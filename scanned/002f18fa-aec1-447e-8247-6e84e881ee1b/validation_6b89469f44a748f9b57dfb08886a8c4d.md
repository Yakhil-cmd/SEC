### Title
Unmetered `canister_inspect_message` Execution Enables Zero-Cost Computational DoS on Subnet Nodes - (File: rs/execution_environment/src/execution/inspect_message.rs)

### Summary
The `canister_inspect_message` system method is executed pre-consensus on every replica node to decide whether to accept an ingress message. This execution consumes real CPU resources proportional to the instruction limit, but **no cycles are charged to any party** — not to the canister, not to the sender. An attacker who deploys a canister with a maximally expensive `canister_inspect_message` that always rejects can send ingress messages at zero net cost, causing all subnet nodes to burn significant CPU time per message.

### Finding Description

The `execute_inspect_message` function in `rs/execution_environment/src/execution/inspect_message.rs` runs the canister's `canister_inspect_message` Wasm method and returns `(NumInstructions, Result<(), UserError>)`. [1](#0-0) 

The caller, `should_accept_ingress_message`, discards the instruction count entirely — it only uses `.1` (the accept/reject result): [2](#0-1) 

No cycle-charging call appears anywhere in this path. The ingress induction cost check that precedes it (lines 3344–3373) is a balance pre-check for the eventual induction fee, which is only charged if the message is **accepted and inducted**. If `canister_inspect_message` rejects the message (i.e., never calls `ic0.accept_message`), the message is never inducted and the canister is never charged: [3](#0-2) 

The instruction limit for inspect-message execution is a configurable ceiling (`max_instructions_for_message_acceptance_calls`), and the metering infrastructure is fully active during this execution: [4](#0-3) 

The `instruction_to_cost` function assigns non-trivial weights to all Wasm instructions, meaning a canister can perform up to hundreds of millions of instructions per inspect-message call: [5](#0-4) 

Because `canister_inspect_message` runs **pre-consensus on every replica node**, one ingress message causes N independent Wasm executions (N = subnet size), amplifying the impact.

### Impact Explanation

An attacker deploys a canister whose `canister_inspect_message` performs the maximum allowed computation and always rejects (never calls `accept_message`). The attacker then floods the subnet with ingress messages targeting this canister. Each message:

1. Triggers a full `canister_inspect_message` execution on every replica node.
2. Is rejected — so the canister pays zero ingress induction cycles.
3. Costs the attacker nothing (users do not pay cycles for sending ingress messages; the induction fee is charged to the receiving canister only on acceptance).

The result is that all subnet nodes continuously burn CPU time executing expensive Wasm, degrading throughput for all other canisters on the subnet. This is a **cycles/resource accounting bug** leading to a **computational denial-of-service** against the subnet.

### Likelihood Explanation

The attack requires only: (a) deploying one canister (one-time cycle cost), and (b) sending ingress messages at whatever rate boundary-node rate limiting allows. Boundary nodes rate-limit by sender identity, not by the computational cost of the inspect-message handler. The attacker can use many identities or stay within per-identity limits while still saturating node CPU. The canister deployment cost is a one-time sunk cost; the ongoing attack cost is zero.

### Recommendation

Charge cycles for `canister_inspect_message` execution against the canister's balance before or after the execution, regardless of whether the message is accepted or rejected. The instruction count returned by `execute_inspect_message` should be used to compute and deduct a cycles fee using the existing `consume_cycles_for_instructions` path, analogous to how update message execution is charged. If the canister cannot cover the cost, the inspect-message execution should be skipped (defaulting to accept) or the message rejected with an out-of-cycles error.

### Proof of Concept

```
1. Deploy a canister with the following canister_inspect_message:
   (func (export "canister_inspect_message")
     ;; Spin for ~max_instructions_for_message_acceptance_calls cycles
     ;; using a tight loop — never call ic0.accept_message
     (loop $spin
       ... expensive computation ...
       br $spin  ;; will be cut off by instruction limit
     )
   )

2. From any user identity, send repeated ingress messages to this canister:
   dfx canister call <canister_id> any_method '()'

3. Observe: each call triggers a full inspect_message execution on every
   replica node. The canister's cycle balance is never decremented.
   The attacker's cost is zero beyond the initial canister deployment.
   Subnet throughput degrades proportionally to the message rate.
```

The root cause is in `rs/execution_environment/src/execution/inspect_message.rs` — the function executes Wasm but returns without charging cycles — and in `rs/execution_environment/src/execution_environment.rs` — the caller discards the instruction count and performs no post-execution accounting.
