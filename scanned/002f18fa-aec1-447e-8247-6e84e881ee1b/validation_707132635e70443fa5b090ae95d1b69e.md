### Title
Unbounded DKG Unvalidated Pool Growth via Remote-Tagged Artifact Flooding — (`rs/artifact_pool/src/dkg_pool.rs`, `rs/consensus/dkg/src/lib.rs`, `rs/replica/setup_ic_network/src/lib.rs`)

### Summary

A Byzantine subnet member (below the fault threshold) can flood the DKG unvalidated pool with arbitrarily many distinct artifacts tagged `target_subnet=Remote(arbitrary_id)` at `start_block_height=current_start_height`. The `DkgBouncer` accepts all of them, the slot table has no limit for DKG, `DkgPoolImpl::insert` has no admission control, and `validate_dealings_for_dealer` silently defers every remote-tagged message whose `NiDkgId` has no matching config — returning `Mutations::new()` without ever removing the artifact. The messages accumulate in the unvalidated pool for the entire DKG interval (typically 500+ blocks), causing unbounded heap growth and eventual OOM crash of honest replicas.

### Finding Description

**Step 1 — No slot table limit for DKG.**

The DKG broadcast channel is registered with `SLOT_TABLE_NO_LIMIT = usize::MAX`: [1](#0-0) [2](#0-1) 

A Byzantine peer can use an unbounded number of distinct slot numbers to advertise N distinct artifacts simultaneously. The slot table enforces a per-peer cap only when `peer_slot_table_len >= self.slot_limit`: [3](#0-2) 

With `slot_limit = usize::MAX`, this branch is never reached for DKG.

**Step 2 — `DkgBouncer` accepts everything at `current_start_height`.**

The bouncer only checks the height field of the `DkgMessageId`, not the content, signer, or `target_subnet`: [4](#0-3) 

Any message whose `id.height == start_height` returns `BouncerValue::Wants`, so all N crafted messages are fetched and forwarded to the pool.

**Step 3 — `DkgPoolImpl::insert` has no admission control.**

The insert path is a direct, unconditional `HashMap` insert with no size check, no per-peer quota, and no content validation: [5](#0-4) 

The key is `DkgMessageId = (crypto_hash(msg), start_block_height)`. Each message with distinct content (different `NiDkgTargetId`, different dealing bytes, or different signature bytes) produces a distinct key, so N distinct messages produce N distinct pool entries. [6](#0-5) 

**Step 4 — `validate_dealings_for_dealer` silently defers remote-tagged messages with unknown configs.**

When `on_state_change` processes the unvalidated pool, it groups messages by `(signer, dkg_id)` and calls `validate_dealings_for_dealer` for each group: [7](#0-6) 

Inside `validate_dealings_for_dealer`, if the message's `dkg_id` is not in the current configs map AND `target_subnet.is_remote()`, the function returns `Mutations::new()` — an empty change set — leaving the artifact in the unvalidated pool with no removal action: [8](#0-7) 

This is intentional for legitimate remote DKG (waiting for the context to appear in state), but it creates an unbounded accumulation window for adversarial messages.

**Step 5 — Purge only fires at DKG interval boundaries.**

The purge action is only emitted when `start_height > dkg_pool.get_current_start_height()`, i.e., when a new DKG summary block is finalized: [9](#0-8) 

The purge removes entries with `id.height < height`, so messages at `current_start_height` survive until the next interval. DKG intervals are typically 500 blocks (~8 minutes). Flooded messages accumulate for the entire interval. [10](#0-9) 

### Impact Explanation

Each `dkg::Message` contains an `NiDkgDealing` with a large cryptographic payload. With N=10,000 messages at ~100 KB each, the unvalidated pool grows by ~1 GB. With N=100,000 messages, it reaches ~10 GB. The replica process is killed by the OS OOM killer, halting consensus on the affected node. If enough honest nodes are targeted simultaneously (still below the Byzantine fault threshold), subnet liveness is violated.

### Likelihood Explanation

The attacker must be a subnet member with a valid TLS node certificate — this is a Byzantine node below the fault threshold, which is explicitly within the IC threat model. No governance majority, no key compromise, and no external infrastructure attack is required. The attack is fully local: craft messages with distinct `NiDkgTargetId` values, send them over the authenticated P2P channel, and wait for the victim's heap to exhaust.

### Recommendation

1. **Add a slot table limit for DKG.** Replace `SLOT_TABLE_NO_LIMIT` with a small constant (e.g., `~2 * subnet_size`) for the DKG broadcast channel, analogous to `SLOT_TABLE_LIMIT_INGRESS` for ingress.
2. **Add admission control in `DkgPoolImpl::insert`.** Enforce a maximum unvalidated pool size (e.g., `MAX_DEALERS * MAX_DKG_CONFIGS_PER_INTERVAL`) and drop artifacts that exceed it.
3. **Eagerly reject remote-tagged messages with unrecognized `dealer_subnet`.** In `validate_dealings_for_dealer`, if `target_subnet.is_remote()` and the `dealer_subnet` does not match the local subnet ID, emit `HandleInvalid` rather than `Mutations::new()`.
4. **Bound the deferred remote DKG set.** Track the number of deferred remote messages per `NiDkgTargetId` and cap it at the maximum number of legitimate dealers.

### Proof of Concept

```rust
// Byzantine node: craft N messages with distinct NiDkgTargetId values
for i in 0..N {
    let target_id = NiDkgTargetId::new([i as u8; 32]); // distinct per iteration
    let dkg_id = NiDkgId {
        start_block_height: current_start_height, // accepted by bouncer
        dealer_subnet: subnet_test_id(999),       // arbitrary
        dkg_tag: NiDkgTag::HighThreshold,
        target_subnet: NiDkgTargetSubnet::Remote(target_id), // triggers deferral
    };
    let msg = craft_dkg_message(dkg_id); // arbitrary dealing bytes, arbitrary signature
    send_via_p2p(victim_node, slot_number=i, msg);
}
// Each message: bouncer returns Wants (height matches), slot table has no limit,
// insert() adds to unvalidated pool, validate_dealings_for_dealer returns Mutations::new().
// After N iterations: pool.get_unvalidated().count() == N, heap grows unboundedly.
// No purge until next DKG summary block (~500 blocks later).
```

    fn purge(&mut self, height: Height) -> Vec<DkgMessageId> {
        self.current_start_height = height;
        // TODO: use drain_filter once it's stable.
        let unvalidated_keys: Vec<_> = self
            .unvalidated
            .keys()
            .filter(|id| id.height < height)
            .cloned()
            .collect();
        for id in unvalidated_keys {
            self.unvalidated.remove(&id);
        }

        let validated_keys: Vec<_> = self
            .validated
            .keys()
            .filter(|id| id.height < height)
            .cloned()
            .collect();
        for hash in &validated_keys {
            self.validated.remove(hash);
        }
        validated_keys
    }
