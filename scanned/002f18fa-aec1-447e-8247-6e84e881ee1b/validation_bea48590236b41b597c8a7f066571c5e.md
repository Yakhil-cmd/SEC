### Title
`deciding_voting_power_adjustment_factor_function` panics when `clear_following_after_seconds` is zero — (`File: rs/nns/governance/src/network_economics.rs`)

---

### Summary

`VotingPowerEconomics::deciding_voting_power_adjustment_factor_function` constructs a `LinearMap` whose `from` range collapses to `begin..begin` when `clear_following_after_seconds` is `0`. `LinearMap::new` unconditionally asserts `from.end != from.start`, causing the NNS governance canister to trap. The `validate()` method for `VotingPowerEconomics` explicitly permits `clear_following_after_seconds = 0`, making this a valid, governance-accepted configuration that silently breaks a core canister function.

---

### Finding Description

`deciding_voting_power_adjustment_factor_function` builds the `from` range for `LinearMap::new` as:

```rust
let begin = self.get_start_reducing_voting_power_after_seconds();
let end   = begin.saturating_add(self.get_clear_following_after_seconds());
begin..end
``` [1](#0-0) 

When `clear_following_after_seconds = 0`, `saturating_add(0)` returns `begin` unchanged, so the range is `begin..begin`. `LinearMap::new` then hits:

```rust
assert!(from.end != from.start, "{from:#?}");
``` [2](#0-1) 

This is an unconditional panic (canister trap). Notably, `apply` already contains a guard for this exact case (`if from.end == from.start { Decimal::ONE }`), but `new` does not — it asserts instead. [3](#0-2) 

The `validate()` method for `VotingPowerEconomics` explicitly documents that zero is a permitted value:

> "They are allowed to be set to 0 though." [4](#0-3) 

There is no check that `clear_following_after_seconds > 0` anywhere in the validation path, so a governance proposal setting it to `0` passes all validation and is accepted into state.

The panicking function is called from `rs/nns/governance/src/neuron/types.rs` during voting-power calculations: [5](#0-4) 

---

### Impact Explanation

Once `clear_following_after_seconds = 0` is committed to governance state, every subsequent call that computes `deciding_voting_power_adjustment_factor` causes the NNS governance canister to trap. Voting-power computation is on the critical path for proposal voting and neuron management. A persistent trap in this path can render the NNS governance canister non-functional for all users, with no self-healing path because the canister cannot process further proposals to fix the parameter.

**Impact: High** — governance canister availability and correctness are broken for all participants.

---

### Likelihood Explanation

**Likelihood: Low** — triggering the bug requires a governance proposal that sets `clear_following_after_seconds` to `0`. This is not a malicious act; the validation code explicitly permits the value. A well-intentioned governance proposal (e.g., "disable the following-clearing grace period") could set this field to `0` without realising it causes a panic. The prerequisite is a governance majority, not an unprivileged ingress sender, which keeps likelihood low.

---

### Recommendation

1. **In `LinearMap::new`**: replace the hard `assert` with a graceful error or, consistent with the existing `apply` guard, silently allow `from.end == from.start` by treating it as a degenerate constant map.

```rust
// Instead of:
assert!(from.end != from.start, "{from:#?}");

// Either return a Result, or handle the degenerate case:
if from.end == from.start {
    // treat as constant map; apply will already handle this correctly
}
``` [2](#0-1) 

2. **In `deciding_voting_power_adjustment_factor_function`**: guard against the degenerate range before constructing `LinearMap`:

```rust
if begin == end {
    return /* constant LinearMap or early return of 1.0 */;
}
``` [1](#0-0) 

3. **In `VotingPowerEconomics::validate`**: add an explicit check that `clear_following_after_seconds > 0`, or document that `0` is intentionally unsupported and reject it. [6](#0-5) 

---

### Proof of Concept

1. Submit a governance proposal (`ManageNetworkEconomics`) setting `voting_power_economics.clear_following_after_seconds = 0`. This passes `VotingPowerEconomics::validate()` because the field is `Some(0)` (set, not `None`) and no lower-bound check exists.
2. After the proposal is adopted, any code path that calls `deciding_voting_power_adjustment_factor` (e.g., computing a neuron's deciding voting power during a vote) invokes `deciding_voting_power_adjustment_factor_function`, which calls `LinearMap::new(begin..begin, 1..0)`.
3. `LinearMap::new` executes `assert!(from.end != from.start)` → assertion fails → canister traps.
4. All subsequent voting-power-dependent operations on the NNS governance canister trap, making the canister non-functional. [7](#0-6) [8](#0-7)
