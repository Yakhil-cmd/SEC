### Title
Semi-Erroneous Median Picks Upper-Middle Element for Even-Length Reward Arrays — (`rs/ethereum/cketh/minter/src/tx.rs`)

### Summary
The private `median` function in the ckETH minter uses `values.len() / 2` as the selection index via `select_nth_unstable`. For even-length slices this always resolves to the upper-middle element rather than the average of the two middle elements, producing an upward-biased `max_priority_fee_per_gas` estimate whenever the flattened reward array has an even number of entries.

### Finding Description
`estimate_transaction_fee` collects all per-block priority-fee rewards from the EVM fee history into a flat `Vec<WeiPerGas>` and passes it to `median`:

```rust
// rs/ethereum/cketh/minter/src/tx.rs  lines 708-718
let mut rewards: Vec<WeiPerGas> = fee_history
    .reward
    .iter()
    .flatten()
    .map(|nat| WeiPerGas::from(nat.clone()))
    .collect();
let historic_max_priority_fee_per_gas =
    *median(&mut rewards).ok_or(...)?;
```

The helper:

```rust
// rs/ethereum/cketh/minter/src/tx.rs  lines 736-742
fn median<T: Ord>(values: &mut [T]) -> Option<&T> {
    if values.is_empty() {
        return None;
    }
    let (_, item, _) = values.select_nth_unstable(values.len() / 2);
    Some(item)
}
```

`select_nth_unstable(n)` partially sorts the slice so that the element at index `n` is in its correct sorted position. For a slice of length `L`:

| L (parity) | index used (`L/2`) | correct median index(es) | result |
|---|---|---|---|
| odd (e.g. 5) | 2 | 2 | ✓ correct |
| even (e.g. 4) | 2 | avg(1, 2) | ✗ upper-middle only |

The total number of reward entries equals `blockCount × len(rewardPercentiles)`. If either factor makes the product even (e.g. 4 blocks × 1 percentile, or 5 blocks × 2 percentiles), the function silently returns the higher of the two middle values instead of their average. [1](#0-0) [2](#0-1) 

### Impact Explanation
`max_priority_fee_per_gas` is set to `historic_max_priority_fee_per_gas.max(MIN_MAX_PRIORITY_FEE_PER_GAS)`. An upward-biased median inflates the priority fee used in every ckETH/ckERC-20 withdrawal transaction. The `max_fee_per_gas` is then computed as `2 × base_fee + max_priority_fee`, so the bias propagates directly into the fee cap paid on-chain. In the worst case (two adjacent even-spaced reward values), the overestimate matches the NFTFloorOracle proof-of-concept: up to ~20% above the true median. This causes ckETH users to overpay transaction fees and may cause the minter to burn more ckETH than necessary for fee reimbursement. [3](#0-2) 

### Likelihood Explanation
The current production call requests rewards for the **last 5 blocks** (odd count) with a single percentile, yielding 5 reward entries — an odd length where the bug is dormant. However, the bug activates the moment either the block count or the number of requested percentiles is changed to produce an even total, or if the EVM RPC returns a different number of entries. The function carries no guard or comment documenting the odd-only assumption, making a future configuration change a realistic trigger. The `query_stats` aggregator in the same codebase correctly handles the even case by explicitly branching on `is_multiple_of(2)`, confirming the pattern is known. [4](#0-3) [1](#0-0) 

### Recommendation
Mirror the correct pattern already used in `get_median` (`rs/query_stats/src/state_machine.rs`):

```rust
fn median(values: &mut [WeiPerGas]) -> Option<WeiPerGas> {
    if values.is_empty() {
        return None;
    }
    values.sort_unstable();
    let mid = values.len() / 2;
    if values.len() % 2 == 0 {
        // average the two middle elements to avoid upward bias
        Some((values[mid - 1] + values[mid]) / WeiPerGas::new(2))
    } else {
        Some(values[mid])
    }
}
```

Add a unit test covering an even-length reward array to lock in the correct behaviour.

### Proof of Concept
Suppose the fee history returns 4 reward entries (e.g. block count changed to 4):

```
rewards (sorted) = [1_000_000_000, 1_500_000_000, 2_250_000_000, 3_375_000_000]
```

Current code: `select_nth_unstable(4/2 = 2)` → returns `2_250_000_000` wei/gas.  
Correct median: `(1_500_000_000 + 2_250_000_000) / 2 = 1_875_000_000` wei/gas.

The minter overstates `max_priority_fee_per_gas` by **375 000 000 wei/gas (~20%)**, inflating every withdrawal's `max_fee_per_gas` and causing users to overpay on-chain fees. [1](#0-0) [5](#0-4)

fn get_median<T: Default + Ord + Copy + Add<Output = T> + Div<Output = T> + From<u8>, F>(
    stats: &[&QueryStats],
    f: F,
) -> T
where
    F: FnMut(&&QueryStats) -> T,
{
    let mut values: Vec<T> = stats.iter().map(f).collect();
    values.sort_unstable();
    let mid = values.len() / 2;

    if values.len().is_multiple_of(2) {
        let left = values
            .get(mid.saturating_sub(1))
            .cloned()
            .unwrap_or(T::default());
        let right = values.get(mid).cloned().unwrap_or(T::default());
        (left + right) / 2_u8.into()
    } else {
        values.get(mid).cloned().unwrap_or(T::default())
    }
}
