### Title
Incomplete OFAC Blocklist Check in ckBTC Minter `retrieve_btc` — Caller's Bitcoin Deposit Address Not Validated Against Blocklist - (File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs)

### Summary

The ckBTC minter's `retrieve_btc` and `retrieve_btc_with_approval` functions only check the **destination** Bitcoin address against the OFAC blocklist, not the **caller's** associated Bitcoin deposit address. The blocklist comment explicitly states the intent is bidirectional ("BTC is not accepted from nor sent to addresses on this list"), but the withdrawal path enforces only the "sent to" half. A user whose derived Bitcoin deposit address is on the OFAC blocklist can still call `retrieve_btc` to a clean destination address and successfully withdraw ckBTC, bypassing OFAC compliance.

### Finding Description

The `BTC_ADDRESS_BLOCKLIST` in `rs/bitcoin/checker/lib/blocklist.rs` carries the comment:

> "BTC is not accepted from nor sent to addresses on this list." [1](#0-0) 

This bidirectional intent is also reflected in the Ethereum blocklist: [2](#0-1) 

The ckBTC minter derives a unique Bitcoin deposit address for each IC account via `runtime.derive_user_address`. During **deposit** (`update_balance`), the minter calls `check_transaction` which checks the UTXO input addresses (the Bitcoin addresses that funded the user's deposit address) against the blocklist. If any input address is blocked, the UTXO is quarantined and ckBTC is not minted. [3](#0-2) 

However, during **withdrawal** (`retrieve_btc`), only the destination address (`args.address`) is checked via `check_address`. The caller's derived Bitcoin deposit address — which is derivable from their IC principal via `runtime.derive_user_address` — is never checked: [4](#0-3) 

The same omission exists in `retrieve_btc_with_approval`: [5](#0-4) 

The `check_address` helper only validates the single address passed to it: [6](#0-5) 

The `check_address` endpoint on the Bitcoin checker canister itself is a simple blocklist lookup: [7](#0-6) 

The minter's `derive_user_address` capability is available on the `CanisterRuntime` trait but is never invoked in the withdrawal path for blocklist screening: [8](#0-7) 

### Impact Explanation

A user whose derived Bitcoin deposit address appears on the OFAC SDN list can still call `retrieve_btc` or `retrieve_btc_with_approval` with any clean destination address and successfully convert ckBTC back to BTC. The minter will only reject the request if the **destination** is blocked, not if the **caller's own deposit address** is blocked. This directly mirrors the ERC1404 analog: the "from" side of the restriction is unenforced. The consequence is a regulatory compliance gap — OFAC-sanctioned entities can exit the ckBTC system freely as long as they route to a non-sanctioned destination address.

### Likelihood Explanation

The attacker entry path is a standard unprivileged ingress call to `retrieve_btc` on the ckBTC minter canister. No special privileges are required. The scenario requires that a user's derived Bitcoin deposit address is present on the OFAC SDN list — either because the address was added to the list after the user already deposited and received ckBTC, or because the minter's deposit-time UTXO check did not catch a tainted deposit (e.g., the blocklist was stale at deposit time). Given that the OFAC SDN list is updated periodically and the ckBTC minter's blocklist is only updated via NNS upgrade proposals, there is a realistic window during which a newly sanctioned address holder retains spendable ckBTC and can withdraw it.

### Recommendation

In both `retrieve_btc` and `retrieve_btc_with_approval`, after obtaining the `caller`, derive the caller's Bitcoin deposit address using `runtime.derive_user_address` and call `check_address` (or the equivalent `check_transaction` path) on it before proceeding with the withdrawal. If the caller's deposit address is found to be blocked, return an appropriate error (analogous to `RetrieveBtcError::GenericError` with `ErrorCode::TaintedAddress`) and do not process the withdrawal.

### Proof of Concept

1. User's derived Bitcoin deposit address `A` (derived from their IC principal) is added to the OFAC SDN list.
2. User already holds ckBTC minted from a prior deposit (before the blocklist update, or via a UTXO whose inputs were not blocked at deposit time).
3. User calls `retrieve_btc` with a clean destination address `B` (not on the blocklist).
4. `retrieve_btc` calls `check_address(btc_checker_principal, B, runtime)` → returns `Clean`.
5. The minter burns the user's ckBTC and schedules a Bitcoin transaction to `B`.
6. Address `A` (the caller's blocked deposit address) was never checked — the withdrawal succeeds in violation of OFAC compliance intent.

The deposit-time test `should_block_deposit_from_blocked_address` (ckETH analog) demonstrates that the system does enforce the "from" direction at deposit time, confirming the asymmetry is not intentional: [9](#0-8) 

No equivalent test exists for the withdrawal path checking the caller's own address, confirming the gap.

#[ic_cdk::query]
fn check_address(args: CheckAddressArgs) -> CheckAddressResponse {
    let config = get_config();
    let btc_network = config.btc_network();
    let address = Address::from_str(args.address.trim())
        .unwrap_or_else(|err| ic_cdk::trap(format!("Invalid Bitcoin address: {err}")))
        .require_network(btc_network.clone().into())
        .unwrap_or_else(|err| ic_cdk::trap(format!("Not a Bitcoin {btc_network} address: {err}")));

    match config.check_mode {
        CheckMode::AcceptAll => CheckAddressResponse::Passed,
        CheckMode::RejectAll => CheckAddressResponse::Failed,
        CheckMode::Normal => {
            if is_blocked(&address) {
                return CheckAddressResponse::Failed;
            }
            CheckAddressResponse::Passed
        }
    }
}
