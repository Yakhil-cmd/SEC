### Title
NeuronManagement Followees Can Replace Themselves via ManageNeuron Proposal — (`rs/nns/governance/src/pb/mod.rs`, `rs/nns/governance/src/governance.rs`)

### Summary

A neuron N's NeuronManagement followees can collectively submit and adopt a `ManageNeuron{SetFollowing{topic: NeuronManagement, followees: [attacker]}}` proposal, which executes with the controller's identity, bypassing the explicit-controller-consent requirement for NeuronManagement following changes. The attacker gains sole control over all future ManageNeuron proposals targeting neuron N.

### Finding Description

**Step 1 — Proposal submission skips authorization.**

`validate_manage_neuron_proposal` handles `SetFollowing` by calling only `validate_intrinsically()`: [1](#0-0) 

`validate_intrinsically` checks only topic uniqueness and followee count — it never checks whether the caller is authorized to change NeuronManagement followees: [2](#0-1) 

**Step 2 — Ballots are computed from the current NeuronManagement followees.**

`compute_ballots_for_manage_neuron_proposal` gives voting power only to the existing NeuronManagement followees of the target neuron: [3](#0-2) 

This means the attacker (already a followee) and their co-followees can adopt the proposal without the controller's involvement.

**Step 3 — Execution impersonates the controller.**

`perform_action` explicitly fetches the managed neuron's controller and passes it as the `caller` to `manage_neuron`: [4](#0-3) 

**Step 4 — `validate_authorized` passes because the impersonated caller IS the controller.**

`validate_authorized` requires `neuron.is_controlled_by(caller)` when the NeuronManagement topic is involved: [5](#0-4) 

Since `caller` was set to the controller in Step 3, this check trivially passes — even though the actual human controller never signed or consented to the change.

**The invariant stated in the code is violated.** The comment at line 71–74 of `mod.rs` explicitly states that NeuronManagement topic changes require the controller directly: [6](#0-5) 

### Impact Explanation

An attacker who is one of N NeuronManagement followees of target neuron N can:
1. Submit a `SetFollowing` proposal replacing all NeuronManagement followees with `[attacker_neuron]`.
2. Gather votes from the existing followee set (including themselves) to adopt it.
3. After execution, become the sole NeuronManagement followee.
4. From that point, unilaterally adopt any future ManageNeuron proposal against neuron N (e.g., `Disburse`, `Split`, `Configure`, `DisburseToNeuron` for not-for-profit neurons, etc.).

The original controller never explicitly consented to the followee replacement.

### Likelihood Explanation

The attacker must already be a NeuronManagement followee of the target neuron (a privileged-but-not-controller role). This is a realistic precondition: neuron owners routinely delegate NeuronManagement following to trusted parties. The attack requires no key compromise, no subnet majority, and no external dependencies — it is fully exercisable via standard governance ingress calls.

### Recommendation

In `validate_manage_neuron_proposal`, add an explicit check that rejects `SetFollowing` commands that include the `NeuronManagement` topic. Specifically, after calling `validate_intrinsically()`, check whether any `FolloweesForTopic` entry targets `Topic::NeuronManagement` and return `NotAuthorized` if so — mirroring the restriction already enforced in `validate_authorized`. Alternatively, treat `SetFollowing{NeuronManagement}` as a command that cannot be issued through the proposal mechanism (similar to how `MakeProposal` is blocked at line 4628–4633). [7](#0-6) 

### Proof of Concept

```
1. Create neuron N with controller C.
2. Set NeuronManagement followees of N to [A, B].
3. As A, call manage_neuron(MakeProposal(ManageNeuron{
       target: N,
       command: SetFollowing{
           topic_following: [{topic: NeuronManagement, followees: [A]}]
       }
   }))
   → validate_manage_neuron_proposal calls validate_intrinsically() only → proposal accepted.
4. A and B register_vote(Yes) → proposal adopted (both are followees, each has 1 VP).
5. perform_action fetches controller C, calls manage_neuron(&C, SetFollowing{...}).
6. set_following → validate() → validate_authorized(&C, neuron) → C.is_controlled_by(C) = true → passes.
7. Assert: neuron N's NeuronManagement followees = [A] only.
8. Assert: C never signed any SetFollowing command.
9. As A alone, now submit and adopt any further ManageNeuron proposal against N.
```
