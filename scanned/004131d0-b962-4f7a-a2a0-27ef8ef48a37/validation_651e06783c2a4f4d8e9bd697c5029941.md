### Title
Unbounded Synchronous BFS in SNS Governance `cast_vote_and_cascade_follow` Lacks Instruction-Limit Protection, Enabling Governance DoS — (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `cast_vote_and_cascade_follow` is a **synchronous, static function** that traverses the entire neuron-following graph via BFS with **no instruction-limit check or async continuation**. The NNS governance has an equivalent function that is `async`, uses explicit soft/hard instruction limits, and can make noop self-calls to spread work across rounds. The SNS version has none of these safeguards. With `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` and up to 15 followees per function/topic per neuron, a sufficiently large following graph can exhaust the per-message instruction budget during a `register_vote` call, causing the canister to trap, rolling back the vote and permanently blocking governance if the condition persists.

---

### Finding Description

**NNS governance** (`rs/nns/governance/src/voting.rs`) defines `cast_vote_and_cascade_follow` as an `async` method with explicit soft and hard instruction limits:

```rust
const HARD_VOTING_INSTRUCTIONS_LIMIT: u64 = 750 * BILLION;
const SOFT_VOTING_INSTRUCTIONS_LIMIT: u64 = BILLION; // 1B in prod
```

It loops, checking `over_soft_message_limit()` after each neuron, and calls `noop_self_call_if_over_instructions(...)` to yield and continue in a subsequent round via a timer. [1](#0-0) [2](#0-1) 

**SNS governance** (`rs/sns/governance/src/governance.rs`) defines `cast_vote_and_cascade_follow` as a **synchronous static function**:

```rust
fn cast_vote_and_cascade_follow(
    proposal_id: &ProposalId,
    voting_neuron_id: &NeuronId,
    ...
    ballots: &mut BTreeMap<String, Ballot>,
    topic: Topic,
) {
    ...
    while !induction_votes.is_empty() {   // ← no instruction check
        ...
    }
}
```

There is no `is_over_instructions_limit()` guard, no async yield, and no timer-based continuation. [3](#0-2) [4](#0-3) 

`register_vote` calls this synchronously and returns immediately after:

```rust
Governance::cast_vote_and_cascade_follow(
    proposal_id, neuron_id, vote, function_id,
    &self.function_followee_index, &self.topic_follower_index,
    &self.proto.neurons, now_seconds,
    &mut proposal.ballots, proposal_topic.unwrap_or_default(),
);
self.process_proposal(proposal_id.id);
``` [5](#0-4) 

**Attacker-controlled entry path**: Any principal with a neuron that has `Vote` permission can call `manage_neuron { command: RegisterVote { ... } }` on the SNS governance canister. This is an unprivileged ingress call.

**Resource bound analysis**:
- `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` [6](#0-5) 
- `MAX_FOLLOWEES_PER_FUNCTION_CEILING = 15` (per function) [7](#0-6) 
- `MAX_FOLLOWEES_PER_TOPIC = 15` (per topic, for the newer topic-based following) [8](#0-7) 

With 200,000 neurons each following 15 others across both legacy function-based and topic-based indexes, the BFS can visit all 200,000 neurons. Each visit requires BTreeMap lookups into `neurons`, `function_followee_index`, and `topic_follower_index`, plus `vote_from_ballots_following` which itself iterates over followees. The IC per-message instruction limit for update calls is ~5 billion instructions; a full traversal of 200,000 neurons with these operations approaches or exceeds that budget.

If the instruction limit is hit, the canister **traps**, all state changes are rolled back, and the vote is never recorded. If the following graph is permanently large enough to always trigger this, **no vote can ever be registered**, permanently DoS-ing the SNS governance.

---

### Impact Explanation

- **Governance DoS**: If the following graph is large enough, every `register_vote` call traps. No neuron can vote directly; proposals cannot reach quorum or be decided. The SNS is effectively frozen until a canister upgrade is deployed.
- **Griefing**: An attacker who stakes enough tokens to create many neurons and sets up a maximally deep/wide following graph can trigger this condition without needing to profit from it.
- **Scope**: Affects the SNS governance canister, which controls upgrades, treasury, and all SNS parameters.

---

### Likelihood Explanation

- Creating 200,000 neurons requires staking the SNS token for each, which has a real economic cost. However, for SNS instances with high token supply or low token price, this cost may be feasible.
- Legitimate organic growth of an SNS to tens of thousands of neurons with complex following relationships (e.g., via swap neuron baskets that now use topic-based following) can also approach this limit without any attacker involvement.
- The recently added topic-based following (`SetFollowing`) multiplies the number of following edges per neuron, increasing BFS cost. [9](#0-8) 

---

### Recommendation

1. **Mirror the NNS pattern**: Make SNS `cast_vote_and_cascade_follow` async and add soft/hard instruction-limit checks with a `VotingStateMachine` and timer-based continuation, exactly as done in `rs/nns/governance/src/voting.rs`.
2. **Add a per-proposal BFS neuron cap**: Limit the number of neurons processed in a single BFS round, deferring the rest to a timer.
3. **Benchmark the current synchronous path** against the maximum neuron count to confirm whether the instruction limit is actually reachable with current parameters.

---

### Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_followees_per_function = 15`.
2. Create N neurons (e.g., 100,000+) and configure them in a chain: neuron[i] follows neuron[i-1] on every registered function and every topic.
3. Submit a proposal. All neurons receive ballots.
4. Have neuron[0] call `manage_neuron { RegisterVote { proposal, vote: Yes } }`.
5. `register_vote` calls `cast_vote_and_cascade_follow` synchronously; the BFS traverses all N neurons in a single message.
6. Observe the call trap due to instruction exhaustion; the vote is rolled back.
7. Repeat: no vote can ever be registered while the following graph remains this large.

The NNS governance avoids this by checking `over_soft_message_limit()` after each neuron and using `noop_self_call_if_over_instructions` to yield. [10](#0-9)  The SNS governance has no equivalent protection. [4](#0-3)

```rust
        let topic_followers = topic_follower_index.get(&topic);

        // Traverse the follow graph using breadth first search (BFS).

        // Each "tier" in the BFS is listed here. Of course, the first tier just
        // contains the original "triggering" ballot.
        let mut induction_votes = BTreeMap::new();
        induction_votes.insert(voting_neuron_id.to_string(), vote_of_neuron);

        // Each iteration of this loop processes one tier in the BFS.
        //
        // This has to terminate, because if we keep going around in a cycle, that
        // means the same neuron keeps getting swayed, but once a neuron is swayed,
        // it does not matter how its "other" followees vote (i.e. those that have
        // not (directly or indirectly) voted yet). That is, once a neuron is swayed,
        // its vote is "locked in". IOW, swaying is "monotonic".
        while !induction_votes.is_empty() {
            // This will be populated with the followers of neurons in the
            // current BFS tier, who might be swayed to indirectly vote, thus
            // forming the next tier in the BFS.
            let mut follower_neuron_ids = BTreeSet::new();

            // Process the current tier in the BFS.
            for (current_neuron_id, current_new_vote) in &induction_votes {
                let current_ballot = match ballots.get_mut(current_neuron_id) {
                    Some(b) => b,
                    None => {
                        // neuron_id has no (blank) ballot, which means they
                        // were not eligible when the proposal was first
                        // created. This is fairly unusual, but does not
                        // indicate a bug (therefore, no log).
                        continue;
                    }
                };

                // Only fill in "blank" ballots. I.e. those with vote ==
                // Unspecified. This check could just as well be done before
                // current_neuron_id is added to induction_votes.
                if current_ballot.vote != (Vote::Unspecified as i32) {
                    continue;
                }

                // Fill in current_ballot.
                assert_ne!(*current_new_vote, Vote::Unspecified);
                current_ballot.vote = *current_new_vote as i32;
                current_ballot.cast_timestamp_seconds = now_seconds;

                // Take note of the followers of current_neuron_id, and add them
                // to the next "tier" in the BFS.

                if let Some(new_follower_neuron_ids) = topic_followers
                    .and_then(|topic_followers| topic_followers.get(current_neuron_id))
                {
                    for follower_neuron_id in new_follower_neuron_ids {
                        follower_neuron_ids.insert(follower_neuron_id.clone());
                    }
                }

                if let Some(new_follower_neuron_ids) =
                    neuron_id_to_follower_neuron_ids.get(current_neuron_id)
                {
                    for follower_neuron_id in new_follower_neuron_ids {
                        follower_neuron_ids.insert(follower_neuron_id.clone());
                    }
                }
            }

            // Prepare for the next iteration of the (outer most) loop by
            // constructing the next BFS tier (from follower_neuron_ids).
            induction_votes.clear();
            for follower_neuron_id in follower_neuron_ids {
                let Some(follower_neuron) = neurons.get(&follower_neuron_id.to_string()) else {
                    // This is a highly suspicious, because currently, we do not
                    // delete neurons, which means that we have an invalid NeuronId
                    // floating around in the system, which indicates that we have a
                    // bug. For now, we deal with that by logging, and pretending like
                    // we did not see follower_neuron_id.
                    log!(
                        ERROR,
                        "Missing neuron {} while trying to record (and cascade) \
                            a vote on proposal {:#?}.",
                        follower_neuron_id,
                        proposal_id,
                    );
                    continue;
                };

                let follower_vote = follower_neuron.vote_from_ballots_following(
                    function_id,
                    topic,
                    ballots,
                    proposal_id,
                );

                if follower_vote != Vote::Unspecified {
                    // follower_neuron would be swayed by its followees!
                    //
                    // This is the other (earlier) point at which we could
                    // consider whether a neuron is already locked in, and that
                    // no recursion is needed.
                    induction_votes.insert(follower_neuron_id.to_string(), follower_vote);
                }
            }
        }
