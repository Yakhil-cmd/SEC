### Title
Markdown Injection via Unsanitized ICRC-1 Memo in ICRC-21 GenericDisplay Consent Messages - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The `add_memo` function in the shared ICRC-21 consent-message library interpolates attacker-controlled ICRC-1 memo bytes directly into a Markdown code span without escaping backtick characters or newlines. Any unprivileged caller can craft a transfer whose memo breaks out of the code span and injects fake Markdown fields (e.g., a spoofed `**Amount:**` or `**To:**` section) into the `GenericDisplayMessage` returned by `icrc21_canister_call_consent_message`. Wallets that render this message to obtain user approval will display content that does not accurately represent the actual transaction being signed.

---

### Finding Description

`ConsentMessage::add_memo` in `packages/icrc-ledger-types/src/icrc21/responses.rs` handles the `GenericMemo::Icrc1Memo` variant as follows:

```rust
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
match self {
    ConsentMessage::GenericDisplayMessage(message) => {
        message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
    }
``` [1](#0-0) 

If the memo bytes are valid UTF-8, `memo_str` is used verbatim. The value is placed inside a Markdown inline code span (`` `{memo_str}` ``), but **backtick characters inside `memo_str` are not escaped**. A single backtick in the memo terminates the code span early; any content after it is rendered as ordinary Markdown. A newline character (`\n`) inside the memo further allows the injected content to start new Markdown block elements (headings, bold fields, etc.).

This function is called unconditionally from `ConsentMessageBuilder::build` whenever a memo is present:

```rust
if let Some(memo) = self.memo {
    message.add_memo(memo);
}
``` [2](#0-1) 

The builder is populated with the caller-supplied `TransferArg.memo` for `icrc1_transfer`, `icrc2_approve`, and `icrc2_transfer_from` calls:

```rust
if let Some(memo) = memo {
    display_message_builder =
        display_message_builder.with_memo(GenericMemo::Icrc1Memo(memo.0));
}
display_message_builder.build()
``` [3](#0-2) 

The same shared library is used by the ICP ledger's `icrc21_canister_call_consent_message` endpoint: [4](#0-3) 

By contrast, the ckBTC and ckDOGE minters explicitly document and defend against this exact class of injection by validating that the destination address parses as a structurally constrained value before interpolation: [5](#0-4) 

No equivalent sanitization exists for the memo field.

---

### Impact Explanation

The `GenericDisplayMessage` is the human-readable string that ICRC-21-compliant wallets (including hardware wallets such as Ledger) display to users before they authorize a canister call. If this string is misleading, the user may approve a transaction whose actual on-chain effect differs from what they believe they are signing.

A crafted memo such as:

```
`\n\n**Amount:** `1000 ICP`\n\n**To:**\n`attacker-account`
```

produces a `GenericDisplayMessage` that, after the real fields, renders injected bold headers `**Amount:**` and `**To:**` with attacker-chosen values. On paginated hardware-wallet displays, the injected page may be the only one the user reads. On GUI wallets that render Markdown, the injected section visually mimics the legitimate fields generated by `add_amount` and `add_account`.

The actual transaction executed on-chain is determined by the `arg` blob, not the consent message. However, the entire purpose of ICRC-21 is to bridge this gap by providing a trustworthy human-readable description. Injecting false fields into that description defeats the security guarantee the standard is designed to provide.

---

### Likelihood Explanation

The ICRC-1 memo field accepts arbitrary bytes up to 32 bytes. Any caller of `icrc1_transfer`, `icrc2_approve`, or `icrc2_transfer_from` on any ledger that uses the shared `build_icrc21_consent_info` library can supply a memo. No privilege, stake, or governance role is required. The attacker only needs to call `icrc21_canister_call_consent_message` with a `TransferArg` whose memo contains a backtick followed by newline-separated Markdown. This is a single unauthenticated query call (the endpoint is `#[update]` on the ICP ledger but callable by anyone). The injection is deterministic and requires no brute-force or timing.

---

### Recommendation

1. **Escape backticks and newlines in `memo_str` before interpolation.** Replace every `` ` `` with `` \` `` (or use a double-backtick span) and strip or replace `\n`/`\r` characters before embedding in the Markdown string.
2. **Alternatively, hex-encode all memo bytes unconditionally** in the `GenericDisplayMessage` path, reserving the UTF-8 display only for the `FieldsDisplayMessage` path where the value is a structured `Value::Text` field and Markdown rendering does not apply.
3. **Add a test analogous to the ckBTC/ckDOGE injection tests** (which already cover this class for addresses) for the memo field, verifying that a memo containing `` ` ``, `\n`, and `**bold**` does not produce injected Markdown sections in the output.

---

### Proof of Concept

Call `icrc21_canister_call_consent_message` on any ICRC-1 ledger (e.g., the ICP ledger) with:

```
method = "icrc1_transfer"
arg = Candid-encode(TransferArg {
    from_subaccount: None,
    to: <legitimate recipient>,
    fee: None,
    amount: 1_000,          // 0.00001 ICP
    created_at_time: None,
    memo: Some(Memo::from(
        b"`\n\n**Amount:** `999999999 ICP`\n\n**To:**\n`attacker-account`".to_vec()
    )),
})
user_preferences.device_spec = GenericDisplay
```

The returned `GenericDisplayMessage` will contain the real fields followed by the injected Markdown:

```
# Send Internet Computer

...

**Amount:** `0.00001 ICP`

**To:**
`legitimate-recipient`

...

**Memo:**
`
**Amount:** `999999999 ICP`

**To:**
`attacker-account`
```

A wallet rendering this Markdown will display two `**Amount:**` and two `**To:**` sections; the injected ones appear last and show attacker-chosen values, potentially causing the user to approve under false pretenses. [6](#0-5) [7](#0-6)

    pub fn add_memo(&mut self, memo: GenericMemo) {
        match memo {
            GenericMemo::Icrc1Memo(memo) => {
                // Check if the memo is a valid UTF-8 string and display it as such if it is.
                let memo_str = match std::str::from_utf8(memo.as_slice()) {
                    Ok(valid_str) => valid_str.to_string(),
                    Err(_) => hex::encode(memo.as_slice()),
                };
                match self {
                    ConsentMessage::GenericDisplayMessage(message) => {
                        message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
                    }
                    ConsentMessage::FieldsDisplayMessage(fields_display) => fields_display
                        .fields
                        .push(("Memo".to_string(), Value::Text { content: memo_str })),
                }
            }
            GenericMemo::IntMemo(memo) => match self {
                ConsentMessage::GenericDisplayMessage(message) => {
                    message.push_str(&format!("\n\n**Memo:**\n`{memo}`"));
                }
                ConsentMessage::FieldsDisplayMessage(fields_display) => {
                    fields_display.fields.push((
                        "Memo".to_string(),
                        Value::Text {
                            content: memo.to_string(),
                        },
                    ))
                }
            },
        };
    }

fn icrc21_canister_call_consent_message(

            memo,

            fee,
            from_subaccount,
            to,
            created_at_time: _,
        } = Decode!(&consent_msg_request.arg, TransferArgs).map_err(|e| {
            Icrc21Error::UnsupportedCanisterCall(ErrorInfo {

            })
        })?;

        let from = if caller() == Principal::anonymous() {
            AccountOrId::AccountIdAddress(None)

                description: format!("Failed to parse receiver account id: {e}"),

            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
            Some(args),
        )
    } else {
        build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints(
            consent_msg_request,
            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
        )
    }

/// Verifies that `address` parses as a valid Bitcoin address on the configured
/// network before it gets interpolated into a consent message. This both
/// guarantees the user is shown a meaningful (parseable) destination and rules
/// out Markdown-injection vectors in the GenericDisplay output (e.g. an
/// "address" that contains newlines or backticks crafted to fake additional
/// fields). Uses the same parser as `retrieve_btc_with_approval`, so any
/// address the consent endpoint accepts is also accepted by the actual call.
fn validate_address(address: &str, network: Network) -> Result<(), Icrc21Error> {
    BitcoinAddress::parse(address, network).map_err(|e| {
        Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
            description: format!("Invalid Bitcoin destination address: {e}"),
        })
    })?;
    Ok(())
}
