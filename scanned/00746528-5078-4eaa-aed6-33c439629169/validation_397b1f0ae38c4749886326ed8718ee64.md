### Title
Stale ICP/XDR Exchange Rate Used for Cycles Minting Without Freshness Check - (`rs/nns/cmc/src/main.rs`)

### Summary
The Cycles Minting Canister (CMC) converts ICP to cycles using a cached `icp_xdr_conversion_rate` that is updated asynchronously via heartbeat (every 5 minutes via the Exchange Rate Canister). The `tokens_to_cycles` function, called by `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles`, reads this cached rate directly without any staleness check. If the heartbeat-driven rate refresh fails persistently (e.g., XRC is unavailable, rate validation fails, or the CMC is freshly deployed), the CMC silently falls back to the hardcoded default rate from May 2021 (`1 ICP = 100 XDR`, timestamp `1_620_633_600`) and mints cycles at that stale price indefinitely.

### Finding Description
The `tokens_to_cycles` function in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` and uses it directly for all cycle-minting operations:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...)
        }
    })
}
```

There is no check on `rate.timestamp_seconds` to verify the rate is recent before using it. The only guard is `None` (which never occurs in practice because the default state always initializes `icp_xdr_conversion_rate` to `Some(...)`).

The default initial rate is hardcoded at canister initialization:

```rust
let initial_icp_xdr_conversion_rate = IcpXdrConversionRate {
    timestamp_seconds: DEFAULT_ICP_XDR_CONVERSION_RATE_TIMESTAMP_SECONDS, // 10 May 2021
    xdr_permyriad_per_icp: DEFAULT_XDR_PERMYRIAD_PER_ICP_CONVERSION_RATE, // 1_000_000 (100 XDR/ICP)
};
```

The rate is refreshed asynchronously via `canister_heartbeat` → `update_exchange_rate` → XRC call, but only every 5 minutes (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`). If the XRC is unavailable or returns invalid data, the CMC retries at the next minute but continues serving `notify_top_up`/`notify_create_canister`/`notify_mint_cycles` calls using the last known (potentially very old) rate.

The three public entry points that call `tokens_to_cycles` — `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` — are all callable by any unprivileged ingress sender.

### Impact Explanation
**Cycles conservation bug / ledger conservation bug.** If the cached rate is stale and significantly lower than the current market rate (e.e., the default 100 XDR/ICP from 2021 vs. a real rate of ~3–8 XDR/ICP today), users receive **far more cycles per ICP** than they should. Conversely, if the rate is stale and higher than market, users receive fewer cycles. In the default-rate scenario (100 XDR/ICP), a user calling `notify_top_up` with 1 ICP would receive `100 * 1e12 = 100T cycles` instead of the correct ~3–8T cycles — a 12–30x over-minting. This directly violates the ICP economic model and drains the network's cycle supply.

The impact is bounded by the `base_cycles_limit` rate limiter (150T cycles/hour for normal users), but the over-minting ratio means the limit is hit much faster than intended, and each minted cycle is backed by less ICP than the protocol requires.

### Likelihood Explanation
The scenario is realistic in two cases:
1. **Fresh CMC deployment or upgrade** where the XRC has not yet been polled: the default rate from 2021 is active until the first successful heartbeat.
2. **Sustained XRC unavailability**: if the XRC canister is unavailable for an extended period (e.g., during a subnet incident), the CMC continues minting at the last cached rate, which may be hours or days old.

Any unprivileged user can trigger this by calling `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` with a valid ICP ledger block index. No special privileges are required.

### Recommendation
Add a staleness check in `tokens_to_cycles` (or in each of the three `process_*` callers) that rejects the conversion if the cached rate's `timestamp_seconds` is older than a configurable threshold (e.g., 1 hour or 1 day):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref()
            .ok_or_else(|| NotifyError::Other { ... })?;

        // Reject if rate is stale (e.g., older than 1 day)
        let now = ic_cdk::api::time() / 1_000_000_000;
        if now.saturating_sub(rate.timestamp_seconds) > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "ICP/XDR conversion rate is stale, retry later".to_string(),
            });
        }

        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

This mirrors the analog fix in the PoolKeeper report: validate the data source before using it, and surface a clear error rather than silently operating on stale data.

### Proof of Concept

1. Deploy a fresh CMC (or simulate one with `State::default()`). The initial `icp_xdr_conversion_rate` is `{ timestamp_seconds: 1_620_633_600, xdr_permyriad_per_icp: 1_000_000 }` — 100 XDR/ICP from May 2021. [1](#0-0) 

2. Disable the XRC (set `exchange_rate_canister_id: None` or make the XRC return errors). The heartbeat will not update the rate. [2](#0-1) 

3. Send 1 ICP to the CMC's subaccount for the caller, then call `notify_top_up`. The CMC calls `tokens_to_cycles`, which reads the stale default rate of 100 XDR/ICP with no timestamp check. [3](#0-2) 

4. The caller receives `1 ICP × 100 XDR/ICP × 1T cycles/XDR = 100T cycles` instead of the correct ~3–8T cycles at current market rates. The over-minting is ~12–30x. [4](#0-3) 

The default rate constant and its 2021 timestamp confirm the magnitude of the discrepancy: [5](#0-4)
