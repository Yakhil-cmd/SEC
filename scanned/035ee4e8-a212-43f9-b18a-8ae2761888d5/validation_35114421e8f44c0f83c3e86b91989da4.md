### Title
Unprivileged Ethereum User Can Permanently Skip a Block and Freeze Depositor Funds via `SkippedBlockForContract` — (`rs/ethereum/cketh/minter/src/deposit.rs`)

### Summary

The `scrape_block_range()` function contains an intentional but exploitable fallback: when the EVM RPC response for a single-block range (`from_block == to_block`) is still too large, the minter permanently records `EventType::SkippedBlockForContract` and advances its scraping cursor past that block. An unprivileged Ethereum user can deliberately trigger this path by packing enough deposit-event-emitting calls into a single Ethereum block, causing all deposits in that block to be permanently ignored — ETH/ERC-20 is locked in the helper contract, no ckETH/ckERC20 is ever minted.

### Finding Description

In `scrape_block_range()`, the binary-bisection loop narrows the query range on `is_response_too_large` errors. When the range cannot be narrowed further (`from_block == to_block`), the minter unconditionally records a skip and advances: [1](#0-0) 

The skipped block is stored in `State::skipped_blocks` (`BTreeMap<Address, BTreeSet<BlockNumber>>`): [2](#0-1) 

There is **no retry or recovery path** for skipped blocks anywhere in the minter. The `skipped_blocks` map is written to but never read back to re-attempt scraping. The `EventType::SkippedBlockForContract` variant is defined as a terminal state: [3](#0-2) 

The existing integration test `should_skip_single_block_containing_too_many_events` explicitly confirms and validates this behavior — 3,100+ log entries in a single block exceed the 2 MB limit and trigger the skip: [4](#0-3) 

### Impact Explanation

An attacker who sends a single Ethereum transaction that emits ~3,100+ `ReceivedEth` or `ReceivedErc20` log events (e.g., via a looping contract that calls the helper contract repeatedly within one transaction, or by flooding the block with many small deposits) causes the minter to permanently skip that block. Every legitimate depositor whose transaction landed in that block loses their funds: ETH/ERC-20 is held by the helper contract forever, and no ckETH/ckERC20 is minted. The `assert!` in `record_skipped_block_for_contract` would panic on a second skip attempt for the same block, but the cursor advance at line 283 prevents re-entry: [5](#0-4) 

### Likelihood Explanation

- The attacker needs only one Ethereum transaction emitting ~3,100 events. At ~375 gas per LOG opcode plus data, this is roughly 1.2M gas — well within Ethereum's ~30M block gas limit and achievable for a moderately funded attacker.
- The attacker does not need to be a depositor themselves; they can grief other depositors by co-locating their flood transaction in the same block.
- The EVM RPC canister's 2 MB HTTP outcall response cap is a hard IC protocol constraint, so the minter cannot work around it without architectural changes.
- The behavior is confirmed by an existing test, meaning it is reproducible locally with certainty.

### Recommendation

1. **Do not permanently skip blocks.** Instead of recording `SkippedBlockForContract` and advancing the cursor, quarantine the block and alert operators, halting scraping progress until manual intervention resolves it.
2. **Alternatively**, implement per-transaction-hash log fetching as a fallback when a single block is too large, so individual deposit events can still be recovered.
3. **Rate-limit or cap** the number of log entries processed per block per contract to bound response sizes before they reach the RPC layer.
4. **Add a governance-controlled recovery endpoint** that allows re-processing a previously skipped block after the underlying data has been chunked or filtered.

### Proof of Concept

```
1. Deploy an Ethereum contract that calls the ckETH helper's deposit()
   function in a loop, emitting 3,100+ ReceivedEth events in one tx.
2. Submit that transaction to Ethereum mainnet/testnet.
3. Wait for the ckETH minter's next scrape_logs() timer tick.
4. Observe: minter bisects the range down to [N, N], still gets
   is_response_too_large, records SkippedBlockForContract{block: N}.
5. Assert: no ckETH is minted for any depositor in block N.
6. Assert: skipped_blocks[helper_contract].contains(N) == true.
7. Assert: last_scraped_block_number advances past N, permanently.
```

The test at `rs/ethereum/cketh/minter/tests/cketh.rs:1087` already proves steps 3–7 are reachable and deterministic. [6](#0-5)
