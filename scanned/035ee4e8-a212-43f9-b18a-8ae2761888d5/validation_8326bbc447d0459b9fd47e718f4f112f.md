### Title
Unbounded Reward-Round Loop in SNS Governance `distribute_rewards` Exhausts Instruction Limit, Permanently Blocking Periodic Tasks - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `distribute_rewards` function iterates over every missed reward round since the last `RewardEvent` in a single synchronous message, with no instruction-limit guard. If `round_duration_seconds` is set to a small value (as low as 1 second, since only 0 is rejected), or if the canister fails to distribute rewards for an extended period, `new_rounds_count` grows proportionally. The resulting loop exhausts the IC instruction limit, causing the message to trap. Because the `latest_reward_event` is only written at the very end of the function, a trap rolls back all state, leaving `new_rounds_count` unchanged or larger on the next invocation. This permanently prevents `distribute_rewards`—and all other periodic tasks in `run_periodic_tasks`—from executing.

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, `distribute_rewards` computes the number of elapsed reward rounds:

```rust
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [1](#0-0) 

It then iterates unconditionally over every round to accumulate the rewards purse:

```rust
for i in 1..=new_rounds_count {
    let current_reward_rate = voting_rewards_parameters.reward_rate_at(...);
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [2](#0-1) 

There is no instruction-limit check inside this loop and no cap on `new_rounds_count`. The only guard against a zero divisor is:

```rust
if round_duration_seconds == 0 { return; }
``` [3](#0-2) 

Any value ≥ 1 is accepted. With `round_duration_seconds = 1`, and `reward_start_timestamp_seconds` defaulting to `0` when `end_timestamp_seconds` is `None`:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();
``` [4](#0-3) 

`new_rounds_count` equals the current Unix timestamp (~1.7 × 10⁹). The loop body involves `Decimal` arithmetic via `reward_rate_at`, which is non-trivial. Even at a conservative 100 instructions per iteration, 1.7 billion iterations far exceeds the IC's ~5 billion instruction limit per message.

The `latest_reward_event` is only written at the very end of the function:

```rust
self.proto.latest_reward_event = Some(RewardEvent { ... })
``` [5](#0-4) 

A trap rolls back all state, so `reward_start_timestamp_seconds` remains 0 on every subsequent call, making the condition permanent.

`distribute_rewards` is called synchronously from `run_periodic_tasks`:

```rust
if should_distribute_rewards {
    match self.ledger.total_supply().await {
        Ok(supply) => { self.distribute_rewards(supply); }
        ...
    }
}
``` [6](#0-5) 

Because `distribute_rewards` traps the entire message, all other periodic tasks (upgrade checks, maturity moves, GC) also stop executing.

**Contrast with NNS governance:** The NNS governance moved reward distribution to a timer-based, batched approach with an explicit instruction-limit check (`DISTRIBUTION_MESSAGE_LIMIT = 1_000_000_000`) inside `continue_processing`:

```rust
if is_over_instructions_limit() { break; }
``` [7](#0-6) 

No equivalent guard exists in the SNS `distribute_rewards` loop.

---

### Impact Explanation

An SNS governance canister configured with a small `round_duration_seconds` (e.g., 1 second) will permanently fail to execute `run_periodic_tasks`. This means:

1. Voting rewards are never distributed to neuron holders.
2. Upgrade checks, maturity moves, and GC never run.
3. The SNS governance canister enters a permanently degraded state from which it cannot self-recover, since the only fix would require an upgrade of the canister itself.

This is a **cycles/resource accounting bug** causing a **permanent DoS of SNS governance periodic tasks**.

---

### Likelihood Explanation

The `round_duration_seconds` parameter is set during SNS initialization by the SNS deployer (a "canister developer" in HackenProof scope) or changed via a governance proposal. The minimum allowed value is 1 second (only 0 is rejected). An SNS deployer who sets `round_duration_seconds = 1` will trigger this on the very first call to `run_periodic_tasks`, since `reward_start_timestamp_seconds` defaults to 0. Even with a more reasonable value (e.g., 1 hour = 3600 seconds), after ~160,000 hours (~18 years) without a successful reward distribution, the same condition is reached. More practically, any SNS that experiences a prolonged failure of `run_periodic_tasks` (e.g., due to a canister upgrade bug) will accumulate missed rounds and eventually hit this limit.

---

### Recommendation

1. **Cap `new_rounds_count`** to a safe maximum (e.g., 1000 rounds) per invocation, rolling over the remainder to the next call, analogous to how the NNS handles missed days.
2. **Add an instruction-limit check** inside the `for i in 1..=new_rounds_count` loop, breaking early and updating `latest_reward_event` with the rounds processed so far.
3. **Enforce a minimum `round_duration_seconds`** (e.g., ≥ 1 day) in `NervousSystemParameters` validation to prevent pathologically small values.
4. **Write `latest_reward_event` incrementally** (or at least write a partial event before the loop) so that a trap does not reset progress.

---

### Proof of Concept

1. Deploy an SNS with `VotingRewardsParameters { round_duration_seconds: Some(1), ... }`.
2. Wait for the SNS governance canister to call `run_periodic_tasks` (triggered by heartbeat or timer).
3. `should_distribute_rewards()` returns `true` immediately (since `now > 0 + 1`).
4. `distribute_rewards` is called; `new_rounds_count = now / 1 ≈ 1,700,000,000`.
5. The loop `for i in 1..=1_700_000_000` exhausts the ~5 billion instruction limit.
6. The message traps; `latest_reward_event` is not updated.
7. Every subsequent call to `run_periodic_tasks` repeats steps 4–6.
8. The SNS governance canister is permanently unable to distribute rewards or run any periodic task.
