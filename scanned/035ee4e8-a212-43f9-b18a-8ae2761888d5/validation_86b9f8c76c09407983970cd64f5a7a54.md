### Title
SNS Voting Reward Rate Change Without Settling Current-Period Rewards - (File: rs/sns/governance/src/governance.rs)

### Summary
When a `ManageNervousSystemParameters` proposal changes `voting_rewards_parameters` (e.g., `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, or `reward_rate_transition_duration_seconds`), the SNS governance canister updates the parameters immediately without first settling the current in-progress reward period at the old rate. The next `distribute_rewards` invocation then retroactively applies the new rate curve to all elapsed-but-unsettled rounds, producing incorrect reward amounts for the period that occurred before the rate change.

### Finding Description
`perform_manage_nervous_system_parameters` in `rs/sns/governance/src/governance.rs` (line 2581) directly overwrites `self.proto.parameters` with the new parameters:

```rust
Ok(()) => {
    self.proto.parameters = Some(new_params);  // line 2597
    Ok(())
}
``` [1](#0-0) 

No call to `distribute_rewards` precedes this write. The `distribute_rewards` function (line 5763) subsequently reads `voting_rewards_parameters` from `self.nervous_system_parameters_or_panic()` — which now returns the **new** parameters — and uses them to compute the reward purse for **all** rounds elapsed since the last `RewardEvent`:

```rust
let current_reward_rate = voting_rewards_parameters.reward_rate_at(
    crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
);
result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
``` [2](#0-1) 

`reward_rate_at` computes the rate as a quadratic function of `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, and `reward_rate_transition_duration_seconds`:

```rust
let dr = self.initial_reward_rate() - self.final_reward_rate();
let variable_reward_rate = s2 * dr;
self.final_reward_rate() + variable_reward_rate
``` [3](#0-2) 

Because the rate function is parameterized by the **current** `VotingRewardsParameters`, changing those parameters mid-period retroactively alters the computed reward for every unsettled round. The proto comment itself acknowledges this: *"Once this is set, it probably should not be changed, because the results would probably be pretty confusing."* [4](#0-3) 

### Impact Explanation
- **Rate increase scenario**: If `initial_reward_rate_basis_points` is raised (e.g., 1 000 → 2 000 bp) before `distribute_rewards` runs, all neurons receive approximately 2× the rewards they legitimately earned during the pre-change period. This is unauthorized token inflation at the expense of the SNS treasury/supply.
- **Rate decrease scenario**: If the rate is lowered, neurons receive fewer rewards than they earned, constituting a silent confiscation of accrued maturity.
- In both cases the `RewardEvent.distributed_e8s_equivalent` and `total_available_e8s_equivalent` fields are permanently recorded with incorrect values, corrupting the on-chain audit trail.

### Likelihood Explanation
Exploiting this requires passing a `ManageNervousSystemParameters` proposal with modified `voting_rewards_parameters`. In many SNS instances — particularly early-stage ones where developer/team neurons hold a majority — a single actor or small coalition can pass such a proposal without broad community consent. The window between the last `distribute_rewards` call and the proposal execution is non-zero because `distribute_rewards` is driven by periodic tasks, not by an atomic pre-execution hook. The proto comment's own warning ("probably should not be changed") confirms the developers are aware of the risk but have not enforced it in code.

### Recommendation
Before applying new `voting_rewards_parameters` in `perform_manage_nervous_system_parameters`, call `distribute_rewards` (or an equivalent settlement function) to close the current reward period at the old rate. This mirrors the pattern recommended in the referenced report: *"Ensure the program calls `increase_share_indexes` to update the share indexes based on the current emission rate and timestamp"* before modifying the rate.

### Proof of Concept
1. Deploy an SNS with `initial_reward_rate_basis_points = 1_000` (10 %/yr), `round_duration_seconds = 86_400`.
2. Advance time by one full round (86 400 s). `distribute_rewards` has not yet been called (or the periodic task fires slightly after the round boundary).
3. Submit and pass a `ManageNervousSystemParameters` proposal setting `initial_reward_rate_basis_points = 2_000` (20 %/yr). `perform_manage_nervous_system_parameters` writes the new parameters immediately.
4. `distribute_rewards` fires. It computes `new_rounds_count = 1` and calls `reward_rate_at(seconds_since_genesis = 86_400)` using the **new** 20 % initial rate, yielding a reward purse ≈ 2× the correct amount.
5. All voting neurons receive approximately double the maturity they legitimately earned during the 10 %-rate period, with no on-chain record of the discrepancy. [5](#0-4) [6](#0-5)

    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }
