### Title
`MintSnsTokens` Proposal Has No Effective 7-Day Minting Rate Limit, Allowing Unlimited Token Inflation - (File: `rs/sns/governance/src/proposal.rs`)

### Summary

The SNS governance canister intentionally designed a 7-day rate-limiting guard for both `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals to prevent a governance majority from draining or inflating the token supply in a single window. For `TransferSnsTreasuryFunds` this guard is fully enforced at both submission time and execution time. For `MintSnsTokens`, the guard is deliberately disabled via a `TODO(NNS1-2982)` comment, causing `recent_amount_total_upper_bound_tokens` to return `Decimal::MAX` (effectively unlimited), and `perform_mint_sns_tokens` performs no execution-time rate-limit check at all. Any SNS governance majority can therefore pass a `MintSnsTokens` proposal to mint an arbitrary quantity of SNS tokens to any address in a single governance cycle, with no protocol-level cap.

### Finding Description

The `TokenProposalAction` trait defines `recent_amount_total_upper_bound_tokens` to cap the 7-day total of sensitive proposals. For `TransferSnsTreasuryFunds`, this is wired to `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens`, which enforces a tiered cap (100% for small treasuries, 25% for medium, 300,000 XDR for large).

For `MintSnsTokens`, the real implementation is commented out:

```rust

    ...
}
*/

// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    Ok(Decimal::MAX)
}

The submission-time check in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` therefore compares the proposed mint amount against `Decimal::MAX`, which always passes. [2](#0-1) 

At execution time, `perform_transfer_sns_treasury_funds` calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` as a second guard: [3](#0-2) 

But `perform_mint_sns_tokens` has no such check — it directly calls `transfer_funds` with the caller-supplied `amount_e8s`: [4](#0-3) 

The dispatch path confirms `MintSnsTokens` goes directly to `perform_mint_sns_tokens` with no interposed guard: [5](#0-4) 

The `mint_sns_tokens_7_day_total_upper_bound_tokens` function exists and is correct in the limit library, but is simply not imported or called: [6](#0-5) [7](#0-6) 

### Impact Explanation

A governance majority in any SNS can submit and pass a `MintSnsTokens` proposal specifying `amount_e8s = u64::MAX` (or any large value) directed to an attacker-controlled principal. Upon execution, the SNS ledger mints that quantity of SNS tokens with no protocol-enforced cap. This:

- Inflates the SNS token supply by an arbitrary amount in a single governance cycle.
- Allows the majority holder to dilute all other token holders to near-zero ownership.
- Effectively constitutes a rug pull of all SNS token holders, analogous to the M-12 `withdrawAll` pattern.

The `TransferSnsTreasuryFunds` path has both submission-time and execution-time guards enforcing the 7-day cap. The `MintSnsTokens` path has neither effective guard, creating an asymmetric and unintended exposure.

### Likelihood Explanation

Any SNS where a single entity or colluding group holds majority voting power (common in early-stage SNS deployments where the developer team retains large neuron stakes) can exploit this immediately. The attack requires only submitting a standard `MintSnsTokens` proposal through the normal `manage_neuron` ingress endpoint — no special access, no key compromise, no external dependency. The disabled guard is explicitly marked as a known TODO, confirming the protection was intended but is absent in the current production code.

### Recommendation

1. Uncomment the real `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (remove the `TODO(NNS1-2982): Delete` stub and uncomment the `TODO(NNS1-2982): Uncomment` block).
2. Add an execution-time check in `perform_mint_sns_tokens` analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, using the `MintSnsTokensActionAuxiliary` valuation stored at proposal submission time.
3. Import `mint_sns_tokens_7_day_total_upper_bound_tokens` from `ic_sns_governance_proposals_amount_total_limit` (currently commented out in the import list).

### Proof of Concept

1. An SNS whale neuron holder calls `manage_neuron` (ingress) on the SNS governance canister with:
   ```
   Command::MakeProposal(Proposal {
       action: Action::MintSnsTokens(MintSnsTokens {
           amount_e8s: Some(u64::MAX),
           to_principal: Some(attacker_principal),
           to_subaccount: None,
           memo: None,
       })
   })
   ```
2. Submission-time validation calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err` → `MintSnsTokens::recent_amount_total_upper_bound_tokens` → returns `Decimal::MAX` → check passes unconditionally. [8](#0-7) 
3. The proposal is adopted by governance vote.
4. `perform_action` dispatches to `perform_mint_sns_tokens`, which calls `self.ledger.transfer_funds(u64::MAX, 0, None, attacker_account, memo)` with no rate-limit check. [9](#0-8) 
5. The SNS ledger mints `u64::MAX` tokens to the attacker, collapsing the ownership share of all other holders.

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
