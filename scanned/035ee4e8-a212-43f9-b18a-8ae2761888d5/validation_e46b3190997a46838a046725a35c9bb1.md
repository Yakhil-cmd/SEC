### Title
Byzantine Block Proposer Can Stall Flexible HTTP Callback via Invalid Candid Content Bypassing Payload Validation — (`rs/https_outcalls/consensus/src/payload_builder.rs`)

---

### Summary

`validate_canister_http_payload_impl` never Candid-decodes the `Success(data)` bytes inside a `FlexibleCanisterHttpResponses` group. `flexible_ok_responses_into_consensus_response`, called only at execution time in `into_messages`, does Candid-decode those bytes and silently returns `None` on failure, causing the entire group to be dropped with no `ConsensusResponse` delivered. A Byzantine subnet node that is both a committee member and the block proposer can exploit this gap to stall a canister's flexible HTTP callback for up to `CANISTER_HTTP_TIMEOUT_INTERVAL` (60 s).

---

### Finding Description

**Validation gap — no Candid decode of `Success(data)` bytes**

`validate_canister_http_payload_impl` validates each `FlexibleCanisterHttpResponseWithProof` entry by calling `validate_flexible_response_with_proof`, which checks:
- callback-id consistency
- committee membership and registry version of the signer
- content hash and content size match
- `is_reject` flag match
- (deferred) cryptographic signature [1](#0-0) 

None of these checks Candid-decode the raw bytes inside `CanisterHttpResponseContent::Success(data)`. [2](#0-1) 

**Execution-time decode in `flexible_ok_responses_into_consensus_response`**

After the block is finalized, `into_messages` calls `flexible_ok_responses_into_consensus_response`. This function Candid-decodes every `Success(data)` entry as `CanisterHttpResponsePayload`. If any entry fails, `Decode!(...).ok()` returns `None`, `collect::<Option<_>>()` short-circuits to `None`, and the `?` propagates `None` out of the function — dropping the entire group. [3](#0-2) 

The `None` result is then silently swallowed by `.flatten()` in `into_messages`: [4](#0-3) 

The code comment explicitly acknowledges this fallback: *"Returns `None` if Candid decoding/encoding fails, which leads to skipping the delivery of this response. This should never occur, but if it does, eventually a timeout will gracefully end the outstanding callback."* [5](#0-4) 

**The attack is already unit-tested and confirmed**

The existing test `flexible_ok_responses_into_messages_decode_failure_is_skipped` directly proves the behavior: a group containing one valid and one invalid-Candid entry produces zero `ConsensusResponse` objects and increments `flexible_ok_responses_candid_failures`. [6](#0-5) 

---

### Impact Explanation

A Byzantine subnet node that is simultaneously a committee member for a specific flexible HTTP request and the block proposer for the next block can:

1. Sign a `CanisterHttpResponseContent::Success(b"invalid candid")` response (valid signature, valid committee membership, valid content hash — all checks pass).
2. Propose a block containing a `FlexibleCanisterHttpResponses` group with that entry.
3. The block passes `validate_canister_http_payload_impl` on all honest nodes and is finalized.
4. `into_messages` drops the group silently.
5. The canister's callback for that `callback_id` receives no `ConsensusResponse` until `CANISTER_HTTP_TIMEOUT_INTERVAL` (60 s) expires, at which point it receives a timeout reject instead of the actual HTTP response.

The canister cannot distinguish this from a genuine timeout and must retry the HTTP request from scratch.

---

### Likelihood Explanation

The attacker must be a single Byzantine subnet node that happens to be both a committee member for the targeted flexible HTTP request and the block proposer for the relevant round. Both roles are drawn from the same set of subnet nodes, so a single Byzantine node can satisfy both conditions simultaneously. No threshold corruption, key compromise, or external infrastructure attack is required — this is within the single-Byzantine-node fault model that the IC protocol is explicitly designed to tolerate at the consensus layer, but this particular path is not covered by that tolerance.

---

### Recommendation

Add a Candid-decode check for `Success(data)` bytes inside `validate_canister_http_payload_impl`'s flexible-response loop, mirroring exactly what `flexible_ok_responses_into_consensus_response` does at execution time:

```rust
if let CanisterHttpResponseContent::Success(data) = &response_with_proof.response.content {
    if Decode!(data, CanisterHttpResponsePayload).is_err() {
        return invalid_artifact(
            InvalidCanisterHttpPayloadReason::FlexibleResponseInvalidCandid { callback_id },
        );
    }
}
```

This closes the gap between what validation accepts and what `into_messages` can actually process. [7](#0-6)

pub(crate) fn validate_flexible_response_with_proof(
    response_with_proof: &FlexibleCanisterHttpResponseWithProof,
    callback_id: CallbackId,
    flex_committee: &BTreeSet<NodeId>,
    seen_signers: &mut HashSet<NodeId>,
    consensus_registry_version: RegistryVersion,
    per_replica_allowance: Cycles,
) -> Result<(), InvalidCanisterHttpPayloadReason> {
    if response_with_proof.response.id != callback_id {
        return Err(
            InvalidCanisterHttpPayloadReason::FlexibleCallbackIdMismatch {
                callback_id,
                mismatched_id: response_with_proof.response.id,
            },
        );
    }

    validate_response_share(
        &response_with_proof.proof,
        callback_id,
        flex_committee,
        seen_signers,
        consensus_registry_version,
        per_replica_allowance,
    )?;

    let calculated_hash = crypto_hash(&response_with_proof.response);
    if &calculated_hash != response_with_proof.proof.content.content_hash() {
        return Err(InvalidCanisterHttpPayloadReason::ContentHashMismatch {
            metadata_hash: response_with_proof.proof.content.content_hash().clone(),
            calculated_hash,
        });
    }

    let calculated_size = response_with_proof.response.content.count_bytes() as u32;
    if calculated_size != response_with_proof.proof.content.content_size() {
        return Err(InvalidCanisterHttpPayloadReason::ContentSizeMismatch {
            metadata_size: response_with_proof.proof.content.content_size(),
            calculated_size,
        });
    }

    let calculated_is_reject = response_with_proof.response.content.is_reject();
    if calculated_is_reject != response_with_proof.proof.content.is_reject() {
        return Err(InvalidCanisterHttpPayloadReason::IsRejectMismatch {
            metadata_is_reject: response_with_proof.proof.content.is_reject(),
            calculated_is_reject,
        });
    }

    Ok(())
}
