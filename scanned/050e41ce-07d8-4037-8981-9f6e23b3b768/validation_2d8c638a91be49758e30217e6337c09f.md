### Title
SNS Swap `Swap::scale` Floor-Division Rounding Permanently Strands SNS Tokens in Swap Canister - (File: rs/sns/swap/src/swap.rs)

### Summary
The `Swap::scale` function in the SNS Swap canister converts each participant's ICP contribution to an SNS token allocation using integer floor division. Because each participant's allocation is independently rounded down, the sum of all allocations is strictly less than the total SNS tokens offered. The residual "dust" tokens remain permanently locked in the swap canister with no recovery path, constituting a ledger conservation bug analogous to the external report's rounding flaw.

### Finding Description
`Swap::scale` computes each participant's SNS token allocation as:

```
(amount_icp_e8s * total_sns_e8s) / total_icp_e8s
```

using 128-bit integer floor division. [1](#0-0) 

This is called independently for every direct participant and every Neurons' Fund neuron during `create_sns_neuron_recipes`: [2](#0-1) [3](#0-2) 

Because floor division is applied per-participant, the mathematical identity

```
sum_i floor(a_i * T / S) ≤ T
```

holds with strict inequality whenever any `a_i * T` is not divisible by `S`. The difference — up to `(num_participants − 1)` e8s — is never distributed. The running total is tracked with `saturating_add`: [4](#0-3) 

but there is no subsequent sweep, refund, or governance proposal mechanism to recover the stranded remainder. The integration test itself acknowledges the approximation is not e8-precise: [5](#0-4) 

### Impact Explanation
After every committed SNS swap, a small but non-zero quantity of SNS tokens (up to `num_participants − 1` e8s) is permanently locked in the swap canister. These tokens are:

- Not distributed to any participant.
- Not returned to the SNS treasury or governance canister.
- Not recoverable via any existing canister method.

This is a **ledger conservation bug**: the total SNS token supply is unchanged, but a portion of the tokens offered for sale is permanently inaccessible. For swaps with thousands of participants, the stranded amount can reach thousands of e8s. The `apportion_approximately_equally` helper used for neuron basket splitting does preserve conservation within a single participant's basket: [6](#0-5) 

but the cross-participant rounding loss from `Swap::scale` is not compensated anywhere.

### Likelihood Explanation
This affects **every committed SNS swap** with more than one participant. The entry path is fully unprivileged: any user who participates in an SNS decentralization swap triggers `create_sns_neuron_recipes` at finalization. No special role or key is required. The condition is deterministic and reproducible.

### Recommendation
1. After `create_sns_neuron_recipes` completes, compute `leftover = sns_being_offered_e8s − total_sns_tokens_sold_e8s` and transfer the remainder to the SNS treasury or governance canister rather than leaving it stranded.
2. Alternatively, assign the rounding remainder to the last participant (or the largest participant) to ensure full distribution, analogous to how `apportion_approximately_equally` distributes the remainder to the last elements.
3. Add an invariant assertion or post-condition check that `total_sns_tokens_sold_e8s == sns_being_offered_e8s` after all recipes are created, or explicitly document and handle the delta.

### Proof of Concept
Consider a swap with `sns_token_e8s = 10` and two participants each contributing `5` ICP out of `10` total ICP:

```
Participant A: scale(5, 10, 10) = (5 * 10) / 10 = 5
Participant B: scale(5, 10, 10) = (5 * 10) / 10 = 5
total_sns_tokens_sold = 10  ← exact, no dust here
```

Now consider `sns_token_e8s = 10` and three participants each contributing `3`, `3`, `4` ICP out of `10` total:

```
Participant A: scale(3, 10, 10) = 3
Participant B: scale(3, 10, 10) = 3
Participant C: scale(4, 10, 10) = 4
total_sns_tokens_sold = 10  ← exact again
```

Now consider `sns_token_e8s = 7` and two participants each contributing `5` ICP out of `10` total:

```
Participant A: scale(5, 7, 10) = floor(35/10) = 3
Participant B: scale(5, 7, 10) = floor(35/10) = 3
total_sns_tokens_sold = 6
stranded = 7 − 6 = 1 e8 permanently locked in swap canister
```

With `N` participants and a non-divisible `sns_token_e8s`, up to `N − 1` e8s are stranded per swap. For a swap with 10,000 participants, up to 9,999 e8s (≈ 0.0001 SNS tokens) are permanently lost per swap finalization. [7](#0-6) [8](#0-7)

pub fn apportion_approximately_equally(total: u64, len: u64) -> Result<Vec<u64>, String> {
    let quotient = total
        .checked_div(len)
        .ok_or_else(|| format!("Unable to divide total={total} by len={len}"))?;
    let remainder = total % len; // For unsigned integers, % cannot overflow.

    // So far, we have only apportioned quotient * len. To reach the desired
    // total, we must still somehow add remainder (per Euclid's Division
    // Theorem). That is accomplished right after this.
    let mut result = vec![quotient; len as usize];

    // Divvy out the remainder: Starting from the last element, increment
    // elements by 1. The number of such increments performed here is remainder,
    // bringing our total back to the desired amount.
    if remainder >= result.len() as u64 {
        return Err(format!("Could not apportion {total} into {len} pieces"));
    }
    let mut iter_mut = result.iter_mut();
    for _ in 0..remainder {
        let element: &mut u64 = iter_mut
            .next_back()
            // We can prove that this will not panic:
            // The number of iterations of this loop is total % len.
            // This must be < len (by Euclid's Division Theorem).
            // Thus, the number of iterations that this loop goes through is < len.
            // Thus, the number of times next_back is called is < len.
            // next_back only returns None after len calls.
            // Therefore, next_back does not return None here.
            // Therefore, this expect will never panic.
            .ok_or_else(
                || format!("Ran out of elements to increment. total={total}, len={len}",),
            )?;

        // This cannot overflow because the result must be <= total. Thus, this
        // will not panic.
        *element = element.checked_add(1).ok_or_else(|| {
            format!("Incrementing element by 1 resulted in overflow. total={total}, len={len}",)
        })?;
    }

    Ok(result)

    /// Computes `amount_icp_e8s` scaled by (`total_sns_e8s` divided by
    /// `total_icp_e8s`), but perform the computation in integer space
    /// by computing `(amount_icp_e8s * total_sns_e8s) /
    /// total_icp_e8s` in 128 bit space.
    fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
        assert!(amount_icp_e8s <= u64::from(total_icp_e8s));
        // Note that the multiplication cannot overflow as both factors fit in 64 bits.
        let r = (amount_icp_e8s as u128)
            .saturating_mul(total_sns_e8s as u128)
            .div(NonZeroU128::from(total_icp_e8s));
        // This follows logically from the initial assert `amount_icp_e8s <= total_icp_e8s`.
        assert!(r <= u64::MAX as u128);
        r as u64
    }
