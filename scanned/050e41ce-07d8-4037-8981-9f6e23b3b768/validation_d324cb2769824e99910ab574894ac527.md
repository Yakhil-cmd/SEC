### Title
Byzantine Dealer Can Inject Arbitrary Shares via All-Identity Randomizers Bypassing All ZK Proof Checks — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/groth20_bls12_381/encryption.rs`)

### Summary
A Byzantine dealer can submit a NiDKG dealing where all `rr[j]` (randomizer chunks) are the G1 identity point. This causes `combined_r = identity` in the sharing proof instance. All three verification layers — `verify_ciphertext_integrity`, `verify_chunking`, and `verify_sharing` — can be satisfied with witness `r = 0`, allowing the dealer to inject arbitrary, attacker-chosen shares that pass every check in `verify_dealing`.

### Finding Description
In `verify_zk_proofs`:

1. `FsEncryptionCiphertext::deserialize` accepts the G1 identity point in `rand_r` (it is a valid compressed BLS12-381 point).
2. `g1_from_big_endian_chunks(ciphertext.randomizers_r())` returns `G1Affine::identity()` when all chunks are identity, because doubling and adding identity always yields identity.
3. `verify_ciphertext_integrity` checks `e(r_i, id) * e(s_i, h) * e(-g1, z_i) == 1`. With `r_i = identity`, the `e(r_i, id)` term vanishes, and the attacker satisfies the remaining equation by setting `s_i = g1^{k_i}`, `z_i = h^{k_i}` for any known scalar `k_i` (since `h` is a public system parameter).
4. `verify_chunking` Thread 1 reduces to `dd_i == g1^{z_r_i}` when all `R_j = identity`. The attacker knows witness `r_j = 0` and constructs a valid proof.
5. `SharingInstance::check_instance` does not check whether `combined_randomizer` is the identity.
6. `verify_sharing` Eq 1 reduces to `F == g1^{z_r}` (trivially satisfiable). Eq 3 is satisfied by choosing `s_i` consistent with attacker-chosen public coefficients `A_k = g2^{a_k}` and using witness `(r=0, s=[s_1..s_n])`.

The attacker controls the entire dealing: arbitrary shares `s_i`, arbitrary public coefficients, and all proofs verify. [9](#0-8) 

### Impact Explanation
When receivers call `dec_chunks`, they decrypt the attacker's chosen shares `s_i` (since `r_j = 0` means `C_{i,j} = g1^{m_{i,j}}` and decryption recovers `m_{i,j}` directly). The resulting threshold key is a function of honest dealers' contributions plus the attacker's injected shares. If the attacker knows their injected shares (they do, by construction), they can bias or fully control the threshold key material, breaking the security guarantee of the NiDKG protocol. [10](#0-9) 

### Likelihood Explanation
The attacker must be a Byzantine dealer — a single compromised subnet node. This is "protocol peer behavior below the consensus fault threshold." The NiDKG ZK proofs exist precisely to prevent Byzantine dealers from injecting arbitrary shares; this attack bypasses all of them. The construction is fully deterministic and requires no brute force.

### Recommendation
- In `verify_zk_proofs` (or `SharingInstance::check_instance`), reject `combined_randomizer` if it is the identity point.
- In `verify_ciphertext_integrity`, explicitly check that each `rr[i]` is not the identity point before the pairing check.
- In `ChunkingInstance::check_instance`, reject any `randomizers_r[j]` that is the identity point.
- Add a test that submits a dealing with all-identity `rand_r` and asserts `verify_zk_proofs` returns `Err`. [6](#0-5) [11](#0-10)
