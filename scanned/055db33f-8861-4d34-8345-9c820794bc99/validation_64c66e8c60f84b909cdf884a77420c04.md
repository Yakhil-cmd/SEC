### Title
Pending Voting Rewards Not Settled Before `VotingRewardsParameters` Update in SNS Governance - (File: rs/sns/governance/src/governance.rs)

### Summary

When a `ManageNervousSystemParameters` proposal is executed in SNS Governance to update `VotingRewardsParameters` (e.g., `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, or `round_duration_seconds`), the function `perform_manage_nervous_system_parameters` applies the new parameters immediately without first distributing pending rewards accrued under the old parameters. The next time `distribute_rewards` runs, it applies the new reward rate to the entire unsettled period retroactively, rather than only from the point of the parameter change.

### Finding Description

`perform_manage_nervous_system_parameters` in `rs/sns/governance/src/governance.rs` directly overwrites `self.proto.parameters` with the new parameters:

```rust
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            self.proto.parameters = Some(new_params);  // ← no distribute_rewards() call first
            Ok(())
        }
        ...
    }
}
``` [1](#0-0) 

The `distribute_rewards` function, called periodically from `run_periodic_tasks`, reads the **current** `voting_rewards_parameters` and applies them to all rounds since the last reward event:

```rust
let voting_rewards_parameters = match &self
    .nervous_system_parameters_or_panic()
    .voting_rewards_parameters { ... };

for i in 1..=new_rounds_count {
    let seconds_since_genesis = round_duration_seconds
        .saturating_mul(i)
        .saturating_add(reward_start_timestamp_seconds)
        .saturating_sub(self.proto.genesis_timestamp_seconds);

    let current_reward_rate = voting_rewards_parameters.reward_rate_at(
        crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
    );
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [2](#0-1) 

The `reward_rate_at` function computes the rate based on `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, and `reward_rate_transition_duration_seconds` — all of which can be changed by a `ManageNervousSystemParameters` proposal: [3](#0-2) 

`VotingRewardsParameters` is a mutable field of `NervousSystemParameters` and is explicitly changeable via governance proposal: [4](#0-3) 

The proto comment even warns: *"Once this is set, it probably should not be changed, because the results would probably be pretty confusing."* — confirming the developers are aware of the sensitivity but the code does not enforce a settlement checkpoint. [5](#0-4) 

### Impact Explanation

**Impact: Medium.** When `initial_reward_rate_basis_points` or `final_reward_rate_basis_points` is changed via a `ManageNervousSystemParameters` proposal, the new rate is applied retroactively to the entire current unsettled reward period (from `latest_reward_event.end_timestamp_seconds` to the next `distribute_rewards` call). Neurons receive more or fewer voting rewards than they actually earned under the old rate for the pre-change portion of the period. This is a direct loss or gain of yield for SNS neuron holders, depending on the direction of the rate change.

### Likelihood Explanation

**Likelihood: Medium.** Any SNS neuron holder with sufficient voting power (or followees) can submit a `ManageNervousSystemParameters` proposal to change `voting_rewards_parameters`. This is a standard governance action. The impact window is bounded to one reward period (typically 1–7 days), but the miscalculation occurs automatically on every such parameter change without any special attacker action beyond passing the proposal.

### Recommendation

Call `distribute_rewards` (or an equivalent reward settlement function) before applying the new `NervousSystemParameters` in `perform_manage_nervous_system_parameters`, specifically when `voting_rewards_parameters` is being changed. This ensures that rewards accrued under the old parameters are settled at the old rate before the new parameters take effect, analogous to adding a `processInterests` modifier in the original report.

### Proof of Concept

1. SNS is deployed with `initial_reward_rate_basis_points = 200` (2%), `final_reward_rate_basis_points = 200`, `round_duration_seconds = 86400` (1 day).
2. At time T (last reward event end), `latest_reward_event.end_timestamp_seconds = T`.
3. At time T + 12 hours (halfway through the current reward period), a `ManageNervousSystemParameters` proposal passes, changing `initial_reward_rate_basis_points` to `0` (disabling rewards).
4. `perform_manage_nervous_system_parameters` immediately sets `self.proto.parameters` with the new rate — no `distribute_rewards` call occurs.
5. At time T + 24 hours, `run_periodic_tasks` triggers `distribute_rewards`.
6. `distribute_rewards` reads the **new** `voting_rewards_parameters` (rate = 0) and computes `rewards_purse_e8s = 0` for the entire 24-hour period.
7. Neurons receive **zero** rewards for the first 12 hours (T to T+12h) during which the old 2% rate was in effect — a direct loss of yield.

The reverse scenario (rate increase) causes neurons to receive excess rewards for the pre-change period, inflating token supply beyond the intended schedule. [6](#0-5) [7](#0-6)

    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }

    pub fn reward_rate_at(&self, now: Instant) -> RewardRate {
        let reward_rate_transition_duration_seconds = self
            .reward_rate_transition_duration_seconds
            .expect("reward_rate_transition_duration_seconds unset");

        let time_since_genesis = {
            let result = now - *GENESIS;
            // For the purposes of determining reward rate, treat times before
            // genesis the same as at genesis. This is not expected to occur in
            // practice. This code is just being extra defensive.
            if result.as_secs() < i2d(0) {
                Duration { days: i2d(0) }
            } else {
                result
            }
        };
        if reward_rate_transition_duration_seconds == 0
            || time_since_genesis.as_secs() >= i2d(reward_rate_transition_duration_seconds)
        {
            return self.final_reward_rate();
        }

        // s linearly varies from 1 -> 0 as seconds_since_genesis varies from 0
        // to reward_rate_transition_duration_seconds.
        let transition = LinearMap::new(
            dec!(0)..i2d(reward_rate_transition_duration_seconds),
            dec!(1)..dec!(0),
        );
        let s = transition.apply(time_since_genesis.as_secs());
        // s2 varies quadratically from 1 -> 0 (again, as seconds_since_genesis
        // varies from 0 to reward_rate_transition_duration_seconds), and
        // flattens out as seconds_since_genesis approaches
        // reward_rate_transition_duration_seconds.
        let s2 = s * s;

        // This looks backwards, but we think of variable rate as being added to
        // final growth rate, not initial, and the amount to add is up to
        // initial - final (where initial is thought of as being greater than
        // final).
        let dr = self.initial_reward_rate() - self.final_reward_rate();
        // variable_reward_rate varies from dr to 0 as round varies from
        // 1 to transition_round_count.
        let variable_reward_rate = s2 * dr;

        self.final_reward_rate() + variable_reward_rate
    }
