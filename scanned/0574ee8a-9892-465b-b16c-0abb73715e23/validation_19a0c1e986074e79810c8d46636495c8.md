### Title
Unbounded `suspended_utxos` Heap Growth via Dust UTXOs Across Distinct Subaccounts — (`rs/bitcoin/ckbtc/minter/src/state.rs`, `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

### Summary

An unprivileged depositor who controls N Bitcoin subaccounts can send a dust UTXO (value below `deposit_btc_min_amount`) to each subaccount's derived Bitcoin address, then call `update_balance(subaccount=i)` once per subaccount. Each call unconditionally inserts a new `Account`-keyed entry into `SuspendedUtxos::utxos`, which has no size bound. The minter's heap grows proportionally to the number of distinct subaccounts ever used with dust UTXOs, with no eviction mechanism.

### Finding Description

**Data structure:**

`SuspendedUtxos` holds three unbounded maps:

```rust
pub struct SuspendedUtxos {
    utxos_without_account: BTreeMap<Utxo, SuspendedReason>,
    utxos: BTreeMap<Account, BTreeMap<Utxo, SuspendedReason>>,
    last_time_checked_cache: BTreeMap<Utxo, Timestamp>,
}
``` [1](#0-0) 

**Attack call chain:**

1. `update_balance` is called with `subaccount=i`. The guard only limits *concurrent* requests (100 max); sequential calls across different subaccounts are unlimited. [2](#0-1) [3](#0-2) 

2. `processable_utxos_for_account` classifies the dust UTXO as a **new** UTXO (not yet in `suspended_utxos`), placing it in `processable_utxos.new_utxos`. [4](#0-3) 

3. The `satoshis_to_mint == 0` early-exit guard — which exists specifically to prevent memory exhaustion — does **not** fire, because the dust UTXO has a non-zero value (e.g., 546 satoshis). The comment at line 218 acknowledges the memory concern but only protects `utxos_state_addresses`, not `suspended_utxos`. [5](#0-4) 

4. The loop detects `utxo.value < deposit_btc_min_amount` and calls `audit::ignore_utxo`. [6](#0-5) 

5. `audit::ignore_utxo` → `discard_utxo` → `state.suspend_utxo` → `suspended_utxos.insert(account, utxo, ValueTooSmall, now)`. This inserts a new `BTreeMap` node keyed by `Account{owner, subaccount=i}` with no capacity check. [7](#0-6) [8](#0-7) 

6. `last_time_checked_cache` also receives a new entry per UTXO. [9](#0-8) 

**No removal path exists** for these entries unless the UTXO is spent on-chain or the deposit minimum is lowered so the UTXO becomes processable. The entries persist indefinitely.

### Impact Explanation

Each entry consumes approximately:
- `utxos` BTreeMap node: `Account` (~61 B) + `Utxo` (~48 B) + `SuspendedReason` (1 B) + BTreeMap overhead (~128 B) ≈ 238 B
- `last_time_checked_cache` node: `Utxo` (~48 B) + `Timestamp` (8 B) + overhead (~64 B) ≈ 120 B
- **Total: ~358 B per subaccount**

At 1,000,000 entries: ~358 MB of heap. At 10,000,000 entries: ~3.58 GB, approaching the Wasm heap limit and causing the minter canister to trap on allocation, halting all ckBTC minting and withdrawal operations.

### Likelihood Explanation

The economic barrier is significant: each UTXO requires a real Bitcoin transaction. At ~750 satoshis per transaction fee and a dust UTXO value of ~500 satoshis, the attacker spends ~$0.075 per entry at $100k/BTC. Causing OOM (~10M entries) would cost ~$750,000 in fees. Causing noticeable memory pressure (100,000 entries, ~35 MB) costs ~$7,500. This is a high but not prohibitive barrier for a motivated attacker targeting a protocol managing hundreds of millions of dollars in BTC. The attack is also slow (requires 6 Bitcoin block confirmations per batch) but can be parallelized across many UTXOs in a single Bitcoin transaction with multiple outputs.

### Recommendation

1. **Enforce a per-account cap** on the number of suspended UTXOs (e.g., 10 per account). Reject or silently drop new entries beyond the cap in `SuspendedUtxos::insert`.
2. **Enforce a global cap** on `suspended_utxos.num_utxos()` and return an error from `update_balance` if the cap is exceeded.
3. **Extend the `satoshis_to_mint == 0` guard** to also cover the case where all processable UTXOs are dust — bail out before calling `audit::ignore_utxo` if the UTXO is already known to be below the minimum and is being seen for the first time (i.e., skip the suspend step for first-time dust UTXOs that have never been checked).
4. Consider a **Bitcoin address reuse fee** or rate-limiting on `update_balance` calls per principal per time window.

impl SuspendedUtxos {
    pub fn insert(
        &mut self,
        account: Account,
        utxo: Utxo,
        reason: SuspendedReason,
        now: Option<Timestamp>,
    ) -> bool {
        if let Some(timestamp) = now {
            self.last_time_checked_cache.insert(utxo.clone(), timestamp);
        }
        if self.utxos.get(&account).and_then(|u| u.get(&utxo)) == Some(&reason) {
            return false;
        }
        self.utxos_without_account.remove(&utxo);
        let utxos = self.utxos.entry(account).or_default();
        utxos.insert(utxo, reason);
        true
    }

pub fn ignore_utxo<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    utxo: Utxo,
    account: Account,
    now: Timestamp,
    runtime: &R,
) {
    discard_utxo(
        state,
        utxo,
        account,
        SuspendedReason::ValueTooSmall,
        now,
        runtime,
    );
}

fn discard_utxo<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    utxo: Utxo,
    account: Account,
    reason: SuspendedReason,
    now: Timestamp,
    runtime: &R,
) {
    // ignored UTXOs are periodically re-evaluated and should not trigger
    // an event if they are still ignored.
    if state.suspend_utxo(utxo.clone(), account, reason, now) {
        record_event(
            EventType::SuspendedUtxo {
                utxo,
                account,
                reason,
            },
            runtime,
        )
    }
}
