### Title
Unbounded `node_providers` Vec with O(N) Linear Search Causes Instruction-Limit Exhaustion in NNS Governance Canister - (File: rs/nns/governance/src/proposals/add_or_remove_node_provider.rs, rs/nns/governance/src/governance.rs)

---

### Summary

The NNS Governance canister stores all registered node providers in an unbounded `Vec<NodeProvider>` (`heap_data.node_providers`). Every critical operation on this list — adding, removing, rewarding, updating, and querying node providers — performs a full O(N) linear scan. There is no enforced upper bound on the list's size. As the IC network grows and more node providers are legitimately added through governance proposals, these operations will eventually exhaust the per-message instruction limit, causing governance proposal execution to fail and permanently blocking node provider removal or reward distribution.

---

### Finding Description

`HeapGovernanceData` stores node providers as a plain `Vec<NodeProvider>` with no size cap: [1](#0-0) 

Node providers are appended unconditionally via `node_providers.push(...)`: [2](#0-1) 

Every subsequent operation performs a full linear scan over this unbounded list:

**1. Remove (governance proposal execution):** `find_existing_node_provider_position` iterates the entire list with `.iter().position(...)`: [3](#0-2) 

**2. Reward validation:** `reward_node_provider_helper` scans with `.iter().any(...)`: [4](#0-3) 

**3. Lookup (query endpoint):** `get_node_provider` scans with `.iter().find(...)`. The code itself contains a `TODO` acknowledging the inefficiency: [5](#0-4) 

**4. Update (direct ingress call):** `update_node_provider` scans with `.iter_mut().find(...)`: [6](#0-5) 

**5. Monthly reward calculation:** `get_node_providers_rewards` iterates the entire list to compute rewards: [7](#0-6) 

---

### Impact Explanation

The Internet Computer enforces a hard per-message instruction limit (5 billion instructions for query calls, 40 billion for update calls). When `node_providers` grows large enough:

- **Governance proposal execution for `RemoveNodeProvider` will trap**, permanently preventing removal of any node provider whose position in the list requires scanning past the instruction limit. The proposal will be marked as failed, and the node provider cannot be removed.
- **`RewardNodeProvider` and `RewardNodeProviders` proposal execution will trap**, blocking monthly reward distribution.
- **`update_node_provider` ingress calls will trap**, preventing node providers from updating their reward accounts.
- **`get_node_provider_by_caller` query calls will trap**, breaking the query interface for all callers.

The `mint_monthly_node_provider_rewards` path iterates the full list and performs async ledger calls per provider, compounding the instruction cost: [7](#0-6) 

---

### Likelihood Explanation

The IC mainnet currently has on the order of hundreds of node providers. The list grows monotonically — node providers are only removed via governance proposals, which themselves require a linear scan to execute. As the network scales toward thousands of node providers (a stated goal), the instruction cost of these operations grows proportionally. The `TODO(NNS1-1168)` comment in the production code confirms the DFINITY team is aware of the inefficiency but has not yet fixed it. No adversarial action is required; organic network growth is sufficient to trigger the condition.

---

### Recommendation

Replace `Vec<NodeProvider>` with a `BTreeMap<PrincipalId, NodeProvider>` or `HashMap<PrincipalId, NodeProvider>` keyed by the node provider's principal ID. This reduces all lookup, insert, and remove operations from O(N) to O(log N) or O(1), eliminating the instruction-limit risk regardless of list size. Alternatively, enforce a hard cap on the number of node providers and reject `AddNodeProvider` proposals that would exceed it.

---

### Proof of Concept

1. Governance accumulates N node providers via `AddOrRemoveNodeProvider` proposals over time (normal network growth).
2. A `RemoveNodeProvider` proposal is submitted for a node provider near the end of the list.
3. `ValidRemoveNodeProvider::execute` calls `find_existing_node_provider_position`, which calls `node_providers.iter().position(...)` — scanning up to N entries.
4. When N is large enough, the instruction counter exceeds the 40-billion-instruction update limit mid-scan.
5. The proposal execution traps; `set_proposal_execution_status` records a failure.
6. The node provider cannot be removed. Any subsequent `RemoveNodeProvider` proposal for the same or later-indexed provider will also fail.
7. Similarly, `reward_node_provider_helper` traps for the same reason, blocking `RewardNodeProvider` proposal execution. [8](#0-7) [3](#0-2)

    fn find_existing_node_provider_position(
        &self,
        node_providers: &[NodeProvider],
    ) -> Result<usize, GovernanceError> {
        node_providers
            .iter()
            .position(|node_provider| node_provider.id == Some(self.id))
            .ok_or_else(|| {
                GovernanceError::new_with_message(
                    ErrorType::PreconditionFailed,
                    format!(
                        "AddOrRemoveNodeProvider ToRemove must target an existing Node Provider but targeted {}",
                        self.id
                    ),
                )
            })
    }
