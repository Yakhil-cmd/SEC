### Title
ckERC20 Minter Mints Based on Event-Logged Amount, Not Actual Received Amount, for Fee-on-Transfer ERC-20 Tokens — (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

Both ckERC20 deposit helper contracts emit the caller-supplied `amount` parameter in the `ReceivedEthOrErc20` / `ReceivedErc20` event, not the actual amount of ERC-20 tokens received by the minter's Ethereum address. For fee-on-transfer (deflationary) ERC-20 tokens, these two values differ. The IC ckETH minter reads the event-logged value and mints that many ckERC20 tokens to the user, creating more ckERC20 than the minter actually holds in ERC-20 collateral. This breaks the 1:1 peg and can drain the minter's ERC-20 reserves.

---

### Finding Description

**Step 1 — Helper contract emits the requested amount, not the received amount.**

In `CkErc20Deposit.deposit()` (`ERC20DepositHelper.sol`):

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal);  // ← emits `amount`, not actual received
}
```

And identically in `CkDeposit.depositErc20()` (`DepositHelperWithSubaccount.sol`):

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);  // ← emits `amount`
```

For a fee-on-transfer ERC-20 token, `safeTransferFrom(from, minterAddress, amount)` causes `minterAddress` to receive `amount - fee`, but the event records `amount`. [1](#0-0) [2](#0-1) 

**Step 2 — The IC minter parses the event-logged value.**

`ReceivedErc20LogParser::parse_log()` in `rs/ethereum/cketh/minter/src/eth_logs/parser.rs` reads the `value` field directly from the Ethereum log data:

```rust
let [value_bytes] = parse_hex_into_32_byte_words(entry.data, event_source)?;
// ...
value: Erc20Value::from_be_bytes(value_bytes),  // ← this is the `amount` from the event, not actual received
``` [3](#0-2) 

**Step 3 — The minter mints based on the event value.**

In `rs/ethereum/cketh/minter/src/deposit.rs`, `mint()` calls the ledger with `amount: event.value()`:

```rust
let block_index = match client
    .transfer(TransferArg {
        // ...
        amount: event.value(),  // ← mints the event-logged amount, not actual received
    })
    .await
``` [4](#0-3) 

The `value()` method on `ReceivedEvent` returns the `value` field parsed from the log, which is the pre-fee `amount` parameter. [5](#0-4) 

---

### Impact Explanation

For any supported ERC-20 token that charges a transfer fee (e.g., USDT with its fee mechanism enabled, or any future fee-on-transfer token added via NNS governance):

- User calls `depositErc20(token, 1000, ...)` where token has a 1% transfer fee.
- Minter's Ethereum address receives **990** tokens.
- Event emits `amount = 1000`.
- IC minter mints **1000 ckERC20** to the user.
- Minter is now undercollateralized by 10 tokens per deposit.
- Repeated deposits drain the minter's ERC-20 reserve.
- When legitimate users attempt to withdraw ckERC20, the minter cannot fulfill all withdrawals because it holds fewer ERC-20 tokens than the total ckERC20 supply.

This breaks the fundamental conservation invariant: `total_ckERC20_supply == minter_ERC20_balance`. The peg is broken and the protocol becomes insolvent for that token.

---

### Likelihood Explanation

The minter only processes events for tokens explicitly whitelisted via NNS governance proposals. Currently supported tokens (USDC, USDT, LINK, etc.) do not have active transfer fees. However:

1. **USDT** has a built-in fee mechanism (`basisPointsRate`, `maximumFee`) that is currently set to zero but can be activated by the USDT contract owner at any time without any IC governance action.
2. Any future NNS proposal to add a new ERC-20 token that happens to be fee-on-transfer would silently trigger this bug.
3. The code contains no guard or check to reject fee-on-transfer tokens.

The attacker-controlled entry path is: call `depositErc20` on the helper contract with a fee-on-transfer token that is supported by the minter. No privileged access is required — any Ethereum address can call the helper contract.

---

### Recommendation

After calling `safeTransferFrom`, measure the actual balance change at `minterAddress` and emit that as the event value, rather than the caller-supplied `amount`:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

Alternatively, the minter canister could verify the actual ERC-20 balance of its Ethereum address before and after processing a deposit event, and reject or quarantine deposits where the balance increase does not match the event-logged value.

### Proof of Concept

1. USDT contract owner calls `setParams(10, 50)` on the USDT contract, enabling a 0.1% fee (10 basis points, max 50 tokens).
2. Attacker calls `depositErc20(USDT_ADDRESS, 1_000_000, principal, subaccount)` on the helper contract.
3. `safeTransferFrom` transfers 1,000,000 USDT from attacker; minter receives 999,950 USDT (fee = 50).
4. Event emits `ReceivedEthOrErc20(..., 1_000_000, ...)`.
5. IC minter scrapes the log, parses `value = 1_000_000`, mints 1,000,000 ckUSDT to attacker.
6. Attacker now holds 1,000,000 ckUSDT but minter only holds 999,950 USDT.
7. Attacker repeats: each deposit of 1,000,000 USDT nets 50 extra ckUSDT.
8. Eventually the minter's USDT reserve is exhausted relative to outstanding ckUSDT, and legitimate withdrawals fail. [6](#0-5) [2](#0-1) [3](#0-2) [4](#0-3)
