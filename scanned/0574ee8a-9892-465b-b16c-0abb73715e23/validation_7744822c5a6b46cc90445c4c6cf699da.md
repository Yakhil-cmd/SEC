### Title
SNS Governance `disburse_neuron` Silently Discards In-Flight `disburse_maturity_in_progress` Entries, Causing Permanent Maturity Loss - (File: rs/sns/governance/src/governance.rs)

### Summary

In the SNS governance canister, a user who has initiated a `DisburseMaturity` operation (which places an entry in `disburse_maturity_in_progress` and deducts the amount from `maturity_e8s_equivalent`) can subsequently call `Disburse` (stake disbursal) on the same dissolved neuron. The `disburse_neuron` function does not check for pending `disburse_maturity_in_progress` entries and does not revert. The neuron's stake is transferred out and `cached_neuron_stake_e8s` drops to zero, but the neuron record persists. However, the periodic task `maybe_finalize_disburse_maturity` iterates over `self.proto.neurons` to find pending disbursements — if the neuron is later removed (e.g., via a future cleanup or re-stake/re-disburse cycle), those in-flight maturity entries are permanently lost. More critically, the maturity was already deducted from `maturity_e8s_equivalent` at `DisburseMaturity` call time, so the funds are neither in the neuron's maturity field nor yet minted to the user's account, and the 7-day finalization window may never complete if the neuron is in an inconsistent state.

### Finding Description

The SNS `disburse_maturity` function deducts `maturity_to_deduct` from `neuron.maturity_e8s_equivalent` immediately and pushes a `DisburseMaturityInProgress` entry into `neuron.disburse_maturity_in_progress`:

```rust
neuron.maturity_e8s_equivalent = neuron
    .maturity_e8s_equivalent
    .saturating_sub(maturity_to_deduct);
neuron
    .disburse_maturity_in_progress
    .push(disbursement_in_progress);
```

The actual minting transfer only happens 7 days later via the periodic `maybe_finalize_disburse_maturity` task. During this 7-day window, the user can call `Disburse` (stake disbursal) on the same neuron. The `disburse_neuron` function performs no check for pending `disburse_maturity_in_progress` entries:

```rust
pub async fn disburse_neuron(
    &mut self,
    id: &NeuronId,
    caller: &PrincipalId,
    disburse: &manage_neuron::Disburse,
) -> Result<u64, GovernanceError> {
    // First check authorized
    let neuron = self.get_neuron_result(id)?;
    neuron.check_authorized(caller, NeuronPermissionType::Disburse)?;

    // Check that the neuron is dissolved.
    let state = neuron.state(self.env.now());
    if state != NeuronState::Dissolved {
        return Err(...);
    }
    // ... no check for disburse_maturity_in_progress ...
```

After `disburse_neuron` succeeds, `cached_neuron_stake_e8s` is reduced to zero. The neuron record still exists in `self.proto.neurons`, so `maybe_finalize_disburse_maturity` can still find it and attempt to finalize the maturity disbursement. However, the `remove_neuron` function, which is called in other flows (e.g., if the neuron is cleaned up or re-staked/re-dissolved), permanently removes the neuron from `self.proto.neurons`, causing the `disburse_maturity_in_progress` entries to be silently dropped with no minting transfer ever occurring.

The maturity was already subtracted from `maturity_e8s_equivalent` at `DisburseMaturity` call time, so the funds are in limbo: not in the neuron's maturity field, not yet minted to the user's account.

### Impact Explanation

**High.** A user who calls `DisburseMaturity` and then `Disburse` (stake) within the 7-day finalization window, and whose neuron is subsequently removed from governance state, permanently loses the maturity that was deducted but never minted. The maturity is neither refunded nor minted — it is simply gone. This is a direct monetary loss of SNS governance tokens for the affected user.

Even without neuron removal, the interaction creates a confusing state where a neuron with zero stake has pending maturity disbursements, which may cause operational issues in the finalization path.

### Likelihood Explanation

**Medium.** The SNS `Disburse` command is callable by any principal with `NeuronPermissionType::Disburse` on a dissolved neuron. A user who has previously called `DisburseMaturity` (which is a separate, explicit action) and then decides to fully exit their position by calling `Disburse` within the 7-day window is a realistic scenario. The SNS governance canister is directly callable by any IC principal without going through a front-end, so there is no UI-level enforcement of the correct ordering.

### Recommendation

1. **Preferred:** In `disburse_neuron`, check whether `neuron.disburse_maturity_in_progress` is non-empty and return a `PreconditionFailed` error if so, requiring the user to wait for all pending maturity disbursements to finalize before disbursing the stake.

2. **Alternative:** When `remove_neuron` is called, iterate over any remaining `disburse_maturity_in_progress` entries and either finalize them immediately or refund the maturity back to the user's account before deletion.

### Proof of Concept

1. User has an SNS neuron in `Dissolved` state with `maturity_e8s_equivalent = 1_000_000_000` (10 SNS tokens).
2. User calls `manage_neuron` → `DisburseMaturity { percentage_to_disburse: 100, to_account: None }`.
   - `maturity_e8s_equivalent` is set to `0`.
   - `disburse_maturity_in_progress` gets one entry with `amount_e8s = 1_000_000_000` and `finalize_disbursement_timestamp_seconds = now + 7 days`.
   - No tokens are minted yet.
3. Within the 7-day window, user calls `manage_neuron` → `Disburse { amount: None, to_account: None }`.
   - `disburse_neuron` checks: neuron is dissolved ✓, caller is authorized ✓, **no check for `disburse_maturity_in_progress`**.
   - Stake is transferred out. `cached_neuron_stake_e8s = 0`.
4. If the neuron is subsequently removed via `remove_neuron` (e.g., through a governance cleanup or re-stake flow that replaces the neuron), the `disburse_maturity_in_progress` entry is dropped with it.
5. The 1_000_000_000 e8s of maturity is permanently lost — never minted, never refunded.
