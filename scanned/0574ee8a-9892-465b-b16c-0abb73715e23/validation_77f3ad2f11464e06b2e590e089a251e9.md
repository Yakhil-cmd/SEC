### Title
No Slippage Protection for ckBTC Withdrawal Fee Deduction Between Request Submission and Transaction Processing - (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

---

### Summary

The ckBTC minter's `retrieve_btc` and `retrieve_btc_with_approval` endpoints immediately and irrevocably burn the full user-specified ckBTC amount at request time, but the actual Bitcoin network fee deducted from the user's BTC output is determined later — at transaction construction time — using the **current** Bitcoin fee rate. No user-controlled slippage parameter (e.g., `min_receive_amount` or `max_fee`) exists. If Bitcoin network fees spike significantly between request submission and batch processing, users receive substantially less BTC than expected. In the extreme case where fees exceed the withdrawal amount, the request is silently finalized as `AmountTooLow` with no ckBTC reimbursement.

---

### Finding Description

**Step 1 — ckBTC burned immediately at request time:**

In `retrieve_btc`, the full `args.amount` is burned from the user's ckBTC balance synchronously before the request is queued: [1](#0-0) 

The `RetrieveBtcRequest` stored in `pending_retrieve_btc_requests` contains only `amount`, `address`, `block_index`, `received_at`, and `reimbursement_account`. There is no `max_fee` or `min_receive_amount` field: [2](#0-1) 

**Step 2 — Fee determined later at batch processing time:**

The minter's timer task calls `submit_pending_requests`, which fetches the **current** Bitcoin fee rate at that moment and passes it to `build_unsigned_transaction`: [3](#0-2) 

**Step 3 — Fee deducted from user's BTC output:**

Inside `build_unsigned_transaction_from_inputs`, the fee is computed from the current fee rate and subtracted from each recipient output: [4](#0-3) 

**Step 4 — No reimbursement on `AmountTooLow`:**

If the current fee rate has spiked so much that `fee + minter_fee > amount`, the request is finalized as `AmountTooLow` with no ckBTC reimbursement — the burned ckBTC is permanently lost: [5](#0-4) 

**Batching delay window:**

Requests sit in the queue until `can_form_a_batch` returns true — either when `MIN_PENDING_REQUESTS` accumulates or when `max_time_in_queue_nanos` elapses for the oldest request. This creates a window of hours during which Bitcoin fees can change dramatically: [6](#0-5) 

**No slippage parameter in the API:**

The `retrieve_btc` DID interface accepts only `address` and `amount` — no `min_receive_amount` or `max_fee_per_vbyte`: [7](#0-6) 

---

### Impact Explanation

A user calling `retrieve_btc(amount = 50_000_000 satoshi)` when the Bitcoin fee rate is 5 sat/vbyte expects to receive approximately `50_000_000 - ~3_000` satoshi. If the request sits in the queue and Bitcoin fees spike to 500 sat/vbyte (a common occurrence during Ordinals inscription waves or halving events), the user receives approximately `50_000_000 - ~300_000` satoshi — a loss of ~0.3% of their withdrawal. For near-minimum withdrawals, the entire amount can be consumed by fees, triggering `AmountTooLow` finalization with **zero reimbursement** of the already-burned ckBTC. This is a direct, non-recoverable ledger conservation loss for the user. [8](#0-7) 

---

### Likelihood Explanation

Bitcoin transaction fees are highly volatile and well-documented to spike 10–100× during network congestion events. The `max_time_in_queue_nanos` parameter (configurable, used in production) means requests can legitimately sit pending for hours. Any user submitting a `retrieve_btc` request during a low-fee period who has their request processed during a high-fee period is exposed. This is a realistic, recurring scenario on Bitcoin mainnet. The `estimate_withdrawal_fee` query endpoint exists but is advisory only — the user cannot bind the minter to honor a fee estimate. [9](#0-8) 

---

### Recommendation

Add an optional `max_fee` or `min_receive_amount` field to `RetrieveBtcArgs` and `RetrieveBtcWithApprovalArgs`. At batch processing time in `build_unsigned_transaction`, check whether the computed fee exceeds the user's specified maximum. If it does, reschedule the request (rather than finalizing it as `AmountTooLow`) and reimburse the user's ckBTC. This mirrors the pattern already used for `InsufficientTransactionFee` in the ckETH minter, where the request is moved back to the end of the queue rather than silently dropped. [10](#0-9) 

---

### Proof of Concept

1. Bitcoin mainnet fee rate is 5 sat/vbyte. User calls `retrieve_btc_with_approval(address="bc1q...", amount=100_000)`. The full 100,000 satoshi of ckBTC is burned immediately.
2. The request enters `pending_retrieve_btc_requests` with `received_at = T`.
3. No other requests arrive. The minter waits until `max_time_in_queue_nanos` elapses.
4. During this window, a Bitcoin congestion event occurs; fee rate rises to 800 sat/vbyte.
5. `submit_pending_requests` fires, calls `estimate_fee_per_vbyte` → returns 800 sat/vbyte.
6. `build_unsigned_transaction` computes `fee ≈ 800 * 140 vbytes = 112_000 satoshi`.
7. Since `112_000 + minter_fee > 100_000`, `BuildTxError::AmountTooLow` is returned.
8. The request is finalized as `FinalizedStatus::AmountTooLow` with no reimbursement.
9. The user has lost 100,000 satoshi of ckBTC with zero BTC received. [11](#0-10) [5](#0-4)

    pub fn can_form_a_batch(&self, min_pending: usize, now: u64) -> bool {
        if self.pending_retrieve_btc_requests.len() >= min_pending {
            return true;
        }

        if let Some(req) = self.pending_retrieve_btc_requests.first()
            && self.max_time_in_queue_nanos < now.saturating_sub(req.received_at)
        {
            return true;
        }

        if let Some(req) = self.pending_retrieve_btc_requests.last()
            && let Some(last_submission_time) = self.last_transaction_submission_time_ns
            && self.max_time_in_queue_nanos < req.received_at.saturating_sub(last_submission_time)
        {
            return true;
        }

        false
    }
