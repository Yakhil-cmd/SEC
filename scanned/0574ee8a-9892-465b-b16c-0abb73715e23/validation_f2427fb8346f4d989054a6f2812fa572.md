### Title
Unbounded Iteration in SNS `distribute_rewards` Can Permanently Halt Voting Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

### Summary
The SNS governance canister's `distribute_rewards` function iterates over all voting neurons in a single message execution with no instruction-limit guard. On a large SNS with many neurons and proposals, this causes the timer callback to trap on every invocation, permanently preventing voting rewards from being distributed. NNS governance already fixed the identical issue via a chunked, multi-message distribution state machine; SNS governance has not received the same fix.

### Finding Description

`distribute_rewards` is called from `run_periodic_tasks` via a timer: [1](#0-0) 

Inside `distribute_rewards`, two unbounded loops execute synchronously within a single message:

**Loop 1** — builds `neuron_id_to_reward_shares` by iterating over every ballot of every `ReadyToSettle` proposal. With `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700` proposals and `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` neurons, this is up to 140 million iterations: [2](#0-1) [3](#0-2) 

**Loop 2** — distributes rewards to each neuron, performing `Decimal` division per entry: [4](#0-3) 

Neither loop checks the IC instruction counter or breaks early. There is no equivalent of `is_message_over_threshold` anywhere in this path.

**Contrast with NNS governance**, which already fixed this exact pattern. NNS `distribute_pending_rewards` checks `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` after each neuron and persists remaining work to a `StableBTreeMap` for the next timer tick: [5](#0-4) [6](#0-5) 

The NNS changelog explicitly documents this fix: [7](#0-6) 

### Impact Explanation

On the IC, if a canister message exceeds the per-message instruction limit (~40 billion instructions for update calls), the execution traps and **all state changes are rolled back**. Because `distribute_rewards` performs all mutations (neuron maturity updates, proposal settlement, `latest_reward_event` update) in a single atomic message:

- If the instruction limit is hit mid-loop, the entire function rolls back.
- Proposals remain in `ReadyToSettle` state.
- `latest_reward_event` is not updated, so `should_distribute_rewards()` returns `true` again.
- The next timer tick calls `distribute_rewards` again, hits the same limit, and traps again.
- **Voting rewards are permanently stuck.** No neuron ever receives maturity for the affected reward period. [8](#0-7) 

### Likelihood Explanation

The default `max_number_of_neurons` is 200,000 and `max_number_of_proposals_with_ballots` is 700: [9](#0-8) 

Any unprivileged principal can create neurons by staking SNS tokens (no special permission required). Any neuron holder meeting the minimum stake threshold can submit proposals. As an SNS grows organically, the neuron count and proposal backlog naturally approach these ceilings. No adversarial action is required — normal operation of a popular SNS is sufficient to trigger the condition. The NNS team already recognized this as a real production risk and shipped the chunked-distribution fix for NNS governance.

### Recommendation

Port the NNS chunked-distribution pattern to SNS governance:

1. Introduce a `RewardsDistributionStateMachine` (or equivalent) backed by stable memory for SNS.
2. In `distribute_rewards`, compute the `neuron_id_to_reward_shares` map and persist it, then return.
3. In a separate periodic task (or the same timer), call `continue_processing` with an `is_message_over_threshold` guard that breaks the loop and re-schedules when the instruction budget is nearly exhausted.
4. Only update `latest_reward_event` and settle proposals after the distribution is fully complete.

### Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_number_of_proposals_with_ballots = 700`.
2. Create 200,000 neurons (each staking the minimum SNS token amount).
3. Submit 700 proposals and have all neurons vote on each.
4. Wait for the reward round to end; `should_distribute_rewards()` returns `true`.
5. The timer fires `run_periodic_tasks` → `distribute_rewards`.
6. Loop 1 iterates ~140 million ballot entries; Loop 2 iterates ~200,000 neurons with `Decimal` arithmetic.
7. The message traps due to instruction limit exhaustion; all state rolls back.
8. Every subsequent timer tick repeats steps 5–7 indefinitely.
9. No neuron ever receives voting rewards; `latest_reward_event` is never advanced. [10](#0-9) [11](#0-10)

    fn continue_processing(
        &mut self,
        neuron_store: &mut NeuronStore,
        is_over_instructions_limit: fn() -> bool,
    ) {
        while let Some((id, reward_e8s)) = self.rewards.pop_first() {
            match neuron_store.with_neuron_mut(&id, |neuron| {
                let auto_stake = neuron.auto_stake_maturity.unwrap_or(false);
                if auto_stake {
                    neuron.staked_maturity_e8s_equivalent = Some(
                        neuron
                            .staked_maturity_e8s_equivalent
                            .unwrap_or_default()
                            .saturating_add(reward_e8s),
                    );
                } else {
                    neuron.maturity_e8s_equivalent =
                        neuron.maturity_e8s_equivalent.saturating_add(reward_e8s);
                }
            }) {
                Ok(_) => {}
                Err(e) => {
                    println!(
                        "{}Error rewarding neuron {:?} during reward_distribution.\
                    This should not be possible as neuron existence is checked when \
                    rewards are calculated: {}",
                        LOG_PREFIX, id, e
                    );
                }
            };
            if is_over_instructions_limit() {
                break;
