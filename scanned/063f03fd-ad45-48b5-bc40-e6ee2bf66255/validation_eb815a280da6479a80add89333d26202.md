### Title
VetKD Key Share `node_signature` Lacks Request-Context Binding, Enabling Cross-Request Replay - (File: `rs/crypto/src/vetkd/mod.rs`)

---

### Summary

The `node_signature` field of `VetKdEncryptedKeyShare` is an Ed25519 basic signature over only the raw bytes of the encrypted key share content (`VetKdEncryptedKeyShareContent`). It does not bind to the `request_id`, `ni_dkg_id`, `caller`, derivation `context`, `input`, or `transport_public_key`. A malicious committee node (a protocol peer below the consensus fault threshold) can therefore replay a valid `(encrypted_key_share, node_signature)` pair produced for one VetKD request as its contribution to a different request. The share-validation step (`verify_encrypted_key_share`) accepts the replayed share because the Ed25519 signature is still valid over the same bytes. The replayed share is then included in the combination step, producing a cryptographically incorrect combined key, causing `verify_encrypted_key` to fail and the VetKD key-derivation request to be denied.

---

### Finding Description

**Root cause — signed bytes contain no request context.**

`VetKdEncryptedKeyShareContent` implements `SignedBytesWithoutDomainSeparator` by appending only the raw BLS12-381 group-element bytes: [1](#0-0) 

The `SignatureDomain` implementation prepends the fixed string `"ic-vetkd-encrypted-key-share-content-domain"`: [2](#0-1) 

Neither the `request_id`, `ni_dkg_id`, `caller`, `context`, `input`, nor `transport_public_key` are included in the signed payload.

**Creation — signature is over bare ciphertext bytes only.**

In `create_encrypted_key_share_internal`, after the BLS12-381 encrypted share is produced, the node signs only the `encrypted_key_share` value: [3](#0-2) 

The `VetKdArgs` fields (`request_id` lives in the outer `VetKdKeyShare` wrapper, `ni_dkg_id`, `caller`, `context`, `input`) are not mixed into the signed bytes.

**Verification — only the Ed25519 signature is checked, not context binding.**

`verify_encrypted_key_share_internal` reconstructs the signature and calls `verify_basic_sig` against `key_share.encrypted_key_share` alone: [4](#0-3) 

The `args` parameter (which carries `ni_dkg_id`, `caller`, `context`, `input`) is used only to look up the registry version; it is never mixed into the verification predicate. A share produced for request R1 with context C1 will pass verification when submitted for request R2 with context C2, because the Ed25519 signature is still valid over the same ciphertext bytes.

**Combination — no per-share cryptographic validity check.**

`combine_encrypted_key_shares_internal` deserializes each share and passes it directly to the BLS12-381 combination routine without calling `EncryptedKeyShare::is_valid`: [5](#0-4) 

A replayed share (computed for C1 but submitted for C2) is therefore included in the Lagrange interpolation, producing a combined key that is incorrect for C2. The subsequent `verify_encrypted_key` call fails.

**Outer artifact — `request_id` is metadata only, not covered by the signature.**

The `VetKdKeyShare` struct carries `request_id` and `signer_id` as unsigned metadata alongside the `VetKdEncryptedKeyShare`: [6](#0-5) 

A malicious node can freely set `request_id` to any target request while reusing a previously produced `(encrypted_key_share, node_signature)` pair.

**Exploit path.**

1. Malicious committee node N creates a legitimate share for request R1 (context C1, input I1, transport key TPK1): bytes B1, signature S = Ed25519_sign(N_sk, domain || B1).
2. N broadcasts `VetKdKeyShare { signer_id: N, request_id: R2, share: { encrypted_key_share: B1, node_signature: S } }` targeting request R2 (context C2 ≠ C1).
3. Honest nodes call `verify_encrypted_key_share(N, share, args_R2)`. The check passes: S is a valid Ed25519 signature over B1 by N.
4. The share is moved to the validated pool for R2.
5. Combination for R2 includes B1 (computed for C1). Lagrange interpolation over mismatched shares produces an incorrect combined key.
6. `verify_encrypted_key` fails; the VetKD key-derivation request for R2 is denied.

---

### Impact Explanation

A single malicious committee node (one protocol peer, well below the consensus fault threshold) can persistently cause VetKD key-derivation requests to fail for any request it targets. Because the replayed share passes the validation gate and enters the validated pool, honest nodes have no mechanism to exclude it before combination. The result is a targeted, sustained denial-of-service against the VetKD feature: canisters that depend on `vetkd_public_key` / `vetkd_encrypted_key` management-canister calls will receive errors for affected requests.

---

### Likelihood Explanation

The attacker must be a node operator whose node is selected into the VetKD NI-DKG committee. Committee membership is determined by governance-controlled NI-DKG, so the attacker must control a legitimate subnet node. This is a realistic threat model for a compromised or malicious node operator. The exploit itself requires no special tooling beyond replaying a previously produced share artifact with a different `request_id` field.

---

### Recommendation

Include the full request context in the signed bytes for `VetKdEncryptedKeyShareContent`. Specifically, the signed payload should commit to at minimum: `ni_dkg_id`, `caller`, `context`, `input`, and `transport_public_key`. This mirrors the pattern used by ECDSA/Schnorr sig shares, where the share bytes are cryptographically derived from the full request inputs and therefore implicitly context-bound. Alternatively, add an explicit context-binding wrapper (analogous to how `IDkgComplaintContent` wraps the complaint with its transcript ID) so that `verify_encrypted_key_share` can reject a share whose signed bytes do not match the `args` of the request being verified.

---

### Proof of Concept

```
// Node N holds a valid share for R1:
let share_r1 = crypto_N.create_encrypted_key_share(args_r1).unwrap();
// share_r1.encrypted_key_share = B1, share_r1.node_signature = S

// N broadcasts the same bytes as its share for R2:
let replayed = VetKdKeyShare {
    signer_id: N,
    request_id: R2,          // different request
    share: VetKdEncryptedKeyShare {
        encrypted_key_share: share_r1.encrypted_key_share.clone(), // B1
        node_signature:      share_r1.node_signature.clone(),       // S
    },
};

// Honest node calls verify_encrypted_key_share(N, replayed.share, args_r2):
// → verify_basic_sig(S, B1, N) → OK  (S is valid over B1 regardless of context)
// → share accepted into validated pool for R2

// Combination for R2 includes B1 (computed for C1):
// → combined key is wrong for C2
// → verify_encrypted_key fails → R2 denied
```

The `verify_encrypted_key_share_internal` function verifies only `BasicSigVerifierInternal::verify_basic_sig(..., &key_share.encrypted_key_share, signer, ...)` and never checks that `encrypted_key_share` was computed under the `args` of the target request. [7](#0-6) [1](#0-0)

fn verify_encrypted_key_share_internal<S: CspSigner>(
    lockable_threshold_sig_data_store: &LockableThresholdSigDataStore,
    registry: &dyn RegistryClient,
    csp_signer: &S,
    key_share: &VetKdEncryptedKeyShare,
    signer: NodeId,
    args: &VetKdArgs,
) -> Result<(), VetKdKeyShareVerificationError> {
    let registry_version_from_store = lockable_threshold_sig_data_store
        .read()
        .transcript_data(args.ni_dkg_id)
        .map(|transcript_data| transcript_data.registry_version())
        .ok_or_else(|| {
            VetKdKeyShareVerificationError::ThresholdSigDataNotFound(
                ThresholdSigDataNotFoundError::ThresholdSigDataNotFound {
                    dkg_id: args.ni_dkg_id.clone(),
                },
            )
        })?;

    let signature = BasicSigOf::new(BasicSig(key_share.node_signature.clone()));
    BasicSigVerifierInternal::verify_basic_sig(
        csp_signer,
        registry,
        &signature,
        &key_share.encrypted_key_share,
        signer,
        registry_version_from_store,
    )
    .map_err(VetKdKeyShareVerificationError::VerificationError)
}
