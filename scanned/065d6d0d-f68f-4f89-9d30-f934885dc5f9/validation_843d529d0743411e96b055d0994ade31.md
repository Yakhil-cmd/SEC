### Title
CMC `ensure_balance` Ignores Existing Balance Before Computing Cycles to Mint — (File: rs/nns/cmc/src/main.rs)

### Summary
The `ensure_balance` function in the Cycles Minting Canister (CMC) computes `cycles_to_mint = cycles - current_balance` without first checking whether the CMC's existing cycles balance already covers the requested amount. When `current_balance > cycles`, this subtraction underflows (wraps to a near-`u128::MAX` value in release builds), causing the minting-rate-limiter check to reject the operation and making `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` fail for any user whose requested amount is smaller than the CMC's accumulated balance.

### Finding Description
`ensure_balance` is the single function that all three ICP-to-cycles conversion paths call before sending cycles:

```rust
// rs/nns/cmc/src/main.rs  lines 2306-2325
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;   // ← no guard; underflows when balance > cycles

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
``` [1](#0-0) 

It is called unconditionally from `deposit_cycles` (top-up path), `do_mint_cycles` (cycles-ledger path), and `do_create_canister` (canister-creation path): [2](#0-1) [3](#0-2) [4](#0-3) 

The function's own doc-comment states the intent: *"Ensure the Cycles Minting canister has at least `cycles` balance of cycles, **otherwise**, mint more."* The "otherwise" guard is missing. In a Rust `--release` build, `Cycles` (a `u128` newtype) wraps on underflow, so `cycles - current_balance` when `current_balance > cycles` produces a value near `u128::MAX`. The subsequent `check_and_add_cycles` call then rejects the operation because that wrapped value exceeds the per-hour minting limit: [5](#0-4) 

The CMC's existing balance — the "buffer" — is never consulted before the minting path is entered, directly mirroring the original report's pattern.

### Impact Explanation
Every call to `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` whose requested cycle amount is smaller than the CMC's current balance will fail with a rate-limit error. The user's ICP has already been transferred to the CMC's subaccount at that point; recovery depends on the refund path executing correctly. At scale this is a complete denial-of-service of the ICP→cycles conversion service.

**Impact: Medium** — operations fail and users cannot top up or create canisters; ICP is temporarily locked pending refund.

### Likelihood Explanation
Any canister on any subnet can call `deposit_cycles` on the management canister targeting the CMC canister ID, depositing an arbitrary number of cycles into the CMC's balance. This is a standard, permissionless management-canister call. An attacker needs only a canister with enough cycles to exceed the smallest common top-up amount (e.g., depositing 1 T cycles blocks all top-ups of < 1 T cycles). The CMC's balance is normally near zero, so the condition is not triggered accidentally, but it is trivially reachable by any unprivileged canister caller.

**Likelihood: Medium** — requires a deliberate but cheap and permissionless action.

### Recommendation
Add an early-return guard at the top of `ensure_balance`:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    if current_balance >= cycles {
        return Ok(());   // existing balance is sufficient; no minting needed
    }
    let cycles_to_mint = cycles - current_balance;
    // ... rest unchanged
}
```

This mirrors the Lido-style "use the buffer first" fix recommended in the original report.

### Proof of Concept
1. Attacker creates a canister with 2 T cycles on any application subnet.
2. Attacker calls `ic00::deposit_cycles({ canister_id: CMC_CANISTER_ID })` with 2 T cycles attached. CMC balance is now 2 T.
3. Legitimate user transfers 1 T ICP-equivalent to the CMC's top-up subaccount and calls `notify_top_up({ block_index, canister_id })`.
4. CMC calls `deposit_cycles(canister_id, 1T, true, BaseLimit)` → `ensure_balance(1T, BaseLimit)`.
5. `current_balance = 2T`, `cycles_to_mint = 1T - 2T` wraps to `u128::MAX - 1T + 1` ≈ `3.4 × 10^38`.
6. `check_and_add_cycles(state, now, ~3.4e38)` fails: amount exceeds `base_cycles_limit` (150 × 10^15).
7. `ensure_balance` returns `Err(...)`, `deposit_cycles` returns `Err(...)`, `process_top_up` triggers `refund_icp`.
8. User's top-up silently fails; attacker repeats step 2 whenever the CMC's balance drops below the target threshold.
