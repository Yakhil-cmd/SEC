### Title
ICRC-106 Standard URL Mismatch in `icrc1_supported_standards` — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

---

### Summary

The `icrc1_supported_standards` / `icrc10_supported_standards` endpoint of the production ICRC ledger canister returns a GitHub **pull-request** URL for the ICRC-106 standard instead of a canonical specification URL. Every other standard in the same list uses a stable `blob/main` or `tree/main` URL. Any wallet, aggregator, or protocol that validates the URL field of the returned `StandardRecord` to confirm ICRC-106 compliance will receive incorrect information.

---

### Finding Description

In `rs/ledger_suite/icrc1/ledger/src/main.rs`, the `supported_standards()` function (exposed as both `icrc1_supported_standards` and `icrc10_supported_standards`) hard-codes the following entry for ICRC-106:

```rust
StandardRecord {
    name: "ICRC-106".to_string(),
    url: "https://github.com/dfinity/ICRC/pull/106".to_string(),  // ← PR URL
},
``` [1](#0-0) 

Every other standard in the same list uses a stable, merged specification URL:

| Standard | URL pattern |
|---|---|
| ICRC-1 | `…/blob/main/standards/ICRC-1` |
| ICRC-10 | `…/blob/main/ICRCs/ICRC-10/ICRC-10.md` |
| ICRC-21 | `…/blob/main/ICRCs/ICRC-21/ICRC-21.md` |
| ICRC-103 | `…/tree/main/ICRCs/ICRC-103` |
| ICRC-152 | `…/blob/main/ICRCs/ICRC-152.md` |
| **ICRC-106** | **`…/pull/106`** ← GitHub PR, not a spec | [2](#0-1) 

The same pattern appears in the ICRC-1 archive canister, where `icrc3_supported_block_types` advertises the ICRC-107 fee-collector block type with a PR URL (`https://github.com/dfinity/ICRC/pull/117`) instead of a merged specification URL:

```rust
SupportedBlockType {
    block_type: BTYPE_107.to_string(),
    url: "https://github.com/dfinity/ICRC/pull/117".to_string(),  // ← PR URL
},
``` [3](#0-2) 

The ICRC-10 standard mandates that the `url` field in a `StandardRecord` points to the **specification document** for the named standard. A PR URL is not a specification URL; it is a mutable, potentially-closeable GitHub page that does not represent a finalized standard.

---

### Impact Explanation

The `icrc1_supported_standards` and `icrc10_supported_standards` endpoints are the IC equivalent of ERC-165's `supportsInterface()`: they are the canonical on-chain mechanism by which wallets, DEXes, aggregators, and cross-chain bridges discover which standards a ledger canister implements. The ICRC-10 specification explicitly defines the `url` field as the normative reference for the standard.

Consequences of the wrong URL:

1. **Compatibility-check failures.** Any off-chain or on-chain tool that validates the URL field (e.g., checks that it is a `blob/main` or `tree/main` GitHub URL, or that it resolves to a document containing the standard's Candid interface) will incorrectly conclude that the ledger does not properly implement ICRC-106.
2. **Broken specification lookup.** Wallets or developer tooling that fetch the specification from the advertised URL to auto-generate consent messages or validate method signatures will receive a GitHub PR page (which may be closed, redirected, or deleted) rather than the stable specification.
3. **Ecosystem fragmentation.** Third-party ledger implementations that copy the reference URL from the DFINITY production ledger will propagate the incorrect URL, widening the mismatch across the ecosystem.

---

### Likelihood Explanation

The affected query endpoints (`icrc1_supported_standards`, `icrc10_supported_standards`) are publicly callable by any unprivileged user or canister with a simple query call — no authentication, cycles, or special role is required. The ICRC ledger is deployed on mainnet and is the reference implementation used by ckBTC, ckETH, ckERC-20, and SNS token ledgers. Any wallet or aggregator that performs standard-discovery against these canisters will observe the incorrect URL on every call.

---

### Recommendation

Replace the PR URL with the canonical merged specification URL for ICRC-106 in `rs/ledger_suite/icrc1/ledger/src/main.rs`:

```rust
StandardRecord {
    name: "ICRC-106".to_string(),
    url: "https://github.com/dfinity/ICRC/blob/main/ICRCs/ICRC-106/ICRC-106.md".to_string(),
},
```

Similarly, update the BTYPE_107 URL in `rs/ledger_suite/icrc1/archive/src/main.rs` to point to the merged ICRC-107 specification rather than the PR.

Consider adding a CI lint rule that rejects any `StandardRecord` or `SupportedBlockType` whose `url` field matches the pattern `github.com/.*/pull/\d+`.

---

### Proof of Concept

Any unprivileged caller can observe the mismatch with a single query:

```bash
dfx canister --network ic call \
  mxzaz-hqaaa-aaaar-qaada-cai \   # ckBTC ledger (ICRC ledger instance)
  icrc1_supported_standards '()'
```

The response will include:

```
record { name = "ICRC-106"; url = "https://github.com/dfinity/ICRC/pull/106" }
```

A conforming ICRC-10 implementation should return a stable specification URL (e.g., `…/blob/main/ICRCs/ICRC-106/…`). The returned PR URL does not satisfy the ICRC-10 requirement that `url` identifies the specification document.
