### Title
Overly Broad Self-Approval Check Blocks All Cross-Subaccount ICRC-2 Approvals — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
`icrc2_approve_not_async` compares only the `owner` (Principal) field when checking for self-approval, ignoring the `subaccount` field. This causes a canister trap for any approval where `from.owner == spender.owner`, even when `from.subaccount != spender.subaccount`. Under ICRC-1, two accounts with the same owner but different subaccounts are distinct accounts; ICRC-2 must permit approvals between them.

### Finding Description

The guard at line 828 in `rs/ledger_suite/icrc1/ledger/src/main.rs`:

```rust
if from_account.owner == arg.spender.owner {
    ic_cdk::trap("self approval is not allowed")
}
```

compares only the `owner` field of the two `Account` structs. [1](#0-0) 

An ICRC-1 `Account` is `{owner: Principal, subaccount: Option<Subaccount>}`. Two accounts with the same `owner` but different `subaccount` values are **distinct accounts** per the ICRC-1 spec. The ICRC-2 spec's self-approval prohibition applies to the full account pair, not just the principal.

The lower-level `AllowanceTable::approve` in `rs/ledger_suite/common/ledger_core/src/approvals.rs` correctly uses full-account equality:

```rust
if account == spender {
    return Err(ApproveError::SelfApproval);
}
``` [2](#0-1) 

But this code is **never reached** for cross-subaccount same-principal calls because the `ic_cdk::trap` at line 829 fires first and unwinds the entire call, returning `CanisterCalledTrap` to the caller instead of a graceful `Err(ApproveError::...)`.

The identical bug exists in the ICP ledger: [3](#0-2) 

The existing `test_approve_self` test only validates the true self-approval case (`spender = from.0.into()`, same principal, same default subaccount) and does not cover the cross-subaccount case: [4](#0-3) 

### Impact Explanation

Any call to `icrc2_approve` where the caller's principal matches the spender's principal — regardless of differing subaccounts — traps unconditionally. This permanently prevents:

- A user's default account `{owner: A, subaccount: None}` from approving their own subaccount `{owner: A, subaccount: Some([1;32])}` as a spender.
- Any DeFi protocol, vault, or delegation pattern that routes funds through same-principal subaccounts using ICRC-2 approvals.

If a protocol has already locked assets under the assumption that cross-subaccount approvals work (a reasonable assumption given the ICRC-2 spec), those assets become permanently inaccessible via the approval pathway.

### Likelihood Explanation

The entrypoint is the public `icrc2_approve` update method, reachable by any unprivileged ingress call. No special role or key is required. The trigger condition (same principal, different subaccounts) is a standard and documented ICRC-1/ICRC-2 usage pattern. Any user or protocol attempting this pattern will hit the trap deterministically.

### Recommendation

Change the self-approval guard in both ledgers from a principal-only comparison to a full-account comparison:

```rust
// Before (incorrect):
if from_account.owner == arg.spender.owner {
    ic_cdk::trap("self approval is not allowed")
}

// After (correct):
if from_account == arg.spender {
    ic_cdk::trap("self approval is not allowed")
}
```

This matches the semantics of the lower-level `AllowanceTable::approve` check and the ICRC-2 spec. Add a state-machine test covering the cross-subaccount case to prevent regression.

### Proof of Concept

State-machine test (pseudocode):

```rust
let principal_a = PrincipalId::new_user_test_id(1);
let subaccount_1 = Some([1u8; 32]);
let subaccount_2 = Some([2u8; 32]);

// Fund the from-account
let (env, canister_id) = setup(ledger_wasm, encode_init_args,
    vec![(Account { owner: principal_a.0, subaccount: subaccount_1 }, 1_000_000)]);

let approve_args = ApproveArgs {
    from_subaccount: subaccount_1,
    spender: Account { owner: principal_a.0, subaccount: subaccount_2 },
    amount: Nat::from(500_000_u64),
    ..default_approve_args()
};

// This should succeed per ICRC-2 spec (distinct accounts), but instead traps:
let result = env.execute_ingress_as(principal_a, canister_id, "icrc2_approve",
    Encode!(&approve_args).unwrap());

// Observed: Err(CanisterCalledTrap("self approval is not allowed"))
// Expected: Ok(block_index)
assert!(result.is_ok(), "cross-subaccount approval must succeed");
```
