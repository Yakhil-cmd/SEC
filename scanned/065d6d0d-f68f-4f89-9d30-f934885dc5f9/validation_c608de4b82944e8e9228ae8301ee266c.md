### Title
Missing stashed pre-signature transcripts in `get_active_transcripts` causes key deletion and signing failures — (`rs/consensus/idkg/src/lib.rs`)

### Summary

`get_active_transcripts` in `lib.rs` omits the actual pre-signature transcripts stored in `pre_signature_stashes` from the active set it passes to `retain_keys_for_transcripts`. Only the stash's `key_transcript` is retained. The `complaints.rs::active_transcripts` function correctly includes both. Because `retain_keys_for_transcripts` deletes every canister threshold key **not** identified by an active transcript, the keys for stashed pre-signature transcripts are deleted in normal operation, causing `create_sig_share` to fail with `SecretSharesNotFound` when those pre-signatures are later consumed.

### Finding Description

**`get_active_transcripts` in `lib.rs` (lines 415–429):**

```rust
if let Some(snapshot) = state_reader.get_certified_state_snapshot() {
    let state = snapshot.get_state();

    // Retain all stashed key transcripts
    for stash in state.pre_signature_stashes().values() {
        active_transcripts.insert((*stash.key_transcript).clone()); // ← only key_transcript
    }

    // Retain transcripts paired with ongoing requests
    for request in state.signature_request_contexts().values() {
        for transcript in request.iter_idkg_transcripts() {
            active_transcripts.insert(transcript.clone());
        }
    }
}
``` [1](#0-0) 

The loop over `pre_signature_stashes` inserts only `stash.key_transcript`. It never iterates `stash.pre_signatures.values().flat_map(|pre_sig| pre_sig.iter_idkg_transcripts())`.

**`complaints.rs::active_transcripts` (lines 752–778) correctly includes them:**

```rust
let stashed_pre_sig_transcripts = state
    .pre_signature_stashes()
    .values()
    .flat_map(|stash| stash.pre_signatures.values())
    .flat_map(|pre_sig| pre_sig.iter_idkg_transcripts())
    .map(|transcript| (transcript.transcript_id, transcript));
``` [2](#0-1) 

**Why the blockchain path does not compensate:** `IDkgPayload::active_transcripts()` covers `available_pre_signatures` and `pre_signatures_in_creation` in the finalized tip. [3](#0-2) 

But `create_data_payload_helper_2` explicitly clears `available_pre_signatures` in every new payload because they were already delivered to execution:

```rust
// We purge available pre-signatures of the parent payload,
// because they were already delivered with the previous payload.
idkg_payload.available_pre_signatures.clear();
``` [4](#0-3) 

Once pre-signatures are in `pre_signature_stashes` in replicated state, they are absent from `available_pre_signatures` in every subsequent payload. The blockchain path provides no coverage.

**`retain_keys_for_transcripts` deletes all keys not in the active set:**

> "IDKG threshold keys not identified by a transcript will be removed." [5](#0-4) 

The vault call `idkg_retain_active_keys(active_key_ids, oldest_public_key)` is the deletion mechanism: [6](#0-5) 

### Impact Explanation

The complete lifecycle that triggers the failure:

1. Pre-signatures complete and are placed in `available_pre_signatures` in the IDKG payload.
2. They are delivered to execution via `AvailablePreSignatures` in the batch and merged into `pre_signature_stashes` in replicated state. [7](#0-6) 
3. The next IDKG payload clears `available_pre_signatures` — the transcripts are now **only** in the stash.
4. `purge_inactive_transcripts` fires (every `INACTIVE_TRANSCRIPT_PURGE_SECS`), calls `get_active_transcripts`, which omits the stashed pre-signature transcripts. [8](#0-7) 
5. `retain_keys_for_transcripts` deletes the canister threshold keys for those transcripts.
6. A signing request is matched with a stashed pre-signature via `match_contexts_with_stashed_pre_signatures`. [9](#0-8) 
7. `create_sig_share` fails with `SecretSharesNotFound` because the keys were deleted.

### Likelihood Explanation

This is **not** a theoretical race condition requiring an attacker. It is a deterministic omission in `get_active_transcripts` that fires on every node in every subnet that uses threshold signing, on every purge cycle after pre-signatures have been stashed. No adversarial action is required. The "delay state certification" framing in the question is a red herring — the bug manifests regardless of certification timing, because even a fresh certified snapshot will have the stash but `get_active_transcripts` still omits the pre-signature transcripts from it.

### Recommendation

In `get_active_transcripts` (`rs/consensus/idkg/src/lib.rs`), add the same `stashed_pre_sig_transcripts` iteration that `complaints.rs::active_transcripts` already performs:

```rust
for stash in state.pre_signature_stashes().values() {
    active_transcripts.insert((*stash.key_transcript).clone());
    // ADD: retain pre-signature transcripts too
    for pre_sig in stash.pre_signatures.values() {
        for transcript in pre_sig.iter_idkg_transcripts() {
            active_transcripts.insert(transcript.clone());
        }
    }
}
```

A differential unit test comparing the active sets produced by both functions for the same state containing stashed pre-signatures would have caught this.

### Proof of Concept

The existing test `test_get_active_transcripts` in `lib.rs` (lines 653–719) already creates stashes with `fake_pre_signature_stash` and checks that `stashed_transcripts` (only `stashes.len()`) are added — it counts only one transcript per stash (the key transcript), not the pre-signature transcripts. Compare with the `complaints.rs` test at line 1172 which asserts `9` transcripts (1 key + 2×4 pre-signature transcripts) for the same stash structure. Running both tests against the same state with stashed pre-signatures and comparing the resulting active sets directly demonstrates the differential. [10](#0-9) [11](#0-10)

                    self.logger,

                    internal_error
                );

                    .client_errors
                    .with_label_values(&["retain_active_transcripts_transient"])
                    .inc();

            Err(error) => {

                    .inc();
            }
            Ok(()) => {
                self.metrics
                    .client_metrics
                    .with_label_values(&["retain_active_transcripts"])
                    .inc();
            }
        }
    }
