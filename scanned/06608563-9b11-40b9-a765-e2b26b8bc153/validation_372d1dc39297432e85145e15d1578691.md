### Title
Unbounded Header Count Causes 2-3× Memory Amplification Per Stream, Enabling OOM Crash via 1,000 Concurrent Streams — (`rs/p2p/quic_transport/src/request_handler.rs`)

---

### Summary

A Byzantine subnet peer with an established QUIC connection can craft `pb::HttpRequest` messages containing a large number of headers whose aggregate wire size stays below `MAX_MESSAGE_SIZE_BYTES` (128 MB). Processing each such message in `read_request` allocates memory in three distinct layers — raw wire bytes, decoded `Vec<HttpHeader>`, and `http::HeaderMap` — producing a 2–3× amplification over the wire size. With `MAX_CONCURRENT_BIDI_STREAMS = 1,000` concurrent streams permitted per connection, a single Byzantine peer can drive the victim replica into OOM.

---

### Finding Description

`read_request` in `rs/p2p/quic_transport/src/request_handler.rs` processes an incoming stream in three sequential allocation steps with no header-count guard:

**Step 1 — raw wire buffer** [1](#0-0) 
`read_to_end(MAX_MESSAGE_SIZE_BYTES)` buffers the entire stream into a `Vec<u8>` of up to 128 MB.

**Step 2 — protobuf decode** [2](#0-1) 
`pb::HttpRequest::decode` allocates a `Vec<HttpHeader>` on the heap. For N headers each with a 1 KB key and 1 KB value, this is another O(N × 2 KB) allocation — a second copy of the header data.

**Step 3 — HeaderMap construction** [3](#0-2) 
The unbounded loop over `request_proto.headers` calls `request_builder.header(key, value)` for every entry, inserting into `http::HeaderMap` (a hash table). This is a third allocation proportional to header count.

There is no check on `request_proto.headers.len()` anywhere in this path.

The proto schema places no constraint on header count either: [4](#0-3) 

The QUIC transport is configured to allow up to 1,000 concurrent bidirectional streams per connection: [5](#0-4) 

These streams are accepted and spawned without backpressure: [6](#0-5) 

The comment in the acceptor loop explicitly acknowledges the unbounded buffering risk: [7](#0-6) 

`MAX_MESSAGE_SIZE_BYTES` is 128 MB: [8](#0-7) 

---

### Impact Explanation

**Per-stream memory budget (concrete example):**

| Layer | Size |
|---|---|
| `read_to_end` raw buffer | ~100 MB |
| `Vec<HttpHeader>` from decode | ~100 MB |
| `http::HeaderMap` insertions | ~50–100 MB |
| **Total per stream** | **~250–300 MB** |

(Example: 50,000 headers × 1 KB key + 1 KB value = ~100 MB wire size, fits within 128 MB limit.)

**Across 1,000 concurrent streams:**
1,000 × ~250 MB = **~250 GB heap pressure** → OOM kill of the replica process.

The `STREAM_RECEIVE_WINDOW = 4 MB` controls QUIC flow-control pacing but does not cap total application-layer buffering; `read_to_end` advances the window continuously until the stream closes or the 128 MB cap is hit. [9](#0-8) 

Impact: **crash of a single replica** (OOM), matching the stated scope of a non-volumetric platform-level crash of a single replica.

---

### Likelihood Explanation

- The attacker must be a Byzantine node that is a current subnet member — its TLS certificate is in the registry and a QUIC connection is already established. This is within the IC's explicit Byzantine fault model (up to f nodes).
- No threshold corruption is required; a single Byzantine peer suffices.
- The attack requires only crafting a valid protobuf message with many headers — trivially constructable.
- No special timing, race condition, or external dependency is needed.
- The attack is repeatable and deterministic.

---

### Recommendation

1. **Add a header count limit** in `read_request` before the decode loop:
   ```rust
   const MAX_HEADERS_PER_REQUEST: usize = 100;
   if request_proto.headers.len() > MAX_HEADERS_PER_REQUEST {
       return Err(P2PError::from("too many headers".to_string()));
   }
   ``` [3](#0-2) 

2. **Add a per-header size limit** on key and value lengths.

3. **Add a concurrent-stream semaphore** in `start_stream_acceptor` to bound total in-flight memory across all streams from a single peer, rather than accepting up to `MAX_CONCURRENT_BIDI_STREAMS` unconditionally. [6](#0-5)

### Proof of Concept

```rust
// Byzantine peer sends 1000 concurrent streams, each with 50_000 headers × 2KB
for _ in 0..1000 {
    let (mut send, _recv) = conn.open_bi().await.unwrap();
    let mut headers = vec![];
    for i in 0..50_000usize {
        headers.push(pb::HttpHeader {
            key: "x".repeat(1024),
            value: vec![0u8; 1024],
        });
    }
    let req = pb::HttpRequest {
        uri: "/".to_string(),
        headers,
        method: pb::HttpMethod::Get as i32,
        body: vec![],
    };
    let bytes = req.encode_to_vec(); // ~100MB, within 128MB limit
    send.write_all(&bytes).await.unwrap();
    send.finish().unwrap();
    // Do NOT await response — keep all streams open concurrently
}
// Victim replica allocates ~250GB → OOM
```

Each stream's `read_request` call allocates ~250 MB. With 1,000 concurrent streams the victim process is killed by the OS OOM killer, crashing the replica.
