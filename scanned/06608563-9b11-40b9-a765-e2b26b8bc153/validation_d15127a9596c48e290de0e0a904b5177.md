Based on my code analysis, this is a real vulnerability. Here is the detailed breakdown:

---

### Title
Unvalidated EquivocationProof artifacts at heights ≤ finalized_height accumulate unboundedly in the unvalidated pool — (`rs/consensus/src/consensus/validator.rs`)

### Summary

`validate_equivocation_proofs` only iterates heights strictly above `finalized_height`, silently skipping (neither validating nor removing) any `EquivocationProof` artifacts at or below `finalized_height`. The purger's equivocation-proof cleanup only targets the **validated** pool. The sole mechanism that cleans the **unvalidated** pool — `PurgeUnvalidatedBelow` — is gated on `expected_batch_height` advancing. A Byzantine P2P peer can exploit this gap to grow the unvalidated pool without bound, causing OOM and replica crash.

### Finding Description

**Validator skips low-height proofs entirely:**

`validate_equivocation_proofs` constructs its iteration range as:

```rust
let range_to_validate = HeightRange::new(finalized_height.increment(), range.max);
``` [1](#0-0) 

Any `EquivocationProof` artifact sitting in the unvalidated pool at a height ≤ `finalized_height` is never touched — no `HandleInvalid`, no `RemoveFromUnvalidated`. It simply stays.

**Purger only cleans the validated pool for equivocation proofs:**

`purge_equivocation_proofs_by_finalized_height` emits only `PurgeValidatedOfTypeBelow`:

```rust
changeset.push(ChangeAction::PurgeValidatedOfTypeBelow(
    PurgeableArtifactType::EquivocationProof,
    finalized_height.increment(),
));
``` [2](#0-1) 

There is no corresponding `PurgeUnvalidatedOfTypeBelow` for equivocation proofs.

**The only unvalidated-pool cleanup is gated on `expected_batch_height` advancing:**

```rust
if *prev_expected_batch_height < expected_batch_height
    && expected_batch_height <= finalized_height.increment()
{
    ...
    changeset.push(ChangeAction::PurgeUnvalidatedBelow(
        expected_batch_height.decrement(),
    ));
}
``` [3](#0-2) 

If `expected_batch_height` does not advance (stalled subnet, slow execution), this block never fires and unvalidated artifacts at low heights are never removed.

**No per-peer quota or admission control found for equivocation proofs.** Searches for quota, per-peer limits, and unvalidated pool size bounds returned no results in the artifact pool or P2P layer for this artifact type.

### Impact Explanation

A Byzantine P2P peer (below the fault threshold — no majority needed) can:
1. Continuously send `EquivocationProof` artifacts (cryptographically invalid ones are fine — the validator never checks them at these heights) at heights ≤ `finalized_height`.
2. The validator silently skips them on every consensus round.
3. The purger never removes them from the unvalidated pool.
4. Memory grows without bound until the replica OOMs and crashes.

Even on a healthy subnet, the attacker can maintain a steady flood faster than `PurgeUnvalidatedBelow` fires. On a stalled subnet (slow execution, state sync in progress), the accumulation is indefinite.

### Likelihood Explanation

- Attacker only needs to be a P2P peer — a single Byzantine node below the fault threshold suffices.
- No cryptographic material is needed; invalid proofs accumulate just as well as valid ones since the validator never reaches the signature-check code for these heights.
- The stalled-subnet precondition amplifies the attack but is not strictly required; even under normal operation the window between purges allows significant accumulation.

### Recommendation

In `validate_equivocation_proofs`, explicitly emit `RemoveFromUnvalidated` for any proof whose height is ≤ `finalized_height`:

```rust
if proof.height <= finalized_height {
    return Some(ChangeAction::RemoveFromUnvalidated(proof.into_message()));
}
```

Alternatively, add a `PurgeUnvalidatedOfTypeBelow(EquivocationProof, finalized_height.increment())` action in `purge_equivocation_proofs_by_finalized_height` alongside the existing validated-pool purge.

### Proof of Concept

State-machine test:
1. Advance the pool to finalized height H.
2. Freeze `expected_batch_height` (simulate stall).
3. Insert 10,000 `EquivocationProof` artifacts at heights 1..H into the unvalidated pool via the P2P path.
4. Run `validate_equivocation_proofs` and `purger.on_state_change` in a loop.
5. Assert: unvalidated pool equivocation proof count remains 10,000 (none removed).
6. Assert: memory usage grows linearly with insertion count.

The root cause is confirmed at:
- [4](#0-3)  — range excludes `finalized_height` and below
- [5](#0-4)  — only validated pool is purged for equivocation proofs
- [3](#0-2)  — unvalidated pool purge gated on `expected_batch_height` advancing

    fn purge_equivocation_proofs_by_finalized_height(
        &self,
        pool: &PoolReader,
        finalized_height: Height,

        if pool
            .pool()
            .validated()
            .equivocation_proof()
            .height_range()
            .is_some()

            changeset.push(ChangeAction::PurgeValidatedOfTypeBelow(
                PurgeableArtifactType::EquivocationProof,
                finalized_height.increment(),

                "Purge validated equivocation proofs at and below {finalized_height:?}"
