### Title
Unauthenticated Caller Can Force SNS Swap Participation on Behalf of Any Principal via `refresh_buyer_tokens` - (File: rs/sns/swap/canister/canister.rs)

### Summary
The `refresh_buyer_tokens` update method on the SNS Swap canister accepts an arbitrary `buyer` principal string from any caller. When the `buyer` field is non-empty, the canister uses the attacker-supplied principal instead of the actual caller's identity. This allows any unprivileged ingress sender to trigger swap participation state changes (including ticket deletion) on behalf of any victim principal, without the victim's consent.

### Finding Description

The `refresh_buyer_tokens` canister endpoint in `rs/sns/swap/canister/canister.rs` resolves the buyer principal as follows:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← attacker-controlled
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The `buyer` field in `RefreshBuyerTokensRequest` is documented as "If not specified, the caller is used," but there is no authorization check preventing an arbitrary caller from specifying any other principal as the `buyer`. [2](#0-1) 

Inside `refresh_buyer_token_e8s`, the function:
1. Checks the ICP ledger balance of the **victim's** subaccount (not the caller's).
2. If the balance meets the minimum, creates or updates a `BuyerState` entry for the victim.
3. **Deletes the victim's open ticket** from `OPEN_TICKETS_MEMORY` if one exists and the balance satisfies it. [3](#0-2) 

The ticket deletion is irreversible and keyed on the victim's principal:

```rust
memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow_mut().remove(&principal));
``` [4](#0-3) 

Additionally, if the victim has already transferred ICP to their swap subaccount but has not yet called `refresh_buyer_tokens` themselves, an attacker can call it on their behalf — potentially with a mismatched `confirmation_text` that causes the call to fail, or with the correct text to force-register the victim's participation without their explicit consent.

### Impact Explanation

**Governance authorization bug / unauthorized state change on behalf of another user.**

1. **Ticket deletion griefing**: A victim creates a sale ticket via `new_sale_ticket` (which is caller-authenticated), transfers ICP to their swap subaccount, and then an attacker calls `refresh_buyer_tokens` with the victim's principal. If the balance satisfies the ticket amount, the ticket is deleted from `OPEN_TICKETS_MEMORY`. The victim's ticket is permanently destroyed without their action. This disrupts the ticketing payment flow.

2. **Forced participation registration**: An attacker can register swap participation for any principal that has ICP sitting in their swap subaccount, bypassing the victim's intent to participate on their own schedule or with their own confirmation text.

3. **Confirmation text bypass**: If the SNS requires a `confirmation_text`, the attacker must know it (it is public in the swap init parameters), so this is not a meaningful barrier. The attacker can supply the correct text and force-register the victim.

4. **Swap outcome manipulation**: By force-registering many principals who have ICP in their subaccounts, an attacker can artificially inflate the direct participation total, potentially pushing the swap to commit prematurely or affecting the Neurons' Fund matched funding calculation.

### Likelihood Explanation

- The `refresh_buyer_tokens` endpoint is a publicly callable `#[update]` method with no ingress authentication beyond the IC's standard message signing (any principal can call it).
- The `buyer` field is a plain string in the Candid interface, trivially set to any target principal's text representation.
- No on-chain rate limiting or caller-vs-buyer identity check exists in the canister handler.
- The attack requires only knowledge of a victim's principal ID (publicly observable on-chain) and that the victim has ICP in their swap subaccount (also observable via ledger queries).
- This is a low-effort, zero-cost attack (no ICP required from the attacker).

### Recommendation

Enforce that the resolved `buyer` principal equals the actual `caller()`. Remove the ability for callers to specify an arbitrary `buyer` field, or add an explicit check:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, requested, "Caller may only refresh their own buyer tokens");
        requested
    };
    ...
}
```

Alternatively, remove the `buyer` field entirely and always use `caller_principal_id()`, matching the mitigation pattern applied to the analogous `Vault.sponsor` bug in the referenced report.

### Proof of Concept

**Attacker-controlled entry path**: Any ingress sender → `refresh_buyer_tokens` update call on the SNS Swap canister with `buyer = victim_principal.to_string()`.

**Scenario — ticket deletion griefing**:

1. Victim calls `new_sale_ticket(amount_icp_e8s: X)` → ticket stored in `OPEN_TICKETS_MEMORY[victim]`.
2. Victim transfers `X` ICP to `swap_canister_subaccount(victim)` on the ICP ledger.
3. Attacker (any principal) calls:
   ```
   refresh_buyer_tokens({ buyer: "<victim_principal>", confirmation_text: <public_text_or_None> })
   ```
4. Inside `refresh_buyer_token_e8s`:
   - Ledger balance of victim's subaccount = X ≥ `min_participant_icp_e8s` → passes minimum check.
   - Ticket found: `amount_ticket = X ≤ requested_increment_e8s = X` → ticket deleted.
   - `BuyerState` for victim set to X e8s.
5. Victim's ticket is permanently deleted. Victim's participation is registered without their explicit action.
6. Victim can no longer use the ticketing flow for this swap round. [1](#0-0) [5](#0-4) [6](#0-5)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
