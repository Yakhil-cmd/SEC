SNS Vesting Neuron Stake Can Be Increased via `ClaimOrRefresh` While `Disburse` and `Split` Are Blocked â€” (`File: rs/sns/governance/src/governance.rs`)

---

### Summary

In the SNS governance canister, vesting neurons block `Disburse` and `Split` operations but explicitly permit `ClaimOrRefresh`. Because `refresh_neuron` requires no caller authorization and does not check whether the target neuron is vesting, any principal can transfer SNS tokens to a vesting neuron's subaccount and call `ClaimOrRefresh` to permanently increase that neuron's cached stake. Those tokens are then locked for the entire remaining vesting period, during which neither `Disburse` nor `Split` can be called. This is the direct IC analog of the `increaseAmount()` gap reported in the veSatin audit.

---

### Finding Description

`check_command_is_valid_if_neuron_is_vesting` is the single gate that enforces which `manage_neuron` commands a vesting neuron may execute. [1](#0-0) 

The function short-circuits and returns `Ok(())` for every `ClaimOrRefresh` variant before any neuron lookup occurs: [2](#0-1) 

Later in the same match, `Disburse` and `Split` are explicitly rejected: [3](#0-2) 

`ClaimOrRefresh` with `By::NeuronId` routes to `refresh_neuron`. That function checks only whether the neuron is Neurons' Fund-controlled; it contains no vesting check and no caller-authorization check: [4](#0-3) 

The result: any principal can send SNS tokens to a vesting neuron's ledger subaccount and then call `ClaimOrRefresh` to fold those tokens into the neuron's `cached_neuron_stake_e8s`. Because `Disburse` and `Split` are blocked for the entire vesting period, those tokens cannot be recovered until vesting expires.

Vesting neurons are created at SNS genesis via `NeuronDistribution` with `vesting_period_seconds` set, which can be multiple years: [5](#0-4) 

---

### Impact Explanation

A principal who transfers SNS tokens to a vesting neuron's subaccount and calls `ClaimOrRefresh` loses control of those tokens for the remainder of the vesting period. The tokens are credited to the neuron's stake but cannot be disbursed or split. If the vesting period is long (e.g., 3â€“4 years, which is common for developer neurons), the effective lock-up is substantial. The neuron owner receives the benefit of the increased stake and eventually the tokens themselves; the sender has no recourse.

Additionally, a developer who intentionally tops up their own vesting neuron may not realize the additional tokens will be subject to the same vesting lock as the original stake, leading to unintended self-locking of funds.

---

### Likelihood Explanation

The attack surface is reachable by any unprivileged ingress sender. No special role or key is required. The `ClaimOrRefresh` endpoint is public, and sending tokens to a neuron subaccount is a standard ledger operation. The scenario where a developer tops up their own vesting neuron (and inadvertently locks additional tokens) is a realistic, low-friction path. The third-party griefing path (sending tokens to someone else's vesting neuron) is self-harmful to the attacker but still reachable. Likelihood is **medium** for the self-locking case and **low** for the third-party case.

---

### Recommendation

Add a vesting check inside `refresh_neuron` (or inside `check_command_is_valid_if_neuron_is_vesting` before the early return) that rejects stake refreshes on vesting neurons, consistent with how `refresh_neuron` already rejects Neurons' Fund-controlled neurons: [6](#0-5) 

Alternatively, remove the blanket `ClaimOrRefresh` exemption from `check_command_is_valid_if_neuron_is_vesting` and instead allow only the `By::MemoAndController` (claim) path while blocking the `By::NeuronId` (refresh/stake-increase) path for vesting neurons.

---

### Proof of Concept

1. An SNS is initialized with a developer neuron having `vesting_period_seconds = 3 years`.
2. An attacker (or the developer themselves) transfers `X` SNS tokens to the neuron's ledger subaccount (derived from the neuron ID).
3. The attacker calls `manage_neuron` with `Command::ClaimOrRefresh { by: By::NeuronId(Empty {}) }` targeting the vesting neuron.
4. `check_command_is_valid_if_neuron_is_vesting` returns `Ok(())` immediately at line 4856â€“4858 without inspecting the neuron.
5. `refresh_neuron` reads the ledger balance, finds `X` extra tokens, and increases `cached_neuron_stake_e8s` by `X`.
6. Any subsequent call to `manage_neuron` with `Command::Disburse` or `Command::Split` on the same neuron is rejected by `check_command_is_valid_if_neuron_is_vesting` at lines 4882â€“4883.
7. The `X` tokens are locked in the vesting neuron for the remaining vesting period with no withdrawal path available to the sender. [7](#0-6)

    /// Returns an error if the given neuron is vesting and the given command cannot be called by
    /// a vesting neuron
    fn check_command_is_valid_if_neuron_is_vesting(
        &self,
        neuron_id: &NeuronId,
        command: &manage_neuron::Command,
    ) -> Result<(), GovernanceError> {
        use manage_neuron::{Command::*, configure::Operation::*};

        // If this is a "claim" call, the neuron doesn't exist yet, so we return (because no checks
        // can be made). A "refresh" call can be made on a vesting neuron, so in this case also
        // results in returning Ok.
        if let ClaimOrRefresh(_) = command {
            return Ok(());
        }

        let neuron = self.get_neuron_result(neuron_id)?;

        if !neuron.is_vesting(self.env.now()) {
            return Ok(());
        }

        let err = |op: &str| -> Result<(), GovernanceError> {
            Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Neuron {neuron_id} is vesting and cannot call {op}"),
            ))
        };

        match command {
            Configure(configure) => match configure.operation {
                Some(IncreaseDissolveDelay(_)) => err("IncreaseDissolveDelay"),
                Some(StartDissolving(_)) => err("StartDissolving"),
                Some(StopDissolving(_)) => err("StopDissolving"),
                Some(SetDissolveTimestamp(_)) => err("SetDissolveTimestamp"),
                Some(ChangeAutoStakeMaturity(_)) => Ok(()),
                None => Ok(()),
            },
            Disburse(_) => err("Disburse"),
            Split(_) => err("Split"),
            Follow(_)
            | SetFollowing(_)
            | MakeProposal(_)
            | RegisterVote(_)
            | ClaimOrRefresh(_)
            | MergeMaturity(_)
            | DisburseMaturity(_)
            | AddNeuronPermissions(_)
            | RemoveNeuronPermissions(_)
            | StakeMaturity(_) => Ok(()),
        }
    }
