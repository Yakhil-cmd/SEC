### Title
Canister Sandbox Can Crash Replica via `ftruncate` on Shared Memory Files Without Seccomp Filtering - (File: `ic-os/guestos/docs/SELinux-Policy.adoc`, `ic-os/components/guestos/selinux/ic-node/ic-node.te`, `rs/replicated_state/src/page_map/page_allocator/mmap.rs`)

---

### Summary

The canister sandbox process holds write-capable file descriptors to shared memory backing files (`ic_canister_mem_t` and `tmpfs_t`). Because no Seccomp-BPF filter is applied to the sandbox, the `ftruncate64` syscall is unrestricted. A malicious canister can cause the sandbox process to call `ftruncate` to shrink these files while the replica has them mapped `MAP_SHARED`. Any subsequent replica access to a now-truncated page delivers `SIGBUS`, crashing the replica process. This is explicitly acknowledged as an unresolved security goal violation in the IC's own SELinux policy documentation.

---

### Finding Description

The IC canister sandbox relies exclusively on SELinux for access control. The SELinux policy for `ic_canister_sandbox_t` grants `{ map read write getattr }` on both `tmpfs_t` (shared memory area) and `ic_canister_mem_t` (heap delta backing files): [1](#0-0) 

The replica's page allocator (`MmapBasedPageAllocatorCore`) maps these same files with `MAP_SHARED | PROT_READ | PROT_WRITE`: [2](#0-1) 

The `truncate_file` helper wraps `ftruncate64` and is used by the allocator to grow files: [3](#0-2) 

Because no Seccomp filter is installed in the sandbox process, the sandbox can call `ftruncate64` to *shrink* any of the file descriptors it holds. When the replica subsequently accesses a page in its `MAP_SHARED` mapping that now falls beyond the truncated file size, the kernel delivers `SIGBUS`, terminating the replica process.

The IC's own SELinux policy documentation explicitly records this as an open security goal violation: [4](#0-3) 

The documentation also notes that SELinux cannot prevent `ftruncate` because the permission model has no `ftruncate` object class, and no remedy has been implemented: [5](#0-4) 

---

### Impact Explanation

A malicious canister can crash the replica process on any node that executes it. Crashing the replica halts that node's consensus participation. If the same canister is scheduled across multiple nodes simultaneously (e.g., during a subnet-wide round), it could crash multiple replicas, potentially dropping a subnet below its fault-tolerance threshold and halting finalization. Even a single-node crash is a node-level denial of service triggered by an unprivileged canister developer with no special access.

---

### Likelihood Explanation

The attack requires only deploying a canister and triggering execution. The canister sandbox process automatically receives the relevant file descriptors from the replica. No kernel exploit, no privileged access, and no threshold corruption is needed. The `ftruncate64` syscall is a standard POSIX call available to any process that holds a writable file descriptor. The only barrier is that the attacker must cause the sandbox to invoke `ftruncate64` with a shrinking offset — achievable through a Wasm-level exploit of the sandbox runtime or through any future code-execution primitive within the sandbox. The IC documentation itself rates this as a known, unmitigated issue.

---

### Recommendation

1. Install a Seccomp-BPF allowlist in the sandbox process at startup (before any untrusted code runs) that explicitly denies `ftruncate` / `ftruncate64` syscalls.
2. As a complementary mitigation, apply `F_SEAL_SHRINK` via `fcntl(F_ADD_SEALS)` on `memfd`-backed files before passing file descriptors to the sandbox, preventing any process from shrinking the file.
3. Alternatively, restructure the architecture so the replica does not hold concurrent `MAP_SHARED` mappings of files that the sandbox can write to, as noted in the existing remedies discussion.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Canister developer deploys a canister to any subnet.
2. The replica spawns a sandbox process (`ic_canister_sandbox_t`) and passes it file descriptors for the canister's heap delta backing files (`ic_canister_mem_t`) and shared memory area (`tmpfs_t`).
3. The sandbox process — with no Seccomp filter installed — calls `ftruncate64(fd, 0)` on one of the received writable file descriptors, shrinking the backing file to zero bytes.
4. The replica's `MmapBasedPageAllocatorCore`, which holds a live `MAP_SHARED` mapping of the same file, accesses a page at any offset > 0.
5. The kernel delivers `SIGBUS` to the replica process, which has no handler for it, causing immediate process termination.

**Relevant code path confirming the replica holds a concurrent MAP_SHARED mapping:** [6](#0-5) 

**SELinux policy confirming the sandbox holds a writable fd to the same file type with no ftruncate denial:** [7](#0-6)
