### Title
Unbounded `list_known_neurons` Query Without Pagination Enables DoS in NNS Governance - (File: `rs/nns/governance/src/neuron_store.rs`)

---

### Summary

The NNS governance canister exposes a `list_known_neurons` query that returns all registered known neurons in a single unbounded response. The known-neuron registry grows monotonically as users register neurons via `RegisterKnownNeuron` governance proposals, with no removal mechanism. As the list grows, the query will eventually exhaust the IC per-call instruction limit, causing a permanent DoS on this endpoint for every caller.

---

### Finding Description

The `list_known_neurons` query is declared in the NNS governance DID with no arguments and therefore no pagination: [1](#0-0) 

The underlying implementation in the neuron store calls `list_known_neuron_ids()`, which returns a `Vec<NeuronId>` of **every** registered known neuron with no upper bound: [2](#0-1) 

Known neurons are neurons that have been registered with a public name and description via the `RegisterKnownNeuron` proposal type. Any neuron holder can submit such a proposal. Once accepted, the neuron is permanently added to the known-neuron index ‚Äî there is no `DeregisterKnownNeuron` proposal type or any other removal path. The index therefore grows monotonically over the lifetime of the NNS.

The canister-level handler (accessible to any unprivileged query caller) invokes this unbounded collection traversal on every call: [3](#0-2) 

Compare this with the paginated `list_neurons` path, which caps results at `MAX_LIST_NEURONS_RESULTS` (500) and supports `start_page_at` / `page_size` parameters: [4](#0-3) 

`list_known_neurons` has no equivalent guard.

---

### Impact Explanation

Once the known-neuron list grows large enough to exhaust the IC query instruction limit (~5 billion instructions), **every** call to `list_known_neurons` will fail with an instruction-limit-exceeded trap. Because the list never shrinks, this becomes a **permanent** DoS on the endpoint ‚Äî no caller (governance UI, tooling, or protocol participant) can retrieve the known-neuron list without a canister upgrade that adds pagination. Governance front-ends that rely on this query to display the neuron directory would be permanently broken.

---

### Likelihood Explanation

The attack surface is open to any NNS neuron holder who can submit `RegisterKnownNeuron` proposals. The NNS has been live since 2021 and the known-neuron list accumulates continuously. While the current count is manageable, there is no architectural cap preventing the list from reaching a size that triggers the instruction limit. A motivated actor could accelerate growth by submitting many `RegisterKnownNeuron` proposals (each requires only a staked neuron and the proposal fee, which is burned on rejection but not on acceptance).

---

### Recommendation

1. **Add pagination** to `list_known_neurons` analogous to `list_neurons`: accept `start_page_at: opt NeuronId` and `limit: nat32` parameters, and cap results at a safe maximum (e.g., 100‚Äì500 per call).
2. **Alternatively**, introduce a hard cap on the total number of known neurons enforced at registration time.
3. **Consider** adding a `DeregisterKnownNeuron` proposal type so stale entries can be pruned.

---

### Proof of Concept

1. Obtain multiple NNS neurons (or coordinate with other neuron holders).
2. Submit `RegisterKnownNeuron` proposals for each neuron; proposals are accepted by majority vote.
3. Repeat until the known-neuron index is large enough that iterating it in a single query call exceeds the IC instruction limit.
4. Call `list_known_neurons` ‚Äî every invocation now traps with an instruction-limit error, permanently breaking the endpoint for all callers.
