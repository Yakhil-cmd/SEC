### Title
Silent ICP Burn Failure in Cycles Minting Canister Allows Ledger Conservation Violation - (File: rs/nns/cmc/src/main.rs)

### Summary

The `burn_and_log` function in the Cycles Minting Canister (CMC) intentionally swallows burn failures after cycles have already been deposited. When the ICP burn fails, the CMC returns a success response to the caller, marks the notification as permanently processed, and leaves the ICP unburned in its subaccount — violating ICP ledger conservation.

### Finding Description

In `rs/nns/cmc/src/main.rs`, the functions `process_top_up`, `process_mint_cycles`, and `process_create_canister` follow a two-step pattern:

1. Deposit cycles to the target canister (or mint to cycles ledger).
2. Call `burn_and_log` to destroy the corresponding ICP from the CMC's subaccount.

The `burn_and_log` function is explicitly designed to never propagate errors:

### Impact Explanation

When `burn_and_log` fails (e.g., due to a transient ledger error, ledger canister unavailability, or the CMC's subaccount being in an unexpected state):

- The caller's cycles are already deposited — they received value.
- The ICP is **not burned** — it remains in the CMC's subaccount keyed by `Subaccount::from(&canister_id)` or `Subaccount::from(&caller())`.
- The notification block is permanently marked as `NotifiedTopUp(Ok(...))` — no retry is possible.
- The orphaned ICP in the CMC's subaccount cannot be reclaimed by the user (notification is closed) and will not be burned automatically.

This is a **ledger conservation violation**: cycles are minted against ICP that was never destroyed. Repeated occurrences inflate the effective ICP supply relative to the cycles minted, breaking the ICP↔cycles economic invariant. If a governance upgrade later adds a withdrawal path from CMC subaccounts, accumulated unburned ICP could be extracted.

### Likelihood Explanation

The ICP ledger `send_pb` call can fail transiently if the ledger canister is temporarily unavailable (e.g., during an upgrade, under heavy load, or due to a subnet issue). Any unprivileged user who calls `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` at such a moment triggers this path. The CMC is a high-value, frequently-called canister, making transient ledger errors a realistic scenario over its operational lifetime.

### Recommendation

**Short term:** After `burn_and_log` fails, do not mark the notification as permanently successful. Instead, record a "burn pending" state and schedule a retry of the burn. Alternatively, if the burn fails, attempt to refund the ICP to the user and mark the notification as refunded.

**Long term:** Separate the "cycles deposited" and "ICP burned" states in `blocks_notified` so that a failed burn can be retried independently without risking double-minting. Consider making `burn_and_log` return a `Result` and handle the failure path explicitly, consistent with how `refund_icp` failures are handled (they propagate as `NotifyError::Other` and allow retry).

### Proof of Concept

1. Alice sends 10 ICP to the CMC's subaccount for canister `C` with memo `MEMO_TOP_UP_CANISTER` at ledger block `B`.
2. Alice calls `notify_top_up(block_index=B, canister_id=C)`.
3. CMC calls `deposit_cycles(C, cycles, ...)` — succeeds, canister `C` receives cycles.
4. CMC calls `burn_and_log(sub, 10 ICP)` — the ledger call fails transiently (e.g., ledger is upgrading).
5. `burn_and_log` logs the error and returns normally (no error propagated).
6. CMC records `blocks_notified[B] = NotifiedTopUp(Ok(cycles))`.
7. Alice's canister `C` has the cycles. The 10 ICP remains unburned in the CMC's subaccount. The notification is permanently closed — no retry, no refund.
8. The ICP supply is inflated by 10 ICP relative to the cycles minted. [6](#0-5) [7](#0-6)

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
