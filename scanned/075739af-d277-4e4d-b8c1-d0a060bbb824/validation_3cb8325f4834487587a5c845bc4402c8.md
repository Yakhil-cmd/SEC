### Title
ICRC-21 Consent Message Displays Caller-Supplied `expected_allowance` as "Existing Allowance" Without On-Chain Verification - (File: `packages/icrc-ledger-types/src/icrc21/lib.rs`)

---

### Summary
The `icrc21_canister_call_consent_message` endpoint on both the ICP ledger and ICRC-1 ledger generates a consent message for `icrc2_approve` that displays the caller-supplied `expected_allowance` field as **"Existing allowance"** — implying it is the current on-chain state — without ever querying the actual current allowance from ledger state. A malicious dApp can supply any arbitrary `expected_allowance` value to mislead the user about their current allowance before manipulating them into approving a different, larger allowance in a follow-up call.

---

### Finding Description

In `build_icrc21_consent_info` (`packages/icrc-ledger-types/src/icrc21/lib.rs`), the `Icrc21Function::Approve` branch decodes `ApproveArgs` from the caller-supplied `consent_msg_request.arg` and passes `expected_allowance` directly into the display builder:

```rust
if let Some(expected_allowance) = expected_allowance {
    display_message_builder =
        display_message_builder.with_expected_allowance(expected_allowance);
}
``` [1](#0-0) 

This value is then rendered in the consent message as **"Existing allowance: X — Until approval, this allowance remains in effect."** The label strongly implies X is the current on-chain allowance. However, X is entirely caller-controlled and is never cross-checked against the ledger's actual stored allowance.

By contrast, the fee IS validated against the ledger's internal state via `icrc21_check_fee`:

```rust
icrc21_check_fee(&fee, &ledger_fee)?;
``` [2](#0-1) 

No equivalent check exists for `expected_allowance`. The callers — both the ICP ledger and the ICRC-1 ledger — pass the request straight through without querying the actual current allowance: [3](#0-2)
