### Title
`ManageNervousSystemParameters` Updates `voting_rewards_parameters` Without First Settling Pending Rewards - (`File: rs/sns/governance/src/governance.rs`)

### Summary
The SNS governance canister's `perform_manage_nervous_system_parameters` function updates `VotingRewardsParameters` (including `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, `reward_rate_transition_duration_seconds`, and `round_duration_seconds`) without first calling `distribute_rewards` to settle pending reward calculations. Any elapsed-but-undistributed reward rounds will subsequently be calculated using the new rate parameters instead of the old ones, causing inaccurate maturity accrual for SNS neuron holders.

### Finding Description

`perform_manage_nervous_system_parameters` in the SNS governance canister directly overwrites `self.proto.parameters` with the new `NervousSystemParameters` (which embeds `VotingRewardsParameters`) without first triggering a reward distribution:

```rust
// rs/sns/governance/src/governance.rs:2581-2617
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            self.proto.parameters = Some(new_params);  // <-- rate updated here
            Ok(())
        }
        ...
    }
}

### Impact Explanation

**Impact: Medium**

When a `ManageNervousSystemParameters` proposal changes `voting_rewards_parameters`, all reward rounds that have elapsed since the last `RewardEvent` but have not yet been distributed will be retroactively calculated using the new rate. Concretely:

- If `initial_reward_rate_basis_points` is **decreased** (e.g., from 200 to 100 bps), neuron holders who voted during the pending period receive **less** maturity than they earned under the old rate.
- If `initial_reward_rate_basis_points` is **increased**, neuron holders receive **more** maturity than they earned.
- Changing `round_duration_seconds` also alters `new_rounds_count` (line 5812–5814), compounding the distortion by changing how many rounds are counted for the pending period.

This is a direct financial accounting error affecting all SNS neuron holders who voted during the pending reward window. The magnitude is bounded by at most one reward round's worth of rewards (typically one day), but can be larger if reward distribution was delayed.

### Likelihood Explanation

**Likelihood: Medium**

Any SNS community can legitimately pass a `ManageNervousSystemParameters` proposal to adjust reward rates — this is a standard governance action. The bug fires silently every time such a proposal is executed while there are pending (elapsed but undistributed) reward rounds. Since `distribute_rewards` runs on a periodic timer and proposals can be executed at any point in the reward cycle, there is almost always a non-zero pending window. The SNS proto comment acknowledges the parameter "probably should not be changed," but provides no enforcement or automatic settlement.

### Recommendation

In `perform_manage_nervous_system_parameters`, call `distribute_rewards` (or an equivalent reward-settling function) **before** overwriting `self.proto.parameters` when `voting_rewards_parameters` is being changed. This ensures all pending reward rounds are settled at the old rate before the new rate takes effect:

```rust
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            // If voting_rewards_parameters is changing, settle pending rewards first.
            if new_params.voting_rewards_parameters
                != self.nervous_system_parameters_or_panic().voting_rewards_parameters
            {
                if let Some(supply) = /* fetch current supply */ {
                    self.distribute_rewards(supply);
                }
            }
            self.proto.parameters = Some(new_params);
            Ok(())
        }
        ...
    }
}
```

### Proof of Concept

1. An SNS is configured with `initial_reward_rate_basis_points = 200` (2%) and `round_duration_seconds = 86400` (1 day).
2. Neuron holders vote on proposals during day 1. The reward round elapses but `distribute_rewards` has not yet been called (e.g., the periodic task fires slightly after the proposal execution).
3. An SNS governance proposal (`ManageNervousSystemParameters`) to change `initial_reward_rate_basis_points` to `100` (1%) is executed via `perform_manage_nervous_system_parameters`.
4. `self.proto.parameters` is updated immediately with the new rate.
5. When `distribute_rewards` is next called (by the periodic task), it reads `voting_rewards_parameters` from `self.nervous_system_parameters_or_panic()` — now returning the new 1% rate — and calculates the reward purse for day 1 at 1% instead of 2%.
6. All neuron holders who voted during day 1 receive half the maturity they were entitled to under the rate in effect when they voted.

The entry path is fully unprivileged from the perspective of the bug trigger: any SNS token holder with sufficient voting power can submit and pass a `ManageNervousSystemParameters` proposal, which is a standard, non-privileged governance action on the IC. [7](#0-6) [8](#0-7) [3](#0-2)

    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }

    pub fn reward_rate_at(&self, now: Instant) -> RewardRate {
        let reward_rate_transition_duration_seconds = self
            .reward_rate_transition_duration_seconds
            .expect("reward_rate_transition_duration_seconds unset");

        let time_since_genesis = {
            let result = now - *GENESIS;
            // For the purposes of determining reward rate, treat times before
            // genesis the same as at genesis. This is not expected to occur in
            // practice. This code is just being extra defensive.
            if result.as_secs() < i2d(0) {
                Duration { days: i2d(0) }
            } else {
                result
            }
        };
        if reward_rate_transition_duration_seconds == 0
            || time_since_genesis.as_secs() >= i2d(reward_rate_transition_duration_seconds)
        {
            return self.final_reward_rate();
        }

        // s linearly varies from 1 -> 0 as seconds_since_genesis varies from 0
        // to reward_rate_transition_duration_seconds.
        let transition = LinearMap::new(
            dec!(0)..i2d(reward_rate_transition_duration_seconds),
            dec!(1)..dec!(0),
        );
        let s = transition.apply(time_since_genesis.as_secs());
        // s2 varies quadratically from 1 -> 0 (again, as seconds_since_genesis
        // varies from 0 to reward_rate_transition_duration_seconds), and
        // flattens out as seconds_since_genesis approaches
        // reward_rate_transition_duration_seconds.
        let s2 = s * s;

        // This looks backwards, but we think of variable rate as being added to
        // final growth rate, not initial, and the amount to add is up to
        // initial - final (where initial is thought of as being greater than
        // final).
        let dr = self.initial_reward_rate() - self.final_reward_rate();
        // variable_reward_rate varies from dr to 0 as round varies from
        // 1 to transition_round_count.
        let variable_reward_rate = s2 * dr;

        self.final_reward_rate() + variable_reward_rate

pub struct VotingRewardsParameters {
    /// The amount of time between reward events.
    ///
    /// Must be > 0.
    ///
    /// During such periods, proposals enter the ReadyToSettle state. Once the round is over, voting
    /// for those proposals entitle voters to voting rewards. Such rewards are calculated in
    /// the governance canister's run_periodic_tasks function.
    ///
    /// This is a nominal amount. That is, the actual time between reward
    /// calculations and distribution cannot be guaranteed to be perfectly
    /// periodic, but actual inter-reward periods are generally expected to be
    /// within a few seconds of this.
    ///
    /// This supersedes super.reward_distribution_period_seconds.
    #[prost(uint64, optional, tag = "1")]
    pub round_duration_seconds: ::core::option::Option<u64>,
    /// The amount of time that the growth rate changes (presumably, decreases)
    /// from the initial growth rate to the final growth rate. (See the two
    /// *_reward_rate_basis_points fields bellow.) The transition is quadratic, and
    /// levels out at the end of the growth rate transition period.
    #[prost(uint64, optional, tag = "3")]
    pub reward_rate_transition_duration_seconds: ::core::option::Option<u64>,
    /// The amount of rewards is proportional to token_supply * current_rate. In
    /// turn, current_rate is somewhere between `initial_reward_rate_basis_points`
    /// and `final_reward_rate_basis_points`. In the first reward period, it is the
    /// initial growth rate, and after the growth rate transition period has elapsed,
    /// the growth rate becomes the final growth rate, and remains at that value for
    /// the rest of time. The transition between the initial and final growth rates is
    /// quadratic, and levels out at the end of the growth rate transition period.
    ///
    /// (A basis point is one in ten thousand.)
    #[prost(uint64, optional, tag = "4")]
    pub initial_reward_rate_basis_points: ::core::option::Option<u64>,
    #[prost(uint64, optional, tag = "5")]
    pub final_reward_rate_basis_points: ::core::option::Option<u64>,
}
