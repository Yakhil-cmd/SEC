### Title
Cycles Accepted But Not Used When `try_fetch_tx` Returns `NotEnoughCycles` - (File: `rs/bitcoin/checker/src/fetch.rs`)

---

### Summary

In the BTC Checker canister, `try_fetch_tx` calls `cycles_accept(cycle_cost)` before confirming that enough cycles are available. When the available cycles are less than `cycle_cost`, the IC system API accepts **all remaining available cycles** (up to `cycle_cost`) and returns the accepted amount. Since that amount is less than `cycle_cost`, the function returns `TryFetchResult::NotEnoughCycles` — but the already-accepted cycles are permanently transferred to the canister's balance without being used for any HTTP outcall and without being refunded to the caller.

---

### Finding Description

In `rs/bitcoin/checker/src/fetch.rs`, the `try_fetch_tx` method of the `FetchEnv` trait contains:

```rust
let cycle_cost = get_tx_cycle_cost(max_response_bytes, num_subnet_nodes);
if self.cycles_accept(cycle_cost) < cycle_cost {
    TryFetchResult::NotEnoughCycles
} else {
    TryFetchResult::ToFetch(self.fetch_tx(guard, provider, txid, max_response_bytes))
}
``` [1](#0-0) 

The `cycles_accept` implementation calls `ic_cdk::api::msg_cycles_accept(cycles)`: [2](#0-1) 

Per the IC protocol, `msg_cycles_accept(max_amount)` moves `min(max_amount, available)` cycles from the incoming message into the canister's balance. If `available = X` and `cycle_cost = Y` where `X < Y`, then `X` cycles are **permanently accepted** into the canister balance, the function returns `X`, the condition `X < Y` is true, and `NotEnoughCycles` is returned — but those `X` cycles are now gone from the caller and are not used for any HTTP outcall.

This is distinct from the IC's automatic refund of **unaccepted** cycles: once `msg_cycles_accept` is called, the accepted portion is irrevocably transferred to the canister. There is no mechanism to return them to the caller after the fact.

The vulnerable path is reached from `check_fetched`, which calls `try_fetch_tx` in a loop for each input transaction of the checked transaction: [3](#0-2) 

The guard in `check_transaction_with` only ensures enough cycles for the **first** HTTP outcall (the main transaction fetch), not for all subsequent input-transaction fetches: [4](#0-3) 

---

### Impact Explanation

A caller who sends exactly `CHECK_TRANSACTION_CYCLES_REQUIRED` cycles for a transaction with many inputs (e.g., a Bitcoin transaction with 5+ inputs) will have their remaining cycles silently consumed by the canister mid-processing when cycles run out during the input-fetching loop. The caller receives `NotEnoughCycles` and is told to retry, but the cycles accepted in the failing `try_fetch_tx` call are permanently lost — they are neither used for an HTTP outcall nor refunded. This is a direct cycles/resource accounting loss for any caller of `check_transaction` or `check_transaction_str`.

---

### Likelihood Explanation

The ckBTC minter calls `check_transaction` as part of its deposit flow. Any Bitcoin transaction with more inputs than `CHECK_TRANSACTION_CYCLES_REQUIRED` covers will trigger this path. A caller (including the ckBTC minter itself) sending the documented minimum cycles for a multi-input transaction will reliably lose cycles on each retry attempt. The entry path is an unprivileged `#[ic_cdk::update]` call reachable by any canister or user. [5](#0-4) 

---

### Recommendation

Check cycle availability **before** accepting, rather than inferring insufficiency from the accepted amount:

```rust
let cycle_cost = get_tx_cycle_cost(max_response_bytes, num_subnet_nodes);
// Check availability before accepting to avoid consuming cycles without use
if ic_cdk::api::msg_cycles_available() < cycle_cost {
    return TryFetchResult::NotEnoughCycles;
}
self.cycles_accept(cycle_cost);
TryFetchResult::ToFetch(self.fetch_tx(guard, provider, txid, max_response_bytes))
```

This requires adding a `cycles_available` method to the `FetchEnv` trait alongside `cycles_accept` so the check is testable. [6](#0-5) 

---

### Proof of Concept

1. Deploy the BTC Checker canister in `Normal` mode.
2. Submit a `check_transaction` call for a Bitcoin transaction that has 5 inputs, attaching exactly `CHECK_TRANSACTION_CYCLES_REQUIRED` cycles.
3. The canister fetches the main transaction (first `try_fetch_tx` call, cycles accepted and used).
4. `check_fetched` iterates over the 5 inputs. For each input, `try_fetch_tx` accepts `cycle_cost` cycles and makes an HTTP outcall.
5. When cycles run out mid-loop (e.g., on the 4th input), `cycles_accept(cycle_cost)` accepts all remaining cycles (say `R < cycle_cost`) and returns `NotEnoughCycles`.
6. The caller receives `NotEnoughCycles` but has lost `R` cycles that were accepted without any HTTP outcall being made.
7. On retry with fresh cycles, the same `R` cycles are lost again if the same boundary is hit. [7](#0-6)
