### Title
Arithmetic Overflow in `spawn_neuron()` Causes Canister Trap, Blocking Maturity Withdrawal - (File: `rs/nns/governance/src/governance.rs`)

---

### Summary

The NNS governance `spawn_neuron()` function and the SNS governance `disburse_maturity()` function both perform an unchecked-in-effect `u64 × u64` multiplication using `checked_mul(...).expect(...)`. When `maturity_e8s_equivalent` is large enough that the product overflows `u64`, the `.expect()` call panics, trapping the canister and permanently blocking the neuron owner from spawning/disbursing their maturity (the IC analog of "withdrawing a large deposit").

---

### Finding Description

In `rs/nns/governance/src/governance.rs`, `spawn_neuron()` computes the amount of maturity to move to the child neuron as:

```rust
let maturity_to_spawn = parent_neuron
    .maturity_e8s_equivalent
    .checked_mul(percentage as u64)
    .expect("Overflow while processing maturity to spawn.");
``` [1](#0-0) 

`maturity_e8s_equivalent` is a `u64`. `percentage` is a caller-supplied `u32` (validated to be 1–100) cast to `u64`. The product `maturity_e8s_equivalent × percentage` is computed as a plain `u64 × u64` multiplication. `checked_mul` returns `None` on overflow, and `.expect()` converts that `None` into a **panic**, which on the Internet Computer causes the canister to **trap** the entire update call.

The identical pattern exists in SNS governance's `disburse_maturity()`:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)
    .expect("Overflow while processing maturity to disburse.")
    .checked_div(100)
    ...
``` [2](#0-1) 

By contrast, `stake_maturity_of_neuron` in both NNS and SNS governance correctly uses `saturating_mul`, which never panics:

```rust
let mut maturity_to_stake =
    (neuron_maturity_e8s_equivalent.saturating_mul(percentage_to_stake as u64)) / 100;
``` [3](#0-2) 

The inconsistency between `saturating_mul` (safe) in `stake_maturity` and `checked_mul(...).expect(...)` (panics) in `spawn_neuron`/`disburse_maturity` is the root cause.

---

### Impact Explanation

If a neuron's `maturity_e8s_equivalent` exceeds `u64::MAX / percentage` (≈ `1.84 × 10¹⁷` e8s for `percentage = 100`), any call to `spawn_neuron` or `disburse_maturity` will trap the governance canister update call. The neuron owner is permanently unable to convert their maturity into ICP (the primary "withdrawal" mechanism for maturity). The neuron's maturity is effectively frozen. This is a **ledger conservation bug**: tokens that should be redeemable become permanently locked.

---

### Likelihood Explanation

For NNS ICP: the total ICP supply is approximately `5 × 10¹⁶` e8s, which is below the overflow threshold of `1.84 × 10¹⁷` e8s. This makes the condition practically unreachable on the NNS today.

For SNS governance: SNS tokens can have arbitrary total supplies and decimal configurations. An SNS token with a large total supply and aggressive maturity accrual could produce a neuron whose `maturity_e8s_equivalent` exceeds `u64::MAX / 100`. The likelihood is **low but non-zero** for SNS deployments with large token supplies, and the code pattern is a latent defect that could be triggered as the ecosystem grows.

---

### Recommendation

Replace `checked_mul(...).expect(...)` with intermediate widening to `u128` before the multiplication, matching the pattern already used in `calculate_split_neuron_effect`:

```rust
// Safe: two u64s multiplied as u128 cannot overflow u128
let maturity_to_spawn = ((parent_neuron.maturity_e8s_equivalent as u128)
    * (percentage as u128)
    / 100) as u64;
``` [4](#0-3) 

Apply the same fix to `disburse_maturity` in SNS governance. Align `spawn_neuron` and `disburse_maturity` with the `saturating_mul`-based approach already used in `stake_maturity_of_neuron`.

---

### Proof of Concept

1. Create an NNS or SNS neuron and artificially set `maturity_e8s_equivalent` to a value greater than `u64::MAX / 100 = 184_467_440_737_095_516` (achievable in a test environment).
2. Call `spawn_neuron` (NNS) or `disburse_maturity` (SNS) with `percentage_to_spawn = 100` / `percentage_to_disburse = 100`.
3. Observe the canister traps with the message `"Overflow while processing maturity to spawn."` / `"Overflow while processing maturity to disburse."`.
4. The neuron owner can no longer withdraw their maturity regardless of how many times they retry. [1](#0-0) [2](#0-1)
