Looking at the exact code paths for this question:

### Title
Unbounded DKG Unvalidated Pool Growth via Byzantine Peer Flooding with Phantom Remote DKG Dealings — (`rs/consensus/dkg/src/lib.rs`)

### Summary

A Byzantine subnet node can flood the DKG unvalidated pool with dealings referencing fabricated remote DKG IDs. Because `validate_dealings_for_dealer` intentionally defers (returns `Mutations::new()`) rather than rejects such dealings, and because the DKG pool has no size limit and the P2P slot table for DKG is configured with `SLOT_TABLE_NO_LIMIT`, these artifacts accumulate unbounded in memory until the next DKG interval purge. Every consensus round, `on_state_change` iterates over the entire unvalidated pool, creating O(N) processing overhead that grows with the flood.

---

### Finding Description

**Root cause 1 — Defer instead of reject for phantom remote DKG IDs:**

In `validate_dealings_for_dealer`, when a dealing's `NiDkgId` has `target_subnet.is_remote() == true` but no matching config exists, the function silently defers: [1](#0-0) 

This is intentional for legitimate remote DKG dealings that arrive before the corresponding `SetupInitialDKG` context appears in certified state. However, it creates no distinction between a legitimately early dealing and a fabricated dealing with a random `NiDkgTargetId` that will never have a matching config.

**Root cause 2 — No size limit on the DKG unvalidated pool:**

`DkgPoolImpl::insert` performs no bounds check: [2](#0-1) 

There is no per-peer quota, no total message count limit, and no byte-size limit on the unvalidated section, unlike the ingress pool which has `ingress_pool_max_count` and `ingress_pool_max_bytes`.

**Root cause 3 — DKG P2P channel uses `SLOT_TABLE_NO_LIMIT`:**

The P2P consensus manager's per-peer slot table, which is the first line of defense against artifact flooding, is configured with `usize::MAX` for DKG: [3](#0-2) 

Compare this to ingress, which uses `SLOT_TABLE_LIMIT_INGRESS = 50_000`: [4](#0-3) 

The slot table in `ConsensusManagerReceiver` enforces a per-peer limit only when `peer_slot_table_len < self.slot_limit`: [5](#0-4) 

With `slot_limit = usize::MAX`, this check never triggers for DKG.

**Root cause 4 — Purging is epoch-gated, not size-gated:**

Purging only fires when the consensus summary block advances past the pool's current start height: [6](#0-5) 

The purge removes only messages whose `id.height < new_height`: [7](#0-6) 

Deferred remote dealings at the current start height are not purged until the next DKG interval (typically ~500 blocks, ~8 minutes).

**Root cause 5 — O(N) processing overhead per round:**

Every call to `on_state_change` iterates the entire unvalidated pool and groups all dealings by `(dealer, DKG ID)`: [8](#0-7) 

With N deferred phantom dealings, this is O(N) work per consensus round, degrading throughput proportionally.

---

### Impact Explanation

A single Byzantine subnet node can:
1. Generate N dealings with distinct `NiDkgId { target_subnet: Remote(random_id_i), start_block_height: current_height }` values — each has a unique hash and thus a unique `DkgMessageId`.
2. Sign them with its own node key (valid TLS peer, valid signature).
3. Advertise them via P2P using distinct slot numbers (no slot limit for DKG).
4. Each dealing passes the `DkgBouncer` (height == current start height → `Wants`).
5. Each is inserted into the unvalidated pool without rejection.
6. Each is deferred indefinitely by `validate_dealings_for_dealer`.
7. Pool memory grows at the rate of (dealing size × N) until the next DKG interval.
8. Every consensus round incurs O(N) iteration overhead.

Impact: replica memory exhaustion; consensus round processing slowdown; potential subnet halt if multiple replicas are affected simultaneously.

---

### Likelihood Explanation

- **Attacker model**: A single Byzantine subnet node — within the standard BFT fault assumption (f < n/3). No threshold corruption required.
- **Entry point**: Standard P2P artifact gossip, authenticated only by TLS node identity.
- **Effort**: Low. The attacker only needs to generate many `NiDkgId` values with random `NiDkgTargetId` bytes and sign them. No cryptographic breaking required.
- **Window**: One full DKG interval (~500 blocks) per attack cycle. The attack can be repeated continuously across intervals.
- **Detectability**: The `on_state_change_processed` metric would spike, but no alert is wired to pool size for DKG.

---

### Recommendation

1. **Reject phantom remote DKG dealings at admission**: In `validate_dealings_for_dealer`, check whether the `dealer_subnet` field of the `NiDkgId` matches the local subnet ID. Dealings from the local subnet for a remote target that has no matching config should be rejected (not deferred), since the local node would know if it initiated such a request.

2. **Apply a slot table limit to DKG**: Change `SLOT_TABLE_NO_LIMIT` to a bounded constant for DKG (e.g., matching the expected maximum number of concurrent DKG dealings per subnet node per interval, which is small — at most a handful of configs × 1 dealing per dealer).

3. **Add a pool size cap**: Enforce a maximum unvalidated pool size in `DkgPoolImpl::insert`, evicting oldest entries or dropping new ones when the cap is reached.

4. **Rate-limit deferred remote dealings per peer**: Track how many deferred remote dealings have been received from each peer and drop new ones beyond a threshold.

        self.current_start_height = height;

        let unvalidated_keys: Vec<_> = self
            .unvalidated
            .keys()
            .filter(|id| id.height < height)
            .cloned()
            .collect();
        for id in unvalidated_keys {
            self.unvalidated.remove(&id);
        }

        let validated_keys: Vec<_> = self
            .validated
            .keys()
            .filter(|id| id.height < height)
            .cloned()
            .collect();
        for hash in &validated_keys {
            self.validated.remove(hash);
        }
        validated_keys
    }
