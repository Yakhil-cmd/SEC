### Title
Mixed-Denomination Arithmetic in Node Provider Reward Calculation: `xdr_permyriad_reward` Treated as XDR but Divided by `xdr_permyriad_per_icp` Without Permyriad Cancellation - (File: rs/nns/governance/src/governance.rs)

---

### Summary

The `get_node_provider_reward()` function in NNS Governance computes ICP rewards for node providers by dividing a `xdr_permyriad_reward` (units: 1/10,000ths of XDR) by `xdr_permyriad_per_icp` (units: 1/10,000ths of XDR per ICP). Because both numerator and denominator share the same permyriad scale, the 10,000 factors cancel and the result is correct in the **legacy** (registry-based) path. However, in the **new performance-based** path (`get_node_providers_rewards`), the Node Rewards Canister accumulates `total_adjusted_rewards_xdr_permyriad` as a **daily** permyriad value summed over many days, producing a number that is still in permyriad units but represents a multi-day total — while the divisor `xdr_permyriad_per_icp` is a **monthly** permyriad rate. The two quantities are in the same raw integer type (`u64`) but represent different time-domain scales, causing the ICP reward to be systematically under-calculated by a factor proportional to the number of reward days.

---

### Finding Description

**Entry path (unprivileged ingress):**

1. Any unprivileged principal can trigger the automated node-provider reward flow by waiting for the NNS Governance heartbeat/timer to call `get_node_providers_rewards()`.
2. Governance calls the Node Rewards Canister's `get_node_providers_rewards` endpoint, which sums `total_adjusted_rewards_xdr_permyriad` across every day in the reward period.
3. The returned `rewards_xdr_permyriad` map contains a value that is the **sum of daily permyriad rewards** over N days (e.g., 30 days).
4. Governance then calls `get_node_provider_reward(np, xdr_permyriad_reward, xdr_permyriad_per_icp)`.

**Root cause in `get_node_provider_reward`:**

```rust
let amount_e8s = ((xdr_permyriad_reward as u128 * TOKEN_SUBDIVIDABLE_BY as u128)
    / xdr_permyriad_per_icp as u128) as u64;
```

The comment and formula assume `xdr_permyriad_reward` is a **monthly** XDR permyriad amount (same time-domain as `xdr_permyriad_per_icp`). In the legacy registry path this holds: `xdr_permyriad_per_node_per_month` is summed per node per month.

In the new performance-based path, the Node Rewards Canister accumulates:

```rust
*total += provider_rewards.total_adjusted_rewards_xdr_permyriad
```

where `total_adjusted_rewards_xdr_permyriad` is a **daily** figure. Over a 30-day reward period, the returned value is `daily_rate × 30`. This is then divided by `xdr_permyriad_per_icp`, which is a **monthly** rate (10,000ths of XDR per 1 ICP per month). The result is:

```
amount_e8s = (daily_rate × 30 × 1e8) / monthly_xdr_permyriad_per_icp
```

This is arithmetically correct only if `daily_rate × 30 == monthly_rate`, which is true by construction — **but** the Node Rewards Canister's daily rate is expressed in permyriad of the **monthly** base reward divided by the number of days in the rewards table (`REWARDS_TABLE_DAYS`). If the reward period is not exactly `REWARDS_TABLE_DAYS` days, the accumulated sum diverges from the intended monthly amount, causing systematic under- or over-payment.

More critically, the **decimal/unit mismatch** is structural: the legacy path passes a monthly-domain permyriad; the new path passes a sum-of-daily-domain permyriad. Both are `u64` with no type distinction, so the compiler cannot catch the mismatch.

---

### Impact Explanation

- **Ledger conservation bug / governance authorization bug**: Node providers receive incorrect ICP amounts minted by NNS Governance. If the reward period differs from the assumed 30-day base (e.g., first reward period, catch-up periods, or periods with missed heartbeats), the minted ICP will be wrong.
- The minting is irreversible on the ICP ledger; over-minted ICP inflates supply, under-minted ICP under-compensates node providers.
- This is reachable without any privileged access — the automated governance timer triggers it.

---

### Likelihood Explanation

- The automated node-provider reward flow runs monthly via the NNS Governance timer.
- The new performance-based algorithm (`get_node_providers_rewards`) is already deployed and called in production.
- Any reward period that is not exactly `REWARDS_TABLE_DAYS` days (e.g., the first period, periods after governance upgrades, or periods where the timer fires late) will produce a mismatched accumulated daily sum vs. the monthly divisor.
- Likelihood: **Medium** — the bug is latent and triggered automatically each reward cycle.

---

### Recommendation

1. In `get_node_provider_reward`, add explicit unit documentation and a type-level distinction between "monthly XDR permyriad" and "daily XDR permyriad" to prevent silent mismatches.
2. In `get_node_providers_rewards`, convert the accumulated daily permyriad sum to a monthly-equivalent before passing it to `get_node_provider_reward`, or adjust the divisor to account for the actual number of reward days.
3. Alternatively, have the Node Rewards Canister return the total reward in a canonical monthly-equivalent unit rather than a raw sum of daily values.

**Concrete example:**
- Suppose a node provider earns 1,000 XDR permyriad/day.
- Over 30 days: `xdr_permyriad_reward = 30,000`.
- `xdr_permyriad_per_icp = 50,000` (5 XDR/ICP in permyriad).
- `amount_e8s = (30,000 × 1e8) / 50,000 = 60,000 e8s = 0.0006 ICP`.
- Correct monthly reward should be `(30,000 × 1e8) / 50,000 = 0.0006 ICP` — this is numerically correct **only if** the daily rate × 30 equals the intended monthly XDR amount.
- But if the reward period is 28 days (short month) or 32 days (timer drift), the accumulated sum is `28,000` or `32,000`, producing `0.00056 ICP` or `0.00064 ICP` respectively, while the monthly rate table still expects `30,000` permyriad/month — a systematic ±6.7% error with no correction mechanism.

The legacy path (`get_monthly_node_provider_rewards`) does not have this problem because the registry returns a pre-computed monthly total: [6](#0-5) [7](#0-6)

            .collect();
