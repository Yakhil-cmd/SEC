### Title
Silent ICP Burn Failure After Cycles Minting Allows ICP Supply Inflation - (File: `rs/nns/cmc/src/main.rs`)

### Summary
The Cycles Minting Canister (CMC) calls `burn_and_log` to burn ICP after successfully minting cycles, but discards the result with `.await;`. If the ICP burn fails, cycles are already deposited to the recipient while the ICP remains unburned in the CMC's subaccount, violating the ICP/cycles conservation invariant.

### Finding Description
In `rs/nns/cmc/src/main.rs`, two async functions — `process_mint_cycles` and `process_top_up` — follow the same pattern: they first perform the cycles-side operation (depositing cycles to the cycles ledger or topping up a canister), and only on success call `burn_and_log` to burn the corresponding ICP. The call is made as a fire-and-forget statement:

```rust
burn_and_log(sub, amount).await;
```

The trailing semicolon discards whatever `burn_and_log` returns. If the ICP ledger call inside `burn_and_log` fails (e.g., due to a transient inter-canister call rejection, ledger trap, or cycles exhaustion on the CMC), the function logs the error internally but the caller (`process_mint_cycles` / `process_top_up`) proceeds to return `Ok(...)` to the ingress sender as if everything succeeded. ### Impact Explanation
The ICP/cycles exchange is supposed to be conservation-preserving: ICP sent to the CMC must be burned when cycles are minted. If `burn_and_log` silently fails:

1. The recipient already holds the newly minted cycles (the deposit to the cycles ledger or the canister top-up succeeded).
2. The ICP remains in the CMC's per-user subaccount — it is neither burned nor refunded.
3. The CMC records the block index as processed (deduplication), so the same block cannot be re-notified.
4. The net effect is that cycles exist without a corresponding ICP burn, inflating the effective ICP supply and breaking the ledger conservation invariant that underpins the ICP/cycles exchange rate.

This is a **ledger conservation bug**: the total ICP supply is larger than it should be by the amount that failed to burn, and the discrepancy is permanent (the ICP is locked in the CMC subaccount with no recovery path).

### Likelihood Explanation
The ICP ledger is a well-tested canister and outright failures are rare under normal conditions. However, transient inter-canister call rejections are a normal part of IC operation (e.g., the ledger canister is stopped for upgrade, the CMC runs out of cycles, or a DTS slice limit is hit during a busy period). Because the burn is the *last* step and its failure is silently swallowed, any such transient event permanently breaks conservation for that transaction. An unprivileged user who times a `notify_top_up` or `notify_mint_cycles` call during a ledger upgrade window can reliably trigger this condition.

### Recommendation
Propagate the result of `burn_and_log` and treat a burn failure as a critical error. Options:

1. **Retry the burn**: persist the "pending burn" state and retry in a timer task, similar to how the CMC already retries other operations.
2. **Revert the cycles deposit**: if the burn cannot be completed, attempt to claw back the deposited cycles (though this is harder given the cycles ledger semantics).
3. **At minimum, alert and halt**: if the burn fails, do not return `Ok` to the caller; return an error so the caller knows the operation is in an inconsistent state and the operator is alerted.

The analog fix in the ERC-20 world is using `SafeERC20` to ensure the transfer reverts on failure; the IC analog is propagating the `Result` from `burn_and_log` and not silently discarding it.

### Proof of Concept
1. User sends 10 ICP to the CMC's subaccount for canister `C` with `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up` for canister `C`.
3. CMC calls `deposit_cycles(C, cycles, ...)` — succeeds; canister `C` now has more cycles.
4. CMC calls `burn_and_log(sub, amount).await;` — the ICP ledger is momentarily unavailable (e.g., mid-upgrade); the call returns an error.
5. `burn_and_log` logs the error and returns `()`.
6. `process_top_up` returns `Ok(cycles)` to the user.
7. Result: canister `C` received cycles, but the 10 ICP was never burned and sits permanently in the CMC's subaccount. The ICP supply is inflated by 10 ICP. [3](#0-2) [4](#0-3)

async fn process_mint_cycles(
    to_account: Account,
    amount: Tokens,
    deposit_memo: Option<Vec<u8>>,
    from: AccountIdentifier,
    sub: Subaccount,
) -> NotifyMintCyclesResult {
    let cycles = tokens_to_cycles(amount)?;
    match do_mint_cycles(to_account, cycles, deposit_memo).await {
        Ok(deposit_result) => {
            burn_and_log(sub, amount).await;
            Ok(NotifyMintCyclesSuccess {
                block_index: deposit_result.block_index,
                minted: cycles.into(),
                balance: deposit_result.balance,
            })
        }
        Err(err) => {
            let refund_block = refund_icp(sub, from, amount, MINT_CYCLES_REFUND_FEE).await?;
            Err(NotifyError::Refunded {
                reason: err,
                block_index: refund_block,
            })
        }
    }
}

async fn process_top_up(
    canister_id: CanisterId,
    from: AccountIdentifier,
    amount: Tokens,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<Cycles, NotifyError> {
    let cycles = tokens_to_cycles(amount)?;

    let sub = Subaccount::from(&canister_id);

    print(format!(
        "Topping up canister {canister_id} by {cycles} cycles."
    ));

    match deposit_cycles(canister_id, cycles, true, limiter_to_use).await {
        Ok(()) => {
            burn_and_log(sub, amount).await;
            Ok(cycles)
        }
        Err(err) => {
            let refund_block = refund_icp(sub, from, amount, TOP_UP_CANISTER_REFUND_FEE).await?;
            Err(NotifyError::Refunded {
                reason: err.to_string(),
                block_index: refund_block,
            })
        }
    }
}
