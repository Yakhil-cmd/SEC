### Title
Unbounded Per-Peer Slot Table Growth via `SLOT_TABLE_NO_LIMIT` Allows Byzantine Node to OOM Honest Replicas — (`rs/p2p/consensus_manager/src/receiver.rs`)

### Summary
A single Byzantine subnet member can exhaust memory on honest replicas by sending slot updates with monotonically increasing unique `SlotNumber` values. The consensus artifact channel is configured with `SLOT_TABLE_NO_LIMIT = usize::MAX`, and the per-peer guard `peer_slot_table_len < self.slot_limit` is effectively disabled, allowing unbounded slot table and task growth.

### Finding Description

The guard in `handle_slot_update_receive` is: [1](#0-0) 

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => {
    empty_slot.insert(new_slot_entry);
    ...
    (true, None)
}
Entry::Vacant(_) => {
    // drop with warning
    (false, None)
}
```

When `slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice (a `usize` HashMap can never reach `usize::MAX` entries before OOM). This is the configured value for consensus artifacts: [2](#0-1) 

Both branches of the consensus path (with and without `HASHES_IN_BLOCKS_ENABLED`) pass `SLOT_TABLE_NO_LIMIT`.

The slot table structure is `HashMap<NodeId, HashMap<SlotNumber, SlotEntry>>`: [3](#0-2) 

Each unique `SlotNumber` from a peer creates a new inner map entry. Critically, when `to_add = true`, the code also inserts into `active_assembles` and spawns a new `process_slot_update` task: [4](#0-3) 

Each spawned task holds a `watch::Receiver<PeerCounter>` and awaits `peer_rx.wait_for(|p| p.is_empty())` — which the Byzantine node never triggers by sending a deletion. Tasks accumulate indefinitely.

The topology-based pruning in `handle_topology_update` only removes nodes that **leave** the topology: [5](#0-4) 

A Byzantine node that remains in the subnet topology is never pruned.

The inbound channel has capacity 100: [6](#0-5) 

This limits burst rate but not total volume — the event loop drains it continuously.

### Impact Explanation
A single Byzantine subnet member can:
1. Send N slot updates with unique `SlotNumber` values and unique artifact IDs
2. Each creates a new `SlotEntry` in the per-peer table, a new `active_assembles` entry, and a new live async task
3. Tasks never terminate (Byzantine node never sends deletions)
4. Memory grows without bound → OOM → replica crash → consensus stalls

This affects every honest replica the Byzantine node connects to. If multiple honest replicas crash, consensus finalization halts subnet-wide.

### Likelihood Explanation
The attacker is a legitimate subnet member (authenticated TLS), which is within the BFT fault model ("protocol peer behavior below the consensus fault threshold"). The attack requires no cryptographic break, no majority, and no privileged access — only membership in the subnet and the ability to send crafted P2P messages. The attack is deterministic and requires no timing or race conditions.

### Recommendation
Replace `SLOT_TABLE_NO_LIMIT` for consensus artifacts with a finite per-peer bound derived from the maximum number of in-flight consensus artifacts a legitimate node would ever advertise (e.g., bounded by the consensus pool's maximum artifact count per round). The existing `SLOT_TABLE_LIMIT_INGRESS` pattern used for ingress at line 257 demonstrates the correct approach: [7](#0-6) 

A similar finite constant should be applied to consensus, certification, DKG, and IDKG channels. Additionally, consider adding a cap on `active_assembles` / `artifact_processor_tasks` per peer.

### Proof of Concept
```rust
// Unit test: send 1_000_000 slot updates with unique SlotNumbers from NODE_1
// with slot_limit = usize::MAX (SLOT_TABLE_NO_LIMIT)
let (mut mgr, _) = ReceiverManagerBuilder::new()
    .with_slot_limit(usize::MAX)
    .build();
let cancellation = CancellationToken::new();
for i in 0..1_000_000u64 {
    mgr.handle_slot_update_receive(
        SlotUpdate {
            slot_number: SlotNumber::from(i),
            commit_id: CommitId::from(i),
            update: Update::Id(i),
        },
        NODE_1,
        ConnId::from(1),
        cancellation.clone(),
    );
}
// Assert: slot_table grows to 1_000_000 entries
// Assert: artifact_processor_tasks grows to 1_000_000 live tasks
// Observe: memory usage is unbounded; process OOMs on real hardware
assert_eq!(mgr.slot_table.get(&NODE_1).unwrap().len(), 1_000_000);
assert_eq!(mgr.artifact_processor_tasks.len(), 1_000_000);
```
