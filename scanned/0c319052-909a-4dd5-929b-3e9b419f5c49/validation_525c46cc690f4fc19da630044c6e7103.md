### Title
Predictable ChaCha20Rng Seed Enables UUID Prediction and Undisclosed Rule Metadata Enumeration — (`rs/boundary_node/rate_limits/canister/random.rs`)

---

### Summary

The `ChaCha20Rng` used to generate rule UUIDs is seeded with a value derived entirely from the publicly observable IC consensus block time. Any observer who monitors block timestamps can reconstruct the exact seed, predict all future rule UUIDs, and call `get_rule_by_id` as an unprivileged query caller to enumerate the existence and metadata of undisclosed rules.

---

### Finding Description

**Seed construction** in `random.rs`:

```rust
thread_local! {
  static RNG: RefCell<ChaCha20Rng> = {
    let mut seed = [42; 32];
    seed[..8].copy_from_slice(&time().to_le_bytes());
    RefCell::new(ChaCha20Rng::from_seed(seed))
  };
}
``` [1](#0-0) 

The 32-byte seed is: `time().to_le_bytes() ++ [42u8; 24]`. Only the first 8 bytes vary; the remaining 24 bytes are the constant `42`. `ic_cdk::api::time()` returns the IC consensus block time in nanoseconds, which is publicly observable on-chain.

The `thread_local!` static is initialized lazily on first access. In the IC Wasm environment, heap memory is reset on canister install/upgrade, so the RNG is re-seeded on the first call to `generate_random_uuid()` after each install/upgrade. That first call happens inside `add_config` when a new rule is introduced:

```rust
let rule_id = RuleId(generate_random_uuid()?);
``` [2](#0-1)

`getrandom` is wired to `custom_getrandom_bytes_impl`, which draws from the `RNG` thread-local: [4](#0-3) 

**Query access for unprivileged callers:**

`get_rule_by_id` is a `#[query]` method. The `inspect_message` hook only restricts ingress update messages; query calls bypass it entirely. The hook explicitly lists only `get_config`, `add_config`, and `disclose_rules`: [5](#0-4) [6](#0-5) 

So `get_rule_by_id` is reachable by any anonymous or unprivileged caller as a query: [7](#0-6) 

**What an unprivileged caller receives:**

For a `RestrictedRead` caller, `RuleGetter` redacts `rule_raw` and `description` for undisclosed rules, but still returns `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version`: [8](#0-7) 

The `NotFound` error is returned only when the UUID does not exist, so the attacker can distinguish "rule exists but undisclosed" from "rule does not exist."

---

### Impact Explanation

An attacker who:
1. Observes the IC block timestamp of the first `add_config` call after canister install/upgrade (publicly available via the IC state tree)
2. Reconstructs the seed: `seed[..8] = block_time_ns.to_le_bytes(); seed[8..] = [42u8; 24]`
3. Instantiates a local `ChaCha20Rng::from_seed(seed)` and replays UUID generation

…can predict every rule UUID generated in subsequent `add_config` calls and query `get_rule_by_id` to learn:
- Whether a confidential rule exists (existence oracle)
- Its `incident_id` (links the rule to a security incident before public disclosure)
- Its `added_in_version` and `removed_in_version`

This breaks the stated invariant that rule UUIDs are unguessable by unprivileged callers and that undisclosed rule metadata is confidential.

---

### Likelihood Explanation

- IC block times are publicly observable with nanosecond precision via certified state reads.
- The attacker does not need to brute-force anything; they observe the exact timestamp directly.
- The attack requires no privileged access, no key material, and no consensus corruption.
- The only prerequisite is monitoring the chain for the first `add_config` call after an install/upgrade, which is trivially detectable.

---

### Recommendation

Replace the time-seeded initialization with a cryptographically unpredictable source. On the IC, the correct approach is to use `ic_cdk::api::management_canister::main::raw_rand()` (an async call to the management canister that returns 32 bytes of threshold-BLS randomness) to seed the RNG during `init`/`post_upgrade`, storing the seed in stable memory or a canister-held variable. The `thread_local!` lazy initialization pattern with `time()` must be removed entirely.

---

        getter.get(&rule_id)
    })?;
    Ok(response)
}
