### Title
Suspended UTXO Cooldown Timestamp Unconditionally Overwritten on Every `update_balance` Call, Extending the Retry Delay - (File: `rs/bitcoin/ckbtc/minter/src/state.rs`)

---

### Summary

The ckBTC minter's `SuspendedUtxos::insert` unconditionally overwrites `last_time_checked_cache` with the current timestamp on every call, regardless of whether the UTXO's suspension state actually changed. Because `update_balance` re-evaluates suspended UTXOs and calls `suspend_utxo` (which calls `insert`) each time a suspended UTXO is still found to be suspended, the 24-hour cooldown window is reset on every `update_balance` invocation. A user who calls `update_balance` repeatedly while their UTXO is suspended will never reach the 24-hour threshold needed to re-evaluate the UTXO, effectively locking it in suspension indefinitely.

---

### Finding Description

The ckBTC minter implements a cooldown mechanism for suspended UTXOs (those that are too small in value or tainted). A suspended UTXO is only re-evaluated after 24 hours have elapsed since `last_time_checked`. The timestamp is stored in `SuspendedUtxos::last_time_checked_cache`, a `BTreeMap<Utxo, Timestamp>`.

The root cause is in `SuspendedUtxos::insert`:

```rust
pub fn insert(
    &mut self,
    account: Account,
    utxo: Utxo,
    reason: SuspendedReason,
    now: Option<Timestamp>,
) -> bool {
    if let Some(timestamp) = now {
        self.last_time_checked_cache.insert(utxo.clone(), timestamp); // always overwrites
    }
    if self.utxos.get(&account).and_then(|u| u.get(&utxo)) == Some(&reason) {
        return false; // early return: UTXO already suspended for same reason
    }
    ...
}
```

The timestamp is written **before** the early-return check. Even when the UTXO is already suspended for the same reason (the `return false` branch), the `last_time_checked_cache` has already been updated to `now`.

The call chain is:

1. User calls `update_balance` → `processable_utxos_for_account` checks elapsed time since `last_time_checked`.
2. If elapsed < 24 hours, the UTXO is returned as still-suspended (not re-evaluated).
3. After `processable_utxos_for_account`, the code in `update_balance.rs` processes each `processable_utxo`. For UTXOs that are still too small, `audit::ignore_utxo` → `discard_utxo` → `state.suspend_utxo` → `SuspendedUtxos::insert` is called with `now`.
4. But wait — suspended UTXOs that are **not yet eligible** for re-evaluation are returned in the `suspended_utxos` vec, not in `processable_utxos`. So they do not go through `ignore_utxo` again in that path.

However, the critical path is when the UTXO **is** eligible (elapsed ≥ 24 hours) and is re-evaluated but still found to be suspended (e.g., still too small, or still tainted). In that case, `ignore_utxo` or `quarantine_utxo` is called again with the current `now`, which calls `insert` with the new timestamp — resetting the 24-hour window. This is the intended behavior for re-evaluation.

The **actual analog bug** is more subtle and directly mirrors the report: the `last_time_checked_cache` timestamp is overwritten even when `insert` returns `false` (no state change). The test `should_be_nop_when_already_suspended_for_same_reason` explicitly documents and asserts this behavior:

```rust
assert!(!suspended_utxos.insert(ledger_account(), utxo(), reason, Some(NOW)));
// ...
assert_eq!(
    suspended_utxos.last_time_checked_cache,
    btreemap! {utxo() => NOW}  // timestamp was updated even though insert returned false
);
```

This means: if a user calls `update_balance` at time T (when the UTXO is already eligible for re-evaluation), the UTXO is re-evaluated, found still suspended, and `insert` is called with timestamp T. The `last_time_checked_cache` is updated to T. The user must now wait another 24 hours from T. If the user calls `update_balance` again at T+1 second (before the re-evaluation result changes), the same thing happens: the timestamp is reset to T+1. Each call to `update_balance` that triggers re-evaluation resets the 24-hour window, forcing the user to wait longer.

---

### Impact Explanation

**Impact: Medium**

A user with a suspended UTXO (e.g., a UTXO whose value is too small, or one that was tainted) who calls `update_balance` repeatedly after the 24-hour window opens will have their cooldown timer reset on each call. This means:

- The UTXO can never be finalized/re-evaluated if the user keeps calling `update_balance` more frequently than once per 24 hours.
- In practice, users who poll `update_balance` frequently (e.g., wallets, automated scripts) will find their suspended UTXOs permanently stuck.
- For `ValueTooSmall` UTXOs, this means the user's BTC deposit is permanently inaccessible through the normal flow.
- For `Quarantined` UTXOs, the re-evaluation that might clear them (if the BTC checker changes its verdict) is indefinitely deferred.

---

### Likelihood Explanation

**Likelihood: Medium**

Any user or wallet that polls `update_balance` more than once per 24 hours (which is common behavior for wallets checking deposit status) will trigger this. The `update_balance` endpoint is publicly callable by any IC principal with no rate limiting beyond the `AlreadyProcessing` guard (which only prevents concurrent calls, not sequential ones). The bug is triggered by normal, expected usage patterns.

---

### Recommendation

In `SuspendedUtxos::insert`, only update `last_time_checked_cache` when the UTXO's suspension state actually changes (i.e., when the function does not return `false`). Move the timestamp write after the early-return check:

```rust
pub fn insert(
    &mut self,
    account: Account,
    utxo: Utxo,
    reason: SuspendedReason,
    now: Option<Timestamp>,
) -> bool {
    if self.utxos.get(&account).and_then(|u| u.get(&utxo)) == Some(&reason) {
        return false; // no state change, do NOT update timestamp
    }
    if let Some(timestamp) = now {
        self.last_time_checked_cache.insert(utxo.clone(), timestamp);
    }
    self.utxos_without_account.remove(&utxo);
    let utxos = self.utxos.entry(account).or_default();
    utxos.insert(utxo, reason);
    true
}
```

Alternatively, only update the timestamp when the UTXO transitions to a new suspension state (new reason or newly suspended), not when it is re-confirmed in the same state.

### Proof of Concept

**Entry path:** Any unprivileged IC principal can call `update_balance` on the ckBTC minter canister.

**Steps:**

1. User deposits a BTC UTXO whose value is below `deposit_btc_min_amount`. The minter calls `audit::ignore_utxo` → `SuspendedUtxos::insert(account, utxo, ValueTooSmall, T0)`. The UTXO is suspended with `last_time_checked = T0`.

2. At time `T0 + 24h + 1s`, the UTXO becomes eligible for re-evaluation. User calls `update_balance`. `processable_utxos_for_account` sees elapsed ≥ 24h and returns the UTXO in `processable_utxos`.

3. In `update_balance`, the UTXO is still too small (value hasn't changed). `audit::ignore_utxo` is called again with `now = T0 + 24h + 1s`. This calls `SuspendedUtxos::insert(account, utxo, ValueTooSmall, T0 + 24h + 1s)`. The timestamp is reset to `T0 + 24h + 1s`.

4. User calls `update_balance` again at `T0 + 24h + 2s`. The elapsed time since `last_time_checked` is only 1 second (< 24h). The UTXO is returned as still-suspended. The user must wait another ~24 hours.

5. If the user calls `update_balance` every second after step 2, the cooldown window is reset each time, and the UTXO is never finalized.

    now: Timestamp,
    runtime: &R,
