### Title
Unbounded Loop Over All Buyers in `canister_post_upgrade` Can Permanently Block SNS Swap Canister Upgrades - (File: `rs/sns/swap/src/swap.rs`)

### Summary
The SNS Swap canister's `canister_post_upgrade` calls `rebuild_indexes()`, which iterates over every entry in `self.buyers` when `BUYERS_LIST_INDEX` is empty. For a large SNS sale with thousands of participants, this unbounded loop can exceed the IC's total instruction limit for `install_code`, causing every subsequent upgrade attempt to fail with `CanisterInstructionLimitExceeded` and permanently bricking the canister's upgrade path.

### Finding Description

In `rs/sns/swap/canister/canister.rs`, `canister_post_upgrade` unconditionally calls `rebuild_indexes()`: [1](#0-0) 

The `rebuild_indexes` implementation in `rs/sns/swap/src/swap.rs` loops over all keys in `self.buyers` whenever `BUYERS_LIST_INDEX` is empty: [2](#0-1) 

The triggering condition is:

```rust
if !self.buyers.is_empty() && buyers_list_index_is_empty {
    for key in self.buyers.keys() {
        if let Some(buyer_principal) = string_to_principal(key) {
            insert_buyer_into_buyers_list_index(buyer_principal).map_err(...)?;
        }
    }
}
```

`self.buyers` is a `BTreeMap<String, BuyerState>` that grows with every participant in the SNS token sale. Each `insert_buyer_into_buyers_list_index` call performs a stable BTreeMap insertion — an operation that is expensive in Wasm instructions due to stable memory I/O. For a popular SNS sale with thousands of buyers, the cumulative instruction cost of this loop can exceed the IC's `max_instructions_per_install_code` limit.

The IC execution environment does support Deterministic Time Slicing (DTS) for `canister_post_upgrade`, allowing the upgrade to span multiple rounds: [3](#0-2) 

However, DTS does not remove the *total* instruction limit for install_code — it only allows the work to be spread across rounds up to that ceiling. When the ceiling is exceeded, the upgrade fails with `CanisterInstructionLimitExceeded`: [4](#0-3) 

The critical self-reinforcing failure: when the upgrade fails, the canister rolls back to its pre-upgrade state. Because `BUYERS_LIST_INDEX` was never populated (the loop never completed), the next upgrade attempt will again find `buyers_list_index_is_empty == true` and attempt the same full loop — failing again. The canister is permanently unupgradeable.

### Impact Explanation

**Impact: High**

A permanently unupgradeable SNS Swap canister means:
- Critical security patches cannot be applied.
- Bug fixes cannot be deployed.
- The canister is frozen at a potentially vulnerable or broken version.
- The SNS project's governance over its own swap canister is effectively nullified.

This is a denial-of-service against the canister's upgrade mechanism, triggered by normal user participation in the sale.

### Likelihood Explanation

**Likelihood: Medium**

The condition triggers only when upgrading from a version that predates `BUYERS_LIST_INDEX` to a version that introduces it. However:
- The code comment explicitly acknowledges this migration path as the expected scenario: *"This most likely indicates that this canister was upgraded from a previous version where BUYERS_LIST_INDEX did not exist."*
- Popular SNS sales routinely attract thousands of participants (the `sale_buyer_count` metric is actively tracked).
- Once triggered, the failure is self-perpetuating — there is no recovery path without a special-purpose hotfix that bypasses the loop. [5](#0-4) 

### Recommendation

Replace the single-message full-scan with a paginated, resumable migration. Store a migration cursor in stable memory (e.g., the last processed principal). On each `post_upgrade`, process only a bounded batch (e.g., 1,000 buyers), then schedule a timer to continue. Mark migration complete when the cursor reaches the end. This is the same pattern the NNS Governance canister uses for neuron migration: [6](#0-5) 

Alternatively, pre-populate `BUYERS_LIST_INDEX` in `pre_upgrade` from the existing buyers map before the new wasm is installed, so `post_upgrade` finds the index already populated and skips the loop entirely.

### Proof of Concept

1. Deploy an SNS sale and accumulate a large number of buyers (e.g., 50,000 participants, achievable in a popular sale).
2. Upgrade the SNS Swap canister from a version without `BUYERS_LIST_INDEX` to the current version.
3. `canister_post_upgrade` calls `rebuild_indexes()`.
4. The loop iterates over all 50,000 `self.buyers` entries, each performing a stable BTreeMap insert.
5. The total instruction count exceeds `max_instructions_per_install_code`.
6. The upgrade fails with `CanisterInstructionLimitExceeded`; the canister rolls back.
7. `BUYERS_LIST_INDEX` remains empty. Every subsequent upgrade attempt repeats steps 3–6.
8. The canister is permanently unupgradeable.

The attacker-controlled entry path is simply participating in the SNS sale as a normal buyer — no privileged access required. Any sufficiently popular sale naturally accumulates enough buyers to trigger this condition.

    pub fn rebuild_indexes(&self) -> Result<(), String> {
        let buyers_list_index_is_empty =
            memory::BUYERS_LIST_INDEX.with(|bli| bli.borrow().is_empty());

        if !self.buyers.is_empty() && buyers_list_index_is_empty {
            log!(
                INFO,
                "Buyers state is populated but BUYERS_LIST_INDEX is not. This most likely indicates \
                that this canister was upgraded from a previous version where BUYERS_LIST_INDEX did not \
                exist. Conducting a best effort rebuild."
            );

            for key in self.buyers.keys() {
                // Try to parse the string representation of the Principal. Logging the error
                // occurs in `string_to_principal`.
                if let Some(buyer_principal) = string_to_principal(key) {
                    // If the index cannot be built due to limitations of the stable memory,
                    // return to the caller to determine how to handle the error.
                    insert_buyer_into_buyers_list_index(buyer_principal).map_err(|grow_failed| {
                        format!(
                            "Failed to add buyer {buyer_principal} to state, the canister's stable memory could not grow: {grow_failed}"
                        )
                    })?;
                }
            }
        }

        Ok(())
    }
