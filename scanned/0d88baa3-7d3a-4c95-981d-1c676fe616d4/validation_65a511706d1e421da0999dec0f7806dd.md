### Title
Predictable ChaCha20 RNG Seed Enables UUID Enumeration and Undisclosed Rule Metadata Leakage — (`rs/boundary_node/rate_limits/canister/random.rs`, `add_config.rs`, `getter.rs`)

---

### Summary

The rate-limit canister seeds its ChaCha20 RNG with only 8 bytes of entropy (IC time in nanoseconds), with the remaining 24 bytes hardcoded to `42`. Because IC time is publicly observable and the RNG is deterministic, any observer who knows the canister's initialization time can reconstruct the exact RNG state, predict every UUID that will ever be assigned as a rule ID, and then call the publicly accessible `get_rule_by_id` query to confirm rule existence and extract undisclosed rule metadata (`incident_id`, `added_in_version`, `removed_in_version`) before official disclosure.

---

### Finding Description

**Step 1 — Weak RNG seed** [1](#0-0) 

The seed is `[42u8; 32]` with only `seed[..8]` overwritten by `time().to_le_bytes()`. The remaining 24 bytes are the constant `42`. The full seed is therefore:

```
[T0, T1, T2, T3, T4, T5, T6, T7, 42, 42, 42, 42, 42, 42, 42, 42,
 42, 42, 42, 42, 42, 42, 42, 42, 42, 42, 42, 42, 42, 42, 42, 42]
```

where `T0..T7` are the little-endian bytes of `ic_cdk::api::time()` at canister init. The effective entropy is 8 bytes — and the value of those 8 bytes is publicly observable.

**Step 2 — All rule UUIDs flow from this RNG**

`generate_random_uuid()` calls `getrandom::getrandom()`, which on `wasm32-unknown-unknown` is routed through the registered custom handler `custom_getrandom_bytes_impl`, which drains bytes from the same `thread_local! RNG`. [2](#0-1) [3](#0-2) 

Every new rule ID is the next 16 bytes from this single, predictable stream.

**Step 3 — `get_rule_by_id` is a plain query, accessible to any caller** [4](#0-3) 

The `inspect_message` hook only intercepts **update** (ingress) calls. It explicitly rejects everything except `get_config`, `add_config`, and `disclose_rules` for update calls. But `get_rule_by_id` is a `#[query]` — query calls bypass `inspect_message` entirely on the IC. Any anonymous or unprivileged principal can call it. [5](#0-4) 

**Step 4 — The response for an undisclosed rule is `Ok`, not an error**

`RuleGetter::get()` returns `Err(NotFound)` only when the rule does not exist in storage. If the rule exists but is undisclosed, it returns `Ok(OutputRuleMetadata{...})` after applying the confidentiality formatter. [6](#0-5) 

The `RuleConfidentialityFormatter` only nulls out `rule_raw` and `description` when `disclosed_at.is_none()`. It does **not** suppress `rule_id`, `incident_id`, `added_in_version`, or `removed_in_version`. [7](#0-6) 

This is confirmed by the existing test, which asserts that an unauthorized caller receives a successful `Ok` response containing `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` for an undisclosed rule: [8](#0-7)

### Impact Explanation

An attacker who knows the canister's init time (publicly readable from IC certified state or canister history) can:

1. Reconstruct the exact ChaCha20 RNG state.
2. Pre-compute the full sequence of rule UUIDs that will be assigned.
3. Call `get_rule_by_id` with each predicted UUID.
4. Distinguish "rule exists but undisclosed" (`Ok` with nulled fields) from "rule does not exist" (`Err(NotFound)`).
5. Extract `incident_id` (linking the rule to a security incident), `added_in_version`, and `removed_in_version` for every undisclosed rule — before official disclosure.

This breaks the confidentiality invariant that undisclosed rule metadata must not be reachable by unprivileged callers.

---

### Likelihood Explanation

- The init time is a single 64-bit value with no additional secret. It is not guessed; it is read from public IC state.
- The attack requires zero privileges, no governance majority, no key material, and no network-level attack.
- It is fully local-testable in a pocket-ic or replica environment.
- The only precondition is knowing the canister's deployment timestamp, which is trivially observable.

---

### Recommendation

Replace the weak seeding strategy with a cryptographically unpredictable source. On the IC the correct approach is to call `ic_cdk::api::management_canister::main::raw_rand()` (which returns 32 bytes of certified randomness from the threshold BLS beacon) during `init`/`post_upgrade` and use those bytes as the seed. Until that async call completes, UUID generation should be deferred or blocked.

Additionally, `get_rule_by_id` should return `Err(NotFound)` (indistinguishable from a non-existent rule) for undisclosed rules when called by `RestrictedRead` callers, rather than returning a successful response with partial metadata.

use uuid::Uuid;

fn predict_rule_ids(init_time_ns: u64, count: usize) -> Vec<String> {
    let mut seed = [42u8; 32];
    seed[..8].copy_from_slice(&init_time_ns.to_le_bytes());
    let mut rng = ChaCha20Rng::from_seed(seed);
    (0..count).map(|_| {
        let mut buf = [0u8; 16];
        rng.fill_bytes(&mut buf);
        Uuid::from_slice(&buf).unwrap().to_string()
    }).collect()
}
