### Title
`SenderInfoContent` canister signature lacks request-context binding, enabling cross-canister replay of identity attestations - (File: `rs/types/types/src/messages/http.rs`)

---

### Summary

The `SenderInfoContent` type, which defines the signed payload for the `sender_info` field in IC ingress messages, only commits to the raw `info` bytes under the `"ic-sender-info"` domain separator. It does not bind to the target canister ID, the sender's principal, or the request expiry. A canister signature produced by an identity provider (e.g., Internet Identity) over `SenderInfoContent` is therefore valid for any request to any canister, enabling cross-canister replay of identity attestations.

---

### Finding Description

`SenderInfoContent` is defined in `rs/types/types/src/messages/http.rs`:

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);   // only the raw info blob
    }
}
``` [1](#0-0) 

The `SignatureDomain` implementation prepends only `"ic-sender-info"`:

```rust
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
``` [2](#0-1) 

The full signed bytes are therefore: `\x0Eic-sender-info || info_bytes`. No target canister ID, no sender principal, and no ingress expiry are included.

In `verify_sender_info_canister_sig`, the verifier constructs the signable content from only `sender_info.info`:

```rust
let sender_info_content = SenderInfoContent(&sender_info.info);
let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

verify_canister_sig_with_fallback!(
    validator,
    &canister_sig,
    &sender_info_content,
    &public_key,
    ...
);
``` [3](#0-2) 

The only binding check performed is that the canister ID encoded in `sender_pubkey` matches `sender_info.signer`:

```rust
if pubkey_canister_id != sender_info.signer {
    return Err(InvalidSenderInfo(...));
}
``` [4](#0-3) 

There is no check that the `sender_info` attestation was produced for this specific target canister, this specific sender, or within a bounded time window.

---

### Impact Explanation

The `sender_info` feature is designed for identity providers (e.g., Internet Identity) to attest to user attributes (e.g., `b"age_verified"`, `b"kyc_passed"`, or richer credential blobs) that the target canister reads via `ic0.msg_caller_info_data()` and `ic0.msg_caller_info_signer()`. Canisters that make authorization decisions based on these values assume the attestation was produced specifically for the current request context.

Because the canister signature over `SenderInfoContent` is not bound to the target canister ID or sender principal, a valid `(info, signer, sig)` triple obtained from one request to canister A can be replayed verbatim in a new request to canister B. The attacker:

1. Makes a legitimate request to canister A, obtaining a valid `sender_info` signed by II.
2. Constructs a new request to canister B, embedding the same `(info, signer, sig)`.
3. Signs the outer envelope with their own key (the `MessageId` covers `sender_info` content, so the outer signature is valid).
4. Canister B's call to `ic0.msg_caller_info_data()` returns the replayed `info` blob, and `ic0.msg_caller_info_signer()` returns the II canister ID — indistinguishable from a fresh attestation.

Any canister that gates privileged actions on `sender_info` content (e.g., "II attests this user is KYC-verified, allow withdrawal") is vulnerable to authorization bypass via replay.

Additionally, `sender_info` carries no expiry field of its own. Once obtained, a signature is valid indefinitely across all canisters on the IC.

---

### Likelihood Explanation

The attack requires only that an adversary obtain one valid `sender_info` signature from the target identity provider — achievable by making a single legitimate request to any canister that accepts `sender_info`. No privileged access, no key compromise, and no threshold corruption is required. The attacker controls the outer envelope and can direct the replayed `sender_info` to any target canister. This is reachable by any unprivileged ingress sender.

---

### Recommendation

Bind the `SenderInfoContent` signed bytes to the target canister ID and sender principal. Extend the struct to carry these fields and include them in `write_signed_bytes_without_domain_separator`:

```rust
pub struct SenderInfoContent<'a> {
    pub info: &'a [u8],
    pub canister_id: &'a CanisterId,
    pub sender: &'a UserId,
}

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.canister_id.get().as_slice());
        bytes.extend_from_slice(self.sender.get().as_slice());
        bytes.extend_from_slice(self.info);
    }
}
```

The `verify_sender_info_canister_sig` function in `rs/validator/src/ingress_validation.rs` must be updated to pass the target canister ID and sender from the request content when constructing `SenderInfoContent`. Identity providers must also update their signing logic accordingly.

Optionally, add an expiry field to `RawSignedSenderInfo` and enforce it during validation, analogous to delegation expiry.

---

### Proof of Concept

An unprivileged user can demonstrate the replay as follows:

1. Send a valid update call to canister A with `sender_info = {info: b"kyc_passed", signer: II_id, sig: <II_sig>}`. The II canister signs `\x0Eic-sender-info || b"kyc_passed"`.

2. Construct a second update call to canister B with the identical `sender_info` triple. Sign the outer envelope with the same user key (the `MessageId` will differ because `canister_id` differs, but `sender_info.sig` is reused verbatim).

3. Submit the second request. `verify_sender_info_canister_sig` in `rs/validator/src/ingress_validation.rs` will accept it: the canister ID in `sender_pubkey` matches `sender_info.signer`, and the canister signature over `SenderInfoContent(&b"kyc_passed")` verifies correctly.

4. Inside canister B, `ic0.msg_caller_info_data()` returns `b"kyc_passed"` and `ic0.msg_caller_info_signer()` returns `II_id` — the same values as in the original request to canister A, despite II never having attested to these attributes for canister B. [5](#0-4) [6](#0-5)

```rust
fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
