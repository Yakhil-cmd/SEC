### Title
Unbounded Body Buffering via `to_bytes(body, usize::MAX)` in `RosettaMetricsMiddleware` Enables Process OOM DoS — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

---

### Summary

The `RosettaMetricsMiddleware` unconditionally buffers the entire HTTP request body into heap memory with no size limit before forwarding the request to any route handler. An unprivileged HTTP client can exploit this to crash the Rosetta process via OOM, causing ledger sync interruption and API unavailability.

---

### Finding Description

In `extract_canister_id`, the body is consumed using `to_bytes(body, usize::MAX)`: [1](#0-0) 

`usize::MAX` is the explicit "no limit" sentinel for axum's `to_bytes`. The entire streaming body is collected into a single `Bytes` allocation before any route handler runs.

This is called unconditionally from `RosettaMetricsMiddleware::call` for every non-`/metrics` request: [2](#0-1) 

The middleware is applied as the innermost layer wrapping all routes in the ICRC1 Rosetta server: [3](#0-2) 

In Axum, `.layer()` calls wrap from inside out — `metrics_layer` is applied first (closest to routes), meaning it runs before any route handler. The outer layers are only `add_request_span()` and `RequestIdLayer`, neither of which impose a body size limit. A search across the entire `rs/rosetta-api/` tree for `RequestBodyLimitLayer`, `body_limit`, `max_body`, or any equivalent guard returns **zero matches** — there is no body size enforcement anywhere in the stack.

The server binds publicly: [4](#0-3) 

---

### Impact Explanation

Any unauthenticated HTTP client reachable to the Rosetta port can POST a multi-gigabyte body to any endpoint (e.g., `/account/balance`, `/construction/submit`). The process will attempt to allocate that many bytes contiguously on the heap. On a memory-constrained host this causes OOM termination of the Rosetta process, interrupting the ledger sync thread and making all Rosetta API endpoints unavailable until the process is restarted.

---

### Likelihood Explanation

The attack requires only a single HTTP connection and a streaming POST body — no authentication, no special knowledge, no privileged access. The endpoint is publicly reachable by design (Rosetta is an exchange-facing API). The bug is in a middleware that runs on every request, so every endpoint is equally affected.

---

### Recommendation

Replace `usize::MAX` with a reasonable maximum body size (e.g., 1 MB for a JSON Rosetta request) in `extract_canister_id`:

```rust
const MAX_BODY_SIZE: usize = 1 * 1024 * 1024; // 1 MB
let bytes = to_bytes(body, MAX_BODY_SIZE).await.map_err(|_| {
    // return 413 or propagate error
})?;
```

Alternatively, add Axum's `DefaultBodyLimit::max(1_048_576)` as an outer layer before `metrics_layer` so oversized requests are rejected with HTTP 413 before any buffering occurs.

---

### Proof of Concept

```bash
# Send a 512 MB body to any Rosetta endpoint
dd if=/dev/zero bs=1M count=512 | \
  curl -X POST http://<rosetta-host>:<port>/account/balance \
       -H "Content-Type: application/json" \
       --data-binary @- &

# Repeat a few times concurrently; monitor RSS of the rosetta process
watch -n1 'ps aux | grep rosetta'
# Process RSS climbs until OOM kill
```

The call chain is: `POST /account/balance` → `RosettaMetricsMiddleware::call` (line 304) → `extract_canister_id(body)` (line 336) → `to_bytes(body, usize::MAX)` (line 389) → heap exhaustion → SIGKILL. [5](#0-4)
