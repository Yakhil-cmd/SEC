### Title
Missing Minimum Cycles Output Validation in CMC ICP-to-Cycles Conversion - (File: rs/nns/cmc/src/main.rs)

### Summary
The Cycles Minting Canister (CMC) implements a two-step ICP-to-cycles conversion flow: (1) the user sends ICP to a CMC subaccount via the ledger, then (2) the user calls `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` to trigger the conversion. None of these notification endpoints accept a `min_cycles_expected` parameter, so the user has no way to enforce a minimum acceptable output at execution time. The conversion uses the ICP/XDR rate that is current at notification time, which may have changed significantly since the ICP was sent.

### Finding Description
The CMC's ICP-to-cycles conversion is split across two separate transactions. In step 1, the user sends ICP to a CMC-controlled subaccount via the ICP ledger. In step 2, the user calls a notification endpoint (`notify_top_up`, `notify_mint_cycles`, or `notify_create_canister`) which reads the current `icp_xdr_conversion_rate` from state and converts the ICP amount to cycles via `tokens_to_cycles`.

The `NotifyTopUp` struct contains only `block_index` and `canister_id`: [1](#0-0) 

The `NotifyMintCyclesArg` struct contains only `block_index`, `to_subaccount`, and `deposit_memo`: [2](#0-1) 

Neither struct, nor `NotifyCreateCanister`, includes a `min_cycles_expected` field. At execution time, `tokens_to_cycles` reads the live exchange rate from state with no floor check against any user-supplied expectation: [3](#0-2) 

This rate is then used directly in `process_top_up`, `process_mint_cycles`, and `process_create_canister` without any comparison to a user-specified minimum: [4](#0-3) [5](#0-4) 

The ICP/XDR rate is updated every five minutes from the Exchange Rate Canister (XRC): [6](#0-5) 

### Impact Explanation
A user who sends ICP expecting a certain number of cycles has no on-chain mechanism to abort the conversion if the rate has moved adversely between the ledger transfer and the notification call. The ICP is already in the CMC subaccount; the only outcome is conversion at whatever rate is current. If the ICP/XDR rate drops sharply (e.g., due to a volatile market move or a delayed/stale XRC update), the user receives materially fewer cycles than anticipated with no recourse. The burned ICP is unrecoverable. This is a **cycles/resource accounting bug** that causes direct, irreversible financial loss to the user.

### Likelihood Explanation
The ICP/XDR rate is refreshed every five minutes. Any non-trivial delay between the ledger transfer and the notification call — caused by network congestion, user latency, or deliberate timing — exposes the user to rate movement. The two-step flow is the only supported path for all three conversion endpoints (`notify_top_up`, `notify_mint_cycles`, `notify_create_canister`), so every user of the CMC is affected. No privileged access is required; any unprivileged ingress sender can trigger this path.

### Recommendation
Add an optional `min_cycles_expected: opt nat` field to `NotifyTopUp`, `NotifyMintCyclesArg`, and `NotifyCreateCanister`. In `tokens_to_cycles` (or in each `process_*` function), after computing the cycles amount, compare it against the caller-supplied minimum and return a `NotifyError::Refunded` (with a descriptive reason) if the computed amount falls below the threshold. This mirrors the fix applied in the Vaultka remediation: the user specifies the minimum acceptable output at request time, and the execution fails fast if the rate has moved beyond that bound.

### Proof of Concept
1. Observe the current ICP/XDR rate via `get_icp_xdr_conversion_rate` on the CMC — e.g., 50,000 XDR permyriad per ICP.
2. Send 1 ICP to the CMC top-up subaccount for a target canister via the ICP ledger, recording the `block_index`.
3. Before calling `notify_top_up`, wait for the XRC to push a new rate that is significantly lower (e.g., 25,000 XDR permyriad per ICP — a 50% drop).
4. Call `notify_top_up` with `{ block_index, canister_id }`. The CMC calls `tokens_to_cycles` using the new, lower rate and deposits only half the expected cycles. The call succeeds and the ICP is burned; the user has no way to prevent this outcome. [7](#0-6) [8](#0-7)

            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles {
                xdr_permyriad_per_icp,
                cycles_per_xdr: state.cycles_per_xdr,
            }
            .to_cycles(amount)),
            None => {
                let error_message =
                    "No conversion rate found in CMC, notification aborted".to_string();
                print(&error_message);
                Err(NotifyError::Other {
                    error_code: NotifyErrorCode::Internal as u64,
                    error_message,
                })
            }
        }
    })
