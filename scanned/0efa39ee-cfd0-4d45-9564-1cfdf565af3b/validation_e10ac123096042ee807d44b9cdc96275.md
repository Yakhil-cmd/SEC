### Title
NNS Governance `would_follow_ballots` Does Not Deduplicate Followees, Allowing Majority-Vote Manipulation - (File: rs/nns/governance/src/neuron/types.rs)

---

### Summary

The NNS governance `would_follow_ballots` function iterates over the raw `followees` list without deduplication. Because the NNS `Follow` command explicitly permits duplicate neuron IDs in the followees list, a neuron owner can inflate the effective weight of a chosen followee by repeating its ID, causing the majority threshold to be computed against a list length that includes duplicates. This violates the stated governance invariant that "once a majority of the followees votes to adopt or reject a proposal, the neuron votes the same way."

---

### Finding Description

`Neuron::would_follow_ballots` in `rs/nns/governance/src/neuron/types.rs` computes the following-based vote by iterating over `self.followees` and counting yes/no occurrences, then comparing against `followees.len()`:

```rust
let mut yes: usize = 0;
let mut no: usize = 0;
for f in followees.iter() {
    if let Some(f_vote) = ballots.get(&f.id) {
        if f_vote.vote == (Vote::Yes as i32) {
            yes = yes.saturating_add(1);
        } else if f_vote.vote == (Vote::No as i32) {
            no = no.saturating_add(1);
        }
    }
}
if yes.saturating_mul(2_usize) > followees.len() { return Vote::Yes; }
if no.saturating_mul(2_usize) >= followees.len() { return Vote::No; }
``` [1](#0-0) 

The `followees` vector is stored as-is from the `Follow` command — no deduplication is performed. The integration test `follow_same_neuron_multiple_times` explicitly confirms that the NNS governance canister accepts and stores duplicate neuron IDs in a followee list: [2](#0-1) 

A storage-level test also documents that duplicates are preserved with full multiplicity: [3](#0-2) 

**Concrete manipulation example:**

| Followees list | B votes Yes, C votes No | Result |
|---|---|---|
| `[B, C]` | yes=1, no=1, len=2 → `1×2=2 ≥ 2` | **Vote::No** |
| `[B, B, C]` | yes=2, no=1, len=3 → `2×2=4 > 3` | **Vote::Yes** |

By submitting `Follow { topic, followees: [B, B, C] }`, a neuron owner makes their neuron vote **Yes** even though the unique-followee majority voted **No**. The same trick works in reverse to force a **No** vote.

The manipulated vote then propagates through the cascade in `rs/nns/governance/src/voting.rs`, which calls `neuron_store.neuron_would_follow_ballots(...)` for every follower of the neuron that just voted: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

A neuron owner can cause their neuron to cast a following-based vote that contradicts the true majority of its unique followees. This violates the core governance invariant documented in the NNS proto and API:

> "Once a majority of the followees votes to adopt or reject a proposal belonging to the specified topic, the neuron votes the same way." [6](#0-5) 

If the manipulated neuron itself has significant voting power, or if many other neurons follow it (liquid-democracy cascade), the distorted vote propagates to all downstream followers, potentially flipping the outcome of a governance proposal. The neuron owner can simultaneously appear to follow a diverse set of neurons (for social/reputational purposes) while covertly giving one followee disproportionate weight — a form of governance deception that is not detectable from the stored followee list alone.

---

### Likelihood Explanation

The attack requires only a standard `manage_neuron` call with a `Follow` command containing duplicate neuron IDs. No privileged access, no threshold corruption, and no external dependency is needed. Any neuron owner can submit this call at any time. The NNS governance canister accepts and stores the duplicate list without error, as confirmed by the existing integration test.

---

### Recommendation

Deduplicate the followees list before computing the majority vote in `would_follow_ballots`:

```rust
let unique_followees: Vec<_> = followees.iter()
    .collect::<std::collections::BTreeSet<_>>()
    .into_iter()
    .collect();
// use unique_followees.len() as denominator and iterate over unique_followees
```

Alternatively, reject duplicate followee IDs in the `Follow` command handler before storing the list, analogous to the SNS governance `ValidatedFolloweesForTopic` validation that already returns `DuplicateFolloweeNeuronId`: [7](#0-6) 

---

### Proof of Concept

1. Neuron owner controls neuron **A**.
2. Owner submits `manage_neuron { Follow { topic: SomeGovernanceTopic, followees: [B, B, C] } }`.
3. NNS governance stores `followees = [B, B, C]` without error (confirmed by `follow_same_neuron_multiple_times` test).
4. A governance proposal is created. Neuron B votes **Yes**; neuron C votes **No**.
5. The cascade calls `would_follow_ballots` for neuron A:
   - `yes = 2` (B counted twice), `no = 1`, `len = 3`
   - `2 × 2 = 4 > 3` → neuron A votes **Yes**
6. Without duplicates (`[B, C]`): `yes=1, no=1, len=2` → `1×2=2 ≥ 2` → neuron A votes **No**.
7. All neurons that follow A inherit the manipulated **Yes** vote through the cascade.
