### Title
Global `MAX_PENDING = 100` Withdrawal Queue Exhaustion Blocks All `withdraw_eth` Calls - (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`)

### Summary

The ckETH minter enforces a global, canister-wide limit of 100 pending withdrawal requests (`MAX_PENDING = 100`). Any caller — including an attacker — who fills this queue causes every subsequent `withdraw_eth` call from any user to fail with `TemporarilyUnavailable`. The attacker's cost is low and the attack is repeatable, creating a sustained liveness failure for ckETH withdrawals.

In `withdraw_eth`, the guard is acquired before any ckETH is burned:

When `GuardError::TooManyPendingRequests` is returned, the call traps (panics), which means the user's call is rejected before any burn occurs. The `pending_withdrawal_requests` queue is only drained when the minter's timer fires `process_retrieve_eth_requests()` and creates Ethereum transactions for the queued requests. [6](#0-5) 

The `pending_withdrawal_requests` VecDeque holds requests from the moment `withdraw_eth` succeeds (ckETH burned, request enqueued) until the minter's timer processes them into signed Ethereum transactions: [7](#0-6) 

### Impact Explanation

Once 100 withdrawal requests are in `pending_withdrawal_requests`, every `withdraw_eth` call from any user — regardless of their principal — is rejected with `TemporarilyUnavailable`. Users who have already burned their ckETH are waiting for their ETH; new users cannot initiate withdrawals at all. This is a complete liveness failure for the ckETH withdrawal path.

The `withdraw_erc20` path uses the same guard mechanism and is equally affected. [8](#0-7) 

### Likelihood Explanation

**Normal-usage case**: The current minimum withdrawal amount is 0.005 ETH (~$10). If ckETH adoption grows and 100 users simultaneously submit withdrawal requests before the minter's timer fires, the queue fills naturally and new users are blocked. The minter processes requests in batches on a timer interval (`PROCESS_ETH_RETRIEVE_TRANSACTIONS_INTERVAL`), so during high-traffic periods the queue can remain saturated.

**Attack case**: An attacker with 100 distinct IC principals (trivially created) and 0.5 ETH (100 × 0.005 ETH minimum) can:
1. Submit 100 `withdraw_eth` calls, one per principal, filling `pending_withdrawal_requests` to `MAX_PENDING`.
2. All subsequent `withdraw_eth` calls from legitimate users fail with `TemporarilyUnavailable`.
3. The minter processes the attacker's requests, sending ETH back to the attacker minus gas fees.
4. The attacker re-deposits ETH to get ckETH and repeats.

Net cost per attack cycle: Ethereum gas fees for 100 transactions (currently cents to low dollars each) plus IC ingress fees. At current Ethereum fees, the yearly cost to permanently block ckETH withdrawals is in the range of hundreds to low thousands of USD — comparable to the H-01 GMX attack cost of ~$3,504/year.

The `MAX_PENDING = 100` limit is hardcoded with no per-user rate limiting or anti-spam mechanism: [1](#0-0) 

### Recommendation

1. **Per-principal pending request cap**: Limit the number of pending withdrawal requests per principal (e.g., max 5 per principal) in addition to the global cap, so a single attacker cannot monopolize the queue.
2. **Raise `MAX_PENDING`**: Increase the global limit to a value that reflects realistic peak demand (e.g., 1000), reducing the probability of accidental saturation.
3. **Minimum deposit/fee to deter spam**: Require a non-refundable IC cycles fee or a higher minimum withdrawal amount to raise the cost of queue-filling attacks.
4. **Queue drain prioritization**: Ensure the minter's timer fires frequently enough that the queue drains faster than an attacker can refill it.

### Proof of Concept

1. Attacker creates 100 IC principals (free on IC).
2. Each principal deposits 0.005 ETH to the ckETH helper contract, receiving 0.005 ckETH minus fees.
3. Each principal calls `withdraw_eth` with `amount = 0.005 ETH` and `recipient = attacker_eth_address`.
4. After 100 such calls, `withdrawal_requests_len() == 100 == MAX_PENDING`.
5. Any legitimate user calling `withdraw_eth` now hits:
   ```
   if PR::pending_requests_count(s) >= MAX_PENDING {
       return Err(GuardError::TooManyPendingRequests);
   }
   ```
   which causes `ic_cdk::trap(...)` in `withdraw_eth`, rejecting the call.
6. The minter's timer eventually processes the 100 attacker requests, sending ETH back to the attacker (minus gas fees).
7. Attacker repeats from step 2. [9](#0-8) [5](#0-4) [4](#0-3)

    fn new(principal: Principal) -> Result<Self, GuardError> {
        mutate_state(|s| {
            if PR::pending_requests_count(s) >= MAX_PENDING {
                return Err(GuardError::TooManyPendingRequests);
            }
            let principals = PR::guarded_principals(s);
            if principals.contains(&principal) {
                return Err(GuardError::AlreadyProcessing);
            }
            if principals.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            principals.insert(principal);
            Ok(Self {
                principal,
                _marker: PhantomData,
            })
        })
    }
