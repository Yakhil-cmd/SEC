### Title
VIP Caller Bypass Broken in `try_borrow_slot`: Unprivileged Callers Can Block Governance Operations via Slot Exhaustion — (`rs/nns/handlers/root/impl/canister/canister.rs`, `rs/nervous_system/clients/src/management_canister_client.rs`)

---

### Summary

The `LimitedOutstandingCallsManagementCanisterClient` in the NNS root canister contains a logic flaw: the "VIP bypass" for NNS canisters (e.g., governance) prevents them from *consuming* slots but does not bypass the zero-check guard. An unprivileged caller can exhaust all 167 slots via the public `canister_status` endpoint, causing the zero-check to reject even VIP governance callers with `SysTransient`, blocking governance-initiated `update_canister_settings`, `take_canister_snapshot`, and `load_canister_snapshot` operations during the attack window.

---

### Finding Description

**The broken VIP guard in `try_borrow_slot`:**

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {   // ← applies to ALL callers, including VIPs
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
``` [1](#0-0) 

VIP callers have `used_slot_count = 0` so they don't deplete the pool, but the `if *available_slot_count == 0` guard fires for *everyone* — including VIPs — when the pool is empty.

**The public attack surface:**

`canister_status` on the NNS root canister is a public `#[update]` endpoint with no authorization check: [2](#0-1) 

**The shared slot pool:**

All methods — `canister_status`, `update_settings`, `stop_canister`, `delete_canister`, `take_canister_snapshot`, `load_canister_snapshot` — share the same `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` thread-local initialized to 167: [3](#0-2) 

**VIP detection:**

Governance is correctly identified as VIP (NNS canister), so `used_slot_count = 0`: [4](#0-3) 

**Affected governance operations (those that call `new_management_canister_client()`):**

- `update_canister_settings` — governance-only
- `take_canister_snapshot` — governance-only
- `load_canister_snapshot` — governance-only
- `change_canister_controllers` — SNS-W-only [5](#0-4) 

**NOT affected** (bypass `new_management_canister_client()` entirely):
- `change_nns_canister` → calls `change_canister()` which uses `Rt::call_with_cleanup` directly
- `stop_or_start_nns_canister` → calls `stop_canister::<CdkRuntime>` directly [6](#0-5) 

---

### Impact Explanation

During the attack window, governance proposals that require `update_canister_settings`, `take_canister_snapshot`, or `load_canister_snapshot` via the root canister will fail with `SysTransient`. This blocks NNS governance execution for those proposal types. The attack is temporary but can be sustained continuously by an attacker who keeps replenishing the 167 concurrent `canister_status` calls as slots free up.

The primary canister upgrade path (`change_nns_canister` → `change_canister`) is **not** affected because it bypasses `LimitedOutstandingCallsManagementCanisterClient` entirely.

---

### Likelihood Explanation

- The `canister_status` endpoint requires no credentials — any principal can call it.
- The IC canister input queue holds up to 500 messages; 167 concurrent callers is well within reach.
- Each `canister_status` call awaits a management canister round-trip, creating a multi-round window where all slots are occupied.
- The attack can be sustained by a single coordinated actor or a small botnet of principals.
- No privileged access, key material, or consensus corruption is required.

---

### Recommendation

Fix `try_borrow_slot` so that VIP callers bypass the zero-check entirely, not just the slot subtraction:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    if self.is_caller_vip {
        // VIPs bypass both the check and the subtraction.
        return Ok(SlotLoan { available_slot_count: self.available_slot_count, used_slot_count: 0 });
    }

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                return Err((code, "Unavailable. Maybe, try again later?".to_string()));
            }
            *available_slot_count -= 1;
            Ok(())
        })?;
    ...
}
```

Alternatively, reserve a dedicated sub-pool of slots exclusively for VIP callers that non-VIP callers cannot touch.

---

### Proof of Concept

State-machine test outline:

1. Deploy NNS root canister with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Spawn 167 concurrent `canister_status` update calls from anonymous/unprivileged principals, each targeting a canister whose management canister response is artificially delayed (e.g., a canister in `Stopping` state).
3. While all 167 calls are in-flight (slots = 0), send an `update_canister_settings` call from the governance canister principal.
4. Assert the governance call returns `Err((SysTransient, "Unavailable. Maybe, try again later?"))`.
5. Wait for the 167 calls to complete (slots restored to 167).
6. Retry the governance `update_canister_settings` call and assert it succeeds.

The existing test infrastructure in `rs/nervous_system/clients/src/management_canister_client/tests.rs` already demonstrates the slot exhaustion mechanics with a capacity-2 pool; scaling to 167 with a VIP caller added to step 3 directly proves the invariant violation. [7](#0-6)
