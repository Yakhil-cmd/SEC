### Title
Unchecked Inter-Canister Call Result in `burn_and_log` Silently Locks ICP in CMC Subaccount - (File: rs/nns/cmc/src/main.rs)

### Summary

The Cycles Minting Canister (CMC) contains a `burn_and_log` function that makes an inter-canister call to the ICP ledger to burn ICP after successfully minting cycles or creating a canister. If the ledger call fails for any reason (e.g., transient unavailability during an upgrade), the error is only logged and the function returns `()`. The ICP remains permanently locked in the CMC's subaccount with no recovery or retry mechanism, breaking ICP supply conservation.

### Finding Description

In `rs/nns/cmc/src/main.rs`, three public-facing notification flows — `process_top_up`, `process_create_canister`, and `process_mint_cycles` — all follow the same pattern:

1. Perform the primary action (deposit cycles, create canister, or mint cycles to cycles ledger).
2. On success, call `burn_and_log(sub, amount).await` to burn the ICP from the CMC's subaccount.

The `burn_and_log` function (lines 2017–2049) is explicitly designed to swallow errors:

```rust

/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.

    ...

        call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;

```

When the ledger call fails, the function logs the failure and returns. The ICP that was transferred to the CMC's subaccount is never burned and never refunded. There is no retry queue, no deferred burn mechanism, and no administrative recovery path exposed in the CMC.

The three callers that trigger this path are:

- `process_create_canister` (line 1945): `burn_and_log(sub, amount).await;`
- `process_top_up` (line 2001): `burn_and_log(sub, amount).await;`
- `process_mint_cycles` (line 1968): `burn_and_log(sub, amount).await;`

In all three cases, the primary action (cycles deposited, canister created, cycles minted) has already succeeded and been committed before `burn_and_log` is called. The notification block is also marked as processed, so the user cannot retry.

### Impact Explanation

**Ledger conservation bug.** ICP that should be burned (destroyed) to back the minted cycles remains in the CMC's subaccount indefinitely. This means:

- The ICP total supply is not reduced as the protocol intends, breaking the ICP↔cycles economic invariant.
- The locked ICP in the CMC subaccount is inaccessible: the user cannot reclaim it (the notification is marked done), and there is no CMC administrative method to sweep or retry failed burns.
- Over repeated failures, the CMC accumulates unburned ICP, silently inflating the effective ICP supply relative to the cycles that were minted against it.

### Likelihood Explanation

The ICP ledger is a canister and can be temporarily unavailable during NNS-governed upgrades, which occur regularly. During a ledger upgrade window (typically seconds to minutes), any `burn_and_log` call that executes will receive a `SysTransient` reject from the ledger. Because the CMC marks the notification as processed before calling `burn_and_log`, the user cannot retry, and the burn is permanently skipped. Any user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during a ledger upgrade window will trigger this condition. No special privilege is required.

### Recommendation

Replace the fire-and-forget pattern in `burn_and_log` with a persistent retry queue. On ledger call failure, store the `(subaccount, amount)` pair in stable state and expose a permissionless `retry_burn` endpoint (or drive retries from the CMC's heartbeat/timer). This is the standard IC pattern for ensuring eventual consistency of cross-canister financial operations. Alternatively, if the design intent is to never block notifications, at minimum the failed burn entries must be persisted so that an operator or governance proposal can drain them.

### Proof of Concept

1. User sends ICP to the CMC's subaccount for a canister top-up and calls `notify_top_up`.
2. CMC calls `deposit_cycles` — succeeds; cycles are credited to the target canister.
3. CMC calls `burn_and_log`; the ledger is mid-upgrade and returns a `SysTransient` reject.
4. `burn_and_log` logs `"Burning of X ICPTs from subaccount Y failed with code 4: ..."` and returns `()`.
5. The notification block is already marked `NotifiedTopUp` in `blocks_notified`; the user's retry of `notify_top_up` returns the cached success result without re-attempting the burn.
6. The ICP remains in the CMC's subaccount. The cycles were minted. The ICP was not burned.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
