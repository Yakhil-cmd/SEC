### Title
Instruction-Limit Exhaustion via Unbounded O(N) Node Scans in Registry Canister Update Calls — (`rs/registry/canister/src/mutations/node_management/do_add_node.rs`, `rs/registry/canister/src/mutations/do_remove_node_operators.rs`)

---

### Summary

The registry canister's `do_add_node` (callable by any registered node operator) and `do_remove_node_operators` (callable via NNS governance execution) each perform multiple unbounded O(N) full scans over all node records in the registry within a single update call. As the IC network grows, the cumulative instruction cost of these scans — compounded by the global invariant checks run on every mutation — will eventually exceed the per-message instruction limit, making these calls permanently uncallable. This is the direct IC analog to the Liquid Collective `_pickNextValidatorsToDepositFromActiveOperators` / `_requestExitsBasedOnRedeemDemandAfterRebalancings` out-of-gas pattern.

---

### Finding Description

**Root cause 1 — `do_add_node` (node-operator-callable):**

`do_add_node_` in `rs/registry/canister/src/mutations/node_management/do_add_node.rs` performs at least three independent full scans of the entire node record family before applying any mutation:

1. `scan_for_nodes_by_ip` — iterates every `NodeRecord` to find IP conflicts.
2. `get_node_operator_nodes` — iterates every `NodeRecord` to count nodes owned by the caller (executed when node rewards are enabled).
3. `node_exists_with_ipv4` — iterates every `NodeRecord` to check IPv4 uniqueness.

After these scans, `maybe_apply_mutation_internal` is called, which unconditionally invokes `check_global_state_invariants`. That function runs a full suite of invariant checkers — including `check_node_crypto_keys_invariants`, which iterates over every node's crypto keys in the snapshot. The crypto invariant code itself contains an explicit acknowledgment of the problem:

> *"It is NOT CHECKED that the crypto keys are fully well-formed or valid, as these checks are expensive in terms of computation (about 200 times more expensive than just parsing, 400M instructions per node vs. 2M instructions), so for the mainnet state with 1K+ nodes the full validation would go over the instruction limit per message."*

Even with the expensive per-key validation disabled, the remaining invariant suite still iterates over all nodes multiple times. Combined with the three pre-mutation scans, the total instruction cost per `add_node` call scales as O(K × N) where K is the number of full-scan passes and N is the total node count.

**Root cause 2 — `filter_out_node_operators_that_have_nodes` (governance-execution path):**

`filter_out_node_operators_that_have_nodes` in `rs/registry/canister/src/mutations/do_remove_node_operators.rs` performs a full scan of every `NodeRecord` for each call to `do_remove_node_operators`. The code carries a self-admitted comment:

> *"This implementation is inefficient, because it does a full scan of all nodes."*

The inner body calls `retain` on the `node_operators` vector for every node record, yielding O(nodes × operators_to_remove) work. This is called synchronously during NNS governance proposal execution of `RemoveNodeOperators` (NNS function 23). If the proposal execution traps due to instruction exhaustion, the proposal is marked failed and the node operator records cannot be removed.

---

### Impact Explanation

**`do_add_node` path:** Once the IC network grows past the instruction-limit threshold, every call to `add_node` on the registry canister will trap. No registered node operator will be able to add new nodes to the network. This permanently halts network expansion and node replacement (including the IP-conflict-based node replacement flow used during hardware refreshes). The registry canister is a system canister on the NNS subnet; its update calls have a higher instruction limit than application subnets, but the limit is finite and the growth is unbounded.

**`do_remove_node_operators` path:** NNS governance proposals of type `RemoveNodeOperators` will permanently fail to execute once the node count crosses the threshold. Node operator records that should be cleaned up will remain in the registry indefinitely, polluting the registry state and potentially interfering with node operator allowance accounting.

---

### Likelihood Explanation

The IC mainnet already has over 1,000 nodes. The crypto invariant code explicitly documents that full per-key validation "would go over the instruction limit per message" at 1K+ nodes — confirming the instruction budget is already tight at current scale. The three additional O(N) pre-mutation scans in `do_add_node` are additive. As the network targets tens of thousands of nodes, the combined cost will cross the limit. No adversarial action is required; organic network growth is sufficient. A node operator who has already been registered (a one-time governance step) can trigger `add_node` at any time without further privileged access.

---

### Recommendation

**For `do_add_node`:**
1. Replace `scan_for_nodes_by_ip` with an indexed lookup (maintain a reverse IP→NodeId index in the registry, updated on node add/remove).
2. Replace `get_node_operator_nodes` with a per-operator counter stored directly in the `NodeOperatorRecord`, incremented/decremented on node add/remove.
3. Replace `node_exists_with_ipv4` with a similar indexed lookup.
4. Consider batching or deferring invariant checks for mutations that do not affect the checked invariant domains, or introduce a per-mutation invariant scope to avoid running all invariant checkers on every mutation.

**For `do_remove_node_operators`:**
1. Maintain a per-operator node count in `NodeOperatorRecord` (same index as above). `filter_out_node_operators_that_have_nodes` then becomes an O(1) lookup per operator instead of an O(nodes) scan.
2. Alternatively, allow the governance proposal to specify a batch size and split the operation across multiple proposals if the operator list is large.

    fn filter_out_node_operators_that_have_nodes(&self, node_operators: &mut Vec<PrincipalId>) {
        if node_operators.is_empty() {
            return;
        }

        // This implementation is inefficient, because it does a full scan of all nodes.
        for (_key, node_record) in get_key_family_iter::<NodeRecord>(self, NODE_RECORD_KEY_PREFIX) {
            // Throw out node operators that operate the node (that this loop is currently considering).
            node_operators
                .retain(|node_operator| node_operator.to_vec() != node_record.node_operator_id);
        }
    }
