### Title
Silent ICP Burn Failure After Cycles Minting Violates ICP/Cycles Conservation Invariant — (File: rs/nns/cmc/src/main.rs)

---

### Summary

The Cycles Minting Canister (CMC) intentionally suppresses errors from its `burn_and_log` function, which is responsible for burning ICP on the ledger **after** cycles have already been minted and deposited. If the ICP ledger call inside `burn_and_log` fails (e.g., during a ledger upgrade or transient NNS subnet unavailability), cycles are permanently created without the corresponding ICP being destroyed. The notification block is then marked as successfully processed, preventing any retry. This violates the ICP/cycles conservation invariant across all three CMC minting paths.

---

### Finding Description

In `rs/nns/cmc/src/main.rs`, three async functions follow the same two-phase pattern:

1. **Phase 1 – Mint/deposit cycles** (irreversible once successful):
   - `process_top_up` (line 1999): calls `deposit_cycles` → management canister `deposit_cycles`
   - `process_create_canister` (line 1943): calls `do_create_canister`
   - `process_mint_cycles` (line 1966): calls `do_mint_cycles` → cycles ledger `deposit`

2. **Phase 2 – Burn ICP** (called only on Phase 1 success):
   - All three call `burn_and_log(sub, amount).await` (lines 2001, 1945, 1968)

The `burn_and_log` function (lines 2017–2049) is explicitly designed to swallow errors:

```rust
// Line 2014-2016 comment:

/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
```


The error is only logged; it is never returned. The callers (`process_top_up`, `process_create_canister`, `process_mint_cycles`) return `Ok(...)` unconditionally after Phase 1 succeeds, regardless of whether the ICP burn succeeded. The notification block is then recorded as `NotifiedTopUp(Ok(cycles))` / `NotifiedCreateCanister(Ok(...))` / `NotifiedMint(Ok(...))`, permanently closing the notification window and preventing any retry.

The ICP that was sent to the CMC's subaccount remains there indefinitely — it is neither burned nor refunded. Meanwhile, the cycles or canister created in Phase 1 are fully live.

---

### Impact Explanation

**Ledger conservation bug**: The IC's economic model requires that every cycle minted corresponds to ICP burned (1 XDR worth of ICP = 1 trillion cycles). When `burn_and_log` fails silently:

- Cycles are permanently minted and deposited to the target canister (or cycles ledger account, or new canister created).
- The corresponding ICP is **not** burned; it remains stranded in the CMC's subaccount (e.g., `Subaccount::from(&canister_id)` for top-ups).
- The total supply of ICP + cycles is inflated relative to the protocol invariant.
- The stranded ICP has no recovery path: the notification is closed, the user cannot re-notify, and there is no background sweep to retry failed burns.

This affects all three minting paths: `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles`.

---

### Likelihood Explanation

The trigger is the ICP ledger being temporarily unreachable from the CMC at the moment `burn_and_log` issues its `send_pb` call. This can occur during:

- **Routine ICP ledger upgrades** on the NNS subnet (the ledger canister is stopped/upgraded, causing inter-canister calls to it to be rejected with a transient error for the duration of the upgrade).
- **NNS subnet message queue saturation** causing the ledger call to be dropped.
- Any transient `CanisterStopped` / `CanisterStopping` reject from the ledger.

Ledger upgrades are routine governance operations that happen multiple times per year. Any user who happens to call `notify_top_up` (or the other two endpoints) at the exact moment the ledger is being upgraded will trigger this condition. The user does not need any special privilege; the three `notify_*` endpoints are fully open to unprivileged callers.

---

### Recommendation

The intentional design choice (comment at line 2014–2016) trades correctness for retry-safety. A safer approach:

1. **Retry the burn asynchronously**: Record a "pending burn" in stable state when `burn_and_log` fails, and have the heartbeat or a timer retry it until it succeeds. This preserves the no-retry guarantee for the user-facing notification while ensuring the burn eventually completes.
2. **Alternatively**, use `created_at_time` deduplication on the burn `send_pb` call so that a retry of the burn is idempotent and safe even if the original call was delivered but the response was lost.

---

### Proof of Concept

1. User sends 10 ICP to `CMC_subaccount(canister_X)` on the ICP ledger (block index `B`).
2. User calls `notify_top_up { block_index: B, canister_id: canister_X }` on the CMC.
3. CMC fetches the transaction, sets block `B` status to `Processing`.
4. CMC calls `deposit_cycles(canister_X, cycles, ...)` → management canister deposits cycles to `canister_X`. **Cycles are now live.**
5. CMC calls `burn_and_log(subaccount_X, 10 ICP)` → ICP ledger is mid-upgrade → `send_pb` returns `Err(CanisterStopped, ...)`.
6. `burn_and_log` logs `"Burning of 10 ICP failed with code -1: ..."` and returns `()`.
7. `process_top_up` returns `Ok(cycles)`.
8. CMC sets block `B` status to `NotifiedTopUp(Ok(cycles))`.

```rust
/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
```
