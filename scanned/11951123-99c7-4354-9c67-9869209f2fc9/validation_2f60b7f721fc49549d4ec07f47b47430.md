### Title
Unbounded `limit` in `services::search_transactions` allows unauthenticated OOM crash of the ICRC-1 Rosetta process — (`rs/rosetta-api/icrc1/src/data_api/services.rs`)

### Summary
An unprivileged HTTP client can send a `SearchTransactionsRequest` with `limit=9223372036854775807` (`i64::MAX`). The server-side code uses `MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST` (10 000) only as a default when `limit` is absent, never as a cap. The attacker-supplied value passes `try_into::<u64>()` without error, becomes the SQL `LIMIT` clause verbatim, and causes the tokio-rusqlite thread to load every matching block into a `Vec<RosettaBlock>` in process memory, crashing the single Rosetta replica with OOM.

### Finding Description
In `rs/rosetta-api/icrc1/src/data_api/services.rs`, `search_transactions` resolves the effective limit as:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST as i64)
    .try_into()  // i64::MAX → u64 succeeds
    ...?;
```

`i64::MAX` is a valid `u64`, so `try_into()` succeeds. No subsequent guard clamps the value. The limit is then used directly:

```rust
command.push_str("LIMIT :limit ");
parameters.push((":limit".to_string(), rusqlite::types::Value::Integer(limit as i64)));
```

SQLite receives `LIMIT 9223372036854775807`, effectively no limit. All matching rows are fetched and pushed into a `Vec` in memory before the response is serialized.

### Impact Explanation
The Rosetta node is a standalone OS process. A single HTTP POST to `/search/transactions` with `limit=9223372036854775807` and no other filters (or a broad account filter) against a node with a large block database will exhaust process heap memory and trigger an OOM kill, taking down the entire Rosetta replica. No authentication is required.

### Likelihood Explanation
The endpoint is publicly reachable (it is the standard Rosetta API surface). The payload is trivial to construct. A single request is sufficient; no volumetric attack is needed.

### Recommendation
Enforce a hard server-side cap immediately after parsing:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST as i64)
    .try_into()
    .map_err(|err| Error::request_processing_error(...))?;

let limit = limit.min(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST);
```

The same fix should be applied to `offset` and `max_block` for defense-in-depth.

### Proof of Concept
```rust
// Unit test sketch
let storage = StorageClient::new_in_memory().await.unwrap();
// insert 100_000 blocks ...
let request = SearchTransactionsRequest {
    limit: Some(i64::MAX),
    ..Default::default()
};
// Without the fix this materializes all 100k blocks into memory;
// with a large enough DB this OOMs the process.
let _ = services::search_transactions(&storage, request, "ICP".into(), 8).await;
```
