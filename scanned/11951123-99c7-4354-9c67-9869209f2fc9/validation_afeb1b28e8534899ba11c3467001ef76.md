### Title
Front-Running ICP/XDR Rate Update Allows Minting Excess Cycles at Stale Favorable Rate - (File: rs/nns/cmc/src/main.rs)

### Summary
The Cycles Minting Canister (CMC) uses a two-step flow for converting ICP to cycles: (1) transfer ICP to a CMC subaccount on the ledger, then (2) call `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister`. The conversion rate applied is the **current** `icp_xdr_conversion_rate` at the time of the notification call, not at the time of the ICP transfer. Because NNS governance proposals to increase the ICP/XDR rate are publicly observable days in advance, an unprivileged user can front-run the rate update: send ICP before the proposal executes, then call notify after the rate has increased, receiving more cycles per ICP than the ICP was worth at transfer time.

### Finding Description
`tokens_to_cycles()` reads `state.icp_xdr_conversion_rate` at the moment the notification is processed:

```rust

This function is called by all three notification paths — `process_top_up`, `process_mint_cycles`, and `process_create_canister` — after the ICP amount has already been locked in the ledger block: [2](#0-1) [3](#0-2) [4](#0-3) 

The rate is updated via `set_icp_xdr_conversion_rate`, callable only by the NNS Governance canister, which executes after a governance proposal passes: [5](#0-4) 

The rate is also updated automatically via the exchange rate canister heartbeat (`update_exchange_rate`), which calls `do_set_icp_xdr_conversion_rate` every five minutes: [6](#0-5) 

There is no check binding the conversion rate to the block timestamp of the ICP transfer. The `blocks_notified` map only prevents double-spending the same block; it does not enforce rate-at-transfer-time semantics: [7](#0-6) 

### Impact Explanation
An attacker who front-runs a rate increase receives more cycles per ICP than the ICP was worth at the time of the on-chain transfer. The cycles supply is inflated relative to the ICP burned. Concretely, if the ICP/XDR rate increases by 20% (e.g., from 10,000 to 12,000 `xdr_permyriad_per_icp`), an attacker who sends 1,000 ICP before the update and notifies after receives 20% more cycles than a user who sent the same ICP after the update. The ICP is still burned, but the cycles minted exceed the economic value of the ICP at transfer time, diluting the cycles economy.

### Likelihood Explanation
NNS governance proposals are fully public and have a multi-day voting period. The `UpdateIcpXdrConversionRatePayload` and its `xdr_permyriad_per_icp` field are visible on-chain before execution: [8](#0-7) 

Any unprivileged principal can execute the attack with two ordinary ingress calls (ICP ledger `transfer` + CMC `notify_top_up` / `notify_mint_cycles`). No special role, key, or majority is required. The notification window is bounded only by `MAX_NOTIFY_HISTORY` blocks, giving the attacker ample time to wait for the rate update before calling notify.

### Recommendation
Bind the conversion rate to the ledger block timestamp of the ICP transfer rather than the notification time. When `fetch_transaction` retrieves the block, record the block's timestamp and look up the ICP/XDR rate that was active at that timestamp (e.g., using `recent_icp_xdr_rates` keyed by day). Alternatively, record the rate in the `blocks_notified` entry at the time the block is first marked `Processing`, and use that cached rate for the conversion regardless of when the notification completes. [9](#0-8) 

### Proof of Concept
1. Monitor the NNS dashboard for a pending `UpdateIcpXdrConversionRate` proposal that increases `xdr_permyriad_per_icp` (e.g., from 10,000 to 15,000).
2. Before the proposal executes, send N ICP to `AccountIdentifier(CMC_ID, Subaccount(caller))` with memo `MEMO_MINT_CYCLES`. Record the resulting `block_index`.
3. Wait for the NNS proposal to execute and `set_icp_xdr_conversion_rate` to update the CMC state.
4. Call `notify_mint_cycles({ block_index, to_subaccount: None, deposit_memo: None })`.
5. Observe that `minted` in `NotifyMintCyclesSuccess` reflects the new higher rate (15,000 XDR/ICP) rather than the rate at transfer time (10,000 XDR/ICP), yielding 50% more cycles than the ICP was worth when transferred. [10](#0-9) [11](#0-10)

pub async fn update_exchange_rate(
    safe_state: &'static LocalKey<RefCell<Option<State>>>,
    env: &impl Environment,
    xrc_client: &impl ExchangeRateCanisterClient,
) -> Result<(), UpdateExchangeRateError> {
    let now_timestamp_seconds = env.now_timestamp_seconds();
    let current_minute_seconds =
        round_down_to_multiple_of(now_timestamp_seconds, ONE_MINUTE_SECONDS);

    UpdateExchangeRateGuard::with_guard(safe_state, current_minute_seconds, async {
        let call_xrc_result = xrc_client.get_icp_to_xdr_exchange_rate(None).await;
        // Check if updating the rate via the exchange rate canister was disabled while retrieving the rate.
        // If it has, exit early.
        let is_updating_rate_disabled = read_state(safe_state, |state| {
            state
                .update_exchange_rate_canister_state
                .unwrap_or_default()
                == UpdateExchangeRateState::Disabled
        });
        if is_updating_rate_disabled {
            return Err(UpdateExchangeRateError::Disabled);
        }

        match call_xrc_result {
            Ok(exchange_rate) => {
                validate_exchange_rate(&exchange_rate)
                    .map_err(|error| UpdateExchangeRateError::InvalidRate(error.to_string()))?;
                let icp_xdr_conversion_rate = IcpXdrConversionRate::from(exchange_rate);
                if let Err(error) =
                    do_set_icp_xdr_conversion_rate(safe_state, env, icp_xdr_conversion_rate)
                {
                    return Err(UpdateExchangeRateError::FailedToSetRate(error));
                }
            }
            Err(error) => {
                return Err(UpdateExchangeRateError::FailedToRetrieveRate(
                    error.to_string(),
                ));
            }
        };

        Ok(())
    })
    .await
}
