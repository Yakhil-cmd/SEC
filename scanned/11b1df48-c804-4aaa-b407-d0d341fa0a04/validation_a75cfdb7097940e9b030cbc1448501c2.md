### Title
SNS Token Treasury Transfer Limits Enforced Using Inaccurate Historical Price Model - (`rs/sns/governance/token_valuation/src/lib.rs`)

### Summary

The `IcpsPerSnsTokenClient::fetch_icps_per_sns_token()` function in `rs/sns/governance/token_valuation/src/lib.rs` computes the current SNS token price using only the **initial swap price** and **supply inflation**, ignoring actual market price movements. This inaccurate price feeds directly into the 7-day treasury transfer and minting limits (`transfer_sns_treasury_funds_7_day_total_upper_bound_tokens`, `mint_sns_tokens_7_day_total_upper_bound_tokens`). When the SNS token has appreciated significantly from its initial swap price, the treasury is systematically undervalued, causing the protective limits to be far too permissive and allowing excessive treasury outflows.

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, `fetch_icps_per_sns_token()` computes the current ICP-per-SNS-token price as:

```
current_price = (1 / initial_sns_tokens_per_icp) / (current_supply / initial_supply)
``` [1](#0-0) 

The model assumes the SNS token price is **purely** a function of its initial swap price (fetched from `get_derived_state` on the swap canister) and supply inflation. It ignores all actual market price movements.

This is directly analogous to the external report's `get_virtual_price` issue: a proxy price is used that assumes a specific mathematical relationship between token value and supply, which does not hold in practice. Just as `get_virtual_price` assumes all pool tokens equal $1, this model assumes the SNS token price only changes due to supply changes from the initial swap price.

The computed price is consumed by `try_get_sns_token_balance_valuation`: [2](#0-1) 

Which feeds into `treasury_valuation_if_proposal_amount_is_small_enough_or_err` in `rs/sns/governance/src/proposal.rs`: [3](#0-2) 

Which enforces the 7-day treasury limits via `ProposalsAmountTotalUpperBound::in_tokens`: [4](#0-3) 

The three treasury size regimes are:
- **Small** (< 100,000 XDR): `NoLimit` — the entire treasury can be transferred
- **Medium** (100,000–1,200,000 XDR): 25% per 7 days
- **Large** (> 1,200,000 XDR): 300,000 XDR per 7 days [5](#0-4) 

Critically, there is no `MAX_XDRS_PER_ICP` clamp and no clamp on `icps_per_token`. The only floor is `MIN_XDRS_PER_ICP = 1`, which protects against ICP undervaluation but does nothing to correct an underestimated SNS token price: [6](#0-5) 

### Impact Explanation

**Concrete scenario:**
- Initial swap: 100 SNS tokens per ICP → `initial_icps_per_sns_token = 0.01`
- 10% supply inflation → `total_inflation = 1.1`
- Model's current price: `0.01 / 1.1 ≈ 0.0091` ICP per SNS token
- Actual market price: 1 ICP per SNS token (100× appreciation, plausible for a successful SNS)
- XDR/ICP rate: 5 XDR/ICP

Treasury with 1,000,000 SNS tokens:
- **Model's valuation**: `0.0091 × 1,000,000 × 5 = 45,500 XDR` → classified **"small"** → `NoLimit`
- **Actual valuation**: `1 × 1,000,000 × 5 = 5,000,000 XDR` → should be **"large"** → limit = 300,000 XDR

A single `TransferSnsTreasuryFunds` proposal could drain the entire treasury (5,000,000 XDR worth), when the intended limit is 300,000 XDR per 7 days. The protective mechanism is completely bypassed.

### Likelihood Explanation

Any SNS token that appreciates significantly from its initial swap price triggers this condition. Successful SNS projects are expected to appreciate over time. The discrepancy grows monotonically as market price diverges from the initial swap price. The model has no mechanism to self-correct. This is not a transient edge case — it is the expected steady-state for any successful SNS.

### Recommendation

Replace the initial-swap-price + inflation model with an actual market price oracle. The Exchange Rate Canister (XRC) already provides ICP/XDR rates; a similar mechanism should be used for SNS token prices. Alternatively, integrate a DEX TWAP (time-weighted average price) for the SNS token. At minimum, add a `MAX_ICPS_PER_TOKEN` clamp symmetric to `MIN_XDRS_PER_ICP` to bound the undervaluation risk.

### Proof of Concept

1. Deploy an SNS with initial swap at 100 SNS tokens per ICP.
2. The SNS token appreciates to 1 ICP per SNS token on the open market (100× from initial swap).
3. An SNS neuron holder submits a `TransferSnsTreasuryFunds` proposal to transfer 1,000,000 SNS tokens.
4. Proposal validation calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err` → `assess_treasury_balance` → `try_get_sns_token_balance_valuation` → `IcpsPerSnsTokenClient::fetch_icps_per_sns_token()`.
5. `fetch_icps_per_sns_token()` returns `≈ 0.0091` ICP/SNS (model price), not `1.0` ICP/SNS (market price).
6. Treasury is valued at `≈ 45,500 XDR` → `NoLimit` regime → proposal passes validation.
7. The governance majority, seeing the proposal is within the automatic limit, votes to approve.
8. 1,000,000 SNS tokens (actual value: 5,000,000 XDR) are transferred — 16× the intended 300,000 XDR cap.

async fn treasury_valuation_if_proposal_amount_is_small_enough_or_err<MyTokenProposalAction>(
    env: &dyn Environment,
    sns_ledger_canister_id: CanisterId,
    swap_canister_id: CanisterId,
    proposals: impl Iterator<Item = &ProposalData>,
    action: &MyTokenProposalAction,
) -> Result<Valuation, String>
where
    MyTokenProposalAction: TokenProposalAction,
{
    let spent_tokens = action.recent_amount_total_tokens(proposals, env.now())?;

    // Get valuation of the tokens in the treasury.
    let token = action.token()?;
    let valuation = assess_treasury_balance(
        token,
        env.canister_id(),
        sns_ledger_canister_id,
        swap_canister_id,
    )
    .await?;

    // From valuation, determine limit on the total from the past 7 days.
    let max_tokens = MyTokenProposalAction::recent_amount_total_upper_bound_tokens(&valuation)
        // Err is most likely a bug.
        .map_err(|treasury_limit_error| {
            format!("Unable to validate amount: {treasury_limit_error:?}",)
        })?;

    // Finally, inspect the proposal's amount: it must not exceed max - spent (remainder). Or if
    // you prefer, equivalently, amount + spent must be <= max.
    let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| {
        format!("Arithmetic error while performing {max_tokens} - {spent_tokens}",)
    })?;
    let proposal_amount_tokens = action.proposal_amount_tokens()?;
    if proposal_amount_tokens > allowance_remainder_tokens {
        // Although it might not be obvious to the user, their proposal is invalid, and we
        // consider it to be "their fault".
        return Err(format!(
            "Amount is too large. Within the past 7 days, a total of {spent_tokens} tokens has already \
             been executed in like proposals. Whereas, at most {max_tokens} is allowed. An additional \
             {proposal_amount_tokens} tokens from this proposal would cause that upper bound to be exceeded. \
             Maybe, try again in a few days?"
        ));
    }

    Ok(valuation)

    fn in_tokens(mut valuation: Valuation) -> Result<Decimal, ProposalsAmountTotalLimitError> {
        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))
                })?;

                // Calculate the inverse conversion rate.
                if xdrs_per_token == Decimal::from(0) {
                    // This is not reachable, because in this case, valuation.to_xdr() would return
                    // 0, and in that case, we would have taken the NoLimit branch.
                    return Err(ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
                }
                let tokens_per_xdr = xdrs_per_token.inv();

                max_xdr.checked_mul(tokens_per_xdr).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Max tokens could not be calculated with valuation: {valuation:?}",
                    ))
                })?
            }
        };

        Ok(result_tokens)
    }
