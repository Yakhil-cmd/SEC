### Title
Floating-Point Precision Loss in ICRC-21 Consent Message Token Amount Display Causes Incorrect Amount Shown to Users - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The `convert_tokens_to_string_representation` function in the ICRC-21 consent message library converts a `Nat` token amount to `f64` before dividing by `10^decimals`. For tokens with many decimal places (e.g., ckETH with 18 decimals), the `f64` conversion silently loses precision for any amount exceeding 2^53 ≈ 9 × 10^15. This causes the ICRC-21 consent message — the security-critical display users rely on to verify what they are signing — to show an incorrect token amount. For extremely large amounts, the display degrades to `"inf"`. The error-handling path (`ok_or`) is also dead code because `BigUint::to_f64()` never returns `None`.

---

### Finding Description

The vulnerable function is:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
``` [1](#0-0) 

`Nat` wraps a `num_bigint::BigUint`. The `ToPrimitive::to_f64()` implementation for `BigUint` **always** returns `Some(f64)` — it never returns `None`. For values that exceed `f64`'s 53-bit mantissa precision (2^53 ≈ 9 × 10^15), it silently rounds to the nearest representable double. For astronomically large values it returns `Some(f64::INFINITY)`. This means:

1. The `ok_or` error guard is dead code — it never fires.
2. Any `Nat` amount exceeding 2^53 is silently rounded before display.
3. Amounts large enough to overflow `f64` are displayed as `"inf"`.

This function is called from `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` in the `GenericDisplayMessage` branch of `ConsentMessage`: [2](#0-1) 

These methods are invoked by `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints`, which is the production entry point used by both the ICP ledger and the ICRC-1 ledger's `icrc21_canister_call_consent_message` update endpoint: [3](#0-2) [4](#0-3) [5](#0-4) 

The `decimals` value passed in is taken directly from `icrc1_decimals()` on the ledger, which for ckETH-style tokens is 18: [6](#0-5) 

---

### Impact Explanation

For a token with 18 decimals (e.g., ckETH, where 1 ETH = 10^18 Wei):

- `f64` can represent integers exactly only up to 2^53 ≈ 9 × 10^15.
- 1 ETH = 10^18 Wei, so **any ckETH amount greater than ~0.009 ETH** (9 × 10^15 Wei) loses precision in the `f64` conversion.
- The consent message shown to the user will display a rounded or incorrect amount.
- For amounts ≥ ~1.8 × 10^308 (practically unreachable but theoretically possible with a `Nat`), the display shows `"inf"`.

The ICRC-21 consent message is the **only** user-facing security display that wallets (e.g., hardware wallets, browser extensions) show before a user signs a transaction. Displaying an incorrect amount undermines the entire purpose of the consent message standard. A user could approve a transfer for a different amount than what they believe they are signing.

Additionally, in the `FieldsDisplayMessage` branch, `nat_to_u64` is used instead: [7](#0-6) 

For ckETH amounts exceeding `u64::MAX` ≈ 18.4 ETH, this returns an `Icrc21Error`, causing the consent message endpoint to fail entirely — a denial-of-service on consent message generation for any transfer above ~18.4 ETH.

---

### Likelihood Explanation

**High.** The ICRC-21 consent message endpoint is a publicly callable update method on every ICRC-1 ledger. Any user initiating a ckETH transfer of more than ~0.009 ETH (essentially all real-world transfers) via a wallet that requests a `GenericDisplayMessage` will receive an imprecise amount in the consent message. No special privileges are required. The attacker-controlled entry path is simply calling `icrc21_canister_call_consent_message` with a large `amount` field in the transfer/approve arguments.

---

### Recommendation

Replace the `f64`-based conversion with exact integer arithmetic. Perform the decimal shift using integer division and modulo on the `BigUint` directly, formatting the whole and fractional parts separately — similar to how `format_amount` is correctly implemented in the ckBTC minter: [8](#0-7) 

Apply the same pattern to `convert_tokens_to_string_representation`: divide the `BigUint` by `10^decimals` to get the whole part, take the remainder for the fractional part, and format both as strings without any floating-point conversion. Also fix `nat_to_u64` in the `FieldsDisplayMessage` path to handle amounts exceeding `u64::MAX` gracefully (e.g., by using a `u128` or `BigUint`-based representation, or by capping and annotating).

---

### Proof of Concept

For a ckETH ledger with `decimals = 18`, call `icrc21_canister_call_consent_message` with a transfer of `amount = 1_000_000_000_000_000_001` Wei (1 ETH + 1 Wei):

1. `tokens.0.to_f64()` returns `Some(1_000_000_000_000_000_000.0)` — the trailing `1` is lost because 10^18 + 1 > 2^53.
2. `format!("{}", 1e18_f64 / 1e18_f64)` → `"1"`.
3. The consent message displays `"1 ckETH"` instead of `"1.000000000000000001 ckETH"`.

For a transfer of `amount = 10_000_000_000_000_000_000` Wei (10 ETH):

1. `to_f64()` returns `Some(10_000_000_000_000_000_000.0)` — representable exactly as `1e19`.
2. `format!("{}", 1e19_f64 / 1e18_f64)` → `"10"` — correct in this case, but adjacent values (e.g., 10 ETH + 1 Wei) are silently rounded.

For the `FieldsDisplayMessage` path with `amount = 20_000_000_000_000_000_000` Wei (20 ETH > `u64::MAX` ≈ 18.4 ETH):

1. `nat_to_u64` returns `Err(Icrc21Error::GenericError { ... })`.
2. The entire `icrc21_canister_call_consent_message` call fails — the user cannot obtain a consent message for this transfer.

fn icrc21_canister_call_consent_message(
    consent_msg_request: ConsentMessageRequest,
) -> Result<ConsentInfo, Icrc21Error> {
    let caller_principal = caller();
    let ledger_fee = Nat::from(LEDGER.read().unwrap().transfer_fee.get_e8s());
    let token_symbol = LEDGER.read().unwrap().token_symbol.clone();
    let token_name = LEDGER.read().unwrap().token_name.clone();
    let decimals = ic_ledger_core::tokens::DECIMAL_PLACES as u8;

    if consent_msg_request.method == "transfer" {
        let TransferArgs {
            memo,
            amount,
            fee,
            from_subaccount,
            to,
            created_at_time: _,
        } = Decode!(&consent_msg_request.arg, TransferArgs).map_err(|e| {
            Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
                description: format!("Failed to decode TransferArgs: {e}"),
            })
        })?;
        icrc21_check_fee(&Some(Nat::from(fee)), &ledger_fee)?;
        let from = if caller() == Principal::anonymous() {
            AccountOrId::AccountIdAddress(None)
        } else {
            let account = Account {
                owner: caller(),
                subaccount: from_subaccount.map(|sa| sa.0),
            };
            AccountOrId::AccountIdAddress(Some(AccountIdentifier::from(account).to_hex()))
        };
        let receiver = AccountIdentifier::from_slice(&to).map_err(|e| {
            Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
                description: format!("Failed to parse receiver account id: {e}"),
            })
        })?;
        let args = GenericTransferArgs {
            from,
            receiver: AccountOrId::AccountIdAddress(Some(receiver.to_hex())),
            amount: Nat::from(amount.get_e8s()),
            memo: Some(GenericMemo::IntMemo(memo.0)),
        };
        build_icrc21_consent_info(
            consent_msg_request,
            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
            Some(args),
        )
    } else {
        build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints(
            consent_msg_request,
            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
        )
    }
}

pub(super) fn format_amount(amount: u64, decimals: u8) -> String {
    let divisor = 10_u64.pow(decimals as u32);
    let whole = amount / divisor;
    let frac = amount % divisor;
    if frac == 0 {
        format!("{whole}")
    } else {
        let frac_str = format!("{frac:0width$}", width = decimals as usize);
        let trimmed = frac_str.trim_end_matches('0');
        format!("{whole}.{trimmed}")
    }
}
