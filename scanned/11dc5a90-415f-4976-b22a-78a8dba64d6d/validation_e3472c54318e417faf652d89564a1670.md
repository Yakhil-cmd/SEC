### Title
Floating-Point Precision Loss in ICRC-21 Consent Message Amount Display Understates Token Amounts for High-Decimal Assets - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The `convert_tokens_to_string_representation` function in the shared ICRC-21 library converts a `Nat` token amount to `f64` before dividing by `10^decimals` to produce a human-readable string. Because `f64` has only 53 bits of mantissa (~15–16 significant decimal digits), large token amounts lose precision silently. Any unprivileged caller of `icrc21_canister_call_consent_message` on any production ICRC-1 ledger (ICP, ckETH, ckUSDC, ckUSDT, etc.) can trigger this path and receive a consent message that displays a rounded/incorrect amount — the exact analog of the EVM vault's failure to scale amounts for non-18-decimal assets.

---

### Finding Description

`convert_tokens_to_string_representation` is the sole function used to render token amounts in the `GenericDisplayMessage` variant of ICRC-21 consent messages:

```rust
// packages/icrc-ledger-types/src/icrc21/responses.rs, lines 318-327
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
``` [1](#0-0) 

The function is called for every displayed field — amount, fee, allowance, and existing allowance — in the `GenericDisplayMessage` path: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The `Nat` type is arbitrary-precision, but `to_f64()` silently rounds to the nearest representable `f64`. IEEE 754 `f64` can represent integers exactly only up to `2^53 ≈ 9.007 × 10^15`. For ckETH (18 decimals), 1 ETH = `10^18` wei; amounts above roughly 9 ETH begin losing sub-wei precision. For amounts of 100 ETH (`10^20` wei), the last ~5 significant digits of wei are silently discarded.

**Concrete example (ckETH, 18 decimals):**
- Actual amount: `100_000_000_000_000_012_345` wei (100 ETH + 12345 wei)
- After `to_f64()`: `100000000000000000000.0` (rounds to nearest f64)
- Displayed: `"100"` ckETH — the 12345 wei difference is invisible to the user

Both call into `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints` / `build_icrc21_consent_info`, which ultimately calls `convert_tokens_to_string_representation` for the `GenericDisplayMessage` path. [8](#0-7) 

---

### Impact Explanation

ICRC-21 consent messages are the mechanism by which hardware wallets and signing devices show users what they are about to approve. If the displayed amount is incorrect, a user may unknowingly approve a transfer or allowance for a different amount than shown. For ckETH and any 18-decimal ICRC-1 token, amounts above ~9 ETH will have their sub-ETH precision silently truncated in the `GenericDisplayMessage` path. The actual on-chain transaction is unaffected, but the consent message — the user's only verification surface — is wrong. This is a **boundary/API validation bypass** (incorrect data returned to a caller relying on the endpoint for security decisions) and a **ledger conservation display bug** (amounts shown to users do not match amounts being authorized).

---

### Likelihood Explanation

Any unprivileged ingress sender can call `icrc21_canister_call_consent_message` on any production ICRC-1 ledger with a large token amount in the argument. No special role, key, or privilege is required. The ckETH ledger (18 decimals) and all ckERC20 ledgers (ckUSDC, ckUSDT — 6 decimals, but still affected for very large amounts) are all deployed on mainnet and expose this endpoint. Wallets implementing ICRC-21 (e.g., hardware wallet integrations) will display the rounded value to users.

---

### Recommendation

Replace the `f64` intermediate with exact integer arithmetic. The correct approach mirrors what the ckBTC/ckDOGE minters already do in their own `format_amount` helpers:

```rust
// rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs, lines 211-221
pub(super) fn format_amount(amount: u64, decimals: u8) -> String {
    let divisor = 10_u64.pow(decimals as u32);
    let whole = amount / divisor;
    let frac = amount % divisor;
    if frac == 0 {
        format!("{whole}")
    } else {
        let frac_str = format!("{frac:0width$}", width = decimals as usize);
        let trimmed = frac_str.trim_end_matches('0');
        format!("{whole}.{trimmed}")
    }
}
``` [9](#0-8) 

`convert_tokens_to_string_representation` should be rewritten to use `BigUint` integer division and modulo (since `tokens` is already a `Nat`/`BigUint`) rather than converting to `f64`, eliminating all precision loss regardless of token decimals or amount magnitude.

1. Call `icrc21_canister_call_consent_message` on the ckETH ICRC-1 ledger with a `GenericDisplay` preference and an `icrc1_transfer` argument where `amount = 100_000_000_000_000_012_345` (100 ETH + 12345 wei).
2. Observe the returned `GenericDisplayMessage` string contains `**Amount:** \`100 ckETH\`` — the 12345 wei is silently dropped.
3. The actual ledger transfer, if submitted, would move the full `100_000_000_000_000_012_345` wei, but the user's consent message showed only `100`.

#[update]
fn icrc21_canister_call_consent_message(
    consent_msg_request: ConsentMessageRequest,
) -> Result<ConsentInfo, Icrc21Error> {
    let caller_principal = caller();
    let ledger_fee = Nat::from(LEDGER.read().unwrap().transfer_fee.get_e8s());
    let token_symbol = LEDGER.read().unwrap().token_symbol.clone();
    let token_name = LEDGER.read().unwrap().token_name.clone();
    let decimals = ic_ledger_core::tokens::DECIMAL_PLACES as u8;

    if consent_msg_request.method == "transfer" {
        let TransferArgs {
            memo,
            amount,
            fee,
            from_subaccount,
            to,
            created_at_time: _,
        } = Decode!(&consent_msg_request.arg, TransferArgs).map_err(|e| {
            Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
                description: format!("Failed to decode TransferArgs: {e}"),
            })
        })?;
        icrc21_check_fee(&Some(Nat::from(fee)), &ledger_fee)?;
        let from = if caller() == Principal::anonymous() {
            AccountOrId::AccountIdAddress(None)
        } else {
            let account = Account {
                owner: caller(),
                subaccount: from_subaccount.map(|sa| sa.0),
            };
            AccountOrId::AccountIdAddress(Some(AccountIdentifier::from(account).to_hex()))
        };
        let receiver = AccountIdentifier::from_slice(&to).map_err(|e| {
            Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
                description: format!("Failed to parse receiver account id: {e}"),
            })
        })?;
        let args = GenericTransferArgs {
            from,
            receiver: AccountOrId::AccountIdAddress(Some(receiver.to_hex())),
            amount: Nat::from(amount.get_e8s()),
            memo: Some(GenericMemo::IntMemo(memo.0)),
        };
        build_icrc21_consent_info(
            consent_msg_request,
            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
            Some(args),
        )
    } else {
        build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints(
            consent_msg_request,
            caller_principal,
            ledger_fee,
            token_symbol,
            token_name,
            decimals,
        )
    }
}
