### Title
Redundant Double-Write of Canister Log Delta to Both `log_memory_store` and `canister_log` After Migration - (File: rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs)

### Summary
In `SystemStateModifications::apply_changes`, when `log_memory_store.is_migrated()` is `true`, the same execution-round log delta is unconditionally written to **both** the new `LogMemoryStore` and the legacy `CanisterLog`. This is the IC analog of the reported double-wrapping pattern: two storage layers receive the same data simultaneously, causing double memory consumption and incorrect resource accounting.

### Finding Description
In `apply_changes` at lines 393–405 of `rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs`:

```rust
// Append delta logs.
if system_state.log_memory_store.is_migrated() {
    system_state
        .log_memory_store
        .append_delta_log(&mut self.canister_log.clone()); // ← writes clone to new store
}
system_state
    .canister_log
    .append_delta_log(&mut self.canister_log); // ← writes original to legacy store (unconditional)
``` [1](#0-0) 

When `is_migrated()` is `true`, the code:
1. Clones `self.canister_log` and drains its records into `log_memory_store` via `LogMemoryStore::append_delta_log`, which calls `delta_log.records_mut().drain(..)`.
2. Then unconditionally drains the **original** `self.canister_log` records into the legacy `system_state.canister_log` via `CanisterLog::append_delta_log`. [2](#0-1) [3](#0-2) 

Both stores receive the identical set of log records from the same execution round. The `canister_log` write is not gated on `!is_migrated()`, so after migration the legacy store continues to accumulate records that are already persisted in `log_memory_store`. The code's own comment acknowledges this transitional state: `// TODO(DSM-11): Move this into append_delta_log() once there is only one of it.` [4](#0-3)

### Impact Explanation
**Cycles/resource accounting bug.** Both `canister_log` and `log_memory_store` are part of the canister's replicated state and their memory usage is charged to the canister. After migration, every log record emitted by a canister is stored twice: once in the new `LogMemoryStore` (backed by a `PageMap` on disk) and once in the legacy `CanisterLog` (in-memory). This causes:

1. **Double memory consumption**: The canister's effective log storage cost is doubled for every round that produces log output, inflating the canister's reported memory usage and the cycles charged for it.
2. **Incorrect memory limit enforcement**: Canisters with a `log_memory_limit` setting may exhaust their allocation at half the expected log volume, because the same bytes are counted in two stores.
3. **Log state inconsistency**: `fetch_canister_logs` callers reading from `canister_log` after migration see a store that is still being populated, while `log_memory_store` is the authoritative source, creating divergent views of the log history.

### Likelihood Explanation
The `log_memory_store` migration is controlled by a `FlagStatus` feature flag. Once enabled on a subnet and a canister's migration flag is set (`is_migrated() == true`), **every single update call or task that emits at least one log record** triggers the double-write. Any canister developer can emit logs via `ic0.debug_print`. No special privilege is required. The entry path is fully attacker-controlled through normal canister execution.

### Recommendation
Gate the legacy `canister_log.append_delta_log` call on `!system_state.log_memory_store.is_migrated()`, so that after migration only `log_memory_store` receives the delta:

```rust
if system_state.log_memory_store.is_migrated() {
    system_state.log_memory_store.append_delta_log(&mut self.canister_log);
} else {
    system_state.canister_log.append_delta_log(&mut self.canister_log);
}
```

This ensures exactly one storage layer receives each log delta, eliminating the double-write and restoring correct memory accounting.

### Proof of Concept
1. Enable the `log_memory_store` feature flag on a subnet.
2. Allow a canister's migration to complete (`is_migrated()` returns `true`).
3. Execute any update call on that canister that calls `ic0.debug_print` (or any logging API).
4. After the round, inspect `system_state.canister_log` and `system_state.log_memory_store`: both contain the same log records from that round.
5. Observe that the canister's total memory usage (as reported by `canister_status`) reflects both stores, charging the canister for the same log bytes twice.
