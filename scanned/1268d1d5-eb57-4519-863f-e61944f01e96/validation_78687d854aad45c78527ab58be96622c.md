### Title
Partial-Window Price Average Computes Maturity Modulation from Insufficient Observations, Skewing ICP Minting - (File: rs/nns/governance/src/timer_tasks/update_icp_xdr_rate_related_data.rs)

### Summary

`compute_average_icp_xdr_rate` in the NNS Governance canister computes a moving average over however many days are available in the buffer, without requiring a minimum number of contributing observations. When the price history is sparse — due to XRC fetch failures during the initial backfill phase or persistent gaps — the 7-day "current price" or 365-day "reference price" window can be computed from as few as one day's data. This is the direct IC analog of the TWAP fallback-to-oldest-observation bug: a single observation dominates the average, producing a skewed maturity modulation that is then applied to ICP minting in `maybe_spawn_neurons` and `finalize_maturity_disbursement`.

### Finding Description

`compute_average_icp_xdr_rate` uses Last Observation Carried Forward (LOCF) to fill gaps within the window, and only returns `None` when `count == 0`:

```rust

    println!(...); // warning only — does NOT abort
}
if count == 0 {
    return None;
}
Some((sum / count as u128) as u64)

When `count >= 1` but `count < window_days`, the function returns a value. `compute_maturity_modulation_permyriad` treats this as a valid full-window average and proceeds to compute the modulation:

```rust

    rates, current_day, MATURITY_MODULATION_CURRENT_ICP_PRICE_WINDOW_DAYS,
).ok_or_else(|| "no rate available for the recent price window".to_string())?;

let reference_icp_price = compute_average_icp_xdr_rate(
    rates, current_day, MATURITY_MODULATION_REFERENCE_ICP_PRICE_WINDOW_DAYS,
).ok_or_else(|| "no rate available for the reference price window".to_string())?;
``` [2](#0-1) 

The critical scenario: if XRC fails for 6 of the last 7 days but succeeds for the remaining 358 days, the 7-day window has only 1 observation (a potentially anomalous day), while the 365-day window has 358 observations. The single-day `recent_price` is compared against the stable `reference_price`, producing a large `target_modulation`. On the **first calculation** (`previous = None`), the speed limit is bypassed entirely:

```rust

    None => target_modulation, // speed limit skipped on first calculation
    ...

### Impact Explanation

Incorrect maturity modulation directly affects the amount of ICP minted when neurons spawn or disburse maturity. The global bounds are [-1000, +200] permyriad (−10% to +2%): [8](#0-7) 

A modulation incorrectly driven to +200 permyriad causes 2% excess ICP to be minted for every spawning/disbursement event until the next daily recalculation. A modulation incorrectly driven to −1000 permyriad causes 10% less ICP to be minted, harming neuron owners. Both outcomes violate ledger conservation: the total ICP supply is inflated or deflated relative to the intended economic model. The first-calculation speed-limit bypass amplifies the impact because the modulation can jump to the extreme value in a single step rather than approaching it gradually at 30 permyriad/day.

### Likelihood Explanation

The NNS Governance changelog explicitly records XRC failure handling as a recurring concern: [9](#0-8) 

The backfill phase (~30 minutes at 5 s/call for 365 days) is a predictable window after every canister upgrade where partial history is normal. An unprivileged neuron owner can observe the sparse-history state by querying `get_maturity_modulation` and checking whether `updated_at_timestamp_seconds` is `None` (indicating the first calculation has not yet run) or whether the value was just set with a suspiciously extreme value. They can then time a `SpawnNeuron` or `DisburseMaturity` call to exploit the incorrect modulation. No privileged access is required; the `spawn_neuron` and `disburse_maturity` endpoints are open to any neuron controller.

### Recommendation

1. **Enforce a minimum observation count** in `compute_average_icp_xdr_rate`: return `None` (or a distinct error) when `count < window_days`, rather than only logging a warning. This mirrors the recommendation in the external report to revert when the oldest observation is less than the requested window.
2. **Alternatively, require a minimum fill ratio** (e.g., at least 50% of the window days must have actual observations, not just LOCF-filled values) before the result is considered valid for modulation computation.
3. **Gate the first-calculation speed-limit bypass** on having a fully populated history (i.e., `count == window_days` for both windows), so that the jump-to-target behavior only occurs when the data is trustworthy.

### Proof of Concept

The existing unit test `test_compute_average_icp_xdr_rate_skips_days_with_no_prior_rate` already demonstrates the partial-window behavior:

```rust
// 7-day window ending at day 100. The earliest rate is day 96.
// Days 94, 95 have no prior rate → skipped. Days 96-100 use day 96's rate via LOCF.
// Average = 80_000 over only 5 contributing days — returned as Some(80_000).
let rates = vec![SampledPrice {
    timestamp_seconds: 96 * ONE_DAY_SECONDS,
    xdr_permyriad_per_icp: 80_000,
}];
assert_eq!(compute_average_icp_xdr_rate(&rates, 100, 7), Some(80_000));
``` [10](#0-9) 

Exploit path:
1. Governance canister is upgraded; `icp_price_history` is empty; `maturity_modulation.updated_at_days_since_epoch = None`.
2. Backfill runs; XRC is intermittently unavailable. After all 365 attempts, the buffer contains, e.g., 358 stable-price days (days 1–358, rate = 50,000) and 1 recent spike day (day 365, rate = 100,000); days 359–364 are missing.
3. `get_day_to_fetch` returns `None`; `update_maturity_modulation` is called with `previous = None`.
4. `compute_average_icp_xdr_rate` for the 7-day window (days 359–365): only day 365 contributes → `recent_price = 100,000`.
5. `compute_average_icp_xdr_rate` for the 365-day window: days 1–358 contribute 50,000; days 359–364 are skipped (no prior rate); day 365 contributes 100,000 → `reference_price ≈ 50,137`.
6. `target_modulation = 2500 * (100,000 − 50,137) / 50,137 ≈ 2,487 permyriad`; clamped to `MAX = 200 permyriad`.
7. Speed limit bypassed (first calculation); `maturity_modulation.current_value_permyriad = 200`.
8. Neuron owner calls `SpawnNeuron` or `DisburseMaturity`; receives 2% excess ICP.

```rust
pub fn apply_maturity_modulation(
    amount_maturity_e8s: u64,
    maturity_modulation_basis_points: i32,
) -> Result<u64, String> {
    let amount_e8s = u128::from(amount_maturity_e8s);

    let adjusted_maturity_modulation_basis_points = saturating_add_or_subtract_u128_i32(
        BASIS_POINTS_PER_UNITY,
        maturity_modulation_basis_points,
    );

    let modulated_amount_e8s: u128 = amount_e8s
        .checked_mul(adjusted_maturity_modulation_basis_points)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?
        .checked_div(BASIS_POINTS_PER_UNITY)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?;

    u64::try_from(modulated_amount_e8s).map_err(|err| err.to_string())
}
