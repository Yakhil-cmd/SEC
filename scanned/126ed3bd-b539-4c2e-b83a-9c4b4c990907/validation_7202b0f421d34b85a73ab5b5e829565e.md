### Title
SNS Token Valuation Mixes Local Swap Supply with Global Initial Supply in Exchange Rate Calculation - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

### Summary
`IcpsPerSnsTokenClient::fetch_icps_per_sns_token` computes the current ICP-per-SNS-token price by dividing a swap-derived initial price (denominated in swap tokens only) by an inflation factor (denominated in the total initial supply). This mixes a local quantity (swap tokens) with a global quantity (total initial supply), producing a valuation inflated by the ratio `total_initial_supply / swap_tokens`. The inflated valuation is then used to enforce the 7-day treasury transfer and minting limits for `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals.

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, `fetch_icps_per_sns_token` concurrently fetches three values and combines them: [1](#0-0) 

1. `initial_sns_tokens_per_icp` — from `swap.get_derived_state()`, computed inside the swap canister as: [2](#0-1) 

   This equals `swap_tokens_available / buyer_total_icp_e8s`, i.e., it is denominated in **swap tokens only** (a subset of the total initial supply).

2. `initial_supply_e8s` — from `initial_supply_e8s()`, which scans the SNS ledger and sums **all** mint transactions at the first timestamp (developer, airdrop, treasury, and swap tokens combined): [3](#0-2) 

3. `current_supply_e8s` — from `icrc1_total_supply`.

The formula assembled is: [4](#0-3) 

```
icps_per_sns_token
  = (1 / initial_sns_tokens_per_icp) / (current_supply / initial_supply)
  = (buyer_icp / swap_tokens) / (current_supply / total_initial_supply)
  = buyer_icp × total_initial_supply / (swap_tokens × current_supply)
```

The correct formula is simply `buyer_icp / current_supply`. The current code uses `swap_tokens` as the denominator of the initial price but `total_initial_supply` as the denominator of the inflation factor. Because `total_initial_supply > swap_tokens` always (developer, airdrop, and treasury tokens are also minted at genesis), the valuation is inflated by the factor `total_initial_supply / swap_tokens`.

This inflated valuation is stored in the proposal's `action_auxiliary` at submission time and reused at execution time: [5](#0-4) [6](#0-5) [7](#0-6) 

The limit computation in `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` uses this valuation to classify the treasury as small/medium/large and derive the maximum transferable tokens: [8](#0-7) 

### Impact Explanation

**Impact: Medium.** The treasury valuation is inflated by `total_initial_supply / swap_tokens`. For a typical SNS where swap tokens represent 20–50% of the total initial supply, the valuation is 2–5× too high. This pushes the treasury into a higher tier (medium → large, or small → medium), making the 7-day transfer and minting limits more restrictive than intended. Legitimate `TransferSnsTreasuryFunds` and `MintSnsTokens` governance proposals are incorrectly rejected at both submission and execution time, constituting a governance denial-of-service. No direct financial loss occurs (the treasury cannot be drained), but SNS DAOs lose the ability to execute legitimate treasury operations within the intended bounds.

### Likelihood Explanation

**Likelihood: High.** Every SNS that has developer, airdrop, or treasury tokens minted at genesis (i.e., virtually all SNS deployments) is affected. The swap tokens are always a strict subset of the total initial supply, so the inflation factor is always wrong. The bug is triggered by any SNS governance participant submitting a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal — no special privilege is required.

### Recommendation

Replace the two-step formula with the correct direct formula. Instead of:

```rust
let initial_icps_per_sns_token = 1 / initial_sns_tokens_per_icp;
let total_inflation = current_supply / initial_supply;
initial_icps_per_sns_token / total_inflation
```

Use a consistent denominator. The simplest correct form is:

```rust
// buyer_icp / current_supply
Decimal::from(buyer_total_icp_e8s) / current_supply_e8s
```

Alternatively, if the inflation-based framing is preferred, use the swap token supply as the baseline for both the initial price and the inflation factor:

```rust
let initial_icps_per_sns_token = 1 / initial_sns_tokens_per_icp;  // buyer_icp / swap_tokens
let inflation = current_supply / swap_tokens;                       // consistent denominator
initial_icps_per_sns_token / inflation                              // = buyer_icp / current_supply
```

### Proof of Concept

**Setup:**
- Total initial supply: 1,000,000 SNS tokens (developer 400k + treasury 400k + swap 200k)
- Swap tokens: 200,000
- ICP raised in swap: 100,000 ICP
- Current supply (after 10% inflation): 1,100,000 SNS tokens

**Correct price:** `100,000 / 1,100,000 ≈ 0.0909 ICP per token`

**Current formula:**
- `initial_sns_tokens_per_icp = 200,000 / 100,000 = 2`
- `initial_icps_per_sns_token = 1 / 2 = 0.5`
- `total_inflation = 1,100,000 / 1,000,000 = 1.1`
- `icps_per_sns_token = 0.5 / 1.1 ≈ 0.4545 ICP per token` **(5× too high)**

**Consequence:** The treasury valuation is inflated 5×. A treasury holding 10,000 SNS tokens appears to be worth `10,000 × 0.4545 × xdrs_per_icp` XDR instead of the correct `10,000 × 0.0909 × xdrs_per_icp` XDR. This pushes the treasury into a higher tier, making the 7-day transfer limit more restrictive than the SNS community intended, and causing legitimate proposals to be rejected.

    let spent_tokens = action.recent_amount_total_tokens(proposals, env.now())?;

    // Get valuation of the tokens in the treasury.

    let valuation = assess_treasury_balance(
        token,
        env.canister_id(),
        sns_ledger_canister_id,
        swap_canister_id,
    )
    .await?;

    // From valuation, determine limit on the total from the past 7 days.
    let max_tokens = MyTokenProposalAction::recent_amount_total_upper_bound_tokens(&valuation)
        // Err is most likely a bug.

            format!("Unable to validate amount: {treasury_limit_error:?}",)

    // Finally, inspect the proposal's amount: it must not exceed max - spent (remainder). Or if
    // you prefer, equivalently, amount + spent must be <= max.
