### Title
Post-Upgrade Heap Reset of `API_BOUNDARY_NODE_PRINCIPALS` Causes Complete Denial of Salt Access to All Boundary Nodes — (`rs/boundary_node/salt_sharing/canister/storage.rs`, `canister.rs`, `helpers.rs`)

---

### Summary

Every canister upgrade resets `API_BOUNDARY_NODE_PRINCIPALS` to an empty `HashSet` because it is stored in heap-only `thread_local!` memory. Both `inspect_message` (for update-path calls) and `get_salt` itself (for query-path calls) gate access on this set. The first repopulation only occurs after `registry_polling_interval_secs` (300 s in production) elapses and the inter-canister call to the registry completes. During this window every API boundary node — regardless of prior authorization — is denied salt access.

---

### Finding Description

**Root cause 1 — heap-only principal store:**

`API_BOUNDARY_NODE_PRINCIPALS` is declared as a `thread_local!` `RefCell<HashSet<Principal>>` initialized to `HashSet::new()`. [1](#0-0) 

On every Wasm upgrade the IC discards heap memory and re-executes the `thread_local!` initializer, so the set is empty the instant `post_upgrade` returns.

**Root cause 2 — `post_upgrade` defers all async work:**

`post_upgrade` delegates to `init`, which schedules `init_async` via `set_timer(Duration::ZERO, …)`. [2](#0-1) [3](#0-2) 

**Root cause 3 — `init_async` never calls `poll_api_boundary_nodes()` eagerly:**

`init_async` only registers a periodic timer; the first actual poll fires after a full `registry_polling_interval_secs` interval (300 s in production). [4](#0-3) 

The production upgrade argument confirms `registry_polling_interval_secs = 300`. [5](#0-4) 

**Root cause 4 — both access paths check the empty set:**

`inspect_message` (update path): [6](#0-5) 

`get_salt` (query path): [7](#0-6) 

Both call `is_api_boundary_node_principal`, which reads from the empty set: [8](#0-7) 

---

### Impact Explanation

For at least 300 seconds after every governance-approved upgrade, every API boundary node receives either an `inspect_message` trap (update path) or `GetSaltError::Unauthorized` (query path). Boundary nodes that exhaust their locally cached salt during this window cannot anonymize sensitive data (e.g., IP addresses) consistently, breaking the core invariant the canister was designed to enforce.

---

### Likelihood Explanation

The condition is triggered deterministically on every canister upgrade. The production upgrade proposal already demonstrates `regenerate_now = false` and `registry_polling_interval_secs = 300`, confirming the 300 s minimum blackout window is the real-world deployment configuration. No adversarial action is required beyond the normal governance upgrade lifecycle.

---

### Recommendation

Either:

1. **Persist `API_BOUNDARY_NODE_PRINCIPALS` in stable memory** (e.g., a `StableBTreeMap` under a new `MemoryId`) so it survives upgrades, or
2. **Call `poll_api_boundary_nodes()` eagerly inside `init_async`** before registering the interval timer, so the set is populated within the first execution round after upgrade rather than after a full polling interval.

Option 1 is preferable because it also survives registry call failures after upgrade.

---

### Proof of Concept

```
1. Install canister with regenerate_now=true, registry_polling_interval_secs=300.
2. Tick 100 rounds → salt populated, API_BOUNDARY_NODE_PRINCIPALS populated with known principal P.
3. Upgrade canister with regenerate_now=false (as in production).
4. Immediately (0 ticks) call get_salt as principal P via update or query.
5. Update path → inspect_message traps with "message_inspection_failed".
   Query path → returns Err(GetSaltError::Unauthorized).
6. Tick 0 rounds (timer fires but inter-canister call hasn't returned) → same result.
7. Tick until poll_api_boundary_nodes() completes → call succeeds.
```

The existing PocketIc test harness in `rs/boundary_node/salt_sharing/integration_tests/` already has all the scaffolding needed to reproduce this in a state-machine test. [9](#0-8)
