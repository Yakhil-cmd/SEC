### Title
ckETH Minter Accumulates Unspent Transaction Fees With No Withdrawal Mechanism - (`rs/ethereum/cketh/minter/src/state.rs`)

---

### Summary

The ckETH minter charges users a conservative `max_tx_fee_estimate` upfront for every `withdraw_eth` and `withdraw_erc20` call. After each withdrawal finalizes on Ethereum, the difference `max_tx_fee_estimate − actual_tx_fee` (the "unspent transaction fee") is credited to an internal accumulator `total_unspent_tx_fees` in the minter's `EthBalance` state. No endpoint exists in the minter to withdraw, redistribute, or reimburse these accumulated ETH funds. They are permanently locked in the minter's Ethereum address until a canister upgrade adds such functionality.

---

### Finding Description

Every ckETH or ckERC20 withdrawal burns the full `withdrawal_amount` of ckETH from the user, but only sends `withdrawal_amount − max_tx_fee_estimate` ETH to the destination address. When the Ethereum transaction finalizes, `update_balance_upon_withdrawal` computes:

```
unspent_tx_fee = charged_tx_fee − actual_tx_fee
```

and calls `total_unspent_tx_fees_add(unspent_tx_fee)`. [1](#0-0) 

The `EthBalance` struct tracks three fields: `eth_balance`, `total_effective_tx_fees`, and `total_unspent_tx_fees`. The `total_unspent_tx_fees` field only ever grows — there is no corresponding subtraction path. [2](#0-1) 

Searching all `#[update]` endpoints in `main.rs` reveals no function that withdraws, redistributes, or reimburses the accumulated unspent fees. The dashboard and metrics expose the value for observability only. [3](#0-2) 

The test suite explicitly encodes the design intent that unused ckERC20 transaction fees are **not** reimbursed: [4](#0-3) 

The documentation confirms: *"Overcharged transaction fees are not reimbursed."* [5](#0-4) 

---

### Impact Explanation

Every finalized ckETH or ckERC20 withdrawal leaves a residual ETH amount (`max_tx_fee_estimate − actual_tx_fee`) in the minter's Ethereum address that is not backed by any outstanding ckETH supply (the corresponding ckETH was already burned). This ETH is permanently inaccessible: there is no governance proposal type, no canister endpoint, and no automated process that can move or mint against it. The funds are locked until a canister upgrade introduces a withdrawal mechanism. Over time, as withdrawal volume grows, the locked ETH value compounds. This is a **chain-fusion ledger conservation bug**: ETH value is collected from users (via ckETH burns) but cannot be recovered or redistributed.

---

### Likelihood Explanation

This condition is triggered by every single successful `withdraw_eth` or `withdraw_erc20` call made by any unprivileged user. Because Ethereum gas estimation must be conservative to handle resubmissions, `actual_tx_fee < max_tx_fee_estimate` is the normal case for the vast majority of withdrawals. The accumulation is continuous and automatic, requiring no special attacker action. [6](#0-5) 

---

### Recommendation

Add a governance-gated endpoint (callable only by the NNS or a designated controller) that mints ckETH proportional to the accumulated `total_unspent_tx_fees` balance to a treasury account, or alternatively reimburses individual users by recording per-withdrawal unspent amounts and providing a `claim_fee_refund` endpoint. At minimum, the minter should expose a mechanism to sweep these funds rather than leaving them permanently stranded.

---

### Proof of Concept

1. User calls `withdraw_eth` with `withdrawal_amount = 39_998_000_000_000_000` wei (as in the documented example).
2. Minter burns `39_998_000_000_000_000` ckETH from the user.
3. Minter estimates `max_tx_fee_estimate = 1_823_126_598_888_000` wei and sends `38_174_873_401_112_000` wei to the destination.
4. Ethereum transaction finalizes with `actual_tx_fee = 899_399_014_248_000` wei.
5. `unspent_tx_fee = 1_823_126_598_888_000 − 899_399_014_248_000 = 923_727_584_640_000` wei is added to `total_unspent_tx_fees`.
6. No ckETH is minted back; no ETH is returned. The `923_727_584_640_000` wei (~0.00092 ETH) is permanently locked.
7. Repeat for every withdrawal. After 1,000 withdrawals at similar gas conditions, ~0.92 ETH is locked with no recovery path. [7](#0-6) [8](#0-7)

#[update]
async fn withdraw_eth(
    WithdrawalArg {
        amount,
        recipient,
        from_subaccount,
    }: WithdrawalArg,
) -> Result<RetrieveEthRequest, WithdrawalError> {
    let caller = validate_caller_not_anonymous();
    let _guard = retrieve_withdraw_guard(caller).unwrap_or_else(|e| {
        ic_cdk::trap(format!(
            "Failed retrieving guard for principal {caller}: {e:?}"
        ))
    });

    let destination = validate_address_as_destination(&recipient).map_err(|e| match e {
        AddressValidationError::Invalid { .. } | AddressValidationError::NotSupported(_) => {
            ic_cdk::trap(e.to_string())
        }
        AddressValidationError::Blocked(address) => WithdrawalError::RecipientAddressBlocked {
            address: address.to_string(),
        },
    })?;

    let amount = Wei::try_from(amount).expect("failed to convert Nat to u256");

    let minimum_withdrawal_amount = read_state(|s| s.cketh_minimum_withdrawal_amount);
    if amount < minimum_withdrawal_amount {
        return Err(WithdrawalError::AmountTooLow {
            min_withdrawal_amount: minimum_withdrawal_amount.into(),
        });
    }

    let client = read_state(LedgerClient::cketh_ledger_from_state);
    let now = ic_cdk::api::time();
    log!(INFO, "[withdraw]: burning {:?}", amount);
    match client
        .burn_from(
            Account {
                owner: caller,
                subaccount: from_subaccount,
            },
            amount,
            BurnMemo::Convert {
                to_address: destination,
            },
        )
        .await
    {
        Ok(ledger_burn_index) => {
            let withdrawal_request = EthWithdrawalRequest {
                withdrawal_amount: amount,
                destination,
                ledger_burn_index,
                from: caller,
                from_subaccount: from_subaccount.and_then(LedgerSubaccount::from_bytes),
                created_at: Some(now),
            };

            log!(
                INFO,
                "[withdraw]: queuing withdrawal request {:?}",
                withdrawal_request,
            );

            mutate_state(|s| {
                process_event(
                    s,
                    EventType::AcceptedEthWithdrawalRequest(withdrawal_request.clone()),
                );
            });
            Ok(RetrieveEthRequest::from(withdrawal_request))
        }
        Err(e) => Err(WithdrawalError::from(e)),
    }
}

.Effective transaction fees vs unspent transaction fees

The minter dashboard displays in the metadata table the following fees

. `Total effective transaction fees`: the sum of all `actual_tx_fee` for all withdrawals.
