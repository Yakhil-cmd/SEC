### Title
Unescaped `token_name` Enables Markdown Injection in ICRC-21 `GenericDisplayMessage` Consent Messages - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The ICRC-21 consent message builder in `packages/icrc-ledger-types/src/icrc21/responses.rs` interpolates the ledger-configured `token_name` and `token_symbol` strings directly into a Markdown `GenericDisplayMessage` without sanitizing Markdown structural characters (newlines, `#`, backticks, `**`). A malicious canister developer who deploys an ICRC-1-compatible ledger with a crafted `token_name` can inject fake Markdown fields into the consent message shown to users by wallets, deceiving them into approving transactions they did not intend to authorize. The ckBTC and ckDOGE minters explicitly guard against the analogous injection vector in the address field, but the shared ICRC-21 library has no equivalent guard for `token_name` or `token_symbol`.

---

### Finding Description

**Root cause — `add_intent` in `packages/icrc-ledger-types/src/icrc21/responses.rs`:**

```rust
// line 53
message.push_str(&format!("# Send {}", token_name.unwrap()));
// line 65
message.push_str(&format!("# Spend {}", token_name.unwrap()));
```

`token_name` is taken verbatim from the ledger's initialization state and concatenated into the Markdown string. No character is escaped or rejected. The same pattern repeats for `token_symbol` in `add_amount` (line 114), `add_fee` (lines 144/149), `add_allowance` (line 188), and `add_existing_allowance` (line 213). [1](#0-0) [2](#0-1) 

**Contrast with the explicit guard in the ckBTC minter:**

The ckBTC minter's `validate_address` function explicitly rejects any address that would break the Markdown structure before it is interpolated, with a comment that directly names the Markdown-injection risk:

```rust
/// rules out Markdown-injection vectors in the GenericDisplay output
/// (e.g. an "address" that contains newlines or backticks crafted to
/// fake additional fields).
fn validate_address(address: &str, network: Network) -> Result<(), Icrc21Error> {
    BitcoinAddress::parse(address, network).map_err(...)
}
``` [3](#0-2) 

No equivalent guard exists for `token_name` or `token_symbol` in `build_icrc21_consent_info` or `ConsentMessageBuilder`. [4](#0-3) 

**`token_name` is set at ledger initialization and is fully controlled by the canister developer:** [5](#0-4) 

The ICRC-1 ledger `UpgradeArgs` also allows `token_name` and `token_symbol` to be changed post-deployment by the controller: [6](#0-5) 

**Secondary surface — `add_memo`:**

The memo field is also interpolated without sanitization:

```rust
// line 294
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

A memo containing a backtick (valid UTF-8) closes the code span and allows injection of additional Markdown fields. The memo is caller-supplied via the encoded `TransferArg` in the `ConsentMessageRequest`. [7](#0-6) 

---

### Impact Explanation

The `GenericDisplayMessage` is a Markdown string returned to wallets (e.g., Ledger hardware wallet, NNS dapp) that render it as the human-readable description of what the user is about to sign. If a malicious canister developer sets:

```
token_name = "ICP\n\n**Amount:** 1000000 ICP\n\n**To:**\n`attacker_address`"
```

the rendered consent message becomes:

```
# Send ICP

**Amount:** 1000000 ICP

**To:**
`attacker_address`
[... actual fields follow ...]
```

The injected fields appear visually identical to legitimate fields. A user reviewing the consent message in a wallet would be deceived into believing they are sending 1,000,000 ICP to `attacker_address`, when the actual encoded transfer may be for a different amount or recipient. This is a direct consent-spoofing / phishing vector at the protocol level, undermining the security guarantee that ICRC-21 is designed to provide.

---

### Likelihood Explanation

Any developer can deploy an ICRC-1 ledger on the IC with arbitrary `token_name` and `token_symbol`. No privileged access, governance majority, or key compromise is required. The attack is triggered the moment a wallet calls `icrc21_canister_call_consent_message` on the malicious ledger — a standard wallet operation. The memo injection surface is reachable by any unprivileged caller of the `icrc21_canister_call_consent_message` update endpoint.

---

### Recommendation

1. **Sanitize `token_name` and `token_symbol`** before interpolation: reject or strip characters that have structural meaning in Markdown (`\n`, `\r`, `` ` ``, `#`, `*`, `_`, `[`, `]`). Apply the same philosophy already used in `validate_address` for the ckBTC/ckDOGE minters.
2. **Sanitize the memo string** before interpolating it into the backtick-delimited code span; at minimum, reject or escape any backtick characters in the UTF-8 memo path.
3. Consider validating `token_name` and `token_symbol` at ledger initialization time (length, character set) so that the invariant is enforced at the source rather than at every call site.

---

### Proof of Concept

**Step 1 — Deploy a malicious ICRC-1 ledger** with:
```
token_name = "ICP\n\n**Amount:** 1000000 ICP\n\n**To:**\n`rrkah-fqaaa-aaaaa-aaaaq-cai`"
token_symbol = "ICP`\n\n**Fees:** 0 ICP\n`"
```

**Step 2 — Call `icrc21_canister_call_consent_message`** with a `ConsentMessageRequest` for `icrc1_transfer`, `GenericDisplay`.

**Step 3 — Observe the returned `GenericDisplayMessage`:**

```
# Send ICP

**Amount:** 1000000 ICP

**To:**
`rrkah-fqaaa-aaaaa-aaaaq-cai`

You are approving a transfer of funds from your account.

**From:**
`<actual sender>`

**Amount:** `<actual amount> ICP`

**Fees:** 0 ICP
`
...
```

The injected lines are visually indistinguishable from the legitimate fields generated by `add_amount` and `add_account`. A wallet rendering this Markdown would show the attacker-controlled values to the user before the legitimate values, causing the user to approve a transaction under false pretenses. /// Verifies that `address` parses as a valid Bitcoin address on the configured
/// network before it gets interpolated into a consent message. This both
/// guarantees the user is shown a meaningful (parseable) destination and rules
/// out Markdown-injection vectors in the GenericDisplay output (e.g. an
/// "address" that contains newlines or backticks crafted to fake additional
/// fields). Uses the same parser as `retrieve_btc_with_approval`, so any
/// address the consent endpoint accepts is also accepted by the actual call.
fn validate_address(address: &str, network: Network) -> Result<(), Icrc21Error> {
    BitcoinAddress::parse(address, network).map_err(|e| {
        Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
            description: format!("Invalid Bitcoin destination address: {e}"),
        })
    })?;
    Ok(())
}
