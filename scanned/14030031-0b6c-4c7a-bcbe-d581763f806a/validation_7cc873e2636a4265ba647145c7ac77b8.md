### Title
SNS Treasury Valuation Uses Live Swap-Derived `sns_tokens_per_icp` Without Time-Weighted Averaging, Enabling Manipulation of 7-Day Treasury Transfer Limits - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

### Summary
The `fetch_icps_per_sns_token` function in the SNS governance token valuation module reads `sns_tokens_per_icp` from the SNS swap canister's `get_derived_state` endpoint — a live, dynamically computed ratio of `tokens_available_for_swap / participant_total_icp_e8s`. During the open swap phase, any unprivileged user can increase `participant_total_icp_e8s` by calling `refresh_buyer_tokens`, which decreases `sns_tokens_per_icp`, increases `icps_per_token`, and inflates the treasury valuation. This inflated valuation is captured at proposal submission time and stored in `ActionAuxiliary`, where it is later used at execution time to enforce the 7-day upper bound on `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals.

### Finding Description

**Root cause — live spot price used as oracle:**

`fetch_icps_per_sns_token` in `rs/sns/governance/token_valuation/src/lib.rs` calls `get_derived_state` on the SNS swap canister:

```rust
call::<_, MyRuntime>(self.swap_canister_id, GetDerivedStateRequest {})
``` [1](#0-0) 

The swap canister's `derived_state()` computes `sns_tokens_per_icp` as a live ratio:

```rust
let sns_tokens_per_icp = i2d(tokens_available_for_swap)
    .checked_div(i2d(participant_total_icp_e8s))
    .and_then(|d| d.to_f32())
    .unwrap_or(0.0);
``` [2](#0-1) 

This value changes in real time as swap participants add ICP via `refresh_buyer_tokens`. The proto comment itself warns: *"Current approximate rate SNS tokens per ICP. Note that this should not be used for super precise financial accounting, because this is floating point."* [3](#0-2) 

**How the manipulated value propagates to the limit:**

The returned `sns_tokens_per_icp` is inverted to `initial_icps_per_sns_token`, then divided by `total_inflation` (current supply / initial supply):

```rust
let initial_icps_per_sns_token = Decimal::from(1)
    .checked_div(initial_sns_tokens_per_icp)...;
let total_inflation = current_supply_e8s.checked_div(initial_supply_e8s)...;
initial_icps_per_sns_token.checked_div(total_inflation)
``` [4](#0-3) 

The resulting `icps_per_token` feeds into `ValuationFactors::to_xdr()`:

```rust
tokens * icps_per_token * xdrs_per_icp
``` [5](#0-4) 

This XDR valuation determines the 7-day transfer ceiling via `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` and `mint_sns_tokens_7_day_total_upper_bound_tokens`. [6](#0-5) 

**Valuation is locked in at proposal submission time:**

`treasury_valuation_if_proposal_amount_is_small_enough_or_err` calls `assess_treasury_balance` (which calls `fetch_icps_per_sns_token`) at proposal submission time and stores the result in `ActionAuxiliary::TransferSnsTreasuryFunds`: [7](#0-6) 

At execution time, `perform_transfer_sns_treasury_funds` retrieves the **stored** valuation and passes it to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`: [8](#0-7) 

The execution-time check uses the stored valuation — not a freshly computed one — so an inflated valuation captured at submission time persists through execution. [9](#0-8) 

**Manipulation vector:**

During the `LIFECYCLE_OPEN` phase, any user can call `refresh_buyer_tokens` to deposit ICP into the swap. Depositing a large amount of ICP increases `participant_total_icp_e8s`, decreasing `sns_tokens_per_icp`, increasing `icps_per_token`, and inflating the treasury valuation. If a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal is submitted while the swap is open, the inflated valuation is captured and stored.

**Uncertainty about `PreInitializationSwap` mode:**

The SNS governance operates in `PreInitializationSwap` mode during the open swap, which restricts certain operations (e.g., `start_dissolving_neuron` fails with a `PreInitializationSwap` error). Whether `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals are also blocked in this mode could not be confirmed from the available code. If they are blocked, the window of exploitability is limited to any gap between swap commitment and the SNS governance mode transitioning to `Normal`. If they are not blocked, the vulnerability is directly exploitable during the open swap.

### Impact Explanation

If exploitable, an attacker who is a swap participant can inflate the treasury valuation at proposal submission time, causing the 7-day upper bound to be computed against an artificially high valuation. For a "large" treasury (>1M XDR), the limit is 1M XDR worth of tokens per 7 days. By inflating `icps_per_token`, the attacker can make the treasury appear larger than it is, raising the ceiling and allowing a larger-than-intended transfer or mint to pass the limit check. This could result in unauthorized draining of the SNS treasury or excessive token minting, harming all SNS token holders.

### Likelihood Explanation

**Low-medium.** The attack requires: (1) the swap to be in the `LIFECYCLE_OPEN` state when a governance proposal is submitted; (2) `TransferSnsTreasuryFunds` or `MintSnsTokens` proposals to be submittable during `PreInitializationSwap` mode (uncertain); (3) the attacker to have sufficient voting power or collude with neuron holders to pass the proposal; (4) the attacker's ICP to be locked in the swap (no same-round reversal as in the EVM analog). The attacker does not lose their ICP (they receive SNS tokens or a refund), but the attack is multi-round and requires governance collusion, making it harder to execute than the original EVM exploit.

### Recommendation

- **Short term:** In `fetch_icps_per_sns_token`, do not use the live `sns_tokens_per_icp` from `get_derived_state`. Instead, read the finalized swap price from the swap canister's committed state (e.g., from `params.sns_token_e8s` and the final `buyer_total_icp_e8s` recorded at commitment time), or store the swap price in the SNS governance canister at the time the swap is finalized.
- **Long term:** Use a time-weighted average price (TWAP) or a price that is certified and snapshotted at a known, manipulation-resistant point in time (e.g., the block at which the swap was committed). Ensure that `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals are explicitly blocked during `PreInitializationSwap` mode.

async fn treasury_valuation_if_proposal_amount_is_small_enough_or_err<MyTokenProposalAction>(
    env: &dyn Environment,
    sns_ledger_canister_id: CanisterId,
    swap_canister_id: CanisterId,
    proposals: impl Iterator<Item = &ProposalData>,
    action: &MyTokenProposalAction,
) -> Result<Valuation, String>
where
    MyTokenProposalAction: TokenProposalAction,
{
    let spent_tokens = action.recent_amount_total_tokens(proposals, env.now())?;

    // Get valuation of the tokens in the treasury.
    let token = action.token()?;
    let valuation = assess_treasury_balance(
        token,
        env.canister_id(),
        sns_ledger_canister_id,
        swap_canister_id,
    )
    .await?;

    // From valuation, determine limit on the total from the past 7 days.
    let max_tokens = MyTokenProposalAction::recent_amount_total_upper_bound_tokens(&valuation)
        // Err is most likely a bug.
        .map_err(|treasury_limit_error| {
            format!("Unable to validate amount: {treasury_limit_error:?}",)
        })?;

    // Finally, inspect the proposal's amount: it must not exceed max - spent (remainder). Or if
    // you prefer, equivalently, amount + spent must be <= max.
    let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| {
        format!("Arithmetic error while performing {max_tokens} - {spent_tokens}",)
    })?;
    let proposal_amount_tokens = action.proposal_amount_tokens()?;
    if proposal_amount_tokens > allowance_remainder_tokens {
        // Although it might not be obvious to the user, their proposal is invalid, and we
        // consider it to be "their fault".
        return Err(format!(
            "Amount is too large. Within the past 7 days, a total of {spent_tokens} tokens has already \
             been executed in like proposals. Whereas, at most {max_tokens} is allowed. An additional \
             {proposal_amount_tokens} tokens from this proposal would cause that upper bound to be exceeded. \
             Maybe, try again in a few days?"
        ));
    }

    Ok(valuation)

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }
