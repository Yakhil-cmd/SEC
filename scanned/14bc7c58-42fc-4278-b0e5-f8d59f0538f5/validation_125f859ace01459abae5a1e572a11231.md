### Title
Unbounded Iteration Over Growing Proposals Map in `get_pending_proposals` Causes Permanent Query Unavailability - (File: rs/nns/governance/src/governance.rs)

### Summary
The NNS Governance canister's `get_pending_proposals` query iterates over the entire `heap_data.proposals` BTreeMap — which grows monotonically and is never pruned — to filter for open proposals. There is no iteration limit. As the NNS accumulates settled proposals over time, the per-query instruction cost grows without bound, eventually causing the query to permanently exceed the IC instruction limit and become unavailable to all callers.

### Finding Description

The `get_pending_proposals` function in `rs/nns/governance/src/governance.rs` iterates over every entry in `self.heap_data.proposals` to find those with `ProposalStatus::Open`:

```rust
self.heap_data
    .proposals
    .values()
    .filter(|data| data.status() == ProposalStatus::Open)
    .map(|data| {
        proposal_data_to_info(...)
    })
    .collect()
``` [1](#0-0) 

The `proposals` field is a `BTreeMap<u64, ProposalData>` that only ever grows. Settled proposals (decided, executed, rejected) are never removed from the map. The NNS has been running since 2021 and has accumulated thousands of proposals. The iteration cost is O(total proposals), not O(open proposals). While `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` caps the number of simultaneously open proposals, it does not cap the total number of settled proposals that must be traversed on every call.

By contrast, the sibling `list_proposals` function correctly applies a `limit` (capped at `MAX_LIST_PROPOSAL_RESULTS`) before iterating: [2](#0-1) 

The `get_pending_proposals` endpoint is exposed as a `#[query]` callable by any unprivileged principal: [3](#0-2) 

Each `ProposalData` entry can be large (containing ballots for all voting neurons). The instruction cost of iterating and deserializing thousands of such entries in a single query call will eventually exceed the IC query instruction limit.

### Impact Explanation

When the total number of proposals in `heap_data.proposals` grows large enough that iterating over all of them in a single query call exceeds the IC instruction limit, `get_pending_proposals` will permanently trap with `CanisterInstructionLimitExceeded`. This endpoint is used by Rosetta API clients, dashboards, and governance tooling to discover open proposals. Its permanent failure would break the ability of any external party to discover pending NNS proposals without pagination, degrading governance observability and tooling that depends on this endpoint. [4](#0-3) 

### Likelihood Explanation

The NNS has been running since 2021 and has accumulated thousands of proposals. The proposals map grows monotonically. The IC query instruction limit is 5 billion instructions. Each `ProposalData` iteration involves BTreeMap traversal and `proposal_data_to_info` conversion (which includes ballot filtering per neuron). As the NNS continues to operate and accumulate proposals over years, the instruction cost of this query will eventually cross the limit. No attacker action is required — natural protocol operation causes the growth. The risk is medium-term and increases monotonically.

### Recommendation

Replace the full-scan iteration in `get_pending_proposals` with a bounded approach. Since the number of open proposals is already bounded by `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`, the canister should maintain a secondary index (e.g., a `BTreeSet<u64>` of open proposal IDs) that is updated on proposal status transitions. `get_pending_proposals` would then iterate only over that bounded set rather than the entire proposals map. Alternatively, apply a hard cap on the number of results returned (similar to `list_proposals`'s `MAX_LIST_PROPOSAL_RESULTS`) and document that callers must paginate.

### Proof of Concept

1. Observe that `heap_data.proposals` is a `BTreeMap<u64, ProposalData>` with no removal of settled proposals: [5](#0-4) 

2. Observe that `get_pending_proposals` iterates the entire map with no limit: [6](#0-5) 

3. Observe that `list_proposals` (the bounded sibling) applies `take(limit)` to avoid this: [7](#0-6) 

4. The crypto invariant check code itself acknowledges that iterating all nodes at ~400M instructions/node would exceed the instruction limit per message with 1K+ nodes — confirming that unbounded per-element iteration is a real instruction-limit risk in IC production canisters: [8](#0-7) 

5. As the NNS accumulates proposals (currently thousands), the per-call instruction cost of `get_pending_proposals` grows linearly with total proposal count. Once it crosses the query instruction limit, the endpoint permanently fails for all callers with no recovery path short of a canister upgrade that adds pruning or indexing.

    pub fn get_pending_proposals(
        &self,
        caller: &PrincipalId,
        req: Option<GetPendingProposalsRequest>,
    ) -> Vec<ProposalInfo> {
        let now = self.env.now();
        let caller_neurons = self.get_neuron_ids_by_principal(caller);
        let return_self_describing_action = req
            .and_then(|r| r.return_self_describing_action)
            .unwrap_or(false);
        self.heap_data
            .proposals
            .values()
            .filter(|data| data.status() == ProposalStatus::Open)
            .map(|data| {
                proposal_data_to_info(
                    data,
                    ProposalDisplayOptions::for_get_pending_proposals(
                        return_self_describing_action,
                    ),
                    &caller_neurons,
                    now,
                    self.voting_period_seconds(),
                )
            })
            .collect()
    }
