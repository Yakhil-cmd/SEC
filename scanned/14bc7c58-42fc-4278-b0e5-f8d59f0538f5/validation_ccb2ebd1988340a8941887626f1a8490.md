### Title
Incorrect Accounting on Fee-on-Transfer ERC20 Tokens in ckERC20 Deposit Helper Contracts - (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol` and `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

Both ERC20 deposit helper contracts (`CkErc20Deposit.deposit()` and `CkDeposit.depositErc20()`) emit a deposit event recording the caller-supplied `amount` rather than the actual tokens received by the minter address. For fee-on-transfer (deflationary) ERC20 tokens, the actual received amount is less than `amount`. The IC ckETH minter canister scrapes these events and mints ckERC20 tokens equal to the event's `amount` field, creating more ckERC20 in circulation than ERC20 tokens locked. This allows early depositors to drain tokens belonging to later depositors upon withdrawal.

---

### Finding Description

**`ERC20DepositHelper.sol` — `CkErc20Deposit.deposit()`:**

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount); // actual received may be < amount
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal);            // records pre-fee amount
}
``` [1](#0-0) 

**`DepositHelperWithSubaccount.sol` — `CkDeposit.depositErc20()`:**

```solidity
function depositErc20(address erc20Address, uint256 amount, ...) public {
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount); // actual received may be < amount
    emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, ...); // records pre-fee amount
}
``` [2](#0-1) 

The IC minter canister scrapes these events via `ReceivedErc20LogParser` and `ReceivedEthOrErc20LogParser`, extracting the `value` field directly from the event data: [3](#0-2) 

The minter then mints exactly `event.value()` ckERC20 tokens to the beneficiary: [4](#0-3) 

Since `event.value()` is the pre-fee `amount` from the Solidity event (not the actual tokens received), the minter mints more ckERC20 than the ERC20 tokens actually held.

---

### Impact Explanation

For every deposit of a fee-on-transfer ERC20 token, the minter mints `amount` ckERC20 but only holds `amount - fee` ERC20. The ckERC20 total supply grows faster than the ERC20 reserve. When users withdraw, the last withdrawers cannot be fully reimbursed because the reserve is insufficient. An attacker who deposits early and withdraws before the reserve is depleted effectively steals tokens from later depositors. The invariant `total_ckERC20_supply == total_ERC20_locked` is broken.

---

### Likelihood Explanation

The helper contracts do not enforce any whitelist of ERC20 tokens beyond what the minter's supported token list allows. If any supported ckERC20 token ever implements a transfer fee (e.g., via an upgrade), or if a future supported token is fee-on-transfer by design, this bug is immediately exploitable by any unprivileged Ethereum user calling `deposit()` or `depositErc20()`. The entry path requires no privileged access — any Ethereum address can call these public functions.

---

### Recommendation

Record the actual received amount by checking the minter's ERC20 balance before and after the `safeTransferFrom`, and emit the difference in the event:

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    uint256 balanceBefore = erc20Token.balanceOf(cketh_minter_main_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    uint256 actualReceived = erc20Token.balanceOf(cketh_minter_main_address) - balanceBefore;
    emit ReceivedErc20(erc20_address, msg.sender, actualReceived, principal);
}
```

Apply the same fix to `depositErc20()` in `DepositHelperWithSubaccount.sol`.

---

### Proof of Concept

1. A fee-on-transfer ERC20 token (e.g., 1% fee on every transfer) is added as a supported ckERC20 token.
2. User A calls `deposit(token, 1000, principalA)`. The minter receives 990 tokens. The event records `amount = 1000`. The IC minter mints 1000 ckERC20 to User A.
3. User B calls `deposit(token, 1000, principalB)`. The minter receives 990 tokens (total held: 1980). The event records `amount = 1000`. The IC minter mints 1000 ckERC20 to User B. Total ckERC20 supply: 2000.
4. User A withdraws 1000 ckERC20. The minter sends 1000 ERC20 to User A (minus Ethereum transfer fee). Minter now holds ~980 ERC20.
5. User B attempts to withdraw 1000 ckERC20. The minter only holds ~980 ERC20 — User B's withdrawal fails or is underpaid, losing ~20 tokens to User A's over-minting.
