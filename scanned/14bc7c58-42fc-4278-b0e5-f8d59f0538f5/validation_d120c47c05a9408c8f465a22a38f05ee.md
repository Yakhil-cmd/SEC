### Title
Unbounded `inbound_connecting` JoinSet Enables Pre-Authentication Resource Exhaustion via QUIC Connection Flooding — (`rs/p2p/quic_transport/src/connection_manager.rs`)

---

### Summary

The QUIC transport layer spawns an unbounded Tokio task into `inbound_connecting` for every accepted `Incoming` connection before any TLS/peer-identity check occurs. An off-subnet attacker who can reach the replica's UDP transport port can flood the endpoint with QUIC Initial packets, accumulating thousands of live tasks that each hold a QUIC connection object and a 10-second timeout, exhausting heap memory and Tokio worker threads before a single authentication check fires.

---

### Finding Description

The `ConnectionManager` struct declares `inbound_connecting` as a plain, unbounded `JoinSet`: [1](#0-0) 

The event loop's `select!` arm unconditionally calls `handle_inbound_conn_attemp` for every `Incoming` returned by `endpoint.accept()`, with no size guard: [2](#0-1) 

Inside `handle_inbound_conn_attemp`, a task is spawned immediately — before any identity check: [3](#0-2) 

The TLS peer-certificate extraction and the `peer_id > node_id` check both happen *inside* the spawned `conn_fut`, only after `incoming.await` completes the full QUIC+TLS handshake: [4](#0-3) 

`CONNECT_TIMEOUT` is 10 seconds, so each malicious task lives for up to 10 s before it is reaped: [5](#0-4) 

No `concurrent_connections` cap is set on the Quinn `ServerConfig`, and no rate-limit or size check exists anywhere in the inbound path: [6](#0-5) 

The metrics infrastructure tracks `connecting_connections` as an observable gauge but imposes no enforcement limit: [7](#0-6) 

---

### Impact Explanation

Each spawned task holds a live `quinn::Incoming` (and subsequently a partially-established `quinn::Connection`), a `tokio::time::Sleep` future, and associated stack/heap. At 1 000 connection attempts per second and a 10 s timeout window, 10 000 tasks accumulate simultaneously. This exhausts:

- **Heap memory**: each `quinn::Connection` object is non-trivial; 10 k of them can consume hundreds of MB.
- **Tokio worker threads**: tasks that are polling their TLS futures compete with legitimate peer-connection tasks and request-handler tasks on the same runtime.

Legitimate subnet peers attempting to connect during the flood will find their `handle_inbound_conn_attemp` tasks starved of scheduling time, causing them to hit `CONNECT_TIMEOUT` and fail. The `outbound_connecting` path (used by the lower-ID dialer) is unaffected, but the higher-ID listener side is completely blocked. This breaks the invariant that both sides of a peer pair can establish a connection, stalling consensus finalization.

---

### Likelihood Explanation

The replica's QUIC UDP port must be reachable by other subnet peers, so it is necessarily reachable from the internet (or at minimum from any co-located attacker). QUIC Initial packets are standard UDP datagrams; no prior handshake state is required. A single commodity server can sustain thousands of QUIC Initial packets per second. The attack requires no privileged access, no key material, and no subnet-majority corruption.

---

### Recommendation

1. **Enforce a hard cap on `inbound_connecting`** before spawning: if `self.inbound_connecting.len() >= MAX_INBOUND_CONNECTING` (e.g., `2 × subnet_size`), call `incoming.refuse()` and drop the `Incoming` immediately.
2. **Set `ServerConfig::concurrent_connections`** to a small multiple of the expected subnet size so Quinn itself drops excess connections at the QUIC layer before they reach the application.
3. **Optionally move the `peer_id > node_id` check earlier**: Quinn's `Incoming` exposes the remote address; a per-IP connection rate limiter (token bucket) can shed load before any TLS work begins.
4. **Add a metric alert** when `inbound_connecting.len()` exceeds a threshold, to detect ongoing attacks.

---

### Proof of Concept

Using the `turmoil` simulation framework already used elsewhere in the IC p2p tests:

```rust
// Spawn N tasks that each open a QUIC connection to the victim replica
// using a self-signed certificate that is NOT in the subnet topology.
for _ in 0..10_000 {
    sim.client("attacker", async move {
        let _ = quinn::Endpoint::client("0.0.0.0:0".parse().unwrap())
            .unwrap()
            .connect(victim_addr, "irrelevant")
            .unwrap()
            .await; // TLS will fail, but the task is already spawned on victim
    });
}
// Now attempt a legitimate peer connection from a real subnet peer.
// Assert it completes within CONNECT_TIMEOUT.
// Without the fix, this assertion fails because the Tokio runtime is saturated.
```

The test would confirm that `inbound_connecting.len()` grows to the flood size and that the legitimate peer's connection task does not complete within `CONNECT_TIMEOUT`.

```rust
    fn handle_inbound_conn_attemp(&mut self, incoming: Incoming) {
        self.metrics.inbound_connection_total.inc();
        let node_id = self.node_id;
        let conn_fut = async move {
            let established =
                incoming
                    .await
                    .map_err(|cause| ConnectionEstablishError::ConnectionError {
                        peer_id: None,
                        cause,
                    })?;

            let rustls_certs = established
                .peer_identity()
                .ok_or(ConnectionEstablishError::AuthenticationFailed(
                    "missing peer identity".to_string(),
                ))?
                .downcast::<Vec<CertificateDer>>()
                .unwrap();
            let rustls_cert =
                rustls_certs
                    .first()
                    .ok_or(ConnectionEstablishError::AuthenticationFailed(
                        "a single cert must be present".to_string(),
                    ))?;
            let peer_id = node_id_from_certificate_der(rustls_cert.as_ref())
                .map_err(|err| ConnectionEstablishError::AuthenticationFailed(err.to_string()))?;

            // Lower ID is dialer. So we reject if this nodes id is higher.
            if peer_id > node_id {
                return Err(ConnectionEstablishError::InvalidIncomingPeerId {
                    client: peer_id,
                    server: node_id,
                });
            }

            Ok::<_, ConnectionEstablishError>(ConnectionWithPeerId {
                peer_id,
                connection: established,
            })
        };

        let timeout_conn_fut = async move {
            match tokio::time::timeout(CONNECT_TIMEOUT, conn_fut).await {
                Ok(connection_res) => connection_res,
                Err(_) => Err(ConnectionEstablishError::Timeout),
            }
        };

        self.inbound_connecting.spawn(timeout_conn_fut);
    }
```
