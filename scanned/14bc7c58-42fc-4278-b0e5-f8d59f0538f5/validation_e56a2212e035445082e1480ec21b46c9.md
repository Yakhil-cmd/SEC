### Title
Unbounded Growth of `retrieve_btc_account_to_block_indices` Causes Per-Account DoS in `retrieve_btc_status_v2_by_account` - (File: rs/bitcoin/ckbtc/minter/src/state.rs)

---

### Summary

The ckBTC Minter canister maintains a `BTreeMap<Account, Vec<u64>>` called `retrieve_btc_account_to_block_indices` that maps each user account to all of its historical retrieve_btc request block indices. Entries are appended to this Vec on every accepted retrieve_btc request but are **never removed** after requests are finalized. The public query `retrieve_btc_status_v2_by_account` iterates over the entire Vec for the queried account, calling `retrieve_btc_status_v2` for each entry. As the Vec grows without bound, this query becomes progressively more expensive and will eventually exceed the IC instruction limit, permanently denying the user access to their withdrawal status.

---

### Finding Description

**Root cause â€” unbounded append, no cleanup:**

In `rs/bitcoin/ckbtc/minter/src/state/audit.rs`, `accept_retrieve_btc_request` appends the new `block_index` to the account's Vec every time a retrieve_btc request is accepted:

```rust
state
    .retrieve_btc_account_to_block_indices
    .entry(account)
    .and_modify(|entry| entry.push(request.block_index))
    .or_insert(vec![request.block_index]);
``` [1](#0-0) 

The same append-only pattern is replicated during event log replay in `rs/bitcoin/ckbtc/minter/src/state/eventlog.rs`: [2](#0-1) 

No code path ever removes a `block_index` from this Vec â€” not when a request is finalized, not when it is confirmed, not when it is reimbursed. The field is declared as:

```rust
pub retrieve_btc_account_to_block_indices: BTreeMap<Account, Vec<u64>>,
``` [3](#0-2) 

**Unbounded iteration in the query handler:**

`retrieve_btc_status_v2_by_account` fetches the full Vec for the target account and maps every element through `retrieve_btc_status_v2`:

```rust
let block_indices: Vec<u64> = self
    .retrieve_btc_account_to_block_indices
    .get(&target_account)
    .unwrap_or(&vec![])
    .to_vec();

let result: Vec<BtcRetrievalStatusV2> = block_indices
    .iter()
    .map(|&block_index| BtcRetrievalStatusV2 {
        block_index,
        status_v2: Some(self.retrieve_btc_status_v2(block_index)),
    })
    .collect();
``` [4](#0-3) 

`retrieve_btc_status_v2` itself performs multiple sequential BTreeMap lookups (`pending_reimbursements`, `pending_withdrawal_reimbursements`, `reimbursed_transactions`, `reimbursed_withdrawals`) and falls through to `retrieve_btc_status`, which linearly scans `pending_retrieve_btc_requests` and `submitted_transactions`: [5](#0-4) 

The total work per call to `retrieve_btc_status_v2_by_account` is therefore **O(n Ă— k)** where `n` is the lifetime count of retrieve_btc requests for the account and `k` is the cost of each status lookup.

---

### Impact Explanation

The IC enforces a hard instruction limit on query calls (~5 billion instructions). Once a user's `retrieve_btc_account_to_block_indices` Vec is large enough, every call to `retrieve_btc_status_v2_by_account` for that account will trap with an instruction-limit exceeded error. The user permanently loses the ability to query the status of their withdrawals through this API. Because the Vec is stored in replicated state and never pruned, the condition is irreversible without a canister upgrade that explicitly cleans the map. Additionally, the ever-growing map itself consumes unbounded heap memory in the ckBTC Minter canister, threatening overall canister stability.

---

### Likelihood Explanation

Any unprivileged principal that holds ckBTC can call `retrieve_btc_with_approval` (or the legacy `retrieve_btc`) repeatedly over time. Each successful call appends one entry to their Vec. The `MAX_CONCURRENT_PENDING_REQUESTS` limit throttles how many requests can be *in-flight simultaneously*, but it does not cap the *lifetime* count of requests per account. A user who makes even a modest number of withdrawals over months or years will accumulate hundreds of entries. A motivated attacker who repeatedly withdraws the minimum amount will reach a DoS threshold faster. The entry path requires no privileged role â€” only a valid ckBTC balance. [6](#0-5) 

---

### Recommendation

After a retrieve_btc request reaches a terminal state (confirmed, reimbursed, or amount-too-low), remove its `block_index` from the account's Vec in `retrieve_btc_account_to_block_indices`. If historical lookup is still desired, cap the Vec to a fixed maximum (e.g., the last 100 entries per account), mirroring the existing `MAX_FINALIZED_REQUESTS = 100` cap applied to `finalized_requests`. [7](#0-6) 

Alternatively, replace the per-account Vec with a bounded ring-buffer or a paginated query that does not require iterating the full history in a single call.

---

### Proof of Concept

1. Alice calls `retrieve_btc_with_approval` N times over her account lifetime (each call with the minimum allowed amount). Each call appends one `block_index` to `retrieve_btc_account_to_block_indices[alice_account]`. [8](#0-7) 

2. After N requests are finalized, `retrieve_btc_account_to_block_indices[alice_account]` contains N entries. None are removed.

3. Alice (or anyone) calls `retrieve_btc_status_v2_by_account(Some(alice_account))`. The canister iterates all N entries, calling `retrieve_btc_status_v2` for each. [4](#0-3) 

4. For sufficiently large N, the query exceeds the IC instruction limit and traps. Alice can no longer retrieve her withdrawal history via this API. The condition persists across all future calls because the Vec is never pruned.

    pub fn retrieve_btc_status_v2(&self, block_index: u64) -> RetrieveBtcStatusV2 {
        // Hack to avoid a Candid breaking change in `ReimbursementReason`
        // which is in the return type of `retrieve_btc_status_v2`
        fn map_reimbursement_reason(reason: &WithdrawalReimbursementReason) -> ReimbursementReason {
            match reason {
                WithdrawalReimbursementReason::InvalidTransaction(
                    InvalidTransactionError::TooManyInputs { .. },
                ) => ReimbursementReason::CallFailed,
            }
        }

        if let Some(reimbursement) = self.pending_reimbursements.get(&block_index) {
            return RetrieveBtcStatusV2::WillReimburse(reimbursement.clone());
        }

        if let Some(reimbursement) = self.pending_withdrawal_reimbursements.get(&block_index) {
            return RetrieveBtcStatusV2::WillReimburse(ReimburseDepositTask {
                account: reimbursement.account,
                amount: reimbursement.amount,
                reason: map_reimbursement_reason(&reimbursement.reason),
            });
        }

        if let Some(reimbursement) = self.reimbursed_transactions.get(&block_index) {
            return RetrieveBtcStatusV2::Reimbursed(reimbursement.clone());
        }

        if let Some(maybe_reimbursed) = self.reimbursed_withdrawals.get(&block_index) {
            return match maybe_reimbursed {
                Ok(reimbursement) => RetrieveBtcStatusV2::Reimbursed(ReimbursedDeposit {
                    account: reimbursement.account,
                    amount: reimbursement.amount,
                    reason: map_reimbursement_reason(&reimbursement.reason),
                    mint_block_index: reimbursement.mint_block_index,
                }),
                Err(err) => match err {
                    ReimbursedError::Quarantined => RetrieveBtcStatusV2::Unknown,
                },
            };
        }

        let status_v2: RetrieveBtcStatusV2 = self.retrieve_btc_status(block_index).into();

        status_v2
    }

    /// Returns the status of the retrieve_btc request with the specified
    /// identifier.
    pub fn retrieve_btc_status(&self, block_index: u64) -> RetrieveBtcStatus {
        if self
            .pending_retrieve_btc_requests
            .iter()
            .any(|req| req.block_index == block_index)
        {
            return RetrieveBtcStatus::Pending;
        }

        if let Some(status) = self.requests_in_flight.get(&block_index).cloned() {
            return match status {
                InFlightStatus::Signing => RetrieveBtcStatus::Signing,
            };
        }

        if let Some(txid) = self
            .submitted_transactions
            .iter()
            .find_map(|tx| match &tx.requests {
                SubmittedWithdrawalRequests::ToConfirm { requests } => {
                    (requests.iter().any(|r| r.block_index == block_index)).then_some(tx.txid)
                }
                SubmittedWithdrawalRequests::ToCancel { .. } => None,
                SubmittedWithdrawalRequests::ToConsolidate { request } => {
                    if request.block_index == block_index {
                        Some(tx.txid)
                    } else {
                        None
                    }
                }
            })
        {
            return RetrieveBtcStatus::Submitted { txid };
        }

        match self
            .finalized_requests
            .iter()
            .find(|finalized_request| finalized_request.request.block_index() == block_index)
            .map(|final_req| final_req.state.clone())
        {
            Some(FinalizedStatus::AmountTooLow) => RetrieveBtcStatus::AmountTooLow,
            Some(FinalizedStatus::Confirmed { txid }) => RetrieveBtcStatus::Confirmed { txid },
            None => RetrieveBtcStatus::Unknown,
        }
    }
