### Title
Unbounded `artifact_processor_tasks` Growth via IDKG `SLOT_TABLE_NO_LIMIT` — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

---

### Summary

`SLOT_TABLE_NO_LIMIT` is set to `usize::MAX` and applied to the IDKG (and DKG, HTTPS-outcalls) channels. A single Byzantine subnet peer can advertise an unbounded number of distinct artifact IDs across distinct slot numbers, causing `ConsensusManagerReceiver` to spawn an unbounded number of tokio tasks in `artifact_processor_tasks`, exhausting memory and stalling the IDKG/chain-key pipeline on every honest replica.

---

### Finding Description

**Root cause — the constant:** [1](#0-0) 

```
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;   // ingress is protected
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;     // IDKG/DKG/HTTPS are not
```

The comment on line 72-74 explicitly acknowledges that `SLOT_TABLE_LIMIT_INGRESS` exists to protect against a malicious peer advertising many messages. Ingress is capped at 50,000 slots per peer. IDKG is not.

**IDKG channel registration with no limit:** [2](#0-1) 

**The slot-limit guard in `handle_slot_update_receive`:** [3](#0-2) 

The guard `peer_slot_table_len < self.slot_limit` is the **only** per-peer admission check. With `slot_limit = usize::MAX`, the `Entry::Vacant(_)` fallthrough (lines 381-390, the "drop" branch) is unreachable in practice.

**Task spawning on every new (slot, artifact_id) pair:** [4](#0-3) 

Every new artifact ID that is not already in `active_assembles` spawns a new tokio task via `artifact_processor_tasks.spawn_on(...)`. The `active_assembles` HashMap and `artifact_processor_tasks` JoinSet are **not independently bounded**; their size is entirely determined by the slot table, which has no effective limit for IDKG.

**Tasks never terminate without peer cooperation:** [5](#0-4) 

Each spawned task waits for `peer_rx` to become empty (all advertising peers removed the artifact). A Byzantine peer that never sends a deletion keeps every task alive indefinitely.

---

### Impact Explanation

A single Byzantine subnet node can:
1. Open a QUIC connection to an honest replica (authenticated, but subnet membership is the only requirement).
2. POST to `/{idkg_prefix}/update` with N distinct `slot_id` values and N distinct artifact IDs.
3. Cause N tokio tasks to be spawned and held in `artifact_processor_tasks`, each holding a `watch::Sender<PeerCounter>`, a cloned `Assembler`, and associated async state.
4. Since the Byzantine peer never sends a removal, tasks accumulate without bound.
5. Result: RSS grows without bound → OOM kill or severe memory pressure → IDKG/chain-key pipeline stalls → subnet cannot produce threshold signatures or chain-key outputs.

---

### Likelihood Explanation

- Requires the attacker to be a registered subnet node (Byzantine peer below the fault threshold). This is explicitly within the IC threat model and the stated scope ("protocol peer behavior below the consensus fault threshold").
- No cryptographic material, governance majority, or privileged access is needed beyond subnet membership.
- The attack is trivially scriptable: loop over slot IDs 0..N, send one HTTP/2 frame per iteration.
- The ingress pipeline has an analogous protection (`SLOT_TABLE_LIMIT_INGRESS = 50_000`) that was deliberately omitted for IDKG, confirming the gap is a design oversight rather than an intentional choice.

---

### Recommendation

Apply a finite per-peer slot limit to IDKG (and DKG, HTTPS-outcalls) channels, analogous to `SLOT_TABLE_LIMIT_INGRESS`. The limit should be derived from the maximum number of in-flight IDKG transcripts a single honest node would legitimately advertise simultaneously (a small, protocol-bounded constant). Replace `SLOT_TABLE_NO_LIMIT` with a value like `SLOT_TABLE_LIMIT_IDKG: usize = <protocol_max>` at the call site in `rs/replica/setup_ic_network/src/lib.rs` line 291.

---

### Proof of Concept

```rust
// Pseudocode: Byzantine peer sends N distinct slot updates to an honest replica
for slot_id in 0..N {
    let update = pb::SlotUpdate {
        commit_id: slot_id as u64,
        slot_id: slot_id as u64,
        update: Some(pb::slot_update::Update::Id(
            encode_idkg_artifact_id(unique_id(slot_id))
        )),
    };
    quic_transport.post(format!("/{}/update", IDKG_PREFIX), update.encode_to_vec()).await;
}
// On the honest replica:
// assert!(receiver.artifact_processor_tasks.len() == N);
// assert!(process_rss() > threshold);
```

Each iteration hits the `Entry::Vacant` branch at receiver.rs:373 (condition always true with `usize::MAX`), inserts into the slot table, finds no existing entry in `active_assembles`, and spawns a new task at receiver.rs:407. With N = ~100,000 iterations, RSS growth is measurable and the IDKG event loop is saturated processing task completions that never arrive.
