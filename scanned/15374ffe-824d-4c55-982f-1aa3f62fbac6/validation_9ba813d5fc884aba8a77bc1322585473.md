### Title
Missing Input Validation of `registry_polling_period_secs` in Canister `init`/`post_upgrade` Leads to Cycles Exhaustion and Rate-Limit Enforcement Failure - (File: `rs/boundary_node/rate_limits/canister/canister.rs`)

---

### Summary

The `init` and `post_upgrade` entry points of the rate-limits canister accept an `InitArg` whose `registry_polling_period_secs` field is passed directly to `set_timer_interval` without any non-zero validation. Supplying `0` creates a timer that fires every consensus round, causing the canister to make unbounded cross-canister calls to the registry, rapidly exhausting its cycles balance and rendering boundary-node rate-limit enforcement inoperative.

---

### Finding Description

In `rs/boundary_node/rate_limits/canister/canister.rs`, the `#[init]` function accepts `InitArg` and immediately passes `init_arg.registry_polling_period_secs` to `periodically_poll_api_boundary_nodes`: [1](#0-0) 

That helper converts the value directly to a `Duration` and hands it to `ic_cdk_timers::set_timer_interval`: [2](#0-1) 

`#[post_upgrade]` is identical — it simply delegates to `init`: [3](#0-2) 

No guard exists anywhere in the call chain to reject `registry_polling_period_secs == 0`. When `Duration::from_secs(0)` is passed to `set_timer_interval`, the IC runtime schedules the callback to execute on every round (approximately every 2 seconds). Each invocation issues a cross-canister `call` to the registry canister: [4](#0-3) 

This is structurally identical to the `PositionsManagerForAaveLogic` finding in the report, where an unvalidated `_maxGas` value causes match/unmatch functions to always fail or revert — here an unvalidated interval causes the periodic task to always fire, consuming all available resources.

---

### Impact Explanation

1. **Cycles exhaustion**: Each timer tick issues a cross-canister call. With a zero interval the canister burns through its entire cycles balance within minutes, after which the canister is frozen.
2. **Rate-limit enforcement failure**: Once frozen, boundary nodes can no longer call `get_config` to retrieve updated rate-limit rules. The `inspect_message` hook that gates `add_config` and `disclose_rules` also stops executing, so the canister's security model collapses entirely.
3. **Persistent broken state**: Because `post_upgrade` calls `init` with the same `InitArg`, a subsequent upgrade that again passes `registry_polling_period_secs = 0` re-creates the same timer, making recovery non-trivial.

---

### Likelihood Explanation

The `InitArg` is a Candid-encoded argument supplied at install/upgrade time. Any controller of the canister (or a governance proposal that upgrades it) can supply `registry_polling_period_secs = 0` — either accidentally (omitting the field defaults to `0` for `u64` in Candid) or intentionally. Because `u64` defaults to `0` in Candid deserialization when the field is absent, a deployer who forgets to set the field will silently trigger the bug on every install or upgrade.

---

### Recommendation

Add an explicit non-zero check at the top of `init` before the timer is created:

```rust
#[init]
fn init(init_arg: InitArg) {
    assert!(
        init_arg.registry_polling_period_secs > 0,
        "registry_polling_period_secs must be > 0"
    );
    // ... rest of init
}
```

Alternatively, enforce a minimum sensible floor (e.g., 60 seconds) and document the constraint in the `InitArg` type definition.

---

### Proof of Concept

1. Deploy the rate-limits canister with `InitArg { registry_polling_period_secs: 0, authorized_principal: None }`.
2. `init` calls `periodically_poll_api_boundary_nodes(0, ...)`.
3. `set_timer_interval(Duration::from_secs(0), callback)` is registered.
4. The IC runtime fires `callback` every round; each invocation calls `registry_canister.get_api_boundary_node_ids`.
5. Cycles balance drops to zero within minutes; the canister is frozen.
6. All subsequent `get_config` queries from boundary nodes are rejected with "canister out of cycles".
7. Rate-limit rules can no longer be fetched or updated; boundary-node rate limiting is disabled.
