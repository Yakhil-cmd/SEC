### Title
Range-scan break condition in `get_allowances` checks only `account.owner`, leaking allowances of sibling subaccounts — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

### Summary
`get_allowances` uses a BTreeMap range scan starting at the requested `from` account. Its termination guard only tests `account_spender.account.owner != from.owner`, ignoring the subaccount. Any caller can therefore enumerate all allowances belonging to every subaccount of a given principal by querying with the lexicographically smallest subaccount of that principal.

### Finding Description
In `rs/ledger_suite/icrc1/ledger/src/lib.rs`, the `get_allowances` function iterates over a stable BTreeMap keyed by `AccountSpender`, which is ordered first by `Account` (owner then effective subaccount) and then by spender. [1](#0-0) 

The loop break condition is:

```rust
if account_spender.account.owner != from.owner {
    break;
}
``` [2](#0-1) 

Because `Account::Ord` sorts by `(owner, effective_subaccount)`, all entries for `{A, subaccount_1}`, `{A, subaccount_2}`, … are contiguous in the map and all share the same `owner`. The break condition never fires for sibling subaccounts of the same principal, so the scan continues past the requested account and returns allowances for every subaccount of `A` whose effective subaccount is `>= from.effective_subaccount()`. [3](#0-2) 

The `Account::Ord` implementation confirms the ordering: [4](#0-3) 

The public endpoint `icrc103_get_allowances` passes the caller-supplied `from_account` directly to `get_allowances` with no access restriction: [5](#0-4) 

### Impact Explanation
An unprivileged caller who knows (or guesses) principal A can call `icrc103_get_allowances` with `from_account = {A, Some([0;32])}` (the minimum subaccount) and receive all allowances for every subaccount of A in a single paginated scan. This leaks spender identities, allowance amounts, and expiration times for subaccounts the caller did not request and that the account holder may consider private.

### Likelihood Explanation
The endpoint is a public query call reachable by any IC principal with no authentication or rate-limiting beyond the standard query cost. The exploit requires only knowledge of a target principal, which is often public (e.g., from transaction history). It is trivially reproducible in a unit test.

### Recommendation
Change the break condition to compare the full `Account` (owner + effective subaccount):

```rust
// Before (buggy):
if account_spender.account.owner != from.owner {
    break;
}

// After (correct):
if account_spender.account != from {
    break;
}
```

`Account::PartialEq` already uses `effective_subaccount()` for comparison, so this single-character change correctly terminates the scan at the subaccount boundary. [6](#0-5)

fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    let max_take_allowances = Access::with_ledger(|ledger| ledger.max_take_allowances());
    let max_results = arg
        .take
        .map(|take| take.0.to_u64().unwrap_or(max_take_allowances))
        .map(|take| std::cmp::min(take, max_take_allowances))
        .unwrap_or(max_take_allowances);
    Ok(get_allowances(
        from_account,
        arg.prev_spender,
        max_results,
        ic_cdk::api::time(),
    ))
