### Title
SNS Governance `approve_treasury_manager` Sets Allowances Without `expected_allowance`, Enabling Allowance Double-Spend by Malicious Treasury Manager - (File: rs/sns/governance/src/extensions.rs)

---

### Summary

`Governance::approve_treasury_manager` calls `icrc2_approve` with `expected_allowance: None` for both SNS and ICP token allowances. An inline comment incorrectly asserts this eliminates double-spending risk. In reality, a malicious treasury manager canister can drain the residual old allowance via `icrc2_transfer_from` before the governance's new `icrc2_approve` message is processed by the ledger, then also consume the freshly set allowance — spending X + Y instead of the intended Y.

---

### Finding Description

In `rs/sns/governance/src/extensions.rs`, `Governance::approve_treasury_manager` (lines 777–831) issues two `icrc2_approve` calls — one to the SNS ledger (lines 794–810) and one to the ICP ledger (lines 812–828) — both with `expected_allowance: None`:

```rust
// If expected_allowance is None, the ledger *blindly* overwrites any existing
// allowance (even if non-zero). Therefore, there is no risk of double spending.

self.ledger
    .icrc2_approve(
        to,
        sns_amount_e8s,
        Some(expiry_time_nsec),
        self.transaction_fee_e8s_or_panic(),
        self.sns_treasury_subaccount(),
        None,   // ← expected_allowance
    )
    .await
``` [1](#0-0) 

The comment's reasoning is wrong. "Blindly overwriting" the allowance does not prevent the treasury manager from spending the *old* allowance in the window between when the governance canister sends the `icrc2_approve` message and when the ledger canister processes it.

`approve_treasury_manager` is called from two sites:

1. **`ValidatedRegisterExtension::execute`** (lines 545–551) — safe, because `ensure_no_code_is_installed` (lines 523–525) guarantees the extension canister has no Wasm installed yet, so it cannot call `icrc2_transfer_from`. [2](#0-1) 

2. **`execute_treasury_manager_deposit`** (lines 1566–1573) — **vulnerable**, because the treasury manager already has code installed and may hold a residual allowance from a prior deposit round. [3](#0-2) 

The ICRC-2 `AllowanceTable::approve` in `rs/ledger_suite/common/ledger_core/src/approvals.rs` (lines 253–291) only enforces the `expected_allowance` guard when the caller supplies a non-`None` value. When `None` is passed, the allowance is unconditionally overwritten regardless of the current on-chain value. [4](#0-3) 

---

### Impact Explanation

A malicious treasury manager canister can spend **X + Y** SNS tokens (and/or ICP) instead of the governance-intended **Y**:

- **X** = residual allowance from a prior deposit that the treasury manager deliberately did not exhaust.
- **Y** = the new allowance set by the current deposit proposal.

This directly undermines the SNS governance's ability to bound treasury outflows. Each successive deposit proposal can be exploited to drain an additional X tokens beyond the approved amount, constituting a **governance authorization bypass** and **ledger conservation bug** for SNS-controlled treasuries.

---

### Likelihood Explanation

**Medium.** The attack requires a malicious treasury manager canister — explicitly within the HackenProof scope as "malicious canister." SNS governance proposals are public and have a multi-day voting period, giving the treasury manager ample time to observe an incoming deposit proposal and pre-stage an `icrc2_transfer_from` call. On the IC, inter-canister message ordering between the treasury manager and the governance canister to the ledger is not guaranteed; the treasury manager's message can arrive at the ledger before the governance's `icrc2_approve` message. The treasury manager only needs to retain a non-zero residual allowance from a prior deposit to mount the attack.

---

### Recommendation

Pass the current on-chain allowance as `expected_allowance` in both `icrc2_approve` calls. Before calling `approve_treasury_manager`, query `icrc2_allowance` for the treasury manager and supply the result as `expected_allowance`. If the ledger returns `AllowanceChanged`, the governance should first zero out the allowance (with the correct `expected_allowance`) before setting the new value. This is the standard two-step mitigation for the approve-frontrunning pattern, and the ICRC-2 `AllowanceChanged` error variant exists precisely for this purpose. [5](#0-4) 

---

### Proof of Concept

1. SNS governance executes a first deposit proposal: `approve_treasury_manager(tm, X_sns, X_icp)` → SNS allowance = X, ICP allowance = X.
2. Treasury manager calls `deposit` but intentionally transfers only `X - ε` tokens, leaving allowance = ε > 0.
3. SNS governance passes a second deposit proposal for Y tokens.
4. Treasury manager observes the proposal during the voting period and submits `icrc2_transfer_from(treasury_sns_account → tm_account, ε)` to the SNS ledger. SNS allowance → 0.
5. Governance's `icrc2_approve(tm, Y, expected_allowance=None)` is processed next: SNS allowance → Y.
6. Treasury manager calls `deposit`, consuming Y tokens.
7. **Net result**: treasury manager extracted `ε + Y` SNS tokens; governance intended only Y.

The same sequence applies to the ICP allowance via `self.nns_ledger.icrc2_approve(…, None)`. [6](#0-5) [7](#0-6)

async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
