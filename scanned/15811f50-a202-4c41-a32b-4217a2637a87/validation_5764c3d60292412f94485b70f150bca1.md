### Title
ckBTC Minter Allows Withdrawal to User Deposit Address, Enabling Circular Re-Minting That Breaks 1:1 BTC Backing - (`File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

---

### Summary

The ckBTC minter's `retrieve_btc` and `retrieve_btc_with_approval` endpoints only block withdrawals to the minter's own **main** Bitcoin address, but do not block withdrawals to any **user deposit address** (which are also derived from the minter's threshold ECDSA key). An unprivileged user can withdraw ckBTC specifying their own deposit address as the destination. The minter sends BTC back to that address, creating new UTXOs there. When the user subsequently calls `update_balance`, the minter sees those new UTXOs and mints ckBTC again — for BTC that was already accounted for — breaking the 1:1 backing invariant.

---

### Finding Description

The ckBTC minter derives Bitcoin addresses for all accounts (both the minter itself and users) from the same threshold ECDSA key. The minter's main address corresponds to `Account { owner: minter_canister_id, subaccount: None }`, while user deposit addresses correspond to `Account { owner: user_principal, subaccount: ... }`.

In `retrieve_btc`, the only forbidden-destination check is:

```rust
let main_address_str = state::read_state(|s| runtime.derive_minter_address_str(s));
if args.address == main_address_str {
    ic_cdk::trap("illegal retrieve_btc target");
}
``` [1](#0-0) 

The same incomplete check exists in `retrieve_btc_with_approval`: [2](#0-1) 

`derive_minter_address_str` only returns the address for the minter's own main account (no subaccount): [3](#0-2) 

User deposit addresses are derived via `account_to_p2wpkh_address_from_state` for `Account { owner: user_principal, subaccount: ... }`: [4](#0-3) 

These user deposit addresses are **not** checked against the withdrawal target. The minter tracks already-processed UTXOs via `minted_outpoints`, but a withdrawal transaction creates **new UTXOs** with new outpoints at the destination address, so they are not in `minted_outpoints` and will be processed as fresh deposits. [5](#0-4) 

---

### Impact Explanation

An attacker can execute the following circular attack to inflate the ckBTC supply beyond the BTC held by the minter:

1. Call `get_btc_address(attacker_principal, None)` → obtain deposit address `A` (derived from the minter's ECDSA key for the attacker's account).
2. Send X BTC to address `A` on Bitcoin; call `update_balance` → minter mints X ckBTC (minus fees).
3. Call `retrieve_btc(address=A, amount=X-fees)` → minter burns ckBTC and sends BTC to address `A`. This creates **new UTXOs** at `A` with new outpoints.
4. Call `update_balance` again → minter sees the new UTXOs at `A` (not in `minted_outpoints`) and mints ckBTC again.

Each cycle produces ckBTC without any net BTC entering the minter's custody, directly violating the protocol invariant that ckBTC supply equals BTC held by the minter. This is the IC analog of the Chakra "bridge to same chain" vulnerability: the source and destination of the chain-fusion operation are the same address, so the burn on one side is immediately re-minted on the same side.

The existing integration test confirms the minter's main address is blocked but does not test user deposit addresses: [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged user can execute this attack with no special access. The only cost is Bitcoin transaction fees (paid twice per cycle: once to deposit, once for the minter's withdrawal transaction). The attacker controls the withdrawal address argument directly via the public `retrieve_btc` or `retrieve_btc_with_approval` ingress endpoint. No governance majority, threshold key compromise, or privileged role is required.

---

### Recommendation

In both `retrieve_btc` and `retrieve_btc_with_approval`, after parsing the destination address, check that it does not correspond to **any** address derivable from the minter's ECDSA key. Concretely, the minter should reject any withdrawal target whose Bitcoin address matches any address it would derive for any account (i.e., any P2WPKH address whose public key is a child of the minter's ECDSA root key). A practical approach is to check that the parsed `BitcoinAddress` does not equal any address the minter controls — at minimum, reject any address whose derivation path prefix matches the minter's key hierarchy. Alternatively, maintain a set of all known user deposit addresses and reject withdrawals to any of them.

---

### Proof of Concept

```
1. Alice calls get_btc_address(alice_principal, None) → address A
2. Alice sends 1 BTC to address A on Bitcoin
3. Alice calls update_balance(alice_principal, None)
   → Minter mints 0.9999 ckBTC to Alice (after fees)
4. Alice calls retrieve_btc(address=A, amount=0.9998)
   → Minter burns 0.9998 ckBTC from Alice's withdrawal account
   → Minter sends 0.9998 BTC to address A (new UTXO, new outpoint)
5. Alice calls update_balance(alice_principal, None)
   → Minter sees new UTXO at A (outpoint not in minted_outpoints)
   → Minter mints ~0.9997 ckBTC to Alice again

Net result: Alice holds ~0.9997 ckBTC but the minter's net BTC custody
has not increased — the same BTC was counted twice.
Repeating steps 4-5 inflates ckBTC supply unboundedly.
```

        let main_account = Account {
            owner: ic_cdk::api::canister_self(),
