### Title
Unchecked `envelope_pairs[0]` Index in `construction_hash` Causes Panic on Empty `Vec<EnvelopePair>` — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

---

### Summary
An unauthenticated HTTP client can crash the ICP Rosetta node by sending a crafted CBOR-encoded `SignedTransaction` to `POST /construction/hash` where a `Request` entry carries `RequestType::Send` paired with an empty `Vec<EnvelopePair>`. The handler unconditionally indexes `envelope_pairs[0]` without a bounds check, triggering a Rust index-out-of-bounds panic.

---

### Finding Description

`Request` is defined as a tuple type: [1](#0-0) 

```rust
pub type Request = (RequestType, Vec<EnvelopePair>);
```

`SignedTransaction::from_str` performs pure CBOR deserialization with no structural validation — it does not assert that the inner `Vec<EnvelopePair>` is non-empty: [2](#0-1) 

In `construction_hash`, after finding the first `RequestType::Send` entry via `.find(|(rt, _)| rt.is_transfer())`, the code immediately indexes position 0 without checking the length: [3](#0-2) 

If the matched entry has an empty `Vec<EnvelopePair>`, `envelope_pairs[0]` panics unconditionally.

**Contrast with the guarded path**: `TryFrom<&models::Request> for Request` (used by `construction_parse`/`construction_submit`) correctly guards the same access: [4](#0-3) 

```rust
let payload: &models::EnvelopePair = calls
    .first()
    .ok_or_else(|| ApiError::invalid_request("No request payload provided."))?;
```

`construction_hash` has no equivalent guard — it is the only production site that uses the bare `[0]` index on attacker-controlled data: [5](#0-4) 

The HTTP handler is registered with actix-web and has no `catch_unwind` wrapper: [6](#0-5) 

---

### Impact Explanation
A Rust index-out-of-bounds panic in an actix-web handler propagates to the worker thread. actix-web worker threads are not automatically restarted on panic in the default configuration; a panic aborts the worker. Repeated requests exhaust the worker pool and crash the Rosetta node process, causing complete availability loss of the ICP Rosetta API. This disrupts ledger-monitoring infrastructure and any exchange or integration relying on the Rosetta node for ICP transaction submission and hash derivation.

---

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. The exploit requires only crafting a valid hex-encoded CBOR blob — a trivial operation using any CBOR library. The payload structure is fully documented by the open-source type definitions. A single malformed request is sufficient to trigger the panic.

---

### Recommendation
Replace the unchecked index with a bounds-checked accessor and return a proper `ApiError` on empty input, mirroring the pattern already used in `request.rs`:

```rust
let first_pair = envelope_pairs
    .first()
    .ok_or_else(|| ApiError::invalid_request("No envelope pairs provided for Send request"))?;
TransactionIdentifier::try_from_envelope(request_type.clone(), &first_pair.update)
``` [7](#0-6) 

---

### Proof of Concept

```rust
// Craft the malicious payload
let malicious = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],  // empty EnvelopePair vec
};
let cbor_bytes = serde_cbor::to_vec(&malicious).unwrap();
let hex_payload = hex::encode(&cbor_bytes);

// POST to /construction/hash
let body = serde_json::json!({
    "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
    "signed_transaction": hex_payload
});
// Expected: ApiError response
// Actual: index out of bounds panic → worker crash → process termination
```

The `find` at line 23 matches the `RequestType::Send` entry (since `is_transfer()` returns `true` for `RequestType::Send`), `envelope_pairs` is the empty `vec![]`, and `envelope_pairs[0]` panics. [8](#0-7)
