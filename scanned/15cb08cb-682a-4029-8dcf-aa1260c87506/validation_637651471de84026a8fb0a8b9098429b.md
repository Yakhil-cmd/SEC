### Title
Sequential Archive Canister Upgrade Loop Lacks Per-Canister Error Isolation, Leaving Archives Stopped and Blocking Remaining Upgrades - (File: rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs)

### Summary

The `UpgradeLedgerSuiteSubtask::UpgradeArchives` execution path in the ledger-suite-orchestrator iterates over all archive canisters for a given ERC-20 token and calls `upgrade_canister` on each sequentially. The loop uses the `?` operator to propagate the first error immediately, with no per-canister error isolation. If any archive canister's stop/upgrade/start sequence fails, the remaining archive canisters are not processed in that timer tick. Additionally, there is no pre-check of each archive canister's operational state (running, cycles balance) before initiating the stop/upgrade/start sequence, mirroring the root cause in the external report where no availability check is performed before calling an external protocol.

### Finding Description

In `rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs`, the `UpgradeLedgerSuiteSubtask::UpgradeArchives` arm iterates over all known archive canisters for a token and calls `upgrade_canister` on each:

```rust
//We expect usually 0 or 1 archive, so a simple sequential strategy is good enough.
for canister_id in archives {
    upgrade_canister::<Archive, _>(canister_id, compressed_wasm_hash, runtime)
        .await?;
}
``` [1](#0-0) 

The `upgrade_canister` function performs three sequential management-canister calls — `stop_canister`, `upgrade_canister` (install_code), and `start_canister` — with no pre-check of the target canister's current state or cycles balance:

```rust
runtime.stop_canister(canister_id).await
    .map_err(UpgradeLedgerSuiteError::StopCanisterError)?;
runtime.upgrade_canister(canister_id, wasm.to_bytes()).await
    .map_err(UpgradeLedgerSuiteError::UpgradeCanisterError)?;
runtime.start_canister(canister_id).await
    .map_err(UpgradeLedgerSuiteError::StartCanisterError)?;
``` [2](#0-1) 

If `stop_canister` succeeds but `upgrade_canister` fails (e.g., the archive canister is out of cycles, the management canister rejects the call, or the archive is in an unexpected state), the archive is left **permanently stopped** until the next retry. All subsequent archive canisters in the `archives` vector are never reached in that timer tick because `?` propagates the error immediately.

The `is_recoverable` classification for `UpgradeCanisterError` is `true`, so the task is rescheduled. However, on retry the same archive is attempted first again. If the failure is persistent (e.g., the archive canister's cycles are chronically low between `maybe_top_up` intervals, or the management canister is temporarily rejecting calls), the loop will keep failing on the same archive canister, leaving it stopped and blocking all subsequent archive upgrades indefinitely.

There is no pre-check analogous to Lido's `getCurrentStakeLimit()` or Rocket Pool's `getDepositEnabled()` — no check that the archive canister is `Running` and has sufficient cycles before initiating the stop/upgrade/start sequence. [3](#0-2) 

The `discover_archives` function (called as `Task::DiscoverArchives`) does handle per-ledger errors gracefully by collecting all results before returning the first error, allowing successful ledgers to be recorded: [4](#0-3) 

But `UpgradeArchives` does not apply the same pattern.

### Impact Explanation

1. **Archive canisters left in stopped state**: If `stop_canister` succeeds but `upgrade_canister` fails, the archive canister is stopped and not restarted. Users querying historical transactions routed to that archive receive errors until the next successful retry.
2. **Remaining archives not upgraded**: All archive canisters after the failing one in the iteration order are skipped entirely in that timer tick, creating a version mismatch between the ledger and its archives.
3. **Persistent blocking**: If the failure condition is not transient (e.g., the archive canister is chronically low on cycles between `maybe_top_up` runs), the upgrade loop retries indefinitely on the same archive, leaving it stopped and blocking all subsequent archive upgrades for that token's ledger suite.
4. **Service disruption for ckERC20 users**: Archive canisters being stopped means ICRC-3 block queries for historical transactions fail, degrading the ckERC20 ledger suite's availability.

### Likelihood Explanation

- Archive canisters can run low on cycles between `maybe_top_up` timer ticks, especially during periods of high transaction volume that cause the ledger to spawn new archives.
- The management canister can transiently reject `stop_canister` or `upgrade_canister` calls due to subnet load.
- The `maybe_top_up` task and `UpgradeArchives` task run on separate timers with no coordination, creating a window where an archive canister may have insufficient cycles when the upgrade is attempted.
- The test `should_discover_archive_and_return_first_error` confirms the codebase is aware of per-canister failure scenarios in `discover_archives` but the same pattern was not applied to `UpgradeArchives`. [5](#0-4) 

### Recommendation

1. **Add per-canister error isolation in the `UpgradeArchives` loop**: Collect errors per archive canister rather than propagating the first error with `?`. Log failures and continue upgrading remaining archives. Re-schedule failed archives for retry independently.

2. **Add a pre-check for archive canister state before stop/upgrade/start**: Query the archive canister's status (running/stopped/stopping) and cycles balance before initiating the sequence. If the canister is already stopped (from a previous failed upgrade), skip `stop_canister` and proceed directly to `upgrade_canister`. If cycles are insufficient, skip and log rather than leaving the canister stopped.

3. **Apply the same error-collection pattern used in `discover_archives`**: The `discover_archives` function already demonstrates the correct pattern — collect all results, record successes, and return only the first error. The `UpgradeArchives` loop should adopt the same approach.

### Proof of Concept

Given a ledger suite with two archive canisters `[archive_A, archive_B]`:

1. The `UpgradeArchives` subtask executes.
2. `upgrade_canister(archive_A, ...)` is called:
   - `stop_canister(archive_A)` → **succeeds** (archive_A is now stopped).
   - `upgrade_canister(archive_A, wasm)` → **fails** (e.g., archive_A is out of cycles, management canister rejects).
   - `?` propagates `UpgradeLedgerSuiteError::UpgradeCanisterError`.
3. `upgrade_canister(archive_B, ...)` is **never called** — archive_B remains on the old wasm.
4. The task is rescheduled (error is recoverable).
5. On retry, `stop_canister(archive_A)` is called again on the already-stopped canister. If the cycles issue persists, the loop fails again at the same point.
6. archive_A remains stopped indefinitely; archive_B is never upgraded.

This mirrors the SafEth pattern exactly: one external call failing (Lido daily limit / archive out of cycles) causes the entire loop to abort, leaving all subsequent items unprocessed and the failed item in a broken intermediate state. [6](#0-5) [7](#0-6)

```rust
async fn upgrade_canister<T: StorableWasm, R: CanisterRuntime>(
    canister_id: Principal,
    wasm_hash: &WasmHash,
    runtime: &R,
) -> Result<(), UpgradeLedgerSuiteError> {
    let wasm = match read_wasm_store(|s| wasm_store_try_get::<T>(s, wasm_hash)) {
        Ok(Some(wasm)) => Ok(wasm),
        Ok(None) => Err(UpgradeLedgerSuiteError::WasmHashNotFound(wasm_hash.clone())),
        Err(e) => Err(UpgradeLedgerSuiteError::WasmStoreError(e)),
    }?;

    log!(DEBUG, "Stopping canister {}", canister_id);
    runtime
        .stop_canister(canister_id)
        .await
        .map_err(UpgradeLedgerSuiteError::StopCanisterError)?;

    log!(
        DEBUG,
        "Upgrading wasm module of canister {} to {}",
        canister_id,
        wasm_hash
    );
    runtime
        .upgrade_canister(canister_id, wasm.to_bytes())
        .await
        .map_err(UpgradeLedgerSuiteError::UpgradeCanisterError)?;

    log!(DEBUG, "Starting canister {}", canister_id);
    runtime
        .start_canister(canister_id)
        .await
        .map_err(UpgradeLedgerSuiteError::StartCanisterError)?;

    log!(
        DEBUG,
        "Upgrade of canister {} to {} completed",
        canister_id,
        wasm_hash
    );
    let now = runtime.time();
    mutate_state(|s| s.record_upgrade_completed(canister_id, wasm_hash.clone(), now));
    Ok(())
