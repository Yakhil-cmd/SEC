### Title
Disabled `MintSnsTokens` 7-Day Rate Limit Allows Unbounded SNS Token Inflation — (File: `rs/sns/governance/src/proposal.rs`)

---

### Summary

The `MintSnsTokens` SNS governance proposal action has its 7-day minting rate limit intentionally disabled via a `TODO` comment, returning `Decimal::MAX` as the effective upper bound. No protocol-level cap exists on how many SNS tokens can be minted via governance proposals within any time window, directly analogous to the SkyWeaver `maxSupply` validation gap where a privileged role can set a supply limit without checking against the current issuance.

---

### Finding Description

In `rs/sns/governance/src/proposal.rs`, the `TokenProposalAction` trait implementation for `MintSnsTokens` overrides `recent_amount_total_upper_bound_tokens` to return `Decimal::MAX` unconditionally: [1](#0-0) 

The real limit function `mint_sns_tokens_7_day_total_upper_bound_tokens` — which computes a treasury-valuation-based cap — is commented out: [2](#0-1) 

The import of that function is also suppressed at the top of the file: [3](#0-2) 

By contrast, `TransferSnsTreasuryFunds` has its rate limit fully enforced: [4](#0-3) 

The validation path for a `MintSnsTokens` proposal calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which in turn calls `recent_amount_total_upper_bound_tokens`. Because that returns `Decimal::MAX`, the check at line 805 (`if proposal_amount_tokens > allowance_remainder_tokens`) can never trigger: [5](#0-4) 

The integration test confirms the disabled state — the assertion that a second large mint should be rejected is commented out, and the line that makes it pass unconditionally is present: [6](#0-5) 

When a `MintSnsTokens` proposal passes governance, `perform_mint_sns_tokens` calls `transfer_funds` on the SNS ledger with no supply cap check: [7](#0-6) 

---

### Impact Explanation

The 7-day rate limit was specifically designed as a protocol-level safety mechanism to constrain even a governance majority from inflating the SNS token supply too rapidly. With it disabled:

- Any SNS governance majority can pass back-to-back `MintSnsTokens` proposals for arbitrarily large amounts (up to `u64::MAX` e8s each) with no cooldown or cap.
- Existing token holders suffer unlimited dilution.
- The asymmetry with `TransferSnsTreasuryFunds` (which is rate-limited) means the minting path is unprotected while the treasury-transfer path is not.

This is a **ledger conservation bug**: the supply invariant that the protocol intends to enforce is absent.

---

### Likelihood Explanation

Medium. Exploitation requires an SNS governance majority. In practice, many SNS deployments have a single large neuron holder ("whale") with >50% voting power, or a small coordinated group. The `MintSnsTokens` proposal type is a standard, publicly documented governance action reachable by any SNS neuron holder via ingress to the SNS governance canister.

---

### Recommendation

Uncomment the `mint_sns_tokens_7_day_total_upper_bound_tokens` call in `recent_amount_total_upper_bound_tokens` for `MintSnsTokens` (as indicated by `TODO(NNS1-2982): Uncomment`) and delete the temporary `Decimal::MAX` stub. Simultaneously, uncomment the corresponding integration-test assertion in `rs/sns/integration_tests/src/sns_treasury.rs` to enforce the rate limit in CI.

---

### Proof of Concept

1. Deploy an SNS with a governance canister where a single principal holds a majority neuron.
2. Submit a `MintSnsTokens` proposal with `amount_e8s = u64::MAX`.
3. Pass the proposal (single-majority neuron votes yes).
4. Observe the proposal executes successfully — `perform_mint_sns_tokens` calls `transfer_funds` with no cap check.
5. Immediately submit a second identical proposal; it also passes and executes.
6. The SNS ledger `total_supply` grows without bound across both proposals, confirmed by `icrc1_total_supply`.

This is directly demonstrated by the existing test at `rs/sns/integration_tests/src/sns_treasury.rs` line 966 (`doomed_make_proposal_result.unwrap()`), where the second mint proposal succeeds when the rate-limit logic says it should fail.

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
