Looking at the production code carefully, I can trace the full attack path.

## Key Code Facts

**`SLOT_TABLE_NO_LIMIT` definition and usage:** [1](#0-0) 

```
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
```

`SLOT_TABLE_NO_LIMIT` is passed as `slot_limit` for **consensus** (lines 237, 246), **certifier** (268), **dkg** (279), **idkg** (291), and **https_outcalls** (303). Only ingress uses the bounded `SLOT_TABLE_LIMIT_INGRESS`. [2](#0-1) [3](#0-2) 

**The guard in `handle_slot_update_receive`:** [4](#0-3) 

The guard is:
```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { ... }
```

When `slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice — no process can hold `usize::MAX` HashMap entries before OOM kills it first. The fallthrough `Entry::Vacant(_)` branch that drops the update and logs a warning is **never reached** for these artifact types.

**The task-spawning path:** [5](#0-4) 

For each slot update that passes the guard, if the artifact ID is not already in `active_assembles`, a new task is unconditionally spawned into `artifact_processor_tasks` (a `JoinSet`). The deduplication is by artifact ID, not by slot number.

**The data structures:** [6](#0-5) 

`slot_table` is `HashMap<NodeId, HashMap<SlotNumber, SlotEntry>>` and `active_assembles` is `HashMap<WireArtifact::Id, watch::Sender<PeerCounter>>`. Both are unbounded for non-ingress types.

## Attack Path

A single Byzantine subnet peer (authenticated via QUIC/TLS, so its `NodeId` is known) sends N slot updates:

- Each update uses a **distinct `slot_number`** → each hits `Entry::Vacant`, passes `peer_slot_table_len < usize::MAX`, and is inserted into the peer's slot table.
- Each update uses a **distinct artifact ID** → `active_assembles.get(&id)` returns `None` for each, so a new `process_slot_update` task is spawned into `artifact_processor_tasks`.

Result: N concurrent assemble tasks, N entries in `active_assembles`, N entries in the peer's slot table — all unbounded.

The comment at lines 72–73 explicitly acknowledges the limit is a protection against malicious peers, but the protection is only applied to ingress: [7](#0-6)

## Verdict

### Title
Unbounded assemble task spawning via `SLOT_TABLE_NO_LIMIT` allows a single Byzantine peer to exhaust replica memory — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

### Summary
`SLOT_TABLE_NO_LIMIT = usize::MAX` is the production `slot_limit` for consensus, DKG, IDKG, certification, and HTTPS outcalls artifact types. The per-peer slot table guard in `handle_slot_update_receive` is effectively disabled, allowing a single Byzantine peer to insert an unbounded number of slot entries and spawn an unbounded number of assemble tasks.

### Finding Description
In `handle_slot_update_receive`, new slot entries are only rejected when `peer_slot_table_len >= self.slot_limit`. With `slot_limit = usize::MAX`, this rejection never fires. For each new `(slot_number, artifact_id)` pair where the artifact ID has not been seen before, a new Tokio task is spawned into `artifact_processor_tasks` (a `JoinSet`) and a new entry is inserted into `active_assembles`. A Byzantine peer can send N updates with N distinct slot numbers and N distinct artifact IDs, spawning N concurrent tasks with no upper bound.

### Impact Explanation
Each spawned `process_slot_update` task holds a `watch::Receiver<PeerCounter>`, a cloned `Assembler`, a cloned `Sender`, and potentially initiates network fetch operations. At scale (10^6 tasks), this exhausts heap memory and Tokio task scheduler capacity, crashing the replica process. A crashed replica halts local consensus participation, and if enough replicas are affected, subnet finalization stalls.

### Likelihood Explanation
The attacker is a single compromised subnet node — a realistic Byzantine adversary below the fault threshold. The QUIC transport authenticates the sender's `NodeId` but imposes no rate limit or slot count limit on slot update messages for non-ingress artifact types. The attack requires only sending crafted P2P slot update messages, which is within the capability of any subnet peer.

### Recommendation
Apply a bounded `slot_limit` for all artifact types, not just ingress. The appropriate bound should reflect the maximum number of in-flight artifacts a legitimate peer would ever advertise simultaneously for each artifact type (e.g., bounded by consensus round depth, DKG epoch size, etc.). The existing `SLOT_TABLE_LIMIT_INGRESS = 50_000` pattern should be replicated with type-appropriate constants for consensus, DKG, IDKG, certification, and HTTPS outcalls.

### Proof of Concept
State-machine test (no network required):
```rust
let (mut mgr, _channels) = ReceiverManagerBuilder::new()
    .with_slot_limit(usize::MAX)
    .build();
let cancellation = CancellationToken::new();
for i in 0..1_000_000u64 {
    mgr.handle_slot_update_receive(
        SlotUpdate {
            slot_number: SlotNumber::from(i),
            commit_id: CommitId::from(i),
            update: Update::Id(i), // distinct artifact ID per slot
        },
        NODE_1,
        ConnId::from(1),
        cancellation.clone(),
    );
}
// Assert: both grow to 1_000_000 with no bound
assert_eq!(mgr.active_assembles.len(), 1_000_000);
assert_eq!(mgr.artifact_processor_tasks.len(), 1_000_000);
```
