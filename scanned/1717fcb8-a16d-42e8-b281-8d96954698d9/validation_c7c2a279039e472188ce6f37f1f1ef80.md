### Title
Unauthenticated `buyer` Override in `refresh_buyer_tokens` Allows Forced Swap Participation Without Victim Consent - (File: rs/sns/swap/canister/canister.rs)

### Summary
The SNS Swap canister's `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal in its request payload without verifying that the caller is that principal. Any unprivileged ingress sender can supply a victim's principal as `buyer`, causing the swap canister to register the victim's ICP subaccount balance as confirmed swap participation — including bypassing the SNS-configured `confirmation_text` consent gate — and locking the victim's ICP in the swap without their explicit agreement.

### Finding Description
The `refresh_buyer_tokens` update method in the SNS Swap canister resolves the effective buyer principal as follows:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
``` [1](#0-0) 

When `arg.buyer` is non-empty, the caller's identity is completely ignored and the supplied principal is used verbatim. There is no check that `arg.buyer == caller`. The resolved principal `p` is then passed directly to `refresh_buyer_token_e8s`:

```rust
swap_mut()
    .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
    .await
``` [2](#0-1) 

Inside `refresh_buyer_token_e8s`, the function reads the ICP balance of `Account { owner: swap_canister, subaccount: principal_to_subaccount(&buyer) }` and, if it meets the minimum, records the victim's participation in `self.buyers` and validates the `confirmation_text` against the SNS-configured text: [3](#0-2) 

The `confirmation_text` is set at SNS initialization and is publicly readable from the swap state. An attacker who knows it (trivially, since it is on-chain) can supply it in the call.

Once `self.buyers` contains an entry for the victim with `transfer_success_timestamp_seconds == 0`, `error_refund_icp` will refuse to refund the victim's ICP:

```rust
if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
    if let Some(transfer) = &buyer_state.icp
        && transfer.transfer_success_timestamp_seconds == 0
    {
        return ErrorRefundIcpResponse::new_precondition_error(format!(
            "ICP cannot be refunded as principal {} has {} ICP (e8s) in escrow",
            ...
        ));
    }
``` [4](#0-3) 

The proto definition confirms the field is optional and caller-supplied:

```
message RefreshBuyerTokensRequest {
  // If not specified, the caller is used.
  string buyer = 1;
  optional string confirmation_text = 2;
}
``` [5](#0-4) 

### Impact Explanation
An attacker who observes that a victim has transferred ICP to the swap canister's subaccount (visible on the public ICP ledger) can immediately call `refresh_buyer_tokens` with `buyer = victim_principal` and the public `confirmation_text`. This:

1. Registers the victim as a confirmed swap participant, locking their ICP.
2. Blocks `error_refund_icp` from returning the victim's ICP until the swap finalizes.
3. If the swap commits, the victim receives SNS tokens instead of their ICP — an asset they may not want and cannot easily liquidate.
4. Bypasses the `confirmation_text` consent mechanism, which is the only explicit user-agreement gate in the participation flow.

The impact is a ledger conservation / governance authorization bug: the victim's ICP is irreversibly committed to a swap outcome without their consent.

### Likelihood Explanation
High. The precondition is only that the victim has transferred ICP to the swap subaccount (a normal, observable on-chain action). The attacker needs no special privileges, no private keys, and no admin access. The `confirmation_text` is public. Any ingress sender can exploit this immediately after observing the victim's ledger transfer.

### Recommendation
Add a caller-identity check before accepting a non-empty `buyer` field. Either:

- Reject calls where `arg.buyer` is non-empty and `arg.buyer != caller_principal_id()`, or
- Remove the ability to supply an arbitrary `buyer` entirely and always derive it from `caller_principal_id()`.

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let supplied = PrincipalId::from_str(&arg.buyer).unwrap();
    if supplied != caller_principal_id() {
        panic!("buyer must match caller");
    }
    supplied
};
```

### Proof of Concept

1. Alice transfers 5 ICP to `Account { owner: swap_canister, subaccount: principal_to_subaccount(alice) }` on the ICP ledger, intending to evaluate participation but not yet ready to commit.
2. The SNS swap has `confirmation_text = Some("I agree to the terms")` (publicly readable via `get_state`).
3. Attacker Bob calls:
   ```
   refresh_buyer_tokens({
     buyer: alice.to_string(),
     confirmation_text: Some("I agree to the terms")
   })
   ```
   from his own principal.
4. The swap canister resolves `p = alice`, reads Alice's subaccount balance (5 ICP ≥ `min_participant_icp_e8s`), validates the confirmation text (matches), and inserts Alice into `self.buyers` with `amount_icp_e8s = 5 * E8`.
5. Alice calls `error_refund_icp({ source_principal_id: alice })` — the canister returns `"ICP cannot be refunded as principal alice has 500000000 ICP (e8s) in escrow"`.
6. The swap commits; Alice receives SNS tokens she never consented to purchase. [6](#0-5) [7](#0-6)
