### Title
Stale Voting Power Snapshot Used for New Proposals Bypasses Current Stake Distribution - (`rs/nns/governance/src/governance/voting_power_snapshots.rs`)

### Summary

The NNS Governance canister's voting power spike detection mechanism uses historical snapshots (up to 7 days old) to create ballots for new proposals when a "spike" is detected. This is the direct IC analog of the Axelar M-04 bug: just as past operator sets could sign new commands after operatorship transfer, past (stale) voting power distributions can be used to authorize new governance proposals after the current stake distribution has changed significantly.

### Finding Description

The `VotingPowerSnapshots` struct in `rs/nns/governance/src/governance/voting_power_snapshots.rs` stores up to `MAX_VOTING_POWER_SNAPSHOTS = 7` daily snapshots of neuron voting power. When a new proposal is submitted, `compute_ballots_for_standard_proposal` checks whether the current total potential voting power exceeds 1.5× the minimum total potential voting power across all stored snapshots. If so, it falls back to using the **oldest snapshot with the minimum voting power** to create ballots for the new proposal.

The critical issue is in `totals_entry_with_minimum_total_potential_voting_power_if_voting_power_spiked`:

```rust
.filter(|(created_at, _)| {
    let age = now_seconds - created_at;  // ← plain subtraction, no saturating_sub
    age <= MAXIMUM_STALENESS_SECONDS
})
.min_by_key(|(_, snapshot)| snapshot.total_potential_voting_power)?;
```

**Root cause 1 — Integer underflow in staleness filter:** `now_seconds - created_at` is a plain `u64` subtraction. If `created_at > now_seconds` (which can happen if the canister's time source is manipulated or if a snapshot was recorded with a future timestamp due to a clock anomaly), this underflows to a very large value, causing the snapshot to be incorrectly treated as stale and excluded — or, conversely, included when it should not be.

**Root cause 2 — Stale distribution bypass (the primary analog):** The mechanism is designed to use the snapshot with the **minimum** total potential voting power when a spike is detected. An attacker who controls a large neuron can:

1. Wait until 7 daily snapshots have accumulated with low total voting power (e.g., before a large legitimate staking event).
2. Temporarily inflate their own neuron's stake (e.g., by staking a large amount of ICP) to trigger the spike detection threshold (>1.5× minimum).
3. Submit a governance proposal. The system falls back to the **oldest snapshot** (up to 7 days old), which reflects the old, lower stake distribution — one that may not include neurons that staked recently, or may include neurons that have since dissolved.
4. The proposal's ballots are created from the stale distribution. Neurons that staked recently are excluded from voting; neurons that dissolved recently may still appear in the ballot set.
5. The attacker then unstakes their ICP (after the proposal is submitted), suffering no lasting cost.

This mirrors the Axelar bug exactly: the past 7 stake distributions (snapshots) are all considered valid for authorizing new governance actions, even after the current distribution has changed.

Additionally, `SnapshotVotingPowerTask::execute` skips recording a new snapshot when `is_latest_snapshot_a_spike` returns true:

```rust
if self.snapshots.with_borrow(|snapshots| snapshots.is_latest_snapshot_a_spike(now_seconds)) {
    return (VOTING_POWER_SNAPSHOT_INTERVAL, self);
}
```

This means a sustained spike (e.g., from a large legitimate staking event) permanently freezes the snapshot set, causing all subsequent proposals to use the stale minimum-voting-power snapshot indefinitely — until the spike resolves or snapshots age out after 3 months (`MAXIMUM_STALENESS_SECONDS = ONE_MONTH_SECONDS * 3`).

### Impact Explanation

An unprivileged NNS governance participant (any neuron controller) can:

1. **Bypass current stake distribution**: Submit proposals whose ballots are computed from a stale snapshot that excludes recently-staked neurons or includes recently-dissolved ones, distorting the effective quorum and voting outcome.
2. **Freeze the governance snapshot indefinitely**: By maintaining a stake that keeps `is_latest_snapshot_a_spike` true, prevent new snapshots from being recorded, causing all proposals to use an increasingly stale distribution.
3. **Exclude legitimate voters**: Neurons that staked ICP after the minimum-voting-power snapshot was taken will not receive ballots for proposals submitted during a spike, effectively disenfranchising them.

The impact is governance authorization bypass — the current stake distribution is not used to authorize new governance actions, matching the severity class of the Axelar M-04 finding.

### Likelihood Explanation

The attack requires:
- Sufficient ICP to stake a neuron large enough to push total potential voting power above 1.5× the historical minimum (a significant but not impossible capital requirement on a live network).
- Waiting ~7 days for the snapshot window to fill with low-baseline snapshots.
- The ability to submit a governance proposal (requires a neuron with sufficient dissolve delay).

This is reachable by any unprivileged NNS participant with sufficient ICP. The cost is temporary (ICP can be unstaked after the proposal is submitted), making the attack economically viable for a motivated adversary.

### Recommendation

1. **Do not use stale snapshots for new proposals**: When a spike is detected, reject the proposal submission rather than falling back to a stale snapshot. The stale snapshot was designed to protect against flash-loan-style attacks, but it introduces the symmetric risk of using outdated distributions.
2. **Use `saturating_sub` for the staleness age calculation**: Replace `now_seconds - created_at` with `now_seconds.saturating_sub(*created_at)` to prevent integer underflow.
3. **Bound the snapshot age used for ballot creation**: If fallback to a previous snapshot is retained, restrict it to snapshots no older than 1–2 days rather than up to 3 months (`MAXIMUM_STALENESS_SECONDS`).
4. **Prevent snapshot freezing**: If the latest snapshot is a spike, still record a new snapshot (or use the current distribution for proposals), rather than halting snapshot collection entirely.

### Proof of Concept

**Setup**: Assume the NNS has been running for 7+ days with a stable total potential voting power of `X`.

**Attack at time T (after 7 snapshots accumulated)**:

1. Attacker stakes `0.51 * X` ICP in a new neuron, pushing total potential voting power to `1.51 * X > 1.5 * X` (spike threshold).
2. Attacker calls `manage_neuron` → `MakeProposal`. `compute_ballots_for_standard_proposal` is invoked.
3. `previous_ballots_if_voting_power_spike_detected(1.51 * X, now)` returns the snapshot with minimum total potential voting power (e.g., 7 days ago with value `X`).
4. Ballots are created from the 7-day-old snapshot. Any neuron that staked ICP in the last 7 days receives no ballot and cannot vote.
5. Attacker immediately begins dissolving their neuron (no lasting cost after the proposal is submitted).
6. The proposal proceeds with a ballot set that excludes all recently-staked neurons, potentially allowing the attacker's followees (who were in the old snapshot) to pass the proposal with a quorum that would not exist under the current distribution.

/// The maximum number of voting power snapshots to keep.
const MAX_VOTING_POWER_SNAPSHOTS: u64 = 7;
/// The multiplier used to define what is a "voting power spike": if the current total voting
/// power is more than this multiplier times the minimum total voting power in the snapshots,
/// then we consider it a spike.
const MULTIPLIER_THRESHOLD_FOR_VOTING_POWER_SPIKE: f64 = 1.5;
/// The maximum staleness of a voting power snapshot. This is usually not needed since
/// the snapshots should be added frequently. However, we do not want to use a snapshot that is too
/// old, in the event of a failure in taking the snapshots.
const MAXIMUM_STALENESS_SECONDS: u64 = ONE_MONTH_SECONDS * 3;

    pub(crate) fn previous_ballots_if_voting_power_spike_detected(
        &self,
        total_potential_voting_power: u64,
        now_seconds: TimestampSeconds,
    ) -> Option<(TimestampSeconds, VotingPowerSnapshot)> {
        // Step 0: skip the check in test mode when the snapshots are not yet full. Otherwise it
        // would be difficult to get around the spike detection in tests, and a lot of test setups
        // involve creating a lot of voting power.
        if cfg!(feature = "test") && self.voting_power_totals.len() < MAX_VOTING_POWER_SNAPSHOTS {
            return None;
        }

        // Step 1: find the voting power totals entry with the minimum total potential voting power,
        // if a spike is detected.
        let Some((
            timestamp_with_minimum_total_potential_voting_power,
            totals_with_minimum_total_potential_voting_power,
        )) = self.totals_entry_with_minimum_total_potential_voting_power_if_voting_power_spiked(
            now_seconds,
            total_potential_voting_power,
        )
        else {
            eprintln!(
                "{}Voting power totals are empty. No voting power spike detected.",
                LOG_PREFIX,
            );
            return None;
        };

        // Step 2: find the voting power map for the timestamp with the minimum potential voting power.
        let Some(voting_power_map) = self
            .neuron_id_to_voting_power_maps
            .get(&timestamp_with_minimum_total_potential_voting_power)
        else {
            eprintln!(
                "{}Voting power map not found for timestamp {} while the totals \
                are found. This should not happen.",
                LOG_PREFIX, timestamp_with_minimum_total_potential_voting_power,
            );
            return None;
        };

        // Step 3: returns one of the previous voting power maps (with minimum total potential
        // voting power) since a voting power spike is detected.
        let previous_voting_power_snapshot = VotingPowerSnapshot::from((
            voting_power_map,
            totals_with_minimum_total_potential_voting_power,
        ));
        println!(
            "{}Voting power spike detected at timestamp {}, total potential voting power: {}, \
            minimum total potential voting power: {}",
            LOG_PREFIX,
            timestamp_with_minimum_total_potential_voting_power,
            total_potential_voting_power,
            totals_with_minimum_total_potential_voting_power.total_potential_voting_power
        );
        Some((
            timestamp_with_minimum_total_potential_voting_power,
            previous_voting_power_snapshot,
        ))
    }

    fn compute_ballots_for_standard_proposal(
        &self,
        now_seconds: u64,
    ) -> Result<
        (
            HashMap<u64, Ballot>,
            u64,         /*potential_voting_power*/
            Option<u64>, /*previous_ballots_timestamp_seconds*/
        ),
        GovernanceError,
    > {
        let current_voting_power_snapshot = self
            .neuron_store
            .compute_voting_power_snapshot_for_standard_proposal(
                self.voting_power_economics(),
                now_seconds,
            )?;

        // Check if there is a voting power spike. If there is, then the return value here
        // will be `Some(...)`.
        let maybe_previous_ballots_if_voting_power_spike_detected = VOTING_POWER_SNAPSHOTS
            .with_borrow(|snapshots| {
                snapshots.previous_ballots_if_voting_power_spike_detected(
                    current_voting_power_snapshot.total_potential_voting_power(),
                    now_seconds,
                )
            });

        let (voting_power_snapshot, previous_ballots_timestamp_seconds) =
            match maybe_previous_ballots_if_voting_power_spike_detected {
                // This is the extraordinary case - we have a voting power spike, and we
                // need to use the previous snapshot.
                Some((previous_snapshot_timestamp, previous_snapshot)) => {
                    (previous_snapshot, Some(previous_snapshot_timestamp))
                }
                // This is the normal case - we have no voting power spike, so we use the
                // current snapshot.
                None => (current_voting_power_snapshot, None),
            };

        let (ballots, total_potential_voting_power) =
            voting_power_snapshot.create_ballots_and_total_potential_voting_power();
        Ok((
            ballots,
            total_potential_voting_power,
            previous_ballots_timestamp_seconds,
        ))
    }
