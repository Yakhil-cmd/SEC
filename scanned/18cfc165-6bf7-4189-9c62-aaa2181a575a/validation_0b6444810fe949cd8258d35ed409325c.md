### Title
Hard-Coded `minimum_fee_per_vbyte` Floor in ckBTC Minter Silently Inflates Fees and Blocks Withdrawals - (File: rs/bitcoin/ckbtc/minter/src/fees/mod.rs)

---

### Summary

The `BitcoinFeeEstimator::minimum_fee_per_vbyte()` function hard-codes a minimum fee floor of 1,500 millisat/vbyte (1.5 sat/vbyte) for Bitcoin Mainnet. This floor is unconditionally applied in `estimate_nth_fee`, overriding the actual market fee whenever it falls below the threshold. The floored fee propagates into `fee_based_retrieve_btc_min_amount`, which is the live gate used by both `retrieve_btc` and `retrieve_btc_with_approval` to accept or reject user withdrawal requests. When actual Bitcoin network fees are legitimately below 1.5 sat/vbyte, users either pay more in transaction fees than the market requires (fund loss) or have small-but-otherwise-valid withdrawals rejected (availability loss). Neither outcome can be avoided by the user, and the floor cannot be tuned without a full canister upgrade.

---

### Finding Description

`BitcoinFeeEstimator::minimum_fee_per_vbyte()` returns a hard-coded constant:

```rust
// rs/bitcoin/ckbtc/minter/src/fees/mod.rs  lines 80-87
const fn minimum_fee_per_vbyte(&self) -> FeeRate {
    let rate = match &self.network {
        Network::Mainnet => 1_500,   // 1.5 sat/vbyte
        Network::Testnet => 1_000,
        Network::Regtest => 0,
    };
    FeeRate::from_millis_per_byte(rate)
}
```

This floor is applied unconditionally in `estimate_nth_fee`:

```rust
// line 112
median_fee.map(|f| f.max(self.minimum_fee_per_vbyte()))
```

`estimate_nth_fee` is called by `estimate_median_fee` (nth = 50), which is called by `estimate_fee_per_vbyte`. That function writes the floored value into two state fields:

```rust
// rs/bitcoin/ckbtc/minter/src/lib.rs  lines 245-248

    s.last_fee_per_vbyte = fees;

```

`fee_based_retrieve_btc_min_amount` is then used as the hard gate in both withdrawal entry points:

```rust
// rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs  lines 166-171

The same check exists in `retrieve_btc_with_approval` at lines 264–270.

Additionally, `fee_based_minimum_withdrawal_amount` contains a second hard-coded constant, `50_000`, that rounds the computed minimum up to the nearest 50,000-satoshi boundary regardless of the configured `retrieve_btc_min_amount`:

```rust

    + PER_REQUEST_MINTER_FEE_BOUND
    + self.check_fee)

```

The TODO comment in the source acknowledges this should be made proportional to `retrieve_btc_min_amount / 2` but has not been fixed.

The floor was introduced as a stop-gap after three mainnet withdrawals became stuck on 2025-06-21 because the minter selected an anomalously low fee of 142 millisat/vbyte. The upgrade proposal (`rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`) explicitly states: *"We currently don't have a satisfying explanation for how this low median fee was computed."* The root cause is unresolved, and the floor remains in production with no mechanism to tune it short of a governance-approved canister upgrade.

---

### Impact Explanation

Two concrete harms arise when actual Bitcoin network fees fall below 1.5 sat/vbyte:

1. **User fund loss (over-charged fees).** The minter builds Bitcoin transactions using the floored fee rate instead of the real market rate. The excess fee is deducted from the user's BTC output. The user receives less BTC than the market rate would require, with no disclosure that the fee was inflated by a hard-coded floor rather than by actual network conditions. The `estimate_withdrawal_fee` query also returns the inflated estimate, so users cannot detect the discrepancy without independently querying the Bitcoin network.

2. **Withdrawal unavailability.** `fee_based_retrieve_btc_min_amount` is computed from the floored fee. When fees are low, the floor inflates this minimum. Users whose withdrawal amounts are above the true fee-based minimum but below the floor-inflated minimum receive `AmountTooLow` errors and cannot withdraw. The 50,000-satoshi rounding step compounds this: a user whose amount exceeds the true minimum by up to 49,999 satoshi may still be rejected.

Both outcomes match the vulnerability class of the reference report: a hard-coded threshold that cannot be tuned causes either silent fund loss or operation unavailability.

---

### Likelihood Explanation

The event that triggers the vulnerability — Bitcoin network fees falling below 1.5 sat/vbyte — already occurred on mainnet on 2025-06-21, causing three real withdrawals to become stuck. The root cause (why the minter computed 142 millisat/vbyte when the true median was ~2,000) is explicitly unresolved in the upgrade proposal. If the same anomaly recurs, the floor will again be the active fee rate, and users will again be over-charged or blocked. Bitcoin fees have also historically been near or below 1 sat/vbyte during low-activity periods, making this a realistic scenario independent of any minter bug.

---

### Recommendation

1. Expose `minimum_fee_per_vbyte` as a configurable field in `UpgradeArgs` (analogous to `retrieve_btc_min_amount`, `check_fee`, etc.) so it can be adjusted via NNS governance proposal without a full code change.
2. Replace the hard-coded `50_000` rounding increment in `fee_based_minimum_withdrawal_amount` with `retrieve_btc_min_amount / 2` as the TODO comment already suggests (DEFI-2187).
3. Resolve the root cause of the anomalous fee estimate so the floor can eventually be removed or set to a safe, well-justified value.

### Proof of Concept

**Step 1 – Trigger condition.** The Bitcoin canister returns fee percentiles where the 50th percentile is below 1,500 millisat/vbyte (e.g., 142 millisat/vbyte as observed on 2025-06-21).

**Step 2 – Floor applied.** `estimate_nth_fee` clamps the returned value to 1,500:
```
median_fee = max(142, 1_500) = 1_500 millisat/vbyte
```

**Step 3 – Minimum withdrawal inflated.** With `retrieve_btc_min_amount = 50_000` and `check_fee = 1_000`:
```
fee_based_min = ((22_100 + ceil(1_500 * 221 / 1_000) + 305 + 1_000) / 50_000) * 50_000 + 50_000
              = ((22_100 + 332 + 305 + 1_000) / 50_000) * 50_000 + 50_000
              = (23_737 / 50_000) * 50_000 + 50_000
              = 0 + 50_000 = 50_000
```
At the true market fee of 142 millisat/vbyte the result would be the same base, but the transaction fee charged to the user is computed using 1,500 instead of 142, meaning the user pays ~10× more in Bitcoin fees than the network requires.

**Step 4 – User impact.** A user calling `retrieve_btc_with_approval` with an amount just above the true fee-based minimum but below the floor-inflated minimum receives `AmountTooLow`. A user whose amount clears the minimum receives less BTC than expected because the transaction is built with the inflated fee rate.

The test at `rs/bitcoin/ckbtc/minter/tests/tests.rs` lines 2882–2887 explicitly demonstrates the floor behavior:
```rust

    FeeRate::from_millis_per_byte(1_500),  // floor applied
