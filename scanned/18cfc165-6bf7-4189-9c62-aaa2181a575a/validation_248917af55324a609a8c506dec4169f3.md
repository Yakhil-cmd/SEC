### Title
Global Migration Rate Limit Exhaustion Without Caller Fee Enables 24-Hour DOS - (File: `rs/migration_canister/src/lib.rs`, `rs/migration_canister/src/migration_canister.rs`)

---

### Summary

The `migration_canister` enforces a single global `RATE_LIMIT = 50` shared across all callers within a 24-hour sliding window. Because no fee is charged to the **caller** of `migrate_canister`, an unprivileged principal who controls 50 pairs of canisters can exhaust this global limit, blocking all other users from submitting migration requests for up to 24 hours.

---

### Finding Description

The `migrate_canister` update endpoint in `rs/migration_canister/src/migration_canister.rs` enforces a global rate limit via `rate_limited()`:

```rust
pub fn rate_limited() -> bool {
    num_requests() + num_successes_in_past_24_h() >= RATE_LIMIT
}
``` [1](#0-0) 

`RATE_LIMIT` is a constant of `50`, shared globally across every caller. [2](#0-1) 

The `migrate_canister` function checks this limit before and after validation, but charges **no fee to the caller**:

```rust
#[update]
async fn migrate_canister(args: MigrateCanisterArgs) -> Result<(), Option<ValidationError>> {
    ...
    if rate_limited() {
        return Err(Some(ValidationError::RateLimited(Reserved)));
    }
    ...
    if rate_limited() {
        return Err(Some(ValidationError::RateLimited(Reserved)));
    }
    insert_request(RequestState::Accepted { request });
    Ok(())
}
``` [3](#0-2) 

The only cycle-related check is a **minimum balance requirement on the migrated canister** (`migrated_canister_status.cycles < CYCLES_COST_PER_MIGRATION`), not a fee deducted from the caller: [4](#0-3) 

The allowlist is optional. When initialized with `allowlist: None`, `caller_allowed` returns `true` for any principal: [5](#0-4) 

Successful migrations are recorded in `LIMITER` and counted by `num_successes_in_past_24_h()` for the full 24-hour window: [6](#0-5) 

---

### Impact Explanation

Once an attacker submits 50 accepted migration requests (active or recently succeeded), `rate_limited()` returns `true` for all subsequent callers. Every call to `migrate_canister` by any other user returns `ValidationError::RateLimited` for up to 24 hours. This completely blocks the canister migration service for all legitimate users during that window.

The test `parallel_migrations` confirms that the 51st request is rejected with `RateLimited`: [7](#0-6) 

---

### Likelihood Explanation

The attack requires the attacker to:
1. Control 50 pairs of canisters (each migrated canister with ≥ 10T cycles, both stopped, migration canister added as controller).
2. Submit 50 `migrate_canister` calls — cheap IC ingress messages.

The cycles in the migrated canisters are a prerequisite on the **attacker's own canisters**, not a fee burned to the protocol. The attacker retains control of the canisters and can repeat the attack after each 24-hour window. The economic cost is low relative to the disruption caused. Any principal on the IC (when the allowlist is `None`) can execute this attack.

---

### Recommendation

1. **Introduce a per-caller sub-limit**: Track how many migrations each caller has submitted in the past 24 hours and cap it (e.g., 5 per caller), so no single principal can exhaust the global limit.
2. **Charge a caller-side fee**: Accept cycles from the `migrate_canister` caller (via `ic_cdk::api::call::msg_cycles_accept128`) proportional to the migration cost, making bulk DOS economically unattractive.
3. **Separate active-request count from success count**: Active in-flight requests should not count toward the 24-hour success window, so a failed or cancelled batch of requests does not permanently consume quota.

---

### Proof of Concept

**Setup**: Attacker creates 50 pairs of canisters `(M_i, R_i)` for `i = 1..50`. For each pair:
- Adds the migration canister as a controller of both `M_i` and `R_i`.
- Stops both canisters.
- Ensures `M_i` holds ≥ 10T cycles.
- Ensures `M_i` and `R_i` are on different subnets.

**Attack**:
```
for i in 1..=50:
    migrate_canister({ migrated_canister_id: M_i, replaced_canister_id: R_i })
    // Each call succeeds → inserted into REQUESTS
```

After 50 calls, `num_requests() = 50 >= RATE_LIMIT`. Any subsequent call by any other user:
```
migrate_canister({ migrated_canister_id: victim_canister, replaced_canister_id: victim_replaced })
// → Err(Some(ValidationError::RateLimited(_)))
``` [8](#0-7) 

Once the 50 requests complete successfully, `num_successes_in_past_24_h() = 50 >= RATE_LIMIT` for the next 24 hours, sustaining the DOS even after the active requests clear: [2](#0-1)

#[update]
async fn migrate_canister(args: MigrateCanisterArgs) -> Result<(), Option<ValidationError>> {
    if migrations_disabled() {
        return Err(Some(ValidationError::MigrationsDisabled(Reserved)));
    }
    // Prevent too many interleaved validations.
    let Ok(_guard) = ValidationGuard::new() else {
        return Err(Some(ValidationError::RateLimited(Reserved)));
    };
    if rate_limited() {
        return Err(Some(ValidationError::RateLimited(Reserved)));
    }
    let caller = msg_caller();
    // For soft rollout purposes
    if !caller_allowed(&caller) {
        return Err(Some(ValidationError::MigrationsDisabled(Reserved)));
    }
    match validate_request(args.migrated_canister_id, args.replaced_canister_id, caller).await {
        Err(e) => {
            println!("Failed to validate request {}: {}", args, e);
            return Err(Some(e));
        }
        Ok((request, _guards)) => {
            // Need to check the rate limit again
            if rate_limited() {
                return Err(Some(ValidationError::RateLimited(Reserved)));
            }
            println!("Accepted request {}", request);
            insert_request(RequestState::Accepted { request });
        }
    }
    Ok(())
}
