### Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_end` in `construction_payloads` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

---

### Summary

The `construction_payloads` function in the ICRC1 Rosetta server accepts attacker-controlled `ingress_start` and `ingress_end` values from the HTTP request body and enters an unbounded `while` loop that pushes one `u64` per iteration into a `Vec`. No guard caps the window size `(ingress_end - ingress_start)`. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes the server to attempt allocating hundreds of gigabytes of memory, triggering OOM or, due to u64 wraparound in release mode, an infinite loop that permanently hangs the handler task.

---

### Finding Description

**Entry point:** `POST /construction/payloads` → `endpoints::construction_payloads` → `services::construction_payloads`.

The endpoint handler passes the raw metadata fields directly to the service function with no pre-validation of the window size: [1](#0-0) 

Inside `services::construction_payloads`, the two guards that exist only check ordering and minimum future reach — neither bounds the window: [2](#0-1) 

With `ingress_start=0` and `ingress_end=u64::MAX`, both guards pass (`0 < u64::MAX` and `u64::MAX >= now + ingress_interval`). Execution then falls into the unbounded loop: [3](#0-2) 

**Step size calculation:**
- `ingress_interval = MAX_INGRESS_TTL − PERMITTED_DRIFT = 300s − 60s = 240s = 240_000_000_000 ns`
- `INGRESS_INTERVAL_OVERLAP = 120s = 120_000_000_000 ns`
- Effective step per iteration = `240_000_000_000 − 120_000_000_000 = 120_000_000_000 ns` [4](#0-3) [5](#0-4) 

**Iteration count with `ingress_end = u64::MAX`:**
`u64::MAX / 120_000_000_000 ≈ 153,722,867,280` iterations — each pushing 8 bytes. The server OOMs long before completion. Additionally, because `ingress_start += step` eventually overflows `u64` in Rust release mode (wrapping semantics), the wrapped value falls below `u64::MAX` again, converting the OOM into a **permanent infinite loop** on the handler thread/task.

The same pattern exists in the ICP Rosetta server: [6](#0-5) 

---

### Impact Explanation

A single unauthenticated HTTP POST to `/construction/payloads` with `metadata.ingress_start=0` and `metadata.ingress_end=18446744073709551615` causes the Rosetta server process to exhaust all available heap memory and be killed by the OS OOM killer, or to spin indefinitely on the handler task. All concurrent and subsequent requests are denied service. Exchange integrations relying on the Rosetta construction API for transfer signing lose the ability to submit transactions for the duration of the outage.

---

### Likelihood Explanation

The endpoint is publicly reachable with no authentication. The payload is a standard JSON body. The attack requires a single HTTP request and is trivially reproducible. No privileged access, key material, or protocol-level attack is needed.

---

### Recommendation

Add an explicit upper bound on the ingress window before entering the loop. For example:

```rust
const MAX_INGRESS_WINDOW_ENTRIES: usize = 100; // ~200 minutes at 120s step

let window_size = ingress_end.saturating_sub(ingress_start);
let max_window = (MAX_INGRESS_WINDOW_ENTRIES as u64)
    .saturating_mul(ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64));
if window_size > max_window {
    return Err(Error::processing_construction_failed(
        "Ingress window exceeds maximum allowed size"
    ));
}
```

Apply the same fix to the ICP Rosetta server's `construction_payloads` in `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`.

---

### Proof of Concept

```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<network_id>"},
    "operations": [<valid_transfer_op>],
    "public_keys": [<valid_pubkey>],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

Expected: server process is killed by OOM or the handler task spins indefinitely; all subsequent requests to the Rosetta server time out or are refused.
