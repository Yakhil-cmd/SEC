### Title
Integer Division Truncation in `memory_cost` Causes Zero Storage Fee for Small Memory Canisters - (File: `rs/cycles_account_manager/src/cycles_account_manager.rs`)

### Summary
The `memory_cost` function in `CyclesAccountManager` performs integer division by `one_gib` (1,073,741,824) before a second integer division in `scale_cost` by `reference_subnet_size`. For canisters with small memory usage, both truncations compound to produce a zero memory-proportional storage fee, directly analogous to the Beedle `_calculateInterest()` rounding-to-zero bug.

### Finding Description
`memory_cost` computes the storage fee as:

```rust
// rs/cycles_account_manager/src/cycles_account_manager.rs, lines 652–658
let one_gib = 1024 * 1024 * 1024;
let cycles = Cycles::from(
    (bytes.get() as u128
        * self.config.gib_storage_per_second_fee.get()
        * duration.as_secs() as u128)
        / one_gib,   // ← first integer truncation
);
self.scale_cost(cycles, subnet_cycles_config)
```

`scale_cost` then performs a second integer division:

```rust
// rs/cycles_account_manager/src/cycles_account_manager.rs, lines 115–116
let real =
    (cycles * subnet_cycles_config.subnet_size) / self.config.reference_subnet_size.max(1);
//                                                ↑ second integer truncation
```

With the production values from `CyclesAccountManagerConfig::application_subnet`:
- `gib_storage_per_second_fee = 317_500` cycles/GiB/s
- `duration_between_allocation_charges = 10` s
- `reference_subnet_size = DEFAULT_REFERENCE_SUBNET_SIZE` (13 for a standard app subnet)

For the first division alone to produce a non-zero result:
```
bytes × 317_500 × 10 ≥ 1_073_741_824
bytes ≥ 338 bytes
```

For the result to survive the second division by 13:
```
(bytes × 317_500 × 10 / 1_073_741_824) × subnet_size / 13 ≥ 1
bytes ≥ 13 × 1_073_741_824 / 3_175_000 ≈ 4_394 bytes
```

Any canister whose memory usage stays below ~4,394 bytes pays **zero** memory-proportional storage cycles per charging period.

The same `memory_cost` path feeds `idle_cycles_burned_rate_by_resource`, which feeds `freeze_threshold_cycles`. Therefore the freeze threshold is also underestimated for such canisters, allowing them to hold a lower cycle balance without being frozen. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
A canister developer can deploy a minimal canister whose total memory footprint stays below ~4,394 bytes and pay **zero** memory-proportional storage fee indefinitely. The `base_per_second_fee` (10,000 cycles/s) still applies, so the canister is not entirely free, but the memory-proportional component — which is the mechanism that scales cost with actual resource consumption — is silently zeroed out. Additionally, `freeze_threshold_cycles` underestimates the required cycle reserve for such canisters, meaning they can operate with a lower balance than the protocol intends, effectively subsidising their operation at the expense of the subnet's economic model. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
The entry path requires no privilege: any unprivileged principal can create a canister via the management canister interface and keep its memory footprint small. The threshold (~4,394 bytes) is achievable with a minimal Wasm module that stores only a few bytes of stable or heap memory. The rounding is deterministic and reproducible on every charging interval (every 10 seconds), so the fee bypass is persistent, not transient. [6](#0-5) 

### Recommendation
Reorder the arithmetic so that multiplication by `subnet_size` occurs **before** division by `one_gib`, eliminating the first truncation for realistic memory sizes:

```rust
let cycles = Cycles::from(
    bytes.get() as u128
        * self.config.gib_storage_per_second_fee.get()
        * duration.as_secs() as u128
        * subnet_cycles_config.subnet_size as u128
        / one_gib
        / self.config.reference_subnet_size.max(1) as u128,
);
```

Alternatively, introduce a minimum fee floor (e.g., 1 cycle) whenever `bytes > 0` and `duration > 0`, mirroring the `canister_base_cost` guard already present at line 668. [5](#0-4) 

### Proof of Concept
Using production config values (`gib_storage_per_second_fee = 317_500`, `duration = 10 s`, `reference_subnet_size = 13`, `subnet_size = 13`):

```
bytes = 4_000  (a minimal canister with 4 KB memory)

Step 1 – memory_cost inner division:
  4_000 × 317_500 × 10 / 1_073_741_824
= 12_700_000_000 / 1_073_741_824
= 11  (truncated from 11.82)

Step 2 – scale_cost division:
  11 × 13 / 13 = 11   ← non-zero, but only barely

bytes = 3_000  (3 KB memory)
  3_000 × 317_500 × 10 / 1_073_741_824
= 9_525_000_000 / 1_073_741_824
= 8  (truncated)
  8 × 13 / 13 = 8   ← still non-zero

bytes = 300  (300 bytes memory)
  300 × 317_500 × 10 / 1_073_741_824
= 952_500_000 / 1_073_741_824
= 0  ← truncated to zero; scale_cost returns 0 regardless of subnet_size
```

A canister holding 300 bytes of memory is charged **0 cycles** for memory storage every 10-second charging period, indefinitely. [7](#0-6) [3](#0-2)

    // Scale cycles cost according to a subnet size.
    fn scale_cost<T: CyclesUseCaseKind>(
        &self,
        cycles: Cycles,
        subnet_cycles_config: CyclesAccountManagerSubnetConfig,
    ) -> CompoundCycles<T> {
        debug_assert_ne!(
            self.config.reference_subnet_size, 0,
            "prevent divide by zero panic"
        );
        let real =
            (cycles * subnet_cycles_config.subnet_size) / self.config.reference_subnet_size.max(1);
        CompoundCycles::<T>::new(real, subnet_cycles_config.cost_schedule)
    }
