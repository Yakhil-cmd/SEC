### Title
Attacker Can Permanently Censor Victim's ckBTC Deposits by Front-Running `update_balance` with Victim's `owner` - (`File: rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

### Summary
The ckBTC minter's `update_balance` endpoint accepts an optional `owner: Option<Principal>` parameter that allows any caller to trigger a deposit-processing flow on behalf of an arbitrary account. The function acquires a per-account concurrency guard keyed on the attacker-supplied `owner` before making async inter-canister calls. An unprivileged attacker can continuously call `update_balance` with a victim's principal as `owner`, keeping the victim's account locked in `update_balance_accounts` indefinitely and preventing the victim from ever minting ckBTC from their Bitcoin deposits.

### Finding Description
`update_balance` in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` constructs the target account from the caller-supplied `args.owner`:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),  // attacker controls this
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;  // locks victim's account
``` [1](#0-0) 

The guard implementation inserts the account into a global `BTreeSet<Account>` (`update_balance_accounts`) and holds it for the entire duration of the async call, which includes at least one round-trip to the Bitcoin canister via `get_utxos`:

```rust
let utxos = get_utxos(btc_network, &address, min_confirmations, CallSource::Client, runtime)
    .await?
    .utxos;
``` [2](#0-1) 

The guard is only released on `Drop`:

```rust
impl<PR: PendingRequests> Drop for Guard<PR> {
    fn drop(&mut self) {
        mutate_state(|s| PR::pending_requests(s).remove(&self.account));
    }
}
``` [3](#0-2) 

While the guard is held, any concurrent call to `update_balance` for the same account returns `GuardError::AlreadyProcessing`:

```rust
if accounts.contains(&account) {
    return Err(GuardError::AlreadyProcessing);
}
``` [4](#0-3) 

which surfaces to the victim as `UpdateBalanceError::AlreadyProcessing`. [5](#0-4) 

The `UpdateBalanceArgs` struct explicitly exposes `owner` as an optional field in the public Candid interface:

```rust
pub struct UpdateBalanceArgs {
    pub owner: Option<Principal>,   // any caller can set this to any principal
    pub subaccount: Option<Subaccount>,
}
``` [6](#0-5) 

There is no authorization check preventing a caller from specifying a different principal as `owner`. The only guard is against specifying the minter's own canister ID:

```rust
if args.owner.unwrap_or(caller) == runtime.id() {
    ic_cdk::trap("cannot update minter's balance");
}
``` [7](#0-6) 

### Impact Explanation
**Impact: High.** A victim who has sent BTC to their ckBTC deposit address cannot call `update_balance` to mint ckBTC. Their Bitcoin funds are locked on-chain with no way to convert them to ckBTC tokens. The attacker can maintain this censorship indefinitely at low cost: each attacker call to `update_balance` with the victim's `owner` will quickly return `NoNewUtxos` (since the attacker's address has no UTXOs at the victim's derived BTC address), but the guard is held for the full async round-trip to the Bitcoin canister. The attacker simply re-calls immediately after each guard release. The `MAX_CONCURRENT` limit of 100 also means an attacker can simultaneously lock up to 100 distinct victim accounts. [8](#0-7) 

### Likelihood Explanation
**Likelihood: Medium.** The attack requires no special privileges — any ingress sender or canister can call `update_balance` with an arbitrary `owner`. The victim's principal is publicly observable (e.g., from their ckBTC ledger account or prior on-chain activity). The attack is cheap: the attacker pays only the cycles cost of repeated inter-canister calls to the Bitcoin canister, with no minimum deposit or stake required. Protocols or whales depositing large BTC amounts are natural targets.

### Recommendation
Restrict `update_balance` so that the effective `owner` must equal the caller, removing the ability to trigger the guard on behalf of another principal:

```rust
let caller_account = Account {
    owner: caller,  // always use caller, ignore args.owner
    subaccount: args.subaccount,
};
```

If third-party deposit notification is a desired feature (e.g., a smart contract depositing on behalf of a user), introduce a separate, non-locking query path or require the owner to pre-authorize the caller via an allowlist.

### Proof of Concept
1. Victim `V` sends BTC to their ckBTC deposit address and waits for confirmations.
2. Attacker `A` (any unprivileged principal) calls `update_balance({ owner: Some(V), subaccount: None })` on the ckBTC minter.
3. The minter constructs `caller_account = { owner: V, subaccount: None }` and inserts it into `update_balance_accounts` via `balance_update_guard`.
4. The minter makes an async `get_utxos` call to the Bitcoin canister for V's derived address. The guard is held during this round-trip.
5. Victim `V` calls `update_balance({ owner: None, subaccount: None })`. The minter constructs the same `caller_account = { owner: V, subaccount: None }`, finds it already in `update_balance_accounts`, and returns `UpdateBalanceError::AlreadyProcessing`.
6. Attacker `A`'s call completes with `UpdateBalanceError::NoNewUtxos` (A has no UTXOs at V's address). The guard is released.
7. Attacker `A` immediately re-calls step 2. Steps 3–7 repeat indefinitely.
8. Victim `V` can never successfully call `update_balance`; their BTC deposit is permanently inaccessible as ckBTC.
