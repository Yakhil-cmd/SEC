### Title
Unbounded heap allocation via oversized `StateSyncId.hash` in `state_sync_advert_handler` — (`rs/p2p/state_sync_manager/src/routes/advert.rs`)

### Summary

`state_sync_advert_handler` decodes an attacker-controlled `pb::Advert` payload with no size guard on the `StateSyncId.hash` bytes field. The QUIC transport caps each raw stream at **128 MiB** (not 8 MiB as the question states — the actual limit is larger, making the impact worse). With a channel capacity of 20, a Byzantine subnet peer can queue up to 20 × ~128 MiB ≈ **2.56 GiB** of decoded `Advert` structs on the victim replica's heap.

### Finding Description

**No hash size validation in the handler:**

`state_sync_advert_handler` calls `pb::Advert::decode(payload)` directly, then `Advert::try_from(advert)`, with no check on the size of `id.hash` at any point. [1](#0-0) 

**`TryFrom<pb::Advert>` also performs no size check:**

The conversion simply wraps `id.hash` into a `CryptoHash(Vec<u8>)` without bounding it. [2](#0-1) 

**`StateSyncId.hash` is an unconstrained `bytes` field in the proto schema:** [3](#0-2) 

The generated Rust struct confirms `hash` is a plain `Vec<u8>` with no length constraint: [4](#0-3) 

**The QUIC transport limit is 128 MiB, not 8 MiB:**

The question understates the per-message cap. The actual value is: [5](#0-4) 

`read_request` enforces this limit on the raw stream, but passes the full body to the handler without further field-level validation: [6](#0-5) 

**The advert channel has capacity 20:**

Decoded `Advert` structs (with their full `Vec<u8>` hash) sit in this bounded channel until the manager loop consumes them: [7](#0-6) 

**The QUIC request handler spawns a task per stream**, so multiple concurrent requests can be in-flight simultaneously, each holding a decoded payload in memory while awaiting channel space: [8](#0-7) 

### Impact Explanation

A single Byzantine subnet peer can:
1. Craft `pb::Advert { id: Some(StateSyncId { height: 1, hash: vec![0u8; 128*1024*1024 - overhead] }) }`.
2. Send 20 such adverts concurrently.
3. Each decoded `Advert` (with ~128 MiB `Vec<u8>` hash) is queued in the channel.
4. Peak heap from the channel alone: **~2.56 GiB**, plus additional in-flight allocations from concurrent QUIC tasks awaiting channel space.

This can cause OOM on the victim replica, crashing it and removing it from the subnet's active set — a targeted availability attack.

### Likelihood Explanation

The attacker must be an authenticated subnet peer (a Byzantine node). The IC protocol is designed to tolerate Byzantine nodes, so a single compromised node causing OOM on a peer is within the realistic threat model. No threshold corruption is required — one Byzantine node suffices.

### Recommendation

Add a hash size check immediately after decoding, before the `Advert` is constructed or sent to the channel. The legitimate `CryptoHashOfState` is always a fixed 32-byte SHA-256 digest. Reject any advert whose `id.hash` length differs from 32 bytes:

```rust
// In TryFrom<pb::Advert> for Advert, or directly in state_sync_advert_handler:
if advert.id.as_ref().map_or(0, |id| id.hash.len()) != 32 {
    return Err(StatusCode::BAD_REQUEST);
}
```

Alternatively, enforce a proto-level field constraint or add a `MAX_HASH_SIZE` guard in `state_sync_advert_handler` before calling `pb::Advert::decode`.

### Proof of Concept

```rust
// Attacker (Byzantine subnet peer) sends 20 concurrent requests:
let hash = vec![0u8; 128 * 1024 * 1024 - 64]; // near MAX_MESSAGE_SIZE_BYTES
let advert = pb::Advert {
    id: Some(pb::StateSyncId { height: 1, hash }),
};
let payload = advert.encode_to_vec();
// Send payload to /state-sync/advert 20 times concurrently via QUIC transport.
// Each decoded Advert with ~128 MiB hash sits in the mpsc channel (capacity 20).
// Total heap pressure: ~2.56 GiB → OOM on victim replica.
```
