### Impact Explanation

`todo!()` in Rust calls `panic!()`. In a tokio async runtime (which the Rosetta node uses), a panic inside a spawned request-handling task is caught at the task boundary by tokio's scheduler, so the **process itself does not necessarily terminate**. However:

- The panicking task is aborted; the client receives a connection-level error (reset/close) rather than a structured API error.
- The invariant that "a well-formed client request must never cause a panic" is violated.
- If the server framework does **not** wrap handlers in `catch_unwind` (e.g., uses `spawn_blocking` or inline async without a panic boundary), the panic propagates to the tokio worker thread and **can** crash the process.

The realistic worst-case is a per-request panic causing connection resets for all clients attempting hash-only Rosetta block lookups, with a non-zero probability of process termination depending on the exact server wiring.

### Likelihood Explanation

- Requires only a valid HTTP client and knowledge of the `/block` endpoint (public Rosetta API spec).
- `RosettaBlocksMode::Enabled` is the intended production configuration for ICP Rosetta nodes that support the Rosetta block format.
- Any hash value that is not a verified ICP block hash (including a valid Rosetta block hash) triggers the panic — the attacker does not need to know any specific hash; a random 32-byte hex string suffices.
