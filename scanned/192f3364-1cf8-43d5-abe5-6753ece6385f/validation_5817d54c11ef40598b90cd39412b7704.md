### Title
Unretracted Byzantine Peer Advertisement Blocks `Remove` Mutation Indefinitely — (`rs/p2p/consensus_manager/src/receiver.rs`)

---

### Summary

A single Byzantine peer (below the consensus fault threshold) that advertises an artifact on a slot and never retracts it causes the corresponding assemble task to block permanently at `peer_rx.wait_for(|p| p.is_empty()).await` after sending `Insert` to the unvalidated pool. The `Remove` mutation is never sent, leaving the artifact in the unvalidated pool indefinitely. The developers have explicitly flagged this with `// TODO: NET-1774` at both call sites.

---

### Finding Description

In `process_slot_update`, after `assemble_artifact` resolves with `AssembleResult::Done`, the code sends `Insert` to the unvalidated pool and then unconditionally awaits:

```rust
// wait for deletion from peers
// TODO: NET-1774
let _ = peer_rx.wait_for(|p| p.is_empty()).await;
``` [1](#0-0) 

This `wait_for` is **outside** the outer `select!` that guards the cancellation token. The outer `select!` only races `cancellation_token.cancelled()`, `assemble_artifact`, and `all_peers_deleted_artifact`: [2](#0-1) 

Once the `assemble_artifact` branch wins the race, execution is committed to the `Done` arm. The cancellation token and the `all_peers_deleted_artifact` future are both dropped. The only way `wait_for` can resolve is:

1. The `PeerCounter` for the artifact becomes empty (all advertising peers retract), or
2. The `watch::Sender` is dropped (which only happens in `handle_artifact_processor_joined` → `active_assembles.remove(&id)`, which only runs after the task *joins*, which never happens because the task is blocked). [3](#0-2) 

The only legitimate escape for a peer that stays in the subnet topology is overwriting slot S with a different artifact ID, which triggers `sender.send_if_modified(|h| h.remove(peer_id))` in `handle_slot_update_receive`: [4](#0-3) 

`handle_topology_update` also removes peers from `PeerCounter`s, but only when they leave the subnet topology: [5](#0-4) 

A Byzantine peer below the fault threshold remains in the topology indefinitely, so neither escape applies.

The same unguarded `wait_for` exists in the `AssembleResult::Unwanted` branch: [6](#0-5) 

Both are annotated `// TODO: NET-1774`, confirming the developers are aware this is unresolved.

### Impact Explanation

- **Unvalidated pool memory leak**: Every artifact assembled from a Byzantine peer's advertisement that the peer never retracts stays in the unvalidated pool. The pool grows by up to `slot_limit` artifacts per Byzantine peer.
- **Assemble task leak**: Each blocked task holds a `JoinSet` slot, a `watch::Receiver`, and a `Sender` channel reference. These are never freed.
- **`active_assembles` leak**: The `watch::Sender` entry for the artifact ID is never removed from `active_assembles`, so the map grows without bound relative to Byzantine peers × slot_limit.
- **Consensus processing overhead**: Artifacts stuck in the unvalidated pool are repeatedly re-evaluated by consensus until it independently purges them via finalization, wasting CPU.

The impact is bounded per peer by `slot_limit` but scales linearly with the number of Byzantine peers, making it a viable resource-exhaustion DoS vector. [7](#0-6) 

---

### Likelihood Explanation

The attacker is a subnet peer operating below the consensus fault threshold — a realistic adversary model for Byzantine fault tolerance analysis. The exploit requires only that the peer advertise an artifact and then stop sending slot updates for that slot. No cryptographic break, no majority corruption, and no privileged access is needed. The attack is trivially reproducible in a local test environment.

---

### Recommendation

Wrap the `wait_for` call inside a `select!` that also listens to the cancellation token, and add a timeout or a separate "consensus finalized past this artifact" signal as an additional escape condition. The referenced ticket NET-1774 should be treated as a security-relevant fix, not merely a cleanup item.

```rust
select! {
    _ = cancellation_token.cancelled() => {}
    _ = peer_rx.wait_for(|p| p.is_empty()) => {}
}
```

This ensures that when the node shuts down or the artifact is no longer relevant, the task unblocks and sends `Remove`.

4. Do **not** call `handle_slot_update_receive` again for `NODE_1` slot 1 (Byzantine peer never retracts).

6. Assert that `mgr.artifact_processor_tasks.len() == 1` (task is permanently blocked).
7. Assert that `mgr.active_assembles.len() == 1` (entry never cleaned up).
