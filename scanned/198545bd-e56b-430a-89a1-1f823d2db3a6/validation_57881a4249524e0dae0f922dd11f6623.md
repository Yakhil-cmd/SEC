Let me check the fetch_artifact module and any size limits in the transport layer.

### Title
Unbounded `JoinSet::spawn` via unchecked `ingress_messages` count in `TryFrom<pb::StrippedBlockProposal>` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

---

### Summary

A Byzantine subnet peer can craft a `pb::StrippedBlockProposal` response carrying an arbitrarily large `ingress_messages` array. Because `TryFrom<pb::StrippedBlockProposal>` performs no count bound check, `BlockProposalAssembler::new` allocates a `Vec` of N entries, and `assemble_message` then spawns N concurrent `get_or_fetch` tokio tasks via `JoinSet::spawn`. This exhausts replica memory and the tokio task scheduler, causing an OOM crash and halting consensus finalization on the affected node.

---

### Finding Description

**1. No count bound in `TryFrom<pb::StrippedBlockProposal>`** [1](#0-0) 

The `ingress_messages` field is iterated and collected into a `Vec<SignedIngressId>` with no upper-bound check. A Byzantine peer can supply N=millions of entries up to the transport's `MAX_MESSAGE_SIZE_BYTES` limit.

**2. `BlockProposalAssembler::new` allocates a Vec of N entries** [2](#0-1) 

The developer comment at line 441 reads *"We can have at most 1000 elements in the vector"* — this is an assumption, not an enforced invariant. [3](#0-2) 

**3. N concurrent tokio tasks spawned unconditionally** [4](#0-3) 

Every entry returned by `missing_stripped_messages()` spawns one `get_or_fetch` task. Each task loops indefinitely (with exponential backoff) trying to download the missing message from peers.

**4. The `FetchArtifact` ID check does not validate content** [5](#0-4) 

The only check before accepting the artifact is `message.id() == id`. For `MaybeStrippedConsensusMessage`, the ID is derived solely from `unstripped_consensus_message_id`, not from the ingress list. A crafted artifact with a valid `unstripped_consensus_message_id` and N fake ingress entries passes this check.

**5. The final `try_assemble()` ID mismatch check is too late** [6](#0-5) 

This check would eventually detect the mismatch, but only after all N tasks have already been spawned and are running.

**6. Transport size limit bounds N but does not prevent the attack**

`connection_handle.rs` uses `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`. [7](#0-6) 

The test `test_real_socket_large_msg` sends 100 MB successfully, confirming `MAX_MESSAGE_SIZE_BYTES` is at least 100 MB. Each `StrippedIngressMessage` proto entry is ~80–150 bytes, so a single crafted response can encode ~660,000–1,250,000 fake entries — far exceeding any reasonable task budget.

**7. Bouncer mitigation is insufficient**

The bouncer refresh period is 3 seconds. [8](#0-7) 

Dropping `join_set` aborts tasks when the artifact becomes unwanted, but the window between task spawning and bouncer invalidation is enough to exhaust memory. If the Byzantine peer is the sole advertiser of the artifact, the tasks persist until the real block arrives from another peer.

---

### Impact Explanation

- **Memory exhaustion**: Each spawned tokio task allocates a stack and heap state. N=1M tasks × ~8 KB stack = ~8 GB minimum stack allocation, plus heap for `Vec` entries and RPC state.
- **Task scheduler exhaustion**: The tokio runtime's work-stealing scheduler degrades severely under millions of concurrent tasks.
- **Result**: Replica OOM crash → node drops out of consensus → subnet finalization halted until the node restarts.

---

### Likelihood Explanation

The attacker must be a legitimate subnet peer (below the fault threshold). They need to:
1. Observe a valid `StrippedConsensusMessageId` being advertised (public P2P information).
2. Be selected as the random peer from which the victim downloads the artifact — probability 1/|advertising peers|, and approaches 1 if the Byzantine peer is the sole advertiser.

This is a realistic, single-peer attack requiring no key material, no majority, and no external infrastructure.

---

### Recommendation

Add a count bound check in `TryFrom<pb::StrippedBlockProposal>` before collecting `ingress_messages`:

```rust
// In stripped.rs, before line 102
const MAX_INGRESS_MESSAGES_PER_BLOCK: usize = 1_000; // match consensus limit
if value.ingress_messages.len() > MAX_INGRESS_MESSAGES_PER_BLOCK {
    return Err(ProxyDecodeError::Other(format!(
        "Too many ingress messages: {} > {}",
        value.ingress_messages.len(),
        MAX_INGRESS_MESSAGES_PER_BLOCK,
    )));
}
```

The same bound should be applied to `stripped_idkg_dealings`. The constant should be derived from the subnet's `max_block_payload_size` / `max_ingress_bytes_per_message` registry parameters rather than hardcoded.

### Proof of Concept

```rust
#[test]
fn unbounded_task_spawn_via_fake_ingress_ids() {
    // Construct a StrippedBlockProposal with N=500_000 fake ingress entries
    let n = 500_000usize;
    let fake_ingress_messages: Vec<pb::StrippedIngressMessage> = (0..n)
        .map(|i| pb::StrippedIngressMessage {
            stripped: Some(/* fake IngressMessageId proto */),
            ingress_bytes_hash: vec![i as u8; 32],
        })
        .collect();

    let proto = pb::StrippedBlockProposal {
        pruned_block_proposal: Some(/* valid pruned proto with empty ingress_payload */),
        unstripped_consensus_message_id: Some(/* valid BlockProposal hash */),
        ingress_messages: fake_ingress_messages,
        stripped_idkg_dealings: vec![],
    };

    // TryFrom succeeds — no count check
    let stripped = StrippedBlockProposal::try_from(proto).unwrap();
    assert_eq!(stripped.stripped_ingress_payload.ingress_messages.len(), n);

    // BlockProposalAssembler::new allocates Vec of n entries
    let assembler = BlockProposalAssembler::new(stripped);
    let missing = assembler.missing_stripped_messages();
    assert_eq!(missing.len(), n); // n tasks would be spawned
    // In assemble_message, join_set.spawn() is called n times → OOM
}
```
