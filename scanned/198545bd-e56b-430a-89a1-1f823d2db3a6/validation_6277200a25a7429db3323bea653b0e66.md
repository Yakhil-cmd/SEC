### Title
SNS `distribute_rewards()` May Exhaust Instruction Limit Due to Unbounded Neuron Iteration - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary
The SNS governance `distribute_rewards()` function iterates over all voting neurons across all settled proposals in a single synchronous execution without any instruction-limit guard or batching mechanism. If the neuron/ballot count is large enough, the function will exceed the IC per-message instruction limit, causing the timer callback to trap and roll back all state changes, permanently halting SNS reward distribution.

---

### Finding Description
`distribute_rewards` in `rs/sns/governance/src/governance.rs` performs two unbounded loops in a single execution slice:

**Loop 1** — collect reward shares from every ballot of every settled proposal: [1](#0-0) 

**Loop 2** — update maturity for every neuron that voted: [2](#0-1) 

Neither loop contains an instruction-limit check or an early-exit mechanism. The function is called synchronously from `run_periodic_tasks` after an `await` on `ledger.total_supply()`, so it begins with a fresh instruction budget: [3](#0-2) 

The NNS governance team already identified and fixed this exact pattern. The NNS CHANGELOG explicitly states:

> *"Distribute rewards is moved to timer, and has a mechanism to distribute in batches in multiple messages."*
> *"Unstaking maturity task has a limit of 100 neurons per message, which prevents it from exceeding instruction limit."* [4](#0-3) 

The NNS fix introduces `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` checked after every neuron update, allowing the work to be spread across multiple timer firings: [5](#0-4) 

The SNS governance has no equivalent protection.

---

### Impact Explanation
If the instruction limit is exceeded, the IC traps the entire message and rolls back all state changes. Because `proto.latest_reward_event` is only written at the very end of `distribute_rewards` (after both neuron loops complete), a mid-execution trap leaves `latest_reward_event` unchanged. The next timer firing will attempt the identical distribution and trap again. The result is a **permanent liveness failure**: SNS voting rewards are never distributed to any neuron, and the reward event is never advanced.

---

### Likelihood Explanation
SNS governance allows up to `max_number_of_neurons` neurons (configurable, with a large default). Each neuron that votes on a proposal contributes one entry to the ballot map. With many neurons voting on multiple proposals simultaneously, the combined ballot iteration in Loop 1 and the neuron-maturity update in Loop 2 can easily reach tens of millions of operations. The NNS team's own benchmarks and changelog confirm this is a realistic concern even at moderate neuron counts, which is why they introduced batching for NNS. SNS has not received the same fix.

---

### Recommendation
Apply the same batching pattern already used in NNS governance:
1. Move SNS reward distribution to a dedicated periodic timer task (analogous to `DistributeRewardsTask` in `rs/nns/governance/src/timer_tasks/distribute_rewards.rs`).
2. Store the pending per-neuron reward map in stable state between timer firings (analogous to `RewardsDistributionStateMachine` in `rs/nns/governance/src/reward/distribution.rs`).
3. In each timer firing, process neurons in a loop that checks `is_message_over_threshold` after each neuron update and breaks early if the limit is approached, resuming in the next firing. [6](#0-5) [7](#0-6) 

---

### Proof of Concept
1. Deploy an SNS with a large number of neurons (e.g., approaching `max_number_of_neurons`).
2. Submit and pass several proposals so that many neurons cast ballots.
3. Advance time past the reward round boundary so `should_distribute_rewards()` returns `true`.
4. When the periodic timer fires `run_periodic_tasks`, it calls `ledger.total_supply().await` (resetting the instruction counter), then immediately calls `distribute_rewards(supply)` synchronously.
5. `distribute_rewards` iterates over every ballot of every settled proposal (Loop 1) and then updates every voting neuron's maturity (Loop 2) with no instruction-limit check.
6. With a sufficiently large neuron/ballot count, the IC instruction limit is exceeded; the message traps; `latest_reward_event` is not updated.
7. Every subsequent timer firing repeats steps 4–6, permanently blocking reward distribution.

impl Governance {
    pub(crate) fn schedule_pending_rewards_distribution(
        &self,
        day_after_genesis: u64,
        distribution: RewardsDistribution,
    ) {
        let result =
            with_rewards_distribution_state_machine_mut(|rewards_distribution_state_machine| {
                rewards_distribution_state_machine
                    .add_rewards_distribution(day_after_genesis, distribution)
            });

        if let Err(e) = result {
            println!("{}Error scheduling rewards distribution: {}", LOG_PREFIX, e);
        }

        // TODO(NNS1-3643) Determine if there is a way we can refactor this so that
        // canbench can call timer setting function stubs (or even immediately execute the work)
        #[cfg(not(feature = "canbench-rs"))]
        run_distribute_rewards_periodic_task();
    }

    // Returns if there is work left to do
    pub fn distribute_pending_rewards(&mut self) -> bool {
        let is_over_instructions_limit = || is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT);
        with_rewards_distribution_state_machine_mut(|rewards_distribution_state_machine| {
            rewards_distribution_state_machine.with_next_distribution(|(_, distribution)| {
                distribution
                    .continue_processing(&mut self.neuron_store, is_over_instructions_limit);
            });
            // Work left?
            !rewards_distribution_state_machine.distributions.is_empty()
        })
    }

    fn continue_processing(
        &mut self,
        neuron_store: &mut NeuronStore,
        is_over_instructions_limit: fn() -> bool,
    ) {
        while let Some((id, reward_e8s)) = self.rewards.pop_first() {
            match neuron_store.with_neuron_mut(&id, |neuron| {
                let auto_stake = neuron.auto_stake_maturity.unwrap_or(false);
                if auto_stake {
                    neuron.staked_maturity_e8s_equivalent = Some(
                        neuron
                            .staked_maturity_e8s_equivalent
                            .unwrap_or_default()
                            .saturating_add(reward_e8s),
                    );
                } else {
                    neuron.maturity_e8s_equivalent =
                        neuron.maturity_e8s_equivalent.saturating_add(reward_e8s);
                }
            }) {
                Ok(_) => {}
                Err(e) => {
                    println!(
                        "{}Error rewarding neuron {:?} during reward_distribution.\
                    This should not be possible as neuron existence is checked when \
                    rewards are calculated: {}",
                        LOG_PREFIX, id, e
                    );
                }
            };
            if is_over_instructions_limit() {
                break;
            }
        }
    }

impl PeriodicSyncTask for DistributeRewardsTask {
    fn execute(self) {
        self.governance.with_borrow_mut(|governance| {
            let work_left = governance.distribute_pending_rewards();
            if !work_left {
                cancel_distribute_pending_rewards_timer();
            }
        });
    }

    const NAME: &'static str = "distribute_rewards";
