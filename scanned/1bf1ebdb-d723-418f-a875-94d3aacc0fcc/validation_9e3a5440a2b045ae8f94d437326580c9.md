### Title
Silent No-Op Overwrite in `insert_sns_specific_upgrade_path_entry` Permanently Locks SNS Custom Upgrade Path - (`rs/nns/sns-wasm/src/sns_wasm.rs`)

### Summary

`insert_sns_specific_upgrade_path_entry` has an asymmetric implementation compared to `insert_upgrade_path_entry`: the `Entry::Occupied` branch logs "being overwritten" but **never calls `occupied.insert(to)`**, silently dropping the new value. A second NNS governance proposal targeting the same `(from, governance_canister_id)` key will return success but leave the original entry unchanged.

### Finding Description

In `insert_sns_specific_upgrade_path_entry` (lines 1927–1951):

```rust
Entry::Occupied(occupied) => {
    println!(
        "Special Entry for {} from {:?} to {:?} is being overwritten with new value {:?}",
        sns_governance_canister_id,
        occupied.key(),
        occupied.get(),
        to   // <-- `to` is captured in the log message but never inserted
    );
    // BUG: occupied.insert(to) is missing here
}
Entry::Vacant(vacant) => {
    vacant.insert(to);
}
``` [1](#0-0) 

Compare to the correct implementation in `insert_upgrade_path_entry` (lines 1954–1970), which does call `occupied.insert(to)`: [2](#0-1) 

The caller `insert_upgrade_path_entries` returns `InsertUpgradePathEntriesResponse { error: None }` unconditionally after calling the buggy function, so the caller receives a success response even though the update was silently dropped: [3](#0-2) 

### Impact Explanation

The SNS-specific upgrade path is the **emergency escape hatch** for SNS instances that cannot follow the standard upgrade path. Once a custom path entry `A -> B` is inserted for governance canister `G`, no subsequent NNS governance proposal can change it to `A -> C`. The SNS is permanently locked to the original custom path until the SNS-W canister itself is upgraded with a code fix. If the original custom path was incorrect, broken, or pointed to a bad WASM version, the SNS cannot be rescued via governance.

### Likelihood Explanation

The trigger path is a standard NNS governance proposal flow (`NnsFunction::InsertSnsWasmUpgradePathEntries`), which is the intended and documented mechanism for managing custom upgrade paths. The existing test at line 2835–2842 even explicitly calls `insert_upgrade_path_entries` twice with the same key "to exercise the 'occupied' code path," but it uses the **same** `next_version` both times, so it never detects that the overwrite is silently dropped: [4](#0-3) 

Any real-world scenario where a governance proposal corrects a previously inserted custom path (e.g., wrong WASM hash, path that dead-ends) will silently fail.

### Recommendation

Add `occupied.insert(to)` in the `Entry::Occupied` branch of `insert_sns_specific_upgrade_path_entry`, matching the behavior of `insert_upgrade_path_entry`:

```rust
Entry::Occupied(mut occupied) => {
    println!(...);
    occupied.insert(to); // add this line
}
```

### Proof of Concept

State-machine test (unit level):
1. Insert entry `A -> B` for governance canister `G` via `insert_upgrade_path_entries`.
2. Insert entry `A -> C` (same `from`, different `to`) for the same `G`.
3. Call `get_next_sns_version` with `current_version = A` and `governance_canister_id = G`.
4. **Expected**: returns `C`. **Actual**: returns `B`.

The function returns `InsertUpgradePathEntriesResponse { error: None }` for both calls, giving no indication that the second update was dropped. [5](#0-4)
