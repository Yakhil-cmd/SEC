### Title
Unbounded Loop Over All Neuron Ballots in SNS Governance `distribute_rewards` Exhausts Instruction Limit, Permanently Blocking Reward Distribution - (File: rs/sns/governance/src/governance.rs)

### Summary

SNS Governance's `distribute_rewards` function iterates over every ballot of every settled proposal in a single synchronous message, with no instruction-limit check or batching mechanism. As the number of neurons and proposals grows, this loop can exhaust the per-message instruction limit, causing the reward distribution to silently fail every reward round and permanently halting SNS voting rewards.

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `distribute_rewards` method is called from `run_periodic_tasks` once per reward round. It contains two nested unbounded loops:

1. An outer loop over all `considered_proposals` (proposals ready to settle).
2. An inner loop over every `ballot` in each proposal's ballot map — one entry per neuron that voted. [1](#0-0) 

After accumulating reward shares, a second unbounded loop iterates over every neuron that voted to credit maturity: [2](#0-1) 

Neither loop checks the instruction counter or breaks early. The entire computation — iterating all ballots across all settled proposals, then crediting all rewarded neurons — must complete within a single message execution. There is no batching, no resumption state machine, and no instruction-limit guard analogous to what NNS Governance added for its own reward distribution. [3](#0-2) 

By contrast, NNS Governance explicitly refactored its reward distribution to use a `RewardsDistributionStateMachine` with a `DISTRIBUTION_MESSAGE_LIMIT` guard and a periodic timer that resumes across multiple messages: [4](#0-3) 

SNS Governance has no equivalent protection.

### Impact Explanation

If the instruction limit is exceeded during `distribute_rewards`, the entire message traps and is rolled back. Because `run_periodic_tasks` is called on a heartbeat/timer, the next invocation will attempt the same full computation again — and fail again — for every subsequent reward round. The result is:

- **Voting rewards are permanently frozen** for all SNS neurons, regardless of participation.
- The SNS governance canister remains otherwise functional (proposals, voting, upgrades), but the reward mechanism is silently broken.
- There is no on-chain error surfaced to users; the failure is invisible except through the absence of maturity accrual.

This is a **cycles/resource accounting bug** (instruction exhaustion) that causes a **governance authorization/liveness failure**: a critical periodic governance function becomes permanently non-executable.

### Likelihood Explanation

The IC per-message instruction limit for update calls is 40 billion instructions on application subnets. The inner ballot loop iterates once per neuron per settled proposal. An SNS with:

- 10,000 neurons (achievable for a popular SNS),
- 10 proposals settling in one reward round,

produces 100,000 ballot iterations plus 10,000 maturity-credit iterations per call. At stable-memory access costs (hundreds of instructions per `get_neuron_result_mut` call), this approaches the limit. The NNS Governance changelog explicitly documents this exact failure mode occurring in production and requiring a hotfix: [5](#0-4) 

SNS instances with large neuron counts (e.g., OpenChat, Kinic, or any SNS that reaches tens of thousands of neurons) are concretely at risk. An attacker can accelerate this by creating many neurons (each with minimum stake) to inflate ballot counts, since any principal can call `refresh_buyer_tokens` during a swap to become a neuron holder.

### Recommendation

1. **Adopt the NNS pattern**: Separate reward calculation (synchronous, produces a `RewardsDistribution` map) from reward disbursement (batched across multiple timer-driven messages with an instruction-limit guard), exactly as done in `rs/nns/governance/src/reward/distribution.rs`.

2. **Add an instruction-limit check** inside the neuron maturity-credit loop, breaking out and persisting remaining work to stable storage for the next timer invocation.

3. **Add a benchmark** (analogous to NNS canbench tests) measuring instruction consumption of `distribute_rewards` at realistic neuron counts (10K, 50K, 100K) to establish a regression baseline.

4. **Cap the number of proposals settled per reward round** if a full batching refactor is not immediately feasible, as a short-term mitigation.

### Proof of Concept

**Entry path**: Any unprivileged user participates in an SNS swap → becomes a neuron holder → their neuron ID appears in proposal ballots → `distribute_rewards` is called by the SNS governance heartbeat/timer → the ballot iteration loop over all neurons × all settled proposals exhausts the instruction limit → the message traps → rewards are never distributed.

**Concrete trigger**:
- Deploy an SNS with `voting_rewards_parameters` enabled.
- Create or acquire ~10,000–50,000 neurons (via swap participation or `ManageNeuron::ClaimOrRefresh`).
- Let several proposals reach `ReadyToSettle` state simultaneously (e.g., by setting a short `reward_round_duration_seconds`).
- Observe that `run_periodic_tasks` → `distribute_rewards` traps on every invocation.
- Verify: `latest_reward_event.end_timestamp_seconds` stops advancing; neuron `maturity_e8s_equivalent` stops increasing for all neurons.
