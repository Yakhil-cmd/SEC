### Title
ckETH Minter `next_transaction_nonce` Setter Overwrites Nonce Without Checking In-Flight Transactions — (File: `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

### Summary
The ckETH minter's `UpgradeArg` exposes a `next_transaction_nonce` field that, when supplied in an NNS governance upgrade proposal, unconditionally overwrites the minter's internal `next_nonce` counter with no guard against existing in-flight (created or sent) Ethereum transactions. This is the direct IC analog of the CurveLP setter bug: a privileged configuration setter replaces a key operational address/parameter while prior state still holds active resources, making those resources inaccessible or corrupted.

---

### Finding Description

`update_next_transaction_nonce` in `EthTransactions` is a one-liner with no invariant checks: [1](#0-0) 

It is called unconditionally from `State::upgrade()` whenever `UpgradeArg::next_transaction_nonce` is `Some`: [2](#0-1) 

The `UpgradeArg` type exposes this field publicly via the canister's Candid interface: [3](#0-2) 

The minter's transaction state machine tracks three live maps keyed by `(TransactionNonce, LedgerBurnIndex)`: [4](#0-3) 

When `next_nonce` is reset to value `N` while `sent_tx` already contains an entry at nonce `N` (for a prior withdrawal), the next call to `create_transactions_batch` creates a new `Eip1559TransactionRequest` with nonce `N` for a different burn index and inserts it into `created_tx`. When that transaction is subsequently signed, `record_signed_transaction` finds nonce `N` already present in `sent_tx` and silently appends the new signed transaction to the old entry's `Vec`: [5](#0-4) 

This corrupts the `sent_tx` map: nonce `N` now carries signed transactions for two different burn indices under a single secondary key. The new burn index is invisible to `sent_tx`, so when the minter later calls `record_finalized_transaction` for the new withdrawal it panics with "BUG: sent transaction not found", halting all withdrawal processing. Meanwhile, on Ethereum the new transaction (with the same nonce) may replace the old one if it carries a higher gas price, causing the original user's ETH transfer to be silently dropped.

The same `upgrade()` function also exposes three contract-address setters (`ethereum_contract_address`, `erc20_helper_contract_address`, `deposit_with_subaccount_helper_contract_address`) that overwrite the scraped contract address without resetting `last_scraped_block_number`: [6](#0-5) 

`set_contract_address` only rejects the zero address: [7](#0-6) 

After the address is replaced, `scrape_until_block` resumes from `last_scraped_block_number + 1` but queries only the **new** contract. Any `ReceivedEth` or `ReceivedErc20` log events emitted by the **old** contract in blocks after `last_scraped_block_number` are permanently skipped; the corresponding user deposits are never minted. [8](#0-7) 

---

### Impact Explanation

**Nonce setter:** Resetting `next_transaction_nonce` to a value already occupied by a `sent_tx` entry corrupts the minter's transaction state machine. The immediate consequence is a canister trap on the next finalization attempt, halting all ckETH and ckERC20 withdrawals. In the window before the trap, the minter may broadcast a second Ethereum transaction at the same nonce, which can replace the original on-chain transaction; the original withdrawal's ETH is then sent to a different recipient while the original user receives only a ckETH reimbursement — a net loss of ETH for that user.

**Contract-address setters:** Replacing any of the three helper contract addresses while unscraped Ethereum blocks remain causes all deposit events from the old contract in those blocks to be permanently ignored. Users who sent ETH or ERC-20 tokens to the old contract in that window receive no ckETH/ckERC20 and have no recovery path within the minter.

---

### Likelihood Explanation

NNS governance proposals to upgrade the ckETH minter are routine and have been executed many times on mainnet. The `next_transaction_nonce` field exists precisely to allow operators to "fix" a stuck nonce — a scenario that has occurred in practice (e.g., when an Ethereum provider went offline). An operator who sets this field to the current on-chain transaction count without accounting for the minter's in-flight `sent_tx` entries triggers the bug. Similarly, contract-address rotation (already done once for the deposit-with-subaccount contract) is a plausible future governance action. Neither setter requires any attacker capability beyond submitting or voting on a governance proposal.

---

### Recommendation

1. **`update_next_transaction_nonce`**: Before accepting a new nonce, assert that `created_tx` and `sent_tx` contain no entries with nonces `>= new_nonce`. If they do, reject the upgrade argument with a descriptive error rather than silently corrupting state.

2. **Contract-address setters**: Before replacing a contract address, verify that `last_scraped_block_number` equals the current observed chain tip (i.e., all blocks have been scraped for the old address). Alternatively, retain the old address in a "draining" state and continue scraping it until fully caught up before switching exclusively to the new address.

---

### Proof of Concept

**Nonce corruption path:**

1. Minter is live with `next_nonce = 10`; `sent_tx` contains entries for nonces 8 and 9 (two pending withdrawals).
2. NNS governance submits an upgrade proposal with `UpgradeArg { next_transaction_nonce: Some(8), ... }`.
3. `post_upgrade` calls `State::upgrade()` → `update_next_transaction_nonce(8)` → `next_nonce = 8`.
4. A new withdrawal request arrives; `create_transactions_batch` creates `Eip1559TransactionRequest { nonce: 8, ... }` for a new burn index and inserts it into `created_tx`.
5. The minter signs the transaction and calls `record_signed_transaction`. Because nonce 8 is already in `sent_tx` (for the old burn index), the branch at line 573 fires and appends the new signed tx to the old entry's `Vec` — the new burn index is never registered as a secondary key.
6. On Ethereum, the new transaction (nonce 8, potentially higher gas) replaces the old one; the old withdrawal's ETH goes to the new recipient.
7. When the minter calls `record_finalized_transaction` for the new burn index, `self.sent_tx.remove_entry_alt(&new_burn_index)` returns `None` → `expect("BUG: sent transaction not found")` → canister trap → all withdrawal processing halts.

**Deposit-loss path:**

1. Minter is scraping `EthDepositWithoutSubaccount` at contract `0xOLD`, `last_scraped_block_number = B`.
2. At block `B+1`, a user sends 5 ETH to `0xOLD`; the log event is not yet scraped.
3. NNS governance submits an upgrade with `UpgradeArg { ethereum_contract_address: Some("0xNEW"), ... }`.
4. `set_contract_address(EthDepositWithoutSubaccount, 0xNEW)` overwrites the address; `last_scraped_block_number` remains `B`.
5. Next `scrape_logs` call scrapes blocks `B+1..current` querying only `0xNEW`; the deposit at `0xOLD` in block `B+1` is never seen.
6. The user's 5 ETH is permanently unrecoverable through the minter.

async fn scrape_until_block<S>(last_block_number: BlockNumber, max_block_spread: u16)
where
    S: LogScraping,
{
    let scrape = match read_state(S::next_scrape) {
        Some(s) => s,
        None => {
            log!(
                DEBUG,
                "[scrape_contract_logs]: skipping scraping {} logs: not active",
                S::ID
            );
            return;
        }
    };
    let block_range = BlockRangeInclusive::new(
        scrape
            .last_scraped_block_number
            .checked_increment()
            .unwrap_or(BlockNumber::MAX),
        last_block_number,
    );
    log!(
        DEBUG,
        "[scrape_contract_logs]: Scraping {} logs in block range {block_range}",
        S::ID
    );
    let rpc_client = read_state(rpc_client);
    for block_range in block_range.into_chunks(max_block_spread) {
        match scrape_block_range::<S>(
            &rpc_client,
            scrape.contract_address,
            scrape.topics.clone(),
            block_range.clone(),
        )
        .await
        {
            Ok(()) => {}
            Err(e) => {
                log!(
                    INFO,
                    "[scrape_contract_logs]: Failed to scrape {} logs in range {block_range}: {e:?}",
                    S::ID
                );
                return;
            }
        }
    }
