### Title
ckERC20 Minter Over-Mints Tokens for Fee-on-Transfer ERC-20 Deposits Due to Missing Balance Verification - (File: `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

The ckETH/ckERC20 minter canister mints ckERC20 tokens based solely on the `amount` field recorded in the Ethereum event log (`ReceivedEthOrErc20` / `ReceivedErc20`), without verifying the actual ERC-20 balance received by the minter's Ethereum address. For fee-on-transfer ERC-20 tokens, the actual received amount is less than the logged `amount`, causing the IC minter to over-mint ckERC20 tokens and break the 1:1 backing guarantee.

---

### Finding Description

The ckERC20 deposit flow works as follows:

1. A user calls `depositErc20(erc20Address, amount, principal, subaccount)` on the Ethereum helper smart contract.
2. The helper contract calls `safeTransferFrom(msg.sender, minterAddress, amount)` on the ERC-20 token.
3. The helper contract emits `ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount)` — using the **input parameter `amount`**, not the actual balance change at `minterAddress`.
4. The IC minter scrapes these logs via HTTPS outcalls and reads `event.value()` (the `amount` from the log).
5. The minter mints exactly `event.value()` ckERC20 tokens to the user.

**Root cause in the Solidity helper contracts:**

In `DepositHelperWithSubaccount.sol`, `depositErc20` emits the input `amount` rather than the actual received amount: [1](#0-0) 

Similarly in `ERC20DepositHelper.sol`: [2](#0-1) 

**Root cause in the IC minter canister:**

The `mint()` function in `rs/ethereum/cketh/minter/src/deposit.rs` mints `event.value()` directly from the scraped log without any balance check before or after: [3](#0-2) 

The `ReceivedErc20Event.value` field is populated directly from the Ethereum log data, which records the input `amount` parameter, not the actual balance delta: [4](#0-3) 

For a fee-on-transfer ERC-20 token, `safeTransferFrom(msg.sender, minterAddress, amount)` transfers `amount - fee` to `minterAddress`, but the event records `amount`. The IC minter then mints `amount` ckERC20 tokens, creating `fee` unbacked ckERC20 tokens per deposit.

---

### Impact Explanation

**Ledger conservation bug / chain-fusion mint/burn/replay bug.** The ckERC20 token's 1:1 backing invariant is violated: the minter mints more ckERC20 than the ERC-20 tokens it actually holds. Over time, repeated deposits of a fee-on-transfer token drain the minter's ERC-20 reserves. When users attempt to withdraw ckERC20 back to ERC-20, the minter will be unable to fulfill all withdrawal requests because it holds fewer ERC-20 tokens than the outstanding ckERC20 supply. This is a direct ledger conservation failure affecting all ckERC20 holders of the affected token.

---

### Likelihood Explanation

Currently supported ckERC20 tokens (ckUSDC, ckUSDT, etc.) do not actively charge transfer fees. However:
- USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) has a fee mechanism in its contract that is currently set to 0 but can be enabled by the USDT issuer at any time.
- Any future NNS proposal to add a fee-on-transfer ERC-20 token as a supported ckERC20 token would immediately make this exploitable by any unprivileged Ethereum user.
- The vulnerability requires no privileged IC access — any Ethereum address can call `depositErc20` on the helper contract.

Likelihood is **medium** given the USDT fee mechanism and the possibility of future token additions.

---

### Recommendation

1. **In the Solidity helper contracts:** Record the actual balance change rather than the input `amount`. Before and after `safeTransferFrom`, read `erc20Token.balanceOf(minterAddress)` and emit the difference as the deposit value:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

2. **In the IC minter canister:** Add a validation step or documentation that explicitly rejects fee-on-transfer tokens when adding new supported ckERC20 tokens via NNS governance proposals.

3. **Governance-level control:** Enforce a check during the `add_ckerc20_token` flow that verifies the candidate ERC-20 token does not implement fee-on-transfer behavior.

---

### Proof of Concept

1. Assume USDT enables its transfer fee (e.g., 1 basis point = 0.01%).
2. Attacker calls `depositErc20(USDT_ADDRESS, 1_000_000_000, principal, subaccount)` on the Ethereum helper contract.
3. `safeTransferFrom` transfers `999_900_000` USDT to the minter (after 0.01% fee).
4. The helper emits `ReceivedEthOrErc20(..., 1_000_000_000, ...)`.
5. The IC minter scrapes the log, reads `value = 1_000_000_000`, and mints `1_000_000_000` ckUSDT.
6. The attacker received `100_000` unbacked ckUSDT per deposit.
7. Repeated at scale, the minter's USDT reserves are drained relative to outstanding ckUSDT supply.
