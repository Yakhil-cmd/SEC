### Title
ckERC20 Minter Determines Withdrawal Success Solely from EVM Receipt Status, Not ERC-20 `transfer()` Return Value — (`File: rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

---

### Summary

The ckETH minter finalizes ckERC20 withdrawals by checking only the EVM-level transaction receipt `status` field (`0x0` = reverted, `0x1` = not reverted). It never inspects the actual return value of the ERC-20 `transfer(address,uint256)` call embedded in the transaction data. For ERC-20 tokens that signal failure by returning `false` without reverting — a behavior explicitly permitted by the ERC-20 standard — the minter incorrectly treats the withdrawal as successful, permanently burning the user's ckERC20 tokens without delivering the underlying ERC-20 tokens.

---

### Finding Description

During a ckERC20 withdrawal, the minter constructs an EIP-1559 transaction whose `data` field encodes `transfer(to, amount)` with selector `0xa9059cbb`: [1](#0-0) 

After the transaction is mined, `finalize_transactions_batch` retrieves the receipt and emits a `FinalizedTransaction` event: [2](#0-1) 

`record_finalized_transaction` then decides whether to reimburse the user based exclusively on `receipt.status`: [3](#0-2) 

`update_balance_upon_withdrawal` similarly gates the internal ERC-20 balance deduction on `receipt.status == TransactionStatus::Success`: [4](#0-3) 

The EVM receipt `status` field only reflects whether the outermost call frame reverted. For ERC-20 tokens that return `false` on failure without reverting (a valid ERC-20 pattern), the receipt status is `0x1` (Success), yet no tokens are transferred. The minter has no mechanism to distinguish this case: it neither decodes the return value of the `transfer()` call nor checks for the presence of a `Transfer(from, to, value)` event in the receipt logs.

The `TransactionCallData::decode` function only parses the *input* calldata, not the output: [5](#0-4) 

The `TransactionReceipt` struct contains no field for the call's return data: [6](#0-5) 

---

### Impact Explanation

When a supported ERC-20 token's `transfer()` returns `false` without reverting:

1. The minter sees `receipt.status = Success`.
2. No reimbursement is scheduled — the user's ckERC20 burn is treated as final.
3. `erc20_balances.erc20_sub()` is called, reducing the minter's internal accounting for that token even though no tokens left the minter's Ethereum address.
4. The user permanently loses their ckERC20 tokens without receiving the underlying ERC-20 tokens.
5. The minter's internal ERC-20 balance becomes understated, corrupting future withdrawal accounting for all users of that token.

This is a **chain-fusion ledger conservation bug**: ckERC20 tokens are destroyed on the IC ledger without a corresponding ERC-20 transfer on Ethereum.

---

### Likelihood Explanation

The currently deployed ckERC20 tokens (ckUSDC, ckUSDT, etc.) revert on failure, so the bug is not immediately exploitable against live tokens. However:

- The ERC-20 standard explicitly permits returning `false` instead of reverting; many tokens in the wild use this pattern.
- Any future ckERC20 token added via governance that follows this pattern would immediately expose all its withdrawal users to fund loss.
- No attacker action is required beyond initiating a normal withdrawal against a token whose `transfer()` returns `false` under some condition (e.g., transfer to a blacklisted address, paused contract, or insufficient balance edge case).
- The entry path is fully unprivileged: any user calling `withdraw_erc20` on the minter canister.

---

### Recommendation

After retrieving the transaction receipt, verify that the ERC-20 transfer actually occurred by checking for the presence of a `Transfer(indexed from, indexed to, value)` log entry in `receipt.logs` matching the minter's address, the recipient, and the withdrawal amount. If the `Transfer` event is absent despite a `Success` receipt status, treat the withdrawal as failed and trigger reimbursement. This mirrors the `SafeERC20._callOptionalReturn` pattern used in the deposit helper contracts: [7](#0-6) 

---

### Proof of Concept

1. A ckERC20 token is added whose `transfer()` implementation returns `false` (without reverting) when the recipient is on a blocklist.
2. A user calls `withdraw_erc20` specifying a blocklisted Ethereum address as the recipient.
3. The minter burns the user's ckERC20 tokens on the IC ledger and sends the Ethereum transaction.
4. The ERC-20 contract executes `transfer(blocklisted_address, amount)`, returns `false`, emits no `Transfer` event, but does not revert.
5. The Ethereum transaction receipt has `status = 0x1` (Success).
6. `record_finalized_transaction` sees `receipt.status != Failure`, so no reimbursement is recorded.
7. `update_balance_upon_withdrawal` sees `receipt.status == Success`, calls `erc20_balances.erc20_sub(token, amount)`.
8. The user's ckERC20 tokens are permanently burned; the ERC-20 tokens remain in the minter's Ethereum address; the minter's internal balance is understated by `amount`.

    pub fn decode<T: AsRef<[u8]>>(data: T) -> Result<Self, String> {
        let data = data.as_ref();
        match data.get(0..4) {
            Some(selector) if selector == ERC_20_TRANSFER_FUNCTION_SELECTOR => {
                if data.len() != 68 {
                    return Err("Invalid data length".to_string());
                }
                let address = <[u8; 32]>::try_from(&data[4..36]).unwrap();
                let to = Address::try_from(&address).unwrap();

                let value = <[u8; 32]>::try_from(&data[36..]).unwrap();
                let value = Erc20Value::from_be_bytes(value);

                Ok(TransactionCallData::Erc20Transfer { to, value })
            }
            Some(selector) => Err(format!(
                "Unknown function selector 0x{:?}",
                hex::encode(selector)
            )),
            None => Err("missing function selector".to_string()),
        }
    }

    #[n(0)]
    pub block_hash: Hash,

    /// The number of the block containing the transaction.
    #[n(1)]
    pub block_number: BlockNumber,

    /// The total base charge plus tip paid for each unit of gas
    #[n(2)]

    /// The amount of gas used by this specific transaction alone
    #[n(3)]
    pub gas_used: GasAmount,

    /// Status of the transaction.
    #[n(4)]
    pub status: TransactionStatus,

    /// The hash of the transaction
    #[n(5)]
    pub transaction_hash: Hash,
}

impl TransactionReceipt {
    pub fn effective_transaction_fee(&self) -> Wei {
        self.effective_gas_price
