### Title
Unpermissioned `update_balance` Allows Any Caller to Exhaust the Per-Account Concurrent Guard and DoS Victim's ckBTC Minting - (File: rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs)

---

### Summary

The ckBTC minter's `update_balance` endpoint accepts an optional `owner` field that, when set, allows any unprivileged caller to trigger the minting flow for an arbitrary victim account. Because the per-account concurrency guard (`balance_update_guard`) is keyed on the target account (not the caller), an attacker can hold the guard slot for a victim's account for the entire duration of an async Bitcoin-checker round-trip, causing the victim's own legitimate `update_balance` call to return `AlreadyProcessing` and fail to mint their ckBTC.

---

### Finding Description

`update_balance` in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` accepts `UpdateBalanceArgs { owner: Option<Principal>, subaccount: Option<Subaccount> }`. When `owner` is `Some(victim_principal)`, the function resolves the target account as:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

There is no check that `caller == args.owner`. The guard is keyed on `caller_account` (the victim's account), not on the actual ingress caller. The guard is held for the entire duration of the async call, which includes at least one inter-canister call to the Bitcoin canister (`get_utxos`) and up to `MAX_CHECK_TRANSACTION_RETRY` (10) calls to the Bitcoin checker canister. This can take many seconds of wall-clock time.

During this window, any subsequent call to `update_balance` for the same victim account — including the victim's own legitimate call — will hit the occupied guard and return `UpdateBalanceError::AlreadyProcessing`.

The `MAX_CONCURRENT` limit is 100 slots shared across all accounts. An attacker with 100 distinct principals (or subaccounts) can fill all 100 slots with victim accounts, causing every new `update_balance` call to return `TooManyConcurrentRequests` for all users.

---

### Impact Explanation

- **Per-victim DoS**: An attacker can continuously call `update_balance` for a specific victim's account, keeping the guard occupied across async await points. The victim's own call returns `AlreadyProcessing` and cannot mint ckBTC until the attacker stops.
- **Global DoS**: With 100 distinct attacker-controlled principals (trivially generated on IC), the attacker fills all 100 `MAX_CONCURRENT` guard slots with arbitrary victim accounts. All legitimate `update_balance` calls then return `TooManyConcurrentRequests`, halting ckBTC minting for all users.
- The attacker pays only IC ingress fees (cycles for message submission). No BTC or ckBTC is required. The attack is cheap and repeatable.
- Funds are not stolen, but users cannot convert their confirmed BTC deposits into ckBTC for the duration of the attack.

---

### Likelihood Explanation

The endpoint is publicly reachable by any ingress sender. The `owner` field is explicitly documented as optional and intended to allow third-party callers to trigger minting for others (the README and integration tests confirm this is a supported use case). The guard is held across multiple async inter-canister calls, giving the attacker a large window. The attack requires no special privileges, no tokens, and no on-chain state beyond submitting ingress messages.

---

### Recommendation

Restrict the guard key to the **caller** rather than the target account, or require that `caller == args.owner` when `owner` is explicitly set. For example:

```rust
// Only allow caller to trigger update_balance for themselves,
// or require explicit opt-in from the owner.
if let Some(owner) = args.owner {
    if owner != caller {
        return Err(UpdateBalanceError::GenericError {
            error_code: ...,
            error_message: "caller is not authorized to update balance for another principal".into(),
        });
    }
}
```

Alternatively, key the guard on the **caller** principal rather than the target account, so that an attacker can only block their own slot, not the victim's.

---

### Proof of Concept

1. Victim `V` sends BTC to their ckBTC deposit address and waits for confirmations.
2. Attacker `A` (any unprivileged IC principal) calls:
   ```
   update_balance(record { owner = opt principal "V"; subaccount = null })
   ```
   This acquires `balance_update_guard` for account `{owner: V, subaccount: null}` and holds it across the async `get_utxos` + `check_transaction` calls.
3. Victim `V` calls `update_balance(record {})` (defaulting to their own account). The guard is already held → returns `AlreadyProcessing`.
4. Attacker repeats step 2 in a loop. Victim cannot mint.
5. For global DoS: attacker uses 100 different subaccounts targeting 100 different victim principals, filling all `MAX_CONCURRENT = 100` slots. All users receive `TooManyConcurrentRequests`.

pub async fn update_balance<R: CanisterRuntime>(
    args: UpdateBalanceArgs,
    runtime: &R,
) -> Result<Vec<UtxoStatus>, UpdateBalanceError> {
    let caller = runtime.caller();
    if args.owner.unwrap_or(caller) == runtime.id() {
        ic_cdk::trap("cannot update minter's balance");
    }

    // Record start time of method execution for metrics
    let start_time = runtime.time();

    // When the minter is in the mode using a whitelist we only want a certain
    // set of principal to be able to mint. But we also want those principals
    // to mint at any desired address. Therefore, the check below is on "caller".
    state::read_state(|s| s.mode.is_deposit_available_for(&caller))
        .map_err(UpdateBalanceError::TemporarilyUnavailable)?;

    init_ecdsa_public_key().await;

    let caller_account = Account {
        owner: args.owner.unwrap_or(caller),
        subaccount: args.subaccount,
    };
    let _guard = balance_update_guard(caller_account)?;

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
