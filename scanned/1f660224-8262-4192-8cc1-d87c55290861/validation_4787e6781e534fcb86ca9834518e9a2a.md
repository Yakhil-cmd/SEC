### Title
Unchecked Zero `principal` Parameter in Chain-Fusion Deposit Helper Contracts Causes Permanent Loss of Deposited Funds - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

The `CkDeposit` contract's `depositEth` and `depositErc20` functions, as well as the legacy `CkEthDeposit.deposit` function, accept a `bytes32 principal` parameter without validating it against `bytes32(0)`. When a user passes a zero principal, the ETH or ERC-20 tokens are irreversibly transferred to the minter address on Ethereum, but the IC minter rejects the deposit as invalid and mints nothing. The deposited funds are permanently unrecoverable.

---

### Finding Description

In `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, the `CkDeposit` contract exposes two public deposit entry points:

```solidity
function depositEth(bytes32 principal, bytes32 subaccount) public payable {
    emit ReceivedEthOrErc20(ZERO_ADDRESS, msg.sender, msg.value, principal, subaccount);
    minterAddress.transfer(msg.value);
}

function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
}
```

Neither function checks `require(principal != bytes32(0), ...)`. The same omission exists in the legacy `CkEthDeposit.deposit(bytes32 _principal)` in `rs/ethereum/cketh/minter/EthDepositHelper.sol`. [1](#0-0) [2](#0-1) [3](#0-2) 

On the IC side, the minter's `parse_principal_from_slice` function in `rs/ethereum/cketh/minter/src/eth_logs/mod.rs` explicitly rejects a zero `bytes32` principal:

```rust
let num_bytes = slice[0] as usize;
if num_bytes == 0 {
    return Err("management canister principal is not allowed".to_string());
}
``` [4](#0-3) 

This causes the deposit to be recorded as `InvalidDeposit` in the minter state, and no ckETH or ckERC20 is ever minted. [5](#0-4) 

The full call chain is:

1. User calls `depositEth(bytes32(0), subaccount)` on the deployed `CkDeposit` contract — ETH is transferred to `minterAddress` on Ethereum (irreversible).
2. The `ReceivedEthOrErc20` event is emitted with `principal = bytes32(0)`.
3. The IC minter scrapes the log and calls `parse_principal` → `parse_principal_from_slice` → returns `Err("management canister principal is not allowed")`.
4. The minter records an `InvalidDeposit` event and skips minting.
5. The deposited ETH is permanently locked in the minter's Ethereum address with no corresponding ckETH minted. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any user who calls `depositEth` or `depositErc20` with `principal = bytes32(0)` permanently loses their deposited ETH or ERC-20 tokens. The Ethereum-side transfer is irreversible once the transaction is finalized. The IC minter will never mint ckETH or ckERC20 for such a deposit because `parse_principal_from_slice` rejects the zero encoding. The funds are locked in the minter's Ethereum address with no recovery mechanism. This is a direct, permanent loss of user funds — a chain-fusion mint/burn accounting bug.

---

### Likelihood Explanation

The `principal` parameter must be manually encoded as a `bytes32` by the caller. A user who is unfamiliar with the encoding format, testing the contract, or using a script with an uninitialized variable can trivially pass `bytes32(0)`. The documentation itself warns: *"It's critical that the encoded IC principal is correct otherwise the funds will be lost"* — acknowledging the risk but relying entirely on user diligence rather than a contract-level guard. The likelihood is low-to-medium for individual users but non-negligible given the permissionless, public-facing nature of the contract. [8](#0-7) 

---

### Recommendation

Add a zero-check guard at the top of each deposit function in both `DepositHelperWithSubaccount.sol` and `EthDepositHelper.sol`:

```solidity
// In CkDeposit (DepositHelperWithSubaccount.sol)
function depositEth(bytes32 principal, bytes32 subaccount) public payable {
    require(principal != bytes32(0), "CkDeposit: zero principal");
    emit ReceivedEthOrErc20(ZERO_ADDRESS, msg.sender, msg.value, principal, subaccount);
    minterAddress.transfer(msg.value);
}

function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    require(principal != bytes32(0), "CkDeposit: zero principal");
    ...
}
```

Apply the same fix to `CkEthDeposit.deposit` in `EthDepositHelper.sol`. This mirrors the existing `erc20Address != ZERO_ADDRESS` guard already present in `depositErc20` and aligns with the IC minter's own rejection logic.

---

### Proof of Concept

1. Deploy or interact with the live `CkDeposit` contract at the address returned by `get_minter_info`.
2. Call `depositEth{value: 1 ether}(bytes32(0), bytes32(0))`.
3. The transaction succeeds on Ethereum — 1 ETH is transferred to `minterAddress`.
4. The IC minter scrapes the `ReceivedEthOrErc20` log, calls `parse_principal_from_slice([0x00, ...])`, receives `Err("management canister principal is not allowed")`.
5. The minter records `InvalidDeposit` for the event source (tx hash + log index).
6. No ckETH is minted. The 1 ETH is permanently locked. The caller receives nothing. [9](#0-8) [10](#0-9)

```rust
fn parse_principal_from_slice(slice: &[u8]) -> Result<Principal, String> {
    const ANONYMOUS_PRINCIPAL_BYTES: [u8; 1] = [4];

    if slice.is_empty() {
        return Err("slice too short".to_string());
    }
    if slice.len() > 32 {
        return Err(format!("Expected at most 32 bytes, got {}", slice.len()));
    }
    let num_bytes = slice[0] as usize;
    if num_bytes == 0 {
        return Err("management canister principal is not allowed".to_string());
    }
    if num_bytes > 29 {
        return Err(format!(
            "invalid number of bytes: expected a number in the range [1,29], got {num_bytes}",
        ));
    }
    if slice.len() < 1 + num_bytes {
        return Err("slice too short".to_string());
    }
    let (principal_bytes, trailing_zeroes) = slice[1..].split_at(num_bytes);
    if !trailing_zeroes
        .iter()
        .all(|trailing_zero| *trailing_zero == 0)
    {
        return Err("trailing non-zero bytes".to_string());
    }
    if principal_bytes == ANONYMOUS_PRINCIPAL_BYTES {
        return Err("anonymous principal is not allowed".to_string());
    }
    Principal::try_from_slice(principal_bytes).map_err(|err| err.to_string())
}

        let from_address = parse_address(&entry.topics[2], event_source)?;

        let [value_bytes, subaccount_bytes] =
            parse_hex_into_32_byte_words(entry.data, event_source)?;

            transaction_hash,
