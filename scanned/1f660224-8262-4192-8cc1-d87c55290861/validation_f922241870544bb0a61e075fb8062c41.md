### Title
Unbounded Ingress Window Causes OOM in ICP Rosetta `construction_payloads` - (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from `ConstructionPayloadsRequestMetadata` and feeds them directly into an unbounded `while` loop with no cap on the window size. A single unauthenticated HTTP POST with `ingress_start=0` and `ingress_end=u64::MAX` causes the loop to push ~153 billion `u64` entries into a `Vec`, exhausting heap memory and crashing the Rosetta process.

---

### Finding Description

In `construction_payloads()`, the interval step is computed as:

```
interval = MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s
         = 300s - 60s - 120s = 120 seconds
``` [1](#0-0) [2](#0-1) 

The attacker-supplied `ingress_start` and `ingress_end` are deserialized from the JSON body and converted to `ic_types::time::Time` with no validation: [3](#0-2) 

They are then fed directly into the loop: [4](#0-3) 

With `ingress_start=0` and `ingress_end=u64::MAX` (18446744073709551615 ns):

```
iterations ≈ u64::MAX / 120_000_000_000 ≈ 153,722,867,280
memory     ≈ 153 billion × 8 bytes      ≈ 1.23 TB
```

The process is killed by the OOM killer long before completion, crashing the Rosetta node.

The `ConstructionPayloadsRequestMetadata` struct accepts both fields as plain `Option<u64>` with no range constraints: [5](#0-4) 

**The ICRC1 Rosetta implementation has explicit guards for exactly this case** — it rejects requests where `ingress_start >= ingress_end` or where the window is unreasonably large — but the ICP Rosetta implementation has no equivalent protection: [6](#0-5) 

The ICP version has no analogous check anywhere before the loop executes.

---

### Impact Explanation

A single small (~100-byte) HTTP POST to `/construction/payloads` crashes the Rosetta node process via OOM. No authentication is required. The Rosetta API becomes unavailable until the process is restarted. This is a non-volumetric, single-request DoS against the Rosetta node.

---

### Likelihood Explanation

The endpoint is publicly reachable. The exploit requires no credentials, no prior state, and no special knowledge — only knowledge of the Rosetta API schema (which is public). The payload is trivially constructable. The ICRC1 counterpart already demonstrates awareness of this class of bug, making the omission in the ICP path a clear oversight.

---

### Recommendation

Add a window-size guard before the loop in `construction_payloads()`, mirroring the ICRC1 implementation:

```rust
let max_window = interval * MAX_REASONABLE_EXPIRY_COUNT; // e.g., 100 intervals
if ingress_end > ingress_start + max_window {
    return Err(ApiError::internal_error(
        "ingress_end exceeds maximum allowed window"
    ));
}
if ingress_start >= ingress_end {
    return Err(ApiError::internal_error(
        "ingress_start must be before ingress_end"
    ));
}
```

Alternatively, cap `ingress_expiries` to a fixed maximum length (e.g., 100 entries) and return an error if the window would exceed it.

---

### Proof of Concept

```
POST /construction/payloads HTTP/1.1
Content-Type: application/json

{
  "network_identifier": { ... },
  "operations": [ ... ],
  "public_keys": [ ... ],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

This triggers the loop at [7](#0-6)  with ~153 billion iterations, exhausting heap memory and crashing the process.

**Note on the question's numbers:** The question states a ~23.5-hour interval and ~1.7×10¹³ entries. The actual interval is **120 seconds** (not 23.5 hours), yielding ~153 billion entries (not 1.7×10¹³). The core vulnerability is real regardless; the question's arithmetic was based on incorrect interval assumptions.
