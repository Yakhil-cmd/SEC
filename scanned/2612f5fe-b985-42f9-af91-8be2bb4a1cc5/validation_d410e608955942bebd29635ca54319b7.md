### Title
ckETH/ckERC20 Minter Has No Chain Reorganization Detection — (File: rs/ethereum/cketh/minter/src/deposit.rs)

---

### Summary

The ckETH/ckERC20 minter scrapes Ethereum logs to detect deposits and mint tokens, but contains no mechanism to detect or handle Ethereum chain reorganizations. Once a block range is processed and `last_scraped_block_number` is advanced, those blocks are never re-queried. The `ensure_not_removed()` guard only catches reorgs reflected in the *current* `eth_getLogs` response — not reorgs that occur after the minter has already processed and minted tokens. No block hash comparison is performed between scraping cycles.

---

### Finding Description

**Scraping loop never revisits processed blocks.**

`scrape_block_range()` in `deposit.rs` processes a block range, registers deposit events, and unconditionally advances `last_scraped_block_number` to `to_block`: [1](#0-0) 

Once `last_scraped_block_number` is advanced, `scrape_until_block()` always starts from `last_scraped_block_number + 1`: [2](#0-1) 

The state only stores a block *number*, never a block *hash*: [3](#0-2) 

There is no comparison of the stored block number's hash against what the RPC returns in the next cycle. If a reorganization replaces a previously-processed block with a different block at the same height, the minter has no way to detect it.

**`ensure_not_removed()` is insufficient.**

The parser does check the `removed` field of each `LogEntry`: [4](#0-3) 

However, this only protects against reorgs that are reflected *within the same `eth_getLogs` response* (i.e., the RPC provider returns `removed: true` for a log in the same batch). If the minter has already processed a block, minted tokens, and advanced `last_scraped_block_number`, a subsequent reorg of that block is never observed — the minter never re-queries it.

**Deposit flow leading to irreversible mint.**

Once a deposit event passes `ensure_not_removed()` and `register_deposit_events()` is called, the event is placed in `events_to_mint` and minting is scheduled immediately: [5](#0-4) 

After `record_successful_mint()` moves the event to `minted_events`, there is no path to reverse the mint: [6](#0-5) 

**Configurable block tag widens the attack surface.**

The production minter defaults to `Finalized`, but `ethereum_block_height` is a governance-upgradeable parameter: [7](#0-6) 

If the minter is upgraded to use `Safe` or `Latest`, reorganizations become realistic without any catastrophic consensus failure. The test `should_skip_scrapping_when_last_seen_block_newer_than_current_height` demonstrates that the minter already handles `Safe`→`Finalized` transitions by simply not re-scraping the gap: [8](#0-7) 

---

### Impact Explanation

If an Ethereum chain reorganization removes a deposit transaction after the minter has already minted ckETH or ckERC20 tokens for it, the minted tokens become permanently unbacked. The attacker retains the minted tokens while the original on-chain deposit no longer exists on the canonical chain. There is no code path to reverse a completed mint (`minted_events` is append-only and `record_successful_mint` panics on double-mint attempts). This directly violates the 1:1 backing invariant of the chain-fusion token.

---

### Likelihood Explanation

**With `Finalized` (production default):** Post-merge Ethereum finalizes blocks via Casper FFG; reorganizing a finalized block requires >1/3 of beacon-chain validators to be slashed simultaneously. This is a catastrophic external event and makes the vulnerability practically unexploitable under normal conditions.

**With `Safe` or `Latest`:** These tags are reachable via a governance `UpgradeArg` without any privileged key or majority corruption — a normal NNS proposal suffices. `Safe` blocks can be reorganized by natural network conditions; `Latest` blocks are reorganized routinely. Under either configuration, an attacker who can time a deposit to a block that is subsequently reorganized away can obtain unbacked ckETH/ckERC20. The attacker's only required action is submitting an Ethereum transaction to the helper contract — a fully unprivileged ingress path.

---

### Recommendation

**Short term:** After each scraping cycle, store the block hash of `last_scraped_block_number` alongside the block number. At the start of the next cycle, fetch the block at that stored height and compare its hash. If the hashes differ, a reorganization has occurred; halt minting and alert operators.

**Long term:** Implement automated reorganization handling: re-scrape the affected block range, identify events that are no longer present on the canonical chain, and design a strategy for handling already-minted tokens (e.g., negative-balance tracking or governance-controlled burn). Until automated handling is in place, restrict `ethereum_block_height` to `Finalized` via a governance invariant check in the upgrade path.

---

### Proof of Concept

1. A governance proposal upgrades the minter with `ethereum_block_height = Safe` (or `Latest`).
2. An attacker calls the ETH helper contract's deposit function on Ethereum, emitting a `ReceivedEth` event in block N.
3. The minter's timer fires, `update_last_observed_block_number()` returns block N as the safe head, and `scrape_block_range()` processes block N — finding the deposit event, passing `ensure_not_removed()` (since `removed = false` at query time), and calling `register_deposit_events()`.
4. `mint()` is scheduled; the ICRC-1 ledger mints ckETH to the attacker's principal. `last_scraped_block_number` is advanced to N.
5. An Ethereum chain reorganization occurs; block N is replaced by a competing block that does not contain the attacker's deposit transaction.
6. The minter's next timer cycle starts from block N+1 — block N is never re-queried. The `removed` flag check is never triggered for the already-processed event.
7. The attacker holds minted ckETH with no corresponding on-chain ETH backing it. The minter's `minted_events` map permanently records the mint with no reversal mechanism. [9](#0-8) [10](#0-9)

pub async fn update_last_observed_block_number() -> Option<BlockNumber> {
    let block_height = read_state(State::ethereum_block_height);
    match read_state(rpc_client)
        .get_block_by_number(block_height.clone())
        .with_cycles(MIN_ATTACHED_CYCLES)
        .try_send()
        .await
        .reduce_with_strategy(NoReduction)
    {
        Ok(latest_block) => {
            let block_number = Some(BlockNumber::from(latest_block.number));
            mutate_state(|s| s.last_observed_block_number = block_number);
            block_number
        }
        Err(e) => {
            log!(
                INFO,
                "Failed to get the latest {block_height:?} block number: {e:?}"
            );
            read_state(|s| s.last_observed_block_number)
        }
    }
}

#[test]
fn should_skip_scrapping_when_last_seen_block_newer_than_current_height() {
    let safe_block_number = LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 100;
    let finalized_block_number = safe_block_number - 32;
    let cketh = CkEthSetup::default().check_audit_logs_and_upgrade(UpgradeArg {
        ethereum_block_height: Some(CandidBlockTag::Safe),
        ..Default::default()
    });
    let received_eth_event_topic = cketh.received_eth_event_topic();
    cketh.env.tick();

    let cketh = cketh
        .deposit(DepositParams::default())
        .with_mock_eth_get_block_by_number(move |mock| {
            mock.with_request_params(json!(["safe", false]))
                .respond_for_all_with(block_response(safe_block_number))
        })
        .with_mock_eth_get_logs(move |mock| {
            mock.with_request_params(json!([{
                "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "toBlock": BlockNumber::from(safe_block_number),
                "address": [ETH_HELPER_CONTRACT_ADDRESS],
                "topics": [received_eth_event_topic]
            }]))
        })
        .expect_mint();

    let cketh = cketh
        .check_audit_logs_and_upgrade(UpgradeArg {
            ethereum_block_height: Some(CandidBlockTag::Finalized),
            ..Default::default()
        })
        .assert_has_unique_events_in_order(&[EventPayload::SyncedToBlock {
            block_number: safe_block_number.into(),
        }]);
    cketh.env.tick();

    MockJsonRpcProviders::when(JsonRpcMethod::EthGetBlockByNumber)
        .with_request_params(json!(["finalized", false]))
        .respond_for_all_with(block_response(finalized_block_number))
        .build()
        .expect_rpc_calls(&cketh);

    cketh
        .assert_has_no_rpc_call(&JsonRpcMethod::EthGetLogs)
        .check_audit_logs_and_upgrade(Default::default())
        .assert_has_no_event_satisfying(|event| {
            matches!(event, EventPayload::SyncedToBlock { block_number }
                if block_number != &Nat::from(safe_block_number) && block_number != &Nat::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL))
        });
}
