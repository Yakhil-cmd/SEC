### Title
Unhandled State-Update Error After Successful Maturity Transfer Enables Double-Disbursement - (File: rs/sns/governance/src/governance.rs)

### Summary
In `rs/sns/governance/src/governance.rs`, the `finalize_disburse_maturity` function transfers minted ICP to a neuron owner and then attempts to remove the corresponding `DisburseMaturityInProgress` entry from the neuron's state. If the post-transfer state update (`get_neuron_result_mut`) fails, the code logs an error and `continue`s — leaving the disbursement entry intact. On the next periodic invocation of `finalize_disburse_maturity`, the same entry is found again and the transfer is re-attempted, causing double-minting.

### Finding Description
After a successful `transfer_funds` call, the code attempts to retrieve the neuron mutably to remove the completed disbursement entry:

```rust
Ok(block_index) => {
    let neuron = match self.get_neuron_result_mut(&neuron_id) {
        Ok(neuron) => neuron,
        Err(e) => {
            log!(ERROR, "Failed updating DisburseMaturityInProgress-entry ...");
            continue;   // ← transfer succeeded, but entry NOT removed
        }
    };
    neuron.disburse_maturity_in_progress.remove(0);
}
``` [1](#0-0) 

If `get_neuron_result_mut` returns `Err`, the `disburse_maturity_in_progress.remove(0)` line is never reached. The disbursement entry persists in the neuron's state. The next time the periodic task calls `finalize_disburse_maturity`, it iterates over all pending entries, finds this one again, and issues a second minting transfer for the same amount.

This contrasts with the newer NNS governance implementation (`try_finalize_maturity_disbursement`), which pops the entry *before* the ledger call and pushes it back only on failure — a correct check-then-act pattern: [2](#0-1) 

The SNS governance path has not been updated to match this safer pattern.

### Impact Explanation
Each successful ledger transfer mints new ICP from the governance minting account to the neuron owner's account. If the disbursement entry is not removed, the same amount is minted again on the next periodic run. Repeated invocations produce unbounded ICP inflation from the SNS governance minting account, draining the token supply and harming all other token holders. This is a ledger conservation bug: tokens are created without a corresponding reduction in maturity.

### Likelihood Explanation
`get_neuron_result_mut` failing after a successful transfer requires the neuron to become inaccessible between the `await` and the state update. Because the neuron lock is held during this window, concurrent deletion is blocked under normal operation. However, the neuron lock is stored in `in_flight_commands` in heap memory; an unexpected canister trap or upgrade between the two steps could clear the lock and leave the entry un-removed. The periodic nature of `finalize_disburse_maturity` means any such residual entry will be retried automatically. Likelihood is low but non-zero, and the code is structurally incorrect regardless.

### Recommendation
Mirror the NNS governance pattern: pop the `DisburseMaturityInProgress` entry *before* the ledger call, and push it back to the front of the queue if the transfer fails. If the post-transfer state restoration itself fails, retain the neuron lock to prevent further processing. This eliminates the window where a successful transfer leaves a stale disbursement entry.

```rust
// Pop entry before transfer
let entry = neuron.disburse_maturity_in_progress.remove(0);

let transfer_result = self.ledger.transfer_funds(...).await;

match transfer_result {
    Ok(_) => { /* done */ }
    Err(e) => {
        // Restore entry so it can be retried
        neuron.disburse_maturity_in_progress.insert(0, entry);
        log!(ERROR, "Transfer failed: {}", e);
    }
}
```

### Proof of Concept
1. Neuron owner calls `disburse_maturity` on an SNS neuron with non-zero `maturity_e8s_equivalent`. A `DisburseMaturityInProgress` entry is appended and maturity is deducted.
2. The periodic `finalize_disburse_maturity` task runs. It acquires the neuron lock, calls `transfer_funds` (minting ICP to the owner), and receives `Ok(block_index)`.
3. Immediately after the `await` returns, `get_neuron_result_mut` returns `Err` (e.g., due to a canister upgrade that cleared `in_flight_commands` or an internal store inconsistency). The code logs the error and calls `continue`.
4. The `DisburseMaturityInProgress` entry remains at index 0 of `disburse_maturity_in_progress`.
5. On the next periodic invocation, the same entry is found, the neuron lock is re-acquired (since it was released by `continue`), and a second minting transfer is issued for the same amount.
6. Steps 3–5 repeat, minting ICP on every periodic run until the entry is manually cleared via a canister upgrade. [3](#0-2) [4](#0-3)
