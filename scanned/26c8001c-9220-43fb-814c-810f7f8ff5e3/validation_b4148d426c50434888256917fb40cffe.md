### Title
`sender_info` Canister Signature Not Bound to Request Sender or Canister — Replay Across Any Request - (File: `rs/types/types/src/messages/http.rs`)

---

### Summary

The `sender_info` field allows a trusted canister (e.g., Internet Identity) to attest user attributes by signing an `info` blob. The signed bytes (`SenderInfoContent`) contain only the raw `info` bytes and a domain separator — they do not include the request sender, target canister, or message ID. A valid `sender_info` signature observed from any request can be replayed verbatim in a completely different request by a different sender targeting a different canister, falsely claiming the attested attributes.

---

### Finding Description

The `SenderInfoContent` type defines the signed payload for `sender_info` canister signatures: [1](#0-0) 

The `write_signed_bytes_without_domain_separator` implementation appends only the raw `info` bytes. Combined with the `SignatureDomain` implementation that prepends `"ic-sender-info"`, the full signed message is:

```
\x0Eic-sender-info || info_bytes
```

No `sender` principal, no `canister_id`, no `message_id`, no `ingress_expiry` is included.

The validation in `verify_sender_info_canister_sig` verifies the canister signature only over this context-free payload: [2](#0-1) 

The outer envelope signature (`sender_sig`) does cover the `sender_info` content (including the `sig` field) via the `MessageId` hash: [3](#0-2) 

This means the envelope signature prevents tampering with `sender_info` within a single request, but it does **not** prevent an attacker from extracting a valid `(info_bytes, sig)` pair from one request and embedding it in a new request with a different sender and different target canister.

---

### Impact Explanation

Any canister that calls `msg_caller_info_data()` and makes authorization or trust decisions based on the attested `info` blob is vulnerable. An attacker who observes a valid `sender_info` signature (e.g., from a public ingress message on the IC, since ingress messages are observable) can:

1. Extract `{info, signer, sig}` from the victim's request.
2. Construct a new request with the attacker's own `sender` principal and any target `canister_id`.
3. Embed the extracted `sender_info` unchanged.
4. Sign the new envelope with the attacker's own key pair (valid envelope signature).
5. Submit the request.

The replica accepts it because:
- The envelope signature is valid (attacker's key).
- The `sender_info` canister signature is valid (it only checks `sig` over `info_bytes`, not the request context).

The target canister sees `msg_caller()` = attacker's principal but `msg_caller_info_data()` = victim's attested attributes. If the canister grants privileges based on those attributes (e.g., "this user is KYC-verified", "this user has role X"), the attacker gains those privileges without authorization.

---

### Likelihood Explanation

**Medium.** Ingress messages on the IC are observable by boundary nodes and replicas. An attacker can passively collect valid `sender_info` signatures from any request that uses the feature. The attacker only needs to know the `info_bytes` content (which is in plaintext in the request) and have a valid signature for them. No privileged access, no threshold corruption, and no social engineering is required — only passive observation of network traffic and construction of a new valid envelope.

---

### Recommendation

Bind the `sender_info` canister signature to the request context by including at minimum the `sender` principal in the signed bytes. Ideally, include the full `MessageId` (which already covers `sender`, `canister_id`, `method_name`, `arg`, `ingress_expiry`, `nonce`, and the `sender_info` content itself):

```rust
impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        // Include sender principal and message_id to prevent cross-request replay
        bytes.extend_from_slice(self.sender.as_slice());
        bytes.extend_from_slice(self.message_id.as_bytes());
        bytes.extend_from_slice(self.info);
    }
}
```

Alternatively, require the signing canister to include the `sender` principal inside the `info` blob itself and validate that it matches `request.sender()` during `verify_sender_info_canister_sig`.

### Proof of Concept

**Setup:**
- Signing canister `C` (e.g., Internet Identity) signs `info_bytes = b"role=admin"` → produces `sig_info`.
- Victim user `U` sends a valid request with `sender_info = {info: info_bytes, signer: C, sig: sig_info}`.

**Attack:**
```
1. Attacker observes victim's request (public ingress), extracts {info_bytes, sig_info}.
2. Attacker constructs:
   HttpCanisterUpdate {
     canister_id: target_canister,   // any canister
     method_name: "privileged_op",
     arg: ...,
     sender: attacker_principal,     // attacker's own key
     ingress_expiry: fresh_expiry,
     nonce: fresh_nonce,
     sender_info: Some(RawSignedSenderInfo {
       info: info_bytes,             // same as victim's
       signer: C,                    // same signing canister
       sig: sig_info,               // REPLAYED from victim's request
     }),
   }
3. Attacker signs the envelope with their own key pair → valid sender_sig.
4. Replica validates:
   - validate_user_id: attacker_principal matches attacker's pubkey ✓
   - validate_signature: attacker's sig over message_id ✓
   - verify_sender_info_canister_sig: sig_info over info_bytes ✓ (no context check)
5. target_canister receives msg_caller() = attacker_principal,
   msg_caller_info_data() = b"role=admin"  ← victim's attested attributes
```
