### Title
Single Malicious Node Can Skew Query Stats Median to Manipulate Canister Resource Accounting - (File: `rs/query_stats/src/state_machine.rs`)

---

### Summary

The `get_median` function in the QueryStats aggregation state machine computes the median of per-node query statistics using only the submitted values. When the number of submitted values is **odd**, a single malicious node (below the Byzantine fault-tolerance threshold) can set its reported value to any point between the two adjacent honest nodes' values, thereby fully controlling the median. This directly corrupts the `total_query_stats` written into every canister's replicated state, which is the authoritative source for query-call resource accounting on the IC.

---

### Finding Description

The QueryStats feature collects per-canister query execution statistics from each node and aggregates them once per epoch by taking the **median** across all submitting nodes. The aggregation is implemented in `rs/query_stats/src/state_machine.rs`.

**`get_median` (lines 84–105):**

```rust
fn get_median<T: ...>(stats: &[&QueryStats], f: F) -> T {
    let mut values: Vec<T> = stats.iter().map(f).collect();
    values.sort_unstable();
    let mid = values.len() / 2;

    if values.len().is_multiple_of(2) {
        // average of two middle values
        (left + right) / 2_u8.into()
    } else {
        values.get(mid).cloned().unwrap_or(T::default())  // exact middle
    }
}
``` [1](#0-0) 

When `values.len()` is **odd**, the result is the single element at index `values.len() / 2`. A node whose submitted value lands exactly at that index controls the output.

**Aggregation threshold (`try_aggregate_one_epoch`, lines 307–311):**

```rust
let need_stats_from = num_nodes.saturating_sub(get_faults_tolerated(num_nodes));
if num_nodes_with_stats < need_stats_from {
    return false;
}
``` [2](#0-1) 

For a 13-node subnet, `get_faults_tolerated(13) = ⌊(13−1)/3⌋ = 4`, so `need_stats_from = 9`. Aggregation fires as soon as 9 nodes have submitted — an **odd** count. The median is then the value at sorted index 4.

**Zero-padding (lines 340–342):**

```rust
let num_empty_stats = num_nodes_with_stats.saturating_sub(stats.len());
stats.append(&mut vec![&empty_stats; num_empty_stats]);
``` [3](#0-2) 

The vector passed to `aggregate_query_stats` always has exactly `num_nodes_with_stats` entries. If a canister was not reported by some submitting nodes, zeros are appended. This means the total count is always `num_nodes_with_stats` — preserving the odd-count property.

**Result written to replicated state (lines 139–147):**

```rust
canister_query_stats.num_calls += aggregated_stats.num_calls as u128 * num_nodes;
canister_query_stats.num_instructions += aggregated_stats.num_instructions as u128 * num_nodes;
``` [4](#0-3) 

The manipulated median is multiplied by `num_nodes` (13) before being accumulated into `total_query_stats`, amplifying any skew by a factor of 13 per epoch.

**Design comment that is incorrect (lines 62–68):**

> "If we omit 1/3 of the records, it is still guaranteed that the median is calculated from values submitted by honest nodes." [5](#0-4) 

This claim is false. Excluding non-submitting nodes only guarantees the median comes from a *submitted* node — it does not prevent a malicious *submitting* node from being the one at the median position.

---

### Impact Explanation

`total_query_stats` is the authoritative, certified, replicated-state record of query-call resource usage for every canister. It is:

1. **Exposed to canister controllers** via `canister_status` (management canister), where it is read as `query_stats.num_calls_total`, `num_instructions_total`, etc. [6](#0-5) 

2. **Stored in certified replicated state** (`SystemState::total_query_stats`), making it part of the state hash and state sync. [7](#0-6) 

3. **Designed as the basis for future cycles charging** — the aggregation comment explicitly states it must "tolerate malicious nodes over- or under-reporting charges." [8](#0-7) 

A malicious node can persistently skew the median each epoch, causing `total_query_stats` to accumulate incorrect values. Because the median is multiplied by `num_nodes` (13) before accumulation, even a small per-epoch skew compounds over time. Any future cycles-charging mechanism built on these stats would be directly exploitable.

---

### Likelihood Explanation

The attack requires a single node on the target subnet to submit a manipulated `QueryStatsPayload` as block proposer. Block proposal rotates among all nodes, so every node gets the opportunity to submit its own stats. A node operator who controls one node (or whose node software is compromised) can execute this attack without any coordination with other nodes and without exceeding the Byzantine fault-tolerance threshold. The attack is persistent and repeats every epoch automatically.

---

### Recommendation

**Short term:** Add monitoring/alerting on per-node query stat submissions to detect outlier values that consistently land near the median position.

**Long term:** The design should acknowledge that a single submitting malicious node can control the median when the submission count is odd. Consider:
- Requiring an **even** minimum submission count (e.g., `need_stats_from` rounded up to the nearest even number) so the median is always the average of two values, forcing the attacker to control two adjacent positions simultaneously.
- Trimming a fixed fraction of extreme values before computing the median (trimmed mean / Winsorized median), which requires controlling more than one node to shift the result.
- Documenting the actual security guarantee: the median is within the range `[v_{⌊n/2⌋}, v_{⌈n/2⌉}]` of honest values, not necessarily equal to an honest value.

---

### Proof of Concept

**Setup:** 13-node subnet, 9 nodes submit (minimum threshold). Honest nodes H₁–H₈ each report `num_calls = 100` for canister C. Malicious node M reports `num_calls = 150`.

**Sorted values:** `[100, 100, 100, 100, 150, 100, 100, 100, 100]` → sorted: `[100, 100, 100, 100, 100, 100, 100, 100, 150]`

Median at index 4 = **100** (M's value is too high to reach the median here).

**Adjusted scenario:** Honest nodes H₁–H₄ report 80, H₅–H₈ report 120. M reports 100.

Sorted: `[80, 80, 80, 80, 100, 120, 120, 120, 120]` → median at index 4 = **100** (M's value).

Without M (8 honest values, even count): median = `(80 + 120) / 2 = 100`. With M reporting 95: sorted `[80, 80, 80, 80, 95, 120, 120, 120, 120]`, median = **95**. M has shifted the median from 100 to 95.

**Amplification:** `apply_query_stats_to_canister` writes `95 × 13 = 1235` instead of `100 × 13 = 1300` to `total_query_stats.num_calls` for this epoch. Over 1000 epochs, the cumulative error is `65,000` calls — a 5% persistent undercount (or overcount if M inflates).

The entry path is: malicious node acts as block proposer → includes `QueryStatsPayload` with manipulated `num_calls`/`num_instructions` → `deliver_query_stats` → `try_aggregate_one_epoch` → `get_median` returns M's value → `apply_query_stats_to_canister` writes corrupted `total_query_stats` to certified replicated state.
