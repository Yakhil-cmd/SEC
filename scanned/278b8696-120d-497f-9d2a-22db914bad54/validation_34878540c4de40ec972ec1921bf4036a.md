### Title
Mint-Before-Burn Ordering in CMC Allows Cycles to Be Minted Without ICP Being Burned if `burn_and_log` Fails - (File: rs/nns/cmc/src/main.rs)

---

### Summary

The Cycles Minting Canister (CMC) in `rs/nns/cmc/src/main.rs` follows the same mint-before-burn anti-pattern identified in the external report. In `process_mint_cycles`, `process_top_up`, and `process_create_canister`, the CMC first delivers the output (cycles deposited to the cycles ledger, or a canister created) and only then attempts to burn the ICP from its subaccount via `burn_and_log`. The burn is explicitly designed to **never propagate errors** and to **never block the success path**. If the burn fails for any reason, cycles are permanently minted without the corresponding ICP being destroyed, violating ICP ledger conservation.

---

### Finding Description

In `process_mint_cycles`:

```rust
match do_mint_cycles(to_account, cycles, deposit_memo).await {
    Ok(deposit_result) => {
        burn_and_log(sub, amount).await;   // burn happens AFTER mint
        Ok(NotifyMintCyclesSuccess { ... })
    }
    ...
}
```

The same pattern appears in `process_top_up` and `process_create_canister`. [1](#0-0) 

The `burn_and_log` function is explicitly documented to swallow all errors:

```rust
/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
``` [2](#0-1) 

Additionally, `burn_and_log` contains an early-return guard that silently skips the burn if `amount < DEFAULT_TRANSFER_FEE`: [3](#0-2) 

The `do_mint_cycles` function calls the cycles ledger's `deposit` endpoint with payment, crediting the user's cycles ledger account before the ICP burn is attempted: [4](#0-3) 

Once `do_mint_cycles` succeeds, the block is marked `NotifiedMint(Ok(...))` in `blocks_notified`, permanently preventing any retry of the notification: [5](#0-4) 

If `burn_and_log` then fails (e.g., the ICP ledger call to `send_pb` returns an error), the result is:
- Cycles are credited to the user's cycles ledger account (irreversible).
- The ICP remains in the CMC's per-user subaccount (not burned, not refunded).
- The block notification is permanently consumed — no retry is possible.
- The ICP total supply is not reduced as it should be.

The same structural flaw exists in `process_top_up` (cycles deposited to a canister before ICP burned) and `process_create_canister` (canister created before ICP burned): [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Ledger conservation violation / ICP supply inflation.** When `burn_and_log` fails after a successful `do_mint_cycles`, the user receives cycles without the backing ICP being destroyed. The ICP remains stranded in the CMC's subaccount with no withdrawal mechanism. The ICP total supply is not reduced, meaning cycles exist without a corresponding ICP burn — a direct violation of the ICP ↔ cycles conservation invariant. The same applies to `notify_top_up` and `notify_create_canister`.

---

### Likelihood Explanation

The `burn_and_log` failure requires the ICP ledger's `send_pb` call to fail. This is not directly attacker-controlled under normal conditions; it would require the ICP ledger to be temporarily unavailable (e.g., under heavy load, during an upgrade, or due to a transient system error). The likelihood of a targeted exploit is low, but the probability of accidental occurrence during ICP ledger disruptions is non-negligible given the volume of CMC operations. The design choice to silence burn errors was intentional (to avoid retryable notifications), but it creates a permanent conservation gap whenever the burn fails.

---

### Recommendation

Reverse the order of operations to follow the Checks-Effects-Interactions pattern: **burn the ICP first, then mint/deposit cycles**. If the burn fails, the operation should be retried rather than silently succeeding. Specifically:

1. In `process_mint_cycles`: call `burn_and_log` (or a failable burn) **before** `do_mint_cycles`.
2. In `process_top_up`: call the burn **before** `deposit_cycles`.
3. In `process_create_canister`: call the burn **before** `do_create_canister`.

If the concern is that a failable burn would allow infinite retries, the solution is to use a two-phase commit: record the burn block index in state before minting, and only mint after confirming the burn succeeded. This is the pattern already used correctly in the ckBTC minter's `reimburse_withdrawals`, which uses a `scopeguard` to quarantine on panic: [8](#0-7) 

---

### Proof of Concept

1. User transfers `N` ICP to the CMC's subaccount keyed by their principal with memo `MEMO_MINT_CYCLES`.
2. User calls `notify_mint_cycles` with the block index.
3. CMC calls `do_mint_cycles` → cycles ledger `deposit` succeeds → user's cycles ledger balance increases by the converted amount.
4. CMC calls `burn_and_log` → ICP ledger `send_pb` fails (e.g., ledger temporarily unavailable).
5. `burn_and_log` logs the error and returns without propagating it.
6. `notify_mint_cycles` returns `Ok(NotifyMintCyclesSuccess { ... })` to the user.
7. The block is marked `NotifiedMint(Ok(...))` — permanently consumed.
8. Result: user holds cycles; `N` ICP remains in CMC subaccount, not burned; ICP total supply is inflated by `N`.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}

async fn do_mint_cycles(
    account: Account,
    cycles: Cycles,
    deposit_memo: Option<Vec<u8>>,
) -> Result<CyclesLedgerDepositResult, String> {
    let Some(cycles_ledger_canister_id) = with_state(|state| state.cycles_ledger_canister_id)
    else {
        return Err("No cycles ledger canister id configured.".to_string());
    };
    // Always use base cycles limit for minting cycles, since the Subnet Rental Canister
    // doesn't call endpoints using this function.
    ensure_balance(cycles, CyclesMintingLimiterSelector::BaseLimit)?;

    let arg = CyclesLedgerDepositArgs {
        to: account,
        memo: deposit_memo,
    };

    let result: CallResult<(CyclesLedgerDepositResult,)> = ic_cdk::api::call::call_with_payment128(
        cycles_ledger_canister_id.get().0,
        "deposit",
        (arg,),
        u128::from(cycles),
    )
    .await;

    result.map(|r| r.0).map_err(|(code, msg)| {
        format!(
            "Cycles ledger rejected deposit call with code {}: {:?}",
            code as i32, msg
        )
    })
}
