### Title
Missing Response-Type/Request-Type Cross-Check in `validate_self_validating_payload_impl` Allows Byzantine Block Proposer to Deliver Mismatched Bitcoin Adapter Response — (`rs/bitcoin/consensus/src/payload_builder.rs`)

---

### Summary

`validate_self_validating_payload_impl` performs no semantic validation of a `SelfValidatingPayload`: it only checks whether the payload is empty and returns its byte size. A single Byzantine block proposer can therefore include a `BitcoinAdapterResponse` whose response variant (`SendTransactionResponse`) does not match the request type stored in state for the given `callback_id` (a `GetSuccessors` context). Honest nodes will notarize and finalize the block. During batch delivery, `push_response_bitcoin` dispatches on the response variant, not on the stored request type, and the `SendTransactionResponse` arm unconditionally pushes `EmptyBlob` to the `consensus_queue` for the supplied `callback_id` — without ever consulting `bitcoin_send_transaction_internal_contexts`. The bitcoin canister's `GetSuccessors` callback therefore receives `EmptyBlob` instead of a `GetSuccessorsResponse`.

---

### Finding Description

**Gap 1 — Validation is a near-no-op.**

`validate_self_validating_payload_impl` checks only for an empty payload and returns the byte count. It never:
- verifies that a `callback_id` exists in state,
- verifies that the response variant matches the request type stored for that `callback_id`, or
- verifies that the `callback_id` has not already been answered. [1](#0-0) 

The `BatchPayloadBuilder::validate_payload` path calls this same function after decoding, adding only a size check. [2](#0-1) 

**Gap 2 — `push_response` dispatches on the response variant, not on the stored request type.**

In `rs/replicated_state/src/bitcoin.rs`, the `GetSuccessorsResponse` arm looks up `bitcoin_get_successors_contexts` and returns `StateError::BitcoinNonMatchingResponse` if the `callback_id` is absent. However, the `SendTransactionResponse` arm performs **no lookup at all**: it unconditionally constructs `EmptyBlob` and pushes a `ConsensusResponse` for whatever `callback_id` was supplied. [3](#0-2) 

Compare with the guarded `GetSuccessorsResponse` arm: [4](#0-3) 

**Attack path:**

1. Byzantine node proposes a block whose `SelfValidatingPayload` contains `BitcoinAdapterResponse { response: SendTransactionResponse {}, callback_id: X }` where `X` is a live `BitcoinGetSuccessorsContext` in state.
2. Honest nodes call `validate_payload` → `validate_self_validating_payload_impl`. The payload is non-empty and within size limits → **validation passes**.
3. Block is notarized (2/3 threshold) and finalized.
4. Batch delivery calls `push_response_bitcoin` with the mismatched response.
5. `SendTransactionResponse` arm fires: `EmptyBlob` is pushed to `consensus_queue` for `callback_id X`.
6. The `BitcoinGetSuccessorsContext` at `X` is **not removed** from `bitcoin_get_successors_contexts` (the arm never touches it).
7. Execution environment delivers `EmptyBlob` to the bitcoin canister's `GetSuccessors` callback. [5](#0-4) 

---

### Impact Explanation

The bitcoin canister's `GetSuccessors` callback receives `EmptyBlob` where it expects a `BitcoinGetSuccessorsResponse`. Depending on the canister's decode logic this causes either:

- **Trap / panic** — execution rolls back; the `ConsensusResponse` is not consumed; the `GetSuccessors` context remains in state; the canister is effectively stalled for that request.
- **Silent misinterpretation** — if the canister treats a decode failure as "no new blocks," it silently skips blocks, corrupting its UTXO set. A corrupted UTXO set is the precondition for false ckBTC minting (UTXOs credited without corresponding on-chain deposits).

The `GetSuccessors` context remaining in state after the mismatched response is delivered means the context can be re-exploited in subsequent blocks.

---

### Likelihood Explanation

The attacker must control one node on the bitcoin subnet. Subnet nodes are selected by NNS governance, so this is not a zero-cost position. However, the IC's Byzantine fault-tolerance model explicitly assumes up to `f < n/3` malicious nodes, and a single malicious block proposer is squarely within that model. The exploit requires no threshold key, no governance majority, and no external dependency — it is fully local to one subnet round.

---

### Recommendation

1. **In `validate_self_validating_payload_impl`**: load the certified state and, for each `BitcoinAdapterResponse` in the payload, verify (a) the `callback_id` exists in the appropriate context map and (b) the response variant matches the stored request type (`GetSuccessors` context ↔ `GetSuccessorsResponse`/`GetSuccessorsReject`; `SendTransaction` context ↔ `SendTransactionResponse`/`SendTransactionReject`).

2. **In `push_response` (`SendTransactionResponse` arm)**: add a lookup against `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if the `callback_id` is absent, mirroring the guard already present in the `GetSuccessorsResponse` arm. [3](#0-2)

    fn validate_self_validating_payload_impl(
        &self,
        payload: &SelfValidatingPayload,
        validation_context: &ValidationContext,
    ) -> Result<NumBytes, SelfValidatingPayloadValidationError> {
        let since = Instant::now();

        // An empty block is always valid.
        if *payload == SelfValidatingPayload::default() {
            return Ok(0.into());
        }

        self.metrics
            .observe_validate_duration(VALIDATION_STATUS_VALID, since);
        let size = NumBytes::new(payload.count_bytes() as u64);

        Ok(size)
    }
