I need to trace the exact cycles accounting through the abort → uninstall → retry path to determine if the invariant actually breaks.

**Tracing the cycles flow:**

**Step 1 – Initial prepay** (`install_code_dts`): When `prepaid_execution_cycles` is `None`, `prepay_execution_cycles` deducts `execution_cost(message_instruction_limit)` from the canister balance and returns the full `CompoundCycles<Instructions>` value. [1](#0-0) 

**Step 2 – First slice executes**: Instructions consumed are tracked by reducing `helper.execution_parameters.instruction_limits` (in-memory only). No balance change occurs mid-execution.

**Step 3 – Checkpoint abort**: `PausedPreUpgradeExecution::abort()` returns `self.original.prepaid_execution_cycles` — the **full original prepaid amount**, unmodified by instructions consumed in the first slice. This is stored verbatim in `ExecutionTask::AbortedInstallCode { prepaid_execution_cycles }`. [2](#0-1) 

**Step 4 – `uninstall_code` executes**: Removes the Wasm module from the canister state. The `AbortedInstallCode` task remains in the task queue untouched. [3](#0-2) 

**Step 5 – Retry via `resume_install_code`**: The `AbortedInstallCode` branch calls `execute_install_code` with `Some(prepaid_execution_cycles)` (the full original prepaid). [4](#0-3) 

**Step 6 – `execute_upgrade` fails immediately**: A fresh `InstallCodeHelper` is created with `instructions_left = message_instruction_limit`. The Wasm module check fails (`execution_state` is `None` after uninstall), so `finish_err` is called with `instructions_left = message_instruction_limit`. [5](#0-4) 

**Step 7 – Full refund issued**: `finish_err` calls `refund_unused_execution_cycles` with `num_instructions = message_instruction_limit` and `num_instructions_initially_charged = message_instruction_limit`, refunding the **entire** `prepaid_execution_cycles`. [6](#0-5) 

**Net result**: `balance_final = balance_initial − execution_cost(limit) + execution_cost(limit) = balance_initial`, even though real instructions were consumed in Step 2.

### Title
Cycles Over-Refund via DTS Abort + Uninstall in Upgrade Path — (`rs/execution_environment/src/execution/upgrade.rs`, `rs/execution_environment/src/execution/install_code.rs`)

### Summary
When a `canister_pre_upgrade` execution is paused across a DTS slice, then aborted at a checkpoint, then `uninstall_code` is called before the retry, the retry of the aborted upgrade fails immediately with `WasmModuleNotFound` and issues a **full refund** of `prepaid_execution_cycles`, even though real instructions were consumed in the aborted slice. The canister controller receives free computation.

### Finding Description
The root cause is in `PausedPreUpgradeExecution::abort()`: it returns `self.original.prepaid_execution_cycles` — the full original prepaid amount — without any partial charge for instructions consumed in the aborted slice. This value is stored in `ExecutionTask::AbortedInstallCode`. When the retry is triggered after `uninstall_code` removes the Wasm module, `execute_upgrade` creates a fresh helper with `instructions_left = message_instruction_limit` and immediately calls `finish_err`, which refunds the full prepaid amount. The instructions consumed in the aborted slice are never charged.

### Impact Explanation
A canister controller can obtain free computation (up to one slice worth of instructions per checkpoint interval) by:
1. Deploying a canister with a long `canister_pre_upgrade`
2. Submitting an upgrade that pauses during `canister_pre_upgrade`
3. Waiting for a checkpoint (which aborts the execution)
4. Calling `uninstall_code`
5. Observing the retry fail with full refund

The attack is repeatable. The free computation per iteration is bounded by the install_code slice instruction limit. The attack rate is bounded by checkpoint frequency (~10 minutes on mainnet).

### Likelihood Explanation
The scenario is reachable by any canister controller without any privileged access. The existing test `dts_uninstall_with_aborted_upgrade` demonstrates the exact sequence is exercised in the test suite, but the cycles balance is never asserted, confirming the accounting gap was not verified.

### Recommendation
In `PausedPreUpgradeExecution::abort()` (and analogous abort paths), issue a partial charge for instructions already consumed before returning `prepaid_execution_cycles`. Alternatively, reduce `prepaid_execution_cycles` by the cost of consumed instructions before storing it in `AbortedInstallCode`, so that any subsequent full refund only covers the truly unused portion. Add a cycles-balance assertion to `dts_uninstall_with_aborted_upgrade` and `dts_uninstall_with_aborted_install_code`.

### Proof of Concept
```
1. Install canister with canister_pre_upgrade that consumes ~slice_limit instructions
2. Submit upgrade → first slice executes (consuming N instructions), execution pauses
3. Trigger checkpoint → PausedInstallCode → AbortedInstallCode (full prepaid stored)
4. Call uninstall_code → Wasm module removed, AbortedInstallCode task remains
5. Scheduler retries AbortedInstallCode → execute_upgrade → WasmModuleNotFound →
   finish_err(instructions_left = message_instruction_limit) → full refund
6. Assert: canister_balance_after == canister_balance_before (N instructions executed for free)
```

The state-machine test `dts_uninstall_with_aborted_upgrade` already sets up steps 1–5; adding a `assert_eq!(balance_after, balance_before)` assertion would demonstrate the over-refund.
