### Title
Byzantine Dealer DoS via Out-of-Range Chunks Bypassing `verify_chunking` Range Bound, Triggering 2 GiB BSGS Allocation in `dec_chunks` — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

---

### Summary

A single Byzantine dealer below the fault threshold can craft a NiDKG dealing whose ZK proofs (`ZKProofDec` / `ZKProofShare`) pass `verify_dealing` while containing out-of-range plaintext chunks. When any receiver calls `load_transcript`, the decryption path falls through to `CheatingDealerDlogSolver::new(n, m)`, which allocates up to 2 GiB and runs for hours per chunk, causing sustained CPU and memory exhaustion on every receiver replica.

---

### Finding Description

**Step 1 — `verify_chunking` does not enforce `[0, CHUNK_MAX]`**

The range check in `verify_chunking` is:

```
ss  = n * m * (CHUNK_SIZE - 1) * CHALLENGE_MASK
zz  = 2 * NUM_ZK_REPETITIONS * ss
``` [1](#0-0) 

With `CHUNK_SIZE = 65536`, `CHALLENGE_MASK = 255`, `NUM_ZK_REPETITIONS = 32`, and a 28-node subnet (`n=28`, `m=16`), `zz ≈ 1.5 × 10¹²`. The honest chunk range is `[0, CHUNK_MAX = 65535]`. Out-of-range chunks (e.g., `0x8000 × delta` for small `delta`) are orders of magnitude below `zz`, so the `z_s < zz` check passes trivially. The proof is a statistical argument over a much wider range than `CHUNK_MAX`; it does **not** enforce `[0, CHUNK_MAX]`.

**Step 2 — `verify_ciphertext_integrity` also passes for out-of-range chunks**

`verify_ciphertext_integrity` only checks pairing equations over the CRSZ structure; it is blind to plaintext chunk values. [2](#0-1) 

The test `should_decrypt_correctly_for_cheating_dealer` explicitly confirms this: it creates chunks where `chunk * delta > CHUNK_MAX`, calls `enc_chunks`, and asserts `verify_ciphertext_integrity` returns `Ok`. [3](#0-2) 

**Step 3 — The dealing passes `verify_dealing` and enters the transcript**

`verify_dealing` calls `verify_zk_proofs`, which calls both `verify_ciphertext_integrity` and `verify_chunking`. [4](#0-3) 

Since both checks pass for out-of-range chunks (as shown above), the dealing is accepted by all honest nodes and included in the transcript via consensus.

**Step 4 — `load_transcript` → `dec_chunks` → `CheatingDealerDlogSolver`**

The call chain is:

`load_transcript` → `compute_threshold_signing_key` → `decrypt` → `dec_chunks` [5](#0-4) 

Inside `dec_chunks`, the honest lookup table covers only `[CHUNK_MIN, CHUNK_MAX]`: [6](#0-5) 

When the honest lookup returns `None` for an out-of-range chunk, the code falls through to:

```rust
let cheating_solver = CheatingDealerDlogSolver::new(n, m);
// It may take hours to brute force a cheater's discrete log.
dlogs[i] = cheating_solver.solve(&powers[i]);
``` [7](#0-6) 

`CheatingDealerDlogSolver::new` allocates a BSGS table capped at 2 GiB: [8](#0-7) 

The ignored test `print_time_for_cheating_dlog_solver_to_run` (marked `#[ignore]` precisely because it causes bazel timeouts) measures this cost explicitly: [9](#0-8) 

---

### Impact Explanation

Every receiver replica in the subnet calls `load_transcript` at each DKG summary block. A single Byzantine dealer's dealing (which passes all verification) causes every receiver to:
- Allocate up to 2 GiB for the BSGS table
- Spend hours of CPU per out-of-range chunk (up to 16 chunks per dealing)

This constitutes sustained CPU and memory exhaustion on all receiver replicas, disrupting DKG key loading and potentially stalling threshold signing for the subnet.

---

### Likelihood Explanation

A Byzantine dealer node is a realistic adversary in the IC threat model (below the fault threshold). The attack requires no privileged access: the dealer simply crafts chunks with `chunk * delta > CHUNK_MAX` and generates a valid ZK proof over those values (which the prover can do, since the proof range is much wider than `CHUNK_MAX`). The attack is repeatable every DKG round.

---

### Recommendation

1. **Tighten the chunking proof range**: Modify `prove_chunking` / `verify_chunking` to enforce that each `s_ij` is in `[0, CHUNK_SIZE - 1]` using a proper range proof (e.g., bit-decomposition or a tighter sigma protocol), so out-of-range chunks are rejected at `verify_dealing` time.
2. **Add a fast pre-check in `dec_chunks`**: Before invoking `CheatingDealerDlogSolver`, verify that the decrypted group element is in the honest range; if not, return `DecErr::InvalidChunk` immediately rather than spending hours on BSGS.
3. **Bound or time-limit `CheatingDealerDlogSolver`**: If the BSGS path must remain, impose a wall-clock or iteration limit and return an error rather than running indefinitely.

---

    crsz: &FsEncryptionCiphertext,
    epoch: Epoch,
    associated_data: &[u8],
    sys: &SysParam,
) -> Result<(), CiphertextIntegrityError> {
    let n = if crsz.cc.is_empty() {
        0
    } else {
        crsz.cc[0].len()
    };
    if crsz.rr.len() != n || crsz.ss.len() != n || crsz.zz.len() != n {
        // In theory, this is unreachable fail because deserialization only succeeds
        // when the vectors of a CRSZ have the same length. (In practice, it's
        // surprising how often "unreachable" code is reached!)
        return Err(CrszVectorsLengthMismatch);
    }

    let id = ftau_extended(&crsz.cc, &crsz.rr, &crsz.ss, sys, epoch, associated_data);

    let g1_neg = G1Affine::generator().neg();
    let precomp_id = G2Prepared::from(&id);

    // TODO(CRP-2550) Each of these checks could be run in parallel

        let s = &crsz.ss[i];

        if !v.is_identity() {
            return Err(InvalidNidkgCiphertext);
        }
    }

fn should_decrypt_correctly_for_cheating_dealer() {
    let epoch = Epoch::from(0);
    let rng = &mut reproducible_rng();
    let associated_data = rng.r#gen::<[u8; 10]>();

    let sys = SysParam::global();
    let key_gen_assoc_data = rng.r#gen::<[u8; 32]>();

    let nodes = 3;

    let mut keys = Vec::new();
    for _ in 0..nodes {
        let key_pair = kgen(&key_gen_assoc_data, sys, rng);
        keys.push(key_pair);
    }
    let pks: Vec<_> = keys.iter().map(|key| key.0.public_key().clone()).collect();

    let mut sij = {
        let mut sij = Vec::with_capacity(nodes);

        for _ in 0..nodes {
            let mut chunks = [0; NUM_CHUNKS];

            for i in 0..NUM_CHUNKS {
                // ensure that multiplying by delta pushes us out of Chunk range
                let chunk = (0x8000 | rng.r#gen::<u16>()) as isize;
                chunks[i] = chunk;
            }
            // this ensures that chunks is the encoding of a scalar less
            // than the group order:
            chunks[0] %= 0x73ee;

            sij.push(chunks);
        }

        sij
    };

    let cheating_i = rng.r#gen::<usize>() % nodes;
    let cheating_j = std::cmp::max(1, rng.r#gen::<usize>() % NUM_CHUNKS);

    let delta = (2 + rng.r#gen::<usize>() % 10) as isize;
    sij[cheating_i][cheating_j] *= delta; // doesn't overflow as delta is small and isize >> u16

    // however the new sij *is* larger than the maximum "legal" chunk
    assert!(sij[cheating_i][cheating_j] > CHUNK_MAX);

    let cheating_chunks = sij
        .iter()
        .map(|c| PlaintextChunks::new_unchecked(*c))
        .collect::<Vec<_>>();

    let pks_and_chunks = pks
        .iter()
        .cloned()
        .zip(cheating_chunks.iter().cloned())
        .collect::<Vec<_>>();

    let (crsz, _witness) = enc_chunks(&pks_and_chunks, epoch, &associated_data, sys, rng);

    // still a valid ciphertext
    assert!(verify_ciphertext_integrity(&crsz, epoch, &associated_data, sys).is_ok());

    for i in 0..keys.len() {
        let secret_key = &keys[i].1;
        let out = dec_chunks(secret_key, i, &crsz, epoch, &associated_data);
        assert_eq!(
            out.unwrap(),
            cheating_chunks[i].recombine_to_scalar(),
            "Message decrypted wrongly"
        );
    }

pub fn verify_dealing(
    dealer_index: NodeIndex,

    epoch: Epoch,
    receiver_keys: &BTreeMap<NodeIndex, FsEncryptionPublicKey>,

    let number_of_receivers =
        number_of_receivers(receiver_keys).map_err(CspDkgVerifyDealingError::SizeError)?;

        .map_err(CspDkgVerifyDealingError::InvalidThresholdError)?;
    verify_receiver_indices(receiver_keys, number_of_receivers)?;
    verify_all_shares_are_present_and_well_formatted(dealing, number_of_receivers)

        epoch,
        receiver_keys,
        &dealing.public_coefficients,
        &dealing.ciphertexts,
        &dealing.zk_proof_decryptability,
        &dealing.zk_proof_correct_sharing,
        &dealer_index.to_be_bytes(),
    )?;
    Ok(())
