### Title
Hardcoded `CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT` of 21,000 Causes Permanent ETH Fee Loss for Smart Contract Recipients - (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

### Summary

The ckETH minter hardcodes a gas limit of `21_000` for all ckETH withdrawal transactions. This is only sufficient for simple EOA-to-EOA ETH transfers. Any user who withdraws ckETH to a smart contract address (multisig, smart contract wallet, DeFi protocol) will have their Ethereum transaction fail with out-of-gas, and will permanently lose the actual Ethereum gas fee paid for the failed transaction.

### Finding Description

In `rs/ethereum/cketh/minter/src/withdraw.rs`, two constants define the gas limits for all withdrawal transactions:

```rust
pub const CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(21_000);
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);
``` [1](#0-0) 

The `estimate_gas_limit` function unconditionally returns `CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT` for all ckETH withdrawals, regardless of whether the destination is an EOA or a smart contract:

```rust
pub fn estimate_gas_limit(withdrawal_request: &WithdrawalRequest) -> GasAmount {
    match withdrawal_request {
        WithdrawalRequest::CkEth(_) => CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
        WithdrawalRequest::CkErc20(_) => CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
    }
}
``` [2](#0-1) 

This gas limit is then used in `create_transaction` to build the EIP-1559 transaction sent to Ethereum: [3](#0-2) 

The `withdraw_eth` endpoint accepts any non-blocked Ethereum address as recipient — including smart contract addresses — without any check for whether the destination is an EOA: [4](#0-3) 

The DID interface itself acknowledges this limitation:

```
// IMPORTANT: The current gas limit is set to 21,000 for a transaction so withdrawals to smart contract addresses will likely fail.
withdraw_eth : (WithdrawalArg) -> (variant { Ok : RetrieveEthRequest; Err : WithdrawalError });
``` [5](#0-4) 

When the Ethereum transaction fails (out-of-gas), the minter reimburses the user `withdrawal_amount - effective_fee_paid`, meaning the user permanently loses the actual Ethereum gas fee consumed by the failed transaction: [6](#0-5) 

This is confirmed by the test `should_record_finalized_transaction_and_reimburse_unused_tx_fee_when_cketh_withdrawal_fails`: [7](#0-6) 

And the integration test `should_reimburse` explicitly quantifies the loss:

```rust
let cost_of_failed_transaction = ...;
assert_eq!(cost_of_failed_transaction, 693_077_873_418_000); // ~0.000693 ETH
``` [8](#0-7) 

### Impact Explanation

**Impact: Medium**

Any user who calls `withdraw_eth` with a smart contract address as recipient (e.g., a Gnosis Safe multisig, an Argent wallet, a DeFi vault, or any contract with a non-trivial `receive`/`fallback`) will:

1. Have their ckETH burned on the IC ledger.
2. Have the minter submit an Ethereum transaction with `gas_limit = 21_000`.
3. Have that transaction fail on Ethereum with out-of-gas (smart contract execution requires >21,000 gas).
4. Receive a reimbursement of `withdrawal_amount - actual_gas_fee`, permanently losing the Ethereum gas fee paid for the failed transaction.

At 30 gwei gas price, the loss per failed withdrawal is approximately `21,000 × 30 × 10⁹ = 630,000,000,000 wei ≈ 0.00063 ETH`. At elevated gas prices (e.g., 100+ gwei during congestion), losses scale proportionally. The user's ckETH is not fully returned; the gas cost is a permanent loss.

### Likelihood Explanation

**Likelihood: Medium**

Smart contract wallets (Gnosis Safe, Argent, etc.) are widely used by DeFi power users and institutions. Many users hold ETH in multisig contracts or protocol treasuries and would naturally attempt to withdraw ckETH directly to those addresses. The `withdraw_eth` endpoint accepts any non-blocked address without warning at the call site (the warning exists only in the DID comment, not as a runtime error or rejection). Users who are unaware of this limitation will trigger the failure path and lose funds.

### Recommendation

1. **Validate destination type at withdrawal time**: Before accepting a `withdraw_eth` request, query the Ethereum network (via the EVM RPC canister) to check whether the destination address has contract code (`eth_getCode`). If it does, reject the request with a descriptive error (e.g., `WithdrawalError::RecipientIsSmartContract`) rather than silently accepting it and burning ckETH.

2. **Alternatively, increase the gas limit**: Allow a configurable or higher gas limit for ckETH withdrawals (similar to how `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` is set to `65_000`). However, this alone does not prevent failures for contracts requiring more than the new limit.

3. **Surface the warning as a runtime error**: At minimum, if on-chain validation is not feasible, add a runtime check or a user-facing warning in the canister response rather than only in the DID comment.

### Proof of Concept

**Attacker-controlled entry path:**

1. User (unprivileged IC principal) holds ckETH and calls `icrc2_approve` on the ckETH ledger to allow the minter to burn their tokens.
2. User calls `withdraw_eth` on the ckETH minter with `recipient = "0x<GnosisSafeAddress>"` (a deployed multisig contract).
3. The minter accepts the request, burns the user's ckETH, and queues a withdrawal.
4. `create_transactions_batch` calls `estimate_gas_limit` → returns `21_000` → `create_transaction` builds an EIP-1559 tx with `gas_limit = 21_000`.
5. The transaction is signed via threshold ECDSA and submitted to Ethereum via `eth_sendRawTransaction`.
6. On Ethereum, the Gnosis Safe's `receive` function requires >21,000 gas; the transaction reverts with out-of-gas. All 21,000 gas is consumed.
7. The minter detects `TransactionStatus::Failure` in the receipt and records a reimbursement of `withdrawal_amount - effective_fee_paid`.
8. The user receives back their ckETH minus the gas fee — a permanent, irreversible loss of funds denominated in ckETH. [9](#0-8) [10](#0-9)

        log!(DEBUG, "[create_transactions_batch]: processing {request:?}",);

            nonce,

                );
