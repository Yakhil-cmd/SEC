The code confirms this is a real vulnerability. Let me lay out the exact findings.

### Title
Unprivileged Caller Can Force Arbitrary Principal Into SNS Swap Participation — (`rs/sns/swap/canister/canister.rs`)

---

### Summary

The `refresh_buyer_tokens` update method on the SNS Swap canister accepts an arbitrary `buyer` principal string from any caller and credits that principal's participation without verifying that the caller **is** the buyer. An attacker can send ICP to `subaccount(swap_canister, victim_principal)` on the ICP ledger and then call `refresh_buyer_tokens` with `buyer=victim_principal`, forcing the victim into the swap without their consent.

---

### Finding Description

In `canister.rs`, the ingress handler for `refresh_buyer_tokens` is:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
``` [1](#0-0) 

When `arg.buyer` is non-empty, the caller's identity is **completely ignored**. The resolved principal `p` is passed directly to `refresh_buyer_token_e8s`:

```rust
swap_mut()
    .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
    .await
``` [2](#0-1) 

Inside `refresh_buyer_token_e8s`, there is no check that the caller matches `buyer`. The function queries the ICP ledger balance of `subaccount(this_canister, buyer)` and credits that balance to `buyer`: [3](#0-2) 

It then unconditionally registers `buyer` in `self.buyers` and updates participation totals: [4](#0-3) 

The `confirmation_text` check is not a mitigation: the required text is set at SNS initialization and is publicly readable from the swap state, so any attacker can supply it. [5](#0-4) 

---

### Impact Explanation

**Forced participation without consent:**
The attacker sends ICP to `subaccount(swap_canister, victim_principal)` on the ICP ledger, then calls `refresh_buyer_tokens(buyer=victim_principal)`. The swap canister sees a non-zero balance in the victim's subaccount and registers the victim as a buyer.

**If the swap commits:**
`sweep_icp` transfers the ICP from the victim's subaccount to SNS Governance and mints SNS neurons for the victim. The victim receives unwanted SNS neurons (with potential tax, regulatory, or reputational consequences) and cannot recover the ICP — `error_refund_icp` is blocked for registered buyers whose ICP has not yet been disbursed: [6](#0-5) 

**Premature swap commitment:**
If the swap is near its `max_direct_participation_icp_e8s` target, the attacker can push it over the threshold, triggering early commitment and locking out other legitimate participants who intended to participate.

**If the swap aborts:**
`sweep_icp` returns the ICP to the victim's account (not the attacker's), so the attacker loses their ICP with no recovery path.

---

### Likelihood Explanation

- Requires no privileged role — any IC principal can call `refresh_buyer_tokens` via ingress.
- The only cost to the attacker is the ICP they send (which they lose regardless of outcome).
- The confirmation text, if required, is publicly readable from the swap state.
- The attack is fully executable on-chain with no off-chain dependencies.

---

### Recommendation

Add a caller-equals-buyer guard at the top of the `refresh_buyer_tokens` handler in `canister.rs`. When `arg.buyer` is non-empty, verify that `PrincipalId::from_str(&arg.buyer).unwrap() == caller_principal_id()`, and reject the call otherwise. This matches the documented intent (the proto comment says "If not specified, the caller is used") and enforces that only the buyer can register their own participation. [7](#0-6) 

---

### Proof of Concept

State-machine test sketch:

```rust
// Setup: swap is OPEN, attacker != victim
let attacker = PrincipalId::new_user_test_id(1);
let victim   = PrincipalId::new_user_test_id(2);

// Step 1: attacker sends ICP to victim's subaccount on the swap canister
icp_ledger_transfer(
    attacker,
    Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&victim)) },
    min_participant_icp_e8s,
);

// Step 2: attacker calls refresh_buyer_tokens with buyer=victim
let response = state_machine.execute_ingress_as(
    attacker,                          // caller = attacker
    swap_canister_id,
    "refresh_buyer_tokens",
    Encode!(&RefreshBuyerTokensRequest {
        buyer: victim.to_string(),     // buyer = victim
        confirmation_text: Some(publicly_known_confirmation_text),
    }).unwrap(),
);

// Step 3: assert victim is now registered as a buyer without ever calling anything
assert!(response.is_ok());
let buyer_state = swap.get_buyer_state(&GetBuyerStateRequest { principal_id: Some(victim) });
assert!(buyer_state.buyer_state.is_some()); // victim forced into swap
```

The canister currently accepts this call and registers the victim. The fix is to assert `attacker == victim` (i.e., `caller == buyer`) before proceeding. [8](#0-7)
