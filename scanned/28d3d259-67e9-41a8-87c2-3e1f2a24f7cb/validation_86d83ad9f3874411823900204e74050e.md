### Title
ckETH Minter Burns Different Gas Fee Than Displayed to User in `withdraw_erc20` - (File: `rs/ethereum/cketh/minter/src/main.rs`, `rs/ethereum/cketh/minter/src/tx.rs`)

---

### Summary

The ckETH minter's `withdraw_erc20` endpoint internally refreshes the Ethereum gas fee estimate from the EVM RPC canister at execution time, which can differ from the fee previously returned to the user by the `eip_1559_transaction_price` query. The amount of ckETH actually burned from the user's balance is determined by this freshly fetched value, not by the value the user observed before initiating the withdrawal. No re-confirmation step exists between the fee display and the burn.

---

### Finding Description

The `eip_1559_transaction_price` query endpoint returns the minter's cached gas fee estimate:

```rust

        result.timestamp = Some(ts);
        result

``` [1](#0-0) 

When the user subsequently calls `withdraw_erc20`, the minter calls `estimate_erc20_transaction_fee()` (line 430), which internally invokes `lazy_refresh_gas_fee_estimate()`: [2](#0-1) 

`lazy_refresh_gas_fee_estimate()` uses a 60-second staleness window. If the cached estimate is older than 60 seconds at the time `withdraw_erc20` executes, it performs a live `eth_feeHistory` call to the EVM RPC canister and stores a new estimate:

```rust

The freshly fetched estimate is then used to burn ckETH from the user's balance: [4](#0-3)

---

### Impact Explanation

A user who queries `eip_1559_transaction_price` to determine the fee, approves the minter for that amount on the ckETH ledger, and then calls `withdraw_erc20` may have a materially different amount of ckETH burned than they observed. The documentation explicitly recommends approving a large blanket allowance (e.g., 1 ETH) to avoid repeated approvals: [7](#0-6) 

With such a large standing allowance, the minter can burn a significantly higher fee than the user anticipated — up to the full approved amount — with no re-confirmation step. Overcharged transaction fees are explicitly **not reimbursed**: [8](#0-7) 

The impact is direct, irreversible financial loss of ckETH for the user.

---

### Likelihood Explanation

Ethereum gas fees are volatile and can spike by multiples within seconds during network congestion. The 60-second cache window is wide enough that a user who queries the fee estimate near the end of a cache period will trigger a fresh fetch during `withdraw_erc20`. This is a normal, reachable condition for any unprivileged user calling the public `withdraw_erc20` endpoint. No special attacker capability is required — the divergence arises from ordinary network fee volatility combined with the cache expiry timing.

---

### Recommendation

**Short term:** Inside `withdraw_erc20`, use the same cached estimate that was returned by `eip_1559_transaction_price` at the time the user queried it, or return the fee that will actually be burned to the caller before executing the burn, allowing the caller to abort if the fee is unacceptable. At minimum, cap the burned fee to the value the user explicitly approved via `icrc2_approve` rather than fetching a potentially higher live value.

**Long term:** Implement a two-phase commit: the user first calls a dry-run endpoint that returns the exact fee to be burned (with a short validity window), then calls `withdraw_erc20` with that fee locked in. If the live fee at execution time exceeds the locked value, the call should fail rather than silently burning more.

---

### Proof of Concept

1. User calls `eip_1559_transaction_price` (query) at time T. The cached estimate is 59 seconds old. The returned `max_transaction_fee` is **F₁ = 0.005 ETH**. The response includes `timestamp` showing the estimate age.
2. User calls `icrc2_approve` on the ckETH ledger, approving the minter for **1 ETH** (large blanket approval as documented).
3. User calls `withdraw_erc20` at time T+2s. The cached estimate is now 61 seconds old, exceeding `MAX_AGE_NS`.
4. `lazy_refresh_gas_fee_estimate()` performs a live `eth_feeHistory` call to the EVM RPC canister. Due to a gas spike, the new estimate yields **F₂ = 0.05 ETH**.
5. The minter burns **0.05 ETH** of ckETH from the user's balance — 10× more than the user observed. The user receives no notification and has no opportunity to cancel.
6. The `Erc20WithdrawalRequest` records `max_transaction_fee = 0.05 ETH`. The overcharge is not reimbursed.

The attacker-controlled entry path is: unprivileged ingress sender → `withdraw_erc20` update call → `lazy_refresh_gas_fee_estimate()` → EVM RPC canister fee history → burn of user ckETH at live rate. No privileged access is required.

    async fn do_refresh() -> Option<GasFeeEstimate> {
        let _guard = match TimerGuard::new(TaskType::RefreshGasFeeEstimate) {
            Ok(guard) => guard,
            Err(e) => {
                log!(
                    DEBUG,
                    "[refresh_gas_fee_estimate]: Failed retrieving guard: {e:?}",
                );
                return None;
            }
        };

        let fee_history = match eth_fee_history().await {
            Ok(fee_history) => fee_history,
            Err(e) => {
                log!(
                    INFO,
                    "[refresh_gas_fee_estimate]: Failed retrieving fee history: {e:?}",
                );
                return None;
            }
        };

        let gas_fee_estimate = match estimate_transaction_fee(&fee_history) {
            Ok(estimate) => {
                mutate_state(|s| {
                    s.last_transaction_price_estimate =
                        Some((ic_cdk::api::time(), estimate.clone()));
                });
                estimate
            }
            Err(e) => {
                log!(
                    INFO,
                    "[refresh_gas_fee_estimate]: Failed estimating gas fee: {e:?}",
                );
                return None;
            }
        };
        log!(
            INFO,
            "[refresh_gas_fee_estimate]: Estimated transaction fee: {:?}",
            gas_fee_estimate,
        );
        Some(gas_fee_estimate)
    }

    async fn eth_fee_history() -> Result<FeeHistory, MultiCallError<FeeHistory>> {
        read_state(rpc_client)
            .fee_history((5_u8, BlockTag::Latest))
            .with_reward_percentiles(vec![20])
            .with_cycles(MIN_ATTACHED_CYCLES)
            .try_send()
            .await
            .reduce_with_strategy(StrictMajorityByKey::new(|fee_history: &FeeHistory| {
                Nat::from(fee_history.oldest_block.clone())
            }))
    }
