### Title
Async Reentrancy in SNS Governance `refresh_neuron` Allows Cached Stake Inflation — (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `refresh_neuron` function makes an inter-canister call to the ledger to fetch a neuron's balance **without first acquiring a neuron lock**. Because IC canisters can process other ingress messages between an `await` point and its response, a concurrent `disburse` (or similar stake-reducing) call can execute against the same neuron while the balance query is in flight. When the ledger response returns, `refresh_neuron` overwrites `cached_neuron_stake_e8s` with the stale, pre-disburse balance, permanently inflating the neuron's recorded stake and voting power beyond what the ledger actually holds.

---

### Finding Description

**Vulnerable function:** `refresh_neuron` in `rs/sns/governance/src/governance.rs` (lines 4237–4297).

```rust
async fn refresh_neuron(&mut self, nid: &NeuronId) -> Result<(), GovernanceError> {
    // ... neurons_fund check (no await) ...

    // ← NO neuron lock acquired here
    let balance = self.ledger.account_balance(account).await?;   // await point

    let neuron = self.get_neuron_result_mut(nid)?;
    match neuron.cached_neuron_stake_e8s.cmp(&balance.get_e8s()) {
        Ordering::Less => { neuron.update_stake(balance.get_e8s(), now); }
        Ordering::Greater => { neuron.update_stake(balance.get_e8s(), now); }
        Ordering::Equal => (),
    };
    Ok(())
}

---

### Impact Explanation

**Impact: High**

An attacker can permanently inflate `cached_neuron_stake_e8s` — the field used to compute voting power — without holding the corresponding tokens on the ledger. Concretely:

- Inflated voting power lets the attacker swing SNS governance proposals (parameter changes, treasury disbursements, canister upgrades) beyond their legitimate stake.
- The inconsistency between the ledger balance and the cached stake persists until the neuron is dissolved or another refresh corrects it, but the attacker can repeat the attack to keep the stake inflated.

---

### Likelihood Explanation

**Likelihood: Medium**

- The attack entry point is a publicly callable `manage_neuron` ingress message — no privileged role required.
- The attacker controls message ordering: they send `ClaimOrRefresh` (triggering `refresh_neuron`) and immediately queue a `Disburse` command. IC message scheduling guarantees the `Disburse` will be processed during the inter-canister call window.
- The attacker only needs an existing SNS neuron and enough tokens to transfer to the neuron subaccount before the attack.
- No threshold corruption, DNS hijack, or social engineering is required.

---

### Recommendation

Acquire a neuron lock (add the neuron to `neurons_with_ongoing_operations` / `in_flight_commands`) **before** the `account_balance` await in `refresh_neuron`, mirroring the NNS governance pattern:

```rust
async fn refresh_neuron(&mut self, nid: &NeuronId) -> Result<(), GovernanceError> {
    // ... neurons_fund check ...

    // ADD: lock the neuron before any await
    let _neuron_lock = self.lock_neuron_for_command(nid, ...)?;

    let balance = self.ledger.account_balance(account).await?;
    // ... rest unchanged ...
}
```

Additionally, after the await, verify that `cached_neuron_stake_e8s` has not been modified by a concurrent operation before applying the update (checks-effects-interactions pattern).

### Proof of Concept

**Setup:** Attacker owns SNS neuron `N` with `cached_neuron_stake_e8s = S` (e.g., 100 tokens). Ledger subaccount for `N` also holds `S`.

**Step 1 — Fund the subaccount:** Attacker transfers `T` additional tokens (e.g., 100) to neuron `N`'s governance subaccount. Ledger balance is now `S + T = 200`.

**Step 2 — Trigger `refresh_neuron`:** Attacker sends `manage_neuron { ClaimOrRefresh { By::MemoAndController } }` to SNS governance. The canister calls `self.ledger.account_balance(account)` and **suspends at the await**. The in-flight ledger query will return `200`.

**Step 3 — Concurrent `Disburse`:** While the ledger query is in flight (at least one IC round), attacker sends `manage_neuron { Disburse { amount: T } }`. Because `refresh_neuron` holds **no lock**, `Disburse` is processed immediately:
- Ledger transfers `T` tokens to attacker's account.
- `cached_neuron_stake_e8s` is updated to `S = 100`.

**Step 4 — Stale balance applied:** The ledger responds to `refresh_neuron` with `200`. `refresh_neuron` resumes, sees `cached_neuron_stake_e8s (100) < balance (200)`, and calls `neuron.update_stake(200, now)`. [4](#0-3) 

**Result:**
- Ledger subaccount holds `S = 100` tokens.
- `cached_neuron_stake_e8s = 200` (inflated by `T`).
- Attacker's external account received `T = 100` tokens from the disburse.
- Attacker retains `2×` the legitimate voting power with only `S` tokens at stake.
