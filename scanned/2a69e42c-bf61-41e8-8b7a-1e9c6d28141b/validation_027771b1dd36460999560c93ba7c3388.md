### Title
Missing Reentrancy Guard in `execute_treasury_manager_deposit` Allows Concurrent Proposals to Bypass 50% Treasury Limit - (File: rs/sns/governance/src/extensions.rs)

### Summary
The `execute_treasury_manager_deposit` function in SNS Governance lacks a reentrancy/concurrency guard. Because the treasury balance check and the actual ICRC-2 approval are separated by multiple inter-canister `await` points, two concurrently executing `ExecuteExtensionOperation` proposals can each independently pass the 50% treasury balance check and together drain more than 50% of the SNS treasury — the exact protection the check was designed to enforce.

### Finding Description

`validate_deposit_operation_impl` performs an async balance check against the SNS and ICP ledgers, rejecting any deposit request that exceeds 50% of the current treasury balance: [1](#0-0) 

This check is async — it awaits two ledger `account_balance` calls. After the check passes, execution continues into `execute_treasury_manager_deposit`, which makes further async calls: [2](#0-1) 

The function contains three distinct `await` suspension points:
1. `governance.treasury_manager_deposit_context().await` — fetches SNS token symbol
2. `governance.approve_treasury_manager(...).await` — two sub-awaits to set ICRC-2 approvals on SNS and ICP ledgers
3. `governance.env.call_canister(extension_canister_id, "deposit", ...).await` — calls the extension canister [3](#0-2) 

At each `await` point, the IC scheduler can interleave other messages. Two adopted `ExecuteExtensionOperation` proposals execute as independent background tasks spawned by `start_proposal_execution`: [4](#0-3) 

The `ExecuteExtensionOperation` action path has **no concurrency lock**: [5](#0-4) 

By contrast, `perform_transfer_sns_treasury_funds` — which performs a similar treasury operation — explicitly acquires a `thread_local` lock to prevent concurrent execution: [6](#0-5) 

**Attack sequence:**

1. Two `ExecuteExtensionOperation` proposals (A and B) are adopted, each requesting 50% of the SNS treasury.
2. Proposal A begins executing and suspends at the first `await` inside `validate_deposit_operation_impl` (ledger balance query).
3. While A is suspended, Proposal B begins executing and also suspends at the same `await`. Both see the same treasury balance; both pass the 50% check.
4. Proposal A resumes, calls `approve_treasury_manager`, setting ICRC-2 approval to 50% of treasury.
5. Proposal B resumes, calls `approve_treasury_manager`, **overwriting** A's approval with its own 50% approval.
6. Proposal A calls `deposit` on the extension canister. The extension canister now holds an approval for 50% (B's approval, not A's). It executes `icrc2_transfer_from` for 50%.
7. Proposal B calls `deposit` on the extension canister. The approval has been consumed; B's deposit fails or uses a stale/zero allowance.

In a variant where the extension canister is adversarial (or buggy), it can exploit the window between step 4 and step 5 to use A's approval before B overwrites it, then use B's approval when B calls deposit — effectively double-spending from the treasury.

Additionally, the balance check in `validate_deposit_operation_impl` is stale by the time `approve_treasury_manager` executes. If treasury funds are moved between the check and the approval (e.g., by a concurrent `TransferSnsTreasuryFunds` proposal), the 50% invariant is violated silently.

### Impact Explanation

The 50% treasury balance limit — the primary protection against a single governance proposal draining the SNS treasury — can be bypassed. Two concurrent `ExecuteExtensionOperation` proposals can together drain up to 100% of the SNS treasury (both SNS tokens and ICP). The extension canister receives ICRC-2 approvals that collectively exceed the intended limit, and the ledger enforces only per-approval limits, not aggregate limits across concurrent proposals.

### Likelihood Explanation

Likelihood is **medium**. It requires two `ExecuteExtensionOperation` proposals to be adopted and executing concurrently. On a live SNS, proposals execute via `run_periodic_tasks` heartbeats. If two such proposals are adopted in the same or adjacent heartbeat cycles, they will execute concurrently. A malicious SNS token holder with sufficient voting power (or a coalition) can deliberately submit two proposals timed to execute concurrently. The extension canister mechanism is new and the missing lock is an oversight relative to the existing `TransferSnsTreasuryFunds` lock.

### Recommendation

Add a `thread_local` reentrancy guard to `perform_execute_extension_operation` (or inside `execute_treasury_manager_deposit`) identical to the one in `perform_transfer_sns_treasury_funds`:

```rust
thread_local! {
    static IN_PROGRESS_PROPOSAL_ID: RefCell<Option<u64>> = const { RefCell::new(None) };
}
let release_on_drop = acquire(&IN_PROGRESS_PROPOSAL_ID, proposal_id);
if let Err(already_in_progress) = release_on_drop {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!("Another ExecuteExtensionOperation proposal ({already_in_progress}) is already in progress."),
    ));
}
```

Additionally, re-validate the treasury balance check immediately before calling `approve_treasury_manager` (after all prior awaits complete) to ensure the 50% invariant holds at the moment of approval, not just at the moment of proposal validation.

### Proof of Concept

```
T=0: Proposal A (request 50% SNS treasury) adopted, starts executing
T=1: Proposal A suspends at ledger.account_balance().await (balance = 1000 SNS)
T=2: Proposal B (request 50% SNS treasury) adopted, starts executing  
T=3: Proposal B suspends at ledger.account_balance().await (balance = 1000 SNS)
T=4: Proposal A resumes: 500 ≤ 500 (50% check passes), proceeds
T=5: Proposal B resumes: 500 ≤ 500 (50% check passes), proceeds
T=6: Proposal A: approve_treasury_manager(ext, sns=500, icp=500) → ledger sets allowance[ext]=500
T=7: Proposal B: approve_treasury_manager(ext, sns=500, icp=500) → ledger OVERWRITES allowance[ext]=500
T=8: Proposal A: call_canister(ext, "deposit") → ext calls icrc2_transfer_from(gov→ext, 500 SNS) ✓
T=9: Proposal B: call_canister(ext, "deposit") → ext calls icrc2_transfer_from(gov→ext, 500 SNS) ✓
     (allowance was reset to 500 at T=7, so this succeeds)

Result: 1000 SNS (100% of treasury) transferred to extension canister.
        The 50% limit was bypassed entirely.
```

async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}

    fn start_proposal_execution(&mut self, proposal_id: u64, action: Action) {
        // `perform_action` is an async method of &mut self.
        //
        // Starting it and letting it run in the background requires knowing that
        // the `self` reference will last until the future has completed.
        //
        // The compiler cannot know that, but this is actually true:
        //
        // - in unit tests, all futures are immediately ready, because no real async
        //   call is made. In this case, the transmutation to a static ref is abusive,
        //   but it's still ok since the future will immediately resolve.
        //
        // - in prod, "self" is a reference to the GOVERNANCE static variable, which is
        //   initialized only once (in canister_init or canister_post_upgrade)
        let governance: &'static mut Governance = unsafe { std::mem::transmute(self) };
        spawn_in_canister_env(governance.perform_action(proposal_id, action));
    }
