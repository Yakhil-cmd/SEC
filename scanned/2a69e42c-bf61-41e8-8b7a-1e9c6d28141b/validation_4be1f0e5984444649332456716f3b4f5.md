### Title
Silent Neuron Deduplication in `NeuronsFundSnapshotPb::validate()` Due to Incomplete `Ord` Implementation - (`rs/nns/governance/src/neurons_fund.rs`, `rs/types/types/src/messages/inter_canister.rs`)

### Summary

`NeuronsFundNeuronPortion` implements `Ord` using only `controller` and `maturity_equivalent_icp_e8s` as sort keys, omitting `id`. `NeuronsFundSnapshotPb::validate()` collects deserialized neuron portions into a `BTreeSet<NeuronsFundNeuronPortion>` before converting to the canonical `BTreeMap`. When two Neurons' Fund neurons share the same controller principal and the same maturity amount, they compare `Ordering::Equal` under `Ord`, causing `BTreeSet` to silently discard one entry. The surviving snapshot is then used to compute refunds and SNS swap participation, permanently excluding the dropped neuron.

### Finding Description

`NeuronsFundNeuronPortion` derives `PartialEq` (all fields) but implements `Ord` manually using only two fields:

```rust
impl Ord for NeuronsFundNeuronPortion {
    fn cmp(&self, other: &Self) -> Ordering {
        match self.controller.cmp(&other.controller) {
            Ordering::Equal => self
                .maturity_equivalent_icp_e8s
                .cmp(&other.maturity_equivalent_icp_e8s),
            ordering => ordering,
        }
    }
}
``` [1](#0-0) 

The `id` field (`NeuronId`) is never consulted. Two distinct neurons owned by the same principal with identical maturity amounts therefore compare `Equal` under `Ord` while comparing unequal under `PartialEq`, violating Rust's ordering contract.

`NeuronsFundSnapshotPb::validate()` uses this type as the element of a `BTreeSet`:

```rust
let neurons_fund = self
    .neurons_fund_neuron_portions
    .iter()
    .enumerate()
    .map(|(i, n)| { n.validate().map_err(...) })
    .collect::<Result<BTreeSet<_>, _>>()?;
Ok(NeuronsFundSnapshot::new(neurons_fund))
``` [2](#0-1) 

`BTreeSet` uses `Ord` exclusively for all membership decisions. When the second of two equal-by-`Ord` neurons is inserted, `BTreeSet::insert` silently discards it (the existing entry is kept, the new one is dropped). The resulting set is then passed to `NeuronsFundSnapshot::new`, which maps each surviving portion by its `id`:

```rust
pub fn new<I>(neurons: I) -> Self
where
    I: IntoIterator<Item = NeuronsFundNeuronPortion>,
{
    let neurons = neurons.into_iter().map(|n| (n.id, n)).collect();
    Self { neurons }
}
``` [3](#0-2) 

The dropped neuron's `NeuronId` never appears in the resulting `BTreeMap`, so all downstream operations — refund computation via `diff()`, maturity restoration via `refund_maturity_to_neurons_fund`, and SNS neuron creation — are performed without it.

### Impact Explanation

The `NeuronsFundSnapshot` is the authoritative record of which NNS neurons reserved maturity for an SNS swap and how much. It is stored in the NNS Governance canister's persistent state as a `NeuronsFundSnapshotPb` protobuf and deserialized (calling `validate()`) on every canister upgrade. If two neurons with the same controller and maturity are present in the snapshot at upgrade time, one is permanently erased from the in-memory snapshot. Subsequent calls to `settle_neurons_fund_participation` compute refunds against the corrupted snapshot: the erased neuron's reserved maturity is never returned, constituting a permanent loss of ICP-equivalent maturity for that neuron's controller. The `diff()` function, which computes `initial_snapshot - final_snapshot` to determine refunds, will simply not produce an entry for the missing neuron. [4](#0-3) 

### Likelihood Explanation

A single NNS principal can control multiple neurons. Two neurons under the same controller reaching identical `maturity_e8s_equivalent` values is plausible over time through normal staking and maturity accrual. Both neurons must have joined the Neurons' Fund and been included in the same `NeuronsFundSnapshot`. The silent drop only manifests when `validate()` is called — i.e., on NNS Governance canister upgrades, which occur regularly as part of the NNS upgrade process. No privileged access or external oracle is required; the triggering condition is ordinary neuron ownership and routine canister upgrades.

### Recommendation

Include `id` as a final tie-breaker in `NeuronsFundNeuronPortion::Ord`, consistent with how other priority structures in the codebase handle ties (e.g., `MessagePool`'s `size_queue` uses message `Id` as a tie-breaker, and `RefundPool` uses `CanisterId`):

```rust
impl Ord for NeuronsFundNeuronPortion {
    fn cmp(&self, other: &Self) -> Ordering {
        self.controller
            .cmp(&other.controller)
            .then_with(|| self.maturity_equivalent_icp_e8s.cmp(&other.maturity_equivalent_icp_e8s))
            .then_with(|| self.id.cmp(&other.id))
    }
}
```

Alternatively, replace the intermediate `BTreeSet` in `validate()` with a `Vec` and detect duplicates explicitly, returning an error rather than silently dropping entries. [5](#0-4) 

### Proof of Concept

1. Principal `P` controls two NNS neurons `N1` and `N2`, both with `maturity_e8s_equivalent = 5_000_000_000`.
2. Both neurons join the Neurons' Fund; an SNS swap opens and both are included in the initial `NeuronsFundSnapshot` with portions `(N1, amount=X)` and `(N2, amount=X)`.
3. The snapshot is serialized to `NeuronsFundSnapshotPb` and stored in the NNS Governance canister's stable memory.
4. The NNS Governance canister is upgraded (a routine operation). During `post_upgrade`, the state is deserialized; `NeuronsFundSnapshotPb::validate()` is called.
5. Inside `validate()`, both `NeuronsFundNeuronPortion` entries for `N1` and `N2` compare `Ordering::Equal` (same `controller = P`, same `maturity_equivalent_icp_e8s = 5_000_000_000`). `BTreeSet` retains whichever was inserted first and discards the other.
6. The resulting `NeuronsFundSnapshot` contains only one of `{N1, N2}`.
7. When the swap finalizes and `settle_neurons_fund_participation` is called, `diff(initial, final)` produces a refund only for the surviving neuron. The dropped neuron's reserved maturity is never refunded and is permanently lost. [2](#0-1) [6](#0-5)
