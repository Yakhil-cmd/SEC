### Title
Third-Party Caller Can Force SNS Swap Participation on Behalf of Any User, Bypassing Confirmation-Text Consent Gate - (File: rs/sns/swap/canister/canister.rs)

### Summary
The SNS Swap canister's `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal in its request payload. Any unprivileged ingress caller can supply a victim's principal as `buyer` together with the publicly-known `confirmation_text`, forcing the victim's ICP (already sitting in the swap subaccount) to be committed to the swap without the victim ever explicitly consenting to the swap terms.

### Finding Description
`refresh_buyer_tokens` is a public `#[update]` endpoint on the SNS Swap canister. Its handler resolves the acting principal from the request body, not from `ic_cdk::api::msg_caller()`:

```rust
// rs/sns/swap/canister/canister.rs  lines 128-143
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()   // ← attacker-supplied
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, ...)  // ← attacker-supplied confirmation_text
        .await
``` [1](#0-0) 

Inside `refresh_buyer_token_e8s`, the only consent gate is `validate_confirmation_text`, which compares the caller-supplied string against the SNS-configured `confirmation_text`:

```rust
// rs/sns/swap/src/swap.rs  lines 1149-1150
self.validate_confirmation_text(confirmation_text)?;
``` [2](#0-1) 

The `confirmation_text` is stored in the public `Init` protobuf field and is readable by anyone via `get_sale_parameters`: [3](#0-2) 

The function then reads the ICP balance of the victim's swap subaccount from the ledger and records the participation:

```rust
// rs/sns/swap/src/swap.rs  lines 1153-1163
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),  // victim's subaccount
};
icp_ledger.account_balance(account).await ...
``` [4](#0-3) 

Attack path:
1. Victim transfers ICP to the swap subaccount (the standard first step of participation) but has not yet called `refresh_buyer_tokens` — perhaps they are still reviewing the swap terms.
2. Attacker reads the public `confirmation_text` from `get_sale_parameters`.
3. Attacker sends an ingress call to `refresh_buyer_tokens` with `buyer = victim_principal` and `confirmation_text = <correct public text>`.
4. The swap canister records the victim's ICP as committed participation, bypassing the victim's need to explicitly confirm the terms.

This is structurally identical to the reported ERC-2612 permit vulnerability: a third party uses the victim's prior authorization (ICP transfer to the subaccount) together with attacker-controlled parameters (the `confirmation_text`) to act on the victim's behalf without their explicit consent at the time of the action.

### Impact Explanation
When an SNS swap has a `confirmation_text` set (the mechanism designed to enforce informed consent), any attacker can bypass that consent gate for any victim who has already transferred ICP to the swap subaccount. The victim's ICP is committed to the swap without them ever explicitly agreeing to the terms. If the swap commits, the victim receives SNS tokens in exchange for their ICP — a financial outcome they did not explicitly authorize. The `confirmation_text` field exists precisely to prevent uninformed or unwilling participation; this bypass nullifies that protection entirely.

### Likelihood Explanation
The attack requires only:
- The SNS to have a `confirmation_text` set (a common governance best-practice for SNS launches).
- The victim to have transferred ICP to the swap subaccount but not yet called `refresh_buyer_tokens` (a normal intermediate state in the two-step participation flow).
- The attacker to read the public `confirmation_text` (zero-effort, it is a query call).

All three conditions are realistic and observable on-chain. No privileged access, no key compromise, and no threshold attack is required. Any unprivileged ingress sender can execute this.

### Recommendation
Resolve the `buyer` principal exclusively from `ic_cdk::api::msg_caller()` when a `confirmation_text` is required. If third-party notification (calling `refresh_buyer_tokens` on behalf of another) must be supported, the `confirmation_text` validation must be skipped or the confirmation must be tied to a separate, caller-authenticated step. Concretely:

```rust
// Only allow third-party refresh when no confirmation_text is required.
// If confirmation_text is set, require buyer == caller.
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let p = PrincipalId::from_str(&arg.buyer).unwrap();
    if swap().init_or_panic().confirmation_text.is_some()
        && p != caller_principal_id()
    {
        panic!("confirmation_text requires buyer == caller");
    }
    p
};
```
