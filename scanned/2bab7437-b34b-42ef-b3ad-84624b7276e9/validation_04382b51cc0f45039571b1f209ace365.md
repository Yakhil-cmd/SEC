### Title
Stale Treasury Balance Check Allows SNS Deposit to Exceed 50% Safety Limit at Execution Time - (File: `rs/sns/governance/src/extensions.rs`)

---

### Summary

The SNS governance framework enforces a 50% treasury balance safety limit on deposit proposals, but this check is performed only at **proposal validation time** (submission), not at **execution time** (after the voting period). Between these two points, treasury balances can legitimately decrease due to other proposals executing, causing `execute_treasury_manager_deposit` to proceed with amounts that now exceed the intended 50% safety cap — the exact analog of the GoodEntry H-01 pattern where a stale value read at one time is used to gate an action that executes under different conditions.

---

### Finding Description

In `rs/sns/governance/src/extensions.rs`, `validate_deposit_operation_impl` fetches the current treasury balances and rejects any deposit request exceeding 50% of either balance: [1](#0-0) 

This validation runs when the proposal is submitted. The validated amounts (`treasury_allocation_sns_e8s`, `treasury_allocation_icp_e8s`) are frozen into a `ValidatedDepositOperationArg` struct. When the proposal is later adopted and executed, `execute_treasury_manager_deposit` uses these frozen amounts directly: [2](#0-1) 

There is **no re-check** of the 50% limit against the current treasury balance at execution time. The `approve_treasury_manager` call proceeds with the original amounts regardless of how the treasury balance has changed during the voting period. [3](#0-2) 

The treasury manager DID itself explicitly acknowledges the related slippage risk that compounds this issue: [4](#0-3) 

---

### Impact Explanation

The 50% safety limit — the primary guard against excessive single-operation treasury exposure — is bypassed at execution time. An SNS treasury deposit proposal validated against a balance of 100 SNS (requesting 50 SNS, exactly 50%) will still execute if the treasury balance has since dropped to 60 SNS, depositing 50 SNS which is now 83% of the remaining balance. Combined with the acknowledged slippage risk (price ratio at execution differs from approval time), this can result in the SNS treasury losing a disproportionate share of its assets in a single governance-approved operation. The safety invariant "no single deposit exceeds 50% of treasury" is not maintained.

---

### Likelihood Explanation

Moderate. No malicious actor is required. SNS governance routinely executes multiple proposals in sequence — a `TransferSnsTreasuryFunds` proposal and a treasury manager deposit proposal can both pass voting and execute in order, with the first reducing the balance below the threshold assumed by the second. This is a natural race condition in any active SNS with concurrent governance activity. The entry path is any SNS token holder submitting a deposit proposal.

---

### Recommendation

Re-validate the 50% balance limit inside `execute_treasury_manager_deposit` immediately before calling `approve_treasury_manager`, using a fresh ledger balance query at execution time. The existing `validate_deposit_operation_impl` logic can be reused or extracted into a shared helper called at both validation and execution time.

---

### Proof of Concept

1. SNS treasury holds **100 SNS** and **200 ICP**.
2. A deposit proposal is submitted requesting **50 SNS** and **100 ICP** (exactly 50% of each). `validate_deposit_operation_impl` queries balances, the check passes, and the `ValidatedDepositOperationArg` is stored with `treasury_allocation_sns_e8s = 50`.
3. During the voting period, a separate `TransferSnsTreasuryFunds` proposal executes, transferring **60 SNS** out of the treasury. Treasury now holds **40 SNS** and **200 ICP**.
4. The deposit proposal is adopted. `execute_treasury_manager_deposit` is called with the frozen `treasury_allocation_sns_e8s = 50`.
5. `approve_treasury_manager` issues an ICRC-2 approval for **50 SNS** against a treasury that only holds **40 SNS** — 125% of the current balance, far exceeding the intended 50% cap.
6. The treasury manager calls `transfer_from` for 50 SNS; if the balance is sufficient (e.g., balance dropped only to 60 SNS), the full 60%+ of remaining treasury is deposited into the DEX at whatever price ratio exists at execution time, with no slippage protection enforced by the framework. async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

    async fn approve_treasury_manager(
        &self,
        treasury_manager_canister_id: CanisterId,
        sns_amount_e8s: u64,
        icp_amount_e8s: u64,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: treasury_manager_canister_id.get().0,
            subaccount: None,
        };

        let expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS);
        let expiry_time_nsec = expiry_time_sec.saturating_mul(NANO_SECONDS_PER_SECOND);

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

        self.ledger
            .icrc2_approve(
                to,
                sns_amount_e8s,
                Some(expiry_time_nsec),
                self.transaction_fee_e8s_or_panic(),
                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making SNS Token treasury transfer: {e}"),
                )
            })?;

        self.nns_ledger
            .icrc2_approve(
                to,
                icp_amount_e8s,
                Some(expiry_time_nsec),
                icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(),
                self.icp_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making ICP Token treasury transfer: {e}"),
                )
            })?;

        Ok(())
    }

async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;
