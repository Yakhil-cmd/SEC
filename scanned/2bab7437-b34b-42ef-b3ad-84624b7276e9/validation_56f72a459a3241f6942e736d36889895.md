### Title
Missing Per-IP/Per-Subnet Rate Limiting on Query and Read\_State Endpoints at the Boundary Node - (File: `rs/boundary_node/ic_boundary/src/core.rs`)

---

### Summary
The boundary node's `setup_router` function wires per-IP and per-subnet rate limiting **exclusively to update-call routes**, leaving query and read\_state routes with no equivalent built-in throttle. An unprivileged attacker with network access to any API boundary node can send unlimited rapid-fire query or read\_state requests to enumerate canister state or brute-force canister-level access controls implemented via query methods.

---

### Finding Description

In `rs/boundary_node/ic_boundary/src/core.rs`, `setup_router` constructs three distinct route groups and applies rate limiting asymmetrically:

**Call routes — optional per-IP and per-subnet rate limiting applied:** [1](#0-0) 

```rust
let canister_call_routes = {
    let mut route = Router::new()
        .route(PATH_CALL_V2, ...)
        .route(PATH_CALL_V3, ...)
        .route(PATH_CALL_V4, ...);

    if let Some(rl) = cli.rate_limiting.rate_limit_per_second_per_ip {
        route = RateLimit::try_from(rl).unwrap().add_ip_rate_limiting(route);
    }
    if let Some(rl) = cli.rate_limiting.rate_limit_per_second_per_subnet {
        route = RateLimit::try_from(rl).unwrap().add_subnet_rate_limiting(route)
    }
    route
};
```

**Query routes — no rate limiting applied:** [2](#0-1) 

**Read\_state routes — no rate limiting applied:** [3](#0-2) 

All three groups are merged and wrapped in `canister_layers`: [4](#0-3) 

`canister_layers` does include `middleware_generic_limiter`, but that limiter is also optional — it is only instantiated when `rate_limit_generic_file` or `rate_limit_generic_canister_id` CLI flags are explicitly set: [5](#0-4) 

The bouncer is similarly opt-in: [6](#0-5) 

The CLI definition confirms both per-IP and per-subnet limits are `Option<u32>` with no default value, meaning they are off unless explicitly configured: [7](#0-6) 

The concrete route paths affected are:
- `POST /api/v2/canister/{canister_id}/query`
- `POST /api/v3/canister/{canister_id}/query`
- `POST /api/v2/canister/{canister_id}/read_state`
- `POST /api/v3/canister/{canister_id}/read_state`
- `POST /api/v3/subnet/{subnet_id}/query`
- `POST /api/v2/subnet/{subnet_id}/read_state`
- `POST /api/v3/subnet/{subnet_id}/read_state` [8](#0-7) 

The `GenericLimiter.evaluate` also defaults to `Decision::Pass` when no rules are loaded, meaning an unconfigured limiter is a no-op: [9](#0-8) 

---

### Impact Explanation

1. **Canister query-method brute-force**: Any canister that implements access control via a query method (e.g., `is_authorized(principal)`, `validate_code(code)`, `check_invite(token)`) can be called at wire speed with no throttle. An attacker can enumerate valid principals, codes, or tokens to bypass canister-level access controls.

2. **Read\_state message-ID enumeration**: The `read_state` endpoint accepts `request_status` paths. An attacker can rapidly probe arbitrary message IDs to determine which ones exist and what their status is, leaking information about in-flight or recently completed calls.

3. **Replica resource exhaustion**: Each query call is executed synchronously by a replica node. Unlimited query flooding consumes replica CPU and memory, degrading service for legitimate users on the same subnet — even if the attacker never exceeds the (absent) boundary-node limit.

The asymmetry is operationally dangerous: an operator who configures `--rate-limit-per-second-per-ip` believing it protects all IC API traffic will find that query and read\_state routes remain completely unthrottled.

---

### Likelihood Explanation

- The boundary node is publicly reachable by any internet user.
- Anonymous query calls require no identity or authentication.
- Authenticated query calls require only a self-generated key pair (no registration, no approval).
- Sending CBOR-encoded query envelopes is trivially automatable with standard HTTP tooling.
- The default deployment has no generic limiter rules and no bouncer enabled, so the attack surface is present in any default boundary-node deployment.

Likelihood: **Medium** — the capability is trivially available to any attacker; exploitation depends on whether a target canister exposes a query method that leaks useful information or implements guessable access control.

---

### Recommendation

1. Apply `add_ip_rate_limiting` to `canister_query_routes` and `canister_read_state_routes` in `setup_router`, mirroring the existing pattern for `canister_call_routes`.
2. Provide non-`None` default values for `rate_limit_per_second_per_ip` and `rate_limit_per_second_per_subnet` so that rate limiting is active by default rather than opt-in.
3. Document clearly that the generic limiter must be configured to cover query/read\_state if per-IP/per-subnet flags are not set.
