### Title
Averaged Reward Pool Across Type3 Nodes Causes Reward Mismatch for Unassigned Nodes Using Extrapolated Failure Rate - (File: rs/node_rewards/rewards_calculation/src/performance_based_algorithm/mod.rs)

---

### Summary

The Internet Computer's performance-based node provider rewards system contains two compounding reward accounting mismatches that are directly analogous to the reported Polygon vulnerability: (1) all Type3/Type3.1 nodes in the same country share a single averaged base reward regardless of their individual regional rates, and (2) unassigned (non-subnet-member) nodes receive a performance penalty derived from the **average** of their co-provider's assigned nodes' failure rates, rather than a neutral or individually-tracked rate. Both mismatches cause rewards to be distributed in a pooled/averaged fashion that does not accurately reflect the individual contribution or cost basis of each node.

---

### Finding Description

**Mismatch 1 — Averaged Type3 Base Rewards Pool**

In `calculate_base_rewards_by_region_and_type`, all Type3 and Type3.1 nodes belonging to the same node provider in the same country are grouped into a single pool. The total reward for the group is computed with a sequential decay coefficient, then **divided equally** among all nodes in the group regardless of their individual regional rate or coefficient:

```rust
// V1: averages rates and coefficients first, then applies reduction
let avg_rate = avg(rates.as_slice()).unwrap_or_default();
let avg_coeff = avg(coeff.as_slice()).unwrap_or_default();
// ...
let region_rewards_avg = avg(&region_rewards).unwrap_or_default();
```

```rust
// V2: sorts by (rate desc, coeff desc), applies sequential reduction, then averages
let region_rewards_avg = total_rewards / Decimal::from(nodes_count);
```

Every node in the group receives `region_rewards_avg` as its `base_rewards_xdr_permyriad`, regardless of whether it is in a high-rate or low-rate sub-region. A node in a high-rate sub-region (e.g., California at 20,000 XDR/day) receives the same base reward as a node in a low-rate sub-region (e.g., Nevada at 10,000 XDR/day) when they are grouped under the same country. [1](#0-0) 

**Mismatch 2 — Extrapolated Failure Rate for Unassigned Nodes**

In `calculate_provider_rewards`, nodes that are not assigned to any subnet on a given day (i.e., `NonSubnetMember` nodes) receive a performance penalty based on the **arithmetic average** of the relative failure rates of the provider's *assigned* nodes:

```rust
let extrapolated_failure_rate = if !relative_nodes_failure_rate.is_empty() {
    let values: Vec<Decimal> = relative_nodes_failure_rate.values().cloned().collect();
    avg(&values).unwrap_or_default()
} else {
    Decimal::ZERO
};
```

This extrapolated rate is then fed into `calculate_performance_multipliers` as the failure rate for all unassigned nodes:

```rust
let daily_failure_rate_used = relative_nodes_failure_rate
    .get(&node.node_id)
    .copied()
    .unwrap_or(*extrapolated_failure_rate);
```

If a provider has one assigned node with a high relative failure rate (e.g., 0.35), all of its unassigned nodes will be penalized with that same rate, even though those nodes had no opportunity to fail or succeed. Conversely, if the assigned nodes all have 0% failure rate, unassigned nodes receive a 0% extrapolated rate and full rewards — even if the unassigned nodes are actually malfunctioning. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Mismatch 1 (Type3 averaging):** A node provider operating nodes in both a high-rate and a low-rate sub-region within the same country will have their high-rate nodes' rewards diluted by the low-rate nodes, and vice versa. The reward paid to each node does not match the rate that was agreed upon for that specific region. This is a **cycles/resource accounting bug** — ICP minted to node providers does not accurately reflect the per-node regional rate commitments encoded in the rewards table. Over a monthly reward period, this can result in systematic over- or under-payment relative to the governance-approved reward table.

**Mismatch 2 (extrapolated failure rate):** A node provider with one poorly-performing assigned node will have all of its unassigned nodes penalized by the same failure rate. This means a provider can be penalized for nodes that had no measurable performance data. Conversely, a provider with all well-performing assigned nodes will have its unassigned nodes receive full rewards regardless of their actual state. This is a **cycles/resource accounting bug** — ICP minted to node providers does not accurately reflect actual node performance.

The governance canister calls `get_node_providers_rewards` which calls the node rewards canister's `get_node_providers_rewards` endpoint, which executes this calculation and returns XDR amounts that are then minted as ICP to node providers. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The performance-based rewards system is live on mainnet (enabled per NNS Proposal 135054, as noted in the governance CHANGELOG). Any node provider with Type3/Type3.1 nodes in multiple sub-regions of the same country, or with a mix of assigned and unassigned nodes, will experience this mismatch every reward period. No special attacker action is required — the mismatch occurs automatically during normal reward calculation. The entry path is the periodic `mint_monthly_node_provider_rewards` call triggered by the governance canister's heartbeat. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **Type3 base rewards:** Instead of assigning every node in the country group the same averaged reward, assign each node its individually-computed reward based on its position in the sorted decay sequence. The total pool is already computed correctly; the fix is to assign `rate * running_coeff_at_position_i` to node `i` rather than `total / n` to all nodes.

2. **Extrapolated failure rate:** For nodes with no subnet assignment on a given day, use a fixed neutral rate (e.g., 0%) rather than the average of the provider's assigned nodes. Alternatively, document clearly that unassigned nodes inherit the provider's average performance, and ensure this is reflected in governance proposals and node provider agreements. The CHANGELOG entry for Proposal 138288 already noted a prior fix for providers with *no* assigned nodes (using 0%), but the case of providers with *some* assigned nodes still uses the average. [8](#0-7)

### Proof of Concept

**Type3 averaging mismatch (V2):**

Consider a node provider with:
- Node A: Type3 in California, rate = 20,000 XDR/day, coeff = 0.80
- Node B: Type3.1 in Nevada, rate = 10,000 XDR/day, coeff = 0.90

V2 sorts by rate desc: processes A first (20,000 × 1.0 = 20,000), then B (10,000 × 0.80 = 8,000). Total = 28,000. Average = 14,000.

Both nodes receive 14,000 XDR/day as base reward. Node A (which should receive 20,000) is underpaid by 6,000 XDR/day. Node B (which should receive 10,000) is overpaid by 4,000 XDR/day. The governance-approved reward table is not honored per-node. [9](#0-8) 

**Extrapolated failure rate mismatch:**

Consider a node provider with:
- Node X: assigned to subnet, relative failure rate = 0.35 (mid-penalty zone)
- Node Y: unassigned (NonSubnetMember)

`extrapolated_failure_rate = avg([0.35]) = 0.35`. Node Y receives `performance_multiplier = 1 - (0.35 - 0.1)/(0.6 - 0.1) * 0.8 = 0.6`, i.e., a 40% reward reduction, despite having no measurable failure data. This is confirmed by the unit test `test_compute_nodes_performance_multiplier` which explicitly shows the unassigned node receiving the same penalty as the mid-failure-rate node. [10](#0-9)

    pub async fn get_node_providers_rewards(
        canister: &'static LocalKey<RefCell<NodeRewardsCanister>>,
        request: GetNodeProvidersRewardsRequest,
    ) -> GetNodeProvidersRewardsResponse {
        let from_date = NaiveDate::try_from(request.from_day)?;
        let to_date = NaiveDate::try_from(request.to_day)?;
        canister.with_borrow(|c| c.validate_reward_period(from_date, to_date))?;

        let mut rewards_per_node_provider: BTreeMap<Principal, u64> = BTreeMap::new();

        let algorithm_version = request.algorithm_version.unwrap_or_default();
        for day in from_date.iter_days().take_while(|d| *d <= to_date) {
            let result_for_day = canister
                .with_borrow(|canister| {
                    canister.calculate_rewards_for_date(&day, request.algorithm_version)
                })
                .map_err(|e| format!("Could not calculate rewards: {e:?}"))?;

            for (provider_id, provider_rewards) in &result_for_day.provider_results {
                rewards_per_node_provider
                    .entry(provider_id.0)
                    .and_modify(|total| {
                        *total += provider_rewards.total_adjusted_rewards_xdr_permyriad
                    })
                    .or_insert(provider_rewards.total_adjusted_rewards_xdr_permyriad);
            }

            #[cfg(target_arch = "wasm32")]
            let _ = ic_cdk::call::Call::bounded_wait(
                ic_cdk::api::canister_self(),
                "reset_instructions",
            )
            .await
            .unwrap();
        }

        Ok(NodeProvidersRewards {
            algorithm_version,
            rewards_xdr_permyriad: rewards_per_node_provider,
        })
    }
