### Title
Missing `http2_max_header_list_size` Cap on SOCKS-Proxied HTTP/2 Connections Allows Memory Exhaustion in HTTPS Outcalls Adapter — (`rs/https_outcalls/adapter/src/rpc_server.rs`)

---

### Summary

`create_socks_proxy_client` builds a hyper `Client` without calling `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)`, unlike the direct client which caps HTTP/2 response header list size at 52 KiB. A malicious HTTPS server reachable via the SOCKS fallback path can send arbitrarily large HTTP/2 HEADERS frames that hyper buffers entirely in memory before any application-level check runs, causing a memory spike or OOM crash of the adapter process on the affected replica.

---

### Finding Description

**Root cause — one call site, one missing line:**

The direct client is constructed with the 52 KiB cap:

```rust
// lines 88-90
let client = Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)   // 52 * 1024
    .build::<_, Full<Bytes>>(direct_https_connector);
``` [1](#0-0) 

The SOCKS client is constructed **without** it:

```rust
// lines 119-127
Client::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(
    builder
        .enable_all_versions()
        .wrap_connector(SocksConnector { ... }),
)
``` [2](#0-1) 

A `grep` across the entire repository confirms `http2_max_header_list_size` appears exactly once — only in the direct-client path. The SOCKS client's default is `u32::MAX` (≈ 4 GiB), as set by hyper-util when the builder option is omitted.

**Why the application-level check does not save it:**

After `http_resp` is returned from hyper, the code counts `headers_size_bytes` and rejects oversized headers:

```rust
// lines 374-417
let mut headers_size_bytes = 0;
for (k, v) in http_resp.headers() { ... }
let remaining_limit = req.max_response_size_bytes
    .checked_sub(headers_size_bytes as u64)
    .ok_or_else(|| { /* LimitExceeded error */ })?;
``` [3](#0-2) 

This check runs **after** hyper has already decoded and heap-allocated the full HEADERS frame. The `http2_max_header_list_size` setting is the only mechanism that causes hyper/h2 to reject oversized headers at the HTTP/2 framing layer, before allocation. Without it on the SOCKS path, the allocation happens unconditionally.

**Trigger path:**

```
canister http_request (IPv4-only target)
  → direct connect fails (replica has no IPv4)
  → or_else closure fires → do_https_outcall_socks_proxy
  → get_socks_client → create_socks_proxy_client (no header cap)
  → socks_client.request(...)
  → malicious server sends HTTP/2 HEADERS frame with e.g. 200 MiB of headers
  → h2 decoder allocates all of it
  → adapter process OOM / memory spike
``` [4](#0-3) 

---

### Impact Explanation

The HTTPS outcalls adapter runs as a separate process. An OOM crash kills that process on the affected replica. During the restart window, all HTTPS outcalls from that replica fail. Because the IC requires a consensus threshold of responses across replicas, losing one replica's adapter degrades (but does not halt) outcall availability for the subnet. Repeated triggering (the SOCKS client is cached per proxy URI, so the same uncapped client is reused) can keep the adapter in a crash-restart loop for as long as the malicious server is reachable.

---

### Likelihood Explanation

- The attacker only needs to operate an HTTPS server that any canister calls via an IPv4-only address (common for many web services).
- The SOCKS fallback is the production path for IPv4 reachability on IC replicas, which have only IPv6 interfaces.
- No privileged access, key material, or majority corruption is required.
- The attack is repeatable and cheap: sending a large HEADERS frame costs the attacker nothing.

---

### Recommendation

Add `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)` to the `Client::builder` call inside `create_socks_proxy_client`, mirroring the direct-client construction:

```rust
Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)   // add this line
    .build::<_, Full<Bytes>>(
        builder.enable_all_versions().wrap_connector(SocksConnector { ... }),
    )
``` [5](#0-4) 

---

### Proof of Concept

1. Stand up a TLS server that speaks HTTP/2 and, upon any GET, responds with a HEADERS frame containing 200 KiB of synthetic headers (e.g., 25 headers each with an 8 KiB value).
2. Configure a local SOCKS5 proxy that forwards to this server.
3. Issue a canister HTTP outcall to an IPv4-only address so the direct connect fails, with `socks_proxy_addrs` pointing at the local proxy.
4. **Expected (direct path):** hyper rejects the response at the h2 layer with a `PROTOCOL_ERROR` before allocating, because `MAX_HEADER_LIST_SIZE = 52 KiB` is enforced.
5. **Actual (SOCKS path):** hyper allocates the full 200 KiB (or more) before the adapter's application-level check runs. Scale to several MiB to trigger OOM.
6. Assert that the SOCKS path returns an error without buffering oversized headers, and that adapter RSS does not spike — this assertion currently fails.

    fn create_socks_proxy_client(
        &self,
        proxy_addr: Uri,
    ) -> Client<HttpsConnector<SocksConnector<HttpConnector>>, Full<Bytes>> {
        let mut http_connector = HttpConnector::new();
        http_connector.enforce_http(false);
        http_connector
            .set_connect_timeout(Some(Duration::from_secs(self.http_connect_timeout_secs)));

        let builder = HttpsConnectorBuilder::new()
            .with_native_roots()
            .expect("Failed to set native roots");

        #[cfg(not(feature = "http"))]
        let builder = builder.https_only();
        #[cfg(feature = "http")]
        let builder = builder.https_or_http();

        Client::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(
            builder
                .enable_all_versions()
                .wrap_connector(SocksConnector {
                    proxy_addr,
                    auth: None,
                    connector: http_connector,
                }),
        )
    }
