### Title
Byzantine Block Proposer Can Inject Cross-Type Bitcoin Adapter Response, Corrupting ckBTC GetSuccessors Callback â€” (`rs/replicated_state/src/bitcoin.rs`, `rs/bitcoin/consensus/src/payload_builder.rs`)

---

### Summary

The `push_response` function in `rs/replicated_state/src/bitcoin.rs` has an asymmetric validation gap: only the `GetSuccessorsResponse` branch verifies that the `callback_id` exists in the correct context map before pushing to `consensus_queue`. The `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` branches push unconditionally with any `callback_id`. Combined with a `validate_payload` implementation that is effectively a no-op (size-only check), a Byzantine block proposer can include a `SendTransactionResponse` carrying a `callback_id` belonging to a `bitcoin_get_successors_contexts` entry, causing the ckBTC canister's `GetSuccessors` callback to be consumed with a malformed `EmptyBlob` payload, permanently losing the real `GetSuccessors` response.

---

### Finding Description

**Gap 1 â€” `push_response` does not validate response-type/context-type alignment for three of four branches.**

The `GetSuccessorsResponse` branch correctly guards:

```rust
let context = state
    .metadata
    .subnet_call_context_manager
    .bitcoin_get_successors_contexts
    .get_mut(&callback_id)
    .ok_or_else(|| StateError::BitcoinNonMatchingResponse { ... })?;
``` [1](#0-0) 

The `SendTransactionResponse` branch does no such lookup:

```rust
BitcoinAdapterResponseWrapper::SendTransactionResponse(_) => {
    let callback_id = CallbackId::from(response.callback_id);
    let payload = Payload::Data(EmptyBlob.encode());
    state.consensus_queue.push(ConsensusResponse::new(callback_id, payload));
    Ok(())
}
``` [2](#0-1) 

`GetSuccessorsReject` and `SendTransactionReject` are identical in this regard â€” they push to `consensus_queue` with no context lookup. [3](#0-2) 

**Gap 2 â€” `validate_self_validating_payload_impl` is a no-op.**

The entire validation body reduces to an empty-check and a byte count. It never reads the state, never checks whether any `callback_id` in the payload exists in either context map, and never checks whether the response variant matches the pending request type:

```rust
fn validate_self_validating_payload_impl(...) -> Result<NumBytes, ...> {
    if *payload == SelfValidatingPayload::default() {
        return Ok(0.into());
    }
    // ... metrics only
    Ok(NumBytes::new(payload.count_bytes() as u64))
}
``` [4](#0-3) 

`validate_payload` (the `BatchPayloadBuilder` entry point called by consensus during block validation) delegates entirely to this function plus a size check: [5](#0-4) 

Honest replicas therefore accept any well-formed protobuf Bitcoin payload regardless of semantic correctness.

---

### Impact Explanation

When a Byzantine proposer includes `SendTransactionResponse{callback_id: X}` where `X âˆˆ bitcoin_get_successors_contexts`:

1. `push_response` pushes `ConsensusResponse{callback_id: X, payload: EmptyBlob}` to `consensus_queue`.
2. The execution layer finds the ckBTC canister's `GetSuccessors` callback at `X`, delivers `EmptyBlob`, and **consumes the callback**.
3. The ckBTC canister traps or returns an error trying to decode `EmptyBlob` as a `BitcoinGetSuccessorsResponse`.
4. The `bitcoin_get_successors_contexts` entry for `X` is **not removed** (the `SendTransactionResponse` branch never touches it).
5. The next honest proposer includes a legitimate `GetSuccessorsResponse{callback_id: X}`, `push_response` finds the context, pushes a second `ConsensusResponse{callback_id: X}` â€” but the callback is already gone, so the response is silently dropped.

Net result: the ckBTC canister's `GetSuccessors` callback is permanently poisoned for that `callback_id`. Bitcoin chain-tip synchronization for ckBTC stalls until the canister issues a new request. [6](#0-5) 

---

### Likelihood Explanation

- Requires exactly **one Byzantine replica** to win a block-proposal slot â€” well within the BFT fault tolerance threshold (`f < n/3`), and explicitly within scope as "protocol peer behavior below the consensus fault threshold."
- No key material, governance majority, or external oracle is needed.
- The attack is deterministic and locally reproducible as a unit test (as the question's proof idea demonstrates).
- The `validate_payload` guard that should catch this is provably absent.

---

### Recommendation

1. **In `push_response`**: Add context-existence checks to all three unguarded branches, mirroring the `GetSuccessorsResponse` branch. For `SendTransactionResponse`/`SendTransactionReject`, look up `callback_id` in `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if absent. For `GetSuccessorsReject`, look up in `bitcoin_get_successors_contexts`. [7](#0-6) 

2. **In `validate_self_validating_payload_impl`**: Load the certified state and verify that each `callback_id` in the payload exists in the correct context map for its response variant. Reject with `InvalidArtifact` on mismatch. [4](#0-3) 

---
