### Title
`Value::TokenAmount` Uses `u64` for ICRC-1 `Nat` Amounts, Blocking `FieldsDisplay` Consent Messages for Large Transfers - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The ICRC-21 consent message system's `FieldsDisplayMessage` variant stores token amounts as `u64` inside `Value::TokenAmount`. ICRC-1 token amounts are typed as `Nat` (arbitrary precision). When `icrc21_canister_call_consent_message` is called with `FieldsDisplay` device spec for a transfer whose amount exceeds `u64::MAX`, the `nat_to_u64` conversion fails and the endpoint returns a `GenericError` instead of a valid consent message. This is a direct analog to the external report's `uint128` vs `uint256` narrowing bug.

---

### Finding Description

The `Value` enum used in `FieldsDisplayMessage` consent messages defines `TokenAmount` with a `u64` amount field:

```rust
// packages/icrc-ledger-types/src/icrc21/responses.rs
Value::TokenAmount {
    decimals,
    amount: nat_to_u64(amount)?,   // ← fails if Nat > u64::MAX
    symbol: token_symbol.to_string(),
},
```

The helper `nat_to_u64` returns `Err(GenericError)` when the `Nat` value does not fit in `u64`:

```rust
fn nat_to_u64(tokens: Nat) -> Result<u64, Icrc21Error> {
    tokens.0.to_u64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })
}
```

This narrowing is applied in four places: `add_amount` (line 120), `add_fee` (line 156), `add_allowance` (line 195), and `add_existing_allowance` (line 219). All four propagate the error upward, causing `icrc21_canister_call_consent_message` to return `Err` for any `FieldsDisplay` request involving an amount > `u64::MAX`.

The DID interface confirms the structural constraint:

```
type Icrc21Value = variant {
  TokenAmount : record {
    decimals : nat8;
    amount : nat64;   // ← narrowed from Nat
    symbol : text
  };
  ...
};
```

The ckETH ledger uses `U256` token amounts (18 decimal places). `u64::MAX` ≈ 1.844 × 10¹⁹ wei ≈ **18.44 ETH**. Any `FieldsDisplay` consent message request for a ckETH transfer, approval, or transfer_from exceeding ~18.44 ETH will fail with a `GenericError`.

---

### Impact Explanation

Any unprivileged user who calls `icrc21_canister_call_consent_message` on the ICRC-1 ledger (or ICP ledger) with `device_spec = FieldsDisplay` and a token amount > `u64::MAX` receives `Err(GenericError { error_code: 500, description: "Failed to convert tokens to u64" })` instead of a valid consent message. For ckETH, this threshold is ~18.44 ETH — a routine transfer size. Hardware wallets and signing devices that rely on `FieldsDisplay` for structured transaction review cannot display or approve such transactions, effectively blocking user access to the consent-message-gated signing flow for large amounts.

---

### Likelihood Explanation

The ckETH ledger is deployed on mainnet and uses `U256` amounts. Transfers of more than 18.44 ETH are common. Any wallet or dApp that requests `FieldsDisplay` consent messages (the structured, machine-readable format) for such transfers will encounter this error on every attempt. The entry path requires no privilege: a single `icrc21_canister_call_consent_message` update call with a large `amount` field in the encoded `TransferArg` is sufficient to trigger the failure.

---

### Recommendation

Change `Value::TokenAmount` to store the amount as `Nat` (or at minimum `u128`) instead of `u64`, matching the actual domain of ICRC-1 token amounts. Update the DID interface accordingly:

```
// Before
TokenAmount : record { decimals : nat8; amount : nat64; symbol : text };

// After
TokenAmount : record { decimals : nat8; amount : nat; symbol : text };
```

Remove `nat_to_u64` from the `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` paths for `FieldsDisplayMessage`, and pass the `Nat` value directly.

---

### Proof of Concept

1. Deploy or use the mainnet ckETH ICRC-1 ledger (which uses `U256` amounts).
2. Call `icrc21_canister_call_consent_message` as any principal with:
   - `method = "icrc1_transfer"`
   - `arg` = Candid-encoded `TransferArg { amount: 20_000_000_000_000_000_000 }` (20 ETH in wei, > `u64::MAX` ≈ 18.44 ETH)
   - `user_preferences.device_spec = FieldsDisplay`
3. Observe the response is `Err(GenericError { error_code: 500, description: "Failed to convert tokens to u64" })` instead of a valid `ConsentInfo`.
4. Repeat with `device_spec = GenericDisplay` — this succeeds, confirming the bug is isolated to the `FieldsDisplayMessage` path.
