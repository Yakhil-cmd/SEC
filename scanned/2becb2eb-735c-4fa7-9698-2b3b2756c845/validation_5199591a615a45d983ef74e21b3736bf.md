### Title
Non-Fatal Memory Size Validation Allows Sandbox-Reported Oversized Memory to Be Applied — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

---

### Summary

A compromised sandbox process can report arbitrarily large `wasm_memory.size` or `stable_memory.size` values in its `ExecutionFinishedRequest`. The replica's `update_execution_state` function calls `verify_size()` on receipt, but treats any error as **non-fatal**: it logs a message and increments a metric, then unconditionally returns the oversized `CanisterStateChanges` to the caller. There is no check against the canister's configured memory allocation or subnet memory limits at this boundary. The TODO comment in the code explicitly acknowledges this gap.

---

### Finding Description

In `update_execution_state`, the sandbox-reported memory sizes are applied directly:

```rust
wasm_memory.size = execution_state_modifications.wasm_memory.size;  // line 1779
// ...
stable_memory.size = execution_state_modifications.stable_memory.size;  // line 1800
``` [1](#0-0) 

`verify_size()` is then called, but on error the code only logs and increments a counter — it does **not** return early or reject the state:

```rust
if let Err(err) = wasm_memory.verify_size() {
    error!(...);
    self.metrics.sandboxed_execution_critical_error_invalid_memory_size.inc();
}
// execution continues — oversized memory is still returned below
``` [2](#0-1) 

The function then unconditionally returns `CanisterStateChanges` containing the oversized memory: [3](#0-2) 

The developers themselves acknowledge this gap with an explicit TODO:

> "TODO: If a canister has broken out of wasm then it might have allocated more wasm or stable memory than allowed. We should add an additional check here that the canister is still within its allowed memory usage." [4](#0-3) 

Compare this to the analogous guard for `num_instructions_left`, which **does** clamp the value rather than just logging: [5](#0-4) 

The `ExecutionStateModifications` struct carries `wasm_memory.size: NumWasmPages` and `stable_memory.size: NumWasmPages` as plain deserialized fields with no bounds enforced at the IPC boundary: [6](#0-5) 

---

### Impact Explanation

If a sandbox process is compromised (e.g., via a Wasm sandbox escape CVE), it can report `wasm_memory.size = NumWasmPages::MAX` or any value exceeding the canister's memory allocation. The replica will apply this to the canister's `ExecutionState` without re-validating against the canister's `memory_allocation` setting or the subnet's available memory. This can:

- **Bypass canister memory limits**: a canister's effective recorded memory size exceeds its allocation, corrupting accounting.
- **Exhaust subnet memory tracking**: if the inflated size propagates into subnet-level memory accounting, it can starve other canisters or cause subnet-wide resource exhaustion.

---

### Likelihood Explanation

The precondition is a compromised sandbox process, which requires a separate Wasm sandbox escape. This is a non-trivial prerequisite. However, the IC's security architecture explicitly treats the sandbox as an untrusted boundary — the replica is supposed to validate all data returned from the sandbox. The TODO comment and the asymmetric treatment (instructions are clamped; memory is only logged) confirm this is an acknowledged, unmitigated gap in the sandbox isolation boundary, not a theoretical concern.

---

### Recommendation

Mirror the pattern used for `num_instructions_left`: instead of logging and continuing, either:
1. **Clamp** the reported size to the canister's configured memory allocation, or
2. **Reject** the execution output entirely and treat it as a trap/error, preventing state from being committed.

Additionally, add an explicit post-sandbox validation step that checks `wasm_memory.size` and `stable_memory.size` against `canister.memory_allocation()` and the subnet's available memory before constructing `CanisterStateChanges`.

---

### Proof of Concept

Write a differential test that:
1. Constructs a `SandboxExecOutput` with `execution_state_modifications.wasm_memory.size = NumWasmPages::MAX`.
2. Calls `update_execution_state` directly.
3. Asserts that `sandboxed_execution_critical_error_invalid_memory_size` counter is incremented (it will be).
4. Asserts that the returned `CanisterStateChanges.execution_state_changes.wasm_memory.size` is **not** `NumWasmPages::MAX` (this assertion will **fail** with current code, demonstrating the bypass).
5. Verify the canister's memory size in replicated state is not updated to the oversized value.

pub struct MemoryModifications {

#[derive(Clone, PartialEq, Debug, Default, Deserialize, Serialize)]
pub struct StateModifications {
    /// Modifications in the execution state of the canister.
    ///
    /// This field is optional because the state changes might or might not
    /// be applied depending on the method executed.
    pub execution_state_modifications: Option<ExecutionStateModifications>,

    /// Modifications in the system state of the canister.
    ///
    /// The system state changes contain parts that are always applied
    /// and parts that are only applied depending on the method executed
    /// (similarly to `execution_state_modifications`).
    pub system_state_modifications: SystemStateModifications,
}

#[derive(Clone, PartialEq, Debug, Deserialize, Serialize)]
pub struct ExecutionStateModifications {
    /// The state of the global variables after execution.
    pub globals: Vec<Global>,

    /// Modifications in the Wasm memory.
    pub wasm_memory: MemoryModifications,

    /// Modifications in the stable memory.
    pub stable_memory: MemoryModifications,
}
