### Title
No Mechanism in `CkDeposit` Helper Contract Rejects Deposits of Unsupported ERC-20 Tokens - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

The `CkDeposit` Ethereum helper contract, part of the IC chain-fusion infrastructure, accepts deposits of any ERC-20 token without validating whether the token is supported by the ckETH minter canister on the IC. When a user deposits an unsupported ERC-20 token, the tokens are irrevocably transferred to the minter's Ethereum address, but the IC minter never processes the deposit and never mints ckERC20. The funds are permanently locked with no recovery path.

---

### Finding Description

The `depositErc20` function in `CkDeposit` (`DepositHelperWithSubaccount.sol`) performs only a zero-address check on the ERC-20 contract address and then unconditionally transfers tokens to the minter's Ethereum address:

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
}
``` [1](#0-0) 

There is no whitelist check against supported ERC-20 tokens. The contract emits `ReceivedEthOrErc20` for any token address.

On the IC side, the minter enforces the whitelist exclusively through log-scraping topic filters. `ReceivedErc20LogScraping` only fetches logs whose second topic matches one of the supported ERC-20 contract addresses:

```rust
topics.push(
    erc20_smart_contracts_addresses_as_topics(state)
        .collect::<Vec<_>>()
        .into(),
);
``` [2](#0-1) 

If a user deposits an unsupported ERC-20 token, the minter's log scraping never retrieves the corresponding `ReceivedEthOrErc20` event. The `register_deposit_events` function in `deposit.rs` is never called for that event, no `AcceptedErc20Deposit` state transition occurs, and no ckERC20 is minted. [3](#0-2) 

The minter has no mechanism to detect, track, or return tokens deposited for unsupported ERC-20 contracts. The documentation itself acknowledges this gap explicitly:

> "Note that the helper smart contract does not enforce any whitelist of allowed ERC-20 tokens. This is enforced by the minter, which fetches logs only for the supported ERC-20 tokens. Therefore, funds of unsupported ERC-20 tokens could be deposited via the helper smart contract, but the minter will not know anything about it." [4](#0-3) 

The same structural gap exists in the older `ERC20DepositHelper.sol` (`CkErc20Deposit` contract), which also emits `ReceivedErc20` for any ERC-20 address without a whitelist check. [5](#0-4) 

---

### Impact Explanation

Any ERC-20 tokens deposited via `depositErc20` for an unsupported token address are transferred to the minter's Ethereum address and permanently locked. The minter canister has no state for these tokens, no event log entry, and no reimbursement path. The user suffers a complete, irreversible loss of the deposited ERC-20 tokens. The supported token list changes over time via NNS proposals (tokens can be added or removed), meaning a token that was supported at deposit time may later become unsupported, or a user may deposit a token that was never supported.

---

### Likelihood Explanation

The likelihood is **medium**. The `depositErc20` function is a public, permissionless entry point callable by any Ethereum user. The list of supported tokens is maintained off-chain by the IC minter and is not visible to the Ethereum contract. A user who:
- deposits a token that was recently removed from the supported list,
- deposits a token they believe is supported but is not,
- or makes a typo in the ERC-20 contract address

will silently lose their funds. The documentation warning is the only safeguard, and it requires users to proactively query `get_minter_info` before every deposit.

---

### Recommendation

Add a supported-token whitelist to the `CkDeposit` helper contract, maintained by the contract owner (the minter's Ethereum address), and revert `depositErc20` calls for unsupported token addresses. This mirrors the recommendation in the external report: add a mechanism at the entry point (L1 Ethereum contract) to reject deposits that would be considered invalid by the IC minter (L2). Alternatively, implement a minimum-amount check in the helper contract to prevent dust deposits that cannot cover minting costs.

---

### Proof of Concept

1. User identifies an ERC-20 token not in the minter's `supported_ckerc20_tokens` list (queryable via `get_minter_info`).
2. User calls `depositErc20(unsupportedTokenAddress, amount, principal, subaccount)` on the `CkDeposit` contract at `0x18901044688D3756C35Ed2b36D93e6a5B8e00E68`.
3. The contract calls `safeTransferFrom(user, minterAddress, amount)` — tokens are transferred to the minter's Ethereum address.
4. The contract emits `ReceivedEthOrErc20(unsupportedTokenAddress, user, amount, principal, subaccount)`.
5. The IC minter's `ReceivedErc20LogScraping` scrapes logs filtered by supported token topics — the event for the unsupported token is never retrieved. [6](#0-5) 

6. `register_deposit_events` is never called for this deposit; no `AcceptedErc20Deposit` event is recorded; no ckERC20 is minted. [7](#0-6) 

7. The user's ERC-20 tokens remain permanently locked in the minter's Ethereum address with no recovery mechanism.

pub enum ReceivedErc20LogScraping {}

impl LogScraping for ReceivedErc20LogScraping {
    const ID: LogScrapingId = LogScrapingId::Erc20DepositWithoutSubaccount;
    type Parser = ReceivedErc20LogParser;

    fn next_scrape(state: &State) -> Option<Scrape> {
        if state.ckerc20_tokens.is_empty() {
            return None;
        }
        let contract_address = *Self::contract_address(state)?;
        let last_scraped_block_number = Self::last_scraped_block_number(state);

        let mut topics: Vec<_> = vec![Topic::Single(Hex32::from(RECEIVED_ERC20_EVENT_TOPIC))];
        // We add token contract addresses as additional topics to match.
        // It has a disjunction semantics, so it will match if event matches any one of these addresses.
        topics.push(
            erc20_smart_contracts_addresses_as_topics(state)
                .collect::<Vec<_>>()
                .into(),
        );

        Some(Scrape {
            contract_address,
            last_scraped_block_number,
            topics,
        })
    }
