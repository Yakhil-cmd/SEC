### Title
`MintSnsTokens` 7-Day Spending Limit Circuit Breaker Is Completely Disabled — (`rs/sns/governance/src/proposal.rs`, `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister implements a 7-day treasury protection limit for both `TransferSnsTreasuryFunds` and `MintSnsTokens` proposal types. For `TransferSnsTreasuryFunds`, the limit is enforced both at proposal submission and at execution time, with a concurrency lock. For `MintSnsTokens`, the circuit breaker is entirely disabled: the submission-time check returns `Decimal::MAX` (no limit), and the execution function `perform_mint_sns_tokens` contains no limit check at all. Any SNS neuron holder with sufficient voting power can submit and execute an unlimited number of `MintSnsTokens` proposals within a 7-day window, minting an unbounded quantity of SNS tokens and bypassing the intended treasury protection entirely.

---

### Finding Description

**`TransferSnsTreasuryFunds` — correctly guarded:**

`perform_transfer_sns_treasury_funds` acquires a per-proposal concurrency lock and calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` before executing the ledger transfer. [1](#0-0) 

At submission time, `validate_and_render_transfer_sns_treasury_funds` calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which computes `spent_tokens + proposal_amount <= max_tokens` using the real valuation-based limit. [2](#0-1) 

**`MintSnsTokens` — circuit breaker disabled:**

The `TokenProposalAction` implementation for `MintSnsTokens` overrides `recent_amount_total_upper_bound_tokens` to return `Decimal::MAX` unconditionally, with a comment acknowledging the real limit is commented out pending ticket NNS1-2982: [3](#0-2) 

The real limit function `mint_sns_tokens_7_day_total_upper_bound_tokens` exists and is correct, but is commented out at the call site: [4](#0-3) 

The import of `mint_sns_tokens_7_day_total_upper_bound_tokens` is also commented out: [5](#0-4) 

At execution time, `perform_mint_sns_tokens` performs no limit check whatsoever — it directly calls `self.ledger.transfer_funds` with the full `amount_e8s`: [6](#0-5) 

Compare this to `TransferSnsTreasuryFunds`, which has both a pre-transfer limit check and a concurrency lock. `MintSnsTokens` has neither.

The dispatch site confirms `MintSnsTokens` goes directly to `perform_mint_sns_tokens` with no auxiliary valuation or limit check: [7](#0-6) 

The integration test explicitly acknowledges the limit is not enforced and the assertion that should fail is commented out, with `doomed_make_proposal_result.unwrap()` replacing the expected error: [8](#0-7) 

---

### Impact Explanation

An SNS neuron holder (or coalition holding majority voting power) can submit an unlimited number of `MintSnsTokens` proposals within a 7-day window. Each proposal passes submission-time validation because `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`. Each proposal executes without any execution-time limit check. The result is unbounded SNS token inflation, diluting all existing token holders. The intended 7-day treasury protection limit — which for a "medium" treasury caps minting at 25% of treasury value per 7 days — is completely bypassed. This is the direct analog of the external report: the circuit breaker fires too late (or not at all), allowing the limit to be exceeded within a single governance cycle.

---

### Likelihood Explanation

Any SNS with a single whale neuron or a coordinated governance majority can exploit this immediately. The code itself acknowledges the issue via `TODO(NNS1-2982)` comments. No special exploit is required — submitting and passing `MintSnsTokens` proposals is a standard governance action. The only prerequisite is sufficient voting power to pass proposals, which is the normal operating condition for any SNS governance participant.

---

### Recommendation

1. Uncomment the real `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (remove the `TODO(NNS1-2982)` stub returning `Decimal::MAX`).
2. Add an execution-time limit check in `perform_mint_sns_tokens`, analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` for `TransferSnsTreasuryFunds`.
3. Add a concurrency lock in `perform_mint_sns_tokens` analogous to the `IN_PROGRESS_PROPOSAL_ID` lock in `perform_transfer_sns_treasury_funds`.
4. Uncomment the import of `mint_sns_tokens_7_day_total_upper_bound_tokens` in `rs/sns/governance/src/proposal.rs`.

---

### Proof of Concept

1. Deploy an SNS with a whale neuron holding majority voting power.
2. Submit `MintSnsTokens` proposal #1 for `amount_e8s = u64::MAX` (or any large amount). Submission succeeds because `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`.
3. Pass and execute proposal #1. `perform_mint_sns_tokens` calls `self.ledger.transfer_funds(u64::MAX, 0, None, to, memo)` with no limit check.
4. Immediately submit and execute proposal #2 for another `u64::MAX` e8s. Again, no limit check at submission or execution.
5. Repeat within the same 7-day window. The SNS token supply inflates without bound, bypassing the intended 25%-of-treasury-per-7-days cap entirely.

The `TransferSnsTreasuryFunds` equivalent would correctly block step 4 with `PreconditionFailed: 7 day upper bound exceeded`. `MintSnsTokens` has no such protection.

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
