### Title
Voting Reward Purse Permanently Lost When Settled Proposals Have Zero Voting Participation - (`rs/nns/governance/src/governance.rs`, `rs/sns/governance/src/governance.rs`)

---

### Summary

In both NNS and SNS governance, when a reward distribution round contains proposals that are ready to settle but no neurons cast eligible votes on them, the entire reward purse for that round is neither distributed to neurons nor rolled over to the next round. It is silently and permanently destroyed. The root cause is a mismatch between the condition that skips reward distribution (`total_voting_rights < 0.001` / `total_reward_shares == 0`) and the condition that triggers rollover (`settled_proposals.is_empty()`). When proposals are settled but no one voted, both conditions are simultaneously true and false in the wrong combination, causing the purse to fall through a gap.

---

### Finding Description

**NNS Governance — `rs/nns/governance/src/governance.rs`**

In `calculate_voting_rewards()`, `sum_weighted_voting_power` is called over the settled proposals to produce `total_voting_rights`. This value is the sum of `reward_weight * total_potential_voting_power` across all considered proposals. If all neurons have zero deciding voting power at proposal creation time (possible under the `VotingPowerEconomics` feature introduced in proposal 132411, where neurons that do not refresh for ~7 months reach deciding voting power = 0), then `total_potential_voting_power == 0` for those proposals, making `total_voting_rights == 0.0 < 0.001`. [1](#0-0) 

When this guard fires, `reward_distribution` is set to `None`. The `RewardEvent` is then constructed with `settled_proposals: considered_proposals` — a **non-empty** list — and stored as `latest_reward_event`. [2](#0-1) 

In the **next** reward round, `e8s_equivalent_to_be_rolled_over()` is called on the previous event: [3](#0-2) 

`rewards_rolled_over()` returns `self.settled_proposals.is_empty()`. Because proposals **were** settled (non-empty list), this returns `false`, so `e8s_equivalent_to_be_rolled_over()` returns `0`. The `total_available_e8s_equivalent` from the skipped round is **never added** to the next round's purse. It is permanently lost. [4](#0-3) 

**SNS Governance — `rs/sns/governance/src/governance.rs`**

The identical structural flaw exists in SNS governance's `distribute_rewards()`. Here, `total_reward_shares` is computed by summing `ballot.voting_power` only for neurons that cast an `eligible_for_rewards()` vote: [5](#0-4) 

If no neuron voted on any settled proposal (all ballots are `Vote::Unspecified`), `total_reward_shares == dec!(0)` and the distribution block is skipped: [6](#0-5) 

The `RewardEvent` is still stored with non-empty `settled_proposals`: [7](#0-6) 

The SNS rollover check is identical in structure: [8](#0-7) 

`rewards_rolled_over()` returns `false` (proposals were settled), so `e8s_equivalent_to_be_rolled_over()` returns `0`. The SNS reward purse for that round is permanently destroyed.

---

### Impact Explanation

The `total_available_e8s_equivalent` computed for the affected round — representing ICP-equivalent maturity that should have been distributed to voting neurons — is permanently zeroed out. It is not minted, not distributed, and not carried forward. This is a **ledger conservation bug**: the governance canister computes a reward pool from the live ICP supply but then silently discards it. For NNS, the daily pool is on the order of 5–10% of total ICP supply annualized (hundreds of thousands of ICP-equivalent maturity per day). For SNS, the magnitude depends on `VotingRewardsParameters` but can be substantial for large SNS treasuries. The loss is irreversible with no recovery path.

---

### Likelihood Explanation

**SNS (higher likelihood):** An SNS can have very few neurons. If all neurons dissolve, or if an SNS is in an early stage where no neuron holds sufficient dissolve delay to vote, proposals can expire with zero eligible votes. This is a realistic operational scenario for any SNS project. The trigger requires no adversarial action — it occurs naturally.

**NNS (lower but non-zero likelihood):** The `VotingPowerEconomics` feature (proposal 132411) introduces `start_reducing_voting_power_after_seconds` (initially ~0.5 years) and `clear_following_after_seconds` (initially ~1/12 year). A neuron that does not refresh its voting power for approximately 7 months reaches `deciding_voting_power == 0`. If a proposal is created at a moment when all neurons have stale voting power, `total_potential_voting_power == 0` for that proposal, making `total_voting_rights == 0.0 < 0.001`. While unlikely for the full NNS, it becomes more plausible for low-participation subnets or during governance inactivity periods. [9](#0-8) 

---

### Recommendation

The rollover condition must be decoupled from whether proposals were settled. The correct invariant is: **if rewards were not distributed (for any reason), the purse must roll over**. The fix is to change `rewards_rolled_over()` to check whether `distributed_e8s_equivalent == 0` (or equivalently, whether `reward_distribution` was `None`), rather than whether `settled_proposals.is_empty()`.

Concretely, in both NNS and SNS governance:

1. Add a boolean field to `RewardEvent` such as `rewards_were_distributed: bool`, set to `true` only when the distribution block actually executes.
2. Change `rewards_rolled_over()` to return `!self.rewards_were_distributed` (or equivalently `self.distributed_e8s_equivalent == 0 && self.settled_proposals.is_empty()` is the current logic — the fix is to remove the `settled_proposals` dependency).

Alternatively, when `total_voting_rights < 0.001` or `total_reward_shares == 0` is detected with non-empty proposals, the proposals should still be settled but the `RewardEvent` should be recorded with an empty `settled_proposals` list so that the rollover fires correctly — though this has the side effect of leaving proposals in `ReadyToSettle` state indefinitely.

---

### Proof of Concept

**NNS path:**

1. All NNS neurons fail to refresh voting power for > 7 months (or a governance bug causes `total_potential_voting_power == 0` at proposal creation).
2. A proposal is created; its `total_potential_voting_power` is recorded as `0`.
3. The proposal expires and enters `ReadyToSettle`.
4. `distribute_voting_rewards_to_neurons()` runs. `sum_weighted_voting_power` returns `total_voting_rights == 0.0`.
5. Line 6712: `total_voting_rights < 0.001` → `reward_distribution = None`.
6. `RewardEvent` stored with `settled_proposals = [proposal_id]`, `total_available_e8s_equivalent = X > 0`, `distributed_e8s_equivalent = 0`.
7. Next round: `e8s_equivalent_to_be_rolled_over()` on previous event → `rewards_rolled_over()` → `settled_proposals.is_empty()` → `false` → returns `0`.
8. `total_available_e8s_equivalent_float` for the new round does NOT include `X`. The `X` e8s are permanently lost.

**SNS path:**

1. An SNS has proposals expiring with no neuron votes (e.g., all neurons dissolved, or SNS just launched with no active voters).
2. `distribute_rewards()` runs. `total_reward_shares == dec!(0)`.
3. Line 5946: distribution block skipped.
4. `RewardEvent` stored with `settled_proposals = [proposal_id]`, `total_available_e8s_equivalent = Some(Y)`, `distributed_e8s_equivalent = 0`.
5. Next round: `e8s_equivalent_to_be_rolled_over()` → `rewards_rolled_over()` → `settled_proposals.is_empty()` → `false` → returns `0`.
6. `Y` SNS-token-equivalent maturity is permanently lost. [10](#0-9) [3](#0-2) [6](#0-5) [8](#0-7)
