### Title
Stale ICP/XDR Exchange Rate Consumed Without Freshness Check in Cycles Minting Canister — (File: rs/nns/cmc/src/main.rs)

---

### Summary

The Cycles Minting Canister (CMC) converts ICP to cycles using a cached `icp_xdr_conversion_rate`. The function that performs this conversion (`tokens_to_cycles`) reads the stored rate and uses it directly without checking whether the rate's `timestamp_seconds` is recent. If the Exchange Rate Canister (XRC) becomes unavailable for any reason, the CMC silently continues minting cycles at the last successfully cached rate indefinitely, with no maximum-age guard.

---

### Finding Description

The CMC heartbeat calls `update_exchange_rate` every five minutes to refresh `state.icp_xdr_conversion_rate` from the XRC. When the XRC is unreachable or returns errors, the update fails and the cached rate is left unchanged. The rate is then consumed by `tokens_to_cycles`:

```rust
// rs/nns/cmc/src/main.rs  lines 1900-1922
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);   // timestamp_seconds is never read
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles {
                xdr_permyriad_per_icp,
                cycles_per_xdr: state.cycles_per_xdr,
            }
            .to_cycles(amount)),
            None => { /* error */ }
        }
    })
}
``` [1](#0-0) 

The `IcpXdrConversionRate` struct carries a `timestamp_seconds` field, but `tokens_to_cycles` only checks whether the `Option` is `Some`; it never compares `rate.timestamp_seconds` against the current canister time. There is no `MAX_RATE_AGE` constant or equivalent guard anywhere in the consumption path.

The setter `do_set_icp_xdr_conversion_rate` enforces only monotonicity (new timestamp must exceed the stored one) and a non-zero value check — it does not enforce that the rate is recent relative to wall-clock time:

```rust

### Impact Explanation

**Vulnerability class**: cycles/resource accounting bug.

If the ICP/XDR rate cached in the CMC is stale and the real ICP price has fallen significantly since the last successful XRC update, any caller of `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` will receive more cycles per ICP than the current market rate warrants. The protocol effectively subsidizes cycles at the expense of the ICP burned, breaking the economic invariant that 1 ICP always converts to cycles at the current market rate. Conversely, if ICP price has risen, users receive fewer cycles than they should, but the protocol-loss direction (ICP price drop + stale high rate) is the security-relevant case.

---

### Likelihood Explanation

The XRC is a system canister that itself performs HTTPS outcalls to external forex and crypto-exchange endpoints. If those external sources are broadly unavailable (e.g., a major exchange API outage), the XRC returns errors such as `CryptoBaseAssetNotFound`, `StablecoinRateTooFewRates`, or `InconsistentRatesReceived`. The CMC's heartbeat then fails to update the rate. The CMC retries every minute, but the stale rate is used for all minting operations during the outage window. An attacker who observes that the XRC is failing (publicly visible via replica logs or by calling the XRC directly) and that ICP price has dropped can immediately exploit the stale rate by submitting ICP-to-cycles conversion notifications. No privileged access is required; `notify_top_up` and `notify_mint_cycles` are open to any principal.

---

### Recommendation

Add a maximum-age check in `tokens_to_cycles` before using the cached rate. If `now - rate.timestamp_seconds` exceeds a defined threshold (e.g., 30 minutes or 1 hour), the function should return an error rather than use the stale rate:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let now = ic_cdk::api::time() / 1_000_000_000; // nanoseconds → seconds
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) if now.saturating_sub(rate.timestamp_seconds) <= MAX_RATE_AGE_SECONDS => {
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            Some(_) => Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "ICP/XDR conversion rate is stale; minting suspended".to_string(),
            }),
            None => Err(/* existing None error */),
        }
    })
}
```

A reasonable `MAX_RATE_AGE_SECONDS` would be 30–60 minutes, giving the heartbeat several retry windows before minting is suspended.

---

### Proof of Concept

1. Observe that the XRC is returning errors (e.g., by calling `get_exchange_rate` on the XRC canister directly and receiving `StablecoinRateTooFewRates`).
2. Note the timestamp on the CMC's current rate via `get_icp_xdr_conversion_rate` — it will stop advancing.
3. Wait for ICP market price to drop below the stale cached rate (or time the attack during a known price drop coinciding with an XRC outage).
4. Call `notify_top_up` (or `notify_mint_cycles`) on the CMC with an ICP ledger block that burns ICP at the current (lower) market price.
5. The CMC calls `tokens_to_cycles` which reads the stale (higher) `xdr_permyriad_per_icp` and mints more cycles than the current ICP value warrants.
6. The attacker receives excess cycles; the ICP burned is worth less than the cycles received at current market rates.
