### Title
Unchecked Index Panic in `construction_hash` on Empty `EnvelopePair` Vec — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

---

### Summary

The `construction_hash` handler directly indexes `envelope_pairs[0]` after finding a `RequestType::Send` entry via `.find(|(rt, _)| rt.is_transfer())`, with no guard against an empty `Vec<EnvelopePair>`. Because `SignedTransaction::from_str` is a pure CBOR deserialization with no structural validation, an attacker can POST a crafted payload containing `(RequestType::Send, vec![])` to the unauthenticated `/construction/hash` endpoint and trigger an index-out-of-bounds panic.

---

### Finding Description

The vulnerable line is:

```rust
&envelope_pairs[0].update,
``` [1](#0-0) 

`Request` is defined as `(RequestType, Vec<EnvelopePair>)`: [2](#0-1) 

`SignedTransaction::from_str` performs only hex-decode + CBOR deserialization — no invariant enforces that the `Vec<EnvelopePair>` is non-empty: [3](#0-2) 

`RequestType::is_transfer()` returns `true` for `RequestType::Send`: [4](#0-3) 

So the full path is: `find(|(rt, _)| rt.is_transfer())` matches the crafted `(Send, vec![])` tuple → `envelope_pairs[0]` panics on the empty vec.

The fix exists in a *different* code path (`TryFrom<&models::Request> for Request`), which correctly guards with `.first().ok_or_else(...)`: [5](#0-4) 

But `construction_hash` never goes through that conversion — it directly indexes the raw `Vec<EnvelopePair>` from the deserialized `SignedTransaction`.

The endpoint is registered with no authentication: [6](#0-5) 

---

### Impact Explanation

In actix-web (the framework used here), a panic in a handler causes the worker thread to abort and be restarted by the framework. With default `panic=unwind`, the process survives but the in-flight request is dropped with a 500. With `panic=abort` (common in production IC builds for determinism), the entire Rosetta process terminates. In either case, the attacker can trigger this repeatedly with a single crafted HTTP POST, causing continuous worker restarts or a persistent crash loop, denying service to all clients depending on this Rosetta instance for ICP transaction hashing and submission.

---

### Likelihood Explanation

The `/construction/hash` endpoint requires no credentials. The CBOR payload is trivially constructable: serialize `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` to CBOR, hex-encode it, and POST. No prior knowledge of the system state is needed. The attacker does not need to be a ledger participant.

---

### Recommendation

Add an emptiness check before indexing, mirroring the guard already present in `TryFrom<&models::Request> for Request`:

```rust
if envelope_pairs.is_empty() {
    return Err(ApiError::invalid_request("No envelope pairs provided for Send request."));
}
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pairs[0].update,
)
``` [7](#0-6) 

---

### Proof of Concept

```rust
#[test]
fn test_construction_hash_empty_envelope_pairs_does_not_panic() {
    use std::str::FromStr;
    use crate::models::SignedTransaction;
    use crate::request_types::RequestType;

    // Craft a SignedTransaction with a Send request but empty envelope pairs
    let crafted = SignedTransaction {
        requests: vec![(RequestType::Send, vec![])],
    };
    let hex_cbor = crafted.to_string(); // hex(cbor(crafted))

    // Simulate what construction_hash does after from_str succeeds
    let parsed = SignedTransaction::from_str(&hex_cbor).unwrap();
    let result = parsed
        .requests
        .iter()
        .rev()
        .find(|(rt, _)| rt.is_transfer());

    // This will panic at envelope_pairs[0] in the current code
    if let Some((_rt, envelope_pairs)) = result {
        let _ = &envelope_pairs[0]; // <-- panics here
    }
}
```

Running this under `cargo test` with `RUSTFLAGS="-C panic=abort"` will crash the process. Under default settings it panics and unwinds, which actix-web catches — but the attacker can loop this indefinitely.
