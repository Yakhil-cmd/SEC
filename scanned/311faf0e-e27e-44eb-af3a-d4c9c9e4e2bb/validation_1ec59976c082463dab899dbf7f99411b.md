### Title
Unbounded `stripped_idkg_dealings` Count Enables Byzantine Peer to Spawn Unbounded Tokio Tasks — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

---

### Summary

A single Byzantine P2P peer can craft a `StrippedBlockProposal` with an arbitrarily large `stripped_idkg_dealings` repeated field. Because `TryFrom<pb::StrippedBlockProposal>` imposes no count bound, `BlockProposalAssembler::new` allocates a proportionally sized `Vec`, and `assemble_message` immediately spawns one unbounded-lifetime Tokio task per entry — all before the bouncer loop can fire. This allows a sub-threshold Byzantine peer to saturate the Tokio task pool and cause sustained memory pressure on any victim replica.

---

### Finding Description

**Step 1 — No count bound in deserialization.**

`TryFrom<pb::StrippedBlockProposal>` in `stripped.rs` iterates `value.stripped_idkg_dealings` and collects it into a `Vec` with no upper-bound check. The only per-entry validation is that each `IDkgArtifactId` is a `Dealing` variant — a trivially satisfiable structural check. [1](#0-0) 

**Step 2 — `BlockProposalAssembler::new` allocates proportionally.**

`BlockProposalAssembler::new` maps every entry in `stripped_idkg_dealings` into a `Vec<((NodeIndex, IDkgArtifactId), Option<SignedIDkgDealing>)>` with no bound. [2](#0-1) 

**Step 3 — N tasks spawned before the bouncer loop.**

`assemble_message` calls `assembler.missing_stripped_messages()` — which returns all N entries — and immediately spawns one `get_or_fetch` task per entry into a `JoinSet`. The bouncer watcher is only consulted inside the subsequent `loop`, meaning all N tasks are live before any abort can occur. [3](#0-2) 

**Step 4 — Each task loops indefinitely.**

`download_stripped_message` is built with `with_max_elapsed_time(None)` — no overall timeout — and loops forever retrying RPC calls to peers. For fake dealing IDs that no honest peer holds, every task will loop until the `JoinSet` is dropped. [4](#0-3) [5](#0-4) 

**Step 5 — Bouncer fires only after tasks are already running.**

The bouncer watcher is set up after the spawn loop and only aborts assembly (dropping the `JoinSet`) when the block becomes unwanted. Until finalization (~1–3 s), all N tasks remain live. [6](#0-5) 

---

### Impact Explanation

With N = 10^5 fake `stripped_idkg_dealings` entries per crafted artifact, and IC block time ~1 s with finalization latency ~2–3 s, the attacker can maintain ~2–3 × 10^5 live Tokio tasks at steady state by repeating the attack each block. Each task holds transport handles, backoff state, and RPC buffers. At scale this causes:

- **Tokio task-pool saturation**: the runtime's thread pool is consumed servicing sleeping/retrying tasks, starving legitimate consensus work.
- **Memory pressure**: each task's stack and associated `Arc` clones accumulate proportionally.

The `DefaultBodyLimit::disable()` on the RPC axum router removes any HTTP-layer size guard on the serving side, and the artifact transport layer does not impose a per-artifact element count limit. [7](#0-6) 

---

### Likelihood Explanation

The attacker needs only to be a single Byzantine peer in the subnet's P2P overlay — well below the consensus fault threshold. The peer must advertise a `StrippedConsensusMessageId` whose underlying block proposal hash passes the bouncer (e.g., by mirroring a legitimately advertised block's hash), then serve a crafted `pb::StrippedBlockProposal` body with an inflated `stripped_idkg_dealings` field. No privileged access, key material, or majority corruption is required.

---

### Recommendation

1. **Add a count bound in `TryFrom<pb::StrippedBlockProposal>`**: Reject any `stripped_idkg_dealings` whose length exceeds the maximum number of verified dealings a valid IC block can contain (bounded by subnet size × transcripts per block, a small constant).

2. **Bound task concurrency**: Cap the number of tasks spawned in `assemble_message` (e.g., via a semaphore or by asserting the count against a protocol-level maximum before spawning).

3. **Add a per-task timeout**: Give `download_stripped_message` a finite `max_elapsed_time` so orphaned tasks self-terminate even if the `JoinSet` is not dropped promptly.

---

### Proof of Concept

```
1. Craft a pb::StrippedBlockProposal where:
   - pruned_block_proposal = a valid (empty) block proto
   - unstripped_consensus_message_id = hash of a currently-advertised honest block proposal
   - stripped_idkg_dealings = 100,000 entries, each with a syntactically valid
     IDkgArtifactId::Dealing(...) but a random/nonexistent hash

2. Serve this artifact over the P2P transport when the victim node requests
   the StrippedConsensusMessage for that block ID.

3. Observe: BlockProposalAssembler::new allocates a Vec of 100,000 entries;
   assemble_message spawns 100,000 get_or_fetch tasks; each task enters
   download_stripped_message's infinite retry loop.

4. Repeat every ~1 s (each new block). Measure Tokio task count and heap
   growth in the victim replica process.
```
