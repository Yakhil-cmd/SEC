### Title
`MintSnsTokens` Proposal Enforces No Effective 7-Day Minting Rate Limit - (`rs/sns/governance/src/proposal.rs`)

---

### Summary

The SNS governance `MintSnsTokens` proposal action has its 7-day rolling minting cap intentionally disabled, returning `Decimal::MAX` as the effective upper bound. While the identical rate-limiting mechanism is correctly enforced for `TransferSnsTreasuryFunds`, the `MintSnsTokens` equivalent is commented out pending ticket NNS1-2982. Any SNS governance majority can therefore mint an unbounded quantity of SNS tokens via repeated proposals, with no treasury-valuation-based cap applied.

---

### Finding Description

In `rs/sns/governance/src/proposal.rs`, the `TokenProposalAction` trait is implemented for both `TransferSnsTreasuryFunds` and `MintSnsTokens`. The trait method `recent_amount_total_upper_bound_tokens` is the mechanism that caps how many tokens can be moved/minted within a rolling 7-day window.

For `TransferSnsTreasuryFunds`, the cap is correctly enforced: [1](#0-0) 

For `MintSnsTokens`, the correct implementation is commented out and replaced with `Decimal::MAX`: [2](#0-1) 

The validation function `treasury_valuation_if_proposal_amount_is_small_enough_or_err` calls this method and rejects proposals only when `proposal_amount_tokens > allowance_remainder_tokens`. Since `allowance_remainder_tokens` is derived from `Decimal::MAX`, the check never rejects any `MintSnsTokens` proposal regardless of amount: [3](#0-2) 

The execution path `perform_mint_sns_tokens` then calls the ledger directly with no amount guard: [4](#0-3) 

The integration test `sns_can_mint_funds_via_proposals` explicitly acknowledges this gap: the assertion that a second large mint should fail is commented out, and `doomed_make_proposal_result.unwrap()` is called instead, confirming that unlimited repeated minting currently succeeds: [5](#0-4) 

The correct rate-limiting function `mint_sns_tokens_7_day_total_upper_bound_tokens` exists and is fully implemented in the `proposals_amount_total_limit` crate — it simply is not wired in: [6](#0-5) 

---

### Impact Explanation

An SNS governance majority can submit and pass `MintSnsTokens` proposals of arbitrary size, repeatedly, with no 7-day cap. This inflates the SNS token supply without bound, diluting all existing token holders. The asymmetry is concrete: `TransferSnsTreasuryFunds` is capped at 25% of treasury per 7 days for medium-sized treasuries, while `MintSnsTokens` has no cap at all. A governance majority that is compromised, captured, or acting adversarially can drain value from all token holders through unlimited inflation.

**Impact: Medium** — token supply inflation, dilution of all SNS token holders, no recovery mechanism once tokens are minted.

---

### Likelihood Explanation

SNS governance is open to any token holder with sufficient voting power. Unlike the external report's "whitelisted managers," SNS neurons are permissionlessly acquired. A whale, a coordinated group, or a governance attack (e.g., flash-loan-style accumulation during low participation) can pass `MintSnsTokens` proposals. The missing cap is not a theoretical gap — the integration test explicitly demonstrates it succeeds today.

**Likelihood: Medium** — requires governance majority, but SNS governance is not restricted to a trusted whitelist.

---

### Recommendation

Uncomment the correct `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (the block marked `TODO(NNS1-2982): Uncomment`) and delete the `Decimal::MAX` stub below it. This will enforce the same treasury-valuation-based 7-day rolling cap on minting that already applies to treasury transfers. [2](#0-1) 

Also uncomment the corresponding assertion in the integration test: [5](#0-4) 

---

### Proof of Concept

1. Deploy an SNS with a governance majority neuron.
2. Submit a `MintSnsTokens` proposal with `amount_e8s = u64::MAX` (≈ 184 billion SNS tokens).
3. Vote to adopt the proposal.
4. `validate_and_render_mint_sns_tokens` calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which calls `MintSnsTokens::recent_amount_total_upper_bound_tokens` → returns `Decimal::MAX`. The check `proposal_amount_tokens > allowance_remainder_tokens` evaluates to `false`. Proposal is accepted.
5. `perform_mint_sns_tokens` calls `self.ledger.transfer_funds(u64::MAX, 0, None, to, memo)` — minting `u64::MAX` e8s of SNS tokens to the attacker's account.
6. Repeat immediately with another proposal. No 7-day cooldown is enforced.

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
