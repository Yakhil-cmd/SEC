### Title
Unbounded Synchronous Loop Over User-Controlled Buyers Collection in `create_sns_neuron_recipes` Can Permanently Block SNS Swap Finalization - (File: rs/sns/swap/src/swap.rs)

### Summary
`Swap::create_sns_neuron_recipes` iterates over the entire `self.buyers` map synchronously with no instruction-limit check or batching. Because any user can call `refresh_buyer_tokens` to insert themselves into `self.buyers`, a sufficiently large participant set causes the function to exhaust the IC instruction limit, permanently preventing the SNS swap from finalizing.

### Finding Description
`create_sns_neuron_recipes` contains two back-to-back synchronous `for` loops: one over `self.buyers` and one over `self.cf_participants`. For each buyer it calls `create_sns_neuron_basket_for_direct_participant`, which allocates a `Vec<SnsNeuronRecipe>` of `neuron_basket_construction_parameters.count` entries and extends `self.neuron_recipes`. [1](#0-0) 

There is no instruction-limit guard anywhere in this function. Compare this with `batch_claim_swap_neurons`, which explicitly batches work into chunks of `CLAIM_SWAP_NEURONS_BATCH_SIZE`, or with `sweep_icp`/`sweep_sns`, which reset the instruction counter at every `await` across a message boundary. [2](#0-1) 

`create_sns_neuron_recipes` is called synchronously (no `await`) from `finalize_inner`: [3](#0-2) 

The idempotency guard (`has_created_neuron_recipes == Some(true)`) does not help: on every retry the loop still traverses the entire `buyers` map from the start to find which entries to skip. If the map is large enough that even this traversal exceeds the 5-billion-instruction limit, no retry can ever succeed.

### Impact Explanation
The SNS swap canister is permanently stuck in `Lifecycle::Committed`. `finalize` is the only path to distribute ICP to the SNS governance canister, distribute SNS tokens to participants, and create neurons. If `create_sns_neuron_recipes` can never complete, none of those steps execute. Participants' ICP and SNS tokens remain locked in the swap canister indefinitely. The only escape is a canister upgrade that rewrites the finalization logic, which requires NNS governance action.

### Likelihood Explanation
The number of direct buyers is bounded by `max_direct_participation_icp_e8s / min_participant_icp_e8s`. With `min_participant_icp_e8s = 1 ICP` (10^8 e8s) and `max_direct_participation_icp_e8s = 100,000 ICP` (10^13 e8s), up to 100,000 distinct principals can participate. A single adversary with many funded accounts, or organic popularity of the SNS, can reach this count. At a neuron basket size of 3, that is 300,000 `SnsNeuronRecipe` allocations in one synchronous call — well within the range that exhausts the 5-billion-instruction limit. No privileged access is required; `refresh_buyer_tokens` is a public endpoint callable by any principal. [4](#0-3) 

### Recommendation
Apply the same batching pattern already used in `batch_claim_swap_neurons`: split `create_sns_neuron_recipes` into chunks, persist progress between calls using the existing `has_created_neuron_recipes` flag, and check the instruction counter (or use a fixed batch size) between iterations. Alternatively, enforce a hard cap on the number of direct participants at `refresh_buyer_tokens` time, sized so that a full `create_sns_neuron_recipes` run always fits within the instruction limit.

### Proof of Concept

1. Deploy an SNS swap with `min_participant_icp_e8s = 1e8` (1 ICP) and `max_direct_participation_icp_e8s = 1e13` (100,000 ICP).
2. Fund 50,000–100,000 distinct principals with 1 ICP each and have each call `refresh_buyer_tokens`. Each call inserts one entry into `self.buyers`.
3. Allow the swap to reach `Lifecycle::Committed` (minimum ICP threshold met).
4. Call `finalize`. The execution reaches `create_sns_neuron_recipes` at: [3](#0-2) 

5. The synchronous loop over `self.buyers` (line 839) allocates `count` `SnsNeuronRecipe` structs per buyer. With 50,000+ buyers and `count = 3`, instruction consumption exceeds 5 billion; the call is rejected by the replica with `InstructionLimitExceeded`.
6. The lock is released; `finalize` can be retried. On every retry the loop restarts from the beginning of `self.buyers`, re-traversing all already-processed entries. Because even the traversal alone exceeds the limit, no retry ever completes `create_sns_neuron_recipes`, and the swap is permanently stuck. [1](#0-0)
