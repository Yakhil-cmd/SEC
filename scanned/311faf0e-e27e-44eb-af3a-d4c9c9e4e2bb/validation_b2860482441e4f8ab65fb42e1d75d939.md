### Title
Gasless Ingress Execution via `UpdateSettings` Delayed-Charge Bypass — (`rs/cycles_account_manager/src/cycles_account_manager.rs`, `rs/execution_environment/src/execution_environment.rs`)

---

### Summary

The IC's `UpdateSettings` management-canister method deliberately defers ingress induction cost collection to allow frozen canisters to be unfrozen. For small payloads (≤ 338 bytes), `ingress_induction_cost()` returns `IngressInductionCost::Free`, bypassing all cycle checks at induction time. The deferred charge attempted during execution silently ignores `CanisterOutOfCyclesError`. The result is that any unprivileged ingress sender can submit unlimited `UpdateSettings` messages targeting any canister — including zero-cycle canisters — and have them inducted and executed at no cost, consuming subnet execution resources without payment.

---

### Finding Description

**Step 1 — Free induction classification.**

`ingress_induction_cost()` in `rs/cycles_account_manager/src/cycles_account_manager.rs` returns `IngressInductionCost::Free` (i.e., `paying_canister = None`) whenever the method is `UpdateSettings` and the payload is ≤ `MAX_DELAYED_INGRESS_COST_PAYLOAD_SIZE` (338 bytes): [1](#0-0) 

The constant is defined at: [2](#0-1) 

**Step 2 — Cycle check skipped at induction.**

In `valid_set_rule.rs`, the `enqueue()` function calls `ingress_induction_cost()` and only charges cycles when the result is `IngressInductionCost::Fee`. When the result is `Free`, the message is pushed directly into the input queue with no cycle withdrawal: [3](#0-2) 

The same `Free` branch is taken in the ingress selector's `validate_ingress()`, so the block-building cycle check is also skipped: [4](#0-3) 

**Step 3 — Cycle check skipped at the HTTP handler.**

`should_accept_ingress_message()` in `execution_environment.rs` also calls `ingress_induction_cost()` and only checks the canister balance when the result is `Fee`. For small `UpdateSettings` payloads the `Fee` branch is never entered: [5](#0-4) 

**Step 4 — Deferred charge silently ignored.**

After `update_settings` executes, the delayed charge is attempted via `consume_cycles()`. The return value is explicitly discarded with `let _ignore_error`: [6](#0-5) 

The comment acknowledges the error is "not actionable at this point." This means a canister with zero cycles will have the charge attempted and silently dropped — no cycles are ever collected.

**Step 5 — Intentional design, unintended consequence.**

The design intent is documented: [7](#0-6) 

The mechanism is correct for the unfreeze use case, but it creates an unrestricted free-execution path for any sender targeting any canister.

---

### Impact Explanation

An unprivileged ingress sender can:

1. Craft `UpdateSettings` messages with payloads ≤ 338 bytes targeting any canister (including zero-cycle canisters they control, or canisters they do not control — authorization is checked only during execution, not induction).
2. Submit these messages at the maximum ingress rate without any cycle cost.
3. Each message consumes: an ingress pool slot, a block payload slot (up to `max_ingress_messages_per_block`), an ingress history entry, and subnet execution instruction budget.

The net effect is a resource-metering bypass analogous to the Optimism M-11 finding: subnet execution resources are consumed without the sender paying cycles. A sustained flood of such messages degrades execution throughput for all other canisters on the subnet without any economic cost to the attacker beyond L1-equivalent transaction fees (IC ingress submission is free to end users).

---

### Likelihood Explanation

The attack path is fully reachable by any unprivileged ingress sender with no special access. The payload required is trivially constructable (a valid Candid-encoded `UpdateSettingsArgs` with only `canister_id` and a `freezing_threshold` field fits well within 338 bytes, as confirmed by the existing test). No threshold corruption, governance majority, or privileged key is required. [8](#0-7) 

---

### Recommendation

1. **Enforce a minimum cycle balance check at induction for `UpdateSettings` even when the charge is deferred.** The induction check and the actual charge can remain decoupled (charge after execution), but the check should still verify the canister has enough cycles to cover the induction cost before the message is admitted.

2. **Do not silently ignore the deferred charge error.** If the canister cannot pay after execution, the error should at minimum be logged at a warning level and counted in a metric so operators can detect abuse.

3. **Rate-limit free-induction management messages per sender or per target canister** to bound the worst-case resource consumption.

---

### Proof of Concept

```
# Attacker controls canister C with 0 cycles.
# Construct a small UpdateSettings payload (well under 338 bytes):
payload = Candid.encode(UpdateSettingsArgs {
    canister_id: C,
    settings: CanisterSettingsArgs { freezing_threshold: Some(1) },
    sender_canister_version: None,
})
# len(payload) << 338 bytes → ingress_induction_cost returns Free

# Submit at maximum ingress rate (no cycle cost):
loop:
    submit_ingress(to=IC_00, method="update_settings", payload=payload)
```

Each submitted message:
- Passes `should_accept_ingress_message` with no cycle check (Free branch).
- Is inducted by `valid_set_rule::enqueue` with no cycle withdrawal (Free branch).
- Is included in a block by the ingress selector with no cycle check (Free branch).
- Executes `update_settings`; the deferred `consume_cycles` call fails silently (`_ignore_error`).

Net cost to attacker: zero cycles. Net cost to subnet: instruction budget + ingress history memory + block payload space per message. [9](#0-8) [10](#0-9)

    pub fn ingress_induction_cost(
        &self,
        ingress: &SignedIngress,
        effective_canister_id: Option<CanisterId>,
        subnet_cycles_config: CyclesAccountManagerSubnetConfig,
    ) -> IngressInductionCost {
        let raw_bytes = NumBytes::from(ingress.binary().len() as u64);
        let ingress = ingress.content();
        let paying_canister = match ingress.is_addressed_to_subnet() {
            // If a subnet message, get effective canister id who will pay for the message.
            true => {
                if let Ok(Method::UpdateSettings) = Method::from_str(ingress.method_name()) {
                    // The fee for `UpdateSettings` with small payload is charged after
                    // applying the settings to allow users to unfreeze canisters
                    // after accidentally setting the freezing threshold too high.
                    if self.is_delayed_ingress_induction_cost(ingress.arg()) {
                        None
                    } else {
                        effective_canister_id
                    }
                } else {
                    effective_canister_id
                }
            }
            // A message to a canister is always paid for by the receiving canister.
            false => Some(ingress.canister_id()),
        };

        match paying_canister {
            Some(paying_canister) => {
                let cost = self.ingress_induction_cost_from_bytes(raw_bytes, subnet_cycles_config);
                IngressInductionCost::Fee {
                    payer: paying_canister,
                    cost: cost.real(),
                }
            }
            None => IngressInductionCost::Free,
        }
    }

fn max_delayed_ingress_cost_payload_size_test() {
    let default_freezing_limit = 30 * 24 * 3600; // 30 days
    let payload = UpdateSettingsArgs {
        canister_id: CanisterId::from_u64(0).into(),
        settings: CanisterSettingsArgsBuilder::new()

            .build(),
        sender_canister_version: None, // ingress messages are not supposed to set this field
    };

    let payload_size = 2 * Encode!(&payload).unwrap().len();
