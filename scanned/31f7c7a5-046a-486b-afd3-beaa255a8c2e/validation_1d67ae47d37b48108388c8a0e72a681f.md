### Title
NNS Governance Neuron Permanently Locked on Double Failure in `try_finalize_maturity_disbursement` - (File: `rs/nns/governance/src/governance/disburse_maturity.rs`)

---

### Summary

When `try_finalize_maturity_disbursement` encounters a double failure — ICP minting fails **and** the subsequent attempt to reverse the neuron state mutation also fails — the function explicitly calls `neuron_lock.retain()`, permanently inserting a `FinalizeDisburseMaturity` entry into `in_flight_commands`. There is no on-chain escape hatch to release this lock. The neuron is frozen indefinitely, blocking all operations (disburse, split, merge, configure) on the user's staked ICP until a canister upgrade with custom reconciliation code is deployed via NNS governance vote.

---

### Finding Description

`try_finalize_maturity_disbursement` in `rs/nns/governance/src/governance/disburse_maturity.rs` executes three sequential steps:

1. **Acquire neuron lock** — inserts a `FinalizeDisburseMaturity` entry into `heap_data.in_flight_commands`.
2. **Pop the maturity disbursement** from the neuron (first mutation).
3. **Call the ICP ledger** to mint ICP to the destination account.

If step 3 fails, the code attempts to reverse step 2 by pushing the disbursement back onto the neuron. If **that reversal also fails**, the code reaches the terminal error branch:

```rust
// Reaching this point means the neuron mutation was performed, the ledger operation failed
// and the neuron mutation could not be reversed. The best we can do at this point is to
// retain the neuron lock.
neuron_lock.retain();
Err(FinalizeMaturityDisbursementError::FailToRestoreMaturityDisbursement { ... })
```

`NeuronAsyncLock::drop()` checks `self.retain` and skips the `unlock_neuron` call, leaving the entry permanently in `in_flight_commands`. [1](#0-0) 

The `NeuronAsyncLock` drop implementation confirms the skip: [2](#0-1) 

The `get_delay_until_next_finalization` function explicitly acknowledges that a neuron can be "locked indefinitely" with no automatic resolution: [3](#0-2) 

The governance proto documents the only recovery path as custom code added on upgrade: [4](#0-3) 

---

### Impact Explanation

Once a neuron's ID is present in `in_flight_commands` with a retained `FinalizeDisburseMaturity` command, `lock_neuron_for_command` and `acquire_neuron_async_lock` both reject all subsequent operations on that neuron with `LedgerUpdateOngoing`: [5](#0-4) 

This blocks every user-facing neuron operation: `Disburse`, `Split`, `DisburseToNeuron`, `Merge`, `Configure`, `Spawn`, and `DisburseMaturity`. The user's staked ICP is frozen with no on-chain recovery path. The only resolution is a canister upgrade carrying custom reconciliation code, which requires an NNS governance proposal and vote — a privileged, time-delayed operation outside the user's control.

---

### Likelihood Explanation

The double-failure path requires:

1. **ICP ledger mint fails** — realistic when the ICP ledger is temporarily unavailable or rejects the call (e.g., cycles exhaustion, transient replica error).
2. **Neuron reversal (`with_neuron_mut`) fails** — requires the neuron to be absent from the neuron store at the moment of reversal, despite being locked. This is unlikely under normal operation but possible under neuron-store corruption, stable-memory exhaustion, or a bug in the neuron store index.

The code explicitly handles this branch and documents it, indicating the developers considered it a real (if rare) possibility. The `FailToRestoreMaturityDisbursement` variant is a named, exported error type with a dedicated display message: *"Neuron lock is retained."* [6](#0-5) 

Likelihood: **Low**, but non-zero and explicitly anticipated by the codebase.

---

### Recommendation

1. **Add an on-chain escape hatch**: Provide a governance-controlled (or time-bounded) method to clear a `FinalizeDisburseMaturity` entry from `in_flight_commands` when the disbursement record is confirmed absent from the neuron. This mirrors the recommendation in the external report to provide an `exitAwardPhase()` function.
2. **Idempotent reversal**: Make `push_front_maturity_disbursement_in_progress` infallible or log-and-continue rather than propagating an error that triggers lock retention, since the neuron is already locked and cannot be concurrently modified.
3. **Upgrade hook**: Add a post-upgrade hook that detects stale `FinalizeDisburseMaturity` entries in `in_flight_commands` and automatically reconciles them, rather than requiring bespoke upgrade code each time.

---

### Proof of Concept

1. An unprivileged user calls `disburse_maturity` on their neuron (valid ingress call). The neuron's `maturity_e8s_equivalent` is decremented and a `MaturityDisbursement` entry is appended.
2. After `DISBURSEMENT_DELAY_SECONDS` (7 days), the recurring timer task `FinalizeMaturityDisbursementsTask` fires and calls `try_finalize_maturity_disbursement`. [7](#0-6) 

3. The function acquires the neuron lock (step 1), pops the disbursement (step 2), and calls `mint_icp_with_ledger` (step 3). The ICP ledger returns an error (e.g., transient unavailability).
4. The reversal call `push_front_maturity_disbursement_in_progress` also fails (e.g., neuron store corruption).
5. `neuron_lock.retain()` is called at line 668. The `FinalizeDisburseMaturity` entry remains in `in_flight_commands` permanently.
6. All subsequent `manage_neuron` calls by the user return `LedgerUpdateOngoing`. The user's staked ICP is frozen indefinitely with no on-chain recovery path. [8](#0-7) [9](#0-8)

impl Drop for NeuronAsyncLock {
    fn drop(&mut self) {
        if self.retain {
            return;
        }
        // In the case of a panic, the state of the ledger account representing the neuron's stake
        // may be inconsistent with the internal state of governance.  In that case, we want to
        // prevent further operations with that neuron until the issue can be investigated and
        // resolved, which will require code changes.
        if ic_cdk::futures::is_recovering_from_trap() {
            return;
        }
        // The lock is released when the NeuronAsyncLock is dropped. This is done to ensure that the lock
        // is released even if the NeuronAsyncLock is not explicitly unlocked.
        self.governance.with_borrow_mut(|governance| {
            governance.unlock_neuron(self.neuron_id.id);
        });
    }
