### Title
Unprivileged caller can enumerate `rule_id`s and `incident_id`s of undisclosed rules via regular query call to `get_config`, bypassing `inspect_message` — (`rs/boundary_node/rate_limits/canister/canister.rs`)

### Summary
`get_config` is declared as `#[query]`, making it callable as a non-replicated query by any principal. The `inspect_message` hook that gates access to this method is only invoked for ingress (update) messages per the IC protocol — it is never invoked for query calls. The confidentiality formatter applied inside the handler redacts `rule_raw` and `description` for undisclosed rules but leaves `rule_id` and `incident_id` intact. Any anonymous caller can therefore enumerate all rule and incident UUIDs, including those for security incidents not yet publicly disclosed.

### Finding Description
The canister defines `REPLICATED_QUERY_METHOD = "get_config"` and uses `inspect_message` to reject callers who are neither the authorized principal nor a registered API boundary node:

```rust
if called_method == REPLICATED_QUERY_METHOD {
    if has_full_access || has_full_read_access {
        ic_cdk::api::call::accept_message();
    } else {
        ic_cdk::api::trap("...");
    }
}
``` [6](#0-5) 

However, `get_config` is annotated `#[query]`, not `#[update]`. On the IC, `inspect_message` is part of the **ingress message inspection** phase, which only runs for update (ingress) calls before they enter consensus. A regular query call to `get_config` never triggers `inspect_message` at all — the IC runtime routes it directly to the query handler.

Inside the handler, `ConfigGetter::get` applies `ConfigConfidentialityFormatter` for unauthorized callers, which only nulls out `description` and `rule_raw`:

```rust
config.rules.iter_mut().for_each(|rule| {
    if rule.disclosed_at.is_none() {
        rule.description = None;
        rule.rule_raw = None;
        // rule.id and rule.incident_id are NOT cleared
    }
});
``` [7](#0-6) 

The `OutputRule` struct always carries `rule_id` and `incident_id`, so the redacted response still leaks both identifiers for every undisclosed rule. [8](#0-7) 

### Impact Explanation
An anonymous principal can call `get_config` as a regular query and receive a response listing all `rule_id` and `incident_id` UUIDs for every rule in the config, including rules whose `disclosed_at` is `None` (i.e., active security incident rules not yet made public). This allows enumeration of the existence and count of undisclosed security incidents before operators choose to disclose them.

### Likelihood Explanation
The exploit requires no special access, no keys, and no privileged role. Any IC client library can issue a query call to `get_config` with an anonymous principal. The bypass is a direct consequence of how the IC protocol works (`inspect_message` ≠ query guard) combined with the formatter not redacting identifiers.

### Recommendation
Two independent fixes are needed:

1. **Add an explicit access check inside `get_config`** (and the other query handlers) that returns an error for callers with `AccessLevel::RestrictedRead` if the intent is to block them entirely — do not rely on `inspect_message` as the sole gate for query methods.
2. **Redact `rule_id` and `incident_id`** in `ConfigConfidentialityFormatter` for non-disclosed rules if those identifiers are considered sensitive, so that even callers who do reach the handler cannot enumerate undisclosed incident metadata. [9](#0-8) [10](#0-9) 

### Proof of Concept
Using `pocket_ic` or any IC agent:

```rust
// Anonymous principal, regular query call — inspect_message is never invoked
let response: GetConfigResponse = pocket_ic
    .query_call(
        canister_id,
        Principal::anonymous(),
        "get_config",
        encode_one(None::<u64>).unwrap(),
    )
    .unwrap();

let config = response.unwrap();
assert!(config.config.is_redacted);  // true — formatter ran
for rule in &config.config.rules {
    // rule_id and incident_id are present even for undisclosed rules
    assert!(!rule.rule_id.is_empty());
    assert!(!rule.incident_id.is_empty());
    // For undisclosed rules: rule_raw and description are None, but IDs leak
}
```

The test in `getter.rs` already documents this behavior for `AccessLevel::RestrictedRead` — `rule_id` and `incident_id` are present in the unauthorized response. [11](#0-10)

impl ConfidentialityFormatting for ConfigConfidentialityFormatter {
    type Input = OutputConfig;

    fn format(&self, config: OutputConfig) -> OutputConfig {
        let mut config = config;
        config.is_redacted = true;
        // Redact (hide) fields of non-disclosed rules
        config.rules.iter_mut().for_each(|rule| {
            if rule.disclosed_at.is_none() {
                rule.description = None;
                rule.rule_raw = None;
            }
        });
        config
    }
