### Title
Missing Proof of Possession for iDKG Dealing Encryption Key in `do_update_node_directly` - (File: `rs/registry/canister/src/mutations/do_update_node_directly.rs`)

---

### Summary

The registry canister's `do_update_node_directly` endpoint accepts an iDKG dealing encryption public key (`MEGaSecp256k1`) from a registered node without requiring any proof of possession (PoP). A registered node on a signing subnet can submit any valid secp256k1 curve point as its iDKG key — including a key for which it does not hold the corresponding private key — and the registry will accept and store it. This is the direct IC analog of the SSV front-running/ownership-bypass vulnerability class: a party registers a public key without proving ownership of the corresponding private key.

---

### Finding Description

The `update_node_directly` canister endpoint is callable by any registered node and is used to rotate the node's iDKG dealing encryption key: [1](#0-0) 

The handler derives the node identity from the IC caller principal and delegates to `do_update_node`: [2](#0-1) 

Step 5 of `do_update_node` validates the submitted iDKG key: [3](#0-2) 

The validation delegates to `ValidIDkgDealingEncryptionPublicKey::try_from`, which **only checks that the key is a valid point on the secp256k1 curve** — no proof of possession is required or checked: [4](#0-3) 

This is explicitly documented in the crate: [5](#0-4) 

By contrast, the committee signing key and the NI-DKG dealing encryption key both require a valid PoP: [6](#0-5) 

The iDKG key proto is explicitly serialized with `proof_data: None`: [7](#0-6) 

The same missing PoP applies to the iDKG key submitted in `do_add_node` via `valid_keys_from_payload`: [8](#0-7) 

---

### Impact Explanation

The iDKG dealing encryption key (`MEGaSecp256k1`) is the long-term encryption key used in the MEGa scheme for threshold ECDSA/Schnorr: dealers encrypt per-node secret shares to each receiver's iDKG public key, and receivers decrypt their shares using the corresponding private key to participate in signing.

If a registered node submits an iDKG public key for which it does not hold the private key:
1. Dealers encrypt threshold signing shares to the wrong key.
2. The node cannot decrypt its share and cannot contribute to signature reconstruction.
3. The node's slot is effectively dead for threshold signing purposes.

If multiple nodes on a signing subnet register iDKG keys they do not own, the subnet's effective honest-node count for threshold signing drops. Depending on subnet size and the threshold `f`, this can prevent the subnet from completing any threshold ECDSA/Schnorr signing requests, breaking chain-key functionality for all canisters relying on that subnet.

Additionally, two nodes could register the same iDKG public key (no uniqueness check is enforced), causing the MEGa scheme to encrypt the same share to the same key twice, which may have further protocol-level consequences.

---

### Likelihood Explanation

The attacker must be a registered node on a signing subnet — a legitimate but not fully trusted role (node operators are external entities). The attack is partially self-harming (the attacker's own node loses signing capacity), but a malicious node operator motivated to disrupt chain-key signing (e.g., to prevent a specific canister from obtaining a threshold signature) can deliberately register a wrong iDKG key. The endpoint is open to any registered node with no additional authorization: [9](#0-8) 

The rate-limiting and timing checks in `do_update_node` do not prevent submission of a key the node does not own — they only throttle how frequently updates occur: [10](#0-9) 

---

### Recommendation

Require a proof of possession for the iDKG dealing encryption key, analogous to what is already done for the committee signing key (`MultiBls12_381` PoP) and the NI-DKG dealing encryption key (`Groth20_Bls12_381` PoP). The MEGa scheme already has the cryptographic infrastructure for PoP (`compute_eph_key_and_pop` / `verify_pop` in `rs/crypto/internal/crypto_lib/threshold_sig/canister_threshold_sig/src/idkg/mega.rs`). The registry canister should:

1. Require a PoP field alongside the iDKG public key in `UpdateNodeDirectlyPayload` and `AddNodePayload`.
2. Verify the PoP in `ValidIDkgDealingEncryptionPublicKey::try_from` before accepting the key.
3. Reject any iDKG key update that does not include a valid PoP proving the submitter holds the corresponding private key.

---

### Proof of Concept

1. Node A (registered on a signing subnet) observes Node B's iDKG public key in the registry.
2. Node A calls `update_node_directly` on the registry canister, submitting Node B's iDKG public key bytes as its own `idkg_dealing_encryption_pk`.
3. `ValidIDkgDealingEncryptionPublicKey::try_from` accepts the key (it is a valid secp256k1 point).
4. The registry stores Node B's iDKG public key as Node A's iDKG key.
5. In subsequent threshold signing rounds, dealers encrypt Node A's share to Node B's key.
6. Node A cannot decrypt its share; Node A's slot is dead for threshold signing.
7. If the subnet is near the fault threshold, signing requests begin to fail.

The test `test_idkg_key_update_success` in `do_update_node_directly.rs` demonstrates that two different nodes can successfully register the same iDKG key bytes with no error, confirming there is no ownership or uniqueness enforcement: [11](#0-10)

/// Validates a node's interactive DKG dealing encryption key.
///
/// See the crate documentation for the exact checks that are performed.
impl TryFrom<PublicKey> for ValidIDkgDealingEncryptionPublicKey {
    type Error = KeyValidationError;

    fn try_from(public_key: PublicKey) -> Result<Self, Self::Error> {
        let curve_type = match AlgorithmIdProto::try_from(public_key.algorithm).ok() {
            Some(AlgorithmIdProto::MegaSecp256k1) => Ok(EccCurveType::K256),
            alg_id => Err(invalid_idkg_dealing_enc_pubkey_error(format!(
                "unsupported algorithm: {alg_id:?}"
            ))),
        }?;
        // `verify_mega_public_key` also ensures that the public key is a valid point on the curve.
        verify_mega_public_key(curve_type, &public_key.key_value).map_err(|e| {
            invalid_idkg_dealing_enc_pubkey_error(format!("verification failed: {e:?}"))
        })?;
        Ok(Self { public_key })
    }
}
