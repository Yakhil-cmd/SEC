### Title
Hardcoded `E8` Divisor in SNS Treasury Valuation Ignores Actual Token Decimals, Bypassing Spending Limits - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

### Summary

`try_get_balance_valuation_factors` in the SNS governance token valuation module always divides the raw ICRC-1 balance by the hardcoded constant `E8` (10^8) to convert it to whole tokens. This is only correct for tokens with exactly 8 decimal places. For SNS tokens with fewer decimal places (e.g., 6), the treasury XDR valuation is systematically understated, causing the treasury to appear in a lower spending regime than it actually is. This allows a malicious SNS governance majority to bypass the 300,000 XDR per-7-day spending cap and drain the entire treasury.

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, the function `try_get_balance_valuation_factors` converts the raw `icrc1_balance_of` response to a whole-token count by dividing by the hardcoded constant `E8 = 100_000_000`:

```rust
let tokens = Decimal::from(u128::try_from(balance_of_response.0)...) / Decimal::from(E8);
``` [1](#0-0) 

The ICRC-1 standard allows any number of decimal places. The SNS ledger initializes with `decimals: decimals.unwrap_or_else(default_decimals)`, where `decimals` is a configurable field: [2](#0-1) 

No validation in the SNS initialization enforces exactly 8 decimal places. If an SNS token has 6 decimal places, `icrc1_balance_of` returns amounts in units of 10^-6, but the code divides by 10^8, producing a token count 100× smaller than the true whole-token count.

The `icps_per_token` value from the swap's `get_derived_state` (`sns_tokens_per_icp`) is expressed in whole tokens and is unaffected by this bug. Therefore the XDR valuation `tokens × icps_per_token × xdrs_per_icp` is 100× smaller than the true value. [3](#0-2) 

The XDR valuation determines which spending regime applies: [4](#0-3) 

- **Small** (< 100,000 XDR): `NoLimit` — the entire balance can be transferred.
- **Medium** (100,000–1,200,000 XDR): `Fraction(0.25)` — at most 25% per 7 days.
- **Large** (> 1,200,000 XDR): `Xdr(300,000)` — at most 300,000 XDR per 7 days.

The proposal amount is also converted using the same hardcoded `E8`: [5](#0-4) [6](#0-5) 

**Concrete exploit scenario** (6-decimal SNS token, 1 token = 10 XDR):

| | Correct | With Bug (6 decimals) |
|---|---|---|
| Treasury balance (raw) | 100,000,000,000 | 100,000,000,000 |
| Whole tokens | 100,000 | 1,000 (÷10^8 instead of ÷10^6) |
| XDR valuation | 1,000,000 XDR | 10,000 XDR |
| Regime | Large → max 300,000 XDR | Small → NoLimit |
| Limit (whole tokens) | 30,000 | 1,000 (full balance in wrong units) |
| Proposal for 100,000 tokens (amount_e8s=10^11) | Rejected (exceeds 30,000) | Allowed (1,000 ≤ 1,000) |

The proposal amount `10^11 / 10^8 = 1,000` equals the (wrong) limit `1,000`, so the check passes and the full treasury is transferred.

The same valuation path is used for `MintSnsTokens` proposals: [7](#0-6) 

### Impact Explanation

A malicious SNS governance majority controlling an SNS whose native token has fewer than 8 decimal places can submit a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal that exceeds the intended 300,000 XDR per-7-day cap. The treasury valuation is understated, placing the treasury in the `NoLimit` or `Fraction` regime instead of the `Xdr` regime. The entire SNS treasury (potentially millions of XDR in value) can be drained in a single proposal, defeating the safety mechanism designed to protect against malicious governance majorities.

### Likelihood Explanation

Creating an SNS requires NNS governance approval, which provides a partial gate. However, the NNS proposal review process does not explicitly validate that the SNS token uses exactly 8 decimal places. The ICRC-1 standard explicitly supports configurable decimals, and the SNS ledger initialization accepts any value. Once an SNS with a non-8-decimal token is deployed and its treasury grows above 1,200,000 XDR, any governance majority can exploit this to bypass the spending cap. The likelihood is **medium**: it requires SNS creation with a non-standard decimal count and sufficient treasury accumulation, but no privileged key or threshold corruption is needed.

### Recommendation

In `try_get_balance_valuation_factors`, query the ledger's `icrc1_decimals` endpoint and use the actual decimal count as the divisor instead of the hardcoded `E8`:

```rust
// Fetch actual decimals from the ledger
let decimals: u8 = icrc1_client.icrc1_decimals().await?;
let divisor = Decimal::from(10_u128.pow(decimals as u32));
let tokens = Decimal::from(u128::try_from(balance_of_response.0)?) / divisor;
```

Apply the same fix to `denominations_to_tokens` calls in `rs/sns/governance/src/proposal.rs` (lines 840 and 2632) and `tokens_to_e8s` in `rs/sns/governance/src/treasury.rs` (line 55), replacing `E8` with the actual token denomination.

Alternatively, enforce at SNS initialization time that the SNS token must have exactly 8 decimal places, and document this as a protocol requirement.

### Proof of Concept

1. Create an SNS with a token configured to 6 decimal places (`decimals: Some(6)` in the ICRC-1 ledger init args).
2. Accumulate a treasury worth > 1,200,000 XDR (e.g., 200,000 whole tokens at 10 XDR each = 2,000,000 XDR).
3. Submit a `TransferSnsTreasuryFunds` proposal with `amount_e8s = 200_000 * 10^6` (full balance in 6-decimal units).
4. The valuation check computes: `balance_tokens = 200_000_000_000 / 10^8 = 2,000` and `XDR = 2,000 × 10 = 20,000` → `NoLimit` regime → limit = `2,000`.
5. The proposal amount computes: `200_000_000_000 / 10^8 = 2,000`.
6. Check: `2,000 > 2,000` → false → proposal is accepted.
7. The full treasury (worth 2,000,000 XDR) is transferred, bypassing the 300,000 XDR cap. [8](#0-7) [9](#0-8)

async fn try_get_balance_valuation_factors(
    account: Account,
    icrc1_client: &mut dyn Icrc1Client,
    icps_per_token_client: &mut dyn IcpsPerTokenClient,
    xdrs_per_icp_client: &mut dyn XdrsPerIcpClient,
) -> Result<ValuationFactors, ValuationError> {
    // Fetch the three ingredients:
    //
    //     1. balance
    //     2. token -> ICP
    //     3. ICP -> XDR
    //
    // No await here. Instead, we use join (right after this).
    let balance_of_request = icrc1_client.icrc1_balance_of(account);
    let icps_per_token_request = icps_per_token_client.get();
    let xdrs_per_icp_request = xdrs_per_icp_client.get();

    // Make all (3) requests (concurrently).
    let (balance_of_response, icps_per_token_response, xdrs_per_icp_response) = join!(
        balance_of_request,
        icps_per_token_request,
        xdrs_per_icp_request,
    );

    // Unwrap/forward errors to the caller.
    let balance_of_response = balance_of_response.map_err(|err| {
        ValuationError::new_external(format!("Unable to obtain balance from ledger: {err:?}"))
    })?;
    let icps_per_token_response = icps_per_token_response.map_err(|err| {
        ValuationError::new_external(format!("Unable to determine ICPs per token: {err:?}"))
    })?;
    let xdrs_per_icp_response = xdrs_per_icp_response.map_err(|err| {
        ValuationError::new_external(format!("Unable to obtain XDR per ICP: {err:?}"))
    })?;

    // Extract and interpret the data we actually care about from the (Ok) responses.
    let tokens = Decimal::from(u128::try_from(balance_of_response.0).map_err(|err| {
        ValuationError::new_arithmetic(format!(
            "Balance of {account:?} does not fit in u128: {err:?}"
        ))
    })?) / Decimal::from(E8);
    let icps_per_token = icps_per_token_response;
    let xdrs_per_icp = xdrs_per_icp_response;

    // Compose the fetched/interpretted data (i.e. multiply them) to construct the final result.
    Ok(ValuationFactors {
        tokens,
        icps_per_token,
        xdrs_per_icp,
    })
}

    fn in_tokens(mut valuation: Valuation) -> Result<Decimal, ProposalsAmountTotalLimitError> {
        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))
                })?;

                // Calculate the inverse conversion rate.
                if xdrs_per_token == Decimal::from(0) {
                    // This is not reachable, because in this case, valuation.to_xdr() would return
                    // 0, and in that case, we would have taken the NoLimit branch.
                    return Err(ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
                }
                let tokens_per_xdr = xdrs_per_token.inv();

                max_xdr.checked_mul(tokens_per_xdr).ok_or_else(|| {

                        "Max tokens could not be calculated with valuation: {valuation:?}",
                    ))
                })?
