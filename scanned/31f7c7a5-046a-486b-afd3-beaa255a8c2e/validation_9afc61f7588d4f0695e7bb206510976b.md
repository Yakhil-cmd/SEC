### Title
ICP Sent to SNS Swap Subaccount During PENDING/ADOPTED Lifecycle Is Temporarily Locked With No Refund Path — (`File: rs/sns/swap/src/swap.rs`)

---

### Summary

The SNS Swap canister accepts ICP transfers to per-user subaccounts at **any** lifecycle state via the ICP ledger (an external, permissionless entry point). However, the only user-facing refund path — `error_refund_icp` — is gated to `Lifecycle::Aborted || Lifecycle::Committed`, and the participation path — `refresh_buyer_token_e8s` — is gated to `Lifecycle::Open`. A user who sends ICP to the swap canister's subaccount while the swap is in `PENDING` or `ADOPTED` state has no callable method to recover those funds until the swap eventually closes. This is the direct IC analog of the reported "liquidity can still be supplied when the protocol is paused" pattern.

---

### Finding Description

The SNS Swap canister defines a strict lifecycle:

```
PENDING → ADOPTED → OPEN → COMMITTED / ABORTED
``` [1](#0-0) 

The intended participation flow requires the swap to be in `OPEN` state. `refresh_buyer_token_e8s` enforces this with two lifecycle checks (before and after the async ledger call): [2](#0-1) 

The ticket creation path (`new_sale_ticket`) also enforces `Lifecycle::Open`: [3](#0-2) 

However, the ICP ledger is a **permissionless external canister**. Any user can call `icrc1_transfer` on the ICP ledger to send ICP to `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&buyer)) }` at **any time**, regardless of the swap's lifecycle state. This is the alternative entry point that bypasses the lifecycle guard.

The only recovery mechanism for such "erroneous" transfers is `error_refund_icp`, which is hard-gated: [4](#0-3) 

This means:

| Lifecycle State | Can participate via `refresh_buyer_token_e8s`? | Can recover via `error_refund_icp`? |
|---|---|---|
| `PENDING` | No | **No** |
| `ADOPTED` | No | **No** |
| `OPEN` | Yes | No |
| `COMMITTED` | No | Yes |
| `ABORTED` | No | Yes |

A user who sends ICP to the swap canister's subaccount while the swap is in `PENDING` or `ADOPTED` state has **no callable method** to recover those funds until the swap eventually transitions to `COMMITTED` or `ABORTED`. The canister entry point `error_refund_icp` is exposed publicly: [5](#0-4) 

The documentation in the swap code itself acknowledges this gap:

> "If a ledger transfer was successfully made, but this call fails (many reasons are possible), the owner of the ICP sent to the subaccount can reclaim their tokens using `error_refund_icp` **once this swap is closed** (committed or aborted)." [6](#0-5) 

---

### Impact Explanation

A user who sends ICP to the swap canister's subaccount during `PENDING` or `ADOPTED` state has their funds **locked with no recovery path** until the swap closes. In the legacy (non-one-proposal) flow, the swap can remain in `PENDING` indefinitely if NNS governance never calls `open`. Even in the one-proposal flow, the `ADOPTED` delay period (specified by `init.swap_start_timestamp_seconds`) can be substantial. During this window, the user's ICP is inaccessible. This is a **ledger conservation / user fund lock** impact: real ICP is committed to the swap canister's subaccount with no mechanism to retrieve it until a future state transition occurs.

---

### Likelihood Explanation

Likelihood is **low but non-zero**. Users may:
- Follow outdated documentation or tooling that instructs them to transfer ICP before the swap opens
- Mistakenly send ICP to the swap canister's subaccount before the swap is open (e.g., confusing the swap start time)
- Be directed by a malicious frontend to send ICP early

The entry path is fully unprivileged: any ICP ledger user can trigger it with a standard `icrc1_transfer` call. No special role or key is required.

---

### Recommendation

Add a lifecycle-aware refund path that allows users to recover ICP sent to the swap canister's subaccount while the swap is in `PENDING` or `ADOPTED` state. Concretely, `error_refund_icp` should be extended to also permit refunds when `lifecycle().is_before_open()` is true, provided the caller's subaccount balance is non-zero and the funds have not been registered via `refresh_buyer_token_e8s`. Alternatively, document clearly that ICP sent before the swap opens is locked until close, and ensure frontends prevent premature transfers.

---

### Proof of Concept

1. An SNS swap is deployed and is in `LIFECYCLE_PENDING` or `LIFECYCLE_ADOPTED` state.
2. An unprivileged user calls `icrc1_transfer` on the ICP ledger, sending ICP to `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&user_principal)) }`.
3. The user attempts to call `refresh_buyer_tokens` on the swap canister → rejected: `"Participation is possible only when the Swap is in the OPEN state."` [7](#0-6) 
4. The user attempts to call `error_refund_icp` on the swap canister → rejected: `"Error refunds can only be performed when the swap is ABORTED or COMMITTED"` [8](#0-7) 
5. The user's ICP remains locked in the swap canister's subaccount with no callable recovery path until the swap eventually reaches `COMMITTED` or `ABORTED`. [9](#0-8)

enum Lifecycle {
  // The canister is incorrectly configured. Not a real lifecycle state.
  LIFECYCLE_UNSPECIFIED = 0;
  // In PENDING state, the canister is correctly initialized. Once SNS
  // tokens have been transferred to the swap canister's account on
  // the SNS ledger, a call to `open` with valid parameters will start
  // the swap.
  LIFECYCLE_PENDING = 1;
  // In ADOPTED state, the proposal to start the decentralization swap
  // has been adopted, and the swap will be automatically opened after a delay.
  // In the legacy (non-one-proposal) flow, the swap delay is specified by
  // params.sale_delay_seconds. In the one-proposal flow, the swap delay is
  // specified by `init.swap_start_timestamp_seconds`.
  LIFECYCLE_ADOPTED = 5;
  // In OPEN state, prospective buyers can register for the token
  // swap. The swap will be committed when the target (max) ICP has
  // been reached or the swap's due date/time occurs, whichever
  // happens first.
  LIFECYCLE_OPEN = 2;
  // In COMMITTED state the token price has been determined; on a call to
  // finalize`, buyers receive their SNS neurons and the SNS governance canister
  // receives the ICP.
  LIFECYCLE_COMMITTED = 3;
  // In ABORTED state the token swap has been aborted, e.g., because the due
  // date/time occurred before the minimum (reserve) amount of ICP has been
  // retrieved. On a call to `finalize`, participants get their ICP refunded.
  LIFECYCLE_ABORTED = 4;
}
