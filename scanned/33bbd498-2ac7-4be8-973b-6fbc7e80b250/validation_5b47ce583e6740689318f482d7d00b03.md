### Title
Read-Only Reentrancy: `get_neuron` Query Observes Inconsistent Intermediate State During `disburse_neuron` Async Operation - (File: rs/sns/governance/canister/canister.rs)

---

### Summary

The SNS Governance canister's `disburse_neuron` update method acquires a neuron lock to prevent concurrent update operations. However, the lock only guards against concurrent *update* calls — it does not prevent `#[query]` calls from executing against the committed intermediate state that exists at each `await` point during the async ledger transfers. The `get_neuron` query function reads `cached_neuron_stake_e8s` directly from canister state without checking whether the neuron lock is held, allowing any caller to observe an inflated or otherwise inconsistent neuron stake during the disburse operation.

---

### Finding Description

`disburse_neuron` in SNS Governance is an async update method that performs two sequential ledger inter-canister calls:

1. **Burn fees** — `transfer_funds(fees_amount_e8s, 0, ...)` → `.await?`
2. **Disburse stake** — `transfer_funds(disburse_amount_e8s, transaction_fee_e8s, ...)` → `.await?`

The neuron lock is acquired at the start via `lock_neuron_for_command`, which inserts an entry into `proto.in_flight_commands`. This prevents concurrent *update* calls from modifying the same neuron. However, at each `await` point, the IC runtime commits the current state and suspends execution. The committed intermediate state is then visible to any subsequent message — including query calls. [1](#0-0) 

The `get_neuron` canister endpoint is a `#[query]` function: [2](#0-1) 

It reads `proto.neurons` directly: [3](#0-2) 

There is no check for whether the neuron's ID is present in `proto.in_flight_commands`. The lock is only enforced for update-path callers.

The two intermediate states that are observable via `get_neuron` during `disburse_neuron`:

- **Between await 1 and await 2**: fees have been burned on the ledger, but `cached_neuron_stake_e8s` has been reduced only by `fees_amount_e8s`. The disburse amount has not yet been deducted. The neuron appears to have more stake than its actual ledger balance. [4](#0-3) 

- **Before await 1**: `cached_neuron_stake_e8s` still reflects the full pre-disburse stake even though a burn is in flight.

The same pattern exists in NNS Governance's `disburse_neuron`, where `cached_neuron_stake_e8s` is updated only *after* each ledger response: [5](#0-4) 

Additionally, during NNS `spawn_neuron`, `maturity_e8s_equivalent` is zeroed and `cached_neuron_stake_e8s` is set to the new stake value *before* the ledger mint call, creating a window where the neuron shows stake that does not yet exist on the ledger. [6](#0-5) 

The neuron lock mechanism is documented as protecting only against concurrent update interleaving: [7](#0-6) 

---

### Impact Explanation

Any external canister or protocol that calls `get_neuron` (a query) during the window between ledger await points will observe a `cached_neuron_stake_e8s` that does not match the actual on-chain ledger balance. Concretely:

- An external DeFi protocol that uses `get_neuron` to determine collateral value for a loan against a neuron's stake could be deceived into accepting a neuron as collateral at its pre-disburse stake value, even though the disburse is already in flight and the ledger balance is being reduced.
- The SNS governance canister itself calls `get_derived_state` on the Swap canister (a query) to determine treasury ICP amounts for valuation; an analogous pattern exists where governance reads query state that may be mid-finalization. [8](#0-7) 

The `get_neuron` query is publicly accessible to any unprivileged caller and is listed in the SNS governance DID as a `query` method: [9](#0-8) 

---

### Likelihood Explanation

The window of exposure is the round-trip latency of an inter-canister ledger call (typically one to a few IC consensus rounds, on the order of seconds). An attacker who initiates a `disburse_neuron` call and simultaneously (or via a coordinated canister) issues a `get_neuron` query can reliably observe the intermediate state. The IC execution model guarantees that state committed before an `await` is visible to subsequent messages, including queries, making this timing deterministic rather than probabilistic. The attack requires no privileged access — any principal can call `get_neuron`.

---

### Recommendation

Introduce a read-only reentrancy guard for `get_neuron` (and similar query endpoints) that checks whether the target neuron ID is present in `proto.in_flight_commands` and returns an appropriate error or a flag indicating that the neuron is mid-operation. This mirrors the recommended fix in the original report: a modifier that only *checks* the reentrancy guard without setting it.

```rust
// In get_neuron:
if self.proto.in_flight_commands.contains_key(&nid.to_string()) {
    return GetNeuronResponse {
        result: Some(get_neuron_response::Result::Error(
            GovernanceError::new_with_message(
                ErrorType::LedgerUpdateOngoing,
                "Neuron has an ongoing ledger update; state may be inconsistent.",
            )
        )),
    };
}
```

Alternatively, expose the `in_flight_commands` status as a field in the `GetNeuronResponse` so callers can detect and handle the intermediate state.

---

### Proof of Concept

1. Attacker (or attacker-controlled canister) calls `manage_neuron` → `Disburse` on a dissolved SNS neuron with a significant stake.
2. Simultaneously, attacker calls `get_neuron` (query) for the same neuron ID.
3. If the query executes between the first and second ledger `transfer_funds` await points, the returned `cached_neuron_stake_e8s` equals `original_stake - fees_amount`, while the actual ledger balance is `original_stake - fees_amount - disburse_amount - tx_fee`.
4. An external lending protocol that reads `get_neuron` to determine collateral value would see an inflated stake and could issue a loan against value that no longer exists on the ledger.

The neuron lock entry in `proto.in_flight_commands` is visible in state but is not checked by `get_neuron`: [10](#0-9) [11](#0-10)

```rust
        &mut self,
        nid: &NeuronId,
        command: NeuronInFlightCommand,
    ) -> Result<LedgerUpdateLock, GovernanceError> {
        let nid = nid.to_string();
        if self.proto.in_flight_commands.contains_key(&nid) {
            return Err(GovernanceError::new_with_message(
                ErrorType::NeuronLocked,
                "Neuron has an ongoing operation.",
            ));
        }

        self.proto.in_flight_commands.insert(nid.clone(), command);

        Ok(LedgerUpdateLock { nid, gov: self })
```

```rust
//! This module defines mechanisms for locking neurons in order to prevent problematic interleaving
//! of neuron operations.
//!
//! The `LedgerUpdateLock` is a legacy mechanism, where the lock contains a `*mut Governance`
//! pointer. An unsafe block is needed to unlock the neuron. In addition, the pointer needs to be
//! `'static` in order for the lock to be used in async contexts. However, using `&'static mut` to
//! access global state is dangerous and should be avoided.
//!
//! The `NeuronAsyncLock` is a new mechanism that uses a `&'static LocalKey<RefCell<Governance>>` to
//! access the global state. This allows for safe access to the global state in async contexts.
//!
//! For sync methods, there is actually no need to acquire the lock, since it's impossible for the
//! lock to be persisted in any case anyway. In the future, a new method on the `Governance` struct
//! can be used to check whether a lock is held for a neuron. However, currently, in order to avoid
//! introducing a 3rd pattern for locking neurons, the recommendation is to keep using
//! `lock_neuron_for_command` with a `SyncCommand`.
//!
//! Note that it's OK for `NeuronAsyncLock` and `LedgerUpdateLock` to co-exist. If a
//! `NeuronAsyncLock` is held for a neuron, and another method tries to acquire a `LedgerUpdateLock`
//! for the same neuron, it will still fail as expected, and vice versa, since their underlying
//! storage is the same `in_flight_commands` map.
```
