### Title
Unbounded `transfer_fee` in `ManageLedgerParameters` Proposal Allows SNS Governance to Freeze or Spam the SNS Ledger - (File: `rs/sns/governance/src/proposal.rs`)

### Summary

The `ManageLedgerParameters` SNS governance proposal action accepts any `transfer_fee` value (including `0` or `u64::MAX`) without bounds validation. A governance majority can pass such a proposal to either freeze all SNS token transfers (fee = `u64::MAX`) or enable fee-free spam (fee = `0`). Additionally, after execution the governance canister silently writes the new fee into `NervousSystemParameters.transaction_fee_e8s` without re-validating the invariant `neuron_minimum_stake_e8s > transaction_fee_e8s`, breaking downstream neuron arithmetic.

### Finding Description

`validate_and_render_manage_ledger_parameters` in `rs/sns/governance/src/proposal.rs` is the sole validation gate for `ManageLedgerParameters` proposals. When `transfer_fee` is `Some(value)`, the function only renders a human-readable string and sets a `change` flag — it performs **no range check** on the value: [1](#0-0) 

Compare this to `token_name`, `token_symbol`, and `token_logo`, which each call dedicated `ledger_validation::validate_*` helpers. No analogous validator exists for `transfer_fee`.

`perform_manage_ledger_parameters` in `rs/sns/governance/src/governance.rs` then upgrades the ledger canister with the raw fee value and, on success, unconditionally overwrites `NervousSystemParameters.transaction_fee_e8s`: [2](#0-1) 

`NervousSystemParameters.validate()` enforces `neuron_minimum_stake_e8s > transaction_fee_e8s`: [3](#0-2) 

But this validation is **never called** after the fee update in `perform_manage_ledger_parameters`. If `transfer_fee` is set to a value ≥ `neuron_minimum_stake_e8s`, the invariant is silently violated in stored state.

Downstream code in `split_neuron` and `disburse_to_neuron` computes `min_stake + transaction_fee_e8s` without overflow protection: [4](#0-3) 

If `transaction_fee_e8s = u64::MAX`, this addition wraps in release mode, producing `min_stake - 1`, corrupting the minimum-stake guard.

### Impact Explanation

**Scenario A — Ledger freeze (`transfer_fee = u64::MAX`):** Every ICRC-1 transfer on the SNS ledger requires paying the fee. With fee = `u64::MAX`, no account can afford a transfer. Token transfers, neuron staking, neuron disbursement, and all token-based SNS operations become permanently impossible until a recovery proposal is passed. The SNS token is effectively bricked.

**Scenario B — Fee-free spam (`transfer_fee = 0`):** All transfers become free. Adversaries can flood the ledger with zero-cost transactions, exhausting ledger memory and cycles, and disrupting normal SNS operation.

**Scenario C — Invariant violation (`transfer_fee ≥ neuron_minimum_stake_e8s`):** The `transaction_fee_e8s` stored in `NervousSystemParameters` exceeds `neuron_minimum_stake_e8s`. The `min_stake + transaction_fee_e8s` addition in `split_neuron` wraps to a small value, bypassing the minimum-stake guard and allowing neurons to be split into dust.

### Likelihood Explanation

Exploiting this requires an SNS governance majority to pass a `ManageLedgerParameters` proposal with a malicious `transfer_fee`. This is the same trust model as the external report (governance/Council ownership). Likelihood is low in practice but non-zero: a compromised SNS governance key, a social-engineering attack on token holders, or a governance capture attack could trigger it. The root cause — the absence of bounds checking — is an independent code defect regardless of governance intent.

### Recommendation

Add explicit bounds validation for `transfer_fee` inside `validate_and_render_manage_ledger_parameters`:

- Reject `transfer_fee = 0` (or enforce a protocol-defined minimum).
- Reject `transfer_fee ≥ neuron_minimum_stake_e8s` (read the current value from `NervousSystemParameters` at proposal creation time and re-validate at execution time, analogously to how NNS `validate_manage_network_economics` calls `apply_changes_and_validate` at both creation and execution).

After `perform_manage_ledger_parameters` updates `transaction_fee_e8s`, call `NervousSystemParameters::validate()` and fail the proposal if the result is invalid.

### Proof of Concept

1. An SNS governance majority submits:
   ```
   ManageLedgerParameters { transfer_fee: Some(u64::MAX), .. }
   ```
2. `validate_and_render_manage_ledger_parameters` accepts it — the only check is `change == true`. [5](#0-4) 
3. The proposal passes voting and `perform_manage_ledger_parameters` upgrades the ledger with `transfer_fee = u64::MAX`. [6](#0-5) 
4. `nervous_system_parameters.transaction_fee_e8s` is set to `u64::MAX` with no re-validation. [2](#0-1) 
5. Every subsequent ICRC-1 transfer on the SNS ledger is rejected (fee unaffordable). Every `split_neuron` call computes `min_stake + u64::MAX`, which wraps to `min_stake - 1` in release mode, corrupting the minimum-stake guard. [4](#0-3)
