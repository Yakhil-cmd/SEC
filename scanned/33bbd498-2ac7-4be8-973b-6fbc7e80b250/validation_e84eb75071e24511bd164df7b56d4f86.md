### Title
SNS `ManageNervousSystemParameters` Can Arbitrarily Increase `neuron_minimum_dissolve_delay_to_vote_seconds`, Retroactively Disenfranchising Existing Neuron Holders - (`File: rs/sns/governance/src/types.rs`)

---

### Summary

In the SNS governance canister, the `neuron_minimum_dissolve_delay_to_vote_seconds` parameter inside `NervousSystemParameters` can be raised to any value up to `max_dissolve_delay_seconds` via a `ManageNervousSystemParameters` proposal. There is no protection preventing the new value from exceeding the dissolve delay of neurons that were already eligible to vote under the previous parameter. A neuron holder who locked tokens for a dissolve delay that satisfied the old threshold can be instantly stripped of voting eligibility — and therefore of voting rewards — by a governance majority passing a single parameter-change proposal.

---

### Finding Description

`NervousSystemParameters.neuron_minimum_dissolve_delay_to_vote_seconds` is the sole gate for voting eligibility in SNS governance. When a new proposal is created, `compute_ballots_for_new_proposal` iterates all neurons and excludes any whose current dissolve delay is below this threshold:

```rust
if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
    // Not eligible due to dissolve delay.
    continue;
}
``` [1](#0-0) 

The only validation applied when this parameter is changed is that the new value must not exceed `max_dissolve_delay_seconds`:

```rust
fn validate_neuron_minimum_dissolve_delay_to_vote_seconds(&self) -> Result<(), String> {
    let max_dissolve_delay_seconds = self.validate_max_dissolve_delay_seconds()?;
    ...
    if neuron_minimum_dissolve_delay_to_vote_seconds > max_dissolve_delay_seconds {
        Err(...)
    } else {
        Ok(())
    }
}
``` [2](#0-1) 

There is no check that the new threshold does not exceed the dissolve delay of existing neurons, and no check that it does not exceed the dissolve delay of the proposing neuron itself at the time of execution. The execution path in `perform_manage_nervous_system_parameters` simply calls `validate()` on the merged parameters and writes them:

```rust
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            self.proto.parameters = Some(new_params);
            Ok(())
        }
        ...
    }
}
``` [3](#0-2) 

The `ManageNervousSystemParameters` action is a standard SNS proposal action: [4](#0-3) 

Any neuron holder with sufficient voting power to pass a proposal can submit and pass a `ManageNervousSystemParameters` proposal that raises `neuron_minimum_dissolve_delay_to_vote_seconds` to any value up to `max_dissolve_delay_seconds` (default 8 years). Neurons that previously satisfied the old threshold but fall below the new one are immediately excluded from all future electoral rolls — losing both voting rights and voting rewards — with no recourse, since dissolve delay can only be increased, not decreased.

---

### Impact Explanation

**Governance authorization bug / governance disenfranchisement.**

A majority coalition (or a single neuron with majority stake) can pass a `ManageNervousSystemParameters` proposal that raises `neuron_minimum_dissolve_delay_to_vote_seconds` to a value higher than the dissolve delay of a targeted minority of neurons. Those neurons are immediately and permanently excluded from voting on all future proposals until they increase their own dissolve delay. Since dissolve delay can only be increased (not decreased), affected neuron holders must lock their tokens for an even longer period than they originally agreed to — or permanently lose their governance participation and associated voting rewards. This is a direct analog to the rage-quit timestamp reduction described in the external report: a privileged role (governance majority) can unilaterally revoke a time-based guarantee (voting eligibility for a given dissolve delay) that token holders relied upon when staking.

---

### Likelihood Explanation

The attack path is fully reachable by any unprivileged ingress sender who controls enough SNS voting power to pass a proposal. In many SNS deployments, founding teams or early investors hold majority stake. The proposal type is a standard, publicly documented SNS action. No special keys, admin access, or threshold corruption is required — only a governance majority, which is the normal operating condition for SNS governance. The scenario is realistic in adversarial or contentious governance situations (e.g., a hostile takeover, a faction seeking to silence opposition neurons).

---

### Recommendation

Add a validation check in `validate_and_render_manage_nervous_system_parameters` (and/or at execution time in `perform_manage_nervous_system_parameters`) that prevents `neuron_minimum_dissolve_delay_to_vote_seconds` from being raised above its current value in a single proposal, or at minimum, prevents it from being raised above the dissolve delay of the proposing neuron. Analogous to the rage-quit report's recommendations:

1. **Monotonicity-only increase with a cap**: Require that any increase to `neuron_minimum_dissolve_delay_to_vote_seconds` does not exceed the current value by more than a bounded percentage per proposal.
2. **Protect existing eligible neurons**: Reject any proposed value that would retroactively disenfranchise neurons whose dissolve delay currently satisfies the existing threshold.
3. **Time-lock on changes**: Allow `neuron_minimum_dissolve_delay_to_vote_seconds` to be changed at most once per governance period.

The relevant validation function to modify is `validate_neuron_minimum_dissolve_delay_to_vote_seconds` in `rs/sns/governance/src/types.rs` and the proposal renderer `validate_and_render_manage_nervous_system_parameters` in `rs/sns/governance/src/proposal.rs`. [5](#0-4) 

---

### Proof of Concept

**Setup:**
- SNS is initialized with `neuron_minimum_dissolve_delay_to_vote_seconds = 6 months` and `max_dissolve_delay_seconds = 8 years`.
- Neuron A (attacker): dissolve delay = 8 years, holds 51% of voting power.
- Neuron B (victim): dissolve delay = 6 months (exactly at the threshold), holds 49% of voting power.

**Attack:**
1. Attacker submits a `ManageNervousSystemParameters` proposal setting `neuron_minimum_dissolve_delay_to_vote_seconds = 7 years`.
2. Validation passes: `7 years < max_dissolve_delay_seconds (8 years)`. [6](#0-5) 
3. Attacker's neuron (51% VP) votes yes; proposal is adopted and executed.
4. `perform_manage_nervous_system_parameters` writes the new parameters. [7](#0-6) 
5. On the next proposal, `compute_ballots_for_new_proposal` reads `min_dissolve_delay_for_vote = 7 years`. [8](#0-7) 
6. Neuron B (6-month dissolve delay) is excluded from the electoral roll. It receives no ballot, earns no voting rewards, and cannot vote on any future proposal — despite having staked tokens under the expectation that 6 months was sufficient.

    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }

/// Validates and renders a proposal with action ManageNervousSystemParameters.
fn validate_and_render_manage_nervous_system_parameters(
    new_parameters: &NervousSystemParameters,
    current_parameters: &NervousSystemParameters,
) -> Result<String, String> {
    if new_parameters == &NervousSystemParameters::default() {
        return Err("NervousSystemParameters: at least one field must be set.".to_string());
    }

    new_parameters.inherit_from(current_parameters).validate()?;

    Ok(format!(
        r"# Proposal to change nervous system parameters:
## Current nervous system parameters:

{:#?}

## New nervous system parameters:

{:#?}",
        &current_parameters, new_parameters
    ))
}
