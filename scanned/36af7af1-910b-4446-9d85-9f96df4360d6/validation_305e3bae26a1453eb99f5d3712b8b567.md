### Title
Incorrect Use of `effective_subaccount()` in ICRC-1 Rosetta Block Storage Causes Aggregated Balance Collision - (`rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs`)

### Summary

In the ICRC-1 Rosetta API's `store_blocks` function, `effective_subaccount()` is unconditionally used instead of the actual `subaccount` field when persisting transaction data to SQLite. Because `effective_subaccount()` maps both `None` and `Some([0u8; 32])` to the same `[0u8; 32]` byte array, two semantically distinct on-chain representations are stored identically. A downstream aggregated-balance SQL query that uses `DISTINCT` on the subaccount column then conflates these two rows, producing an incorrect (lower) total balance for any principal that has received transactions to both representations.

### Finding Description

In `store_blocks`, every account's subaccount is stored via `effective_subaccount()`:

```rust
// Mint
Some(*to.effective_subaccount()),   // line 567
// Transfer from/to
Some(*from.effective_subaccount()), // line 584
Some(*to.effective_subaccount()),   // line 586
// Burn
Some(*from.effective_subaccount()), // line 598
// Approve
Some(*from.effective_subaccount()), // line 619
Some(*spender.effective_subaccount()), // line 623
// AuthorizedMint / AuthorizedBurn
Some(*to.effective_subaccount()),   // line 651
Some(*from.effective_subaccount()), // line 664
```

`effective_subaccount()` is defined as:

```rust
pub fn effective_subaccount(&self) -> &Subaccount {
    self.subaccount.as_ref().unwrap_or(DEFAULT_SUBACCOUNT)
}
```

So `Account { subaccount: None }` and `Account { subaccount: Some([0u8; 32]) }` are both stored with subaccount = `[0u8; 32]`. The aggregated-balance query uses `DISTINCT` on `(principal, subaccount)`, so when both representations appear in the ledger history, the query collapses them into a single row and returns a balance that is lower than the true cumulative total.

The test `test_debug_aggregated_balance_sql` in `rs/rosetta-api/icrc1/src/data_api/services.rs` explicitly documents this:

> "BUG CONFIRMED: Aggregated balance mismatch: got {aggregated_balance}, expected {expected_total}. This happens because both None and Some([0;32]) are stored as [0;32] in the database. The DISTINCT clause in the aggregation SQL treats them as one account instead of two."

The fix is analogous to the external report's recommendation: pass the actual `subaccount` field (preserving `None` vs `Some([0;32])` distinction) rather than always resolving through `effective_subaccount()`.

### Impact Explanation

Any user or exchange querying `get_aggregated_balance_for_principal_at_block_idx` via the ICRC-1 Rosetta API for a principal that has received transfers to both `Account { subaccount: None }` and `Account { subaccount: Some([0u8; 32]) }` will receive an understated total balance. This incorrect data can cause:
- Incorrect portfolio valuations reported to end users
- Incorrect balance checks in exchange deposit/withdrawal flows that rely on the Rosetta API
- Systematic under-reporting of holdings for any principal affected by this storage collision

### Likelihood Explanation

The ICRC-1 standard explicitly treats `None` and `Some([0u8; 32])` as the same effective account, so any ledger that has processed transactions using both representations (which is common when different clients use different encodings of the default subaccount) will trigger this bug. The Rosetta API is the standard interface used by exchanges and wallets to query ICRC-1 ledger state, making this a realistic and reachable path for any boundary/API user.

### Recommendation

In `store_blocks`, store the raw `subaccount` field (preserving `None` as `NULL` in the database) rather than always resolving through `effective_subaccount()`. The aggregated-balance query should then use `COALESCE(subaccount, zeroes)` only at query time, or the `DISTINCT` clause should be removed in favour of a `GROUP BY` that correctly handles the `NULL` vs `[0;32]` distinction.

```diff
- Some(*to.effective_subaccount()),
+ to.subaccount,
```

### Proof of Concept

The test `test_debug_aggregated_balance_sql` in `rs/rosetta-api/icrc1/src/data_api/services.rs` constructs exactly this scenario:
1. Mint 6,000,000 to `Account { subaccount: None }`
2. Mint 1,000,000 to `Account { subaccount: Some([0u8; 32]) }`
3. Mint 1,000,000 to `Account { subaccount: Some([0,…,0,1]) }`

Expected aggregated balance: **8,000,000**. Actual result due to the bug: **less than 8,000,000**, because the `DISTINCT` clause in the aggregation SQL collapses rows 1 and 2 into a single entry.
