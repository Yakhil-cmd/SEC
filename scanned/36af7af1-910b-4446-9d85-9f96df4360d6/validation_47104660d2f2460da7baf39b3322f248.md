### Title
Unbounded Neuron Iteration in `compute_ballots_for_new_proposal` Causes Instruction Limit Exhaustion, Permanently Blocking Proposal Submission — (`rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `compute_ballots_for_new_proposal` function iterates over every neuron in `self.proto.neurons` in a single synchronous loop with no instruction-limit guard. When the neuron count is large enough, any call to `make_proposal` traps with `CanisterInstructionLimitExceeded`, permanently disabling governance for the SNS.

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, `compute_ballots_for_new_proposal` (line 5226) performs an unbounded iteration:

```rust
for (k, v) in self.proto.neurons.iter() {
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power(...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { ... });
}
``` [1](#0-0) 

This loop has no instruction-limit check, no pagination, and no early exit. It is called synchronously and unconditionally from `make_proposal`:

```rust
let (_, electoral_roll) = self
    .compute_ballots_for_new_proposal()
    .map_err(...)?;
``` [2](#0-1) 

`make_proposal` is an update call reachable by any neuron holder (unprivileged ingress sender). The IC enforces a hard instruction limit of 40 billion instructions per update message: [3](#0-2) 

The NNS governance canister already recognized this exact problem and addressed it by pre-computing voting power snapshots in a timer task with instruction-limit-aware batching (`compute_voting_power_snapshot_for_standard_proposal`), using `is_message_over_threshold` guards: [4](#0-3) 

The SNS governance has no equivalent protection. Its neurons are stored in a heap `BTreeMap<String, Neuron>` (`self.proto.neurons`), and the entire map is traversed in one shot per proposal submission.

The same pattern also appears in `distribute_rewards` (line 5894), which iterates over all proposals' ballots and all rewarded neurons in a single synchronous call from `run_periodic_tasks`: [5](#0-4) 

---

### Impact Explanation

Once the SNS neuron count crosses the instruction-limit threshold for a single `make_proposal` execution, **every proposal submission traps**. Because `compute_ballots_for_new_proposal` is called before any state mutation is committed, the trap is clean (no partial state corruption), but the practical effect is that the SNS governance canister becomes permanently unable to accept new proposals. This is a critical availability impact: the SNS cannot govern itself, cannot upgrade its own canisters, and cannot execute any governance action.

---

### Likelihood Explanation

SNS token swaps can create large numbers of neurons — one per swap participant. A popular SNS with tens of thousands of participants produces tens of thousands of neurons. The NNS governance benchmark (`compute_ballots_for_new_proposal_with_stable_neurons`) projects that with `MAX_NUMBER_OF_NEURONS` neurons the instruction cost reaches ~25 billion instructions: [6](#0-5) 

The SNS version uses heap memory (faster per access than stable memory), but the loop body still computes dissolve delay and multi-factor voting power for each neuron and inserts into a `BTreeMap`. With a sufficiently large SNS (tens of thousands of neurons), the 40B instruction ceiling is reachable. The attacker-controlled entry path is simply submitting a proposal — any neuron holder with sufficient dissolve delay can trigger this.

---

### Recommendation

1. **Pre-compute and cache the electoral roll** in a timer task (analogous to NNS's `compute_voting_power_snapshot_for_standard_proposal`), using `is_message_over_threshold` / `noop_self_call_if_over_instructions` to break the work across multiple messages: [7](#0-6) 

2. **Use the cached snapshot** in `make_proposal` instead of recomputing from scratch on every call.

3. **Add a benchmark** analogous to the NNS one that projects instruction usage at the maximum neuron count and asserts it stays within the 40B limit.

---

### Proof of Concept

1. Deploy an SNS with a large token swap, creating N neurons (where N is large enough to exhaust 40B instructions in `compute_ballots_for_new_proposal`; the exact threshold depends on per-neuron instruction cost in the SNS heap-based loop).
2. Any neuron holder with sufficient dissolve delay calls `make_proposal` via ingress.
3. The replica executes `make_proposal` → `compute_ballots_for_new_proposal` → iterates all N neurons.
4. The instruction counter exceeds 40B; the execution environment traps the message with `CanisterInstructionLimitExceeded`.
5. No proposal can ever be submitted again; the SNS governance is permanently frozen.

        &self,
        voting_power_economics: &VotingPowerEconomics,
        now_seconds: u64,
    ) -> Result<VotingPowerSnapshot, NeuronStoreError> {
        let mut voting_power_map = HashMap::new();
        let mut total_deciding_voting_power: u128 = 0;
        let mut total_potential_voting_power: u128 = 0;

        let default_min_dissolve_delay = if is_mission_70_voting_rewards_enabled() {
            VotingPowerEconomics::MISSION_70_DEFAULT_NEURON_MINIMUM_DISSOLVE_DELAY_TO_VOTE_SECONDS
        } else {
            VotingPowerEconomics::DEFAULT_NEURON_MINIMUM_DISSOLVE_DELAY_TO_VOTE_SECONDS
        };
        let min_dissolve_delay_seconds = voting_power_economics
            .neuron_minimum_dissolve_delay_to_vote_seconds
            .unwrap_or(default_min_dissolve_delay);

        let mut process_neuron = |neuron: &Neuron| {
            if neuron.is_inactive(now_seconds)
                || neuron.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_seconds
            {
                return;
            }

            let (potential_voting_power, deciding_voting_power) =
                neuron.potential_and_deciding_voting_power(voting_power_economics, now_seconds);
            // We don't handle overflow here, as in `get_voting_power_as_u64` below,
            // the input arguments bigger than u64::MAX will result in an error.
            total_deciding_voting_power =
                total_deciding_voting_power.saturating_add(deciding_voting_power as u128);
            total_potential_voting_power =
                total_potential_voting_power.saturating_add(potential_voting_power as u128);
            voting_power_map.insert(neuron.id().id, deciding_voting_power);
        };

        // Active neurons iterator already makes distinctions between stable and heap neurons.
        self.with_active_neurons_iter_sections(
            |iter| {
                for neuron in iter {
                    process_neuron(&neuron);
                }
            },
            NeuronSections::NONE,
        );

pub async fn noop_self_call_if_over_instructions(
    message_threshold: u64,
    call_context_threshold: Option<u64>,
) -> Result<(), OverCallContextError> {
    // We may still need a new message context for whatever cleanup is needed, but also we will
    // return an error if the call context is over the threshold.
    if is_message_over_threshold(message_threshold) {
        make_noop_call().await;
    }

    if let Some(upper_bound) = call_context_threshold
        && is_call_context_over_threshold(upper_bound)
    {
        return Err(OverCallContextError { limit: upper_bound });
    }

    Ok(())
