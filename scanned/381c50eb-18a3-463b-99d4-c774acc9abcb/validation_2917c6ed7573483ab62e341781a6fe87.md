### Title
CMC `ensure_balance()` Uses Total Canister Balance Instead of Amount-to-Distribute, Enabling Rate-Limiter Bypass and `total_cycles_minted` Undercount - (File: rs/nns/cmc/src/main.rs)

---

### Summary

The `ensure_balance()` function in the Cycles Minting Canister (CMC) computes `cycles_to_mint` as the difference between the requested amount and the CMC's **current total balance**. Because `Cycles` subtraction is saturating, any pre-existing balance in the CMC silently reduces `cycles_to_mint` to zero. The minting rate limiter and the `total_cycles_minted` accounting counter are both gated on this derived value, not on the actual amount being distributed to users. Any canister can deposit cycles into the CMC via the management canister's `deposit_cycles` endpoint, inflating the CMC's balance before a user's `notify_top_up` or `notify_create_canister` call, causing the rate limiter to be checked with zero and the accounting counter to go unincremented.

---

### Finding Description

`ensure_balance()` in `rs/nns/cmc/src/main.rs` reads the CMC's live balance and computes how many cycles need to be freshly minted:

```rust
fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128()); // total CMC balance
    let cycles_to_mint = cycles - current_balance;                           // saturating sub → 0 if balance ≥ cycles

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;   // limiter sees 0
        state.total_cycles_minted += cycles_to_mint;                         // counter unchanged
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);                 // mints 0
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}

---

### Impact Explanation

**1. Rate-limiter bypass (primary):** The hourly minting cap (`base_cycles_limit`, default 150 × 10¹⁵ cycles) is supposed to bound how many cycles users can obtain from ICP per hour. If an attacker pre-loads the CMC with cycles equal to or exceeding the amount a user is converting, `cycles_to_mint` saturates to zero, the limiter records nothing, and the full `cycles` amount is paid out to the user via `call_with_payment128`. The user receives cycles in exchange for ICP without the rate limiter being incremented. Repeated pre-loading allows sustained bypass of the hourly cap.

**2. `total_cycles_minted` undercount (secondary):** `state.total_cycles_minted` is the canonical on-chain record of all cycles ever distributed by the CMC, exposed via the `total_cycles_minted` query and the `cmc_cycles_minted_total` Prometheus metric used for governance and economic analysis. When pre-existing balance covers the request, this counter is not incremented, permanently understating the true amount of cycles distributed. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

Any canister on any subnet can call `deposit_cycles` on the management canister targeting the CMC's canister ID. No privileged role is required. The attacker permanently loses the deposited cycles (they are consumed to fulfill user requests), making large-scale exploitation expensive (~$195 000 USD equivalent to pre-load a full 150 × 10¹⁵ cycle window). However, a motivated actor (e.g., a large ICP holder wishing to convert beyond the hourly cap, or an entity seeking to corrupt governance metrics) has a clear, permissionless path. The `total_cycles_minted` undercount is reachable even with small pre-existing balances left over from normal CMC operation.

---

### Recommendation

Decouple the rate-limiter and accounting update from the amount-to-mint. The limiter and `total_cycles_minted` should be gated on the full `cycles` amount being distributed to the user, regardless of how much of that comes from pre-existing balance versus fresh minting:

```rust
fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance; // saturating; may be 0

    with_state_mut(|state| {
        // Rate-limit and account for the full amount being distributed,
        // not just the freshly minted portion.
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        state.total_cycles_minted += cycles_to_mint; // only count truly new cycles
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

Alternatively, if the intent is that `total_cycles_minted` tracks only ICP-sourced minting, the rate limiter should still be checked against `cycles` (the full distribution amount) to prevent bypass.

### Proof of Concept

1. Attacker canister calls:
   ```
   ic00::deposit_cycles(CMC_CANISTER_ID, 150_000_000_000_000_000u128)
   ```
   This is a standard management canister call available to any canister. The CMC's balance is now ≥ 150 × 10¹⁵ cycles.

2. User sends 1 500 ICP to the CMC's subaccount and calls `notify_top_up` (or `notify_create_canister`). At the current exchange rate this converts to ~150 × 10¹⁵ cycles, which would normally hit the hourly cap.

3. Inside `process_top_up` → `deposit_cycles(canister_id, cycles, true, BaseLimit)` → `ensure_balance(cycles, BaseLimit)`:
   - `current_balance = canister_balance128()` ≥ `cycles`
   - `cycles_to_mint = cycles - current_balance = Cycles(0)` (saturating)
   - `base_limiter.check_and_add_cycles(now, Cycles(0), limit)` → passes, records 0
   - `total_cycles_minted += 0`
   - `ic0_mint_cycles128(0)` → no new cycles minted
   - `call_with_payment128(..., u128::from(cycles))` → 150 × 10¹⁵ cycles sent to user's canister from pre-loaded balance

4. The rate limiter counter remains at 0. The user received 150 × 10¹⁵ cycles in exchange for ICP with no rate-limit enforcement. `total_cycles_minted` is unchanged. [1](#0-0) [5](#0-4) [3](#0-2) [2](#0-1)

    })?;

    Ok(())
}

    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }
