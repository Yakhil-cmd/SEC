### Title
Unprivileged Caller Can DoS `update_balance` by Exhausting Guard Slots via Arbitrary `owner` Parameter - (File: `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

---

### Summary

The ckBTC minter's `update_balance` endpoint accepts an optional `owner: Option<Principal>` parameter and keys its concurrent-execution guard on the caller-supplied `owner` rather than on the actual `msg_caller()`. Any unprivileged ingress sender can supply arbitrary victim principals as `owner`, acquiring guard slots for those accounts. By flooding 100 concurrent calls with distinct principals, an attacker exhausts all available guard slots (`MAX_CONCURRENT = 100`), causing every subsequent legitimate `update_balance` call to fail with `TooManyConcurrentRequests` until the attacker's calls complete. The attacker can re-submit continuously to maintain the denial of service indefinitely.

---

### Finding Description

In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, the guard account is constructed from the caller-supplied `args.owner`:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),   // user-supplied parameter
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;  // guard keyed on user-supplied account
``` [1](#0-0) 

`balance_update_guard` calls `Guard::new(account)`, which inserts the account into `state.update_balance_accounts` (a `BTreeSet<Account>`). It returns `GuardError::AlreadyProcessing` if the account is already present, or `GuardError::TooManyConcurrentRequests` if the set already holds `MAX_CONCURRENT = 100` entries:

```rust
pub fn new(account: Account) -> Result<Self, GuardError> {
    mutate_state(|s| {
        let accounts = PR::pending_requests(s);
        if accounts.contains(&account) {
            return Err(GuardError::AlreadyProcessing);
        }
        if accounts.len() >= MAX_CONCURRENT {
            return Err(GuardError::TooManyConcurrentRequests);
        }
        accounts.insert(account);
        ...
    })
}
``` [2](#0-1) 

The guard is released only when the `Guard` value is dropped (i.e., when the async `update_balance` call completes): [3](#0-2) 

Because `args.owner` is entirely attacker-controlled, any caller can:

1. **Targeted attack**: Call `update_balance(owner = Some(victim_principal))` before the victim. The guard for the victim's account is acquired. The victim's own call returns `AlreadyProcessing`. The attacker re-submits immediately after their call completes to keep the victim perpetually blocked.

2. **Global DoS**: Submit 100 concurrent `update_balance` calls, each with a distinct `owner = Some(principal_i)` (e.g., 100 freshly generated principals with no UTXOs). Each call acquires one guard slot and then awaits an async `get_utxos` call to the Bitcoin canister. While all 100 calls are in-flight, the `update_balance_accounts` set is full. Every legitimate user calling `update_balance` receives `TooManyConcurrentRequests`. The attacker re-submits as slots free up to maintain the DoS.

The `update_balance` function is publicly callable by any non-anonymous principal with no preconditions: [4](#0-3) 

The DID interface confirms `owner` is optional and open to any caller: [5](#0-4) 

The same pattern exists in the ckDOGE minter, which exposes an identical `update_balance` signature: [6](#0-5) 

---

### Impact Explanation

`update_balance` is the **only** mechanism by which users can mint ckBTC after depositing BTC on-chain. Blocking it prevents all users from completing BTC→ckBTC conversions. The attacker does not need to steal funds; the DoS alone is sufficient to render the minter non-functional. The `MAX_CONCURRENT = 100` cap is a hard ceiling: [7](#0-6) 

An attacker holding 100 in-flight calls (each awaiting a `get_utxos` response from the Bitcoin canister) fully saturates this cap. Calls to accounts with no UTXOs complete quickly, so the attacker can cheaply cycle through slots to maintain the DoS.

**Impact: High** — the deposit flow for ckBTC (and ckDOGE) is completely blocked for all users.

---

### Likelihood Explanation

**Likelihood: High** — there are no preconditions. Any non-anonymous principal can call `update_balance` with an arbitrary `owner`. No special permissions, tokens, or prior state are required. The attack requires only the ability to submit ingress messages to the minter canister, which is open to the public on mainnet.

---

### Recommendation

Key the guard on the actual `msg_caller()` rather than on `args.owner`. The `owner` parameter is intended to allow third parties to trigger minting on behalf of others (a legitimate use case), but the concurrency guard should track the actual caller to prevent abuse:

```rust
// Before (vulnerable):
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;

// After (fixed):
let _guard = balance_update_guard(Account { owner: caller, subaccount: None })?;
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
```

This mirrors the approach used in `retrieve_btc`, which correctly guards on `caller` directly: [8](#0-7) 

And the ckETH minter's `withdraw_eth`, which also guards on `caller`: [9](#0-8) 

---

### Proof of Concept

1. Attacker generates 100 distinct IC principals `P_1 … P_100` (e.g., via `Principal::self_authenticating` with 100 different public keys).
2. Attacker submits 100 concurrent ingress update calls to the ckBTC minter's `update_balance` endpoint, each with `owner = Some(P_i)` and `subaccount = None`. These principals have no UTXOs, so each call will eventually return `NoNewUtxos`, but only after awaiting the async `get_utxos` response from the Bitcoin canister.
3. While all 100 calls are in-flight, `state.update_balance_accounts.len() == 100`.
4. Any legitimate user calling `update_balance` (with or without an `owner` argument) receives `UpdateBalanceError::TemporarilyUnavailable("too many concurrent requests")`.
5. As each attacker call completes and frees a slot, the attacker immediately re-submits a new call for a fresh principal to keep the set full.
6. The victim's BTC deposit is confirmed on-chain but ckBTC cannot be minted for as long as the attacker maintains the DoS.
