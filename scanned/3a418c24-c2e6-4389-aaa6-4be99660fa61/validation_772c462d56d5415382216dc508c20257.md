### Title
Ingress Memory Byte Limit Not Enforced During Block Validation, Only During Building — (`File: rs/ingress_manager/src/ingress_selector.rs`)

---

### Summary

The IC's `validate_ingress_payload` enforces message count and per-message size limits but never validates the **total in-memory ingress payload size** against `max_ingress_bytes_per_block`. When the hashes-in-blocks feature is active, the validation path returns only the tiny wire-size (sum of message IDs, ~40 bytes each) to the outer block validator, which then checks it against `max_block_payload_size`. A Byzantine block proposer below the consensus fault threshold can therefore include up to 1,000 messages of 2 MB each (total ~2 GB in memory) in a single block, and every honest node will accept and induct that block, exhausting replica memory.

---

### Finding Description

**Block building** (`get_ingress_payload`, `rs/ingress_manager/src/ingress_selector.rs`) correctly enforces two independent limits:

```
// Break criterion #1: global byte limit
if accumulated_wire_size + size_estimates.wire > wire_byte_limit { break 'outer; }
if accumulated_memory_size + size_estimates.memory > memory_byte_limit { break 'outer; }
``` [1](#0-0) 

`memory_byte_limit` is derived from `max_ingress_bytes_per_block` (default 4 MB) and is independent of the wire limit when hashes-in-blocks is enabled:

```rust
fn memory_byte_limit(&self, wire_byte_limit: NumBytes, ...) -> Result<NumBytes, String> {
    ...
    if self.hashes_in_blocks_enabled() {
        Ok(NumBytes::new(memory_byte_limit))   // independent of wire limit
    } else {
        Ok(NumBytes::new(std::cmp::min(wire_byte_limit.get(), memory_byte_limit)))
    }
}
``` [2](#0-1) 

**Block validation** (`validate_ingress_payload`) checks only message count and per-message size, then returns the **wire** size estimate — never the memory size:

```rust
if payload.message_count() > settings.max_ingress_messages_per_block { return Err(...); }
// per-message checks (size, cycles, expiry) ...
let size_estimates = self.payload_size_estimates(payload);
Ok(size_estimates.wire)   // ← memory size is never checked or returned
``` [3](#0-2) 

`payload_size_estimates` computes two distinct values, but only `wire` is ever used in the validation return:

```rust
fn payload_size_estimates(&self, payload: &IngressPayload) -> SizeEstimates {
    let memory_bytes = payload.total_messages_size_estimate() + payload.total_ids_size_estimate();
    let wire_bytes = if self.hashes_in_blocks_enabled() {
        payload.total_ids_size_estimate()   // ← only IDs, ~40 bytes per message
    } else { memory_bytes };
    SizeEstimates { memory: memory_bytes, wire: wire_bytes }
}
``` [4](#0-3) 

The outer `PayloadBuilderImpl::validate_payload` accumulates the wire sizes returned by each section validator and checks only against `max_block_payload_size`:

```rust
let mut accumulated_size = NumBytes::new(0);
for builder in &self.section_builder {
    accumulated_size +=
        builder.validate_payload(height, batch_payload, proposal_context, past_payloads)?;
}
if accumulated_size > max_block_payload_size * 2 { return Err(...); }
Ok(())
``` [5](#0-4) 

`max_block_payload_size` is a **wire-size** limit (default 4 MB). `max_ingress_bytes_per_block` is a **memory-size** limit (also default 4 MB, but explicitly allowed to be larger, and semantically distinct). The proto comment confirms the separation:

> "Note that this value can be larger than `max_block_payload_size`." [6](#0-5) 

---

### Impact Explanation

With hashes-in-blocks enabled:

| Quantity | Value |
|---|---|
| `max_ingress_messages_per_block` | 1,000 |
| `max_ingress_bytes_per_message` | 2 MB |
| Maximum memory footprint | 1,000 × 2 MB = **~2 GB** |
| Wire size of same payload | 1,000 × ~40 bytes = **~40 KB** |
| `max_block_payload_size` check | 40 KB ≪ 4 MB → **passes** |
| `max_ingress_bytes_per_block` check | **never performed during validation** |

Every honest node validates the block, accepts it, and inducting the full 2 GB of ingress messages into replicated state. This can cause out-of-memory conditions and subnet halt across all nodes simultaneously. [7](#0-6) 

---

### Likelihood Explanation

The attacker requires one Byzantine block proposer node — a single node below the consensus fault threshold (f out of 3f+1). This is explicitly within the IC's Byzantine fault tolerance threat model. The ingress messages themselves are submitted by unprivileged users; each is individually valid (≤ 2 MB, valid signature, sufficient cycles). The block proposer simply selects more of them than the memory limit allows. No threshold corruption, no key compromise, and no governance majority is needed.

---

### Recommendation

In `validate_ingress_payload`, compute the memory size estimate and reject payloads that exceed `max_ingress_bytes_per_block`, mirroring the check already present in `get_ingress_payload`:

```rust
let size_estimates = self.payload_size_estimates(payload);

// Mirror the memory limit check from get_ingress_payload
let memory_byte_limit = self
    .get_ingress_message_settings(context.registry_version)
    .expect("...")
    .max_ingress_bytes_per_block as u64;

if size_estimates.memory > NumBytes::new(memory_byte_limit) {
    return Err(ValidationError::InvalidArtifact(
        InvalidIngressPayloadReason::IngressPayloadTooLarge(...),
    ));
}

Ok(size_estimates.wire)
``` [8](#0-7) 

---

### Proof of Concept

1. Submit 1,000 ingress messages, each with a ~2 MB payload, to a target subnet canister. Each message is individually valid (size ≤ `MAX_INGRESS_BYTES_PER_MESSAGE_APP_SUBNET` = 2 MB, valid sender signature, canister has sufficient cycles).

2. Wait for the messages to be gossiped to all subnet nodes and enter their validated ingress pools.

3. On the Byzantine block proposer node, override `get_ingress_payload` (or directly craft a `BatchPayload`) to include all 1,000 messages, bypassing the local `memory_byte_limit` guard.

4. Propose the block. Its wire size is ~40 KB (1,000 × ~40-byte IDs), well within `max_block_payload_size` = 4 MB.

5. Every honest node calls `validate_ingress_payload`:
   - `payload.message_count()` = 1,000 ≤ `max_ingress_messages_per_block` = 1,000 → **passes**
   - Each message individually ≤ 2 MB → **passes**
   - Returns `size_estimates.wire` ≈ 40 KB
   - Outer check: 40 KB ≤ 4 MB × 2 → **passes**

6. Block is finalized. All nodes induct ~2 GB of ingress messages into replicated state, triggering OOM and subnet halt. [3](#0-2) [5](#0-4)
