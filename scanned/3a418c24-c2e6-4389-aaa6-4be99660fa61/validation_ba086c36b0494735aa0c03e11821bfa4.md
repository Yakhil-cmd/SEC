### Title
Malicious NNS Node Operator Can Permanently Block Governance Upgrade Proposal Rejection via Repeated Proposal Overwrite - (File: rs/nns/handlers/root/impl/src/root_proposals.rs)

### Summary

The `submit_root_proposal_to_upgrade_governance_canister` function in the NNS root canister unconditionally overwrites any existing pending proposal from the same proposer principal, resetting all accumulated votes and the 7-day expiry timer. A malicious NNS node operator can exploit this to prevent their governance upgrade proposal from ever being rejected by repeatedly overwriting it whenever enough "No" votes accumulate, keeping the proposal alive indefinitely.

### Finding Description

Root proposals to upgrade the governance canister are stored in a `BTreeMap<PrincipalId, GovernanceUpgradeRootProposal>` keyed by proposer principal: [1](#0-0) 

When a proposer submits a new proposal, any existing proposal from the same principal is silently overwritten with no guard: [2](#0-1) 

This overwrite:
1. Resets all `node_operator_ballots` to `Undecided` (except the proposer's own automatic `Yes`)
2. Resets `submission_timestamp_seconds` to `now`, restarting the 7-day expiry clock

The `vote_on_root_proposal_to_upgrade_governance_canister` function is `async` and yields control at line 311 during `get_nns_membership(...).await`: [3](#0-2) 

During this yield point, the IC canister runtime can process other ingress messages. A malicious proposer can interleave a `submit_root_proposal_to_upgrade_governance_canister` call here, overwriting the proposal before the voter's ballot is recorded against the accumulated state.

The rejection threshold requires `f + 1` "No" votes where `f = (N-1)/3`: [4](#0-3) 

**Attack flow:**
1. Malicious node operator A submits a proposal to upgrade governance to a malicious Wasm
2. Other node operators vote "No" — the `No` count accumulates toward `f + 1`
3. When the "No" count reaches `f` (one vote away from rejection), A calls `submit_root_proposal_to_upgrade_governance_canister` again with the same (or any valid) Wasm
4. All accumulated "No" ballots are reset to `Undecided`; the 7-day timer resets to `now`
5. A repeats this whenever rejection is imminent

The proposer can also exploit the TOCTOU window: a voter's `vote_on_root_proposal_to_upgrade_governance_canister` call reads the proposal at line 309, then yields at line 311. If A overwrites the proposal during that yield, the voter's ballot is applied to the fresh proposal (with reset vote counts), not the accumulated one.

### Impact Explanation

A malicious NNS node operator can keep a governance upgrade proposal alive indefinitely and prevent it from ever being rejected. The 7-day expiry is neutralized because each overwrite resets `submission_timestamp_seconds`. If the attacker eventually controls or coordinates enough nodes to reach the `N - f` "Yes" threshold, the malicious Wasm upgrade executes — compromising the governance canister and, by extension, the entire NNS. Even without reaching execution, the attack permanently occupies the proposer's proposal slot and forces honest node operators into an indefinite re-voting loop.

### Likelihood Explanation

The attacker must be a node operator on the NNS subnet — a role reachable by any entity that operates NNS nodes, without requiring admin keys or a majority. The attack requires only monitoring the public proposal state (via `get_pending_root_proposals_to_upgrade_governance_canister`) and submitting a new proposal before the rejection threshold is crossed. No cryptographic break or privileged access beyond the node operator role is needed. The motivation is high: a malicious node operator who wants to push through a governance takeover has strong incentive to prevent rejection.

### Recommendation

1. **Prevent overwriting a proposal that has accumulated non-`Undecided` votes.** If any ballot has been cast, reject the overwrite with an error.
2. **Do not reset `submission_timestamp_seconds` on overwrite.** Preserve the original submission time so the 7-day window cannot be extended.
3. **Alternatively, disallow resubmission entirely while a proposal is pending.** Require the proposer to explicitly withdraw before resubmitting.

### Proof of Concept

```
// Setup: NNS subnet with N=7 nodes, f=2, rejection threshold = f+1 = 3 "No" votes.

1. Node operator A calls submit_root_proposal_to_upgrade_governance_canister(malicious_wasm)
   → PROPOSALS[A] = { ballots: [(A,Yes),(B,Undecided),(C,Undecided),...], timestamp: T1 }

2. B votes No → ballots: [(A,Yes),(B,No),(C,Undecided),...]
3. C votes No → ballots: [(A,Yes),(B,No),(C,No),...]
   // One more "No" vote would reject the proposal.

4. A calls submit_root_proposal_to_upgrade_governance_canister(same_or_any_valid_wasm)
   → PROPOSALS[A] = { ballots: [(A,Yes),(B,Undecided),(C,Undecided),...], timestamp: T2 }
   // All "No" votes erased. Timer reset.

5. Repeat from step 2 indefinitely.
   // Proposal is never rejected. Timer never expires.
```
