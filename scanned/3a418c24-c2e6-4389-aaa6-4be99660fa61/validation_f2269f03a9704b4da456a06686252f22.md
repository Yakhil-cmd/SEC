### Title
Malformed `bytes32` Principal Encoding in Ethereum Deposit Helper Contracts Causes Irrecoverable ETH/ERC20 Loss — (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/EthDepositHelper.sol`)

---

### Summary

The Ethereum helper smart contracts (`CkEthDeposit` and `CkDeposit`) accept any arbitrary `bytes32` value as the IC principal without performing any on-chain validation of the encoding. When a user provides a malformed `bytes32`, the ETH or ERC-20 tokens are irrevocably transferred to the minter's Ethereum address, but the IC minter's `parse_principal_from_slice()` subsequently fails to decode the principal. The result is an `InvalidDeposit` event with no ckETH/ckERC20 minted and no on-chain recovery path for the deposited funds.

---

### Finding Description

**Entry points with no principal validation:**

`EthDepositHelper.sol` (`CkEthDeposit.deposit`): [1](#0-0) 

`DepositHelperWithSubaccount.sol` (`CkDeposit.depositEth` and `CkDeposit.depositErc20`): [2](#0-1) 

Both functions accept any `bytes32 principal` and immediately transfer ETH/ERC20 to the minter's Ethereum address and emit the deposit event — with zero validation that the `bytes32` encodes a valid IC principal.

**Decoding happens only after funds are committed, on the IC side:**

The IC minter scrapes the emitted `ReceivedEth` / `ReceivedEthOrErc20` log events and calls `parse_principal_from_slice()`: [3](#0-2) 

This function enforces the full encoding contract (first byte = length in `[1,29]`, trailing bytes must be zero, anonymous/management principals rejected). If any check fails, it returns `Err(...)`.

**The parser propagates the failure as `InvalidPrincipal`:** [4](#0-3) 

**`register_deposit_events` records `InvalidDeposit` and mints nothing:** [5](#0-4) 

**`InvalidDeposit` is a terminal state — no retry, no reimbursement:** [6](#0-5) 

The `InvalidDeposit` variant of `InvalidEventReason` is explicitly documented as "Deposit is invalid and was never minted." There is no `revertReceive`-equivalent path; the ETH/ERC20 remains permanently in the minter's Ethereum address.

**The documentation acknowledges the risk but does not enforce it in code:**

The ckETH and ckERC20 documentation warns: *"It's critical that the encoded IC principal is correct otherwise the funds will be lost."* This is a documentation-only mitigation with no enforcement at the smart-contract layer. [7](#0-6) 

---

### Impact Explanation

Any user who calls `deposit(bytes32)`, `depositEth(bytes32, bytes32)`, or `depositErc20(address, uint256, bytes32, bytes32)` with a `bytes32` that does not correctly encode an IC principal (e.g., first byte = 0, first byte > 29, non-zero trailing bytes, or bytes that form an invalid principal) will:

1. Permanently transfer their ETH or ERC-20 tokens to the minter's Ethereum address.
2. Receive no ckETH or ckERC20 in return.
3. Have no on-chain recovery path — the `InvalidDeposit` state is final and no reimbursement mechanism exists.

This is a **chain-fusion mint/burn conservation bug**: real assets are consumed on the Ethereum side with no corresponding issuance on the IC side, and no refund.

---

### Likelihood Explanation

The `bytes32` encoding of an IC principal is non-trivial and error-prone:
- First byte must equal the number of principal bytes (1–29).
- Remaining bytes after the principal must be zero-padded.
- The encoding is not a standard Ethereum type; users must use a specific tool (`principal-to-hex`, the minter dashboard, or the JS helper).

Users who manually construct the `bytes32`, use a wrong tool, copy-paste a truncated or corrupted value, or confuse the principal encoding with a raw hex address will silently produce an invalid `bytes32`. The Ethereum transaction succeeds (no revert), giving no indication of the error until the IC minter processes the log — at which point the funds are already gone. This is a realistic, unprivileged user-triggered path requiring no special access.

---

### Recommendation

Add on-chain validation in the Ethereum helper contracts to verify the `bytes32` principal encoding before accepting the deposit. At minimum:

```solidity
function _validatePrincipal(bytes32 principal) internal pure {
    uint8 len = uint8(principal[0]);
    require(len >= 1 && len <= 29, "Invalid principal: length out of range");
    for (uint i = 1 + len; i < 32; i++) {
        require(principal[i] == 0, "Invalid principal: non-zero trailing bytes");
    }
}
```

Call `_validatePrincipal(principal)` at the top of `deposit`, `depositEth`, and `depositErc20` and revert if the check fails. This mirrors the recommendation in the external report: enforce the exact format constraint at the entry point, before any asset transfer occurs.

Alternatively, implement a reimbursement path in the IC minter for `InvalidDeposit` events (analogous to the `revertReceive` path recommended in the external report), so that ETH/ERC20 deposited with a malformed principal can be returned to the sender's Ethereum address.

---

### Proof of Concept

1. User calls `depositEth(bytes32(0), bytes32(0))` on `CkDeposit` with `msg.value = 1 ETH`. The first byte of `bytes32(0)` is `0x00`, which `parse_principal_from_slice` rejects as "management canister principal is not allowed."
2. The Solidity function does not revert; it transfers 1 ETH to `minterAddress` and emits `ReceivedEthOrErc20(address(0), msg.sender, 1e18, bytes32(0), bytes32(0))`.
3. The IC minter scrapes the log. `parse_principal` calls `parse_principal_from_slice(&[0x00, 0x00, ..., 0x00])`, which returns `Err("management canister principal is not allowed")`.
4. `register_deposit_events` records `EventType::InvalidDeposit { event_source, reason: "failed to decode principal from bytes 0x0000...0000" }`.
5. No ckETH is minted. The user's 1 ETH is permanently held by the minter's Ethereum address with no recovery mechanism.
6. The same scenario applies to `depositErc20` with any ERC-20 token and a malformed `bytes32` principal (e.g., first byte `> 29`, or non-zero trailing bytes).

```rust
fn parse_principal_from_slice(slice: &[u8]) -> Result<Principal, String> {
    const ANONYMOUS_PRINCIPAL_BYTES: [u8; 1] = [4];

    if slice.is_empty() {
        return Err("slice too short".to_string());
    }
    if slice.len() > 32 {
        return Err(format!("Expected at most 32 bytes, got {}", slice.len()));
    }
    let num_bytes = slice[0] as usize;
    if num_bytes == 0 {
        return Err("management canister principal is not allowed".to_string());
    }
    if num_bytes > 29 {
        return Err(format!(
            "invalid number of bytes: expected a number in the range [1,29], got {num_bytes}",
        ));
    }
    if slice.len() < 1 + num_bytes {
        return Err("slice too short".to_string());
    }
    let (principal_bytes, trailing_zeroes) = slice[1..].split_at(num_bytes);
    if !trailing_zeroes
        .iter()
        .all(|trailing_zero| *trailing_zero == 0)
    {
        return Err("trailing non-zero bytes".to_string());
    }
    if principal_bytes == ANONYMOUS_PRINCIPAL_BYTES {
        return Err("anonymous principal is not allowed".to_string());
    }
    Principal::try_from_slice(principal_bytes).map_err(|err| err.to_string())
```
