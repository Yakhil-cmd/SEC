### Title
Permanent Unvalidated-Pool Memory Leak via `start_block_height = u64::MAX` in DKG Message — (`rs/artifact_pool/src/dkg_pool.rs`)

---

### Summary

A Byzantine P2P peer can craft a `dkg::Message` whose `content.dkg_id.start_block_height` is set to `u64::MAX`. Because `DkgMessageId.height` is derived directly from that field with no bounds check, and the purge filter is a strict less-than comparison (`id.height < height`), the message can never be evicted from the unvalidated pool by any finite purge height. The validator silently skips (rather than invalidates) messages whose `start_block_height` does not match the current DKG interval, so the message is also never removed through the validation path. The result is a permanent, unbounded memory leak in the unvalidated pool that violates the stated pool invariant.

---

### Finding Description

**Step 1 — ID derivation, no bounds check**

`DkgMessageId::from` copies `start_block_height` verbatim into `DkgMessageId.height`: [1](#0-0) 

No validation or clamping of the field occurs here.

**Step 2 — Unvalidated pool insertion, no admission guard**

`DkgPoolImpl::insert` places any received artifact directly into the unvalidated section: [2](#0-1) 

There is no check on `start_block_height` at this point.

**Step 3 — Validator silently skips, does not invalidate**

`validate_dealings_for_dealer` compares `start_block_height` against the current DKG start height. When they differ it returns an empty mutation set — the message is left in the unvalidated pool, not removed: [3](#0-2) 

A message with `start_block_height = u64::MAX` will never equal any real `dkg_start_height`, so it is skipped on every validation pass, forever.

**Step 4 — Purge filter never fires**

`purge` evicts entries whose `id.height < height`. For `id.height = u64::MAX`, this predicate is false for every finite purge height, including `u64::MAX - 1`: [4](#0-3) [5](#0-4) 

**Step 5 — Invariant violated**

The `DkgPool` interface explicitly states the invariant that all messages in both sections must correspond to the current DKG start height: [6](#0-5) 

A message with `height = u64::MAX` permanently violates this invariant.

---

### Impact Explanation

Every honest replica that receives such a crafted message accumulates it in its unvalidated pool indefinitely. Because `DkgMessageId` is keyed by `(hash, height)`, a single Byzantine peer can inject many distinct messages (varying the dealing content to produce different hashes) all sharing `height = u64::MAX`. None are ever purged. Over time this constitutes an unbounded memory leak on every honest replica in the subnet, potentially contributing to OOM conditions and replica instability.

---

### Likelihood Explanation

The attack requires only a single Byzantine P2P peer — a replica that has been compromised but is below the fault threshold. No privileged access, no key material, no governance majority, and no threshold corruption is needed. The peer simply sends crafted `DkgMessage` protobuf artifacts over the existing P2P gossip channel. The path is fully local-testable with a unit test (as the question's proof idea outlines).

---

### Recommendation

1. **Invalidate, do not skip, messages with `start_block_height` that cannot match any current or future DKG interval.** In `validate_dealings_for_dealer`, replace the silent `Mutations::new()` return for height mismatches with `ChangeAction::HandleInvalid` when the height is provably unreachable (e.g., greater than the current finalized height by more than the maximum DKG interval length).
2. **Add an upper-bound admission check in `DkgPoolImpl::insert`** that rejects messages whose `start_block_height` exceeds a reasonable future horizon relative to `current_start_height`.
3. **Alternatively, change the purge predicate** from strict less-than to less-than-or-equal, and ensure `Purge(u64::MAX)` is issued at subnet shutdown — though this is a weaker fix.

---

### Proof of Concept

```rust
// Pseudocode — directly exercisable as a unit test in dkg_pool.rs
let mut pool = DkgPoolImpl::new(MetricsRegistry::new(), no_op_logger(), Height::from(0));

// Byzantine peer crafts a message with start_block_height = u64::MAX
let msg = make_message(Height::from(u64::MAX), node_test_id(1));
pool.insert(UnvalidatedArtifact { message: msg, peer_id: node_test_id(99), timestamp: UNIX_EPOCH });

// Purge at u64::MAX - 1: message survives (u64::MAX < u64::MAX-1 is false)
pool.apply(vec![ChangeAction::Purge(Height::from(u64::MAX - 1))]);
assert_eq!(pool.get_unvalidated().count(), 1); // still present

// Even purge at u64::MAX does NOT remove it (strict less-than, not <=)
pool.apply(vec![ChangeAction::Purge(Height::from(u64::MAX))]);
assert_eq!(pool.get_unvalidated().count(), 1); // still present — permanent leak
```

The message is never evicted. Repeating the insert with varied dealing content (different hash) accumulates entries without bound.
