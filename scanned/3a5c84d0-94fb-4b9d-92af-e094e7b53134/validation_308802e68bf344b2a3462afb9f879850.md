### Title
Shared Allowance Mapping Between `transfer_from` and Burn-via-`transfer_from` Enables Spender to Destroy Tokens Beyond Transfer Intent - (`rs/ledger_suite/icrc1/src/lib.rs` and `rs/ledger_suite/icp/src/lib.rs`)

---

### Summary

Both the ICRC-1 ledger and the ICP ledger use a single `AllowanceTable` for two semantically distinct delegated operations: a regular `transfer_from` (which consumes `amount + fee` from the allowance) and a burn-via-`transfer_from` (triggered when `to == minting_account`, which consumes only `amount` from the allowance with zero fee). A spender holding an allowance granted for the purpose of transferring tokens can instead call `icrc2_transfer_from` with `to = minting_account`, triggering the burn path and destroying more tokens from the approver's account than a transfer would have permitted, while consuming the same allowance budget.

---

### Finding Description

In `execute_transfer_not_async` in the ICRC-1 ledger, the `to` address is inspected to decide whether the operation is a burn or a transfer: [1](#0-0) 

When `to == minting_account`, the transaction is constructed as `Operation::Burn` with `effective_fee = Tokens::zero()`. When `to` is any other account, it is `Operation::Transfer` with `effective_fee = ledger.transfer_fee()`.

In `Transaction::apply` for the ICRC-1 ledger, the `Burn` arm checks and deducts only `amount` from the allowance: [2](#0-1) 

The `Transfer` arm checks and deducts `amount + fee`: [3](#0-2) 

Both arms draw from the same `AllowanceTable`. There is no separate burn-allowance mapping. The identical pattern exists in the ICP ledger's `apply_operation`: [4](#0-3) [5](#0-4) 

The `icrc2_transfer_from` endpoint in the ICRC-1 ledger passes the caller-supplied `to` directly into `execute_transfer` with the spender set: [6](#0-5) 

There is no guard preventing a spender from supplying `to = minting_account` in a `transfer_from` call.

---

### Impact Explanation

With an allowance of `A` and a transfer fee of `F`:

| Path | Allowance consumed | Tokens removed from `from` | Tokens received by recipient |
|---|---|---|---|
| `transfer_from` (normal) | `A` | `A` (`A-F` to recipient + `F` fee) | `A - F` |
| `transfer_from` to minting account (burn) | `A` | `A` (all burned, no fee) | 0 |

The spender can destroy `F` more tokens from the approver's account than a transfer would have moved to a recipient, while consuming the identical allowance. More critically, the approver's intent (tokens going to a recipient) is violated: the spender burns the tokens instead. This is a **ledger conservation bug and approval-confusion bug**: the approver's funds are destroyed rather than transferred, and the total supply is reduced by `A` instead of `F`.

---

### Likelihood Explanation

Any principal holding an ICRC-2 allowance on any ICRC-1 or ICP ledger canister can exploit this immediately by calling `icrc2_transfer_from` with `to` set to the ledger's minting account. No privileged role, governance majority, or threshold corruption is required. The entry path is a standard ingress update call reachable by any unprivileged user.

---

### Recommendation

Introduce a separate burn-allowance mapping (analogous to the TRSRY report's `loanApproval`) or add an explicit guard in `execute_transfer_not_async` / `icrc1_send` that rejects `transfer_from` calls where `to == minting_account`, requiring a dedicated `burn_from` endpoint with its own approval type. At minimum, document that an ICRC-2 allowance implicitly grants burn permission, so approvers are aware.

---

### Proof of Concept

1. `from` holds 200,000 tokens. Transfer fee = 10,000.
2. `from` calls `icrc2_approve(spender, amount=110,000)` — intending to allow one transfer of 100,000 tokens.
3. `spender` calls `icrc2_transfer_from(from=from, to=minting_account, amount=110,000)`.
4. `execute_transfer_not_async` detects `to == minting_account` and constructs `Operation::Burn` with `effective_fee = 0`.
5. The burn arm checks `allowance (110,000) >= amount (110,000)` — passes.
6. `from`'s balance decreases by 110,000 (all burned). Allowance consumed: 110,000.
7. Result: `from` loses 110,000 tokens (all destroyed), whereas a legitimate `transfer_from` of 100,000 would have sent 100,000 to a recipient and burned only 10,000 as fee.
8. The spender has destroyed 10,000 more tokens from `from`'s account than the approver intended, and the recipient receives nothing. [1](#0-0) [7](#0-6)

async fn icrc2_transfer_from(arg: TransferFromArgs) -> Result<Nat, TransferFromError> {
    let spender_account = Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: arg.spender_subaccount,
    };
    execute_transfer(
        arg.from,
        arg.to,
        Some(spender_account),
        arg.fee,
        arg.amount,
        arg.memo,
        arg.created_at_time,
    )
    .await
    .map_err(convert_transfer_error)
    .map_err(|err| {
        let err: TransferFromError = match err.try_into() {
            Ok(err) => err,
            Err(err) => ic_cdk::trap(&err),
        };
        err
    })
}
