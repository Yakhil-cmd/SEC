### Title
Moving-Average Price Lag in Maturity Modulation Enables Favorable ICP Minting Timing - (File: `rs/nns/governance/src/timer_tasks/update_icp_xdr_rate_related_data.rs`)

---

### Summary

The NNS Governance canister computes a `MaturityModulation` factor from a **7-day moving average** of ICP/XDR prices, further rate-limited to change by at most **30 permyriad (0.3%) per day**. This factor is applied directly to the ICP amount minted when neuron holders disburse or spawn maturity. The double lag (moving average + daily speed limit) means the modulation systematically under-reacts to sharp ICP price movements, creating a predictable window during which any unprivileged neuron holder can time their `disburse_maturity` or `spawn_neuron` calls to receive more ICP than the current price warrants.

---

### Finding Description

`compute_maturity_modulation_permyriad` in `rs/nns/governance/src/timer_tasks/update_icp_xdr_rate_related_data.rs` computes the modulation as:

```
target = sensitivity * (7-day-avg - 365-day-avg) / 365-day-avg
```

with `MATURITY_MODULATION_SENSITIVITY_PERMYRIAD = 2_500`, then clamps the daily change to `MATURITY_MODULATION_DAILY_SPEED_LIMIT_PERMYRIAD = 30` permyriad/day, and finally clamps the result to `[-1000, +200]` permyriad (−10% to +2%). [1](#0-0) 

The 7-day window is the "current price" signal: [2](#0-1) 

The daily speed limit further slows the modulation's response: [3](#0-2) 

Additionally, `compute_average_icp_xdr_rate` uses **Last Observation Carried Forward (LOCF)** for days where XRC failed to return a rate, meaning a stale rate can be silently repeated across multiple days, amplifying the lag: [4](#0-3) 

The `validate_exchange_rate` function that gates all XRC data entering the price history validates only the **number of sources**, not the **age** of the returned rate: [5](#0-4) 

The resulting `MaturityModulation.current_value_permyriad` is consumed directly in two ICP-minting paths:

1. **Neuron spawning** — `maybe_spawn_neurons` applies `apply_maturity_modulation(original_maturity, maturity_modulation)` to determine the ICP stake minted: [6](#0-5) 

2. **Maturity disbursement finalization** — `next_maturity_disbursement_to_finalize` applies the same modulation to the disbursement amount: [7](#0-6) 

The maturity disbursement has a **7-day finalization delay** (`DISBURSEMENT_DELAY_SECONDS = ONE_DAY_SECONDS * 7`): [8](#0-7) 

This means the modulation applied at finalization is the one computed 7 days after initiation — a window during which the modulation may still be favorable even if the spot price has moved sharply against the user.

---

### Impact Explanation

When ICP price drops sharply (e.g., 20% in a single day), the maturity modulation should move toward its minimum (−10%) to discourage selling. However:

- The 7-day moving average dilutes the price drop: after day 1, the 7-day average reflects only 1/7 of the drop.
- The daily speed limit of 30 permyriad/day means the modulation can only move 0.3% per day toward the new target.
- If the modulation was at +2% before the drop, it takes at least 40 days to reach −10%, even if the price stays depressed.

During this multi-day to multi-week window, any neuron holder can initiate a `disburse_maturity` call and receive ICP minted at a modulation that is materially higher than the current price warrants. The maximum discrepancy between the lagged modulation and the warranted modulation is bounded by the full range: up to **12 percentage points** (from +2% to −10%). For a holder with 10 million ICP-equivalent maturity, this represents up to **1.2 million ICP** of excess minting — real ICP inflation at the expense of the protocol.

The ICP minted in excess is not backed by any corresponding value, constituting a **ledger conservation violation** (excess ICP supply).

---

### Likelihood Explanation

- The ICP price and the current maturity modulation value are both publicly observable (via `get_maturity_modulation` query and on-chain price history).
- Any neuron holder — an unprivileged ingress sender — can call `disburse_maturity` at any time with no special permissions.
- Sharp ICP price movements occur regularly. The lag is structural and deterministic, not probabilistic.
- The 7-day disbursement delay actually helps the attacker: they can initiate during a favorable modulation window and the modulation at finalization is still lagged.

Likelihood: **Medium-High**. Requires monitoring price and modulation, but no privileged access, no key material, and no threshold corruption.

---

### Recommendation

**Short term:** When finalizing a maturity disbursement, compare the modulation's `updated_at_days_since_epoch` against the current day. If the modulation is stale by more than a configurable threshold (e.g., 2 days), block finalization or apply a conservative fallback (e.g., the minimum modulation). This mirrors the recommendation in the external report to be "punitive when pricing debt."

**Medium term:** Add a staleness check to `validate_exchange_rate` — reject rates whose `timestamp` is older than a defined maximum age (e.g., 2 × the XRC polling interval). This prevents LOCF from silently propagating stale rates across multiple days.

**Long term:** Consider using the spot ICP/XDR rate (or a shorter moving average, e.g., 1-day) for the "current price" signal in `compute_maturity_modulation_permyriad`, reserving the 7-day average only for the reference price. This reduces the lag without eliminating the stabilizing intent of the mechanism.

### Proof of Concept

**Setup:** ICP price has been stable at 5 XDR for 365 days. Modulation = 0.

**Day 0:** ICP price drops 20% to 4 XDR.

**Day 1 modulation computation:**
- 7-day average = (6 × 5 + 1 × 4) / 7 = **4.857 XDR**
- 365-day average = (364 × 5 + 1 × 4) / 365 = **4.997 XDR**
- Target = 2500 × (4.857 − 4.997) / 4.997 ≈ **−70 permyriad**
- Speed-limited from 0: actual modulation = **−30 permyriad (−0.3%)**

**Warranted modulation** (if spot price were used): 2500 × (4 − 5) / 5 = **−500 permyriad (−5%)**

**Gap on Day 1:** modulation is −0.3% instead of −5% → **4.7 percentage points of excess ICP minting**.

A neuron holder with 1,000,000 ICP-equivalent maturity who initiates `disburse_maturity` on Day 0 and finalizes on Day 7 receives approximately **47,000 ICP more** than the current price warrants, minted from thin air.

The attacker-controlled entry path is:
1. Call `manage_neuron { DisburseMaturity { percentage_to_disburse: 100, ... } }` on the NNS Governance canister (unprivileged ingress).
2. Wait 7 days for `try_finalize_maturity_disbursement` to execute with the still-lagged modulation.
3. Receive excess ICP, sell at current market price. [9](#0-8) [10](#0-9)

/// Compute the average `xdr_permyriad_per_icp` over the most recent `window_days` days ending
/// at `current_day` (exclusive of `current_day - window_days`, inclusive of `current_day`).
///
/// For each day in the window, uses that day's rate if present, otherwise falls back to the most
/// recent prior day's rate (Last Observation Carried Forward). This keeps the average meaningful
/// even when XRC fails to return a rate for one or more days. The fallback is computation-only;
/// nothing is written back to `rates`.
///
/// Returns `None` only when LOCF never finds a value to carry — i.e., no rate in the buffer has
/// a timestamp at or before `current_day` (so every day in the window is skipped). If a rate
/// appears partway into the window, leading days that precede it are skipped and the average is
/// computed over the days that do have a value.
pub(crate) fn compute_average_icp_xdr_rate(
    rates: &[SampledPrice],
    current_day: u64,
    window_days: usize,
) -> Option<u64> {
    if window_days == 0 {
        return None;
    }
    let oldest_day_in_window = current_day
        .saturating_sub(window_days as u64)
        .saturating_add(1);

    // Single linear pass: walk days and rates together, carrying the latest seen value forward.
    // For each day, advance through every rate at or before that day so `current_value` holds
    // the LOCF value when we sum. Days iterated before the first rate appears contribute
    // nothing (LOCF has no prior to carry forward).
    let mut rate_idx = 0;
    let mut current_value: Option<u64> = None;
    let mut sum: u128 = 0;
    let mut count: u64 = 0;
    for day in oldest_day_in_window..=current_day {
        let midnight = day * ONE_DAY_SECONDS;
        while rate_idx < rates.len() && rates[rate_idx].timestamp_seconds <= midnight {
            current_value = Some(rates[rate_idx].xdr_permyriad_per_icp);
            rate_idx += 1;
        }
        if let Some(v) = current_value {
            sum += v as u128;
            count += 1;
        }
    }

    if (count as usize) < window_days {
        println!(
            "{}compute_average_icp_xdr_rate: only {} of {} days have a rate available \
             (current_day={})",
            LOG_PREFIX, count, window_days, current_day
        );
    }
    if count == 0 {
        return None;
    }
    Some((sum / count as u128) as u64)

fn compute_maturity_modulation_permyriad(
    rates: &[SampledPrice],
    current_day: u64,
    previous: Option<(i64, u64)>,
) -> Result<i64, String> {
    let recent_icp_price = compute_average_icp_xdr_rate(
        rates,
        current_day,
        MATURITY_MODULATION_CURRENT_ICP_PRICE_WINDOW_DAYS,
    )
    .ok_or_else(|| "no rate available for the recent price window".to_string())?;

    let reference_icp_price = compute_average_icp_xdr_rate(
        rates,
        current_day,
        MATURITY_MODULATION_REFERENCE_ICP_PRICE_WINDOW_DAYS,
    )
    .ok_or_else(|| "no rate available for the reference price window".to_string())?;

    if reference_icp_price == 0 {
        return Err("reference price averaged to zero".to_string());
    }

    let target_modulation = {
        let recent = recent_icp_price as i128;
        let reference = reference_icp_price as i128;
        let sensitivity = MATURITY_MODULATION_SENSITIVITY_PERMYRIAD as i128;
        sensitivity * (recent - reference) / reference
    };
