### Title
SNS Treasury Manager Deposit Lacks Minimum-Output Slippage Guard at Execution Time — (`rs/sns/governance/src/extensions.rs`, `rs/sns/treasury_manager/treasury_manager.did`)

---

### Summary

The SNS Treasury Manager deposit flow (`execute_treasury_manager_deposit`) validates the deposit amount only at **proposal submission time** (checking that the requested amount does not exceed 50% of the current treasury balance), but provides **no minimum-output or price-ratio guarantee** at the time the deposit is actually executed against the DEX. The `treasury_manager.did` interface itself explicitly acknowledges this risk. Because SNS governance proposals can take days to pass, the DEX pool state at execution time can differ materially from the state at proposal creation time, and no on-chain slippage floor is enforced anywhere in the IC-side execution path.

---

### Finding Description

The deposit flow is:

1. An SNS governance proposal of type `ExecuteExtensionOperation` with `operation_name = "deposit"` is submitted, specifying `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s`.
2. At proposal **submission**, `validate_deposit_operation_impl` checks that each requested amount does not exceed 50% of the current treasury balance.
3. After the proposal passes (potentially days later), `execute_treasury_manager_deposit` is called. It calls `approve_treasury_manager` to set ICRC-2 allowances on both the SNS and ICP ledgers, then calls `deposit` on the Treasury Manager canister.
4. The Treasury Manager canister calls the DEX with the approved tokens.

The critical gap: **no minimum-output amount is specified or enforced anywhere in the IC-side code**. The `treasury_manager.did` file explicitly documents this as a known risk:

> "Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."

The `validate_deposit_operation_impl` function only checks that the *input* amount is ≤ 50% of treasury balance — it does not capture or enforce any expected output ratio or minimum LP tokens to receive. [1](#0-0) 

The `execute_treasury_manager_deposit` function passes the original proposal argument blob directly to the Treasury Manager's `deposit` endpoint with no additional slippage parameters added by governance. [2](#0-1) 

The `approve_treasury_manager` function sets ICRC-2 allowances with a 1-hour expiry but no price constraint. [3](#0-2) 

The `treasury_manager.did` `DepositRequest` type contains only `allowances` — there is no `min_lp_tokens_out` or equivalent field in the interface. [4](#0-3) 

The proposal rendering code in `validate_and_render_register_extension` acknowledges the sandwich risk in a warning string but does not enforce any mitigation. [5](#0-4) 

---

### Impact Explanation

An adversary who can observe the IC message queue (or simply time their actions to coincide with proposal execution) can manipulate the DEX pool state immediately before the Treasury Manager's `deposit` call executes. Because no minimum-output is enforced, the Treasury Manager will accept whatever LP tokens the DEX returns — potentially near zero. The SNS treasury loses the deposited SNS tokens and ICP while receiving negligible LP position value. The loss is permanent and proportional to the deposit size (up to 50% of treasury per proposal). Multiple proposals could be chained to drain the treasury further.

---

### Likelihood Explanation

IC does not have a public mempool in the Ethereum sense, but proposal execution timing is **publicly observable** via the NNS/SNS governance canister state. An attacker can query when a proposal is about to be executed (e.g., by monitoring `executed_timestamp_seconds` becoming non-zero, or by watching the voting tally approach the threshold). On IC, canister-to-canister calls within a round are ordered deterministically, but the DEX pool state can be manipulated by any canister caller in a prior round. A well-resourced attacker can front-run the deposit by submitting a large trade to the DEX pool in the round immediately before governance executes the deposit proposal. This is a realistic, low-barrier attack requiring only the ability to call the DEX canister — no privileged access needed.

---

### Recommendation

1. **Add a `min_lp_tokens_out` (or equivalent minimum-output) field to `DepositRequest`** in `treasury_manager.did` and require Treasury Manager implementations to enforce it when calling the DEX.
2. **Capture the expected price ratio at proposal submission time** (inside `validate_deposit_operation_impl`) and embed it in the validated argument, then re-check it at execution time in `execute_treasury_manager_deposit`.
3. **Require Treasury Manager implementations to reject deposits where the received LP tokens fall below the caller-specified minimum**, analogous to the `minimum-amount-out` pattern in AMM interfaces.
4. **Document the execution-time slippage window** as a known risk in the SNS governance proposal flow, and consider adding a maximum staleness check (e.g., reject execution if more than N days have elapsed since proposal submission).

---

### Proof of Concept

1. Alice's SNS submits a `TreasuryManagerDeposit` proposal for 1,000,000 SNS tokens + 500 ICP into a DEX liquidity pool. At submission, the pool price is fair and the 50% balance check passes.
2. The proposal enters the voting period (e.g., 4 days).
3. Eve monitors the SNS governance canister. When the proposal's vote tally crosses the adoption threshold, Eve submits a large trade to the DEX pool in the same or immediately preceding IC round, moving the pool price significantly against the SNS/ICP pair.
4. The proposal executes: `execute_treasury_manager_deposit` calls `approve_treasury_manager` (setting ICRC-2 allowances), then calls `deposit` on the Treasury Manager.
5. The Treasury Manager calls the DEX with the approved tokens. Because no `min_lp_tokens_out` is specified, the DEX accepts the deposit at the manipulated price and returns a tiny LP position.
6. Eve reverses her trade (or profits from arbitrage), extracting value from the SNS treasury. The SNS treasury has lost up to 50% of its SNS tokens and ICP in exchange for near-worthless LP tokens.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
