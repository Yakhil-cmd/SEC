### Title
`NeuronInfo.stake_e8s` Omits `staked_maturity_e8s_equivalent`, Contradicting Its Own Documentation - (File: `rs/nns/governance/src/neuron/types.rs`)

### Summary

The `get_neuron_info` function in the NNS governance canister populates the `stake_e8s` field of `NeuronInfo` using `self.minted_stake_e8s()`, which excludes `staked_maturity_e8s_equivalent`. The canister's own Candid interface documentation explicitly states that `stake_e8s` must equal `cached_neuron_stake_e8s - neuron_fees_e8s + staked_maturity_e8s_equivalent`. The omission is structurally identical to M-6: a value-reporting function silently drops one addend, causing every consumer of the query result to undercount the neuron's true economic stake.

### Finding Description

`get_neuron_info` is a publicly callable query method. It constructs a `NeuronInfo` struct and sets:

```rust
stake_e8s: self.minted_stake_e8s(),
``` [1](#0-0) 

`minted_stake_e8s` is defined as:

```rust
pub fn minted_stake_e8s(&self) -> u64 {
    self.cached_neuron_stake_e8s
        .saturating_sub(self.neuron_fees_e8s)
}
``` [2](#0-1) 

The Candid interface documentation for `NeuronInfo.stake_e8s` explicitly states the correct formula:

```
// The amount of ICP (and staked maturity) locked in this neuron.
// This is the foundation of the neuron's voting power.
// cached_neuron_stake_e8s - neuron_fees_e8s + staked_maturity_e8s_equivalent
stake_e8s : nat64;
``` [3](#0-2) 

The correct function that matches the documented formula is `stake_e8s()`, not `minted_stake_e8s()`:

```rust
pub fn stake_e8s(&self) -> u64 {
    neuron_stake_e8s(
        self.cached_neuron_stake_e8s,
        self.neuron_fees_e8s,
        self.staked_maturity_e8s_equivalent,
    )
}
``` [4](#0-3) 

where `neuron_stake_e8s` adds `staked_maturity_e8s_equivalent`:

```rust
fn neuron_stake_e8s(
    cached_neuron_stake_e8s: u64,
    neuron_fees_e8s: u64,
    staked_maturity_e8s_equivalent: Option<u64>,
) -> u64 {
    cached_neuron_stake_e8s
        .saturating_sub(neuron_fees_e8s)
        .saturating_add(staked_maturity_e8s_equivalent.unwrap_or(0))
}
``` [5](#0-4) 

The on-chain voting power calculation itself correctly uses `self.stake_e8s()` (including staked maturity):

```rust
let stake_e8s = self.stake_e8s();
``` [6](#0-5) 

This creates a direct contradiction: the voting power reported in `NeuronInfo.voting_power` / `deciding_voting_power` is computed from a stake that includes staked maturity, but `NeuronInfo.stake_e8s` — documented as "the foundation of the neuron's voting power" — excludes it. Any caller that tries to reproduce or verify the voting power from the reported `stake_e8s` will get a wrong answer.

The same omission propagates into `compute_neuron_metrics`, where `total_staked_e8s` is accumulated using `minted_stake_e8s()` rather than `stake_e8s()`:

```rust
metrics.total_staked_e8s = metrics
    .total_staked_e8s
    .saturating_add(neuron.minted_stake_e8s());
``` [7](#0-6) 

### Impact Explanation

Any unprivileged query caller — an off-chain dashboard, a third-party canister, or a governance integration — that reads `NeuronInfo.stake_e8s` and treats it as the neuron's total locked economic value (as the documentation instructs) will systematically undercount by the full `staked_maturity_e8s_equivalent` amount. For neurons that have auto-staked large maturity balances over years, this gap can be substantial. Downstream logic that gates decisions on the reported stake (e.g., minimum-stake checks, collateral calculations, or Neurons' Fund eligibility assessments performed off-chain) will operate on a deflated figure. The `GovernanceCachedMetrics.total_staked_e8s` metric exposed to the public is similarly understated, misrepresenting the total ICP economically committed to governance.

### Likelihood Explanation

The vulnerability is reachable by any caller of the `get_neuron_info` query or the `get_metrics` query — no privilege is required. Staked maturity is a normal, widely-used feature (neurons with `auto_stake_maturity` accumulate it continuously). The discrepancy between the documented formula and the implementation is silent: callers following the Candid documentation will write code that is wrong without any error signal.

### Recommendation

Replace `self.minted_stake_e8s()` with `self.stake_e8s()` in `get_neuron_info` so that the returned `stake_e8s` matches the documented formula and the value actually used in voting power calculations:

```rust
// Before (incorrect):
stake_e8s: self.minted_stake_e8s(),

// After (correct):
stake_e8s: self.stake_e8s(),
```

Apply the same correction to `compute_neuron_metrics` so that `total_staked_e8s` reflects the true total economic stake including staked maturity.

### Proof of Concept

1. Create a neuron with `cached_neuron_stake_e8s = 100_000_000` (1 ICP) and `staked_maturity_e8s_equivalent = 50_000_000` (0.5 ICP equivalent).
2. Call `get_neuron_info` on the NNS governance canister.
3. Observe `NeuronInfo.stake_e8s = 100_000_000` (only minted stake).
4. The documentation states the value should be `150_000_000`.
5. Observe `NeuronInfo.deciding_voting_power` is computed from `150_000_000` (correct on-chain value).
6. The reported `stake_e8s` and the actual stake used for voting power are inconsistent by exactly `staked_maturity_e8s_equivalent`.
