### Title
Unbounded `ingress_messages` Allocation in `TryFrom<pb::StrippedBlockProposal>` Enables Byzantine-Peer OOM DoS — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

### Summary

When `HASHES_IN_BLOCKS_ENABLED` is `true`, a Byzantine peer below the fault threshold can send a crafted `StrippedBlockProposal` protobuf with an arbitrarily large `ingress_messages` repeated field. The `TryFrom<pb::StrippedBlockProposal>` implementation performs no length cap before allocating a `Vec<SignedIngressId>` proportional to attacker-controlled input. The subsequent `BlockProposalAssembler::new` mirrors that allocation, and `assemble_message` spawns one `get_or_fetch` Tokio task per entry into a `JoinSet`, causing unbounded memory and task-handle growth on the receiving replica.

### Finding Description

`HASHES_IN_BLOCKS_ENABLED` is unconditionally `true` in production: [1](#0-0) 

The `TryFrom<pb::StrippedBlockProposal>` implementation at lines 100–108 iterates `value.ingress_messages` and collects into a `Vec<SignedIngressId>` with **no length check**: [2](#0-1) 

The developer comment in `try_insert` acknowledges the implicit assumption of ≤1000 elements, but this is never enforced at deserialization time: [3](#0-2) 

`BlockProposalAssembler::new` then mirrors the allocation into a second `Vec<(SignedIngressId, Option<SignedIngress>)>`: [4](#0-3) 

`assemble_message` then calls `missing_stripped_messages()` and spawns one `get_or_fetch` task per entry into a `JoinSet`: [5](#0-4) 

The `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` limit exists only in consensus payload validation, which runs **after** assembly — not during deserialization: [6](#0-5) [7](#0-6) 

The P2P RPC endpoint explicitly disables body size limits: [8](#0-7) 

The `id()` of a `StrippedBlockProposal` is derived solely from `unstripped_consensus_message_id`, which the attacker controls: [9](#0-8) 

This means the `message.id() == id` check in `FetchArtifact::download_artifact` passes for a crafted artifact whose `unstripped_consensus_message_id` is set to any block proposal ID the bouncer considers wanted. [10](#0-9) 

### Impact Explanation

With 10^6 entries per `StrippedIngressMessage` (~100 bytes each on wire), the attack sends ~100 MB. On the receiving replica:
- `Vec<SignedIngressId>` allocation: ~10^6 × ~100 bytes ≈ 100 MB heap
- `Vec<(SignedIngressId, Option<SignedIngress>)>`: another ~100 MB
- 10^6 Tokio task handles in `JoinSet`: ~1 KB overhead each ≈ ~1 GB

This causes OOM or severe memory pressure on a single replica. The `try_assemble()` hash mismatch check fires only after all tasks complete, by which time the damage is done. [11](#0-10) 

### Likelihood Explanation

- `HASHES_IN_BLOCKS_ENABLED = true` is hardcoded in production.
- A single Byzantine peer (below the f-threshold) can execute this attack.
- The attacker only needs to advertise a `StrippedConsensusMessageId` matching any block proposal ID the bouncer considers wanted at the current height — observable from the public P2P network.
- The P2P RPC endpoint has no body size limit, so the 100 MB payload is accepted.
- No privileged access, key material, or majority corruption is required.

### Recommendation

Add a length cap in `TryFrom<pb::StrippedBlockProposal>` before the `collect()`:

```rust
const MAX_INGRESS_MESSAGES: usize = 1000; // matches MAX_INGRESS_MESSAGES_PER_BLOCK
if value.ingress_messages.len() > MAX_INGRESS_MESSAGES {
    return Err(ProxyDecodeError::Other(format!(
        "ingress_messages length {} exceeds maximum {}",
        value.ingress_messages.len(), MAX_INGRESS_MESSAGES
    )));
}
```

Apply the same cap to `stripped_idkg_dealings`. This mirrors the existing consensus-layer limit and prevents unbounded allocation before any consensus validation occurs. [12](#0-11) 

### Proof of Concept

1. Observe the P2P network to find a `StrippedConsensusMessageId` the target replica's bouncer considers `Wants` (any current-height block proposal hash).
2. Craft a `pb::StrippedConsensusMessage` with `msg = StrippedBlockProposal { pruned_block_proposal: <valid empty proto>, unstripped_consensus_message_id: <observed ID>, ingress_messages: [<repeated 10^6 StrippedIngressMessage entries>], stripped_idkg_dealings: [] }`.
3. Send this via the P2P transport RPC endpoint (no body size limit).
4. The receiving replica deserializes, passes the bouncer check, allocates ~1 GB of task state, and enters OOM.
5. Assert: RSS grows proportionally to `ingress_messages.len()` with no cap enforced at lines 100–108 of `stripped.rs`.
