### Title
Missing `charge_for_cpu()` in `ic0.mint_cycles128` Host Function Registration - (File: rs/embedders/src/wasmtime_embedder/linker.rs)

---

### Summary

The `ic0.mint_cycles128` host function is registered in the Wasmtime linker **without** a `charge_for_cpu()` call, unlike every other `ic0.*` system API function. This means a canister can invoke `ic0.mint_cycles128` in a tight loop and pay only the WASM instruction cost of the loop itself, not the per-call host function overhead that all other system API calls incur.

---

### Finding Description

In `rs/embedders/src/wasmtime_embedder/linker.rs`, every `ic0.*` host function charges for its CPU overhead via `charge_for_cpu()` or `charge_for_cpu_and_mem()` before executing. For example, `cycles_burn128` — the function immediately adjacent to `mint_cycles128` — correctly charges overhead: [1](#0-0) 

However, `mint_cycles128` omits this charge entirely: [2](#0-1) 

The `charge_for_cpu` / `charge_for_cpu_and_mem` helpers are the mechanism by which the IC accounts for the overhead of crossing the WASM-to-host boundary and executing host-side logic: [3](#0-2) 

All other `ic0.*` functions — `msg_reply`, `call_cycles_add`, `canister_cycle_balance128`, `msg_cycles_accept128`, `cycles_burn128`, `time`, `debug_print`, etc. — consistently call `charge_for_cpu` before dispatching: [4](#0-3) [5](#0-4) 

The WASM-level instruction metering (injected by `inject_metering`) only charges for WASM opcodes — it does not account for host function overhead: [6](#0-5) 

So a canister calling `ic0.mint_cycles128` in a loop pays only 1 instruction for the WASM `call` opcode, not the full overhead that `charge_for_cpu(overhead::MINT_CYCLES128)` would impose.

---

### Impact Explanation

A canister can call `ic0.mint_cycles128` in a tight loop for the duration of its instruction limit. Each call crosses the WASM sandbox boundary and executes host-side Rust code (`ic0_mint_cycles128` in `system_api.rs`) without paying the corresponding instruction overhead: [7](#0-6) 

For non-CMC canisters the call returns an error immediately, but the host-side work (API type dispatch, error construction, trace logging) is still performed on every iteration. The attacker pays only WASM loop instructions, not the host overhead, allowing them to perform far more host-boundary crossings per message execution than the instruction limit is designed to permit. This drains replica CPU disproportionately relative to the cycles charged, slowing block execution across the subnet.

---

### Likelihood Explanation

Any canister developer can deploy a canister that imports and calls `ic0.mint_cycles128` in a loop. No privileged access is required. The canister will be rejected by the system API for minting (non-CMC), but the unmetered host invocations still occur. The entry path is a standard ingress update call to the attacker's canister.

---

### Recommendation

Add `charge_for_cpu(&mut caller, overhead::MINT_CYCLES128)?;` as the first statement inside the `mint_cycles128` closure, consistent with every other `ic0.*` host function registration in the same file:

```rust
linker
    .func_wrap("ic0", "mint_cycles128", {
        move |mut caller: Caller<'_, StoreData>, amount_high: u64, amount_low: u64, dst: I| {
            charge_for_cpu(&mut caller, overhead::MINT_CYCLES128)?;  // ADD THIS
            with_memory_and_system_api(&mut caller, |s, memory| {
                let dst: usize = dst.try_into().expect("Failed to convert I to usize");
                s.ic0_mint_cycles128(Cycles::from_parts(amount_high, amount_low), dst, memory)
            })
            .map_err(|e| wasmtime::Error::msg(format!("ic0_mint_cycles128 failed: {e}")))
        }
    })
    .unwrap();
```

A corresponding `MINT_CYCLES128` constant must be added to `system_api_complexity.rs` with an appropriate overhead value, consistent with similar functions like `CYCLES_BURN128`.

---

### Proof of Concept

```wat
(module
  (import "ic0" "mint_cycles128"
    (func $mint_cycles128 (param i64 i64 i32)))
  (memory 1)
  (func (export "canister_update exploit")
    (local $i i32)
    (local.set $i (i32.const 0))
    (block $break
      (loop $loop
        ;; call mint_cycles128 — pays only 1 WASM instruction for `call`,
        ;; not the host-function overhead that charge_for_cpu would impose
        (call $mint_cycles128 (i64.const 0) (i64.const 1) (i32.const 0))
        (local.set $i (i32.add (local.get $i) (i32.const 1)))
        (br_if $loop (i32.lt_s (local.get $i) (i32.const 2000000000)))
      )
    )
  )
)
```

This canister, when called via ingress, will exhaust its instruction limit performing far more host-boundary crossings than the instruction budget is designed to allow, consuming disproportionate replica CPU per message.

/// Charge for system api call that doesn't involve touching memory
#[inline(always)]
fn charge_for_cpu(
    caller: &mut Caller<'_, StoreData>,
    overhead: NumInstructions,
) -> Result<(), wasmtime::Error> {
    charge_for_system_api_call(caller, overhead, 0).map_err(|e| process_err(caller, e))
}

/// Charge for system api call that involves writing/reading heap

    num_bytes: usize,
) -> Result<(), wasmtime::Error> {

}
