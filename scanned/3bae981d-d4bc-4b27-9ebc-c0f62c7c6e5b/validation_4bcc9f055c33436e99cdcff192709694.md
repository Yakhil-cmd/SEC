### Title
Stale Maturity Modulation Applied to ICP Minting Without Freshness Check — (`rs/nns/governance/src/governance.rs`, `rs/nns/governance/src/governance/disburse_maturity.rs`)

---

### Summary

`maybe_spawn_neurons()` and `try_finalize_maturity_disbursement()` in the NNS Governance canister read `heap_data.maturity_modulation.current_value_permyriad` to determine how much ICP to mint, but neither function checks `updated_at_days_since_epoch` to verify the cached value is fresh. If the `UpdateIcpXdrRateRelatedData` timer task fails to update the modulation for an extended period (e.g., due to persistent XRC unavailability), the stale modulation factor is silently applied to all ICP minting operations, causing incorrect ledger conservation.

---

### Finding Description

The `MaturityModulation` protobuf message stores two fields:
- `current_value_permyriad`: the cached modulation factor applied to ICP minting
- `updated_at_days_since_epoch`: the day on which the value was last computed [1](#0-0) 

The `UpdateIcpXdrRateRelatedData` recurring timer task is the sole mechanism that refreshes this value. It fetches daily ICP/XDR rates from the Exchange Rate Canister (XRC), maintains a 365-day price history, and recomputes the modulation once per day. [2](#0-1) 

However, both consumers of the modulation value read only `current_value_permyriad` and ignore `updated_at_days_since_epoch`:

**`maybe_spawn_neurons()`** — reads the cached modulation with no freshness check: [3](#0-2) 

**`try_finalize_maturity_disbursement()`** — same pattern: [4](#0-3) 

The `updated_at_days_since_epoch` field is only consulted by the timer task itself (to avoid recomputing on the same day) and is never checked by the minting paths to determine whether the cached value is still valid. [5](#0-4) 

Additionally, the canister is initialized with `current_value_permyriad: Some(0)` and `updated_at_days_since_epoch: None` by design, so that minting operations work immediately before the price history is populated. This means the system is explicitly designed to use a potentially stale (or never-computed) modulation value rather than block operations. [6](#0-5) 

If the XRC is persistently unavailable, `fetch_and_validate_rate()` returns `None` on every attempt, the timer retries at `ERROR_RETRY_INTERVAL_SECONDS` (60 s), and `maturity_modulation` is never updated. The stale `current_value_permyriad` from the last successful computation is then applied indefinitely to all neuron spawning and maturity disbursement finalization. [7](#0-6) 

---

### Impact Explanation

The maturity modulation factor is applied multiplicatively to the ICP amount minted when a neuron owner disburses maturity or spawns a neuron:

```
ICP_minted = maturity * (1 + current_value_permyriad / 10_000)
``` [8](#0-7) 

If the modulation is stale at a favorable value (e.g., +500 permyriad = +5%), all disbursements during the outage mint 5% more ICP than the current market conditions warrant — a ledger conservation violation. If stale at an unfavorable value (−5%), users receive less ICP than they should. The global bounds clamp the modulation to `[MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70, MATURITY_MODULATION_MAX_PERMYRIAD_MISSION_70]`, so the maximum per-disbursement deviation is ±5%. [9](#0-8) 

---

### Likelihood Explanation

The XRC canister is a system canister on the IC. Persistent unavailability can occur due to subnet issues, canister upgrades, or other infrastructure events. The timer task has no fallback mechanism — if `fetch_and_validate_rate()` fails, the modulation is simply not updated and the stale value persists. The CHANGELOG confirms this was a known failure mode that was partially addressed (tolerating gaps via LOCF), but the consumer-side freshness check was never added. [10](#0-9) 

Any neuron owner (unprivileged ingress sender) who initiates a `DisburseMaturity` or `SpawnNeuron` call during a period of XRC unavailability will have the stale modulation applied at finalization time (7 days later), with no ability to detect or prevent this.

---

### Recommendation

Before applying `current_value_permyriad` in `maybe_spawn_neurons()` and `try_finalize_maturity_disbursement()`, check that `updated_at_days_since_epoch` is recent (e.g., within the last N days). If the modulation is too stale, either:
1. Block the minting operation and return an appropriate error, or
2. Fall back to a neutral modulation of 0 permyriad.

A concrete guard in `try_finalize_maturity_disbursement()`:

```rust
let maturity_modulation = governance.heap_data.maturity_modulation.as_ref();
let current_day = governance.env.now() / ONE_DAY_SECONDS;
let modulation_value = maturity_modulation
    .filter(|mm| {
        mm.updated_at_days_since_epoch
            .map(|d| current_day.saturating_sub(d) <= MAX_STALE_DAYS)
            .unwrap_or(false)
    })
    .and_then(|mm| mm.current_value_permyriad);
```

---

### Proof of Concept

1. XRC canister becomes unavailable (subnet issue, upgrade, etc.).
2. `UpdateIcpXdrRateRelatedData::execute()` calls `fetch_and_validate_rate()`, which returns `None` on every attempt.
3. `maturity_modulation.updated_at_days_since_epoch` remains at day D (the last successful update), while the current day advances to D+30.
4. A neuron owner calls `DisburseMaturity` with 100 ICP worth of maturity.
5. After 7 days, `try_finalize_maturity_disbursement()` reads `current_value_permyriad = Some(500)` (stale +5% from 30 days ago).
6. `apply_maturity_modulation(100_0000_0000, 500)` mints 105 ICP instead of the correct amount based on current market conditions.
7. The protocol has minted 5 ICP more than it should have, with no record of the staleness. [11](#0-10)

fn update_maturity_modulation(
    icp_price_history: &IcpPriceHistory,
    maturity_modulation: &mut MaturityModulation,
    current_day: u64,
) {
    if maturity_modulation.updated_at_days_since_epoch == Some(current_day) {
        return;
    }

    let previous = match (
        maturity_modulation.current_value_permyriad,
        maturity_modulation.updated_at_days_since_epoch,
    ) {
        (Some(p), Some(d)) => Some((p as i64, d)),
        _ => None,
    };

    match compute_maturity_modulation_permyriad(
        &icp_price_history.icp_xdr_rates,
        current_day,
        previous,
    ) {
        Ok(new_permyriad) => {
            maturity_modulation.current_value_permyriad = Some(new_permyriad as i32);
            maturity_modulation.updated_at_days_since_epoch = Some(current_day);
        }
        Err(reason) => {
            // Reaches this branch only when the buffer has no rate at or before any day in the
            // recent window (e.g., a fresh canister where every backfill fetch has failed so far,
            // or every fetched rate was zero). Log and leave the prior modulation untouched —
            // subsequent rounds will retry the missing days.
            println!(
                "{}update_maturity_modulation: skipping update: {}; leaving prior modulation \
                 unchanged",
                LOG_PREFIX, reason
            );
        }
    }
}

// The maturity modulation factor is applied when disbursing (unstaked) maturity to ICP.
//
// When a neuron owner disburses maturity, the amount of ICP received is:
//   maturity * (1 + current_value_permyriad / 10_000)
//
// This factor stabilizes ICP price: it is positive when ICP is above its long-term average
// (encouraging selling pressure), and negative when below (discouraging selling).
//
// This might be unpopulated, which indicates that no value is currently available.
message MaturityModulation {
  // Current maturity modulation in permyriad (0.01% per unit).
  optional int32 current_value_permyriad = 1;

  // Day (days_since_epoch) when current_value_permyriad was last computed.
  optional uint64 updated_at_days_since_epoch = 2;
}
