### Title
Unchecked Index Panic on Empty `EnvelopePair` Vec in `construction_parse` Crashes Rosetta Process - (`rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

---

### Summary

An unauthenticated attacker can POST a crafted CBOR-encoded `SignedTransaction` containing a `(RequestType, Vec<EnvelopePair>)` tuple where the `Vec<EnvelopePair>` is empty to the `/construction/parse` endpoint. This triggers an unchecked `updates[0]` index access on an empty slice, causing a Rust panic that crashes the Rosetta process.

---

### Finding Description

In `construction_parse`, when handling a signed transaction, the code iterates over `signed_transaction.requests` and unconditionally indexes into the inner `Vec<EnvelopePair>` at position `[0]`: [1](#0-0) 

```rust
ParsedTransaction::Signed(signed_transaction) => signed_transaction
    .requests
    .iter()
    .map(
        |(request_type, updates)| match updates[0].update.content.clone() {
            HttpCallContent::Call { update } => (request_type.clone(), update),
        },
    )
    .collect(),
```

The `updates[0]` at line 42 is an unchecked slice index. If `updates` is empty, Rust panics with an index-out-of-bounds.

The deserialization path (`ParsedTransaction::try_from`) performs no structural validation — it only calls `serde_cbor::from_slice`: [2](#0-1) 

`serde_cbor` will successfully deserialize a `SignedTransaction` whose `requests` field contains a tuple with an empty `Vec<EnvelopePair>`, because the type definition imposes no minimum-length constraint: [3](#0-2) 

```rust
pub type Request = (RequestType, Vec<EnvelopePair>);
```

---

### Impact Explanation

Rosetta runs as a single OS process. A Rust `panic!` (from an out-of-bounds slice index) is unrecoverable in the default configuration and terminates the process. Any client that can reach the Rosetta HTTP endpoint can crash it with a single malformed request, causing complete availability loss until the process is restarted.

---

### Likelihood Explanation

The Rosetta HTTP API is intentionally public-facing (it is the Rosetta standard API). No authentication is required to POST to `/construction/parse`. The crafted payload is trivial to construct: a CBOR-encoded struct with one request entry containing an empty envelope list. The panic is deterministic and reproducible on every invocation.

---

### Recommendation

Add an explicit guard before the `updates[0]` access. Return an `ApiError::invalid_request` if the inner vec is empty:

```rust
.map(|(request_type, updates)| {
    let first = updates.first().ok_or_else(|| {
        ApiError::invalid_request("SignedTransaction request has no envelope pairs")
    })?;
    match first.update.content.clone() {
        HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
    }
})
.collect::<Result<Vec<_>, _>>()?
```

---

### Proof of Concept

```rust
// Construct a SignedTransaction with one request and zero EnvelopePairs
let signed_tx = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],  // empty Vec<EnvelopePair>
};
let cbor_bytes = serde_cbor::to_vec(&signed_tx).unwrap();
let hex_tx = hex::encode(&cbor_bytes);

// POST to /construction/parse with signed=true
let req = ConstructionParseRequest {
    network_identifier: /* valid network id */,
    signed: true,
    transaction: hex_tx,
};

// This panics at updates[0] in construction_parse.rs:42
handler.construction_parse(req);
// => thread 'main' panicked at 'index out of bounds: the len is 0 but the index is 0'
``` [4](#0-3)
