### Title
SNS Swap Auto-Finalization Permanently Blocked by Any Transient External Canister Error - (File: rs/sns/swap/src/swap.rs)

### Summary

`try_auto_finalize` in the SNS swap canister unconditionally sets `already_tried_to_auto_finalize = true` **before** invoking `finalize`, which chains through five or more external canister calls. If any one of those calls returns a transient error, the entire auto-finalization attempt is recorded as permanently exhausted. The swap can never be auto-finalized again, mirroring M-04's pattern where a broad error-catch in a chained settlement path causes the operation to fail permanently due to an unexpected error from an external component.

### Finding Description

`try_auto_finalize` in `rs/sns/swap/src/swap.rs` sets the one-shot guard flag before calling `finalize`:

```rust
self.already_tried_to_auto_finalize = Some(true);   // line 719
let auto_finalize_swap_response = self.finalize(now_fn, environment).await;  // line 722
```

`finalize` delegates to `finalize_inner`, which chains the following external canister calls in sequence, halting on the first error:

1. `sweep_icp` → ICP ledger (`transfer`)
2. `settle_neurons_fund_participation` → NNS governance
3. `sweep_sns` → SNS ledger (`transfer`)
4. `claim_swap_neurons` → SNS governance
5. `set_sns_governance_to_normal_mode` → SNS governance
6. `take_sole_control_of_dapp_controllers_for_finalize` → SNS root

`can_auto_finalize` permanently gates future attempts:

```rust
if self.already_tried_to_auto_finalize.unwrap_or(true) {
    return Err(format!(
        "self.already_tried_to_auto_finalize is {:?}, indicating that an attempt has already been made to auto-finalize. No further attempts will be made automatically. Manually calling finalize is still allowed.",
        self.already_tried_to_auto_finalize
    ));
}
```

The IC's `SysTransient` reject code (code 2xx) is the platform's standard signal for "retry later" — queue full, canister temporarily unavailable during upgrade, etc. None of the six external calls above distinguish transient from permanent errors before halting `finalize_inner`. Because `already_tried_to_auto_finalize` is already `true` when the transient error surfaces, `run_periodic_tasks` will never re-enter the auto-finalization branch again.

The log message at line 716 even acknowledges this explicitly:

> "Will not automatically attempt again even if this fails."

This is the direct IC analog to M-04's try-catch that conflates OOG errors with other empty errors: both patterns treat a transient, retryable failure from an external component as a permanent terminal condition for a critical settlement operation.

### Impact Explanation

When auto-finalization is triggered (the normal production path via `run_periodic_tasks`) and any one of the five-plus external canisters returns a transient error, the SNS swap is left in a committed-but-unfinalized state indefinitely. Participants' ICP contributions remain locked in the swap canister's subaccounts. SNS tokens remain undistributed. SNS governance remains in pre-initialization mode. The dapp controllers are not transferred. All of this persists until a human operator discovers the failure and manually calls `finalize_swap`. There is no on-chain alert or automatic recovery path.

### Likelihood Explanation

Transient errors from the ICP ledger, NNS governance, SNS ledger, SNS governance, or SNS root are realistic during routine canister upgrades, which happen on a regular cadence on the IC mainnet. A canister upgrade causes the canister to be temporarily stopped, and any in-flight call to it during that window receives a `SysTransient` rejection. Because `finalize_inner` calls five or more distinct canisters, the probability that at least one of them is mid-upgrade during the single auto-finalization window is non-trivial, especially for SNS launches that attract high traffic and coincide with NNS upgrade proposals. No attacker is required; the condition arises from ordinary platform operations.

### Recommendation

Move the `already_tried_to_auto_finalize = Some(true)` assignment to **after** `finalize` returns, and only set it to `true` when the response contains no error message (i.e., finalization fully succeeded). Alternatively, inspect the `FinalizeSwapResponse.error_message` and only set the flag permanently when the error is non-transient. This mirrors the correct pattern used in the ckETH minter's `Reason::from_call_failed`, which explicitly maps `SysTransient` to `TransientInternalError` and retries, rather than treating all errors identically.

### Proof of Concept

1. An SNS swap reaches `Lifecycle::Committed` with `should_auto_finalize = true`.
2. `run_periodic_tasks` fires and calls `try_auto_finalize`.
3. `try_auto_finalize` sets `already_tried_to_auto_finalize = Some(true)` at line 719.
4. `finalize` → `finalize_inner` calls `sweep_icp` successfully (ICP ledger responds).
5. `finalize_inner` calls `settle_neurons_fund_participation`, which calls NNS governance.
6. NNS governance is mid-upgrade; the replica returns `SysTransient` (reject code 2).
7. `settle_neurons_fund_participation` returns `SettleNeuronsFundParticipationResult::Err`.
8. `set_settle_neurons_fund_participation_result` sets `error_message` on `FinalizeSwapResponse`.
9. `finalize_inner` returns early; `finalize` returns the error response.
10. `try_auto_finalize` stores the failed response in `auto_finalize_swap_response`.
11. All future calls to `can_auto_finalize` return `Err` because `already_tried_to_auto_finalize` is `Some(true)`.
12. `run_periodic_tasks` never re-enters the auto-finalization branch.
13. Participants' ICP and SNS tokens remain locked; SNS governance stays in pre-initialization mode until a human manually calls `finalize_swap`.
