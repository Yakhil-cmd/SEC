### Title
Cache Key Collision Between QueryV2 and QueryV3 Allows Wrong-Version Response Serving — (`rs/boundary_node/ic_boundary/src/routes.rs`)

### Summary

The custom `Hash` and `PartialEq` implementations on `RequestContext` deliberately omit the `request_type` field. Both `QueryV2` and `QueryV3` pass the cache bypasser's `is_query()` guard. An unprivileged client can therefore cause the boundary node to serve a cached `QueryV2` response body to a `QueryV3` caller (or vice versa), violating the invariant that cache keys must uniquely identify the full request identity including API version.

### Finding Description

`RequestContext` is the cache key type used by the boundary-node query cache: [1](#0-0) 

The custom `Hash` implementation hashes only `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg`/`http_request` — **`request_type` is never fed into the hasher**: [2](#0-1) 

The custom `PartialEq` implementation compares the same subset of fields and also **omits `request_type`**: [3](#0-2) 

The cache bypasser only skips caching when `!ctx.request_type.is_query()`: [4](#0-3) 

And `is_query()` returns `true` for **both** `QueryV2` and `QueryV3`: [5](#0-4) 

This means two structurally distinct requests — one to `/api/v2/canister/{id}/query` and one to `/api/v3/canister/{id}/query` — with identical `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg` will produce the same hash value and compare as equal, causing a cache hit across API versions.

### Impact Explanation

`QueryV2` and `QueryV3` have different response formats: `QueryV3` returns a `CertifiedQueryResponse` containing a certificate that clients use to cryptographically verify the result. `QueryV2` returns an uncertified `QueryResponse`. When a `QueryV3` caller receives a cached `QueryV2` response:

- Clients that enforce certificate verification will receive a malformed/missing certificate and fail.
- Clients that do not strictly validate may silently accept an uncertified response as if it were certified, undermining the integrity guarantee that `QueryV3` is designed to provide.

The collision is also symmetric: a `QueryV2` caller could receive a `QueryV3` response body, causing CBOR parse failures.

### Likelihood Explanation

The precondition is simply that the boundary node has `cache_size > 0` (the default production configuration). No privilege is required. Any two clients — or a single client — making the same logical query over both API versions triggers the collision deterministically. The path is fully local and testable.

### Recommendation

Include `request_type` in both the `Hash` and `PartialEq` implementations of `RequestContext`:

```rust
impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.request_type.hash(state); // ADD THIS
        self.canister_id.hash(state);
        // ...
    }
}

impl PartialEq for RequestContext {
    fn eq(&self, other: &Self) -> bool {
        self.request_type == other.request_type // ADD THIS
        && self.canister_id == other.canister_id
        // ...
    }
}
```

The comment at line 91–93 states "They should both work on the same fields" — `request_type` must be one of those fields.

### Proof of Concept

```rust
use std::collections::hash_map::DefaultHasher;
use std::hash::{Hash, Hasher};

fn hash_of(ctx: &RequestContext) -> u64 {
    let mut h = DefaultHasher::new();
    ctx.hash(&mut h);
    h.finish()
}

let a = RequestContext { request_type: RequestType::QueryV2, canister_id: Some(p), sender: Some(s), method_name: Some("m".into()), ingress_expiry: Some(0), arg: Some(vec![1,2,3]), ..Default::default() };
let b = RequestContext { request_type: RequestType::QueryV3, ..a.clone() };

assert_eq!(hash_of(&a), hash_of(&b)); // passes — collision confirmed
assert_eq!(a, b);                      // passes — equality confirmed
```

An integration test sending a `QueryV2` request first (populating the cache) and then a `QueryV3` request with identical parameters will observe the `QueryV3` response carrying the `QueryV2` body, confirmed by the `X-Cache: hit` header and the absence of a certificate field in the CBOR response.
