### Title
Byzantine Block Proposer Can Stall Flexible HTTP Callback via Invalid Candid Content Bypassing Payload Validation — (`rs/https_outcalls/consensus/src/payload_builder.rs`)

### Summary

There is a semantic gap between `validate_canister_http_payload_impl` and `flexible_ok_responses_into_consensus_response`: validation never Candid-decodes the raw bytes inside `CanisterHttpResponseContent::Success(data)`, but `into_messages` does. A Byzantine block proposer can craft a `FlexibleCanisterHttpResponses` group whose `Success` bytes are structurally valid (pass all signature/membership/count checks) but are not valid Candid for `CanisterHttpResponsePayload`. The block passes validation on every honest node, is finalized, and then `into_messages` silently drops the entire group, delivering no `ConsensusResponse` for the callback. The callback stalls until `CANISTER_HTTP_TIMEOUT_INTERVAL` (60 s) expires.

### Finding Description

**Validation side — no Candid decode of response content bytes:**

`validate_canister_http_payload_impl` iterates `payload.flexible_responses` and for each group performs: duplicate-ID check, context lookup, response-count range check, per-entry `validate_flexible_response_with_proof` (signer membership, registry version, refund allowance), reject-not-allowed check, and deferred signature verification. At no point does it inspect the raw bytes of `CanisterHttpResponseContent::Success(data)`. [1](#0-0) 

**Execution side — Candid decode with silent `None` on failure:**

`flexible_ok_responses_into_consensus_response` calls `Decode!(&data, CanisterHttpResponsePayload).ok()` for every `Success` entry. If any decode returns `None`, `collect::<Option<_>>()` short-circuits and the function returns `None`. [2](#0-1) 

**`into_messages` silently drops the `None`:**

The result is passed through `.inspect(...)` (which increments `flexible_ok_responses_candid_failures`) and then `.flatten()`, which discards the `None` entirely — no `ConsensusResponse` is emitted for that `callback_id`. [3](#0-2) 

**The code comment acknowledges the fallback but not the Byzantine trigger:**

The function's doc comment says "This should never occur, but if it does, eventually a timeout will gracefully end the outstanding callback." This reasoning holds only under the assumption that honest nodes build the payload; it breaks when the block proposer is Byzantine. [4](#0-3) 

**Existing test confirms the exact drop behavior:**

The test `flexible_ok_responses_into_messages_decode_failure_is_skipped` constructs a group with one valid and one `b"this is invalid candid"` entry, calls `into_messages`, and asserts `responses.len() == 0` and `flexible_ok_responses_candid_failures == 1`. This is a direct proof-of-concept of the drop path. [5](#0-4) 

### Impact Explanation

Every honest replica executes `into_messages` deterministically on the finalized block. Because the drop is deterministic, no replica delivers a `ConsensusResponse` for `callback_id X`. The canister's callback is suspended until the 60-second `CANISTER_HTTP_TIMEOUT_INTERVAL` fires a timeout. A Byzantine proposer can repeat this on every block it is scheduled to propose, targeting any open flexible HTTP callback. [3](#0-2) 

### Likelihood Explanation

Block proposership rotates deterministically across all subnet nodes. A single Byzantine node within the BFT fault-tolerance threshold (`f < n/3`) will be scheduled as proposer regularly. No threshold collusion is required — one node suffices. The crafted payload requires only that the `Success` bytes be non-Candid (e.g., arbitrary bytes), which is trivially constructable. All other validation checks (signatures, membership, count) can be satisfied with legitimate cryptographic material from the node's own share. [6](#0-5) 

### Recommendation

Add a Candid-decode check for `CanisterHttpResponseContent::Success(data)` inside `validate_canister_http_payload_impl`'s `flexible_responses` loop, mirroring exactly what `flexible_ok_responses_into_consensus_response` does at execution time. If `Decode!(&data, CanisterHttpResponsePayload)` fails for any entry, return `invalid_artifact(InvalidCanisterHttpPayloadReason::FlexibleResponseCandidDecodeError { callback_id })`. This closes the semantic gap between validation and execution and ensures that any payload that passes validation will also produce a `ConsensusResponse` in `into_messages`. [7](#0-6) 

### Proof of Concept

The existing test at `tests.rs:3002–3026` is already a local-runnable proof of concept. It demonstrates that a `FlexibleCanisterHttpResponses` group containing `b"this is invalid candid"` as `Success` bytes passes `bytes_to_payload` (protobuf decode), is accepted by `into_messages`, and produces zero `ConsensusResponse` outputs with `flexible_ok_responses_candid_failures == 1`. To complete the end-to-end exploit demonstration, a companion test should call `validate_canister_http_payload_impl` on the same payload and assert it returns `Ok(())` — confirming the validation gap. [8](#0-7)

#[test]
fn flexible_ok_responses_into_messages_decode_failure_is_skipped() {
    let callback_id = CallbackId::from(42);

    let valid_data = Encode!(&CanisterHttpResponsePayload {
        status: 200,
        headers: vec![],
        body: vec![],
    })
    .unwrap();
    let valid_entry = flexible_response(42, 0, &valid_data);
    let invalid_entry = flexible_response(42, 1, b"this is invalid candid");

    let payload = flexible_payload(vec![FlexibleCanisterHttpResponses {
        callback_id,
        responses: vec![valid_entry, invalid_entry],
    }]);
    let bytes = payload_to_bytes_max_4mb(payload);

    let (responses, stats) = CanisterHttpPayloadBuilderImpl::into_messages(&bytes);

    assert_eq!(responses.len(), 0);
    assert_eq!(stats.flexible_ok_responses, 0);
    assert_eq!(stats.flexible_ok_responses_candid_failures, 1);
}
