Let me look at the `should_purge` logic and any P2P-level admission controls more carefully.

### Title
Unbounded IDKG Unvalidated Pool Growth via Crafted `source_height = u64::MAX` Dealings — (`rs/artifact_pool/src/idkg_pool.rs`, `rs/consensus/idkg/src/pre_signer.rs`)

---

### Summary

A Byzantine subnet peer can flood honest replicas with `SignedIDkgDealing` artifacts whose `transcript_id.source_height` is set to `u64::MAX`. These artifacts are unconditionally inserted into the unvalidated pool (no size guard), permanently deferred by `validate_dealings` (because `u64::MAX > tip_height` is always true), and never purged by `purge_artifacts` (because `source_height <= current_height` is never satisfied). The unvalidated pool grows without bound, eventually causing replica OOM and halting consensus finalization.

---

### Finding Description

**Step 1 — Unchecked pool insertion.**

`IDkgPoolImpl::insert` applies no height or size check:

```rust
fn insert(&mut self, artifact: UnvalidatedArtifact<IDkgMessage>) {
    let mut ops = IDkgPoolSectionOps::new();
    ops.insert(artifact.into_inner());
    self.unvalidated.mutate(ops);   // no bounds check
}
``` [1](#0-0) 

`MutableIDkgPoolSection::mutate` likewise performs no validation: [2](#0-1) 

**Step 2 — Permanent deferral in `validate_dealings`.**

`Action::action` in the pre-signer returns `Action::Defer` whenever `msg_height > block_reader.tip_height()`:

```rust
if let Some(height) = msg_height
    && height > block_reader.tip_height()
{
    return Action::Defer;
}
``` [3](#0-2) 

`Action::Defer` maps to `None` in `validate_dealings`, so the artifact is never removed from the unvalidated pool: [4](#0-3) 

With `source_height = u64::MAX`, the condition `u64::MAX > tip_height` is permanently true for any realistic chain height.

**Step 3 — `purge_artifacts` never removes future-height artifacts.**

`should_purge` requires `source_height() <= current_height` as a precondition:

```rust
transcript_id.source_height() <= current_height && !in_progress.contains(transcript_id)
``` [5](#0-4) 

The existing test suite explicitly documents this as intentional — "Dealing 3: height > current_height (not purged)": [6](#0-5) 

An artifact with `source_height = u64::MAX` satisfies neither the purge condition nor the validation condition, so it accumulates permanently.

**Step 4 — No P2P slot-table limit for IDKG.**

The IDKG broadcast channel is registered with `SLOT_TABLE_NO_LIMIT` (`usize::MAX`), meaning the P2P layer imposes no cap on how many IDKG artifact slots a peer may occupy: [7](#0-6) 

The ingress pool has an explicit per-peer byte/count limit (`exceeds_limit`); the IDKG pool has no equivalent: [8](#0-7) 

**Step 5 — `IDkgTranscriptId` fields are unchecked u64.**

`source_height` is a plain `Height` (wrapping `u64`) with no upper-bound validation at construction: [9](#0-8) 

A Byzantine peer can generate arbitrarily many unique `IDkgTranscriptId` values (varying the `id: u64` field) and sign each dealing with their legitimate node key, producing distinct pool entries that bypass deduplication.

---

### Impact Explanation

Each crafted dealing occupies memory in `InMemoryIDkgPoolSection` indefinitely. With no pool size cap and no purge path for `source_height > current_height`, a single Byzantine peer can exhaust replica heap memory. OOM kills the replica process, halting block finalization on that node. If enough replicas are targeted (still within the f < n/3 Byzantine budget), subnet liveness is lost.

---

### Likelihood Explanation

The attacker must be a legitimate subnet member (Byzantine peer), which is within the IC's explicit threat model. No threshold corruption is required — a single Byzantine node suffices to target individual honest replicas. The attack requires only the ability to send P2P gossip messages, which every subnet peer can do. No cryptographic break is needed; the peer signs dealings with their own valid key.

---

### Recommendation

1. **Add a height sanity check at pool admission.** Reject any `IDkgMessage::Dealing` whose `transcript_id.source_height` exceeds `tip_height + MAX_FUTURE_HEIGHT_DELTA` before inserting into the unvalidated pool.

2. **Add a per-peer or total unvalidated pool size cap for IDKG**, analogous to `ingress_pool_max_count`/`ingress_pool_max_bytes` in `IngressPoolImpl`.

3. **Extend `purge_artifacts` to evict artifacts that have been deferred beyond a configurable age** (e.g., `source_height > tip_height + PURGE_THRESHOLD`), so that even legitimately future-height artifacts cannot accumulate indefinitely.

4. **Replace `SLOT_TABLE_NO_LIMIT` for IDKG** with a bounded slot table limit in `rs/replica/setup_ic_network/src/lib.rs`.

### Proof of Concept

```rust
// Byzantine peer constructs N unique dealings with source_height = u64::MAX
for i in 0..1_000_000u64 {
    let transcript_id = IDkgTranscriptId::new(
        subnet_id,
        i,                          // unique id per dealing
        Height::new(u64::MAX),      // source_height = u64::MAX
    );
    let dealing = create_and_sign_dealing(transcript_id, byzantine_node_key);
    p2p_send_to_honest_replica(IDkgMessage::Dealing(dealing));
}

// On the honest replica:
// - Each dealing is inserted unconditionally (no bounds check)
// - validate_dealings: Action::Defer (u64::MAX > tip_height always)
// - purge_artifacts: should_purge returns false (u64::MAX <= current_height never)
// - Pool grows by ~N * sizeof(SignedIDkgDealing) bytes with no eviction path
// Assert: replica OOM before purge_artifacts removes any of the N entries
