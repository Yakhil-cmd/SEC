### Title
SNS Governance Proposal Queue Exhaustion Enables Temporary DOS of Non-Exempt Proposal Types - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance `make_proposal()` enforces a hard-coded ceiling of 700 unsettled proposals. Ballots are only cleared during periodic reward distribution events — not when proposals are rejected or executed. An unprivileged neuron holder with sufficient SNS tokens can fill this queue with junk proposals, blocking all non-exempt governance actions for the entire voting period (up to 30 days, extendable by wait-for-quiet).

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, `make_proposal()` rejects new proposals when the count of proposals with non-empty ballots reaches `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`: [1](#0-0) 

The constant is defined as: [2](#0-1) 

Ballots are only cleared inside `distribute_rewards()`, which runs on a configurable reward-round schedule (floor: 1 day, ceiling: 30 days): [3](#0-2) 

This means that even after 700 proposals are decided (rejected or executed), their ballots persist until the next reward event fires. The NNS governance exhibits the same pattern — the existing unit test explicitly documents that rejected proposals still block new submissions until ballots are cleared: [4](#0-3) 

The only proposals that bypass the limit are those returning `true` from `allowed_when_resources_are_low()`. For SNS, this covers only three action types: [5](#0-4) 

All other native SNS actions — `Motion`, `ManageNervousSystemParameters`, `RegisterDappCanisters`, `DeregisterDappCanisters`, `TransferSnsTreasuryFunds`, `MintSnsTokens`, `ManageSnsMetadata`, `ManageLedgerParameters` — are blocked when the queue is full.

There is no per-neuron proposal limit in `make_proposal()`. The only economic guard is that the proposer's stake must be at least `reject_cost_e8s`: [6](#0-5) 

The default `reject_cost_e8s` is 1 governance token: [7](#0-6) 

SNS communities can lower this further via a `ManageNervousSystemParameters` proposal. With `reject_cost_e8s` set to a small value, a single neuron holding 700 tokens can fill the entire queue at minimal cost.

Additionally, the configurable `NervousSystemParameters.max_number_of_proposals_with_ballots` (which the proto documents as controlling this limit) is **not** used in the `make_proposal()` check — the check always uses the hard-coded constant 700: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

An attacker can block all non-exempt SNS governance proposals for the full duration of the voting period plus wait-for-quiet extension. The maximum voting period is 30 days (`INITIAL_VOTING_PERIOD_SECONDS_CEILING`) and wait-for-quiet can extend it by up to another 30 days, yielding a potential DOS window of up to 60 days per attack cycle. Blocked proposal types include treasury transfers, token minting, dapp registration/deregistration, and parameter changes — core governance functions of any SNS. [10](#0-9) 

---

### Likelihood Explanation

The attack requires a neuron with stake ≥ `reject_cost_e8s` and the ability to submit 700 proposals. For SNS instances that have lowered `reject_cost_e8s` (or where the token price is low), the economic cost is negligible. Even at the default of 1 token per proposal, 700 tokens is a modest holding for a motivated attacker in a small SNS. The attack can be repeated each reward cycle. No privileged access, no key compromise, and no coordination beyond a single principal is required.

---

### Recommendation

1. **Per-neuron proposal cap**: Reject `make_proposal()` if the proposing neuron already has `N` open proposals (e.g., `N = 1` or a small configurable value), mirroring the Salty.IO mitigation.
2. **Clear ballots on settlement**: Clear ballots when a proposal reaches a final decision status (rejected/executed/failed), rather than waiting for the reward distribution event. This decouples the queue limit from the reward schedule.
3. **Use the configurable parameter**: Replace the hard-coded `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` constant in the `make_proposal()` check with `nervous_system_parameters.max_number_of_proposals_with_ballots`, so the SNS community can tune the limit. Currently the configurable field is documented as controlling this limit but has no effect on it.

    sns_governance.manage_neuron(ManageNeuron {
