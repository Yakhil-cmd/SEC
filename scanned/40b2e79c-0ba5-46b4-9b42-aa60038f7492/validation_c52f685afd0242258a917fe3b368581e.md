The code is now fully traced. Here is the analysis:

**`verify_encrypted_key_share_internal`** (lines 272–301 of `rs/crypto/src/vetkd/mod.rs`) does exactly one thing: verify the Ed25519 `node_signature` over the raw `encrypted_key_share` bytes. It never touches the curve-point content of those bytes. [1](#0-0) 

**`combine_encrypted_key_shares_internal`** (lines 347–366) calls `EncryptedKeyShare::deserialize` on every share's raw bytes. That function calls `G1Affine::deserialize` and `G2Affine::deserialize`, which return `Err` for any byte sequence that does not encode a valid prime-order curve point, even if the byte length is exactly `EncryptedKeyShare::BYTES`. A single deserialization failure causes the entire `.collect::<Result<_, _>>()` to short-circuit and return `VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare`. [2](#0-1) [3](#0-2) 

The gap is structural and confirmed by the code: `verify_encrypted_key_share` is a signature-only check; `combine_encrypted_key_shares` is a signature-plus-curve-point check. There is no curve-point validation in the verification path.

### Title
Byzantine node submits signature-valid but curve-invalid `VetKdEncryptedKeyShare`, causing `combine_encrypted_key_shares` to abort with `InvalidArgumentEncryptedKeyShare` — (`rs/crypto/src/vetkd/mod.rs`)

### Summary
`verify_encrypted_key_share` only verifies the Ed25519 node signature over the raw share bytes. It does not deserialize or validate the embedded BLS12-381 curve points. `combine_encrypted_key_shares` does deserialize those points and fails hard if any share contains an invalid encoding. A single Byzantine subnet node can exploit this gap to permanently block VetKD key derivation for any canister request that includes its share.

### Finding Description
`verify_encrypted_key_share_internal` constructs a `BasicSigOf` from `key_share.node_signature` and calls `BasicSigVerifierInternal::verify_basic_sig` over `key_share.encrypted_key_share`. The content of `encrypted_key_share` is treated as an opaque byte blob for signing purposes; no structural validation is performed. [1](#0-0) 

`combine_encrypted_key_shares_internal` iterates over all shares and calls `EncryptedKeyShare::deserialize`, which checks that the bytes decode to three valid BLS12-381 points (`c1: G1Affine`, `c2: G2Affine`, `c3: G1Affine`). If any point fails, the function returns `Err(VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare)` immediately, before any combination is attempted. [3](#0-2) 

`EncryptedKeyShare::BYTES` is `2 * G1Affine::BYTES + G2Affine::BYTES` = 192 bytes. A Byzantine node can trivially produce a 192-byte payload whose first 48 bytes do not represent a valid compressed G1 point (e.g., all-zero bytes, or any byte string with an invalid tag/coordinate), sign it with their legitimate node key, and submit it. The share passes `verify_encrypted_key_share` (signature is valid) but causes `combine_encrypted_key_shares` to abort. [4](#0-3) 

The `combine_valid_shares` fallback path (lines 388–430 of `rs/crypto/src/vetkd/mod.rs`) is only reached when `combine_all` returns `EncryptedKeyCombinationError::InvalidShares` — which requires successful deserialization of all shares first. It does not rescue a deserialization failure. [5](#0-4) 

### Impact Explanation
Any canister that requests VetKD key derivation and whose request is assigned to a combination round that includes the Byzantine node's share will receive a failed response. The `combine_shares` call in `rs/consensus/chain_key/src/lib.rs` propagates the error upward without a per-share retry mechanism. [6](#0-5) 

### Likelihood Explanation
The attacker is a legitimate subnet node (below the Byzantine fault threshold). No privileged access, key leakage, or majority corruption is required. The crafted payload is trivial to construct: any 192-byte string with an invalid BLS12-381 point encoding suffices. The node signs it with their own valid key. The attack is repeatable for every new VetKD request.

### Recommendation
Add `EncryptedKeyShare::deserialize` validation inside `verify_encrypted_key_share_internal`, after the signature check. If deserialization fails, return a new `VetKdKeyShareVerificationError` variant (e.g., `InvalidEncryptedKeyShare`). This closes the gap and restores the invariant that any share passing verification can always be deserialized during combination. [7](#0-6)

            (Ok(c1), Ok(c2), Ok(c3)) => Ok(Self { c1, c2, c3 }),
