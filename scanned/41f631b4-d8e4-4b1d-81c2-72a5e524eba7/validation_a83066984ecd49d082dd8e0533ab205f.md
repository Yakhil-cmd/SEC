### Title
NNS Governance `disburse_neuron` Unconditionally Zeroes `neuron_fees_e8s` Without Burning — Tokens Permanently Stuck in Neuron Subaccount - (File: `rs/nns/governance/src/governance.rs`)

---

### Summary

In `rs/nns/governance/src/governance.rs`, the `disburse_neuron` function always sets `neuron.neuron_fees_e8s = 0` in governance state regardless of whether the fee burn on the ICP ledger was actually executed. When `fees_amount_e8s <= transaction_fee_e8s`, the ledger burn is intentionally skipped (the ledger rejects burns below the minimum), but the governance accounting is still updated as if the burn occurred. This leaves the fee tokens permanently stranded in the neuron's subaccount on the ICP ledger with no retrieval path.

---

### Finding Description

In `disburse_neuron`, the fee-burning logic is guarded by a threshold check:

```rust
if fees_amount_e8s > transaction_fee_e8s {
    let _result = self.ledger.transfer_funds(
        fees_amount_e8s, 0,
        Some(neuron_subaccount),
        governance_minting_account(),
        now,
    ).await?;
}
``` [1](#0-0) 

Immediately after — unconditionally — the governance state is updated:

```rust
self.with_neuron_mut(id, |neuron| {
    if neuron.cached_neuron_stake_e8s > fees_amount_e8s {
        neuron.cached_neuron_stake_e8s -= fees_amount_e8s;
    } else {
        neuron.cached_neuron_stake_e8s = 0;
    }
    neuron.neuron_fees_e8s = 0;   // ← always zeroed, even when no burn happened
})
``` [2](#0-1) 

The disburse amount sent to the user was already computed as `minted_stake = cached_stake - fees`, so the user receives `cached_stake - fees - tx_fee`. After the ledger transfer completes, the neuron subaccount balance on the ICP ledger is exactly `fees_amount_e8s`. Governance, however, records `cached_neuron_stake_e8s = 0` and `neuron_fees_e8s = 0`, believing the neuron is fully empty. The `fees_amount_e8s` tokens in the subaccount are now orphaned: governance will never issue another transfer from that subaccount, and there is no admin sweep function.

The SNS governance canister identified and fixed this exact pattern in proposal 137687 (2025-08-01):

> *"Fees are now only recorded as burned when they exceed the transaction fee threshold and are actually burned."* [3](#0-2) 

The SNS fix conditionally updates `neuron_fees_e8s` only inside the burn branch:

```rust
if max_burnable_fee > transaction_fee_e8s {
    // burn ...
    neuron.neuron_fees_e8s = neuron.neuron_fees_e8s.saturating_sub(max_burnable_fee);
}
``` [4](#0-3) 

The NNS governance has not received the equivalent fix.

---

### Impact Explanation

For every NNS neuron that is disbursed while holding `0 < neuron_fees_e8s <= transaction_fee_e8s` (i.e., 1–10,000 e8s = up to 0.0001 ICP), the fee tokens are permanently locked in the neuron's ICP ledger subaccount. The governance canister controls those subaccounts but has no code path to recover the stranded balance. The ICP ledger total supply is conserved (tokens exist), but they are inaccessible — a ledger conservation bug identical in class to the Curves M-01 finding.

---

### Likelihood Explanation

The NNS `reject_cost_e8s` is currently set to 10 ICP (1,000,000,000 e8s), far above the 10,000 e8s transaction fee, so the condition `fees_amount_e8s <= transaction_fee_e8s` is rarely satisfied in normal operation. However, it can be reached if:
- A neuron accumulated fees from a governance period when `reject_cost_e8s` was lower, or
- Partial fee refunds (from accepted follow-up proposals) reduced `neuron_fees_e8s` to a sub-threshold value before disbursement.

Any dissolved NNS neuron holder can trigger this path by calling `disburse_neuron` — no privileged access is required.

---

### Recommendation

Mirror the SNS governance fix: only update `neuron_fees_e8s` (and the corresponding `cached_neuron_stake_e8s` reduction) inside the `if fees_amount_e8s > transaction_fee_e8s` branch, so governance state is only modified when the ledger burn actually executes.

```rust
if fees_amount_e8s > transaction_fee_e8s {
    self.ledger.transfer_funds(fees_amount_e8s, 0, ...).await?;

    self.with_neuron_mut(id, |neuron| {
        neuron.cached_neuron_stake_e8s =
            neuron.cached_neuron_stake_e8s.saturating_sub(fees_amount_e8s);
        neuron.neuron_fees_e8s = 0;
    }).expect("...");
}
// else: leave neuron_fees_e8s unchanged; the tokens remain in the subaccount
//       and will be disbursed to the user as part of the stake transfer.
```

### Proof of Concept

**Setup:** NNS neuron with `cached_neuron_stake_e8s = 1_000_000` and `neuron_fees_e8s = 5_000` (below the 10,000 e8s transaction fee).

**Step 1 — disburse amount computed:**
`disburse_amount_e8s = (1_000_000 - 5_000) - 10_000 = 985_000`

**Step 2 — burn skipped** (`5_000 <= 10_000`): no ledger call.

**Step 3 — governance state updated unconditionally:**
`cached_neuron_stake_e8s = 995_000`, `neuron_fees_e8s = 0`

**Step 4 — stake transfer to user:**
Ledger debits `985_000 + 10_000 = 995_000` from the subaccount.

**Result:**
- ICP ledger subaccount balance: `1_000_000 - 995_000 = 5_000` ← permanently stuck
- Governance: `cached_neuron_stake_e8s = 0`, `neuron_fees_e8s = 0` ← believes neuron is empty
- No governance function can issue a transfer from this subaccount again. [5](#0-4)
