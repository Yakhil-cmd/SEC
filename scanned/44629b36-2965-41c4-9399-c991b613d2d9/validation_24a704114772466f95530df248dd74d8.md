### Title
NNS/SNS Governance Neuron Lock Retained Indefinitely on Trap with No Timeout Escape Hatch — (`rs/nns/governance/src/neuron_lock.rs`, `rs/sns/governance/src/types.rs`)

---

### Summary

Both NNS and SNS Governance intentionally retain the neuron lock (`in_flight_commands`) when a trap occurs during an async ledger operation, with no timeout or user-accessible recovery path. The only resolution requires a governance upgrade (code change). This is structurally identical to the `potCreator` blocking `rotateLiquidity()`: a single failure point permanently freezes the neuron owner's staked ICP/SNS tokens, and there is no time-bounded escape hatch for the user.

---

### Finding Description

**Root cause — NNS Governance (`rs/nns/governance/src/neuron_lock.rs`):**

Both `NeuronAsyncLock::drop()` and `LedgerUpdateLock::drop()` check `ic_cdk::futures::is_recovering_from_trap()` and **skip** the `unlock_neuron` call when a trap is detected:

```rust
// NeuronAsyncLock::drop (lines 52-54)
if ic_cdk::futures::is_recovering_from_trap() {
    return;   // lock is NOT released
}
```

```rust
// LedgerUpdateLock::drop (lines 88-90)
if ic_cdk::futures::is_recovering_from_trap() {
    return;   // lock is NOT released
}
```

The `in_flight_commands` map entry is persisted in stable proto state across upgrades. The code itself acknowledges the consequence:

> *"If something goes fundamentally wrong (say we trap at some point after issuing a transfer call) the neuron(s) involved are left in a 'locked' state, meaning new operations can't be applied without reconciling the state … will require code changes."*

**Root cause — SNS Governance (`rs/sns/governance/src/types.rs:2163-2164`):**

The SNS `LedgerUpdateLock::drop()` has the identical pattern:

```rust
if ic_cdk::futures::is_recovering_from_trap() {
    return;   // lock is NOT released
}
```

**Most concrete trigger — `disburse_maturity.rs:668`:**

There is an explicit, non-trap code path that deliberately retains the lock permanently. In `finalize_maturity_disbursement`, if the ledger mint fails **and** the neuron mutation reversal also fails, the code calls:

```rust
// Reaching this point means the neuron mutation was performed, the ledger operation failed
// and the neuron mutation could not be reversed. The best we can do at this point is to
// retain the neuron lock.
neuron_lock.retain();
```

The scheduler in `get_delay_until_next_finalization` (lines 697–701) even acknowledges the consequence: *"it can be locked indefinitely."*

**No timeout or escape hatch exists anywhere in the codebase.** There is no periodic cleanup of stale `in_flight_commands` entries, no user-callable unlock after a deadline, and no automatic expiry.

---

### Impact Explanation

A neuron's ICP (NNS) or SNS token stake is permanently frozen. The neuron owner cannot disburse, split, spawn, merge, or perform any other operation on the neuron — all such operations check `in_flight_commands` and return `LedgerUpdateOngoing` / `NeuronLocked`. The staked tokens are inaccessible until a governance upgrade is deployed to clear the lock entry, a process that takes days to weeks on NNS and is subject to governance voting. For SNS neurons, the SNS governance itself must pass a proposal and upgrade, which may be impossible if the SNS is in a degraded state.

---

### Likelihood Explanation

**Trap path (both NNS and SNS):** Any async neuron operation (disburse, split, spawn, merge, disburse-maturity) that traps in its callback permanently locks the neuron. Traps can be caused by bugs in callback logic, out-of-memory conditions, or — for SNS — a ledger canister that returns a malformed response causing a decode panic. This is low-probability per operation but non-zero across the full user population and operation history.

**Explicit retain path (`disburse_maturity.rs:668`):** Requires two sequential failures: (1) the ICP ledger mint fails, and (2) the neuron mutation reversal also fails (e.g., the neuron was concurrently modified or deleted). This is a realistic double-fault scenario under ledger congestion or canister bugs.

Neither path requires privileged access. Any neuron owner submitting a standard `manage_neuron` ingress message is on the entry path.

---

### Recommendation

Add a timestamp-based expiry to `NeuronInFlightCommand`. During periodic tasks or on any neuron operation attempt, check whether the lock timestamp is older than a configurable threshold (e.g., 24–48 hours). If so, automatically release the lock and emit an audit event. This mirrors the external report's recommendation of "adding a time duration after which the privileged action cannot block user withdrawals."

Alternatively, expose a governance-proposal-gated `clear_neuron_lock` endpoint that can be invoked without a full canister upgrade.

---

### Proof of Concept

**Trap path:**
1. Neuron owner submits `manage_neuron { command: Disburse { ... } }` (unprivileged ingress).
2. NNS Governance acquires `LedgerUpdateLock` via `lock_neuron_for_command`, inserting into `in_flight_commands`.
3. Governance makes async call to ICP ledger.
4. Governance callback traps (e.g., due to a decode bug or OOM).
5. `LedgerUpdateLock::drop()` fires; `is_recovering_from_trap()` returns `true`; `unlock_neuron` is skipped.
6. `in_flight_commands[neuron_id]` persists in stable state.
7. All subsequent `manage_neuron` calls for this neuron return `LedgerUpdateOngoing` indefinitely.
8. Neuron owner's ICP stake is inaccessible until a governance upgrade clears the entry.

**Explicit retain path (`disburse_maturity`):**
1. Neuron owner calls `disburse_maturity` to schedule a maturity disbursement.
2. Periodic task `finalize_maturity_disbursement` fires; acquires `NeuronAsyncLock`; pops the disbursement from the neuron's queue (neuron mutation performed).
3. ICP ledger mint call fails (e.g., transient ledger error).
4. Governance attempts to reverse the neuron mutation (`push_front_maturity_disbursement_in_progress`); this also fails (e.g., neuron not found due to concurrent modification).
5. `neuron_lock.retain()` is called at `disburse_maturity.rs:668`.

impl Drop for NeuronAsyncLock {
    fn drop(&mut self) {
        if self.retain {
            return;
        }
        // In the case of a panic, the state of the ledger account representing the neuron's stake
        // may be inconsistent with the internal state of governance.  In that case, we want to
        // prevent further operations with that neuron until the issue can be investigated and
        // resolved, which will require code changes.
        if ic_cdk::futures::is_recovering_from_trap() {
            return;
        }
        // The lock is released when the NeuronAsyncLock is dropped. This is done to ensure that the lock
        // is released even if the NeuronAsyncLock is not explicitly unlocked.
        self.governance.with_borrow_mut(|governance| {
            governance.unlock_neuron(self.neuron_id.id);
        });
    }
