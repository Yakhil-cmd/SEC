### Title
Unbounded Loop with Wrapping Arithmetic in `/construction/payloads` Causes Infinite CPU/OOM DoS on ICP Rosetta Node — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The ICP Rosetta node's `construction_payloads` handler contains an unbounded `while` loop that iterates over attacker-controlled `ingress_start`/`ingress_end` values with no input validation. When `ingress_end = u64::MAX`, the `Time::add` implementation wraps around (truncating cast via `as u64`), making the loop **infinite**. Any unauthenticated HTTP client can trigger this to permanently hang the Rosetta node thread.

---

### Finding Description

The loop in question: [1](#0-0) 

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;   // interval = 120 seconds
}
```

`ingress_start` and `ingress_end` are taken directly from the client-supplied JSON metadata with no bounds validation: [2](#0-1) 

The metadata struct accepts raw `u64` nanosecond values with no range checks: [3](#0-2) 

The `interval` is computed as: [4](#0-3) 

```
MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120_000_000_000 ns
``` [5](#0-4) 

**The critical defect is in `Time::add` / `Time::add_assign`:** [6](#0-5) 

```rust
fn add(self, dur: Duration) -> Time {
    Time::from_duration(Duration::from_nanos(self.0) + dur)
}
fn from_duration(t: Duration) -> Self {
    Time(t.as_nanos() as u64)   // ← truncating cast, wraps on overflow
}
```

`Duration::as_nanos()` returns `u128`. When `now` is near `u64::MAX` and `interval` (120s) is added, the resulting `u128` exceeds `u64::MAX`, and the `as u64` cast **wraps around** to a small value (≈60 seconds in nanoseconds). Since this small value is still `< ingress_end = u64::MAX`, the loop condition remains true and the loop **never terminates**.

**Contrast with ICRC1 Rosetta**, which has explicit guards before its equivalent loop: [7](#0-6) 

```rust
if ingress_start >= ingress_end { return Err(...) }
if ingress_end < now + ingress_interval { return Err(...) }
```

The ICP Rosetta handler has **none of these guards**.

---

### Impact Explanation

- **Infinite loop / CPU exhaustion**: With `ingress_end = u64::MAX`, `now` wraps around repeatedly and the loop never exits, consuming 100% of the handling thread indefinitely.
- **OOM path**: Even without wrapping (e.g., `ingress_end = u64::MAX / 2`), the loop runs ~76 billion iterations, each pushing a `u64` to `ingress_expiries`, allocating ~600 GB of heap before OOM-killing the process.
- **Scope**: The Rosetta node process is rendered unavailable. All subsequent API requests are dropped. This is a complete availability loss for the Rosetta service.

---

### Likelihood Explanation

- The `/construction/payloads` endpoint is a public, unauthenticated HTTP POST endpoint.
- The attack requires a single crafted JSON request with two integer fields set to extreme values.
- No credentials, keys, or privileged access are needed.
- The exploit is trivially reproducible locally.

---

### Recommendation

1. **Add input validation** before the loop, mirroring the ICRC1 Rosetta guards:
   - Reject if `ingress_start >= ingress_end`
   - Reject if `ingress_end - ingress_start > some_reasonable_max` (e.g., 24 hours = `86_400_000_000_000` ns, yielding at most 720 iterations)
2. **Use `checked_add`** instead of the unchecked `+=` in the loop to detect and break on overflow.
3. **Cap `ingress_expiries`** to a maximum length (e.g., 1000 entries) as a defense-in-depth measure.
