### Title
Race Condition in `LauncherChildWatch`: Unknown PID Defaults `should_panic = true`, Crashing the Launcher — (`rs/canister_sandbox/src/launcher.rs`)

---

### Summary

A real TOCTOU race condition exists between `launch_compiler`'s `spawn_socketed_process` call and its subsequent `info_map.insert`. If a compiler child process exits with a non-zero status in that window, the `LauncherChildWatch` thread reaps it via `wait()`, finds `None` in the map, and defaults `should_panic` to `true`, triggering `panic!("Launcher detected sandbox exit")` and crashing the launcher — which in turn crashes the replica.

---

### Finding Description

In `rs/canister_sandbox/src/launcher.rs`, `launch_compiler` spawns a child process and only inserts its PID into `pid_to_process_info` after acquiring the lock — several instructions later: [1](#0-0) 

The gap between line 230 (`spawn_socketed_process`) and line 245 (`info_map.insert`) is the race window. During this window, the compiler process can exit.

Concurrently, the `LauncherChildWatch` thread — if the map was already non-empty — has already passed the `wait_while` guard and dropped the lock: [2](#0-1) 

It then calls `wait()` (line 118), which is a non-specific POSIX `waitpid(-1, ...)` that reaps **any** child. If it reaps the compiler PID before `launch_compiler` inserts it: [3](#0-2) 

`info_map.remove(&pid)` returns `None`, and `.unwrap_or(true)` at line 146 sets `should_panic = true`. The panic fires unconditionally at line 156: [4](#0-3) 

This is the defect: the `unwrap_or(true)` sentinel is wrong. For compiler processes, `panic_on_failure` is intentionally `false`: [5](#0-4) 

But that `false` is never consulted if the PID races out of the map before insertion.

---

### Impact Explanation

The launcher process panics, which is detected by the replica's watcher thread (`panic_due_to_exit`), causing the replica process itself to panic. This is a single-replica crash (medium scope). No data is corrupted; the replica restarts, but availability is disrupted. [6](#0-5) 

---

### Likelihood Explanation

**Low.** The race window is extremely narrow — it spans only the time between `spawn_socketed_process` returning and `info_map.lock().unwrap()` completing (lines 230–235), typically sub-microsecond. For the race to fire:

1. The map must already be non-empty (watcher past `wait_while`).
2. The compiler child must exit with a non-zero status **within that window**.
3. The watcher's `wait()` must return before `launch_compiler` acquires the lock.

Condition 2 is the hardest to satisfy: the compiler binary must fail almost immediately after `execve`. This could happen under extreme resource pressure (OOM, missing binary, `SIGKILL` from the OS), but is not reliably triggerable by an unprivileged attacker submitting crafted Wasm — Wasm validation occurs before the compiler is invoked, and the compiler binary itself is trusted infrastructure.

An unprivileged attacker can trigger canister compilation via `install_code` ingress, satisfying the entry-point requirement, but cannot reliably win the race or force the compiler to crash in the sub-microsecond window.

---

### Recommendation

Change the `unwrap_or` default from `true` to `false`:

```rust
// rs/canister_sandbox/src/launcher.rs, line 146
let should_panic = process_info
    .as_ref()
    .map(|x| x.panic_on_failure)
    .unwrap_or(false);  // unknown PID: do not panic
```

Additionally, insert the PID into the map **before** the process can be reaped — i.e., acquire the lock before calling `spawn_socketed_process`, or use `WNOHANG`-style deferred reaping. The `unwrap_or(false)` fix is the minimal safe correction; the structural fix eliminates the race entirely.
