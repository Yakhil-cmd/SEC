### Title
Zero-Cost Repeated `refresh_buyer_tokens` Calls Drain SNS Swap Canister Cycles via Unconditional ICP Ledger Inter-Canister Call - (File: rs/sns/swap/src/swap.rs)

### Summary
The `refresh_buyer_token_e8s` function in the SNS Swap canister unconditionally makes an inter-canister call to the ICP ledger to read the caller's balance **before** checking whether any ICP has been deposited. An unprivileged attacker can invoke `refresh_buyer_tokens` an unlimited number of times with zero ICP deposited, paying nothing themselves while forcing the swap canister to pay for an inter-canister call on every invocation. This is the direct IC analog of the `craftAmount = 0` bug: the "amount" check (minimum ICP balance) is gated after the expensive operation (the ledger call), not before it.

### Finding Description
In `rs/sns/swap/canister/canister.rs`, the public `#[update]` endpoint `refresh_buyer_tokens` is callable by any unprivileged principal with no preconditions on token ownership. [1](#0-0) 

It delegates to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`. The function performs three cheap, local checks (lifecycle, participation possibility, confirmation text), then **unconditionally** issues an `await`-ed inter-canister call to the ICP ledger to read the buyer's subaccount balance: [2](#0-1) 

Only **after** the inter-canister call returns does the function check whether the returned balance meets the minimum participation threshold: [3](#0-2) 

If the caller has deposited zero ICP, `e8s` will be `0`, which is always less than `params.min_participant_icp_e8s`, so the call returns an error — but the inter-canister call to the ICP ledger has already been made and its cycles cost has already been charged to the swap canister.

There is no rate-limiting guard, no per-caller cooldown, and no minimum-deposit pre-check before the ledger call. The ckBTC `update_balance` function, by contrast, uses a `balance_update_guard` to prevent concurrent abuse: [4](#0-3) 

No equivalent guard exists in `refresh_buyer_token_e8s`.

### Impact Explanation
Every call to `refresh_buyer_tokens` with zero ICP deposited forces the SNS Swap canister to pay for one inter-canister call to the ICP ledger. Because the endpoint is open to any principal with no deposit precondition, an attacker controlling many principals (or a single principal calling sequentially) can issue an unbounded stream of such calls. The cumulative cycles cost is borne entirely by the swap canister. If the swap canister's cycle balance is exhausted, it becomes frozen and the SNS decentralization swap fails — permanently locking or losing participant funds and preventing the SNS from launching. Impact: **5/10**.

### Likelihood Explanation
The attack requires no tokens, no special permissions, and no coordination. Any IC user can send ingress messages to the swap canister at negligible personal cost (ingress fees are charged to the receiving canister on IC, not the sender). The attack window is bounded by the swap's open lifecycle, which can last days to weeks. Sustained automated calling is straightforward. Likelihood: **3/10**.

### Recommendation
Add a minimum-balance pre-check **before** the inter-canister call. Specifically, require the caller to have a non-zero (or at least `min_participant_icp_e8s`-sized) balance in the swap subaccount before issuing the ledger query. Alternatively, introduce a per-caller rate-limiting guard analogous to the `balance_update_guard` used in the ckBTC minter: [4](#0-3) 

A simpler mitigation is to reorder the logic: check whether the caller already has a recorded buyer state that is up-to-date, or require a small proof-of-intent (e.g., a non-zero on-chain balance check via a query call) before committing to the update call.

### Proof of Concept

1. Wait for an SNS swap to enter `Lifecycle::Open`.
2. From any principal that has **not** deposited any ICP to the swap subaccount, call:
   ```
   dfx canister call <swap_canister_id> refresh_buyer_tokens \
     '(record { buyer = ""; confirmation_text = null })'
   ```
3. Observe that the call returns an error (`Amount transferred: 0; minimum required to participate: ...`), but the swap canister has paid for one inter-canister call to the ICP ledger.
4. Repeat step 2 in a tight loop from a script. Each iteration costs the attacker only the ingress induction fee (charged to the swap canister, not the caller), while the swap canister pays for the ledger `account_balance` call on every iteration.
5. After sufficiently many iterations, the swap canister's cycle balance approaches its freezing threshold, causing it to stop processing messages — aborting the SNS launch.

The root cause is at: [5](#0-4) 

with the amount guard placed too late at: [3](#0-2)

async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    log!(INFO, "refresh_buyer_tokens");

        caller_principal_id()

        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
    {
        Ok(r) => r,
        Err(msg) => panic!("{}", msg),
    }
}
