### Title
Silent Clamping of Over-Range `icrc2_approve` Amount to `Tokens::MAX` Grants Unintended Maximum Allowance — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
An unprivileged caller can submit an `icrc2_approve` call with `amount` encoded as a Candid `Nat` exceeding `u64::MAX`. Instead of rejecting the call, the ledger silently replaces the amount with `Tokens::MAX` (`u64::MAX` e8s), recording an allowance the caller never explicitly authorized. The spender can then drain the approver's full balance via `icrc2_transfer_from`.

### Finding Description
In `icrc2_approve_not_async` [1](#0-0) , the conversion of the user-supplied `amount` field uses:

```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```

`Tokens::try_from(Nat)` returns `Err` for any value that does not fit in `u64` [2](#0-1) , but the `unwrap_or_else` branch discards the error and substitutes `Tokens::MAX` (`u64::MAX` e8s) [3](#0-2) . The resulting `amount` is then embedded directly into the `Approve` transaction and committed to the ledger [4](#0-3) .

The same function handles the `expected_allowance` field correctly — a conversion failure there returns `ApproveError::AllowanceChanged` [5](#0-4)  — making the inconsistency in the `amount` path unambiguously a defect.

### Impact Explanation
Any unprivileged ledger user (no special role required) can:
1. Encode `ApproveArgs { amount: Nat(2^64), spender: <attacker_account>, ... }` in Candid.
2. Call `icrc2_approve`. The ledger records an allowance of `u64::MAX` e8s (~184 billion tokens) for the spender.
3. The spender calls `icrc2_transfer_from` repeatedly to drain the approver's entire balance.

The approver's intent was to approve an amount that should have been rejected; instead they unknowingly grant the maximum possible allowance.

### Likelihood Explanation
The attack requires only a standard ingress `icrc2_approve` call with a crafted `Nat` value. No privileged access, key material, or governance majority is needed. Any user of any ICRC-1/ICRC-2 ledger deployed with this code is affected.

### Recommendation
Replace the silent fallback with an explicit error return:

```rust
let amount = Tokens::try_from(arg.amount).map_err(|_| ApproveError::GenericError {
    error_code: Nat::from(0u64),
    message: "amount exceeds maximum token value".to_string(),
})?;
```

This mirrors the correct handling already applied to `expected_allowance` in the same function. [6](#0-5)
