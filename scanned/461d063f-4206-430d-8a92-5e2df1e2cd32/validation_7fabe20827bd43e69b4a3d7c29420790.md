### Title
Missing Execution-Time 7-Day Rolling Amount Re-Check in `perform_mint_sns_tokens` Allows Limit Bypass - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

`perform_transfer_sns_treasury_funds` re-validates the 7-day rolling treasury-amount cap at proposal execution time, but the analogous `perform_mint_sns_tokens` function performs no such re-check. Multiple `MintSnsTokens` proposals can be submitted and approved concurrently, each passing the per-proposal limit at submission time, then all execute without any cumulative re-check, collectively minting far beyond the intended 7-day cap.

---

### Finding Description

SNS governance enforces a 7-day rolling upper bound on both treasury transfers and token minting. At proposal submission time, both `validate_and_render_transfer_sns_treasury_funds` and `validate_and_render_mint_sns_tokens` call the shared helper `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which checks the rolling total and stores a `Valuation` in `ActionAuxiliary`.

At **execution time**, however, the two action handlers diverge:

`perform_transfer_sns_treasury_funds` (lines 3000â€“3005) calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, which re-computes the rolling total against the stored valuation and aborts if the cap would be exceeded: [1](#0-0) 

`perform_mint_sns_tokens` (lines 3062â€“3088) performs **no equivalent check** â€” it resolves the target account and immediately calls `ledger.transfer_funds`: [2](#0-1) 

The codebase itself contains two explicit markers confirming this gap is known but unresolved:

1. The import of `mint_sns_tokens_7_day_total_upper_bound_tokens` is commented out with `TODO(NNS1-2982)`: [3](#0-2) 

2. The helper `total_minting_amount_tokens` is annotated `#[allow(unused)]` with `TODO(NNS1-2910): Delete this`, indicating the execution-time enforcement path for minting was never wired up: [4](#0-3) 

The execution-time check for `TransferSnsTreasuryFunds` explicitly documents why it is necessary â€” the rolling total at execution time can differ from what was computed at submission time: [5](#0-4) 

---

### Impact Explanation

The 7-day rolling cap on `MintSnsTokens` is a rate-limit designed to protect SNS token holders even against a governance majority acting in bad faith or under compromise. Without the execution-time re-check, an attacker who controls or influences SNS governance can:

1. Submit N `MintSnsTokens` proposals simultaneously, each for an amount just below the 7-day cap.
2. Each proposal passes the submission-time check independently (the rolling total is still low when each is submitted).
3. All N proposals are voted through and executed in sequence.
4. `perform_mint_sns_tokens` executes each without re-checking the cumulative total.
5. The SNS ledger mints N Ă— (cap amount) tokens â€” a multiple of the intended limit â€” inflating supply and diluting all token holders.

---

### Likelihood Explanation

Any SNS governance participant who can achieve a voting majority (the normal operational threshold for any SNS proposal) can exploit this. No privileged key, admin access, or external oracle is required. The attack requires only the ability to submit and pass multiple `MintSnsTokens` proposals within a voting window, which is a standard governance flow. The gap is confirmed by two TODO markers in production code, indicating it has not been closed.

---

### Recommendation

Add an execution-time re-check to `perform_mint_sns_tokens` analogous to the one in `perform_transfer_sns_treasury_funds`. Specifically:

1. Retrieve the stored `Valuation` from `ActionAuxiliary::MintSnsTokens` (already stored at proposal submission time, as seen in `validate_and_render_mint_sns_tokens`).
2. Implement and call a `mint_sns_tokens_amount_is_small_enough_at_execution_time_or_err` function (the `mint_sns_tokens_7_day_total_upper_bound_tokens` function already exists in `proposals_amount_total_limit` â€” uncomment the import per `TODO(NNS1-2982)`).
3. Wire the check into `perform_mint_sns_tokens` before the ledger transfer, mirroring lines 3000â€“3005 of `perform_transfer_sns_treasury_funds`.

---

### Proof of Concept

**Setup:** An SNS with a treasury valuation placing it in the "medium" regime, where the 7-day cap is 25% of treasury (e.g., 25,000 tokens on a 100,000-token treasury).

**Attack:**
1. Submit `MintSnsTokens` proposal A for 24,000 tokens. Submission-time check passes (0 + 24,000 â‰¤ 25,000).
2. Submit `MintSnsTokens` proposal B for 24,000 tokens. Submission-time check passes (0 + 24,000 â‰¤ 25,000, because A is not yet executed).
3. Vote both proposals through governance.
4. Proposal A executes: `perform_mint_sns_tokens` mints 24,000 tokens. No execution-time re-check.
5. Proposal B executes: `perform_mint_sns_tokens` mints another 24,000 tokens. No execution-time re-check.
6. **Result:** 48,000 tokens minted â€” nearly double the 25,000-token 7-day cap â€” with no error returned.

Compare: if both were `TransferSnsTreasuryFunds` proposals, step 5 would be caught by `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` and aborted with `ErrorType::PreconditionFailed`. [6](#0-5)

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }

    Ok(())
