### Title
Vote Overwriting Without "Already Voted" Guard in Root Proposal Governance Upgrade - (File: `rs/nns/handlers/root/impl/src/root_proposals.rs`)

---

### Summary

The `vote_on_root_proposal_to_upgrade_governance_canister` function in the NNS root canister unconditionally overwrites a node operator's ballot on every call, with no guard preventing a voter from changing a previously cast vote. This is the direct IC analog of the reported "votes not cleared / vote state not protected" class: a participant in the active voting set can flip their own vote at will, corrupting the tally and potentially forcing or blocking a governance canister upgrade.

---

### Finding Description

In `rs/nns/handlers/root/impl/src/root_proposals.rs`, the voting loop that records a node operator's ballot contains no idempotency or "already voted" check:

```rust
// Add the ballots for this node operator.
let mut voted_on: i32 = 0;
for (p, b) in &mut proposal.node_operator_ballots {
    if p == &caller {
        *b = ballot.clone();   // ← unconditional overwrite
        voted_on += 1;
    }
}
``` [1](#0-0) 

Every time a node operator calls `vote_on_root_proposal_to_upgrade_governance_canister`, their entry in `node_operator_ballots` is silently replaced with the new `ballot` value, regardless of what it was before. There is no equivalent of the `NeuronAlreadyVoted` guard that the NNS governance canister enforces for normal proposals:

```rust
if neuron_ballot.vote != (Vote::Unspecified as i32) {
    return Err(GovernanceError::new_with_message(
        ErrorType::NeuronAlreadyVoted,
        "Neuron already voted on proposal.",
    ));
}
``` [2](#0-1) 

The root proposal's tally functions (`is_byzantine_majority_yes` / `is_byzantine_majority_no`) count the current state of `node_operator_ballots` at the moment of each vote call: [3](#0-2) 

Because the ballot is mutable and unguarded, the tally is computed over a mutable, attacker-controlled value rather than an immutable record of each operator's first cast vote.

The `node_operator_ballots` field is a `Vec<(PrincipalId, RootProposalBallot)>` populated at proposal submission time from the NNS subnet registry: [4](#0-3) 

The canister entry point is exposed as an update call reachable by any node operator principal: [5](#0-4) 

---

### Impact Explanation

A node operator on the NNS subnet who has already cast a `Yes` vote can call `vote_on_root_proposal_to_upgrade_governance_canister` again with `No` (or `Undecided`) to retract or flip their vote. Symmetrically, a node operator who voted `No` can flip to `Yes`. Because the byzantine-majority threshold is `N - f` (e.g., 15 out of 22 nodes), a single operator controlling multiple node slots, or a small coalition of operators, can:

1. **Block a legitimate upgrade** — vote `Yes` to help reach quorum, then flip to `No` or `Undecided` before the final tally is checked, dropping the count below threshold.
2. **Force through a rejected upgrade** — vote `No` to appear to block a proposal, then flip to `Yes` at a strategic moment to push it over the threshold.

The governance canister is the most critical NNS canister; an unauthorized or manipulated upgrade could break the entire NNS governance system, freeze neuron management, or install malicious logic controlling ICP token flows.

---

### Likelihood Explanation

The `vote_on_root_proposal_to_upgrade_governance_canister` update method is callable by any principal that appears in the `node_operator_ballots` list, i.e., any node operator running a node on the NNS subnet at the time the proposal was submitted. The NNS subnet has ~40 nodes operated by a relatively small set of node providers. A single compromised or malicious node operator key is sufficient to exploit this. The function is reachable via standard ingress (no special tooling beyond `ic-admin` or a direct canister call), and the attack requires no coordination beyond the attacker controlling their own principal.

---

### Recommendation

Add an "already voted" guard inside the ballot loop, mirroring the pattern used in NNS governance's `register_vote`:

```rust
for (p, b) in &mut proposal.node_operator_ballots {
    if p == &caller {
        // Prevent vote changes after the initial cast.
        if !matches!(b, RootProposalBallot::Undecided) {
            return Err(format!(
                "{LOG_PREFIX}Caller: {caller} has already cast a ballot on this proposal.",
            ));
        }
        *b = ballot.clone();
        voted_on += 1;
    }
}
``` [1](#0-0) 

This ensures that once a node operator's ballot transitions from `Undecided` to `Yes` or `No`, it cannot be overwritten by a subsequent call.

### Proof of Concept

**Setup:** NNS subnet with 7 nodes (f = 2, threshold = 5 yes votes). Node operators: NO1…NO7. NO1 submits a root proposal (auto-votes `Yes`). NO2–NO5 vote `Yes`. Tally: 5 Yes — proposal is at the threshold.

**Attack (blocking a legitimate upgrade):**
1. NO2 calls `vote_on_root_proposal_to_upgrade_governance_canister(proposer=NO1, wasm_sha256=..., ballot=Yes)` — tally: 5 Yes.
2. Before `is_byzantine_majority_yes()` is evaluated in a subsequent vote call, NO2 calls the same function again with `ballot=No`.
3. NO2's entry in `node_operator_ballots` is now `No`. Tally: 4 Yes, 1 No — below threshold.
4. The proposal stalls or is rejected, despite having legitimately reached 5 Yes votes.

**Attack (forcing through a rejected upgrade):**
1. NO2–NO4 vote `No`. Tally: 1 Yes, 3 No — `is_byzantine_majority_no()` requires > 2 No votes, so 3 No votes rejects the proposal.
2. Before the rejection branch executes (i.e., before another vote triggers the check), NO2 flips to `Yes`.
3. Tally: 2 Yes, 2 No — no majority either way; proposal remains open.
4. NO2 and NO3 both flip to `Yes`. Tally: 3 Yes, 1 No — still below 5.
5. NO5 and NO6 vote `Yes`. Tally: 5 Yes, 1 No — `is_byzantine_majority_yes()` passes; upgrade executes.
