### Title
Unbounded Frame Size Allocation in `FrameDecoder::decode()` Enables Compromised Sandbox to Crash Replica Process — (`File: rs/canister_sandbox/src/frame_decoder.rs`)

---

### Summary

The `FrameDecoder::decode()` function in the canister sandbox IPC layer reads a raw `u64` frame-size field from the wire and calls `data.reserve(size)` with no upper-bound check. A compromised sandbox process can send a crafted frame header containing a near-`u64::MAX` size, causing the replica controller thread to attempt an enormous heap allocation, triggering an OOM kill or a Rust allocator panic. Additionally, `bincode::deserialize(&frame).unwrap()` on the same path panics on any malformed payload, providing a second crash vector. Neither path is bounded by `SocketReaderConfig`, which carries no `max_message_size` field at all.

---

### Finding Description

`FrameDecoder<Message>` is the sole message-framing layer for all sandbox ↔ replica IPC. Its `decode()` method:

1. Reads 8 bytes and interprets them as a `u64` frame size with no validation.
2. Casts that `u64` directly to `usize` and calls `data.reserve(size)` — no upper bound.
3. Calls `bincode::deserialize(&frame).unwrap()` — panics on any deserialization error. [1](#0-0) 

`socket_read_messages()` in `transport.rs` drives this decoder in a tight loop with no interposed size guard: [2](#0-1) 

`SocketReaderConfig` controls only idle-trim and malloc-trim behaviour; it has no `max_message_size` field: [3](#0-2) 

The replica controller thread that reads from each sandbox socket is spawned in `spawn_canister_sandbox_process()` and uses `SocketReaderConfig::default()`, which adds no protection: [4](#0-3) 

---

### Impact Explanation

A compromised sandbox process sends a single 8-byte frame header containing `0xFFFFFFFFFFFFFFFF`. The replica's `FrameDecoder` reads the size, casts it to `usize`, and calls `data.reserve(18_446_744_073_709_551_615)`. On a 64-bit Linux host this triggers an immediate OOM kill of the replica process (or a Rust allocator panic, which also terminates the process). Because the replica process is the consensus and state-management engine for the subnet, its crash halts block production for the affected node, degrading subnet liveness. The same crash is reachable via the `bincode::deserialize(...).unwrap()` path by sending a syntactically valid length followed by garbage bytes. [5](#0-4) 

---

### Likelihood Explanation

The IC's security model explicitly treats the sandbox process as an untrusted boundary: canister Wasm runs inside the sandbox, and any Wasmtime or instrumentation bug that grants native code execution inside the sandbox process immediately enables this attack. The attack requires only writing 8 bytes to the Unix-domain socket that the sandbox already holds open toward the replica. No network access, no privileged credentials, and no consensus-level corruption are needed — only native code execution inside one sandbox process, which is the exact threat the sandbox boundary is designed to contain. [6](#0-5) 

---

### Recommendation

1. Add a `max_frame_size: usize` field to `SocketReaderConfig` and enforce it in `FrameDecoder::decode()` before calling `data.reserve()`. A reasonable limit is the maximum inter-canister payload size (currently 2 MiB for responses, with a small overhead for framing).
2. Replace `bincode::deserialize(&frame).unwrap()` with a `match` that returns `None` (or closes the socket) on deserialization failure instead of panicking.
3. Propagate the limit through all call sites: `SocketReaderConfig::default()`, `SocketReaderConfig::for_sandbox()`, and `SocketReaderConfig::for_testing()`. [5](#0-4) [3](#0-2) 

---

### Proof of Concept

From inside a compromised sandbox process (one that has escaped Wasm isolation), write the following 8 bytes to file descriptor 3 (the Unix-domain socket to the replica controller):

```rust
// Inside compromised sandbox process — fd 3 is the controller socket
use std::os::unix::io::FromRawFd;
use std::io::Write;
let mut sock = unsafe { std::os::unix::net::UnixStream::from_raw_fd(3) };
// Frame header: u64 big-endian value of u64::MAX
sock.write_all(&u64::MAX.to_be_bytes()).unwrap();
```

The replica's `CanisterSandboxIPC` thread calls `FrameDecoder::decode()`, reads the size `18446744073709551615`, casts it to `usize`, and calls `data.reserve(18446744073709551615)`. The allocator immediately fails, the thread panics, and the replica process terminates. [7](#0-6) [8](#0-7)
