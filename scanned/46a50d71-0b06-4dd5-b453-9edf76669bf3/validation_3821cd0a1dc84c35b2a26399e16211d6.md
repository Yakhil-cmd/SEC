### Title
Missing `break`/`return` After Disconnect Causes Infinite `Disconnected` Event Loop — (`rs/bitcoin/adapter/src/stream.rs`)

### Summary

The `handle_stream` task loop never exits after emitting a `StreamEventKind::Disconnected` event. A malicious (or simply abruptly-closing) Bitcoin peer can trigger this by closing the TCP connection, causing the task to spin indefinitely and flood the shared `stream_event_sender` channel with duplicate `Disconnected` events, starving the router event loop.

### Finding Description

In `handle_stream`, after a disconnect-class error is detected and a `Disconnected` event is sent, the loop has no `break` or `return`: [1](#0-0) 

The `tick()` method calls `read_message()`, which calls `try_read()` on the closed socket. A closed TCP socket returns 0 bytes on every subsequent read, which maps to `StreamError::Io(UnexpectedEof)`: [2](#0-1) 

`UnexpectedEof` is not `WouldBlock`, so `disconnect = true` on every iteration: [3](#0-2) 

The `send(...).await.ok()` call blocks when the channel is full (capacity `DEFAULT_CHANNEL_BUFFER_SIZE = 64`) and resumes each time the router drains one slot, creating a steady-state flood: [4](#0-3) 

### Impact Explanation

The router's main event loop in `start_main_event_loop` uses a `tokio::select!` over several branches, one of which is `connection_manager.receive_stream_event()`: [5](#0-4) 

When the `stream_event_sender` channel is continuously saturated with duplicate `Disconnected` events from one peer's leaked task, the `select!` loop preferentially drains those events. This starves the `network_message_receiver` branch (carrying block/header data from all other peers) and the `tick_interval` branch (which drives `connection_manager.tick` and `blockchain_manager.tick`): [6](#0-5) 

The `JoinHandle` returned by `handle_stream` never completes. Each additional peer that closes its connection leaks another spinning task, compounding the starvation. Under enough such peers, ckBTC/ckDOGE block ingestion stalls entirely.

### Likelihood Explanation

The attacker only needs to be a Bitcoin P2P peer that the adapter connects to — any node on the public Bitcoin network qualifies. No privileged access, no key material, and no consensus-level corruption is required. The trigger is a single TCP `FIN`/`RST` after the version handshake, which is trivially achievable.

### Recommendation

Add `break` (or `return`) immediately after the `Disconnected` event is sent:

```rust
if disconnect {
    stream_event_sender
        .send(StreamEvent { address, kind: StreamEventKind::Disconnected })
        .await
        .ok();
    break;  // <-- missing
}
``` [7](#0-6) 

### Proof of Concept

1. Spawn a `TcpListener` on a local port.
2. Call `handle_stream` with that address as the peer.
3. Accept the connection on the listener side, perform the Bitcoin version handshake, then immediately drop the socket (closing the TCP connection).
4. Collect all `StreamEvent`s from the receiver with a timeout.
5. Assert that exactly one `StreamEventKind::Disconnected` is received and the `JoinHandle` completes within the timeout.

Currently, the `JoinHandle` never completes and the receiver accumulates an unbounded stream of `Disconnected` events until the channel fills and the task blocks permanently, matching the described starvation scenario.
