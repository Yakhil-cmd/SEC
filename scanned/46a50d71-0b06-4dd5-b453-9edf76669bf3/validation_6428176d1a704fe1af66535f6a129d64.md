### Title
Missing Callback-ID/Response-Type Cross-Check in `validate_self_validating_payload_impl` Allows Byzantine Block Proposer to Deliver `EmptyBlob` to Bitcoin Canister's `GetSuccessors` Callback — (`rs/bitcoin/consensus/src/payload_builder.rs`, `rs/replicated_state/src/bitcoin.rs`)

---

### Summary

A Byzantine subnet node (a single node, below the consensus fault threshold) can propose a block whose `SelfValidatingPayload` contains a `BitcoinAdapterResponse` with `response = SendTransactionResponse` but `callback_id = X`, where X is a live `BitcoinGetSuccessors` context. Validation passes because `validate_self_validating_payload_impl` performs no cross-check between the response variant and the request type stored in state. After finalization, `push_response` for `SendTransactionResponse` blindly enqueues `EmptyBlob` for callback_id X without verifying X belongs to a `SendTransaction` context. The bitcoin canister then receives `EmptyBlob` for its `GetSuccessors` callback, causing a trap and disrupting UTXO synchronization.

---

### Finding Description

**Guard 1 — `validate_self_validating_payload_impl`** is essentially a no-op:

```rust
fn validate_self_validating_payload_impl(
    &self,
    payload: &SelfValidatingPayload,
    validation_context: &ValidationContext,
) -> Result<NumBytes, SelfValidatingPayloadValidationError> {
    let since = Instant::now();
    if *payload == SelfValidatingPayload::default() {
        return Ok(0.into());
    }
    self.metrics.observe_validate_duration(VALIDATION_STATUS_VALID, since);
    let size = NumBytes::new(payload.count_bytes() as u64);
    Ok(size)
}
``` [1](#0-0) 

It checks only: (a) is the payload empty? (b) is it within the byte limit? It never checks whether each `BitcoinAdapterResponse.callback_id` exists in state, nor whether the response variant (`GetSuccessorsResponse` vs. `SendTransactionResponse`) matches the request type stored for that `callback_id`.

**Guard 2 — `validate_payload`** in `BatchPayloadBuilder` delegates entirely to `validate_self_validating_payload_impl` after a decode + size check: [2](#0-1) 

No callback_id or response-type validation is added here either.

**Guard 3 — `push_response` for `SendTransactionResponse`** has no callback_id validation at all:

```rust
BitcoinAdapterResponseWrapper::SendTransactionResponse(_) => {
    let callback_id = CallbackId::from(response.callback_id);
    let payload = Payload::Data(EmptyBlob.encode());
    state.consensus_queue.push(ConsensusResponse::new(callback_id, payload));
    Ok(())
}
``` [3](#0-2) 

It does not check whether `callback_id` exists in `bitcoin_send_transaction_internal_contexts`. It always returns `Ok(())` and always enqueues `EmptyBlob`.

**Contrast with `GetSuccessorsResponse`**, which does validate:

```rust
let context = state
    .metadata
    .subnet_call_context_manager
    .bitcoin_get_successors_contexts
    .get_mut(&callback_id)
    .ok_or_else(|| StateError::BitcoinNonMatchingResponse {
        callback_id: callback_id.get(),
    })?;
``` [4](#0-3) 

The `GetSuccessorsResponse` branch returns `Err(StateError::BitcoinNonMatchingResponse)` if the callback_id is not in `bitcoin_get_successors_contexts`. The `SendTransactionResponse` branch has no equivalent guard.

**`retrieve_context`** removes the `BitcoinGetSuccessors` context from `bitcoin_get_successors_contexts` when the execution environment processes the `consensus_queue` entry: [5](#0-4) 

This means once the malformed `EmptyBlob` response is processed, the `GetSuccessors` context is permanently removed from state, and no honest proposer can re-deliver the real response.

---

### Impact Explanation

The concrete impact is:

1. The bitcoin canister's `GetSuccessors` callback receives `EmptyBlob` (Candid-encoded `EmptyBlob`) instead of a `BitcoinGetSuccessorsResponse`. Decoding fails; the canister traps.
2. `retrieve_context` removes the `BitcoinGetSuccessors` context from `bitcoin_get_successors_contexts` as part of processing the `consensus_queue` entry.
3. `bitcoin_requests_iter` will no longer iterate over that callback_id, so no honest proposer will ever re-deliver the real response.
4. The bitcoin canister's UTXO chain-tip synchronization stalls for that request slot.

The claim of **false ckBTC minting** is not directly achievable from this path alone: the canister traps and state is rolled back for that callback execution. However, **permanent loss of a `GetSuccessors` request** and **stalling of UTXO synchronization** are concrete, deterministic outcomes.

---

### Likelihood Explanation

Requires a single Byzantine subnet node (below the consensus fault threshold), which is an explicitly in-scope attacker model per "protocol peer behavior below the consensus fault threshold." The attack is deterministic: propose one block with a crafted `SelfValidatingPayload`, and all honest nodes will accept it because `validate_payload` performs no type/callback_id cross-check.

---

### Recommendation

1. In `validate_self_validating_payload_impl`, load the certified state and for each `BitcoinAdapterResponse` in the payload, verify that `callback_id` exists in the correct context map (`bitcoin_get_successors_contexts` for `GetSuccessorsResponse`/`GetSuccessorsReject`, `bitcoin_send_transaction_internal_contexts` for `SendTransactionResponse`/`SendTransactionReject`). Return `InvalidArtifact` on mismatch.

2. In `push_response` for `SendTransactionResponse`, add the same guard that `GetSuccessorsResponse` has: look up `callback_id` in `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if not found.
