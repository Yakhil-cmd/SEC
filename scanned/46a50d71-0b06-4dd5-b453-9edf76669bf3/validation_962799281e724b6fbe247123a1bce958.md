### Title
Race Condition in `fail_stuck_upgrade_in_progress` Allows Any Caller to Prematurely Mark SNS Upgrade as Failed — (`rs/sns/governance/canister/canister.rs`)

### Summary
The SNS governance canister exposes `fail_stuck_upgrade_in_progress` as an unrestricted public `#[update]` endpoint. It marks an in-progress upgrade as failed based solely on a timestamp check (`now > mark_failed_at_seconds`), without first verifying whether the upgrade actually succeeded. This creates a race condition with `check_upgrade_status` (called from `run_periodic_tasks`): an unprivileged attacker can call `fail_stuck_upgrade_in_progress` after the deadline but before `check_upgrade_status` confirms success, causing the upgrade to be incorrectly marked as failed, leaving `deployed_version` stale, and triggering a panic in the `check_upgrade_status` callback.

### Finding Description

When an SNS upgrade proposal executes, `perform_upgrade_to_next_sns_version_legacy` sets `pending_version` with `mark_failed_at_seconds = now + 5 * 60`: [1](#0-0) 

The `check_upgrade_status` function (called from `run_periodic_tasks`) periodically queries `get_running_version` via an inter-canister call. Before the `await`, it increments `checking_upgrade_lock` to signal a check is in progress: [2](#0-1) 

After the `await` returns, it resets the lock with an unconditional `unwrap()`: [3](#0-2) 

The `fail_stuck_upgrade_in_progress` function is a public endpoint with **no access control**: [4](#0-3) 

Its implementation only checks the timestamp and then calls `complete_sns_upgrade_to_next_version` with `deployed_version = None` (i.e., without updating `deployed_version`): [5](#0-4) 

`complete_sns_upgrade_to_next_version` clears `pending_version` unconditionally: [6](#0-5) 

**Race sequence:**
1. An SNS upgrade proposal passes; `pending_version` is set with a 5-minute deadline.
2. The upgrade completes on-chain (canister is running the new version).
3. `check_upgrade_status` increments `checking_upgrade_lock` to 1 and suspends at the `await` for `get_running_version`.
4. `mark_failed_at_seconds` passes while the async call is in-flight.
5. An attacker calls `fail_stuck_upgrade_in_progress`; it sees `now > mark_failed_at_seconds`, clears `pending_version`, and marks the proposal as FAILED — all committed atomically.
6. The `check_upgrade_status` callback resumes and executes `self.proto.pending_version.as_mut().unwrap()` — **panics** because `pending_version` is now `None`.
7. The callback traps; its state changes are rolled back, but step 5's changes are already committed.

The comment in `check_upgrade_status` explicitly assumes `pending_version` cannot be cleared between `should_check_upgrade_status` and the `unwrap()` call — an assumption violated by `fail_stuck_upgrade_in_progress`: [7](#0-6) 

### Impact Explanation

- The upgrade proposal is incorrectly marked as **FAILED** even though the upgrade succeeded on-chain.
- `deployed_version` remains at the old version (stale), causing the SNS governance to have an incorrect view of the running software.
- Future `UpgradeSnsToNextVersion` proposals will compute upgrade steps from the wrong `deployed_version`, potentially causing them to fail or upgrade to an incorrect target.
- The `check_upgrade_status` callback panics, causing a one-time trap of `run_periodic_tasks` (reward distribution and other periodic tasks are skipped for that round).
- The `checking_upgrade_lock` field is left in an incremented state (the reset was rolled back), though this is moot since `pending_version` is cleared.

### Likelihood Explanation

- `fail_stuck_upgrade_in_progress` is callable by **any principal** with no authentication or authorization check.
- An attacker can query the governance canister's state to read `pending_version.mark_failed_at_seconds` and time the call precisely.
- The exploitable window is the interval between `mark_failed_at_seconds` passing and the next successful `check_upgrade_status` confirmation. Because `check_upgrade_status` makes an inter-canister call (which spans multiple rounds), this window can be several seconds to minutes — wide enough for a determined attacker to exploit.
- The attack is repeatable: every SNS upgrade proposal creates a new `pending_version` with a new deadline.

### Recommendation

1. **Add access control** to `fail_stuck_upgrade_in_progress`. Restrict callers to SNS governance neurons with sufficient voting power, or to the SNS root canister, analogous to how other privileged operations are gated.
2. **Check `checking_upgrade_lock` before proceeding**: if `checking_upgrade_lock > 0`, a status check is already in flight; `fail_stuck_upgrade_in_progress` should either return early or wait.
3. **Verify upgrade success before marking failure**: before calling `complete_sns_upgrade_to_next_version` with a failure status, `fail_stuck_upgrade_in_progress` should first attempt to confirm the running version (analogous to the bounty report's recommendation to call `_checkForUpgrade()` inside `closeBounty()` before checking the end timestamp).
4. **Defensive `unwrap` in callback**: replace the unconditional `unwrap()` after the `await` in `check_upgrade_status` with a graceful check, since `pending_version` can be externally cleared between the `await` suspension and the callback resumption.

### Proof of Concept

```
1. SNS upgrade proposal (UpgradeSnsToNextVersion) passes.
   → pending_version = { target_version: V2, mark_failed_at_seconds: T+300, checking_upgrade_lock: 0 }
   → deployed_version = V1

2. Canister upgrade to V2 completes on-chain.

3. run_periodic_tasks heartbeat fires → check_upgrade_status():
   - checking_upgrade_lock set to 1
   - inter-canister call to get_running_version() issued
   - canister suspends at `await`

4. Clock advances past T+300 (mark_failed_at_seconds).

5. Attacker sends: fail_stuck_upgrade_in_progress({})
   - now > mark_failed_at_seconds → true
   - complete_sns_upgrade_to_next_version(proposal_id, Failure, None) called
   - pending_version = None  [COMMITTED]
   - upgrade proposal marked FAILED  [COMMITTED]
   - deployed_version = V1 (unchanged)  [COMMITTED]

6. get_running_version() response arrives; check_upgrade_status callback resumes:
   - self.proto.pending_version.as_mut().unwrap() → PANIC (pending_version is None)
   - callback traps; all callback state changes rolled back

Result:
  - deployed_version = V1 (stale; canister actually runs V2)
  - pending_version = None
  - Upgrade proposal = FAILED (incorrect)
  - Future upgrade proposals compute steps from V1, not V2
```

    pub fn fail_stuck_upgrade_in_progress(
        &mut self,
        _: FailStuckUpgradeInProgressRequest,
    ) -> FailStuckUpgradeInProgressResponse {
        let pending_version = match self.proto.pending_version.as_ref() {
            None => return FailStuckUpgradeInProgressResponse {},
            Some(pending_version) => pending_version,
        };

        // Maybe, we should look at the checking_upgrade_lock field and only
        // proceed if it is false, or the request has force set to true.

        let now = self.env.now();

        if now > pending_version.mark_failed_at_seconds {
            let message = format!(
                "Upgrade marked as failed at {}. \
                Governance upgrade was manually aborted by calling fail_stuck_upgrade_in_progress \
                after mark_failed_at_seconds ({}). Setting upgrade to failed to unblock retry.",
                format_timestamp_for_humans(now),
                pending_version.mark_failed_at_seconds,
            );
            let status = upgrade_journal_entry::upgrade_outcome::Status::ExternalFailure(Empty {});

            self.complete_sns_upgrade_to_next_version(
                pending_version.proposal_id,
                status,
                message,
                None,
            );
        }

        FailStuckUpgradeInProgressResponse {}
    }
