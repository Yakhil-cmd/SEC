### Title
ckERC20 Minter Does Not Verify ERC20 `transfer()` Return Value, Enabling Silent Fund Loss for Non-Reverting Token Failures - (File: rs/ethereum/cketh/minter/src/state.rs)

---

### Summary

The ckETH minter's withdrawal finalization logic checks only the Ethereum transaction-level receipt `status` (0x1/0x0) when processing ckERC20 → ERC20 withdrawals. It never inspects the return value of the ERC20 `transfer()` call embedded in the transaction data, nor does it verify a `Transfer` event was emitted. For non-compliant ERC20 tokens that return `false` on failure without reverting, the minter incorrectly treats a failed token transfer as successful: it burns the user's ckERC20 tokens on the IC ledger, decrements its internal ERC20 balance, and issues no reimbursement.

---

### Finding Description

During a ckERC20 withdrawal, the minter constructs an EIP-1559 transaction whose `data` field encodes an ERC20 `transfer(address,uint256)` call using the selector `0xa9059cbb`. [1](#0-0) 

After the transaction is mined, `finalize_transactions_batch` retrieves the receipt via `eth_getTransactionReceipt` and records it. [2](#0-1) 

The finalization path calls `record_finalized_transaction`, which decides whether to reimburse the user based solely on `receipt.status == TransactionStatus::Failure`. [3](#0-2) 

Then `update_balance_upon_withdrawal` decrements the minter's internal ERC20 balance whenever `receipt.status == TransactionStatus::Success` and the transaction data is non-empty — with no check of the ERC20 `transfer()` return value or emitted logs. [4](#0-3) 

The `TransactionStatus` is derived exclusively from the receipt's `status` byte (0 = Failure, 1 = Success). [5](#0-4) 

For a non-compliant ERC20 token that returns `false` on transfer failure without reverting (e.g., older USDT, BNB, OMG), the Ethereum transaction status is `0x1` (no revert occurred), so the minter records `TransactionStatus::Success`. The actual token transfer never happened, but the minter has no mechanism to detect this: it does not decode the ABI return data of the `transfer()` call, and it does not inspect the `logs` field of the receipt for a `Transfer(from, to, value)` event.

---

### Impact Explanation

A user who calls `withdraw_erc20` with a supported non-compliant ERC20 token will have their ckERC20 tokens permanently burned on the IC ledger (via `icrc2_transfer_from` burn), but will not receive the corresponding ERC20 tokens at their Ethereum destination address. The minter's internal `erc20_balances` will be understated by the withdrawal amount, creating a permanent accounting discrepancy between the minter's on-chain ERC20 holdings and its internal state. No reimbursement is triggered because `record_finalized_transaction` only reimburses on `TransactionStatus::Failure`. The user suffers a total, unrecoverable loss of the withdrawn amount.

**Vulnerability class:** chain-fusion mint/burn/replay bug — incorrect finalization of ERC20 withdrawal leads to ckERC20 burn without ERC20 delivery and corrupted internal balance.

---

### Likelihood Explanation

The minter only processes governance-whitelisted ERC20 tokens. However, several widely-deployed ERC20 tokens are known to return `false` on failure rather than reverting. If any such token is added as a supported ckERC20 token, any unprivileged user calling `withdraw_erc20` can trigger this scenario by initiating a withdrawal that the ERC20 contract silently rejects (e.g., due to a transfer restriction, fee-on-transfer hook, or blacklist). The attacker-controlled entry path is the public `withdraw_erc20` endpoint. [6](#0-5) 

---

### Recommendation

The minter should verify that the ERC20 transfer actually succeeded by inspecting the receipt's `logs` field for a `Transfer(indexed from, indexed to, value)` event matching the minter's address as `from`, the user's destination as `to`, and the withdrawal amount as `value`. Only if such a log is present should the minter treat the withdrawal as successful and skip reimbursement. This mirrors the fix applied in the original report (SafeERC20's `_callOptionalReturn` pattern) but adapted to the post-hoc receipt-inspection model used by the minter.

---

### Proof of Concept

1. A non-compliant ERC20 token (returns `false` on failure, no revert) is added as a supported ckERC20 token via NNS governance proposal.
2. User calls `withdraw_erc20(ckerc20_ledger_id, amount, eth_destination)` on the minter.
3. Minter burns the user's ckERC20 tokens via `icrc2_transfer_from` and enqueues an Ethereum `transfer(eth_destination, amount)` transaction.
4. The ERC20 `transfer()` call returns `false` (e.g., destination is blacklisted), but the Ethereum transaction does not revert — receipt `status = 0x1`.
5. `finalize_transactions_batch` retrieves the receipt; `record_finalized_transaction` sees `TransactionStatus::Success` and does **not** enqueue a reimbursement request.
6. `update_balance_upon_withdrawal` calls `erc20_balances.erc20_sub(erc20_contract, withdrawal_amount)`, decrementing the minter's internal balance. [4](#0-3) 

7. The user's ckERC20 tokens are permanently burned; the ERC20 tokens remain at the minter's Ethereum address; the user receives nothing and has no reimbursement path.

#[derive(Clone, Eq, PartialEq, Debug)]
pub enum TransactionCallData {
    Erc20Transfer { to: Address, value: Erc20Value },
}

impl TransactionCallData {
    /// Encode the transaction call data to interact with an Ethereum smart contract.
    /// See the [Contract ABI Specification](https://docs.soliditylang.org/en/develop/abi-spec.html#contract-abi-specification).
    pub fn encode(&self) -> Vec<u8> {

            TransactionCallData::Erc20Transfer { to, value } => {

                data.extend(ERC_20_TRANSFER_FUNCTION_SELECTOR);
                data.extend(<[u8; 32]>::from(to));
                data.extend(value.to_be_bytes());
                data
            }

#[derive(Copy, Clone, Eq, PartialEq, Debug, Decode, Deserialize, Encode, Serialize)]
#[serde(try_from = "ethnum::u256", into = "ethnum::u256")]
pub enum TransactionStatus {
    /// Transaction was mined and executed successfully.
    #[n(0)]
    Success,

    /// Transaction was mined but execution failed (e.g., out-of-gas error).
    /// The amount of the transaction is returned to the sender but gas is consumed.
    /// Note that this is different from a transaction that is not mined at all: a failed transaction
    /// is part of the blockchain and the next transaction from the same sender should have an incremented
    /// transaction nonce.
    #[n(1)]
    Failure,
}

impl From<TransactionStatus> for ethnum::u256 {
    fn from(value: TransactionStatus) -> Self {
        match value {
            TransactionStatus::Success => ethnum::u256::ONE,
            TransactionStatus::Failure => ethnum::U256::ZERO,
        }
    }
}

impl TryFrom<u8> for TransactionStatus {
    type Error = String;

    fn try_from(value: u8) -> Result<Self, Self::Error> {
        match value {
            0 => Ok(TransactionStatus::Failure),
            1 => Ok(TransactionStatus::Success),
            _ => Err(format!("invalid transaction status: {value}")),
        }
    }
