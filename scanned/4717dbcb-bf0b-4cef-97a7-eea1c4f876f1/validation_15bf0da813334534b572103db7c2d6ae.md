### Title
Cycles Permanently Burned Before Canister-Status Check in `ValidSetRuleImpl::enqueue` — (`rs/messaging/src/scheduling/valid_set_rule.rs`)

### Summary
`ValidSetRuleImpl::enqueue` charges ingress induction cycles **before** checking whether the target canister is in `Stopping` or `Stopped` state. When the canister is `Stopping`, the charge is committed and the function returns `IngressInductionError::CanisterStopping` with no refund, permanently destroying cycles without inducting the message.

### Finding Description
In `enqueue` [1](#0-0) , `charge_ingress_induction_cost` is called unconditionally for any canister that is the fee payer. For a canister without a paused execution, this resolves to `consume_with_threshold` [2](#0-1) , which directly burns cycles from `system_state.cycles_balance`. Only **after** this irreversible deduction does the code check canister status [3](#0-2) . If the canister is `Stopping`, the function returns an error — but the cycles are already gone and there is no compensating credit anywhere in the error path.

### Impact Explanation
Any unprivileged principal can send a valid signed ingress to a canister that is in `Stopping` state. Each such message burns the ingress induction cost (base fee + per-byte fee) from the canister's balance without inducting the message. Because the `Stopping` window can be arbitrarily long (it persists until all outstanding inter-canister calls complete), an attacker can repeatedly send messages during this window to drain the canister's cycles. The canister owner has no way to prevent this: they cannot accept the messages (canister is stopping) and cannot avoid the charge (it happens before the status check).

### Likelihood Explanation
The `Stopping` state is reachable by any canister whose controller calls `stop_canister`. The window is observable on-chain via `canister_status`. The attack requires only a valid signed ingress — no privileged access, no key material, no consensus manipulation. It is fully local-testable with a state-machine test.

### Recommendation
Move the canister-status check to **before** `charge_ingress_induction_cost` in `enqueue`. The corrected order should be:

1. Resolve payer canister (already done at line 295).
2. **Check `canister.status()`** — return `CanisterStopping`/`CanisterStopped` immediately if not `Running`.
3. Only then call `charge_ingress_induction_cost`.
4. Call `state.push_ingress(ingress)`. [4](#0-3) 

### Proof of Concept
State-machine test outline:

```rust
// 1. Create canister with exactly threshold + induction_cost cycles.
// 2. Call stop_canister to put it in Stopping state (do not complete outstanding calls).
// 3. Record balance_before = canister.system_state.balance().
// 4. Submit a valid signed ingress to the canister via induct_messages().
// 5. Assert the returned IngressStatus is Failed(CanisterStopping).
// 6. Record balance_after = canister.system_state.balance().
// 7. Assert balance_before == balance_after  <-- currently FAILS; cycles are deducted.
```

The existing test `canister_on_application_subnet_charges_for_ingress` [5](#0-4)  demonstrates the test harness pattern. A parallel test for a `Stopping` canister would reproduce the bug deterministically.

fn canister_on_application_subnet_charges_for_ingress() {

        .canister_id(canister_test_id(0))

    let balance_before = state
        .canister_state(&canister_test_id(0))
        .unwrap()
        .system_state
