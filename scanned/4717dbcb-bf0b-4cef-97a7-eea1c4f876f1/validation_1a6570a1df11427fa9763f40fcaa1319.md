### Title
Silent Clamping of Over-Max `icrc2_approve` Amount to `Tokens::MAX` — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
An unprivileged caller can invoke `icrc2_approve` with a `Nat` amount exceeding `u64::MAX`. Instead of being rejected, the conversion error is silently swallowed and replaced with `Tokens::max_value()` (`u64::MAX`), granting the spender an allowance of `u64::MAX` regardless of the owner's intent.

### Finding Description
`TryFrom<Nat> for Tokens` correctly returns `Err` when the `Nat` value does not fit in a `u64`: [1](#0-0) 

However, the caller in `icrc2_approve_not_async` discards that error via `unwrap_or_else`: [2](#0-1) 

```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```

The resulting `amount` (silently set to `u64::MAX`) is then embedded directly into the `Operation::Approve` transaction and committed to the ledger: [3](#0-2) 

The public `#[update]` endpoint `icrc2_approve` is reachable by any principal with no privilege requirement: [4](#0-3) 

### Impact Explanation
Any account holder can grant a spender an allowance of `u64::MAX` (~18.4 × 10¹⁸ e8s, i.e., effectively unlimited) by encoding `amount = Nat(u64::MAX + 1)` in a Candid `ApproveArgs`. The spender can then call `icrc2_transfer_from` to drain the approver's entire balance. The invariant "stored allowance equals requested amount, or the call fails" is broken.

### Likelihood Explanation
The attack requires only a single ingress `icrc2_approve` call with a crafted `Nat` value. No privileged access, key material, or governance majority is needed. The Candid encoding of an arbitrary-precision `Nat` larger than `u64::MAX` is trivially constructable by any client.

### Recommendation
Replace the silent fallback with an explicit error return:

```rust
let amount = Tokens::try_from(arg.amount)
    .map_err(|_| ApproveError::GenericError {
        error_code: Nat::from(0u64),
        message: "amount exceeds maximum token value".to_string(),
    })?;
```

The `expected_allowance` branch at lines 842–853 already demonstrates the correct pattern — it returns `Err(ApproveError::AllowanceChanged { ... })` on conversion failure rather than substituting a fallback value. [5](#0-4) 

### Proof of Concept
1. Encode `ApproveArgs { amount: Nat(2^64), spender: attacker_account, from_subaccount: None, ... }`.
2. Submit via `icrc2_approve` ingress call — the call succeeds and returns a block index.
3. Query `icrc2_allowance` for `(owner, attacker_account)` — the returned allowance is `u64::MAX`, not `2^64` and not an error.
4. Call `icrc2_transfer_from` from the attacker's principal to transfer the owner's full balance to any destination.
