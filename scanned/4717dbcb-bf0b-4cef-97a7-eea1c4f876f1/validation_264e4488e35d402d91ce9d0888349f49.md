### Title
Quarantined Withdrawal Reimbursement State Permanently Locks User ckBTC With No User-Accessible Recovery Path - (File: `rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`)

### Summary
The ckBTC minter's `QuarantinedWithdrawalReimbursement` state, triggered when a panic occurs during the `reimburse_withdrawals` async callback, permanently locks a user's ckBTC with no user-callable recovery path. Once a reimbursement is quarantined, the user's ckBTC has been burned, no BTC was sent, no ckBTC is reminted, and the status is silently set to `RetrieveBtcStatusV2::Unknown`. Recovery requires a canister upgrade via NNS governance — an action entirely outside the user's control.

### Finding Description

**Normal withdrawal-reimbursement flow:**

1. User calls `retrieve_btc_with_approval` → ckBTC is burned on the ledger.
2. If the withdrawal fails (e.g., `TooManyInputs`), a `ReimburseWithdrawalTask` is inserted into `pending_withdrawal_reimbursements`.
3. The periodic task `reimburse_withdrawals` iterates over pending reimbursements and calls `runtime.mint_ckbtc(...)` to return the ckBTC to the user.

**The quarantine path:**

Before calling `mint_ckbtc`, a `scopeguard` is armed:

```rust
let prevent_double_minting_guard = scopeguard::guard(burn_index, |index| {
    state::mutate_state(|s| {
        state::audit::quarantine_withdrawal_reimbursement(s, index, runtime)
    });
});
``` [1](#0-0) 

If a panic occurs anywhere in the callback after this guard is armed (including inside `reimburse_withdrawal_completed` after a successful `mint_ckbtc`), the IC rolls back all state mutations in the callback, then fires the cleanup callback which executes the scopeguard. The scopeguard calls `quarantine_withdrawal_reimbursement`:

```rust
pub fn quarantine_withdrawal_reimbursement(&mut self, burn_index: LedgerBurnIndex) {
    self.pending_withdrawal_reimbursements.remove(&burn_index);
    self.reimbursed_withdrawals
        .insert(burn_index, Err(ReimbursedError::Quarantined));
}
``` [2](#0-1) 

The event log records this as a terminal state:

```
/// The minter unexpectedly panicked while processing a reimbursement.
/// The reimbursement is quarantined to prevent any double minting and
/// will not be processed without further manual intervention.
QuarantinedWithdrawalReimbursement { burn_block_index: u64 }
``` [3](#0-2) 

**The locked state has no user-accessible exit.** After quarantine:
- The entry is removed from `pending_withdrawal_reimbursements` — the periodic task will never retry it.
- The entry is stored as `Err(ReimbursedError::Quarantined)` in `reimbursed_withdrawals`.
- `retrieve_btc_status_v2` returns `RetrieveBtcStatusV2::Unknown` — the user cannot even observe the quarantine. [4](#0-3) 

There is no `retry_quarantined_reimbursement` endpoint, no `claim_quarantined` function, and no state transition back to `pending`. The `ReimbursedError::Quarantined` variant is defined as a permanent error:

```rust
pub enum ReimbursedError {
    /// Whether reimbursement was minted or not is unknown...
    /// will not be further processed without manual intervention.
    Quarantined,
}
``` [5](#0-4) 

The identical pattern exists in the ckDOGE minter: [6](#0-5) 

### Impact Explanation

A user whose withdrawal reimbursement is quarantined has:
- Their ckBTC permanently burned (the ledger burn is committed and irreversible).
- No BTC sent to their Bitcoin address.
- No ckBTC reminted.
- No visible status — `retrieve_btc_status_v2` returns `Unknown`, giving no indication that funds are stuck.
- No callable function to recover funds.

This is a **ledger conservation bug**: ckBTC supply is reduced (burn) without a corresponding BTC output or ckBTC remint, breaking the 1:1 backing invariant for the affected user. Recovery requires an NNS governance proposal to upgrade the minter canister — a process entirely outside the user's control and not guaranteed to be executed promptly.

### Likelihood Explanation

The quarantine fires on any panic inside the `reimburse_withdrawals` callback. Concrete panic paths exist in `reimburse_withdrawal_completed`:

```rust
assert_ne!(burn_index, mint_index, "BUG: mint index cannot be the same as the burn index");
// ...
.unwrap_or_else(|| panic!("BUG: missing pending reimbursement of withdrawal {burn_index}."))
// ...
assert_eq!(..., None, "BUG: Reimbursement of withdrawal {reimbursement:?} was already completed!")
``` [7](#0-6) 

If `mint_ckbtc` returns `Ok` and then `reimburse_withdrawal_completed` panics (e.g., ledger returns a block index equal to the burn index, or state is inconsistent due to a concurrent upgrade), the ckBTC is already minted on the ledger but the minter state is rolled back and then quarantined. In this specific sub-case the user retains their ckBTC but the minter state is permanently inconsistent. For the user to lose funds, the panic must occur after `mint_ckbtc` returns `Err` — the error path is simpler but not immune to panics from logging infrastructure or future code changes. The ckBTC minter has had multiple real-world stuck withdrawal incidents requiring emergency upgrades, confirming this class of failure is operationally realistic. [8](#0-7) 

Likelihood: **Low-to-Medium** — the panic trigger requires a code bug or unexpected runtime condition, but the code explicitly anticipates and handles this scenario, and real-world stuck withdrawals have occurred.

### Recommendation

1. **Add a user-callable `retry_quarantined_reimbursement(burn_index)` endpoint** that allows the original burner to re-trigger minting for a quarantined reimbursement, guarded by a check that the caller matches the original `reimbursement.account.owner`.
2. **Expose quarantine status** in `retrieve_btc_status_v2` — return a distinct `Quarantined` variant instead of `Unknown` so users can identify affected withdrawals and seek support.
3. **Replace `assert!`/`panic!` calls in `reimburse_withdrawal_completed`** with recoverable errors that log a critical metric and skip the entry rather than panicking, preventing the scopeguard from firing on recoverable state inconsistencies.
