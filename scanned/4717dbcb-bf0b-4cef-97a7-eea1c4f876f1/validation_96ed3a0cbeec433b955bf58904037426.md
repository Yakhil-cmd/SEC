### Title
Broken Idempotency Guard in `settle_neurons_fund_participation` When NNS Returns Zero NF Participants — (`rs/sns/swap/src/swap.rs`)

### Summary

The idempotency guard protecting `settle_neurons_fund_participation` from being called more than once relies on `self.cf_participants` being non-empty after a successful settlement. When NNS governance returns an empty `neurons_fund_neuron_portions` list (zero NF participants), the guard fails to detect prior completion, allowing a second call to NNS governance on a subsequent `finalize_swap` invocation.

### Finding Description

The guard at line 2329 is:

```rust
if !self.cf_participants.is_empty() {
    // already settled, return early
}
``` [1](#0-0) 

After a successful call where NNS governance returns an empty `neurons_fund_neuron_portions` list, the processing loop at line 2440 iterates over nothing, `cf_participant_map` stays empty, and the assignment at line 2485 writes an empty `Vec` back to `self.cf_participants`: [2](#0-1) 

So after the first successful settlement with zero NF participants:
- `self.cf_participants` remains `[]` (empty)
- The guard `!self.cf_participants.is_empty()` evaluates to `false`
- The function does **not** short-circuit on the next call
- A second `settle_neurons_fund_participation` RPC is issued to NNS governance

The finalize lock (mentioned in the doc comment at line 2309) only prevents **concurrent** calls. Once the first `finalize_swap` completes and releases the lock, a second sequential call by any unprivileged caller proceeds through the full flow again. [3](#0-2) 

There is also a post-await interleaving guard at line 2396, but it only fires if `cf_participants` is non-empty — which it never is in this scenario: [4](#0-3) 

### Impact Explanation

The invariant stated in the code's own documentation — that NNS governance must be called **at most once** per swap finalization — is violated. A second `settle_neurons_fund_participation` call is issued to NNS governance for the same committed swap. If NNS governance is not itself idempotent for this call (e.g., it processes the committed result a second time and mints ICP again), this results in double-minting of ICP to the swap canister's account. Even if NNS governance is idempotent, the swap canister's own invariant is broken and the behavior is undefined.

### Likelihood Explanation

- `finalize_swap` is publicly callable by any unprivileged principal.
- The triggering condition (NNS governance returning zero NF participants for a committed swap with `neurons_fund_participation=true`) is a legitimate, reachable state — it occurs when no NNS neurons chose to participate in the Neurons' Fund for this SNS.
- No special access, key material, or majority corruption is required.
- The path is fully local-testable via a state-machine test with a mock NNS governance client.

### Recommendation

Replace the idempotency guard with a dedicated boolean flag (e.g., `neurons_fund_participation_settled: bool`) that is set to `true` after any successful call to NNS governance, regardless of whether the returned participant list is empty. The guard should check this flag rather than `cf_participants.is_empty()`.

Alternatively, distinguish between "never settled" and "settled with zero participants" by using `Option<Vec<CfParticipant>>` for `cf_participants`, where `None` means unsettled and `Some([])` means settled with zero participants.

### Proof of Concept

1. Initialize a swap with `neurons_fund_participation = true`.
2. Commit the swap (lifecycle → `Committed`).
3. Configure the mock NNS governance client to return `Ok` with an empty `neurons_fund_neuron_portions` list.
4. Call `finalize_swap` — it completes successfully; `self.cf_participants` is `[]`.
5. Call `finalize_swap` a second time — the guard at line 2329 does not fire; NNS governance client receives a **second** `settle_neurons_fund_participation` call.
6. Assert the mock NNS governance client was called exactly once — the assertion **fails**, confirming the bug. [5](#0-4)
