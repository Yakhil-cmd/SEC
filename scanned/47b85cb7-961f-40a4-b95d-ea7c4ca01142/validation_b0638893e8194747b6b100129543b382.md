### Title
Rule Disclosure Cannot Be Revoked — (`rs/boundary_node/rate_limits/canister/disclose.rs`)

### Summary
The `disclose_rules` update method in the rate-limits canister permanently sets `disclosed_at` on rules and `is_disclosed = true` on incidents with no corresponding revoke/un-disclose function. Once the authorized principal accidentally discloses a rule or incident, the disclosure is irreversible at runtime, and a secondary policy check permanently blocks new rules from being linked to that incident.

### Finding Description
In `rs/boundary_node/rate_limits/canister/disclose.rs`, the `disclose_rules` function performs two one-way state mutations:

```rust
// line 74
rule.disclosed_at = Some(time);
canister_api.upsert_rule(*rule_id, rule);
``` [1](#0-0) 

```rust
// line 105
incident.is_disclosed = true;
canister_api.upsert_incident(incident_id, incident);
``` [2](#0-1) 

Neither `disclosed_at` nor `is_disclosed` can ever be reset. The `CanisterApi` trait exposes only `upsert_rule` and `upsert_incident` — there is no `remove_rule`, `remove_incident`, or any "un-disclose" method. [3](#0-2) 

A second, compounding consequence is enforced in `add_config`: any attempt to add a new rule that references an already-disclosed incident is permanently rejected with `AddConfigError::LinkingRuleToDisclosedIncident`. [4](#0-3) 

The access-control model grants `RestrictedRead` to ordinary callers (rule content hidden) and `FullRead` to API boundary nodes. Disclosure promotes a rule from hidden to fully visible for all callers. [5](#0-4) 

The `authorized_principal` — the only identity permitted to call `disclose_rules` — is a single principal set at `init`/`post_upgrade` and cannot be rotated at runtime without a canister upgrade. [6](#0-5) 

### Impact Explanation
1. **Permanent rule exposure**: Once a rule is disclosed, its raw bytes and description are visible to every ingress caller forever. An attacker who queries `get_config` or `get_rules_by_incident_id` after an accidental disclosure obtains the exact rate-limit logic and can craft requests that evade it.
2. **Permanent incident lock-out**: Once an incident is disclosed, `add_config` will reject any future config that references that incident ID. The operator must create a new incident UUID to add follow-up rules, breaking the incident-tracking workflow and potentially leaving a gap in coverage while the new incident is wired up.
3. There is no runtime recovery path; the only remediation is a full canister upgrade (requiring an NNS governance proposal), which is a heavyweight, time-delayed operation.

### Likelihood Explanation
The `authorized_principal` is a human-operated or automated key that calls `disclose_rules` to publish resolved incidents. Operational mistakes — disclosing the wrong rule ID, disclosing an incident before the corresponding mitigation is fully deployed, or disclosing under time pressure during an active attack — are realistic. Because the canister provides no "undo" path, any such mistake has permanent consequences. The pattern is identical to the original finding: a privileged setter exists (`disclose_rules`) but the corresponding revoke operation (`undisclose_rules`) does not.

### Recommendation
Add a privileged `undisclose_rules` (or `set_rule_disclosure`) update method that allows the `authorized_principal` to reset `disclosed_at` to `None` and `is_disclosed` to `false`, mirroring the `adjustBurningAccess(address, bool)` fix recommended in the original report. The `add_config` policy check for `LinkingRuleToDisclosedIncident` should also be re-evaluated once un-disclosure is possible.

```rust
fn undisclose_rules(
    &self,
    arg: rate_limits_api::UndiscloseRulesArg,
) -> Result<(), UndiscloseRulesError> {
    // reset disclosed_at = None and is_disclosed = false
}
```

### Proof of Concept
1. The authorized principal calls `disclose_rules(DiscloseRulesArg::IncidentIds([incident_id_X]))` for an incident whose mitigation is not yet fully deployed.
2. `incident.is_disclosed` is set to `true` and all associated rules have `disclosed_at` stamped.
3. Any ingress caller now queries `get_config` and receives the full `rule_raw` bytes for those rules.
4. The authorized principal realizes the mistake and attempts to call `disclose_rules` again — there is no inverse method; the call is a no-op on already-disclosed rules (`if rule.disclosed_at.is_none()` guard). [1](#0-0) 
5. Any subsequent `add_config` call that references `incident_id_X` is permanently rejected with `LinkingRuleToDisclosedIncident`, blocking follow-up rule updates for that incident. [4](#0-3) 
6. The attacker uses the exposed `rule_raw` to understand the exact matching criteria and crafts traffic that bypasses the rate-limit rule, evading boundary-node protection until a new incident is created and a canister upgrade is executed.
