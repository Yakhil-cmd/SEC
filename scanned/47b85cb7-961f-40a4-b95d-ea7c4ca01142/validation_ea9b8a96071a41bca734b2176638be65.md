### Title
SNS Treasury Manager Deposit/Withdraw Exposes SNS Treasury to Unlimited Slippage — (`rs/sns/governance/src/extensions.rs`, `rs/sns/treasury_manager/treasury_manager.did`)

### Summary

The SNS governance canister's `execute_treasury_manager_deposit` and `execute_treasury_manager_withdraw` functions, together with the `TreasuryManager` Candid interface, perform DEX liquidity pool deposits and withdrawals with no slippage protection. Neither the `DepositRequest` type nor the `WithdrawRequest` type carries a minimum-received-amount parameter, and the governance execution path never validates the returned balances against any floor. A malicious actor who can manipulate the DEX pool reserves between proposal approval and execution can cause the SNS treasury to receive arbitrarily fewer LP tokens (or fewer underlying assets on withdrawal) than the governance voters expected.

### Finding Description

The `DepositRequest` type in the canonical `TreasuryManager` Candid interface contains only `allowances` — the amounts to deposit — with no `min_lp_tokens_out` or equivalent field: [1](#0-0) 

The `WithdrawRequest` type is similarly bare: [2](#0-1) 

The interface itself acknowledges this as a "Known Security Risk": [3](#0-2) 

On the governance side, `ValidatedDepositOperationArg` — the struct that carries all validated deposit parameters — contains only the amounts to send, with no minimum-received field: [4](#0-3) 

The validation step (`validate_deposit_operation_impl`) only checks that the requested amounts do not exceed 50% of the current treasury balance; it performs no slippage check: [5](#0-4) 

At execution time, `execute_treasury_manager_deposit` approves the treasury manager, calls `deposit`, receives back a `Balances` struct, logs it, and returns `Ok(())` — it never compares the returned balances against any minimum: [6](#0-5) 

`execute_treasury_manager_withdraw` has the same structure — it calls `withdraw`, receives balances, and returns without any floor check: [7](#0-6) 

The proposal rendering for `RegisterExtension` even explicitly warns voters about this gap, confirming the design is aware of the risk but has not closed it at the protocol level: [8](#0-7) 

The KongSwap adaptor (`kongswap-adaptor-canister` v1.0.0) is already referenced as a production artifact in the build system, meaning this code path is live: [9](#0-8) 

### Impact Explanation

An attacker who can move the KongSwap (or any blessed DEX) pool reserves — which any unprivileged user can do by making ordinary swaps — between the moment an SNS governance deposit proposal is adopted and the moment the governance canister executes it can cause the SNS treasury to receive a fraction of the LP tokens that voters expected. On withdrawal, the same manipulation in the opposite direction causes the treasury to receive fewer underlying tokens back. In both cases, the difference is silently lost: the governance canister returns `Ok(())` and logs the actual (degraded) balances without reverting or alerting. This is a direct, permanent loss of SNS treasury funds with no on-chain recourse.

### Likelihood Explanation

The attack window is the latency between proposal adoption and canister execution, which on the IC is at least one round-trip (seconds to minutes). Any principal with enough tokens to move the DEX pool can exploit this window. The attack is profitable whenever the sandwich profit exceeds the DEX trading fees, which is routinely achievable for pools with low liquidity relative to the treasury deposit size. SNS treasuries are publicly visible, so the deposit size and timing are known in advance. Likelihood is **medium**: the attack requires active monitoring and capital, but no privileged access.

### Recommendation

1. **Add a `min_lp_tokens_out` (or `min_amount_out`) field to `DepositRequest` and a `min_amount_out` field to `WithdrawRequest`** in `treasury_manager.did`, making slippage limits a first-class part of the interface.
2. **Extend `ValidatedDepositOperationArg`** to carry the minimum acceptable LP tokens, validated at proposal submission time against a reasonable price oracle or a governance-specified tolerance.
3. **In `execute_treasury_manager_deposit` and `execute_treasury_manager_withdraw`**, compare the returned `Balances` against the minimum and return `Err` (triggering a governance failure record) if the floor is not met.
4. Until the interface is updated, document the risk prominently in the `ExecuteExtensionOperation` proposal rendering (analogous to the existing `RegisterExtension` warning) so voters can make informed decisions.

### Proof of Concept

1. An SNS DAO submits and adopts a governance proposal: `ExecuteExtensionOperation { operation_name: "deposit", operation_arg: { treasury_allocation_sns_e8s: 10_000_000, treasury_allocation_icp_e8s: 5_000_000 } }`.
2. The proposal enters the execution queue. Before the governance canister fires the inter-canister call to the KongSwap adaptor, the attacker executes a large swap on KongSwap that skews the SNS/ICP price ratio significantly (e.g., dumps SNS tokens to make SNS appear cheap).
3. `execute_treasury_manager_deposit` calls `governance.approve_treasury_manager(...)` then `call_canister(..., "deposit", ...)`. The adaptor deposits at the manipulated price and receives far fewer LP tokens than the market rate would imply.
4. The adaptor returns a `Balances` struct reflecting the degraded position. `execute_treasury_manager_deposit` logs it and returns `Ok(())` — no error, no revert.
5. The attacker reverses the swap (sandwich), extracting profit at the SNS treasury's expense. The treasury's LP position is permanently worth less than the deposited assets. [10](#0-9) [11](#0-10)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
