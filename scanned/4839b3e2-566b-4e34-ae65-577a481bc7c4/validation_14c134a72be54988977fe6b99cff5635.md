### Title
Silent ICP Burn Failure in Cycles Minting Canister Allows Cycles Minting Without ICP Conservation — (`rs/nns/cmc/src/main.rs`)

---

### Summary

The `burn_and_log` function in the Cycles Minting Canister (CMC) intentionally swallows errors from the ICP burn transfer after cycles have already been deposited or minted. If the ICP ledger returns a transient error at that moment, cycles are created without the corresponding ICP being burned, violating the ICP-to-cycles conservation invariant.

---

### Finding Description

In `rs/nns/cmc/src/main.rs`, the function `burn_and_log` is responsible for burning the ICP held in a CMC subaccount after cycles have been successfully deposited. The function explicitly does not propagate errors:

/// notification because then it could be retried.

    ...
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;
    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),

            let code = code as i32;

``` [1](#0-0) 

This function is called in three success paths, all after cycles have already been irrevocably deposited or minted:

- `process_top_up` — after `deposit_cycles` succeeds
- `process_create_canister` — after `do_create_canister` succeeds
- `process_mint_cycles` — after `do_mint_cycles` succeeds [2](#0-1) [3](#0-2) [4](#0-3) 

In all three cases, the notification block is subsequently marked as fully processed (`NotifiedTopUp`, `NotifiedCreateCanister`, or `NotifiedMint`), preventing any retry. If `burn_and_log` fails silently, the ICP remains in the CMC's subaccount indefinitely — unburned and unrecoverable — while the cycles have already been delivered.

The design rationale (stated in the comment) is to avoid double-minting: if the burn failure caused the notification to return an error, the caller could retry and receive cycles again. However, the consequence of the chosen design is that a transient ledger error at this exact point causes a permanent conservation violation.

---

### Impact Explanation

**Ledger conservation bug.** Cycles are minted without the corresponding ICP being burned. The ICP total supply is not reduced as required by the economic invariant that every cycle is backed by burned ICP. The unburned ICP accumulates in the CMC's per-canister or per-principal subaccounts. Over time, repeated transient ledger failures during `burn_and_log` inflate the effective ICP supply relative to cycles in circulation.

---

### Likelihood Explanation

The ICP ledger can legitimately return a transient error (e.g., during a scheduled canister upgrade, or under high message load causing `TemporarilyUnavailable`). The window is narrow — the failure must occur after cycles are deposited but during the `burn_and_log` call — but it is a real operational condition, not a theoretical one. Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during such a window triggers the condition without any special privilege.

---

### Recommendation

1. **Persist failed burns and retry them.** Store the `(subaccount, amount)` pair in stable state when `burn_and_log` fails, and retry in a heartbeat or timer until the burn succeeds.
2. **Alternatively**, expose a privileged endpoint that allows governance to trigger a re-burn for a specific subaccount, so the conservation invariant can be restored after a ledger outage.
3. At minimum, emit a certified metric or counter for failed burns so that conservation violations are observable and auditable.

---

### Proof of Concept

1. User sends ICP to the CMC subaccount corresponding to canister `C` with `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id: C }`.
3. CMC calls `deposit_cycles(C, cycles, ...)` — this succeeds; cycles are now in canister `C`.
4. CMC calls `burn_and_log(sub, amount)` — the ICP ledger is temporarily unavailable (e.g., mid-upgrade) and returns an error.
5. `burn_and_log` logs the error and returns `()` — no error is propagated.
6. CMC records `NotifiedTopUp(Ok(cycles))` for the block — the notification is permanently closed.
7. Result: canister `C` received cycles; the ICP in the CMC subaccount is never burned. The ICP supply is inflated by `amount`. [5](#0-4)

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
