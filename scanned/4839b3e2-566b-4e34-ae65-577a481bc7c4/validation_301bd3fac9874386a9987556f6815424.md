### Title
`sender_info.sig` Included in MessageId Computation Enables Deduplication Bypass via Multiple Valid Canister Signatures — (`rs/types/types/src/messages/http.rs`)

### Summary

The `sender_info.sig` field (a canister signature over the `info` blob) is included in the `representation_independent_hash_call_or_query` computation that produces the `MessageId`. Because the IC canister signature scheme is non-deterministic across rounds (the same `(canister_id, seed, msg)` triple can produce multiple distinct valid `sig` byte strings using different IC certificates), an attacker who controls the signer canister can produce two logically identical ingress messages with different `sender_info.sig` values, yielding two distinct `MessageId`s. Both messages pass all validation, are not deduplicated against each other, and are both executed.

### Finding Description

**Root cause — `sig` in the hash:**

In `rs/types/types/src/messages/http.rs`, `representation_independent_hash_call_or_query` inserts the full `sender_info` nested map — including `sig` — into the hash:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, sig }) = sender_info {
    map.insert(
        "sender_info",
        Map(btreemap! {
            "info" => Bytes(info),
            "signer" => Bytes(signer),
            "sig" => Bytes(sig),   // ← signature bytes affect MessageId
        }),
    );
}
``` [1](#0-0) 

This is consumed by `HttpCanisterUpdate::id()` and `SignedIngressContent::id()`: [2](#0-1) [3](#0-2) 

**Deduplication is keyed on `MessageId`:**

`ValidSetRuleImpl::is_duplicate` checks `state.get_ingress_status(&msg.content().id())`. Two messages with different `sig` bytes produce different `id()` values and are treated as distinct messages: [4](#0-3) 

`IngressHistorySet::contains` in the ingress selector does the same: [5](#0-4) 

**Canister signatures are non-deterministic across rounds:**

`rs/crypto/iccsa/src/api.rs` shows that a canister signature is valid if the IC certificate covers the canister's `certified_data` and the witness tree contains an empty leaf at `["sig", SHA256(seed), SHA256(msg)]`. The certificate itself (which is part of the `sig` bytes) changes every IC round. Two certificates issued at different rounds for the same `certified_data` both pass `verify_certified_data`, producing two distinct but valid `sig` byte strings for the same `(seed, msg)` pair: [6](#0-5) [7](#0-6) 

**`sender_info.sig` is verified against `sender_pubkey`:**

`verify_sender_info_canister_sig` verifies `sender_info.sig` using the envelope-level `sender_pubkey` (which encodes `(canister_id, seed)`). The `sender_info.signer` must match the canister_id in `sender_pubkey`: [8](#0-7) 

**Concrete attack steps:**

1. Attacker deploys canister `C` on the IC. The `sender_pubkey` encodes `(C, seed)`. The `sender` principal is `SHA224(DER(sender_pubkey)) || 0x02`.
2. `C` certifies path `["sig", SHA256(seed), SHA256(info_blob)]` in its `certified_data`. At round T1, the IC issues certificate `cert1` → attacker constructs `sig_info_1 = CBOR{cert1, tree}`.
3. At round T2 (≥1 second later), the IC issues certificate `cert2` for the same `certified_data` → attacker constructs `sig_info_2 = CBOR{cert2, tree}`. Both `sig_info_1` and `sig_info_2` pass `verify_sender_info_canister_sig`.
4. `MessageId1 = hash(canister_id, method, arg, expiry, sender, {info, C, sig_info_1})` and `MessageId2 = hash(..., sig_info_2)` are distinct.
5. `C` certifies `["sig", SHA256(seed), SHA256(MessageId1)]` and `["sig", SHA256(seed), SHA256(MessageId2)]` → valid envelope signatures for both messages.
6. Both messages are submitted. Both pass `validate_request_content` (expiry, envelope sig, sender_info sig all valid). Both have distinct `MessageId`s. Neither is in the ingress history when the other is checked. Both are inducted and executed.

### Impact Explanation

Any canister that relies on the IC's `MessageId`-based deduplication (via ingress history) to prevent double-execution of a state-mutating call is vulnerable. If the target canister performs a financial operation (minting cycles, crediting tokens, claiming rewards) without its own application-level deduplication, the attacker can execute the same logical operation twice. The ICP ledger has its own `created_at_time`/`memo` deduplication and is not directly affected, but application canisters that trust `MessageId` uniqueness are.

### Likelihood Explanation

The attacker only needs to deploy a canister on the IC (unprivileged, costs cycles) and wait one IC round (~1 second) between obtaining the two certificates. No privileged access, no key compromise, no threshold attack is required. The `sender_info` feature is actively tested and deployed.

### Recommendation

Remove `sig` from the `MessageId` computation. Only `info` and `signer` carry logical message identity; `sig` is a proof of `info` and should not affect the hash. The fix is to change the nested map in `representation_independent_hash_call_or_query` to:

```rust
Map(btreemap! {
    "info"   => Bytes(info),
    "signer" => Bytes(signer),
    // sig intentionally excluded
})
``` [1](#0-0) 

### Proof of Concept

```
1. Deploy canister C on the IC.
2. Have C set certified_data = root_hash(tree) where tree contains
   leaf at ["sig", SHA256(seed), SHA256(info_blob)] = "".
3. Fetch IC certificate cert1 at round T1 → build sig_info_1.
4. Wait one round. Fetch IC certificate cert2 at round T2 → build sig_info_2.
5. Assert sig_info_1 != sig_info_2 (different cert timestamps/signatures).
6. Assert both pass ic_crypto_iccsa::verify(info_blob, sig_info_1, pk, root_pk) == Ok(())
   and ic_crypto_iccsa::verify(info_blob, sig_info_2, pk, root_pk) == Ok(()).
7. Compute MessageId1 and MessageId2 — assert MessageId1 != MessageId2.
8. Build envelope signatures for both MessageIds using C.
9. Submit both to the target canister — observe both are executed.
```

fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
