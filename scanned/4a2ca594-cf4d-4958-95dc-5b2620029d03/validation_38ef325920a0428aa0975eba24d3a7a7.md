Based on my investigation, I found a clear analog in the ckBTC minter. Let me present the finding.

### Title
`retrieve_btc_account_to_block_indices` Never Pruned on Request Finalization — Stale Index Causes Unbounded Growth and Misleading Status Queries - (File: `rs/bitcoin/ckbtc/minter/src/state/audit.rs`)

### Summary

The ckBTC minter maintains `retrieve_btc_account_to_block_indices`, a map from `Account` to a `Vec<u64>` of block indices, used to answer `retrieve_btc_status_v2_by_account` queries. Block indices are appended on every accepted `retrieve_btc` request but are **never removed** when requests are finalized, cancelled, or reimbursed. This is the direct IC analog of the Knox `epochsByBuyer[]` bug: a tracking index that is only ever grown, never pruned, causing the query function to return stale/misleading records indefinitely.

### Finding Description

**Addition path** — `accept_retrieve_btc_request` in `rs/bitcoin/ckbtc/minter/src/state/audit.rs` lines 27–33:

```rust
if let Some(account) = request.reimbursement_account {
    state
        .retrieve_btc_account_to_block_indices
        .entry(account)
        .and_modify(|entry| entry.push(request.block_index))
        .or_insert(vec![request.block_index]);
}
```

Every accepted request whose `reimbursement_account` is `Some` appends its `block_index` to the per-account vector. [1](#0-0) 

**Removal paths — all missing the cleanup:**

1. `remove_retrieve_btc_request` (audit.rs lines 67–83) finalizes a request with `AmountTooLow` but does **not** remove the block_index from `retrieve_btc_account_to_block_indices`. [2](#0-1) 

2. `finalize_transaction` (state.rs lines 1036–1047) confirms requests on-chain but does **not** remove their block indices from the map. [3](#0-2) 

3. The event-log replay handler for `RemovedRetrieveBtcRequest` (eventlog.rs lines 405–418) calls `remove_pending_retrieve_btc_request` but never touches `retrieve_btc_account_to_block_indices`. [4](#0-3) 

4. The event-log replay handler for `SentBtcTransaction` (eventlog.rs lines 419–473) similarly removes requests from the pending queue without pruning the index. [5](#0-4) 

**Query path** — `retrieve_btc_status_v2_by_account` (state.rs lines 798–822) reads the full, ever-growing vector and maps every stored block_index to a status:

```rust
let block_indices: Vec<u64> = self
    .retrieve_btc_account_to_block_indices
    .get(&target_account)
    .unwrap_or(&vec![])
    .to_vec();
``` [6](#0-5) 

The field declaration confirms it is a plain `BTreeMap<Account, Vec<u64>>` with no eviction policy: [7](#0-6) 

### Impact Explanation

1. **Unbounded state growth**: Every `retrieve_btc` call by the same account permanently grows the per-account vector. A user who makes N withdrawals over the lifetime of the minter will have a vector of length N that is never trimmed. Because the minter is a long-lived canister on mainnet, this constitutes a monotonically growing memory leak in stable state.

2. **Misleading query results**: `retrieve_btc_status_v2_by_account` is the primary user-facing API for checking withdrawal status. It returns every historical block_index ever associated with an account. For finalized requests, `retrieve_btc_status_v2` falls through to `retrieve_btc_status`, which returns `Unknown` once the request has aged out of the bounded `finalized_requests` VecDeque. A user querying their account will see old, completed withdrawals reported as `Unknown`, which is incorrect and misleading — directly analogous to Knox's `getEpochsByBuyer()` returning inaccurate results.

3. **Reimbursement-account confusion**: The index is keyed on `reimbursement_account`, not the caller. A user who uses different subaccounts for different withdrawals will have separate vectors, but a user who always uses the same account accumulates all historical indices in one vector, making the response payload grow without bound.

### Likelihood Explanation

This is triggered by any unprivileged user calling `retrieve_btc` or `retrieve_btc_with_approval` repeatedly over time. Both entry points set `reimbursement_account` to `Some(...)` unconditionally, so every successful withdrawal appends to the index. The ckBTC minter is a production mainnet canister with real user traffic; the condition is already occurring on every withdrawal. [8](#0-7) [9](#0-8) 

### Recommendation

In `accept_retrieve_btc_request`, `remove_retrieve_btc_request`, and `finalize_transaction`, maintain the index symmetrically: remove the block_index from `retrieve_btc_account_to_block_indices` when a request transitions to a terminal state (`AmountTooLow`, `Confirmed`, or reimbursed). Specifically:

- In `remove_retrieve_btc_request` (audit.rs), after `push_finalized_request`, remove `request.block_index` from the account's vector and drop the account key if the vector becomes empty.
- In `finalize_transaction` (state.rs), for each confirmed request, similarly prune the index.
- Mirror this cleanup in the event-log replay handlers so state rebuilt from the log is consistent.

Alternatively, cap the per-account vector to a fixed maximum (e.g., the last N entries) and document that only recent withdrawals are tracked.

### Proof of Concept

1. User calls `retrieve_btc_with_approval` with `from_subaccount = Some([1;32])` → request accepted, `block_index = 100` appended to `retrieve_btc_account_to_block_indices[{owner, subaccount=[1;32]}]`.
2. Minter processes the request; `finalize_transaction` is called → request moves to `finalized_requests`, but `retrieve_btc_account_to_block_indices` still contains `[100]`.
3. User calls `retrieve_btc_with_approval` again → `block_index = 200` appended → vector is now `[100, 200]`.
4. After enough time, `block_index = 100` ages out of the bounded `finalized_requests` VecDeque.
5. User calls `retrieve_btc_status_v2_by_account(Some({owner, subaccount=[1;32]}))` → response contains both `block_index=100` with status `Unknown` (stale, misleading) and `block_index=200` with the correct current status.
6. Repeated over many withdrawals, the vector grows without bound, consuming increasing heap memory and producing increasingly large (and misleading) query responses.
