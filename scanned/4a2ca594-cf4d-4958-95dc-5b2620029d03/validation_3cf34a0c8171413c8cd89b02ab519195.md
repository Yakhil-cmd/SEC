### Title
Logo-omission guard bypassed in `convert_proposal` when `return_self_describing_action=true` — (`rs/nns/governance/src/pb/proposal_conversions.rs`)

### Summary
When `list_proposals` is called with `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)`, the `show_action` flag is set to `false`, causing `action` to be `None`. The `is_create_service_nervous_system_proposal` flag is then derived from that `None` value and is always `false`, so the logo-omission guard passed to `convert_self_describing_action` is always `false`. The full logo bytes are returned in `self_describing_action.value` regardless of `omit_large_fields`.

### Finding Description
`ProposalDisplayOptions::for_list_proposals` sets `show_action = !return_self_describing_action`. When `return_self_describing_action=true`, `show_action=false`, so `convert_proposal` sets `action = None`. The subsequent check:

```rust
let is_create_service_nervous_system_proposal = action.as_ref().is_some_and(|action| {
    matches!(action, api::proposal::Action::CreateServiceNervousSystem(_))
});
```

always evaluates to `false` because `action` is `None`. The omit flag passed to `convert_self_describing_action` is therefore `false && display_options.omit_create_service_nervous_system_large_fields()` = `false`, and the logo paths (`"logo"`, `"ledger_parameters"/"token_logo"`) are never added to `paths_to_omit`. [6](#0-5) 

The `self_describing_action` is stored at proposal-submission time with the full logo bytes embedded in the `SelfDescribingValue` blob tree. [7](#0-6) 

### Impact Explanation
The IC query response size limit is 2 MB. A `CreateServiceNervousSystem` proposal can carry a logo up to ~256 KiB. A single `list_proposals` call returns up to 100 proposals. If several such proposals exist, the unstripped `self_describing_action` blobs can push the serialized response over the 2 MB limit, causing the query execution to trap and return an error to the caller. The `omit_large_fields` flag — whose documented purpose is precisely to prevent this — is silently ineffective in this code path. [8](#0-7) 

### Likelihood Explanation
The trigger is a standard, unauthenticated query call (`list_proposals`) with two boolean flags set. No privileged access, no key material, and no governance majority is required. Any caller can reproduce this deterministically. The precondition — at least one `CreateServiceNervousSystem` proposal with a non-trivial logo — is routinely satisfied on mainnet (every SNS launch produces one).

### Recommendation
Derive `is_create_service_nervous_system_proposal` from the raw `pb::Proposal.action` field directly, not from the already-converted (and potentially `None`) `action` output:

```rust
let is_create_service_nervous_system_proposal = matches!(
    item.action.as_ref(),
    Some(pb::proposal::Action::CreateServiceNervousSystem(_))
);
```

This makes the type-detection independent of `show_action`.

### Proof of Concept
State-machine test outline:
1. Submit a `CreateServiceNervousSystem` proposal with a 256 KiB base64-encoded logo.
2. Call `list_proposals` with `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)`.
3. Assert that the returned `self_describing_action.value` map entry for `"logo"` is `SelfDescribingValue::Null` (currently it is the full blob, demonstrating the bypass).
