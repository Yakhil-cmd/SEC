Looking at the actual code to verify each claim in the exploit chain.

### Title
Predictable ChaCha20Rng Seed Enables Confidential Rule ID Enumeration via Unauthenticated `get_rule_by_id` Query — (`rs/boundary_node/rate_limits/canister/random.rs`, `canister.rs`, `getter.rs`)

---

### Summary

Three independently verifiable weaknesses compose into a concrete exploit: (1) the canister's ChaCha20Rng is seeded with only 8 bytes of observable IC time and 24 fixed bytes, making the UUID sequence predictable; (2) `get_rule_by_id` is a `#[query]` method that any anonymous caller can invoke without restriction, because `inspect_message` only applies to update/ingress calls; (3) `RuleGetter` returns a distinguishable `Ok(redacted)` for existing confidential rules versus `Err(NotFound)` for absent ones, leaking rule existence. Together these allow an unprivileged attacker to enumerate the IDs of confidential, non-disclosed rate-limit rules.

---

### Finding Description

**Step 1 — Weak, predictable RNG seed.**

`random.rs` initializes the thread-local `ChaCha20Rng` as follows:

```rust
let mut seed = [42; 32];
seed[..8].copy_from_slice(&time().to_le_bytes());
RefCell::new(ChaCha20Rng::from_seed(seed))
``` [1](#0-0) 

The seed is 32 bytes: the first 8 bytes are the IC time in nanoseconds at the moment of first RNG access; the remaining 24 bytes are the constant `0x2A` (42). IC time is part of the certified state and is publicly observable. The `thread_local!` initializer runs lazily on the first call to `generate_random_uuid`, which occurs during the first `add_config` invocation that introduces a new rule. The timestamp of that call is observable from the canister's version history and the IC block timestamps.

**Step 2 — `get_rule_by_id` is freely accessible as a query.**

`get_rule_by_id` is declared `#[query]`:

```rust
#[query]
fn get_rule_by_id(rule_id: RuleId) -> GetRuleByIdResponse { ... }
``` [2](#0-1) 

The `inspect_message` hook only intercepts update (ingress) calls. It covers `add_config`, `disclose_rules`, and `get_config`-as-update. Query calls bypass `inspect_message` entirely at the IC protocol level — they are served by a single replica without consensus and without the hook firing. `get_rule_by_id` and `get_rules_by_incident_id` are absent from the `UPDATE_METHODS` and `REPLICATED_QUERY_METHOD` constants and are never mentioned in the hook: [3](#0-2) 

Any anonymous principal can call `get_rule_by_id` as a query with no restriction.

**Step 3 — Response leaks rule existence.**

Inside `RuleGetter::get`, the lookup happens before the access-level check:

```rust
let stored_rule = self
    .canister_api
    .get_rule(&rule_id)
    .ok_or_else(|| GetEntityError::NotFound(rule_id.0.to_string()))?;
``` [4](#0-3) 

If the rule does not exist → `Err(NotFound)`. If it exists but the caller is unauthorized → `Ok(redacted_rule)` where `rule_raw` and `description` are `None` but the response variant is `Ok`. The `RuleConfidentialityFormatter` only nulls out content fields; it never converts the response to an error: [5](#0-4) 

The two response variants are structurally distinct. An attacker can trivially distinguish "rule exists (confidential)" from "rule does not exist."

**Exploit chain:**

1. Observe the IC block timestamp of the first `add_config` call (public, certified state).
2. Reconstruct the seed: `seed[0..8] = observed_time.to_le_bytes(); seed[8..32] = [42; 24]`.
3. Instantiate `ChaCha20Rng::from_seed(seed)` locally and generate the UUID sequence.
4. For each predicted UUID, call `get_rule_by_id(uuid)` as an anonymous query.
5. `Ok(...)` → rule exists and is confidential. `Err(NotFound)` → UUID not assigned.

The IC time is in nanoseconds. Within a single block the time is fixed and deterministic. If the exact nanosecond is not immediately known, the search space is bounded by the block duration (~1 second = 10⁹ values), which is feasible to brute-force offline before submitting queries.

---

### Impact Explanation

An unprivileged attacker can confirm the existence of confidential, non-disclosed rate-limit rules — rules that protect against active security incidents. Knowing that a rule with a specific UUID exists (even without seeing its content) reveals that an incident is being actively mitigated. This leaks the fact of an ongoing security response before public disclosure, allowing adversaries to probe or exploit the protected surface before patches are applied.

---

### Likelihood Explanation

The attack requires no privileged access, no key material, and no social engineering. It requires only: (a) observing a public IC block timestamp, and (b) issuing anonymous query calls. The brute-force over nanosecond timestamps is entirely offline and parallelizable. The query endpoint is publicly reachable on mainnet. Likelihood is **high** given the low barrier and the public accessibility of all required inputs.

---

### Recommendation

1. **Fix the RNG seed**: Replace the time-based seed with a cryptographically unpredictable source. On IC, use `ic_cdk::api::management_canister::main::raw_rand()` (an async call to the management canister) to obtain 32 bytes of verifiable randomness during `init`/`post_upgrade`, store it in stable memory, and use it as the seed.

2. **Fix the existence oracle**: `RuleGetter::get` should return `Err(NotFound)` for confidential rules when the caller is unauthorized, rather than `Ok(redacted)`. This prevents distinguishing "exists but hidden" from "does not exist."

3. **Alternatively, gate `get_rule_by_id` on access level**: Return `Err(Unauthorized)` or `Err(NotFound)` for any caller without `FullAccess` or `FullRead` when the rule is not disclosed.

### Proof of Concept

```rust
use rand_chacha::ChaCha20Rng;
use rand_chacha::rand_core::{RngCore, SeedableRng};
use uuid::Uuid;

fn predict_uuids(canister_init_time_ns: u64, count: usize) -> Vec<Uuid> {
    let mut seed = [42u8; 32];
    seed[..8].copy_from_slice(&canister_init_time_ns.to_le_bytes());
    let mut rng = ChaCha20Rng::from_seed(seed);
    (0..count).map(|_| {
        let mut buf = [0u8; 16];
        rng.fill_bytes(&mut buf);
        Uuid::from_slice(&buf).unwrap()
    }).collect()
}

// Then for each uuid in predict_uuids(observed_time, 100):
//   call get_rule_by_id(uuid.to_string()) as an anonymous query
//   Ok(_)  => confidential rule confirmed to exist
//   Err(_) => not assigned
```

The `observed_time` is read from the IC certified state (block timestamp of the first `add_config` transaction). If the exact nanosecond is uncertain, iterate over the ~10⁹ values in the block window offline before issuing queries.
