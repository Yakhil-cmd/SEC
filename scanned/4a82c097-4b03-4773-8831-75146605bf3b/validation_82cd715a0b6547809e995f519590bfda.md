### Title
`get_pending_proposals` Iterates All Open Proposals Without a Result Limit, Enabling Resource-Exhaustion DoS - (File: rs/nns/governance/src/governance.rs)

### Summary

The NNS Governance canister exposes a `get_pending_proposals` query that iterates the entire proposal map and returns **all** open proposals with no pagination cap. Unlike `list_proposals`, which enforces `MAX_LIST_PROPOSAL_RESULTS = 100`, `get_pending_proposals` collects every open proposal into a single `Vec<ProposalInfo>` and returns it. Because manage-neuron proposals have a separate ceiling of `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS = 10,000`, an attacker who controls a neuron can accumulate thousands of open proposals and cause every call to `get_pending_proposals` to either exhaust the query instruction budget or produce a response that exceeds the IC's per-message size limit, making the endpoint permanently unavailable.

### Impact Explanation

When the open-proposal count is driven to the 10,000-proposal ceiling, every invocation of `get_pending_proposals` must:

1. Scan the entire `heap_data.proposals` BTreeMap (O(n) over all proposals ever created).
2. Call `proposal_data_to_info` for each open proposal, including ballot serialization.
3. Collect and return an unbounded `Vec<ProposalInfo>`.

This either exhausts the query instruction budget (causing a trap) or produces a response that exceeds the IC per-message size limit. Either outcome makes `get_pending_proposals` permanently unavailable for as long as the attacker's proposals remain open, breaking the Rosetta API `/call` endpoint and any other consumer of this query.

### Likelihood Explanation

Creating a neuron requires staking at least 1 ICP. Submitting 10,000 manage-neuron proposals costs the rejection fee per proposal only if each proposal is rejected; proposals that are simply ignored (never voted on) do not consume stake. A motivated attacker therefore needs only a single neuron and the patience to submit 10,000 transactions. The attack is persistent: the proposals remain open indefinitely unless the NNS votes to reject them, which itself requires governance participation.

### Recommendation

Apply the same result cap used by `list_proposals` to `get_pending_proposals`, or convert it to a paginated API. At minimum, add a hard ceiling (e.g., `MAX_LIST_PROPOSAL_RESULTS = 100`) inside `get_pending_proposals` so that the response size is bounded regardless of how many proposals are open:

```rust
self.heap_data
    .proposals
    .values()
    .filter(|data| data.status() == ProposalStatus::Open)
    .take(MAX_LIST_PROPOSAL_RESULTS as usize)   // add this
    .map(|data| proposal_data_to_info(...))
    .collect()
```

Callers that need the full list should use the paginated `list_proposals` API with `include_status = [Open]`.

### Proof of Concept

1. Stake 1 ICP to create a neuron `N` with sufficient dissolve delay.
2. In a loop, submit 10,000 `ManageNeuron` proposals targeting `N` itself (e.g., `IncreaseDissolveDelay` by 1 second). Each proposal is accepted by the governance canister because the manage-neuron open-proposal count has not yet reached `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS`.
3. Once 10,000 proposals are open, call `get_pending_proposals` from any principal (including anonymous).
4. Observe that the query either traps (instruction limit exceeded) or returns a response that the IC rejects for exceeding the per-message size limit.
5. The Rosetta API `/call?method=get_pending_proposals` now returns an error for every request, confirming the DoS.
