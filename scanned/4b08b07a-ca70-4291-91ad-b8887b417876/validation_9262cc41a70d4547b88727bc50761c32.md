### Title
Unbounded Growth of `retrieve_btc_account_to_block_indices` Never Pruned After Withdrawal Finalization — (File: `rs/bitcoin/ckbtc/minter/src/state.rs`)

---

### Summary

The ckBTC minter maintains a per-account map `retrieve_btc_account_to_block_indices: BTreeMap<Account, Vec<u64>>` that records every withdrawal (retrieve_btc) block index ever associated with an account. Entries are appended on every accepted withdrawal request and on every deposit reimbursement, but the map is **never pruned**. This is the direct IC analog of the reported "deposit cap not updated on withdrawals" pattern: a monotonically growing per-account data structure with no corresponding decrement on the reverse operation. Over time, an unprivileged user can cause unbounded heap memory growth in the ckBTC minter canister, and the `retrieve_btc_status_v2_by_account` query becomes increasingly expensive per account.

---

Meanwhile, `finalized_requests` is a bounded `VecDeque` that drops old entries, meaning `retrieve_btc_account_to_block_indices` will accumulate block indices for requests whose status is no longer available, returning `Unknown` for them — but the indices themselves are never removed. [7](#0-6) 

---

### Impact Explanation

The ckBTC minter is a critical chain-fusion canister. Its heap memory is finite (currently capped at 4 GiB for IC canisters). Because `retrieve_btc_account_to_block_indices` grows without bound:

1. **Memory exhaustion**: A user who repeatedly makes withdrawal requests (each finalized before the next) permanently grows their entry in the map. Across many users over the lifetime of the minter, this map can consume a significant and unbounded portion of heap memory, eventually causing the minter to trap on memory allocation and halt all ckBTC operations.

2. **Query performance degradation**: `retrieve_btc_status_v2_by_account` iterates over all stored block indices for an account. An account with thousands of historical withdrawals causes this query to perform thousands of status lookups, potentially exhausting the query execution budget.

3. **Stale data accumulation**: Because `finalized_requests` is bounded and drops old entries, the map accumulates block indices that resolve to `Unknown`, providing no useful information while still consuming memory.

---

### Likelihood Explanation

**Medium-low.** Each withdrawal requires burning real ckBTC (backed by real BTC), making deliberate exploitation costly. However, the growth is also triggered by legitimate heavy users of the ckBTC bridge over its operational lifetime. The minter has been live on mainnet and accumulates state continuously. No operator action can prune this map without a canister upgrade that explicitly migrates state.

The attacker-controlled entry path is: any unprivileged principal calls `retrieve_btc` or `retrieve_btc_with_approval` (publicly accessible update endpoints), waits for finalization, and repeats. No privileged role is required. [8](#0-7) 

---

### Recommendation

1. **Prune stale entries**: After a withdrawal request reaches a terminal state (`Confirmed`, `Reimbursed`, `AmountTooLow`), remove its block index from `retrieve_btc_account_to_block_indices`, or cap the Vec per account to a fixed maximum (e.g., the last 100 entries).

2. **Bound the map**: Introduce a maximum number of stored block indices per account, evicting the oldest entries when the limit is reached.

3. **Separate hot and cold state**: Move historical (finalized) block indices to stable memory with a bounded in-memory cache, analogous to how `finalized_requests` is already bounded.

---

### Proof of Concept

```
1. User A calls retrieve_btc_with_approval(amount=min_retrieve_amount, address=...).
   → retrieve_btc_account_to_block_indices[A] = [block_index_1]

2. Minter finalizes the BTC transaction (ConfirmedBtcTransaction event).
   → finalized_requests may eventually drop block_index_1 (bounded VecDeque).
   → retrieve_btc_account_to_block_indices[A] = [block_index_1]  ← NOT pruned.

3. User A repeats N times.
   → retrieve_btc_account_to_block_indices[A] = [block_index_1, ..., block_index_N]
   → Each entry: 8 bytes (u64) + BTreeMap overhead.
   → For N = 1,000,000 withdrawals across all users: ~8 MB+ of map data, growing forever.

4. retrieve_btc_status_v2_by_account(Some(A)) now iterates N entries,
   most returning Unknown (finalized_requests dropped them), consuming O(N) query cycles.
```

pub fn accept_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    runtime: &R,
) {
    record_event(
        EventType::AcceptedRetrieveBtcRequest(request.clone()),
        runtime,
    );
    state.pending_retrieve_btc_requests.push(request.clone());
    if let Some(account) = request.reimbursement_account {
        state
            .retrieve_btc_account_to_block_indices
            .entry(account)
            .and_modify(|entry| entry.push(request.block_index))
            .or_insert(vec![request.block_index]);
    }
    if let Some(kyt_provider) = request.kyt_provider {
        *state.owed_kyt_amount.entry(kyt_provider).or_insert(0) += state.check_fee;
    }
}
