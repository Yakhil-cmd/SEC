### Title
Attacker-Controlled Gzip Footer Inflates Compilation Cost Charged in Error Path, Draining Round Instruction Limit - (File: rs/embedders/src/wasm_utils/decoding.rs, rs/execution_environment/src/hypervisor.rs)

---

### Summary

The `wasm_encoding_and_size()` function reads the uncompressed Wasm size from the gzip footer — a 4-byte field fully controlled by the submitter. This value is used in `Hypervisor::create_execution_state()` to compute a compilation cost estimate that is charged in the **error path** when the actual decompression/size check fails. An unprivileged canister developer can craft a gzip Wasm with a maximally inflated footer (`u32::MAX` ≈ 4.3 GB), causing an enormous instruction cost to be deducted from the shared `round_limits.instructions` budget, starving other canisters in the same execution round.

---

### Finding Description

**Root cause — `wasm_encoding_and_size()` in `rs/embedders/src/wasm_utils/decoding.rs`:**

```rust
isize_bytes.copy_from_slice(&module_bytes[module_bytes.len() - 4..module_bytes.len()]);
let uncompressed_size = u32::from_le_bytes(isize_bytes) as usize;
return Ok((WasmEncoding::Gzip, uncompressed_size));
```

The function itself carries an explicit warning: *"If the Wasm is gzipped, then the returned size cannot be trusted. It would come from the gzip footer which could have been manipulated."* [1](#0-0) 

**Propagation — `Hypervisor::create_execution_state()` in `rs/execution_environment/src/hypervisor.rs`:**

```rust
let wasm_size_result = decoded_wasm_size(canister_module.as_slice());
let wasm_size = match wasm_size_result {
    Ok(size) => std::cmp::max(size, canister_module.len()),
    Err(_) => canister_module.len(),
};
let compilation_cost = self.cost_to_compile_wasm_instruction * wasm_size as u64;
``` [2](#0-1) 

The attacker-controlled `wasm_size` (up to `u32::MAX` ≈ 4.3 billion) is multiplied by `cost_to_compile_wasm_instruction` to produce `compilation_cost`. This estimate is then charged in the **error branch**:

```rust
Err(err) => {
    let total_cost = self.create_execution_state_base_cost + compilation_cost;
    round_limits.instructions -= as_round_instructions(total_cost);
    (total_cost, Err(err))
}
``` [3](#0-2) 

**Why the error path is reachable:** `decode_wasm()` — called inside `wasm_executor.create_execution_state()` — performs the legitimate size check against `max_size`:

```rust
if uncompressed_size as u64 > max_size.get() {
    return Err(WasmValidationError::ModuleTooLarge { ... });
}
``` [4](#0-3) 

If the attacker sets the footer to any value exceeding `wasm_max_size` (the configured maximum, e.g. ~100 MB), `decode_wasm` returns `ModuleTooLarge`, triggering the error branch in `create_execution_state` — which charges the inflated `compilation_cost` computed from the attacker-controlled footer value.

**Crucially, on the success path the actual compilation cost from the compiler is used** (not the estimate), so the manipulation only affects the error path:

```rust
Ok((execution_state, compilation_cost, compilation_result)) => {
    let total_cost = self.create_execution_state_base_cost
        + compilation_cost_handling.adjusted_compilation_cost(compilation_cost);
    ...
}
``` [5](#0-4) 

---

### Impact Explanation

`round_limits.instructions` is a **shared per-round budget** across all canisters scheduled in that execution round. By draining it with an artificially inflated compilation cost, the attacker prevents other canisters from executing in the same round. With `cost_to_compile_wasm_instruction = 1` and footer = `u32::MAX`, the deduction is ~4.3 billion instructions — a significant fraction of or exceeding the round budget. With higher `cost_to_compile_wasm_instruction` values (the production default is non-trivial), the deduction saturates at `u64::MAX`. This constitutes a **cycles/resource accounting bug** and a **round-level DoS** against other canisters.

---

### Likelihood Explanation

Any canister developer can submit a gzip-compressed Wasm module via `install_code` or `install_chunked_code`. Crafting a gzip file with an arbitrary 4-byte footer is trivial (standard gzip format, RFC 1952). No privileged access, governance majority, or threshold corruption is required. The attack is repeatable across rounds. [6](#0-5) 

---

### Recommendation

1. **Do not use the gzip footer size for cost estimation.** In `create_execution_state`, use `canister_module.len()` (the actual compressed size) as the fallback estimate when the module is gzip-encoded, rather than the untrusted footer value. The comment already acknowledges the footer is untrustworthy.

2. **Cap the estimated size** used for cost calculation at `wasm_max_size` before multiplying by `cost_to_compile_wasm_instruction`, so even a manipulated footer cannot produce a cost exceeding what a legitimately maximum-sized module would cost.

3. Alternatively, perform the `decode_wasm` size check **before** computing the compilation cost estimate, so the error path never charges based on the attacker-controlled value.

---

### Proof of Concept

1. Attacker creates a minimal valid gzip stream (valid header + deflate body + crafted footer).
2. Footer bytes (last 4 bytes, little-endian) are set to `0xFF 0xFF 0xFF 0xFF` (`u32::MAX` = 4,294,967,295).
3. Attacker submits this as the `wasm_module` field of an `install_code` call to any canister they control.
4. `decoded_wasm_size` returns 4,294,967,295.
5. `wasm_size = max(4,294,967,295, actual_compressed_len)` = 4,294,967,295.
6. `compilation_cost = cost_to_compile_wasm_instruction * 4,294,967,295` (enormous).
7. `wasm_executor.create_execution_state` calls `decode_wasm`, which returns `ModuleTooLarge` (since 4 GB > `wasm_max_size`).
8. Error branch executes: `round_limits.instructions -= as_round_instructions(base_cost + compilation_cost)`.
9. `round_limits.instructions` is drained by billions of instructions, starving other canisters scheduled in the same round. [7](#0-6) [8](#0-7)
