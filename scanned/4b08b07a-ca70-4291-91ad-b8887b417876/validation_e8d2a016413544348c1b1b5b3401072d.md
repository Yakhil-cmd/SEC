### Title
ICP Sent to SNS Swap Canister's Default Subaccount Is Permanently Unrecoverable - (File: rs/sns/swap/src/swap.rs)

### Summary

The SNS Swap canister's `error_refund_icp` rescue function exclusively checks the **principal-derived subaccount** (`principal_to_subaccount(source_principal_id)`) when attempting to refund ICP. If a user mistakenly sends ICP to the swap canister's **default subaccount** (no subaccount / all-zero subaccount) instead of their correct buyer-specific subaccount, those tokens are permanently locked with no on-chain recovery path.

### Finding Description

The SNS swap participation flow requires two steps:

1. Transfer ICP to the buyer's **principal-derived subaccount** of the swap canister: `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&buyer)) }`
2. Call `refresh_buyer_tokens` to register the participation.

The `error_refund_icp` function is the designated rescue mechanism for ICP sent in error. However, it unconditionally constructs the source subaccount as:

```rust
let source_subaccount = principal_to_subaccount(source_principal_id);
let account_balance_result = icp_ledger
    .account_balance(Account {
        owner: self_canister_id.into(),
        subaccount: Some(source_subaccount),  // always principal-derived, never default
    })
    .await;
``` [1](#0-0) 

There is no code path in `error_refund_icp`, `sweep_icp`, or any other swap canister function that reads from or transfers out of the swap canister's **default subaccount** (subaccount = `None`). The `sweep_icp` function only iterates over `self.buyers`, which are populated exclusively by `refresh_buyer_tokens` calls that check principal-derived subaccounts. [2](#0-1) 

A user who sends ICP directly to the swap canister's principal ID (default subaccount) — a natural mistake for someone unfamiliar with the subaccount-based participation model — has no on-chain mechanism to recover those funds. The `error_refund_icp` endpoint requires a `source_principal_id`, but will always look at `principal_to_subaccount(source_principal_id)`, never at the default subaccount where the ICP actually landed. [3](#0-2) [4](#0-3) 

The `ErrorRefundIcpRequest` proto itself documents the intent as refunding from "this canister's subaccount of the caller," confirming the design only covers principal-derived subaccounts. [5](#0-4) 

### Impact Explanation

ICP tokens sent to the swap canister's default subaccount are permanently locked. There is no admin rescue function, no sweep mechanism, and no `error_refund_icp` path that covers the default subaccount. The funds are conserved on-chain but inaccessible to both the user and the swap canister's normal finalization logic. This is a **ledger conservation bug**: tokens exist on the ledger but are effectively burned from the user's perspective with no recourse.

**Impact: High** — permanent, irreversible loss of user ICP funds.

### Likelihood Explanation

The two-step participation flow (transfer to a subaccount, then call `refresh_buyer_tokens`) is non-obvious. A user who reads documentation saying "send ICP to the swap canister" may send to the canister's principal ID without a subaccount. The NNS Governance README itself explicitly warns that the destination subaccount is NOT the governance's main/default subaccount, indicating this is a known user confusion vector. [6](#0-5) 

The SNS swap participation helper in the agent library shows the correct two-step flow, but nothing prevents a user from skipping the subaccount derivation. [7](#0-6) 

**Likelihood: Medium** — realistic user error with no on-chain safeguard.

### Recommendation

Extend `error_refund_icp` to also check and refund from the swap canister's **default subaccount** when `source_principal_id` is absent or when explicitly requested. Alternatively, add a separate admin-callable function (callable only after `Committed` or `Aborted`) that can sweep arbitrary subaccounts — including the default subaccount — back to a designated recipient, analogous to the fix applied in the NFTfi report (allowing rescue of tokens not associated with any registered record).

### Proof of Concept

1. SNS swap is in `Open` lifecycle.
2. User `P` sends 10 ICP to `Account { owner: swap_canister_id, subaccount: None }` (the default subaccount) via `icrc1_transfer` on the ICP ledger.
3. User `P` calls `refresh_buyer_tokens` — this checks `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&P)) }`, which has zero balance. The call either fails or registers 0 ICP.
4. Swap reaches `Committed` or `Aborted`.
5. User `P` calls `error_refund_icp { source_principal_id: Some(P) }`.
6. The function checks `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&P)) }` — balance is 0. Transfer fails with an `External` error.
7. The 10 ICP in `Account { owner: swap_canister_id, subaccount: None }` is never touched by any swap canister function and remains permanently locked. [8](#0-7) [9](#0-8)
