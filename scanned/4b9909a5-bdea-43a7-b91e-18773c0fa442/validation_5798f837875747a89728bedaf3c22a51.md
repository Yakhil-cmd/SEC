### Title
Unbounded `disburse_maturity_in_progress` Array Growth Causes Instruction-Limit DoS on SNS Neuron - (File: rs/sns/governance/src/governance.rs)

### Summary
The SNS governance `disburse_maturity` function appends to a neuron's `disburse_maturity_in_progress` vector without any cap on its length. A neuron controller with sufficient maturity can call `DisburseMaturity` repeatedly with small percentages, growing this array unboundedly. This makes the neuron's serialized state so large that subsequent operations on it — including the periodic finalization task — trap on the IC instruction limit, permanently bricking the neuron. The NNS governance already has an identical fix (`MAX_NUM_DISBURSEMENTS = 10`) that the SNS governance is missing.

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `disburse_maturity` function unconditionally pushes a new `DisburseMaturityInProgress` entry onto the neuron's vector:

```rust
neuron
    .disburse_maturity_in_progress
    .push(disbursement_in_progress);
``` [1](#0-0) 

There is no check on the current length of `disburse_maturity_in_progress` before appending. The only guards present are:
- `percentage_to_disburse` must be 1–100
- The worst-case modulated amount must exceed `transaction_fee_e8s`
- The caller must hold `DisburseMaturity` permission [2](#0-1) 

Each call deducts from `maturity_e8s_equivalent`, so the maximum number of entries is bounded by `maturity / min_disbursement_e8s`. With a default SNS transaction fee of 10,000 e8s and worst-case modulation of −5%, the minimum disbursement is ~10,527 e8s. A neuron holding 1 ICP of maturity can create ~9,500 entries; a neuron with 100 ICP of maturity can create ~950,000 entries.

By contrast, the NNS governance already has this exact protection:

```rust
const MAX_NUM_DISBURSEMENTS: usize = 10;
// ...
if num_disbursements >= MAX_NUM_DISBURSEMENTS {
    return Err(InitiateMaturityDisbursementError::TooManyDisbursements);
}
``` [3](#0-2) [4](#0-3) 

The SNS neuron's `disburse_maturity_in_progress` field is a `repeated` protobuf field stored inline in the neuron: [5](#0-4) 

The SNS governance stores neurons in a heap `HashMap<String, Neuron>` (`proto.neurons`). Every access to the neuron requires full deserialization of the entire struct, including the unbounded vector. The periodic finalization task processes one disbursement per heartbeat invocation: [6](#0-5) 

When the vector is very large, deserializing the neuron, iterating over the vector to find the oldest entry, and re-serializing the neuron all consume instructions proportional to the vector length, eventually trapping on the 40B/50B instruction limit.

### Impact Explanation

A neuron controller who has accumulated sufficient maturity can:
1. Call `manage_neuron { DisburseMaturity { percentage_to_disburse: 1 } }` repeatedly until maturity is exhausted.
2. Grow `disburse_maturity_in_progress` to thousands of entries.
3. Cause every subsequent `manage_neuron` call on that neuron, every periodic finalization heartbeat touching that neuron, and every `get_neuron` query to trap on the instruction limit.
4. The maturity already deducted from `maturity_e8s_equivalent` is stuck in the queue and cannot be recovered — the neuron is permanently bricked.

Additionally, if the SNS governance canister's periodic task iterates over all neurons to find ones ready for finalization, a single bloated neuron can cause the entire periodic task to trap, delaying finalization for all other neurons in the SNS.

### Likelihood Explanation

Any SNS neuron controller with sufficient accumulated maturity can trigger this. SNS neurons accumulate maturity through voting rewards over time. Active SNS participants with long-locked neurons will naturally accumulate large maturity balances. The attack requires no special privilege beyond owning a neuron with `DisburseMaturity` permission, which is the default for the neuron controller. The attacker harms primarily themselves (their own neuron is bricked), but the periodic-task disruption affects all SNS participants.

### Recommendation

Add a cap on `disburse_maturity_in_progress.len()` in `rs/sns/governance/src/governance.rs` `disburse_maturity()`, mirroring the NNS governance fix:

```rust
const MAX_DISBURSE_MATURITY_IN_PROGRESS: usize = 10;

if neuron.disburse_maturity_in_progress.len() >= MAX_DISBURSE_MATURITY_IN_PROGRESS {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!(
            "Too many disbursements in progress (max {MAX_DISBURSE_MATURITY_IN_PROGRESS}). \
             Wait for existing disbursements to finalize before initiating new ones."
        ),
    ));
}
```

This should be checked before the push at line 1697, after the authorization and amount checks.

### Proof of Concept

1. Deploy an SNS with a neuron holding large maturity (e.g., 100 ICP equivalent).
2. As the neuron controller, call `manage_neuron` with `DisburseMaturity { percentage_to_disburse: 1 }` in a loop. Each call succeeds, deducting ~1% of remaining maturity and appending one entry to `disburse_maturity_in_progress`.
3. After ~100+ calls (or more with smaller percentages if the SNS transaction fee is low), attempt `get_neuron` or another `manage_neuron` call on the same neuron.
4. Observe that the call traps due to instruction limit exhaustion during neuron deserialization/processing.
5. Observe that the periodic finalization task also fails to process this neuron, and the maturity stuck in the queue is unrecoverable.
