### Title
Missing Fee Amount in ckBTC/ckDOGE Minter ICRC-21 Consent Messages for `retrieve_btc_with_approval` / `retrieve_doge_with_approval` - (File: `rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs`, `rs/dogecoin/ckdoge/minter/src/updates/icrc21.rs`)

---

### Summary

The ICRC-21 consent messages generated by the ckBTC and ckDOGE minters for `retrieve_btc_with_approval` and `retrieve_doge_with_approval` do not display the actual fee amounts that will be deducted from the user's withdrawal. In the `FieldsDisplay` variant (used by hardware wallets such as Ledger), no fee field is present at all. In the `GenericDisplay` variant, fees are only mentioned as a vague parenthetical with no concrete amount. A user approving the transaction via a hardware wallet sees only the ckBTC/ckDOGE amount being burned, with no indication of the actual BTC/DOGE they will receive after fees are deducted.

---

### Finding Description

In `build_retrieve_btc_with_approval_message` in `rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs`, the `FieldsDisplay` branch constructs a `FieldsDisplay` struct with only two fields: `Amount` (the ckBTC being burned) and `BTC address` (the destination). No fee field is added:

```rust
let mut fields = vec![
    ("Amount".to_string(), Value::TokenAmount { ... amount: args.amount, symbol: ckbtc }),
    (format!("{btc} address"), Value::Text { content: args.address.clone() }),
];
``` [1](#0-0) 

The `GenericDisplay` branch mentions fees only as a vague parenthetical `"(minus network and minter fees)"` without showing any concrete amount:

```
send the equivalent amount in {btc} (minus network and minter fees) to the Bitcoin address below.
``` [2](#0-1) 

The identical pattern exists in the ckDOGE minter's `build_retrieve_doge_with_approval_message`: [3](#0-2) 

This contrasts with the ICRC-1/ICRC-2 ledger consent messages, which always include an explicit fee field via `message.add_fee(...)`: [4](#0-3) [5](#0-4) 

The minter exposes `estimate_withdrawal_fee` as a query endpoint that returns both `bitcoin_fee` and `minter_fee`: [6](#0-5) 

However, this information is never surfaced in the ICRC-21 consent message.

---

### Impact Explanation

A user approving `retrieve_btc_with_approval` via a hardware wallet (which uses `FieldsDisplay`) sees only:
- `Amount: X ckBTC` (the amount being burned from their account)
- `BTC address: <destination>`

They have no visibility into the `bitcoin_fee` or `minter_fee` that will be deducted from the BTC they receive. During periods of high Bitcoin network congestion, the `bitcoin_fee` can be substantial — potentially consuming a significant fraction of the withdrawal amount. The user has approved burning `X ckBTC` but may receive far less than `X BTC` with no prior disclosure of the deduction. This is a ledger conservation / chain-fusion user transparency issue: the user cannot make an informed decision about whether to proceed with the withdrawal at the current fee level.

The `FieldsDisplay` path is specifically designed for constrained devices (hardware wallets) where the user cannot easily navigate to external fee estimation tools before approving. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The ckBTC minter is a production chain-fusion canister with significant TVL. Hardware wallet users (Ledger) are a primary target audience for ICRC-21 consent messages. The `FieldsDisplay` path is exercised in integration tests and is the path used by hardware wallets. Bitcoin network fees are dynamic and can spike significantly. Any user performing a ckBTC-to-BTC withdrawal via a hardware wallet during high-fee periods is affected. The entry path requires no special privileges — any unprivileged user calling `retrieve_btc_with_approval` is exposed. [8](#0-7) 

---

### Recommendation

1. In `build_retrieve_btc_with_approval_message`, call `estimate_withdrawal_fee` (or read the current fee estimate from state) and include a `"Estimated fees"` field in both `GenericDisplay` and `FieldsDisplay` consent messages, showing the estimated `bitcoin_fee + minter_fee` in ckBTC/satoshi units.
2. Apply the same fix to `build_retrieve_doge_with_approval_message` in the ckDOGE minter.
3. If a precise fee cannot be computed synchronously at consent-message time, include at minimum the fixed `minter_fee` and a note that the `bitcoin_fee` is variable, consistent with how the ICRC-1/ICRC-2 ledger consent messages always show the ledger fee.

---

### Proof of Concept

A hardware wallet user calls `icrc21_canister_call_consent_message` with `method = "retrieve_btc_with_approval"` and `device_spec = FieldsDisplay`. The returned `FieldsDisplay` contains exactly two fields — `Amount` and `BTC address` — with no fee field:

```
fields: [
  ("Amount", TokenAmount { amount: 250_000, symbol: "ckBTC", decimals: 8 }),
  ("BTC address", Text { content: "bc1q..." }),
]
```

This is confirmed by the integration test assertion:

```rust
assert_eq!(
    fields_display.fields,
    vec![
        ("Amount".to_string(), Value::TokenAmount { decimals: 8, amount: 250_000, symbol: "ckBTC".to_string() }),
        ("BTC address".to_string(), Value::Text { content: WITHDRAWAL_ADDRESS.to_string() }),
    ]
);
``` [9](#0-8) 

The user approves this on their hardware wallet. The minter then deducts `bitcoin_fee + minter_fee` from the BTC sent to the destination address. The user receives less BTC than the `Amount` field implied, with no prior disclosure of the fee magnitude.
