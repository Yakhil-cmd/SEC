### Title
Missing Concurrency Limit on Synchronous Call Endpoints (`call_v3_router`, `call_v4_router`, `subnet_call_v4_router`) - (File: `rs/http_endpoints/public/src/lib.rs`)

---

### Summary

The IC replica's public HTTP endpoint applies `GlobalConcurrencyLimitLayer` to every route family **except** the three synchronous call routers (`call_v3_router`, `call_v4_router`, `subnet_call_v4_router`). This is acknowledged in the code with a `TODO` comment but left unresolved. An unprivileged external sender can flood these endpoints without hitting any per-route concurrency cap, exhausting replica-side resources and degrading availability for all users of the affected replica node.

---

### Finding Description

In `rs/http_endpoints/public/src/lib.rs`, the `make_router` function assembles the final Axum router and wraps each route family with a `GlobalConcurrencyLimitLayer` + load-shedder via `service_builder(...)`. Every route receives this protection **except** the three synchronous call routers:

```rust
// rs/http_endpoints/public/src/lib.rs  lines 603–609
.merge(http_handler.call_v2_router.layer(service_builder(
    GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests),
)))
// TODO(CON-1574): see if there is any reasonable explicit concurrency limit we could use here.
.merge(http_handler.call_v3_router)
.merge(http_handler.call_v4_router)
.merge(http_handler.subnet_call_v4_router)
```

`call_v2_router` is protected with `max_call_concurrent_requests` (default 50). The v3/v4/subnet-v4 variants are merged **bare**, with no concurrency limit layer at all. The `TODO(CON-1574)` comment confirms this is a known gap, not an intentional design choice.

The only global protection that applies to these routes is the outer `load_shed()` + `timeout` applied to the entire router stack, which sheds load only when the Tokio task queue itself is saturated — a much coarser and later-stage backstop than a per-route concurrency cap.

The `call_v3_router` and `call_v4_router` are the **synchronous** call paths: they hold the connection open while waiting for the ingress message to be certified (up to `ingress_message_certificate_timeout_seconds`, default 10 s). This makes each request significantly more resource-intensive than the async v2 path, amplifying the impact of unbounded concurrency.

---

### Impact Explanation

An unprivileged attacker sending a high volume of POST requests to `/api/v3/canister/{id}/call`, `/api/v4/canister/{id}/call`, or `/api/v4/subnet/{id}/call` can:

- Hold an unbounded number of long-lived connections open simultaneously (each waiting up to 10 s for certification).
- Exhaust the replica's Tokio worker threads, file descriptors, and memory.
- Degrade or deny service to legitimate users on the same replica node for **all** endpoint types (query, read_state, call), since they share the same async runtime.

Impact: **Availability degradation / DoS of a replica node's public HTTP endpoint.**

---

### Likelihood Explanation

The endpoints are publicly reachable by any Internet user without authentication. No special role, key, or governance majority is required. The attacker only needs to send standard CBOR-encoded IC call envelopes (even with invalid content — the concurrency limit fires before deep validation). The synchronous nature of v3/v4 calls (long-held connections) makes the attack more efficient per request than flooding a stateless endpoint.

---

### Recommendation

Apply the same `service_builder(GlobalConcurrencyLimitLayer::new(...))` pattern to `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` as is already done for `call_v2_router`. A dedicated config field (e.g., `max_call_sync_concurrent_requests`) should be introduced in `rs/config/src/http_handler.rs` to allow independent tuning, since the synchronous paths hold connections much longer than the async v2 path and may warrant a lower limit.
