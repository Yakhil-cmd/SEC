### Title
ckBTC Minter Burns User ckBTC Without Reimbursement When `BuildTxError::AmountTooLow` Occurs After Fee Spike - (File: rs/bitcoin/ckbtc/minter/src/lib.rs)

### Summary

The ckBTC minter's `submit_pending_requests` function permanently finalizes withdrawal requests with `FinalizedStatus::AmountTooLow` when a Bitcoin transaction cannot be built because the withdrawal amount is too low to cover the current network fees. However, the user's ckBTC has **already been burned** from the ledger at the time of the `retrieve_btc` call. The minter discards the request without minting ckBTC back to the user, resulting in a permanent loss of user funds.

### Finding Description

When a user calls `retrieve_btc` or `retrieve_btc_with_approval`, the minter first burns the user's ckBTC from the ledger, then enqueues a `RetrieveBtcRequest`. Later, during `submit_pending_requests`, if the Bitcoin network fee has spiked such that the withdrawal amount no longer covers the fee, `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`. The minter handles this by calling `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` — permanently discarding the request with no reimbursement to the user.

The analogous `BuildTxError::InvalidTransaction` path correctly calls `reimburse_canceled_requests`, which schedules a mint-back to the user. The `AmountTooLow` path does not.

### Impact Explanation

A user who submits a valid `retrieve_btc` request (passing the minimum amount check at submission time) can have their ckBTC permanently burned with no BTC sent and no ckBTC refunded. This occurs when Bitcoin network fees spike between the time the request is accepted and the time the minter attempts to build the transaction. The `FinalizedStatus::AmountTooLow` state is terminal — there is no recovery path, no reimbursement account is consulted, and the user's funds are destroyed. This is a **ledger conservation bug**: ckBTC supply decreases without a corresponding BTC output.

The `DustOutput` variant of the same error also calls `remove_retrieve_btc_request` with `AmountTooLow` for the affected request without reimbursement: [7](#0-6) 

### Likelihood Explanation

Bitcoin network fees are volatile and can spike significantly between the time a user submits a withdrawal and the time the minter batches and processes it (up to `MAX_TIME_IN_QUEUE`). Any user who submits a withdrawal near the minimum threshold during a low-fee period and whose request is processed during a high-fee period is affected. This is a realistic, externally-triggerable scenario requiring no privileged access — only a standard `retrieve_btc` call followed by a fee spike.

### Recommendation

The `BuildTxError::AmountTooLow` and `BuildTxError::DustOutput` branches in `submit_pending_requests` should call `reimburse_canceled_requests` (or an equivalent reimbursement function) instead of silently finalizing with `FinalizedStatus::AmountTooLow`. The `reimbursement_account` field already exists on `RetrieveBtcRequest` for exactly this purpose, as demonstrated by the `InvalidTransaction` branch. The `FinalizedStatus` enum should either be extended with a `Reimbursed` variant or the `AmountTooLow` case should be removed in favor of the reimbursement flow.

### Proof of Concept

1. User calls `retrieve_btc(amount = min_retrieve_amount)` — ckBTC is burned from the ledger, `RetrieveBtcRequest` is enqueued.
2. Bitcoin network fees spike before the minter processes the batch.
3. `submit_pending_requests` calls `build_unsigned_transaction`, which returns `BuildTxError::AmountTooLow`.
4. The minter executes the `AmountTooLow` branch: calls `remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)`.
5. No reimbursement is scheduled. The user's ckBTC is permanently gone.
6. `retrieve_btc_status(block_index)` returns `RetrieveBtcStatus::AmountTooLow` — a terminal state with no recovery. [8](#0-7)

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
