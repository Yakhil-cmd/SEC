### Title
Byzantine Dealer DoS via Out-of-Range Chunk Values Bypassing `verify_chunking` — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

### Summary

A Byzantine dealer below the fault threshold can craft a dealing whose ciphertext chunk plaintexts lie outside `[0, CHUNK_MAX]` while still passing `verify_chunking`. When any receiver calls `load_threshold_signing_key` → `compute_threshold_signing_key` → `decrypt` → `dec_chunks`, the `HonestDealerDlogLookupTable` returns `None`, triggering `CheatingDealerDlogSolver::new(n, m)` followed by up to 255 BSGS iterations over a range of ~2·Z−1 ≈ 2^40 elements. The production code itself documents this: *"It may take hours to brute force a cheater's discrete log."*

---

### Finding Description

**Step 1 — `verify_chunking` does not bound individual chunk values.**

`verify_chunking` in `nizk_chunking.rs` performs exactly one explicit range check:

```rust
let ss = n * m * (CHUNK_SIZE - 1) * CHALLENGE_MASK;
let zz = 2 * NUM_ZK_REPETITIONS * ss;
let zz_big = Scalar::from_usize(zz);

for z_sk in nizk.z_s.iter() {
    if z_sk >= &zz_big {
        return Err(ZkProofChunkingError::InvalidProof);
    }
}
``` [1](#0-0) 

This bounds the *response scalars* `z_s`, which are linear combinations `Σ e_ijk · s_ij + σ_k`. It does **not** bound the individual witness values `s_ij`. A cheating dealer can choose `s_ij = v / Δ` (scalar-field division) for `v` in the honest range and small `Δ ∈ [2, 255]`, producing `z_s` values that still satisfy the bound while the actual chunk value is outside `[0, CHUNK_MAX]`. The test comment in `dlog_recovery.rs` confirms this explicitly:

> *"The bounds of the NIZK chunking proof are loose, so a malicious DKG participant can force us to search around 2^40 candidates for a discrete log."* [2](#0-1) 

**Step 2 — The bad dealing passes `verify_dealing` and enters the transcript.**

`verify_dealing` calls `verify_zk_proofs`, which calls `verify_ciphertext_integrity` and `verify_chunking`. Neither check bounds individual plaintext chunk values. [3](#0-2) [4](#0-3) 

**Step 3 — Slow path triggered in `dec_chunks`.**

When a receiver loads the transcript, `compute_threshold_signing_key` calls `decrypt` for every dealer's ciphertext. Inside `dec_chunks`, `HonestDealerDlogLookupTable::solve_several` only covers `[CHUNK_MIN, CHUNK_MAX]` = `[0, 65535]`. For an out-of-range chunk it returns `None`, triggering:

```rust
if dlogs.iter().any(|x| x.is_none()) {
    // Cheating dealer case
    let cheating_solver = CheatingDealerDlogSolver::new(n, m);
    for i in 0..dlogs.len() {
        if dlogs[i].is_none() {
            // It may take hours to brute force a cheater's discrete log.
            dlogs[i] = cheating_solver.solve(&powers[i]);
        }
    }
}
``` [5](#0-4) 

**Step 4 — `CheatingDealerDlogSolver::new` builds a 2 GiB BSGS table; `solve` loops 255 times.**

```rust

``` [6](#0-5) 

For a 28-node subnet with 16 chunks per receiver, the worst case is 16 × 255 BSGS searches over ~2^40 elements each. The `#[ignore]`-tagged benchmark `print_time_for_cheating_dlog_solver_to_run` measures this directly. [7](#0-6) 

---

### Impact Explanation

Every receiver node that calls `load_threshold_signing_key` on a transcript containing the malicious dealing will stall in `dec_chunks` for tens of minutes to hours. Since the transcript is shared, **all** receivers are affected simultaneously. This delays or prevents the subnet from completing DKG transcript loading, blocking threshold signing key availability and degrading subnet liveness during key rotation or initialization.

---

### Likelihood Explanation

- Requires only a single Byzantine dealer that is an eligible dealer in an active NI-DKG round — well within the `f = ⌊(n−1)/3⌋` fault budget.
- No privileged access, no key leakage, no governance majority needed.
- The attack is constructive: choose `s_ij = v · Δ⁻¹` in the scalar field for any `v ∈ [0, CHUNK_MAX]` and `Δ ∈ [2, 255]`, compute `C_ij = y_i^{r_j} · g1^{s_ij}`, generate the NIZK normally. The `z_s` bound is satisfied because `s_ij < CHUNK_MAX` in scalar-field magnitude.
- The `CheatingDealerDlogSolver` test confirms the solver works for exactly this class of inputs. [8](#0-7) 

---

### Recommendation

Add an explicit range check on the individual plaintext chunk values during dealing verification. After decrypting (or as part of `verify_chunking`), verify that each `s_ij ∈ [0, CHUNK_MAX]`. Alternatively, tighten the NIZK chunking proof to use a range proof that directly bounds individual witness values rather than only their linear combinations, eliminating the looseness that allows the cheating-dealer search space.

---

### Proof of Concept

```rust
// 1. Create a dealing with s_ij = v / delta (scalar field) for delta = 251 (largest prime < 256)
let delta = Scalar::from_u64(251);
let delta_inv = delta.inverse().unwrap();
let v = Scalar::from_u64(12345); // honest value in [0, CHUNK_MAX]
let s_cheat = v * delta_inv;     // outside [0, CHUNK_MAX] as an integer

// 2. Compute C_ij = y_i^r_j * g1^s_cheat — valid ciphertext with out-of-range chunk
// 3. Generate NIZK chunking proof with witness s_cheat — z_s values satisfy bound
//    because s_cheat < CHUNK_MAX in scalar magnitude
// 4. verify_chunking passes
// 5. Call dec_chunks — HonestDealerDlogLookupTable returns None
// 6. CheatingDealerDlogSolver::new(n, m) builds 2 GiB table
// 7. solve() iterates delta=1..255, each doing BSGS over ~2^40 range
// 8. Wall-clock time >> honest-case bound by >100x
```

The `cheating_dlog_instance` helper in the existing test suite already constructs exactly this scenario. [9](#0-8)

    pub fn new(n: usize, m: usize) -> Self {
        let scale_range = 1 << CHALLENGE_BITS;
        let ss = n * m * (CHUNK_SIZE - 1) * (scale_range - 1);
        let zz = 2 * NUM_ZK_REPETITIONS * ss;

        let bsgs_lo = 1 - zz as isize;
        let bsgs_range = 2 * zz - 1;

        let baby_giant = BabyStepGiantStep::new(
            Gt::generator(),
            bsgs_lo,
            bsgs_range,
            Self::MAX_TABLE_MBYTES,
            Self::LARGEST_TABLE_MUL,
        );
        Self {
            baby_giant,
            scale_range,
        }
    }

    /// Searches for discrete log for a malicious DKG participant whose NIZK
    /// chunking proof checks out, which implies certain bounds on the search
    ///
    /// This function is not constant time, but it only leaks information in the
    /// event that a dealer is already dishonest, and the only information it
    /// leaks is the value that the dishonest dealer sent.
    pub fn solve(&self, target: &Gt) -> Option<Scalar> {
        /*
        For some Delta in [1..E - 1] the answer s satisfies (Delta * s) in
        [1 - Z..Z - 1].

        For each delta in [1..E - 1] we compute target*delta and use
        baby-step-giant-step to find `scaled_answer` such that:
           base*scaled_answer = target*delta

         Then `base * (scaled_answer / delta) = target`
          (here division is modulo the group order
         That is, the discrete log of target is `scaled_answer / delta`.
        */
        let mut target_power = Gt::identity();
        for delta in 1..self.scale_range {
            target_power += target;

            if let Some(scaled_answer) = self.baby_giant.solve(&target_power) {
                let inv_delta = Scalar::from_usize(delta)
                    .inverse()
                    .expect("Delta is always invertible");
                let result = scaled_answer * inv_delta;
                return Some(result);
            }
        }
        None
    }

```rust
fn cheating_dlog_instance<R: rand::RngCore + rand::CryptoRng>(
    m: usize,
    rng: &mut R,
    use_max_delta: bool,
) -> (Scalar, Gt) {
    let z = 1069531200 * 16 * m as u64;

    let s = Scalar::from_u64(rng.r#gen::<u64>() % z);

    let delta = if use_max_delta {
        // Instead of maximum delta (255) we use the largest
        // delta that is prime. This leads to consistent results
        // from run to run, and is only slightly faster than
        // the most pessimal delta.
        251
    } else {
        std::cmp::max(1, rng.r#gen::<u16>() % 10)
    } as u64;
    let delta = Scalar::from_u64(delta);

    let delta_inv = delta.inverse().expect("Delta not invertible");

    let s_div_delta = s * delta_inv;

    let p = Gt::generator() * &s_div_delta;
    (s_div_delta, p)
}
