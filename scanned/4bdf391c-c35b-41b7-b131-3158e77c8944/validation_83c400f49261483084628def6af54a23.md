### Title
No Mechanism to Remove or Pause a Registered ckERC20 Token from the Ledger Suite Orchestrator - (`rs/ethereum/ledger-suite-orchestrator/src/state/mod.rs`)

---

### Summary

The Ledger Suite Orchestrator (LSO) for ckERC20 tokens provides a one-way `AddErc20Arg` registration path with no corresponding removal or pause capability. Once an ERC-20 token is registered via an NNS upgrade proposal, it is permanently embedded in the orchestrator's state. There is no `RemoveErc20Arg` or `PauseErc20Arg` variant in the governance interface, and the internal state data structure enforces insert-only semantics. This is a direct analog to the Frankencoin M-13 finding: a privileged entity (here, a ckERC20 ledger suite) can be registered but never revoked by governance.

---

### Finding Description

The `OrchestratorArg` Candid type — the sole governance interface for the LSO — exposes only three variants:

```
type OrchestratorArg = variant {
    UpgradeArg : UpgradeArg;
    InitArg : InitArg;
    AddErc20Arg : AddErc20Arg;
};
```

There is no `RemoveErc20Arg`, `PauseErc20Arg`, or equivalent. [1](#0-0) 

The internal state enforces this at the data-structure level. `ManagedCanisters::insert_once` asserts that the token is not already present and panics if it is, but there is no corresponding `remove` method anywhere in the type:

```rust
pub fn insert_once(&mut self, token_id: TokenId, canisters: Canisters) {
    assert_eq!(
        self.find_by_id(&token_id),
        None,
        "BUG: token {token_id:?} is already managed"
    );
    ...
}
``` [2](#0-1) 

The `State` struct exposes `record_new_erc20_token`, `record_manage_other_canisters`, `record_created_canister`, and `record_installed_canister` — all additive — but no removal or pause method. [3](#0-2) 

When an NNS proposal passes with `AddErc20Arg`, `lifecycle::add_erc20` is called in `post_upgrade`, which schedules `Task::InstallLedgerSuite` and calls `setup_tasks_and_timers()`. From that point on, the orchestrator's timer loop perpetually manages (tops up cycles, upgrades) the registered token's ledger and index canisters. [4](#0-3) 

The `post_upgrade` entry point confirms that `AddErc20Arg` is the only add-path and there is no remove-path: [5](#0-4) 

Additionally, `validate_add_erc20` explicitly rejects re-registration of an already-managed contract, confirming the one-way nature of the registration: [6](#0-5) 

---

### Impact Explanation

**Governance authorization bug.** Once an ERC-20 token is registered:

1. **Permanent cycle drain**: The orchestrator's `MaybeTopUp` timer task perpetually tops up the ledger and index canisters for every managed token. A token registered with incorrect parameters (e.g., wrong contract address, wrong decimals) will consume cycles indefinitely with no governance path to stop it.

2. **Minter notification is irrevocable**: On registration, the ckETH minter is notified of the new token and begins scraping Ethereum logs for it. There is no mechanism to un-notify the minter for a specific token without a full minter upgrade that patches internal state.

3. **No emergency stop for individual tokens**: The ckBTC/ckETH minter supports a `Mode::ReadOnly` that halts all operations, but this is a global switch affecting all tokens. There is no per-token pause in the orchestrator. A buggy or exploited ERC-20 ledger suite cannot be isolated without halting the entire minter.

4. **Upgrade-only remediation is fragile**: The only recovery path is an NNS upgrade proposal that manually patches the orchestrator's stable state — a non-standard, error-prone operation with no clean governance interface.

---

### Likelihood Explanation

**Medium.** Adding a new ckERC20 token requires an NNS proposal to pass, which requires significant voting power. However:

- The NNS has historically passed many `AddErc20Arg` proposals (ckUSDC, ckUSDT, ckPEPE, ckWSTETH, ckEURC, ckXAUT, etc.), demonstrating this is a routine governance action.
- A proposal with a subtly incorrect ERC-20 contract address or misconfigured parameters could pass review and be executed.
- Once executed, the absence of a removal path means the error is permanent unless a bespoke state-patching upgrade is crafted and passed — a much higher governance bar.
- The LSO README explicitly documents that adding a token is a multi-step, irreversible process with no documented rollback procedure. [7](#0-6) 

---

### Recommendation

1. **Add a `RemoveErc20Arg` variant** to `OrchestratorArg` that allows NNS governance to deregister a managed ERC-20 token, stopping cycle top-ups and notifying the minter to cease scraping for that token.

2. **Add a per-token pause mechanism** (analogous to `Mode::ReadOnly` in the ckBTC/ckETH minter) so that a specific token's ledger suite can be frozen without affecting others.

3. **Implement a `remove` method** on `ManagedCanisters` alongside `insert_once`, and expose it through a new `State::remove_erc20_token` method.

---

### Proof of Concept

**Step 1**: Submit an NNS upgrade proposal for the LSO with `AddErc20Arg` pointing to a valid but misconfigured ERC-20 contract (e.g., wrong `decimals`, or a contract that has a bug in its `Transfer` event).

**Step 2**: Proposal passes. `post_upgrade` calls `lifecycle::add_erc20`, which calls `record_new_erc20_token` → `insert_once`. The token is now permanently in `State.managed_canisters`. [8](#0-7) 

**Step 3**: Attempt to submit a new NNS proposal to remove the token. No `RemoveErc20Arg` variant exists in `OrchestratorArg`. The only available variants are `UpgradeArg`, `InitArg`, and `AddErc20Arg`. [9](#0-8) 

**Step 4**: The orchestrator's timer continues to top up cycles for the misconfigured token's ledger and index canisters on every `MaybeTopUp` tick. The ckETH minter continues to scrape Ethereum logs for the misconfigured contract address. There is no on-chain governance action that can stop this without a custom state-patching upgrade.

    read_state(|s| s.validate_config().expect("ERROR: invalid state"));

}

#[post_upgrade]
fn post_upgrade(orchestrator_arg: Option<OrchestratorArg>) {
    match orchestrator_arg {
        Some(OrchestratorArg::InitArg(_)) => {
            ic_cdk::trap("cannot upgrade canister state with init args");
        }
        Some(OrchestratorArg::UpgradeArg(upgrade_arg)) => {
            lifecycle::post_upgrade(Some(upgrade_arg))
        }
        Some(OrchestratorArg::AddErc20Arg(erc20)) => {
            lifecycle::add_erc20(erc20);
        }
        None => lifecycle::post_upgrade(None),
    }
}
