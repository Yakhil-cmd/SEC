### Title
ICP Rosetta Server Process Crash via Empty `ingress_expiries` Propagated Through payloads→combine→parse Pipeline — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The ICP Rosetta API's `/construction/payloads` endpoint does not validate that `ingress_start < ingress_end`. When a client supplies `ingress_start == ingress_end`, the `ingress_expiries` vec is silently left empty. This empty vec propagates through `/construction/combine` (producing a `SignedTransaction` with zero `EnvelopePair`s per request) and then causes an unconditional index-out-of-bounds **panic** at `updates[0]` in `/construction/parse`, crashing the Rosetta server process.

---

### Finding Description

**Stage 1 — `/construction/payloads` (missing guard)**

In `construction_payloads.rs`, `ingress_expiries` is populated by a `while now < ingress_end` loop: [1](#0-0) 

If the client supplies `ingress_start == ingress_end` (or `ingress_start > ingress_end`), the condition `now < ingress_end` is immediately false and the loop body never executes, leaving `ingress_expiries = []`. There is **no guard** that rejects this condition. By contrast, the ICRC1 Rosetta implementation has an explicit early-return at the equivalent location: [2](#0-1) 

Despite the empty `ingress_expiries`, the `updates` vec is still populated normally (e.g., for a transfer): [3](#0-2) 

`add_payloads` iterates over `ingress_expiries` and adds nothing to `payloads` when it is empty: [4](#0-3) 

The function returns successfully with `UnsignedTransaction { updates: [one entry], ingress_expiries: [] }` and an empty `payloads` list. [5](#0-4) 

---

**Stage 2 — `/construction/combine` (silent empty-envelope propagation)**

`construction_combine` iterates over `updates`, and for each update iterates over `ingress_expiries`: [6](#0-5) 

With `ingress_expiries` empty, the inner loop never executes. The signature-lookup code (lines 50–65) is inside that inner loop and is therefore never reached — no error is returned. `request_envelopes` stays empty, and the result is pushed: [7](#0-6) 

This produces `SignedTransaction { requests: [(RequestType::Send, vec![])] }` — a structurally valid CBOR blob that deserializes without error.

---

**Stage 3 — `/construction/parse` (unconditional panic)**

In `construction_parse.rs`, the signed-transaction branch maps each `(request_type, updates)` tuple (where `updates: Vec<EnvelopePair>`) by indexing `updates[0]`: [8](#0-7) 

When `updates` is an empty `Vec<EnvelopePair>`, `updates[0]` triggers a Rust index-out-of-bounds **panic**. This panic occurs inside a closure passed to `.map()` and propagates as an unrecoverable panic, crashing the Rosetta server process.

The `SignedTransaction` type and its `Request` alias confirm that the inner vec is `Vec<EnvelopePair>`, which can be empty: [9](#0-8) 

---

### Impact Explanation

An unprivileged HTTP client can crash the ICP Rosetta server process with a three-request sequence requiring no authentication, no funds, and no prior state. The crash is deterministic and repeatable. Because the panic is unrecovered, the process exits and the Rosetta service becomes unavailable until restarted. Any operator or exchange relying on the ICP Rosetta API for ledger operations (transfers, neuron management) is fully denied service.

---

### Likelihood Explanation

The attack requires only knowledge of the Rosetta API spec and the ability to set `ingress_start == ingress_end` in the metadata JSON. No privileged access, no key material, and no volumetric traffic are needed. The three-step sequence (`payloads` → `combine` → `parse`) is the standard Rosetta construction flow, making it trivially reachable by any API client.

---

### Recommendation

Add a guard in the ICP Rosetta `construction_payloads` handler that rejects requests where `ingress_start >= ingress_end`, mirroring the existing check in the ICRC1 implementation:

```rust
// In construction_payloads.rs, after computing ingress_start and ingress_end:
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(
        "ingress_start must be strictly before ingress_end"
    ));
}
```

Additionally, add a defensive check in `construction_parse` before indexing `updates[0]`, and in `construction_combine` before pushing an empty `request_envelopes` vec, to prevent any future empty-vec from reaching the indexing site.
