### Title
Small-Order Ed25519 Public Key Accepted in COSE/WebAuthn Ingress Path Without Torsion Check — (`rs/crypto/internal/crypto_lib/basic_sig/cose/src/lib.rs`)

### Summary

`parse_eddsa_ed25519` calls `ic_ed25519::PublicKey::deserialize_raw` which explicitly documents it does **not** check for prime-order subgroup membership. The all-zeros 32-byte encoding (`[0u8; 32]`) is a valid small-order (order-8) Ed25519 curve point — confirmed by production test code — and `deserialize_raw` accepts it. No `is_torsion_free()` guard is applied anywhere in the COSE → ingress verification call chain. The custom `verify_signature` uses ZIP215 cofactor-clearing semantics, which means a forged signature verifies against any small-order public key.

### Finding Description

**Step 1 — `[0u8; 32]` is a valid small-order curve point.**

The test at `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api/tests.rs:342-355` explicitly asserts:
- `CompressedEdwardsY([0; 32]).decompress()` succeeds (it IS on the curve)
- The decompressed point `.is_small_order()` returns `true` [1](#0-0) 

**Step 2 — `deserialize_raw` accepts it with no torsion check.**

The `deserialize_raw` docstring explicitly warns it does not check subgroup membership: [2](#0-1) 

`VerifyingKey::from_bytes` only checks that the bytes decode to a valid curve point — which `[0u8; 32]` satisfies — so `deserialize_raw([0u8; 32])` returns `Ok(...)`.

**Step 3 — `parse_eddsa_ed25519` has no torsion guard.** [3](#0-2) 

The ECDSA P-256 path calls `ic_secp256r1::PublicKey::deserialize_from_xy` which validates the point. The Ed25519 path calls only `deserialize_raw` — no `is_torsion_free()` call follows.

**Step 4 — The ingress verification path also has no torsion guard.**

`verify_basic_sig_by_public_key` → `verify_basic_sig_by_public_key` (standalone) calls `deserialize_raw` again without any torsion check: [4](#0-3) 

Compare with `verify_public_key` in the basic_sig API, which correctly calls `is_torsion_free()` — but this function is **not** invoked in the COSE/WebAuthn ingress path: [5](#0-4) 

**Step 5 — `verify_signature` uses ZIP215 cofactor-clearing, enabling forgery.** [6](#0-5) 

The check is: `([S]B − [k]A − R) · 8 == identity`

With A = all-zeros point (order divides 8): `[8][k]A = identity` for all k. The equation reduces to `[8][S]B = [8]R`. Choosing S=0 and R=identity (`[1,0,…,0]`): `identity = identity` ✓. The forged 64-byte signature `[1,0,…,0] ∥ [0,…,0]` verifies against the all-zeros key for **any message**.

**Step 6 — Full ingress call chain.** [7](#0-6) 

`validate_signature` → `public_key_from_bytes` → `parse_cose_public_key` → `parse_eddsa_ed25519` (accepts all-zeros key) → `validate_webauthn_sig` → `verify_basic_sig_by_public_key` → `verify_signature` (accepts forged sig).

### Impact Explanation

An unprivileged attacker can:
1. Compute the self-authenticating principal `P = new_self_authenticating(DER([0u8;32]))` — a fixed, deterministic value.
2. Craft a COSE OKP/EdDSA key with x=`[0u8;32]`, DER-wrap it, and submit ingress messages with sender=P.
3. Forge the signature as `[1,0,…,0,0,…,0]` (R=identity, S=0) — this verifies for any message under the all-zeros key.
4. Authenticate as principal P and call any canister as that principal.

The impact is scoped to principal P specifically. No legitimate user would derive their principal from the all-zeros key, so direct financial/asset theft is unlikely in practice. However, the invariant that small-order keys must be rejected is violated, and the forgery is unconditional and message-independent.

### Likelihood Explanation

The exploit is fully constructible from public information, requires no privileged access, and is locally testable. The only limiting factor is that the compromised principal P is a fixed, known value with no pre-existing assets. Likelihood of meaningful harm is **low** in practice, but the cryptographic invariant violation is **certain**.

### Recommendation

Add an `is_torsion_free()` check in `parse_eddsa_ed25519` immediately after `deserialize_raw` succeeds, mirroring the pattern already used in `verify_public_key` and `ValidNodeSigningPublicKey::try_from`:

```rust
let pk = ic_ed25519::PublicKey::deserialize_raw(x).map_err(|_| {
    CosePublicKeyParseError::MalformedPublicKey(AlgorithmId::Ed25519)
})?;
if !pk.is_torsion_free() {
    return Err(CosePublicKeyParseError::MalformedPublicKey(AlgorithmId::Ed25519));
}
```

The same guard should be added in `user_public_key_from_bytes` for the Ed25519 DER path.
