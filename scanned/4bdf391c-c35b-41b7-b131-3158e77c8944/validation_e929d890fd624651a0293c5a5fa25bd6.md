ckBTC Minter `estimate_withdrawal_fee` Panics Instead of Returning a Structured Error for Oversized Withdrawal Amounts, Masking the Effective Maximum Withdrawal Limit — (File: `rs/bitcoin/ckbtc/minter/src/main.rs`)

---

### Summary

The ckBTC minter exposes no programmatic way to determine the maximum withdrawal amount that can be processed without being reimbursed. `get_minter_info` only advertises `retrieve_btc_min_amount` with no corresponding maximum. The `estimate_withdrawal_fee` query endpoint — which callers naturally use to pre-validate a withdrawal — **panics (traps)** instead of returning a structured error when the requested amount would require more UTXOs than `max_num_inputs_in_transaction` allows. Meanwhile, `retrieve_btc` / `retrieve_btc_with_approval` accept any amount above the minimum, burn the caller's ckBTC, and only later discover the transaction is invalid, reimbursing the user minus `COST_OF_ONE_BILLION_CYCLES`. This is a direct analog to the ERC-4626 `maxDeposit()` violation: a public query interface that fails to reflect actual operational limits, causing callers to commit funds to operations that will silently fail at cost.

---

### Finding Description

**Root cause — `estimate_withdrawal_fee` panics instead of returning an error:** [1](#0-0) 

```rust
#[query]
fn estimate_withdrawal_fee(arg: EstimateFeeArg) -> WithdrawalFee {
    match mutate_state(|s| { ... }) {
        Ok(fee) => fee,
        Err(BuildTxError::NotEnoughFunds) => {
            panic!("ERROR: withdrawal amount is too large for the minter")
        }
        Err(BuildTxError::InvalidTransaction(
            e @ InvalidTransactionError::TooManyInputs { .. },
        )) => panic!(
            "ERROR: the minter cannot currently process such a large withdrawal amount ..."
        ),
    }
}
```

When the requested amount would require more inputs than `max_num_inputs_in_transaction`, the query **traps**. The caller receives an unstructured IC reject, not a typed error. There is no way to distinguish "amount too large" from any other failure, and no way to binary-search for the actual maximum.

**Root cause — `MinterInfo` exposes no maximum withdrawal amount:** [2](#0-1) 

`MinterInfo` carries `retrieve_btc_min_amount` and `deposit_btc_min_amount` but no `retrieve_btc_max_amount`. The effective maximum is a dynamic function of the current UTXO set composition and `max_num_inputs_in_transaction`, neither of which is surfaced.

**Root cause — `retrieve_btc` accepts any amount above the minimum with no upper-bound check:** [3](#0-2) 

Only `AmountTooLow` is checked at ingress time. No check for whether the amount would require too many inputs is performed before burning the caller's ckBTC and enqueuing the request.

**Contrast with ckDOGE minter (correct design):** [4](#0-3) 

The ckDOGE `estimate_withdrawal_fee` returns `Result<WithdrawalFee, EstimateWithdrawalFeeError>` and maps `TooManyInputs` → `Err(AmountTooHigh)`. Callers can handle this programmatically. The ckBTC minter has no equivalent.

---

### Impact Explanation

A user or integrator who:
1. Calls `estimate_withdrawal_fee` with a large amount → receives an IC trap (no structured error, no maximum hint).
2. Calls `retrieve_btc` / `retrieve_btc_with_approval` with that amount → ckBTC is **burned** from their account, the request is accepted (`RetrieveBtcOk` returned).
3. The minter later discovers the transaction would require too many inputs → schedules reimbursement.
4. User receives back `withdrawal_amount − COST_OF_ONE_BILLION_CYCLES` — a **permanent, unrecoverable financial loss**. [5](#0-4) 

The test `should_cancel_and_reimburse_large_withdrawal` confirms: the withdrawal is accepted, no BTC transaction is ever broadcast, and the user is reimbursed minus `COST_OF_ONE_BILLION_CYCLES`. This fee is non-trivial and is charged even though the failure was entirely predictable from the minter's own state.

---

### Likelihood Explanation

- **Reachable by any unprivileged ingress sender** — no special role required.
- **Naturally triggered** when a user accumulates many small UTXOs (e.g., from many small deposits) and then attempts a large withdrawal. The minter's own UTXO consolidation upgrade notes acknowledge this is a real operational scenario with ~60k UTXOs observed on mainnet. [6](#0-5) 

- Integrators who call `estimate_withdrawal_fee` to pre-validate amounts before calling `retrieve_btc` will receive a trap and have no structured signal to fall back on, making correct integration impossible without out-of-band knowledge of `max_num_inputs_in_transaction` and the current UTXO set.

---

### Recommendation

1. **Change `estimate_withdrawal_fee` return type** to `Result<WithdrawalFee, EstimateWithdrawalFeeError>` (matching the ckDOGE design) and map `NotEnoughFunds` and `TooManyInputs` to a typed `AmountTooHigh` variant instead of panicking.

2. **Add `retrieve_btc_max_amount: opt nat64` to `MinterInfo`** — computed dynamically as the largest amount the current UTXO set can satisfy within `max_num_inputs_in_transaction`. This gives integrators a queryable upper bound analogous to `retrieve_btc_min_amount`.

3. **Add an upfront check in `retrieve_btc` / `retrieve_btc_with_approval`** that calls `estimate_retrieve_btc_fee` before burning ckBTC, and returns `AmountTooLow` / a new `AmountTooHigh` error variant synchronously, so the user's funds are never committed to a request that is guaranteed to be reimbursed.

---

### Proof of Concept

```
1. Deposit (MAX_NUM_INPUTS + 100) UTXOs of small value to the ckBTC minter.
2. Call estimate_withdrawal_fee with amount = (MAX_NUM_INPUTS + 1) * utxo_value.
   → Query traps. No structured error. No maximum hint.
3. Call retrieve_btc_with_approval with the same amount.
   → Returns RetrieveBtcOk { block_index }.
   → ckBTC is burned from caller's account.
4. Advance time past MAX_TIME_IN_QUEUE.
   → No Bitcoin transaction is broadcast.
   → retrieve_btc_status_v2 returns Reimbursed with
     amount = withdrawal_amount − COST_OF_ONE_BILLION_CYCLES.
   → Caller has permanently lost COST_OF_ONE_BILLION_CYCLES of ckBTC.
```

#[query]
fn estimate_withdrawal_fee(
    arg: EstimateFeeArg,
) -> Result<WithdrawalFee, EstimateWithdrawalFeeError> {
    // This is a **query** endpoint, so mutating the state is not an issue
    // (even when called in replicated mode) since any change will be discarded.
    ic_ckbtc_minter::state::mutate_state(|s| {
        let fee_estimator = DOGECOIN_CANISTER_RUNTIME.fee_estimator(s);
        let withdrawal_amount = arg.amount.unwrap_or(s.fee_based_retrieve_btc_min_amount);

        ic_ckdoge_minter::fees::estimate_retrieve_doge_fee(
            &mut s.available_utxos,
            withdrawal_amount,
            s.last_median_fee_per_vbyte
                .expect("Bitcoin current fee percentiles not retrieved yet."),
            s.max_num_inputs_in_transaction,
            &fee_estimator,
        )
        .map_err(|e| match e {
            BuildTxError::NotEnoughFunds
            | BuildTxError::InvalidTransaction(InvalidTransactionError::TooManyInputs { .. }) => {
                EstimateWithdrawalFeeError::AmountTooHigh
            }
            BuildTxError::AmountTooLow | BuildTxError::DustOutput { .. } => {
                EstimateWithdrawalFeeError::AmountTooLow {
                    min_amount: s.fee_based_retrieve_btc_min_amount,
                }
            }
        })
    })
}

#[test]
fn should_cancel_and_reimburse_large_withdrawal() {
    let ckbtc = CkBtcSetup::new();
    let user = Principal::from(ckbtc.caller);
    let subaccount: Option<[u8; 32]> = Some([1; 32]);
    let user_account = Account {
        owner: user,
        subaccount,
    };

    // Step 1: deposit enough small UTXOs to exceed the max inputs limit.
    // We need at least max + 1 UTXOs for the withdrawal to trigger TooManyInputs,
    // plus a small buffer so there are leftover UTXOs in the set.
    const MAX_INPUTS: usize = ic_ckbtc_minter::state::DEFAULT_MAX_NUM_INPUTS_IN_TRANSACTION;
    const NUM_UTXOS: usize = MAX_INPUTS + 100;
    let deposit_value = 100_000_u64;
    let _deposited_utxos =
        ckbtc.deposit_utxos_with_value(user_account, &[deposit_value; NUM_UTXOS]);
    let balance_after_deposit = ckbtc.balance_of(user_account);
    assert_eq!(
        balance_after_deposit,
        Nat::from(NUM_UTXOS as u64 * (deposit_value - CHECK_FEE))
    );

    let withdrawal_amount = (MAX_INPUTS as u64 + 1) * deposit_value;
    ckbtc.approve_minter(user, withdrawal_amount, subaccount);
    let balance_before_withdrawal = ckbtc.balance_of(user_account);

    let RetrieveBtcOk { block_index } = ckbtc
        .retrieve_btc_with_approval(
            WITHDRAWAL_ADDRESS.to_string(),
            withdrawal_amount,
            subaccount,
        )
        .expect("retrieve_btc failed");

    let balance_after_withdrawal = ckbtc.balance_of(user_account);
    assert_eq!(
        balance_after_withdrawal,
        balance_before_withdrawal.clone() - Nat::from(withdrawal_amount)
    );

    assert_eq!(
        ckbtc.retrieve_btc_status_v2(block_index),
        RetrieveBtcStatusV2::Pending
    );

    ckbtc.env.advance_time(MAX_TIME_IN_QUEUE);

    let mempool = ckbtc.mempool();
    assert_eq!(
        mempool.len(),
        0,
        "no transaction should appear when being reimbursed"
    );

    let reimbursement_block_index = block_index + 1;
    let reimbursement_amount = withdrawal_amount - BitcoinFeeEstimator::COST_OF_ONE_BILLION_CYCLES;

    assert_matches!(
        ckbtc.retrieve_btc_status_v2(block_index),
        RetrieveBtcStatusV2::Reimbursed(reimbursement) if
        reimbursement.account == user_account &&
        reimbursement.amount == reimbursement_amount &&
        reimbursement.mint_block_index == reimbursement_block_index
    );

Upgrade the ckBTC minter canister to start the periodic consolidation of the set of UTXOs:
 * Currently, the minter has too many UTXOS of small value: it has around 60k UTXOs for a total of around 385 BTC.
 * This is a problem for large withdrawal requests (ckBTC -> BTC) involving typically more than 10 BTC, since a transaction created by the minter to cover the total withdrawal amount may need such a large number of inputs that the transaction becomes non-standard. As a stop-gap solution, proposal [137930](https://dashboard.internetcomputer.org/proposal/137930) limited the number of inputs in a transaction to 1000.
 * On the other hand, the minter needs to have sufficiently many UTXOs to be able to serve multiple withdrawal requests in parallel.
