### Title
Provider-Level Leaky Bucket Exhaustion by Single Node Operator Blocks All Sibling Operators — (`rs/registry/canister/src/rate_limits.rs`)

---

### Summary

The provider-level rate-limit bucket (`NODE_PROVIDER_MAX_SPIKE = 140`) has the **same maximum capacity** as each individual operator-level bucket (`NODE_OPERATOR_MAX_SPIKE = 140`). Because every rate-limited registry call commits one token from **both** the per-operator bucket and the shared per-provider bucket, a single node operator can legitimately exhaust the entire provider-level budget, blocking every sibling operator under the same provider from performing any rate-limited registry operation for up to 7 days.

---

### Finding Description

In `rs/registry/canister/src/rate_limits.rs`, both rate-limit tiers are configured with identical spike capacity: [1](#0-0) 

`NODE_PROVIDER_MAX_SPIKE = 20 × 7 = 140` and `NODE_OPERATOR_MAX_SPIKE = 20 × 7 = 140`.

Every call to `try_reserve_capacity_for_node_operator_operation` atomically reserves one token from the per-operator bucket **and** one token from the shared per-provider bucket: [2](#0-1) 

The provider bucket is keyed solely by provider ID, so it is shared across all operators registered under that provider. Because the per-operator spike limit equals the per-provider spike limit (both 140), a single operator can commit all 140 tokens from their own bucket while simultaneously committing all 140 tokens from the shared provider bucket.

The `do_update_node_domain` function (and every other rate-limited node-management call) follows the reserve-then-commit pattern: [3](#0-2) 

Each successful call commits 1 unit from both buckets. An operator with at least one node can toggle the domain value back and forth (e.g., `"a.com"` → `"b.com"` → `"a.com"` …) 140 times, exhausting the provider bucket without any per-call failure.

---

### Impact Explanation

Once the provider bucket is at zero, any call by a sibling operator to `try_reserve_capacity_for_node_operator_operation` fails at the provider reservation step with `RateLimiterError::NotEnoughCapacity`, regardless of how much capacity remains in the sibling's own operator bucket. The affected operations are:

- `do_update_node_domain_directly`
- `do_update_node_ipv4_config_directly`
- `do_add_node`
- `do_remove_node_directly`

The provider bucket refills at 1 token per `ONE_DAY_SECONDS / 20 ≈ 72 minutes`, so full recovery from a complete drain takes 7 days. During this window, sibling operators cannot add, remove, or reconfigure their nodes via direct calls. [4](#0-3) 

---

### Likelihood Explanation

The attacker must be a registered node operator under the same provider as the victim. This is a realistic scenario: large node providers commonly have multiple independent operators. The attack costs nothing beyond 140 cheap ingress calls to the registry canister and requires no special tooling. The `do_update_node_domain_directly` entry point is publicly callable by any registered node operator for their own nodes. [5](#0-4) 

---

### Recommendation

The provider-level bucket capacity should be sized proportionally to the number of operators it covers, not set equal to a single operator's budget. One approach: set `NODE_PROVIDER_MAX_SPIKE` to a multiple of `NODE_OPERATOR_MAX_SPIKE` (e.g., `N × 140` where N is the expected maximum number of operators per provider), or decouple the provider bucket's `max_capacity` from the operator bucket's `max_capacity` entirely. Alternatively, enforce a per-operator sub-quota within the provider bucket so that no single operator can consume more than their fair share of the shared budget. [6](#0-5) 

---

### Proof of Concept

State-machine test sketch (mirrors the existing `test_combined_rate_limiting` structure):

```rust
// Setup: two operators under one provider
let shared_provider = PrincipalId::new_user_test_id(1000);
let op1 = PrincipalId::new_user_test_id(1);
let op2 = PrincipalId::new_user_test_id(2);
// ... register both operators under shared_provider ...

let now = now_system_time();

// Step 1: exhaust provider bucket via operator 1
let available = registry.get_available_node_provider_op_capacity(shared_provider, now); // 140
let reservation = registry
    .try_reserve_capacity_for_node_operator_operation(now, op1, available)
    .unwrap();
registry
    .commit_used_capacity_for_node_operator_operation(now, reservation)
    .unwrap();

// Step 2: operator 2 attempts any rate-limited operation
let result = registry.try_reserve_capacity_for_node_operator_operation(now, op2, 1);
assert_eq!(result.unwrap_err(), RateLimiterError::NotEnoughCapacity);
// Operator 2's own bucket still has 140 capacity, but provider bucket is 0.
```

This directly mirrors the pattern already used in `test_do_update_node_domain_directly_fails_when_rate_limits_exceeded` and `test_combined_rate_limiting`. [7](#0-6) [8](#0-7)

    pub fn try_reserve_capacity_for_node_operator_operation(
        &self,
        now: SystemTime,
        node_operator_id: PrincipalId,
        requested_capacity: u64,
    ) -> Result<RateLimitReservation, RateLimiterError> {
        // Find the associated node provider ID for this node operator
        let node_provider_id = get_node_provider_id_for_operator_id(self, node_operator_id)
            .map_err(RateLimiterError::InvalidArguments)?;

        // First reserve from node operator rate limiter (primary)
        let operator_reservation = with_node_operator_rate_limiter(|rate_limiter| {
            rate_limiter.try_reserve(now, node_operator_key(node_operator_id), requested_capacity)
        })?;

        // Then reserve from node provider rate limiter (secondary)
        let provider_reservation = with_node_provider_rate_limiter(|rate_limiter| {
            rate_limiter.try_reserve(now, node_provider_key(node_provider_id), requested_capacity)
        })?;

        Ok(RateLimitReservation {
            operator_reservation,
            provider_reservation,
        })
    }

        let initial_operator_1_capacity =

        let initial_provider_capacity =
            registry.get_available_node_provider_op_capacity(shared_node_provider, now);

            .try_reserve_capacity_for_node_operator_operation(now, node_operator_1, 5)

        let after_operator_1_capacity =
            registry.get_available_node_operator_op_capacity(node_operator_1, now);
        let after_provider_capacity =

        assert_eq!(initial_operator_1_capacity - 5, after_operator_1_capacity);

            registry.get_available_node_provider_op_capacity(shared_node_provider, now);
        assert_eq!(initial_operator_1_capacity, restored_operator_1_capacity);
        assert_eq!(initial_provider_capacity - 3, restored_provider_capacity); // Only operator 2's reservation remains

        // Drop second reservation - should restore everything
        drop(reservation_2);

        let final_operator_1_capacity =
            registry.get_available_node_operator_op_capacity(node_operator_1, now);
        let final_operator_2_capacity =
            registry.get_available_node_operator_op_capacity(node_operator_2, now);
        let final_provider_capacity =

        assert_eq!(initial_operator_2_capacity, final_operator_2_capacity);

    #[test]
    fn test_do_update_node_domain_directly_fails_when_rate_limits_exceeded() {
        let (mut registry, node_id, node_operator_id, node_provider_id) = setup_registry_for_test();

        let payload = UpdateNodeDomainDirectlyPayload {
            node_id,
            domain: Some("example.com".to_string()),
        };

        let now = now_system_time();

        // Exhaust the rate limit capacity
        let available_operator =
            registry.get_available_node_operator_op_capacity(node_operator_id, now);
        let available_provider =
            registry.get_available_node_provider_op_capacity(node_provider_id, now);
        let available = available_operator.min(available_provider);
        let reservation = registry
            .try_reserve_capacity_for_node_operator_operation(now, node_operator_id, available)
            .unwrap();
        registry
            .commit_used_capacity_for_node_operator_operation(now, reservation)
            .unwrap();

        let error = registry
            .do_update_node_domain(payload, node_operator_id, now)
            .unwrap_err();
        assert_eq!(
            error,
            "Rate Limit Capacity exceeded. Please wait and try again later."
        );
    }
