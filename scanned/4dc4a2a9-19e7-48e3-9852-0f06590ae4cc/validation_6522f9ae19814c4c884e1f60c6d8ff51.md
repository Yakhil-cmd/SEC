### Title
Cycles Accounting Inconsistency: Insufficient Balance Silently Forgives Ingress Induction Debit During DTS Execution - (File: rs/replicated_state/src/canister_state/system_state.rs)

---

### Summary

`apply_ingress_induction_cycles_debit` in `SystemState` silently drops the portion of `ingress_induction_cycles_debit` that exceeds the canister's current balance, logging only an error and continuing. This is structurally identical to the NomadFacet pattern: a sub-operation (collecting the owed ingress fee) fails silently, the remaining debt is forgiven, and execution continues with incorrect accounting. A canister under Deterministic Time Slicing (DTS) can deliberately burn cycles during execution to drive its balance below the accumulated debit, causing the subnet to induct ingress messages for free.

---

### Finding Description

During DTS execution, when a canister has a paused execution, new ingress messages addressed to it are still inducted. Their induction cost is not immediately charged; instead it is accumulated in `ingress_induction_cycles_debit` via `add_postponed_charge_to_ingress_induction_cycles_debit`. This function asserts that the charge does not exceed `debited_balance()` at the time of the call, so the invariant `ingress_induction_cycles_debit ≤ cycles_balance` holds at induction time.

However, the canister's balance can decrease between induction and settlement:
- The canister can call `ic0.cycles_burn128` during execution.
- Storage reservation cycles can be deducted.
- Execution costs are prepaid and reduce the balance before `apply_ingress_induction_cycles_debit` is called.

When `apply_ingress_induction_cycles_debit` is finally called at the end of execution (in `finish` for both `call_or_task` and `install_code`), it computes:

```rust
// rs/replicated_state/src/canister_state/system_state.rs:1018-1038
let remaining_debit = self.ingress_induction_cycles_debit - self.cycles_balance;
debug_assert_eq!(remaining_debit.get(), 0);
if remaining_debit.get() > 0 {
    charging_from_balance_error.inc();
    error!(
        log,
        "[EXC-BUG]: Debited cycles exceed the cycles balance of {} by {} in install_code",
        canister_id,
        remaining_debit,
    );
    // Continue the execution by dropping the remaining debit, which makes
    // some of the postponed charges free.
}
self.consume_cycles(CompoundCycles::<IngressInduction>::new(
    self.ingress_induction_cycles_debit,
    cost_schedule,
));
self.ingress_induction_cycles_debit = Cycles::zero();
```

Because `Cycles` uses saturating arithmetic (as the comment on line 1018 confirms), `consume_cycles` called with `ingress_induction_cycles_debit > cycles_balance` saturates the balance to zero. The canister pays only `cycles_balance` cycles instead of the full `ingress_induction_cycles_debit`. The difference — `remaining_debit` — is silently forgiven. The debit counter is then cleared to zero, permanently losing the record of the owed amount.

This is the direct IC analog of the NomadFacet pattern: a failed sub-operation (collecting the full debt) causes the system to continue with an incorrect accounting state rather than reverting or deferring. [1](#0-0) 

The `handle_wasm_execution_of_cleanup_callback` in `response.rs` explicitly acknowledges and intentionally handles this same race for cleanup callbacks by reducing the debit before applying it. No equivalent protection exists for the general `apply_ingress_induction_cycles_debit` path used by update calls and `install_code`. [2](#0-1) 

The settlement call sites are:

- `rs/execution_environment/src/execution/call_or_task.rs` lines 483–490 (update/replicated-query finish)

### Impact Explanation

A canister that deliberately burns cycles during a DTS execution slice (via `ic0.cycles_burn128`) can drive its balance below the accumulated `ingress_induction_cycles_debit`. When execution finishes, the subnet silently forgives the shortfall. The canister receives ingress message induction services without paying the full fee. The subnet's cycle accounting records the full `ingress_induction_cycles_debit` as consumed (via `consume_cycles`), but the actual balance reduction is capped at `cycles_balance`, creating a permanent discrepancy between nominal and real cycle consumption. At scale, this erodes the subnet's cycle revenue and corrupts per-canister accounting metrics used for resource management.

---

### Likelihood Explanation

The preconditions are reachable by any unprivileged canister developer:

1. Deploy a canister with a long-running update method (triggering DTS slicing).
2. While the canister is paused, send ingress messages to it — this is an open, permissionless operation. Each inducted message adds to `ingress_induction_cycles_debit`.
3. The canister's Wasm code calls `ic0.cycles_burn128` during execution to reduce the balance below the accumulated debit.
4. On execution completion, `apply_ingress_induction_cycles_debit` silently forgives the shortfall.

No privileged access, governance majority, or threshold corruption is required. The only constraint is that the canister must burn at least as many cycles as the ingress induction cost it wishes to avoid — making this economically neutral for the attacker in the simple case, but still causing a real accounting inconsistency that compounds across many canisters or rounds.

---

### Recommendation

Replace the silent-forgive path in `apply_ingress_induction_cycles_debit` with the same cap-and-apply pattern already used in `handle_wasm_execution_of_cleanup_callback`: before calling `consume_cycles`, cap `ingress_induction_cycles_debit` to `min(ingress_induction_cycles_debit, cycles_balance)` and record the forgiven amount as a separate metric. Alternatively, defer the remaining debit rather than dropping it, so it is collected in a subsequent round. The `debug_assert_eq!(remaining_debit.get(), 0)` on line 1020 confirms the developers consider this state a bug; the fix should enforce the invariant rather than silently continuing past it. [5](#0-4)

### Proof of Concept

```
1. Deploy canister C with a DTS-eligible update method that:
   a. Calls ic0.instruction_counter_is_at_least(large_value) to force DTS slicing.
   b. After the slice boundary, calls ic0.cycles_burn128(D + 1) where D is the
      expected ingress_induction_cycles_debit.

2. Submit the update call to C. The first DTS slice starts; C is paused.

3. While C is paused, submit N ingress messages to C from any user principal.
   Each message adds cost_per_message to ingress_induction_cycles_debit.
   Total debit D = N * cost_per_message.

4. The next DTS slice resumes. C burns D + 1 cycles via ic0.cycles_burn128.
   Now cycles_balance < ingress_induction_cycles_debit.

5. Execution finishes. apply_ingress_induction_cycles_debit is called.
   remaining_debit = ingress_induction_cycles_debit - cycles_balance > 0.
   The error counter increments; the remaining debit is silently dropped.
   C's balance saturates to 0; the full debit is cleared.

6. C has received N ingress messages inducted without paying the full fee.
   The subnet's consumed_cycles metric overstates actual cycle collection by
   remaining_debit cycles.
``` [6](#0-5) [7](#0-6)
