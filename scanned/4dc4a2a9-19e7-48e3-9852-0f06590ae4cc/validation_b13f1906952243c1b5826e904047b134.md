### Title
Arbitrary `buyer` Principal in `refresh_buyer_tokens` Bypasses Confirmation-Text Consent — (`rs/sns/swap/canister/canister.rs`)

---

### Summary

The SNS Swap canister's `refresh_buyer_tokens` endpoint accepts a caller-supplied `buyer` string that is used without any authorization check to credit swap participation on behalf of an arbitrary principal. Because the confirmation text (the swap's legal-consent gate) is a public value, any unprivileged ingress sender can call `refresh_buyer_tokens` with a victim's principal and the correct confirmation text, forcing the victim's ICP — already sitting in the swap canister's per-principal subaccount — into a registered participation state without the victim's explicit consent.

---

### Finding Description

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` update endpoint resolves the buyer principal as follows:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()   // ← arbitrary principal, no auth check
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The resolved `p` is then passed directly into `refresh_buyer_token_e8s`, which:

1. Validates the confirmation text against the SNS-configured value (a public string).
2. Reads the ICP balance of `Account { owner: swap_canister, subaccount: principal_to_subaccount(&buyer) }`.
3. Credits that balance as the buyer's accepted participation.

```rust
self.validate_confirmation_text(confirmation_text)?;
...
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),
};
icp_ledger.account_balance(account).await
``` [2](#0-1) 

There is no check that `caller == buyer`. The `RefreshBuyerTokensRequest.buyer` proto field is documented as "If not specified, the caller is used," implying it is an optional convenience field, but the canister imposes no restriction on who may supply it. [3](#0-2) 

The confirmation text is set at SNS initialization time and is publicly readable. Any caller who knows it (trivially obtained from the swap's public state) can supply it on behalf of any victim.

---

### Impact Explanation

A victim (Alice) may have transferred ICP to the swap canister's per-principal subaccount — either intentionally while still deciding whether to agree to the confirmation text, or accidentally. During the swap's `Open` lifecycle, Alice cannot withdraw those funds; `error_refund_icp` is only available after the swap closes.

An attacker calls:

```
refresh_buyer_tokens({ buyer: "Alice's principal", confirmation_text: "<public text>" })
```

This:
- Registers Alice's ICP as accepted participation without Alice ever calling `refresh_buyer_tokens` herself.
- Bypasses the confirmation-text consent gate, which is the only mechanism ensuring a participant has explicitly agreed to the swap's terms.
- Locks Alice's ICP in the swap until finalization.
- If the swap commits, Alice receives SNS tokens she never consented to receive in exchange for her ICP.

The analog to the original report is direct: just as `deligateStake()` used an arbitrary `from` address to stake funds on behalf of a victim who had only approved the contract, `refresh_buyer_tokens` uses an arbitrary `buyer` principal to register participation on behalf of a victim who had only deposited ICP into the subaccount.

---

### Likelihood Explanation

- The `buyer` field is a plain string in a publicly callable update endpoint — no special privilege is required.
- The confirmation text is a public value readable from the swap's init state.
- Any ICP deposited to the swap canister's subaccount (even accidentally) is immediately exploitable.
- The attack requires a single ingress message and zero on-chain cost beyond the IC ingress fee.

---

### Recommendation

Enforce that the resolved buyer principal equals the ingress caller:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, requested, "caller must match buyer");
        requested
    };
    ...
}
```

Alternatively, remove the `buyer` override field entirely and always use `caller_principal_id()`, since the only legitimate use case for the override (third-party notification) does not require bypassing the caller check — the ICP balance is read from the ledger, not from the caller.

---

### Proof of Concept

1. Alice calls `icrc1_transfer` on the ICP ledger, sending 10 ICP to `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(Alice) }`.
2. Alice has not yet called `refresh_buyer_tokens` because she is still reviewing the confirmation text.
3. Attacker reads the swap's public `confirmation_text` from `get_sale_parameters`.
4. Attacker submits an ingress message to the swap canister:
   ```
   refresh_buyer_tokens({
     buyer: Alice.to_string(),
     confirmation_text: Some("<public confirmation text>")
   })
   ```
5. The swap canister resolves `p = Alice`, reads Alice's subaccount balance (10 ICP), validates the confirmation text (passes, because the attacker supplied the correct public string), and credits Alice's participation.
6. Alice's 10 ICP is now locked as accepted swap participation. Alice never explicitly agreed to the confirmation text. [4](#0-3) [5](#0-4)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
