### Title
SNS Swap Canister `refresh_buyer_token_e8s` - Swap Permanently Stuck When Remaining Capacity < `min_participant_icp_e8s` and All Participants at Per-Participant Maximum - (`rs/sns/swap/src/swap.rs`)

### Summary

The SNS Swap canister enforces a per-participant minimum (`min_participant_icp_e8s`) both on the raw transferred amount and on the effective accepted amount after capping to the remaining swap capacity. When the remaining direct participation capacity (`available_direct_participation_e8s()`) falls below `min_participant_icp_e8s` and all existing participants have already reached `max_participant_icp_e8s`, the swap enters a permanently stuck state: no new participant can join, no existing participant can top up, `min_direct_participation_icp_e8s` cannot be reached, and the swap cannot commit. Participants' ICP is locked until `swap_due_timestamp_seconds` (up to 14 days), after which the swap aborts.

### Finding Description

In `refresh_buyer_token_e8s`, the swap first checks that the raw ledger balance meets the minimum: [1](#0-0) 

It then caps the accepted increment to the remaining capacity: [2](#0-1) 

Then it applies a second minimum check on the effective (post-cap) balance: [3](#0-2) 

For a **new participant** (`old_amount_icp_e8s = 0`), `new_balance_e8s = actual_increment_e8s`. If `available_direct_participation_e8s() < min_participant_icp_e8s`, then `actual_increment_e8s < min_participant_icp_e8s`, and the second check always rejects them.

For an **existing participant**, `new_balance_e8s = old_amount + actual_increment`. If `old_amount_icp_e8s >= min_participant_icp_e8s`, the check passes regardless of increment size. However, if the existing participant has already reached `max_participant_icp_e8s`, the cap at line 1237 prevents any further increase: [4](#0-3) 

The initialization validation in `validate_participation_constraints` does **not** enforce the invariant `min_participant_icp_e8s <= max_direct_participation_icp_e8s - min_direct_participation_icp_e8s`: [5](#0-4) 

The checks performed are only: `max_direct >= min_direct`, `max_participant >= min_participant`, `max_participant <= max_direct`, and `max_direct >= min_participants * min_participant`. The critical missing constraint allows a valid configuration to produce a stuck swap.

The `sufficient_participation` check that gates `can_commit` requires `min_direct_participation_icp_e8s` to be reached: [6](#0-5) [7](#0-6) 

The swap can only commit early if the ICP target is reached or the swap is due: [8](#0-7) 

The developers are partially aware of the scenario where remaining capacity < `min_participant_icp_e8s` (as evidenced by the test `test_swap_cannot_finalize_via_new_participation_if_remaining_lt_minimal_participation_amount` at line 5707), but the test only covers the case where an existing participant below `max_participant_icp_e8s` can rescue the swap. It does not cover the case where all participants are already at `max_participant_icp_e8s`. [9](#0-8) 

### Impact Explanation

When the stuck state is reached:
- No new participant can call `refresh_buyer_tokens` successfully (effective amount < `min_participant_icp_e8s`)
- No existing participant at `max_participant_icp_e8s` can top up
- `min_direct_participation_icp_e8s` is unreachable, so `sufficient_participation()` returns false
- `can_commit()` returns false; `can_abort()` returns false until `swap_due_timestamp_seconds`
- All participants' ICP is locked in the swap canister for up to `MAX_SALE_DURATION_SECONDS` (14 days) [10](#0-9) 

After expiry the swap aborts and participants can reclaim ICP via `error_refund_icp`, so funds are not permanently lost. The impact is a bounded DoS (temporary lock of participant ICP).

### Likelihood Explanation

The condition `min_participant_icp_e8s > max_direct_participation_icp_e8s - min_direct_participation_icp_e8s` is not prevented by any validation. A swap creator who sets, for example, `min_participant_icp_e8s = 3 ICP`, `min_direct_participation_icp_e8s = 9 ICP`, `max_direct_participation_icp_e8s = 10 ICP`, `max_participant_icp_e8s = 4 ICP` passes all current checks. Two participants each contributing 4 ICP (both at max) leave 2 ICP of capacity, which is below the 3 ICP minimum, and the swap is stuck. This can occur organically or be deliberately triggered by an early participant who fills the swap to the critical state.

### Recommendation

Add a validation in `SnsInitPayload::validate_participation_constraints` (in `rs/sns/init/src/lib.rs`) to enforce:

```
min_participant_icp_e8s <= max_direct_participation_icp_e8s - min_direct_participation_icp_e8s
```

This ensures that there is always at least one minimum-sized contribution of headroom between `min_direct` and `max_direct`, preventing the stuck state. Alternatively, in `refresh_buyer_token_e8s`, when `available_direct_participation_e8s() < min_participant_icp_e8s`, the swap could be automatically transitioned to a state where `validate_possibility_of_direct_participation` blocks further participation and triggers early abort/commit evaluation.

### Proof of Concept

Concrete stuck configuration (all validations in `validate_participation_constraints` pass):

```
min_participant_icp_e8s          = 3 ICP
min_direct_participation_icp_e8s = 9 ICP
max_direct_participation_icp_e8s = 10 ICP
max_participant_icp_e8s          = 4 ICP
min_participants                 = 1
```

Checks that pass:
- `max_direct (10) >= min_direct (9)` ✓
- `max_participant (4) >= min_participant (3)` ✓
- `max_participant (4) <= max_direct (10)` ✓
- `max_direct (10) >= min_participants (1) * min_participant (3) = 3` ✓

Attack sequence:
1. Participant A calls `refresh_buyer_tokens` with 4 ICP → accepted (4 ICP total)
2. Participant B calls `refresh_buyer_tokens` with 4 ICP → accepted (8 ICP total)
3. Both A and B are at `max_participant_icp_e8s = 4 ICP`; `available = 2 ICP`
4. New participant C tries 3 ICP: raw check passes (3 >= 3), but `actual_increment = min(2, 3) = 2`, `new_balance = 0 + 2 = 2 < 3` → **rejected** at line 1241
5. New participant C tries 2 ICP: raw check fails (2 < 3) → **rejected** at line 1202
6. Participant A tries to top up 1 ICP: `new_balance = 4 + min(2,1) = 5 > max_participant (4)` → capped to 4, no change
7. `current_direct = 8 < min_direct = 9` → `sufficient_participation() = false` → `can_commit() = false`
8. Swap is stuck until `swap_due_timestamp_seconds`; all ICP locked [11](#0-10) [12](#0-11)
