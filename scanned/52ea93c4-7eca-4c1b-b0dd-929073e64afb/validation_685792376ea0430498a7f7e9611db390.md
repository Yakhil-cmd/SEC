### Title
Voting Rewards Permanently Lost When Proposals Are Settled With Zero Voter Participation - (`rs/sns/governance/src/governance.rs`)

### Summary

In the SNS governance canister, when one or more proposals reach `ReadyToSettle` state but no neuron casts a vote, the entire rewards purse for that round is permanently destroyed rather than rolled over to the next reward event. The rollover gate (`rewards_rolled_over()`) checks only whether `settled_proposals` is empty, not whether any rewards were actually distributed. This is the direct IC analog of the BlueberryStaking M-03 finding: a timestamp/epoch marker is advanced even when no distribution can occur, causing the accumulated reward pool to be silently discarded.

### Finding Description

`distribute_rewards` in `rs/sns/governance/src/governance.rs` computes a `rewards_purse_e8s` from the token supply and elapsed rounds, then attempts to distribute it proportionally to neurons that voted. The distribution is gated on `total_reward_shares`:

```rust
if total_reward_shares == dec!(0) {
    log!(ERROR, "Warning: total_reward_shares is 0. Therefore, we skip increasing \
         neuron maturity. ...");
} else {
    // distribute to neurons
}
``` [1](#0-0) 

When `total_reward_shares == 0`, `distributed_e8s_equivalent` remains `0`. Execution then falls through to settle all proposals and write the new `RewardEvent`:

```rust
self.proto.latest_reward_event = Some(RewardEvent {
    round: new_reward_event_round,
    settled_proposals: considered_proposals,   // ← non-empty
    distributed_e8s_equivalent,               // ← 0
    total_available_e8s_equivalent,           // ← full purse, but never used
    ...
})
``` [2](#0-1) 

The rollover decision for the **next** round is made by `e8s_equivalent_to_be_rolled_over`, which delegates to `rewards_rolled_over`:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty()
}
``` [3](#0-2) 

Because `settled_proposals` is non-empty (the proposals were settled even though nobody voted), `rewards_rolled_over()` returns `false`, and `e8s_equivalent_to_be_rolled_over()` returns `0`:

```rust
pub(crate) fn e8s_equivalent_to_be_rolled_over(&self) -> u64 {
    if self.rewards_rolled_over() {
        self.total_available_e8s_equivalent.unwrap_or_default()
    } else {
        0   // ← returned here; purse is gone
    }
}
``` [4](#0-3) 

The next call to `distribute_rewards` starts with a fresh purse of `0` rolled-over e8s, so the entire `rewards_purse_e8s` from the affected round is permanently destroyed.

The same structural defect exists in NNS governance: `calculate_voting_rewards` returns `None` for the distribution when `total_voting_rights < 0.001`, but the `RewardEvent` it returns still carries the non-empty `settled_proposals` list, causing the NNS rollover gate (`rewards_rolled_over` in `rs/nns/governance/src/reward/calculation.rs`) to return `false` and discard the purse. [5](#0-4) [6](#0-5) 

### Impact Explanation

Every reward round in which at least one proposal is settled but zero neurons voted results in the complete loss of that round's reward purse. The purse is proportional to the total governance token supply and the current reward rate. For a mid-sized SNS with 1 000 000 tokens and a 10 % annual rate, each affected daily round destroys ≈ 274 tokens of maturity that should have been available to future voters. Repeated occurrences compound: because the lost purse is never rolled over, the deficit is not recovered in subsequent rounds. Neuron holders receive permanently reduced maturity, degrading the economic incentive to participate in governance.

### Likelihood Explanation

The condition `total_reward_shares == 0` with non-empty `considered_proposals` is reachable in two ways:

1. **Organic low-participation**: A newly launched or dormant SNS may have proposals that expire without any neuron casting a vote. This requires no adversarial action.
2. **Deliberate griefing**: Any principal holding enough SNS tokens to create a proposal can submit proposals on topics that the community ignores or is unable to vote on before the voting period closes. The proposal rejection fee is the only cost, and it is paid regardless of whether the attacker intends to lose it. The attacker does not need any privileged role.

The entry path is fully unprivileged: `manage_neuron` → `make_proposal` (canister call), followed by waiting for the voting period to expire with no votes, then the periodic `run_periodic_tasks` → `distribute_rewards` execution path. [7](#0-6) 

### Recommendation

`rewards_rolled_over()` should reflect whether rewards were actually distributed, not merely whether proposals were settled. One correct fix is to change the predicate to also check `distributed_e8s_equivalent == 0`:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty() || self.distributed_e8s_equivalent == 0
}
```

Alternatively, when `total_reward_shares == 0` and proposals exist, the code should explicitly carry the full `rewards_purse_e8s` forward by setting `settled_proposals` to empty in the emitted `RewardEvent` (treating the round as a rollover), or by storing the undistributed purse in a dedicated field that is always added to the next round's starting balance regardless of `settled_proposals`.

The same fix must be applied symmetrically to the NNS governance rollover logic in `rs/nns/governance/src/reward/calculation.rs`.

### Proof of Concept

1. Deploy an SNS with `voting_rewards_parameters` set (non-zero reward rate).
2. Create a proposal via `manage_neuron` → `make_proposal` from any neuron with sufficient stake.
3. Allow the proposal's voting period to expire without any neuron casting a vote. The proposal transitions to `ReadyToSettle`.
4. Wait for `run_periodic_tasks` to call `distribute_rewards` (one reward round later).
5. Observe that `latest_reward_event.distributed_e8s_equivalent == 0` and `latest_reward_event.settled_proposals` is non-empty.
6. Wait for the next reward round. Observe that `latest_reward_event.total_available_e8s_equivalent` does **not** include the purse from step 4 — it starts fresh from the new round's supply fraction only, confirming the prior round's purse was permanently destroyed.
