### Title
Unvalidated `ingress_bytes_hash` Length in `TryFrom<pb::StrippedIngressMessage>` Enables Byzantine Peer to Force Bounded Assembly Stall — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types.rs`)

---

### Summary

`TryFrom<pb::StrippedIngressMessage> for SignedIngressId` wraps the raw protobuf byte vector directly into a `CryptoHashOf` with no length check. A Byzantine peer can supply a zero-length or arbitrarily large `ingress_bytes_hash`, which is accepted as a valid `SignedIngressId`. Every subsequent attempt to match that hash against a real ingress message (which always produces a 32-byte SHA-256 digest) fails permanently, driving `download_stripped_message` into its indefinite retry loop for the lifetime of the block proposal.

---

### Finding Description

**Root cause — `types.rs` line 47:**

```rust
let ingress_bytes_hash = CryptoHashOf::from(CryptoHash(value.ingress_bytes_hash));
```

`CryptoHash` is a plain `Vec<u8>` newtype; `CryptoHashOf::from` is an infallible wrapper. No length invariant is enforced. [1](#0-0) 

**Acceptance path — `stripped.rs` lines 100–107:**

`TryFrom<pb::StrippedBlockProposal>` iterates over `value.ingress_messages` and calls `SignedIngressId::try_from` for each entry. Because the inner conversion never rejects a bad-length hash, a stripped block proposal carrying N ingress entries with 0-byte or 1 MB hashes is fully accepted. [2](#0-1) 

**Pool-check bypass — `assembler.rs` lines 327–344:**

`get_or_fetch` looks up the ingress by `ingress_message_id` in the pool, then compares `SignedIngressId::from(&ingress_message)` (real 32-byte hash) against the stored `signed_ingress_id` (malformed hash). The comparison always fails, so control falls through to `download_stripped_message`. [3](#0-2) 

**Infinite inner loop — `download.rs` lines 301–305, 332–375:**

`download_stripped_message` is built with `with_max_elapsed_time(None)` and loops forever. It contains no bouncer check of its own. [4](#0-3) 

**`parse_response` always mismatches — `download.rs` lines 388–407:**

Every honest peer returns the real ingress bytes; `SignedIngressId::from(&ingress)` recomputes the correct 32-byte hash, which never equals the malformed hash stored in `ingress_id`. The function always returns `Err(ParseResponseError::MessageIdMismatch)`, keeping the loop alive. [5](#0-4) 

**Outer bouncer escape hatch — `assembler.rs` lines 233–249:**

The outer `assemble_message` loop does have a `tokio::select!` arm that aborts when the bouncer marks the artifact `Unwanted`, dropping `join_set` and cancelling all spawned `get_or_fetch` tasks. This is the only termination path. [6](#0-5) 

---

### Impact Explanation

For every ingress entry carrying a malformed hash, the victim node spawns a `get_or_fetch` task that drives `download_stripped_message` in an infinite exponential-backoff loop (5 s → 120 s intervals, no elapsed-time cap). The node cannot assemble the block proposal and cannot deliver it to the consensus layer until the bouncer eventually marks the artifact `Unwanted` (i.e., after the block is finalized or superseded by honest peers). During this window:

- The victim node wastes CPU and network bandwidth retrying unanswerable RPC requests.
- Block assembly for that proposal is completely stalled on the victim node.
- If the Byzantine peer targets multiple nodes simultaneously (each receiving the malformed stripped proposal before honest peers respond), the effective finalization latency for that round increases proportionally.

The stall is **bounded** — it terminates when the bouncer fires — so this is not a permanent denial of service. However, it is a concrete, repeatable degradation of consensus throughput achievable by a single below-threshold Byzantine peer.

---

### Likelihood Explanation

The attacker must be a subnet peer (reachable via QUIC transport). They observe the `ConsensusMessageId` of a real block proposal being advertised on the network, craft a `pb::StrippedBlockProposal` with that ID, a syntactically valid (but content-arbitrary) `pruned_block_proposal`, and one or more `pb::StrippedIngressMessage` entries whose `ingress_bytes_hash` is not 32 bytes. They then advertise this artifact and respond to the victim's download request before honest peers. No privileged access, no key material, and no majority corruption is required.

---

### Recommendation

Add an explicit length check in `TryFrom<pb::StrippedIngressMessage> for SignedIngressId`:

```rust
const CRYPTO_HASH_LEN: usize = 32;

impl TryFrom<pb::StrippedIngressMessage> for SignedIngressId {
    type Error = ProxyDecodeError;

    fn try_from(value: pb::StrippedIngressMessage) -> Result<Self, Self::Error> {
        let ingress_message_id =
            try_from_option_field(value.stripped, "StrippedIngressMessage::stripped")?;
        if value.ingress_bytes_hash.len() != CRYPTO_HASH_LEN {
            return Err(ProxyDecodeError::Other(format!(
                "ingress_bytes_hash must be {} bytes, got {}",
                CRYPTO_HASH_LEN,
                value.ingress_bytes_hash.len()
            )));
        }
        let ingress_bytes_hash = CryptoHashOf::from(CryptoHash(value.ingress_bytes_hash));
        Ok(SignedIngressId { ingress_message_id, ingress_bytes_hash })
    }
}
```

This makes the `TryFrom` conversion reject any malformed stripped block proposal at the deserialization boundary, before any assembly task is spawned.

### Proof of Concept

```rust

for bad_len in [0usize, 1, 31, 33, 1024, 1_048_576] {
    let proto = pb::StrippedIngressMessage {
        stripped: Some(/* valid IngressMessageId proto */),
        ingress_bytes_hash: vec![0u8; bad_len],
    };
    assert!(
        SignedIngressId::try_from(proto).is_err(),
        "Expected rejection for ingress_bytes_hash length {bad_len}"
    );
}
```

Without the fix, all six cases return `Ok`, and any downstream `download_stripped_message` call for such an ID loops until the bouncer fires.
