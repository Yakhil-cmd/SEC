### Title
Reducing `VotingPowerEconomics` Refresh Window via `ManageNetworkEconomics` Retroactively Clears Following of Existing Neurons - (`rs/nns/governance/src/neuron/types.rs`)

---

### Summary

The NNS Governance canister's `prune_following` function evaluates whether a neuron's following should be cleared using the **current** global `VotingPowerEconomics` parameters (`start_reducing_voting_power_after_seconds` and `clear_following_after_seconds`). These parameters are mutable via `ManageNetworkEconomics` governance proposals. Reducing either parameter retroactively reclassifies neurons that were previously "fresh" as "stale," causing their following to be irreversibly cleared in the next `PruneFollowingTask` execution — which runs every 10 seconds.

---

### Finding Description

**Root cause — mutable global parameter applied retroactively to existing neuron state:**

The `prune_following` method computes the freshness threshold using the current global `VotingPowerEconomics`:

```rust
// rs/nns/governance/src/neuron/types.rs
pub(crate) fn prune_following(
    &mut self,
    voting_power_economics: &VotingPowerEconomics,
    now_seconds: u64,
) -> u64 {
    let is_fresh = self.voting_power_refreshed_timestamp_seconds
        >= now_seconds
            .saturating_sub(
                voting_power_economics.get_start_reducing_voting_power_after_seconds(),
            )
            .saturating_sub(voting_power_economics.get_clear_following_after_seconds());
    if is_fresh {
        return 0;
    }
    // Clear all following except ManageNeuron.
    self.followees
        .retain(|topic, _| *topic == Topic::NeuronManagement as i32);
    ...
}

The `PruneFollowingTask` runs every 10 seconds and calls `Governance::prune_some_following`, which fetches the **current** `VotingPowerEconomics` from `self.heap_data.economics` at execution time:

``` [2](#0-1) 

The `PruneFollowingTask` is scheduled to run every 10 seconds: [3](#0-2) 

The `ManageNetworkEconomics` proposal type, executed via `perform_manage_network_economics_impl`, allows governance to update `VotingPowerEconomics` — including reducing `start_reducing_voting_power_after_seconds` (default: 6 months) and `clear_following_after_seconds` (default: 1 month): [4](#0-3) 

The validation bounds for `neuron_minimum_dissolve_delay_to_vote_seconds` are enforced, but **no bounds are enforced on `start_reducing_voting_power_after_seconds` or `clear_following_after_seconds`** — they are only required to be `Some`: [5](#0-4) 

The `VotingPowerEconomics` parameters are mutable via governance proposals and are explicitly designed to be changeable: [6](#0-5) 

A test explicitly demonstrates that reducing these parameters to very small values causes **all** neurons — including those that were previously "fresh" — to have their following cleared: [7](#0-6) 

The test comment at line 595 states: *"This shows that VotingPowerEconomics is used when pruning following, not the old constant(s)."* This confirms the design: the current global parameter is always used, with no per-neuron snapshot of the parameter at the time of refresh.

---

### Impact Explanation

When `start_reducing_voting_power_after_seconds` is reduced via a `ManageNetworkEconomics` proposal:

1. **Immediate, irreversible following loss**: Neurons that refreshed their voting power within the old window but outside the new (shorter) window are reclassified as stale. Within 10 seconds, the `PruneFollowingTask` clears their following on all topics except `NeuronManagement`.

2. **Voting reward loss**: Neurons that vote via following receive voting rewards. Losing following means they must vote manually or re-establish following to continue receiving rewards. Neurons that do not notice the change will silently stop receiving rewards.

3. **Governance participation disruption**: If many neurons lose their following simultaneously (e.g., a large reduction in the refresh window), proposals may fail to reach quorum via the following mechanism, disrupting NNS governance.

4. **No warning or grace period**: The clearing is immediate and irreversible without neuron owner action. There is no notification mechanism.

The impact is a **governance authorization bug**: a legitimate governance parameter change has an unintended retroactive effect on existing neuron state, analogous to the Particle LAMM `LOAN_TERM` reduction causing existing loans to be liquidated early.

---

### Likelihood Explanation

- A `ManageNetworkEconomics` proposal requires a majority of NNS voting power to pass. This is a high bar, but such proposals are a normal part of NNS operation (the Mission 70 changes included exactly this type of parameter update).
- The retroactive effect is a **design flaw** that could be triggered unintentionally by a well-intentioned governance change. The proposers may intend to tighten the refresh requirement for *future* behavior, not realizing it immediately clears following for neurons that refreshed within the old window.
- The NNS CHANGELOG confirms that `VotingPowerEconomics` parameters have already been changed in production (Mission 70, Proposal 141441), demonstrating that such changes are realistic and have occurred. [8](#0-7) 

---

### Recommendation

1. **Store the effective refresh window at the time of each neuron's last refresh** (analogous to the Particle report's recommendation to store `loanTerm` in the lien struct). When pruning, use the per-neuron stored window rather than the current global parameter.

2. **Alternatively, implement a grace period**: when `VotingPowerEconomics` is updated to stricter values, apply the new parameters only to neurons that refresh *after* the update. Neurons that refreshed before the update should be evaluated against the parameter value that was in effect at the time of their refresh.

3. **At minimum, add validation bounds** on `start_reducing_voting_power_after_seconds` and `clear_following_after_seconds` to prevent extreme reductions, and document the retroactive effect in the proposal type.

---

### Proof of Concept

1. Neuron A refreshes its voting power 4 months ago. Under the current `start_reducing_voting_power_after_seconds = 6 months` and `clear_following_after_seconds = 1 month`, the freshness threshold is `now - 7 months`. Neuron A's `voting_power_refreshed_timestamp_seconds` is `now - 4 months`, which is ≥ `now - 7 months`, so it is "fresh." Its following is safe.

2. A `ManageNetworkEconomics` proposal passes, reducing `start_reducing_voting_power_after_seconds` to 3 months (keeping `clear_following_after_seconds` at 1 month). The new freshness threshold is `now - 4 months`.

3. Within 10 seconds, the `PruneFollowingTask` runs. It calls `prune_following` with the new `VotingPowerEconomics`. Neuron A's `voting_power_refreshed_timestamp_seconds` is `now - 4 months`, which equals the new threshold `now - 4 months` — it is now classified as stale (the check is `>=`, so equality is still "fresh," but any neuron that refreshed even 1 second before the threshold is cleared).

4. Any neuron that refreshed between 3 months and 7 months ago (the old window) has its following on all topics except `NeuronManagement` cleared immediately.

5. Neuron A's owner is not notified. Their following-based voting stops, their voting rewards drop, and they must manually re-establish following.

const PRUNE_FOLLOWING_INTERVAL: Duration = Duration::from_secs(10);

// Once this amount of instructions is used by the
// Governance::prune_some_following, it stops, saves where it is, schedules more
// pruning later, and returns.
//
// Why this value seems to make sense:
//
// I think we can conservatively estimate that it takes 2e6 instructions to pull
// a neuron from stable memory. If we assume 200e3 neurons are in stable memory,
// then 400e9 instructions are needed to read all neurons in stable memory.
// 400e9 instructions / 50e6 instructions per batch = 8e3 batches. If we process
// 1 batch every 10 s (see PRUNE_FOLLOWING_INTERVAL), then it would take less
// than 23 hours to complete a full pass.
//
// This comes to 1.08 full passes per day. If each full pass uses 400e9
// instructions, then we use 432e9 instructions per day doing
// prune_some_following. If we assume 1 terainstruction costs 1 XDR,
// prune_some_following uses less than half an XDR per day.
const MAX_PRUNE_SOME_FOLLOWING_INSTRUCTIONS: u64 = 50_000_000;

pub(super) struct PruneFollowingTask {
    governance: &'static LocalKey<RefCell<Governance>>,
    begin: Bound<NeuronId>,
}

impl PruneFollowingTask {
    pub fn new(governance: &'static LocalKey<RefCell<Governance>>) -> Self {
        Self {
            governance,
            begin: Bound::Unbounded,
        }
    }
}

impl RecurringSyncTask for PruneFollowingTask {
    fn execute(self) -> (Duration, Self) {
        let new_begin = self.governance.with_borrow_mut(|governance| {
            let carry_on = || !is_message_over_threshold(MAX_PRUNE_SOME_FOLLOWING_INSTRUCTIONS);
            governance.prune_some_following(self.begin, carry_on)
        });

        (
            PRUNE_FOLLOWING_INTERVAL,
            Self {
                governance: self.governance,
                begin: new_begin,
            },
        )
    }

message VotingPowerEconomics {
  // If a neuron has not "refreshed" its voting power after this amount of time,
  // its deciding voting power starts decreasing linearly. See also
  // clear_following_after_seconds.
  //
  // For explanation of what "refresh" means in this context, see
  // https://dashboard.internetcomputer.org/proposal/132411
  //
  // Initially, set to 0.5 years. (The nominal length of a year is 365.25 days).
  optional uint64 start_reducing_voting_power_after_seconds = 1;

  // After a neuron has experienced voting power reduction for this amount of
  // time, a couple of things happen:
  //
  //     1. Deciding voting power reaches 0.
  //
  //     2. Its following on topics other than NeuronManagement are cleared.
  //
  // Initially, set to 1/12 years.
  optional uint64 clear_following_after_seconds = 2;

  // The minimum dissolve delay a neuron must have in order to be eligible to vote.
  //
  // Neurons with a dissolve delay lower than this threshold will not have
  // voting power, even if they are otherwise active.
  //
  // This value is an essential part of the staking mechanism, promoting
  // long-term alignment with the network's governance.
  optional uint64 neuron_minimum_dissolve_delay_to_vote_seconds = 3;
}

#[test]
fn test_prune_some_following_super_strict_voting_power_refresh() {
    // Step 1: Prepare the world. (This is exactly the same as the previous test.)

    let followees = hashmap! {
        Topic::Governance as i32 => Followees {
            followees: vec![NeuronId { id: 99 }],
        },
        Topic::NeuronManagement as i32 => Followees {
            followees: vec![NeuronId { id: 101 }],
        },
    };

    let mut fresh_neuron = simple_neuron_builder(1)
        .with_followees(followees.clone())
        .build();
    fresh_neuron.refresh_voting_power(CREATED_TIMESTAMP_SECONDS - 7 * ONE_MONTH_SECONDS + 1);

    // Similar to fresh_neuron, except voting power was refrshed a "long" time
    // ago.
    let mut stale_neuron = simple_neuron_builder(3)
        .with_followees(followees.clone())
        .build();
    stale_neuron.refresh_voting_power(CREATED_TIMESTAMP_SECONDS - 7 * ONE_MONTH_SECONDS - 1);

    let mut neuron_store = NeuronStore::new(btreemap! {
        fresh_neuron.id().id => fresh_neuron.clone(),
        stale_neuron.id().id => stale_neuron.clone(),
    });

    // Control the perception of time by neuron_store.
    #[derive(Debug, Clone)]
    struct DummyClock {}
    impl Clock for DummyClock {
        fn now(&self) -> u64 {
            CREATED_TIMESTAMP_SECONDS
        }

        fn set_time_warp(&mut self, _: TimeWarp) {
            unimplemented!();
        }
    }
    impl PracticalClock for DummyClock {}
    let clock = DummyClock {};
    neuron_store.clock = Box::new(clock);

    // Step 2: Call code under test. (This is where things start looking
    // different, compared to the previous test.)

    assert_eq!(
        prune_some_following(
            &VotingPowerEconomics {
                // These are much smaller than the normal values. As a result, all
                // neurons suddenly look stale. As a result, all following is
                // supposed to be cleared.
                start_reducing_voting_power_after_seconds: Some(42),
                clear_following_after_seconds: Some(58),
                neuron_minimum_dissolve_delay_to_vote_seconds: Some(42)
            },
            &mut neuron_store,
            Bound::Unbounded, // Start new cycle.
            || true,          // Do a full cycle.
        ),
        Bound::Unbounded,
    );

    // Step 3: Inspect results.

    // Assert that everyone's following got cleared, due to super strict
    // VotingPowerEconomics.
    for neuron_id in [fresh_neuron.id(), stale_neuron.id()] {
        neuron_store
            .with_neuron(&neuron_id, |observed_neuron| {

                        // NeuronManagement did not get touched.
                        Topic::NeuronManagement as i32 => Followees {
                            followees: vec![NeuronId { id: 101 }],
                        },
                    },
                );
            })
            .unwrap();
    }

    assert_eq!(neuron_store.len(), 2);
