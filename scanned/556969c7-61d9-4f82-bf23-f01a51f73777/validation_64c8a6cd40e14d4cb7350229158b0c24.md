### Title
Missing Global Rate Limit Enforcement in `notify_mint_cycles` Allows Bypass of Cycles Minting Cap — (File: `rs/nns/cmc/src/main.rs`)

### Summary
The Cycles Minting Canister (CMC) enforces a global cycles minting rate limit (`base_cycles_limit`) for `notify_top_up` and `notify_create_canister`, but the `notify_mint_cycles` endpoint — which mints cycles directly to the cycles ledger — completely omits this check. Any unprivileged user with ICP can call `notify_mint_cycles` to mint cycles without bound, bypassing the intended global rate cap.

### Finding Description
The CMC implements a global sliding-window rate limiter (`Limiter` in `rs/nns/cmc/src/limiter.rs`) that tracks total cycles minted across all callers within a rolling time window. This limiter is enforced via `ensure_balance` → `check_and_add_cycles` in the `notify_top_up` and `notify_create_canister` code paths.

However, `notify_mint_cycles` (introduced to mint cycles into the cycles ledger) calls `process_mint_cycles` → `do_mint_cycles` directly, with no call to `ensure_balance` or any limiter:

```rust
// rs/nns/cmc/src/main.rs:1958-1983
async fn process_mint_cycles(...) -> NotifyMintCyclesResult {
    let cycles = tokens_to_cycles(amount)?;
    match do_mint_cycles(to_account, cycles, deposit_memo).await {
        Ok(deposit_result) => { ... }
        Err(err) => { ... }
    }
}
```

Compare with `process_top_up`, which passes through `deposit_cycles` → `ensure_balance` → `limiter_to_use.check_and_add_cycles`:

```rust
// rs/nns/cmc/src/main.rs:1985-2012
async fn process_top_up(..., limiter_to_use: CyclesMintingLimiterSelector) -> ... {
    let cycles = tokens_to_cycles(amount)?;
    match deposit_cycles(canister_id, cycles, true, limiter_to_use).await { ... }
}
```

The `ensure_balance` function is the sole enforcement point for the global rate limit:

```rust
// rs/nns/cmc/src/main.rs:2306-2324
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    ...
    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;
    ...
}
```

`notify_mint_cycles` never reaches this path.

### Impact Explanation
The global cycles minting rate limit (`base_cycles_limit`, currently 150 petacycles/hour) is intended to prevent runaway ICP-to-cycles conversion that could destabilize the ICP/cycles exchange rate. By using `notify_mint_cycles` instead of `notify_top_up`, any number of users can each mint up to the full ICP balance they hold in cycles, collectively far exceeding the intended hourly cap. The `total_cycles_minted` counter is also not updated for `notify_mint_cycles` calls, so the accounting diverges from reality.

**Impact: Medium** — economic rate-limit bypass; does not break consensus or steal funds, but undermines the CMC's stability mechanism.

### Likelihood Explanation
`notify_mint_cycles` is a publicly documented endpoint in `rs/nns/cmc/cmc.did` and is callable by any principal with ICP. The only prerequisite is sending ICP to the CMC's subaccount with `MEMO_MINT_CYCLES`. No privileged access is required.

**Likelihood: Medium** — requires ICP capital and knowledge of the endpoint, but is straightforward to exploit.

### Recommendation
Apply the same `ensure_balance` / `check_and_add_cycles` rate-limit check inside `process_mint_cycles` (or `do_mint_cycles`) that is already applied in `process_top_up`. The `CyclesMintingLimiterSelector::BaseLimit` variant should be used, consistent with the treatment of non-SRC top-ups.

**Attack scenario:**
1. Attacker sends ICP to `CMC_PRINCIPAL / Subaccount::from(&caller)` with `MEMO_MINT_CYCLES`.
2. Attacker calls `notify_mint_cycles` with the resulting `block_index`.
3. `process_mint_cycles` → `do_mint_cycles` executes with no rate-limit check.
4. Steps 1–3 are repeated by multiple callers simultaneously; each call succeeds regardless of how many cycles have already been minted in the current window.
5. The `base_limiter` counter is never incremented for these calls, so `notify_top_up` callers also see an artificially low counter and may be permitted to mint more than intended.

    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }
