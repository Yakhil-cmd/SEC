### Title
Unbounded `inbound_connecting` JoinSet Allows Byzantine Subnet Peer to Exhaust Replica Resources via QUIC Connection Flood — (`rs/p2p/quic_transport/src/connection_manager.rs`)

---

### Summary

`handle_inbound_conn_attemp` unconditionally spawns a Tokio task into the unbounded `inbound_connecting` JoinSet for every accepted QUIC `Incoming`, with no per-peer cap, no global cap, and no deduplication. A single Byzantine subnet peer (holding a valid TLS certificate) can open N concurrent QUIC connections, causing N tasks to accumulate in the JoinSet, each alive for up to `CONNECT_TIMEOUT` (10 s), exhausting Tokio task heap and QUIC connection state.

---

### Finding Description

**Entrypoint — `handle_inbound_conn_attemp`:**

Every iteration of the `run` event loop calls `self.endpoint.accept()`. When an `Incoming` arrives it is passed directly to `handle_inbound_conn_attemp`: [1](#0-0) 

Inside that function, a timeout-wrapped future is spawned with no guard: [2](#0-1) 

The field itself is a plain, unbounded `JoinSet`: [3](#0-2) 

**Contrast with outbound path:** `outbound_connecting` uses a `JoinMap` keyed by `NodeId`, so at most one outbound task per peer exists at any time. `inbound_connecting` uses a plain `JoinSet` — no key, no deduplication, no cap. [4](#0-3) 

**TLS does not prevent the flood:** The server config is updated to only allow subnet members: [5](#0-4) 

This blocks non-subnet nodes from completing TLS. However, a Byzantine peer *is* a subnet member and holds a valid certificate. Its connections complete TLS successfully, so every spawned task runs to completion — it is not rejected early. The TLS guard only filters outsiders, not misbehaving insiders.

**Event loop processes one completion per iteration:** The `select!` loop handles exactly one branch per iteration. If the attacker opens connections faster than the loop drains `inbound_connecting.join_next()`, the JoinSet grows without bound. [6](#0-5) 

**`CONNECT_TIMEOUT` = 10 s** means each task lives for up to 10 seconds, giving the attacker a sustained 10-second window to accumulate tasks: [7](#0-6) 

---

### Impact Explanation

Each spawned task holds:
- A Tokio task stack allocation
- A `quinn::Incoming` / `quinn::Connection` object (QUIC connection state, crypto context, stream buffers)
- A reference to the endpoint's internal connection table entry

With N concurrent connections from one Byzantine peer, the replica accumulates N tasks × (task overhead + QUIC connection state). At scale this can exhaust Tokio task heap memory or OS-level resources (file descriptors, UDP socket receive-buffer entries), stalling or OOM-crashing the replica's event loop. A crashed replica loses its role in consensus, and if enough replicas are targeted simultaneously, subnet availability and consensus safety are at risk.

---

### Likelihood Explanation

The attacker is a Byzantine subnet peer — a realistic fault model explicitly within the IC's threat model (up to `f` Byzantine nodes in a subnet of `3f+1`). The peer holds a valid node TLS certificate issued by the NNS. Opening many concurrent QUIC connections from a single process is trivial (e.g., using the `quinn` client library in a loop). No privileged access, no key compromise, and no majority corruption is required. The attack is local-testable and reproducible.

---

### Recommendation

1. **Add a per-peer inbound connection cap** before spawning: track how many in-flight inbound tasks exist per source `NodeId` (extractable from the `Incoming`'s remote address mapped to topology) and reject or drop `Incoming` objects that exceed the limit (e.g., 2–3 concurrent attempts per peer).
2. **Add a global cap** on `inbound_connecting.len()` (e.g., `2 × subnet_size`) and call `incoming.refuse()` when the cap is reached.
3. **Consider migrating `inbound_connecting` to a `JoinMap`** keyed by source IP or peer identity once the identity is extractable pre-handshake, mirroring the deduplication already present in `outbound_connecting`.

---

### Proof of Concept

```rust
// Byzantine peer opens N concurrent QUIC connections to victim replica
let client_config = make_client_config_with_valid_node_cert();
let mut handles = Vec::new();
for _ in 0..N {
    let cfg = client_config.clone();
    handles.push(tokio::spawn(async move {
        let endpoint = quinn::Endpoint::client("0.0.0.0:0".parse().unwrap()).unwrap();
        let _ = endpoint.connect_with(cfg, victim_addr, "irrelevant")
                        .unwrap()
                        .await; // completes TLS, task stays in inbound_connecting
    }));
}
// Assert: victim's inbound_connecting.len() grows to N
// Assert: victim RSS grows proportionally to N
// Assert: victim event loop latency degrades as N increases
```
