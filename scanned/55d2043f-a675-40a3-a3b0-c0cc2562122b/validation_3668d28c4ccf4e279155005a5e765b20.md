### Title
Hardcoded `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` With No Setter Causes Permanent ckERC20 Withdrawal Unavailability and ckETH Loss - (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

---

### Summary

The ckETH minter applies a single compile-time constant gas limit (`65_000`) to every ckERC20 withdrawal transaction, with no runtime setter or per-token override. If any supported ERC-20 token requires more than 65,000 gas for a transfer, every withdrawal for that token will fail on Ethereum. The minter reimburses the burned ckERC20 tokens but explicitly does **not** reimburse the ckETH burned for gas fees, causing permanent user fund loss and complete withdrawal unavailability for the affected token.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/withdraw.rs`, line 44, the gas limit for all ckERC20 withdrawal transactions is a compile-time constant:

```rust
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);
``` [1](#0-0) 

The function `estimate_gas_limit()` unconditionally returns this constant for every ckERC20 withdrawal, regardless of which specific ERC-20 token is being withdrawn:

```rust
pub fn estimate_gas_limit(withdrawal_request: &WithdrawalRequest) -> GasAmount {
    match withdrawal_request {
        WithdrawalRequest::CkEth(_) => CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
        WithdrawalRequest::CkErc20(_) => CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
    }
}
``` [2](#0-1) 

This value is then passed directly into `create_transaction()` inside `create_transactions_batch()`: [3](#0-2) 

The minter's own documentation acknowledges this hardcoding: *"The `gas_limit` for ckERC20 withdrawals is currently fixed to `65_000` and should be sufficient for standard ERC-20 contracts."* [4](#0-3) 

There is no runtime setter, no per-token gas limit field in the minter state, and no upgrade-arg mechanism (unlike `minimum_withdrawal_amount`, which is configurable via upgrade args) to change this value without a full canister code upgrade and NNS governance proposal.

The `withdraw_erc20` flow burns ckETH for gas fees **before** the Ethereum transaction is submitted. If the Ethereum transaction fails due to out-of-gas, the minter reimburses the ckERC20 tokens but explicitly does not reimburse the ckETH gas fees: [5](#0-4) 

The `finalize_transactions_batch()` function processes `FinalizedTransaction` events for failed receipts, which triggers reimbursement of ckERC20 but not ckETH: [6](#0-5) 

---

### Impact Explanation

If any supported ERC-20 token requires more than 65,000 gas per transfer:

1. Every `withdraw_erc20` call for that token burns the user's ckETH for gas fees and their ckERC20 tokens.
2. The Ethereum transaction is submitted with `gas_limit = 65_000` and reverts with out-of-gas.
3. The minter detects the failed receipt and reimburses the ckERC20 tokens.
4. The ckETH burned for gas is **permanently lost** ("Overcharged transaction fees are not reimbursed").
5. The withdrawal queue for that token is stuck in a permanent failure loop — every retry burns more ckETH.

This constitutes both **permanent user fund loss** (ckETH) and **complete withdrawal unavailability** for the affected ckERC20 token, matching the impact class of the reference report.

---

### Likelihood Explanation

The minter currently supports tokens including wstETH (wraps stETH with rebasing logic), XAUt (Tether Gold with complex transfer logic), and USDT (with blacklist checks). Several of these have transfer gas costs that approach or can exceed 65,000 gas under certain conditions (e.g., first-time storage slot initialization, proxy delegate calls, or contract upgrades via proxy patterns). Additionally, ERC-20 contracts deployed behind upgradeable proxies (USDC, USDT) can have their transfer logic changed by their issuers, potentially increasing gas requirements above the hardcoded limit at any time without any action by the IC minter governance. The likelihood is **medium**: it does not require an attacker, only a routine contract upgrade or an edge-case execution path in a supported token.

---

### Recommendation

1. **Add a per-token configurable gas limit** stored in the minter state alongside each supported ckERC20 token's metadata, settable via the existing `add_ckerc20_token` governance path.
2. **At minimum**, expose `ckerc20_withdrawal_gas_limit` as a configurable upgrade argument (analogous to `minimum_withdrawal_amount`) so it can be updated via NNS proposal without a code change.
3. Update `estimate_gas_limit()` to look up the per-token gas limit from state rather than returning a global constant.

---

### Proof of Concept

**Entry path** (unprivileged ingress sender):

1. User calls `withdraw_erc20` on the ckETH minter for a supported token (e.g., wstETH) whose `transfer()` requires >65,000 gas.
2. Minter burns user's ckETH (gas fee) and ckERC20 tokens.
3. `create_transactions_batch()` calls `estimate_gas_limit()` → returns `65_000` for all ckERC20 tokens.
4. `create_transaction()` builds an EIP-1559 tx with `gas_limit = 65_000`.
5. Ethereum miners execute the tx; it runs out of gas and reverts.
6. `finalize_transactions_batch()` retrieves the failed receipt and emits `FinalizedTransaction`.
7. Minter reimburses ckERC20 tokens but not the ckETH gas fee.
8. User has permanently lost ckETH. Repeating the withdrawal produces the same result indefinitely until a canister upgrade is deployed. [7](#0-6) [8](#0-7)
