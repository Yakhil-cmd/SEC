### Title
`inspect_message` Guard Bypassed for `get_config` via Non-Replicated Query Path â€” (`rs/boundary_node/rate_limits/canister/canister.rs`)

### Summary

`get_config` is annotated `#[query]` (non-replicated query), but `inspect_message` on the Internet Computer is **only invoked for ingress update messages**. Non-replicated query calls never trigger `inspect_message`. The constant `REPLICATED_QUERY_METHOD = "get_config"` and the guard logic in `inspect_message` are therefore dead code for the actual call path. Any anonymous principal can invoke `get_config` as a query, receiving `rule_id` and `incident_id` for every confidential (non-disclosed) rule.

### Finding Description

**`inspect_message` guard (intended to block unauthorized access):** [1](#0-0) [2](#0-1) 

The guard checks `called_method == "get_config"` and traps for unauthorized callers. However, this hook is never invoked for non-replicated query calls â€” it is an IC protocol invariant that `inspect_message` fires only for ingress update messages.

**`get_config` is a non-replicated query:** [3](#0-2) 

The `#[query]` macro registers a non-replicated query endpoint. `inspect_message` is never called for it.

**Secondary defense (formatter) does NOT redact `rule_id` / `incident_id`:** [4](#0-3) 

`ConfigConfidentialityFormatter::format` only nulls out `rule_raw` and `description` for non-disclosed rules. `rule_id` and `incident_id` are always emitted.

This is confirmed by the existing unit test for the unauthorized path: [5](#0-4) 

`rule_id` and `incident_id` appear in the `RestrictedRead` response.

**`AccessLevelResolver` returns `RestrictedRead` for any anonymous principal:** [6](#0-5) 

No principal check blocks the call; it simply downgrades to `RestrictedRead` and proceeds.

### Impact Explanation

An anonymous caller can enumerate the `rule_id` (UUID) and `incident_id` (UUID) of every confidential rate-limit rule that has not yet been disclosed. While `rule_raw` and `description` are redacted, the structural metadata â€” which rules exist, how many there are, and which incident they belong to â€” is fully exposed. This violates the stated confidentiality model where unauthorized callers should be completely blocked from `get_config`.

### Likelihood Explanation

Trivially reachable. Any HTTP client can send a query call to the canister's `get_config` endpoint anonymously. No special tooling, keys, or privileges are required. The bypass is a direct consequence of using `#[query]` instead of `#[update]` for a method that was intended to be gated by `inspect_message`.

### Recommendation

Change `get_config` from `#[query]` to `#[update]` (replicated query semantics) so that `inspect_message` is actually invoked for it â€” consistent with the existing `REPLICATED_QUERY_METHOD` constant name. Alternatively, add an explicit authorization check at the top of `get_config` itself (before any data is read) that traps for callers without `FullAccess` or `FullRead`, making the function self-guarding regardless of call type.

### Proof of Concept

```rust
// In a canister integration test:
// 1. Install the rate-limits canister with an authorized principal.
// 2. Add a config with a confidential (non-disclosed) rule via `add_config`.
// 3. Call `get_config` as an anonymous principal using a *query* call (not update).
// 4. Assert that the response contains the rule_id and incident_id of the
//    non-disclosed rule, with rule_raw=None and description=None.

let response = canister.query_call("get_config", anonymous_principal, None).unwrap();
let config_response: GetConfigResponse = decode(response);
let rule = &config_response.config.rules[0]; // non-disclosed rule
assert!(rule.rule_id != "");       // leaks: UUID present
assert!(rule.incident_id != "");   // leaks: UUID present
assert_eq!(rule.rule_raw, None);   // redacted
assert_eq!(rule.description, None); // redacted
// inspect_message was never called; the guard was completely bypassed.
```
