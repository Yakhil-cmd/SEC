### Title
Hot Key Holder Becomes Controller of Child Neuron via `split_neuron` — (`rs/nns/governance/src/governance.rs`)

### Summary

The NNS `split_neuron` function authorizes the call using `is_controlled_by(caller)`, which accepts both the controller **and** hot key holders. However, the child neuron is then constructed with `*caller` as its controller — not `parent_neuron.controller()`. A hot key holder who is not the actual controller can therefore split a parent neuron and receive a child neuron they fully control.

### Finding Description

In `split_neuron`, the authorization guard at line 2175 is:

```rust
if !parent_neuron.is_controlled_by(caller) {
    return Err(GovernanceError::new(ErrorType::NotAuthorized));
}
``` [1](#0-0) 

`is_controlled_by` in NNS governance returns `true` for both the controller principal **and** registered hot key holders. This means a hot key holder B (where the actual controller is A) passes this check.

Immediately after, the child neuron is built with `*caller` as the controller principal:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    *caller,          // ← caller, not parent_neuron.controller()
    parent_neuron.dissolve_state_and_age(),
    created_timestamp_seconds,
)
``` [2](#0-1) 

The subaccount derivation for the memo path also correctly uses `parent_neuron.controller()`:

```rust
let to_subaccount = Subaccount(ledger::compute_neuron_split_subaccount_bytes(
    parent_neuron.controller(),
    memo,
));
``` [3](#0-2) 

…but the **controller field** of the new neuron is set to `*caller`, not `parent_neuron.controller()`. These are inconsistent.

### Impact Explanation

Hot key holder B calls `manage_neuron` → `Split` on a neuron owned by A. The authorization check passes (hot keys are permitted). The child neuron is created with B as its controller. B now has full controller rights over the child neuron and its ICP stake — they can disburse, merge, or dissolve it. A retains the reduced parent neuron but has permanently lost the split portion to B.

### Likelihood Explanation

Hot keys are a standard NNS feature used by staking services and neuron managers. Any hot key holder of any neuron can trigger this. The call path is a standard ingress `manage_neuron` message — no privileged access required beyond being a registered hot key.

### Recommendation

Replace `*caller` with `parent_neuron.controller()` in the `NeuronBuilder::new(...)` call:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    parent_neuron.controller(),  // ← fix: inherit parent's controller
    parent_neuron.dissolve_state_and_age(),
    created_timestamp_seconds,
)
``` [4](#0-3) 

Additionally, add a unit test asserting `child_neuron.controller == parent_neuron.controller` when the caller is a hot key, not the controller.

### Proof of Concept

1. Create neuron N with `controller = A`, add `hot_key = B`.
2. B sends `manage_neuron { id: N, command: Split { amount_e8s: X, memo: None } }`.
3. `is_controlled_by(B)` returns `true` (B is a hot key).
4. `NeuronBuilder::new(..., *caller, ...)` sets child neuron controller to B.
5. Child neuron with stake `X - fee` is now fully controlled by B.
6. Assert: `child_neuron.controller == B`, but `parent_neuron.controller == A`. Invariant broken. [1](#0-0) [2](#0-1)
