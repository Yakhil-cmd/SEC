### Title
`get_btc_address` Does Not Check Minter Mode, Misleading Users Into Irreversible BTC Sends When Deposits Are Disabled - (File: rs/bitcoin/ckbtc/minter/src/updates/get_btc_address.rs)

---

### Summary

The ckBTC minter's `get_btc_address` endpoint always returns a valid Bitcoin deposit address regardless of the minter's operational `Mode`. When the minter is in `ReadOnly`, `RestrictedTo`, or `DepositsRestrictedTo` mode, deposits are disabled and `update_balance` will reject the caller. However, `get_btc_address` never checks the mode and returns an address unconditionally. A user who calls `get_btc_address`, receives a valid address, and sends BTC to it will find their on-chain Bitcoin transaction irreversible and their funds locked until the minter's mode is restored.

---

### Finding Description

The ckBTC minter defines a `Mode` enum that controls which operations are permitted:

- `ReadOnly` — no state modifications allowed
- `RestrictedTo(Vec<Principal>)` — only listed principals can deposit or withdraw
- `DepositsRestrictedTo(Vec<Principal>)` — only listed principals can deposit
- `GeneralAvailability` — no restrictions [1](#0-0) 

The `update_balance` function, which actually mints ckBTC, correctly enforces the mode restriction at line 159:

```rust
state::read_state(|s| s.mode.is_deposit_available_for(&caller))
    .map_err(UpdateBalanceError::TemporarilyUnavailable)?;
``` [2](#0-1) 

The `is_deposit_available_for` method returns an error for `ReadOnly`, and for `RestrictedTo`/`DepositsRestrictedTo` when the caller is not in the allow-list: [3](#0-2) 

However, `get_btc_address` performs **no mode check whatsoever**. It only validates that the owner is non-anonymous, then unconditionally derives and returns a Bitcoin address:

```rust
pub async fn get_btc_address(args: GetBtcAddressArgs) -> String {
    let owner = args.owner.unwrap_or_else(ic_cdk::api::msg_caller);
    assert_ne!(owner, Principal::anonymous(), "the owner must be non-anonymous");
    init_ecdsa_public_key().await;
    read_state(|s| {
        account_to_p2wpkh_address_from_state(s, &Account { owner, subaccount: args.subaccount })
    })
}
``` [4](#0-3) 

The same pattern exists in the ckDOGE minter's `get_doge_address`, which also performs no mode check before returning a deposit address: [5](#0-4) 

The documented deposit flow is: call `get_btc_address` → send BTC → call `update_balance`. The address-fetching step is the user's signal that the deposit path is open. When the minter is in a restricted mode, `get_btc_address` silently returns a valid address, giving no indication that the subsequent `update_balance` call will be rejected. [6](#0-5) 

---

### Impact Explanation

A user who calls `get_btc_address` while the minter is in `ReadOnly` or `RestrictedTo` or `DepositsRestrictedTo` mode receives a valid Bitcoin address. If they send BTC to that address, the Bitcoin transaction is **irreversible on the Bitcoin network**. When they subsequently call `update_balance`, the minter rejects them with `TemporarilyUnavailable`. Their BTC is locked in the minter's deposit address until the governance-controlled mode is restored to `GeneralAvailability`. The duration of this lockup is unknown to the user and could be extended. This is a direct, user-facing fund-lockup impact on the chain-fusion ckBTC bridge.

---

### Likelihood Explanation

The minter is placed into restricted modes during upgrades, emergency pauses, or phased rollouts — all of which are real operational events on mainnet. Any user who queries `get_btc_address` during such a window and follows the documented deposit flow will be affected. The entry path requires only an unprivileged ingress call to `get_btc_address` followed by a standard Bitcoin send. No privileged access is needed by the attacker/victim.

---

### Recommendation

`get_btc_address` should check `s.mode.is_deposit_available_for(&caller)` before returning an address, and return an error (or trap) when deposits are disabled for the caller. Since `get_btc_address` is an `#[update]` call (not a query), it can return a `Result` type. Alternatively, the function should at minimum check `Mode::ReadOnly` and any applicable allow-list restriction for the resolved owner, mirroring the check already present in `update_balance`. The same fix applies to `get_doge_address` in the ckDOGE minter.

---

### Proof of Concept

1. Governance sets the ckBTC minter mode to `ReadOnly` (or `RestrictedTo([some_other_principal])`).
2. An unprivileged user calls `get_btc_address({owner: None, subaccount: None})` — the call succeeds and returns a valid Bitcoin P2WPKH address.
3. The user sends 0.01 BTC to that address on the Bitcoin network. The transaction is confirmed and irreversible.
4. The user calls `update_balance({owner: None, subaccount: None})`.
5. The minter returns `Err(UpdateBalanceError::TemporarilyUnavailable("the minter is in read-only mode"))`.
6. The user's BTC is locked in the minter's deposit address with no recourse until the mode is changed back by governance. [2](#0-1) [7](#0-6) [4](#0-3)

impl Mode {
    /// Returns Ok if the specified principal can convert BTC to ckBTC.
    pub fn is_deposit_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("access to the minter is temporarily restricted".to_string());
                }
                Ok(())
            }
            Self::DepositsRestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC deposits are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }
