### Title
`MintSnsTokens` 7-Day Rolling Rate Limit Disabled via `Decimal::MAX` Upper Bound — (`rs/sns/governance/src/proposal.rs`)

### Summary

The SNS governance canister enforces a 7-day rolling rate limit on `TransferSnsTreasuryFunds` proposals (capped at 25% of a medium treasury per window). The same limit is defined for `MintSnsTokens` but is intentionally disabled: `TokenProposalAction::recent_amount_total_upper_bound_tokens` for `MintSnsTokens` returns `Decimal::MAX` instead of calling `mint_sns_tokens_7_day_total_upper_bound_tokens`. There is also no execution-time re-check for `MintSnsTokens` (unlike `TransferSnsTreasuryFunds`). An SNS governance majority meeting the Critical threshold can therefore mint an unlimited quantity of SNS tokens within a single 7-day window, bypassing the rate-limiting protection that exists for treasury transfers.

### Finding Description

`TransferSnsTreasuryFunds` and `MintSnsTokens` both implement the `TokenProposalAction` trait, which is used to enforce a 7-day rolling cap on treasury outflows. For `TransferSnsTreasuryFunds`, `recent_amount_total_upper_bound_tokens` correctly delegates to `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens`, which returns a fraction of the treasury (e.g., 25% for a medium-sized treasury). [1](#0-0) 

For `MintSnsTokens`, the correct implementation is commented out with a `TODO(NNS1-2982): Uncomment` marker, and the active implementation unconditionally returns `Decimal::MAX`: [2](#0-1) 

Because `allowance_remainder_tokens = Decimal::MAX - spent_tokens ≈ Decimal::MAX`, the guard `if proposal_amount_tokens > allowance_remainder_tokens` in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` never fires for any `MintSnsTokens` proposal: [3](#0-2) 

Additionally, `perform_mint_sns_tokens` at execution time performs no rate-limit re-check at all — unlike `perform_transfer_sns_treasury_funds`, which calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`: [4](#0-3) 

Compare with the execution-time guard present for `TransferSnsTreasuryFunds`: [5](#0-4) 

The `mint_sns_tokens_7_day_total_upper_bound_tokens` function is fully implemented and ready to use: [6](#0-5) 

The `total_minting_amount_tokens` aggregation function is also implemented (and correctly aggregates across past executed proposals), but is marked `#[allow(unused)]` because the upper bound it feeds into is `Decimal::MAX`: [7](#0-6) 

### Impact Explanation

An SNS governance majority that meets the Critical threshold (67% of exercised voting power, 20% of total) can submit and execute `MintSnsTokens` proposals of arbitrary size, repeated each 7-day window, with no rate-limiting constraint. This allows unlimited dilution of SNS token holders. For example, on an SNS with 10 million tokens in its treasury, a single proposal could mint 10 million additional tokens (100% dilution) where the intended limit would be ~2.5 million (25% per 7 days). The `TransferSnsTreasuryFunds` path has this protection; `MintSnsTokens` does not.

### Likelihood Explanation

The entry path is a standard SNS governance ingress call (`manage_neuron` → `make_proposal` with `Action::MintSnsTokens`). Any SNS neuron holder can submit such a proposal. The Critical threshold (67% of exercised) is high, but in many SNS DAOs a small number of large neuron holders control the majority of voting power. A colluding or malicious majority can exploit this without any privileged access. The vulnerability is reachable on every deployed SNS canister running this code.

### Recommendation

1. Uncomment the correct `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (the `TODO(NNS1-2982): Uncomment` block at line 1025–1032).
2. Delete the `Decimal::MAX` placeholder (lines 1035–1041).
3. Add an execution-time re-check in `perform_mint_sns_tokens` analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, to guard against race conditions between concurrent proposals.
4. Remove the `#[allow(unused)]` annotation from `total_minting_amount_tokens` once the limit is active.

### Proof of Concept

The following sequence demonstrates the bypass:

1. Deploy an SNS with a treasury of 10,000 SNS tokens.
2. Submit a `MintSnsTokens` proposal for 5,000 SNS tokens (50% of supply — above the intended 25% cap).
3. Observe that `validate_and_render_mint_sns_tokens` calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err` with `MintSnsTokens` as the action.
4. `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`; `allowance_remainder_tokens = Decimal::MAX`.
5. `5000 > Decimal::MAX` is false; the proposal is accepted without error.
6. The proposal passes with Critical quorum and executes via `perform_mint_sns_tokens`, which mints 5,000 tokens with no execution-time rate-limit check.
7. Repeat immediately with another 5,000-token proposal in the same 7-day window — both succeed, for a total of 10,000 minted tokens (100% dilution), where the intended limit was 2,500 tokens.

The `TransferSnsTreasuryFunds` path correctly rejects step 2 for amounts above 25% of the treasury: [8](#0-7)

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }

    Ok(())

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
