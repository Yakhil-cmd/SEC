### Title
Timer-Based Migration Processing Bypasses `disable_api` Pause — (File: rs/migration_canister/src/processing.rs)

### Summary
The migration canister exposes a `disable_api` / `enable_api` mechanism (toggled by NNS governance via `PauseCanisterMigrations` / `UnpauseCanisterMigrations` proposals) that sets a `DISABLED` flag in stable memory. The `migrate_canister` update endpoint correctly checks this flag before accepting new requests. However, the timer-based processing functions that advance already-accepted requests through the migration state machine do **not** check `migrations_disabled()`, allowing in-flight migrations to continue executing even after an operator has paused the API.

### Finding Description
`migrate_canister` (the user-facing update endpoint) checks `migrations_disabled()` as its first guard: [1](#0-0) 

The `DISABLED` flag is set by `disable_api` / `enable_api`, which are restricted to controllers and the NNS governance canister: [2](#0-1) 

The flag itself is persisted in stable memory: [3](#0-2) 

However, the timer-based processing functions — `process_accepted`, `process_controllers_changed`, `process_stopped`, `process_renamed`, `process_routing_table`, `process_migrated_canister_deleted`, etc. — are imported and scheduled periodically: [4](#0-3) 

None of these processing functions import or call `migrations_disabled()`: [5](#0-4) 

A concrete example — `process_routing_table` — proceeds unconditionally to delete the migrated canister and advance state without any pause check: [6](#0-5) 

The DID interface exposes `disable_api` as a top-level control knob, and the NNS governance integration names the proposals `PauseCanisterMigrations` / `UnpauseCanisterMigrations`, indicating the intent is to halt all migration activity — not merely block new submissions: [7](#0-6) 

### Impact Explanation
When an operator calls `disable_api` (e.g., in response to a discovered bug or security incident), any migration request that was already accepted into the `REQUESTS` store continues to be processed by the timer. During processing, the migration canister:
- Takes **exclusive controller** of both the migrated and replaced canisters (removing the original owner's control)
- Modifies the **routing table** on the registry canister
- **Deletes** the migrated canister from its original subnet

If these operations should be halted due to an emergency, the `disable_api` mechanism fails to stop them. A user who submitted a migration request immediately before the pause can have their migration complete in full, including irreversible steps (canister deletion, routing table change), defeating the operator's intent.

### Likelihood Explanation
The `migrate_canister` endpoint is callable by any principal on the allowlist (which defaults to all principals when no allowlist is set). A user need only submit a valid migration request before the governance proposal to pause takes effect. Given that governance proposals take time to pass, there is a realistic window for this to occur. The processing timer fires continuously, so no additional attacker action is needed after the initial submission.

### Recommendation
Add a `migrations_disabled()` check at the start of each timer-based processing function (or in the central dispatch loop that calls them), and skip or abort processing of pending requests when the flag is set. This mirrors the guard already present in `migrate_canister` and ensures that `disable_api` fully halts all migration activity, not just new submissions.

### Proof of Concept
1. Unprivileged user calls `migrate_canister` with a valid pair of canister IDs → request is inserted into `REQUESTS` as `RequestState::Accepted`.
2. Operator (controller or NNS governance) calls `disable_api` → `DISABLED` flag set to `true`.
3. Timer fires → `process_accepted` (and subsequent processing functions) run without checking `migrations_disabled()`.
4. The migration advances through `ControllersChanged → StoppedAndReady → RenamedReplacedCanister → UpdatedRoutingTable → RoutingTableChangeAccepted → MigratedCanisterDeleted → RestoredControllers`, completing in full.
5. The operator's pause had no effect on the in-flight migration.
