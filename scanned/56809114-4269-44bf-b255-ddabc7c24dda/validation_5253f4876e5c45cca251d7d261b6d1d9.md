### Title
Unprivileged Attacker Can Force Arbitrary Principal Into SNS Swap Participation Without Consent — (`rs/sns/swap/canister/canister.rs`, `rs/sns/swap/src/swap.rs`)

---

### Summary

The `refresh_buyer_tokens` ingress endpoint accepts an arbitrary `buyer` principal in its request payload and uses it directly — with no check that `buyer == caller`. An attacker can pre-fund a victim's swap subaccount and then call this endpoint with `buyer = victim`, causing the swap canister to register the victim as a committed participant and, upon swap finalization, lock the victim into SNS neurons they never consented to.

---

### Finding Description

**Entrypoint — no caller/buyer identity check:**

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` update handler resolves the effective buyer principal as follows: [1](#0-0) 

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // ← no check that arg.buyer == caller
};
```

When `arg.buyer` is non-empty, the caller's identity is completely ignored. Any principal can be substituted.

**Core logic — balance lookup keyed on attacker-supplied principal:**

`refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` then queries the ICP ledger balance of `swap_canister[subaccount(buyer)]`: [2](#0-1) 

If that balance meets `min_participant_icp_e8s`, the victim's `BuyerState` is created/updated: [3](#0-2) 

**Ticket check is explicitly optional:**

The only other guard — the open-ticket check — is explicitly bypassed when no ticket exists for the buyer: [4](#0-3) 

The inline comment at line 1271 confirms: *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket."* An attacker calling on behalf of a victim will have no ticket for the victim, so this check is a no-op.

**Confirmation text is not a barrier:**

`validate_confirmation_text` is called, but the confirmation text is a public parameter set at SNS initialization time — readable by anyone from the swap's `init` state. It provides no access control.

**Finalization locks the victim in:**

Once registered, `sweep_icp` on commit transfers ICP from `swap_canister[subaccount(victim)]` to SNS governance and creates locked SNS neuron recipes for the victim: [5](#0-4) 

The victim cannot reverse this after the swap commits.

---

### Impact Explanation

- **Forced financial participation**: The victim is registered as a swap participant without any action or consent on their part.
- **Locked SNS neurons**: Upon swap commitment, the victim receives SNS neurons with dissolve delays (typically months to years) that they did not choose to acquire.
- **No victim recourse during OPEN state**: `error_refund_icp` is only available after the swap is COMMITTED or ABORTED. During the OPEN phase, the victim cannot reclaim the ICP from their subaccount.
- **Attacker cost**: The attacker sacrifices ICP (it either flows to SNS governance on commit, or is refunded to the victim on abort), making this a griefing/coercion attack rather than a theft.

---

### Likelihood Explanation

- Requires no privileged access — any principal can call `refresh_buyer_tokens` via ingress.
- Requires only the ability to transfer ICP to an arbitrary ICRC-1 account (the victim's subaccount of the swap canister), which is a standard ledger operation.
- The confirmation text is publicly readable from swap state.
- The attack is fully executable on mainnet against any open SNS swap.

---

### Recommendation

Add a caller-equals-buyer authorization check in the canister entry point. If the `buyer` field is specified and differs from the caller, reject the call unless the caller is the swap canister itself (for inter-canister notification flows):

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    let caller = caller_principal_id();
    if specified != caller {
        panic!("buyer must match caller");
    }
    specified
};
``` [1](#0-0)
