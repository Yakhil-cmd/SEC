### Title
Unbounded Linear Iteration Over All SNS Neurons in `compute_ballots_for_new_proposal` Causes Instruction-Limit Exhaustion, Permanently Blocking Proposal Creation - (File: rs/sns/governance/src/governance.rs)

---

### Summary

`compute_ballots_for_new_proposal` in SNS Governance iterates over every neuron in the canister's in-memory `BTreeMap` to build the electoral roll for each new proposal. Because the neuron set grows with every swap participant and every neuron split, a sufficiently large SNS can reach a state where this single-message loop exceeds the IC instruction limit, making it permanently impossible for any neuron holder to submit a proposal.

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, `compute_ballots_for_new_proposal` performs an unbounded linear scan over `self.proto.neurons`:

```rust
for (k, v) in self.proto.neurons.iter() {          // O(n) over ALL neurons
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power( … );
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { … });
}
``` [1](#0-0) 

This is called unconditionally from `make_proposal` on every proposal submission. `self.proto.neurons` is an in-memory `BTreeMap<String, Neuron>` whose size grows monotonically as users:

1. Participate in the SNS swap (each participant receives a basket of neurons, typically 5–7 per participant).
2. Split existing neurons (any neuron holder can call `split_neuron`).
3. Stake additional SNS tokens to create new neurons.

The SNS governance parameter `max_number_of_neurons` provides a nominal ceiling, but its default is on the order of 200,000 neurons — far above the threshold at which the loop exhausts the IC instruction budget.

Compare this directly to the UpDown `isParticipating` modifier:

```solidity
for (uint i = 0; i < UpPlayers.length; i++) {
    require(UpPlayers[i] != player, "Already participating");
}
```

Both patterns perform an O(n) scan over a collection that grows with user participation, on a hot path executed for every user action. In Solidity the resource is gas; in the IC the resource is the per-message instruction limit (≈ 5 billion instructions on application subnets).

The NNS Governance canister already recognised this problem and addressed it by introducing a `VotingStateMachine` with explicit soft/hard instruction limits and timer-based continuation:

```rust
noop_self_call_if_over_instructions(
    SOFT_VOTING_INSTRUCTIONS_LIMIT,
    Some(HARD_VOTING_INSTRUCTIONS_LIMIT),
).await
``` [2](#0-1) 

SNS Governance's `compute_ballots_for_new_proposal` has no equivalent guard.

---

### Impact Explanation

Once the neuron count crosses the instruction-budget threshold, every call to `make_proposal` traps with `CanisterInstructionLimitExceeded`. Because ballot computation is a prerequisite for proposal creation, **no new governance proposals can ever be submitted again**. This permanently freezes SNS governance: no parameter changes, no treasury disbursements, no upgrades, and no emergency actions can be enacted through the normal governance path. The SNS is effectively bricked from a governance perspective.

---

### Likelihood Explanation

A large SNS swap with 10,000 direct participants, each receiving a 5-neuron basket, produces 50,000 neurons immediately at swap close. Subsequent neuron splits (which any holder can perform permissionlessly) push the count higher. At roughly 100,000 instructions per neuron iteration (BTreeMap traversal + `voting_power` arithmetic + `electoral_roll.insert`), 50,000 neurons already approaches the 5-billion-instruction limit on application subnets. Popular SNS projects routinely attract tens of thousands of participants, making this threshold reachable in normal operation without any adversarial intent.

---

### Recommendation

Replace the full-scan ballot construction with one of the following:

1. **Pre-computed / cached electoral roll**: Maintain a running snapshot of eligible voters and their voting power, updated incrementally on neuron state changes, so `make_proposal` only reads a pre-built structure.
2. **Chunked construction with timer continuation**: Mirror the NNS Governance approach — build ballots in slices across multiple messages using `noop_self_call_if_over_instructions`, storing partial state between slices.
3. **Stable-structure index**: Store neurons in a `StableBTreeMap` indexed by eligibility, so only eligible neurons are iterated, and use pagination to spread the work.

---

### Proof of Concept

1. Launch an SNS with `max_number_of_neurons = 200_000` and `min_participant_icp_e8s` set low.
2. Run the SNS swap to completion with ≥ 10,000 direct participants (each receiving a 5-neuron basket → 50,000+ neurons).
3. Have any neuron holder call `make_proposal` with a valid `Motion` action.
4. Observe the call rejected with `CanisterInstructionLimitExceeded` because `compute_ballots_for_new_proposal` exhausts the ≈ 5 B instruction budget iterating over all neurons.
5. Confirm no proposal can ever be submitted again regardless of caller, proposal type, or neuron stake. [3](#0-2)
