### Title
Unbounded Global State Growth via Unvalidated `network_identifier.network` in Rosetta Metrics Middleware — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

---

### Summary

The `RosettaMetricsMiddleware` unconditionally inserts attacker-controlled strings from `network_identifier.network` into two unbounded process-global data structures: the `CANISTER_DISPLAY_NAMES` `Mutex<HashMap<String, String>>` and the Prometheus metric registry (via high-cardinality label values). No validation, no eviction, no rate limiting. An unprivileged HTTP client can exhaust heap memory and crash the Rosetta process.

---

### Finding Description

**Root cause 1 — `CANISTER_DISPLAY_NAMES` HashMap:**

`extract_canister_id` performs no validation on the extracted string: [1](#0-0) 

It returns any JSON string at `network_identifier.network` verbatim. The middleware then calls `RosettaMetrics::new` with this raw string: [2](#0-1) 

`RosettaMetrics::new` unconditionally inserts into the global map: [3](#0-2) 

The global map has no size cap, no TTL, and no eviction: [4](#0-3) 

**Root cause 2 — Prometheus label cardinality explosion:**

After each request, `inc_api_status_count` and `observe_request_duration` are called using the attacker-controlled `token_display_name` as a Prometheus label: [5](#0-4) 

Each unique label value creates new time series in the Prometheus default registry (11 series per unique label for the `HistogramVec` with 9 buckets). These are never evicted.

**Middleware is applied to all routes with no authentication or rate limiting:** [6](#0-5) 

The body size limit is `usize::MAX`, offering no protection: [7](#0-6) 

---

### Impact Explanation

Each HTTP POST to any Rosetta endpoint (e.g., `/block`) with a unique `network_identifier.network` string permanently adds one entry to `CANISTER_DISPLAY_NAMES` and 11+ Prometheus time series to the global registry. Neither structure is bounded or evicted. Sustained sending of unique strings exhausts heap memory and crashes the process. Because the attack requires only a low, sustained request rate (not volumetric), it persists across restarts as long as the attacker continues. An OOM crash halts all ledger block synchronization and asset bridge operations served by this Rosetta instance.

---

### Likelihood Explanation

The Rosetta HTTP API is publicly reachable with no authentication. The attack requires only a standard HTTP client sending POST requests with incrementing JSON strings. No privileged access, no key material, no protocol knowledge beyond the Rosetta API spec is needed. The attack is trivially scriptable.

---

### Recommendation

1. **Validate `network_identifier.network`** in `extract_canister_id` against the set of configured canister IDs before accepting it as a label value. Reject or ignore strings that do not match a known canister ID.
2. **Cap `CANISTER_DISPLAY_NAMES`** with a maximum size or use a bounded LRU cache.
3. **Never use attacker-controlled strings as Prometheus label values.** Map unknown IDs to a fixed sentinel label (e.g., `"unknown"`) rather than the raw input string.
4. **Add a request body size limit** in `extract_canister_id` (replace `usize::MAX` with a reasonable bound).
