### Title
SNS Treasury Manager Deposit/Withdraw API Lacks Slippage Protection, Enabling Price Manipulation Attacks on SNS Treasury Funds - (`rs/sns/treasury_manager/treasury_manager.did`, `rs/sns/governance/src/extensions.rs`)

---

### Summary

The `TreasuryManager` API and the SNS governance execution path for treasury deposit and withdrawal operations contain no slippage protection parameters. The `DepositRequest` and `WithdrawRequest` types carry no minimum-received-amount or price-limit fields, and the governance execution functions pass no such constraints to the treasury manager canister. Because governance proposal execution times are publicly known days in advance, an attacker can manipulate the price on the underlying DEX canister before execution, causing the SNS treasury to deposit or withdraw at a severely unfavorable price ratio.

---

### Finding Description

The `TreasuryManager` API, defined in `rs/sns/treasury_manager/treasury_manager.did`, specifies two financial operations:

```
type DepositRequest = record {
  allowances : vec Allowance;   // only specifies amounts to send, no minimum to receive
};

type WithdrawRequest = record {
  withdraw_accounts : opt vec record { principal; Account };  // only destination, no minimum amounts
};
```

Neither type includes a slippage parameter (minimum LP tokens to receive on deposit, minimum tokens to receive on withdrawal, or any price limit). [1](#0-0) 

The developers themselves acknowledge this gap in the same file:

```
// Known Security Risks:
// ====================
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
``` [2](#0-1) 

The governance execution function `execute_treasury_manager_deposit` in `rs/sns/governance/src/extensions.rs` first grants an ICRC-2 allowance to the treasury manager canister via `approve_treasury_manager`, then calls `deposit` with no price constraints whatsoever:

```rust
governance
    .approve_treasury_manager(
        extension_canister_id,
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
    )
    .await?;

let balances = governance
    .env
    .call_canister(extension_canister_id, "deposit", arg_blob)
    .await
    ...
``` [3](#0-2) 

Similarly, `execute_treasury_manager_withdraw` calls `withdraw` with no minimum-received-amount check: [4](#0-3) 

The proposal validation step (`validate_deposit_operation_impl`) only checks that the requested amount does not exceed 50% of the current treasury balance — it performs no price or slippage validation: [5](#0-4) 

The proposal rendering code in `rs/sns/governance/src/proposal.rs` even warns voters about this, but the framework provides no mechanism to specify or enforce slippage bounds:

```
## WARNING
Some Decentralized Exchanges lack slippage protection during deposits. Consequently, 
deposited asset ratios may deviate from those specified in the proposal. 
This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running 
or sandwich attacks.
``` [6](#0-5) 

---

### Impact Explanation

An attacker who manipulates the DEX canister price just before a governance proposal executes can cause the SNS treasury to:
- **On deposit**: receive far fewer LP tokens than the deposited value warrants, with the difference extracted by the attacker trading back after execution.
- **On withdraw**: receive far fewer tokens back from the DEX than the LP position is worth.

Because the `TreasuryManager` API provides no mechanism to specify a minimum received amount, and the governance framework enforces no such constraint, the treasury manager canister (e.g., a KongSwap adaptor) will execute at whatever price the DEX offers at execution time. The SNS treasury — holding community funds — can lose a significant fraction of the deposited or withdrawn value.

---

### Likelihood Explanation

The attack is realistic for the following reasons:

1. **Predictable execution time**: SNS governance proposals have a public voting period (default 4 days, configurable). The exact block/round at which a proposal will execute is observable by anyone monitoring the governance canister. [7](#0-6) 

2. **No privileged access required**: Any canister or user can interact with a DEX canister to trade and shift the pool price. No admin key or governance majority is needed to manipulate the DEX state.

3. **IC consensus does not prevent pre-execution price manipulation**: While the IC has no traditional mempool with MEV bots, an attacker can submit transactions to the DEX canister in the rounds immediately preceding proposal execution, moving the price to an unfavorable level before the governance call lands.

4. **The risk is self-documented**: The production DID file and proposal rendering code both acknowledge this attack surface exists, confirming it is a real, reachable condition. [2](#0-1) 

---

### Recommendation

1. **Add slippage parameters to the `TreasuryManager` API**: Extend `DepositRequest` with a `min_lp_tokens_received` field and `WithdrawRequest` with `min_tokens_received` fields per asset. Treasury manager implementations must enforce these bounds and revert if not met.

2. **Enforce slippage bounds at the governance layer**: The `ExecuteExtensionOperation` proposal argument should allow the proposer to specify acceptable price bounds. The governance execution functions `execute_treasury_manager_deposit` and `execute_treasury_manager_withdraw` should pass these bounds through to the treasury manager and treat a slippage violation as a proposal execution failure (not a silent success).

3. **Use a time-weighted average price (TWAP) oracle**: If the underlying DEX supports it, the treasury manager implementation should compare the execution price against a TWAP and reject execution if the deviation exceeds a governance-configured threshold.

---

### Proof of Concept

**Setup**: An SNS has a treasury manager extension backed by a DEX liquidity pool (e.g., KongSwap). The SNS governance submits a `TreasuryManagerDeposit` proposal to deposit 1,000 ICP and 500,000 SNS tokens into the pool.

**Attack**:
1. The proposal is adopted and will execute in ~10 minutes (after the voting period ends).
2. The attacker observes the pending proposal and the current pool price (e.g., 1 ICP = 500 SNS).
3. The attacker submits a large trade to the DEX canister, buying SNS tokens with ICP, shifting the price to 1 ICP = 100 SNS (a 5× manipulation).
4. The governance canister executes `execute_treasury_manager_deposit`:
   - `approve_treasury_manager` grants the treasury manager an ICRC-2 allowance for 1,000 ICP and 500,000 SNS. [8](#0-7) 
   - `call_canister(..., "deposit", arg_blob)` is called with no price limit. [9](#0-8) 
5. The treasury manager deposits at the manipulated price (1 ICP = 100 SNS), receiving LP tokens worth far less than the deposited value.
6. The attacker trades back (selling SNS for ICP), restoring the price and pocketing the spread — effectively extracting value from the SNS treasury.
7. The governance call returns `Ok(Balances {...})` with no error, because no minimum-received check exists anywhere in the call chain.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

    async fn approve_treasury_manager(
        &self,
        treasury_manager_canister_id: CanisterId,
        sns_amount_e8s: u64,
        icp_amount_e8s: u64,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: treasury_manager_canister_id.get().0,
            subaccount: None,
        };

        let expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS);
        let expiry_time_nsec = expiry_time_sec.saturating_mul(NANO_SECONDS_PER_SECOND);

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

        self.ledger
            .icrc2_approve(
                to,
                sns_amount_e8s,
                Some(expiry_time_nsec),
                self.transaction_fee_e8s_or_panic(),
                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making SNS Token treasury transfer: {e}"),
                )
            })?;

        self.nns_ledger
            .icrc2_approve(
                to,
                icp_amount_e8s,
                Some(expiry_time_nsec),
                icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(),
                self.icp_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making ICP Token treasury transfer: {e}"),
                )
            })?;

        Ok(())
    }

/// Execute a treasury manager withdraw operation
async fn execute_treasury_manager_withdraw(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedWithdrawOperationArg,
) -> Result<(), GovernanceError> {
    let arg_blob = construct_treasury_manager_withdraw_payload(arg.original).map_err(|err| {
        GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!("Failed to construct treasury manager withdraw payload: {err}"),
        )
    })?;

    let balances = governance
        .env
        .call_canister(extension_canister_id, "withdraw", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.withdraw failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!(
                        "Error decoding TreasuryManager.withdraw response: {err:?}"
                    ),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.withdraw failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.withdraw succeeded with response: {:?}",
        balances
    );

    Ok(())
}
