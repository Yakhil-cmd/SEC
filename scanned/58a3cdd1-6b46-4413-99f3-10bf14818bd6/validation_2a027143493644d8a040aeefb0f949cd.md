### Title
SNS Swap Participants Permanently Lose ICP When `sweep_icp` Succeeds But Subsequent Finalization Steps Permanently Fail — (`File: rs/sns/swap/src/swap.rs`)

---

### Summary

In the SNS Swap canister's `finalize_inner` function (COMMITTED path), `sweep_icp` is called **first** and irreversibly transfers all buyers' ICP from their per-buyer subaccounts to the SNS governance treasury. If any subsequent finalization step (`settle_neurons_fund_participation`, `create_sns_neuron_recipes`, `sweep_sns`, or `claim_swap_neurons`) permanently fails, there is **no rollback mechanism**. Buyers' ICP is permanently locked in SNS governance, and they never receive SNS neurons. There is no "emergency withdrawal" path for the COMMITTED lifecycle state.

---

### Finding Description

`finalize_inner` in `rs/sns/swap/src/swap.rs` executes a strict sequential pipeline for the COMMITTED path:

```
sweep_icp → settle_neurons_fund_participation → create_sns_neuron_recipes → sweep_sns → claim_swap_neurons → set_mode
``` [1](#0-0) 

`sweep_icp` transfers ICP from each buyer's subaccount (`principal_to_subaccount(buyer)`) to the SNS governance canister's account. This is an **irreversible on-chain ledger transfer**. Once it succeeds, the ICP is no longer in the buyer's subaccount. [2](#0-1) 

After `sweep_icp` succeeds, `set_sweep_icp_result` checks `is_successful_sweep()`. If the result contains any `failure`, `invalid`, or `global_failures` entries, an error message is set and finalization halts: [3](#0-2) 

The same halt-on-error pattern applies to every subsequent step: [4](#0-3) 

**Permanent failure scenarios after `sweep_icp` succeeds:**

1. **`create_sns_neuron_recipes` with `global_failures`**: If `neuron_basket_construction_parameters` is `None` (misconfiguration or state corruption), `create_sns_neuron_recipes` always returns `global_failures: 1`. Every subsequent call to `finalize` will halt at this step. ICP is already in SNS governance.

2. **`sweep_sns` with permanent `invalid` entries**: If a `SnsNeuronRecipe` has corrupted `neuron_attributes`, `investor`, or `sns` fields, `sweep_sns` returns `invalid > 0`. `is_successful_sweep()` returns false, finalization halts permanently. ICP is already in SNS governance; SNS tokens remain in the swap canister.

3. **`claim_swap_neurons` with `ClaimedStatus::Invalid`**: If SNS governance returns `ClaimedSwapNeuronStatus::Invalid` for a neuron, the recipe's `claimed_status` is set to `Invalid`. On every subsequent `finalize` call, `to_neuron_recipe` returns `ConversionError::Invalid` for that recipe, `sweep_result.invalid` is incremented, and finalization always halts. ICP is in SNS governance; SNS tokens are in neuron staking subaccounts but neurons are never claimed. [5](#0-4) 

**The `error_refund_icp` escape hatch does not help**: This function only refunds ICP from a buyer's own subaccount of the swap canister. Once `sweep_icp` has succeeded for a buyer (`transfer_success_timestamp_seconds != 0`), their subaccount is empty. Furthermore, for buyers whose `sweep_icp` transfer has started but not yet succeeded (`transfer_success_timestamp_seconds == 0`), `error_refund_icp` explicitly blocks the refund with "ICP in escrow": [6](#0-5) 

There is no mechanism to retrieve ICP from the SNS governance treasury back to buyers.

---

### Impact Explanation

Buyers who participated in a committed SNS swap permanently lose their ICP contribution and receive no SNS neurons. The ICP is locked in the SNS governance treasury with no on-chain recovery path. The lifecycle remains `COMMITTED` indefinitely; the swap canister cannot be deleted until all tokens are disbursed, but disbursement is permanently blocked.

---

### Likelihood Explanation

The likelihood is **low but non-zero**. The permanent failure conditions include:

- State corruption in `neuron_basket_construction_parameters` (e.g., upgrade bug zeroing optional fields)
- SNS governance returning `ClaimedSwapNeuronStatus::Invalid` for valid neurons (SNS governance bug)
- Corrupted `SnsNeuronRecipe` fields after a canister upgrade

The `finalize_swap` endpoint is publicly callable by any principal, meaning any user can trigger the finalization pipeline. The vulnerability manifests when the system reaches a state where a post-`sweep_icp` step permanently fails — a condition that does not require attacker action but can arise from bugs in the SNS canister suite. [7](#0-6) 

---

### Recommendation

1. **Reorder finalization**: Validate all preconditions (neuron recipe creation, NF settlement readiness) **before** calling `sweep_icp`. Only transfer ICP after confirming all downstream steps can proceed.

2. **Add a recovery path**: Introduce a governance-controlled "abort finalization" mechanism that, if triggered after a permanent failure, allows the swap to transition to a special recovery state where ICP can be returned to buyers from the SNS governance treasury.

3. **Distinguish transient vs. permanent failures**: The `SweepResult.invalid` and `global_failures` fields indicate permanent failures. These should be treated differently from `failure` (transient). Permanent failures should trigger an alert and a defined recovery procedure rather than silently halting finalization.

---

### Proof of Concept

The existing test `test_finalization_halts_when_create_sns_neuron_recipes_fails` in `rs/sns/swap/tests/swap.rs` demonstrates the halting behavior: [8](#0-7) 

In this test, `sweep_icp` succeeds (ICP transferred to SNS governance), but `create_sns_neuron_recipes` fails permanently. The test confirms finalization halts and all subsequent steps are skipped. Extending this test to verify that `error_refund_icp` also fails for the affected buyers (because their subaccounts are now empty) would confirm the complete loss scenario.

/// Tests that if create sns neuron recipes fails finalize will halt finalization
#[tokio::test]
async fn test_finalization_halts_when_create_sns_neuron_recipes_fails() {
    // Step 1: Prepare the world

    // Setup the necessary buyers for the test.
    let mut swap = Swap {
        lifecycle: Committed as i32,
        init: Some(init()),
        params: Some(Params {
            // This should cause neuron recipe creation to fail.
            neuron_basket_construction_parameters: None,
            ..params()
        }),
        buyers: buyers(),
        ..Default::default()
    };

    let mut clients = CanisterClients {
        icp_ledger: SpyLedger::new(vec![LedgerReply::TransferFunds(Ok(1000))]),
        nns_governance: SpyNnsGovernanceClient::new(vec![
            NnsGovernanceClientReply::SettleNeuronsFundParticipation(
                SettleNeuronsFundParticipationResponse {
                    result: Some(settle_neurons_fund_participation_response::Result::Ok(
                        settle_neurons_fund_participation_response::Ok {
                            neurons_fund_neuron_portions: vec![],
                        },
                    )),
                },
            ),
        ]),
        ..spy_clients()
    };

    // Step 2: Call finalize
    let result = swap.finalize(now_fn, &mut clients).await;

    // Assert that previous subtasks execute correctly
    assert!(result.sweep_icp_result.is_some());
    assert!(result.settle_neurons_fund_participation_result.is_some());

    assert_eq!(
        result.create_sns_neuron_recipes_result,
        Some(SweepResult {
            success: 0,
            skipped: 0,
            failure: 0,
            invalid: 0,
            global_failures: 1,
        })
    );

    assert_eq!(
        result.error_message,
        Some(String::from(
            "Creating SnsNeuronRecipes did not complete fully, some data was invalid or failed. Halting swap finalization"
        ))
    );

    // Assert all other fields are set to None because finalization was halted
    assert!(result.set_dapp_controllers_call_result.is_none());
    assert!(result.sweep_sns_result.is_none());
    assert!(result.set_mode_call_result.is_none());
    assert!(result.claim_neuron_result.is_none());
}
