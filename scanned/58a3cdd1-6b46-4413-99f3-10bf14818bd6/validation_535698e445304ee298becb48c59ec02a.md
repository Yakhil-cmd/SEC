### Title
Untested Custom Arithmetic in Maturity Modulation Library Used for ICP Minting - (File: rs/nervous_system/governance/src/maturity_modulation/mod.rs)

### Summary
The `rs/nervous_system/governance/src/maturity_modulation/mod.rs` module implements two custom arithmetic functions — `apply_maturity_modulation` and the private helper `saturating_add_or_subtract_u128_i32` — that directly govern how much ICP is minted when NNS and SNS neuron holders disburse maturity. Neither function has any unit tests in its own module. This is the direct IC analog of the reported "untested custom SafeMath library" vulnerability class.

### Finding Description
The file `rs/nervous_system/governance/src/maturity_modulation/mod.rs` contains the entire maturity-modulation arithmetic library in 39 lines with no `#[cfg(test)]` block and no companion test file (confirmed by glob search returning only the single source file). [1](#0-0) 

`apply_maturity_modulation` computes the ICP amount to mint from a neuron's maturity e8s and a signed basis-point modulation value: [2](#0-1) 

It delegates mixed-sign arithmetic to the private helper `saturating_add_or_subtract_u128_i32`: [3](#0-2) 

The helper contains a latent edge-case: when `delta == i32::MIN`, `delta.saturating_abs()` returns `i32::MAX` (2 147 483 647) rather than the true absolute value (2 147 483 648), producing an off-by-one subtraction. While the current caller bounds `maturity_modulation_basis_points` to `[-500, 500]`, this invariant is enforced only at the call site in the CMC canister — not inside the library itself — and is nowhere tested.

The absence of unit tests means:
1. The edge-case above is never exercised.
2. Any future refactor of the arithmetic (e.g., changing the basis-point scale, adjusting bounds, or porting to a different integer width) will not be caught by the test suite before deployment.

Higher-level integration tests in NNS governance, SNS governance, and the CMC canister do exercise paths that call `apply_maturity_modulation`, but they test end-to-end disbursement flows with a small set of fixed modulation values (e.g., `200`, `0`, `-300`) and do not constitute unit coverage of the arithmetic library itself. [4](#0-3) 

### Impact Explanation
`apply_maturity_modulation` is the sole function that converts a neuron's maturity e8s into the ICP amount minted to the user's account during a `DisburseMaturity` call. An arithmetic error — whether latent today or introduced by a future undetected regression — directly controls how many ICP tokens are minted. Incorrect results produce either under-minting (user loss) or over-minting (ledger conservation violation / inflation). Both outcomes are irreversible on-chain.

### Likelihood Explanation
The current code is arithmetically correct for all inputs within the enforced `[-500, 500]` permyriad range. However, the lack of unit tests creates a persistent regression risk: any developer modifying the arithmetic, changing the scale, or relaxing the bounds has no automated safety net. The IC governance codebase is actively developed (Mission 70 voting-rewards changes, maturity-modulation algorithm overhaul visible in the same codebase), making future modifications likely. Likelihood of a silent regression reaching production is **medium**.

### Recommendation
Add a dedicated `#[cfg(test)]` module (or a companion `tests.rs`) inside `rs/nervous_system/governance/src/maturity_modulation/` that covers:

1. **Identity**: `apply_maturity_modulation(x, 0) == x` for representative values of `x`.
2. **Positive modulation**: `apply_maturity_modulation(10_000 * E8, 500)` equals `10_050 * E8`.
3. **Negative modulation**: `apply_maturity_modulation(10_000 * E8, -500)` equals `9_950 * E8`.
4. **Boundary of `saturating_add_or_subtract_u128_i32`**: `delta = i32::MIN`, `delta = i32::MAX`, `initial_value = 0`, `initial_value = u128::MAX`.
5. **Overflow guard**: `apply_maturity_modulation(u64::MAX, 500)` returns `Err(...)` rather than silently truncating.

use std::cmp::Ordering;

/// The maturity modulation range in basis points.
pub const MIN_MATURITY_MODULATION_PERMYRIAD: i32 = -500;
pub const MAX_MATURITY_MODULATION_PERMYRIAD: i32 = 500;

pub const BASIS_POINTS_PER_UNITY: u128 = 10_000;

/// Modulate amount_e8s. That is, multiply by 1 + X where
/// X = maturity_modulation_basis_points / 10_000.
pub fn apply_maturity_modulation(
    amount_maturity_e8s: u64,
    maturity_modulation_basis_points: i32,
) -> Result<u64, String> {
    let amount_e8s = u128::from(amount_maturity_e8s);

    let adjusted_maturity_modulation_basis_points = saturating_add_or_subtract_u128_i32(
        BASIS_POINTS_PER_UNITY,
        maturity_modulation_basis_points,
    );

    let modulated_amount_e8s: u128 = amount_e8s
        .checked_mul(adjusted_maturity_modulation_basis_points)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?
        .checked_div(BASIS_POINTS_PER_UNITY)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?;

    u64::try_from(modulated_amount_e8s).map_err(|err| err.to_string())
}

/// Adds or subtracts a i32 from a u128, resulting in a u128. Safety check allows for `as`
/// conversions inline.
fn saturating_add_or_subtract_u128_i32(initial_value: u128, delta: i32) -> u128 {
    match delta.cmp(&0) {
        Ordering::Less => initial_value.saturating_sub(delta.saturating_abs() as u128),
        Ordering::Equal => initial_value,
        Ordering::Greater => initial_value.saturating_add(delta as u128),
    }
}
