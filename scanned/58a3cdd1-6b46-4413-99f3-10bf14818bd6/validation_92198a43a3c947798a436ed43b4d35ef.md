### Title
Unbounded `ingress_expiries` Loop in ICP Rosetta `construction_payloads` Allows OOM DoS ŌĆö (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The `construction_payloads` handler in the ICP Rosetta node accepts attacker-controlled `ingress_start` and `ingress_end` u64 nanosecond timestamps with no server-side cap on their span. The while-loop at lines 99ŌĆō107 iterates `(ingress_end ŌłÆ ingress_start) / interval` times, pushing one `u64` per iteration into `ingress_expiries`, then `add_payloads` pushes two heap-allocated `SigningPayload` structs per entry. With `ingress_start=0` and `ingress_end=u64::MAX`, this produces ~153 million iterations and gigabytes of heap allocation in a single synchronous request, crashing the Rosetta process via OOM.

---

### Finding Description

**Vulnerable loop** ŌĆö `construction_payloads.rs` lines 99ŌĆō107:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    let ingress_expiry = (now
        + ic_limits::MAX_INGRESS_TTL.saturating_sub(ic_limits::PERMITTED_DRIFT))
    .as_nanos_since_unix_epoch();
    ingress_expiries.push(ingress_expiry);
    now += interval;
}
``` [1](#0-0) 

**Interval value** ŌĆö `MAX_INGRESS_TTL` (5 min) ŌłÆ `PERMITTED_DRIFT` (60 s) ŌłÆ 120 s = **120 seconds = 120,000,000,000 ns**: [2](#0-1) [3](#0-2) 

**Iteration count with max span**: `u64::MAX / 120_000_000_000 Ōēł 153,722,867` iterations (~153 million). Each iteration pushes a `u64` (8 bytes) ŌåÆ ~1.2 GB for `ingress_expiries` alone.

**Amplification via `add_payloads`** ŌĆö for each entry in `ingress_expiries`, two heap-allocated `SigningPayload` structs (each containing a hex-encoded 32-byte hash string, an `AccountIdentifier`, and a `SignatureType`) are pushed into `payloads`: [4](#0-3) 

Total heap allocation per request: conservatively **tens of gigabytes**.

**No validation of the span** ŌĆö `ingress_start` and `ingress_end` are deserialized directly from the JSON metadata with no cap: [5](#0-4) 

**Contrast with ICRC1 Rosetta** ŌĆö the ICRC1 variant of the same function explicitly rejects oversized spans before entering the loop: [6](#0-5) 

The ICP handler has no equivalent guard.

**No per-request timeout in the Rosetta server** ŌĆö the actix-web server is configured with only a 4 MB JSON body limit and no request-level timeout: [7](#0-6) 

The attack payload is a tiny JSON object (two u64 numbers), well under 4 MB, so the body limit provides no protection.

---

### Impact Explanation

The Rosetta process is a long-running service that ICP ecosystem tooling (exchanges, wallets, staking UIs) depends on for ledger queries and transaction construction. A single malformed POST request causes the process to allocate gigabytes of heap memory synchronously on the request-handling thread, triggering the OS OOM killer or an allocator panic. This denies service to all legitimate users of the Rosetta API until the process is manually restarted.

---

### Likelihood Explanation

The endpoint is publicly reachable (no authentication required by the Rosetta spec). The attack requires a single HTTP POST with a two-field JSON body. No special privileges, keys, or network position are needed. The exploit is trivially reproducible locally.

---

### Recommendation

Add a span cap before the loop, mirroring the ICRC1 implementation:

```rust
let max_span = interval * MAX_INGRESS_WINDOWS; // e.g., MAX_INGRESS_WINDOWS = 10
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch()) > max_span.as_nanos() as u64 {
    return Err(ApiError::invalid_request(
        "ingress_end ŌłÆ ingress_start exceeds the maximum allowed span",
    ));
}
```

Or equivalently, cap `ingress_expiries.len()` to a small constant (e.g., Ōēż 10) and return an error if the requested span would exceed it.
