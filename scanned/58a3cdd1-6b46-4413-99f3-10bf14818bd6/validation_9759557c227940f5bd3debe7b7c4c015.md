### Title
Unbounded ingress-expiry loop in ICP Rosetta `construction_payloads` allows unauthenticated OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

### Summary
The ICP Rosetta `/construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` values from the JSON metadata with no range validation. Setting `ingress_start=0` and `ingress_end=u64::MAX` causes the while loop at lines 99–107 to execute ~153 billion iterations, exhausting process memory and crashing the Rosetta server.

### Finding Description
`construction_metadata` returns `ConstructionPayloadsRequestMetadata::default()` (all-`None` fields) as the metadata response. [1](#0-0) 

A client then calls `/construction/payloads` and may supply arbitrary `ingress_start`/`ingress_end` values in the metadata `ObjectMap`. These are deserialized as plain `Option<u64>` with no validation: [2](#0-1) 

The handler computes `interval = MAX_INGRESS_TTL(300s) − PERMITTED_DRIFT(60s) − 120s = 120s = 120,000,000,000 ns`, then enters an unbounded loop: [3](#0-2) 

There is no cap, no guard, and no check that `ingress_end − ingress_start` is within a reasonable bound. With `ingress_start=0` and `ingress_end=u64::MAX`, the loop runs ≈153 billion iterations, each `push`ing a `u64` onto `ingress_expiries`, allocating ~1.2 TB of heap before OOM kills the process.

By contrast, the ICRC1 Rosetta implementation explicitly validates the range before entering its equivalent loop: [4](#0-3) 

The ICP Rosetta path has no such guard.

### Impact Explanation
Any unauthenticated HTTP client can crash the ICP Rosetta process with a single request. Rosetta is the standard exchange-integration gateway for ICP; its crash denies service to all users of that instance (exchange deposits/withdrawals, wallet integrations, etc.) until the process is restarted. The attack is repeatable — the process can be crashed again immediately after restart.

### Likelihood Explanation
The endpoint is public, requires no authentication, and the exploit is a single well-formed JSON POST. The attacker needs no special knowledge beyond the Rosetta API spec. The `ingress_start` and `ingress_end` fields are documented in the metadata struct.

### Recommendation
Add an explicit upper bound on the computed range before entering the loop, mirroring the ICRC1 guard pattern:
```rust
let max_expiries = 1000usize; // or derive from a reasonable window
if (ingress_end - ingress_start) / interval_nanos > max_expiries as u64 {
    return Err(ApiError::invalid_request("ingress window too large"));
}
```
Alternatively, cap `ingress_end` to `ingress_start + N * interval` for some small constant `N`.
