### Title
Cycles Minting Rate Limiter Permanently Consumed by Failed `notify_top_up` Calls to Non-Existent Canisters - (File: rs/nns/cmc/src/main.rs, rs/nns/cmc/src/limiter.rs)

### Summary

The Cycles Minting Canister (CMC) maintains a time-windowed rate limiter (`Limiter`) that tracks how many cycles have been minted in the last hour. When `notify_top_up` is called for a non-existent canister, the limiter is permanently incremented even though the cycles are never delivered and the ICP is refunded. Because the `Limiter` has no decrement operation, an unprivileged attacker can exhaust the global minting rate limit at very low cost (only ICP transfer fees), blocking all legitimate users from minting cycles for up to one hour.

### Finding Description

The `Limiter` struct in `rs/nns/cmc/src/limiter.rs` tracks cumulative cycles minted in a sliding time window. Its only public mutation method is `check_and_add_cycles`, which increments `total_count` and never decrements it. [1](#0-0) 

The call chain for `notify_top_up` is:

```
notify_top_up → process_top_up → deposit_cycles → ensure_balance
                                                      ↓
                                          check_and_add_cycles  ← limiter incremented
                                          ic0_mint_cycles128    ← cycles minted into CMC balance
                                                      ↓
                                    management_canister::deposit_cycles  ← may FAIL
                                                      ↓ (on failure)
                                    refund_icp        ← ICP returned to user
                                    (limiter NOT decremented)
```

In `ensure_balance`, the limiter is incremented and cycles are physically minted into the CMC's own balance **before** the inter-canister call to the management canister's `deposit_cycles` method: [2](#0-1) 

In `deposit_cycles`, `ensure_balance` is called first; if the subsequent management canister call fails (e.g., target canister does not exist), the error propagates but the limiter is already incremented: [3](#0-2) 

In `process_top_up`, a failed `deposit_cycles` triggers `refund_icp`, returning the ICP to the caller. The limiter is never decremented: [4](#0-3) 

The `Limiter` struct has no `subtract` or `decrease` method — only `check_and_add_cycles` and `get_count`: [5](#0-4) 

The global base limit is 150 petacycles per hour: [6](#0-5) 

### Impact Explanation

An attacker can exhaust the entire 150P cycles/hour minting rate limit, causing all subsequent `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` calls from legitimate users to be rejected with `"More than N cycles have been minted in the last 3600 seconds, please try again later"` for up to one full hour. This is a denial-of-service against the CMC's core function of converting ICP to cycles. [7](#0-6) 

### Likelihood Explanation

The attack is cheap. Each iteration costs only two ICP ledger transfer fees (≈0.0002 ICP): one to send ICP to the CMC subaccount and one for the refund. To exhaust 150P cycles at 207T cycles per iteration requires ≈725 iterations, costing ≈0.145 ICP total. The attacker recovers their principal ICP each time. The existing integration test explicitly models this attack scenario and confirms the limiter is consumed by failed top-ups: [8](#0-7) 

The test title claims the rate limit is "not" consumed by invalid top-ups, but the test body shows 720 iterations consuming 149.04P cycles of the 150P limit — confirming the limiter IS permanently incremented by failed deposits. The test only passes because the good account's 400T request fits within the remaining ~960T headroom. With a slightly larger attack (≈725 iterations instead of 720), the full limit is exhausted.

Any unprivileged ingress sender with a small ICP balance can trigger this. No privileged access, flash loan, or threshold attack is required.

### Recommendation

Add a `subtract` method to `Limiter` that decrements `total_count` (clamped to zero) and call it when `deposit_cycles` to the management canister fails, before the ICP refund is issued. This mirrors the recommended fix in the external report (an `increase` method on the Alchemix `Limiters` library called during `burn`):

```rust
// In rs/nns/cmc/src/limiter.rs
pub fn subtract(&mut self, cycles: Cycles) {
    if self.total_count >= cycles {
        self.total_count -= cycles;
        // Also subtract from the most recent window
        if let Some(back) = self.time_windows.back_mut() {
            back.count = back.count.saturating_sub(cycles);
        }
    } else {
        self.total_count = Cycles::zero();
    }
}
```

Call this in `deposit_cycles` when the management canister call fails, passing back the `cycles_to_mint` that was charged to the limiter in `ensure_balance`. Alternatively, move the limiter increment to occur only after a successful deposit.

### Proof of Concept

1. Attacker identifies a non-existent canister ID (e.g., `CanisterId::from_u64(123_456_789)`).
2. Attacker transfers `N` ICP to the CMC subaccount corresponding to that canister ID on the ICP ledger.
3. Attacker calls `notify_top_up { block_index: <block>, canister_id: <non_existent> }`.
4. CMC calls `ensure_balance` → limiter incremented by `cycles_equivalent(N)` → cycles minted into CMC balance.
5. CMC calls management canister `deposit_cycles` for the non-existent canister → call fails.
6. CMC calls `refund_icp` → attacker receives `N - fees` ICP back.
7. Limiter remains incremented. Attacker repeats from step 2 with the refunded ICP.
8. After ≈725 iterations (≈0.145 ICP in fees), the 150P/hour limit is exhausted.
9. All legitimate `notify_top_up` / `notify_create_canister` / `notify_mint_cycles` calls fail with the rate-limit error for up to one hour. [9](#0-8) [3](#0-2) [10](#0-9)

#[derive(Clone, Eq, PartialEq, Debug, CandidType, Deserialize, Serialize)]
pub struct Limiter {
    time_windows: VecDeque<TimeWindowCount>,
    total_count: Cycles,
    resolution: Duration,
    max_age: Duration,
}

impl Limiter {
    pub fn new(resolution: Duration, max_age: Duration) -> Self {
        Self {
            time_windows: VecDeque::new(),
            total_count: Cycles::zero(),
            resolution,
            max_age,
        }
    }

    /// This function clears old events, checks against the provided limit, and returns an
    /// error if the cycles cannot be added due to exceeding the limit.
    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }

    /// Record a cycles minting event at time `now`. It's expected
    /// that `now` is monotonically non-decreasing.
    fn add(&mut self, now: SystemTime, cycles: Cycles) {
        self.purge_old(now);

        let window = self.time_to_window(now);

        if self
            .time_windows
            .back()
            .filter(|w| w.window >= window)
            .is_none()
        {
            self.time_windows.push_back(TimeWindowCount {
                window,
                count: Cycles::zero(),
            });
        };

        self.time_windows.back_mut().unwrap().count += cycles;
        self.total_count += cycles;
    }

    /// Forget about all cycles minting events older than `now -
    /// self.max_age`.
    fn purge_old(&mut self, now: SystemTime) {
        while let Some(oldest) = self.time_windows.front() {
            if self.window_to_time(oldest.window + 1) + self.max_age <= now {
                self.total_count -= oldest.count;
                self.time_windows.pop_front();
            } else {
                break;
            }
        }
    }

    fn time_to_window(&self, time: SystemTime) -> TimeWindow {
        (time.duration_since(UNIX_EPOCH).unwrap().as_secs() / self.resolution.as_secs())
            .try_into()
            .unwrap()
    }

    fn window_to_time(&self, window: TimeWindow) -> SystemTime {
        UNIX_EPOCH + self.resolution * window
    }

    /// Return the total number of cycles minted in the last
    /// `self.max_age` period.
    pub fn get_count(&self) -> Cycles {
        self.total_count
    }

    fn get_max_age(&self) -> Duration {
        self.max_age
    }
}

async fn process_top_up(
    canister_id: CanisterId,
    from: AccountIdentifier,
    amount: Tokens,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<Cycles, NotifyError> {
    let cycles = tokens_to_cycles(amount)?;

    let sub = Subaccount::from(&canister_id);

    print(format!(
        "Topping up canister {canister_id} by {cycles} cycles."
    ));

    match deposit_cycles(canister_id, cycles, true, limiter_to_use).await {
        Ok(()) => {
            burn_and_log(sub, amount).await;
            Ok(cycles)
        }
        Err(err) => {
            let refund_block = refund_icp(sub, from, amount, TOP_UP_CANISTER_REFUND_FEE).await?;
            Err(NotifyError::Refunded {
                reason: err.to_string(),
                block_index: refund_block,
            })
        }
    }
}

async fn deposit_cycles(
    canister_id: CanisterId,
    cycles: Cycles,
    mint_cycles: bool,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    if mint_cycles {
        ensure_balance(cycles, limiter_to_use)?;
    }

    let res: CallResult<()> = ic_cdk::api::call::call_with_payment128(
        candid::Principal::management_canister(),
        METHOD_DEPOSIT_CYCLES,
        (CanisterIdRecord {
            canister_id: canister_id.get().0,
        },),
        u128::from(cycles),
    )
    .await;

    res.map_err(|(code, msg)| {
        format!(
            "Depositing cycles failed with code {}: {:?}",
            code as i32, msg
        )
    })?;

    Ok(())
}

fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    // unused because of check above
    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}

#[test]
fn cmc_notify_top_up_not_rate_limited_by_invalid_top_up() {
    let good_account = AccountIdentifier::new(*TEST_USER1_PRINCIPAL, None);
    let good_account_icpts = Tokens::new(10, 0).unwrap();
    let bad_account = AccountIdentifier::new(*TEST_USER2_PRINCIPAL, None);
    let bad_account_icpts = Tokens::new(1, 0).unwrap();

    let state_machine = state_machine_builder_for_nns_tests().build();
    let nns_init_payloads = NnsInitPayloadsBuilder::new()
        .with_test_neurons()
        .with_ledger_accounts(vec![
            (good_account, good_account_icpts),
            (bad_account, bad_account_icpts),
        ])
        .build();
    setup_nns_canisters(&state_machine, nns_init_payloads);
    let non_existing_canister_id = CanisterId::from_u64(123_456_789);

    // First make sure topping up 400T cycles on a valid canister works.
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(4, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(400_000_000_000_000_u128));

    // Advance time by 1 hour to make sure the rate limit is reset.
    state_machine.advance_time(Duration::from_secs(60 * 60));

    // Now the attack begins - the bad account sends 0.69 tokens per 5 seconds to the non-existing
    // canister, which makes it 207T * 12 * 60 = 149.04P cycles per hour, close to the 150P limit,
    // while getting the 0.69 tokens back each time (the account only has 1 token in the
    // beginning).
    for _ in 0..(12 * 60) {
        let error = notify_top_up(
            &state_machine,
            non_existing_canister_id,
            Tokens::from_e8s(207_000_000),
        )
        .unwrap_err();
        assert_matches!(error, NotifyError::Refunded { .. });
        state_machine.advance_time(Duration::from_secs(5));
    }

    // Now the good account tries to top up 400T cycles on the governance canister again, which
    // should still succeed.
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(4, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(400_000_000_000_000_u128));
}
