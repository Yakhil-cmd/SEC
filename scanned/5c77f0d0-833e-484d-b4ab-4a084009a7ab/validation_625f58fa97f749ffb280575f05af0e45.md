### Title
SNS Swap DoS on Finalization via Front-Running When `min_direct_participation == max_direct_participation` - (File: `rs/sns/swap/src/swap.rs`)

---

### Summary

The SNS swap canister's initialization validation uses a non-strict `<` comparison that permits `min_direct_participation_icp_e8s == max_direct_participation_icp_e8s`. An unprivileged attacker can exploit this by front-running a finalizing contribution to leave exactly 1 e8 of capacity remaining. Any new participant's effective balance is then capped to 1 e8, which is less than `min_participant_icp_e8s`, causing `refresh_buyer_token_e8s` to reject the contribution. If the attacker is the only participant who can fill the remaining 1 e8 (because all others have hit `max_participant_icp_e8s`), and the attacker refuses to do so, the swap is permanently stuck below `min_direct_participation` until the deadline, forcing an abort and causing the SNS launch to fail.

---

### Finding Description

**Root Cause 1 — Non-strict validation allows `min == max`:**

In `rs/sns/init/src/lib.rs`, `validate_participation_constraints` uses a strict less-than check:

```rust
if max_direct_participation_icp_e8s < min_direct_participation_icp_e8s {
    return Err(...)
}
```

This allows `max_direct_participation_icp_e8s == min_direct_participation_icp_e8s`, meaning the swap has exactly one valid total ICP amount at which it can succeed. [1](#0-0) 

**Root Cause 2 — Capped increment checked against `min_participant_icp_e8s`:**

In `refresh_buyer_token_e8s`, the accepted increment is capped to `available_direct_participation_e8s()`:

```rust
let max_increment_e8s = self.available_direct_participation_e8s();
// ...
let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);
let new_balance_e8s = old_amount_icp_e8s.saturating_add(actual_increment_e8s);
```

Then the resulting `new_balance_e8s` (the post-cap value) is checked against `min_participant_icp_e8s`:

```rust
if new_balance_e8s < params.min_participant_icp_e8s {
    return Err(format!(
        "Rejecting participation of effective amount {}; minimum required to participate: {}",
        new_balance_e8s, params.min_participant_icp_e8s
    ));
}
``` [2](#0-1) [3](#0-2) [4](#0-3) 

**Attack Scenario (concrete example):**

- `min_direct_participation_icp_e8s = max_direct_participation_icp_e8s = 10 ICP`
- `min_participant_icp_e8s = 1 ICP`, `max_participant_icp_e8s = 4 ICP`

1. Alice contributes `4 ICP` (hits her per-participant max).
2. Bob contributes `4 ICP` (hits his per-participant max).
3. `current_direct_participation = 8 ICP`. Remaining capacity = `2 ICP`.
4. Attacker front-runs the next contribution with `2 ICP - 1 e8`. This is ≥ `min_participant_icp_e8s`, so it is accepted. `current_direct_participation = 10 ICP - 1 e8`.
5. Any new participant calls `refresh_buyer_token_e8s`:
   - `max_increment_e8s = 1`
   - `new_balance_e8s = 0 + 1 = 1 e8 < min_participant_icp_e8s = 1 ICP` → **REJECTED**
6. Alice and Bob have hit `max_participant_icp_e8s` → cannot contribute more.
7. Only the attacker can contribute the remaining 1 e8 (their `new_balance_e8s = 2 ICP ≥ min_participant_icp_e8s`), but they refuse.
8. `current_direct_participation = 10 ICP - 1 e8 < min_direct_participation = 10 ICP`.

The swap cannot commit because `min_direct_participation_icp_e8s_reached()` returns `false`:

```rust
pub fn min_direct_participation_icp_e8s_reached(&self) -> bool {
    // ...
    return self.current_direct_participation_e8s() >= min_direct_participation_icp_e8s;
}
``` [5](#0-4) 

`can_commit` requires `sufficient_participation()`, which requires `min_direct_participation_icp_e8s_reached()`: [6](#0-5) 

The swap is stuck until `swap_due_timestamp_seconds`, at which point it aborts.

---

### Impact Explanation

The SNS decentralization swap is permanently blocked from committing. All participants (Alice, Bob, and the attacker) have their ICP returned on abort. The SNS launch fails entirely. Since `min_direct_participation_icp_e8s`, `max_direct_participation_icp_e8s`, and `min_participant_icp_e8s` are immutable after swap creation, there is no recovery path during the swap duration. [7](#0-6) 

---

### Likelihood Explanation

The attack requires:
1. A swap configured with `min_direct_participation_icp_e8s == max_direct_participation_icp_e8s` — this is explicitly permitted by the current validation.
2. An attacker willing to lock up ICP for the swap duration (recovered on abort).
3. Timing the front-run so that all other participants have hit `max_participant_icp_e8s`.

Condition 1 is a valid and documented configuration (the swap targets an exact ICP amount). Conditions 2 and 3 are achievable by a motivated attacker. The attacker bears no permanent financial loss (ICP is returned on abort) and gains a competitive advantage by preventing a rival SNS from launching.

---

### Recommendation

Change the validation in `validate_participation_constraints` from a strict less-than to a less-than-or-equal check, mirroring the fix suggested in the referenced ETH crowdfund report:

```rust
// Before (allows min == max):
if max_direct_participation_icp_e8s < min_direct_participation_icp_e8s {

// After (prevents min == max):
if max_direct_participation_icp_e8s <= min_direct_participation_icp_e8s {
``` [1](#0-0) 

This ensures there is always a gap between `min` and `max`, so that even if `current_direct_participation = max - 1`, the swap can still abort gracefully (since `current_direct_participation >= min`) rather than being stuck in a state where it can neither commit nor accept new contributions.

---

### Proof of Concept

The following scenario can be verified against `rs/sns/swap/src/swap.rs` using the existing test infrastructure (`SwapBuilder`):

```rust
// Setup: min == max == 10 ICP, min_participant = 1 ICP, max_participant = 4 ICP
// 1. Alice contributes 4 ICP -> accepted (hits max_participant)
// 2. Bob contributes 4 ICP -> accepted (hits max_participant)
// 3. Attacker contributes (2 ICP - 1 e8) -> accepted (>= min_participant)
//    current_direct_participation = 10 ICP - 1 e8
// 4. New participant tries to contribute 1 ICP:
//    max_increment_e8s = 1
//    new_balance_e8s = 1 < min_participant_icp_e8s (1 ICP)
//    -> Err("Rejecting participation of effective amount 1; minimum required: 100000000")
// 5. swap.can_commit(now) == false (min_direct_participation not reached)
// 6. After deadline: swap.try_abort(deadline) == true -> Lifecycle::Aborted
```
