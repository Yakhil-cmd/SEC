### Title
Incomplete `PreInitializationSwap` Proposal Restriction Allows Governance Bypass During SNS Token Swap - (`File: rs/sns/governance/src/types.rs`)

### Summary

The SNS governance `PreInitializationSwap` mode is designed to restrict powerful proposal actions while a token swap is in progress, protecting swap participants from the SNS team making unilateral changes. However, the disallowed-action list in `proposal_action_is_allowed_in_pre_initialization_swap_or_err` is incomplete: it blocks six specific action types but silently allows all others, including `UpgradeSnsToNextVersion` (which upgrades the governance canister itself) and `ManageLedgerParameters` (which changes the ledger's transfer fee and token metadata). This is the direct IC analog of the GuardCM `delegatecall` bypass: the guard only checks a subset of operation types and returns `Ok(())` for everything else.

### Finding Description

`governance::Mode::PreInitializationSwap` enforces its restrictions through two functions in `rs/sns/governance/src/types.rs`.

`functions_disallowed_in_pre_initialization_swap()` returns a hard-coded list of six `NervousSystemFunction` values: [1](#0-0) 

`proposal_action_is_allowed_in_pre_initialization_swap_or_err()` converts the incoming `Action` to a `NervousSystemFunction` and checks whether its `id` appears in that list. If the action is not `ExecuteGenericNervousSystemFunction` and its id is not in the list, the function returns `Ok(())` unconditionally: [2](#0-1) 

The `NativeAction` enum defines all action ids. The following actions exist in production but are absent from the disallowed list: [3](#0-2) 

Concretely:

- `UpgradeSnsToNextVersion` (id 7) — upgrades SNS core canisters (governance, ledger, root, swap, archive, index) to the next version on the blessed path.
- `ManageLedgerParameters` (id 13) — changes the ledger's transfer fee, token name, and token symbol.
- `ManageDappCanisterSettings` (id 14) — changes compute/memory/freeze settings for dapp canisters.
- `AdvanceSnsTargetVersion` (id 15) — advances the SNS upgrade target version.
- `RegisterExtension` / `UpgradeExtension` / `ExecuteExtensionOperation` (ids 17–19) — register and execute SNS extension canisters.

`MakeProposal` is explicitly allowed during `PreInitializationSwap`: [4](#0-3) 

So any neuron holder can submit a proposal with any of the above actions, and `allows_proposal_action_or_err` will pass it through. The proposal then executes normally via `perform_proposal_action`: [5](#0-4) 

### Impact Explanation

**`UpgradeSnsToNextVersion`** is the highest-severity omission. It upgrades the governance canister itself to the next blessed version. If the new governance Wasm does not enforce `PreInitializationSwap` restrictions (e.g., a future version relaxes the list), the upgrade during the swap period effectively removes the guard — exactly the GuardCM pattern. Even without that, upgrading the governance canister mid-swap can disrupt the swap state machine, corrupt in-flight proposal data, or change voting rules while swap participants are committing ICP.

**`ManageLedgerParameters`** allows changing the transfer fee and token name/symbol while the swap is live. Raising the transfer fee mid-swap increases the cost for participants to move tokens after the swap closes, directly harming them. Changing the token name/symbol causes confusion and could constitute fraud. [6](#0-5) 

### Likelihood Explanation

The attacker entry path requires only neuron voting power — no privileged keys, no admin access. During `PreInitializationSwap`, the SNS team (or any coalition of neuron holders with majority voting power) can submit and pass these proposals. The SNS team itself is the most realistic actor, since they hold the founding neurons. The swap period is a known, bounded window, making targeted exploitation straightforward. The `MakeProposal` command is explicitly permitted during this mode, so there is no additional barrier.

### Recommendation

Extend `functions_disallowed_in_pre_initialization_swap()` to include all action types that can alter governance behavior, ledger parameters, or SNS canister code during the swap period:

```rust
pub fn functions_disallowed_in_pre_initialization_swap() -> Vec<NervousSystemFunction> {
    vec![
        NervousSystemFunction::manage_nervous_system_parameters(),
        NervousSystemFunction::transfer_sns_treasury_funds(),
        NervousSystemFunction::mint_sns_tokens(),
        NervousSystemFunction::upgrade_sns_controlled_canister(),
        NervousSystemFunction::register_dapp_canisters(),
        NervousSystemFunction::deregister_dapp_canisters(),
        // Missing entries:
        NervousSystemFunction::upgrade_sns_to_next_version(),
        NervousSystemFunction::manage_ledger_parameters(),
        NervousSystemFunction::manage_dapp_canister_settings(),
        NervousSystemFunction::advance_sns_target_version(),
        NervousSystemFunction::register_extension(),
        NervousSystemFunction::upgrade_extension(),
        NervousSystemFunction::execute_extension_operation(),
    ]
}
```

Alternatively, invert the logic to an explicit allowlist (only `Motion`, `AddGenericNervousSystemFunction`, `RemoveGenericNervousSystemFunction`, and `ExecuteGenericNervousSystemFunction` targeting non-SNS canisters are currently intended to be allowed), so that newly added action types are blocked by default rather than silently permitted.

### Proof of Concept

1. An SNS is deployed and enters `PreInitializationSwap` mode (swap is live).
2. A neuron holder calls `manage_neuron` with `Command::MakeProposal` containing `Action::UpgradeSnsToNextVersion({})`. This passes `allows_manage_neuron_command_or_err` because `MakeProposal` is explicitly allowed.
3. `make_proposal` calls `allows_proposal_action_or_err`. `UpgradeSnsToNextVersion` is not `ExecuteGenericNervousSystemFunction`, and its id (7) is not in `functions_disallowed_in_pre_initialization_swap()`, so the function returns `Ok(())`.
4. The proposal is created and voted on. If it passes, `perform_upgrade_sns_to_next_version` is called, upgrading the governance canister mid-swap.
5. Similarly, a `ManageLedgerParameters` proposal with `transfer_fee = Some(u64::MAX)` passes the same check and executes `perform_manage_ledger_parameters`, setting the transfer fee to an extreme value while the swap is live. [7](#0-6) [8](#0-7)

    fn proposal_action_is_allowed_in_pre_initialization_swap_or_err(
        action: &Action,
        disallowed_target_canister_ids: &HashSet<CanisterId>,
        id_to_nervous_system_function: &BTreeMap<u64, NervousSystemFunction>,
    ) -> Result<(), GovernanceError> {
        // ExecuteGenericNervousSystemFunction is special in that it
        // is only disallowed in some cases.
        if let Action::ExecuteGenericNervousSystemFunction(execute) = action {
            return Self::execute_generic_nervous_system_function_is_allowed_in_pre_initialization_swap_or_err(
                    execute,
                    disallowed_target_canister_ids,
                    id_to_nervous_system_function,
                );
        }

        let nervous_system_function = NervousSystemFunction::from(action.clone());

        let is_action_disallowed = Self::functions_disallowed_in_pre_initialization_swap()
            .into_iter()
            .any(|t| t.id == nervous_system_function.id);

        if is_action_disallowed {
            Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Proposal type for {:?} is not allowed while governance is in \
                     PreInitializationSwap ({}) mode.",
                    nervous_system_function,
                    Mode::PreInitializationSwap as i32,
                ),
            ))
        } else {
            Ok(())
        }
    }
