### Title
Large `Nat` Allowance Amount Causes `FieldsDisplay` ICRC-21 Consent Message to Fail — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

When a user calls `icrc21_canister_call_consent_message` for an `icrc2_approve` transaction with an `amount` exceeding `u64::MAX` (a valid value on U256-token ledgers) and requests `FieldsDisplay` output, the `nat_to_u64` helper unconditionally returns a `GenericError`. This mirrors the original report's pattern: a sentinel/large value triggers a code path that assumes a narrower type, causing the operation to fail for a legitimate special case.

---

### Finding Description

The ICRC-21 consent message library in `packages/icrc-ledger-types/src/icrc21/responses.rs` contains two rendering paths for token amounts:

- **`GenericDisplay`**: calls `convert_tokens_to_string_representation`, which uses `BigUint::to_f64()` — succeeds for any value (though with precision loss for very large numbers).
- **`FieldsDisplay`**: calls `nat_to_u64(amount)?`, which hard-fails for any `Nat > u64::MAX`. [1](#0-0) 

The `add_allowance` method, invoked for every `icrc2_approve` consent message, takes this `FieldsDisplay` branch: [2](#0-1) 

The same failure occurs in `add_existing_allowance` (for the `expected_allowance` field) and in `add_amount` / `add_fee` for `icrc1_transfer` and `icrc2_transfer_from`: [3](#0-2) 

The `icrc2_approve` argument type accepts an unbounded `Nat`: [4](#0-3) 

ICRC-1 ledgers compiled with the `u256-tokens` feature (e.g., ckETH, ckERC20 ledgers) use `U256` as their token type, whose `MAX` is `2^256 - 1`, far exceeding `u64::MAX`: [5](#0-4) 

The `Value::TokenAmount` struct that `FieldsDisplay` populates stores `amount: u64`, making it structurally incapable of representing U256-range values: [6](#0-5) 

The public endpoint `icrc21_canister_call_consent_message` is exposed on all ICRC-1/ICRC-2 ledgers and routes directly into `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints`: [7](#0-6) 

---

### Impact Explanation

Any unprivileged caller can submit a well-formed `icrc2_approve` Candid argument with `amount > u64::MAX` and `device_spec = FieldsDisplay` to `icrc21_canister_call_consent_message` on any U256-token ledger. The endpoint returns `GenericError { error_code: 500, description: "Failed to convert tokens to u64" }` instead of a valid consent message.

Hardware wallets (e.g., Ledger devices) that use `FieldsDisplay` to render structured transaction fields will receive this error and refuse to sign the transaction. This effectively blocks users of hardware wallets from approving large (> `u64::MAX`) allowances on U256-token ledgers such as ckETH or ckERC20 ledgers, even though the underlying `icrc2_approve` call would succeed. The `GenericDisplay` path is unaffected, so software wallets that fall back to it are not blocked.

---

### Likelihood Explanation

The trigger is straightforward and requires no privilege: any user with a hardware wallet that requests `FieldsDisplay` and wishes to set a large allowance (e.g., "infinite approval" patterns common in DeFi, or any amount exceeding `u64::MAX ≈ 1.8 × 10^19`) will hit this path. U256-token ledgers (ckETH, ckERC20) are production-deployed on mainnet. The `FieldsDisplay` path is the structured display mode explicitly designed for hardware wallets per the ICRC-21 spec.

---

### Recommendation

Replace the hard `nat_to_u64` conversion in `add_allowance`, `add_existing_allowance`, `add_amount`, and `add_fee` with a graceful fallback for values exceeding `u64::MAX`. Options include:

1. **Saturate at `u64::MAX`** and annotate the display with a note such as "≥ max u64 (unlimited)".
2. **Add a new `Value` variant** (e.g., `TokenAmountLarge { amount: String, symbol: String }`) to carry the full decimal string representation for large amounts.
3. **Fall back to `GenericDisplay` string rendering** (`convert_tokens_to_string_representation`) when the amount does not fit in `u64`, so `FieldsDisplay` degrades gracefully rather than erroring.

The `Value::TokenAmount` struct's `amount: u64` field is the structural root cause and should be widened or supplemented for U256-token ledger compatibility.

---

### Proof of Concept

```
// Attacker (any unprivileged user) calls on a U256-token ledger (e.g., ckETH ledger):
icrc21_canister_call_consent_message({
    method: "icrc2_approve",
    arg: Candid-encode(ApproveArgs {
        spender: { owner: <any principal>, subaccount: None },
        amount: 2^128,          // valid Nat, exceeds u64::MAX
        from_subaccount: None,
        expected_allowance: None,
        expires_at: None,
        fee: None,
        memo: None,
        created_at_time: None,
    }),
    user_preferences: {
        metadata: { language: "en", utc_offset_minutes: None },
        device_spec: Some(FieldsDisplay),   // hardware wallet path
    }
})

// Result: Err(GenericError { error_code: 500, description: "Failed to convert tokens to u64" })
// instead of a valid FieldsDisplayMessage consent info.
```
