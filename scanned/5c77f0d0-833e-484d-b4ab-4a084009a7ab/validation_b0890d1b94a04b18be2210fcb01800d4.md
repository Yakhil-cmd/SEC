Based on my analysis of the production code, the described vulnerability is real and exploitable.

### Title
Unbounded `crypto_verify_complaint()` Calls Per Cycle via Unvalidated Pool Flooding — (`rs/consensus/idkg/src/complaints.rs`)

### Summary

`IDkgComplaintHandlerImpl::validate_complaints` contains a deduplication gap: the in-batch `validated_complaints` BTreeSet only receives a `ComplaintKey` insertion when a complaint is **successfully** validated (`MoveToValidated`). When complaints fail validation with a permanent error (`HandleInvalid`), the key is never inserted, so every subsequent complaint with the same `(transcript_id, dealer_id, complainer_id)` key but different `internal_complaint_raw` bytes triggers a full `crypto_verify_complaint()` call within the same `on_state_change` cycle.

### Finding Description

In `validate_complaints`, the dedup logic is:

```rust
let mut validated_complaints = BTreeSet::new();   // empty at cycle start
// ...
if validated_complaints.contains(&key) {           // in-batch dedup
    ret.push(IDkgChangeAction::RemoveUnvalidated(id));
    continue;
}
// ...
let action = self.crypto_verify_complaint(id, transcript, signed_complaint);
if let Some(IDkgChangeAction::MoveToValidated(_)) = action {
    validated_complaints.insert(key);              // only inserted on SUCCESS
}
``` [1](#0-0) 

The cross-batch dedup (`has_complainer_issued_complaint`) queries only the **validated** pool: [2](#0-1) 

When `crypto_verify_complaint` returns `HandleInvalid` (permanent error — e.g., valid signature over invalid complaint content), the key is **not** inserted into `validated_complaints`. The next complaint with the same `ComplaintKey` but different content passes both dedup checks and triggers another full crypto call. [3](#0-2) 

The pool (`IDkgObjectPool`) is keyed by `IDkgMessageId` derived from `IDkgArtifactId::from(&message)`, so K complaints with different `internal_complaint_raw` bytes produce K distinct IDs and are all admitted: [4](#0-3) 

There is no observed per-sender admission cap or pool size limit on the unvalidated section.

### Impact Explanation

A Byzantine subnet node (within the f < n/3 fault tolerance budget) can sign K complaints with the same `(transcript_id, dealer_id)` but distinct `internal_complaint_raw` bytes. Each carries a valid BLS signature but invalid complaint content. In a single `on_state_change` cycle:

1. Signature verification passes for each (expensive BLS verify).
2. `verify_complaint()` is called for each (involves threshold crypto over the full transcript).
3. Each returns `HandleInvalid` → key never enters `validated_complaints`.
4. All K complaints trigger full `crypto_verify_complaint()`.

Result: O(K) expensive threshold-crypto operations per cycle, stalling the complaint handler and delaying transcript loading and threshold-signature availability.

A non-subnet P2P peer can execute a cheaper variant (invalid signatures → signature-verify-only cost × K), but the expensive path requires a Byzantine subnet node.

### Likelihood Explanation

A single Byzantine subnet node suffices. No threshold corruption, no key leakage, no external dependency. The attacker only needs to craft and gossip K signed artifacts, which is normal P2P behavior. The pool has no apparent size cap on unvalidated complaints.

### Recommendation

Insert the `ComplaintKey` into `validated_complaints` on **any terminal outcome** (both `MoveToValidated` and `HandleInvalid`), not only on success. This bounds crypto calls to O(1) per `(complainer, transcript, dealer)` per cycle regardless of unvalidated pool size:

```rust
let action = self.crypto_verify_complaint(id, transcript, signed_complaint);
match &action {
    Some(IDkgChangeAction::MoveToValidated(_))
    | Some(IDkgChangeAction::HandleInvalid(_, _)) => {
        validated_complaints.insert(key);
    }
    _ => {}
}
```

### Proof of Concept

Inject 1000 `SignedIDkgComplaint` messages into the unvalidated pool, all sharing the same `(transcript_id, dealer_id, complainer_id)` but with distinct `internal_complaint_raw` bytes (valid BLS signatures, invalid complaint content). Call `validate_complaints` once. Assert that the crypto mock's `verify_complaint` counter equals exactly 1000. With the fix applied, assert the counter equals 1. [5](#0-4)

    fn validate_complaints(
        &self,
        idkg_pool: &dyn IDkgPool,
        block_reader: &dyn IDkgBlockReader,
        active_transcripts: &BTreeMap<IDkgTranscriptId, &IDkgTranscript>,
    ) -> IDkgChangeSet {
        let requested_transcripts = self.requested_transcripts(block_reader);

        // Collection of validated complaints <complainer Id, transcript Id, dealer Id>
        let mut validated_complaints = BTreeSet::new();

        let mut ret = Vec::new();
        for (id, signed_complaint) in idkg_pool.unvalidated().complaints() {
            let complaint = signed_complaint.get();
            // Remove the duplicate entries
            let key = ComplaintKey::from(&signed_complaint);
            if validated_complaints.contains(&key) {
                self.metrics
                    .complaint_errors_inc("duplicate_complaints_in_batch");
                ret.push(IDkgChangeAction::RemoveUnvalidated(id));
                continue;
            }

            match Action::action(
                block_reader,
                active_transcripts,
                &requested_transcripts,
                complaint.idkg_complaint.transcript_id.source_height(),
                &complaint.idkg_complaint.transcript_id,
            ) {
                Action::Process(transcript) => {
                    if self.has_complainer_issued_complaint(
                        idkg_pool,
                        &complaint.idkg_complaint,
                        &signed_complaint.signature.signer,
                    ) {
                        self.metrics.complaint_errors_inc("duplicate_complaint");
                        ret.push(IDkgChangeAction::RemoveUnvalidated(id));
                    } else {
                        let action = self.crypto_verify_complaint(id, transcript, signed_complaint);
                        if let Some(IDkgChangeAction::MoveToValidated(_)) = action {
                            validated_complaints.insert(key);
                        }
                        ret.append(&mut action.into_iter().collect());
                    }
                }
                Action::Drop => ret.push(IDkgChangeAction::RemoveUnvalidated(id)),
                Action::Defer => {}
            }
        }

        ret
    }

    fn has_complainer_issued_complaint(
        &self,
        idkg_pool: &dyn IDkgPool,
        idkg_complaint: &IDkgComplaint,
        complainer_id: &NodeId,
    ) -> bool {
        let prefix = complaint_prefix(
            &idkg_complaint.transcript_id,
            &idkg_complaint.dealer_id,
            complainer_id,
        );
        idkg_pool
            .validated()
            .complaints_by_prefix(prefix)
            .any(|(_, signed_complaint)| {
                let complaint = signed_complaint.get();
                signed_complaint.signature.signer == *complainer_id
                    && complaint.idkg_complaint.transcript_id == idkg_complaint.transcript_id
                    && complaint.idkg_complaint.dealer_id == idkg_complaint.dealer_id
            })
    }
