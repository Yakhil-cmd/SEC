### Title
`get_btc_address` Accepts Minter's Own Principal While `update_balance` Traps, Creating an Unrecoverable BTC Black-Hole Address — (`rs/bitcoin/ckbtc/minter/src/updates/get_btc_address.rs`)

---

### Summary

`get_btc_address` has no guard against the minter's own principal being passed as `owner`, but `update_balance` unconditionally traps when the resolved owner equals the minter's canister ID. Any unprivileged caller can obtain the minter's deposit address, send BTC there, and permanently lock those funds with no on-chain recovery path short of a governance-controlled canister upgrade.

---

### Finding Description

`get_btc_address` resolves the owner and checks only for the anonymous principal: [1](#0-0) 

There is no guard preventing `owner = minter_canister_id`. The call succeeds and returns a deterministic P2WPKH address derived from the minter's ECDSA key and the minter's own principal.

`update_balance`, however, unconditionally traps for that same principal: [2](#0-1) 

The existing integration test `test_illegal_caller` confirms this trap is intentional and permanent: [3](#0-2) 

The minter's internal background task `fetch_main_utxos` only fetches UTXOs for the minter's *main/change address* (derived from the root ECDSA key with an empty derivation path), not for the deposit address derived from `owner=minter_id`: [4](#0-3) 

These are two distinct Bitcoin addresses. UTXOs at the deposit address derived from `owner=minter_id` are never swept into `available_utxos` by any background task, and `update_balance` is the only user-facing path to process deposit UTXOs.

---

### Impact Explanation

BTC sent to the address returned by `get_btc_address(owner=minter_id)` is permanently unspendable by the minter's normal operation:

- No ckBTC is minted (the depositor receives nothing).
- The UTXOs are not added to `available_utxos` and cannot be used for withdrawals.
- The only recovery path is a governance-approved canister upgrade that adds a special-case sweep for that address.

The minter's overall ckBTC supply invariant is not broken (no inflation), but real BTC value is destroyed from the perspective of the depositor.

---

### Likelihood Explanation

The attack requires no privilege. `get_btc_address` is a public update endpoint callable by any non-anonymous principal. The attacker must spend their own BTC (or socially engineer another user into sending BTC to the published address), making this a griefing/fund-destruction attack rather than a theft. The cost to the attacker equals the amount locked, which limits casual exploitation but does not prevent a motivated actor.

---

### Recommendation

Add the same minter-principal guard to `get_btc_address` that already exists in `update_balance`:

```rust
// in get_btc_address, after resolving `owner`:
let minter_id = ic_cdk::api::canister_self();
assert_ne!(owner, minter_id, "the minter's own principal cannot be used as a deposit owner");
```

This mirrors the existing pattern and closes the asymmetry.

---

### Proof of Concept

```rust
// 1. Any unprivileged caller obtains the minter's deposit address:
let minter_deposit_addr = get_btc_address(GetBtcAddressArgs {
    owner: Some(minter_canister_id),
    subaccount: None,
}).await; // succeeds — no guard

// 2. Attacker sends BTC to `minter_deposit_addr` on-chain.

// 3. Attempt to credit the deposit:
let result = update_balance(UpdateBalanceArgs {
    owner: Some(minter_canister_id),
    subaccount: None,
}, &runtime).await;
// → ic_cdk::trap("cannot update minter's balance")
// BTC is now permanently locked; no ckBTC is minted; no recovery path exists.
```

The existing test at `rs/bitcoin/ckbtc/minter/tests/tests.rs:684` already proves step 3 traps. Step 1 is confirmed by the absence of any minter-principal check in `get_btc_address`. [5](#0-4) [6](#0-5)

    args: UpdateBalanceArgs,
    runtime: &R,

    if args.owner.unwrap_or(caller) == runtime.id() {
        ic_cdk::trap("cannot update minter's balance");

async fn fetch_main_utxos<R: CanisterRuntime>(
    main_account: &Account,
    main_address: &String,
    runtime: &R,
) -> Vec<Utxo> {
    let (btc_network, min_confirmations) =
        state::read_state(|s| (s.btc_network, s.min_confirmations));

    let utxos = match management::get_utxos(
        btc_network,
        main_address,
        min_confirmations,
        management::CallSource::Minter,
        runtime,
    )
    .await
    {
        Ok(response) => response.utxos,
        Err(e) => {
            log!(
                Priority::Info,
                "[fetch_main_utxos]: failed to fetch UTXOs for the main address {main_address}: {e}",
            );
            return vec![];
        }
    };

    state::read_state(|s| match s.utxos_state_addresses.get(main_account) {
        Some(known_utxos) => utxos
            .into_iter()
            .filter(|u| !known_utxos.contains(u))
            .collect(),
        None => utxos,
    })
}
