### Title
Single `ManagePrincipals` Holder Can Evict All Other Principals and Take Over a Shared SNS Neuron - (`rs/sns/governance/src/neuron.rs`)

---

### Summary

In the SNS governance canister, any principal holding the `ManagePrincipals` permission on a neuron can call `manage_neuron` → `RemoveNeuronPermissions` to strip all permissions from every other principal on that neuron, including removing `ManagePrincipals` from all peers. There is no hierarchical "owner" authority above the flat set of `ManagePrincipals` holders. A single compromised or malicious `ManagePrincipals` holder can therefore take sole control of a shared neuron and all its staked SNS tokens.

---

### Finding Description

SNS neurons support a multi-principal access control list. Each principal can hold a set of `NeuronPermissionType` values. The highest-privilege permission is `ManagePrincipals` (value `2`), which grants the ability to add or remove any permission for any principal on the neuron.

The authorization check for `RemoveNeuronPermissions` is performed in `check_principal_authorized_to_change_permissions`:

```rust
// rs/sns/governance/src/neuron.rs
pub(crate) fn check_principal_authorized_to_change_permissions(
    &self,
    caller: &PrincipalId,
    permissions_to_change: NeuronPermissionList,
) -> Result<(), GovernanceError> {
    let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
        vec![ManagePrincipals, ManageVotingPermission]
    } else {
        vec![ManagePrincipals]
    };
    let caller_authorized = sufficient_permissions
        .iter()
        .any(|p| self.is_authorized(caller, *p));
    if caller_authorized { Ok(()) } else { Err(...) }
}
```

This check only verifies that the **caller** holds `ManagePrincipals`. It does **not** verify:
- Whether the caller is removing `ManagePrincipals` from another peer (a lateral removal)
- Whether the operation would leave the caller as the sole `ManagePrincipals` holder
- Whether any "owner" principal consented to the change

The `remove_neuron_permissions` function then proceeds unconditionally:

```rust
// rs/sns/governance/src/governance.rs
neuron.check_principal_authorized_to_change_permissions(
    caller,
    permissions_to_remove.clone(),
)?;
// No further guard — proceeds to remove all listed permissions from any target principal
let principal_id_was_removed = self
    .get_neuron_result_mut(neuron_id)?
    .remove_permissions_for_principal(
        principal_id,
        permissions_to_remove.permissions.clone(),
    )?;
```

The proto documentation itself acknowledges the danger: *"This is a dangerous operation as it's possible to remove all permissions for a neuron and no longer be able to modify its state."*

The attack sequence for a shared neuron with principals `A`, `B`, `C` all holding `ManagePrincipals`:

1. `A` (malicious/compromised) sends `manage_neuron` → `RemoveNeuronPermissions` targeting `B` with `NeuronPermissionList::all()`.
2. `A` sends `manage_neuron` → `RemoveNeuronPermissions` targeting `C` with `NeuronPermissionList::all()`.
3. `A` is now the sole `ManagePrincipals` holder and can exercise all other permissions (e.g., `Disburse`, `Split`, `DisburseMaturity`) to drain the neuron's staked tokens.

---

### Impact Explanation

A single malicious or compromised `ManagePrincipals` holder can:
- Remove all other principals' permissions from a shared neuron in two ingress calls.
- Become the sole controller of the neuron.
- Disburse or split the neuron's staked SNS tokens to an attacker-controlled account.
- Permanently lock out all other co-owners from the neuron.

The staked tokens are at direct risk. The neuron's governance voting power is also hijacked, which can affect SNS proposal outcomes if the neuron has significant stake.

---

### Likelihood Explanation

Multi-principal neuron management is an explicitly supported and documented feature of SNS governance (the `max_number_of_principals_per_neuron` parameter exists precisely for this). Teams, DAOs, and organizations routinely share `ManagePrincipals` across multiple principals to avoid single-key risk. Any one of those principals being compromised (phishing, key leak, insider threat) triggers this takeover. The attack requires only two standard `manage_neuron` ingress calls with no special preconditions beyond already holding `ManagePrincipals`.

---

### Recommendation

Introduce a hierarchical authority model for `ManagePrincipals` operations:

1. **Designate a neuron "owner" principal** (e.g., the original claimer) whose `ManagePrincipals` permission cannot be removed by peers — only by themselves or by a governance proposal.
2. **Prevent lateral removal of `ManagePrincipals`**: A `ManagePrincipals` holder should not be able to remove `ManagePrincipals` from another `ManagePrincipals` holder without the target's consent or a supermajority of current `ManagePrincipals` holders.
3. **Alternatively**, require that removing `ManagePrincipals` from any principal requires the caller to be the neuron's designated owner principal, not merely any `ManagePrincipals` holder.

---

### Proof of Concept

**Attacker-controlled entry path**: Standard ingress `manage_neuron` call to the SNS governance canister, authenticated as a legitimate `ManagePrincipals` holder.

**Step 1** — Remove peer B's permissions:
```
manage_neuron({
  subaccount: <neuron_subaccount>,
  command: RemoveNeuronPermissions({
    principal_id: <B's principal>,
    permissions_to_remove: NeuronPermissionList::all()
  })
})
```
Authorized by `check_principal_authorized_to_change_permissions` because caller A holds `ManagePrincipals`. [1](#0-0) 

**Step 2** — Repeat for all other co-owners. No guard prevents this. [2](#0-1) 

**Step 3** — A is now sole `ManagePrincipals` holder. A calls `manage_neuron` → `Disburse` to drain staked tokens. The `remove_permissions_for_principal` function removes the entry entirely when all permission types are stripped: [3](#0-2) 

The proto definition confirms `ManagePrincipals` is the only gate for this operation, with no owner-level override: [4](#0-3) 

The `remove_neuron_permissions` function in governance confirms no additional check exists between the authorization gate and the mutation: [5](#0-4)

    pub(crate) fn check_principal_authorized_to_change_permissions(
        &self,
        caller: &PrincipalId,
        permissions_to_change: NeuronPermissionList,
    ) -> Result<(), GovernanceError> {
        // If the permissions to change are exclusively voting related,
        // ManagePrincipals or ManageVotingPermission is sufficient.
        // Otherwise, only ManagePrincipals is sufficient.
        let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
            vec![
                NeuronPermissionType::ManagePrincipals,
                NeuronPermissionType::ManageVotingPermission,
            ]
        } else {
            vec![NeuronPermissionType::ManagePrincipals]
        };

        // The caller is authorized if they have any of the sufficient permissions
        let caller_authorized = sufficient_permissions
            .iter()
            .any(|sufficient_permission| self.is_authorized(caller, *sufficient_permission));

        if caller_authorized {
            Ok(())
        } else {
            let caller_permissions = self.permissions_for_principal(caller);
            Err(GovernanceError::new_with_message(
                ErrorType::NotAuthorized,
                format!(
                    "Caller '{caller:?}' is not authorized to modify permissions {permissions_to_change} for neuron '{}' as it does not have any of {sufficient_permissions:?}. (Caller's permissions are {caller_permissions})",
                    self.id.as_ref().expect("Neuron must have a NeuronId"),
                ),
            ))
        }
    }

    fn remove_neuron_permissions(
        &mut self,
        neuron_id: &NeuronId,
        caller: &PrincipalId,
        remove_neuron_permissions: &RemoveNeuronPermissions,
    ) -> Result<(), GovernanceError> {
        let neuron = self.get_neuron_result(neuron_id)?;

        let permissions_to_remove = remove_neuron_permissions
            .permissions_to_remove
            .as_ref()
            .ok_or_else(|| {
                GovernanceError::new_with_message(
                    ErrorType::InvalidCommand,
                    "RemoveNeuronPermissions command must provide permissions to remove",
                )
            })?;

        // A simple check to prevent DoS attack with large number of permission changes.
        if permissions_to_remove.permissions.len() > NeuronPermissionType::all().len() {
            return Err(GovernanceError::new_with_message(
                ErrorType::InvalidCommand,
                "RemoveNeuronPermissions command provided more permissions than exist in the system",
            ));
        }

        let principal_id = remove_neuron_permissions
            .principal_id
            .ok_or_else(|| {
                GovernanceError::new_with_message(
                    ErrorType::InvalidCommand,
                    "RemoveNeuronPermissions command must provide a PrincipalId to remove permissions from",
                )
            })?;

        neuron.check_principal_authorized_to_change_permissions(
            caller,
            permissions_to_remove.clone(),
        )?;

        // Re-borrow the neuron mutably to update now that the preconditions have been met
        let principal_id_was_removed = self
            .get_neuron_result_mut(neuron_id)?
            .remove_permissions_for_principal(
                principal_id,
                permissions_to_remove.permissions.clone(),
            )?;

        if principal_id_was_removed == RemovePermissionsStatus::AllPermissionTypesRemoved {
            GovernanceProto::remove_neuron_from_principal_in_principal_to_neuron_ids_index(
                &mut self.principal_to_neuron_ids_index,
                neuron_id,
                &principal_id,
            )
        }

        Ok(())
    }
