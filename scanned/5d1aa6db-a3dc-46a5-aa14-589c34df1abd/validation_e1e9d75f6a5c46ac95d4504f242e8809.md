### Title
Unconditional `.unwrap()` in `From<PrincipalIdProto> for PrincipalId` Causes Replica Panic on Oversized `serialized_id` — (`rs/types/base_types/src/principal_id.rs`)

---

### Summary

`From<PrincipalIdProto> for PrincipalId` unconditionally calls `.unwrap()` on a `TryFrom` that returns `Err` for any `serialized_id` longer than 29 bytes. The `prost::Message::merge_field` implementation for `PrincipalId` calls this conversion during protobuf decoding. Any protobuf message containing a `PrincipalId` field that is decoded from attacker-controlled bytes before authentication will trigger a replica panic.

---

### Finding Description

**Root cause — line 422–426:** [1](#0-0) 

```rust
impl From<PrincipalIdProto> for PrincipalId {
    fn from(pb: PrincipalIdProto) -> Self {
        PrincipalId::try_from(pb.serialized_id).unwrap()   // panics if len > 29
    }
}
```

`PrincipalId::try_from(Vec<u8>)` delegates to `Principal::try_from`, which enforces the 29-byte cap and returns `Err` for longer inputs. [2](#0-1) 

The error is silently converted to a panic by `.unwrap()` — there is no `?` propagation, no length guard, and no fallback.

**Propagation through `prost::Message::merge_field` — lines 441–452:** [3](#0-2) 

```rust
fn merge_field(...) -> Result<(), prost::DecodeError> {

    *self = Self::from(pid_proto);   // <-- calls the unwrap() path
```

`merge_field` is the prost hook called for every field during `prost::Message::decode`. Any outer protobuf struct that embeds `PrincipalId` directly (using the `prost::Message` impl) will invoke this path when decoded. The function signature returns `Result`, but the panic inside `Self::from(pid_proto)` unwinds the stack before `Ok(())` is ever reached.

### Impact Explanation

A replica thread that panics during protobuf decoding of a consensus artifact, xnet stream slice, or state-sync chunk will crash. In the IC, replicas run as single processes; a panic in the decoding path terminates the replica. Repeated delivery of the same malformed message to every honest node on a subnet causes a subnet halt. This violates the BFT invariant that a single Byzantine actor (below the ⌊(n−1)/3⌋ fault threshold) cannot crash honest replicas.

---

### Likelihood Explanation

**Deserialization precedes authentication in all three named paths:**

- **Consensus artifacts**: A block proposal or notarization is received as raw bytes over P2P. The bytes must be deserialized to extract the payload and the embedded signature before the signature can be verified. A panic during deserialization fires before any crypto check.
- **XNet stream slices**: The slice is decoded from the block payload. Certification verification happens after the slice is parsed out of the block.
- **State-sync chunks**: Chunks are decoded from raw bytes received from peers; Merkle-proof verification follows decoding.

A single Byzantine node — which holds a valid TLS certificate and is admitted to the P2P overlay — can craft a protobuf blob with a `PrincipalId` field whose `serialized_id` is 30+ bytes and gossip it to every honest replica. No threshold of corrupted nodes is required; one is sufficient.

**Confirmed absent guards:**
- No length check before `.unwrap()` in `From<PrincipalIdProto>`.
- No `TryFrom` variant of the conversion used in `merge_field`.
- No catch-unwind wrapper around the prost decode call in the artifact-pool admission path (not visible in this repo, but the panic propagates to the thread boundary regardless).

### Recommendation

1. Replace `From<PrincipalIdProto> for PrincipalId` with `TryFrom<PrincipalIdProto> for PrincipalId` returning `Result<PrincipalId, PrincipalIdError>`.
2. Update `prost::Message::merge_field` to propagate the error as a `prost::DecodeError` instead of panicking:

```rust
fn merge_field(...) -> Result<(), prost::DecodeError> {
    let mut pid_proto = PrincipalIdProto::from(*self);
    pid_proto.merge_field(tag, wire_type, buf, ctx)?;
    *self = Self::try_from(pid_proto)
        .map_err(|e| prost::DecodeError::new(e.to_string()))?;
    Ok(())
}
```

3. Audit every call site of `From<PrincipalIdProto> for PrincipalId` (the `into()` / `from()` call chain) and replace with the fallible variant.
