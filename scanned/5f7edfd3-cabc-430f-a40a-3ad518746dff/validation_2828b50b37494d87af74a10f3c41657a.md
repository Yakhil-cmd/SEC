### Title
Compromised NeuronManagement Followee Can Permanently Lock a Neuron - (File: rs/nns/governance/src/governance.rs)

### Summary

In NNS governance, a neuron whose controller key has been destroyed (a documented and supported use case) can only be managed through `ManageNeuron` proposals. These proposals are voted on exclusively by the neuron's followees on the `Topic::NeuronManagement` topic, each with equal voting power of 1. A compromised followee holding ≥ 50% of the followee votes can immediately and permanently reject any `ManageNeuron` proposal — including proposals to remove it from the followee list — creating an irrecoverable deadlock identical in structure to the CANCELLER_ROLE bug.

### Finding Description

The NNS governance canister implements a `ManageNeuron` proposal type that allows a group of followee neurons to collectively control a target neuron. The `compute_ballots_for_manage_neuron_proposal` function assigns each followee exactly 1 voting power: [1](#0-0) 

The `can_make_decision` function immediately rejects a proposal when NO votes reach ≥ 50% of total: [2](#0-1) 

Changing the `NeuronManagement` followees directly requires the neuron's controller key: [3](#0-2) 

The official documentation for `ManageNeuron` proposals explicitly endorses destroying the controller key so the neuron is controlled only by its followees: [4](#0-3) 

Once the controller key is destroyed, the only path to change the `NeuronManagement` followee list is through a `ManageNeuron` proposal voted on by those same followees. A compromised followee holding ≥ 50% of the followee votes can immediately vote NO on any such proposal, including the proposal to remove it, creating a permanent deadlock.

**Concrete scenario (2-of-2 setup)**:
1. Neuron N has controller key destroyed; followees = [A, B] on `NeuronManagement` topic.
2. Followee A is compromised.
3. Followee B submits a `ManageNeuron` proposal (action: `Follow`, new followees = [B]).
4. A immediately votes NO. With total=2, no=1: `1 >= 2-1` → true → proposal immediately rejected.
5. Every subsequent attempt is rejected the same way. Neuron N is permanently unmanageable.

The `neuron_managers` helper confirms the followee list is the sole source of management authority when the controller key is gone: [5](#0-4) 

### Impact Explanation

The target neuron becomes permanently unmanageable. `ManageNeuron` proposals can execute nearly any command on the target neuron — disburse, split, change dissolve state, add/remove hotkeys, register votes — so the neuron's staked ICP and maturity are permanently frozen. Any organization that relied on this neuron for governance participation or treasury management loses all access with no recovery path.

### Likelihood Explanation

The scenario requires two conditions: (1) the neuron's controller key has been destroyed, and (2) a followee holding ≥ 50% of the followee votes is compromised. Condition (1) is an explicitly documented and encouraged use case for high-value neurons. Condition (2) is realistic for small followee sets (2-of-2 or 2-of-3 where 2 are compromised), which are the natural choice for team-managed neurons. The attack requires no special privileges — any principal controlling a followee neuron can execute it by calling `manage_neuron` with `RegisterVote: No` on any pending `ManageNeuron` proposal.

### Recommendation

Allow the neuron's controller (or a designated emergency principal) to change `NeuronManagement` followees directly without requiring a `ManageNeuron` proposal, even when the controller key is technically "destroyed" (e.g., by requiring a time-delayed override path that bypasses the followee vote). Alternatively, require a supermajority (e.g., > 2/3) for `ManageNeuron` proposals that modify the `NeuronManagement` followee list itself, so that a single compromised minority followee cannot block removal. At minimum, document that destroying the controller key while using a small followee set creates an irrecoverable single-point-of-failure.

### Proof of Concept

**Setup**: Neuron N with controller key destroyed; followees on `NeuronManagement` = [A, B].

**Step 1**: Followee B submits a `ManageNeuron` proposal to change followees to [B]:
```
manage_neuron(
  neuron_id = B,
  command = MakeProposal(
    action = ManageNeuron(
      target = N,
      command = Follow(topic = NeuronManagement, followees = [B])
    )
  )
)
```

**Step 2**: `compute_ballots_for_manage_neuron_proposal` creates ballots: `{A: {vote: Unspecified, power: 1}, B: {vote: Yes, power: 1}}` (B auto-votes YES as proposer). [6](#0-5) 

**Step 3**: Compromised A calls `manage_neuron(neuron_id=A, command=RegisterVote(proposal=P, vote=No))`.

**Step 4**: `can_make_decision` evaluates: `tally = {yes:1, no:1, total:2}`. Condition `no >= total - no` → `1 >= 1` → true → proposal immediately rejected. [7](#0-6) 

**Step 5**: B repeats Step 1 indefinitely; A repeats Step 3 each time. Neuron N is permanently locked. No recovery path exists because the controller key is gone and the only mechanism to change followees (a `ManageNeuron` proposal) is controlled by the same compromised followee set.
