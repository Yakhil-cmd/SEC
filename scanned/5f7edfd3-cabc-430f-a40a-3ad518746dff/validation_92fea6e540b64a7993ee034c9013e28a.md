### Title
Unprivileged Caller Can Panic `icrc21_canister_call_consent_message` via Out-of-Range `utc_offset_minutes` in `add_expiration` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

`ConsentMessage::add_expiration` calls `time::UtcOffset::from_whole_seconds(...).expect("Invalid offset")` with an attacker-controlled value derived from `utc_offset_minutes: Option<i16>`. Any value whose absolute magnitude exceeds 1439 minutes (e.g., `i16::MAX = 32767`) produces a seconds value outside the library's accepted range `−86_399..=86_399`, causing `.expect()` to panic and trap the update call. No validation of the offset is performed anywhere in the call chain before this point.

---

### Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs`, the `add_expiration` method for the `GenericDisplayMessage` variant computes a UTC offset from the caller-supplied `utc_offset_minutes`:

```rust
let offset = time::UtcOffset::from_whole_seconds(
    (utc_offset_minutes.unwrap_or(0) * 60).into(),
)
.expect("Invalid offset");   // panics if |seconds| > 86_399
``` [1](#0-0) 

`i16::MAX * 60 = 1_966_020`, which is far outside `−86_399..=86_399`. The `time` crate returns `Err(...)` for out-of-range values, and `.expect()` converts that into a Rust panic, which on the IC becomes a canister trap.

The `utc_offset_minutes` value flows from the ingress message through `build_icrc21_consent_info` with no range check:

```rust
if let Some(offset) = consent_msg_request.user_preferences.metadata.utc_offset_minutes {
    display_message_builder = display_message_builder.with_utc_offset_minutes(offset);
}
``` [2](#0-1) 

`add_expiration` is called unconditionally in the `Icrc21Function::Approve` branch of `ConsentMessageBuilder::build`: [3](#0-2) 

The panic path is only reachable when:
1. `device_spec` is `None` or `GenericDisplay` (the `FieldsDisplayMessage` branch ignores `utc_offset_minutes` entirely), **and**
2. `expires_at` is `Some(...)` in the decoded `ApproveArgs` (the offset computation is inside a `.map()` closure on `expires_at`). [4](#0-3) 

Both conditions are trivially satisfied by an attacker-crafted request.

---

### Impact Explanation

On the Internet Computer, a canister trap in an update call rolls back state and returns an error to the caller — it does **not** permanently disable the endpoint for other callers. The "permanent DoS for all users" framing in the question is incorrect; subsequent calls with valid inputs succeed normally.

The real impacts are:

1. **ICRC-21 spec violation**: The spec requires `icrc21_canister_call_consent_message` to return `Icrc21Error` variants on all invalid inputs, never trap. This invariant is broken.
2. **Cycles drain**: Each trapped update call still consumes cycles from the canister. An attacker can spam crafted requests to drain the canister's cycle balance, eventually causing it to be uninstalled. For ledger canisters holding ICP/ckBTC/ckETH this is a meaningful concern.
3. **Legitimate user breakage**: Any wallet or signer that passes a `utc_offset_minutes` value outside ±1439 (the full i16 range permits this) will receive a trap instead of a graceful error.

---

### Likelihood Explanation

The attack requires no privileges, no keys, and no coordination. A single ingress message with a crafted `ConsentMessageRequest` is sufficient. The Candid interface accepts `opt int16` for `utc_offset_minutes`, so the value `32767` is a valid wire-level input. The exploit is deterministic and locally testable.

---

### Recommendation

Replace `.expect("Invalid offset")` with proper error handling that returns an `Icrc21Error`:

```rust
let offset = time::UtcOffset::from_whole_seconds(
    (utc_offset_minutes.unwrap_or(0) as i32) * 60,
)
.map_err(|_| /* return a fallback string or propagate Icrc21Error */)?;
```

Since `add_expiration` currently returns `()`, the signature should be changed to `Result<(), Icrc21Error>` (or the offset computation should clamp/default on error). Alternatively, validate `utc_offset_minutes` at the entry point in `build_icrc21_consent_info` before it reaches the builder, rejecting values outside `−1439..=1439`.

---

### Proof of Concept

```rust
#[test]
fn add_expiration_does_not_panic_on_extreme_offset() {
    let mut msg = ConsentMessage::GenericDisplayMessage(String::new());
    // Should return gracefully, not panic
    msg.add_expiration(Some(u64::MAX), Some(i16::MAX));
}
```

This unit test will panic (trap) with the current code, demonstrating the bug. A state-machine test sending:

```
icrc21_canister_call_consent_message({
    method: "icrc2_approve",
    arg: Encode!(ApproveArgs { expires_at: Some(1_000_000_000_000_000_000u64), amount: 0, ..Default }),
    user_preferences: { metadata: { utc_offset_minutes: Some(32767) }, device_spec: None }
})
```

from `Principal::anonymous()` will produce a canister trap instead of an `Icrc21Error` response.
