### Title
SNS `neuron_minimum_stake_e8s` Governance Increase Retroactively Blocks Refresh and Split on Previously-Valid Neurons — (`rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS `NervousSystemParameters.neuron_minimum_stake_e8s` is a governance-changeable parameter. When a `ManageNervousSystemParameters` proposal increases this value, neurons that were previously valid (stake ≥ old minimum) but now fall below the new minimum are retroactively blocked from `refresh_neuron` and `split_neuron`. The root cause is structurally identical to the GMX M-4 bug: a changeable minimum requirement is applied retroactively to existing positions without a graceful exit path. Unlike GMX where liquidation actively seizes funds, here the user's tokens are locked until the dissolve delay expires, but the governance-parameter-change-as-retroactive-harm pattern is the same.

---

### Finding Description

`NervousSystemParameters.neuron_minimum_stake_e8s` is explicitly changeable via a `ManageNervousSystemParameters` proposal: [1](#0-0) 

The proto comment even acknowledges the parameter affects existing neurons but only discusses the *decrease* direction: [2](#0-1) 

When the parameter is **increased**, two operations immediately break for neurons whose stake falls between old_min and new_min:

**1. `refresh_neuron` — blocked by new minimum:** [3](#0-2) 

The check compares the ledger balance against the *current* (post-increase) `neuron_minimum_stake_e8s`. A neuron that was valid before the governance change is now permanently rejected from refreshing.

**2. `split_neuron` — blocked by new minimum:** [4](#0-3) 

A neuron with stake between old_min and new_min cannot split because `stake_e8s() < min_stake + split.amount_e8s` for any valid split amount.

The parameter update path itself has no guard against retroactive harm: [5](#0-4) 

`perform_manage_nervous_system_parameters` only validates that the *new* parameters are internally consistent — it does not check whether existing neurons would be stranded.

The same pattern exists in NNS governance: `neuron_minimum_stake_e8s` in `NetworkEconomics` is changeable via `ManageNetworkEconomics`, and `refresh_neuron` applies the same check: [6](#0-5) 

Crucially, `disburse_neuron` in SNS governance does **not** check minimum stake — it only requires the neuron to be dissolved: [7](#0-6) 

This means funds are not permanently lost, but the user is locked out of all neuron management operations until the dissolve delay expires.

---

### Impact Explanation

For any SNS neuron with stake S where `old_min ≤ S < new_min` after a `ManageNervousSystemParameters` proposal increases `neuron_minimum_stake_e8s`:

- **`refresh_neuron` is permanently blocked** — the user cannot top up the cached stake even if they transfer additional tokens to the subaccount (unless they bring the ledger balance above `new_min`).
- **`split_neuron` is permanently blocked** — the neuron cannot be split into two neurons.
- **Funds are locked until dissolve delay expires** — `disburse_neuron` requires `NeuronState::Dissolved`. If the neuron has a long dissolve delay or is in a vesting period, the user's tokens are inaccessible for that entire duration.
- **Vesting neurons are worst-case** — SNS neurons can have `vesting_period_seconds`; during vesting the neuron cannot start dissolving, compounding the lock-up.

The user was fully compliant before the governance change. The parameter change retroactively penalizes them with no recourse other than waiting.

---

### Likelihood Explanation

Any SNS community with a governance majority can pass a `ManageNervousSystemParameters` proposal to increase `neuron_minimum_stake_e8s` for legitimate reasons (anti-spam, economic rebalancing, token price changes). The entry path is a standard unprivileged governance ingress call — no admin key or malicious majority is required; a well-intentioned majority suffices. The likelihood of this occurring accidentally is low but non-zero, and the affected population is every neuron with stake between old_min and new_min at the time of the change.

---

### Recommendation

1. In `refresh_neuron` and `split_neuron`, skip the minimum-stake check for neurons that already exist in governance state (i.e., were created before the parameter change). The minimum-stake check should only apply at neuron *creation* time.
2. Alternatively, when executing a `ManageNervousSystemParameters` proposal that increases `neuron_minimum_stake_e8s`, emit a warning or reject if any existing neuron would fall below the new minimum.
3. At minimum, correct the proto comment to explicitly state that *increasing* `neuron_minimum_stake_e8s` retroactively affects existing neurons, so SNS communities are aware of the consequence before voting.

---

### Proof of Concept

1. SNS is initialized with `neuron_minimum_stake_e8s = 100_000_000` (1 token).
2. User stakes exactly 1 token and claims a neuron with `dissolve_delay = 1 year`.
3. SNS governance passes `ManageNervousSystemParameters { neuron_minimum_stake_e8s: Some(200_000_000) }`.
4. User calls `ClaimOrRefresh` → `refresh_neuron` → fails: *"Account does not have enough funds to refresh a neuron. Please make sure that account has at least 200000000 e8s (was 100000000 e8s)"* — blocked at [8](#0-7) 
5. User calls `Split { amount_e8s: 1 }` → `split_neuron` → fails: *"minimum allowed stake"* — blocked at [4](#0-3) 
6. User calls `Disburse` → fails: *"Neuron is NOT dissolved. It is in state NotDissolving"* — blocked at [9](#0-8) 
7. User must wait 1 full year for the dissolve delay to expire before recovering their tokens — with no ability to manage the neuron in the interim.

    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }
