### Title
Any Unprivileged Caller Can Force-Claim a Neuron for Any Principal - (`rs/nns/governance/src/governance.rs` and `rs/sns/governance/src/governance.rs`)

### Summary

Both NNS Governance and SNS Governance allow any unprivileged ingress sender to force-claim (stake) a neuron on behalf of any arbitrary principal by supplying that principal's ID in the `controller` field of `MemoAndController`. This is the direct IC analog of STK-2: a staking action can be performed for any address without that address's consent.

### Finding Description

The `ClaimOrRefresh` command in both NNS and SNS Governance accepts a `MemoAndController` struct with an optional `controller` field. When `controller` is set to an arbitrary `PrincipalId`, the governance canister computes the staking subaccount as `hash(controller, memo)`, checks the ledger balance of that subaccount, and if sufficient, creates a neuron owned by that `controller` — regardless of who the actual caller is.

**NNS Governance** (`rs/nns/governance/src/governance.rs`):

```rust
async fn claim_or_refresh_neuron_by_memo_and_controller(
    &mut self,
    caller: &PrincipalId,
    memo_and_controller: MemoAndController,
    claim_or_refresh: &ClaimOrRefresh,
) -> Result<NeuronId, GovernanceError> {
    let controller = memo_and_controller.controller.unwrap_or(*caller);
    // ...
    None => {
        self.claim_neuron(subaccount, controller, claim_or_refresh).await
    }
}

### Impact Explanation

**NNS (higher severity):** Any unprivileged ingress sender can force-claim a neuron for any principal P, provided tokens are sitting in the governance subaccount `hash(P, memo)`. The neuron is created in a `NotDissolving` state with `INITIAL_NEURON_DISSOLVE_DELAY`. The victim's tokens become locked in a neuron they did not consent to create at that time. The attacker gains no control over the neuron, but:

1. **Canister-based staking disruption**: A canister (e.g., a DeFi protocol, DAO treasury, or liquid staking protocol) that sends ICP to a governance subaccount as part of a multi-step staking workflow can have its neuron force-claimed before the canister is ready to proceed. The canister's internal state machine may not expect the neuron to exist yet, breaking its logic.
2. **Forced lockup**: A user who sent ICP to a governance subaccount intending to claim later (e.g., waiting for a governance parameter change, or timing their dissolve delay) has their tokens locked immediately with the initial dissolve delay.
3. **Interaction with STK-1 analog**: If a warmup period or other time-sensitive governance mechanism is added, force-claiming disrupts the intended timing.

**SNS (lower severity):** Same forced-claim behavior, but the initial dissolve delay is 0, so the victim can immediately dissolve and disburse. The disruption to canister-based SNS staking workflows is still real.

### Likelihood Explanation

**High.** The attack requires only:
1. Knowledge of the victim's principal ID (public information on-chain).
2. Knowledge of the memo/nonce used in the staking transfer (observable on the ledger).
3. A single `manage_neuron` ingress call with `ClaimOrRefresh { by: MemoAndController { memo, controller: Some(victim) } }`.

No privileged access, no key material, and no threshold corruption is required. The attacker only needs to observe a pending staking transfer on the ledger and race to claim before the victim does. This is particularly easy to automate against canister-based staking systems whose staking subaccounts and memos are deterministic and publicly computable.

### Recommendation

Require that when `controller` is explicitly set in `MemoAndController`, the caller must equal the specified controller:

```rust
if let Some(controller) = memo_and_controller.controller {
    if controller != *caller {
        return Err(GovernanceError::new(ErrorType::NotAuthorized));
    }
}
let controller = memo_and_controller.controller.unwrap_or(*caller);
```

If the proxy-claim feature is intentionally required (e.g., for hardware wallet flows), it should be carefully documented and its interaction with any future warmup period or time-sensitive governance mechanisms should be explicitly analyzed. The SNS governance already demonstrates that the neuron is correctly assigned to the specified controller, so the fix does not break the ownership model — it only restricts who can trigger the claim.

### Proof of Concept

**Entry path (NNS):**

```
Attacker (any principal) → NNS Governance canister (manage_neuron)
  ManageNeuron {
    command: ClaimOrRefresh {
      by: MemoAndController {
        memo: <victim_memo>,           // observable on ledger
        controller: Some(<victim_principal>)  // public information
      }
    }
  }
```

**Step-by-step:**

1. Victim canister C sends 10 ICP to `governance_subaccount(C.principal, nonce=42)` as step 1 of a multi-step staking workflow.
2. Attacker observes the ledger transfer (public).
3. Attacker immediately calls `manage_neuron` on NNS Governance with `MemoAndController { memo: 42, controller: Some(C.principal) }`.
4. `claim_or_refresh_neuron_by_memo_and_controller` resolves `controller = C.principal`, computes the same subaccount, finds the balance ≥ minimum stake, and calls `claim_neuron(subaccount, C.principal, ...)`.
5. A neuron is created for C with `INITIAL_NEURON_DISSOLVE_DELAY` in `NotDissolving` state.
6. When C's canister later tries to claim the neuron (step 2 of its workflow), it finds the neuron already exists. Depending on C's internal logic, this may cause a panic, incorrect state, or loss of the staking operation.

The existing test `test_claim_neuron_memo_and_controller_by_proxy` in `rs/nns/governance/tests/governance.rs` (lines 4822–4826) confirms this succeeds without error. [8](#0-7) [9](#0-8) [10](#0-9)
