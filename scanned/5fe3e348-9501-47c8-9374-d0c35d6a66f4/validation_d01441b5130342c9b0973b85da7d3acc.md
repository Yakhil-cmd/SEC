### Title
Stale Token-Ratio in SNS Treasury Manager Deposit Proposal Causes Pool Imbalance and Treasury Loss - (File: rs/sns/governance/src/extensions.rs)

### Summary

The SNS governance `ExecuteExtensionOperation` deposit flow fixes `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s` at proposal-creation time. Because SNS proposals have a multi-day voting period, the DEX pool's price ratio can shift substantially before execution. When the proposal finally executes, `execute_treasury_manager_deposit` forwards the original, now-stale amounts to the treasury manager without any on-chain re-validation of the current pool ratio. The codebase itself acknowledges this risk in two separate places but provides no mitigation at the execution layer.

### Finding Description

The deposit flow has three distinct phases:

**Phase 1 – Proposal creation / validation**
`validate_deposit_operation_impl` reads the current treasury balances and rejects requests exceeding 50 % of each balance, but it does **not** check whether the SNS/ICP ratio in the proposal matches the current DEX pool ratio. [1](#0-0) 

The validated struct simply stores the caller-supplied amounts verbatim: [2](#0-1) 

**Phase 2 – Voting period (days)**
During the voting period, arbitrage bots or normal market activity can move the DEX pool price significantly. No on-chain mechanism freezes or re-checks the ratio.

**Phase 3 – Proposal execution**
`execute_treasury_manager_deposit` uses the frozen `treasury_allocation_sns_e8s` / `treasury_allocation_icp_e8s` from the proposal directly, approves those exact amounts via ICRC-2, and calls `deposit` on the treasury manager canister — all without re-querying the current pool price: [3](#0-2) 

The ICRC-2 approval grants the treasury manager canister the right to pull exactly those stale amounts: [4](#0-3) 

The `DepositRequest` carries the stale allowances straight to the DEX: [5](#0-4) 

The codebase explicitly acknowledges the risk in the Treasury Manager interface spec: [6](#0-5) 

And again in the proposal-rendering path: [7](#0-6) 

Both acknowledgements are documentation-only; neither triggers any protective logic at execution time.

### Impact Explanation

When the DEX pool price has moved between proposal submission and execution, the treasury manager deposits tokens at the stale ratio. The DEX returns the excess of whichever token is over-represented, but the LP position is minted at the disadvantaged ratio. The SNS treasury effectively sells one token at below-market price, suffering a direct, irreversible financial loss proportional to the price drift and the deposit size (up to 50 % of each treasury balance per proposal). Because the `DepositRequest` carries no slippage bound, the loss is unbounded within that 50 % cap.

### Likelihood Explanation

SNS governance voting periods are typically several days. DEX prices for SNS/ICP pairs are volatile and actively arbitraged. A motivated attacker can sandwich the proposal execution: manipulate the pool price just before the proposal executes (e.g., via a large swap), let the governance canister deposit at the skewed ratio, then reverse the manipulation to extract the arbitrage profit. No privileged access is required; the attacker only needs to be able to trade on the DEX and observe the IC mempool / proposal queue. The attack is fully permissionless.

### Recommendation

1. **Re-validate the ratio at execution time.** Before calling `approve_treasury_manager`, query the current DEX pool reserves and verify that the proposal's SNS/ICP ratio is within an acceptable slippage tolerance (e.g., ±1 %) of the live ratio. Reject execution if the drift exceeds the threshold.
2. **Encode a slippage bound in the proposal.** Extend `ValidatedDepositOperationArg` with a `max_slippage_bps: u64` field that voters explicitly approve. The execution path must enforce this bound on-chain.
3. **Compute the deposit amounts on-chain at execution time** rather than freezing them at proposal creation, analogous to the remediation described in the referenced report.

### Proof of Concept

1. Attacker observes a pending SNS `ExecuteExtensionOperation` deposit proposal with `treasury_allocation_sns_e8s = X` and `treasury_allocation_icp_e8s = Y`, reflecting the pool ratio at proposal creation.
2. Attacker waits until the proposal reaches the execution block.
3. Immediately before execution, attacker submits a large swap on the DEX that moves the SNS/ICP price significantly (e.g., dumps SNS tokens, making SNS cheap relative to ICP).
4. The governance canister executes `execute_treasury_manager_deposit` with the original `X` SNS and `Y` ICP. Because the pool now has excess SNS, the DEX accepts all `Y` ICP but only a fraction of `X` SNS at the new ratio; the remainder of SNS is returned. The LP position is minted at the skewed ratio.
5. Attacker reverses the swap, profiting from the arbitrage. The SNS treasury's LP position is worth less than the deposited tokens' fair-market value. [8](#0-7) [6](#0-5)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

                sns_amount_e8s,
                Some(expiry_time_nsec),

                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,

                )

async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)

/// Validated deposit operation arguments
#[derive(Debug, Clone)]
pub struct ValidatedDepositOperationArg {
    /// Amount of SNS tokens to allocate from treasury
    pub treasury_allocation_sns_e8s: u64,
    /// Amount of ICP tokens to allocate from treasury
    pub treasury_allocation_icp_e8s: u64,
    /// Original Precise value with all fields
    pub original: Precise,
}

impl TryFrom<Option<Precise>> for ValidatedDepositOperationArg {
    type Error = String;

    fn try_from(value: Option<Precise>) -> Result<Self, Self::Error> {
        let Some(original) = value else {
            return Err("Deposit operation arguments must be provided".to_string());
        };

        let map = match &original.value {
            Some(precise::Value::Map(PreciseMap { map })) => map,
            _ => return Err("Deposit operation arguments must be a PreciseMap".to_string()),
        };

        let treasury_allocation_sns_e8s = map
            .get("treasury_allocation_sns_e8s")
            .and_then(|p| match &p.value {
                Some(precise::Value::Nat(n)) => Some(*n),
                _ => None,
            })
            .ok_or_else(|| "treasury_allocation_sns_e8s must be a Nat value".to_string())?;

        let treasury_allocation_icp_e8s = map
            .get("treasury_allocation_icp_e8s")
            .and_then(|p| match &p.value {
                Some(precise::Value::Nat(n)) => Some(*n),
                _ => None,
            })
            .ok_or_else(|| "treasury_allocation_icp_e8s must be a Nat value".to_string())?;

        Ok(Self {
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
            original,
        })
    }
