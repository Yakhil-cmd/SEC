### Title
Signed Integer Division Truncates Toward Zero in `compute_maturity_modulation`, Causing Maturity Modulation to Be Slightly Less Negative Than Correct - (File: `rs/nns/cmc/src/main.rs`)

### Summary

The `compute_maturity_modulation` function in the Cycles Minting Canister (CMC) averages four signed rate terms using Rust's integer division operator `/`, which truncates toward zero. When the sum of the four terms is negative (ICP price has dropped), the result is slightly less negative than the mathematically correct floor value. This causes the stored `maturity_modulation_permyriad` to be slightly less negative than it should be, so neuron holders receive slightly more ICP than they should when disbursing maturity during a price-drop period.

### Finding Description

In `rs/nns/cmc/src/main.rs`, `compute_maturity_modulation` computes:

```rust
fn compute_maturity_modulation(rates: &[IcpXdrConversionRate], time_s: u64) -> i32 {
    let day = time_s / 86_400;
    let rate1 = compute_capped_maturity_modulation(rates, day - 7, day);
    let rate2 = compute_capped_maturity_modulation(rates, day - 14, day - 7);
    let rate3 = compute_capped_maturity_modulation(rates, day - 21, day - 14);
    let rate4 = compute_capped_maturity_modulation(rates, day - 28, day - 21);
    (rate1 + rate2 + rate3 + rate4) / 4   // <-- signed integer division, truncates toward zero
}
``` [1](#0-0) 

Rust's `/` on signed integers truncates toward zero (not toward negative infinity). When the sum of the four terms is negative — which happens when the ICP price has declined — the result is rounded toward zero (i.e., made less negative). For example, a sum of `-7` yields `-1` instead of the mathematically correct `-2`.

The same truncation-toward-zero pattern appears inside `compute_capped_maturity_modulation` at the inner `checked_div`:

```rust
match difference_permyriad.checked_div(start_rate_value) {
    Some(relative_change_permyriad) => relative_change_permyriad.clamp(...)
``` [2](#0-1) 

The result is stored in `state.maturity_modulation_permyriad` and served via the `neuron_maturity_modulation()` query endpoint, which SNS Governance (and NNS Governance via its CMC polling path) consumes to apply modulation to maturity disbursements. [3](#0-2) 

The downstream consumer, `apply_maturity_modulation`, then computes:

```rust
let modulated_amount_e8s: u128 = amount_e8s
    .checked_mul(adjusted_maturity_modulation_basis_points)  // 10_000 + modulation
    .checked_div(BASIS_POINTS_PER_UNITY)                     // / 10_000
``` [4](#0-3) 

A maturity modulation that is 1 permyriad less negative than it should be causes `adjusted_maturity_modulation_basis_points` to be 1 unit higher than correct, so the disbursed amount is `amount_e8s / 10_000` e8s more than it should be per disbursement.

### Impact Explanation

When the ICP price has dropped, the maturity modulation should be negative (users receive less ICP on disbursement). Due to truncation toward zero, the stored modulation is slightly less negative than the correct value. Every neuron holder who disburses maturity during a price-drop period receives a fractionally larger ICP mint than the protocol intends. The error is at most `3/4` permyriad per computation cycle (≤ 0.0075%), which is dust-level per individual disbursement but accumulates across all disbursements system-wide. This is a ledger conservation bug: the protocol mints slightly more ICP than it should.

### Likelihood Explanation

The condition (ICP price declining) is common and recurring. The rounding error fires on every CMC heartbeat that updates the rate during a declining-price period. Any neuron holder who calls `disburse_maturity` or triggers spawning during such a period benefits from the rounding error. The entry path is fully unprivileged: any principal holding a neuron can call `manage_neuron { DisburseMaturity }` or `manage_neuron { Spawn }`.

### Recommendation

Replace the final averaging division with a floor-toward-negative-infinity division to ensure the modulation is never rounded in the user's favor when negative:

```rust
// Instead of:
(rate1 + rate2 + rate3 + rate4) / 4

// Use floor division (rounds toward -∞):
let sum = rate1 + rate2 + rate3 + rate4;
if sum >= 0 { sum / 4 } else { (sum - 3) / 4 }
```

Apply the same correction inside `compute_capped_maturity_modulation` where `difference_permyriad.checked_div(start_rate_value)` is called with a potentially negative numerator.

### Proof of Concept

Concrete numeric example:

- Suppose `rate1 = -3`, `rate2 = -2`, `rate3 = -1`, `rate4 = -1` (all negative, ICP price dropped in each 7-day window).
- Sum = `-7`.
- Rust: `-7_i32 / 4 == -1` (truncates toward zero).
- Correct floor: `floor(-7/4) = floor(-1.75) = -2`.
- Stored modulation: `-1` permyriad instead of `-2` permyriad.
- For a disbursement of `1_000_000_000` e8s (10 ICP), `apply_maturity_modulation` computes:
  - With `-1`: `1_000_000_000 * 9_999 / 10_000 = 999_900_000` e8s minted.
  - With `-2` (correct): `1_000_000_000 * 9_998 / 10_000 = 999_800_000` e8s minted.
  - Excess minted: `100_000` e8s (0.001 ICP) per 10 ICP disbursed.

The attacker-controlled entry path: any principal calls `manage_neuron { DisburseMaturity { percentage_to_disburse: 100, ... } }` on NNS or SNS Governance during a period when the CMC has computed a slightly-less-negative modulation due to this rounding error. The protocol mints the excess ICP unconditionally.
