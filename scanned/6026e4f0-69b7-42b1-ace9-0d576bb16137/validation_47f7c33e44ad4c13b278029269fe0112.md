### Title
Unauthorized Caller Can Bypass SNS Swap Confirmation Text and Force Participation for Arbitrary Buyers via `refresh_buyer_tokens` - (File: `rs/sns/swap/canister/canister.rs`)

---

### Summary

The `refresh_buyer_tokens` update endpoint in the SNS Swap canister accepts a caller-supplied `buyer` principal with no check that the caller equals the specified buyer. Because the SNS confirmation text is publicly readable from canister state, any unprivileged ingress sender can call `refresh_buyer_tokens` on behalf of any victim who has already transferred ICP to the swap subaccount, bypassing the confirmation-text gate and forcing the victim's participation to be committed and their open ticket to be deleted — without the victim's explicit agreement.

---

### Finding Description

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` endpoint resolves the effective buyer principal as follows:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
``` [1](#0-0) 

There is no assertion that `caller == p` when `arg.buyer` is non-empty. The resolved principal `p` is then passed directly into `refresh_buyer_token_e8s`, which:

1. Reads the ICP ledger balance of `p`'s subaccount on the swap canister.
2. Validates and consumes the `confirmation_text` argument — the text the SNS project set to require explicit participant agreement.
3. Updates `self.buyers[p]` with the new participation amount.
4. **Deletes `p`'s open ticket** from `OPEN_TICKETS_MEMORY`. [2](#0-1) [3](#0-2) 

The confirmation text is set during SNS initialization and is publicly readable via `get_init`. The proto comment for `RefreshBuyerTokensRequest.buyer` states "If not specified, the caller is used," implying third-party calls are accepted by design, but no guard prevents abuse of the confirmation-text mechanism. [4](#0-3) 

---

### Impact Explanation

An SNS project may set a `confirmation_text` (e.g., a legal disclaimer or risk warning) to ensure every participant explicitly agrees before their ICP is committed. Because `refresh_buyer_tokens` accepts an arbitrary `buyer` without a caller check:

- **Confirmation-text bypass**: An attacker who knows the public confirmation text (trivially obtained from `get_init`) can call `refresh_buyer_tokens(buyer=victim, confirmation_text=<public_text>)` for any victim who has already transferred ICP to the swap subaccount. The victim's participation is registered and their ICP is committed to the swap without the victim ever having agreed to the confirmation text.
- **Forced ticket deletion**: The victim's open ticket is deleted, removing their ability to cancel or adjust participation through the normal ticket-based payment flow.
- **Irreversible commitment**: Once participation is registered, the victim cannot withdraw. If the swap commits, the victim receives SNS tokens they never explicitly agreed to purchase.

---

### Likelihood Explanation

The attack requires only that the victim has transferred ICP to their swap subaccount (a normal step in the participation flow, done before calling `refresh_buyer_tokens`). The confirmation text is publicly readable. No privileged access, key material, or network-level attack is needed. Any unprivileged ingress sender can execute this against any in-progress buyer.

---

### Recommendation

Add a caller-identity check inside `refresh_buyer_tokens` when a non-empty `buyer` is specified:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    // Only allow a third-party caller if no confirmation text is required.
    if swap().init_or_panic().confirmation_text.is_some()
        && specified != caller_principal_id()
    {
        panic!("Caller must match buyer when a confirmation text is required");
    }
    specified
};
```

Alternatively, restrict the `buyer` override to the buyer themselves unconditionally, matching the security model of the confirmation-text feature.

---

### Proof of Concept

1. An SNS is deployed with `confirmation_text = "I accept the risk"` (readable via `get_init`).
2. Victim calls `new_sale_ticket(amount_icp_e8s = X)` — a ticket is created.
3. Victim transfers `X` ICP to `swap_canister_subaccount(victim_principal)` on the ICP ledger, intending to review the confirmation text before committing.
4. **Attacker** (any principal) calls:
   ```
   refresh_buyer_tokens({
     buyer: "<victim_principal>",
     confirmation_text: Some("I accept the risk")
   })
   ``` [1](#0-0) 
5. `refresh_buyer_token_e8s` reads the victim's ICP balance, validates the (attacker-supplied) confirmation text, registers the victim's participation, and deletes the victim's open ticket. [5](#0-4) 
6. The victim's ICP is now committed to the swap. The victim never explicitly agreed to the confirmation text. If the swap commits, the victim receives SNS tokens without consent.

        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        self.validate_confirmation_text(confirmation_text)?;

            };

                .account_balance(account)
