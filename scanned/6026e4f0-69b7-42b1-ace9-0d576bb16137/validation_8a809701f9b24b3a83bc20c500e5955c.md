### Title
Paused DTS Execution Acts as Unintended Grace Period, Delaying Canister Uninstallation on Cycles Exhaustion - (File: rs/execution_environment/src/scheduler.rs)

---

### Summary

The IC scheduler explicitly skips resource-allocation charging for any canister that has a paused (Deterministic Time Slicing) execution. This creates a bounded but real "grace period" during which a canister that has exhausted its cycles balance is not uninstalled, continues to hold subnet memory and compute allocation, and blocks those resources from other canisters. The analog to the Scroll report is direct: the paused execution is the "sequencer grace period," and the delayed uninstallation is the delayed "liquidation."

---

### Finding Description

In `charge_canisters_for_resource_allocation_and_usage`, the scheduler iterates over all canisters and charges them for memory and compute usage. However, any canister with an active paused or aborted DTS execution is unconditionally skipped:

```rust
// Postpone charging for resources when a canister has a paused execution
// to avoid modifying the balance of a canister during an unfinished operation.
if canister.has_paused_execution_or_install_code() {
    return;
}
``` [1](#0-0) 

Critically, `time_of_last_allocation_charge` is **not** updated when a canister is skipped, so the accumulated unpaid duration grows silently. The code's own TODO acknowledges this is unresolved:

```rust
// TODO(DSM-103): Charge all canisters every N rounds / seconds (and otherwise
// do nothing). Ensure that paused execution canisters are charged eventually.
``` [2](#0-1) 

Paused executions are aborted at checkpoint rounds, after which charging is applied:

```rust
// Abort (some) paused executions.
self.finish_round(&mut final_state, current_round_type);

// Charge canisters after (some) paused executions were aborted.
self.charge_canisters_for_resource_allocation_and_usage(
    &mut final_state,
    current_round,
    current_round_type,
);
``` [3](#0-2) 

The comment in the charging function itself confirms the design intent and its limitation:

> "The latter ensures that (because we skip canisters with paused executions), every canister is charged at least once per checkpoint interval." [4](#0-3) 

The uninstallation path (the "liquidation") only fires when `charge_canister_for_resource_allocation_and_usage` returns `Err`, which cannot happen while the canister is skipped: [5](#0-4) 

The `freeze_threshold_cycles` function computes the minimum balance a canister must maintain; a canister below this threshold should be uninstalled, but the paused-execution skip prevents that check from running: [6](#0-5) 

---

### Impact Explanation

A canister developer can deploy a canister with a large memory allocation, fund it with cycles just sufficient to start a long DTS update execution, and allow the balance to fall below the freezing threshold during that execution. Because the scheduler skips charging for paused canisters, the canister:

1. Continues to hold allocated subnet memory and compute allocation without paying.
2. Cannot be uninstalled ("liquidated") until the paused execution is aborted at the next checkpoint round.
3. Keeps open call contexts alive, potentially blocking response delivery to other canisters awaiting replies.

The subnet's available memory and callback slots are finite resources. A canister holding them without paying for up to one full checkpoint interval degrades subnet capacity for all other users on that subnet. If multiple canisters coordinate this behavior, the effect compounds.

---

### Likelihood Explanation

The scenario requires a canister to simultaneously have a long DTS execution in progress and a cycles balance that drops below the freezing threshold during that execution. DTS long executions are a normal operational feature (any update method exceeding the per-slice instruction limit triggers them). A canister developer who intentionally sizes their initial balance to exhaust during a long execution can reliably trigger this window. The delay is bounded by the checkpoint interval (typically minutes), limiting the severity, but the root cause is a confirmed open TODO in production code.

---

### Recommendation

1. Track `time_of_last_allocation_charge` even for skipped canisters, so the accumulated duration is not silently deferred.
2. Implement a read-only solvency check (without modifying the balance) for paused canisters: if the canister's balance is already below the freezing threshold, abort the paused execution immediately and proceed with uninstallation rather than waiting for the next checkpoint round.
3. Resolve `TODO(DSM-103)` by charging paused canisters via the `ingress_induction_cycles_debit` postponed-charge mechanism already used for concurrent ingress induction, so the balance is debited without racing with the in-flight execution.

---

### Proof of Concept

1. Deploy a canister with `memory_allocation = 1 GiB` and `compute_allocation = 1%` on an application subnet.
2. Fund it with exactly enough cycles to cover the freezing threshold plus one DTS slice of execution cost, but not enough to cover the full resource-allocation charge for one checkpoint interval.
3. Send an ingress update message whose Wasm body requires more instructions than `max_instructions_per_slice`, triggering a paused DTS execution.
4. Observe via `canister_status` that the canister's balance falls below the freezing threshold mid-execution.
5. Observe that the canister is **not** uninstalled during the ordinary rounds following the balance drop.
6. Observe that uninstallation only occurs at the next checkpoint round, after `finish_round` aborts the paused execution and `charge_canisters_for_resource_allocation_and_usage` runs.

The test `dont_charge_allocations_for_paused_canisters` in the scheduler test suite directly confirms this behavior: [7](#0-6)

    /// taking the reserved balance into account.
    pub fn freeze_threshold_cycles(
        &self,
        freeze_threshold: NumSeconds,
        memory_allocation: MemoryAllocation,
        memory_usage: NumBytes,
        message_memory_usage: MessageMemoryUsage,
        compute_allocation: ComputeAllocation,
        subnet_cycles_config: CyclesAccountManagerSubnetConfig,
        reserved_balance: Cycles,
    ) -> Cycles {
        let idle_cycles_burned_rate = self.idle_cycles_burned_rate(
            memory_allocation,
            memory_usage,
            message_memory_usage,
            compute_allocation,
            subnet_cycles_config,
        );

        let threshold = idle_cycles_burned_rate * freeze_threshold.get() as u128 / SECONDS_PER_DAY;

        // Here we rely on the saturating subtraction for Cycles.
        threshold - reserved_balance
    }

#[test]
fn dont_charge_allocations_for_paused_canisters() {
    const T0: Time = Time::from_nanos_since_unix_epoch(1_000_000_000);
    const INITIAL_CYCLES: Cycles = Cycles::new(10_000_000);
    const MEMORY_ALLOCATION: NumBytes = NumBytes::new(1 << 30);

    let mut test = SchedulerTestBuilder::new().build();

    let mut create_canister_with_memory_allocation = || -> CanisterId {
        test.create_canister_with(
            INITIAL_CYCLES,
            ComputeAllocation::zero(),
            MemoryAllocation::from(MEMORY_ALLOCATION),
            None,
            Some(T0),
            None,
        )
    };
    let canister = create_canister_with_memory_allocation();
    let paused_canister = create_canister_with_memory_allocation();
    let paused_install_canister = create_canister_with_memory_allocation();

    test.canister_state_mut(paused_canister)
        .system_state
        .task_queue
        .enqueue(ExecutionTask::PausedExecution {
            id: PausedExecutionId(0),
            input: CanisterMessageOrTask::Task(CanisterTask::Heartbeat),
        });
    test.canister_state_mut(paused_install_canister)
        .system_state
        .task_queue
        .enqueue(ExecutionTask::PausedInstallCode(PausedExecutionId(0)));

    let duration_between_allocation_charges = test.duration_between_allocation_charges();
    test.set_time(T0 + duration_between_allocation_charges);

    test.charge_for_resource_allocations();

    fn assert_balance_change(test: &SchedulerTest, canister: CanisterId, duration: Duration) {
        assert_eq!(
            test.canister_state(canister).system_state.balance(),
            INITIAL_CYCLES
                - test.memory_cost(MEMORY_ALLOCATION, duration).real()
                - test.canister_base_cost(MEMORY_ALLOCATION, duration).real()
        );
    }
    // Balance has changed for the canister with no paused execution.
    assert_balance_change(&test, canister, duration_between_allocation_charges);
    // Balance has not changed for the canisters with paused execution/install code.
    assert_balance_change(&test, paused_canister, Duration::from_secs(0));
    assert_balance_change(&test, paused_install_canister, Duration::from_secs(0));

    // One second later, the two long executions complete.
    let duration_plus_one_second = duration_between_allocation_charges + Duration::from_secs(1);
    test.set_time(T0 + duration_plus_one_second);
    test.canister_state_mut(paused_canister)
        .system_state
        .task_queue
        .pop_front();
    test.canister_state_mut(paused_install_canister)
        .system_state
        .task_queue
        .pop_front();

    test.charge_for_resource_allocations();

    // The balance has not changed for the canister that was already charged.
    assert_balance_change(&test, canister, duration_between_allocation_charges);
    // The balance has changed for the canisters with paused execution/install code.
    assert_balance_change(&test, paused_canister, duration_plus_one_second);
    assert_balance_change(&test, paused_install_canister, duration_plus_one_second);
}
