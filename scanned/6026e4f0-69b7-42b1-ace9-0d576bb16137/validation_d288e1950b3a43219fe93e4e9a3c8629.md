### Title
Unbounded `ingress_expiries` Vec allocation via uncapped `ingress_end` in `construction_payloads` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

---

### Summary

The `construction_payloads` function accepts attacker-controlled `ingress_start` and `ingress_end` values from the HTTP request metadata. The only guard on `ingress_end` is a **lower-bound** check (line 154), with no upper-bound cap. Submitting `ingress_start=0, ingress_end=u64::MAX` passes all guards and causes the while loop at line 163 to iterate ~153 million times, pushing one entry per iteration into an unbounded `Vec`, exhausting process memory and crashing the Rosetta node.

---

### Finding Description

In `construction_payloads`:

**Step 1 — Attacker-controlled inputs pass deserialization with no bounds:**
`ConstructionPayloadsRequestMetadata.ingress_end` is a plain `Option<u64>` with no range validation at the type level. [1](#0-0) 

**Step 2 — Guard at line 148 passes trivially:**
```
ingress_start=0 >= ingress_end=u64::MAX  →  FALSE  →  no error
``` [2](#0-1) 

**Step 3 — Guard at line 154 also passes without any overflow:**
The check is `if ingress_end < now + ingress_interval`. With `ingress_end = u64::MAX ≈ 1.84e19` and `now ≈ 1.76e18` (nanoseconds since epoch in 2026), the expression `now + ingress_interval` is approximately `1.76e18 + 2.4e11`, which is far smaller than `u64::MAX`. The condition is `FALSE`, so no error is returned and execution falls through to the loop. **No overflow is required for this attack path.** [3](#0-2) 

**Step 4 — While loop runs ~153 million iterations:**
```
step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP)
     = (240s - 120s) in ns = 1.2e11 ns
iterations ≈ u64::MAX / 1.2e11 ≈ 153,000,000
```
Each iteration pushes one `u64` onto `ingress_expiries`, allocating ~1.2 GB of heap before OOM. [4](#0-3) 

The constants confirm `INGRESS_INTERVAL_OVERLAP = 120s`: [5](#0-4) 

---

### Impact Explanation

A single unauthenticated HTTP POST to `/construction/payloads` with `ingress_end` set to `u64::MAX` causes the Rosetta process to allocate ~1.2 GB in a tight loop and crash with OOM. This is a complete denial-of-service against the ICRC1 Rosetta node. The process does not recover without a restart.

---

### Likelihood Explanation

The `/construction/payloads` endpoint is a standard public Rosetta API endpoint. No authentication or privileged role is required. The payload is a simple JSON object. Any client that can reach the Rosetta HTTP port can trigger this with a single request.

---

### Recommendation

Add an explicit upper-bound check on `ingress_end` before the while loop. For example:

```rust
let max_ingress_end = now + MAX_ALLOWED_INGRESS_WINDOW; // e.g., 24h in ns
if ingress_end > max_ingress_end {
    return Err(Error::processing_construction_failed(&format!(
        "ingress_end {ingress_end} exceeds maximum allowed window"
    )));
}
```

Alternatively, cap the number of iterations in the while loop directly:

```rust
const MAX_INGRESS_EXPIRIES: usize = 1000;
while ingress_start < ingress_end {
    if ingress_expiries.len() >= MAX_INGRESS_EXPIRIES {
        return Err(Error::processing_construction_failed("Too many ingress intervals"));
    }
    ...
}
```

### Proof of Concept

```rust
#[test]
fn test_construction_payloads_oom_dos() {
    use std::time::{Duration, SystemTime, UNIX_EPOCH};
    // Simulate a normal server time (~2026)
    let now = UNIX_EPOCH + Duration::from_secs(1_750_000_000);
    let result = construction_payloads(
        vec![/* valid transfer operation */],
        Some(ConstructionPayloadsRequestMetadata {
            ingress_start: Some(0),
            ingress_end: Some(u64::MAX),
            created_at_time: None,
            memo: None,
        }),
        &Principal::anonymous(),
        vec![/* valid public key */],
        now,
    );
    // Should return Err, but instead loops ~153M times → OOM
    assert!(result.is_err());
}
```

The function will not return an error — it will exhaust memory before completing.
