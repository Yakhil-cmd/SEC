### Title
Missing Context Validation in `push_response` `SendTransactionResponse` Arm Allows Malicious Block Proposer to Inject Spurious `ConsensusResponse` — (`rs/replicated_state/src/bitcoin.rs`)

### Summary

The `SendTransactionResponse` arm of `push_response` unconditionally pushes a `ConsensusResponse` to `consensus_queue` without verifying the `callback_id` exists in `bitcoin_send_transaction_internal_contexts`. This is an asymmetric omission compared to the `GetSuccessorsResponse` arm, which enforces this check. Combined with a `validate_self_validating_payload_impl` that is effectively a no-op (only checks size), a single malicious block proposer can finalize a block containing a fabricated `SendTransactionResponse` with an arbitrary `callback_id`.

### Finding Description

**Bug 1 — Missing context lookup in `push_response`:**

In `rs/replicated_state/src/bitcoin.rs`, the `GetSuccessorsResponse` arm (lines 32–39) validates the `callback_id` against `bitcoin_get_successors_contexts` and returns `StateError::BitcoinNonMatchingResponse` if no match is found: [1](#0-0) 

The `SendTransactionResponse` arm (lines 64–75) performs **no such lookup**. It blindly constructs and enqueues a `ConsensusResponse` for any `callback_id`: [2](#0-1) 

The same omission exists in the `SendTransactionReject` arm (lines 90–101): [3](#0-2) 

**Bug 2 — `validate_self_validating_payload_impl` is a no-op:**

The validation function used by all nodes when notarizing a block proposal does not check `callback_id` values against the replicated state at all. It only checks whether the payload is empty or returns the byte size: [4](#0-3) 

This means a block containing a `SendTransactionResponse` with `callback_id = u64::MAX` (or any other fabricated value) passes validation on all honest nodes and gets notarized and finalized.

**Bug 3 — `consensus_queue` must be fully drained each round:**

In `rs/messaging/src/state_machine.rs`, after execution, there is a `fatal!` assertion that the `consensus_queue` must be empty: [5](#0-4) 

Every item injected into `consensus_queue` — including fabricated ones — must be processed by the execution scheduler.

### Impact Explanation

**Scenario A — Orphaned `callback_id` (no matching context):**
The execution environment calls `retrieve_context(callback_id)` which returns `None`. Depending on how the execution layer handles this (graceful drop vs. unrecoverable error), this could cause a replica crash (DoS) or a silent state divergence. The `fatal!` invariant at the end of the round creates pressure for all items to be consumed.

**Scenario B — Targeted `callback_id` (matching an existing pending context):**
The malicious proposer observes the monotonically increasing `next_callback_id` in `SubnetCallContextManager` (which is part of the certified state) and targets a live `bitcoin_send_transaction_internal_contexts` entry. The fabricated `ConsensusResponse` causes `retrieve_context` to remove the context and deliver a fake `EmptyBlob` reply to the bitcoin canister. The bitcoin canister treats the `send_transaction` as acknowledged and responds to the ckBTC minter, which may advance its internal state machine (e.g., marking a pending withdrawal as submitted).

The `send_transaction` response is an adapter-level acknowledgment, not a Bitcoin confirmation. However, premature removal of the context means the legitimate adapter response (when it arrives) will find no matching context and be silently dropped, permanently orphaning the real transaction's lifecycle tracking.

### Likelihood Explanation

- Any node in the Bitcoin subnet can become block proposer (round-robin / rank-based selection).
- A single malicious node below the fault threshold suffices — no collusion required.
- The fabricated payload passes `validate_self_validating_payload_impl` on all honest nodes because it performs no callback_id-to-state cross-check.
- The `callback_id` counter is observable from the certified state, making targeted attacks feasible.

### Recommendation

Add the same context lookup guard to the `SendTransactionResponse` and `SendTransactionReject` arms as exists in `GetSuccessorsResponse`:

```rust
BitcoinAdapterResponseWrapper::SendTransactionResponse(_) => {
    let callback_id = CallbackId::from(response.callback_id);
    state
        .metadata
        .subnet_call_context_manager
        .bitcoin_send_transaction_internal_contexts
        .get(&callback_id)
        .ok_or_else(|| StateError::BitcoinNonMatchingResponse {
            callback_id: callback_id.get(),
        })?;
    let payload = Payload::Data(EmptyBlob.encode());
    state.consensus_queue.push(ConsensusResponse::new(callback_id, payload));
    Ok(())
}
```

Additionally, `validate_self_validating_payload_impl` should validate that every `callback_id` in the payload corresponds to a pending context in the state at `validation_context.certified_height`, mirroring how `bitcoin_requests_iter` only iterates over known contexts during payload building. [6](#0-5) 

### Proof of Concept

```rust
// State-machine test sketch:
let mut state = ReplicatedState::new(SUBNET_ID, SubnetType::Application);
// No bitcoin_send_transaction_internal_contexts entries.
assert!(state.metadata.subnet_call_context_manager
    .bitcoin_send_transaction_internal_contexts.is_empty());

// Inject fabricated SendTransactionResponse with arbitrary callback_id.
let result = push_response(
    &mut state,
    BitcoinAdapterResponse {
        response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
            SendTransactionResponse {}
        ),
        callback_id: u64::MAX,
    },
);

// Bug: returns Ok(()) instead of Err(StateError::BitcoinNonMatchingResponse)
assert!(result.is_ok());

// Bug: consensus_queue now contains a spurious response.
assert_eq!(state.consensus_queue.len(), 1);
assert_eq!(
    state.consensus_queue[0].callback_id,
    CallbackId::from(u64::MAX)
);
```

This test passes against the current code and demonstrates the invariant violation. Replaying the same batch on a second replica produces an identical state hash, confirming the bug is deterministic and consensus-safe from the protocol's perspective — meaning all honest nodes will accept and execute the fabricated payload identically.
