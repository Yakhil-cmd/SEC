### Title
Off-by-One Index Out of Bounds in `get_key_family` When Search Key Exceeds All Registry Records - (`File: rs/registry/client/src/client.rs`)

### Summary

`RegistryClientImpl::get_key_family` (and its mirror in `FakeRegistryClient`) uses `binary_search_by_key` to locate the first matching record. When the search key is lexicographically greater than every entry in the sorted `records` vector, `binary_search_by_key` returns `Err(records.len())`. The code then immediately indexes `records[idx]` without a bounds check, causing a Rust index-out-of-bounds panic.

### Finding Description

In `rs/registry/client/src/client.rs`, `get_key_family` constructs a composite search key `(key_prefix, &RegistryVersion::from(1))` and performs a binary search over the sorted `records` vector:

```rust
let first_match_index = match cache_state
    .records
    .binary_search_by_key(search_key, |r| (&r.key, &r.version))
{
    Ok(idx) => idx,
    Err(idx) => {
        // comment claims idx is always a valid index — this is wrong
        if !cache_state.records[idx].key.starts_with(key_prefix) {
            return Ok(vec![]);
        }
        idx
    }
};
```

`binary_search_by_key` returns `Err(insertion_point)` where `insertion_point` ranges from `0` to `records.len()` inclusive. When `key_prefix` is lexicographically greater than every key currently stored in the registry, `insertion_point == records.len()`, and `records[records.len()]` panics.

The inline comment — *"The entry at idx cannot be lexicographically less than key_prefix, otherwise the correctness assumption about bin search would not hold"* — conflates the ordering guarantee with a bounds guarantee. The ordering guarantee is correct; the bounds guarantee is not: `Err(records.len())` is a perfectly valid return value from `binary_search_by_key`.

The identical bug exists in `rs/registry/fake/src/lib.rs` at the same logical location. [1](#0-0) [2](#0-1) 

### Impact Explanation

`get_key_family` is called on every hot path that needs to enumerate a key family from the registry — for example `get_chain_key_enabled_subnets` (chain-key routing), node-record enumeration, API boundary-node discovery, and more. [3](#0-2) 

A panic in `RegistryClientImpl::get_key_family` propagates up through the replica's consensus and execution threads. Because the replica does not wrap registry calls in `std::panic::catch_unwind`, the thread unwinds and the replica process crashes or becomes unable to make progress, causing the node to drop out of consensus. If the condition is reproducible (e.g., tied to a specific registry state), the node cannot recover without a code fix.

### Likelihood Explanation

The trigger condition — all registry keys being lexicographically less than a queried prefix — can arise in two realistic scenarios:

1. **Freshly bootstrapped subnet**: During early subnet setup the registry may contain only a small set of keys (e.g., only `api_boundary_node_record_*` entries starting with `'a'`). Any `get_key_family` call with a prefix starting with a character later in the alphabet (e.g., `subnet_record_*` starting with `'s'`) would hit the bug before those keys are populated.

2. **New feature key prefix added to code before registry is populated**: When a new key family is introduced in a replica upgrade (e.g., a prefix starting with `'z'`), the first call to `get_key_family` with that prefix against an older registry state that has no such keys — and whose highest key is lexicographically less than the new prefix — triggers the panic.

Neither scenario requires an unprivileged attacker to directly manipulate the registry; both can arise from normal operational sequencing.

### Recommendation

Add an explicit bounds check before indexing:

```rust
Err(idx) => {
    if idx >= cache_state.records.len()
        || !cache_state.records[idx].key.starts_with(key_prefix)
    {
        return Ok(vec![]);
    }
    idx
}
```

This mirrors the correct pattern already used in `get_versioned_value` in the same file, which guards `idx > 0` before accessing `records[idx - 1]`. [4](#0-3) 

### Proof of Concept

```
registry records = [ ("aaa", v1), ("bbb", v1) ]   // all keys < "zzz"
call: get_key_family("zzz", v1)

binary_search_by_key(("zzz", v1), ...) => Err(2)   // 2 == records.len()
records[2]  =>  ** index out of bounds: the len is 2 but the index is 2 **
                ** replica thread panics **
```
