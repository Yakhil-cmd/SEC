### Title
CMC Rate-Limit Bypass via Cycles Donation to `ensure_balance` — (File: `rs/nns/cmc/src/main.rs`)

### Summary

The `ensure_balance` function in the Cycles Minting Canister (CMC) computes how many cycles to freshly mint by reading the live `canister_balance128()` and subtracting it from the requested amount. Because `Cycles::Sub` is a saturating subtraction, if the CMC's balance already meets or exceeds the requested amount, `cycles_to_mint` silently becomes zero. Both the rate limiter and the `total_cycles_minted` accounting counter are updated with this zero value. Any unprivileged canister can pre-fund the CMC by depositing cycles via the management canister's `deposit_cycles` endpoint, causing all subsequent `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` calls to bypass the hourly rate limit entirely.

### Finding Description

`ensure_balance` is the single gate that enforces the CMC's cycles-minting rate limit:

```rust
fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128()); // live balance
    let cycles_to_mint = cycles - current_balance;                           // saturating sub

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;   // rate-limit check
        state.total_cycles_minted += cycles_to_mint;                         // accounting
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);                 // actual mint
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
``` [1](#0-0) 

The `Cycles` type implements `Sub` as a **saturating** operation:

```rust
impl Sub for Cycles {
    type Output = Self;
    fn sub(self, rhs: Self) -> Self {
        Self(self.0.saturating_sub(rhs.0))   // saturates to zero on underflow
    }
}
``` [2](#0-1) 

When `current_balance >= cycles`, `cycles_to_mint` saturates to `Cycles(0)`. The rate limiter receives `0`, the `total_cycles_minted` counter is incremented by `0`, and `ic0_mint_cycles128(0)` mints nothing — yet the CMC still sends `cycles` worth of cycles to the target from its existing balance. The rate limit is never consumed.

The management canister exposes a `deposit_cycles` endpoint that any canister can call to increase the cycle balance of any target canister, including the CMC:

```rust
async fn deposit_cycles(
    canister_id: CanisterId,
    cycles: Cycles,
    mint_cycles: bool,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    if mint_cycles {
        ensure_balance(cycles, limiter_to_use)?;
    }
    let res: CallResult<()> = ic_cdk::api::call::call_with_payment128(
        candid::Principal::management_canister(),
        METHOD_DEPOSIT_CYCLES,
        ...
    ).await;
``` [3](#0-2) 

The NNS subnet imposes no cycle-balance ceiling on canisters, so the CMC can absorb an arbitrarily large donation.

### Impact Explanation

1. **Rate-limit bypass**: The hourly cycles-minting rate limits (`CYCLES_MINTING_LIMIT` for the base limiter, `SUBNET_RENTAL_CYCLES_MINTING_LIMIT` for the subnet-rental limiter) are enforced only on freshly minted cycles. An attacker who pre-funds the CMC with enough cycles can allow unlimited ICP-to-cycles conversion during the donation window without the limiter ever triggering.

2. **`total_cycles_minted` under-counting**: The `total_cycles_minted` state variable, exposed via the `total_cycles_minted` query endpoint, will not reflect cycles dispensed from the donated balance. This corrupts the on-chain accounting metric used to audit cycles inflation. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

The attack is reachable by any unprivileged canister on any subnet. The attacker's cost is the donated cycles themselves, but those cycles are not destroyed — they are simply transferred to the CMC's balance and then re-dispensed to legitimate callers. An attacker who wants to convert a large ICP position to cycles without hitting the hourly cap can:

1. Acquire cycles (e.g., from a DEX or by borrowing from another canister).
2. Deposit those cycles to the CMC via `management_canister::deposit_cycles(CMC_ID, amount)`.
3. Call `notify_top_up` / `notify_mint_cycles` to convert ICP to cycles — `ensure_balance` computes `cycles_to_mint = 0`, the rate limiter is not charged, and the CMC pays out from its donated balance.
4. Net effect: ICP is burned, cycles are received, rate limit is untouched.

No privileged access, no governance vote, and no threshold-crypto compromise is required.

### Recommendation

Replace the live-balance-based computation with an explicit check that skips minting only when the balance is sufficient, but **always charges the rate limiter and accounting counter for the full amount being dispensed**, regardless of whether new cycles are freshly minted or paid from an existing balance:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    // Always charge the full dispensed amount against the rate limit and accounting.
    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        state.total_cycles_minted += cycles;
        Ok::<_, String>(())
    })?;

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    if current_balance < cycles {
        let cycles_to_mint = cycles - current_balance;
        let _minted = ic0_mint_cycles128(cycles_to_mint);
    }
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

Alternatively, track a separate `cycles_dispensed` counter that includes both freshly minted and balance-sourced cycles, and enforce the rate limit against that counter.

### Proof of Concept

```
// Attacker canister pseudocode (Motoko/Rust canister on any subnet)

// Step 1: Deposit a large amount of cycles to the CMC to pre-fund it.
//         This bypasses the rate limit for all subsequent notify_top_up calls.
management_canister.deposit_cycles({
    canister_id = CYCLES_MINTING_CANISTER_ID
}, cycles = 500_000_000_000_000_000);  // 500 quadrillion cycles

// Step 2: Now call notify_top_up repeatedly.
//         Each call: ensure_balance sees current_balance >= cycles,
//         cycles_to_mint = 0, rate limiter is charged 0, limit never hit.
for _ in 0..1000 {
    cmc.notify_top_up({ block_index = ..., canister_id = target });
    // Each call converts 1500T cycles worth of ICP without consuming the rate limit.
}
// Total cycles dispensed: 1000 * 1500T = 1.5 quadrillion cycles
// Rate limit consumed: 0 (because cycles_to_mint was 0 each time)
```

The existing test `cmc_notify_top_up_not_rate_limited_by_invalid_top_up` demonstrates awareness of rate-limit manipulation, but it does not cover the cycles-donation vector described here. [6](#0-5)

        state_machine.advance_time(Duration::from_secs(5));

    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(4, 0).unwrap(),
    )
