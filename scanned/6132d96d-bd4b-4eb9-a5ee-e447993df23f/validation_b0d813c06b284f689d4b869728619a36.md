### Title
Missing Input Validation for `transfer_fee` in `ManageLedgerParameters` Proposal - (File: `rs/sns/governance/src/proposal.rs`)

### Summary
The `validate_and_render_manage_ledger_parameters` function in the SNS governance canister accepts any `transfer_fee` value without bounds checking. Any SNS governance participant can submit a `ManageLedgerParameters` proposal setting `transfer_fee` to `0` or `u64::MAX`. If passed, the ledger fee is applied immediately and `transaction_fee_e8s` in `NervousSystemParameters` is overwritten without re-running `NervousSystemParameters::validate()`, violating the invariant `neuron_minimum_stake_e8s > transaction_fee_e8s`.

### Finding Description
In `validate_and_render_manage_ledger_parameters` the `transfer_fee` field is accepted and rendered with no range check:

```rust
if let Some(transfer_fee) = transfer_fee {
    render += &format!("# Set token transfer fee: {transfer_fee} token-quantums. \n",);
    change = true;
}
``` [1](#0-0) 

After the proposal executes, the governance canister directly overwrites `transaction_fee_e8s` in `NervousSystemParameters` without calling `NervousSystemParameters::validate()`:

```rust
if let Some(nervous_system_parameters) = self.proto.parameters.as_mut()
    && let Some(transfer_fee) = manage_ledger_parameters.transfer_fee
{
    nervous_system_parameters.transaction_fee_e8s = Some(transfer_fee);
}
``` [2](#0-1) 

By contrast, `ManageNervousSystemParameters` proposals always call `NervousSystemParameters::validate()` before committing, which enforces `neuron_minimum_stake_e8s > transaction_fee_e8s`: [3](#0-2) 

The `validate_transaction_fee_e8s` helper only checks that the field is `Some`, with no floor or ceiling: [4](#0-3) 

### Impact Explanation
- **`transfer_fee = 0`**: All SNS token transfers become free. This breaks the economic model of the SNS (no fee revenue, potential spam of micro-transfers, archive/ledger bloat).
- **`transfer_fee >= neuron_minimum_stake_e8s`**: The invariant `neuron_minimum_stake_e8s > transaction_fee_e8s` is silently violated. Subsequent `ManageNervousSystemParameters` proposals that call `validate()` will fail with a confusing error, and neuron staking logic that relies on this invariant may behave incorrectly.
- **`transfer_fee = u64::MAX`**: All transfers become economically impossible, freezing the SNS ledger.

The `NervousSystemParameters` struct documents this invariant explicitly: [5](#0-4) 

### Likelihood Explanation
Any SNS token holder with sufficient stake to submit and pass a governance proposal can trigger this. Early-stage SNS deployments commonly have a single founding team with majority stake. A mistaken or malicious `ManageLedgerParameters` proposal with an extreme `transfer_fee` value would pass on-chain validation without any rejection, since no bounds check exists at the proposal validation layer.

### Recommendation
Add explicit floor and ceiling validation for `transfer_fee` inside `validate_and_render_manage_ledger_parameters`, analogous to how `NervousSystemParameters::validate_transaction_fee_e8s` and `validate_neuron_minimum_stake_e8s` enforce their relationship. Additionally, after executing a `ManageLedgerParameters` proposal that changes `transfer_fee`, call `NervousSystemParameters::validate()` on the updated parameters before committing, to ensure the `neuron_minimum_stake_e8s > transaction_fee_e8s` invariant is preserved.

### Proof of Concept
1. An SNS token holder submits a `ManageLedgerParameters` proposal with `transfer_fee = u64::MAX`.
2. `validate_and_render_manage_ledger_parameters` is called — it finds `transfer_fee` is `Some`, appends a render string, sets `change = true`, and returns `Ok(render)` with no error. [1](#0-0) 
3. The proposal passes governance vote and executes. The ledger is upgraded with `transfer_fee = u64::MAX`, and `nervous_system_parameters.transaction_fee_e8s` is set to `u64::MAX` without calling `validate()`. [2](#0-1) 
4. All subsequent ledger transfers fail because no user can afford the fee. The SNS ledger is effectively frozen.
