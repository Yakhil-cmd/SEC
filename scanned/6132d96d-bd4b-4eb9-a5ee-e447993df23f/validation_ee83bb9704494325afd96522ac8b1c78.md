### Title
FD Accumulation in `install_file_descriptors` Allows Compromised Sandbox to Exhaust Replica FD Table â€” (`rs/canister_sandbox/src/transport.rs`)

---

### Summary

`install_file_descriptors` does not close excess SCM_RIGHTS file descriptors when a received message carries more FDs than its `enumerate_fds()` slot count declares. Because all `ctlsvc::Request` variants (sandboxâ†’controller direction) declare **zero** FD slots, any FDs a compromised sandbox attaches to those messages accumulate permanently in the replica's `fds` accumulator vector and are never closed, exhausting the replica's FD table.

---

### Finding Description

**The accumulator vector is shared across the entire read loop.**

`socket_read_messages` allocates one `fds: Vec<RawFd>` that persists for the lifetime of the connection: [1](#0-0) 

`receive_message` pushes every SCM_RIGHTS FD it receives onto this vector. `install_file_descriptors` is then called to distribute them into the deserialized message's slots: [2](#0-1) 

The critical branch is:

```rust
if fd_locs.len() < fds.len() {
    fds.drain(0..fd_locs.len());   // only removes the consumed slots
} else {
    fds.clear();
}
```

When `fd_locs.len() == 0` (message declares no FD slots), `fds.drain(0..0)` drains **nothing**. Every FD that arrived with that message stays in `fds` and is never closed.

**All sandboxâ†’controller message types declare zero FD slots.**

`ctlsvc::Request` covers every message the sandbox can send to the controller (`ExecutionFinished`, `ExecutionPaused`, `LogViaReplica`), and its `enumerate_fds` implementation is a no-op: [3](#0-2) 

`sbxsvc::Reply` (the other half of `SandboxToController`) similarly declares no FD slots for any variant.

**`RawFd` is `i32`; dropping `Vec<RawFd>` does not close the FDs.**

The integers accumulate in the heap-allocated vector. They represent open kernel file-description references in the replica process. They are never passed to `libc::close`.

**Per-message FD injection rate.**

`CONTROL_MESSAGE_SIZE = 4096` bytes; each FD is 4 bytes; Linux kernel caps SCM_RIGHTS at ~253 FDs per `sendmsg`. A compromised sandbox can inject ~253 open FDs per message. With a default `RLIMIT_NOFILE` of 1024 for the replica, fewer than five crafted messages suffice to exhaust the table. [4](#0-3) 

---

### Impact Explanation

Once the replica's FD table is exhausted:
- `socketpair()` calls for new sandbox launches fail â†’ no new canister executions can start.
- `accept()` / `connect()` for peer-to-peer and HTTP outcall sockets fail.
- `open()` for state-sync and page-allocator memfds fails.
- The replica node effectively stops processing new messages, causing canister integrity loss through execution unavailability.

---

### Likelihood Explanation

**Prerequisite:** the sandbox process must be compromised (i.e., attacker has achieved code execution inside the sandbox). This requires a separate vulnerability (e.g., a Wasmtime memory-safety bug triggered by a malicious Wasm module submitted via ingress). That is a non-trivial bar.

**Once the sandbox is compromised**, the FD exhaustion path is trivially reachable: the sandbox already holds the write end of the Unix socket and can call `sendmsg` with arbitrary SCM_RIGHTS payloads. No additional privilege is needed. The sandbox isolation boundary is specifically the security guarantee being violated here.

---

### Recommendation

In `install_file_descriptors`, explicitly close every FD that remains in `fds` after the message's declared slots have been filled:

```rust
if fd_locs.len() < fds.len() {
    fds.drain(0..fd_locs.len());
    // Close excess FDs to prevent accumulation in the replica FD table.
    for &fd in fds.iter() {
        unsafe { libc::close(fd); }
    }
    fds.clear();
} else {
    fds.clear();
}
```

Additionally, consider enforcing a hard cap on the number of FDs accepted per `recvmsg` call in `receive_message`, rejecting (closing + dropping) the connection if the count exceeds the maximum declared by any legitimate message type.

fn install_file_descriptors<Message: EnumerateInnerFileDescriptors>(
    msg: &mut Message,
    fds: &mut Vec<RawFd>,
) {
    let mut fd_locs = vec![];
    msg.enumerate_fds(&mut fd_locs);
    for n in 0..fd_locs.len() {
        if n < fds.len() {
            *fd_locs[n] = fds[n];
        } else {
            *fd_locs[n] = -1;
        }
    }
    if fd_locs.len() < fds.len() {
        fds.drain(0..fd_locs.len());
    } else {
        fds.clear();
    }
}
