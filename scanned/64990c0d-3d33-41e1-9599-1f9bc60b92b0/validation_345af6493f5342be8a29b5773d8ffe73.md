### Title
Hardcoded 8-Decimal Assumption in SNS Treasury Valuation Causes Incorrect Transfer Limits - (`rs/sns/governance/token_valuation/src/lib.rs`)

---

### Summary

`try_get_balance_valuation_factors` in the SNS governance token valuation library hardcodes `E8` (10^8) as the divisor when converting a raw `icrc1_balance_of` response to a human-readable token count. The ICRC-1 standard allows any number of decimals, so SNS tokens with non-8 decimals produce a systematically wrong treasury XDR valuation. This valuation is the sole input to the regime classifier that enforces 7-day treasury transfer and minting limits, meaning the protection can be bypassed or incorrectly tightened.

---

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, the function `try_get_balance_valuation_factors` converts the raw ledger balance to a `Decimal` token count by dividing by the constant `E8 = 100_000_000`:

```rust
let tokens = Decimal::from(u128::try_from(balance_of_response.0)...) / Decimal::from(E8);
``` [1](#0-0) 

`icrc1_balance_of` returns the raw amount in the token's smallest indivisible unit. The correct divisor is `10^decimals`, where `decimals` is returned by `icrc1_decimals()`. The code never calls `icrc1_decimals()` and always uses `10^8`.

The resulting `tokens` field is multiplied by `icps_per_token` and `xdrs_per_icp` to produce the treasury's XDR value:

```rust
tokens * icps_per_token * xdrs_per_icp
``` [2](#0-1) 

This XDR value is passed to `ProposalsAmountTotalUpperBound`, which classifies the treasury as small (< 100 K XDR → no limit), medium (< 1.2 M XDR → 25 % cap), or large (≥ 1.2 M XDR → absolute 300 K XDR cap): [3](#0-2) 

The valuation is computed at proposal submission time for both `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals: [4](#0-3) 

and re-checked at execution time: [5](#0-4) 

The `ICRC1Client` interface already exposes `icrc1_decimals()` but it is never called in the valuation path: [6](#0-5) 

---

### Impact Explanation

**Scenario A – SNS token with 6 decimals (e.g., a USDC-style token):**

- Raw balance is in units of 10^-6 tokens.
- Code divides by 10^8, producing a `tokens` value 100× smaller than reality.
- A treasury actually worth 1,200,000 XDR (medium regime, 25 % cap) appears to be worth 12,000 XDR (small regime, no cap).
- The SNS community can pass a proposal to drain 100 % of the treasury instead of being limited to 25 %.

**Scenario B – SNS token with 18 decimals:**

- `tokens` is 10^10× larger than reality.
- Even a tiny treasury appears "large" and hits the absolute 300 K XDR cap.
- The cap is applied in wrong units: `allowance_tokens = 300_000 / xdrs_per_token` is computed in the wrong denomination, making the effective limit 10^10× tighter than intended, potentially blocking all treasury transfers.

In both cases the governance protection that is supposed to prevent rapid treasury drain is either bypassed or rendered inoperative.

---

### Likelihood Explanation

The ICRC-1 standard explicitly allows any `u8` value for `icrc1_decimals`. An SNS can be launched with a non-8-decimal token through the standard SNS initialization flow. No privileged access, no key compromise, and no majority attack is required. Any SNS neuron holder can submit a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal, which triggers the valuation path. The bug is deterministic and reproducible for any SNS whose token ledger reports `decimals != 8`.

---

### Recommendation

Replace the hardcoded `E8` divisor with a dynamic call to `icrc1_decimals()` on the ledger canister. The `Icrc1Client` trait should be extended (or the existing `ICRC1Client` in `packages/icrc-ledger-client` used) to fetch the actual decimal count, and the balance conversion should use `10^decimals` as the divisor:

```rust
let decimals = icrc1_client.icrc1_decimals().await?;
let tokens = Decimal::from(raw_balance) / Decimal::from(10_u128.pow(decimals as u32));
```

The same fix should be applied to `try_get_icp_balance_valuation` for consistency, though ICP is always 8 decimals in practice.

---

### Proof of Concept

1. Deploy an SNS with a token ledger configured with `decimals = 6`.
2. Fund the SNS treasury with enough tokens so the true XDR value is between 100,000 and 1,200,000 XDR (medium regime).
3. Submit a `TransferSnsTreasuryFunds` proposal requesting 100 % of the treasury balance.
4. During proposal validation, `try_get_sns_token_balance_valuation` is called: [7](#0-6) 
5. `try_get_balance_valuation_factors` divides the raw balance by `E8 = 10^8` instead of `10^6`, producing a `tokens` value 100× too small. [1](#0-0) 
6. `valuation.to_xdr()` returns a value 100× below the true XDR value, classifying the treasury as "small" (< 100,000 XDR). [8](#0-7) 
7. `ProposalsAmountTotalUpperBound` returns `NoLimit`, and the proposal passes validation. [9](#0-8) 
8. The proposal executes, draining the full treasury — bypassing the intended 25 % cap.

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }
