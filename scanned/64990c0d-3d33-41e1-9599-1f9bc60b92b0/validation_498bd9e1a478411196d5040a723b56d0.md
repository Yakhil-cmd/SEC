### Title
`MintSnsTokens` 7-Day Rate Limit Intentionally Disabled — Governance Majority Can Mint Unlimited SNS Tokens - (File: `rs/sns/governance/src/proposal.rs`)

---

### Summary

The `MintSnsTokens` SNS governance proposal action has its 7-day minting rate limit deliberately disabled via a `TODO` stub that returns `Decimal::MAX` as the upper bound, and its execution path (`perform_mint_sns_tokens`) performs no execution-time rate-limit check. By contrast, the analogous `TransferSnsTreasuryFunds` action enforces the limit at both submission time and execution time. A governance majority (including a single whale neuron holder with >50% voting power) can therefore pass `MintSnsTokens` proposals of arbitrary size, minting unlimited SNS tokens to any principal and diluting all other token holders.

---

### Finding Description

**`TransferSnsTreasuryFunds` — properly rate-limited at both stages:**

At submission, `treasury_valuation_if_proposal_amount_is_small_enough_or_err` is called, which enforces the 7-day window via `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens`. [1](#0-0) 

At execution, `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` is called again inside `perform_transfer_sns_treasury_funds`, re-checking the 7-day total against the valuation captured at submission time. [2](#0-1) 

**`MintSnsTokens` — rate limit disabled:**

The `TokenProposalAction` implementation for `MintSnsTokens` has the real upper-bound function commented out with `TODO(NNS1-2982): Uncomment`, and the active stub unconditionally returns `Decimal::MAX`:

```rust
// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    Ok(Decimal::MAX)
}
``` [3](#0-2) 

The commented-out real implementation that should be active: [4](#0-3) 

At execution, `perform_mint_sns_tokens` directly calls `self.ledger.transfer_funds(...)` with no rate-limit check whatsoever — unlike `perform_transfer_sns_treasury_funds`: [5](#0-4) 

The dispatch in `perform_action` confirms `MintSnsTokens` goes directly to `perform_mint_sns_tokens` with no auxiliary valuation lookup: [6](#0-5) 

Whereas `TransferSnsTreasuryFunds` retrieves the stored valuation and passes it to the rate-limit check: [7](#0-6) 

The `mint_sns_tokens_7_day_total_upper_bound_tokens` function exists and is fully implemented in the rate-limit library — it simply is not wired in: [8](#0-7)

### Impact Explanation

A governance majority can submit and pass a `MintSnsTokens` proposal for any `amount_e8s` (up to `u64::MAX`). Because `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`, the submission-time check always passes. Because `perform_mint_sns_tokens` has no execution-time check, the minting executes unconditionally. The minted tokens go to any `to_principal` the proposal specifies. This allows:

- Unlimited dilution of all existing SNS token holders.
- Effective drain of the SNS treasury's economic value (the treasury holds SNS tokens whose value collapses as supply is inflated).
- Repeated proposals within the same 7-day window, each minting the full token supply again.

This is the direct IC analog of the `withDrawFunds()` owner-drain vulnerability: a privileged actor (governance majority / whale) can extract unbounded value from the protocol.

---

### Likelihood Explanation

Many SNS deployments launch with a small number of founding neuron holders who collectively or individually hold >50% of voting power. A single whale neuron holder with majority voting power can unilaterally pass `MintSnsTokens` proposals. The attack requires no special keys, no admin access, and no external oracle manipulation — only sufficient SNS voting power, which is a realistic condition in early-stage or low-participation SNS DAOs. The codebase itself acknowledges the missing limit via the `TODO(NNS1-2982)` comment, confirming this is a known open issue.

---

### Recommendation

1. **Uncomment** the real `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (tracked as `NNS1-2982`) so that `mint_sns_tokens_7_day_total_upper_bound_tokens` is enforced at proposal submission.
2. **Add an execution-time rate-limit check** in `perform_mint_sns_tokens`, mirroring the pattern used in `perform_transfer_sns_treasury_funds` with `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`.
3. **Retain executed `MintSnsTokens` proposals** for at least 7 days (already done for `TransferSnsTreasuryFunds` via `EXECUTED_MINT_SNS_TOKENS_PROPOSAL_RETENTION_DURATION_SECONDS`) so the execution-time check can correctly sum recent minting totals. [9](#0-8) 

---

### Proof of Concept

1. Deploy an SNS where a single principal controls >50% of voting power (common in early SNS launches).
2. Submit a `MintSnsTokens` proposal:
   ```
   MintSnsTokens {
       amount_e8s: Some(u64::MAX),  // ~184 billion tokens
       to_principal: Some(attacker_principal),
       to_subaccount: None,
       memo: None,
   }
   ```
3. The submission-time check calls `recent_amount_total_upper_bound_tokens` which returns `Decimal::MAX` — proposal is accepted.
4. The whale votes yes; proposal passes.
5. `perform_mint_sns_tokens` calls `self.ledger.transfer_funds(u64::MAX, 0, None, attacker_account, 0)` with no rate-limit check.
6. `u64::MAX` SNS tokens are minted to the attacker. All existing holders are diluted to near-zero relative value.
7. Repeat immediately — no 7-day cooldown is enforced.

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
