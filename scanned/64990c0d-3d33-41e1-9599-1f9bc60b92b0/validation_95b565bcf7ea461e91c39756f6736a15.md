### Title
ckERC20 Minter Mints Based on Event-Logged Amount, Not Actual Received Amount, Enabling Over-Minting with Fee-on-Transfer ERC-20 Tokens - (File: `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

The ckETH/ckERC20 minter canister mints ckERC20 tokens based on the `amount` field emitted in the `ReceivedEthOrErc20` / `ReceivedErc20` Ethereum log event, which is set by the helper smart contract to the *requested* transfer amount — not the *actual* amount received by the minter address. For fee-on-transfer ERC-20 tokens, these two values differ. The minter has no mechanism to verify the actual on-chain balance change, so it over-mints ckERC20 tokens relative to the ERC-20 tokens it actually holds, breaking the 1:1 backing invariant.

---

### Finding Description

The deposit flow for ckERC20 proceeds as follows:

**Step 1 — Solidity helper contract (`DepositHelperWithSubaccount.sol`):**

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);  // actual received = amount - fee
    emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, ...);  // logs full `amount`, not actual received
}
```

The event is emitted with the caller-supplied `amount` parameter, not with the balance delta of `minterAddress`. For a fee-on-transfer token, `minterAddress` receives `amount - fee`, but the event records `amount`. [1](#0-0) 

The same pattern exists in the older helper: [2](#0-1) 

**Step 2 — IC minter log parser (`eth_logs/parser.rs`):**

The minter reads `value` directly from the event data bytes, which is the `amount` field set by the Solidity contract:

```rust
let value = Erc20Value::from_be_bytes(value_bytes);
Ok(ReceivedErc20Event { ..., value, ... }.into())
``` [3](#0-2) 

**Step 3 — IC minter mint logic (`deposit.rs`):**

The minter calls `icrc1_transfer` with `amount: event.value()` — the event-logged value — as the mint amount, with no cross-check against the minter's actual ERC-20 balance:

```rust
let block_index = match client
    .transfer(TransferArg {
        ...
        amount: event.value(),   // ← taken verbatim from the Ethereum log
    })
    .await
``` [4](#0-3) 

There is no step anywhere in the IC minter that queries the minter's actual ERC-20 balance on Ethereum before or after the deposit to verify the received amount matches the event-logged amount.

---

### Impact Explanation

For any fee-on-transfer ERC-20 token added as a supported ckERC20 token:

- The minter mints `amount` ckERC20 tokens but only holds `amount - fee` ERC-20 tokens.
- The ckERC20 total supply exceeds the minter's ERC-20 balance by `fee` per deposit.
- Accumulated over many deposits, the shortfall grows unboundedly.
- Eventually, some ckERC20 holders cannot redeem their tokens because the minter's ERC-20 reserve is insufficient — a direct ledger conservation violation.
- An attacker who knows a supported token has fees enabled can repeatedly deposit to systematically inflate the ckERC20 supply beyond the backing reserve, then redeem early while later redeemers are left with unbacked tokens.

USDT (Tether) has a fee mechanism in its contract that can be toggled on by the contract owner. If ckUSDT were added as a supported token and Tether later enabled the fee, every subsequent deposit would silently over-mint.

---

### Likelihood Explanation

The vulnerability requires a fee-on-transfer ERC-20 token to be active as a supported ckERC20 token. Two realistic paths exist:

1. A new fee-on-transfer token is proposed and accepted via NNS governance as a supported ckERC20 token. The NNS proposal process does not currently include automated on-chain verification that the token is fee-free.
2. An already-supported token (e.g., USDT, which has a dormant fee switch) has its fee enabled by the token contract owner after being added to the minter. This requires no IC governance action and is entirely outside IC control.

Path 2 is particularly concerning: it is triggered by a third-party action (the ERC-20 token owner enabling fees) and requires no IC-side change. Once triggered, any unprivileged Ethereum user can exploit it by simply calling `depositErc20`.

---

### Recommendation

1. **In the Solidity helper contracts**: Measure the minter's balance before and after `safeTransferFrom` and emit the actual received amount (balance delta) in the event, not the caller-supplied `amount`:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

2. **In the IC minter**: Add a validation step (or documentation/governance gate) that explicitly rejects fee-on-transfer tokens when adding new ckERC20 tokens. The minter could query the ERC-20 token's balance before and after a test transfer during the `add_ckerc20_token` governance proposal execution.

3. **Governance process**: Require that any NNS proposal to add a new ckERC20 token includes explicit verification that the token does not implement fee-on-transfer or rebasing mechanics.

---

### Proof of Concept

1. USDT is added as a supported ckERC20 token via NNS proposal.
2. Tether enables the USDT transfer fee (e.g., 1 basis point = 0.01%).
3. Attacker calls `depositErc20(USDT_ADDRESS, 1_000_000_000, principal, subaccount)` on the helper contract.
4. `safeTransferFrom` moves `1_000_000_000 - 100_000 = 999_900_000` USDT to the minter address (fee = 100,000 units).
5. The helper emits `ReceivedEthOrErc20(..., 1_000_000_000, ...)` with the full requested amount.
6. The IC minter scrapes the log, parses `value = 1_000_000_000` from the event data.
7. The minter calls `icrc1_transfer` minting `1_000_000_000` ckUSDT to the attacker.
8. The attacker received 100,000 more ckUSDT than the USDT deposited.
9. Repeated deposits accumulate a growing shortfall; the last redeemers cannot withdraw their ckUSDT.
