### Title
Stale Ballot Voting Power Used as Reward Share Denominator in SNS Governance — (`rs/sns/governance/src/governance.rs`)

### Summary

In the SNS governance canister, `distribute_rewards` computes each neuron's reward share using the voting power recorded in proposal ballots at proposal-creation time. The denominator (`total_reward_shares`) is the sum of only the *voting* neurons' ballot powers, not the total potential voting power of all eligible neurons at proposal creation. Because `staked_maturity_e8s_equivalent` contributes to voting power and grows over time as rewards are auto-staked, neurons that auto-stake maturity and vote on many proposals accumulate progressively higher voting power relative to neurons that do not. Since the ballot voting power is frozen at proposal creation but the denominator only sums voters, a neuron that votes on every proposal in a reward round receives a disproportionately large share of the reward purse compared to what its fair proportional stake would entitle it to.

### Finding Description

In `rs/sns/governance/src/governance.rs`, `distribute_rewards` computes reward shares as follows:

```rust
// Add up reward shares based on voting power that was exercised.
let mut neuron_id_to_reward_shares: HashMap<NeuronId, Decimal> = HashMap::new();
for proposal_id in &considered_proposals {
    if let Some(proposal) = self.get_proposal_data(*proposal_id) {
        for (voter, ballot) in &proposal.ballots {
            if !Vote::try_from(ballot.vote)...eligible_for_rewards() { continue; }
            let reward_shares = i2d(ballot.voting_power);
            *neuron_id_to_reward_shares.entry(neuron_id)... += reward_shares;
        }
    }
}
let total_reward_shares: Decimal = neuron_id_to_reward_shares.values().sum();
// ...
let neuron_reward_e8s = rewards_purse_e8s * (neuron_reward_shares / total_reward_shares);
``` [1](#0-0) 

The `total_reward_shares` denominator is the **sum of ballot voting powers of neurons that actually voted**, not the total potential voting power of all eligible neurons at proposal creation time. This is structurally different from the NNS governance, which explicitly stores `total_potential_voting_power` on each `ProposalData` and uses it as the denominator:

```rust
// Increment global total.
total_weighted_voting_power += reward_weight
    * proposal
        .total_potential_voting_power
        .unwrap_or(total_ballots_voting_power) as f64;
``` [2](#0-1) 

The SNS `ProposalData` has no `total_potential_voting_power` field. The SNS `compute_ballots_for_new_proposal` records each eligible neuron's voting power at proposal creation time into ballots, and the total power is returned but **not stored on the proposal**:

```rust
fn compute_ballots_for_new_proposal(&self) -> Result<(u64, BTreeMap<String, Ballot>), String> {
    // ...
    Ok((total_power as u64, electoral_roll))
}
``` [3](#0-2) 

The returned `total_power` is used only for overflow checking and is discarded — it is never stored on the `ProposalData`. As a result, when `distribute_rewards` runs, it cannot use the correct denominator (total eligible voting power at proposal creation). Instead it sums only the voting neurons' ballot powers, which is always ≤ the true total.

Additionally, `staked_maturity_e8s_equivalent` contributes to voting power in SNS:

```rust
fn voting_power_stake_e8s(&self) -> u64 {
    self.cached_neuron_stake_e8s
        .saturating_sub(self.neuron_fees_e8s)
        .saturating_add(self.staked_maturity_e8s_equivalent.unwrap_or(0))
}
``` [4](#0-3) 

When `auto_stake_maturity` is enabled, rewards are added directly to `staked_maturity_e8s_equivalent`:

```rust
if neuron.auto_stake_maturity.unwrap_or(false) {
    neuron.staked_maturity_e8s_equivalent = Some(
        neuron.staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s,
    );
}
``` [5](#0-4) 

This means a neuron with `auto_stake_maturity=true` that votes on every proposal will have its voting power grow each reward round, and since the denominator only counts voters, it will capture an ever-increasing fraction of the reward purse relative to neurons that vote less frequently or have `auto_stake_maturity=false`.

### Impact Explanation

Voting rewards in SNS governance are computed with an incorrect denominator. Neurons that vote on every proposal and have `auto_stake_maturity` enabled receive a disproportionately large share of the reward purse. Neurons that vote less frequently, or that do not auto-stake, receive less than their fair share. Over many reward rounds, this compounds: the auto-staking neuron's voting power grows faster than it should, further skewing future reward calculations. This is a governance accounting bug that causes incorrect maturity distribution — the analog of the Wild Credit H-01 "wrong rewards minted for users which do not represent their real fair share."

The impact is bounded to SNS governance instances (not NNS, which correctly stores `total_potential_voting_power` on proposals). Any SNS with multiple neurons of different voting activity levels and `auto_stake_maturity` configurations is affected.

### Likelihood Explanation

This is reachable by any SNS governance participant. No privileged access is required. Any neuron holder can enable `auto_stake_maturity` and vote on every proposal. The effect is automatic and continuous — it occurs every reward round as long as the SNS has proposals and neurons with differing voting participation rates. The entry path is: unprivileged neuron controller calls `manage_neuron` with `ChangeAutoStakeMaturity { requested_setting_for_auto_stake_maturity: true }`, then votes on proposals via `manage_neuron`. The reward skew accumulates automatically each reward round when `distribute_rewards` is called from `run_periodic_tasks`.

### Recommendation

Store `total_potential_voting_power` on SNS `ProposalData` at proposal creation time (analogous to the NNS field at `rs/nns/governance/proto/ic_nns_governance/pb/v1/governance.proto` line 1246), and use it as the denominator in `distribute_rewards` instead of `total_reward_shares` (the sum of only voting neurons' ballot powers). The `compute_ballots_for_new_proposal` already computes `total_power` — it should be stored on the proposal. In `distribute_rewards`, the per-proposal denominator should be `total_potential_voting_power` (scaled by proposal weight), not the sum of exercised ballot powers.

### Proof of Concept

**Setup**: SNS with two neurons, A and B, each with 1000 tokens staked, `auto_stake_maturity=true` for A, `auto_stake_maturity=false` for B. Both vote on every proposal.

**Round 1**: Both neurons have voting power 1000. Proposal P1 is created; both vote. `total_reward_shares = 1000 + 1000 = 2000`. Reward purse = 100 e8s. A gets 50, B gets 50. A's `staked_maturity` becomes 50, so A's voting power for the next proposal is 1050.

**Round 2**: Proposal P2 is created. A's ballot = 1050, B's ballot = 1000. Both vote. `total_reward_shares = 1050 + 1000 = 2050`. A gets `100 * 1050/2050 ≈ 51.2`, B gets `100 * 1000/2050 ≈ 48.8`. A's staked maturity is now 101.2, voting power 1101.2.

**Round N**: A's share of rewards keeps growing above 50% even though A and B deposited equal amounts at the same time. The correct denominator (total potential voting power at proposal creation, including non-voters) would keep the ratio fair. The incorrect denominator (sum of voters only) amplifies the compounding effect of `auto_stake_maturity`.

The root cause is in `rs/sns/governance/src/governance.rs` at the `distribute_rewards` function: `total_reward_shares` sums only voting neurons' ballot powers rather than using the total eligible voting power at proposal creation time. [6](#0-5)

    /// Computes the total potential voting power of the governance canister and ballots.
    fn compute_ballots_for_new_proposal(&self) -> Result<(u64, BTreeMap<String, Ballot>), String> {
        let now_seconds = self.env.now();

        let nervous_system_parameters = self.nervous_system_parameters_or_panic();

        // Voting power bonus parameters.
        let max_dissolve_delay = nervous_system_parameters
            .max_dissolve_delay_seconds
            .expect("NervousSystemParameters must have max_dissolve_delay_seconds");

        let max_age_bonus = nervous_system_parameters
            .max_neuron_age_for_age_bonus
            .expect("NervousSystemParameters must have max_neuron_age_for_age_bonus");

        let max_dissolve_delay_bonus_percentage = nervous_system_parameters
            .max_dissolve_delay_bonus_percentage
            .expect("NervousSystemParameters must have max_dissolve_delay_bonus_percentage");

        let max_age_bonus_percentage = nervous_system_parameters
            .max_age_bonus_percentage
            .expect("NervousSystemParameters must have max_age_bonus_percentage");

        let min_dissolve_delay_for_vote = nervous_system_parameters
            .neuron_minimum_dissolve_delay_to_vote_seconds
            .expect("NervousSystemParameters must have min_dissolve_delay_for_vote");

        let mut electoral_roll = BTreeMap::<String, Ballot>::new();
        let mut total_power: u128 = 0;

        for (k, v) in self.proto.neurons.iter() {
            // If this neuron is eligible to vote, record its
            // voting power at the time of proposal creation (now).
            if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
                // Not eligible due to dissolve delay.
                continue;
            }

            let voting_power = v.voting_power(
                now_seconds,
                max_dissolve_delay,
                max_age_bonus,
                max_dissolve_delay_bonus_percentage,
                max_age_bonus_percentage,
            );

            total_power += voting_power as u128;
            electoral_roll.insert(
                k.clone(),
                Ballot {
                    vote: Vote::Unspecified as i32,
                    voting_power,
                    cast_timestamp_seconds: 0,
                },
            );
        }

        if total_power >= (u64::MAX as u128) {
            // The way the neurons are configured, the total voting
            // power on this proposal would overflow a u64!
            return Err("Voting power overflow.".to_string());
        }
        if electoral_roll.is_empty() {
            // Cannot make a proposal with no eligible voters.  This
            // is a precaution that shouldn't happen as we check that
            // the voter is allowed to vote.
            return Err("No eligible voters.".to_string());
        }

        Ok((total_power as u64, electoral_roll))
    }
