### Title
Unbounded `ingress_expiries` Loop in `RosettaRequestHandler::construction_payloads` Causes OOM Crash — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

### Summary

The ICP Rosetta server's `construction_payloads` endpoint accepts client-controlled `ingress_start` and `ingress_end` values with no bounds validation before entering a loop that allocates one `u64` per step. Setting `ingress_start=0` and `ingress_end=u64::MAX` causes ~154 billion loop iterations, exhausting all available memory and crashing the server.

### Finding Description

The loop at lines 99–107 of `construction_payloads.rs` iterates from `ingress_start` to `ingress_end` in steps of `interval`: [1](#0-0) [2](#0-1) 

`interval` = `MAX_INGRESS_TTL(300s) − PERMITTED_DRIFT(60s) − 120s` = **120 seconds** = `120,000,000,000` nanoseconds. [3](#0-2) 

Both `ingress_start` and `ingress_end` are `Option<u64>` fields deserialized directly from the JSON request body: [4](#0-3) 

They are accepted without any range validation before the loop: [5](#0-4) 

With `ingress_start = 0` and `ingress_end = u64::MAX = 18,446,744,073,709,551,615`:

```
iterations = 18,446,744,073,709,551,615 / 120,000,000,000 ≈ 153,722,867,280
memory     = 153,722,867,280 × 8 bytes ≈ 1.23 TB
```

The server OOM-crashes after exhausting available RAM (typically a few GB), which occurs after only a few hundred million iterations — well before the loop terminates.

**The ICRC1 Rosetta counterpart has explicit guards that the ICP handler is missing:** [6](#0-5) 

The ICP handler has no equivalent check.

### Impact Explanation

The Rosetta server is a standalone HTTP process. Any unauthenticated client can POST to `/construction/payloads` with a crafted metadata object. The resulting unbounded `Vec<u64>` allocation exhausts process memory, triggering an OOM kill. This takes down the entire Rosetta server, blocking all ICP exchange operations (transfers, neuron management, etc.) that depend on it until the process is manually restarted.

### Likelihood Explanation

The attack requires a single unauthenticated HTTP POST with two integer fields set to extreme values. No credentials, keys, or special access are needed. The Rosetta API is publicly exposed by exchanges and infrastructure operators. The exploit is trivially reproducible locally.

### Recommendation

Add a maximum ingress window size check before the loop, mirroring the ICRC1 Rosetta implementation. For example:

```rust
// Reject windows larger than a reasonable maximum (e.g., 24 hours)
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60);
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end exceeds maximum allowed window from ingress_start",
    ));
}
```

Additionally, add a guard that `ingress_start < ingress_end` (as the ICRC1 handler does at line 148).

### Proof of Concept

```http
POST /construction/payloads HTTP/1.1
Host: <rosetta-server>:8080
Content-Type: application/json

{
  "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
  "operations": [{ "operation_identifier": {"index": 0}, "type": "TRANSACTION", ... }],
  "public_keys": [{ "hex_bytes": "<valid_pubkey>", "curve_type": "edwards25519" }],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

The server allocates memory unboundedly in the loop at lines 99–107 and is killed by the OS OOM killer within seconds, returning no response and leaving the server unavailable. [2](#0-1)
