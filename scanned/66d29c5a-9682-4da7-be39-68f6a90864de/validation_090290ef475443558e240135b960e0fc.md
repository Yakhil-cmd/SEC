### Title
`SenderInfoContent` Canister Signature Lacks Request-Context Binding, Enabling Cross-Request Replay of Identity Attestations — (File: `rs/types/types/src/messages/http.rs`)

---

### Summary

The `sender_info` canister signature verified during ingress validation signs only the raw `info` blob under the domain `"ic-sender-info"`. It does not commit to the sender's principal, the target canister ID, the `MessageId`, or any expiry. A valid `sender_info` canister signature obtained for one request is cryptographically valid for any other request by the same user to any canister, indefinitely. This is the direct IC analog of the Revolver `joinGame()` finding: the signed payload omits all session/context fields.

---

### Finding Description

`SenderInfoContent` is the signable type used to verify the `sender_info` field in ingress messages. Its signed-bytes implementation appends only the raw `info` blob: [1](#0-0) 

The `SignatureDomain` for `SenderInfoContent` prepends `"ic-sender-info"`, so the full signed preimage is:

```
\x0Eic-sender-info || info_bytes
```

The preimage does **not** include:
- The sender's principal (`UserId`) — who is making the request
- The target `canister_id` — which canister is being called
- The `MessageId` — which encodes `canister_id`, `method_name`, `arg`, `sender`, `ingress_expiry`, and `nonce`
- Any expiry or nonce specific to the attestation

The validator in `verify_sender_info_canister_sig` checks three things: [2](#0-1) 

1. `sender_pubkey` is a valid canister signature public key.
2. The canister ID encoded in `sender_pubkey` matches `sender_info.signer`.
3. The canister signature over `SenderInfoContent(&sender_info.info)` is valid.

It does **not** verify that the `sender_info` signature is bound to the current request's `MessageId`, sender principal, or target canister. The `sender_info` blob is included in the `MessageId` computation (so the envelope signature covers it), but the `sender_info`'s own canister signature is not bound to the request. [3](#0-2) 

The `SignatureDomain` implementation confirms the domain is a static string with no per-request context: [4](#0-3) 

---

### Impact Explanation

A valid `sender_info` canister signature issued by Internet Identity (or any signing canister) for `info_bytes = [some_claim]` is cryptographically valid for **every** future request by the same user (same `sender_pubkey`) to **any** canister, with **any** method, **any** arg, and **any** `ingress_expiry`, indefinitely.

Concrete harm:
- **Cross-canister replay**: II issues a `sender_info` attesting user attributes for Canister X. The user (or an observer who captured the `sender_info` blob) replays the same `(info, signer, sig)` in a request to Canister Y. Canister Y receives and trusts the attestation even though II never intended to vouch for this context.
- **Indefinite reuse**: The `sender_info` canister signature has no expiry of its own. Once issued, it is valid forever (until the signing canister's certified state changes), undermining any "one-shot" gatekeeping the signing canister intended.
- **Weakened identity attestation model**: The `sender_info` feature is designed to let a canister (e.g., II) vouch for user attributes on a per-request basis. Without request-context binding, the attestation is effectively a permanent, context-free credential — the opposite of the intended design.

---

### Likelihood Explanation

The `sender_info` feature is reachable by any unprivileged ingress sender. Any user who has obtained a valid `sender_info` signature (e.g., by authenticating with II for one request) can immediately replay it in subsequent requests to different canisters. No privileged access, key compromise, or threshold attack is required. The attacker only needs to capture the `(info, signer, sig)` triple from their own prior request (or from an observed request if the transport is not confidential).

---

### Recommendation

Bind the `sender_info` canister signature to the request context by including the `MessageId` (or at minimum the sender principal and target canister ID) in the signed preimage:

```rust
// In SenderInfoContent or a new RequestBoundSenderInfoContent:
impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        // Bind to: sender principal, target canister_id, and info blob
        bytes.extend_from_slice(self.sender.as_slice());
        bytes.extend_from_slice(self.canister_id.as_ref());
        bytes.extend_from_slice(self.info);
    }
}
```

At minimum, include the sender's principal and the target canister ID so the attestation cannot be replayed across users or canisters. Including the full `MessageId` would provide the strongest binding (preventing replay across any two distinct requests).

### Proof of Concept

**Step 1**: User A authenticates with II and obtains a `sender_info` for Request R1 targeting Canister X:
```
sender_info = {
  info: [role: "premium_user"],
  signer: II_canister_id,
  sig: canister_sig_over("ic-sender-info" || [role: "premium_user"])
}
```

**Step 2**: User A submits R1 to Canister X. The validator accepts it because the canister signature over `SenderInfoContent(&[role: "premium_user"])` is valid.

**Step 3**: User A constructs Request R2 targeting Canister Y (a different canister with different access controls), reusing the identical `(info, signer, sig)` triple. R2 has a different `canister_id`, `method_name`, `arg`, and `ingress_expiry` — so its `MessageId` is completely different from R1's.

**Step 4**: `verify_sender_info_canister_sig` in `rs/validator/src/ingress_validation.rs` verifies:
- `sender_pubkey` encodes II's canister ID ✓
- `pubkey_canister_id == sender_info.signer` ✓
- Canister signature over `SenderInfoContent(&[role: "premium_user"])` is valid ✓

The validator accepts R2's `sender_info` even though II never issued an attestation for Canister Y. Canister Y receives `msg_caller_info_data() = [role: "premium_user"]` and grants User A premium access. [5](#0-4) [6](#0-5)

/// Verifies that the sender_info canister signature is valid:
/// 1. The envelope-level sender_pubkey is a valid canister signature public key.
/// 2. The canister ID encoded in sender_pubkey matches the signer field.
/// 3. The signature over the info blob is valid against the root of trust.
fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
}
