### Title
Voting Rewards Permanently Lost When Neuron Is Dissolved Before Reward Distribution — (`File: rs/nns/governance/src/governance.rs`)

---

### Summary

In NNS governance (and analogously in SNS governance), when a neuron votes on a proposal and is subsequently dissolved and disbursed before the periodic reward distribution runs, its entire voting reward share is silently skipped and permanently lost. The undistributed amount is **not** rolled over to the next reward period, causing a ledger conservation bug where `total_available_e8s_equivalent` persistently exceeds `distributed_e8s_equivalent` with no recovery path.

---

### Finding Description

**Root cause — NNS governance `calculate_voting_rewards`:**

In `rs/nns/governance/src/governance.rs`, the function `calculate_voting_rewards` iterates over the ballots of all settled proposals to compute per-neuron reward shares. For each voter it checks whether the neuron still exists in the neuron store at the moment of reward calculation:

```rust
for (neuron_id, used_voting_rights) in voters_to_used_voting_right {
    if self.neuron_store.contains(neuron_id) {
        let reward = (used_voting_rights * total_available_e8s_equivalent_float
            / total_voting_rights) as u64;
        reward_distribution.add_reward(neuron_id, reward);
        actually_distributed_e8s_equivalent += reward;
    } else {
        println!(
            "{}Cannot find neuron {}, despite having voted with power {} \
                in the considered reward period. The reward that should have been \
                distributed to this neuron is simply skipped, so the total amount \
                of distributed reward for this period will be lower than the maximum \
                allowed.",
            LOG_PREFIX, neuron_id.id, used_voting_rights
        );
    }
}
``` [1](#0-0) 

If the neuron has been removed from the store (via `disburse_neuron` → `remove_neuron`), its reward share is silently dropped and `actually_distributed_e8s_equivalent` is not incremented for it. [2](#0-1) 

**No rollover of the gap:**

The `e8s_equivalent_to_be_rolled_over` function in `rs/nns/governance/src/reward/calculation.rs` returns the `total_available_e8s_equivalent` only when `rewards_rolled_over()` is true — which is only when `settled_proposals.is_empty()`. When proposals are settled (the normal case), it returns `0`:

```rust
pub(crate) fn e8s_equivalent_to_be_rolled_over(&self) -> u64 {
    if self.rewards_rolled_over() {
        self.total_available_e8s_equivalent
    } else {
        0
    }
}
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty()
}
``` [3](#0-2) 

Therefore, the gap between `total_available_e8s_equivalent` and `distributed_e8s_equivalent` is permanently discarded — it is neither distributed nor carried forward.

**Neuron deletion path:**

`disburse_neuron` (callable by any neuron controller once the neuron is in `Dissolved` state) performs no check for pending reward eligibility before removing the neuron from the store: [4](#0-3) 

**Same pattern in SNS governance:**

The SNS `distribute_rewards` function contains the identical skip-and-log pattern with the same absence of rollover: [5](#0-4) 

The SNS `e8s_equivalent_to_be_rolled_over` also returns 0 when proposals are settled: [6](#0-5) 

**Scenario (step-by-step):**

1. Neuron A (dissolve delay ≥ 6 months) votes on proposal P — ballot recorded in `P.ballots` with `voting_power = V`.
2. Proposal P is decided; its `reward_status` transitions to `ReadyToSettle`.
3. Neuron A's dissolve delay reaches 0 (it enters `Dissolved` state).
4. Neuron A's controller calls `disburse_neuron`; the neuron is removed from the neuron store.
5. The daily `distribute_voting_rewards_to_neurons` timer fires.
6. `calculate_voting_rewards` finds Neuron A's ballot in `P.ballots` but `neuron_store.contains(A)` returns `false`.
7. Neuron A's proportional share of `total_available_e8s_equivalent` is skipped.
8. The `RewardEvent` records `distributed_e8s_equivalent < total_available_e8s_equivalent`; the gap is not rolled over.

---

### Impact Explanation

The rewards purse for each distribution round is calculated from the total ICP token supply. When one or more neurons are deleted between voting and reward distribution, the actual maturity minted is permanently less than the protocol-intended amount. The `total_available_e8s_equivalent` field in the `RewardEvent` records the intended amount, but the shortfall is silently absorbed — it is never minted and never rolled over. This is a **ledger conservation bug**: the governance canister's accounting of available rewards diverges from what is actually distributed, with no recovery mechanism. Repeated occurrences compound the deficit over time. [7](#0-6) 

---

### Likelihood Explanation

The trigger is a neuron that (a) voted on a proposal (directly or via following) and (b) is in `Dissolved` state and disbursed before the next daily reward distribution. Neurons with a dissolve delay that has already elapsed are immediately disburse-eligible. Because NNS reward distribution runs once per day and proposal voting periods can be as short as hours, the window is realistic. Any neuron controller who disburses a dissolved neuron that voted in the current reward period will silently forfeit that reward share, and the protocol will not compensate other neurons or roll the amount forward. Likelihood is **low-to-medium** for any individual event but non-zero and grows with the number of active neurons that dissolve and disburse.

---

### Recommendation

1. **Roll over the gap**: When `actually_distributed_e8s_equivalent < total_available_e8s_equivalent` and the shortfall is due to missing neurons (not rounding), add the difference to the next period's rollover pool rather than discarding it. The existing `e8s_equivalent_to_be_rolled_over` mechanism already supports this for the no-proposals case; extend it to cover partial-distribution cases.

2. **Guard at disburse time**: Before removing a neuron from the store, check whether any `ReadyToSettle` proposals contain a ballot for that neuron. If so, either defer disbursement until after the next reward event or pre-credit the neuron's maturity before deletion.

3. **Apply the same fix to SNS governance** `distribute_rewards` in `rs/sns/governance/src/governance.rs`.

---

### Proof of Concept

```
1. Deploy NNS governance with Neuron A (dissolve_delay = 6 months, stake = S).
2. Submit proposal P; Neuron A votes YES (via following or directly).
3. Proposal P is decided; reward_status = ReadyToSettle.
4. Advance time so Neuron A's dissolve_delay = 0 (Dissolved state).
5. Call manage_neuron { Disburse } for Neuron A → neuron removed from store.
6. Advance time by REWARD_DISTRIBUTION_PERIOD_SECONDS to trigger distribute_voting_rewards_to_neurons.
7. Observe RewardEvent:
     distributed_e8s_equivalent  = X
     total_available_e8s_equivalent = X + gap   (gap = Neuron A's share)
8. Advance another REWARD_DISTRIBUTION_PERIOD_SECONDS (no proposals).
9. Observe next RewardEvent:
     rolling_over_from_previous = 0   ← gap is NOT carried forward
     total_available_e8s_equivalent = fresh_round_amount only
10. Confirm: Neuron A's reward share is permanently lost; no other neuron receives it.
