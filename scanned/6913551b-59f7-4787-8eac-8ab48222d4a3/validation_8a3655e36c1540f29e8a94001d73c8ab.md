### Title
Secp256k1 Private Key Exposed via CLI Argument in Shell History and Process List - (File: rs/boundary_node/rate_limits/canister_client/src/main.rs)

### Summary
The `rate-limiting-canister-client` tool accepts a raw Secp256k1 private key (SEC1 PEM) as a plain-text `--identity-key` command-line argument. This causes the key to be stored in shell history and visible in the OS process list (`ps`), allowing any local user or attacker with shell access to steal the signing identity used to submit rate-limiting configurations to the IC boundary node rate-limiting canister.

### Finding Description
In `rs/boundary_node/rate_limits/canister_client/src/main.rs`, the `Cli` struct defines `identity_key` as an `Option<String>` CLI argument:

```rust
/// Identity key
#[arg(long)]
identity_key: Option<String>,
``` [1](#0-0) 

The value is then directly parsed as a `SecretKey` and used to construct a `Secp256k1Identity` for authenticating calls to the IC:

```rust
let identity = Secp256k1Identity::from_private_key(
    SecretKey::from_sec1_pem(&identity_key).context("failed to parse the identity key")?,
);
``` [2](#0-1) 

This identity is then used to call `submit_config`, which submits rate-limiting rules to the canister: [3](#0-2) 

Additionally, the `Cli` struct derives `Debug`: [4](#0-3) 

This means any debug-level logging of the parsed `Cli` struct will print the full private key in plaintext.

A secondary exposure exists in `rs/sys/src/utility_command.rs`, where the HSM PIN is passed as a `--pin` process argument to a `pkcs11-tool` subprocess (visible via `ps`), and the `Display` implementation for `UtilityCommand` includes all arguments — meaning the PIN appears in error log messages: [5](#0-4) [6](#0-5) 

### Impact Explanation
An attacker who gains local access to the boundary node operator's machine (e.g., via a compromised shell session, shared server, or audit of shell history files) can recover the raw Secp256k1 private key from:
- Shell history files (e.g., `~/.bash_history`, `~/.zsh_history`)
- Live process listing via `ps aux` or `/proc/<pid>/cmdline`
- Debug log output if the `Cli` struct is ever logged

With the stolen key, the attacker can impersonate the legitimate operator and call `submit_config` to push arbitrary rate-limiting rules to the IC boundary node rate-limiting canister. This can be used to suppress rate limits for malicious traffic or impose rules that deny service to legitimate IC users.

### Likelihood Explanation
The tool is a production boundary node management utility. Operators running it on shared infrastructure or logging terminals are directly exposed. Shell history leakage is a well-known, low-effort attack vector requiring only read access to the operator's home directory — no special privileges are needed beyond local access.

### Recommendation
**Short term:** Remove the `--identity-key` string argument. Replace it with `--identity-key-file <path>` that accepts a path to a PEM file, keeping the secret off the command line entirely (analogous to how `--secret_key_pem` is handled in `rs/registry/admin/bin/main.rs`).

**Long term:** Ensure the `Cli` struct does not derive `Debug` while it contains secret fields, or implement a custom `Debug` that redacts sensitive fields. For `UtilityCommand::sign_message`, avoid passing the HSM PIN as a process argument; use `pkcs11-tool`'s `--pin-source` (file descriptor or file) mechanism instead, and redact PIN values from the `Display` implementation.

### Proof of Concept
```bash
# Operator submits rate-limiting config with private key on the command line
rate-limiting-canister-client \
  --canister-id <canister_id> \
  --config-file rules.json \
  --identity-key "-----BEGIN EC PRIVATE KEY-----\nMHQCAQEEI...\n-----END EC PRIVATE KEY-----"

# Attacker on the same machine reads shell history
cat ~/.bash_history | grep identity-key
# Output: --identity-key "-----BEGIN EC PRIVATE KEY-----\nMHQCAQEEI...\n-----END EC PRIVATE KEY-----"

# Or via live process inspection
ps aux | grep rate-limiting-canister-client
# Full command line including the private key is visible

# Attacker reuses the key to submit malicious rate-limiting rules
rate-limiting-canister-client \
  --canister-id <canister_id> \
  --config-file malicious_rules.json \
  --identity-key "<stolen_key>"
```
