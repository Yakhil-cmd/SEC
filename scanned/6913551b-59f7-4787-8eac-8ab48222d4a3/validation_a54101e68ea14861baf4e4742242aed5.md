### Title
Cross-Subaccount Allowance Enumeration via Incomplete Break Condition in `get_allowances` — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

### Summary
The `get_allowances` loop terminates only when `account_spender.account.owner != from.owner`, but the `StableBTreeMap` is ordered by full `Account` (owner + effective_subaccount). After exhausting allowances for the queried subaccount, iteration silently continues into adjacent subaccounts of the same owner. Any unprivileged caller can enumerate all allowances across every subaccount of a target principal.

### Finding Description
`get_allowances` in `rs/ledger_suite/icrc1/ledger/src/lib.rs` (line 1174) iterates a `StableBTreeMap<AccountSpender, ...>` starting from `{account: from, spender: <min>}`. The loop's only exit guard is:

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

Because `AccountSpender::Ord` sorts by the full `Account` (owner then effective_subaccount), once all spenders for `{owner:X, sub:[0;32]}` are yielded, the iterator advances to `{owner:X, sub:[1;32]}`, `{owner:X, sub:[2;32]}`, etc. The guard never fires because the owner field is unchanged. The correct guard would compare the full `Account`:

```rust
if account_spender.account != from {
    break;
}
```

The public `icrc103_get_allowances` query endpoint (line 1214 of `rs/ledger_suite/icrc1/ledger/src/main.rs`) accepts an arbitrary `from_account` with no caller restriction, making this reachable by any unprivileged principal. [5](#0-4) 

### Impact Explanation
An attacker who knows (or guesses) a target principal X can call `icrc103_get_allowances` with `from_account = {owner:X, subaccount:Some([0u8;32])}` and receive allowances belonging to every subaccount of X that has ever called `icrc2_approve`. The response includes spender identity, approved amount, and expiry for each allowance — information that the subaccount owners may consider private. No funds can be moved; the impact is confined to allowance metadata disclosure.

### Likelihood Explanation
The endpoint is a public query, callable by any principal with no authentication. The only precondition is that the target principal has approved from more than one subaccount. This is a straightforward single-call exploit requiring no special privileges, timing, or state manipulation.

### Recommendation
Replace the owner-only comparison with a full `Account` equality check:

```rust
// Before (line 1204):
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```

`Account::PartialEq` already uses `effective_subaccount()` for comparison, so this correctly handles both `subaccount: None` and `subaccount: Some([0u8;32])` as equivalent. [6](#0-5) 

### Proof of Concept

fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    let max_take_allowances = Access::with_ledger(|ledger| ledger.max_take_allowances());
    let max_results = arg
        .take
        .map(|take| take.0.to_u64().unwrap_or(max_take_allowances))
        .map(|take| std::cmp::min(take, max_take_allowances))
        .unwrap_or(max_take_allowances);
    Ok(get_allowances(
        from_account,
        arg.prev_spender,
        max_results,
        ic_cdk::api::time(),
    ))
