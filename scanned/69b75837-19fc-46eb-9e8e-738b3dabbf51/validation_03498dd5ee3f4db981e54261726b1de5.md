### Summary

The ckETH/ckERC20 minter's deposit blocklist check operates exclusively on the `from_address` field parsed from Ethereum log events. Both deposit helper contracts emit `msg.sender` â€” the **immediate caller** â€” as `from_address`. A blocked address can deploy an unlisted proxy contract, route funds through it, and cause the helper to emit the proxy's address instead of the blocked address. The minter then checks `is_blocked(proxy)` â†’ `false` and mints ckETH/ckERC20 to the sanctioned party.

### Finding Description

**Step 1 â€” Solidity emits `msg.sender`, not `tx.origin`**

```solidity

``` [1](#0-0)

In Solidity, `msg.sender` is the **immediate caller** of the function. If a proxy contract `P` calls `deposit()`, then `msg.sender == address(P)`, regardless of who called `P`.

### Impact Explanation

A sanctioned/blocked Ethereum address can receive minted ckETH or ckERC20 tokens on the Internet Computer by routing deposits through a freshly deployed, unlisted proxy contract. This directly violates the stated invariant: *"ETH is not accepted from nor sent to addresses on this list."* [9](#0-8) 

---

### Likelihood Explanation

The attack requires only:
1. Deploying a proxy contract on Ethereum (permissionless, costs ~$5â€“$50 in gas).
2. Transferring ETH or pre-approving ERC20 tokens to the proxy.
3. Calling `deposit()` / `depositErc20()` from the proxy.

No privileged access, no key compromise, no governance majority. Any blocked address can execute this immediately.

---

### Recommendation

The minter cannot retrieve `tx.origin` from log data (it is not emitted). Mitigations include:

1. **Emit and check `tx.origin`**: Modify the Solidity helpers to emit `tx.origin` alongside `msg.sender` and have the minter check both against the blocklist.
2. **Dual-field blocklist check**: If both `msg.sender` and `tx.origin` are emitted, reject the deposit if either is blocked.
3. **Acknowledge as a design limitation**: Document that the blocklist only covers direct callers and cannot prevent proxy-mediated deposits â€” but this would mean the compliance guarantee is weaker than stated.

Option 1 is the most robust fix, though it requires a coordinated upgrade of both the Solidity helper contracts and the Rust minter parser.

### Proof of Concept

```solidity
// ProxyDeposit.sol â€” deployed by blocked address A
contract ProxyDeposit {
    CkEthDeposit helper = CkEthDeposit(HELPER_ADDRESS);

    function doDeposit(bytes32 principal) external payable {
        // msg.sender inside helper.deposit() == address(this), NOT address(A)
        helper.deposit{value: msg.value}(principal);
    }
}
```

Attack flow:
1. Blocked address `A` deploys `ProxyDeposit` â†’ address `P` (not on blocklist).
2. `A` calls `P.doDeposit{value: 1 ether}(principal_of_A)`.
3. `P` calls `helper.deposit{value: 1 ether}(principal_of_A)`.
4. Helper emits `ReceivedEth(P, 1 ether, principal_of_A)`.
5. Minter scrapes log, parses `from_address = P`.
6. `is_blocked(P)` â†’ `false`.
7. Minter mints 1 ckETH to `principal_of_A` â€” controlled by blocked address `A`.

The same pattern applies to ERC20 via `depositErc20` in `DepositHelperWithSubaccount.sol`. [10](#0-9)
