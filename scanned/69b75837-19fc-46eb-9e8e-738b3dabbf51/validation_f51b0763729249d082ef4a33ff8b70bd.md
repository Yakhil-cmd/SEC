### Title
Silent Clamping of Oversized `icrc2_approve` Amount to `U256::MAX` Instead of Rejection — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

---

### Summary

`icrc2_approve_not_async` silently converts any `amount` that overflows `U256` into `Tokens::max_value()` (= `U256::MAX`, i.e. 2^256−1) rather than returning an error. A caller submitting `amount = 2^256` expects a rejection; instead the ledger records an approval for the maximum possible token amount, which a spender can immediately exploit to drain the approver's full balance.

---

### Finding Description

**Root cause — line 840 of `rs/ledger_suite/icrc1/ledger/src/main.rs`:**

```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```

`TryFrom<Nat> for U256` (in `rs/ledger_suite/icrc1/tokens_u256/src/lib.rs`) correctly rejects any value whose little-endian byte representation exceeds 32 bytes:

```rust
if le_bytes.len() > 32 {
    return Err(format!("amount {n} does not fit into u256 token type"));
}
```

`Nat(2^256)` encodes to 33 little-endian bytes, so `TryFrom` returns `Err`. The call site swallows that error and substitutes `Tokens::max_value()` = `U256::MAX`. No `ApproveError` is ever returned to the caller; the ledger proceeds to record an approval for `U256::MAX` tokens.

**Contrast with `expected_allowance` handling (lines 841–853):** the same `TryFrom` failure on `expected_allowance` correctly surfaces `ApproveError::AllowanceChanged`. The inconsistency is the bug. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Once the approval for `U256::MAX` is on-chain, the designated spender can call `icrc2_transfer_from` repeatedly (up to the approver's actual balance) and drain the approver's account entirely. The approver's actual intent (pass an out-of-range value, expect rejection) is violated; the ledger records a semantically different, maximally permissive approval without the approver's knowledge.

---

### Likelihood Explanation

The attack is reachable by any unprivileged principal via the public `icrc2_approve` update call. No admin key, governance majority, or subnet compromise is required. A malicious dApp frontend can craft the oversized `Nat` argument and present the call to a user as "this will fail — just sign to confirm the rejection," then exploit the silently-recorded `U256::MAX` approval. The only precondition is that the approver holds enough balance to pay the fee (so the fee-deduction step in `apply_transaction` succeeds). [4](#0-3) [5](#0-4) 

---

### Recommendation

Replace the silent clamp with an explicit error return:

```rust
// Before (line 840):
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());

// After:
let amount = Tokens::try_from(arg.amount).map_err(|_| ApproveError::GenericError {
    error_code: Nat::from(0_u64),
    message: "amount exceeds maximum token value".to_string(),
})?;
```

This mirrors the already-correct handling of `expected_allowance` and ensures the ledger never records an approval the caller did not explicitly request. [6](#0-5) 

---

```rust
    from: victim_account,
    to: attacker_account,
    amount: victim_balance,
    ..Default::default()
});
assert!(drain.is_ok()); // succeeds — full balance drained
```

```rust
impl TryFrom<Nat> for U256 {
    type Error = String;

    fn try_from(n: Nat) -> Result<Self, Self::Error> {
        let le_bytes = n.0.to_bytes_le();
        if le_bytes.len() > 32 {
            return Err(format!("amount {n} does not fit into u256 token type"));
        }
        let mut bytes = [0_u8; 32];
        bytes[0..le_bytes.len()].copy_from_slice(&le_bytes[..]);
        Ok(Self::new(u256::from_le_bytes(bytes)))
    }
```
