### Title
Missing Slippage Protection in SNS Treasury Manager `deposit` and `withdraw` Operations - (File: `rs/sns/treasury_manager/src/lib.rs`, `rs/sns/governance/src/extensions.rs`)

---

### Summary

The `DepositRequest` and `WithdrawRequest` types in the SNS Treasury Manager interface carry no minimum-output (slippage) parameters. When SNS governance executes a treasury deposit into a DEX liquidity pool, the actual LP tokens received — or assets returned on withdrawal — are entirely at the mercy of the pool's price at execution time. A malicious canister or any unprivileged actor with sufficient capital can sandwich the governance-executed deposit, causing the SNS DAO treasury to receive far fewer LP tokens than the proposal implied, with no on-chain guard to abort the transaction.

---

### Finding Description

The `DepositRequest` struct in `rs/sns/treasury_manager/src/lib.rs` contains only a list of `allowances`:

```rust
pub struct DepositRequest {
    pub allowances: Vec<Allowance>,
}
``` [1](#0-0) 

There is no `min_lp_tokens_out`, `min_shares_minted`, or any equivalent slippage bound. The same is true for `WithdrawRequest`:

```rust
pub struct WithdrawRequest {
    pub withdraw_accounts: Option<BTreeMap<Principal, Account>>,
}
``` [2](#0-1) 

No `min_assets_out` field exists. The Candid interface mirrors this exactly: [3](#0-2) 

The codebase itself acknowledges this as a **Known Security Risk** in the interface specification:

> *"Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."* [4](#0-3) 

The proposal rendering code in `rs/sns/governance/src/proposal.rs` also warns:

> *"Some Decentralized Exchanges lack slippage protection during deposits … making them vulnerable to front-running or sandwich attacks."* [5](#0-4) 

Despite these acknowledged risks, the `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` calls `deposit` on the treasury manager canister with no post-call slippage check — it only logs the returned `Balances` and returns `Ok(())`: [6](#0-5) 

The proposal validation step (`validate_deposit_operation_impl`) only enforces that the requested amounts do not exceed 50% of the current treasury balance. It performs no check on the expected LP tokens to be received: [7](#0-6) 

---

### Impact Explanation

**Impact: Medium**

An attacker can cause the SNS DAO treasury to receive significantly fewer LP tokens than the governance proposal implied, or to receive fewer assets back on withdrawal. Because the `DepositRequest` carries no `min_lp_tokens_out` field, neither the SNS Governance canister nor the Treasury Manager canister can abort the deposit if the DEX price has moved adversarially. The 50%-of-balance cap limits the maximum loss per proposal execution, but the attack is repeatable across multiple proposals. SNS DAO treasury funds (ICP and SNS tokens) are directly at risk.

---

### Likelihood Explanation

**Likelihood: Medium**

Any unprivileged canister or boundary-node user with sufficient capital can monitor the SNS governance canister for pending `ExecuteExtensionOperation` proposals targeting a DEX-backed Treasury Manager. Once a proposal passes and enters the execution queue, the attacker can front-run the deposit by manipulating the DEX pool's price ratio before the governance canister's `execute_treasury_manager_deposit` call lands. On the Internet Computer, inter-canister calls are asynchronous and subject to scheduling delays, providing a realistic window for such manipulation. No privileged access, key compromise, or subnet-majority corruption is required.

---

### Recommendation

1. **Add a `min_lp_tokens_out` field to `DepositRequest`** and a `min_assets_out` field to `WithdrawRequest` in `rs/sns/treasury_manager/src/lib.rs` and `rs/sns/treasury_manager/treasury_manager.did`. Treasury Manager implementations must enforce these bounds before calling the DEX.

2. **Enforce the slippage bound in `execute_treasury_manager_deposit`** (`rs/sns/governance/src/extensions.rs`): after the `deposit` call returns, compare the `external_custodian` balance in the returned `Balances` against the proposer-specified minimum. If the minimum is not met, treat the operation as failed.

3. **Include the minimum LP tokens in the governance proposal payload** so that SNS token holders can evaluate the acceptable slippage when voting.

---

### Proof of Concept

1. An SNS DAO submits an `ExecuteExtensionOperation` proposal with `operation_name = "deposit"` and `treasury_allocation_sns_e8s = X`, `treasury_allocation_icp_e8s = Y` targeting a DEX-backed Treasury Manager.
2. The proposal passes the 50%-balance validation in `validate_deposit_operation_impl` and is approved by voters.
3. Before the governance canister executes the proposal, the attacker calls the DEX canister directly to swap a large amount of one token for the other, skewing the pool's price ratio.
4. `execute_treasury_manager_deposit` calls `approve_treasury_manager` (granting ICRC-2 allowances) and then calls `deposit` on the Treasury Manager canister.
5. The Treasury Manager deposits into the DEX at the manipulated price, receiving far fewer LP tokens than the proposal implied.
6. The attacker reverses their swap, profiting from the price impact while the SNS treasury holds devalued LP tokens.
7. No check in `execute_treasury_manager_deposit` or `validate_deposit_operation_impl` detects or reverts this outcome. [8](#0-7) [1](#0-0) [4](#0-3)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
