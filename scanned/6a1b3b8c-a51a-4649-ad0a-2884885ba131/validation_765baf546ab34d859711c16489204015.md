### Title
Hardcoded Stale Token-Price Assumption in `COST_OF_ONE_BILLION_CYCLES` Causes Systematic Under-Charging of Minter Reimbursement Fees - (File: `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

---

### Summary

The ckDOGE minter hardcodes a price assumption for DOGE that is used to convert IC cycles cost into a DOGE-denominated reimbursement fee. The constant `COST_OF_ONE_BILLION_CYCLES = 5_000_000` koinu (0.05 DOGE) is derived from the assumption that "50 DOGE = 1 XDR." This assumption is baked in at compile time with no on-chain governance mechanism to update it. When the real DOGE/XDR price diverges significantly from this assumption — which it already has — the minter systematically under-charges users for failed withdrawal reimbursements, causing the minter canister to absorb real cycles costs that are not recovered from users.

---

### Finding Description

In `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`, the `DogecoinFeeEstimator` struct defines:

```rust
/// Cost in koinu of 1B cycles.
///
/// Use a lower bound on the price of Doge of 50 DOGE = 1 XDR, so that 5M koinus correspond to 1B cycles.
pub const COST_OF_ONE_BILLION_CYCLES: u64 = 5_000_000;
```

This constant is used in `reimbursement_fee_for_pending_withdrawal_requests`:

```rust
fn reimbursement_fee_for_pending_withdrawal_requests(&self, num_requests: u64) -> u64 {
    num_requests.saturating_mul(Self::COST_OF_ONE_BILLION_CYCLES)
}
```

This function is called in the ckBTC minter's `submit_pending_requests` (shared with ckDOGE via the shared minter library) when a batch of withdrawal requests cannot be processed (e.g., `BuildTxError::InvalidTransaction` / `TooManyInputs`). The reimbursement fee is deducted from the user's refund to compensate the minter for cycles spent.

The price assumption "50 DOGE = 1 XDR" was set at the time of the ckDOGE minter's initial deployment (January 2026, per `rs/dogecoin/ckdoge/mainnet/proposals/minter_install_2026_01_30.md`). At that time, DOGE was approximately $0.35 USD and 1 XDR ≈ $1.33 USD, making 1 XDR ≈ 3.8 DOGE — already far below the assumed 50 DOGE/XDR. The constant is therefore already stale by a factor of ~13x at deployment, meaning the minter charges users only ~1/13th of the actual cycles cost for failed withdrawals.

Unlike the ckBTC minter's `COST_OF_ONE_BILLION_CYCLES = 10` satoshi (which uses a conservative lower bound of 1 BTC = 10,000 XDR, a floor that BTC has historically stayed well above), the ckDOGE constant uses a floor of 50 DOGE = 1 XDR that is already violated by the actual market price. There is no governance-callable update path for this constant — it is a compile-time `const` with no upgrade argument to change it.

The analogous ckBTC constant in `rs/bitcoin/ckbtc/minter/src/fees/mod.rs` has the same structural issue but is less immediately impactful because BTC's price floor assumption is more conservative.

---

### Impact Explanation

When a ckDOGE withdrawal request fails (e.g., due to `TooManyInputs`), the minter reimburses the user minus `COST_OF_ONE_BILLION_CYCLES` per request to cover cycles spent. With the constant set to 5,000,000 koinu (0.05 DOGE ≈ $0.017 at $0.35/DOGE), but the actual cycles cost being ~13x higher in real terms, the minter canister absorbs the difference. Over many failed withdrawals, this creates a systematic cycles drain on the minter canister. The minter's cycles balance is depleted faster than expected, potentially causing the canister to run out of cycles and halt operations — freezing all ckDOGE deposits and withdrawals for all users. Any unprivileged user can trigger this path by submitting a large withdrawal that requires more UTXOs than `max_num_inputs_in_transaction`, causing the minter to process and reimburse the request at a loss.

---

### Likelihood Explanation

The entry path is fully reachable by any unprivileged user: call `retrieve_doge_with_approval` with an amount large enough to require more UTXOs than the configured maximum. The minter will attempt to build the transaction, fail with `TooManyInputs`, and reimburse the user while deducting only the under-priced `COST_OF_ONE_BILLION_CYCLES`. The DOGE price assumption is already violated at mainnet deployment, making this an active issue rather than a theoretical one. Repeated triggering by a single attacker or organic high-volume usage both lead to the same outcome.

---

### Recommendation

1. Make `COST_OF_ONE_BILLION_CYCLES` a runtime-configurable parameter in the minter state, settable via `UpgradeArgs` (analogous to how `retrieve_doge_min_amount` and `deposit_doge_min_amount` are already configurable).
2. Alternatively, derive the cost dynamically from the CMC's `get_average_icp_xdr_conversion_rate` and an on-chain DOGE/XDR price oracle, similar to how the SNS governance treasury limits use `get_average_icp_xdr_conversion_rate`.
3. At minimum, update the hardcoded constant to reflect a realistic lower bound on DOGE's price (e.g., 3 DOGE = 1 XDR instead of 50 DOGE = 1 XDR).

---

### Proof of Concept

1. Deploy ckDOGE minter on mainnet (already deployed at `eqltq-xqaaa-aaaar-qb3vq-cai`).
2. Deposit enough small DOGE UTXOs to the minter so that a single withdrawal requires more than `max_num_inputs_in_transaction` (500) UTXOs.
3. Call `retrieve_doge_with_approval` for an amount requiring >500 UTXOs.
4. The minter calls `reimbursement_fee_for_pending_withdrawal_requests(1)` which returns `5_000_000` koinu (0.05 DOGE ≈ $0.017).
5. The actual cycles consumed by the minter for building the transaction candidate and burning on the ledger is ~1B cycles ≈ 0.65 DOGE at current prices.
6. The minter reimburses `withdrawal_amount - 5_000_000` koinu, absorbing ~0.60 DOGE in unrecovered cycles cost per failed request.
7. Repeat to drain the minter's cycles balance.
