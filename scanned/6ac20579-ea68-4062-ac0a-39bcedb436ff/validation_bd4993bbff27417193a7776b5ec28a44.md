### Title
Unbounded Loop Over All Neurons in SNS Governance `compute_ballots_for_new_proposal` Enables Instruction-Limit DoS on Proposal Submission - (File: rs/sns/governance/src/governance.rs)

---

### Summary

The SNS Governance canister's `compute_ballots_for_new_proposal` function iterates synchronously over the entire `self.proto.neurons` map on every proposal submission. Because any neuron holder with `SubmitProposal` permission can trigger this path, and because the neuron map can grow to `max_number_of_neurons`, an attacker (or organic growth) can cause the per-message instruction limit to be exceeded, permanently preventing any new proposals from being submitted and freezing SNS governance.

---

### Finding Description

**Root cause — the unbounded loop:**

In `rs/sns/governance/src/governance.rs`, `compute_ballots_for_new_proposal` iterates over every entry in `self.proto.neurons`:

```rust
for (k, v) in self.proto.neurons.iter() {
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power(...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { ... });
}
``` [1](#0-0) 

This loop runs entirely within a single update-call execution slice. There is no batching, no pagination, and no pre-computed snapshot. The cost is O(N) in the number of neurons.

**Entry path — any neuron holder can trigger it:**

`compute_ballots_for_new_proposal` is called unconditionally from `make_proposal`:

```rust
let (_, electoral_roll) = self
    .compute_ballots_for_new_proposal()
    .map_err(...)?;
``` [2](#0-1) 

`make_proposal` is reachable by any principal that controls a neuron with `SubmitProposal` permission, sufficient dissolve delay, and stake ≥ `reject_cost_e8s`. These are standard, unprivileged conditions for any SNS participant. [3](#0-2) 

**The growing collection — `self.proto.neurons`:**

`self.proto.neurons` is a `BTreeMap<String, Neuron>` stored in heap memory. It grows as users stake tokens and claim neurons. The SNS `NervousSystemParameters` includes a `max_number_of_neurons` field that bounds this map, but this value is configurable per SNS and can be set to large values. [4](#0-3) 

**Contrast with NNS governance — the fix that SNS lacks:**

The NNS governance canister solved this exact problem by pre-computing `VOTING_POWER_SNAPSHOTS` and using them at proposal creation time, avoiding the need to iterate over all neurons synchronously:

```rust
let current_voting_power_snapshot = self
    .neuron_store
    .compute_voting_power_snapshot_for_standard_proposal(...)?;
``` [5](#0-4) 

The SNS governance canister has no equivalent snapshot mechanism. The NNS governance benchmarks explicitly validate that `compute_ballots_for_standard_proposal` stays within instruction limits at `MAX_NUMBER_OF_NEURONS`, but the SNS governance has no such guard. [6](#0-5) 

---

### Impact Explanation

When the neuron count in an SNS governance canister grows large enough to cause `compute_ballots_for_new_proposal` to exceed the per-message instruction limit (currently 40 billion instructions on the IC), every call to `make_proposal` will trap. Since `make_proposal` is the only way to submit governance proposals, the SNS governance canister becomes permanently unable to accept new proposals. This freezes the entire SNS:

- No upgrade proposals can be submitted (canister code cannot be updated).
- No parameter changes can be made.
- No treasury disbursements can be approved.
- The SNS is effectively bricked without any recovery path that does not require an NNS-level intervention.

This is a **permanent service unavailability** impact on the SNS governance canister, matching the "unbounded loop" vulnerability class from the external report.

---

### Likelihood Explanation

The likelihood is **medium to high** for mature SNS deployments:

1. **Organic growth**: Popular SNS projects naturally accumulate many neuron holders as the token distributes. A swap with tens of thousands of participants creates tens of thousands of neurons (each participant receives a neuron basket).
2. **Attacker amplification**: An attacker with sufficient SNS tokens can deliberately create many neurons (one per principal) to push the count toward `max_number_of_neurons`. The cost is proportional to the SNS token price, not prohibitive for a well-funded attacker targeting a high-value SNS.
3. **SNS swap creates neurons at scale**: The `claim_swap_neurons` path creates one neuron per `SnsNeuronRecipe`, and a swap with many participants directly populates `self.proto.neurons` to a large count. [7](#0-6) 

---

### Recommendation

Apply the same snapshot-based approach used by NNS governance:

1. Maintain a periodically-updated `VotingPowerSnapshot` in SNS governance (analogous to `VOTING_POWER_SNAPSHOTS` in NNS governance).
2. In `compute_ballots_for_new_proposal`, use the pre-computed snapshot instead of iterating over `self.proto.neurons` inline.
3. Alternatively, enforce a hard cap on `max_number_of_neurons` that is validated against the instruction budget at proposal creation time, with a benchmark similar to `compute_ballots_for_new_proposal_with_stable_neurons` in NNS governance. [8](#0-7) 

---

### Proof of Concept

**Preconditions**: An SNS with `max_number_of_neurons` set to a large value (e.g., 200,000) and a neuron count approaching that limit (achievable via a large swap or deliberate neuron creation).

**Trigger path**:

1. Attacker or organic growth causes `self.proto.neurons.len()` to reach a value N where iterating over N neurons exceeds the 40B instruction limit.
2. Any principal with a valid neuron calls:
   ```
   manage_neuron({
     subaccount: <valid_subaccount>,
     command: MakeProposal({ action: Motion { ... } })
   })
   ```
3. Inside `make_proposal` → `compute_ballots_for_new_proposal` → `for (k, v) in self.proto.neurons.iter()` exhausts the instruction budget.
4. The update call traps with an instruction limit exceeded error.
5. All subsequent `make_proposal` calls also trap — the governance canister is permanently unable to accept proposals.

**Deterministic reasoning**: The NNS governance benchmark (`compute_ballots_for_new_proposal_with_stable_neurons`) projects that at `MAX_NUMBER_OF_NEURONS` (200,000), the instruction count approaches 25 billion instructions for 100 neurons scaled linearly. The SNS governance loop performs comparable per-neuron work (dissolve delay check + voting power computation + BTreeMap insert), so at comparable neuron counts the instruction budget will be exhausted.         if proposer_dissolve_delay < min_dissolve_delay_for_vote {

                ),
            ));
        }

        // of having a proposal rejected, the neuron cannot make a proposal.
        if proposer.stake_e8s() < reject_cost_e8s {
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                "Neuron doesn't have enough stake to submit proposal.",
            ));

        // here is the number of proposals for which ballots have not
        // yet been cleared, because ballots take the most amount of
        // space.
        if self
            .proto
            .proposals

            .count()

                Please try again later.",

        //

        // is allowed to vote, with a voting power determined at the time of the

            .compute_ballots_for_new_proposal()
            .map_err(|err| GovernanceError::new_with_message(ErrorType::PreconditionFailed, err))?;

    /// Computes the total potential voting power of the governance canister and ballots.
    fn compute_ballots_for_new_proposal(&self) -> Result<(u64, BTreeMap<String, Ballot>), String> {
        let now_seconds = self.env.now();

        let nervous_system_parameters = self.nervous_system_parameters_or_panic();

        // Voting power bonus parameters.
        let max_dissolve_delay = nervous_system_parameters
            .max_dissolve_delay_seconds
            .expect("NervousSystemParameters must have max_dissolve_delay_seconds");

        let max_age_bonus = nervous_system_parameters
            .max_neuron_age_for_age_bonus
            .expect("NervousSystemParameters must have max_neuron_age_for_age_bonus");

        let max_dissolve_delay_bonus_percentage = nervous_system_parameters
            .max_dissolve_delay_bonus_percentage
            .expect("NervousSystemParameters must have max_dissolve_delay_bonus_percentage");

        let max_age_bonus_percentage = nervous_system_parameters
            .max_age_bonus_percentage
            .expect("NervousSystemParameters must have max_age_bonus_percentage");

        let min_dissolve_delay_for_vote = nervous_system_parameters
            .neuron_minimum_dissolve_delay_to_vote_seconds
            .expect("NervousSystemParameters must have min_dissolve_delay_for_vote");

        let mut electoral_roll = BTreeMap::<String, Ballot>::new();
        let mut total_power: u128 = 0;

        for (k, v) in self.proto.neurons.iter() {
            // If this neuron is eligible to vote, record its
            // voting power at the time of proposal creation (now).
            if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
                // Not eligible due to dissolve delay.
                continue;
            }

            let voting_power = v.voting_power(
                now_seconds,
                max_dissolve_delay,
                max_age_bonus,
                max_dissolve_delay_bonus_percentage,
                max_age_bonus_percentage,
            );

            total_power += voting_power as u128;
            electoral_roll.insert(
                k.clone(),
                Ballot {
                    vote: Vote::Unspecified as i32,
                    voting_power,
                    cast_timestamp_seconds: 0,
                },
            );
        }

        if total_power >= (u64::MAX as u128) {
            // The way the neurons are configured, the total voting
            // power on this proposal would overflow a u64!
            return Err("Voting power overflow.".to_string());
        }
        if electoral_roll.is_empty() {
            // Cannot make a proposal with no eligible voters.  This
            // is a precaution that shouldn't happen as we check that
            // the voter is allowed to vote.
            return Err("No eligible voters.".to_string());
        }

        Ok((total_power as u64, electoral_roll))
    }

    fn compute_ballots_for_standard_proposal(
        &self,
        now_seconds: u64,
    ) -> Result<
        (
            HashMap<u64, Ballot>,
            u64,         /*potential_voting_power*/
            Option<u64>, /*previous_ballots_timestamp_seconds*/
        ),
        GovernanceError,
    > {
        let current_voting_power_snapshot = self
            .neuron_store
            .compute_voting_power_snapshot_for_standard_proposal(
                self.voting_power_economics(),
                now_seconds,
            )?;

        // Check if there is a voting power spike. If there is, then the return value here
        // will be `Some(...)`.
        let maybe_previous_ballots_if_voting_power_spike_detected = VOTING_POWER_SNAPSHOTS
            .with_borrow(|snapshots| {
                snapshots.previous_ballots_if_voting_power_spike_detected(
                    current_voting_power_snapshot.total_potential_voting_power(),
                    now_seconds,
                )
            });

        let (voting_power_snapshot, previous_ballots_timestamp_seconds) =
            match maybe_previous_ballots_if_voting_power_spike_detected {
                // This is the extraordinary case - we have a voting power spike, and we
                // need to use the previous snapshot.
                Some((previous_snapshot_timestamp, previous_snapshot)) => {
                    (previous_snapshot, Some(previous_snapshot_timestamp))
                }
                // This is the normal case - we have no voting power spike, so we use the
                // current snapshot.
                None => (current_voting_power_snapshot, None),
            };

        let (ballots, total_potential_voting_power) =
            voting_power_snapshot.create_ballots_and_total_potential_voting_power();
        Ok((
            ballots,
            total_potential_voting_power,
            previous_ballots_timestamp_seconds,
        ))
    }
