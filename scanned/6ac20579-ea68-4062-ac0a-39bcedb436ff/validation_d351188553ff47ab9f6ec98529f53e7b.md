### Title
Unbounded Iteration in `get_pending_proposals` Enables Query Instruction-Limit Exhaustion - (File: `rs/nns/governance/src/governance.rs`)

---

### Summary

The NNS Governance canister exposes a `get_pending_proposals` query endpoint that iterates over **every entry** in the entire proposals map without any pagination or result-count cap. This is the direct IC analog of the RabbitHole `getOwnedTokenIdsOfQuest` unbounded-loop vulnerability: a publicly reachable function performs O(N) work over a collection that an unprivileged actor can grow, with no upper bound on the per-call instruction cost.

---

### Finding Description

`get_pending_proposals` is registered as a `#[query]` handler in the NNS Governance canister: [1](#0-0) 

Its implementation unconditionally iterates over the full `heap_data.proposals` BTreeMap, filters for `ProposalStatus::Open`, and calls `proposal_data_to_info` on every matching entry before collecting the entire result into a `Vec`: [2](#0-1) 

There is no `limit`, `take(n)`, or pagination parameter of any kind. Compare this with `list_proposals`, which explicitly caps results at `MAX_LIST_PROPOSAL_RESULTS`: [3](#0-2) 

`list_known_neurons` has the same structural problem — it iterates over every known-neuron ID and fetches full neuron data with no bound: [4](#0-3) 

---

### Impact Explanation

Each call to `get_pending_proposals` must:
1. Scan the entire `heap_data.proposals` BTreeMap.
2. Call `proposal_data_to_info` for every open proposal — a non-trivial operation that processes ballot maps, payload bytes, and voting-period arithmetic.

As the number of simultaneously open proposals grows toward the system's `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` ceiling, the per-query instruction cost grows proportionally. If that ceiling is reached (or approached), the query can exhaust the IC instruction limit for a single message, causing it to be rejected with `CanisterInstructionLimitExceeded`. This makes the `get_pending_proposals` endpoint — which is relied upon by Rosetta, governance dashboards, and the Rosetta client library — permanently unavailable until proposals close. [5](#0-4) 

---

### Likelihood Explanation

Any principal holding a neuron with sufficient dissolve delay can submit NNS proposals. The NNS currently has hundreds of thousands of neurons. A coordinated set of neuron holders (or a single holder with many neurons) can submit proposals across all available NNS function types simultaneously, keeping a large number of proposals in `Open` state for the full voting period (~4 days). Because `get_pending_proposals` is a query (not an update), it does not consume cycles from the governance canister, so the attacker bears no on-chain cost beyond the initial proposal submission fees. The Rosetta API and governance front-ends call this endpoint continuously, so even a transient instruction-limit failure causes observable service disruption.

---

### Recommendation

Apply the same fix pattern used by `list_proposals`: add a `limit` / `before_proposal` pagination parameter to `get_pending_proposals` and cap the result set at `MAX_LIST_PROPOSAL_RESULTS`. The existing `GetPendingProposalsRequest` struct already provides an extension point:

```rust
// rs/nns/governance/src/governance.rs
pub fn get_pending_proposals(
    &self,
    caller: &PrincipalId,
    req: Option<GetPendingProposalsRequest>,
) -> Vec<ProposalInfo> {
    let limit = /* read from req, default MAX_LIST_PROPOSAL_RESULTS */;
    self.heap_data
        .proposals
        .values()
        .filter(|data| data.status() == ProposalStatus::Open)
        .take(limit)          // <-- add this
        .map(|data| proposal_data_to_info(...))
        .collect()
}
```

Apply the same treatment to `list_known_neurons`.

---

### Proof of Concept

1. An attacker controls neurons across multiple principals.
2. The attacker submits proposals of every available `NnsFunction` type, keeping them open for the full voting period.
3. A legitimate caller (e.g., the Rosetta API server) calls `get_pending_proposals`.
4. The governance canister iterates over all open proposals, calling `proposal_data_to_info` for each. With a large enough open-proposal count, the query hits `CanisterInstructionLimitExceeded`.
5. The Rosetta `/construction/submit` flow that depends on `pending_proposals()` fails, and governance dashboards stop showing pending proposals. [6](#0-5) [5](#0-4)

    pub fn get_pending_proposals(
        &self,
        caller: &PrincipalId,
        req: Option<GetPendingProposalsRequest>,
    ) -> Vec<ProposalInfo> {
        let now = self.env.now();
        let caller_neurons = self.get_neuron_ids_by_principal(caller);
        let return_self_describing_action = req
            .and_then(|r| r.return_self_describing_action)
            .unwrap_or(false);
        self.heap_data
            .proposals
            .values()
            .filter(|data| data.status() == ProposalStatus::Open)
            .map(|data| {
                proposal_data_to_info(
                    data,
                    ProposalDisplayOptions::for_get_pending_proposals(
                        return_self_describing_action,
                    ),
                    &caller_neurons,
                    now,
                    self.voting_period_seconds(),
                )
            })
            .collect()
    }

            ApiError::InvalidRequest(

                Details::from(format!("Could not decode PendingProposals response: {err}")),
            )
        })
    }
