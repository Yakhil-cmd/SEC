### Title
`sender_info` Canister Signature Lacks Sender-Principal Binding, Enabling Cross-User Attribute Replay - (File: `rs/validator/src/ingress_validation.rs`)

### Summary

The IC's new `sender_info` field allows a canister (e.g., Internet Identity) to attest to user attributes via a canister signature. The signed message (`SenderInfoContent`) contains only the raw `info` bytes under the domain `"ic-sender-info"` — it does not bind to the `sender` principal, `canister_id`, `method_name`, or `ingress_expiry`. Any unprivileged user who observes a valid `(info, signer, sig)` tuple from another user's request can replay it verbatim in their own request to any canister, causing the receiving canister to believe the signer canister attested to the attacker's identity having the victim's attributes.

### Finding Description

`SenderInfoContent` is defined as a wrapper over raw bytes:

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);  // only the raw info bytes
    }
}
``` [1](#0-0) 

The `SignatureDomain` implementation prepends only `"ic-sender-info"`:

```rust
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
``` [2](#0-1) 

The signed bytes are therefore: `\x0Eic-sender-info || info_bytes` — with no inclusion of the `sender` principal, `canister_id`, `method_name`, or `ingress_expiry`.

The validator in `verify_sender_info_canister_sig` performs three checks:
1. `sender_pubkey` is a valid canister signature public key
2. The canister ID in `sender_pubkey` matches `sender_info.signer`
3. The canister signature over `SenderInfoContent(info_bytes)` is valid [3](#0-2) 

There is **no check** that the `sender` principal in the outer request corresponds to the user for whom the signer canister issued the attestation. The `sender_info` is included in the `MessageId` hash (binding it to the outer envelope signature), but the inner `sender_info.sig` is freely portable across any sender, any canister, and any method. [4](#0-3) 

### Impact Explanation

A canister receiving a request with `sender_info` calls `ic0.msg_caller_info_data()` and `ic0.msg_caller_info_signer()` to read the attested attributes and the attesting canister. If the receiving canister uses these to make authorization decisions (e.g., "Internet Identity attests this caller is a premium user / is over 18 / has KYC status"), an attacker who replays a victim's `sender_info` will cause the canister to grant the attacker the victim's privileges. The attacker's own `sender` principal is correctly reported by `ic0.msg_caller()`, but the attested `info` bytes belong to the victim. Any canister that does not independently re-derive the user's identity from the `info` bytes is vulnerable to this impersonation.

### Likelihood Explanation

The `sender_info` field is part of the public wire format (`HttpCanisterUpdate`). Any request containing a `sender_info` is observable from the public API (boundary nodes, archive nodes). An attacker needs only to:
1. Observe a valid `(info, signer, sig)` tuple from any prior request (e.g., by monitoring the public `/api/v2/canister/.../call` endpoint or reading certified state)
2. Construct a new request with their own `sender` principal and the replayed `sender_info`
3. Sign the new `MessageId` (which includes the replayed `sender_info`) with their own key

No privileged access is required. The attack is fully reachable by an unprivileged ingress sender.

### Recommendation

Bind the `sender_info` canister signature to the `sender` principal (and optionally `canister_id` and `ingress_expiry`) so that the signed message is:

```
"ic-sender-info" || sender_principal_bytes || info_bytes
```

This mirrors the EIP-712 fix in the external report: include enough context in the signed payload to make the signature non-portable across users and contexts. The `verify_sender_info_canister_sig` function in `rs/validator/src/ingress_validation.rs` and the `SenderInfoContent` type in `rs/types/types/src/messages/http.rs` would both need to be updated, along with any signer canister (e.g., Internet Identity) that produces these signatures.

### Proof of Concept

**Setup**: Internet Identity (canister `II`) issues a `sender_info` for User A (principal `P_A`) attesting `info_bytes = b"premium_user"`. User A makes a call to canister X; the `(info, signer=II, sig=S)` tuple is visible on-chain.

**Attack**:
1. Attacker (principal `P_B`) constructs a new `HttpCanisterUpdate` targeting canister Y with:
   - `sender = P_B`
   - `sender_info = { info: b"premium_user", signer: II, sig: S }` (replayed verbatim)
2. Attacker computes `MessageId = representation_independent_hash(canister_id=Y, method=..., sender=P_B, sender_info={info, signer, sig}, ...)` — this is a fresh `MessageId` that the attacker signs with their own key.
3. The replica's `verify_sender_info_canister_sig` verifies that `S` is a valid canister signature over `SenderInfoContent(b"premium_user")` under `II`'s key — **this passes**, because `S` is a valid signature over those bytes regardless of who is sending.
4. Canister Y executes with `msg_caller() = P_B` and `msg_caller_info_data() = b"premium_user"` and `msg_caller_info_signer() = II`.
5. Canister Y grants `P_B` premium access, believing II attested to `P_B` being a premium user. [5](#0-4) [6](#0-5)

```rust
fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
