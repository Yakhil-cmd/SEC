### Title
Stale `maturity_e8s_equivalent` Read in Neuron Operations During Asynchronous Reward Distribution Window - (File: rs/nns/governance/src/reward/distribution.rs)

### Summary
NNS Governance's two-phase asynchronous reward distribution (introduced in proposal 135702) creates a window where neuron operations — `merge_neurons`, `initiate_maturity_disbursement`, and `stake_maturity_of_neuron` — read a stale `maturity_e8s_equivalent` from the neuron store. Rewards that have been calculated and scheduled but not yet applied to neurons are silently excluded from these operations, causing the user's operation to act on less maturity than they actually own.

### Finding Description

**Phase 1 — Reward calculation and scheduling (synchronous):**

`distribute_voting_rewards_to_neurons` calculates per-neuron rewards and calls `schedule_pending_rewards_distribution`, which inserts the reward amounts into a `RewardsDistributionStateMachine` (a stable BTreeMap). The reward event is immediately marked as settled and proposals are cleared. At this point, neurons' `maturity_e8s_equivalent` fields have **not** been updated. [1](#0-0) [2](#0-1) 

**Phase 2 — Reward application (asynchronous, deferred):**

A timer task (`DistributeRewardsTask`) runs every 2 seconds and calls `distribute_pending_rewards` → `continue_processing`, which iterates the pending map and increments each neuron's `maturity_e8s_equivalent` (or `staked_maturity_e8s_equivalent` if auto-stake is on). [3](#0-2) [4](#0-3) 

**The window:**

Between Phase 1 and Phase 2 (up to ~2 seconds, longer if the timer is delayed or multiple reward events are queued), any neuron operation that reads `maturity_e8s_equivalent` will observe a value that does not include the pending rewards.

Affected operations:

1. **`merge_neurons`** — `calculate_merge_neurons_effect` captures `source.maturity_e8s_equivalent` at calculation time. Pending rewards for the source neuron are not included in the transfer to the target. [5](#0-4) 

2. **`initiate_maturity_disbursement`** — reads `neuron.maturity_e8s_equivalent` to compute the disbursement amount. Pending rewards are not included, so the disbursement is smaller than the user's actual accrued maturity. [6](#0-5) 

3. **`stake_maturity_of_neuron`** — reads `maturity_e8s_equivalent` to compute the amount to stake. Pending rewards are not staked. [7](#0-6) 

### Impact Explanation

After the timer fires, the pending rewards are applied to the neuron's `maturity_e8s_equivalent`. The funds are not permanently destroyed, but:

- For **`disburse_maturity`**: the disbursement amount was fixed at initiation using the stale value. The pending rewards land in the neuron's maturity after the timer fires, requiring the user to initiate a second disbursement and wait another 7-day delay before receiving them. A user who intended to disburse 100% of their maturity receives less than 100%.
- For **`merge_neurons`**: the source neuron retains the pending rewards after the merge completes, requiring a second merge operation to move them to the target.
- For **`stake_maturity_of_neuron`**: pending rewards land in unstaked maturity after the timer fires, defeating the user's intent to stake 100% of their maturity.

In all cases the user's intended operation is not fulfilled and additional steps are required to recover the pending rewards.

### Likelihood Explanation

The timer interval is 2 seconds. Any user who submits one of the above `manage_neuron` calls within approximately 2 seconds of a daily reward distribution event will be affected. Because reward distribution is triggered by a periodic timer (not by user action), the user has no way to know whether pending rewards exist in the `RewardsDistributionStateMachine` at the time they submit their call. The window is narrow but deterministic and repeats every reward period. Under high-load conditions or if the timer is delayed, the window can be longer.

### Recommendation

Before reading `maturity_e8s_equivalent` in `calculate_merge_neurons_effect`, `initiate_maturity_disbursement`, and `stake_maturity_of_neuron`, first flush any pending rewards for the affected neuron(s) from the `RewardsDistributionStateMachine`. Alternatively, expose a query that lets users check whether pending rewards exist for their neuron and document that they should wait for the timer to fire before performing maturity-sensitive operations.

### Proof of Concept

1. Reward distribution fires: `distribute_voting_rewards_to_neurons` is called with a supply that yields 500 e8s for neuron N. `schedule_pending_rewards_distribution` stores `{N: 500}` in the `RewardsDistributionStateMachine`. Neuron N's `maturity_e8s_equivalent` is still X (e.g., 1000 e8s). [1](#0-0) 

2. Within the next ~2 seconds, the neuron controller calls `manage_neuron` with `DisburseMaturity { percentage_to_disburse: 100, ... }`.

3. `initiate_maturity_disbursement` reads `maturity_e8s_equivalent = 1000` (stale; the 500 e8s pending reward is not yet applied). The disbursement is initiated for 1000 e8s and the neuron's maturity is set to 0. [8](#0-7) 

4. The `DistributeRewardsTask` timer fires, calls `continue_processing`, and adds 500 e8s to neuron N's `maturity_e8s_equivalent` (now 500 e8s). [9](#0-8) 

5. The disbursement finalizes for 1000 e8s. Neuron N still holds 500 e8s of maturity that was not disbursed. The user must initiate a second disbursement and wait another 7 days to receive the 500 e8s they were entitled to at the time of their original call.

    fn continue_processing(
        &mut self,
        neuron_store: &mut NeuronStore,
        is_over_instructions_limit: fn() -> bool,
    ) {
        while let Some((id, reward_e8s)) = self.rewards.pop_first() {
            match neuron_store.with_neuron_mut(&id, |neuron| {
                let auto_stake = neuron.auto_stake_maturity.unwrap_or(false);
                if auto_stake {
                    neuron.staked_maturity_e8s_equivalent = Some(
                        neuron
                            .staked_maturity_e8s_equivalent
                            .unwrap_or_default()
                            .saturating_add(reward_e8s),
                    );
                } else {
                    neuron.maturity_e8s_equivalent =
                        neuron.maturity_e8s_equivalent.saturating_add(reward_e8s);
                }
            }) {
                Ok(_) => {}
                Err(e) => {
                    println!(
                        "{}Error rewarding neuron {:?} during reward_distribution.\
                    This should not be possible as neuron existence is checked when \
                    rewards are calculated: {}",
                        LOG_PREFIX, id, e
                    );
                }
            };
            if is_over_instructions_limit() {
                break;
            }
        }
    }
