### Title
ckERC20 Withdrawal Permanently Stuck When Ethereum Gas Fees Spike Above Locked `max_transaction_fee` - (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, `rs/ethereum/cketh/minter/src/withdraw.rs`)

---

### Summary

The ckERC20 minter burns a user's ckETH as `max_transaction_fee` at `withdraw_erc20` call time. If Ethereum gas fees subsequently spike above that locked ceiling — due to network congestion, an EVM-RPC outage, or any delay — the withdrawal request cycles indefinitely in the pending queue with no reimbursement and no user-accessible remedy. A sent transaction that cannot be resubmitted at the higher fee is silently abandoned, blocking all subsequent nonces and permanently locking both the user's burned ckETH and ckERC20 tokens.

---

### Finding Description

**Step 1 — Fee locked at withdrawal creation.**

When a user calls `withdraw_erc20`, the minter estimates the current Ethereum gas fee and immediately burns that amount of ckETH from the user's account as `erc20_tx_fee`, storing it as `max_transaction_fee` in the `Erc20WithdrawalRequest`: [1](#0-0) [2](#0-1) 

**Step 2 — Transaction creation fails silently when gas spikes.**

`create_transactions_batch` periodically dequeues pending requests and calls `create_transaction`. For a `CkErc20` request, the function computes `request_max_fee_per_gas = max_transaction_fee / gas_limit` and compares it against the live `actual_min_max_fee_per_gas`. If the live fee exceeds the locked ceiling, it returns `CreateTransactionError::InsufficientTransactionFee`: [3](#0-2) 

The caller in `create_transactions_batch` handles this by rescheduling the request to the back of the queue — no reimbursement, no cancellation, no user notification: [4](#0-3) 

The request loops indefinitely. If gas fees remain elevated, the user's burned ckETH and burned ckERC20 tokens are locked with no exit path.

**Step 3 — Sent transactions that cannot be resubmitted are silently abandoned.**

If a transaction was already sent but gas fees spike enough that resubmission would exceed `allowed_max_transaction_fee` (set from `ckerc20.max_transaction_fee`), `SignedTransactionRequest::resubmit` returns `ResubmitTransactionError::InsufficientTransactionFee`: [5](#0-4) 

`resubmit_transactions_batch` only logs the error and returns — no resubmission, no reimbursement: [6](#0-5) 

Critically, `create_resubmit_transactions` stops at the first failure because all higher-nonce transactions are blocked by the stuck nonce: [7](#0-6) 

This means one stuck ckERC20 withdrawal can cascade and block every subsequent withdrawal for every user sharing the minter's nonce sequence.

**Step 4 — No user-accessible remedy exists.**

There is no endpoint on the minter that allows a user to top up the `max_transaction_fee` of a pending `Erc20WithdrawalRequest`. The `Erc20WithdrawalRequest` struct has no mutable fee field after creation: [2](#0-1) 

The reimbursement mechanism only triggers on ckERC20 ledger burn failure (before queuing) or on Ethereum transaction finalization with a failure receipt — not on the indefinite-reschedule or silent-abandon paths: [8](#0-7) 

---

### Impact Explanation

A user who calls `withdraw_erc20` during a period of normal gas prices, but whose request is delayed (e.g., by an EVM-RPC outage, IC subnet slowdown, or Ethereum congestion spike), suffers:

1. **Permanent ckETH loss**: The gas-fee ckETH burn is irreversible. No reimbursement is issued for the indefinite-reschedule path.
2. **Permanent ckERC20 loss**: The ERC20 token burn is also irreversible. The tokens are gone from the ledger and the Ethereum transfer never executes.
3. **Cascading nonce blockage**: A single stuck ckERC20 sent transaction blocks all subsequent minter withdrawals for all users until the stuck transaction is resolved — which it cannot be without a protocol-level intervention.

This matches the M-9 impact class: a fee set at request creation becomes insufficient due to a post-creation gas spike, causing the user's position (here: their token balances) to suffer permanent loss with no recourse.

---

### Likelihood Explanation

Ethereum gas fee spikes are routine and well-documented. The EVM-RPC canister introduces additional latency (HTTPS outcalls, consensus rounds) between when the user queries `eip_1559_transaction_price` and when the minter actually constructs the transaction. Any spike during that window — or during a period when the minter's timer is delayed — triggers the bug. The ckBTC minter upgrade notes from 2025-06-27 confirm that stuck-withdrawal scenarios due to fee mismatches have already occurred in production: [9](#0-8) 

A prior ckETH minter upgrade also explicitly fixed "ckERC20 withdrawals which were unnecessarily delayed as soon as the estimated transaction fees increased," confirming this class of bug has manifested before: [10](#0-9) 

---

### Recommendation

1. **Add a `top_up_withdrawal_fee` endpoint** that allows the original caller to burn additional ckETH to increase the `max_transaction_fee` of a pending `Erc20WithdrawalRequest`, analogous to the GMX recommendation of allowing execution-fee top-ups.
2. **Add a reimbursement path for the indefinite-reschedule case**: after N failed reschedule attempts (or after a configurable timeout), cancel the request and reimburse both the ckETH gas fee and the ckERC20 tokens to the user.
3. **For the sent-transaction stuck case**: when `ResubmitTransactionError::InsufficientTransactionFee` is returned, trigger a reimbursement rather than silently logging, and unblock the nonce sequence.

---

### Proof of Concept

1. User calls `eip_1559_transaction_price` — minter returns a fee estimate based on current Ethereum gas.
2. User approves minter for that fee amount on the ckETH ledger and calls `withdraw_erc20`. Minter burns `max_transaction_fee` ckETH and `withdrawal_amount` ckERC20.
3. Ethereum gas fees spike 3× (e.g., due to a popular NFT mint or DeFi liquidation cascade).
4. Minter's next heartbeat calls `create_transactions_batch`. `create_transaction` computes `actual_min_max_fee_per_gas > request_max_fee_per_gas` and returns `InsufficientTransactionFee`.
5. `create_transactions_batch` calls `reschedule_withdrawal_request` — request goes to back of queue.
6. Steps 4–5 repeat on every heartbeat. Gas fees remain elevated. The request never executes.
7. User's ckETH and ckERC20 are permanently burned. No reimbursement is issued. No endpoint exists to increase the fee budget.

For the cascading case: if the request had already been sent (step 4 occurs during resubmission), `resubmit_transactions_batch` logs `"Failed to resubmit transaction"` and returns. All subsequent nonces are blocked. Every other user's pending ckERC20 withdrawal is also stuck until a governance upgrade intervenes. [11](#0-10) [12](#0-11) [13](#0-12)

    /// We stop on the first error since if a transaction with nonce n could not be resubmitted
    /// (e.g., the transaction amount does not cover the new fees),
    /// then the next transactions with nonces n+1, n+2, ... are blocked anyway
    /// and trying to resubmit them would only artificially increase their transaction fees.
    pub fn create_resubmit_transactions(
        &self,
        latest_transaction_count: TransactionCount,
        current_gas_fee: GasFeeEstimate,
    ) -> Vec<Result<(LedgerBurnIndex, Eip1559TransactionRequest), ResubmitTransactionError>> {
        // If transaction count at block height H is c > 0, then transactions with nonces
        // 0, 1, ..., c - 1 were mined. If transaction count is 0, then no transactions were mined.
        // The nonce of the first pending transaction is then exactly c.
        let first_pending_tx_nonce: TransactionNonce = latest_transaction_count.change_units();
        let mut transactions_to_resubmit = Vec::new();
        for (nonce, burn_index, signed_tx) in self
            .sent_tx
            .iter()
            .filter(|(nonce, _burn_index, _signed_tx)| *nonce >= &first_pending_tx_nonce)
        {
            let last_signed_tx = signed_tx.last().expect("BUG: empty sent transactions list");
            match last_signed_tx.resubmit(current_gas_fee.clone()) {
                Ok(Some(new_tx)) => {
                    transactions_to_resubmit.push(Ok((*burn_index, new_tx)));
                }
                Ok(None) => {
                    // the transaction fee is still up-to-date but because the transaction did not get included,
                    // we re-send it as is to be sure that it remains known to the mempool and hopefully be included at some point.
                    // Since we always re-send the last non-included transactions in sent_tx, there is nothing to do.
                }
                Err(crate::tx::ResubmitTransactionError::InsufficientTransactionFee {
                    allowed_max_transaction_fee,
                    actual_max_transaction_fee,
                }) => {
                    transactions_to_resubmit.push(Err(
                        ResubmitTransactionError::InsufficientTransactionFee {
                            ledger_burn_index: *burn_index,
                            transaction_nonce: *nonce,
                            allowed_max_transaction_fee,
                            max_transaction_fee: actual_max_transaction_fee,
                        },
                    ));
                    return transactions_to_resubmit;
                }

fn create_transactions_batch(gas_fee_estimate: GasFeeEstimate) {
    for request in read_state(|s| {
        s.eth_transactions
            .withdrawal_requests_batch(WITHDRAWAL_REQUESTS_BATCH_SIZE)
    }) {
        log!(DEBUG, "[create_transactions_batch]: processing {request:?}",);
        let ethereum_network = read_state(State::ethereum_network);
        let nonce = read_state(|s| s.eth_transactions.next_transaction_nonce());
        let gas_limit = estimate_gas_limit(&request);
        match create_transaction(
            &request,
            nonce,
            gas_fee_estimate.clone(),
            gas_limit,
            ethereum_network,
        ) {
            Ok(transaction) => {
                log!(
                    DEBUG,
                    "[create_transactions_batch]: created transaction {transaction:?}",
                );

                mutate_state(|s| {
                    process_event(
                        s,
                        EventType::CreatedTransaction {
                            withdrawal_id: request.cketh_ledger_burn_index(),
                            transaction,
                        },
                    );
                });
            }
            Err(CreateTransactionError::InsufficientTransactionFee {
                cketh_ledger_burn_index: ledger_burn_index,
                allowed_max_transaction_fee: withdrawal_amount,
                actual_max_transaction_fee: max_transaction_fee,

                log!(
                    INFO,

                );
                mutate_state(|s| s.eth_transactions.reschedule_withdrawal_request(request));
            }
        };

impl SignedTransactionRequest {
    pub fn resubmit(
        &self,
        new_gas_fee: GasFeeEstimate,
    ) -> Result<Option<Eip1559TransactionRequest>, ResubmitTransactionError> {
        let transaction_request = self.transaction.transaction();
        let last_tx_price = transaction_request.transaction_price();
        let new_tx_price = last_tx_price
            .clone()
            .resubmit_transaction_price(new_gas_fee);
        if new_tx_price == last_tx_price {
            return Ok(None);
        }

        if new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee() {
            return Err(ResubmitTransactionError::InsufficientTransactionFee {
                allowed_max_transaction_fee: self.resubmission.allowed_max_transaction_fee(),
                actual_max_transaction_fee: new_tx_price.max_transaction_fee(),
            });
        }
        let new_amount = match self.resubmission {
            ResubmissionStrategy::ReduceEthAmount { withdrawal_amount } => {
                withdrawal_amount.checked_sub(new_tx_price.max_transaction_fee())
                    .expect("BUG: withdrawal_amount covers new transaction fee because it was checked before")
            }
            ResubmissionStrategy::GuaranteeEthAmount { .. } => transaction_request.amount,
        };
        Ok(Some(Eip1559TransactionRequest {
            max_priority_fee_per_gas: new_tx_price.max_priority_fee_per_gas,
            max_fee_per_gas: new_tx_price.max_fee_per_gas,
            gas_limit: new_tx_price.gas_limit,
            amount: new_amount,
            ..transaction_request.clone()
        }))
    }
