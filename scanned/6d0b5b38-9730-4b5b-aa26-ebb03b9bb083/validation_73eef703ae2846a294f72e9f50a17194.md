### Title
NNS Governance `disburse_neuron` Burns Neuron Fees Before Confirming Stake Transfer Success - (File: rs/nns/governance/src/governance.rs)

### Summary

In `disburse_neuron`, the NNS governance canister burns a neuron's accumulated management fees (Transfer 1) and immediately zeroes `neuron_fees_e8s` in state before attempting the stake disbursal to the user (Transfer 2). If Transfer 2 fails, the fees are permanently destroyed on the ledger and the governance state reflects zero fees, but the user receives no ICP. This is a direct analog to the reported pattern of crediting a fee recipient before confirming the underlying operation succeeds.

### Finding Description

`disburse_neuron` in `rs/nns/governance/src/governance.rs` performs two sequential async ledger calls separated by a committed state mutation:

**Step 1 — Fee burn (Transfer 1):** The function calls `ledger.transfer_funds` to burn `fees_amount_e8s` from the neuron's subaccount to the minting account. [1](#0-0) 

**Step 2 — State committed before Transfer 2:** Immediately after the `await?` on Transfer 1, the governance state is mutated: `neuron.neuron_fees_e8s = 0` and `cached_neuron_stake_e8s` is reduced by `fees_amount_e8s`. This state is committed to replicated state before Transfer 2 is attempted. [2](#0-1) 

**Step 3 — Stake disbursal (Transfer 2):** The function then attempts to transfer `disburse_amount_e8s` to the user's account. The code comment explicitly acknowledges that this transfer can fail while the burn has already succeeded: [3](#0-2) 

The existing test `test_cant_disburse_without_paying_fees` confirms this behavior: after a failed disbursal attempt, `neuron_fees_e8s` is asserted to be `0` (fees burned) while the user received nothing. [4](#0-3) 

The trigger condition for Transfer 2 to fail: the user supplies an explicit `amount` field in the `Disburse` command that exceeds the actual on-ledger balance of the neuron's subaccount. The governance code explicitly does not enforce that the requested amount is within the cached stake: [5](#0-4) 

The `disburse_amount_e8s` computation uses `a.e8s.saturating_sub(fees_amount_e8s)`, so a user-supplied amount larger than `cached_neuron_stake_e8s` produces a disburse amount that exceeds the post-burn subaccount balance, causing Transfer 2 to fail with an `External` error from the ledger. [6](#0-5) 

### Impact Explanation

A neuron owner who calls `manage_neuron { Disburse { amount: Some(Amount { e8s: X }) } }` where `X > cached_neuron_stake_e8s` will:
1. Have their `neuron_fees_e8s` permanently burned on the ICP ledger (real ICP destroyed).
2. Receive `neuron_fees_e8s = 0` in governance state.
3. Receive no ICP disbursement (Transfer 2 fails and returns an error).

The user permanently loses their accumulated neuron management fees — real ICP — without receiving their stake. The governance state and ledger state are left inconsistent: governance records zero fees but the user's subaccount balance is `original_stake - fees_burned` with no disbursement having occurred.

### Likelihood Explanation

The entry path is reachable by any neuron owner (unprivileged ingress sender) calling `manage_neuron` with a `Disburse` command and an explicit `amount`. The condition is triggered when the explicit amount exceeds the actual subaccount balance. This can occur through:
- User miscalculation of their available stake (e.g., not accounting for fees already accrued).
- A race where additional fees accrued between the user reading their neuron state and submitting the disburse call.
- Deliberate self-harm (user testing limits).

The likelihood is low-to-medium: it requires a specific combination of explicit amount and fee state, but the code explicitly documents and permits this failure mode without protecting the user from fee loss.

### Recommendation

Move the fee burn (Transfer 1) to occur **after** Transfer 2 succeeds, or alternatively, if Transfer 2 fails, issue a compensating mint back to the neuron's subaccount for the already-burned fees. The SNS governance implementation in `rs/sns/governance/src/governance.rs` has the same structural issue and should be reviewed identically. [7](#0-6) 

### Proof of Concept

1. Create a neuron with `cached_neuron_stake_e8s = 1_000_000_000` and `neuron_fees_e8s = 100_000` (fees from rejected proposals).
2. Call `manage_neuron { id: neuron_id, command: Disburse { amount: Some(Amount { e8s: 2_000_000_000 }), to_account: None } }` — requesting twice the actual stake.
3. Observe: Transfer 1 burns `100_000` e8s from the neuron subaccount to the minting account (succeeds). Governance sets `neuron_fees_e8s = 0` and `cached_neuron_stake_e8s = 900_000_000`.
4. Transfer 2 attempts to send `2_000_000_000 - 100_000 = 1_999_900_000` e8s from the subaccount (which only holds `900_000_000`). The ledger rejects with insufficient funds.
5. `disburse_neuron` returns `Err(GovernanceError { error_type: External })`.
6. Final state: `neuron_fees_e8s = 0`, neuron subaccount holds `900_000_000` e8s, user received `0` ICP. The `100_000` e8s in fees are permanently destroyed.

This is confirmed by the existing test at `rs/nns/governance/tests/governance.rs:5262` which asserts `neuron_fees_e8s == 0` after a failed disbursal. [8](#0-7)

#[cfg_attr(feature = "tla", with_tla_trace_check)]
fn test_cant_disburse_without_paying_fees() {

    let id = neuron.id.unwrap();
    let from = neuron.controller.unwrap();
