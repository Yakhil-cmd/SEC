### Title
Unrestricted Sale Ticket Creation Enables Stable Memory Exhaustion DoS in SNS Swap Canister - (File: `rs/sns/swap/src/swap.rs`)

### Summary
The `Swap::new_sale_ticket()` function in the SNS Swap canister allows any non-anonymous principal to create a sale ticket during an open swap without any fee, deposit, or rate limit. Because tickets do not commit ICP (the actual transfer happens separately via `refresh_buyer_tokens`), an attacker can generate arbitrarily many IC identities and create tickets with each one, filling the canister's `OPEN_TICKETS_MEMORY` stable memory. The global `next_ticket_id` counter is incremented on every successful ticket creation and is never reset, directly mirroring the global ID exhaustion pattern from the reference report.

### Finding Description
`Swap::new_sale_ticket()` is exposed as a public `#[update]` endpoint with no caller-side fee or rate limit. The only guards are:

1. Lifecycle must be `Lifecycle::Open`
2. Caller must not be anonymous
3. Subaccount, if provided, must be 32 bytes
4. No existing open ticket for this principal
5. Requested amount must be within swap participation bounds (computed against **committed** ICP, not pending tickets) [1](#0-0) 

Because tickets do not commit ICP, `current_direct_participation_e8s()` (the committed total used in the bounds check) does not increase when a ticket is created. An attacker can therefore create tickets with many distinct principals simultaneously, all passing the bounds check, without ever transferring ICP.

The global ticket ID counter increments on every successful creation: [2](#0-1) 

Each ticket is persisted in `OPEN_TICKETS_MEMORY` (stable memory): [3](#0-2) 

The only cleanup mechanism is `try_purge_old_tickets`, which only activates when there are **more than 100 million tickets** AND they are older than 2 days: [4](#0-3) 

The comment in the code itself acknowledges the scale: `100M * ~size(ticket) = ~25GB`. Below the 100M threshold, no purging occurs at all, leaving the stable memory to grow unbounded.

The canister endpoint is declared with no access control: [5](#0-4) 

### Impact Explanation
An attacker who generates N unique IC identities (key generation is free) and sends N ingress `new_sale_ticket` calls—without ever transferring ICP—accumulates N entries in `OPEN_TICKETS_MEMORY`. As stable memory fills:

- The swap canister's heartbeat (`run_periodic_tasks`) becomes increasingly expensive in cycles, eventually causing the canister to run out of cycles and stop processing.
- Legitimate participants calling `new_sale_ticket` or `refresh_buyer_tokens` may encounter canister-level failures if stable memory operations fail.
- The SNS decentralization sale is effectively blocked for the duration of the attack window (up to 2 days per batch of tickets before purging begins, and purging itself only starts at 100M tickets).

This is a direct DoS on a live SNS swap, preventing token decentralization and harming both the SNS project and its prospective participants.

### Likelihood Explanation
- IC principal identities are generated client-side from key pairs at zero on-chain cost.
- Each `new_sale_ticket` call costs only standard IC ingress fees (~1.2M cycles, a fraction of a cent), paid by the attacker.
- The swap canister bears the storage and execution cost of every ticket.
- A high-value SNS swap is a financially motivated target; an attacker who profits from blocking the sale (e.g., a competitor) has clear incentive.
- The 100M-ticket purge threshold is extremely high, providing no protection for the realistic attack range of 1M–10M tickets (50 MB–1 GB of stable memory growth).

Likelihood: **Medium** — economically rational for a targeted attack on a high-value swap.

### Recommendation
1. **Require ICP to be present before ticket creation**: Check the caller's ICP balance on the ledger (or require a prior transfer) before issuing a ticket, analogous to the original report's recommendation of a refundable deposit.
2. **Add the caller's principal to the ticket's uniqueness domain more aggressively**: The current one-ticket-per-principal check is correct but insufficient; consider a per-principal cooldown after ticket expiry.
3. **Lower the purge threshold**: `NUMBER_OF_TICKETS_THRESHOLD = 100_000_000` is far too high. A threshold of 10,000–100,000 would limit stable memory growth while still being operationally practical.
4. **Charge a small cycle fee at the canister level** for ticket creation, making spam economically prohibitive.

### Proof of Concept
```
1. Wait for an SNS swap to enter Lifecycle::Open.
2. Generate K unique secp256k1 key pairs offline (free, no on-chain action needed).
3. For each key pair, derive the IC self-authenticating principal.
4. Send a new_sale_ticket ingress message from each principal with
   amount_icp_e8s = params.min_participant_icp_e8s.
   - Lifecycle check passes (swap is Open).
   - Anonymous check passes (self-authenticating principal).
   - No existing ticket for this principal.
   - compute_participation_increment passes because current_direct_participation_e8s
     reflects only committed ICP (zero for these principals), not pending tickets.
5. Never call refresh_buyer_tokens — no ICP is ever transferred.
6. K tickets accumulate in OPEN_TICKETS_MEMORY.
7. Repeat with fresh identities every 2 days (before purge age is reached)
   or keep below the 100M purge threshold to avoid any cleanup.
8. Stable memory grows, canister cycle consumption increases, and legitimate
   participants are denied service.
```

    pub fn new_sale_ticket(
        &mut self,
        request: &NewSaleTicketRequest,
        caller: PrincipalId,
        time: u64,
    ) -> NewSaleTicketResponse {
        // Return an error if we are not in Lifecycle::Open.
        if self.lifecycle().is_before_open() {
            return NewSaleTicketResponse::err_sale_not_open();
        }
        if self.lifecycle().is_after_open() {
            return NewSaleTicketResponse::err_sale_closed();
        }
        if self.lifecycle() != Lifecycle::Open {
            // It must be that we are in Lifecycle::Unspecified, but this also
            // accounts for cases that might have been overlooked.
            log!(
                ERROR,
                "We are not in Lifecycle::Open. Swap:\n{:#?}",
                SwapDigest::new(self),
            );
            return NewSaleTicketResponse::err_sale_not_open();
        }

        if caller.is_anonymous() {
            return NewSaleTicketResponse::err_invalid_principal();
        }

        // subaccounts must be 32 bytes
        if request
            .subaccount
            .as_ref()
            .is_some_and(|subaccount| subaccount.len() != 32)
        {
            return NewSaleTicketResponse::err_invalid_subaccount();
        }
        let principal = Blob::from_bytes(caller.as_slice().into());
        if let Some(ticket) = memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow().get(&principal)) {
            return NewSaleTicketResponse::err_ticket_exists(ticket);
        }
