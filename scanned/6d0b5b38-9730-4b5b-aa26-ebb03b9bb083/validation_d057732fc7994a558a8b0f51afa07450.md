### Title
Sandbox RSS-Based Eviction Suppressed for Canisters with Large Stable Memory — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

### Summary

In `monitor_and_evict_sandbox_processes`, only the anonymous RSS (`get_anon_rss`) is stored in `sandbox_processes_rss` and propagated to `stats.rss`. The page-allocator RSS (`get_page_allocator_rss`), which accounts for canister/stable memory held in memfd or file-backed regions, is collected for metrics only and never included in the per-canister RSS stored for eviction decisions. As a result, `total_sandboxes_rss` — the value used as the gate condition in `trigger_sandbox_eviction` — systematically undercounts actual sandbox memory consumption for any canister that holds significant stable memory. The RSS-based eviction branch is therefore never triggered for such canisters, even when the node is under real memory pressure.

### Finding Description

In `monitor_and_evict_sandbox_processes`:

```
get_anon_rss(pid)  → pushed to sandbox_processes_rss  (line 1405)
get_page_allocator_rss(pid) → metrics only, NOT pushed (lines 1409-1414)
``` [1](#0-0) 

`update_sandbox_processes_rss` then writes only the anon RSS into `stats.rss`: [2](#0-1) 

`total_sandboxes_rss` sums only `stats.rss` (anon-only): [3](#0-2) 

`trigger_sandbox_eviction` uses a conjunctive (AND) gate:

```
total_sandboxes_rss > max_sandboxes_rss   ← anon-only, undercounted
AND
available_memory < 250 GiB               ← real system memory
``` [4](#0-3) 

For a canister with large stable memory (stored in memfd/file-backed regions):
- Condition 1 (`total_sandboxes_rss > max_sandboxes_rss`): **always FALSE** — anon RSS is small
- Condition 2 (`available_memory < 250 GiB`): **TRUE** — real memory is consumed by memfd
- AND(FALSE, TRUE) = **FALSE** → RSS-based eviction is never triggered

The monitor thread's own eviction call passes `max_sandboxes_rss = u64::MAX`, so it performs no RSS-based eviction either: [5](#0-4) 

`get_page_allocator_rss` correctly reads memfd and file-backed `.mem (deleted)` regions from `/proc/<pid>/smaps`: [6](#0-5) 

### Impact Explanation

A canister developer deploys a canister that allocates large stable memory (up to the IC protocol limit). The sandbox process for that canister has near-zero anon RSS but large page-allocator RSS. `total_sandboxes_rss` never exceeds `max_sandboxes_rss`, so the RSS-based eviction branch in `trigger_sandbox_eviction` is permanently suppressed. The node can accumulate physical memory pressure from memfd/file-backed regions without the RSS eviction mechanism engaging, potentially exhausting node RAM. The count-based and idle-time eviction paths remain functional but do not account for per-canister memory magnitude.

### Likelihood Explanation

Any unprivileged canister developer can deploy a canister that writes to stable memory via the public IC API. No privileged access, governance majority, or infrastructure compromise is required. The condition is deterministic and reproducible in a local replica environment.

### Recommendation

Include page-allocator RSS in the per-canister RSS stored for eviction. In `monitor_and_evict_sandbox_processes`, accumulate both `get_anon_rss` and `get_page_allocator_rss` before pushing to `sandbox_processes_rss`:

```rust
// Instead of pushing only anon RSS:
sandbox_processes_rss.push((*canister_id, NumBytes::new((anon_kib + page_alloc_kib) * 1024)));
```

This ensures `total_sandboxes_rss` reflects actual sandbox memory consumption and the eviction gate condition fires correctly.

### Proof of Concept

State-machine test outline:
1. Start a local replica with `max_sandboxes_rss` set to a small value (e.g., 100 MiB).
2. Deploy a canister that writes ~200 MiB of stable memory (via `stable_grow` + `stable_write`).
3. Keep the Wasm heap allocation minimal (near-zero anon RSS).
4. Observe via `/proc/<pid>/smaps` that the sandbox process has ~200 MiB of memfd/file-backed RSS but near-zero `RssAnon`.
5. Confirm that `stats.rss` for the backend is near zero (only anon RSS stored).
6. Confirm `total_sandboxes_rss` is below `max_sandboxes_rss`.
7. Simulate low available memory (condition 2 = TRUE).
8. Call `trigger_sandbox_eviction` and observe no eviction occurs despite actual memory exceeding the configured limit.

fn update_sandbox_processes_rss(
    backends: &mut HashMap<CanisterId, Backend>,
    sandbox_processes_rss: Vec<(CanisterId, NumBytes)>,
) {
    for (id, rss) in sandbox_processes_rss {
        backends.entry(id).and_modify(|backend| match backend {
            Backend::Active { stats, .. } | Backend::Evicted { stats, .. } => stats.rss = rss,
            Backend::Empty => {}
        });
    }
}

fn total_sandboxes_rss(backends: &HashMap<CanisterId, Backend>) -> NumBytes {
    backends
        .values()
        .map(|backend| match backend {
            Backend::Active { stats, .. } => stats.rss,
            Backend::Evicted { .. } | Backend::Empty => 0.into(),
        })
        .sum()
}

fn compute_memory_allocator_rss_total(vma_infos: &[VMAInfo]) -> u64 {
    let mut total: u64 = 0;
    for vma_info in vma_infos {
        // Check if memfds are present and add their RSS.
        // This is working for the original memory-based allocator.
        const PREFIX: &[u8] = b"/memfd:";
        let matching = vma_info
            .pathname
            .iter()
            .zip(PREFIX.iter())
            .filter(|&(x, y)| x == y)
            .count();
        if matching == PREFIX.len() {
            if let Ok(size) = get_named_field_kb(&vma_info.fields, "Rss") {
                total += size;
            }
        } else {
            // Check if file-backed memory regions are present and add their RSS.
            // This is working for the newer file-backed allocator. All files
            // of the file-backed allocator are of form /var/lib/ic/data/ic_state/page_deltas/[uuid].mem .
            // The (deleted) part appears because the allocator deletes the file as soon as it is created,
            // and thus the space on disk will be freed as soon as the last file descriptor pointing to
            // that file is closed.
            const EXTENSION: &[u8] = b".mem (deleted)";
            let matching = vma_info
                .pathname
                .iter()
                .rev()
                .zip(EXTENSION.iter().rev())
                .filter(|&(x, y)| x == y)
                .count();

            if matching == EXTENSION.len()
                && let Ok(size) = get_named_field_kb(&vma_info.fields, "Rss")
            {
                total += size;
            }
        }
    }

    total
}
