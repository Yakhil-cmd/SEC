### Title
Hardcoded Registry Canister ID in Migration Canister Causes Indefinitely Stuck Migrations if Registry Call Fails — (File: `rs/migration_canister/src/external_interfaces/registry.rs`)

---

### Summary

The migration canister hardcodes the Registry canister ID and calls it as a mandatory, non-bypassable step during the `process_renamed` processing phase. If the Registry canister's `migrate_canisters` method is broken or unavailable, in-flight migrations are retried indefinitely via `or_retry()` and never transition to `Failed`. During this time, the migrated and replaced canisters remain stopped with their controllers forcibly changed to the migration canister, locking them from their original controllers with no recovery path.

---

### Finding Description

In `rs/migration_canister/src/external_interfaces/registry.rs`, the Registry canister principal is hardcoded:

```rust
const REGISTRY_CANISTER_ID: &str = "rwlgt-iiaaa-aaaaa-aaaaa-cai";
``` [1](#0-0) 

The migration state machine in `rs/migration_canister/src/processing.rs` calls `migrate_canister` (which calls the Registry's `migrate_canisters` update method) in `process_renamed`:

```rust
migrate_canister(request.migrated_canister, request.replaced_canister_subnet)
    .await
    .map_success(|registry_version| RequestState::UpdatedRoutingTable { ... })
    .or_retry()
``` [2](#0-1) 

The `or_retry()` combinator converts any `FatalFailure` into `NoProgress`, meaning the state machine never transitions to `Failed` — it simply retries every second indefinitely:

```rust
ProcessingResult::FatalFailure(f) => {
    println!("Unreachable: Ignore failure {} and retry.", f);
    ProcessingResult::NoProgress
}
``` [3](#0-2) 

And `NoProgress` in `transition()` leaves the request state unchanged:

```rust
ProcessingResult::NoProgress => {}
``` [4](#0-3) 

By the time `process_renamed` is reached, `process_accepted` has already called `set_exclusive_controller` on **both** the migrated and replaced canisters, making the migration canister their sole controller:

```rust
set_exclusive_controller(request.migrated_canister).await ...
set_exclusive_controller(request.replaced_canister).await ...
``` [5](#0-4) 

The controller-restoration recovery logic only runs when a request reaches `RequestState::Failed`:

```rust
let requests = list_by(|r| matches!(r, RequestState::Failed { .. }));
``` [6](#0-5) 

Because `or_retry()` prevents the transition to `Failed`, the recovery mechanism never fires. There is no privileged method to force-fail a stuck migration, and the Registry canister ID cannot be updated at runtime.

---

### Impact Explanation

Any migration that has passed the `ControllersChanged` state and is stuck in `RenamedReplacedCanister` leaves both the migrated canister and the replaced canister:

- **Permanently stopped** — they cannot be started because the migration canister is their sole controller.
- **Inaccessible to original controllers** — original controllers were removed by `set_exclusive_controller`.
- **Unrecoverable without external intervention** — the `Failed`-state recovery path that would restore controllers is never reached.

This is a functional DOS on every canister pair involved in an in-flight migration at the time the Registry call begins failing.

---

### Likelihood Explanation

Low-to-medium. The Registry canister (`rwlgt-iiaaa-aaaaa-aaaaa-cai`) is a system canister upgraded via NNS governance proposals. A regression in the `migrate_canisters` update method — introduced by a buggy upgrade — would cause every in-flight migration in the `RenamedReplacedCanister` state to stall indefinitely. The `migrate_canisters` method and the migration canister itself are relatively new additions to the NNS stack, making regressions plausible. The Registry canister also serves many other callers, so a breaking change in its interface or internal invariants is a realistic failure mode.

---

### Recommendation

1. **Allow governance to update the Registry canister ID** stored in the migration canister, analogous to the original report's recommendation to allow the platform manager to update `operatorFilterRegistry`.
2. **Add a timeout-based fallback**: if a migration remains in `RenamedReplacedCanister` for longer than a configurable threshold, transition it to `Failed` so the controller-restoration recovery path fires.
3. **Expose a privileged `force_fail_migration` method** callable by NNS governance to manually transition a stuck migration to `Failed`, enabling controller recovery without requiring the Registry canister to be fixed first.

---

### Proof of Concept

1. Submit a valid `migrate_canister` call. The migration enters `Accepted` → `ControllersChanged` (both canisters now have the migration canister as sole controller) → `StoppedAndReady` → `RenamedReplacedCanister`.
2. At this point, introduce a failure in the Registry canister's `migrate_canisters` method (e.g., via an NNS upgrade that causes it to trap).
3. Observe via `migration_status` that the migration remains indefinitely in `InProgress { status: "RenamedReplacedCanister" }`.
4. Attempt to start or interact with either canister from the original controller — all calls are rejected because the migration canister is the sole controller.
5. The `process_all_failed` timer never fires for this request because `or_retry()` keeps it in `NoProgress`, so controllers are never restored. [1](#0-0) [7](#0-6) [8](#0-7)

pub async fn process_renamed(
    request: RequestState,
) -> ProcessingResult<RequestState, RequestState> {
    let RequestState::RenamedReplacedCanister {
        request,
        stopped_since,
    } = request
    else {
        println!("Error: list_by RenamedReplacedCanister returned bad variant");
        return ProcessingResult::NoProgress;
    };

    migrate_canister(request.migrated_canister, request.replaced_canister_subnet)
        .await
        .map_success(|registry_version| RequestState::UpdatedRoutingTable {
            request,
            stopped_since,
            registry_version,
        })
        .or_retry()
}

    if requests.is_empty() {

        tasks.push(process_failed(request.clone()));
    }
    let results = join_all(tasks).await;
    let mut success_counter = 0;
    for (req, res) in zip(requests, results) {
        if res.is_success() {
            success_counter += 1;
        }
        res.transition(req);
    }
    println!(

        success_counter
    );
}
