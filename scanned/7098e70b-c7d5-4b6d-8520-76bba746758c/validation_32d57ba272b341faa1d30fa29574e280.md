### Title
Negative Amount Bypass via Unsafe `i128`→`u64` Cast in ICP Rosetta API Amount Validation - (File: `rs/rosetta-api/icp/src/models/amount.rs`)

---

### Summary

The ICP Rosetta API's `from_amount` function validates that `val.abs()` fits in `u64` but does not reject negative `i128` values. The downstream `ledgeramount_from_amount` function then casts the returned `i128` directly to `u64` without a sign check, causing negative amounts to silently wrap to astronomically large `u64` values. A second instance exists in `convert.rs` where a positive fee amount is negated and cast to `u64`, producing the same wrap. Both paths bypass the Rosetta API's own input validation and forward malformed `Tokens` values to the ICP ledger or NNS governance canister.

---

### Finding Description

**Root cause — `from_amount` allows negative values through its validation:** [1](#0-0) 

```rust
let val: i128 = value.parse()...?;
let _ = u64::try_from(val.abs())
    .map_err(|_| "Amount does not fit in u64".to_string())?;
Ok(val)   // val can still be negative here
```

The guard `u64::try_from(val.abs())` only checks that the *magnitude* fits in `u64`. It does not reject negative values. For example, `val = -1` passes: `(-1_i128).abs() = 1`, `u64::try_from(1)` succeeds, and `-1_i128` is returned.

**First unsafe cast — `ledgeramount_from_amount`:** [2](#0-1) 

```rust
pub fn ledgeramount_from_amount(amount: &Amount, token_name: &str) -> Result<Tokens, String> {
    let inner = from_amount(amount, token_name)?;
    Ok(Tokens::from_e8s(inner as u64))   // negative inner wraps to huge u64
}
```

`inner as u64` is an unchecked Rust cast. If `inner = -1`, the result is `u64::MAX = 18_446_744_073_709_551_615`. This is used for the `Disburse` operation: [3](#0-2) 

**Second unsafe cast — `Fee` operation negation:** [4](#0-3) 

```rust
let amount = from_amount(amount, token_name)?;
state.fee(account, Tokens::from_e8s((-amount) as u64))?;
```

The Rosetta convention is that fee amounts are negative (e.g., `-10_000`). The code negates them before casting. If a caller supplies a *positive* fee amount (e.g., `+10_000`), `from_amount` accepts it (magnitude fits in `u64`), then `(-10_000_i128) as u64 = u64::MAX - 9_999`, producing a fee of `~1.8 × 10¹⁹` e8s.

**Third unsafe cast — `state.transaction` credit branch:** [5](#0-4) 

```rust
if amount > 0 || ... {
    self.credit = Some(AccountTokens {
        tokens: Tokens::from_e8s(amount as u64),  // safe only because amount > 0 here
    });
} else {
    self.debit = Some(AccountTokens {
        tokens: Tokens::from_e8s((-amount) as u64),  // wraps if amount == i128::MIN
    });
}
```

The debit branch is safe for normal negative values but would wrap on `i128::MIN` (though `from_amount`'s `abs()` check mitigates this in practice).

---

### Impact Explanation

An unprivileged Rosetta API caller submitting a `Disburse` operation with amount `"-1"` causes `Tokens::from_e8s(u64::MAX)` to be forwarded to the NNS governance canister's disburse endpoint. The governance canister will reject the request with an "insufficient maturity/stake" error rather than the Rosetta API returning a proper `InvalidTransaction` error. The same pattern applies to the `Fee` path: a positive fee amount produces a `BadFee` rejection from the ICP ledger instead of early rejection at the API boundary.

While the downstream ledger and governance canisters do reject these malformed values (no tokens can be stolen), the Rosetta API's own validation contract is broken:

- The API silently accepts semantically invalid inputs that it is responsible for rejecting.
- Error attribution is shifted from the API layer to the ledger/governance layer, producing misleading error codes to callers.
- Any future consumer of `ledgeramount_from_amount` that does not have a downstream balance check (e.g., a new operation type, a mock, or an off-chain accounting path) would silently operate on a wrapped `u64::MAX` token value.

**Vulnerability class:** boundary/API validation bypass (ledger conservation adjacent — the API's amount guard is defeated by a sign-blind cast, directly analogous to the Uniswap `int128 → uint128` slippage bypass).

---

### Likelihood Explanation

Any unprivileged HTTP caller of the Rosetta API's `/construction/payloads` or `/construction/preprocess` endpoints can trigger this path by supplying a negative amount in a `Disburse` operation or a positive amount in a `Fee` operation. No authentication, key material, or privileged role is required to reach `operations_to_requests` → `ledgeramount_from_amount`. The Rosetta API is a publicly reachable boundary node service.

---

### Recommendation

1. **In `from_amount`** — reject negative values explicitly before returning:

```rust
if val < 0 {
    return Err("Amount must be non-negative".to_string());
}
```

Or replace `u64::try_from(val.abs())` with `u64::try_from(val)`, which already rejects negatives.

2. **In `ledgeramount_from_amount`** — replace the unchecked cast with a checked conversion:

```rust
let e8s = u64::try_from(inner)
    .map_err(|_| "Amount is negative or overflows u64".to_string())?;
Ok(Tokens::from_e8s(e8s))
```

3. **In `convert.rs` Fee path** — validate that fee amounts are negative before negating, or apply the same `u64::try_from` guard after negation.

---

### Proof of Concept

**Trigger path for `Disburse` with negative amount:**

1. POST to Rosetta `/construction/payloads` with operations containing a `Disburse` operation and `amount.value = "-1"`.
2. `operations_to_requests` → `ledgeramount_from_amount("-1", "ICP")` is called.
3. `from_amount` parses `val = -1_i128`; `u64::try_from((-1_i128).abs()) = u64::try_from(1)` succeeds; returns `-1_i128`.
4. `ledgeramount_from_amount` computes `(-1_i128) as u64 = 18_446_744_073_709_551_615`; returns `Tokens::from_e8s(u64::MAX)`.
5. The constructed request is submitted to NNS governance with disburse amount `u64::MAX` e8s.
6. Governance rejects with a stake/maturity error instead of the Rosetta API returning `InvalidTransaction`. [6](#0-5) [3](#0-2) [7](#0-6)
