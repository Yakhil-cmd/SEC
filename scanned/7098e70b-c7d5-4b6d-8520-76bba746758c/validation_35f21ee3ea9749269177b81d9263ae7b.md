### Title
Lack of Caller Authorization in `refresh_buyer_tokens` Allows Anyone to Force Participation and Bypass Confirmation Text on Behalf of Any Buyer - (`rs/sns/swap/canister/canister.rs`)

### Summary

The SNS Swap canister's `refresh_buyer_tokens` update endpoint accepts an arbitrary `buyer` principal in its request payload without verifying that the actual IC caller matches that principal. Any unprivileged ingress sender can supply a victim's principal as `buyer`, causing the swap canister to credit the victim's ICP participation, delete the victim's open ticket, and bypass the SNS-configured confirmation text consent requirement — all without the victim's explicit action.

### Finding Description

The `refresh_buyer_tokens` canister endpoint reads the `buyer` field from the caller-supplied `RefreshBuyerTokensRequest`. If the field is non-empty, it is parsed directly into a `PrincipalId` and used as the buyer identity for the entire operation, with no check that `caller == buyer`:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()   // ← no caller == buyer check
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The `RefreshBuyerTokensRequest` proto explicitly documents the field as "If not specified, the caller is used", implying the intent that a non-empty value is a convenience for the buyer themselves — but no enforcement of that intent exists in the canister code. [2](#0-1) 

Inside `refresh_buyer_token_e8s`, the supplied `buyer` principal is used to:
1. Derive the ICP ledger subaccount to query (`principal_to_subaccount(&buyer)`)
2. Credit the buyer's participation amount in `self.buyers`
3. Delete the buyer's open ticket from `OPEN_TICKETS_MEMORY` if the balance satisfies the ticket amount
4. Validate and consume the `confirmation_text` on behalf of the buyer [3](#0-2) 

The ticket deletion path in `notify_payment_failure` correctly uses `caller_principal_id()` and cannot be called by a third party, but `refresh_buyer_tokens` has no equivalent guard. [4](#0-3) 

### Impact Explanation

An attacker who knows a victim's principal and observes (on-chain) that the victim has transferred ICP to the swap subaccount can:

1. **Bypass the SNS confirmation text consent mechanism.** The confirmation text is a public value set at SNS initialization. The attacker can supply it on behalf of the victim, recording the victim's "consent" without the victim ever sending the call. This undermines the legal/regulatory purpose of the confirmation text.

2. **Force the victim's participation and delete their open ticket.** Once `refresh_buyer_tokens` succeeds, the victim's open ticket is removed from `OPEN_TICKETS_MEMORY`. The victim can no longer call `notify_payment_failure` to cancel and reclaim their ICP before the swap closes. Their ICP is irrevocably committed to the swap.

3. **Manipulate swap lifecycle.** By forcing multiple victims' participation simultaneously, an attacker can push the swap to its ICP target faster, triggering early commitment and preventing other users from participating or withdrawing. [5](#0-4) 

### Likelihood Explanation

The attack requires only:
- Knowledge of a victim's principal (publicly observable from any prior on-chain interaction)
- The victim having already transferred ICP to the swap subaccount (observable on the ICP ledger)
- The correct confirmation text (publicly readable from the swap's `get_init` query endpoint)

No privileged access, key material, or social engineering is needed. The attacker sends a single ingress update call. The SNS swap is a high-value, time-limited contract, making this a realistic attack during an active sale.

### Recommendation

Add a caller authorization check at the top of `refresh_buyer_tokens`. The `buyer` field should only be accepted if it matches the actual IC caller:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Caller {} is not authorized to refresh tokens for buyer {}", caller, buyer);
        }
        buyer
    };
    ...
}
``` [1](#0-0) 

### Proof of Concept

1. Alice (victim) transfers 10 ICP to `swap_canister[subaccount = principal_to_subaccount(Alice)]` and creates an open ticket via `new_sale_ticket`.
2. The SNS swap requires a `confirmation_text = "I agree to the terms"` (readable via `get_init`).
3. Bob (attacker) calls:
   ```
   refresh_buyer_tokens({
     buyer: "<Alice's principal>",
     confirmation_text: "I agree to the terms"
   })
   ```
   as an ingress message from Bob's own identity.
4. The swap canister queries Alice's subaccount balance (10 ICP), credits Alice's participation, and deletes Alice's open ticket.
5. Alice's ICP is now locked in the swap. Alice's call to `notify_payment_failure` returns `None` (no ticket found). Alice never sent the confirmation text herself. [6](#0-5) [1](#0-0)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };

    // was removed successfully. Returns None if no ticket was found for the caller.
    // Only the owner of a ticket can remove it.
    pub fn notify_payment_failure(&mut self, caller: &PrincipalId) -> NotifyPaymentFailureResponse {
        let principal = Blob::from_bytes(caller.as_slice().into());
        let ticket = match memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow().get(&principal)) {
            Some(ticket) => ticket,
            None => return NotifyPaymentFailureResponse { ticket: None },
        };

        // process ticket.
        memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow_mut().remove(&principal));
        log!(

            "{}",
            format!(
                "Ticket with ID: {} was deleted successfully. Ticket: {:?}",
                ticket.ticket_id, ticket
            )
        );

        }
