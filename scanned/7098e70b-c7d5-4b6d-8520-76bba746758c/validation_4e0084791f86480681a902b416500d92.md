I need to trace the full execution path: parse → validate → execute, and check whether any layer catches duplicate `callback_id` values within a single Bitcoin payload.

### Title
Missing Intra-Payload Duplicate `callback_id` Check in Bitcoin `SelfValidatingPayload` Validation — (`rs/bitcoin/consensus/src/payload_builder/parse.rs`)

---

### Summary

`bytes_to_payload` collects `BitcoinAdapterResponse` entries into a `Vec` with no uniqueness check on `callback_id`. The `validate_payload` path calls only `validate_self_validating_payload_impl`, which checks nothing beyond emptiness and byte size. A Byzantine block proposer (single node, within the consensus fault threshold) can craft a payload containing two entries with the same `callback_id`, which passes all validation, gets finalized, and causes the execution layer to push two `ConsensusResponse` entries for the same callback into the `consensus_queue`, delivering the Bitcoin response twice.

---

### Finding Description

**Parse layer — no deduplication:**

`bytes_to_payload` decodes length-delimited protobuf messages into a plain `Vec`: [1](#0-0) 

No uniqueness check on `callback_id` is performed. Compare this to `bytes_to_chain_key_payload`, which explicitly errors on duplicate `callback_id`: [2](#0-1) 

And canister HTTP validation, which tracks a `delivered_ids` set and returns `DuplicateResponse` on collision: [3](#0-2) 

**Validation layer — no state cross-check:**

`validate_payload` calls `bytes_to_payload` then `validate_self_validating_payload_impl`. The impl does nothing beyond an emptiness check: [4](#0-3) 

It does not verify that `callback_id` values exist in the state, are unique within the payload, or are absent from past payloads. The `delivered_ids` computed from past payloads is never used to cross-check the current payload's entries: [5](#0-4) 

**Execution layer — unconditional double-push:**

`DemuxImpl::process_payload` iterates over all `bitcoin_adapter_responses` and calls `push_response_bitcoin` for each, with no deduplication: [6](#0-5) 

`push_response` for `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` performs no context lookup — it unconditionally pushes to `consensus_queue`: [7](#0-6) 

For `GetSuccessorsResponse`, `get_mut` is used (not `remove`), so the context remains in the map and both duplicate calls succeed, each pushing a `ConsensusResponse` to `consensus_queue`: [8](#0-7) 

---

### Impact Explanation

Two `ConsensusResponse` entries with the same `callback_id` enter the `consensus_queue`. The execution environment processes both, invoking the Bitcoin management canister's callback twice for a single request. For `GetSuccessors`, this means the canister processes the same Bitcoin block data twice, corrupting its chain-tip tracking state. For `SendTransaction`, the canister receives two confirmations for one broadcast. This violates the invariant that each `callback_id` maps to exactly one pending request and is delivered exactly once.

---

### Likelihood Explanation

A single Byzantine block proposer — one node within the `f < n/3` fault threshold — can craft and propose such a block. Because `validate_payload` has no duplicate check, all honest nodes accept and notarize the block. No threshold corruption, no key compromise, and no external dependency is required. The attacker only needs to control one subnet node, which is the standard Byzantine fault model the IC is designed to tolerate at the consensus layer but must guard against at the payload-validation layer.

---

### Recommendation

Add an intra-payload duplicate `callback_id` check in `validate_payload` (or in `bytes_to_payload` itself, mirroring `bytes_to_chain_key_payload`):

```rust
// In validate_payload, after bytes_to_payload:
let mut seen_ids = std::collections::BTreeSet::new();
for response in &payload {
    if !seen_ids.insert(response.callback_id) {
        return Err(ValidationError::InvalidArtifact(
            consensus::InvalidPayloadReason::InvalidSelfValidatingPayload(
                InvalidSelfValidatingPayloadReason::DuplicateCallbackId(response.callback_id),
            ),
        ));
    }
}
```

Additionally, cross-check each `callback_id` against `delivered_ids` (already computed from past payloads) to reject responses for already-delivered callbacks.

---

### Proof of Concept

1. Encode two `PbBitcoinAdapterResponse` protobuf messages with identical `callback_id = X` and valid `response` fields (e.g., `SendTransactionReject`) as length-delimited bytes.
2. Submit this as the `SelfValidatingPayload` bytes in a block proposal from a controlled block-proposer node.
3. Observe that `validate_payload` returns `Ok(())` on all honest nodes.
4. After finalization, observe that `consensus_queue` contains two entries with `callback_id = X`.
5. Observe the Bitcoin management canister's callback is invoked twice.

Fuzz assertion: `bytes_to_payload(data)` should never return a `Vec` containing two entries with the same `callback_id`. [1](#0-0) [5](#0-4) [6](#0-5) [7](#0-6)
