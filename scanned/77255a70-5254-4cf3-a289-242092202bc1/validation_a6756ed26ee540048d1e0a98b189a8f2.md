### Title
Query Cache `ignore_canister_balances` Flag Omits `ic0.canister_liquid_cycle_balance128` — Stale Balance Reads Served to Callers - (File: `rs/execution_environment/src/query_handler/query_cache.rs`)

---

### Summary

The IC replica-side query cache uses a flag `ignore_canister_balances` to decide whether a cached query result can be served even when a canister's cycle balance has changed. This flag is computed by checking only two of the three balance-related System API call counters. The counter for `ic0.canister_liquid_cycle_balance128()` is tracked in `SystemApiCallCounters` but is never consulted when computing `ignore_canister_balances`. As a result, any query that reads the liquid balance exclusively via `ic0.canister_liquid_cycle_balance128()` will have its result cached as if it were balance-independent, and subsequent queries will receive stale balance data even after the canister's cycles have changed.

---

### Finding Description

`SystemApiCallCounters` tracks four balance/time-sensitive System API calls: [1](#0-0) 

When a query result is pushed into the `QueryCache`, `EntryValue::new` computes the `ignore_canister_balances` flag: [2](#0-1) 

The logic checks `canister_cycle_balance` and `canister_cycle_balance128`, but **not** `canister_liquid_cycle_balance128`. The `canister_liquid_cycle_balance128` field is fully tracked in `SystemApiCallCounters` and is even propagated through `saturating_add`: [3](#0-2) 

The future-proof guard test explicitly lists `SystemApiCallId::CanisterLiquidCycleBalance128` alongside the other balance calls and warns that adding balance-dependent calls can break cache coherency: [4](#0-3) [5](#0-4) 

Despite this warning, the `ignore_canister_balances` computation was never updated to include `canister_liquid_cycle_balance128`. The cache validation path checks the stored canister balance against the current state only when `ignore_canister_balances` is `false`: [6](#0-5) 

When a query exclusively uses `ic0.canister_liquid_cycle_balance128()`, `ignore_canister_balances` is set to `true`, so the balance check is skipped entirely on subsequent cache lookups, and the stale result is returned as a hit.

---

### Impact Explanation

A canister that exposes a query method reading its liquid cycle balance via `ic0.canister_liquid_cycle_balance128()` — for example, to report available liquid cycles or to gate logic on whether sufficient liquid cycles exist — will have its result incorrectly cached as balance-independent. After the canister's balance changes (e.g., cycles are burned by an update call), any user querying the same method with the same arguments will receive the old, stale balance value from the cache. This is a **boundary/API validation bypass**: the query cache's coherency invariant is violated, and the replica serves certified-looking query responses that do not reflect actual on-chain state. Canisters that use `ic0.canister_liquid_cycle_balance128()` to make security-relevant decisions (e.g., "do I have enough liquid cycles to proceed?") expose incorrect information to callers.

---

### Likelihood Explanation

`ic0.canister_liquid_cycle_balance128()` is a recently added System API call. Any canister that uses it in a query method and is deployed on a subnet with query caching enabled is affected. The query cache is enabled by default on the IC. An unprivileged user only needs to send two identical queries bracketing an update call that changes the canister's balance to observe the stale result. No special privileges, keys, or majority corruption are required.

---

### Recommendation

Add `canister_liquid_cycle_balance128` to the `ignore_canister_balances` check in `EntryValue::new`:

```rust
let ignore_canister_balances = system_api_call_counters.canister_cycle_balance == 0
    && system_api_call_counters.canister_cycle_balance128 == 0
    && system_api_call_counters.canister_liquid_cycle_balance128 == 0; // add this
```

Add a corresponding regression test analogous to the existing `query_cache_returns_different_results_for_different_canister_balance128s` test, but using `ic0.canister_liquid_cycle_balance128()`.

---

### Proof of Concept

1. Deploy a canister with a query method that calls `ic0.canister_liquid_cycle_balance128()` and returns the result.
2. Send query Q₁ → result `R₁` (liquid balance = X) is cached with `ignore_canister_balances = true`.
3. Send an update call that burns cycles from the canister (balance changes to X − δ).
4. Send the identical query Q₁ again → the cache returns `R₁` (balance = X) as a hit, even though the actual liquid balance is now X − δ.

The root cause is confirmed at: [2](#0-1) 

where `canister_liquid_cycle_balance128` is absent from the guard, and at: [7](#0-6) 

where the counter is defined but ignored by the cache coherency logic.
