### Title
Fixed Gas Limit for All ckERC20 Withdrawals Causes Permanent ckETH Loss for High-Gas ERC-20 Tokens - (File: rs/ethereum/cketh/minter/src/withdraw.rs)

### Summary

The ckETH minter applies a single hardcoded gas limit of `65_000` to every ckERC20 withdrawal transaction on Ethereum, regardless of the specific ERC-20 token's actual gas requirements. When a supported ckERC20 token's `transfer()` call requires more than 65,000 gas, the Ethereum transaction runs out of gas and fails. The ckERC20 principal is reimbursed, but the ckETH burned to pay the transaction fee is permanently lost, as the protocol explicitly does not reimburse overcharged transaction fees.

### Finding Description

In `rs/ethereum/cketh/minter/src/withdraw.rs`, two gas limit constants are defined:

```rust
pub const CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(21_000);
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);
``` [1](#0-0) 

The function `estimate_gas_limit` returns `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` for every single ckERC20 withdrawal request, with no per-token differentiation:

```rust
pub fn estimate_gas_limit(withdrawal_request: &WithdrawalRequest) -> GasAmount {
    match withdrawal_request {
        WithdrawalRequest::CkEth(_) => CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
        WithdrawalRequest::CkErc20(_) => CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
    }
}
``` [2](#0-1) 

This constant is used in `create_transactions_batch` to build every EIP-1559 transaction for ckERC20 withdrawals: [3](#0-2) 

The same constant is used in `eip_1559_transaction_price` to estimate the fee that is burned from the user's ckETH balance upfront: [4](#0-3) 

The `withdraw_erc20` endpoint burns ckETH from the user based on this estimate before the Ethereum transaction is ever submitted: [5](#0-4) 

The protocol documentation explicitly states the gas limit is fixed and acknowledges the risk: *"The `gas_limit` for ckERC20 withdrawals is currently fixed to `65_000` and should be sufficient for standard ERC-20 contracts."* [1](#0-0) 

### Impact Explanation

When a user calls `withdraw_erc20` for a ckERC20 token whose underlying ERC-20 `transfer()` requires more than 65,000 gas (e.g., tokens with fee-on-transfer logic, hooks, complex storage layouts, or rebasing mechanics):

1. The minter burns ckETH from the user to cover the estimated fee (computed using `65_000` gas).
2. The minter burns the ckERC20 tokens from the user.
3. The minter submits an Ethereum transaction with `gas_limit = 65_000`.
4. The transaction runs out of gas on Ethereum and fails.
5. The ckERC20 tokens are reimbursed to the user (confirmed by `ReimbursedErc20Withdrawal` event path).
6. The ckETH burned for the transaction fee is **permanently lost**. The documentation explicitly states: *"Overcharged transaction fees are not reimbursed."* [6](#0-5) 

The reimbursement logic confirms only ckERC20 tokens are returned on failure, not the ckETH fee: [7](#0-6) 

This constitutes a **ledger conservation bug** and **chain-fusion burn/replay bug**: ckETH (backed 1:1 by real ETH held by the minter) is permanently destroyed without a corresponding successful Ethereum transfer.

### Likelihood Explanation

The IC's ckERC20 system is designed to support any ERC-20 token via NNS proposals. Tokens already added include ckUSDC, ckSHIB, ckUNI, and others. As the ecosystem expands, tokens with non-standard transfer logic (fee-on-transfer, hooks, rebasing) are increasingly likely to be added. Any unprivileged user holding such a ckERC20 token can trigger this loss simply by calling `withdraw_erc20`. No special privileges, governance majority, or external compromise is required.

### Recommendation

Replace the single global `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` constant with a per-token configurable gas limit stored in the `CkErc20Token` state. The `AddCkErc20Token` message and the `add_ckerc20_token` endpoint should accept an optional `gas_limit` field. The `estimate_gas_limit` function should look up the per-token value, falling back to `65_000` for tokens that do not specify one. This mirrors the fix recommended in the external report: pass the parameter explicitly rather than hardcoding it.

```diff
- pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);

  pub fn estimate_gas_limit(withdrawal_request: &WithdrawalRequest) -> GasAmount {
      match withdrawal_request {
          WithdrawalRequest::CkEth(_) => CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
-         WithdrawalRequest::CkErc20(_) => CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
+         WithdrawalRequest::CkErc20(r) => r.gas_limit
+             .unwrap_or(GasAmount::new(65_000)),
      }
  }
```

### Proof of Concept

1. An NNS proposal adds a ckERC20 token whose ERC-20 `transfer()` consumes ~80,000 gas (e.g., a fee-on-transfer token or a token with an ERC-777 hook).
2. A user calls `withdraw_erc20` on the ckETH minter canister (`sv3dd-oaaaa-aaaar-qacoa-cai`) specifying that token.
3. The minter calls `estimate_erc20_transaction_fee()`, which uses `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000` to compute the fee and burns the corresponding ckETH from the user.
4. The minter burns the ckERC20 tokens and submits an Ethereum transaction with `gas_limit = 65_000`.
5. The Ethereum transaction reverts with out-of-gas.
6. The minter detects the failed receipt via `FinalizedTransaction` event and schedules a `ReimbursedErc20Withdrawal` — returning only the ckERC20 tokens.
7. The ckETH burned in step 3 is never returned. The user has permanently lost real ETH value. [2](#0-1) [8](#0-7)
