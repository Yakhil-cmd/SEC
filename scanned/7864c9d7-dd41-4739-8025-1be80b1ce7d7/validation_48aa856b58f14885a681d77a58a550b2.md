### Title
Fee Accounting Cleared Unconditionally Before Stake Transfer Succeeds - (`File: rs/nns/governance/src/governance.rs`)

### Summary
In NNS Governance's `disburse_neuron`, `neuron_fees_e8s` is zeroed out unconditionally after the fee-burn ledger call succeeds, regardless of whether the fee burn was actually performed (i.e., when `fees_amount_e8s <= transaction_fee_e8s`, the burn is skipped but the fees are still cleared). This is the direct analog of the Overlay M-05 bug: fees are "accounted" (cleared from governance state) without the corresponding on-chain burn having occurred, creating a permanent discrepancy between the governance-tracked fee balance and the actual ledger balance.

### Finding Description

In `disburse_neuron` in `rs/nns/governance/src/governance.rs`, the flow is:

1. Read `fees_amount_e8s = neuron.neuron_fees_e8s` from neuron state.
2. **Conditionally** burn fees on the ledger — only if `fees_amount_e8s > transaction_fee_e8s`:
   ```rust
   if fees_amount_e8s > transaction_fee_e8s {
       self.ledger.transfer_funds(fees_amount_e8s, 0, ...).await?;
   }
   ```
3. **Unconditionally** zero out `neuron_fees_e8s` and reduce `cached_neuron_stake_e8s`:
   ```rust
   self.with_neuron_mut(id, |neuron| {
       if neuron.cached_neuron_stake_e8s > fees_amount_e8s {
           neuron.cached_neuron_stake_e8s -= fees_amount_e8s;
       } else {
           neuron.cached_neuron_stake_e8s = 0;
       }
       neuron.neuron_fees_e8s = 0;  // ← always zeroed
   })
   ```

When `fees_amount_e8s <= transaction_fee_e8s`, the burn is skipped (the ledger retains those tokens in the neuron's subaccount), but governance still:
- Subtracts `fees_amount_e8s` from `cached_neuron_stake_e8s`
- Sets `neuron_fees_e8s = 0`

This means governance believes the fees were burned and the stake was reduced, but the ledger still holds those tokens. The governance-tracked `cached_neuron_stake_e8s` is now **lower** than the actual ledger balance by `fees_amount_e8s`. The fee tokens are silently "leaked" — they remain in the neuron's subaccount but are no longer tracked by governance. [1](#0-0) 

### Impact Explanation

**Ledger/governance state divergence (governance conservation bug):**

- After a disburse where `fees_amount_e8s <= transaction_fee_e8s`, the neuron's ledger subaccount holds `fees_amount_e8s` more tokens than governance's `cached_neuron_stake_e8s` reflects.
- These tokens are permanently stranded in the neuron's subaccount: governance no longer tracks them (fees zeroed, stake reduced), so no future operation will attempt to burn or transfer them.
- The total ICP supply tracked by governance diverges from the actual ledger supply — the "burned" fees were never actually burned.
- A subsequent `refresh_neuron` call would detect the discrepancy (ledger balance > cached stake) and silently re-credit the stake, potentially allowing the neuron owner to recover and re-disburse those tokens — effectively double-spending the fee amount. [2](#0-1) 

### Likelihood Explanation

This is reachable by any NNS neuron owner who calls `disburse_neuron` on a dissolved neuron that has accumulated `neuron_fees_e8s > 0` but `neuron_fees_e8s <= transaction_fee_e8s` (10,000 e8s). This is a realistic scenario: a neuron that submitted one rejected proposal accumulates exactly `reject_cost_e8s` in fees, which can be configured to be at or below the transaction fee. The caller is an unprivileged ingress sender (the neuron controller). No privileged access is required. [3](#0-2) 

### Recommendation

Mirror the SNS governance fix: only zero `neuron_fees_e8s` and reduce `cached_neuron_stake_e8s` by the amount that was **actually burned**. When the burn is skipped (fees too small), do not modify `neuron_fees_e8s` or subtract from `cached_neuron_stake_e8s`.

```rust
if fees_amount_e8s > transaction_fee_e8s {
    self.ledger.transfer_funds(fees_amount_e8s, 0, ...).await?;
    // Only update state if burn actually happened
    self.with_neuron_mut(id, |neuron| {
        neuron.cached_neuron_stake_e8s =
            neuron.cached_neuron_stake_e8s.saturating_sub(fees_amount_e8s);
        neuron.neuron_fees_e8s = neuron.neuron_fees_e8s.saturating_sub(fees_amount_e8s);
    })?;
}
// Do NOT unconditionally zero neuron_fees_e8s here
```

The SNS governance implementation already does this correctly — it only updates `neuron_fees_e8s` and `cached_neuron_stake_e8s` inside the `if max_burnable_fee > transaction_fee_e8s` block. [4](#0-3) 

### Proof of Concept

1. Create a dissolved NNS neuron with `cached_neuron_stake_e8s = 100_000` and `neuron_fees_e8s = 5_000` (below `transaction_fee_e8s = 10_000`).
2. Call `disburse_neuron` on this neuron.
3. The condition `fees_amount_e8s (5_000) > transaction_fee_e8s (10_000)` is **false**, so no burn ledger call is made.
4. The code unconditionally executes:
   - `cached_neuron_stake_e8s = 100_000 - 5_000 = 95_000`
   - `neuron_fees_e8s = 0`
5. The ledger subaccount still holds `100_000` e8s (no burn occurred).
6. Governance now believes the neuron has `95_000` stake and `0` fees, but the ledger has `100_000`.
7. The 5,000 e8s fee tokens were never burned — they remain in the subaccount, untracked.
8. A subsequent `refresh_neuron` call will detect `ledger_balance (100_000 - disburse_amount - tx_fee) > cached_stake` and silently re-inflate the stake, allowing the fee tokens to be disbursed again. [2](#0-1) [5](#0-4)
