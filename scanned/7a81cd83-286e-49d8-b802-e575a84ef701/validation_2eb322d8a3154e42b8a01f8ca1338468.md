NNS/SNS Governance Voting Rewards Permanently Lost When No Neurons Vote During a Reward Period With Settled Proposals - (`rs/sns/governance/src/governance.rs`, `rs/nns/governance/src/governance.rs`)

---

### Summary

In both SNS and NNS governance, when no neurons vote on any proposals during a reward period (resulting in `total_reward_shares == 0` in SNS, or `total_voting_rights < 0.001` in NNS), the entire reward purse for that period is permanently destroyed rather than rolled over to the next period. This occurs because the proposals are still marked as settled, which causes `e8s_equivalent_to_be_rolled_over()` to return 0, silently discarding the accumulated reward budget.

---

### Finding Description

**Root cause — SNS (`rs/sns/governance/src/governance.rs`):**

The `distribute_rewards` function computes `total_reward_shares` by summing `ballot.voting_power` for every neuron that cast an eligible vote across all `ReadyToSettle` proposals. [1](#0-0) 

When `total_reward_shares == dec!(0)`, the function logs a warning and skips incrementing any neuron's maturity. However, execution continues unconditionally into the proposal-settlement loop: [2](#0-1) 

Every proposal in `considered_proposals` has its `reward_event_end_timestamp_seconds` and `reward_event_round` set and its ballots cleared. The resulting `RewardEvent` is written with a non-empty `settled_proposals` list and `distributed_e8s_equivalent = 0`.

**The rollover gate — SNS (`rs/sns/governance/src/types.rs`):** [3](#0-2) 

`e8s_equivalent_to_be_rolled_over()` returns `total_available_e8s_equivalent` only when `rewards_rolled_over()` is `true`, which requires `settled_proposals.is_empty()`. Because the proposals were settled above, this returns `0`. On the next reward round, `rolling_over_from_previous_reward_event_e8s_equivalent` is therefore `0`, and the entire reward purse from the zero-voter period is silently discarded.

**Root cause — NNS (`rs/nns/governance/src/governance.rs`):**

The identical structural flaw exists in `calculate_voting_rewards`. When `total_voting_rights < 0.001`, `reward_distribution` is set to `None` (no maturity incremented), but `reward_event.settled_proposals` is still populated with `considered_proposals`: [4](#0-3) 

The NNS rollover gate is identical in logic: [5](#0-4) 

The NNS code even contains an explicit acknowledgment of the unresolved risk: *"Not sure if that is theoretically possible, but even if it isn't, it might occur due to some bug."*

**The missing else-case analogy:**

This is the direct IC analog of the Syndicate's `// todo - check else case for any ETH lost` comment. In the Syndicate, the `else` block of `updateAccruedETHPerShares` was empty when `numberOfRegisteredKnots == 0`. Here, the `if total_reward_shares == dec!(0)` branch skips distribution but does not prevent proposal settlement, so the reward purse is consumed without being distributed or preserved.

---

### Impact Explanation

Any SNS token maturity rewards calculated for a reward round in which no neuron cast an eligible vote are permanently destroyed. The `rewards_purse_e8s` computed for that round — proportional to the total SNS token supply and the configured reward rate — is never added to any neuron's `maturity_e8s_equivalent` or `staked_maturity_e8s_equivalent`, and is never carried forward. Neuron holders lose maturity they are entitled to, with no recovery path. For NNS the same loss applies to ICP-equivalent maturity. Because maturity is the precursor to minting new governance tokens, this constitutes a ledger conservation violation: the protocol promises a reward rate but silently fails to honour it.

---

### Likelihood Explanation

**SNS:** Realistic for small or declining SNS DAOs. If all neurons have dissolved (dissolve delay expired, voting power = 0), their ballots carry `voting_power = 0`, so `total_reward_shares` sums to zero even if every neuron "voted." Alternatively, if no neuron voted (directly or via following) on any proposal that reached `ReadyToSettle`, the same condition is triggered. An unprivileged actor who holds a neuron can submit a proposal; if the SNS community is inactive, the reward purse for that round is lost. No privileged access is required.

**NNS:** The guard comment concedes the condition "might occur due to some bug." A bug in `sum_weighted_voting_power` or in vote propagation that zeroes out `total_voting_rights` while leaving `considered_proposals` non-empty would silently destroy the day's reward pot. The likelihood is low under normal operation but non-zero under fault conditions.

---

### Recommendation

When `total_reward_shares == dec!(0)` (SNS) or `total_voting_rights < 0.001` (NNS) and there are proposals to settle, the reward purse must not be silently discarded. Two complementary fixes:

1. **Do not settle proposals when no rewards can be distributed.** Skip the settlement loop when `total_reward_shares == 0`, leaving the proposals in `ReadyToSettle` so they are reconsidered in the next round. This mirrors the Syndicate mitigation of handling the zero-knots case explicitly.

2. **Alternatively, decouple rollover eligibility from proposal settlement.** Change `rewards_rolled_over()` (and thus `e8s_equivalent_to_be_rolled_over()`) to return `true` whenever `distributed_e8s_equivalent == 0`, regardless of whether `settled_proposals` is empty. This ensures the purse is always carried forward when nothing was actually paid out.

---

### Proof of Concept

**SNS scenario (no neurons vote):**

1. Deploy an SNS with `voting_rewards_parameters` configured (non-zero reward rate).
2. Create a proposal via a neuron; let it reach `ReadyToSettle` (voting period expires with no votes cast, or all neurons have dissolved so `voting_power = 0` in every ballot).
3. Advance time past one reward round so `distribute_rewards` is called.
4. Observe: `total_reward_shares == dec!(0)` → the `if` branch logs a warning and skips maturity increments.
5. Observe: the proposal's `reward_event_end_timestamp_seconds` is set and ballots are cleared — it is permanently settled.
6. Advance time past a second reward round.
7. Observe: `rolling_over_from_previous_reward_event_e8s_equivalent` is `0` because `e8s_equivalent_to_be_rolled_over()` returned `0` (settled_proposals was non-empty).
8. The reward purse from step 3 is gone; no neuron ever received it.
