### Title
Silent Integer Overflow in `add_expiration` Produces Epoch-0 Expiration in `FieldsDisplayMessage` Consent — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

### Summary

The `FieldsDisplayMessage` branch of `add_expiration` casts a `u64` nanosecond timestamp to `i64` without overflow checking. For any `expires_at > i64::MAX` (e.g., `u64::MAX`), the cast wraps to a negative value, integer division then truncates it to `0`, and the re-cast to `u64` produces `TimestampSeconds { amount: 0 }` — epoch 0 (1970-01-01). The `GenericDisplayMessage` branch at least surfaces an error string; the `FieldsDisplayMessage` branch is silent.

Arithmetic trace for `expires_at = u64::MAX = 18446744073709551615`:

| Step | Expression | Result |
|---|---|---|
| 1 | `u64::MAX as i64` | `-1` (wrapping two's-complement) |
| 2 | `(-1_i64) / 1_000_000_000_i64` | `0` (Rust truncates toward zero) |
| 3 | `0_i64 as u64` | `0` |
| Output | `TimestampSeconds { amount: 0 }` | epoch 0 = 1970-01-01 |

The `GenericDisplayMessage` branch performs the same cast but then calls `replace_nanosecond` with the wrapped remainder (`4294967295`), which fails the range check and returns the string `"Invalid nanosecond: 4294967295"` — at least an observable error. The `FieldsDisplayMessage` branch has no such guard. [2](#0-1) 

The full call chain is:

`icrc21_canister_call_consent_message` → `build_icrc21_consent_info` → `ConsentMessageBuilder::build` → `add_expiration` [3](#0-2) 

The `expires_at` field flows directly from the decoded `ApproveArgs` into `with_expires_at` with no range validation before `add_expiration` is called. [4](#0-3) 

### Impact Explanation

ICRC-21 consent messages are the canonical, canister-generated descriptions that wallets display to users before signing. A malicious dApp can craft an `icrc2_approve` with `expires_at = u64::MAX` (a near-permanent approval expiring ~year 2554), request the consent message with `device_spec = FieldsDisplay`, and receive back `TimestampSeconds { amount: 0 }`. A wallet rendering this field would display "Approval expiration: 1970-01-01" (or equivalent), leading the user to believe the approval is already expired or trivially short-lived, while the actual on-chain approval is effectively permanent. This directly undermines the security guarantee ICRC-21 is designed to provide.

### Likelihood Explanation

The overflow threshold is `i64::MAX` nanoseconds ≈ year 2262. Current real-world timestamps (~1.7 × 10¹⁸ ns) are safely below this. However, `u64::MAX` is a valid `u64` value accepted by the Candid decoder for `ApproveArgs.expires_at`, and no ledger-side guard rejects it before the consent message is generated. Any unprivileged caller can trigger this path with a single crafted ingress message.

### Recommendation

Replace the unchecked cast with a checked conversion and return an error (or a fallback display string) for out-of-range values, consistent with the `GenericDisplayMessage` branch:

```rust
// FieldsDisplayMessage branch
Some(expires_at) => {
    match i64::try_from(expires_at) {
        Ok(ts_i64) => {
            let seconds = ts_i64 / 1_000_000_000_i64;
            fields_display.fields.push((
                "Approval expiration".to_string(),
                Value::TimestampSeconds { amount: seconds as u64 },
            ));
        }
        Err(_) => fields_display.fields.push((
            "Approval expiration".to_string(),
            Value::Text { content: format!("Invalid timestamp: {expires_at}") },
        )),
    }
}
