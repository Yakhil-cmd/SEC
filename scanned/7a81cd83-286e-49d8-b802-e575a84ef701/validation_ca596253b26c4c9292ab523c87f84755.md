### Title
Voting Power Decay Restriction Bypass via Neuron Split — (File: `rs/nns/governance/src/governance.rs`)

### Summary
The NNS governance's `voting_power_refreshed_timestamp_seconds` mechanism is designed to reduce the deciding voting power of neurons that have not actively participated in governance for more than 6 months. However, when a neuron is split, the child neuron is always created with `voting_power_refreshed_timestamp_seconds = now()`, regardless of the parent's stale value. This allows any neuron holder whose voting power has decayed to bypass the restriction by splitting their neuron and using the child neuron to vote with full deciding power.

### Finding Description

The `split_neuron` function constructs the child neuron using `NeuronBuilder::new`, passing `created_timestamp_seconds` (which equals `self.env.now()`) as the creation time: [1](#0-0) 

The builder initialises `voting_power_refreshed_timestamp_seconds` from `created_timestamp_seconds`, so the child always receives a fresh timestamp equal to the current time. The parent's `voting_power_refreshed_timestamp_seconds` is never propagated to the child and is never updated by the split operation itself.

The deciding voting power of any neuron is computed as:

```
deciding_voting_power = adjustment_factor(time_since_last_refreshed) × potential_voting_power
``` [2](#0-1) 

When `time_since_last_refreshed` exceeds 6 months, `adjustment_factor` decreases linearly toward 0 over the following month. After 7 months of inactivity the neuron's deciding voting power reaches 0 and its followees are cleared.

The only documented ways to refresh `voting_power_refreshed_timestamp_seconds` are direct voting and setting followees: [3](#0-2) 

Splitting is not a refresh action, yet the child neuron unconditionally receives `voting_power_refreshed_timestamp_seconds = now()`. This is confirmed by the existing test suite: [4](#0-3) 

**Attack path:**
1. Attacker holds a neuron whose `voting_power_refreshed_timestamp_seconds` is stale (> 6 months ago), giving it a reduced or zero deciding voting power.
2. Attacker calls `manage_neuron` → `Split { amount_e8s: X }` to split any portion of stake into a child neuron.
3. The child neuron is created with `voting_power_refreshed_timestamp_seconds = now()` and therefore has full deciding voting power proportional to its stake.
4. Attacker uses the child neuron to vote on proposals with full deciding power, bypassing the decay restriction entirely.

**Quantified impact:** If the parent's adjustment factor is α (0 ≤ α < 1) and k e8s are split to the child:
- Before split: deciding VP = α × VP(N)
- After split: deciding VP = α × VP(N−k) + 1.0 × VP(k)
- Net gain = (1−α) × VP(k) > 0

In the extreme case (α = 0, all stake split), the attacker recovers full deciding voting power from zero.

### Impact Explanation
An unprivileged neuron holder whose deciding voting power has been reduced or eliminated due to governance inactivity can unilaterally restore full deciding voting power by splitting their neuron. This undermines the NNS's mechanism for incentivising active governance participation and allows inactive stakeholders to exert disproportionate influence on proposals without performing any of the legitimate refresh actions (direct voting or setting followees).

**Impact: Low–Medium** (governance voting power manipulation; no direct fund loss, but distorts NNS vote outcomes).

### Likelihood Explanation
**Likelihood: High.** Any neuron holder with a stale `voting_power_refreshed_timestamp_seconds` can perform this bypass with a single `manage_neuron` call. The operation costs only the ICP transaction fee for the ledger transfer. No special privileges, coordination, or timing are required.

### Recommendation
In `split_neuron`, propagate the parent's `voting_power_refreshed_timestamp_seconds` to the child neuron instead of defaulting to `now()`. Specifically, the child should inherit the **minimum** of `now()` and the parent's timestamp, ensuring the child never receives a fresher timestamp than the parent:

```rust
// In NeuronBuilder construction for split_neuron:
.with_voting_power_refreshed_timestamp_seconds(
    parent_neuron.voting_power_refreshed_timestamp_seconds
        .min(created_timestamp_seconds)
)
```

The same fix should be applied to any other neuron-creation paths (`disburse_to_neuron`, `spawn_neuron`) that similarly default to `now()`.

### Proof of Concept

```
1. Create neuron N with 1_000_000_000 e8s stake.
2. Advance time by 7+ months (> 6 months decay window + 1 month linear decay).
   → N.deciding_voting_power ≈ 0; N.voting_power_refreshed_timestamp_seconds is stale.
3. Call manage_neuron(N, Split { amount_e8s: 500_000_000, memo: None }).
   → Child neuron C is created.
   → C.voting_power_refreshed_timestamp_seconds = now()  [fresh]
   → N.voting_power_refreshed_timestamp_seconds unchanged [stale]
4. C.deciding_voting_power = 1.0 × VP(500_000_000 e8s)  [full power]
5. Use C to vote on any open proposal with full deciding power.
```

The existing test at `rs/nns/governance/tests/governance.rs:5602–5616` already asserts that `child_neuron.voting_power_refreshed_timestamp_seconds == Some(driver.now())` and that it is strictly greater than the parent's value, confirming the behaviour is present in the current codebase.
