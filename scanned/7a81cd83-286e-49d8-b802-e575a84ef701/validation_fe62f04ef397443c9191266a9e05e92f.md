### Title
Cycles Permanently Burned Without Message Induction for Stopping/Stopped Canisters — (`rs/messaging/src/scheduling/valid_set_rule.rs`)

---

### Summary

In `ValidSetRuleImpl::enqueue`, `charge_ingress_induction_cost` is called and permanently burns cycles from the receiving canister **before** the canister-status guard is checked. When the canister is `Stopping` or `Stopped`, the function returns `Err` with no refund path, violating the invariant that cycles must not be permanently deducted for messages that are never inducted.

---

### Finding Description

The ordering in `enqueue` is:

1. **Charge cycles** (lines 305–315) — calls `charge_ingress_induction_cost`, which (for a canister without a paused execution) calls `consume_with_threshold::<IngressInduction>` → `system_state.consume_cycles()`, permanently reducing the balance. The doc comment on `charge_ingress_induction_cost` is explicit: *"Note that this method reports the cycles withdrawn as consumed (i.e. burnt)."*

2. **Check canister status** (lines 317–332) — only after the charge succeeds does the code check whether the canister is `Running`. If it is `Stopping` or `Stopped`, it returns `Err(CanisterStopping)` / `Err(CanisterStopped)` with **no refund**. [1](#0-0) 

The `charge_ingress_induction_cost` function itself confirms the permanent burn: [2](#0-1) 

`consume_with_threshold` calls `system_state.consume_cycles()`, which subtracts from `cycles_balance` with no deferred/reversible mechanism for the `IngressInduction` use case: [3](#0-2) 

The payer is always the **receiving** canister, not the sender: [4](#0-3) 

The existing tests `ingress_to_stopping_canister_is_rejected` and `ingress_to_stopped_canister_is_rejected` only assert the returned error variant — they never assert that the canister's cycle balance is unchanged after the rejection, so this drain goes undetected: [5](#0-4) 

---

### Impact Explanation

Any unprivileged user can send ingress messages to a canister in `Stopping` or `Stopped` state. Each message causes `induction_cost` cycles to be permanently burned from the **target** canister's balance, with no message ever entering the input queue. Repeated across many rounds and many messages per round, an attacker can drain the entire cycle balance of any canister that passes through the `Stopping` state (e.g., during an upgrade or explicit stop). On application subnets, induction cost scales with message size, so large-payload messages amplify the drain rate.

---

### Likelihood Explanation

- The `Stopping` state is a normal, observable protocol state reachable by any canister undergoing a stop or upgrade.
- Any user can submit ingress messages to any canister — no privileged role is required.
- The canister status is publicly queryable, so an attacker can time the attack precisely.
- The `Stopped` state persists indefinitely until the canister is restarted, giving an unbounded attack window.
- No rate-limiting on ingress per target canister per round exists that would prevent bulk submission.

---

### Recommendation

Reorder the checks in `enqueue`: perform the canister-status guard **before** calling `charge_ingress_induction_cost`. The corrected logic should be:

```
1. Check canister status → return Err early if Stopping/Stopped (no cycles touched)
2. Charge cycles
3. Push ingress to queue
```

Alternatively, if the charge must happen first for some reason, add an explicit refund call (via `system_state.refund_cycles` or equivalent) on every early-return error path after the charge succeeds.

---

### Proof of Concept

State-machine test outline:

```rust
// 1. Create an application-subnet canister with known balance B.
// 2. Transition it to Stopping state (stop_canister via management canister).
// 3. Submit N distinct ingress messages (different nonces) to the canister.
// 4. Run induct_messages for the batch.
// 5. Assert: ingress_queue_size == 0 (no messages inducted).
// 6. Assert: canister balance == B - N * induction_cost  (cycles drained).
// 7. The delta B - balance equals N * induction_cost even though zero work was done.
```

The same test with a `Running` canister would show balance `B - N * induction_cost` AND `ingress_queue_size == N`, confirming the charge is correct only when messages are actually inducted. For `Stopping`/`Stopped`, the charge without induction is the bug.

    ///
    /// Note that this method reports the cycles withdrawn as consumed (i.e.
    /// burnt).
    ///
    /// # Errors
    ///
    /// Returns a `CanisterOutOfCyclesError` if the
    /// requested amount is greater than the currently available.
    pub fn charge_ingress_induction_cost(
        &self,
        canister: &mut CanisterState,
        canister_current_memory_usage: NumBytes,
        canister_current_message_memory_usage: MessageMemoryUsage,
        canister_compute_allocation: ComputeAllocation,
        cycles: Cycles,
        subnet_cycles_config: CyclesAccountManagerSubnetConfig,
        reveal_top_up: bool,
    ) -> Result<(), CanisterOutOfCyclesError> {
        let threshold = self.freeze_threshold_cycles(
            canister.system_state.freeze_threshold,
            canister.system_state.memory_allocation,
            canister_current_memory_usage,
            canister_current_message_memory_usage,
            canister_compute_allocation,
            subnet_cycles_config,
            canister.system_state.reserved_balance(),
        );
        if canister.has_paused_execution_or_install_code() {
            if canister.system_state.debited_balance() < cycles + threshold {
                return Err(CanisterOutOfCyclesError {
                    canister_id: canister.canister_id(),
                    available: canister.system_state.debited_balance(),
                    requested: cycles,
                    threshold,
                    reveal_top_up,
                });
            }
            canister
                .system_state
                .add_postponed_charge_to_ingress_induction_cycles_debit(cycles);
            Ok(())
        } else {
            self.consume_with_threshold::<IngressInduction>(
                &mut canister.system_state,
                CompoundCycles::new(cycles, subnet_cycles_config.cost_schedule),
                threshold,
                reveal_top_up,
            )
        }
