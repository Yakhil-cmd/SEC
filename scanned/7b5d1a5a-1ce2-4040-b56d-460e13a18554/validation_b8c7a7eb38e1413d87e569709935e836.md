### Title
`Mode::ReadOnly` and `Mode::RestrictedTo` Block ckBTC Withdrawals, Trapping Existing User Funds - (File: `rs/bitcoin/ckbtc/minter/src/state.rs`)

---

### Summary

The ckBTC minter's `Mode` enum conflates "restrict new deposits" with "restrict existing withdrawals." When governance sets the minter to `Mode::ReadOnly` or `Mode::RestrictedTo([])`, users who already hold ckBTC cannot redeem it back to BTC via `retrieve_btc` or `retrieve_btc_with_approval`. This directly mirrors the Isomorph H-6 vulnerability class: a single administrative state flag blocks both new-position creation and existing-position closure, trapping user funds.

---

### Finding Description

The `Mode` enum in `rs/bitcoin/ckbtc/minter/src/state.rs` controls which operations the minter permits: [1](#0-0) 

The `is_withdrawal_available_for` method returns an error for both `ReadOnly` and `RestrictedTo` variants: [2](#0-1) 

Both `retrieve_btc` and `retrieve_btc_with_approval` enforce this check as their very first gate before any other logic: [3](#0-2) [4](#0-3) 

If the NNS upgrades the minter with `Mode::ReadOnly` (e.g., for a maintenance window or in response to a deposit-side bug), **all** withdrawal requests are rejected with `TemporarilyUnavailable`. Users who have already burned their BTC and received ckBTC on the ledger cannot redeem it. The ckBTC tokens remain on the ledger but the minter refuses to process any BTC-side payout.

The `DepositsRestrictedTo` variant correctly decouples the two concerns — it blocks deposits while allowing withdrawals: [5](#0-4) 

But `ReadOnly` and `RestrictedTo` do not follow this pattern, creating an asymmetry where a legitimate governance maintenance action silently traps all existing ckBTC holders.

---

### Impact Explanation

Users holding ckBTC cannot convert it back to BTC while the minter is in `ReadOnly` or `RestrictedTo` mode. Their ckBTC tokens are live on the ICRC-1 ledger but the minter — the only entity that can authorize a BTC payout — refuses to process redemptions. This is a direct fund-freezing impact: the ckBTC peg is broken for existing holders for the duration of the restricted mode, which could be indefinite if governance does not act promptly to restore `GeneralAvailability`.

---

### Likelihood Explanation

The NNS governance canister is the controller of the ckBTC minter and can upgrade it with a new `mode` argument at any time via a standard NNS proposal. A legitimate maintenance scenario (e.g., a discovered bug in the deposit/UTXO-check flow) would motivate setting `ReadOnly` to halt all state changes. This is not a hypothetical attack — it is the documented intended use of `ReadOnly` mode. The test `test_upgrade_read_only` in `rs/bitcoin/ckbtc/minter/tests/tests.rs` explicitly confirms that `retrieve_btc` is rejected in this mode: [6](#0-5) 

The likelihood is moderate-to-high because `ReadOnly` is a standard operational tool and its side-effect of blocking withdrawals is not documented as a known trade-off.

---

### Recommendation

Decouple withdrawal availability from the `ReadOnly` and `RestrictedTo` modes, mirroring the fix applied in Isomorph. Specifically, `is_withdrawal_available_for` should return `Ok(())` for `ReadOnly` (and optionally for `RestrictedTo` with an empty allow-list), so that existing ckBTC holders can always redeem their tokens. New deposits can remain blocked. A separate `WithdrawalsRestrictedTo` variant can be introduced if full withdrawal restriction is ever needed, making the intent explicit rather than implicit.

---

### Proof of Concept

1. User deposits BTC → receives ckBTC on the ICRC-1 ledger.
2. NNS passes a proposal to upgrade the ckBTC minter with `MinterArg::Upgrade(Some(UpgradeArgs { mode: Some(Mode::ReadOnly), .. }))`.
3. User calls `retrieve_btc` with their ckBTC amount.
4. The minter executes `s.mode.is_withdrawal_available_for(&caller)` → returns `Err("the minter is in read-only mode")` → the call returns `RetrieveBtcError::TemporarilyUnavailable`.
5. The user's ckBTC remains on the ledger with no path to BTC redemption until governance issues another upgrade to restore `GeneralAvailability`. [7](#0-6) [3](#0-2)

    pub fn is_withdrawal_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability | Self::DepositsRestrictedTo(_) => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC withdrawals are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }

fn test_upgrade_read_only() {
    let env = new_state_machine();
    let ledger_id = install_ledger(&env);
    let minter_id = install_minter(&env, ledger_id);

    let authorized_principal =
        Principal::from_str("k2t6j-2nvnp-4zjm3-25dtz-6xhaa-c7boj-5gayf-oj3xs-i43lp-teztq-6ae")
            .unwrap();

    // upgrade
    let upgrade_args = UpgradeArgs {
        mode: Some(Mode::ReadOnly),
        ..Default::default()
    };
    let minter_arg = MinterArg::Upgrade(Some(upgrade_args));
    env.upgrade_canister(minter_id, minter_wasm(), Encode!(&minter_arg).unwrap())
        .expect("Failed to upgrade the minter canister");

    // when the mode is ReadOnly then the minter should reject all update calls.

    // 1. update_balance
    let update_balance_args = UpdateBalanceArgs {
        owner: None,
        subaccount: None,
    };
    let res = env
        .execute_ingress_as(
            authorized_principal.into(),
            minter_id,
            "update_balance",
            Encode!(&update_balance_args).unwrap(),
        )
        .expect("Failed to call update_balance");
    let res = Decode!(&res.bytes(), Result<Vec<UtxoStatus>, UpdateBalanceError>).unwrap();
    assert!(
        matches!(res, Err(UpdateBalanceError::TemporarilyUnavailable(_))),
        "unexpected result: {res:?}"
    );

    // 2. retrieve_btc
    let retrieve_btc_args = RetrieveBtcArgs {
        amount: 10,
        address: "".into(),
    };
    let res = env
        .execute_ingress_as(
            authorized_principal.into(),
            minter_id,
            "retrieve_btc",
            Encode!(&retrieve_btc_args).unwrap(),
        )
        .expect("Failed to call retrieve_btc");
    let res = Decode!(&res.bytes(), Result<RetrieveBtcOk, RetrieveBtcError>).unwrap();
    assert!(
        matches!(res, Err(RetrieveBtcError::TemporarilyUnavailable(_))),
        "unexpected result: {res:?}"
    );
