**Complete ingress path for Ed25519:**

1. `validate_user_id_and_signature` → `validate_signature` → `public_key_from_bytes` → `user_public_key_from_bytes`
2. In `user_public_key_from_bytes`: calls `ic_ed25519::PublicKey::deserialize_rfc8410_der` — **no `is_torsion_free()` check**
3. → `validate_signature_plain` → `verify_basic_sig_by_public_key` (standalone)
4. In standalone verifier: calls `ic_ed25519::PublicKey::deserialize_raw` — **no `is_torsion_free()` check**
5. → `pk.verify_signature` — uses ZIP215 cofactor equation: `(recomputed_r - sig.r()).mul_by_cofactor().is_identity()`

**The forgery:** For a torsion-component key A (order divides 8) and signature (R=A, S=0):
- `recomputed_r = [k]·(-A) + [0]·B = -k·A`
- `(recomputed_r - R).mul_by_cofactor() = (-k·A - A).mul_by_cofactor() = 8·(-(k+1)·A) = 0` ✓ (since 8·A = 0 for any torsion element)

This holds for **any message** M (k = H(R||A||M) is irrelevant since it's multiplied by A which has order dividing 8).

The `verify_public_key` function that does call `is_torsion_free()` is only used for node signing key validation — it is **never called** in the ingress validation path.

### Title
Ed25519 Torsion-Component Key Forgery Bypasses Ingress Authentication — (`rs/crypto/standalone-sig-verifier/src/lib.rs`, `rs/validator/src/ingress_validation.rs`)

### Summary
The ingress signature verification path accepts Ed25519 public keys with a torsion component and uses ZIP215 cofactor-clearing verification. For any of the 18 known torsion-component keys, the trivial signature `(R=torsion_key, S=0)` satisfies the ZIP215 equation for **every possible message**. An unprivileged attacker can derive a self-authenticating principal from such a key and submit authenticated ingress messages without possessing any private key.

### Finding Description

`verify_basic_sig_by_public_key` in the standalone sig verifier deserializes the Ed25519 public key with `deserialize_raw`, which explicitly documents that it does not check `is_torsion_free()`: [1](#0-0) 

The standalone verifier then calls `verify_signature` directly with no subgroup guard: [2](#0-1) 

`verify_signature` uses the ZIP215 cofactor equation: [3](#0-2) 

The `mul_by_cofactor()` call clears all torsion components. For a torsion-component key A (8·A = 0) and signature (R=A, S=0), the check becomes `8·(-k·A - A) = 0`, which is identically zero for any k (and thus any message).

The ingress path in `user_public_key_from_bytes` also performs no torsion check when parsing the DER key: [4](#0-3) 

The function `verify_public_key` that does enforce `is_torsion_free()` is only used for node key registration, not ingress: [5](#0-4) 

The ingress validation chain `validate_user_id_and_signature` → `validate_signature` → `validate_signature_plain` → `verify_basic_sig_by_public_key` never invokes it: [6](#0-5) 

The codebase itself confirms all 18 torsion-component keys are accepted by `deserialize_raw` and detectable only via the opt-in `is_torsion_free()`: [7](#0-6) 

### Impact Explanation
An attacker can:
1. Pick any of the 18 known torsion-component keys (e.g. `c7176a703d4dd84fba3c0b760d10670f2a2053fa2c39ccc64ec7fd7792ac03fa`)
2. DER-encode it via `convert_raw32_to_der` (no validity check)
3. Derive a self-authenticating principal: `SHA224(DER_key) || 0x02`
4. Construct the universal forgery: signature bytes = `torsion_key_bytes || [0u8; 32]` (R=torsion_key, S=0)
5. Submit any ingress update call authenticated as that principal

All replicas accept this because they all use the same ZIP215 path. The attacker has permanent, exclusive, keyless control over the derived principal and can mutate state on any canister that grants permissions to it.

### Likelihood Explanation
The attack requires no privileged access, no private key, no network-level attack, and no threshold corruption. It is fully constructable offline from public information (the 18 torsion keys are published in the test suite). The forgery works for every message, so it is not a one-time exploit.

### Recommendation
Add a `is_torsion_free()` check in `verify_basic_sig_by_public_key` in `rs/crypto/standalone-sig-verifier/src/lib.rs` immediately after `deserialize_raw`, returning `CryptoError::MalformedPublicKey` if the check fails. Equivalently, add the check in `user_public_key_from_bytes` in `sign_utils.rs` for the Ed25519 branch. The guard already exists and is correct — it is simply not called on the ingress path.
