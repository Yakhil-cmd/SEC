### Title
SNS Swap Early-Abort Mechanism Exhibits Counterproductive Incentives Under Permissible Parameter Combinations - (File: `rs/sns/swap/src/swap.rs`)

### Summary
The SNS swap canister contains an early-abort safety mechanism that triggers when `max_direct_participation_icp_e8s` is reached without `sufficient_participation`. Under a permissible parameter configuration (`max_participant_icp_e8s == max_direct_participation_icp_e8s` with `min_participants > 1`), any single unprivileged participant can fill the entire swap capacity via `refresh_buyer_token_e8s`, which immediately triggers the early-abort path on the next heartbeat. The participant's contribution — intended to help the swap succeed — instead permanently kills it. This is the IC analog of Aave H08: a safety mechanism that, under specific parameter conditions, acts counterproductively and worsens the outcome it was designed to prevent.

### Finding Description

The SNS swap lifecycle transitions to `ABORTED` when:

```
(swap_due || icp_target_reached) && !sufficient_participation
``` [1](#0-0) 

Where `icp_target_reached` means `current_direct_participation_e8s >= max_direct_participation_e8s`: [2](#0-1) 

And `sufficient_participation` requires both `min_participants` direct buyers AND `min_direct_participation_icp_e8s` reached: [3](#0-2) 

The swap parameter validation (`validate_participation_constraints`) enforces:
- `max_participant_icp_e8s >= min_participant_icp_e8s`
- `max_direct_participation_icp_e8s >= min_direct_participation_icp_e8s`
- `max_direct_participation_icp_e8s >= min_participants * min_participant_icp_e8s` [4](#0-3) 

But **no constraint prevents** `max_participant_icp_e8s == max_direct_participation_icp_e8s` when `min_participants > 1`. The proto comment even says `max_participant_icp_e8s` "can effectively be disabled by setting it to `max_icp_e8s`": [5](#0-4) 

When this configuration is used, a single caller invoking `refresh_buyer_token_e8s` with `max_direct_participation_icp_e8s` ICP fills the entire swap capacity in one call: [6](#0-5) 

On the next heartbeat, `run_periodic_tasks` calls `try_abort`, which succeeds because `icp_target_reached && !sufficient_participation` (only 1 participant, `min_participants > 1`): [7](#0-6) 

This is confirmed by the existing test `test_try_commit_or_abort_insufficient_participation_with_max_icp`, which explicitly documents that a single buyer reaching `max_icp` with `min_participants=2` causes an abort: [8](#0-7) 

The proto lifecycle diagram itself documents this abort path: [9](#0-8) 

### Impact Explanation

A single unprivileged principal can permanently abort any SNS decentralization swap whose parameters satisfy `max_participant_icp_e8s >= max_direct_participation_icp_e8s` and `min_participants > 1`. The attacker:

1. Transfers `max_direct_participation_icp_e8s` ICP to their subaccount on the swap canister
2. Calls `refresh_buyer_token_e8s` — accepted because the ICP target is not yet reached
3. The swap's direct participation is now at maximum; the next heartbeat aborts the swap
4. The attacker receives a full ICP refund (minus two transfer fees)

The SNS launch fails permanently. The dapp's control is returned to `fallback_controller_principal_ids`. All legitimate participants are refunded but the SNS is never created. The cost to the attacker is two ICP ledger transfer fees (~0.0002 ICP). This is a governance/SNS-launch denial-of-service with negligible cost.

### Likelihood Explanation

The condition is reachable in practice. Many real SNS launches set `max_participant_icp_e8s` equal or close to `max_direct_participation_icp_e8s` to allow whale participation. The `min_participants > 1` requirement is standard for decentralization. The attack requires no privileged access, no key material, and no governance majority — only the ability to hold and transfer ICP. The attacker recovers their ICP, making the attack essentially free. The swap canister is a publicly callable canister on mainnet.

### Recommendation

Add a validation constraint in `validate_participation_constraints` that enforces:

```
max_participant_icp_e8s * min_participants <= max_direct_participation_icp_e8s
```

This ensures that no single participant can fill the entire swap capacity when `min_participants > 1`, preventing the counterproductive early-abort. Alternatively, the early-abort logic could be modified to only trigger when it is **impossible** for the remaining capacity to accommodate `min_participants - current_participants` additional participants each contributing at least `min_participant_icp_e8s`, rather than triggering immediately when the ICP target is reached. [10](#0-9) 

### Proof of Concept

**Parameters (all pass current validation):**
- `min_participants = 2`
- `min_participant_icp_e8s = 1_000_000_000` (10 ICP)
- `max_participant_icp_e8s = 100_000_000_000` (1000 ICP)
- `min_direct_participation_icp_e8s = 10_000_000_000` (100 ICP)
- `max_direct_participation_icp_e8s = 100_000_000_000` (1000 ICP)

**Attack steps:**
1. Attacker transfers 1000 ICP to `swap_canister_subaccount(attacker_principal)` on the ICP ledger
2. Attacker calls `swap.refresh_buyer_token_e8s(attacker_principal, None, swap_canister_id, icp_ledger)`
3. `available_direct_participation_e8s()` = 1000 ICP; attacker's balance is set to 1000 ICP
4. `icp_target_progress()` now returns `Reached(100_000_000_000)`
5. Next heartbeat: `can_abort(now)` = `true` (target reached, only 1 participant < `min_participants=2`)
6. `try_abort(now)` succeeds → swap transitions to `ABORTED`
7. Attacker calls `finalize` → receives 1000 ICP refund minus two transfer fees

The existing test at `rs/sns/swap/src/swap.rs:4672-4708` already demonstrates this exact abort path with `min_participants=2` and a single buyer at `max_icp`. [8](#0-7)

    pub fn icp_target_progress(&self) -> IcpTargetProgress {

            let current_direct_participation_e8s = self.current_direct_participation_e8s();
            let max_direct_participation_e8s = self.max_direct_participation_e8s();
            match current_direct_participation_e8s.cmp(&max_direct_participation_e8s) {

                },
