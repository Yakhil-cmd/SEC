### Title
Unprivileged Caller Can Force-Fail SNS Upgrade and Create Inconsistent `pending_version` State - (File: rs/sns/governance/canister/canister.rs)

### Summary
The `fail_stuck_upgrade_in_progress` endpoint in SNS Governance is exposed as an unrestricted `#[update]` method callable by any unprivileged ingress sender. After the `mark_failed_at_seconds` deadline passes, any caller can force-clear `pending_version` and mark the associated upgrade proposal as failed — without updating `deployed_version` — creating an inconsistent governance state analogous to the GMToken `setFailedState` pattern.

### Finding Description
`fail_stuck_upgrade_in_progress` in `rs/sns/governance/canister/canister.rs` is declared as a plain `#[update]` with no caller guard:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(
        sns_gov_pb::FailStuckUpgradeInProgressRequest::from(request),
    ))
}
``` [1](#0-0) 

The implementation in `rs/sns/governance/src/governance.rs` only checks a temporal condition — `now > pending_version.mark_failed_at_seconds` — before clearing `pending_version` and marking the proposal failed:

```rust
if now > pending_version.mark_failed_at_seconds {
    ...
    self.complete_sns_upgrade_to_next_version(
        pending_version.proposal_id,
        status,
        message,
        None,   // <-- deployed_version NOT updated
    );
}
``` [2](#0-1) 

`complete_sns_upgrade_to_next_version` clears `self.proto.pending_version` but only updates `deployed_version` when the `deployed_version` argument is `Some(...)`. Passing `None` leaves `deployed_version` pointing to the pre-upgrade version even if the actual canister upgrade completed just before the call: [3](#0-2) 

The developers themselves noted the missing `checking_upgrade_lock` guard in a TODO comment inside the function body:

> "Maybe, we should look at the `checking_upgrade_lock` field and only proceed if it is false, or the request has force set to true." [4](#0-3) 

The `pending_version` field tracks the in-flight upgrade target and deadline: [5](#0-4) 

### Impact Explanation
After `mark_failed_at_seconds` elapses, any unprivileged principal can:

1. **Force-fail a still-completing upgrade.** If the upgrade heartbeat or root-canister call is slow but has not yet confirmed, the attacker clears `pending_version` and marks the proposal failed. The actual canister may then finish upgrading to the new Wasm, leaving `deployed_version` stale (pointing to the old version).

2. **Create a persistent governance/canister version mismatch.** Subsequent `UpgradeSnsToNextVersion` proposals use `deployed_version` to determine the upgrade path. A stale value causes the governance to compute an incorrect next-version step, potentially rejecting valid upgrades or accepting a version-skip.

3. **Unblock a new upgrade proposal immediately.** With `pending_version` cleared, a new `UpgradeSnsToNextVersion` proposal can be adopted and executed right away, racing against the still-in-flight original upgrade and potentially double-upgrading canisters.

The `Lifecycle` enum for the SNS Swap canister and the `sns_token_swap_lifecycle` field in NNS `ProposalData` are both downstream consumers of correct SNS version state, so a corrupted `deployed_version` can propagate into swap and governance decisions. [6](#0-5) [7](#0-6) 

### Likelihood Explanation
SNS upgrades are time-bounded by `mark_failed_at_seconds`. Any upgrade that takes longer than its deadline (e.g., due to subnet load, slow inter-canister calls, or a transient root-canister error) becomes vulnerable the moment the deadline passes. An attacker monitoring the SNS governance canister state can observe `pending_version.mark_failed_at_seconds` via the public `get_running_sns_version` query and call `fail_stuck_upgrade_in_progress` at the exact right moment. No special privilege, key, or social engineering is required — a single ingress update call suffices.

### Recommendation
1. **Add a caller restriction.** Only the SNS governance canister itself (via heartbeat/timer), or a whitelisted set of principals (e.g., SNS root), should be permitted to call `fail_stuck_upgrade_in_progress`. Reject calls from arbitrary principals.
2. **Check `checking_upgrade_lock`.** As the in-code comment acknowledges, the function should refuse to act (or require an explicit `force` flag) when `checking_upgrade_lock > 0`, indicating an active upgrade check is in flight.
3. **Update `deployed_version` on forced failure.** If the upgrade may have partially completed, the function should query the actual running Wasm hash from the root canister before clearing `pending_version`, or at minimum emit a warning that `deployed_version` may be stale.

### Proof of Concept
```
# 1. Observe pending_version.mark_failed_at_seconds via public query:
dfx canister call <sns_governance_id> get_running_sns_version '(record {})'
# -> returns pending_version { mark_failed_at_seconds = T }

# 2. Wait until now > T (or observe it has already passed).

# 3. Any principal calls fail_stuck_upgrade_in_progress:
dfx canister call <sns_governance_id> fail_stuck_upgrade_in_progress '(record {})'
# -> returns empty record {}

# 4. Verify inconsistent state:
dfx canister call <sns_governance_id> get_running_sns_version '(record {})'
# -> pending_version is now None; deployed_version still shows old version
#    even though the actual canister Wasm may have been upgraded.

# 5. A new UpgradeSnsToNextVersion proposal can now be submitted immediately,
#    racing against the still-in-flight original upgrade.
```
