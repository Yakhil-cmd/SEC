### Title
Silent ICP Burn Failure in CMC Allows Cycles Minting Without Ledger Conservation - (`rs/nns/cmc/src/main.rs`)

### Summary

The `burn_and_log` function in the Cycles Minting Canister (CMC) silently discards inter-canister call failures when burning ICP from the ledger after successfully minting and depositing cycles. If the ledger `send_pb` call fails for any reason, cycles are permanently minted and delivered to the target canister while the corresponding ICP remains unburned in the CMC's subaccount. The notification is then permanently marked as successful, preventing any retry or recovery. This breaks the ICP/cycles conservation invariant.

### Finding Description

In `process_top_up`, `process_create_canister`, and `process_mint_cycles`, the CMC first mints cycles and deposits them to the target canister (or cycles ledger), then calls `burn_and_log` to destroy the corresponding ICP from its own subaccount on the ICP ledger. The `burn_and_log` function is explicitly designed to swallow all errors: [1](#0-0) 

The comment at line 2015 states the rationale: "Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried." However, this design creates a one-way door: if the ledger call at line 2040 fails (transient error, ledger unavailability, or insufficient subaccount balance due to a race), the error is only printed and the function returns normally.

The callers of `burn_and_log` treat its return as unconditional success: [2](#0-1) [3](#0-2) [4](#0-3) 

After `burn_and_log` returns (whether the burn succeeded or not), the notification is stored as a permanent success in `blocks_notified`: [5](#0-4) 

This means the block index is permanently marked `NotifiedTopUp(Ok(cycles))`, preventing any future retry or recovery of the failed burn.

The `ensure_balance` function, called inside `deposit_cycles`, mints new cycles into the CMC's own balance before the deposit: [6](#0-5) 

By the time `burn_and_log` is called, cycles have already been irreversibly minted and transferred. The ICP in the CMC's subaccount is now orphaned: it cannot be reclaimed by the original sender (the notification is finalized), and future `notify_top_up` calls for the same canister only burn their own transaction's `amount`, not the accumulated leftover.

### Impact Explanation

**Ledger conservation violation**: Cycles are minted (ICP supply should decrease by the equivalent amount) but the ICP is not burned. The total ICP supply is higher than it should be. Over repeated failures, unburned ICP accumulates in CMC subaccounts indefinitely. This directly breaks the economic invariant that underpins the ICP/cycles exchange: every cycle minted must correspond to burned ICP.

Unlike the Solidity analog where stuck ETH is redistributed to the next caller, here the stuck ICP is not redistributed — it is permanently stranded in the CMC's subaccount with no on-chain mechanism to recover or burn it. The conservation violation is permanent per failed notification.

### Likelihood Explanation

The ICP ledger is a trusted NNS canister, so outright failures are uncommon. However, realistic failure scenarios include:

1. **Transient inter-canister call errors**: The IC runtime can return `SysTransient` reject codes for temporary resource exhaustion or queue overflow, causing `call_protobuf` to return `Err`.
2. **Ledger-side rejection**: If the CMC's subaccount balance is lower than expected (e.g., due to a concurrent partial burn or a prior failed burn that left the subaccount in an inconsistent state), the ledger returns `InsufficientFunds`, which `burn_and_log` silently swallows.
3. **CMC upgrade during processing**: A canister upgrade between `deposit_cycles` and `burn_and_log` could cause the in-flight call to fail.

Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` can trigger this path. No special privilege is required.

### Recommendation

The design trade-off is real: propagating the burn error would allow the notification to be retried, risking double-minting. However, the current approach silently accepts conservation violations. A safer approach:

1. **Retry the burn separately**: Store failed burns in persistent state and retry them in a background timer task, independent of the notification idempotency mechanism. The notification remains finalized (preventing double-minting of cycles) while the burn is retried until it succeeds.
2. **Alert on burn failure**: At minimum, emit a certified metric or structured log that can be monitored and acted upon by operators, rather than only a `print` statement that is not observable on-chain.
3. **Verify subaccount balance before minting**: Before calling `ensure_balance`/`deposit_cycles`, verify that the CMC's subaccount holds at least `amount` ICP, reducing the chance of a burn failure due to balance mismatch.

### Proof of Concept

1. User A sends 10 ICP to the CMC's subaccount for Canister X (with `MEMO_TOP_UP_CANISTER`).
2. User A calls `notify_top_up { block_index: B, canister_id: X }`.
3. CMC calls `process_top_up` → `deposit_cycles` succeeds → Canister X receives cycles.
4. CMC calls `burn_and_log(sub_X, 10_ICP)`.
5. The ICP ledger's `send_pb` call returns `Err` (e.g., transient error or ledger queue full).
6. `burn_and_log` prints the error and returns `()` — no error propagated.
7. `notify_top_up` stores `NotifiedTopUp(Ok(cycles))` in `blocks_notified[B]`.
8. User A's 10 ICP remains in the CMC's subaccount for Canister X, unburned.
9. Canister X has received cycles equivalent to 10 ICP.
10. The ICP total supply is 10 ICP higher than it should be — a permanent conservation violation.
11. User A cannot retry (the notification is finalized). The ICP is permanently stranded. [7](#0-6)

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }

fn ensure_balance(

    // unused because of check above
    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
