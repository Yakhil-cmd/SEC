### Title
Failed ICRC-1/ICRC-2 Ledger Transactions Not Recorded in Deduplication Window, Enabling Replay - (`rs/ledger_suite/common/ledger_canister_core/src/ledger.rs`)

### Summary

The `apply_transaction` function in the shared IC ledger core only inserts a transaction hash into the deduplication map (`transactions_by_hash`) after the transaction **succeeds**. If the transaction fails at the `transaction.apply()` step (e.g., `InsufficientFunds`, `InsufficientAllowance`), the hash is never recorded. This allows the identical transaction — same `created_at_time`, same parameters, same hash — to be replayed within the deduplication window once the blocking condition is resolved, bypassing the replay protection the caller opted into by setting `created_at_time`.

### Finding Description

In `rs/ledger_suite/common/ledger_canister_core/src/ledger.rs`, `apply_transaction` performs deduplication only for transactions that include `created_at_time`: [1](#0-0) 

If no duplicate is found, the transaction is applied: [2](#0-1) 

The `?` operator at line 275 causes an early return on failure. The hash is only recorded **after** this point, on the success path: [3](#0-2) 

Because the hash is never inserted into `transactions_by_hash` for a failed transaction, a subsequent submission of the identical transaction (same `created_at_time` + same fields = same hash) within the deduplication window will pass the duplicate check at line 249 and proceed to execution.

This affects both the ICP ledger (`rs/ledger_suite/icp/ledger/src/lib.rs`) and the ICRC-1 ledger (`rs/ledger_suite/icrc1/ledger/src/main.rs`), both of which call `apply_transaction`: [4](#0-3) [5](#0-4) 

The deduplication window for the ICP ledger is 24 hours: [6](#0-5) 

### Impact Explanation

The highest-impact scenario is **ICRC-2 `transfer_from`**: a spender canister (e.g., a DeFi protocol) calls `icrc2_transfer_from` with `created_at_time` set, targeting an owner account with insufficient funds. The call fails with `InsufficientFunds`. The hash is not recorded. Later — within the 24-hour window — the owner's balance is replenished. The spender replays the identical `transfer_from` call (same `created_at_time`, same args). The ledger does not detect it as a duplicate and executes the transfer, draining the owner's tokens **without fresh consent from the owner**. The owner's approval may still be valid (not expired), making this a silent unauthorized debit.

For `icrc1_transfer`, the caller is the sender, so only the sender can replay their own failed transaction — lower severity but still violates the deduplication guarantee the user opted into.

### Likelihood Explanation

- The attacker is an unprivileged canister caller (spender) with a valid existing ICRC-2 approval.
- No privileged access, no key compromise, no governance majority required.
- The condition (failed `transfer_from` followed by balance replenishment within 24 hours) is realistic in DeFi flows where users receive tokens from multiple sources.
- The spender only needs to retain the original call arguments and `created_at_time` value, which is trivial for any canister.
- The deduplication window is 24 hours, giving a wide replay opportunity.

### Recommendation

Record the transaction hash in `transactions_by_hash` even when `transaction.apply()` fails, provided `created_at_time` was set. This mirrors the intent of the external report's recommendation: a failed transaction should still consume its deduplication slot to prevent future re-execution. Alternatively, document clearly that `created_at_time`-based deduplication does **not** protect against replay of failed transactions, and advise callers to use a fresh `created_at_time` on retry.

### Proof of Concept

1. Owner `O` has 500 tokens. Spender `S` has an ICRC-2 approval from `O` for 1000 tokens.
2. `S` calls `icrc2_transfer_from` with `created_at_time = T`, `amount = 1000`. The call fails: `InsufficientFunds { balance: 500 }`. The tx hash `H` is **not** inserted into `transactions_by_hash`.
3. `O` receives 1000 tokens from another source. Balance is now 1000.
4. Within 24 hours of `T`, `S` replays the identical `icrc2_transfer_from` call with `created_at_time = T`, `amount = 1000`.
5. `apply_transaction` checks `transactions_by_hash.get(&H)` → `None` (not a duplicate). The call succeeds.
6. `O` loses 1000 tokens without having authorized a new transfer. [7](#0-6)
