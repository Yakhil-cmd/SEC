### Title
Disabled Minting Cap Allows Unbounded SNS Token Inflation via `MintSnsTokens` Governance Proposals - (File: rs/sns/governance/src/proposal.rs)

### Summary

The `MintSnsTokens` SNS governance proposal action has its 7-day minting upper-bound enforcement intentionally disabled via a commented-out code block, causing `recent_amount_total_upper_bound_tokens` to return `Decimal::MAX` for every proposal. This means any SNS governance majority can pass an unlimited number of `MintSnsTokens` proposals within a 7-day window, minting an unbounded quantity of SNS tokens to any arbitrary account with no collateral or supply constraint.

### Finding Description

In `rs/sns/governance/src/proposal.rs`, the `TokenProposalAction` implementation for `MintSnsTokens` deliberately bypasses the treasury-valuation-based minting cap:

```rust
// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    // Ideally, we'd return infinity, but Decimal does not have that. This is the next best
    // thing, and should be good enough, because we have already planned the obselences of this
    // code (see tickets NNS1-298(1|2)).
    Ok(Decimal::MAX)
}
```

The correct, cap-enforcing implementation is commented out:

```rust
/* TODO(NNS1-2982): Uncomment.
fn recent_amount_total_upper_bound_tokens(valuation: &Valuation) -> Result<Decimal, String> {
    mint_sns_tokens_7_day_total_upper_bound_tokens(*valuation)
        ...
}
*/
```

The generic enforcement path in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` computes `max_tokens` from `recent_amount_total_upper_bound_tokens` and then checks `proposal_amount_tokens > allowance_remainder_tokens`. Because `max_tokens = Decimal::MAX`, the check `proposal_amount_tokens > Decimal::MAX - spent_tokens` is always false, so every `MintSnsTokens` proposal passes validation regardless of amount.

At execution time, `perform_mint_sns_tokens` in `rs/sns/governance/src/governance.rs` calls `self.ledger.transfer_funds(amount_e8s, 0, None, to, mint.memo())` directly, with no secondary amount check — unlike `TransferSnsTreasuryFunds`, which has a dedicated `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` guard called at execution time. No analogous execution-time guard exists for `MintSnsTokens`.

The `mint_sns_tokens_7_day_total_upper_bound_tokens` function in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` is fully implemented and correct, but is not imported or called because the import is also commented out at the top of `proposal.rs`:

```rust
use ic_sns_governance_proposals_amount_total_limit::{
    // TODO(NNS1-2982): Uncomment. mint_sns_tokens_7_day_total_upper_bound_tokens,
    transfer_sns_treasury_funds_7_day_total_upper_bound_tokens,
};
```

### Impact Explanation

Any SNS governance majority (a set of neuron holders controlling >50% of voting power, which is an unprivileged on-chain role reachable by any token holder who accumulates sufficient stake) can pass repeated `MintSnsTokens` proposals minting an arbitrary number of SNS tokens to any account. This inflates the SNS token supply without bound, diluting all existing holders, and breaks the economic invariant that minting is bounded relative to treasury size. The impact is equivalent to the original report: a privileged role (here, SNS governance majority) can mint unlimited tokens to arbitrary addresses with no collateral backing.

### Likelihood Explanation

Any SNS where a single entity or coordinated group holds a governance majority can exploit this immediately. The entry path is a standard governance proposal submission (`make_proposal` ingress call), which is open to any neuron holder. The disabled cap is a production code path, not a test artifact. The commented-out TODO references ticket NNS1-2982, indicating this is a known incomplete state shipped to production.

### Recommendation

1. Uncomment the `mint_sns_tokens_7_day_total_upper_bound_tokens` import and the `recent_amount_total_upper_bound_tokens` implementation in `rs/sns/governance/src/proposal.rs` to enforce the 7-day treasury-valuation-based cap at proposal submission time.
2. Add an execution-time guard in `perform_mint_sns_tokens` analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` to re-validate the cap at execution time (since time passes between proposal submission and execution).

### Proof of Concept

**Entry path:** Any unprivileged neuron holder submits a `MintSnsTokens` proposal via the SNS governance canister's `make_proposal` ingress endpoint.

**Root cause chain:**

1. `validate_and_render_mint_sns_tokens` calls `treasury_valuation_if_proposal_amount_is_small_enough_or_err` with the `MintSnsTokens` action. [1](#0-0) 

2. That function calls `MintSnsTokens::recent_amount_total_upper_bound_tokens(&valuation)`, which unconditionally returns `Decimal::MAX`. [2](#0-1) 

3. The cap check `proposal_amount_tokens > allowance_remainder_tokens` is always false because `allowance_remainder_tokens ≈ Decimal::MAX`. [3](#0-2) 

4. The disabled implementation that would enforce the real cap: [4](#0-3) 

5. The import of the cap function is also commented out: [5](#0-4) 

6. At execution time, `perform_mint_sns_tokens` mints directly with no amount re-check: [6](#0-5) 

7. The correctly implemented cap function exists but is unused: [7](#0-6)
