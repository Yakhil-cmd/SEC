### Title
SNS Governance `early_decision()` Uses Wrong Denominator for Majority Check, Preventing Early Proposal Resolution - (File: `rs/sns/governance/src/proposal.rs`)

### Summary

`ProposalData::early_decision()` in SNS governance passes `tally.total` (all eligible voting power, including undecided/uncast votes) as the denominator to `majority_decision()`, while the threshold parameter it supplies is `minimum_yes_proportion_of_exercised`. The correct denominator for an *exercised* proportion check is `tally.yes + tally.no`. The sibling function `is_accepted()` in the same file correctly uses `tally.yes + tally.no` for the same threshold. This mismatch means proposals that have achieved an absolute majority of exercised votes are never decided early; they are forced to wait until the voting deadline expires.

---

### Finding Description

`early_decision()` is responsible for detecting when a proposal can be resolved before its deadline because the remaining uncast votes cannot change the outcome. It delegates to `majority_decision()`:

```rust
// rs/sns/governance/src/proposal.rs  line 2350-2363
pub fn early_decision(&self) -> Vote {
    let tally = &self.latest_tally.as_ref()...;
    let minimum_yes_proportion_of_exercised = self.minimum_yes_proportion_of_exercised();

    Self::majority_decision(
        tally.yes,
        tally.no,
        tally.total,                          // ← BUG: includes undecided voters
        minimum_yes_proportion_of_exercised,  // ← threshold is for *exercised* votes
    )
}
```

`majority_decision()` treats its third argument as the denominator for both the yes and no threshold checks:

```rust
// line 2426-2429
if yes * 10_000 > total * required_yes_of_total_basis_points {
    Vote::Yes
} else if no * 10_000 >= total * required_no_of_total_basis_points {
    Vote::No
```

With `tally.total` as the denominator and `minimum_yes_proportion_of_exercised = 50%` (the default), the yes-branch fires only when `yes > tally.total / 2`, i.e., yes exceeds half of **all eligible voters**. The correct semantic is `yes > (yes + no) / 2`, i.e., yes exceeds half of **voters who actually voted**.

By contrast, `is_accepted()` — which runs the same check after the deadline — correctly passes `tally.yes + tally.no`:

```rust
// line 2291-2296
let majority_met = Self::majority_decision(
    tally.yes,
    tally.no,
    tally.yes + tally.no,                 // ← correct: exercised votes only
    minimum_yes_proportion_of_exercised,
) == Vote::Yes;
```

The two functions therefore apply inconsistent semantics for the same threshold, creating a gap where `early_decision` returns `Unspecified` even though `is_accepted` would return `true` for the same tally.

**Concrete example** (default 50% exercised threshold):

| Scenario | yes | no | undecided | total |
|---|---|---|---|---|
| Values | 26 | 0 | 74 | 100 |

- `is_accepted` check: `26 * 10000 > 26 * 5000` → `260000 > 130000` → **Yes** (correct)
- `early_decision` check: `26 * 10000 > 100 * 5000` → `260000 > 500000` → **Unspecified** (wrong)

The proposal has an unambiguous yes-majority among exercised votes, but `early_decision` fails to detect it.

---

### Impact Explanation

`can_make_decision()` calls `early_decision()` to determine whether an absolute majority has been reached:

```rust
// line 2322
let absolute_majority = self.early_decision() != Vote::Unspecified;
```

When `early_decision` incorrectly returns `Unspecified`, `can_make_decision` returns `false` and the proposal is not resolved early. The proposal must wait until its full voting deadline expires before `is_accepted()` is consulted. This delays execution of legitimately adopted SNS governance proposals (including upgrades, treasury transfers, and parameter changes) by the full remaining voting period, which can be days. For time-sensitive SNS actions (e.g., emergency upgrades, treasury operations), this delay is a material governance failure.

---

### Likelihood Explanation

The bug is triggered whenever:
1. A neuron casts a vote on any SNS proposal, and
2. The yes (or no) votes constitute a strict majority of *exercised* votes but not yet a majority of *total eligible* voting power (i.e., there are significant undecided voters).

This is the normal operating condition for most SNS proposals, where voter participation is well below 100%. Any unprivileged neuron holder submitting or voting on a proposal can reach this code path. No special privileges are required.

---

### Recommendation

Change `early_decision()` to pass `tally.yes + tally.no` as the denominator, consistent with `is_accepted()`:

```rust
pub fn early_decision(&self) -> Vote {
    let tally = &self.latest_tally.as_ref()
        .expect("expected latest_tally to not be None");
    let minimum_yes_proportion_of_exercised = self.minimum_yes_proportion_of_exercised();

    Self::majority_decision(
        tally.yes,
        tally.no,
        tally.yes + tally.no,             // ← fix: exercised votes only
        minimum_yes_proportion_of_exercised,
    )
}
```
