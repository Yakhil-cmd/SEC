### Title
Guard Slot Exhaustion via Subaccount Enumeration Blocks All ckBTC Withdrawals - (`rs/bitcoin/ckbtc/minter/src/guard.rs`, `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

---

### Summary

The `retrieve_btc_accounts` guard uses a global `BTreeSet<Account>` capped at `MAX_CONCURRENT = 100`. Because `retrieve_btc_with_approval` keys the guard on the full `Account { owner, subaccount }` tuple — and `from_subaccount` is attacker-supplied — a single unprivileged principal can exhaust all 100 slots using 100 distinct subaccounts, with zero ckBTC balance required to hold each slot.

---

### Finding Description

**Guard implementation** — `rs/bitcoin/ckbtc/minter/src/guard.rs`:

The global limit is a flat count of entries in `retrieve_btc_accounts`: [1](#0-0) [2](#0-1) 

The duplicate check is an exact-`Account` membership test, so two calls from the same principal with different subaccounts are treated as two independent accounts: [3](#0-2) 

**Guard acquisition in `retrieve_btc_with_approval`** — `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`:

The guard key is built directly from the caller-controlled `from_subaccount` field: [4](#0-3) 

The guard is acquired **before** any funds check, and is then held across the `check_address` inter-canister call: [5](#0-4) [6](#0-5) 

The `burn_ckbtcs_icrc2` call that actually requires ckBTC funds only executes **after** `check_address` returns: [7](#0-6) 

**Attack window:** Each in-flight call holds its guard slot for the entire round-trip duration of the `check_address` inter-canister call. During that window the attacker needs no ckBTC balance.

The existing unit test in `guard.rs` already proves the exact exhaustion pattern — 50 guards from subaccounts of a single principal plus 50 from other principals fills all 100 slots: [8](#0-7) 

---

### Impact Explanation

With all 100 guard slots occupied, every subsequent call to either `retrieve_btc` or `retrieve_btc_with_approval` by any legitimate user returns `TemporarilyUnavailable("too many concurrent requests")`: [9](#0-8) 

All ckBTC → BTC withdrawals are blocked for the duration of the attack. The attacker can sustain the DoS by continuously re-submitting 100 calls as previous ones complete, requiring no ckBTC balance at any point.

---

### Likelihood Explanation

- No privileged access, no funds, no social engineering required.
- The attacker only needs a valid Bitcoin address and `amount >= min_retrieve_amount` (a public parameter).
- The `from_subaccount` field is a free 32-byte value; generating 100 distinct values is trivial.
- The attack is repeatable indefinitely and cheap to sustain.

---

### Recommendation

1. **Per-principal slot accounting**: Count guard slots per `owner` principal rather than per `Account`. A single principal should not be able to hold more than one (or a small fixed number of) concurrent slots regardless of subaccount.
2. **Alternatively**: Key the guard on `owner` only (ignoring subaccount), consistent with how `retrieve_btc` already does it: [10](#0-9) 

   `retrieve_btc` passes `subaccount: None` unconditionally, so a single principal can hold at most one slot there. `retrieve_btc_with_approval` should apply the same policy.

3. **Move the guard before the first `await`**: Currently `init_ecdsa_public_key().await` runs before the guard is acquired, which is a separate (minor) ordering issue. The guard should be the very first operation.

---

### Proof of Concept

State-machine test sketch (no ckBTC balance needed):

```rust
// Acquire 100 guard slots from a single principal using distinct subaccounts
let attacker = Principal::from_text("...").unwrap();
let guards: Vec<_> = (0u8..100)
    .map(|i| {
        retrieve_btc_guard(Account {
            owner: attacker,
            subaccount: Some([i; 32]),
        })
        .expect("slot should be available")
    })
    .collect();

// 101st call from any legitimate user is now rejected
let victim = Principal::from_text("...").unwrap();
assert_eq!(
    retrieve_btc_guard(Account { owner: victim, subaccount: None }),
    Err(GuardError::TooManyConcurrentRequests)
);
```

This mirrors the pattern already demonstrated in the existing test at `guard.rs` lines 141–162. [8](#0-7)

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
