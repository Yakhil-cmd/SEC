### Title
SNS Swap `refresh_buyer_tokens` Accepts Arbitrary `buyer` Principal Without Caller Verification - (File: `rs/sns/swap/canister/canister.rs`)

---

### Summary

The `refresh_buyer_tokens` update endpoint in the SNS Swap canister accepts a caller-supplied `buyer` string field that is **never verified against the actual ingress caller**. Any unprivileged principal can invoke this endpoint with any victim's principal as `buyer`, forcing that victim's ICP participation in the swap without their explicit consent and bypassing the SNS-mandated `confirmation_text` requirement.

---

### Finding Description

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` handler resolves the participant principal from the request argument rather than from the authenticated caller:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← attacker-controlled
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The resolved `p` is then passed directly into `refresh_buyer_token_e8s`, which:

1. Validates the caller-supplied `confirmation_text` against the SNS-configured text (a public value).
2. Queries the ICP ledger balance of `p`'s subaccount on the swap canister.
3. Credits that balance to `p` in the swap's `buyers` map, registering `p` as a participant. [2](#0-1) 

There is no check anywhere that `p == caller`. The `confirmation_text` is set by SNS governance and is publicly readable, so an attacker can trivially supply the correct text on behalf of any victim.

The analogous root cause to the DODO report: just as `decodeData.payer` was never checked against `msg.sender` in `D3Proxy.d3MMSwapCallBack`, `arg.buyer` is never checked against `caller_principal_id()` in `refresh_buyer_tokens`.

---

### Impact Explanation

An attacker can force any user's ICP into an SNS swap without that user's explicit consent:

- **Confirmation text bypass**: SNS governance can require a `confirmation_text` so that participants must explicitly agree to swap terms. Because the attacker supplies both `buyer` (victim's principal) and `confirmation_text` (public), the victim's participation is registered as if they confirmed — without them ever sending a message.
- **Forced participation**: If a victim transferred ICP to the swap subaccount but had not yet decided to participate (or transferred by mistake), an attacker can lock that ICP into the swap by calling `refresh_buyer_tokens` with the victim's principal.
- **Swap outcome manipulation**: An attacker can push a swap past its minimum-participation threshold by force-registering multiple victims who have ICP sitting in their swap subaccounts, causing the swap to commit when it otherwise would have aborted.

The victim's ICP is credited to the victim (not the attacker), so this is not direct fund theft — but it constitutes unauthorized commitment of the victim's funds and a complete bypass of the consent mechanism the SNS governance intended to enforce.

---

### Likelihood Explanation

- `refresh_buyer_tokens` is a public `#[update]` endpoint reachable by any ingress sender with no access control.
- The `buyer` field is a plain string; no signature or proof-of-ownership is required.
- The `confirmation_text` is a public governance parameter, trivially readable by anyone.
- Any user who has transferred ICP to the swap subaccount (a normal prerequisite step) is vulnerable before they call `refresh_buyer_tokens` themselves.

---

### Recommendation

Add a check at the top of `refresh_buyer_tokens` that rejects any request where `arg.buyer` is non-empty and does not match the caller:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(requested, caller, "buyer must match caller");
        requested
    };
    ...
}
```

Alternatively, remove the `buyer` field entirely and always derive the participant from `caller_principal_id()`.

---

### Proof of Concept

1. Alice transfers 5 ICP to the swap canister's subaccount keyed by her principal (the standard deposit step). She has not yet called `refresh_buyer_tokens` — she is still reading the swap terms.
2. The SNS has set `confirmation_text = "I agree to the SNS swap terms"` (publicly visible in swap state).
3. Attacker (Bob) sends an ingress update to the swap canister:
   ```
   refresh_buyer_tokens({
     buyer: "<Alice's principal>",
     confirmation_text: Some("I agree to the SNS swap terms")
   })
   ```
4. The swap canister resolves `p = Alice`, validates the confirmation text (passes, because Bob supplied the correct public string), queries Alice's subaccount balance (5 ICP), and writes `buyers["Alice"] = BuyerState { amount_icp_e8s: 5_0000_0000 }`.
5. Alice's 5 ICP is now committed to the swap. She never sent a message to the swap canister. Her explicit confirmation was bypassed. [1](#0-0) [3](#0-2)

```rust
    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
