### Title
Unbounded Tokio Task Accumulation via `MaybeWantsLater` Stash with `SLOT_TABLE_NO_LIMIT` in HTTPS-Outcalls P2P Layer — (`rs/p2p/consensus_manager/src/receiver.rs`, `rs/https_outcalls/consensus/src/gossip.rs`)

---

### Summary

A Byzantine subnet peer can exhaust async task memory on honest replicas by advertising an unbounded number of unique `CanisterHttpResponseId` values with callback IDs above `highest_accepted_request_id`. Because https-outcalls uses `SLOT_TABLE_NO_LIMIT`, the per-peer slot table has no admission cap. Each unique artifact ID spawns a permanent tokio task stashed in `MaybeWantsLater`, waiting indefinitely on a watch channel that never resolves for arbitrarily high IDs.

---

### Finding Description

**Step 1 — No slot table cap for https-outcalls.**

`SLOT_TABLE_NO_LIMIT = usize::MAX` is assigned to the https-outcalls channel: [1](#0-0) [2](#0-1) 

The receiver enforces the limit only as `peer_slot_table_len < self.slot_limit`: [3](#0-2) 

With `slot_limit = usize::MAX`, a Byzantine peer can send slot updates using slot numbers 1, 2, …, N — each a new `HashMap` entry — and the guard never fires.

**Step 2 — Each unique artifact ID spawns a new tokio task.**

In `handle_slot_update_receive`, if the artifact ID is not already in `active_assembles`, a new `process_slot_update` task is spawned unconditionally: [4](#0-3) 

A Byzantine peer can craft N distinct `CanisterHttpResponseId` values (different hashes, all with high callback IDs), causing N tasks to be spawned.

**Step 3 — Bouncer permanently returns `MaybeWantsLater` for high callback IDs.**

The https-outcalls bouncer returns `MaybeWantsLater` for any ID whose callback ID exceeds `highest_accepted_request_id = next_callback_id + MAX_NUMBER_OF_REQUESTS_AHEAD (345)`: [5](#0-4) 

For IDs near `u64::MAX`, `highest_accepted_request_id` can never catch up (it only grows as fast as the subnet processes real requests), so the bouncer returns `MaybeWantsLater` on every 3-second refresh indefinitely.

**Step 4 — Stashed tasks wait forever on the watch channel.**

`should_download` drops the artifact payload but keeps the task alive, blocking on `bouncer_watcher.changed()`: [6](#0-5) 

The bouncer refreshes every 3 seconds: [7](#0-6) 

Each refresh re-evaluates the high-ID artifact and returns `MaybeWantsLater` again. The task never exits `should_download`. The peer counter dropping to zero (e.g., after topology update removes the Byzantine peer) does not cancel the task — `should_download` only observes the bouncer watch channel, not the peer counter.

---

### Impact Explanation

Each stashed task is a live tokio task holding a stack allocation, a cloned `bouncer_rx` watch receiver, and the artifact ID. With N = 100,000 tasks, memory consumption grows proportionally. On a replica with constrained memory, this causes OOM, crashing the honest node. If enough honest nodes crash, the subnet falls below the BFT fault-tolerance threshold, halting consensus and compromising subnet liveness/integrity.

---

### Likelihood Explanation

The attacker must be a subnet member (protocol peer), which is within the Byzantine fault model the IC is designed to tolerate. A single Byzantine node suffices — no threshold corruption is required. The attack requires only sending crafted slot-update messages over the existing QUIC transport connection, which is a normal peer operation. No cryptographic forgery is needed; the Byzantine node signs its own shares with its legitimate key and can freely choose the callback ID field and hash content of the `CanisterHttpResponseId` it advertises.

---

### Recommendation

1. **Apply a per-peer slot limit for https-outcalls.** Replace `SLOT_TABLE_NO_LIMIT` with a bounded constant (e.g., matching the maximum number of concurrent canister HTTP requests per subnet, which is bounded by the execution environment). The ingress channel already uses `SLOT_TABLE_LIMIT_INGRESS = 50_000` as a precedent.

2. **Terminate stashed tasks when peer counter reaches zero.** In `should_download`, also select on a signal derived from the peer counter watch channel so that tasks with no remaining advertisers are cancelled rather than waiting indefinitely for the bouncer.

3. **Bound `MaybeWantsLater` stash size explicitly.** Track the count of stashed tasks globally and reject new adverts that would exceed the bound.

---

### Proof of Concept

```
1. Byzantine node B is a subnet member.
2. B opens its QUIC transport connection to honest node H.
3. B sends slot-update messages:
     for slot_number in 1..=N:
         id = CanisterHttpResponseId {
             callback_id: u64::MAX - slot_number,  // always > highest_accepted_request_id
             hash: random_bytes(),                  // unique per slot
             registry_version: current_version,
         }
         send SlotUpdate { slot_number, commit_id: slot_number, update: Update::Id(id) }
4. H's receiver: slot_limit = usize::MAX → all N entries admitted to slot table.
5. H spawns N process_slot_update tasks.
6. Each task calls should_download → bouncer returns MaybeWantsLater → task blocks on bouncer_watcher.changed().
7. Bouncer refreshes every 3s → still MaybeWantsLater for all N IDs → tasks remain stashed.
8. H's memory grows as O(N); at sufficient N, OOM kills the replica process.
```
