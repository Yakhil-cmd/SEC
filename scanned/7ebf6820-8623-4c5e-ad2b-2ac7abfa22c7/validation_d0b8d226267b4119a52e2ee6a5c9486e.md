### Title
Missing Path Deduplication in `read_state` Allows Amplified CPU Work Per Request - (File: `rs/http_endpoints/public/src/read_state.rs`, `rs/validator/src/ingress_validation.rs`)

---

### Summary

The IC `read_state` endpoint validates path count (≤ 1,000) and depth (≤ 127 labels) but performs **no deduplication** of identical paths before processing. An unprivileged, unauthenticated caller can submit a single `read_state` request containing 1,000 identical paths, causing the replica to perform O(n) work — including repeated state lookups and tree-sort operations — for what is effectively O(1) unique work. This is the IC analog of the GraphQL "Field Duplication" and "Alias Overloading" vulnerabilities described in the report.

---

### Finding Description

The `read_state` request validation in `rs/validator/src/ingress_validation.rs` enforces two structural limits: [1](#0-0) 

```
MAXIMUM_NUMBER_OF_PATHS     = 1,000
MAXIMUM_NUMBER_OF_LABELS_PER_PATH = 127
```

The validation function `validate_paths_width_and_depth` checks only count and depth: [2](#0-1) 

There is **no deduplication step**. After validation passes, the handler calls `verify_paths`, which iterates over every path in the request — including all duplicates — performing per-path work: [3](#0-2) 

For `request_status` paths, each iteration calls `state.get_ingress_status(&message_id)` (a `BTreeMap` lookup into ingress history): [4](#0-3) 

For `canister/*/metadata/*` paths, each iteration calls `can_read_canister_metadata(...)` (reads canister state): [5](#0-4) 

After `verify_paths`, `get_certificate_and_create_response` calls `sparse_labeled_tree_from_paths` on all 1,000 paths. This function sorts the full path list (O(n log n)) before building the tree: [6](#0-5) 

Duplicate paths collapse to a single node in the resulting tree (confirmed by the existing test `sparse_labeled_tree_duplicate_paths`): [7](#0-6) 

So 1,000 identical paths produce the same certified response as 1 path, but the server performs 1,000× the per-path work to get there.

The entire handler runs inside `tokio::task::spawn_blocking`: [8](#0-7) 

Exhausting the blocking thread pool with many concurrent amplified requests degrades the `read_state` endpoint for all users.

---

### Impact Explanation

An unprivileged, unauthenticated caller (anonymous sender is explicitly allowed for `read_state` — expiry is not checked for anonymous requests): [9](#0-8) 

can send a stream of `read_state` requests each containing 1,000 identical `request_status` or `canister/metadata` paths. Each request causes:
- 1,000 `BTreeMap` lookups into ingress history (or 1,000 canister-state reads)
- An O(n log n) sort of 1,000 paths in `sparse_labeled_tree_from_paths`
- Occupation of one blocking thread for the full amplified duration

This amplifies per-request CPU cost by up to 1,000× relative to a single-path request, degrading `read_state` response latency for all users on the replica node. The impact is bounded by `max_request_size_bytes` (which limits total label data per request) but the count-based amplification is independent of label size.

**Impact: 3** (application-level DoS on the `read_state` endpoint; no state corruption or consensus impact)

---

### Likelihood Explanation

The attack requires only crafting a CBOR-encoded `read_state` envelope with 1,000 identical paths — trivially achievable with any HTTP client. No authentication, no cycles, no canister deployment needed. The anonymous sender is explicitly permitted.

**Likelihood: 3** (trivially exploitable by any external caller with HTTP access to a boundary node)

---

### Recommendation

Add a deduplication step in `validate_paths_width_and_depth` (or immediately before `verify_paths`) to reject or deduplicate requests containing repeated paths:

```rust
fn validate_paths_width_and_depth(paths: &[Path]) -> Result<(), RequestValidationError> {
    if paths.len() > MAXIMUM_NUMBER_OF_PATHS {
        return Err(TooManyPaths { ... });
    }
    let mut seen = HashSet::new();
    for path in paths {
        if path.len() > MAXIMUM_NUMBER_OF_LABELS_PER_PATH {
            return Err(PathTooLong { ... });
        }
        // Reject or silently deduplicate duplicate paths
        if !seen.insert(path) {
            return Err(DuplicatePath { ... }); // or just skip
        }
    }
    Ok(())
}
```

Alternatively, deduplicate paths before passing them to `verify_paths` and `sparse_labeled_tree_from_paths`, analogous to the recommendation in the report for GraphQL field duplication: validate incoming requests and reject (or collapse) those containing duplicate fields.

---

### Proof of Concept

Send the following CBOR-encoded `read_state` request (anonymous sender, no signature required) to any replica's public endpoint:

```
POST /api/v2/canister/<canister_id>/read_state
Content-Type: application/cbor

{
  "content": {
    "request_type": "read_state",
    "sender": <anonymous>,
    "ingress_expiry": <any>,
    "paths": [
      ["request_status", <32-byte-message-id>],  // repeated 1,000 times
      ["request_status", <32-byte-message-id>],
      ... (998 more identical entries)
    ]
  }
}
```

The request passes `validate_paths_width_and_depth` (1,000 ≤ 1,000, depth 2 ≤ 127). The `verify_paths` loop then calls `state.get_ingress_status()` 1,000 times and `sparse_labeled_tree_from_paths` sorts 1,000 paths, all to produce a response identical to a single-path request. Sending this in a tight loop from multiple clients amplifies load on the replica's blocking thread pool. [10](#0-9) [11](#0-10) [12](#0-11)
