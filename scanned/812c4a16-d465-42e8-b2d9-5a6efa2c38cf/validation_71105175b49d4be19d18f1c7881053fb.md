### Title
Missing `delivered_ids` Duplicate-Callback Check in `validate_payload` Allows Byzantine Block Proposer to Inject Duplicate Bitcoin Adapter Responses — (`rs/bitcoin/consensus/src/payload_builder.rs`)

### Summary

`validate_payload` (the `BatchPayloadBuilder` impl for `BitcoinPayloadBuilder`) computes `delivered_ids` from `past_payloads` at line 369 but never uses that set to reject duplicate `callback_id` values in the proposed payload. A single Byzantine block proposer can craft a bitcoin payload containing a `callback_id` already present in `past_payloads`; honest validators accept the block, and the execution layer receives a duplicate `ConsensusResponse` for the same callback, violating the at-most-once delivery invariant.

### Finding Description

In `validate_payload`: [1](#0-0) 

`delivered_ids` is computed but then silently discarded. The call to `validate_self_validating_payload_impl` at line 379 receives only the payload and the validation context — never `delivered_ids`.

`validate_self_validating_payload_impl` itself only checks for an empty payload and validates byte size: [2](#0-1) 

There is no guard anywhere in `validate_payload` that rejects a response whose `callback_id` already appears in `past_payloads`. By contrast, `build_payload` correctly passes `delivered_ids` to `get_self_validating_payload_impl` and skips already-delivered IDs: [3](#0-2) 

The `SelfValidatingPayloadBuilder` impl has the same gap — `_past_payloads` is explicitly ignored: [4](#0-3) 

### Impact Explanation

After a duplicate-callback block is finalized, `demux.process_payload` calls `push_response_bitcoin` for every response in the payload: [5](#0-4) 

Inside `push_response`, the behavior differs by response type:

**`GetSuccessorsResponse`** — uses `get_mut` (not `remove`) on `bitcoin_get_successors_contexts`, so the context is still present for the duplicate call: [6](#0-5) 

The second call succeeds and **overwrites** `bitcoin_get_successors_follow_up_responses` with the duplicate's follow-up blocks, corrupting the paginated block data the bitcoin canister will serve to callers.

**`SendTransactionResponse`, `GetSuccessorsReject`, `SendTransactionReject`** — perform **no context lookup at all** and unconditionally push to `consensus_queue`: [7](#0-6) 

Both the original and the duplicate `ConsensusResponse` land in `consensus_queue`. The scheduler drains the queue each round; the second entry targets a callback that has already been consumed, so it fails — but the `bitcoin_get_successors_follow_up_responses` corruption for `GetSuccessorsResponse` is already committed to replicated state before execution runs.

### Likelihood Explanation

Exploiting this requires a single Byzantine block proposer — one malicious replica node acting within its normal turn. No threshold corruption, no key compromise, and no external infrastructure attack is needed. The proposer simply crafts a `SelfValidatingPayload` whose `callback_id` list overlaps with a `past_payload` entry. All honest validators call `validate_payload`, compute `delivered_ids`, and then ignore it, so they notarize and finalize the block normally.

### Recommendation

In `validate_payload`, after computing `delivered_ids`, iterate over the decoded payload and reject any response whose `callback_id` is in `delivered_ids`:

```rust
let delivered_ids = parse::parse_past_payload_ids(past_payloads, &self.log);
// ... decode payload ...
for response in &payload {
    if delivered_ids.contains(&response.callback_id) {
        return Err(ValidationError::InvalidArtifact(
            consensus::InvalidPayloadReason::InvalidSelfValidatingPayload(
                InvalidSelfValidatingPayloadReason::DuplicateResponse,
            ),
        ));
    }
}
```

Apply the same fix to `validate_self_validating_payload` (stop ignoring `_past_payloads`). Additionally, `push_response` for `GetSuccessorsResponse` should use `remove` rather than `get_mut` so that a duplicate response at the state layer is caught by `BitcoinNonMatchingResponse` rather than silently overwriting follow-up data.
