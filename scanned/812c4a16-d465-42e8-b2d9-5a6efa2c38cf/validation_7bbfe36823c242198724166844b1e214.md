### Title
Unbounded `ingress_expiries` Loop in `construction_payloads` Causes Rosetta Node OOM Crash — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

---

### Summary

An unprivileged HTTP client can crash the ICRC-1 Rosetta node by sending a single `POST /construction/payloads` request with a crafted `ingress_end` value (e.g., `u64::MAX`). The server-side loop that generates ingress expiry timestamps has no upper bound on the number of iterations, causing unbounded heap allocation that exhausts process memory.

---

### Finding Description

In `construction_payloads` (`rs/rosetta-api/icrc1/src/construction_api/services.rs`), the function accepts attacker-controlled `ingress_start` and `ingress_end` from the request metadata and uses them to drive a `while` loop: [1](#0-0) 

```rust
let mut ingress_expiries = vec![];
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
```

The only two guards before this loop are:

1. **`ingress_start >= ingress_end`** → returns an error. [2](#0-1) 

2. **`ingress_end < now + ingress_interval`** → returns an error (ensures `ingress_end` is at least one interval in the future). [3](#0-2) 

There is **no upper bound** on `ingress_end`. An attacker sets `ingress_end = u64::MAX` (≈ 1.8 × 10¹⁹ ns), which trivially passes both guards. The step size per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP`, a fixed positive value. With `MAX_INGRESS_TTL ≈ 5 min` and `PERMITTED_DRIFT ≈ 60 s`, `ingress_interval ≈ 240,000,000,000 ns`. The loop runs approximately:

```
(u64::MAX - now) / step ≈ (1.8e19 - 1.7e18) / 2.4e11 ≈ 67,000,000 iterations
```

Each iteration allocates an `EnvelopeContent::Call` struct (containing `canister_id`, `method_name`, `arg` bytes, `nonce`, `sender`, `ingress_expiry`) and a `SigningPayload` into `signing_payloads` and `envelope_contents` Vecs: [4](#0-3) 

The accumulated `Vec<EnvelopeContent>` is then serialized into the `unsigned_transaction` string via CBOR+hex: [5](#0-4) 

The `ConstructionPayloadsResponse` holds both the unbounded `unsigned_transaction: String` and `payloads: Vec<SigningPayload>`: [6](#0-5) 

---

### Impact Explanation

The Rosetta node process exhausts heap memory and crashes (OOM kill or panic). This takes down the entire ICRC-1 Rosetta node, making transaction construction, signing, and submission unavailable. A single HTTP request with a ~100-byte JSON body is sufficient. No authentication is required.

---

### Likelihood Explanation

The Rosetta construction API is a public HTTP endpoint. The exploit requires no privileges, no keys, no prior state, and no volumetric traffic — one request suffices. The crafted values (`ingress_start`, `ingress_end`) are plain `u64` fields in a JSON metadata object with no server-side range validation beyond the two weak guards shown above.

---

### Recommendation

Add a hard cap on the number of ingress expiries that can be generated. For example:

```rust
const MAX_INGRESS_EXPIRIES: usize = 100; // ~24h / 4min interval = ~360 max legitimate

if ingress_expiries.len() >= MAX_INGRESS_EXPIRIES {
    return Err(Error::processing_construction_failed(
        &"ingress_end exceeds maximum allowed window"
    ));
}
```

Alternatively, validate that `ingress_end - ingress_start` does not exceed a maximum window (e.g., 24 hours = `86_400_000_000_000 ns`) before entering the loop.

---

### Proof of Concept

```bash
# Minimal PoC: single HTTP request to crash the Rosetta node
curl -X POST http://<rosetta-node>:8081/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [<valid_transfer_operation>],
    "public_keys": [<valid_public_key>],
    "metadata": {
      "ingress_start": 1700000000000000000,
      "ingress_end": 18446744073709551615
    }
  }'
# Expected: Rosetta process OOM-crashes before responding
```

The `ingress_end = 18446744073709551615` (`u64::MAX`) passes both guards (it is greater than `ingress_start` and greater than `now + ingress_interval`), triggering ~67 million loop iterations and exhausting heap memory.
