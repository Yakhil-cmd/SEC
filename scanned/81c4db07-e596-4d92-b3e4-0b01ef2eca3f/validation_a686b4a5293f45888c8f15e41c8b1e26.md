### Title
SNS Swap Canister Accepts Participation from Restricted Countries Without Enforcement - (File: rs/sns/swap/src/swap.rs)

### Summary
The SNS Swap canister stores a `restricted_countries` field in its on-chain `Init` configuration but never checks it inside `refresh_buyer_token_e8s` or `new_sale_ticket`. Any unprivileged ingress caller from a restricted country can transfer ICP to their swap subaccount and call `refresh_buyer_tokens` directly, bypassing the country restriction entirely.

### Finding Description
The `Init` message of the SNS Swap canister includes an `optional ic_nervous_system.pb.v1.Countries restricted_countries` field (proto field 16), described as "An optional set of countries that should not participate in the swap." [1](#0-0) 

This field is stored in the canister's replicated state and is part of the on-chain `Init` struct: [2](#0-1) 

However, a `grep` of `restricted_countries` across all of `rs/sns/swap/src/swap.rs` returns **zero matches**. Neither `refresh_buyer_token_e8s` nor `new_sale_ticket` consult this field at any point during execution.

`refresh_buyer_token_e8s` performs the following checks before accepting a buyer's ICP: [3](#0-2) 

None of these checks include a `restricted_countries` lookup. The ticket system — which is the only other participation gate — is explicitly documented as optional:

> "The ticketing system is optional in the current implementation, so the participants are free to choose if they use it or not." [4](#0-3) 

The comment inside `refresh_buyer_token_e8s` at the ticket-check block confirms that skipping the ticket is a supported code path: [5](#0-4) 

`new_sale_ticket` similarly performs no `restricted_countries` check: [6](#0-5) 

### Impact Explanation
An SNS creator who sets `restricted_countries` (e.g., to comply with securities regulations in specific jurisdictions) receives no on-chain enforcement. Any user from a restricted country can:

1. Transfer ICP directly to `swap_canister_id` with their principal-derived subaccount on the ICP ledger.
2. Call `refresh_buyer_tokens` as a standard ingress update call.
3. Receive SNS tokens and governance neurons upon swap finalization.

The `restricted_countries` field is part of the canister's replicated state and is surfaced in the public `get_init` query, creating a false expectation that the restriction is enforced on-chain. The actual impact is that legally mandated geographic restrictions are silently unenforced, exposing SNS creators to regulatory liability and violating the intended access-control invariant of the swap.

### Likelihood Explanation
Likelihood is **high**. The attack requires no privileged access, no key material, and no subnet-majority corruption. Any user with an ICP wallet can perform the two-step sequence (ledger transfer + `refresh_buyer_tokens` ingress call) from any jurisdiction. The ticketing system being optional means there is no mandatory gate that could indirectly enforce the restriction.

### Recommendation
Inside `refresh_buyer_token_e8s` (and `new_sale_ticket`), add a check against `self.init.restricted_countries` using the caller's self-declared or IP-derived country code, or — if on-chain country verification is not feasible — remove the `restricted_countries` field from the on-chain `Init` and document that enforcement is purely off-chain, so SNS creators are not misled into believing the canister enforces it.

At minimum, `refresh_buyer_token_e8s` should reject callers whose principal appears in a restricted-country allowlist maintained by the SNS, mirroring the pattern used by `new_sale_ticket`'s anonymous-caller check: [7](#0-6) 

### Proof of Concept

1. An SNS is launched with `restricted_countries: ["US"]` in its `Init`.
2. A US-based user `alice` transfers `min_participant_icp_e8s` ICP to:
   ```
   Account { owner: swap_canister_id, subaccount: principal_to_subaccount(alice) }
   ```
3. `alice` submits an ingress update call to `refresh_buyer_tokens` with `buyer = alice.to_string()` and no `confirmation_text` (or the correct one if set).
4. `refresh_buyer_token_e8s` executes:
   - Checks lifecycle is `Open` ✓
   - Checks `validate_possibility_of_direct_participation` ✓
   - Checks `validate_confirmation_text` ✓
   - Reads ICP balance from ledger ✓
   - Checks participant count and min/max amounts ✓
   - **Never reads `self.init.restricted_countries`** ✓
   - Inserts `alice` into `self.buyers` and updates participation totals.
5. Upon swap finalization, `alice` receives SNS neurons despite being from a restricted country. [8](#0-7)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };

        // Recheck lifecycle state and ICP target after async call because the swap could have
        // been closed (committed or aborted) while the call to get the account balance was
        // outstanding.
        self.validate_lifecycle_is_open()
            .map_err(context_after_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_after_awaiting_icp_ledger_response)?;
