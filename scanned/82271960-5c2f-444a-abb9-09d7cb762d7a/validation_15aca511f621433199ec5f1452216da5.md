### Title
Missing Slippage Protection in `withdraw_erc20` Allows Unbounded ckETH Fee Burn - (File: `rs/ethereum/cketh/minter/src/main.rs`)

---

### Summary

The `withdraw_erc20` function in the ckETH minter canister lacks a user-specified maximum fee parameter. The ckETH gas fee is estimated and burned at execution time without any caller-controlled upper bound, exposing users to paying significantly more ckETH than anticipated when Ethereum gas prices are volatile.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/main.rs`, the `withdraw_erc20` update function accepts a `WithdrawErc20Arg` that contains no `max_cketh_fee` field: [1](#0-0) 

Inside the function, `estimate_erc20_transaction_fee()` is called to determine the ckETH amount to burn as the Ethereum gas fee: [2](#0-1) 

This helper fetches a gas fee estimate (cached for up to 60 seconds) and computes `(2 × base_fee_per_gas + max_priority_fee_per_gas) × 65,000` as the maximum transaction fee: [3](#0-2) 

The resulting `erc20_tx_fee` is immediately burned from the caller's ckETH account with no check against any caller-supplied ceiling: [4](#0-3) 

The `GasFeeEstimate::to_price` path uses a conservative `2 × base_fee_per_gas` multiplier to ensure the estimate covers the next few blocks: [5](#0-4) 

The documentation explicitly acknowledges that the fee is variable and recommends pre-approving a large allowance (e.g., 1 ETH): [6](#0-5) 

Because the user pre-approves a large ckETH allowance and the minter burns whatever the current estimate is, there is no on-chain mechanism for the user to say "only proceed if the fee is ≤ X ckETH."

The `WithdrawErc20Error` type has no variant for "fee too high," confirming the minter never rejects a withdrawal on fee grounds from the user's perspective: [7](#0-6) 

The same structural gap exists in `withdraw_eth`: the ETH amount received equals `withdrawal_amount − max_tx_fee_estimate`, where `max_tx_fee_estimate` is determined asynchronously after the ckETH is already burned, and there is no `min_eth_received` parameter in `WithdrawalArg`: [8](#0-7) 

---

### Impact Explanation

**Impact: High.**

A user calling `withdraw_erc20` pre-approves a large ckETH allowance. The minter burns `erc20_tx_fee` ckETH at call time based on the current Ethereum gas estimate. During periods of gas price spikes (e.g., NFT mints, network congestion), the fee formula `2 × base_fee_per_gas × 65,000` can be several times higher than the user anticipated. The user has no on-chain recourse: the burn is irreversible, and overcharged transaction fees are explicitly documented as non-reimbursable: [9](#0-8) 

For `withdraw_eth`, the user receives `withdrawal_amount − max_tx_fee_estimate` ETH, where `max_tx_fee_estimate` is determined after the ckETH is already burned. If gas prices rise between submission and processing, the user receives less ETH than expected with no protection.

---

### Likelihood Explanation

**Likelihood: Low.**

Ethereum gas prices are volatile but spikes of 2–5× within a 60-second window (the cache TTL) are uncommon under normal conditions. However, during high-activity events (large NFT drops, DeFi liquidation cascades), such spikes are historically observed. The IC has no mempool, so traditional sandwich attacks are not applicable, but natural gas price volatility is sufficient to trigger the issue. Users who pre-approve large ckETH allowances (as the documentation recommends) are fully exposed.

---

### Recommendation

1. Add an optional `max_cketh_fee: opt nat` field to `WithdrawErc20Arg` in `cketh_minter.did`.
2. In `withdraw_erc20`, after computing `erc20_tx_fee`, check `if erc20_tx_fee > max_cketh_fee { return Err(...) }` before burning.
3. Add an optional `min_eth_received: opt nat` field to `WithdrawalArg` for `withdraw_eth`, and reject the request if `withdrawal_amount − max_tx_fee_estimate < min_eth_received`.
4. Expose a `WithdrawErc20Error::FeeTooHigh { actual_fee: nat; max_fee: nat }` variant so callers can distinguish this rejection.

---

### Proof of Concept

1. User queries `eip1559_transaction_price` and sees a fee of 0.005 ckETH. They approve the minter for 1 ckETH on the ckETH ledger.
2. User calls `withdraw_erc20` with their ckERC20 amount and recipient.
3. Between the query and the call execution (or within the 60-second cache window), Ethereum gas prices spike 10×.
4. `estimate_erc20_transaction_fee()` returns 0.05 ckETH (10× the expected amount).
5. The minter burns 0.05 ckETH from the user's account — 10× more than the user anticipated — with no revert path.
6. The ERC20 withdrawal proceeds normally, but the user has lost 0.045 ckETH more than expected, with no reimbursement. [10](#0-9) [11](#0-10)

pub async fn lazy_refresh_gas_fee_estimate() -> Option<GasFeeEstimate> {
    const MAX_AGE_NS: u64 = 60_000_000_000_u64; //60 seconds

    async fn do_refresh() -> Option<GasFeeEstimate> {
        let _guard = match TimerGuard::new(TaskType::RefreshGasFeeEstimate) {
            Ok(guard) => guard,
            Err(e) => {
                log!(
                    DEBUG,
                    "[refresh_gas_fee_estimate]: Failed retrieving guard: {e:?}",
                );
                return None;
            }
        };

        let fee_history = match eth_fee_history().await {
            Ok(fee_history) => fee_history,
            Err(e) => {
                log!(
                    INFO,
                    "[refresh_gas_fee_estimate]: Failed retrieving fee history: {e:?}",
                );
                return None;
            }
        };

        let gas_fee_estimate = match estimate_transaction_fee(&fee_history) {
            Ok(estimate) => {
                mutate_state(|s| {
                    s.last_transaction_price_estimate =
                        Some((ic_cdk::api::time(), estimate.clone()));
                });
                estimate
            }
            Err(e) => {
                log!(
                    INFO,
                    "[refresh_gas_fee_estimate]: Failed estimating gas fee: {e:?}",
                );
                return None;
            }
        };
        log!(
            INFO,
            "[refresh_gas_fee_estimate]: Estimated transaction fee: {:?}",
            gas_fee_estimate,
        );
        Some(gas_fee_estimate)
    }

    async fn eth_fee_history() -> Result<FeeHistory, MultiCallError<FeeHistory>> {
        read_state(rpc_client)
            .fee_history((5_u8, BlockTag::Latest))
            .with_reward_percentiles(vec![20])
            .with_cycles(MIN_ATTACHED_CYCLES)
            .try_send()
            .await
            .reduce_with_strategy(StrictMajorityByKey::new(|fee_history: &FeeHistory| {
                Nat::from(fee_history.oldest_block.clone())
            }))
    }

    let now_ns = ic_cdk::api::time();
    match read_state(|s| s.last_transaction_price_estimate.clone()) {
        Some((last_estimate_timestamp_ns, estimate))
            if now_ns < last_estimate_timestamp_ns.saturating_add(MAX_AGE_NS) =>
        {
            Some(estimate)
        }
        _ => do_refresh().await,
    }
}

type WithdrawErc20Error = variant {
    // The user provided ckERC20 token is not supported by the minter.
    TokenNotSupported : record {supported_tokens : vec CkErc20Token};

    // Recipient's address is blocked.
    // No withdrawal can be made to that address.
    RecipientAddressBlocked : record { address : text };

    // The minter could not burn the required amount of ckETH to pay for the transaction fees.
    CkEthLedgerError : record { error : LedgerError };

    // The minter could not burn the requested amount of ckERC20 tokens.
    // The `cketh_block_index` identifies the burn that occurred on the ckETH ledger and that will be reimbursed.
    CkErc20LedgerError : record { cketh_block_index : nat; error : LedgerError };

    // The minter is temporarily unavailable, retry the request.
    // The payload contains a human-readable message explaining what caused the unavailability.
    TemporarilyUnavailable : text;
};
