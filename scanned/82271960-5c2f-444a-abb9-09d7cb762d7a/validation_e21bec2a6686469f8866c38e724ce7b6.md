### Title
BTC Depositors Locked Out of Minting ckBTC When Minter Mode Changes After On-Chain Deposit — (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

---

### Summary

The ckBTC minter's `update_balance` endpoint enforces the current `Mode` access restriction at call time, even for users who have already irreversibly sent BTC to the minter's derived Bitcoin address. If the minter mode is changed to `RestrictedTo` or `DepositsRestrictedTo` (excluding the depositor) or `ReadOnly` after the BTC is on-chain but before the user calls `update_balance`, the user is permanently blocked from minting their ckBTC — with no recourse — until the mode is changed back.

---

### Finding Description

The ckBTC minter defines a `Mode` enum that controls which principals may interact with the minter: [1](#0-0) 

`is_deposit_available_for` enforces this restriction: [2](#0-1) 

`update_balance` — the only user-callable endpoint to mint ckBTC for deposited UTXOs — checks this at the very start of execution: [3](#0-2) 

The check is performed against the **caller** (the IC principal), not against whether the UTXO was deposited before or after the mode change. This means the restriction is applied retroactively to users who already sent BTC on-chain.

The BTC deposit flow is:
1. User sends BTC to a deterministic address derived from their IC principal (irreversible on-chain action).
2. User calls `update_balance` to notify the minter and receive ckBTC.

If the minter mode changes between steps 1 and 2 — removing the user from the allowlist — step 2 fails with `TemporarilyUnavailable`. The user's BTC is locked at the minter-controlled address with no user-callable path to recover it.

This is structurally identical to the reported `withdrawAvailableCollateral` issue: a user who committed assets before a whitelist change is blocked from reclaiming them by a restriction that should only apply to new actions.

The same pattern exists in the ckDoge minter, which uses an identical `Mode` enum and `update_balance` flow: [4](#0-3) 

---

### Impact Explanation

A user who has already sent BTC to the minter's Bitcoin address cannot mint ckBTC if:
- The minter is upgraded to `Mode::RestrictedTo(vec![...])` not including them.
- The minter is upgraded to `Mode::DepositsRestrictedTo(vec![...])` not including them.
- The minter is upgraded to `Mode::ReadOnly`.

The BTC is held at a minter-controlled address. The user has no alternative path to claim ckBTC or recover the BTC. The funds remain locked until the mode is changed back — which may never happen if the restriction is permanent (e.g., a migration to a new minter). This is a direct ledger conservation bug: BTC is deposited but ckBTC is never minted.

**Impact: Medium** — user funds are locked, not permanently destroyed, but the user has no recourse.

---

### Likelihood Explanation

The minter mode is changed via canister upgrade arguments (`UpgradeArgs { mode: Some(...) }`), which is a routine NNS governance action. The window between a user sending BTC on-chain (which requires Bitcoin confirmations — at least 12 by default) and calling `update_balance` is non-trivial. Any governance upgrade that restricts the mode during this confirmation window affects all in-flight depositors not in the new allowlist. [5](#0-4) 

**Likelihood: Medium** — realistic during minter migrations, security incidents, or routine mode changes.

---

### Recommendation

Remove the `is_deposit_available_for` check from `update_balance`, or scope it only to UTXOs that arrived **after** the mode change. The mode restriction is intended to prevent new deposits, not to block users from claiming ckBTC for BTC they have already irreversibly committed on-chain. Canceling or claiming already-deposited assets should never be gated by a whitelist.

Analogously, `retrieve_btc` / `retrieve_btc_with_approval` check `is_withdrawal_available_for` — the same issue applies if a user has already transferred ckBTC to the minter's withdrawal account but the mode changes before they call `retrieve_btc`. [6](#0-5) 

---

### Proof of Concept

1. User sends 1 BTC to their minter-derived address on Bitcoin mainnet (irreversible).
2. NNS governance upgrades the ckBTC minter with `UpgradeArgs { mode: Some(Mode::RestrictedTo(vec![other_principal])), .. }` — the user's principal is not in the list.
3. After 12 Bitcoin confirmations, user calls `update_balance`.
4. `is_deposit_available_for` returns `Err("access to the minter is temporarily restricted")`.
5. `update_balance` returns `Err(UpdateBalanceError::TemporarilyUnavailable(...))`.
6. User's 1 BTC is locked at the minter's address. No ckBTC is minted. No recovery path exists for the user. [7](#0-6) [8](#0-7)

impl Mode {
    /// Returns Ok if the specified principal can convert BTC to ckBTC.
    pub fn is_deposit_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("access to the minter is temporarily restricted".to_string());
                }
                Ok(())
            }
            Self::DepositsRestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC deposits are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }

    /// Returns Ok if the specified principal can convert ckBTC to BTC.
    pub fn is_withdrawal_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability | Self::DepositsRestrictedTo(_) => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC withdrawals are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }

/// Notifies the ckBTC minter to update the balance of the user subaccount.
pub async fn update_balance<R: CanisterRuntime>(
    args: UpdateBalanceArgs,
    runtime: &R,
) -> Result<Vec<UtxoStatus>, UpdateBalanceError> {
    let caller = runtime.caller();
    if args.owner.unwrap_or(caller) == runtime.id() {
        ic_cdk::trap("cannot update minter's balance");
    }

    // Record start time of method execution for metrics
    let start_time = runtime.time();

    // When the minter is in the mode using a whitelist we only want a certain
    // set of principal to be able to mint. But we also want those principals
    // to mint at any desired address. Therefore, the check below is on "caller".
    state::read_state(|s| s.mode.is_deposit_available_for(&caller))
        .map_err(UpdateBalanceError::TemporarilyUnavailable)?;
