### Title
Unauthorized SNS Swap Participation Registration via Arbitrary `buyer` Parameter Bypasses Confirmation-Text Consent Mechanism - (File: `rs/sns/swap/canister/canister.rs`)

---

### Summary

The `refresh_buyer_tokens` update endpoint in the SNS Swap canister accepts an arbitrary `buyer` principal in its request payload without verifying that the caller is that principal. Any unprivileged ingress sender can call this endpoint specifying a victim's principal as `buyer`, registering the victim's ICP participation in the swap and bypassing the `confirmation_text` consent mechanism — even when the SNS explicitly requires explicit user agreement before participation.

---

### Finding Description

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` handler resolves the effective buyer principal from the request payload rather than from `msg_caller()`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← no caller == buyer check
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The `buyer` field is documented as "If not specified, the caller is used," but when it is specified, there is no authorization check that the caller equals the supplied principal. [2](#0-1) 

Inside `refresh_buyer_token_e8s`, the swap canister checks the ICP balance of the subaccount derived from the supplied `buyer` principal and records that principal as a participant:

```rust
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),
};
icp_ledger.account_balance(account).await ...
``` [3](#0-2) 

The `confirmation_text` field — the only consent gate — is validated against the SNS's publicly readable init configuration. Since the text is retrievable by anyone via the `get_init` query endpoint, an attacker can supply it on behalf of the victim: [4](#0-3) 

The proto definition confirms the field is public and caller-agnostic: [5](#0-4) 

---

### Impact Explanation

When an SNS configures a `confirmation_text` (e.g., "I agree to the token sale terms"), the intent is that each participant must explicitly affirm consent before their ICP is locked into the swap. Because `refresh_buyer_tokens` accepts an arbitrary `buyer` without verifying the caller, an attacker can:

1. Read the public `confirmation_text` from `get_init`.
2. Call `refresh_buyer_tokens` with `buyer = victim_principal` and the correct `confirmation_text`.
3. The swap canister reads the ICP balance already sitting in the victim's swap subaccount and records the victim as a committed buyer.

If the swap commits, the victim's ICP is swept to the SNS treasury and the victim receives SNS tokens they never explicitly agreed to purchase. The victim cannot reverse this — they must wait for a swap abort (which may never happen) to reclaim ICP via `error_refund_icp`. This is a direct ledger conservation impact: ICP is irreversibly exchanged for SNS tokens without the owner's explicit consent at the time of the call.

---

### Likelihood Explanation

The precondition — that the victim has already transferred ICP to the swap subaccount — is the normal first step of the participation flow. Users routinely transfer ICP before calling `refresh_buyer_tokens`. The window between the ICP transfer and the user's own `refresh_buyer_tokens` call is the attack surface. The attacker needs only the victim's principal (publicly observable on-chain) and the `confirmation_text` (readable via `get_init` query). No privileged access, key material, or threshold corruption is required. Any unprivileged ingress sender can execute this.

---

### Recommendation

Enforce that the resolved buyer principal equals the caller's principal. If third-party notification (e.g., a bot notifying on behalf of a user) is a desired feature, it must be a separate endpoint that does **not** accept or validate a `confirmation_text` on the victim's behalf — the confirmation text must only be accepted when `buyer` is empty (i.e., the caller is the buyer):

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let p = PrincipalId::from_str(&arg.buyer).unwrap();
    // If a third-party is notifying, confirmation_text must not be accepted
    if arg.confirmation_text.is_some() && p != caller_principal_id() {
        panic!("confirmation_text can only be submitted by the buyer themselves");
    }
    p
};
```

Alternatively, remove the ability to specify an arbitrary `buyer` entirely and always use `caller_principal_id()`.

---

### Proof of Concept

1. Alice transfers 100 ICP to the swap canister's subaccount for her principal (`swap_canister_subaccount = principal_to_subaccount(alice_principal)`).
2. Alice intends to read the SNS terms before confirming participation.
3. Attacker queries `get_init` on the swap canister (a public query) and reads `confirmation_text = "I agree to the SNS token sale terms"`.
4. Attacker sends an ingress message to `refresh_buyer_tokens` with:
   - `buyer = alice_principal.to_string()`
   - `confirmation_text = Some("I agree to the SNS token sale terms")`
5. The swap canister resolves `p = alice_principal`, validates the confirmation text (passes, because the text matches), reads Alice's subaccount balance (100 ICP), and records Alice as a buyer with 100 ICP committed.
6. The swap commits. Alice's 100 ICP is swept to the SNS treasury; Alice receives SNS tokens she never explicitly agreed to purchase. [1](#0-0) [6](#0-5)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
