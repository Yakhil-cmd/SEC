### Title
ICP Ledger `account_balance` Silently Returns Zero for Invalid-Length Account Identifiers Instead of Rejecting Them - (File: rs/ledger_suite/icp/ledger/src/main.rs)

### Summary
The ICP Ledger canister's `account_balance` Candid endpoint silently returns `Tokens::ZERO` when the caller supplies an account identifier blob of incorrect length (anything other than 32 bytes), instead of trapping or returning an error. This is a direct analog to the Starknet `validateAndParseAddress` lax-validation bug: a non-canonical or malformed address input is silently accepted and produces a misleading result rather than being rejected.

### Finding Description

The `account_balance` query endpoint is defined in `rs/ledger_suite/icp/ledger/src/main.rs`:

```rust
#[query(name = "account_balance")]
fn account_balance_candid_(arg: AccountIdentifierByteBuf) -> Tokens {
    match BinaryAccountBalanceArgs::try_from(arg) {
        Ok(arg) => {
            let account = AccountIdentifier::from_address(arg.account).unwrap_or_else(|e| {
                trap(format!("Invalid account identifier: {e}"));
            });
            account_balance(account)
        }
        Err(_) => Tokens::ZERO,   // <-- silent failure path
    }
}
``` [1](#0-0) 

The `Err(_) => Tokens::ZERO` branch is reached whenever `BinaryAccountBalanceArgs::try_from(arg)` fails. That conversion fails for any blob that is not exactly 32 bytes:

```rust
impl TryFrom<AccountIdentifierByteBuf> for BinaryAccountBalanceArgs {
    type Error = String;
    fn try_from(value: AccountIdentifierByteBuf) -> Result<Self, Self::Error> {
        Ok(BinaryAccountBalanceArgs {
            account: AccountIdBlob::try_from(value.account.as_slice()).map_err(|_| {
                format!(
                    "Invalid account identifier length (expected 32, got {})",
                    value.account.len()
                )
            })?,
        })
    }
}
``` [2](#0-1) 

The behavior is confirmed by an existing integration test that explicitly asserts a zero-length account identifier returns `Tokens::from_e8s(0)` with no error: [3](#0-2) 

The canonical `AccountIdentifier` format is a 32-byte blob: 4-byte CRC32 checksum followed by 28-byte hash. When the 32-byte format is provided, the checksum is verified and a bad checksum causes a `trap`. But when a non-32-byte blob is provided, the endpoint silently returns zero with no indication to the caller that the query was invalid.

The `account_balance_pb` protobuf endpoint has a related but distinct issue: it accepts 28-byte account identifiers (raw hash, no checksum) via `AccountIdentifier::from_slice()`, which explicitly allows the 28-byte format without any integrity check:

```rust
pub fn from_slice(v: &[u8]) -> Result<AccountIdentifier, AccountIdParseError> {
    match v.try_into() {
        Ok(h) => check_sum(h).map_err(AccountIdParseError::InvalidChecksum),
        Err(_) => match v.try_into() {
            Ok(hash) => Ok(AccountIdentifier { hash }),  // 28-byte: no checksum
            Err(_) => Err(AccountIdParseError::InvalidLength(v.to_vec())),
        }
    }
}
``` [4](#0-3) 

This is used by `AccountIdentifier::from_proto()` which feeds both `account_balance_pb` and `send_pb`: [5](#0-4) 

### Impact Explanation

A dApp or canister caller that supplies a malformed (wrong-length) account identifier blob to `account_balance` receives `Tokens::ZERO` with no error signal. The caller cannot distinguish "this account genuinely has zero balance" from "this query was invalid." A dApp that gates a transaction on a balance check (e.g., "only proceed if balance > 0") would silently skip the transaction when given a malformed identifier, with no indication of the input error. For `send_pb` and `account_balance_pb`, the 28-byte format bypasses checksum validation entirely: a typo in any of the 28 hash bytes is not caught, and a transfer would proceed to the wrong account.

### Likelihood Explanation

The `account_balance` Candid endpoint is publicly callable by any unprivileged ingress sender or canister. Any client that constructs the `AccountIdentifierByteBuf` argument incorrectly (e.g., passes a hex-decoded string of wrong length, or passes a 28-byte hash instead of the 32-byte canonical form) will silently receive zero. The `send_pb` protobuf endpoint is used by legacy clients (older dfx versions, some wallets) and accepts 28-byte identifiers without checksum, making typo-induced misdirected transfers undetectable at the API layer.

### Recommendation

1. The `account_balance` Candid endpoint should `trap` (or return a `Result` error) when `BinaryAccountBalanceArgs::try_from(arg)` fails, rather than silently returning `Tokens::ZERO`. The error message is already constructed inside `try_from`; it should be surfaced to the caller.
2. The `account_balance_pb` and `send_pb` protobuf endpoints should require the 32-byte canonical format (with checksum) and reject 28-byte inputs, or at minimum document clearly that 28-byte inputs bypass integrity checking.
3. Callers and dApps should be advised to always use the 32-byte canonical account identifier format (4-byte CRC32 + 28-byte hash) to benefit from checksum protection.

### Proof of Concept

An unprivileged ingress sender calls `account_balance` on the ICP Ledger canister with a `AccountIdentifierByteBuf { account: ByteBuf::from(vec![0u8; 0]) }` (zero-length blob). The canister returns `Tokens { e8s: 0 }` with no error, as confirmed by the existing test at: [3](#0-2) 

The same behavior occurs for any blob length other than 32 (e.g., 1, 16, 28, 31, 33 bytes). The caller receives `Tokens::ZERO` and has no way to distinguish this from a genuine zero balance. The root cause is the `Err(_) => Tokens::ZERO` arm at: [6](#0-5)
