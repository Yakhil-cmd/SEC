### Title
`icrc1_transfer` and `icrc2_transfer_from` trap instead of returning typed errors on `can_send` failure, breaking ICRC-1/ICRC-2 standard composability - (File: `rs/ledger_suite/icp/ledger/src/main.rs`)

---

### Summary

The ICP ledger's `icrc1_transfer` and `icrc2_transfer_from` endpoints call `trap()` — a canister-level panic — when the caller is not permitted to hold tokens (via the `can_send()` guard), instead of returning a typed `TransferError` or `TransferFromError`. The ICRC-1 and ICRC-2 standards mandate that all failure conditions be communicated through typed error variants in the `Result` return value. Trapping instead deviates from the standard and breaks composability with any canister that integrates with the ICP ledger using the standard interface.

---

### Finding Description

In `rs/ledger_suite/icp/ledger/src/main.rs`, both `icrc1_transfer` and `icrc2_transfer_from` begin with a `can_send` guard:

```rust
// icrc1_transfer — lines 811–817
if !LEDGER.read().unwrap().can_send(&PrincipalId::from(caller())) {
    trap("Caller cannot hold tokens on the ledger.");
}

// icrc2_transfer_from — lines 847–857
if !LEDGER.read().unwrap().can_send(&PrincipalId::from(caller())) {
    trap("Caller cannot hold tokens on the ledger.");
}
if !LEDGER.read().unwrap().feature_flags.icrc2 {
    trap("ICRC-2 features are not enabled on the ledger.");
}
``` [1](#0-0) [2](#0-1) 

Both functions are declared with the standard ICRC-1/ICRC-2 return types (`Result<Nat, TransferError>` and `Result<Nat, TransferFromError>` respectively). The ICRC-1 standard requires that `icrc1_transfer` communicate every failure through a typed `TransferError` variant — it must never trap on a caller-induced condition. Trapping is a system-level canister abort, not an application-level error, and it propagates to the caller as a `CanisterCalledTrap` rejection code rather than a decoded `Err(TransferError)`.

The `icrc2_transfer_from` function compounds this by also trapping unconditionally when the `icrc2` feature flag is disabled, confirmed by the integration test `expect_icrc2_disabled`:

```rust
err.assert_contains(
    ErrorCode::CanisterCalledTrap,
    "ICRC-2 features are not enabled on the ledger.",
);
``` [3](#0-2) 

---

### Impact Explanation

Any canister that integrates with the ICP ledger by calling `icrc1_transfer` or `icrc2_transfer_from` and pattern-matches on `Result<Nat, TransferError>` / `Result<Nat, TransferFromError>` will receive a system-level `CanisterCalledTrap` rejection instead of a typed error when the `can_send` guard fires. The calling canister cannot distinguish this from an internal ledger crash. Its `Err(TransferError::GenericError { … })` handler is never reached; instead the inter-canister call itself fails, potentially causing the calling canister's own transaction to roll back or enter an unexpected error path. This is the direct IC analog of the ERC-1155 composability break: any protocol that calls `icrc1_transfer` expecting a `Result` will silently fail for callers that are on the ledger's send-restriction list, even when all supplied arguments are otherwise valid.

---

### Likelihood Explanation

The `can_send` restriction applies to specific principals (e.g., certain system canisters or blocked accounts). Any third-party canister that happens to be on that list — or that acts on behalf of such a principal — will trigger the trap on every call to `icrc1_transfer`. Because the ICP ledger is a core system canister widely integrated across the ecosystem, the surface area is large. An unprivileged canister caller is the direct entry path; no privileged access is required to reach the trap.

---

### Recommendation

Replace each `trap()` call inside the standard ICRC-1/ICRC-2 endpoints with a typed error return that conforms to the standard:

```rust
// icrc1_transfer
if !LEDGER.read().unwrap().can_send(&PrincipalId::from(caller())) {
    return Err(TransferError::GenericError {
        error_code: Nat::from(0u64),
        message: "Caller cannot hold tokens on the ledger.".to_string(),
    });
}

// icrc2_transfer_from — feature-flag guard
if !LEDGER.read().unwrap().feature_flags.icrc2 {
    return Err(TransferFromError::GenericError {
        error_code: Nat::from(0u64),
        message: "ICRC-2 features are not enabled on the ledger.".to_string(),
    });
}
```

This preserves the rejection semantics while keeping the call result decodable as a standard `Result`, restoring composability for all integrating canisters.

---

### Proof of Concept

1. Deploy a canister `C` whose principal is on the ICP ledger's send-restriction list (i.e., `can_send()` returns `false` for it).
2. From canister `C`, issue an inter-canister call to the ICP ledger's `icrc1_transfer` endpoint with otherwise valid arguments (valid `to`, positive `amount`, correct `fee`).
3. Observe that the call does **not** return `Err(TransferError::GenericError { … })`; instead the inter-canister call itself is rejected with `CanisterCalledTrap`.
4. Any error-handling logic in `C` that matches on `TransferError` variants is never executed, and `C`'s own state may be left inconsistent depending on how it handles the unexpected rejection. [4](#0-3) [5](#0-4)
