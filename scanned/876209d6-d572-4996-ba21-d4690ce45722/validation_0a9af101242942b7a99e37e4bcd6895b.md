### Title
Unbounded `ingress_expiries` Loop in ICRC1 Rosetta `construction_payloads` Causes OOM DoS — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

---

### Summary

An unauthenticated attacker can POST to `/construction/payloads` with an arbitrarily large `ingress_end` value. The server enters an unbounded `while` loop that allocates one `EnvelopeContent::Call` and one `SigningPayload` per iteration, exhausting process memory and crashing the Rosetta server.

---

### Finding Description

In `construction_payloads` (ICRC1 Rosetta), the only guards on the caller-supplied `ingress_start`/`ingress_end` are: [1](#0-0) 

Neither check bounds the *range* `ingress_end - ingress_start`. The loop that follows has no iteration cap: [2](#0-1) 

The step size per iteration is:

```
step = ingress_interval − INGRESS_INTERVAL_OVERLAP
     = (MAX_INGRESS_TTL − PERMITTED_DRIFT) − INGRESS_INTERVAL_OVERLAP
     = (300s − 60s) − 120s
     = 120 seconds = 1.2 × 10¹¹ ns
``` [3](#0-2) [4](#0-3) 

With `ingress_start = now` (≈ 1.7 × 10¹⁸ ns) and `ingress_end = u64::MAX` (≈ 1.84 × 10¹⁹ ns):

```
iterations ≈ (1.84×10¹⁹ − 1.7×10¹⁸) / 1.2×10¹¹ ≈ 139,000,000
```

Each iteration pushes one `EnvelopeContent::Call` (containing cloned `canister_method_args` bytes, method name string, sender, nonce, etc.) into `envelope_contents`, and one `SigningPayload` into `signing_payloads` inside `handle_construction_payloads`: [5](#0-4) 

At even a conservative 300 bytes per pair of entries, 139 million iterations allocates **~42 GB** before the function returns — far beyond any realistic server memory budget.

The HTTP endpoint is fully public (no authentication, no rate-limit visible in the handler): [6](#0-5) 

The same structural issue exists in the ICP Rosetta variant: [7](#0-6) 

---

### Impact Explanation

The ICRC1 Rosetta server process is killed by the OS OOM killer (or panics on allocation failure). All in-flight requests are dropped. Any operator or user relying on the Rosetta API for transaction construction, submission, or monitoring is denied service until the process is restarted. A single malicious HTTP request is sufficient; the attack is repeatable.

---

### Likelihood Explanation

The endpoint is unauthenticated, reachable over the network, and requires only a valid JSON body with a large `ingress_end` field. No privileged access, key material, or protocol-level attack is needed. The exploit is trivially reproducible with `curl`.

---

### Recommendation

Add an explicit iteration cap immediately after the existing guards:

```rust
const MAX_INGRESS_EXPIRIES: usize = 100; // ~200 minutes of coverage

if ingress_start >= ingress_end { ... }
if ingress_end < now + ingress_interval { ... }

// NEW: reject if the requested window would generate too many expiries
let range = ingress_end.saturating_sub(ingress_start);
if range / step > MAX_INGRESS_EXPIRIES as u64 {
    return Err(Error::processing_construction_failed(
        "ingress window too large"
    ));
}
```

Alternatively, truncate the loop after `MAX_INGRESS_EXPIRIES` entries are collected. Apply the same fix to the ICP Rosetta variant.

---

### Proof of Concept

```bash
# ingress_start = current time in nanoseconds
NOW=$(python3 -c "import time; print(int(time.time()*1e9))")
INGRESS_END=18446744073709551615  # u64::MAX

curl -X POST http://rosetta-server:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\":\"Internet Computer\",\"network\":\"<ledger_id>\"},
    \"operations\": [<valid_transfer_op>],
    \"public_keys\": [<valid_pubkey>],
    \"metadata\": {
      \"ingress_start\": $NOW,
      \"ingress_end\": $INGRESS_END
    }
  }"
# Server enters ~139M-iteration loop, allocates ~42 GB, crashes.
```
