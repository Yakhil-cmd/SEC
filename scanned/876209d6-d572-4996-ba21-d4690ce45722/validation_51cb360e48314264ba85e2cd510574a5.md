### Title
Stale Yes-Votes Persist in Root Proposal Ballot After Node Operator Controls Multiple Nodes — (`rs/nns/handlers/root/impl/src/root_proposals.rs`)

### Summary

In `vote_on_root_proposal_to_upgrade_governance_canister`, when a node operator who controls multiple NNS subnet nodes casts a `Yes` vote, all of their per-node ballot entries are set to `Yes`. If that same operator later calls the function again to change their vote to `No`, all matching entries are updated. However, the `is_byzantine_majority_yes` tally counts every ballot entry in `node_operator_ballots` independently — one per node, not one per operator. A node operator controlling K nodes gets K yes-votes counted toward the threshold. This is the intended design. The analog vulnerability is: the `is_byzantine_majority_yes` threshold denominator `num_nodes` is derived from `node_operator_ballots.len()`, which is the total number of node-ballot entries (one per node). But the threshold formula `votes_yes >= (num_nodes - max_faults)` is designed for a model where each node casts exactly one independent vote. When a single operator controls multiple nodes and votes Yes, they contribute multiple yes-votes to the numerator while the denominator stays fixed at the total node count. This means a single operator controlling enough nodes can single-handedly satisfy the byzantine majority threshold — without any other operator's participation — even though the security model assumes the threshold requires agreement from independent node operators.

### Finding Description

In `submit_root_proposal_to_upgrade_governance_canister`, the ballot list is built by iterating over every node in the NNS subnet and appending one `(node_operator_pid, ballot)` entry per node:

```rust
for node in nns_nodes {
    total_votes += 1;
    let node_operator_pid = get_node_operator_pid_of_node(...).await?;
    if node_operator_pid == caller {
        voted_on += 1;
        node_operator_ballots.push((node_operator_pid, RootProposalBallot::Yes));
    } else {
        node_operator_ballots.push((node_operator_pid, RootProposalBallot::Undecided));
    }
}
``` [1](#0-0) 

If a single node operator controls K nodes, they appear K times in `node_operator_ballots`. When `is_byzantine_majority_yes` is evaluated:

```rust
fn is_byzantine_majority_yes(&self) -> bool {
    let num_nodes = self.node_operator_ballots.len();
    let max_faults = (num_nodes - 1) / 3;
    let votes_yes: usize = self.node_operator_ballots.iter()
        .map(|(_, b)| match b { RootProposalBallot::Yes => 1, _ => 0 })
        .sum();
    votes_yes >= (num_nodes - max_faults)
}
``` [2](#0-1) 

`num_nodes` equals the total ballot entries (one per node), and `votes_yes` counts all entries marked `Yes`. A single operator controlling K nodes contributes K yes-votes. For a subnet of N nodes where one operator controls K nodes, the threshold is `N - floor((N-1)/3)`. If K ≥ that threshold, the proposer alone — by submitting the proposal — immediately satisfies `is_byzantine_majority_yes` without any other operator voting.

The analog to the external report is: just as endorsed votes in the Olympus report were not decremented when the underlying token balance changed (stale vote count vs. total supply mismatch), here the yes-vote count in the root proposal is inflated relative to the number of independent operators, because the denominator (`num_nodes`) counts physical nodes while the security model assumes independent operator principals. The proposer's K initial yes-votes are never "revoked" or re-weighted against the actual number of distinct operators.

### Impact Explanation

A single node operator who controls a byzantine-fault-threshold-crossing number of NNS subnet nodes can submit a root proposal and have it immediately pass — without any other operator voting — because the proposer's K initial yes-votes (one per node they control) already satisfy `votes_yes >= num_nodes - max_faults`. This allows that operator to unilaterally upgrade the NNS governance canister to arbitrary code, bypassing the intended multi-party approval requirement. The governance canister controls the entire NNS, so this is a critical governance authorization bypass.

### Likelihood Explanation

The NNS subnet currently has a fixed set of node operators. The likelihood depends on whether any single operator controls ≥ `N - floor((N-1)/3)` nodes. For a 28-node subnet, the threshold is 28 - 9 = 19 nodes. For a 7-node subnet (as used in tests), the threshold is 7 - 2 = 5 nodes. If the NNS subnet is small or if node operator concentration is high, this is reachable by a single operator without any collusion. The entry path is a direct ingress call to `submit_root_proposal_to_upgrade_governance_canister` on the root canister — no privileged key or admin access beyond being a registered NNS node operator is required.

### Recommendation

Change `is_byzantine_majority_yes` and `is_byzantine_majority_no` to count distinct operator principals rather than raw ballot entries. Deduplicate `node_operator_ballots` by principal when computing `num_nodes` and `votes_yes`/`votes_no`, or restructure the ballot as a `HashMap<PrincipalId, RootProposalBallot>` so each operator has exactly one vote regardless of how many nodes they control. The threshold formula should operate over the number of distinct operators, not the number of nodes.

### Proof of Concept

Consider a 7-node NNS subnet where one operator (the proposer) controls 5 nodes. The threshold is `7 - floor(6/3) = 7 - 2 = 5`. When the proposer calls `submit_root_proposal_to_upgrade_governance_canister`:

1. `node_operator_ballots` is populated with 5 entries `(proposer, Yes)` and 2 entries `(other_op, Undecided)`.
2. `is_byzantine_majority_yes` computes `num_nodes = 7`, `max_faults = 2`, `votes_yes = 5`, checks `5 >= 5` → **true**.
3. The proposal immediately executes `change_canister(payload)` upgrading the governance canister — with zero votes from any other operator. [3](#0-2) [4](#0-3) [5](#0-4)
