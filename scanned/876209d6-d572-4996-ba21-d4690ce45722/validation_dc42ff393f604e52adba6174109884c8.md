### Title
Hardcoded Block-Rate Assumption in `INGRESS_HISTORY_MAX_MESSAGES` Causes Premature Ingress Rejection if IC Round Rate Changes - (`rs/limits/src/lib.rs`)

### Summary
`INGRESS_HISTORY_MAX_MESSAGES` is computed at compile time by multiplying a hardcoded per-block message count by `MAX_INGRESS_TTL`, under the explicit assumption that the IC produces exactly **1 block per second**. If the IC consensus round rate is ever increased (e.g., via a protocol upgrade reducing `UNIT_DELAY_APP_SUBNET`), the constant becomes too small, and the `enqueue` gate in `ValidSetRuleImpl` will start rejecting legitimate ingress messages with `IngressHistoryFull` before the true TTL window is exhausted.

### Finding Description

`rs/limits/src/lib.rs` defines:

```rust
/// The value is the product of the default `max_ingress_messages_per_block`
/// configured in the subnet record; and the `MAX_INGRESS_TTL`
/// (assuming a block rate of 1 block per second). Times 2 …
pub const INGRESS_HISTORY_MAX_MESSAGES: usize =
    2 * 1000 * MAX_INGRESS_TTL.as_secs() as usize;
``` [1](#0-0) 

This evaluates to `2 × 1000 × 300 = 600 000`. The factor `1000` is `MAX_INGRESS_MESSAGES_PER_BLOCK` and the factor `300` is `MAX_INGRESS_TTL` in seconds, with the implicit assumption that one block is produced per second. [2](#0-1) 

The constant is wired directly into `ValidSetRuleImpl::new` without any registry override path:

```rust
Self {
    ingress_history_max_messages: INGRESS_HISTORY_MAX_MESSAGES,
    …
}
``` [3](#0-2) 

It is then used as a hard gate in `enqueue`:

```rust
if state.metadata.own_subnet_type != SubnetType::System
    && state.metadata.ingress_history.len() >= self.ingress_history_max_messages
{
    return Err(IngressInductionError::IngressHistoryFull { … });
}
``` [4](#0-3) 

The file header itself acknowledges that registry overrides are the intended mechanism for adjusting system limits, yet `ingress_history_max_messages` is never read from the registry: [5](#0-4) 

### Impact Explanation

If the IC consensus round rate increases above 1 block/second (e.g., `UNIT_DELAY_APP_SUBNET` is reduced in a future upgrade), the actual number of in-flight ingress messages within one `MAX_INGRESS_TTL` window grows proportionally, while `INGRESS_HISTORY_MAX_MESSAGES` stays fixed. Legitimate ingress messages submitted by unprivileged users are then rejected with `ErrorCode::IngressHistoryFull` even though the subnet has not truly exhausted its deduplication capacity. This is an availability impact: any user submitting update calls to an application subnet is affected.

The inverse is also true for the NNS subnet: `UNIT_DELAY_NNS_SUBNET = 3000 ms` means the actual block rate is ~0.33 blocks/second, so the limit is already ~3× larger than necessary, wasting memory.

### Likelihood Explanation

The IC protocol has evolved its consensus timing parameters before and may do so again. The constant is not registry-overridable, so a block-rate change cannot be compensated without a replica software upgrade. The likelihood is low in the short term but non-negligible over the lifetime of the protocol, matching the same class of risk as the Basin finding (future protocol parameter change invalidating a hardcoded assumption).

### Recommendation

Derive `ingress_history_max_messages` from the actual per-subnet `unit_delay` read from the registry at runtime, or expose it as a registry-configurable parameter (consistent with the stated intent of `rs/limits/src/lib.rs`). At minimum, document that this constant must be updated whenever `UNIT_DELAY_APP_SUBNET` changes.

### Proof of Concept

1. IC governance passes a proposal reducing `UNIT_DELAY_APP_SUBNET` from 1000 ms to 500 ms (2 blocks/second).
2. At 2 blocks/second × 1000 messages/block × 300 s TTL = 600 000 in-flight messages — exactly the current cap.
3. Any further increase in block rate, or any burst of legitimate traffic, causes `enqueue` to return `IngressHistoryFull` for valid messages, denying service to unprivileged callers.
4. The cap cannot be adjusted without a full replica software upgrade because `ingress_history_max_messages` is hardcoded in `ValidSetRuleImpl::new` and has no registry override path. [1](#0-0) [3](#0-2) [6](#0-5)
