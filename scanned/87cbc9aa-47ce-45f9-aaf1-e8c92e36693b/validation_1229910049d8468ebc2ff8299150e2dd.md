### Title
No "Withdraw Maximum" Option in ckBTC Minter While Minimum Withdrawal Amount Is Dynamically Adjusted - (File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs)

### Summary

The ckBTC minter's `retrieve_btc` and `retrieve_btc_with_approval` endpoints require callers to specify an exact satoshi amount with no "withdraw all" sentinel value. Simultaneously, the effective minimum withdrawal amount (`fee_based_retrieve_btc_min_amount`) is updated dynamically by a background timer based on current Bitcoin network fees. When a user's balance is close to the dynamic minimum, or when fees are volatile, the user cannot reliably determine the exact amount they can withdraw, causing repeated failed calls and wasted cycles.

### Finding Description

Both withdrawal endpoints accept only a fixed `amount: u64` argument:

```rust
pub struct RetrieveBtcArgs {
    pub amount: u64,
    pub address: String,
}
``` [1](#0-0) 

Inside `retrieve_btc`, the minter reads the current `fee_based_retrieve_btc_min_amount` and rejects the call if the user's amount is below it:

```rust
let (min_retrieve_amount, btc_network) =
    read_state(|s| (s.fee_based_retrieve_btc_min_amount, s.btc_network));

if args.amount < min_retrieve_amount {
    return Err(RetrieveBtcError::AmountTooLow(min_retrieve_amount));
}
``` [2](#0-1) 

The same check exists in `retrieve_btc_with_approval`: [3](#0-2) 

The `fee_based_retrieve_btc_min_amount` is not static. It is recomputed from the Bitcoin network's median fee percentiles every time the background timer fires `RefreshFeePercentiles`:

```rust
mutate_state(|s| {
    s.last_fee_per_vbyte = fees;
    s.last_median_fee_per_vbyte = Some(median_fee);
    s.fee_based_retrieve_btc_min_amount = fee_based_retrieve_btc_min_amount;
});
``` [4](#0-3) 

The formula for the dynamic minimum is:

```rust
fn fee_based_minimum_withdrawal_amount(&self, median_fee_rate: FeeRate) -> Satoshi {
    ((PER_REQUEST_RBF_BOUND
        + median_fee_rate.fee_ceil(PER_REQUEST_VSIZE_BOUND)
        + PER_REQUEST_MINTER_FEE_BOUND
        + self.check_fee)
        / 50_000)
        * 50_000
        + self.retrieve_btc_min_amount
}
``` [5](#0-4) 

The `get_minter_info` query exposes the current `fee_based_retrieve_btc_min_amount` as `retrieve_btc_min_amount`: [6](#0-5) 

However, between the moment a user queries `get_minter_info` (or `estimate_withdrawal_fee`) and the moment their `retrieve_btc` update call executes, the timer can fire and raise the minimum. There is no atomic "withdraw all available" path.

The `estimate_withdrawal_fee` query endpoint itself uses the stale in-memory `fee_based_retrieve_btc_min_amount` as the default amount when none is provided: [7](#0-6) 

### Impact Explanation

A user whose ckBTC balance is close to the dynamic minimum faces the following scenario:

1. User queries `get_minter_info` → sees `retrieve_btc_min_amount = 100_000` sats.
2. User's balance is 105_000 sats. They call `retrieve_btc(105_000)`.
3. Between steps 1 and 2, the Bitcoin fee spikes and the timer updates `fee_based_retrieve_btc_min_amount` to 150_000 sats.
4. The call fails with `AmountTooLow(150_000)`. The user's balance (105_000) is now below the new minimum — they cannot withdraw at all until fees drop.

Even when the user's balance exceeds the minimum, there is no way to specify "withdraw everything minus fees." The user must manually compute `balance - bitcoin_fee - minter_fee`, but both fees are dynamic and can change between the fee query and the withdrawal call. Repeated failed calls waste cycles on each ingress message.

During periods of high Bitcoin fee volatility (a realistic and recurring condition), users with balances near the minimum threshold can be temporarily locked out of withdrawing their ckBTC entirely.

### Likelihood Explanation

Bitcoin network fees are well-known to be highly volatile, with the median fee per vbyte varying by an order of magnitude within hours during congestion events. The `fee_based_retrieve_btc_min_amount` is updated on every timer tick, which fires regularly. Any user whose balance is within one fee-increment step (50,000 sats per the rounding formula) of the current minimum is at risk. This is a realistic condition for users who have deposited modest amounts of BTC. [8](#0-7) 

### Recommendation

**Short term:** Add a sentinel value (e.g., `amount = u64::MAX` or a new `RetrieveBtcArgs` variant) that instructs the minter to compute and use the maximum withdrawable amount given the caller's current balance and the current `fee_based_retrieve_btc_min_amount`. Alternatively, expose a query endpoint that returns the exact maximum amount a given account can currently withdraw.

**Long term:** Document the dynamic nature of `fee_based_retrieve_btc_min_amount` prominently in the API, and consider returning the current minimum in every `AmountTooLow` error response (which `RetrieveBtcError::AmountTooLow(u64)` already does) along with the user's current balance, so callers can self-correct in a single round-trip.

### Proof of Concept

1. Alice deposits 110,000 sats of BTC → receives ~110,000 ckBTC (minus check fee).
2. Alice queries `get_minter_info` → `retrieve_btc_min_amount = 100_000`.
3. Alice calls `retrieve_btc({ amount = 110_000, address = "..." })`.
4. Before the call executes, the timer fires and updates `fee_based_retrieve_btc_min_amount` to 150_000 (Bitcoin fees spiked).
5. Alice's call returns `Err(AmountTooLow(150_000))`.
6. Alice's balance (110,000) is now below the new minimum. She cannot withdraw any ckBTC until Bitcoin fees drop below the threshold that makes `fee_based_minimum_withdrawal_amount` ≤ 110,000.
7. Alice must wait and retry repeatedly, wasting cycles on each failed ingress call, with no way to know when the minimum will drop back below her balance. [9](#0-8) [10](#0-9)

pub async fn retrieve_btc<R: CanisterRuntime>(
    args: RetrieveBtcArgs,
    runtime: &R,
) -> Result<RetrieveBtcOk, RetrieveBtcError> {
    let caller = ic_cdk::api::msg_caller();

    state::read_state(|s| s.mode.is_withdrawal_available_for(&caller))
        .map_err(RetrieveBtcError::TemporarilyUnavailable)?;

    let _ecdsa_public_key = init_ecdsa_public_key().await;
    let main_address_str = state::read_state(|s| runtime.derive_minter_address_str(s));

    if args.address == main_address_str {
        ic_cdk::trap("illegal retrieve_btc target");
    }

    let _guard = retrieve_btc_guard(Account {
        owner: caller,
        subaccount: None,
    })?;
    let (min_retrieve_amount, btc_network) =
        read_state(|s| (s.fee_based_retrieve_btc_min_amount, s.btc_network));

    if args.amount < min_retrieve_amount {
        return Err(RetrieveBtcError::AmountTooLow(min_retrieve_amount));
    }

    let parsed_address = BitcoinAddress::parse(&args.address, btc_network)?;
    if read_state(|s| s.count_incomplete_retrieve_btc_requests() >= MAX_CONCURRENT_PENDING_REQUESTS)
    {
        return Err(RetrieveBtcError::TemporarilyUnavailable(
            "too many pending retrieve_btc requests".to_string(),
        ));
    }

    let balance = balance_of(caller).await?;
    if args.amount > balance {
        return Err(RetrieveBtcError::InsufficientFunds { balance });
    }

#[query]
fn estimate_withdrawal_fee(arg: EstimateFeeArg) -> WithdrawalFee {
    // This is a **query** endpoint, so mutating the state is not an issue
    // (even when called in replicated mode) since any change will be discarded.
    match mutate_state(|s| {
        let fee_estimator = IC_CANISTER_RUNTIME.fee_estimator(s);
        let withdrawal_amount = arg.amount.unwrap_or(s.fee_based_retrieve_btc_min_amount);
        ic_ckbtc_minter::estimate_retrieve_btc_fee(
            &mut s.available_utxos,
            withdrawal_amount,
            s.last_median_fee_per_vbyte
                .expect("Bitcoin current fee percentiles not retrieved yet."),
            s.max_num_inputs_in_transaction,
            &fee_estimator,
        )
    }) {
        Ok(fee) => fee,
        Err(BuildTxError::NotEnoughFunds) => {
            panic!("ERROR: withdrawal amount is too large for the minter")
        }
        Err(e @ BuildTxError::DustOutput { .. } | e @ BuildTxError::AmountTooLow) => panic!(
            "BUG: withdrawal amount is too low ({e:?}), but the withdrawal amount should be large enough to prevent this"
        ),
        Err(BuildTxError::InvalidTransaction(
            e @ InvalidTransactionError::TooManyInputs { .. },
        )) => panic!(
            "ERROR: the minter cannot currently process such a large withdrawal amount because it would require too many inputs ({e:?}), \
            resulting in the transaction being potentially non-standard"
        ),
    }
