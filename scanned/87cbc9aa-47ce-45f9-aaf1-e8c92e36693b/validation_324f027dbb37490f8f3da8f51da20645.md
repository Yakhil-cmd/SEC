### Title
Unbounded Per-Buyer Inter-Canister Calls in `sweep_icp` During SNS Swap Finalization - (File: rs/sns/swap/src/swap.rs)

### Summary
The `sweep_icp` function in the SNS Swap canister iterates over the entire `self.buyers` map and issues one inter-canister ICP ledger transfer call per buyer, with no batch size limit. Although the number of direct participants is bounded by `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`, this ceiling is large enough that a single `finalize_swap` call can exhaust the per-message instruction limit, causing the canister to trap and permanently stall finalization for that invocation. Because `finalize_swap` holds a re-entrancy lock while executing, a trap inside the async body releases the lock (via the post-upgrade hook), but the partial state left behind means repeated calls must re-do all already-skipped transfers, and each call still faces the same instruction-budget wall.

### Finding Description

`sweep_icp` iterates over every entry in `self.buyers` and `await`s a ledger `transfer_funds` call for each one:

```rust
for (principal_str, buyer_state) in self.buyers.iter_mut() {
    ...
    let result = icp_transferable_amount
        .transfer_helper(now_fn, DEFAULT_TRANSFER_FEE, Some(subaccount), &dst, icp_ledger)
        .await;
    ...
}
``` [1](#0-0) 

There is no batch-size cap on this loop. The number of buyers is bounded only by `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` (imported from `ic_nervous_system_common`), which is enforced in `refresh_buyer_token_e8s`:

```rust
if (num_direct_participants + 1) * num_sns_neurons_per_basket
    > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(...);
}
``` [2](#0-1) 

`MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` is a large constant designed to cap SNS Governance memory, not to cap the instruction budget of a single `sweep_icp` call. [3](#0-2) 

By contrast, `sweep_sns` iterates over `self.neuron_recipes` (also unbounded per call), and `claim_swap_neurons` explicitly batches its work into chunks of `CLAIM_SWAP_NEURONS_BATCH_SIZE`:

```rust
while !neuron_recipes.is_empty() {
    let current_batch_limit =
        std::cmp::min(CLAIM_SWAP_NEURONS_BATCH_SIZE, neuron_recipes.len());
    ...
}
``` [4](#0-3) 

`sweep_icp` and `sweep_sns` have no equivalent batching.

The minimum participation amount (`min_participant_icp_e8s`) is set per-SNS at launch time and can be as low as `MIN_PARTICIPANT_ICP_LOWER_BOUND_E8S`. A low minimum allows a large number of distinct principals to participate, each with a separate entry in `self.buyers`, maximising the iteration count. [5](#0-4) 

### Impact Explanation

**Impact: Medium** — When the `buyers` map is large (many distinct participants each at or near `min_participant_icp_e8s`), a single `finalize_swap` call will attempt to issue one ledger transfer per buyer in a single async loop. On the Internet Computer, each canister message has a fixed instruction budget. Exceeding it causes the message to trap. Because `sweep_icp` is called inside `finalize_inner`, which is called inside `finalize` while holding the `finalize_swap_in_progress` lock, a trap releases the lock (via upgrade) but leaves `transfer_start_timestamp_seconds` set for already-processed buyers. On the next call, those buyers are skipped (`AlreadyStarted`), but the remaining unprocessed buyers still form a loop that may again exceed the budget. Finalization can be permanently stalled until the SNS is upgraded to process buyers in smaller batches. ICP and SNS tokens held in the swap canister are locked until finalization completes.

### Likelihood Explanation

**Likelihood: Medium** — An SNS with a low `min_participant_icp_e8s` and a popular token can legitimately accumulate thousands of direct participants. No attacker action is required; normal participation at scale triggers the condition. An attacker who can participate with many principals at the minimum amount can deliberately maximise the buyer count to guarantee the DoS.

### Recommendation

Apply the same batching pattern used in `claim_swap_neurons` to both `sweep_icp` and `sweep_sns`. Process a fixed maximum number of buyers per `finalize_swap` invocation, persist progress via the existing `transfer_start_timestamp_seconds` / `transfer_success_timestamp_seconds` idempotency markers, and return a partial `SweepResult` so that callers can retry until all buyers are processed. The batch size should be chosen empirically to stay well within the per-message instruction budget.

### Proof of Concept

1. Deploy an SNS swap with `min_participant_icp_e8s` set to the minimum allowed value.
2. Have N distinct principals (where N is large, e.g. 10,000) each transfer exactly `min_participant_icp_e8s` ICP to their respective swap subaccounts and call `refresh_buyer_tokens`. Each call adds one entry to `self.buyers`.
3. Allow the swap to reach `Committed` or `Aborted` state.
4. Call `finalize_swap`. The canister enters `sweep_icp`, which loops over all N buyers and issues N sequential `await`-ed ledger calls. The accumulated instruction count exceeds the per-message budget; the message traps.
5. The `finalize_swap_in_progress` lock is released by the post-upgrade hook. Repeated calls to `finalize_swap` each process a prefix of the remaining buyers before trapping again, making forward progress only if the batch that fits within the budget is non-zero — but with a sufficiently large N, even the remaining unprocessed buyers may exceed the budget on every call, permanently stalling finalization.
