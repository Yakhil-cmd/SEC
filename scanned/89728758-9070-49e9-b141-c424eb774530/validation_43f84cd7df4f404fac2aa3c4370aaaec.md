### Title
ICRC-21 Consent Message for `retrieve_btc_with_approval` / `retrieve_doge_with_approval` Does Not Display the Caller's Account - (`File: rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs`, `rs/dogecoin/ckdoge/minter/src/updates/icrc21.rs`)

### Summary

The ICRC-21 consent messages generated by the ckBTC and ckDOGE minters for `retrieve_btc_with_approval` / `retrieve_doge_with_approval` do not include the caller's principal (the account that will authorize the ckBTC/ckDOGE burn). A malicious dapp can present a consent message that appears to be for one account while actually executing the call from a different account, and the user has no way to detect the discrepancy from the consent dialog alone.

### Finding Description

The `build_consent_info` function in the ckBTC minter's ICRC-21 implementation accepts only `consent_msg_request` and `network` — it never receives or uses the caller's principal:

```rust
pub(super) fn build_consent_info(
    consent_msg_request: ConsentMessageRequest,
    network: Network,
) -> Result<ConsentInfo, Icrc21Error> {
``` [1](#0-0) 

The downstream `build_retrieve_btc_with_approval_message` function constructs the consent message showing only the amount, the Bitcoin destination address, and optionally the source subaccount bytes — but never the caller's IC principal:

```rust
let mut message = format!(
    "# Convert {ckbtc} to {btc}\n\n\
     Authorize the {ckbtc} minter to burn {ckbtc} from your account and \
     send the equivalent amount in {btc} (minus network and minter fees) to \
     the Bitcoin address below.\n\n\
     **Amount to convert:** `{amount} {ckbtc}`\n\n\
     **Bitcoin destination address:**\n`{address}`",
    address = args.address,
);
``` [2](#0-1) 

The same pattern is present in the ckDOGE minter: [3](#0-2) [4](#0-3) 

By contrast, the ICRC ledger implementations (ICP ledger, ICRC-1 ledger) correctly receive `caller_principal` and include it in the consent message under a `From:` field, suppressing it only when the caller is anonymous: [5](#0-4) [6](#0-5) 

The actual `retrieve_btc_with_approval` execution path does correctly capture the caller: [7](#0-6) 

So the consent message and the execution path are decoupled: the consent message is caller-agnostic, but the actual burn is caller-specific.

### Impact Explanation

A malicious dapp can:
1. Call `icrc21_canister_call_consent_message` on the ckBTC/ckDOGE minter to obtain a consent message for a small amount from a low-value account.
2. Present this consent message to a victim user who approves it.
3. Then call `retrieve_btc_with_approval` from a *different* account (e.g., the victim's high-value account, if the dapp has obtained an ICRC-2 approval from that account).

Because the consent message never shows *which* IC principal is authorizing the burn, the user cannot detect the substitution. The user believes they are authorizing a burn from account A, but the actual on-chain call burns from account B. This leads to unauthorized ckBTC/ckDOGE burns and loss of user funds.

### Likelihood Explanation

The ICRC-21 standard is specifically designed to let hardware wallets and signing devices show users exactly what they are authorizing before they sign. The ckBTC and ckDOGE minters advertise ICRC-21 support via `icrc10_supported_standards`. Any dapp integrating with these minters via ICRC-21 will encounter this gap. The attack requires only that the victim has previously granted an ICRC-2 allowance to a dapp-controlled principal, which is a normal usage pattern for `retrieve_btc_with_approval`. No privileged access is required.

### Recommendation

Pass the caller's principal into `build_consent_info` (and `build_retrieve_btc_with_approval_message` / `build_retrieve_doge_with_approval_message`) and include it as a `From account:` field in both `GenericDisplay` and `FieldsDisplay` outputs, mirroring the pattern used in the ICRC ledger implementations:

```rust
pub(super) fn build_consent_info(
    consent_msg_request: ConsentMessageRequest,
    network: Network,
    caller: Principal,          // <-- add
) -> Result<ConsentInfo, Icrc21Error> { ... }
```

The `icrc21_canister_call_consent_message` entry point already has access to the caller via `ic_cdk::api::msg_caller()` (as done in the ICRC-1 ledger) and should forward it. Suppress the field only when the caller is `Principal::anonymous()`, consistent with the ledger behaviour.

### Proof of Concept

1. Alice holds 1 BTC worth of ckBTC in her main account `alice_main`.
2. Alice previously called `icrc2_approve` on the ckBTC ledger, granting `malicious_dapp` an allowance to spend from `alice_main`.
3. `malicious_dapp` calls `icrc21_canister_call_consent_message` with `retrieve_btc_with_approval` args specifying `amount = 1_000` satoshis (a small amount) and `from_subaccount = None`.
4. The returned consent message reads:

   ```
   # Convert ckBTC to BTC
   Authorize the ckBTC minter to burn ckBTC from your account and send the
   equivalent amount in BTC (minus network and minter fees) to the Bitcoin
   address below.
   **Amount to convert:** `0.00001 ckBTC`
   **Bitcoin destination address:** `bc1q...attacker`
   ```

   No `From account:` field is present.
5. `malicious_dapp` shows this consent message to Alice. Alice approves, believing she is authorizing a trivial 1,000-satoshi burn.
6. `malicious_dapp` then calls `retrieve_btc_with_approval` from `alice_main` with `amount = 100_000_000` (1 BTC). The minter uses `msg_caller()` = `alice_main` and the pre-existing ICRC-2 allowance to burn 1 BTC.
7. Alice's entire ckBTC balance is drained. The consent message she approved gave her no indication of which account was being used. [8](#0-7) [9](#0-8) [10](#0-9) [11](#0-10)
