### Title
No Slippage Protection in SNS Treasury Manager Deposit Validation Allows Front-Running for LP Token Loss - (`rs/sns/governance/src/extensions.rs`)

### Summary

The `validate_deposit_operation_impl` function in the SNS governance extension system validates deposit amounts only against individual treasury balance caps (50% per token), but never checks the ratio of SNS-to-ICP tokens against the current DEX pool ratio. Because SNS governance proposals have a non-trivial delay between approval and execution, an unprivileged DEX user can front-run the deposit by manipulating the pool ratio, causing the SNS treasury to receive significantly fewer LP tokens than expected.

### Finding Description

The `validate_deposit_operation_impl` function in `rs/sns/governance/src/extensions.rs` performs two independent checks on a deposit proposal:

1. `sns_requested > sns_balance / 2` → reject
2. `icp_requested > icp_balance / 2` → reject [1](#0-0) 

Neither check involves the current pool ratio on the external DEX. The amounts `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s` are fixed at proposal-creation time and are forwarded verbatim to the treasury manager's `deposit` call at execution time. [2](#0-1) 

The `treasury_manager.did` interface itself explicitly acknowledges this gap as a "Known Security Risk":

> *Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved.* [3](#0-2) 

The same warning is surfaced in the proposal rendering for `RegisterExtension`: [4](#0-3) 

### Impact Explanation

When the SNS treasury deposits SNS and ICP tokens to a DEX liquidity pool with an asymmetric ratio (relative to the pool's current reserves), the AMM mints fewer LP tokens than it would for a balanced deposit. The SNS DAO permanently loses LP position value proportional to the imbalance. A front-runner who manipulates the pool ratio before execution can then arbitrage the pool back to equilibrium, extracting profit at the SNS treasury's expense. The note "any undeposited tokens are automatically returned" only applies to excess tokens that the DEX refuses entirely; it does not compensate for the LP token shortfall caused by asymmetric pricing.

### Likelihood Explanation

SNS governance proposals require a voting period (typically days). Any DEX participant can observe a pending `ExecuteExtensionOperation` deposit proposal on-chain, calculate the expected deposit ratio, and trade to skew the pool ratio before the proposal executes. This requires no privileged access — only the ability to trade on the DEX. The attack is more profitable when the deposit is large relative to pool liquidity, which is precisely the scenario the SNS treasury manager is designed for.

### Recommendation

1. **Enforce a minimum LP token output**: The deposit call to the treasury manager should include a `min_lp_tokens_out` parameter derived from the pool ratio at proposal-creation time. If the actual LP tokens minted fall below this threshold, the deposit should revert.
2. **Re-validate pool ratio at execution time**: Before approving the ICRC-2 allowance and calling `deposit`, `execute_treasury_manager_deposit` should query the DEX for the current pool ratio and reject the deposit if it deviates beyond a governance-configured slippage tolerance.
3. **Split deposit into smaller tranches**: Reduce the per-deposit size to limit the profitability of front-running.

### Proof of Concept

1. SNS governance creates a deposit proposal: `treasury_allocation_sns_e8s = 1_000_000_000`, `treasury_allocation_icp_e8s = 10_000_000_000` (matching the current 1:10 pool ratio).
2. Proposal passes the `validate_deposit_operation_impl` check: both amounts are ≤ 50% of treasury balances. [5](#0-4) 
3. Before execution, an attacker swaps a large amount of SNS into the pool, shifting the ratio to 1:5.
4. `execute_treasury_manager_deposit` is called. It approves the allowance and calls `deposit` with the original amounts unchanged. [6](#0-5) 
5. The DEX receives 1_000_000_000 SNS and 10_000_000_000 ICP at a 1:5 pool ratio. The ICP side is 2× the balanced amount; the AMM mints LP tokens only for the balanced portion, wasting the excess ICP (or returning it, but the LP position is still smaller than expected).
6. The attacker swaps ICP back for SNS, arbitraging the pool to equilibrium and extracting profit from the price impact the SNS treasury absorbed.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
