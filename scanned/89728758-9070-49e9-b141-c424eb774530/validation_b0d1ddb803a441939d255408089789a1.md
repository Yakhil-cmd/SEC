### Title
`estimate_withdrawal_fee` Query Endpoint Permanently Removes UTXOs from Available Set When Called in Replicated Mode — (File: `rs/bitcoin/ckbtc/minter/src/main.rs`, `rs/dogecoin/ckdoge/minter/src/main.rs`)

---

### Summary

The `estimate_withdrawal_fee` endpoint in both the ckBTC and ckDOGE minters is declared as `#[query]` but internally calls `mutate_state`, which invokes `utxos_selection` → `greedy`. The `greedy` function **permanently removes UTXOs from `available_utxos`** in the success path and never re-inserts them. A misleading inline comment claims state changes are discarded even in replicated mode — this is incorrect. When called as an ingress update call (replicated execution), state changes are committed, causing UTXOs to be silently drained from the minter's available UTXO pool.

---

### Finding Description

**Root cause — `estimate_withdrawal_fee` calls `mutate_state` and removes UTXOs:**

In `rs/bitcoin/ckbtc/minter/src/main.rs`:

```rust
#[query]
fn estimate_withdrawal_fee(arg: EstimateFeeArg) -> WithdrawalFee {
    // This is a **query** endpoint, so mutating the state is not an issue
    // (even when called in replicated mode) since any change will be discarded.
    match mutate_state(|s| {
        ...
        ic_ckbtc_minter::estimate_retrieve_btc_fee(
            &mut s.available_utxos,   // ← mutable borrow
            ...
        )
    }) { ... }
}
``` [1](#0-0) 

The same pattern exists in `rs/dogecoin/ckdoge/minter/src/main.rs`: [2](#0-1) 

**The call chain actually mutates `available_utxos`:**

`estimate_retrieve_btc_fee` → `queries::estimate_withdrawal_fee` → `utxos_selection` → `greedy`:

```rust
fn greedy(target: u64, available_utxos: &mut UtxoSet) -> Vec<Utxo> {
    ...
    Some(utxo) => {
        let utxo = available_utxos.remove(&utxo).expect("BUG: missing UTXO");
        // UTXOs are removed here and NEVER re-inserted in the success path
        solution.push(utxo);
    }
    ...
}
``` [3](#0-2) 

`utxos_selection` also calls `available_utxos.pop_first()` when the UTXO count exceeds `UTXOS_COUNT_THRESHOLD`: [4](#0-3) 

In `queries::estimate_withdrawal_fee`, the selected UTXOs are passed by reference to `build_unsigned_transaction_from_inputs` and then **dropped** — they are never re-inserted into `available_utxos`: [5](#0-4) 

Compare this to `build_unsigned_transaction`, which explicitly undoes the mutation on error: [6](#0-5) 

`estimate_withdrawal_fee` has no such undo logic.

**The comment is factually wrong about replicated mode:**

The same file contains `http_request`, which explicitly guards against replicated execution:

```rust
#[query(hidden = true)]
fn http_request(req: HttpRequest) -> HttpResponse {
    if ic_cdk::api::in_replicated_execution() {
        ic_cdk::trap("update call rejected");
    }
``` [7](#0-6) 

This guard exists precisely because state changes **are** committed when a `#[query]` method is called via an ingress update message (replicated execution). `estimate_withdrawal_fee` has no such guard.

---

### Impact Explanation

When `estimate_withdrawal_fee` is called as an ingress update call (replicated execution), `mutate_state` commits the removal of UTXOs from `available_utxos` to the canister's replicated state. Each call with a satisfiable withdrawal amount permanently removes a batch of UTXOs from the pool. An attacker can:

1. Repeatedly call `estimate_withdrawal_fee` as an update call with varying amounts.
2. Drain `available_utxos` over time.
3. Cause all subsequent legitimate `retrieve_btc` / `retrieve_doge_with_approval` calls to fail with `NotEnoughFunds`, halting withdrawals for all users.

This is a **chain-fusion mint/burn/replay bug** class: the minter's UTXO accounting is corrupted without any corresponding Bitcoin transaction, breaking the invariant that `available_utxos` only shrinks when a real withdrawal is submitted.

---

### Likelihood Explanation

- `estimate_withdrawal_fee` is a public, permissionless endpoint callable by any principal.
- No authentication, rate-limiting, or `in_replicated_execution()` guard is present.
- Calling a `#[query]` method as an update (ingress) call is a standard IC operation available to any boundary node user.
- The misleading comment actively discourages future developers from adding the necessary guard.

---

### Recommendation

1. Add an `in_replicated_execution()` guard identical to `http_request`:
   ```rust
   #[query]
   fn estimate_withdrawal_fee(arg: EstimateFeeArg) -> WithdrawalFee {
       if ic_cdk::api::in_replicated_execution() {
           ic_cdk::trap("update call rejected");
       }
       ...
   }
   ```
2. Alternatively, refactor `queries::estimate_withdrawal_fee` to take `&UtxoSet` (immutable) and use a cloned or read-only simulation path, eliminating the need for `mutate_state` entirely.
3. Remove the incorrect comment claiming state changes are discarded in replicated mode.
4. Apply the same fix to `rs/dogecoin/ckdoge/minter/src/main.rs`.

fn greedy(target: u64, available_utxos: &mut UtxoSet) -> Vec<Utxo> {
    #[cfg(feature = "canbench-rs")]
    let _scope = canbench_rs::bench_scope("greedy");

    let mut solution = vec![];
    let mut goal = target;
    while goal > 0 {
        let candidate_utxo = available_utxos
            .find_lower_bound(goal)
            .or_else(|| available_utxos.last())
            .cloned();
        match candidate_utxo {
            Some(utxo) => {
                let utxo = available_utxos.remove(&utxo).expect("BUG: missing UTXO");
                goal = goal.saturating_sub(utxo.value);
                solution.push(utxo);
            }
            None => {
                // Not enough available UTXOs to satisfy the request.
                for u in solution {
                    available_utxos.insert(u);
                }
                return vec![];
            }
        }
    }

    debug_assert!(solution.is_empty() || solution.iter().map(|u| u.value).sum::<u64>() >= target);

    solution
}

pub fn estimate_withdrawal_fee<F: FeeEstimator>(
    available_utxos: &mut UtxoSet,
    withdrawal_amount: u64,
    median_fee_millisatoshi_per_vbyte: FeeRate,
    minter_address: BitcoinAddress,
    recipient_address: BitcoinAddress,
    max_num_inputs_in_transaction: usize,
    fee_estimator: &F,
) -> Result<WithdrawalFee, BuildTxError> {
    // We simulate the algorithm that selects UTXOs for the
    // specified amount.
    let selected_utxos = utxos_selection(withdrawal_amount, available_utxos, 1);

    build_unsigned_transaction_from_inputs(
        &selected_utxos,
        vec![(recipient_address, withdrawal_amount)],
        &minter_address,
        max_num_inputs_in_transaction,
        median_fee_millisatoshi_per_vbyte,
        fee_estimator,
    )
    .map(|(unsigned_tx, _change_output, fee)| {
        assert_eq!(
            unsigned_tx.outputs.len(),
            2,
            "BUG: expected 1 output to the recipient and one change output to the minter, \
                so that the totality of the fee is paid in full by the recipient"
        );
        fee
    })
}
