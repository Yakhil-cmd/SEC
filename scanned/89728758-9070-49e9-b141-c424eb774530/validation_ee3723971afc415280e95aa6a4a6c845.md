### Title
Stale ICP/XDR Exchange Rate Used Without Freshness Check in Cycles Minting - (`rs/nns/cmc/src/main.rs`)

### Summary

The Cycles Minting Canister (CMC) converts ICP to cycles using a cached `icp_xdr_conversion_rate` that is never validated for freshness at the point of use. Any user calling `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` will receive cycles computed from a potentially arbitrarily stale rate, directly analogous to the Chainlink `latestRoundData()` staleness issue in H-01.

### Finding Description

`tokens_to_cycles` in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` and extracts only `xdr_permyriad_per_icp`, completely ignoring the accompanying `timestamp_seconds` field: [1](#0-0) 

The rate struct carries a `timestamp_seconds` field (visible in the Candid interface and state definition), but no staleness guard is applied before the rate is used for financial computation: [2](#0-1) 

The CMC refreshes the rate via heartbeat every `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes): [3](#0-2) 

However, the heartbeat only fires if `exchange_rate_canister_id` is set: [4](#0-3) 

And `exchange_rate_canister_id` is `Option<CanisterId>` — it can be absent: [5](#0-4) 

When absent, the CMC falls back to the hardcoded default rate initialized at genesis (May 10, 2021, 1 ICP = 100 XDR): [6](#0-5) 

The `validate_exchange_rate` function called during rate ingestion checks only source counts, not the rate's own timestamp: [7](#0-6) 

`do_set_icp_xdr_conversion_rate` only enforces monotonically increasing timestamps (new > old), not that the new rate is recent relative to wall-clock time: [8](#0-7) 

### Impact Explanation

All three public cycle-minting flows call `tokens_to_cycles` unconditionally:

- `process_create_canister` → `tokens_to_cycles` → wrong cycles minted for canister creation [9](#0-8) 
- `process_top_up` → `tokens_to_cycles` → wrong cycles deposited to target canister [10](#0-9) 
- `process_mint_cycles` → `tokens_to_cycles` → wrong cycles minted to cycles ledger account [11](#0-10) 

If the XRC canister is unavailable for an extended period (subnet issues, canister upgrade, etc.), the CMC silently continues using the last cached rate. If `exchange_rate_canister_id` is unset, the 2021 default rate (1 ICP = 100 XDR) is used indefinitely. Given ICP's price history, this default rate would cause massive over-issuance of cycles (ICP was worth far less than 100 XDR for most of its history after 2021), or under-issuance depending on direction. Users burning ICP receive an incorrect number of cycles, breaking the economic invariant that 1 XDR = 1 trillion cycles.

### Likelihood Explanation

The XRC canister is a system canister on the NNS subnet. Any transient NNS subnet degradation, XRC canister upgrade, or misconfiguration (missing `exchange_rate_canister_id`) causes the CMC to silently use a stale rate. The entry path requires no special privilege: any principal can call `notify_top_up` or `notify_create_canister` after making an ICP ledger transfer. The default rate scenario is reachable on any deployment where the XRC canister ID has not been configured.

### Recommendation

Add a maximum-age check in `tokens_to_cycles` before using the cached rate. Reject the conversion (returning an error to the caller) if `now - icp_xdr_conversion_rate.timestamp_seconds` exceeds a configurable threshold (e.g., 30 minutes):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref()
            .ok_or_else(|| NotifyError::Other { ... })?;

        let age = now_seconds().saturating_sub(rate.timestamp_seconds);
        if age > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: format!("ICP/XDR rate is stale ({age}s old)"),
            });
        }

        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

Additionally, the `validate_exchange_rate` function should optionally accept a `now` parameter and reject rates whose `timestamp` is too far in the past.

### Proof of Concept

1. Deploy a CMC without setting `exchange_rate_canister_id` (or simulate XRC unavailability by disabling the heartbeat path).
2. The CMC retains `icp_xdr_conversion_rate` with `timestamp_seconds = 1_620_633_600` (May 2021) and `xdr_permyriad_per_icp = 1_000_000` (100 XDR/ICP).
3. Call `notify_top_up` with 1 ICP. `tokens_to_cycles` computes cycles using the 2021 rate without any staleness check.
4. The caller receives cycles computed at 100 XDR/ICP regardless of the actual current market rate, with no error or warning.

The `tokens_to_cycles` function reads only `xdr_permyriad_per_icp` and never inspects `timestamp_seconds`: [12](#0-11)
