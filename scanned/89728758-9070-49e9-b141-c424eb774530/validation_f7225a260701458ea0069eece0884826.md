### Title
Integer Division Truncation in `compute_maturity_modulation_permyriad` Silently Zeroes Small Price-Change Signals — (`rs/nns/governance/src/timer_tasks/update_icp_xdr_rate_related_data.rs`)

---

### Summary

`compute_maturity_modulation_permyriad` computes the NNS maturity modulation using integer arithmetic. The formula `sensitivity * (recent - reference) / reference` truncates to zero whenever the price deviation is smaller than `reference / sensitivity`. Because `sensitivity = 2_500` and typical ICP prices are in the tens-of-thousands of permyriad, any 7-day vs. 365-day price divergence smaller than ~0.4% is silently discarded. The modulation stays at its previous value (or zero on first run) instead of tracking the true signal, causing every NNS neuron owner who disburses or spawns maturity during that period to receive a slightly wrong ICP amount.

---

### Finding Description

`compute_maturity_modulation_permyriad` computes:

```rust
let target_modulation = {
    let recent = recent_icp_price as i128;
    let reference = reference_icp_price as i128;
    let sensitivity = MATURITY_MODULATION_SENSITIVITY_PERMYRIAD as i128; // = 2_500
    sensitivity * (recent - reference) / reference   // ← integer division
};
``` [1](#0-0) 

`MATURITY_MODULATION_SENSITIVITY_PERMYRIAD` is `2_500` and `reference_icp_price` is a `u64` in permyriad units (e.g., 10 XDR/ICP → 100,000 permyriad). [2](#0-1) 

The division is Rust integer division (floor toward zero). The result is zero whenever:

```
|sensitivity * (recent - reference)| < reference
⟺  |recent - reference| < reference / sensitivity
                        = reference / 2_500
```

**Concrete example** — ICP at 10 XDR/ICP (`reference = 100_000`):

| Price deviation | Expected modulation | Computed modulation |
|---|---|---|
| +0.39% (+390 permyriad) | +9.75 permyriad | **0** (truncated) |
| +0.04% (+40 permyriad) | +1.0 permyriad | **0** (truncated) |
| +0.01% (+10 permyriad) | +0.25 permyriad | **0** (truncated) |

Any day where the 7-day average is within 0.4% of the 365-day average produces `target_modulation = 0`, regardless of the true signal. The speed-limit clamp then keeps the stored modulation unchanged, so the error accumulates silently across many days. [3](#0-2) 

The stored modulation is later applied to every maturity disbursement and neuron spawn via `apply_maturity_modulation`:

```rust
let modulated_amount_e8s: u128 = amount_e8s
    .checked_mul(adjusted_maturity_modulation_basis_points)
    ...
    .checked_div(BASIS_POINTS_PER_UNITY)
``` [4](#0-3) 

---

### Impact Explanation

Every NNS neuron owner who calls `spawn_neuron` or `disburse_maturity` while the ICP price is within ~0.4% of its 365-day average receives an ICP amount computed from a stale or zero modulation instead of the correct one. The error per disbursement is bounded by the maximum modulation range (±1,000 permyriad = ±10%), but the systematic bias — always rounding the signal toward zero — means the modulation chronically under-reacts to small, sustained price trends. Over weeks of small-but-consistent price movement, the stored modulation can lag the true value by many permyriad, silently mis-pricing every maturity conversion during that window.

---

### Likelihood Explanation

ICP price changes of less than 0.4% relative to the 365-day moving average are extremely common — the 365-day average is a slow-moving anchor, and the 7-day average frequently hovers within a fraction of a percent of it during low-volatility periods. This is not an edge case; it is the normal operating regime for extended stretches of time. The code path runs automatically every day via the recurring timer task, so no attacker action is required — the precision loss occurs passively on every daily tick where the price deviation is below the threshold.

---

### Recommendation

Multiply the numerator by a fixed scale factor before dividing, then carry that scale through to the consumer, or switch to a rational/fixed-point intermediate:

```rust
// Option A: scale up before dividing, keep result in sub-permyriad units
const SCALE: i128 = 10_000;
let target_modulation_scaled = sensitivity * (recent - reference) * SCALE / reference;
// use target_modulation_scaled / SCALE when applying the clamp and speed limit,
// and store the full-precision value until the final conversion.

// Option B: use Decimal (already used elsewhere in the codebase)
use rust_decimal::Decimal;
let target_modulation = (Decimal::from(sensitivity)
    * (Decimal::from(recent) - Decimal::from(reference))
    / Decimal::from(reference))
    .round()
    .to_i64()
    .unwrap_or(0);
```

The `Decimal` type is already used in SNS governance reward calculations for exactly this reason. [5](#0-4)

### Proof of Concept

```
reference_icp_price = 100_000   (10 XDR/ICP, a realistic mainnet value)
recent_icp_price    = 100_039   (0.039% above reference — well within normal daily noise)

sensitivity = 2_500

numerator   = 2_500 * (100_039 - 100_000) = 2_500 * 39 = 97_500
denominator = 100_000

target_modulation = 97_500 / 100_000 = 0   ← Rust integer division

Expected (floating-point): 97_500 / 100_000 = 0.975 permyriad ≈ 1 permyriad

Result: the modulation signal for this day is completely discarded.
If this condition persists for 30 consecutive days, the modulation
remains at 0 instead of gradually climbing toward ~30 permyriad,
meaning every maturity disbursement during that month is computed
with a 0.3% error in the ICP conversion rate.
```
