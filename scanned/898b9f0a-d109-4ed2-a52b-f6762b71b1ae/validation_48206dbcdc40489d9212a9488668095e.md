### Title
Missing Concurrency Lock in `perform_mint_sns_tokens` Allows Concurrent Execution and Future Spending-Limit Bypass - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS Governance canister's `perform_mint_sns_tokens` function makes an inter-canister call to the ledger without acquiring any concurrency lock. Its sibling function `perform_transfer_sns_treasury_funds` explicitly acquires such a lock before its ledger call. This asymmetry means multiple `MintSnsTokens` proposals can execute concurrently across `.await` suspension points, and when the 7-day spending limit for `MintSnsTokens` is enforced (currently disabled via a `TODO`), the limit can be bypassed.

---

### Finding Description

`perform_mint_sns_tokens` in `rs/sns/governance/src/governance.rs` performs a minting inter-canister call with no guard:

```rust
async fn perform_mint_sns_tokens(
    &mut self,
    mint: MintSnsTokens,
) -> Result<(), GovernanceError> {
    // ... build `to` and `amount_e8s` ...
    self.ledger
        .transfer_funds(amount_e8s, 0, None, to, mint.memo())
        .await?;   // <-- suspension point, no lock held
    Ok(())
}
``` [1](#0-0) 

By contrast, `perform_transfer_sns_treasury_funds` acquires a `thread_local` lock before its ledger call:

```rust
thread_local! {
    static IN_PROGRESS_PROPOSAL_ID: RefCell<Option<u64>> = const { RefCell::new(None) };
}
let release_on_drop = acquire(&IN_PROGRESS_PROPOSAL_ID, proposal_id);
if let Err(already_in_progress_proposal_id) = release_on_drop {
    return Err(...);
}
``` [2](#0-1) 

Both functions are spawned in the background by `start_proposal_execution` via `spawn_in_canister_env`: [3](#0-2) 

On the IC, a canister suspends at every `.await` and processes other queued messages before resuming. Two independently adopted `MintSnsTokens` proposals can therefore interleave: proposal P1 suspends at `transfer_funds(...).await`, the scheduler runs P2's execution, P2 also suspends at `transfer_funds(...).await`, and both eventually complete — neither having seen the other as "executed" (`executed_timestamp_seconds == 0` for both during the overlap).

The 7-day spending limit for `MintSnsTokens` is currently disabled (returns `Decimal::MAX`): [4](#0-3) 

The enforced version is commented out with `TODO(NNS1-2982): Uncomment`: [5](#0-4) 

The limit enforcement reads `executed_timestamp_seconds` from past proposals to compute the rolling 7-day total: [6](#0-5) 

Because `executed_timestamp_seconds` is only written by `set_proposal_execution_status` *after* the ledger call returns, two concurrently executing proposals will each see the other's amount as zero when computing the rolling total, allowing the combined mint to exceed the limit. [7](#0-6) 

---

### Impact Explanation

**Governance authorization bug / ledger conservation bug.** When the `MintSnsTokens` 7-day spending cap is activated, two (or more) concurrently executing `MintSnsTokens` proposals can each independently pass the cap check and both complete their ledger mints, collectively exceeding the intended cap. This undermines the economic safety invariant that limits how many SNS tokens can be minted within a rolling 7-day window, potentially allowing unbounded token inflation within a single window.

Even before the cap is enforced, the structural absence of the lock is inconsistent with the explicit protection applied to the analogous `TransferSnsTreasuryFunds` path, and represents a latent governance authorization bypass.

---

### Likelihood Explanation

**Moderate.** No malicious governance majority is required. Any two legitimately adopted `MintSnsTokens` proposals whose execution futures are both in-flight simultaneously (which is the normal operating mode when `run_periodic_tasks` processes multiple adopted proposals in one heartbeat) will interleave at the ledger `.await`. The likelihood increases once the spending cap is enforced, at which point an SNS with a high proposal throughput or a proposer who deliberately times two proposals to be adopted close together can reliably trigger the bypass.

---

### Recommendation

Add the same `thread_local` concurrency lock pattern used in `perform_transfer_sns_treasury_funds` to `perform_mint_sns_tokens`, keyed on `proposal_id`. Additionally, add an execution-time amount check (analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`) so that the rolling 7-day total is re-validated while the lock is held, immediately before the ledger call.

---

### Proof of Concept

1. SNS governance has two adopted `MintSnsTokens` proposals P1 and P2, each for an amount equal to 90% of the 7-day cap.
2. `run_periodic_tasks` calls `process_proposal(P1)` → `start_proposal_execution(P1)` → spawns `perform_action(P1)` in background.
3. `run_periodic_tasks` calls `process_proposal(P2)` → `start_proposal_execution(P2)` → spawns `perform_action(P2)` in background.
4. P1's future runs: no lock acquired, calls `self.ledger.transfer_funds(...).await` — suspends.
5. P2's future runs: no lock acquired, checks rolling 7-day total — P1's `executed_timestamp_seconds` is still `0` so P2 sees total = 0, passes the cap check, calls `self.ledger.transfer_funds(...).await` — suspends.
6. P1's ledger response arrives: `set_proposal_execution_status` sets `executed_timestamp_seconds` for P1.
7. P2's ledger response arrives: `set_proposal_execution_status` sets `executed_timestamp_seconds` for P2.
8. Both mints succeed; combined minted amount = 180% of the 7-day cap, violating the limit.
