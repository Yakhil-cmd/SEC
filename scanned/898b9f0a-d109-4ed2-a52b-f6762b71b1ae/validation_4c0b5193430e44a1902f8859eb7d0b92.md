Looking at the M-8 vulnerability class — **cycles/resource accounting bug where fees are tracked but never actually paid out** — I need to find an IC analog where cycles owed to a party are recorded but have no mechanism for actual disbursement.

My investigation focused on the canister HTTP outcall pay-as-you-go (`PayAsYouGo`) refund mechanism, which is the closest structural analog to the `lossFee` pattern.

---

### Title
Cycles Permanently Lost Due to Integer Division Truncation in HTTP Outcall Pay-As-You-Go `per_replica_allowance` Calculation - (`rs/execution_environment/src/execution_environment.rs`)

### Summary

When a canister makes an HTTP outcall under `PricingVersion::PayAsYouGo`, the execution environment computes `per_replica_allowance = refundable_cycles / node_count` using integer division. The remainder `refundable_cycles % node_count` is recorded in `refundable_cycles` as "available to be refunded" but can never be returned to the canister — there is no code path that disburses this remainder.

### Finding Description

In `rs/execution_environment/src/execution_environment.rs`, when processing a canister HTTP request under pay-as-you-go pricing:

1. The entire payment is taken upfront: `canister_http_request_context.request.payment.take()`.
2. `refundable_cycles = payment - base_fee` is computed — this is the amount the canister is owed back.
3. `per_replica_allowance = refundable_cycles / node_count` is computed using **integer division**, truncating the remainder.
4. These values are stored in `RefundStatus`. [1](#0-0) 

The `RefundStatus` struct documents `refundable_cycles` as "the amount of cycles that are available to be refunded": [2](#0-1) 

The maximum total refund that can ever be returned to the canister is `per_replica_allowance * node_count = (refundable_cycles / node_count) * node_count`. When `refundable_cycles % node_count != 0`, the remainder is permanently unrefundable. No code path in the payload builder, pool manager, or execution environment returns this remainder: [3](#0-2) 

The payload validation only enforces `refund <= per_replica_allowance` per replica — it never accounts for the truncated remainder at the aggregate level. [4](#0-3) 

### Impact Explanation

For every HTTP outcall with `PayAsYouGo` pricing where `refundable_cycles % node_count != 0`, the canister permanently loses `refundable_cycles % node_count` cycles. The maximum loss per request is `node_count - 1` cycles (e.g., up to 12 cycles on a 13-node subnet). While small per request, this accumulates across all HTTP outcalls on the subnet and is a systematic, deterministic loss with no recovery mechanism — directly analogous to M-8's `lossFee` being added to `commonData` with no withdrawal path.

### Likelihood Explanation

Any canister invoking `http_request` or `FlexibleHttpRequest` with `PayAsYouGo` pricing on a subnet where `refundable_cycles % node_count != 0` is affected. This is triggered by any unprivileged canister caller and requires no special privileges. The condition `refundable_cycles % node_count != 0` holds for the vast majority of real payments. [5](#0-4) 

### Recommendation

Before storing `RefundStatus`, adjust `refundable_cycles` downward to `per_replica_allowance * node_count` and immediately credit the remainder back to the canister's balance. Alternatively, add a final settlement step when the request completes (response, divergence, or timeout) that returns `refundable_cycles - refunded_cycles` to the caller.

### Proof of Concept

- `payment = 1_000_000_014 cycles`
- `base_fee = 1_000_000_000 cycles`
- `refundable_cycles = 14 cycles`
- `node_count = 13` (typical application subnet)
- `per_replica_allowance = 14 / 13 = 1 cycle` (integer division)
- Maximum total refund = `1 * 13 = 13 cycles`
- **Permanently lost = `14 - 13 = 1 cycle`** — tracked in `refundable_cycles` but no code path can return it [6](#0-5) [7](#0-6)

/// Enforces the per-replica refund allowance from the request context: the
/// `refund` claimed in the payment receipt must never exceed the
/// `per_replica_allowance` derived from the request's context.
pub(crate) fn check_refund_allowance(
    receipt: &CanisterHttpPaymentReceipt,
    per_replica_allowance: Cycles,
) -> Result<(), InvalidCanisterHttpPayloadReason> {
    if receipt.refund > per_replica_allowance {
        return Err(InvalidCanisterHttpPayloadReason::RefundExceedsAllowance {
            refund: receipt.refund,
            per_replica_allowance,
        });
    }
    Ok(())
}
