### Title
Unchecked Subtraction in `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` Permanently Blocks Governance-Approved Treasury Transfers - (File: rs/sns/governance/src/proposal.rs)

### Summary

In `rs/sns/governance/src/proposal.rs`, the function `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` performs a plain (unchecked) subtraction `allowance_tokens - spent_tokens` at line 2631. The `allowance_tokens` is derived from the treasury valuation **stored at proposal submission time**, while `spent_tokens` is the **current** 7-day total of executed transfers. When token price volatility causes the treasury to be in a lower-value regime at submission time than it was when prior proposals were executed, `spent_tokens` can exceed `allowance_tokens`. Because `rust_decimal::Decimal` is a signed type, the subtraction silently produces a negative `remainder_tokens`, causing every subsequent `transfer_amount_tokens > remainder_tokens` check to be trivially true, permanently blocking execution of the governance-approved proposal for the entire 7-day window.

### Finding Description

The execution-time guard in `perform_transfer_sns_treasury_funds` calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, which:

1. Computes `allowance_tokens` from the `Valuation` stored in `ActionAuxiliary::TransferSnsTreasuryFunds` at proposal submission time.
2. Computes `spent_tokens` as the live 7-day total of all executed `TransferSnsTreasuryFunds` proposals.
3. Performs **plain subtraction** at line 2631:

```rust
let remainder_tokens = allowance_tokens - spent_tokens;
```

The submission-time path (line 801) correctly uses `checked_sub`:

```rust
let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| { ... })?;
```

The execution-time path does not. Since `rust_decimal::Decimal` is signed, when `spent_tokens > allowance_tokens`, `remainder_tokens` becomes a negative `Decimal`. The subsequent check:

```rust
if transfer_amount_tokens > remainder_tokens { return Err(...PreconditionFailed...); }
```

is always true for any positive `transfer_amount_tokens`, so the proposal always fails with `PreconditionFailed`.

**Concrete trigger scenario:**

|---|---|---|---|---|---|
| 1 | 1,000 | 2,000 | 2,000,000 | Large | Proposal B submitted: 140 tokens (limit = 150) |

### Impact Explanation

A governance-approved `TransferSnsTreasuryFunds` proposal is permanently blocked from executing for the remainder of the 7-day window. The proposal is marked `Failed` with `ErrorType::PreconditionFailed`. The SNS treasury transfer mechanism is effectively DoS'd for any proposal submitted during a lower-value regime that overlaps with a higher-value spending window. This constitutes a **governance authorization bug**: a legitimately voted-on proposal cannot be executed due to incorrect arithmetic, causing loss of governance liveness for treasury operations.

### Likelihood Explanation

SNS token prices are volatile. The scenario requires only that: (a) a `TransferSnsTreasuryFunds` proposal is submitted when the treasury is in a lower-value regime (medium: 25% fraction), and (b) other proposals were executed in the same 7-day window when the treasury was in a higher-value regime (large: fixed 300,000 XDR cap). Both conditions are realistic given normal market conditions. No privileged access is required; any SNS governance participant can submit a proposal via the standard ingress path.

### Recommendation

Replace the plain subtraction at line 2631 with a checked subtraction, mirroring the submission-time check at line 801:

```rust
// Before (line 2631):
let remainder_tokens = allowance_tokens - spent_tokens;

// After:
let remainder_tokens = allowance_tokens.checked_sub(spent_tokens).ok_or_else(|| {
    GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!(
            "The 7-day spending total ({spent_tokens} tokens) already exceeds the \
             allowance ({allowance_tokens} tokens) computed from the valuation at \
             proposal submission time. Try again after the 7-day window resets.",
        ),
    )
})?;
```
