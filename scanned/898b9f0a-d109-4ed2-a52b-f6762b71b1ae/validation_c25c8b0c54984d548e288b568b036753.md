### Title
Missing Slippage Protection in SNS Treasury Manager Deposit Interface — (`File: rs/sns/treasury_manager/treasury_manager.did`)

### Summary
The SNS Treasury Manager `deposit` interface and the SNS Governance deposit validation logic lack any slippage protection parameter. The `DepositRequest` type carries no `min_amount_out` or minimum LP-token field, and the governance-level `validate_deposit_operation` only checks that deposit amounts do not exceed 50% of treasury balance — it never enforces a minimum output. Any DEX participant can front-run a pending deposit proposal execution, manipulate the pool price, and cause the SNS treasury to receive far fewer LP tokens than expected.

### Finding Description

The `DepositRequest` type in the Treasury Manager interface contains only `allowances` (the assets to deposit) and no slippage bound: [1](#0-0) 

The interface itself explicitly acknowledges this gap in its "Known Security Risks" section: [2](#0-1) 

The SNS Governance deposit validation (`validate_deposit_operation`) enforces only that the requested amounts are valid `Nat` values and do not exceed 50% of the current treasury balance: [3](#0-2) [4](#0-3) 

No post-execution check on LP tokens received, no minimum output ratio, and no slippage tolerance is enforced at any layer — neither in the `DepositRequest` type, nor in the governance proposal validation, nor in the `deposit` API contract itself: [5](#0-4) 

### Impact Explanation

An SNS governance proposal to deposit treasury assets into a DEX liquidity pool is approved at a specific price ratio. Between proposal approval and on-chain execution, any unprivileged DEX participant can:

1. Observe the pending deposit call to the Treasury Manager canister.
2. Front-run it by swapping a large amount into the pool, moving the price adversely.
3. Allow the Treasury Manager's `deposit` to execute — receiving far fewer LP tokens than the ratio at proposal time.
4. Immediately reverse the front-run trade, extracting the price difference as profit.

The SNS treasury permanently loses value proportional to the price impact, with no on-chain mechanism to detect or revert the loss. The economic loss is bounded only by the size of the deposit and the depth of the DEX pool.

### Likelihood Explanation

The attack is straightforward for any participant who can observe canister update calls (which are public on the IC). SNS governance proposals are publicly visible on-chain before execution. The execution delay between proposal adoption and the actual `deposit` call gives ample time for a front-runner to act. No privileged access is required — any principal with sufficient liquidity on the target DEX can execute this attack.

### Recommendation

Add a `min_amount_out` (or equivalent minimum LP token) field to `DepositRequest`:

```candid
type DepositRequest = record {
  allowances : vec Allowance;
  min_lp_tokens_out : opt nat;   // slippage guard
};
```

The SNS Governance `validate_deposit_operation` should require this field to be set and non-zero when the deposit targets an external DEX custodian. The Treasury Manager implementation must revert (return `Err` with `Postcondition` kind) if the LP tokens actually received are below `min_lp_tokens_out`.

### Proof of Concept

1. SNS governance adopts a proposal: deposit 1 000 000 SNS tokens + 2 000 000 ICP into a DEX pool at the current 1:2 ratio.
2. Attacker observes the pending `deposit` call to the Treasury Manager canister.
3. Attacker swaps a large amount of ICP into the pool, shifting the ratio to 1:3 (SNS becomes cheaper).
4. Treasury Manager's `deposit` executes — it deposits at the 1:3 ratio, receiving ~33% fewer LP tokens than expected. No check prevents this; `DepositRequest` carries no `min_lp_tokens_out`.
5. Attacker reverses the swap, profiting from the spread. The SNS treasury has permanently lost ~33% of the deposit value in LP token terms. [1](#0-0) [6](#0-5)
