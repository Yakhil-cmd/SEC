### Title
SNS Governance `ManagePrincipals` Holder Can Evade Permission Revocation by Pre-Granting to Sybil Accounts - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

In SNS governance, any principal holding the `ManagePrincipals` permission on a neuron can grant `ManagePrincipals` to an arbitrary number of additional principals — including sybil accounts they control — before their own permission is revoked. Because there is no restriction preventing a `ManagePrincipals` holder from propagating that same permission to others, revocation of a malicious `ManagePrincipals` holder is ineffective: the attacker retains full neuron control through the newly empowered sybil accounts.

---

### Finding Description

`add_neuron_permissions` in `rs/sns/governance/src/governance.rs` enforces two authorization checks before granting permissions to a target principal:

1. The caller must hold `ManagePrincipals` (or `ManageVotingPermission` for voting-only grants).
2. The permissions to add must be a subset of `neuron_grantable_permissions`. [1](#0-0) 

The authorization gate in `check_principal_authorized_to_change_permissions` only checks whether the caller holds one of the sufficient permissions: [2](#0-1) 

The grantability gate in `check_permissions_are_grantable` only checks against the system-wide `neuron_grantable_permissions` parameter: [3](#0-2) 

Neither check prevents a `ManagePrincipals` holder from:
- Granting `ManagePrincipals` to additional principals they control.
- Doing so preemptively before their own `ManagePrincipals` is revoked.

`ManagePrincipals` is included in `PERMISSIONS_RELATED_TO_VOTING`? No — it is not. `PERMISSIONS_RELATED_TO_VOTING` is: [4](#0-3) 

`ManagePrincipals` is therefore only grantable by a caller who already holds `ManagePrincipals`. But there is no cap on how many times a `ManagePrincipals` holder can exercise this right before their own permission is removed. The only limit is `max_number_of_principals_per_neuron`: [5](#0-4) 

The default value of `max_number_of_principals_per_neuron` is large enough (typically 5–10) to allow a malicious holder to add multiple sybil accounts before revocation completes.

---

### Impact Explanation

`ManagePrincipals` is the highest-privilege neuron permission. A holder can:
- Add or remove any principal's permissions (`AddNeuronPermissions` / `RemoveNeuronPermissions`).
- Disburse the neuron's staked tokens (`Disburse`).
- Split, merge maturity, and configure dissolve state.

By pre-granting `ManagePrincipals` to sybil accounts before their own permission is revoked, a malicious holder retains full, permanent control over the neuron — including its staked SNS tokens and voting power. The legitimate neuron owner cannot effectively revoke access without also discovering and revoking every sybil account, which may not be possible if the attacker acts faster than the victim.

**Impact: High** — permanent unauthorized control over neuron assets and governance voting power.

---

### Likelihood Explanation

**Likelihood: Low** — the attacker must already legitimately hold `ManagePrincipals` on the target neuron (e.g., as a co-manager in a team-controlled neuron or DAO). The attack requires anticipating that revocation is imminent and acting before the `RemoveNeuronPermissions` transaction is processed. On the Internet Computer, all update calls are processed sequentially within a subnet round, so the attacker must submit their `AddNeuronPermissions` call in a prior round. This is realistic in adversarial multi-party neuron management scenarios.

---

### Recommendation

Restrict the ability to grant `ManagePrincipals` to only the neuron's original claimer, or enforce a **"cannot grant permissions you do not hold"** invariant: a caller may only grant permissions that are a strict subset of their own current permissions on the neuron. This prevents a `ManagePrincipals` holder from propagating that permission to sybil accounts before revocation.

Alternatively, remove `ManagePrincipals` from `neuron_grantable_permissions` by default, requiring an explicit governance proposal to enable it.

---

### Proof of Concept

**Setup**: Neuron N has two `ManagePrincipals` holders: legitimate owner B and co-manager A (attacker).

**Step 1**: B decides to revoke A's `ManagePrincipals` and submits:
```
manage_neuron(RemoveNeuronPermissions {
    subaccount: N,
    principal_id: A,
    permissions_to_remove: [ManagePrincipals]
})
```

**Step 2**: Before B's transaction is processed (or in a prior round), A submits:
```
manage_neuron(AddNeuronPermissions {
    subaccount: N,
    principal_id: C,  // sybil account controlled by A
    permissions_to_add: [ManagePrincipals]
})
```

This call succeeds because:
- A holds `ManagePrincipals` → `check_principal_authorized_to_change_permissions` passes. [6](#0-5) 
- `ManagePrincipals` is in `neuron_grantable_permissions` → `check_permissions_are_grantable` passes. [7](#0-6) 
- Principal C is new, and the principal count is below `max_number_of_principals_per_neuron`. [8](#0-7) 

**Step 3**: B's `RemoveNeuronPermissions` executes, removing A's `ManagePrincipals`.

**Result**: A no longer has `ManagePrincipals` directly, but C — controlled by A — now holds `ManagePrincipals` on neuron N. A retains full control: they can disburse staked tokens, vote, submit proposals, and add further sybil accounts through C. The revocation is completely ineffective.

    pub(crate) fn check_principal_authorized_to_change_permissions(
        &self,
        caller: &PrincipalId,
        permissions_to_change: NeuronPermissionList,
    ) -> Result<(), GovernanceError> {
        // If the permissions to change are exclusively voting related,
        // ManagePrincipals or ManageVotingPermission is sufficient.
        // Otherwise, only ManagePrincipals is sufficient.
        let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
            vec![
                NeuronPermissionType::ManagePrincipals,
                NeuronPermissionType::ManageVotingPermission,
            ]
        } else {
            vec![NeuronPermissionType::ManagePrincipals]
        };

        // The caller is authorized if they have any of the sufficient permissions
        let caller_authorized = sufficient_permissions
            .iter()
            .any(|sufficient_permission| self.is_authorized(caller, *sufficient_permission));

    pub fn check_permissions_are_grantable(
        &self,
        neuron_permission_list: &NeuronPermissionList,
    ) -> Result<(), GovernanceError> {
        let mut illegal_permissions = HashSet::new();

        let grantable_permissions: HashSet<&i32> = self
            .neuron_grantable_permissions
            .as_ref()
            .expect("NervousSystemParameters.neuron_grantable_permissions must be present")
            .permissions
            .iter()
            .collect();

        for permission in &neuron_permission_list.permissions {
            if !grantable_permissions.contains(&permission) {
                illegal_permissions.insert(NeuronPermissionType::try_from(*permission).ok());
            }
        }

        if !illegal_permissions.is_empty() {
            return Err(GovernanceError::new_with_message(
                ErrorType::AccessControlList,
                format!(
                    "Cannot grant permissions as one or more permissions is not \
                    allowed to be granted. Illegal Permissions: {illegal_permissions:?}"
                ),
            ));
        }

        Ok(())
    }
