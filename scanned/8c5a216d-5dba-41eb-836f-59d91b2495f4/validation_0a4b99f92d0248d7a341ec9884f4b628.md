### Title
Missing Cycles Attachment in `create_canister_and_install_code` Causes Zero-Cycle Frozen Canister - (File: `rs/nns/handlers/root/impl/src/canister_management.rs`)

### Summary

The `create_canister_and_install_code` function in the NNS Root canister calls the management canister's `create_canister` method without attaching any cycles. The IC management canister's `create_canister` requires cycles to be forwarded in order to fund the new canister. Without cycles, the newly created canister starts with zero cycles, is immediately frozen, and cannot process any messages. This is the direct IC analog of the Wormhole report: a function that calls a fee-requiring external function without forwarding the required payment.

### Finding Description

In `rs/nns/handlers/root/impl/src/canister_management.rs`, the `create_canister_and_install_code` function (lines 269–334) constructs a `create_canister` call using `Call::bounded_wait` but never calls `.with_cycles(...)`:

```rust
// Step 1.1: Send create_canister call (to the subnet, not "aaaaa-aa", and attach cycles).
let create_canister_result = Call::bounded_wait(callee, "create_canister")
    .with_arg(create_canister_args)
    .await;                          // ← no .with_cycles(amount) here
```

The comment on line 287 explicitly says "attach cycles," but the implementation omits the `.with_cycles(payment)` call entirely. [1](#0-0) 

Contrast this with the older `try_to_create_and_install_canister` function in the same file, which correctly uses `call_with_payment` and passes `request.initial_cycles`:

```rust
let (id,): (CanisterIdRecord,) = call_with_payment(
    CanisterId::ic_00().get().0,
    "create_canister",
    (create_args,),
    request.initial_cycles,   // ← cycles properly forwarded
)
``` [2](#0-1) 

Furthermore, the `CreateCanisterAndInstallCodeRequest` type (defined in the DID interface) has no `initial_cycles` field at all, so there is no mechanism to supply cycles even if the code tried to forward them:

```
type CreateCanisterAndInstallCodeRequest = record {
  host_subnet_id : principal;
  canister_settings : opt CanisterSettings;
  wasm_module : blob;
  install_arg : blob;
};
``` [3](#0-2) 

The `create_canister_and_install_code` update endpoint on the Root canister is only callable by NNS Governance: [4](#0-3) 

The NNS Governance canister dispatches this call via the `CreateCanisterAndInstallCode` proposal type, which calls Root with no cycles attached: [5](#0-4) 

### Impact Explanation

Any NNS governance proposal of type `CreateCanisterAndInstallCode` that passes will result in a canister being created with exactly 0 cycles. On the IC, a canister with 0 cycles is immediately below its freezing threshold and cannot accept or process any ingress or inter-canister messages. The `install_code` step that follows may succeed (controllers can install code regardless of cycle balance), but the resulting canister is permanently non-functional until manually topped up. The entire `CreateCanisterAndInstallCode` governance proposal type is therefore broken: it silently produces a frozen, unusable canister instead of a working one. This is a **cycles/resource accounting bug** with a concrete governance-level impact.

### Likelihood Explanation

The bug is triggered by any legitimate NNS governance proposal of type `CreateCanisterAndInstallCode` that achieves a voting majority and executes. No malicious actor is required; any neuron submitting such a proposal in good faith will trigger the defect. The proposal type is documented and exposed in the Root canister's public DID interface, making it a reachable code path.

### Recommendation

1. Add an `initial_cycles: opt nat` field to `CreateCanisterAndInstallCodeRequest` in both the Candid interface (`root.did`) and the corresponding Rust struct.
2. In `create_canister_and_install_code`, forward those cycles to the `create_canister` call using `.with_cycles(initial_cycles)`, mirroring the pattern already used in `try_to_create_and_install_canister` with `call_with_payment(..., request.initial_cycles)`.
3. In the NNS Governance canister's `CreateCanisterAndInstallCode` proposal handler, attach the specified cycles when calling Root's `create_canister_and_install_code`.

### Proof of Concept

1. Submit an NNS governance proposal of type `CreateCanisterAndInstallCode` targeting any application subnet, with a valid `wasm_module` and `install_arg`.
2. The proposal passes and NNS Governance calls `Root::create_canister_and_install_code`.
3. Root calls `create_canister` on the target subnet with **zero cycles attached** (no `.with_cycles()` in the `Call::bounded_wait` chain).
4. The management canister creates the new canister with a cycle balance of 0.
5. The new canister is immediately frozen (cycle balance < freezing threshold).
6. `install_code` may succeed (controller privilege), but the installed canister cannot respond to any messages.
7. Observe: `canister_status` on the new canister reports `status = frozen` and `cycles = 0`. [6](#0-5)

pub async fn create_canister_and_install_code(
    request: CreateCanisterAndInstallCodeRequest,
) -> CreateCanisterAndInstallCodeResponse {
    let CreateCanisterAndInstallCodeRequest {
        host_subnet_id,
        canister_settings,
        wasm_module,
        install_arg,
    } = request;

    // We call host_subnet_id directly (rather than the Management (virtual)
    // canister) so that the canister is created on that specific subnet
    // (not the one where this canister (i.e. Root) lives).
    let callee = host_subnet_id.0;

    let main = async {
        // Step 1: Create canister.

        // Step 1.1: Send create_canister call (to the subnet, not "aaaaa-aa", and attach cycles).
        let create_canister_args = CreateCanisterArgs {
            settings: canister_settings.map(management_canister::CanisterSettingsArgs::from),
            sender_canister_version: Some(ic_cdk::api::canister_version()),
        };
        let create_canister_result = Call::bounded_wait(callee, "create_canister")
            .with_arg(create_canister_args)
            .await;

        // Step 1.2: Handle the create_canister result. This consists of decoding
        // the reply, and handling errors.
        let response = create_canister_result.map_err(|err| {
            convert_call_failed_to_create_canister_and_install_code_error("create_canister", err)
        })?;
        let create_canister_reply: CanisterIdRecord = response.candid().map_err(|err| {
            // Show raw bytes to aid debugging when the response can't be decoded.
            let response = humanize_blob(response.as_ref(), 200);
            CreateCanisterAndInstallCodeError {
                code: None,
                description: format!(
                    "Failed to decode create_canister response: {err}. Raw reply (hex): {response}"
                ),
            }
        })?;
        // This will be returned (assuming install_code goes well).
        let new_canister_id = create_canister_reply.get_canister_id();

        // Step 2: Install code into the new canister.
        let install_code_args = InstallCodeArgs {
            mode: Install,
            canister_id: new_canister_id.get(),
            wasm_module,
            arg: install_arg,
            sender_canister_version: None,
        };
        Call::bounded_wait(callee, "install_code")
            .with_arg(install_code_args)
            .await
            .map_err(|err| {
                convert_call_failed_to_create_canister_and_install_code_error("install_code", err)
            })?;

        Ok(new_canister_id)
    };

    let result = main.await;
    CreateCanisterAndInstallCodeResponse::from(result)
}
