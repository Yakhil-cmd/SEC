### Title
SNS Treasury Manager `DepositRequest` Lacks Minimum LP Token Output, Exposing SNS Treasury to Timing-Based Price Manipulation - (File: rs/sns/governance/src/extensions.rs)

### Summary
The SNS Treasury Manager framework's `execute_treasury_manager_deposit` function and the `DepositRequest` type contain no minimum LP token output (slippage) parameter. When an SNS governance proposal to deposit treasury assets into a DEX liquidity pool is approved and scheduled for execution, an attacker with knowledge of the predictable execution time can manipulate the DEX pool ratio beforehand, causing the SNS treasury to receive fewer LP tokens than expected. The codebase itself acknowledges this as a "Known Security Risk" but provides no on-chain enforcement.

### Finding Description

The `DepositRequest` type in the Treasury Manager interface accepts only `allowances` (amounts to deposit) with no minimum output constraint: [1](#0-0) 

The `ValidatedDepositOperationArg` struct, which is the validated form used during governance proposal execution, only carries `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s` — no minimum LP tokens out: [2](#0-1) 

The `execute_treasury_manager_deposit` function approves the treasury manager and calls `deposit` without any post-call check on the minimum LP tokens received: [3](#0-2) 

The `validate_deposit_operation_impl` function only validates that the requested amounts do not exceed 50% of the current treasury balance. It performs no check on minimum output: [4](#0-3) 

The codebase explicitly acknowledges this gap as a "Known Security Risk" in the interface definition: [5](#0-4) 

The proposal rendering for `RegisterExtension` also warns about this but provides no enforcement: [6](#0-5) 

### Impact Explanation

An SNS DAO's treasury assets (SNS tokens + ICP) deposited into a DEX liquidity pool can be manipulated so the treasury receives significantly fewer LP tokens than the fair-market ratio at the time the proposal was voted on. The attacker profits by withdrawing from the pool at the favorable ratio after the governance deposit executes. This is a direct, quantifiable loss of SNS treasury funds — the same class of impact as the original H-05 report.

### Likelihood Explanation

IC governance proposals have a publicly known, predictable execution time (after the voting period ends). Any principal with sufficient capital can interact with a DEX canister on the IC to manipulate the pool ratio in the rounds immediately before the governance proposal executes. Unlike Ethereum, there is no mempool to observe, but the attack window is actually *larger* on the IC: the attacker knows the exact block at which the proposal will execute and can prepare the manipulation in advance. No privileged access is required — only the ability to call the DEX canister as an ordinary user.

### Recommendation

1. Add a `min_lp_tokens_out : opt nat` field to `DepositRequest` in `rs/sns/treasury_manager/treasury_manager.did` so that the SNS governance proposal can encode a voter-approved minimum output.
2. Add a corresponding `minimum_lp_tokens_out_e8s` field to `ValidatedDepositOperationArg` in `rs/sns/governance/src/extensions.rs`.
3. In `execute_treasury_manager_deposit`, after calling `deposit`, verify that the returned `external_custodian` balance in `Balances` meets the minimum LP token threshold specified in the proposal. If not, revert or log a governance error.
4. Include the minimum output value in the proposal rendering so SNS voters can review and approve the slippage tolerance before the proposal passes.

### Proof of Concept

1. An SNS DAO submits a governance proposal to deposit 1,000,000 SNS tokens and 500 ICP into a DEX liquidity pool. The proposal passes voting and is scheduled to execute at a known future block.
2. An attacker observes the scheduled execution time (public information on the IC dashboard).
3. Before the proposal executes, the attacker calls the DEX canister directly to swap a large amount of ICP for SNS tokens, skewing the pool ratio heavily in favor of ICP.
4. The governance proposal executes: `execute_treasury_manager_deposit` calls `approve_treasury_manager` then `deposit` on the treasury manager canister, which in turn deposits into the DEX at the now-skewed ratio. The SNS treasury receives far fewer LP tokens than expected.
5. The attacker immediately calls the DEX canister to swap back, profiting from the price impact. The SNS treasury has permanently received fewer LP tokens for the same deposited assets.

No minimum output check exists anywhere in the call chain — `DepositRequest` carries no `min_lp_tokens_out`, `ValidatedDepositOperationArg` carries no such field, and `execute_treasury_manager_deposit` performs no post-deposit balance assertion against a minimum threshold. [7](#0-6) [8](#0-7)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
