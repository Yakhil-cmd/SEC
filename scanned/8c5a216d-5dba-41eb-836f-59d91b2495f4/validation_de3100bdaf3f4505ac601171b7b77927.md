### Title
ckDOGE Minter Never Schedules `ConsolidateUtxos` Task, Enabling UTXO Fragmentation DoS on Large Withdrawals — (`rs/dogecoin/ckdoge/minter/src/main.rs`)

### Summary
ckDOGE's `setup_tasks()` omits `schedule_now(TaskType::ConsolidateUtxos, ...)` that ckBTC includes. Because the task queue is a thread-local that starts empty and is only populated via explicit `schedule_now`/`schedule_after` calls, `consolidate_utxos()` is never invoked at runtime. The `utxo_consolidation_threshold` init parameter is therefore dead configuration. An unprivileged depositor can accumulate many small UTXOs in the minter's `available_utxos` set, eventually causing any withdrawal that requires more than `max_num_inputs_in_transaction` (500) inputs to fail with `BuildTxError::InvalidTransaction(TooManyInputs)`.

### Finding Description
`rs/dogecoin/ckdoge/minter/src/main.rs` `setup_tasks()` (lines 46–49) schedules only `ProcessLogic` and `RefreshFeePercentiles`. The ckBTC equivalent (lines 54–58) additionally calls `schedule_now(TaskType::ConsolidateUtxos, ...)`, which bootstraps the hourly self-rescheduling consolidation loop. Without that initial `schedule_now`, `TaskType::ConsolidateUtxos` is never inserted into the `TASKS` thread-local queue, so `consolidate_utxos()` in `rs/bitcoin/ckbtc/minter/src/lib.rs` (lines 1431–1548) is never called. The full consolidation infrastructure (the `ConsolidateUtxosRequest` type, `CreatedConsolidateUtxosRequest` event, `utxo_consolidation_threshold` state field, `select_utxos_to_consolidate`) is present but unreachable.

Every `update_balance` call for a new deposit inserts a UTXO into `available_utxos`. With no consolidation, this set grows monotonically. When `retrieve_doge_with_approval` is called, `build_unsigned_transaction` enforces `max_num_inputs_in_transaction = 500`. If covering the requested withdrawal amount requires more than 500 inputs (possible when the UTXO set is dominated by small-value entries), the transaction build fails.

### Impact Explanation
Large withdrawals (ckDOGE → DOGE) fail with `TooManyInputs` when the minter's UTXO set is sufficiently fragmented. Users whose withdrawal requests fail are reimbursed (the minter has reimbursement logic), so funds are not permanently lost. However, the minter becomes unable to service large withdrawals until a canister upgrade adds the missing `schedule_now(TaskType::ConsolidateUtxos, ...)` call. The stated invariant — that the minter can always process any withdrawal above `retrieve_doge_min_amount` — is violated. The impact is a targeted DoS on large withdrawals, not permanent fund loss.

### Likelihood Explanation
The attacker cost is non-trivial: each deposit creates one UTXO, and the minimum deposit is ~50 DOGE (~$7 at staging deployment time). Creating enough fragmentation to block large withdrawals (e.g., 10,000 UTXOs) costs ~500,000 DOGE (~$70,000). However, organic usage alone (many small legitimate deposits) will naturally fragment the UTXO set over time without any attacker, exactly as happened with ckBTC (60k UTXOs in production). The omission is therefore a latent operational vulnerability that activates under normal growth.

### Recommendation
Add `schedule_now(TaskType::ConsolidateUtxos, &DOGECOIN_CANISTER_RUNTIME);` to ckDOGE's `setup_tasks()` function, mirroring the ckBTC implementation. This will bootstrap the hourly consolidation loop and make the already-configured `utxo_consolidation_threshold` parameter effective.

### Proof of Concept
A state-machine test can:
1. Initialize the ckDOGE minter.
2. Deposit 501+ UTXOs each worth exactly `deposit_doge_min_amount` via `update_balance`.
3. Assert `available_utxos.len() > max_num_inputs_in_transaction` and that no `ConsolidateUtxos` task is present in the task queue.
4. Call `retrieve_doge_with_approval` for an amount requiring all UTXOs.
5. Assert the result is `Err(BuildTxError::InvalidTransaction(TooManyInputs { .. }))`.
