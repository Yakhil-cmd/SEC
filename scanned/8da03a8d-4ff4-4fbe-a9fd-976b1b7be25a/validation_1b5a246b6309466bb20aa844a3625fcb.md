### Title
ckERC20 Minter Mints Event-Declared Amount Instead of Actual Received Amount for Rebasing ERC20 Tokens, Breaking 1:1 Backing Invariant — (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

The ckETH/ckERC20 helper contracts emit the caller-supplied `amount` parameter in the deposit event rather than the actual balance difference received by the minter address. For rebasing ERC20 tokens (e.g., stETH), `safeTransferFrom` succeeds but the minter address receives `amount - 1` or `amount - 2` tokens due to share-rounding. The IC minter reads the event value and mints exactly that declared amount of ckERC20, creating unbacked tokens and breaking the 1:1 conservation invariant.

---

### Finding Description

**Root cause — Solidity helper contract (`DepositHelperWithSubaccount.sol`):**

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);  // actual received may be amount-2

    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        amount,          // ← emits the requested amount, NOT the actual received amount
        principal,
        subaccount
    );
}
``` [1](#0-0) 

The same pattern exists in the older helper: [2](#0-1) 

**Root cause — IC minter log parser (`parser.rs`):**

The minter reads `value_bytes` directly from the event data and stores it as `Erc20Value`:

```rust
let [value_bytes] = parse_hex_into_32_byte_words(entry.data, event_source)?;
// ...
value: Erc20Value::from_be_bytes(value_bytes),  // ← trusts the event-declared amount
``` [3](#0-2) 

**Root cause — IC minter deposit handler (`deposit.rs`):**

The minter then mints exactly `event.value()` ckERC20 tokens to the beneficiary:

```rust
.transfer(TransferArg {
    // ...
    amount: event.value(),   // ← mints the event-declared amount, not actual received
})
``` [4](#0-3) 

---

### Impact Explanation

**Vulnerability class: chain-fusion mint/burn/replay bug (ledger conservation break)**

For a rebasing ERC20 token (e.g., stETH, which uses share-based accounting and rounds down on `transferFrom`):

1. User calls `depositErc20(stETH, 1_000_000_000_000_000_000, principal, subaccount)`.
2. `safeTransferFrom` succeeds but minter address receives `999_999_999_999_999_998` (2 wei short due to share rounding).
3. Event emits `amount = 1_000_000_000_000_000_000`.
4. IC minter reads the event and mints `1_000_000_000_000_000_000` ckstETH.
5. **Discrepancy**: 2 wei of ckstETH are unbacked.

After N deposits, the minter holds `N × (amount - δ)` stETH but has minted `N × amount` ckstETH. When users withdraw, the minter constructs an Ethereum ERC20 `transfer(destination, withdrawal_amount)` call. Once the accumulated shortfall exceeds the minter's stETH balance, the Ethereum transaction reverts (insufficient balance), permanently blocking withdrawals for that token.

Additionally, the ckERC20 total supply exceeds the actual ERC20 backing, violating the 1:1 peg guarantee that the entire chain-fusion system is built on.

---

### Likelihood Explanation

**Medium.** The vulnerability is latent: it activates only when a rebasing ERC20 token is added as a supported ckERC20 token via governance. The current supported tokens (USDC, USDT, ckBTC-wrapped tokens) are not rebasing. However:

- Any governance proposal to add stETH or another rebasing token would immediately expose this path.
- The attacker-controlled entry path is fully unprivileged: any Ethereum user can call `depositErc20` on the deployed helper contract with a rebasing token.
- No special role, key, or majority is required.
- The effect is cumulative and silent — each deposit adds a small shortfall that eventually causes withdrawal failures.

---

### Recommendation

1. **In the Solidity helper contracts**: Measure the actual balance difference around the `safeTransferFrom` call and emit that as the event value, rather than the caller-supplied `amount`:

```solidity
uint256 balanceBefore = IERC20(erc20Address).balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = IERC20(erc20Address).balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

2. **In the IC minter**: No change is needed on the IC side if the Solidity fix is applied, since the minter correctly trusts the event value. However, consider adding a minimum deposit threshold to prevent dust deposits that could arise from rounding.

3. **Governance**: Until the Solidity contracts are upgraded, avoid adding rebasing ERC20 tokens as supported ckERC20 tokens.

---

### Proof of Concept

1. A rebasing ERC20 token (e.g., stETH) is added as a supported ckERC20 token via governance.
2. Attacker (or any user) calls `depositErc20(stETH_address, 10^18, principal, subaccount)` on the deployed `DepositHelperWithSubaccount` contract.
3. Due to stETH share rounding ([lido-dao/issues/442](https://github.com/lidofinance/lido-dao/issues/442)), the minter address receives `10^18 - 1` or `10^18 - 2` stETH.
4. The event emits `amount = 10^18`.
5. The IC minter scrapes the log via `scrape_block_range` → `ReceivedEthOrErc20LogParser::parse_log` → stores `value = 10^18`.
6. `mint()` calls `icrc1_transfer` with `amount = 10^18`, minting 1–2 wei of unbacked ckstETH.
7. After `K` such deposits, the minter's stETH balance is `K × (10^18 - δ)` but ckstETH total supply is `K × 10^18`.
8. When any user withdraws `K × 10^18` ckstETH, the minter constructs an Ethereum `transfer(destination, K × 10^18)` call that reverts because the minter only holds `K × (10^18 - δ)` stETH — permanently blocking all withdrawals for that token.
