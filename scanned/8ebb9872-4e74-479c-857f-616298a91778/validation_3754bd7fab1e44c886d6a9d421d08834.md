### Title
Disabled `MintSnsTokens` 7-Day Rate Limit Allows Unbounded SNS Token Minting via Governance Proposals - (File: `rs/sns/governance/src/proposal.rs`)

### Summary
The SNS Governance canister contains a deliberately disabled rate-limiting guard for `MintSnsTokens` proposals. The `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` unconditionally returns `Decimal::MAX` instead of enforcing the intended 7-day treasury-proportional cap. This means any SNS token holder with sufficient voting power can pass an unlimited number of `MintSnsTokens` proposals within a 7-day window, minting arbitrary quantities of SNS tokens to any recipient — directly analogous to the ERC-20 `rescueTokens`/unilateral-owner-mint rug-pull vector in the external report.

### Finding Description

The `MintSnsTokens` proposal action allows SNS governance to mint new SNS tokens to an arbitrary principal. A rate-limiting framework exists (`transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` / `mint_sns_tokens_7_day_total_upper_bound_tokens`) that is correctly applied to `TransferSnsTreasuryFunds` proposals, but is **explicitly commented out** for `MintSnsTokens` via a `TODO(NNS1-2982)` marker.

The `TokenProposalAction` trait implementation for `MintSnsTokens` in `rs/sns/governance/src/proposal.rs` reads:

```rust
// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    // Ideally, we'd return infinity, but Decimal does not have that. This is the next best
    // thing, and should be good enough, because we have already planned the obselences of this
    // code (see tickets NNS1-298(1|2)).
    Ok(Decimal::MAX)
}
```

The correct implementation is commented out:

```rust
/* TODO(NNS1-2982): Uncomment.
fn recent_amount_total_upper_bound_tokens(valuation: &Valuation) -> Result<Decimal, String> {
    mint_sns_tokens_7_day_total_upper_bound_tokens(*valuation)
    ...
}
*/
```

The `perform_mint_sns_tokens` execution path in `rs/sns/governance/src/governance.rs` performs **no secondary rate-limit check at execution time** (unlike `perform_transfer_sns_treasury_funds`, which calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`). The integration test `sns_can_mint_funds_via_proposals` in `rs/sns/integration_tests/src/sns_treasury.rs` explicitly confirms this: the second minting proposal that should be blocked is instead allowed to succeed (`doomed_make_proposal_result.unwrap()`), with the blocking assertion commented out under the same `TODO(NNS1-2982)`.

The attacker-controlled entry path is:
1. An unprivileged ingress sender stakes SNS tokens to obtain a neuron with sufficient voting power (or accumulates followees).
2. The attacker submits a `MintSnsTokens` proposal targeting any principal with an arbitrary `amount_e8s`.
3. Because `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`, the proposal passes validation regardless of amount.
4. After adoption, `perform_mint_sns_tokens` calls `self.ledger.transfer_funds(amount_e8s, 0, None, to, mint.memo())` with no execution-time cap.
5. The attacker can repeat this across multiple proposals within the same 7-day window with no limit.

### Impact Explanation

An attacker controlling a majority of SNS voting power (or an SNS whose neuron follow graph concentrates power) can mint an unbounded quantity of SNS tokens to any account within a 7-day window. This inflates the token supply without bound, diluting all existing holders to near-zero value — the IC-native equivalent of the ERC-20 rug-pull via `rescueTokens`/owner-mint. The SNS ledger's `transfer_funds` call with `fee=0` and `from=None` (minting path) has no on-chain cap other than the disabled governance guard.

### Likelihood Explanation

Any SNS where a single entity or coordinated group holds a governance majority can exploit this immediately. SNS DAOs launched via the decentralization swap often have concentrated early neuron distributions. The attack requires no special technical capability beyond submitting a governance proposal — a standard ingress call to `manage_neuron`. The disabled guard is documented in the codebase itself as a known incomplete state.

### Recommendation

1. **Immediately enable the rate limit**: Uncomment the `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (remove the `TODO(NNS1-2982)` block) so that `mint_sns_tokens_7_day_total_upper_bound_tokens` is called during proposal validation.
2. **Add an execution-time check**: Mirror the pattern used by `perform_transfer_sns_treasury_funds`, which calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` before executing. Add an analogous `mint_sns_tokens_amount_is_small_enough_at_execution_time_or_err` call inside `perform_mint_sns_tokens`.
3. **Remove the permissive test stub**: Delete the `doomed_make_proposal_result.unwrap()` line in `sns_can_mint_funds_via_proposals` and uncomment the assertion block that verifies the second proposal is rejected.

        &mut self,
        proposal_id: u64, // This is just to control concurrency.
        valuation: Result<Valuation, GovernanceError>,
        transfer: &TransferSnsTreasuryFunds,
    ) -> Result<(), GovernanceError> {
        // Only execute one proposal of this type at a time.
        thread_local! {
            static IN_PROGRESS_PROPOSAL_ID: RefCell<Option<u64>> = const { RefCell::new(None) };
        }
        let release_on_drop = acquire(&IN_PROGRESS_PROPOSAL_ID, proposal_id);
        if let Err(already_in_progress_proposal_id) = release_on_drop {

                ErrorType::PreconditionFailed,
                format!(
                    "Another TransferSnsTreasuryFunds proposal (ID = {already_in_progress_proposal_id}) is already in progress.",
                ),
            ));
        }

            transfer,
            valuation?,
            self.proto.proposals.values(),
            self.env.now(),
        )?;

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
