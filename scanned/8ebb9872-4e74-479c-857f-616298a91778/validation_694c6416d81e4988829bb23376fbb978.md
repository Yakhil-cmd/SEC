### Title
ckBTC Minter Permanently Burns User ckBTC Without Reimbursement When Bitcoin Fees Spike After `retrieve_btc` Submission — (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

---

### Summary

The ckBTC minter uses a **dynamic** minimum withdrawal threshold (`fee_based_retrieve_btc_min_amount`) that is recomputed from live Bitcoin network fee percentiles each time `submit_pending_requests` runs. A user can call `retrieve_btc` or `retrieve_btc_with_approval` when fees are low, pass the minimum-amount gate, have their ckBTC irreversibly burned on the ledger, and then have the queued request silently dropped with `FinalizedStatus::AmountTooLow` — **with no reimbursement** — if Bitcoin fees spike before the minter processes the batch. This is a direct structural analog to M-9: a minimum-size check passes at creation time but the same effective check fails at execution time due to changed external conditions, causing the operation to be permanently abandoned and the user to lose funds.

---

### Finding Description

**Step 1 — Minimum check at submission time (dynamic threshold)**

In `retrieve_btc` and `retrieve_btc_with_approval`, the minter reads the current `fee_based_retrieve_btc_min_amount` and rejects requests below it:

```rust
let (min_retrieve_amount, btc_network) =
    read_state(|s| (s.fee_based_retrieve_btc_min_amount, s.btc_network));

if args.amount < min_retrieve_amount {
    return Err(RetrieveBtcError::AmountTooLow(min_retrieve_amount));
}
``` [1](#0-0) 

If the check passes, ckBTC is burned from the caller's account and the request is pushed to `pending_retrieve_btc_requests`. The burn is **irreversible** at this point. [2](#0-1) 

**Step 2 — Fee update overwrites the threshold at processing time**

`submit_pending_requests` calls `estimate_fee_per_vbyte`, which fetches live Bitcoin fee percentiles and **mutates** `fee_based_retrieve_btc_min_amount` in state:

```rust
mutate_state(|s| {
    s.last_fee_per_vbyte = fees;
    s.last_median_fee_per_vbyte = Some(median_fee);
    s.fee_based_retrieve_btc_min_amount = fee_based_retrieve_btc_min_amount;
});
``` [3](#0-2) 

The `fee_based_minimum_withdrawal_amount` formula adds a per-request RBF bound, vsize bound, minter fee, and check fee on top of the base `retrieve_btc_min_amount`, so the effective minimum can jump by 50 000-satoshi increments as fees rise: [4](#0-3) 

**Step 3 — `AmountTooLow` path drops the request without reimbursement**

`build_unsigned_transaction` is called with the **new, higher** fee rate. If the queued request amount can no longer cover fees, it returns `BuildTxError::AmountTooLow`. The minter handles this by calling `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`:

```rust
Err(BuildTxError::AmountTooLow) => {
    // There is no point in retrying the request because the amount is too low.
    for request in batch {
        state::audit::remove_retrieve_btc_request(
            s,
            request,
            state::FinalizedStatus::AmountTooLow,
            runtime,
        );
    }
    None
}
``` [5](#0-4) 

**Step 4 — `remove_retrieve_btc_request` does not reimburse**

`remove_retrieve_btc_request` in the audit module only records an event and pushes to the finalized queue. It does **not** call `reimburse_canceled_requests` and does not mint ckBTC back to the user: [6](#0-5) 

Contrast this with the `BuildTxError::InvalidTransaction` path, which **does** call `reimburse_canceled_requests` and mints ckBTC back: [7](#0-6) 

The same `AmountTooLow` finalization without reimbursement also occurs in the `BuildTxError::DustOutput` branch for the specific request that caused the dust output. [8](#0-7) 

---

### Impact Explanation

A user who submits a `retrieve_btc` request at a moment when Bitcoin fees are low (e.g., `fee_based_retrieve_btc_min_amount` = 100 000 sat) with an amount of exactly 100 001 sat will have their ckBTC burned. If Bitcoin fees then spike (e.g., `fee_based_retrieve_btc_min_amount` rises to 200 000 sat) before the minter processes the batch, the request is permanently dropped with `FinalizedStatus::AmountTooLow` and the user receives no reimbursement. The user permanently loses their ckBTC. This is a **ledger conservation bug**: ckBTC is burned but no corresponding BTC is ever sent and no ckBTC is minted back.

The `fee_based_retrieve_btc_min_amount` field in state confirms this is a live, mutable value distinct from the static `retrieve_btc_min_amount`: [9](#0-8) 

---

### Likelihood Explanation

Bitcoin fee markets are volatile. The `fee_based_minimum_withdrawal_amount` formula can increase the effective minimum by 50 000-satoshi steps. A user submitting a request near the current minimum during a low-fee period faces a realistic risk of their request being dropped if fees spike before the minter's next `submit_pending_requests` cycle. The entry path is fully unprivileged: any caller of `retrieve_btc` or `retrieve_btc_with_approval` is exposed. The `max_time_in_queue_nanos` parameter (set to 10 minutes on mainnet) means requests can sit in the queue long enough for fees to change materially.

---

### Recommendation

When `build_unsigned_transaction` returns `BuildTxError::AmountTooLow` for a queued request, the minter should reimburse the user by minting ckBTC back to the `reimbursement_account`, exactly as it does for `BuildTxError::InvalidTransaction`. The `reimburse_canceled_requests` helper already exists for this purpose. Alternatively, the minter could re-queue the request and retry when fees drop, rather than permanently dropping it.

---

### Proof of Concept

1. Bitcoin fees are low; `fee_based_retrieve_btc_min_amount` = 100 000 sat.
2. User calls `retrieve_btc` with `amount` = 100 001 sat. The check at line 169 of `retrieve_btc.rs` passes. ckBTC is burned at line 210.
3. Bitcoin fees spike; `estimate_fee_per_vbyte` updates `fee_based_retrieve_btc_min_amount` to 200 000 sat (line 248 of `lib.rs`).
4. `submit_pending_requests` calls `build_unsigned_transaction` with the new fee rate. It returns `BuildTxError::AmountTooLow`.
5. The `AmountTooLow` branch (lib.rs line 412) calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`.
6. `remove_retrieve_btc_request` (audit.rs line 67) records an event and finalizes the request — no ckBTC is minted back.
7. User's ckBTC is permanently lost. The request status is `AmountTooLow` with no reimbursement.
