### Title
Unbounded Loop Over Archive Canisters With Sequential Inter-Canister Upgrade Calls Can Trap the `UpgradeArchives` Task — (`File: rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs`)

---

### Summary

The Ledger Suite Orchestrator (LSO) canister, which manages all ckERC20 token ledger suites on the Internet Computer, contains an unbounded sequential loop in `UpgradeLedgerSuiteSubtask::UpgradeArchives`. For each token's archive canisters, the loop calls `upgrade_canister` (which itself makes three sequential inter-canister calls: `stop_canister`, `install_code`, `start_canister`) for every archive in the `archives` `Vec`. Because the `archives` list is unbounded and grows as ledger activity increases, and because each iteration performs three blocking async inter-canister calls, the entire `UpgradeArchives` subtask can exhaust the IC's per-message instruction limit, permanently trapping the upgrade path for that token's archives.

---

### Finding Description

In `UpgradeLedgerSuiteSubtask::UpgradeArchives`, the orchestrator iterates over all known archive canisters for a given ERC-20 token and upgrades them one by one in a sequential `for` loop:

```rust
//We expect usually 0 or 1 archive, so a simple sequential strategy is good enough.
for canister_id in archives {
    upgrade_canister::<Archive, _>(canister_id, compressed_wasm_hash, runtime)
        .await?;
}
``` [1](#0-0) 

Each call to `upgrade_canister` performs three sequential inter-canister calls to the management canister: `stop_canister`, `install_code` (upgrade), and `start_canister`:

```rust
runtime.stop_canister(canister_id).await ...
runtime.upgrade_canister(canister_id, wasm.to_bytes()).await ...
runtime.start_canister(canister_id).await ...
``` [2](#0-1) 

The `archives` list is populated by `discover_archives`, which calls `icrc3_get_archives` on each ledger and stores the result without any upper bound:

```rust
mutate_state(|s| s.record_archives(&token_id, archives.into_iter().collect()));
``` [3](#0-2) 

The `record_archives` function simply overwrites the stored `Vec<Principal>` with whatever the ledger reports:

```rust
pub fn record_archives(&mut self, token_id: &TokenId, archives: Vec<Principal>) {
    let canisters = self
        .managed_canisters_mut(token_id)
        .unwrap_or_else(|| panic!("BUG: token {token_id:?} is not managed"));
    canisters.archives = archives;
}
``` [4](#0-3) 

The `Canisters` struct stores archives as a plain `Vec<Principal>` with no size cap:

```rust
pub struct State {
    managed_canisters: ManagedCanisters,
    ...
}
``` [5](#0-4) 

The comment in the code itself acknowledges the assumption: `"We expect usually 0 or 1 archive, so a simple sequential strategy is good enough."` This assumption breaks as a high-volume token (e.g., ckUSDC, ckUSDT) accumulates many archive canisters over time.

The `UpgradeArchives` subtask is executed as a single timer-fired message. On the IC, a single message execution is bounded by the instruction limit (~40 billion instructions for update calls). Each `upgrade_canister` call involves three round-trips to the management canister. As the number of archives grows, the cumulative instruction cost of the loop body (including the async suspension/resume overhead and the management canister calls) can exceed the per-message limit, causing the message to be killed mid-loop. Because the task uses `?` propagation and the retry logic reschedules from the beginning of the `UpgradeArchives` subtask (not from the last successfully upgraded archive), the task will retry and fail again at the same point, permanently blocking archive upgrades for that token.

---

### Impact Explanation

- **Permanent DoS of archive upgrades for any ckERC20 token with many archives.** Once the archive count grows large enough, the `UpgradeArchives` subtask will always exhaust the instruction limit and never complete. The orchestrator's retry mechanism reschedules the entire subtask from scratch, so the failure is permanent, not transient.
- **Security/correctness risk from stale archive code.** If a critical bug is found in the archive canister WASM, the NNS cannot push a fix to archives of high-volume tokens. This is a direct analog to the `safeApproveAllTokens` trap in the original report.
- **No loss of funds directly**, but the inability to upgrade archive canisters means security patches cannot be applied to the archive layer of ckERC20 ledger suites.

---

### Likelihood Explanation

- **Medium-to-high likelihood over time.** The IC ICRC-1 ledger spawns a new archive canister every time the ledger's local block buffer exceeds `trigger_threshold` (default 2,000 blocks). High-volume tokens like ckUSDC or ckUSDT process thousands of transactions per day. Over months to years, a single token can accumulate dozens of archive canisters. The code comment itself acknowledges the assumption of "0 or 1 archive" which is already violated in production for high-volume tokens.
- The attacker-controlled entry path is **indirect but realistic**: any user can drive transaction volume on a ckERC20 token (by transferring tokens), which causes the ledger to spawn more archives, which grows the `archives` list, which eventually traps the upgrade loop. No privileged access is required.

---

### Recommendation

1. **Process archives in batches across multiple timer firings.** Store a `next_archive_index` in the `UpgradeArchives` subtask state (similar to how `UpgradeLedgerSuite` already uses `next_subtask_index`) and upgrade one archive per timer invocation, rescheduling the remainder.
2. **Alternatively, parallelize with `join_all`** (as `discover_archives` already does for discovery calls), but add an instruction-count check between iterations.
3. **Add a hard cap** on the number of archives that can be stored per token, or enforce a maximum batch size per upgrade cycle.

---

### Proof of Concept

1. Deploy the LSO on mainnet (already deployed as `sv3dd-oaaaa-aaaar-qacoa-cai`).
2. Over time, ckUSDC accumulates N archive canisters (N grows with transaction volume; each archive holds ~2,000 blocks by default).
3. An NNS proposal is submitted to upgrade the archive WASM to a new version.
4. The LSO's `post_upgrade` schedules `Task::UpgradeLedgerSuite` for each managed token, including ckUSDC.
5. The timer fires and executes subtasks in order: `UpgradeIndex` → `UpgradeLedger` → `DiscoverArchives` → `UpgradeArchives`.
6. `UpgradeArchives` enters the loop at line 415 and calls `upgrade_canister` for each of the N archives sequentially.
7. When N is large enough, the message exhausts the IC instruction limit mid-loop and is killed.
8. The retry logic reschedules `Task::UpgradeLedgerSuite` with the same `UpgradeArchives` subtask from the beginning.
9. The task retries and fails again at the same point — archive upgrades for ckUSDC are permanently blocked. [6](#0-5) [7](#0-6) [8](#0-7)

async fn upgrade_canister<T: StorableWasm, R: CanisterRuntime>(
    canister_id: Principal,
    wasm_hash: &WasmHash,
    runtime: &R,
) -> Result<(), UpgradeLedgerSuiteError> {
    let wasm = match read_wasm_store(|s| wasm_store_try_get::<T>(s, wasm_hash)) {
        Ok(Some(wasm)) => Ok(wasm),
        Ok(None) => Err(UpgradeLedgerSuiteError::WasmHashNotFound(wasm_hash.clone())),
        Err(e) => Err(UpgradeLedgerSuiteError::WasmStoreError(e)),
    }?;

    log!(DEBUG, "Stopping canister {}", canister_id);
    runtime
        .stop_canister(canister_id)
        .await
        .map_err(UpgradeLedgerSuiteError::StopCanisterError)?;

    log!(
        DEBUG,
        "Upgrading wasm module of canister {} to {}",
        canister_id,
        wasm_hash
    );
    runtime
        .upgrade_canister(canister_id, wasm.to_bytes())
        .await
        .map_err(UpgradeLedgerSuiteError::UpgradeCanisterError)?;

    log!(DEBUG, "Starting canister {}", canister_id);
    runtime
        .start_canister(canister_id)
        .await
        .map_err(UpgradeLedgerSuiteError::StartCanisterError)?;

    log!(
        DEBUG,
        "Upgrade of canister {} to {} completed",
        canister_id,
        wasm_hash
    );
    let now = runtime.time();
    mutate_state(|s| s.record_upgrade_completed(canister_id, wasm_hash.clone(), now));
    Ok(())
