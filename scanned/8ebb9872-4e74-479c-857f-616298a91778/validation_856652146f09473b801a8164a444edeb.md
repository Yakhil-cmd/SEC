**Step 1: Validation weakness confirmed**

`validate_self_validating_payload_impl` at `rs/bitcoin/consensus/src/payload_builder.rs:234-251` only checks `payload == SelfValidatingPayload::default()` (empty check) and returns the byte size. It never validates callback_id existence, response-type/request-type correspondence, or any content. [1](#0-0) 

**Step 2: `push_response` type-dispatch asymmetry**

In `rs/replicated_state/src/bitcoin.rs`, the `GetSuccessorsResponse` arm performs a mandatory context lookup and returns `StateError::BitcoinNonMatchingResponse` if the callback_id is absent from `bitcoin_get_successors_contexts`. [2](#0-1) 

The `SendTransactionResponse` arm performs **no context lookup at all** — it unconditionally constructs `Payload::Data(EmptyBlob.encode())` and pushes it to `consensus_queue` for whatever `callback_id` is in the response. [3](#0-2) 

**Step 3: Error handling in demux**

`DemuxImpl::process_payload` in `rs/messaging/src/routing/demux.rs` calls `push_response_bitcoin` and silently swallows errors with a `debug!` log — so even if an error were returned, it would not halt block finalization. [4](#0-3) 

**Step 4: Attacker model**

The attacker is a single Byzantine block proposer — a subnet node acting below the consensus fault threshold (< 1/3). Because `validate_payload` (the `BatchPayloadBuilder` path) also delegates to the same weak `validate_self_validating_payload_impl`, honest nodes will notarize the malicious block without objection. [5](#0-4) 

**Step 5: Concrete impact**

A `BitcoinAdapterResponse { response: SendTransactionResponse(()), callback_id: X }` where X is a pending `GetSuccessors` context ID will:
1. Pass all validation (size-only check).
2. Cause `push_response` to push `ConsensusResponse { callback_id: X, payload: EmptyBlob }` to `consensus_queue`.
3. The execution environment delivers `EmptyBlob` to the ckBTC minter callback registered for X, which expects `BitcoinGetSuccessorsResponse`.
4. Candid decoding fails; the minter callback receives a malformed response, potentially corrupting minter state or causing a trap.
5. The `GetSuccessors` context for X is **not removed** (the `SendTransactionResponse` arm never touches `bitcoin_get_successors_contexts`), so the context lingers and may be re-queried, creating a double-delivery scenario.

The same asymmetry applies to `GetSuccessorsReject` and `SendTransactionReject` — neither validates callback_id existence before pushing to `consensus_queue`. [6](#0-5)

### Title
Weak `validate_self_validating_payload_impl` allows Byzantine block proposer to inject type-mismatched Bitcoin adapter responses, corrupting ckBTC minter state — (`rs/bitcoin/consensus/src/payload_builder.rs`)

### Summary
`validate_self_validating_payload_impl` performs no content validation beyond an empty-payload check. A single Byzantine block proposer can include a `BitcoinAdapterResponse` carrying a `SendTransactionResponse` (or any reject variant) paired with a `callback_id` belonging to a `GetSuccessors` context. Because `push_response`'s `SendTransactionResponse` arm never validates the callback_id against `bitcoin_send_transaction_internal_contexts`, it unconditionally pushes `EmptyBlob` to the `consensus_queue` under the wrong callback_id. Honest nodes notarize the block because their own validation is equally weak.

### Finding Description
`validate_self_validating_payload_impl` (`rs/bitcoin/consensus/src/payload_builder.rs:234`) only checks `payload == SelfValidatingPayload::default()`. It does not verify that each `BitcoinAdapterResponse.callback_id` exists in the state, nor that the response variant matches the request type registered for that callback_id.

`push_response` (`rs/replicated_state/src/bitcoin.rs:23`) dispatches on the response variant. The `GetSuccessorsResponse` arm guards with a context lookup and returns `StateError::BitcoinNonMatchingResponse` on mismatch. The `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` arms perform **no such guard** — they push to `consensus_queue` unconditionally.

`DemuxImpl::process_payload` (`rs/messaging/src/routing/demux.rs:86`) silently discards any `StateError` returned by `push_response_bitcoin`, so even the partial guard on `GetSuccessorsResponse` cannot prevent finalization.

### Impact Explanation
The ckBTC minter (bitcoin canister) receives `EmptyBlob` (or a spurious reject) for a callback_id it registered for a `GetSuccessors` response. Candid decoding of `EmptyBlob` as `BitcoinGetSuccessorsResponse` fails. Depending on the minter's error handling, this causes a trap (state rollback for that round) or silent minter state corruption. Additionally, the `GetSuccessors` context is never removed, so the context lingers and may produce a double-delivery when an honest proposer later includes the legitimate adapter response.

### Likelihood Explanation
Requires a single Byzantine subnet node to act as block proposer — a realistic scenario within the IC fault model (below the 1/3 threshold). No privileged access, key material, or majority collusion is needed. The attack is deterministic and locally testable.

### Recommendation
1. In `validate_self_validating_payload_impl`, load the certified state and verify that each `callback_id` in the payload exists in the appropriate context map (`bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`) and that the response variant matches the registered request type.
2. In `push_response`, add a context lookup for `SendTransactionResponse` (and both reject variants) analogous to the existing guard on `GetSuccessorsResponse`, returning `StateError::BitcoinNonMatchingResponse` on mismatch.

    ),
    callback_id: 0,  // belongs to a GetSuccessors context
};
let payload = SelfValidatingPayload::new(vec![malicious_response]);

assert!(bitcoin_payload_builder
    .validate_self_validating_payload(&payload, &ctx, &[])
    .is_ok());

push_response(&mut state, malicious_response).unwrap();

// 5. Assert consensus_queue[0] has EmptyBlob, not BitcoinGetSuccessorsResponse.
assert_eq!(
    state.consensus_queue[0].payload,
    Payload::Data(EmptyBlob.encode())  // wrong type for a GetSuccessors callback
);
// The GetSuccessors context is still present (not consumed).
assert!(state.metadata.subnet_call_context_manager

        &self,
        payload: &SelfValidatingPayload,
        validation_context: &ValidationContext,

        let since = Instant::now();

        }

        self.metrics
            .observe_validate_duration(VALIDATION_STATUS_VALID, since);
        let size = NumBytes::new(payload.count_bytes() as u64);

        Ok(size)
