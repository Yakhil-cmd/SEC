### Title
SNS Swap Canister Never Calls `ic0.certified_data_set`, Enabling Malicious Boundary Nodes to Serve Manipulated Query Responses About Swap State - (File: rs/sns/swap/src/swap.rs)

---

### Summary
The SNS Swap canister exposes multiple query endpoints that return financially sensitive state (lifecycle, buyer participation amounts, derived state). Because the canister never calls `ic0.certified_data_set` at any point in its lifecycle, all query responses are uncertified. A malicious boundary node can intercept and replace these responses with arbitrary data, misleading users about whether a swap is open, committed, or aborted, and about their own participation balances.

---

### Finding Description
The IC's security model distinguishes between **update calls** (replicated, tamper-proof) and **query calls** (non-replicated, not certified by default). For a canister to allow clients to cryptographically verify query responses, it must call `ic0.certified_data_set` during update execution to commit a hash of its critical state, which the subnet then signs into the state tree. Clients can then verify query responses against the certified data via `ic0.data_certificate_copy`.

A grep across the entire SNS Swap canister source tree (`rs/sns/swap/**/*.rs`) returns **zero matches** for `set_certified_data`. This means:

- `get_lifecycle` — returns the swap's current lifecycle (Pending/Open/Committed/Aborted) — **uncertified**
- `get_buyer_state` — returns a buyer's ICP participation amount — **uncertified**
- `get_derived_state` — returns SNS tokens per ICP, total participation — **uncertified**
- `get_state` — returns the full swap state — **uncertified**
- `get_buyers_total` — returns total ICP committed — **uncertified** [1](#0-0) 

The core state-mutating functions — `refresh_buyer_token_e8s` (buyer participation), `try_commit`, `try_abort`, `try_open`, and `finalize_inner` — all mutate critical swap state but never call `set_certified_data` afterward. [2](#0-1) [3](#0-2) [4](#0-3) 

By contrast, the CMC canister — another financial canister in the same repository — explicitly implements `set_certified_data` through its `Environment` trait: [5](#0-4) 

The SNS Swap canister's DID interface confirms no `icrc3_get_blocks` endpoint exists either, meaning there is no ICRC-3 block log providing an alternative certified audit trail of swap events. [6](#0-5) 

---

### Impact Explanation
A malicious boundary node controlling the query path to the SNS Swap canister can:

1. **Lifecycle spoofing**: Return `Lifecycle::Open` to a user when the swap is actually `Committed` or `Aborted`. The user sends ICP to the swap subaccount, calls `refresh_buyer_tokens` (an update call), which fails because the swap is closed. The user's ICP is locked in the subaccount until `error_refund_icp` is called, causing temporary fund lockup and transaction fee loss.

2. **Balance spoofing**: Return `icp_accepted_participation_e8s = 0` for a buyer who already has a large committed balance. The user, believing their prior transfer was not recorded, sends additional ICP and calls `refresh_buyer_tokens` again. If the swap is still open, this succeeds and the user over-commits ICP beyond their intent.

3. **Derived state manipulation**: Return a false `sns_tokens_per_icp` ratio, causing users to miscalculate the value of their participation and make economically irrational decisions.

The financial impact is direct and concrete: users can be induced to send ICP to a closed swap or to over-commit ICP based on false balance information.

---

### Likelihood Explanation
The IC's security model explicitly treats boundary nodes as potentially adversarial for query calls. Boundary nodes are operated by independent parties and are not subject to the same consensus guarantees as replica nodes. Any boundary node on the path between a user and the SNS Swap canister can serve manipulated query responses. Because the SNS Swap canister provides no certified data, clients have no cryptographic mechanism to detect such manipulation. The attack requires no special privileges — any boundary node operator can execute it against users routed through their node.

---

### Recommendation
After each significant state transition in the SNS Swap canister, call `ic_cdk::api::set_certified_data` with a hash of the critical state fields (lifecycle, total ICP committed, swap parameters hash). Specifically:

- After `try_open`, `try_commit`, `try_abort` succeed
- After `refresh_buyer_token_e8s` updates a buyer's balance
- After `finalize_inner` completes

This allows clients to verify query responses against the subnet-signed state tree via `icrc3_get_tip_certificate`-style patterns, consistent with how the CMC and Registry canisters already operate.

**Step 1 — Confirm absence of `set_certified_data` in the SNS Swap canister:**

```
grep -r "set_certified_data" rs/sns/swap/
# Returns: No matches
```

**Step 2 — Confirm the CMC canister (same repo, same financial domain) does use it:**

grep -r "set_certified_data" rs/nns/cmc/
# Returns: rs/nns/cmc/src/main.rs, rs/nns/cmc/src/environment.rs, rs/nns/cmc/src/exchange_rate_canister.rs
``` [7](#0-6) 

**Step 3 — Attacker scenario (malicious boundary node):**

```
# User queries swap lifecycle via boundary node B (malicious)
dfx canister --network https://B.boundary.ic0.app call <swap_id> get_lifecycle '(record {})'
# B returns: variant { lifecycle = opt 2 }  (Open)
# Actual on-chain state: variant { lifecycle = opt 4 }  (Committed)

# User, believing swap is open, transfers ICP to swap subaccount
# User calls refresh_buyer_tokens (update call — goes through consensus, cannot be spoofed)
# Result: Error — swap is not in Open state
# User's ICP is now locked in the subaccount
```

Because `get_lifecycle` returns an uncertified response and the canister never sets certified data, the user has no way to detect the boundary node's lie before committing funds. [8](#0-7)

    pub async fn finalize(
        &mut self,
        now_fn: fn(bool) -> u64,
        environment: &mut impl CanisterEnvironment,
    ) -> FinalizeSwapResponse {
        // Acquire the lock or return a FinalizeSwapResponse with an error message.
        if let Err(error_message) = self.lock_finalize_swap() {
            return FinalizeSwapResponse::with_error(error_message);
        }

        // The lock is now acquired and asynchronous calls to finalize are blocked.
        // Perform all subactions.
        let finalize_swap_response = self.finalize_inner(now_fn, environment).await;

        if finalize_swap_response.has_error_message() {
            log!(
                ERROR,
                "The swap did not finalize successfully. \n\
                finalize_swap_response: {finalize_swap_response:?}"
            );
        } else {
            log!(
                INFO,
                "The swap finalized successfully. \n\
                finalize_swap_response: {finalize_swap_response:?}"
            );
        }

        // Release the lock. Note, if there is a panic, the lock will
        // not be released. In that case, the Swap canister will need
        // to be upgraded to release the lock.
        self.unlock_finalize_swap();

        finalize_swap_response
    }
