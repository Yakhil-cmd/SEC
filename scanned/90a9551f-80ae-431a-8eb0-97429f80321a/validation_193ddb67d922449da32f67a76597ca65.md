### Title
`SenderInfoContent` Signed Bytes Lack Target-Canister Binding, Enabling Cross-Canister Replay of Caller Attribute Attestations - (File: `rs/types/types/src/messages/http.rs`)

### Summary

The `SenderInfoContent` structure, which is the signed payload for the `sender_info` field in IC HTTP requests, contains only the raw `info` blob. It does not bind the signature to the target canister ID or the sender's principal. This is the direct IC analog of the ParaSpace `Credit` structure missing a `MarketplaceAddress` field: a signature produced by a trusted signing canister (e.g., Internet Identity) for one context can be replayed verbatim in a request targeting a completely different canister, allowing an attacker to present forged caller-attribute attestations to any canister that trusts `sender_info`.

### Finding Description

`SenderInfoContent` is defined as a newtype wrapper over a raw byte slice:

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);  // only the raw info blob
    }
}
```

The full signed bytes are therefore: `"\x0Eic-sender-info" || info_bytes` — nothing else. [1](#0-0) 

The `SignatureDomain` implementation confirms the domain separator is `"ic-sender-info"` with no canister-specific component: [2](#0-1) 

The validator `verify_sender_info_canister_sig` performs three checks:
1. `sender_pubkey` is a valid canister signature public key.
2. The canister ID encoded in `sender_pubkey` matches `sender_info.signer`.
3. The canister signature over `SenderInfoContent(info_bytes)` is cryptographically valid. [3](#0-2) 

Critically, **no check binds the signature to the target `canister_id` of the request, the `sender` principal, or any other request-specific field**. The `validate_sender_info` function receives the full `HttpRequest<C>` but never passes the target canister or sender to the verification logic: [4](#0-3) 

The `SenderInfoContent` signed bytes are identical regardless of which canister the request targets, which method is called, or which user is sending the request.

### Impact Explanation

The `sender_info` feature is designed to let a trusted signing canister (e.g., Internet Identity) attest to caller attributes (e.g., verified age, membership tier, KYC status) that the target canister can read via `msg_caller_info_data()`. Canisters are expected to use this for authorization decisions.

Because the signed payload contains only `info_bytes` with no canister binding:

- A valid `(info, signer, sig)` tuple produced by Internet Identity for a request to canister A is cryptographically valid for any request to canister B, as long as the attacker authenticates using a canister signature from the same signing canister (Internet Identity) with the same seed used to produce the `sender_info` signature.
- An attacker who observes a victim's `sender_info` from a public HTTP envelope (the envelope is transmitted in plaintext over HTTPS) can replay it in their own request to a different canister.
- Any canister that grants elevated privileges based on `sender_info` attributes (e.g., "this user is KYC-verified", "this user is a premium member") can be bypassed.

The `sender_info` is included in the `MessageId` hash, so the outer envelope signature is bound to the specific request. However, the *inner* canister signature over `SenderInfoContent` is not, which is the root cause. [5](#0-4) 

### Likelihood Explanation

The attack is reachable by any unprivileged boundary-node user who:
1. Observes a valid `sender_info` from any public HTTP request (e.g., by submitting a request to the same canister and inspecting the envelope, or by monitoring network traffic).
2. Authenticates to the IC using a canister signature from the same signing canister (e.g., any Internet Identity anchor).
3. Constructs a new request to a target canister with the replayed `sender_info`.

No privileged access, no key compromise, and no threshold attack is required. The `sender_info` field is part of the public HTTP envelope and is not encrypted. The feature is actively deployed (tests confirm it is accepted for both update and query calls). [6](#0-5) 

### Recommendation

Bind the `SenderInfoContent` signed bytes to the target canister ID (and optionally the sender's principal). The `write_signed_bytes_without_domain_separator` implementation should include the target canister ID:

```rust
pub struct SenderInfoContent<'a> {
    pub info: &'a [u8],
    pub target_canister_id: &'a [u8],  // add binding
}

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        // length-prefix each field for prefix-freeness
        bytes.extend_from_slice(&(self.target_canister_id.len() as u64).to_be_bytes());
        bytes.extend_from_slice(self.target_canister_id);
        bytes.extend_from_slice(self.info);
    }
}
```

The `verify_sender_info_canister_sig` function must then pass the request's `canister_id` into the `SenderInfoContent` construction at line 530, and the signing canister must include the target canister ID when producing the signature. [7](#0-6) 

### Proof of Concept

1. Alice (Internet Identity anchor A) sends a request to canister X with `sender_info = {info: b"premium_user", signer: II_canister_id, sig: <II_sig>}`. The II canister signs `SenderInfoContent(b"premium_user")` = `"\x0Eic-sender-info" || b"premium_user"`.

2. Attacker Bob (Internet Identity anchor B, non-premium) observes Alice's HTTP envelope and extracts `sender_info`.

3. Bob constructs a request to canister Y (a DeFi canister that grants premium features based on `sender_info`), using his own II canister signature for authentication but including Alice's `sender_info` tuple unchanged.

4. `verify_sender_info_canister_sig` checks:
   - Bob's `sender_pubkey` canister ID == `II_canister_id` == `sender_info.signer` ✓
   - Signature over `SenderInfoContent(b"premium_user")` is valid (it was produced by II) ✓

5. Canister Y calls `msg_caller_info_data()`, receives `b"premium_user"`, and grants Bob premium access — even though Bob is not a premium user. [8](#0-7) [9](#0-8)

    };

        sender_info,
        sender_pubkey,
        ingress_signature_verifier,
        root_of_trust_provider,
    )

/// Verifies that the sender_info canister signature is valid:
/// 1. The envelope-level sender_pubkey is a valid canister signature public key.
/// 2. The canister ID encoded in sender_pubkey matches the signer field.
/// 3. The signature over the info blob is valid against the root of trust.
fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
}
