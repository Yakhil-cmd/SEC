### Title
Missing Zero-Address Validation in Chain-Fusion Deposit Helper Contract Constructors - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol, EthDepositHelper.sol, ERC20DepositHelper.sol)

### Summary
The Ethereum-side chain-fusion deposit helper contracts (`CkDeposit`, `CkEthDeposit`, `CkErc20Deposit`) accept a `_minterAddress` / `_cketh_minter_main_address` constructor parameter and store it as an immutable `payable` address without validating it is non-zero. This is the direct IC-codebase analog of the Pickle Finance missing `require(variableAddress != address(0))` constructor check.

### Finding Description
All three Ethereum helper contracts in the ckETH/ckERC20 chain-fusion system store the minter's Ethereum address as an immutable field set only at construction time, with no zero-address guard:

**`EthDepositHelper.sol` â€” `CkEthDeposit` constructor:** [1](#0-0) 

**`ERC20DepositHelper.sol` â€” `CkErc20Deposit` constructor:** [2](#0-1) 

**`DepositHelperWithSubaccount.sol` â€” `CkDeposit` constructor:** [3](#0-2) 

The inconsistency is most glaring in `DepositHelperWithSubaccount.sol`: the contract explicitly defines `ZERO_ADDRESS = address(0)` and uses it to guard `depositErc20`'s `erc20Address` parameter, yet the constructor never applies the same guard to `_minterAddress`: [4](#0-3) 

The `depositErc20` guard that exists: [5](#0-4) 

### Impact Explanation
**Vulnerability class: chain-fusion mint/burn/replay bug.**

If any of these contracts is deployed with `_minterAddress = address(0)`:

1. `depositEth()` / `deposit()` emits a `ReceivedEth` or `ReceivedEthOrErc20` event and calls `minterAddress.transfer(msg.value)`, sending ETH to `address(0)` (permanently burned). The IC ckETH minter scrapes logs from the contract address, observes the event, and proceeds to mint ckETH to the encoded IC principal â€” **minting unbacked ckETH with no corresponding ETH received by the minter**. [6](#0-5) 

2. `depositErc20()` transfers ERC-20 tokens to `address(0)` (burned), while the emitted event still triggers ckERC20 minting on the IC side â€” **minting unbacked ckERC20**.

The IC minter canister itself does validate its own `ledger_id` principal (correctly rejecting anonymous): [7](#0-6) 

But there is no corresponding guard on the Ethereum side for the minter address in the constructor.

### Likelihood Explanation
**Medium-low.** The official contracts are deployed by DFINITY, so exploitation requires either (a) a deployment-time error where `address(0)` is accidentally passed, or (b) a user being directed to interact with a maliciously or erroneously deployed copy of the contract. Because `minterAddress` is `immutable`, there is no recovery path once deployed with a zero address â€” the contract is permanently broken and any ETH/ERC20 sent to it is lost while ckETH/ckERC20 may still be minted. The absence of a guard makes this class of deployment error silently catastrophic.

### Recommendation
Add a zero-address check at the top of each constructor, mirroring the pattern already used in `depositErc20`:

```solidity
// EthDepositHelper.sol / ERC20DepositHelper.sol
constructor(address _cketh_minter_main_address) {
    require(_cketh_minter_main_address != address(0), "minter address cannot be zero");
    cketh_minter_main_address = payable(_cketh_minter_main_address);
}

// DepositHelperWithSubaccount.sol
constructor(address _minterAddress) {
    require(_minterAddress != ZERO_ADDRESS, "minter address cannot be zero");
    minterAddress = payable(_minterAddress);
}
```

### Proof of Concept

1. Deploy `CkDeposit` (from `DepositHelperWithSubaccount.sol`) with `_minterAddress = address(0)`. The constructor succeeds silently.
2. Call `depositEth{value: 1 ether}(encodedPrincipal, subaccount)`. The call succeeds: the `ReceivedEthOrErc20` event is emitted with the caller's IC principal, and 1 ETH is transferred to `address(0)` (burned).
3. The IC ckETH minter, scraping logs from this contract address, observes the `ReceivedEthOrErc20` event and mints 1 ckETH (minus fee) to the specified IC principal.
4. Result: 1 ckETH exists on the IC with no backing ETH held by the minter â€” a chain-fusion conservation violation.
