### Title
Per-Participant Investment Limit Bypass via Arbitrary `buyer` Parameter in SNS Swap - (File: `rs/sns/swap/canister/canister.rs`)

---

### Summary

The `refresh_buyer_tokens` endpoint of the SNS Swap canister accepts an arbitrary `buyer` principal in its request payload. The per-participant ICP cap (`max_participant_icp_e8s`) is enforced against the `buyer` address rather than the actual ingress caller. An unprivileged user who has reached their own participation limit can bypass it by funding and registering participation under fresh principals they control, acquiring disproportionate SNS tokens and governance power.

---

### Finding Description

The `refresh_buyer_tokens` update method in `rs/sns/swap/canister/canister.rs` resolves the participant identity from the caller-supplied `arg.buyer` string field rather than from `caller_principal_id()`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← any principal accepted
    };
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

Inside `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`, the ICP balance lookup, the per-participant cap check, and the state update all operate on the caller-supplied `buyer` principal:

```rust

    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),  // ← buyer, not caller

// State written under buyer's key
self.buyers.entry(buyer.to_string())...set_amount_icp_e8s(new_balance_e8s);
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The ticket system (`new_sale_ticket`) is entirely optional. The comment in `refresh_buyer_token_e8s` at line 1271 explicitly states: *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket."* [5](#0-4) 

The `RefreshBuyerTokensRequest` proto documents the `buyer` field as *"If not specified, the caller is used"*, confirming the design intentionally allows specifying a different beneficiary. [6](#0-5) 

---

### Impact Explanation

The `max_participant_icp_e8s` parameter is the SNS swap's primary mechanism for ensuring fair token distribution and preventing any single entity from acquiring outsized governance power. By routing participation through multiple fresh buyer principals, a single attacker can:

1. Accumulate SNS tokens far beyond the intended per-participant cap.
2. Gain disproportionate voting power in the newly created SNS DAO from day one.
3. Undermine the decentralization goals that the SNS swap is designed to enforce.

The `max_direct_participation_icp_e8s` total cap still applies, so the attacker cannot raise more ICP than the swap allows in aggregate — but within that cap, one entity can crowd out legitimate participants and dominate the distribution.

---

### Likelihood Explanation

The attack requires no privileged access, no admin key, and no threshold corruption. Any unprivileged ingress sender can:

- Generate arbitrarily many IC principals (self-authenticating, derived from fresh key pairs).
- Transfer ICP to each fresh principal's subaccount of the swap canister via the ICP ledger.
- Call `refresh_buyer_tokens` with each fresh principal as `buyer` — no ticket required.

The only practical constraint is the `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` ceiling on total participant count, which limits how many distinct buyer principals can be registered. However, this ceiling is large enough that a well-resourced attacker can still acquire a dominant share before it is reached.

---

### Recommendation

Enforce the per-participant limit against the **ingress caller** (`caller_principal_id()`), not the caller-supplied `buyer` field. Specifically:

- In `refresh_buyer_tokens` (canister.rs), always use `caller_principal_id()` as the principal whose accumulated balance and limit are checked.
- If the design intent is to allow third-party participation on behalf of another address, track and limit the **funding source** (caller) rather than the beneficiary (`buyer`).
- Alternatively, require that `buyer == caller` unless the caller is a whitelisted canister (e.g., the Neurons' Fund).

---

### Proof of Concept

```
// Attacker Alice has already reached max_participant_icp_e8s = 5 ICP

// Step 1: Generate fresh principals Bob1, Bob2, ... BobN (self-authenticating)
let bob1 = fresh_principal();
let bob2 = fresh_principal();

// Step 2: Transfer ICP to each BobN's subaccount of the swap canister
icp_ledger.transfer({
    to: Account { owner: swap_canister_id, subaccount: principal_to_subaccount(bob1) },
    amount: 5 ICP
});
icp_ledger.transfer({
    to: Account { owner: swap_canister_id, subaccount: principal_to_subaccount(bob2) },
    amount: 5 ICP
});

// Step 3: Call refresh_buyer_tokens with each fresh buyer — no ticket needed
// Alice is the ingress caller; Bob1/Bob2 are the buyers
swap.refresh_buyer_tokens({ buyer: bob1.to_string(), confirmation_text: None });
swap.refresh_buyer_tokens({ buyer: bob2.to_string(), confirmation_text: None });

// Result: Alice has effectively participated with 10 ICP (2× the 5 ICP cap)
// Bob1 and Bob2 receive SNS neurons; Alice controls Bob1 and Bob2's keys
// Repeating with BobN allows Alice to acquire an unbounded share of the swap
```

The root cause is in:
- `rs/sns/swap/canister/canister.rs` lines 130–134: caller-supplied `buyer` is used as the participant identity without restriction.
- `rs/sns/swap/src/swap.rs` lines 1210–1237: per-participant limit is checked and applied against `buyer`, not the ingress caller.
