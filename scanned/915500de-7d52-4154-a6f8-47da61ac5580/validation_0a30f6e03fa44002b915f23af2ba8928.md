### Title
Unbounded Unvalidated DKG Pool Growth via Deferred Remote Dealings with Unknown Target IDs — (`rs/consensus/dkg/src/lib.rs`)

### Summary

A Byzantine subnet member (below the fault threshold) can inject an unbounded number of DKG dealing messages with `dkg_id.target_subnet = NiDkgTargetSubnet::Remote(unknown_id)` into the unvalidated pool. Because `validate_dealings_for_dealer` silently defers these messages instead of rejecting them, and because the DKG pool has no size cap and the P2P layer uses `SLOT_TABLE_NO_LIMIT` for DKG artifacts, these messages accumulate in memory until the next DKG interval purge.

---

### Finding Description

**Deferred return for unknown remote target IDs:**

In `validate_dealings_for_dealer`, when a dealing's `dkg_id` is not found in the current configs map and `target_subnet.is_remote()` is true, the function returns `Mutations::new()` (no-op) rather than `HandleInvalid`: [1](#0-0) 

This early return happens *before* any dealer identity or signature check, so any subnet peer (not just legitimate dealers) can inject deferred messages.

**No size limit on the DKG unvalidated pool:**

`DkgPoolImpl::insert()` places artifacts into a `PoolSection` with no quota or eviction policy: [2](#0-1) 

**P2P layer uses `SLOT_TABLE_NO_LIMIT` for DKG:**

The DKG broadcast channel is constructed with `SLOT_TABLE_NO_LIMIT = usize::MAX`, meaning no per-peer cap on how many DKG slot advertisements are accepted: [3](#0-2) 

Compare this to ingress, which uses `SLOT_TABLE_LIMIT_INGRESS = 50_000`: [4](#0-3) 

**Purge only occurs at DKG interval boundaries:**

The pool is only purged when `start_height > dkg_pool.get_current_start_height()`, i.e., at the start of a new DKG interval (default ~60 blocks). Deferred messages accumulate for the entire interval: [5](#0-4) 

**Bouncer accepts all messages at the current start height:**

`DkgBouncer` returns `BouncerValue::Wants` for any `DkgMessageId` whose height equals `current_start_height`, with no check on `target_subnet`: [6](#0-5) 

---

### Impact Explanation

Each unique `NiDkgTargetId` (32-byte array) produces a distinct `DkgMessageId` hash, so messages are not deduplicated across different target IDs. A Byzantine peer can advertise thousands of unique DKG artifacts using different slot numbers (no slot limit), each fetched and inserted into the unvalidated pool. DKG dealings contain non-trivial cryptographic material. Across a 60-block interval, this can exhaust replica memory and degrade or halt consensus participation.

---

### Likelihood Explanation

The attacker must be a legitimate subnet member (Byzantine node within the fault threshold). This is the standard Byzantine adversary model for IC subnet protocols. No external access, leaked keys, or governance majority is required — only the ability to sign messages with the attacker's own node key and advertise them over the P2P layer.

---

### Recommendation

1. **Reject unknown remote target IDs immediately** if the `start_block_height` matches the current interval but no config exists for the `dkg_id`. The current deferral was designed for the case where the remote DKG request has not yet appeared in certified state, but this window should be bounded.
2. **Apply a per-peer slot table limit for DKG** analogous to `SLOT_TABLE_LIMIT_INGRESS`, or add a maximum unvalidated pool size.
3. **Add a count-based cap** inside `DkgPoolImpl::insert()` to reject insertions beyond a configurable threshold.

---

### Proof of Concept

```
1. Byzantine node B is a legitimate member of subnet S.
2. B crafts N DKG dealing messages, each with:
   - content.dkg_id.start_block_height = current DKG interval start height
   - content.dkg_id.target_subnet = NiDkgTargetSubnet::Remote([i as u8; 32]) for i in 0..N
   - signature.signer = B's own NodeId (valid signature)
3. B advertises each message on a distinct P2P slot number.
4. Receiving replicas: DkgBouncer returns Wants (height matches), SLOT_TABLE_NO_LIMIT allows all slots.
5. Each message is inserted into the unvalidated pool.
6. on_state_change calls validate_dealings_for_dealer for each; all return Mutations::new() (deferred).
7. Pool grows by N entries, each holding a full DKG dealing, until the next interval purge.
8. Repeat across multiple intervals to sustain memory pressure.
``` [7](#0-6) [8](#0-7)

    fn validate_dealings_for_dealer(
        &self,
        dkg_pool: &dyn DkgPool,
        configs: &BTreeMap<&NiDkgId, &NiDkgConfig>,
        dkg_start_height: Height,
        messages: &[&Message],
    ) -> Mutations {
        // Because dealing generation is not entirely deterministic, it is
        // actually possible to receive multiple dealings from an honest dealer.
        // As such, we simply try validating the first message in the list, and
        // get a result for that message. Other messages will be dealt with by
        // subsequent calls to this function.
        let message = if let Some(message) = messages.first() {
            message
        } else {
            return Mutations::new();
        };

        if message.content.version != ReplicaVersion::default() {
            return Mutations::from(ChangeAction::RemoveFromUnvalidated((*message).clone()));
        }

        let message_dkg_id = &message.content.dkg_id;

        // If the dealing refers to a DKG interval starting at a different height,
        // we skip it.
        if message_dkg_id.start_block_height != dkg_start_height {
            return Mutations::new();
        }

        // If the dealing refers a config which is not among the ongoing DKGs,
        // we reject it, unless it is a remote DKG, in which case we defer it
        // until the request appears in the state, or the dealing is purged.
        let config = match configs.get(message_dkg_id) {
            Some(config) => config,
            None if message_dkg_id.target_subnet.is_remote() => {
                return Mutations::new();
            }
            None => {
                return get_handle_invalid_change_action(
                    message,
                    format!("No DKG configuration for Id={message_dkg_id:?} was found."),
                )
                .into();
            }
        };

impl<Pool: DkgPool> BouncerFactory<DkgMessageId, Pool> for DkgBouncer {
    fn new_bouncer(&self, dkg_pool: &Pool) -> Bouncer<DkgMessageId> {
        let _timer = self.metrics.update_duration.start_timer();

        let start_height = dkg_pool.get_current_start_height();
        Box::new(move |id| {
            use std::cmp::Ordering;
            match id.height.cmp(&start_height) {
                Ordering::Equal => BouncerValue::Wants,
                Ordering::Greater => BouncerValue::MaybeWantsLater,
                Ordering::Less => BouncerValue::Unwanted,
            }
        })

    fn purge(&mut self, height: Height) -> Vec<DkgMessageId> {
        self.current_start_height = height;
        // TODO: use drain_filter once it's stable.
        let unvalidated_keys: Vec<_> = self
            .unvalidated
            .keys()
            .filter(|id| id.height < height)
            .cloned()
            .collect();
        for id in unvalidated_keys {
            self.unvalidated.remove(&id);
        }

        let validated_keys: Vec<_> = self
            .validated
            .keys()
            .filter(|id| id.height < height)
            .cloned()
            .collect();
        for hash in &validated_keys {
            self.validated.remove(hash);
        }
        validated_keys
    }
