### Title
Unbounded `ingress_expiries` Vec allocation via attacker-controlled `ingress_start`/`ingress_end` in ICP Rosetta `/construction/payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

### Summary
The ICP Rosetta server's `construction_payloads` handler accepts arbitrary `ingress_start` and `ingress_end` nanosecond timestamps from any unauthenticated HTTP client and feeds them directly into an unbounded `while` loop that allocates one `u64` per `interval`-sized step. There is no cap on the window size before allocation begins. A single crafted request with `ingress_start=0, ingress_end=u64::MAX` causes ~153 billion loop iterations and ~1.2 TB of attempted heap allocation, crashing the Rosetta server process via OOM.

### Finding Description
In `construction_payloads` (lines 59–107), `interval` is fixed at 120 s (120,000,000,000 ns). Both `ingress_start` and `ingress_end` are deserialized from the caller-supplied JSON metadata as raw `u64` nanosecond values with no validation. The loop:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;          // += 120_000_000_000
}
```

has no guard on `(ingress_end - ingress_start)`. With `ingress_start=0` and `ingress_end=u64::MAX`, the iteration count is:

```
18_446_744_073_709_551_615 / 120_000_000_000 ≈ 153_722_867_280 iterations
```

Each iteration pushes 8 bytes, totalling ~1.2 TB. The process OOMs before completing. Even a modest window of 1 billion seconds (~31 years) produces ~8.3 million iterations (~66 MB) per request; a handful of concurrent requests exhausts typical server RAM.

The ICRC1 Rosetta sibling function (`rs/rosetta-api/icrc1/src/construction_api/services.rs`, lines 148–158) correctly rejects requests where `ingress_start >= ingress_end` or `ingress_end` is not at least one interval ahead of `now`. The ICP Rosetta handler has no equivalent guard.

### Impact Explanation
An unauthenticated attacker can crash the ICP Rosetta server process (OOM kill) with a single HTTP POST. This denies service to all users of the Rosetta API (exchanges, wallets, integrators) until the process is restarted. Repeated requests prevent recovery.

### Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication, no API key, and no prior state. The malicious payload is a small, valid JSON object. The attack is trivially reproducible locally and requires no special knowledge beyond the Rosetta API spec.

### Recommendation
Add a window-size cap before the loop, mirroring the ICRC1 implementation:

```rust
// Reject if window exceeds a reasonable maximum (e.g., 24 hours)
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60);
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed window"
    ));
}
// Also reject if ingress_start >= ingress_end
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request("ingress_start must be before ingress_end"));
}
```

Alternatively, cap `ingress_expiries` with a hard `Vec::with_capacity` limit and break early.
