### Title
Single Controller Can Mint Unlimited Tokens Without Supply Cap or Multi-Party Authorization — (File: `rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
The `icrc152_mint` endpoint in the ICRC-1 ledger canister allows any single canister controller to mint an arbitrary amount of tokens to any account with no configurable supply cap and no multi-party authorization requirement. This is the direct IC analog of the Cumulative Merkle Drop vulnerability where a single owner could set a merkle root granting themselves an unlimited claimable amount.

### Finding Description
`icrc152_mint_not_async` at `rs/ledger_suite/icrc1/ledger/src/main.rs` lines 905–988 performs the following checks before executing a mint:

1. The `icrc152` feature flag is enabled.
2. `ic_cdk::api::is_controller(&caller)` — the caller is **any single** controller principal.
3. `args.amount != 0` — amount is nonzero.
4. `Tokens::try_from(args.amount)` — amount fits in the `Tokens` type (bounded by `Tokens::max_value()`).
5. Target is not anonymous and not the minting account.

There is no configurable supply cap, no per-period rate limit, and no requirement for multi-party authorization. A single controller can call `icrc152_mint` in a loop, minting up to `Tokens::max_value()` total supply in a single session. The controller can designate their own account as the recipient.

The underlying `mint` function in `rs/ledger_suite/common/ledger_core/src/balances.rs` lines 145–156 enforces only the hard arithmetic ceiling (`token_pool.checked_sub` panics on underflow), not any governance-configured cap:

```rust
pub fn mint(&mut self, to: &S::AccountId, amount: S::Tokens) -> Result<(), BalanceError<S::Tokens>> {
    self.token_pool = self.token_pool.checked_sub(&amount)
        .expect("total token supply exceeded");
    self.credit(to, amount);
    Ok(())
}
```

The `Icrc152MintArgs` struct (`packages/icrc-ledger-types/src/icrc152/mod.rs` lines 7–12) accepts an unbounded `Nat` as the amount, and the only rejection is if it exceeds `Tokens::max_value()`.

### Impact Explanation
A single controller of any ICRC-1 ledger with `icrc152: true` can unilaterally inflate the token supply to the maximum representable value. For SNS tokens, custom chain-fusion tokens (e.g., ckDOGE), or any ICRC-1 ledger where the controller is a single developer principal rather than a governance canister, this allows:

- Complete destruction of token economics by minting near-`u64::MAX` tokens to the controller's own account.
- Bypassing any intended token distribution model (fixed supply, vesting schedules, swap allocations).
- Draining liquidity pools or DEX pairs that hold the token, since the controller can flood supply.

This is a **ledger conservation bug / governance authorization bug**: the total supply invariant is only bounded by the arithmetic maximum, not by any policy-enforced cap.

### Likelihood Explanation
Medium. The ICRC-152 feature must be explicitly enabled at init (`feature_flags: Some(FeatureFlags { icrc152: true, ... })`). However, any SNS project, ckDOGE-style chain-fusion token, or custom ICRC-1 deployment that enables ICRC-152 and has a single-principal controller is immediately exposed. The attacker path requires only that the controller principal sends a single ingress `icrc152_mint` call — no leaked keys, no social engineering, no consensus attack.

### Recommendation
1. Add an optional `max_supply: Option<Nat>` field to `InitArgs` and enforce it inside `icrc152_mint_not_async` before applying the transaction.
2. Alternatively, require that `icrc152_mint` calls above a configurable threshold be authorized by a governance proposal or an M-of-N multisig controller set, mirroring the original report's recommendation of a multisig wallet for the owner.
3. At minimum, document that enabling `icrc152` on a ledger controlled by a single principal is equivalent to granting that principal an uncapped mint authority.

### Proof of Concept
```
// Attacker is the sole controller of an ICRC-1 ledger with icrc152: true.
// Single ingress call — no governance, no multisig, no cap check:
icrc152_mint({
  to: { owner: attacker_principal, subaccount: null },
  amount: 18_446_744_073_709_551_615,  // u64::MAX — Tokens::max_value()
  created_at_time: <now_nanos>,
  reason: null

// Returns Ok(block_idx). Total supply is now u64::MAX.
// All prior holders' balances are economically worthless.
```

```rust
fn icrc152_mint_not_async(
    caller: Principal,
    args: Icrc152MintArgs,
) -> Result<u64, Icrc152MintError> {
    let block_idx = Access::with_ledger_mut(|ledger| {
        if !ledger.feature_flags().icrc152 {
            return Err(Icrc152MintError::GenericError {
                error_code: Nat::from(0_u64),
                message: "ICRC-152 is not enabled".to_string(),
            });
        }
        if !ic_cdk::api::is_controller(&caller) {
            return Err(Icrc152MintError::Unauthorized(
                "caller is not a controller".to_string(),
            ));
        }
        if args.amount == 0_u64 {
            return Err(Icrc152MintError::GenericError {
                error_code: Nat::from(0_u64),
                message: "amount must be greater than 0".to_string(),
            });
        }
        let amount =
            Tokens::try_from(args.amount.clone()).map_err(|_| Icrc152MintError::GenericError {
                error_code: Nat::from(0_u64),
                message: "amount is too large".to_string(),
            })?;
```
