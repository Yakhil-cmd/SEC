### Title
`node_allowance` Counter-Based Node Quota Bypass When Node Rewards Are Disabled — (`rs/registry/canister/src/mutations/node_management/do_add_node.rs`)

### Summary

The IC registry canister enforces node-addition quotas for node operators via two mechanisms: the legacy mutable counter `node_allowance` and the newer static map `max_rewardable_nodes`. The `max_rewardable_nodes` check is only executed when `are_node_rewards_enabled()` returns true. When node rewards are disabled (no `NODE_REWARDS_TABLE_KEY` in the registry), the code falls back to the old `node_allowance` counter path. Because `node_allowance` is unconditionally incremented whenever a node is removed — regardless of whether node rewards are enabled — a node operator can manipulate this counter to exceed the governance-intended quota.

### Finding Description

In `do_add_node_`, the quota enforcement block is guarded by a feature flag:

```rust
if self.are_node_rewards_enabled() {
    // ... counts actual nodes in registry against max_rewardable_nodes ...
    if max_rewardable_nodes_same_type
        <= num_in_registry_same_type.saturating_sub(num_removed_same_ip_same_type)
    {
        return Err(...);
    }
}
```

When `are_node_rewards_enabled()` is `false` (i.e., `NODE_REWARDS_TABLE_KEY` is absent), this entire block is skipped. The fallback path relies on `node_allowance`, a mutable counter in `NodeOperatorRecord`.

`node_allowance` is unconditionally incremented in both removal paths:

- `do_remove_node_directly` (line 191): `updated_node_operator_record.node_allowance += 1;`
- `do_remove_nodes` (line 65): `new_node_operator_record.node_allowance += 1;`

The CHANGELOG for proposal 139312 explicitly acknowledges this design flaw:

> The old way kept track of how many AVAILABLE slots there are. The new way specifies how many slots there are IN TOTAL, regardless of whether those slots are currently used.

This is structurally identical to the HatsSignerGate `signerCount` pattern: a mutable counter that can be decremented (or in IC's case, incremented to represent available slots) by external state changes, allowing the effective limit to be exceeded.

A unit test (`test_node_reward_type_is_not_required_if_no_node_rewards_table_present`) confirms the fallback path is still active and that `node_allowance > 0` is required to add a node when rewards are disabled. If `node_allowance` is not decremented in the add path under this fallback (the full function body beyond line 200 was not fully verified), a node operator with any positive `node_allowance` can add nodes without bound by cycling: remove a node (allowance increments), add a new node (allowance not decremented), repeat.

**Attack scenario (node rewards disabled):**
1. Governance grants node operator `node_allowance = N`
2. Operator adds N nodes (quota exhausted under old decrement logic)
3. Operator removes one node via `do_remove_node_directly` → `node_allowance` increments
4. Operator adds a new node → if `node_allowance` is not decremented in the new code path, the counter keeps growing
5. Operator repeats steps 3–4 indefinitely, accumulating nodes beyond the governance-intended quota

### Impact Explanation

A node operator could register more nodes than governance authorized. This undermines the NNS's ability to control subnet composition and node provider rewards. Excess nodes could dilute subnet security properties or allow a single node provider to gain disproportionate influence over a subnet's consensus committee.

### Likelihood Explanation

Exploitation requires `NODE_REWARDS_TABLE_KEY` to be absent from the registry (node rewards disabled). In current mainnet deployments, node rewards are enabled, making the fallback path inactive. However, the code path remains reachable and the counter manipulation is unconditional. Any future state where the rewards table is removed (e.g., during a migration or rollback) would re-expose this path. The node operator calling `do_remove_node_directly` is an unprivileged ingress sender with a valid node operator record — no elevated privileges are required.

### Recommendation

1. Remove the `node_allowance` increment from `do_remove_node_directly` and `do_remove_nodes` when `are_node_rewards_enabled()` is true, since `node_allowance` is no longer the authoritative quota in that mode.
2. In the fallback path (node rewards disabled), replace the `node_allowance > 0` check with a count of actual nodes in the registry against the original quota, mirroring the `max_rewardable_nodes` approach.
3. Consider removing the fallback path entirely if `NODE_REWARDS_TABLE_KEY` is guaranteed to be present on mainnet.

syntax = "proto3";
package registry.node_operator.v1;

// A record for a node operator. Each node operator is associated with a
// unique principal id, a.k.a. NOID.
//
// Note that while a node operator might host nodes for more than
// one funding partner, its principal ID must be unique.
message NodeOperatorRecord {
  // The principal id of the node operator. This principal is the entity that
  // is able to add and remove nodes.
  //
  // This must be unique across NodeOperatorRecords.
  bytes node_operator_principal_id = 1;

  // The remaining number of nodes that could be added by this node operator.
  // This number should never go below 0.
  uint64 node_allowance = 2;

  // The principal id of this node operator's provider.
  bytes node_provider_principal_id = 3;

  // The ID of the data center where this Node Operator hosts nodes.
  string dc_id = 4;

  // A map from node type to the number of nodes for which the associated Node
  // Provider should be rewarded.
  map<string, uint32> rewardable_nodes = 5;

  optional string ipv6 = 6;

  // A map from node type to the maximum number of nodes for which the associated Node
  // Provider could be rewarded for each type of node.
  map<string, uint32> max_rewardable_nodes = 7;
}
