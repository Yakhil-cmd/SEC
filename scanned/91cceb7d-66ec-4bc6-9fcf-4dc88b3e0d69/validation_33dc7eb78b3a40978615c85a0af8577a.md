### Title
Boundary Node Cache Key Collision Between `QueryV2`/`QueryV3` and `QuerySubnetV3` Due to Missing `request_type` in `RequestContext` Hash/Eq — (`rs/boundary_node/ic_boundary/src/routes.rs`)

---

### Summary

The `RequestContext` struct used as the cache key omits `request_type` from both its `Hash` and `PartialEq` implementations. Because `is_query()` returns `true` for `QueryV2`, `QueryV3`, **and** `QuerySubnetV3`, all three pass the bypasser check and are eligible for caching. Two requests of different types but identical CBOR fields produce the same cache key, causing the boundary node to serve a cached response from one request type to a caller of a different type.

---

### Finding Description

**Root cause — `RequestContext::Hash` and `PartialEq` omit `request_type`:** [1](#0-0) 

The hash covers only `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg`/`http_request`. `request_type` is a field of the struct but is never hashed or compared: [2](#0-1) 

**`QuerySubnetV3` passes the bypasser and enters the cache:**

`is_query()` explicitly includes `QuerySubnetV3`: [3](#0-2) 

The bypasser only skips non-query types: [4](#0-3) 

**The key extractor returns the full `RequestContext` as the key**, relying entirely on the broken `Hash`/`Eq`: [5](#0-4) 

**Collision scenario:**

For `QuerySubnetV3`, the CBOR body carries `canister_id = ic_00` (management canister), `sender`, `method_name`, `ingress_expiry`, and `arg`. A `QueryV2` request to the management canister with identical CBOR fields produces a `RequestContext` that is `Hash`-equal and `PartialEq`-equal to the `QuerySubnetV3` context. The cache treats them as the same entry.

---

### Impact Explanation

The boundary node serves a cached `QueryV2` (or `QueryV3`) response body to a `QuerySubnetV3` caller, or vice versa. The response formats differ: `QueryV2` returns a bare CBOR reply with no certificate; `QueryV3`/`QuerySubnetV3` return a response envelope with a certificate. A `QuerySubnetV3` client receiving a `QueryV2` response gets wrong certified data for subnet-level state reads. Clients that do not independently verify the certificate (e.g., lightweight clients, agents in non-strict mode) may silently accept the wrong response.

---

### Likelihood Explanation

The collision requires matching `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg` across two different request types. For management-canister queries (`canister_id = ic_00`), the same method and argument can legitimately appear in both `QueryV2` and `QuerySubnetV3` requests. An attacker can deliberately craft a `QueryV2` to the management canister to pre-populate the cache, then any `QuerySubnetV3` with matching fields hits the poisoned entry. The `ingress_expiry` field narrows the window but does not prevent the attack since the attacker controls it.

---

### Recommendation

Add `request_type` to both `Hash` and `PartialEq` for `RequestContext`:

```rust
// In Hash impl
self.request_type.hash(state);

// In PartialEq impl
&& self.request_type == other.request_type
``` [6](#0-5)

impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.canister_id.hash(state);
        self.sender.hash(state);
        self.method_name.hash(state);
        self.ingress_expiry.hash(state);

        // Hash http_request if it's present, arg otherwise
        // They're mutually exclusive
        if self.http_request.is_some() {
            self.http_request.hash(state);
        } else {
            self.arg.hash(state);
        }
    }
}

impl PartialEq for RequestContext {
    fn eq(&self, other: &Self) -> bool {
        let r = self.canister_id == other.canister_id
            && self.sender == other.sender
            && self.method_name == other.method_name
            && self.ingress_expiry == other.ingress_expiry;

        // Same as in hash()
        if self.http_request.is_some() {
            r && self.http_request == other.http_request
        } else {
            r && self.arg == other.arg
        }
    }
}
