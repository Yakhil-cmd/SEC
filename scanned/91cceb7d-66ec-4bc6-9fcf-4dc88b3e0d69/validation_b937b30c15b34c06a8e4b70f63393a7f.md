### Title
Missing Minimum Received Amount in `withdraw_eth` Allows Users to Receive Significantly Less ETH Than Expected Due to Dynamic Gas Fee Deduction - (File: `rs/ethereum/cketh/minter/src/main.rs`, `rs/ethereum/cketh/minter/src/tx.rs`)

---

### Summary

The ckETH minter's `withdraw_eth` endpoint immediately burns the user's ckETH at call time, but the actual ETH amount delivered to the Ethereum destination is only determined later — asynchronously — when the minter creates and potentially resubmits the Ethereum transaction. Because `WithdrawalArg` contains no `min_received_amount` field, users have no way to bound how much ETH they will actually receive. When Ethereum gas prices spike between the `withdraw_eth` call and transaction creation (or across multiple resubmissions), the user receives substantially less ETH than they anticipated, with no recourse.

---

### Finding Description

**Step 1 — Immediate burn, deferred fee deduction.**

`withdraw_eth` in `rs/ethereum/cketh/minter/src/main.rs` burns the full `amount` of ckETH from the caller's account immediately:

```rust
match client.burn_from(Account { owner: caller, subaccount: from_subaccount }, amount, ...).await {
    Ok(ledger_burn_index) => { /* queue withdrawal request */ }
```

The only guard at call time is `amount >= minimum_withdrawal_amount` (a static floor). No fee is estimated or deducted at this point. [1](#0-0) 

**Step 2 — Fee deducted asynchronously at transaction creation.**

Later (up to ~6 minutes, or longer during congestion), the minter's background task calls `create_transaction`, which deducts the *current* `max_transaction_fee` from the withdrawal amount:

```rust
let tx_amount = match request.withdrawal_amount.checked_sub(max_transaction_fee) {
    Some(tx_amount) => tx_amount,
    None => return Err(CreateTransactionError::InsufficientTransactionFee { .. }),
};
``` [2](#0-1) 

The fee used here is the live Ethereum gas fee estimate at the moment the minter processes the queue — not the fee at the time the user called `withdraw_eth`.

**Step 3 — Resubmission further reduces the ETH amount.**

If the Ethereum transaction is not mined quickly enough, the minter resubmits it with a higher fee using `ResubmissionStrategy::ReduceEthAmount`. Each resubmission further reduces the ETH value sent to the user:

```rust
ResubmissionStrategy::ReduceEthAmount { withdrawal_amount } => {
    withdrawal_amount.checked_sub(new_tx_price.max_transaction_fee())
        .expect("BUG: withdrawal_amount covers new transaction fee ...")
}
``` [3](#0-2) 

The `resubmit_transaction_price` logic bumps fees by at least 10% per resubmission, compounding the reduction. [4](#0-3) 

**Step 4 — No `min_received_amount` parameter exists.**

`WithdrawalArg` in the Candid interface only accepts `amount`, `recipient`, and `from_subaccount`:

```candid
type WithdrawalArg = record {
    recipient : text;
    amount : nat;
    from_subaccount : opt Subaccount;
};
``` [5](#0-4) 

There is no field for the user to specify the minimum ETH they are willing to receive at the destination. The `eip_1559_transaction_price` query endpoint exists for pre-call estimation, but its result can be stale by the time the minter processes the withdrawal. [6](#0-5) 

**Contrast with ckERC20 design.**

For `withdraw_erc20`, the fee is pre-paid separately via a ckETH burn, and if the live fee exceeds `max_transaction_fee`, the transaction is rejected with `InsufficientTransactionFee` and the user is reimbursed. The ckETH withdrawal path has no equivalent protection. [7](#0-6) 

---

### Impact Explanation

A user who queries `eip_1559_transaction_price`, sees a fee of F, and calls `withdraw_eth(amount)` expecting to receive `amount - F` ETH at their address may instead receive `amount - F'` where `F' >> F` if:

- Ethereum gas prices spike in the ~6-minute window before the minter creates the transaction, or
- The transaction requires one or more resubmissions, each of which further reduces the ETH value by the incremental fee increase.

The documentation itself acknowledges this: *"The exact fee deducted depends on the dynamic Ethereum transaction fees used at the time the transaction was created."* The ckETH is already burned and non-recoverable at this point. The user has no mechanism to abort the withdrawal if the fee exceeds their tolerance. [8](#0-7) 

---

### Likelihood Explanation

Ethereum gas prices are volatile and can spike by multiples within minutes during network congestion. The minter processes withdrawals in batches on a timer, creating a structural gap between when the user commits (burn) and when the fee is determined. The resubmission path (`ReduceEthAmount`) is exercised whenever the Ethereum network is congested — exactly the conditions under which gas prices are highest. Any unprivileged user who calls `withdraw_eth` is exposed to this risk on every withdrawal. [9](#0-8) 

---

### Recommendation

Add an optional `min_received_amount : opt nat` field to `WithdrawalArg`. In `create_transaction`, after computing `tx_amount = withdrawal_amount - max_transaction_fee`, check:

```rust
if let Some(min) = withdrawal_request.min_received_amount {
    if tx_amount < min {
        return Err(CreateTransactionError::BelowMinReceivedAmount { .. });
    }
}
```

If the check fails, cancel the withdrawal and reimburse the user (analogous to the existing `InsufficientTransactionFee` reimbursement path for ckERC20). Apply the same check in the resubmission path before reducing the ETH amount.

---

### Proof of Concept

1. Query `eip_1559_transaction_price()` → observe `max_transaction_fee = F` (e.g., 0.001 ETH).
2. Call `withdraw_eth({ amount = 0.1 ETH, recipient = "0x..." })` → ckETH burned immediately.
3. Ethereum gas prices spike 5× before the minter's next processing cycle.
4. Minter calls `create_transaction` with new `max_transaction_fee = 5F` → `tx_amount = 0.1 - 5F`.
5. Transaction is not mined; minter resubmits with 10% fee bump → `tx_amount = 0.1 - 5.5F`.
6. User receives `0.1 - 5.5F` ETH instead of the expected `0.1 - F` ETH, with no recourse.

The user had no way to specify "revert if I receive less than `0.1 - 2F`", because `WithdrawalArg` provides no such field. [10](#0-9) [11](#0-10)

        }
        AddressValidationError::Blocked(address) => WithdrawalError::RecipientAddressBlocked {

        },

#[derive(Clone, Eq, PartialEq, Debug)]
pub enum ResubmissionStrategy {
    ReduceEthAmount { withdrawal_amount: Wei },
    GuaranteeEthAmount { allowed_max_transaction_fee: Wei },
}

impl ResubmissionStrategy {
    pub fn allowed_max_transaction_fee(&self) -> Wei {
        match self {
            ResubmissionStrategy::ReduceEthAmount { withdrawal_amount } => *withdrawal_amount,
            ResubmissionStrategy::GuaranteeEthAmount {
                allowed_max_transaction_fee,
            } => *allowed_max_transaction_fee,
        }
    }
}

#[derive(Clone, Eq, PartialEq, Debug)]
pub enum ResubmitTransactionError {
    InsufficientTransactionFee {
        allowed_max_transaction_fee: Wei,
        actual_max_transaction_fee: Wei,
    },
}

impl SignedTransactionRequest {
    pub fn resubmit(
        &self,
        new_gas_fee: GasFeeEstimate,
    ) -> Result<Option<Eip1559TransactionRequest>, ResubmitTransactionError> {
        let transaction_request = self.transaction.transaction();
        let last_tx_price = transaction_request.transaction_price();
        let new_tx_price = last_tx_price
            .clone()
            .resubmit_transaction_price(new_gas_fee);
        if new_tx_price == last_tx_price {
            return Ok(None);
        }

        if new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee() {
            return Err(ResubmitTransactionError::InsufficientTransactionFee {
                allowed_max_transaction_fee: self.resubmission.allowed_max_transaction_fee(),
                actual_max_transaction_fee: new_tx_price.max_transaction_fee(),
            });
        }
        let new_amount = match self.resubmission {
            ResubmissionStrategy::ReduceEthAmount { withdrawal_amount } => {
                withdrawal_amount.checked_sub(new_tx_price.max_transaction_fee())
                    .expect("BUG: withdrawal_amount covers new transaction fee because it was checked before")
            }
            ResubmissionStrategy::GuaranteeEthAmount { .. } => transaction_request.amount,
        };
        Ok(Some(Eip1559TransactionRequest {
            max_priority_fee_per_gas: new_tx_price.max_priority_fee_per_gas,
            max_fee_per_gas: new_tx_price.max_fee_per_gas,
            gas_limit: new_tx_price.gas_limit,
            amount: new_amount,
            ..transaction_request.clone()
        }))
    }

    pub fn resubmit_transaction_price(self, new_gas_fee: GasFeeEstimate) -> Self {
        let plus_10_percent = |amount: WeiPerGas| {
            amount
                .checked_add(
                    amount
                        .checked_div_ceil(10_u8)
                        .expect("BUG: must be Some() because divisor is non-zero"),
                )
                .unwrap_or(WeiPerGas::MAX)
        };

        if self.max_fee_per_gas >= new_gas_fee.min_max_fee_per_gas()
            && self.max_priority_fee_per_gas >= new_gas_fee.max_priority_fee_per_gas
        {
            self
        } else {
            // At this point the transaction price needs to be updated
            // which involves a minimum increase of 10% in the max_priority_fee_per_gas.
            // We also need to ensure that the new max_fee_per_gas covers the new max_priority_fee_per_gas,
            // but it would be counter-productive to increase it further than the minimum required.
            // The reason is that any increase in the max_fee_per_gas may render the corresponding transaction
            // not resubmittable due to the user not having enough funds to cover the new transaction price,
            // which could potentially block the minter further. In other words, having a stuck transaction with a higher
            // max_priority_fee_per_gas, is better than having a stuck transaction with a lower max_priority_fee_per_gas,
            // since the first one will go through sooner than the second one when the transaction prices decrease.
            // In case of steep increasing transaction fees, several resubmissions each involving costly operations
            // (various HTTPs outcalls, tECDSA signatures, etc.) might be required, which potentially could be avoided,
            // if one were to increase the max_fee_per_gas more than the minimum required. However,
            // this seems less important than getting the minter unstuck as soon as possible.
            let updated_max_priority_fee_per_gas = plus_10_percent(self.max_priority_fee_per_gas)
                .max(new_gas_fee.max_priority_fee_per_gas);
            let new_gas_fee = GasFeeEstimate {
                max_priority_fee_per_gas: updated_max_priority_fee_per_gas,
                ..new_gas_fee
            };
            let new_max_fee_per_gas = new_gas_fee.min_max_fee_per_gas().max(self.max_fee_per_gas);
            TransactionPrice {
                gas_limit: self.gas_limit,
                max_fee_per_gas: new_max_fee_per_gas,
                max_priority_fee_per_gas: updated_max_priority_fee_per_gas,
            }
        }

```text
Note that the transaction will be made at the cost of the beneficiary meaning that the resulting received amount will be less than the specified withdrawal amount.
The exact fee deducted depends on the dynamic Ethereum transaction fees used at the time the transaction was created.

In more detail, assume that a user calls `withdraw_eth` (after having approved the minter) to withdraw `withdraw_amount` (e.g. 1ckETH) to some address.
Then the minter is going to do the following

. Burn `withdraw_amount` on the ckETH ledger for the IC principal (the caller of `withdraw_eth`).
. Estimate the maximum current cost of a transaction on Ethereum, say `max_tx_fee_estimate`. This `max_tx_fee_estimate` is expected to be large enough to be valid for the few next blocks.
. Issue an Ethereum transaction (via threshold ECDSA) with the value `withdraw_amount - max_tx_fee_estimate`. This requires of course that `withdraw_amount >= max_tx_fee_estimate` and that's why we currently have a conservative minimum value for withdrawals of `30_000_000_000_000_000` wei. This ensures that the minter can always send the transaction to Ethereum if one or several resubmissions are needed if the Ethereum network is congested and fees are increasing rapidly (each resubmission requires an increase of at least 10% of the transaction fee).
