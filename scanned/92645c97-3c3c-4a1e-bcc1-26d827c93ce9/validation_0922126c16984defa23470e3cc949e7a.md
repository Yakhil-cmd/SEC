### Title
Bare `.unwrap()` on `version_for_timestamp_nanoseconds` in `get_rewardable_nodes_per_provider` causes canister trap on public query — (`rs/node_rewards/canister/src/registry_querier.rs`)

---

### Summary

`get_rewardable_nodes_per_provider` calls `version_for_timestamp_nanoseconds(...).unwrap()` without any fallback. `version_for_timestamp_nanoseconds` explicitly returns `Option<RegistryVersion>` and returns `None` when the local `timestamp_to_versions_map` has no entry at or before the requested timestamp. Any unprivileged caller can reach this code path through the public `get_node_providers_rewards_calculation` query endpoint, triggering a canister trap.

---

### Finding Description

**Root cause — bare `.unwrap()` in `get_rewardable_nodes_per_provider`:** [1](#0-0) 

```rust
let registry_version = self
    .version_for_timestamp_nanoseconds(last_unix_timestamp_nanoseconds(date))
    .unwrap();   // panics if None
```

`version_for_timestamp_nanoseconds` returns `None` when the BTreeMap range query finds no entry ≤ the given timestamp: [2](#0-1) 

The same canister already handles this correctly in `get_rewards_table`, using `.ok_or_else(...)` instead of `.unwrap()`: [3](#0-2) 

The inconsistency confirms the `.unwrap()` in `get_rewardable_nodes_per_provider` is an oversight.

**Public entry point — no governance guard on `get_node_providers_rewards_calculation`:** [4](#0-3) 

This is a `#[query]` method with no `panic_if_caller_not_governance()` check. Compare with `get_node_providers_rewards`, which is gated: [5](#0-4) 

**`validate_reward_period` does not close the gap:** [6](#0-5) 

It only checks that `last_day_synced >= to_date` and `to_date < today`. It does **not** verify that the local registry's `timestamp_to_versions_map` contains an entry for the requested date. The registry sync (`schedule_registry_sync`) and the metrics sync (`schedule_metrics_sync`) are independent operations: [7](#0-6) 

If the local registry was first populated on date D but metrics were collected for dates before D (setting `last_day_synced` to a date < D), any query for those earlier dates passes `validate_reward_period` but hits the `.unwrap()` panic.

**Call chain:**

```
get_node_providers_rewards_calculation   [public #[query], no auth]
  → validate_reward_period               [passes if last_day_synced >= date]
  → calculate_rewards_for_date
    → get_rewardable_nodes
      → get_rewardable_nodes_per_provider
          → version_for_timestamp_nanoseconds(...).unwrap()  ← PANIC
``` [8](#0-7) 

---

### Impact Explanation

A canister trap in a non-replicated query call causes that specific query to fail with a trap error returned to the caller. It does not corrupt canister state. However, the trap is deterministic and repeatable: every caller supplying a date before the first registry sync will receive a trap, making the `get_node_providers_rewards_calculation` endpoint permanently broken for that date range until the registry is backfilled. This constitutes a targeted denial-of-service on the rewards calculation query for any date predating the canister's first registry sync.

---

### Likelihood Explanation

The window exists concretely during initial canister deployment: the `HourlySyncTask` first syncs the registry (populating `timestamp_to_versions_map` from the current point forward), then syncs metrics (which may collect historical data for past days, setting `last_day_synced` to a date before the first registry entry). Any caller who queries a date in that gap triggers the panic. The inconsistency with `get_rewards_table` (which uses `.ok_or_else`) confirms this is an unintentional omission, not a deliberate design choice.

---

### Recommendation

Replace the `.unwrap()` with proper error propagation, matching the pattern already used in `get_rewards_table`:

```rust
// rs/node_rewards/canister/src/registry_querier.rs
let registry_version = self
    .version_for_timestamp_nanoseconds(last_unix_timestamp_nanoseconds(date))
    .ok_or_else(|| RegistryClientError::/* appropriate variant */)?;
```

Since `get_rewardable_nodes_per_provider` returns `Result<_, RegistryClientError>`, the `?` operator propagates the error cleanly to the caller without panicking.

---

### Proof of Concept

```rust
#[test]
fn test_get_rewardable_nodes_per_provider_empty_registry_returns_err_not_panic() {
    // Build a RegistryQuerier backed by an empty registry
    // (no entries in timestamp_to_versions_map)
    let store = REGISTRY_STORE.with(|store| store.clone());
    let querier = RegistryQuerier { registry_client: store };

    let date = NaiveDate::from_ymd_opt(2025, 1, 1).unwrap();

    // This currently panics with .unwrap() on None.
    // After the fix it should return Err(...).
    let result = querier.get_rewardable_nodes_per_provider(&date, None);
    assert!(result.is_err(), "Expected Err, got panic");
}
```

    fn validate_reward_period(
        &self,
        from_date: NaiveDate,
        to_date: NaiveDate,
    ) -> Result<(), String> {
        let last_day_synced = self
            .get_last_day_synced()
            .ok_or("Metrics and registry are not synced up")?;

        if last_day_synced < to_date {
            return Err("Metrics and registry are not synced up to to_date".to_string());
        }

        if from_date > to_date {
            return Err("from_date must be before to_date".to_string());
        }

        let today =
            DateTime::from_timestamp_nanos(current_time().as_nanos_since_unix_epoch() as i64)
                .date_naive();

        if to_date >= today {
            return Err("to_date must be earlier than today".to_string());
        }

        Ok(())
    }
