### Title
Cycles Minting Canister Uses Stale ICP/XDR Spot Rate Without Staleness Enforcement in `tokens_to_cycles` - (File: rs/nns/cmc/src/main.rs)

### Summary
The Cycles Minting Canister (CMC) converts ICP to cycles using a cached spot `icp_xdr_conversion_rate` that is updated only via a periodic heartbeat calling the Exchange Rate Canister (XRC). The `tokens_to_cycles` function, called by all three cycle-minting paths (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`), reads this cached rate with no staleness check. If the XRC fails to return a valid rate for an extended period, the CMC silently continues minting cycles at an arbitrarily old rate, allowing any unprivileged user to mint cycles at a rate that no longer reflects the current ICP market price.

### Finding Description

The `tokens_to_cycles` function reads `state.icp_xdr_conversion_rate` and uses it directly for all cycle minting operations:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        ...
    })
}
``` [1](#0-0) 

There is no check on `rate.timestamp_seconds` against the current time. The function only fails if the rate is `None`, which cannot happen in practice because the CMC initializes with a hardcoded default rate.

The rate is updated exclusively via the `canister_heartbeat`, which calls `update_exchange_rate` every 5 minutes under normal conditions: [2](#0-1) 

The `UpdateExchangeRateGuard` enforces a 5-minute refresh interval (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`). On XRC failure, it retries every minute, but if the XRC consistently returns errors (e.g., `StablecoinRateTooFewRates`, `CryptoBaseAssetNotFound`, or insufficient source counts), the rate is never updated: [3](#0-2) 

The `do_set_icp_xdr_conversion_rate` function only enforces that a new rate has a strictly greater timestamp than the current one — it does not enforce any maximum age on the existing stored rate: [4](#0-3) 

All three public cycle-minting entry points call `tokens_to_cycles` with the stale rate:

- `process_top_up` (called by `notify_top_up`) [5](#0-4) 
- `process_create_canister` (called by `notify_create_canister`) [6](#0-5) 
- `process_mint_cycles` (called by `notify_mint_cycles`) [7](#0-6) 

The XRC validation requirements (minimum ICP and CXDR source counts) are a known failure mode that causes the CMC to skip the rate update and retain the old value: [8](#0-7) 

### Impact Explanation

If the ICP market price drops significantly while the XRC is failing (e.g., due to insufficient exchange data sources), the CMC continues minting cycles at the old, higher ICP price. Any unprivileged user calling `notify_top_up` or `notify_create_canister` receives more cycles per ICP than the current market rate warrants. This is a **cycles conservation bug**: cycles are minted in excess of what the burned ICP is worth at current market rates, diluting the value of cycles for the entire network. The inverse (ICP price rising while rate is stale) causes users to receive fewer cycles, which is less severe but still incorrect.

The CMC does have a per-hour cycle minting rate limiter (`base_limiter`), which partially bounds the total damage in any given window, but does not prevent the per-transaction mispricing. [9](#0-8) 

### Likelihood Explanation

The XRC is known to fail when exchange data sources are insufficient, as explicitly tested in the integration test suite: [10](#0-9) 

When the XRC returns an error, the CMC retains the old rate and schedules a retry at the next minute. If the XRC fails for an extended period (e.g., 30+ minutes due to market data unavailability), the stored rate becomes significantly stale. Any unprivileged user can call `notify_top_up` at any time — no special role or privilege is required. The attacker does not need to cause the XRC failure; they only need to observe that the rate is stale (by comparing `icp_xdr_conversion_rate.timestamp_seconds` from `get_icp_xdr_conversion_rate` against the current time) and then submit ICP for conversion during the stale window.

### Recommendation

Add a maximum staleness check inside `tokens_to_cycles` that rejects conversions when the stored rate's `timestamp_seconds` is older than a defined threshold (e.g., 30 minutes or 1 hour) relative to the current canister time. Return a retriable `NotifyError::Other` in that case so callers can retry once the rate is refreshed. This mirrors the pattern already used in governance's `should_refresh_xdr_rate`: [11](#0-10) 

### Proof of Concept

1. The XRC begins returning `StablecoinRateTooFewRates` errors. The CMC heartbeat calls `update_exchange_rate`, which fails with `UpdateExchangeRateError::FailedToRetrieveRate`. The stored `icp_xdr_conversion_rate` is not updated.
2. ICP market price drops 40% over the next hour while the XRC remains unavailable. The CMC's stored rate still reflects the old (higher) price.
3. An unprivileged user queries `get_icp_xdr_conversion_rate` (a public query endpoint) and observes that `timestamp_seconds` is 60+ minutes old.
4. The user sends ICP to the CMC subaccount and calls `notify_top_up`. `process_top_up` calls `tokens_to_cycles`, which reads the stale `icp_xdr_conversion_rate` with no age check and mints ~1.67× more cycles than the current market rate warrants.
5. The user repeats until the hourly `base_limiter` is exhausted, extracting excess cycles at the protocol's expense. [12](#0-11)

    fn should_refresh_xdr_rate(&self) -> bool {
