### Title
UTXO Consolidation Consumes Liquidity Reserved for Pending User Withdrawals Without Accounting for Pending Requests - (File: `rs/bitcoin/ckbtc/minter/src/lib.rs`)

---

### Summary

The ckBTC minter's `consolidate_utxos()` function removes UTXOs from `available_utxos` without first checking whether those UTXOs are needed to fulfill pending user `retrieve_btc` withdrawal requests. This is structurally identical to the reported GeniusVaultCore bug: an internal "rebalancing" operation (UTXO consolidation) consumes the liquidity pool without reserving funds for outstanding user claims, causing those claims to be blocked for the duration of Bitcoin transaction confirmation (hours to days).

---

### Finding Description

`consolidate_utxos()` in `rs/bitcoin/ckbtc/minter/src/lib.rs` is triggered by the canister timer when `available_utxos.len() >= utxo_consolidation_threshold`. Its guard conditions check only:

1. Whether the UTXO count exceeds the threshold
2. Whether the minimum consolidation interval has elapsed
3. Whether a consolidation is already in progress [1](#0-0) 

None of these guards check `pending_retrieve_btc_requests`. Immediately after passing the guards, `select_utxos_to_consolidate()` is called, which destructively removes up to `max_num_inputs_in_transaction` UTXOs from `available_utxos` via `pop_first()`: [2](#0-1) [3](#0-2) 

These UTXOs are then locked inside a `current_consolidate_utxos_request` until the Bitcoin transaction is confirmed on-chain: [4](#0-3) 

Meanwhile, `submit_pending_requests()` calls `build_batch()`, which computes `available_utxos_value` directly from the now-depleted `available_utxos` set: [5](#0-4) 

If the remaining UTXOs are insufficient to cover pending withdrawal amounts, those requests are silently re-queued and cannot be processed until the consolidation transaction is confirmed on Bitcoin. There is no check in `submit_pending_requests()` that prevents it from running while consolidation is in progress, and no check in `consolidate_utxos()` that prevents it from running while there are pending withdrawal requests.

The `CkBtcMinterState` holds both `pending_retrieve_btc_requests` and `available_utxos` as separate fields, but `consolidate_utxos()` reads only the latter: [6](#0-5) 

---

### Impact Explanation

A user who calls `retrieve_btc` has their ckBTC burned on the ledger — an irreversible on-chain action. If UTXO consolidation is triggered immediately after (or concurrently with) the withdrawal request being queued, the UTXOs needed to fulfill the withdrawal are removed from `available_utxos` and locked in the consolidation transaction. The user's withdrawal is then stuck in `pending_retrieve_btc_requests` with no BTC sent and no ckBTC remaining. The delay lasts until the consolidation Bitcoin transaction receives `min_confirmations` confirmations, which can range from tens of minutes to multiple days if the transaction is stuck or the mempool is congested. During this window the user holds neither ckBTC nor BTC.

---

### Likelihood Explanation

The scenario is reachable under normal operational conditions:

- The `utxo_consolidation_threshold` is a configurable parameter (set via upgrade args), and the test suite itself sets it to `1 + max_num_inputs_in_transaction` to trigger consolidation aggressively.
- On mainnet the minter currently holds tens of thousands of UTXOs, making consolidation a routine periodic event.
- Any user can submit a `retrieve_btc` request at any time; the timer-driven consolidation and the withdrawal submission pipeline share the same `available_utxos` pool with no mutual exclusion or reservation logic.
- The upgrade notes for the consolidation feature explicitly acknowledge the tension: "the minter needs to have sufficiently many UTXOs to be able to serve multiple withdrawal requests in parallel." [7](#0-6) 

---

### Recommendation

Before removing UTXOs in `consolidate_utxos()`, compute the total satoshi value required by all entries in `pending_retrieve_btc_requests` and ensure that the UTXOs left in `available_utxos` after consolidation are sufficient to cover that amount. Concretely:

```rust
// Inside consolidate_utxos(), before select_utxos_to_consolidate():
let pending_amount: u64 = read_state(|s| {
    s.pending_retrieve_btc_requests.iter().map(|r| r.amount).sum()
});
let available_value: u64 = read_state(|s| {
    s.available_utxos.iter().map(|u| u.value).sum()
});
if available_value.saturating_sub(pending_amount) < MIN_CONSOLIDATION_VALUE {
    return Err(ConsolidateUtxosError::InsufficientLiquidityForPendingWithdrawals);
}
```

This mirrors the fix recommended in the GeniusVaultCore report: introduce a reservation threshold that accounts for outstanding user claims before any rebalancing/consolidation operation is allowed to proceed.

---

### Proof of Concept

The following sequence reproduces the issue:

1. User A deposits BTC → receives ckBTC (minter now has N UTXOs, N > `utxo_consolidation_threshold`).
2. User A calls `retrieve_btc(amount = total_available_value - epsilon)` → ckBTC is burned; request enters `pending_retrieve_btc_requests`.
3. Timer fires → `consolidate_utxos()` passes all guards (UTXO count still above threshold), calls `select_utxos_to_consolidate()`, removes `max_num_inputs_in_transaction` UTXOs from `available_utxos`.
4. Timer fires → `submit_pending_requests()` calls `build_batch()`: `available_utxos_value` is now less than `req.amount`; the request is re-queued.
5. User A's withdrawal is stuck until the consolidation Bitcoin transaction is confirmed.

The existing test `test_utxo_consolidation_multiple` demonstrates that consolidation successfully removes UTXOs from `available_utxos` and that the count drops by `MAX_NUM_INPUTS_IN_TRANSACTION - 2` per round, confirming the depletion mechanism is real and measurable: [8](#0-7)

// Return UTXOs for consolidation and remove them from available_utxos.
fn select_utxos_to_consolidate(available_utxos: &mut UtxoSet, num_inputs: usize) -> Vec<Utxo> {
    let mut utxos = Vec::with_capacity(num_inputs);
    while utxos.len() < num_inputs {
        if let Some(utxo) = available_utxos.pop_first() {
            utxos.push(utxo);
        } else {
            break;
        }
    }
    utxos
}

    pub fn build_batch(&mut self, max_size: usize) -> BTreeSet<RetrieveBtcRequest> {
        let available_utxos_value = self.available_utxos.iter().map(|u| u.value).sum::<u64>();
        let mut batch = BTreeSet::new();
        let mut tx_amount = 0;
        for req in std::mem::take(&mut self.pending_retrieve_btc_requests) {
            if available_utxos_value < req.amount + tx_amount || batch.len() >= max_size {
                // Put this request back to the queue until we have enough liquid UTXOs.
                self.pending_retrieve_btc_requests.push(req);
            } else {
                tx_amount += req.amount;
                batch.insert(req);
            }
        }
