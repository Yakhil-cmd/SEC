### Title
Unbounded `data.reserve(size)` in `FrameDecoder::decode` Allows Compromised Sandbox to OOM-Abort Replica Controller — (`rs/canister_sandbox/src/frame_decoder.rs`)

---

### Summary

`FrameDecoder::decode` reads an attacker-controlled u64 length prefix from the wire and calls `data.reserve(size)` with no upper-bound guard. The replica controller calls `socket_read_messages` → `FrameDecoder::decode` to read messages from each sandbox process over a Unix-domain socket. A compromised sandbox process can send 8 crafted bytes to trigger an allocation of `usize::MAX` bytes, aborting the replica controller process.

---

### Finding Description

In `rs/canister_sandbox/src/frame_decoder.rs`, the `decode` method reads an 8-byte u64 length prefix and immediately uses it as a reservation size: [1](#0-0) 

```rust
let size = data.get_u64();                    // attacker-controlled
self.state = FrameDecoderState::Length(size);
// ...
let size: usize = *size as usize;             // cast, no bounds check
if data.len() < size {
    data.reserve(size);                       // ← no upper-bound guard
    return None;
}
```

There is no maximum frame size check before `data.reserve(size)`. Sending `0xFFFFFFFFFFFFFFFF` as the 8-byte prefix causes `size as usize = usize::MAX`, and `data.reserve(usize::MAX)` invokes the global allocator with an impossible request. Rust's allocator calls `handle_alloc_error` → `abort()`, killing the process.

The controller side **does** call `socket_read_messages` (and therefore `FrameDecoder::decode`) to read `SandboxToController` messages from each sandbox child process: [2](#0-1) 

The same pattern exists for the launcher IPC thread: [3](#0-2) 

The socket is a `UnixStream::pair()` — the sandbox process holds one end as fd 3 and can write arbitrary bytes to it: [4](#0-3) 

---

### Impact Explanation

A compromised sandbox process (e.g., a malicious canister exploiting a Wasm executor bug to achieve code execution inside the sandbox) can write 8 bytes — `FF FF FF FF FF FF FF FF` — to its socket fd. The controller's `CanisterSandboxIPC` thread calls `FrameDecoder::decode`, hits `data.reserve(usize::MAX)`, and the replica process aborts. This halts the replica node. If the same canister is executed across multiple nodes in a subnet simultaneously, all affected nodes crash, breaking consensus liveness.

---

### Likelihood Explanation

The canister sandbox is explicitly a security boundary: the design assumes the sandbox process may be compromised and the controller must remain safe. The IPC socket is the only communication channel and the controller unconditionally trusts the length prefix. No authentication, HMAC, or size cap exists on incoming frames. Exploiting this requires only writing 8 bytes to a file descriptor the sandbox process legitimately holds.

---

### Recommendation

Add a maximum frame size guard before `data.reserve(size)` in `FrameDecoder::decode`:

```rust
FrameDecoderState::Length(size) => {
    let size: usize = *size as usize;
    const MAX_FRAME_SIZE: usize = 256 * 1024 * 1024; // e.g. 256 MiB
    if size > MAX_FRAME_SIZE {
        panic!("frame size {} exceeds maximum", size); // or return Err
    }
    if data.len() < size {
        data.reserve(size);
        return None;
    }
```

The guard should be placed at `rs/canister_sandbox/src/frame_decoder.rs` line 45, before the `data.reserve(size)` call. [5](#0-4) 

---

### Proof of Concept

```rust
// Fuzz target: feed crafted bytes into FrameDecoder
let mut decoder = FrameDecoder::<SomeMessage>::new();
let mut buf = BytesMut::new();
// Write a u64 length of usize::MAX in big-endian
buf.extend_from_slice(&0xFFFFFFFFFFFFFFFFu64.to_be_bytes());
// This call aborts the process via OOM handler:
decoder.decode(&mut buf);
```

On a real node: write `\xFF\xFF\xFF\xFF\xFF\xFF\xFF\xFF` to fd 3 from within a compromised sandbox process. The controller's `CanisterSandboxIPC` thread aborts immediately.
