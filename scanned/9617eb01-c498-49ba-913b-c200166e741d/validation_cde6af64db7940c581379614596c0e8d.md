### Title
`inspect_message`-only authorization in rate-limit canister bypassed by inter-canister calls - (File: `rs/boundary_node/rate_limits/canister/canister.rs`)

---

### Summary

The rate-limit canister places its entire authorization model inside the `canister_inspect_message` hook. On the Internet Computer, `inspect_message` is **only invoked for ingress messages** (user-originated calls); it is never called for inter-canister calls. Any canister on the IC can therefore call the privileged `add_config` and `disclose_rules` update methods without restriction.

---

### Finding Description

The `inspect_message` hook in `rs/boundary_node/rate_limits/canister/canister.rs` is the sole authorization gate for the two privileged update methods:

```
const UPDATE_METHODS: [&str; 2] = ["add_config", "disclose_rules"];
``` [1](#0-0) 

The hook checks that `caller_id == authorized_principal` before accepting any call to `add_config` or `disclose_rules`. For all other callers it traps with `"message_inspection_failed: unauthorized caller"`.

However, the IC Interface Specification explicitly states that `canister_inspect_message` is **not** invoked for inter-canister calls — it is a pre-consensus filter for ingress messages only. The actual `add_config` and `disclose_rules` implementations carry no independent authorization check. A malicious canister that calls either method via an inter-canister update call will reach the implementation directly, with `inspect_message` never executed.

The registry canister test suite explicitly documents this exact pitfall:

```rust
// Going through a real canister (rather than an ingress message)
// ensures the access control cannot be bypassed by, e.g., only guarding
// ingress messages in `inspect_message`.
``` [2](#0-1) 

The integration test for the rate-limit canister only exercises ingress-path rejection and never tests the inter-canister path: [3](#0-2) 

The salt-sharing canister exhibits the same structural pattern — `inspect_message` as the only guard: [4](#0-3) 

---

### Impact Explanation

An unprivileged canister can:

1. Call `add_config` via inter-canister call and inject arbitrary rate-limit rules (including rules that suppress legitimate traffic or expose sensitive incident IDs).
2. Call `disclose_rules` via inter-canister call and force-disclose incident information that the authorized operator intended to keep hidden.

Both actions mutate persistent canister state and affect the behavior of boundary nodes that consume the rate-limit configuration. The impact is a **governance authorization bypass** leading to unauthorized state mutation in a production boundary-node canister.

---

### Likelihood Explanation

The attack requires only the ability to deploy a canister on any subnet and make an inter-canister call — capabilities available to any IC user. No privileged role, leaked key, or social engineering is needed. The call path is a single inter-canister update call, making exploitation trivial once the gap is known.

---

### Recommendation

Add an explicit caller check at the top of the `add_config` and `disclose_rules` update handlers themselves, independent of `inspect_message`. For example:

```rust
#[update]
fn add_config(config: InputConfig) -> AddConfigResponse {
    let caller = ic_cdk::api::caller();
    with_canister_state(|state| {
        if Some(caller) != state.get_authorized_principal() {
            ic_cdk::trap("unauthorized caller");
        }
    });
    // ... existing logic
}
```

This mirrors the pattern used correctly in the registry canister, where authorization is checked inside the handler body and separately verified via inter-canister call tests. [5](#0-4) 

---

### Proof of Concept

1. Deploy an attacker canister `A` on any subnet.
2. From `A`, issue an inter-canister update call to the rate-limit canister's `add_config` method with an arbitrary `InputConfig`.
3. Because `inspect_message` is not invoked for inter-canister calls, the IC runtime delivers the call directly to the `add_config` implementation.
4. The implementation has no independent authorization check, so the call succeeds and the malicious config is written to stable state.
5. Boundary nodes polling `get_config` will subsequently receive and apply the attacker-injected rate-limit rules.

const UPDATE_METHODS: [&str; 2] = ["add_config", "disclose_rules"];
const REPLICATED_QUERY_METHOD: &str = "get_config";

// Inspect the ingress messages in the pre-consensus phase and reject early, if the conditions are not met
#[inspect_message]
fn inspect_message() {
    // In order for this hook to succeed, accept_message() must be invoked.
    let caller_id: Principal = ic_cdk::api::caller();
    let called_method = ic_cdk::api::call::method_name();

    let (has_full_access, has_full_read_access) = with_canister_state(|state| {
        let authorized_principal = state.get_authorized_principal();
        (
            Some(caller_id) == authorized_principal,
            state.is_api_boundary_node_principal(&caller_id),
        )
    });

    if called_method == REPLICATED_QUERY_METHOD {
        if has_full_access || has_full_read_access {
            ic_cdk::api::call::accept_message();
        } else {
            ic_cdk::api::trap(
                "message_inspection_failed: method call is prohibited in the current context",
            );
        }
    } else if UPDATE_METHODS.contains(&called_method.as_str()) {
        if has_full_access {
            ic_cdk::api::call::accept_message();
        } else {
            ic_cdk::api::trap("message_inspection_failed: unauthorized caller");
        }
    } else {
        // All others calls are rejected
        ic_cdk::api::trap(
            "message_inspection_failed: method call is prohibited in the current context",
        );
    }
}
