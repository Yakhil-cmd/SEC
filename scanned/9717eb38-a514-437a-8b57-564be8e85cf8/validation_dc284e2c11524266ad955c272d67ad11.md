### Title
Unbounded Linear Scan Over `processed_withdrawal_requests` in `withdrawal_status` Query Causes Permanent Instruction-Limit DoS — (`File: rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

---

### Summary

The ckETH minter canister's `withdrawal_status` query endpoint iterates linearly over the entire `processed_withdrawal_requests` `BTreeMap` on every call. This map is never pruned and grows monotonically with every withdrawal ever processed. As the protocol accumulates withdrawals through normal operation, the query's instruction cost grows without bound, eventually causing it to permanently exceed the IC query instruction limit (5 billion instructions) and become unavailable to all callers.

---

### Finding Description

`EthTransactions::withdrawal_status` in `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` performs two full linear scans per call:

1. Over `pending_withdrawal_requests` (a `VecDeque<WithdrawalRequest>`)
2. Over `processed_withdrawal_requests` (a `BTreeMap<LedgerBurnIndex, WithdrawalRequest>`) [1](#0-0) 

The `pending_withdrawal_requests` queue is bounded in practice because the minter's timer drains it in batches. However, `processed_withdrawal_requests` is **never pruned**: every withdrawal request for which a transaction has been created is inserted and never removed. [2](#0-1) 

The struct definition confirms `processed_withdrawal_requests` is a plain `BTreeMap` with no eviction policy: [3](#0-2) 

The public query endpoint `withdrawal_status` in `rs/ethereum/cketh/minter/src/main.rs` is callable by any unprivileged user and directly invokes this O(n) scan: [4](#0-3) 

The DID interface confirms this is an unrestricted query: [5](#0-4) 

Additionally, `transaction_status` — called from the `retrieve_eth_status` update endpoint — also performs a linear scan over `pending_withdrawal_requests` on every invocation: [6](#0-5) [7](#0-6) 

And `record_withdrawal_request` performs an O(n) duplicate check over `pending_withdrawal_requests` on every new withdrawal submission: [8](#0-7) 

---

### Impact Explanation

The IC query instruction limit is 5 billion instructions. [9](#0-8) 

Each element visited in the `withdrawal_status` linear scan costs a small but fixed number of instructions. As `processed_withdrawal_requests` grows with every withdrawal ever processed on mainnet, the per-call cost increases monotonically. Once the map is large enough, every call to `withdrawal_status` will trap with `CanisterInstructionLimitExceeded`, permanently disabling the endpoint for all callers — including legitimate users checking their own withdrawal status. The `retrieve_eth_status` update endpoint (40B limit) is less immediately threatened but follows the same growth trajectory.

---

### Likelihood Explanation

The ckETH minter is a production canister processing real withdrawals continuously. `processed_withdrawal_requests` grows with every withdrawal ever finalized — this is normal protocol operation, not an attack. No adversarial action is required; the endpoint degrades automatically over time. An attacker wishing to accelerate the degradation can submit many minimum-value withdrawals (each burning the minimum ckETH amount), but even without deliberate attack, the endpoint will eventually fail as cumulative withdrawal volume grows. The `withdrawal_status` query has no pagination, no size cap, and no pruning of historical entries.

---

### Recommendation

1. **Prune `processed_withdrawal_requests`**: Remove entries once a withdrawal reaches a terminal state (finalized and reimbursement settled). Store only the minimal status needed for historical queries in a separate, bounded structure.
2. **Index by search key**: Instead of iterating all entries, maintain secondary indexes (e.g., `BTreeMap<Address, BTreeSet<LedgerBurnIndex>>` for `ByRecipient`) so `withdrawal_status` can perform O(log n) lookups rather than O(n) full scans.
3. **Add a result cap**: Limit the number of results returned per call and require callers to paginate, bounding per-call instruction cost regardless of total state size.
4. **Replace the `pending_withdrawal_requests` duplicate check**: Use a `BTreeSet<LedgerBurnIndex>` shadow index so `record_withdrawal_request` performs an O(log n) membership test instead of an O(n) linear scan.

---

### Proof of Concept

The following describes the reachable path:

1. Over time (or via deliberate minimum-value withdrawals), `processed_withdrawal_requests` accumulates N entries.
2. Any caller invokes `withdrawal_status(ByRecipient("0x..."))` — an unrestricted query.
3. `EthTransactions::withdrawal_status` iterates all N entries in `processed_withdrawal_requests.values()` regardless of how many match.
4. When N is large enough, the query traps with instruction limit exceeded, permanently disabling the endpoint.

The same path applies to `retrieve_eth_status` (update, 40B limit) via `transaction_status` scanning `pending_withdrawal_requests`, and to `withdraw_eth` (update) via `record_withdrawal_request` scanning `pending_withdrawal_requests` on every new submission. [10](#0-9) [11](#0-10)

#[query]
async fn withdrawal_status(parameter: WithdrawalSearchParameter) -> Vec<WithdrawalDetail> {
    use transactions::WithdrawalRequest::*;
    let parameter = transactions::WithdrawalSearchParameter::try_from(parameter).unwrap();
    read_state(|s| {
        s.eth_transactions
            .withdrawal_status(&parameter)
            .into_iter()
            .map(|(request, status, tx)| WithdrawalDetail {
                withdrawal_id: *request.cketh_ledger_burn_index().as_ref(),
                recipient_address: request.payee().to_string(),
                token_symbol: match request {
                    CkEth(_) => CkTokenSymbol::cketh_symbol_from_state(s).to_string(),
                    CkErc20(r) => s
                        .ckerc20_tokens
                        .get_alt(&r.erc20_contract_address)
                        .unwrap()
                        .to_string(),
                },
                withdrawal_amount: match request {
                    CkEth(r) => r.withdrawal_amount.into(),
                    CkErc20(r) => r.withdrawal_amount.into(),
                },
                max_transaction_fee: match (request, tx) {
                    (CkEth(_), None) => None,
                    (CkEth(r), Some(tx)) => {
                        r.withdrawal_amount.checked_sub(tx.amount).map(|x| x.into())
                    }
                    (CkErc20(r), _) => Some(r.max_transaction_fee.into()),
                },
                from: request.from(),
                from_subaccount: request
                    .from_subaccount()
                    .cloned()
                    .map(LedgerSubaccount::to_bytes),
                status,
            })
            .collect()
    })
}
