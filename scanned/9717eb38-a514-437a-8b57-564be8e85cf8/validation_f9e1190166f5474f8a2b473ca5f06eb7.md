### Title
Unescaped Memo Content Enables Markdown Injection into ICRC-21 GenericDisplayMessage — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The `add_memo` function in `responses.rs` interpolates attacker-controlled memo bytes directly into a Markdown-formatted `GenericDisplayMessage` string without escaping backticks or newlines. A malicious dApp can craft a UTF-8 memo containing a backtick that closes the surrounding code span early, causing subsequent content to render as real Markdown — including fake bold fields like `**Amount:**` or `**To:**` — in any wallet that renders `GenericDisplayMessage` as Markdown per the ICRC-21 spec.

---

### Finding Description

In `add_memo`, when the memo bytes are valid UTF-8, the decoded string is placed verbatim inside a single-backtick code span:

```rust
// packages/icrc-ledger-types/src/icrc21/responses.rs, line 294
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
``` [1](#0-0) 

No escaping of backticks or newlines is performed on `memo_str`. In CommonMark, a single-backtick code span ends at the **next** single backtick. If `memo_str` begins with a backtick (e.g., `` ` ``), the opening backtick and the first character of `memo_str` form a double-backtick sequence (`` `` ``). Since no matching `` `` `` follows, CommonMark treats them as literal characters and the remainder of `memo_str` is parsed as raw Markdown.

**Concrete injection example:**

memo bytes (valid UTF-8) = `` `\n\n**Amount:** `999999 ICP ``

Resulting `GenericDisplayMessage` fragment:
```
**Memo:**
``

**Amount:** `999999 ICP`
```

The wallet renders this as a bold `**Amount:**` field followed by a code-span `999999 ICP` — visually indistinguishable from the real `**Amount:**` field generated by `add_amount`. [2](#0-1) 

The call path from a public ingress is direct:

1. Any caller submits `icrc21_canister_call_consent_message` with a `TransferArg` whose `memo` field contains the crafted bytes.
2. `build_icrc21_consent_info` decodes the arg and passes `GenericMemo::Icrc1Memo(memo.0)` to the builder.
3. `ConsentMessageBuilder::build` calls `message.add_memo(memo)` unconditionally if a memo is present.
4. `add_memo` decodes the bytes as UTF-8 and interpolates without escaping. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

The `GenericDisplayMessage` is the primary human-readable consent surface for ICRC-21. The ICRC-21 standard specifies that wallets **should** render it as Markdown. A malicious dApp can inject fake `**Amount:**`, `**To:**`, or `**Fees:**` fields that are visually identical to the real ones generated by the ledger. Because the memo section is appended last, the injected fields appear after the real ones; a user who reads only the bottom of a long consent message (or a wallet UI that truncates the top) may act on the injected values instead of the real ones. This breaks the core security invariant of ICRC-21: that the consent message faithfully and exclusively represents the transaction the user is authorizing.

---

### Likelihood Explanation

- Any unprivileged caller can submit a `ConsentMessageRequest` with a crafted memo via the public `icrc21_canister_call_consent_message` update endpoint.
- The memo field in `TransferArg` / `TransferFromArgs` / `ApproveArgs` accepts arbitrary bytes up to 32 bytes (ICRC-1 limit), which is sufficient to inject a fake field.
- Wallets implementing ICRC-21 (Plug, NFID, hardware wallets) are expected to render `GenericDisplayMessage` as Markdown.
- The injected content appears after the real fields, so a user who reads the full message carefully will see the real values first. However, wallet UIs that paginate, truncate, or scroll the consent message may present the injected content more prominently.

---

### Recommendation

Escape all Markdown control characters in `memo_str` before interpolation. At minimum, replace backticks (`` ` ``) and newlines (`\n`, `\r`) in `memo_str`:

```rust
let safe_memo = memo_str
    .replace('`', "\\`")
    .replace('\n', " ")
    .replace('\r', " ");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, hex-encode all memo bytes unconditionally for `GenericDisplayMessage` (as is already done for non-UTF-8 bytes), since memo bytes have no semantic meaning that requires human-readable display.

---

### Proof of Concept

```rust
// Unit test demonstrating injection
let memo_bytes = b"`\n\n**Amount:** `999999 ICP".to_vec();
let memo = GenericMemo::Icrc1Memo(memo_bytes);
let mut msg = ConsentMessage::GenericDisplayMessage(String::new());
msg.add_intent(Icrc21Function::Transfer, Some("ICP".to_string()));
msg.add_amount(Some(Nat::from(1u64)), 8, &"ICP".to_string()).unwrap();
msg.add_memo(memo);

if let ConsentMessage::GenericDisplayMessage(text) = &msg {
    // The injected bold field appears in the rendered output
    assert!(
        !text.contains("\n\n**Amount:** `999999 ICP`"),
        "FAIL: injected fake Amount field present in consent message:\n{text}"
    );
}

    pub fn add_memo(&mut self, memo: GenericMemo) {
        match memo {
            GenericMemo::Icrc1Memo(memo) => {
                // Check if the memo is a valid UTF-8 string and display it as such if it is.
                let memo_str = match std::str::from_utf8(memo.as_slice()) {
                    Ok(valid_str) => valid_str.to_string(),
                    Err(_) => hex::encode(memo.as_slice()),
                };
                match self {
                    ConsentMessage::GenericDisplayMessage(message) => {

                    }
                    ConsentMessage::FieldsDisplayMessage(fields_display) => fields_display
                        .fields
                        .push(("Memo".to_string(), Value::Text { content: memo_str })),
                }
