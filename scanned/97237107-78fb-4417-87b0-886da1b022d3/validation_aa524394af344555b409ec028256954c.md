### Title
CMC `purge_old_notifications` Silently Drops In-Flight `Processing` Entries, Permanently Locking User ICP - (File: `rs/nns/cmc/src/main.rs`)

### Summary

The Cycles Minting Canister (CMC) maintains a `blocks_notified` map bounded at `MAX_NOTIFY_HISTORY = 1_000_000` entries. When the map is full, `purge_old_notifications` evicts the oldest entries **regardless of their `NotificationStatus`**, including entries in `Processing` state that represent in-flight async ledger calls. If such an entry is evicted while the async call is still pending and the call later returns a transient error, the entry is deleted and `last_purged_notification` is already advanced past that block index, making the user's ICP permanently unrecoverable.

### Finding Description

The CMC's `purge_old_notifications` function is called at the start of every `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` invocation:

```rust
// rs/nns/cmc/src/main.rs
const MAX_NOTIFY_HISTORY: usize = 1_000_000;
const MAX_NOTIFY_PURGE: usize = 100_000;

fn purge_old_notifications(&mut self, max_history: usize) {
    let mut last_purged = 0;
    let mut cnt = 0;
    while self.blocks_notified.len() > max_history && cnt < MAX_NOTIFY_PURGE {
        let block_height = *self.blocks_notified.iter().next().unwrap().0;
        self.blocks_notified.remove(&block_height);   // ← removes ANY status, including Processing
        last_purged = block_height;
        cnt += 1;
    }
    self.last_purged_notification = last_purged.max(self.last_purged_notification);
}

The `Processing` variant signals that an async inter-canister call is in-flight:

```rust
pub enum NotificationStatus {
    Processing,   // ← in-flight async call
    NotifiedTopUp(Result<Cycles, NotifyError>),
    ...
}
``` [2](#0-1) 

The attack/failure path in `notify_top_up` is:

1. User calls `notify_top_up(block_index = X)`. `purge_old_notifications` runs, then `blocks_notified[X]` is set to `Processing`.
2. CMC awaits `process_top_up` (an async call to the ledger/management canister).
3. While awaiting, another caller triggers `notify_top_up(block_index = Y > X)`. If `blocks_notified.len() > MAX_NOTIFY_HISTORY`, `purge_old_notifications` removes block index `X` (the oldest entry) and advances `last_purged_notification` to `X`.
4. The original async call returns with a transient error. The post-await handler removes the entry:

```rust
with_state_mut(|state| {
    state.blocks_notified.insert(block_index, NotificationStatus::NotifiedTopUp(result.clone()));
    if is_transient_error(&result) {
        state.blocks_notified.remove(&block_index);  // entry deleted
    }
});
``` [3](#0-2) 

5. User retries `notify_top_up(X)`. The guard at the top of the function fires:

```rust
if block_index <= state.last_purged_notification {
    return Some(Err(NotifyError::TransactionTooOld(
        state.last_purged_notification + 1,
    )));
}
``` [4](#0-3) 

The user receives `TransactionTooOld` permanently. Their ICP, which was transferred to the CMC's subaccount before calling `notify_top_up`, is now locked with no recovery path. The CMC exposes no function to return ICP from a subaccount to its owner, and the automatic-refund path only applies to transactions with non-meaningful memos.

The same flaw exists in `notify_create_canister` and `notify_mint_cycles`, which share the identical `purge_old_notifications` + `last_purged_notification` guard pattern. [5](#0-4) [6](#0-5) 

### Impact Explanation

A user who sent ICP to the CMC's subaccount (with `MEMO_TOP_UP_CANISTER`, `MEMO_CREATE_CANISTER`, or `MEMO_MINT_CYCLES`) and whose `notify_*` call was in `Processing` state when the purge ran will permanently lose that ICP. There is no on-chain recovery mechanism: the ICP sits in the CMC's subaccount, inaccessible to the user. This is a direct ledger conservation violation — ICP is neither minted into cycles nor returned to the sender.

### Likelihood Explanation

The `blocks_notified` map is bounded at 1,000,000 entries. The CMC is a high-traffic system canister that has been live since mainnet genesis. Once the map reaches capacity (which happens in steady-state operation), every new `notify_*` call evicts the oldest entry. A `Processing` entry becomes the eviction target whenever it holds the smallest block index in the map — a realistic condition when many concurrent notifications are in-flight. An adversary can accelerate this by submitting many small ICP transactions to the CMC's subaccount (each costing only the ICP transfer fee) to flood `blocks_notified` and force eviction of a targeted victim's `Processing` entry.

### Recommendation

`purge_old_notifications` must skip entries whose status is `Processing`:

```rust
fn purge_old_notifications(&mut self, max_history: usize) {
    let mut last_purged = 0;
    let mut cnt = 0;
    while self.blocks_notified.len() > max_history && cnt < MAX_NOTIFY_PURGE {
        let block_height = *self.blocks_notified.iter().next().unwrap().0;
        // Do NOT evict in-flight entries
        if self.blocks_notified[&block_height] == NotificationStatus::Processing {
            break;
        }
        self.blocks_notified.remove(&block_height);
        last_purged = block_height;
        cnt += 1;
    }
    self.last_purged_notification = last_purged.max(self.last_purged_notification);
}
```

Alternatively, track in-flight block indices in a separate set and exclude them from the eviction candidate set. Additionally, `last_purged_notification` should only advance past block indices that were fully processed (not merely evicted), so that users whose entries were evicted mid-flight can still retry.

### Proof of Concept

```
State: blocks_notified has 1_000_000 entries; smallest key = block_index X.
blocks_notified[X] = Processing  (set by User A's notify_top_up, async call in-flight)

Step 1: User B calls notify_top_up(block_index = Y, Y > X).
        → purge_old_notifications fires because len > MAX_NOTIFY_HISTORY
        → blocks_notified.remove(X)
        → last_purged_notification = X

Step 2: User A's async call (process_top_up) returns Err(transient).
        → blocks_notified.insert(X, NotifiedTopUp(Err(transient)))
        → is_transient_error → blocks_notified.remove(X)

Step 3: User A retries notify_top_up(X).
        → block_index X <= last_purged_notification
        → returns Err(TransactionTooOld(X+1))

Result: User A's ICP (sent to CMC subaccount before step 1) is permanently locked.
        No on-chain recovery path exists.
``` [7](#0-6) [1](#0-0) [8](#0-7)
