### Title
IP-Based `add_node` Rate Limiter Resets on Canister Upgrade Due to Heap-Only Storage — (`rs/registry/canister/src/rate_limits.rs`)

### Summary

The `ADD_NODE_IP_RATE_LIMITER` is stored exclusively in heap memory (`InMemoryRateLimiter<String>` backed by a `BTreeMap`). On every registry canister upgrade, the IC wipes heap memory, resetting this limiter to its initial state. The operator/provider rate limiters use `StableMemoryCapacityStorage` and survive upgrades; the IP limiter does not. No `post_upgrade` logic reconstructs it from stable state.

### Finding Description

In `rs/registry/canister/src/rate_limits.rs`, the three rate limiters are declared as `thread_local!`:

- `NODE_PROVIDER_RATE_LIMITER` — uses `RateLimiter::new_stable(...)` with `StableMemoryCapacityStorage` → **survives upgrades**
- `NODE_OPERATOR_RATE_LIMITER` — uses `RateLimiter::new_stable(...)` with `StableMemoryCapacityStorage` → **survives upgrades**
- `ADD_NODE_IP_RATE_LIMITER` — uses `InMemoryRateLimiter::new_in_memory(...)` with `InMemoryCapacityStorage` (a plain `BTreeMap`) → **reset on every upgrade** [1](#0-0) 

The code comment explicitly acknowledges this: *"Stored in heap memory (not stable memory)."*

`InMemoryRateLimiter<K>` is a type alias for `RateLimiter<K, InMemoryCapacityStorage<K>>`, where `InMemoryCapacityStorage` is a plain `BTreeMap` with no stable-memory backing. [2](#0-1) [3](#0-2) 

`canister_post_upgrade` restores registry data from stable storage but performs no reconstruction of the IP rate limiter state. [4](#0-3) 

In `do_add_node_`, the IP reservation is checked at line 89–90: [5](#0-4) 

After an upgrade, `ADD_NODE_IP_RATE_LIMITER` starts fresh with `max_capacity = ADD_NODE_IP_MAX_SPIKE = 7`, meaning up to 7 nodes can be registered at the same IP immediately after any upgrade. [6](#0-5) 

### Impact Explanation

An attacker who has exhausted the IP rate limit can bypass it by waiting for any legitimate registry canister upgrade (which occurs routinely as part of IC development — no governance majority needed by the attacker). After the upgrade, the IP limiter resets and the attacker can register up to 7 nodes at the same IP again. Combined with the same-IP replacement logic in `do_add_node`, this could be used to repeatedly re-register nodes at the same IP address.

The impact is **bounded** by the other guards that do survive upgrades:
- `NODE_OPERATOR_RATE_LIMITER` and `NODE_PROVIDER_RATE_LIMITER` are in stable memory and are not reset.
- `node_allowance` and `max_rewardable_nodes` in the node operator record constrain total registrations.

So the attacker cannot register unlimited nodes — they are still constrained by their operator/provider rate limits and allowance. The IP rate limit bypass allows them to exceed the 1-node/day/IP policy in bursts around upgrade events.

### Likelihood Explanation

Registry canister upgrades happen regularly. The attacker does not need to submit or pass any governance proposal — they only need to observe when an upgrade occurs (which is publicly visible on-chain) and act immediately after. The precondition of "influencing a governance proposal" stated in the question is actually unnecessary; passive timing against routine upgrades suffices.

### Recommendation

Migrate `ADD_NODE_IP_RATE_LIMITER` to use `StableMemoryCapacityStorage` (as the operator and provider limiters already do), backed by a dedicated virtual memory region from the memory manager. Alternatively, reconstruct the IP rate limiter state from the existing node records in `canister_post_upgrade` by scanning all registered node IPs and their registration timestamps.

### Proof of Concept

1. Exhaust the IP rate limit for IP `X` (register 7 nodes, consuming `max_capacity`).
2. Observe a registry canister upgrade on mainnet (or simulate one in a state-machine test).
3. After upgrade, call `add_node` with the same IP `X`.
4. The call succeeds — `ADD_NODE_IP_RATE_LIMITER` has been reset to zero usage, so `try_reserve_add_node_capacity` returns `Ok`.
5. The operator/provider rate limits (in stable memory) still apply and will eventually block further registrations, but the IP-based limit has been bypassed.

    // Registry data migrations should be implemented as follows:
    let mutation_batches_due_to_data_migrations = {
        let mut total_batches = 0;

        let mutations = fix_node_operators_corrupted(registry);
        if !mutations.is_empty() {
            registry.maybe_apply_mutation_internal(mutations);
            total_batches += 1;
        }

        let mutations = fix_vetkd_pre_signatures_field(registry);
        if !mutations.is_empty() {
            registry.maybe_apply_mutation_internal(mutations);
            total_batches += 1;
        }

        let mutations = convert_type1dot1_nodes_to_type4dot5(registry);
        if !mutations.is_empty() {
            registry.maybe_apply_mutation_internal(mutations);
            total_batches += 1;
        }

        total_batches

    //
    // When there are no migrations, `mutation_batches_due_to_data_migrations` should be set to `0`.

    registry.check_global_state_invariants(&[]);

    recertify_registry(registry);
