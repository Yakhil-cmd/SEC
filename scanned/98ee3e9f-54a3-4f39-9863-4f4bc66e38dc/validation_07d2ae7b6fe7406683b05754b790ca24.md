### Title
Unbounded State Sync Restart Loop via Byzantine Peer Returning HTTP 204 — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

### Summary
A single Byzantine subnet peer can force a victim replica into an infinite state-sync start/stop cycle by advertising a valid state height and then returning HTTP 204 (No Content) for every chunk request. Each cycle creates and immediately destroys an `IncompleteState` object, wastes CPU and filesystem I/O, and—critically—prevents honest peers from ever making download progress on that height.

### Finding Description
`OngoingStateSync::run` exits its event loop whenever `self.active_downloads.is_empty()` (line 173–176). The only way a peer is removed from `active_downloads` is via `handle_downloaded_chunk_result` on `DownloadChunkError::NoContent` or `RequestError`. A Byzantine peer that always responds with HTTP 204 will be evicted after all `PARALLEL_CHUNK_DOWNLOADS` (10) in-flight tasks complete, leaving `active_downloads` empty and causing an immediate break with no retry, no backoff, and no minimum-time guard.

When the `OngoingStateSync` task finishes, the `Arc<Mutex<Box<dyn Chunkable>>>` tracker is dropped. `IncompleteState::drop` calls `state_sync_refs.active.write().take()`, clearing the only guard that prevents `maybe_start_state_sync` from returning `Some` again. The next advert from the same Byzantine peer (or any peer) causes `handle_advert` to call `maybe_start_state_sync` again, which succeeds, and the cycle repeats indefinitely.

There is no rate limit on advert processing, no per-peer failure counter, no minimum state-sync lifetime, and no detection of repeated start/stop cycles for the same height.

### Impact Explanation
- **CPU/IO waste**: Each cycle allocates a scratchpad directory, spawns 10 async RPC tasks, and then tears everything down. At sub-second cycle times this is a tight busy-loop.
- **State sync starvation**: Because the Byzantine peer's 10 downloads complete (with 204) in milliseconds—before honest peers' adverts arrive (honest peers re-advertise every 5 seconds)—the Byzantine peer can ensure it is always the sole entry in `active_downloads` when the loop exits. Honest peers never get a chance to contribute chunks, so the lagging node cannot catch up.
- **Consensus participation delayed**: A node that cannot complete state sync cannot participate in consensus for the target height, degrading subnet liveness proportionally to the number of affected nodes.

### Likelihood Explanation
The attacker is any single Byzantine subnet peer (well below the BFT fault threshold). The only capability required is the ability to return HTTP 204 instead of 200 for chunk requests—a trivial modification to a malicious replica binary. No key material, governance majority, or network-level attack is needed. The attack is self-sustaining: the Byzantine peer only needs to keep re-advertising the same `StateSyncArtifactId`.

### Recommendation
1. **Minimum state-sync lifetime**: Do not exit the `OngoingStateSync` loop immediately when `active_downloads` becomes empty. Instead, wait for a minimum duration (e.g., 30 seconds) or until a cancellation token fires, so that honest peers' adverts have time to arrive and be added.
2. **Per-peer failure tracking**: Count consecutive `NoContent`/`RequestError` responses per peer across state-sync restarts. Suppress re-adding a peer that has failed repeatedly within a short window.
3. **Rate-limit `maybe_start_state_sync` per height**: Track the last time a state sync was started for a given height and enforce a minimum inter-start interval (e.g., 60 seconds).

### Proof of Concept
```
1. Spin up a test replica that needs state at height H.
2. Connect a "Byzantine" peer that:
   a. Advertises StateSyncArtifactId { height: H, hash: <valid> } continuously.
   b. Returns HTTP 204 for every /chunk request.
3. Instrument maybe_start_state_sync with a call counter.
4. Run for 60 seconds.
5. Observe: maybe_start_state_sync is called O(thousands) of times;
   no chunks are ever successfully delivered; the node never advances past height H.
```
