### Title
Residual ICRC-2 Allowance Not Revoked After Partial Treasury Manager Deposit - (`File: rs/sns/governance/src/extensions.rs`)

### Summary

`execute_treasury_manager_deposit` in SNS Governance sets a full ICRC-2 allowance for the treasury manager canister before calling `deposit`, but never resets the allowance to zero after the call completes. If the treasury manager's `deposit` function only partially consumes the approved amount, the residual allowance persists on the ledger for up to one hour, during which the treasury manager canister can unilaterally drain the remaining approved tokens from the SNS treasury without any additional governance approval.

### Finding Description

In `rs/sns/governance/src/extensions.rs`, the function `execute_treasury_manager_deposit` follows a two-step pattern:

1. Call `approve_treasury_manager`, which issues `icrc2_approve` on both the SNS ledger and the ICP ledger, granting the treasury manager canister an allowance of exactly `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s` respectively, with a 1-hour expiry.

2. Call `deposit` on the treasury manager canister, passing the same amounts as the `DepositRequest`. [1](#0-0) 

The `approve_treasury_manager` function sets the allowance with `expected_allowance: None`, which blindly overwrites any prior allowance: [2](#0-1) 

The `TreasuryManager` API contract (as documented in `treasury_manager.did`) explicitly acknowledges that a deposit may not consume the full allowance — the `Allowance` type includes an `owner_account` field specifically "needed to refund excess assets that cannot be managed at this time": [3](#0-2) 

After `deposit` returns (successfully or partially), **no code in `execute_treasury_manager_deposit` resets the ICRC-2 allowance to zero**. The treasury manager canister retains the ability to call `icrc2_transfer_from` on the SNS and ICP ledgers for the full originally-approved amount minus only what it consumed during `deposit`, for up to one hour.

The ICRC-2 `use_allowance` logic in the ledger core correctly decrements the allowance by the consumed amount but leaves any remainder intact: [4](#0-3) 

### Impact Explanation

A treasury manager canister that partially consumes its approved allowance (e.g., due to DEX slippage, liquidity constraints, or intentional partial fills) retains a live ICRC-2 allowance on both the SNS token ledger and the ICP ledger. Within the 1-hour expiry window, the treasury manager canister can call `icrc2_transfer_from` directly on those ledgers to transfer the remaining approved tokens out of the SNS governance treasury subaccount to any destination — without any additional governance proposal or approval. This constitutes unauthorized token drainage beyond the amount sanctioned by the governance vote.

### Likelihood Explanation

The `ALLOWED_EXTENSIONS` list is currently empty (the Kong Swap extension was removed), so no active treasury manager extension exists today. However, the design flaw is structural: any future blessed treasury manager that performs a partial deposit (a scenario explicitly anticipated by the API design) will leave a residual allowance. The treasury manager canister is a third-party canister controlled by its own developers, and even a non-malicious implementation that partially fills a deposit creates this window. The 1-hour expiry (`ONE_HOUR_SECONDS`) provides a bounded but meaningful exploitation window. [5](#0-4) 

### Recommendation

After `deposit` returns (whether successfully or with partial consumption), immediately reset the ICRC-2 allowance to zero on both ledgers by calling `icrc2_approve` with `amount = 0` for the treasury manager canister. This mirrors the mitigation recommended in the original Malt Finance report: reset the approval if the external call did not consume the full approved amount.

Alternatively, use `expected_allowance` in the subsequent `icrc2_approve` call (if a second deposit proposal is executed) to detect and handle any residual allowance, though this does not close the intra-proposal window.

### Proof of Concept

1. SNS governance passes a proposal to deposit `X` SNS tokens and `Y` ICP tokens into a treasury manager.
2. `execute_treasury_manager_deposit` calls `approve_treasury_manager(extension_canister_id, X, Y)`.
3. The SNS ledger now has allowance `(sns_treasury_subaccount → treasury_manager_canister) = X` and the ICP ledger has `(icp_treasury_subaccount → treasury_manager_canister) = Y`, both expiring in 1 hour.
4. `deposit` is called on the treasury manager. Due to DEX slippage or partial liquidity, it only calls `icrc2_transfer_from` for `X/2` SNS tokens and `Y/2` ICP tokens.
5. `execute_treasury_manager_deposit` returns `Ok(())`. No allowance reset is performed.
6. The treasury manager canister now has a live allowance of `X/2` SNS tokens and `Y/2` ICP tokens remaining.
7. Within the next hour, the treasury manager canister calls `icrc2_transfer_from` on both ledgers to transfer the remaining `X/2` SNS and `Y/2` ICP to any account it chooses — with no governance approval for this second transfer. [6](#0-5) [7](#0-6)

        treasury_manager_canister_id: CanisterId,
        sns_amount_e8s: u64,

        let to = Account {
            owner: treasury_manager_canister_id.get().0,
            subaccount: None,
        };

        let expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS);
        let expiry_time_nsec = expiry_time_sec.saturating_mul(NANO_SECONDS_PER_SECOND);

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

        self.ledger
            .icrc2_approve(
                to,
                sns_amount_e8s,
                Some(expiry_time_nsec),
                self.transaction_fee_e8s_or_panic(),
                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making SNS Token treasury transfer: {e}"),
                )
            })?;

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
