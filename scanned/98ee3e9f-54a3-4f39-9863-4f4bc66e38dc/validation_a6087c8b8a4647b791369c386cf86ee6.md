### Title
Missing Slippage Protection in SNS Treasury Manager Deposit Execution Allows Sandwich Attacks on DAO Treasury - (`rs/sns/treasury_manager/treasury_manager.did`, `rs/sns/governance/src/extensions.rs`)

---

### Summary

The SNS Treasury Manager framework allows SNS DAOs to deposit treasury funds (SNS tokens + ICP) into external DEX liquidity pools via governance proposals. The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` executes these deposits without any slippage protection. The `DepositRequest` type in `treasury_manager.did` contains only token allowances and no expected price ratio field. A malicious actor who can trade on the target DEX can sandwich the governance proposal execution — skewing the pool price before the deposit and reversing after — causing the SNS treasury to deposit at an unfavorable ratio and suffer a direct financial loss. This risk is explicitly acknowledged in the codebase but remains unmitigated.

---

### Finding Description

When an SNS governance proposal of type `ExecuteExtensionOperation` (operation: `TreasuryManagerDeposit`) is adopted and executed, the following call chain runs:

1. `execute_treasury_manager_deposit` in `rs/sns/governance/src/extensions.rs` is called with a `ValidatedDepositOperationArg` containing `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s`.
2. It calls `governance.approve_treasury_manager(...)` to grant ICRC-2 allowances to the Treasury Manager canister.
3. It then calls `governance.env.call_canister(extension_canister_id, "deposit", arg_blob)` — forwarding a `DepositRequest` to the Treasury Manager.

The `DepositRequest` type, as defined in `treasury_manager.did`, contains only `allowances` (token amounts):

```candid
type DepositRequest = record {
  allowances : vec Allowance;
};
```

There is no field for an expected price ratio, minimum received LP tokens, or any other slippage bound. The governance canister has no mechanism to enforce that the DEX pool price at execution time matches the price at proposal approval time.

The `validate_deposit_operation_impl` function only checks that the requested amounts do not exceed 50% of the current treasury balance — it performs no price or ratio validation whatsoever.

The codebase itself acknowledges this in two places:

- `treasury_manager.did` lines 35–40: *"Known Security Risks: Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."*
- `rs/sns/governance/src/proposal.rs` lines 1542–1545: *"Some Decentralized Exchanges lack slippage protection during deposits. Consequently, deposited asset ratios may deviate from those specified in the proposal. This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running or sandwich attacks."*

Despite this acknowledgment, no mitigation is implemented in the production execution path.

---

### Impact Explanation

An attacker who can trade on the target DEX can execute a sandwich attack against any SNS DAO that uses the Treasury Manager to deposit into a liquidity pool:

1. Attacker observes a pending (adopted but not yet executed) governance proposal to deposit `X` SNS + `Y` ICP into a DEX pool.
2. Attacker submits a large trade to the DEX, skewing the pool's price ratio significantly in one direction.
3. The governance canister executes `execute_treasury_manager_deposit`, depositing at the manipulated ratio. The SNS treasury receives far fewer LP tokens than it should, or deposits a highly imbalanced ratio of assets.
4. Attacker reverses their trade, profiting from the price impact while the SNS treasury has permanently deposited at the unfavorable ratio.

The SNS treasury suffers a direct, irreversible financial loss proportional to the attacker's price manipulation. The magnitude scales with the deposit size and the attacker's capital. Since governance proposals are public and their execution is predictable, the attack window is well-defined and reliable.

---

### Likelihood Explanation

- SNS governance proposals are publicly visible on-chain before execution. Any observer can see the pending deposit and its parameters.
- The attacker only needs to be able to trade on the target DEX — no privileged access, no governance majority, no key compromise is required.
- The IC does not have a traditional mempool, but the attack window is the entire period between proposal adoption and execution (which can be hours or days for governance proposals), making timing trivial compared to EVM frontrunning.
- The codebase explicitly documents this as a known risk, confirming the developers consider it realistic.
- Any SNS that registers a Treasury Manager extension and passes a deposit proposal is immediately exposed.

---

### Recommendation

1. **Add a `min_lp_tokens_out` or `expected_price_ratio` field to `DepositRequest`** in `treasury_manager.did`, allowing the caller to specify a slippage bound.
2. **Require Treasury Manager implementations to enforce this bound** — rejecting the deposit if the actual execution price deviates beyond the specified tolerance.
3. **Propagate the slippage parameter through the governance proposal** — the `ValidatedDepositOperationArg` and `construct_treasury_manager_deposit_payload` should include and forward the expected ratio so voters can see and approve the slippage tolerance.
4. As a defense-in-depth measure, **require Treasury Manager implementations to return the actual execution price** in the `Balances` response so governance can verify post-execution.

---

### Proof of Concept

**Step 1**: SNS governance adopts a proposal: deposit 100,000 SNS + 10,000 ICP into DEX pool at current ratio 10:1.

**Step 2**: Attacker observes the adopted proposal. Before execution, attacker buys a large amount of SNS from the DEX pool, shifting the ratio to 20:1 (SNS now worth half as much in ICP terms).

**Step 3**: Governance canister executes `execute_treasury_manager_deposit`:

```rust
// rs/sns/governance/src/extensions.rs ~line 1566-1578
governance
    .approve_treasury_manager(
        extension_canister_id,
        treasury_allocation_sns_e8s,   // 100,000 SNS
        treasury_allocation_icp_e8s,   // 10,000 ICP
    )
    .await?;

let balances = governance
    .env
    .call_canister(extension_canister_id, "deposit", arg_blob)
    .await  // No price check — deposits at manipulated 20:1 ratio
```

The Treasury Manager deposits at the manipulated 20:1 ratio. The SNS treasury effectively loses ~50% of the SNS token value relative to what voters approved.

**Step 4**: Attacker sells their SNS back to the pool, restoring the price and pocketing the profit. The SNS treasury's LP position is permanently at the unfavorable entry price.

The `DepositRequest` carries no slippage bound:

```candid
// rs/sns/treasury_manager/treasury_manager.did line 84-86
type DepositRequest = record {
  allowances : vec Allowance;  // Only amounts, no price/ratio constraint
};
```

And `execute_treasury_manager_deposit` performs no price validation before or after the call:

```rust
// rs/sns/governance/src/extensions.rs ~line 1545-1609
async fn execute_treasury_manager_deposit(...) -> Result<(), GovernanceError> {
    // No slippage check here or anywhere in this function
    governance.approve_treasury_manager(...).await?;
    let balances = governance.env.call_canister(..., "deposit", ...).await...?;
    // No post-execution price validation either
    Ok(())
}

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())

/// Validated deposit operation arguments
#[derive(Debug, Clone)]
pub struct ValidatedDepositOperationArg {
    /// Amount of SNS tokens to allocate from treasury
    pub treasury_allocation_sns_e8s: u64,
    /// Amount of ICP tokens to allocate from treasury
    pub treasury_allocation_icp_e8s: u64,
    /// Original Precise value with all fields
    pub original: Precise,
}

impl TryFrom<Option<Precise>> for ValidatedDepositOperationArg {
    type Error = String;

    fn try_from(value: Option<Precise>) -> Result<Self, Self::Error> {
        let Some(original) = value else {
            return Err("Deposit operation arguments must be provided".to_string());
        };

        let map = match &original.value {
            Some(precise::Value::Map(PreciseMap { map })) => map,
            _ => return Err("Deposit operation arguments must be a PreciseMap".to_string()),
        };

        let treasury_allocation_sns_e8s = map
            .get("treasury_allocation_sns_e8s")
            .and_then(|p| match &p.value {
                Some(precise::Value::Nat(n)) => Some(*n),
                _ => None,
            })
            .ok_or_else(|| "treasury_allocation_sns_e8s must be a Nat value".to_string())?;

        let treasury_allocation_icp_e8s = map
            .get("treasury_allocation_icp_e8s")
            .and_then(|p| match &p.value {
                Some(precise::Value::Nat(n)) => Some(*n),
                _ => None,
            })
            .ok_or_else(|| "treasury_allocation_icp_e8s must be a Nat value".to_string())?;

        Ok(Self {
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
            original,
        })
    }
