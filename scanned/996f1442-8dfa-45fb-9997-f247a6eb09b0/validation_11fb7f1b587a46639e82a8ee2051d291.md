### Title
Silent ICP Burn Failure in CMC Leaves Funds Permanently Stuck in Subaccounts - (File: rs/nns/cmc/src/main.rs)

### Summary
The `burn_and_log` function in the Cycles Minting Canister (CMC) explicitly swallows ICP burn failures after a successful canister creation or top-up. If the ledger call to burn ICP fails, the ICP remains permanently locked in the CMC's subaccount with no recovery path, while the block notification is already marked as successfully processed and cannot be retried.

### Finding Description

The `burn_and_log` function at `rs/nns/cmc/src/main.rs` lines 2017–2048 is called after `do_create_canister` succeeds inside `process_create_canister` (line 1945). It issues a `send_pb` call to the ICP ledger to transfer the user's ICP to the minting account (effectively burning it). The result is matched, but on failure only a log message is emitted — no error is returned, no retry is scheduled, and no state is updated to reflect the failed burn:

```rust
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    // ...
    let res: CallResult<BlockIndex> =
        call_protobuf(ledger_canister_id, "send_pb", send_args).await;
    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}

The caller `process_create_canister` returns `Ok(canister_id)` regardless of whether `burn_and_log` succeeds:

```rust
Ok(canister_id) => {
    burn_and_log(sub, amount).await;
    Ok(canister_id)
}
``` [2](#0-1) 

The notification is then stored as `NotificationStatus::NotifiedCreateCanister(Ok(canister_id))`: [3](#0-2) 

Because the block index is recorded as successfully processed, any subsequent call to `notify_create_canister` with the same block index returns the cached `Ok` result immediately, preventing any retry of the burn. The ICP in the CMC subaccount (`Subaccount::from(&controller)`) is controlled solely by the CMC and has no user-accessible withdrawal path.

The same pattern applies to `refund_icp`, which calls `burn_and_log` after sending a refund: [4](#0-3) 

The design comment explicitly acknowledges this: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* This prevents double-spending of the notification, but at the cost of permanently stranding ICP. [5](#0-4) 

### Impact Explanation

ICP sent to a CMC subaccount for canister creation or top-up can become permanently irrecoverable if the ledger burn call fails transiently. The ICP is not returned to the user (the notification is finalized), not burned (the ledger call failed), and not retried (no retry mechanism exists). This constitutes a **ledger conservation bug**: the circulating supply of ICP is not reduced as the protocol intends, and real token value is permanently locked in a CMC-controlled subaccount with no accessor. Over time, repeated transient ledger failures accumulate stuck ICP. The user received their canister or cycles, so direct user harm is indirect (protocol-level inflation of ICP supply), but the stuck ICP is unrecoverable by any party.

### Likelihood Explanation

The ICP ledger can transiently reject calls due to: the ledger being temporarily unavailable (`TemporarilyUnavailable`), the CMC being at its message queue limit, or a subnet-level issue causing the inter-canister call to be dropped. These are realistic operational conditions on mainnet. Any user who calls `notify_create_canister` or `notify_top_up` during such a window triggers this path. No special privilege is required — any unprivileged ingress sender can reach `notify_create_canister`. [6](#0-5) 

### Recommendation

1. **Retry queue**: Record failed burns in persistent CMC state and retry them in a heartbeat or timer task, similar to how the ckBTC minter handles pending operations via its event log.
2. **Separate burn from notification finalization**: Only mark the notification as `NotifiedCreateCanister(Ok(...))` after the burn succeeds. If the burn fails transiently, store the notification as `Processing` and retry, returning `NotifyError::Processing` to the caller.
3. **At minimum**, emit a metric counter for failed burns so operators can detect accumulation and manually intervene via governance upgrade.

### Proof of Concept

1. User sends 10 ICP to `AccountIdentifier::new(CMC_ID, Some(Subaccount::from(&controller)))` with `MEMO_CREATE_CANISTER`.
2. User calls `notify_create_canister` with the resulting block index.
3. CMC calls `do_create_canister` — succeeds, canister is created.
4. CMC calls `burn_and_log(sub, amount)` — the ICP ledger returns a transient error (e.g., `TemporarilyUnavailable`).
5. `burn_and_log` logs the error and returns `()`.
6. `process_create_canister` returns `Ok(canister_id)`.
7. CMC stores `NotificationStatus::NotifiedCreateCanister(Ok(canister_id))` for the block index.
8. Any retry of `notify_create_canister` with the same block index returns the cached `Ok(canister_id)` immediately — the burn is never retried.
9. The 10 ICP (minus fees) remains in `AccountIdentifier::new(CMC_ID, Some(sub))` indefinitely, inaccessible to any party and not reflected as burned in the ICP ledger's total supply accounting. [7](#0-6) [8](#0-7)

async fn process_create_canister(
    controller: PrincipalId,
    from: AccountIdentifier,
    amount: Tokens,
    subnet_selection: Option<SubnetSelection>,
    settings: Option<CanisterSettings>,
) -> Result<CanisterId, NotifyError> {
    let cycles = tokens_to_cycles(amount)?;

    let sub = Subaccount::from(&controller);

    print(format!(
        "Creating canister with controller {controller} with {cycles} cycles.",
    ));

    // Create the canister. If this fails, refund. Either way,
    // return a result so that the notification cannot be retried.
    // If refund fails, we allow to retry.

        Ok(canister_id) => {
            burn_and_log(sub, amount).await;
            Ok(canister_id)

            let refund_block = refund_icp(sub, from, amount, CREATE_CANISTER_REFUND_FEE).await?;
            Err(NotifyError::Refunded {
                reason: err,
                block_index: refund_block,

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
