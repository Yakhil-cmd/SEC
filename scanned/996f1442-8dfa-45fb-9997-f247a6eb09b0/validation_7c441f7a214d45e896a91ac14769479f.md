### Title
Unbounded `MintSnsTokens` Governance Proposal Allows SNS Token Inflation Without Rate Limit - (File: rs/sns/governance/src/proposal.rs)

### Summary
The `MintSnsTokens` SNS governance proposal action intentionally has its 7-day minting rate-limit enforcement **commented out** (marked `TODO(NNS1-2982): Delete`), causing `recent_amount_total_upper_bound_tokens` to return `Decimal::MAX` for every proposal. An SNS governance majority can therefore pass an unlimited number of `MintSnsTokens` proposals in rapid succession, minting an unbounded quantity of SNS tokens to any target principal, with no on-chain rate-limit protection.

### Finding Description
`TransferSnsTreasuryFunds` proposals are protected by a 7-day rolling cap enforced both at proposal submission time and at execution time. The analogous cap for `MintSnsTokens` is fully implemented in the library (`mint_sns_tokens_7_day_total_upper_bound_tokens` in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs`) but is deliberately disabled in the governance canister.

In `rs/sns/governance/src/proposal.rs`, the `TokenProposalAction` implementation for `MintSnsTokens` overrides `recent_amount_total_upper_bound_tokens` to return `Decimal::MAX` unconditionally:

```rust
// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    Ok(Decimal::MAX)
}
```

The correct implementation is commented out:

```rust
/* TODO(NNS1-2982): Uncomment.
fn recent_amount_total_upper_bound_tokens(valuation: &Valuation) -> Result<Decimal, String> {
    mint_sns_tokens_7_day_total_upper_bound_tokens(*valuation)
    ...
}
*/
```

The import of `mint_sns_tokens_7_day_total_upper_bound_tokens` is also commented out at the top of the file. The integration test in `rs/sns/integration_tests/src/sns_treasury.rs` confirms this: the second `MintSnsTokens` proposal that should be rejected is instead allowed to succeed (`doomed_make_proposal_result.unwrap()`), and the whale's balance doubles to `2 * 2_222` tokens.

Additionally, `perform_mint_sns_tokens` in `rs/sns/governance/src/governance.rs` performs no execution-time rate-limit check (unlike `perform_transfer_sns_treasury_funds` which calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`).

### Impact Explanation
An SNS governance majority (which can be a small coordinated group of token holders, or even a single whale neuron) can pass back-to-back `MintSnsTokens` proposals to mint an arbitrary total amount of SNS tokens to any target principal. This is a direct analog of the IMA rug-pull: instead of draining a deposit box, the attacker inflates the token supply to dilute all existing holders to near-zero and/or transfer massive value to themselves. The SNS token ledger minting account is controlled by the governance canister, so the minted tokens are real and immediately transferable.

### Likelihood Explanation
The entry path is a standard SNS governance proposal, reachable by any principal with enough voting power to pass a proposal. No privileged key, admin access, or threshold corruption is required — only a governance majority, which is the normal operating condition for SNS DAOs. The missing guard is not a configuration issue; it is absent from the production code path for every deployed SNS.

### Recommendation
Uncomment the rate-limit enforcement for `MintSnsTokens`:

1. In `rs/sns/governance/src/proposal.rs`, uncomment `mint_sns_tokens_7_day_total_upper_bound_tokens` in the import block and replace the `Decimal::MAX` stub with the real implementation.
2. Add an execution-time check in `perform_mint_sns_tokens` (analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`) to prevent the race condition where two proposals are submitted under the limit but together exceed it.
3. Remove the `TODO(NNS1-2982)` markers once the fix is in place.

            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
