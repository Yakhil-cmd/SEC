### Title
Node Provider Rewards Silently Dropped Due to Mismatched Iteration Set in `get_node_providers_rewards` / `get_monthly_node_provider_rewards` - (File: rs/nns/governance/src/governance.rs)

### Summary
Both `get_node_providers_rewards` and `get_monthly_node_provider_rewards` iterate over `heap_data.node_providers` (governance's internal list) to distribute rewards, while the reward amounts are computed by the Node Rewards Canister (or Registry) based on the registry's node operators. These two sets can diverge: a node provider removed from governance's list while their node operators remain active in the registry will have their earned rewards silently dropped with no error, no log, and no redistribution.

### Finding Description
In `get_node_providers_rewards` (performance-based path):

```rust
for np in &self.heap_data.node_providers {          // iterates governance's list
    if let Some(np_id) = &np.id {
        let xdr_permyriad_reward =
            *rewards_per_node_provider.get(np_id).unwrap_or(&0);  // silent zero if absent
        ...
    }
}
```

And identically in `get_monthly_node_provider_rewards` (legacy path):

```rust
for np in &self.heap_data.node_providers {
    if let Some(np_id) = &np.id {
        let xdr_permyriad_reward = *reg_rewards.get(np_id).unwrap_or(&0);
        ...
    }
}
```

`rewards_per_node_provider` / `reg_rewards` is keyed by node provider principal IDs derived from the registry's `NodeOperatorRecord.node_provider_principal_id` fields. `heap_data.node_providers` is governance's own list, managed exclusively via `AddOrRemoveNodeProvider` proposals. These are two independent data structures with no automatic synchronization.

The critical asymmetry: `validate_assign_noid_payload` enforces that a node provider **must** be in `heap_data.node_providers` before a node operator can be added to the registry. However, `ValidRemoveNodeProvider::execute` removes a node provider from `heap_data.node_providers` with **no check** that the provider has zero active node operators in the registry. After removal, the Node Rewards Canister continues computing rewards for that provider (their node operators still exist), but governance's iteration never reaches them — their rewards are silently discarded via `.unwrap_or(&0)`.

This is the direct analog of the reported bug: the code iterates over the wrong set (`heap_data.node_providers` / `valset_execution`) instead of the authoritative reward-bearing set (`rewards_per_node_provider` / `valset_snapshot`).

### Impact Explanation
Any node provider removed from `heap_data.node_providers` while retaining active node operators in the registry permanently loses their monthly ICP rewards. The rewards are computed by the Node Rewards Canister, passed to governance, and then silently discarded — they are neither distributed to the node provider nor to any other party. This is a **ledger conservation bug**: ICP that should be minted and transferred is never minted. The `mint_monthly_node_provider_rewards` call path is the only mechanism for distributing these rewards; once the monthly window passes, the rewards are gone.

### Likelihood Explanation
The divergence is created by executing an `AddOrRemoveNodeProvider` (ToRemove) governance proposal against a node provider who still has live node operators in the registry. This is a realistic operational scenario: governance may vote to deregister a node provider (e.g., for non-compliance or exit) before the separate registry mutations to remove their node operators are processed. The `ValidRemoveNodeProvider::execute` function performs no cross-check against the registry. No malicious actor is required — a sequencing error between two legitimate governance proposals is sufficient. The `mint_monthly_node_provider_rewards` function is called automatically by the governance heartbeat, so the loss occurs without any further action.

### Recommendation
Two complementary fixes:

1. **Enforce invariant at removal time**: In `ValidRemoveNodeProvider::validate` (or `execute`), query the registry to confirm the node provider has no active `NodeOperatorRecord` entries before allowing removal from `heap_data.node_providers`.

2. **Iterate over the authoritative reward set**: In both `get_node_providers_rewards` and `get_monthly_node_provider_rewards`, iterate over the keys of `rewards_per_node_provider` / `reg_rewards` rather than `heap_data.node_providers`. For each key, look up the corresponding `NodeProvider` struct (for the reward account) from `heap_data.node_providers`; if absent, log a warning and skip rather than silently dropping.

### Proof of Concept

1. Submit and pass an `AddOrRemoveNodeProvider { ToAdd: NP_X }` governance proposal → NP_X is added to `heap_data.node_providers`.
2. Submit and pass an `ExecuteNnsFunction { AssignNoid }` proposal adding a node operator for NP_X to the registry → `validate_assign_noid_payload` passes because NP_X is in `heap_data.node_providers`.
3. Submit and pass an `AddOrRemoveNodeProvider { ToRemove: NP_X }` governance proposal → `ValidRemoveNodeProvider::execute` removes NP_X from `heap_data.node_providers` with no registry check.
4. At the next monthly reward cycle, `mint_monthly_node_provider_rewards` calls `get_node_providers_rewards` (or `get_monthly_node_provider_rewards`). The Node Rewards Canister returns a non-zero reward for NP_X (their node operators are still active). The governance loop iterates `heap_data.node_providers`, which no longer contains NP_X. The reward entry for NP_X in `rewards_per_node_provider` is never visited. NP_X receives zero ICP. The reward is permanently lost.

            .iter()
            .any(|np| np.id.unwrap() == decoded_payload.node_provider_principal_id.unwrap());
        if !is_registered {
            return Err("The node provider specified in the payload is not registered".to_string());
        }

        Ok(())
