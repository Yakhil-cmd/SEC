### Title
Unsanitized Markdown Injection in SNS/NNS Governance Proposal Text Fields Enables Voter Deception - (File: rs/sns/governance/src/proposal.rs)

### Summary
The SNS governance canister's `validate_and_render_motion` function and the NNS governance's `validate_proposal_summary` perform only length-based validation on user-controlled text fields. These fields are directly interpolated into a Markdown rendering that is stored on-chain in `payload_text_rendering` and explicitly documented as intended for display in UI frontends. Any unprivileged SNS neuron holder can craft a Motion proposal whose `motion_text` injects Markdown that mimics the rendering of a more dangerous proposal type (e.g., `UpgradeSnsControlledCanister` or `TransferSnsTreasuryFunds`), misleading voters into approving a proposal they believe is benign.

### Finding Description
In `rs/sns/governance/src/proposal.rs`, `validate_and_render_motion` validates `motion_text` only for byte length, then directly interpolates it into a Markdown template:

```rust
fn validate_and_render_motion(motion: &Motion) -> Result<String, String> {
    validate_len("motion.motion_text", &motion.motion_text, 0, PROPOSAL_MOTION_TEXT_BYTES_MAX)?;
    Ok(format!(
        r"# Motion Proposal:
## Motion Text:

{}",
        &motion.motion_text   // ← unsanitized user input
    ))
}

### Impact Explanation
An attacker who controls an SNS neuron can submit a Motion proposal (action id 1) whose `motion_text` is crafted to render identically to a high-impact proposal type. For example, injecting:

```
Approve community guidelines update

---

# Proposal to Upgrade an SNS Controlled Canister

## Target canister: <attacker-controlled canister id>

## Wasm info

Embedded module with 8 bytes and SHA256 `<hash>`.

## Mode: Upgrade
```

When the SNS dapp renders `payload_text_rendering` as Markdown, voters see what appears to be a legitimate canister upgrade proposal. If the Motion passes, no canister is actually upgraded (a Motion has no on-chain execution effect), but the deception can be used as a social-engineering step: e.g., to establish a false precedent, to confuse audit trails, or—combined with following/liquid democracy—to cause neurons that auto-follow "Upgrade" proposals to vote on a Motion they would otherwise ignore.

More critically, the same injection can be used in reverse: a genuine `TransferSnsTreasuryFunds` or `UpgradeSnsControlledCanister` proposal can have its `summary` field (also unsanitized) crafted to look like a harmless Motion, hiding the true action from voters who skim the rendered text rather than inspecting the raw Candid payload. [6](#0-5) 

### Likelihood Explanation
The entry path requires only that the attacker holds a staked SNS neuron, which is an unprivileged role open to any token holder. The `make_proposal` canister method is a standard ingress call. No privileged key, admin role, or subnet-majority corruption is needed. The SNS dapp is the primary UI used by voters and is known to render `payload_text_rendering` as Markdown. The attack requires no special tooling beyond constructing a Candid-encoded `Proposal` message with a crafted `motion_text` or `summary` string.

### Recommendation
1. **Strip or escape Markdown control characters** from all user-supplied text fields (`motion_text`, `summary`, `title`, `name`, `description`) before interpolating them into the rendered Markdown string in `validate_and_render_motion`, `validate_and_render_manage_sns_metadata`, and equivalent render functions.
2. **Render each field in a structurally isolated Markdown block** (e.g., inside a fenced code block or using a dedicated UI component) so that injected headers, horizontal rules, and links cannot escape the field's visual boundary.
3. **Apply the same sanitization to `validate_proposal_summary`** in `rs/nns/governance/api/src/proposal_validation.rs` for NNS proposals. [5](#0-4) 

### Proof of Concept
**Attacker-controlled entry path:**
1. Acquire SNS tokens, stake a neuron with sufficient dissolve delay to meet `neuron_minimum_dissolve_delay_to_vote_seconds`.
2. Call `manage_neuron` → `MakeProposal` on the SNS governance canister with:

```candid
record {
  title = "Routine community vote";
  summary = "";
  url = "";
  action = variant {
    Motion = record {
      motion_text = "Approve community guidelines.\n\n---\n\n# Proposal to Upgrade an SNS Controlled Canister\n\n## Target canister: <victim-canister-id>\n\n## Wasm info\n\nEmbedded module with 8 bytes and SHA256 `deadbeef...`.\n\n## Mode: Upgrade\n\n## Argument info\n\nNo upgrade argument."
    }
  }
}
```

3. The SNS governance canister calls `validate_and_render_motion`, which passes the length check (well under `PROPOSAL_MOTION_TEXT_BYTES_MAX`) and stores the injected string verbatim in `payload_text_rendering`. [7](#0-6) 

4. Any caller of `list_proposals` or `get_proposal` receives `payload_text_rendering` containing the injected Markdown. The SNS dapp renders this as a canister upgrade proposal, while the actual on-chain action is a harmless Motion. [8](#0-7)

pub fn validate_proposal_summary(summary: &str) -> Result<(), String> {
    if summary.len() > PROPOSAL_SUMMARY_BYTES_MAX {
        return Err(format!(
            "The maximum proposal summary size is {} bytes, this proposal is: {} bytes",
            PROPOSAL_SUMMARY_BYTES_MAX,
            summary.len(),
        ));
    }

    Ok(())
}
