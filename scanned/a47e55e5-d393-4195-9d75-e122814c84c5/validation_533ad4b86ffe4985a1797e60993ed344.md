### Title
Unbounded Iteration in `get_pending_proposals` Query Will Eventually Exceed IC Instruction Limit - (File: rs/nns/governance/src/governance.rs)

### Summary

`Governance::get_pending_proposals` iterates over the entire `heap_data.proposals` map — which grows unboundedly as the NNS accumulates proposals over its lifetime — without any pagination or limit. As the total number of historical proposals grows, the per-call instruction cost grows proportionally, and the function will eventually exceed the IC query instruction limit, permanently breaking the endpoint for all callers.

### Finding Description

The `get_pending_proposals` function in the NNS Governance canister is exposed as a public `#[query]` endpoint:

```rust
// rs/nns/governance/canister/canister.rs
#[query]
fn get_pending_proposals(req: Option<GetPendingProposalsRequest>) -> Vec<ProposalInfo> {
    with_governance(|governance| governance.get_pending_proposals(&caller(), req))
}
```

Its implementation iterates over **every proposal ever created** in `heap_data.proposals`, filtering lazily for open ones:

```rust
// rs/nns/governance/src/governance.rs
pub fn get_pending_proposals(
    &self,
    caller: &PrincipalId,
    req: Option<GetPendingProposalsRequest>,
) -> Vec<ProposalInfo> {
    let now = self.env.now();
    let caller_neurons = self.get_neuron_ids_by_principal(caller);
    ...
    self.heap_data
        .proposals
        .values()
        .filter(|data| data.status() == ProposalStatus::Open)
        .map(|data| {
            proposal_data_to_info(
                data,
                ProposalDisplayOptions::for_get_pending_proposals(...),
                &caller_neurons,
                now,
                self.voting_period_seconds(),
            )
        })
        .collect()
}
``` [1](#0-0) 

The Rust iterator is lazy, but the `.filter()` predicate is evaluated for every entry in `heap_data.proposals`. The `proposal_data_to_info` call is non-trivial (it computes reward status, voting period, filters ballots, etc.). The `heap_data.proposals` map is never pruned — settled proposals (executed, rejected, failed) remain in it indefinitely. The NNS has been live since 2021 and has accumulated thousands of proposals.

By contrast, `list_proposals` correctly applies a `limit` (capped at `MAX_LIST_PROPOSAL_RESULTS`) and uses cursor-based pagination via `before_proposal`. `get_pending_proposals` has no equivalent guard. [2](#0-1) 

The canister DID exposes this as a public query:

```
get_pending_proposals : (opt GetPendingProposalsRequest) -> (vec ProposalInfo) query;
``` [3](#0-2) 

### Impact Explanation

When the total number of proposals in `heap_data.proposals` grows large enough that iterating and calling `proposal_data_to_info` on each one exhausts the IC query instruction limit (~5 billion instructions), every call to `get_pending_proposals` will trap. This permanently breaks:

- The Rosetta API's `get_pending_proposals` endpoint (used by exchanges and tooling)
- Any governance dashboard or dapp that polls pending proposals
- The NNS dapp's ability to display open proposals to users

The function returns no partial result — it either succeeds fully or traps entirely. There is no fallback. [4](#0-3) 

### Likelihood Explanation

The NNS has been running since 2021 and already has thousands of proposals. The `heap_data.proposals` map grows monotonically. The `proposal_data_to_info` function performs non-trivial per-proposal work (ballot filtering, reward status computation, voting period lookup). As governance activity continues, the instruction cost per call increases linearly with total proposal count. This is not a theoretical concern — it is a matter of when, not if. [5](#0-4) 

### Recommendation

Add a `limit` and cursor-based pagination to `get_pending_proposals`, mirroring the pattern already used in `list_proposals`. Alternatively, maintain a separate index of currently-open proposal IDs so the function only iterates over open proposals rather than the full historical map. The fix is analogous to what was applied to `BondAggregator.liveMarketsBy`: add `start` and `stop` indices (or a `before_proposal` cursor with a `limit` cap).

### Proof of Concept

1. The NNS governance canister's `heap_data.proposals` is a `BTreeMap<u64, ProposalData>` that is never pruned.
2. `get_pending_proposals` calls `.values()` on this map — iterating all entries — then lazily filters for `ProposalStatus::Open`.
3. For each entry (open or not), the iterator must evaluate `data.status()`. For open entries, it additionally calls `proposal_data_to_info`, which computes `voting_period_seconds`, filters ballots, and serializes `ProposalInfo`.
4. With N total proposals, the instruction cost is O(N). As N grows past the IC query instruction limit threshold, every call traps.
5. Any unprivileged user can trigger this by simply calling `get_pending_proposals` once the threshold is crossed — no special permissions required. [5](#0-4) [4](#0-3)
