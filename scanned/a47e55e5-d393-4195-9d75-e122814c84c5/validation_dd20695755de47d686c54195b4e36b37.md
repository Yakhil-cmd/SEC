### Title
`SenderInfoContent` Canister Signature Is Not Bound to the Ingress Request ŌĆö Replay Across Arbitrary Requests - (File: `rs/validator/src/ingress_validation.rs`)

### Summary

The `sender_info` field introduced in the IC HTTP interface allows a signing canister (e.g., Internet Identity) to attest to user attributes by producing a canister signature over `SenderInfoContent(info_bytes)`. Because the signed payload contains **only the raw `info` bytes** and a static domain separator ŌĆö with no binding to the `MessageId`, target canister, method, arguments, or `ingress_expiry` ŌĆö the same `(info, sig)` pair is cryptographically valid for **any** ingress message that uses the same `sender_pubkey`. An unprivileged sender can capture one valid `sender_info` and replay it verbatim across arbitrarily many different requests to different canisters, methods, and argument payloads, for the full lifetime of the canister signature's certified-data proof.

---

### Finding Description

`SenderInfoContent` is the signable type used to verify the `sender_info` field of an ingress message. Its full signed-byte representation is:

```
len("ic-sender-info") || "ic-sender-info" || info_bytes
```

Critically, **none of the following request-specific fields are included in the signed content**:
- `MessageId` (which commits to `canister_id`, `method_name`, `arg`, `sender`, `ingress_expiry`, `nonce`)
- Target `canister_id`
- `ingress_expiry`
- `sender` principal

The `validate_sender_info` function receives the full `HttpRequest<C>` but passes only `sender_info.info` into the signed content: [4](#0-3) 

Because the canister signature scheme certifies `hash(seed) || hash(msg)` in the canister's certified data tree, and `msg` here is only `domain || info_bytes`, the same certified-data proof is valid for every ingress message that carries the same `(sender_pubkey, info, sig)` triple ŌĆö regardless of which canister is being called, what method is invoked, what arguments are passed, or when the request expires.

---

### Impact Explanation

Any canister that reads `msg_caller_info_data()` and makes authorization or business-logic decisions based on the attested `info` bytes is vulnerable to replay. A concrete scenario:

1. A signing canister (e.g., Internet Identity) issues a `sender_info` attesting `info = "role:admin"` for a user's session.
2. The user legitimately calls canister A with this `sender_info`.
3. The user replays the **identical** `(info, sig)` in a new request to canister B (different canister, different method, different arguments, different `ingress_expiry`).
4. Canister B's `validate_sender_info` check passes because the canister signature over `SenderInfoContent(info_bytes)` is still valid ŌĆö the signing canister's certified data has not changed.
5. Canister B grants the user `role:admin` access it was never intended to have.

The signing canister has no mechanism at the protocol level to restrict the attestation to a specific target canister, method, or time window ŌĆö the protocol does not include those fields in the signed content, so no amount of care by the signing canister can prevent cross-context replay unless it encodes the full request context inside `info_bytes` itself (which the protocol does not require or document).

---

### Likelihood Explanation

The attack requires only a valid `sender_info` obtained through any normal interaction with the signing canister. No privileged access, no key compromise, and no threshold corruption is needed. The attacker is the legitimate sender ŌĆö they simply reuse the same `(info, sig)` bytes in a different request. The canister signature's certified-data proof remains valid until the signing canister rotates its certified data (i.e., until it explicitly revokes the proof by updating its certified data tree). This is a low-effort, zero-privilege replay that any user of the `sender_info` feature can perform.

---

### Recommendation

Bind the `SenderInfoContent` signed payload to the specific ingress request by including the `MessageId` (or at minimum the `ingress_expiry` and target `canister_id`) in the signed bytes:

```rust
pub struct SenderInfoContent<'a> {
    pub info: &'a [u8],
    pub message_id: &'a MessageId,   // binds to the exact request
}

impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.message_id.as_bytes());
        bytes.extend_from_slice(self.info);
    }
}
```

`validate_sender_info` already has access to the full `HttpRequest<C>` and can supply `request.id()` to the content constructor. This change makes each `sender_info` signature valid for exactly one ingress message, directly analogous to the Connext fix of tracking used signatures per `transferId`.

---

### Proof of Concept

```
// Step 1: obtain a valid sender_info for any request R1
let info_bytes = b"role:admin".to_vec();
let sender_info_content = SenderInfoContent(&info_bytes);
// signing canister certifies hash(seed) || hash(domain || info_bytes)
let sig = signing_canister.sign(&sender_info_content).await;

let valid_sender_info = RawSignedSenderInfo {
    info: Blob(info_bytes.clone()),
    signer: Blob(signing_canister_id.get().into_vec()),
    sig: Blob(sig),
};

// Step 2: send R1 to canister_A ŌĆö accepted
send_update(canister_A, "privileged_method", args_A, valid_sender_info.clone()).await;

// Step 3: replay the IDENTICAL sender_info to canister_B with different args
// validate_sender_info passes because SenderInfoContent(info_bytes) is unchanged
send_update(canister_B, "other_privileged_method", args_B, valid_sender_info).await;
// canister_B sees msg_caller_info_data() == b"role:admin" ŌĆö unintended grant
```

The `verify_sender_info_canister_sig` call at [5](#0-4) 
succeeds for both requests because the signed content `SenderInfoContent(&sender_info.info)` is identical in both cases, and the canister signature proof has not been invalidated.

    request: &HttpRequest<C>,
    ingress_signature_verifier: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    let Some(sender_info) = request.sender_info() else {
        return Ok(());
    };

    // Per the spec, the sender_info signature must verify using the
    // envelope-level sender_pubkey as a canister signature public key.

        Authentication::Authenticated(sig) => &sig.signer_pubkey,
        Authentication::Anonymous => {

                "sender_info requires an authenticated request with sender_pubkey".to_string(),
            ));
        }
    };

    verify_sender_info_canister_sig(
        sender_info,
        sender_pubkey,
        ingress_signature_verifier,
        root_of_trust_provider,
    )
}
