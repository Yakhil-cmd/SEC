### Title
Excess ICP Sent to SNS Swap Subaccount Is Not Returned During Active Swap - (File: `rs/sns/swap/src/swap.rs`)

### Summary
The `refresh_buyer_token_e8s` function in the SNS swap canister accepts ICP participation up to `max_participant_icp_e8s` per buyer. When a user transfers more ICP than this cap to their swap subaccount, the excess is silently left in the subaccount and cannot be retrieved until the swap reaches a terminal lifecycle state (COMMITTED or ABORTED). The codebase itself acknowledges this with an open TODO. This directly mirrors the external report's class: excess payment accepted by a contract but not returned to the sender.

---

### Finding Description

`refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` reads the ICP balance of the caller's subaccount on the swap canister, then caps the accepted amount at `max_participant_icp_e8s`:

```rust
/// TODO(NNS1-1682): attempt to refund ICP that cannot be accepted.
pub async fn refresh_buyer_token_e8s(...)
``` [1](#0-0) 

After computing the new balance, the function silently clamps it:

```rust
let new_balance_e8s = std::cmp::min(new_balance_e8s, max_participant_icp_e8s);
``` [2](#0-1) 

Any ICP in the subaccount beyond `max_participant_icp_e8s` is neither accepted into the buyer's state nor returned to the user. It remains stranded in the subaccount.

The only recovery path is `error_refund_icp`, which is gated behind a terminal lifecycle check:

```rust
if !(self.lifecycle() == Lifecycle::Aborted || self.lifecycle() == Lifecycle::Committed) {
    return ErrorRefundIcpResponse::new_precondition_error(
        "Error refunds can only be performed when the swap is ABORTED or COMMITTED",
    );
}
``` [3](#0-2) 

So during the entire OPEN phase of the swap (which can span days or weeks), the excess ICP is inaccessible to the user.

The existing test suite confirms the capping behavior but does not assert that excess ICP is returned:

```rust
// Should only get 5 as that's the max per participant.
assert_eq!(
    swap.buyers.get(&TEST_USER1_PRINCIPAL.to_string()).unwrap().amount_icp_e8s(),
    5 * E8
);
``` [4](#0-3) 

The canister entry point `refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` exposes this function directly to any caller: [5](#0-4) 

---

### Impact Explanation

Any user who transfers more ICP than `max_participant_icp_e8s` to their swap subaccount and calls `refresh_buyer_tokens` will have their excess ICP locked for the duration of the swap. SNS swaps can remain OPEN for days or weeks. During this window the user:

- Cannot retrieve the excess ICP via `error_refund_icp` (blocked by lifecycle guard)
- Cannot re-use the locked ICP elsewhere (opportunity cost, liquidity loss)
- Has no on-chain mechanism to recover funds until the swap finalizes

If the swap is never finalized due to a separate bug, the ICP could be locked indefinitely. The magnitude of loss scales with the excess amount and the swap duration.

---

### Likelihood Explanation

The attack path requires no privilege: any unprivileged principal can transfer ICP to their swap subaccount and call `refresh_buyer_tokens`. Users commonly overshoot participation caps (e.g., sending a round number that exceeds the per-participant limit), and the function provides no warning or immediate refund. The open TODO (`NNS1-1682`) confirms the team is aware this path exists and has not yet been remediated. Likelihood is **medium** — it requires a user to send excess ICP, which is a realistic and common mistake.

---

### Recommendation

Implement the acknowledged TODO `NNS1-1682`: after computing `actual_increment_e8s`, if `requested_increment_e8s > actual_increment_e8s`, immediately initiate an ICP ledger transfer to return the difference (`requested_increment_e8s - actual_increment_e8s`) to the buyer's principal account. This should be attempted before returning `RefreshBuyerTokensResponse`, analogous to how `error_refund_icp` performs the transfer after the swap ends.

---

### Proof of Concept

1. An SNS swap is in the OPEN state with `max_participant_icp_e8s = 5 * E8` and `min_participant_icp_e8s = 1 * E8`.
2. User transfers `10 * E8` ICP to `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&user)) }`.
3. User calls `refresh_buyer_tokens({ buyer: "", confirmation_text: None })`.
4. `refresh_buyer_token_e8s` reads the ledger balance as `10 * E8`, computes `new_balance_e8s = min(10 * E8, 5 * E8) = 5 * E8`, and records `5 * E8` as the buyer's participation.
5. The remaining `5 * E8` ICP stays in the subaccount with no refund issued.
6. User immediately calls `error_refund_icp({ source_principal_id: Some(user) })`.
7. The call returns: `"Error refunds can only be performed when the swap is ABORTED or COMMITTED"`.
8. The user's `5 * E8` ICP is locked for the entire remaining duration of the swap. [6](#0-5) [7](#0-6)
