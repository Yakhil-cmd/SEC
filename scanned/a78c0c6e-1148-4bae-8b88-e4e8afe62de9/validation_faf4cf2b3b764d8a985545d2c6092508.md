### Title
ICP Ledger `icrc2_allowance` Traps Instead of Returning Zero When ICRC-2 Feature Is Disabled — (`rs/ledger_suite/icp/ledger/src/main.rs`)

---

### Summary

The ICP Ledger's `icrc2_allowance` query endpoint calls `trap()` when `feature_flags.icrc2 = false`, instead of returning a zero allowance. The ICRC-2 standard requires this endpoint to return the current allowance for any caller. When ICRC-2 is disabled, no approvals can exist, so the correct return value is `Allowance { allowance: 0, expires_at: None }`. Trapping is a non-compliant response that breaks any integrator relying on this query.

---

### Finding Description

In `rs/ledger_suite/icp/ledger/src/main.rs`, the `icrc2_allowance` query function unconditionally traps when the ICRC-2 feature flag is false:

```rust
#[query]
fn icrc2_allowance(arg: AllowanceArgs) -> Allowance {
    if !LEDGER.read().unwrap().feature_flags.icrc2 {
        trap("ICRC-2 features are not enabled on the ledger.");
    }
    let from = AccountIdentifier::from(arg.account);
    let spender = AccountIdentifier::from(arg.spender);
    get_allowance(from, spender)
}
``` [1](#0-0) 

The ICRC-2 standard mandates that `icrc2_allowance` is a pure read query that returns the current allowance. When ICRC-2 is disabled, no approvals can have been created, so the allowance is definitionally zero. The endpoint should return `Allowance { allowance: 0, expires_at: None }` rather than trapping.

By contrast, the ICRC-1 ledger's `icrc2_allowance` correctly handles this case — it does not gate on the feature flag and always returns the stored allowance (zero if none exists):

```rust
#[query]
fn icrc2_allowance(arg: AllowanceArgs) -> Allowance {
    Access::with_ledger(|ledger| {
        let now = TimeStamp::from_nanos_since_unix_epoch(ic_cdk::api::time());
        let allowance = ledger.approvals().allowance(&arg.account, &arg.spender, now);
        Allowance {
            allowance: allowance.amount.into(),
            expires_at: allowance.expires_at.map(|t| t.as_nanos_since_unix_epoch()),
        }
    })
}
``` [2](#0-1) 

The test helper `expect_icrc2_disabled` confirms the ICP Ledger's trap behavior is present and tested, but the test accepts it as expected rather than flagging it as non-compliant: [3](#0-2) 

The `icrc2_allowance` endpoint is permanently exposed in the ICP Ledger's Candid interface regardless of the feature flag state, meaning any caller can reach it and trigger the trap: [4](#0-3) 

---

### Impact Explanation

Any integrator or canister that calls `icrc2_allowance` on the ICP Ledger while `feature_flags.icrc2 = false` receives a `CanisterCalledTrap` rejection instead of a zero allowance. This breaks:

- Allowance-checking logic that gates downstream operations (e.g., "check allowance, then call `icrc2_transfer_from`")
- Wallets or DeFi protocols that query allowances to display balances or validate state
- Any canister that treats a non-error response from `icrc2_allowance` as a precondition for further execution

Because the endpoint is always present in the interface, integrators have no way to distinguish "ICRC-2 disabled" from "ICRC-2 enabled but allowance is zero" without separately calling `icrc1_supported_standards`. Protocols that skip this check — a common pattern — will encounter unexpected traps.

---

### Likelihood Explanation

The ICP Ledger on mainnet currently has ICRC-2 enabled. However:

1. The `feature_flags.icrc2` field can be set to `false` via an upgrade argument (`UpgradeArgs { feature_flags: Some(FeatureFlags { icrc2: false }), ... }`), which any controller of the ledger can trigger.
2. SNS-deployed token ledgers derived from the same codebase may be initialized with `icrc2: false`.
3. Any unprivileged ingress caller can call `icrc2_allowance` at any time; no special role is required to trigger the trap. [5](#0-4) 

---

### Recommendation

Change `icrc2_allowance` in the ICP Ledger to return a zero allowance when ICRC-2 is disabled, rather than trapping:

```rust
#[query]
fn icrc2_allowance(arg: AllowanceArgs) -> Allowance {
    if !LEDGER.read().unwrap().feature_flags.icrc2 {
        return Allowance {
            allowance: Nat::from(0u64),
            expires_at: None,
        };
    }
    let from = AccountIdentifier::from(arg.account);
    let spender = AccountIdentifier::from(arg.spender);
    get_allowance(from, spender)
}
```

This aligns the ICP Ledger's behavior with the ICRC-1 ledger's `icrc2_allowance` implementation and with the ICRC-2 standard's requirement that the query always returns a value.

---

### Proof of Concept

1. Deploy the ICP Ledger with `feature_flags: FeatureFlags { icrc2: false }`.
2. Call `icrc2_allowance` with any valid `AllowanceArgs`.
3. Observe: the call returns `CanisterCalledTrap` with message `"ICRC-2 features are not enabled on the ledger."` instead of `Allowance { allowance: 0, expires_at: None }`.

The existing test helper `expect_icrc2_disabled` already demonstrates this trap behavior: [6](#0-5)
