### Title
Silent Permanent Loss of Neuron Maturity Rewards When Reward Distribution Scheduling Fails - (File: `rs/nns/governance/src/reward/distribution.rs`)

### Summary

In the NNS governance canister, when `schedule_pending_rewards_distribution` fails to insert a reward distribution into stable memory (because an entry for that `day_after_genesis` already exists), the error is only logged and silently swallowed. The calling function `distribute_voting_rewards_to_neurons` then unconditionally proceeds to mark all settled proposals as `Settled` and update `latest_reward_event`, permanently closing the window for any retry. The calculated rewards for all voting neurons in that round are irrecoverably lost.

### Finding Description

`distribute_voting_rewards_to_neurons` orchestrates the NNS reward cycle in three sequential steps:

1. **Write to stable memory**: `schedule_pending_rewards_distribution` inserts the computed `RewardsDistribution` into the `REWARDS_DISTRIBUTION_STATE_MACHINE` (`StableBTreeMap` backed by stable memory).
2. **Write to heap memory**: Proposals are marked `Settled` and `latest_reward_event` is updated. [1](#0-0) 

The insertion in step 1 is guarded by a duplicate-key check: [2](#0-1) 

When `add_rewards_distribution` returns `Err`, `schedule_pending_rewards_distribution` only emits a `println!` and returns normally — the `Result` is not propagated: [3](#0-2) 

`distribute_voting_rewards_to_neurons` has no awareness of this failure and continues unconditionally to settle proposals and advance `latest_reward_event`: [4](#0-3) 

**Root cause — stable vs. heap memory divergence on trap**: On the Internet Computer, stable memory writes are **not** rolled back when a canister message traps; only heap memory is reverted. If `distribute_voting_rewards_to_neurons` traps (e.g., instruction limit exceeded, panic in `process_proposal` or `record_known_neuron_abstentions`) after `schedule_pending_rewards_distribution` has written to stable memory but before `latest_reward_event` is updated on the heap, the canister enters a split state:

- **Stable memory**: distribution for day N is present.
- **Heap memory**: `latest_reward_event.day_after_genesis` still equals N−1.

On the next invocation of `CalculateDistributableRewardsTask`: [5](#0-4) 

`calculate_voting_rewards` sees `latest_reward_event` at day N−1 and proceeds. `schedule_pending_rewards_distribution` is called again for day N, `add_rewards_distribution` returns `Err` (key already exists), the error is logged and dropped, and execution continues. Proposals are marked `Settled` and `latest_reward_event` is advanced to day N. The newly computed distribution (dist2) is permanently discarded; the stale distribution (dist1, from the failed prior attempt) remains in the state machine and is what gets processed — with potentially different supply figures, different proposal sets, and different per-neuron amounts than what the `RewardEvent` records.

The `RewardsDistributionStateMachine` is stored in stable memory at `REWARDS_DISTRIBUTION_STATE_MACHINE_MEMORY_ID`: [6](#0-5) 

The CHANGELOG itself records a prior rollback of neuron migration "due to issues with reward distribution," confirming that this stable/heap divergence class of bug has manifested in production: [7](#0-6) 

### Impact Explanation

All neurons that voted on proposals settled in the affected round permanently lose their maturity increment. Because proposals are marked `Settled` and `latest_reward_event` is advanced, there is no mechanism to retry or recover the dropped distribution. The `distributed_e8s_equivalent` field in the recorded `RewardEvent` will also be inconsistent with the maturity actually credited to neurons, breaking the accounting invariant of the governance ledger.

### Likelihood Explanation

The trigger requires `distribute_voting_rewards_to_neurons` to trap after the stable memory write but before the heap write. This can occur due to instruction-limit exhaustion (the function iterates over all settled proposals and calls `process_proposal` for each), a panic in `record_known_neuron_abstentions`, or any other unexpected condition. The CHANGELOG evidence of a prior reward-distribution rollback indicates this class of failure is not purely theoretical. Likelihood is **low** but non-zero, and the impact when triggered is permanent and unrecoverable without a governance upgrade.

### Recommendation

1. **Propagate the error**: `schedule_pending_rewards_distribution` should return `Result` and `distribute_voting_rewards_to_neurons` should abort (not settle proposals, not advance `latest_reward_event`) if the insertion fails.
2. **Atomic state update**: Consider updating `latest_reward_event` in stable memory (not heap) so that both the distribution and the event marker are either both present or both absent after a trap.
3. **Idempotent retry**: If a distribution for `day_after_genesis` already exists in stable memory when `distribute_voting_rewards_to_neurons` is called for the same day, merge or replace it rather than silently dropping the new computation.

### Proof of Concept

1. Governance canister is running normally; `latest_reward_event.day_after_genesis = N−1` (heap).
2. `CalculateDistributableRewardsTask` fires, fetches `total_supply`, calls `distribute_voting_rewards_to_neurons`.
3. `schedule_pending_rewards_distribution(N, dist1)` succeeds — stable memory now contains entry for day N.
4. `distribute_voting_rewards_to_neurons` traps (e.g., instruction limit) before reaching line 6852.
5. Heap is rolled back: `latest_reward_event.day_after_genesis` reverts to N−1. Stable memory retains day N entry.
6. `CalculateDistributableRewardsTask` fires again. `calculate_voting_rewards` returns `Some` (day N−1 < N).
7. `schedule_pending_rewards_distribution(N, dist2)` → `add_rewards_distribution` returns `Err("already exists")` → logged, dropped.
8. Proposals settled, `latest_reward_event` advanced to N.
9. `DistributeRewardsTask` processes dist1 (stale). dist2 is gone. Neurons receive wrong amounts; `RewardEvent` records different amounts. No recovery path exists. [2](#0-1) [3](#0-2) [1](#0-0)

    fn add_rewards_distribution(
        &mut self,
        day_after_genesis: u64,
        distribution: RewardsDistribution,
    ) -> Result<(), String> {
        if self.distributions.contains_key(&day_after_genesis) {
            return Err(format!(
                "{LOG_PREFIX}Rewards distribution already exists for day_after_genesis: {day_after_genesis}"
            ));
        }
        self.distributions.insert(
            day_after_genesis,
            RewardsDistributionInProgress::from(distribution),
        );
        Ok(())
    }

impl RecurringAsyncTask for CalculateDistributableRewardsTask {
    async fn execute(self) -> (Duration, Self) {
        let total_supply = self
            .governance
            .with_borrow(|governance| governance.get_ledger())
            .total_supply()
            .await;
        match total_supply {
            Ok(total_supply) => {
                self.governance.with_borrow_mut(|governance| {
                    governance.distribute_voting_rewards_to_neurons(total_supply);
                });
            }
            Err(err) => {
                println!(
                    "{}Error when getting total ICP supply: {}",
                    LOG_PREFIX,
                    GovernanceError::from(err)
                )
            }
        }

        let next_run = self.next_reward_task_from_now();
        (next_run, self)
    }
