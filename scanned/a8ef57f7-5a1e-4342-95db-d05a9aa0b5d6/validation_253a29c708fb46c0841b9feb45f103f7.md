### Title
NNS Governance Maturity Disbursement and Neuron Spawning Frozen When XRC Oracle Is Unavailable — (File: `rs/nns/governance/src/governance/disburse_maturity.rs`)

### Summary
When the Exchange Rate Canister (XRC) is unavailable or returns invalid/zero rates, `heap_data.maturity_modulation.current_value_permyriad` remains `None` in NNS governance. Both `try_finalize_maturity_disbursement` and `maybe_spawn_neurons` hard-block on this field being `Some`, causing all pending maturity disbursements and neuron spawning to be frozen — even though user maturity has already been deducted at initiation time.

### Finding Description

The NNS governance canister's maturity disbursement finalization path reads the maturity modulation from `heap_data.maturity_modulation`, which is populated exclusively by the `UpdateIcpXdrRateRelatedData` timer task via calls to the XRC.

In `try_finalize_maturity_disbursement`: [1](#0-0) 

The extracted `maturity_modulation` (which is `None` when XRC has never successfully returned a rate) is passed directly to `next_maturity_disbursement_to_finalize`: [2](#0-1) 

This returns `Err(FinalizeMaturityDisbursementError::NoMaturityModulation)` immediately, aborting the finalization with no fallback. [3](#0-2) 

The same hard-block exists in `maybe_spawn_neurons`: [4](#0-3) 

The `heap_data.maturity_modulation` field is populated only by `UpdateIcpXdrRateRelatedData`, which calls the XRC: [5](#0-4) 

If the XRC call fails (network error, exchange downtime, HTTPS outcalls disabled on the subnet), the task returns `None` and the field is never set: [6](#0-5) 

The XRC itself depends on HTTPS outcalls to external cryptocurrency exchanges: [7](#0-6) 

On a fresh NNS governance canister, `heap_data.maturity_modulation` starts as `None`. If the XRC is unavailable during the initial backfill period, the field is never populated and all maturity disbursements are permanently blocked until the XRC recovers.

Critically, maturity is **deducted from the neuron at initiation time** (not at finalization): [8](#0-7) 

This means a user's maturity is already gone from their neuron, but the ICP mint never fires — funds are locked in `maturity_disbursements_in_progress` with no recourse until the XRC recovers.

The test suite explicitly confirms this failure mode: [9](#0-8) 

### Impact Explanation

Users who have initiated maturity disbursements (a 7-day delayed operation) find their maturity already deducted but ICP never minted when the XRC is unavailable. The funds are locked in `maturity_disbursements_in_progress` with no user-accessible escape hatch. Similarly, neurons with `spawn_at_timestamp_seconds` set cannot be spawned. The longer the XRC is unavailable, the larger the backlog of frozen disbursements. Unlike the Blueberry case (protocol insolvency), the funds are not permanently lost — but they are inaccessible for the duration of the outage, which could span days or weeks if the XRC's upstream HTTPS outcall sources are disrupted.

### Likelihood Explanation

The XRC aggregates rates via HTTPS outcalls to external exchanges. Any sustained failure of those outcalls (exchange downtime, HTTPS outcall feature disruption on the XRC subnet, or a bug in the XRC canister itself) causes the XRC to return `CryptoBaseAssetNotFound`, `StablecoinRateTooFewRates`, or similar errors. On a freshly deployed or state-reset NNS governance canister, `heap_data.maturity_modulation` starts as `None` and requires a successful XRC round-trip to become `Some`. The window of vulnerability is the period between canister deployment and the first successful XRC fetch.

### Recommendation

1. **Fallback to last-known value**: When `current_value_permyriad` is `None`, fall back to the most recently cached value from the old `cached_daily_maturity_modulation_basis_points` field, or use `0` (no modulation) as a safe default.
2. **Decouple finalization from oracle availability**: Allow maturity disbursements to finalize with a default modulation of `0` when the XRC has been unavailable beyond a configurable staleness threshold, rather than blocking indefinitely.
3. **Preserve maturity on finalization failure**: If finalization is blocked, ensure the maturity is restored to the neuron so the user is not left in a deducted-but-unminted state.

### Proof of Concept

1. Deploy a fresh NNS governance canister (or simulate with `heap_data.maturity_modulation = None`).
2. Stop the XRC canister or disable HTTPS outcalls on its subnet so `UpdateIcpXdrRateRelatedData` always fails.
3. A user calls `disburse_maturity` — maturity is immediately deducted from the neuron and stored in `maturity_disbursements_in_progress`.
4. After 7 days, the `FinalizeMaturityDisbursement` timer fires.
5. `try_finalize_maturity_disbursement` reads `maturity_modulation = None`, calls `next_maturity_disbursement_to_finalize` with `None`, which returns `Err(NoMaturityModulation)`.
6. No ICP is minted. The user's maturity is gone and their ICP is not delivered. The timer retries indefinitely with no resolution until the XRC recovers.
