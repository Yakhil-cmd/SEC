### Title
Stale Supply Snapshot Used for Multi-Round Reward Backfill Causes Over-Distribution of Voting Rewards - (`rs/nns/governance/src/governance.rs`, `rs/nns/governance/src/timer_tasks/calculate_distributable_rewards.rs`)

---

### Summary

When the NNS governance canister misses one or more reward distribution rounds, it backfills all missed rounds using the **current** ICP total supply fetched at the moment of calculation. Because the ICP supply grows over time (via CMC minting, etc.), applying the current (inflated) supply retroactively to past rounds causes neurons to receive more maturity than they should have earned during those periods. This is the direct IC analog of the reported `_currentAccountedNativeRewards()` bug: a live balance/supply value is used to compute rewards for past time intervals.

---

### Finding Description

The reward distribution task in `rs/nns/governance/src/timer_tasks/calculate_distributable_rewards.rs` performs an async cross-canister call to fetch the current ICP total supply, then passes it synchronously into `distribute_voting_rewards_to_neurons`:

```rust
// calculate_distributable_rewards.rs lines 52-71
async fn execute(self) -> (Duration, Self) {
    let total_supply = self
        .governance
        .with_borrow(|governance| governance.get_ledger())
        .total_supply()   // <-- fetches CURRENT supply at time of call
        .await;
    match total_supply {
        Ok(total_supply) => {
            self.governance.with_borrow_mut(|governance| {
                governance.distribute_voting_rewards_to_neurons(total_supply);
            });
        }
        ...
    }
}
```

Inside `calculate_voting_rewards` in `rs/nns/governance/src/governance.rs`, when multiple rounds have been missed (`new_rounds_count > 1`), the same single `supply` value is applied uniformly across **all** missed rounds:

```rust
// governance.rs lines 6647-6654
let fraction: f64 = days
    .map(crate::reward::calculation::rewards_pool_to_distribute_in_supply_fraction_for_one_day)
    .sum();   // sum of fractions for ALL missed days

let total_available_e8s_equivalent_float = (supply.get_e8s() as f64) * fraction
    + rolling_over_from_previous_reward_event_e8s_equivalent as f64;
```

Here `days = last_event_day_after_genesis..day_after_genesis` spans all missed rounds, and `supply` is the current supply fetched moments ago — not the historical supply at the time each past round should have been settled.

The same pattern exists in SNS governance `distribute_rewards` in `rs/sns/governance/src/governance.rs`:

```rust
// sns/governance.rs lines 5859-5871
let supply = i2d(supply.get_e8s());
for i in 1..=new_rounds_count {
    // ...
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
    // same current supply applied to every missed round
}
```

The code even explicitly logs when this backfill occurs:

```
"More than one reward round (i.e. days) has passed since the last RewardEvent.
 This could mean that rewards are being rolled over, or earlier rounds were missed."
```

---

### Impact Explanation

When reward distribution is delayed (e.g., during a canister upgrade, a timer failure, or a rollover period) and the ICP supply has grown in the interim (via CMC minting, spawning neurons, etc.), the backfill calculation applies the current inflated supply to all missed past rounds. This causes neurons to receive more maturity (e8s equivalent) than they would have if distribution had occurred on schedule. The over-distribution is bounded by:

```
over_distribution ≈ (supply_growth_e8s) × (sum_of_fractions_for_missed_rounds)
```

For example, if 2 days are missed and the supply grew by 1% during that period, neurons receive ~1% more maturity for those 2 days than they should. Since maturity is eventually converted to ICP via spawning, this represents a small but real inflation of the ICP supply beyond the intended schedule.

**Impact: Low** — The over-distribution is bounded by the supply growth rate and the number of missed rounds. It does not allow arbitrary minting, but it does cause the actual reward distribution to deviate from the protocol-specified schedule.

---

### Likelihood Explanation

Missed reward rounds are explicitly handled by the code and acknowledged in comments and logs. They occur during:
- Canister upgrades (governance canister is stopped briefly)
- Timer failures or scheduling delays
- Rollover rounds (no proposals to settle)

The ICP supply grows continuously via CMC minting and neuron spawning. Therefore, the conditions for this bug (missed rounds + supply growth) are regularly met in normal operation.

**Likelihood: Medium** — Missed rounds are uncommon but documented and expected. Supply growth is continuous.

---

### Recommendation

Mirror the fix suggested in the original report: snapshot the supply at the boundary of each reward round rather than using the live supply for all rounds. Concretely:

- Store a `supply_at_last_reward_event` field in governance state, updated each time a reward event is committed.
- When backfilling missed rounds, use the stored snapshot for past rounds and the current supply only for the current round.
- Alternatively, accept the small error (as the original report suggests) and document it explicitly, since the spec already acknowledges supply variability.

---

### Proof of Concept

1. At day N, ICP supply = 500M e8s. Reward distribution runs normally.
2. Governance canister is upgraded; timer is delayed for 2 days.
3. During those 2 days, CMC mints 50M ICP → supply = 550M e8s.
4. At day N+2, `CalculateDistributableRewardsTask::execute()` fires:
   - Fetches `total_supply()` → 550M e8s (current).
   - Calls `calculate_voting_rewards(550M)`.
   - `days = N..N+2` → `new_rounds_count = 2`.
   - `fraction = fraction_for_day_N + fraction_for_day_N+1` (sum for both missed days).
   - `total_available = 550M × fraction` — but day N's reward should have been `500M × fraction_for_day_N`.
5. Neurons receive maturity computed on 550M for both days, instead of 500M for day N and 550M for day N+1.
6. Over-distribution for day N ≈ `50M × fraction_for_day_N` e8s equivalent in maturity. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    fn calculate_voting_rewards(
        &self,
        supply: Tokens,
    ) -> Option<(RewardEvent, Option<RewardsDistribution>)> {
        let now = self.env.now();

        let latest_reward_event = self.latest_reward_event();

        // Which reward rounds (i.e. days) require rewards? (Usually, there is
        // just one of these, but we support rewarding many consecutive rounds.)
        let day_after_genesis =
            (now - self.heap_data.genesis_timestamp_seconds) / REWARD_DISTRIBUTION_PERIOD_SECONDS;
        let last_event_day_after_genesis = latest_reward_event.day_after_genesis;
        let days = last_event_day_after_genesis..day_after_genesis;
        let new_rounds_count = days.clone().count();

        if new_rounds_count == 0 {
            // This may happen, in case consider_distributing_rewards was called
            // several times at almost the same time. This is
            // harmless, just abandon.
            return None;
        }

        if new_rounds_count > 1 {
            println!(
                "{}More than one reward round (i.e. days) has passed since the last \
                 RewardEvent. This could mean that rewards are being rolled over, \
                 or earlier rounds were missed. It is now {} full days since \
                 IC genesis, and the last distribution nominally happened at {} \
                 full days since IC genesis.",
                LOG_PREFIX, day_after_genesis, last_event_day_after_genesis
            );
        }

        // Only used for metrics.
        let latest_day_fraction: f64 = days
            .clone()
            .last()
            .map(|day| {
                crate::reward::calculation::rewards_pool_to_distribute_in_supply_fraction_for_one_day(day)
            })
            .unwrap_or(0.0);
        let latest_round_available_e8s_equivalent_float =
            (supply.get_e8s() as f64) * latest_day_fraction;

        let fraction: f64 = days
            .map(crate::reward::calculation::rewards_pool_to_distribute_in_supply_fraction_for_one_day)
            .sum();

        let rolling_over_from_previous_reward_event_e8s_equivalent =
            latest_reward_event.e8s_equivalent_to_be_rolled_over();
        let total_available_e8s_equivalent_float = (supply.get_e8s() as f64) * fraction
            + rolling_over_from_previous_reward_event_e8s_equivalent as f64;
