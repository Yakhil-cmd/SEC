### Title
Unsupported ERC-20 Tokens Deposited via Helper Contract Are Permanently Locked in Minter's Ethereum Address - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

The `DepositHelperWithSubaccount.sol` Ethereum helper contract accepts any ERC-20 token via `depositErc20()` without enforcing a whitelist of supported tokens. The IC ckETH minter, however, only scrapes and processes deposit logs for ERC-20 tokens that are explicitly supported (whitelisted). Any ERC-20 token deposited via the helper contract that is not on the minter's supported list is silently ignored: the ERC-20 tokens are transferred to the minter's Ethereum address and permanently locked there, with no ckERC-20 minted and no recovery path.

---

### Finding Description

**Deposit side (Ethereum):** The `depositErc20` function in `DepositHelperWithSubaccount.sol` accepts any ERC-20 contract address and any amount. It performs a `safeTransferFrom` to move the tokens to the minter's Ethereum address and emits a `ReceivedEthOrErc20` event — with no check that the ERC-20 token is supported by the IC minter. [1](#0-0) 

**Withdrawal side (IC minter):** The minter's log scraping for the old ERC-20 helper contract (`ReceivedErc20LogScraping`) filters at the Ethereum topic level — it only fetches events whose ERC-20 contract address matches a supported token. Unsupported ERC-20 tokens are never fetched. [2](#0-1) 

For the new combined helper contract (`ReceivedEthOrErc20LogScraping`), the minter's own documentation explicitly acknowledges the gap:

> *"Note that the helper smart contract does not enforce any whitelist of allowed ERC-20 tokens. This is enforced by the minter, which fetches logs only for the supported ERC-20 tokens. Therefore, funds of unsupported ERC-20 tokens could be deposited via the helper smart contract, but the minter will not know anything about it."* [3](#0-2) 

The minter's internal state transition also asserts that any ERC-20 event it processes must correspond to a supported token — meaning if an unsupported token somehow reached `record_event_to_mint`, the minter would panic. The design therefore relies entirely on filtering at the scraping layer, with no fallback or recovery path. [4](#0-3) 

There is no endpoint, governance proposal path, or minter function that allows recovery of ERC-20 tokens held at the minter's Ethereum address for unsupported token types.

---

### Impact Explanation

A user who deposits an ERC-20 token that is not on the minter's supported list (e.g., a newly listed token not yet added via NNS proposal, or a token the user mistakenly believes is supported) will:

1. Permanently lose their ERC-20 tokens — they are transferred to the minter's Ethereum address and cannot be recovered.
2. Receive no ckERC-20 tokens on the IC side.
3. Have no recourse: the minter has no withdrawal or sweep function for unsupported ERC-20 balances.

This is a **ledger conservation bug / chain-fusion locked-funds bug**: the deposit side (Ethereum helper contract) accepts a class of input (unsupported ERC-20 tokens) that the IC minter side permanently ignores, with no recovery mechanism.

---

### Likelihood Explanation

- The helper contract is publicly callable by any Ethereum user.
- The list of supported ERC-20 tokens changes over time via NNS governance proposals. A token that is unsupported today may be added later, but deposits made before the token was added are already lost.
- Users interacting with the helper contract directly (e.g., via Etherscan's write interface) may not check `get_minter_info` first.
- The documentation warning exists but is easy to miss for users interacting programmatically or via third-party frontends.

---

### Recommendation

1. **Preferred fix (Ethereum side):** Add a whitelist of supported ERC-20 contract addresses to `DepositHelperWithSubaccount.sol` and revert `depositErc20` calls for unsupported tokens. The minter's address should be the authoritative source for this whitelist (queried off-chain and updated via contract upgrade or governance).

2. **Fallback fix (IC minter side):** Add a privileged `sweep_unsupported_erc20` endpoint to the minter that allows the minter's controller (NNS) to issue an Ethereum transaction transferring any unsupported ERC-20 balance held at the minter's address back to a designated recovery address.

---

### Proof of Concept

```
1. User checks minter info: supported_ckerc20_tokens = [USDC, USDT, WBTC, ...]
2. User deposits TOKEN_X (not in supported list) via:
   DepositHelperWithSubaccount.depositErc20(
       TOKEN_X_address,   // unsupported ERC-20
       1_000_000,
       encoded_principal,
       subaccount
   )
3. TOKEN_X is transferred to minter's Ethereum address. ReceivedEthOrErc20 event emitted.
4. IC minter scrapes logs. ReceivedEthOrErc20LogScraping processes the event.
   - TOKEN_X is not in ckerc20_tokens → event is filtered/ignored.
5. No ckTOKEN_X is minted. No error is returned to the user.
6. User's 1_000_000 TOKEN_X is permanently locked in the minter's Ethereum address.
7. No recovery path exists on the IC minter side.
``` [1](#0-0) [2](#0-1) [3](#0-2)

        address erc20Address,
        uint256 amount,
        bytes32 principal,
        bytes32 subaccount
    ) public {
        require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
        IERC20 erc20Token = IERC20(erc20Address);
        erc20Token.safeTransferFrom(
            msg.sender,
            minterAddress,
            amount
        );

        emit ReceivedEthOrErc20(
            erc20Address,
            msg.sender,
            amount,
            principal,
            subaccount
        );
    }

    fn next_scrape(state: &State) -> Option<Scrape> {
        if state.ckerc20_tokens.is_empty() {
            return None;
        }
        let contract_address = *Self::contract_address(state)?;
        let last_scraped_block_number = Self::last_scraped_block_number(state);

        let mut topics: Vec<_> = vec![Topic::Single(Hex32::from(RECEIVED_ERC20_EVENT_TOPIC))];
        // We add token contract addresses as additional topics to match.
        // It has a disjunction semantics, so it will match if event matches any one of these addresses.
        topics.push(
            erc20_smart_contracts_addresses_as_topics(state)
                .collect::<Vec<_>>()
                .into(),
        );

        Some(Scrape {
            contract_address,
            last_scraped_block_number,
            topics,
        })
    }
