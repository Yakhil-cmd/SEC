### Title
Merkle Root Guard Short-Circuits on Empty `txdata`, Allowing Cache Poisoning by Malicious Bitcoin Peer — (`rs/bitcoin/adapter/src/blockchainstate.rs`)

---

### Summary

The `add_block` guard in `BlockchainState` uses a compound condition that short-circuits when `txdata` is empty, bypassing merkle root validation entirely. A malicious Bitcoin peer can respond to a solicited `getdata` request with a structurally stripped block (valid header, empty `txdata`) to poison the adapter's block cache with an invalid block that will never be re-fetched.

---

### Finding Description

The guard at line 237 of `blockchainstate.rs` is:

```rust
if block.compute_merkle_root().is_some() && !block.check_merkle_root() {
    return Err(AddBlockError::InvalidMerkleRoot(block_hash));
}
``` [1](#0-0) 

`compute_merkle_root()` returns `Option<TxMerkleNode>` — it returns `None` when `txdata` is empty. [2](#0-1) 

When `txdata: vec![]`, `compute_merkle_root().is_some()` evaluates to `false`, short-circuiting the `&&`. The block is never rejected regardless of what the header's `merkle_root` field contains. The block then proceeds unconditionally to be serialized and inserted into `block_cache`. [3](#0-2) 

The `received_block_message` handler in `blockchainmanager.rs` gates on the block hash being present in `getdata_request_info`, and **removes** that entry on first receipt:

```rust
let request = match self.getdata_request_info.remove(&block_hash) {
    Some(request) => request,
    None => return Err(ReceivedBlockMessageError::UnknownBlock),
};
``` [4](#0-3) 

Because the block hash is derived from the header (not from `txdata`), a stripped block with the same header has the same hash. Once the malicious peer's response is accepted, the `getdata_request_info` entry is consumed. `enqueue_new_blocks_to_download` will not re-queue the block because `get_block(&hash)` now returns `Some(...)`:

```rust
if self.blockchain.get_block(&hash).is_none()
    && !self.block_sync_queue.contains(&hash)
    && !self.getdata_request_info.contains_key(&hash)
{
    self.block_sync_queue.insert(hash);
}
``` [5](#0-4) 

The invalid block is permanently cached and will be forwarded to the Bitcoin canister. The correct block is never re-fetched.

The existing test `test_sorted_tip` confirms the guard permits empty-`txdata` blocks through — it calls `add_block` with `txdata: Vec::new()` and asserts `Ok(())`: [6](#0-5) 

---

### Impact Explanation

A malicious Bitcoin peer that the adapter has connected to can respond to any solicited `getdata` request with a stripped block. The adapter permanently caches the invalid block, stops re-fetching it, and forwards it to the Bitcoin canister. Depending on whether the canister independently validates the merkle root, this causes either:

- A **persistent stall**: the canister rejects the invalid block but the adapter never re-fetches the correct one, halting Bitcoin block ingestion at that height.
- **Delivery of a structurally invalid block** to the replica if the canister does not independently re-validate the merkle root.

---

### Likelihood Explanation

The adapter connects to external Bitcoin peers by design. Any peer in the adapter's peer list (including a peer an operator was socially-engineered into adding, or a peer on a permissive testnet peer list) can trigger this with a single crafted `block` P2P message in response to a legitimate `getdata`. No PoW mining is required — the attacker reuses a header the adapter already accepted and strips the transactions.

---

### Recommendation

Replace the compound guard with a direct call to `check_merkle_root()`, which already handles the `None` case by returning `false`:

```rust
// bitcoin::Block::check_merkle_root() returns false when txdata is empty
if !block.check_merkle_root() {
    return Err(AddBlockError::InvalidMerkleRoot(block_hash));
}
```

This correctly rejects both: (a) non-empty `txdata` with a mismatched merkle root, and (b) empty `txdata` with any non-zero merkle root in the header.

---

### Proof of Concept

Unit test against `blockchainstate::add_block`:

```rust
#[tokio::test]
async fn test_empty_txdata_with_nonzero_merkle_root_is_rejected() {
    let state = BlockchainState::new(
        Network::Regtest,
        None,
        &MetricsRegistry::default(),
        no_op_logger(),
    );
    let genesis = state.genesis();
    // Build a header whose merkle_root is non-zero (e.g. all-ones)
    let mut header = generate_header(genesis.block_hash(), genesis.time, 0);
    header.merkle_root = TxMerkleNode::from_raw_hash(bitcoin::hashes::Hash::all_zeros());
    // Craft a block with that header but no transactions
    let bad_block = Block { header, txdata: vec![] };
    let result = state.add_block(bad_block).await;
    // Currently passes (Ok(())) — should be Err(InvalidMerkleRoot)
    assert!(
        matches!(result, Err(AddBlockError::InvalidMerkleRoot(_))),
        "Expected rejection, got: {:?}", result
    );
}
```

Under the current code this assertion **fails** — `add_block` returns `Ok(())` and caches the invalid block.
