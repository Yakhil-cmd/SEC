### Title
Coinbase Transaction Input Causes Permanent Retriable Error Loop in ckBTC Checker ŌĆö (`rs/bitcoin/checker/src/state.rs`, `rs/bitcoin/checker/src/fetch.rs`)

---

### Summary

The ckBTC checker canister has no coinbase-awareness. When `check_transaction` is called with the txid of a coinbase transaction, `TransactionCheckData::from_transaction` stores the coinbase input as `PreviousOutput{txid:[0;32], vout:0xFFFFFFFF}`. `check_fetched` then unconditionally attempts to fetch that phantom txid from external providers. Every provider returns a non-200 response (404), which is classified as a **retriable** `TransientInternalError`. The status for txid `[0;32]` cycles through all three providers on each retry and never transitions to an irrecoverable error, creating a permanent retry loop that blocks the deposit and drains minter cycles.

---

### Finding Description

**Step 1 ŌĆö No coinbase guard in `from_transaction`** [1](#0-0) 

All inputs, including the coinbase sentinel (`previous_output.txid = [0;32]`, `vout = 0xFFFFFFFF`), are stored verbatim as `PreviousOutput`. There is no `is_coinbase()` check anywhere in the checker codebase (confirmed: zero matches for "coinbase" across all checker source files).

**Step 2 ŌĆö `check_fetched` unconditionally fetches every input** [2](#0-1) 

For every input whose address is `None`, `try_fetch_tx(input.txid)` is called. For a coinbase transaction, `input.txid` is `[0;32]`.

**Step 3 ŌĆö 404 from providers maps to a retriable error** [3](#0-2) 

Any non-200 HTTP status (including 404 for the non-existent txid `[0;32]`) becomes `HttpGetTxError::Rejected`. [4](#0-3) 

`Rejected` maps to `CheckTransactionRetriable::TransientInternalError` ŌĆö a **retriable**, not irrecoverable, response.

**Step 4 ŌĆö Error state cycles through providers indefinitely** [5](#0-4) 

When `try_fetch_tx` sees `FetchTxStatus::Error`, it advances to the next provider and retries. With three providers (Btcscan, Blockstream, MempoolSpace), all of which return 404 for txid `[0;32]`, the cycle never terminates and never produces an irrecoverable result.

---

### Impact Explanation

- The coinbase UTXO deposit is permanently stuck: the checker never returns `Passed` or an irrecoverable `Error`, only `TransientInternalError`.
- The ckBTC minter retries `check_transaction` indefinitely, paying the `CHECK_TRANSACTION_CYCLES_SERVICE_FEE` on every call.
- A single coinbase UTXO deposit is sufficient to trigger this; the attacker does not need to repeat the action.

---

### Likelihood Explanation

Coinbase UTXOs mature after 100 blocks and are freely spendable. Any miner (or anyone who acquires a mature coinbase output) can send it to the ckBTC minter deposit address. This is an unprivileged, on-chain action requiring no special access. The attack is cheap to execute and the loop is self-sustaining once triggered.

---

### Recommendation

In `TransactionCheckData::from_transaction`, detect coinbase inputs using the Bitcoin protocol sentinel values (`txid == [0;32]` and `vout == 0xFFFFFFFF`) and either:

1. **Skip them** ŌĆö do not add them to `inputs`, so `check_fetched` never tries to fetch them; or
2. **Return an irrecoverable error** ŌĆö if the transaction being checked *is itself* a coinbase transaction, immediately return `CheckTransactionIrrecoverableError` (e.g., `InvalidTransaction("coinbase transactions cannot be checked")`).

Option 2 is safer because it surfaces the condition explicitly to the minter, allowing it to handle coinbase deposits with a defined policy rather than silently looping. [6](#0-5) 

---

### Proof of Concept

1. Obtain a mature coinbase UTXO (any mined block reward after 100 confirmations).
2. Send it to the ckBTC minter deposit address on mainnet (or testnet).
3. The minter calls `check_transaction(coinbase_txid)`.
4. The checker fetches the coinbase transaction successfully (it exists on-chain).
5. `check_fetched` calls `try_fetch_tx([0;32])`.
6. All three providers return 404 ŌåÆ `TransientInternalError`.
7. On every subsequent `check_transaction` call, `try_fetch_tx` sees `FetchTxStatus::Error` for `[0;32]`, advances to the next provider, fetches again, gets 404 again.
8. The response is always `TransientInternalError`; the deposit never resolves; cycles drain on every retry.

An integration test can reproduce this by injecting a mock coinbase transaction into the checker's `check_transaction` endpoint and asserting the response is `CheckTransactionIrrecoverableError` rather than `TransientInternalError`.

    pub fn from_transaction(
        btc_network: &BtcNetwork,
        tx: Transaction,
    ) -> Result<Self, bitcoin::address::FromScriptError> {
        let inputs = tx
            .input
            .iter()
            .map(|input| PreviousOutput {
                txid: Txid::from(*(input.previous_output.txid.as_ref() as &[u8; 32])),
                vout: input.previous_output.vout,
            })
            .collect();
        let mut outputs = Vec::new();
        for output in tx.output.iter() {
            // Some outputs do not have addresses. These outputs will never be
            // inputs of other transactions, so it is okay to treat them as `None`.
            outputs.push(
                Address::from_script(
                    &output.script_pubkey,
                    bitcoin::Network::from(btc_network.clone()),
                )
                .ok(),
            )
        }
        Ok(Self { inputs, outputs })
    }
