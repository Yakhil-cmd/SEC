### Title
Missing `last_eth_scraped_block_number` in `UpgradeArg` Causes Permanent ETH Deposit Loss When Helper Contract Address Is Changed — (File: `rs/ethereum/cketh/minter/src/lifecycle/upgrade.rs`)

---

### Summary

The ckETH minter's `UpgradeArg` allows changing `ethereum_contract_address` (the ETH helper smart contract on Ethereum) but provides **no corresponding field to reset the ETH scraping position** (`last_eth_scraped_block_number`). When the ETH helper contract address is changed to a contract deployed at a block number lower than the current `last_scraped_block_number`, all ETH deposits made to the new contract in the already-scraped block range are permanently missed and never minted as ckETH. This is a direct analog of the FantiumMinterV1 issue: changing a contract reference without clearing/resetting the associated state that depends on it.

---

### Finding Description

The `UpgradeArg` struct exposes three contract-address/scraping-position pairs:

| Contract | Address field | Block-reset field |
|---|---|---|
| ETH helper | `ethereum_contract_address` | **MISSING** |
| ERC-20 helper | `erc20_helper_contract_address` | `last_erc20_scraped_block_number` ✓ |
| Deposit-with-subaccount helper | `deposit_with_subaccount_helper_contract_address` | `last_deposit_with_subaccount_scraped_block_number` ✓ | [1](#0-0) 

The ERC-20 and deposit-with-subaccount scrapings each have a paired `last_*_scraped_block_number` field so operators can reset the scraping cursor when the contract address changes. The ETH scraping has no such field.

In `State::upgrade()`, when `ethereum_contract_address` is provided, only `set_contract_address()` is called — the `last_scraped_block_number` for `LogScrapingId::EthDepositWithoutSubaccount` is left unchanged: [2](#0-1) 

Compare with the ERC-20 path, which correctly allows resetting the block cursor: [3](#0-2) 

The `LogScrapingState` stores `last_scraped_block_number` independently per scraping ID, and the minter resumes scraping from `last_scraped_block_number + 1`: [4](#0-3) 

There is no mechanism in `UpgradeArg` to reset the ETH scraping cursor. The Candid interface confirms the asymmetry: [5](#0-4) 

---

### Impact Explanation

**Vulnerability class:** Chain-fusion mint/burn/replay bug — missed mint.

If the ETH helper contract address is changed to a contract deployed at Ethereum block N, and the minter's current `last_scraped_block_number` for `EthDepositWithoutSubaccount` is M ≥ N:

1. The minter resumes scraping from block M+1.
2. All `ReceivedEth` events emitted by the new contract in blocks N through M are **permanently skipped** — they will never appear in `events_to_mint` and will never be minted.
3. Users who deposited ETH to the new contract in that block range have their ETH locked in the Ethereum helper contract with no corresponding ckETH ever minted on the IC side. The funds are unrecoverable without a further governance intervention.

The `minted_events` and `events_to_mint` maps are keyed by `EventSource` (transaction hash + log index), so there is no deduplication path that would catch these missed events later. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The trigger is an NNS governance proposal that changes `ethereum_contract_address`. This is a legitimate, expected operation (it has already occurred in production — see the mainnet upgrade history). The dangerous scenario arises when:

- The new contract is deployed at a block number lower than the current scraping position (e.g., when reverting to a previous contract version, or when a significant delay exists between contract deployment and the governance proposal passing and being executed).

The real-world upgrade history shows that `ethereum_contract_address` has been changed in production proposals: [7](#0-6) 

Each such upgrade is a potential trigger. The window of missed blocks equals the difference between the current scraping position and the new contract's deployment block.

---

### Recommendation

Add a `last_eth_scraped_block_number : opt nat` field to `UpgradeArg` (both in the Candid interface and the Rust struct), and handle it in `State::upgrade()` analogously to `last_erc20_scraped_block_number`:

```rust
if let Some(block_number) = last_eth_scraped_block_number {
    self.log_scrapings.set_last_scraped_block_number(
        LogScrapingId::EthDepositWithoutSubaccount,
        BlockNumber::try_from(block_number)...,
    );
}
```

This mirrors the existing pattern for ERC-20 and deposit-with-subaccount scrapings and allows operators to correctly reset the ETH scraping cursor whenever the helper contract address is changed.

---

### Proof of Concept

1. Current minter state: `last_scraped_block_number` for `EthDepositWithoutSubaccount` = **21,000,000**.
2. A new ETH helper contract is deployed on Ethereum at block **20,900,000**.
3. Users deposit ETH to the new contract between blocks 20,900,000 and 21,000,000 (100,000 blocks of deposits).
4. An NNS governance proposal passes, upgrading the minter with `UpgradeArg { ethereum_contract_address: Some("0xNewContract..."), ... }`. There is no field to set `last_eth_scraped_block_number = 20_900_000`.
5. After upgrade, `post_upgrade` replays events and applies the upgrade arg. The ETH scraping cursor remains at 21,000,000.
6. The minter resumes scraping from block 21,000,001 for the new contract.
7. All `ReceivedEth` events from the new contract in blocks 20,900,000–21,000,000 are permanently skipped.
8. Affected users' ETH is locked in the new Ethereum helper contract; no ckETH is ever minted for them. [8](#0-7) [9](#0-8)

```rust
pub fn post_upgrade(upgrade_args: Option<UpgradeArg>) {
    let start = ic_cdk::api::instruction_counter();

    STATE.with(|cell| {
        *cell.borrow_mut() = Some(replay_events());
    });
    if let Some(args) = upgrade_args {
        mutate_state(|s| process_event(s, EventType::Upgrade(args)))
    }

    let end = ic_cdk::api::instruction_counter();

    let event_count = total_event_count();
    let instructions_consumed = end - start;

    log!(
        INFO,
        "[upgrade]: replaying {event_count} events consumed {instructions_consumed} instructions ({} instructions per event on average)",
        instructions_consumed / event_count
    );
```

```text
type UpgradeArg = record {
    // Change the nonce of the next transaction to be sent to the Ethereum network.
    next_transaction_nonce : opt nat;
```

```markdown
## Motivation

This proposal upgrades the ckETH minter to support ledger subaccounts on the minter:
* for deposits:
    * A new helper smart contract was deployed that can be used both for depositing ETH or ERC20 while specifying a full ledger account (principal and subaccount).
    * The minter scraps the logs of the new helper smart contract at the same frequency as the other ones.
    * The previous helper smart contracts are marked as deprecated since the new contract can handle all types of deposits supported by the minter. Future proposals may decide to reduce the frequency at which the logs of the deprecated helper smart contracts are being scraped.
* for withdrawals: subaccounts can be specified when withdrawing ckETH or ckERC20.
```
