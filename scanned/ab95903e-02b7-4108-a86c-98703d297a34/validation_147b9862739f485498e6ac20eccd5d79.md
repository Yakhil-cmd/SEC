### Title
Incorrect Upgrade Lock Order in SNS Governance Allows Concurrent Upgrade Proposals to Bypass Guard - (File: `rs/sns/governance/src/governance.rs`)

### Summary

In `perform_upgrade_to_next_sns_version_legacy`, the `pending_version` upgrade lock is set **after** all async inter-canister upgrade calls complete, rather than before them. This means the `check_no_upgrades_in_progress` guard is ineffective during the async window, allowing two concurrently executing `UpgradeSnsToNextVersion` proposals to both bypass the guard and proceed simultaneously. The newer automatic upgrade path (`initiate_upgrade_if_sns_behind_target_version`) correctly sets the lock **before** the async call, making the inconsistency between the two code paths the root cause.

### Finding Description

In `perform_upgrade_to_next_sns_version_legacy`:

```
check_no_upgrades_in_progress(...)   // ← guard checks pending_version == None
...
get_upgrade_params(...).await        // ← yields control; pending_version still None
...
get_wasm(...).await                  // ← yields control; pending_version still None
...
upgrade_non_root_canister(...).await // ← yields control; pending_version still None
...
self.proto.pending_version = Some(PendingVersion { ... }); // ← lock set AFTER
``` [1](#0-0) 

During every `await` point, the IC canister runtime can process other incoming messages. If a second `UpgradeSnsToNextVersion` proposal execution is spawned (via `spawn_in_canister_env`) and begins while the first is awaiting an inter-canister call, it will also call `check_no_upgrades_in_progress`, find `pending_version == None`, and proceed with its own upgrade.

By contrast, `initiate_upgrade_if_sns_behind_target_version` correctly sets `pending_version` **before** the async upgrade call:

```rust
self.proto.pending_version = Some(PendingVersion { ... }); // ← lock set BEFORE
let upgrade_attempt = self.upgrade_sns_framework_canister(...).await;
``` [2](#0-1) 

This inconsistency between the two upgrade paths is the root cause. The `pending_version` field is the sole upgrade-in-progress lock: [3](#0-2) 

Proposal execution is spawned as a background future: [4](#0-3) 

### Impact Explanation

Two concurrent `UpgradeSnsToNextVersion` proposals can both pass `check_no_upgrades_in_progress` and execute simultaneously. This results in:

1. Two sequential WASM installs on the same SNS canister (e.g., ledger or governance), potentially with different WASM hashes if the proposals target different versions.
2. The `pending_version` field being set twice — the second write overwrites the first, causing `check_upgrade_status` to track the wrong target version.
3. The `deployed_version` being updated to an incorrect version after `check_upgrade_status` resolves, permanently corrupting the SNS's version tracking state.
4. Subsequent upgrade proposals being blocked or incorrectly evaluated because `deployed_version` no longer reflects reality. [5](#0-4) 

### Likelihood Explanation

SNS governance allows any neuron holder to submit `UpgradeSnsToNextVersion` proposals. If two such proposals are adopted (reach voting threshold) within the same voting period — which can happen legitimately when the SNS-W upgrade path has multiple pending steps — both are spawned as background futures via `start_proposal_execution`. The async window during `upgrade_non_root_canister` (which involves stop → install_code → start inter-canister calls through root) spans multiple IC rounds, providing a realistic interleaving window. No malicious governance majority is required; the condition arises from normal concurrent proposal adoption.

### Recommendation

Set `self.proto.pending_version` **before** the first async call in `perform_upgrade_to_next_sns_version_legacy`, mirroring the pattern already used in `initiate_upgrade_if_sns_behind_target_version`. If the subsequent upgrade calls fail, clear `pending_version` in the error path (as `initiate_upgrade_if_sns_behind_target_version` already does at line 5650). [6](#0-5) 

### Proof of Concept

1. SNS has `deployed_version = V1` and `pending_version = None`.
2. Two `UpgradeSnsToNextVersion` proposals (P1 targeting V2, P2 targeting V3) are adopted simultaneously.
3. Both call `perform_upgrade_to_next_sns_version_legacy` via `spawn_in_canister_env`.
4. P1 passes `check_no_upgrades_in_progress` (pending_version is None), begins `get_upgrade_params(...).await`.
5. During P1's await, P2 begins execution, also passes `check_no_upgrades_in_progress` (pending_version still None).
6. P2 calls `upgrade_non_root_canister` for V3 while P1 is still upgrading to V2.
7. P1 completes, sets `pending_version = {target: V2}`.
8. P2 completes, overwrites `pending_version = {target: V3}`.
9. `check_upgrade_status` runs, sees running version = V3 (from P2's install), marks upgrade as succeeded, sets `deployed_version = V3`.
10. P1's proposal is never marked executed (its `proposal_id` was in the overwritten `pending_version`), leaving a permanently adopted-but-unresolved proposal and `deployed_version` skipping V2 entirely — corrupting the SNS upgrade journal and blocking future step-by-step upgrades.     fn complete_sns_upgrade_to_next_version(
        &mut self,
        proposal_id: Option<u64>,
        status: upgrade_journal_entry::upgrade_outcome::Status,
        message: String,
        deployed_version: Option<Version>,
    ) {
        use upgrade_journal_entry::upgrade_outcome::Status;

        let result = match &status {
            Status::Success(_) => Ok(()),
            Status::InvalidState(_) => Err(GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                message.to_string(),
            )),
            Status::ExternalFailure(_) | Status::Timeout(_) => Err(
                GovernanceError::new_with_message(ErrorType::External, message.to_string()),
            ),
        };

        self.push_to_upgrade_journal(upgrade_journal_entry::UpgradeOutcome {
            human_readable: Some(message),
            status: Some(status),
        });

        if let Some(proposal_id) = proposal_id {
            self.set_proposal_execution_status(proposal_id, result);
        }

        self.proto.pending_version = None;

        if let Some(deployed_version) = deployed_version {
            self.proto.deployed_version.replace(deployed_version);
        }
