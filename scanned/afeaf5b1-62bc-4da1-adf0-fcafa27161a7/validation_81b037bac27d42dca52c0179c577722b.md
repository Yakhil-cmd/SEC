### Title
Missing Staleness Check on ICP/XDR Conversion Rate Before Minting Cycles - (`rs/nns/cmc/src/main.rs`)

### Summary

The Cycles Minting Canister (CMC) uses a stored `icp_xdr_conversion_rate` to convert ICP to cycles in `tokens_to_cycles`. This function checks only whether the rate is `None`, but never checks whether the rate's `timestamp_seconds` is recent. When the XRC update pipeline is disabled (e.g., after a `DivergedRate` governance proposal) or persistently failing, the stored rate can become arbitrarily stale. Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during this window will have their ICP converted at the stale rate, causing over-minting (loss to the protocol) or under-minting (loss to the user).

### Finding Description

`validate_exchange_rate` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` checks source counts (the validity metadata) before a rate is stored: [1](#0-0) 

`update_exchange_rate` in `rs/nns/cmc/src/exchange_rate_canister.rs` correctly calls `validate_exchange_rate` before storing a new rate: [2](#0-1) 

However, `tokens_to_cycles` in `rs/nns/cmc/src/main.rs` reads the stored rate and uses it directly, with no check on `rate.timestamp_seconds`: [3](#0-2) 

`do_set_icp_xdr_conversion_rate` only enforces that a new rate has a strictly greater timestamp than the current one — it does not enforce that the stored rate is recent relative to the current canister time: [4](#0-3) 

The XRC update pipeline can be permanently halted by a governance proposal carrying `DivergedRate`: [5](#0-4) 

Once `UpdateExchangeRateState::Disabled` is set, the CMC stops polling the XRC entirely. The stored rate is frozen at whatever value it held at the time of disablement. Governance must submit a manual `set_icp_xdr_conversion_rate` proposal to unfreeze it. During the gap between disablement and the governance proposal, `tokens_to_cycles` continues to accept and process `notify_top_up` / `notify_create_canister` / `notify_mint_cycles` calls using the frozen rate.

The three public entry points that call `tokens_to_cycles` are: [6](#0-5) [7](#0-6) [8](#0-7) 

### Impact Explanation

If ICP's market price moves significantly while the stored rate is frozen:

- **Over-minting (loss to the protocol):** If ICP price falls after the rate is frozen, the stale rate is too high. Users receive more cycles per ICP than the current market warrants. Cycles are minted against a burned ICP balance, so the protocol permanently over-issues cycles relative to the true ICP/XDR value.
- **Under-minting (loss to users):** If ICP price rises after the rate is frozen, users receive fewer cycles than they should. ICP is still burned, so users suffer a direct financial loss.

Both outcomes are analogous to the Synthetix report's under-liquidation / over-liquidation impact.

### Likelihood Explanation

The `DivergedRate` governance path is an explicitly designed operational mode, not a theoretical edge case. It is triggered whenever the XRC-reported rate diverges from the governance-submitted rate. The governance proposal execution and the subsequent manual rate-correction proposal are not atomic — there is always a window (potentially hours to days, depending on governance voting periods) during which the rate is frozen. Any user who submits a `notify_top_up` transaction during this window is affected. The entry path requires no special privileges: `notify_top_up` is a public update call. [9](#0-8) 

### Recommendation

Add a maximum-age guard in `tokens_to_cycles` (or in the callers) that rejects conversions when `now_seconds() - rate.timestamp_seconds` exceeds a defined threshold (e.g., `2 * REFRESH_RATE_INTERVAL_SECONDS` or a governance-configurable value). Return a retriable `NotifyError` so users are not permanently locked out, but cycles are not minted against a stale rate.

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) => {
                let age = now_seconds().saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: format!(
                            "ICP/XDR conversion rate is stale (age: {}s)", age
                        ),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(NotifyError::Other { ... })
        }
    })
}
```

### Proof of Concept

1. Governance submits a `UpdateIcpXdrConversionRatePayload` with `reason = DivergedRate`. This sets `update_exchange_rate_canister_state = Disabled`.
2. The CMC stops polling the XRC. `state.icp_xdr_conversion_rate.timestamp_seconds` is now frozen.
3. ICP market price drops 30% over the next 48 hours (governance voting period for the corrective proposal).
4. An attacker sends ICP to the CMC subaccount and calls `notify_top_up`. `tokens_to_cycles` reads the frozen rate (30% too high) and mints 30% more cycles than the current ICP/XDR value warrants.
5. The attacker receives over-minted cycles; the ICP is burned. The protocol has permanently issued excess cycles. [3](#0-2) [5](#0-4)
