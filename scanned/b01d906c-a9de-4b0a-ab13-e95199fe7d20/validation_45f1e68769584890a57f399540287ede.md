### Title
UTF-8 Byte Count vs. Character Count Confusion in Token Symbol/Name Validation Allows Single-Character Tokens — (`rs/nervous_system/common/src/ledger_validation.rs`)

### Summary

Both `validate_token_symbol` and `validate_token_name` use Rust's `.len()` on `&str`, which returns the **UTF-8 byte count**, not the Unicode scalar value (character) count. This allows a submitter to craft a token symbol or name consisting of a single multibyte character that satisfies the byte-length minimum while containing far fewer visible characters than intended.

### Finding Description

In `rs/nervous_system/common/src/ledger_validation.rs`, both validation functions compare `.len()` (bytes) against the minimum thresholds: [1](#0-0) [2](#0-1) 

The constants are defined as: [3](#0-2) 

**Concrete bypass cases:**

| Input | UTF-8 bytes (`.len()`) | Unicode chars (`.chars().count()`) | Passes? |
|---|---|---|---|
| `"€"` (U+20AC) as symbol | 3 | 1 | ✅ (3 >= 3) |
| `"😀"` (U+1F600) as name | 4 | 1 | ✅ (4 >= 4) |
| `"€€"` as symbol | 6 | 2 | ✅ (6 >= 3) |

The error message itself says `"given character count: {}"` but reports bytes — a misleading diagnostic that confirms the intent was to count characters.

### Impact Explanation

There are **two reachable paths**:

1. **SNS creation**: `SnsInitPayload::validate_pre_execution()` and `validate_post_execution()` both delegate to `ledger_validation::validate_token_symbol/name`. A crafted `SnsInitPayload` with a single-emoji token symbol or name passes all programmatic validation and proceeds to ledger initialization. [4](#0-3) 

2. **Post-deployment SNS governance (`ManageLedgerParameters`)**: An SNS neuron holder can submit a governance proposal to change the token symbol/name. The same buggy validation is applied: [5](#0-4) 

In both cases, the SNS ledger is initialized or updated with a token symbol/name that has fewer visible characters than the stated minimum invariant. This enables confusion and impersonation attacks — e.g., a token with symbol `"€"` (1 visible char) passes the "3 character minimum" check.

### Likelihood Explanation

The exploit requires no privileged access. Any SNS creator submitting an `SnsInitPayload`, or any SNS neuron holder submitting a `ManageLedgerParameters` proposal, can trigger this. The NNS/SNS governance voters may trust the programmatic validation and not scrutinize the byte-vs-char distinction.

### Recommendation

Replace `.len()` with `.chars().count()` in both minimum-length checks:

```rust
// validate_token_symbol
if token_symbol.chars().count() < MIN_TOKEN_SYMBOL_LENGTH { ... }
if token_symbol.chars().count() > MAX_TOKEN_SYMBOL_LENGTH { ... }

// validate_token_name
if token_name.chars().count() < MIN_TOKEN_NAME_LENGTH { ... }
if token_name.chars().count() > MAX_TOKEN_NAME_LENGTH { ... }
```

Also fix the error message to say "character count" accurately.

### Proof of Concept

fn test_single_multibyte_char_bypasses_minimum() {
    // "€" = U+20AC = 3 UTF-8 bytes, 1 visible char
    // Should fail MIN_TOKEN_SYMBOL_LENGTH == 3 (chars), but passes because .len() == 3 (bytes)
    assert!(validate_token_symbol("€").is_err(), "single € should fail symbol min");

    // "😀" = U+1F600 = 4 UTF-8 bytes, 1 visible char
    // Should fail MIN_TOKEN_NAME_LENGTH == 4 (chars), but passes because .len() == 4 (bytes)
    assert!(validate_token_name("😀").is_err(), "single 😀 should fail name min");
}

Both assertions currently **fail** (i.e., the functions return `Ok(())` for these inputs), confirming the bypass.
