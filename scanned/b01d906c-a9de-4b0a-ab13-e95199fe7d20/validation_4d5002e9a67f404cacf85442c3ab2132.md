### Title
Inverted Boundary Check in `SwapParametersError` Allows Equal Min/Max Participation Limits to Silently Pass Validation â€” (`File: rs/nns/governance/src/neurons_fund.rs`)

---

### Summary

The `SwapParticipationLimits::validate()` function in the NNS Governance canister uses strict `>` comparisons to detect invalid min/max participation bounds. This means that when `min_direct_participation_icp_e8s == max_direct_participation_icp_e8s` or `min_participant_icp_e8s == max_participant_icp_e8s`, the validation passes silently. However, the error variant names (`MaxIsLessThanOrEqualMinParticipationIcp`, `MinIsLessThanOrEqualMaxParticipantIcp`) and their display messages explicitly state that the invariant being enforced is `min <= max` â€” meaning the code's intent is to reject the equal case, but the implementation does not.

---

### Finding Description

In `rs/nns/governance/src/neurons_fund.rs`, the `SwapParticipationLimits::validate()` function is:

```rust
pub fn validate(&self) -> Result<(), SwapParametersError> {
    if self.min_direct_participation_icp_e8s > self.max_direct_participation_icp_e8s {
        return Err(SwapParametersError::MaxIsLessThanOrEqualMinParticipationIcp { ... });
    }
    if self.min_participant_icp_e8s > self.max_participant_icp_e8s {
        return Err(SwapParametersError::MinIsLessThanOrEqualMaxParticipantIcp { ... });
    }
    Ok(())
}
```

The error variant names are `MaxIsLessThanOrEqualMinParticipationIcp` and `MinIsLessThanOrEqualMaxParticipantIcp`. Their `Display` implementations say:

> `"invariant violated: min_direct_participation_icp_e8s ({min}) <= max_direct_participation_icp_e8s ({max})."`

This message is printed when `min <= max` is violated â€” i.e., the intent is to reject the case where `min >= max` (including equality). But the guard condition `min > max` only catches the strict case, allowing `min == max` to pass validation.

The equal case (`min == max`) is a degenerate swap configuration: a swap where the minimum and maximum direct participation are identical means there is exactly one valid participation amount, and a swap where `min_participant_icp_e8s == max_participant_icp_e8s` means every participant must contribute exactly that amount. While the tests in `neurons_fund_participation_constraints_tests.rs` explicitly exercise the equal case and show it passes, the error variant naming and display text reveal the developer's intent was to reject it. The `try_from_swap_parameters` function calls `validate()` and is the entry point used when the NNS Governance canister processes a `CreateServiceNervousSystem` proposal that triggers Neurons' Fund matched funding: [4](#0-3) 

---

### Impact Explanation

If `min_direct_participation_icp_e8s == max_direct_participation_icp_e8s`, the Neurons' Fund matched-funding computation proceeds with a degenerate range. The `compute_linear_scaling_coefficients` function and downstream matched-funding math operate on this range. The test `compute_linear_scaling_coefficients_max_min_direct_participation_eqaul` (note the typo in the test name) confirms this case is reachable and produces non-trivial output â€” meaning the Neurons' Fund will compute and commit maturity for a swap with a single valid direct-participation amount. [5](#0-4) 

The impact is a governance/ledger conservation bug: the Neurons' Fund can be committed to participate in a degenerate SNS swap (one where the participation window is a single point), which the protocol's own error naming indicates should be rejected. This can result in maturity being drawn from NNS neurons for a swap that was intended to be invalid.

---

### Likelihood Explanation

Any unprivileged principal can submit a `CreateServiceNervousSystem` NNS proposal. The proposal payload includes `SwapParameters` with `minimum_direct_participation_icp` and `maximum_direct_participation_icp`. If a proposer sets these equal, the `try_from_swap_parameters` validation passes, and the Neurons' Fund participation is computed and committed at proposal execution time. This is a realistic, low-effort misconfiguration that bypasses the intended guard. [6](#0-5) 

---

### Recommendation

Change the strict `>` comparisons to `>=` in `SwapParticipationLimits::validate()` to match the intent expressed by the error variant names and display messages:

```diff
- if self.min_direct_participation_icp_e8s > self.max_direct_participation_icp_e8s {
+ if self.min_direct_participation_icp_e8s >= self.max_direct_participation_icp_e8s {
     return Err(SwapParametersError::MaxIsLessThanOrEqualMinParticipationIcp { ... });
 }
- if self.min_participant_icp_e8s > self.max_participant_icp_e8s {
+ if self.min_participant_icp_e8s >= self.max_participant_icp_e8s {
     return Err(SwapParametersError::MinIsLessThanOrEqualMaxParticipantIcp { ... });
 }
```

Update the test `compute_linear_scaling_coefficients_max_min_direct_participation_eqaul` and `compute_linear_scaling_coefficients_max_min_participant_icp_equal` to expect an error result instead of a successful participation. [7](#0-6) 

---

### Proof of Concept

1. Submit a `CreateServiceNervousSystem` NNS proposal with `SwapParameters` where `minimum_direct_participation_icp == maximum_direct_participation_icp` (e.g., both set to `100_000_000_000` e8s).
2. `SwapParticipationLimits::try_from_swap_parameters` is called at proposal execution; `validate()` passes because `100_000_000_000 > 100_000_000_000` is `false`.
3. `NeuronsFundParticipation::new_impl` proceeds with the degenerate range.
4. Neurons' Fund maturity is drawn from NNS neurons for a swap with a single valid participation point â€” a configuration the error variant name `MaxIsLessThanOrEqualMinParticipationIcp` was designed to reject. [2](#0-1) [5](#0-4)
