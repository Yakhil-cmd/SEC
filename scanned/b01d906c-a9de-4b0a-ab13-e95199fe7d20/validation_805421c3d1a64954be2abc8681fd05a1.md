### Title
SNS Governance `VotingRewardsParameters` Change Retroactively Impacts Already-Elapsed Unsettled Reward Rounds - (`rs/sns/governance/src/governance.rs`)

---

### Summary

In the SNS governance canister, `VotingRewardsParameters` (including `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, `reward_rate_transition_duration_seconds`, and `round_duration_seconds`) can be updated at any time via a `ManageNervousSystemParameters` governance proposal. When `distribute_rewards` is subsequently called, it reads the **current** parameters and applies them to **all** elapsed rounds since the last reward event — including rounds that elapsed before the parameter change. No reward settlement occurs before the parameter update is committed. This is a direct structural analog to the Zaros retrospective funding-rate bug.

---

### Finding Description

`perform_manage_nervous_system_parameters` immediately overwrites `self.proto.parameters` with the new `NervousSystemParameters` (including `voting_rewards_parameters`) without first triggering a reward settlement: [1](#0-0) 

The next time `distribute_rewards` is invoked (via `run_periodic_tasks`), it reads the **current** (post-change) `voting_rewards_parameters`: [2](#0-1) 

It then computes `new_rounds_count` using the **current** `round_duration_seconds`, which may differ from the value in effect during those elapsed rounds: [3](#0-2) 

Finally, it loops over every elapsed round and calls `reward_rate_at` with the **current** `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, and `reward_rate_transition_duration_seconds`: [4](#0-3) 

`reward_rate_at` itself is entirely parameterised by the current `VotingRewardsParameters`: [5](#0-4) 

The codebase itself acknowledges the hazard in a comment on the field: [6](#0-5) 

> *"Once this is set, it probably should not be changed, because the results would probably be pretty confusing."*

Despite this warning, no enforcement or pre-settlement guard exists in `perform_manage_nervous_system_parameters`.

---

### Impact Explanation

Any SNS governance majority that passes a `ManageNervousSystemParameters` proposal altering `voting_rewards_parameters` will cause the new rate schedule to be applied retroactively to all reward rounds that elapsed since the last `RewardEvent`. Concretely:

- **Increasing `initial_reward_rate_basis_points`** inflates the rewards purse for already-elapsed rounds, diluting the token supply beyond what neuron holders anticipated when they voted.
- **Decreasing `initial_reward_rate_basis_points`** deflates the rewards purse, depriving voters of maturity they accrued under the prior schedule.
- **Changing `round_duration_seconds`** alters `new_rounds_count` for the elapsed window, compressing or expanding the number of rounds counted and therefore the total rewards distributed.
- **Changing `reward_rate_transition_duration_seconds`** shifts the quadratic decay curve, changing the effective rate for every past round in the unsettled window.

The effect is proportional to the gap between the last `RewardEvent` and the parameter change. SNS reward rounds can be up to one year long (`MAX_REWARD_ROUND_DURATION_SECONDS`), so the unsettled window can be large. [7](#0-6) 

---

### Likelihood Explanation

The entry path is a standard SNS governance proposal submitted by any token holder with sufficient stake — explicitly a "ledger/governance/chain-fusion user" entry point. No admin key, leaked secret, or threshold-majority corruption is required. The governance majority may act in good faith (e.g., adjusting inflation policy) without realising the retroactive effect on already-elapsed rounds. The longer the SNS has been running without a reward event (e.g., due to no settled proposals), the larger the unsettled window and the greater the impact.

---

### Recommendation

Before committing new `voting_rewards_parameters` in `perform_manage_nervous_system_parameters`, the canister should trigger an immediate reward settlement (equivalent to calling `distribute_rewards` with the **old** parameters). This ensures that all rounds elapsed under the prior schedule are settled before the new schedule takes effect, matching the principle of "settle before changing parameters."

Alternatively, each `RewardEvent` could snapshot the `VotingRewardsParameters` in effect at the time it was generated, so that future backfill calculations use the historically correct parameters rather than the current ones.

---

### Proof of Concept

1. Deploy an SNS with `initial_reward_rate_basis_points = 200` (2 %) and `round_duration_seconds = ONE_WEEK`.
2. Advance time by 3 weeks without triggering a reward event (e.g., no settled proposals, so rewards roll over).
3. Pass a `ManageNervousSystemParameters` proposal setting `initial_reward_rate_basis_points = 2000` (20 %).
4. Advance time by one more second so `should_distribute_rewards` returns `true`.
5. Observe that `distribute_rewards` calculates the rewards purse for all 3 elapsed weeks at the new 20 % rate, not the 2 % rate that was in effect during those weeks.

The relevant loop in `distribute_rewards` iterates `i in 1..=new_rounds_count` and calls `voting_rewards_parameters.reward_rate_at(...)` using the post-change parameters for every past round: [4](#0-3)

    pub fn reward_rate_at(&self, now: Instant) -> RewardRate {
        let reward_rate_transition_duration_seconds = self
            .reward_rate_transition_duration_seconds
            .expect("reward_rate_transition_duration_seconds unset");

        let time_since_genesis = {
            let result = now - *GENESIS;
            // For the purposes of determining reward rate, treat times before
            // genesis the same as at genesis. This is not expected to occur in
            // practice. This code is just being extra defensive.
            if result.as_secs() < i2d(0) {
                Duration { days: i2d(0) }
            } else {
                result
            }
        };
        if reward_rate_transition_duration_seconds == 0
            || time_since_genesis.as_secs() >= i2d(reward_rate_transition_duration_seconds)
        {
            return self.final_reward_rate();
        }

        // s linearly varies from 1 -> 0 as seconds_since_genesis varies from 0
        // to reward_rate_transition_duration_seconds.
        let transition = LinearMap::new(
            dec!(0)..i2d(reward_rate_transition_duration_seconds),
            dec!(1)..dec!(0),
        );
        let s = transition.apply(time_since_genesis.as_secs());
        // s2 varies quadratically from 1 -> 0 (again, as seconds_since_genesis
        // varies from 0 to reward_rate_transition_duration_seconds), and
        // flattens out as seconds_since_genesis approaches
        // reward_rate_transition_duration_seconds.
        let s2 = s * s;

        // This looks backwards, but we think of variable rate as being added to
        // final growth rate, not initial, and the amount to add is up to
        // initial - final (where initial is thought of as being greater than
        // final).
        let dr = self.initial_reward_rate() - self.final_reward_rate();
        // variable_reward_rate varies from dr to 0 as round varies from
        // 1 to transition_round_count.
        let variable_reward_rate = s2 * dr;

        self.final_reward_rate() + variable_reward_rate
    }
