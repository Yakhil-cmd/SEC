### Title
Unenforced Access Control in `icrc103_get_allowances` Allows Any Caller to Enumerate Any Account's Full Allowance Table — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
The `icrc103_get_allowances` query endpoint accepts an arbitrary `from_account` argument and returns all allowances for that account without verifying that the caller is the account owner. The `AccessDenied` error variant and the `icrc103:public_allowances` metadata flag both exist in the codebase but are never consulted at runtime.

### Finding Description
At lines 1215–1231 of `rs/ledger_suite/icrc1/ledger/src/main.rs`, the handler:

```rust
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    // ... computes max_results ...
    Ok(get_allowances(from_account, arg.prev_spender, max_results, ic_cdk::api::time()))
}
```

There is no check of the form `if caller != from_account.owner && !public_allowances { return Err(GetAllowancesError::AccessDenied {...}) }`. Any principal can supply `from_account: Some(victim_account)` and receive the complete paginated allowance list for that account. Repeated calls with `prev_spender` set to the last returned spender enumerate the full table.

The ICRC-103 standard design intent is that `icrc103:public_allowances = false` (or absent) means only the account owner may query their own allowances. The implementation ignores this entirely.

### Impact Explanation
- **Direct impact:** Complete allowance graph disclosure for any account on the ledger. An attacker learns every (spender, amount, expiry) tuple for any target account.
- **Corrected financial impact claim:** The question's claim of direct `transfer_from` drains exceeding $1M is **overstated**. Knowing an allowance exists does not grant the ability to execute `transfer_from` — the attacker must already be the approved spender. The real impact is a **privacy/information disclosure** violation: the attacker can identify which accounts have large outstanding approvals to which spenders, enabling targeted social engineering, front-running, or exploitation of compromised spender keys.

### Likelihood Explanation
The endpoint is a public query call — no cycles, no authentication, no rate limit beyond `max_take_allowances`. Any IC principal can call it immediately. Pagination makes full enumeration trivial.

### Recommendation
Add a caller-identity guard before invoking `get_allowances`:

```rust
let caller = ic_cdk::api::msg_caller();
let public = Access::with_ledger(|l| l.public_allowances());
if !public && caller != from_account.owner {
    return Err(GetAllowancesError::AccessDenied {
        reason: "caller is not the account owner".to_string(),
    });
}
```

Also ensure `icrc1_metadata` correctly reflects the `icrc103:public_allowances` value so clients can reason about the ledger's policy.

### Proof of Concept
1. Deploy the ICRC-1 ledger canister.
2. Account A approves spender S for 1000 tokens via `icrc2_approve`.
3. From any unrelated principal P (P ≠ A.owner), call:
   ```
   icrc103_get_allowances({ from_account: Some(A), prev_spender: None, take: None })
   ```
4. Observe that the response is `Ok([{from_account: A, to_spender: S, allowance: 1000, ...}])` with no `AccessDenied` error, regardless of the `icrc103:public_allowances` metadata value.
