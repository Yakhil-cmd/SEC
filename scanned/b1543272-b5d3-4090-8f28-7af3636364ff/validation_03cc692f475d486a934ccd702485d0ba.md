### Title
Time-Seeded PRNG with Constant Padding Used as Global `getrandom` Implementation for UUID Generation - (File: `rs/boundary_node/rate_limits/canister/random.rs`)

---

### Summary
The rate-limit canister registers a custom `getrandom` implementation that seeds a `ChaCha20Rng` with only 8 bytes of IC canister time (`time().to_le_bytes()`) and 24 bytes of the constant `0x2a` (`42`). This low-entropy, time-predictable RNG is the sole source of randomness for `generate_random_uuid()`, which generates `RuleId` identifiers for every rate-limit rule stored in the canister. An attacker who can observe or approximate the canister's initialization time can enumerate the full PRNG output stream and predict all past and future `RuleId` values.

---

### Finding Description

In `rs/boundary_node/rate_limits/canister/random.rs`, the canister registers a custom `getrandom` handler:

```rust
thread_local! {
  static RNG: RefCell<ChaCha20Rng> = {
    let mut seed = [42; 32];
    seed[..8].copy_from_slice(&time().to_le_bytes());
    RefCell::new(ChaCha20Rng::from_seed(seed))
  };
}
``` [1](#0-0) 

The seed is constructed by overwriting only the first 8 bytes of a 32-byte buffer (pre-filled with `42`) with the IC canister time in nanoseconds. The remaining 24 bytes are the constant `0x2a`. Effective entropy is therefore at most 64 bits of IC consensus time — a value that is deterministic, replicated identically across all subnet nodes, and publicly observable from block metadata.

This RNG is registered as the global `getrandom` provider for the canister: [2](#0-1) 

`generate_random_uuid()` in `add_config.rs` calls `getrandom::getrandom()`, which routes through this time-seeded RNG to produce 16 bytes used to construct a `RuleId` UUID for every newly submitted rate-limit rule:

```rust
fn generate_random_uuid() -> Result<Uuid, anyhow::Error> {
    let mut buf = [0_u8; 16];
    getrandom::getrandom(&mut buf)
        ...
    let uuid = Uuid::from_slice(&buf)...;
    Ok(uuid)
}
``` [3](#0-2) 

This function is called for every new rule submitted via `add_config`: [4](#0-3) 

The `add_config` update method is callable by any authorized principal (boundary node operator), and the canister is a production system-subnet canister managing security-sensitive rate-limit rules for the IC API boundary layer: [5](#0-4) 

---

### Impact Explanation

An attacker who knows the approximate time at which the rate-limit canister was initialized (observable from on-chain block timestamps) can reconstruct the PRNG seed space (64-bit IC nanosecond time + 24 constant bytes) and enumerate all `RuleId` UUIDs the canister has generated or will generate. While `RuleId` values are publicly visible in query responses even for undisclosed rules, the predictability breaks the assumption of UUID unpredictability that the canister design relies on for rule identity integrity. Specifically:

- An attacker can pre-compute future `RuleId` values before rules are submitted, enabling targeted manipulation of rule-lookup logic or collision-based denial of service against the `get_rule` / `get_rules_by_incident_id` endpoints.
- The `RuleId` is the only canister-generated identifier for confidential rules; if it were ever used as an access-control token in a future extension, the weak RNG would directly enable unauthorized access.

The `RuleId` type is defined as a `Uuid` wrapper used as the stable key for all rule storage and retrieval: [6](#0-5) 

---

### Likelihood Explanation

The IC canister time (`ic_cdk::api::time()`) is the consensus time of the subnet, identical across all replicas and derivable from public block headers. The seed space is at most 2^64 values (nanosecond timestamps), but in practice the initialization window is narrow (canister deployment time is observable). An attacker with knowledge of the deployment block can reduce the search space to milliseconds of nanosecond timestamps, making full enumeration computationally trivial. No privileged access is required; the attacker only needs to observe public chain state.

---

### Recommendation

Replace the time-seeded custom `getrandom` implementation with one backed by `ic00`'s `raw_rand` management canister call, seeded once at initialization and periodically reseeded (as done in the NNS governance canister's `SeedingTask`). The 24 constant bytes of `42` in the seed provide zero additional entropy and must be replaced with cryptographically random material. Until `raw_rand` seeding is implemented, at minimum mix additional entropy sources (canister ID, subnet ID, block height) into the seed alongside the timestamp.

---

### Proof of Concept

1. Observe the IC block timestamp at the time the rate-limit canister was deployed (publicly available from the IC dashboard or replica logs).
2. Reconstruct the seed: `seed[0..8] = deployment_time_nanos.to_le_bytes(); seed[8..32] = [42u8; 24]`.
3. Instantiate `ChaCha20Rng::from_seed(seed)` and call `fill_bytes` repeatedly in 16-byte chunks.
4. Interpret each 16-byte chunk as a UUID — these are the `RuleId` values the canister will generate in order.
5. Confirm by querying `get_config` and comparing the returned `rule_id` fields against the predicted sequence.
