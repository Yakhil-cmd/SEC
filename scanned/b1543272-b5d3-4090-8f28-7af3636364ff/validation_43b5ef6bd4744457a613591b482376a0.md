### Title
SNS Treasury Manager Deposit Lacks Slippage Protection, Enabling Sandwich Attacks on DAO Treasury Funds - (File: rs/sns/governance/src/extensions.rs)

### Summary
The SNS Treasury Manager deposit flow (`execute_treasury_manager_deposit`) approves and transfers treasury funds to a DEX-backed extension canister without any minimum-output or slippage-protection parameter. Because SNS governance proposals are public and have a predictable voting window, an attacker can observe a pending deposit proposal, manipulate the DEX pool price just before execution, and cause the SNS treasury to receive significantly fewer LP tokens than expected. The codebase itself acknowledges this risk in the DID specification but provides no on-chain mitigation.

### Finding Description
When an SNS DAO passes an `ExecuteExtensionOperation` proposal with `operation_name = "deposit"`, the following sequence executes in `execute_treasury_manager_deposit`:

1. `validate_deposit_operation_impl` is called **at proposal-submission time** to check that the requested amounts do not exceed 50% of the current treasury balance.
2. After the governance voting period (days to weeks later), `execute_treasury_manager_deposit` is called at execution time.
3. `approve_treasury_manager` issues ICRC-2 allowances for the **exact amounts fixed in the proposal** to the extension canister.
4. `deposit` is called on the treasury manager canister, which forwards the funds to a DEX liquidity pool.

No minimum-output or slippage-tolerance parameter is passed at any step. The deposit amounts are fixed at proposal-submission time and are not re-validated against current market conditions at execution time. The `treasury_manager.did` specification explicitly acknowledges this:

> "Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."

The proposal-rendering code in `validate_and_render_register_extension` also warns about sandwich attacks but provides no enforcement mechanism.

### Impact Explanation
**Impact: High.** An attacker who monitors public SNS governance proposals can:
1. Identify a pending deposit proposal specifying exact SNS-token and ICP amounts.
2. Know the approximate execution time (end of voting period).
3. Submit a large swap on the target DEX pool just before the deposit executes, skewing the pool price.
4. Allow the deposit to execute at the manipulated price — the SNS treasury receives far fewer LP tokens than the fair-market equivalent.
5. Reverse the manipulation and extract the price difference as profit.

The loss is borne entirely by the SNS DAO treasury. Because the amounts are fixed in the proposal and no minimum-output check exists, there is no on-chain guard that can abort the deposit if the price has moved adversarially.

### Likelihood Explanation
**Likelihood: Medium.** SNS governance proposals are fully public and have deterministic voting windows. Any observer can read the exact deposit amounts and predict the execution block. The attacker needs only sufficient capital to move the DEX pool price, which is a well-known and commonly executed attack. The risk increases as treasury deposits grow larger (e.g., after a period during which deposits were blocked by a temporarily unavailable DEX, causing reward accumulation).

### Recommendation
1. **Add a `min_lp_tokens_out` (or equivalent slippage tolerance) parameter** to the `ExecuteExtensionOperation` deposit proposal. This value should be validated at execution time against the actual LP tokens received.
2. **Re-validate treasury amounts at execution time**, not only at proposal-submission time, to account for price drift during the voting period.
3. **Restrict deposit execution** to a short time window after proposal adoption, reducing the attacker's ability to time their manipulation.
4. **Require DEX-backed Treasury Manager implementations** to enforce a minimum-output check inside their `deposit` method before accepting the ICRC-2 allowance.

### Proof of Concept

**Step 1 – Attacker observes the proposal.**
An SNS governance proposal is submitted with:
```
ExecuteExtensionOperation {
  operation_name: "deposit",
  operation_arg: { treasury_allocation_sns_e8s: 1_000_000_000, treasury_allocation_icp_e8s: 500_000_000 }
}
```
This is publicly visible on-chain.

**Step 2 – Validation at submission time (no execution-time guard).**
`validate_deposit_operation_impl` checks only that `1_000_000_000 ≤ sns_balance / 2` and `500_000_000 ≤ icp_balance / 2`. No price or slippage check is performed. [1](#0-0) 

**Step 3 – Attacker front-runs execution.**
Just before the proposal executes, the attacker swaps a large amount into the DEX pool, moving the SNS/ICP price adversarially.

**Step 4 – Deposit executes without slippage protection.**
`execute_treasury_manager_deposit` approves the fixed amounts and calls `deposit`: [2](#0-1) 

`approve_treasury_manager` issues ICRC-2 allowances for the exact proposal amounts with no minimum-output parameter: [3](#0-2) 

**Step 5 – Treasury receives fewer LP tokens.**
The DEX executes the deposit at the manipulated price. The SNS treasury receives far fewer LP tokens than the fair-market value of the deposited assets. The attacker reverses their swap and profits.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

    async fn approve_treasury_manager(
        &self,
        treasury_manager_canister_id: CanisterId,
        sns_amount_e8s: u64,
        icp_amount_e8s: u64,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: treasury_manager_canister_id.get().0,
            subaccount: None,
        };

        let expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS);
        let expiry_time_nsec = expiry_time_sec.saturating_mul(NANO_SECONDS_PER_SECOND);

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

        self.ledger
            .icrc2_approve(
                to,
                sns_amount_e8s,
                Some(expiry_time_nsec),
                self.transaction_fee_e8s_or_panic(),
                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making SNS Token treasury transfer: {e}"),
                )
            })?;

        self.nns_ledger
            .icrc2_approve(
                to,
                icp_amount_e8s,
                Some(expiry_time_nsec),
                icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(),
                self.icp_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making ICP Token treasury transfer: {e}"),
                )
            })?;

        Ok(())
    }
