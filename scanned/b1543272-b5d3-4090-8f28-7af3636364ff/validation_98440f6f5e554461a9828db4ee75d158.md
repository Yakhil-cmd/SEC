### Title
No Way to Remove a Managed ERC-20 Token from the Ledger Suite Orchestrator Once Registered - (File: rs/ethereum/ledger-suite-orchestrator/src/state/mod.rs)

---

### Summary

The Ledger Suite Orchestrator permanently registers ERC-20 token ledger suites via `insert_once` with no corresponding removal function. Once a ckERC20 token is added, neither the NNS nor any other actor can remove it from the orchestrator's managed set without a full orchestrator canister upgrade that manually patches state. If a managed ledger suite is later found to be compromised or irreparably broken, the orchestrator will continue to top up the compromised canisters with cycles indefinitely.

---

### Finding Description

`record_new_erc20_token` and `record_manage_other_canisters` both call `insert_once` on the `managed_canisters` map, which panics if the key already exists and has no inverse operation: [1](#0-0) 

The canister's only state-mutation entry points are `post_upgrade` (handling `UpgradeArg` and `AddErc20Arg`) and the `add_erc20` lifecycle function: [2](#0-1) 

Neither `UpgradeArg` nor any other argument type exposes a `remove_erc20` or `unmanage_ledger_suite` field. The Candid interface confirms this — `OrchestratorArg` has `AddErc20Arg` and `UpgradeArg` but no removal variant: [3](#0-2) 

The orchestrator's timer-driven scheduler continuously tops up all managed canisters and schedules upgrade tasks for every token in `all_managed_tokens_ids_iter()`: [4](#0-3) 

There is no `remove_wasm`, `remove_erc20_token`, or `unmanage_canisters` function anywhere in the orchestrator source.

---

### Impact Explanation

If a ckERC20 ledger canister (e.g., ckUSDC, ckUSDT) is found to contain a critical vulnerability — such as an unauthorized minting bug or a state-corruption flaw — the orchestrator cannot be instructed to stop managing it. Concretely:

1. The orchestrator's `MaybeTopUp` task will continue sending cycles to the compromised canister on every timer tick, keeping it alive and funded.
2. The `UpgradeLedgerSuite` task will continue attempting to upgrade it, potentially re-installing a still-vulnerable WASM if the upgrade path has not been separately patched in SNS-WASM.
3. Users holding the affected ckERC20 token face continued exposure because the orchestrator actively sustains the compromised canister's operation.

The only recovery path is an emergency NNS upgrade proposal that replaces the orchestrator binary with a version that manually removes the token from state — a slow, high-friction process that extends the window of exposure.

---

### Likelihood Explanation

The ckERC20 ledger canisters are complex ICRC-1 implementations. A post-deployment vulnerability discovery (e.g., via audit or exploit) is a realistic scenario. The NNS has already demonstrated the need for emergency response mechanisms (e.g., `NNS_FUNCTION_UNINSTALL_CODE`, `StopOrStartCanister` proposals). The absence of a `remove_erc20_token` path in the orchestrator means the standard NNS emergency toolkit is insufficient to cleanly isolate a compromised ckERC20 ledger suite from the orchestrator's active management loop.

---

### Recommendation

Add a privileged `remove_erc20_token` (or `unmanage_ledger_suite`) variant to `OrchestratorArg` that:

1. Removes the `TokenId` entry from `managed_canisters`.
2. Cancels any pending `UpgradeLedgerSuite` or `MaybeTopUp` tasks for that token.
3. Is gated to NNS-only invocation (identical to the existing `AddErc20Arg` path).

This mirrors the recommendation in the original report: add a privileged removal function so that a compromised or deprecated resource can be cleanly deregistered and, if needed, re-initialized.

---

### Proof of Concept

1. NNS submits an upgrade proposal with `OrchestratorArg::AddErc20Arg` for a new ERC-20 token. The orchestrator calls `record_new_erc20_token`, which calls `insert_once`: [5](#0-4) 

2. A critical vulnerability is discovered in the spawned ledger canister.

3. The NNS attempts to respond. Searching the entire `OrchestratorArg` enum and `UpgradeArg` struct, there is no field to remove or deregister a token: [6](#0-5) 

4. On every timer tick, `Task::MaybeTopUp` iterates `all_managed_tokens_ids_iter()` — which includes the compromised token — and sends cycles to its ledger and index canisters, sustaining the attacker's position.

5. The only remediation is an emergency NNS upgrade of the orchestrator binary itself with a patched state, which requires a full governance cycle and leaves the compromised canister funded in the interim.

        read_wasm_store(|w| InstallLedgerSuiteArgs::validate_add_erc20(s, w, token.clone()))
    }) {

    read_state(|s| s.validate_config().expect("ERROR: invalid state"));

type OrchestratorArg = variant {
    UpgradeArg : UpgradeArg;
    InitArg : InitArg;
    AddErc20Arg : AddErc20Arg;
};

type InitArg = record {
    // All canisters that will be spawned off by the orchestrator will be controlled by the orchestrator
    // and *additionally* by the following controllers.
    more_controller_ids : vec principal;

    // Canister ID of the minter that will be notified when new ERC-20 tokens are added.
    minter_id: opt principal;

    // Controls the cycles management of the canisters managed by the orchestrator.
    cycles_management: opt CyclesManagement;
};

type UpgradeArg = record {
    // Hexadecimal encoding of the SHA-1 git commit hash used for this upgrade, e.g.,
    // "51d01d3936498d4010de54505d6433e9ad5cc62b", corresponding to a git revision in the
    // [IC repository](https://github.com/dfinity/ic).
    // If this field is present, the orchestrator will register all embedded wasms (ledger, index, and archive) in its stable memory,
    // so that those exact wasms can be used to upgrade managed canisters as specified below.
    git_commit_hash: opt text;

   // Hexadecimal encoding of the SHA2-256 ledger compressed wasm hash, e.g.,
   // "3148f7a9f1b0ee39262c8abe3b08813480cf78551eee5a60ab1cf38433b5d9b0".
   // This exact version will be used for upgrading all existing ledger canisters managed by the orchestrator.
   // Leaving this field empty will not upgrade the ledger canisters.
   ledger_compressed_wasm_hash: opt text;

   // Hexadecimal encoding of the SHA2-256 index compressed wasm hash, e.g.,
   // "3a6d39b5e94cdef5203bca62720e75a28cd071ff434d22b9746403ac7ae59614".
   // This exact version will be used for upgrading all existing index canisters managed by the orchestrator.
   // Leaving this field empty will not upgrade the index canisters.
   index_compressed_wasm_hash: opt text;

   // Hexadecimal encoding of the SHA2-256 archive compressed wasm hash, e.g.,
   // "b24812976b2cc64f12faf813cf592631f3062bfda837334f77ab807361d64e82".
   // This exact version will be used for upgrading all existing archive canisters managed by the orchestrator.
   // Leaving this field empty will not upgrade the archive canisters.
   archive_compressed_wasm_hash: opt text;

   // Update the cycles management of the canisters managed by the orchestrator.
   cycles_management: opt UpdateCyclesManagement;

   // Add already installed ledger suites to the canisters managed by the orchestrator.
   // Those ledger suites are *NOT* necessarily ckERC20 tokens.
   // This assumes that the orchestrator is a controller of all the canisters in the list.
   manage_ledger_suites: opt vec InstalledLedgerSuite;
};

type AddErc20Arg = record {
   contract: Erc20Contract;
   ledger_init_arg: LedgerInitArg;
};
