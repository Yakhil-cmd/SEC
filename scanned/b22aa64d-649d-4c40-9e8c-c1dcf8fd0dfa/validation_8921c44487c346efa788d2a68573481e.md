### Title
`inspect_message` Guard Bypassed via Non-Replicated Query — Undisclosed `rule_id`/`incident_id` Enumeration - (`rs/boundary_node/rate_limits/canister/canister.rs`)

---

### Summary

`get_config` is annotated `#[query]`, making it a **non-replicated query** on the IC. The IC protocol only invokes `inspect_message` for **update (ingress) calls** — never for query calls. The `inspect_message` guard that was intended to restrict `get_config` to authorized principals is therefore completely inert. Any anonymous principal can call `get_config` as a query, receive the full rule list, and observe the `rule_id` and `incident_id` of every non-disclosed rule (with only `rule_raw` and `description` redacted).

---

### Finding Description

**Root cause — wrong method type:**

The constant name makes the developer intent explicit:

```rust
const REPLICATED_QUERY_METHOD: &str = "get_config";
``` [1](#0-0) 

A *replicated* query would be submitted as an ingress message, go through consensus, and trigger `inspect_message`. Instead, `get_config` is declared as a plain non-replicated query:

```rust
#[query]
fn get_config(version: Option<Version>) -> GetConfigResponse {
``` [2](#0-1) 

**The guard that is never reached:**

`inspect_message` explicitly checks `REPLICATED_QUERY_METHOD` and traps for unauthorized callers:

```rust
if called_method == REPLICATED_QUERY_METHOD {
    if has_full_access || has_full_read_access {
        ic_cdk::api::call::accept_message();
    } else {
        ic_cdk::api::trap("message_inspection_failed: ...");
    }
}
``` [3](#0-2) 

Because `get_config` is `#[query]`, this block is **never executed** for any query call. The IC runtime simply routes the query directly to execution.

**What the formatter leaks:**

`ConfigConfidentialityFormatter` only nulls out `description` and `rule_raw` for non-disclosed rules. It does **not** remove the rule entry, `rule_id`, or `incident_id`:

```rust
config.rules.iter_mut().for_each(|rule| {
    if rule.disclosed_at.is_none() {
        rule.description = None;
        rule.rule_raw = None;
    }
});
``` [4](#0-3) 

This is confirmed by the existing unit test, which asserts that a `RestrictedRead` caller receives entries with `rule_id` and `incident_id` populated for non-disclosed rules:

```rust
api::OutputRule {
    rule_id: rule_id_1.0.to_string(),
    incident_id: incident_id.0.to_string(),
    rule_raw: None,
    description: None,
},
``` [5](#0-4) 

---

### Impact Explanation

An anonymous principal calling `get_config` as a query receives the complete list of active `rule_id`s and their associated `incident_id`s — including those for rules that have never been disclosed. By correlating `incident_id`s across repeated queries (e.g., before and after a new security incident is added), an attacker can enumerate the existence, count, and grouping of undisclosed security incidents. The `rule_raw` (the actual rate-limit payload) and `description` remain hidden, so the attacker cannot learn the specific traffic patterns being blocked, but the metadata exposure violates the stated confidentiality invariant. [6](#0-5) 

---

### Likelihood Explanation

Exploitation requires no privileges, no keys, and no special tooling — a single `dfx canister call --query get_config` from an anonymous principal suffices. The bypass is structural (wrong IC method type), not conditional on any runtime state.

---

### Recommendation

Change `get_config` from `#[query]` to `#[update]` so that it is submitted as an ingress message and `inspect_message` is actually invoked. This matches the developer's own naming (`REPLICATED_QUERY_METHOD`) and restores the intended access control. Alternatively, if a non-replicated query is required for performance, move the authorization check **inside** `get_config` itself (before calling the getter) and return an error for unauthorized callers, rather than relying on `inspect_message`. [7](#0-6)

### Proof of Concept

```bash
# As anonymous principal, call get_config as a non-replicated query
dfx canister call --query <rate_limit_canister_id> get_config '(null)'
```

The response will contain `OutputRule` entries where `rule_raw` and `description` are `null` but `rule_id` and `incident_id` are populated — confirming that undisclosed rule metadata is visible to any unauthenticated caller. An integration test can assert:

```rust
// anonymous query call
let response = query_call(Principal::anonymous(), "get_config", None).await;
let rules = response.config.rules;
// non-disclosed rule: IDs are present, content is redacted
assert!(rules.iter().any(|r| r.rule_raw.is_none() && !r.rule_id.is_empty()));
assert!(rules.iter().any(|r| r.incident_id != ""));
``` [8](#0-7)
