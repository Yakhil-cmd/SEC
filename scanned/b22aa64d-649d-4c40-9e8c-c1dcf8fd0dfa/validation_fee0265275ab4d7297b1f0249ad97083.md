### Title
Stale Cached ICP/XDR Conversion Rate Used for Cycles Minting — (`rs/nns/cmc/src/main.rs`)

### Summary
The Cycles Minting Canister (CMC) caches the ICP/XDR conversion rate in `state.icp_xdr_conversion_rate` and updates it only via a periodic heartbeat task (every 5 minutes). All user-triggered cycles-minting operations — `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` — read this cached rate at execution time without refreshing it. If the rate has not been updated recently (e.g., the XRC is temporarily unavailable, or the heartbeat has not fired), users receive fewer cycles than the current market rate entitles them to, or more cycles than they should receive.

### Finding Description

The `tokens_to_cycles` function in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` directly from cached state:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        ...
    })
}
```

This function is called synchronously inside `process_top_up`, `process_create_canister`, and `process_mint_cycles` — all of which are invoked from the three public update endpoints. The cached rate is only refreshed by the periodic `update_exchange_rate` task, which runs at most once every 5 minutes (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`). If the XRC call fails, the next retry is scheduled 1 minute later, but the stale rate remains in use for all minting operations in the interim.

The state field `icp_xdr_conversion_rate` is set only in `do_set_icp_xdr_conversion_rate`, which is called either by the heartbeat-driven `update_exchange_rate` or by the governance-gated `set_icp_xdr_conversion_rate`. No on-demand refresh occurs at the time of a user's minting call. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

**Impact: Medium**

Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during a period when the cached rate is stale will receive a cycles amount computed from an outdated ICP/XDR rate. If the ICP price has risen since the last cache update, users receive fewer cycles than the current market rate entitles them to (economic loss to users). If the ICP price has fallen, users receive more cycles than they should (economic loss to the protocol/treasury). The magnitude is bounded by the price movement within the staleness window (up to 5 minutes under normal operation, potentially longer during XRC outages). The ICP is burned at the stale rate, so the discrepancy is permanent and unrecoverable for the affected block. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

**Likelihood: Low-Medium**

Under normal operation the rate is refreshed every 5 minutes, so the window of staleness is small. However, the XRC is an external dependency; if it returns errors, the CMC retries every minute but continues using the stale rate for all minting operations during the outage. The `UpdateExchangeRateError::FailedToRetrieveRate` path explicitly schedules a retry at the next minute without updating the rate, meaning the stale rate persists. Any user who mints cycles during an XRC outage of any duration is affected. Given that `notify_top_up` and `notify_mint_cycles` are high-frequency public endpoints, the probability of a user hitting a stale rate during even a brief XRC disruption is non-trivial. [7](#0-6) [8](#0-7) 

### Recommendation

1. **Expose the rate timestamp to callers**: Include `icp_xdr_conversion_rate.timestamp_seconds` in the `NotifyMintCyclesSuccess` / `NotifyTopUp` response so users can verify the rate age used for their conversion.
2. **Enforce a maximum rate age at minting time**: In `tokens_to_cycles`, check that `now_seconds - icp_xdr_conversion_rate.timestamp_seconds` does not exceed a threshold (e.g., 30 minutes). If the rate is too stale, return a retriable `NotifyError` rather than minting at an outdated rate.
3. **Increase XRC retry frequency during outages**: The current 1-minute retry on failure is reasonable, but the CMC could also attempt an on-demand XRC fetch when a minting call arrives and the cached rate is older than a configurable threshold. [9](#0-8) [3](#0-2) 

### Proof of Concept

1. The CMC is initialized with a default rate from May 2021 (`DEFAULT_ICP_XDR_CONVERSION_RATE_TIMESTAMP_SECONDS = 1_620_633_600`).
2. The XRC canister is configured to return errors (e.g., `ExchangeRateError::StablecoinRateTooFewRates`).
3. The CMC heartbeat fires, calls `update_exchange_rate`, receives an error, and schedules a retry in 1 minute — but does **not** update `icp_xdr_conversion_rate`. The stale rate remains.
4. A user calls `notify_top_up` with 1 ICP. `tokens_to_cycles` reads the stale `xdr_permyriad_per_icp` and mints cycles at the old rate.
5. If the real market rate has moved 5% since the last successful update, the user receives ~5% fewer (or more) cycles than the current market rate entitles them to.

This scenario is directly demonstrated by the existing integration test at `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs` lines 162–185, which shows that after an XRC error the CMC retains the previous rate for subsequent minting operations. [10](#0-9) [11](#0-10)

/// The periodic task for collecting the ICP/XDR rate from the Exchange Rate Canister.
/// To avoid having multiple calls sent to the Exchange Rate Canister,
/// this function contains a guard to ensure multiple calls cannot be made until
/// the prior call is complete.
pub async fn update_exchange_rate(
    safe_state: &'static LocalKey<RefCell<Option<State>>>,
    env: &impl Environment,
    xrc_client: &impl ExchangeRateCanisterClient,
) -> Result<(), UpdateExchangeRateError> {
    let now_timestamp_seconds = env.now_timestamp_seconds();
    let current_minute_seconds =
        round_down_to_multiple_of(now_timestamp_seconds, ONE_MINUTE_SECONDS);

    UpdateExchangeRateGuard::with_guard(safe_state, current_minute_seconds, async {
        let call_xrc_result = xrc_client.get_icp_to_xdr_exchange_rate(None).await;
        // Check if updating the rate via the exchange rate canister was disabled while retrieving the rate.
        // If it has, exit early.
        let is_updating_rate_disabled = read_state(safe_state, |state| {
            state
                .update_exchange_rate_canister_state
                .unwrap_or_default()
                == UpdateExchangeRateState::Disabled
        });
        if is_updating_rate_disabled {
            return Err(UpdateExchangeRateError::Disabled);
        }

        match call_xrc_result {
            Ok(exchange_rate) => {
                validate_exchange_rate(&exchange_rate)
                    .map_err(|error| UpdateExchangeRateError::InvalidRate(error.to_string()))?;
                let icp_xdr_conversion_rate = IcpXdrConversionRate::from(exchange_rate);
                if let Err(error) =
                    do_set_icp_xdr_conversion_rate(safe_state, env, icp_xdr_conversion_rate)
                {
                    return Err(UpdateExchangeRateError::FailedToSetRate(error));
                }
            }
            Err(error) => {
                return Err(UpdateExchangeRateError::FailedToRetrieveRate(
                    error.to_string(),
                ));
            }
        };

        Ok(())
    })
    .await
}
