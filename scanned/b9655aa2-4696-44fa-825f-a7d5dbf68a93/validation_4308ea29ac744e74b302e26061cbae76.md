### Title
ckBTC Minter Withdrawal Guard Bypass via Inconsistent Guard Key Between `retrieve_btc` and `retrieve_btc_with_approval` - (File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs)

---

### Summary

The ckBTC minter uses a per-account guard (`retrieve_btc_accounts`) to prevent a single principal from having multiple concurrent withdrawal requests in flight. However, the two withdrawal endpoints use **different guard keys** for the same principal, allowing an attacker to bypass the concurrency protection entirely by using different subaccounts across the two endpoints.

---

### Finding Description

The ckBTC minter exposes two withdrawal endpoints: `retrieve_btc` and `retrieve_btc_with_approval`. Both are protected by a `Guard<RetrieveBtcUpdates>` that inserts an `Account` into `state.retrieve_btc_accounts` and rejects a second request if the same `Account` is already present.

**`retrieve_btc` always hardcodes `subaccount: None` as the guard key:**

```rust
let _guard = retrieve_btc_guard(Account {
    owner: caller,
    subaccount: None,   // ← always None, regardless of actual operation
})?;
``` [1](#0-0) 

**`retrieve_btc_with_approval` uses the caller-supplied `from_subaccount` as the guard key:**

```rust
let caller_account = Account {
    owner: caller,
    subaccount: args.from_subaccount,  // ← attacker-controlled
};
let _guard = retrieve_btc_guard(caller_account)?;
``` [2](#0-1) 

The guard is backed by a `BTreeSet<Account>` keyed on the full `(owner, subaccount)` pair: [3](#0-2) 

The guard state is stored per-account in `retrieve_btc_accounts`: [4](#0-3) 

Because IC canister execution is interleaved at `await` points, a caller can:

1. Call `retrieve_btc` → guard acquired for `{owner: caller, subaccount: None}`.
2. While `retrieve_btc` is suspended at any of its `await` points (`init_ecdsa_public_key`, `balance_of`, `check_address`, `burn_ckbtcs`), the canister processes the next ingress message.
3. Call `retrieve_btc_with_approval` with `from_subaccount: Some([1u8; 32])` → guard acquired for `{owner: caller, subaccount: Some([1u8; 32])}`.
4. These are **distinct keys** in the `BTreeSet`, so the second guard succeeds — both withdrawals proceed concurrently.

Furthermore, a caller can call `retrieve_btc_with_approval` multiple times with distinct non-zero subaccounts (e.g., `Some([1; 32])`, `Some([2; 32])`, …), each acquiring a separate guard slot, bypassing the "one concurrent request per account" invariant entirely.

The guard's own test acknowledges that `None` and `Some([0u8; 32])` are treated as the same account (ICRC-1 canonical equivalence), but any non-zero subaccount is a distinct key: [5](#0-4) 

---

### Impact Explanation

An unprivileged ingress caller can:

- **Bypass the per-principal concurrency limit** for ckBTC withdrawals, submitting many concurrent `retrieve_btc_with_approval` requests (one per distinct subaccount) while the guard only blocks exact-key duplicates.
- **Exhaust `MAX_CONCURRENT_PENDING_REQUESTS`** (5,000) faster than intended, degrading service for other users.
- **Combine `retrieve_btc` and `retrieve_btc_with_approval` concurrently** for the same principal, violating the invariant that a principal has at most one in-flight withdrawal.

The ledger enforces per-account balance checks independently, so direct double-spending of the same tokens is prevented. However, the minter's own concurrency invariant is broken, and the `retrieve_btc_account_to_block_indices` tracking map can accumulate more entries per account than intended. [6](#0-5) 

---

### Likelihood Explanation

- Requires only an unprivileged ingress caller with ckBTC in multiple subaccounts (or ICRC-2 approvals from multiple subaccounts).
- The exploit path is straightforward: send two ingress messages in the same IC round targeting the two different endpoints with different subaccounts.
- No privileged access, no key material, no governance majority required.
- The IC's interleaved async execution model makes the race window reliable and repeatable.

---

### Recommendation

Normalize the guard key to the **principal only** (ignoring subaccount) for both endpoints, so that any concurrent withdrawal attempt by the same principal is blocked regardless of which subaccount is used:

```rust
// retrieve_btc
let _guard = retrieve_btc_guard(Account {
    owner: caller,
    subaccount: None,
})?;

// retrieve_btc_with_approval — change to:
let _guard = retrieve_btc_guard(Account {
    owner: caller,
    subaccount: None,  // normalize to None for guard purposes
})?;
```

Alternatively, if per-subaccount isolation is desired, ensure both endpoints use the same normalization strategy and document the intended invariant explicitly.

---

### Proof of Concept

1. Attacker holds ckBTC in two subaccounts: `sub_A = [1u8; 32]` and `sub_B = [2u8; 32]`, and has approved the minter to spend from both via `icrc2_approve`.
2. Attacker sends ingress message M1: `retrieve_btc_with_approval { amount: X, from_subaccount: Some(sub_A), address: "btc_addr_1" }`.
3. M1 is accepted; guard for `{caller, Some(sub_A)}` is inserted into `retrieve_btc_accounts`. M1 suspends at `init_ecdsa_public_key().await`.
4. Attacker sends ingress message M2: `retrieve_btc_with_approval { amount: Y, from_subaccount: Some(sub_B), address: "btc_addr_2" }`.
5. M2 is processed while M1 is suspended. Guard check: `retrieve_btc_accounts.contains({caller, Some(sub_B)})` → **false** (different key). Guard for `{caller, Some(sub_B)}` is inserted. M2 proceeds.
6. Both M1 and M2 complete successfully, burning from `sub_A` and `sub_B` respectively — two concurrent withdrawal requests from the same principal, bypassing the intended one-at-a-time protection. [7](#0-6) [8](#0-7)

impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }

pub fn accept_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    runtime: &R,
) {
    record_event(
        EventType::AcceptedRetrieveBtcRequest(request.clone()),
        runtime,
    );
    state.pending_retrieve_btc_requests.push(request.clone());
    if let Some(account) = request.reimbursement_account {
        state
            .retrieve_btc_account_to_block_indices
            .entry(account)
            .and_modify(|entry| entry.push(request.block_index))
            .or_insert(vec![request.block_index]);
    }
    if let Some(kyt_provider) = request.kyt_provider {
        *state.owed_kyt_amount.entry(kyt_provider).or_insert(0) += state.check_fee;
    }
}
