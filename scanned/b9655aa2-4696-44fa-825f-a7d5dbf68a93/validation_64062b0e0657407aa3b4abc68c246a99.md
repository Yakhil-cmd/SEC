### Title
Pre-Authorization Inter-Canister Call in `make_proposal` â€” (`rs/nns/governance/src/governance.rs`)

### Summary

In `Governance::make_proposal`, the call to `action.to_self_describing(self.env.clone()).await` â€” which for most `ExecuteNnsFunction` variants issues an outbound inter-canister call to the management canister (`ic_00`) to fetch `candid:service` metadata â€” executes **before** the `is_proposer_authorized_to_vote` check. Any principal who knows a valid neuron ID (publicly readable from governance state) can trigger this outbound call without owning or controlling that neuron.

### Finding Description

The execution order in `make_proposal` is:

1. `validate_proposal` (synchronous, no inter-canister call) â€” line 5141
2. `action.to_self_describing(self.env.clone()).await` â€” lines 5157â€“5171 â€” **outbound inter-canister call for `ExecuteNnsFunction` proposals**
3. `neuron.is_authorized_to_vote(caller)` check â€” lines 5178â€“5197 [1](#0-0) 

For `ExecuteNnsFunction` proposals where the NNS function's target is not `ic_00` itself (i.e., all variants except `UninstallCode`), `to_self_describing` calls `get_candid_source`, which issues `env.call_canister_method(CanisterId::ic_00(), "canister_metadata", ...)` to the management canister: [2](#0-1) 

This call is guarded by `cfg!(target_arch = "wasm32")`, so it only fires in the deployed canister, not in native tests: [3](#0-2) 

The authorization check that should gate all of this comes later: [4](#0-3) 

### Impact Explanation

An attacker can:
1. Read any valid `NeuronId` from governance public state.
2. Submit `make_proposal` with that neuron ID and a structurally valid `ExecuteNnsFunction` action (e.g., `UpdateSubnetType` with any payload).
3. The governance canister issues a `canister_metadata` call to `ic_00` before checking authorization.
4. The call failure (or success) is non-fatal â€” the error is caught and logged, and execution continues to the authorization check, which then returns `NotAuthorized`. [5](#0-4) 

The target of the outbound call is **not** attacker-controlled â€” it is always `ic_00` asking for the `candid:service` metadata of a fixed NNS canister determined by the `ValidNnsFunction` enum: [6](#0-5) 

The practical impact is **cycles drain** on the governance canister: each unauthorized submission triggers one `canister_metadata` call to `ic_00`. The cost per call is small, the governance canister has a large cycles balance, and IC ingress rate limits apply. The call cannot be directed to an arbitrary canister and causes no state mutation. This makes the impact **low severity** â€” a real invariant violation (unauthenticated callers triggering outbound calls) with limited practical consequence.

### Likelihood Explanation

The attack path is trivially reachable: neuron IDs are public, the proposal format is documented, and no special privilege is required. However, the low per-call cost and IC-level rate limiting make meaningful cycles exhaustion impractical against a well-funded system canister.

### Recommendation

Move the `to_self_describing` call to **after** the authorization check (after line 5197). Since `to_self_describing` is purely for enriching proposal metadata and its failure is already non-fatal, this reordering has no functional impact on the proposal submission flow and eliminates the pre-auth outbound call entirely.

### Proof of Concept

State-machine test (wasm32 target):
1. Create a governance instance with one neuron controlled by principal A.
2. Submit `make_proposal` from principal B (not controller/hot-key of the neuron) with a valid `ExecuteNnsFunction { nns_function: UpdateSubnetType, payload: ... }`.
3. Assert that a `canister_metadata` call to `ic_00` was recorded **before** the `NotAuthorized` error is returned.
4. Assert the returned error is `ErrorType::NotAuthorized`.

The `MockEnvironment` pattern already used in `execute_nns_function_tests.rs` can record and assert on `call_canister_method` invocations to confirm the call fires pre-authorization. [7](#0-6)
