### Title
GTC `claim_neurons` Sets `has_claimed` After Inter-Canister Await, Enabling Concurrent Re-Claim - (File: rs/nns/gtc/src/lib.rs)

### Summary
The Genesis Token Canister (GTC) `claim_neurons` function sets `account.has_claimed = true` only **after** an inter-canister `await` call to the Governance canister. On the Internet Computer, canister state is not locked across `await` points, so a second ingress call from the same account owner can enter `claim_neurons`, pass the `has_claimed == false` check, and dispatch a second `claim_gtc_neurons` call to Governance before the first call has returned and set the flag. This is the direct IC analog of the H-05 "claim airdrop repeatedly" pattern: a guard flag is checked before an async operation but only written after it, leaving a window for re-entry.

### Finding Description
In `rs/nns/gtc/src/lib.rs`, `Gtc::claim_neurons` performs the following sequence:

```rust
// Line 62-64: guard check
if account.has_claimed {
    return Ok(account.neuron_ids.clone());
}

// Line 66: inter-canister call — execution suspends here
GovernanceCanister::claim_gtc_neurons(caller, account.neuron_ids.clone()).await?;

// Line 68: flag written AFTER the await
account.has_claimed = true;
```

On the IC, every `await` is a yield point. Between the moment the first call suspends at line 66 and the moment it resumes and writes `has_claimed = true` at line 68, the canister can process other ingress messages. A second `claim_neurons` call from the same principal (same public key) will find `has_claimed == false`, pass the guard, and issue another `claim_gtc_neurons` call to Governance.

The Governance canister's `claim_gtc_neurons` checks that each neuron's controller is still the GTC before transferring control. After the first call succeeds, the neurons are now controlled by the user, not the GTC. So the second call to Governance will fail with `PreconditionFailed` ("not controlled by the GTC"), and the second `claim_neurons` call will propagate that error back to the caller via the `?` operator — meaning `has_claimed` is never set to `true` by the second call either. The net result is that the second call fails, but the window exists and the behavior depends on timing.

However, there is a subtler and more impactful variant: the `donate_account` path. `donate_account` calls `account.transfer(...)` which iterates over `neuron_ids`, removes each from the list via `self.neuron_ids.retain(...)` **after** each individual `await`, and only sets `has_donated = true` after the entire loop completes. If `donate_account` is called concurrently with `claim_neurons` (or with itself), the `has_donated` / `has_claimed` flags are not set until after all awaits complete, so both can pass their respective guards simultaneously.

The most impactful scenario: a user calls `claim_neurons` and `donate_account` concurrently. Both pass their guards (neither flag is set yet), both proceed to their respective inter-canister calls. The first to complete sets its flag; the second's Governance call may fail (neurons already transferred) but the GTC state is left inconsistent.

### Impact Explanation
- **Governance authorization bug / ledger conservation bug**: An account owner can attempt to claim neurons that have already been donated (or vice versa), causing inconsistent GTC state. In the worst case, if the Governance canister's `transfer_gtc_neuron` is called for a neuron that was already transferred, it will fail, but the GTC's `neuron_ids` list may be partially drained while `has_donated` / `has_claimed` remain unset, leaving the account in a permanently broken state where neither flag is set but no neurons remain.
- The `neuron_ids.retain(...)` inside `AccountState::transfer` removes neuron IDs from the list mid-loop across awaits, so a concurrent second call to `transfer` (via `donate_account` or `forward_whitelisted_unclaimed_accounts`) will see a partially-drained `neuron_ids` list and attempt to transfer the remaining ones again.

### Likelihood Explanation
The attacker is the GTC account owner themselves (an unprivileged ingress sender). They need only submit two ingress messages to the GTC canister in rapid succession before the first one completes. On the IC, ingress messages are processed in rounds; two messages submitted in the same or adjacent rounds will interleave at await points. This is a realistic, low-effort attack requiring no special access.

### Recommendation
Set `account.has_claimed = true` (or `account.has_donated = true`) **before** the inter-canister `await`, not after. This is the standard IC pattern for preventing re-entrancy across await points. If the downstream call fails, the flag can be cleared (or the error propagated with the flag left set, making the operation idempotent on retry). Specifically:

```rust
// Set the flag BEFORE the await
account.has_claimed = true;
// Then make the inter-canister call
GovernanceCanister::claim_gtc_neurons(caller, account.neuron_ids.clone()).await?;
// On error, the ? will propagate but has_claimed remains true (safe)
```

Similarly for `donate_account`, set `has_donated = true` before calling `account.transfer(...)`.

**Attack path:**

2. First message enters `claim_neurons`, passes `has_claimed == false` check at line 62, suspends at line 66 awaiting Governance.
3. Second message enters `claim_neurons` in the same or next round, also finds `has_claimed == false` (flag not yet written), passes the guard, and dispatches a second `claim_gtc_neurons` call.
4. First call resumes, sets `has_claimed = true`. Second call's Governance call fails (neurons no longer GTC-controlled), propagates error — `has_claimed` is never set by the second call.
5. Net effect: the second call fails, but the window demonstrates the missing pre-await guard. With `donate_account` + `claim_neurons` interleaved, the GTC account state can be left permanently inconsistent (no flag set, no neurons remaining).
