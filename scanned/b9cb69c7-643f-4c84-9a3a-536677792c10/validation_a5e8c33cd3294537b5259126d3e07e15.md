### Title
Byzantine Peer Can Permanently Block `process_slot_update` Task After Assembly, Leaking Unvalidated Pool Entry and Ignoring Cancellation — (`rs/p2p/consensus_manager/src/receiver.rs`)

---

### Summary

After `AssembleResult::Done` is returned by the assembler, `process_slot_update` exits the outer `select!` and enters a sequential code path that calls `peer_rx.wait_for(|p| p.is_empty()).await` with no cancellation guard. A Byzantine subnet peer that advertises an artifact but never sends a slot retraction keeps the `PeerCounter` permanently non-empty, causing the task to block indefinitely at that await point. The assembled artifact is inserted into the unvalidated pool but never removed, and the task does not terminate even after `cancellation_token.cancel()` is called. The developers have already flagged both call sites with `// TODO: NET-1774`.

---

### Finding Description

In `process_slot_update` the outer `select!` races three futures:

```
select! {
    _ = cancellation_token.cancelled() => {}          // branch A
    assemble_result = assemble_artifact => { ... }    // branch B
    _ = all_peers_deleted_artifact => { ... }         // branch C
}
``` [1](#0-0) 

Once branch B wins (assembly completes), the `select!` is resolved and the cancellation token is no longer polled. Inside the `AssembleResult::Done` arm the code:

1. Sends `Insert` to the unvalidated pool (line 489).
2. Calls `peer_rx.wait_for(|p| p.is_empty()).await` (line 500) — **outside any `select!`**.
3. Only if that returns does it send `Remove` (line 505). [2](#0-1) 

The same pattern exists for `AssembleResult::Unwanted` at line 521. [3](#0-2) 

`peer_rx.wait_for` unblocks only when the `PeerCounter` becomes empty **or** when the `watch::Sender` is dropped. The sender lives in `active_assembles` and is only removed in `handle_artifact_processor_joined` when the task has already joined: [4](#0-3) 

Since the task is blocked, it never joins, so the sender is never dropped, so `wait_for` never returns via the error path either. The only other way the counter reaches zero is via `handle_topology_update`, which removes peers that leave the subnet topology: [5](#0-4) 

A Byzantine peer that remains a valid subnet member and simply withholds the slot retraction message is never removed by the topology update path.

---

### Impact Explanation

- **Unvalidated pool pollution**: `UnvalidatedArtifactMutation::Remove` (line 505) is never sent, so the assembled artifact remains in the unvalidated pool for the lifetime of the replica process. A Byzantine peer can repeat this for every slot it occupies, accumulating stale entries.
- **Task leak**: The `process_slot_update` Tokio task is permanently suspended. Because it never returns, `artifact_processor_tasks.join_next()` never yields it, so `handle_artifact_processor_joined` is never called, and `active_assembles` retains the entry indefinitely.
- **Cancellation invariant broken**: Calling `cancellation_token.cancel()` does not terminate the task. The `JoinSet` holding the task is dropped when the event loop exits, which causes Tokio to abort the task — but the `Remove` message is still never sent, leaving the pool entry permanently.

---

### Likelihood Explanation

The attacker is a Byzantine subnet peer operating below the consensus fault threshold — a standard adversary in the IC threat model. The required action is trivially simple: send one slot advertisement (with or without the artifact body) and then stop responding. No cryptographic material, no privileged access, and no coordination with other nodes is required. The slot limit per peer (`slot_limit`) bounds how many slots a single peer can occupy, but each occupied slot that completes assembly produces one permanent pool entry and one leaked task.

---

### Recommendation

Replace the bare `peer_rx.wait_for(...)` calls at lines 500 and 521 with a `select!` that also arms the cancellation token, for example:

```rust
select! {
    _ = cancellation_token.cancelled() => { return (peer_rx, id_c); }
    _ = peer_rx.wait_for(|p| p.is_empty()) => {}
}
```

This is already tracked as `TODO: NET-1774`. [6](#0-5) [7](#0-6)

### Proof of Concept

```rust
#[tokio::test]
async fn byzantine_peer_blocks_task_after_assembly() {
    use tokio::time::{timeout, Duration};

    fn make_assembler() -> MockArtifactAssembler {
        let mut a = MockArtifactAssembler::default();
        a.expect_assemble_message()
            .returning(|id, _, _: PeerWatcher| {
                Box::pin(async move {
                    AssembleResult::Done {
                        message: U64Artifact::id_to_msg(id, 100),
                        peer_id: NODE_1,
                    }
                })
            });
        a
    }

    let (mut mgr, mut channels) = ReceiverManagerBuilder::new()
        .with_artifact_assembler_maker(make_assembler)
        .build();
    let cancellation = CancellationToken::new();

    // Byzantine peer advertises artifact on slot 1.
    mgr.handle_slot_update_receive(
        SlotUpdate { slot_number: SlotNumber::from(1), commit_id: CommitId::from(1), update: Update::Id(0) },
        NODE_1, ConnId::from(1), cancellation.clone(),
    );

    // Wait for Insert to arrive — assembly completed.
    assert!(matches!(
        channels.unvalidated_artifact_receiver.recv().await.unwrap(),
        UnvalidatedArtifactMutation::Insert(_)
    ));

    // Cancel the system — task should terminate.
    cancellation.cancel();

    // Task does NOT terminate within 5 s: Remove is never sent.
    let task_result = timeout(
        Duration::from_secs(5),
        mgr.artifact_processor_tasks.join_next(),
    ).await;

    // This assertion FAILS — the task is still blocked at wait_for.
    assert!(task_result.is_ok(), "Task leaked: did not terminate after cancellation");

    // Pool entry is also never cleaned up.
    assert!(
        timeout(Duration::from_millis(100),
            channels.unvalidated_artifact_receiver.recv()).await.is_err(),
        "Remove was never sent; artifact leaks in unvalidated pool"
    );
}
```
