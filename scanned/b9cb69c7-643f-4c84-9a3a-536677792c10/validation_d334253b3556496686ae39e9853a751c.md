### Title
Missing `is_canonical()` Check Allows Identity-Element TLS Certificate Registration and Potential Node Impersonation — (`rs/crypto/node_key_validation/tls_cert_validation/src/lib.rs`)

---

### Summary

`verify_ed25519_public_key` only calls `is_torsion_free()` and never `is_canonical()`. The three non-canonical encodings of the Ed25519 identity element are torsion-free (confirmed by the codebase's own tests), so they pass the public-key check. Because `ic_ed25519::PublicKey::verify_signature` follows ZIP215 rules, the identity element as a public key degenerates the verification equation to `8·[S]B == 8·R`, which is trivially satisfiable by any attacker. This allows a registered node operator to submit an `add_node` payload with a crafted, self-signed TLS certificate whose public key is the identity element, have it accepted by `ValidTlsCertificate::try_from`, and store it in the registry. Downstream TLS verifiers that use `verify_signature` (ZIP215) to authenticate the certificate's self-signature will then accept any forged signature against that registered key.

---

### Finding Description

**Root cause — missing canonicality check:** [1](#0-0) 

`verify_ed25519_public_key` calls only `is_torsion_free()`. The `ic_ed25519` crate explicitly documents and tests that `is_torsion_free` and `is_canonical` are separate properties: [2](#0-1) 

The three non-canonical encodings of the identity element are confirmed torsion-free by the codebase's own test: [3](#0-2) 

**Self-signature forgery under ZIP215:**

`verify_ed25519_signature` uses `ic_ed25519::PublicKey::verify_signature`, which implements ZIP215: [4](#0-3) 

The ZIP215 verification equation is: [5](#0-4) 

With `A = identity`:
- `minus_a = -identity = identity`
- `recomputed_r = [S]B + [k]·identity = [S]B`
- Check: `([S]B − R)·8 == identity`, i.e., `8·[S]B == 8·R`

This is satisfied for **any message** by choosing any scalar `S` and setting `R = [S]B`. The attacker constructs the TBSCertificate (containing the identity-element public key), picks `S=1`, sets `R=B` (the base point), and the self-signature `(R, S)` passes `verify_ed25519_signature` regardless of the TBSCertificate content.

**Registration path:** [6](#0-5) 

`valid_keys_from_payload` → `ValidNodePublicKeys::try_from` → `ValidTlsCertificate::try_from`. The crafted certificate passes all checks and is stored in the registry via `make_add_node_registry_mutations`. [7](#0-6) 

**Downstream impact — boundary node TlsVerifier:** [8](#0-7) 

`verify_server_cert` extracts the registry public key (the identity element) and calls `verify_signature` (ZIP215). With the identity element, this accepts any presented certificate self-signature. The `verify_tls13_signature` method delegates to `rustls::crypto::ring::default_provider()`. Ring's Ed25519 uses cofactor-cleared verification (BoringSSL heritage), which with `A = identity` also degenerates to `8·[S]B == 8·R`, making CertificateVerify forgeable.

**Downstream impact — node-to-node TLS verifier:** [9](#0-8) 

`ensure_certificates_equal` requires the presented certificate to byte-match the registry certificate, so the attacker must present the exact identity-element certificate. `ensure_node_certificate_is_valid` re-runs `ValidTlsCertificate::try_from` (which passes for the same reasons). `verify_tls13_signature` again uses ring.

---

### Impact Explanation

A registered node operator can register a node whose TLS certificate carries the identity-element public key. Because the identity element degenerates both ZIP215 and cofactor-cleared verification, any party can subsequently impersonate that node in TLS handshakes without possessing a private key. If the node is admitted to a subnet, this enables unauthorized participation in consensus-layer P2P communication under that node's identity.

---

### Likelihood Explanation

The prerequisite is being a registered node operator (requires governance approval). However, node operators are a relatively broad class of participants on the IC. The cryptographic forgery is deterministic and requires no brute force — it is a direct consequence of the identity element's algebraic properties. The three non-canonical encodings are publicly documented. The attack is locally testable as described in the proof of concept.

**Uncertainty:** Whether ring's `UnparsedPublicKey::verify` rejects the identity element as a small-order point before reaching the verification equation is not confirmed from the available source. If ring does reject it, the TLS 1.3 CertificateVerify step would block the handshake impersonation, limiting impact to registry pollution (a weak TLS certificate stored for a node). The registration bypass and ZIP215 self-signature forgery are confirmed regardless.

---

### Recommendation

Add an `is_canonical()` check to `verify_ed25519_public_key`:

```rust
fn verify_ed25519_public_key(
    public_key: &ic_ed25519::PublicKey,
) -> Result<(), TlsCertValidationError> {
    if !public_key.is_torsion_free() || !public_key.is_canonical() {
        return Err(invalid_tls_certificate_error(
            "public key verification failed",
        ));
    }
    Ok(())
}
```

The same gap exists in `ValidNodeSigningPublicKey::try_from` in `rs/crypto/node_key_validation/src/lib.rs` (line 236), which also only calls `is_torsion_free()`. [10](#0-9)

### Proof of Concept

```rust
// 1. Build a TBSCertificate with subject_public_key =
//    0x0100000000000000000000000000000000000000000000000000000000000080
//    (non-canonical identity element, confirmed torsion-free)
//
// 2. Forge self-signature: S=1, R=B (base point).
//    ZIP215 check: 8·[1]B == 8·B  ✓  (independent of TBSCertificate content)
//
// 3. Call ValidTlsCertificate::try_from((cert, node_id, time))
//    → assert Ok(_)
//
// 4. Call ic_ed25519::PublicKey::deserialize_raw(&IDENTITY_NONCANONICAL)
//       .unwrap()
//       .verify_signature(arbitrary_msg, &forged_sig_with_R_equals_SB)
//    → assert Ok(())   // confirms universal forgery
```

```rust
    x509_cert: &X509Certificate,
    public_key: &ic_ed25519::PublicKey,
) -> CryptoResult<()> {
    let sig = x509_cert.signature_value.data.as_ref();

            algorithm: AlgorithmId::Ed25519,
            public_key_bytes: public_key.serialize_raw().to_vec(),
            sig_bytes: sig.to_vec(),
            internal_error: e.to_string(),
        })
```

```rust
    /// Return true if and only if the key is contained within the prime
    /// order subgroup
    pub fn is_torsion_free(&self) -> bool {
        // We don't need to call is_weak here since that is subsumed by the
        // test that the point is torsion free - is_weak just checks if the
        // point is within the size-8 cofactor group.
        self.pk.to_edwards().is_torsion_free()
    }

    /// Return true if and only if the public key uses a canonical encoding
    pub fn is_canonical(&self) -> bool {
        self.pk.to_bytes() == self.pk.to_edwards().compress().0
    }
```

```rust
    pub fn verify_signature(&self, msg: &[u8], signature: &[u8]) -> Result<(), SignatureError> {
        let signature = Signature::from_slice(signature)?;

        let k = Self::compute_challenge(&signature, self, msg);
        let minus_a = -self.pk.to_edwards();
        let recomputed_r =
            EdwardsPoint::vartime_double_scalar_mul_basepoint(&k, &minus_a, signature.s());

        use curve25519_dalek::traits::IsIdentity;

        if (recomputed_r - signature.r())
            .mul_by_cofactor()
            .is_identity()
        {
            Ok(())
        } else {
            Err(SignatureError::InvalidSignature)
        }
    }
```

```rust
        return Err(String::from("transport_tls_cert is empty"));
    };

    // 2. get the keys for verification -- for that, we need to create
    // NodePublicKeys first
    let node_signing_pk = PublicKey::decode(&payload.node_signing_pk[..])
        .map_err(|e| format!("node_signing_pk is not in the expected format: {e:?}"))?;
    let committee_signing_pk = PublicKey::decode(&payload.committee_signing_pk[..])

        .map_err(|e| format!("transport_tls_cert is not in the expected format: {e:?}"))?;

        .map_err(|e| {
            format!("ni_dkg_dealing_encryption_pk is not in the expected format: {e:?}")
        })?;

    // TODO(NNS1-1197): Refactor when nodes are provisioned for threshold ECDSA subnets
    let idkg_dealing_encryption_pk = match &payload.idkg_dealing_encryption_pk {

            return Err(String::from("idkg_dealing_encryption_pk is empty"));
        }

        Some(pk) => PublicKey::decode(&pk[..]).map_err(|e| {
            format!("idkg_dealing_encryption_pk is not in the expected format: {e:?}")
        })?,
    };

    // 3. get the node id from the node_signing_pk
    let node_id = crypto_basicsig_conversions::derive_node_id(&node_signing_pk)
        .map_err(|e| format!("node signing public key couldn't be converted to a NodeId: {e:?}"))?;

    // 4. get the keys for verification -- for that, we need to create
    let node_pks = CurrentNodePublicKeys {
        node_signing_public_key: Some(node_signing_pk),
        committee_signing_public_key: Some(committee_signing_pk),
        tls_certificate: Some(tls_certificate),
        dkg_dealing_encryption_public_key: Some(dkg_dealing_encryption_pk),
        idkg_dealing_encryption_public_key: Some(idkg_dealing_encryption_pk),
    };

    // 5. validate the keys and the node_id
    match ValidNodePublicKeys::try_from(node_pks, node_id, now()?) {
        Ok(valid_pks) => Ok((node_id, valid_pks)),
        Err(e) => Err(format!("Could not validate public keys, due to {e:?}")),
    }
}
```

```rust
fn verify_node_cert(
    end_entity_der: &CertificateDer,
    intermediates: &[CertificateDer],
    allowed_nodes: &SomeOrAllNodes,
    registry_client: &dyn RegistryClient,
    registry_version: RegistryVersion,
    current_time: Time,
) -> Result<(), TLSError> {
    ensure_intermediate_certs_empty(intermediates)?;
    let end_entity_node_id =
        node_id_from_certificate_der(end_entity_der.as_ref()).map_err(|err| match err {
            NodeIdFromCertificateDerError::InvalidCertificate(_) => {
                TLSError::InvalidCertificate(CertificateError::BadEncoding)
            }
            NodeIdFromCertificateDerError::UnexpectedContent(e) => TLSError::InvalidCertificate(
                CertificateError::Other(OtherError(Arc::from(Box::from(e)))),
            ),
        })?;

    ensure_node_id_in_allowed_nodes(end_entity_node_id, allowed_nodes)?;
    let node_cert_from_registry =
        node_cert_from_registry(end_entity_node_id, registry_client, registry_version)?;
    ensure_certificates_equal(
        end_entity_der.as_ref(),
        end_entity_node_id,
        node_cert_from_registry.as_der(),
    )?;
    // It's important to do the validity check after checking equality to the
    // registry cert because the cert validation uses a different parser
    // (`x509_parser` as opposed to OpenSSL that is used above) and it is safer
    // to not just pass any untrusted data to it. We consider the DER here trusted
    // because it is equal to the certificate DER stored in the registry, as checked
    // above.
    ensure_node_certificate_is_valid(end_entity_der.to_vec(), end_entity_node_id, current_time)?;
    Ok(())
}
```
