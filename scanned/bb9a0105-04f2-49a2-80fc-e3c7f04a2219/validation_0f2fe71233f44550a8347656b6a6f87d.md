### Title
Malicious ICRC-1 Ledger Canister Can Permanently Halt Index-NG Canister via Balance Overflow/Underflow Trap - (File: `rs/ledger_suite/icrc1/index-ng/src/main.rs`)

---

### Summary

The ICRC-1 index-ng canister's `credit()` and `debit()` functions call `ic_cdk::trap()` on balance overflow or underflow during block processing. Because `build_index()` is an async function that sets `is_build_index_running = true` before its first `await` point (committing that state), and because `ic_cdk::trap()` terminates Wasm execution without running Rust destructors (including the `scopeguard` guard that is supposed to reset the flag), a malicious ICRC-1 ledger canister can craft a block that permanently halts the index-ng canister's indexing loop.

---

### Finding Description

**Root cause — `credit()` and `debit()` call `ic_cdk::trap()` on attacker-controlled arithmetic:** [1](#0-0) [2](#0-1) 

Both functions call `ic_cdk::trap()` when a block's token amounts cause overflow or underflow. The amounts come directly from blocks returned by the configured ledger canister — an externally controlled source.

**State commitment before the trap:**

`build_index()` sets `is_build_index_running = true` and installs a `scopeguard` to reset it, then immediately hits an `await` point: [3](#0-2) 

In IC's execution model, state changes before an `await` are committed when the inter-canister call is dispatched. The `is_build_index_running = true` mutation is committed at that point.

**Scopeguard destructor never runs on trap:**

`ic_cdk::trap()` calls `ic0.trap()`, a Wasm host function that terminates execution immediately without unwinding the stack. Rust destructors — including the `scopeguard` guard — are never invoked. The flag `is_build_index_running` remains `true` permanently in committed state.

**Timer is never rescheduled:**

The timer reschedule call (`set_build_index_timer`) only executes after `num_indexed` is resolved: [4](#0-3) 

A trap inside `fetch_blocks_via_get_blocks()` or `fetch_blocks_via_icrc3()` prevents this code from ever running. Combined with `is_build_index_running = true`, every future invocation of `build_index()` returns `None` immediately: [5](#0-4) 

**Attacker-controlled entry path:**

A canister developer deploys a malicious ICRC-1-compatible ledger canister. An index-ng canister is initialized pointing to it. The malicious ledger initially behaves normally, then returns a crafted block — for example:
- A `Mint` block crediting `Tokens::MAX` to an account that already has balance `> 0`, triggering overflow in `credit()`.
- A `Burn` block debiting `amount > balance` from an account with insufficient balance, triggering underflow in `debit()`.

The call chain is: `build_index()` → `fetch_blocks_via_get_blocks()` → `get_blocks_from_ledger().await` → (callback) `append_blocks()` → `append_block()` → `process_balance_changes()` → `credit()` / `debit()` → `ic_cdk::trap()`. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The index-ng canister permanently stops indexing blocks. All user-facing queries — `get_account_transactions`, `icrc1_balance_of`, `list_subaccounts` — return stale or empty data. The canister cannot be recovered without an upgrade that resets `is_build_index_running`. Any SNS or token project relying on the index-ng canister for account history loses that service permanently until an operator intervenes.

---

### Likelihood Explanation

Any canister developer can deploy a malicious ICRC-1 ledger canister on the IC. The attack requires no privileged access, no consensus corruption, and no key compromise. The attacker only needs to control the ledger canister that the target index-ng canister is configured to poll. The crafted block is trivial to construct (e.g., a mint of `Tokens::MAX` to any account with nonzero balance, or a burn exceeding any account's balance). The attack is one-shot and irreversible without an operator upgrade.

---

### Recommendation

1. **Replace `ic_cdk::trap()` with error propagation** in `credit()` and `debit()`. Return a `Result` and propagate the error up through `process_balance_changes()` → `append_block()` → `append_blocks()` as a non-retriable `SyncError`, so the indexing timer stops gracefully without corrupting the `is_build_index_running` flag.

2. **Reset `is_build_index_running` in the timer callback wrapper**, not via a scopeguard, so it is reset even if the async body traps. For example, wrap the timer body in a pattern that unconditionally resets the flag in a separate message after the async work completes.

3. **Validate block amounts before processing**: reject blocks where `amount > Tokens::MAX - current_balance` before calling `credit()`, returning a non-retriable sync error rather than trapping.

---

### Proof of Concept

1. Deploy a malicious ICRC-1 ledger canister `M` on the IC.
2. Deploy an index-ng canister `I` with `ledger_id = M`.
3. `M` initially returns valid blocks so `I` indexes normally.
4. `M` then returns a `Mint` block: `{ to: account_A, amount: Tokens::MAX }` where `account_A` already has balance `1` from a prior block.
5. `I`'s timer fires → `build_index()` sets `is_build_index_running = true` → awaits `find_get_blocks_method()` → **state committed**.
6. Callback fetches blocks from `M` → `append_block()` → `process_balance_changes()` → `credit(account_A, Tokens::MAX)` → `1 + Tokens::MAX` overflows → `ic_cdk::trap()`.
7. Trap rolls back callback changes; `is_build_index_running = true` remains committed; scopeguard never runs; timer never rescheduled.
8. All future `build_index()` calls return `None` immediately. `I` is permanently halted.

fn append_block(block_index: BlockIndex64, block: GenericBlock) -> Result<(), SyncError> {
    measure_span(&PROFILING_DATA, "append_blocks", move || {
        let original_block = block.clone();

        let block = match generic_block_to_encoded_block(block) {
            Ok(block) => block,
            Err(e) => {
                let message = format!(
                    "Unable to decode generic block at index {block_index}: {}. Error: {e}",
                    original_block
                );
                return Err(SyncError {
                    message,
                    retriable: false,
                });
            }
        };

        let decoded_block = match Block::<Tokens>::decode(block.clone()) {
            Ok(block) => block,
            Err(e) => {
                let message = format!(
                    "Unable to decode encoded block at index {block_index}: {}. Error: {e}",
                    original_block
                );
                return Err(SyncError {
                    message,
                    retriable: false,
                });
            }
        };
        let decoded_value = encoded_block_to_generic_block(&decoded_block.clone().encode());
        if original_block.hash() != decoded_value.hash() {
            let message = format!(
                "Block at index {block_index} has unknown fields. Original block: {}, decoded block: {}.",
                original_block, decoded_value
            );
            return Err(SyncError {
                message,
                retriable: false,
            });
        }

        // append the encoded block to the block log
        with_blocks(|blocks| {
            blocks
                .append(&block.0)
                .unwrap_or_else(|_| trap("no space left"))
        });

        // add the block idx to the indices
        with_account_block_ids(|account_block_ids| {
            for account in get_accounts(&decoded_block) {
                account_block_ids.insert(account_block_ids_key(account, block_index), ());
            }
        });

        // change the fee collector if block is a 107 block
        process_fee_collector_block(&decoded_block);

        // add the block to the fee_collector if one is set
        index_fee_collector(block_index);

        // change the balance of the involved accounts
        process_balance_changes(block_index, &decoded_block);

        Ok(())
    })
}
