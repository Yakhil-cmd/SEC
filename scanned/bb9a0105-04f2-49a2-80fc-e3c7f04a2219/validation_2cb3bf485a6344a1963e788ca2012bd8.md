### Title
SNS Governance `maybe_finalize_disburse_maturity` Silently Continues After Successful Mint but Failed State Update, Enabling Double-Minting - (File: rs/sns/governance/src/governance.rs)

### Summary

The SNS governance canister's `maybe_finalize_disburse_maturity` function mints SNS tokens to a neuron owner and then attempts to remove the processed `DisburseMaturityInProgress` entry from the neuron's state. If the post-mint state update fails, the function silently `continue`s to the next disbursement without removing the entry. The disbursement entry remains in the queue and will be re-processed on the next heartbeat invocation, causing the same amount to be minted again.

### Finding Description

`maybe_finalize_disburse_maturity` in `rs/sns/governance/src/governance.rs` iterates over all neurons with pending maturity disbursements that have passed their finalization timestamp. For each entry it:

1. Calls `transfer_funds` (a minting transfer) to send tokens to the beneficiary account.
2. On success, calls `get_neuron_result_mut` to obtain a mutable reference to the neuron.
3. Calls `neuron.disburse_maturity_in_progress.remove(0)` to mark the disbursement as complete.

The critical flaw is at step 2–3:

```rust
// rs/sns/governance/src/governance.rs lines 5047-5069
match transfer_result {
    Ok(block_index) => {
        // Tokens already minted to user at this point
        let neuron = match self.get_neuron_result_mut(&neuron_id) {
            Ok(neuron) => neuron,
            Err(e) => {
                log!(ERROR, "Failed updating DisburseMaturityInProgress-entry ...");
                continue; // <-- entry NOT removed; will be re-processed
            }
        };
        neuron.disburse_maturity_in_progress.remove(0); // skipped on error above
    }
    ...
}
```

If `get_neuron_result_mut` returns an error after a successful mint, the `continue` skips `remove(0)`. The disbursement entry persists in `disburse_maturity_in_progress`, and the next heartbeat call to `maybe_finalize_disburse_maturity` will find the same entry still eligible for finalization and mint the tokens a second time.

The NNS governance canister already fixed this exact pattern in its own maturity disbursement path (`rs/nns/governance/src/governance/disburse_maturity.rs`) by reversing the order: it pops the entry from state *before* calling the ledger, and pushes it back if the ledger call fails. The SNS governance has not received the same fix. [1](#0-0) [2](#0-1) 

### Impact Explanation

A successful mint followed by a failed `get_neuron_result_mut` leaves the disbursement entry in place. Every subsequent heartbeat execution of `maybe_finalize_disburse_maturity` will re-mint the same amount until the entry is eventually removed (which never happens automatically under this code path). This inflates the SNS token supply without a corresponding reduction in neuron maturity, constituting a **ledger conservation bug**: tokens are created without burning the backing maturity, diluting all other token holders. [3](#0-2) 

### Likelihood Explanation

`maybe_finalize_disburse_maturity` is called on every SNS governance heartbeat for every SNS deployment. [4](#0-3) 

`get_neuron_result_mut` can fail if the neuron is absent from `self.proto.neurons`. Although the neuron is locked via `_neuron_lock` for the duration of the loop iteration, the lock only prevents concurrent *commands* on the neuron; it does not prevent the neuron from being absent due to a prior inconsistency or a future code change that removes neurons. Between the `await` on `transfer_funds` and the subsequent synchronous `get_neuron_result_mut`, the IC execution model allows other ingress messages to run, which could alter canister state. Any SNS neuron holder who initiates a `DisburseMaturity` request is an unprivileged entry point that places an entry into `disburse_maturity_in_progress`, making this reachable without any privileged access. [5](#0-4) 

### Recommendation

Adopt the same pattern used by the NNS governance fix: **pop the disbursement entry from state before calling the ledger**, and push it back to the front of the queue if the ledger call fails. This ensures the entry is never present in the queue while tokens have already been minted for it.

```rust
// Step 1: remove the entry from state first (no ledger call yet)
let neuron = self.get_neuron_result_mut(&neuron_id)?;
neuron.disburse_maturity_in_progress.remove(0);

// Step 2: call the ledger
let transfer_result = self.ledger.transfer_funds(...).await;

// Step 3: on failure, restore the entry
if let Err(e) = transfer_result {
    if let Ok(neuron) = self.get_neuron_result_mut(&neuron_id) {
        neuron.disburse_maturity_in_progress.insert(0, disbursement);
    }
    continue;
}
```

This mirrors the NNS governance implementation at `rs/nns/governance/src/governance/disburse_maturity.rs` lines 612–663. [6](#0-5) 

### Proof of Concept

1. A neuron owner calls `DisburseMaturity` on an SNS governance canister, placing an entry in `disburse_maturity_in_progress` with `finalize_disbursement_timestamp_seconds = now + MATURITY_DISBURSEMENT_DELAY_SECONDS`.
2. After the delay, the heartbeat triggers `maybe_finalize_disburse_maturity`.
3. `transfer_funds` succeeds — tokens are minted to the owner's account.
4. `get_neuron_result_mut` returns `Err` (e.g., due to a neuron store inconsistency introduced by a concurrent message processed during the `await`).
5. The code logs the error and executes `continue`; `disburse_maturity_in_progress.remove(0)` is never called.
6. On the next heartbeat, the same entry is still present and still past its finalization timestamp.
7. `transfer_funds` is called again — a second identical mint is issued to the owner's account.
8. Steps 4–7 repeat indefinitely, minting unbounded tokens from the SNS treasury. [7](#0-6) [8](#0-7)
