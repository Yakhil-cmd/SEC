### Title
Query Stats Aggregation Multiplies Median by Total Subnet Size Instead of Reporting-Node Count, Inflating Canister `total_query_stats` - (File: rs/query_stats/src/state_machine.rs)

### Summary
In `try_aggregate_one_epoch`, the per-canister median query stat is computed from `num_nodes_with_stats` values (padded with zeros), but then passed to `apply_query_stats_to_canister` with `num_nodes` (the full subnet size) as the scale factor. Because `num_nodes_with_stats` can be as low as ⌈2/3 · num_nodes⌉, the stored `total_query_stats` for every canister is systematically inflated by up to `num_nodes / num_nodes_with_stats` — up to ~50 % on a 13-node subnet — every time an epoch is aggregated.

### Finding Description
`try_aggregate_one_epoch` in `rs/query_stats/src/state_machine.rs` performs the following steps:

1. Reads `num_nodes` = total subnet size from `replicated_state.system_metadata().own_subnet_size()`.
2. Counts `num_nodes_with_stats` = the number of nodes that have fully committed a record for the epoch being aggregated (must be ≥ `num_nodes − get_faults_tolerated(num_nodes)`, i.e. > 2/3 of `num_nodes`).
3. For each canister, pads the per-node stat vector with zeros **up to `num_nodes_with_stats`** (not up to `num_nodes`).
4. Computes the median of those `num_nodes_with_stats` values.
5. Calls `apply_query_stats_to_canister(..., num_nodes, ...)`.

Inside `apply_query_stats_to_canister` the median is scaled:

```rust

The median was derived from a sample of size `num_nodes_with_stats`, but the multiplier is `num_nodes`. The two quantities are inconsistent: the design comment at lines 61–68 explicitly states that non-submitting nodes are **excluded** from the median calculation, yet their count is silently included in the multiplier. [2](#0-1) 

The padding step that normalises the per-canister vector uses `num_nodes_with_stats`, not `num_nodes`:

```rust
// line 341
let num_empty_stats = num_nodes_with_stats.saturating_sub(stats.len());
stats.append(&mut vec![&empty_stats; num_empty_stats]);
``` [3](#0-2) 

The `num_nodes` value is fetched from the subnet-level metadata, which is the aggregate count of all nodes — analogous to reading a pool-level value rather than a position-specific one. [4](#0-3) 

### Impact Explanation
`total_query_stats` is persisted in every canister's `SystemState` and is already exposed through `canister_status` (management canister). It is the intended basis for future query-call cycles billing. [5](#0-4) 

The inflation factor per epoch is `num_nodes / num_nodes_with_stats`. On a 13-node subnet where exactly 9 nodes submit (the minimum required), the factor is 13/9 ≈ 1.44 — a 44 % overcount applied to every canister's cumulative query statistics every epoch. Because the error accumulates with each epoch, long-running canisters will carry a proportionally larger inflated total.
