### Title
Unbounded Iteration in `get_pending_proposals` Enables Query-Level DoS via Proposal Flooding - (File: `rs/nns/governance/src/governance.rs`)

### Summary

The NNS Governance canister exposes a public query endpoint `get_pending_proposals` that iterates over **all** open proposals without any result limit or instruction-count guard. Because `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS` permits up to 10,000 simultaneously open manage-neuron proposals, an attacker who controls neurons with sufficient stake can flood the proposal queue and cause `get_pending_proposals` to exhaust the 5 billion-instruction query limit, permanently breaking the endpoint for all callers.

---

### Finding Description

`get_pending_proposals` in `rs/nns/governance/src/governance.rs` scans the entire `heap_data.proposals` map, filters for `ProposalStatus::Open`, converts each matching entry to a `ProposalInfo` (including ballot filtering), and collects the results into a `Vec` with no upper bound on the number of entries returned:

```rust
// rs/nns/governance/src/governance.rs  lines 3469-3495
pub fn get_pending_proposals(
    &self,
    caller: &PrincipalId,
    req: Option<GetPendingProposalsRequest>,
) -> Vec<ProposalInfo> {
    let now = self.env.now();
    let caller_neurons = self.get_neuron_ids_by_principal(caller);
    ...
    self.heap_data
        .proposals
        .values()
        .filter(|data| data.status() == ProposalStatus::Open)
        .map(|data| {
            proposal_data_to_info(
                data,
                ProposalDisplayOptions::for_get_pending_proposals(...),
                &caller_neurons,
                now,
                self.voting_period_seconds(),
            )
        })
        .collect()   // ← no .take(limit) anywhere
}
``` [1](#0-0) 

The canister enforces two separate open-proposal caps:

- Regular (non-manage-neuron) proposals: `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 200`
- Manage-neuron proposals: `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS = 10_000` [2](#0-1) 

Both categories are included in the `ProposalStatus::Open` filter, so the loop can touch up to **10,200 proposals** in a single query call. Each `proposal_data_to_info` call must iterate over the proposal's ballot map (which for regular proposals can contain entries for every eligible neuron) to filter the caller's ballots. The combined work across all open proposals can easily exceed the 5 billion-instruction query limit (`MAX_INSTRUCTIONS_PER_QUERY_MESSAGE`). [3](#0-2) 

The canister endpoint is declared as a `#[query]` in `rs/nns/governance/canister/canister.rs` and is listed in the public Candid interface: [4](#0-3) [5](#0-4) 

By contrast, `list_proposals` — the paginated sibling — correctly applies a `MAX_LIST_PROPOSAL_RESULTS` cap and a `before_proposal` cursor, so it is not affected. [6](#0-5) 

---

### Impact Explanation

An attacker who fills the manage-neuron proposal queue to its 10,000-entry limit causes every subsequent call to `get_pending_proposals` to iterate over all 10,000+ open proposals in a single query execution. Because query calls on the IC are bounded by `MAX_INSTRUCTIONS_PER_QUERY_MESSAGE = 5 × 10⁹` instructions, the call traps with `CanisterInstructionLimitExceeded`. The endpoint becomes permanently unusable until the open proposals are resolved — which requires the voting period to expire (days to weeks). External integrations that depend on `get_pending_proposals` (e.g., the ICP Rosetta API, which calls this endpoint to serve `/construction/parse` voting data) are also broken. [7](#0-6) 

---

### Likelihood Explanation

The attacker needs neurons with enough stake and dissolve delay to submit manage-neuron proposals. The NNS proposal fee is 0.01 ICP per proposal; submitting 10,000 proposals costs ~100 ICP plus the neuron stake requirement. This is economically feasible for a motivated adversary. The attack is also self-sustaining: once the queue is full, the attacker only needs to replace proposals that expire, keeping the queue saturated indefinitely at low ongoing cost. No privileged access, no threshold corruption, and no external dependency is required — only standard NNS neuron operations.

---

### Recommendation

Apply the same fix recommended in H-03: add pagination to `get_pending_proposals`. Specifically:

1. Accept an optional `before_proposal` cursor and a `limit` parameter (mirroring `list_proposals`).
2. Apply `.take(limit)` before `.collect()`, capping results at a safe constant (e.g., `MAX_LIST_PROPOSAL_RESULTS = 100` already used by `list_proposals`).
3. Alternatively, add an instruction-count guard inside the loop (analogous to `is_message_over_threshold` already used in `process_voting_state_machines` and `distribute_pending_rewards`) and return a partial result with a continuation cursor. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker creates or controls a neuron with sufficient stake and dissolve delay to submit manage-neuron proposals.
2. Attacker submits manage-neuron proposals targeting any neuron (e.g., their own) up to `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS = 10_000`.
3. The check in `make_proposal` only rejects the *next* proposal once the limit is reached; the 10,000 already-open proposals remain. [10](#0-9) 

4. Any caller now invokes `get_pending_proposals` (a public query, no authentication required).
5. The function iterates over all 10,000+ open proposals, calling `proposal_data_to_info` for each, exhausting the 5B-instruction query budget.
6. The query traps; the endpoint is DoS'd for all callers until the proposals expire.

The existing test `test_max_number_of_proposals_with_ballots` confirms that 200 regular proposals can be open simultaneously and that `get_pending_proposals` returns all of them without any limit — demonstrating the unbounded collection behavior: [11](#0-10)
