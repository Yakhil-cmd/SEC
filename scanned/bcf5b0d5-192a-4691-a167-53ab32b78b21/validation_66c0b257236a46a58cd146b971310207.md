### Title
No Minimum Deposit Amount in ckETH Minter Creates Permanently Irrecoverable ETH Dust — (`File: rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

The ckETH minter accepts ETH deposits of any size, including dust amounts far below the minimum withdrawal threshold. Because the minter enforces a `minimum_withdrawal_amount` on the withdrawal side but has no corresponding `minimum_deposit_amount` on the deposit side, any ETH deposited below the withdrawal floor is permanently locked in the minter's Ethereum address and can never be redeemed. This is the direct IC analog of the "dust deposit → bad debt" pattern described in the reference report.

---

### Finding Description

The ckBTC and ckDOGE minters both enforce an explicit minimum deposit amount before minting. In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, UTXOs below `deposit_btc_min_amount` are rejected with `UtxoStatus::ValueTooSmall`. [1](#0-0) 

The ckDOGE minter has the same guard via `deposit_doge_min_amount`. [2](#0-1) 

The ckETH minter, however, has **no equivalent minimum deposit check**. The `register_deposit_events` function in `rs/ethereum/cketh/minter/src/deposit.rs` only filters deposits from OFAC-blocked addresses. Any non-blocked deposit of any size — including 1 wei — is unconditionally accepted and queued for minting: [3](#0-2) 

The minter's `InitArg` and `UpgradeArg` expose only `minimum_withdrawal_amount`; there is no `minimum_deposit_amount` field at all: [4](#0-3) [5](#0-4) 

The `MinterInfo` response similarly exposes only `minimum_withdrawal_amount`, confirming no deposit floor exists: [6](#0-5) 

The `mint()` function in `rs/ethereum/cketh/minter/src/deposit.rs` calls `icrc1_transfer` with `amount: event.value()` — the raw deposited amount — with no floor check before attempting the ledger call: [7](#0-6) 

---

### Impact Explanation

The current `minimum_withdrawal_amount` is `5_000_000_000_000_000` wei (0.005 ETH, ~$10 at current prices), as set by the most recent upgrade proposal: [8](#0-7) 

Any ETH deposit below this threshold results in:

1. The ETH being transferred to and held by the minter's Ethereum address (controlled via threshold ECDSA).
2. The minter minting the corresponding dust ckETH amount to the depositor.
3. The depositor being **permanently unable to redeem** the dust ckETH back to ETH, because `withdraw_eth` enforces `amount >= minimum_withdrawal_amount`: [9](#0-8) 

The deposited ETH is therefore permanently locked — it cannot be withdrawn by the user, and the protocol has no mechanism to sweep or consolidate dust ETH balances (unlike ckBTC's UTXO consolidation). This is the direct analog of "bad debt": the minter holds real ETH that is economically irrecoverable.

The same issue applies to ckERC20 deposits: the `register_deposit_events` path handles both `ReceivedEth` and `ReceivedErc20` events with the same no-minimum logic. [10](#0-9) 

---

### Likelihood Explanation

**Medium.** The attack path requires only a standard Ethereum transaction to the public helper contract. The `deposit()` function on `CkEthDeposit.sol` accepts any `msg.value`, including 1 wei: [11](#0-10) 

While an intentional attacker would pay Ethereum gas (~$0.10–$1 per transaction) to lock negligible ETH, accidental dust deposits are a realistic everyday occurrence (e.g., rounding errors in DeFi integrations, test transactions, wallet software bugs). Each such deposit permanently removes ETH from circulation. At scale, accumulated dust represents a ledger conservation imbalance: total ckETH supply includes dust tokens that can never be burned via the normal withdrawal path.

---

### Recommendation

Add a `minimum_deposit_amount` field to `InitArg` and `UpgradeArg` (mirroring the ckBTC pattern), and enforce it inside `register_deposit_events` before calling `process_event(s, event.into_deposit())`. Deposits below the floor should be recorded as `InvalidDeposit` with a descriptive reason, consistent with how ckBTC handles `UtxoStatus::ValueTooSmall`. The minimum should be set to at least `minimum_withdrawal_amount` to guarantee every accepted deposit can eventually be redeemed.

---

### Proof of Concept

1. Deploy or use the existing mainnet ckETH helper contract at `0x18901044688D3756C35Ed2b36D93e6a5B8e00E68`.
2. Call `deposit(bytes32 principal)` with `msg.value = 1` (1 wei) and a valid IC principal encoded as `bytes32`.
3. Wait for the minter to scrape the `ReceivedEth` log (next scraping cycle, typically within minutes).
4. Observe that `register_deposit_events` accepts the event without any minimum check and queues it for minting.
5. The minter mints 1 wei of ckETH to the specified principal.
6. Attempt `withdraw_eth` with `amount = 1`. The call returns `WithdrawalError::AmountTooLow { min_withdrawal_amount: 5_000_000_000_000_000 }`.
7. The 1 wei of ETH is now permanently locked in the minter's Ethereum address. The 1 wei of ckETH cannot be redeemed.

* Reduce the minimum ETH withdrawal amount by a factor of 6, from 0.03 ETH (`30_000_000_000_000_000` wei) to 0.005 ETH (`5_000_000_000_000_000` wei) — approximately $10 at current prices. The reasoning is as follows:

At that time ETH traded in a similar USD range (around $2000), but Ethereum mainnet transaction fees were averaging $5–$10 per transaction ([source](https://bitinfocharts.com/comparison/ethereum-transactionfees.html#3y)).
