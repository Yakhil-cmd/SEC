### Title
SNS Swap Integer Division Truncation in `Swap::scale()` Permanently Locks SNS Tokens - (File: `rs/sns/swap/src/swap.rs`)

### Summary
The `Swap::scale()` function uses integer (floor) division to compute each participant's SNS token allocation during swap finalization. The sum of all per-participant allocations is always less than or equal to `sns_being_offered_e8s`, with the remainder ("change") permanently locked in the swap canister's SNS ledger account. No mechanism exists to redistribute or return these tokens after finalization.

### Finding Description
During SNS swap finalization, `create_sns_neuron_recipes()` calls `Swap::scale()` for every direct participant and every Neurons' Fund neuron to compute their proportional SNS token allocation: [1](#0-0) 

The computation is `floor((amount_icp_e8s * total_sns_e8s) / total_icp_e8s)`. Integer floor division discards the remainder for every participant. The cumulative undistributed amount is tracked in `total_sns_tokens_sold_e8s` and the code explicitly logs the "change": [2](#0-1) 

The "change" (`sns_being_offered_e8s - total_sns_tokens_sold_e8s`) is only logged — it is never included in any `SnsNeuronRecipe`, never swept via `sweep_sns()`, and never returned to the SNS governance canister or treasury. After `sweep_sns()` completes, these tokens remain stranded in the swap canister's SNS ledger subaccount with no recovery path.

The same truncation applies to Neurons' Fund participants at line 920: [3](#0-2) 

The `apportion_approximately_equally()` helper used inside `generate_vesting_schedule()` does correctly preserve the total within a single basket: [4](#0-3) 

However, the cross-participant truncation in `Swap::scale()` is the outer layer that is never corrected.

### Impact Explanation
Every successfully committed SNS swap permanently locks between 0 and `(N_direct + N_nf - 1)` SNS e8s in the swap canister, where `N_direct` is the number of direct buyers and `N_nf` is the number of Neurons' Fund neurons. These tokens are not burned, not returned to the SNS treasury, and not distributed to any participant. With hundreds of participants the locked amount is bounded but non-zero and irrecoverable. The vulnerability class is **ledger conservation bug**: tokens that should be distributed are silently discarded by integer truncation, directly analogous to the `amount.sub(amount.mod(lotSize))` pattern in the reference report.

### Likelihood Explanation
This condition is triggered on every SNS swap that commits with more than one participant whose ICP contribution is not an exact divisor of `total_participant_icp_e8s`. In practice this is virtually every real swap. The entry path requires no privileged access: any unprivileged principal can participate in an open SNS swap via `refresh_buyer_tokens`, and the truncation occurs automatically during `finalize_swap` / `create_sns_neuron_recipes`.

### Recommendation
After `sweep_sns()` completes, query the swap canister's remaining SNS ledger balance and transfer any residual amount back to the SNS governance canister treasury account. Alternatively, assign the remainder to the last participant's basket (similar to how `apportion_approximately_equally` assigns the remainder to the last element).

### Proof of Concept
Consider a swap with `sns_being_offered_e8s = 100` and three participants each contributing 1 e8s ICP (`total_participant_icp_e8s = 3`):

```
Swap::scale(1, 100, 3) = floor(100 / 3) = 33   // participant A
Swap::scale(1, 100, 3) = floor(100 / 3) = 33   // participant B
Swap::scale(1, 100, 3) = floor(100 / 3) = 33   // participant C
total_sns_tokens_sold_e8s = 99
change = 100 - 99 = 1 SNS e8s  ← permanently locked in swap canister
```

The log at line 985 will print `change 1` but no code path ever transfers this 1 e8s back to the SNS treasury or to any participant. After `sweep_sns()` the swap canister's SNS ledger account retains this balance indefinitely. [5](#0-4) [6](#0-5)

    fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
        assert!(amount_icp_e8s <= u64::from(total_icp_e8s));
        // Note that the multiplication cannot overflow as both factors fit in 64 bits.
        let r = (amount_icp_e8s as u128)
            .saturating_mul(total_sns_e8s as u128)
            .div(NonZeroU128::from(total_icp_e8s));
        // This follows logically from the initial assert `amount_icp_e8s <= total_icp_e8s`.
        assert!(r <= u64::MAX as u128);
        r as u64
    }
