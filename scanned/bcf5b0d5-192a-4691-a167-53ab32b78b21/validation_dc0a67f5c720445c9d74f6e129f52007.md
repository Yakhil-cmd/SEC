### Title
`ManageNervousSystemParameters` Updates `voting_rewards_parameters` Without First Settling Pending Rewards - (`File: rs/sns/governance/src/governance.rs`)

### Summary

The SNS governance canister's `perform_manage_nervous_system_parameters` function updates `VotingRewardsParameters` (including `initial_reward_rate_basis_points` and `final_reward_rate_basis_points`) without first calling `distribute_rewards`. As a result, any elapsed-but-unsettled reward rounds are subsequently calculated using the newly updated rate rather than the rate that was in effect when those rounds occurred, leading to inaccurate reward distribution for all SNS neuron holders.

### Finding Description

When a `ManageNervousSystemParameters` proposal is adopted and executed, the SNS governance canister calls `perform_manage_nervous_system_parameters`:

```rust
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            self.proto.parameters = Some(new_params); // ← rate updated immediately
            Ok(())
        }
        ...
    }
}
``` [1](#0-0) 

This directly overwrites `self.proto.parameters` — which contains `voting_rewards_parameters` — without first distributing pending rewards.

The periodic `distribute_rewards` function later calculates the rewards purse for all elapsed rounds since the last `RewardEvent` using the **current** (already updated) `voting_rewards_parameters`:

```rust
let current_reward_rate = voting_rewards_parameters.reward_rate_at(
    crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
);
result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
``` [2](#0-1) 

`reward_rate_at` derives the rate from `self.initial_reward_rate_basis_points` and `self.final_reward_rate_basis_points` of the **current** `VotingRewardsParameters`:

```rust
let dr = self.initial_reward_rate() - self.final_reward_rate();
let variable_reward_rate = s2 * dr;
self.final_reward_rate() + variable_reward_rate
``` [3](#0-2) 

Because `distribute_rewards` is driven by a periodic timer (not by proposal execution), there will always be at least a partial elapsed window between the last `RewardEvent` and the moment the parameter change takes effect. Any complete rounds in that window are retroactively recalculated at the new rate. [4](#0-3) 

### Impact Explanation

**Governance reward conservation bug.** All SNS neuron holders who voted on proposals in the pending window receive incorrect maturity increments. If `initial_reward_rate_basis_points` is lowered (e.g., from 1000 bps to 100 bps), neuron holders are under-rewarded for rounds that occurred at the higher rate. If it is raised, they are over-rewarded. Because maturity is the precursor to minting SNS tokens, this directly affects token economics and violates the invariant that rewards are calculated at the rate in effect during the period they were earned.

### Likelihood Explanation

**Medium.** Any SNS that legitimately updates its `voting_rewards_parameters` via a `ManageNervousSystemParameters` proposal — a standard, documented governance action — triggers this condition. The SNS proto comment itself warns: *"Once this is set, it probably should not be changed, because the results would probably be pretty confusing."* [5](#0-4) 

This warning acknowledges the problem but provides no enforcement or mitigation. Any SNS community that votes to adjust reward rates (a common governance action) will silently miscalculate rewards for the pending window.

### Recommendation

`perform_manage_nervous_system_parameters` should call `distribute_rewards` (or an equivalent settlement step) **before** overwriting `self.proto.parameters` whenever `voting_rewards_parameters` is being changed. This ensures all rounds elapsed under the old rate are settled at that rate before the new rate takes effect. [6](#0-5) 

### Proof of Concept

1. An SNS is initialized with `initial_reward_rate_basis_points = 1000` (10 %) and `final_reward_rate_basis_points = 500` (5 %).
2. One full reward round elapses. `distribute_rewards` has not yet been called for this round (the timer fires slightly after the round boundary).
3. A `ManageNervousSystemParameters` proposal is adopted that sets `initial_reward_rate_basis_points = 50` (0.5 %) and `final_reward_rate_basis_points = 10` (0.1 %).
4. `perform_manage_nervous_system_parameters` immediately writes the new parameters to `self.proto.parameters`.
5. The periodic timer fires and calls `distribute_rewards`. It computes `new_rounds_count = 1` and calls `voting_rewards_parameters.reward_rate_at(...)` — which now returns ≈ 0.5 % instead of ≈ 10 %.
6. All neuron holders receive ~20× less maturity than they were entitled to for that round, with no recourse. [7](#0-6)
