### Title
Cross-Subaccount Allowance Enumeration via Incomplete Break Condition in `get_allowances` — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

### Summary
The `get_allowances` loop terminates only when `account_spender.account.owner != from.owner`. Because `Account::cmp` orders by `(owner, effective_subaccount)`, a range scan starting at `{owner:X, subaccount:[0u8;32]}` will continue through `{owner:X, subaccount:[1u8;32]}`, `{owner:X, subaccount:[2u8;32]}`, etc. — all of which pass the owner-only break check. Any unprivileged caller can therefore enumerate allowances belonging to subaccounts of X that were never queried.

### Finding Description

`Account::cmp` sorts first by `owner`, then by `effective_subaccount()`: [5](#0-4) 

So the `ALLOWANCES_MEMORY` `StableBTreeMap` is ordered as:
```
{owner:X, sub:[0;32], spender:S1}
{owner:X, sub:[0;32], spender:S2}
{owner:X, sub:[1;32], spender:S3}   ← different subaccount, same owner
{owner:X, sub:[2;32], spender:S4}   ← different subaccount, same owner
{owner:Y, ...}
```

A call with `from_account = {owner:X, subaccount:Some([0u8;32])}` starts the range at `{account:{owner:X, sub:[0;32]}, spender:empty_principal}`. The loop iterates forward and hits entries for `sub:[1;32]`, `sub:[2;32]`, etc. The break guard:

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

evaluates to `false` for all of them (owner is still X), so they are all appended to `result` and returned to the caller. [6](#0-5) 

The correct guard must compare the full `Account` (which uses `effective_subaccount()` in its `PartialEq`):
```rust
if account_spender.account != from {
    break;
}
``` [7](#0-6) 

### Impact Explanation
An unprivileged attacker can call `icrc103_get_allowances` with `from_account = {owner:X, subaccount:Some([0u8;32])}` and receive allowances stored under `{owner:X, subaccount:Some([1u8;32])}`, `{owner:X, subaccount:Some([2u8;32])}`, etc. The response leaks: spender identities, approved amounts, and expiration timestamps for every subaccount of X that is lexicographically greater than the queried subaccount. This is allowance metadata disclosure across subaccount boundaries of the same principal.

### Likelihood Explanation
The endpoint is a public query with no authentication or authorization check. [4](#0-3) 
Any principal can call it with an arbitrary `from_account`. The precondition (owner X has approved from multiple subaccounts) is a normal usage pattern for wallets and DeFi protocols. Exploitation requires only a single query call.

### Recommendation
Replace the owner-only break condition with a full-account comparison:

```rust
// Before (line 1204):
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```

`Account::PartialEq` already normalizes `None` and `Some([0u8;32])` via `effective_subaccount()`, so this handles the default-subaccount equivalence correctly. [3](#0-2)

fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    let max_take_allowances = Access::with_ledger(|ledger| ledger.max_take_allowances());
    let max_results = arg
        .take
        .map(|take| take.0.to_u64().unwrap_or(max_take_allowances))
        .map(|take| std::cmp::min(take, max_take_allowances))
        .unwrap_or(max_take_allowances);
    Ok(get_allowances(
        from_account,
        arg.prev_spender,
        max_results,
        ic_cdk::api::time(),
    ))
