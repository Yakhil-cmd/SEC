### Title
Overly Broad Self-Approval Check Blocks Legitimate Cross-Subaccount `icrc2_approve` Operations - (File: `rs/ledger_suite/icrc1/ledger/src/main.rs` and `rs/ledger_suite/icp/ledger/src/main.rs`)

---

### Summary

Both the ICRC-1 ledger and the ICP ledger implement a self-approval guard in `icrc2_approve_not_async` that compares only the `owner` (Principal) field of the `from` account and the `spender` account, rather than the full ICRC-1 account identity (owner + subaccount). This causes the ledger to unconditionally trap and reject any approval where the caller's principal matches the spender's principal, even when the two accounts are distinct (i.e., they have different subaccounts). This is the IC analog of the reported vulnerability: an overly broad guard in a shared validation path blocks a class of legitimate token operations.

---

### Finding Description

In ICRC-1, an account is uniquely identified by the pair `{owner: Principal, subaccount: Option<[u8; 32]>}`. Two accounts sharing the same `owner` but having different `subaccount` values are semantically distinct accounts. A principal controlling multiple subaccounts (e.g., a canister managing treasury sub-pools) has a legitimate need to approve one subaccount to spend from another.

The `icrc2_approve_not_async` function in the ICRC-1 ledger performs the following check:

```rust
if from_account.owner == arg.spender.owner {
    ic_cdk::trap("self approval is not allowed")
}
``` [1](#0-0) 

The identical pattern appears in the ICP ledger:

```rust
if from_account.owner == arg.spender.owner {
    trap("self approval is not allowed");
}
``` [2](#0-1) 

Both checks compare only the `owner` field (a `Principal`), not the full account. This means any call where `from_subaccount != spender.subaccount` but `caller == spender.owner` is unconditionally trapped, even though the two accounts are different.

The lower-level `approvals.approve()` function in the shared ledger core correctly uses full-account equality:

```rust
if account == spender {
    return Err(ApproveError::SelfApproval);
}
``` [3](#0-2) 

The upper-level guard is therefore more restrictive than the lower-level guard, and it prevents legitimate cross-subaccount approvals from ever reaching the correct check.

---

### Impact Explanation

Any principal that holds balances across multiple subaccounts and wishes to grant one subaccount spending authority over another subaccount of the same principal is permanently blocked. The `ic_cdk::trap` call is unrecoverable — it does not return a structured `ApproveError`; it aborts the call entirely. This means:

- Cross-subaccount delegation via `icrc2_approve` is impossible for any principal.
- Downstream `icrc2_transfer_from` flows that depend on such approvals are also blocked.
- Canisters that manage multiple subaccounts (e.g., per-user treasury accounts under a single canister principal) cannot use the ICRC-2 approval mechanism for internal delegation.

This is a ledger conservation / token-operation-blocking bug: a class of valid, standard-compliant operations is silently forbidden by an overly broad guard, directly analogous to the reported `_update` override blocking all mint and burn operations.

---

### Likelihood Explanation

The entry path is a direct, unprivileged ingress call to `icrc2_approve` on either the ICRC-1 ledger canister or the ICP ledger canister. No special role, key, or governance action is required. Any user or canister that:

1. Controls a principal with tokens in two different subaccounts, and
2. Calls `icrc2_approve` with `from_subaccount = Some(subaccount_A)` and `spender = { owner: same_principal, subaccount: Some(subaccount_B) }`

will trigger the trap. This is a realistic and common pattern for canisters managing per-user or per-pool subaccounts.

---

### Recommendation

Replace the owner-only comparison with a full-account comparison in both ledgers:

**ICRC-1 ledger** (`rs/ledger_suite/icrc1/ledger/src/main.rs`):
```rust
// Before (incorrect):
if from_account.owner == arg.spender.owner {
    ic_cdk::trap("self approval is not allowed")
}

// After (correct):
if from_account == arg.spender {
    ic_cdk::trap("self approval is not allowed")
}
```

**ICP ledger** (`rs/ledger_suite/icp/ledger/src/main.rs`):
```rust
// Before (incorrect):
if from_account.owner == arg.spender.owner {
    trap("self approval is not allowed");
}

// After (correct):
if from_account == arg.spender {
    trap("self approval is not allowed");
}
```

This aligns the upper-level guard with the lower-level `approvals.approve()` check and with the ICRC-2 standard's intent that "self" means the same full account, not the same principal.

---

### Proof of Concept

A canister `C` with principal `alice` holds tokens in two subaccounts:
- `from_account = { owner: alice, subaccount: Some([0x01; 32]) }`
- `spender = { owner: alice, subaccount: Some([0x02; 32]) }`

Calling `icrc2_approve` with these arguments:

```
icrc2_approve({
  from_subaccount: Some([0x01; 32]),
  spender: { owner: alice, subaccount: Some([0x02; 32]) },
  amount: 1_000_000,
  ...
})
```

The ledger evaluates `from_account.owner == arg.spender.owner` → `alice == alice` → `true`, and traps with `"self approval is not allowed"`, even though `from_account != arg.spender` (different subaccounts). The approval is permanently rejected. Any subsequent `icrc2_transfer_from` that would have relied on this allowance also fails. [4](#0-3) [5](#0-4)
