### Title
ckETH Minter `withdraw_eth` Panics on `BadBurn` Due to Hardcoded Ledger Fee Validation — (`rs/ethereum/cketh/minter/src/endpoints.rs`)

---

### Summary

The ckETH minter's `validate_config` guards against `BadBurn` by checking `minimum_withdrawal_amount` against a **hardcoded** ledger fee constant rather than the actual on-chain fee. If the ckETH ledger's `transfer_fee` is raised via NNS governance above `minimum_withdrawal_amount`, the guard passes silently, but any subsequent `withdraw_eth` call **panics** (traps the canister message) instead of returning a graceful `WithdrawalError::AmountTooLow` to the caller. This permanently breaks the core ckETH withdrawal endpoint for all users until the minter is upgraded.

---

### Finding Description

**Step 1 — Hardcoded fee in `validate_config`**

`State::validate_config` in `rs/ethereum/cketh/minter/src/state.rs` hardcodes the expected ckETH ledger `transfer_fee`:

```rust
let cketh_ledger_transfer_fee = match self.ethereum_network {
    EthereumNetwork::Mainnet => Wei::new(2_000_000_000_000),
    EthereumNetwork::Sepolia => Wei::new(10_000_000_000),
};
if self.cketh_minimum_withdrawal_amount < cketh_ledger_transfer_fee {
    return Err(InvalidStateError::InvalidMinimumWithdrawalAmount(
        "minimum_withdrawal_amount must cover ledger transaction fee, \
        otherwise ledger can return a BadBurn error that should be returned to the user"
            .to_string(),
    ));
}
```

The comment explicitly acknowledges that `BadBurn` **"should be returned to the user"**, but the hardcoded constant `2_000_000_000_000` is never updated when the actual ledger fee changes. The minter never queries the ledger's real `transfer_fee` at runtime.

**Step 2 — `From<LedgerBurnError> for WithdrawalError` panics on `AmountTooLow`**

`rs/ethereum/cketh/minter/src/endpoints.rs` converts `LedgerBurnError` to `WithdrawalError` for `withdraw_eth`. Every variant is handled gracefully **except** `AmountTooLow`, which panics:

```rust
LedgerBurnError::AmountTooLow {
    minimum_burn_amount,
    failed_burn_amount,
    ledger,
} => {
    panic!(
        "BUG: withdrawal amount {failed_burn_amount} on the ckETH ledger {ledger:?} \
        should always be higher than the ledger transaction fee {minimum_burn_amount}"
    )
}
```

**Step 3 — `withdraw_eth` propagates the panic**

`rs/ethereum/cketh/minter/src/main.rs` calls `Err(WithdrawalError::from(e))` on the burn error, which triggers the panic path above:

```rust
Err(e) => Err(WithdrawalError::from(e)),
```

**Step 4 — Inconsistency with `withdraw_erc20`**

`withdraw_erc20` in the same file handles `LedgerBurnError::AmountTooLow` gracefully by returning `WithdrawErc20Error::CkEthLedgerError { error: LedgerError::AmountTooLow { ... } }`. The `withdraw_eth` path has no equivalent graceful path.

**Trigger condition**

The ICRC-1 ledger returns `BadBurn` when `amount < min(transfer_fee, balance)`. If the ckETH ledger's `transfer_fee` is raised via NNS governance to a value above `cketh_minimum_withdrawal_amount`, then:

- `validate_config` still passes (it checks against the stale hardcoded constant)
- Any `withdraw_eth` call with `amount >= minimum_withdrawal_amount` but `amount < new_transfer_fee` reaches `burn_from`, which returns `LedgerBurnError::AmountTooLow`
- The conversion to `WithdrawalError` panics, trapping the canister message

---

### Impact Explanation

Every `withdraw_eth` call in the affected amount range traps. Since `ic_cdk::trap` (triggered by `panic!` in canister context) aborts the message execution, the caller receives a reject response with no useful error. The core ckETH withdrawal functionality is broken for all users until the minter is upgraded with a corrected `minimum_withdrawal_amount` or the panic is replaced with a graceful return. The `WithdrawalError` type already has an `AmountTooLow` variant that is never populated by this code path, confirming the omission.

---

### Likelihood Explanation

The NNS can raise the ckETH ledger's `transfer_fee` via a standard governance proposal — the same mechanism used to set the fee originally (proposal [126309](https://dashboard.internetcomputer.org/proposal/126309)) and to reduce `minimum_withdrawal_amount` (proposal [139665](https://dashboard.internetcomputer.org/proposal/139665)). A fee adjustment is a routine governance action (e.g., in response to ETH price changes). The minter's `validate_config` would not catch the mismatch because it compares against a hardcoded constant, not the live ledger fee. Once the ledger fee exceeds `minimum_withdrawal_amount`, every `withdraw_eth` call in the affected range panics until the minter is upgraded.

---

### Recommendation

1. **Replace the panic with a graceful return** in `From<LedgerBurnError> for WithdrawalError`:

```rust
LedgerBurnError::AmountTooLow { minimum_burn_amount, .. } => {
    Self::AmountTooLow { min_withdrawal_amount: minimum_burn_amount }
}
```

2. **Remove the hardcoded fee from `validate_config`** or replace it with a live query to the ledger's `icrc1_fee` endpoint during minter initialization and upgrade, so the guard reflects the actual constraint.

3. Align `withdraw_eth` error handling with `withdraw_erc20`, which already propagates `AmountTooLow` as a structured error rather than panicking.

---

### Proof of Concept

1. NNS passes a proposal to raise the ckETH ledger `transfer_fee` from `2_000_000_000_000` wei to, e.g., `10_000_000_000_000_000` wei (a plausible adjustment if ETH price drops 5000×).
2. The ckETH minter is **not** upgraded; `cketh_minimum_withdrawal_amount` remains at `5_000_000_000_000_000` wei.
3. `validate_config` is called on the next minter upgrade — it checks `5_000_000_000_000_000 >= 2_000_000_000_000` (hardcoded), which passes.
4. A user calls `withdraw_eth` with `amount = 5_000_000_000_000_000` wei (exactly `minimum_withdrawal_amount`).
5. `withdraw_eth` passes the `amount < minimum_withdrawal_amount` guard (line 292).
6. `burn_from` calls `transfer_from` on the ledger; the ledger returns `BadBurn { min_burn_amount: 10_000_000_000_000_000 }` because `5_000_000_000_000_000 < 10_000_000_000_000_000`.
7. `burn_from` converts this to `LedgerBurnError::AmountTooLow`.
8. `withdraw_eth` calls `WithdrawalError::from(e)`, which hits the `panic!` branch.
9. The canister message is trapped; the user receives a reject with no actionable error.
10. All subsequent `withdraw_eth` calls in this range trap identically until the minter is upgraded.
