### Title
Unchecked Integer Overflow in SNS Governance Reward Distribution Corrupts Neuron Maturity - (File: rs/sns/governance/src/governance.rs)

### Summary
The SNS governance canister's `distribute_rewards` function uses bare, unchecked `u64` arithmetic when crediting voting rewards to neuron maturity fields. In Rust compiled to WebAssembly (release mode), integer overflow wraps silently. A neuron that accumulates maturity near `u64::MAX` will have its maturity silently zeroed on the next reward distribution, permanently destroying accumulated governance rewards without any error or revert.

### Finding Description

In `rs/sns/governance/src/governance.rs`, the reward distribution loop credits each voting neuron with its proportional share of the rewards purse using bare `+` and `+=` operators on `u64` fields:

```rust
if neuron.auto_stake_maturity.unwrap_or(false) {
    neuron.staked_maturity_e8s_equivalent = Some(
        neuron.staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s,
    );
} else {
    neuron.maturity_e8s_equivalent += neuron_reward_e8s;
}
distributed_e8s_equivalent += neuron_reward_e8s;
``` [1](#0-0) 

All three additions — to `staked_maturity_e8s_equivalent`, to `maturity_e8s_equivalent`, and to the local `distributed_e8s_equivalent` counter — are unchecked. The rest of the codebase consistently uses `saturating_add` or `checked_add` for maturity fields (e.g., in `RewardsDistribution::continue_processing` for NNS, and in `add_staked_maturity`): [2](#0-1) [3](#0-2) 

The SNS reward path is the only place that omits this protection.

The `neuron_reward_e8s` value is computed as a `Decimal` fraction of the rewards purse and then converted to `u64`: [4](#0-3) 

The `rewards_purse_e8s` itself is a `Decimal` derived from the SNS token supply. Because both `maturity_e8s_equivalent` and `staked_maturity_e8s_equivalent` are `u64` fields in the protobuf-defined `Neuron` struct: [5](#0-4) 

and because WebAssembly integer arithmetic wraps on overflow (Rust release mode does not trap on `u64` overflow in Wasm), a neuron whose accumulated maturity is near `u64::MAX` will silently wrap to a small value when the next reward is added.

The `debug_assert!` guard at line 6004 only fires in debug builds and is a no-op in production: [6](#0-5) 

### Impact Explanation

**Ledger conservation bug / governance accounting bug.** A neuron's `maturity_e8s_equivalent` or `staked_maturity_e8s_equivalent` silently wraps to near-zero. The neuron owner permanently loses all accumulated maturity — the SNS governance token equivalent of earned voting rewards — without any error, log, or revert. Because maturity is the precursor to minting new SNS tokens (via `disburse_maturity` or `merge_maturity`), this constitutes a direct loss of funds for the neuron holder. The SNS token supply invariant is also violated: maturity that was earned but never disbursed disappears from the accounting without a corresponding burn.

### Likelihood Explanation

**Low.** Exploitation requires a neuron to accumulate `maturity_e8s_equivalent` approaching `u64::MAX` (≈ 1.84 × 10¹⁹ e8s). This is bounded by the total SNS token supply, which in practice is far below `u64::MAX`. However:

1. The condition is not impossible — an SNS with a large initial supply and a dominant neuron that never disburses maturity could approach this over years of operation.
2. The bug is structural: the missing `saturating_add` is a one-character fix, and the inconsistency with every other maturity-crediting path in the codebase (NNS `RewardsDistribution`, `add_staked_maturity`, `stake_maturity_of_neuron`) confirms this is an oversight rather than an intentional design choice.
3. No attacker action is required to trigger it — the overflow occurs automatically during the governance canister's periodic reward distribution heartbeat, reachable by any neuron holder who votes.

### Recommendation

Replace the three unchecked additions with `saturating_add`, consistent with the rest of the codebase:

```rust
if neuron.auto_stake_maturity.unwrap_or(false) {
    neuron.staked_maturity_e8s_equivalent = Some(
        neuron.staked_maturity_e8s_equivalent
            .unwrap_or(0)
            .saturating_add(neuron_reward_e8s),
    );
} else {
    neuron.maturity_e8s_equivalent =
        neuron.maturity_e8s_equivalent.saturating_add(neuron_reward_e8s);
}
distributed_e8s_equivalent =
    distributed_e8s_equivalent.saturating_add(neuron_reward_e8s);
```

Alternatively, use `checked_add` and log an error (without panicking) if overflow would occur, mirroring the pattern in `rs/nns/governance/src/reward/distribution.rs`.

### Proof of Concept

1. Deploy an SNS with a large token supply (e.g., `u64::MAX / 2` tokens).
2. Create a single neuron holding all tokens with `auto_stake_maturity = true`.
3. Vote on every proposal for a sufficient number of reward periods until `staked_maturity_e8s_equivalent` approaches `u64::MAX`.
4. On the next reward distribution, `staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s` overflows `u64::MAX` and wraps to a value near zero.
5. The neuron's staked maturity is silently destroyed; the governance canister records a `RewardEvent` with a `distributed_e8s_equivalent` that also wraps, masking the accounting error.
