### Title
Slot-pool exhaustion via public `canister_status` blocks VIP governance calls due to broken zero-check guard — (`rs/nns/handlers/root/impl/canister/canister.rs`, `rs/nervous_system/clients/src/management_canister_client.rs`)

### Summary

The `LimitedOutstandingCallsManagementCanisterClient` uses a single 167-slot `thread_local` pool shared across all management canister methods. The public `canister_status` endpoint on NNS root has no authorization check, allowing any unprivileged caller to exhaust all 167 slots with concurrent requests. Due to a logic bug in `try_borrow_slot`, the early-return guard (`if *available_slot_count == 0 { return Err(...) }`) fires before the VIP path (`used_slot_count = 0`) takes effect, meaning NNS Governance-initiated calls to `update_canister_settings`, `take_canister_snapshot`, `load_canister_snapshot`, and SNS-W's `change_canister_controllers` are also rejected with `SysTransient` during the attack window.

### Finding Description

`try_borrow_slot` computes `used_slot_count = 0` for VIP callers, but the pool-empty guard is evaluated unconditionally before the subtraction:

```rust
// management_canister_client.rs, try_borrow_slot
let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

self.available_slot_count.with_borrow_mut(|available_slot_count| {
    if *available_slot_count == 0 {          // ← fires for VIPs too
        return Err((SysTransient, ...));
    }
    *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
    Ok(())
})?;
``` [3](#0-2) 

An unprivileged caller sends 167 concurrent `canister_status` ingress messages. Each one enters the canister, consumes one slot, then awaits the management canister response. While all 167 are in-flight, `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` reaches 0. Any subsequent call — including from Governance (a VIP) — hits the `== 0` guard and returns `Err(SysTransient, ...)`. [6](#0-5) 

### Impact Explanation

The affected governance-routed operations are those that call `new_management_canister_client()`:

| Endpoint | Caller | Impact |
|---|---|---|
| `update_canister_settings` | NNS Governance | Blocked |
| `take_canister_snapshot` | NNS Governance | Blocked |
| `load_canister_snapshot` | NNS Governance | Blocked |
| `change_canister_controllers` | SNS-W | Blocked |

**Not affected** (bypass the slot limiter entirely): `change_nns_canister` → `change_canister`, and `stop_or_start_nns_canister` → `stop_canister::<CdkRuntime>` — these call the management canister directly without going through `new_management_canister_client()`. [4](#0-3) 

The claim in the question that canister upgrades (`change_nns_canister`) and `stop_canister` are blocked is **incorrect** — those paths are unaffected. The real impact is denial of settings updates and snapshot operations during the attack window.

### Likelihood Explanation

- Sending 167 concurrent ingress messages is feasible; the IC's per-canister queue limit is ~500.
- The management canister's `canister_status` response takes at least one consensus round, giving the attacker a window of several seconds per batch.
- The attacker must continuously refresh the batch (send new calls as old ones complete) to sustain the attack, which costs cycles but is not prohibitive.
- The attack is temporary and self-limiting; once the attacker stops, slots drain naturally.

### Recommendation

Fix `try_borrow_slot` to skip the zero-check entirely for VIP callers:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    if self.is_caller_vip {
        return Ok(SlotLoan { available_slot_count: self.available_slot_count, used_slot_count: 0 });
    }
    self.available_slot_count.with_borrow_mut(|available_slot_count| {
        if *available_slot_count == 0 {
            return Err((RejectCode::SysTransient as i32, "Unavailable. Maybe, try again later?".to_string()));
        }
        *available_slot_count -= 1;
        Ok(())
    })?;
    Ok(SlotLoan { available_slot_count: self.available_slot_count, used_slot_count: 1 })
}
```

Additionally, consider giving `canister_status` its own separate slot pool, isolated from lifecycle-management operations.

### Proof of Concept

State-machine test outline:
1. Set up NNS root canister.
2. Send 167 concurrent `canister_status` calls from an unprivileged principal, targeting a canister for which NNS root is a controller (so the management canister call succeeds and stays in-flight for at least one round).
3. Before those calls complete, have Governance call `update_canister_settings` on NNS root.
4. Assert the `update_canister_settings` call returns `Err` with `SysTransient`.
5. Wait for the 167 `canister_status` calls to drain, then retry `update_canister_settings` and assert it succeeds.
