### Title
Hardcoded `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` Causes Permanent ckETH Loss on Out-of-Gas ERC-20 Withdrawals — (`File: rs/ethereum/cketh/minter/src/withdraw.rs`)

---

### Summary

The ckETH minter canister uses a single compile-time constant `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000` for every ckERC20 withdrawal transaction, regardless of which ERC-20 token is being withdrawn. There is no per-token gas limit, no admin endpoint to update it, and no field in `UpgradeArg` to change it without a full canister upgrade. If any supported ckERC20 token's transfer requires more than 65,000 gas (e.g., after a proxy contract upgrade, or when a new token with complex transfer logic is added), the Ethereum transaction will fail with out-of-gas. When this happens, the ckERC20 principal amount is reimbursed, but the ckETH gas fee burned upfront is **permanently lost** — the protocol explicitly does not reimburse transaction fees.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/withdraw.rs`, two gas limits are defined as Rust compile-time constants:

```rust
pub const CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(21_000);
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);
``` [1](#0-0) 

The function `estimate_gas_limit` dispatches uniformly to `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` for every ckERC20 withdrawal, with no per-token differentiation:

```rust
pub fn estimate_gas_limit(withdrawal_request: &WithdrawalRequest) -> GasAmount {
    match withdrawal_request {
        WithdrawalRequest::CkEth(_) => CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
        WithdrawalRequest::CkErc20(_) => CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT,
    }
}
``` [2](#0-1) 

This constant is used in `eip_1559_transaction_price` (the public query endpoint users call to estimate fees before withdrawing) and in the actual transaction construction path: [3](#0-2) 

The `CkErc20Token` struct carries no `gas_limit` field: [4](#0-3) 

The `UpgradeArg` type (the only mechanism to change minter parameters without a full Wasm upgrade) has no field for gas limits: [5](#0-4) 

The documentation explicitly acknowledges the hardcoded limit and its assumption: *"The `gas_limit` for ckERC20 withdrawals is currently fixed to `65_000` and should be sufficient for standard ERC-20 contracts."* [6](#0-5) 

---

### Impact Explanation

The ckERC20 withdrawal flow burns ckETH for gas fees **before** the Ethereum transaction is submitted. If the Ethereum transaction fails (e.g., out-of-gas because the token requires more than 65,000 gas), the protocol reimburses the ckERC20 withdrawal amount but **does not reimburse the ckETH gas fee**:

> *"The minter retrieves the receipt of the finalized transaction and will reimburse the ckERC20 tokens in case the transaction failed. Overcharged transaction fees are not reimbursed."* [7](#0-6) 

This is confirmed by the reimbursement logic: on `TransactionStatus::Failure`, only the ckERC20 withdrawal amount is queued for reimbursement; the ckETH burn index is not refunded: [8](#0-7) 

Concretely:
1. User burns ckETH (e.g., ~0.004 ETH equivalent) for gas fees.
2. User burns ckERC20 for the withdrawal amount.
3. Minter submits an Ethereum transaction with `gas_limit = 65_000`.
4. If the ERC-20 `transfer()` call consumes more than 65,000 gas, the transaction reverts on-chain.
5. ckERC20 tokens are minted back to the user; ckETH gas fees are **permanently burned**.

---

### Likelihood Explanation

The 65,000 gas assumption holds for simple ERC-20 contracts today, but is fragile in several realistic scenarios:

- **Proxy contract upgrades**: USDC (the primary ckERC20 token) is a UUPS proxy. If Circle upgrades the implementation to add compliance hooks, blacklist checks, or fee logic, the gas cost of `transfer()` can exceed 65,000. This is entirely outside DFINITY's control.
- **New ckERC20 tokens**: Any ERC-20 token can be added via a governance proposal targeting the ledger suite orchestrator. Tokens with rebasing logic, transfer hooks (ERC-777 style), or fee-on-transfer mechanics routinely consume 80,000–150,000+ gas per transfer.
- **No per-token override**: Because `CkErc20Token` carries no `gas_limit` field and `UpgradeArg` has no gas limit parameter, there is no way to set a higher limit for a specific token without a full canister Wasm upgrade — and even then, the single constant applies to all tokens simultaneously.

---

### Recommendation

1. Add an optional `gas_limit` field to `CkErc20Token` (and to `AddCkErc20Token` / `UpgradeArg`) so that per-token gas limits can be set at registration time and updated via upgrade without changing the Wasm binary.
2. Fall back to `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` when no per-token override is present, preserving backward compatibility.
3. Consider adding a governance-accessible setter endpoint (restricted to the NNS or ledger suite orchestrator) to update the gas limit for a specific token without a full canister upgrade.

---

### Proof of Concept

**Step 1 — Token with high gas cost is added.**
A governance proposal adds a new ckERC20 token whose ERC-20 `transfer()` function consumes 90,000 gas (e.g., a token with a compliance hook). This is a valid governance action via `add_ckerc20_token`.

**Step 2 — User initiates withdrawal.**
User calls `eip_1559_transaction_price` with the new token's ledger ID. The minter returns a price estimate based on `gas_limit = 65_000`: [3](#0-2) 

User approves the minter for the quoted ckETH fee and calls `withdraw_erc20`. The minter burns the ckETH gas fee and the ckERC20 withdrawal amount.

**Step 3 — Ethereum transaction fails.**
The minter constructs and submits an EIP-1559 transaction with `gas_limit = 65_000`: [9](#0-8) 

The Ethereum node executes the transaction; the ERC-20 `transfer()` call consumes 90,000 gas and reverts with out-of-gas at the 65,000 limit. The transaction receipt has `status = Failure`.

**Step 4 — Asymmetric reimbursement.**
The minter processes the failed receipt. The ckERC20 withdrawal amount is reimbursed. The ckETH gas fee is not: [8](#0-7) 

The user permanently loses the ckETH gas fee. All future withdrawal attempts for this token will fail identically until a canister Wasm upgrade changes the single shared constant — which also affects all other ckERC20 tokens simultaneously.
