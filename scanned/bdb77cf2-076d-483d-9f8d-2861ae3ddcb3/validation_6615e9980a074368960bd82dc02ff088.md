### Title
SNS `neuron_minimum_dissolve_delay_to_vote_seconds` Lacks Minimum Floor Enforcement, Allowing Governance Time-Lock Bypass - (File: rs/sns/init/src/lib.rs, rs/sns/governance/src/types.rs)

---

### Summary

The SNS governance system's validation of `neuron_minimum_dissolve_delay_to_vote_seconds` enforces only an upper bound (must not exceed `max_dissolve_delay_seconds`) but imposes no lower-bound floor. An SNS can therefore be initialized with this parameter set to `0`, eliminating the dissolve-delay time-lock that is the IC's analog of the ERC-20 `timeLockPeriod`. Once deployed, any token holder — including one who just acquired tokens in the decentralization swap — can immediately vote and submit proposals with zero lock-up, bypassing the governance security model.

---

### Finding Description

The external report describes a `timeLockPeriod` that defaults to `0` and is never configured post-deployment, allowing a passthrough that skips the intended waiting period before privileged operations take effect. The direct IC analog is `neuron_minimum_dissolve_delay_to_vote_seconds` in SNS governance: it is the parameter that enforces how long tokens must be locked before a neuron may vote or propose. If it is `0`, any neuron — regardless of dissolve delay — can participate in governance immediately.

**Root cause 1 — SNS init validation (`rs/sns/init/src/lib.rs`):**

```rust
fn validate_neuron_minimum_dissolve_delay_to_vote_seconds(&self) -> Result<(), String> {
    let max_dissolve_delay_seconds = *NervousSystemParameters::with_default_values()
        .max_dissolve_delay_seconds
        .as_ref()
        .unwrap();

    let neuron_minimum_dissolve_delay_to_vote_seconds = self
        .neuron_minimum_dissolve_delay_to_vote_seconds
        .ok_or_else(|| { ... })?;

    if neuron_minimum_dissolve_delay_to_vote_seconds > max_dissolve_delay_seconds {
        return Err(...);
    }
    Ok(())   // ← value of 0 passes here
}
```

Only the upper bound is checked. A value of `0` is accepted without error. [1](#0-0) 

**Root cause 2 — SNS governance `NervousSystemParameters` validation (`rs/sns/governance/src/types.rs`):**

```rust
fn validate_neuron_minimum_dissolve_delay_to_vote_seconds(&self) -> Result<(), String> {
    let max_dissolve_delay_seconds = self.validate_max_dissolve_delay_seconds()?;
    let neuron_minimum_dissolve_delay_to_vote_seconds = self
        .neuron_minimum_dissolve_delay_to_vote_seconds
        .ok_or_else(|| { ... })?;

    if neuron_minimum_dissolve_delay_to_vote_seconds > max_dissolve_delay_seconds {
        Err(...)
    } else {
        Ok(())   // ← value of 0 passes here
    }
}
```

Same pattern: only the upper bound is enforced. [2](#0-1) 

By contrast, the NNS governance enforces a strict range `(14 * ONE_DAY_SECONDS)..=(6 * ONE_MONTH_SECONDS)` for its equivalent parameter, explicitly rejecting values outside that window. [3](#0-2) 

The SNS `NervousSystemParameters` default is `6 * ONE_MONTH_SECONDS`, but nothing prevents an SNS deployer from overriding it to `0` — and the validation will accept it. [4](#0-3) 

The governance enforcement path that uses this parameter to gate voting eligibility is:

```rust
if neuron.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_seconds {
    return;  // neuron gets no ballot
}
```

When `min_dissolve_delay_seconds == 0`, this guard is never triggered and every neuron — including dissolved ones — receives a ballot. [5](#0-4) 

---

### Impact Explanation

An SNS deployed with `neuron_minimum_dissolve_delay_to_vote_seconds = 0` has no governance time-lock whatsoever. Any principal who holds SNS tokens — even tokens acquired seconds ago in the decentralization swap — can immediately:

- Submit `UpgradeSnsControlledCanister` proposals to replace dapp canister code
- Submit `ManageNervousSystemParameters` proposals to further weaken governance guards
- Submit `TransferSnsTreasuryFunds` proposals to drain the SNS treasury

This enables flash-loan-style governance attacks: acquire tokens, vote, execute, exit — all within a single round of inter-canister calls. The SNS treasury, controlled dapp canisters, and governance parameters are all at risk.

---

### Likelihood Explanation

The SNS deployment flow via `SnsInitPayload` accepts any non-`None` value for `neuron_minimum_dissolve_delay_to_vote_seconds` as long as it does not exceed `max_dissolve_delay_seconds`. A deployer who misreads the documentation, copies a test configuration (the test init params file sets `neuron_minimum_dissolve_delay_to_vote_seconds: 15778800` but nothing prevents a `0`), or deliberately wants "open" governance can set this to `0` and the system will accept it. Because the NNS proposal that creates an SNS does not validate this parameter against a floor, the misconfiguration reaches mainnet undetected.

---

### Recommendation

- **Short term:** Add a minimum floor check in both `rs/sns/init/src/lib.rs` (`validate_neuron_minimum_dissolve_delay_to_vote_seconds`) and `rs/sns/governance/src/types.rs` (`validate_neuron_minimum_dissolve_delay_to_vote_seconds`), mirroring the NNS pattern:
  ```rust
  const NEURON_MINIMUM_DISSOLVE_DELAY_TO_VOTE_SECONDS_FLOOR: u64 = ONE_DAY_SECONDS; // or similar
  if neuron_minimum_dissolve_delay_to_vote_seconds < FLOOR {
      return Err(...);
  }
  ```
- **Long term:** Require `neuron_minimum_dissolve_delay_to_vote_seconds > 0` as a hard invariant enforced at the NNS level before an SNS proposal is executed, analogous to how the NNS enforces its own `NEURON_MINIMUM_DISSOLVE_DELAY_TO_VOTE_SECONDS_BOUNDS`.

---

### Proof of Concept

1. Construct an `SnsInitPayload` with `neuron_minimum_dissolve_delay_to_vote_seconds: Some(0)`.
2. Call `validate_post_execution()` — it returns `Ok(())` because `validate_neuron_minimum_dissolve_delay_to_vote_seconds` only checks the upper bound. [6](#0-5) 
3. Deploy the SNS via NNS `CreateServiceNervousSystem` proposal.
4. Acquire any amount of SNS tokens (e.g., via the decentralization swap).
5. Stake a neuron with `dissolve_delay_seconds = 0`.
6. Immediately call `make_proposal` — the neuron passes the dissolve-delay eligibility check because `0 >= 0`.
7. With sufficient voting power, the proposal is adopted and executed in the same round, with no waiting period.
