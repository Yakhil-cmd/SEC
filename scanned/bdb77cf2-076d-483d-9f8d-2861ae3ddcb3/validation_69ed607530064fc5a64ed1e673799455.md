### Title
Stale Treasury Valuation at Execution Time Causes SNS `TransferSnsTreasuryFunds` 7-Day Limit to Be Inconsistently Enforced — (`rs/sns/governance/src/proposal.rs`)

---

### Summary

SNS governance enforces a 7-day treasury transfer limit via two checks: one at proposal submission and one at execution. The execution-time check (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`) computes `allowance_tokens` from a **stale valuation** frozen at submission time, while `spent_tokens` is computed from the **current** executed-proposal list. This mismatch — identical in structure to the ZKSync `_verifyWithdrawalLimit` bug — means the limit base and the accumulated spend are drawn from different points in time, producing two failure modes: (1) valid proposals are incorrectly rejected when the treasury grows between submission and execution, and (2) the limit can be silently exceeded when the token price rises significantly during the voting window.

---

### Finding Description

At **proposal submission time**, `treasury_valuation_if_proposal_amount_is_small_enough_or_err` fetches a live `Valuation` (containing `balance_tokens`, `icps_per_token`, `xdrs_per_icp`) and stores it in `action_auxiliary`: [1](#0-0) 

At **proposal execution time**, `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` reuses that frozen valuation to compute `allowance_tokens`: [2](#0-1) 

But `spent_tokens` is computed live from the current proposal list: [3](#0-2) 

The code itself acknowledges the inconsistency in a comment:

> *"The total calculated here could be different from what was calculated at proposal submission/creation time. A difference would result from the execution of (another) TransferSnsTreasuryFunds proposal between now and then."*

The limit formula in `ProposalsAmountTotalUpperBound::in_tokens` uses `balance_tokens` directly for the `Fraction` (medium treasury, 25%) branch: [4](#0-3) 

And uses `icps_per_token` × `xdrs_per_icp` for the `Xdr` (large treasury, 300,000 XDR cap) branch: [5](#0-4) 

Both values are stale.

---

### Impact Explanation

**Mode 1 — Valid proposal incorrectly rejected (direct ZKSync analog):**

Suppose the SNS treasury holds 100,000 ICP (medium, limit = 25,000 ICP). Two proposals are submitted simultaneously, each for 24,000 ICP; both pass the submission-time check (spent = 0, max = 25,000). Between submission and execution the treasury receives a large deposit (e.g., 1,000,000 ICP from a fundraise). Proposal A executes fine (allowance = 25,000, spent = 0). Proposal B then executes: `allowance_tokens` = 25,000 (stale, based on 100,000 ICP), `spent_tokens` = 24,000 (current), remainder = 1,000. The 24,000 ICP transfer is rejected. Yet based on the current treasury (≈1,076,000 ICP) the limit is ≈269,000 ICP and both proposals together (48,000 ICP) are well within it. The proposal fails with `PreconditionFailed` and must be resubmitted, disrupting governance.

**Mode 2 — Limit silently exceeded when token price rises:**

For a large treasury the cap is 300,000 XDR worth of tokens. If the SNS token trades at 2 XDR/token at submission time, the stale valuation allows 150,000 tokens. If the price rises to 20 XDR/token by execution time, the current limit would be 15,000 tokens. A proposal for 140,000 tokens passes the stale check (140,000 ≤ 150,000) but transfers 2,800,000 XDR of value — nearly 10× the intended cap — bypassing the safety mechanism entirely. [6](#0-5) 

---

### Likelihood Explanation

SNS governance voting windows are typically 4–7 days. During that window:

- Treasury deposits (from protocol revenue, token sales, or fundraising) are routine and can be large.
- Token prices routinely move 2–10× over a week for smaller SNS projects.

Mode 1 is triggered by any treasury deposit between submission and execution of concurrent proposals — a common operational pattern. Mode 2 is triggered by a price increase during the voting window, which is also common. Neither requires privileged access; any token holder can submit a `TransferSnsTreasuryFunds` proposal. [7](#0-6) 

---

### Recommendation

At execution time, re-fetch a fresh valuation (as is already done at submission time via `assess_treasury_balance`) instead of reusing the stale one stored in `action_auxiliary`. The execution-time guard should call `assess_treasury_balance` live and compute `allowance_tokens` from the current balance and price, then compare against the current `spent_tokens`. This mirrors the fix applied in the ZKSync case: record the base value at the start of the window and use it consistently, or re-evaluate it at enforcement time. [8](#0-7) 

---

### Proof of Concept

**Mode 1 (valid proposal rejected):**

1. SNS treasury: 100,000 ICP (medium treasury, 25% limit = 25,000 ICP).
2. Submit Proposal A: transfer 24,000 ICP. Submission check: spent=0, max=25,000. Passes. Stale valuation stored: `balance_tokens=100,000`.
3. Submit Proposal B: transfer 24,000 ICP. Same check. Passes. Stale valuation stored: `balance_tokens=100,000`.
4. Treasury receives 1,000,000 ICP deposit (now 1,100,000 ICP).
5. Proposal A executes: `allowance_tokens=25,000`, `spent_tokens=0`, remainder=25,000. 24,000 ≤ 25,000. ✓
6. Proposal B executes: `allowance_tokens=25,000` (stale), `spent_tokens=24,000` (current), remainder=1,000. 24,000 > 1,000. **Rejected with `PreconditionFailed`.**
7. Current treasury limit is ≈269,000 ICP; both proposals together (48,000 ICP) are valid. The rejection is incorrect.

**Mode 2 (limit bypassed):**

1. SNS treasury: 1,000,000 SNS tokens at 2 XDR/token = 2,000,000 XDR (large treasury). Current limit = 300,000 XDR / 2 = 150,000 tokens.
2. Submit Proposal: transfer 140,000 tokens. Submission check: spent=0, max=150,000. Passes. Stale valuation stored: `icps_per_token` and `xdrs_per_icp` reflecting 2 XDR/token.
3. Token price rises to 20 XDR/token during voting. Current limit = 300,000 XDR / 20 = 15,000 tokens.
4. Proposal executes: `allowance_tokens=150,000` (stale), `spent_tokens=0`, remainder=150,000. 140,000 ≤ 150,000. **Passes.**
5. 140,000 tokens transferred at 20 XDR/token = 2,800,000 XDR — 9.3× the intended 300,000 XDR cap. [9](#0-8) [10](#0-9)

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }

    Ok(())
}

    fn in_tokens(mut valuation: Valuation) -> Result<Decimal, ProposalsAmountTotalLimitError> {
        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))
                })?;

                // Calculate the inverse conversion rate.
                if xdrs_per_token == Decimal::from(0) {
                    // This is not reachable, because in this case, valuation.to_xdr() would return
                    // 0, and in that case, we would have taken the NoLimit branch.
                    return Err(ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
                }
                let tokens_per_xdr = xdrs_per_token.inv();

                max_xdr.checked_mul(tokens_per_xdr).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Max tokens could not be calculated with valuation: {valuation:?}",
                    ))
                })?
            }
        };

        Ok(result_tokens)
    }

pub(crate) async fn assess_treasury_balance(
    token: Token,
    sns_governance_canister_id: CanisterId,
    sns_ledger_canister_id: CanisterId,
    swap_canister_id: CanisterId,
) -> Result<Valuation, String> {
    let treasury_account = token.treasury_account(sns_governance_canister_id)?;
    let valuation = token
        .assess_balance(sns_ledger_canister_id, swap_canister_id, treasury_account)
        .await
        .map_err(|valuation_error| {
            format!("Unable to assess current treasury balance: {valuation_error:?}")
        })?;
    Ok(valuation)
}
