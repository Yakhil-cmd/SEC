### Title
Byzantine Block Proposer Injects Spurious `ConsensusResponse` via Unvalidated Bitcoin Callback IDs â€” (`rs/replicated_state/src/bitcoin.rs`, `rs/bitcoin/consensus/src/payload_builder.rs`)

### Summary

`validate_payload` (the `BatchPayloadBuilder` impl) performs no cross-check of `callback_id` values against pending requests in replicated state. Three of four arms in `push_response` also perform no such check. A single Byzantine block proposer â€” within the BFT fault-tolerance threshold â€” can therefore finalize a payload containing fabricated `callback_id` values and inject arbitrary `ConsensusResponse` entries into the `consensus_queue`.

### Finding Description

**Gap 1 â€” `validate_payload` performs no state lookup.**

### Impact Explanation

A finalized block containing a `SendTransactionResponse`, `GetSuccessorsReject`, or `SendTransactionReject` with a fabricated `callback_id` causes `push_response` to unconditionally enqueue a `ConsensusResponse` into `state.consensus_queue`. The execution scheduler drains this queue and dispatches responses by `callback_id`. Depending on execution behavior:

- If the fabricated ID matches a live canister callback (possible since `CallbackId` is a global monotonic counter observable by any node), the wrong canister receives a spurious response, corrupting its execution state.
- If the ID matches no live callback, execution may panic or silently drop it â€” the former halts the subnet deterministically across all honest nodes.

### Likelihood Explanation

The attacker is a **single Byzantine block proposer** â€” one compromised subnet node, which is explicitly within the BFT fault-tolerance bound (f < n/3). No threshold corruption, governance majority, or external infrastructure attack is required. The proposer can observe the current `CallbackId` counter from state and craft a targeted payload. All honest validators will accept the block because `validate_payload` performs no state cross-check.

### Recommendation

1. **In `validate_payload`**: After parsing, retrieve state at `proposal_context.validation_context.certified_height` and reject any `callback_id` not present in `bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`. Also use the already-computed `delivered_ids` to reject duplicates.
2. **In `push_response`**: Add the same `.get(&callback_id).ok_or(StateError::BitcoinNonMatchingResponse{â€¦})?` guard to the `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` arms, mirroring the existing guard in the `GetSuccessorsResponse` arm.
