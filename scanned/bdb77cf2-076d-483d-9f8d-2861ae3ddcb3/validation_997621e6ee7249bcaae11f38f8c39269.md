### Title
Weak Entropy Seed for Canister-Wide PRNG Enables Prediction of Confidential Rate-Limit Rule IDs - (File: rs/boundary_node/rate_limits/canister/random.rs)

### Summary
The rate-limits canister registers a custom `getrandom` implementation seeded almost entirely with the constant byte `0x2a` (42), with only 8 bytes replaced by the publicly observable IC canister time. This makes the entire canister-wide PRNG output predictable, allowing an unprivileged caller to enumerate generated `RuleId` UUIDs and query metadata of undisclosed (confidential) rate-limit rules via the public `get_rule_by_id` query endpoint.

### Finding Description

In `rs/boundary_node/rate_limits/canister/random.rs`, the global `getrandom` implementation for the rate-limits canister WASM is initialized as follows:

```rust
thread_local! {
  static RNG: RefCell<ChaCha20Rng> = {
    let mut seed = [42; 32];
    seed[..8].copy_from_slice(&time().to_le_bytes());
    RefCell::new(ChaCha20Rng::from_seed(seed))
  };
}
``` [1](#0-0) 

The 32-byte seed is constructed as:
- Bytes `[0..8]`: `ic_cdk::api::time()` — the IC consensus time in nanoseconds (publicly observable, deterministic per block)
- Bytes `[8..32]`: constant `0x2a` (the literal `42`)

This RNG is then registered as the canister-wide `getrandom` provider via `getrandom::register_custom_getrandom!`: [2](#0-1) 

Any call to `Uuid::new_v4()` within the canister (e.g., in `add_config.rs` when generating `RuleId` values for newly submitted rate-limit rules) draws from this RNG. The `InputRule` type does not include a `rule_id` field, confirming the canister generates `RuleId` UUIDs internally: [3](#0-2) 

The `get_rule_by_id` query method is publicly accessible (not gated by `inspect_message`) and returns metadata including `incident_id`, `added_in_version`, and `removed_in_version` even for undisclosed rules — only `rule_raw` and `description` are redacted: [4](#0-3) 

The confidentiality formatting hides `rule_raw` and `description` but always returns `rule_id`, `incident_id`, and version metadata for unauthorized callers: [5](#0-4) 

### Impact Explanation

An unprivileged attacker who knows the approximate canister initialization time (which is public on-chain) can reconstruct the exact 32-byte seed (8 bytes of known timestamp + 24 bytes of constant `0x2a`), replay the ChaCha20 stream, and predict all `RuleId` UUIDs generated since canister initialization. By calling `get_rule_by_id` with predicted UUIDs, the attacker can:

1. Confirm the existence of undisclosed (confidential) rate-limit rules before they are publicly disclosed.
2. Retrieve the `incident_id`, `added_in_version`, and `removed_in_version` of confidential rules — metadata that is intended to be hidden until the authorized principal calls `disclose_rules`.

This breaks the confidentiality model of the rate-limits canister, which is designed to hide active incident-response rules from the public until explicitly disclosed.

### Likelihood Explanation

The IC canister time (`ic_cdk::api::time()`) is the consensus timestamp of the block in which the canister was initialized. This value is recorded on-chain and is trivially observable by any party. The remaining 24 bytes of the seed are the constant `0x2a`. The attacker needs only to:
1. Observe the canister's initialization block time (public).
2. Reconstruct the seed deterministically.
3. Run ChaCha20 locally to generate the UUID sequence.
4. Query `get_rule_by_id` for each predicted UUID.

No privileged access, social engineering, or threshold attack is required. The attack is fully executable by any unprivileged ingress caller.

### Recommendation

Replace the time-seeded initialization with a call to `ic_cdk::api::management_canister::main::raw_rand()` to obtain cryptographically secure randomness from the IC management canister. Since `raw_rand` is asynchronous, the canister should defer UUID generation until after the first successful seeding, or use a lazy-initialization pattern that seeds the RNG from `raw_rand` on the first `add_config` call. Alternatively, accept `RuleId` values from the authorized caller (removing the need for canister-side UUID generation), or derive rule IDs deterministically from a hash of the rule content plus a canister-held secret seeded from `raw_rand`.

### Proof of Concept

```
1. Observe the IC block time T_ns at which the rate-limits canister was initialized
   (available from any IC block explorer or replica state).

2. Reconstruct the seed:
   seed = T_ns.to_le_bytes() ++ [0x2a; 24]

3. Initialize ChaCha20Rng with this seed and generate UUID v4 values
   (each UUID consumes 16 bytes of RNG output).

4. For each predicted UUID `u`:
   Call: get_rule_by_id(u)  [public query, no authentication required]

5. For any hit, the response reveals:
   - incident_id (UUID of the associated security incident)
   - added_in_version / removed_in_version
   even if the rule has never been disclosed via disclose_rules().
```

/// Retrieves a specific rate-limit rule by its ID, applying confidentiality formatting, based on caller's access level and rule's confidentiality status
#[query]
fn get_rule_by_id(rule_id: RuleId) -> GetRuleByIdResponse {
    let caller_id = ic_cdk::api::caller();
    let response = with_canister_state(|state| {
        let access_resolver = AccessLevelResolver::new(caller_id, state.clone());
        let formatter = RuleConfidentialityFormatter;
        let getter = RuleGetter::new(state, formatter, access_resolver);
        getter.get(&rule_id)
    })?;
    Ok(response)
}
