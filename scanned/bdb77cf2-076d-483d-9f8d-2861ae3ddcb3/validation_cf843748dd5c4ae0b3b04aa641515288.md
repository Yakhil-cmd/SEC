### Title
Unverified Bitcoin Checker Canister Identity Enables Taint-Check Bypass in ckBTC Minter - (File: rs/bitcoin/ckbtc/minter/src/lib.rs)

### Summary
The ckBTC minter's security model for Bitcoin address/UTXO taint checking relies entirely on the authenticity of the canister stored in `btc_checker_principal`. This principal is accepted as a plain `CanisterId` parameter at init/upgrade time with no on-chain verification that the supplied canister is a genuine, NNS-controlled Bitcoin checker. A malicious or compromised deployer can point the minter at an attacker-controlled canister that always returns `Passed`, bypassing the entire AML/sanctions screening gate and allowing tainted BTC to be minted into ckBTC.

### Finding Description
The ckBTC minter stores a `btc_checker_principal: Option<CanisterId>` in its state. This value is set directly from the `InitArgs` or `UpgradeArgs` supplied at installation/upgrade time, with no verification that the canister at that address is a legitimate Bitcoin checker.

At init time, `btc_checker_principal` is accepted verbatim:

```rust
// rs/bitcoin/ckbtc/minter/src/lifecycle/init.rs
pub struct InitArgs {
    ...
    pub btc_checker_principal: Option<CanisterId>,
    ...
}
```

At upgrade time, it is similarly accepted without validation:

```rust
// rs/bitcoin/ckbtc/minter/src/state.rs
if let Some(btc_checker_principal) = btc_checker_principal {
    self.btc_checker_principal = Some(btc_checker_principal);
}
```

The only check performed is that the field is `Some(...)` (not `None`):

```rust
// rs/bitcoin/ckbtc/minter/src/state.rs
pub fn validate_config(&self) {
    ...
    if self.btc_checker_principal.is_none() {
        ic_cdk::trap("Bitcoin checker principal is not set");
    }
}
```

When `update_balance` or `retrieve_btc` is called, the minter blindly calls `check_address` or `check_transaction` on whatever canister is stored in `btc_checker_principal`:

```rust
// rs/bitcoin/ckbtc/minter/src/lib.rs
ic_cdk::call::Call::bounded_wait(btc_checker_principal, "check_address")
    .with_arg(CheckAddressArgs { address: address.clone() })
    .await
    ...
    .map(|res: CheckAddressResponse| match res {
        CheckAddressResponse::Failed => BtcAddressCheckStatus::Tainted,
        CheckAddressResponse::Passed => BtcAddressCheckStatus::Clean,
    })
```

An attacker-controlled canister can implement `check_address` and `check_transaction` to unconditionally return `Passed`/`CheckTransactionResponse::Passed`, making every address and UTXO appear clean regardless of its actual sanctions status.

### Impact Explanation
The Bitcoin checker is the sole AML/sanctions gate for the ckBTC minter. If it is bypassed:

- **`update_balance`**: Tainted UTXOs (from sanctioned addresses) are accepted and minted as ckBTC, violating the protocol's compliance guarantees and potentially exposing the NNS/ckBTC system to regulatory and reputational risk.
- **`retrieve_btc`**: Withdrawals to sanctioned Bitcoin addresses are permitted, again bypassing the OFAC SDN screening.

The impact is a **chain-fusion mint/burn security bypass**: tainted BTC enters the ckBTC supply and tainted BTC addresses receive withdrawals, undermining the integrity of the entire ckBTC bridge.

### Likelihood Explanation
The `btc_checker_principal` is set at install/upgrade time by whoever controls the minter canister. On mainnet, the ckBTC minter is controlled by the NNS root canister, so changing `btc_checker_principal` requires an NNS proposal — this significantly limits the attack surface for the production deployment. However:

1. Any SNS or third-party deploying a ckBTC-style minter (e.g., ckDOGE, which shares the same codebase pattern) with a misconfigured or attacker-controlled `btc_checker_principal` is fully vulnerable.
2. The code contains no allowlist or on-chain verification of the checker canister's identity, controllers, or code hash — analogous exactly to the reported Solidity pattern where `deployer()` is called on a caller-supplied contract.
3. The comment in the code itself acknowledges the dependency: `"BUG: upgrade procedure must ensure that the Bitcoin checker principal is set"` — but there is no enforcement that it is set to a *legitimate* checker.

### Recommendation
1. **Verify the checker canister's controllers at init/upgrade time**: After setting `btc_checker_principal`, call `canister_info` on the management canister to assert that the checker canister is controlled exclusively by the NNS root (or another trusted principal), rejecting the configuration if not.
2. **Verify the checker canister's code hash**: Store the expected Wasm hash of the Bitcoin checker and verify it matches the installed module hash via `canister_info` before accepting the principal.
3. **Alternatively, use a hardcoded allowlist**: Maintain a hardcoded set of known-good Bitcoin checker canister IDs (analogous to the Spearbit recommendation to use an allowlist for oracles), rejecting any `btc_checker_principal` not in the list.

### Proof of Concept

**Attacker-controlled checker canister (pseudocode):**
```rust
#[query]
fn check_address(_args: CheckAddressArgs) -> CheckAddressResponse {
    CheckAddressResponse::Passed  // always passes, regardless of SDN status
}

#[update]
fn check_transaction(_args: CheckTransactionArgs) -> CheckTransactionResponse {
    CheckTransactionResponse::Passed
}
```

**Attack flow:**
1. Deploy the above malicious canister; obtain its `CanisterId`.
2. At minter init (or via an upgrade proposal if the attacker controls the minter), supply the malicious canister's ID as `btc_checker_principal`.
3. The minter's `validate_config` passes because `btc_checker_principal` is `Some(...)`.
4. Call `update_balance` with a UTXO from a sanctioned address — the minter calls `check_transaction` on the malicious canister, receives `Passed`, and mints ckBTC.
5. Call `retrieve_btc` with a sanctioned destination address — the minter calls `check_address`, receives `Passed`, and processes the withdrawal.
