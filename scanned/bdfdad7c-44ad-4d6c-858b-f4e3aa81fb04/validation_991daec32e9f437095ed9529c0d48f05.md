### Title
Insufficient Slippage Protection in SNS Treasury Manager Deposit/Withdraw API - (File: rs/sns/treasury_manager/treasury_manager.did)

---

### Summary

The SNS Treasury Manager API (`treasury_manager.did`) and its governance integration (`rs/sns/governance/src/extensions.rs`) lack any slippage protection parameters for `deposit` and `withdraw` operations. The `DepositRequest` type only carries token allowance amounts; there is no field for a minimum LP token output, minimum price ratio, or any other bound on the exchange rate at execution time. The codebase itself acknowledges this as a "Known Security Risk" in the DID file. Because SNS governance proposals are subject to a voting delay before execution, the DEX price at execution time can differ materially from the price at proposal creation time, causing the SNS DAO treasury to receive fewer LP tokens than voters approved.

---

### Finding Description

The `DepositRequest` type in `treasury_manager.did` is defined as:

```candid
type DepositRequest = record {
  allowances : vec Allowance;
};
```

It contains only the token amounts to deposit — no `min_lp_tokens_out`, no `min_price_ratio`, no `max_price_ratio`, and no deadline. The same absence applies to `WithdrawRequest`, which carries only optional withdrawal accounts.

The DID file itself explicitly flags this:

```
// Known Security Risks:
// ====================
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
```

In `rs/sns/governance/src/extensions.rs`, the `ValidatedDepositOperationArg` struct only stores `treasury_allocation_sns_e8s`, `treasury_allocation_icp_e8s`, and the raw `original` Precise blob — no slippage bounds:

```rust
pub struct ValidatedDepositOperationArg {
    pub treasury_allocation_sns_e8s: u64,
    pub treasury_allocation_icp_e8s: u64,
    pub original: Precise,
}
```

The `validate_deposit_operation_impl` function checks only that the requested amounts do not exceed 50% of the current treasury balance. It performs no check on the current DEX price ratio, no minimum LP token output, and no deadline:

```rust
if sns_requested > sns_balance.checked_div(2).unwrap() { ... }
if icp_requested > icp_balance.checked_div(2).unwrap() { ... }
```

`execute_treasury_manager_deposit` then approves the treasury manager and calls `deposit` on the extension canister with no slippage guard:

```rust
governance.approve_treasury_manager(...).await?;
governance.env.call_canister(extension_canister_id, "deposit", arg_blob).await ...
```

The `construct_treasury_manager_deposit_payload` function builds a `DepositRequest` containing only the allowances — no slippage fields are added at any point in the call chain.

---

### Impact Explanation

An SNS governance proposal to deposit treasury funds into a DEX liquidity pool is voted on over a period of days. Between proposal adoption and execution, any market participant can move the DEX price (e.g., by making large trades on the pool). When the proposal executes, the treasury manager deposits the approved token amounts at the prevailing (manipulated) price ratio, receiving far fewer LP tokens than the DAO voters expected. The SNS treasury suffers a direct, permanent financial loss proportional to the price deviation. Because the `WithdrawRequest` also carries no minimum token-out bounds, the same risk applies on withdrawal: the DAO may receive fewer underlying tokens than expected when exiting a liquidity position.

---

### Likelihood Explanation

The IC's asynchronous governance model (multi-day voting periods) makes this structurally more severe than in synchronous EVM contexts. Any unprivileged canister caller or market participant who can trade on the same DEX pool can front-run the proposal execution. The DEX canister itself (if malicious or compromised) could also manipulate the price immediately before the deposit call. The codebase's own "Known Security Risks" comment confirms the developers are aware of the gap. The likelihood is **high** whenever a Treasury Manager extension is registered and a deposit proposal is executed against a DEX pool with meaningful liquidity.

---

### Recommendation

1. Add slippage protection fields to `DepositRequest` and `WithdrawRequest` in `treasury_manager.did`:
   - `min_lp_tokens_out : opt nat` — minimum LP tokens the treasury must receive for the deposit to succeed.
   - `max_price_ratio : opt record { numerator: nat; denominator: nat }` — maximum acceptable token price ratio at execution time.
   - `min_token_a_out` / `min_token_b_out` for `WithdrawRequest`.
2. Add corresponding fields to `ValidatedDepositOperationArg` in `extensions.rs` and enforce them in `validate_deposit_operation_impl` and `execute_treasury_manager_deposit`.
3. Require Treasury Manager implementations to pass these bounds through to the underlying DEX call and revert if the DEX cannot satisfy them.
4. Consider adding a proposal execution deadline (e.g., `execute_before_timestamp_seconds`) so that stale proposals cannot be executed after the market has moved significantly.

---

### Proof of Concept

1. An SNS DAO submits a `RegisterExtension` + `ExecuteExtensionOperation` (deposit) proposal specifying `treasury_allocation_sns_e8s = X` and `treasury_allocation_icp_e8s = Y` at a time when the DEX pool price ratio is `X:Y`.
2. The proposal enters the voting period (e.g., 4 days).
3. An attacker (any unprivileged principal) executes large trades on the DEX pool, shifting the price ratio to `X:2Y` (doubling the ICP price of SNS tokens).
4. The proposal passes and `execute_treasury_manager_deposit` is called. It calls `governance.approve_treasury_manager(extension_canister_id, X, Y)` and then `deposit` with `allowances = [{SNS, X}, {ICP, Y}]`.
5. The Treasury Manager deposits `X` SNS and `Y` ICP into the pool at the manipulated ratio. Because the pool now prices SNS at twice the ICP value, the DAO receives LP tokens worth only ~50% of the intended value.
6. No check in `validate_deposit_operation_impl`, `execute_treasury_manager_deposit`, or `construct_treasury_manager_deposit_payload` rejects or reverts this outcome.

```rust
async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
```

```rust
/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
```
