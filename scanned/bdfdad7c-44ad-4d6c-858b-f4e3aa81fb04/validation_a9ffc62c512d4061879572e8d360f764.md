### Title
Missing Minimum Output Amount Check in SNS Treasury Manager Deposit Operation Enables Partial-Fill Value Loss - (File: rs/sns/governance/src/extensions.rs)

### Summary

The SNS Treasury Manager `deposit` operation framework lacks a `min_lp_tokens_out` (minimum output amount) parameter in the `DepositRequest` type and the `execute_treasury_manager_deposit` function does not verify the minimum LP tokens received after calling the DEX. This is the direct IC analog of the `amountOutMinimum = 0` partial-trading vulnerability: the SNS governance canister approves the full token allowance and calls `deposit`, but never checks whether the LP tokens returned meet any minimum threshold. The `treasury_manager.did` interface itself explicitly labels this a "Known Security Risk."

### Finding Description

The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` executes in two steps:

1. **Approve** the treasury manager canister to spend `treasury_allocation_sns_e8s` SNS tokens and `treasury_allocation_icp_e8s` ICP from the SNS treasury via ICRC-2 allowances.
2. **Call** `deposit` on the treasury manager canister and log the returned `Balances` struct. [1](#0-0) 

The `DepositRequest` type forwarded to the treasury manager contains only `allowances` (input amounts) and no `min_lp_tokens_out` field: [2](#0-1) 

The proposal validation function `validate_deposit_operation_impl` only checks structural validity and that the requested amount does not exceed 50% of the current treasury balance. It performs no minimum output check: [3](#0-2) 

After the `deposit` call returns, `execute_treasury_manager_deposit` only logs the response and returns `Ok(())` — it never inspects the `external_custodian` balance in the returned `Balances` struct to verify that a minimum number of LP tokens was received: [4](#0-3) 

The `treasury_manager.did` interface itself explicitly acknowledges this gap: [5](#0-4) 

The `validate_and_render_register_extension` proposal renderer also warns voters about this risk but does not enforce any on-chain guard: [6](#0-5) 

### Impact Explanation

When an SNS DAO executes a `deposit` proposal, the governance canister grants the full ICRC-2 allowance for both SNS tokens and ICP to the treasury manager, then calls `deposit`. If the underlying DEX pool price has moved adversely between proposal creation and execution — or if the DEX itself does not implement slippage protection — the treasury manager may deposit the full approved token amounts but receive far fewer LP tokens than the DAO expected at proposal time. Because `execute_treasury_manager_deposit` never checks the returned LP token balance against any minimum, the transaction succeeds and the SNS treasury silently suffers a loss of value. The approved but unused token amounts are returned to the treasury (as noted in the warning), but the LP tokens received can be arbitrarily small, representing a permanent loss of the deposited value relative to what was voted on.

**Vulnerability class**: Ledger conservation bug / governance authorization bug — the SNS governance canister executes a treasury operation without enforcing the minimum output invariant that the DAO implicitly assumed when voting.

### Likelihood Explanation

The `ALLOWED_EXTENSIONS` whitelist is currently empty (KongSwap ceased operations April 6, 2026), so no new extensions can be registered at this moment: [7](#0-6) 

However, the framework is production code designed for future extensions. When a new blessed DEX adaptor is registered, any SNS DAO that votes to deposit treasury funds into a DEX that does not implement per-deposit slippage protection is exposed. On the IC, there is no mempool-based MEV, but:

- Any unprivileged canister can interact with the DEX pool between the proposal's adoption and its execution to move the price.
- Natural market movement during the governance voting period (which can last days) can cause the same effect.
- A malicious or buggy DEX canister can accept the full allowance and return minimal LP tokens.

The `treasury_manager.did` and the proposal renderer both acknowledge this is a realistic scenario, not a theoretical one.

### Recommendation

1. **Add `min_lp_tokens_out` to `DepositRequest`**: Extend the `DepositRequest` type in `treasury_manager.did` with an optional `min_lp_tokens_out : opt nat` field, analogous to `amountOutMinimum` in Uniswap V3.

2. **Enforce the minimum in `execute_treasury_manager_deposit`**: After the `deposit` call returns, inspect the `external_custodian` balance in the returned `Balances` struct and revert (return `Err`) if it is below the minimum specified in the proposal.

3. **Enforce the minimum in `validate_deposit_operation_impl`**: Require that `min_lp_tokens_out` is present and non-zero in the proposal arguments, so voters can see and approve the slippage tolerance on-chain.

4. **Balance-delta check**: As an additional defense, record the treasury manager's LP token balance before and after the `deposit` call and assert the delta meets the minimum, mirroring the "check balance before and after" recommendation from the original report.

### Proof of Concept

1. An SNS DAO votes to deposit 1,000 ICP + 500,000 SNS tokens into a DEX liquidity pool via `ExecuteExtensionOperation { operation_name: "deposit", operation_arg: { treasury_allocation_icp_e8s: 100_000_000_000, treasury_allocation_sns_e8s: 50_000_000_000_000 } }`.
2. The proposal passes and `execute_treasury_manager_deposit` is called.
3. `approve_treasury_manager` grants the full ICRC-2 allowance to the treasury manager canister for both tokens.
4. Before the `deposit` call reaches the DEX, an adversarial canister (or natural price movement) shifts the pool price significantly.
5. The DEX accepts the full token amounts at the new (unfavorable) price and mints a tiny number of LP tokens.
6. `execute_treasury_manager_deposit` receives the `Balances` response, logs it, and returns `Ok(())` — no minimum check is performed.
7. The SNS treasury has lost the deposited value relative to what the DAO voted on, with no on-chain revert.

The root cause lines are:

- Missing `min_lp_tokens_out` field: [2](#0-1) 
- No minimum output check after deposit: [8](#0-7) 
- No minimum output validation at proposal time: [3](#0-2)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
