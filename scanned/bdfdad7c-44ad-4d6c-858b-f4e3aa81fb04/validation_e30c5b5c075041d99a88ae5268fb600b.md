### Title
NNS Governance `would_follow_ballots` Unchecked Duplicate Followee IDs Allow Following-Majority Manipulation - (`rs/nns/governance/src/neuron/types.rs`)

---

### Summary

NNS governance allows a neuron owner to register the same followee neuron ID multiple times in a `Follow` or `SetFollowing` command. Because `would_follow_ballots` iterates over the raw `Vec<NeuronId>` without deduplication, a duplicated followee's vote is counted multiple times in the majority calculation, allowing the neuron owner to make their neuron follow a specific neuron's vote even when the true majority of unique followees voted the opposite way.

---

### Finding Description

The `would_follow_ballots` function in `rs/nns/governance/src/neuron/types.rs` computes the following vote by iterating over `followees` (a `Vec<NeuronId>`) and counting yes/no votes:

```rust
for f in followees.iter() {
    if let Some(f_vote) = ballots.get(&f.id) {
        if f_vote.vote == (Vote::Yes as i32) {
            yes = yes.saturating_add(1);
        } else if f_vote.vote == (Vote::No as i32) {
            no = no.saturating_add(1);
        }
    }
}
if yes.saturating_mul(2_usize) > followees.len() { return Vote::Yes; }
if no.saturating_mul(2_usize) >= followees.len() { return Vote::No; }
``` [1](#0-0) 

If the same neuron ID appears N times in the `followees` Vec, its vote is counted N times while `followees.len()` is also N times larger. This skews the majority threshold in favor of the duplicated followee.

The NNS governance `Follow` handler validates only visibility/existence of followees, not uniqueness: [2](#0-1) 

The `SetFollowing.validate_intrinsically()` checks only that topics are unique and that the followee count does not exceed `MAX_FOLLOWEES_PER_TOPIC`, but never checks for duplicate neuron IDs within a topic's followee list: [3](#0-2) 

The integration test `follow_same_neuron_multiple_times` explicitly confirms that NNS governance accepts and stores duplicate followees without error: [4](#0-3) 

By contrast, SNS governance explicitly rejects duplicate neuron IDs in `ValidatedFolloweesForTopic::try_from`, returning a `DuplicateFolloweeNeuronId` error: [5](#0-4) 

---

### Impact Explanation

A neuron owner can make their neuron follow a specific neuron's vote even when the majority of unique followees voted the opposite way. Concrete example:

- Neuron A registers followees `[B, B, C]` (B duplicated twice).
- B votes Yes, C votes No.
- `yes=2, no=1, len=3` → `2×2 > 3` → **Vote::Yes** (follows B).
- Without duplicates `[B, C]`: `yes=1, no=1, len=2` → `1×2 >= 2` → **Vote::No** (follows C).

The attacker cannot inflate their own ballot weight (voting power is fixed at proposal creation), but they can cause their neuron to vote contrary to the intended majority of their unique followees. On high-stakes NNS proposals (e.g., subnet management, protocol upgrades, treasury transfers), this allows a neuron owner to deviate from the following configuration they publicly advertise, undermining the trust model of liquid democracy on the NNS.

---

### Likelihood Explanation

Any neuron controller or hot key can call `manage_neuron` with a `Follow` command containing duplicate neuron IDs. No special privilege, admin key, or threshold corruption is required. The attack is a single ingress message. The integration test confirms the path is reachable and accepted today.

---

### Recommendation

Add a duplicate-neuron-ID check in the NNS governance `Follow` and `SetFollowing` handlers, mirroring the check already present in SNS governance. Specifically:

1. In `SetFollowing::validate_intrinsically` (`rs/nns/governance/src/pb/mod.rs`), add a per-topic uniqueness check over the `followees` Vec, analogous to `get_duplicate_followee_groups` in SNS governance.
2. In the `Follow` command handler (`rs/nns/governance/src/governance.rs`), reject requests where `new_followees.followees` contains duplicate `NeuronId` values.
3. Alternatively, deduplicate the stored followees Vec before persisting, so that even if duplicates are submitted they have no effect.

---

### Proof of Concept

1. Neuron A (controller = Alice) currently follows `[B, C]` on topic `Governance`. B and C are expected to split 50/50.
2. Alice calls `manage_neuron` → `Follow { topic: Governance, followees: [B, B, C] }`.
3. NNS governance accepts the call (no duplicate check), stores `[B, B, C]`.
4. A proposal is submitted. B votes Yes, C votes No.
5. `would_follow_ballots` computes: `yes=2` (B counted twice), `no=1`, `len=3`. `2×2=4 > 3` → Neuron A votes **Yes**.
6. Without the duplicate, the correct result would be `yes=1, no=1, len=2` → `1×2=2 >= 2` → Neuron A votes **No**.

Alice has manipulated her neuron's following outcome by duplicating a single followee ID, with no special access required beyond controlling her own neuron.
