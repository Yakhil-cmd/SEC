### Title
SNS Governance `maybe_finalize_disburse_maturity` Does Not Remove Disbursement Entry After Successful Transfer - (File: `rs/sns/governance/src/governance.rs`)

### Summary
In the SNS governance canister, `maybe_finalize_disburse_maturity` fails to remove a `disburse_maturity_in_progress` entry from a neuron after a successful ledger minting transfer when `get_neuron_result_mut` returns an error post-await. The disbursement record persists in state, and the next periodic invocation re-processes the same entry, causing double-minting of SNS tokens to the beneficiary.

### Finding Description
`maybe_finalize_disburse_maturity` (called from `run_periodic_tasks`) collects all neurons with ready disbursements, then for each one: acquires a per-neuron lock, calls `transfer_funds` across an `await` boundary, and only removes the disbursement entry **after** the transfer succeeds. [1](#0-0) 

The global re-entry guard is set here. Then the critical post-transfer cleanup: [2](#0-1) 

After `transfer_funds` returns `Ok`, the code calls `get_neuron_result_mut` to obtain a mutable reference to the neuron. If this lookup fails (line 5058–5067), the code executes `continue` — **skipping `neuron.disburse_maturity_in_progress.remove(0)` at line 5069**. The transfer already occurred on the ledger, but the disbursement entry remains in the neuron's `disburse_maturity_in_progress` list. [3](#0-2) 

The global lock is then cleared. On the next timer tick, `maybe_finalize_disburse_maturity` finds the same entry still present (its `finalize_disbursement_timestamp_seconds` is already in the past), acquires the per-neuron lock again, and issues a second `transfer_funds` for the same amount to the same account — double-minting SNS tokens.

**Contrast with NNS governance**: The NNS `try_finalize_maturity_disbursement` correctly pops the disbursement **before** the transfer and pushes it back on failure: [4](#0-3) 

The SNS implementation inverts this order, leaving a window where a successful transfer is not matched by state cleanup.

### Impact Explanation
An SNS neuron holder who calls `disburse_maturity` receives a minting transfer for the same disbursement entry on every subsequent periodic-task execution until the entry is eventually removed (or the canister is upgraded). Each extra transfer mints new SNS tokens from the minting account, inflating the token supply and draining value from all other token holders. The attacker's gain is unbounded across timer ticks.

### Likelihood Explanation
The trigger condition is `get_neuron_result_mut` returning `Err` after a successful `transfer_funds`. This requires the neuron to be absent from `self.proto.neurons` at the moment the callback resumes — possible if the neuron is deleted by another message processed during the `transfer_funds` inter-canister await (the per-neuron lock blocks commands but does not block all deletion paths). The window is narrow and requires an unusual concurrent state change, making this **low likelihood** in practice, but the structural defect is present in production code and the impact when triggered is severe.

### Recommendation
Mirror the NNS governance pattern: remove the `disburse_maturity_in_progress` entry **before** calling `transfer_funds`, and re-insert it at the front of the list if the transfer fails. This eliminates the possibility of a successful transfer leaving a stale disbursement entry that will be re-processed.

```rust
// Step 1: remove the entry before the transfer
neuron.disburse_maturity_in_progress.remove(0);

// Step 2: transfer
let transfer_result = self.ledger.transfer_funds(...).await;

// Step 3: on failure, restore
if transfer_result.is_err() {
    if let Ok(neuron) = self.get_neuron_result_mut(&neuron_id) {
        neuron.disburse_maturity_in_progress.insert(0, disbursement);
    }
}
```

### Proof of Concept

1. SNS neuron holder calls `manage_neuron` → `DisburseMaturity` for 100% of maturity. The entry is appended to `disburse_maturity_in_progress` with `finalize_disbursement_timestamp_seconds = now + MATURITY_DISBURSEMENT_DELAY_SECONDS`. [5](#0-4) 

2. After the delay, `run_periodic_tasks` fires and calls `maybe_finalize_disburse_maturity`. [6](#0-5) 

3. The neuron is collected, the per-neuron lock is acquired, and `transfer_funds` is called and succeeds — SNS tokens are minted to the beneficiary. [7](#0-6) 

4. `get_neuron_result_mut` returns `Err` (e.g., neuron deleted by a concurrent message during the await). The `continue` at line 5066 skips `remove(0)`. [8](#0-7) 

5. `is_finalizing_disburse_maturity` is cleared to `None`. [3](#0-2) 

6. On the next timer tick, the same disbursement entry is found again (timestamp already past), the lock is re-acquired, and `transfer_funds` is called again — minting a second batch of SNS tokens for the same original maturity amount. This repeats on every subsequent tick until the entry is manually removed or the canister is upgraded.
