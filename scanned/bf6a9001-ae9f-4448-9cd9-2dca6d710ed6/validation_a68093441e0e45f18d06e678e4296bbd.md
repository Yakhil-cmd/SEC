### Title
Unbounded `disburse_maturity_in_progress` Vec Growth in SNS Governance Enables Storage Exhaustion - (File: `rs/sns/governance/src/governance.rs`)

### Summary
The SNS governance `disburse_maturity` function unconditionally pushes entries to a neuron's `disburse_maturity_in_progress` Vec with no per-neuron limit. The NNS governance equivalent has an explicit `MAX_NUM_DISBURSEMENTS` guard, but SNS governance does not. A neuron owner with `DisburseMaturity` permission and sufficient maturity can repeatedly call `manage_neuron` with small percentages to grow this Vec without bound, inflating the SNS governance canister's heap memory.

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `disburse_maturity` function at line 1697 unconditionally pushes a new `DisburseMaturityInProgress` entry to the neuron's Vec:

```rust
neuron
    .disburse_maturity_in_progress
    .push(disbursement_in_progress);
``` [1](#0-0) 

The only guards present are:
- `percentage_to_disburse` must be 1–100
- `worst_case_maturity_modulation >= transaction_fee_e8s`

Neither guard limits the *count* of pending disbursements. The NNS governance equivalent (`rs/nns/governance/src/governance/disburse_maturity.rs`) explicitly checks `num_disbursements >= MAX_NUM_DISBURSEMENTS` and returns `TooManyDisbursements`: [2](#0-1) 

SNS governance has no such check. The `DisburseMaturityInProgress` struct is stored inline in the neuron protobuf: [3](#0-2) 

### Impact Explanation

Each `DisburseMaturityInProgress` entry is serialized as part of the neuron's protobuf and stored in the SNS governance canister's heap. With a neuron holding large maturity (e.g., 10^9 e8s), an attacker calling `disburse_maturity` with `percentage_to_disburse = 1` repeatedly can accumulate hundreds of entries before the remaining maturity falls below the minimum disbursement threshold. This inflates the SNS governance canister's heap memory proportionally. If many neurons are exploited simultaneously, the canister could exhaust its memory allocation, causing the SNS governance canister to trap on state writes or become unable to accept new neurons — a denial-of-service against the entire SNS instance.

### Likelihood Explanation

Any holder of an SNS neuron with the `DisburseMaturity` permission (which is a standard, user-grantable permission) can trigger this. No privileged role is required. The attacker must own a neuron with meaningful maturity, which requires staking SNS tokens — a real but low cost for a targeted attack. The attack is reachable via the standard `manage_neuron` ingress endpoint. [4](#0-3) 

### Recommendation

Add a per-neuron limit on the number of pending maturity disbursements in SNS governance, mirroring the `MAX_NUM_DISBURSEMENTS` guard already present in NNS governance:

```rust
const MAX_DISBURSE_MATURITY_IN_PROGRESS: usize = 7; // or similar

if neuron.disburse_maturity_in_progress.len() >= MAX_DISBURSE_MATURITY_IN_PROGRESS {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Too many maturity disbursements in progress.",
    ));
}
``` [5](#0-4) 

### Proof of Concept

1. Deploy an SNS with default parameters and a neuron holding 10^9 e8s of maturity.
2. Repeatedly call `manage_neuron` with `Command::DisburseMaturity { percentage_to_disburse: 1, to_account: None }` from the neuron controller.
3. Each call succeeds (maturity decreases by ~1% each time) and appends one `DisburseMaturityInProgress` entry.
4. With a default SNS transaction fee of 10,000 e8s, approximately 680+ entries can be accumulated before the remaining maturity is too small to disburse.
5. Observe that `neuron.disburse_maturity_in_progress.len()` grows without bound, inflating the SNS governance canister's heap.
6. Repeat across multiple neurons to exhaust the canister's memory budget. [6](#0-5)
