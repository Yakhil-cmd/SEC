### Title
Unbounded `ingress_end` in `construction_payloads` Enables OOM DoS via Unbounded Envelope-Generation Loop — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

---

### Summary

An unauthenticated HTTP client can POST to `/construction/payloads` with `metadata.ingress_end = u64::MAX`. The two existing guards do not bound the upper value of `ingress_end`, so the while-loop at lines 163–167 runs ~139 million iterations, allocating gigabytes of heap memory and crashing the Rosetta node.

---

### Finding Description

`construction_payloads` in `services.rs` computes an `ingress_interval` of 240 seconds (= `MAX_INGRESS_TTL` − `PERMITTED_DRIFT` = 300s − 60s) and an `INGRESS_INTERVAL_OVERLAP` of 120 seconds, giving a loop step of 120 × 10⁹ ns per iteration. [1](#0-0) [2](#0-1) 

The only two guards before the loop are:

1. `ingress_start >= ingress_end` → error [3](#0-2) 
2. `ingress_end < now + ingress_interval` → error [4](#0-3) 

Neither guard imposes an **upper bound** on `ingress_end`. With `ingress_end = u64::MAX ≈ 1.844 × 10¹⁹ ns` and `ingress_start = now ≈ 1.75 × 10¹⁸ ns`, both guards pass, and the loop executes:

```
(u64::MAX − now) / 120e9  ≈  139,000,000 iterations
```

Each iteration pushes a `u64` into `ingress_expiries`: [5](#0-4) 

That alone is ~1.1 GB. The entire `ingress_expiries` vec is then passed to `handle_construction_payloads`, which allocates a full `EnvelopeContent::Call` struct (containing `Vec<u8>` fields for `arg`, `nonce`, method name, etc.) for every entry: [6](#0-5) 

The endpoint is reachable with no authentication: [7](#0-6) 

The same unbounded loop pattern also exists in the ICP Rosetta implementation: [8](#0-7) 

---

### Impact Explanation

A single malformed HTTP request causes the Rosetta process to exhaust available memory (OOM kill) or spin for an extended period, making the node unavailable. All legitimate ledger transfers routed through that Rosetta node are blocked for the duration of the crash/restart cycle.

---

### Likelihood Explanation

The endpoint requires no credentials. The malicious payload is a standard JSON body with one field set to a large integer. Any attacker who can reach the Rosetta HTTP port can trigger this with a single `curl` command.

---

### Recommendation

Add an explicit upper-bound check immediately after the existing guards, before the loop:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // e.g. 24 h
if ingress_end > now + MAX_INGRESS_WINDOW.as_nanos() as u64 {
    return Err(Error::processing_construction_failed(
        "ingress_end exceeds maximum allowed window"
    ));
}
```

This caps the loop to at most `24h / 120s = 720` iterations, which is safe.

---

### Proof of Concept

```rust
#[test]
fn test_construction_payloads_dos() {
    use std::time::{Duration, SystemTime};
    let now = SystemTime::now()
        .duration_since(SystemTime::UNIX_EPOCH)
        .unwrap()
        .as_nanos() as u64;

    // Both existing guards pass: ingress_start < ingress_end, and
    // ingress_end >> now + ingress_interval
    let result = construction_payloads(
        vec![/* valid transfer operation */],
        Some(ConstructionPayloadsRequestMetadata {
            ingress_start: Some(now),
            ingress_end: Some(u64::MAX),   // ← attacker-controlled
            created_at_time: None,
            memo: None,
        }),
        &Principal::anonymous(),
        vec![/* valid public key */],
        SystemTime::now(),
    );
    // Without the fix this hangs / OOMs; with the fix it must return Err.
    assert!(result.is_err(), "must reject astronomically large ingress_end");
}
```
