### Title
Single `ProcessLogic` Timer Task as Sole Automated Mechanism for ckBTC Withdrawal Processing — (File: rs/bitcoin/ckbtc/minter/src/tasks.rs)

---

### Summary
The ckBTC minter relies on a single `ProcessLogic` timer task to drive all BTC withdrawal processing (`submit_pending_requests`, `finalize_requests`, `reimburse_withdrawals`). All three task types share one global `TimerLogicGuard` lock. There is no redundancy, no health monitoring, and no alerting when the task fails persistently — directly analogous to the external report's single-bot liquidation design.

---

### Finding Description

In `rs/bitcoin/ckbtc/minter/src/tasks.rs`, the `run_task` function handles three task types. The `ProcessLogic` branch is the sole automated path for processing pending BTC withdrawal requests:

```rust
TaskType::ProcessLogic => {
    const INTERVAL_PROCESSING: Duration = Duration::from_secs(5);
    let _enqueue_followup_guard = guard((), |_| {
        schedule_after(INTERVAL_PROCESSING, TaskType::ProcessLogic, &runtime)
    });
    let _guard = match crate::guard::TimerLogicGuard::new() {
        Some(guard) => guard,
        None => return,
    };
    submit_pending_requests(&runtime).await;
    finalize_requests(&runtime).await;
    reimburse_withdrawals(&runtime).await;
}
``` [1](#0-0) 

All three task types (`ProcessLogic`, `RefreshFeePercentiles`, `ConsolidateUtxos`) compete for the same single `TimerLogicGuard`, backed by the `is_timer_running` boolean in the minter state:

```rust
pub struct TimerLogicGuard(());
impl TimerLogicGuard {
    pub fn new() -> Option<Self> {
        mutate_state(|s| {
            if s.is_timer_running { return None; }
            s.is_timer_running = true;
            Some(TimerLogicGuard(()))
        })
    }
}
``` [2](#0-1) 

The minter is initialized with `setup_tasks()` which schedules all three task types:

```rust
fn setup_tasks() {
    schedule_now(TaskType::ProcessLogic, &IC_CANISTER_RUNTIME);
    schedule_now(TaskType::RefreshFeePercentiles, &IC_CANISTER_RUNTIME);
    schedule_now(TaskType::ConsolidateUtxos, &IC_CANISTER_RUNTIME);
}
``` [3](#0-2) 

**Root cause:** There is no redundancy for the `ProcessLogic` task. If `submit_pending_requests`, `finalize_requests`, or `reimburse_withdrawals` fails persistently — for example, because the BTC canister or ckBTC ledger changes its interface — the task retries every 5 seconds indefinitely but never succeeds. There is no health metric tracking consecutive failures, no alerting threshold, and no documented manual fallback procedure in the code. The only observable signal is the `ckbtc_minter_global_timer_timestamp` gauge, which shows the next scheduled timer deadline but not whether the task is actually succeeding: [4](#0-3) 

Additionally, the single `TimerLogicGuard` means that if `RefreshFeePercentiles` or `ConsolidateUtxos` holds the lock, `ProcessLogic` is silently skipped for that cycle (returns `None` → early return), with no indication that a withdrawal processing cycle was missed. [5](#0-4) 

---

### Impact Explanation

A chain-fusion user who calls `retrieve_btc` or `retrieve_btc_with_approval` has their ckBTC burned and a withdrawal request queued. If the `ProcessLogic` task fails persistently (e.g., due to an interface change in the BTC canister), the BTC transaction is never submitted to the Bitcoin network. The user's funds are locked in a pending state indefinitely — ckBTC already burned, BTC not yet sent — until the minter canister is manually upgraded to fix the interface mismatch. There is no in-protocol manual fallback path. [6](#0-5) 

---

### Likelihood Explanation

Low. Interface changes to the BTC canister or ckBTC ledger are rare and typically coordinated with the minter upgrade. However, the risk is non-zero: the external report demonstrates exactly this failure mode (bot stopped due to interface change), and the IC's chain-fusion stack has multiple independently upgradeable components (BTC canister, ckBTC ledger, ckBTC minter) whose upgrade sequencing could transiently break the `ProcessLogic` task's assumptions. [1](#0-0) 

---

### Recommendation

1. **Add a health metric** tracking the timestamp of the last *successful* `ProcessLogic` execution (distinct from the next scheduled timer deadline), so operators can detect persistent failures via Prometheus alerting.
2. **Add a consecutive-failure counter** inside `ProcessLogic`; emit a high-priority log or metric when it exceeds a threshold (e.g., 10 consecutive failures).
3. **Document the manual fallback procedure**: which minter state fields to inspect, how to diagnose the stuck withdrawal queue, and how to trigger a manual retry after fixing the interface mismatch.
4. **Decouple the `TimerLogicGuard`** per task type (as the ckETH minter already does with `TimerGuard` keyed by `TaskType`) so that a slow `RefreshFeePercentiles` or `ConsolidateUtxos` execution cannot silently starve `ProcessLogic` cycles. [7](#0-6) 

---

### Proof of Concept

1. User calls `retrieve_btc_with_approval(address, amount)` on the ckBTC minter; ckBTC is burned and a `RetrieveBtcRequest` is queued in `pending_retrieve_btc_requests`.
2. The BTC canister is upgraded with a breaking interface change (e.g., a new mandatory field in `send_transaction`).
3. `ProcessLogic` fires every 5 seconds; `submit_pending_requests` calls the BTC canister and receives a `CallError::CandidDecodeFailed` or `Reject`.
4. The error is logged at `Priority::Info`; `_enqueue_followup_guard` reschedules the task after 5 seconds. The cycle repeats indefinitely.
5. No metric crosses an alerting threshold; no operator is paged.
6. The user's BTC withdrawal remains stuck until a minter upgrade is deployed — with no in-protocol recovery path and no documented manual procedure. [6](#0-5) [2](#0-1)

#[derive(Eq, PartialEq, Debug)]
pub struct TimerGuard {
    task: TaskType,
}

impl TimerGuard {
    pub fn new(task: TaskType) -> Result<Self, TimerGuardError> {
        mutate_state(|s| {
            if !s.active_tasks.insert(task) {
                return Err(TimerGuardError::AlreadyProcessing);
            }
            Ok(Self { task })
        })
    }
}

impl Drop for TimerGuard {
    fn drop(&mut self) {
        mutate_state(|s| {
            s.active_tasks.remove(&self.task);
        });
    }
}
