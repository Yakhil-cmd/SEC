### Title
Shared `QueryScheduler` Allows User Queries to Starve Ingress Filter Checks for the Same Canister — (`rs/execution_environment/src/ingress_filter.rs`)

### Summary
The `QueryScheduler` is shared between the query handler and the ingress filter. Both push closures into the same per-canister FIFO queue. The scheduler is round-robin *between canisters*, not between request types. An attacker who floods canister A's per-canister queue with max-instruction user queries can delay ingress filter checks for canister A beyond the HTTP request timeout (300 s), causing `504 Gateway Timeout` on `/api/v2/call` for that canister.

### Finding Description

**Shared scheduler — confirmed in production code:**

`setup_execution_helper` creates one `QueryScheduler` and clones it to both the ingress filter and the query handler: [1](#0-0) 

Both `IngressFilterServiceImpl::call` and `HttpQueryHandler::call` call `self.query_scheduler.push(canister_id, ...)`: [2](#0-1) [3](#0-2) 

**Unbounded per-canister queue — no capacity check:**

`QuerySchedulerCore::push` appends unconditionally to `canister.incoming` (a `VecDeque<Query>`) with no size limit: [4](#0-3) 

**Round-robin is between canisters, not request types:**

When canister A is scheduled, it drains its own FIFO queue (queries + ingress filter checks interleaved in arrival order) for one `time_slice_per_canister` (20 ms), then yields to the next canister: [5](#0-4) 

**Thread concurrency per canister is capped at 2:** [6](#0-5) 

**Total query execution threads = 4:** [7](#0-6) 

**HTTP-level concurrency limits are separate per endpoint but share the same scheduler:**

- Query endpoint: `max_query_concurrent_requests = QUERY_EXECUTION_THREADS_TOTAL * 100 = 400`
- Call endpoint (ingress filter): `max_call_concurrent_requests = 50` [8](#0-7) [9](#0-8) 

**Attack path:**

1. Attacker opens one HTTP/2 connection (within the nftables 400-connection-per-IP limit) and sends 400 concurrent max-instruction queries to canister A. Each query is pushed to canister A's `incoming` queue.
2. Victim submits an ingress message to canister A. The ingress filter check is pushed *behind* the 400 queries in canister A's queue.
3. With `max_threads_per_canister = 2` and max-instruction queries (~5 s each): drain time ≈ 400 / 2 × 5 s = **1000 s**.
4. The HTTP request timeout is **300 s** (`request_timeout_seconds`): [10](#0-9) 
5. The ingress filter check times out → victim receives `504 Gateway Timeout` → ingress message is not submitted.

The attacker does not need to keep sending new queries after the initial 400; those 400 are sufficient to exceed the timeout. New queries pushed after the ingress filter check go *behind* it in the FIFO queue and do not worsen the delay.

### Impact Explanation
An unprivileged attacker can prevent any user from submitting ingress messages to a specific canister for as long as the attacker maintains 400 concurrent max-instruction queries. The impact is canister-scoped (other canisters are unaffected) and sustained for the duration of the attack. Victims receive `504 Gateway Timeout` on `/api/v2/call`.

### Likelihood Explanation
The attack requires only a single HTTP/2 connection (400 streams) and a canister with a query method that can be made to run near the instruction limit. This is a realistic precondition on mainnet. The nftables rate limit (2000 new connections/min) does not prevent this because HTTP/2 multiplexes 400 streams over one TCP connection.

### Recommendation
1. **Separate the ingress filter from the query scheduler.** Give `IngressFilterServiceImpl` its own dedicated thread pool or a higher-priority queue, so ingress filter checks are never blocked by user queries.
2. **Alternatively, add a priority lane** inside `QuerySchedulerCore` for ingress filter tasks, processed before `incoming` user queries.
3. **Or bound the per-canister queue depth** in `QuerySchedulerCore::push` and reject (or back-pressure) new queries when the queue is full, preventing unbounded accumulation.

### Proof of Concept
```
1. Deploy canister A with a query method that loops to the instruction limit.
2. Open one HTTP/2 connection to a replica node.
3. Send 400 concurrent query calls to canister A's slow method.
4. Concurrently, attempt to submit an ingress message to canister A via /api/v2/call.
5. Observe: the /api/v2/call request returns 504 Gateway Timeout after 300 s.
6. Assert: the latency of the ingress filter response is NOT bounded independently of query load — it is directly proportional to the depth of canister A's query queue.
```

    fn push(&mut self, canister_id: CanisterId, query: Query) {
        let canister = self
            .canisters
            .entry(canister_id)
            .or_insert_with(CanisterData::new);
        canister.incoming.push_back(query);

        self.metrics
            .queue_length
            .observe((canister.incoming.len() + canister.leftover.len()) as f64);

        if !canister.has_been_scheduled
            && canister.should_be_scheduled(self.max_threads_per_canister)
        {
            canister.has_been_scheduled = true;
            self.scheduled.push_back(canister_id);
        }

        #[cfg(debug_assertions)]
        self.verify_invariants();
    }

    /// Returns a batch of queries to execute if there are any.
    fn pop(&mut self) -> Option<(CanisterId, Vec<Query>)> {
        let canister_id = self.scheduled.pop_front()?;
        // It is safe to unwrap here because of the invariants in
        // `validate_invariants()`: each canister in the round-robin list must
        // be present in the canister table.
        let canister = self.canisters.get_mut(&canister_id).unwrap();
        debug_assert!(canister.has_been_scheduled);
        canister.has_been_scheduled = false;

        let total = canister.queries_per_time_slice(self.time_slice_per_canister);

        // Collect queries from the `leftover` queue first.
        let from_leftover = total.min(canister.leftover.len());
        let mut result: Vec<_> = canister.leftover.drain(0..from_leftover).collect();

        // Get the remaining queries from the `incoming` queue.
        let from_incoming = (total - from_leftover).min(canister.incoming.len());
        result.extend(canister.incoming.drain(0..from_incoming));

        // Follows from the main invariant.
        debug_assert!(!result.is_empty());

        canister.active_threads += 1;

        // We removed the canister from the schedule at the beginning of this
        // method and cleared `has_been_scheduled`.
        debug_assert!(!canister.has_been_scheduled);
        if canister.should_be_scheduled(self.max_threads_per_canister) {
            canister.has_been_scheduled = true;
            self.scheduled.push_back(canister_id);
        }

        #[cfg(debug_assertions)]
        self.verify_invariants();

        Some((canister_id, result))
    }
