### Title
Deferred Async Initialization Leaves `API_BOUNDARY_NODE_PRINCIPALS` Empty, Causing Incorrect Access Rejection in Salt-Sharing Canister During Every Install/Upgrade Window - (`File: rs/boundary_node/salt_sharing/canister/canister.rs`)

---

### Summary

The `salt_sharing` canister defers its entire initialization — including populating the `API_BOUNDARY_NODE_PRINCIPALS` access-control set — to an async timer callback (`init_async`). Because `API_BOUNDARY_NODE_PRINCIPALS` is a non-stable `thread_local!` variable that starts as an empty `HashSet`, there is a guaranteed multi-round window after every `#[init]` and `#[post_upgrade]` during which the `inspect_message` hook and the `get_salt` query both incorrectly treat every caller — including legitimate API boundary nodes — as unauthorized.

---

### Finding Description

**Root cause — deferred async init:**

In `rs/boundary_node/salt_sharing/canister/canister.rs`, the `#[init]` handler (also called verbatim from `#[post_upgrade]`) does not perform any initialization synchronously. It only schedules a zero-delay timer:

```rust
#[init]
fn init(init_arg: InitArg) {
    set_timer(Duration::ZERO, async { init_async(init_arg).await });
    ...
}
``` [1](#0-0) 

`init_async` is the function that actually populates `API_BOUNDARY_NODE_PRINCIPALS` — but only after it first awaits a call to the management canister (`raw_rand`) and then sets up a periodic timer whose first firing calls the registry canister (`get_api_boundary_node_ids`):

```rust
pub async fn init_async(init_arg: InitArg) {
    if (!is_salt_init() || init_arg.regenerate_now)
        && let Err(err) = try_regenerate_salt().await   // ← async, awaits raw_rand
    { ... }
    ...
    set_timer_interval(period, async || poll_api_boundary_nodes().await);
    // ↑ poll_api_boundary_nodes is NOT called immediately; it fires after `period` seconds
}
``` [2](#0-1) 

`poll_api_boundary_nodes` is the only function that writes to `API_BOUNDARY_NODE_PRINCIPALS`:

```rust
API_BOUNDARY_NODE_PRINCIPALS.with(|cell| *cell.borrow_mut() = principals);
``` [3](#0-2) 

**The storage variable starts empty and is never persisted to stable memory:**

```rust
pub static API_BOUNDARY_NODE_PRINCIPALS: RefCell<HashSet<Principal>> =
    RefCell::new(HashSet::new());
``` [4](#0-3) 

Because this is a `thread_local!` heap variable (not stable memory), it is reset to an empty `HashSet` on every upgrade.

**Access control depends on this empty set:**

The `inspect_message` hook — which gates all ingress update-mode calls — reads from this set:

```rust
fn inspect_message() {
    let caller_id = caller();
    let called_method = method_name();
    if called_method == REPLICATED_QUERY_METHOD && is_api_boundary_node_principal(&caller_id) {
        accept_message();
    } else {
        trap("message_inspection_failed: method call is prohibited in the current context");
    }
}
``` [5](#0-4) 

The `get_salt` query function also reads from this set:

```rust
fn get_salt() -> GetSaltResponse {
    let caller_id = caller();
    if is_api_boundary_node_principal(&caller_id) {
        ...
    }
    Err(GetSaltError::Unauthorized)
}
``` [6](#0-5) 

`is_api_boundary_node_principal` simply checks the in-memory set:

```rust
pub fn is_api_boundary_node_principal(principal: &Principal) -> bool {
    API_BOUNDARY_NODE_PRINCIPALS.with(|cell| cell.borrow().contains(principal))
}
``` [7](#0-6) 

**Timeline of the vulnerable window:**

1. `#[init]` / `#[post_upgrade]` returns → canister is live, accepting messages.
2. Zero-delay timer fires in the next round → `init_async` starts.
3. `init_async` awaits `try_regenerate_salt()` (async call to management canister `raw_rand`) — multiple rounds.
4. `init_async` then calls `set_timer_interval(period, ...)` — the first `poll_api_boundary_nodes` fires only after `registry_polling_interval_secs` seconds (configured to 300 s on mainnet).
5. Only after step 4's first interval fires and the registry call returns does `API_BOUNDARY_NODE_PRINCIPALS` get populated.

During the entire window from step 1 to step 5 (which is at minimum `registry_polling_interval_secs` = 300 seconds on mainnet), `API_BOUNDARY_NODE_PRINCIPALS` is empty.

The integration test itself confirms this is the expected (but unguarded) behavior:

```rust
// Salt should not be initialized immediately after canister's installation
assert_eq!(salt_id, 0);
// But once some rounds pass, salt should be initialized
tick_n_times(&pocket_ic, TICKS).await;
``` [8](#0-7) 

---

### Impact Explanation

During the initialization window (guaranteed on every install and every upgrade):

- **All update-mode calls to `get_salt` are rejected** by `inspect_message` with `"message_inspection_failed"`, including calls from legitimate API boundary nodes.
- **All query calls to `get_salt` return `GetSaltError::Unauthorized`** for every caller, including legitimate API boundary nodes — even if the salt itself is already present in stable memory from a previous install.

The second point is particularly impactful on upgrades: the salt value survives in stable memory (`SALT` is a `StableBTreeMap`), so `is_salt_init()` returns `true`, but `API_BOUNDARY_NODE_PRINCIPALS` is wiped. API boundary nodes calling `get_salt` immediately after an upgrade receive `Unauthorized` (not `SaltNotInitialized`), giving them no signal to retry with a different strategy. On mainnet, with `registry_polling_interval_secs = 300`, this window is at least 5 minutes per upgrade.

The salt is used by API boundary nodes to consistently anonymize sensitive data (IP addresses). Denial of the salt during the upgrade window causes inconsistent or failed anonymization across the boundary node fleet.

---

### Likelihood Explanation

This condition is triggered deterministically on every canister install and every upgrade — no attacker action is required. The mainnet deployment proposal confirms `registry_polling_interval_secs = 300`, making the window at least 5 minutes. [9](#0-8) 

Any canister upgrade (which requires only an NNS governance proposal) triggers the window. The upgrade path is:

```rust
#[post_upgrade]
fn post_upgrade(init_arg: InitArg) {
    init(init_arg);  // same deferred async pattern
}
``` [10](#0-9) 

---

### Recommendation

1. **Persist `API_BOUNDARY_NODE_PRINCIPALS` in stable memory** so it survives upgrades and is immediately available after `post_upgrade`, eliminating the window for the upgrade case.
2. **Perform an eager synchronous registry poll** (or at minimum schedule `poll_api_boundary_nodes` with `Duration::ZERO` in addition to the periodic interval) so the set is populated in the first round after init, not after `registry_polling_interval_secs`.
3. **Return `SaltNotInitialized` instead of `Unauthorized`** when `API_BOUNDARY_NODE_PRINCIPALS` is empty but the canister is still initializing, so callers can distinguish a transient state from a permanent access denial.

---

### Proof of Concept

1. Deploy the salt-sharing canister with `registry_polling_interval_secs = 300`.
2. Immediately (within the first 300 seconds) call `get_salt` as a query from a principal that is a registered API boundary node.
3. Observe `GetSaltError::Unauthorized` — even though the caller is a legitimate API boundary node and the salt may already be in stable memory.
4. Upgrade the canister. Repeat step 2 immediately after upgrade. Observe the same `Unauthorized` response for up to 5 minutes, despite the salt being present in stable memory from before the upgrade.

    if (!is_salt_init() || init_arg.regenerate_now)
        && let Err(err) = try_regenerate_salt().await

    set_timer_interval(period, async || poll_api_boundary_nodes().await);
}
