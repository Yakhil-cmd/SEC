### Title
ckERC20 Withdrawal: Overcharged ckETH Transaction Fees Not Reimbursed on Successful Ethereum Transaction - (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

---

### Summary

When a user calls `withdraw_erc20`, the ckETH minter burns a `max_transaction_fee` amount of ckETH upfront to cover the estimated Ethereum gas cost. Because the estimate is deliberately conservative (to allow for resubmissions), the actual gas consumed is almost always less than `max_transaction_fee`. The difference — `max_transaction_fee - actual_tx_fee` — is permanently lost to the user on every **successful** ckERC20 withdrawal. This is the direct IC analog of the M-10 finding: a user pays more tokens than are actually consumed, and the excess is never returned.

---

### Finding Description

In `withdraw_erc20`, the minter first burns `erc20_tx_fee` (the estimated maximum transaction fee) from the user's ckETH balance: [1](#0-0) 

This `erc20_tx_fee` is stored as `max_transaction_fee` in the `Erc20WithdrawalRequest`: [2](#0-1) 

When the Ethereum transaction is finalized, `record_finalized_transaction` is called. For a **successful** ckERC20 withdrawal, no reimbursement is scheduled at all — the `if receipt.status == TransactionStatus::Failure` guard means the success path is a no-op: [3](#0-2) 

The `update_balance_upon_withdrawal` function correctly computes `unspent_tx_fee = max_transaction_fee - actual_tx_fee` and tracks it in `total_unspent_tx_fees`, but this value is **only** used for accounting/metrics — it is never minted back to the user: [4](#0-3) 

This is explicitly confirmed by the test `should_not_reimburse_unused_transaction_fee_when_ckerc20_withdrawal_successful`, which asserts that `reimbursement_requests` is empty after a successful ckERC20 withdrawal even when gas was not fully consumed: [5](#0-4) 

The design documentation explicitly acknowledges this as intentional: *"Overcharged transaction fees are not reimbursed."* [6](#0-5) 

By contrast, for **failed** ckERC20 withdrawals, the ckERC20 tokens are reimbursed in full, but the overcharged ckETH fee is still not returned. For **ckETH** (non-ERC20) withdrawals, the unspent fee is also not returned on success (the user simply receives `withdrawal_amount - max_tx_fee_estimate` at the destination, and the unspent portion stays in the minter's ETH balance).

---

### Impact Explanation

Every successful `withdraw_erc20` call results in the user permanently losing `max_transaction_fee - actual_tx_fee` ckETH. The magnitude depends on Ethereum gas price volatility: the minter uses a conservative estimate that includes a safety margin for resubmissions (at least 10% headroom on `max_priority_fee_per_gas`). In practice, the documentation's own example shows an unspent fee of `923,727,584,640,000` wei (~0.00092 ETH) on a single withdrawal. Across many users and withdrawals, this constitutes a systematic, permanent fund loss for every ckERC20 withdrawer. The funds accumulate in the minter's ETH balance and are not recoverable by users.

This matches the M-10 vulnerability class: the user pays `original_amount` (= `max_transaction_fee`) but only `actual_tx_fee` is consumed, and the difference is never refunded.

---

### Likelihood Explanation

This affects **every** successful `withdraw_erc20` call. Since the gas estimate is intentionally conservative and Ethereum's EIP-1559 mechanism means `actual_tx_fee < max_fee_per_gas * gas_limit` in virtually all cases, the overpayment occurs on essentially every transaction. Any unprivileged IC principal with ckETH and ckERC20 tokens can trigger this by calling `withdraw_erc20` on the ckETH minter canister.

---

### Recommendation

After a successful ckERC20 Ethereum transaction is finalized, compute `unspent_tx_fee = max_transaction_fee - actual_tx_fee` and schedule a ckETH reimbursement mint to the user for that amount (minus the ckETH ledger transfer fee). This mirrors how failed ckETH withdrawals already reimburse `withdrawal_amount - actual_tx_fee` back to the user.

Concretely, in `record_finalized_transaction` in `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, add a reimbursement branch for the `CkErc20` success case analogous to the existing failure-case reimbursement, crediting `max_transaction_fee - effective_transaction_fee` back to the user's ckETH account. [3](#0-2) 

---

### Proof of Concept

1. User calls `withdraw_erc20` with `max_transaction_fee = 5_000_000_000_000_000` wei ckETH (a typical conservative estimate for 65,000 gas).
2. Minter burns `5_000_000_000_000_000` ckETH from user's account.
3. Ethereum transaction is mined with `actual_tx_fee = 2_000_000_000_000_000` wei.
4. `record_finalized_transaction` is called with `TransactionStatus::Success`.
5. The `CkErc20` success branch has no reimbursement logic — `reimbursement_requests` remains empty.
6. `update_balance_upon_withdrawal` records `unspent_tx_fee = 3_000_000_000_000_000` wei in `total_unspent_tx_fees` (metrics only).
7. User permanently loses `3_000_000_000_000_000` wei ckETH (~0.003 ETH) with no recourse.

The test `should_not_reimburse_unused_transaction_fee_when_ckerc20_withdrawal_successful` at line 1574 of `rs/ethereum/cketh/minter/src/state/transactions/tests.rs` explicitly encodes this behavior as expected, confirming the root cause is in the design, not a test gap. [5](#0-4)
