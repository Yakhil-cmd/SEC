### Title
ICRC-1 Archive Canister's `icrc3_supported_block_types` Does Not Advertise ICRC-152 Block Types Despite Storing Them - (`rs/ledger_suite/icrc1/archive/src/main.rs`)

---

### Summary

The ICRC-1 archive canister's `icrc3_supported_block_types` endpoint returns a hardcoded list of block types that permanently omits `"122mint"` and `"122burn"`. When the parent ledger has ICRC-152 enabled, it produces and archives blocks of these types. The archive stores them faithfully but never advertises them, breaking ICRC-3 standard compliance and causing any indexer or protocol that relies on `icrc3_supported_block_types` to silently miss all authorized-mint and authorized-burn transactions.

---

### Finding Description

The ICRC-1 ledger conditionally advertises `"122mint"` and `"122burn"` block types in `icrc3_supported_block_types` only when the `icrc152` feature flag is enabled: [1](#0-0) 

The archive canister's `icrc3_supported_block_types` is hardcoded and never includes `"122mint"` or `"122burn"`, regardless of what the parent ledger is configured to produce: [2](#0-1) 

The archive receives and stores all blocks the ledger sends via `append_blocks`, including `"122mint"` and `"122burn"` blocks produced by `icrc152_mint` and `icrc152_burn`: [3](#0-2) 

The ICRC-3 standard requires `icrc3_supported_block_types` to enumerate all block types the canister can return via `icrc3_get_blocks`. The archive can and does return `"122mint"` and `"122burn"` blocks through `icrc3_get_blocks`, but never declares them. This is the direct IC analog of `ERC1155Bridgable` supporting the `ERC1155Metadata_URI` extension (via `supportsInterface`) while failing to emit the required `URI` event on state change.

The archive's `icrc3_get_blocks` will return these blocks to any caller: [4](#0-3) 

---

### Impact Explanation

Any indexer, wallet, or protocol that calls `icrc3_supported_block_types` on the archive to discover what block types it holds will receive a list that omits `"122mint"` and `"122burn"`. Such a consumer will:

1. Not register handlers for authorized-mint and authorized-burn operations.
2. Silently skip or misclassify those blocks when iterating via `icrc3_get_blocks`.
3. Compute incorrect account balances and transaction histories for any account that received tokens via `icrc152_mint` or had tokens removed via `icrc152_burn`.

The ICRC-1 index-ng canister already demonstrates this dependency: it queries `icrc1_supported_standards` from the ledger to decide which block types to handle, and the index-ng tests confirm that `"122mint"` and `"122burn"` blocks must be correctly parsed for balance tracking: [5](#0-4) 

---

### Likelihood Explanation

The trigger path is straightforward and requires no privileged access:

1. The ledger is deployed with `icrc152: true` in its feature flags (a standard deployment option).
2. Any caller invokes `icrc152_mint` or `icrc152_burn` — these are public update endpoints callable by the designated minting authority, which is a canister-level role, not a subnet-level privilege.
3. After enough blocks accumulate, the ledger automatically archives them, sending `"122mint"` / `"122burn"` blocks to the archive.
4. From that point on, every call to `icrc3_supported_block_types` on the archive returns a non-compliant response.

The ICRC-152 feature flag is already present in the production ledger DID and the `supported_standards` test suite confirms it is a live, testable code path: [6](#0-5) 

---

### Recommendation

The archive canister does not have direct access to the parent ledger's feature flags at initialization time. Two approaches can fix this:

1. **Pass feature flags at spawn time**: When the ledger spawns an archive, include the active feature flags (including `icrc152`) in the archive's init arguments so the archive can conditionally include `"122mint"` and `"122burn"` in `icrc3_supported_block_types`.

2. **Always advertise all block types the archive can store**: Since the archive is a passive store for whatever the ledger sends, the simplest fix is to unconditionally include `"122mint"` and `"122burn"` in the archive's `icrc3_supported_block_types`, matching the full set of block types the ledger can produce.

---

### Proof of Concept

1. Deploy the ICRC-1 ledger with `icrc152: true`.
2. Call `icrc152_mint` enough times to exceed the archive trigger threshold (`ARCHIVE_TRIGGER_THRESHOLD`).
3. The ledger spawns an archive and sends the `"122mint"` blocks to it.
4. Call `icrc3_supported_block_types` on the archive canister.
5. Observe that the response contains `["1burn", "1mint", "1xfer", "2approve", "2xfer", "107feecol"]` — `"122mint"` and `"122burn"` are absent.
6. Call `icrc3_get_blocks` on the archive for the same range.
7. Observe that the returned blocks have `btype: "122mint"` — the archive stores them but does not advertise them, violating ICRC-3 compliance. [2](#0-1) [7](#0-6)

#[query]
fn icrc3_get_blocks(reqs: Vec<GetBlocksRequest>) -> GetBlocksResult {
    const MAX_BLOCKS_PER_RESPONSE: u64 = 100;

    let mut blocks = vec![];
    for req in reqs {
        let mut id = req.start.clone();
        let (start, length) = req
            .as_start_and_length()
            .unwrap_or_else(|msg| ic_cdk::api::trap(&msg));
        let max_length = MAX_BLOCKS_PER_RESPONSE.saturating_sub(blocks.len() as u64);
        if max_length == 0 {
            break;
        }
        let length = length.min(max_length);
        let decoded_block_range = decode_block_range(start, length, decode_icrc1_block);
        for block in decoded_block_range {
            blocks.push(BlockWithId {
                id: id.clone(),
                block: ICRC3Value::from(block),
            });
            id += 1_u64;
        }
    }
    GetBlocksResult {
        // We return the local log length because the archive
        // knows only about its local blocks.
        log_length: candid::Nat::from(with_blocks(|blocks| blocks.len())),
        blocks,
        archived_blocks: vec![],
    }
}

fn test_authorized_mint_and_burn_indexing() {
    let env = &StateMachine::new();
    let ledger_id = install_icrc3_test_ledger(env);
    let index_id = install_index_ng(env, index_init_arg_without_interval(ledger_id));

    let account_1 = account(1, 0);
    let account_2 = account(2, 0);
    let caller = PrincipalId::new_user_test_id(99);

    // Block 0: regular mint to give account_1 initial balance
    let block0 = BlockBuilder::new(0, 1000)
        .mint(account_1, Tokens::from(10_000_000_u64))
        .build();
    assert_eq!(
        Nat::from(0_u64),
        add_block(env, ledger_id, &block0).expect("failed to add block 0")
    );

    // Block 1: authorized mint to account_2 (ICRC-152 style, with caller/mthd/ts)
    let block1 = BlockBuilder::new(1, 2000)
        .authorized_mint(account_2, Tokens::from(5_000_000_u64))
        .with_caller(caller.0)
        .with_mthd("152mint".to_string())
        .with_created_at_time(1500)
        .build();
    assert_eq!(
        Nat::from(1_u64),
        add_block(env, ledger_id, &block1).expect("failed to add block 1")
    );

    // Block 2: authorized burn from account_1
    let block2 = BlockBuilder::new(2, 3000)
        .authorized_burn(account_1, Tokens::from(3_000_000_u64))
        .with_caller(caller.0)
        .with_mthd("152burn".to_string())
        .with_created_at_time(2500)
        .with_reason("compliance".to_string())
        .build();
    assert_eq!(
        Nat::from(2_u64),
        add_block(env, ledger_id, &block2).expect("failed to add block 2")
    );

    wait_until_sync_is_completed(env, index_id, ledger_id);

    // Verify balances
    // account_1: 10_000_000 (mint) - 3_000_000 (authorized burn) = 7_000_000
    assert_eq!(icrc1_balance_of(env, index_id, account_1), 7_000_000);
    // account_2: 5_000_000 (authorized mint)
    assert_eq!(icrc1_balance_of(env, index_id, account_2), 5_000_000);

    // Verify account_2 transactions (should have the authorized mint)
    let txs = get_account_transactions(env, index_id, account_2, None, u64::MAX);
    assert_eq!(txs.transactions.len(), 1);
    assert_eq!(txs.transactions[0].id, Nat::from(1_u64));
    let tx = &txs.transactions[0].transaction;
    assert_eq!(tx.kind, "122mint");
    assert!(tx.authorized_mint.is_some());

    // Verify account_1 transactions (should have the mint and the authorized burn)
    let txs = get_account_transactions(env, index_id, account_1, None, u64::MAX);
    assert_eq!(txs.transactions.len(), 2);
    // Transactions are in descending order
    assert_eq!(txs.transactions[0].id, Nat::from(2_u64));
    assert_eq!(txs.transactions[0].transaction.kind, "122burn");
    assert!(txs.transactions[0].transaction.authorized_burn.is_some());
    assert_eq!(txs.transactions[1].id, Nat::from(0_u64));
    assert_eq!(txs.transactions[1].transaction.kind, "mint");
