### Title
Stale ICP Total Supply Snapshot Used for Voting Reward Calculation Due to Async Interleave - (File: rs/nns/governance/src/timer_tasks/calculate_distributable_rewards.rs)

---

### Summary

The NNS Governance canister's `CalculateDistributableRewardsTask` fetches the ICP total supply from the ledger via an async inter-canister call, then uses that stale snapshot to calculate voting rewards. Because the IC execution model allows other messages to execute during the `await` point, the actual ICP supply at the time `distribute_voting_rewards_to_neurons` is called may differ from the snapshot, causing the reward pool to be calculated against an incorrect supply figure. This is the direct IC analog of the Beedle `update()` ordering bug.

---

### Finding Description

In `rs/nns/governance/src/timer_tasks/calculate_distributable_rewards.rs`, the `CalculateDistributableRewardsTask::execute` method performs the following sequence:

```rust
// Step 1: fetch supply — async inter-canister call to ICP ledger
let total_supply = self
    .governance
    .with_borrow(|governance| governance.get_ledger())
    .total_supply()   // <-- await point; other messages can execute here
    .await;

// Step 2: use the snapshot to compute and commit rewards
match total_supply {
    Ok(total_supply) => {
        self.governance.with_borrow_mut(|governance| {
            governance.distribute_voting_rewards_to_neurons(total_supply);
        });
    }
    ...
}
``` [1](#0-0) 

Between the `.total_supply().await` call and the subsequent `distribute_voting_rewards_to_neurons(total_supply)` call, the IC scheduler can interleave any number of other update messages against the governance canister or the ICP ledger. Specifically:

- A user can call `disburse_neuron`, which transfers ICP out of a neuron subaccount and reduces the effective circulating supply.
- A user can stake a new neuron (transfer ICP into a neuron subaccount), increasing the supply.
- The CMC can mint new ICP (e.g., for node provider rewards), increasing total supply.

The reward pool is computed as:

```
total_available_e8s = supply.get_e8s() * fraction
``` [2](#0-1) 

where `supply` is the stale snapshot. If the real supply at distribution time is higher (e.g., due to minting between the snapshot and the distribution), neurons receive fewer rewards than they are entitled to. If the real supply is lower (e.g., due to burns), neurons receive more than entitled.

The SNS governance canister has the same pattern in `run_periodic_tasks`:

```rust
if should_distribute_rewards {
    match self.ledger.total_supply().await {   // <-- await point
        Ok(supply) => {
            self.distribute_rewards(supply);   // uses stale snapshot
        }
        ...
    }
}
``` [3](#0-2) 

The SNS `distribute_rewards` uses the same supply-as-principal formula:

```rust
result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
``` [4](#0-3) 

---

### Impact Explanation

The reward pool distributed to all voting neurons for a given day is proportional to the ICP total supply snapshot. If the snapshot is stale:

- **Supply inflated between snapshot and distribution** (e.g., CMC mints ICP for node providers): the reward pool is over-estimated; neurons receive more maturity than the protocol intends, violating the economic model.
- **Supply deflated between snapshot and distribution** (e.g., large disburse or burn): the reward pool is under-estimated; all voting neurons receive less maturity than they are entitled to for that round. The shortfall is not rolled over — it is permanently lost.

Since the ICP ledger's `total_supply` is a query call that returns the live supply at the moment of the call, and the governance canister's reward distribution is a separate update message that executes after the await, the two values are not guaranteed to be consistent. The magnitude of the discrepancy scales with the amount of ICP minted or burned in the window between the two execution points.

---

### Likelihood Explanation

This is a structural property of the IC execution model: every `await` on an inter-canister call is a yield point. The CMC mints ICP for node provider rewards monthly, and individual users disburse neurons continuously. The window between `total_supply().await` and `distribute_voting_rewards_to_neurons` is always present in every reward distribution cycle (daily for NNS, per-round for SNS). The discrepancy is small per event but accumulates over time and is not correctable after the fact because the reward event is marked as settled.

---

### Recommendation

Fetch the total supply and immediately use it within the same atomic execution step, or re-fetch the supply inside `distribute_voting_rewards_to_neurons` itself (which is synchronous). Alternatively, document and accept the known approximation, and add a rollover mechanism for any shortfall caused by supply changes between the snapshot and distribution.

For NNS governance, the fix would be to move the `total_supply` call inside the synchronous `distribute_voting_rewards_to_neurons` by making it accept a closure or by caching the supply at a well-defined checkpoint (e.g., end of the previous reward round) rather than at the moment of distribution.

---

### Proof of Concept

1. At time T, `CalculateDistributableRewardsTask::execute` calls `ledger.total_supply().await`. The ledger returns `S = 500_000_000 ICP`.
2. While the governance canister awaits the response, the CMC processes a node provider reward payment and mints `1_000_000 ICP`. The real supply is now `501_000_000 ICP`.
3. The governance canister resumes and calls `distribute_voting_rewards_to_neurons(S)` with the stale `S = 500_000_000`.
4. The reward pool for the day is computed as `500_000_000 * fraction` instead of `501_000_000 * fraction`.
5. All voting neurons collectively receive `1_000_000 * fraction` fewer e8s of maturity than the protocol specifies. This shortfall is not rolled over.

The same scenario applies in reverse: if a large neuron disburse occurs between steps 1 and 3, the snapshot is higher than the real supply, and neurons receive more maturity than intended. [5](#0-4) [6](#0-5)
