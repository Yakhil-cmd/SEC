### Title
Cycles Permanently Locked in SNS-WASM Canister on Failed Deployment — (`rs/nns/sns-wasm/canister/canister.rs`)

### Summary
The `deploy_new_sns` update endpoint accepts cycles from the caller via `msg_cycles_accept` before attempting SNS deployment. If deployment fails mid-way, the accepted cycles are not refunded to the caller. Additionally, the `delete_canister` cleanup path explicitly skips cycle recovery from partially-created canisters before deleting them, causing a second layer of permanent cycle loss. The code itself acknowledges this with a TODO comment.

### Finding Description

`CanisterApiImpl::accept_message_cycles` unconditionally calls `ic_cdk::api::call::msg_cycles_accept(cycles)`, which irrevocably transfers cycles from the caller's message into the SNS-WASM canister's balance. [1](#0-0) 

Once `msg_cycles_accept` is called, the IC protocol does not refund those cycles to the caller even if the enclosing update call subsequently returns an error. The `deploy_new_sns` `#[update]` handler passes `&canister_api()` (a `CanisterApiImpl`) into `SnsWasmCanister::deploy_new_sns`, which uses `accept_message_cycles` to consume the caller's attached cycles before attempting multi-step canister creation. [2](#0-1) 

When deployment fails after cycles have been accepted, the cleanup path calls `delete_canister`. That function explicitly skips cycle recovery with a TODO comment: [3](#0-2) 

The comment at line 99 reads: `// TODO(NNS1-1524) We need to collect the cycles from the canister before we delete it`. This confirms that cycles forwarded to partially-created SNS canisters via `create_canister` (which uses `call_with_payment`) are burned when those canisters are deleted without first withdrawing their balance. [4](#0-3) 

There is no withdrawal or refund mechanism exposed to the original caller anywhere in the canister interface.

### Impact Explanation

An unprivileged canister or user who calls `deploy_new_sns` and attaches cycles suffers permanent, unrecoverable cycle loss in two ways:

1. **Accepted cycles not refunded**: Cycles consumed by `msg_cycles_accept` remain in the SNS-WASM canister with no caller-accessible withdrawal path.
2. **Forwarded cycles burned on cleanup**: Cycles forwarded to partially-created SNS canisters via `call_with_payment` are destroyed when those canisters are deleted without prior cycle recovery.

Cycles are the IC's native resource/currency unit. Permanent loss of caller-supplied cycles is the direct analog of ETH being locked in a Solidity contract with no withdrawal function.

### Likelihood Explanation

`deploy_new_sns` is a multi-step async operation involving multiple management canister calls (create, install, set_controllers). Any transient subnet error, resource exhaustion, or logic rejection at any step triggers the cleanup path. The code's own TODO comment confirms the cycle-loss path is known and reachable in production. Any caller — including an SNS launch initiator — who attaches cycles to a deployment that partially fails will lose those cycles.

### Recommendation

1. Before calling `msg_cycles_accept`, check whether the full deployment is likely to succeed, or defer acceptance until after all fallible steps complete.
2. In `delete_canister`, call `deposit_cycles` or `uninstall_code` + cycle withdrawal before deletion to recover cycles from partially-created canisters (resolving TODO NNS1-1524).
3. On any error path after `accept_message_cycles`, use `ic_cdk::api::call::msg_cycles_refunded` patterns or explicitly re-send the accepted amount back to the original caller via a cycles transfer.

### Proof of Concept

1. Caller attaches 180T cycles and calls `deploy_new_sns`.
2. `accept_message_cycles` calls `msg_cycles_accept(180T)` — cycles are now in SNS-WASM's balance, not refundable.
3. `create_canister` forwards, say, 50T cycles via `call_with_payment` to create the first SNS canister.
4. A subsequent step (e.g., `install_wasm` or `set_controllers`) fails.
5. Cleanup calls `delete_canister` on the partially-created canister — the 50T cycles inside it are burned (TODO NNS1-1524 not fixed).
6. The remaining ~130T cycles stay in SNS-WASM with no mechanism for the original caller to retrieve them.
7. Net result: caller loses all 180T cycles permanently. [5](#0-4) [1](#0-0)

    async fn create_canister(

        target_subnet: SubnetId,
        controller_id: PrincipalId,
        cycles: Cycles,

        let result: CallResult<(CanisterIdRecord,)> = ic_cdk::api::call::call_with_payment(

                sender_canister_version: Some(ic_cdk::api::canister_version()),
            },),
            cycles.get().try_into().unwrap(),
        )
        .await;

        result
            .map_err(handle_call_error(format!(
                "Creating canister in subnet {target_subnet} failed"
            )))
            .map(|record| record.0.get_canister_id())
    }

        // Try to stop the canister first
        self.stop_canister(canister).await?;

        // TODO(NNS1-1524) We need to collect the cycles from the canister before we delete it
        let response: CallResult<()> = ic_cdk::call(
            CanisterId::ic_00().get().0,
            "delete_canister",
            (CanisterIdRecord::from(canister),),
        )
        .await;

        response.map_err(handle_call_error(format!(
            "Failed to delete canister {canister}"
        )))
    }

    fn accept_message_cycles(&self, cycles: Option<u64>) -> Result<u64, String> {
        let cycles = cycles.unwrap_or_else(ic_cdk::api::call::msg_cycles_available);
        self.message_has_enough_cycles(cycles)?;

        let accepted = ic_cdk::api::call::msg_cycles_accept(cycles);
