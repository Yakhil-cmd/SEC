### Title
VetKD Transport Public Key Identity Point Accepted, Enabling Trivial Key Recovery - (`rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs`, `rs/crypto/internal/crypto_service_provider/src/vault/local_csp_vault/vetkd/mod.rs`)

---

### Summary

`TransportPublicKey::deserialize` accepts the BLS12-381 G1 identity point (point at infinity) as a valid transport public key. When the identity is used in `EncryptedKeyShare::create`, the encryption term vanishes and `c3` becomes the raw BLS signature of the derived key material. An attacker who supplies the identity point as `transport_public_key` can recover the vetKD derived key directly from the ciphertext without possessing any transport secret key.

---

### Finding Description

**Call chain:**

`create_encrypted_key_share` → `create_encrypted_key_share_internal` → `vault.create_encrypted_vetkd_key_share` → `create_encrypted_vetkd_key_share_internal` → `TransportPublicKey::deserialize` → `G1Affine::deserialize` → `ic_bls12_381::G1Affine::from_compressed`

**Step 1 – No identity-point guard in `TransportPublicKey::deserialize`:** [1](#0-0) 

`TransportPublicKey::deserialize` delegates entirely to `G1Affine::deserialize`. There is no subsequent check that the resulting point is non-identity.

**Step 2 – `G1Affine::deserialize` accepts the identity:** [2](#0-1) 

`from_compressed` is documented to verify prime-order subgroup membership. The identity element (encoded as `0xC0` followed by 47 zero bytes) **is** in every subgroup — it is the neutral element — so `from_compressed` returns `CtOption::some(G1Affine::identity())` and deserialization succeeds. Arbitrary low-order non-identity points are correctly rejected by the subgroup check; the identity is the sole exception.

**Step 3 – Encryption degenerates when `transport_pk` is the identity:** [3](#0-2) 

With `transport_pk = identity`:

```
c1 = generator * r          (random blinding)
c2 = G2::generator * r      (random blinding)
c3 = identity * r + msg * dsk
   = identity + msg * dsk
   = msg * dsk              ← raw BLS signature, no encryption
```

`c3` is now the BLS signature of `msg` under the derived secret key `dsk`, written in plaintext inside the ciphertext.

**Step 4 – Attacker recovers the key without the transport secret:**

Normal decryption is `k = c3 − c1 * tsk`. When `tsk = 0` (the scalar corresponding to the identity public key), `k = c3 − 0 = c3`. The attacker simply reads `c3` from the returned `VetKdEncryptedKeyShare` or combined `VetKdEncryptedKey` bytes.

**Step 5 – The vault propagates the bad key without additional checks:** [4](#0-3) 

After `TransportPublicKey::deserialize` succeeds, `create_encrypted_vetkd_key_share_internal` proceeds directly to `EncryptedKeyShare::create` with no further validation. [5](#0-4) 

**Step 6 – `create_encrypted_key_share_internal` passes the attacker-supplied key verbatim:** [6](#0-5) 

No validation of `args.transport_public_key` occurs before it reaches the vault.

---

### Impact Explanation

The vetKD derived key is the output of the BLS threshold signature scheme applied to the canister's derivation context and input. If the attacker recovers this key, they can decrypt any data that was encrypted to the corresponding derived public key. For chain-fusion and canister applications relying on vetKD for confidentiality (e.g., encrypted canister state, cross-chain asset keys), this constitutes a full confidentiality break for any key derived under the attacker-chosen context.

---

### Likelihood Explanation

The attack requires only a single malformed API call. The attacker is an unprivileged canister or user who submits a vetKD request with `transport_public_key = [0xC0, 0x00, …, 0x00]` (48 bytes). No privileged access, threshold corruption, or social engineering is required. The identity encoding is a well-known constant. The attack is deterministic and locally testable.

---

### Recommendation

In `TransportPublicKey::deserialize`, after successfully deserializing the G1 point, add an explicit identity check:

```rust
pub fn deserialize(bytes: &[u8]) -> Result<Self, TransportPublicKeyDeserializationError> {
    let public_key = G1Affine::deserialize(&bytes)
        .map_err(|_| TransportPublicKeyDeserializationError::InvalidPublicKey)?;
    if bool::from(public_key.is_identity()) {
        return Err(TransportPublicKeyDeserializationError::InvalidPublicKey);
    }
    Ok(Self { public_key })
}
```

This is the correct location because it is the single deserialization point used by both the vault (`create_encrypted_vetkd_key_share_internal`) and the higher-level functions (`combine_encrypted_key_shares_internal`, `verify_encrypted_key_internal`). [1](#0-0)

### Proof of Concept

```rust

// Encoding: compression bit (0x80) | infinity bit (0x40) = 0xC0, rest zeros
let identity_tpk: Vec<u8> = {
    let mut v = vec![0u8; 48];
    v[0] = 0xC0;
    v
};

// Verify it passes deserialization (it will)
assert!(TransportPublicKey::deserialize(&identity_tpk).is_ok());

// Submit vetKD request with identity_tpk as transport_public_key.
// The returned VetKdEncryptedKeyShare has:
//   c3 = msg * dsk   (bytes 48..144 of the 144-byte share)
// The attacker reads c3 directly — that IS the derived vetKD key.
// No transport secret key needed.
```

The existing test at `rs/crypto/tests/vetkd.rs:256` only tests a *corrupted* encoding (XOR of 0x40 onto a non-identity point, producing an inconsistent encoding), not the properly-encoded identity point `[0xC0, 0x00, …]`. [7](#0-6)
