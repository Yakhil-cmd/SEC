### Title
Legacy `function_followee_index` Not Cleaned Up When `set_following` Removes Legacy Following — (`rs/sns/governance/src/governance.rs`)

---

### Summary

In `Governance::set_following`, when legacy function-based following is removed from a neuron's `followees` map, the corresponding entries in `function_followee_index` are never actually removed. The helper `remove_neuron_from_function_followee_index_for_function` reads `neuron.followees.get(&function)` to discover which followees to evict from the index, but `neuron.followees.remove(function)` is called **first**, so the helper finds nothing and silently returns. The stale index entries cause the neuron to continue automatically casting votes alongside its old followees on every subsequent proposal, even though the neuron owner believes the following relationship has been severed.

---

### Finding Description

`Governance::set_following` (SNS) is the new topic-based following command. As part of its contract, it is also supposed to clear any **legacy** function-based following that falls under the topics being modified. The relevant code block is:

```rust
// Fifth, remove any legacy following (based on individual proposal types under the topics
// that were modified by this command).
for topic in &mentioned_topics {
    let native_functions = topic.native_functions();
    let custom_functions = GovernanceProto::get_custom_functions_for_topic(...);
    for function in native_functions.union(&custom_functions) {
        neuron.followees.remove(function);                          // ← (A) primary map updated

        legacy::remove_neuron_from_function_followee_index_for_function(
            &mut self.function_followee_index,
            neuron,
            *function,
        );                                                          // ← (B) index update attempted
    }
}
``` [1](#0-0) 

Step (A) removes the function key from `neuron.followees`. Step (B) then calls `remove_neuron_from_function_followee_index_for_function`, whose first substantive action is:

```rust
let Some(followees) = neuron.followees.get(&function) else {
    return;   // ← early return: entry was already removed in step (A)
};
``` [2](#0-1) 

Because step (A) already deleted the key, `neuron.followees.get(&function)` returns `None`, the function returns immediately, and the `function_followee_index` is **never updated**. The same ordering defect is repeated for the catch-all function:

```rust
neuron.followees.remove(&catchall_function);          // ← primary map updated first

legacy::remove_neuron_from_function_followee_index_for_function(
    &mut self.function_followee_index,
    neuron,
    catchall_function,
);                                                    // ← index update is a no-op
``` [3](#0-2) 

The `function_followee_index` is a `BTreeMap<u64, BTreeMap<String, BTreeSet<NeuronId>>>` that maps `function_id → followee_id → set of follower neuron IDs`. It is the authoritative reverse index used during vote cascading to find all neurons that should automatically vote when a followee votes. [4](#0-3) 

The correct pattern — used in the topic-based index update just above — is to remove from the index **before** mutating the neuron's primary data:

```rust
// Second, remove the neuron from the follower index, which needs to be done before
// replacing the topic followees.
remove_neuron_from_follower_index(&mut self.topic_follower_index, neuron);

// Third, save the new followees.
neuron.topic_followees.replace(new_topic_followees);
``` [5](#0-4) 

The legacy cleanup block inverts this order, making the index removal a no-op identical in structure to the H-11 `removeDelegation` bug: the function operates on the wrong state (already-mutated neuron data) and therefore does nothing.

---

### Impact Explanation

The `function_followee_index` is consumed by `cast_vote_and_cascade_follow` to automatically propagate votes from a followee to all its registered followers. [6](#0-5) 

Because stale entries remain in the index after `set_following`, a neuron that has called `set_following` to remove its legacy following will **continue to automatically vote** alongside its old followees on every proposal whose action falls under the affected functions. The neuron owner has no way to prevent this: the primary `neuron.followees` map is correctly cleared (so re-reading the neuron shows no following), but the cascade logic uses only the index, not the primary map. The discrepancy is invisible to the neuron owner and cannot be corrected without a governance upgrade.

This allows a followee neuron to permanently retain delegated voting power from any follower that has attempted to remove legacy following via `set_following`, enabling governance manipulation: a well-positioned followee can accumulate and hold voting power from neurons that believe they have opted out.

---

### Likelihood Explanation

`SetFollowing` is the actively promoted new command for SNS neurons (documented in the SNS governance changelog as of Proposal 136227). Any SNS neuron owner who migrates from legacy function-based following to topic-based following by calling `SetFollowing` triggers this bug. The entry path is a standard unprivileged `manage_neuron` ingress call — no special role or key is required. The bug fires on every invocation of `set_following` where the neuron has pre-existing legacy followees for the mentioned topics. [7](#0-6) 

---

### Recommendation

Swap the order of operations in the legacy cleanup loop so that the index is updated **before** the primary map is mutated, mirroring the correct pattern used for topic-based following:

```diff
 for function in native_functions.union(&custom_functions) {
-    neuron.followees.remove(function);
-
-    legacy::remove_neuron_from_function_followee_index_for_function(
+    legacy::remove_neuron_from_function_followee_index_for_function(
         &mut self.function_followee_index,
         neuron,
         *function,
     );
+
+    neuron.followees.remove(function);
 }
```

Apply the same fix to the catch-all block:

```diff
+legacy::remove_neuron_from_function_followee_index_for_function(
+    &mut self.function_followee_index,
+    neuron,
+    catchall_function,
+);
 neuron.followees.remove(&catchall_function);
-
-legacy::remove_neuron_from_function_followee_index_for_function(
-    &mut self.function_followee_index,
-    neuron,
-    catchall_function,
-);
```

### Proof of Concept

1. SNS neuron A has legacy following: `followees = { function_id_X: [neuron_B] }`.
2. Neuron A's owner calls `manage_neuron` → `SetFollowing` with a topic that covers `function_id_X` and an empty followees list for that topic.
3. Inside `set_following`:
   - `neuron.followees.remove(&function_id_X)` executes — `neuron.followees` is now `{}`.
   - `remove_neuron_from_function_followee_index_for_function(index, neuron, function_id_X)` executes — `neuron.followees.get(&function_id_X)` returns `None` → early return. Index unchanged.
4. `function_followee_index[function_id_X][neuron_B]` still contains neuron A's ID.
5. Neuron B votes Yes on proposal P (action type `function_id_X`).
6. `cast_vote_and_cascade_follow` looks up `function_followee_index[function_id_X][neuron_B]` → finds neuron A → automatically casts neuron A's vote as Yes.
7. Neuron A's owner observes their neuron voted Yes despite having removed the following relationship. [8](#0-7) [1](#0-0)

    pub fn remove_neuron_from_function_followee_index_for_function(
        index: &mut FollowerIndex,
        neuron: &Neuron,
        function: u64,
    ) {
        let Some(neuron_id) = neuron.id.as_ref() else {
            log!(ERROR, "Neuron {:?} does not have an ID!", neuron);
            return;
        };

        let Some(followees) = neuron.followees.get(&function) else {
            return;
        };

        let Some(followee_index) = index.get_mut(&function) else {
            return;
        };

        for followee in followees.followees.iter() {
            let nid = followee.to_string();

            if let Some(followee_set) = followee_index.get_mut(&nid) {
                followee_set.remove(neuron_id);

                if followee_set.is_empty() {
                    followee_index.remove(&nid);
                }
            }
        }
    }
