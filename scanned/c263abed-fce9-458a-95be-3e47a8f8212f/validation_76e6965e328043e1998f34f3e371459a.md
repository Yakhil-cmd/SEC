### Title
SNS Swap Canister Silently Locks Excess ICP in Buyer Subaccounts When Participation Is Capped - (`rs/sns/swap/src/swap.rs`)

### Summary
The SNS Swap canister's `refresh_buyer_token_e8s` function caps the accepted ICP participation amount at both the per-participant maximum (`max_participant_icp_e8s`) and the global remaining capacity (`available_direct_participation_e8s`). When a buyer transfers more ICP than either cap allows, the excess ICP is silently left in the buyer's subaccount of the Swap canister and cannot be recovered until the swap reaches a terminal state (COMMITTED or ABORTED). During an active (OPEN) swap — which can last days or weeks — the excess funds are inaccessible to the user. The codebase itself acknowledges this with an open TODO.

### Finding Description

In `refresh_buyer_token_e8s`, after reading the buyer's subaccount balance (`e8s`), the function computes the accepted amount by applying two caps:

```rust
let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);
let new_balance_e8s = old_amount_icp_e8s.saturating_add(actual_increment_e8s);
// ...
let new_balance_e8s = std::cmp::min(new_balance_e8s, max_participant_icp_e8s);
```

Only `new_balance_e8s` is recorded in `BuyerState`. The difference `e8s - new_balance_e8s` remains in the subaccount but is never returned to the buyer during the OPEN state. The function returns `icp_accepted_participation_e8s: new_balance_e8s` and `icp_ledger_account_balance_e8s: e8s`, making the discrepancy visible in the response but taking no corrective action.

The code itself contains an explicit acknowledgment of this gap:

The only recovery path is `error_refund_icp`, which is gated behind the swap reaching ABORTED or COMMITTED state:

```rust
if !(self.lifecycle() == Lifecycle::Aborted || self.lifecycle() == Lifecycle::Committed) {
    return ErrorRefundIcpResponse::new_precondition_error(
        "Error refunds can only be performed when the swap is ABORTED or COMMITTED",
    );
}
```

During `sweep_icp`, only `amount_e8s` (the capped accepted amount) is transferred out of the subaccount — the excess remains behind, recoverable only post-finalization via `error_refund_icp`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

Any buyer who transfers more ICP than the per-participant cap or the remaining global cap has their excess ICP locked in the Swap canister's subaccount for the entire duration of the swap. SNS swaps can run for days or weeks. The locked amount is not dust — it is the full difference between what was transferred and what was accepted. For example, if only 2 ICP of global capacity remains and a buyer sends 6 ICP, 4 ICP is locked until finalization. The buyer cannot use, transfer, or stake those funds during this period. This is a ledger conservation bug: tokens committed to the protocol exceed what the protocol actually uses, and the excess is not promptly returned. [5](#0-4) 

### Likelihood Explanation

This is reachable by any unprivileged ingress sender. The scenario is common: a buyer who sends the maximum per-participant amount when the swap is nearly full, or any buyer who sends more than `max_participant_icp_e8s`. The `test_min_max_icp_per_buyer` and `test_max_icp` unit tests explicitly demonstrate the capping behavior, confirming it is a reachable code path on every swap. The integration test in `sns_lifecycle.rs` also documents "Case C" where excess ICP is expected to be refunded only after finalization. [6](#0-5) [7](#0-6) 

### Recommendation

Implement the acknowledged TODO (NNS1-1682): after capping the accepted amount in `refresh_buyer_token_e8s`, immediately attempt to refund the excess ICP (`e8s - new_balance_e8s`) back to the buyer's principal account on the ICP ledger. If the refund transfer fails, the current behavior (leaving funds in the subaccount for later `error_refund_icp` recovery) can serve as a fallback, but the attempt should be made proactively during the OPEN state rather than requiring the user to wait for swap finalization. [8](#0-7) 

### Proof of Concept

1. A swap is configured with `max_participant_icp_e8s = 5 * E8` and `max_direct_participation_icp_e8s = 10 * E8`.
2. Buyer A transfers 6 ICP to `subaccount(swap_canister, buyer_A)` on the ICP ledger.
3. Buyer A calls `refresh_buyer_tokens`. The function reads balance = 6 ICP, caps to 5 ICP, records `BuyerState { amount_e8s: 5 * E8 }`. Returns `icp_accepted_participation_e8s: 5 * E8, icp_ledger_account_balance_e8s: 6 * E8`.
4. 1 ICP remains in `subaccount(swap_canister, buyer_A)` on the ICP ledger.
5. Buyer A calls `error_refund_icp` → receives `"Error refunds can only be performed when the swap is ABORTED or COMMITTED"`.
6. The 1 ICP is inaccessible until the swap finalizes. If the swap runs for 14 days, the funds are locked for 14 days.

This is confirmed by the existing test `test_min_max_icp_per_buyer` which shows 6 ICP transferred but only 5 ICP accepted, with no automatic refund of the 1 ICP excess. [9](#0-8) [10](#0-9)
