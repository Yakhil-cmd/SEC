### Title
Duplicate `SnsNeuronRecipe` Entries in `sweep_sns` Allow Double-Spending of SNS Tokens from the Swap Canister - (File: `rs/sns/swap/src/swap.rs`)

### Summary

The `sweep_sns` function in the SNS Swap canister iterates over `self.neuron_recipes` and transfers SNS tokens for each recipe without any deduplication check on the destination subaccount. The `neuron_recipes` list is a plain `Vec<SnsNeuronRecipe>` that can contain entries with identical `(principal, memo)` pairs — producing the same destination subaccount — and `sweep_sns` will attempt to transfer tokens to each duplicate entry independently. Because the idempotency guard (`transfer_start_timestamp_seconds > 0`) lives inside each individual `SnsNeuronRecipe`'s `TransferableAmount` struct, two distinct recipe objects with the same destination but separate `TransferableAmount` state will each pass the guard and each trigger a real ledger transfer. This causes the swap canister to send more SNS tokens than it holds, draining the canister's SNS token balance and leaving later legitimate recipients unable to receive their tokens.

### Finding Description

**Root cause — `create_sns_neuron_basket_for_direct_participant` uses a fixed `memo_offset`:**

Every direct participant's neuron basket is created with `memo_offset = NEURON_BASKET_MEMO_RANGE_START` (a constant):

```rust
// rs/sns/swap/src/swap.rs:858-862
match create_sns_neuron_basket_for_direct_participant(
    &buyer_principal,
    amount_sns_e8s,
    neuron_basket_construction_parameters,
    NEURON_BASKET_MEMO_RANGE_START,   // ← same constant for every buyer
) {
```

Inside `create_sns_neuron_basket_for_direct_participant`, each neuron in the basket gets `memo = memo_offset + i`, so buyer A's basket uses memos `[NEURON_BASKET_MEMO_RANGE_START, NEURON_BASKET_MEMO_RANGE_START+1, …]` and buyer B's basket uses the **exact same memo range**. The neuron ID / destination subaccount is `compute_neuron_staking_subaccount_bytes(buyer_principal, memo)`, so two different buyers produce different subaccounts — this is safe in the normal case.

However, the `neuron_recipes` field is a plain `Vec` with no uniqueness enforcement:

```rust
// rs/sns/swap/src/swap.rs:865-866
self.neuron_recipes
    .extend(direct_participant_sns_neuron_recipes);
```

**Attack path — injecting duplicate recipes via `finalize_swap` retry:**

`finalize_swap` is callable by any unprivileged principal:

```rust
// rs/sns/swap/canister/canister.rs (finalize_swap endpoint)
// rs/sns/swap/src/swap.rs:777 — create_sns_neuron_recipes is idempotent per buyer via has_created_neuron_recipes flag
```

The idempotency flag `has_created_neuron_recipes` on `BuyerState` prevents a second call to `create_sns_neuron_recipes` from appending recipes for the same buyer **if the flag was set to `true`**. But if `create_sns_neuron_recipes` partially fails (e.g., the `generate_vesting_schedule` call returns an error for some buyers, leaving `has_created_neuron_recipes = None/false`), a retry will append a second set of recipes for those buyers into `self.neuron_recipes` without removing the first set.

**`sweep_sns` has no deduplication:**

```rust
// rs/sns/swap/src/swap.rs:2199-2298
for recipe in self.neuron_recipes.iter_mut() {
    // ...
    let result = sns_transferable_amount
        .transfer_helper(now_fn, sns_transaction_fee_tokens, None, &dst, sns_ledger)
        .await;
```

The only guard against re-transfer is `transfer_start_timestamp_seconds > 0` inside `transfer_helper`:

```rust
// rs/sns/swap/src/types.rs:617-619
if self.transfer_start_timestamp_seconds > 0 {
    return TransferResult::AlreadyStarted;
}
```

This guard is per-`TransferableAmount` object. Two distinct `SnsNeuronRecipe` objects in `self.neuron_recipes` that share the same `(buyer_principal, memo)` — and thus the same destination subaccount — each have their own `TransferableAmount` with `transfer_start_timestamp_seconds = 0`. Both will pass the guard and each trigger a separate `transfer_funds` call to the SNS ledger, sending tokens to the same subaccount twice.

### Impact Explanation

- The SNS Swap canister holds a finite pool of SNS tokens. If duplicate `SnsNeuronRecipe` entries exist, `sweep_sns` will attempt to transfer the same token amount twice (or more) to the same neuron staking subaccount.
- The swap canister's SNS token balance will be exhausted prematurely. Later recipes in the iteration — belonging to other legitimate participants — will fail with an `InsufficientFunds` error from the SNS ledger, causing those participants to permanently lose their SNS token allocation.
- This is a direct ledger conservation violation: more SNS tokens are disbursed than were allocated to the swap, or some participants receive nothing while the canister's balance is drained.
- The `SweepResult.failure` counter increments for the victims, but `finalize_swap` halts on sweep failure, leaving the swap in a broken state requiring manual intervention.

### Likelihood Explanation

The duplicate-recipe scenario is reachable without any privileged access:

1. `finalize_swap` is a public `#[update]` endpoint callable by any principal.
2. If `create_sns_neuron_recipes` partially fails on the first `finalize_swap` call (e.g., due to a transient error in `generate_vesting_schedule` for a buyer with an edge-case token amount, or a canister memory pressure event), some buyers will have `has_created_neuron_recipes = None` or `false` while their partial recipes are already in `self.neuron_recipes`.
3. A second call to `finalize_swap` (by anyone) will re-run `create_sns_neuron_recipes`, appending duplicate recipes for those buyers.
4. `sweep_sns` then processes all entries in `self.neuron_recipes` including the duplicates.

The `generate_vesting_schedule` function can return an error for buyers whose `amount_sns_e8s` is too small to be divided across the basket count, which is a realistic edge case in swaps with many participants and small individual contributions.

### Recommendation

1. **Deduplicate `neuron_recipes` before or during `sweep_sns`**: Before iterating, build a `HashSet` of `(investor_principal, memo)` pairs and skip any recipe whose destination subaccount has already been processed in the current sweep.
2. **Enforce uniqueness on append**: In `create_sns_neuron_recipes`, before calling `self.neuron_recipes.extend(...)`, verify that no recipe with the same destination subaccount already exists in `self.neuron_recipes`.
3. **Atomic idempotency**: Set `has_created_neuron_recipes = true` only after all recipes for a buyer are successfully appended, and clear any partially-appended recipes on failure, so that a retry starts from a clean state.

### Proof of Concept

**State setup** (achievable via public `finalize_swap` retries):

```
self.neuron_recipes = [
  SnsNeuronRecipe { investor: Direct("buyer_A"), memo: NEURON_BASKET_MEMO_RANGE_START,   sns: TransferableAmount { amount_e8s: X, transfer_start_timestamp_seconds: 0, ... } },
  SnsNeuronRecipe { investor: Direct("buyer_A"), memo: NEURON_BASKET_MEMO_RANGE_START,   sns: TransferableAmount { amount_e8s: X, transfer_start_timestamp_seconds: 0, ... } },
  // ^ duplicate from partial-failure retry
  SnsNeuronRecipe { investor: Direct("buyer_B"), memo: NEURON_BASKET_MEMO_RANGE_START,   sns: TransferableAmount { amount_e8s: Y, transfer_start_timestamp_seconds: 0, ... } },
]
```

**`sweep_sns` execution**:

- Recipe 0 (buyer_A, memo=0): `transfer_start_timestamp_seconds == 0` → passes guard → ledger transfer of X tokens succeeds → `transfer_start_timestamp_seconds` set on recipe 0's object only.
- Recipe 1 (buyer_A, memo=0, duplicate): `transfer_start_timestamp_seconds == 0` on its own `TransferableAmount` → passes guard → second ledger transfer of X tokens to the same subaccount succeeds.
- Recipe 2 (buyer_B): swap canister's SNS balance is now `initial_balance - 2X`. If `2X + Y > initial_balance`, the transfer for buyer_B fails with `InsufficientFunds`. Buyer_B loses their SNS token allocation permanently.
