### Title
Unbounded SNS Ledger Transfer Fee via `ManageLedgerParameters` Allows Token Holders to Be Griefed - (File: rs/sns/governance/src/proposal.rs)

### Summary

The SNS governance `ManageLedgerParameters` proposal action allows an SNS community to set the SNS ledger's `transfer_fee` to any arbitrary `u64` value, including values equal to or exceeding the total token supply. The validation function `validate_and_render_manage_ledger_parameters` performs **no upper-bound check** on `transfer_fee`. Once a malicious or compromised SNS governance majority passes such a proposal, every subsequent `icrc1_transfer` or `icrc2_transfer_from` on the SNS ledger will deduct the inflated fee, making token transfers economically impossible or causing the entire token balance to be consumed as fees. This is the IC analog of the "front-run deposits to steal pool tokens" class: a privileged actor (SNS governance majority) can change a fee parameter without any ceiling or delay, directly harming unprivileged token holders who have no slippage/minimum-received protection.

### Finding Description

`ManageLedgerParameters` is a native SNS governance proposal action (Id = 13) that upgrades the SNS ledger canister with new parameters, including `transfer_fee: Option<u64>`.

The validation path in `rs/sns/governance/src/proposal.rs` at `validate_and_render_manage_ledger_parameters` (lines 1761–1799) only checks that **at least one field is set**; it applies no ceiling to `transfer_fee`:

```rust
if let Some(transfer_fee) = transfer_fee {
    render += &format!("# Set token transfer fee: {transfer_fee} token-quantums. \n",);
    change = true;
}
``` [1](#0-0) 

No maximum value is enforced. Any `u64` value — including `u64::MAX` — passes validation and is forwarded to the ledger upgrade.

The execution path in `rs/sns/governance/src/governance.rs` at `perform_manage_ledger_parameters` (lines 3090–3212) directly converts the `ManageLedgerParameters` into `LedgerUpgradeArgs` and upgrades the ledger canister:

```rust
let ledger_upgrade_arg = candid::encode_one(Some(LedgerArgument::Upgrade(Some(
    UpgradeArgs::from(manage_ledger_parameters.clone()),
))))
.unwrap();
self.upgrade_non_root_canister(
    ledger_canister_id,
    Wasm::Bytes(ledger_wasm),
    ledger_upgrade_arg,
    CanisterInstallMode::Upgrade,
)
.await?;
``` [2](#0-1) 

The ICRC-1 ledger's `upgrade` function in `rs/ledger_suite/icrc1/ledger/src/lib.rs` (lines 927–933) accepts any `transfer_fee` value without bounds checking:

```rust
if let Some(transfer_fee) = args.transfer_fee {
    self.transfer_fee = Tokens::try_from(transfer_fee.clone()).unwrap_or_else(|e| {
        ic_cdk::trap(...)
    });
}
``` [3](#0-2) 

After the upgrade, every `icrc1_transfer` and `icrc2_transfer_from` call deducts the new fee from the sender's balance. If the fee equals or exceeds the token supply, all transfers fail with `InsufficientFunds`.

The `NervousSystemParameters.transaction_fee_e8s` is also updated to match the new fee after the ledger upgrade succeeds:

```rust
if let Some(nervous_system_parameters) = self.proto.parameters.as_mut()
    && let Some(transfer_fee) = manage_ledger_parameters.transfer_fee
{
    nervous_system_parameters.transaction_fee_e8s = Some(transfer_fee);
}
``` [4](#0-3) 

There is no time-lock, no delay mechanism, and no upper-bound ceiling on the `transfer_fee` field in `ManageLedgerParameters`. [5](#0-4) 

### Impact Explanation

- **Token transfers permanently broken**: Setting `transfer_fee = u64::MAX` makes every `icrc1_transfer` and `icrc2_transfer_from` fail with `InsufficientFunds`, effectively freezing all token movement.
- **Neuron staking/disbursing broken**: SNS neuron staking and disbursing rely on ledger transfers; an inflated fee breaks these operations, locking staked tokens.
- **Swap/treasury operations broken**: Any SNS treasury transfer or swap finalization that calls the SNS ledger will fail.
- **No user-side protection**: Token holders have no "minimum received" or slippage parameter to protect against a fee hike between their intent and execution.

### Likelihood Explanation

This requires an SNS governance majority (>50% of voting power) to pass a `ManageLedgerParameters` proposal with a malicious `transfer_fee`. This is a **governance authorization bug** — the attacker role is a malicious SNS governance majority, which is a realistic threat model for SNS DAOs with concentrated token holdings, compromised neuron keys, or low participation. The SNS framework explicitly allows any token holder with sufficient stake to submit and pass such proposals. No admin key, threshold crypto compromise, or external dependency is required.

### Recommendation

1. **Add an upper-bound ceiling** on `transfer_fee` in `validate_and_render_manage_ledger_parameters`. A reasonable ceiling would be a small fraction of `neuron_minimum_stake_e8s` (e.g., no more than 1% of the minimum neuron stake, or a hard cap such as `1_000_000_000` e8s).
2. **Add a time-lock delay** for `ManageLedgerParameters` proposals that change `transfer_fee`, analogous to the slippage protection recommended in the external report, so token holders have time to react before the fee change takes effect.
3. **Validate consistency**: Ensure the new `transfer_fee` is strictly less than `neuron_minimum_stake_e8s` (this relationship is already required at SNS init time in `rs/sns/init/src/lib.rs` but is not re-enforced on fee changes via `ManageLedgerParameters`). [6](#0-5) 

### Proof of Concept

1. An SNS DAO with a majority holder (or a coalition) submits a `ManageLedgerParameters` proposal:
   ```
   ManageLedgerParameters { transfer_fee: Some(u64::MAX), .. }
   ```
2. The proposal passes `validate_and_render_manage_ledger_parameters` — no ceiling check exists.
3. `perform_manage_ledger_parameters` upgrades the SNS ledger with `transfer_fee = u64::MAX`.
4. All subsequent `icrc1_transfer` calls on the SNS ledger return `InsufficientFunds { balance: <user_balance> }` because `balance < u64::MAX`.
5. SNS neuron disbursements, treasury transfers, and swap finalizations all fail.
6. Token holders have no recourse — there is no slippage parameter, no minimum-received check, and no time-lock that would have allowed them to exit before the fee change took effect.
