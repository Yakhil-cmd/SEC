### Title
Byzantine Dealer Out-of-Range Plaintext Chunks Bypass `verify_dealing`, Triggering Hours-Long `CheatingDealerDlogSolver` BSGS Stall on Honest Receivers — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/groth20_bls12_381/dealing.rs`)

---

### Summary

A single Byzantine dealer (below the DKG fault threshold) can craft a dealing whose ciphertext chunks contain plaintext values outside `[0, CHUNK_MAX]` yet pass every check in `verify_dealing` — including `verify_ciphertext_integrity`, the NIZK chunking proof, and the NIZK sharing proof. When any honest receiver later calls `compute_threshold_signing_key`, the `dec_chunks` function's fast-path lookup (`HonestDealerDlogLookupTable`) fails, unconditionally falling back to `CheatingDealerDlogSolver`, which performs Baby-Step Giant-Step over a ~2^40 search space with up to 256 delta iterations. The production code itself comments this "may take hours." This stalls the receiver replica for the duration of key loading, blocking subnet participation.

---

### Finding Description

**Root cause — `verify_chunking` does not enforce `[0, CHUNK_MAX]` on plaintexts.**

The NIZK chunking proof (`verify_chunking`) only checks that the response scalars `z_s` satisfy `z_s < zz`, where:

```
ss  = n * m * (CHUNK_SIZE - 1) * CHALLENGE_MASK
zz  = 2 * NUM_ZK_REPETITIONS * ss
``` [1](#0-0) 

`zz` is astronomically larger than `CHUNK_MAX = 0xFFFF`. The proof never binds the prover to use chunks in `[0, CHUNK_MAX]`; a Byzantine dealer can choose any chunk value that keeps `z_s` within `[-zz, zz)` (the prover loop retries until this holds). The `verify_dealing` call chain — `verify_all_shares_are_present_and_well_formatted` → `verify_zk_proofs` → `verify_chunking` — contains no range check on the plaintext chunk values themselves. [2](#0-1) 

**The codebase's own test confirms the bypass.**

`should_decrypt_correctly_for_cheating_dealer` explicitly constructs chunks with values `> CHUNK_MAX` via `PlaintextChunks::new_unchecked`, encrypts them, and asserts `verify_ciphertext_integrity` still returns `Ok(())`: [3](#0-2) 

**`dec_chunks` unconditionally invokes `CheatingDealerDlogSolver` on any out-of-range chunk.**

After the fast-path `HonestDealerDlogLookupTable` (which covers only `[0, 0xFFFF]`) returns `None` for any chunk, `dec_chunks` constructs a `CheatingDealerDlogSolver` and runs BSGS. The production comment is explicit: [4](#0-3) 

**`CheatingDealerDlogSolver` is designed for a ~2^40 search space.**

The solver allocates up to 2 GiB of table memory and iterates `scale_range = 1 << CHALLENGE_BITS` (= 256) delta values, each requiring a full BSGS pass over a range of `2*zz - 1 ≈ 2^40`: [5](#0-4) 

The test suite itself documents the search space: "a malicious DKG participant can force us to search around 2^40 candidates for a discrete log": [6](#0-5) 

---

### Impact Explanation

During a DKG round, the Byzantine dealer's dealing passes `verify_dealing` on all honest nodes and is included in the transcript. Every honest receiver that calls `compute_threshold_signing_key` must call `dec_chunks` for each dealing in the transcript. For the Byzantine dealing, `dec_chunks` triggers `CheatingDealerDlogSolver::new(n, m)` — allocating ~2 GiB and running hours of BSGS — **per affected chunk, per receiver**. The receiver replica is CPU-stalled and cannot participate in consensus or sign threshold signatures for the duration. With `NUM_CHUNKS = 16` chunks per dealing, a maximally malicious dealing can force 16 sequential BSGS invocations. The stall is non-volumetric: a single crafted dealing suffices.

---

### Likelihood Explanation

The attacker needs only to be a registered DKG dealer (a subnet node) — no privileged key, no governance majority, no threshold corruption. The attack requires crafting a dealing with out-of-range chunks and a valid NIZK proof, which is straightforward given that `prove_chunking` already supports this (the prover loop retries until `z_s` is in range regardless of chunk magnitude). The dealing passes all protocol-level verification. The attack is local-testable as demonstrated by the existing test `should_decrypt_correctly_for_cheating_dealer`.

---

### Recommendation

Add an explicit plaintext-range check inside `verify_chunking` or `verify_zk_proofs` that rejects any dealing whose ciphertext chunks, when decrypted, would fall outside `[0, CHUNK_MAX]`. Since the plaintext values are committed to in the ciphertext group elements, a range proof or a tighter bound on `z_s` (clamped to `n * m * CHUNK_MAX`) can enforce this without breaking the zero-knowledge property. Alternatively, `verify_dealing` can be extended to verify that the combined ciphertext group elements lie in the subgroup generated by `[g^0, g^1, ..., g^CHUNK_MAX]` — rejecting any dealing that would require `CheatingDealerDlogSolver` at decryption time.

---

#[test]
fn should_decrypt_correctly_for_cheating_dealer() {
    let epoch = Epoch::from(0);
    let rng = &mut reproducible_rng();
    let associated_data = rng.r#gen::<[u8; 10]>();

    let sys = SysParam::global();
    let key_gen_assoc_data = rng.r#gen::<[u8; 32]>();

    let nodes = 3;

    let mut keys = Vec::new();
    for _ in 0..nodes {
        let key_pair = kgen(&key_gen_assoc_data, sys, rng);
        keys.push(key_pair);
    }
    let pks: Vec<_> = keys.iter().map(|key| key.0.public_key().clone()).collect();

    let mut sij = {
        let mut sij = Vec::with_capacity(nodes);

        for _ in 0..nodes {
            let mut chunks = [0; NUM_CHUNKS];

            for i in 0..NUM_CHUNKS {
                // ensure that multiplying by delta pushes us out of Chunk range
                let chunk = (0x8000 | rng.r#gen::<u16>()) as isize;
                chunks[i] = chunk;
            }
            // this ensures that chunks is the encoding of a scalar less
            // than the group order:
            chunks[0] %= 0x73ee;

            sij.push(chunks);
        }

        sij
    };

    let cheating_i = rng.r#gen::<usize>() % nodes;
    let cheating_j = std::cmp::max(1, rng.r#gen::<usize>() % NUM_CHUNKS);

    let delta = (2 + rng.r#gen::<usize>() % 10) as isize;
    sij[cheating_i][cheating_j] *= delta; // doesn't overflow as delta is small and isize >> u16

    // however the new sij *is* larger than the maximum "legal" chunk
    assert!(sij[cheating_i][cheating_j] > CHUNK_MAX);

    let cheating_chunks = sij
        .iter()
        .map(|c| PlaintextChunks::new_unchecked(*c))
        .collect::<Vec<_>>();

    let pks_and_chunks = pks
        .iter()
        .cloned()
        .zip(cheating_chunks.iter().cloned())
        .collect::<Vec<_>>();

    let (crsz, _witness) = enc_chunks(&pks_and_chunks, epoch, &associated_data, sys, rng);

    // still a valid ciphertext
    assert!(verify_ciphertext_integrity(&crsz, epoch, &associated_data, sys).is_ok());

    for i in 0..keys.len() {
        let secret_key = &keys[i].1;
        let out = dec_chunks(secret_key, i, &crsz, epoch, &associated_data);
        assert_eq!(
            out.unwrap(),
            cheating_chunks[i].recombine_to_scalar(),
            "Message decrypted wrongly"
        );
    }
