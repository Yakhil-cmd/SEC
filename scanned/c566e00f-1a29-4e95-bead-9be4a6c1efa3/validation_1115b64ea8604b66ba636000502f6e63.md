### Title
SNS Governance Marks Proposal Executed Without Verifying Application-Level Action Completion - (File: rs/sns/governance/src/canister_control.rs)

### Summary

`perform_execute_generic_nervous_system_function_call` in SNS governance treats any non-error inter-canister reply from the `target_canister` as a successful proposal execution. A target canister that is upgraded (or designed from the start) to return a success reply without performing the intended action will cause the proposal to be permanently marked as `Executed`, with voting rewards distributed, even though the action never ran.

### Finding Description

When an `ExecuteGenericNervousSystemFunction` proposal is adopted, SNS governance calls `perform_execute_generic_nervous_system_function_call`, which makes an inter-canister call to the registered `target_canister_id` at `target_method_name`. The function then inspects only the transport-level result: [1](#0-0) 

Any `Ok(_reply)` — regardless of what the reply bytes contain — is returned as `Ok(())`, which propagates back through `perform_action` and causes `set_proposal_execution_status` to set `executed_timestamp_seconds` on the proposal: [2](#0-1) [3](#0-2) [4](#0-3) 

The proposal's `executed_timestamp_seconds` is then set permanently: [5](#0-4) 

The code itself acknowledges this gap with a `TODO` comment noting that application-level errors are indistinguishable from success: [6](#0-5) 

The `GenericNervousSystemFunction` proto allows any `target_canister_id` and `target_method_name` to be registered: [7](#0-6) 

### Impact Explanation

Once the proposal is marked `Executed`:

1. The proposal status is permanently `Executed` — it cannot be re-executed or retried.
2. Voting rewards are distributed to all voters as if the action succeeded.
3. Any downstream state that was supposed to change (e.g., a treasury transfer, a dapp configuration update, a parameter change in an external canister) never changes, but the governance record shows it did.

This is directly analogous to the EVM report's `LimitedCallsEnforcer` / `NonceEnforcer` scenario: the stateful side-effect (proposal marked executed, rewards distributed) persists even though the action was not performed.

### Likelihood Explanation

The attack requires:

1. A `GenericNervousSystemFunction` to be registered pointing to a canister whose controller is not exclusively the SNS root. This is realistic: many SNS deployments register functions targeting dapp canisters that may have external or multi-sig controllers during transition periods, or target third-party canisters entirely.
2. The target canister to be upgraded (or designed from the start) to return an empty/success reply without performing the intended action. Any canister controller can do this unilaterally, without a governance vote.

The scenario where a legitimately registered target canister is later upgraded by its controller to silently no-op is reachable by any canister developer who controls the target canister, without requiring a governance majority.

### Recommendation

Decode the reply bytes from the target canister and require them to match a `Result<String, String>` (or similar) success variant before marking the proposal as executed. The code already contains the exact suggestion:

```rust
// candid::Decode(&reply, Result<String, String>)
// This could then be converted into a Result<(), GovernanceError>.
```

Implementing this would make application-level failures propagate as `Err`, causing `set_proposal_execution_status` to record a failure instead of success. Additionally, SNS governance should document and enforce that `target_canister_id` must be a canister controlled exclusively by the SNS root, or provide a registry of approved target canisters.

### Proof of Concept

1. Register a `GenericNervousSystemFunction` (via `AddGenericNervousSystemFunction` proposal) pointing to a canister `C` at method `do_action`. Canister `C` is controlled by an external party.
2. Pass an `ExecuteGenericNervousSystemFunction` proposal to call `C::do_action`.
3. Before the proposal executes, upgrade `C` so that `do_action` immediately returns `()` (empty reply) without performing any state change.
4. The proposal executes: `perform_execute_generic_nervous_system_function_call` receives `Ok(vec![])` from the inter-canister call.
5. `Ok(())` is returned; `set_proposal_execution_status` sets `executed_timestamp_seconds = now`.
6. The proposal is permanently `Executed`. Voting rewards are distributed. The intended action never ran. [8](#0-7) [9](#0-8)

/// Executes a generic nervous system function (i.e., a non-native SNS proposal).
pub async fn perform_execute_generic_nervous_system_function_call(
    env: &dyn Environment,
    function: NervousSystemFunction,
    call: ExecuteGenericNervousSystemFunction,
) -> Result<(), GovernanceError> {
    // Get the canister id and the method against which we execute the proposal.
    let valid_function = ValidGenericNervousSystemFunction::try_from(&function)
        .map_err(|e| GovernanceError::new_with_message(ErrorType::InvalidProposal, e))?;

    let result = env
        .call_canister(
            valid_function.target_canister_id,
            &valid_function.target_method,
            call.payload,
        )
        .await;

    // Convert result.
    match result {
        Err(err) => Err(GovernanceError::new_with_message(
            ErrorType::External,
            format!("Canister method call to execute proposal failed: {err:?}"),
        )),

        Ok(_reply) => {
            // TODO: Do something with reply. E.g. store it in the proposal,
            // and/or deserialize it so that we can detect whether there was an
            // application-level error, as opposed to a communication
            // error. Detecting application error could be done as follows:
            //
            //   candid::!Decode(&reply, Result<String, String>)
            //
            // This could then be converted into a Result<(), GovernanceError>.
            // For now, any reply is considered a success.
            Ok(())
        }
    }
}

    /// Executes a (non-native) nervous system function as a result of an adopted proposal.
    async fn perform_execute_generic_nervous_system_function(
        &self,
        call: ExecuteGenericNervousSystemFunction,
    ) -> Result<(), GovernanceError> {
        match self
            .proto
            .id_to_nervous_system_functions
            .get(&call.function_id)
        {
            None => Err(GovernanceError::new_with_message(
                ErrorType::NotFound,
                format!(
                    "There is no generic NervousSystemFunction with id: {}",
                    call.function_id
                ),
            )),
            Some(function) => {
                perform_execute_generic_nervous_system_function_call(
                    &*self.env,
                    function.clone(),
                    call,
                )
                .await
            }
        }
    }
