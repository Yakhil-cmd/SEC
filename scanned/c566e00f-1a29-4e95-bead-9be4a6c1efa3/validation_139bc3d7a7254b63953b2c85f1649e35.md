### Title
Cycles Charged Before Canister Status Check in Ingress Induction, No Refund on Rejection - (File: rs/messaging/src/scheduling/valid_set_rule.rs)

### Summary

In `ValidSetRuleImpl::enqueue()`, the ingress induction fee is **charged from the canister's cycle balance before the canister's lifecycle status is checked**. When the status check subsequently finds the canister is `Stopping` or `Stopped`, the function returns an error — but the already-consumed cycles are never refunded. Any ingress message that reaches induction for a canister that transitioned to `Stopping`/`Stopped` after the ingress filter check will permanently burn the payer's cycles with no message delivered.

### Finding Description

In `rs/messaging/src/scheduling/valid_set_rule.rs`, the `enqueue()` function follows this order:

1. **Line 305**: `charge_ingress_induction_cost()` is called, which calls `consume_with_threshold::<IngressInduction>()` — permanently deducting cycles from `canister.system_state.cycles_balance`.
2. **Lines 318–331**: Only *after* the charge succeeds does the code check `canister.status()`. If the canister is `Stopping` or `Stopped`, it returns `Err(IngressInductionError::CanisterStopping(...))` or `Err(IngressInductionError::CanisterStopped(...))`.
3. The caller `induct_message()` records the failure in ingress history. **No refund path exists.** [1](#0-0) 

The `charge_ingress_induction_cost()` function in `CyclesAccountManager` directly consumes cycles via `consume_with_threshold::<IngressInduction>()` for non-paused canisters, permanently reducing `cycles_balance`. [2](#0-1) 

The ingress filter (`should_accept_ingress_message`) does check canister status before accepting a message at the boundary node level: [3](#0-2) 

However, this filter runs on a **stale snapshot** of state. Between the filter accepting the message and the actual induction round executing `enqueue()`, the canister can legitimately transition from `Running` → `Stopping` (e.g., a controller sends `StopCanister` in an intervening round). The induction path has no guard preventing this race.

### Impact Explanation

Any ingress message that was accepted while the canister was `Running` but is inducted after the canister transitions to `Stopping` or `Stopped` will:
- Permanently burn the induction fee cycles from the canister's balance.
- Deliver no message (the ingress is marked `Failed`).

The canister controller cannot recover these cycles. On a busy subnet with many in-flight ingress messages, a controller stopping a canister for security or operational reasons could inadvertently cause all queued-but-not-yet-inducted messages to each burn their induction fee. The `ingress_induction_cycles_debit` mechanism (which defers charges for paused DTS executions) is not applied here — the charge is immediate and irrecoverable.

**Impact: 4** — Permanent, unrecoverable loss of cycles from a canister's balance, triggered by a normal operational transition (stopping a canister). No funds are transferred to an attacker; cycles are simply burned.

### Likelihood Explanation

The race window exists in every round where a canister transitions from `Running` to `Stopping`. Ingress messages can be queued for multiple rounds before induction. A controller stopping a canister with pending ingress traffic will reliably trigger this. No special attacker capability is needed — any user who has sent ingress to a canister that is subsequently stopped by its controller will lose their induction fee.

**Likelihood: 3** — Requires a timing coincidence (ingress accepted before stop, inducted after stop), but this is a routine operational scenario, not an exotic edge case.

### Recommendation

Reorder the checks in `enqueue()` so the canister status is validated **before** `charge_ingress_induction_cost()` is called. The status check at lines 318–331 should be moved to before line 305. This mirrors the guard already present in `should_accept_ingress_message()` and in `ingress_selector.rs`: [4](#0-3) 

Alternatively, if the charge must happen first, a refund must be issued before returning the `CanisterStopping`/`CanisterStopped` error.

### Proof of Concept

**Attacker-controlled entry path** (unprivileged ingress sender + canister controller):

1. Canister `C` is `Running`. User `U` submits ingress message `M` to `C`. The ingress filter accepts it (canister is Running at filter time).
2. Before the next execution round, `C`'s controller sends `StopCanister` to `ic:00`. The canister transitions to `Stopping`.
3. In the next round, `ValidSetRuleImpl::induct_messages()` calls `induct_message()` → `enqueue()` for message `M`.
4. `charge_ingress_induction_cost()` is called at line 305. Since `C` has no paused DTS execution, `consume_with_threshold::<IngressInduction>()` is called, permanently deducting the induction fee from `C.cycles_balance`.
5. The status check at line 319 finds `CanisterStatusType::Stopping` and returns `Err(IngressInductionError::CanisterStopping(C))`.
6. `induct_message()` records `IngressState::Failed(CanisterStopping)` in ingress history. No refund is issued.
7. `C` has permanently lost the induction fee cycles. `M` was never executed. [5](#0-4) [6](#0-5)

    pub fn charge_ingress_induction_cost(
        &self,
        canister: &mut CanisterState,
        canister_current_memory_usage: NumBytes,
        canister_current_message_memory_usage: MessageMemoryUsage,
        canister_compute_allocation: ComputeAllocation,
        cycles: Cycles,
        subnet_cycles_config: CyclesAccountManagerSubnetConfig,
        reveal_top_up: bool,
    ) -> Result<(), CanisterOutOfCyclesError> {
        let threshold = self.freeze_threshold_cycles(
            canister.system_state.freeze_threshold,
            canister.system_state.memory_allocation,
            canister_current_memory_usage,
            canister_current_message_memory_usage,
            canister_compute_allocation,
            subnet_cycles_config,
            canister.system_state.reserved_balance(),
        );
        if canister.has_paused_execution_or_install_code() {
            if canister.system_state.debited_balance() < cycles + threshold {
                return Err(CanisterOutOfCyclesError {
                    canister_id: canister.canister_id(),
                    available: canister.system_state.debited_balance(),
                    requested: cycles,
                    threshold,
                    reveal_top_up,
                });
            }
            canister
                .system_state
                .add_postponed_charge_to_ingress_induction_cycles_debit(cycles);
            Ok(())
        } else {
            self.consume_with_threshold::<IngressInduction>(
                &mut canister.system_state,
                CompoundCycles::new(cycles, subnet_cycles_config.cost_schedule),
                threshold,
                reveal_top_up,
            )
        }
    }
