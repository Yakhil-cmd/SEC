### Title
Withdrawal Liveness DoS via `MAX_NUM_PENDING_TRANSACTION_NONCES` Exhaustion — (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

---

### Summary

An unprivileged attacker holding sufficient ckETH can permanently stall all other users' withdrawal requests by filling the 1000-nonce cap enforced in `withdrawal_requests_batch`. The cap is measured against `created_tx + sent_tx` entries, not against `pending_withdrawal_requests`, so once the attacker's 1000 requests are promoted to in-flight transactions, the batch function returns an empty slice for every subsequent call — blocking all honest users until the attacker's 1000 Ethereum transactions finalize.

---

### Finding Description

`withdrawal_requests_batch` enforces a hard ceiling of 1000 in-flight nonces: [1](#0-0) 

```rust
const MAX_NUM_PENDING_TRANSACTION_NONCES: usize = 1000;
let unique_pending_transaction_nonces: BTreeSet<_> =
    self.created_tx.keys().chain(self.sent_tx.keys()).collect();
let actual_batch_size = min(
    MAX_NUM_PENDING_TRANSACTION_NONCES
        .saturating_sub(unique_pending_transaction_nonces.len()),
    requested_batch_size,
);
```

When `unique_pending_transaction_nonces.len() >= 1000`, `actual_batch_size` saturates to 0 and the function returns `[]` regardless of how many requests sit in `pending_withdrawal_requests`.

`record_withdrawal_request` imposes **no per-user limit and no global cap** on `pending_withdrawal_requests`: [2](#0-1) 

The public `withdraw_eth` endpoint burns ckETH and enqueues the request for any non-anonymous caller whose amount meets the minimum: [3](#0-2) 

The only guard (`retrieve_withdraw_guard`) is a per-caller concurrency lock that prevents two simultaneous calls from the **same** principal, but it is released after each call completes. An attacker can therefore submit 1000 requests sequentially with no obstacle.

---

### Impact Explanation

Once the attacker's 1000 requests are promoted from `pending_withdrawal_requests` into `created_tx` / `sent_tx` (each consuming one nonce), `withdrawal_requests_batch` returns `[]` on every timer tick. All honest users whose requests are already in `pending_withdrawal_requests` are frozen until the attacker's 1000 Ethereum transactions finalize. On a congested network, or if the minter's Ethereum transaction throughput is low, this can last hours.

---

### Likelihood Explanation

**Attack cost:**  
The current minimum withdrawal amount is 0.005 ETH (5 000 000 000 000 000 wei), reduced by the 2026-05-29 upgrade: [4](#0-3) 

1000 × 0.005 ETH = **5 ETH** of principal, which the attacker **recovers** when their transactions finalize (minus gas). The net cost is only ~1000 × Ethereum gas fees for simple ETH transfers — at current mainnet fees (cents per transfer) this is on the order of **$1–$10 total**. The attack is therefore economically trivial.

**No privileged access required.** Any non-anonymous IC principal with 5 ETH worth of ckETH can execute this.

---

### Recommendation

1. **Cap `pending_withdrawal_requests` per principal** — reject `withdraw_eth` if the caller already has N unprocessed requests queued.
2. **Cap the global `pending_withdrawal_requests` queue** — return `WithdrawalError::TemporarilyUnavailable` when the queue exceeds a safe bound, preventing the attacker from pre-loading the queue before nonces are consumed.
3. **Interleave requests fairly** — when selecting the next batch, round-robin across distinct principals rather than taking the front of a single FIFO queue, so a single principal cannot monopolize all 1000 nonce slots.

---

### Proof of Concept

```
1. Attacker calls withdraw_eth 1000 times (sequentially, each ≥ 5_000_000_000_000_000 wei).
   → pending_withdrawal_requests now contains 1000 attacker entries.

2. Minter timer fires → withdrawal_requests_batch(N) returns all 1000 attacker requests
   (created_tx + sent_tx is still empty at this point).

3. Minter calls record_created_transaction for each → moves all 1000 into created_tx.
   → unique_pending_transaction_nonces.len() == 1000.

4. Honest user calls withdraw_eth → request appended to pending_withdrawal_requests.

5. Minter timer fires → withdrawal_requests_batch(N):
   MAX_NUM_PENDING_TRANSACTION_NONCES.saturating_sub(1000) == 0
   → returns [].

6. Honest user's request is never processed until all 1000 attacker transactions
   finalize on Ethereum (potentially hours).

Invariant assertion:
   after step 3, for any N: withdrawal_requests_batch(N) == []
```

```rust
    pub fn withdrawal_requests_batch(&self, requested_batch_size: usize) -> Vec<WithdrawalRequest> {
        // The number of pending transaction nonces is counted and not the number of pending transactions
        // because a nonce may be associated with several distinct transactions (due to re-submission and dynamic fees).
        // However, once a nonce is chosen for a withdrawal request, it's in our interest that the corresponding transaction be finalized asap.
        // Limiting the number of transactions would be counter-productive.
        const MAX_NUM_PENDING_TRANSACTION_NONCES: usize = 1000;
        let unique_pending_transaction_nonces: BTreeSet<_> =
            self.created_tx.keys().chain(self.sent_tx.keys()).collect();
        let actual_batch_size = min(
            MAX_NUM_PENDING_TRANSACTION_NONCES
                .saturating_sub(unique_pending_transaction_nonces.len()),
            requested_batch_size,
        );
        self.withdrawal_requests_iter()
            .take(actual_batch_size)
            .cloned()
            .collect()
    }
```
