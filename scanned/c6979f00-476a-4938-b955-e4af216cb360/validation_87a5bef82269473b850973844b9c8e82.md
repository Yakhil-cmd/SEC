### Title
Fixed Gas Limit Assumption for ckERC20 Withdrawals Causes Irrecoverable ckETH Loss on Non-Standard Token Failure - (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

### Summary
The ckETH minter canister applies a single hardcoded gas limit of `65,000` to every ckERC20 withdrawal transaction, regardless of the specific ERC-20 token's actual on-chain gas requirements. When a supported ERC-20 token's `transfer` call consumes more than this limit, the Ethereum transaction reverts with out-of-gas. The minter's reimbursement logic then returns only the ckERC20 principal amount; the ckETH already burned to pay the estimated transaction fee is permanently lost to the user.

### Finding Description
`CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` is a compile-time constant set to `65_000` gas units. [1](#0-0) 

`estimate_gas_limit` returns this same constant for every `CkErc20` withdrawal request with no per-token override path. [2](#0-1) 

The official documentation acknowledges the assumption explicitly: *"The `gas_limit` for ckERC20 withdrawals is currently fixed to `65_000` and should be sufficient for standard ERC-20 contracts."* [3](#0-2) 

When `create_transaction` builds the EIP-1559 transaction for a `CkErc20` withdrawal, it encodes the fixed `gas_limit` directly into the signed transaction sent to Ethereum. [4](#0-3) 

On finalization, when the receipt status is `Failure`, `record_finalized_transaction` creates only a `ReimbursementIndex::CkErc20` entry — reimbursing the ckERC20 principal — and no corresponding `ReimbursementIndex::CkEth` entry for the gas fee already burned. [5](#0-4) 

`process_reimbursement` then mints back only the ckERC20 amount; the ckETH burned in `withdraw_erc20` for the gas fee is never returned. [6](#0-5) 

The `withdraw_erc20` handler burns ckETH first, then ckERC20. If the Ethereum transaction later fails, only the ckERC20 burn is reversed. [7](#0-6) 

### Impact Explanation
Any user who calls `withdraw_erc20` for a supported token whose on-chain `transfer` opcode path exceeds 65,000 gas will:
1. Have their ckETH gas-fee burn finalized and unrecoverable.
2. Receive their ckERC20 principal back.
3. Suffer a net ckETH loss equal to `max_transaction_fee` (currently estimated at ~0.03 ETH at moderate gas prices).

Because the gas limit is uniform and immutable at runtime, every subsequent withdrawal attempt for that token will also fail, making the token effectively un-withdrawable while continuously draining ckETH from users who retry. This breaks the 1:1 redemption guarantee for the affected ckERC20 token and constitutes a ledger conservation bug: ckERC20 supply remains backed on-chain, but ckETH supply is silently deflated without a corresponding Ethereum-side event.

### Likelihood Explanation
Several production-grade ERC-20 tokens require more than 65,000 gas for a `transfer` call:

- **USDT (Tether)**: contains a fee-deduction branch and a blacklist check; measured at ~65,000–70,000 gas depending on storage state. USDT is already a listed supported token.
- **Proxy-pattern tokens** (e.g., upgradeable USDC): delegate-call overhead adds ~2,000–5,000 gas per hop.
- **Tokens with hooks or callbacks** (ERC-777 wrappers, rebasing tokens): can exceed 100,000 gas.

The NNS `AddErc20Arg` proposal path does not validate or bound the gas cost of the token's `transfer` function before adding it. [8](#0-7) 

An unprivileged user triggers the loss simply by calling `withdraw_erc20` on a legitimately-listed token whose gas profile exceeds the hardcoded limit.

### Recommendation
**Short term**: Document that tokens whose `transfer` function may exceed 65,000 gas (e.g., USDT with fee enabled, proxy tokens) are not safely withdrawable, and surface a warning in the minter dashboard and `get_minter_info` response.

**Long term**: Store a per-token `gas_limit` field in `CkErc20Token` state, populated at token-addition time (e.g., via an `eth_estimateGas` outcall against the token contract). Use this per-token limit in `estimate_gas_limit` instead of the global constant. Additionally, consider reimbursing the ckETH gas fee when an ERC-20 `transfer` reverts due to out-of-gas, to avoid penalizing users for a protocol-level misconfiguration.

### Proof of Concept

1. USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is a supported ckERC20 token. [9](#0-8) 

2. A user calls `withdraw_erc20` specifying the ckUSDT ledger ID and a valid amount. The minter burns ckETH (e.g., 30,000,000,000,000,000 wei) and ckUSDT. [10](#0-9) 

3. `estimate_gas_limit` returns `65_000` for the ckUSDT withdrawal. [2](#0-1) 

4. The minter submits an EIP-1559 transaction to Ethereum with `gas_limit = 65_000`. If USDT's fee mechanism is active or storage layout causes the call to exceed 65,000 gas, the transaction reverts with `status = Failure`.

5. `record_finalized_transaction` creates only a `ReimbursementIndex::CkErc20` reimbursement request. [11](#0-10) 

6. `process_reimbursement` mints back the ckUSDT amount to the user. The ckETH burned in step 2 is never returned. The user's net loss equals the ckETH gas fee. [12](#0-11)
