### Title
Unauthorized Third-Party Can Force SNS Swap Participation on Behalf of Any Buyer, Bypassing Confirmation Text Consent - (`rs/sns/swap/canister/canister.rs`)

### Summary
The `refresh_buyer_tokens` endpoint in the SNS swap canister accepts an arbitrary `buyer` principal in its request body, allowing any unprivileged caller to register swap participation on behalf of another principal. When the SNS requires a `confirmation_text` (a legal/compliance acceptance mechanism), the attacker can read this publicly stored text from the swap canister's state and supply it in the call, bypassing the explicit consent requirement. A buyer who has transferred ICP to their swap subaccount (as a preparatory step) can be forced into participation without their knowledge or consent.

### Finding Description

The `refresh_buyer_tokens` canister endpoint in `rs/sns/swap/canister/canister.rs` resolves the buyer principal from the request argument rather than from the authenticated caller identity:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
    };
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, ...)
        .await
}
```

When `arg.buyer` is non-empty, the function uses the provided principal without verifying that the caller is that principal. The `confirmation_text` argument is also passed through from the caller, not from the actual buyer.

The `confirmation_text` is set during SNS initialization and is publicly readable from the swap canister's state via the `get_init` query endpoint. The `validate_confirmation_text` function in `rs/sns/swap/src/swap.rs` only checks that the provided text matches the stored text — it does not verify that the caller is the buyer:

```rust
pub fn validate_confirmation_text(
    &self,
    confirmation_text: Option<String>,
) -> Result<(), String> {
    match (self.init_or_panic().confirmation_text.as_ref(), confirmation_text) {
        (Some(expected_text), Some(text)) => {
            if &text != expected_text { Err(...) } else { Ok(()) }
        }
        // ...
    }
}
```

An attacker can therefore:
1. Read the `confirmation_text` from the swap canister's public `get_init` state.
2. Wait for a target buyer to transfer ICP to their swap subaccount (subaccount = `principal_to_subaccount(buyer)`).
3. Call `refresh_buyer_tokens({buyer: victim.to_string(), confirmation_text: Some(public_text)})` from any identity.
4. The victim is now registered as a participant, having "accepted" the confirmation text without their knowledge.

This is confirmed by the integration test in `rs/tests/nns/sns/lib/src/sns_deployment.rs` which explicitly demonstrates that after an ICP transfer, a third-party caller (`default_sns_agent`) can successfully call `refresh_buyer_tokens` on behalf of the wealthy user (`wealthy_user_identity.principal_id`), and the call succeeds.

### Impact Explanation

The `confirmation_text` is a legal/compliance acceptance mechanism (e.g., "I confirm I am not a US person", "I confirm I understand the risks"). By allowing a third party to call `refresh_buyer_tokens` on behalf of a buyer and supply the correct confirmation text, the system allows bypassing this explicit consent requirement. Once registered:

- The buyer's ICP is locked in the swap until it concludes.
- If the swap is committed, the buyer receives SNS tokens instead of their ICP, potentially at an unfavorable or worthless exchange rate.
- The buyer has been recorded as having "accepted" legal/compliance terms they never explicitly agreed to.

### Likelihood Explanation

- Any unprivileged ingress sender can call `refresh_buyer_tokens` with any buyer's principal — no special role is required.
- The `confirmation_text` is publicly readable from the swap canister's `get_init` query endpoint.
- The attack requires the target buyer to have already transferred ICP to their swap subaccount, which is a realistic scenario (e.g., a user who transferred funds but is still deliberating, or who transferred accidentally).
- The `RefreshBuyerTokensRequest.buyer` field is explicitly documented as "If not specified, the caller is used," signaling that third-party calls were considered but the consent bypass was not addressed.

### Recommendation

When `arg.buyer` is non-empty and differs from `caller_principal_id()`, either:
1. Reject the call outright (require `caller == buyer`), or
2. Accept the call but ignore the `confirmation_text` from the third-party caller — only allow `confirmation_text` to be accepted when `caller == buyer`.

Option 1 is simpler and safer. If third-party notification is a desired feature (e.g., to allow anyone to notify the swap of a buyer's transfer), the `confirmation_text` must never be accepted from a non-buyer caller.

### Proof of Concept

1. SNS swap is initialized with `confirmation_text = "I confirm I am not a US person"`.
2. Alice transfers 100 ICP to her swap subaccount (`swap_canister_id`, subaccount = `principal_to_subaccount(alice)`).
3. Alice has not yet called `refresh_buyer_tokens` — she is still deciding whether to participate.
4. Bob queries `get_init` on the swap canister and reads `confirmation_text = "I confirm I am not a US person"`.
5. Bob calls `refresh_buyer_tokens({buyer: alice.principal_id.to_string(), confirmation_text: Some("I confirm I am not a US person")})` from his own identity.
6. The call succeeds: Alice is now registered as a swap participant with 100 ICP, having "accepted" the confirmation text without her knowledge or consent.
7. If the swap commits, Alice receives SNS tokens and cannot recover her ICP.

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
