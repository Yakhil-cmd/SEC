### Title
Dissolve Delay Bypass via Pre-Dissolved Neuron Stake Refresh - (`rs/nns/governance/src/governance.rs`, `rs/sns/governance/src/governance.rs`)

### Summary

Both NNS and SNS governance allow a user to bypass the dissolve delay (the core withdrawal lock mechanism) by pre-dissolving a minimal-stake neuron, waiting for the delay to expire, then adding arbitrary stake via `ClaimOrRefresh`. Because `refresh_neuron` calls `update_stake_adjust_age` / `update_stake` — which only update `cached_neuron_stake_e8s` and `aging_since_timestamp_seconds` but never reset the dissolve state — a neuron that is already `Dissolved` remains `Dissolved` after any stake increase. The attacker can then immediately call `disburse_neuron` on the newly-inflated stake without serving any dissolve delay.

### Finding Description

The dissolve delay is the fundamental time-lock of NNS/SNS governance. To disburse a neuron's stake, the neuron must be in the `Dissolved` state. The intended flow is: stake → set dissolve delay → start dissolving → wait → disburse.

The vulnerability is that the dissolve state and the stake amount are completely decoupled. When a neuron's stake is refreshed via `ClaimOrRefresh`, the governance canister calls `refresh_neuron`, which calls `update_stake_adjust_age` (NNS) or `update_stake` (SNS). Neither function touches the `dissolve_state` field.

### Impact Explanation

An attacker can bypass the dissolve delay for an arbitrarily large stake. The dissolve delay is the core security mechanism ensuring long-term alignment of governance participants. Bypassing it means:

- A user can stake a large amount of ICP/SNS tokens and immediately disburse them without serving any lock period.
- The intended invariant — that tokens staked in a neuron are locked for at least the dissolve delay — is violated for any stake added to an already-dissolved neuron.
- This undermines the economic security model of NNS/SNS governance (no direct theft of other users' funds, but the attacker's own tokens are freed from the lock).

### Likelihood Explanation

The attack is straightforward and requires no privileged access:

1. Any unprivileged principal can create a neuron with the minimum stake (`neuron_minimum_stake_e8s`).
2. They immediately call `StartDissolving`.
3. They wait for the dissolve delay (as short as 7 days for NNS via `INITIAL_NEURON_DISSOLVE_DELAY`).
4. The neuron is now permanently `Dissolved`.
5. They transfer any amount of ICP/SNS tokens to the neuron's subaccount on the ledger.
6. They call `ClaimOrRefresh` → `refresh_neuron` → `update_stake_adjust_age` / `update_stake`.
7. They immediately call `disburse_neuron` — succeeds because the neuron is still `Dissolved`.

Steps 5–7 can be repeated indefinitely. The attacker only needs to serve the dissolve delay once.

### Recommendation

When `refresh_neuron` increases a neuron's stake, the governance canister should reset the dissolve state to `NotDissolving` with the configured dissolve delay (or at minimum, the minimum dissolve delay), analogous to the Telcoin recommendation of resetting `withdrawalRequestTimestamps` on stake. Specifically:

- In `update_stake_adjust_age` (NNS) and `update_stake` (SNS), when the stake is being **increased** and the neuron is in the `DissolvingOrDissolved` state with `when_dissolved_timestamp_seconds <= now` (i.e., already dissolved), the dissolve state should be reset to `NotDissolving` with a fresh dissolve delay.
- Alternatively, `refresh_neuron` itself should check whether the neuron is dissolved and, if so, reset the dissolve state before updating the stake.

### Proof of Concept

**NNS (applies equally to SNS with SNS-specific API calls):**

```
// Step 1: Create a neuron with minimum stake (e.g., 1 ICP = 100_000_000 e8s)
// Transfer minimum_stake to governance subaccount(controller, memo=0)
// Call manage_neuron(ClaimOrRefresh { by: MemoAndController { memo: 0, controller: None } })
// → neuron created with INITIAL_NEURON_DISSOLVE_DELAY (7 days), state: NotDissolving

// Step 2: Start dissolving immediately
// Call manage_neuron(Configure { operation: StartDissolving {} })
// → neuron state: Dissolving, when_dissolved_timestamp_seconds = now + 7 days

// Step 3: Wait 7 days
// → neuron state: Dissolved (when_dissolved_timestamp_seconds <= now)

// Step 4: Transfer large amount (e.g., 10,000 ICP) to the same governance subaccount
// (same subaccount = governance_canister_id + subaccount(controller, memo=0))

// Step 5: Refresh the neuron
// Call manage_neuron(ClaimOrRefresh { by: NeuronIdOrSubaccount {} })
// → refresh_neuron() → update_stake_adjust_age(10_000 ICP, now)
// → cached_neuron_stake_e8s = 10_000 ICP
// → dissolve_state UNCHANGED: still DissolvingOrDissolved { when_dissolved_timestamp_seconds = past }
// → neuron is still Dissolved

// Step 6: Immediately disburse
// Call manage_neuron(Disburse { amount: None, to_account: None })
// → disburse_neuron checks: neuron.state(now) == Dissolved ✓
// → 10,000 ICP disbursed immediately, no dissolve delay served
```

The dissolve delay for the 10,000 ICP was never served. The attacker only waited 7 days for 1 ICP of minimum stake.

    pub fn update_stake_adjust_age(&mut self, updated_stake_e8s: u64, now: u64) {
        // If the updated stake is less than the original stake, preserve the
        // age and distribute it over the new amount. This should not happen
        // in practice, so this code exists merely as a defensive fallback.
        //
        // TODO(NNS1-954) Consider whether update_stake_adjust_age (and other
        // similar methods) should use a neurons effective stake rather than
        // the cached stake.
        if updated_stake_e8s < self.cached_neuron_stake_e8s {
            println!(
                "{}Reducing neuron {:?} stake via update_stake_adjust_age: {} -> {}",
                LOG_PREFIX,
                self.id(),
                self.cached_neuron_stake_e8s,
                updated_stake_e8s
            );
            self.cached_neuron_stake_e8s = updated_stake_e8s;
        } else {
            // If one looks at "stake * age" as describing an area, the goal
            // at this point is to increase the stake while keeping the area
            // constant. This means decreasing the age in proportion to the
            // additional stake, which is the purpose of combine_aged_stakes.
            let (new_stake_e8s, new_age_seconds) = combine_aged_stakes(
                self.cached_neuron_stake_e8s,
                self.age_seconds(now),
                updated_stake_e8s.saturating_sub(self.cached_neuron_stake_e8s),
                0,
            );
            // A consequence of the math above is that the 'new_stake_e8s' is
            // always the same as the 'updated_stake_e8s'. We use
            // 'combine_aged_stakes' here to make sure the age is
            // appropriately pro-rated to accommodate the new stake.
            assert!(new_stake_e8s == updated_stake_e8s);
            self.cached_neuron_stake_e8s = new_stake_e8s;

            let new_aging_since_timestamp_seconds = now.saturating_sub(new_age_seconds);
            let new_disolved_dissolve_state_and_age = self
                .dissolve_state_and_age()
                .adjust_age(new_aging_since_timestamp_seconds);
            self.set_dissolve_state_and_age(new_disolved_dissolve_state_and_age);
        }

    pub fn update_stake(&mut self, new_stake_e8s: u64, now: u64) {
        // If this neuron has an age and its stake is being increased, adjust the
        // neuron's age
        if self.aging_since_timestamp_seconds < now && self.cached_neuron_stake_e8s <= new_stake_e8s
        {
            let old_stake = self.cached_neuron_stake_e8s as u128;
            let old_age = now.saturating_sub(self.aging_since_timestamp_seconds) as u128;
            let new_age = (old_age * old_stake) / (new_stake_e8s as u128);

            // new_age * new_stake = old_age * old_stake -
            // (old_stake * old_age) % new_stake. That is, age is
            // adjusted in proportion to the stake, but due to the
            // discrete nature of u64 numbers, some resolution is
            // lost due to the division above. This means the age
            // bonus is derived from a constant times age times
            // stake, minus up to new_stake - 1 each time the
            // neuron is refreshed. Only if old_age * old_stake is
            // a multiple of new_stake does the age remain
            // constant after the refresh operation. However, in
            // the end, the most that can be lost due to rounding
            // from the actual age, is always less 1 second, so
            // this is not a problem.
            self.aging_since_timestamp_seconds = now.saturating_sub(new_age as u64);
            // Note that if new_stake == old_stake, then
            // new_age == old_age, and
            // now - new_age =
            // now-(now-neuron.aging_since_timestamp_seconds)
            // = neuron.aging_since_timestamp_seconds.
        }

        self.cached_neuron_stake_e8s = new_stake_e8s;
    }

fn test_update_stake_adjust_age_for_dissolved_neuron_variant_a_now() {
    // WhenDissolvedTimestampSeconds(NOW) ==> dissolved
    let mut neuron = create_neuron_with_stake_dissolve_state_and_age(
        10 * E8,
        DissolveStateAndAge::DissolvingOrDissolved {
            when_dissolved_timestamp_seconds: NOW,
        },
    );

    let new_stake_e8s = 1_500_000_000_u64; // 15 ICP
    neuron.update_stake_adjust_age(new_stake_e8s, NOW);

    assert_eq!(neuron.cached_neuron_stake_e8s, new_stake_e8s);
    assert_eq!(
        neuron.dissolve_state_and_age(),
        DissolveStateAndAge::DissolvingOrDissolved {
            when_dissolved_timestamp_seconds: NOW,
        }
    );
