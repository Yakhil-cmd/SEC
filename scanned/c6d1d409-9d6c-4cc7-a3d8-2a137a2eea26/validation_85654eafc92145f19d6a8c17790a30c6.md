Based on my analysis of the production code, this is a real vulnerability. Here is my assessment:

---

### Title
Unbounded `active_assembles` / `artifact_processor_tasks` Growth via `SLOT_TABLE_NO_LIMIT` — (`rs/p2p/consensus_manager/src/receiver.rs`)

### Summary

The `ConsensusManagerReceiver` contains a per-peer slot table limit guard that is the sole defense against a Byzantine peer exhausting receiver-side memory and Tokio task capacity. When channels are configured with `SLOT_TABLE_NO_LIMIT` (`usize::MAX`), this guard is rendered inoperative, allowing a single Byzantine subnet peer to grow `active_assembles` and `artifact_processor_tasks` without bound.

### Finding Description

In `handle_slot_update_receive`, the only admission control for new slot entries is: [1](#0-0) 

```rust
// Only insert slot update if we are below peer slot table limit.
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { ... }
Entry::Vacant(_) => {
    // warn and drop
}
```

When `self.slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice (a `usize` counter cannot reach `usize::MAX` before OOM). Every new `(slot_number, artifact_id)` pair from a peer is unconditionally admitted.

For each admitted new artifact ID that is not already in `active_assembles`, the code:

1. Inserts a new entry into `self.slot_table[peer_id]`
2. Inserts a new `watch::Sender` into `self.active_assembles`
3. Spawns a new Tokio task into `self.artifact_processor_tasks` [2](#0-1) 

These tasks are long-lived. In `process_slot_update`, after the assembler returns (even `AssembleResult::Unwanted`), the task waits for the peer counter to reach zero: [3](#0-2) 

```rust
AssembleResult::Unwanted => {
    let _ = peer_rx.wait_for(|p| p.is_empty()).await;
    ...
}
```

The peer counter only reaches zero when the peer overwrites the slot or leaves the topology. A Byzantine peer that never overwrites its slots keeps every task alive indefinitely. Furthermore, `handle_artifact_processor_joined` restarts the task if the peer counter is still non-empty when the task finishes: [4](#0-3) 

So the Byzantine peer can pin N tasks alive by holding N distinct slots open.

The `SLOT_TABLE_NO_LIMIT` constant is applied to production channels in `rs/replica/setup_ic_network/src/lib.rs` (7 usages confirmed), covering consensus, certifier, DKG, iDKG, and HTTPS outcalls channels. [5](#0-4) 

The `AvailableSlotSet` on the sender side also has no hard cap — it grows `next_free_slot` unboundedly past `SLOT_TABLE_THRESHOLD` (which only emits a warning): [6](#0-5) 

### Impact Explanation

A single Byzantine subnet peer (below the BFT fault threshold) can:
- Grow `active_assembles` (a `HashMap`) and `artifact_processor_tasks` (a `JoinSet`) to an arbitrary size
- Exhaust heap memory on the honest replica
- Exhaust the Tokio task scheduler's capacity, starving legitimate consensus work
- Cause the honest replica to crash or become unresponsive, violating the liveness guarantee that a single Byzantine peer cannot exhaust honest replica resources

### Likelihood Explanation

The attacker must be a compromised subnet node authenticated via TLS — a realistic Byzantine adversary within the IC threat model. No threshold corruption is required; a single node suffices. The attack is mechanically simple: send `N` `SlotUpdate` messages each with a distinct `slot_id` and a distinct artifact `id`. The `update_handler` performs only protobuf decoding before forwarding to the event loop with no rate limiting or slot count check at the transport layer. [7](#0-6) 

### Recommendation

1. **Remove `SLOT_TABLE_NO_LIMIT`**: Replace `usize::MAX` with a protocol-appropriate bound (e.g., the maximum number of in-flight artifacts a correct peer can legitimately have per artifact type). For consensus this is bounded by the pipeline depth.
2. **Enforce the limit at the transport/handler layer**: Reject slot updates from a peer that has already filled its quota before they reach the event loop.
3. **Add a global cap** on `active_assembles.len()` and `artifact_processor_tasks.len()` as a defense-in-depth backstop.

### Proof of Concept

State-machine test (no network required):

```rust
let (mut mgr, _) = ReceiverManagerBuilder::new()
    .with_slot_limit(usize::MAX)  // SLOT_TABLE_NO_LIMIT
    .build();
let cancellation = CancellationToken::new();
for i in 0u64..100_000 {
    mgr.handle_slot_update_receive(
        SlotUpdate {
            slot_number: SlotNumber::from(i),
            commit_id: CommitId::from(i),
            update: Update::Id(i),   // distinct ID per slot
        },
        NODE_1,
        ConnId::from(1),
        cancellation.clone(),
    );
}
// Both grow to 100_000 — unbounded
assert_eq!(mgr.active_assembles.len(), 100_000);
assert_eq!(mgr.artifact_processor_tasks.len(), 100_000);
```

The existing test `slot_table_limit_exceeded` already demonstrates the guard works when `slot_limit` is finite: [8](#0-7) 

The vulnerability is precisely that `SLOT_TABLE_NO_LIMIT = usize::MAX` disables this guard in production.
