### Title
Cross-Subaccount Allowance Disclosure in `icrc103_get_allowances` Due to Owner-Only Break Condition — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

### Summary
The `get_allowances` function's BTree range scan terminates only when the `owner` field changes, not when the full `Account` (owner + subaccount) changes. Any unprivileged caller can retrieve allowances belonging to subaccounts of owner X that they never requested.

### Finding Description
`get_allowances` constructs a range starting at `(from_account, min_spender)` and iterates forward through the `ALLOWANCES_MEMORY` StableBTreeMap. The loop's only termination guard for account scope is:

```rust
if account_spender.account.owner != from.owner {
    break;
}
``` [1](#0-0) 

Because `Account` is ordered by `(owner, effective_subaccount)`, entries for `{owner: X, subaccount: [2;32]}` sort immediately after entries for `{owner: X, subaccount: [1;32]}`. The missing guard `account_spender.account != from` (full equality including subaccount) means the scan crosses subaccount boundaries and pushes foreign entries into the result. [2](#0-1) 

The `Account` ordering that makes this exploitable: [3](#0-2) 

The public query endpoint applies no caller restriction: [4](#0-3) 

### Impact Explanation
An unprivileged attacker can enumerate all allowances for every subaccount of any owner by issuing a single `icrc103_get_allowances` call with `from_account = {owner: X, subaccount: None}` (effective subaccount `[0;32]`, the BTree minimum). This discloses: which subaccounts have active approvals, which spenders are authorized, and the allowance amounts and expiry times — information that is supposed to be scoped to the exact account specified.

### Likelihood Explanation
The endpoint is a public `#[query]` call with no authentication or rate-limiting beyond `max_take`. The exploit requires only knowledge of a target principal, which is public on-chain. It is trivially reproducible in a local state-machine test.

### Recommendation
Replace the owner-only break condition with a full account equality check:

```rust
// Before (line 1204):
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```

`Account::eq` already uses `effective_subaccount()` for comparison, so `None` and `Some([0;32])` remain equivalent as intended. [5](#0-4)
