### Title
Silent Allowance Amount Truncation to `u64::MAX` in ICP Ledger `icrc2_approve` - (`File: rs/ledger_suite/icp/ledger/src/main.rs`)

---

### Summary

The ICP ledger's `icrc2_approve_not_async` function silently clamps an oversized `Nat` allowance amount to `u64::MAX` instead of returning an error. Any unprivileged user who submits an `amount` exceeding `u64::MAX` via `icrc2_approve` will have their allowance recorded as `u64::MAX` — not the value they requested — while still paying the approval fee, with no error returned.

---

### Finding Description

In `rs/ledger_suite/icp/ledger/src/main.rs` at line 1350, the `icrc2_approve_not_async` function converts the caller-supplied `Nat` amount to `Tokens` (backed by `u64`) using:

```rust
let allowance = Tokens::from_e8s(arg.amount.0.to_u64().unwrap_or(u64::MAX));
```

The ICRC-2 standard defines `amount` as a `Nat` (arbitrary-precision integer). When `arg.amount` exceeds `u64::MAX`, `to_u64()` returns `None`, and `unwrap_or(u64::MAX)` silently substitutes the maximum `u64` value. No `ApproveError` is returned to the caller.

The same pattern appears in the ICRC-1 ledger at line 840:

```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```

Notably, the `expected_allowance` field in the same function is handled *correctly* — when it exceeds `u64::MAX`, an `ApproveError::AllowanceChanged` is returned:

```rust
let expected_allowance = match arg.expected_allowance {
    Some(n) => match n.0.to_u64() {
        Some(n) => Some(Tokens::from_e8s(n)),
        None => {
            // returns ApproveError::AllowanceChanged
        }
    },
    None => None,
};
```

This asymmetry confirms the `amount` path is the defective one. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

- The caller pays the approval fee (deducted from their balance).
- The on-chain allowance is recorded as `u64::MAX` instead of the requested value.
- The caller receives `Ok(block_index)` — no indication that the stored allowance differs from the requested amount.
- Any downstream logic that reads the allowance and compares it to the originally requested value will observe a discrepancy.
- A spender relying on the exact approved amount (e.g., for a protocol that checks `allowance == requested_amount`) will observe an unexpected value.

This is a **ledger accounting bug**: the stored state diverges from the caller's intent without any error signal, analogous to the reference report where the recorded deposit amount diverged from the transferred amount. [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged user can call `icrc2_approve` via ingress with a `Nat` amount exceeding `u64::MAX`. The ICRC-2 interface explicitly accepts `Nat`, so this is a valid and reachable input. No special privileges, keys, or majority corruption are required. The trigger condition is a single crafted ingress message. [5](#0-4) 

---

### Recommendation

Replace the silent `unwrap_or(u64::MAX)` with an explicit bounds check that returns a proper error when the amount exceeds the representable range:

```rust
let allowance = match arg.amount.0.to_u64() {
    Some(e8s) => Tokens::from_e8s(e8s),
    None => return Err(ApproveError::GenericError {
        error_code: Nat::from(0_u64),
        description: "amount exceeds maximum token value".to_string(),
    }),
};
```

Apply the same fix to the ICRC-1 ledger's `icrc2_approve_not_async` at line 840, replacing `unwrap_or_else(|_| Tokens::max_value())` with an explicit error return. [1](#0-0) [3](#0-2) 

---

### Proof of Concept

**Entry path:** Unprivileged user → ingress → `icrc2_approve` → `icrc2_approve_not_async`

**Steps:**

1. Call `icrc2_approve` on the ICP ledger with:
   ```
   amount = Nat::from(u64::MAX) + Nat::from(1u64)   // i.e., 2^64
   ```
2. Observe the return value: `Ok(block_index)` — no error.
3. Query `icrc2_allowance` for the same `(from, spender)` pair.

**Expected result:** `ApproveError` indicating the amount is out of range.

**Actual result:** Allowance is set to `u64::MAX` (= `18446744073709551615`), not `2^64`. The approval fee is deducted from the caller's balance. The caller has no way to distinguish this outcome from a successful exact-amount approval without re-querying the allowance. [6](#0-5) [7](#0-6)

fn icrc2_approve_not_async(
    caller: Principal,
    arg: ApproveArgs,
    override_spender: Option<AccountIdentifier>,
) -> Result<Nat, ApproveError> {
    if !LEDGER.read().unwrap().can_send(&PrincipalId::from(caller)) {
        trap("Caller cannot approve token transfers on the ledger.");
    }

    if !LEDGER.read().unwrap().feature_flags.icrc2 {
        trap("ICRC-2 features are not enabled on the ledger.");
    }
    let now = TimeStamp::from_nanos_since_unix_epoch(time());

    let from_account = Account {
        owner: caller,
        subaccount: arg.from_subaccount,
    };
    let from = AccountIdentifier::from(from_account);
    if from_account.owner == arg.spender.owner {
        trap("self approval is not allowed");
    }
    let spender = if let Some(override_spender) = override_spender {
        override_spender
    } else {
        AccountIdentifier::from(arg.spender)
    };
    let minting_acc = LEDGER
        .read()
        .unwrap()
        .minting_account_id
        .expect("Minting canister id not initialized");

    if from == minting_acc {
        trap("the minting account cannot delegate mints")
    }
    match arg.memo.as_ref() {
        Some(memo) if memo.0.len() > MEMO_SIZE_BYTES => trap("the memo field is too large"),
        _ => {}
    };

    let allowance = Tokens::from_e8s(arg.amount.0.to_u64().unwrap_or(u64::MAX));
