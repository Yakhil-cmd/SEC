### Title
Unauthenticated `read_state` Certificate Parsing When `root_pk` Is `None` Allows State Certification Forgery - (File: `rs/canister_client/read_state_response_parser/src/lib.rs`)

### Summary
`parse_read_state_response` and `parse_subnet_read_state_response` accept `root_pk: Option<&ThresholdSigPublicKey>`. When `None` is supplied, the functions skip all BLS threshold-signature verification and deserialize the raw CBOR certificate bytes directly, then proceed to extract and return `RequestStatus` or `SubnetMetrics` from the unverified tree. This is structurally identical to the reported mobile-wallet bug: a single function handles both the "verify-then-parse" and the "skip-verify-and-parse" paths based on the presence of a single optional field, so any caller that omits the root key receives unauthenticated state data without any error.

### Finding Description

In `rs/canister_client/read_state_response_parser/src/lib.rs`, both public functions branch on `root_pk`:

```rust
// parse_read_state_response (lines 48-55)
let certificate = match root_pk {
    Some(pk) => {
        ic_certification::verify_certificate(&response.certificate, effective_canister_id, pk)
            .map_err(|source| format!("verifying certificate failed: {source}"))?
    }
    None => serde_cbor::from_slice(response.certificate.as_slice())   // ← no verification
        .map_err(|source| format!("decoding Certificate failed: {source}"))?,
};
```

The same pattern appears in `parse_subnet_read_state_response` (lines 93-101). When `root_pk` is `None`, the function:
1. Deserializes the raw CBOR bytes into a `Certificate` struct without any signature check.
2. Looks up `request_status` or `subnet/metrics` paths in the unverified tree.
3. Returns the result to the caller as if it were certified state.

No error is returned; the caller receives a `RequestStatus` or `SubnetMetrics` value that may have been entirely fabricated by a malicious responder.

The analog to the mobile-wallet bug is exact:
- Mobile wallet: if `nonce` field is absent → skip decryption → parse unauthenticated payload.
- IC client library: if `root_pk` is `None` → skip BLS verification → parse unauthenticated certificate tree.

### Impact Explanation

Any IC infrastructure component (orchestrator, NNS delegation manager, recovery tooling, Rosetta, ledger synchronizer) that calls `parse_read_state_response` or `parse_subnet_read_state_response` with `root_pk = None` will silently accept a forged certificate from a malicious boundary node or a compromised/impersonated replica. The attacker can fabricate arbitrary `RequestStatus` values (e.g., reporting a call as `replied` with a crafted payload, or `rejected` to suppress a legitimate result) or arbitrary `SubnetMetrics`. Depending on the consuming component, this can lead to:
- Incorrect governance or ledger decisions based on spoofed state.
- Suppression or fabrication of canister call results visible to tooling.
- Incorrect subnet metrics influencing operational decisions.

### Likelihood Explanation

The `None` path is a first-class production code path in a public API, not gated by a feature flag or compile-time constant. The function signature makes it trivially easy for a caller to omit the root key (e.g., before the key is fetched, or in error-handling fallback paths). The `rs/rosetta-api/icp/ledger_canister_blocks_synchronizer/src/canister_access.rs` already shows a pattern where `root_key` is `Option<Vec<u8>>` and a warning is emitted when it is absent, indicating real production callers operate without a root key in some configurations. A network-level attacker positioned between the client and a boundary node (or a malicious boundary node itself) can serve a crafted CBOR response with no cryptographic material at all.

### Recommendation

Remove the `Option` wrapper from `root_pk` in both `parse_read_state_response` and `parse_subnet_read_state_response`. Certificate verification must be unconditional. Callers that do not yet have the root key should fetch it before calling these functions, not bypass verification. If a "parse-only, no-verify" path is genuinely needed for testing, it should be a separate function in a `#[cfg(test)]` module, mirroring the recommendation in the external report to split decryption from parsing.

### Proof of Concept

An attacker serving as a malicious boundary node responds to a `read_state` request with the following crafted CBOR (pseudocode):

```
HttpReadStateResponse {
  certificate: CBOR({
    tree: HashTree { /* fabricated tree with request_status/<id>/status = "replied", reply = <attacker_bytes> */ },
    signature: <omitted or zeroed>,
  })
}
```

A client calling:
```rust
parse_read_state_response(&request_id, &canister_id, None, cbor_response)
```
will return `Ok(RequestStatus { status: "replied", reply: Some(<attacker_bytes>), .. })` with no error, presenting the attacker-controlled payload as certified IC state. [1](#0-0) [2](#0-1)

pub fn parse_subnet_read_state_response(
    subnet_id: &SubnetId,
    root_pk: Option<&ThresholdSigPublicKey>,
    message: CBOR,
) -> Result<SubnetMetrics, String> {
    let response = serde_cbor::value::from_value::<HttpReadStateResponse>(message)
        .map_err(|source| format!("decoding to HttpReadStateResponse failed: {source}"))?;

    let certificate = match root_pk {
        Some(pk) => ic_certification::verify_certificate_for_subnet_read_state(
            &response.certificate,
            subnet_id,
            pk,
        )
        .map_err(|source| format!("verifying certificate failed: {source}"))?,
        None => serde_cbor::from_slice(response.certificate.as_slice())
            .map_err(|source| format!("decoding Certificate failed: {source}"))?,
    };
