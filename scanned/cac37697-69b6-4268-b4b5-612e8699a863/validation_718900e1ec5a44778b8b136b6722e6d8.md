### Title
Boundary Node DoS via CRLF in `method_name` Causing Panic in `postprocess_response` — (`rs/boundary_node/ic_boundary/src/http/middleware/process.rs`)

### Summary

An unprivileged attacker can send a CBOR-encoded IC API request with a `method_name` containing CRLF bytes (`\r\n`). The only validation applied is a length check. In `postprocess_response`, the truncated `method_name` is passed to `HeaderValue::from_maybe_shared(...).unwrap()`, which panics when the `http` crate's validator rejects the CRLF bytes, aborting the connection handler task. No panic-catching layer exists in the middleware stack.

---

### Finding Description

**Validation gap — `check_method_name_length` only checks length, not content:** [1](#0-0) 

The deserializer custom function rejects names longer than 20,000 bytes but performs zero content sanitization. A `method_name` of `"AAAAAAAAAA\r\nX-Evil: injected"` (30 bytes) passes unconditionally.

**Unsanitized value inserted into response header with `.unwrap()`:** [2](#0-1) 

`Bytes::from(truncated.to_string())` creates a fresh `Bytes` object from the attacker-controlled string. `HeaderValue::from_maybe_shared` validates every byte when given a fresh `Bytes`; bytes `\r` (0x0D) and `\n` (0x0A) are explicitly invalid per the HTTP/1.1 header value grammar enforced by the `http` crate. The function returns `Err(InvalidHeaderValue)`, and `.unwrap()` panics.

**No panic-catching layer in the middleware stack:** [3](#0-2) 

The `common_layers` `ServiceBuilder` chain includes no `CatchPanicLayer` or equivalent. A panic in `postprocess_response` propagates to the Tokio connection handler task, aborting it and dropping the connection.

---

### Impact Explanation

Each crafted request causes a task-level panic and connection drop on the boundary node. An attacker can send a continuous stream of such requests (no authentication required, no rate limiting on the `method_name` content path) to cause repeated panics across connection handler tasks. This degrades or denies service to legitimate users. Header injection is **not** achievable because `from_maybe_shared` rejects CRLF before any injection can occur — the actual impact is exclusively DoS via panic.

---

### Likelihood Explanation

- Requires no credentials, no privileged role, no key material.
- Entry point is any public IC API endpoint (`/api/v2/canister/.../call`, `/query`, `/read_state`).
- The CBOR body is trivially constructable; the CRLF must appear within the first `MAX_LOGGING_METHOD_NAME_LENGTH` bytes of `method_name`.
- No existing guard sanitizes or rejects non-printable bytes in `method_name` before the header insertion.
- Reproducible locally with a single crafted request.

---

### Recommendation

Replace the bare `.unwrap()` with explicit error handling. Either:

1. **Sanitize before insertion**: strip or percent-encode non-printable bytes from `method_name` before constructing the `HeaderValue`.
2. **Handle the error gracefully**: use `.ok()` or `if let Ok(v) = ... { response.headers_mut().insert(..., v); }` to silently skip insertion on invalid values rather than panicking.

The same pattern appears for `X_IC_ERROR_CAUSE` (line 203), `X_IC_CACHE_STATUS` (line 211), and `X_IC_CACHE_BYPASS_REASON` (line 217) — those values come from internal `enum`-derived strings and are safe in practice, but the `method_name` case is attacker-controlled and must be treated differently.

---

### Proof of Concept

```python
import cbor2, requests

# method_name with CRLF within first 50 bytes
method_name = "AAAAAAAAAA\r\nX-Evil: injected"

envelope = {
    "content": {
        "request_type": "call",
        "sender": bytes(1),           # anonymous principal
        "canister_id": bytes(b'\x00' * 10),
        "method_name": method_name,
        "arg": b'',
        "ingress_expiry": 9999999999999999999,
        "nonce": b'\x01',
    }
}

body = cbor2.dumps(envelope)
resp = requests.post(
    "https://<boundary-node>/api/v2/canister/aaaaa-aa/call",
    data=body,
    headers={"Content-Type": "application/cbor"},
)
# Expected: connection reset / 500 due to panic in postprocess_response
# (not a 4xx rejection, because the panic occurs after the inner handler runs)
print(resp.status_code)
```

The `method_name` passes `check_method_name_length` (30 < 20,000), is stored in `RequestContext`, and when `postprocess_response` calls `HeaderValue::from_maybe_shared(Bytes::from("AAAAAAAAAA\r\nX-Evil: injected")).unwrap()`, the `http` crate rejects the CRLF and the `.unwrap()` panics.

fn check_method_name_length<'de, D>(deserializer: D) -> Result<Option<String>, D::Error>
where
    D: Deserializer<'de>,
{
    let s: Option<String> = Option::<String>::deserialize(deserializer)?;
    if let Some(val) = &s
        && val.len() > MAX_METHOD_NAME_LENGTH
    {
        return Err(D::Error::custom(format!(
            "Method name exceeds maximum allowed length of {MAX_METHOD_NAME_LENGTH}"
        )));
    }

    Ok(s)
}
