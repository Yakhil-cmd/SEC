### Title
Unilateral `node_provider_principal_id` Transfer Without Recipient Consent — (`rs/registry/canister/src/mutations/do_update_node_operator_config_directly.rs`)

---

### Summary

`do_update_node_operator_config_directly` allows the current `node_provider_principal_id` to reassign that field to **any arbitrary principal** without the recipient's consent, without verifying the recipient is a registered node provider, and without governance approval. The described two-hop path is valid, but is actually unnecessary — a single call from NP-A directly to NP-C suffices.

---

### Finding Description

The sole authorization gate in `do_update_node_operator_config_directly_` is: [1](#0-0) 

```rust
// 2. Make sure that the caller is authorized to make the requested changes to node_operator_record.
if caller
    != PrincipalId::try_from(&node_operator_record.node_provider_principal_id).unwrap()
{
    return Err(format!(
        "Caller {caller} not equal to the node_provider_princpal_id for this record."
    ));
}
```

After passing this check, the new `node_provider_id` from the payload is written unconditionally: [2](#0-1) 

There is **no check** that:
- The new `node_provider_id` is a registered node provider
- The new `node_provider_id` has consented to the transfer
- Governance approval is required

The canister entry point explicitly comments `// This method can be called by anyone`: [3](#0-2) 

The two-hop attack works exactly as described:
1. NP-A (current `node_provider_principal_id`) calls with `node_provider_id=NP-B` → succeeds, NP-B is now the `node_provider_principal_id`.
2. NP-B calls with `node_provider_id=NP-C` → succeeds, NP-C is now the `node_provider_principal_id`.

But the simpler single-hop — NP-A directly calling with `node_provider_id=NP-C` — is equally valid and requires no intermediate identity.

---

### Impact Explanation

The `node_provider_principal_id` field in `NodeOperatorRecord` is the direct input to reward attribution. The node rewards canister groups rewardable nodes by this field: [4](#0-3) 

Setting it to NP-C without NP-C's knowledge redirects ICP node rewards for all nodes under that node operator to NP-C. NP-C **can** reclaim control (since they are now the `node_provider_principal_id` and can call the function themselves), but may be unaware the change occurred. The question's claim of "unauthorized neuron/governance access" is overstated — there is no direct neuron linkage — but the **reward redirection is a concrete, on-chain financial impact** with no governance oversight.

---

### Likelihood Explanation

Any registered node provider can execute this with one or two standard ingress update calls. The rate limit is 20 ops/day with a max spike of 140: [5](#0-4) 

The rate limit is charged to the **caller** (NP-A and NP-B), not to NP-C, and does not prevent the attack. No privileged access, key compromise, or governance majority is required.

---

### Recommendation

Route `node_provider_principal_id` changes through the governance proposal path (`do_update_node_operator_config`, which already requires `check_caller_is_governance_and_log`): [6](#0-5) 

Alternatively, implement a two-step accept pattern: the new `node_provider_id` must submit a separate confirmation call before the change is committed.

---

### Proof of Concept

**Single-hop (simpler):**
1. NP-A is the current `node_provider_principal_id` for node operator X.
2. NP-A calls `update_node_operator_config_directly({ node_operator_id: X, node_provider_id: NP-C })`.
3. Authorization passes (`caller NP-A == current node_provider_principal_id NP-A`).
4. NP-C is now the `node_provider_principal_id` for node operator X without their consent.
5. The node rewards canister now attributes all rewards for node operator X's nodes to NP-C.

**Two-hop (as described):**
1. NP-A → `node_provider_id=NP-B` (NP-A controls NP-B).
2. NP-B → `node_provider_id=NP-C`.
3. NP-C is now the `node_provider_principal_id`.

Both paths succeed because the only check is `caller == current node_provider_principal_id`, with no validation of the incoming `node_provider_id`.

**Correction to the question's invariant claim**: NP-C *can* reclaim control after the attack, since they are now the `node_provider_principal_id` and can call the function themselves. The invariant that is actually broken is the absence of recipient consent, not permanent loss of control.
