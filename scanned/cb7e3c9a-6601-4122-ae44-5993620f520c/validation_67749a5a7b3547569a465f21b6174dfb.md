### Title
SNS Token Supply Manipulation Inflates Treasury Valuation, Blocking `TransferSnsTreasuryFunds` and `MintSnsTokens` Proposals - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

---

### Summary

The SNS governance system computes the current price of SNS tokens on-chain at proposal submission time by reading the live `icrc1_total_supply` from the SNS ledger. Any unprivileged SNS token holder can burn their own tokens to artificially reduce the total supply, which inflates the computed `icps_per_sns_token`, which inflates the treasury valuation in XDR, which can push the treasury into a more restrictive rate-limit regime. This causes legitimate `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals to be rejected at submission time — a denial-of-service analogous to the `calcImpact()` manipulation described in the external report.

---

### Finding Description

`fetch_icps_per_sns_token()` in `rs/sns/governance/token_valuation/src/lib.rs` computes the current SNS token price as:

```
icps_per_sns_token = (1 / initial_sns_tokens_per_icp) / (current_supply / initial_supply)
```

It fetches three live values concurrently:
1. `sns_tokens_per_icp` from the Swap canister's `get_derived_state` (fixed at swap finalization)
2. `initial_supply_e8s` from the SNS ledger's first mint transactions (fixed at genesis)
3. `current_supply_e8s` from `icrc1_total_supply` (live, mutable) [1](#0-0) 

The computed `icps_per_sns_token` is then used by `assess_treasury_balance()` → `treasury_valuation_if_proposal_amount_is_small_enough_or_err()` to determine the maximum amount that can be transferred from the SNS treasury in a 7-day window: [2](#0-1) 

The rate-limit regime is determined by `ProposalsAmountTotalUpperBound::from_valuation_xdr()`:
- **Small** (< 100,000 XDR): `NoLimit` — any amount allowed
- **Medium** (100,000–1,200,000 XDR): 25% of treasury per 7 days
- **Large** (> 1,200,000 XDR): hard cap of 300,000 XDR per 7 days [3](#0-2) 

**The attack**: Any SNS token holder can call `icrc1_burn` on the SNS ledger to reduce `current_supply_e8s`. This reduces `total_inflation = current_supply / initial_supply`, which increases `icps_per_sns_token = initial_icps_per_sns_token / total_inflation`, which inflates `valuation_xdr = tokens × icps_per_token × xdrs_per_icp`. If the treasury is near the 1,200,000 XDR boundary, a targeted burn can push it from "medium" (25% cap) to "large" (300,000 XDR hard cap), causing any proposal that exceeds the new lower cap to be rejected. [4](#0-3) 

Critically, there is a `MIN_XDRS_PER_ICP` floor to prevent artificially low XDR prices from inflating the "small" regime, but **there is no corresponding `MAX_ICPS_PER_SNS_TOKEN` ceiling** to prevent artificially high token prices from inflating the "large" regime: [5](#0-4) 

The valuation is computed and stored at proposal **submission** time (in `ActionAuxiliary`), not at execution time: [6](#0-5) 

---

### Impact Explanation

An unprivileged SNS token holder can permanently suppress the 7-day treasury transfer allowance for `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals. If the SNS treasury is in the "medium" regime and the attacker burns enough tokens to push it into "large", the per-7-day cap drops from 25% of the treasury to a fixed 300,000 XDR. Any proposal exceeding this new cap is rejected at submission with the error:

> "Amount is too large. Within the past 7 days, a total of X tokens has already been executed in like proposals. Whereas, at most Y is allowed." [7](#0-6) 

This is a governance-level denial-of-service: legitimate SNS treasury operations are blocked without any privileged access.

---

### Likelihood Explanation

**Medium.** The attacker must hold and permanently burn SNS tokens (economic cost). However:
- SNS tokens are publicly acquirable via the SNS swap
- The burn operation (`icrc1_burn`) is a standard, unprivileged ledger call
- The attack is most effective when the treasury valuation is near the 1,200,000 XDR boundary — a condition that can be monitored on-chain
- The attacker does not need to front-run a specific transaction; they only need to burn before a proposal is submitted
- The effect is permanent (burned tokens cannot be recovered), making repeated attacks costly but a single well-timed attack sufficient to block a specific proposal

---

### Recommendation

Apply a symmetric clamp on `icps_per_sns_token` analogous to the existing `MIN_XDRS_PER_ICP` floor. Specifically, define a `MAX_ICPS_PER_SNS_TOKEN` ceiling in `fetch_icps_per_sns_token()` or in `ProposalsAmountTotalUpperBound::in_tokens()` to prevent the computed price from exceeding a realistic upper bound. This mirrors the existing protection: [8](#0-7) 

Alternatively, use a time-weighted average supply or a governance-set reference supply rather than the live `icrc1_total_supply` to compute the inflation factor, so that short-term supply manipulation does not affect the treasury rate limit.

---

### Proof of Concept

1. SNS treasury holds 500,000 SNS tokens. Initial swap price: 1 SNS = 0.5 ICP. ICP = 10 XDR. Initial supply: 10,000,000 SNS.
2. Current supply after legitimate minting: 12,000,000 SNS. `total_inflation = 1.2`. `icps_per_sns_token = (1/2) / 1.2 ≈ 0.417`. Treasury valuation: 500,000 × 0.417 × 10 ≈ 2,083,500 XDR → **large regime**, cap = 300,000 XDR/7 days.
3. SNS wants to transfer 400,000 XDR worth (≈ 95,900 SNS tokens). This exceeds the 300,000 XDR cap → proposal rejected.
4. Attacker burns 2,000,000 SNS tokens (their own). Current supply: 10,000,000. `total_inflation = 1.0`. `icps_per_sns_token = 0.5`. Treasury valuation: 500,000 × 0.5 × 10 = 2,500,000 XDR → still large, cap still 300,000 XDR.
5. More targeted: attacker burns enough to push supply below initial (e.g., to 8,000,000). `total_inflation = 0.8`. `icps_per_sns_token = 0.625`. Treasury valuation: 500,000 × 0.625 × 10 = 3,125,000 XDR → large, cap = 300,000 XDR. A proposal for 350,000 XDR worth is now blocked.
6. Without the burn (current supply 12,000,000, valuation 2,083,500 XDR), the same 350,000 XDR proposal would also be blocked. The attack is most impactful when the treasury is in the "medium" regime (100,000–1,200,000 XDR) and the attacker burns enough to push it into "large", reducing the cap from 25% of treasury to 300,000 XDR. async fn treasury_valuation_if_proposal_amount_is_small_enough_or_err<MyTokenProposalAction>(
    env: &dyn Environment,
    sns_ledger_canister_id: CanisterId,
    swap_canister_id: CanisterId,
    proposals: impl Iterator<Item = &ProposalData>,
    action: &MyTokenProposalAction,
) -> Result<Valuation, String>
where
    MyTokenProposalAction: TokenProposalAction,
{
    let spent_tokens = action.recent_amount_total_tokens(proposals, env.now())?;

    // Get valuation of the tokens in the treasury.
    let token = action.token()?;
    let valuation = assess_treasury_balance(
        token,
        env.canister_id(),
        sns_ledger_canister_id,
        swap_canister_id,
    )
    .await?;

    // From valuation, determine limit on the total from the past 7 days.
    let max_tokens = MyTokenProposalAction::recent_amount_total_upper_bound_tokens(&valuation)
        // Err is most likely a bug.
        .map_err(|treasury_limit_error| {
            format!("Unable to validate amount: {treasury_limit_error:?}",)
        })?;

    // Finally, inspect the proposal's amount: it must not exceed max - spent (remainder). Or if
    // you prefer, equivalently, amount + spent must be <= max.
    let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| {
        format!("Arithmetic error while performing {max_tokens} - {spent_tokens}",)
    })?;
    let proposal_amount_tokens = action.proposal_amount_tokens()?;
    if proposal_amount_tokens > allowance_remainder_tokens {
        // Although it might not be obvious to the user, their proposal is invalid, and we
        // consider it to be "their fault".
        return Err(format!(
            "Amount is too large. Within the past 7 days, a total of {spent_tokens} tokens has already \
             been executed in like proposals. Whereas, at most {max_tokens} is allowed. An additional \
             {proposal_amount_tokens} tokens from this proposal would cause that upper bound to be exceeded. \
             Maybe, try again in a few days?"
        ));
    }

    Ok(valuation)

    fn in_tokens(mut valuation: Valuation) -> Result<Decimal, ProposalsAmountTotalLimitError> {
        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))
                })?;

                // Calculate the inverse conversion rate.
                if xdrs_per_token == Decimal::from(0) {
                    // This is not reachable, because in this case, valuation.to_xdr() would return
                    // 0, and in that case, we would have taken the NoLimit branch.
                    return Err(ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
                }
                let tokens_per_xdr = xdrs_per_token.inv();

                max_xdr.checked_mul(tokens_per_xdr).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Max tokens could not be calculated with valuation: {valuation:?}",
                    ))
                })?
            }
        };

        Ok(result_tokens)
    }

    fn from_valuation_xdr(valuation_xdr: Decimal) -> Self {
        // Ideally, this would be checked at compile time. In principal, this should be possible,
        // since all the inputs are const, but I'm not sure how to do that. Therefore,
        // debug_assert_eq is used instead, and should be very nearly as good, because this will be
        // run during CI.
        debug_assert_eq!(
            Self::MAX_MEDIUM_TREASURY_SIZE_XDR.checked_mul(ONE_QUARTER),
            Some(Self::MAX_XDR),
        );

        if valuation_xdr <= Self::MAX_SMALL_TREASURY_SIZE_XDR {
            return Self::NoLimit;
        }

        if valuation_xdr <= Self::MAX_MEDIUM_TREASURY_SIZE_XDR {
            return Self::Fraction(ONE_QUARTER);
        }

        Self::Xdr(Self::MAX_XDR)
    }
