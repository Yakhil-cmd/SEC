### Title
Missing 7-Day Mint Amount Upper Bound Enforcement for `MintSnsTokens` Proposals - (`rs/sns/governance/src/proposal.rs`)

### Summary

The SNS governance canister intentionally defines a 7-day rolling upper bound on how many tokens can be minted via `MintSnsTokens` proposals (analogous to the limit already enforced for `TransferSnsTreasuryFunds`). However, the enforcement of this limit is explicitly commented out and replaced with `Decimal::MAX` (i.e., no limit), meaning any governance-approved `MintSnsTokens` proposal can mint an unbounded number of SNS tokens regardless of the treasury valuation.

### Finding Description

The `TokenProposalAction` trait defines `recent_amount_total_upper_bound_tokens` to cap the 7-day rolling total of token-minting or treasury-transfer proposals. For `TransferSnsTreasuryFunds`, this is fully enforced: the real bound from `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` is used both at proposal submission time and at execution time.

For `MintSnsTokens`, the correct implementation exists in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` as `mint_sns_tokens_7_day_total_upper_bound_tokens`, and the correct `impl TokenProposalAction for MintSnsTokens` block is written and commented out with `TODO(NNS1-2982): Uncomment`. In its place, a stub returns `Decimal::MAX`:

```rust
// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    Ok(Decimal::MAX)
}
```

This stub is used inside `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which is called from `validate_and_render_mint_sns_tokens` at proposal submission. Because `Decimal::MAX` is always returned as the limit, the check `if proposal_amount_tokens > allowance_remainder_tokens` never triggers for `MintSnsTokens`.

Additionally, unlike `TransferSnsTreasuryFunds`, the execution path for `MintSnsTokens` in `perform_mint_sns_tokens` performs **no secondary check** at execution time — it directly calls `self.ledger.transfer_funds(amount_e8s, ...)` with no limit validation. The `TransferSnsTreasuryFunds` path, by contrast, calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` before executing.

The integration test `sns_can_mint_funds_via_proposals` explicitly documents this: the second proposal that should be rejected by the minting limit is instead allowed to succeed, with the assertion commented out and replaced by `.unwrap()`.

### Impact Explanation

Any SNS community that passes a `MintSnsTokens` governance proposal can mint an arbitrarily large number of SNS tokens — far beyond the treasury-valuation-based 7-day cap that was designed to protect token holders from rapid dilution. A whale neuron holder (or a coalition) that controls enough voting power can submit and pass a proposal to mint tokens exceeding the intended limit (e.g., 25% of treasury value for medium-sized treasuries, or 300,000 XDR cap for large ones), causing unbounded inflation of the SNS token supply. This directly harms all existing SNS token holders through dilution.

### Likelihood Explanation

The entry path is a standard SNS governance proposal submission (`make_proposal`) followed by a vote. Any principal with sufficient neuron voting power — a realistic scenario for SNS DAOs with concentrated stake — can trigger this. The code explicitly acknowledges the missing enforcement with `TODO(NNS1-2982)` comments and the integration test confirms the bypass is live in production.

### Recommendation

1. Uncomment the correct `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` in `rs/sns/governance/src/proposal.rs` and remove the `Decimal::MAX` stub.
2. Also uncomment the import of `mint_sns_tokens_7_day_total_upper_bound_tokens` at the top of the file.
3. Add an execution-time check in `perform_mint_sns_tokens` analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, using the `MintSnsTokensActionAuxiliary` valuation stored in `action_auxiliary`.
4. Uncomment the integration test assertions in `sns_can_mint_funds_via_proposals`.

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
