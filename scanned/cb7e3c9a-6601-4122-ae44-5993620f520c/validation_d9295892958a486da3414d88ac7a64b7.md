### Title
Incorrect Accounting in `calculate_rewards_v0` Due to Lexicographic Node Operator Ordering Corrupting Type-3 Reward Decay Sequence - (File: `rs/registry/node_provider_rewards/src/lib.rs`)

### Summary
The `calculate_rewards_v0` function in `rs/registry/node_provider_rewards/src/lib.rs` iterates over node operator records in **lexicographic key order** rather than insertion/registration order. For type-3 nodes, the running decay coefficient (`np_coeff`) is accumulated across data centers within the same country. Because the iteration order is determined by the lexicographic sort of node operator registry keys — not by the chronological order in which node operators were registered — the decay sequence applied to each node operator's type-3 nodes is non-deterministic from the node provider's perspective and can be manipulated by a node provider who controls the naming of their node operator records.

### Finding Description

In `calculate_rewards_v0`, the outer loop iterates over `node_operators` in the order they are passed in, which is the lexicographic order of their registry keys:

```rust
for (key_string, node_operator) in node_operators.iter() {
    ...
    for (node_type, node_count) in node_operator.rewardable_nodes.iter() {
        ...
        t if t.starts_with("type3") => {
            let mut np_coeff = *np_coefficients.get(&np_coefficients_key).unwrap_or(&1.0);
            ...
            for i in 0..*node_count {
                let node_reward = (reward_base * np_coeff) as u64;
                dc_reward += node_reward;
                np_coeff *= dc_reward_coefficient_percent;
            }
            np_coefficients.insert(np_coefficients_key, np_coeff);
```

The `np_coeff` state is shared across all node operators belonging to the same node provider in the same country. The order in which node operators are processed determines which data center gets the higher (earlier, less-decayed) coefficient and which gets the lower (later, more-decayed) coefficient. Since the order is lexicographic by registry key string, a node provider who registers node operator `"aaa_dc_high_reward"` before `"zzz_dc_low_reward"` will have `aaa` processed first (receiving the full coefficient) and `zzz` processed second (receiving the decayed coefficient) — regardless of which was actually registered first on-chain.

The code itself acknowledges this as a known issue in a comment at lines 85–99, but it is not merely a fairness concern: it constitutes an **exploitable accounting error** where a node provider can systematically inflate their total monthly XDR reward by choosing node operator key names that cause their higher-reward-rate data centers to be processed first in the decay sequence.

The analog to the `LockingController.applyLosses` bug is exact: in both cases, a running accumulator (the decay coefficient `np_coeff` here; the denominator `_globalReceiptToken` there) is mutated inside a loop and the mutated value is used for subsequent iterations, causing the total allocation to diverge from the intended amount. Here, the order of mutation is attacker-influenced via key naming.

### Impact Explanation

A node provider who controls the string names of their node operator registry keys can arrange for their highest-reward-rate data centers to be processed first in the decay sequence, maximizing their total monthly ICP reward. Conversely, a node provider whose keys happen to sort in the "wrong" order receives less than the intended reward. This corrupts the ledger-conservation invariant of the node provider reward system: the total ICP minted for node rewards diverges from what the governance-approved reward table intends.

The `calculate_rewards_v0` result is used directly by `Registry::get_node_providers_monthly_xdr_rewards`, which feeds into `Governance::get_node_providers_rewards` and ultimately into the ICP minting path for node provider rewards. Incorrect totals here result in incorrect ICP minted on-chain.

### Likelihood Explanation

Node providers register node operators via governance proposals (`do_add_node_operator`). The registry key for a node operator is derived from the node operator's principal ID, which is chosen by the node provider. A node provider with multiple data centers in the same country can choose principal IDs whose string representations sort in a favorable lexicographic order, causing their high-reward DCs to be processed first. This requires no privileged access beyond what a legitimate node provider already has.

### Recommendation

Replace the lexicographic iteration order with a deterministic, registration-time-ordered iteration, or — more robustly — compute the total type-3 reward for a node provider in a country as a function of the **set** of nodes (independent of processing order), as is done in the newer `RewardsCalculationV1`/`V2` algorithms in `rs/node_rewards/rewards_calculation/`. The `calculate_rewards_v0` path used by the registry canister should either be deprecated in favor of the order-independent algorithm, or the iteration should be sorted by a canonical, non-manipulable key (e.g., node operator principal ID bytes, not the registry key string).

### Proof of Concept

Consider a node provider with two node operators in the same country:
- Node operator key `"aaa_op"`: DC with `reward_base = 22_000_000`, `reward_coefficient_percent = 70`, 1 node
- Node operator key `"zzz_op"`: DC with `reward_base = 10_000_000`, `reward_coefficient_percent = 70`, 1 node

**Lexicographic order (aaa first):**
- `aaa_op`: `np_coeff = 1.0` → reward = 22,000,000; `np_coeff` becomes 0.7
- `zzz_op`: `np_coeff = 0.7` → reward = 7,000,000
- **Total = 29,000,000**

**Reverse order (zzz first):**
- `zzz_op`: `np_coeff = 1.0` → reward = 10,000,000; `np_coeff` becomes 0.7
- `aaa_op`: `np_coeff = 0.7` → reward = 15,400,000
- **Total = 25,400,000**

The same node provider with the same nodes receives **3,600,000 XDR more** simply by choosing node operator key names that sort favorably. By naming node operators so that the highest-`reward_base` DC sorts first, a node provider maximizes their reward beyond what the governance-approved table intends.
