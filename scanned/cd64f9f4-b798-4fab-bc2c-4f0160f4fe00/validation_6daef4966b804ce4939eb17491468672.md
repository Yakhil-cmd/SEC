### Title
Missing Memory Size Bounds Validation on Sandbox-Provided ExecutionFinishedRequest Allows Memory Accounting Corruption — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

### Summary
A compromised canister sandbox process can send a forged `ExecutionFinishedRequest` with an arbitrarily large `wasm_memory.size` or `stable_memory.size`. The replica controller applies these values directly to the canister's execution state without validating them against the canister's allocated memory limit. The only guard (`verify_size()`) checks the wrong direction and passes for inflated values. The developers explicitly acknowledge this gap in a TODO comment at the exact location.

### Finding Description
In `update_execution_state` (sandboxed_execution_controller.rs:1779, 1800), the replica unconditionally assigns:

```rust
wasm_memory.size = execution_state_modifications.wasm_memory.size;
stable_memory.size = execution_state_modifications.stable_memory.size;
```

The subsequent `verify_size()` call only asserts `page_map_bytes <= memory_bytes` — i.e., that the page map does not exceed the declared size. An inflated `size` trivially satisfies this (the page map is smaller than the inflated declared size), so the error path is never triggered. The inflated `Memory` struct is then returned in `CanisterStateChanges` and applied unconditionally in `apply_canister_state_changes` (common.rs:588-589).

Contrast this with the instructions path (line 1713-1724), where the code explicitly comments *"If sandbox is compromised this value could be larger than the initial limit"* and clamps the value. No equivalent clamp exists for memory size, despite the TODO at line 1772-1774 calling for exactly this.

### Impact Explanation
- The replica records an inflated `wasm_memory.size` / `stable_memory.size` for the canister in its replicated state.
- Storage cost accounting (`wasm_memory_usage()`) uses this size field, so the canister can manipulate its apparent memory footprint.
- The `update_on_low_wasm_memory_hook_status` call (common.rs:630) uses the now-inflated size, corrupting hook scheduling.
- Subnet-level memory invariants that depend on per-canister memory sizes become inconsistent.
- A canister could inflate its recorded size to `NumWasmPages::MAX`, making the subnet believe it has consumed all available memory, starving other canisters of memory allocation.

### Likelihood Explanation
The precondition is code execution inside a canister sandbox process, which requires exploiting a Wasm engine (wasmtime) vulnerability. This is a non-trivial prerequisite. However:
- The IC sandbox is explicitly designed as a security boundary against exactly this scenario.
- The developer TODO and the instructions-guard comment both confirm this threat model is intended to be defended against.
- The asymmetry (instructions are guarded, memory is not) is a concrete, auditable gap — not a theoretical one.

### Recommendation
Apply the same pattern used for instructions: after receiving `execution_state_modifications`, clamp `wasm_memory.size` and `stable_memory.size` to the maximum allowed by the canister's memory allocation (pre-execution limit). Specifically, replace the TODO comment with an actual bounds check:

```rust
// Guard against compromised sandbox inflating memory size.
let max_wasm_pages = /* canister's allocated wasm memory limit in pages */;
if execution_state_modifications.wasm_memory.size > max_wasm_pages {
    // log critical error, clamp or reject
}
```

### Proof of Concept
1. Deploy a canister that exploits a wasmtime sandbox escape.
2. From within the sandbox process, forge an `ExecutionFinishedRequest` with `exec_output.state_modifications.wasm_memory.size = NumWasmPages::MAX`.
3. Send it over the controller IPC socket.
4. `ControllerServiceImpl::execution_finished` (controller_service_impl.rs:48-80) passes it to the registered completion closure without inspection.
5. `update_execution_state` assigns the inflated size; `verify_size()` passes (page_map_bytes < inflated memory_bytes).
6. `apply_canister_state_changes` writes the inflated `wasm_memory` into `execution_state`.
7. Assert that `execution_state.wasm_memory.size == NumWasmPages::MAX` — memory accounting is now corrupted.

    pub fn verify_size(&self) -> Result<(), String> {
        let page_map_bytes = self.page_map.num_host_pages() * PAGE_SIZE;
        let memory_bytes = self.size.get() * WASM_PAGE_SIZE_IN_BYTES;
        if page_map_bytes <= memory_bytes {
            Ok(())
        } else {
            Err(format!(
                "The page map size {page_map_bytes} exceeds the memory size {memory_bytes}"
            ))
        }
    }
