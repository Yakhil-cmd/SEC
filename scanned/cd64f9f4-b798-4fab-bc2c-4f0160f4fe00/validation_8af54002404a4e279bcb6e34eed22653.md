### Title
Unverified `buyer` Parameter in `refresh_buyer_tokens` Allows Bypassing Swap Participation Consent Mechanism - (File: rs/sns/swap/canister/canister.rs)

---

### Summary

The `refresh_buyer_tokens` update method in the SNS Swap canister accepts an arbitrary `buyer` principal in its request payload without verifying it against the actual `ic_cdk::caller()`. The `confirmation_text` field — the sole consent mechanism that proves a participant explicitly agreed to the swap's terms — is validated only against the swap's global init text, not against the identity of the actual caller. Any unprivileged ingress sender who knows the public confirmation text can call `refresh_buyer_tokens` with a victim's principal as `buyer`, bypassing the victim's consent and forcibly committing the victim's ICP into the swap.

---

### Finding Description

In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` handler resolves the effective buyer principal from the request payload without any caller-identity check:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()   // ← no check that arg.buyer == caller()
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

The resolved principal `p` is then passed directly into `refresh_buyer_token_e8s`, which:

1. Validates the `confirmation_text` against the swap's **global** init text — not against the actual caller:

```rust
pub fn validate_confirmation_text(
    &self,
    confirmation_text: Option<String>,
) -> Result<(), String> {
    match (self.init_or_panic().confirmation_text.as_ref(), confirmation_text) {
        (Some(expected_text), Some(text)) => {
            if &text != expected_text { Err(...) } else { Ok(()) }
        }
        ...
    }
}
``` [2](#0-1) 

2. Checks the ICP balance of `buyer`'s subaccount and credits that balance to `buyer`'s `BuyerState`: [3](#0-2) 

3. Inserts or updates the `buyer` entry in the `self.buyers` map, locking the ICP into the swap: [4](#0-3) 

The `confirmation_text` is an **optional consent field** defined in the SNS init payload, described as "an optional text that swap participants should confirm before they may participate in the swap": [5](#0-4) 

The `RefreshBuyerTokensRequest` proto explicitly documents that `buyer` defaults to the caller if unset, implying third-party calls are possible: [6](#0-5) 

Because the confirmation text is stored in the swap's public `Init` state (readable by anyone via `get_state`), any attacker can read it and supply it on behalf of a victim.

---

### Impact Explanation

**Consent bypass / forced swap participation.** The `confirmation_text` mechanism exists to ensure each participant explicitly agrees to the swap's legal or operational terms before their ICP is locked. Because `refresh_buyer_tokens` does not verify that `arg.buyer == caller()`, an attacker can:

1. Read the swap's `confirmation_text` from the public `get_state` query.
2. Identify a victim who has already transferred ICP to the swap canister's subaccount (e.g., a user who sent ICP but has not yet called `refresh_buyer_tokens` because they have not agreed to the terms).
3. Call `refresh_buyer_tokens { buyer: victim_principal, confirmation_text: <known_text> }` from any identity.
4. The swap canister credits the victim's ICP balance into `self.buyers`, locking the victim into the swap without their consent.

Once committed, the victim's ICP is held by the swap canister until finalization. If the swap commits, the victim receives SNS tokens and loses their ICP — an outcome they never agreed to. The `confirmation_text` consent mechanism is rendered entirely ineffective.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no key material, and no social engineering. The only prerequisites are:

- The swap has a non-empty `confirmation_text` (a common configuration for legally-sensitive SNS launches).
- A victim has transferred ICP to the swap subaccount but has not yet called `refresh_buyer_tokens` (a normal intermediate state in the participation flow).
- The attacker can read the public `confirmation_text` from `get_state`.

All three conditions are routinely satisfied during active SNS swaps. The call is a standard ingress update message callable by any anonymous or authenticated principal.

---

### Recommendation

When `arg.buyer` is non-empty and differs from `caller()`, the canister should either:

1. **Reject the call outright** — require `arg.buyer == caller()` unless the caller is a whitelisted trusted intermediary (analogous to the ZealousSwap fix):

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Caller must match the specified buyer");
        }
        buyer
    };
    ...
}
```

2. **Or**, if third-party notification is intentionally supported (e.g., a frontend canister notifying on behalf of a user), strip the `confirmation_text` requirement when the caller is not the buyer — since the consent must come from the buyer, not a third party.

---

### Proof of Concept

1. Alice transfers 10 ICP to the SNS swap canister's subaccount for her principal (`swap_canister_subaccount(alice_principal)`), intending to review the confirmation text before committing.
2. Attacker Bob queries `get_state` on the swap canister and reads `init.confirmation_text = "I agree to the SNS swap terms"`.
3. Bob sends an ingress update:
   ```
   refresh_buyer_tokens({
     buyer: alice_principal_text,
     confirmation_text: Some("I agree to the SNS swap terms")
   })
   ```
4. The swap canister resolves `p = alice_principal`, validates the confirmation text against the global init text (passes), reads Alice's subaccount balance (10 ICP), and inserts `alice_principal → BuyerState { amount_icp_e8s: 10_0000_0000 }` into `self.buyers`.
5. Alice's ICP is now locked in the swap. She never provided her consent. When the swap finalizes, her ICP is swept to the SNS governance canister and she receives SNS tokens she never agreed to purchase. [1](#0-0) [7](#0-6) [4](#0-3)
