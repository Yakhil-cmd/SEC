### Title
SNS Governance Child Neuron Inherits All Parent Permissions Including Third-Party Permissions After Split - (File: rs/sns/governance/src/governance.rs)

### Summary

When an SNS neuron is split via `ManageNeuron::Split`, the child neuron unconditionally inherits the **entire permissions list** from the parent neuron, including any permissions previously granted to third-party principals. If the parent neuron had a malicious or compromised principal with `Disburse` or `ManagePrincipals` permission, that principal retains identical permissions on the child neuron. When the child neuron is subsequently "transferred" to a new owner (by removing the splitter's permissions and adding the new owner's), the third-party permissions silently persist, allowing the old third party to drain or hijack the child neuron.

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `split_neuron` function constructs the child neuron by cloning the parent's full `permissions` field:

```rust
let child_neuron = Neuron {
    id: Some(child_nid.clone()),
    permissions: parent_neuron.permissions.clone(),  // ALL permissions copied verbatim
    ...
};
```

This is explicitly documented in the proto as intended behavior:

> "The child neuron inherits from the parent neuron the permissions (i.e., principals that can change the neuron), the age, the followees, and the dissolve state."

However, the permissions list may contain principals that were granted access by a **previous `ManagePrincipals` holder** — not the current neuron owner. The current owner may be unaware of these third-party entries. When the current owner later "transfers" the child neuron to a new owner (by removing their own permissions and adding the new owner's via `AddNeuronPermissions`/`RemoveNeuronPermissions`), the third-party permissions are not cleared and remain fully active on the child neuron.

**Attack scenario:**
1. Alice has a neuron with `ManagePrincipals` permission.
2. Alice grants Bob `ManagePrincipals`. Bob then grants Mallory `Disburse` permission.
3. Alice removes Bob. Alice is now unaware of Mallory's `Disburse` entry.
4. Alice splits the neuron — the child neuron inherits Mallory's `Disburse` permission.
5. Alice "transfers" the child neuron to Carol by removing Alice's own permissions and adding Carol's.
6. Mallory still holds `Disburse` on the child neuron and can drain Carol's staked tokens at any time.

### Impact Explanation

A principal holding `Disburse` permission can call `ManageNeuron::Disburse` to transfer the neuron's entire staked token balance to an arbitrary ledger account once the neuron is dissolved. A principal holding `ManagePrincipals` can add themselves back with full permissions, effectively taking over the neuron. The new owner (Carol) suffers direct financial loss — loss of staked SNS governance tokens — with no on-chain mechanism to detect the residual permission before accepting the neuron.

### Likelihood Explanation

SNS neurons are increasingly used in multi-party setups (DAOs, investment clubs, delegated voting services) where `ManagePrincipals` is shared. The `Split` operation is a routine neuron management action. The victim must trust the transferring party without independently auditing the full permissions list. While permissions are readable on-chain via `get_neuron`, there is no protocol-level warning or automatic cleanup when a neuron is split or when permissions are restructured to simulate a transfer. The likelihood is low-to-medium, matching the LUKSO M-06 assessment.

### Recommendation

1. **Warn at split time**: When `split_neuron` is called, emit a log or return metadata indicating that the child neuron has inherited N permissions from the parent, prompting the caller to review them.
2. **Provide a "clean split" option**: Allow the caller to specify that the child neuron should be initialized with only the caller's own permissions (i.e., only the principal invoking `Split` receives the default claimer permissions on the child), rather than inheriting the full parent permission set.
3. **Document the risk explicitly**: The proto and API documentation should warn that splitting a neuron propagates all third-party permissions to the child, and that any "transfer" of the child neuron requires manually auditing and revoking inherited permissions.
