### Title
Unbounded Per-RPC Response Allocation Allows Byzantine Peer to Exhaust Victim Replica Memory — (`rs/p2p/quic_transport/src/connection_handle.rs`)

### Summary

A Byzantine authenticated subnet node acting as an RPC responder can send maximally-sized responses (up to `MAX_MESSAGE_SIZE_BYTES = 128 MiB`) to every concurrent artifact-fetch RPC initiated by a victim replica. Because `read_to_end(MAX_MESSAGE_SIZE_BYTES)` allocates up to 128 MiB per in-flight RPC, and the number of concurrent RPCs is bounded only by the slot table (which is `SLOT_TABLE_NO_LIMIT` for consensus/DKG/IDKG artifact types), a single Byzantine peer can drive the victim replica into OOM.

### Finding Description

**Exact allocation site:** [1](#0-0) 

`recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)` allocates a `Vec<u8>` that grows up to the limit as bytes arrive. The limit is: [2](#0-1) 

128 MiB, intentionally large to accommodate large consensus summary blocks.

**Concurrency source — one Tokio task per distinct artifact ID:** [3](#0-2) 

Each new artifact ID seen in the slot table spawns a `process_slot_update` task. Each such task calls `assemble_message` → `download_artifact` → `transport.rpc()`, which reaches the `read_to_end` call above. These tasks run concurrently with no aggregate memory cap.

**Slot table limit is `SLOT_TABLE_NO_LIMIT` for consensus/DKG/IDKG:** [4](#0-3) 

A Byzantine peer can advertise an unbounded number of distinct artifact IDs, each triggering a new concurrent RPC task.

**Artifact download loop — one RPC per task, no concurrency throttle:** [5](#0-4) 

Each task issues one `transport.rpc()` at a time in a retry loop. With N concurrent tasks, N simultaneous `read_to_end` allocations of up to 128 MiB each are live.

### Impact Explanation

- **Memory exhaustion**: N concurrent RPCs × 128 MiB = N × 128 MiB heap pressure on the victim replica. With `SLOT_TABLE_NO_LIMIT`, N is unbounded from a single Byzantine peer.
- **OOM kill**: Linux OOM killer terminates the replica process, halting its participation in consensus.
- **Subnet liveness**: If multiple replicas are targeted simultaneously (one Byzantine peer per victim), subnet liveness can be degraded below the fault threshold.

### Likelihood Explanation

- The attacker is a Byzantine subnet node — a valid adversary in the IC threat model (below f < n/3 fault threshold).
- The attack requires no privileged access beyond subnet membership.
- The attack path is direct: advertise many artifact IDs → victim spawns concurrent RPC tasks → Byzantine peer responds with 128 MiB bodies → victim OOMs.
- No existing guard prevents this: there is no per-peer RPC concurrency limit, no aggregate response-bytes budget, and no backpressure from the slot table for `SLOT_TABLE_NO_LIMIT` artifact types.

### Recommendation

1. **Add a per-peer outbound RPC concurrency semaphore** in `ConnectionHandle` or at the `FetchArtifact` layer, limiting simultaneous in-flight RPCs to a small constant (e.g., 4–8 per peer).
2. **Reduce `MAX_MESSAGE_SIZE_BYTES` for the response path** or use a separate, smaller limit for artifact-fetch responses, distinct from the request-receive path.
3. **Apply `SLOT_TABLE_NO_LIMIT` only where truly necessary** and set a reasonable per-peer slot cap for consensus artifact types to bound the number of concurrent assembler tasks.
4. **Stream-and-validate** responses rather than buffering the entire body before decoding, so memory is released incrementally.

### Proof of Concept

```
1. Byzantine node B joins subnet S.
2. B advertises N distinct (but invalid) consensus artifact IDs to victim V
   via the slot-update push path (no slot limit for consensus type).
3. V spawns N concurrent process_slot_update tasks, each calling transport.rpc(&B, ...).
4. B's rpc_handler responds to each RPC by streaming exactly 128 MiB of zeros
   before finishing the QUIC send stream.
5. V's read_to_end(MAX_MESSAGE_SIZE_BYTES) on each of the N recv_streams
   allocates N × 128 MiB simultaneously.
6. With N = 32, peak heap = 4 GiB; typical replica memory budget is 4–8 GiB.
7. OOM killer terminates the replica process.
```

Fuzz-test variant: instrument a turmoil-based test (similar to `test_sending_large_message` in `rs/p2p/quic_transport/tests/test.rs`) with a Byzantine responder that always returns a 128 MiB body, and measure peak RSS with N = 10, 20, 32 concurrent artifact IDs. [6](#0-5)
