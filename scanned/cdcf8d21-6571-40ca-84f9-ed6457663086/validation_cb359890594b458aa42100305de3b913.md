### Title
Metrics Endpoint Leaks Count of Confidential Rate-Limit Rules and Incidents to Anonymous Callers — (`rs/boundary_node/rate_limits/canister/metrics.rs`, `canister.rs`)

---

### Summary

The `/metrics` HTTP endpoint of the rate-limits canister is publicly accessible with no caller authentication. It exposes Prometheus gauges that are recomputed directly from raw canister state, including `active_rules_count` and `stored_incidents_count`, which count **all** rules and incidents regardless of their confidentiality status. This allows any anonymous caller to observe when new confidential rate-limit rules or incidents are added, violating the system's explicit confidentiality invariant.

---

### Finding Description

The `http_request` query handler routes `/metrics` to `export_metrics_as_http_response` with no access-level check: [1](#0-0) 

`export_metrics_as_http_response` calls `recompute_metrics`, which reads raw state counts and sets gauges: [2](#0-1) 

The two critical state reads are:

- `active_rules_count()` — returns `value.rule_ids.len()` from the **latest config**, counting every rule regardless of `disclosed_at`: [3](#0-2) 

- `incidents_count()` — returns the length of the entire incidents map, including undisclosed incidents: [4](#0-3) 

The confidentiality formatters (`ConfigConfidentialityFormatter`, `RuleConfidentialityFormatter`) only redact the **content** (`rule_raw`, `description`) of non-disclosed rules in `get_config` / `get_rule_by_id` responses: [5](#0-4) 

They have no effect on the metrics path. The metrics are also registered and named explicitly: [6](#0-5) 

Additionally, `get_config` (the only other path that would expose version/count information) is gated by `inspect_message` to API boundary nodes and the authorized principal only: [7](#0-6) 

The `/metrics` endpoint has no equivalent gate.

---

### Impact Explanation

An anonymous caller polling `/metrics` periodically observes:
- `active_rules_count` — total rules in the active config (including confidential ones)
- `stored_incidents_count` — total incidents ever created (including confidential ones)
- `active_config_version` — increments on every `add_config` call
- `stored_configs_count` — total configs stored

When a privileged operator deploys a new confidential rule in response to an ongoing attack, all four gauges change simultaneously and are immediately visible to the attacker. The attacker learns: (a) a new rule was deployed, (b) approximately when, and (c) how many rules are now active. This allows the attacker to detect that their traffic pattern was flagged and rules were deployed against them, enabling them to pause, adapt, or time their attack around rule deployment windows.

The confidentiality system's stated purpose is to prevent exactly this: the existence and count of undisclosed rules must not be observable by unprivileged callers.

---

### Likelihood Explanation

The exploit requires only an unauthenticated HTTP query call to a public canister endpoint — no special access, no keys, no social engineering. Any caller on the internet can poll `GET /metrics` repeatedly. The canister is deployed on the IC mainnet and the endpoint is reachable via `https://<canister-id>.raw.ic0.app/metrics`.

---

### Recommendation

Filter the metrics exposed at `/metrics` to exclude counts that reflect confidential state. Specifically:

1. Replace `active_rules_count` with a count of only **disclosed** rules in the active config.
2. Replace `stored_incidents_count` with a count of only **disclosed** incidents.
3. Either remove `active_config_version` from the public metrics or replace it with a "disclosed config version" counter.
4. Alternatively, restrict the `/metrics` endpoint to callers with `FullAccess` or `FullRead` access level, consistent with how `get_config` is gated.

---

### Proof of Concept

State-machine test (no mainnet access required):

1. Initialize the canister with an authorized principal.
2. Call `add_config` with a confidential rule (no `disclosed_at` set) as the authorized principal.
3. Call `http_request { method: "GET", url: "/metrics", ... }` as the anonymous principal.
4. Decode the response body and assert that `active_rules_count 1` and `stored_incidents_count 1` appear in the Prometheus text output.
5. The test passes — confirming that the count of confidential, undisclosed rules is visible to anonymous callers via the metrics endpoint.

pub fn recompute_metrics(canister_api: impl CanisterApi) {
    METRICS.with(|cell| {
        let mut cell = cell.borrow_mut();

        let memory = (ic_cdk::stable::stable_size() * WASM_PAGE_SIZE_IN_BYTES) as f64;

        cell.stable_memory_size.borrow_mut().set(memory);
        cell.api_boundary_nodes_count

        cell.active_version

        cell.incidents_count
            .set(canister_api.incidents_count() as i64);
        cell.active_rate_limit_rules_count

impl ConfidentialityFormatting for ConfigConfidentialityFormatter {
    type Input = OutputConfig;

    fn format(&self, config: OutputConfig) -> OutputConfig {
        let mut config = config;
        config.is_redacted = true;
        // Redact (hide) fields of non-disclosed rules
        config.rules.iter_mut().for_each(|rule| {
            if rule.disclosed_at.is_none() {
                rule.description = None;
                rule.rule_raw = None;
            }
        });
        config
    }
