### Title
Rounding-Induced Permanent Token Lock in SNS Swap Finalization - (File: rs/sns/swap/src/swap.rs)

### Summary
The SNS Swap canister uses integer floor division in `Swap::scale()` to compute each participant's SNS token allocation. Because each participant's allocation is independently floored, the sum of all allocations is always strictly less than `sns_token_e8s` whenever the ICP contributions do not divide evenly. The resulting "change" tokens remain permanently locked in the swap canister's SNS ledger account after finalization, with no recovery path.

### Finding Description

**Root cause — `Swap::scale()`:** [1](#0-0) 

The function computes `floor(amount_icp_e8s * total_sns_e8s / total_icp_e8s)` in 128-bit integer space. For each participant `i`, the rounding error is `(amount_icp_e8s_i * total_sns_e8s) % total_icp_e8s`, which is in `[0, total_icp_e8s - 1]`. Dividing by `total_icp_e8s` gives a per-participant loss of up to 1 e8 (the smallest token unit).

**Accumulation in `create_sns_neuron_recipes()`:** [2](#0-1) [3](#0-2) 

The code itself logs the "change" (`sns_being_offered_e8s - total_sns_tokens_sold_e8s`) but takes no action on it. No neuron recipe is created for the remainder, so it is never included in `sweep_sns`.

**No recovery in `finalize_inner()`:** [4](#0-3) 

After `sweep_sns` completes, the finalization sequence ends. There is no step that transfers the leftover SNS balance from the swap canister's account to the SNS treasury or governance canister.

**Exploit path:**
1. Any unprivileged user calls `refresh_buyer_token_e8s` to participate in an SNS swap.
2. The swap commits and `finalize` is called (permissionless).
3. `create_sns_neuron_recipes` computes per-participant allocations via `Swap::scale()`, accumulating rounding dust.
4. `sweep_sns` distributes only the allocated amounts; the remainder stays in the swap canister's SNS ledger account.
5. The swap canister has no `transfer` endpoint for these residual tokens; they are permanently inaccessible.

### Impact Explanation

SNS tokens are permanently locked in the swap canister's SNS ledger account after every committed swap. The locked amount is bounded by `(number_of_participants - 1)` e8s (at most 1 e8 lost per participant per the floor division). For a swap with 10,000 participants, up to 10,000 e8s (0.0001 SNS tokens at 8 decimals) are permanently lost. For high-value SNS tokens or large participant counts, this is a non-trivial ledger conservation violation: tokens are minted into the swap canister at genesis but can never be recovered or redistributed.

### Likelihood Explanation

This occurs in **every** SNS swap with more than one participant where `amount_icp_e8s_i * total_sns_e8s` is not perfectly divisible by `total_icp_e8s`. In practice, this is virtually every real swap. The effect is deterministic and reproducible. Any user who participates in a swap triggers this path.

### Recommendation

After `sweep_sns` completes, transfer any remaining SNS token balance from the swap canister's account to the SNS governance treasury account. Concretely, after the `sweep_sns` call in `finalize_inner`, query the swap canister's remaining SNS balance and issue a transfer to the SNS governance canister's default account. This mirrors the recommendation in the external report: provide a mechanism to recover the residual amount rather than leaving it permanently locked.

### Proof of Concept

Consider a swap with `sns_token_e8s = 100` and two participants:
- Participant A: `amount_icp_e8s = 1`, `total_icp_e8s = 3`
  - `scale(1, 100, 3) = floor(100/3) = 33`
- Participant B: `amount_icp_e8s = 2`, `total_icp_e8s = 3`
  - `scale(2, 100, 3) = floor(200/3) = 66`

Total distributed: `33 + 66 = 99`. Remaining: `1` e8 permanently locked in the swap canister. [1](#0-0) [5](#0-4) [3](#0-2)

    fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
        assert!(amount_icp_e8s <= u64::from(total_icp_e8s));
        // Note that the multiplication cannot overflow as both factors fit in 64 bits.
        let r = (amount_icp_e8s as u128)
            .saturating_mul(total_sns_e8s as u128)
            .div(NonZeroU128::from(total_icp_e8s));
        // This follows logically from the initial assert `amount_icp_e8s <= total_icp_e8s`.
        assert!(r <= u64::MAX as u128);
        r as u64
    }
