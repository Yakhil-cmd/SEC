### Title
Incorrect Participant Count in Max-Participant Guard Denies Existing Buyers at Boundary — (`File: rs/sns/swap/src/swap.rs`)

### Summary
In `Swap::refresh_buyer_token_e8s`, the guard that enforces the `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` ceiling unconditionally adds `+1` to the current participant count before comparing against the limit. For an **existing** buyer (already in `self.buyers`), this `+1` is wrong: the buyer is already counted. At the exact boundary where `num_direct_participants * num_sns_neurons_per_basket == MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`, an existing participant who tries to increase their ICP contribution is incorrectly rejected with the error message that explicitly promises the opposite: *"existing participants may still increase their ICP participation amount."*

### Finding Description
The guard in `refresh_buyer_token_e8s` reads:

```rust
let num_direct_participants = self.buyers.len() as u64;
let num_sns_neurons_per_basket = params
    .neuron_basket_construction_parameters
    .as_ref()
    .expect("neuron_basket_construction_parameters must be specified")
    .count;
if (num_direct_participants + 1) * num_sns_neurons_per_basket
    > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(format!(
        "The swap has reached the maximum number of direct participants \
         ({num_direct_participants}) and does not accept new participants; \
         existing participants may still increase their ICP participation amount. …"
    ));
}
``` [1](#0-0) 

The check fires **before** the buyer is inserted into `self.buyers`. For a **new** buyer, `self.buyers.len()` does not yet include them, so `+1` correctly simulates the post-insertion count. For an **existing** buyer, `self.buyers.len()` already includes them; the `+1` therefore simulates a phantom extra participant that does not exist.

Consequence: when the swap is at exactly the maximum capacity (`num_direct_participants * num_sns_neurons_per_basket == MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`), the expression `(num_direct_participants + 1) * num_sns_neurons_per_basket` exceeds the limit, and the existing buyer is rejected — directly contradicting the stated invariant.

The buyer's existing-vs-new status is determined later in the function:

```rust
let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
``` [2](#0-1) 

This check is never consulted by the participant-count guard.

### Impact Explanation
An existing SNS swap participant who has already committed ICP is incorrectly denied the ability to increase their participation amount when the swap has reached exactly `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS / num_sns_neurons_per_basket` participants. The stated protocol guarantee — that existing participants can always top up — is silently violated at this boundary. In a popular swap that fills to capacity, every existing participant is locked out of increasing their contribution for the remainder of the swap's open period, potentially reducing total ICP raised and distorting the swap outcome.

### Likelihood Explanation
`MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` is a fixed constant. [3](#0-2) 

For any swap whose `neuron_basket_count` evenly divides `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`, the boundary condition is reachable in normal operation. A popular swap with many participants will naturally approach and hit this limit. No adversarial action is required; any existing participant calling `refresh_buyer_tokens` at that moment triggers the bug.

### Recommendation
Before the participant-count guard, determine whether the caller is already a buyer. Apply the `+1` only for new buyers:

```rust
let is_new_buyer = !self.buyers.contains_key(&buyer.to_string());
if is_new_buyer
    && (num_direct_participants + 1) * num_sns_neurons_per_basket
        > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(format!("…"));
}
```

This preserves the intended invariant: new participants are blocked once the ceiling is reached, while existing participants can always increase their contribution.

### Proof of Concept
1. Deploy a swap with `neuron_basket_count = K` such that `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` is divisible by `K`. Let `N = MAX_NEURONS_FOR_DIRECT_PARTICIPANTS / K`.
2. Have `N` distinct principals each call `refresh_buyer_tokens` successfully. Now `self.buyers.len() == N` and `N * K == MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`.
3. Have principal #1 (already in `self.buyers`) call `refresh_buyer_tokens` again with a higher ICP balance.
4. The guard evaluates `(N + 1) * K > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` → `N*K + K > N*K` → `true`.
5. The call returns the error *"The swap has reached the maximum number of direct participants … existing participants may still increase their ICP participation amount"* — the error message itself contradicts the rejection, confirming the bug. [4](#0-3)
