### Title
Post-Removal Peer Insertion Race: Topology-Change Wins `select!` Before Completed Handshake Is Consumed — (`rs/p2p/quic_transport/src/connection_manager.rs`)

---

### Summary

`ConnectionManager::run()` is a single-threaded `tokio::select!` event loop. When a TLS handshake task completes and a topology-change watcher fires in the same loop iteration, Tokio's `select!` randomly picks one branch. If `watcher.changed()` wins, `handle_topology_change` updates `self.topology` to exclude the removed peer and scans `peer_map` to close connections — but the peer is not yet in `peer_map`, so nothing is closed. On the next iteration, `handle_established_connection` unconditionally inserts the peer into `peer_map` without re-checking `self.topology.is_member(&peer_id)`. The stale entry persists until a subsequent topology change fires, which may never happen if the removed node was the only change (because `PeerManager` uses `send_if_modified`).

---

### Finding Description

**Root cause — missing post-handshake membership guard in `handle_established_connection`:** [1](#0-0) 

The function performs no check against `self.topology` before writing to `peer_map`. Compare with `handle_outbound_conn_attemp`, which calls `can_i_dial_to` (which calls `self.topology.is_member`) before proceeding. [2](#0-1) 

**The race window in the `select!` loop:**

Both `inbound_connecting.join_next()` and `watcher.changed()` can be ready simultaneously. Tokio's `select!` without the `biased` keyword selects among ready branches pseudo-randomly. [3](#0-2) [4](#0-3) 

**`handle_topology_change` only closes connections already in `peer_map`:** [5](#0-4) 

If the peer is not yet in `peer_map` when this runs, the close is skipped entirely.

**`PeerManager` uses `send_if_modified` — no follow-up topology event fires if topology is stable:** [6](#0-5) 

With `TOPOLOGY_UPDATE_INTERVAL = 3s` and no further membership changes, the stale `peer_map` entry is never evicted by a second topology event. [7](#0-6) 

**QUIC keep-alives keep the connection alive indefinitely:** [8](#0-7) 

With `KEEP_ALIVE_INTERVAL = 1s` and `IDLE_TIMEOUT = 5s`, an active peer keeps the connection alive past the idle timeout, so the stale entry does not self-expire.

---

### Impact Explanation

The removed node's `ConnectionHandle` remains in `peer_map` and its stream-acceptor task remains in `active_connections`. Other nodes continue routing P2P artifacts (consensus block proposals, notarizations, DKG dealings, catch-up packages) to the removed node via `peer_map` lookups. The removed node can also push artifacts inbound through the still-open QUIC streams. This violates the invariant stated in the module comment: *"Only if all these steps successfully complete do we add the connection to the active set"* — the topology membership check is one of those steps, but it is not re-verified at insertion time.

---

### Likelihood Explanation

The race requires:
1. A node legitimately in the subnet (governance-controlled entry — high bar).
2. A registry update removing that node propagating to the local replica.
3. The TLS handshake completing in the same `select!` poll cycle as the topology watcher firing.

Condition 3 is not exotic: the `PeerManager` fires every 3 seconds; QUIC handshakes complete in tens of milliseconds; the overlap window is narrow but non-zero and deterministically reproducible in a test harness. A Byzantine operator who controls the timing of their own removal (e.g., by triggering a self-removal proposal while simultaneously initiating a connection) can increase the probability of hitting the window.

---

### Recommendation

Add a topology membership guard at the top of `handle_established_connection`:

```rust
fn handle_established_connection(&mut self, connection: Connection, peer_id: NodeId) {
    // Re-check topology: a topology-change event may have fired between
    // the handshake completing and this handler running.
    if !self.topology.is_member(&peer_id) {
        connection.close(VarInt::from_u32(0), b"peer not in current topology");
        return;
    }
    // ... existing logic
}
```

This mirrors the guard already present in `handle_outbound_conn_attemp` via `can_i_dial_to`.

---

### Proof of Concept

Deterministic replay test sketch:

1. Create a `ConnectionManager` with peer P in topology.
2. Spawn an inbound handshake task for P; let it complete but do not yet poll `inbound_connecting`.
3. Before polling, update the topology watcher to remove P (simulating a registry update).
4. Drive the `select!` loop: the first iteration picks `watcher.changed()` → `handle_topology_change` runs, P absent from `peer_map`, no close issued.
5. Drive the loop again: `inbound_connecting.join_next()` fires → `handle_established_connection` inserts P into `peer_map`.
6. Assert `peer_map.read().unwrap().contains_key(&peer_id)` — assertion passes, demonstrating the invariant violation.
7. Assert `self.topology.is_member(&peer_id)` returns `false` — confirming P is in `peer_map` but not in topology.
