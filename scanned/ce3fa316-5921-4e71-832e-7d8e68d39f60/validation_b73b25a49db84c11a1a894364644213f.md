### Title
Generic Nervous System Function Execution Marks Proposal as Executed Despite Application-Level Failure - (`rs/sns/governance/src/canister_control.rs`)

### Summary
`perform_execute_generic_nervous_system_function_call` treats any non-rejected reply from the target canister as a successful execution, ignoring application-level errors encoded in the reply body. A proposal is permanently marked "executed" even when the target canister's method returns a `Result::Err` or any other application-level failure indicator.

### Finding Description
In `rs/sns/governance/src/canister_control.rs`, the function `perform_execute_generic_nervous_system_function_call` dispatches a call to a registered target canister and method. It correctly propagates IC-level rejections (communication errors) as `Err`. However, when the call **replies** — regardless of the reply's content — it unconditionally returns `Ok(())`:

```rust

This `Ok(())` propagates up through `perform_execute_generic_nervous_system_function` → `perform_action` → `set_proposal_execution_status`, which permanently sets `executed_timestamp_seconds` on the proposal: [2](#0-1) [3](#0-2) 

Once `executed_timestamp_seconds != 0`, the proposal is immutably marked as executed and cannot be retried: [4](#0-3) 

The analog to the original report is direct: just as `_removeModule` ignores the revert of `disable()` and proceeds as if removal succeeded, SNS governance ignores the application-level error in the reply and proceeds as if the proposal action succeeded.

### Impact Explanation
Any SNS governance proposal using `ExecuteGenericNervousSystemFunction` that targets a canister method returning a `Result<T, E>` (or any Candid-encoded error) will be permanently marked as "executed" even when the target canister's logic rejected the operation. Governance participants observe a false "executed" status. The intended state change (e.g., a parameter update, access revocation, treasury operation on a dapp canister) silently does not occur. Because `executed_timestamp_seconds` is set, the proposal cannot be re-executed, and the governance system has no mechanism to detect or surface the discrepancy.

**Impact: Medium** — Governance decisions are not enforced; critical SNS-controlled dapp operations may silently fail while appearing successful.

### Likelihood Explanation
Any SNS that registers a generic nervous system function pointing to a canister method that returns a `Result` type (a common Candid pattern) is affected. The target canister need not be malicious — a routine application-level validation failure (e.g., insufficient balance, wrong state, rate limit) is sufficient to trigger the bug. SNS generic functions are a documented, production-used feature.

**Likelihood: Medium** — Requires an adopted proposal targeting a method that returns an application-level error, which is a realistic operational scenario.

### Recommendation
Decode the reply bytes and check for application-level errors before returning `Ok(())`. The existing TODO comment in the code already identifies the fix:

```rust
Ok(reply) => {
    match candid::Decode!(&reply, Result<String, String>) {
        Ok(Ok(_)) => Ok(()),
        Ok(Err(e)) => Err(GovernanceError::new_with_message(
            ErrorType::External,
            format!("Target canister returned application error: {e}"),
        )),
        Err(_) => Ok(()), // Non-Result reply type; treat as success
    }
}
```

Alternatively, SNS governance could define a standard reply convention (e.g., `Result<_, String>`) for all registered generic nervous system function targets and enforce it at execution time.

### Proof of Concept
1. Register a generic nervous system function in an SNS pointing to a canister method that returns `Result<(), String>`.
2. Submit and adopt a proposal with `ExecuteGenericNervousSystemFunction` whose payload causes the target method to return `Err("precondition failed")`.
3. Observe that `perform_execute_generic_nervous_system_function_call` receives the reply bytes containing the encoded `Err`, discards them at line 302 (`Ok(_reply) => { ... Ok(()) }`), and returns `Ok(())`.
4. `set_proposal_execution_status` receives `Ok(())`, sets `executed_timestamp_seconds = now`, and logs "Execution of proposal succeeded."
5. The proposal is permanently marked executed; the target canister's state is unchanged; no retry is possible. [5](#0-4) [6](#0-5)

pub async fn perform_execute_generic_nervous_system_function_call(
    env: &dyn Environment,
    function: NervousSystemFunction,
    call: ExecuteGenericNervousSystemFunction,
) -> Result<(), GovernanceError> {
    // Get the canister id and the method against which we execute the proposal.
    let valid_function = ValidGenericNervousSystemFunction::try_from(&function)
        .map_err(|e| GovernanceError::new_with_message(ErrorType::InvalidProposal, e))?;

    let result = env

            valid_function.target_canister_id,
            &valid_function.target_method,
            call.payload,

    pub fn set_proposal_execution_status(&mut self, pid: u64, result: Result<(), GovernanceError>) {
        match self.proto.proposals.get_mut(&pid) {
            Some(proposal) => {
                // The proposal has to be adopted before it is executed.
                assert_eq!(proposal.status(), ProposalDecisionStatus::Adopted);
                match result {
                    Ok(_) => {
                        log!(INFO, "Execution of proposal: {} succeeded.", pid);
                        // The proposal was executed 'now'.
                        proposal.executed_timestamp_seconds = self.env.now();
                        // If the proposal was executed it has not failed,
                        // thus we set the failed_timestamp_seconds to zero
                        // (it should already be zero, but let's be defensive).
                        proposal.failed_timestamp_seconds = 0;
                        proposal.failure_reason = None;
                    }
                    Err(error) => {
                        log!(
                            ERROR,
                            "Execution of proposal: {} failed. Reason: {:?}",
                            pid,
                            error
                        );
                        // To ensure that we don't update the failure timestamp
                        // if there has been success, check if executed_timestamp_seconds
                        // is set to a non-zero value (this should not happen).
                        // Then, record that the proposal failed 'now' with the
                        // given error.
                        if proposal.executed_timestamp_seconds == 0 {
                            proposal.failed_timestamp_seconds = self.env.now();
                            proposal.failure_reason = Some(error);
                        }
                    }
                }
