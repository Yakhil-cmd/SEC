### Title
ckERC20 Deposit Helper Emits Requested Amount Instead of Actual Received Amount for Fee-on-Transfer ERC20 Tokens, Breaking 1:1 Backing Invariant — (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

Both Ethereum-side helper contracts used for ckERC20 deposits emit the caller-supplied `amount` parameter in the deposit event rather than the actual amount received by the minter address. The IC-side ckETH minter canister reads this event value and mints exactly that many ckERC20 tokens. For any fee-on-transfer ERC20 token, the minter mints more ckERC20 than the ERC20 it actually holds, permanently breaking the 1:1 backing invariant.

---

### Finding Description

The deposit flow for ERC20 → ckERC20 works as follows:

1. A user calls `depositErc20` (or `deposit`) on the Ethereum helper contract.
2. The helper calls `safeTransferFrom(msg.sender, minterAddress, amount)` on the ERC20 contract.
3. The helper emits a `ReceivedErc20` / `ReceivedEthOrErc20` event with the **caller-supplied** `amount`.
4. The IC minter scrapes this event log and mints `event.value()` ckERC20 tokens to the user.

In `ERC20DepositHelper.sol` (`CkErc20Deposit.deposit`):

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal); // ← uses requested amount
}
``` [1](#0-0) 

In `DepositHelperWithSubaccount.sol` (`depositErc20`):

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount); // ← uses requested amount
``` [2](#0-1) 

Neither contract checks the minter's balance before and after the transfer to determine the actual received amount. For a fee-on-transfer ERC20 token (e.g., one that deducts 1% on every transfer), `safeTransferFrom` delivers `amount * 0.99` to the minter, but the event records `amount`.

The IC minter's `mint()` function in `deposit.rs` uses `event.value()` directly as the mint amount:

```rust
amount: event.value(),  // sourced from the Ethereum log, not from actual balance change
``` [3](#0-2) 

`event.value()` returns the `value` field of `ReceivedErc20Event`, which is parsed directly from the emitted log data: [4](#0-3) [5](#0-4) 

There is no cross-check between the minted amount and the minter's actual ERC20 balance at any point in the IC-side processing.

---

### Impact Explanation

The ckERC20 system is explicitly designed to maintain a 1:1 backing invariant: every ckERC20 token in circulation must be backed by exactly one ERC20 token held by the minter's Ethereum address. [6](#0-5) 

When a fee-on-transfer ERC20 token is deposited:
- The minter holds `amount - fee` ERC20 tokens on Ethereum.
- The IC minter mints `amount` ckERC20 tokens on the IC ledger.
- The deficit accumulates with every deposit.
- When users attempt to withdraw ckERC20 back to ERC20, the minter will eventually be unable to fulfill withdrawals because it does not hold enough ERC20 to cover all outstanding ckERC20 tokens.

This is a **ledger conservation bug** / **chain-fusion mint/burn accounting bug**: the total ckERC20 supply exceeds the ERC20 held in escrow, permanently and irreversibly (since the event is already finalized on Ethereum and the minter marks it as processed).

---

### Likelihood Explanation

The supported ckERC20 token list includes **USDT** (`0xdAC17F958D2ee523a2206206994597C13D831ec7`), which has a fee-on-transfer mechanism in its contract that is currently set to zero but can be activated by the USDT issuer at any time. [7](#0-6) 

Additionally, any new ERC20 token added via NNS proposal could have fee-on-transfer behavior. The documentation explicitly states that the helper contract does not enforce a whitelist of token types, and there is no check in the minter or the orchestrator that rejects fee-on-transfer tokens. [8](#0-7) 

The attacker-controlled entry path is fully permissionless: any user can call `depositErc20` on the helper contract with a supported fee-on-transfer ERC20 token. No privileged role is required.

---

### Recommendation

In both helper contracts, record the minter's ERC20 balance before and after the `safeTransferFrom` call, and emit the **actual received amount** (the balance difference) in the event:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

This mirrors the mitigation recommended in the Rubicon Finance report and ensures the IC minter only mints ckERC20 tokens equal to the ERC20 tokens actually received. Alternatively, the NNS governance process for adding new ckERC20 tokens should explicitly screen for and reject fee-on-transfer tokens, and the minter should verify that currently supported tokens (e.g., USDT) have not activated their fee mechanisms.

---

### Proof of Concept

1. USDT activates its fee-on-transfer mechanism (e.g., 1% fee per transfer).
2. A user calls `depositErc20(USDT_ADDRESS, 1_000_000, principal, subaccount)` on the `DepositHelperWithSubaccount` contract.
3. `safeTransferFrom` transfers 990,000 USDT to the minter (1% fee deducted).
4. The contract emits `ReceivedEthOrErc20(USDT_ADDRESS, user, 1_000_000, principal, subaccount)`.
5. The IC minter scrapes the log, parses `value = 1_000_000` from the event data. [9](#0-8) 
6. The minter calls `icrc1_transfer` with `amount: 1_000_000` to mint 1,000,000 ckUSDT to the user. [3](#0-2) 
7. The minter holds 990,000 USDT but has minted 1,000,000 ckUSDT — a 10,000 ckUSDT deficit per deposit.
8. Repeated deposits accumulate the deficit until the minter cannot honor all ckUSDT → USDT withdrawals.
