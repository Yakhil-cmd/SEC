### Title
Unbounded `limit` in `POST /search/transactions` bypasses `MAX_SEARCH_LIMIT`, enabling OOM crash of the Rosetta process — (`rs/rosetta-api/icp/src/request_handler.rs`)

### Summary
An unprivileged HTTP client can supply `"limit": 9223372036854775807` (`i64::MAX`) in a `POST /search/transactions` request. The handler converts this to `u64` successfully and passes it verbatim to a SQLite `LIMIT` clause, causing the entire `blocks` table to be streamed and deserialized into a `Vec<HashedBlock>` in process memory, exhausting RAM and crashing the Rosetta process.

### Finding Description
In `search_transactions`, `MAX_SEARCH_LIMIT` is only the **fallback default** for a missing `limit` field. [1](#0-0) 

When the client provides any explicit value, the sole validation is `try_into::<u64>()`, which only rejects negative numbers. `i64::MAX` (9,223,372,036,854,775,807) converts without error. The resulting `u64` is then bound directly to the parameterized query: [2](#0-1) 

The HTTP handler in `rosetta_server.rs` applies no additional validation before delegating to `req_handler.search_transactions`: [3](#0-2) 

The only body-size guard is a 4 MB JSON payload cap (line 298), which does not constrain the numeric value of `limit`. [4](#0-3) 

### Impact Explanation
SQLite will attempt to return up to `9.2 × 10^18` rows. On a production Rosetta node synced to the ICP ledger (tens of millions of blocks), this causes the process to allocate a `Vec<HashedBlock>` proportional to the full block table size, exhausting available RAM and triggering an OOM kill. The Rosetta process crashes, making the API unavailable until restarted.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. The payload is a single-line JSON object. No rate limiting or authentication is visible in the server setup. Any network-reachable attacker can trigger this repeatedly to keep the process down.

### Recommendation
After the `try_into()` conversion, enforce the cap unconditionally:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_SEARCH_LIMIT as i64)
    .try_into()
    .map_err(|err| ApiError::invalid_request(...))?;

let limit = limit.min(MAX_SEARCH_LIMIT as u64);  // enforce cap regardless of source
```

Alternatively, reject requests where the provided `limit` exceeds `MAX_SEARCH_LIMIT` with an explicit `ApiError::invalid_request`.

### Proof of Concept
```
POST /search/transactions HTTP/1.1
Content-Type: application/json

{
  "network_identifier": { "blockchain": "Internet Computer", "network": "00000000000000020101" },
  "account_identifier": { "address": "<any valid address>" },
  "limit": 9223372036854775807
}
```

With a non-empty local SQLite store, the process will attempt to load all blocks into memory. Assert that either the response is rejected with an error, or that the returned `transactions` array contains at most `MAX_SEARCH_LIMIT` entries and process RSS does not spike.
