### Title
ckERC-20 Minter Mints Based on Event-Logged Amount Rather Than Actual Received Balance for Fee-on-Transfer ERC-20 Tokens - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol)

---

### Summary

The ckETH/ckERC-20 deposit helper smart contract emits the `ReceivedEthOrErc20` event using the caller-supplied `amount` argument — not the actual balance change received by the minter address. The IC minter canister scrapes these logs and mints ckERC-20 tokens 1:1 against the event's `amount` field. For ERC-20 tokens with fee-on-transfer capability (e.g., USDT, which has a configurable fee currently set to 0%), the minter would mint more ckERC-20 than the ERC-20 tokens it actually holds, breaking the 1:1 peg and creating unbacked supply.

---

### Finding Description

**Root cause — helper contract (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, lines 511–532):**

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(
        msg.sender,
        minterAddress,
        amount          // ← requested amount, not actual received
    );
    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        amount,         // ← same requested amount emitted verbatim
        principal,
        subaccount
    );
}
``` [1](#0-0) 

After `safeTransferFrom` completes, the contract emits `amount` — the value the caller passed in — without checking how many tokens the minter address actually received. For a fee-on-transfer token, the minter receives `amount − fee`, but the event records `amount`.

**Consumption in the IC minter (`rs/ethereum/cketh/minter/src/deposit.rs`, lines 73–81):**

```rust

`event.value()` returns the `value` field of `ReceivedErc20Event`, which is parsed directly from the log's data word:

```rust
let value = Erc20Value::from_be_bytes(value_bytes);
Ok(ReceivedErc20Event { ..., value, ... }.into())
``` [3](#0-2) 

There is no step that compares the minter's ERC-20 balance before and after the `transferFrom` call; the minter unconditionally trusts the log's `amount` field.

---

### Impact Explanation

For any supported ERC-20 token that activates a fee-on-transfer mechanism:

- The minter's Ethereum address receives `amount − fee` tokens.
- The IC minter mints `amount` ckERC-20 tokens.
- The ckERC-20 total supply exceeds the actual ERC-20 collateral held by the minter.
- Subsequent withdrawals drain the minter's reserves; the last withdrawers cannot redeem their ckERC-20 at par.

This is a **chain-fusion mint/burn/replay bug** (ledger conservation violation): the invariant `ckERC-20 total supply ≤ ERC-20 balance of minter address` is broken.

---

### Likelihood Explanation

USDT (Tether) is the canonical example: its contract contains a configurable fee mechanism currently set to 0%, but the fee can be enabled by the contract owner at any time. The M04 report cited this exact token. If USDT or any other fee-on-transfer ERC-20 is added to the minter's supported-token whitelist — or if a currently supported token activates such a fee — every deposit would silently over-mint ckERC-20. The minter documentation explicitly warns that the helper contract does not enforce a token whitelist; enforcement is done by the minter's log-scraping filter, meaning the attack surface is any token the NNS governance approves.

Likelihood is **medium**: it requires a governance decision to add a fee-on-transfer token, or an existing supported token enabling its fee, but the structural flaw is present today and requires no privileged IC access to exploit once the precondition is met.

---

### Recommendation

In `depositErc20`, record the minter's balance before and after the `safeTransferFrom` call and emit the delta as the event amount:

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        actualReceived,   // ← actual balance change, not caller-supplied amount
        principal,
        subaccount
    );
}
```

This mirrors the fix recommended in the M04 report (check balance before/after) and is the same pattern used by Curve's `FEE_INDEX` logic for USDT.

---

### Proof of Concept

1. USDT activates its transfer fee (e.g., 1 basis point = 0.01%).
2. NNS governance adds USDT as a supported ckERC-20 token.
3. Attacker calls `depositErc20(USDT, 1_000_000, principal, subaccount)`.
4. Helper contract calls `USDT.transferFrom(attacker, minter, 1_000_000)`.
   - Due to the 0.01% fee, minter receives `999_900` USDT.
5. Helper contract emits `ReceivedEthOrErc20(USDT, attacker, 1_000_000, ...)`.
6. IC minter scrapes the log, reads `value = 1_000_000`, and mints `1_000_000` ckUSDT to the attacker.
7. Attacker now holds `1_000_000` ckUSDT backed by only `999_900` USDT.
8. Repeated deposits accumulate unbacked ckUSDT; eventually the minter cannot honour all redemptions, and the last redeemers lose funds.

The entry path is fully unprivileged: any user can call `depositErc20` on the helper contract. The IC minter's log-scraping timer (`mint()` in `deposit.rs`) processes the event automatically. [4](#0-3) [1](#0-0)
