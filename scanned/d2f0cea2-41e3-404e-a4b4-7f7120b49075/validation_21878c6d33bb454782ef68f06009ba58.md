### Title
SNS Treasury Deposit to DEX Lacks On-Chain Slippage Guard, Enabling Front-Running of DAO Liquidity Deposits - (File: rs/sns/governance/src/extensions.rs)

### Summary
The SNS governance extension framework allows a DAO to deposit treasury funds (SNS tokens + ICP) into a DEX liquidity pool via a `TreasuryManager` extension canister. The deposit flow validates only that the requested amounts do not exceed 50% of the current treasury balance, but imposes **no slippage bound or minimum-received-LP-tokens check** at execution time. Because IC canister calls are asynchronous and proposal execution can be observed in the mempool (via the public `read_state` endpoint), an adversary who controls or monitors the DEX pool can manipulate the pool's reserve ratio between proposal adoption and execution, causing the SNS treasury to receive far fewer LP tokens than expected — or to deposit at a severely mispriced ratio — with no on-chain protection.

### Finding Description
The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` (lines 1545–1610) performs two sequential async steps:

1. **`approve_treasury_manager`** — issues ICRC-2 allowances on both the SNS ledger and the ICP ledger for the exact amounts specified in the proposal (`treasury_allocation_sns_e8s`, `treasury_allocation_icp_e8s`).
2. **`call_canister(extension_canister_id, "deposit", arg_blob)`** — calls the TreasuryManager's `deposit` method, which in turn calls the DEX.

The validation step (`validate_deposit_operation_impl`, lines 276–321) only checks that the requested amounts are ≤ 50% of the current treasury balance at proposal-submission time. There is no check at execution time for:
- The current pool reserve ratio vs. the ratio implied by the proposal amounts.
- A minimum amount of LP tokens to receive.
- A maximum acceptable price impact.

The `treasury_manager.did` interface itself explicitly documents this as a **Known Security Risk** (lines 35–40):
> "Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."

The `validate_and_render_register_extension` function (lines 1540–1545 of `rs/sns/governance/src/proposal.rs`) also warns voters about this in the proposal rendering, but the warning is informational only — no enforcement exists in the execution path.

The `approve_treasury_manager` function (lines 777–831 of `rs/sns/governance/src/extensions.rs`) sets the ICRC-2 allowance with a 1-hour expiry and `expected_allowance = None`, meaning the allowance is unconditionally overwritten. Once the allowance is set, the TreasuryManager canister can call `icrc2_transfer_from` on both ledgers for the full approved amounts regardless of the current pool state.

### Impact Explanation
An adversary who controls a significant position in the target DEX pool (or who can observe the pending proposal and act before execution) can:

1. Observe the adopted `ExecuteExtensionOperation` deposit proposal via the public `read_state` endpoint.
2. Before the SNS governance canister executes the proposal, manipulate the pool's reserve ratio (e.g., by removing liquidity to bring one reserve near zero, or by making large swaps to skew the price).
3. The SNS treasury's deposit then executes at the manipulated ratio, receiving far fewer LP tokens than expected, or depositing at a price that is immediately arbitrageable.
4. The adversary restores the pool state and profits from the arbitrage, effectively extracting value from the SNS treasury.

The `treasury_manager.did` comment "any undeposited tokens are automatically returned to the SNS treasury account" only applies to tokens that the TreasuryManager cannot deposit (e.g., due to ratio constraints in the DEX). If the DEX accepts the deposit at the manipulated ratio, no tokens are returned — the DAO simply receives LP tokens worth less than the deposited assets.

**Impact**: Direct loss of SNS treasury funds (ICP and SNS tokens) proportional to the price manipulation. For large SNS treasuries, this can be material. The loss is permanent once the deposit is executed at the manipulated ratio.

### Likelihood Explanation
- The proposal execution flow is fully observable on-chain via `read_state` before execution completes.
- IC consensus rounds are ~1–2 seconds, but proposal execution involves multiple async canister calls (ledger balance query → ICRC-2 approve SNS → ICRC-2 approve ICP → TreasuryManager.deposit → DEX call), creating a window of several seconds to minutes during which the pool can be manipulated.
- Any canister or user with sufficient liquidity in the target DEX pool can perform this attack without any privileged access.
- The attack is explicitly acknowledged in the codebase as a known risk, increasing the likelihood that a sophisticated actor will attempt it.
- The SNS extension framework is new and targets high-value DAO treasuries, making it an attractive target.

### Recommendation
1. **Short term**: Require the `ExecuteExtensionOperation` deposit proposal to include a `min_lp_tokens_out` or `max_price_impact_bps` parameter. The TreasuryManager's `deposit` method should enforce this bound before calling the DEX, and revert (returning funds to the treasury) if the bound is violated.
2. **Short term**: Add a re-validation of the pool's current reserve ratio at execution time (inside `execute_treasury_manager_deposit`) and abort if the ratio has deviated beyond a governance-specified tolerance from the ratio at proposal submission.
3. **Long term**: Standardize slippage protection as a mandatory field in the `DepositRequest` type in `treasury_manager.did`, so that all TreasuryManager implementations are required to enforce it.

### Proof of Concept

**Attacker-controlled entry path**: Any unprivileged canister or user with a position in the DEX pool targeted by the SNS treasury deposit.

**Step-by-step**:

1. SNS governance adopts an `ExecuteExtensionOperation` proposal to deposit `X` SNS tokens and `Y` ICP into a DEX pool at ratio `X:Y`.

2. The proposal execution begins. The SNS governance canister calls `validate_deposit_operation_impl` (which only checks the 50% balance limit) and then calls `approve_treasury_manager`: [1](#0-0) 

3. The adversary observes the ICRC-2 approval transactions on the ledger (or monitors the pending proposal via `read_state`) and front-runs by manipulating the DEX pool — e.g., removing liquidity to bring one reserve near zero, then re-adding at a skewed ratio (analogous to the Solidity exploit in the reference report).

4. The SNS governance canister calls `deposit` on the TreasuryManager: [2](#0-1) 

5. The TreasuryManager calls the DEX with the full approved amounts. The DEX accepts the deposit at the manipulated ratio. The SNS treasury receives LP tokens worth significantly less than `X` SNS + `Y` ICP.

6. No slippage check exists anywhere in the IC-side execution path: [3](#0-2) 

7. The known risk is documented but unenforced: [4](#0-3) [5](#0-4)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
