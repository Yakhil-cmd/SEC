### Title
Dust-Triggered Participation Rejection in SNS Swap Canister - (`File: rs/sns/swap/src/swap.rs`)

### Summary
The SNS Swap canister's `refresh_buyer_token_e8s` function reads the raw ICP ledger balance of a buyer's subaccount to determine their participation amount. Because the ICP ledger is a public, permissionless ledger, any third party can send a tiny "dust" amount of ICP directly to a victim buyer's subaccount. This causes the balance check at line 1202 to pass (the total balance now meets `min_participant_icp_e8s`), but the subsequent `max_participant_icp_e8s` cap at line 1237 can reduce `new_balance_e8s` below `min_participant_icp_e8s`, causing the call to return an error at line 1241–1245 and permanently blocking the victim's participation increment.

### Finding Description

In `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`, the function:

1. Reads the raw ledger balance `e8s` of the buyer's subaccount (line 1153–1163).
2. Checks `e8s >= min_participant_icp_e8s` (line 1202). This check passes even if the balance includes unsolicited dust.
3. Computes `requested_increment_e8s = e8s - old_amount_icp_e8s` (line 1223).
4. Computes `actual_increment_e8s = min(max_increment_e8s, requested_increment_e8s)` (line 1224).
5. Computes `new_balance_e8s = old_amount_icp_e8s + actual_increment_e8s` (line 1225).
6. Caps `new_balance_e8s = min(new_balance_e8s, max_participant_icp_e8s)` (line 1237).
7. **Rejects** if `new_balance_e8s < min_participant_icp_e8s` (line 1241–1245).

The critical scenario: suppose a buyer has already committed `old_amount_icp_e8s = max_participant_icp_e8s - 1` (one e8 below their per-participant cap). They want to top up by exactly 1 e8. An attacker sends 1 additional dust e8 to the buyer's subaccount before the buyer's own transfer arrives. Now `e8s = max_participant_icp_e8s + 1` (attacker dust + buyer's top-up). The check at line 1202 passes. Then:

- `requested_increment_e8s = (max_participant_icp_e8s + 1) - (max_participant_icp_e8s - 1) = 2`
- `new_balance_e8s = (max_participant_icp_e8s - 1) + 2 = max_participant_icp_e8s + 1`
- After cap: `new_balance_e8s = max_participant_icp_e8s` — this is fine in this case.

A more impactful scenario: a **new buyer** (no prior participation, `old_amount_icp_e8s = 0`) transfers exactly `min_participant_icp_e8s` ICP. An attacker also sends a tiny dust amount (e.g., 1 e8) to the same subaccount. Now `e8s = min_participant_icp_e8s + 1`. The check at line 1202 passes. Then:

- `requested_increment_e8s = min_participant_icp_e8s + 1`
- `actual_increment_e8s = min(max_increment_e8s, min_participant_icp_e8s + 1)`

If `max_increment_e8s` (remaining swap capacity) is very small — e.g., only 1 e8 remains — then:
- `actual_increment_e8s = 1`
- `new_balance_e8s = 0 + 1 = 1`
- Check at line 1241: `1 < min_participant_icp_e8s` → **Error returned**, participation rejected.

The buyer's ICP is now stuck in the subaccount until the swap closes (committed or aborted), at which point they must use `error_refund_icp`. The attacker's 1 e8 dust cost is negligible (plus the ICP transfer fee of 10,000 e8s). [1](#0-0) [2](#0-1) 

### Impact Explanation

- **Affected users**: Any buyer participating in an SNS swap when the remaining swap capacity (`max_increment_e8s`) is less than `min_participant_icp_e8s`. This is a realistic end-of-swap condition.
- **Effect**: The buyer's `refresh_buyer_token_e8s` call returns an error. Their ICP is locked in the swap subaccount until the swap closes. They cannot participate in the swap at all, or cannot increase their participation.
- **Funds at risk**: The buyer's ICP is not permanently lost — it can be recovered via `error_refund_icp` after the swap closes — but the participation opportunity is permanently denied.
- **Swap integrity**: An attacker can selectively block specific buyers from participating, potentially preventing the swap from reaching `min_participants` or `min_direct_participation_icp_e8s`, causing the swap to abort instead of commit. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

- **Realistic trigger**: Near the end of a popular SNS swap, the remaining capacity (`available_direct_participation_e8s`) is often small. An attacker can monitor the swap state via `get_derived_state` (a public query) and time the attack precisely.
- **Low cost**: The attacker only needs to send a 1 e8 dust transfer (plus the 10,000 e8 ICP transfer fee) to the victim's subaccount. The victim's subaccount address is deterministic: `principal_to_subaccount(buyer_principal)` on the swap canister.
- **No privileged access required**: The ICP ledger is permissionless; anyone can transfer to any subaccount.
- **Targeted**: The attacker can selectively target specific buyers (e.g., large participants) to influence swap outcome. [5](#0-4) [6](#0-5) 

### Recommendation

In `refresh_buyer_token_e8s`, instead of using the raw ledger balance `e8s` directly, compute the effective increment as `min(e8s, old_amount_icp_e8s + max_participant_icp_e8s) - old_amount_icp_e8s` before applying the `max_increment_e8s` cap. This ensures that dust sent to the subaccount beyond the participant's cap does not affect the increment calculation. Alternatively, cap `e8s` to `old_amount_icp_e8s + max_participant_icp_e8s` before computing `requested_increment_e8s`, so that excess balance (whether from dust or overpayment) is silently ignored rather than propagated into the increment logic. [7](#0-6) 

### Proof of Concept

**Setup**:
- SNS swap is `Open` with `min_participant_icp_e8s = 100_000_000` (1 ICP), `max_participant_icp_e8s = 500_000_000` (5 ICP), `max_direct_participation_icp_e8s = 1_000_000_000` (10 ICP).
- Current total direct participation: `999_999_999` e8s (1 e8 below the cap), so `max_increment_e8s = 1`.

**Attack**:
1. Victim (new buyer, `old_amount_icp_e8s = 0`) transfers `100_000_000` e8s (1 ICP, the minimum) to their swap subaccount on the ICP ledger.
2. Attacker sends `1` e8 dust to the same subaccount.
3. Victim calls `refresh_buyer_tokens`.

**Execution in `refresh_buyer_token_e8s`**:
- `e8s = 100_000_001` (victim's 1 ICP + attacker's 1 e8 dust).
- Line 1202: `100_000_001 >= 100_000_000` → check passes.
- `old_amount_icp_e8s = 0`.
- `requested_increment_e8s = 100_000_001 - 0 = 100_000_001`.
- `actual_increment_e8s = min(1, 100_000_001) = 1`.
- `new_balance_e8s = 0 + 1 = 1`.
- Line 1241: `1 < 100_000_000` → **Error: "Rejecting participation of effective amount 1; minimum required to participate: 100000000"**.

**Result**: Victim's 1 ICP is locked in the swap subaccount. They cannot participate. The attacker spent 10,001 e8s (1 dust + 10,000 fee) to deny the victim's participation. [8](#0-7) [2](#0-1)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };

        icp_ledger: &dyn ICRC1Ledger,
    ) -> ErrorRefundIcpResponse {
        // Fail if the request is premature.
        if !(self.lifecycle() == Lifecycle::Aborted || self.lifecycle() == Lifecycle::Committed) {
            return ErrorRefundIcpResponse::new_precondition_error(
                "Error refunds can only be performed when the swap is ABORTED or COMMITTED",
            );
        }

        // Unpack request.
        let source_principal_id = match request {
            ErrorRefundIcpRequest {
                source_principal_id: Some(source_principal_id),
            } => source_principal_id,
            _ => {
                return ErrorRefundIcpResponse::new_invalid_request_error(format!(
                    "Invalid request. Must have source_principal_id. Request:\n{request:#?}",
                ));
            }
        };

        if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
            if let Some(transfer) = &buyer_state.icp
                && transfer.transfer_success_timestamp_seconds == 0
            {
                // This buyer has ICP not yet disbursed using the normal mechanism.
                return ErrorRefundIcpResponse::new_precondition_error(format!(
                    "ICP cannot be refunded as principal {} has {} ICP (e8s) in escrow",
                    source_principal_id,
                    buyer_state.amount_icp_e8s()
                ));
            }
            // This buyer has participated in the swap, but all ICP
            // has already been disbursed, either back to the buyer
            // (aborted) or to the SNS Governance canister
            // (committed). Any ICP in this buyer's subaccount must
            // belong to the buyer.
        } else {
            // This buyer is not known to the swap canister. Any
            // balance in a subaccount belongs to the buyer.
        }

        let source_subaccount = principal_to_subaccount(source_principal_id);

        // Figure out how much to send back to source_principal_id based on
        // what's left in the subaccount.
        let account_balance_result = icp_ledger
            .account_balance(Account {
                owner: self_canister_id.into(),
                subaccount: Some(source_subaccount),

            .await;

            Ok(balance) => balance.get_e8s(),

                return ErrorRefundIcpResponse::new_external_error(format!(
                    "Unable to get the balance for the subaccount of {source_principal_id}: {err:?}",
                ));
            }
        };

        // Make transfer.
        let amount_e8s = balance_e8s.saturating_sub(DEFAULT_TRANSFER_FEE.get_e8s());
