### Title
Missing Minimum-Amount Check on SNS Treasury Manager Withdraw Operation — (`File: rs/sns/governance/src/extensions.rs`)

---

### Summary

The SNS Governance canister's `ExecuteExtensionOperation` proposal action supports a `withdraw` operation that triggers the treasury manager to pull liquidity from an external DEX. Unlike the `deposit` operation, which validates that the requested amount does not exceed 50% of the current treasury balance, the `withdraw` operation performs **no amount or minimum-return validation whatsoever**. Furthermore, the returned balances from the withdraw call are only logged and never checked against any minimum threshold. An attacker who can interact with the DEX canister between the governance proposal passing and its execution can manipulate the pool price, causing the SNS treasury to receive far less than expected.

---

### Finding Description

The `validate_withdraw_operation` function in `rs/sns/governance/src/extensions.rs` accepts any `ExtensionOperationArg` and only checks that the argument is empty (no parameters accepted):

```rust
fn validate_withdraw_operation(
    _governance: &Governance,
    arg: ExtensionOperationArg,
) -> BoxFuture<'_, Result<ValidatedOperationArg, String>> {
    Box::pin(async move {
        let ExtensionOperationArg { value } = arg;
        ValidatedWithdrawOperationArg::try_from(value)
            .map(ValidatedOperationArg::TreasuryManagerWithdraw)
    })
}
``` [1](#0-0) 

No balance check, no minimum-return check, and no slippage parameter is validated. Compare this to `validate_deposit_operation_impl`, which at least checks that the requested amount does not exceed 50% of the current treasury balance: [2](#0-1) 

At execution time, `execute_treasury_manager_withdraw` calls the treasury manager's `withdraw` endpoint and only **logs** the returned balances — it never validates that the amounts returned meet any minimum threshold: [3](#0-2) 

The `WithdrawRequest` type itself carries no minimum-amount fields: [4](#0-3) 

The codebase itself acknowledges this class of risk in two places. The `treasury_manager.did` interface file explicitly lists it as a "Known Security Risk": [5](#0-4) 

And the proposal rendering for `RegisterExtension` warns voters about front-running and sandwich attacks: [6](#0-5) 

The `perform_execute_extension_operation` function re-runs `validate_execute_extension_operation` at execution time, but this only re-validates the deposit amount against the treasury balance — it does not add any minimum-return check for withdrawals: [7](#0-6) 

---

### Impact Explanation

An attacker who can submit transactions to the DEX canister (the `external_custodian` in the treasury manager model) can manipulate the pool price between the time the governance proposal passes and the time it executes. Because the IC governance canister makes multiple async inter-canister calls during proposal execution, other messages — including the attacker's swap — can be processed between those calls. The treasury manager then withdraws liquidity at the manipulated price, and the SNS treasury receives far less than the fair-market value of its LP position. There is no on-chain check that would cause the proposal execution to revert or fail if the returned amounts are below any threshold. The loss is permanent and directly reduces the SNS treasury's token holdings.

---

### Likelihood Explanation

The attack is realistic on the Internet Computer because:
1. Governance proposal execution timing is publicly observable and predictable (proposals execute after they pass the voting threshold, triggered by the governance canister's timer).
2. The DEX canister is a permissionless IC canister; any principal can submit swap messages to it.
3. The governance canister's execution of a `withdraw` proposal involves multiple async inter-canister calls (governance → treasury manager → DEX), and between those calls, other messages to the DEX canister can be processed in the same or subsequent rounds.
4. No minimum-amount parameter exists in `WithdrawRequest`, so there is no way for the SNS community to specify acceptable slippage bounds even if they wanted to.

---

### Recommendation

1. Add a `min_amounts` field to `WithdrawRequest` (and correspondingly to `ValidatedWithdrawOperationArg`) specifying the minimum token amounts the SNS treasury must receive for the withdrawal to succeed.
2. In `validate_withdraw_operation`, validate that the `min_amounts` field is present and structurally valid.
3. In `execute_treasury_manager_withdraw`, after the withdraw call returns, compare the actual returned balances against the minimum amounts and return a `GovernanceError` if they are not met.
4. For the deposit path, similarly add a `min_lp_tokens` or `min_amounts_deposited` field to `DepositRequest` so that the DEX deposit can be rejected if the price has moved unfavorably.

---

### Proof of Concept

1. SNS governance passes an `ExecuteExtensionOperation` proposal with `operation_name = "withdraw"` targeting a KongSwap-style treasury manager extension.
2. The proposal passes the voting threshold and is queued for execution.
3. An attacker observes the proposal and submits a large swap to the DEX canister (e.g., selling a large amount of the SNS token for ICP), moving the pool price significantly.
4. The governance canister's timer fires and calls `perform_execute_extension_operation` → `validate_execute_extension_operation` (passes, since `validate_withdraw_operation` only checks that the arg is empty) → `execute_treasury_manager_withdraw` → `call_canister("withdraw", ...)`.
5. The treasury manager calls the DEX's remove-liquidity function at the manipulated price.
6. The SNS treasury receives, for example, 10% of the ICP it should have received at fair market price.
7. `execute_treasury_manager_withdraw` logs the returned balances and returns `Ok(())` — no error is raised.
8. The attacker reverses their swap on the DEX, profiting from the price impact at the SNS treasury's expense.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager withdraw operation
async fn execute_treasury_manager_withdraw(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedWithdrawOperationArg,
) -> Result<(), GovernanceError> {
    let arg_blob = construct_treasury_manager_withdraw_payload(arg.original).map_err(|err| {
        GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!("Failed to construct treasury manager withdraw payload: {err}"),
        )
    })?;

    let balances = governance
        .env
        .call_canister(extension_canister_id, "withdraw", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.withdraw failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!(
                        "Error decoding TreasuryManager.withdraw response: {err:?}"
                    ),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.withdraw failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.withdraw succeeded with response: {:?}",
        balances
    );

    Ok(())
}
