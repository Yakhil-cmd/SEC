### Title
SNS Treasury Manager Deposit API Lacks Slippage Protection, Enabling Front-Running/Sandwich Attacks on DAO Treasury Funds - (File: rs/sns/treasury_manager/treasury_manager.did)

### Summary
The SNS Treasury Manager's `deposit()` API and its execution path in SNS Governance do not enforce any slippage protection when depositing DAO treasury funds into DEX liquidity pools. The `DepositRequest` type only carries `allowances` (maximum amounts), with no minimum LP token output or price-bound field. The codebase self-acknowledges this gap but provides no on-chain mitigation. An unprivileged attacker who can interact with the target DEX canister can front-run the governance-executed deposit, manipulating the pool price so the SNS DAO receives far fewer LP tokens than the proposal intended, permanently losing treasury value.

### Finding Description
The SNS Treasury Manager extension system allows SNS DAOs to deposit treasury funds into external DEX liquidity pools via governance proposals. The execution path is:

1. `execute_treasury_manager_deposit` in `rs/sns/governance/src/extensions.rs` (lines 1545–1610) calls `approve_treasury_manager` to set ICRC-2 allowances on both the SNS and ICP ledgers for the treasury manager canister.
2. It then calls `deposit()` on the treasury manager canister, passing a `DepositRequest` that contains only `allowances` — the maximum amounts the manager may pull.

The `DepositRequest` type defined in `rs/sns/treasury_manager/treasury_manager.did` (lines 84–86) is:
```
type DepositRequest = record {
  allowances : vec Allowance;
};
```

There is no field for minimum LP tokens to receive, no price bound, and no deadline beyond the 1-hour ICRC-2 allowance expiry. The API specification itself acknowledges the gap at lines 35–40:
```
// Known Security Risks:
// ====================
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
```

The proposal rendering in `rs/sns/governance/src/proposal.rs` (lines 1540–1545) also warns:
> "Some Decentralized Exchanges lack slippage protection during deposits. Consequently, deposited asset ratios may deviate from those specified in the proposal. This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running or sandwich attacks."

Despite these warnings, neither the `TreasuryManager` trait in `rs/sns/treasury_manager/src/lib.rs` (lines 250–282) nor the governance execution layer enforces any slippage bound. The `approve_treasury_manager` function in `rs/sns/governance/src/extensions.rs` (lines 777–831) simply sets ICRC-2 allowances and returns; the treasury manager canister is then free to deposit at any price the DEX quotes.

### Impact Explanation
An attacker who can submit transactions to the target DEX canister can execute a sandwich attack:
1. Observe the pending SNS governance proposal specifying the deposit amounts.
2. Before the proposal executes, submit a large buy of one asset in the pool, moving the price adversely.
3. The governance-triggered `deposit()` executes at the manipulated price, so the SNS DAO receives significantly fewer LP tokens than the proposal intended.
4. The attacker sells their position after the deposit, profiting from the price impact.

Because the `DepositRequest` carries no minimum-output constraint, the treasury manager canister has no on-chain basis to reject a deposit that returns far fewer LP tokens than expected. The SNS DAO permanently loses the difference in treasury value. The magnitude scales with the deposit size and pool depth.

### Likelihood Explanation
SNS governance proposals are fully public and their execution timing is predictable (proposal passes → execution in the next heartbeat/timer). Any canister or user that can call the DEX canister can front-run the deposit. No privileged access, key material, or governance majority is required of the attacker. The attack is economically rational whenever the deposit is large relative to pool liquidity, which is precisely the scenario SNS DAOs are likely to encounter when first seeding a liquidity pool.

### Recommendation
1. **Add slippage parameters to `DepositRequest`**: Extend the type with a `min_lp_tokens_out` or per-asset `min_amount_out` field. The governance proposal validation step (`validate_deposit_operation_impl` in `rs/sns/governance/src/extensions.rs`, lines 276–321) should require these fields to be set and non-zero.
2. **Enforce slippage in the `TreasuryManager` trait**: The `deposit()` implementation must pass the minimum-output constraint to the DEX call and return an error (not silently succeed) if the DEX would return fewer tokens than the bound.
3. **Snapshot price at proposal creation**: Record the pool price at proposal submission time and reject execution if the price has moved beyond a governance-configurable tolerance (e.g., 1–5%).
4. **Consult DEX teams**: Analogous to the Balancer recommendation in the external report, engage with each supported DEX to confirm which deposit API path provides atomic slippage protection and use that path exclusively.

### Proof of Concept
1. SNS DAO submits a `TreasuryManagerDeposit` proposal: deposit 500,000 SNS tokens + 500,000 ICP into DEX pool X.
2. Proposal passes; `execute_treasury_manager_deposit` is called.
3. `approve_treasury_manager` sets ICRC-2 allowances of 500,000 SNS and 500,000 ICP for the treasury manager canister (expiry: 1 hour).
4. Attacker observes the proposal and, in the round before governance executes `deposit()`, calls the DEX canister to buy a large quantity of SNS tokens, shifting the SNS/ICP price ratio significantly.
5. Governance calls `deposit()` on the treasury manager; the treasury manager calls the DEX's add-liquidity endpoint at the manipulated price. The DEX accepts the full 500,000 SNS + 500,000 ICP but issues LP tokens reflecting the skewed ratio — the DAO receives LP tokens worth materially less than 1,000,000 ICP-equivalent.
6. Attacker sells their SNS position, restoring the price and pocketing the spread.
7. The `deposit()` call returns `Ok(Balances{...})` with no error, because `DepositRequest` contained no minimum-output constraint to violate.

The root cause is the absence of any slippage bound in `DepositRequest` [1](#0-0)  and the self-acknowledged gap in the API specification [2](#0-1) , compounded by the governance execution layer passing no price constraint to the treasury manager [3](#0-2)  and the proposal warning being advisory only [4](#0-3) .
