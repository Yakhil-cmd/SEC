### Title
`ProvisionalWhitelist::All` Disables Cycles-Guard for Provisional Management APIs, Enabling Free Cycle Minting by Any Caller - (File: rs/registry/provisional_whitelist/src/lib.rs)

### Summary
The `ProvisionalWhitelist` enum contains an `All` variant explicitly documented as existing "to facilitate local development and testing." When this variant is stored in the registry, the whitelist check is completely bypassed and any principal — including unprivileged canisters — can call `provisional_create_canister_with_cycles` and `provisional_top_up_canister` to mint arbitrary cycles for free. This is the direct IC analog of the `enabled = false` bypass in the Solidity `onlyEOAorWhitelist` modifier: a testing toggle that disables the guard entirely.

### Finding Description
`ProvisionalWhitelist` in `rs/registry/provisional_whitelist/src/lib.rs` has two variants:

```rust
pub enum ProvisionalWhitelist {
    Set(BTreeSet<PrincipalId>),
    /// All PrincipalIds are allowed to use the provisional API.
    /// This option exists to facilitate local development and testing.
    All,
}
```

The `contains()` method returns `true` unconditionally when the variant is `All`:

```rust
pub fn contains(&self, id: &PrincipalId) -> bool {
    match self {
        Self::All => true,
        Self::Set(set) => set.contains(id),
    }
}

When `All` is set, `provisional_create_canister_with_cycles` creates a canister with an arbitrary caller-specified cycle balance without any ICP payment, and `provisional_top_up_canister` adds arbitrary cycles to any canister — both out of thin air. [3](#0-2) 

The `ProvisionalWhitelist::All` variant is a valid, serializable protobuf registry value (`LIST_TYPE_ALL = 1`) that can be stored in and read from the live registry: [4](#0-3) 

Multiple production-adjacent bootstrap paths write `ProvisionalWhitelist::All` into the registry:

- The system-test driver bootstrap (`rs/tests/driver/src/driver/bootstrap.rs:194`) hardcodes `ProvisionalWhitelist::All` for every IC instance it creates: [5](#0-4) 

- `rs/state_machine_tests/src/lib.rs:345` sets `All` as the initial registry record for every `StateMachine` instance: [6](#0-5) 

- `rs/replica_tests/src/lib.rs:257` sets `All` in `get_ic_config()`: [7](#0-6) 

- The `ic-prep` binary (`rs/prep/src/bin/prep.rs:355`) accepts a wildcard `"*"` in the whitelist JSON file and maps it to `ProvisionalWhitelist::All`: [8](#0-7) 

The `ic-prep` comment in `rs/prep/src/internet_computer.rs` acknowledges the risk explicitly: *"In production settings, the whitelist ought to be set explicitly."* [9](#0-8) 

The message-routing layer reads the whitelist from the registry at every block and defaults to an empty set only on read error — not on `All`: [10](#0-9) 

The boundary-node ingress path (`rs/http_endpoints/public/src/call.rs`) similarly reads the live registry value and passes it directly to the ingress filter: [11](#0-10) 

### Impact Explanation
If `ProvisionalWhitelist::All` is present in the live registry — whether written by a misconfigured `ic-prep` invocation using `"*"`, by a test-driver bootstrap that was accidentally pointed at a production subnet, or by a governance proposal — any unprivileged ingress sender or canister can:

1. Call `provisional_create_canister_with_cycles` to create canisters with an arbitrary cycle balance without paying ICP.
2. Call `provisional_top_up_canister` to add arbitrary cycles to any existing canister without paying ICP.

Both operations mint cycles out of thin air, bypassing the normal ICP-to-cycles conversion enforced by the CMC. This is a **cycles/resource accounting bug** with subnet-wide impact: an attacker can exhaust subnet resources, subsidize arbitrary computation, or inflate canister balances to defeat cycle-based rate limits.

### Likelihood Explanation
The likelihood is **low on mainnet** but non-negligible:

- The `ic-prep` wildcard `"*"` path and the test-driver bootstrap both write `All` into real IC registry instances. A misconfigured deployment or a test network accidentally reachable from the public internet would expose this.
- The `ProvisionalWhitelist::All` protobuf value (`list_type = 1, set = []`) is indistinguishable from a legitimate registry entry; there is no runtime enforcement preventing it from being written to a production registry via a governance proposal.
- The `ClearProvisionalWhitelist` NNS function only empties the `Set` — it does not prevent `All` from being re-introduced.

### Recommendation
1. **Remove `ProvisionalWhitelist::All` from production code paths.** The `All` variant should be gated behind a compile-time or runtime feature flag that is disabled in production builds, analogous to removing the `enabled = false` path from the Solidity modifier.
2. **Add a registry invariant check** that rejects `LIST_TYPE_ALL` in the provisional whitelist record on non-test subnets.
3. **Audit all `ic-prep` invocations** used for production deployments to confirm no `"*"` wildcard is present in the whitelist configuration.
4. **Add a governance-level guard** that prevents `ProvisionalWhitelist::All` from being written to the NNS registry via any proposal type.

### Proof of Concept
1. Bootstrap an IC instance using the test driver (`rs/tests/driver/src/driver/bootstrap.rs`) or invoke `ic-prep` with a whitelist file containing `["*"]`. Both write `ProvisionalWhitelist::All` (protobuf `list_type = 1`) into the registry.
2. From any canister (or directly via ingress), call the management canister method `provisional_top_up_canister` with `canister_id = <target>` and `amount = 10_000_000_000_000` (10T cycles).
3. Because `ProvisionalWhitelist::All.contains(any_principal)` returns `true`, the call succeeds and the target canister receives 10T free cycles.
4. Repeat indefinitely to mint unlimited cycles, bypassing all ICP-to-cycles accounting.
