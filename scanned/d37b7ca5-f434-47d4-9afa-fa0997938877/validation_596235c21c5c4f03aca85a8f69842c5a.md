### Title
Single-Step Canister Controller Replacement Without Two-Step Confirmation - (File: rs/nns/handlers/root/impl/src/canister_management.rs)

### Summary
The NNS Root canister's `change_canister_controllers` function, and the SNS Root canister's `set_dapp_controllers` function, both overwrite the complete controller list of target canisters in a single, immediately-committed step. There is no pending-acceptance mechanism: the new controllers are never required to confirm receipt of the role. A governance proposal or SNS swap finalization that accidentally supplies a wrong principal in `new_controllers` permanently and irrevocably strips the legitimate parties of control over the affected canister(s).

### Finding Description
`change_canister_controllers` in `rs/nns/handlers/root/impl/src/canister_management.rs` (lines 213–235) constructs an `UpdateSettings` payload with `controllers: Some(change_canister_controllers_request.new_controllers)` and immediately dispatches it to the IC management canister via `update_settings`. The entire controller list is atomically replaced; there is no staging step, no pending-acceptance state, and no rollback path. [1](#0-0) 

The same single-step pattern is present in `update_canister_settings` (lines 237–259 of the same file) and, more broadly, in `rs/sns/root/src/lib.rs`'s `set_dapp_controllers` (lines 837–850), which iterates over every registered dapp canister and calls `management_canister_client.update_settings` with the caller-supplied `controller_principal_ids` — replacing all controllers in one shot per canister, with no confirmation from the incoming controllers. [2](#0-1) 

The `ChangeCanisterControllersRequest` interface (defined in `rs/nns/handlers/root/interface/src/lib.rs` lines 22–31) makes the risk explicit: `new_controllers` is a plain `Vec<PrincipalId>` with no validation that any of the listed principals are reachable or have accepted the role. [3](#0-2) 

### Impact Explanation
If a governance proposal (or SNS swap finalization) supplies even one incorrect principal in `new_controllers`, the target canister's controller list is permanently overwritten. The IC management canister has no "undo controller change" primitive. The canister becomes permanently inaccessible to its legitimate owners: no upgrades, no cycle top-ups, no recovery. For dapp canisters registered with an SNS, `set_dapp_controllers` can freeze the entire dapp suite in a single call. The impact is **irrevocable loss of canister control**, equivalent to the Reserve Protocol scenario where Alice permanently loses access to the contract.

### Likelihood Explanation
SNS deployment and dapp-transfer proposals are complex multi-field Candid payloads reviewed under time pressure. A single transposed byte in a principal text encoding, a copy-paste error, or a stale principal from a prior deployment is sufficient to trigger the irrevocable loss. The `change_canister_controllers` path is exercised on every SNS launch and every dapp canister transfer — a regularly executed, production code path with real-world exposure. The entry point is a **governance/chain-fusion user** (neuron holder) submitting a proposal, which is explicitly within scope. No malicious majority is required; an honest majority approving a well-intentioned but mis-encoded proposal is sufficient.

### Recommendation
- **Short term:** Implement a two-step controller transfer: store the proposed new controllers in canister state as a *pending* change, and only commit the `update_settings` call after the new controller principal has made a confirming ingress call (or after a time-locked follow-up governance proposal). This mirrors the `Ownable2Step` pattern recommended in the original report.
- **Long term:** Audit all single-shot privileged state mutations across NNS Root, SNS Root, and the engine controller canister (`rs/engine_controller/canister/canister.rs` lines 66–81, where `AUTHORIZED_CALLER` is overwritten in `apply_init_args` with no confirmation from the new principal). Document the recovery path for each. Where no recovery path exists, mandate a two-step confirmation. [4](#0-3) 

### Proof of Concept
1. A neuron holder submits an NNS governance proposal to deploy a new SNS, specifying a dapp canister to transfer. The `new_controllers` field in the resulting `ChangeCanisterControllersRequest` contains a principal with a one-character typo (e.g., a miscopied SNS Root canister ID).
2. The honest governance majority approves the proposal.
3. SNS-W calls `change_canister_controllers` on NNS Root with the malformed principal.
4. NNS Root calls `management.update_settings` with `controllers: [<typo_principal>]` — line 220 of `rs/nns/handlers/root/impl/src/canister_management.rs`.
5. The dapp canister's controller list is permanently set to the typo principal.
6. No subsequent call can recover control; the canister is permanently frozen and unrecoverable without a subnet-level intervention. [5](#0-4)

pub async fn change_canister_controllers(
    change_canister_controllers_request: ChangeCanisterControllersRequest,
    management_canister_client: &mut impl ManagementCanisterClient,
) -> ChangeCanisterControllersResponse {
    let update_settings_args = UpdateSettings {
        canister_id: change_canister_controllers_request.target_canister_id,
        settings: CanisterSettings {
            controllers: Some(change_canister_controllers_request.new_controllers),
            ..Default::default()
        },
        sender_canister_version: management_canister_client.canister_version(),
    };

    match management_canister_client
        .update_settings(update_settings_args)
        .await
    {
        Ok(()) => ChangeCanisterControllersResponse::ok(),
        Err((code, description)) => {
            ChangeCanisterControllersResponse::error(Some(code), description)
        }
    }
}
