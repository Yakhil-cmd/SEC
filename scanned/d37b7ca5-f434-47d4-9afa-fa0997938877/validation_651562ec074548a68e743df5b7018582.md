### Title
Unchecked ICP Burn Result in `burn_and_log` Silently Allows Cycles Minting Without ICP Destruction - (`File: rs/nns/cmc/src/main.rs`)

### Summary

The Cycles Minting Canister (CMC) intentionally discards the error result of the ICP burn ledger call inside `burn_and_log`. When the burn fails, cycles have already been deposited/minted and the block is permanently marked as processed, so the ICP remains unburned in the CMC subaccount with no recovery path. This is the direct IC analog of calling `token.transfer()` without checking the return value: the transfer (burn) can fail silently while the downstream effect (cycles minted) has already been committed.

### Finding Description

After a successful `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` call, the CMC calls `burn_and_log` to destroy the ICP that funded the operation. The function explicitly discards the error:

### Impact Explanation

**Ledger conservation bug.** If the `send_pb` burn call fails (e.g., the ICP ledger is temporarily unavailable during a subnet upgrade, the CMC is low on cycles for the outgoing call, or any transient inter-canister error), the following invariant is broken:

- Cycles are minted / canister is created / top-up is applied — the downstream effect is committed.
- The ICP in the CMC's subaccount is **not** burned.
- The block is permanently marked processed; neither the user nor the protocol can retry the burn.
- The ICP remains stranded in the CMC subaccount indefinitely, inflating the effective ICP supply relative to the cycles issued.

This violates the core economic invariant of the IC: every cycle minted must correspond to burned ICP.

### Likelihood Explanation

The ICP ledger (`ryjl3-tyaaa-aaaaa-aaaba-cai`) is a system canister on the NNS subnet. Inter-canister calls to it can fail transiently during:

- NNS subnet upgrades (the ledger is briefly stopped).
- CMC running low on cycles (the outgoing call is rejected before it reaches the ledger).
- Any `SysTransient` reject from the messaging layer.

These are realistic, non-adversarial conditions that occur in normal protocol operation. No privileged access or attacker control is required; the failure window is narrow but real and repeatable across every notification processed during such an event.

### Recommendation

`burn_and_log` should propagate its error and the caller should handle it. One safe approach is to record a "burn pending" flag in the block's notification status before returning success to the user, then retry the burn in a heartbeat or timer until it succeeds. Alternatively, the burn can be retried immediately with a bounded number of attempts before giving up and emitting a metric/alert that operators can act on. The key invariant to enforce is: **no notification should be finalized as successful until the corresponding ICP burn is confirmed by the ledger**.

### Proof of Concept

1. User sends ICP to CMC subaccount and calls `notify_top_up`.
2. CMC successfully calls `deposit_cycles` — cycles are added to the target canister.
3. CMC calls `burn_and_log`; at this exact moment the NNS subnet is mid-upgrade and the ledger rejects the call with `SysTransient`.
4. `burn_and_log` logs the error and returns `()`.
5. The block is recorded as `NotifiedTopUp(Ok(cycles))`.
6. The ICP remains in the CMC subaccount. The user's canister has the cycles. The ICP is never burned.
7. Any subsequent call to `notify_top_up` with the same block index returns the cached `Ok(cycles)` without attempting the burn again. [6](#0-5) [7](#0-6)

async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
