### Title
No Staleness Guard on Stored ICP/XDR Rate at Point of Cycles Minting - (File: rs/nns/cmc/src/main.rs)

---

### Summary

The Cycles Minting Canister (CMC) converts ICP to cycles using a stored `icp_xdr_conversion_rate` that carries no maximum-age enforcement at the point of consumption. If the Exchange Rate Canister (XRC) becomes unable to supply a fresh rate — because its HTTPS outcalls to external exchanges return fewer than the required minimum sources — the CMC silently retains the last accepted rate and continues minting cycles at that price indefinitely. Any unprivileged user who calls `notify_top_up` or `notify_mint_cycles` during such a period receives cycles computed from a potentially arbitrarily stale rate.

---

### Finding Description

**Vulnerability class**: cycles/resource accounting bug (stale oracle data used without liveness guard at consumption site).

**Root cause — `tokens_to_cycles` in `rs/nns/cmc/src/main.rs`**

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);   // ← timestamp never read
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...)
        }
    })
}
```

The function extracts only `xdr_permyriad_per_icp` from the stored rate; `rate.timestamp_seconds` is never compared against the current time. There is no maximum-age threshold that would cause the function to refuse conversion when the rate is stale.

**Validation gap — `validate_exchange_rate` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs`**

```rust
pub fn validate_exchange_rate(exchange_rate: &ExchangeRate) -> Result<(), ValidateExchangeRateError> {
    if exchange_rate.metadata.base_asset_num_received_rates < MINIMUM_ICP_SOURCES { ... }
    if exchange_rate.metadata.quote_asset_num_received_rates < MINIMUM_CXDR_SOURCES { ... }
    Ok(())
}
```

`validate_exchange_rate` checks only that at least 4 ICP sources and 4 CXDR sources responded. It performs no check on `exchange_rate.timestamp` relative to the current time. A rate that is hours or days old passes validation as long as it was originally collected from enough sources.

**Staleness accumulation path**

The CMC heartbeat calls `update_exchange_rate` every 5 minutes (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`). When the XRC call fails or returns a rate with insufficient sources, the CMC schedules a retry one minute later and **leaves `state.icp_xdr_conversion_rate` unchanged**. There is no upper bound on how long the CMC will continue using the last accepted rate while retries fail.

**Attacker-controlled entry path**

Any principal can call the public update methods `notify_top_up` and `notify_mint_cycles`. Both call `process_top_up` / `process_mint_cycles`, which call `tokens_to_cycles(amount)` directly. No privileged role is required.

---

### Impact Explanation

If the ICP/XDR rate stored in the CMC is stale and the real ICP price has fallen, every call to `notify_top_up` or `notify_mint_cycles` mints more cycles than the deposited ICP is worth at the current market price. Cycles are the IC's unit of computation cost; over-minting cycles at a stale high price is a direct conservation violation — the ICP burned is worth less than the cycles created. At scale this drains the economic backing of the cycles supply.

---

### Likelihood Explanation

The XRC fetches ICP and CXDR rates via HTTPS outcalls to external exchanges. The minimum-source threshold is 4 for each asset class. During periods of exchange downtime, rate-limiting, or network partitions affecting multiple exchanges simultaneously, the XRC returns `CryptoBaseAssetNotFound`, `StablecoinRateTooFewRates`, or similar errors. The CMC then retries every minute but keeps the old rate. Such conditions are not hypothetical: exchange APIs are regularly rate-limited during high-volatility periods — exactly when price staleness is most dangerous. The window of exposure is unbounded; there is no circuit-breaker that halts minting after, say, 30 minutes without a fresh rate.

---

### Recommendation

Add a maximum-age check inside `tokens_to_cycles` before using the stored rate:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref().ok_or_else(|| {
            NotifyError::Other { ... "No conversion rate" ... }
        })?;

        // Reject if the rate is older than MAX_RATE_AGE_SECONDS (e.g. 3600).
        let age = now_seconds().saturating_sub(rate.timestamp_seconds);
        if age > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: format!("ICP/XDR rate is stale ({age}s old); minting suspended"),
            });
        }

        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

Additionally, extend `validate_exchange_rate` to reject rates whose `exchange_rate.timestamp` is older than a configurable threshold relative to the current time.

### Proof of Concept

1. The CMC has a stored rate of, say, 50 000 XDR/ICP (ICP = $10).
2. The XRC's HTTPS outcalls to exchanges begin failing (fewer than 4 sources respond). The CMC logs `FailedToRetrieveRate` or `InvalidRate` every minute but keeps the old rate.
3. ICP price falls to $5 (25 000 XDR/ICP). The XRC remains unavailable.
4. An attacker sends 1 ICP to the CMC subaccount and calls `notify_top_up`.
5. `tokens_to_cycles` reads `xdr_permyriad_per_icp = 50_000` from the stale stored rate — no timestamp check occurs.
6. The attacker receives cycles equivalent to $10 worth of ICP while only depositing $5 worth, a 2× over-mint.
7. The ICP is burned; the excess cycles are permanently in circulation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
