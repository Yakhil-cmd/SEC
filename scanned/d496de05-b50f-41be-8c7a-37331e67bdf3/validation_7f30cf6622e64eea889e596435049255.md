### Title
Hot Keys Not Reset When Spawning a Neuron with a New Controller - (File: `rs/nns/governance/src/governance.rs`)

### Summary
In the NNS Governance canister, the `spawn_neuron` function allows a parent neuron's controller to spawn a child neuron and assign it to a **different** `new_controller`. However, the parent neuron's `hot_keys` are unconditionally copied to the child neuron without being cleared. This means the old controller's previously-registered hot key principals retain voting and governance authority over the newly-spawned neuron, even though the new controller never authorized them.

### Finding Description
The `spawn_neuron` function in `rs/nns/governance/src/governance.rs` constructs the child neuron as follows:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    child_controller,          // ← new controller (can differ from parent's)
    DissolveStateAndAge::DissolvingOrDissolved { ... },
    created_timestamp_seconds,
)
.with_spawn_at_timestamp_seconds(...)
.with_hot_keys(parent_neuron.hot_keys.clone())  // ← hot keys NOT reset
.with_followees(parent_neuron.followees.clone())
...
.build();
``` [1](#0-0) 

The `child_controller` is taken from `spawn.new_controller` and can be any self-authenticating principal — entirely different from the parent's controller. [2](#0-1) 

Yet the parent's `hot_keys` are cloned verbatim into the child. Hot keys are principals that are authorized to vote and perform certain governance operations on behalf of the neuron: [3](#0-2) 

Specifically, hot keys can: register votes, follow other neurons, and join/leave the Neuron Fund — all on behalf of the neuron whose voting power they do not own. [4](#0-3) 

The same pattern also exists in `split_neuron`, though it is less impactful there because the child's controller is always `*caller` (the same person who initiated the split and who set the hot keys). [5](#0-4) 

### Impact Explanation
A hot key holder on the parent neuron retains the ability to vote on NNS governance proposals using the child neuron's voting power after the child is spawned to a new controller. This is a **governance authorization bug**: the new controller's voting power can be silently hijacked by principals they never authorized. The new controller may be unaware that inherited hot keys exist and may never remove them, especially since the spawned neuron starts in `Spawning` state and the new controller may not immediately inspect it.

### Likelihood Explanation
The `spawn_neuron` endpoint is publicly callable by any neuron controller via the `manage_neuron` ingress message — no privileged role is required. The attacker only needs to:
1. Own a neuron with maturity.
2. Add their own alternate principal (or a colluding principal) as a hot key.
3. Spawn a child neuron to a victim's principal as `new_controller`.

This is a realistic scenario in any neuron marketplace or delegation arrangement where maturity is sold/transferred via `Spawn`. The new controller has no on-chain notification that inherited hot keys exist.

### Recommendation
Clear `hot_keys` on the child neuron when `spawn.new_controller` differs from the parent's controller. Specifically, in `spawn_neuron`, replace:

```rust
.with_hot_keys(parent_neuron.hot_keys.clone())
```

with:

```rust
.with_hot_keys(if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
})
```

This mirrors the fix recommended in the external report: reset the privileged access field when ownership changes.

### Proof of Concept
1. Alice controls neuron N with maturity. Alice adds principal `BOB` as a hot key on N.
2. Alice calls `manage_neuron` → `Spawn { new_controller: Some(CAROL), ... }`.
3. Governance creates child neuron C with `controller = CAROL` and `hot_keys = [BOB]`. [1](#0-0) 
4. BOB calls `manage_neuron` → `RegisterVote` on neuron C. Governance checks `is_authorized_to_vote`: [6](#0-5) 
   which resolves to `is_hotkey_or_controller` → `true` for BOB. [7](#0-6) 
5. BOB successfully casts votes using CAROL's neuron's voting power. CAROL never authorized BOB.
