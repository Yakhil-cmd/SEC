### Title
Markdown Injection via Unsanitized UTF-8 Memo in ICRC-21 `GenericDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The `add_memo` function in `responses.rs` inserts a user-controlled memo byte buffer directly into a markdown format string without sanitizing backtick characters. When the memo is valid UTF-8 and contains a backtick, it terminates the surrounding inline code span prematurely, allowing arbitrary markdown to render outside the code fence in the `GenericDisplayMessage` consent message returned by `icrc21_canister_call_consent_message`.

---

### Finding Description

In `add_memo`, the `GenericDisplayMessage` branch formats the memo as:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
``` [1](#0-0) 

The only transformation applied to the raw bytes is a UTF-8 validity check; if valid, the string is used verbatim:

```rust
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
``` [2](#0-1) 

No backtick escaping or markdown sanitization is performed. If `memo_str` contains a backtick (e.g., `x`), the format string produces:

```
\n\n**Memo:**\n`x`\n\n**Amount:** `0.00001 ICP`
```

In CommonMark, the first `` ` `` opens a code span that closes at the second `` ` `` (after `x`). Everything after that — `\n\n**Amount:** \`0.00001 ICP\`` — renders as normal markdown outside any code span, producing a bold `**Amount:**` label followed by an inline code value.

The call chain is fully reachable by any unprivileged principal:

1. `icrc21_canister_call_consent_message` (public `#[update]` endpoint) [3](#0-2) 

2. → `build_icrc21_consent_info` [4](#0-3) 

3. → `ConsentMessageBuilder::build` → `message.add_memo(memo)` [5](#0-4) 

4. → `add_memo` with `GenericMemo::Icrc1Memo(memo.0)` sourced directly from the decoded `TransferArg` [6](#0-5) 

---

### Impact Explanation

The `GenericDisplayMessage` is a markdown string that ICRC-21-compliant wallets render and display to users as the human-readable description of a transaction they are about to sign. A malicious dApp can:

1. Construct an `icrc1_transfer` with a large amount to an attacker-controlled address.
2. Embed a memo such as (within the 32-byte limit): `x`\n\n**Amount:** `1 ICP` (≈ 25 bytes).
3. Pass this as the `arg` in a `ConsentMessageRequest` to `icrc21_canister_call_consent_message`.
4. The returned `ConsentInfo.consent_message` will contain injected markdown that renders a fake `**Amount:**` field visually indistinguishable from the real one.
5. A user who skims the consent message may approve a transaction for a much larger amount than displayed.

The `FieldsDisplayMessage` variant is **not** affected because the memo is stored as a structured `Value::Text` field, not rendered as markdown. [7](#0-6) 

---

### Likelihood Explanation

- The endpoint is public and callable by any principal with no authentication requirement beyond a valid caller identity.
- The memo field accepts up to 32 bytes of arbitrary bytes by default (configurable upward), which is sufficient to inject convincing fake fields.
- The attack requires only crafting a `ConsentMessageRequest` with a malicious `TransferArg` memo — no privileged access, no key material, no governance majority.
- Wallets that implement ICRC-21 and render `GenericDisplayMessage` as markdown are directly exposed.

---

### Recommendation

Escape backtick characters in `memo_str` before interpolating into the markdown format string, or replace the inline code span with a fenced code block that cannot be broken by embedded backticks of a different run length. A minimal fix:

```rust
let safe_memo = memo_str.replace('`', "\\`");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, hex-encode all memo values unconditionally in the `GenericDisplayMessage` path to eliminate the UTF-8 display path entirely.

---

### Proof of Concept

Craft a `TransferArg` with memo bytes equal to the UTF-8 string:

```
x`\n\n**Amount:** `1 ICP
```

(24 bytes — within the default 32-byte limit)

Call `icrc21_canister_call_consent_message` with `method = "icrc1_transfer"` and this arg. The returned `GenericDisplayMessage` will contain:

```markdown
# Send ICP

...

**Amount:** `1000 ICP`

**To:**
`attacker-address`

...

**Memo:**
`x`

**Amount:** `1 ICP
```

The injected `**Amount:** \`1 ICP` renders as a bold label with inline code, visually identical to the legitimate `**Amount:**` field above it, potentially misleading a user into believing the transfer is for 1 ICP rather than 1000 ICP.
