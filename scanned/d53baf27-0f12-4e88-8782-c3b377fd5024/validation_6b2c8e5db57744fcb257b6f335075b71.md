### Title
SNS Governance Lacks Voting Power Spike Detection, Enabling Immediate Majority Treasury Drain After Swap Finalization - (File: rs/sns/governance/src/governance.rs)

---

### Summary

SNS governance does not implement the voting power spike detection mechanism present in NNS governance. An attacker who participates in an SNS swap with a sufficiently large ICP contribution acquires full, immediately usable voting power upon swap finalization. Because SNS governance computes ballots from the live neuron set with no historical comparison, the attacker can immediately submit and pass a `TransferSnsTreasuryFunds` proposal — a critical proposal type — to drain the SNS treasury up to the 7-day rolling limit, with no cooldown between acquiring voting power and exercising it.

---

### Finding Description

**NNS governance has a voting power spike detection mechanism.** In `rs/nns/governance/src/governance.rs`, `compute_ballots_for_standard_proposal` calls `VOTING_POWER_SNAPSHOTS.with_borrow(|snapshots| snapshots.previous_ballots_if_voting_power_spike_detected(...))`. The `VotingPowerSnapshots` structure (in `rs/nns/governance/src/governance/voting_power_snapshots.rs`) stores up to 7 daily snapshots and detects a spike when the current total potential voting power exceeds 1.5× the minimum total in the retained snapshots. When a spike is detected, ballots are created from the previous (lower) snapshot, so a newly staked large neuron cannot immediately pass proposals. [1](#0-0) [2](#0-1) 

**SNS governance has no equivalent mechanism.** `compute_ballots_for_new_proposal` in `rs/sns/governance/src/governance.rs` iterates over all live neurons at the current timestamp with no historical snapshot comparison and no spike detection. [3](#0-2) 

**SNS swap neurons receive full, immediately usable voting power.** In `claim_swap_neurons`, each neuron is created with `voting_power_percentage_multiplier: DEFAULT_VOTING_POWER_PERCENTAGE_MULTIPLIER` (100%) and `vesting_period_seconds: None`. There is no lock-up or cooldown before the neuron can vote. [4](#0-3) 

**`TransferSnsTreasuryFunds` is a critical proposal.** Its topic is `TreasuryAssetManagement` with `ProposalCriticality::Critical`, requiring 67% of exercised voting power and 20% of total voting power for early decision. [5](#0-4) [6](#0-5) 

**Early decision is triggered immediately upon voting.** `ProposalData::early_decision()` is evaluated during `cast_vote_and_cascade_follow`. If the attacker holds enough voting power and votes YES, the proposal is decided and executed in the same round — no minimum waiting period applies to early decision. [7](#0-6) 

**A 7-day rolling limit on treasury transfers exists but is only a partial mitigation.** `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` caps the total transferred within 7 days based on treasury valuation. For a large treasury this still permits draining a significant fraction (e.g., 25% for a "medium" treasury). The attacker can repeat the drain every 7 days. [8](#0-7) 

---

### Impact Explanation

An attacker who acquires a controlling fraction of SNS tokens through the swap can, in the same round that swap finalization completes:

1. Submit a `TransferSnsTreasuryFunds` proposal targeting their own account.
2. Vote YES with their newly acquired neurons.
3. Trigger early decision immediately (no waiting period).
4. Drain the SNS treasury up to the 7-day rolling cap.

They retain majority voting power permanently and can repeat the drain every 7 days, or use other critical proposal types (`MintSnsTokens`, `DeregisterDappCanisters`, `UpgradeSnsControlledCanister`) to cause further harm. Unlike the EVM flash-loan variant, no same-block repayment is needed — the attacker keeps both the tokens and the treasury funds.

---

### Likelihood Explanation

The attack requires the attacker to acquire a controlling fraction of SNS tokens during the swap. This is constrained by `max_participant_icp_e8s` in the swap parameters. However:

- Many SNS deployments set `max_participant_icp_e8s` to a high value or equal to `max_icp_e8s`.
- The attacker can coordinate across multiple principals to bypass per-participant caps.
- The NNS Neurons' Fund participates in swaps and its contribution is counted toward total ICP, which can dilute the attacker's fraction — but if the fund's participation is small relative to the attacker's, the attack remains viable.
- No on-chain mechanism in SNS governance prevents immediate use of swap-acquired voting power.

---

### Recommendation

Implement a voting power spike detection mechanism in SNS governance analogous to the NNS `VotingPowerSnapshots` mechanism:

1. Maintain periodic snapshots of total SNS voting power in `rs/sns/governance/src/governance.rs`.
2. In `compute_ballots_for_new_proposal`, compare the current total voting power against the historical minimum. If a spike (e.g., >1.5×) is detected, use the previous snapshot for ballot creation.
3. Alternatively, enforce a minimum cooldown period (e.g., one `initial_voting_period_seconds`) between swap finalization (`set_sns_governance_to_normal_mode`) and the ability to pass critical proposals.

---

### Proof of Concept

**Setup:**
- SNS swap with `max_icp_e8s = 100_000 ICP`, `max_participant_icp_e8s = 100_000 ICP`, `min_participants = 1`.
- SNS treasury holds 10,000 ICP and 10,000 SNS tokens.
- ICP/XDR rate set so treasury is "medium" (25% drain allowed per 7 days = 2,500 ICP).

**Steps:**

1. Attacker contributes 100,000 ICP to the SNS swap, acquiring ~100% of swap tokens (or a controlling majority if developer neurons have reduced `voting_power_percentage_multiplier`).

2. Swap commits. `finalize_swap` → `claim_swap_neurons` creates attacker's neurons with `voting_power_percentage_multiplier = 100`, `vesting_period_seconds = None`, and `dissolve_delay_seconds` meeting `neuron_minimum_dissolve_delay_to_vote_seconds`. [4](#0-3) 

3. SNS governance mode is set to `Normal` via `set_sns_governance_to_normal_mode`. Attacker's neurons are now eligible to vote. [9](#0-8) 

4. Attacker immediately calls `manage_neuron` → `MakeProposal` with `Action::TransferSnsTreasuryFunds { from_treasury: IcpTreasury, amount_e8s: 2_500 * E8, to_principal: attacker }`.

5. `compute_ballots_for_new_proposal` iterates over all live neurons — attacker's neurons dominate. No spike detection is performed. [3](#0-2) 

6. Attacker votes YES. `early_decision()` evaluates: attacker's yes votes exceed the critical threshold (67% of exercised). Proposal is decided immediately. [7](#0-6) 

7. `perform_transfer_sns_treasury_funds` executes, transferring 2,500 ICP to the attacker. [10](#0-9) 

8. Attacker waits 7 days and repeats, draining another 25% of the remaining treasury.

**Contrast with NNS:** If the same attacker staked a large NNS neuron, `compute_ballots_for_standard_proposal` would detect the voting power spike and use the previous snapshot, excluding the attacker's new neuron from the ballot — the proposal would not pass immediately. [11](#0-10)

    /// Computes the total potential voting power of the governance canister and ballots.
    fn compute_ballots_for_new_proposal(&self) -> Result<(u64, BTreeMap<String, Ballot>), String> {
        let now_seconds = self.env.now();

        let nervous_system_parameters = self.nervous_system_parameters_or_panic();

        // Voting power bonus parameters.
        let max_dissolve_delay = nervous_system_parameters
            .max_dissolve_delay_seconds
            .expect("NervousSystemParameters must have max_dissolve_delay_seconds");

        let max_age_bonus = nervous_system_parameters
            .max_neuron_age_for_age_bonus
            .expect("NervousSystemParameters must have max_neuron_age_for_age_bonus");

        let max_dissolve_delay_bonus_percentage = nervous_system_parameters
            .max_dissolve_delay_bonus_percentage
            .expect("NervousSystemParameters must have max_dissolve_delay_bonus_percentage");

        let max_age_bonus_percentage = nervous_system_parameters
            .max_age_bonus_percentage
            .expect("NervousSystemParameters must have max_age_bonus_percentage");

        let min_dissolve_delay_for_vote = nervous_system_parameters
            .neuron_minimum_dissolve_delay_to_vote_seconds
            .expect("NervousSystemParameters must have min_dissolve_delay_for_vote");

        let mut electoral_roll = BTreeMap::<String, Ballot>::new();
        let mut total_power: u128 = 0;

        for (k, v) in self.proto.neurons.iter() {
            // If this neuron is eligible to vote, record its
            // voting power at the time of proposal creation (now).
            if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
                // Not eligible due to dissolve delay.
                continue;
            }

            let voting_power = v.voting_power(
                now_seconds,
                max_dissolve_delay,
                max_age_bonus,
                max_dissolve_delay_bonus_percentage,
                max_age_bonus_percentage,
            );

            total_power += voting_power as u128;
            electoral_roll.insert(
                k.clone(),
                Ballot {
                    vote: Vote::Unspecified as i32,
                    voting_power,
                    cast_timestamp_seconds: 0,
                },
            );
        }

        if total_power >= (u64::MAX as u128) {
            // The way the neurons are configured, the total voting
            // power on this proposal would overflow a u64!
            return Err("Voting power overflow.".to_string());
        }
        if electoral_roll.is_empty() {
            // Cannot make a proposal with no eligible voters.  This
            // is a precaution that shouldn't happen as we check that
            // the voter is allowed to vote.
            return Err("No eligible voters.".to_string());
        }

        Ok((total_power as u64, electoral_roll))

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }

    Ok(())
