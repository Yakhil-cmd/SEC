### Title
Unbounded Neuron Scan in `maybe_finalize_disburse_maturity` Can Exhaust Instruction Limit and Freeze SNS Disbursements - (File: rs/sns/governance/src/governance.rs)

### Summary

The SNS governance canister's `maybe_finalize_disburse_maturity` function performs an unbounded full scan of every neuron in the canister's state before processing any maturity disbursements. As the number of neurons grows, this scan consumes an ever-increasing number of instructions. If the instruction limit is exhausted, the disbursement finalization timer permanently fails to make progress, freezing all pending maturity disbursements for the SNS.

### Finding Description

`maybe_finalize_disburse_maturity` is called unconditionally on every `run_periodic_tasks` invocation. It begins by iterating over the entire `self.proto.neurons` map to collect all neurons with a ready disbursement: [1](#0-0) 

This is an O(N) scan over all neurons with no instruction-limit guard, no batching, and no early-exit mechanism. After collecting the ready set, it then loops over every collected disbursement and issues an async ledger transfer for each one: [2](#0-1) 

Neither the scan nor the disbursement loop contains any call to `is_message_over_threshold` or any equivalent instruction-budget check. The entire function is called from `run_periodic_tasks` with no surrounding guard: [3](#0-2) 

The NNS governance canister has already addressed this exact pattern. It uses a stable-memory index (`get_neuron_ids_ready_to_finalize_maturity_disbursement`) to avoid a full scan and processes exactly one disbursement per timer invocation: [4](#0-3) 

The SNS governance retains the old, unguarded design.

### Impact Explanation

When the number of SNS neurons is large enough that the full scan of `self.proto.neurons.values()` exhausts the per-message instruction limit, the IC runtime traps the message and rolls back all state changes. The timer fires again on the next interval and immediately traps again for the same reason. The result is a permanent livelock: no maturity disbursement for any neuron in the SNS can ever be finalized. Funds that users have already committed to disburse (and whose maturity has already been deducted from their neuron) are permanently locked in the `disburse_maturity_in_progress` queue with no recovery path available to unprivileged users.

### Likelihood Explanation

SNS governance canisters are permissionlessly deployed and can accumulate thousands of neurons over time. Each neuron can hold multiple `disburse_maturity_in_progress` entries (one per `DisburseMaturity` call). Any SNS token holder can call `DisburseMaturity` to add entries. A sufficiently popular SNS with many active stakers who have all initiated disbursements around the same time (e.g., after a large reward event) can organically reach the threshold. The IC's per-message instruction limit for update calls is 40 billion instructions; reading a neuron from heap memory costs on the order of thousands of instructions, so tens of thousands of neurons with pending disbursements would be sufficient to trigger the condition.

### Recommendation

1. **Replace the full scan with an index**, analogous to the NNS fix: maintain a secondary index of neurons that have a ready disbursement, and look up only those neurons instead of scanning all of them.
2. **Process at most one disbursement per timer invocation**, as the NNS `FinalizeMaturityDisbursementsTask` does, so that each execution is O(1) in the number of neurons.
3. **Add an instruction-budget guard** using `is_message_over_threshold` (already used in NNS timer tasks such as `PruneFollowingTask`) so that if the scan does grow large, the function exits gracefully and reschedules itself rather than trapping. [5](#0-4) 

### Proof of Concept

1. Deploy an SNS with a large number of neurons (e.g., 50,000+).
2. Have a significant fraction of those neurons call `DisburseMaturity`, populating `disburse_maturity_in_progress` entries.
3. Advance time past the 7-day disbursement delay so all entries become eligible.
4. Observe that `run_periodic_tasks` → `maybe_finalize_disburse_maturity` traps on every invocation due to instruction exhaustion during the `self.proto.neurons.values().filter_map(...)` scan at lines 4938–4975.
5. Confirm that no disbursement is ever finalized and the `disburse_maturity_in_progress` queues remain permanently non-empty, with no callable function available to an unprivileged user to unblock the situation. [6](#0-5) [7](#0-6)

    /// Runs periodic tasks that are not directly triggered by user input.
    pub async fn run_periodic_tasks(&mut self) {
        use ic_cdk::println;

        self.process_proposals();

        // None of the upgrade-related tasks should interleave with one another or themselves, so we acquire a global
        // lock for the duration of their execution. This will return `false` if the lock has already been acquired less
        // than 10 minutes ago by a previous invocation of `run_periodic_tasks`, in which case we skip the
        // upgrade-related tasks.
        if self.acquire_upgrade_periodic_task_lock() {
            // We only want to check the upgrade status if we are currently executing an upgrade.
            if self.should_check_upgrade_status() {
                self.check_upgrade_status().await;
            }

            if self.should_refresh_cached_upgrade_steps() {
                match self.try_temporarily_lock_refresh_cached_upgrade_steps() {
                    Err(err) => {
                        log!(ERROR, "{}", err);
                    }
                    Ok(deployed_version) => {
                        self.refresh_cached_upgrade_steps(deployed_version).await;
                    }
                }
            }

            self.initiate_upgrade_if_sns_behind_target_version().await;

            self.release_upgrade_periodic_task_lock();
        }

        let should_distribute_rewards = self.should_distribute_rewards();

        // Getting the total governance token supply from the ledger is expensive enough
        // that we don't want to do it on every call to `run_periodic_tasks`. So
        // we only fetch it when it's needed, which is when rewards should be
        // distributed
        if should_distribute_rewards {
            match self.ledger.total_supply().await {
                Ok(supply) => {
                    // Distribute rewards
                    self.distribute_rewards(supply);
                }
                Err(e) => log!(
                    ERROR,
                    "Error when getting total governance token supply: {}",
                    GovernanceError::from(e)
                ),
            }
        }

        if self.should_update_maturity_modulation() {
            self.update_maturity_modulation().await;
        }

        self.maybe_finalize_disburse_maturity().await;

        self.maybe_move_staked_maturity();

        self.compute_cached_metrics().await;

        self.maybe_gc();
    }
