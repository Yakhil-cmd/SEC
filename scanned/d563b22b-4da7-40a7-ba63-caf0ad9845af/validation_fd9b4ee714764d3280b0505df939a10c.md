### Title
SNS Governance Voting Reward Purse Permanently Lost When All Voters Have Zero Voting Power — (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

In SNS governance, when all neurons that voted on proposals in a reward round have zero effective voting power, the entire reward purse for that round — including any accumulated rollover from prior rounds — is permanently destroyed with no recovery path. The proposals are marked `Settled`, preventing any future rollover, yet `distributed_e8s_equivalent` is zero. The developers themselves wrote a test (`zero_total_reward_shares`) that confirms this exact outcome.

---

### Finding Description

`distribute_rewards` in `rs/sns/governance/src/governance.rs` computes `total_reward_shares` by summing the `voting_power` field of every ballot whose `vote` is `Yes` or `No` across all `ReadyToSettle` proposals:

```rust
// rs/sns/governance/src/governance.rs ~5892-5934

for proposal_id in &considered_proposals {
    if let Some(proposal) = self.get_proposal_data(*proposal_id) {
        for (voter, ballot) in &proposal.ballots {
            if !Vote::try_from(ballot.vote)...eligible_for_rewards() {
                continue;
            }
            ...
            *neuron_id_to_reward_shares.entry(neuron_id)... += i2d(ballot.voting_power);
        }
    }
}
let total_reward_shares: Decimal = neuron_id_to_reward_shares.values().sum();
``` [1](#0-0) 

When `total_reward_shares == dec!(0)` — which occurs whenever every ballot that was cast carries `voting_power = 0` — the distribution branch is skipped entirely:

```rust
if total_reward_shares == dec!(0) {
    log!(ERROR, "Warning: total_reward_shares is 0. ...");
} else {
    // distribute maturity to neurons
}
``` [2](#0-1) 

Regardless of whether rewards were distributed, the function then **settles all considered proposals** and writes a new `RewardEvent` with `settled_proposals` populated:

```rust
// rs/sns/governance/src/governance.rs ~6084-6092

    ...
    settled_proposals: considered_proposals,   // non-empty
    distributed_e8s_equivalent,               // 0
    total_available_e8s_equivalent,           // non-zero purse
});
``` [3](#0-2) 

The rollover gate in `rs/sns/governance/src/types.rs` is keyed exclusively on whether `settled_proposals` is empty:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty()
}
pub(crate) fn e8s_equivalent_to_be_rolled_over(&self) -> u64 {
    if self.rewards_rolled_over() { self.total_available_e8s_equivalent.unwrap_or_default() }
    else { 0 }
}
``` [4](#0-3) 

Because `settled_proposals` is non-empty, `e8s_equivalent_to_be_rolled_over()` returns **0**. The next call to `distribute_rewards` starts with a fresh purse and the current round's `total_available_e8s_equivalent` is gone forever. Any accumulated rollover from prior rounds (included in `rewards_purse_e8s` via `e8s_equivalent_to_be_rolled_over()` at the start of the function) is also consumed and lost. [5](#0-4) 

The developers explicitly tested and accepted this behavior in `rs/sns/integration_tests/src/neuron.rs`:

```rust
// zero_total_reward_shares test
assert_eq!(reward_event.distributed_e8s_equivalent, 0, ...);
// settled_proposals is [proposal_id] — non-empty — so no rollover occurs
``` [6](#0-5) 

The test constructs a neuron where `cached_neuron_stake_e8s == neuron_fees_e8s`, giving `voting_power = 0`. The neuron votes `Yes`, so the ballot is eligible for rewards but contributes zero to `total_reward_shares`. The reward purse is silently discarded. [7](#0-6) 

---

### Impact Explanation

**Impact: High**

The entire `rewards_purse_e8s` for the affected round — which includes the current round's inflation-derived maturity **plus** any accumulated rollover from previous rounds where no proposals were settled — is permanently destroyed. There is no admin escape hatch, no governance proposal to recover it, and no retry path. Neurons that should have received maturity receive nothing, and the shortfall does not carry forward. For an SNS with a large accumulated rollover (e.g., many consecutive rounds with no proposals), a single zero-voting-power round can wipe out a substantial maturity pool.

---

### Likelihood Explanation

**Likelihood: Low**

The trigger condition — all ballots on all `ReadyToSettle` proposals carrying `voting_power = 0` — can arise in at least two realistic ways:

1. **Depleted neurons**: Any SNS neuron where `neuron_fees_e8s >= cached_neuron_stake_e8s` has `voting_power = 0`. If every neuron that voted in a round is in this state (possible in early-stage or low-activity SNS instances), the condition fires.
2. **Universal abstention**: If no neuron casts a `Yes` or `No` vote (all remain `Unspecified`), `neuron_id_to_reward_shares` is empty and `total_reward_shares == 0`. The SNS governance `compute_ballots_for_new_proposal` initialises all ballots to `Vote::Unspecified`, so a round where no neuron actively votes satisfies the condition.

The condition is not attacker-gated — it fires automatically via the `run_periodic_tasks` heartbeat — but it requires a degenerate participation state that is uncommon on healthy, active SNS instances.

---

### Recommendation

When `total_reward_shares == dec!(0)` **and** `considered_proposals` is non-empty, treat the round as a rollover rather than a settled event. Concretely, either:

- Do not advance `settled_proposals` in the `RewardEvent` (leave it empty so `rewards_rolled_over()` returns `true`), or
- Explicitly set `total_available_e8s_equivalent` in the new event to the full purse value so `e8s_equivalent_to_be_rolled_over()` returns it to the next round.

This mirrors the existing rollover logic that already handles the "no proposals to settle" case correctly.

---

### Proof of Concept

1. Deploy an SNS governance canister with `VotingRewardsParameters` enabled.
2. Create a neuron `N` with `cached_neuron_stake_e8s = 1_000_000_000` and `neuron_fees_e8s = 1_000_000_000` → `voting_power = 0`.
3. Submit a `Motion` proposal; `N`'s ballot is created with `voting_power = 0, vote = Unspecified`.
4. Have `N` vote `Yes` (ballot becomes `vote = Yes, voting_power = 0`).
5. Advance time past the voting deadline so the proposal enters `ReadyToSettle`.
6. Trigger `run_periodic_tasks` (heartbeat or direct call).
7. Observe: `latest_reward_event.settled_proposals = [proposal_id]`, `distributed_e8s_equivalent = 0`, `total_available_e8s_equivalent > 0`.
8. Trigger `run_periodic_tasks` again in the next round.
9. Observe: `e8s_equivalent_to_be_rolled_over()` returns **0** for the previous event (because `settled_proposals` was non-empty), so the purse from step 7 is gone.

This is the exact scenario exercised by the `zero_total_reward_shares` integration test at `rs/sns/integration_tests/src/neuron.rs:1168`. [8](#0-7)
