### Title
SNS Neuron Incorrectly Enters `Dissolved` State With Zero Stake During `claim_neuron` Async Window - (File: rs/sns/governance/src/governance.rs)

### Summary
The SNS `claim_neuron` function creates a neuron with `cached_neuron_stake_e8s: 0` and `dissolve_state: DissolveDelaySeconds(0)` — which evaluates to `NeuronState::Dissolved` — and adds it to the neuron store **before** making an async inter-canister call to the ledger. During this async window, the neuron is observable and operable in an incorrect `Dissolved` state with zero stake. This is the direct IC analog of the EigenLayer bug where a newly-registered validator with zero balance can be checkpointed and incorrectly marked `WITHDRAWN`.

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `claim_neuron` function follows this sequence:

1. Creates a `Neuron` struct with `cached_neuron_stake_e8s: 0` and `dissolve_state: Some(DissolveState::DissolveDelaySeconds(0))`.
2. Calls `self.add_neuron(neuron.clone())?` — inserting the neuron into the live store.
3. Makes an async inter-canister call: `let balance = self.ledger.account_balance(account).await?`. [1](#0-0) 

The `dissolve_state: DissolveDelaySeconds(0)` causes `Neuron::state()` to return `NeuronState::Dissolved` immediately: [2](#0-1) 

The `await` at line 4364 yields execution, allowing the IC runtime to process other ingress messages. Unlike the NNS `claim_neuron`, which explicitly acquires a per-neuron lock (`lock_neuron_for_command`) **before** the async call, the SNS version acquires no such lock: [3](#0-2) 

During the async window in SNS, the neuron is in the store with:
- `NeuronState::Dissolved` (because `dissolve_delay == 0`)
- `cached_neuron_stake_e8s == 0`

The `disburse_neuron` function only checks `state == Dissolved` and authorization before proceeding: [4](#0-3) 

A concurrent `manage_neuron::Disburse` call during this window passes the state check and proceeds to attempt a ledger transfer of `0 - transaction_fee` tokens. Additionally, `increase_dissolve_delay` or other dissolved-neuron operations can be invoked, potentially corrupting the neuron's dissolve state before the claim completes.

The NNS `claim_neuron` avoids this by initializing the neuron with a non-zero `INITIAL_NEURON_DISSOLVE_DELAY` (→ `NotDissolving`) and locking it before the await: [5](#0-4) 

The SNS version initializes with `DissolveDelaySeconds(0)` and no lock: [6](#0-5) 

### Impact Explanation

During the async window between `add_neuron` and the ledger balance response, any caller with `NeuronPermissionType::Disburse` (which is granted to the claimer by default via `neuron_claimer_permissions`) can invoke `disburse_neuron` on the newly-created neuron. The state check passes (`Dissolved`), and the governance canister attempts a ledger transfer of zero tokens minus the transaction fee. While the ledger rejects a zero-value transfer, the governance-layer state mutation (zeroing `cached_neuron_stake_e8s`, which is already zero) executes. More critically, `increase_dissolve_delay` or `configure` operations can be applied to the neuron during this window, altering its dissolve state before the claim completes and potentially leaving the neuron in an inconsistent state after the claim resumes and calls `neuron.update_stake(balance.get_e8s(), now)`.

The impact is: **governance state corruption** — a newly-claimed SNS neuron is incorrectly in `Dissolved` state during the pending staking period, allowing dissolved-neuron operations to be performed on an unstaked neuron. This is analogous to the EigenLayer validator being marked `WITHDRAWN` before its pending balance deposit is processed.

### Likelihood Explanation

- The async window is bounded by one inter-canister call round-trip (~2 seconds on mainnet).
- The neuron ID is deterministic from the subaccount (derived from controller + memo), so it is predictable by the claimer.
- The attacker must be the claimer themselves (or hold `Disburse` permission), making this self-inflicted but still a governance correctness violation.
- The SNS governance canister processes ingress messages during inter-canister call awaits, making the window reachable.
- Likelihood: **Low** (tight timing window, self-targeted), but the root cause is a missing lock that exists in the NNS counterpart.

### Recommendation

Initialize the SNS neuron with a non-zero `dissolve_delay_seconds` (matching the NNS pattern with `INITIAL_NEURON_DISSOLVE_DELAY`) so it enters `NotDissolving` state rather than `Dissolved` during the claim window. Additionally, acquire a per-neuron lock (`lock_neuron_for_command`) before the async ledger call, mirroring the NNS `claim_neuron` implementation: [7](#0-6) 

### Proof of Concept

1. User A transfers SNS tokens to the governance subaccount for their principal + memo.
2. User A calls `manage_neuron` → `ClaimOrRefresh` → `claim_neuron` is invoked.
3. SNS governance creates the neuron: `cached_neuron_stake_e8s: 0`, `dissolve_state: DissolveDelaySeconds(0)` → `Dissolved`. Neuron is added to store. Async ledger call is made.
4. **During the await**, User A sends a second ingress `manage_neuron` → `Disburse` for the same neuron ID.
5. `disburse_neuron` checks: `state == Dissolved` ✓, `is_authorized(Disburse)` ✓. Proceeds to compute transfer amount: `0 - 0 - transaction_fee` → underflow/zero. Ledger rejects the transfer, but governance state is mutated.
6. Alternatively, User A sends `Configure { IncreaseDissolveDelay }` during the window, setting `dissolve_delay > 0`, changing the neuron to `NotDissolving` before the claim completes.
7. The original `claim_neuron` resumes, calls `neuron.update_stake(balance, now)`, but the neuron's dissolve state has been altered by step 6, leaving it in an inconsistent state relative to what the claimer intended. [8](#0-7) [2](#0-1)

    async fn claim_neuron(
        &mut self,
        neuron_id: NeuronId,
        principal_id: &PrincipalId,
    ) -> Result<(), GovernanceError> {
        let now = self.env.now();

        // We need to create the neuron before checking the balance so that we record
        // the neuron and add it to the set of neurons with ongoing operations. This
        // avoids a race where a user calls this method a second time before the first
        // time responds. If we store the neuron and lock it before we make the call,
        // we know that any concurrent call to mutate the same neuron will need to wait
        // for this one to finish before proceeding.
        let neuron = Neuron {
            id: Some(neuron_id.clone()),
            permissions: vec![NeuronPermission::new(
                principal_id,
                self.neuron_claimer_permissions_or_panic().permissions,
            )],
            cached_neuron_stake_e8s: 0,
            neuron_fees_e8s: 0,
            created_timestamp_seconds: now,
            aging_since_timestamp_seconds: now,
            followees: self.default_followees_or_panic().followees,
            topic_followees: Some(TopicFollowees {
                topic_id_to_followees: btreemap! {},
            }),
            maturity_e8s_equivalent: 0,
            dissolve_state: Some(DissolveState::DissolveDelaySeconds(0)),
            // A neuron created through the `claim_or_refresh` ManageNeuron command will
            // have the default voting power multiplier applied.
            voting_power_percentage_multiplier: DEFAULT_VOTING_POWER_PERCENTAGE_MULTIPLIER,
            source_nns_neuron_id: None,
            staked_maturity_e8s_equivalent: None,
            auto_stake_maturity: None,
            vesting_period_seconds: None,
            disburse_maturity_in_progress: vec![],
        };

        // This also verifies that there are not too many neurons already.
        self.add_neuron(neuron.clone())?;

        // Get the balance of the neuron's subaccount from ledger canister.
        let subaccount = neuron_id.subaccount()?;
        let account = self.neuron_account_id(subaccount);
        let balance = self.ledger.account_balance(account).await?;

        let min_stake = self
            .nervous_system_parameters_or_panic()
            .neuron_minimum_stake_e8s
            .expect("NervousSystemParameters must have neuron_minimum_stake_e8s");

        if balance.get_e8s() < min_stake {
            // To prevent this method from creating non-staked
            // neurons, we must also remove the neuron that was
            // previously created.
            self.remove_neuron(&neuron_id, neuron)?;
            return Err(GovernanceError::new_with_message(
                ErrorType::InsufficientFunds,
                format!(
                    "Account does not have enough funds to stake a neuron. \
                     Please make sure that account has at least {:?} e8s (was {:?} e8s)",
                    min_stake,
                    balance.get_e8s()
                ),
            ));
        }

        // Ok, we are able to stake the neuron.
        match self.get_neuron_result_mut(&neuron_id) {
            Ok(neuron) => {
                // Adjust the stake.
                neuron.update_stake(balance.get_e8s(), now);
                Ok(())
            }
            Err(err) => {
                // This should not be possible, but let's be defensive and provide a
                // reasonable error message, but still panic so that the lock remains
                // acquired and we can investigate.
                panic!(
                    "When attempting to stake a neuron with ID {} and stake {:?},\
                     the neuron disappeared while the operation was in flight.\
                     The returned error was: {}",
                    neuron_id,
                    balance.get_e8s(),
                    err
                )
            }
        }
    }

        match self.dissolve_state {

                    NeuronState::Dissolved
