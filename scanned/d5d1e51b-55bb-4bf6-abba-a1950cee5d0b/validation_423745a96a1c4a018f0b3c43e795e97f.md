### Title
Floating-Point Precision Loss in ICRC-21 Consent Message Token Amount Display Causes Incorrect User-Facing Amounts - (File: packages/icrc-ledger-types/src/icrc21/responses.rs)

---

### Summary

`convert_tokens_to_string_representation` in the ICRC-21 consent message library converts an arbitrary-precision `Nat` token amount to `f64` before dividing by `10^decimals`. Because `f64` has only 53 bits of mantissa (~15–16 significant decimal digits), any token amount exceeding `2^53 ≈ 9 × 10^15` is silently rounded. For tokens with 18 decimals (ckETH, ckERC-20), this threshold is crossed at amounts as small as ~0.009 ETH. The rounded value is what the wallet displays to the user in the `GenericDisplayMessage` consent screen — the exact security boundary ICRC-21 exists to protect.

---

### Finding Description

The private helper function `convert_tokens_to_string_representation` is called by `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` whenever the consent message variant is `GenericDisplayMessage`:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```

`Nat` is an arbitrary-precision unsigned integer. `to_f64()` maps it to the nearest representable IEEE-754 double. For any integer value `N > 2^53 = 9_007_199_254_740_992`, consecutive integers are no longer representable in `f64`, so the conversion silently rounds. The rounded value is then divided by `10^decimals` and formatted as the human-readable amount shown to the user.

For ckETH (18 decimals), `2^53` wei ≈ 0.009 ETH, meaning **every ckETH amount above ~0.009 ETH** is displayed with a precision error. For a `u64::MAX` wei approval (`18_446_744_073_709_551_615`), the nearest `f64` is `18_446_744_073_709_551_616.0` — off by 1 wei. For amounts near `2^64` or larger (valid for ICRC-1 `Nat` allowances), the rounding error grows to thousands of wei or more.

The `FieldsDisplayMessage` variant is unaffected — it uses `nat_to_u64` which is exact (though limited to `u64`). Only `GenericDisplayMessage` is vulnerable. [1](#0-0) 

The four affected call sites are `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance`: [2](#0-1) [3](#0-2) [4](#0-3) 

These are assembled by `ConsentMessageBuilder::build` and served to wallets via the `icrc21_consent_message` endpoint on every ICRC-1/ICRC-2 ledger: [5](#0-4) 

---

### Impact Explanation

ICRC-21 consent messages are the **sole user-visible security boundary** between a dApp-constructed transaction and the user's signature. Hardware wallets (Ledger, etc.) and software wallets display exactly this string before the user approves. If the displayed amount is rounded down, a malicious dApp can craft an `icrc2_approve` or `icrc1_transfer` argument whose actual on-chain amount is larger than what the user sees and approves. The user signs believing they approved `X` tokens; the ledger records approval for `X + Δ` tokens (where `Δ` is the rounding error). The dApp can then `icrc2_transfer_from` the full `X + Δ`.

For ckETH (18 decimals), the rounding error at `u64::MAX` wei is 1 wei (~$0.000000003). For tokens with fewer decimals or for `Nat`-typed allowances exceeding `u64::MAX`, the error grows. More importantly, the security property of ICRC-21 — that the displayed amount exactly matches the signed amount — is broken for any `GenericDisplayMessage` consumer.

---

### Likelihood Explanation

Any unprivileged caller can invoke `icrc21_consent_message` on any ICRC-1/ICRC-2 ledger canister (ICP ledger, ckETH ledger, ckERC-20 ledgers, SNS ledgers) with a crafted argument whose amount is just above `2^53` in the token's smallest denomination. No special privileges, keys, or governance majority are required. The attacker only needs to construct a transaction argument with a specific numeric value and present it to a user via a dApp.

---

### Recommendation

Replace the lossy `f64` conversion with exact integer arithmetic. Use the same approach as `format_amount` in the ckBTC/ckDOGE minters — integer division and modulo:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let divisor = num_bigint::BigUint::from(10u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let frac = &tokens.0 % &divisor;
    if frac.is_zero() {
        Ok(format!("{whole}"))
    } else {
        let frac_str = format!("{frac:0>width$}", width = decimals as usize);
        let trimmed = frac_str.trim_end_matches('0');
        Ok(format!("{whole}.{trimmed}"))
    }
}
```

This mirrors the exact integer approach already used in `format_amount`: [6](#0-5)

### Proof of Concept

1. Call `icrc21_consent_message` on any ICRC-1 ledger (e.g., ckETH) with a `TransferArg` where `amount = Nat::from(9_007_199_254_740_993u64)` (= `2^53 + 1` wei, just above the f64 exact-integer threshold).
2. Request `GenericDisplayMessage` display type.
3. Observe the returned consent message displays `**Amount:** \`0.009007199254740992 ckETH\`` (rounded down by 1 wei) instead of the correct `0.009007199254740993 ckETH`.
4. A user who approves this consent message has signed a transfer for 1 wei more than the displayed amount.

The rounding is deterministic and reproducible for any amount `> 2^53` in the token's base denomination. For `icrc2_approve` with `amount = Nat::from(u64::MAX)`, the displayed allowance is `18.446744073709552 ckETH` while the actual allowance recorded on-chain is `18.446744073709551615 ckETH` — the consent message overstates the allowance by `~3.85 × 10^-16 ckETH`.
