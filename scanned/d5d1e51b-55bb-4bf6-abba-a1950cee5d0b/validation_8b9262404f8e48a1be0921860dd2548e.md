### Title
SNS Dapp Canister State Can Be Manipulated by Original Controller During Registration Window — (File: `rs/sns/root/src/lib.rs`)

---

### Summary

During SNS dapp canister registration, `register_dapp_canister` performs an async `canister_status` check followed by an async `update_settings` call to make SNS Root the sole controller. Between these two inter-canister calls, the original dapp controller — who retains full controller rights — can install malicious code, drain cycles, or delete state. SNS participants who voted to acquire the dapp receive it in a state that may differ materially from what was evaluated at proposal time.

---

### Finding Description

`SnsRootCanister::register_dapp_canister` in `rs/sns/root/src/lib.rs` executes the following sequence across two async await points:

1. **Status check** — calls `management_canister_client.canister_status(canister_to_register)` to verify SNS Root is already a controller.
2. **Guard check** — if SNS Root is not the sole controller, proceeds to evict all other controllers.
3. **Controller eviction** — calls `management_canister_client.update_settings(...)` with `controllers: Some(vec![root_canister_id])`. [1](#0-0) 

Between the `.await` at step 1 and the `.await` at step 3, the IC message-routing layer can interleave ingress messages from any principal. The original dapp controller — still listed as a controller at the time of the status check — can submit an ingress `install_code` call to the management canister (`ic:00`) targeting the dapp canister. Because any controller may install code on a canister, this call succeeds before SNS Root's `update_settings` removes the original controller.

The same structural window exists in the SNS-W deployment path, where `canister_status` calls are issued per-dapp before the batch `change_canister_controllers` calls: [2](#0-1) 

The `register_dapp_canister` function explicitly acknowledges the async gap but provides no re-validation after the controller change: [3](#0-2) 

---

### Impact Explanation

An original dapp controller who is "selling" their dapp into an SNS can, during the inter-canister call window:

- **Install malicious Wasm** — backdoors, token-draining logic, or logic that reverts the canister to a prior state.
- **Drain cycles** — causing the dapp to stop executing shortly after SNS participants take ownership.
- **Delete stable memory or heap state** — destroying user data or protocol state that was the basis of the SNS valuation.

SNS participants receive a canister whose code and state differ from what was evaluated during the governance proposal, directly analogous to a buyer receiving an NFT-owned vault from which the seller has removed collateral before settlement.

**Vulnerability class**: governance authorization bug / state manipulation during pending transfer.

---

### Likelihood Explanation

- The original dapp controller is an unprivileged ingress sender with no special trust assumption beyond being the current canister controller.
- The exploit window is one inter-canister call round-trip (~2 seconds on mainnet), which is sufficient for an ingress `install_code` message to be included in a subsequent consensus round.
- The attacker has a direct financial incentive: they receive ICP/SNS tokens from the swap while retaining the ability to degrade the asset being transferred.
- No threshold corruption, governance majority, or privileged key is required.

---

### Recommendation

1. **Re-validate after eviction**: After `update_settings` completes, call `canister_status` again and verify the module hash matches the hash recorded at proposal submission time. Reject registration if it differs.
2. **Snapshot module hash at proposal time**: Record the dapp canister's Wasm module hash in the SNS init payload and enforce it at registration time.
3. **Document the risk**: Until a technical fix is deployed, clearly communicate to SNS launch participants that the original dapp controller retains the ability to modify the dapp between the governance proposal passing and the controller transfer completing.

---

### Proof of Concept

```
1. Original dapp controller D adds SNS Root R as a co-controller of dapp canister C.
   (Pre-condition required by the SNS launch process.)

2. SNS Root R calls register_dapp_canister(C).

3. R issues: management_canister.canister_status(C)  [async, awaits response]
   → Response confirms controllers = [D, R]. R proceeds.

4. ** WINDOW OPENS **
   D submits ingress to ic:00: install_code { canister_id: C, wasm_module: <malicious_wasm>, ... }
   This is accepted because D is still a controller of C.
   The IC processes D's ingress in the next round while R's continuation is queued.
   ** WINDOW CLOSES when R's update_settings executes **

5. R issues: management_canister.update_settings(C, controllers=[R])
   → D is removed. R is now sole controller.

6. R adds C to dapp_canister_ids. Registration succeeds.

7. SNS participants finalize the swap and receive governance over C,
   which now runs D's malicious Wasm.
```

The attacker-controlled entry path is a standard ingress `install_code` call to `ic:00` from principal D, which is a fully reachable, unprivileged operation for any canister controller. The IC code is the necessary vulnerable step because `register_dapp_canister` does not hold any lock or freeze on the dapp canister during the async gap. [4](#0-3)

    async fn change_controllers_of_nns_root_owned_canisters(
        nns_root_canister_client: &impl NnsRootCanisterClient,
        dapp_canisters_to_new_controllers: &BTreeMap<Canister, Vec<PrincipalId>>,
    ) -> Result<Vec<Canister>, FailedChangeCanisterControllersResult> {
        let mut result = FailedChangeCanisterControllersResult::new();

        // TODO: parallelize
        for (canister, new_controllers) in dapp_canisters_to_new_controllers {
            // This condition should never be reached due to prior validation, but in case the
            // new controllers is empty, do not change the controllers of the dapp canister.
            if new_controllers.is_empty() {
                result.failed.push(FailedChangeCanisterControllersRequest {
                    canister: *canister,
                    reason: "Tried to request NNS Root to set controllers to an empty set"
                        .to_string(),
                });
                continue;
            }

            match Self::change_controllers_of_nns_root_owned_canister(
                nns_root_canister_client,
                *canister,
                new_controllers.clone(),
            )
            .await
            {
                Ok(_) => result.successful.push(*canister),
                Err(reason) => result.failed.push(FailedChangeCanisterControllersRequest {
                    canister: *canister,
                    reason,
                }),
            };
        }

        if result.failed.is_empty() {
            Ok(dapp_canisters_to_new_controllers.keys().cloned().collect())
        } else {
            println!(
                "{}Failed to transfer control of dapp_canisters. Reason: {}",
                LOG_PREFIX,
                result.join_failed_reasons()
            );
            Err(result)
        }
    }
