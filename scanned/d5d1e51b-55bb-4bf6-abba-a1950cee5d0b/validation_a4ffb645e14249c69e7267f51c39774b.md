### Title
Integer Division Truncation in `Swap::scale` Allows SNS Swap Participants to Lose ICP Without Receiving SNS Tokens — (`rs/sns/swap/src/swap.rs`)

### Summary

In the SNS Swap canister, the `Swap::scale` function computes each participant's SNS token allocation via integer division. When the numerator is smaller than the denominator, the result truncates to zero. The calling function `create_sns_neuron_recipes` does not guard against a zero result, so neuron recipes with `amount_e8s = 0` are silently created and marked as successfully processed. The participant's ICP is swept to SNS governance regardless, leaving them with no SNS tokens or valid neurons.

### Finding Description

`Swap::scale` computes a participant's SNS token share as:

```
(amount_icp_e8s * sns_being_offered_e8s) / total_participant_icp_e8s
``` [1](#0-0) 

When the numerator is smaller than the denominator, Rust integer division truncates to zero. The caller `create_sns_neuron_recipes` performs no zero-check on the returned `amount_sns_e8s` before passing it to `create_sns_neuron_basket_for_direct_participant`: [2](#0-1) 

`generate_vesting_schedule(0)` calls `apportion_approximately_equally(0, count)`, which succeeds and returns `vec![0; count]`. Neuron recipes with `amount_e8s = 0` are created and `has_created_neuron_recipes` is set to `Some(true)`, permanently marking the participant as processed. [3](#0-2) 

The same silent-zero path exists for Neurons' Fund participants: [4](#0-3) 

**Validation gap:** The pre-swap validation in `validate_participation_constraints` (check 8) computes the minimum SNS tokens a participant receives using `max_direct_participation_icp_e8s` as the denominator: [5](#0-4) 

However, at finalization the actual denominator is `total_participant_icp_e8s`, which includes Neurons' Fund participation. When NF participation is non-zero, the actual SNS tokens received are strictly less than what the validation computed, and the validation's guarantee no longer holds.

### Impact Explanation

A direct swap participant contributes ICP (which is swept to SNS governance via `sweep_icp`), but receives neuron recipes with `amount_e8s = 0`. When `sweep_sns` subsequently attempts to transfer `0 - fee` tokens, the transfer underflows and fails. The participant's ICP is permanently transferred to SNS governance with no SNS tokens or neurons created in return — a direct loss of funds.

### Likelihood Explanation

The condition `amount_icp_e8s * sns_token_e8s < total_participant_icp_e8s` is required for `amount_sns_e8s = 0`. This becomes reachable when Neurons' Fund participation is large relative to direct participation, causing `total_participant_icp_e8s` to significantly exceed `max_direct_participation_icp_e8s`. The validation gap (using `max_direct_participation_icp_e8s` instead of `max_direct + max_NF`) means the pre-swap check can pass while the finalization computation still produces zero. Likelihood is low but non-zero for swaps with substantial NF participation.

### Recommendation

1. Add an explicit zero-check in `create_sns_neuron_recipes` after calling `Swap::scale`. If `amount_sns_e8s == 0`, mark the participant as `invalid` (not `success`) and do not set `has_created_neuron_recipes = Some(true)`, so the ICP sweep can be aborted or refunded.

2. Fix the validation in `validate_participation_constraints` to use `max_direct_participation_icp_e8s + max_neurons_fund_participation_icp_e8s` as the denominator, matching the actual finalization computation.

### Proof of Concept

Given a swap with:
- `sns_token_e8s = 1_000_000` (10 SNS tokens)
- `min_participant_icp_e8s = 1` (1 e8)
- `max_direct_participation_icp_e8s = 1_000_000`
- NF participation = `2_000_000` e8s

At finalization: `total_participant_icp_e8s = direct + NF = 1_000_001 + 2_000_000 = 3_000_001`.

For a participant with `amount_icp_e8s = 1`:
```
scale(1, 1_000_000, 3_000_001) = (1 * 1_000_000) / 3_000_001 = 0
```

The validation passed (using `max_direct_participation_icp_e8s = 1_000_000` as denominator), but `amount_sns_e8s = 0`. The participant's ICP is swept; no SNS neurons are created. [1](#0-0) [6](#0-5) [7](#0-6)

                    dissolve_delay_seconds,
                    amount_e8s,
                },
            )

    fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
        assert!(amount_icp_e8s <= u64::from(total_icp_e8s));
        // Note that the multiplication cannot overflow as both factors fit in 64 bits.
        let r = (amount_icp_e8s as u128)
            .saturating_mul(total_sns_e8s as u128)
            .div(NonZeroU128::from(total_icp_e8s));
        // This follows logically from the initial assert `amount_icp_e8s <= total_icp_e8s`.
        assert!(r <= u64::MAX as u128);
        r as u64
    }
