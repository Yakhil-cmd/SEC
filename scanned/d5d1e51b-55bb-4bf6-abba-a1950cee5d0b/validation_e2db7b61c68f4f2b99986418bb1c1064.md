### Title
No Validation of Minter Address Parameter in Deposit Helper Contract Constructors - (File: rs/ethereum/cketh/minter/EthDepositHelper.sol, ERC20DepositHelper.sol, DepositHelperWithSubaccount.sol)

### Summary
Three chain-fusion deposit helper Solidity contracts (`CkEthDeposit`, `CkErc20Deposit`, `CkDeposit`) store a critical `immutable` minter address set exclusively at construction time, with no zero-address or validity check on the constructor parameter. Because the field is `immutable`, there is no recovery path after deployment. If the wrong address is supplied, all subsequent ETH/ERC20 deposits are forwarded to the wrong destination while the deposit events are still emitted, causing the IC ckETH minter canister to mint ckETH/ckERC20 tokens against ETH/ERC20 that it never received.

### Finding Description

**`EthDepositHelper.sol` — `CkEthDeposit`:**

The constructor assigns `_cketh_minter_main_address` directly to the `immutable` field with no check:

```solidity
constructor(address _cketh_minter_main_address) {
    cketh_minter_main_address = payable(_cketh_minter_main_address);
}
``` [1](#0-0) 

The `deposit()` function then emits the `ReceivedEth` event **before** transferring ETH to the stored address:

```solidity
function deposit(bytes32 _principal) public payable {
    emit ReceivedEth(msg.sender, msg.value, _principal);
    cketh_minter_main_address.transfer(msg.value);
}
``` [2](#0-1) 

**`ERC20DepositHelper.sol` — `CkErc20Deposit`:**

Same pattern: constructor stores `_cketh_minter_main_address` without validation:

```solidity
constructor(address _cketh_minter_main_address) {
    cketh_minter_main_address = _cketh_minter_main_address;
}
``` [3](#0-2) 

The `deposit()` function transfers ERC20 tokens to the stored address:

```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
``` [4](#0-3) 

**`DepositHelperWithSubaccount.sol` — `CkDeposit`:**

Same pattern for `minterAddress`:

```solidity
constructor(address _minterAddress) {
    minterAddress = payable(_minterAddress);
}
``` [5](#0-4) 

Notably, this contract **does** validate the ERC20 token address in `depositErc20()`:

```solidity
require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
``` [6](#0-5) 

...but applies no equivalent check to `minterAddress` in the constructor, creating an inconsistency where the token address is guarded but the destination minter address is not.

### Impact Explanation

The `cketh_minter_main_address` / `minterAddress` field is declared `immutable` in all three contracts: [7](#0-6) [8](#0-7) [9](#0-8) 

There is no setter, no upgrade path, and no owner mechanism. A deployment with `address(0)` or any incorrect address is **permanent and irrecoverable** without redeploying the contract and migrating users.

The chain-fusion impact is:
- ETH/ERC20 sent by users is forwarded to the wrong address (burned or stolen), while the `ReceivedEth` / `ReceivedErc20` / `ReceivedEthOrErc20` events are still emitted on-chain.
- The IC ckETH minter canister observes these events via canister HTTP outcalls and mints ckETH/ckERC20 to the depositing user.
- Result: ckETH/ckERC20 tokens are minted without the corresponding ETH/ERC20 backing, breaking the 1:1 collateralization invariant of the chain-fusion bridge and causing direct financial loss to the protocol reserve.

### Likelihood Explanation

Likelihood is **low but non-negligible**. The deployer is the DFINITY team, which is a trusted role. However:
- The `immutable` pattern provides zero defense-in-depth: a single deployment mistake has no recovery path.
- The inconsistency in `DepositHelperWithSubaccount.sol` (validating `erc20Address` but not `minterAddress`) demonstrates that the pattern of zero-address checks is already partially applied, making the omission for the minter address an oversight rather than a deliberate design choice.
- These contracts are intended to be deployed on Ethereum mainnet; any misconfiguration is permanent and publicly visible only after user funds are lost.

### Recommendation

Add a zero-address guard in each constructor:

```solidity
// EthDepositHelper.sol
constructor(address _cketh_minter_main_address) {
    require(_cketh_minter_main_address != address(0), "minter address cannot be zero");
    cketh_minter_main_address = payable(_cketh_minter_main_address);
}
```

Apply the same pattern to `ERC20DepositHelper.sol` and `DepositHelperWithSubaccount.sol`. This mirrors the existing `erc20Address` check already present in `DepositHelperWithSubaccount.sol::depositErc20()`.

### Proof of Concept

1. Deploy `CkEthDeposit` with `_cketh_minter_main_address = address(0)`.
2. Any user calls `deposit(principal)` with `msg.value = 1 ETH`.
3. The `ReceivedEth` event is emitted with `from=user, value=1 ETH, principal=principal`.
4. `address(0).transfer(1 ETH)` succeeds — ETH is burned.
5. The IC ckETH minter canister observes the `ReceivedEth` event and mints 1 ckETH to the user.
6. The ckETH supply increases by 1 with no ETH backing, breaking the bridge's collateralization invariant.
