### Title
Unprivileged Caller Can Abort Any In-Progress SNS Upgrade via `fail_stuck_upgrade_in_progress` - (File: rs/sns/governance/canister/canister.rs)

### Summary
The `fail_stuck_upgrade_in_progress` update method on the SNS Governance canister has no caller authorization check. Any unprivileged ingress sender can invoke it to abort an in-progress SNS upgrade and mark the associated governance proposal as failed, once the 5-minute deadline has elapsed. This is a direct analog to the reported pattern: a state-mutating function callable by anyone without restriction.

### Finding Description
The canister endpoint `fail_stuck_upgrade_in_progress` is declared as a public `#[update]` method with no caller check: [1](#0-0) 

The underlying implementation in `rs/sns/governance/src/governance.rs` performs only a time-based guard — it checks whether `now > pending_version.mark_failed_at_seconds` — and if so, calls `complete_sns_upgrade_to_next_version`: [2](#0-1) 

`complete_sns_upgrade_to_next_version` unconditionally clears `pending_version` (the upgrade lock) and marks the associated governance proposal as failed: [3](#0-2) 

The `mark_failed_at_seconds` deadline is set to exactly 5 minutes after the upgrade starts: [4](#0-3) 

There is no check on `msg_caller()` anywhere in the call path. The developers themselves noted the missing guard in a comment at line 6337: *"Maybe, we should look at the checking_upgrade_lock field and only proceed if it is false, or the request has force set to true."* [5](#0-4) 

### Impact Explanation
Any unprivileged principal on the Internet Computer can send an ingress update call to `fail_stuck_upgrade_in_progress` on any deployed SNS Governance canister. Once 5 minutes have elapsed since an upgrade started, the attacker can:

1. Abort the in-progress SNS upgrade by clearing `pending_version`.
2. Force the associated governance proposal into a `Failed` state.
3. Repeat this for every subsequent upgrade attempt, indefinitely blocking the SNS from upgrading its canisters.

This is a governance disruption attack: the SNS community must re-submit and re-vote on upgrade proposals each time, while the attacker can abort each attempt with a single zero-cost ingress call after the 5-minute window. The `checking_upgrade_lock` field (which causes the periodic task to skip the check when `lock > 1`) is not checked by `fail_stuck_upgrade_in_progress`, meaning the attacker can bypass the periodic task's own re-entrancy guard.

### Likelihood Explanation
The attack requires no privileged access, no tokens, and no neuron. The only precondition is that an SNS upgrade proposal has been adopted and is executing, and that 5 minutes have elapsed. SNS upgrades that encounter any inter-canister call latency or transient failure can remain in `pending_version` state beyond the 5-minute window. The attack is trivially scriptable: monitor `get_running_sns_version` for a non-null `pending_version` with an elapsed `mark_failed_at_seconds`, then call `fail_stuck_upgrade_in_progress`.

### Recommendation
Add a caller authorization check to `fail_stuck_upgrade_in_progress`. The method should only be callable by a principal with governance authority (e.g., a neuron holder via `manage_neuron`, or restricted to the SNS governance canister itself). Alternatively, require the caller to hold a neuron with sufficient voting power, consistent with how other sensitive SNS operations are gated. At minimum, emit an upgrade journal entry that records the caller's principal ID so that unauthorized invocations are auditable on-chain.

### Proof of Concept
1. Observe an SNS with an active upgrade proposal (check `get_running_sns_version` for a non-null `pending_version`).
2. Wait until `now > pending_version.mark_failed_at_seconds` (5 minutes after upgrade start).
3. Send an ingress update call to the SNS Governance canister's `fail_stuck_upgrade_in_progress` method with an empty `FailStuckUpgradeInProgressRequest {}` from any principal (including the anonymous identity).
4. Observe that `pending_version` is now `None` and the upgrade proposal is marked `Failed`.
5. Repeat for each subsequent upgrade attempt to permanently block SNS upgrades.

    pub fn fail_stuck_upgrade_in_progress(
        &mut self,
        _: FailStuckUpgradeInProgressRequest,
    ) -> FailStuckUpgradeInProgressResponse {
        let pending_version = match self.proto.pending_version.as_ref() {
            None => return FailStuckUpgradeInProgressResponse {},
            Some(pending_version) => pending_version,
        };

        // Maybe, we should look at the checking_upgrade_lock field and only
        // proceed if it is false, or the request has force set to true.

        let now = self.env.now();

        if now > pending_version.mark_failed_at_seconds {
            let message = format!(
                "Upgrade marked as failed at {}. \
                Governance upgrade was manually aborted by calling fail_stuck_upgrade_in_progress \
                after mark_failed_at_seconds ({}). Setting upgrade to failed to unblock retry.",
                format_timestamp_for_humans(now),
                pending_version.mark_failed_at_seconds,
            );
            let status = upgrade_journal_entry::upgrade_outcome::Status::ExternalFailure(Empty {});

            self.complete_sns_upgrade_to_next_version(
                pending_version.proposal_id,
                status,
                message,
                None,
            );
        }

        FailStuckUpgradeInProgressResponse {}
    }
