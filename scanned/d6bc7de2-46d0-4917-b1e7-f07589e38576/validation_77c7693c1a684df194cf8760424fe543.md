### Title
Duplicate Followees in NNS Neuron Following Inflate Cascade-Vote Majority Threshold — (`rs/nns/governance/src/neuron/types.rs`)

---

### Summary

The NNS governance `Follow` command accepts and stores duplicate `NeuronId` entries in a neuron's followees list without deduplication. The cascade-vote function `would_follow_ballots` then iterates over this list and counts yes/no votes against `followees.len()` — which includes duplicates — as the denominator. An unprivileged ingress sender who owns a neuron can exploit this to artificially lower the majority threshold required to trigger a cascade vote, making their neuron vote in a direction that does not reflect the true majority of unique followees.

---

### Finding Description

**Step 1 — Duplicate followees are accepted without deduplication.**

The `follow` function in `rs/nns/governance/src/governance.rs` validates only that the followees list does not exceed `MAX_FOLLOWEES_PER_TOPIC` in length. It performs no uniqueness check on the `NeuronId` values supplied: [1](#0-0) 

The followees are stored verbatim: [2](#0-1) 

This is explicitly acknowledged in the codebase. The neuron data validation test confirms that duplicate followees are considered valid state: [3](#0-2) 

**Step 2 — Cascade voting counts duplicates without deduplication.**

`would_follow_ballots` iterates over the raw `followees` vector and counts yes/no votes, then compares against `followees.len()` — which includes any duplicates: [4](#0-3) 

**Step 3 — Threshold manipulation.**

Consider a neuron with followees `[A, A, B]` (neuron A duplicated). `followees.len() == 3`. If A votes Yes:
- `yes = 2` (A counted twice)
- `yes * 2 = 4 > 3` → cascade **Yes**

Without duplicates `[A, B]`, if A votes Yes:
- `yes = 1`
- `yes * 2 = 2 > 2` is **false** → no cascade

An attacker can fill all `MAX_FOLLOWEES_PER_TOPIC` slots with the same `NeuronId`, giving that single neuron 100% weight in the cascade decision and guaranteeing their neuron always follows it, regardless of what other unique followees vote.

---

### Impact Explanation

An unprivileged neuron owner can manipulate the NNS cascade-vote mechanism so that their neuron votes in a direction that does not reflect the true majority of unique followees. In the extreme case (all slots filled with one `NeuronId`), a single neuron can be given 100% cascade weight. This undermines the integrity of the NNS governance process: a neuron can appear to follow a diverse set of neurons while actually being controlled by a single one, enabling covert governance manipulation without direct voting.

---

### Likelihood Explanation

Any principal who owns an NNS neuron can submit a `manage_neuron` `Follow` command with duplicate `NeuronId` entries via a standard ingress message. No special privileges, admin keys, or threshold corruption are required. The attack is cheap, repeatable, and leaves no on-chain evidence distinguishing it from legitimate following.

---

### Recommendation

1. **Deduplicate followees at ingestion.** In the `follow` function, convert the incoming `followees` list to a `HashSet<NeuronId>` before storing, or reject requests containing duplicate `NeuronId` values with an `InvalidCommand` error.
2. **Deduplicate at vote counting.** In `would_follow_ballots`, deduplicate the followees slice before iterating, so that `followees.len()` reflects the number of unique followees.
3. **Add a validation rule** to `NeuronDataValidator` that flags neurons with duplicate followees as a data integrity issue, enabling detection and remediation of existing state.

---

### Proof of Concept

An attacker with a neuron calls `manage_neuron` with:

```
Follow {
    topic: <any topic>,
    followees: [NeuronId { id: X }, NeuronId { id: X }, NeuronId { id: Y }]
}
```

where X is a neuron the attacker controls or expects to vote Yes, and Y is a neuron expected to vote No.

When a proposal is created and X votes Yes, `would_follow_ballots` computes:
- `yes = 2` (X counted twice), `no = 1`, `followees.len() = 3`
- `2 * 2 = 4 > 3` → attacker's neuron cascades **Yes**

Without duplicates (`[X, Y]`), the same votes yield:
- `yes = 1`, `no = 1`, `followees.len() = 2`
- `1 * 2 = 2 > 2` is false → **no cascade**

The attacker's neuron votes Yes despite Y (a unique followee) voting No, violating the majority-of-unique-followees invariant that the cascade mechanism is designed to enforce. [5](#0-4) [6](#0-5)

    pub(crate) fn would_follow_ballots(
        &self,
        topic: Topic,
        ballots: &HashMap<u64, Ballot>,
    ) -> Vote {
        // Compute the list of followees for this topic. If no
        // following is specified for the topic, use the followees
        // from the 'Unspecified' topic.
        if let Some(followees) = self
            .followees
            .get(&(topic as i32))
            .or_else(|| self.followees.get(&(Topic::Unspecified as i32)))
            // extract plain vector from 'Followees' proto
            .map(|x| &x.followees)
        {
            // If, for some reason, a list of followees is specified
            // but empty (this is not normal), don't vote 'no', as
            // would be the natural result of the algorithm below, but
            // instead don't cast a vote.
            if followees.is_empty() {
                return Vote::Unspecified;
            }
            let mut yes: usize = 0;
            let mut no: usize = 0;
            for f in followees.iter() {
                if let Some(f_vote) = ballots.get(&f.id) {
                    if f_vote.vote == (Vote::Yes as i32) {
                        yes = yes.saturating_add(1);
                    } else if f_vote.vote == (Vote::No as i32) {
                        no = no.saturating_add(1);
                    }
                }
            }
            if yes.saturating_mul(2_usize) > followees.len() {
                return Vote::Yes;
            }
            if no.saturating_mul(2_usize) >= followees.len() {
                return Vote::No;
            }
        }
        // No followees specified.
        Vote::Unspecified
    }
