### Title
Deprecated 64-bit `ic0_msg_cycles_available` System API in Production Canisters Enables Targeted Denial-of-Service via Oversized Cycle Attachment - (`rs/nns/cmc/src/main.rs`, `rs/nns/sns-wasm/canister/canister.rs`, `rs/bitcoin/checker/src/main.rs`)

---

### Summary

Multiple production system canisters use the deprecated 64-bit `ic0_msg_cycles_available` system API instead of its 128-bit successor `ic0_msg_cycles_available128`. Per the IC interface specification, `ic0_msg_cycles_available` **traps** if the cycle amount attached to the incoming message cannot fit in a 64-bit value. An unprivileged canister caller who attaches more than `u64::MAX` cycles to a call targeting these canisters will cause the message handler to trap, resulting in a targeted denial-of-service on that specific endpoint invocation.

---

### Finding Description

The IC system API explicitly marks `ic0_msg_cycles_available` as deprecated:

> `(deprecated) Please use ic0_msg_cycles_available128 instead. This API supports only 64-bit values. Traps if the amount of cycles available cannot fit in a 64-bit value.` [1](#0-0) 

The 128-bit successor `ic0_msg_cycles_available128` was introduced precisely to handle the full 128-bit cycle range supported by the IC protocol. The deprecated 64-bit variant is a hard trap boundary, not a silent truncation.

Despite this, the following production canister files call `msg_cycles_available()` (the 64-bit wrapper over `ic0_msg_cycles_available`):

- **Cycles Minting Canister** (`rs/nns/cmc/src/main.rs`) — 2 call sites
- **SNS-WASM Canister** (`rs/nns/sns-wasm/canister/canister.rs`) — 2 call sites
- **Bitcoin Checker Canister** (`rs/bitcoin/checker/src/main.rs`) — 1 call site

The `dfn_core` library exposes the 64-bit wrapper `call_cycles_add` and `msg_cycles_available` without deprecation warnings at the call site: [2](#0-1) 

The CMC library file even suppresses deprecation warnings globally with `#![allow(deprecated)]`: [3](#0-2) 

This confirms the deprecated API usage is known but unaddressed.

---

### Impact Explanation

The IC protocol supports 128-bit cycle values in messages. `u64::MAX` is approximately `1.84 × 10^19` cycles. Any canister caller that attaches more than `u64::MAX` cycles to a call targeting the CMC, SNS-WASM, or Bitcoin checker will cause the receiving canister's message handler to **trap** at the `msg_cycles_available()` call. The trap:

1. Aborts the entire message execution for that invocation.
2. Refunds the attached cycles to the caller (since the call failed before `msg_cycles_accept` was reached).
3. Does not affect other concurrent or subsequent messages — this is a per-invocation DoS, not a persistent state corruption.

For the **Cycles Minting Canister** specifically, this means an attacker can reliably cause any CMC endpoint that reads `msg_cycles_available()` to fail on demand, disrupting the cycle-minting flow for that call. For the **Bitcoin checker**, which charges cycles as payment for BTC transaction verification, the trap prevents fee collection and service delivery for that request.

---

### Likelihood Explanation

`u64::MAX` cycles ≈ 18.4 million XDR worth of cycles at the standard 1 XDR = 10^12 cycles rate. This is a large but finite amount. The attack requires:

1. A canister (not a user principal) as the caller — only canisters can attach cycles to calls.
2. The attacking canister must hold or be topped up with >u64::MAX cycles.

The cost is high (~$24M USD equivalent at current XDR rates), making this unlikely for opportunistic attackers. However, a well-funded adversary targeting critical NNS infrastructure (CMC controls all cycle minting on the IC) has a clear incentive. The attack is also **repeatable at zero marginal cost** after the initial cycle acquisition, since cycles are refunded on trap.

---

### Recommendation

Replace all uses of the deprecated 64-bit `msg_cycles_available()` with `msg_cycles_available128()` in production canister code:

- `rs/nns/cmc/src/main.rs` — replace both `msg_cycles_available()` call sites with `msg_cycles_available128()`
- `rs/nns/sns-wasm/canister/canister.rs` — replace both call sites
- `rs/bitcoin/checker/src/main.rs` — replace the single call site
- `rs/rust_canisters/dfn_core/src/api.rs` — mark `call_cycles_add` and `msg_cycles_available` as `#[deprecated]` to surface usage at compile time

Similarly audit `ic0_canister_cycle_balance` usage (also deprecated, traps if balance > u64::MAX) in `rs/ethereum/cketh/minter/src/main.rs`, `rs/bitcoin/ckbtc/minter/src/metrics.rs`, `rs/ledger_suite/icrc1/ledger/src/main.rs`, and related files. [4](#0-3) 

---

### Proof of Concept

1. Deploy an attacker canister on the IC mainnet.
2. Top up the attacker canister with `u64::MAX + 1` cycles (e.g., via the CMC's `notify_top_up` with sufficient ICP).
3. From the attacker canister, call any CMC endpoint that reads `msg_cycles_available()` (e.g., `mint_cycles`) and attach `u64::MAX + 1` cycles via `ic0.call_cycles_add128`.
4. The CMC handler reaches `msg_cycles_available()` → `ic0_msg_cycles_available` → **trap** because the 64-bit API cannot represent the attached amount.
5. The call fails; cycles are refunded to the attacker canister. The attack is repeatable.
