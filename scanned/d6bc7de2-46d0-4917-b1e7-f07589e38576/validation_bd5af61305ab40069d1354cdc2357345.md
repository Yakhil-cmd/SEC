### Title
Confidential Security-Incident Metadata Leaked to Any Unprivileged Query Caller via Incomplete Redaction in Rate-Limit Canister - (File: `rs/boundary_node/rate_limits/canister/canister.rs`)

---

### Summary

The rate-limit canister's `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` query methods expose `rule_id` and `incident_id` fields for **non-disclosed (confidential) security incidents** to any unprivileged, unauthenticated caller. The `ConfigConfidentialityFormatter` and `RuleConfidentialityFormatter` only redact `rule_raw` and `description`, leaving the existence and UUID identity of every confidential security incident enumerable by any principal on the Internet Computer.

---

### Finding Description

The rate-limit canister (`rs/boundary_node/rate_limits/canister/`) is designed to store confidential rate-limit rules tied to security incidents. Rules remain confidential until explicitly disclosed via `disclose_rules`. The intent is that only authorized principals (the `authorized_principal` or registered API boundary nodes) can see the full content of non-disclosed rules.

The `inspect_message` hook enforces this for **replicated** (update-call) invocations of `get_config`: [1](#0-0) 

However, `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` are all declared `#[query]`, meaning they are reachable as **non-replicated query calls** that entirely bypass `inspect_message`. The access control then falls to the in-method `AccessLevelResolver`, which calls `ConfigConfidentialityFormatter::format` for unauthorized callers: [2](#0-1) 

This formatter only nulls out `rule_raw` and `description`. It leaves `rule_id` and `incident_id` intact for every non-disclosed rule: [3](#0-2) 

The `OutputRule` type returned to the caller therefore always contains `rule_id` and `incident_id` even when the rule has never been disclosed: [4](#0-3) 

An identical partial-redaction gap exists in `RuleGetter` (used by `get_rule_by_id`) and `IncidentGetter` (used by `get_rules_by_incident_id`): [5](#0-4) 

---

### Impact Explanation

Any unprivileged principal — including the anonymous principal — can issue a non-replicated query call to `get_config` and receive a complete list of every `rule_id` and `incident_id` stored in the canister, including those for security incidents that have never been publicly disclosed. This reveals:

1. **The existence and count** of confidential security incidents.
2. **The UUIDs** of those incidents, which can be correlated with external information (e.g., forum posts, GitHub issues, timing of boundary-node deployments).
3. **Version metadata** (`added_in_version`, `removed_in_version`) that reveals when each confidential incident was introduced and retired.

The design intent — keeping non-disclosed rules confidential until `disclose_rules` is called — is violated. This is directly analogous to the ZecWallet finding: sensitive metadata (there, transaction addresses/amounts; here, security-incident UUIDs and version timestamps) persists in an accessible store and is readable by any unprivileged party.

---

### Likelihood Explanation

**High.** The `get_config` query endpoint is publicly reachable by any principal on the IC without any authentication. No cycles, no special identity, and no prior knowledge are required. The call is a standard non-replicated query, the cheapest and most accessible call type on the IC. Any observer who wishes to enumerate confidential security incidents can do so at any time.

---

### Recommendation

**Short term:** Extend `ConfigConfidentialityFormatter::format` and `RuleConfidentialityFormatter::format` to also redact `incident_id` (and optionally `rule_id`) for non-disclosed rules when the caller has `AccessLevel::RestrictedRead`. A stable opaque placeholder (e.g., a hash of the real UUID salted with a canister secret) can be returned instead of the raw UUID to preserve structural consistency without leaking the real identifier.

**Long term:** Audit all `OutputRule` / `OutputRuleMetadata` fields for information that could reveal the existence or timing of confidential security incidents. Consider whether `added_in_version` and `removed_in_version` should also be suppressed for non-disclosed rules.

---

### Proof of Concept

```
# Any principal (including anonymous) can call:
dfx canister call <rate_limit_canister_id> get_config '(null)' --query

# Response for an unauthorized caller will include, for every non-disclosed rule:
# { rule_id = "b97730ac-4879-47f2-9fea-daf20b8d4b64";
#   incident_id = "389bbff8-bffa-4430-bb70-8ce1ea399c07";
#   rule_raw = null;          # correctly redacted
#   description = null;       # correctly redacted
# }
# The incident_id UUID is exposed despite the rule never having been disclosed.
```

The integration test at `rs/boundary_node/rate_limits/integration_tests/tests/rate_limit_canister_tests.rs` (step 5) confirms that an unauthorized principal receives a response with `is_redacted = true` but still obtains `incident_id` for the non-disclosed rule: [6](#0-5)
