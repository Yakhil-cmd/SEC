### Title
Unauthenticated `reset_timers` Bypasses Cool-Down When `timers` Is `None`, Allowing Any Caller to Disrupt SNS Periodic Task Scheduling - (Files: `rs/sns/governance/canister/canister.rs`, `rs/sns/root/canister/canister.rs`, `rs/sns/swap/canister/canister.rs`)

---

### Summary

The `reset_timers` update method exposed on the SNS Governance, SNS Root, and SNS Swap canisters performs no caller authentication. Any unprivileged ingress sender can invoke it. In the SNS Governance and SNS Root implementations the cool-down guard is wrapped in a doubly-nested `if let Some(...)` chain: when `timers` or `last_reset_timestamp_seconds` is `None` the guard is silently skipped and `init_timers()` executes unconditionally — directly analogous to the `emergencyWithdrawableEffectiveTime == 0` always-true bypass in the reference report. Even when the `Option` fields are populated, the absence of any `caller()` check means any principal can trigger a timer reset once per cool-down window.

---

### Finding Description

**Root cause — missing `None` guard (direct analog to the reference report):**

In `rs/sns/governance/canister/canister.rs` and `rs/sns/root/canister/canister.rs` the cool-down check reads:

```rust
if let Some(timers) = governance_mut().proto.timers          // guard 1
    && let Some(last_reset_timestamp_seconds) = timers.last_reset_timestamp_seconds  // guard 2
{
    assert!(
        now_seconds().saturating_sub(last_reset_timestamp_seconds)
            >= reset_timers_cool_down_interval_seconds, ...
    );
}
init_timers();   // always reached when either guard fails
``` [1](#0-0) [2](#0-1) 

If `proto.timers` is `None` **or** `last_reset_timestamp_seconds` is `None`, both guards fail, the `assert!` is never reached, and `init_timers()` runs immediately — identical in structure to `timestamp >= emergencyWithdrawableEffectiveTime` being trivially true when the field is zero/unset.

The `Timers` struct stores `last_reset_timestamp_seconds` as `Option<u64>`: [3](#0-2) 

A canister upgraded from a version that predates the `timers` field will have `proto.timers == None` until `post_upgrade` completes. During the window between message acceptance and `init_timers()` completing, or if state is partially migrated, the guard is vacuously bypassed.

**Root cause — no caller authentication (primary issue):**

None of the three `reset_timers` implementations check `caller()`. The function is decorated `#[update]` with no `inspect_message` guard and no principal allowlist: [1](#0-0) [4](#0-3) [2](#0-1) 

The SNS Swap variant uses a slightly different condition structure (chained `&&` that panics only when all three sub-conditions are true), but equally lacks any caller check: [5](#0-4) 

`init_timers()` clears the existing `TimerId`, registers a fresh `set_timer_interval`, and resets `last_reset_timestamp_seconds` to `now()` while zeroing `last_spawned_timestamp_seconds`: [6](#0-5) [7](#0-6) 

Cool-down constants are 600 s (governance/swap) and 1 week (root): [8](#0-7) [9](#0-8) 

---

### Impact Explanation

Every call to `reset_timers` cancels the running periodic-task timer and schedules a new one from `now`. Periodic tasks on SNS Governance include reward distribution, neuron maturity spawning, proposal execution, and upgrade-lock checks. On SNS Root they include canister-status polling and dapp-controller management. On SNS Swap they include lifecycle advancement and auto-finalization.

An attacker calling `reset_timers` every 600 seconds on SNS Governance/Swap continuously shifts the next periodic-task execution forward by up to 600 s, creating a sustained delay of governance operations. On SNS Root (1-week cool-down, 1-day task interval) a single call delays the next root periodic task by up to 24 hours. Because the function is publicly callable with no cycles cost beyond the ingress fee, the attack is cheap and repeatable.

---

### Likelihood Explanation

The method is listed in the public Candid interface, callable by any ingress sender including anonymous principal. No staking, neuron ownership, or privileged key is required. The attacker needs only to know the canister ID and call `reset_timers` with an empty `ResetTimersRequest`. This is trivially discoverable from on-chain Candid metadata. [10](#0-9) 

---

### Recommendation

1. **Add a `None` guard** analogous to the reference report's fix — if `timers` is `None`, treat it as "no prior reset has occurred" and **reject** rather than silently bypass:

```rust
let timers = governance_mut().proto.timers
    .expect("timers not initialized; call reset_timers only after canister_init");
let last_reset = timers.last_reset_timestamp_seconds
    .expect("last_reset_timestamp_seconds not set");
assert!(
    now_seconds().saturating_sub(last_reset) >= reset_timers_cool_down_interval_seconds,
    "..."
);
```

2. **Restrict the caller** to a known set of principals (e.g., the SNS root canister, NNS governance, or the canister itself) or at minimum reject the anonymous principal.

### Proof of Concept

```
# Any principal (including anonymous) can call reset_timers on SNS Governance:
dfx canister --network ic call <sns_governance_canister_id> reset_timers '(record {})'
# Returns: (record {})  -- succeeds with no authorization check

# Repeat every 600 seconds to continuously delay periodic task execution.
# On SNS Root, call once per week to delay root periodic tasks by up to 24 hours.
```

The `None`-bypass variant: deploy an SNS canister from a WASM that omits the `timers` field in its state proto, then upgrade to the current WASM. On the first `reset_timers` call after upgrade (before `post_upgrade` sets `timers`), the cool-down check is skipped entirely and `init_timers()` executes unconditionally regardless of how recently the previous reset occurred.
