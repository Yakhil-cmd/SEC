### Title
Lack of Minimum Expected ETH Return Value in `withdraw_eth` Allows Users to Receive Less Than Expected - (File: rs/ethereum/cketh/minter/src/main.rs)

### Summary

The `withdraw_eth` endpoint on the ckETH minter canister accepts a withdrawal `amount` and immediately burns the full amount of ckETH from the user's ledger account, but the actual ETH delivered to the recipient is `amount - max_tx_fee_estimate`, where `max_tx_fee_estimate` is computed asynchronously at processing time based on current Ethereum gas prices. No `min_expected_eth` argument exists, so a user has no on-chain mechanism to bound how much less than `amount` they are willing to accept. If Ethereum gas fees spike significantly between the time the user submits `withdraw_eth` and the time the minter builds and signs the transaction, the user receives materially less ETH than anticipated, with no recourse.

### Finding Description

`withdraw_eth` in `rs/ethereum/cketh/minter/src/main.rs` accepts only `amount`, `recipient`, and an optional `from_subaccount`:

```rust
async fn withdraw_eth(
    WithdrawalArg { amount, recipient, from_subaccount }: WithdrawalArg,
) -> Result<RetrieveEthRequest, WithdrawalError>
``` [1](#0-0) 

The function immediately burns the full `amount` from the caller's ckETH ledger account: [2](#0-1) 

The withdrawal request is then queued. Later, when the minter's background task processes the queue, it calls `create_transaction`, which deducts the current `max_transaction_fee` from `withdrawal_amount`:

```rust
let tx_amount = match request.withdrawal_amount.checked_sub(max_transaction_fee) {
    Some(tx_amount) => tx_amount,
    None => return Err(CreateTransactionError::InsufficientTransactionFee { ... }),
};
``` [3](#0-2) 

The `max_transaction_fee` is derived from the live Ethereum gas fee estimate at processing time, not at call time. The ckETH documentation explicitly acknowledges this:

> "The exact fee deducted depends on the dynamic Ethereum transaction fees used at the time the transaction was created." [4](#0-3) 

There is no field in `WithdrawalArg` (nor in the Candid interface) for a caller-specified minimum acceptable ETH output. The `WithdrawalError` type has no variant for "received less than minimum expected": [5](#0-4) 

The `minimum_withdrawal_amount` stored in minter state is a floor on the input `amount`, not a guarantee on the output: [6](#0-5) 

This parameter is itself changeable via NNS upgrade proposals (as demonstrated by the 2026-05-29 upgrade that reduced it from 0.03 ETH to 0.005 ETH): [7](#0-6) 

### Impact Explanation

A user who calls `withdraw_eth` with `amount = X` ckETH expects to receive approximately `X - current_fee` ETH. Because the ckETH minter processes withdrawals asynchronously (typically within ~6 minutes, but potentially longer during congestion), and because Ethereum gas fees can spike by an order of magnitude within minutes during high-demand periods, the user may receive substantially less ETH than anticipated. The ckBTC burn is irrevocable at the point of the call; there is no cancellation mechanism. The user bears the full downside of fee volatility with no on-chain protection.

### Likelihood Explanation

Ethereum gas fees are well-documented to spike dramatically during periods of high network activity (NFT mints, DeFi events, etc.). The ckETH minter's asynchronous processing model means there is always a window between burn and transaction creation. Any unprivileged user calling `withdraw_eth` is exposed to this risk on every withdrawal. The scenario is realistic and has occurred historically on Ethereum mainnet.

### Recommendation

Add an optional `min_expected_eth: opt nat` field to `WithdrawalArg`. In `withdraw_eth`, after the burn succeeds and before queuing the request, or alternatively in `create_transaction` at processing time, check that `withdrawal_amount - max_transaction_fee >= min_expected_eth`. If the check fails, reimburse the user (minus the ledger transfer fee) and return an appropriate error variant such as `WithdrawalError::SlippageExceeded { received: nat; minimum: nat }`. This mirrors the recommendation in the referenced report and is consistent with standard DeFi slippage-protection patterns.

### Proof of Concept

1. Alice queries `eip_1559_transaction_price` (a query endpoint) and observes `max_transaction_fee = 0.001 ETH`.
2. Alice calls `withdraw_eth` with `amount = 0.1 ETH`, expecting to receive ~0.099 ETH.
3. The minter immediately burns 0.1 ckETH from Alice's ledger account.
4. Before the minter's background task runs, Ethereum gas fees spike 10× (e.g., due to a popular NFT mint).
5. The minter calls `create_transaction` with the new `max_transaction_fee = 0.01 ETH`.
6. Alice receives only 0.09 ETH — 10% less than expected — with no recourse.

The entry path is fully unprivileged: any non-anonymous principal with a sufficient ckETH balance can call `withdraw_eth` via ingress message. The vulnerable code path is:

`withdraw_eth` (burns full amount) → queued `EthWithdrawalRequest` → `create_transaction` (deducts live gas fee) → signed Ethereum transaction sent. [8](#0-7)

/// Creates an EIP-1559 transaction for the given withdrawal request.
/// The transaction fees are paid by the beneficiary,
/// meaning that the fees will be deducted from the withdrawal amount.
///
/// # Errors
/// * `CreateTransactionError::InsufficientTransactionFee` if the ETH withdrawal amount does not cover the transaction fee.
pub fn create_transaction(
    withdrawal_request: &WithdrawalRequest,
    nonce: TransactionNonce,
    gas_fee_estimate: GasFeeEstimate,
    gas_limit: GasAmount,
    ethereum_network: EthereumNetwork,
) -> Result<Eip1559TransactionRequest, CreateTransactionError> {
    assert!(
        gas_limit > GasAmount::ZERO,
        "BUG: gas limit should be non-zero"
    );
    match withdrawal_request {
        WithdrawalRequest::CkEth(request) => {
            let transaction_price = gas_fee_estimate.to_price(gas_limit);
            let max_transaction_fee = transaction_price.max_transaction_fee();
            let tx_amount = match request.withdrawal_amount.checked_sub(max_transaction_fee) {
                Some(tx_amount) => tx_amount,
                None => {
                    return Err(CreateTransactionError::InsufficientTransactionFee {
                        cketh_ledger_burn_index: request.ledger_burn_index,
                        allowed_max_transaction_fee: request.withdrawal_amount,
                        actual_max_transaction_fee: max_transaction_fee,
                    });
                }
            };
            Ok(Eip1559TransactionRequest {
                chain_id: ethereum_network.chain_id(),
                nonce,
                max_priority_fee_per_gas: transaction_price.max_priority_fee_per_gas,
                max_fee_per_gas: transaction_price.max_fee_per_gas,
                gas_limit: transaction_price.gas_limit,
                destination: request.destination,
                amount: tx_amount,
                data: Vec::new(),
                access_list: Default::default(),
            })

**File:** rs/ethereum/cketh/docs/cketh.adoc (L200-208)
```text
Note that the transaction will be made at the cost of the beneficiary meaning that the resulting received amount will be less than the specified withdrawal amount.
The exact fee deducted depends on the dynamic Ethereum transaction fees used at the time the transaction was created.

In more detail, assume that a user calls `withdraw_eth` (after having approved the minter) to withdraw `withdraw_amount` (e.g. 1ckETH) to some address.
Then the minter is going to do the following

. Burn `withdraw_amount` on the ckETH ledger for the IC principal (the caller of `withdraw_eth`).
. Estimate the maximum current cost of a transaction on Ethereum, say `max_tx_fee_estimate`. This `max_tx_fee_estimate` is expected to be large enough to be valid for the few next blocks.
. Issue an Ethereum transaction (via threshold ECDSA) with the value `withdraw_amount - max_tx_fee_estimate`. This requires of course that `withdraw_amount >= max_tx_fee_estimate` and that's why we currently have a conservative minimum value for withdrawals of `30_000_000_000_000_000` wei. This ensures that the minter can always send the transaction to Ethereum if one or several resubmissions are needed if the Ethereum network is congested and fees are increasing rapidly (each resubmission requires an increase of at least 10% of the transaction fee).

* Reduce the minimum ETH withdrawal amount by a factor of 6, from 0.03 ETH (`30_000_000_000_000_000` wei) to 0.005 ETH (`5_000_000_000_000_000` wei) — approximately $10 at current prices. The reasoning is as follows:
    * The current minimum dates back to December 2023, when the ckETH minter was installed (see proposal [126171](https://dashboard.internetcomputer.org/proposal/126171)). At that time ETH traded in a similar USD range (around $2000), but Ethereum mainnet transaction fees were averaging $5–$10 per transaction ([source](https://bitinfocharts.com/comparison/ethereum-transactionfees.html#3y)).
    * Today, Ethereum mainnet fees are in the order of cents and rarely exceed $1.
    * As explained [here](https://github.com/dfinity/ic/blob/14382b5abb14b8e7de2bd4a3fb402ba069b82861/rs/ethereum/cketh/docs/cketh.adoc?plain=1#L208), an order-of-magnitude safety margin is preserved so the minter can always submit the transaction even when the Ethereum network is congested and one or more resubmissions are needed (each resubmission requires at least a 10% fee bump). With current Ethereum fees of ~$0.10–$1, a $10 minimum still preserves the ~10× safety margin even after several fee bumps.
