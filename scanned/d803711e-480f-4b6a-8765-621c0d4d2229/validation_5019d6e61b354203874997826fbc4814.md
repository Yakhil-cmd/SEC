The key question is whether `verify()` in the ingress path rejects small-order public keys. Let me trace the exact code path.

### Title
Ingress Signature Forgery via Small-Order Ed25519 Public Key — (`rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs`, `packages/ic-ed25519/src/lib.rs`)

---

### Summary

The `verify()` function used in the ingress validation path does not check whether the Ed25519 public key is in the prime-order subgroup. The ZIP215 verification equation used by `verify_signature` is trivially satisfiable for any small-order public key, allowing an attacker to forge signatures for the self-authenticating principal derived from such a key without possessing any secret scalar.

---

### Finding Description

**Step 1 — `TryFrom<Vec<u8>>` accepts `[0u8; 32]` without curve validation**

`PublicKeyBytes::try_from` only checks length: [1](#0-0) 

No curve-point or subgroup check is performed. `[0u8; 32]` (y=0, a canonical point of order 4) passes.

**Step 2 — `deserialize_raw` accepts small-order (torsion) points**

The library explicitly documents and tests that `deserialize_raw` accepts torsion points without rejection: [2](#0-1) [3](#0-2) 

18 canonical torsion points are confirmed to pass `deserialize_raw().unwrap()` while failing `is_torsion_free()`. The `[0u8; 32]` point is confirmed to be small-order by the `api/tests.rs` test: [4](#0-3) 

**Step 3 — `verify()` calls `verify_signature` with no torsion guard** [5](#0-4) 

`verify()` calls `deserialize_raw` then immediately calls `verify_signature`. It never calls `verify_public_key()` (which does call `is_torsion_free()`). These are two separate functions and the ingress path only uses `verify()`.

**Step 4 — `verify_signature` uses ZIP215 cofactor equation** [6](#0-5) 

The check is: `8 * (s·B − k·A − R) = 0`

For a small-order public key A with `ord(A) | 8`:
- `8·(k·A) = k·(8·A) = k·0 = 0`
- The equation collapses to: `8·(s·B − R) = 0`
- Setting `R = s·B` for any scalar `s` satisfies this for **any message**, since `s·B − R = 0` (identity), and `8·0 = 0`.

**Step 5 — Ingress validation path never calls `verify_public_key`**

`grep_search` for `verify_public_key` in `rs/validator/**/*.rs` returns **zero matches**.

The path is:

```
validate_user_id_and_signature
  → validate_signature
      → user_public_key_from_bytes   [deserialize_rfc8410_der, no torsion check]
      → validate_signature_plain
          → verify_basic_sig_by_public_key
              → ic_ed25519::PublicKey::deserialize_raw  [no torsion check]
              → pk.verify_signature                     [ZIP215, no torsion check]
``` [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9) 

---

### Impact Explanation

An attacker can:
1. Pick any small-order public key A (e.g., any of the 18 confirmed torsion points, or `[0u8; 32]` if it passes `deserialize_raw` in the deployed version)
2. DER-encode it and derive a self-authenticating principal P = SHA224(DER(A)) ‖ 0x02
3. For any ingress message M, choose any scalar `s`, compute `R = s·B`, submit signature `(R ‖ s)`
4. `verify_signature` returns `Ok(())` for every message, every time

The attacker can submit arbitrary ingress calls attributed to P, causing state mutations on any canister that P has permissions on. The principal P is deterministic and publicly computable.

**Uncertainty note**: Whether `[0u8; 32]` specifically passes `VerifyingKey::from_bytes` in the deployed ed25519-dalek version cannot be confirmed from static analysis alone — the `api/tests.rs` test at line 344 uses `CompressedEdwardsY::decompress()` (not `deserialize_raw`) to confirm decompressibility, so it does not definitively prove `deserialize_raw` succeeds for that exact byte string. However, the attack is definitively real for any of the 18 confirmed torsion points in the `WITH_TORSION` test.

---

### Likelihood Explanation

- No privileged access required; any unprivileged user can submit ingress
- The forged signature is computationally trivial (one scalar multiplication)
- The principal P is deterministic and can be pre-computed
- No secret key material is needed

---

### Recommendation

Add a torsion-freeness check inside `verify()` itself (not only in `verify_public_key()`):

```rust
pub fn verify(sig: &types::SignatureBytes, msg: &[u8], pk: &types::PublicKeyBytes) -> CryptoResult<()> {
    let public_key = ic_ed25519::PublicKey::deserialize_raw(&pk.0)...?;
    if !public_key.is_torsion_free() {
        return Err(CryptoError::MalformedPublicKey { ... });
    }
    public_key.verify_signature(msg, &sig.0)...
}
```

Alternatively, enforce the check in `user_public_key_from_bytes` in `sign_utils.rs` so that any torsion-component key is rejected before reaching verification. [5](#0-4) [9](#0-8)

// Verifies correct user and signature.
fn validate_user_id_and_signature<R: RootOfTrustProvider>(
    ingress_signature_verifier: &dyn IngressSigVerifier,
    sender: &UserId,
    message_id: &MessageId,
    signature: Option<&UserSignature>,
    current_time: Time,
    root_of_trust_provider: &R,
) -> Result<CanisterIdSet, RequestValidationError>
where
    R::Error: std::error::Error,
{
    match signature {
        None => {
            if sender.get().is_anonymous() {
                return Ok(CanisterIdSet::all());
            }
            Err(MissingSignature(*sender))
        }
        Some(signature) => {
            if sender.get().is_anonymous() {
                Err(AnonymousSignatureNotAllowed)
            } else {
                let sender_pubkey = &signature.signer_pubkey;
                validate_user_id(sender_pubkey, sender).and_then(|()| {
                    validate_signature(
                        ingress_signature_verifier,
                        message_id,
                        signature,
                        current_time,
                        root_of_trust_provider,
                    )
                })
            }
        }
    }
}
