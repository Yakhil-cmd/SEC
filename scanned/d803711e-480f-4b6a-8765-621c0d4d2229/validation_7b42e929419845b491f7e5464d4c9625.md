### Title
Missing Slippage Protection in SNS Treasury Manager Deposit Execution Allows Sandwich Attacks - (`rs/sns/governance/src/extensions.rs`)

### Summary

The SNS governance `execute_treasury_manager_deposit` function approves token allowances and calls `deposit` on a TreasuryManager extension canister (DEX adaptor) without any slippage check or price validation at execution time. An unprivileged attacker who monitors public governance proposals can front-run the proposal execution by manipulating the DEX price, causing the SNS treasury to deposit at an unfavorable ratio and suffer a loss of value. The codebase itself acknowledges this risk in a WARNING comment but provides no on-chain mitigation.

---

### Finding Description

The SNS governance extension system allows SNS DAOs to deposit treasury assets into DEXes via `ExecuteExtensionOperation` proposals. When such a proposal executes, `execute_treasury_manager_deposit` is called: [1](#0-0) 

The function performs two steps:
1. Calls `approve_treasury_manager` to grant ICRC-2 allowances to the extension canister for the amounts specified in the proposal.
2. Calls `deposit` on the extension canister (the DEX adaptor).

```rust
// 1. Transfer funds from treasury to treasury manager
governance
    .approve_treasury_manager(
        extension_canister_id,
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
    )
    .await?;

// 2. Call deposit on treasury manager
let balances = governance
    .env
    .call_canister(extension_canister_id, "deposit", arg_blob)
    .await
```

There is **no slippage check, no price oracle validation, and no calm-period guard** at execution time. The only validation that occurs is `validate_deposit_operation`, which checks that the requested amounts do not exceed 50% of the current treasury balance — but this check runs at **proposal submission time**, not at execution time. [2](#0-1) 

The `treasury_manager.did` interface specification explicitly documents this known risk: [3](#0-2) 

And the proposal rendering for `RegisterExtension` includes a WARNING: [4](#0-3) 

However, this WARNING is only displayed in the proposal text for human voters — it is not enforced on-chain. The `validate_and_render_execute_extension_operation` function (used for subsequent deposit proposals) does not include this warning at all: [5](#0-4) 

---

### Impact Explanation

An attacker can sandwich the governance-executed deposit:

1. **Observe**: The attacker monitors the public SNS governance canister for an `ExecuteExtensionOperation` deposit proposal that has passed voting and is about to execute.
2. **Front-run**: The attacker manipulates the DEX price (e.g., buys large amounts of the SNS token to push the price up) immediately before the proposal executes.
3. **Trigger**: The governance canister executes the deposit at the manipulated price, providing liquidity at an unfavorable ratio.
4. **Back-run**: The attacker sells their position, extracting value from the SNS treasury.

The SNS treasury loses value proportional to the price impact of the manipulation. The `treasury_manager.did` acknowledges that "the price ratio at the time of execution may differ from the ratio at the time the proposal was approved." [3](#0-2) 

The mitigation cited ("any undeposited tokens are automatically returned to the SNS treasury account") only handles the case where the DEX cannot absorb the full deposit — it does not prevent the attacker from profiting from the price manipulation itself.

---

### Likelihood Explanation

**Medium.** The attack requires:
- Capital to manipulate the DEX price (feasible for well-funded actors targeting high-value SNS treasuries).
- Monitoring of public governance proposals (trivially achievable — all proposals are publicly visible on-chain).
- Timing the front-run to the proposal execution block (predictable, since governance execution is deterministic after the voting period ends).

The voting period (typically days) gives attackers ample time to prepare. The attack is permissionless and requires no privileged access.

---

### Recommendation

1. **Add a slippage parameter to `ExecuteExtensionOperation` deposit proposals.** The proposal should include a `min_lp_tokens_out` or equivalent minimum-output parameter that the TreasuryManager extension must enforce during `deposit`.

2. **Re-validate treasury balance and price bounds at execution time**, not only at proposal submission time. The `execute_treasury_manager_deposit` function should re-check that the current DEX price is within an acceptable range of the price at proposal submission.

3. **Enforce a calm-period check** (analogous to `_onlyCalmPeriods` in the external report) by requiring that the DEX price has not moved beyond a configurable threshold in recent blocks before executing the deposit.

4. **Expose the WARNING** about sandwich attack risk in the `validate_and_render_execute_extension_operation` rendering as well, not only in `validate_and_render_register_extension`. [6](#0-5) 

---

### Proof of Concept

**Setup**: An SNS has registered a KongSwap TreasuryManager extension and holds 1,000,000 SNS tokens and 10,000 ICP in its treasury.

**Step 1**: SNS governance votes on and passes an `ExecuteExtensionOperation` proposal to deposit 500,000 SNS + 5,000 ICP into KongSwap.

**Step 2**: The attacker observes the proposal has passed and calculates the execution block.

**Step 3**: The attacker submits a large buy order for SNS on KongSwap, pushing the SNS/ICP price up by 20%.

**Step 4**: The governance canister executes `execute_treasury_manager_deposit`:
- `approve_treasury_manager` grants allowances for 500,000 SNS + 5,000 ICP.
- `deposit` is called on the KongSwap adaptor, which deposits at the manipulated price.
- The SNS treasury receives LP tokens representing a position worth ~20% less than the deposited value at fair market price.

**Step 5**: The attacker sells their SNS position, profiting from the price impact.

The root cause is that `execute_treasury_manager_deposit` performs no price validation between the proposal approval and the actual DEX deposit call: [7](#0-6)

async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
