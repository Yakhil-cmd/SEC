### Title
Partial Neuron Transfer Silently Locks GTC Account Permanently — (`rs/nns/gtc/src/lib.rs`)

---

### Summary

`AccountState::transfer` swallows per-neuron governance errors and unconditionally returns `Ok(())`. `donate_account` then sets `has_donated = true` regardless of how many neurons actually transferred. Any neuron whose `transfer_gtc_neuron` call fails is simultaneously removed from `neuron_ids` and placed in `failed_transferred_neurons`, while the account flag permanently blocks all future recovery paths.

---

### Finding Description

**Root cause 1 — `transfer()` always returns `Ok(())`**

The loop at lines 188–207 calls `GovernanceCanister::transfer_gtc_neuron` for every neuron. On failure the error is recorded in `failed_transferred_neurons`, but the function falls through to `Ok(())` unconditionally: [1](#0-0) 

**Root cause 2 — neuron removed from `neuron_ids` before the result is checked**

Line 192 removes the neuron from the live list regardless of success or failure: [2](#0-1) 

After the loop, `neuron_ids` is empty whether transfers succeeded or not.

**Root cause 3 — `donate_account` sets `has_donated = true` unconditionally**

Because `transfer()` never returns `Err`, the `?` on line 89 never fires, and line 90 always executes: [3](#0-2) 

**Permanent lock-out**

`has_donated = true` is checked and rejected in every recovery path:
- `claim_neurons` — lines 54–56 [4](#0-3) 
- `transfer()` itself (re-entry guard) — lines 177–178 [5](#0-4) 
- `forward_whitelisted_unclaimed_accounts` — line 114 [6](#0-5) 

Neurons in `failed_transferred_neurons` are not owned by the custodian (the governance call failed), not owned by the GTC account (removed from `neuron_ids`, account locked), and not reachable by any other canister path. They are permanently orphaned.

---

### Impact Explanation

Any GTC account whose `donate_account` call coincides with a transient governance error on even one neuron suffers permanent, irrecoverable loss of the ICP staked in those neurons. The failed neurons are neither in the custodian's control nor accessible to the original owner. The account is permanently locked against all future operations.

---

### Likelihood Explanation

The governance canister is a live, upgradeable canister. Transient inter-canister call failures (reject codes, canister-busy responses, upgrade windows) are a normal operational reality on the IC. No adversarial action is required; the bug fires on ordinary transient errors. The account owner has no way to detect the partial failure before it is too late, because `donate_account` returns `Ok(())` to the caller.

---

### Recommendation

`transfer()` must propagate per-neuron failures to the caller. Two options:

1. **Fail-fast**: return `Err(...)` on the first failed `transfer_gtc_neuron` call and do not remove the neuron from `neuron_ids` until the call succeeds.
2. **Fail-aggregate**: collect all errors and return `Err(aggregate)` at the end if any neuron failed; only remove a neuron from `neuron_ids` after a confirmed successful transfer.

`donate_account` must only set `has_donated = true` after `transfer()` confirms that every neuron was successfully transferred.

---

### Proof of Concept

```
1. Create a GTC account with neuron_ids = [N1, N2, N3].
2. Configure a mock governance that succeeds for N1, fails for N2, succeeds for N3.
3. Call donate_account.
4. Observe:
   - transfer() returns Ok(()).
   - has_donated is set to true.
   - successfully_transferred_neurons = [N1, N3]
   - failed_transferred_neurons     = [N2]   ← not owned by custodian
   - neuron_ids                     = []     ← N2 removed despite failure
5. Call claim_neurons  → Err("Account has previously donated its funds")
6. Call donate_account → Err("Account has already donated its funds")
7. N2's stake is permanently inaccessible.
```
