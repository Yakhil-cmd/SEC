### Title
MintSnsTokens 7-Day Upper Bound Intentionally Disabled, Allowing Unlimited SNS Token Minting via Governance Proposals - (File: `rs/sns/governance/src/proposal.rs`)

---

### Summary

The `MintSnsTokens` governance proposal action in SNS Governance has its 7-day minting upper bound check deliberately disabled (replaced with `Decimal::MAX`), while the analogous `TransferSnsTreasuryFunds` action correctly enforces its cap. Any SNS neuron holder with sufficient voting power can submit and execute repeated `MintSnsTokens` proposals for arbitrarily large amounts, bypassing the intended treasury protection limit entirely.

---

### Finding Description

The `TokenProposalAction` trait in `rs/sns/governance/src/proposal.rs` defines `recent_amount_total_upper_bound_tokens()` to cap the 7-day total of token-related proposals. For `TransferSnsTreasuryFunds`, this is properly enforced: [1](#0-0) 

For `MintSnsTokens`, however, the real enforcement function `mint_sns_tokens_7_day_total_upper_bound_tokens` is commented out at the import level: [2](#0-1) 

And the implementation of `recent_amount_total_upper_bound_tokens` for `MintSnsTokens` unconditionally returns `Decimal::MAX` — effectively no limit: [3](#0-2) 

The validation function `treasury_valuation_if_proposal_amount_is_small_enough_or_err` calls this method and compares the proposal amount against the returned bound: [4](#0-3) 

Since `Decimal::MAX` is always greater than any proposal amount, the check never rejects a `MintSnsTokens` proposal regardless of amount or how many have already been executed in the past 7 days.

The integration test that would have verified the limit is enforced is also commented out, and the line `doomed_make_proposal_result.unwrap()` explicitly expects the second unlimited mint to **succeed**: [5](#0-4) 

The real upper bound function exists and is correct in the library crate: [6](#0-5) 

But it is never called for `MintSnsTokens` proposals.

---

### Impact Explanation

Any SNS neuron holder with sufficient voting power (or who can assemble a voting majority) can submit back-to-back `MintSnsTokens` proposals for any amount — including amounts exceeding the entire existing token supply — with no 7-day cap enforcement. This allows:

- Unlimited dilution of all existing SNS token holders.
- Draining the economic value of the SNS treasury by inflating supply without bound.
- Bypassing the treasury protection mechanism that was explicitly designed to prevent large-scale token extraction via governance.

The `TransferSnsTreasuryFunds` action is protected; `MintSnsTokens` is not, creating an asymmetric and exploitable gap.

---

### Likelihood Explanation

The entry path requires only that an attacker control or assemble a governance majority in an SNS — a realistic scenario for SNS DAOs with concentrated neuron ownership (e.g., a "whale" neuron holder as shown in the test at line 900–914). No privileged system access, key compromise, or subnet-level attack is required. The vulnerability is reachable via standard ingress governance proposal submission. [7](#0-6) 

---

### Recommendation

Uncomment the real upper bound enforcement for `MintSnsTokens` as indicated by the `TODO(NNS1-2982)` markers:

1. Uncomment the import of `mint_sns_tokens_7_day_total_upper_bound_tokens` in `rs/sns/governance/src/proposal.rs` line 52.
2. Replace the `Decimal::MAX` stub implementation of `recent_amount_total_upper_bound_tokens` for `MintSnsTokens` (lines 1035–1041) with the real implementation (lines 1025–1033).
3. Uncomment the integration test assertion in `rs/sns/integration_tests/src/sns_treasury.rs` (lines 942–965) and delete the `unwrap()` bypass at line 966.

---

### Proof of Concept

1. Deploy an SNS with a whale neuron holding majority voting power.
2. Submit a `MintSnsTokens` proposal for `u64::MAX` e8s (maximum possible).
3. Vote to pass it. Observe it executes successfully — no cap rejection.
4. Immediately submit a second `MintSnsTokens` proposal for `u64::MAX` e8s.
5. Vote to pass it. Observe it also executes — the 7-day window accumulation is tracked (`recent_amount_total_tokens` works) but the comparison against `Decimal::MAX` always passes.
6. Repeat indefinitely within the same 7-day window with no rejection.

The test at `rs/sns/integration_tests/src/sns_treasury.rs:966` already documents this: `doomed_make_proposal_result.unwrap()` — the second mint that should be rejected instead succeeds. [3](#0-2) [8](#0-7)
