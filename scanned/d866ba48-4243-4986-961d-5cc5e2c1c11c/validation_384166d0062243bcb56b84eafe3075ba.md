### Title
Ingress Induction Fee Charged Before Canister Status Check Causes Irrecoverable Cycle Loss on Stopped/Stopping Canisters - (File: rs/messaging/src/scheduling/valid_set_rule.rs)

### Summary
In `ValidSetRuleImpl::enqueue`, the ingress induction fee is permanently deducted from a canister's cycle balance **before** the canister's running status is verified. When the canister is `Stopping` or `Stopped`, the function returns an error without refunding the already-consumed cycles. The caller `induct_message` only records a `Failed` ingress status and performs no cycle refund. This creates a window — exploitable within a single block — where cycles are silently burned from a canister that never processes the message.

### Finding Description

`enqueue` in `rs/messaging/src/scheduling/valid_set_rule.rs` handles the actual induction of ingress messages into the replicated state. For messages with a fee, the logic is:

```
IngressInductionCost::Fee { payer, cost } => {
    let canister = state.canister_state_make_mut(&payer)?;

    // Step 1 — cycles permanently deducted here:
    self.cycles_account_manager.charge_ingress_induction_cost(canister, ..., cost, ...)?;

    // Step 2 — status checked AFTER deduction:
    if !ingress.is_addressed_to_subnet() {
        match canister.status() {
            CanisterStatusType::Running => {}
            CanisterStatusType::Stopping => return Err(IngressInductionError::CanisterStopping(...)),
            CanisterStatusType::Stopped  => return Err(IngressInductionError::CanisterStopped(...)),
        }
    }
    state.push_ingress(ingress)
}
``` [1](#0-0) 

`charge_ingress_induction_cost` calls `consume_with_threshold::<IngressInduction>` for non-paused canisters, which permanently decrements `cycles_balance` via `consume_cycles`. [2](#0-1) 

When `enqueue` returns `Err(CanisterStopping)` or `Err(CanisterStopped)`, the caller `induct_message` only records a `Failed` ingress status — no cycle refund is issued. [3](#0-2) 

The ingress selector (`ingress_selector.rs`) correctly checks canister status **before** cycles during block proposal, so messages to already-stopped canisters are normally excluded from blocks. [4](#0-3) 

However, a single block can contain both a `stop_canister` management call targeting canister X and one or more ingress messages addressed to canister X. Subnet management messages are processed before ingress messages within the same round. After `stop_canister` transitions canister X to `Stopping`, the ingress messages for X reach `enqueue`, which charges their induction fees and then immediately returns `CanisterStopping` — the cycles are gone.

### Impact Explanation

Every ingress message that reaches `enqueue` while the target canister is `Stopping` or `Stopped` causes a permanent, unrecoverable cycle deduction from that canister's balance. The deducted amount equals the ingress induction fee (proportional to message size). No refund path exists in `induct_message`. [5](#0-4) 

The impact class is **cycles/resource accounting bug**: a canister's cycle balance is reduced without any corresponding service being rendered, and the reduction is irreversible. For a canister with many pending ingress messages at the moment it is stopped, the aggregate cycle loss can be significant.

### Likelihood Explanation

The trigger condition — a block containing both a `stop_canister` call and ingress messages for the same canister — is a normal operational scenario. Any canister controller who issues `stop_canister` while users are actively sending messages to that canister will reproduce this. No adversarial coordination is required; it occurs naturally during routine canister lifecycle management. The ingress selector's status pre-check does not protect against this because it runs against the state **before** the `stop_canister` call in the same block is applied.

### Recommendation

In `enqueue`, move the canister status check to **before** `charge_ingress_induction_cost`. The corrected order should be:

1. Verify `canister.status() == Running`; return `CanisterStopping`/`CanisterStopped` immediately if not.
2. Only then call `charge_ingress_induction_cost`.

This mirrors the ordering already used correctly in `ingress_selector.rs` (status check at lines 536–548, cycles check at lines 556–584). [6](#0-5) 

### Proof of Concept

1. Deploy canister X on an application subnet with sufficient cycles.
2. Construct a block containing:
   - A `stop_canister` management ingress for canister X (processed first as a subnet message).
   - N ingress messages addressed to canister X (processed second via `induct_messages`).
3. Execute the block. After execution:
   - Canister X is in `Stopping` state.
   - Each of the N ingress messages has status `Failed(CanisterStopping)`.
   - Canister X's cycle balance has been reduced by `N × ingress_induction_cost` with no corresponding work performed and no refund issued.

The existing test `induct_message_fails_for_stopping_canister` confirms the `CanisterStopping` error path is reachable but does not assert that cycles are **not** deducted, leaving the cycle-loss side-effect undetected. [7](#0-6)

fn induct_message_fails_for_stopping_canister() {
    with_test_replica_logger(|log| {

            .canister_id(canister_id)

        let msg_id = msg.id();
        let mut ingress_history_writer = MockIngressHistory::new();
        ingress_history_writer
            .expect_set_status()
            .with(
                always(),
                eq(msg.id()),
                eq(IngressStatus::Known {
                    receiver: canister_id.get(),
                    user_id: user_test_id(2),
                    time: UNIX_EPOCH,
                    state: IngressState::Failed(UserError::new(
                        ErrorCode::CanisterStopping,
                        format!("Canister {canister_id} is stopping"),
                    )),
                }),
                always(),
            )
            .times(1)
            .returning(move |state, _, status, _| {
                state.set_ingress_status(msg_id.clone(), status, NumBytes::from(u64::MAX), |_| {});
                IngressStatus::Unknown.into()
            });
        let ingress_history_writer = Arc::new(ingress_history_writer);
        let metrics_registry = MetricsRegistry::new();
        let valid_set_rule = ValidSetRuleImpl::new(
            ingress_history_writer,
            Arc::new(CyclesAccountManagerBuilder::new().build()),
            &metrics_registry,
            log,
        );

        let mut state = ReplicatedState::new(subnet_test_id(1), SubnetType::Application);
        state.put_canister_state(get_stopping_canister(canister_id));

        valid_set_rule.induct_message(&mut state, signed_ingress, ExecutionRound::from(0));
        assert_eq!(ingress_queue_size(&state, canister_id), 0);
        assert_inducted_ingress_messages_eq(
            metric_vec(&[(&[(LABEL_STATUS, LABEL_VALUE_CANISTER_STOPPING)], 1)]),
            &metrics_registry,
        );
        assert_eq!(
            0,
            fetch_inducted_payload_size_stats(&metrics_registry).count
        );
    });
