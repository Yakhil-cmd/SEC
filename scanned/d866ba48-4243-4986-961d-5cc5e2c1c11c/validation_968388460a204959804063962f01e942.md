### Title
Unbounded Per-Peer Slot Table Growth via `SLOT_TABLE_NO_LIMIT` on Certification/DKG/IDKG/HTTPS-Outcalls Channels Enables OOM Crash — (`rs/replica/setup_ic_network/src/lib.rs`)

### Summary

A single Byzantine subnet peer (below the BFT fault threshold) can exhaust heap memory on any victim replica by advertising an unbounded stream of distinct `SlotUpdate` messages on the certification, DKG, IDKG, HTTPS-outcalls, or consensus P2P channels. These channels are all wired with `SLOT_TABLE_NO_LIMIT = usize::MAX`, making the per-peer slot table guard a no-op. Each distinct slot+ID pair grows `slot_table`, `active_assembles`, and `artifact_processor_tasks` without bound.

### Finding Description

In `rs/replica/setup_ic_network/src/lib.rs`, two constants are defined: [1](#0-0) 

```
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
```

The comment explicitly acknowledges the ingress limit exists to protect against malicious peers. However, every other artifact channel is wired with `SLOT_TABLE_NO_LIMIT`: [2](#0-1) 

- **Certification**: `abortable_broadcast_channel(assembler, SLOT_TABLE_NO_LIMIT)` (line 268)
- **DKG**: `abortable_broadcast_channel(assembler, SLOT_TABLE_NO_LIMIT)` (line 279)
- **IDKG**: `abortable_broadcast_channel(assembler, SLOT_TABLE_NO_LIMIT)` (line 291)
- **HTTPS-outcalls**: `abortable_broadcast_channel(assembler, SLOT_TABLE_NO_LIMIT)` (line 303)
- **Consensus**: `abortable_broadcast_channel(assembler, SLOT_TABLE_NO_LIMIT)` (line 237/246)

In `ConsensusManagerReceiver::handle_slot_update_receive`, the guard is: [3](#0-2) 

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => {
    empty_slot.insert(new_slot_entry);
    ...
    (true, None)
}
Entry::Vacant(_) => {
    // slot_limit exceeded — drop
    (false, None)
}
```

When `self.slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice (memory is exhausted long before `usize::MAX` entries are inserted). Every new `(slot_number, id)` pair from a peer:

1. Inserts into `self.slot_table[peer_id]` (unbounded per-peer HashMap)
2. If the artifact ID is new, inserts into `self.active_assembles` and spawns a new Tokio task in `self.artifact_processor_tasks` [4](#0-3) 

The three data structures that grow without bound:

- `slot_table: HashMap<NodeId, HashMap<SlotNumber, SlotEntry<...>>>` — inner map grows per peer
- `active_assembles: HashMap<WireArtifact::Id, watch::Sender<PeerCounter>>` — grows with unique IDs
- `artifact_processor_tasks: JoinSet<...>` — one Tokio task per unique ID

### Impact Explanation

A Byzantine peer sends `N` messages of the form `SlotUpdate { slot_number: i, commit_id: i, update: Id(unique_id_i) }` for `i` in `0..N` on any of the unlimited channels. Each message is accepted, grows all three data structures, and spawns a new async task. At sufficient `N`, the replica process is OOM-killed. A single Byzantine node can crash any number of victim replicas sequentially or simultaneously. If `f` replicas crash (where `f` is the fault threshold), subnet liveness is lost.

### Likelihood Explanation

The attack requires TLS-authenticated access as a subnet node — a Byzantine peer below the fault threshold. This is an explicitly in-scope threat model for IC's BFT protocol. The attack is mechanical: send a tight loop of distinct slot updates. No cryptographic material needs to be forged. The asymmetry is already documented in the code (ingress is protected; others are not), confirming the developers were aware of the concern but did not extend the protection.

### Recommendation

Apply a per-peer slot table limit to all artifact channels, not only ingress. The appropriate limit for each channel type should reflect the maximum number of in-flight artifacts a legitimate peer would ever advertise simultaneously (e.g., bounded by consensus round size, DKG epoch size, etc.). The existing `SLOT_TABLE_LIMIT_INGRESS` pattern is the correct fix — define analogous constants for each channel type and pass them instead of `SLOT_TABLE_NO_LIMIT`.

### Proof of Concept

```
1. Byzantine peer P authenticates via TLS to victim replica V.
2. P sends N=10^6 HTTP POST requests to /{certificationmessage,dkg,idkg,canisterhttpresponse}/update
   each with a distinct SlotUpdate { slot_number: i, commit_id: i, update: Id(unique_id_i) }.
3. ConsensusManagerReceiver::handle_slot_update_receive processes each:
   - slot_table[P] grows to N entries (no limit check fires)
   - active_assembles grows to N entries
   - artifact_processor_tasks grows to N Tokio tasks
4. Replica V's RSS grows proportionally to N until OOM kill.
5. Differential: repeat with ingress channel → capped at 50,000 entries, no OOM.
```

The exact code path is confirmed by the existing test `slot_table_limit_exceeded` in `rs/p2p/consensus_manager/src/receiver.rs` (line 847), which demonstrates the guard works correctly when a finite limit is set — but that test uses `slot_limit: 2` and is never exercised with `usize::MAX` for the non-ingress channels. [5](#0-4)
