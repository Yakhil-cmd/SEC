### Title
No-Slippage-Parameter Treasury Manager Deposit Exposes SNS Funds to Front-Running - (File: rs/sns/treasury_manager/src/lib.rs, rs/sns/governance/src/extensions.rs)

### Summary
The SNS Treasury Manager deposit flow, executed via `ExecuteExtensionOperation` governance proposals, does not include any slippage protection parameters in the `DepositRequest` type or in the `execute_treasury_manager_deposit` execution path. An unprivileged DEX participant who observes a pending deposit proposal can manipulate the DEX pool price between proposal approval and execution, causing the SNS treasury to deposit at an unfavorable asset ratio and suffer a financial loss. The codebase itself acknowledges this risk in two separate locations.

### Finding Description

The `TreasuryManager` trait's `deposit` function accepts a `DepositRequest` that contains only `allowances` — token amounts and owner accounts — with no `min_amount_out`, `min_lp_tokens`, or any other slippage bound:

```rust
pub struct DepositRequest {
    pub allowances: Vec<Allowance>,
}
``` [1](#0-0) 

The `execute_treasury_manager_deposit` function in SNS Governance approves the treasury manager for the exact token amounts specified in the proposal, then calls `deposit` on the extension canister with no slippage guard:

```rust
// 1. Transfer funds from treasury to treasury manager
governance.approve_treasury_manager(extension_canister_id, treasury_allocation_sns_e8s, treasury_allocation_icp_e8s).await?;

// 2. Call deposit on treasury manager
let balances = governance.env.call_canister(extension_canister_id, "deposit", arg_blob).await ...
``` [2](#0-1) 

The proposal validation step (`validate_deposit_operation_impl`) only checks that the requested amounts do not exceed 50% of the current treasury balance. It does not capture any price ratio, minimum LP token output, or slippage bound at proposal-submission time: [3](#0-2) 

The codebase explicitly acknowledges this gap in two places. In the Treasury Manager interface specification:

> "Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved." [4](#0-3) 

And in the rendered proposal warning for `RegisterExtension`:

> "Some Decentralized Exchanges lack slippage protection during deposits. Consequently, deposited asset ratios may deviate from those specified in the proposal. This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running or sandwich attacks." [5](#0-4) 

### Impact Explanation

An SNS treasury deposit proposal specifies fixed token amounts (`treasury_allocation_sns_e8s`, `treasury_allocation_icp_e8s`) but no price ratio constraint. Between the time the proposal is adopted by governance and the time it is executed on-chain, an attacker can:

1. Observe the pending `ExecuteExtensionOperation` deposit proposal in the SNS governance canister.
2. Manipulate the target DEX pool price by making large trades (e.g., buying up one side of the pool).
3. Allow the proposal to execute — the SNS treasury deposits at the now-skewed ratio, receiving far fewer LP tokens than expected.
4. Reverse the trade, extracting profit at the SNS treasury's expense.

The financial loss falls directly on the SNS DAO treasury. Because the `DepositRequest` API carries no slippage parameters, no Treasury Manager implementation can enforce a minimum output even if it wanted to, unless it independently queries the DEX price — which itself is manipulable at execution time (the same root cause as the external report's `quoteExactInputSingle` pattern).

### Likelihood Explanation

SNS governance proposals have a mandatory voting period (days), giving any DEX participant ample time to observe and prepare a sandwich. The attack requires no privileged access: any principal with enough tokens to move the DEX pool price can execute it. The IC's deterministic execution model means the attacker can predict the exact block in which the proposal executes and front-run it precisely. The codebase's own warnings confirm the developers consider this a realistic scenario.

### Recommendation

1. Extend `DepositRequest` (and the `TreasuryManager` trait's `deposit` signature) to include optional slippage parameters such as `min_lp_tokens_out` or `min_asset_ratio`.
2. Require that `ExecuteExtensionOperation` deposit proposals include a `min_amount_out` field that is validated and forwarded to the Treasury Manager at execution time.
3. Alternatively, enforce a maximum time-to-live between proposal adoption and execution for treasury deposit proposals, and require the Treasury Manager implementation to re-query and validate the current pool price against a governance-approved bound immediately before depositing.

### Proof of Concept

1. An SNS DAO submits a proposal: `ExecuteExtensionOperation { operation_name: "deposit", operation_arg: { treasury_allocation_sns_e8s: 1_000_000_000, treasury_allocation_icp_e8s: 500_000_000 } }`.
2. The proposal enters the voting period. An attacker observes it.
3. The attacker buys a large amount of ICP from the SNS/ICP DEX pool, skewing the price so that ICP is now overpriced relative to SNS tokens.
4. The proposal passes and `execute_treasury_manager_deposit` is called. It calls `approve_treasury_manager` with the fixed amounts, then calls `deposit` on the Treasury Manager extension canister with no slippage bound.
5. The Treasury Manager deposits into the DEX at the manipulated price, receiving far fewer LP tokens than the fair-price equivalent.
6. The attacker sells their ICP back, restoring the price and pocketing the difference — a direct loss from the SNS treasury.

The root cause is in `rs/sns/treasury_manager/src/lib.rs` (`DepositRequest` has no slippage field) and `rs/sns/governance/src/extensions.rs` (`execute_treasury_manager_deposit` passes no slippage bound to the extension canister). [1](#0-0) [6](#0-5) [3](#0-2) [4](#0-3)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
