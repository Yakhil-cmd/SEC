### Title
Arithmetic Overflow in `disburse_maturity` Causes SNS Governance Canister Trap - (File: rs/sns/governance/src/governance.rs)

### Summary
In `rs/sns/governance/src/governance.rs`, the `disburse_maturity` function multiplies `maturity_e8s_equivalent` (a `u64`) by `percentage_to_disburse` (cast to `u64`, max 100) using `u64::checked_mul`, then calls `.expect("Overflow while processing maturity to disburse.")`. When `maturity_e8s_equivalent > u64::MAX / percentage_to_disburse`, the multiplication overflows, `checked_mul` returns `None`, and `.expect()` panics, trapping the canister update call. The NNS governance equivalent correctly casts to `u128` before multiplying.

### Finding Description
In `rs/sns/governance/src/governance.rs` at lines 1643–1649:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)
    .expect("Overflow while processing maturity to disburse.")
    .checked_div(100)
    .expect("Error when processing maturity to disburse.")
    as u128;
```

`maturity_e8s_equivalent` is a `u64`. `percentage_to_disburse` is validated to be in `[1, 100]`. The multiplication `u64 * u64` overflows when `maturity_e8s_equivalent > u64::MAX / percentage_to_disburse`. For `percentage_to_disburse = 100`, the threshold is `u64::MAX / 100 ≈ 1.84 × 10¹⁷` e8s (≈ 1.84 billion tokens with 8 decimals). The `.expect()` call panics on overflow, trapping the update call.

By contrast, the NNS governance implementation in `rs/nns/governance/src/governance/disburse_maturity.rs` at lines 229–231 correctly widens to `u128` before multiplying:

```rust
(total_maturity_e8s as u128)
    .checked_mul(percentage_to_disburse as u128)
    .and_then(|result| result.checked_div(100))
```

### Impact Explanation
Any SNS neuron holder whose `maturity_e8s_equivalent` exceeds `u64::MAX / percentage_to_disburse` cannot disburse maturity — the update call traps and is rejected. For SNS tokens with large supplies (e.g., a token with a 10-trillion-unit supply where a whale neuron accumulates billions of tokens in maturity), this threshold is reachable. The user is permanently unable to disburse maturity via any `percentage_to_disburse` value that causes overflow, effectively locking their maturity rewards. The canister state is rolled back on trap, so there is no state corruption, but the operation is permanently denied for affected neurons.

### Likelihood Explanation
SNS tokens are permissionlessly deployable with arbitrary supplies and decimal configurations. A token with a large supply (e.g., 10¹² tokens at 8 decimals = 10²⁰ e8s total) can have neurons accumulating maturity well above the `~1.84 × 10¹⁷` e8s threshold. An unprivileged neuron holder with `DisburseMaturity` permission on their own neuron can trigger this by simply calling `disburse_maturity` — no special privileges required. The entry path is a standard governance update call.

### Recommendation
Cast `maturity_e8s_equivalent` to `u128` before multiplying, mirroring the NNS governance implementation:

```rust
let maturity_to_deduct = (neuron.maturity_e8s_equivalent as u128)
    .checked_mul(disburse_maturity.percentage_to_disburse as u128)
    .and_then(|r| r.checked_div(100))
    .and_then(|r| u64::try_from(r).ok())
    .ok_or_else(|| GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Overflow while processing maturity to disburse.",
    ))?;
```

This eliminates the overflow and replaces the panic with a graceful error return.

### Proof of Concept
1. Deploy an SNS with a token supply of 10¹² tokens (8 decimals).
2. Create a neuron and accumulate `maturity_e8s_equivalent = 2 × 10¹⁷` (achievable with a large stake over time, or via direct state manipulation in a test environment).
3. Call `disburse_maturity` with `percentage_to_disburse = 100`.
