### Title
Missing Duplicate `callback_id` Check in `validate_self_validating_payload_impl` Enables Double-Delivery of Bitcoin Adapter Responses â€” (`rs/bitcoin/consensus/src/payload_builder.rs`)

### Summary
`validate_self_validating_payload_impl` performs no content validation beyond an emptiness check. A Byzantine block proposer can include two `BitcoinAdapterResponse` entries with the same `callback_id` in a `SelfValidatingPayload`. This passes all validation, gets finalized, and causes `push_response` to enqueue two `ConsensusResponse` entries for the same callback into `consensus_queue`, violating the one-response-per-request invariant.

### Finding Description
`validate_self_validating_payload_impl` only checks `if *payload == SelfValidatingPayload::default()` and returns `Ok(size)` for any non-empty payload. [4](#0-3) 

`validate_payload` computes `delivered_ids` from past payloads but never uses it in the validation call, and performs no intra-payload deduplication. [5](#0-4) 

`push_response` for `GetSuccessorsResponse` uses `get_mut` rather than `remove`, leaving the context in the map after the first response, so a second call with the same `callback_id` succeeds and pushes a second entry. [6](#0-5) 

For `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject`, there is no lookup at all â€” both duplicate entries push unconditionally. [7](#0-6) 

### Impact Explanation
Two `ConsensusResponse` entries for the same `callback_id` are placed in `consensus_queue`. The execution environment processes both, delivering two responses to the bitcoin system canister for a single request. In the context of ckBTC/ckDOGE, this could cause the canister to process the same block data twice, potentially corrupting chain-fusion state or enabling double-crediting of wrapped assets.

### Likelihood Explanation
Requires a single Byzantine block proposer â€” one compromised subnet node acting within its normal role. No subnet majority, no key compromise, no external dependency. The crafted payload is deterministically accepted by all honest validators because the validation gap exists on every node.

### Recommendation
1. In `validate_self_validating_payload_impl` (or `validate_payload`), collect all `callback_id` values from the payload into a `HashSet` and return `InvalidArtifact` if any duplicate is detected.
2. Cross-check each `callback_id` against the in-flight requests in the replicated state (as `get_self_validating_payload_impl` already does via `past_callback_ids`).
3. In `push_response` for `GetSuccessorsResponse`, use `remove` instead of `get_mut` so the context is consumed on first delivery, making a second call return `BitcoinNonMatchingResponse`.

pub fn push_response(
    state: &mut ReplicatedState,
    response: BitcoinAdapterResponse,
) -> Result<(), StateError> {
    match response.response {
        BitcoinAdapterResponseWrapper::GetSuccessorsResponse(r) => {
            // Received a response to a request from the bitcoin wasm canister.
            // Retrieve the associated request.
            let callback_id = CallbackId::from(response.callback_id);
            let context = state
                .metadata
                .subnet_call_context_manager
                .bitcoin_get_successors_contexts
                .get_mut(&callback_id)
                .ok_or_else(|| StateError::BitcoinNonMatchingResponse {
                    callback_id: callback_id.get(),
                })?;

            let payload = match maybe_split_response(r) {
                Ok((initial_response, follow_ups)) => {
                    // Store the follow-ups for later (overwrites previous ones).
                    state
                        .metadata
                        .bitcoin_get_successors_follow_up_responses
                        .insert(context.request.sender(), follow_ups);

                    Payload::Data(initial_response.encode())
                }
                Err(err) => Payload::Reject(RejectContext::new(
                    RejectCode::CanisterError,
                    format!("Received invalid response from adapter: {err:?}"),
                )),
            };

            // Add response to the consensus queue.
            state
                .consensus_queue
                .push(ConsensusResponse::new(callback_id, payload));

            Ok(())
        }
        BitcoinAdapterResponseWrapper::SendTransactionResponse(_) => {
            // Retrieve the associated request from the call context manager.
            let callback_id = CallbackId::from(response.callback_id);
            // The response to a `send_transaction` call is always the empty blob.
            let payload = Payload::Data(EmptyBlob.encode());

            // Add response to the consensus queue.
            state
                .consensus_queue
                .push(ConsensusResponse::new(callback_id, payload));

            Ok(())
        }
        BitcoinAdapterResponseWrapper::GetSuccessorsReject(reject) => {
            // Retrieve the associated request from the call context manager.
            let callback_id = CallbackId::from(response.callback_id);
            let reject_payload =
                Payload::Reject(RejectContext::new(reject.reject_code, reject.message));

            // Add response to the consensus queue.
            state
                .consensus_queue
                .push(ConsensusResponse::new(callback_id, reject_payload));

            Ok(())
        }
        BitcoinAdapterResponseWrapper::SendTransactionReject(reject) => {
            // Retrieve the associated request from the call context manager.
            let callback_id = CallbackId::from(response.callback_id);
            let reject_payload =
                Payload::Reject(RejectContext::new(reject.reject_code, reject.message));

            // Add response to the consensus queue.
            state
                .consensus_queue
                .push(ConsensusResponse::new(callback_id, reject_payload));

            Ok(())
        }
    }
