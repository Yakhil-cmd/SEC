### Title
Attacker-Induced Single-Block Log Overflow Causes Permanent Deposit Loss in ckETH/ckERC20 Minter — (`rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

The ckETH minter's `scrape_block_range` function permanently skips any Ethereum block whose deposit-event log response exceeds the 2 MB HTTP-outcall size ceiling. Once skipped, the block is recorded in `SkippedBlockForContract` and `last_scraped_block_number` is advanced past it — with no recovery path. An unprivileged Ethereum user who floods the helper contract with enough deposit calls in a single block can cause every legitimate deposit in that block to be permanently unprocessed, resulting in irreversible loss of user funds.

---

### Finding Description

`scrape_block_range` uses the EVM-RPC canister's `get_logs` with an initial response-size estimate of `ETH_GET_LOGS_INITIAL_RESPONSE_SIZE_ESTIMATE + HEADER_SIZE_LIMIT = 100 + 2048 = 2148 bytes`. [1](#0-0) 

The `DoubleCycles` retry strategy doubles the estimate on each retry, capping at **2,000,000 bytes (2 MB)**. [2](#0-1) 

When the response for a single block still exceeds 2 MB after all retries, the code reaches the terminal branch:

```rust
if from_block == to_block {
    mutate_state(|s| {
        process_event(s, EventType::SkippedBlockForContract {
            contract_address,
            block_number: to_block,
        });
    });
    mutate_state(|s| S::update_last_scraped_block_number(s, to_block));
``` [3](#0-2) 

`update_last_scraped_block_number` advances the scraping cursor past the block, and `record_skipped_block_for_contract` inserts it into the `skipped_blocks` set with an `assert!` that prevents re-insertion — making the skip permanent and irrecoverable. [4](#0-3) 

The only active scraping path is `ReceivedEthOrErc20LogScraping` (`LogScrapingId::EthOrErc20DepositWithSubaccount`); the other two IDs are `Deprecated`. [5](#0-4) 

This scraping covers **both** ETH and ERC-20 deposits through a single helper contract address, so a flood of either token type in one block triggers the skip for all deposits in that block. [6](#0-5) 

---

### Impact Explanation

Every deposit event emitted by the helper contract in the skipped block is permanently ignored. The depositors' ETH or ERC-20 tokens are locked in the Ethereum helper contract with no corresponding ckETH/ckERC20 minted on the IC. There is no admin endpoint, governance proposal path, or state-machine mechanism to force re-scraping of a block already recorded in `skipped_blocks`. The `assert!` in `record_skipped_block_for_contract` would even panic the canister if a re-skip were attempted. [7](#0-6) 

**Impact class**: chain-fusion mint/burn bug — permanent, unrecoverable loss of user funds for all depositors whose transactions landed in the targeted block.

---

### Likelihood Explanation

Each `ReceivedEthOrErc20WithSubaccount` log entry is approximately 600 bytes (per the inline test comment). Exceeding 2 MB therefore requires roughly **3,334 matching log entries in a single Ethereum block**. [2](#0-1) 

An Ethereum block's gas limit (~30 M gas) bounds the number of independent transactions, but a single transaction can emit many events via a batch-deposit contract or a contract that loops and calls the helper. An attacker willing to spend the gas can concentrate thousands of events in one block. The attacker's own deposits are also lost, making this a griefing/vandalism attack rather than a profitable one, which lowers likelihood — but the permanent, irreversible nature of the impact and the absence of any on-chain or off-chain recovery path make it a meaningful risk for a high-value bridge.

---

### Recommendation

1. **Remove the permanent-skip branch.** Instead of advancing `last_scraped_block_number` past an oversized single block, return an error and let the timer retry. Introduce a per-block retry counter; only skip after N consecutive failures, and emit an alert.
2. **Add a governance-controlled re-scrape endpoint** that allows the NNS or minter controller to reset `last_scraped_block_number` to a prior value so a previously skipped block can be re-attempted after the root cause is resolved.
3. **Remove the `assert!` in `record_skipped_block_for_contract`** (or replace it with a no-op) so that a re-scrape attempt does not panic the canister.
4. **Raise the 2 MB ceiling** or make it configurable via upgrade argument, so that a temporary spike in log volume does not immediately trigger the skip path.

---

### Proof of Concept

The existing test `should_skip_single_block_containing_too_many_events` already demonstrates the exact mechanism: 3,500 log entries (~2.1 MB) in a single block cause the minter to emit `SkippedBlock` and advance past the block, with no further attempt to process those deposits. [8](#0-7) 

An attacker replicates this on mainnet by submitting a batch transaction (or a sequence of transactions within one block) that calls the `ReceivedEthOrErc20WithSubaccount` helper contract enough times to produce ≥ 3,334 log entries. The next scraping cycle of the ckETH minter will:

1. Attempt `get_logs` for the block range containing the flooded block.
2. Receive a `LimitExceeded` HTTP-outcall error at every doubling of the response-size estimate up to 2 MB.
3. Bisect down to the single flooded block (`from_block == to_block`).
4. Emit `SkippedBlockForContract` and advance `last_scraped_block_number` past it — permanently discarding all deposits in that block. [9](#0-8)

                .collect::<Vec<_>>()

        );

            contract_address,
            last_scraped_block_number,
