### Title
ICRC-2 Approve Deduplication Bypass via `fee=None` vs `fee=Some(F)` — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary
The ICRC-1 ledger's `icrc2_approve_not_async` preserves the `None`/`Some` distinction of the caller-supplied `fee` field into the stored `Operation::Approve`. Because the transaction hash is computed over this field (and `None` serializes to an absent CBOR key while `Some(F)` serializes to a present key), two semantically identical approve operations with the same `created_at_time` but differing only in `fee=None` vs `fee=Some(F)` produce distinct hashes and both pass deduplication, each debiting the full fee.

### Finding Description
In `icrc2_approve_not_async`:

```rust
fee: arg.fee.map(|_| expected_fee_tokens),  // None stays None
```

The `fee` field in `Operation::Approve` is `Option<Tokens>`. `FlattenedTransaction` serializes it with `#[serde(skip_serializing_if = "Option::is_none")]`, so `None` produces a shorter CBOR map (no `fee` key) while `Some(F)` includes the `fee` key. The representation-independent hash over these two CBOR maps differs. The deduplication store (`transactions_by_hash`) therefore sees two distinct hashes and admits both transactions within the same deduplication window.

### Impact Explanation
An approver who submits TX1 (`fee=None`) and TX2 (`fee=Some(F)`) with the same `created_at_time` and all other fields identical will have their balance debited by `2*F` instead of `F`. The allowance is set twice (idempotent in final value but the double fee is the real harm). This is most likely to occur in retry scenarios: a client submits with `fee=None`, receives no response, then retries with `fee=Some(F)` believing it is a safe retry — both succeed.

### Likelihood Explanation
Any ICRC-1 ledger client that retries approve calls and switches between omitting and including the explicit fee (a common pattern in SDK retry logic) will trigger this. The attacker is the approver themselves, so no third-party authorization is needed. The precondition is simply having sufficient balance for two fees.

### Recommendation
Normalize the `fee` field before constructing the `Transaction`. Replace:
```rust
fee: arg.fee.map(|_| expected_fee_tokens),
```
with:
```rust
fee: Some(expected_fee_tokens),
```
This ensures both `fee=None` and `fee=Some(F)` produce the same `Operation::Approve` struct and thus the same transaction hash, restoring the deduplication invariant. This matches how the ICP ledger handles it — it always stores `fee: expected_fee` unconditionally. [6](#0-5) 

### Proof of Concept
```rust
// Both calls use the same created_at_time T, same from/spender/amount/memo/expires_at
let args_no_fee = ApproveArgs { fee: None, created_at_time: Some(T), ..base };
let args_with_fee = ApproveArgs { fee: Some(F), created_at_time: Some(T), ..base };

let r1 = icrc2_approve_not_async(caller, args_no_fee);   // Ok(block_0)
let r2 = icrc2_approve_not_async(caller, args_with_fee); // Ok(block_1) — NOT Duplicate

// Assert both succeeded and balance decreased by 2*F
assert!(r1.is_ok() && r2.is_ok());
assert_eq!(balance_of(caller), initial_balance - 2 * F);
```
