### Title
Stable Memory Dirty Pages Excluded from DTS Yield Threshold — Page Copy Monopolizes Execution Round Without Yielding - (`rs/embedders/src/wasm_executor.rs`)

### Summary
The dirty-page yield optimization in `process()` checks only `res.wasm_dirty_pages.len()` to decide whether to yield before copying dirty pages. Stable memory dirty pages (`res.stable_memory_dirty_pages`) are never counted. A canister that writes exclusively to stable memory can dirty up to 2 GiB (the `STABLE_MEMORY_DIRTY_PAGE_LIMIT_MESSAGE` cap) without triggering the yield, causing all stable page copying to occur inline in a single execution slice and monopolizing the round for other canisters.

### Finding Description

**Root cause — line 682:**

```rust
let dirty_pages = NumOsPages::from(res.wasm_dirty_pages.len() as u64);
```

`res` is `InstanceRunResult`, which carries both `wasm_dirty_pages: Vec<PageIndex>` and `stable_memory_dirty_pages: Vec<PageIndex>`. Only the Wasm heap field is read here. [1](#0-0) 

The yield threshold is `DEFAULT_MAX_DIRTY_PAGES_WITHOUT_OPTIMIZATION` = 1 GiB / 4 KiB = **262,144 pages**. [2](#0-1) 

The stable memory dirty page limit for regular messages is `STABLE_MEMORY_DIRTY_PAGE_LIMIT_MESSAGE` = 2 GiB / 4 KiB = **524,288 pages** — enforced inline by the instrumented bytemap counter, which traps with `MemoryWriteLimitExceeded` only when this higher limit is exceeded. [3](#0-2) 

When a canister writes exclusively to stable memory:
- `res.wasm_dirty_pages.len() == 0` → `dirty_pages == 0` → yield condition is false
- `yield_for_dirty_memory_copy()` is never called
- Execution falls through to the stable page copy at lines 757–761 in the same slice [4](#0-3) 

The `compute_page_delta` call for stable memory copies all dirty pages from the Wasm instance into the `PageMap` synchronously, with no round boundary. [5](#0-4) 

Additionally, the `dirty_page_copy_overhead` (3,000 instructions/page) is only charged when `num_dirty_pages > 0`, which only happens when the yield IS triggered. For the stable-memory-only path, `num_dirty_pages` stays 0 and no overhead is charged, so the page copy work is entirely unaccounted in the instruction budget. [6](#0-5) 

**How the yield mechanism works (confirmed):** `DeterministicTimeSlicingHandler::yield_for_dirty_memory_copy()` calls `try_yield_for_dirty_memory_copy()` (transitions to `Paused`), fires the `pause_callback` to notify the replica, then blocks on `wait_for_resume_or_abort()`. The replica schedules a new "dirty page copy" slice for the next round, resumes the thread, and only then does `process()` continue to the page copy code. This inter-round pause is what the stable memory path bypasses entirely. [7](#0-6) [8](#0-7) 

### Impact Explanation
A single unprivileged canister can dirty up to 2 GiB of stable memory in one update message. The resulting page copy (up to 524,288 × 4 KiB = 2 GiB of memory reads/writes) executes inline in the same execution slice without yielding, without instruction overhead, and without any round boundary. This extends the wall-clock duration of that execution round, starving all other canisters on the subnet from being scheduled during that time. The existing test `yield_for_dirty_pages_copy_works` demonstrates the correct behavior for Wasm heap writes; the equivalent stable memory path has no such protection. [9](#0-8) 

### Likelihood Explanation
The attack requires only deploying a canister and sending a single update message that calls `stable64_write` across a range exceeding 1 GiB. No privileged access, governance majority, or key material is needed. The stable memory dirty page limit (2 GiB) is enforced by the bytemap counter, so the canister cannot exceed that bound, but the gap between the yield threshold (1 GiB) and the stable limit (2 GiB) is exactly the exploitable window. [10](#0-9) 

### Recommendation
At line 682, include `res.stable_memory_dirty_pages.len()` in the dirty page count used for the yield threshold check:

```rust
let dirty_pages = NumOsPages::from(
    (res.wasm_dirty_pages.len() + res.stable_memory_dirty_pages.len()) as u64
);
```

This ensures that a canister writing exclusively or predominantly to stable memory triggers the same yield-and-resume behavior as one writing to the Wasm heap, bounding the page copy work per round to `max_dirty_pages_without_optimization` pages regardless of which memory type was dirtied. [11](#0-10)
