### Title
Delegated `ManagePrincipals` Holder Can Strip Original Neuron Owner's Permissions — (`File: rs/sns/governance/src/governance.rs`)

---

### Summary

In the SNS governance canister, any principal granted `ManagePrincipals` permission on a neuron can call `RemoveNeuronPermissions` to strip **any other principal's** permissions from that neuron — including the original owner's. There is no check that prevents a delegated `ManagePrincipals` holder from removing the neuron owner's own `ManagePrincipals` (or all permissions), permanently locking the owner out of their neuron. This is the direct IC analog of the SKALE "node can unlink validator" bug.

---

### Finding Description

The SNS governance `remove_neuron_permissions` function authorizes the operation solely by checking whether the caller holds `ManagePrincipals` (or `ManageVotingPermission` for voting-only changes). It does not verify whether the caller is the original neuron owner, nor does it prevent the caller from targeting the original owner as the subject of the removal.

The authorization gate is `check_principal_authorized_to_change_permissions`:

```rust
// rs/sns/governance/src/neuron.rs:144-164
pub(crate) fn check_principal_authorized_to_change_permissions(
    &self,
    caller: &PrincipalId,
    permissions_to_change: NeuronPermissionList,
) -> Result<(), GovernanceError> {
    let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
        vec![ManagePrincipals, ManageVotingPermission]
    } else {
        vec![ManagePrincipals]
    };
    let caller_authorized = sufficient_permissions
        .iter()
        .any(|p| self.is_authorized(caller, *p));
    ...
}
```

This check is called in `remove_neuron_permissions` before executing the removal:

```rust
// rs/sns/governance/src/governance.rs:4694-4697
neuron.check_principal_authorized_to_change_permissions(
    caller,
    permissions_to_remove.clone(),
)?;
```

The `principal_id` field in `RemoveNeuronPermissions` — the **target** of the removal — is never compared against the original neuron owner. Any caller with `ManagePrincipals` can set `principal_id` to the original owner and remove all of the owner's permissions.

The code itself acknowledges the danger in the docstring:

```rust
// rs/sns/governance/src/governance.rs:4648-4650
/// This is a dangerous operation as it is possible to remove all permissions for a neuron
/// and no longer be able to modify its state, i.e. disbursing the neuron back into the
/// governance token.
```

But no guard is implemented to prevent a delegated principal from exploiting this against the original owner.

---

### Impact Explanation

A malicious or compromised delegated principal (one that was granted `ManagePrincipals` by the neuron owner) can:

1. **Remove the original owner's `ManagePrincipals`** — permanently locking the owner out of their own neuron's access control.
2. **Remove all of the owner's permissions** — stripping the owner of the ability to disburse, vote, dissolve, or otherwise manage the neuron.
3. **Remove their own permissions afterward** — leaving the neuron in a permanently uncontrollable state with staked tokens frozen forever.

The neuron's staked SNS tokens (which may represent significant governance power and economic value) become permanently inaccessible. The attacker gains full unilateral control over the neuron's voting power during the window between step 1 and step 3.

---

### Likelihood Explanation

The attack requires the neuron owner to have previously granted `ManagePrincipals` to another principal — a common and intended workflow for delegating neuron management (e.g., to a hot key or a co-manager). Once that grant is made, the delegate can execute the attack with a single ingress call to the SNS governance canister's `manage_neuron` endpoint. No privileged access, no threshold corruption, and no social engineering beyond the initial delegation is required. The entry path is a standard unprivileged ingress message.

---

### Recommendation

Add a check in `remove_neuron_permissions` that prevents any caller from removing permissions from the **original neuron owner** (the principal who initially claimed the neuron) unless the caller **is** that owner. Specifically:

- Track the original claimer/owner of the neuron.
- In `remove_neuron_permissions`, if `principal_id == original_owner`, require `caller == original_owner`.
- Alternatively, prevent any caller from removing `ManagePrincipals` from a principal who is the sole remaining holder of that permission, mirroring the SKALE fix of requiring `msg.sender == validators[validatorId].validatorAddress`.

---

### Proof of Concept

1. **Alice** claims a neuron; she receives `ManagePrincipals` + all other permissions via `neuron_claimer_permissions`.
2. **Alice** calls `manage_neuron` → `AddNeuronPermissions` to grant `ManagePrincipals` (and optionally all permissions) to **Bob**.
3. **Bob** (unprivileged ingress sender) calls `manage_neuron` → `RemoveNeuronPermissions` with:
   - `principal_id = Alice`
   - `permissions_to_remove = [ManagePrincipals, Disburse, ConfigureDissolveState, ...]` (all permissions)
4. `check_principal_authorized_to_change_permissions` passes because Bob holds `ManagePrincipals`. [1](#0-0) 
5. `remove_permissions_for_principal` removes all of Alice's permissions. [2](#0-1) 
6. Alice is now permanently locked out of her own neuron. Bob holds sole `ManagePrincipals` and can disburse, vote, or abandon the neuron at will.
7. Optionally, Bob calls `RemoveNeuronPermissions` on himself, leaving the neuron with zero principals and staked tokens frozen forever. [3](#0-2) 

The root cause is the absence of an owner-identity check in `remove_neuron_permissions` — the function only verifies the caller's permission type, not the relationship between caller and target. [4](#0-3)
