### Title
Append-Only `dapp_canister_ids` Registry Combined with Panicking Pre-flight Check Permanently Blocks SNS Swap Finalization — (File: `rs/sns/root/src/lib.rs`)

### Summary
`SnsRootCanister::dapp_canister_ids` is append-only: entries are added via `register_dapp_canisters` but can only be removed through `set_dapp_controllers`. `set_dapp_controllers` performs a pre-flight loop that unconditionally `panic!`s if `canister_status` fails for any registered dapp canister. If a registered dapp canister is deleted (e.g., runs out of cycles), there is no on-chain path to remove it from `dapp_canister_ids`: `set_dapp_controllers` panics before making any change, and `DeregisterDappCanisters` proposals also route through `set_dapp_controllers`. The SNS swap finalization, which calls `set_dapp_controllers` with `canister_ids: None` (expanding to the full list), is permanently blocked.

### Finding Description

`register_dapp_canisters` appends to `dapp_canister_ids` and is the only way to add entries: [1](#0-0) 

The only removal path is `set_dapp_controllers`, which first iterates over all target canisters in a pre-flight check and unconditionally panics if `canister_status` returns an error: [2](#0-1) 

The swap finalization calls `set_dapp_controllers` with `canister_ids: None`, which expands to the full `dapp_canister_ids` list: [3](#0-2) 

The `DeregisterDappCanisters` governance proposal also routes through `set_dapp_controllers` via `perform_deregister_dapp_canisters`: [4](#0-3) 

There is no direct governance proposal to remove an entry from `dapp_canister_ids` without going through `set_dapp_controllers`. The pre-flight panic is explicitly acknowledged as a known defect: [5](#0-4) 

### Impact Explanation

- SNS swap finalization is permanently blocked: ICP contributed by participants and SNS tokens allocated for the swap cannot be distributed.
- The SNS cannot complete its lifecycle transition from `Open` to `Committed`.
- `DeregisterDappCanisters` proposals targeting the deleted canister also fail, leaving it permanently stuck in the registry.
- Recovery requires an SNS upgrade proposal to patch the Root canister, which itself requires governance participation and introduces delay.

### Likelihood Explanation

- Dapp canisters can run out of cycles due to normal operation, neglect, or deliberate cycle-draining by an adversary who sends many calls to a public dapp canister.
- Any single registered dapp canister deletion triggers the blockage for the entire swap finalization.
- The SNS swap lifecycle is a critical, time-sensitive operation affecting all swap participants.

### Recommendation

- Remove the `panic!` in the pre-flight check of `set_dapp_controllers` and return a graceful error instead, as already noted by the `TODO(NNS1-1993)` comment.
- Add a dedicated governance proposal type (e.g., `PruneDeletedDappCanisters`) that directly removes entries from `dapp_canister_ids` without requiring `canister_status` to succeed.
- Consider adding a heartbeat-based cleanup that attempts `canister_status` and removes unreachable canisters from `dapp_canister_ids`.

### Proof of Concept

1. An SNS is deployed and a dapp canister is registered via a `RegisterDappCanisters` governance proposal — it is appended to `dapp_canister_ids`.
2. The dapp canister runs out of cycles and is deleted by the IC.
3. The SNS swap finalization is triggered (e.g., via `finalize_swap`).
4. `finalize_swap` calls `take_sole_control_of_dapp_controllers`, which calls `set_dapp_controllers` on Root with `canister_ids: None`.
5. The pre-flight loop in `set_dapp_controllers` calls `management_canister.canister_status(deleted_canister)`.
6. The management canister returns an error (canister does not exist).
7. The code panics: `"Could not get the status of canister: {dapp_canister_id}. Root may not be a controller."` — the call is trapped and rejected.
8. Every subsequent `finalize_swap` retry also fails identically.
9. A `DeregisterDappCanisters` proposal targeting the deleted canister also panics at the same pre-flight check.
10. The SNS swap is permanently blocked with no on-chain recovery path short of a Root canister upgrade.
