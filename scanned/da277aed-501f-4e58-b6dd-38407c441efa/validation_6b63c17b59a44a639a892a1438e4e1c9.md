The core question is whether Rust's `char::is_whitespace()` returns `false` for U+200B, causing the filter to keep it in the string, so `"internet\u{200b}computer"` never matches `"internetcomputer"` in the banned list. Let me verify the exact filter logic and any additional validation layers.

### Title
Unicode Non-Whitespace Invisible Character Bypass in `validate_token_name` Allows SNS Token to Impersonate "Internet Computer" — (`rs/nervous_system/common/src/ledger_validation.rs`)

---

### Summary

The banned-name check in `validate_token_name` strips only characters for which Rust's `char::is_whitespace()` returns `true`. U+200B (ZERO WIDTH SPACE), U+00AD (SOFT HYPHEN), and U+FEFF (BOM/ZWNBSP) all have Unicode `White_Space=No`, so `is_whitespace()` returns `false` for them. They are therefore **kept** in the filtered string, preventing a match against `"internetcomputer"` or `"internetcomputerprotocol"`, and the function returns `Ok(())`.

---

### Finding Description

The guard is at: [1](#0-0) 

```
token_name
    .to_lowercase()
    .chars()
    .filter(|c| !c.is_whitespace())   // ← only strips Unicode White_Space=Yes chars
    .collect::<String>()
```

`BANNED_TOKEN_NAMES` contains `"internetcomputer"` and `"internetcomputerprotocol"`. [2](#0-1) 

For input `"internet\u{200b}computer"`:

| Step | Result |
|---|---|
| `to_lowercase()` | `"internet\u{200b}computer"` |
| `filter(!is_whitespace)` | `"internet\u{200b}computer"` (U+200B kept) |
| `collect::<String>()` | `"internet\u{200b}computer"` |
| `BANNED_TOKEN_NAMES.contains(…)` | **false** → `Ok(())` |

The `trim()` guard at line 68 only catches leading/trailing whitespace; an embedded invisible character in the middle is unaffected. [3](#0-2) 

---

### Impact Explanation

An unprivileged SNS creator submits a `CreateServiceNervousSystem` NNS proposal with `token_name = "internet\u{200b}computer"`. The proposal passes NNS governance validation: [4](#0-3) 

which converts to `SnsInitPayload` and calls `validate_pre_execution()`: [5](#0-4) 

which calls `ledger_validation::validate_token_name`: [6](#0-5) 

The SNS ledger is then initialized with a token name that renders identically to "Internet Computer" in every standard UI (wallets, dashboards, explorers), because U+200B is zero-width and invisible. Users can be deceived into treating the SNS token as ICP.

The same bypass applies to post-deployment token name updates via `ManageLedgerParameters`: [7](#0-6) 

---

### Likelihood Explanation

- The attacker only needs enough ICP staked to submit an NNS proposal (currently 10 ICP).
- The bypass is trivially constructable: a single invisible Unicode character embedded in the name.
- No privileged access, no key material, no social engineering of validators is required.
- The NNS proposal itself would display the name visually as "Internet Computer", making it hard for voters to detect the embedded character without raw-byte inspection.

---

### Recommendation

Replace the `is_whitespace()` filter with a Unicode normalization + category-based strip that removes **all** non-visible, non-printable, and format characters before comparison. Concretely:

1. Apply Unicode NFKC normalization (collapses compatibility equivalents).
2. Strip all characters in Unicode General Categories `Cf` (Format), `Cc` (Control), `Zs`/`Zl`/`Zp` (Separators), and any character with `White_Space=Yes`.
3. Alternatively, allowlist only `[a-z0-9 ]` after lowercasing before the banned-name comparison.

### Proof of Concept

```rust
#[test]
fn test_banned_name_unicode_bypass() {
    use crate::ledger_validation::validate_token_name;

    // U+200B ZERO WIDTH SPACE — White_Space=No in Unicode, is_whitespace()=false in Rust
    assert!(validate_token_name("internet\u{200b}computer").is_err(),
        "U+200B bypass: should be banned");

    // U+00AD SOFT HYPHEN — White_Space=No
    assert!(validate_token_name("internet\u{00ad}computer").is_err(),
        "U+00AD bypass: should be banned");

    // U+FEFF BOM / ZWNBSP — White_Space=No
    assert!(validate_token_name("internet\u{feff}computer").is_err(),
        "U+FEFF bypass: should be banned");

    // Protocol variant
    assert!(validate_token_name("internetcomputer\u{200b}protocol").is_err(),
        "Protocol U+200B bypass: should be banned");
}
```

All four assertions currently **fail** (i.e., `validate_token_name` returns `Ok(())` for each), confirming the bypass is real and locally testable without any network access.
