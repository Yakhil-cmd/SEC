### Title
Aggregated Balance Underreports Total Due to `effective_subaccount()` Normalization Collapsing Two Distinct ICRC-1 Accounts Into One DB Row - (`rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs`)

---

### Summary

The ICRC-1 Rosetta API's `get_aggregated_balance_for_principal_at_block_idx` function returns an incorrect (lower-than-actual) aggregated balance for a principal when that principal holds tokens in both `Account { subaccount: None }` and `Account { subaccount: Some([0u8; 32]) }`. These are two semantically distinct ICRC-1 accounts, but both are stored under the same `subaccount` bytes (`[0u8; 32]`) in the SQLite `account_balances` table because `effective_subaccount()` normalizes `None` to `[0u8; 32]` at write time. The SQL aggregation query then groups by `subaccount`, counting only one entry and silently dropping the other account's balance.

---

### Finding Description

**Root cause — write path:**

When `update_account_balances` inserts balance records, it calls `account.effective_subaccount().as_slice()`: [1](#0-0) 

`effective_subaccount()` is defined as: [2](#0-1) 

This means both `Account { owner: P, subaccount: None }` and `Account { owner: P, subaccount: Some([0u8; 32]) }` are written to the `account_balances` table with the same `subaccount` column value (`[0u8; 32]`). The table schema uses `(principal, subaccount, block_idx)` as the primary key: [3](#0-2) 

**Root cause — read path:**

`get_aggregated_balance_for_principal_at_block_idx` groups by `subaccount` implicitly (via the correlated subquery on `a2.subaccount = a1.subaccount`). Since both accounts share the same stored subaccount bytes, only one row exists in the DB for both, and the SQL query returns only one balance entry — the one that was written last: [4](#0-3) 

The same normalization bug also affects `get_account_balance_at_block_idx` when queried with `effective_subaccount()`: [5](#0-4) 

**The bug is explicitly acknowledged in the codebase itself:** [6](#0-5) 

The `account_balance_with_metadata` service function exposes this via the public `/account/balance` Rosetta endpoint when `aggregate_all_subaccounts: true` is set in the request metadata: [7](#0-6) 

---

### Impact Explanation

Any unprivileged caller of the ICRC-1 Rosetta `/account/balance` endpoint with `aggregate_all_subaccounts: true` will receive an incorrect (understated) balance for any principal that holds tokens in both the default subaccount (`None`) and the explicit zero subaccount (`Some([0u8; 32])`). The missing balance is silently dropped — not an error, just a wrong number. Exchanges, wallets, or DeFi applications relying on the Rosetta API for balance checks will observe a lower-than-actual balance, potentially blocking withdrawals or causing incorrect accounting decisions.

---

### Likelihood Explanation

The ICRC-1 standard treats `subaccount: None` and `subaccount: Some([0u8; 32])` as the same account at the protocol level (via `effective_subaccount()`), so in practice most users will not have funds split across both. However, any application that explicitly constructs `Some([0u8; 32])` as a subaccount (e.g., a canister using a zero-filled byte array as a default) while the same principal also has a `None`-subaccount balance will trigger this. The Rosetta API is a public, unauthenticated read endpoint, so any boundary/API user can observe the incorrect result.

---

### Recommendation

At write time in `update_account_balances`, store the raw subaccount value (preserving `None` vs `Some([0u8; 32])` distinctly) rather than always calling `effective_subaccount()`. Alternatively, add a sentinel byte or flag column to distinguish the two cases in the DB. The `get_aggregated_balance_for_principal_at_block_idx` SQL query must then correctly enumerate all distinct stored subaccounts.

---

             FROM account_balances \
             WHERE principal = :principal \
             AND subaccount = :subaccount \
             AND block_idx <= :block_idx \
             ORDER BY block_idx \

}

pub fn get_aggregated_balance_for_principal_at_block_idx(
    connection: &Connection,
    principal: &PrincipalId,
    block_idx: u64,
) -> anyhow::Result<Nat> {
    // Query to get the latest balance for each subaccount of the principal at or before the given block index
    let mut stmt = connection.prepare_cached(
        "SELECT a1.subaccount, a1.amount
         FROM account_balances a1
         WHERE a1.principal = :principal
         AND a1.block_idx = (
             SELECT MAX(a2.block_idx)
             FROM account_balances a2
             WHERE a2.principal = a1.principal
             AND a2.subaccount = a1.subaccount
             AND a2.block_idx <= :block_idx
         )",
    )?;

    let rows = stmt.query_map(
        named_params! {
            ":principal": principal.as_slice(),
            ":block_idx": block_idx
        },
        |row| {
            let amount_str: String = row.get(1)?;
            Nat::from_str(&amount_str).map_err(|_| {
                rusqlite::Error::InvalidColumnType(
                    1,
                    "amount".to_string(),
                    rusqlite::types::Type::Text,
                )
            })
        },
    )?;

    let mut total_balance = Nat(BigUint::zero());
    for balance_result in rows {
        let balance = balance_result?;
        total_balance = Nat(total_balance
            .0
            .checked_add(&balance.0)
            .with_context(|| "Overflow while aggregating balances")?);
    }

    Ok(total_balance)
}
