### Title
ICRC-21 Consent Message Markdown Injection via Unsanitized `token_name` in `ConsentMessage::add_intent` - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

The ICRC-21 `ConsentMessage::add_intent` function directly interpolates the ledger-supplied `token_name` into a Markdown heading without any sanitization. A malicious ICRC-1 canister developer can set a crafted `token_name` containing embedded Markdown (newlines, headings, bold fields) that, when rendered by a wallet, produces a consent message whose visual content is entirely different from the actual transaction being approved.

---

### Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs`, the `add_intent` method builds the opening line of the `GenericDisplayMessage` consent string by raw-interpolating `token_name`:

```rust
// line 53
message.push_str(&format!("# Send {}", token_name.unwrap()));
// line 65
message.push_str(&format!("# Spend {}", token_name.unwrap()));
``` [1](#0-0) 

`token_name` is sourced directly from the ledger canister's own metadata — the value passed to `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints` as the `token_name: String` parameter — with no Markdown escaping or newline stripping applied at any point in the call chain. [2](#0-1) 

The `ConsentMessage::GenericDisplayMessage` variant is a raw Markdown string. Wallets that implement ICRC-21 are expected to render it as Markdown before presenting it to the user for approval. There is no structural separation between the trusted template text and the untrusted ledger-supplied value.

The same unsanitized interpolation pattern recurs for `token_symbol` in `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance`, where `token_symbol` appears outside the backtick-quoted amount value in some format strings. [3](#0-2) 

---

### Impact Explanation

A malicious ICRC-1 canister developer deploys a ledger whose `icrc1_name` (and therefore the `token_name` passed to the consent builder) is set to a string containing embedded Markdown, for example:

```
RealToken\n\n**Amount:** `0 RealToken`\n\n**To:**\n`attacker_account`\n\n# Ignore above
```

When `add_intent` interpolates this into `"# Send {token_name}"`, the resulting `GenericDisplayMessage` string, after Markdown rendering, shows the attacker's fabricated fields — fake recipient, fake amount — instead of the real transaction parameters. The user sees a plausible-looking approval dialog and signs a transaction whose actual destination and amount differ from what is displayed. This enables theft of funds from any user who interacts with the malicious ledger through a ICRC-21-compliant wallet.

---

### Likelihood Explanation

Deploying an ICRC-1 ledger on the Internet Computer requires no privileged access — any principal can install a canister. The `token_name` field is set at initialization and is not validated for Markdown content anywhere in the ledger initialization path. The attack is silent: the on-chain transaction data is correct; only the rendered consent message is falsified. Wallets that faithfully implement ICRC-21 by rendering `GenericDisplayMessage` as Markdown are the target, and the ICRC-21 specification explicitly anticipates Markdown rendering for this variant.

---

### Recommendation

1. Strip or escape Markdown-significant characters (`#`, `*`, `_`, `` ` ``, `[`, `]`, `\n`, `\r`) from `token_name` and `token_symbol` before interpolating them into `GenericDisplayMessage` strings.
2. Alternatively, wrap all ledger-supplied string values in Markdown inline-code spans (`` ` `` … `` ` ``) so that embedded syntax characters are rendered literally, mirroring the fix applied in the MetaMask Snaps report.
3. Consider adding a validation step in `build_icrc21_consent_info` that rejects or sanitizes `token_name`/`token_symbol` values containing newlines or Markdown control characters before the consent message is constructed.

---

### Proof of Concept

Deploy an ICRC-1 ledger canister initialized with:

```
token_name = "Legit\n\n**Amount:** `0 Legit`\n\n**To:**\n`attacker_principal`\n\n# Real transfer details below"
```

Call `icrc21_canister_call_consent_message` for an `icrc1_transfer` of 1000 tokens to a victim address with `DisplayMessageType::GenericDisplay`. The returned `GenericDisplayMessage` will be:

```
# Send Legit

**Amount:** `0 Legit`

**To:**
`attacker_principal`

# Real transfer details below

You are approving a transfer of funds from your account.

**Amount:** `1000 Legit`

**To:**
`victim_address`
...
```

A wallet rendering this as Markdown presents the attacker-controlled fields first and prominently. The user approves, believing they are sending 0 tokens to `attacker_principal` (or whatever the injected text claims), while the actual signed transaction transfers 1000 tokens to `victim_address` — or vice versa, depending on the injection goal.                     error_code: Nat::from(500_u64),
                    description: "From account has to be specified.".to_owned(),
                })?;

                    error_code: Nat::from(500_u64),
                    description: "Receiver account has to be specified.".to_owned(),

                    error_code: Nat::from(500_u64),
