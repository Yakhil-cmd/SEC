### Title
Unbounded Tokio task spawning via oversized `StrippedBlockProposal` in `assemble_message` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

### Summary
A Byzantine subnet peer (below the fault threshold) can craft a `StrippedBlockProposal` that carries a valid `unstripped_consensus_message_id` (one the bouncer returns `Wants` for) but contains an arbitrarily large number of fake `ingress_messages` or `stripped_idkg_dealings` entries. Because `assemble_message` spawns one `get_or_fetch` Tokio task per entry with no upper-bound check, the receiving replica can be forced to spawn tens of thousands of concurrent tasks, exhausting heap memory or the Tokio thread pool.

### Finding Description
In `assemble_message` (`assembler.rs` lines 209–226), `BlockProposalAssembler::new` is called with the peer-supplied `StrippedBlockProposal`, and then a `for` loop immediately spawns one `JoinSet` task per element returned by `missing_stripped_messages()`. Neither `BlockProposalAssembler::new` nor the deserialization path (`TryFrom<pb::StrippedBlockProposal>`) enforces any limit on the number of entries. The bouncer check that could abort the work only runs inside the subsequent result-processing loop — after all N tasks have already been spawned. The transport layer has its body-size limit explicitly disabled. The only downstream check (`try_assemble` ID mismatch) fires after all tasks complete, far too late to prevent resource exhaustion.

### Impact Explanation
Each spawned `get_or_fetch` task acquires a read lock on the ingress pool, allocates async stack frames, and (when the message is absent from the pool) enters a retry loop against peers. With N = 100,000 entries, this causes: (a) heap exhaustion from task state, (b) lock contention on the ingress/IDKG pool `RwLock`, and (c) a flood of outbound RPC requests. The bouncer refresh period is 3 seconds, so tasks live for at least that long before being aborted. The attack can be repeated every consensus round, causing a sustained crash of a single replica.

### Likelihood Explanation
The attacker needs only a single subnet membership slot (below the Byzantine fault threshold). Block IDs are publicly advertised over P2P, so the attacker can always supply a currently-wanted ID. The crafted response passes the only content check (ID equality). No privileged access, key material, or majority corruption is required.

### Recommendation
Add a count guard immediately after `BlockProposalAssembler::new` (or inside `TryFrom<pb::StrippedBlockProposal>`) that rejects any `StrippedBlockProposal` whose `ingress_messages.len()` exceeds `MAX_INGRESS_MESSAGES_PER_BLOCK` or whose `stripped_dealings.len()` exceeds the configured `dkg_dealings_per_block` limit. This mirrors the existing check in `validate_ingress_payload` and closes the gap before any tasks are spawned.

### Proof of Concept
Construct a `pb::StrippedBlockProposal` with a valid `unstripped_consensus_message_id` (copied from a real advertised block) and `ingress_messages` populated with 100,000 synthetic `StrippedIngressMessage` entries. Serve this from a mock peer. Call `assemble_message` with the bouncer returning `Wants`. Observe that 100,000 tasks are spawned in the `JoinSet` before any bouncer check fires, and that memory usage spikes proportionally.
