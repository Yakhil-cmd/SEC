### Summary

The ckERC20 deposit helper smart contracts emit the caller-supplied `amount` parameter in the `ReceivedErc20` / `ReceivedEthOrErc20` event rather than the actual balance delta received by the minter's Ethereum address. The IC ckETH minter canister scrapes these events and unconditionally mints ckERC20 tokens equal to the event-logged `amount`. For any ERC20 token that charges a transfer fee (fee-on-transfer), the minter's Ethereum address receives strictly less than `amount`, but the IC ledger is credited the full `amount`. This inflates the ckERC20 supply beyond the actual ERC20 collateral held, making the system undercollateralized and causing the last withdrawers to fail.

### Impact Explanation

For every deposit of a fee-on-transfer ERC20 token, the ckERC20 ledger total supply grows by `amount` while the minter's Ethereum address only holds `amount - fee`. The deficit compounds with each deposit. When users attempt to withdraw ckERC20 back to ERC20, the minter constructs an Ethereum transaction sending the full withdrawal amount. Once cumulative fees exceed the minter's surplus, withdrawal transactions will fail on-chain (insufficient ERC20 balance at the minter address), permanently locking ckERC20 tokens for the last set of depositors. The minter's internal `erc20_balances` accounting will also diverge from reality, causing the minter to believe it holds more collateral than it does.

**Vulnerability class:** chain-fusion mint/burn/replay bug — ledger conservation invariant broken (ckERC20 total supply > actual ERC20 collateral).

---

### Likelihood Explanation

The minter enforces a whitelist of supported ERC20 tokens via NNS governance proposals. However:

1. USDT (Tether) has a built-in fee mechanism that is currently set to 0 but can be activated by the Tether issuer at any time without any IC governance action. If USDT is added as a supported ckERC20 token and Tether later activates its fee, the vulnerability becomes live immediately.
2. Any future NNS proposal to add a fee-on-transfer token (intentionally or by oversight) would trigger the issue.
3. The deposit flow is fully permissionless once a token is supported — any unprivileged Ethereum user can trigger deposits.

Likelihood is **Medium**: requires a supported token to have an active transfer fee, which is not the current state for USDC/USDT but is a realistic future condition.

---

### Recommendation

In both helper contracts, measure the actual balance delta instead of emitting the input `amount`:

```solidity
uint256 balanceBefore = IERC20(erc20Address).balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 received = IERC20(erc20Address).balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, received, principal, subaccount);
```

This ensures the event-logged value always equals the actual collateral received, so the IC minter mints exactly the backed amount. Alternatively, the minter canister could periodically reconcile its internal `erc20_balances` against on-chain `eth_call` balance queries and refuse to mint when a deficit is detected.

pub fn register_deposit_events(
    scraping_id: LogScrapingId,
    transaction_events: Vec<ReceivedEvent>,
    errors: Vec<ReceivedEventError>,
) {
    for event in transaction_events {
        log!(
            INFO,
            "Received event {event:?}; will mint {} {scraping_id} to {}",
            event.value(),
            event.beneficiary()
        );
        if crate::blocklist::is_blocked(&event.from_address()) {
            log!(
                INFO,
                "Received event from a blocked address: {} for {} {scraping_id}",
                event.from_address(),
                event.value(),
            );
            mutate_state(|s| {
                process_event(
                    s,
                    EventType::InvalidDeposit {
                        event_source: event.source(),
                        reason: format!("blocked address {}", event.from_address()),
                    },
                )
            });
        } else {
            mutate_state(|s| process_event(s, event.into_deposit()));
        }
    }
    if read_state(State::has_events_to_mint) {
        ic_cdk_timers::set_timer(Duration::from_secs(0), async { mint().await });
    }
    for error in errors {
        if let ReceivedEventError::InvalidEventSource { source, error } = &error {
            mutate_state(|s| {
                process_event(
                    s,
                    EventType::InvalidDeposit {
                        event_source: *source,
                        reason: error.to_string(),
                    },
                )
            });
        }
        report_transaction_error(error);
    }
}

        }
