### Title
NNS Governance `Follow` Command Allows Duplicate Followee IDs, Skewing Majority Vote Calculation - (`rs/nns/governance/src/governance.rs`)

### Summary

The NNS governance `Follow` command accepts a `Vec<NeuronId>` for the `followees` field without any deduplication check. The `would_follow_ballots` function iterates over this raw vector and counts votes against `followees.len()`. Listing the same neuron ID N times inflates that neuron's vote weight N-fold relative to other followees, violating the stated "majority of followees" semantics and allowing any neuron owner to manipulate their neuron's automatic vote outcome.

### Finding Description

The `follow()` function in `rs/nns/governance/src/governance.rs` validates only authorization, topic validity, followee existence/visibility, and the `MAX_FOLLOWEES_PER_TOPIC` count. It performs no deduplication of the submitted `followees` list. [1](#0-0) 

The stored `Followees { followees: Vec<NeuronId> }` preserves multiplicity. The `would_follow_ballots` function then iterates over this raw vector: [2](#0-1) 

Because `yes` and `no` are incremented once per list entry (not per unique neuron), and the majority threshold is computed against `followees.len()` (the raw length including duplicates), listing neuron B three times gives B's vote 3× the weight of any other followee. The integration test explicitly confirms this is accepted behavior: [3](#0-2) 

The `SetFollowing` command in NNS governance (`rs/nns/governance/src/pb/mod.rs`) also does not check for duplicate followee IDs within a topic — `validate_intrinsically()` only checks topic uniqueness and followee count: [4](#0-3) 

Note: the SNS governance `SetFollowing` path (`rs/sns/governance/src/following.rs`) does correctly reject duplicate neuron IDs via `get_duplicate_followee_groups`, but the NNS `Follow` and NNS `SetFollowing` paths do not. [5](#0-4) 

### Impact Explanation

**Governance voting manipulation.** The NNS documentation states: *"Once a majority of the followees votes to adopt or reject a proposal belonging to the specified topic, the neuron votes the same way."* The intent is equal weight per followee. With duplicates, a neuron owner can give one followee N× the weight of others.

Concrete example — neuron A follows `[B, B, B, C, D]`:
- B votes Yes, C and D vote No
- `yes = 3`, `no = 2`, `followees.len() = 5` → `3×2 > 5` → **Vote::Yes**
- With unique followees `[B, C, D]`: `yes = 1`, `no = 2`, `len = 3` → `1×2 > 3` is false, `2×2 >= 3` → **Vote::No**

A neuron with large voting power that follows this pattern will cast an automatic Yes vote even though 2 of 3 unique followees voted No. This breaks the liquid democracy invariant and can affect NNS proposal outcomes if enough neurons with significant voting power adopt such configurations.

### Likelihood Explanation

Any neuron owner can submit a `manage_neuron` ingress message with a `Follow` command containing duplicate IDs. No special privilege, admin key, or governance majority is required. The operation is cheap (no fee for `Follow`), and the integration test confirms the canister accepts it without error. The attack is fully self-service.

### Recommendation

In `follow()` (`rs/nns/governance/src/governance.rs`), deduplicate the `followees` list before storing it, or reject requests containing duplicate `NeuronId` values — analogous to the check already present in SNS governance's `ValidatedFolloweesForTopic::try_from` (`rs/sns/governance/src/following.rs`). Apply the same fix to `SetFollowing`'s `validate_intrinsically()` in `rs/nns/governance/src/pb/mod.rs`.

### Proof of Concept

1. Neuron owner calls `manage_neuron` with:
   ```
   Follow { topic: NetworkEconomics, followees: [B, B, B, C, D] }
   ```
2. `follow()` accepts this — no duplicate check exists.
3. `would_follow_ballots` is called when B votes Yes and C, D vote No:
   - `yes = 3` (B counted 3×), `no = 2`, `followees.len() = 5`
   - `3 * 2 > 5` → returns `Vote::Yes`
4. With unique followees `[B, C, D]`, the same votes yield `Vote::No`.
5. The neuron's automatic vote is Yes instead of No, violating the majority-of-unique-followees invariant. [6](#0-5) [7](#0-6)
