### Title
Malicious or Hacked ICRC-1 Ledger Controller Can Immediately Drain All User Token Balances via `icrc152_burn` - (File: rs/ledger_suite/icrc1/ledger/src/main.rs)

---

### Summary

The ICRC-1 ledger canister exposes two privileged endpoints, `icrc152_mint` and `icrc152_burn`, gated solely by `ic_cdk::api::is_controller(&caller)`. When the `icrc152` feature flag is enabled, any canister controller can immediately burn tokens from **any user's account** without the user's consent and without any timelock or delay. This is the direct IC analog of M-02: a privileged role can immediately steal all assets on the platform with no opportunity for users to react.

---

### Finding Description

The ICRC-1 ledger in `rs/ledger_suite/icrc1/ledger/src/main.rs` implements `icrc152_burn_not_async` and `icrc152_mint_not_async`. The sole authorization check for both is:

```rust
if !ic_cdk::api::is_controller(&caller) {
    return Err(Icrc152BurnError::Unauthorized(
        "caller is not a controller".to_string(),
    ));
}
``` [1](#0-0) 

Once this check passes, `icrc152_burn` applies an `AuthorizedBurn` operation that debits **any specified account** — not just the caller's own account — with no user signature, no approval, and no timelock:

```rust
let tx = Transaction {
    operation: Operation::AuthorizedBurn {
        from: args.from,   // attacker-controlled target account
        amount,
        caller: Some(caller),
        ...
    },
    ...
};
let (block_idx, _) = apply_transaction(ledger, tx, now, Tokens::zero())...
``` [2](#0-1) 

Similarly, `icrc152_mint` allows a controller to mint arbitrary tokens to any account: [3](#0-2) 

The feature is activated via `FeatureFlags { icrc152: true }` at init or upgrade time: [4](#0-3) 

The `UpgradeArgs` type allows the `icrc152` flag to be toggled on via a ledger upgrade: [5](#0-4) 

And the upgrade path applies the flag immediately with no delay: [6](#0-5) 

There is **no timelock**, **no governance vote**, and **no user-facing warning period** between a controller enabling `icrc152` and being able to drain all balances.

---

### Impact Explanation

A malicious or hacked controller of any ICRC-1 ledger with `icrc152: true` can:

1. Call `icrc152_burn` with `from` set to any user account and `amount` set to the full balance.
2. Repeat for every account on the ledger.
3. All user token balances are destroyed immediately, with no recourse.

Alternatively, the controller can call `icrc152_mint` to inflate supply to themselves, diluting all holders to near-zero value.

The `icrc152_burn` operation bypasses the normal ICRC-1/ICRC-2 transfer flow entirely — no user approval, no allowance, no signature from the account owner is required. The only requirement is being a canister controller. [7](#0-6) 

---

### Likelihood Explanation

This is a centralization risk. The ledger controller is a trusted role — for SNS-governed ledgers this is the SNS root canister (itself governed by SNS token holders), and for independently deployed ledgers it may be a single developer key. The risk is:

- A developer key is compromised (private key leak, phishing, supply-chain attack).
- A malicious SNS governance proposal enables `icrc152` and immediately drains balances before token holders can react — there is no mandatory delay between proposal execution and the drain.
- A ledger deployed with `icrc152: true` from the start gives the deployer immediate, permanent power to drain all users.

The `icrc152` feature is already present in the production ledger codebase and is opt-in, meaning any ledger operator can activate it. Users holding tokens in such a ledger have no on-chain mechanism to detect or prevent a drain. [8](#0-7) 

---

### Recommendation

1. **Timelock for `icrc152_burn`**: Require a mandatory delay (e.g., 7 days) between a controller announcing intent to burn and the burn executing, giving users time to exit.
2. **Restrict `icrc152_burn` to the minting account only**: Align with the existing ICRC-1 model where only the designated minting account can perform privileged supply operations, rather than any controller.
3. **Governance-only activation**: Require that enabling `icrc152` go through a governance proposal with a mandatory voting and execution delay, not a simple upgrade argument.
4. **User-visible disclosure**: Expose `icrc152` status prominently in `icrc1_supported_standards` and metadata so users can assess the risk before holding tokens.

---

### Proof of Concept

**Attacker preconditions**: Attacker controls (or compromises) the controller principal of an ICRC-1 ledger with `feature_flags.icrc152 = true`.

**Trigger path**:

```
// Attacker calls icrc152_burn as the ledger controller
icrc152_burn({
    from: { owner: victim_principal, subaccount: null },
    amount: victim_full_balance,
    created_at_time: current_time_nanos,
    reason: null
})
```

**Result**: `apply_transaction` records an `AuthorizedBurn` operation, debiting `victim_principal`'s balance to zero. No user consent, no approval, no timelock. The call succeeds immediately. [9](#0-8) [10](#0-9) 

The test suite confirms this is the intended behavior for a controller: [11](#0-10)

fn icrc152_burn_not_async(
    caller: Principal,
    args: Icrc152BurnArgs,
) -> Result<u64, Icrc152BurnError> {
    let block_idx = Access::with_ledger_mut(|ledger| {
        if !ledger.feature_flags().icrc152 {
            return Err(Icrc152BurnError::GenericError {
                error_code: Nat::from(0_u64),
                message: "ICRC-152 is not enabled".to_string(),
            });
        }
        if !ic_cdk::api::is_controller(&caller) {
            return Err(Icrc152BurnError::Unauthorized(
                "caller is not a controller".to_string(),
            ));
        }
        if args.amount == 0_u64 {
            return Err(Icrc152BurnError::GenericError {
                error_code: Nat::from(0_u64),
                message: "amount must be greater than 0".to_string(),
            });
        }
        let amount =
            Tokens::try_from(args.amount.clone()).map_err(|_| Icrc152BurnError::GenericError {
                error_code: Nat::from(0_u64),
                message: "amount is too large".to_string(),
            })?;
        if args.from.owner == Principal::anonymous() {
            return Err(Icrc152BurnError::InvalidAccount(
                "anonymous principal is not allowed".to_string(),
            ));
        }
        if &args.from == ledger.minting_account() {
            return Err(Icrc152BurnError::InvalidAccount(
                "cannot burn from the minting account".to_string(),
            ));
        }
        if let Some(ref reason) = args.reason
            && reason.len() > MAX_REASON_LENGTH
        {
            return Err(Icrc152BurnError::GenericError {
                error_code: Nat::from(0_u64),
                message: format!("reason must be at most {} bytes", MAX_REASON_LENGTH),
            });
        }
        let now = TimeStamp::from_nanos_since_unix_epoch(ic_cdk::api::time());
        let tx = Transaction {
            operation: Operation::AuthorizedBurn {
                from: args.from,
                amount,
                caller: Some(caller),
                mthd: Some(MTHD_152_BURN.to_string()),
                reason: args.reason,
            },
            created_at_time: Some(args.created_at_time),
            memo: None,
        };
        let (block_idx, _) =
            apply_transaction(ledger, tx, now, Tokens::zero()).map_err(|err| match err {
                CoreTransferError::TxDuplicate { duplicate_of } => Icrc152BurnError::Duplicate {
                    duplicate_of: Nat::from(duplicate_of),
                },
                CoreTransferError::InsufficientFunds { balance } => {
                    Icrc152BurnError::InsufficientBalance {
                        balance: balance.into(),
                    }
                }
                CoreTransferError::TxTooOld { .. } => Icrc152BurnError::GenericError {
                    error_code: Nat::from(0_u64),
                    message: "transaction too old".to_string(),
                },
                CoreTransferError::TxCreatedInFuture { .. } => Icrc152BurnError::GenericError {
                    error_code: Nat::from(0_u64),
                    message: "transaction created in the future".to_string(),
                },
                CoreTransferError::TxThrottled => Icrc152BurnError::GenericError {
                    error_code: Nat::from(0_u64),
                    message: "temporarily unavailable".to_string(),
                },
                other => Icrc152BurnError::GenericError {
                    error_code: Nat::from(0_u64),
                    message: format!("unexpected error: {:?}", other),
                },
            })?;
        update_total_volume(amount, false);
        Ok(block_idx)
    })?;
    Ok(block_idx)
}

pub fn test_icrc152_mint_and_burn<T>(ledger_wasm: Vec<u8>, encode_init_args: fn(InitArgs) -> T)

    T: CandidType,
{
    let p1 = PrincipalId::new_user_test_id(1);
    let (env, canister_id) = setup_icrc152(ledger_wasm, encode_init_args, vec![]);
    let controller = PrincipalId::new_anonymous();

    let supply_before = total_supply(&env, canister_id);
    assert_eq!(supply_before, 0);

    // --- Mint ---
    let mint_amount = 5_000_000_u64;
    let mint_result = icrc152_mint(

        canister_id,
        controller,
        &Icrc152MintArgs {
            to: Account::from(p1.0),
            amount: Nat::from(mint_amount),
            created_at_time: now_nanos(&env),
            reason: Some("test mint".to_string()),
        },
    );
    let mint_block_idx = mint_result.expect("icrc152_mint should succeed");
    assert_eq!(mint_block_idx, Nat::from(0_u64));

    assert_eq!(balance_of(&env, canister_id, p1.0), mint_amount);
    assert_eq!(total_supply(&env, canister_id), mint_amount);
