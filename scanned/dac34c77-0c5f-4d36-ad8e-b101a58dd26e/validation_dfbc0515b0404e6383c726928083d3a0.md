### Title
Unbounded Ingress Expiry Loop in ICP Rosetta `construction_payloads` Enables Denial-of-Service — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

An unprivileged client can supply arbitrarily large `ingress_start`/`ingress_end` values in a `ConstructionPayloadsRequest` metadata object, causing the ICP Rosetta node to execute an unbounded loop that allocates millions of `SigningPayload` objects, exhausting memory and CPU.

---

### Finding Description

The `construction_payloads` function computes a step interval and then loops from `ingress_start` to `ingress_end`, pushing one entry per step:

```
interval = MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s
         = 300s - 60s - 120s = 120 seconds
``` [1](#0-0) [2](#0-1) 

The loop itself:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;
}
``` [3](#0-2) 

Both `ingress_start` and `ingress_end` are taken directly from the client-supplied metadata with no range validation before the loop: [4](#0-3) 

The metadata fields are plain `Option<u64>` nanosecond timestamps with no server-side cap: [5](#0-4) 

After the loop, `add_payloads` is called once per transaction and iterates over **all** `ingress_expiries`, creating **two** `SigningPayload` objects per entry (one update payload, one read-state payload): [6](#0-5) 

The HTTP handler is unauthenticated and synchronous: [7](#0-6) 

---

### Impact Explanation

With `interval = 120s` and `ingress_end = ingress_start + 10 years`:

- Iterations ≈ 315,576,000 / 120 ≈ **2,629,800**
- With one transaction: **~5.26 million `SigningPayload` objects** allocated
- Each `SigningPayload` contains a hex-encoded 32-byte hash, an `AccountIdentifier`, and a `SignatureType` — easily 200+ bytes each
- Total allocation: **>1 GB** per request
- CPU: hashing and hex-encoding ~5M times per request

A single malicious HTTP POST to `/construction/payloads` with a tiny JSON body (two `u64` fields) triggers this. Multiple concurrent requests trivially OOM-kill the Rosetta node.

---

### Likelihood Explanation

- No authentication is required to call `/construction/payloads`
- The request body is tiny (two JSON integers); no body-size limit prevents it
- The attack is trivially reproducible with `curl`
- The ICRC1 Rosetta implementation has partial validation (`ingress_start >= ingress_end` check) but the ICP Rosetta implementation has **none** [8](#0-7) 

---

### Recommendation

Add a maximum allowed window check before the loop in `construction_payloads`:

```rust
let max_window = interval * MAX_INGRESS_EXPIRIES; // e.g., MAX_INGRESS_EXPIRIES = 100
if ingress_end > ingress_start + max_window {
    return Err(ApiError::invalid_request(
        "ingress_end exceeds maximum allowed window"
    ));
}
```

Alternatively, truncate `ingress_expiries` to a hard cap (e.g., 100 entries) after the loop.

---

### Proof of Concept

```bash
curl -X POST http://<rosetta-node>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<ledger_id>"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<addr>"},"amount":{"value":"-1","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 1000000000000000000,
      "ingress_end":   1315576000000000000
    }
  }'
```

`ingress_end - ingress_start` = 315,576,000 seconds (10 years). With `interval = 120s`, the loop executes ~2.63 million times before the process runs out of memory.

fn add_payloads(
    payloads: &mut Vec<SigningPayload>,
    ingress_expiries: &[u64],
    account_identifier: &AccountIdentifier,
    update: &HttpCanisterUpdate,
    signature_type: SignatureType,
) {
    for ingress_expiry in ingress_expiries {
        let mut update = update.clone();
        update.ingress_expiry = *ingress_expiry;
        let message_id = update.id();
        let transaction_payload = SigningPayload {
            address: None,
            account_identifier: Some(account_identifier.clone()),
            hex_bytes: hex::encode(make_sig_data(&message_id)),
            signature_type: Some(signature_type),
        };
        payloads.push(transaction_payload);
        let read_state = make_read_state_from_update(&update);
        let read_state_message_id = MessageId::from(read_state.representation_independent_hash());
        let read_state_payload = SigningPayload {
            address: None,
            account_identifier: Some(account_identifier.clone()),
            hex_bytes: hex::encode(make_sig_data(&read_state_message_id)),
            signature_type: Some(signature_type),
        };
        payloads.push(read_state_payload);
    }
}
