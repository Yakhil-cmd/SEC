### Title
Fee-on-Transfer ERC-20 Deposit Emits Unreduced Amount, Causing ckERC20 Over-Minting - (File: rs/ethereum/cketh/minter/ERC20DepositHelper.sol, rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol)

---

### Summary

Both ckERC20 helper smart contracts emit the deposit event using the caller-supplied `amount` parameter rather than the actual balance delta received by the minter address. The IC ckETH minter canister parses this event value and mints exactly that many ckERC20 tokens. For any fee-on-transfer or deflationary ERC-20 token, the minter receives fewer tokens than the event records, breaking the 1:1 backing invariant and inflating the ckERC20 supply beyond the actual ERC-20 collateral held.

---

### Finding Description

**`ERC20DepositHelper.sol` — `CkErc20Deposit.deposit()`** (the legacy helper):

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal);  // ← uses input `amount`
}
```

**`DepositHelperWithSubaccount.sol` — `CkDeposit.depositErc20()`** (the subaccount-aware helper):

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);  // ← uses input `amount`
}
```

In both cases, the emitted `amount` field is the *requested* transfer amount, not the actual balance increase at `minterAddress`. For a fee-on-transfer token, `minterAddress` receives `amount - fee`, but the event records `amount`.

The IC minter canister scrapes these logs and parses the `value` field directly from the event data in `ReceivedErc20LogParser::parse_log()`:

```rust
let [value_bytes] = parse_hex_into_32_byte_words(entry.data, event_source)?;
// ...
value: Erc20Value::from_be_bytes(value_bytes),  // ← trusts the event-logged amount
```

The `mint()` function in `deposit.rs` then mints exactly `event.value()` ckERC20 tokens to the beneficiary:

```rust
amount: event.value(),  // ← mints the event-logged amount, not actual received
```

There is no on-chain or off-chain reconciliation step that compares the minted ckERC20 supply against the actual ERC-20 balance held by the minter address.

---

### Impact Explanation

For any fee-on-transfer ERC-20 token that is added to the minter's supported list, every deposit mints more ckERC20 than the ERC-20 tokens actually held by the minter. The ckERC20 supply becomes partially unbacked. When users later attempt to withdraw ckERC20 back to ERC-20, the minter will eventually be unable to fulfill withdrawals because it holds less ERC-20 than the outstanding ckERC20 supply. This is a direct ledger conservation / chain-fusion mint bug: the 1:1 peg invariant is broken by the over-minting, and the deficit accumulates with every deposit of such a token.

**Impact: 3/5** — Breaks the core 1:1 backing invariant of the chain-fusion bridge for affected tokens; funds of later withdrawers are at risk.

---

### Likelihood Explanation

The IC minter enforces a whitelist of supported ERC-20 tokens (checked in `deposit.rs` before minting). Currently supported tokens (USDC, USDT) are not fee-on-transfer. However, the helper contracts themselves enforce no whitelist, and the minter's supported token list is governed by NNS proposals and can be extended. Any future addition of a fee-on-transfer or rebasing ERC-20 token to the supported list would immediately trigger this bug for all deposits of that token. The vulnerability is latent and requires no privileged access to exploit — any unprivileged Ethereum user calling `deposit()` or `depositErc20()` with a fee-on-transfer token triggers it.

**Likelihood: 2/5** — Requires a fee-on-transfer token to be added to the supported list; not exploitable with current supported tokens, but the code path is fully reachable.

---

### Recommendation

Both helper contracts should measure the actual balance delta at the minter address rather than emitting the caller-supplied `amount`. The fix mirrors the Beanstalk recommendation exactly:

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    uint256 balanceBefore = erc20Token.balanceOf(cketh_minter_main_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    uint256 actualReceived = erc20Token.balanceOf(cketh_minter_main_address) - balanceBefore;
    emit ReceivedErc20(erc20_address, msg.sender, actualReceived, principal);
}
```

Apply the same pattern to `CkDeposit.depositErc20()` in `DepositHelperWithSubaccount.sol`. The IC minter canister code in `deposit.rs` and `parser.rs` requires no changes — it already trusts the event value, and the fix is to make the event value accurate.

### Proof of Concept

1. Deploy a fee-on-transfer ERC-20 token (e.g., 1% fee on every transfer) on Ethereum.
2. Have the NNS add this token to the ckETH minter's supported list.
3. Approve the helper contract for `1,000,000` units and call `depositErc20(tokenAddr, 1_000_000, principal, subaccount)`.
4. The helper calls `safeTransferFrom(..., 1_000_000)`. Due to the 1% fee, the minter address receives `990,000` tokens. The helper emits `ReceivedEthOrErc20(..., 1_000_000, ...)`.
5. The IC minter scrapes the log, parses `value = 1_000_000`, and mints `1,000,000` ckERC20 tokens to the depositor.
6. The depositor now holds `1,000,000` ckERC20 but the minter only holds `990,000` ERC-20 tokens — a `10,000`-unit deficit per deposit.
7. Repeating this inflates the ckERC20 supply unboundedly relative to the actual ERC-20 collateral, eventually making the bridge insolvent for that token. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
