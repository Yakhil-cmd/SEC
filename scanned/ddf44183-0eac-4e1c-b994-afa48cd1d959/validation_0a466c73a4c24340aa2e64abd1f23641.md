### Title
Truncation-Per-Iteration Accumulation in Type3 Node Provider Reward Loop Causes Systematic Underpayment - (File: `rs/registry/node_provider_rewards/src/lib.rs`)

---

### Summary

In `calculate_rewards_v0`, the type3 node reward calculation truncates a floating-point product to `u64` on **every iteration** of the per-node loop. This is structurally identical to the BarnBridge `compound` function pattern: a truncating cast inside a loop causes the fractional remainder to be silently discarded at each step, and the total error grows linearly with the number of nodes. Node providers with many type3 nodes are systematically underpaid each monthly reward cycle.

---

### Finding Description

In `rs/registry/node_provider_rewards/src/lib.rs`, the `calculate_rewards_v0` function handles type3 node rewards with a per-node loop:

```rust
let reward_base = rate.xdr_permyriad_per_node_per_month as f64;
let mut np_coeff = *np_coefficients.get(&np_coefficients_key).unwrap_or(&1.0);
let dc_reward_coefficient_percent =
    rate.reward_coefficient_percent.unwrap_or(80) as f64 / 100.0;

let mut dc_reward = 0;
for i in 0..*node_count {
    let node_reward = (reward_base * np_coeff) as u64;  // ← truncation every iteration
    dc_reward += node_reward;
    np_coeff *= dc_reward_coefficient_percent;
}
``` [1](#0-0) 

On line 128, `(reward_base * np_coeff) as u64` truncates the floating-point product toward zero on every loop iteration. The fractional part (up to 1 xdr_permyriad unit) is discarded silently at each step. Because `np_coeff` is itself the product of repeated floating-point multiplications by `dc_reward_coefficient_percent` (e.g., `0.98`), it is not exactly representable in binary f64, so the truncation error is non-zero for most iterations. The total accumulated error across `node_count` iterations is bounded by `node_count` xdr_permyriad units.

The correct approach — accumulate in floating-point and truncate only once at the end — is not used here. This is the same structural defect as the BarnBridge `compound` function: division/truncation inside the loop rather than deferred to the end.

This function is called from two production paths:

- `rs/registry/canister/src/get_node_providers_monthly_xdr_rewards.rs` (Registry canister, legacy monthly rewards)
- `rs/node_rewards/canister/src/canister/mod.rs` (Node Rewards canister, performance-based rewards) [2](#0-1) [3](#0-2) 

The XDR reward total produced by `calculate_rewards_v0` is then passed to `get_node_provider_reward` in `rs/nns/governance/src/governance.rs`, which converts it to ICP e8s via a single integer division:

```rust
let amount_e8s = ((xdr_permyriad_reward as u128 * TOKEN_SUBDIVIDABLE_BY as u128)
    / xdr_permyriad_per_icp as u128) as u64;
``` [4](#0-3) 

The already-truncated XDR total is the input here, so the underpayment propagates directly into the minted ICP amount.

---

### Impact Explanation

**Vulnerability class**: Ledger conservation bug — systematic underpayment of node provider rewards.

Each monthly reward cycle, every node provider operating type3 nodes receives less ICP than the mathematically correct amount. The shortfall is bounded by `node_count` xdr_permyriad per node operator per month. For a node provider with 100 type3 nodes, the maximum accumulated truncation error is 100 xdr_permyriad ≈ 0.01 XDR ≈ 0.001 ICP per month. While the per-cycle magnitude is small, the error is systematic (always rounds down, never up), permanent (lost ICP is never recovered), and grows with the number of nodes. The discarded fractional amounts are not redistributed — they simply vanish from the reward pool.

---

### Likelihood Explanation

This executes on every monthly node provider reward cycle. The NNS governance canister calls `get_monthly_node_provider_rewards` or `get_node_providers_rewards` on a fixed schedule, which ultimately invokes `calculate_rewards_v0`. Any node provider with type3 nodes is affected every cycle. No special conditions are required; the bug is always active. [5](#0-4) 

---

### Recommendation

Accumulate the reward in `f64` (or preferably `rust_decimal::Decimal`, as already used in SNS reward calculations) and perform the truncating cast to `u64` **once** after the loop completes:

```rust
let mut dc_reward_f64: f64 = 0.0;
for _ in 0..*node_count {
    dc_reward_f64 += reward_base * np_coeff;
    np_coeff *= dc_reward_coefficient_percent;
}
let dc_reward = dc_reward_f64 as u64;
```

This eliminates the per-iteration truncation and reduces the total error to a single rounding at the end, matching the recommendation from the external report to "always perform divisions as late as possible."

---

### Proof of Concept

Using the values from the existing test in `rs/registry/node_provider_rewards/src/lib.rs`:

- `reward_base = 27_491_250`, `dc_reward_coefficient_percent = 0.98`, `node_count = 3`

**Current behavior (truncation per iteration)**:
- Node 0: `floor(27491250 * 1.0) = 27491250`
- Node 1: `floor(27491250 * 0.98) = 26941425`
- Node 2: `floor(27491250 * 0.9604) = 26402596`
- Total: `80835271`

**Correct behavior (single truncation at end)**:
- Sum in f64: `27491250.0 + 26941425.0 + 26402596.5 = 80835271.5`
- `floor(80835271.5) = 80835271`

For 3 nodes the difference is negligible (0.5 xdr_permyriad). With 100 nodes and coefficient 0.98, the accumulated truncation error reaches ~50 xdr_permyriad ≈ 0.005 XDR per node operator per month. The error is always negative (underpayment) and scales linearly with `node_count`. [6](#0-5)
