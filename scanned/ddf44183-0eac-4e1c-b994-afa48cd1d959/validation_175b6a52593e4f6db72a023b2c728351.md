### Title
SNS Token Precision Loss Permanently Traps Tokens in Swap Canister - (File: `rs/sns/swap/src/swap.rs`)

### Summary
The `Swap::scale()` function in the SNS Swap canister uses integer (floor) division to compute each participant's SNS token allocation. Because Rust integer division always truncates, the sum of all per-participant allocations is strictly less than the total SNS tokens offered whenever the ICP contributions do not divide evenly. The resulting "change" tokens remain permanently locked in the Swap canister's SNS ledger account. No withdrawal or redistribution mechanism exists.

### Finding Description
In `create_sns_neuron_recipes()`, each participant's SNS token share is computed by `Swap::scale()`: [1](#0-0) 

The formula `(amount_icp_e8s * total_sns_e8s) / total_icp_e8s` is evaluated with 128-bit integer arithmetic, which truncates (floors) the result. When this is applied independently to every buyer and every Neurons' Fund neuron, the accumulated truncation means:

```
sum_over_all_participants(scale(participant_i_icp, total_sns, total_icp))
  ≤  total_sns_offered  (strict inequality whenever any division is inexact)
```

The code itself acknowledges the discrepancy by logging it as "change": [2](#0-1) 

After `sweep_sns()` transfers the recipe amounts to participants, the "change" tokens remain in the Swap canister's **default account** on the SNS ledger. The Swap canister exposes no SNS-token withdrawal endpoint. `error_refund_icp` only handles ICP: [3](#0-2) 

The SNS governance treasury subaccount is distinct from the Swap canister's account, so a `TransferSnsTreasuryFunds` governance proposal cannot reach these tokens either.

The integration test that asserts `swap_canister_balance_sns_e8s == 0` after finalization: [4](#0-3) 

passes only because the test is constructed with participant amounts that divide evenly. In production, with arbitrary ICP contributions, the remainder is non-zero and irrecoverable.

### Impact Explanation
SNS tokens are permanently locked in the Swap canister after every committed swap where ICP contributions do not divide evenly into the SNS token supply. The maximum stuck amount per swap is `(N − 1)` e8s, where N is the number of participants. These tokens are irrecoverable: the Swap canister has no SNS-token withdrawal path, and the SNS governance canister cannot reach the Swap canister's ledger account. This is a direct ledger conservation violation — tokens that were deposited into the Swap canister for distribution to participants are instead permanently destroyed from the perspective of all participants and the SNS treasury.

### Likelihood Explanation
Exact divisibility of arbitrary ICP contribution amounts into the SNS token supply is statistically negligible. Every real-world SNS swap with multiple participants will produce a non-zero remainder. The entry path is fully unprivileged: any principal can participate by calling `refresh_buyer_tokens`, and `finalize_swap` (which triggers `create_sns_neuron_recipes` and `sweep_sns`) can be called by any principal once the swap is committed. [5](#0-4) [6](#0-5) 

### Recommendation
1. After `sweep_sns()` completes successfully, query the Swap canister's remaining SNS ledger balance and transfer it to the SNS governance treasury subaccount (or to the largest contributor) to ensure full token conservation.
2. Alternatively, assign the remainder to the last (or largest) participant's recipe before `sweep_sns()` runs, so that `sum(recipes) == sns_being_offered_e8s` exactly — analogous to how `apportion_approximately_equally` distributes remainders: [7](#0-6) 

### Proof of Concept
**Setup:**
- 3 direct participants each contribute exactly 1 ICP (100,000,000 e8s); total ICP = 300,000,000 e8s.
- SNS tokens offered: 10 SNS = 1,000,000,000 e8s (`sns_token_e8s`).

**Execution of `scale()` per participant:**
```
scale(100_000_000, 1_000_000_000, 300_000_000)
  = (100_000_000 × 1_000_000_000) / 300_000_000
  = 100_000_000_000_000_000 / 300_000_000
  = 333_333_333  (truncated; exact = 333_333_333.333…)
```

**Total distributed:** `3 × 333_333_333 = 999_999_999` e8s

**Tokens permanently stuck in Swap canister:** `1_000_000_000 − 999_999_999 = 1` e8s

This 1 e8s (0.00000001 SNS token) remains in the Swap canister's SNS ledger account after `sweep_sns()` completes. No subsequent call can recover it. With 1,000 participants, up to 999 e8s can be permanently lost per swap. The "change" is logged but silently discarded: [8](#0-7)

pub fn apportion_approximately_equally(total: u64, len: u64) -> Result<Vec<u64>, String> {
    let quotient = total
        .checked_div(len)
        .ok_or_else(|| format!("Unable to divide total={total} by len={len}"))?;
    let remainder = total % len; // For unsigned integers, % cannot overflow.

    // So far, we have only apportioned quotient * len. To reach the desired
    // total, we must still somehow add remainder (per Euclid's Division
    // Theorem). That is accomplished right after this.
    let mut result = vec![quotient; len as usize];

    // Divvy out the remainder: Starting from the last element, increment
    // elements by 1. The number of such increments performed here is remainder,
    // bringing our total back to the desired amount.
    if remainder >= result.len() as u64 {
        return Err(format!("Could not apportion {total} into {len} pieces"));
    }
    let mut iter_mut = result.iter_mut();
    for _ in 0..remainder {
        let element: &mut u64 = iter_mut
            .next_back()
            // We can prove that this will not panic:
            // The number of iterations of this loop is total % len.
            // This must be < len (by Euclid's Division Theorem).
            // Thus, the number of iterations that this loop goes through is < len.
            // Thus, the number of times next_back is called is < len.
            // next_back only returns None after len calls.
            // Therefore, next_back does not return None here.
            // Therefore, this expect will never panic.
            .ok_or_else(
                || format!("Ran out of elements to increment. total={total}, len={len}",),
            )?;

        // This cannot overflow because the result must be <= total. Thus, this
        // will not panic.
        *element = element.checked_add(1).ok_or_else(|| {
            format!("Incrementing element by 1 resulted in overflow. total={total}, len={len}",)
        })?;
    }

    Ok(result)

    fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
        assert!(amount_icp_e8s <= u64::from(total_icp_e8s));
        // Note that the multiplication cannot overflow as both factors fit in 64 bits.
        let r = (amount_icp_e8s as u128)
            .saturating_mul(total_sns_e8s as u128)
            .div(NonZeroU128::from(total_icp_e8s));
        // This follows logically from the initial assert `amount_icp_e8s <= total_icp_e8s`.
        assert!(r <= u64::MAX as u128);
        r as u64
    }
