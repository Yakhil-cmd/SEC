### Title
Single Principal Monopolizes All `MAX_CONCURRENT` Guard Slots via Distinct Subaccounts in `retrieve_btc_with_approval` — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

---

### Summary

The `Guard<RetrieveBtcUpdates>` is keyed on the full ICRC-1 `Account` (principal + subaccount), not on the principal alone. A single unprivileged caller can therefore hold all 100 `MAX_CONCURRENT` guard slots simultaneously by submitting 100 concurrent `retrieve_btc_with_approval` calls each with a distinct `from_subaccount`, completely locking out every other user for the duration of those async calls, and — across repeated waves — exhausting the global `MAX_CONCURRENT_PENDING_REQUESTS` (5000) queue.

---

### Finding Description

**Guard key is the full `Account`:** [1](#0-0) 

The `Guard<PR>` struct stores an `Account` (owner + subaccount). The admission check is: [2](#0-1) 

- `AlreadyProcessing` fires only if the **exact same** `Account` is already present.
- `TooManyConcurrentRequests` fires only when the set reaches `MAX_CONCURRENT = 100`.

Because the set is keyed on the full `Account`, a single principal holding 100 different subaccounts occupies 100 distinct set entries — all 100 slots — without ever triggering `AlreadyProcessing`.

**`retrieve_btc_with_approval` constructs the guard key from `from_subaccount`:** [3](#0-2) 

The caller fully controls `args.from_subaccount`, so they can trivially supply 100 distinct values.

**The queue exhaustion check comes after guard acquisition:** [4](#0-3) 

Each successful call appends a `RetrieveBtcRequest` to the pending queue. After the guard drops (function returns), the slot is freed and the next wave can begin.

**The codebase's own test confirms the bypass is possible:** [5](#0-4) 

The test explicitly constructs `MAX_CONCURRENT` guards for the **same principal** using different subaccounts and asserts they all succeed — proving the design allows one principal to saturate the global limit.

**Contrast with `retrieve_btc` (non-approval path):** that function always passes `subaccount: None`, so the same principal can only hold one slot there. [6](#0-5) 

---

### Impact Explanation

**Phase 1 — Concurrent DoS (no economic cost):**
While 100 in-flight calls are executing (BTC checker inter-canister call + ICRC-2 `transfer_from`), the guard set is full. Every other user's `retrieve_btc_with_approval` call returns `TemporarilyUnavailable`. The attacker only needs ICRC-2 approvals in place; no ckBTC needs to be burned yet (the burn happens inside the guarded async section).

**Phase 2 — Queue exhaustion (economic cost = fees only):**
Each completed wave of 100 calls adds 100 entries to `retrieve_btc_accounts` (pending queue). After 50 waves the queue reaches `MAX_CONCURRENT_PENDING_REQUESTS = 5000`. The attacker recovers their ckBTC as BTC; the net cost is only miner fees and ledger fees. Once the queue is full, every user (including legitimate ones) receives `TemporarilyUnavailable("too many pending retrieve_btc requests")` until the backlog drains — which depends on the Bitcoin network and the minter's processing rate.

---

### Likelihood Explanation

The guard bypass requires no privileged access, no key material, and no protocol-level attack. It is executable by any ICRC-2 account holder. The economic barrier (holding enough ckBTC for 5000 minimum-amount withdrawals) is non-trivial but not prohibitive for a motivated attacker, especially since the capital is returned as BTC. The concurrent-slot monopolization (Phase 1) has essentially zero cost.

---

### Recommendation

1. **Key the guard on `caller` (principal) only**, not on the full `Account`. One in-flight `retrieve_btc_with_approval` per principal is sufficient.
2. **Add a per-principal cap on pending queue entries** — count how many `RetrieveBtcRequest` entries in the pending queue belong to a given principal and reject if it exceeds a reasonable per-user limit before accepting a new request.
3. Alternatively, track the guard as `BTreeMap<Principal, usize>` (count of in-flight calls per principal) and cap it at a small value (e.g., 1 or 5).

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
