### Title
Spawn-Neuron Minimum-Stake Guard Uses Hardcoded 5% Worst-Case While Mission 70 Modulation Can Reach -10% — (`rs/nns/governance/src/governance.rs`)

### Summary

The `spawn_neuron` function gates spawning on a hardcoded 5% worst-case modulation check, but `maybe_spawn_neurons` applies the actual maturity modulation (which under Mission 70 can reach -10%). A neuron owner whose maturity satisfies the 5% guard but not the 10% guard can successfully call `spawn_neuron`, wait 7 days, and have `maybe_spawn_neurons` mint a child neuron with `cached_neuron_stake_e8s < neuron_minimum_stake_e8s`.

### Finding Description

**Guard in `spawn_neuron`** (line 2666):

```rust
let least_possible_stake = (maturity_to_spawn as f64 * (1_f64 - 0.05)) as u64;
if least_possible_stake < economics.neuron_minimum_stake_e8s {
    return Err(...)
}
```

This hardcodes a 5% worst-case, matching the standard `MIN_MATURITY_MODULATION_PERMYRIAD = -500` defined in `maturity_modulation/mod.rs`. [1](#0-0) [2](#0-1) 

**`apply_maturity_modulation` enforces no bounds** — it accepts any `maturity_modulation_basis_points: i32` and applies it directly:

```rust
let adjusted = saturating_add_or_subtract_u128_i32(BASIS_POINTS_PER_UNITY, maturity_modulation_basis_points);
let modulated = amount_e8s * adjusted / BASIS_POINTS_PER_UNITY;
``` [3](#0-2) 

**Mission 70 constant** (`MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70`) is referenced in `rs/nns/governance/src/governance.rs` and `rs/nns/governance/src/timer_tasks/update_icp_xdr_rate_related_data.rs`. The question asserts its value is `-1000` permyriad (-10%). I was unable to read the exact numeric value from the index, which is a **critical uncertainty** in this analysis. [4](#0-3) 

**The structural mismatch is real if the premise holds:**

| Condition | Value |
|---|---|
| `spawn_neuron` worst-case guard | 5% (`1_f64 - 0.05`) |
| Standard `MIN_MATURITY_MODULATION_PERMYRIAD` | -500 permyriad (5%) |
| Claimed Mission 70 minimum | -1000 permyriad (10%) |

If `MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70 = -1000` and `maybe_spawn_neurons` passes this value to `apply_maturity_modulation`, then any neuron with maturity in the band:

```
floor(min_stake / 0.95) + 1  <=  maturity  <  floor(min_stake / 0.90) + 1
```

passes `spawn_neuron`'s guard but produces a child neuron with `stake < neuron_minimum_stake_e8s` after minting.

### Impact Explanation

Systematic creation of sub-minimum-stake neurons for every pending spawn in the affected maturity band when Mission 70 modulation is at its minimum. This violates the protocol invariant `cached_neuron_stake_e8s >= neuron_minimum_stake_e8s` for spawned neurons and breaks ICP conservation accounting across the batch.

### Likelihood Explanation

The attacker path is fully unprivileged: any neuron owner can call `spawn_neuron` via ingress, wait for the 7-day dissolve delay, and `maybe_spawn_neurons` runs automatically as a timer task. No privileged access is required. The precondition (Mission 70 modulation at -1000 permyriad) is a protocol-level state, not attacker-controlled, but when it occurs the exploit is automatic for all neurons in the band.

### Critical Caveat

I was unable to confirm the exact numeric value of `MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70` or read the full body of `maybe_spawn_neurons` from the index. **If `MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70` is actually `-500` (matching the standard range), the `spawn_neuron` guard is correct and there is no vulnerability.** The finding is contingent on the Mission 70 constant being `-1000` as claimed.

### Recommendation

Update the `spawn_neuron` guard to use `MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70` (or the actual deployed minimum) instead of the hardcoded `0.05`:

```rust
let worst_case_factor = 1.0 - (MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70.abs() as f64 / 10_000.0);
let least_possible_stake = (maturity_to_spawn as f64 * worst_case_factor) as u64;
```

Alternatively, add a secondary minimum-stake check inside `maybe_spawn_neurons` before minting, and handle the failure gracefully (e.g., return maturity to parent neuron).

### Proof of Concept

```
min_stake = 100_000_000 e8s (1 ICP)
maturity   = floor(100_000_000 / 0.95) + 1 = 105_263_159 e8s

spawn_neuron check:
  least_possible_stake = (105_263_159 * 0.95) as u64 = 100_000_001 >= 100_000_000  ✓ PASSES

maybe_spawn_neurons with modulation = -1000 permyriad:
  apply_maturity_modulation(105_263_159, -1000)
  = 105_263_159 * (10_000 - 1_000) / 10_000
  = 105_263_159 * 9_000 / 10_000
  = 94_736_843 e8s  <  100_000_000  ✗ VIOLATES INVARIANT
```

---

**Verdict:** This is a **conditionally valid vulnerability**. The structural mismatch between the hardcoded 5% guard in `spawn_neuron` and a potential -10% Mission 70 modulation is real and demonstrable from the production code. Confirmation requires verifying that `MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70 = -1000` and that `maybe_spawn_neurons` passes this value to `apply_maturity_modulation` without a secondary minimum-stake guard.

```rust
pub fn apply_maturity_modulation(
    amount_maturity_e8s: u64,
    maturity_modulation_basis_points: i32,
) -> Result<u64, String> {
    let amount_e8s = u128::from(amount_maturity_e8s);

    let adjusted_maturity_modulation_basis_points = saturating_add_or_subtract_u128_i32(
        BASIS_POINTS_PER_UNITY,
        maturity_modulation_basis_points,
    );

    let modulated_amount_e8s: u128 = amount_e8s
        .checked_mul(adjusted_maturity_modulation_basis_points)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?
        .checked_div(BASIS_POINTS_PER_UNITY)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?;

    u64::try_from(modulated_amount_e8s).map_err(|err| err.to_string())
}
```
