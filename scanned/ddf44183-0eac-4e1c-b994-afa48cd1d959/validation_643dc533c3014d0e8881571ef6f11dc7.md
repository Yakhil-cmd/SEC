### Title
Unbounded Iteration in SNS Governance `distribute_rewards` Over All Proposal Ballots and Neurons Without Instruction-Limit Guard - (File: `rs/sns/governance/src/governance.rs`)

### Summary
The SNS governance canister's `distribute_rewards` function iterates synchronously over all `considered_proposals`, and for each proposal over all `ballots` (one per eligible neuron), and then over all `neuron_id_to_reward_shares`, with no instruction-limit check at any point. With a sufficiently large neuron set, this single-message execution will exceed the IC instruction limit, causing the periodic timer task to trap and permanently blocking SNS voting reward distribution.

### Finding Description

`distribute_rewards` in `rs/sns/governance/src/governance.rs` performs three nested/sequential unbounded iterations in one synchronous call:

**Phase 1 – Accumulate reward shares** (lines 5894–5930): iterates over every `considered_proposal`, and for each proposal iterates over every `ballot` (one per eligible neuron at proposal creation time):

```rust
for proposal_id in &considered_proposals {
    if let Some(proposal) = self.get_proposal_data(*proposal_id) {
        for (voter, ballot) in &proposal.ballots {
            ...
        }
    }
}
``` [1](#0-0) 

**Phase 2 – Distribute maturity** (lines 5954–5997): iterates over every entry in `neuron_id_to_reward_shares` (one per voting neuron):

```rust
for (neuron_id, neuron_reward_shares) in neuron_id_to_reward_shares {
    let neuron: &mut Neuron = match self.get_neuron_result_mut(&neuron_id) { ... };
    ...
}
``` [2](#0-1) 

**Phase 3 – Settle proposals** (lines 6013–6081): iterates over `considered_proposals` again, calling `process_proposal` for each: [3](#0-2) 

None of these loops contain any call to `is_message_over_threshold`, `is_over_instructions_limit`, or any equivalent guard. The entire function runs atomically in one message execution.

`distribute_rewards` is called from `run_periodic_tasks` (line 5513), which is scheduled as a repeating timer at `RUN_PERIODIC_TASKS_INTERVAL`: [4](#0-3) [5](#0-4) 

**Contrast with NNS governance**, which was explicitly fixed for this exact problem. The NNS `distribute_voting_rewards_to_neurons` now schedules a `RewardsDistributionStateMachine` that processes one neuron at a time and checks `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` after each neuron: [6](#0-5) 

The NNS CHANGELOG explicitly documents this fix: *"Distribute rewards is moved to timer, and has a mechanism to distribute in batches in multiple messages."* [7](#0-6) 

The SNS governance has received no equivalent fix.

### Impact Explanation

When the total work (number of proposals × neurons per proposal) exceeds the IC per-message instruction limit (~40 billion instructions), the timer callback traps. Because the timer is a repeating interval timer and the state is not checkpointed mid-loop, every subsequent firing of the timer will also trap on the same oversized distribution. **SNS voting reward distribution becomes permanently broken**: no neuron in the SNS ever receives maturity again, and the `latest_reward_event` is never updated, causing the reward purse to roll over indefinitely without being paid out.

### Likelihood Explanation

SNS neuron creation is permissionless. Any user can stake SNS tokens and create neurons. A popular SNS (e.g., one with tens of thousands of participants) will naturally accumulate enough neurons that the O(proposals × neurons) work in `distribute_rewards` exceeds the instruction limit. The IC instruction limit per message is 40 billion; with 100,000 neurons and even 10 proposals, the ballot iteration alone (10 × 100,000 = 1,000,000 entries, each requiring a `HashMap` lookup and arithmetic) combined with the `get_neuron_result_mut` calls in Phase 2 can plausibly exhaust this budget. No adversarial action is required — organic growth of a successful SNS is sufficient.

### Recommendation

Apply the same fix used in NNS governance:

1. Replace the synchronous `distribute_rewards` loop with a state-machine approach (analogous to `RewardsDistributionStateMachine`) that persists partially-completed distributions across messages.
2. Check an instruction-limit guard (e.g., `is_message_over_threshold`) after each neuron processed in Phase 2, and break out of the loop, re-scheduling the remainder for the next timer tick.
3. Separate Phase 1 (ballot accumulation) from Phase 2 (maturity crediting) so each can be interrupted and resumed independently.

### Proof of Concept

1. Deploy an SNS with `voting_rewards_parameters` enabled.
2. Create N neurons (e.g., N = 200,000) by having many users stake SNS tokens.
3. Submit and pass a governance proposal so that `considered_proposals` is non-empty.
4. Wait for `should_distribute_rewards()` to return `true`.
5. Observe that the next `run_periodic_tasks` timer invocation traps with an instruction-limit exceeded error.
6. Observe that `latest_reward_event` is never updated and no neuron's `maturity_e8s_equivalent` increases, confirming permanent reward distribution failure.

## Changed

* Voting Rewards will be scheduled by a timer instead of by heartbeats.
* Unstaking maturity task will be processing up to 100 neurons in a single message, to avoid
  exceeding the instruction limit in a single execution.
* Voting Rewards will be distributed asynchronously in the background after being calculated.
    * This will allow rewards to be compatible with neurons being stored in Stable Memory.
