### Title
Treasury Deposit 50% Balance Check Evaluated at Proposal Validation Time Enables Balance-Inflation Bypass and TOCTOU Safety Limit Violation - (`rs/sns/governance/src/extensions.rs`)

---

### Summary

In `validate_deposit_operation_impl`, the SNS governance canister fetches the **live treasury balance** at proposal validation time and enforces a 50% cap on deposit amounts. This check is **never re-evaluated at execution time** in `execute_treasury_manager_deposit`. Because ICRC-1 ledgers allow anyone to transfer tokens to any account, an unprivileged actor can inflate the treasury balance before proposal submission, causing the 50% check to pass for amounts that exceed 50% of the treasury's genuine pre-inflation balance. Separately, a concurrent governance proposal that reduces the treasury balance between validation and execution silently violates the same limit at execution time.

---

### Finding Description

`validate_deposit_operation_impl` in `rs/sns/governance/src/extensions.rs` (lines 276–321) queries the live SNS and ICP treasury balances and rejects any deposit proposal whose requested amounts exceed 50% of those balances:

```rust
if sns_requested > sns_balance.checked_div(2).unwrap() {
    return Err(format!(
        "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
    ));
}
if icp_requested > icp_balance.checked_div(2).unwrap() {
    return Err(format!(
        "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
    ));
}
``` [1](#0-0) 

This check runs only during proposal validation. The execution path, `execute_treasury_manager_deposit` (lines 1545–1610), receives the already-validated `ValidatedDepositOperationArg` and immediately calls `approve_treasury_manager` and then `deposit` on the treasury manager canister — **with no re-check of the 50% limit**: [2](#0-1) 

This creates two exploitable conditions:

**1. Balance-inflation bypass (unprivileged entry path)**

ICRC-1 ledgers permit any principal to transfer tokens to any account. The SNS treasury subaccount is a deterministic ICRC-1 account derived from the governance canister ID: [3](#0-2) 

An attacker can transfer SNS tokens directly to this account before a deposit proposal is submitted. The inflated balance causes the 50% check to pass for a deposit amount that exceeds 50% of the treasury's pre-inflation balance. After the governance voting period, the proposal executes and deposits the inflated-approved amount into the DEX. The attacker's donated tokens remain in the treasury and are not deposited, but the governance-approved deposit now represents more than 50% of the treasury's original holdings.

**2. TOCTOU — balance decrease between validation and execution**

The code comment itself acknowledges the race:

```rust
// Fail if either is asking for more than 50% of current balance.  The balance could have changed
// since the proposal was created, and we don't assume that the proposal should work
``` [4](#0-3) 

If a concurrent governance proposal (e.g., `TransferSnsTreasuryFunds`) reduces the treasury balance between validation and execution, the deposit amount — validated as ≤ 50% of the old balance — may now exceed 50% of the current balance. Because `execute_treasury_manager_deposit` never re-checks, the deposit proceeds silently past the intended limit.

The `treasury_manager.did` interface itself acknowledges the broader slippage risk in this flow:

```
// Known Security Risks:
// ====================
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
``` [5](#0-4) 

---

### Impact Explanation

The 50% safety cap — the only on-chain guard preventing a single governance proposal from depositing the majority of the SNS treasury into a DEX — can be bypassed. An attacker who inflates the treasury balance before proposal submission can cause governance to approve a deposit of, for example, 60% of the real treasury. Once the ICRC-2 approval is granted and `deposit` is called on the treasury manager canister, the funds are committed to the DEX. Recovery requires a separate withdrawal governance proposal, during which the DEX position is exposed to price movement and front-running. The `proposal.rs` warning confirms this exposure:

> "This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running or sandwich attacks." [6](#0-5) 

---

### Likelihood Explanation

The balance-inflation path requires only an ICRC-1 token transfer to the treasury account — an operation available to any token holder with no governance role. The attacker does not need to hold voting power; they only need to coordinate with (or be) a governance participant who submits the deposit proposal after the inflation. SNS governance proposals require a voting period of days, giving ample time for the inflated balance to be observed and the proposal to pass. The TOCTOU path can occur naturally whenever two treasury-affecting proposals are in flight simultaneously.

---

### Recommendation

Re-evaluate the 50% limit at execution time inside `execute_treasury_manager_deposit`, querying the live treasury balance immediately before calling `approve_treasury_manager`. This mirrors the pattern used in the original check and closes both the inflation bypass and the TOCTOU window. Alternatively, snapshot the treasury balance at proposal creation time and bind the execution to that snapshot, rejecting execution if the current balance has fallen below the snapshot value.

---

### Proof of Concept

1. SNS treasury holds 100 SNS tokens (real balance).
2. Attacker calls `icrc1_transfer` on the SNS ledger, sending 30 SNS tokens to the treasury subaccount. Treasury balance is now 130 SNS.
3. A governance participant submits a deposit proposal with `treasury_allocation_sns_e8s = 60_000_000` (60 SNS).
4. `validate_deposit_operation_impl` fetches balance = 130 SNS; `60 ≤ 130/2 = 65` → check passes.
5. Proposal enters the voting period and passes.
6. `execute_treasury_manager_deposit` calls `approve_treasury_manager` granting the treasury manager an ICRC-2 allowance of 60 SNS, then calls `deposit`.
7. 60 SNS tokens are deposited into the DEX — representing 60% of the original 100 SNS treasury, violating the intended 50% cap.
8. The attacker's 30 donated SNS tokens remain in the treasury and are not deposited. [7](#0-6) [8](#0-7)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;
