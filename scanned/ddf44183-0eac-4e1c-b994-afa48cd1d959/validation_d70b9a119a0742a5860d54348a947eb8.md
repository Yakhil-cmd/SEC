### Title
SNS Governance Distributes Equal Voting Rewards Regardless of Proposal Criticality - (File: rs/sns/governance/src/governance.rs)

---

### Summary

The SNS governance `distribute_rewards` function assigns identical reward weight to every proposal regardless of its topic or criticality level. Critical proposals (requiring 67% of exercised voting power to pass) receive the same reward weight as non-critical proposals (requiring only 50%), creating misaligned incentives that discourage active participation in the most consequential governance decisions and allow reward dilution through low-effort proposal spam.

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `distribute_rewards` function accumulates reward shares purely based on raw `ballot.voting_power` with no topic-based multiplier: [1](#0-0) 

```rust
let reward_shares = i2d(ballot.voting_power);
*neuron_id_to_reward_shares
    .entry(neuron_id)
    .or_insert_with(|| dec!(0)) += reward_shares;
```

The SNS has seven topics with explicitly differentiated criticality levels: [2](#0-1) 

- **Critical**: `DaoCommunitySettings`, `TreasuryAssetManagement`, `CriticalDappOperations` (and custom critical proposals)
- **Non-critical**: `SnsFrameworkManagement`, `DappCanisterManagement`, `ApplicationBusinessLogic`, `Governance`

Critical proposals require **67% of exercised voting power** and **20% of total voting power** to pass, while non-critical proposals require only **50% of exercised** and **3% of total**: [3](#0-2) 

Despite this significant structural difference in governance importance and participation requirements, the reward system treats all proposals identically.

This stands in direct contrast to the NNS governance, which explicitly implements topic-based reward weights via `Topic::reward_weight()`: [4](#0-3) 

```rust
pub fn reward_weight(&self) -> f64 {
    match self {
        Topic::Governance => 20.0,
        Topic::SnsAndCommunityFund => 20.0,
        Topic::ExchangeRate => 0.01,
        _ => 1.0,
    }
}
```

The NNS then applies this weight in `sum_weighted_voting_power`: [5](#0-4) 

The SNS `distribute_rewards` has no equivalent weighting step.

---

### Impact Explanation

**Reward dilution via low-effort proposal spam**: An SNS neuron holder can submit many non-critical proposals (e.g., `Motion` under the `Governance` topic) and vote on all of them. Each vote accumulates `ballot.voting_power` reward shares. A neuron voting on 20 non-critical proposals earns 20× the reward shares of a neuron that votes only on 1 critical `TreasuryAssetManagement` proposal — despite the latter requiring far more scrutiny and carrying far greater protocol risk.

**Governance apathy on critical proposals**: Because there is no reward premium for voting on critical proposals, neurons following automated liquid democracy have no financial incentive to override their followees and cast independent votes on critical decisions. This reduces the effective participation rate on the most consequential proposals.

**Protocol value leakage**: The reward pool is distributed proportionally to reward shares. When low-effort non-critical proposals dominate the reward period, the fraction of the reward pool going to neurons that carefully evaluate critical proposals shrinks. The protocol mints ICP/SNS tokens to reward governance participation, but the incentive structure does not align that minting with the governance actions that most benefit the protocol.

---

### Likelihood Explanation

- This affects **every SNS deployment** that has `voting_rewards_parameters` set.
- The entry path requires only an unprivileged SNS neuron holder — no special role, no key compromise.
- The effect is amplified by liquid democracy (following): followers automatically vote on all proposals, so a neuron that follows on all topics accumulates reward shares on every proposal without any additional effort.
- The SNS topic system was introduced recently (2025), making this a newly created asymmetry that has not been compensated for in the reward logic.

---

### Recommendation

Implement topic-based reward weights in the SNS `distribute_rewards` function, analogous to the NNS `Topic::reward_weight()` mechanism. Specifically:

1. Add a `reward_weight()` method to `pb::Topic` in the SNS that returns a higher multiplier for critical topics (e.g., `TreasuryAssetManagement`, `CriticalDappOperations`, `DaoCommunitySettings`) and a lower or unit multiplier for non-critical topics.
2. In `distribute_rewards`, multiply `ballot.voting_power` by the proposal's topic reward weight before accumulating into `neuron_id_to_reward_shares`, mirroring the NNS pattern in `sum_weighted_voting_power`.
3. Expose the reward weights as part of `NervousSystemParameters` so SNS DAOs can tune them via governance proposal.

---

### Proof of Concept

**Setup**: An SNS with voting rewards enabled. Two neurons, A and B, each with 1,000 voting power. One reward period contains 11 proposals: 10 non-critical `Motion` proposals (Governance topic) and 1 critical `TransferSnsTreasuryFunds` proposal (TreasuryAssetManagement topic).

**Behavior under current code**:

- Neuron A votes on all 10 non-critical Motion proposals → accumulates `10 × 1,000 = 10,000` reward shares.
- Neuron B votes only on the 1 critical treasury proposal → accumulates `1 × 1,000 = 1,000` reward shares.
- Total reward shares: `11,000`.
- Neuron A receives `10,000 / 11,000 ≈ 90.9%` of the reward purse.
- Neuron B receives `1,000 / 11,000 ≈ 9.1%` of the reward purse.

The neuron that participated in the most consequential governance decision (treasury transfer, which requires 67% supermajority) receives ~10× less reward than the neuron that voted on trivial motion proposals.

**Root cause in code**: [6](#0-5) 

The loop at line 5894–5931 iterates over all proposals and ballots, accumulating `i2d(ballot.voting_power)` with no topic multiplier. The NNS equivalent at `rs/nns/governance/src/proposals/mod.rs:373` multiplies by `reward_weight` before accumulating. [7](#0-6)

    pub fn is_critical(&self) -> bool {
        // Handled explicitly to avoid any doubts.
        //
        // We used to fall back to non-critical proposal criticality for backward compatibility,
        // since when custom proposals were introduced, they were not categorized into topics
        // and were all considered non-critical. Since the SNS now enforces that all newly submitted
        // proposals are have topics, their criticality is guaranteed to be explicitly defined
        // (by the topic). Note that for native proposals, the criticality needs to be defined
        // via the topic assigned statically in `Governance::topic_descriptions`. We take
        // some measures to enforce that all native functions have topics. If this assumption
        // is still somehow violated, we now err on the side of caution.
        if *self == Self::Unspecified {
            return true;
        }

        topic_descriptions()
            .iter()
            .any(|topic| *self == Self::from(topic.topic) && topic.is_critical)
    }

        let reward_weight = proposal.topic().reward_weight();

        // This is used in lieu of total_potential_voting_power. That is, this
        // gets used if proposal does not have total_potential_voting_power.
        // This fall back will only be used during a short transition period
        // when there is a backlog of "old" proposals that were created without
        // total_potential_voting_power, but all new proposals have it. In the
        // case of such legacy proposals, their total potential voting power
        // would have been equal to this anyway, so this is a sound substitute
        // for total_potential_voting_power.
        let mut total_ballots_voting_power = 0;

        for (neuron_id, ballot) in &proposal.ballots {
            total_ballots_voting_power += ballot.voting_power;

            // Don't reward neurons that did not actually vote. (Whereas, ALL
            // eligible neurons get an "empty" ballot when the proposal is first
            // created. An "empty" ballot is one where the vote field is set to
            // Unspecified.)
            let vote = Vote::try_from(ballot.vote).unwrap_or_else(|err| {
                println!(
                    "{}ERROR: Unrecognized Vote {} in ballot by neuron {} \
                     on proposal {:?}: {:?}",
                    LOG_PREFIX, ballot.vote, neuron_id, proposal.id, err,
                );
                Vote::Unspecified
            });
            if !vote.eligible_for_rewards() {
                continue;
            }

            // Increment neuron.
            *neuron_id_to_exercised_weighted_voting_power
                .entry(NeuronId { id: *neuron_id })
                .or_insert(0.0) += (ballot.voting_power as f64) * reward_weight;
        }

        // Increment global total.
        total_weighted_voting_power += reward_weight
            * proposal
                .total_potential_voting_power
                .unwrap_or(total_ballots_voting_power) as f64;
