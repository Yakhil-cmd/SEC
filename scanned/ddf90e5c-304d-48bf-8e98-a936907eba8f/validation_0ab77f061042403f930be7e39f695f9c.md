### Title
Snapshot Restore Blindly Destroys Canister Wasm Without Validating Snapshot Wasm Existence — (File: rs/state_manager/src/tip.rs)

---

### Summary

The `restore` function in `rs/state_manager/src/tip.rs` deletes the canister's current wasm file before verifying that the snapshot's wasm file exists on disk. `WasmFile::hardlink_file` in `rs/state_layout/src/state_layout.rs` silently returns `Ok(())` when the source file is absent. If the snapshot's wasm file is missing on disk at restore time, the canister's wasm is permanently destroyed with no error surfaced, leaving the on-disk state inconsistent with the in-memory state and causing deterministic execution divergence across subnet nodes.

---

### Finding Description

The `restore` function performs a non-atomic, non-validated restore:

```rust
// rs/state_manager/src/tip.rs, lines 792-793
canister_layout.wasm().try_delete_file()?;                                    // (1) destroys canister wasm
WasmFile::hardlink_file(&snapshot_layout.wasm(), &canister_layout.wasm())?;  // (2) silently succeeds if snapshot wasm missing
``` [1](#0-0) 

Step (1) unconditionally deletes the canister's current wasm file. Step (2) calls `WasmFile::hardlink_file`, which contains the following guard:

```rust
// rs/state_layout/src/state_layout.rs, lines 2748-2750
if !src_path.exists() {
    return Ok(());   // silent success — no error, no wasm restored
}
``` [2](#0-1) 

This silent-success path was designed for the symmetric `backup` direction (where a canister with no wasm legitimately produces a snapshot with no wasm file). However, in the `restore` direction the same silent-success path is reused after the canister's wasm has already been deleted, meaning a missing snapshot wasm file causes the restore to complete with `Ok(())` while leaving the canister with no wasm on disk.

The `restore` function is invoked from `flush_unflushed_checkpoint_ops` whenever a `LoadSnapshot` operation is dequeued:

```rust
// rs/state_manager/src/tip.rs, lines 710-711
UnflushedCheckpointOp::LoadSnapshot(canister_id, snapshot_id) => {
    restore(log, &tip_handler.tip(height)?, canister_id, snapshot_id)?;
``` [3](#0-2) 

The `LoadSnapshot` operation is enqueued by the execution layer's `load_canister_snapshot` management-canister endpoint, which validates snapshot existence only in the in-memory `ReplicatedState`, not on disk:

```rust
// rs/execution_environment/src/execution_environment.rs, lines 2955-2969
let snapshot_id = args.snapshot_id();
let snapshot_canister_id = snapshot_id.get_canister_id();
let snapshot_canister: Arc<CanisterState> =
    match state.canister_state_arc(&snapshot_canister_id) {
        Some(snapshot_canister) => snapshot_canister,
        None => { ... return error ... }
    };
``` [4](#0-3) 

The disk-level backup (`backup` function) runs asynchronously in the tip thread after the in-memory snapshot is committed. If the disk-level backup fails partway through (e.g., after copying page maps but before hardlinking the wasm), the snapshot directory is left without a wasm file while the in-memory `CanisterSnapshot` still holds a valid `wasm_binary` reference. A subsequent `load_canister_snapshot` call then:

1. Passes all in-memory validation (snapshot exists in `canister_snapshots`)
2. Enqueues `LoadSnapshot`
3. `restore` deletes the canister's current wasm
4. `hardlink_file` silently returns `Ok(())` (snapshot wasm absent on disk)
5. Canister has no wasm file on disk; in-memory state still references a wasm binary

---

### Impact Explanation

The canister's wasm module is permanently destroyed without any error being reported. The in-memory state diverges from the on-disk state: the in-memory `CanisterSnapshot` and the restored `CanisterState` both reference a wasm binary, but no wasm file exists on disk. When the state is next loaded from disk (e.g., after a node restart), `load_canister_state` will fail to find the wasm file, causing the node to crash or diverge from peers that still have the wasm file. This constitutes a **deterministic execution divergence** across subnet nodes: nodes that successfully wrote the snapshot wasm to disk will restore correctly, while nodes that experienced a partial backup failure will silently corrupt the canister and subsequently diverge.

---

### Likelihood Explanation

The snapshot's wasm file being absent on disk requires either a partial failure in the `backup` disk path (e.g., an I/O error after page maps are written but before the wasm hardlink is created) or disk corruption. The `backup` function itself uses the same silent-success `hardlink_file`, so a partial failure leaves no error trace. The `load_canister_snapshot` management-canister endpoint is reachable by any canister controller without elevated privileges, making the trigger path externally accessible. The disk-level failure is not directly attacker-controlled, placing likelihood at **Low-Medium**.

---

### Recommendation

1. In `restore`, validate that the snapshot's wasm file exists on disk **before** deleting the canister's current wasm file. Return a `LayoutError` if the snapshot wasm is absent but the in-memory snapshot references a wasm binary.
2. Consider making the restore atomic: copy snapshot files to a scratchpad directory first, verify all expected files are present, then atomically replace the canister directory.
3. In `WasmFile::hardlink_file`, distinguish between the `backup` context (where a missing source is legitimate) and the `restore` context (where a missing source is an error). A separate `hardlink_file_required` variant that returns an error on missing source would prevent silent data loss in the restore path.

---

### Proof of Concept

1. Canister controller calls `take_canister_snapshot` on a canister with a wasm module. The in-memory `CanisterSnapshot` is created with a valid `wasm_binary`.
2. The tip thread runs `backup`, which calls `WasmFile::hardlink_file(&canister_layout.wasm(), &snapshot_layout.wasm())`. An I/O error occurs after page maps are copied but before the wasm hardlink is created. The error is propagated up and logged as fatal, but the snapshot directory is left without a wasm file.
3. After recovery, the in-memory state still contains the snapshot (it was committed before the tip flush).
4. The canister controller calls `load_canister_snapshot`. The execution layer validates the snapshot exists in memory and enqueues `UnflushedCheckpointOp::LoadSnapshot`.
5. The tip thread calls `flush_unflushed_checkpoint_ops` → `restore`.
6. `restore` calls `canister_layout.wasm().try_delete_file()` — the canister's wasm is deleted.
7. `restore` calls `WasmFile::hardlink_file(&snapshot_layout.wasm(), &canister_layout.wasm())`.
8. `hardlink_file` checks `if !src_path.exists() { return Ok(()); }` — the snapshot wasm is absent, so it silently returns `Ok(())`.
9. `restore` returns `Ok(())`. No error is surfaced.
10. The canister now has no wasm file on disk. On the next node restart, loading the canister state from disk will fail or diverge from peers. [5](#0-4) [6](#0-5) [7](#0-6)

fn flush_unflushed_checkpoint_ops(
    log: &ReplicaLogger,
    tip_handler: &mut TipHandler,
    height: Height,
    operations: Vec<UnflushedCheckpointOp>,

    // This loop is not parallelized as there are combinations such as creating then restoring from a snapshot within the same flush.

fn restore<T>(
    log: &ReplicaLogger,
    layout: &CheckpointLayout<RwPolicy<T>>,
    canister_id: CanisterId,
    snapshot_id: SnapshotId,
) -> Result<(), LayoutError> {
    let canister_layout = layout.canister(&canister_id)?;
    let snapshot_layout = layout.snapshot(&snapshot_id)?;

    canister_layout.vmemory_0().delete_files()?;
    PageMapLayout::copy_or_hardlink_files(
        log,
        &snapshot_layout.vmemory_0(),
        &canister_layout.vmemory_0(),
    )?;
    canister_layout.stable_memory().delete_files()?;
    PageMapLayout::copy_or_hardlink_files(
        log,
        &snapshot_layout.stable_memory(),
        &canister_layout.stable_memory(),
    )?;
    canister_layout.wasm_chunk_store().delete_files()?;
    PageMapLayout::copy_or_hardlink_files(
        log,
        &snapshot_layout.wasm_chunk_store(),
        &canister_layout.wasm_chunk_store(),
    )?;

    canister_layout.wasm().try_delete_file()?;
    WasmFile::hardlink_file(&snapshot_layout.wasm(), &canister_layout.wasm())?;

    Ok(())
}

    pub fn hardlink_file<W>(src: &WasmFile<T>, dst: &WasmFile<W>) -> Result<(), LayoutError>
    where
        W: WritePolicy,
    {
        let src_path = src.raw_path();
        let dst_path = dst.raw_path();

        if !src_path.exists() {
            return Ok(());
        }

        #[cfg(debug_assertions)]
        {
            let src_metadata = src_path.metadata().map_err(|err| LayoutError::IoError {
                path: src_path.to_path_buf(),
                message: "Failed to read metadata".to_string(),
                io_err: err,
            })?;
            debug_assert!(src_metadata.permissions().readonly());
        }

        std::fs::hard_link(src_path, dst_path).map_err(|err| LayoutError::IoError {
            path: src_path.to_path_buf(),
            message: format!(
                "Failed to hardlink {src_path:?} to {dst_path:?} while making a canister snapshot",
            ),
            io_err: err,
        })?;
        Ok(())
    }
