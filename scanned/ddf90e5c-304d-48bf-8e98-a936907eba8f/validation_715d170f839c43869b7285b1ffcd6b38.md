### Title
Missing Slippage Protection in SNS Treasury Manager Withdraw Operation - (`rs/sns/governance/src/extensions.rs`)

### Summary

The SNS Treasury Manager framework's `withdraw` operation provides no mechanism for the SNS DAO to specify minimum expected token amounts when withdrawing liquidity from an external DEX custodian. The `WithdrawRequest` type carries no minimum-amount fields, and `ValidatedWithdrawOperationArg` explicitly rejects any arguments. After a governance proposal is approved (a process that takes days), any DEX participant can manipulate the pool price so that the SNS treasury receives far fewer tokens than the DAO voted to recover.

### Finding Description

The `TreasuryManager` API defines `WithdrawRequest` with only an optional destination account map and no minimum-amount fields:

```candid
type WithdrawRequest = record {
  withdraw_accounts : opt vec record { principal; Account };
};
``` [1](#0-0) 

The Rust-side validation in `ValidatedWithdrawOperationArg::try_from` actively rejects any non-empty argument, making it structurally impossible to pass slippage bounds through the governance proposal:

```rust
// For now, only allow empty arguments
// This ensures withdraw operations don't accept parameters yet
if original.value.is_some() {
    return Err("Withdraw operation does not accept arguments at this time".to_string());
}
``` [2](#0-1) 

The execution function `execute_treasury_manager_withdraw` calls `withdraw` on the extension canister and only checks whether the call itself succeeded or returned an error — it never inspects the returned `Balances` to verify that the treasury received at least a minimum amount of each token:

```rust
let balances = governance
    .env
    .call_canister(extension_canister_id, "withdraw", arg_blob)
    ...
    ?;

log!(INFO, "TreasuryManager.withdraw succeeded with response: {:?}", balances);
Ok(())
``` [3](#0-2) 

The codebase itself acknowledges the analogous slippage risk for deposits in two places. The `treasury_manager.did` file notes:

```
// Known Security Risks:
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
``` [4](#0-3) 

And the proposal rendering code warns about sandwich attacks on deposits: [5](#0-4) 

Neither warning nor protection is applied to the **withdraw** path.

### Impact Explanation

When the SNS DAO passes a `TreasuryManagerWithdraw` governance proposal, the proposal is queued and executed after the voting period (typically several days). During that window, any DEX participant can move the pool price so that the liquidity position becomes single-sided or heavily skewed. When `execute_treasury_manager_withdraw` fires, the treasury manager withdraws from the DEX at the manipulated price and returns whatever tokens result — with no on-chain check that the amounts meet any minimum. The SNS treasury (DAO-owned funds) can receive significantly fewer tokens than the DAO intended to recover, with no revert or error.

The `BalanceBook` returned by `withdraw` contains `treasury_owner`, `external_custodian`, and `fee_collector` balances, but `execute_treasury_manager_withdraw` discards this information entirely after logging it. [6](#0-5) 

### Likelihood Explanation

The SNS governance voting period creates a mandatory multi-day delay between proposal approval and execution. This is a well-known MEV/sandwich window. Any unprivileged DEX participant (canister caller or user) can execute large swaps against the pool during this window. The attack requires no privileged access, no key compromise, and no consensus-level corruption — only the ability to trade on the same DEX pool that the Treasury Manager extension uses. The `TreasuryManager` API is explicitly designed to interact with DEXes such as KongSwap (as shown in integration tests). [7](#0-6) 

### Recommendation

1. Add `min_token_amounts` fields to `WithdrawRequest` in `treasury_manager.did`, analogous to how `DepositRequest` carries `allowances`:

```candid
type WithdrawRequest = record {
  withdraw_accounts : opt vec record { principal; Account };
  min_token_amounts : opt vec record { principal; nat }; // ledger_id -> minimum amount
};
```

2. Remove the blanket rejection of withdraw arguments in `ValidatedWithdrawOperationArg::try_from` and instead parse and validate minimum-amount fields from the governance proposal arg. [8](#0-7) 

3. In `execute_treasury_manager_withdraw`, after the `withdraw` call succeeds, inspect the returned `Balances` and compare `treasury_owner` balances against the minimums specified in the proposal. Return a `GovernanceError` if any minimum is not met. [9](#0-8) 

### Proof of Concept

1. SNS DAO approves a `TreasuryManagerWithdraw` proposal targeting a KongSwap pool holding 1000 SNS + 1000 ICP.
2. During the voting period, an attacker executes a large swap on the same pool, moving the price so the position becomes 2000 SNS + 0 ICP (single-sided).
3. The proposal executes: `execute_treasury_manager_withdraw` calls `withdraw` with an empty `WithdrawRequest`.
4. The Treasury Manager withdraws from KongSwap at the manipulated price and returns 2000 SNS + 0 ICP to the SNS treasury.
5. The SNS treasury receives 0 ICP instead of the ~1000 ICP the DAO expected. The call returns `Ok(())` with no error.
6. The attacker reverses their swap on the now-rebalanced pool, profiting at the DAO's expense.

/// Execute a treasury manager withdraw operation
async fn execute_treasury_manager_withdraw(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedWithdrawOperationArg,
) -> Result<(), GovernanceError> {
    let arg_blob = construct_treasury_manager_withdraw_payload(arg.original).map_err(|err| {
        GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!("Failed to construct treasury manager withdraw payload: {err}"),
        )
    })?;

    let balances = governance
        .env
        .call_canister(extension_canister_id, "withdraw", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.withdraw failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!(
                        "Error decoding TreasuryManager.withdraw response: {err:?}"
                    ),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.withdraw failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.withdraw succeeded with response: {:?}",
        balances
    );

    Ok(())
}
