### Title
Unbounded `new_rounds_count` Loop in `distribute_rewards` Permanently Bricks SNS Governance on First Reward Distribution — (`rs/sns/governance/src/governance.rs`)

---

### Summary

When an SNS is initialized with `round_duration_seconds=1` (the minimum valid value) and no prior reward event has been recorded, the `distribute_rewards` function computes `new_rounds_count ≈ now / 1 ≈ 1.7 billion` and then unconditionally iterates that many times in a Decimal-arithmetic loop. There is no cap on `new_rounds_count`. The first automatic `run_periodic_tasks` invocation exhausts the Wasm instruction budget, trapping the canister and permanently preventing reward distribution.

---

### Finding Description

`distribute_rewards` computes the number of missed reward rounds as:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();          // → 0 for a brand-new SNS
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [1](#0-0) 

With `end_timestamp_seconds = None` (no prior reward event), `unwrap_or_default()` yields `0`. With `round_duration_seconds = 1`, `new_rounds_count = now ≈ 1_700_000_000`.

The only guard present is a zero-divisor check: [2](#0-1) 

There is **no upper-bound cap** on `new_rounds_count`. The code then enters:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = round_duration_seconds
        .saturating_mul(i)
        .saturating_add(reward_start_timestamp_seconds)
        .saturating_sub(self.proto.genesis_timestamp_seconds);

    let current_reward_rate = voting_rewards_parameters.reward_rate_at(
        crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
    );
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [3](#0-2) 

Each iteration performs multiple `rust_decimal::Decimal` multiplications and a `reward_rate_at` call (which itself does a `LinearMap` evaluation). ~1.7 billion such iterations will exhaust the Wasm instruction limit long before completion.

`round_duration_seconds = 1` is explicitly within the validated range: [4](#0-3) 

---

### Impact Explanation

The SNS governance canister traps on the first `run_periodic_tasks` call after initialization. Because the reward event is never written, `latest_reward_event().end_timestamp_seconds` remains `None` forever, so every subsequent periodic task call recomputes the same enormous `new_rounds_count` and traps again. Reward distribution is **permanently and irrecoverably broken** for the lifetime of the canister.

---

### Likelihood Explanation

`round_duration_seconds = 1` is a valid, accepted configuration value. A developer testing a rapid-reward SNS, or one following documentation that lists `1` as the minimum, would hit this. No exploit is needed beyond SNS initialization with this parameter — the crash is automatic and requires no further action from any participant.

---

### Recommendation

Cap `new_rounds_count` to a safe maximum before the loop. For example:

```rust
const MAX_ROUNDS_PER_DISTRIBUTION: u64 = 10_000; // or similar safe bound

let new_rounds_count = new_rounds_count.min(MAX_ROUNDS_PER_DISTRIBUTION);
```

Alternatively, compute the reward purse analytically (closed-form integral over the reward rate curve) instead of iterating round-by-round, which is the correct long-term fix. At minimum, add a validation that rejects `round_duration_seconds` values so small that `now / round_duration_seconds` could exceed a safe iteration budget.

---
