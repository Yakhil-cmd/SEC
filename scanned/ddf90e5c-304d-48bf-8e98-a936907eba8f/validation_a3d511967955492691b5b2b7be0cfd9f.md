### Title
Unmetered Synchronous BFS Traversal Over Unbounded Follower Graph in SNS Governance Vote Cascade - (File: rs/sns/governance/src/governance.rs)

### Summary
SNS governance's `cast_vote_and_cascade_follow` performs a synchronous, unbounded BFS traversal over the neuron follower graph with no instruction-limit check. An unprivileged neuron holder can set up a large follower chain (up to `max_number_of_neurons` = 200,000 neurons) and trigger a vote cascade that exhausts the per-message instruction limit, causing `register_vote` and `make_proposal` update calls to trap and permanently fail for any neuron at the root of such a graph.

---

### Finding Description

The SNS governance canister's `cast_vote_and_cascade_follow` is a **synchronous** function that performs a BFS traversal over the entire follower graph without any instruction-limit guard: [1](#0-0) 

The outer `while !induction_votes.is_empty()` loop iterates over every BFS tier, and the inner loops iterate over every follower of every neuron in that tier. There is no call to any `is_over_instructions_limit()` function, no `async`/`await` self-call to break up the work, and no hard-limit escape hatch.

This is in direct contrast to the NNS governance canister, which was refactored to use a `ProposalVotingStateMachine` with explicit soft and hard instruction limits: [2](#0-1) [3](#0-2) 

The NNS version checks `over_soft_message_limit()` after each neuron processed and uses `noop_self_call_if_over_instructions` to break the work across multiple messages. The SNS version has no equivalent mechanism.

The SNS `cast_vote_and_cascade_follow` is called synchronously from two update-call paths:

1. `register_vote` → `cast_vote_and_cascade_follow` (line 3931): [4](#0-3) 

2. `make_proposal` → `cast_vote_and_cascade_follow` (line 3658): [5](#0-4) 

Both are reachable from the public `manage_neuron` update endpoint: [6](#0-5) 

---

### Impact Explanation

When a neuron at the root of a large follower graph votes (or submits a proposal), the synchronous BFS traversal must complete within a single message execution. The IC per-message instruction limit (~5–40 billion instructions) is finite. With up to `MAX_NUMBER_OF_NEURONS_CEILING` = 200,000 neurons: [7](#0-6) 

…each requiring BTreeMap lookups and `vote_from_ballots_following` calls, the traversal can exceed the instruction limit. When this happens:

- The entire update call traps and is rolled back.
- The vote is never recorded.
- The proposer's automatic yes-vote during `make_proposal` also fails, preventing proposal creation.
- Any neuron that has accumulated a large follower base (legitimately or via attacker-controlled neurons) is permanently unable to vote or propose.

This is a permissionless DoS on the SNS governance voting mechanism.

---

### Likelihood Explanation

**Medium.** The attacker must:
1. Create many neurons (each requiring a minimum stake of `neuron_minimum_stake_e8s`, default 1 governance token).
2. Set each neuron to follow the target neuron using `manage_neuron::Follow` or `SetFollowing`.

`max_followees_per_function` is capped at 15 per neuron per function: [8](#0-7) 

But the **follower** side is unbounded — any number of neurons can follow a single neuron. The attacker only needs to create N neurons each following the target, not the other way around. With `max_number_of_neurons` = 200,000 and a minimum stake of 1 token per neuron, the cost is 200,000 governance tokens. For SNS tokens with low market value, this is feasible. The attack is also self-reinforcing: a popular neuron (e.g., a well-known validator) naturally accumulates followers over time without any attacker involvement.

---

### Recommendation

Apply the same fix used in NNS governance: refactor SNS `cast_vote_and_cascade_follow` into a state-machine-based approach with explicit soft and hard instruction limits, and use async self-calls to break the work across multiple messages when the soft limit is exceeded. Concretely:

1. Introduce a `ProposalVotingStateMachine` equivalent for SNS governance (analogous to `rs/nns/governance/src/voting.rs`).
2. Make `register_vote` and the proposal-creation vote cascade async, checking `is_message_over_threshold` after each neuron processed.
3. Use a timer task to drain any remaining cascade work that did not complete within the initial message.

Alternatively (simpler but less complete): add a hard cap on the total number of neurons processed per BFS traversal, returning early and deferring remaining work to a timer.

---

### Proof of Concept

**Setup:**
1. Deploy an SNS with default parameters (`max_number_of_neurons = 200_000`, `max_followees_per_function = 15`).
2. Create a "target" neuron T with a valid stake.
3. Create N attacker neurons A₁…Aₙ, each calling `manage_neuron::Follow` to follow T on the relevant `function_id`.
4. Submit a proposal from any neuron (triggering T's automatic yes-vote if T is the proposer, or have T call `RegisterVote`).

**Trigger:**
```
manage_neuron({
  subaccount: T_subaccount,
  command: RegisterVote({
    proposal: Some(proposal_id),
    vote: Yes,
  })
})
```

**Result:** The synchronous BFS in `cast_vote_and_cascade_follow` iterates over all Aᵢ followers of T. With N large enough to exhaust the instruction budget, the update call traps. The vote is never recorded. Repeated calls always trap. T is permanently unable to vote on any proposal.

        while !induction_votes.is_empty() {
            // This will be populated with the followers of neurons in the
            // current BFS tier, who might be swayed to indirectly vote, thus
            // forming the next tier in the BFS.
            let mut follower_neuron_ids = BTreeSet::new();

            // Process the current tier in the BFS.
            for (current_neuron_id, current_new_vote) in &induction_votes {
                let current_ballot = match ballots.get_mut(current_neuron_id) {
                    Some(b) => b,
                    None => {
                        // neuron_id has no (blank) ballot, which means they
                        // were not eligible when the proposal was first
                        // created. This is fairly unusual, but does not
                        // indicate a bug (therefore, no log).
                        continue;
                    }
                };

                // Only fill in "blank" ballots. I.e. those with vote ==
                // Unspecified. This check could just as well be done before
                // current_neuron_id is added to induction_votes.
                if current_ballot.vote != (Vote::Unspecified as i32) {
                    continue;
                }

                // Fill in current_ballot.
                assert_ne!(*current_new_vote, Vote::Unspecified);
                current_ballot.vote = *current_new_vote as i32;
                current_ballot.cast_timestamp_seconds = now_seconds;

                // Take note of the followers of current_neuron_id, and add them
                // to the next "tier" in the BFS.

                if let Some(new_follower_neuron_ids) = topic_followers
                    .and_then(|topic_followers| topic_followers.get(current_neuron_id))
                {
                    for follower_neuron_id in new_follower_neuron_ids {
                        follower_neuron_ids.insert(follower_neuron_id.clone());
                    }
                }

                if let Some(new_follower_neuron_ids) =
                    neuron_id_to_follower_neuron_ids.get(current_neuron_id)
                {
                    for follower_neuron_id in new_follower_neuron_ids {
                        follower_neuron_ids.insert(follower_neuron_id.clone());
                    }
                }
            }

            // Prepare for the next iteration of the (outer most) loop by
            // constructing the next BFS tier (from follower_neuron_ids).
            induction_votes.clear();
            for follower_neuron_id in follower_neuron_ids {
                let Some(follower_neuron) = neurons.get(&follower_neuron_id.to_string()) else {
                    // This is a highly suspicious, because currently, we do not
                    // delete neurons, which means that we have an invalid NeuronId
                    // floating around in the system, which indicates that we have a
                    // bug. For now, we deal with that by logging, and pretending like
                    // we did not see follower_neuron_id.
                    log!(
                        ERROR,
                        "Missing neuron {} while trying to record (and cascade) \
                            a vote on proposal {:#?}.",
                        follower_neuron_id,
                        proposal_id,
                    );
                    continue;
                };

                let follower_vote = follower_neuron.vote_from_ballots_following(
                    function_id,
                    topic,
                    ballots,
                    proposal_id,
                );

                if follower_vote != Vote::Unspecified {
                    // follower_neuron would be swayed by its followees!
                    //
                    // This is the other (earlier) point at which we could
                    // consider whether a neuron is already locked in, and that
                    // no recursion is needed.
                    induction_votes.insert(follower_neuron_id.to_string(), follower_vote);
                }
            }
        }
