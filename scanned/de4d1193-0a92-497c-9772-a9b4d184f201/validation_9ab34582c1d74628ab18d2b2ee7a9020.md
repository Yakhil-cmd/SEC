### Title
Incorrect u64→i64 Timestamp Conversion Causes Silent Data Corruption in ICRC-1 Rosetta API SQL Storage - (File: rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs)

### Summary
The `convert_timestamp_to_db` function in the ICRC-1 Rosetta API converts `u64` nanosecond timestamps to `i64` before storing them in SQLite. For any timestamp value exceeding `i64::MAX` (9,223,372,036,854,775,807), the stored value becomes a negative `i64`. This causes all SQL comparisons and ordering on the `transaction_created_at_time` and `approval_expires_at` columns to silently produce incorrect results, because SQLite treats the stored negative integer as numerically less than any positive current timestamp. The `approval_expires_at` field is fully user-controlled via ICRC-2 `approve` calls, making this directly triggerable by an unprivileged user.

### Finding Description

`convert_timestamp_to_db` at lines 714–725 of `rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs` converts a `u64` timestamp to `i64` for SQLite storage:

```rust
fn convert_timestamp_to_db(timestamp: u64) -> i64 {
    if timestamp > i64::MAX as u64 {
        ((timestamp & 0x7fffffffffffffff) as i64) | i64::MIN
    } else {
        timestamp as i64
    }
}
``` [1](#0-0) 

While the bit-reinterpretation arithmetic is equivalent to a plain `timestamp as i64` wrapping cast (both produce the same bit pattern), the result for any `timestamp > i64::MAX` is a **negative** `i64`. This negative value is then stored in the `blocks` table under the `transaction_created_at_time` and `approval_expires_at` columns: [2](#0-1) 

The codebase itself acknowledges the problem with a TODO comment:

> "TODO: Change the timestamps to a text type. Keeping timestamps as signed integers can cause issues if someone tries to compare them directly in the DB." [3](#0-2) 

There is **no corresponding inverse function** (`convert_timestamp_from_db`) anywhere in the codebase. The timestamp columns are used for SQL-level filtering and ordering, not for reconstructing block data (blocks are reconstructed from `serialized_block`). Any SQL query of the form `WHERE approval_expires_at > :current_time` or `ORDER BY transaction_created_at_time` will silently produce wrong results when the stored value is negative.

### Impact Explanation

An ICRC-2 `approve` call allows any user to set `expires_at` to an arbitrary `u64` value. If a user sets `expires_at = u64::MAX` (18,446,744,073,709,551,615), `convert_timestamp_to_db` stores it as `-1` in SQLite. Any SQL query checking whether the approval is still active — e.g., `WHERE approval_expires_at > :now` — evaluates `-1 > 1_700_000_000_000_000_000` as **false**, making a perpetual approval appear immediately expired to the Rosetta API. Conversely, a `transaction_created_at_time` set to a value just above `i64::MAX` would be stored as a large negative number, breaking deduplication window queries and transaction ordering in Rosetta API responses. Exchanges and custodians that rely on the ICRC-1 Rosetta API for approval status or transaction deduplication checks receive silently incorrect data, which can lead to incorrect financial accounting or double-processing of transactions.

### Likelihood Explanation

The `approval_expires_at` field in ICRC-2 is a user-supplied `u64`. Any user can submit an `approve` transaction with `expires_at` set to a value above `i64::MAX` (e.g., `u64::MAX`) at any time with no special privileges. The Rosetta API will silently store and mishandle it. Current wall-clock IC timestamps (~1.7 × 10¹⁸ ns) are below `i64::MAX` (~9.2 × 10¹⁸ ns), so the `timestamp` column is not yet affected by normal block ingestion, but the `approval_expires_at` column is immediately exploitable by any unprivileged user.

### Recommendation

Replace `convert_timestamp_to_db` with a simple wrapping cast (`timestamp as i64`) and add a symmetric `convert_timestamp_from_db(v: i64) -> u64 { v as u64 }` for all read paths. More robustly, store timestamps as `TEXT` (decimal string) as the TODO already suggests, which avoids all signed/unsigned comparison issues in SQLite entirely. All SQL queries on `transaction_created_at_time` and `approval_expires_at` must be audited to ensure they use the correct inverse conversion when comparing against current time.

### Proof of Concept

1. Submit an ICRC-2 `approve` transaction with `expires_at = u64::MAX` (18,446,744,073,709,551,615).
2. The Rosetta API calls `convert_timestamp_to_db(u64::MAX)`:
   - `u64::MAX & 0x7fffffffffffffff = 0x7FFFFFFFFFFFFFFF`
   - `as i64 = i64::MAX`
   - `| i64::MIN = -1`
   - Stored value: `-1`
3. Any SQL query `WHERE approval_expires_at > :current_timestamp` evaluates `-1 > 1_700_000_000_000_000_000` → **false**.
4. The Rosetta API reports the approval as expired, even though `expires_at = u64::MAX` means "never expires."
5. An exchange relying on Rosetta API approval status will incorrectly reject valid spend operations backed by this approval.

The test at lines 2041–2053 of `rs/rosetta-api/icp/ledger_canister_blocks_synchronizer/src/blocks.rs` independently confirms the same pattern — large u64 values stored as negative i64 — and shows the codebase is aware of the bit-reinterpretation behavior, but no SQL-comparison correctness is verified for the ICRC-1 Rosetta path. // Helper function to convert u64 timestamp to i64 for database storage
fn convert_timestamp_to_db(timestamp: u64) -> i64 {
    // Check if timestamp exceeds i64::MAX
    if timestamp > i64::MAX as u64 {
        // For values exceeding i64::MAX, we need to preserve the lower bits
        // but represent them as a negative number in two's complement
        ((timestamp & 0x7fffffffffffffff) as i64) | i64::MIN
    } else {
        // Safe conversion when within i64 range
        timestamp as i64
    }
}
