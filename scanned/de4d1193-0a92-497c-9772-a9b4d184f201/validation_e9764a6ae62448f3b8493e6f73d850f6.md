### Title
Missing Execution-Time Cumulative Cap Enforcement for `MintSnsTokens` Proposals Allows Unlimited SNS Token Minting - (File: `rs/sns/governance/src/proposal.rs`, `rs/sns/governance/src/governance.rs`)

---

### Summary

The `MintSnsTokens` SNS governance proposal action is subject to a designed 7-day cumulative minting cap, analogous to the cap enforced for `TransferSnsTreasuryFunds`. However, two compounding defects completely disable this cap: (1) the submission-time upper bound is hardcoded to `Decimal::MAX`, and (2) unlike `TransferSnsTreasuryFunds`, there is no execution-time re-check of the cumulative amount. Multiple `MintSnsTokens` proposals can be submitted and executed within the same 7-day window, collectively minting an unbounded number of SNS tokens with no protocol-level enforcement of the intended rate limit.

---

### Finding Description

**Vulnerability class:** Ledger conservation bug / chain-fusion mint/burn accounting bug — cumulative accounting not enforced at execution time.

The IC SNS governance system defines a `TokenProposalAction` trait with a `recent_amount_total_upper_bound_tokens` method that is supposed to cap the total tokens minted or transferred within a rolling 7-day window.

For `TransferSnsTreasuryFunds`, this cap is enforced at **two points**:

1. **Submission time** — `treasury_valuation_if_proposal_amount_is_small_enough_or_err` checks `proposal_amount + spent_in_last_7_days <= max_tokens`.
2. **Execution time** — `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` re-checks the cumulative total just before the ledger transfer executes, because other proposals may have been executed between submission and execution. [1](#0-0) 

For `MintSnsTokens`, **neither check is operative**:

**Defect 1 — Submission-time upper bound is `Decimal::MAX`:**

The `TokenProposalAction` implementation for `MintSnsTokens` overrides `recent_amount_total_upper_bound_tokens` to return `Decimal::MAX` unconditionally, with a TODO comment acknowledging this is a placeholder: [2](#0-1) 

This means the check in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` — `if proposal_amount_tokens > allowance_remainder_tokens` — can never trigger for `MintSnsTokens`, regardless of how many tokens have already been minted in the window. [3](#0-2) 

**Defect 2 — No execution-time cumulative check in `perform_mint_sns_tokens`:**

`perform_mint_sns_tokens` directly calls `ledger.transfer_funds` with no pre-execution cumulative validation: [4](#0-3) 

Compare this to `perform_transfer_sns_treasury_funds`, which calls `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` before executing: [5](#0-4) 

The execution dispatch for `MintSnsTokens` passes no valuation and performs no re-check: [6](#0-5) 

**The race condition (analogous to the timelock::claim bug):**

Even if Defect 1 were fixed, Defect 2 alone is sufficient. Two `MintSnsTokens` proposals can be submitted simultaneously. At submission time, `spent_tokens = 0` for both (neither has executed yet), so both pass the `proposal_amount <= max - 0` check. Both are adopted by governance. Both execute sequentially, each minting up to the maximum allowed amount — collectively minting 2× the cap. This is the exact same pattern as `timelock::claim` checking `amount <= unlocked_tokens` without checking `amount + tokens_transferred <= unlocked_tokens`.

The infrastructure for the correct limit (`mint_sns_tokens_7_day_total_upper_bound_tokens`, `total_minting_amount_tokens`) exists and is correct — it is simply never enforced: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

An SNS governance participant (or coalition holding sufficient voting power) can submit multiple `MintSnsTokens` proposals within a 7-day window. Because the upper bound is `Decimal::MAX` at submission time and there is no execution-time check, each proposal passes validation independently. When executed, they collectively mint an unbounded number of SNS tokens, inflating the token supply far beyond the intended 7-day cap. This directly undermines the rate-limiting safety mechanism designed to protect SNS token holders from rapid treasury inflation or governance-level token dilution attacks.

---

### Likelihood Explanation

Medium. Exploiting this requires passing multiple governance proposals, which requires sufficient voting power. However, the rate-limiting cap is specifically designed as a defense-in-depth mechanism against scenarios where governance is temporarily compromised or where multiple proposals are submitted in good faith but collectively exceed the safe limit. The absence of the execution-time check means even non-malicious concurrent proposals can exceed the cap. The TODO comments (`NNS1-2982`, `NNS1-2981`) confirm this is a known, unresolved gap in production code.

---

### Recommendation

1. Replace the `Decimal::MAX` stub in `MintSnsTokens::recent_amount_total_upper_bound_tokens` with the real implementation (uncomment the block guarded by `TODO(NNS1-2982)`):

```rust
fn recent_amount_total_upper_bound_tokens(valuation: &Valuation) -> Result<Decimal, String> {
    mint_sns_tokens_7_day_total_upper_bound_tokens(*valuation)
        .map_err(|e| format!("Unable to validate amount: {:?}", e))
}
```

2. Add an execution-time cumulative check in `perform_mint_sns_tokens`, mirroring `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, before calling `ledger.transfer_funds`.

---

### Proof of Concept

1. Deploy an SNS with a large token treasury.
2. Submit two `MintSnsTokens` proposals simultaneously, each with `amount_e8s` equal to the maximum 7-day allowance (e.g., 25% of treasury for a medium-sized SNS).
3. At submission time, `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`, so both proposals pass validation regardless of amount.
4. Both proposals are adopted by governance.
5. `perform_mint_sns_tokens` executes for proposal 1: mints the full allowance. No state is updated to record this against the cap.
6. `perform_mint_sns_tokens` executes for proposal 2: mints the full allowance again. No cumulative check is performed.
7. Result: 2× the intended 7-day cap has been minted, with no protocol-level rejection. [9](#0-8) [10](#0-9)

    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }

    Ok(())

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
