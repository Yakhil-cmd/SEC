### Title
Ingress Induction Fee Bypass via Cleanup Callback Debit Forgiveness During DTS — (`rs/execution_environment/src/execution/response.rs`)

---

### Summary

During Deterministic Time Slicing (DTS), ingress induction fees for messages arriving while a canister is paused are deferred as `ingress_induction_cycles_debit`. When a response callback traps and triggers the cleanup callback, the execution layer deliberately forgives the entire pending debit if the canister's balance is insufficient to cover it. This is an exact structural analog to the `_postOp`/`postOpReverted` fee bypass: a "normal" path charges fees in full, while the "error/fallback" path silently drops them.

---

### Finding Description

**Two-phase fee deferral during DTS**

When a canister has a paused execution, `charge_ingress_induction_cost` cannot immediately debit the balance. Instead it records the fee as a postponed charge:

```rust
// rs/cycles_account_manager/src/cycles_account_manager.rs:335-348
if canister.has_paused_execution_or_install_code() {
    // ...
    canister.system_state
        .add_postponed_charge_to_ingress_induction_cycles_debit(cycles);
    Ok(())
}
``` [1](#0-0) 

**Normal path — debit applied in full**

When the response callback completes (success or trap-before-cleanup), `apply_ingress_induction_cycles_debit` is called unconditionally, charging the full accumulated debit:

```rust
// rs/execution_environment/src/execution/response.rs:388-395
self.canister.system_state
    .apply_ingress_induction_cycles_debit(
        self.canister.canister_id(),
        round.cost_schedule, round.log,
        round.counters.charging_from_balance_error,
    );
``` [2](#0-1) 

**Error/fallback path — debit silently forgiven**

When the response callback traps and a cleanup callback runs, `handle_wasm_execution_of_cleanup_callback` checks whether the balance covers the debit. If not, it **reduces the debit** before applying it:

```rust
// rs/execution_environment/src/execution/response.rs:490-496
if self.canister.system_state.balance()
    < ingress_induction_cycles_debit + removed_cycles
{
    self.canister.system_state
        .remove_charge_from_ingress_induction_cycles_debit(
            ingress_induction_cycles_debit - removed_cycles,
        );
}
``` [3](#0-2) 

The code comment at lines 481–484 explicitly acknowledges the consequence: *"This allows the cleanup callback to always succeed at the expense of some ingress messages being inducted for free in this edge case."* [4](#0-3) 

When `removed_cycles = 0` (the cleanup callback does not remove any cycles, which is the common case since cleanup callbacks cannot make calls or accept cycles), the entire `ingress_induction_cycles_debit` is forgiven: the new debit becomes `removed_cycles = 0`, and `apply_ingress_induction_cycles_debit` charges nothing. [5](#0-4) 

---

### Impact Explanation

An unprivileged canister controller can induct an arbitrary number of ingress messages into their canister without paying the ingress induction fee. The subnet processes those messages (consuming real resources), but the canister's balance is not reduced by the corresponding fee. This is a direct cycles/resource accounting bypass: the subnet pays the cost of message induction while the canister pays nothing.

---

### Likelihood Explanation

The attack requires only capabilities available to any canister controller on the IC mainnet:

1. Deploy a canister whose response callback traps and whose cleanup callback runs across multiple DTS slices (achievable by performing many stable-memory operations in the cleanup callback).
2. Trigger the response callback via a normal inter-canister call.
3. While the cleanup callback is paused between DTS slices, send many ingress messages to the canister. Each one is accepted (the `debited_balance` check passes) and recorded as `ingress_induction_cycles_debit`.
4. Ensure the canister's actual `cycles_balance` is below the accumulated debit (e.g., by burning cycles or simply not topping up).
5. When the cleanup callback finishes, the entire debit is forgiven.

No privileged access, no threshold corruption, and no external oracle is required. The only constraint is that the cleanup callback must span enough DTS slices to allow the ingress messages to be inducted — a condition the attacker fully controls by choosing the cleanup callback's instruction count.

---

### Recommendation

The debit forgiveness in `handle_wasm_execution_of_cleanup_callback` should be removed or replaced with a stricter policy. Options include:

1. **Apply the full debit regardless**: charge `min(balance, ingress_induction_cycles_debit)` and let the balance saturate to zero, rather than forgiving the remainder. The cleanup callback can still run; it simply cannot remove additional cycles.
2. **Cap the debit at the balance before the cleanup callback starts**: record the balance at the start of the cleanup phase and ensure the debit never exceeds it, rather than silently dropping it at the end.
3. **Reject ingress messages while a cleanup callback is running**: prevent new ingress induction debits from accumulating during the cleanup phase, eliminating the race condition entirely.

---

### Proof of Concept

```
1. Attacker deploys canister C with:
   - on_reply: trap()                          // forces cleanup callback
   - on_cleanup: stable64_grow(N) + loop()     // runs many DTS slices

2. C calls any canister D; D replies.

3. DTS begins executing C's response callback → traps immediately.
   DTS begins executing C's cleanup callback (slice 1 of N).

4. While C is paused (NextExecution::ContinueLong), attacker sends
   K ingress messages to C from an external user identity.
   Each message passes the debited_balance check and adds
   ingress_induction_cost(msg) to C.ingress_induction_cycles_debit.

5. Attacker ensures C.cycles_balance < ingress_induction_cycles_debit
   (e.g., by pre-burning cycles or simply not topping up).

6. DTS finishes the cleanup callback.
   handle_wasm_execution_of_cleanup_callback fires:
     balance (low) < ingress_induction_cycles_debit + 0
     → remove_charge_from_ingress_induction_cycles_debit(full debit)
     → apply_ingress_induction_cycles_debit charges 0

7. K ingress messages were inducted; C paid 0 cycles in induction fees.
```

```rust
    /// Charges the pending 'ingress_induction_cycles_debit' from the balance.
    ///
    /// Precondition:
    /// - The balance is large enough to cover the debit.
    pub fn apply_ingress_induction_cycles_debit(
        &mut self,
        canister_id: CanisterId,
        cost_schedule: CanisterCyclesCostSchedule,
        log: &ReplicaLogger,
        charging_from_balance_error: &IntCounter,
    ) {
        // We rely on saturating operations of `Cycles` here.
        let remaining_debit = self.ingress_induction_cycles_debit - self.cycles_balance;
        debug_assert_eq!(remaining_debit.get(), 0);
        if remaining_debit.get() > 0 {
            // This case is unreachable and may happen only due to a bug: if the
            // caller has reduced the cycles balance below the cycles debit.
            charging_from_balance_error.inc();
            error!(
                log,
                "[EXC-BUG]: Debited cycles exceed the cycles balance of {} by {} in install_code",
                canister_id,
                remaining_debit,
            );
            // Continue the execution by dropping the remaining debit, which makes
            // some of the postponed charges free.
        }
        self.consume_cycles(CompoundCycles::<IngressInduction>::new(
            self.ingress_induction_cycles_debit,
            cost_schedule,
        ));
        self.ingress_induction_cycles_debit = Cycles::zero();
    }
