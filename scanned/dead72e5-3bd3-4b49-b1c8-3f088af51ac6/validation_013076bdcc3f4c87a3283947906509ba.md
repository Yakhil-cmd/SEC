### Title
Unsafe Signed-to-Unsigned Cast in `ledgeramount_from_amount` Produces Massively Overstated Token Amounts - (File: `rs/rosetta-api/icp/src/models/amount.rs`)

### Summary
`from_amount` validates only that `val.abs()` fits in `u64`, allowing negative `i128` values to pass through. `ledgeramount_from_amount` then casts the negative `i128` to `u64` via a bare `as u64` cast, which in Rust is a wrapping (two's-complement) reinterpretation — producing a massively overstated token amount, directly analogous to the Tracer Protocol `int256 → uint256` bug.

### Finding Description
In `rs/rosetta-api/icp/src/models/amount.rs`, `from_amount` parses the Rosetta `Amount.value` string as `i128` (which can be negative) and performs the following guard:

```rust
let _ = u64::try_from(val.abs()).map_err(|_| "Amount does not fit in u64".to_string())?;
Ok(val)
``` [1](#0-0) 

The guard checks that `val.abs()` fits in `u64`, but **does not check that `val >= 0`**. A negative value such as `-1` has `abs() = 1`, which trivially passes the check, and `Ok(-1_i128)` is returned.

`ledgeramount_from_amount` then performs:

```rust
pub fn ledgeramount_from_amount(amount: &Amount, token_name: &str) -> Result<Tokens, String> {
    let inner = from_amount(amount, token_name)?;
    Ok(Tokens::from_e8s(inner as u64))
}
``` [2](#0-1) 

In Rust, `negative_i128 as u64` is a wrapping cast (defined behavior, but semantically wrong): `-1_i128 as u64 == u64::MAX == 18_446_744_073_709_551_615`. This is called in `rs/rosetta-api/icp/src/convert.rs` for the `Disburse` operation:

```rust
Some(ledgeramount_from_amount(amount, token_name).map_err(|e| { ... })?)
``` [3](#0-2) 

A second unsafe cast exists in the same file for `OperationType::Fee`:

```rust
state.fee(account, Tokens::from_e8s((-amount) as u64))?;
``` [4](#0-3) 

If a caller submits a positive fee amount (not rejected by `from_amount`), `-amount` is negative and `(-amount) as u64` wraps to a huge value.

### Impact Explanation
An unprivileged Rosetta API caller submitting a `Disburse` or `Fee` operation with a negative amount string (e.g., `"-1"`) causes `ledgeramount_from_amount` / the fee path to produce `Tokens::from_e8s(u64::MAX)` — approximately 184 billion ICP. The Rosetta API constructs and forwards a governance/ledger call with this massively overstated amount. The downstream NNS governance canister will reject the disburse because the neuron balance is insufficient, so direct token theft is not possible. However, the Rosetta API itself silently misrepresents the amount, and any code path that trusts the constructed `Tokens` value before submission (e.g., pre-flight balance checks, fee estimation, logging, or future code changes) will operate on a wildly incorrect figure. The incorrect `Tokens` value also propagates into the `Request` object returned to the caller, corrupting the construction response.

### Likelihood Explanation
The Rosetta API is publicly reachable. Any user constructing a `TRANSACTION`, `FEE`, or `DISBURSE` operation via the Rosetta construction API can supply an arbitrary `value` string in the `Amount` field. No authentication or privilege is required. The `from_amount` guard explicitly allows negative values through (it only checks `abs()`), making this trivially triggerable.

### Recommendation
1. In `from_amount`, add an explicit non-negativity check before returning:
   ```rust
   if val < 0 {
       return Err(format!("Amount must be non-negative, got {val}"));
   }
   ```
2. In `ledgeramount_from_amount`, replace `inner as u64` with a checked conversion:
   ```rust
   let e8s = u64::try_from(inner).map_err(|_| format!("Amount {inner} does not fit in u64"))?;
   Ok(Tokens::from_e8s(e8s))
   ```
3. In the `Fee` branch of `operations_to_requests`, replace `(-amount) as u64` with a checked negation and conversion, rejecting non-negative fee amounts explicitly.

### Proof of Concept
Attacker submits a Rosetta `/construction/payloads` or `/construction/combine` request with:
```json
{
  "operation_identifier": {"index": 0},
  "type": "DISBURSE",
  "account": {"address": "<valid_account>"},
  "amount": {"value": "-1", "currency": {"symbol": "ICP", "decimals": 8}}
}
```

Execution trace:
1. `operations_to_requests` calls `ledgeramount_from_amount(&amount, "ICP")`.
2. `from_amount` parses `"-1"` → `val = -1_i128`; `val.abs() = 1`; `u64::try_from(1)` succeeds; returns `Ok(-1_i128)`.
3. `ledgeramount_from_amount` computes `-1_i128 as u64 = 18_446_744_073_709_551_615`.
4. `Tokens::from_e8s(18_446_744_073_709_551_615)` is constructed and placed into the `Disburse` request.
5. The Rosetta API returns a construction response containing a disburse of ≈184 billion ICP. [5](#0-4) [6](#0-5)
