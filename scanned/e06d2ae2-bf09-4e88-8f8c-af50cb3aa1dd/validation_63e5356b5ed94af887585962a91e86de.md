### Title
Unbounded `.await` on Bounded Advert Channel Enables Resource Exhaustion via Authenticated Subnet Peer — (`rs/p2p/state_sync_manager/src/routes/advert.rs`)

---

### Summary

The `state_sync_advert_handler` performs an unbounded `.await` on a bounded `mpsc` channel (capacity=20). The single-threaded manager event loop processes adverts sequentially. If the event loop is slow (e.g., `maybe_start_state_sync` performs blocking I/O), the channel fills and all subsequent handler tasks suspend indefinitely, holding QUIC streams open. The claimed impact of blocking *all* P2P messages is overstated due to per-connection QUIC stream limits, but the resource exhaustion path is real and the code evidence is concrete.

---

### Finding Description

**Channel capacity and unbounded await:**

The channel is created with capacity 20: [1](#0-0) 

The HTTP handler sends into this channel with an unbounded `.await` — no timeout, no `try_send`, no backpressure: [2](#0-1) 

**Single-threaded sequential event loop:**

The manager event loop processes one advert at a time with `.await`: [3](#0-2) 

**Slow consumer path — `maybe_start_state_sync` is synchronous:**

`maybe_start_state_sync` is a synchronous function called directly inside the async `handle_advert`. If the implementation performs blocking I/O (file system operations), it blocks the tokio thread running the event loop: [4](#0-3) [5](#0-4) 

**Developer awareness of the blocking concern — but only partially applied:**

The developers explicitly used `try_send` (non-blocking) when forwarding peer IDs to the ongoing state sync, with a comment acknowledging that file system operations can block: [6](#0-5) 

This same discipline was **not** applied to the HTTP handler's `advert_sender.send().await`.

**QUIC stream concurrency limit:**

The QUIC transport sets `MAX_CONCURRENT_BIDI_STREAMS = 1000` **per connection**: [7](#0-6) 

---

### Impact Explanation

**What is real:**
- An authenticated subnet peer can send 20+ concurrent POST `/state-sync/advert` requests.
- If the event loop is blocked in `maybe_start_state_sync` (blocking I/O), the channel stays full.
- All subsequent handler tasks suspend on `.await`, each holding a QUIC stream open.
- Up to 1000 tokio tasks can be blocked per connection (the per-connection QUIC stream limit), consuming memory and executor resources on the victim.

**What is overstated in the question:**
- The QUIC stream limit of 1000 is **per-connection**, not global. A single Byzantine peer can exhaust only their own connection's stream budget. Consensus messages and state sync from *other* peers arrive on separate QUIC connections and are unaffected.
- The claim that "all P2P message handlers" are blocked is incorrect — only the attacker's connection is impacted.

**Actual scoped impact:**
- Resource exhaustion: up to 1000 suspended tokio tasks per attacking peer, each holding memory.
- Loss of state sync contribution from the attacking peer's connection.
- If `maybe_start_state_sync` blocks the tokio thread (not just the task), the impact widens to all tasks scheduled on that thread — but this is a separate blocking-in-async anti-pattern issue.
- Consensus liveness from other honest peers is **not** directly threatened.

---

### Likelihood Explanation

- Requires an authenticated subnet peer (TLS-verified), placing the attacker within the Byzantine fault threshold — within the stated threat model.
- Requires `maybe_start_state_sync` to be slow enough for the channel to stay full. The state manager implementation performs checkpoint and manifest operations that can involve file I/O, making this realistic under load.
- The attack is repeatable and requires no special knowledge beyond valid protobuf advert construction.

---

### Recommendation

Replace the unbounded `.await` in `state_sync_advert_handler` with `try_send` (matching the pattern already used for `ongoing.sender`), returning `StatusCode::TOO_MANY_REQUESTS` when the channel is full:

```rust
// In state_sync_advert_handler:
state
    .advert_sender
    .try_send((advert, peer))
    .map_err(|_| StatusCode::TOO_MANY_REQUESTS)?;
```

Additionally, `maybe_start_state_sync` should be offloaded to a blocking thread pool via `spawn_blocking` (as is already done for `available_states` in `send_state_adverts`) to prevent blocking the tokio executor thread. [8](#0-7) 

---

### Proof of Concept

State-machine test:
1. Construct a `MockStateSync` whose `maybe_start_state_sync` sleeps for 60 seconds.
2. Send 25 concurrent POST `/state-sync/advert` requests with distinct valid `StateSyncArtifactId` values from a mock authenticated peer.
3. Assert: after the channel fills (20 items), the 21st–25th handler futures never return within a 5-second timeout.
4. Assert: the event loop does not process any further adverts during the sleep.

The channel capacity (20) and the unbounded `.await` are directly observable in the production source at: [1](#0-0) [2](#0-1)
