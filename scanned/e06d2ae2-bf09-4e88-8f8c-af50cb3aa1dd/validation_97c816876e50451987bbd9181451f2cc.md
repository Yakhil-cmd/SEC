### Title
Absence of Slippage Protection in SNS Treasury Manager Deposit Exposes Treasury Funds to Sandwich Attacks - (File: rs/sns/governance/src/extensions.rs)

---

### Summary

The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` performs no slippage check when depositing SNS treasury funds into a DEX via a treasury manager extension. Because SNS governance proposals are publicly visible for days before execution, an attacker can observe a pending deposit proposal, manipulate the DEX price just before execution, and cause the SNS treasury to receive significantly fewer LP tokens than expected. The codebase itself acknowledges this as a known risk but provides no on-chain mitigation.

---

### Finding Description

When an SNS governance proposal to deposit treasury funds into a DEX is executed, the call chain is:

1. `execute_treasury_manager_deposit` calls `approve_treasury_manager`, which issues ICRC-2 allowances for the exact token amounts specified in the proposal (`treasury_allocation_sns_e8s`, `treasury_allocation_icp_e8s`).
2. It then calls `deposit` on the treasury manager canister, passing a `DepositRequest` that contains only `allowances` (token amounts) — no price ratio, no slippage tolerance, no minimum LP tokens out. [1](#0-0) 

The `DepositRequest` type itself carries no slippage field: [2](#0-1) 

The proposal validation step (`validate_deposit_operation_impl`) only checks that the requested amounts do not exceed 50% of the current treasury balance. It performs no check on the current DEX price ratio or any acceptable deviation from a reference price: [3](#0-2) 

The codebase explicitly acknowledges this gap in two places. The treasury manager DID interface lists it as a "Known Security Risk": [4](#0-3) 

And the proposal rendering for `RegisterExtension` includes a warning to voters: [5](#0-4) 

Despite these acknowledgements, no enforcement mechanism exists in the SNS governance execution path to bound the price deviation at deposit time.

---

### Impact Explanation

**Impact: Medium**

An attacker who successfully sandwiches a treasury deposit can cause the SNS treasury to receive materially fewer LP tokens than the token amounts deposited would warrant at a fair price. The deposited SNS and ICP tokens are fully consumed (the allowance is spent), but the LP position returned is worth less. For large deposits (the proposal validation permits up to 50% of the treasury balance), the absolute loss can be substantial. The attacker profits by reversing their price-manipulation trade after the deposit settles.

---

### Likelihood Explanation

**Likelihood: Medium**

SNS governance proposals are publicly visible on-chain for their entire voting period (typically days). Any canister or user can observe a pending deposit proposal, compute the expected deposit amounts, and prepare a sandwich. On the Internet Computer, the attacker submits a large DEX swap in a message that executes before the governance heartbeat/timer fires `execute_treasury_manager_deposit`. Because IC message ordering within a subnet is deterministic and the governance execution is triggered by a timer, the attacker can reliably sequence their swap ahead of the deposit by submitting it in the same or an immediately preceding round. No privileged access is required.

---

### Recommendation

1. **Add a `min_lp_tokens_out` or `max_price_deviation_bps` field to the deposit proposal arguments** (`ValidatedDepositOperationArg` / `DepositRequest`). The SNS governance execution path should pass this bound to the treasury manager and reject the result if the returned LP balance falls below it.

2. **Re-validate the price ratio at execution time** inside `execute_treasury_manager_deposit` by querying the DEX for the current spot price and comparing it against a TWAP or the price at proposal submission time, reverting if the deviation exceeds a configurable threshold.

3. **Require treasury manager implementations to enforce slippage protection** as a mandatory condition of the NNS blessing process, and surface this requirement in the `TreasuryManager` trait in `rs/sns/treasury_manager/src/lib.rs`. [6](#0-5) 

---

### Proof of Concept

```
1. An SNS governance proposal is submitted:
      treasury_allocation_sns_e8s = 1_000_000_000   // 10 SNS tokens
      treasury_allocation_icp_e8s = 500_000_000     // 5 ICP

2. The proposal enters the voting period (publicly visible for days).

3. The proposal passes. The governance timer fires execute_treasury_manager_deposit.

4. Before the governance canister's deposit message is processed, the attacker
   submits a large ICP→SNS swap on the DEX, moving the SNS/ICP price significantly
   (e.g., doubling the ICP price of SNS).

5. execute_treasury_manager_deposit calls approve_treasury_manager:
      icrc2_approve(treasury_manager, 1_000_000_000 SNS, expiry=now+1h)
      icrc2_approve(treasury_manager, 500_000_000 ICP, expiry=now+1h)
   No price check is performed.

6. governance.env.call_canister(extension_canister_id, "deposit", arg_blob) is called.
   The treasury manager deposits at the manipulated price.
   The SNS treasury receives LP tokens worth ~50% of the deposited value at fair price.

7. The attacker reverses their swap (SNS→ICP), restoring the price and pocketing
   the profit from the round-trip price impact.

8. execute_treasury_manager_deposit returns Ok(()), logging the Balances response
   with no check on whether the LP tokens received are reasonable.
``` [7](#0-6) [8](#0-7)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)

    async fn approve_treasury_manager(
        &self,
        treasury_manager_canister_id: CanisterId,
        sns_amount_e8s: u64,
        icp_amount_e8s: u64,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: treasury_manager_canister_id.get().0,
            subaccount: None,
        };

        let expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS);
        let expiry_time_nsec = expiry_time_sec.saturating_mul(NANO_SECONDS_PER_SECOND);

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

        self.ledger
            .icrc2_approve(
                to,
                sns_amount_e8s,
                Some(expiry_time_nsec),
                self.transaction_fee_e8s_or_panic(),
                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making SNS Token treasury transfer: {e}"),
                )
            })?;

        self.nns_ledger
            .icrc2_approve(
                to,
                icp_amount_e8s,
                Some(expiry_time_nsec),
                icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(),
                self.icp_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making ICP Token treasury transfer: {e}"),
                )
            })?;

        Ok(())
    }
