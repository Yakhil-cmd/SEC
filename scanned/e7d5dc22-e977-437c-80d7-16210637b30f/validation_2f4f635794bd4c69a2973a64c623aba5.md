### Title
ckETH Minter `EthereumNetwork` Chain ID Hardcoded and Immutable After Initialization, Causing Stale Chain ID on Ethereum Fork - (File: `rs/ethereum/cketh/minter/src/lifecycle.rs`)

### Summary
The ckETH minter's `EthereumNetwork` enum hardcodes Ethereum chain IDs and stores the chosen network in the minter `State` at `init` time. The `UpgradeArg` type provides no field to change `ethereum_network` post-deployment. If Ethereum forks and the canonical chain ID changes, the minter permanently uses the pre-fork chain ID in every new EIP-1559 transaction it creates, mirroring the `DOMAIN_SEPARATOR` staleness pattern from the external report.

### Finding Description
`EthereumNetwork::chain_id()` returns a compile-time constant:

```rust
// rs/ethereum/cketh/minter/src/lifecycle.rs
impl EthereumNetwork {
    pub fn chain_id(&self) -> u64 {
        match self {
            EthereumNetwork::Mainnet => 1,
            EthereumNetwork::Sepolia => 11155111,
        }
    }
}
``` [1](#0-0) 

`TryFrom<u64>` rejects any chain ID that is not one of these two values, so the type cannot represent a post-fork chain:

```rust
impl TryFrom<u64> for EthereumNetwork {
    fn try_from(value: u64) -> Result<Self, Self::Error> {
        match value {
            1 => Ok(EthereumNetwork::Mainnet),
            11155111 => Ok(EthereumNetwork::Sepolia),
            _ => Err("Unknown Ethereum Network".to_string()),
        }
    }
}
``` [2](#0-1) 

`ethereum_network` is written to `State` at `init` and persisted in stable memory: [3](#0-2) 

The `UpgradeArg` Candid type contains no `ethereum_network` field, so there is no upgrade path to change the chain ID: [4](#0-3) 

Every withdrawal transaction is created with the stored chain ID:

```rust
Ok(Eip1559TransactionRequest {
    chain_id: ethereum_network.chain_id(),
    ...
})
``` [5](#0-4) 

The transaction hash is then memoized inside `SignedEip1559TransactionRequest` and used to match finalization receipts: [6](#0-5) [7](#0-6) 

### Impact Explanation
If Ethereum undergoes a contentious hard fork that assigns a new chain ID to the canonical chain (as happened with ETC/ETH in 2016), the ckETH minter cannot be upgraded to track the new chain ID — it would require a full canister reinstall, wiping all pending withdrawal state. During the window before reinstall:

1. **Withdrawal DoS**: All new EIP-1559 transactions carry the old chain ID. Nodes on the new chain reject them as invalid, so no withdrawal can be finalized. Users' ckETH is locked.
2. **Deposit/withdrawal asymmetry**: If the EVM RPC canister is updated to follow the new chain while the minter still uses the old chain ID, the minter can mint ckETH for deposits observed on the new chain but cannot burn/withdraw on it, breaking the 1:1 peg invariant.
3. **Replay risk**: Signed transactions with the old chain ID remain valid on the legacy chain, enabling replay of minter-signed withdrawals on the fork the minter is no longer tracking.

### Likelihood Explanation
An Ethereum chain fork that changes the canonical chain ID is a rare but historically precedented event (ETH/ETC split). The ckETH minter is a production canister holding real user funds on Ethereum Mainnet (`sv3dd-oaaaa-aaaar-qacoa-cai`). The absence of any upgrade path for `ethereum_network` means the impact window is unbounded until a governance proposal reinstalls the canister. Likelihood is low but non-zero, and the consequence is a complete halt of the withdrawal path.

### Recommendation
1. Add an `ethereum_network: opt EthereumNetwork` field to `UpgradeArg` so the chain ID can be updated via a governance-controlled canister upgrade without losing state.
2. Alternatively, store the raw `chain_id: u64` in `State` (decoupled from the enum) so any post-fork chain ID can be set via upgrade, analogous to OpenZeppelin's EIP-712 pattern of re-reading `block.chainid` on each call.
3. Document explicitly that the minter does not automatically support Ethereum forks, and define the incident-response procedure (governance proposal timeline, state migration plan) in advance.

### Proof of Concept
1. The ckETH minter is initialized with `ethereum_network = Mainnet` (chain ID 1).
2. Ethereum forks; the new canonical chain adopts chain ID 2.
3. `EthereumNetwork::try_from(2)` returns `Err("Unknown Ethereum Network")` — the minter type system cannot represent the new chain.
4. `create_transaction(…, EthereumNetwork::Mainnet)` still embeds `chain_id: 1` in every new `Eip1559TransactionRequest`.
5. The EVM RPC canister is updated to query the new chain; it returns receipts from chain 2.
6. `SignedEip1559TransactionRequest::try_finalize` compares the memoized hash (computed over chain ID 1) against the receipt's `transaction_hash` from chain 2 — they never match, so no withdrawal is ever finalized.
7. All pending ckETH withdrawals are permanently stuck until the canister is reinstalled, losing all in-flight transaction state.

pub struct SignedEip1559TransactionRequest {
    inner: InnerSignedTransactionRequest,
    /// Hash of the signed transaction. Since computation of the hash is an expensive operation,

    /// and memoized. It is safe to memoize the hash because the transaction is immutable.
    /// Note: Serialization should ignore this field and deserialization should call
    /// the constructor to create the correct value.
    memoized_hash: Hash,
}
