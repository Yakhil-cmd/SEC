### Title
Gzip Footer Spoofing Inflates `compilation_cost` in `create_execution_state`, Exhausting Round Instruction Budget — (`rs/embedders/src/wasm_utils/decoding.rs`, `rs/execution_environment/src/hypervisor.rs`)

---

### Summary

An unprivileged canister developer can submit a minimal valid gzip-compressed Wasm module whose RFC-1952 footer encodes `0xFFFFFFFF` as the uncompressed size. `decoded_wasm_size` returns this attacker-controlled value without any cap. `create_execution_state` multiplies it by `cost_to_compile_wasm_instruction` (6,000) to produce a `compilation_cost` of ~25.7 trillion instructions. When `decode_wasm` subsequently rejects the module as `ModuleTooLarge`, the error path subtracts this inflated cost from `round_limits.instructions`, driving it deeply negative and exhausting the round instruction budget for subnet messages in that execution round.

---

### Finding Description

**Step 1 — Attacker-controlled size with no guard in `decoded_wasm_size`:** [1](#0-0) [2](#0-1) 

The last 4 bytes of any gzip stream are read as a `u32` and cast directly to `usize`. The function's own doc-comment warns the value "cannot be trusted" and "could have been manipulated," yet no cap is applied before returning it.

**Step 2 — Inflated `compilation_cost` computed before `decode_wasm` is called:** [3](#0-2) 

`wasm_size` is set to `max(4_294_967_295, actual_compressed_len)` = `4_294_967_295`. With `cost_to_compile_wasm_instruction = 6_000`:

```
compilation_cost = 6_000 × 4_294_967_295 = 25_769_803_770_000
``` [4](#0-3) 

**Step 3 — `decode_wasm` rejects the module as `ModuleTooLarge`:** [5](#0-4) 

`4_294_967_295 > 104_857_600` (100 MiB), so `decode_wasm` returns `Err(ModuleTooLarge)` immediately, before any actual decompression.

**Step 4 — Error path subtracts the inflated cost from `round_limits.instructions`:** [6](#0-5) 

The `compilation_cost` used here is the one computed from the spoofed footer (25.7 trillion), not the actual compilation cost. `as_round_instructions` converts it to `i64` without saturation (25.7 trillion fits in `i64`): [7](#0-6) 

The round instruction budget for subnet messages is `max_instructions_per_round / 16 = 250M`. Subtracting 25.7 trillion drives it to approximately `-25_769_553_770_000`, making `instructions_reached()` return `true`: [8](#0-7) 

**Step 5 — Scheduler stops processing subnet messages:** [9](#0-8) 

All subsequent `install_code` and other subnet messages in the same round are skipped.

---

### Impact Explanation

- Any canister developer (unprivileged) can submit a crafted gzip Wasm via `install_code`.
- One such call per round exhausts the subnet message instruction budget for that round.
- All other `install_code`, `update_settings`, and similar subnet messages queued in the same round are starved.
- The attack is repeatable every round, creating a sustained DoS on subnet management operations.
- The `install_code` itself fails with `ModuleTooLarge` (no canister state change), so the attacker pays only the cycles for the rejected call, making this cheap to repeat.

---

### Likelihood Explanation

The exploit requires only: a valid gzip header (`\x1f\x8b\x08`) + any deflate body ≥ 8 bytes + footer `\xff\xff\xff\xff` + any 4-byte CRC32. This is trivially constructable. No privileged access, no key material, no social engineering. The path is deterministic and locally reproducible.

---

### Recommendation

Cap the value returned by `decoded_wasm_size` (or the `wasm_size` derived from it in `create_execution_state`) to `WASM_MAX_SIZE` before computing `compilation_cost`. Specifically, in `create_execution_state`:

```rust
let wasm_size = wasm_size.min(config.wasm_max_size.get() as usize);
let compilation_cost = self.cost_to_compile_wasm_instruction * wasm_size as u64;
```

This mirrors the guard already present in `decode_wasm` and ensures the pre-decompression cost estimate cannot exceed the cost of a legitimately maximum-sized module.

---

### Proof of Concept

```rust
// Craft a minimal gzip with footer = 0xFFFFFFFF
let mut wasm: Vec<u8> = vec![
    0x1f, 0x8b, 0x08, 0x00, // magic + method
    0x00, 0x00, 0x00, 0x00, // mtime
    0x00, 0xff,             // xfl + OS
    0x03, 0x00,             // empty deflate stream (BFINAL=1, BTYPE=00)
    0x00, 0x00, 0x00, 0x00, // CRC32 (any value)
    0xff, 0xff, 0xff, 0xff, // ISIZE = u32::MAX
];
// Submit via install_code ingress message.
// Observe: round_limits.instructions drops by ~25.7 trillion.
// All other install_code messages in the same round are not executed.
```
