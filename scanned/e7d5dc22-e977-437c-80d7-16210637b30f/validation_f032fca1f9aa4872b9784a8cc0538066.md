### Title
No Mechanism to Remove Supported ckERC20 Tokens from Minter and Ledger Suite Orchestrator - (`rs/ethereum/cketh/minter/src/state.rs`, `rs/ethereum/ledger-suite-orchestrator/ledger_suite_orchestrator.did`)

---

### Summary

The ckERC20 chain-fusion system provides a one-way `AddErc20Arg` path to register ERC-20 tokens for bridging, but neither the Ledger Suite Orchestrator (LSO) nor the ckETH minter exposes any mechanism to remove or disable a previously registered ckERC20 token. Once a token is added, it is permanently tracked and actively processed by the minter (log scraping, minting, withdrawals) with no selective off-switch available even to NNS governance.

---

### Finding Description

The `OrchestratorArg` Candid variant in the LSO only defines three arms:

```
type OrchestratorArg = variant {
    UpgradeArg : UpgradeArg;
    InitArg : InitArg;
    AddErc20Arg : AddErc20Arg;
};
```

There is no `RemoveErc20Arg` or equivalent. [1](#0-0) 

On the orchestrator side, `ManagedCanisters` only exposes `insert_once()` for adding a token; there is no removal method. Once `record_new_erc20_token()` is called, the entry is permanent in stable state. [2](#0-1) [3](#0-2) 

On the minter side, `record_add_ckerc20_token()` inserts into `self.ckerc20_tokens` with no corresponding removal function: [4](#0-3) 

The minter's public endpoint `add_ckerc20_token` (callable only by the orchestrator) has no counterpart `remove_ckerc20_token`: [5](#0-4) 

The event log in `audit.rs` records `AddedCkErc20Token` transitions but defines no `RemovedCkErc20Token` event type, meaning even a state-migration upgrade cannot cleanly replay a removal: [6](#0-5) 

The minter uses `supported_ck_erc20_tokens()` to drive active log scraping for every registered ERC-20 contract address. A registered token cannot be excluded from this loop without removing it from `ckerc20_tokens`, which is impossible through any current interface: [7](#0-6) 

---

### Impact Explanation

If a supported ERC-20 token's contract is compromised (e.g., its upgrade key is stolen and the contract is modified to emit fraudulent `Deposit` events), the minter will continue to scrape those logs and mint ckERC20 tokens backed by nothing. NNS governance has no way to selectively halt processing for that one token. The only recourse is to stop the entire minter canister — halting ALL ckETH and ckERC20 operations — or perform an emergency state-migration upgrade that manually rewrites stable memory, a complex and error-prone operation under time pressure. This can result in unbacked ckERC20 tokens being minted and circulating on ICP, breaking the 1:1 backing guarantee for all holders of that token.

---

### Likelihood Explanation

The supported tokens (USDC, USDT, WBTC, LINK, etc.) are high-value targets. ERC-20 contracts with upgradeable proxies (e.g., USDC uses a proxy pattern) can have their logic replaced. A compromised ERC-20 contract that emits synthetic `Deposit` log events would be processed by the minter with no ability to stop it selectively. The likelihood of any single supported token being compromised is low but non-zero, and the consequence of having no emergency removal path is a systemic risk to the integrity of all ckERC20 balances.

---

### Recommendation

1. Add a `RemoveErc20Arg` variant to `OrchestratorArg` in `ledger_suite_orchestrator.did` that accepts an `Erc20Contract` identifier.
2. Implement a corresponding removal path in `lifecycle/mod.rs` that removes the entry from `ManagedCanisters` and notifies the minter.
3. Add a `remove_ckerc20_token` endpoint to the minter (callable only by the orchestrator) that removes the entry from `ckerc20_tokens` and records a `RemovedCkErc20Token` event in the audit log.
4. Ensure the log-scraping loop in the minter respects the updated token list immediately after removal.

---

### Proof of Concept

1. NNS governance submits an `AddErc20Arg` upgrade proposal for a token (e.g., a hypothetical `ckFOO` backed by ERC-20 at address `0xFOO`). The orchestrator calls `add_ckerc20_token` on the minter; `ckerc20_tokens` now contains `0xFOO`.
2. The ERC-20 contract at `0xFOO` is later compromised and begins emitting synthetic `ReceivedErc20` log events.
3. The minter's scraping loop (`supported_ck_erc20_tokens()` → scrape logs for `0xFOO`) picks up these events and calls `record_event_to_mint`, minting unbacked `ckFOO` to attacker-controlled ICP principals.
4. NNS governance attempts to stop this by submitting an upgrade proposal — but `OrchestratorArg` has no `RemoveErc20Arg`, and the minter has no `remove_ckerc20_token` endpoint. The only option is to stop the entire minter canister, halting all ckETH and ckERC20 operations for all users. [1](#0-0) [7](#0-6) [5](#0-4)

    pub fn record_add_ckerc20_token(&mut self, ckerc20_token: CkErc20Token) {
        assert_eq!(
            self.ethereum_network, ckerc20_token.erc20_ethereum_network,
            "ERROR: Expected {}, but got {}",
            self.ethereum_network, ckerc20_token.erc20_ethereum_network
        );
        let ckerc20_with_same_symbol = self
            .supported_ck_erc20_tokens()
            .filter(|ckerc20| ckerc20.ckerc20_token_symbol == ckerc20_token.ckerc20_token_symbol)
            .collect::<Vec<_>>();
        assert_eq!(
            ckerc20_with_same_symbol,
            vec![],
            "ERROR: ckERC20 token symbol {} is already used by {:?}",
            ckerc20_token.ckerc20_token_symbol,
            ckerc20_with_same_symbol
        );
        assert_eq!(
            self.ckerc20_tokens.try_insert(
                ckerc20_token.ckerc20_ledger_id,
                ckerc20_token.erc20_contract_address,
                ckerc20_token.ckerc20_token_symbol,
            ),
            Ok(()),
            "ERROR: some ckERC20 tokens use the same ckERC20 ledger ID or ERC-20 address"

#[update]
async fn add_ckerc20_token(erc20_token: AddCkErc20Token) {
    let orchestrator_id = read_state(|s| s.ledger_suite_orchestrator_id)
        .unwrap_or_else(|| ic_cdk::trap("ERROR: ERC-20 feature is not activated"));
    if orchestrator_id != ic_cdk::api::msg_caller() {
        ic_cdk::trap(format!(
            "ERROR: only the orchestrator {orchestrator_id} can add ERC-20 tokens"
        ));
    }
    let ckerc20_token = erc20::CkErc20Token::try_from(erc20_token)
        .unwrap_or_else(|e| ic_cdk::trap(format!("ERROR: {e}")));
    mutate_state(|s| process_event(s, EventType::AddedCkErc20Token(ckerc20_token)));
}
