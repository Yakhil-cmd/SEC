### Title
`ManageVotingPermission` Can Grant Itself to Other Principals Beyond Its Documented Scope - (`rs/sns/governance/src/neuron.rs`)

### Summary

In SNS governance, the `ManageVotingPermission` neuron permission is documented to allow a principal to "grant/revoke permission to vote and submit proposals on behalf of the neuron to other principals." However, because `ManageVotingPermission` is included in the `PERMISSIONS_RELATED_TO_VOTING` constant, a holder of only `ManageVotingPermission` can also grant `ManageVotingPermission` itself to arbitrary other principals. This is a broader permission than what the documentation describes, directly analogous to the external report's finding that `DEPOSIT_VAULT_ADMIN_ROLE` could pause vaults beyond its documented scope.

### Finding Description

`Neuron::PERMISSIONS_RELATED_TO_VOTING` is defined as:

```rust
pub const PERMISSIONS_RELATED_TO_VOTING: &'static [NeuronPermissionType] = &[
    NeuronPermissionType::Vote,
    NeuronPermissionType::SubmitProposal,
    NeuronPermissionType::ManageVotingPermission,  // <-- includes itself
];
``` [1](#0-0) 

The authorization check in `check_principal_authorized_to_change_permissions` allows a caller with `ManageVotingPermission` to add or remove any permission that `is_exclusively_voting_related()`:

```rust
let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
    vec![
        NeuronPermissionType::ManagePrincipals,
        NeuronPermissionType::ManageVotingPermission,
    ]
} else {
    vec![NeuronPermissionType::ManagePrincipals]
};
``` [2](#0-1) 

`is_exclusively_voting_related()` checks membership against `PERMISSIONS_RELATED_TO_VOTING`:

```rust
pub fn is_exclusively_voting_related(&self) -> bool {
    let permissions_related_to_voting = Neuron::PERMISSIONS_RELATED_TO_VOTING
        .iter()
        .map(|p| *p as i32)
        .collect::<Vec<_>>();
    self.permissions
        .iter()
        .all(|p| permissions_related_to_voting.contains(p))
}
``` [3](#0-2) 

Because `ManageVotingPermission` is itself in `PERMISSIONS_RELATED_TO_VOTING`, a request to add `ManageVotingPermission` to another principal passes the `is_exclusively_voting_related()` check, and a caller holding only `ManageVotingPermission` is authorized to perform it. The `add_neuron_permissions` function then applies a second check against `neuron_grantable_permissions`: [4](#0-3) 

When `ManageVotingPermission` is included in `neuron_grantable_permissions` (which is the case in the default SNS configuration and any SNS that grants all permissions), a principal holding only `ManageVotingPermission` can grant `ManageVotingPermission` to an unlimited number of other principals. This is confirmed by the existing test: [5](#0-4) 

The documentation for `ManageVotingPermission` states it grants "permission to vote and submit proposals on behalf of the neuron to other principals" — it does not mention the ability to propagate `ManageVotingPermission` itself. [6](#0-5) 

### Impact Explanation

A neuron owner who grants `ManageVotingPermission` to a hotkey (e.g., a trading bot or a third-party service) intends to allow that hotkey to manage who can vote and submit proposals. However, the hotkey can also grant `ManageVotingPermission` to arbitrary additional principals, who can in turn grant it to yet more principals. This allows unbounded propagation of the `ManageVotingPermission` role beyond the neuron owner's intent, potentially allowing principals the neuron owner never authorized to gain the ability to manage voting permissions on the neuron. This breaks the documented permission boundary.

### Likelihood Explanation

Any SNS that includes `ManageVotingPermission` in its `neuron_grantable_permissions` (a common configuration) is affected. A neuron owner who grants `ManageVotingPermission` to a hotkey — a standard operational pattern — is exposed. The entry path requires only a standard `manage_neuron` ingress call with `AddNeuronPermissions`, which is a normal user-facing operation.

### Recommendation

Remove `ManageVotingPermission` from `PERMISSIONS_RELATED_TO_VOTING`. A separate, explicit check should be added if a `ManageVotingPermission` holder is intended to be able to propagate that permission. Alternatively, the documentation should be updated to explicitly state that `ManageVotingPermission` holders can grant `ManageVotingPermission` to others, so neuron owners are aware of this when delegating.

### Proof of Concept

The existing test `test_manage_voting_permission_allows_adding_permissions_related_to_voting` in `rs/sns/governance/tests/governance.rs` already demonstrates this: a caller with only `ManageVotingPermission` successfully adds all `PERMISSIONS_RELATED_TO_VOTING` (including `ManageVotingPermission` itself) to another principal. [5](#0-4)

fn test_manage_voting_permission_allows_adding_permissions_related_to_voting() {

    let permissions_to_add: NeuronPermissionList =
        Neuron::PERMISSIONS_RELATED_TO_VOTING.to_vec().into();

        let permissions: &[(PrincipalId, NeuronPermissionList)] = &[(
            caller,
            vec![NeuronPermissionType::ManageVotingPermission].into(),

        let user_principal = PrincipalId::new_user_test_id(0);

            .with_neuron_grantable_permissions(NeuronPermissionList::all())
            .add_neuron_with_permissions(permissions, neuron_id.clone())
            .create();

        (governance_fixture, neuron_id)
    };

    // Attempt to add voting-related permissions to `target` - should succeed
    // since `caller` has ManageVotingPermission.
    governance
