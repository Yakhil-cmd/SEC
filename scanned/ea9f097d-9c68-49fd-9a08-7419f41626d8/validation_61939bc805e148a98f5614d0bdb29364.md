### Title
Treasury Deposit 50% Safety Limit Bypassed via Concurrent Proposals — Missing Execution-Time Re-validation - (File: `rs/sns/governance/src/extensions.rs`)

### Summary

The SNS treasury deposit safety check — which is intended to prevent more than 50% of the treasury from being committed to an external DEX in a single operation — is evaluated only at proposal submission time against a point-in-time ledger balance snapshot. Because no equivalent check is performed at execution time and no concurrency guard exists for `ExecuteExtensionOperation` deposit proposals, two simultaneously submitted proposals can each independently pass the 50% check and collectively drain 100% of the treasury into the treasury manager.

### Finding Description

`validate_deposit_operation_impl` in `rs/sns/governance/src/extensions.rs` is the sole enforcement point for the 50% treasury deposit limit. It queries the live ledger balance of the SNS governance treasury account and rejects any deposit request exceeding half that balance: [1](#0-0) 

This validation runs at **proposal submission time** (inside `validate_execute_extension_operation`, which is called during proposal creation). The actual fund movement happens later, inside `execute_treasury_manager_deposit`, which calls `approve_treasury_manager` and then `deposit` on the extension canister — with no re-validation of the 50% limit: [2](#0-1) 

The `TreasuryManager` trait explicitly documents that the `balances` query returns a **locally cached** view of balances, updated only periodically by `refresh_balances`: [3](#0-2) 

The `BalanceBook` type acknowledges that funds can be "in transit" via the `suspense` field, and the protocol invariant `suspense[k] == 0` is only guaranteed under "normal operations": [4](#0-3) 

This is structurally identical to the external report's `totalAssets()` problem: the balance figure used for the safety check does not account for funds already committed to pending proposals (analogous to funds "outside the vault but not yet in a UniswapV3 position").

By contrast, the older `TransferSnsTreasuryFunds` action has both an execution-time re-check (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`) and a mutex (`IN_PROGRESS_PROPOSAL_ID`) preventing concurrent execution: [5](#0-4) 

No equivalent protections exist for `ExecuteExtensionOperation` deposit proposals.

### Impact Explanation

An SNS neuron holder can submit two deposit proposals simultaneously, each requesting exactly 50% of the treasury. Both pass `validate_deposit_operation_impl` independently because each sees the full treasury balance. After governance voting, both execute sequentially, collectively committing 100% of the SNS and ICP treasury to the treasury manager (external DEX). The 50% safety limit — the only protection against over-committing treasury assets — is completely bypassed. Funds are not lost outright (they move to the treasury manager), but the entire treasury is now locked in an external DEX position, and the SNS DAO loses direct control over its assets until a withdrawal proposal is executed.

**Impact: High** — the entire treasury can be committed to an external custodian against the intended 50% limit.

### Likelihood Explanation

**Likelihood: Low** — exploiting this requires:
1. Submitting two proposals that both pass governance voting (requires sufficient voting power or community approval).
2. The proposals must be submitted before either is executed.

However, no special privilege beyond normal neuron ownership is required. The SNS governance voting period can span days, making it feasible for two proposals to coexist in the voting queue simultaneously. A motivated actor with sufficient voting power (or one who can convince the community) can trigger this without any admin key or privileged access.

### Recommendation

1. **Add an execution-time re-validation**: In `execute_treasury_manager_deposit`, re-query the current treasury ledger balance and verify the requested amount still satisfies the 50% limit before calling `approve_treasury_manager`.

2. **Add a concurrency guard**: Introduce a mutex (similar to `IN_PROGRESS_PROPOSAL_ID` used for `TransferSnsTreasuryFunds`) to prevent two `ExecuteExtensionOperation` deposit proposals from executing concurrently.

3. **Track committed-but-unexecuted amounts**: At proposal submission time, sum the amounts requested by all open/adopted deposit proposals and include that committed amount when evaluating the 50% limit, so the check reflects total committed treasury exposure, not just the instantaneous ledger balance.

### Proof of Concept

1. SNS treasury holds 1,000 ICP and 500 SNS tokens.
2. Neuron holder submits **Proposal A**: deposit 500 ICP + 250 SNS (exactly 50% of each). `validate_deposit_operation_impl` queries the ledger: 1,000 ICP ≥ 2×500 ✓, 500 SNS ≥ 2×250 ✓. Proposal A is accepted.
3. Before Proposal A is executed, neuron holder submits **Proposal B**: deposit 500 ICP + 250 SNS (again 50%). `validate_deposit_operation_impl` queries the ledger again — the balance is still 1,000 ICP / 500 SNS (Proposal A has not yet executed). Proposal B is accepted.
4. Both proposals pass governance voting.
5. Proposal A executes: `approve_treasury_manager` grants a 500 ICP + 250 SNS ICRC-2 allowance; `deposit` is called; treasury manager pulls the funds into the DEX.
6. Proposal B executes: `approve_treasury_manager` grants another 500 ICP + 250 SNS allowance; `deposit` is called; treasury manager pulls the remaining funds.
7. Result: 100% of the treasury (1,000 ICP + 500 SNS) is now held by the external custodian (DEX), violating the 50% safety limit.

The root cause is that `validate_deposit_operation_impl` reads a point-in-time ledger balance snapshot without accounting for funds already committed to other pending proposals, and `execute_treasury_manager_deposit` performs no re-check at execution time. [6](#0-5) [7](#0-6)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
