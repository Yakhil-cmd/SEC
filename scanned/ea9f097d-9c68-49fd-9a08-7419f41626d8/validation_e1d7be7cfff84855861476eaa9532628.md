### Title
Unprivileged Caller Can Invoke `refresh_buyer_tokens` for Any Buyer, Forcing Participation and Preventing Cancellation - (`rs/sns/swap/canister/canister.rs`)

### Summary
The `refresh_buyer_tokens` endpoint in the SNS Swap canister accepts an arbitrary `buyer` principal without verifying that the caller is that buyer. Any unprivileged ingress sender can trigger participation registration for any other user, deleting their open ticket and preventing them from canceling their participation before the swap closes.

### Finding Description
In `rs/sns/swap/canister/canister.rs`, the `refresh_buyer_tokens` update method resolves the target buyer from the request field rather than from the caller:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
    };
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

There is no check that `caller == buyer`. The `buyer` field in `RefreshBuyerTokensRequest` is documented as "If not specified, the caller is used," but any non-empty string is accepted without authorization. [2](#0-1) 

Inside `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`, when the buyer's ICP subaccount balance is sufficient to satisfy the open ticket amount, the ticket is unconditionally deleted and participation is registered:

```rust
if amount_ticket > requested_increment_e8s {
    return Err(...);
}
// Delete fully executed ticket
memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow_mut().remove(&principal));
``` [3](#0-2) 

The `notify_payment_failure` function, which is the only way for a user to voluntarily cancel their pending participation (delete their own ticket), operates on the caller's principal:

```rust
pub fn notify_payment_failure(&mut self, caller: &PrincipalId) -> NotifyPaymentFailureResponse {
    let principal = Blob::from_bytes(caller.as_slice().into());
    ...
    memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow_mut().remove(&principal));
``` [4](#0-3) 

Once the ticket is deleted by the attacker's call to `refresh_buyer_tokens`, the victim's participation is irrevocably registered. The victim can no longer call `notify_payment_failure` to cancel (the ticket is already gone), and there is no mechanism to withdraw registered participation during the OPEN phase.

The `confirmation_text` field provides no meaningful protection: it is a public value set at SNS initialization, readable by anyone, and the attacker simply includes it in the request. [5](#0-4) 

### Impact Explanation
A malicious actor who knows a victim's principal (all principals are public on-chain) can:
1. Call `refresh_buyer_tokens` with `buyer = <victim>` at any time during the OPEN phase, provided the victim has already transferred ICP to their subaccount and holds an open ticket.
2. The victim's participation is registered and their ticket deleted without their consent.
3. The victim can no longer cancel via `notify_payment_failure`.
4. If the swap commits, the victim's ICP is swept to SNS governance and they receive SNS tokens they did not intend to acquire. Their ICP is not recoverable via `error_refund_icp` (which only refunds unregistered ICP).
5. If the swap aborts, the victim recovers ICP, so harm is conditional on the swap committing.

This is a governance authorization bug: a state-mutating operation that should be restricted to the resource owner is callable by any unprivileged ingress sender.

### Likelihood Explanation
The preconditions are routinely met during any active SNS swap:
- Victim's principal is public.
- Victim transfers ICP before calling `refresh_buyer_tokens` (the normal participation flow).
- The confirmation text (if any) is a public string visible in the SNS init config.
- No cycles, tokens, or special permissions are required by the attacker.

The attack window is the entire OPEN phase of the swap. It does not require front-running or a mempool; the attacker can act at any point after the victim transfers ICP.

### Recommendation
Add a caller-equals-buyer check in the canister endpoint before dispatching to `refresh_buyer_token_e8s`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Only the buyer can refresh their own tokens");
        }
        buyer
    };
    ...
}
```

Alternatively, remove the `buyer` field entirely and always use `caller_principal_id()`, since the only legitimate use case for specifying a different buyer is to help another user — which can be accomplished by the user themselves.

### Proof of Concept
1. User A calls `new_sale_ticket` to create a ticket for amount X ICP.
2. User A transfers X ICP to their swap subaccount on the ICP ledger.
3. User A changes their mind and intends to call `notify_payment_failure` to delete their ticket and later reclaim ICP via `error_refund_icp` after the swap closes.
4. Attacker (any principal) calls `refresh_buyer_tokens` with `buyer = User_A_principal` (and the correct `confirmation_text` if required, which is public).
5. Inside `refresh_buyer_token_e8s`: balance = X ≥ ticket amount X → ticket deleted, participation registered for User A.
6. User A calls `notify_payment_failure` → returns `None` (no ticket found); cancellation is impossible.
7. Swap commits → User A's ICP is swept to SNS governance; User A receives unwanted SNS tokens and cannot recover ICP. [1](#0-0) [6](#0-5) [3](#0-2) [4](#0-3)
