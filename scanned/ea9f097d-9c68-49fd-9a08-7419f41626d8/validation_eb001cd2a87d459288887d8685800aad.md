### Title
Stale ICP/XDR Exchange Rate Consumed Without Timestamp Staleness Check in SNS Treasury Valuation — (`rs/sns/governance/token_valuation/src/lib.rs`)

---

### Summary

The `CmcBased30DayMovingAverageXdrsPerIcpClient` in `rs/sns/governance/token_valuation/src/lib.rs` fetches the ICP/XDR conversion rate from the Cycles Minting Canister (CMC) to value SNS treasury assets for `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals. After receiving the response, the code extracts `xdr_permyriad_per_icp` directly without checking whether the rate's `timestamp_seconds` is recent. If the CMC's stored rate is arbitrarily old (e.g., the CMC's own XRC update pipeline has been stalled or disabled), the SNS governance canister will silently use a stale price to compute treasury transfer limits, potentially allowing a much larger or smaller transfer than the current market price would permit.

---

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, the `get` method of `CmcBased30DayMovingAverageXdrsPerIcpClient` calls `get_average_icp_xdr_conversion_rate` on the CMC and immediately uses the returned `xdr_permyriad_per_icp` value:

```rust
let xdr_per_icp =
    Decimal::from(response.data.xdr_permyriad_per_icp) * *UNITS_PER_PERMYRIAD;
Ok(xdr_per_icp)
``` [1](#0-0) 

There is no check that `response.data.timestamp_seconds` is within an acceptable window of the current time. The comment in the code explicitly notes that certificate validation is skipped for inter-canister calls:

```rust
// No need to validate the cerificate in response, because query is not used in this
// case (specifically, canister A in subnet X is calling (another) canister B in
// (another) subnet Y).
``` [2](#0-1) 

The CMC's `do_set_icp_xdr_conversion_rate` only enforces that a new rate has a strictly greater timestamp than the current one, and that the rate is non-zero: [3](#0-2) 

There is no upper bound on how old the stored rate can be. The CMC's automatic XRC update mechanism can be disabled via a governance proposal (`UpdateExchangeRateState::Disabled`): [4](#0-3) 

When disabled, the CMC stops refreshing its rate, and the stored `average_icp_xdr_conversion_rate` can become arbitrarily stale. The SNS governance canister's `validate_and_render_transfer_sns_treasury_funds` and `validate_and_render_mint_sns_tokens` both call `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which calls `try_get_icp_balance_valuation` / `try_get_sns_token_balance_valuation`, which in turn calls `new_standard_xdrs_per_icp_client`: [5](#0-4) 

The resulting `Valuation` is stored in `action_auxiliary` at proposal submission time and re-used at execution time to enforce the 7-day treasury transfer limit: [6](#0-5) 

---

### Impact Explanation

The ICP/XDR rate is the key input to `ProposalsAmountTotalUpperBound`, which determines whether an SNS treasury transfer or mint is within the allowed 7-day window limit. The three regimes (small/medium/large treasury) and the absolute cap of 300,000 XDR all depend on this rate: [7](#0-6) 

If the rate is stale and artificially low (e.g., ICP was worth 2 XDR a year ago but is now worth 10 XDR), the treasury is classified as "small" and the `NoLimit` branch applies — allowing the full treasury balance to be transferred in a single 7-day window, bypassing the intended 25% or 300,000 XDR caps. Conversely, a stale high rate would over-restrict transfers. The `MIN_XDRS_PER_ICP = 1` floor only partially mitigates this; it does not bound staleness.

---

### Likelihood Explanation

The CMC's XRC update can be disabled by an NNS governance proposal with `DivergedRate` reason, which is a legitimate operational action. Additionally, the CMC's heartbeat-driven update pipeline (`update_exchange_rate`) can fail silently for extended periods if the XRC canister is unavailable. In either case, the SNS governance canister has no independent way to detect that the rate it received is stale. Any SNS neuron holder who can submit a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal during a period of rate staleness can exploit the miscalculated limit. The attacker-controlled entry path is: submit a proposal during a stale-rate window → proposal passes governance vote → execution uses stale valuation → transfer limit is miscalculated.

---

### Recommendation

In `CmcBased30DayMovingAverageXdrsPerIcpClient::get`, after receiving the CMC response, validate that `response.data.timestamp_seconds` is within a configurable maximum age (e.g., 48 hours) relative to the current canister time. If the rate is too old, return a `ValuationError::new_external` rather than silently using the stale value. Example:

```rust
let now_seconds = ic_cdk::api::time() / 1_000_000_000;
let max_age_seconds: u64 = 48 * 3600;
if now_seconds.saturating_sub(response.data.timestamp_seconds) > max_age_seconds {
    return Err(ValuationError::new_external(format!(
        "ICP/XDR rate from CMC is stale: timestamp={}, now={}",
        response.data.timestamp_seconds, now_seconds,
    )));
}
```

Additionally, consider adding a zero-rate guard analogous to the one in `do_set_icp_xdr_conversion_rate`. [8](#0-7) 

---

### Proof of Concept

1. NNS governance passes a `DivergedRate` proposal, setting `UpdateExchangeRateState::Disabled` in the CMC. The CMC stops refreshing its ICP/XDR rate. [4](#0-3) 

2. Time passes. The CMC's `average_icp_xdr_conversion_rate.timestamp_seconds` is now days or weeks old. The stored rate reflects a historical (low) ICP price.

3. An SNS neuron holder submits a `TransferSnsTreasuryFunds` proposal. SNS governance calls `try_get_icp_balance_valuation`, which calls `get_average_icp_xdr_conversion_rate` on the CMC. [9](#0-8) 

4. The stale low rate causes `valuation.to_xdr()` to return a value ≤ 100,000 XDR, placing the treasury in the "small" regime → `ProposalsAmountTotalUpperBound::NoLimit`. [10](#0-9) 

5. The proposal passes and executes. `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` uses the stale valuation stored at submission time and allows the full treasury balance to be transferred, bypassing the intended 25%/300,000 XDR cap. [6](#0-5)

impl ProposalsAmountTotalUpperBound {
    // A treasury can be small, medium, or large. These are the boundaries between those regimes.
    const MAX_SMALL_TREASURY_SIZE_XDR: Decimal = dec!(100_000);
    const MAX_MEDIUM_TREASURY_SIZE_XDR: Decimal = dec!(1_200_000);

    // No matter how large the treasury is, not more than this amount can be removed (within a 7 day
    // window).
    const MAX_XDR: Decimal = dec!(300_000);

    /// A price quote less than this is considered "unrealistically" low. When that happens, we use
    /// this instead of the quoted value.
    ///
    /// # Motivation
    ///
    /// Low XDRs per ICP quotes would tend to cause our valuations to be in the "small" regime,
    /// where an SNS is allowed to take the biggest actions relative to their size. This is to
    /// minmize the damage caused by wacky price quotes.
    ///
    /// # What Value to Use
    ///
    /// Currently, the minimum XDRs per ICP used by NNS governance is 1. This is simply copied from
    /// there, specifically from the minimum_icp_xdr_rate field in NetworkEconomics.
    ///
    /// As of Mar 2024, the price of ICP is around 10 XDR. The lowest it has ever been is around 2.2
    /// XDR. FWIW, this is less than that.
    ///
    /// # Why Not Also Define MAX?
    ///
    /// Currently, we do not have/enforce a MAX_XDRS_PER_ICP, because this would tend to cause our
    /// valuations to be in the "large" regime, where actions are more limited.
    const MIN_XDRS_PER_ICP: Decimal = dec!(1);
