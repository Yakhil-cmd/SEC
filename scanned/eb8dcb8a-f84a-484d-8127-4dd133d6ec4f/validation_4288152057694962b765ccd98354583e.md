### Title
Voting Power Spike Detection Permanently Frozen After First Spike — (`rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`)

### Summary

The `SnapshotVotingPowerTask` checks whether the **latest stored** snapshot is a spike before deciding to take a new one. Once a spike snapshot is recorded, the task permanently stops recording new snapshots. After `MAXIMUM_STALENESS_SECONDS` (3 months), all stored snapshots are filtered out as stale, silently disabling the spike detection mechanism entirely. An attacker who can cause a single voting power spike can exploit this to permanently neutralize the anti-frontrunning defense and later submit proposals with inflated voting power that goes unchecked.

### Finding Description

`SnapshotVotingPowerTask::execute` begins by calling `is_latest_snapshot_a_spike`: [1](#0-0) 

If that returns `true`, the task returns immediately without recording a new snapshot: [2](#0-1) 

`is_latest_snapshot_a_spike` reads the **last key-value** from the stable `voting_power_totals` map and checks whether that stored value exceeds 1.5× the minimum across all stored snapshots: [3](#0-2) 

Because the task never computes the **current** voting power before deciding to skip, and because the spike snapshot remains the latest stored entry, every subsequent daily execution of the task will see the same spike and skip again — indefinitely. The snapshots are never refreshed.

`MAXIMUM_STALENESS_SECONDS` is set to 3 months: [4](#0-3) 

The staleness filter in `totals_entry_with_minimum_total_potential_voting_power_if_voting_power_spiked` silently drops all entries older than this threshold: [5](#0-4) 

Once all entries are dropped, `previous_ballots_if_voting_power_spike_detected` returns `None` for any input, and `compute_ballots_for_standard_proposal` falls through to the "normal case" path, using the live (potentially inflated) voting power for every new proposal: [6](#0-5) 

There is no production code path that calls `VotingPowerSnapshots::clear()` or otherwise resets the frozen state; the only call to `clear()` is in a test marked `TODO(NNS1-4323): Remove this test after Mission 70 is fully deployed`: [7](#0-6) 

The task is correctly registered in `schedule_tasks`: [8](#0-7) 

The bug is not that the task is never scheduled — it is that the task's own guard condition causes it to permanently self-suppress after the first spike snapshot is written.

### Impact Explanation

The voting power spike detection is the NNS governance canister's primary defense against a flash-loan-style attack: an actor temporarily acquires a large ICP stake, submits a proposal, and the system falls back to a pre-spike snapshot for ballot creation, neutralizing the inflated voting power. Once the snapshots are stale, `compute_ballots_for_standard_proposal` uses the live neuron store directly: [9](#0-8) 

With spike detection disabled, a proposal submitted while the attacker holds inflated voting power will have ballots created from that inflated power, potentially allowing the attacker to unilaterally adopt or block NNS proposals — including proposals that control subnet upgrades, canister installations, and economic parameters of the entire Internet Computer.

### Likelihood Explanation

The attacker must:
1. Acquire enough ICP to push total potential voting power above 1.5× the historical minimum (a large but finite threshold on a public market).
2. Stake it in neurons with sufficient dissolve delay so it counts toward `compute_voting_power_snapshot_for_standard_proposal`.
3. Wait up to 24 hours for the daily task to record the spike snapshot.
4. Hold the position for 3 months until all prior snapshots expire.

Steps 1–4 represent a significant but not impossible economic commitment for a well-resourced adversary (e.g., an exchange, a large fund, or a coalition). The attack is fully permissionless — no privileged role is required. Likelihood is **low-to-medium** given the capital requirement, but the consequence of success is critical.

### Recommendation

Replace the pre-skip check with a check on the **current** computed voting power rather than the latest stored snapshot. Specifically, `execute` should:

1. Compute the current `VotingPowerSnapshot`.
2. Check whether that current snapshot would be a spike relative to the stored history.
3. Only skip recording if the **current** power is a spike (to avoid polluting the history with spike data).
4. If the current power is normal, record it — even if the latest stored snapshot was a spike — so the history can recover.

Additionally, introduce a bounded recovery window: if no snapshot has been recorded for longer than, say, 2× `VOTING_POWER_SNAPSHOT_INTERVAL`, force a new snapshot regardless of spike status, to prevent permanent freeze.

impl RecurringSyncTask for SnapshotVotingPowerTask {
    fn execute(self) -> (Duration, Self) {
        let now_seconds = self
            .governance
            .with_borrow(|governance| governance.env.now());
        if self
            .snapshots
            .with_borrow(|snapshots| snapshots.is_latest_snapshot_a_spike(now_seconds))
        {
            return (VOTING_POWER_SNAPSHOT_INTERVAL, self);
        }

        let voting_power_snapshot = self.governance.with_borrow_mut(|governance| {
            let voting_power_economics = governance.voting_power_economics();
            governance
                .neuron_store
                .compute_voting_power_snapshot_for_standard_proposal(
                    voting_power_economics,
                    now_seconds,
                )
                .expect("Voting power snapshot failed")
        });

        self.snapshots.with_borrow_mut(|snapshots| {
            snapshots.record_voting_power_snapshot(now_seconds, voting_power_snapshot);
        });

        (VOTING_POWER_SNAPSHOT_INTERVAL, self)
    }

    pub fn is_latest_snapshot_a_spike(&self, now_seconds: TimestampSeconds) -> bool {
