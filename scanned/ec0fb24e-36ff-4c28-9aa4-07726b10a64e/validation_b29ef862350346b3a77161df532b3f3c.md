### Title
SNS Swap Canister: Direct ICP Donation to Buyer Subaccount Bypasses `direct_participation_icp_e8s` Accounting, Enabling Swap Manipulation - (File: `rs/sns/swap/src/swap.rs`)

### Summary

The SNS Swap canister tracks direct participation via an internal state variable `direct_participation_icp_e8s` that is only updated through `refresh_buyer_token_e8s`. An unprivileged attacker can transfer ICP directly to any buyer's subaccount on the ICP ledger without calling `refresh_buyer_token_e8s`, causing the on-chain ledger balance to diverge from the swap's internal accounting. When the victim (or attacker) later calls `refresh_buyer_token_e8s`, the swap reads the inflated ledger balance and credits the buyer with more participation than they actually contributed, consuming `available_direct_participation_e8s` and potentially triggering early swap commitment or blocking other legitimate participants.

### Finding Description

The SNS Swap canister uses a two-step participation flow:
1. A buyer transfers ICP to their personal subaccount on the ICP ledger (subaccount derived from their principal).
2. The buyer calls `refresh_buyer_tokens`, which reads the ledger balance and updates the internal `buyers` map and `direct_participation_icp_e8s`.

The critical accounting state variable `direct_participation_icp_e8s` is only updated inside `update_total_participation_amounts`, which is called exclusively from `refresh_buyer_token_e8s`. [1](#0-0) 

The `available_direct_participation_e8s` is computed as `max_direct_participation_e8s - current_direct_participation_e8s`. [1](#0-0) 

Inside `refresh_buyer_token_e8s`, the swap reads the **full ledger balance** of the buyer's subaccount: [2](#0-1) 

It then computes the increment as `e8s - old_amount_icp_e8s` and caps it at `available_direct_participation_e8s`: [3](#0-2) 

**The attack:** An attacker can send ICP directly to **victim_principal**'s subaccount (i.e., `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&victim)) }`) on the ICP ledger without calling `refresh_buyer_tokens`. When the victim later calls `refresh_buyer_tokens`, the swap sees a higher balance than the victim actually deposited, and credits the victim with the donated amount — consuming `available_direct_participation_e8s` from the swap's cap.

Alternatively, the attacker can donate ICP to their **own** subaccount and then call `refresh_buyer_tokens` themselves, consuming the swap's remaining capacity without having legitimately contributed that ICP (the donated ICP is credited to their `BuyerState`).

The `BuyerState` invariant documented in the proto explicitly acknowledges this asymmetry: [4](#0-3) 

The invariant states `icp.amount_e8 <= icp_ledger.balance_of(subaccount(swap_canister, P))` — meaning the ledger balance can exceed the recorded amount, and a donation makes the ledger balance exceed what the buyer actually contributed.

### Impact Explanation

1. **Swap cap exhaustion / early commitment trigger**: An attacker donates ICP to fill the remaining `available_direct_participation_e8s`. When any buyer calls `refresh_buyer_tokens`, the donated ICP is credited, consuming the cap. If `max_direct_participation_icp_e8s` is reached, `can_commit` returns true and the swap can be committed early — before the intended number of legitimate participants have joined.

2. **Blocking legitimate participants**: Once `available_direct_participation_e8s` reaches zero, new participants are rejected. Existing participants can still increase their stake, but new principals cannot join.

3. **Distorted SNS token distribution**: The `sns_tokens_per_icp` ratio and neuron basket sizes are computed from `buyer_total_icp_e8s` (which includes donated ICP credited to buyers). Buyers who received donated ICP get more SNS tokens than they paid for, at the expense of other participants' proportional share.

4. **Attacker-controlled early swap finalization**: By precisely filling the remaining cap, an attacker can force the swap to commit at a moment of their choosing, before the minimum participation period ends or before enough legitimate participants have joined. [5](#0-4) 

### Likelihood Explanation

- **Reachable entry path**: Any unprivileged principal can call `icrc1_transfer` on the ICP ledger to send ICP to `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&any_principal)) }`. This is a standard ledger transfer requiring no special permissions.
- **No privileged role required**: The attacker only needs ICP tokens and knowledge of the swap canister ID and target principal.
- **Economically motivated**: An attacker who holds SNS tokens or has a stake in the swap outcome (e.g., a competing project, or someone who wants to force early commitment before the Neurons' Fund participation is maximized) has clear incentive.
- **Likelihood**: Medium. The attack requires capital (ICP to donate), but the donated ICP is credited to the victim's `BuyerState` and is locked in the swap — the attacker loses the donated ICP. However, for a well-capitalized attacker seeking to manipulate swap outcomes, this is a realistic attack.

### Recommendation

1. **Track net contributions, not ledger balance**: Instead of reading the raw ledger balance in `refresh_buyer_token_e8s`, the swap should only accept increments that the buyer explicitly initiates (e.g., via a ticket/nonce mechanism that ties a specific transfer to a specific `refresh_buyer_tokens` call). The existing ticket system (`OPEN_TICKETS_MEMORY`) partially addresses this but is optional and can be bypassed.

2. **Enforce ticket usage**: Make the ticket flow mandatory so that `refresh_buyer_tokens` only accepts increments up to the ticket amount, not the full ledger balance. This prevents donated ICP from being credited.

3. **Alternatively, ignore excess balance**: If `e8s > old_amount_icp_e8s + ticket_amount`, reject or cap at `old_amount_icp_e8s + ticket_amount` rather than crediting the full ledger balance. [6](#0-5) 

### Proof of Concept

```
Setup:
  - SNS swap is OPEN with max_direct_participation_icp_e8s = 1000 ICP
  - current_direct_participation_e8s = 900 ICP (9 legitimate buyers × 100 ICP each)
  - available_direct_participation_e8s = 100 ICP
  - min_participant_icp_e8s = 10 ICP

Attack:
  1. Attacker calls ICP ledger icrc1_transfer:
       from: attacker_account
       to: Account { owner: swap_canister_id, subaccount: principal_to_subaccount(victim_principal) }
       amount: 100 ICP

  2. Victim (or attacker using victim's principal) calls swap.refresh_buyer_tokens(victim_principal).

  3. swap.refresh_buyer_token_e8s reads ledger balance = 100 ICP for victim's subaccount.
     old_amount_icp_e8s = 0 (victim never participated before)
     requested_increment_e8s = 100 ICP
     max_increment_e8s = available_direct_participation_e8s = 100 ICP
     actual_increment_e8s = min(100, 100) = 100 ICP
     new_balance_e8s = 100 ICP → credited to victim's BuyerState

  4. update_total_participation_amounts() sets:
     direct_participation_icp_e8s = 1000 ICP = max_direct_participation_icp_e8s
     available_direct_participation_e8s = 0

  5. swap.can_commit() now returns true (max reached).
     Swap is committed early, before additional legitimate participants could join.

  6. Victim receives SNS neurons backed by 100 ICP they did not pay.
     Attacker loses 100 ICP but has manipulated the swap outcome.

    pub fn available_direct_participation_e8s(&self) -> u64 {
        let max_direct_participation_e8s = self.max_direct_participation_e8s();
        let current_direct_participation_e8s = self.current_direct_participation_e8s();

                    < current_direct_participation_e8s ({current_direct_participation_e8s})"
                );
                0
            })
    }

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };

    /// `sns_tokens_per_icp` will be 0 if `participant_total_icp_e8s` is 0.
    pub fn derived_state(&self) -> DerivedState {
        let participant_total_icp_e8s = self.current_total_participation_e8s();
        let direct_participant_count = Some(self.buyers.len() as u64);
        let cf_participant_count = Some(self.cf_participants.len() as u64);
        let cf_neuron_count = Some(self.cf_neuron_count());
        let tokens_available_for_swap = match self.sns_token_e8s() {
            Ok(tokens) => tokens,
            Err(err) => {
                log!(ERROR, "{}", err);
                0
            }

        let sns_tokens_per_icp = i2d(tokens_available_for_swap)
            .checked_div(i2d(participant_total_icp_e8s))

            .unwrap_or(0.0);

            buyer_total_icp_e8s: participant_total_icp_e8s,
            direct_participant_count,
            cf_participant_count,
            cf_neuron_count,
            sns_tokens_per_icp,
            direct_participation_icp_e8s,
            neurons_fund_participation_icp_e8s,
        }
