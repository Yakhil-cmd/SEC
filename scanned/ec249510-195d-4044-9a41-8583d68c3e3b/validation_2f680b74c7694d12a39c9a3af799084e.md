### Title
Wrong Subaccount Stored in `Erc20WithdrawalRequest::from_subaccount` Breaks Withdrawal Status Lookup and Emits Incorrect Audit Events - (`File: rs/ethereum/cketh/minter/src/main.rs`)

---

### Summary

In `withdraw_erc20`, the `Erc20WithdrawalRequest` struct is constructed with `from_ckerc20_subaccount` stored in the field `from_subaccount`, which is explicitly documented as *"The subaccount from which the minter burned ckETH."* The correct value should be `from_cketh_subaccount`. This mirrors the BBLeverage::sellCollateral pattern exactly: the wrong identifier is silently stored in a persistent record, causing downstream operations that depend on that field to behave incorrectly.

---

### Finding Description

The `withdraw_erc20` update endpoint accepts two distinct optional subaccounts from the caller:

- `from_cketh_subaccount` — the subaccount from which ckETH is burned to pay the Ethereum gas fee
- `from_ckerc20_subaccount` — the subaccount from which ckERC20 tokens are burned for the withdrawal

Both are used correctly for the actual burns: [1](#0-0) 

The ckETH burn uses `cketh_account` (built from `from_cketh_subaccount`) and the ckERC20 burn uses `ckerc20_account` (built from `from_ckerc20_subaccount`). However, when the resulting `Erc20WithdrawalRequest` is constructed and persisted to state, the wrong subaccount is stored: [2](#0-1) 

The struct field is unambiguously documented as the ckETH subaccount: [3](#0-2) 

This persisted `from_subaccount` value is then used in two downstream operations:

**1. `BySenderAccount` withdrawal status search** — `match_parameter` compares the caller-supplied account's subaccount against `request.from_subaccount`: [4](#0-3) 

When a user queries their withdrawal status by their ckETH account (owner + `from_cketh_subaccount`), the comparison fails because `from_subaccount` holds `from_ckerc20_subaccount`. The withdrawal request is invisible to the user.

**2. ckERC20 reimbursement destination on Ethereum transaction failure** — `record_finalized_transaction` uses `request.from_subaccount` as `to_subaccount` for the ckERC20 mint-back: [5](#0-4) 

Since `from_subaccount` accidentally holds `from_ckerc20_subaccount`, the ckERC20 reimbursement lands in the correct subaccount by coincidence. However, the field semantics are violated, and any future refactor that reads `from_subaccount` expecting the ckETH subaccount will silently misbehave.

**3. Incorrect on-chain audit event** — The `AcceptedErc20WithdrawalRequest` event emitted to the minter's audit log records `from_subaccount` as the ckERC20 subaccount, not the ckETH subaccount, misleading off-chain indexers and dashboards. [6](#0-5) 

---

### Impact Explanation

Any user who calls `withdraw_erc20` with **different** `from_cketh_subaccount` and `from_ckerc20_subaccount` values will be unable to retrieve their withdrawal status via the `BySenderAccount` search parameter using their ckETH account. The withdrawal request exists in state but is unreachable through the standard query path. Additionally, the minter's certified audit log permanently records the wrong subaccount for every such withdrawal, corrupting the on-chain history used by wallets and explorers.

---

### Likelihood Explanation

The `withdraw_erc20` endpoint is a publicly reachable update call requiring no privileged role. Any anonymous-to-authenticated IC principal can trigger it. The bug manifests whenever `from_cketh_subaccount != from_ckerc20_subaccount`, which is a documented and intentional feature of the API (both fields exist precisely to allow independent subaccounts). Users managing treasury or multi-subaccount setups will routinely hit this. [7](#0-6) 

---

### Recommendation

Change line 489 of `rs/ethereum/cketh/minter/src/main.rs` to use `from_cketh_subaccount` instead of `from_ckerc20_subaccount`:

```diff
-        from_subaccount: from_ckerc20_subaccount
+        from_subaccount: from_cketh_subaccount
             .and_then(LedgerSubaccount::from_bytes),
```

If the intent of `from_subaccount` is actually to track the ckERC20 burn subaccount (for reimbursement), the field must be renamed and re-documented accordingly, and the `BySenderAccount` search logic must be updated to match against the ckETH subaccount separately.

---

### Proof of Concept

1. User holds ckETH in subaccount `A` and ckUSDC in subaccount `B` (`A ≠ B`).
2. User approves the minter on both ledgers and calls:
   ```
   withdraw_erc20({
     amount = X,
     ckerc20_ledger_id = ckUSDC_ledger,
     recipient = "0x...",
     from_cketh_subaccount = Some(A),
     from_ckerc20_subaccount = Some(B)
   })
   ```
3. Both burns succeed. The returned `RetrieveErc20Request` contains the burn indices.
4. User queries withdrawal status via `BySenderAccount { owner: caller, subaccount: Some(A) }`.
5. `match_parameter` compares `Some(A)` against `from_subaccount = Some(B)` → **no match → empty result**.
6. The user cannot track their withdrawal. The audit log records `from_subaccount = B` (ckERC20 subaccount) for an event that is supposed to record the ckETH subaccount. [8](#0-7) [9](#0-8)
