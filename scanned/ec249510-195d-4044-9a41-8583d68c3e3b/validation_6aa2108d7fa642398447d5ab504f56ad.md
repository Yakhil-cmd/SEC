### Title
Unbounded Per-Canister Query Queue in `QuerySchedulerCore` Enables Replica Memory Exhaustion - (File: rs/execution_environment/src/query_handler/query_scheduler/internal.rs)

### Summary
The `QuerySchedulerCore` maintains a per-canister `incoming: VecDeque<Query>` with no capacity bound. Any unprivileged user can flood the replica's HTTP query endpoint, causing the scheduler to accumulate an unbounded number of query closures in heap memory, ultimately crashing the replica process with OOM and degrading subnet liveness.

### Finding Description

`CanisterData` holds two unbounded `VecDeque<Query>` fields — `incoming` and `leftover` — with no maximum size enforced anywhere in the push path: [1](#0-0) 

The `push()` method unconditionally appends to `incoming` without any capacity check: [2](#0-1) 

The public HTTP query handler calls `query_scheduler.push()` for every incoming query request, and its `poll_ready` always returns `Poll::Ready(Ok(()))`, meaning there is no backpressure from the scheduler back to the HTTP layer: [3](#0-2) [4](#0-3) 

By contrast, the update-call (ingress) path has explicit load-shedding: it checks `ingress_throttler.exceeds_threshold()` and rejects with `503` when the pool is full: [5](#0-4) 

No equivalent guard exists for the query path. The boundary-node rate limiter is optional (CLI-flag-gated) and applies only to the boundary layer, not to the replica itself: [6](#0-5) 

The `QUERY_EXECUTION_THREADS_TOTAL` is only 4, meaning the 4 worker threads drain the queue far slower than an attacker can fill it: [7](#0-6) 

### Impact Explanation

An attacker sending a sustained flood of HTTP query requests targeting a single canister causes the replica process to accumulate an unbounded number of heap-allocated query closures (each capturing an `Arc<ReplicatedState>` and other data). This leads to replica OOM, crashing the node. If enough nodes in a subnet are targeted simultaneously (below the fault threshold), subnet liveness is degraded. Even a single-node attack degrades that node's availability for all users.

### Likelihood Explanation

The HTTP query endpoint is publicly reachable by any unprivileged user with no authentication required. No cycles are charged for queries. The attack requires only a high-rate HTTP client (e.g., `wrk`, `ab`) sending repeated query requests to a single canister. The replica has no self-protection at this layer; the only mitigations (boundary-node rate limiting) are optional and external.

### Recommendation

1. Enforce a maximum capacity on `CanisterData::incoming` (e.g., proportional to `QUERY_EXECUTION_THREADS_TOTAL × queries_per_time_slice`). Return an error or drop the query when the per-canister queue is full.
2. Implement `poll_ready` backpressure in `HttpQueryHandler` so the HTTP layer stops accepting new queries when the scheduler queue is saturated, analogous to the ingress pool throttler check.
3. Apply per-IP query rate limiting at the replica HTTP endpoint (not only at the boundary node), consistent with how update calls are protected.

### Proof of Concept

```
# Flood a single canister's query queue on a replica node
wrk -t 16 -c 512 -d 60s \
  -s query_payload.lua \
  https://<replica-ip>:8080/api/v2/canister/<canister-id>/query
```

Where `query_payload.lua` sends a valid signed query. Each request causes `QuerySchedulerCore::push()` to append one `Query` closure to `CanisterData::incoming`. With 512 concurrent connections and no bound on the queue, the replica's heap grows without limit. After sustained load, the replica process is killed by the OS OOM killer, removing the node from the subnet. [8](#0-7)

/// Contains the query queues and execution stats of a canister.
pub(crate) struct CanisterData {
    // All incoming queries are pushed into this FIFO queue.
    incoming: VecDeque<Query>,

    // Queries are executed in batches. This FIFO queue contains queries from
    // the previous batch that did not get enough execution time. These queries
    // will be executed in subsequent batches and take priority over `incoming`
    // queries.
    leftover: VecDeque<Query>,

    // The average query execution duration observed so far.
    // For simplicity, it is initialized with `DEFAULT_QUERY_DURATION`.
    average_query_duration: Duration,

    // The current number of threads executing queries of this canister.
    active_threads: usize,

    // Indicates whether this canister has been added to the `scheduled`
    // canister queue of the scheduler. This flag is needed to ensure that a
    // canister is not added multiple times to the queue.
    has_been_scheduled: bool,
}

    /// Adds the given query to the `incoming` queue of the given canister.
    /// It also adds the canister to the round-robin queue if needed.
    fn push(&mut self, canister_id: CanisterId, query: Query) {
        let canister = self
            .canisters
            .entry(canister_id)
            .or_insert_with(CanisterData::new);
        canister.incoming.push_back(query);

        self.metrics
            .queue_length
            .observe((canister.incoming.len() + canister.leftover.len()) as f64);

        if !canister.has_been_scheduled
            && canister.should_be_scheduled(self.max_threads_per_canister)
        {
            canister.has_been_scheduled = true;
            self.scheduled.push_back(canister_id);
        }

        #[cfg(debug_assertions)]
        self.verify_invariants();
    }
