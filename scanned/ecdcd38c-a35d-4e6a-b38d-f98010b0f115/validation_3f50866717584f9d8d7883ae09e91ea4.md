### Title
SNS Governance `ManageLedgerParameters` Allows Setting Unbounded `transfer_fee`, Blocking All Token Transfers - (File: `rs/sns/governance/src/proposal.rs`)

---

### Summary

The SNS governance `ManageLedgerParameters` proposal action allows setting the SNS ICRC-1 ledger's `transfer_fee` to any arbitrary `u64` value with no upper-bound validation. A passed proposal setting `transfer_fee = u64::MAX` (or any value exceeding all token holders' balances) permanently blocks every SNS token holder from transferring their tokens, since the ledger enforces the fee exactly and returns `InsufficientFunds` for any transfer that cannot cover it.

---

### Finding Description

`ManageLedgerParameters` is a non-critical SNS governance proposal action (action ID 13) that upgrades the SNS ICRC-1 ledger with new parameters. Its validation function `validate_and_render_manage_ledger_parameters` in `rs/sns/governance/src/proposal.rs` performs **no range check** on the `transfer_fee` field:

```rust
if let Some(transfer_fee) = transfer_fee {
    render += &format!("# Set token transfer fee: {transfer_fee} token-quantums. \n",);
    change = true;
    // ← no upper-bound check whatsoever
}
``` [1](#0-0) 

The `ManageLedgerParameters` struct carries `transfer_fee: Option<u64>`, accepting the full `u64` range: [2](#0-1) 

On proposal execution, `perform_manage_ledger_parameters` converts the value directly into `UpgradeArgs` and upgrades the ledger canister with no additional sanitization: [3](#0-2) 

The ICRC-1 ledger's `upgrade` function then stores whatever fee value is provided: [4](#0-3) 

During a transfer, the ledger enforces the stored fee exactly. If a caller omits the `fee` field, the ledger uses the stored fee as `expected_fee_tokens`. If the fee exceeds the sender's balance, the transfer fails with `InsufficientFunds`: [5](#0-4) 

---

### Impact Explanation

If a `ManageLedgerParameters` proposal sets `transfer_fee` to `u64::MAX` (18,446,744,073,709,551,615 e8s), every SNS token holder's balance is smaller than the required fee. All `icrc1_transfer` and `icrc2_transfer_from` calls will return `InsufficientFunds`. Token holders lose the ability to move their SNS tokens indefinitely until a corrective proposal is passed and executed. This is a **ledger conservation bug**: tokens are permanently locked in accounts, destroying the utility of the SNS token.

---

### Likelihood Explanation

`ManageLedgerParameters` is **not** a critical proposal in the SNS governance framework (unlike `TransferSnsTreasuryFunds` which is explicitly critical). Non-critical proposals require a lower voting threshold. Any SNS neuron holder who accumulates sufficient voting power — including the founding team during early SNS operation, a whale, or a coordinated group — can pass this proposal without the supermajority required for critical actions. The attack is a single on-chain proposal submission and vote, with no off-chain prerequisites.

---

### Recommendation

1. Add an upper-bound check in `validate_and_render_manage_ledger_parameters` to reject `transfer_fee` values that exceed a reasonable maximum (e.g., the total token supply or a protocol-defined cap such as `1_000_000_000_000` e8s).
2. Consider classifying `ManageLedgerParameters` as a critical proposal, requiring the same supermajority threshold as `TransferSnsTreasuryFunds`, since it can have an equally severe impact on token holders.
3. Document the intended range for `transfer_fee` in the proposal rendering so voters can identify malicious proposals.

---

### Proof of Concept

1. An SNS neuron holder with sufficient voting power submits:
   ```
   ManageLedgerParameters { transfer_fee: Some(u64::MAX), .. }
   ```
2. The proposal passes `validate_and_render_manage_ledger_parameters` — the only check is that at least one field is `Some`. [6](#0-5) 
3. After the vote passes, `perform_manage_ledger_parameters` upgrades the SNS ledger canister with `UpgradeArgs { transfer_fee: Some(Nat::from(u64::MAX)), .. }`. [7](#0-6) 
4. The ledger stores `transfer_fee = u64::MAX`. [8](#0-7) 
5. Any subsequent `icrc1_transfer` call from any token holder returns `Err(InsufficientFunds { balance: <their_balance> })` because no account holds `u64::MAX` tokens. [9](#0-8) 
6. All SNS token holders are permanently unable to transfer tokens until a corrective governance proposal is passed.
