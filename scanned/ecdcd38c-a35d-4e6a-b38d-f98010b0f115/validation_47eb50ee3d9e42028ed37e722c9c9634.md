**The bug is real and the logic is clear:**

In `process_pong_message`, when `conn.ping_state()` is `PingState::Idle`, `valid_pong` is set to `false`, and `conn.disconnect()` is unconditionally called: [1](#0-0) 

The `PingState::Idle` is the **default and normal state** of every connection between ping intervals (120 seconds): [2](#0-1) 

Any Bitcoin peer connected to the adapter can send an unsolicited `Pong` message at any time during the 120-second idle window. The Bitcoin protocol (BIP-31 / Bitcoin Core behavior) specifies that unsolicited Pong messages must be silently ignored, not cause disconnection.

### Title
Unsolicited Pong Message Causes Forced Disconnection from Bitcoin Peer — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

### Summary
`process_pong_message` disconnects a connection whenever a `Pong` is received while `PingState::Idle`, which is the normal state between ping intervals. Any Bitcoin peer can exploit this to force the adapter to drop its connection.

### Finding Description
In `process_pong_message`, the `valid_pong` flag is `false` for `PingState::Idle { .. }`, and the `else` branch unconditionally calls `conn.disconnect()`. [3](#0-2) 

`PingState::Idle` is the initial and inter-ping state of every connection. Pings are sent every 120 seconds (`PING_INTERVAL`), so the window during which a peer can trigger this is nearly continuous. [4](#0-3) 

The Bitcoin P2P protocol (BIP-31, Bitcoin Core reference implementation) explicitly states that unsolicited `pong` messages must be silently ignored. The adapter instead treats them as misbehavior and disconnects.

### Impact Explanation
A malicious Bitcoin node (or any node on the public Bitcoin P2P network that the adapter connects to) can:
1. Complete the version handshake normally (connection reaches `HandshakeComplete`).
2. Immediately send a `Pong` message with any nonce while the adapter is in `PingState::Idle`.
3. The adapter calls `conn.disconnect()`, transitioning the connection to `NodeDisconnected`.
4. The connection is reaped in the next `reap_disconnected()` call. [5](#0-4) 

With `min_connections = 2` and `max_connections = 5`, an attacker operating or influencing multiple Bitcoin nodes can keep the adapter below `min_connections`, causing `send_to_all` to silently drop all outbound messages (block/transaction relay blocked): [6](#0-5) 

### Likelihood Explanation
- The attacker entry point is the public Bitcoin P2P network — no privileged access required.
- The attack window is the full 120-second idle interval between pings, making it trivially easy to time.
- The adapter reconnects after disconnection, but an attacker controlling or Sybil-attacking the address book can repeatedly force reconnections to malicious peers.
- No cryptographic or consensus-level protection prevents a peer from sending arbitrary P2P messages.

### Recommendation
Replace the `conn.disconnect()` in the `PingState::Idle` branch with a silent ignore (no-op), consistent with the Bitcoin protocol specification:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong — ignore per Bitcoin protocol spec (BIP-31)
}
```

Only disconnect on `PingState::ExpectingPong` with a nonce mismatch.

### Proof of Concept
Unit test (no fuzzing required — deterministic):
1. Create a `Connection` in `HandshakeComplete` state with `PingState::Idle` (the default).
2. Call `process_pong_message` with any nonce (e.g., `42`).
3. Assert `conn.state()` is `ConnectionState::NodeDisconnected`.

This is directly provable from the code path at lines 516–531 of `connectionmanager.rs` without any probabilistic assumptions. The nonce=0 scenario mentioned in the question is a red herring — the `Idle` branch disconnects regardless of nonce value. [7](#0-6)
