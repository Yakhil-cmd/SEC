### Title
Stale Query Stats from Removed Nodes Skew Median Aggregation and Canister Cycles Accounting - (`rs/query_stats/src/state_machine.rs`)

---

### Summary

When nodes are removed from a subnet, their previously submitted `QueryStats` records persist in `RawQueryStats` and continue to influence the median calculation during epoch aggregation. The `purge_records` function only removes records for already-aggregated epochs, not records belonging to nodes that are no longer subnet members. This causes the aggregation threshold check to be satisfied using stale data from removed nodes, and the median to be computed over a participant set that no longer reflects the active subnet, leading to incorrect cycles accounting for canister query calls.

---

### Finding Description

The `try_aggregate_one_epoch` function in `rs/query_stats/src/state_machine.rs` aggregates per-node query statistics by computing the median across all nodes that have submitted records for the current epoch. The function fetches the current subnet size via `own_subnet_size()` and uses it both to compute the aggregation threshold and as a multiplier when applying the aggregated stats to canister state. [1](#0-0) 

The threshold for triggering aggregation is: [2](#0-1) 

`num_nodes_with_stats` is counted by iterating over **all** entries in `state.stats`, which is a `BTreeMap<NodeId, ...>` that is never filtered against the current subnet membership: [3](#0-2) 

The only cleanup performed is `purge_records`, which removes records for epochs ≤ `highest_aggregated_epoch` and then removes node entries with no remaining records: [4](#0-3) 

Critically, there is **no step that removes records belonging to nodes that have been removed from the subnet**. When a node is removed via `do_change_subnet_membership` or `do_remove_nodes_from_subnet`, its `RawQueryStats` entries for the current (not-yet-aggregated) epoch remain in the state. [5](#0-4) 

The aggregated median is then multiplied by the **current** `num_nodes` (the post-removal subnet size), not the size of the participant set whose data was actually used: [6](#0-5) 

The code acknowledges the `num_nodes` multiplier inaccuracy but does not address the median skew from stale node data: [7](#0-6) 

---

### Impact Explanation

**Incorrect cycles accounting for canister query calls.** The `TotalQueryStats` accumulated per canister (accessible via the management canister's `canister_status`) is used to charge canisters for query execution resources. When the median is computed over a mix of current and removed nodes' data, and then multiplied by a `num_nodes` that does not match the actual participant set, the resulting `num_calls`, `num_instructions`, `ingress_payload_size`, and `egress_payload_size` credited to each canister are incorrect.

A malicious node that knows it is about to be removed (e.g., a governance proposal has passed) can submit artificially inflated or deflated `QueryStats` for the current epoch immediately before removal. Its data persists in `RawQueryStats` and shifts the median for that epoch. Because the median is robust only against a minority of outliers, a single removed node can shift the median by at most one position — but with multiple colluding removed nodes (still below the fault threshold), the skew can be larger.

Additionally, `num_nodes_with_stats` counting removed nodes makes the aggregation threshold easier to satisfy with fewer active nodes than the protocol intends, potentially allowing aggregation to proceed with an insufficient quorum of current-member data.

---

### Likelihood Explanation

Subnet membership changes (node additions and removals) are infrequent but are a normal part of IC operations, executed via NNS governance proposals. Any subnet membership change that occurs mid-epoch triggers this condition. A node operator controlling a node that is being removed is a protocol peer below the consensus fault threshold and can exploit the window between proposal execution and epoch boundary to submit manipulated stats. No external attacker capability beyond controlling a single subnet node is required.

---

### Recommendation

1. **Filter `state.stats` against current subnet membership** before computing `num_nodes_with_stats` and `aggregatable_records` in `try_aggregate_one_epoch`. Records from `NodeId`s not present in the current `network_topology` should be excluded from both the threshold check and the median computation.

2. **Purge records from removed nodes** in `purge_records` (or a new companion function called at the same point) by cross-referencing `state.stats` keys against `replicated_state.system_metadata().network_topology`.

3. Alternatively, **snapshot the subnet membership at epoch start** and use that snapshot for both the threshold check and the `num_nodes` multiplier, ensuring consistency between the participant set and the scaling factor.

---

### Proof of Concept

**Setup**: Subnet with 13 nodes (`num_nodes = 13`, `get_faults_tolerated(13) = 4`, `need_stats_from = 9`).

**Step 1**: All 13 nodes submit `QueryStats` for epoch E. `RawQueryStats.stats` has 13 entries.

**Step 2**: A governance proposal removes 5 nodes. `own_subnet_size()` now returns 8. `need_stats_from = 8 - 2 = 6`.

**Step 3**: `try_aggregate_one_epoch` is called. It iterates `state.stats` which still has 13 entries (5 removed nodes' records not purged). `num_nodes_with_stats` counts up to 13 (if removed nodes have records signaling epoch completion). Threshold `6` is easily met.

**Step 4**: `aggregatable_records` includes the 5 removed nodes' data. The median of 13 values is computed, potentially dominated by the removed nodes' (possibly manipulated) values.

**Step 5**: `apply_query_stats_to_canister` multiplies the median by `num_nodes = 8`, producing a result that is neither the correct 13-node aggregate nor the correct 8-node aggregate. [8](#0-7) [9](#0-8) [10](#0-9)

/// Removes all records for epochs equal or lower than `highest_aggregated_epoch`.
/// Then it removes all node_id entries which are empty.
fn purge_records(state: &mut ReplicatedState) {
    let state = &mut state.epoch_query_stats;

    let Some(highest_aggregated_epoch) = state.highest_aggregated_epoch else {
        return;
    };

    // Delete records for epoch that are already aggregated
    state.stats.iter_mut().for_each(|(_node_id, records)| {
        records.retain(|&epoch, _| epoch > highest_aggregated_epoch)
    });

    // Delete node_ids that don't have any entries
    state.stats.retain(|_node_id, records| !records.is_empty());
}

impl Registry {
    /// Changes membership of nodes in a subnet record in the registry.
    ///
    /// This method is called by the governance canister, after a proposal
    /// for modifying a subnet by changing the membership (adding/removing) has been accepted.
    pub fn do_change_subnet_membership(&mut self, payload: ChangeSubnetMembershipPayload) {
        println!("{LOG_PREFIX}do_change_subnet_membership started: {payload:?}");

        let nodes_to_add = payload.node_ids_add.clone();
        let subnet_id = SubnetId::from(payload.subnet_id);
        let mut subnet_record = self.get_subnet_or_panic(subnet_id);

        let current_subnet_nodes: Vec<NodeId> = subnet_record
            .membership
            .iter()
            .map(|bytes| NodeId::from(PrincipalId::try_from(bytes).unwrap()))
            .collect();

        // Verify that nodes requested to be removed belong to the subnet provided in the payload
        if !payload
            .node_ids_remove
            .iter()
            .all(|n| current_subnet_nodes.contains(n))
        {
            panic!("Nodes that should be removed do not belong to the provided subnet.")
        }

        // Calculate a complete list of nodes in this subnet after the change of subnet membership is executed
        let subnet_membership_after_change = nodes_to_add
            .iter()
            .cloned()
            .chain(current_subnet_nodes)
            .filter(|node_id_in_subnet| {
                payload
                    .node_ids_remove
                    .iter()
                    .all(|node_id_to_remove| node_id_in_subnet != node_id_to_remove)
            })
            .collect();

        self.replace_subnet_record_membership(
            subnet_id,
            &mut subnet_record,
            subnet_membership_after_change,
        );
        let mutations = vec![upsert(
            make_subnet_record_key(subnet_id),
            subnet_record.encode_to_vec(),
        )];

        // Check the invariants and apply the mutations if invariants are satisfied
        self.maybe_apply_mutation_internal(mutations);

        println!("{LOG_PREFIX}do_change_subnet_membership finished: {payload:?}");
    }
