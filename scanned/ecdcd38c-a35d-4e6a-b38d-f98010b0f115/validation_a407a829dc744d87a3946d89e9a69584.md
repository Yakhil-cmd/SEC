I need to trace the exact code path to determine if forced participation is achievable and what the real impact is.

### Title
Unauthorized Third-Party Registration of Swap Participation via Unchecked `buyer` Field in `refresh_buyer_tokens` - (`rs/sns/swap/canister/canister.rs`, `rs/sns/swap/src/swap.rs`)

---

### Summary

The `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal in the request body with no check that the caller matches the buyer. Any unprivileged principal can call this endpoint on behalf of any other principal, forcing that principal's `BuyerState` to be created and their open ticket deleted. The **cap bypass claim is false** — `max_participant_icp_e8s` is always enforced. However, the **forced participation** is a real, concrete vulnerability: once a victim's `BuyerState` is registered, `error_refund_icp` blocks them from reclaiming their ICP until after `sweep_icp` runs, meaning a committed swap permanently converts their ICP to SNS neurons without their consent.

---

### Finding Description

**Entry point — no caller/buyer identity check:** [1](#0-0) 

The canister handler reads `arg.buyer` from the request body and, if non-empty, parses it as the target principal — completely ignoring `caller_principal_id()`. There is no assertion that `caller == buyer`.

**Core logic — BuyerState written for the supplied principal:** [2](#0-1) 

The ICP ledger balance is queried for the **supplied** `buyer`'s subaccount, not the caller's. [3](#0-2) 

The `BuyerState` is written for the supplied `buyer` principal.

**Cap bypass claim is FALSE — cap is always enforced:** [4](#0-3) 

`new_balance_e8s` is always clamped to `max_participant_icp_e8s`. The attacker cannot cause the victim to exceed the cap.

**Ticket deletion — side effect of forced call:** [5](#0-4) 

If the victim has an open ticket and the subaccount balance ≥ ticket amount, the ticket is deleted. The victim's intended participation amount is overridden by the full subaccount balance (capped at `max_participant_icp_e8s`).

**The real impact — `error_refund_icp` is blocked for registered buyers:** [6](#0-5) 

`error_refund_icp` returns a precondition error ("ICP in escrow") for any principal present in `self.buyers` whose ICP has not yet been disbursed via `sweep_icp`. This creates a decisive fork:

| Scenario | Victim NOT in `buyers` | Victim IS in `buyers` (attacker called) |
|---|---|---|
| Swap commits | `error_refund_icp` succeeds → victim reclaims ICP | `error_refund_icp` blocked → ICP swept to SNS governance → victim receives SNS neurons |
| Swap aborts | `error_refund_icp` succeeds → victim reclaims ICP | ICP swept back via `sweep_icp` → same outcome |

The only scenario with material harm is a committed swap: the victim is forced to receive SNS neurons instead of ICP, with no recourse.

---

### Impact Explanation

A victim who transferred ICP to the swap subaccount but had not yet called `refresh_buyer_tokens` (e.g., changed their mind, or was waiting) can be forcibly registered as a participant by any third party who knows their principal. If the swap subsequently commits, the victim's ICP is swept to SNS governance and they receive SNS neurons they did not consent to. The `error_refund_icp` escape hatch is unavailable to them because they appear in `self.buyers` with undisbursed ICP.

The **cap bypass** framing in the question is incorrect — `max_participant_icp_e8s` is always enforced and cannot be circumvented via this path.

---

### Likelihood Explanation

- Swap is OPEN: public state, trivially observable.
- Victim's principal: public on the IC (visible from ledger transactions to the swap subaccount).
- Attacker needs no special role, key, or privilege.
- Confirmation text (if required) is public SNS init data, not a secret.
- The attack requires the victim to have transferred ICP but not yet called `refresh_buyer_tokens` — a narrow but realistic window, especially for users who transfer ICP and then reconsider.

---

### Recommendation

Add a caller-equals-buyer guard in the canister handler:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        // Enforce: only the buyer themselves may register their participation.
        if buyer != caller {
            panic!("Caller {} is not authorized to refresh tokens for buyer {}", caller, buyer);
        }
        buyer
    };
    // ...
}
