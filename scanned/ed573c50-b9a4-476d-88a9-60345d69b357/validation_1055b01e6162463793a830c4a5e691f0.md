### Title
Governance `do_remove_nodes` Unconditionally Increments Malicious Node Operator's `node_allowance`, Enabling Immediate Re-registration - (File: rs/registry/canister/src/mutations/node_management/do_remove_nodes.rs)

### Summary
The NNS governance-gated `do_remove_nodes` function (`NNS_FUNCTION_REMOVE_NODES`) unconditionally increments the removed node's operator `node_allowance` by 1 for every node removed. When governance removes a malicious node, this side effect grants the malicious operator an additional registration slot, allowing them to immediately re-register a replacement node via the permissionless `add_node` endpoint. This mirrors the external report's root cause: the removal mechanism has an unintended property that partially negates the governance action.

### Finding Description

In `do_remove_nodes`, step 6 unconditionally increments `node_allowance` for the node operator of every removed node:

```rust
// 6. Retrieve the NO record, cache it and increment its node allowance by 1
new_node_operator_record.node_allowance += 1;
``` [1](#0-0) 

This mutation is then written back to the registry:

```rust
for (node_operator_id, updated_node_operator) in principal_to_node_operator_record {
    mutations.push(make_update_node_operator_mutation(node_operator_id, &updated_node_operator));
}
``` [2](#0-1) 

The same increment exists in `do_remove_node_directly` (voluntary self-removal by the operator), where it is semantically correct — the operator is freeing a slot they own. However, `do_remove_nodes` is the governance-initiated forcible removal path, and applying the same increment here is semantically incorrect: it rewards the malicious operator with a new registration slot. [3](#0-2) 

The `add_node` canister endpoint is callable by any principal that holds a valid node operator record — no governance approval is required: [4](#0-3) 

The `remove_nodes` endpoint, by contrast, is strictly governance-gated: [5](#0-4) 

This asymmetry means that after a governance proposal executes `do_remove_nodes`, the malicious operator can immediately call `add_node` without any further governance approval, using the allowance slot that was just granted to them by the removal itself.

Additionally, `do_remove_nodes` panics (causing the entire governance proposal to fail) if any node in the payload is currently assigned to a subnet, meaning governance cannot atomically remove a subnet-assigned malicious node in a single proposal — it must first pass a `RemoveNodesFromSubnet` proposal, then a `RemoveNodes` proposal, widening the window for re-registration: [6](#0-5) 

### Impact Explanation

A malicious node operator whose node is forcibly removed by NNS governance receives a `node_allowance` increment as a side effect. In the legacy mode (node rewards disabled), `node_allowance` directly gates `add_node` calls. The malicious operator can immediately call `add_node` with fresh cryptographic keys to register a replacement node, restoring their presence in the unassigned node pool. If governance is slow to notice or does not simultaneously reduce the allowance via a separate `UpdateNodeOperatorConfig` proposal, the malicious operator can cycle through this loop indefinitely. Even when node rewards are enabled and `max_rewardable_nodes` is the primary gate, the `node_allowance` field is still written back, creating a persistent inconsistency and a latent re-registration vector if the rewards system is ever disabled or bypassed.

The secondary impact is that governance has no atomic single-proposal path to remove a malicious node AND prevent re-registration. It requires at minimum two sequential proposals (`RemoveNodes` + `UpdateNodeOperatorConfig`), and three if the node is subnet-assigned (`RemoveNodesFromSubnet` + `RemoveNodes` + `UpdateNodeOperatorConfig`), with a live re-registration window between each.

### Likelihood Explanation

The malicious operator must already hold a valid node operator record (which requires prior NNS governance approval), so the precondition is non-trivial. However, once a node operator is acting maliciously, they clearly have the technical capability to generate new node keys and call `add_node`. The rate limit on `add_node` (per-operator and per-IP) provides a short-term brake but resets over time. The likelihood is **medium**: the precondition is governance-gated, but the exploit is trivially executable by any operator who has already been onboarded and is now acting adversarially.

### Recommendation

1. **Do not increment `node_allowance` in `do_remove_nodes`.** Governance-initiated forcible removal should not grant the operator a new slot. The allowance increment is semantically correct only for voluntary self-removal (`do_remove_node_directly`).
2. **Provide an atomic governance proposal type** that removes a node AND decrements (or zeroes) the operator's `node_allowance` in a single registry mutation batch, eliminating the re-registration window.
3. **Consider adding a `RemoveNodeOperator` step** as part of the malicious-node removal flow, or at minimum enforce that `do_remove_nodes` does not write back an incremented allowance when called from the governance path.

### Proof of Concept

1. Malicious node operator `M` has `node_allowance = 0` (all slots consumed).
2. NNS governance passes `NNS_FUNCTION_REMOVE_NODES` targeting `M`'s node.
3. `do_remove_nodes` executes: node record and crypto keys are deleted; `M.node_allowance` is written back as `1`. [7](#0-6) 
4. `M` immediately calls `canister_update add_node` with fresh cryptographic keys (new `NodeId`). No governance approval is required. [4](#0-3) 
5. `do_add_node` succeeds: `M.node_allowance` is decremented back to `0`, and a new node record appears in the registry.
6. Governance must detect the new node and pass another `RemoveNodes` proposal, repeating from step 2. Each cycle costs governance one proposal round; the malicious operator pays only a `add_node` call.
