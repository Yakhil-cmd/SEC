### Title
NNS Governance `disburse_neuron` Unconditionally Clears `neuron_fees_e8s` Even When Fee Burn Is Skipped, Stranding Tokens in Neuron Subaccount - (File: rs/nns/governance/src/governance.rs)

---

### Summary

In the NNS Governance canister's `disburse_neuron` function, `neuron.neuron_fees_e8s` is unconditionally set to `0` even when the fee-burn ledger transfer is skipped because `fees_amount_e8s <= transaction_fee_e8s`. This causes a divergence between the ledger state (fee tokens remain in the neuron's subaccount) and governance state (fees recorded as zero/burned). The fee tokens are permanently stranded — not burned, not disbursed to the user — inflating the effective ICP supply by the unburned amount.

---

### Finding Description

In `disburse_neuron`, the code first conditionally burns the neuron's accumulated management fees:

```rust
if fees_amount_e8s > transaction_fee_e8s {
    let _result = self.ledger.transfer_funds(
        fees_amount_e8s,
        0, // Burning transfers don't pay a fee.
        Some(neuron_subaccount),
        governance_minting_account(),
        now,
    ).await?;
}
```

Immediately after — regardless of whether the burn happened — the governance state is updated:

```rust
self.with_neuron_mut(id, |neuron| {
    if neuron.cached_neuron_stake_e8s > fees_amount_e8s {
        neuron.cached_neuron_stake_e8s -= fees_amount_e8s;
    } else {
        neuron.cached_neuron_stake_e8s = 0;
    }
    neuron.neuron_fees_e8s = 0;  // ← always zeroed
})
``` [1](#0-0) 

When `fees_amount_e8s <= transaction_fee_e8s` (i.e., the burn is skipped), the `fees_amount_e8s` tokens remain in the neuron's subaccount on the ICP ledger. The disburse amount was already computed as `neuron_minted_stake_e8s = cached_neuron_stake_e8s - fees_amount_e8s`, so the fee tokens are neither burned nor sent to the user. After the disburse completes, governance records `cached_neuron_stake_e8s = 0` and `neuron_fees_e8s = 0`, but the ledger still holds `fees_amount_e8s` tokens in the neuron's subaccount — permanently stranded.

The SNS Governance canister explicitly avoids this mistake with the comment:

> "We only update the `cached_neuron_stake_e8s` and `neuron_fees_e8s` if we actually burn fees, otherwise this leads to ledger and governance getting out of sync." [2](#0-1) 

The SNS implementation only updates the neuron's fee fields when the burn actually occurs. The NNS implementation lacks this guard.

---

### Impact Explanation

- **Ledger conservation bug**: Tokens that should be burned (as a penalty for rejected proposals) are not burned. The effective ICP supply is inflated by the stranded amount per affected neuron.
- **Governance/ledger state divergence**: After `disburse_neuron` completes, the governance canister records `neuron_fees_e8s = 0` and `cached_neuron_stake_e8s = 0`, but the ICP ledger retains `fees_amount_e8s` tokens in the neuron's subaccount. These tokens are inaccessible through any normal governance operation since the neuron is fully disbursed.
- **Per-occurrence impact**: At most `transaction_fee_e8s - 1` = 9,999 e8s (≈ 0.0001 ICP) per neuron disbursement. Individually small ("dust"), but the pattern is structurally identical to the reported M-06 vulnerability.

---

### Likelihood Explanation

The condition `0 < fees_amount_e8s <= transaction_fee_e8s` (10,000 e8s) is required. In the current NNS configuration, `reject_cost_e8s` is 1 ICP = 100,000,000 e8s, making this condition unlikely for a single rejected proposal. However:

1. `reject_cost_e8s` is a governance-controlled parameter and could be reduced by a governance vote.
2. Any neuron that has accumulated fees through a governance parameter change that lowered `reject_cost_e8s` below 10,000 e8s would be affected.
3. The entry path is fully unprivileged: any neuron controller calls `manage_neuron` → `Disburse` on a dissolved neuron with small accumulated fees. [3](#0-2) 

---

### Recommendation

Mirror the SNS Governance fix: only update `neuron_fees_e8s` and `cached_neuron_stake_e8s` when the burn actually occurred. When `fees_amount_e8s <= transaction_fee_e8s`, do not clear `neuron_fees_e8s` — leave it for a future burn opportunity, or explicitly handle the residual by disbursing the remaining fee tokens to the user (analogous to the M-06 recommended mitigation).

```rust
if fees_amount_e8s > transaction_fee_e8s {
    // ... ledger burn ...
    self.with_neuron_mut(id, |neuron| {
        if neuron.cached_neuron_stake_e8s > fees_amount_e8s {
            neuron.cached_neuron_stake_e8s -= fees_amount_e8s;
        } else {
            neuron.cached_neuron_stake_e8s = 0;
        }
        neuron.neuron_fees_e8s = 0;
    })?;
}
// Only update state when burn actually happened
``` [4](#0-3)

### Proof of Concept

**Trigger condition**: A dissolved NNS neuron with `0 < neuron_fees_e8s <= 9999` e8s (e.g., accumulated from a governance parameter era where `reject_cost_e8s` was small).

**Steps**:
1. Neuron controller calls `manage_neuron` → `Disburse` on a dissolved neuron with `neuron_fees_e8s = F` where `F <= 9999`.
2. `disburse_neuron` skips the burn transfer (line 2046 condition is false).
3. Governance unconditionally sets `neuron_fees_e8s = 0` and reduces `cached_neuron_stake_e8s` by `F` (lines 2067–2075).
4. Disburse transfer sends `cached_neuron_stake_e8s - F - tx_fee` to the user.
5. **Result**: ICP ledger retains `F` tokens in the neuron's subaccount; governance records zero. `F` tokens are permanently stranded — not burned, not accessible. [5](#0-4)
