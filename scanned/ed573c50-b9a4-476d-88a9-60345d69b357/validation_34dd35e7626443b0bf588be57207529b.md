### Title
`fee_based_minimum_withdrawal_amount` uses fixed 2-input size assumption, enabling `AmountTooLow` finalization without reimbursement on fragmented UTXO sets — (`rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

---

### Summary

`fee_based_minimum_withdrawal_amount` computes the minimum withdrawal amount using a hardcoded 374-byte transaction size (2-input/2-output P2PKH). The actual transaction size — and critically the **minter fee**, which scales at 1.46 DOGE per input — is not bounded by this assumption. When the minter's UTXO set is fragmented, the actual transaction requires more inputs, the actual `fee + minter_fee` can exceed the withdrawal amount, and `BuildTxError::AmountTooLow` is returned. This path finalizes the request with `FinalizedStatus::AmountTooLow` **without any reimbursement**, burning the user's ckDOGE.

---

### Finding Description

**Step 1 — Minimum amount computation (fixed 2-input assumption):** [1](#0-0) 

`PER_REQUEST_SIZE_BOUND = 374` bytes and `PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu are hardcoded for a 2-input/2-output transaction. The actual minter fee for 2 inputs is `2×146_000_000 + 2×4_000_000 + 26_000_000 = 326_000_000` koinu — the bound is 1000× smaller and effectively ignored. The result is `retrieve_doge_min_amount + small_rounding` (e.g., 50 DOGE).

**Step 2 — Actual fee uses real transaction size:** [2](#0-1) 

`evaluate_transaction_fee` calls `DogecoinTransactionSigner::fake_sign(tx).len()` — the real serialized size with all actual inputs.

**Step 3 — Minter fee scales linearly with inputs:** [3](#0-2) 

`MINTER_FEE_PER_INPUT = 146_000_000` koinu = 1.46 DOGE per input. With 35 inputs: `35×146_000_000 + 34_000_000 = 5_144_000_000` koinu = **51.44 DOGE**, which exceeds the 50 DOGE minimum withdrawal amount.

**Step 4 — AmountTooLow check in transaction builder:** [4](#0-3) 

If `fee + minter_fee > amount`, `BuildTxError::AmountTooLow` is returned.

**Step 5 — AmountTooLow finalizes WITHOUT reimbursement:** [5](#0-4) 

The comment explicitly states "There is no point in retrying." The request is removed via `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. No reimbursement task is enqueued. Compare with `BuildTxError::InvalidTransaction`, which **does** call `reimburse_canceled_requests`. [6](#0-5) 

**Step 6 — TooManyInputs check happens before AmountTooLow, but only at 500+ inputs:** [7](#0-6) [8](#0-7) 

`DOGECOIN_MAX_NUM_INPUTS_IN_TRANSACTION = 500`. The dangerous zone is **35–499 inputs**: enough to push `minter_fee > amount`, but not enough to trigger the reimbursed `TooManyInputs` path.

---

### Impact Explanation

A user submitting a withdrawal at exactly `fee_based_minimum_withdrawal_amount` (e.g., 50 DOGE) when the minter's UTXO set contains UTXOs smaller than ~1.43 DOGE each will have their ckDOGE burned with no reimbursement. The ledger burn is irreversible. The `FinalizedStatus::AmountTooLow` state is terminal. [9](#0-8) 

---

### Likelihood Explanation

- The minter accumulates UTXOs from all user deposits. Many small deposits (e.g., 1 DOGE each) naturally fragment the UTXO set.
- An adversary can deliberately deposit many small UTXOs to fragment the set, then trigger the bug for themselves or other users.
- The `fee_based_minimum_withdrawal_amount` is advertised to users as the safe minimum; users have no way to know the actual UTXO composition.
- The threshold is reachable with only 35 inputs of <1.43 DOGE each — a realistic operational state.

---

### Recommendation

1. **In `fee_based_minimum_withdrawal_amount`**: use a conservative upper bound on the number of inputs (e.g., based on the actual UTXO set composition or a configurable worst-case), not a fixed 2-input assumption.
2. **In the `AmountTooLow` handler**: add reimbursement (minus a small processing fee), consistent with the `InvalidTransaction` handler. The user submitted a valid amount per the published minimum; the failure is due to minter state, not user error.
3. **Alternatively**: before finalizing with `AmountTooLow`, attempt to re-select UTXOs or defer the request rather than burning it.

---

### Proof of Concept

```
1. Minter state: 60 UTXOs of 1 DOGE each (60 DOGE total available).
2. fee_based_minimum_withdrawal_amount(median_fee) returns 50 DOGE
   (using PER_REQUEST_SIZE_BOUND=374 bytes, 2-input assumption).
3. User calls retrieve_doge_with_approval(50_DOGE). Passes ingress check.
4. submit_pending_requests calls build_unsigned_transaction:
   - utxos_selection selects 50 UTXOs of 1 DOGE to cover 50 DOGE.
   - minter_fee = 50×146_000_000 + 2×4_000_000 + 26_000_000
               = 7_334_000_000 koinu = 73.34 DOGE
   - fee + minter_fee = ~73.34 DOGE > 50 DOGE → AmountTooLow
5. remove_retrieve_btc_request(request, FinalizedStatus::AmountTooLow)
   → no reimbursement, ckDOGE burned.
```

Proptest: generate a `UtxoSet` with many UTXOs of value < 1.43 DOGE, set withdrawal amount = `fee_based_minimum_withdrawal_amount(fee_rate)`, assert `build_unsigned_transaction` does not return `AmountTooLow`. This assertion will fail.

    fn fee_based_minimum_withdrawal_amount(&self, median_fee_rate: FeeRate) -> u64 {
        match self.network {
            Network::Mainnet => {
                //in Koinu
                const PER_REQUEST_RBF_BOUND: u64 = 374_000;
                // in Bytes
                // Size of a typical transaction made by the minter,
                // which is a P2PKH transaction with 2 inputs and 2 outputs
                const PER_REQUEST_SIZE_BOUND: u64 = 374;
                //in Koinu
                const PER_REQUEST_MINTER_FEE_BOUND: u64 = 326_000;

                let min_withdrawal_amount_increment = self.retrieve_doge_min_amount >> 1;
                ((PER_REQUEST_RBF_BOUND
                    + median_fee_rate.fee_ceil(PER_REQUEST_SIZE_BOUND)
                    + PER_REQUEST_MINTER_FEE_BOUND)
                    / min_withdrawal_amount_increment)
                    * min_withdrawal_amount_increment
                    + self.retrieve_doge_min_amount
            }
            Network::Regtest => self.retrieve_doge_min_amount,
        }
    }
