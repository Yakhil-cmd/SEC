### Title
Unbounded `ingress_expiries` Loop in `services::construction_payloads` Allows Unauthenticated DoS via Memory Exhaustion — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

### Summary
The `construction_payloads` service function iterates from a caller-supplied `ingress_start` to a caller-supplied `ingress_end` with a fixed step of ~120 seconds, with no cap on the number of iterations. A single unauthenticated HTTP request with a large `ingress_end` value causes unbounded heap allocation, exhausting the Rosetta process's memory.

### Finding Description
In `services::construction_payloads`, `ingress_start` and `ingress_end` are read directly from the caller-supplied request metadata: [1](#0-0) 

The only validation applied is: [2](#0-1) 

This enforces `ingress_start < ingress_end` and `ingress_end >= now + ingress_interval`, but places **no upper bound** on `ingress_end`. The loop then runs without any iteration cap: [3](#0-2) 

The step per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP`:
- `ingress_interval` = (`MAX_INGRESS_TTL` − `PERMITTED_DRIFT`) = 300s − 60s = **240s**
- `INGRESS_INTERVAL_OVERLAP` = **120s**
- Step = **120s = 120,000,000,000 ns** [4](#0-3) [5](#0-4) 

The endpoint is fully unauthenticated — it accepts any `ConstructionPayloadsRequest` from any HTTP client: [6](#0-5) 

### Impact Explanation
For `ingress_end = now + u64::MAX/2` (~292 years in nanoseconds):
- Iterations ≈ (u64::MAX/2) / 120,000,000,000 ≈ **76.8 billion**
- Memory for `ingress_expiries` alone: ~614 GB (8 bytes × 76.8B entries)

Even a modest attack with `ingress_end = now + 10 years`:
- Iterations ≈ 2,628,000
- `handle_construction_payloads` then iterates `ingress_expiries` and pushes **2 `SigningPayload` structs per entry** (each containing hex-encoded message IDs), multiplying the allocation further

The Rosetta process OOMs and crashes, causing complete availability loss.

### Likelihood Explanation
The endpoint is a standard Rosetta Construction API endpoint, publicly accessible to any HTTP client with no authentication, no rate limiting in the service layer, and no request-level size guard. The attack requires a single HTTP POST with a crafted JSON body — trivially reproducible.

### Recommendation
Add an explicit cap on the number of ingress expiry entries before the loop, e.g.:

```rust
const MAX_INGRESS_EXPIRIES: usize = 1000; // ~33 hours at 120s step
if ingress_start < ingress_end {
    let span = ingress_end - ingress_start;
    let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
    let count = (span / step) as usize + 1;
    if count > MAX_INGRESS_EXPIRIES {
        return Err(Error::processing_construction_failed(
            "ingress_end exceeds maximum allowed window"
        ));
    }
}
```

Alternatively, enforce a maximum `ingress_end - ingress_start` window (e.g., 24 hours) before entering the loop.
