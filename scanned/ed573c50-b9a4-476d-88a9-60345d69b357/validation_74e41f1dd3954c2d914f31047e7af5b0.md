### Title
Missing Supply Limit Enforcement on `MintSnsTokens` Proposals Allows Unbounded SNS Token Inflation — (File: `rs/sns/governance/src/proposal.rs`)

---

### Summary

The `MintSnsTokens` SNS governance proposal action has its 7-day rolling supply cap intentionally disabled. The real enforcement function is commented out and replaced with a stub returning `Decimal::MAX`, meaning any SNS governance majority can pass an unlimited number of minting proposals in rapid succession with no protocol-level rate limit, inflating the SNS token supply without bound.

---

### Finding Description

The SNS governance system defines two token-moving proposal actions: `TransferSnsTreasuryFunds` and `MintSnsTokens`. Both share the same validation framework (`treasury_valuation_if_proposal_amount_is_small_enough_or_err`) which is supposed to enforce a 7-day rolling cap calibrated to treasury valuation.

For `TransferSnsTreasuryFunds`, the cap is fully enforced. The `recent_amount_total_upper_bound_tokens` implementation calls `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens`, which computes a real limit (up to 25% of medium treasuries, or 300,000 XDR max for large ones): [1](#0-0) 

For `MintSnsTokens`, the analogous implementation is **commented out** and replaced with a stub that returns `Decimal::MAX`: [2](#0-1) 

The real limit function `mint_sns_tokens_7_day_total_upper_bound_tokens` exists and is fully implemented in the limit library: [3](#0-2) 

It is even imported in the file but commented out: [4](#0-3) 

Additionally, `TransferSnsTreasuryFunds` has a second enforcement check at **execution time** (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`): [5](#0-4) 

No analogous execution-time check exists for `MintSnsTokens`.

The integration test explicitly confirms the bypass is live: the assertion that a second large mint should be rejected is commented out, and `doomed_make_proposal_result.unwrap()` is used instead, confirming the second mint succeeds: [6](#0-5) 

The whale's balance after both mints is `2 * 2_222 = 4_444` tokens, confirming both passed: [7](#0-6) 

---

### Impact Explanation

An SNS governance majority (a whale neuron holder or coordinated group) can submit back-to-back `MintSnsTokens` proposals with no protocol-enforced ceiling on the 7-day minting total. Each proposal that passes mints new SNS tokens from thin air, diluting all existing token holders. Because the SNS token is the governance and economic unit of the SNS DAO, unbounded inflation:

- Dilutes the voting power and economic stake of all existing token holders.
- Depresses the token price, harming holders who cannot exit quickly.
- Undermines the economic model of any SNS-governed dapp that relies on token scarcity.

This is a **ledger conservation / governance authorization bug**: the protocol-level safeguard that was designed to bound minting rate relative to market liquidity is absent, directly analogous to the "lack of supply limit" class described in the external report.

---

### Likelihood Explanation

Any SNS neuron holder who accumulates or coordinates sufficient voting power to pass proposals can exploit this. The SNS governance model is permissionless — anyone can acquire tokens and neurons. The missing limit is not a configuration gap but a code-level bypass (`Decimal::MAX` stub) that is active on every deployed SNS canister using this governance code. The TODO markers (`NNS1-2982`) confirm this is a known incomplete implementation shipped to production.

---

### Recommendation

1. **Uncomment** the real `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (remove the `/* TODO(NNS1-2982): Uncomment. */` block and delete the `Decimal::MAX` stub).
2. **Add an execution-time check** for `MintSnsTokens` analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, so that even if a proposal was submitted before the limit was reached, it cannot execute if the window is now full.
3. **Uncomment** the integration test assertion in `rs/sns/integration_tests/src/sns_treasury.rs` that verifies the second mint is rejected.
4. **Delete** the `#[allow(unused)]` annotation on `total_minting_amount_tokens` once the limit is active.

---

### Proof of Concept

1. Deploy an SNS with a treasury valued above 100,000 XDR (medium or large regime).
2. Acquire or coordinate enough neuron voting power to pass proposals.
3. Submit `MintSnsTokens` proposal #1 for an amount equal to 25% of the treasury token supply. It passes and executes.
4. Immediately submit `MintSnsTokens` proposal #2 for the same amount. Under the intended design, this should be rejected at submission time because `spent_tokens + proposal_amount > max_tokens`. Instead, because `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`, the check at line 805 (`proposal_amount_tokens > allowance_remainder_tokens`) is always false, and the proposal is accepted. [8](#0-7) 

5. Repeat indefinitely within the same 7-day window. Each iteration mints new tokens with no protocol resistance, confirmed by the integration test at line 966 and the resulting `4_444` token balance at line 983. [9](#0-8)
