### Title
Missing Concurrency Limit on Synchronous Call Endpoints (`/api/v3` and `/api/v4`) Allows Resource Exhaustion - (File: `rs/http_endpoints/public/src/lib.rs`)

### Summary
The IC replica's public HTTP server applies `GlobalConcurrencyLimitLayer` to every endpoint except the synchronous call endpoints `/api/v3/canister/{id}/call`, `/api/v4/canister/{id}/call`, and `/api/v4/subnet/{id}/call`. These endpoints are long-running (they hold `IngressWatcher` subscriptions and block on certified state reads for up to 10 seconds each), and an unprivileged sender can flood them without bound, exhausting shared replica resources.

### Finding Description

In `rs/http_endpoints/public/src/lib.rs`, the `make_router` function applies a `GlobalConcurrencyLimitLayer` (backed by a load-shedder) to every route:

- `/api/v2/call` → `max_call_concurrent_requests` (default 50)
- `/api/v2/query`, `/api/v3/query`, `/api/v3/subnet/query` → `max_query_concurrent_requests`
- All four `read_state` variants → `max_read_state_concurrent_requests` (default 100)
- `/api/v2/status`, `/_/catch_up_package`, `/_/dashboard`, `/_/pprof/*`, `/_/tracing/*` → their respective limits

However, the three synchronous call routers are merged **without any concurrency limit layer**, with an explicit developer TODO acknowledging the gap:

```rust
// TODO(CON-1574): see if there is any reasonable explicit concurrency limit we could use here.
.merge(http_handler.call_v3_router)
.merge(http_handler.call_v4_router)
.merge(http_handler.subnet_call_v4_router)
``` [1](#0-0) 

Each request handled by `call_sync` in `rs/http_endpoints/public/src/call/call_sync.rs`:

1. Acquires the shared `Arc<Mutex<IngressFilterService>>` lock (line 343 of `call.rs`).
2. Calls `ingress_watcher_handle.subscribe_for_certification(...)` — consuming a slot in the `IngressWatcher`.
3. Calls `state_reader.get_certified_state_snapshot()` via `tokio::task::spawn_blocking`, consuming a blocking thread.
4. Blocks for up to `ingress_message_certificate_timeout_seconds` (default **10 seconds**) waiting for certification. [2](#0-1) 

The `ingress_filter` is a single shared `Arc<Mutex<IngressFilterService>>` created once in `start_server`: [3](#0-2) 

And locked inside `validate_ingress_message`: [4](#0-3) 

The `Config` struct has no field for a v3/v4 call concurrency limit, confirming no limit is configured: [5](#0-4) 

### Impact Explanation

An unprivileged attacker sending a flood of POST requests to `/api/v3/canister/{any_canister_id}/call` or `/api/v4/canister/{any_canister_id}/call` can:

1. **Exhaust the Tokio blocking thread pool**: Each request spawns a `spawn_blocking` call to `get_certified_state_snapshot`. With no concurrency cap, this can saturate all blocking threads, stalling other replica subsystems that also use `spawn_blocking` (e.g., crypto operations, state reads for `read_state` endpoints).
2. **Exhaust `IngressWatcher` subscription capacity**: Each in-flight request holds a certification subscription for up to 10 seconds. Flooding creates unbounded subscriptions, degrading the `IngressWatcher`'s ability to serve legitimate callers.
3. **Contend on the shared `ingress_filter` mutex**: Concurrent requests serialise through the single `Mutex<IngressFilterService>`, causing legitimate ingress validation to queue behind attacker requests.
4. **Degrade replica availability**: The above resource pressure can cause legitimate `query`, `read_state`, and `call` (v2) requests to be load-shed (429) even though their own concurrency limits have not been reached, because shared underlying resources are exhausted.

This is a **boundary/API validation bypass** (missing concurrency enforcement) with a **cycles/resource accounting** impact on the replica node.

### Likelihood Explanation

The `/api/v3/call` and `/api/v4/call` endpoints are publicly reachable via API boundary nodes. Any anonymous or authenticated user can send valid or minimally-valid CBOR-encoded call envelopes. The attacker does not need any privileged key or role. The 10-second per-request hold time means even a modest rate (e.g., 100 req/s sustained) creates thousands of concurrent in-flight requests. The TODO comment in the source confirms the developers are aware of the gap but have not resolved it.

### Recommendation

Apply a `GlobalConcurrencyLimitLayer` (with load-shedding) to `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` in `make_router`, analogous to the existing `max_call_concurrent_requests` limit on `call_v2_router`. Add a corresponding `max_sync_call_concurrent_requests` field to `Config` in `rs/config/src/http_handler.rs`.

### Proof of Concept

1. Start a replica node (or use a test harness via `HttpEndpointBuilder`).
2. Send a large number of concurrent POST requests to `/api/v3/canister/aaaaa-aa/call` with a valid CBOR envelope (any valid `HttpRequestEnvelope<HttpCallContent>`).
3. Observe that all requests are accepted and held for up to 10 seconds each (the `ingress_message_certificate_timeout_seconds` default).
4. Concurrently send a legitimate `/api/v2/canister/{id}/call` request and observe it is blocked or degraded due to `ingress_filter` mutex contention and blocking thread exhaustion, even though the v2 concurrency limit (50) has not been reached.
5. Contrast with `/api/v2/call`: sending 51 concurrent requests to that endpoint causes the 51st to receive `429 Too Many Requests` immediately, as demonstrated in `rs/http_endpoints/public/tests/load_shed_test.rs` — no equivalent protection exists for v3/v4. [6](#0-5)

/// Test concurrency limiter for `/v2/.../call` endpoint and that when the load shedder kicks in
/// we return 429.
/// Test scenario:
/// 1. Set the concurrency limiter for the call service, `max_call_concurrent_requests`, to 1.
/// 2. Send an ingress message where we wait with responding for the update call
///    inside the ingress filter service handle.
/// 3. Concurrently make another update call, and assert it hits the load shedder.
#[test]
fn test_load_shedding_update_call() {
    let rt = Runtime::new().unwrap();
    let addr = get_free_localhost_socket_addr();

    let config = Config {
        listen_addr: addr,
        max_call_concurrent_requests: 1,
        ..Default::default()
    };

    let mut handlers = HttpEndpointBuilder::new(rt.handle().clone(), config).run();

    let ingress_filter_running = Arc::new(Notify::new());
    let load_shedder_returned = Arc::new(Notify::new());

    let ingress_filter_running_clone = ingress_filter_running.clone();
    let load_shedder_returned_clone = load_shedder_returned.clone();

    let call_agent = Call::V2;

    let load_shedded_request_handle = rt.spawn(async move {
        ingress_filter_running_clone.notified().await;
        let response = call_agent.call(addr, IngressMessage::default()).await;
        load_shedder_returned_clone.notify_one();
        response
    });

    // Mock ingress filter
    rt.spawn(async move {
        let (_, resp) = handlers.ingress_filter.next_request().await.unwrap();
        ingress_filter_running.notify_one();
        load_shedder_returned.notified().await;
        resp.send_response(Ok(Ok(())))
    });

    rt.block_on(async {
        wait_for_status_healthy(&addr).await.unwrap();
        let response = call_agent.call(addr, IngressMessage::default()).await;
        assert_eq!(
            response.status(),
            StatusCode::ACCEPTED,
            "{:?}",
            response.text().await.unwrap()
        );

        let load_shedded_response = load_shedded_request_handle.await.unwrap();

        assert_eq!(
            StatusCode::TOO_MANY_REQUESTS,
            load_shedded_response.status()
        );
    })
}
