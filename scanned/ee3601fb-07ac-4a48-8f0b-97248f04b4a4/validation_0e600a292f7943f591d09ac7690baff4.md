The code confirms the vulnerability is real. Here is the analysis:

**Key facts from the code:**

1. `MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST = 10000` is defined as a constant. [1](#0-0) 

2. The constant is used **only as a default** via `unwrap_or`, not as a cap. If `request.limit` is `Some(i64::MAX)`, the `unwrap_or` branch is never taken. [2](#0-1) 

3. `i64::MAX` (9223372036854775807) is a valid `u64` value, so `try_into()` succeeds without error — no rejection occurs.

4. After line 391, there is **no upper-bound enforcement** on `limit`. The only post-computation check is `if limit == 0` (early return with empty result), which does not help here. [3](#0-2) 

5. The attacker-controlled `limit` flows directly into the SQL `LIMIT` clause, and is cast back to `i64` as `limit as i64` — which is `i64::MAX` — a valid SQLite LIMIT value that effectively means "return all rows." [4](#0-3)

### Title
Missing Upper-Bound Enforcement on `limit` in `search_transactions` Allows Memory Exhaustion via Unbounded SQLite Query — (`rs/rosetta-api/icrc1/src/data_api/services.rs`)

### Summary
An unprivileged HTTP client can POST to `/search/transactions` with `limit=i64::MAX`. The `MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST=10000` guard is applied only as a default when `limit` is absent (`None`), not as a cap on explicit values. The `i64::MAX → u64` conversion succeeds silently, and the value is forwarded verbatim to the SQLite `LIMIT` clause, causing the database to return all matching rows and loading them all into the Rosetta process heap.

### Finding Description
In `rs/rosetta-api/icrc1/src/data_api/services.rs` lines 385–391, the limit is resolved as:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST as i64)
    .try_into()
    .map_err(|err| { ... })?;
```

`unwrap_or` only substitutes the default when `request.limit` is `None`. When a caller supplies `Some(9223372036854775807)` (`i64::MAX`), the default is bypassed entirely. The subsequent `try_into::<u64>()` succeeds because `i64::MAX` is representable as `u64`. No subsequent check enforces `limit <= MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST`. The value is then used at line 490 as `rusqlite::types::Value::Integer(limit as i64)`, which is `i64::MAX` — a valid SQLite integer that SQLite treats as an effectively unbounded `LIMIT`.

### Impact Explanation
All rows matching the query are fetched from SQLite and deserialized into `Vec<BlockTransaction>` in the Rosetta process heap. On a node with 100k+ blocks, a single request can exhaust available memory, crashing or severely degrading the Rosetta API process. This is a remotely triggerable, unauthenticated DoS.

### Likelihood Explanation
The Rosetta HTTP API is publicly reachable. No authentication is required. The exploit requires a single crafted HTTP POST. Any attacker aware of the Rosetta API spec can trigger this.

### Recommendation
Replace the `unwrap_or` default pattern with an explicit cap after the limit is resolved:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST as i64)
    .try_into()
    .map_err(|err| {
        Error::request_processing_error(&format!("Limit has to be a valid u64: {err}"))
    })?;

// Enforce the cap regardless of whether the caller supplied a value
let limit = limit.min(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST);
```

Or reject requests that exceed the maximum:

```rust
if limit > MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST {
    return Err(Error::request_processing_error(&format!(
        "Limit exceeds maximum allowed value of {MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST}"
    )));
}
```

### Proof of Concept
Send the following to a Rosetta instance with ≥1 block:

```json
POST /search/transactions
{
  "network_identifier": { ... },
  "limit": 9223372036854775807
}
```

The response will attempt to return all blocks in the database. With 100k+ blocks, the process will allocate memory proportional to the full block set, causing OOM or severe latency. A bounded response or an error is the expected correct behavior.
