### Title
No Minimum Deposit Amount Enforced for ckBTC Deposits Allows Dust UTXO Spam That Permanently Bloats Minter State - (`rs/bitcoin/ckbtc/minter/src/state.rs`)

---

### Summary

The ckBTC minter's `deposit_btc_min_amount` field defaults to `0` when not explicitly set during initialization. When `deposit_btc_min_amount == 0`, the only guard against dust UTXOs is `check_fee > 0`. However, `check_fee` can also be set to `0` (e.g., on Regtest, or if misconfigured). Even when `check_fee > 0`, the effective minimum is only `check_fee + 1` satoshi — a trivially small amount. An attacker can send thousands of 1-satoshi UTXOs to minter-controlled deposit addresses, causing the minter to process, check, and permanently store each one in `available_utxos`, `outpoint_account`, and `utxos_state_addresses`, bloating the minter's heap state without bound.

---

### Finding Description

In `rs/bitcoin/ckbtc/minter/src/state.rs`, `deposit_btc_min_amount` is initialized via `deposit_btc_min_amount.unwrap_or_default()`, which resolves to `0` when the optional field is absent:

```rust
deposit_btc_min_amount: deposit_btc_min_amount.unwrap_or_default(),
```

The `validate_config` method only checks that `check_fee` does not exceed `retrieve_btc_min_amount`, but imposes **no lower bound on `deposit_btc_min_amount`**:

```rust
pub fn validate_config(&self) {
    if self.check_fee > self.retrieve_btc_min_amount {
        ic_cdk::trap("check_fee cannot be greater than retrieve_btc_min_amount");
    }
    ...
}
```

The `effective_deposit_min_btc_amount()` helper computes the runtime floor:

```rust
pub fn effective_deposit_min_btc_amount(&self) -> u64 {
    self.deposit_btc_min_amount
        .max(self.check_fee.saturating_add(1))
}
```

This means when `check_fee = 0` and `deposit_btc_min_amount = 0`, the effective minimum is `1` satoshi — a dust amount. When `check_fee > 0`, the floor is `check_fee + 1`, but `check_fee` itself has no enforced minimum.

In `update_balance.rs`, the UTXO filtering reads the raw `deposit_btc_min_amount` (not the effective value) directly from state:

```rust
let (deposit_btc_min_amount, check_fee) =
    read_state(|s| (s.deposit_btc_min_amount, s.check_fee));
...
if utxo.value < deposit_btc_min_amount { ... }
else if utxo.value <= check_fee { ... }
```

UTXOs that pass these checks are permanently added to `available_utxos`, `outpoint_account`, and `utxos_state_addresses` in `add_utxos()`. There is no cap on the number of UTXOs stored per account or globally.

---

### Impact Explanation

An attacker who controls Bitcoin addresses can send thousands of dust UTXOs (e.g., 1–2 satoshi each) to any ckBTC deposit address. Each UTXO that passes the `deposit_btc_min_amount` and `check_fee` filters is:

1. Checked via the Bitcoin checker canister (consuming cycles per check).
2. Permanently stored in the minter's heap state (`available_utxos`, `outpoint_account`, `utxos_state_addresses`).
3. Minted as ckBTC (1 satoshi = 1 ckBTC satoshi unit), creating dust ckBTC balances.

The minter's heap grows without bound. On the IC, canisters have a heap memory limit (currently ~4 GiB). Filling the minter's heap with dust UTXO entries can cause the minter to trap on upgrade (due to serialization cost) or exhaust available memory, halting all ckBTC operations — deposits, withdrawals, and consolidations — for all users. This is a **chain-fusion availability break** affecting the entire ckBTC protocol.

---

### Likelihood Explanation

The attack is cheap: Bitcoin dust outputs cost only the Bitcoin transaction fee (a few hundred satoshis per transaction, with many outputs per transaction). An attacker can pack hundreds of dust outputs into a single Bitcoin transaction. The ckBTC minter's deposit addresses are publicly derivable. No privileged access is required — any unprivileged Bitcoin user can execute this attack. The `deposit_btc_min_amount` defaulting to `0` is confirmed by the production upgrade history: the mainnet minter only set `deposit_btc_min_amount = 300` in January 2026 (proposal `minter_upgrade_2026_01_23.md`), meaning prior deployments ran with `deposit_btc_min_amount = 0`.

---

### Recommendation

1. **Enforce a non-zero lower bound on `deposit_btc_min_amount` in `validate_config`**: reject configurations where `deposit_btc_min_amount < check_fee + 1` or where `deposit_btc_min_amount` is below the Bitcoin dust relay threshold (546 satoshis for P2PKH, 294 for P2WPKH).
2. **Use `effective_deposit_min_btc_amount()` consistently** in `update_balance.rs` instead of reading `deposit_btc_min_amount` directly.
3. **Add a per-account or global cap** on the number of UTXOs stored in `available_utxos` to bound heap growth.

    pub(crate) fn add_utxos<I: CheckInvariants>(&mut self, account: Account, utxos: Vec<Utxo>) {
        if utxos.is_empty() {
            return;
        }

        self.tokens_minted += utxos.iter().map(|u| u.value).sum::<u64>();

        let account_bucket = self.utxos_state_addresses.entry(account).or_default();

        for utxo in utxos {
            self.minted_outpoints.insert(utxo.outpoint.clone());
            self.outpoint_account.insert(utxo.outpoint.clone(), account);
            self.available_utxos.insert(utxo.clone());
            self.checked_utxos.remove(&utxo);
            account_bucket.insert(utxo);
        }

        if cfg!(debug_assertions) {
            I::check_invariants(self).expect("state invariants are violated");
        }
    }
