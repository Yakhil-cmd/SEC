### Title
ERC-721 / ERC-20 `transferFrom` Selector Collision Allows NFTs to Be Permanently Locked in the ckETH Minter's Ethereum Address - (Files: `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

Both IC chain-fusion helper contracts (`CkErc20Deposit` and `CkDeposit`) accept any Ethereum address as the ERC-20 token contract without validating that it actually implements the ERC-20 interface. Because ERC-721's `transferFrom(address,address,uint256)` shares the identical 4-byte function selector (`0x23b872dd`) with ERC-20's `transferFrom(address,address,uint256)`, a user can deposit an ERC-721 NFT through the ERC-20 deposit path. The NFT is transferred to the minter's Ethereum address, but the minter canister has no mechanism to return it: it only constructs ERC-20 `transfer` calls (selector `0xa9059cbb`) for withdrawals, and it ignores the deposit event because the ERC-721 contract address is not in its supported-token whitelist. The NFT is permanently locked.

---

### Finding Description

**Root cause — no token-type validation in the helper contracts:**

`CkErc20Deposit.deposit()` in `ERC20DepositHelper.sol` accepts any `erc20_address` with zero validation:

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
}
``` [1](#0-0) 

`CkDeposit.depositErc20()` in `DepositHelperWithSubaccount.sol` only rejects the zero address:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
}
``` [2](#0-1) 

**Why the call succeeds for ERC-721:**

`SafeERC20.safeTransferFrom` dispatches `transferFrom(from, to, value)` via `_callOptionalReturn`. ERC-721's `transferFrom(address,address,uint256)` has the same 4-byte selector `0x23b872dd` as ERC-20's. ERC-721's `transferFrom` returns no value, so `returndata.length == 0`, which `_callOptionalReturn` treats as success. The NFT is moved to the minter's Ethereum address and the deposit event is emitted. [3](#0-2) 

**Why the minter ignores the event:**

The minter canister's log scraping filters events by topic, including only supported ERC-20 contract addresses. An ERC-721 address is never in `state.ckerc20_tokens`, so the event is silently dropped and no ckERC20 is minted. [4](#0-3) [5](#0-4) 

**Why the NFT cannot be recovered:**

The minter's only withdrawal transaction type for ERC-20 tokens encodes `TransactionCallData::Erc20Transfer` using the ERC-20 `transfer(address,uint256)` selector `0xa9059cbb`. There is no code path to construct an ERC-721 `safeTransferFrom` or `transferFrom` call. The minter's Ethereum address is controlled exclusively via threshold ECDSA, and no "rescue" endpoint exists. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any ERC-721 NFT deposited via `CkErc20Deposit.deposit()` or `CkDeposit.depositErc20()` is permanently locked in the minter's Ethereum address. The user receives no ckERC20 tokens in return and has no on-chain mechanism to recover the NFT. The minter's own withdrawal logic (`withdraw_erc20`) only supports tokens in its supported-token registry and only constructs ERC-20 `transfer` calls, making recovery impossible without a minter upgrade that adds an explicit rescue path.

This is a **chain-fusion asset lockup bug**: user funds (NFTs) are irreversibly lost through the IC chain-fusion deposit interface.

---

### Likelihood Explanation

The likelihood is **low-medium**. The scenario requires a user to supply an ERC-721 contract address where an ERC-20 address is expected. This can occur through:
- User error (confusing an NFT contract address for a fungible token address).
- A malicious third-party UI or integration that passes an ERC-721 address.
- A developer testing the deposit interface with an arbitrary contract.

The official documentation explicitly warns that the helper contract enforces no whitelist and that unsupported tokens will be lost, but this warning does not cover the ERC-721 selector-collision scenario specifically. [8](#0-7) 

---

### Recommendation

1. **Add ERC-165 interface detection** in both helper contracts before calling `safeTransferFrom`. Reject any address that reports `supportsInterface(0x80ac58cd)` (ERC-721) or does not report `supportsInterface(0x36372b07)` (ERC-20, if the token supports ERC-165).
2. **Alternatively, maintain an on-chain whitelist** of approved ERC-20 contract addresses in the helper contracts, rejecting any address not on the list. This mirrors the minter-side whitelist already enforced via log-topic filtering.
3. **Add a minter-side rescue endpoint** (governance-gated) that can construct arbitrary ERC-20 or ERC-721 transfer transactions from the minter's Ethereum address, to recover tokens locked by this or similar bugs.

---

### Proof of Concept

1. User holds ERC-721 NFT at contract `0xNFT`, token ID `42`.
2. User calls `0xNFT.approve(helper_contract, 42)` (ERC-721 approval).
3. User calls `CkErc20Deposit.deposit(0xNFT, 42, encoded_principal)`.
4. Helper calls `IERC20(0xNFT).safeTransferFrom(user, minter_eth_addr, 42)`.
5. ERC-721's `transferFrom(user, minter_eth_addr, 42)` executes (same selector `0x23b872dd`), NFT moves to minter's Ethereum address.
6. `ReceivedErc20(0xNFT, user, 42, principal)` event is emitted.
7. Minter scrapes logs with topic filter `[RECEIVED_ERC20_EVENT_TOPIC, [supported_erc20_addresses...]]`; `0xNFT` is not in the filter, event is ignored.
8. No ckERC20 minted. NFT is permanently locked. `withdraw_erc20` cannot be called for `0xNFT` (returns `TokenNotSupported`). No other recovery path exists. [9](#0-8) [10](#0-9)

// First 4 bytes of keccak256(transfer(address,uint256))
const ERC_20_TRANSFER_FUNCTION_SELECTOR: [u8; 4] = hex_literal::hex!("a9059cbb");

#[derive(Clone, Eq, PartialEq, Debug)]
pub enum TransactionCallData {
    Erc20Transfer { to: Address, value: Erc20Value },
}

impl TransactionCallData {
    /// Encode the transaction call data to interact with an Ethereum smart contract.
    /// See the [Contract ABI Specification](https://docs.soliditylang.org/en/develop/abi-spec.html#contract-abi-specification).
    pub fn encode(&self) -> Vec<u8> {
        match self {
            TransactionCallData::Erc20Transfer { to, value } => {
                let mut data = Vec::with_capacity(68);
                data.extend(ERC_20_TRANSFER_FUNCTION_SELECTOR);
                data.extend(<[u8; 32]>::from(to));
                data.extend(value.to_be_bytes());
                data
            }
        }
    }
