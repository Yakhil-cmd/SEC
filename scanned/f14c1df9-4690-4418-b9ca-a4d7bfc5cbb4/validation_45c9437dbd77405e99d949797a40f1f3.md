### Title
`sender_info.sig` Included in MessageId Enables Double-Execution via Distinct Valid Canister Signatures — (`rs/types/types/src/messages/http.rs`, `rs/types/types/src/messages/ingress_messages.rs`)

### Summary

The `sender_info.sig` field is included in the representation-independent hash that produces the `MessageId`. Because a canister can produce multiple distinct valid signatures over the same `SenderInfoContent` blob (one per block height at which the certified_data was set), an attacker who controls a canister can submit two ingress messages that are logically identical but carry different `sender_info.sig` values. Each message receives a distinct `MessageId`, both pass `validate_sender_info`, and both bypass every deduplication guard, resulting in double execution of the same logical call.

### Finding Description

**Root cause — `sig` in the hash:**

`representation_independent_hash_call_or_query` in `rs/types/types/src/messages/http.rs` inserts the full `sender_info` sub-map — including `sig` — into the hash:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, sig }) = sender_info {
    map.insert(
        "sender_info",
        Map(btreemap! {
            "info"   => Bytes(info),
            "signer" => Bytes(signer),
            "sig"    => Bytes(sig),   // ← signature bytes in the hash
        }),
    );
}
``` [1](#0-0) 

`SignedIngressContent::id()` calls this function and passes `sender_info.sig` directly: [2](#0-1) 

This is in deliberate contrast to the envelope-level `sender_sig`, which is explicitly **excluded** from the MessageId (confirmed by the `message_id_icf_reference` test): [3](#0-2) 

**Deduplication is keyed on MessageId only:**

`ValidSetRuleImpl::is_duplicate` checks the ingress history keyed on `msg.content().id()`: [4](#0-3) 

`validate_ingress` in the ingress selector checks `past_ingress_set.contains(&ingress_id)` where `ingress_id` is the `IngressMessageId` wrapping the same `MessageId`: [5](#0-4) 

`validate_ingress_pool_object` checks `ingress_message_status(&ingress_object.message_id)`: [6](#0-5) 

None of these guards compare the logical content of the message independently of the signature.

**Two distinct valid canister signatures over the same message are trivially obtainable:**

In the IC canister signature scheme (`rs/crypto/iccsa/src/api.rs`), a signature is a certificate proving that the canister's `certified_data` equals `hash(seed || message)` at a specific block height: [7](#0-6) 

Because the certificate embeds the block height and the subnet's threshold signature over the state root at that height, two certificates obtained at heights H1 and H2 (with the same `certified_data`) are different byte strings — `sig1 ≠ sig2` — yet both verify correctly against the same canister public key and root of trust. The canister does not need to change its `certified_data` between the two requests.

**`validate_sender_info` / `verify_sender_info_canister_sig` accepts both:** [8](#0-7) 

Both signatures verify against the same `SenderInfoContent(info_bytes)` and the same canister public key, so both pass.

### Impact Explanation

An attacker who controls any canister C can:

1. Set `certified_data = hash(seed || SenderInfoContent(info_bytes))` in C.
2. Obtain certificate `sig1` at block height H1 and `sig2` at block height H2 (no state change needed between the two).
3. Submit `msg1` with `sender_info = {info, signer=C, sig=sig1}` → `MessageId1 = hash(..., sig1)`.
4. Submit `msg2` with `sender_info = {info, signer=C, sig=sig2}` → `MessageId2 = hash(..., sig2)`.
5. Both messages pass `validate_sender_info`, pass ingress pool admission, pass consensus-level deduplication, and pass `ValidSetRuleImpl::is_duplicate`.
6. Both are inducted and executed.

For governance votes, ledger transfers, or any state-changing call, this causes double execution of the same logical operation, violating the at-most-once execution invariant.

### Likelihood Explanation

Any principal who can deploy or control a canister on the IC can execute this attack. Obtaining two certificates at different heights is a normal operation (no special privileges, no threshold corruption, no key leakage). The attack requires no timing precision: the two messages can be submitted seconds apart within the same `ingress_expiry` window.

### Recommendation

Remove `sig` from the representation-independent hash. The `MessageId` should be computed from the logical content of the request — `{info, signer}` — but not from the authentication proof `sig`. This is consistent with the existing design decision to exclude the envelope-level `sender_sig` from the hash. The `sig` field should remain validated by `verify_sender_info_canister_sig` but must not influence the `MessageId`.

Concretely, in `representation_independent_hash_call_or_query`:

```rust
// Only hash the content fields, not the signature
map.insert(
    "sender_info",
    Map(btreemap! {
        "info"   => Bytes(info),
        "signer" => Bytes(signer),
        // sig excluded — authentication, not content
    }),
);
```

fn validate_sender_info<C: HttpRequestContent, R: RootOfTrustProvider>(
    request: &HttpRequest<C>,
    ingress_signature_verifier: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    let Some(sender_info) = request.sender_info() else {
        return Ok(());
    };

    // Per the spec, the sender_info signature must verify using the
    // envelope-level sender_pubkey as a canister signature public key.
    let sender_pubkey = match request.authentication() {
        Authentication::Authenticated(sig) => &sig.signer_pubkey,
        Authentication::Anonymous => {
            return Err(InvalidSenderInfo(
                "sender_info requires an authenticated request with sender_pubkey".to_string(),
            ));
        }
    };

    verify_sender_info_canister_sig(
        sender_info,
        sender_pubkey,
        ingress_signature_verifier,
        root_of_trust_provider,
    )
}

/// Verifies that the sender_info canister signature is valid:
/// 1. The envelope-level sender_pubkey is a valid canister signature public key.
/// 2. The canister ID encoded in sender_pubkey matches the signer field.
/// 3. The signature over the info blob is valid against the root of trust.
fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
}
