### Title
ckBTC Minter Withdrawal Processing Permanently Halts When Bitcoin Canister Returns Fewer Than 100 Fee Percentiles — (`rs/bitcoin/ckbtc/minter/src/fees/mod.rs`)

### Summary

The ckBTC minter's `submit_pending_requests` function depends on the Bitcoin canister (`bitcoin_get_current_fee_percentiles`) returning a vector of exactly 100 fee-rate entries. If the Bitcoin canister returns fewer than 100 entries, `estimate_median_fee` returns `None`, and the minter silently skips all pending ckBTC → BTC withdrawal processing with no on-chain override or kill-switch. This is the direct IC analog of M-22: an external oracle returning a degenerate value permanently blocks all "swap" (withdrawal) operations, and there is no governance-accessible circuit-breaker to bypass the throttle without a full canister upgrade.

### Finding Description

**Root cause — `estimate_nth_fee` hard-requires exactly 100 percentiles:**

```rust
// rs/bitcoin/ckbtc/minter/src/fees/mod.rs  lines 103-112
fn estimate_nth_fee(&self, fee_percentiles: &[FeeRate], nth: usize) -> Option<FeeRate> {
    let median_fee = match &self.network {
        Network::Mainnet | Network::Testnet => {
            if fee_percentiles.len() < 100 || nth >= 100 {
                return None;          // ← returns None for any vec shorter than 100
            }
            Some(fee_percentiles[nth])
        }
        ...
    };
    median_fee.map(|f| f.max(self.minimum_fee_per_vbyte()))
}
```

`estimate_median_fee` calls `estimate_nth_fee(..., 50)`, so any response from the Bitcoin canister with fewer than 100 entries yields `None`.

**Propagation — `submit_pending_requests` silently aborts on `None`:**

```rust
// rs/bitcoin/ckbtc/minter/src/lib.rs  lines 358-361
let fee_millisatoshi_per_vbyte = match estimate_fee_per_vbyte(runtime).await {
    Some(fee) => fee,
    None => return,          // ← all pending withdrawals are silently skipped
};
```

`estimate_fee_per_vbyte` itself returns `None` whenever `estimate_median_fee` returns `None`:

```rust
// rs/bitcoin/ckbtc/minter/src/lib.rs  lines 235-253
Ok(fees) => {
    match fee_estimator.estimate_median_fee(&fees) {
        Some(median_fee) => { ... Some(median_fee) }
        None => None,        // ← propagated upward
    }
}
```

The timer-driven task loop calls `submit_pending_requests` on every tick. When the Bitcoin canister returns a short vector, every tick returns early, leaving every queued `RetrieveBtcRequest` permanently in `Pending` state. Users who already burned their ckBTC on the ledger (the burn is committed before the Bitcoin transaction is sent) have no recourse until the minter is upgraded via an NNS governance proposal.

**No on-chain bypass exists.** The minter's `Mode` enum (`ReadOnly`, `RestrictedTo`, `DepositsRestrictedTo`, `GeneralAvailability`) controls access but provides no way to force-process pending requests with a cached or operator-supplied fee rate. The only recovery path is an NNS upgrade proposal, which takes days.

**Secondary panic surface — `estimate_withdrawal_fee` query endpoint:**

```rust
// rs/bitcoin/ckbtc/minter/src/main.rs  lines 229-230
s.last_median_fee_per_vbyte
    .expect("Bitcoin current fee percentiles not retrieved yet.")
```

Any user calling `estimate_withdrawal_fee` before fee percentiles have been fetched (or after a state reset) triggers a canister trap, making the fee-estimation query permanently unavailable until the state is refreshed.

### Impact Explanation

All pending ckBTC → BTC withdrawals are frozen. Because the ckBTC burn on the ledger is committed atomically before the Bitcoin transaction is constructed, affected users have already lost their ckBTC balance with no BTC received and no automatic reimbursement path. The freeze persists across every timer tick until either the Bitcoin canister resumes returning ≥ 100 percentiles or an NNS upgrade proposal is executed (days of latency). This constitutes a permanent lock of user funds reachable through normal chain-fusion user flows.

### Likelihood Explanation

This scenario has already occurred on mainnet. The upgrade proposal `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` documents that on 2025-06-21 the Bitcoin canister returned an anomalously low fee (142 ms/vbyte vs. the expected ~2 000 ms/vbyte), causing three withdrawal transactions to be stuck and requiring an emergency NNS upgrade. The minimum-fee floor added in PR #5742 mitigates the "fee too low" sub-case but does not protect against the Bitcoin canister returning a vector shorter than 100 entries, which is the structural root cause. Low Bitcoin network activity (mempool near-empty) is a realistic trigger for a short percentile vector.

### Recommendation

1. **Fallback to cached percentiles**: When `get_current_fee_percentiles` returns fewer than 100 entries, fall back to `state.last_fee_per_vbyte` (already stored) rather than returning `None`, so pending withdrawals continue to be processed using the last known good fee.
2. **Governance-accessible fee override**: Add an upgrade-arg or NNS-callable setter that allows operators to supply a manual fee rate, bypassing the Bitcoin canister oracle entirely — analogous to the M-22 recommendation of allowing governance to disable the throttle.
3. **Explicit error surfacing**: Replace the silent `return` in `submit_pending_requests` with a logged, metered error so operators are alerted immediately when the fee oracle is degraded.

### Proof of Concept

1. User calls `retrieve_btc` / `retrieve_btc_with_approval`; ckBTC is burned on the ledger; `RetrieveBtcRequest` enters `Pending` queue.
2. Bitcoin canister's `bitcoin_get_current_fee_percentiles` returns a vector with fewer than 100 entries (e.g., empty mempool or canister bug).
3. Every timer tick: `estimate_fee_per_vbyte` → `estimate_median_fee` → `estimate_nth_fee` returns `None` → `submit_pending_requests` returns at line 360 of `lib.rs` without touching the pending queue.
4. State: user's ckBTC is burned, no BTC transaction is ever submitted, `retrieve_btc_status` remains `Pending` indefinitely.

pub async fn estimate_fee_per_vbyte<R: CanisterRuntime>(runtime: &R) -> Option<FeeRate> {
    let btc_network = state::read_state(|s| s.btc_network);
    match runtime
        .get_current_fee_percentiles(&bitcoin_canister::GetCurrentFeePercentilesRequest {
            network: btc_network.into(),
        })
        .await
    {
        Ok(fees) => {
            let fee_estimator = state::read_state(|s| runtime.fee_estimator(s));

                Some(median_fee) => {
                    let fee_based_retrieve_btc_min_amount =
                        fee_estimator.fee_based_minimum_withdrawal_amount(median_fee);
                    log!(
                        Priority::Debug,
