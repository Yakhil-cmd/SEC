### Title
Quarantined Reimbursements Permanently Lock User Tokens With No Recovery Path - (`rs/ethereum/cketh/minter/src/withdraw.rs`)

### Summary

The ckETH/ckERC20 minter's `process_reimbursement()` function uses a `scopeguard` to quarantine a reimbursement if an unexpected panic occurs after contacting the ledger. Once quarantined, the reimbursement is permanently removed from the processing queue and stored as `Err(ReimbursedError::Quarantined)` with no on-chain recovery mechanism. The user's tokens — already burned from the ledger — are permanently lost with no way to reclaim them without a canister upgrade and manual intervention.

### Finding Description

In `rs/ethereum/cketh/minter/src/withdraw.rs`, the `process_reimbursement()` function iterates over pending reimbursement requests and attempts to mint tokens back to users whose withdrawals failed. To prevent double-minting, a `scopeguard` is set up before the async ledger call:

```rust
let prevent_double_minting_guard = scopeguard::guard(index.clone(), |index| {
    mutate_state(|s| process_event(s, EventType::QuarantinedReimbursement { index }));
});
```

If the minter panics after calling the ledger but before recording the successful mint, the guard fires and emits `EventType::QuarantinedReimbursement`. This transitions the reimbursement to `Err(ReimbursedError::Quarantined)` in the state.

The `QuarantinedReimbursement` event is defined in `rs/ethereum/cketh/minter/src/state/event.rs` as:

> "The minter unexpectedly panic while processing a reimbursement. The reimbursement is quarantined to prevent any double minting and will not be processed without further manual intervention."

Once quarantined, the state machine in `rs/ethereum/cketh/minter/src/state/audit.rs` calls `record_quarantined_reimbursement()`, which moves the entry to the `reimbursed` map as `Err(ReimbursedError::Quarantined)`. There is no public endpoint, timer task, or automated mechanism to retry or recover a quarantined reimbursement. The user's ckETH or ckERC20 tokens were already burned from the ledger at withdrawal time, and the minter has no way to re-mint them without a governance-approved canister upgrade.

The same pattern exists in the ckBTC minter (`rs/bitcoin/ckbtc/minter/src/state.rs`, `quarantine_withdrawal_reimbursement()`) and ckDOGE minter (`rs/dogecoin/ckdoge/minter/src/event/mod.rs`, `QuarantinedWithdrawalReimbursement`).

### Impact Explanation

Any user who initiates a ckETH or ckERC20 withdrawal that:
1. Fails on-chain (Ethereum transaction reverts), triggering a reimbursement, AND
2. Causes the minter to panic during the reimbursement mint callback

will have their tokens permanently burned with no automated recovery. The burned tokens are gone from the ledger supply, and the minter's state records the reimbursement as quarantined. The user has no callable endpoint to trigger re-processing. This is a **ledger conservation bug**: tokens are destroyed without being returned to the user.

The ckETH minter is a production canister holding real user funds. The quarantine mechanism is explicitly designed to fire on unexpected panics — a class of bugs that has occurred in practice (as evidenced by the ckBTC stuck-withdrawal incidents documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` and `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2026_03_20.md`).

### Likelihood Explanation

The trigger requires a panic inside the reimbursement callback after the ledger call is dispatched. This is a realistic scenario:
- The IC execution environment can deliver a response that causes a panic in the callback (e.g., unexpected ledger response format, integer overflow in block index conversion via `.expect("block index should fit into u64")`).
- Any future code change that introduces a panic path in the callback will silently quarantine all in-flight reimbursements.
- The `process_reimbursement()` function processes all pending reimbursements in a single timer invocation; a single panic quarantines the current item and stops processing the rest.

The entry path is fully unprivileged: any user can trigger a ckETH or ckERC20 withdrawal, which if it fails on Ethereum, creates a reimbursement request that enters this code path automatically via the minter's timer.

### Recommendation

1. **Add a public `retry_quarantined_reimbursement` endpoint** (callable by the affected user or any principal) that re-queues a quarantined reimbursement for processing, guarded by a deduplication check using `created_at_time` on the ledger transfer to prevent double-minting.
2. **Replace the `.expect()` panic** on block index conversion at line 98 (`block_index should fit into u64`) with a recoverable error path that defuses the guard rather than triggering it.
3. **Add an automated retry timer** that periodically attempts to re-process quarantined reimbursements using idempotent ledger calls (leveraging `created_at_time` deduplication).

### Proof of Concept

**Entry path (unprivileged ingress):**

1. User calls `withdraw_eth` or `withdraw_erc20` on the ckETH minter — burns their ckETH/ckERC20 tokens.
2. The Ethereum transaction fails (reverts on-chain).
3. The minter's `process_retrieve_eth_requests` timer detects the failure and calls `record_finalized_transaction`, which enqueues a `ReimbursementRequest`.
4. The minter's `process_reimbursement` timer fires and calls `client.transfer(args).await` to mint tokens back.
5. If the minter panics at any point after the `await` returns (e.g., at line 98: `.expect("block index should fit into u64")` if the ledger returns a block index > u64::MAX), the `scopeguard` fires.
6. `EventType::QuarantinedReimbursement { index }` is emitted and applied via `apply_state_transition` → `record_quarantined_reimbursement`.
7. The reimbursement is now stored as `Err(ReimbursedError::Quarantined)` — permanently inaccessible.

**Verification:** Query `get_events` on the minter canister and observe `QuarantinedReimbursement` entries. Query `retrieve_eth_status` for the affected withdrawal ID — it will show `TxFinalized(PendingReimbursement(...))` indefinitely, never transitioning to `Reimbursed`.     /// Quarantine the reimbursement request identified by its index to prevent double minting.
    /// WARNING!: It's crucial that this method does not panic,
    /// since it's called inside the clean-up callback, when an unexpected panic did occur before.
    pub fn quarantine_withdrawal_reimbursement(&mut self, burn_index: LedgerBurnIndex) {
        self.pending_withdrawal_reimbursements.remove(&burn_index);
        self.reimbursed_withdrawals
            .insert(burn_index, Err(ReimbursedError::Quarantined));
