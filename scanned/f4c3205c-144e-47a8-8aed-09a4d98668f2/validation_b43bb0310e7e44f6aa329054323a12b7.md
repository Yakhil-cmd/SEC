### Title
Missing Ethereum Precompile Address Guard in `validate_address_as_destination` Allows Permanent ETH Lock — (`rs/ethereum/cketh/minter/src/address.rs`)

### Summary
`validate_address_as_destination` rejects only `Address::ZERO` and OFAC-style blocklisted addresses. Ethereum precompile addresses `0x0000000000000000000000000000000000000001` through `0x0000000000000000000000000000000000000009` pass validation, allowing any non-anonymous caller to burn ckETH and have the minter dispatch a real ETH transfer to a precompile address from which ETH is permanently unrecoverable.

### Finding Description
The validation function has exactly two guards: [1](#0-0) 

1. Reject `Address::ZERO`.
2. Reject addresses present in `ETH_ADDRESS_BLOCKLIST`.

The blocklist is a sorted OFAC/sanctions list whose lowest entry is `0330070FD38Ec3bB94F58FA55D40368271E9e54A`. [2](#0-1) 

Precompile addresses `0x0000000000000000000000000000000000000001`–`0x0000000000000000000000000000000000000009` are numerically far below this range and are not present in the list. No other guard exists — the word "precompile" does not appear anywhere in the ckETH minter source.

The public `withdraw_eth` endpoint calls `validate_address_as_destination` directly and proceeds to burn the caller's ckETH and queue an Ethereum transaction on `Ok`: [3](#0-2) [4](#0-3) 

The same gap exists in `withdraw_erc20`. [5](#0-4) 

### Impact Explanation
Ethereum precompile addresses have no associated private key and no Solidity `withdraw` path. A plain ETH transfer to `0x0000000000000000000000000000000000000001` succeeds at the EVM protocol level (gas cost ≤ 21,000, well within the minter's limit) and credits the balance to that address permanently. The ckETH is burned on the IC ledger, the ETH is gone from the minter's Ethereum account, and neither the user nor the minter can recover it. The ckETH ↔ ETH conservation invariant is broken for the affected withdrawal.

### Likelihood Explanation
The path requires only a non-anonymous IC principal and the minimum withdrawal amount. No privileged role, no governance action, and no threshold corruption is needed. The call is a single `withdraw_eth` invocation with a well-known address. Accidental triggering is unlikely, but deliberate triggering (e.g., to grief the system's accounting or to demonstrate the gap) is trivially reproducible.

### Recommendation
Add an explicit precompile range check inside `validate_address_as_destination`:

```rust
// Ethereum built-in precompiles: 0x01 – 0x09
let raw = address.as_ref(); // [u8; 20]
if raw[..19] == [0u8; 19] && raw[19] >= 1 && raw[19] <= 9 {
    return Err(AddressValidationError::NotSupported(address));
}
```

This mirrors the existing `Address::ZERO` guard and closes the gap for all nine precompile addresses. [6](#0-5) 

### Proof of Concept
Unit test (drop into `rs/ethereum/cketh/minter/src/address/tests.rs`):

```rust
#[test]
fn should_reject_precompile_addresses() {
    for i in 1u8..=9 {
        let mut bytes = [0u8; 20];
        bytes[19] = i;
        let addr = Address::new(bytes);
        assert_eq!(
            validate_address_as_destination(&addr.to_string()),
            Err(AddressValidationError::NotSupported(addr)),
            "precompile 0x{:02x} should be rejected", i
        );
    }
}
```

Under the current code this test **fails** for all nine addresses — each returns `Ok(addr)` — confirming the missing guard. [7](#0-6)
