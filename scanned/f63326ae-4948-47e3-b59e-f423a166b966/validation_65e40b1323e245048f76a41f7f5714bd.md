### Title
SNS Governance `transaction_fee_e8s` in `NervousSystemParameters` Not Updated Atomically with Ledger Fee Change, Allowing Fee-Mismatch During Pending Neuron Operations - (File: rs/sns/governance/src/governance.rs)

### Summary

The SNS governance canister's `perform_manage_ledger_parameters` function upgrades the SNS ledger canister to apply a new `transfer_fee`, and then — only after confirming the upgrade succeeded — updates `NervousSystemParameters.transaction_fee_e8s` in governance state. During the window between the ledger upgrade completing and the governance state being updated (or if the governance canister traps/restarts before the update), the `transaction_fee_e8s` cached in governance diverges from the actual ledger fee. This stale fee value is used by governance to compute neuron operation costs (e.g., `reject_cost_e8s` checks, neuron disbursement, and neuron staking validation), causing unpredictable fee deductions for users whose operations span this window.

### Finding Description

In `perform_manage_ledger_parameters` in `rs/sns/governance/src/governance.rs`, the SNS governance canister:

1. Upgrades the SNS ledger canister with the new `transfer_fee` via `upgrade_non_root_canister`.
2. Polls `canister_info` in a loop until it confirms the upgrade succeeded.
3. Only then updates `nervous_system_parameters.transaction_fee_e8s` in governance state. [1](#0-0) 

The `transaction_fee_e8s` field in `NervousSystemParameters` is the governance canister's cached copy of the ledger fee. It is used in multiple places — including neuron staking validation, proposal submission fee checks, and neuron disbursement — to compute how many tokens are deducted as fees for ledger transfers initiated by governance. [2](#0-1)

The `perform_manage_ledger_parameters` function is called during proposal execution: [4](#0-3) 

The vulnerability window exists because:
- The ledger fee changes **immediately** when the ledger canister is upgraded (step 1).
- The governance cached `transaction_fee_e8s` is only updated **after** the polling loop confirms success (step 3).
- Between these two points, any governance-initiated ledger transfer (e.g., neuron disbursement, maturity distribution) uses the **old** cached fee value when calling `transfer_funds`, but the ledger enforces the **new** fee. This causes transfers to fail with `BadFee` or succeed with an incorrect fee deduction.

Additionally, the `NervousSystemParameters.transaction_fee_e8s` is used to validate that a neuron's minimum stake exceeds the fee: [5](#0-4) 

If the fee increases, neurons that were previously valid may now be below the minimum stake threshold, and governance will use the stale (lower) fee to allow operations that the ledger will reject.

The integration test confirms this fee-change mechanism is live and reachable: [6](#0-5) 

### Impact Explanation

Any SNS token holder can trigger this window by observing when a `ManageLedgerParameters` proposal is executing. During the window:

- **Neuron disbursements** initiated by governance use the stale `transaction_fee_e8s` when calling `transfer_funds`, but the ledger enforces the new fee. If the fee increased, the transfer fails with `BadFee`, leaving the neuron in a broken state until governance retries. If the fee decreased, the governance over-deducts from the neuron's stake.
- **Proposal submission fee checks** (`reject_cost_e8s`) use the stale fee, allowing proposals to be submitted that the ledger will reject when governance tries to execute the associated transfer.
- **Neuron minimum stake validation** uses the stale fee, potentially allowing neurons to be created that are immediately invalid under the new fee regime.

The impact is **ledger conservation bug** / **governance authorization bug**: users' funds can be incorrectly deducted or operations can be incorrectly rejected/accepted based on a fee value that does not match the actual ledger state.

### Likelihood Explanation

- `ManageLedgerParameters` is a standard SNS governance proposal type, reachable by any SNS token holder with sufficient stake.
- The polling loop in `perform_manage_ledger_parameters` introduces a real time window (potentially seconds to minutes) during which the ledger fee and governance cached fee diverge.
- Any concurrent neuron operation (disbursement, staking, maturity distribution) during this window is affected.
- No privileged access, key compromise, or threshold attack is required — only a valid governance proposal execution.

### Recommendation

The `transaction_fee_e8s` in `NervousSystemParameters` should be updated **before** the ledger upgrade is initiated, or the governance canister should re-read the actual ledger fee at the point of use rather than relying on the cached value. Alternatively, all governance-initiated ledger transfers should pass `fee: None` (letting the ledger apply its current fee) and adjust the deduction from neuron stake accordingly, rather than computing the fee from the stale cached parameter.

### Proof of Concept

1. An SNS community submits and passes a `ManageLedgerParameters` proposal to change `transfer_fee` from `10_000` to `50_000`.
2. `perform_manage_ledger_parameters` begins executing: it calls `upgrade_non_root_canister` to upgrade the ledger with the new fee. The ledger now enforces `transfer_fee = 50_000`.
3. Before the polling loop at line 3162 confirms the upgrade and updates `nervous_system_parameters.transaction_fee_e8s`, a neuron owner calls `disburse_neuron`. Governance computes the ledger transfer fee as `10_000` (stale cached value) and calls `transfer_funds(amount, 10_000, ...)`.
4. The ledger rejects the transfer with `BadFee { expected_fee: 50_000 }`, causing the disbursement to fail. The neuron remains locked and the user's funds are inaccessible until the operation is retried after governance state is updated.
5. Conversely, if the fee decreased from `50_000` to `10_000`, governance would over-deduct `50_000` from the neuron's stake for a transfer that only costs `10_000`, permanently destroying `40_000` e8s of the user's tokens. [7](#0-6) [8](#0-7) [9](#0-8)
