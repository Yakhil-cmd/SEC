### Title
Unbounded `ingress_expiries` Allocation via Attacker-Controlled `ingress_start`/`ingress_end` in ICP Rosetta `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The ICP Rosetta `/construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` nanosecond timestamps from the JSON request body with no range validation. These values drive an unbounded `while` loop that pushes one `u64` per ~2-minute interval into a heap-allocated vector. With `ingress_start=0` and `ingress_end=u64::MAX`, the loop executes ~153 billion iterations, allocating ~1.2 TB of heap memory, OOM-killing the Rosetta process. No authentication is required.

---

### Finding Description

**Entrypoint:** `POST /construction/payloads` — registered in `rosetta_server.rs` with no authentication middleware. [1](#0-0) 

The handler calls `req_handler.construction_payloads(msg.into_inner())` synchronously on the actix worker thread.

**Metadata parsing:** `ingress_start` and `ingress_end` are plain `Option<u64>` fields deserialized from JSON with no constraints. [2](#0-1) 

**Vulnerable loop:** The interval is computed as `MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120,000,000,000 ns`. [3](#0-2) 

With `ingress_start=0` and `ingress_end=u64::MAX (≈1.84×10¹⁹ ns)`, the loop runs `⌊u64::MAX / 120_000_000_000⌋ ≈ 153,722,867,280` iterations, each pushing an 8-byte `u64`, requiring ~1.23 TB of heap.

**No guard exists in the ICP Rosetta path.** The ICRC1 Rosetta implementation of the same function has explicit guards that reject oversized windows: [4](#0-3) 

These guards are entirely absent from the ICP Rosetta version.

**Body size limit does not help:** The actix-web JSON body limit is 4 MB, but the malicious payload is just two small integers (`ingress_start: 0, ingress_end: 18446744073709551615`) — well under any body size limit. [5](#0-4) 

---

### Impact Explanation

The Rosetta process is a single, non-replicated off-chain process. OOM-killing it takes down the entire ICP Rosetta service until it is restarted. Any exchange or wallet relying on this Rosetta node for ICP transfers, neuron management, or balance queries loses service. A single unauthenticated HTTP request is sufficient; no volumetric attack is needed.

---

### Likelihood Explanation

The endpoint is publicly reachable on any deployed ICP Rosetta node. The attack requires no credentials, no prior state, and no special knowledge beyond the Rosetta API spec. The malicious values fit in a normal JSON body. The ICRC1 Rosetta sibling already has the fix, confirming the developers are aware of the pattern — the ICP version was simply not updated.

---

### Recommendation

Apply the same guard present in the ICRC1 Rosetta implementation to the ICP Rosetta `construction_payloads` function. Before the while loop, add:

1. Reject if `ingress_end <= ingress_start`.
2. Reject if `(ingress_end - ingress_start)` exceeds a compile-time maximum window (e.g., 24 hours = `86_400_000_000_000 ns`), which caps `ingress_expiries.len()` to at most `86_400_000_000_000 / 120_000_000_000 = 720` entries.

Reference the existing guard in `rs/rosetta-api/icrc1/src/construction_api/services.rs` lines 148–158.
