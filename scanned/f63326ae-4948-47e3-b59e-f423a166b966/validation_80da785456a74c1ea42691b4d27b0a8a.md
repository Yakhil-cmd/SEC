### Title
Missing Length Validation on `Transform.context` and `Transform.method_name` Allows Replicated State Bloat via Canister HTTP Outcalls - (File: `rs/types/types/src/canister_http.rs`)

### Summary
The `CanisterHttpRequestContext` stored in replicated state includes a `Transform` struct with two unbounded string/blob fields — `method_name` and `context` — that are never individually size-checked during request validation. Unlike the URL, headers, and body (all of which have enforced length limits), the transform fields are silently accepted at any size up to the inter-canister message limit (~2 MiB). A malicious canister can submit up to `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT` (3,000) requests each carrying a ~2 MiB transform context, bloating the subnet's replicated state by up to ~6 GiB from this map alone.

### Finding Description

`CanisterHttpRequestContext::generate_from_args` and `generate_from_flexible_args` perform three validation steps before accepting a request:

```
validate_transform_principal(...)   // only checks principal ID, not size
validate_url_length(...)            // enforces MAX_CANISTER_HTTP_URL_SIZE
validate_http_headers_and_body(...) // enforces limits on headers + body
``` [1](#0-0) 

The `validate_http_headers_and_body` function checks header count, individual header name/value lengths, total header size, and total `headers + body` size against `MAX_CANISTER_HTTP_REQUEST_BYTES`. However, it does **not** include `transform.method_name` or `transform.context` in any of these totals: [2](#0-1) 

The `variable_parts_size()` method — used only for fee calculation — does account for the transform fields: [3](#0-2) 

But fee accounting is not a hard cap. A well-funded canister can pay the cycles cost and still store arbitrarily large transform contexts in replicated state.

The accepted `CanisterHttpRequestContext` (including the unbounded transform fields) is inserted directly into `SubnetCallContextManager.canister_http_request_contexts`, which is part of the subnet's replicated state: [4](#0-3) 

The only bound on the number of in-flight contexts is `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT = 3000`: [5](#0-4) [6](#0-5) 

### Impact Explanation

A malicious canister submits 3,000 HTTP outcall requests, each with a `TransformContext.context` blob of ~2 MiB (the inter-canister message size limit). Each request passes all validation checks. The resulting `canister_http_request_contexts` map in replicated state grows to ~6 GiB. This causes:

1. **State sync degradation**: Nodes catching up must download and verify the bloated state, significantly slowing recovery and onboarding.
2. **Checkpoint pressure**: Periodic checkpointing of a multi-GiB metadata section increases latency for all canisters on the subnet.
3. **Memory exhaustion**: Subnets with constrained memory may be unable to hold the full replicated state in memory, causing replica crashes or OOM kills.

Unlike the URL and headers — which are bounded and cannot individually exceed their respective limits — the transform context has no enforced ceiling, making this the only unbounded field that persists in replicated state per HTTP outcall.

### Likelihood Explanation

Any canister on a subnet that has HTTP outcalls enabled can trigger this. No privileged role, governance majority, or key material is required. The attacker only needs sufficient cycles to pay the fee for 3,000 requests with large transform contexts. The cycles cost scales with `variable_parts_size()`, so the attack is not free, but it is economically feasible for a motivated attacker. The HTTP outcalls feature is enabled on production subnets.

### Recommendation

Enforce an explicit maximum length on `Transform.method_name` and `Transform.context` during request validation in `generate_from_args` and `generate_from_flexible_args`. A natural place is alongside the existing `validate_url_length` call:

```rust
fn validate_transform_context_size(
    transform: &Option<TransformContext>,
) -> Result<(), CanisterHttpRequestContextError> {
    if let Some(tc) = transform {
        let method_len = tc.function.0.method.len();
        let context_len = tc.context.len();
        if method_len > MAX_CANISTER_HTTP_TRANSFORM_METHOD_NAME_LENGTH {
            return Err(...);
        }
        if context_len > MAX_CANISTER_HTTP_TRANSFORM_CONTEXT_LENGTH {
            return Err(...);
        }
    }
    Ok(())
}
```

Reasonable limits (e.g., 256 bytes for `method_name`, 64 KiB for `context`) would align with the existing header limits and prevent replicated state bloat while preserving legitimate use cases.

### Proof of Concept

1. Deploy a canister on a subnet with HTTP outcalls enabled.
2. From the canister, call `ic00::http_request` 3,000 times with:
   - `url`: any valid URL (e.g., `"https://example.com"`)
   - `headers`: empty
   - `body`: empty
   - `transform`: `TransformContext { function: <self>, context: vec![0u8; 2_000_000] }`
3. Each call passes `validate_transform_principal` (principal matches), `validate_url_length` (URL is short), and `validate_http_headers_and_body` (headers and body are empty — total = 0 bytes, well under `MAX_CANISTER_HTTP_REQUEST_BYTES`).
4. Each `CanisterHttpRequestContext` with a 2 MiB `transform.context` is inserted into `SubnetCallContextManager.canister_http_request_contexts`.
5. After 3,000 requests, the map holds ~6 GiB of transform context data in replicated state, causing state sync slowdowns and checkpoint pressure on the subnet. [7](#0-6) [1](#0-0) [2](#0-1)

pub fn validate_http_headers_and_body(
    headers: &[HttpHeader],
    body: &[u8],
) -> Result<(), CanisterHttpRequestContextError> {
    let mut headers_num = 0;
    let mut headers_max_name_len = 0;
    let mut headers_max_value_len = 0;
    let mut headers_size_bytes = 0;
    for h in headers.iter() {
        headers_num += 1;
        let name_len = h.name.len();
        let value_len = h.value.len();
        headers_max_name_len = name_len.max(headers_max_name_len);
        headers_max_value_len = value_len.max(headers_max_value_len);
        headers_size_bytes += name_len + value_len;
    }

    if headers_num > MAX_CANISTER_HTTP_HEADER_NUM {
        return Err(CanisterHttpRequestContextError::TooManyHeaders(headers_num));
    }

    if headers_max_name_len > MAX_CANISTER_HTTP_HEADER_NAME_VALUE_LENGTH {
        return Err(CanisterHttpRequestContextError::TooLongHeaderName(
            headers_max_name_len,
        ));
    }

    if headers_max_value_len > MAX_CANISTER_HTTP_HEADER_NAME_VALUE_LENGTH {
        return Err(CanisterHttpRequestContextError::TooLongHeaderValue(
            headers_max_value_len,
        ));
    }

    if headers_size_bytes > MAX_CANISTER_HTTP_HEADER_TOTAL_SIZE {
        return Err(CanisterHttpRequestContextError::TooLargeHeaders(
            headers_size_bytes,
        ));
    }

    let body_size_bytes = body.len();
    let request_total_bytes = headers_size_bytes + body_size_bytes;
    if request_total_bytes > (MAX_CANISTER_HTTP_REQUEST_BYTES as usize) {
        return Err(CanisterHttpRequestContextError::TooLargeRequest(
            request_total_bytes,
        ));
    }

    Ok(())

impl CanisterHttpRequestContext {
    /// Calculate the size of all unbounded struct elements.
    pub fn variable_parts_size(&self) -> NumBytes {
        let request_size = self.url.len()
            + self
                .headers
                .iter()
                .map(|header| header.name.len() + header.value.len())
                .sum::<usize>()
            + self.body.as_ref().map_or(0, |body| body.len())
            + self.transform.as_ref().map_or(0, |transform| {
                transform.method_name.len() + transform.context.len()
            });
        NumBytes::from(request_size as u64)
    }
