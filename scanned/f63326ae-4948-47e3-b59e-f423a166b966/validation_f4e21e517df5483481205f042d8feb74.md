### Title
Unbounded CBOR Deserialization Depth in `SignedTransaction::from_str` Enables Stack Overflow DoS — (`rs/rosetta-api/icp/src/models.rs`)

### Summary
An unprivileged attacker can crash the ICP Rosetta process by POSTing a hex-encoded, deeply nested CBOR payload to `/construction/hash`. The `serde_cbor::from_slice` call has no recursion depth limit, and the 4MB JSON body cap is insufficient to prevent a stack overflow.

### Finding Description
`SignedTransaction::from_str` [1](#0-0)  decodes the hex string and passes the raw bytes directly to `serde_cbor::from_slice` with no depth guard. On failure it retries with `LegacySignedTransaction`. The `serde_cbor` crate's parser is recursive; each CBOR nesting level consumes a stack frame. A CBOR array-of-arrays structure 100,000 levels deep fits comfortably within 2MB of binary data (each level costs ~20 bytes), which in turn fits within the 4MB JSON body limit set at [2](#0-1) . The `/construction/hash` endpoint is wired with no authentication: [3](#0-2) 

### Impact Explanation
A single malformed request causes a stack overflow in the Rosetta process, crashing it. The Rosetta node becomes unavailable until restarted. This does not affect the IC consensus layer or user funds directly, but it is a concrete, repeatable DoS against the Rosetta availability service.

### Likelihood Explanation
The endpoint is publicly reachable, requires no credentials, and the exploit requires only crafting a valid hex string containing deeply nested CBOR — achievable with a few lines of Python using the `cbor2` library. The 4MB body limit provides no meaningful protection against depth-based stack exhaustion.

### Recommendation
- Replace `serde_cbor` (deprecated) with `ciborium`, which supports configurable recursion limits.
- Alternatively, pre-validate the hex-decoded byte length and enforce a maximum CBOR nesting depth before calling `from_slice` (e.g., using a streaming CBOR scanner that counts depth without recursion).
- Apply the same fix to the identical pattern in `ConstructionParseRequest::try_from` at [4](#0-3)  and `UnsignedTransaction::from_str` at [5](#0-4) .
