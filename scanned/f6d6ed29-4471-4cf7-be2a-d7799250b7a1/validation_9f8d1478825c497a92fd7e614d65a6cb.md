### Title
SNS Swap `refresh_buyer_token_e8s` Blocks Existing Participants from Increasing Participation When Neuron-Basket Limit Is Reached — (`rs/sns/swap/src/swap.rs`)

---

### Summary

The `refresh_buyer_token_e8s` function in the SNS Swap canister applies the `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` guard to **all** callers — including buyers already recorded in `self.buyers` — without first checking whether the caller is a new or existing participant. Once the limit is reached, existing participants are silently denied the ability to increase their ICP contribution, directly contradicting the error message the code itself emits.

---

### Finding Description

In `rs/sns/swap/src/swap.rs`, the guard reads:

```rust
let num_direct_participants = self.buyers.len() as u64;
let num_sns_neurons_per_basket = params
    .neuron_basket_construction_parameters
    .as_ref()
    .expect("neuron_basket_construction_parameters must be specified")
    .count;
if (num_direct_participants + 1) * num_sns_neurons_per_basket
    > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(format!(
        "The swap has reached the maximum number of direct participants \
         ({num_direct_participants}) and does not accept new participants; \
         existing participants may still increase their ICP participation amount. …",
    ));
}
``` [1](#0-0) 

`self.buyers.len()` already includes every buyer who has previously participated, so when an **existing** buyer calls `refresh_buyer_tokens`, `num_direct_participants` already counts them. The expression `(num_direct_participants + 1)` therefore asks "what if one more *new* participant joined?" — but the answer is used to gate **all** callers, new and existing alike.

The public endpoint that feeds this function accepts an arbitrary `buyer` string:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
``` [2](#0-1) 

Any ingress sender or canister can call `refresh_buyer_tokens` with their own principal. Once the neuron-basket ceiling is hit, every subsequent call — including top-ups from existing participants — is rejected.

The error message itself acknowledges the intended behaviour ("existing participants may still increase their ICP participation amount") but the code does not implement it: there is no `self.buyers.contains_key(&buyer.to_string())` guard before the check.

The analogous test (`test_refresh_buyer_tokens_no_sns_neuron_baskets_available`) only verifies that a **new** user is blocked and that the existing buyer's *current* balance is unchanged; it never attempts a top-up from an existing participant after the limit is reached, so the regression is untested. [3](#0-2) 

---

### Impact Explanation

When a popular SNS swap fills its neuron-basket quota, every existing participant who tries to increase their ICP contribution receives the same rejection as a brand-new participant. This:

- Permanently denies top-up functionality to all existing participants for the remainder of the swap's open window.
- Contradicts the documented invariant stated in the error string.
- Can be deliberately triggered: a malicious actor who controls enough principals can fill the remaining participant slots, locking out all existing participants from topping up.

---

### Likelihood Explanation

The condition is reachable under normal operation for any SNS swap that configures a large `neuron_basket_count` (e.g., the test uses 33 000 baskets per participant, reaching the ceiling with only three participants). No privileged access, governance majority, or threshold attack is required. Any unprivileged ingress sender can call `refresh_buyer_tokens`; the bug manifests as soon as `(buyers.len() + 1) * basket_count > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`.

---

### Recommendation

Before applying the neuron-basket guard, check whether the buyer is already a participant:

```rust
let is_new_buyer = !self.buyers.contains_key(&buyer.to_string());
if is_new_buyer
    && (num_direct_participants + 1) * num_sns_neurons_per_basket
        > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(…);
}
```

This mirrors the intent stated in the error message and matches the behaviour verified by the existing test for the ICP-target-reached path, where existing participants are correctly allowed to top up. [4](#0-3) 

---

### Proof of Concept

1. Deploy a swap with `neuron_basket_count = 33_000` and `max_direct_participation_icp_e8s = 100 * E8`.
2. Have `user2`, `user3`, `user4` each participate; after three participants `(3+1)*33_000 > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`.
3. `user2` (already in `self.buyers`) calls `refresh_buyer_tokens` with a higher ICP balance in their subaccount.
4. The call returns `"The swap has reached the maximum number of direct participants … existing participants may still increase their ICP participation amount."` — the error message promises the operation is allowed, but the function rejects it.
5. `user2`'s ICP remains locked in the swap subaccount with no way to be accepted until the swap closes and `error_refund_icp` is available. [5](#0-4)

#[test]
fn test_refresh_buyer_tokens_no_sns_neuron_baskets_available() {
    let user1 = PrincipalId::new_user_test_id(1);
    let user2 = PrincipalId::new_user_test_id(2);
    let user3 = PrincipalId::new_user_test_id(3);
    let user4 = PrincipalId::new_user_test_id(4);

    let mut swap = SwapBuilder::new()
        .with_sns_governance_canister_id(SNS_GOVERNANCE_CANISTER_ID)
        .with_lifecycle(Open)
        .with_swap_start_due(Some(START_TIMESTAMP_SECONDS), Some(END_TIMESTAMP_SECONDS))
        .with_min_participants(1)
        .with_min_max_participant_icp(2 * E8, 40 * E8)
        .with_min_max_direct_participation(5 * E8, 100 * E8)
        .with_sns_tokens(100_000 * E8)
        // An extremely large basket size, so we can reach MAX_NEURONS_FOR_DIRECT_PARTICIPANTS with
        // a relatively small number of participants.
        .with_neuron_basket_count(33_000)
        .with_neurons_fund_participation()
        .build();

    let params = swap.params.unwrap();

    let amount_user1_0 = 5 * E8;
    let amount_user2_0 = 40 * E8;
    let amount_user3_0 = 40 * E8;
    let amount_user4_0 = 99 * E8 - (amount_user2_0 + amount_user3_0);

    // All tokens but one should be already bought up by users 2 to 4 --> 99 Tokens were bought
    buy_token_ok(&mut swap, &user2, &amount_user2_0, &amount_user2_0);
    buy_token_ok(&mut swap, &user3, &amount_user3_0, &amount_user3_0);
    buy_token_ok(&mut swap, &user4, &amount_user4_0, &amount_user4_0);

    // Make sure the 99 tokens were registered
    assert_eq!(
        swap.get_buyers_total().buyers_total,
        amount_user2_0 + amount_user3_0 + amount_user4_0
    );

    // Make sure that only an amount smaller than the minimum amount to be bought per user is available
    assert!(
        params.max_direct_participation_icp_e8s.unwrap() - swap.get_buyers_total().buyers_total
            < params.min_participant_icp_e8s
    );

    // No user that has not participated in the swap yet can buy this one token left
    buy_token_err(
        &mut swap,
        &user1,
        &amount_user1_0,
        "The swap has reached the maximum number of direct participants",
    );

    // The one token should still be left fur purchase
    check_final_conditions(
        &mut swap,
        &user2,
        &amount_user2_0,
        &(params.max_direct_participation_icp_e8s.unwrap() - E8),
    );
}
