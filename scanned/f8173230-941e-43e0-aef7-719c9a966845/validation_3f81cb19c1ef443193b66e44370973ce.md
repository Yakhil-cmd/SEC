### Title
Unbounded `disburse_maturity_in_progress` Array Growth in SNS Governance Enables Instruction-Limit DoS - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `disburse_maturity` function appends to a neuron's `disburse_maturity_in_progress` Vec without any cap on its length. An authorized neuron controller can repeatedly call `DisburseMaturity` with small percentages, growing this array arbitrarily. Because the entire neuron proto (including all pending disbursements) is serialized/deserialized on every neuron access, a sufficiently large array causes subsequent operations on that neuron to exceed the IC instruction limit, permanently bricking the neuron's disbursement pipeline. The NNS governance canister has an explicit `MAX_NUM_DISBURSEMENTS = 10` guard for the equivalent operation; SNS governance has no such guard.

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `disburse_maturity` function unconditionally pushes a new `DisburseMaturityInProgress` entry onto the neuron's `disburse_maturity_in_progress` Vec: [1](#0-0) 

There is no check on the current length of `disburse_maturity_in_progress` before this push. The only guards present are:

1. `percentage_to_disburse` must be 1–100.
2. The worst-case modulated amount must exceed the transaction fee. [2](#0-1) 

Neither guard limits how many times the operation can be called. Because each call deducts a percentage of the *remaining* maturity, an attacker with initial maturity `M` can make approximately `log(MINIMUM_DISBURSEMENT_E8S / M) / log(0.99)` calls at 1% per call before the minimum disbursement floor is hit. For a neuron with 1000 ICP of maturity, this is ~686 entries.

The `disburse_maturity_in_progress` field is a `repeated` protobuf field stored inline in the SNS neuron proto: [3](#0-2) 

Every neuron read/write (`get_neuron_result`, `get_neuron_result_mut`) deserializes and re-serializes the entire neuron proto, making each operation O(n) in the number of pending disbursements. The periodic finalization task also iterates over all neurons: [4](#0-3) 

By contrast, the NNS governance canister has an explicit cap: [5](#0-4) 

enforced at: [6](#0-5) 

SNS governance has no equivalent.

---

### Impact Explanation

Once the `disburse_maturity_in_progress` array is large enough, any message that touches the affected neuron (including the periodic `maybe_finalize_disburse_maturity` timer) will exceed the IC instruction limit (40B for application subnets, 50B for system subnets). The neuron becomes permanently stuck: its pending disbursements can never be finalized, and the maturity already deducted from `maturity_e8s_equivalent` is effectively lost. If the SNS governance canister runs on a system subnet, the periodic timer task itself could be disrupted, degrading the entire SNS instance's maturity disbursement pipeline for all neurons.

---

### Likelihood Explanation

The attack requires only that the attacker control an SNS neuron with `DisburseMaturity` permission and sufficient maturity (≥ a few hundred ICP equivalent). This is an unprivileged ingress path — any neuron holder with enough maturity can trigger it. The cost is the maturity itself (which is deducted), making it a self-funded DoS. The attacker does not need any special role, governance majority, or key material.

---

### Recommendation

Add a cap on `disburse_maturity_in_progress` length in `disburse_maturity`, mirroring the NNS governance pattern:

```rust
const MAX_DISBURSE_MATURITY_IN_PROGRESS: usize = 10;

if neuron.disburse_maturity_in_progress.len() >= MAX_DISBURSE_MATURITY_IN_PROGRESS {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!("Too many disbursements in progress (max {MAX_DISBURSE_MATURITY_IN_PROGRESS})"),
    ));
}
```

This check should be performed before the push at line 1696.

---

### Proof of Concept

1. Deploy an SNS instance and stake a neuron with ≥ 1000 SNS tokens of maturity.
2. Repeatedly call `manage_neuron` with `Command::DisburseMaturity { percentage_to_disburse: 1, to_account: None }` until the minimum disbursement floor is reached (~686 calls for 1000 ICP).
3. Observe that `neuron.disburse_maturity_in_progress.len()` grows to ~686.
4. Attempt any subsequent operation that reads/writes the neuron (e.g., another `DisburseMaturity`, or wait for the periodic `maybe_finalize_disburse_maturity` timer to process the neuron).
5. The operation will trap with `CanisterInstructionLimitExceeded` because deserializing and re-serializing a neuron with 686 pending disbursements exceeds the per-message instruction budget.
6. The neuron's pending disbursements are permanently unprocessable; the deducted maturity is lost.
