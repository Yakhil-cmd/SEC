### Title
SNS Governance Reward Purse Inflation via Zero `end_timestamp_seconds` Fallback in `distribute_rewards` - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `distribute_rewards()` function computes the reward start timestamp by calling `.unwrap_or_default()` on the optional `end_timestamp_seconds` field of `latest_reward_event`. When that field is `None` — which occurs for any SNS instance whose persisted `latest_reward_event` predates the introduction of `end_timestamp_seconds` — the fallback value is `0` (Unix epoch). This causes the reward purse to be calculated for every round from epoch 0 to the present, inflating it by orders of magnitude and distributing massively disproportionate maturity to all voters.

---

### Finding Description

**Root cause — `distribute_rewards` in `rs/sns/governance/src/governance.rs`:**

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();          // ← returns 0 when field is None
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [1](#0-0) 

When `end_timestamp_seconds` is `None`, `reward_start_timestamp_seconds = 0`, so `new_rounds_count = now / round_duration_seconds`. For a daily-round SNS running since mainnet launch (~1.7 × 10⁹ seconds), this yields roughly **19,675 rounds** instead of 1.

The rewards purse loop then accumulates interest for every one of those rounds:

```rust
for i in 1..=new_rounds_count {
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [2](#0-1) 

The same zero-fallback appears in `should_distribute_rewards()`, which also uses `.unwrap_or_default()` on `end_timestamp_seconds`, causing the function to immediately return `true` and trigger `distribute_rewards()`: [3](#0-2) 

**Why the initialization guard does not protect against this:**

`Governance::new()` only inserts a properly-timestamped genesis `RewardEvent` when `proto.latest_reward_event` is entirely `None`:

```rust
if proto.latest_reward_event.is_none() {
    proto.latest_reward_event = Some(RewardEvent {
        end_timestamp_seconds: Some(now),   // ← correct
        ...
    })
}
``` [4](#0-3) 

If `latest_reward_event` is already `Some(RewardEvent { end_timestamp_seconds: None, ... })` — the state of any SNS instance whose stable memory was written before `end_timestamp_seconds` (proto tag 5, declared `optional`) was introduced — the guard is skipped and the field remains `None`. [5](#0-4) 

The `round` field is explicitly marked **DEPRECATED** in favour of `end_timestamp_seconds`, confirming the field was added after initial deployment: [6](#0-5) 

---

### Impact Explanation

On the first `run_periodic_tasks()` call after an upgrade that leaves `end_timestamp_seconds: None` in a live SNS:

1. `should_distribute_rewards()` returns `true` immediately (elapsed time appears to be ~54 years).
2. `distribute_rewards()` computes a rewards purse for ~19,675 rounds (daily cadence) instead of 1.
3. The entire inflated purse is distributed as **maturity** to all neurons that voted on any `ReadyToSettle` proposal.
4. Maturity can be converted to governance tokens via `disburse_maturity`, constituting a direct financial loss to the SNS treasury and dilution of all token holders.

No cap on `new_rounds_count` or the purse exists in the code path. [7](#0-6) 

---

### Likelihood Explanation

Any SNS governance canister that:
- Was deployed before `end_timestamp_seconds` was added to `RewardEvent`, **and**
- Has not had a bespoke migration that back-fills `end_timestamp_seconds` in stable memory

will trigger this on the next upgrade + periodic task cycle. The field is `optional` in the proto, so old serialised state deserialises with `end_timestamp_seconds = None`. No migration code was found in `Governance::new()` or elsewhere that handles the case where `latest_reward_event` is `Some` but `end_timestamp_seconds` is `None`. The entry path requires no privileged access: any neuron holder who voted on a proposal receives the inflated maturity automatically.

---

### Recommendation

In `Governance::new()`, extend the guard to also fix a present-but-unset `end_timestamp_seconds`:

```rust
if proto.latest_reward_event.is_none()
    || proto.latest_reward_event
           .as_ref()
           .unwrap()
           .end_timestamp_seconds
           .is_none()
{
    let ts = proto
        .latest_reward_event
        .as_ref()
        .and_then(|e| e.actual_timestamp_seconds.checked_sub(0))
        .unwrap_or(now);
    proto.latest_reward_event = Some(RewardEvent {
        end_timestamp_seconds: Some(ts),
        ..proto.latest_reward_event.clone().unwrap_or_default()
    });
}
```

Alternatively, replace `.unwrap_or_default()` in both `should_distribute_rewards()` and `distribute_rewards()` with a fallback to `genesis_timestamp_seconds` rather than `0`, so that even a missing field cannot produce an epoch-0 start time.

---

### Proof of Concept

1. Deploy an SNS governance canister with a `latest_reward_event` that has `end_timestamp_seconds: None` (simulating a pre-migration state, as used in existing integration tests at `rs/sns/integration_tests/src/neuron.rs` lines 1293–1301 and 1535–1543). [8](#0-7) 
2. Ensure at least one proposal is in `ReadyToSettle` state.
3. Call `run_periodic_tasks()` (triggered automatically by the canister timer).
4. Observe that `should_distribute_rewards()` returns `true` immediately and `distribute_rewards()` computes `new_rounds_count = now / round_duration_seconds` (~19,675 for a daily SNS).
5. Observe that all voting neurons receive maturity equal to ~19,675× the expected single-round reward.
6. Call `disburse_maturity` to convert the inflated maturity to SNS tokens.
