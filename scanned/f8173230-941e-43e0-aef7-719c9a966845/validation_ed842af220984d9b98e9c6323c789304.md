### Title
`HttpClient` Accepts TLS 1.2 and Any Server Certificate, Enabling MITM/Downgrade Attacks - (File: `rs/canister_client/src/http_client.rs`)

### Summary
The production `HttpClient` in `rs/canister_client/src/http_client.rs` is built with `rustls::ClientConfig::builder()` (which defaults to TLS 1.2 + TLS 1.3) and a `DangerAcceptInvalidCerts` verifier that unconditionally accepts any server certificate. This combination allows a network-positioned attacker to intercept connections from the canister client to IC replicas, present a self-signed certificate, and downgrade the session to TLS 1.2, stripping the security guarantees that TLS 1.3 provides.

### Finding Description
In `rs/canister_client/src/http_client.rs`, `HttpClient::new_with_config` constructs a rustls client configuration as follows:

```rust
let mut rustls_config = rustls::ClientConfig::builder()
    .dangerous()
    .with_custom_certificate_verifier(Arc::new(DangerAcceptInvalidCerts {}))
    .with_no_client_auth();
rustls_config.enable_sni = false;
```

`rustls::ClientConfig::builder()` (without `builder_with_protocol_versions`) uses rustls's `DEFAULT_VERSIONS`, which includes both TLS 1.2 and TLS 1.3. The `DangerAcceptInvalidCerts` verifier returns `Ok(ServerCertVerified::assertion())` unconditionally for any certificate, performing no chain, hostname, or identity validation. SNI is also disabled.

The test file `rs/canister_client/tests/tls.rs` explicitly proves this: the test `should_perform_tls_1_2_handshake_with_server_with_bogus_cert` successfully completes a TLS 1.2 handshake with a server presenting a bogus certificate, and the test comment acknowledges: *"This highlights that the canister client trusts ANY server certificate. Depending on the context where the client is used, this may be a security issue since anyone could act as server / MITM."*

By contrast, all production IC node-to-node TLS (replicas, boundary nodes, xnet) is correctly restricted to TLS 1.3 only via `builder_with_protocol_versions(&[&TLS13])`.

### Impact Explanation
An attacker with a network-level position between the canister client and an IC replica (e.g., on the same LAN, a rogue Wi-Fi access point, or a compromised ISP segment) can:
1. Intercept the outgoing HTTPS connection from the `HttpClient`.
2. Present any self-signed certificate — accepted unconditionally by `DangerAcceptInvalidCerts`.
3. Negotiate TLS 1.2 instead of TLS 1.3, losing TLS 1.3's forward secrecy guarantees and exposing the session to known TLS 1.2 weaknesses (BEAST, LUCKY13, POODLE-like attacks on CBC modes).
4. Read or modify query responses, update call acknowledgements, and any other data exchanged between the client and the replica.

This affects all users of `ic_canister_client::HttpClient` and `Agent`, including tooling that submits ingress messages or reads canister state from IC replicas.

### Likelihood Explanation
Medium. The attacker requires a network-level MITM position, which is feasible in many real-world deployment scenarios (shared networks, cloud environments, compromised routers). No privileged IC access is required. The vulnerability is confirmed by a passing test in the repository.

### Recommendation
- **Short term:** Replace `rustls::ClientConfig::builder()` with `rustls::ClientConfig::builder_with_protocol_versions(&[&rustls::version::TLS13])` to restrict the `HttpClient` to TLS 1.3 only, consistent with all other IC TLS configurations.
- **Long term:** Implement proper server certificate verification against the IC registry (as done in `TlsVerifier` for the boundary node) rather than using `DangerAcceptInvalidCerts`. At minimum, pin to known IC root certificates.

// This highlights that the canister client trusts ANY server certificate. Depending on the context

async fn should_perform_tls_1_2_handshake_with_server_with_bogus_cert() {
    let rng = &mut reproducible_rng();
    let server = CustomServer::builder()
        .with_protocol_versions(vec![TlsVersion::TLS1_2])
        .with_allowed_cipher_suites(vec![CipherSuite::TLS12_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256])
        .build(CertWithPrivateKey::builder().build_prime256v1(rng));

    let agent = agent_for(&server);

    let id = CanisterId::from_u64(42);
    let (client_result, server_result) = tokio::join!(
        agent.execute_query(&id, "some method", vec![]),
        server.run()
    );

    assert!(server_result.is_ok());
    // The test server closes the channel without responding after successful TLS handshake,
    // so the agent's client will receive a conn closed error:
    assert!(client_result.err().unwrap().contains("connection closed"));
}
