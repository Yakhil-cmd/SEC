### Title
XNet Stream Delivery Can Be Delayed by Malicious Block Proposer Supplying an Elevated `registry_version` — (File: rs/xnet/payload_builder/src/lib.rs)

---

### Summary

The `registry_version` used to cryptographically verify incoming `CertifiedStreamSlice` certifications during XNet payload validation is taken from the block's `ValidationContext`, which is set by the block proposer and is **not bound to the certification's own height**. A single malicious block proposer (a protocol peer below the consensus fault threshold) can set a `registry_version` that post-dates a DKG key rotation on the sending subnet, causing all previously-certified stream slices to fail signature verification. Because `registry_version` is monotonically non-decreasing across finalized blocks, every subsequent honest block proposer is also forced to use the elevated version, halting XNet message induction until the sending subnet re-certifies its stream under the new key.

---

### Finding Description

**Root cause — `registry_version` is not bound to the certification height.**

In `validate_slice()`, the XNet payload builder decodes and verifies the incoming `CertifiedStreamSlice` using `validation_context.registry_version`: [1](#0-0) 

That call reaches `StateManagerImpl::decode_certified_stream_slice()`, which passes the same `registry_version` to the threshold-signature verifier: [2](#0-1) 

`VerifierImpl::validate()` then calls `verify_combined_threshold_sig_by_public_key` with the caller-supplied `registry_version` to look up the **sending subnet's threshold public key**: [3](#0-2) 

The `CertifiedStreamSlice` itself carries a `Certification` whose `ThresholdSignature.signer` field is a `NiDkgId` identifying the exact DKG transcript used to produce the signature, but **this field is never consulted during XNet slice verification**. The only key-selection input is the externally-supplied `registry_version`. [4](#0-3) 

**How `registry_version` enters the block.**

The block maker sets `ValidationContext.registry_version` to the current "stable" registry version: [5](#0-4) 

The consensus validator only requires that the proposed context is monotonically non-decreasing relative to the parent block and that the local node has already seen the version: [6](#0-5) 

A malicious block proposer can therefore set `registry_version` to any version it has locally observed — including one that post-dates a DKG key rotation on the sending subnet — without violating any consensus rule.

**Attack scenario.**

1. Sending subnet S certifies stream slices at height H using threshold key K₁ (registered at registry version R₁).
2. Subnet S completes a DKG rotation; the new key K₂ is recorded in the NNS registry at version R₂ > R₁.
3. A malicious block proposer on the receiving subnet proposes a block with `ValidationContext.registry_version = R₂`.
4. `validate_slice()` calls `decode_certified_stream_slice(subnet_id, R₂, slice)`. The verifier looks up K₂ at R₂ and attempts to verify a signature produced with K₁ → **`InvalidSignature`**.
5. The block is finalized with no XNet slices from S.
6. Because `registry_version` is monotonically non-decreasing, all subsequent honest block proposers must also use `registry_version ≥ R₂`. They too fail to include slices certified under K₁.
7. XNet message induction from S is halted until S certifies a new state under K₂ and the receiving subnet fetches and pools the new slice.

---

### Impact Explanation

Cross-subnet canister calls (XNet messages) from the affected sending subnet are blocked from being inducted on the receiving subnet for the duration of the delay. Any canister relying on timely cross-subnet responses — DeFi protocols, governance calls, chain-fusion operations — experiences stalled execution. The delay is bounded by the sending subnet's next certification cycle (typically seconds to low minutes), but the attack can be repeated by any block proposer who observes a subsequent DKG rotation, making it a persistent availability degradation tool for a malicious subnet member.

---

### Likelihood Explanation

- **Attacker role**: A single node on the receiving subnet that wins a block-proposal slot. This is a normal protocol participant below the consensus fault threshold (f < n/3), not a privileged role.
- **Precondition**: The sending subnet must have completed a DKG key rotation between R₁ and R₂. IC subnets rotate DKG transcripts at every DKG interval (configurable, typically every ~500 blocks, i.e., roughly every few minutes). This precondition is routinely satisfied in production.
- **Execution**: The attacker simply proposes a block with `context.registry_version` set to a version it has already observed locally. No cryptographic material needs to be forged. The block passes all consensus validation checks.

---

### Recommendation

1. **Bind verification to the certification's own height.** Derive the `registry_version` used for XNet slice signature verification from the `certification.height` embedded in the `CertifiedStreamSlice`, using `registry_version_at_height()` (already used for local certifications in `certifier.rs`), rather than from the block's `ValidationContext.registry_version`. [7](#0-6) 

2. **Alternatively, include the `registry_version` inside the `CertifiedStreamSlice`.** The sending subnet should embed the registry version under which it signed the certification into the slice (analogous to the EVM fix of including `validatorSetId` in the block header). The receiving subnet then validates that this embedded version is consistent with the block's `ValidationContext.registry_version` before using it for signature verification.

---

### Proof of Concept

```
Precondition:
  - Sending subnet S has threshold key K₁ at NNS registry version R₁.
  - S completes a DKG rotation; K₂ is registered at R₂ > R₁.
  - S has certified stream slices at height H using K₁.
  - Receiving subnet R has observed registry version R₂ locally.

Attack:
  1. Malicious node M on R wins a block-proposal slot.
  2. M constructs a BlockProposal with:
       context.registry_version = R₂   // elevated past K₁→K₂ rotation
       context.certified_height  = <current>
       context.time              = <monotonically increasing>
     The block contains no XNet payload from S (or includes the slice,
     which will be rejected during validate_payload).
  3. Other nodes on R validate the proposal:
       local_context.registry_version >= R₂  ✓  (they have also seen R₂)
       context.greater(&parent.context)       ✓
     → Block is notarized and finalized.
  4. All subsequent block proposers on R must use registry_version ≥ R₂.
     validate_slice() calls decode_certified_stream_slice(S, R₂, slice_K₁).
     verify_combined_threshold_sig_by_public_key looks up K₂ at R₂.
     Signature was made with K₁ → InvalidSignature → slice rejected.
  5. XNet induction from S is halted until S certifies a new state under K₂.

            // than local time, assuming the clocks are perfectly in-sync. We choose
            // `delta = initial_notary_delay + 1ns`, because all nodes have to wait at least
            // `initial_notary_delay` time to notarize (and therefore propose subsequent)
            // blocks. The additional 1ns makes no practical difference in that regard.
            time: std::cmp::max(
                self.time_source.get_relative_time(),
                parent.as_ref().context.time + monotonic_block_increment,

        // time increases *strictly* monotonically.
        let proposal = proposal.as_ref();
        if !proposal.context.greater(&parent.context) {
            return Err(InvalidArtifactReason::NonStrictlyIncreasingValidationContext.into());
        }

        let local_context = ValidationContext {
            certified_height: self.state_manager.latest_certified_height(),
            registry_version: self.registry_client.get_latest_version(),
            time: self.time_source.get_relative_time(),
        };

        // If we don't find an instant for the parent block, we fall back to the origin
        // instant which we recorded at validator initialization.
        // The only scenario in which we may have a notarized parent but no instant for
        // the block are replica restarts due to updates or crashes, or while the replica
        // is catching up via CUP. This is not a big problem in practice, because we will
        // always wait at most `proposal.time - parent.time` before validating the
        // proposal. Any heights after that point will have instants, so there will be no
        // further slowdown for other rounds.
        let parent_block_instant = pool_reader
            .get_block_instant(&proposal.parent)
            .unwrap_or(self.time_source.get_origin_instant());
        let duration_since_received_parent = self
            .time_source
            .get_instant()
            .saturating_duration_since(parent_block_instant);

        // Check that our locally available validation context is sufficient for
        // validating the proposal. We require all fields of our local context - with
        // the exception of time - to be greater or equal to the proposal's context.
        //
        // We allow out-of-sync validation, assuming the block proposal's timestamp is
        // not further than the time since we received the parent block.
        // We do this to shield against clock issues, to prevent nodes with lagging
        // clocks to stall a subnet with f malicious replicas.
        let sufficient_local_ctx = local_context.registry_version
            >= proposal.context.registry_version
            && local_context.certified_height >= proposal.context.certified_height
            && std::cmp::max(
                local_context.time,
                parent.context.time + duration_since_received_parent,
            ) >= proposal.context.time;

        if !sufficient_local_ctx {
            return Err(ValidationFailure::ValidationContextNotReached(
                proposal.context.clone(),
                local_context,
            )
            .into());

pub struct ValidationContext {
    /// The registry version to be associated with the payload.
    pub registry_version: RegistryVersion,
    /// The certified state height necessary for the validation of a payload.
    pub certified_height: Height,
    /// The clock time being used for the payload.
    pub time: Time,
}

impl ValidationContext {
    /// The derived PartialOrd trait implementation uses a lexicographic
    /// ordering over its fields, which is not what we want in the case of
    /// ValidationContext. We need every single field to be equal or greater
    /// than those of 'other' to return true. Otherwise, we return false.
    pub fn greater_or_equal(&self, other: &ValidationContext) -> bool {
        self.registry_version >= other.registry_version
            && self.certified_height >= other.certified_height
            && self.time >= other.time
    }

    /// Same as [`Self::greater_or_equal`], except that we require time to be strictly
    /// greater.
    pub fn greater(&self, other: &ValidationContext) -> bool {
        self.registry_version >= other.registry_version
            && self.certified_height >= other.certified_height
            && self.time > other.time
    }
}
