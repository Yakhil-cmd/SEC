### Title
Permanent Minting Failure Locks User ETH/ERC20 Funds with No User-Accessible Recovery Path - (File: rs/ethereum/cketh/minter/src/deposit.rs)

### Summary
The ckETH/ckERC20 minter's `mint()` function treats every ICRC-1 ledger error — including permanent ones such as `TransferError::GenericError` — as transient and retries indefinitely. If the ledger returns a permanent error for a deposit event, that event remains stuck in `events_to_mint` forever. Because the user's ETH/ERC20 is already locked on Ethereum at the point of failure, there is no user-accessible mechanism to recover funds; only an admin-driven minter upgrade can resolve the situation.

### Finding Description

In `rs/ethereum/cketh/minter/src/deposit.rs`, the private `mint()` function iterates over all pending deposit events and calls `icrc1_transfer` on the ICRC-1 ledger to mint ckETH or ckERC20 tokens:

```rust
let block_index = match client
    .transfer(TransferArg { ... })
    .await
{
    Ok(Ok(block_index)) => block_index.0.to_u64()...,
    Ok(Err(err)) => {
        log!(INFO, "Failed to mint {token_symbol}: {event:?} {err}");
        error_count += 1;
        ScopeGuard::into_inner(prevent_double_minting_guard);
        continue;                          // event stays in events_to_mint
    }
    Err(err) => {
        ...
        error_count += 1;
        ScopeGuard::into_inner(prevent_double_minting_guard);
        continue;                          // event stays in events_to_mint
    }
};
```

After the loop, any non-zero `error_count` causes an unconditional reschedule:

```rust
if error_count > 0 {
    ic_cdk_timers::set_timer(crate::MINT_RETRY_DELAY, async { mint().await });
}
```

Both error arms — a ledger-level `TransferError` and a message-delivery failure — are handled identically: the event is left in `events_to_mint` and the timer fires again after `MINT_RETRY_DELAY`. There is **no distinction between transient errors** (e.g., `TemporarilyUnavailable`) **and permanent errors** (e.g., `GenericError` carrying a permanent condition such as a supply cap, a blocked account, or a ledger-level invariant violation introduced by an upgrade).

The ICRC-1 standard explicitly reserves `GenericError { error_code, message }` as a catch-all for conditions not covered by the other variants. If the ckETH or any ckERC20 ledger returns this variant for a permanent reason, the deposit event will cycle through `events_to_mint → mint() → ledger error → events_to_mint` indefinitely.

At the moment the deposit event is first scraped from Ethereum, the user's ETH/ERC20 has already been transferred to the minter's Ethereum address by the helper smart contract. There is no on-chain Ethereum mechanism to reverse this transfer, and the IC minter exposes no endpoint that allows a user to cancel or reclaim a pending deposit. The `QuarantinedDeposit` state — the only escape hatch — is only triggered by an unexpected **panic** in the callback, not by a ledger-returned error:

```rust
let prevent_double_minting_guard = scopeguard::guard(event.clone(), |event| {
    mutate_state(|s| {
        process_event(s, EventType::QuarantinedDeposit { event_source: event.source() })
    });
});
// guard is defused on any error path — quarantine never fires for ledger errors
ScopeGuard::into_inner(prevent_double_minting_guard);
```

The same structural problem exists in `process_reimbursement()` in `rs/ethereum/cketh/minter/src/withdraw.rs` (lines 95–116), where reimbursement mints are also retried indefinitely on any ledger error, meaning a user who burned ckETH for a failed withdrawal could lose both their ckETH and their ETH.

### Impact Explanation

A user who deposits ETH or ERC-20 tokens via the Ethereum helper contract has their funds irrevocably transferred to the minter's Ethereum address. If the corresponding ICRC-1 ledger mint permanently fails (for any reason the ledger encodes as `GenericError`), the user receives neither ckETH/ckERC20 nor a refund of their original assets. The deposit event loops in `events_to_mint` indefinitely. Resolution requires an admin-driven minter upgrade — exactly the same admin-only remediation described in the reference report for sUSX's mintCap scenario.

### Likelihood Explanation

The standard ICRC-1 ledger deployed for ckETH does not currently enforce a supply cap. However, the likelihood is non-negligible because:

1. **Ledger upgrades**: A governance proposal could upgrade the ckETH or any ckERC20 ledger to add supply constraints (e.g., a maximum total supply), causing `GenericError` to be returned for mints that exceed the cap.
2. **Custom ckERC20 ledgers**: Each ckERC20 token has its own ledger canister. A ledger added via the orchestrator could have custom logic that permanently rejects certain mints.
3. **Ledger bugs**: A regression in the ICRC-1 ledger implementation could cause permanent errors for specific mint amounts or accounts.
4. **No retry limit**: Because there is no maximum retry count, even a transient condition that becomes permanent (e.g., a ledger that is stopped and never restarted) produces the same indefinite lock.

The entry path is fully unprivileged: any user can trigger it by depositing ETH to the helper contract.

### Recommendation

1. **Classify errors**: Distinguish `TemporarilyUnavailable` (retry) from `GenericError` / other permanent variants (do not retry; quarantine or alert).
2. **Bounded retries**: Introduce a maximum retry count per deposit event. After exhaustion, emit a `QuarantinedDeposit` event and alert operators, rather than looping forever.
3. **User recovery endpoint**: Expose a canister endpoint (callable by the original depositor) that, for a quarantined deposit, initiates an Ethereum refund transaction back to the depositor's address.
4. **Apply the same fix to `process_reimbursement()`** in `rs/ethereum/cketh/minter/src/withdraw.rs` to prevent the symmetric loss of both burned ckETH and the underlying ETH.

### Proof of Concept

1. User calls `depositEth(amount, principal)` on the Ethereum helper contract; ETH is transferred to the minter's Ethereum address and a `ReceivedEth` log event is emitted.
2. The ckETH minter scrapes the log and records the event via `process_event(s, EventType::AcceptedDeposit(...))`, placing it in `state.events_to_mint`.
3. The ckETH ledger is upgraded (via NNS governance) to enforce a maximum total supply that has already been reached, returning `TransferError::GenericError { error_code: 1, message: "supply cap exceeded" }` for any mint.
4. The minter's timer fires `mint()`. The call to `client.transfer(...)` returns `Ok(Err(TransferError::GenericError {...}))`.
5. The `Ok(Err(err))` arm logs the error, increments `error_count`, defuses the guard (leaving the event in `events_to_mint`), and continues.
6. After the loop, `error_count > 0` causes `ic_cdk_timers::set_timer(MINT_RETRY_DELAY, mint())`.
7. Steps 4–6 repeat indefinitely. The user's ETH remains locked on Ethereum; no ckETH is ever minted; no user-callable endpoint exists to reclaim the ETH.
8. Resolution requires an admin to upgrade the minter (e.g., to quarantine the event and arrange an Ethereum refund), mirroring the admin-only `mintCap` increase required in the sUSX scenario. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
