### Title
ICRC-1 Index-NG Canister Permanently Halted by Mint Block Where Fee Exceeds Amount - (File: `rs/ledger_suite/icrc1/index-ng/src/main.rs`)

### Summary
The `process_balance_changes` function in the ICRC-1 Index-NG canister unconditionally calls `trap()` when processing a `Mint` block whose `fee` field is greater than or equal to the `amount` field. Because the trap rolls back state and the block remains at the same index, every subsequent sync heartbeat re-enters the same code path and traps again, permanently halting the index canister's block processing.

### Finding Description
In `process_balance_changes`, the `Mint` arm computes the net credit to the recipient by subtracting the effective fee from the gross mint amount:

```rust
// rs/ledger_suite/icrc1/index-ng/src/main.rs  lines 1044-1058
Operation::Mint { to, amount, fee } => {
    let mut amount_without_fee = amount;
    let effective_fee = block.effective_fee.or(fee);
    if let Some(fee) = effective_fee {
        amount_without_fee = amount.checked_sub(&fee).unwrap_or_else(|| {
            trap(format!(
                "token amount underflow while indexing block {block_index}"
            ))
        });
        ...
    }
    credit(block_index, to, amount_without_fee)
}
```

The `amount` field in an ICRC-3 Mint block is the **gross** amount (principal + fee). The code subtracts the fee to obtain the net credit. If `fee >= amount`, `checked_sub` returns `None` and `trap()` is called. A trap in a canister message causes the entire message to be rolled back; the index canister's sync cursor does not advance. On the next heartbeat the canister attempts the same block, traps again, and so on indefinitely.

The ICRC-3 standard explicitly permits fees in Mint blocks, as demonstrated by the production test `should_take_mint_block_fee_into_account` (which only exercises the happy path where `MINT_FEE = 10_000 < MINT_AMOUNT = 10_000_000`). The standard ICRC-1 ledger rejects mint-with-fee at the ledger level (`TxApplyError::BurnOrMintFee`), but the index canister is designed to consume **any** ICRC-3-compliant ledger, including custom ones. A malicious or buggy canister implementing ICRC-3 can emit a Mint block with `fee >= amount`.

The identical pattern exists in the Rosetta API at `rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs` lines 363–366, where `checked_sub` failure propagates as an error via `?`, potentially stalling the Rosetta balance-update loop as well. [1](#0-0) [2](#0-1) 

### Impact Explanation
Any ICRC-1 Index-NG canister configured to sync from a ledger that emits a Mint block with `fee >= amount` is permanently halted: it can no longer advance its block cursor, serve `icrc1_balance_of` queries with up-to-date data, or return transaction history. All users relying on that index canister lose access to accurate balance and history information for as long as the canister remains unupgraded. [3](#0-2) 

### Likelihood Explanation
The standard ICRC-1 ledger rejects mint fees at the ledger layer, so the attack surface is limited to custom ICRC-3-compliant ledgers. A canister developer (in scope per the audit rules) who deploys a custom ledger and a paired index canister can trivially trigger this. A legitimate ledger with a fee-accounting bug could also produce such a block accidentally. The ICRC-3 test infrastructure (`BlockBuilder`) already supports constructing Mint blocks with arbitrary fees, confirming the block format is valid at the protocol level. [4](#0-3) 

### Recommendation
Before performing the subtraction, validate that `fee < amount`. If the invariant is violated, the index canister should log a critical error and either skip the block (recording it as unprocessable) or halt gracefully with a human-readable error rather than trapping in a tight loop. The fix mirrors the recommendation in the external report: isolate the fee component before using the gross amount for balance accounting.

```rust
// Proposed guard
if fee > amount {
    ic_cdk::trap(format!(
        "Block {block_index}: mint fee {fee} exceeds mint amount {amount}; \
         this block cannot be processed"
    ));
    // OR: log + skip, depending on desired recovery semantics
}
let amount_without_fee = amount.checked_sub(&fee).expect("validated above");
```

### Proof of Concept
1. Deploy a custom canister that implements the ICRC-3 `icrc3_get_blocks` interface and returns a single Mint block with `amount = 100` and `fee = 200` (or `effective_fee = 200`).
2. Deploy an ICRC-1 Index-NG canister (`rs/ledger_suite/icrc1/index-ng`) configured to sync from the custom canister.
3. Observe that on every heartbeat the index canister traps with `"token amount underflow while indexing block 0"` and its block cursor never advances past block 0.
4. All `icrc1_balance_of` queries to the index canister return stale (zero) balances indefinitely. [5](#0-4)

    fn should_take_mint_block_fee_into_account() {
        const TEST_USER_1: PrincipalId = PrincipalId::new_user_test_id(1);

            owner: TEST_USER_1.0,
            subaccount: None,
        };

        const FEE_COLLECTOR_ACCOUNT: Account = Account {

        };

        const MINT_AMOUNT: u64 = 10_000_000;
        const MINT_FEE: u64 = 10_000;

        let env = StateMachine::new();

        let ledger_id = install_icrc3_test_ledger(&env);

        let mint = BlockBuilder::new(0, GENESIS.as_nanos_since_unix_epoch())
            .with_fee(Tokens::from(MINT_FEE))
            .with_fee_collector(FEE_COLLECTOR_ACCOUNT)
            .mint(TEST_ACCOUNT_1, Tokens::from(MINT_AMOUNT))
            .build();

        let expected_balance = MINT_AMOUNT - MINT_FEE;

        assert_eq!(
            Nat::from(0_u64),
            add_block(&env, ledger_id, &mint)
                .expect("error adding mint block to ICRC-3 test ledger")
        );

        let index_id = install_index_ng(&env, index_init_arg_without_interval(ledger_id));

        wait_until_sync_is_completed(&env, index_id, ledger_id);

        let actual_user_balance =
            icrc1_balance_of(&env, index_id, Account::from(Principal::from(TEST_USER_1)));

        // Verify that the full mint amount, minus the fee, was credited to the user account.
        assert_eq!(
            actual_user_balance, expected_balance,
            "Actual user balance does not match expected balance after mint block ({} vs {})",
            actual_user_balance, expected_balance
        );

        let actual_fee_collector_balance = icrc1_balance_of(
            &env,
            index_id,
            Account::from(Principal::from(FEE_COLLECTOR)),
        );

        // Verify that the mint fee was credited to the fee collector.
        assert_eq!(
            actual_fee_collector_balance, MINT_FEE,
            "Actual fee collector balance does not match expected balance after mint block ({} vs {})",
            actual_fee_collector_balance, MINT_FEE
        );
    }
