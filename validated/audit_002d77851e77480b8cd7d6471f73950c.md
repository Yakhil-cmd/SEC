### Title
Incomplete Refund Guard in `exit_to_near_precompile_callback` Causes Permanent Token Loss on wNEAR Unwrap Failure - (`engine/src/contract_methods/connector.rs`)

### Summary

`exit_to_near_precompile_callback` performs a refund only when `args.refund` is `Some`. When the `error_refund` Cargo feature is disabled (which is the case in the default and `contract` production builds), `args.refund` is always `None`. For the wNEAR-unwrap exit path, a callback is still attached (because `transfer_near` is `Some`), but if the underlying `near_withdraw` promise fails, the callback's `else { None }` branch silently returns `Ok(None)` with no refund. The ERC-20 wNEAR tokens were already burned in the EVM before the promise was created, so they are permanently lost.

### Finding Description

**Root cause — conditional refund check that does not cover all situations:**

In `engine-precompiles/src/native.rs`, the `ExitToNearPrecompileCallbackArgs` struct is populated as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

The callback is attached to the promise only when `callback_args != default()`:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { base: transfer_promise, callback: ... })
};
``` [2](#0-1) 

For the wNEAR-unwrap exit path, `transfer_near_args` is `Some(...)`, so `callback_args != default()` and the callback **is** attached. However, `refund` is `None` because `error_refund` is not a default or `contract` feature: [3](#0-2) 

Inside `exit_to_near_precompile_callback`, the three-branch logic is:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    if let Some(args) = args.transfer_near { /* transfer NEAR */ }
    None
} else if let Some(args) = args.refund {
    // refund on error — only reached when refund is Some
    engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent success even though promise failed and tokens are burned
};
``` [4](#0-3) 

The refund branch is guarded by `if let Some(args) = args.refund`. When `args.refund` is `None` (production build), the `else { None }` branch is taken unconditionally on any promise failure, returning `Ok(None)` with no error and no refund.

**Exploit path:**

1. An unprivileged EVM user holds wNEAR ERC-20 tokens inside Aurora.
2. The user calls the `exitToNear` precompile with the `:unwrap` suffix, triggering the wNEAR-unwrap branch in `exit_erc20_token_to_near`.
3. The precompile burns the wNEAR ERC-20 tokens in the EVM and schedules a `near_withdraw` promise on the wNEAR NEP-141 contract, with `exit_to_near_precompile_callback` as the callback.
4. The `near_withdraw` promise fails (e.g., the wNEAR contract is paused, the Aurora engine's wNEAR balance is insufficient, or any other NEAR-side error).
5. `exit_to_near_precompile_callback` is invoked. `handler.promise_result(0)` is not `Successful`, and `args.refund` is `None`.
6. The `else { None }` branch executes, returning `Ok(None)`.
7. The wNEAR ERC-20 tokens are permanently burned with no NEAR transferred and no ERC-20 refund. [5](#0-4) 

### Impact Explanation

**Permanent freezing/loss of user funds.** Any user who triggers the wNEAR-unwrap exit path and whose `near_withdraw` promise fails loses their wNEAR ERC-20 tokens permanently. The tokens are burned in the EVM before the NEAR-side promise is even attempted, and the callback provides no recovery path when `error_refund` is disabled. This matches the "Permanent freezing of funds" critical impact category.

### Likelihood Explanation

The `near_withdraw` promise can fail for reasons outside the user's control: the wNEAR contract being paused, a NEAR runtime error, or an edge-case balance discrepancy. The user only needs to call the `exitToNear` precompile with the `:unwrap` suffix — a standard, documented, unprivileged operation. The `error_refund` feature is not enabled in the default or `contract` production build, so the gap is always present in deployed code.

### Recommendation

1. Enable `error_refund` in the `contract` feature set, or make it a default feature, so that `args.refund` is always populated for exit paths that burn tokens.
2. Alternatively, treat a `None` refund with a failed promise as an explicit error rather than a silent `Ok(None)`: replace the `else { None }` branch with an explicit error return (e.g., `return Err(errors::ERR_REFUND_FAILURE.into())`), which at minimum prevents silent data loss and surfaces the failure.
3. Ensure that for every exit path that burns ERC-20 tokens before the NEAR-side promise, a refund path exists regardless of feature flags.

### Proof of Concept

1. Deploy Aurora with the `contract` feature (no `error_refund`).
2. Mint wNEAR ERC-20 tokens to an EVM address `A`.
3. From address `A`, call the `exitToNear` precompile with flag `0x01` (ERC-20), the wNEAR ERC-20 address as caller, and recipient `some.near:unwrap`.
4. The precompile burns the wNEAR ERC-20 balance of `A` and schedules `near_withdraw` on the wNEAR NEP-141 contract.
5. Arrange for the `near_withdraw` promise to fail (e.g., pause the wNEAR contract before the promise executes).
6. Observe that `exit_to_near_precompile_callback` is called, hits the `else { None }` branch, and returns `Ok(None)`.
7. Verify that address `A`'s wNEAR ERC-20 balance is zero and no NEAR was transferred — funds are permanently lost. [6](#0-5) [1](#0-0)
