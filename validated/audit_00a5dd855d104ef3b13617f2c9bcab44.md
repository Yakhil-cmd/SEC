### Title
`native_per_pubdata` Integer Division Truncation to Zero Silently Waives All Pubdata Fees — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio is computed with floor division (`wrapping_div`). When `pubdata_price < native_price`, the result truncates to zero. Every downstream consumer of this value then treats pubdata as completely free, bypassing the native-resource charge for all pubdata generated by the transaction. This is the direct ZKsync OS analog of the EUSD H-06 class: an integer-division result of zero is silently assumed to mean "no cost" when it can also arise from truncation of a non-zero ratio.

---

### Finding Description

In `validation_impl.rs` the two per-unit ratios are computed back-to-back:

```rust
// line 135 — uses div_ceil: result is 0 only when gas_price == 0
u256_try_to_u64(&gas_price.div_ceil(native_price))

// line 142 — uses wrapping_div (floor): result is 0 whenever pubdata_price < native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`native_per_gas` deliberately uses `div_ceil` so that any non-zero gas price always produces at least 1, preventing the "unlimited native" branch from being entered spuriously. `native_per_pubdata` uses `wrapping_div` (floor), so any `pubdata_price` in the range `[1, native_price - 1]` silently produces 0.

Every place that consumes `native_per_pubdata` then behaves as if pubdata is free:

**`create_resources_for_tx` (gas_helpers.rs line 352)**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0; no native deducted for intrinsic pubdata
```

**`get_resources_to_charge_for_pubdata` (gas_helpers.rs line 430)**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // → 0; no native charged for execution pubdata
    .ok_or(out_of_native_resources!())?;
```

**`check_enough_resources_for_pubdata` (gas_helpers.rs line 454)**
```rust
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata == 0 → always true; pubdata check never fails
```

The same floor-division path is replicated in `api/src/helpers.rs` line 427:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

---

### Impact Explanation

When `pubdata_price` is set to any value strictly less than `native_price` (e.g., `pubdata_price = 5`, `native_price = 10`):

1. `native_per_pubdata = 0`.
2. The intrinsic pubdata overhead charged during resource creation is 0, giving the transaction more native budget than it should have.
3. `check_enough_resources_for_pubdata` always returns `enough = true`, so the post-execution pubdata gate never rejects the transaction regardless of how much pubdata it generates.
4. The actual native charged for pubdata is 0 for every byte, even though the operator intended to charge `pubdata_price / native_price` native per byte.

A user can therefore submit transactions that produce the maximum allowed pubdata per block at zero native cost, effectively stealing block pubdata capacity without paying for it. The operator's pubdata revenue is entirely zeroed out for any block context where `pubdata_price < native_price`.

---

### Likelihood Explanation

The operator supplies `pubdata_price` and `native_price` as block-level oracle inputs. During periods of low L1 data-availability cost, `pubdata_price` can legitimately be set below `native_price`. The `BlockMetadataFromOracle::new_for_test()` default already sets `pubdata_price = 0`, confirming this is a normal operating range. Any sequencer configuration where `pubdata_price < native_price` silently activates the vulnerability for every L2 transaction in that block. No special attacker capability is required beyond submitting a normal L2 transaction.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. This ensures that any non-zero `pubdata_price` always produces at least 1 native unit per pubdata byte, preventing the silent zero-cost path.

---

### Proof of Concept

**Setup**: operator sets `pubdata_price = 5`, `native_price = 10`.

**Step 1** — `native_per_pubdata` calculation:
```
native_per_pubdata = floor(5 / 10) = 0
```

**Step 2** — resource creation (`create_resources_for_tx`):
```
intrinsic_pubdata_overhead = 0 * L2_TX_INTRINSIC_PUBDATA = 0
→ native_limit is NOT reduced; transaction gets full native budget
```

**Step 3** — post-execution pubdata check (`check_enough_resources_for_pubdata`):
```
resources_for_pubdata = 0 * pubdata_bytes_used = 0
→ resources.has_enough(0) == true always
→ transaction passes regardless of pubdata generated
```

**Step 4** — refund calculation (`compute_gas_refund`):
```
to_charge_for_pubdata = 0
→ no native deducted for pubdata at refund time
```

**Result**: a transaction generating 10,000 bytes of pubdata pays 0 native for it. With `div_ceil`, the same transaction would pay `ceil(5/10) * 10,000 = 10,000` native units, correctly reflecting the operator's intended pricing.
