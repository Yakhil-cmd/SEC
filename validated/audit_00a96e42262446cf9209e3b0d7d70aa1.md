### Title
SNS Governance `follow` Command Permanently Blocks Neuron Owners from Clearing Followees After Generic Function Deregistration - (`rs/sns/governance/src/governance.rs`)

### Summary
When a `RemoveGenericNervousSystemFunction` proposal is executed in SNS governance, any neuron that previously set followees for that function ID can no longer clear those followees via the `follow` command. The `is_registered_function_id` guard fires unconditionally — even when the caller's intent is to *unfollow* (pass an empty followees list) — permanently locking stale following relationships into the neuron's state with no alternative removal path.

### Finding Description
The SNS `follow` function validates the requested `function_id` against the live registry before doing anything else: [1](#0-0) 

`is_registered_function_id` returns `false` for any function that has been removed via `perform_remove_generic_nervous_system_function`, which inserts a `NERVOUS_SYSTEM_FUNCTION_DELETION_MARKER` rather than deleting the entry: [2](#0-1) [3](#0-2) 

Because the guard fires before the branch that handles an empty followees list (the unfollow path), a neuron owner who wants to clear their followees for a deregistered function receives:

> `"Function with id: X is not present among the current set of functions."`

The `set_following` (topic-based) command is also not a workaround: it only removes legacy per-function followees for functions returned by `get_custom_functions_for_topic`, which itself skips deregistered functions: [4](#0-3) 

The stale `followees` entries therefore persist in the neuron's on-chain state indefinitely.

### Impact Explanation
1. **Permanent stale state**: Neuron owners cannot remove followees for deregistered functions. The neuron's `followees` map retains entries that can never be cleaned up through any user-accessible path.
2. **Silent re-registration risk**: `perform_add_generic_nervous_system_function` uses `is_registered_function_id` to gate re-registration; because the deletion marker makes the check return `false`, the same numeric ID *can* be re-registered with a completely different target canister. At that point the stale followees silently apply to the new function — the neuron owner's vote on proposals for the new function is now influenced by whoever they previously followed, without their knowledge or consent.
3. **No temporary workaround**: Unlike the Fenix Finance case (where the ManagedNFT operator can recast votes to exclude the killed gauge), there is no privileged operator who can clear a neuron's followees on behalf of the owner.

### Likelihood Explanation
Any SNS that legitimately removes a generic nervous system function (a routine governance operation) triggers this condition for every neuron that had followees set for that function. The entry path is an unprivileged ingress call to `manage_neuron` with a `Follow` command, reachable by any neuron owner. No malicious majority is required to expose the bug; a single legitimate removal proposal suffices.

### Recommendation
Move the `is_registered_function_id` check so it only applies when the caller is *adding* followees (non-empty list). When the followees list is empty (unfollow intent), the function should proceed regardless of whether the function ID is currently registered:

```rust
if !f.followees.is_empty()
    && !is_registered_function_id(f.function_id, &self.proto.id_to_nervous_system_functions)
{
    return Err(GovernanceError::new_with_message(
        ErrorType::NotFound,
        format!("Function with id: {} is not present ...", f.function_id),
    ));
}
```

Alternatively, when `perform_remove_generic_nervous_system_function` executes, iterate over all neurons and remove followees for the deregistered function ID, analogous to the Fenix Finance recommendation of skipping killed gauges gracefully.

### Proof of Concept

1. Neuron A calls `manage_neuron { Follow { function_id: 1000, followees: [NeuronB] } }` — succeeds while function 1000 is registered.
2. A governance proposal `RemoveGenericNervousSystemFunction(1000)` is adopted and executed; `NERVOUS_SYSTEM_FUNCTION_DELETION_MARKER` is inserted for ID 1000.
3. Neuron A calls `manage_neuron { Follow { function_id: 1000, followees: [] } }` to unfollow — the `is_registered_function_id` check at line 3998 returns `false` and the call is rejected with `ErrorType::NotFound`.
4. Neuron A's `followees` map permanently retains `{ 1000 → [NeuronB] }` with no removal path.
5. If a new `AddGenericNervousSystemFunction` proposal registers a different canister under ID 1000, Neuron A's votes on those proposals are silently influenced by NeuronB without Neuron A's owner being able to prevent it.

```rust
pub fn is_registered_function_id(
    function_id: u64,
    nervous_system_functions: &BTreeMap<u64, NervousSystemFunction>,
) -> bool {
    // Check if the function id is present among the native actions.
    if Action::native_function_ids().contains(&function_id) {
        return true;
    }

    match nervous_system_functions.get(&function_id) {
        None => false,
        Some(function) => function != &*NERVOUS_SYSTEM_FUNCTION_DELETION_MARKER,
    }
}
```
