### Title
Stale wNEAR Address in Pending XCC Promise Causes Permanent User Fund Freeze - (File: `engine/src/contract_methods/xcc.rs`, `engine-precompiles/src/xcc.rs`)

---

### Summary

`factory_set_wnear_address` allows the owner to update the wNEAR ERC-20 address stored in engine state at any time. The XCC precompile reads this address during EVM execution to `transferFrom` the user's wNEAR tokens into the engine's implicit address. A subsequent NEAR callback (`withdraw_wnear_to_router`) re-reads the wNEAR address from storage to call `withdrawToNear`. If the owner changes the wNEAR address between these two steps, the callback calls `withdrawToNear` on the wrong contract, the withdrawal fails, and the user's wNEAR tokens are permanently frozen inside the engine's implicit EVM address with no recovery path.

---

### Finding Description

The XCC cross-contract call flow involves two separate on-chain transactions:

**Transaction A — EVM execution (`submit`):**

1. The XCC precompile (`engine-precompiles/src/xcc.rs`) reads `wnear_address` from storage: [1](#0-0) 

2. It calls `transferFrom` on that address, moving the user's wNEAR tokens from their EVM address to the engine's implicit address: [2](#0-1) 

3. `handle_precompile_promise` (in `engine/src/xcc.rs`) schedules a `withdraw_wnear_to_router` NEAR callback: [3](#0-2) 

**Between Transaction A and Transaction B — owner calls `factory_set_wnear_address`:**

The owner can change the wNEAR address at any time with no guard against in-flight XCC promises: [4](#0-3) 

This immediately overwrites the stored address: [5](#0-4) 

**Transaction B — `withdraw_wnear_to_router` callback:**

The callback re-reads `wnear_address` from storage — now the new address — and calls `withdrawToNear` on it: [6](#0-5) 

The engine's implicit address holds a balance at the **old** wNEAR contract, not the new one. The `withdrawToNear` call on the new contract fails. The callback returns `ERR_WITHDRAW_FAILED`: [7](#0-6) 

The user's wNEAR tokens remain in the engine's implicit EVM address at the old wNEAR contract. There is no refund path in the callback and no mechanism for a user to recover tokens held by the engine's implicit address (which has no corresponding private key).

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

The user's wNEAR tokens are transferred from their EVM account to the engine's implicit address during EVM execution. If the `withdraw_wnear_to_router` callback fails due to a stale wNEAR address, those tokens are stranded. The engine's implicit address (`near_account_to_evm_address(engine_account_id.as_bytes())`) has no private key and cannot be used to sign EVM transactions. No user-accessible function exists to sweep or recover ERC-20 tokens held by this address. Unless the owner reverts the wNEAR address change and the user retries (which is not guaranteed and has no protocol enforcement), the funds are permanently frozen.

---

### Likelihood Explanation

**Likelihood: Medium.**

The wNEAR address is expected to change when the wNEAR contract is upgraded or migrated — a realistic operational event. The NEAR callback (`withdraw_wnear_to_router`) executes in a subsequent block after the initial `submit` transaction. Any `factory_set_wnear_address` call landing in that inter-block window (or even in the same block before the callback) will trigger the bug. Because XCC is a production feature used by real users, and wNEAR contract migrations are plausible, the intersection of these events is realistic. No attacker action is required; the owner's legitimate administrative action is sufficient.

---

### Recommendation

Record the wNEAR address at the time of the initial XCC precompile execution and pass it through the promise chain rather than re-reading it from storage in the `withdraw_wnear_to_router` callback. Concretely, include the wNEAR address in the `WithdrawWnearToRouterArgs` struct so the callback uses the address that was active when the user's tokens were transferred: [8](#0-7) 

Alternatively, add a guard in `factory_set_wnear_address` that prevents the address from being changed while any XCC router sub-accounts have pending in-flight operations, or emit an event so off-chain tooling can detect and remediate affected in-flight calls.

---

### Proof of Concept

1. User calls an EVM contract that invokes the XCC precompile with `required_near > 0`.
2. XCC precompile reads `wnear_address = A`, calls `transferFrom(user, engine_implicit, amount)` on contract `A`. User's wNEAR balance at `A` decreases; engine's implicit address balance at `A` increases.
3. `handle_precompile_promise` schedules a `withdraw_wnear_to_router` NEAR callback targeting the engine.
4. Before the callback executes, the owner calls `factory_set_wnear_address(B)`.
5. `withdraw_wnear_to_router` callback fires: reads `wnear_address = B`, calls `withdrawToNear` on contract `B`. Engine's implicit address has zero balance at `B`. The EVM call fails.
6. Callback returns `ERR_WITHDRAW_FAILED`. The XCC call is aborted.
7. User's wNEAR tokens remain locked in the engine's implicit address at contract `A`. No recovery function exists. Funds are permanently frozen.

pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));
        Ok(())
    })
}
