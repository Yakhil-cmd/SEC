### Title
NEP-141 Fee-on-Transfer Token Overbridging Causes Bridge Insolvency - (File: `engine/src/engine.rs`)

### Summary

`receive_erc20_tokens` unconditionally mints ERC-20 tokens equal to `args.amount` — the value reported by the calling NEP-141 contract — without verifying the actual NEP-141 balance change credited to Aurora. If the NEP-141 token deducts a transfer fee, Aurora holds fewer NEP-141 tokens than the ERC-20 supply it has minted, making the bridge insolvent for that token.

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` is called by a NEP-141 token contract after a user invokes `ft_transfer_call`. The `FtOnTransferArgs.amount` field is supplied by the NEP-141 contract itself and represents the nominal transfer amount, not the net amount actually credited to Aurora's NEP-141 balance. [1](#0-0) 

The engine then routes to `receive_erc20_tokens`, which reads `args.amount` directly and passes it to `setup_receive_erc20_tokens_input` to build an ERC-20 `mint` call: [2](#0-1) 

Specifically, line 803 extracts the nominal amount: [3](#0-2) 

And line 831 uses it verbatim to mint: [4](#0-3) 

`setup_receive_erc20_tokens_input` encodes this amount directly into the ERC-20 `mint` selector call: [5](#0-4) 

There is no balance snapshot taken before and after the NEP-141 transfer to verify the actual amount received. The engine never queries its own NEP-141 balance to reconcile the difference.

### Impact Explanation

If a NEP-141 token registered with Aurora charges a transfer fee (e.g., 1%), a user bridging 1000 tokens causes Aurora to receive 990 tokens but mint 1000 ERC-20 tokens. The ERC-20 total supply on Aurora exceeds the NEP-141 reserve held by Aurora. When users attempt to exit (burn ERC-20 → receive NEP-141), the last users to exit will find Aurora's NEP-141 balance exhausted. Their ERC-20 tokens are permanently frozen with no corresponding NEP-141 to redeem — a direct bridge insolvency and permanent fund freeze.

**Impact: Critical — Permanent freezing of funds / Insolvency.**

### Likelihood Explanation

Any NEP-141 token that (a) has been registered with Aurora via `deploy_erc20_token` and (b) charges a fee on `ft_transfer_call` triggers this path. The attacker-controlled entry is `ft_transfer_call` on the NEP-141 contract, which is a standard, permissionless user action. No admin access is required. The discrepancy accumulates with every bridge-in transaction, so even a small fee causes progressive insolvency. A malicious token deployer can set a fee of up to 100% and drain the bridge reserve in a single transaction.

### Recommendation

In `receive_erc20_tokens`, record Aurora's NEP-141 balance before and after the transfer and use the delta as the mint amount, not `args.amount`. Concretely:

1. Before calling `ft_on_transfer` (or at the start of `receive_erc20_tokens`), query Aurora's current NEP-141 balance for the token.
2. After the transfer is credited, query the balance again.
3. Use `balance_after - balance_before` as the `amount` passed to `setup_receive_erc20_tokens_input`.

Alternatively, enforce at token registration time that only NEP-141 tokens with no transfer fee are eligible for bridging, and document this as a hard invariant in the asset listing checklist.

### Proof of Concept

1. Deploy a NEP-141 token `fee_token.near` that deducts 10% on every `ft_transfer_call`.
2. Register it with Aurora: call `deploy_erc20_token` with `fee_token.near`.
3. As attacker, call `ft_transfer_call` on `fee_token.near` with `receiver_id = aurora`, `amount = 1000`, `msg = <attacker_evm_address>`.
4. `fee_token.near` credits 900 tokens to Aurora's balance and calls `ft_on_transfer` with `amount = 1000`.
5. Aurora's `receive_erc20_tokens` reads `args.amount = 1000` at line 803 and mints 1000 ERC-20 tokens to the attacker.
6. Aurora holds 900 NEP-141 tokens but has issued 1000 ERC-20 tokens.
7. Attacker calls the `ExitToNear` precompile to burn 1000 ERC-20 and withdraw 1000 NEP-141 — Aurora's balance goes to -100 (or the exit fails for the last 100 tokens of any other user).
8. Any honest user who bridged before the attacker now has ERC-20 tokens they cannot redeem — permanent fund freeze.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[must_use]
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);

    [selector, tail.as_slice()].concat()
}
