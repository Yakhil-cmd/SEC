Audit Report

## Title
Unbounded Ingress Expiry Loop with `u64` Wrap-Around Enables DoS via Infinite Loop or OOM — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from request metadata and feeds them directly into an unbounded `while` loop with no validation. The `Time::add_assign` implementation performs a truncating `u128 → u64` cast, causing `now` to wrap to a small value when it approaches `u64::MAX`, making the loop condition permanently true. A single unauthenticated HTTP POST with `ingress_end = u64::MAX` permanently blocks an actix-web worker thread; a moderately large window (e.g., 10 years) causes massive heap allocation leading to OOM.

## Finding Description

**Vulnerable loop** — `construction_payloads.rs` lines 59–107: The `interval` is computed as `MAX_INGRESS_TTL (300s) - PERMITTED_DRIFT (60s) - 120s = 120s = 120,000,000,000 ns`. Both `ingress_start` and `ingress_end` are taken directly from attacker-supplied `u64` nanosecond values with no validation before the loop:

```rust
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;   // Time::add_assign — wraps silently
}
``` [1](#0-0) 

**`Time::add_assign` wraps on overflow** — `time.rs` lines 55–58 and 102–105: `AddAssign` calls `from_duration`, which casts `Duration::as_nanos()` (a `u128`) to `u64` with a truncating `as` cast. When `now` is near `u64::MAX` and `interval` (120,000,000,000 ns) is added, the `u128` result exceeds `u64::MAX` and wraps to approximately `120,000,000,000` — far below `u64::MAX` — making `now < ingress_end` true again, producing an infinite loop. [2](#0-1) [3](#0-2) 

**No guards in ICP Rosetta** — The ICRC1 Rosetta `services.rs` lines 148–158 at least checks `ingress_start >= ingress_end` before its loop. ICP Rosetta has no equivalent guard whatsoever. [4](#0-3) 

**Handler is synchronous on the actix-web worker thread** — `rosetta_server.rs` lines 124–131: `req_handler.construction_payloads(msg.into_inner())` is called without `.await`, meaning the entire loop executes synchronously on the actix-web thread pool with no per-request timeout. The 4 MB JSON body limit at lines 293–303 applies only to the incoming request body, not to server-side computation. [5](#0-4) [6](#0-5) 

**`MAX_INGRESS_TTL` and `PERMITTED_DRIFT` values confirmed** — `rs/limits/src/lib.rs` lines 17 and 21 confirm `MAX_INGRESS_TTL = 300s` and `PERMITTED_DRIFT = 60s`, making `interval = 120s`. [7](#0-6) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Rosetta … security impact with concrete user or protocol harm."*

| Payload | Outcome |
|---|---|
| `ingress_start=0, ingress_end=u64::MAX` | Infinite loop; worker thread blocked permanently |
| `ingress_start=now, ingress_end=now+10yr` | ~2.6M iterations; ~1 GB+ heap allocation; OOM crash |

The ICP Rosetta is a single process. Blocking all actix-web worker threads or triggering OOM takes down the entire service, preventing exchanges and users from constructing or submitting ICP transactions via Rosetta.

## Likelihood Explanation

No authentication is required for `POST /construction/payloads`. The malicious payload is a small (~200 byte) JSON object with two integer fields. A single request is sufficient to trigger either the infinite loop or OOM condition. The attack is trivially reproducible locally and requires no special privileges, victim interaction, or external dependencies.

## Recommendation

Add a maximum ingress window guard before the loop, analogous to the ICRC1 Rosetta pattern, and additionally cap the window size to a reasonable maximum (e.g., 24 hours):

```rust
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request("ingress_start must be before ingress_end"));
}
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 3600);
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request("ingress window exceeds maximum allowed duration"));
}
```

This bounds the loop to at most `24*3600 / 120 = 720` iterations. Additionally, consider using `Time::checked_add` instead of `AddAssign` in the loop to avoid silent wrap-around.

## Proof of Concept

```bash
# Infinite loop variant (u64 wrap):
curl -s -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<ledger-canister-id>"},
    "operations": [
      {"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<addr>"},"amount":{"value":"-100000000","currency":{"symbol":"ICP","decimals":8}}},
      {"operation_identifier":{"index":1},"type":"TRANSACTION","account":{"address":"<addr2>"},"amount":{"value":"100000000","currency":{"symbol":"ICP","decimals":8}}}
    ],
    "public_keys": [{"hex_bytes":"<valid-ed25519-pubkey>","curve_type":"edwards25519"}],
    "metadata": {"ingress_start": 0, "ingress_end": 18446744073709551615}
  }'
```

Expected: server hangs indefinitely; subsequent requests stall or time out. For the OOM variant, substitute `ingress_end` with `current_time_ns + 315360000000000000` (10 years). A unit test can confirm the wrap-around by constructing a `Time` near `u64::MAX`, calling `add_assign` with `interval`, and asserting the result is less than the original value.
