### Title
Unprotected `new` Initialization Function Allows Front-Running to Seize Full Contract Ownership - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The `new` function that initializes the Aurora Engine contract state carries no caller access control. Any unprivileged NEAR account can race the legitimate deployer to call it first, install themselves as `owner_id`, and subsequently exercise every owner-gated privilege — including deploying arbitrary replacement code and adding a full-access key — resulting in permanent theft of all user funds held by the engine.

### Finding Description
`new` is the sole initialization entry point for the Aurora Engine. It writes the `EngineState` (including `owner_id`, `chain_id`, `upgrade_delay_blocks`, etc.) to persistent storage. Its only guard is a check that state does not already exist:

```rust
// engine/src/contract_methods/admin.rs  lines 56-88
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
```

It is exported as an unrestricted public NEAR function:

```rust
// engine/src/lib.rs  lines 76-83
/// Sets the configuration for the Engine.
/// Should be called on deployment.
#[unsafe(no_mangle)]
pub extern "C" fn new() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::admin::new(io, &env)...sdk_unwrap();
}
```

The comment "Should be called on deployment" confirms that deployment and initialization are expected to be two separate transactions. Between the moment the WASM binary is deployed to the `aurora` account and the moment the deployer's `new` transaction is confirmed, the contract is live but uninitialized. Any observer of the NEAR mempool or block stream can submit their own `new` call with attacker-controlled `NewCallArgs` (including an attacker-controlled `owner_id`) and have it land first. Once that succeeds, the deployer's `new` reverts with `ERR_ALREADY_INITIALIZED` and the attacker is permanently recorded as owner.

The contrast with the XCC router is instructive: its `initialize` function explicitly documents that "the Create, Deploy and Initialize actions are done in a single NEAR batch," and the code enforces that only the original first caller can re-initialize. No equivalent protection exists for the main engine's `new`.

### Impact Explanation
With ownership of the Aurora Engine the attacker can immediately:

1. Call `upgrade` (owner-only) to deploy arbitrary replacement bytecode, draining all ETH balances and ERC-20 mirror balances held by the engine.
2. Call `attach_full_access_key` (owner-only) to add their own full-access key to the `aurora` NEAR account, giving them permanent, irrevocable control of the account independent of the contract code.
3. Call `set_owner` to lock out the legitimate team from ever reclaiming ownership through the contract's own governance path.

Every user's ETH, wNEAR, and bridged ERC-20 tokens stored in the engine are at risk of direct theft. This satisfies the **Critical — direct theft of user funds** impact class.

### Likelihood Explanation
The window exists whenever deployment and initialization are not atomic. NEAR batch transactions make atomicity possible but the contract code does not enforce it. Any deployment script, CI pipeline, or manual procedure that issues `DeployContract` and `new` as separate transactions (even in the same block) creates the exploitable gap. A sophisticated attacker monitoring the NEAR RPC for newly deployed contracts with no state can detect and exploit this window with a single transaction.

### Recommendation
Enforce atomicity at the code level: check inside `new` that `env.predecessor_account_id() == env.current_account_id()` (i.e., the call originates from the account itself, which is only possible in a batch action). Alternatively, accept an explicit `expected_owner` argument and verify `env.predecessor_account_id() == expected_owner` before writing state, so a front-runner cannot substitute their own account.

### Proof of Concept
1. Attacker monitors NEAR for a `DeployContract` action targeting the `aurora` account.
2. Attacker immediately submits a `new` call to `aurora` with `NewCallArgs { owner_id: attacker.near, ... }`.
3. Attacker's transaction is included before the deployer's `new` call; state is written with `owner_id = attacker.near`.
4. Deployer's `new` reverts: `ERR_ALREADY_INITIALIZED`.
5. Attacker calls `attach_full_access_key` (passes `require_owner_only` because `state.owner_id == attacker.near`) to add their key to the `aurora` account.
6. Attacker calls `upgrade` with malicious bytecode that transfers all ETH balances to the attacker's address.
7. All user funds are stolen; the legitimate team has no on-chain path to recover ownership.

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
