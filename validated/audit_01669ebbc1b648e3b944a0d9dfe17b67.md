Audit Report

## Title
Stale ICP/XDR Exchange Rate Used Without Freshness Check in Cycles Minting â€” (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles` function in the Cycles Minting Canister reads `state.icp_xdr_conversion_rate` and uses it directly without checking the age of the stored rate. If the Exchange Rate Canister (XRC) becomes unavailable, the CMC continues accepting `notify_top_up`, `notify_mint_cycles`, and `notify_create_canister` calls using an arbitrarily stale rate. An unprivileged attacker who monitors the public `get_icp_xdr_conversion_rate` endpoint can detect rate staleness and mint cycles at an inflated rate if ICP's market price has dropped during the XRC outage.

## Finding Description
`tokens_to_cycles` (rs/nns/cmc/src/main.rs, L1899â€“1923) extracts only `rate.xdr_permyriad_per_icp` from the stored rate and never reads `rate.timestamp_seconds`. The sole guard is that the `Option` is `Some(...)`:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...)
        }
    })
}
```

The rate is refreshed via `update_exchange_rate` (rs/nns/cmc/src/exchange_rate_canister.rs, L236â€“280) on a heartbeat. On XRC failure (`FailedToRetrieveRate`, `InvalidRate`), `schedule_next_attempt` reschedules the next attempt at the next minute (L153â€“161) but does **not** clear or invalidate the stored rate. The stored rate therefore persists indefinitely.

`validate_exchange_rate` (rs/nervous_system/clients/src/exchange_rate_canister_client.rs, L110â€“129) checks only the number of data sources, not the age of the rate. `do_set_icp_xdr_conversion_rate` (rs/nns/cmc/src/main.rs, L1022â€“1030) only enforces monotonically increasing timestamps on updates; it imposes no maximum age at minting time. No other staleness guard exists anywhere in the minting path (`process_top_up` L1985â€“2012, `process_mint_cycles` L1958â€“1983, `process_create_canister` L1925â€“1956 all call `tokens_to_cycles` directly).

Exploit flow:
1. XRC becomes unavailable (canister freeze, subnet degradation, or sustained `StablecoinRateTooFewRates` errors). The CMC heartbeat retries every minute but cannot update the rate.
2. Attacker polls the public `get_icp_xdr_conversion_rate` query endpoint and observes that `timestamp_seconds` has stopped advancing â€” confirming the rate is stale.
3. ICP's market price drops materially (e.g., from 10 XDR to 5 XDR) during the outage.
4. Attacker sends ICP to the CMC subaccount and calls `notify_top_up` or `notify_mint_cycles`.
5. `tokens_to_cycles` reads the stale 10 XDR/ICP rate and mints 2Ã— the cycles justified by the current 5 XDR/ICP market rate.
6. Attacker repeats up to the hourly rate limiter, extracting cycles at below-market cost.

## Impact Explanation
This constitutes illegal minting of cycles â€” a protocol resource backed by burned ICP â€” at a rate that exceeds what the burned ICP justifies at current market prices. Cycles represent real computational resources on the IC. The ICP burned is insufficient to cover the cycles issued at the true current rate, constituting value extraction from the protocol. This matches the allowed High impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm." The per-hour rate limiter bounds but does not eliminate the damage; at 150P cycles/hour and a 2Ã— stale-rate multiplier, the excess cycles issued per hour represent a concrete, quantifiable protocol loss. The attacker requires no special privileges â€” any principal can call `notify_top_up`.

## Likelihood Explanation
Two concurrent conditions are required: (1) XRC unavailability for a sustained period, and (2) a material ICP price drop during that window. XRC unavailability is realistic â€” the CMC's own error-handling code already anticipates `StablecoinRateTooFewRates` and similar failures, and the retry-every-minute logic confirms this is an expected operational condition. ICP price volatility is a normal market condition. Critically, the attacker does not need to cause the XRC outage; they only need to monitor the public `get_icp_xdr_conversion_rate` endpoint (zero cost, zero privilege) to detect when the rate has gone stale and time their minting accordingly. The attack is repeatable up to the hourly rate limit for as long as the XRC remains unavailable.

## Recommendation
Add a maximum-age check inside `tokens_to_cycles` that rejects minting if `rate.timestamp_seconds` is older than a configurable threshold (e.g., 30 minutes):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) => {
                let now = ic_cdk::api::time() / 1_000_000_000;
                let age = now.saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: format!("ICP/XDR rate is stale ({age}s); minting suspended"),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(/* existing error */)
        }
    })
}
```

`MAX_RATE_AGE_SECONDS` should be set to a value meaningfully larger than the 5-minute refresh interval (e.g., 30 minutes) to tolerate transient XRC hiccups while still bounding the staleness window.

## Proof of Concept
A deterministic PocketIC integration test can reproduce this without touching mainnet:

1. Initialize a CMC instance with a stored `icp_xdr_conversion_rate` of 100_000 XDR permyriad (10 XDR/ICP) and a `timestamp_seconds` set to `T`.
2. Advance the PocketIC clock to `T + 7200` (2 hours later) without calling `update_exchange_rate` (simulating XRC unavailability).
3. Call `notify_top_up` or `notify_mint_cycles` with 1 ICP.
4. Assert that `tokens_to_cycles` succeeds and returns cycles computed at the stale 10 XDR/ICP rate.
5. Confirm that no freshness error is returned despite the 2-hour-old rate.
6. Optionally: set the "true" rate to 50_000 permyriad (5 XDR/ICP) and show the minted cycles are 2Ã— what the current rate would produce.
