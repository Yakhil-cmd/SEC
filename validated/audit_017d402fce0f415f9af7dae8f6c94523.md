Audit Report

## Title
Unprivileged Caller Can Drain ckBTC Minter Cycles via `update_balance` with Arbitrary Owner Principals - (File: rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs)

## Summary
The `update_balance` endpoint accepts an arbitrary `owner` principal from any non-anonymous caller and derives a Bitcoin address for that account, unconditionally triggering one or two `get_utxos` Bitcoin API calls billed to the minter's cycle balance. Because the per-account concurrency guard is keyed on the supplied `{owner, subaccount}` pair rather than the actual ingress sender, an attacker can saturate the global 100-slot guard with 100 distinct novel principals per batch, forcing up to 200 Bitcoin API calls per batch at zero cost to themselves. Sustained repetition progressively depletes the minter's cycle balance, eventually freezing or uninstalling it and halting all ckBTC deposit and withdrawal operations.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, the `update_balance` function resolves the target account as `Account { owner: args.owner.unwrap_or(caller), subaccount: args.subaccount }` with no requirement that `args.owner == caller`. [1](#0-0) 

The guard `balance_update_guard(caller_account)` is keyed on this resolved account, not on the actual ingress sender. [2](#0-1) 

`MAX_CONCURRENT = 100` means 100 distinct `{owner, subaccount}` pairs can be in-flight simultaneously; a single attacker with 100 distinct principals fills all 100 slots. [3](#0-2) 

After the guard is acquired, a first `get_utxos` call is made unconditionally. [4](#0-3) 

When the address has no UTXOs (the universal case for freshly-generated principals), `satoshis_to_mint == 0` and a second `get_utxos` call is made with `min_confirmations = 0` to populate the pending-UTXO error response. [5](#0-4) 

The `get_utxos_cache` is keyed on `GetUtxosRequest { address, network, filter }`. Each novel principal derives a unique Bitcoin address, so every call is a guaranteed cache miss. [6](#0-5) 

The only caller restriction is `check_anonymous_caller()`, which only blocks `Principal::anonymous()`. [7](#0-6) 

**Exploit flow:**
1. Attacker generates 100 fresh IC key pairs (principals `p_1…p_100`).
2. Attacker submits 100 concurrent ingress calls to `update_balance` with `owner = Some(p_i)`.
3. All 100 pass the guard (each is a distinct account), each triggers 2 `get_utxos` calls → 200 Bitcoin API calls billed to the minter.
4. After all calls complete, the guard set empties. Attacker immediately submits the next batch with 100 new principals.
5. Repeat indefinitely.

## Impact Explanation
Each batch of 100 concurrent calls forces approximately 200 Bitcoin-canister `get_utxos` round-trips billed entirely to the minter's cycle balance. The attacker bears only the standard IC ingress fee, which is orders of magnitude smaller than the Bitcoin API cost. Sustained batching progressively depletes the minter's cycle balance. When the balance falls below the freeze threshold the minter stops executing; if it reaches zero the subnet uninstalls it, halting all ckBTC deposit and withdrawal operations for every user. This constitutes an **application/platform-level DoS of a critical Chain Fusion financial component**, matching the High ($2,000–$10,000) allowed impact: "Application/platform-level DoS… not based on raw volumetric DDoS."

## Likelihood Explanation
The attack requires only a valid (non-anonymous) IC principal and the ability to submit ingress messages — capabilities available to any IC user at negligible cost. No BTC deposit, privileged key, or governance access is needed. Batch submission can be fully automated. The `MAX_CONCURRENT = 100` cap limits per-batch parallelism but imposes no throughput limit over time; after each batch completes the guard set is empty and the next batch can begin immediately.

## Recommendation
1. **Restrict the `owner` field**: enforce `args.owner.unwrap_or(caller) == caller` so a caller can only trigger balance checks for their own account, eliminating the ability to enumerate arbitrary addresses.
2. **Per-caller rate limiting**: if cross-account queries must remain supported, track call counts per ingress sender and reject requests exceeding a per-second or per-minute quota.
3. **Charge the caller**: attach a cycle payment requirement to `update_balance` proportional to the expected Bitcoin API cost, refunding any surplus, so the minter is not the net payer for user-initiated queries.

## Proof of Concept
```bash
# Attacker generates 100 distinct principals p_1…p_100 (fresh key pairs)
# and submits 100 concurrent ingress calls:

for i in $(seq 1 100); do
    dfx canister call ckbtc_minter update_balance \
        "(record { owner = opt principal \"<p_$i>\"; subaccount = null })" &
done
wait
# Each of the 100 calls triggers 2 get_utxos calls from the minter's cycles (200 total).
# Repeat indefinitely with fresh principals — no BTC deposit required.
```

A deterministic integration test using PocketIC can verify this by:
1. Initializing the minter with a known cycle balance.
2. Submitting 100 concurrent `update_balance` calls with distinct novel owner principals from a single caller identity.
3. Asserting that the minter's cycle balance decreases by approximately `200 × get_utxos_cost` per batch.
4. Asserting that repeating the pattern with 100 new principals in the next batch again decreases the balance by the same amount, confirming no per-caller throttle exists.

        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
