### Title
No Incentive for Third Parties to Execute Scheduled XCC Promises, Enabling Permanent Freezing of NEAR in Router Contracts — (File: `etc/xcc-router/src/lib.rs`)

### Summary
When an EVM contract uses `CrossContractCallArgs::Delayed`, NEAR is transferred to the user's XCC router sub-account and a promise is stored there. Executing that promise requires a separate call to `execute_scheduled`. The function is open to anyone but offers no reward to the caller. For autonomous EVM smart contracts that have no associated NEAR account and whose wNEAR balance was fully consumed by the scheduling step, there is no reachable path to trigger `execute_scheduled`, and the NEAR is permanently frozen inside the router with no recovery mechanism.

### Finding Description

The `CrossContractCallArgs::Delayed` variant of the XCC precompile is designed to let callers schedule a NEAR cross-contract call for later execution when the current transaction has insufficient gas remaining. [1](#0-0) 

When the precompile processes a `Delayed` call, it:

1. Deducts wNEAR from the caller's EVM balance equal to `call.total_near()`.
2. Calls `withdraw_wnear_to_router` to convert that wNEAR into real NEAR and deposit it into the router sub-account (`{evm_address}.aurora`).
3. Calls `schedule` on the router (restricted to the Aurora engine parent) to store the `PromiseArgs`. [2](#0-1) 

The stored promise, including its `attached_balance`, is later executed by `execute_scheduled`: [3](#0-2) 

The comment explicitly acknowledges that **anyone** may call this function. However, the function provides **no reward** to the caller — the caller pays NEAR gas and receives nothing. The router contract has no `withdraw`, `recover`, or `cancel` method. The only other way to move funds out of the router is `execute` (restricted to the Aurora engine parent) or `send_refund` (restricted to the parent and only triggered during the storage-staking refund flow). [4](#0-3) [5](#0-4) 

### Impact Explanation

**Permanent freezing of funds (Critical).**

Consider an autonomous EVM smart contract (no human-controlled NEAR account) that calls the XCC precompile with `CrossContractCallArgs::Delayed` and attaches NEAR (e.g., for `ft_transfer` with `attached_balance > 0`). After scheduling:

- The contract's wNEAR balance is reduced by `required_near`.
- The NEAR now sits in the router's account balance.
- To execute the promise, `execute_scheduled` must be called on the NEAR layer.
- The EVM contract cannot call NEAR functions directly; it must use XCC again, which requires wNEAR.
- If the contract's wNEAR was fully consumed by the scheduling step, it cannot afford another XCC call to trigger `execute_scheduled`.
- No third party has any economic incentive to call `execute_scheduled` on the contract's behalf.
- The router has no `withdraw` or `cancel` entrypoint.

The NEAR is permanently locked in the router sub-account with no on-chain recovery path available to an unprivileged caller.

### Likelihood Explanation

**Medium.** The XCC precompile is explicitly designed to be called from EVM smart contracts. Any EVM contract that:
- uses `CrossContractCallArgs::Delayed` with a non-zero `attached_near`, **and**
- has no residual wNEAR balance after the scheduling step, **and**
- has no associated human-controlled NEAR account that can call `execute_scheduled` out-of-band

will have its NEAR permanently frozen. DeFi protocols, DAOs, or any contract that programmatically manages cross-chain calls are realistic candidates. The condition is reachable without any privileged access.

### Recommendation

Add an incentive mechanism to `execute_scheduled` so that third parties are compensated for triggering execution. One approach: allow the scheduler to attach a small "tip" (stored alongside the promise) that is paid to whoever calls `execute_scheduled`. Alternatively, expose a `cancel_scheduled` method restricted to the parent engine so that the Aurora engine can return the NEAR to the originating EVM address when the promise is abandoned.

### Proof of Concept

1. Deploy an autonomous EVM contract `Vault` on Aurora that holds wNEAR.
2. `Vault` calls the XCC precompile with `CrossContractCallArgs::Delayed(PromiseArgs::Create(call))` where `call.attached_balance = all_wnear`.
3. The precompile deducts all wNEAR from `Vault`'s EVM balance and deposits it as NEAR into `{vault_address}.aurora` (the router).
4. The promise is stored at nonce `0` in the router's `scheduled_promises` map.
5. `Vault` now has zero wNEAR; it cannot afford another XCC call to invoke `execute_scheduled`.
6. No external NEAR account is associated with `Vault`; no third party has incentive to call `execute_scheduled` and pay gas for free.
7. Observe: the NEAR remains in `{vault_address}.aurora` indefinitely. The router exposes no `withdraw` or `cancel` method. The promise is never executed. Funds are permanently frozen. [6](#0-5) [7](#0-6) [8](#0-7)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }
