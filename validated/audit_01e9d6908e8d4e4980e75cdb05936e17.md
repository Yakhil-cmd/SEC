### Title
Missing Error-Refund Callback in `ExitToNear` Precompile Causes Permanent Fund Freeze When NEAR-Side Transfer Fails - (File: engine-precompiles/src/native.rs)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH to the precompile address) on the EVM side before scheduling an asynchronous NEAR promise. When the `error_refund` feature is not compiled in, no callback is attached to the outgoing promise for ETH and non-wNEAR ERC-20 exits. If the NEAR-side `ft_transfer` or `ft_transfer_call` promise fails, the EVM-side assets are permanently unrecoverable — an exact analog of the "incomplete fix" pattern in the external report, where only the explicitly named issue was addressed while the broader failure-path accounting was left open.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` method constructs a `callback_args` struct and decides whether to attach a callback promise:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};

let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)          // ← no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { … })
};
``` [1](#0-0) 

For the two most common exit paths — ETH (base token) exit and non-wNEAR ERC-20 exit — `transfer_near_args` is `None` and, without `error_refund`, `refund` is also `None`. Therefore `callback_args` equals `default()` and **no callback is ever attached**.

The EVM-side state is already committed before the NEAR promise executes:

- **ERC-20 exit**: `EvmErc20.withdrawToNear` calls `_burn(msg.sender, amount)` first, then calls the precompile. [2](#0-1) 

- **ETH exit**: The EVM transfers `apparent_value` ETH from the caller to the precompile address as part of the call, before the precompile body runs.

If the downstream NEAR `ft_transfer` (or `ft_transfer_call`) fails — for example because the recipient NEAR account has no storage deposit on the eth-connector — the EVM-side burn/transfer is already final and there is no callback path to invoke `refund_on_error`.

The `refund_on_error` function that *would* recover the funds exists and is correct:

```rust
// ETH exit; transfer ETH back from precompile address
let exit_address = exit_to_near::ADDRESS;
engine.call(&exit_address, &refund_address, amount, …)
``` [3](#0-2) 

But it is only reachable through `exit_to_near_precompile_callback`, which is only scheduled when `callback_args != default()`. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the NEAR-side `ft_transfer` fails without `error_refund` enabled:

- **ERC-20 tokens** are permanently burned on the EVM side with no corresponding NEP-141 transfer and no re-mint.
- **ETH** is permanently locked at the `ExitToNear` precompile address (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with no mechanism to recover it, because the precompile address has no withdraw function and `refund_on_error` is never called.

The total supply of the bridged asset on the NEAR side remains unchanged while the EVM-side supply is reduced, creating a permanent accounting discrepancy and irrecoverable user loss.

---

### Likelihood Explanation

**Medium.** The NEAR NEP-141 `ft_transfer` call will revert if the recipient account has not registered a storage deposit with the eth-connector. This is a realistic condition for any NEAR account that has never interacted with the eth-connector. A user who exits ETH or an ERC-20 token to such an account will trigger the failure path. No privileged access is required; any EVM user can reach this path by calling `withdrawToNear` on an ERC-20 contract or sending ETH to the `ExitToNear` precompile address with a recipient that lacks a storage deposit.

---

### Recommendation

1. **Enable `error_refund` unconditionally** in the production build, or promote the refund callback from a feature-gated path to the default code path for all exit types.
2. For the ETH base-token exit, always attach a callback that invokes `exit_to_near_precompile_callback` so that a failed `ft_transfer` triggers `refund_on_error` and returns ETH from the precompile address to the original sender.
3. For non-wNEAR ERC-20 exits, always attach a callback that re-mints the burned tokens on failure.

---

### Proof of Concept

1. Alice holds 1 ETH in her Aurora EVM address.
2. Alice calls the `ExitToNear` precompile (flag `0x00`) with `apparent_value = 1 ETH` and `recipient = "new_account.near"` — an account that has never registered a storage deposit on the eth-connector.
3. The EVM transfers 1 ETH from Alice's address to `exit_to_near::ADDRESS` (`0xe9217bc7…`). Alice's EVM balance is now 0.
4. The precompile constructs `callback_args = { refund: None, transfer_near: None }` which equals `default()`, so `PromiseArgs::Create(transfer_promise)` is used — **no callback**. [5](#0-4) 
5. The NEAR promise calls `ft_transfer` on the eth-connector for `new_account.near`. The eth-connector reverts because `new_account.near` has no storage deposit.
6. Because no callback was attached, `exit_to_near_precompile_callback` is never called, `refund_on_error` is never invoked, and the 1 ETH remains permanently locked at `0xe9217bc7…` with no recovery path.

```rust
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
