Audit Report

## Title
Unbounded ingress-expiry accumulation loop in ICP Rosetta `/construction/payloads` enables OOM/infinite-loop DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The `construction_payloads` handler in ICP Rosetta accepts attacker-controlled `ingress_start` and `ingress_end` timestamps with no upper-bound validation. Setting `ingress_start=0` and `ingress_end=u64::MAX` causes the ingress-expiry accumulation loop to allocate ~1.2 TB of heap before an OOM kill, or — if the process somehow survives to wrap-around — to spin infinitely due to a truncating `as u64` cast in `Time::AddAssign`. Either outcome renders the Rosetta node permanently unavailable until restarted, and the attack is immediately repeatable.

## Finding Description
In `construction_payloads` the interval is computed as:

```rust
let interval =
    ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT - Duration::from_secs(120);
```

With `MAX_INGRESS_TTL = 300s` and `PERMITTED_DRIFT = 60s` this yields `interval = 120s = 120_000_000_000 ns`. [1](#0-0) [2](#0-1) 

The loop that follows has no guard on the range `ingress_end - ingress_start`:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;
}
``` [3](#0-2) 

With `ingress_start = Time(0)` and `ingress_end = Time(u64::MAX)`, the loop requires approximately `u64::MAX / 120_000_000_000 ≈ 153.7 billion` iterations. Each iteration pushes one `u64` (8 bytes) onto `ingress_expiries`, requiring ~1.23 TB of heap — an OOM kill long before completion.

The overflow path that produces an infinite loop is rooted in `Time::AddAssign`:

```rust
impl std::ops::AddAssign<Duration> for Time {
    fn add_assign(&mut self, other: Duration) {
        *self = Time::from_duration(Duration::from_nanos(self.0) + other)
    }
}
``` [4](#0-3) 

`from_duration` performs a truncating `u128 → u64` cast:

```rust
fn from_duration(t: Duration) -> Self {
    Time(t.as_nanos() as u64)  // truncating, not saturating
}
``` [5](#0-4) 

When `now` approaches `u64::MAX` and `interval` is added, the `u128` intermediate exceeds `u64::MAX` and the cast wraps `now` back to a small value still less than `ingress_end`, making the loop infinite. The safe `checked_add` method exists on `Time` but is not used here. [6](#0-5) 

The ICRC1 Rosetta counterpart explicitly prevents this with two guards that are entirely absent from the ICP Rosetta path:

```rust
if ingress_start >= ingress_end { return Err(...); }
if ingress_end < now + ingress_interval { return Err(...); }
``` [7](#0-6) 

## Impact Explanation
A single unauthenticated HTTP POST causes the ICP Rosetta process to either OOM-crash (before wrap-around) or spin in an infinite loop (after wrap-around). This is a concrete application-level DoS against an in-scope financial integration component (ICP Rosetta). It maps to the **High** bounty impact: *"Application/platform-level DoS, crash … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … Rosetta … security impact with concrete user or protocol harm."*

## Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication. The malicious payload is a trivially crafted JSON body with two integer fields. The ICRC1 Rosetta fix demonstrates the developers are aware of the need for this guard — it was simply not applied to the ICP Rosetta path. The attack is repeatable immediately after any restart, making sustained unavailability trivial to maintain.

## Recommendation
1. Add the same guards present in the ICRC1 counterpart before the loop: reject if `ingress_start >= ingress_end`, and enforce a maximum window (e.g., cap `ingress_end - ingress_start` to a reasonable bound such as 24 hours).
2. Replace the truncating `as u64` cast in `Time::from_duration` with a saturating or checked conversion (the existing `Time::checked_add` already implements the correct pattern) to eliminate the infinite-loop path.
3. Consider adding a hard cap on `ingress_expiries.len()` before the loop as a defense-in-depth measure.

## Proof of Concept
```
POST /construction/payloads HTTP/1.1
Content-Type: application/json

{
  "network_identifier": { ... },
  "operations": [ <any valid transfer op> ],
  "public_keys": [ <any valid key> ],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```
The Rosetta process will exhaust available memory (OOM kill) within seconds of receiving this request. A deterministic unit test asserting the function returns an error or completes in bounded time with `ingress_start=0, ingress_end=u64::MAX` will reproduce the issue without requiring a live node.
