### Title
`ic0.mint_cycles128` System API Import Missing Instruction Charge - (`rs/embedders/src/wasmtime_embedder/linker.rs`)

### Summary

The `ic0.mint_cycles128` system API binding in the IC's wasmtime embedder linker does not call `charge_for_cpu` (or `charge_for_cpu_and_mem`) before executing, unlike every other comparable system API import. This means any canister can invoke `mint_cycles128` in a tight loop without paying the fixed per-call instruction overhead, allowing slightly more unmetered work than the instruction budget should permit.

### Finding Description

In `rs/embedders/src/wasmtime_embedder/linker.rs`, the `mint_cycles128` binding is registered as:

```rust
linker
    .func_wrap("ic0", "mint_cycles128", {
        move |mut caller: Caller<'_, StoreData>, amount_high: u64, amount_low: u64, dst: I| {
            with_memory_and_system_api(&mut caller, |s, memory| {
                let dst: usize = dst.try_into().expect("Failed to convert I to usize");
                s.ic0_mint_cycles128(Cycles::from_parts(amount_high, amount_low), dst, memory)
            })
            .map_err(|e| wasmtime::Error::msg(format!("ic0_mint_cycles128 failed: {e}")))
        }
    })
    .unwrap();
``` [1](#0-0) 

There is no `charge_for_cpu` or `charge_for_cpu_and_mem` call before `with_memory_and_system_api`. Compare this directly with the immediately adjacent `cycles_burn128` binding, which correctly charges before executing:

```rust
linker
    .func_wrap("ic0", "cycles_burn128", {
        move |mut caller: Caller<'_, StoreData>, amount_high: u64, amount_low: u64, dst: I| {
            charge_for_cpu(&mut caller, overhead::CYCLES_BURN128)?;  // <-- present
            ...
        }
    })
``` [2](#0-1) 

The `system_api_complexity.rs` overhead table defines `CYCLES_BURN128` (500 instructions) but has **no `MINT_CYCLES128` entry at all**, confirming the charge was never defined or applied: [3](#0-2) 

All other system API calls that touch memory or perform any work call either `charge_for_cpu` or `charge_for_cpu_and_mem` before executing. Examples: `cost_call`, `cost_create_canister`, `cost_http_request`, `data_certificate_copy`, `certified_data_set`, `is_controller`, `in_replicated_execution`, `canister_liquid_cycle_balance128`, etc. [4](#0-3) [5](#0-4) 

The underlying `ic0_mint_cycles128` implementation writes 16 bytes to the canister heap via `copy_cycles_to_heap` (on the success path for CMC), which should also be charged via `charge_for_cpu_and_mem`: [6](#0-5) 

### Impact Explanation

**For non-CMC canisters (any canister on any subnet):** The call fails immediately with a `CanisterContractViolation` error for non-CMC canisters. However, the system API dispatch itself is unmetered — no fixed overhead (the 500-instruction equivalent charged by all other system API calls) is deducted. A malicious canister can call `mint_cycles128` in a tight loop, and each invocation avoids the per-call overhead charge. The WASM loop and call instructions are still charged by the instrumentation, but the system API overhead is not. This allows a canister to make more invocations of this system API within its instruction budget than the accounting model intends.

**For the CMC canister (system subnet):** The CMC is the only canister that can successfully execute `mint_cycles128`. On the success path, 16 bytes are written to the heap via `copy_cycles_to_heap`, but neither the fixed overhead nor the per-byte memory charge is applied. This is a minor accounting inconsistency for a trusted canister.

The primary concern is the non-CMC path: any canister developer can deploy a canister that calls `mint_cycles128` in a loop, bypassing the per-call instruction overhead. While the work done per failed call is small (a match statement and error return), the missing charge is a systematic accounting gap reachable by any unprivileged canister.

### Likelihood Explanation

Any canister on any subnet can import and call `ic0.mint_cycles128` — it is a valid system API import accepted by the validator. The missing charge is unconditional and requires no special privileges. The attack path is straightforward: deploy a canister, call `mint_cycles128` in a loop. Likelihood is high for the missing-charge class of bug, though the practical resource drain per call is small.

### Recommendation

Add a `charge_for_cpu_and_mem` call (to account for both the fixed overhead and the 16-byte heap write on the success path) at the start of the `mint_cycles128` linker binding, consistent with all other system API imports:

```rust
linker
    .func_wrap("ic0", "mint_cycles128", {
        move |mut caller: Caller<'_, StoreData>, amount_high: u64, amount_low: u64, dst: I| {
            charge_for_cpu_and_mem(&mut caller, overhead::MINT_CYCLES128, 16)?;
            with_memory_and_system_api(&mut caller, |s, memory| {
                let dst: usize = dst.try_into().expect("Failed to convert I to usize");
                s.ic0_mint_cycles128(Cycles::from_parts(amount_high, amount_low), dst, memory)
            })
            .map_err(|e| wasmtime::Error::msg(format!("ic0_mint_cycles128 failed: {e}")))
        }
    })
    .unwrap();
```

And add the corresponding constant to `system_api_complexity.rs`:

```rust
pub const MINT_CYCLES128: NumInstructions = NumInstructions::new(500);
```

### Proof of Concept

The root cause is directly visible by comparing the two adjacent bindings in `linker.rs`:

- `mint_cycles128` (lines 1205–1215): no `charge_for_cpu` call before `with_memory_and_system_api`
- `cycles_burn128` (lines 1217–1228): `charge_for_cpu(&mut caller, overhead::CYCLES_BURN128)?;` present [7](#0-6) 

A canister that calls `mint_cycles128` in a loop will consume fewer instructions per iteration than the accounting model intends, because the 500-instruction system API overhead is never deducted. The missing overhead constant is confirmed by the absence of any `MINT_CYCLES128` entry in the overhead table: [3](#0-2)

    fn ic0_mint_cycles128(
        &mut self,
        amount: Cycles,
        dst: usize,
        heap: &mut [u8],
    ) -> HypervisorResult<()> {
        let result = match self.api_type {
            ApiType::Start { .. }
            | ApiType::Init { .. }
            | ApiType::PreUpgrade { .. }
            | ApiType::Cleanup { .. }
            | ApiType::CompositeCleanup { .. }
            | ApiType::ReplicatedQuery { .. }
            | ApiType::CompositeQuery { .. }
            | ApiType::NonReplicatedQuery { .. }
            | ApiType::InspectMessage { .. }
            | ApiType::CompositeReplyCallback { .. }
            | ApiType::CompositeRejectCallback { .. } => Err(self.error_for("ic0_mint_cycles128")),
            ApiType::Update { .. }
            | ApiType::SystemTask { .. }
            | ApiType::ReplyCallback { .. }
            | ApiType::RejectCallback { .. } => {
                let actually_minted = self.sandbox_safe_system_state.mint_cycles(amount)?;
                copy_cycles_to_heap(actually_minted, dst, heap, "ic0_mint_cycles_128")?;
                Ok(())
            }
        };
        trace_syscall!(self, MintCycles128, result, amount);
        result
    }
