Audit Report

## Title
Missing Callback-ID Validation in `push_response` `SendTransactionResponse` Arm Allows Byzantine Block Proposer to Inject Spurious Subnet Responses — (`rs/replicated_state/src/bitcoin.rs`)

## Summary

The `SendTransactionResponse` arm of `push_response` unconditionally enqueues a `ConsensusResponse` without verifying that the supplied `callback_id` corresponds to any pending `bitcoin_send_transaction_internal` context. `validate_self_validating_payload_impl` performs only a size check on non-empty payloads, so honest nodes will notarize a block containing a forged response. A single malicious block proposer below the consensus fault threshold can therefore inject a fabricated `SendTransactionResponse` carrying a real, observable `callback_id`, causing the execution environment to remove the matching context and deliver a success reply to the Bitcoin canister — making it believe a Bitcoin transaction was broadcast when it was not.

## Finding Description

**Gap 1 — Asymmetric validation in `push_response`**

The `GetSuccessorsResponse` arm validates the `callback_id` against `bitcoin_get_successors_contexts` and returns `StateError::BitcoinNonMatchingResponse` if absent: [1](#0-0) 

The `SendTransactionResponse` arm performs no such check. Any `callback_id` — including one with no matching context — is unconditionally wrapped in a `ConsensusResponse` and pushed to `state.consensus_queue`: [2](#0-1) 

The same omission applies to `GetSuccessorsReject` and `SendTransactionReject`: [3](#0-2) 

**Gap 2 — No-op payload validation**

`validate_self_validating_payload_impl` returns `Ok(size)` for any non-empty payload without inspecting the `callback_id` values inside it: [4](#0-3) 

Honest nodes call this function before notarizing a block proposal. Because it is a no-op for non-empty payloads, they will accept a block containing a forged `SendTransactionResponse` with an arbitrary `callback_id`.

**Gap 3 — Execution environment delivers the forged response**

The scheduler drains the entire `consensus_queue` each round: [5](#0-4) 

Each entry is passed to `execute_subnet_message` as a `SubnetMessage::Response`. `retrieve_context` searches all subnet call context maps in order, including `bitcoin_send_transaction_internal_contexts`: [6](#0-5) 

If the `callback_id` matches a pending context, the context is **removed** and `push_subnet_output_response` delivers the forged payload to the original requester: [7](#0-6) 

If the `callback_id` matches nothing, the response is silently dropped (line 704). This is the only existing mitigation, and it is defeated whenever the attacker uses a real, observable `callback_id`.

**Complete exploit flow:**

1. Attacker (malicious block proposer) reads certified state and extracts a live `callback_id` N from `bitcoin_send_transaction_internal_contexts`.
2. Attacker crafts a `SelfValidatingPayload` containing `SendTransactionResponse { callback_id: N }`.
3. Honest nodes call `validate_self_validating_payload_impl` — it returns `Ok(size)` unconditionally; block is notarized and finalized.
4. During execution, `push_response` is called; the `SendTransactionResponse` arm pushes `ConsensusResponse(N, EmptyBlob)` to `consensus_queue` without any context lookup.
5. Scheduler delivers the response to `execute_subnet_message`.
6. `retrieve_context(N)` finds and removes the `BitcoinSendTransactionInternal` context.
7. `push_subnet_output_response` delivers `EmptyBlob` (the expected success payload for `send_transaction`) to the Bitcoin canister.
8. The Bitcoin canister and ckBTC minter advance their state machines as if the Bitcoin transaction was successfully broadcast.

## Impact Explanation

This maps to **High ($2,000–$10,000)**: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm." The ckBTC minter relies on the Bitcoin canister's `send_transaction` success signal to advance its state machine. A forged success response causes the minter to treat a pending transaction as confirmed even though the underlying Bitcoin transaction was never broadcast, enabling double-spend scenarios or permanent loss of track of a pending transaction (funds locked or minted without on-chain backing). Depending on the value of in-flight transactions at the time of the attack, the financial impact could reach the Critical threshold (>$1M), but the subnet-membership prerequisite constrains the realistic severity to High.

## Likelihood Explanation

- The attacker must hold a block proposer slot on the Bitcoin subnet, which requires NNS-approved subnet membership. This is a meaningful prerequisite.
- However, the IC threat model explicitly tolerates up to f < n/3 Byzantine nodes, and a single malicious block proposer is within that threshold.
- Only one malicious block proposer turn is required; no coordination with other nodes is needed.
- `callback_id` values are observable from certified state reads, which are publicly accessible.
- `validate_self_validating_payload_impl` is a confirmed no-op for non-empty payloads, so honest nodes will notarize the malicious block without any additional conditions.
- The attack is fully deterministic and repeatable.

## Recommendation

1. **`push_response` — add context lookup for `SendTransactionResponse`** (and both `Reject` variants) mirroring the `GetSuccessorsResponse` arm: look up the `callback_id` in `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if absent.

2. **`validate_self_validating_payload_impl` — validate callback IDs against state**: load the certified state at `validation_context.certified_height` (as already done in `get_self_validating_payload_impl` via `bitcoin_requests_iter`) and reject any response whose `callback_id` is not present in the corresponding context map.

## Proof of Concept

```rust
// State-machine test sketch
let mut state = ReplicatedState::new(SUBNET_ID, SubnetType::Application);
// No bitcoin_send_transaction_internal_contexts registered.

let result = push_response(
    &mut state,
    BitcoinAdapterResponse {
        response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
            SendTransactionResponse {},
        ),
        callback_id: u64::MAX,
    },
);

// Bug: returns Ok(()) instead of Err(BitcoinNonMatchingResponse)
assert!(result.is_ok());
// Bug: consensus_queue now contains a spurious response
assert_eq!(state.consensus_queue.len(), 1);
assert_eq!(state.consensus_queue[0].callback.get(), u64::MAX);
```

Contrast with `GetSuccessorsResponse` under identical conditions, which returns `Err(StateError::BitcoinNonMatchingResponse { callback_id: u64::MAX })` and leaves the queue empty, demonstrating the asymmetry directly in the existing test harness. [8](#0-7) [2](#0-1)
