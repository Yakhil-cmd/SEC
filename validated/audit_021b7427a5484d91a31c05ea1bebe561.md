### Title
Fee-on-Transfer NEP-141 Token Over-Minting Causes Bridge Insolvency - (File: engine/src/engine.rs)

### Summary
`receive_erc20_tokens` unconditionally mints ERC-20 tokens equal to `args.amount` — the amount declared by the NEP-141 `ft_on_transfer` callback — without verifying the actual balance change on the NEP-141 contract. For a fee-on-transfer NEP-141 token, Aurora receives fewer tokens than declared, but mints the full declared amount as ERC-20 tokens. This creates a permanent accounting deficit: more ERC-20 tokens exist than NEP-141 tokens held by Aurora, making the bridge insolvent for that token.

### Finding Description

When a NEP-141 token is bridged into Aurora, the flow is:

1. User calls `ft_transfer_call(receiver_id=aurora, amount=X, msg=<evm_address>)` on the NEP-141 contract.
2. The NEP-141 contract transfers tokens and calls `ft_on_transfer` on Aurora with `amount=X`.
3. Aurora's `ft_on_transfer` dispatches to `receive_erc20_tokens`.

The minted amount is always `args.amount` — the value declared in the callback — not the actual NEP-141 balance change on Aurora's account. For a fee-on-transfer NEP-141 token, the NEP-141 contract deducts a fee during transfer, so Aurora's actual NEP-141 balance increases by `X - fee`, but `ft_on_transfer` is still called with `amount = X`. Aurora mints `X` ERC-20 tokens while only holding `X - fee` NEP-141 tokens.

The `ft_on_transfer` handler in `connector.rs` passes `args.amount` directly without any balance verification: [3](#0-2) 

### Impact Explanation

Each deposit of a fee-on-transfer NEP-141 token inflates the ERC-20 supply by `fee` units beyond what Aurora holds. Over multiple deposits, the cumulative deficit grows. When users attempt to exit (burn ERC-20 tokens and receive NEP-141 tokens back via `withdrawToNear`), the last users cannot exit because Aurora's NEP-141 balance is insufficient to cover all outstanding ERC-20 tokens. This is **bridge insolvency** and **permanent freezing of funds** for the last holders.

**Impact: Critical — Insolvency / Permanent freezing of funds.**

### Likelihood Explanation

Any NEP-141 token that implements a transfer fee (a non-standard but valid extension of the NEP-141 interface) and is registered with Aurora via `deploy_erc20_token` triggers this bug. The attacker-controlled entry path is simply calling `ft_transfer_call` on such a token with Aurora as the receiver — a fully unprivileged, externally reachable action. No admin compromise is required. The likelihood is **Medium**: fee-on-transfer NEP-141 tokens are non-standard but possible, and the impact is severe once such a token is used.

### Recommendation

In `receive_erc20_tokens`, instead of trusting `args.amount`, Aurora should query its own NEP-141 balance before and after the transfer to determine the actual amount received, and mint only that delta. Alternatively, Aurora should explicitly reject (return the full `args.amount`) for any NEP-141 token whose actual balance change does not match `args.amount`. This mirrors the recommendation in the original report: use the actual received amount rather than the declared amount.

### Proof of Concept

1. Deploy a NEP-141 token that deducts a 1% fee on every transfer.
2. Register it with Aurora via `deploy_erc20_token`.
3. Call `ft_transfer_call(receiver_id=aurora, amount=1000, msg=<evm_address>)`.
4. The NEP-141 contract transfers 990 tokens to Aurora (deducting 10 as fee) and calls `ft_on_transfer` with `amount=1000`.
5. Aurora mints 1000 ERC-20 tokens for `<evm_address>`.
6. Aurora holds 990 NEP-141 tokens but has issued 1000 ERC-20 tokens — a deficit of 10.
7. Repeat 100 times: Aurora holds 99,000 NEP-141 tokens but has issued 100,000 ERC-20 tokens.
8. The last ~1,000 ERC-20 token holders cannot exit; their funds are permanently frozen. [4](#0-3) [5](#0-4)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
