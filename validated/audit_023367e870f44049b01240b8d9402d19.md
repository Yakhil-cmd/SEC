### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent ERC-20 Token Burn Without Underlying Asset Release — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** invoking the ExitToNear / ExitToEthereum precompile via inline assembly, and never inspect the return value of that `call`. If the precompile call fails at the EVM level (returns `0`), the burn is already committed, the NEAR-side promise is never created, and the protocol's own refund callback is never triggered. The user permanently loses their bridged tokens with no recourse.

---

### Finding Description

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear` and `withdrawToEthereum` follow the same unsafe pattern:

```solidity
// EvmErc20.sol – withdrawToNear (lines 53-63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← burn is irreversible

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum` (targeting `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) and in both functions of `EvmErc20V2.sol`: [2](#0-1) [3](#0-2) [4](#0-3) 

The Aurora Engine tracks paused precompiles via `EnginePrecompilesPauser` / `PrecompileFlags`. When a precompile address is in the `paused_precompiles` set, the SputnikVM executor returns a failure result for that `call` opcode (return value `0`) without reverting the calling contract's execution frame: [5](#0-4) 

Because the `_burn` executes **before** the assembly `call`, and because the assembly block never checks `res`, the following sequence is possible:

1. `_burn(sender, amount)` — ERC-20 balance is permanently reduced.
2. `call(…precompile…)` — precompile is paused (or fails for any other reason); returns `0`.
3. `res` is silently discarded; the function returns normally.
4. No NEAR promise is ever created, so `exit_to_near_precompile_callback` is never invoked.
5. The refund branch inside the callback (`engine::refund_on_error`) is therefore unreachable: [6](#0-5) 

The refund path only fires when a NEAR-level `ft_transfer` promise **fails after being created**. It cannot compensate for an EVM-level precompile call that silently returns `0` and never creates a promise at all.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `withdrawToNear` or `withdrawToEthereum` while the corresponding precompile is paused (or otherwise fails at the EVM level) will have their ERC-20 tokens burned with no corresponding release of the underlying NEP-141 or Ethereum-side asset. The tokens are unrecoverable: the ERC-20 supply decreases, the NEP-141 balance held by the Aurora contract does not decrease, and no refund mechanism is triggered.

---

### Likelihood Explanation

**High.** The `EnginePrecompilesPauser` is an explicit, documented administrative capability. Pausing `ExitToNear` or `ExitToEthereum` is a realistic operational action (e.g., during an incident response or upgrade). Any user who attempts a withdrawal during such a pause loses funds permanently. Additionally, the precompile call can fail for reasons beyond pausing (e.g., out-of-gas in the sub-call, malformed input), making the window of exposure broader than just administrative pauses.

---

### Recommendation

1. **Check the return value** of every assembly `call` to a precompile and revert if it returns `0`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Invert the order of operations**: perform the precompile call first, verify success, and only then burn the tokens. This mirrors the checks-effects-interactions pattern and ensures the burn is atomic with a confirmed transfer.

3. Apply the fix identically to `withdrawToNear` and `withdrawToEthereum` in **both** `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) and mint tokens to `user`.
2. As the engine admin, pause the `ExitToNear` precompile via `EnginePrecompilesPauser`.
3. As `user`, call `withdrawToNear(recipientBytes, amount)`.
4. Observe: `balanceOf(user)` decreases by `amount`; the NEP-141 balance held by the Aurora contract is unchanged; no NEAR promise is created; `exit_to_near_precompile_callback` is never called; `refund_on_error` is never reached.
5. Unpause the precompile. The burned tokens are gone with no recovery path.
