### Title
Missing Validation of `refund_recipient` Address in L1→L2 Transactions Causes Permanent Loss of Refund Tokens - (File: `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The `reserved[1]` field of ABI-encoded L1→L2 and upgrade transactions encodes the `refund_recipient` address. The `validate_structure` function explicitly skips validation of this field with a `TODO: validate address?` comment. When `refund_recipient` is `address(0)` (or any uncontrolled address), the bootloader unconditionally mints the refund portion of the deposit to that address, permanently destroying the tokens with no recovery path.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`, the `validate_structure` method for ABI-encoded transactions explicitly leaves the `refund_recipient` field (`reserved[1]`) unvalidated:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
``` [1](#0-0) 

Later, in `process_l1_transaction`, the refund recipient is read directly from `transaction.reserved[1]` without any zero-address check, and `mint_base_token` is called unconditionally to send the refund to it:

```rust
let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
mint_base_token::<S, Config>(
    system,
    system_functions,
    memories.reborrow(),
    &to_refund_recipient,
    &refund_recipient,
    ...
)?;
``` [2](#0-1) 

The `ZKsyncL1Tx` struct exposes `refund_recipient` as a plain `Address` field with no constraints enforced at the encoding layer: [3](#0-2) 

A regression test in the test suite explicitly uses `address(0)` as `refund_recipient` and treats it as valid behavior, confirming the protocol accepts and processes it without error: [4](#0-3) 

---

### Impact Explanation

When an L1→L2 transaction sets `refund_recipient = address(0)`, the bootloader mints the unused-gas refund (computed as `(gas_limit - gas_used) * gas_price`) to `address(0)`. These tokens are permanently unrecoverable — there is no `recoverTokens` or equivalent mechanism. The treasury (`BASE_TOKEN_HOLDER_ADDRESS`) is debited for the full amount, and the refund is destroyed. For transactions with large gas limits and low actual gas usage, this can represent a significant token loss. [5](#0-4) 

---

### Likelihood Explanation

L1→L2 transactions are submitted by unprivileged users on L1. A user who mistakenly passes `address(0)` as `refund_recipient` (e.g., by omitting the field, using a default-initialized struct, or making a copy-paste error) will permanently lose their refund. The `ZKsyncL1Tx` struct's `refund_recipient` field defaults to `Address::ZERO` (Rust's `Default` for `Address`), making accidental zero-address submission a realistic scenario. The `TODO: validate address?` comment in the production code confirms the developers themselves identified this gap. [6](#0-5) 

---

### Recommendation

1. In `validate_structure`, reject L1/upgrade transactions where `reserved[1]` decodes to `address(0)`, replacing the `TODO` with an explicit check:
   ```rust
   Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
       let recipient = u256_to_b160_checked(self.reserved[1].read());
       if recipient == B160::ZERO {
           return Err(());
       }
   }
   ```
2. If `address(0)` must remain valid (e.g., to redirect refunds to the sender), document this explicitly and handle it as a special case in `process_l1_transaction` by substituting `transaction.from()` when `reserved[1]` is zero.
3. Remove the `TODO: validate address?` comment once the decision is made and implemented. [1](#0-0) 

---

### Proof of Concept

1. Construct a `ZKsyncL1Tx` with `refund_recipient = Address::ZERO` (the Rust default), `gas_limit = 1_000_000`, `gas_price = 1000`, and a simple ETH transfer as the body.
2. Submit the transaction to the L2 chain.
3. The transaction executes successfully (e.g., using 21,000 gas).
4. The bootloader computes `to_refund_recipient = (1_000_000 - 21_000) * 1000 = 979_000_000` tokens.
5. `mint_base_token` is called with `recipient = B160::ZERO`.
6. The treasury balance decreases by `979_000_000` tokens; `address(0)` balance increases by `979_000_000` tokens; the user receives nothing.
7. The existing regression test at line 1843 of `tests/instances/transactions/src/lib.rs` already demonstrates this exact scenario and confirms the tokens are credited to `address(0)`. [7](#0-6)

        // If the transaction reverts, then the minting of the deposit
        // reverted too. Thus, we need to refund the entire deposit minus

            .ok_or(internal_error!("td-pto"))

        // If the transaction succeeds, then it is assumed that the
        // mint to `from` address was transferred correctly too.
        // In this case, we just refund the unused gas that the

            .checked_mul(U256::from(transaction.gas_limit.read()))
            .ok_or(internal_error!("gp*gl"))?;

            .ok_or(internal_error!("pf-pto"))
