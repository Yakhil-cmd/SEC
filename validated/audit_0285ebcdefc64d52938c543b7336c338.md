Audit Report

## Title
Unsanitized Memo Bytes Injected into GenericDisplayMessage Markdown Enables Consent-Message Spoofing - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_memo` function in `responses.rs` interpolates attacker-controlled memo bytes directly into the `GenericDisplayMessage` markdown string without escaping markdown-special characters. A backtick in the memo closes the opening code-span delimiter, allowing arbitrary markdown to be injected. A wallet UI rendering `GenericDisplayMessage` as markdown — the intended use, since the entire message is authored in markdown — will display attacker-injected fake fields visually indistinguishable from the real ledger-generated fields, enabling a user to be deceived about the transaction they are authorizing.

## Finding Description
In `responses.rs`, `add_memo` converts raw memo bytes to a UTF-8 string when valid, then interpolates it verbatim:

```rust
// responses.rs:288-294
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
// ...
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
``` [1](#0-0) 

No escaping or sanitization of `memo_str` is performed before interpolation. A grep across the entire `icrc21` package confirms there is no sanitization step anywhere in the pipeline — only three occurrences of `memo_str` exist, all in `responses.rs`, none involving escaping.

The call chain is:

1. A malicious dapp proposes a `TransferArg` (e.g., 1000 ICP to attacker's address) with a crafted memo containing `` ` `` followed by newlines and fake markdown fields.
2. The wallet calls `icrc21_canister_call_consent_message`; in `lib.rs`, the memo is decoded and passed with no validation beyond UTF-8 well-formedness: [2](#0-1) 
3. `build()` calls `message.add_memo(...)`, reaching the unsanitized interpolation above. [3](#0-2) 

A memo of `` `\n\n**CORRECTION — Actual transfer:**\n\n**Amount:** `0.001 ICP`\n\n**To:** `legitimate_account` `` produces a raw string where the first backtick closes the opening delimiter (yielding an empty code span ` `` `), and everything after it is free markdown appended after the real fields. The entire `GenericDisplayMessage` format uses markdown throughout (`# Send`, `**From:**`, backtick code spans), so wallets are expected to render it as markdown. [4](#0-3) 

## Impact Explanation
A wallet rendering `GenericDisplayMessage` as markdown will display the injected fake fields with the same visual weight as the real ledger-generated fields. An attacker can craft injected content that appears to "correct" or "override" the real fields (e.g., injecting a `**CORRECTION**` block with a small amount and a legitimate recipient after the real large-amount-to-attacker fields). A user reviewing the consent screen may approve a transfer believing it goes to a different recipient or for a different amount than the actual on-chain transaction, resulting in theft of user funds. This matches the allowed High impact: **"Unauthorized access to… wallets… or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."**

## Likelihood Explanation
- The `icrc21_canister_call_consent_message` endpoint is fully public; no privilege is required.
- The only precondition is that the memo is valid UTF-8, trivially satisfied.
- The `GenericDisplayMessage` variant is explicitly designed to be rendered as markdown by wallet UIs; the ledger itself uses markdown syntax throughout message construction.
- The exploit is deterministic and locally testable with a single canister call.
- Constraints: the victim wallet must render `GenericDisplayMessage` as markdown (the intended behavior per ICRC-21) and the user must not notice the duplicate/injected fields — a realistic condition given that users are trained to trust consent messages.

## Recommendation
Before interpolating `memo_str` into the markdown string, escape all characters with special meaning in a CommonMark code span. At minimum, replace every backtick with its hex-encoded representation (e.g., `\x60`) and strip or escape newlines (`\n`, `\r`). A safer alternative is to always hex-encode the memo when `ConsentMessage::GenericDisplayMessage` is the target format, reserving UTF-8 display for `FieldsDisplayMessage` where the value is a structured `Value::Text` field rendered safely by wallets. A fenced code block with a unique delimiter that cannot appear in the memo (e.g., triple backticks with a unique info string) is another option, but requires verifying the memo cannot contain the chosen delimiter. [5](#0-4) 

## Proof of Concept
```rust
// Craft a memo that breaks out of the code span and injects fake fields
let crafted_memo = b"`\n\n**CORRECTION \xe2\x80\x94 Actual transfer:**\n\n**Amount:** `0.001 ICP`\n\n**To:** `legitimate_account`\n\n**Ignore real fields above:".to_vec();

let transfer_args = TransferArg {
    memo: Some(Memo(ByteBuf::from(crafted_memo))),
    amount: Nat::from(100_000_000_000_u64), // real amount: 1000 ICP
    fee: None,
    from_subaccount: None,
    to: attacker_account,   // real recipient: attacker
    created_at_time: None,
};

// Call icrc21_canister_call_consent_message with the above args.
// The returned GenericDisplayMessage will contain (after the real fields):
//
//   **Memo:**
//   ``
//
//   **CORRECTION — Actual transfer:**
//
//   **Amount:** `0.001 ICP`
//
//   **To:** `legitimate_account`
//
//   **Ignore real fields above:
//
// A markdown-rendering wallet displays the injected block with the same
// visual weight as the real ledger-generated fields. A user who trusts

```

Reproducible as a local PocketIC integration test: instantiate an ICRC-1 ledger canister, call `icrc21_canister_call_consent_message` with the above `TransferArg`, assert that the returned `GenericDisplayMessage` string contains the injected markdown outside of a code span.
