Audit Report

## Title
Fees Double-Deducted When User Specifies Explicit Disbursement Amount in `disburse_neuron` - (File: `rs/nns/governance/src/governance.rs`)

## Summary
When a neuron controller calls `disburse_neuron` with an explicit `amount = Some(X)`, the NNS governance canister subtracts `neuron_fees_e8s` from `X` before computing the disbursement transfer, causing the user to receive `X - fees - tx_fee` instead of the expected `X - tx_fee`. The fees are also burned separately from the neuron's subaccount, meaning the fees are effectively deducted twice from the user's perspective. This behavior is inconsistent with the SNS governance implementation of the same operation and causes a direct, unannounced financial loss to any neuron controller with accumulated fees who specifies an explicit disbursement amount.

## Finding Description
In `rs/nns/governance/src/governance.rs` at lines 2016–2021, when `disburse.amount = Some(X)`:

```rust
let mut disburse_amount_e8s = disburse
    .amount
    .as_ref()
    .map_or(neuron_minted_stake_e8s, |a| {
        a.e8s.saturating_sub(fees_amount_e8s)  // fees subtracted from user-specified X
    });
```

`disburse_amount_e8s` becomes `X - fees`. Then at lines 2025–2027, `transaction_fee_e8s` is also subtracted, yielding `X - fees - tx_fee`. Separately, at lines 2046–2064, `fees_amount_e8s` is burned from the neuron's subaccount via a ledger transfer. The final disbursement transfer at lines 2091–2100 sends `disburse_amount_e8s = X - fees - tx_fee` to the user.

The total deducted from the neuron's on-chain balance is `fees + (X - fees) + tx_fee = X + tx_fee`, but the user only receives `X - fees - tx_fee`. The `fees` amount is burned, not credited anywhere — it is a net loss to the user beyond what they specified.

The comment at lines 2012–2015 claims "symmetry" because `minted_stake_e8s() = cached_stake - fees`, so both branches of `map_or` subtract fees. However, this symmetry only holds if the user passes `Some(cached_stake)` — i.e., the full cached stake including fees. If the user passes `Some(minted_stake)` (the logical "available" amount), they receive `minted_stake - fees - tx_fee = cached_stake - 2·fees - tx_fee`, which is `fees` fewer tokens than the `amount = None` path for the same logical amount.

The SNS governance implementation at lines 1160–1166 of `rs/sns/governance/src/governance.rs` does not subtract fees from the user-specified amount:

```rust
let mut disburse_amount_e8s = disburse
    .amount
    .as_ref()
    .map_or(neuron.stake_e8s(), |a| a.e8s);  // no fee subtraction
disburse_amount_e8s = disburse_amount_e8s.min(neuron.stake_e8s());
```

This confirms the NNS behavior is inconsistent and incorrect relative to the intended design.

## Impact Explanation
Any NNS neuron controller with `neuron_fees_e8s > 0` (accumulated from rejected governance proposals) who calls `disburse_neuron` with an explicit `amount` suffers a permanent, unannounced loss of `neuron_fees_e8s` ICP beyond the expected transaction fee. The lost tokens are burned. For neurons with large accumulated fees, this can be a material amount of ICP. This constitutes a significant NNS security impact with concrete, demonstrable user financial harm, fitting the **High ($2,000–$10,000)** bounty category for "Significant NNS... security impact with concrete user or protocol harm."

## Likelihood Explanation
Any neuron controller who has had governance proposals rejected (accumulating `neuron_fees_e8s > 0`) and calls `disburse_neuron` with an explicit `amount` will trigger this. This is a normal, expected usage pattern — users commonly specify explicit amounts when partially disbursing. No special privileges, technical skill, or attacker involvement is required; the neuron controller triggers the loss themselves through ordinary use of the governance API. Likelihood is **Medium** given that accumulated fees require prior rejected proposals, but the trigger condition is straightforward and repeatable.

## Recommendation
Remove the fee subtraction from the user-specified amount branch. Mirror the SNS implementation: when `amount = Some(X)`, use `X` directly (capped at the neuron's available stake) without subtracting `fees_amount_e8s`. The fees are already handled separately via the burn transfer. The fix at lines 2016–2021 should be:

```rust
let mut disburse_amount_e8s = disburse
    .amount
    .as_ref()
    .map_or(neuron_minted_stake_e8s, |a| a.e8s);
// Cap at minted stake to prevent over-disbursement
disburse_amount_e8s = disburse_amount_e8s.min(neuron_minted_stake_e8s);
```

Also update or remove the misleading "symmetry" comment at lines 2012–2015.

## Proof of Concept
A deterministic unit test can prove this:

1. Create a neuron with `cached_neuron_stake_e8s = 1_000_000_000` (10 ICP) and `neuron_fees_e8s = 100_000_000` (1 ICP).
2. Call `disburse_neuron` with `amount = Some(500_000_000)` (5 ICP).
3. Observe that the ledger disbursement transfer amount is `500_000_000 - 100_000_000 - tx_fee = 400_000_000 - tx_fee` instead of the expected `500_000_000 - tx_fee`.
4. Confirm the fee burn transfer of `100_000_000` also occurred.
5. Compare with calling `disburse_neuron` with `amount = None` on a neuron with `cached_neuron_stake_e8s = 500_000_000` and `neuron_fees_e8s = 0` — the user receives `500_000_000 - tx_fee`, demonstrating the inconsistency.

Existing test infrastructure in `rs/nns/governance/src/governance_tests/` can be used to mock the ledger and assert the exact transfer amounts.
