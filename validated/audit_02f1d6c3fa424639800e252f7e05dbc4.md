### Title
Truncating Floor Division in `native_per_pubdata` Calculation Systematically Undercharges Pubdata Fees — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` rate used to charge native resources for every ZK L2 transaction's pubdata is computed with truncating (floor) integer division (`wrapping_div`), while the analogous `native_per_gas` rate is computed with ceiling division (`div_ceil`). This asymmetry causes a systematic, per-transaction undercharge of pubdata costs, benefiting every transaction sender at the operator's expense.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two key per-unit rates are computed as follows:

```rust
// native_per_gas uses ceiling division — rounds UP (conservative)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata uses floor division — rounds DOWN (biased)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [1](#0-0) 

The same floor division is replicated in the public API helper:

```rust
// native_per_pubdata = pubdata_price / native_price  (floor, not ceil)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

The computed `native_per_pubdata` is then used directly to charge native resources for every pubdata byte consumed by the transaction:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

The correct formula is `ceil(pubdata_price / native_price)`. Using `floor` instead introduces a systematic downward bias of up to `(native_price − 1)` native units per pubdata byte. This is the direct analog of the biased `1/T` estimator in the external report: a consistent but biased formula that always underestimates the true cost.

---

### Impact Explanation

Every ZK L2 transaction that writes pubdata is undercharged by `(pubdata_price mod native_price)` native units per pubdata byte. For a transaction writing `B` pubdata bytes, the total undercharge is:

```
undercharge = (pubdata_price mod native_price) × B  native units
```

In the worst case (`pubdata_price mod native_price = native_price − 1`), a transaction writing 1 000 pubdata bytes is undercharged by `(native_price − 1) × 1000` native units. The operator absorbs this loss on every transaction. Because `native_per_pubdata` is also used to compute the intrinsic pubdata overhead deducted from the native budget, the undercharge also reduces the effective native resource guard against pubdata-heavy transactions, weakening the DoS protection that the native resource model is designed to provide.

---

### Likelihood Explanation

This affects every ZK L2 transaction that generates pubdata (i.e., any transaction that writes to storage). No special conditions are required. Any unprivileged transaction sender benefits automatically. The effect is deterministic and repeatable across all blocks.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is already computed:

```rust
// Before (biased — floor division):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (unbiased — ceiling division):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [2](#0-1) 

---

### Proof of Concept

Suppose the block oracle sets:
- `pubdata_price = 101` (wei per pubdata byte)
- `native_price = 10` (wei per native unit)

Then:
- `native_per_pubdata` (floor) = `101 / 10 = 10`
- `native_per_pubdata` (ceil) = `ceil(101 / 10) = 11`

A transaction writing 500 pubdata bytes is charged `10 × 500 = 5 000` native units instead of the correct `11 × 500 = 5 500`. The operator is undercompensated by 500 native units per such transaction. At scale (many transactions per block), this represents a continuous, predictable financial loss to the operator that any transaction sender exploits passively by simply submitting transactions.
