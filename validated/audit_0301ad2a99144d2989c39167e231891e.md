Audit Report

## Title
Repeated `ManifestChunk` Requests Trigger Uncached `encode_manifest` Per Call, Enabling CPU Exhaustion — (`rs/state_manager/src/state_sync/types.rs`)

## Summary

Every `ManifestChunk` request served by `StateSyncMessage::get_chunk` unconditionally invokes `encode_manifest(&self.manifest)`, performing a full protobuf serialization of the manifest with no result caching. The P2P chunk handler dispatches each request directly to `tokio::task::spawn_blocking` with no per-peer rate limiting, no deduplication, and no concurrency cap at the application layer. A single Byzantine subnet peer can flood the serving replica with repeated manifest-chunk requests, causing O(manifest\_size) CPU work per request and sustained blocking thread pool exhaustion.

## Finding Description

In `StateSyncMessage::get_chunk`, the `StateSyncChunk::ManifestChunk(index)` branch calls `encode_manifest(&self.manifest)` on every invocation: [1](#0-0) 

`encode_manifest` performs a full protobuf serialization, allocating a new `Vec<u8>` each time: [2](#0-1) 

The result is never stored. `StateSyncMessage` holds only the raw `Manifest` with no cached encoded form: [3](#0-2) 

The P2P chunk handler in `state_sync_chunk_handler` dispatches every incoming request directly to `spawn_blocking` with no admission control: [4](#0-3) 

The router in `build_state_sync_manager` registers the chunk route with no middleware layers for rate limiting or concurrency control: [5](#0-4) 

The `TOO_MANY_REQUESTS` / `Overloaded` handling that exists in the codebase is exclusively on the **client** side (`parse_chunk_handler_response`), confirming the server never emits rate-limit responses: [6](#0-5) 

## Impact Explanation

This enables a Byzantine subnet peer to cause sustained CPU exhaustion and Tokio blocking thread pool starvation on the victim replica. Exhausting the blocking thread pool stalls all other `spawn_blocking` operations — checkpoint reads, file chunk serving, and other I/O — degrading consensus participation and legitimate state-sync serving. This matches the allowed High bounty impact: **Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation

The attacker must be a single Byzantine subnet node operating below the consensus fault threshold — no governance majority, no threshold corruption, and no leaked keys are required. The exploit is mechanically trivial: send repeated POST requests to `/state-sync/chunk` with a valid `ManifestChunk` ID. The number of valid manifest chunk IDs equals `meta_manifest.sub_manifest_hashes.len()` (one per ~1 MiB of encoded manifest), each triggering a full re-serialization. The attack is repeatable and requires no victim interaction.

## Recommendation

Pre-compute and cache the encoded manifest bytes inside `StateSyncMessage` at construction time (e.g., as an `Arc<Vec<u8>>` or `once_cell::sync::OnceCell<Vec<u8>>`), so `encode_manifest` is called at most once per checkpoint regardless of request volume. Additionally, add per-peer request rate limiting (e.g., a `tower::limit::RateLimitLayer` or a per-peer token bucket) in `state_sync_chunk_handler` to bound the serving rate for any single peer.

## Proof of Concept

```rust
// Unit benchmark — no network required
let msg: StateSyncMessage = /* construct with large checkpoint manifest */;
let chunk_id = ChunkId::new(MANIFEST_CHUNK_ID_OFFSET); // valid ManifestChunk(0)

let start = Instant::now();
for _ in 0..1000 {
    let _ = msg.get_chunk(chunk_id); // each call invokes encode_manifest()
}
println!("Total: {:?}", start.elapsed());
// Each iteration re-serializes the full manifest; total ≈ 1000 × T(encode_manifest)
```

A Byzantine peer achieves the same effect by sending 1000 POST requests to `/state-sync/chunk` with `chunk_id = MANIFEST_CHUNK_ID_OFFSET` over QUIC to the serving replica's P2P endpoint, each dispatched to a new `spawn_blocking` task with no backpressure.

```rust
pub(crate) async fn state_sync_chunk_handler<T: 'static>(
    State(state): State<Arc<StateSyncChunkHandler<T>>>,
    payload: Bytes,
) -> Result<Bytes, StatusCode> {
    // Parse payload
    let pb::StateSyncChunkRequest { id, chunk_id } =
        pb::StateSyncChunkRequest::decode(payload).map_err(|_| StatusCode::BAD_REQUEST)?;
    let artifact_id: StateSyncArtifactId = id.map(From::from).ok_or(StatusCode::BAD_REQUEST)?;
    let chunk_id = ChunkId::from(chunk_id);

    let jh =
        tokio::task::spawn_blocking(
            move || match state.state_sync.chunk(&artifact_id, chunk_id) {
                Some(data) => {
                    let pb_chunk = pb::StateSyncChunkResponse { data: data.take() };
                    let mut raw = BytesMut::with_capacity(pb_chunk.encoded_len());
                    pb_chunk.encode(&mut raw).expect("Allocated enough memory");
                    let raw = raw.freeze();

                    let compressed = zstd::bulk::compress(&raw, zstd::DEFAULT_COMPRESSION_LEVEL)
                        .expect("Compression failed");
                    state
                        .metrics
                        .compression_ratio
                        .observe(raw.len() as f64 / compressed.len() as f64);
                    Ok(compressed)
                }
                None => Err(StatusCode::NO_CONTENT),
            },
        );
    let data = jh.await.map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)??;

    Ok(data.into())
}
```
