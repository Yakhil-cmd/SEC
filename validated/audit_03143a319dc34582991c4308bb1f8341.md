Audit Report

## Title
`FlexibleHttpRequest` Permanently Loses Refundable Cycles Due to Unconditional `PayAsYouGo` Rejection in Adapter Client - (File: `rs/https_outcalls/client/src/client.rs`)

## Summary

`FlexibleHttpRequest` hardcodes `PricingVersion::PayAsYouGo` on every context it creates. The execution layer takes the entire payment upfront under `PayAsYouGo`, storing `payment - base_fee` as `refundable_cycles`. The adapter client then unconditionally rejects every `PayAsYouGo` request with `SysFatal` and returns a zero-refund `CanisterHttpPaymentReceipt` (produced by `LegacyTracker`, which is always instantiated regardless of pricing version). The stored refundable cycles are never returned, causing permanent cycles loss on every `FlexibleHttpRequest` call.

## Finding Description

**Step 1 — `FlexibleHttpRequest` hardcodes `PayAsYouGo`.**
`CanisterHttpRequestContext::generate_from_flexible_args` in `rs/types/types/src/canister_http.rs` unconditionally sets `pricing_version: PricingVersion::PayAsYouGo` (line 692). There is no caller-controlled override.

**Step 2 — Execution layer takes entire payment upfront.**
`try_add_http_context_to_replicated_state` in `rs/execution_environment/src/execution_environment.rs` (lines 2213–2219) matches on `PricingVersion::PayAsYouGo` and calls `canister_http_request_context.request.payment.take()`, zeroing the payment field. Before doing so, it computes `refundable_cycles = payment - base_fee` and stores it in `refund_status.refundable_cycles` (lines 2189–2204). The base fee is non-refundable; the remainder is supposed to be returned via the receipt mechanism.

**Step 3 — Adapter client unconditionally rejects `PayAsYouGo`.**
In `rs/https_outcalls/client/src/client.rs` (lines 155–178), the spawned async task checks `request_pricing_version == PricingVersion::PayAsYouGo` and immediately sends back a `SysFatal` reject paired with `budget.create_payment_receipt()`.

**Step 4 — `budget` is always a `LegacyTracker` and always returns zero refund.**
`PricingFactory::new_tracker` in `rs/https_outcalls/pricing/src/lib.rs` (lines 56–60) unconditionally returns `Box::new(LegacyTracker::new(...))` with an explicit TODO comment: `// TODO(IC-1937): This should take into account context.pricing_version`. `LegacyTracker::create_payment_receipt` (lines 48–52 of `rs/https_outcalls/pricing/src/legacy.rs`) always returns `CanisterHttpPaymentReceipt::default()`, i.e., `refund: Cycles::new(0)`.

**Step 5 — Zero-refund receipt passes consensus validation; refundable cycles are never returned.**
`check_refund_allowance` in `rs/https_outcalls/consensus/src/payload_builder/utils.rs` (lines 81–92) only rejects receipts where `refund > per_replica_allowance`. A zero refund always passes. The `refundable_cycles` stored in the context are never disbursed because the receipt claims nothing.

The execution-layer test `execute_flexible_canister_http_request` in `rs/execution_environment/src/execution_environment/tests.rs` (lines 3584–3586) explicitly confirms that under `Normal` cost schedule, `request.payment` becomes `Cycles::new(0)` and `refundable_cycles` is `payment - base_fee`, proving the upfront deduction is real.

## Impact Explanation

Any canister that calls `FlexibleHttpRequest` with a non-zero payment on a subnet with a normal cost schedule permanently loses `payment - base_fee` cycles on every invocation. The base fee itself is also consumed. Because the rejection is unconditional and the refund path is structurally broken (wrong tracker type, zero receipt), there is no code path that returns the refundable cycles. This constitutes permanent, irreversible loss of cycles for every caller of the `FlexibleHttpRequest` API — a concrete financial harm to canister developers matching the "permanent loss of cycles" impact class.

## Likelihood Explanation

The `FlexibleHttpRequest` endpoint is implemented and reachable in the production codebase. The rejection in the adapter is unconditional — every single `PayAsYouGo` request is rejected at line 155 of `client.rs` with no fallback. Any canister developer who calls `FlexibleHttpRequest` triggers the loss on every call. No special privileges, no race conditions, and no external dependencies are required.

## Recommendation

Two independent fixes are needed:

1. **Adapter client**: Implement a `PayAsYouGo`-aware `BudgetTracker` in `rs/https_outcalls/pricing/` and update `PricingFactory::new_tracker` (resolving TODO IC-1937) to return it when `context.pricing_version == PayAsYouGo`. The tracker must compute the actual refund and return it via `create_payment_receipt`.

2. **Short-term guard**: Until the tracker is implemented, reject `PayAsYouGo` requests *before* the payment is deducted in `try_add_http_context_to_replicated_state` — return a `UserError` so the full payment is refunded to the caller rather than taken upfront and lost.

## Proof of Concept

1. Deploy a canister on a subnet with `FlexibleHttpRequest` enabled and a normal cost schedule.
2. Call `FlexibleHttpRequest` with payment `P` (e.g., 1,000,000,000 cycles).
3. Observe the canister's cycle balance decreases by `P` (entire payment taken upfront).
4. The canister receives a `SysFatal` reject: `"Canister HTTP request with PayAsYouGo pricing is not supported"`.
5. Observe the canister's cycle balance does **not** recover by `P - base_fee`.
6. Repeat: every call loses `P - base_fee` cycles permanently.

Deterministic unit test path: extend `execute_flexible_canister_http_request` in `rs/execution_environment/src/execution_environment/tests.rs` to simulate the adapter's zero-refund receipt being processed and assert that `refund_status.refundable_cycles` is never credited back to the canister's balance.
