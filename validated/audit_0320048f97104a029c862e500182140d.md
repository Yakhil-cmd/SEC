### Title
Race Condition in SNS Governance `refresh_neuron` Allows Cached Stake Inflation via Mismatched Subaccount Lock - (File: rs/sns/governance/src/governance.rs)

### Summary
SNS Governance's `refresh_neuron` function makes an async ledger balance query without acquiring a lock on the neuron being refreshed. The outer `manage_neuron_internal` acquires a lock only on the `NeuronId` derived from `ManageNeuron.subaccount`, but when `ClaimOrRefresh` is invoked via `By::MemoAndController`, the actual neuron being refreshed is identified by a separately computed `NeuronId` from `compute_neuron_staking_subaccount_bytes(controller, memo)`. If these two identifiers differ — which an attacker can deliberately cause — the actual neuron is unprotected during the async await, enabling a concurrent disburse to race with the balance snapshot, ultimately inflating `cached_neuron_stake_e8s` above the true ledger balance.

### Finding Description

In `manage_neuron_internal`, a lock is acquired on the neuron identified by `ManageNeuron.subaccount`: [1](#0-0) 

This lock is on `neuron_id` derived from the user-supplied `ManageNeuron.subaccount` field. When the command is `ClaimOrRefresh(By::MemoAndController(...))`, execution flows to: [2](#0-1) 

Which calls: [3](#0-2) 

Here, `nid` is computed independently from `compute_neuron_staking_subaccount_bytes(controller, memo)`. If the attacker supplies a `ManageNeuron.subaccount` that does not match the memo+controller derivation, `nid ≠ neuron_id`. The lock held by `manage_neuron_internal` is on the wrong neuron. `refresh_neuron` then proceeds to make an async ledger call with no lock on `nid`: [4](#0-3) 

After the await returns, the balance snapshot is written unconditionally: [5](#0-4) 

**Contrast with NNS Governance**, which correctly acquires its own lock on the neuron before the async call, independent of any outer lock: [6](#0-5) 

SNS `refresh_neuron` has no equivalent lock acquisition.

### Impact Explanation

During the async `account_balance` await, the IC scheduler can interleave other messages. A concurrent `Disburse` on the same neuron (sent with the correct subaccount, so it acquires the real neuron lock) can complete — reducing the ledger balance and updating `cached_neuron_stake_e8s` to the correct lower value — before the refresh callback resumes. The refresh callback then overwrites `cached_neuron_stake_e8s` with the stale, pre-disburse balance snapshot, inflating the neuron's recorded stake above its actual ledger balance. Since `cached_neuron_stake_e8s` directly determines voting power in SNS governance, this constitutes a **governance authorization / ledger conservation bug**: the attacker's neuron votes with more weight than its actual token backing justifies.

### Likelihood Explanation

The attack is reachable by any unprivileged ingress sender via the public `manage_neuron` endpoint: [7](#0-6) 

The attacker controls both the `ManageNeuron.subaccount` field (to misdirect the lock) and the memo+controller (to target their own neuron). The only probabilistic element is the interleaving of the disburse callback before the refresh callback, which is a natural consequence of concurrent message submission and is achievable by submitting both messages in the same round. No privileged access, key material, or majority corruption is required.

### Recommendation

`refresh_neuron` in SNS Governance should acquire its own lock on `nid` before the async `account_balance` call, mirroring the NNS Governance pattern:

```rust
async fn refresh_neuron(&mut self, nid: &NeuronId) -> Result<(), GovernanceError> {
    // ... existing checks ...
    let _neuron_lock = self.lock_neuron_for_command(
        nid,
        NeuronInFlightCommand { timestamp: now, command: Some(...) },
    )?;
    let balance = self.ledger.account_balance(account).await?;
    // ... rest of function ...
}
```

This ensures the neuron is protected regardless of what lock the outer caller holds.

### Proof of Concept

1. Attacker owns a neuron with subaccount `S` (derived from controller `C`, memo `M`).
2. Attacker sends `manage_neuron` with `ManageNeuron.subaccount = S'` (arbitrary dummy bytes, `S' ≠ S`) and `command = ClaimOrRefresh(By::MemoAndController(C, M))`.
3. `manage_neuron_internal` acquires lock on neuron `S'` (dummy); `refresh_neuron` is called for neuron `S` with no lock on `S`.
4. `refresh_neuron` issues `account_balance(S_account)` and suspends at the await.
5. Attacker concurrently sends `manage_neuron` with `ManageNeuron.subaccount = S` and `command = Disburse(amount)`. This acquires the real lock on `S`, executes the ledger transfer, reduces `cached_neuron_stake_e8s` to `B - amount`, and releases the lock.
6. The refresh callback resumes with the pre-disburse balance `B` and writes `cached_neuron_stake_e8s = B`, overwriting the correct post-disburse value.
7. Neuron `S` now reports `cached_neuron_stake_e8s = B` while its actual ledger balance is `B - amount`, inflating its voting power.
