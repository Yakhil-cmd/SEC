Audit Report

## Title
Unbounded Header Count in `read_request` Enables Memory Amplification via `http::HeaderMap` — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary

The `read_request` function in the QUIC transport layer caps raw stream bytes at `MAX_MESSAGE_SIZE_BYTES` (128 MiB) but applies no limit on the number of protobuf-encoded headers. A Byzantine peer can craft a 128 MiB protobuf message containing millions of minimal headers; when the header-building loop constructs the `http::HeaderMap`, per-entry struct overhead amplifies memory usage by roughly 16×, reaching ~2 GB per stream. With up to 1,000 concurrent bidirectional streams per connection, this can drive a victim replica into OOM and crash it.

## Finding Description

In `rs/p2p/quic_transport/src/request_handler.rs`, `read_request` reads raw bytes bounded by `MAX_MESSAGE_SIZE_BYTES`:

```rust
// line 173-178
let request_bytes = recv_stream
    .read_to_end(MAX_MESSAGE_SIZE_BYTES)   // 128 MiB cap
    .await ...?;
``` [1](#0-0) 

After protobuf decode, the header loop has no count guard:

```rust
// lines 201-204
for h in request_proto.headers {
    let pb::HttpHeader { key, value } = h;
    request_builder = request_builder.header(key, value);  // allocates per entry
}
``` [2](#0-1) 

`MAX_MESSAGE_SIZE_BYTES` is defined as 128 MiB: [3](#0-2) 

A protobuf `HttpHeader` with a 1-byte key and 1-byte value encodes in ~8 bytes, so a 128 MiB message can carry ~16.8 million headers. Each call to `request_builder.header(key, value)` causes the `http` crate to allocate a `Bucket<HeaderValue>` (~60 bytes), a key heap block (~32 bytes), and a value heap block (~32 bytes), totalling ~124 bytes per entry. 16.8M × 124 bytes ≈ **2.08 GB** from a 128 MiB input (~16× amplification). The existing `MAX_MESSAGE_SIZE_BYTES` check is entirely insufficient because it only bounds the wire bytes, not the decoded in-memory representation.

Each accepted bidirectional stream is spawned as an independent tokio task with no backpressure: [4](#0-3) 

The stream concurrency limit is 1,000 per connection: [5](#0-4) 

1,000 concurrent streams × ~2.08 GB = ~2 TB → OOM → replica crash.

## Impact Explanation

A single Byzantine subnet node can crash any honest replica it is connected to by exhausting its memory. Crashing enough honest replicas disrupts subnet consensus and halts finalization. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

The attack requires only one compromised subnet node holding a valid TLS node certificate — a single Byzantine peer, well below the consensus fault threshold. The crafted message is valid protobuf and passes all existing checks. No race conditions, timing windows, or victim mistakes are required. The attacker freely controls header count and sizes within the 128 MiB raw-byte budget. The attack is repeatable: after a victim restarts, the attacker can immediately re-trigger it.

## Recommendation

Add a header count guard immediately after protobuf decode, before the loop:

```rust
const MAX_HEADERS_PER_REQUEST: usize = 100;

if request_proto.headers.len() > MAX_HEADERS_PER_REQUEST {
    return Err(P2PError::from("too many headers".to_string()));
}
```

Additionally, bound individual header key and value byte lengths independently of total message size to prevent analogous amplification via large individual header strings.

## Proof of Concept

Construct a `pb::HttpRequest` protobuf with `N = floor(128 MiB / 8)` headers, each with key `"a"` (1 byte) and value `"b"` (1 byte), encoded to ≤128 MiB. Open 1,000 QUIC bidirectional streams to the victim node (using a valid Byzantine node TLS certificate) and send this message on each stream simultaneously. Monitor victim RSS — it will grow toward ~2 TB before the OOM killer terminates the replica process. A deterministic unit test can verify the amplification by calling `read_request` with a crafted `RecvStream` mock containing the above protobuf payload and asserting that the resulting `Request` holds ~16.8M header entries.
