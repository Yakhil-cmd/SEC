### Title
Incorrect `MAX_BLOBS_PER_BLOCK` Cap of 6 Instead of Pectra-Correct 9 Silently Rejects Valid EIP-4844 Blob Transactions in the ZK Path - (File: `zk_ee/src/system/constants.rs`)

---

### Summary

`zk_ee/src/system/constants.rs` defines `MAX_BLOBS_PER_BLOCK = 6`, the pre-Pectra value. Post-Pectra (EIP-4844 Prague/Electra), the correct maximum is 9. The ZK transaction validation path explicitly imports and enforces this stale constant, causing any EIP-4844 transaction carrying 7–9 blobs to be unconditionally rejected with `BlobListTooLong`. The Ethereum block-flow path already uses the correct value of 9 in two separate locations, creating an internal inconsistency. The ZK DA commitment generator buffer is also already sized for 9 blobs, confirming the intended design supports 9.

---

### Impact Explanation

Any EIP-4844 transaction submitted through the ZK path with 7, 8, or 9 blob versioned hashes is unconditionally rejected at validation time with `InvalidTransaction::BlobListTooLong`, even though the Pectra specification permits up to 9 blobs per block. This silently discards 33% of the available blob-carrying capacity per block. Users who construct valid Pectra-compliant blob transactions lose fees paid for submission and cannot use the full DA bandwidth the protocol is designed to support. The ZK DA commitment generator buffer already accommodates 9 blobs, so the infrastructure is ready but the validation gate is wrong.

---

### Likelihood Explanation

Any unprivileged user who submits a ZK-path EIP-4844 transaction with more than 6 blobs triggers the bug deterministically. No special privileges, oracle manipulation, or governance access is required. The entry path is a standard signed transaction. As blob usage grows post-Pectra, the probability of hitting this limit increases.

---

### Recommendation

Update `zk_ee/src/system/constants.rs` to reflect the Pectra maximum:

```rust
// Was: pub const MAX_BLOBS_PER_BLOCK: usize = 6;
pub const MAX_BLOBS_PER_BLOCK: usize = 9;
pub const TARGET_BLOBS_PER_BLOCK: usize = 6; // EIP-4844 target (unchanged)
```

This aligns `zk_ee` with the values already used in `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs` and `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs`, and matches the 9-blob capacity already allocated in the ZK DA commitment generator buffer.

---

### Proof of Concept

1. Construct a ZKsync ZK-path EIP-4844 transaction with 7 blob versioned hashes (all with valid `0x01` version prefix).
2. Submit it to the ZK bootloader.
3. Execution reaches `validate_and_compute_fee_for_transaction` → `parse_blobs_list::<6>` → `blobs_list.count (7) > MAX_BLOBS_IN_TX (6)` → returns `Err(TxError::Validation(InvalidTransaction::BlobListTooLong))`.
4. The transaction is rejected despite being valid under Pectra rules.
5. The same transaction submitted through the Ethereum path would pass block-header validation (which uses `MAX_BLOBS_PER_BLOCK = 9`), demonstrating the inconsistency.
