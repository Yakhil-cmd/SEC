Audit Report

## Title
`SenderInfoContent` canister signature lacks request-context binding, enabling cross-canister replay of identity attestations - (File: `rs/types/types/src/messages/http.rs`)

## Summary
`SenderInfoContent` signs only the raw `info` bytes under the `"ic-sender-info"` domain separator, with no binding to the target canister ID, sender principal, or ingress expiry. A canister signature produced by an identity provider (e.g., Internet Identity) over a given `info` blob is therefore valid for any request to any canister on the IC. An unprivileged attacker who obtains one valid `(info, signer, sig)` triple can replay it verbatim to any other canister, causing that canister's `ic0.msg_caller_info_data()` and `ic0.msg_caller_info_signer()` calls to return the replayed attestation as if it were freshly produced for that request.

## Finding Description

**Root cause — `SenderInfoContent` signs only the raw blob:**

`rs/types/types/src/messages/http.rs` L342–348 defines:

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);   // only the raw info blob
    }
}
```

`rs/types/types/src/crypto/sign.rs` L161–165 prepends only the `"ic-sender-info"` domain separator:

```rust
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
```

The full signed bytes are therefore `\x0Eic-sender-info || info_bytes`. No target canister ID, no sender principal, and no ingress expiry are included.

**Verifier performs no context-binding check:**

`rs/validator/src/ingress_validation.rs` L529–544 constructs the signable content from only `sender_info.info` and verifies it:

```rust
let sender_info_content = SenderInfoContent(&sender_info.info);
let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

verify_canister_sig_with_fallback!(
    validator,
    &canister_sig,
    &sender_info_content,
    &public_key,
    ...
);
```

The only binding check performed (L516–521) is that the canister ID encoded in `sender_pubkey` matches `sender_info.signer`. There is no check that the attestation was produced for the specific target canister, the specific sender principal, or within a bounded time window.

**Exploit path:**

1. Attacker sends a legitimate update call to canister A with `sender_info = {info: b"kyc_passed", signer: II_id, sig: <II_sig>}`. II signs `\x0Eic-sender-info || b"kyc_passed"`.
2. Attacker constructs a new update call to canister B with the identical `sender_info` triple. The outer envelope is signed with the attacker's own key; the `MessageId` differs (because `canister_id` differs), so the outer signature is valid.
3. `verify_sender_info_canister_sig` accepts the request: the canister ID in `sender_pubkey` matches `sender_info.signer`, and the canister signature over `SenderInfoContent(&b"kyc_passed")` verifies correctly.
4. Inside canister B, `ic0.msg_caller_info_data()` returns `b"kyc_passed"` and `ic0.msg_caller_info_signer()` returns `II_id` — indistinguishable from a fresh attestation, despite II never having attested these attributes for canister B.

The `sender_info` feature is wired into production execution in `rs/execution_environment/src/execution/call_or_task.rs`, `rs/execution_environment/src/query_handler/query_context.rs`, and the system API in `rs/embedders/src/wasmtime_embedder/system_api.rs`, confirming the replay reaches live canister execution.

## Impact Explanation

Any canister that gates privileged actions on `sender_info` content (e.g., "II attests this user is KYC-verified, allow withdrawal") is vulnerable to authorization bypass via replay. The `sender_info` feature is explicitly designed for Internet Identity to attest user attributes to target canisters; the absence of context binding breaks the core security guarantee of that mechanism. This constitutes a significant Internet Identity and IC infrastructure security impact with concrete user harm, matching the High ($2,000–$10,000) impact class: "Significant... Internet Identity... security impact with concrete user or protocol harm."

## Likelihood Explanation

The attack requires only that the adversary obtain one valid `sender_info` signature from the target identity provider — achievable by making a single legitimate request to any canister that accepts `sender_info`. No privileged access, no key compromise, and no threshold corruption is required. The attacker controls the outer envelope and can direct the replayed `sender_info` to any target canister. The attack is repeatable indefinitely because the signature carries no expiry of its own.

## Recommendation

Extend `SenderInfoContent` to include the target canister ID and sender principal, and include them in the signed bytes:

```rust
pub struct SenderInfoContent<'a> {
    pub info: &'a [u8],
    pub canister_id: &'a CanisterId,
    pub sender: &'a UserId,
}

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.canister_id.get().as_slice());
        bytes.extend_from_slice(self.sender.get().as_slice());
        bytes.extend_from_slice(self.info);
    }
}
```

Update `verify_sender_info_canister_sig` in `rs/validator/src/ingress_validation.rs` to pass the target canister ID and sender from the request content when constructing `SenderInfoContent`. Identity providers must update their signing logic accordingly. Optionally, add an expiry field to `SignedSenderInfo` and enforce it during validation, analogous to delegation expiry.

## Proof of Concept

1. Send a valid update call to canister A with `sender_info = {info: b"kyc_passed", signer: II_id, sig: <II_sig>}`.
2. Construct a second update call to canister B with the identical `sender_info` triple. Sign the outer envelope with the same user key (the `MessageId` will differ because `canister_id` differs, but `sender_info.sig` is reused verbatim).
3. Submit the second request. `verify_sender_info_canister_sig` in `rs/validator/src/ingress_validation.rs` L494–544 will accept it: the canister ID in `sender_pubkey` matches `sender_info.signer`, and the canister signature over `SenderInfoContent(&b"kyc_passed")` verifies correctly.
4. Inside canister B, `ic0.msg_caller_info_data()` returns `b"kyc_passed"` and `ic0.msg_caller_info_signer()` returns `II_id`.

A deterministic integration test using PocketIC can confirm this by: (a) deploying two canisters, (b) submitting a request to canister A with a mock II canister signature over `\x0Eic-sender-info || b"kyc_passed"`, (c) replaying the identical `sender_info` in a request to canister B, and (d) asserting that canister B's `ic0.msg_caller_info_data()` returns `b"kyc_passed"` and validation passes.
