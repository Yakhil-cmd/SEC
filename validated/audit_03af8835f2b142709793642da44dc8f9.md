Audit Report

## Title
Unbounded Synchronous Neuron Iteration in `compute_ballots_for_new_proposal` Can Exhaust Instruction Budget, Permanently Blocking SNS Proposal Creation - (File: rs/sns/governance/src/governance.rs)

## Summary

`compute_ballots_for_new_proposal` performs a fully synchronous, unbounded O(n) scan over every neuron in `self.proto.neurons` within a single message execution. As the neuron count grows through swap participation and neuron splits, this loop can exhaust the IC per-message instruction limit (~5 billion on application subnets), causing every subsequent `make_proposal` call to trap permanently. NNS Governance already recognized and mitigated the same pattern via `noop_self_call_if_over_instructions`; SNS Governance has no equivalent guard.

## Finding Description

At `rs/sns/governance/src/governance.rs` lines 5255–5280, `compute_ballots_for_new_proposal` iterates unconditionally over `self.proto.neurons` (an in-memory `BTreeMap<String, Neuron>`):

```rust
for (k, v) in self.proto.neurons.iter() {
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power(...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { ... });
}
```

This function is synchronous (no `async`/`await`, no yield points) and is called on every `make_proposal` invocation. There are no instruction-count checks, no chunking, and no timer-based continuation anywhere in the SNS governance path. The neuron map grows monotonically: each swap participant receives a basket of neurons (typically 5–7), and any holder can call `split_neuron` permissionlessly to increase the count further.

By contrast, `rs/nns/governance/src/voting.rs` uses `noop_self_call_if_over_instructions(SOFT_VOTING_INSTRUCTIONS_LIMIT, Some(HARD_VOTING_INSTRUCTIONS_LIMIT))` to break long voting loops across multiple messages. SNS Governance has no such mechanism.

Once the neuron count crosses the instruction-budget threshold, `make_proposal` traps on every invocation regardless of caller, proposal type, or neuron stake. The partial state is never committed, so the canister is not corrupted — but proposal creation is permanently unavailable.

## Impact Explanation

This is a **High** severity finding matching the allowed impact class: *"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant SNS... security impact with concrete user or protocol harm."*

Permanent inability to submit governance proposals freezes all SNS governance actions: no parameter changes, no treasury disbursements, no canister upgrades, and no emergency interventions can be enacted through the normal governance path. The SNS is effectively governance-bricked without any on-chain recovery path through proposals.

## Likelihood Explanation

An SNS swap with 10,000 direct participants each receiving a 5-neuron basket produces 50,000 neurons at swap close — a realistic scale for popular SNS projects. Subsequent permissionless neuron splits push the count higher. The `max_number_of_neurons` governance parameter provides a nominal ceiling but its default is on the order of 200,000, far above the exhaustion threshold. No adversarial intent is required; normal participation in a successful SNS swap is sufficient to approach or exceed the limit. Any neuron holder (unprivileged) triggers the condition simply by calling `make_proposal`.

## Recommendation

Mirror the NNS Governance approach: introduce chunked ballot construction with timer-based continuation using `noop_self_call_if_over_instructions`, storing partial electoral roll state between slices. Alternatively, maintain a pre-computed eligible-voter index updated incrementally on neuron state changes, so `make_proposal` reads a pre-built structure rather than scanning all neurons. As an immediate mitigation, lower the enforced `max_number_of_neurons` limit to a value benchmarked to stay within the instruction budget for a single message.

## Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and a low `min_participant_icp_e8s`.
2. Complete the SNS swap with ≥ 10,000 direct participants (producing ≥ 50,000 neurons via the standard neuron basket).
3. Optionally have holders call `split_neuron` to push the count higher.
4. Have any neuron holder submit a valid `Motion` proposal via `make_proposal`.
5. Observe the call trap with `CanisterInstructionLimitExceeded` due to `compute_ballots_for_new_proposal` exhausting the ~5 billion instruction budget.
6. Confirm the condition is permanent: no subsequent `make_proposal` call from any caller succeeds.
7. A deterministic integration test or canbench benchmark (analogous to `rs/nns/governance/canbench/`) measuring instructions consumed by `compute_ballots_for_new_proposal` at 50,000, 100,000, and 200,000 neurons provides reproducible proof without mainnet deployment.
