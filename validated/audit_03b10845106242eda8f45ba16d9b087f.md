### Title
Disordered EVM Refund Cap Calculation Causes Users to Overpay Gas When Native Resource Consumption Exceeds EVM Gas - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the EIP-3529 EVM refund cap (`gas_used / 5`) is computed before the `delta_gas` native-resource adjustment is applied. When `delta_gas > 0` (native resource consumption exceeds EVM gas consumption), the final `gas_used` is higher than the value used to compute the cap, making the cap artificially small. Users with SSTORE refunds and high native resource consumption receive less refund than they are entitled to, overpaying gas to the operator.

---

### Finding Description

`compute_gas_refund` in `refund_calculation.rs` performs two sequential corrections to `gas_used`:

1. **EVM refund** (EIP-3529 SSTORE refund, capped at `gas_used / 5`)
2. **`delta_gas` adjustment** (adds native resource overhead to `gas_used` when native consumption exceeds EVM gas)

The current execution order is:

```
// Step 1: compute gas_used from remaining ergs
gas_used = gas_limit - remaining_ergs_as_gas          // e.g. 100,000

// Step 2: cap EVM refund at gas_used / 5  ← uses pre-delta gas_used
max_refund = gas_used / 5                              // e.g. 20,000
evm_refund = min(full_refund_gas, max_refund)          // e.g. 20,000

// Step 3: apply EVM refund
gas_used -= evm_refund                                 // 80,000

// Step 4: apply minimal_gas_used floor
gas_used = max(gas_used, minimal_gas_used)

// Step 5: compute native_used and delta_gas
native_used = full_native_limit - remaining_native     // e.g. 200,000
delta_gas = (native_used / native_per_gas) - gas_used  // 200,000 - 80,000 = 120,000

// Step 6: apply delta_gas
if delta_gas > 0: gas_used += delta_gas                // 200,000
``` [1](#0-0) 

Because the EVM refund cap is computed at step 2 using the pre-`delta_gas` `gas_used`, but the final `gas_used` after step 6 is `native_used / native_per_gas` (which is larger), the cap is too small. The user receives a refund capped at `gas_used_from_ergs / 5` instead of the correct `gas_used_final / 5`.

The `delta_gas` path is active whenever `native_per_gas != 0` and native resource consumption exceeds EVM gas consumption — a normal condition for transactions that do significant proving work (cryptographic precompiles, many storage writes generating pubdata). [2](#0-1) 

The resulting `gas_used` is returned in `RefundInfo` and directly drives the token transfer to the operator and the refund to the sender: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug — order-of-operations error in fee/refund calculation.

**Impact:** Users who both (a) generate SSTORE refunds (e.g., by zeroing previously non-zero storage slots) and (b) trigger `delta_gas > 0` (native resource consumption exceeds EVM gas) are charged more gas than they should be. The excess is transferred to the operator (coinbase). This is a direct, quantifiable loss of user funds on every such transaction.

**Concrete example:**
- `gas_limit = 1,000,000`, `gas_used_from_ergs = 100,000`, `native_per_gas = 1`, `native_used = 200,000`, `full_refund_gas = 30,000`
- **Current (buggy):** `evm_refund = min(30,000, 20,000) = 20,000`; final `gas_used = 200,000`. User pays for 200,000 gas.
- **Correct:** `delta_gas` applied first → `gas_used = 200,000`; `evm_refund = min(30,000, 40,000) = 30,000`; final `gas_used = 170,000`. User pays for 170,000 gas.
- **Overpayment:** 30,000 gas × `gas_price` per transaction.

---

### Likelihood Explanation

The condition `delta_gas > 0` is a normal operating condition in ZKsync OS. The double resource accounting model is designed so that native resource consumption can exceed EVM gas consumption, and `delta_gas` is the mechanism to reconcile this. Any transaction that:
- Performs native-resource-intensive operations (cryptographic precompiles, many storage writes for pubdata), AND
- Has SSTORE refunds (e.g., zeroing storage slots)

will trigger this bug. Both conditions are reachable by any unprivileged transaction sender with no special access required.

---

### Recommendation

Compute `delta_gas` and apply it to `gas_used` **before** computing the EVM refund cap. The corrected order in `compute_gas_refund`:

```rust
// 1. Compute gas_used from ergs
let mut gas_used = gas_limit - remaining_ergs_as_gas;

// 2. Apply minimal_gas_used floor
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

// 3. Compute and apply delta_gas FIRST
let native_used = full_native_limit - resources.native().remaining().as_u64();
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}

// 4. NOW compute EVM refund cap based on final gas_used
let evm_refund = {
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};
gas_used -= evm_refund;
```

This ensures the EIP-3529 cap is proportional to the total gas actually charged to the user, including native resource overhead.
