### Title
Silo Whitelist Bypass: `assert_access` Checks Sender But Not Recipient, Enabling Permanent Fund Freeze - (File: `engine/src/engine.rs`)

### Summary

In Silo mode, the `assert_access` function enforces whitelist checks only on the **transaction sender** (`transaction.address` and `env.predecessor_account_id()`), but never on the **recipient** (`transaction.to`). A whitelisted address can freely transfer ETH or ERC-20 tokens to any non-whitelisted address. Because non-whitelisted addresses cannot submit transactions, any funds they receive are permanently frozen.

### Finding Description

`assert_access` in `engine/src/engine.rs` is the sole gate for Silo whitelist enforcement on submitted EVM transactions:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
``` [1](#0-0) 

`transaction.address` is the **sender's** EVM address (recovered from the ECDSA signature). `transaction.to` is used only to branch between `is_allow_submit` and `is_allow_deploy` — it is never itself checked against the `Address` whitelist.

The Silo module exposes `is_allow_receive_erc20_tokens` specifically for checking whether a recipient is whitelisted:

```rust
/// Check if a user has the right to receive erc20 tokens.
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [2](#0-1) 

This function is only invoked in the NEP-141 bridge deposit path, not in the general EVM transaction path. For any standard EVM call or ETH transfer, the recipient address is never validated against the `Address` whitelist.

The `Address` whitelist, when enabled, gates all transaction submission:

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [3](#0-2) 

A non-whitelisted address that receives funds cannot pass `assert_access` and therefore cannot spend those funds.

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a whitelisted address sends ETH or ERC-20 tokens to a non-whitelisted address (e.g., via a plain ETH transfer or an ERC-20 `transfer(to, amount)` call), the recipient holds a balance but is blocked from submitting any transaction by `assert_access`. Without admin intervention to whitelist the recipient, those funds are permanently frozen. The Silo whitelist is designed to be a closed permissioned environment; this gap allows value to flow into addresses that are structurally unable to move it out.

### Likelihood Explanation

**High.** Any whitelisted participant can trigger this unilaterally with a standard ETH transfer or ERC-20 transfer call. No special privileges, no admin compromise, and no external dependency is required. The attacker-controlled entry path is a normal signed EVM transaction submitted via the `submit` entrypoint. The only precondition is that the sender is whitelisted, which is the normal operating state for any legitimate Silo participant.

### Recommendation

1. In `assert_access`, when `transaction.to` is `Some(recipient)`, additionally call `silo::is_allow_receive_erc20_tokens` (or an equivalent check) on the recipient address before allowing the transaction to proceed.
2. Alternatively, enforce the recipient whitelist check inside the EVM execution layer at the point of balance transfer, so that ERC-20 `transfer` calls to non-whitelisted addresses are also rejected.
3. Ensure `is_allow_receive_erc20_tokens` is consistently applied across all token-movement paths, not only the NEP-141 bridge deposit path.

### Proof of Concept

1. Silo mode is active; `Address` and `Account` whitelists are enabled.
2. `alice.near` / EVM address `0xAlice` is whitelisted; `0xBob` is not.
3. `alice.near` submits a signed EVM transaction: `to = 0xBob`, `value = 1 ETH`.
4. `assert_access` checks `0xAlice` (sender) — passes. `0xBob` (recipient) — never checked.
5. The transaction executes; `0xBob` now holds 1 ETH.
6. `0xBob` attempts to submit any transaction. `assert_access` checks `0xBob` against the `Address` whitelist — fails with `NotAllowed`.
7. The 1 ETH in `0xBob` is permanently frozen with no recovery path absent admin action. [4](#0-3) [5](#0-4)
