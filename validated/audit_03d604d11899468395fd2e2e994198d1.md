Audit Report

## Title
Unbounded Per-Stream Heap Accumulation via Concurrent QUIC Streams Enables Replica OOM - (File: `rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
The IC QUIC transport layer accepts up to 1,000 concurrent bidirectional streams per peer connection and spawns an unbounded tokio task per stream, each calling `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)` where `MAX_MESSAGE_SIZE_BYTES = 128 MiB`. A TLS-authenticated Byzantine subnet peer can open the maximum number of streams and slowly stream data on each, causing the receiving replica to accumulate heap memory across all concurrent `Vec<u8>` buffers far beyond available RAM, triggering an OOM kill and node outage.

## Finding Description

**Root cause 1 — `MAX_MESSAGE_SIZE_BYTES = 128 MiB`:** [1](#0-0) 

**Root cause 2 — `MAX_CONCURRENT_BIDI_STREAMS = 1,000` and `RECEIVE_WINDOW = 200 MiB`:** [2](#0-1) 

**Root cause 3 — Unbounded `JoinSet` with no per-connection task cap:** [3](#0-2) 

The `start_stream_acceptor` loop accepts every incoming bidirectional stream and immediately spawns a task into `inflight_requests` with no bound on the number of concurrent tasks: [4](#0-3) 

**Root cause 4 — Each task calls `read_to_end(MAX_MESSAGE_SIZE_BYTES)`:** [5](#0-4) 

`quinn::RecvStream::read_to_end` accumulates all received bytes into a heap-allocated `Vec<u8>` before returning. As the receiver reads data from each stream into the Vec, the QUIC implementation sends window-update frames to the sender, replenishing the receive window and permitting more data to flow. The connection-level `RECEIVE_WINDOW = 200 MiB` limits data *in-flight at any instant*, but does not cap the *total heap memory accumulated* across all concurrent `read_to_end` calls over time: each Vec grows continuously as data arrives and windows are replenished. With 1,000 concurrent stream tasks each accumulating up to 128 MiB, the theoretical heap ceiling is **128 GiB** from a single peer connection.

The same unbounded accumulation pattern exists on the response path: [6](#0-5) 

**Why existing guards are insufficient:**
- `STREAM_RECEIVE_WINDOW = 4 MiB` and `RECEIVE_WINDOW = 200 MiB` throttle instantaneous in-flight bytes, not total heap allocated across all concurrent tasks.
- The `JoinSet` has no capacity limit; the code comment at line 52–56 explicitly acknowledges this design gap and notes that the stream limit is the only backstop.
- There is no per-peer memory budget or back-pressure mechanism.

## Impact Explanation

A Byzantine subnet peer crashing a replica via OOM directly maps to the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** A crashed replica cannot participate in consensus, degrading subnet liveness. If multiple replicas on the same subnet are targeted simultaneously (each subnet peer can be attacked by any other peer), subnet liveness can be fully disrupted below the Byzantine fault threshold.

## Likelihood Explanation

The attacker must hold a valid TLS certificate as a subnet peer — a meaningful but in-scope constraint. No governance majority, admin key, or external oracle is required. The attack uses only standard QUIC client behavior (opening streams up to the protocol-enforced maximum and sending data at a controlled rate). The constants `MAX_CONCURRENT_BIDI_STREAMS = 1,000` and `MAX_MESSAGE_SIZE_BYTES = 128 MiB` are both reachable by any authenticated peer without special privilege. The attack is gradual and sustained, evading simple rate-limit heuristics. It is repeatable: after a replica restarts, the attacker can reconnect and repeat.

## Recommendation

1. **Cap the `JoinSet`**: Enforce a maximum number of concurrently inflight request tasks per connection (e.g., 50–100), back-pressuring `accept_bi` when the cap is reached. The existing comment at `request_handler.rs` lines 52–56 already identifies this as the correct fix direction.
2. **Reduce `MAX_MESSAGE_SIZE_BYTES`** or apply per-URI handler limits at the router layer rather than a single global ceiling shared across all protocol handlers.
3. **Enforce per-peer memory accounting**: Track total bytes currently buffered across all in-flight `read_to_end` calls for a given peer; reject or reset new streams when the per-peer budget is exceeded.
4. **Consider chunked/streaming reads** instead of `read_to_end` to avoid materializing the full message in heap before processing.

## Proof of Concept

```
1. Attacker is a TLS-authenticated subnet peer.
2. Establish a QUIC connection to the victim replica.
3. Open 1,000 bidirectional streams (MAX_CONCURRENT_BIDI_STREAMS limit).
4. On each stream, send a protobuf-encoded HttpRequest whose body is
   ~127 MiB of arbitrary bytes, paced to stay within QUIC flow-control
   windows (STREAM_RECEIVE_WINDOW=4 MiB, RECEIVE_WINDOW=200 MiB).
5. The victim spawns 1,000 tokio tasks, each calling:
       recv_stream.read_to_end(134_217_728 /* 128 MiB */).await
   Each task's Vec<u8> grows as data arrives and window updates are sent.
6. As each Vec is read into, QUIC window-update frames replenish the
   connection window, permitting the attacker to send more data.
7. Over time, each of the 1,000 Vecs grows toward 128 MiB.
   Total heap allocation approaches 128 GiB, triggering OOM.
8. The replica process is killed; the node drops out of consensus.

Verifiable with a local integration test:
- Spin up a test QuicTransport endpoint.
- Connect a test peer that opens 1,000 bidi streams and streams data
  on each at a controlled rate.
- Observe heap growth via /proc/<pid>/status VmRSS or tokio metrics
  until OOM or task count reaches 1,000.
```
