Audit Report

## Title
Unbounded Permanent Allowance Accumulation in `ALLOWANCES_MEMORY` Stable Storage Enables Ledger DoS - (File: `rs/ledger_suite/common/ledger_core/src/approvals.rs`, `rs/ledger_suite/icp/ledger/src/lib.rs`, `rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary

The `AllowanceTable::approve()` function stores allowances with `expires_at: None` permanently in `ALLOWANCES_MEMORY` without inserting any entry into the expiration queue. The `prune()` function operates exclusively on the expiration queue and returns immediately when the queue is empty, making it structurally incapable of reclaiming permanent allowances. There is no per-account or global cap on the number of stored allowances anywhere in the ledger suite. An unprivileged attacker paying only the transfer fee per call can inflate `ALLOWANCES_MEMORY` without bound, eventually exhausting the ledger canister's stable memory and halting all ledger operations.

## Finding Description

In `rs/ledger_suite/common/ledger_core/src/approvals.rs`, the `approve()` function at lines 265–275 inserts an expiry queue entry only when `expires_at` is `Some`:

```rust
if let Some(expires_at) = expires_at {
    table.allowances_data.insert_expiry(expires_at, key.clone());
}
table.allowances_data.set_allowance(key, Allowance { amount, expires_at, arrived_at: now });
```

When `expires_at` is `None`, the allowance is written to `ALLOWANCES_MEMORY` with no corresponding expiry queue entry. The `prune()` function at lines 372–399 iterates only the expiry queue:

```rust
match table.allowances_data.first_expiry() {
    Some((ts, _key)) => { if ts > now { return pruned; } }
    None => { return pruned; }  // exits immediately if queue is empty
}
```

If all stored allowances are permanent, `first_expiry()` returns `None` and `prune()` returns 0 immediately — it has no mechanism to enumerate or remove entries absent from the expiry queue.

Both ledgers back the allowance table with an uncapped `StableBTreeMap`:
- ICP ledger: `rs/ledger_suite/icp/ledger/src/lib.rs` lines 136–137
- ICRC-1 ledger: `rs/ledger_suite/icrc1/ledger/src/lib.rs` lines 526–527

A grep across the entire `rs/ledger_suite/` tree for `max_allowances` or `MAX_ALLOWANCES` returns zero matches. The only relevant constant, `MAX_TAKE_ALLOWANCES = 500` (`rs/ledger_suite/icrc1/ledger/src/lib.rs` line 74), limits query result size, not storage. The `icrc2_approve` handlers in both `rs/ledger_suite/icp/ledger/src/main.rs` (lines 1313–1425) and `rs/ledger_suite/icrc1/ledger/src/main.rs` (lines 820–903) perform no check on the current number of stored allowances before inserting a new one.

## Impact Explanation

This matches the allowed ICP bounty impact: **Application/platform-level DoS — High ($2,000–$10,000)**. The ICP and ICRC-1 ledgers are critical financial infrastructure. Exhausting their stable memory prevents all subsequent transactions, approvals, burns, and transfers. Even before full exhaustion, a very large `StableBTreeMap` degrades every operation that touches stable memory. The attack is triggered by an unprivileged ingress call with no special privileges required.

## Likelihood Explanation

Medium. Each `icrc2_approve` call costs the transfer fee (e.g., 10,000 e8s for ICP), which is non-recoverable. A single principal with many subaccounts (up to 2^256 per principal) can create distinct `(approver, spender)` pairs at scale without needing separate identities. The attacker can subsequently drain token balances from each subaccount via transfer, leaving zero-balance accounts with permanent allowances that impose no further cost. The attack is gradual and produces no anomalous individual call. There is no on-chain rate limiting or per-account allowance count cap to impede it.

## Recommendation

1. **Enforce a per-account cap** on active allowances: reject `icrc2_approve` if the approver already has N active allowances (e.g., N = 500, matching `MAX_TAKE_ALLOWANCES`).
2. **Add a global allowance count limit** as a configurable ledger parameter, analogous to `MAX_TRANSACTIONS_IN_WINDOW` for the deduplication window.
3. **Charge a higher fee for permanent allowances** (no `expires_at`) to make the attack economically prohibitive relative to the damage potential.
4. **Implement a cleanup function** that removes allowances whose approver account has a zero balance, callable by anyone, to provide a reclamation path for the zero-balance-approver scenario described in the PoC.

## Proof of Concept

1. Attacker controls principal `P` with subaccounts `sub_0 … sub_N`.
2. Attacker mints a small amount of ICP to each subaccount (enough to pay the approval fee once).
3. For each `sub_i`, attacker calls `icrc2_approve` with `expires_at: None` and a distinct spender `S_j`.
4. Each call succeeds: a permanent entry is written to `ALLOWANCES_MEMORY`; no expiry queue entry is created.
5. Attacker transfers the remaining balance out of each subaccount, leaving zero-balance accounts with permanent allowances.
6. `prune()` is called on every subsequent ledger operation but returns 0 immediately because the expiry queue is empty.
7. After `N × M` calls, `ALLOWANCES_MEMORY` contains `N × M` entries that can never be automatically reclaimed.
8. A deterministic integration test using `StateMachine` / PocketIC can verify this by: (a) creating many subaccounts, (b) calling `icrc2_approve` with `expires_at: None` from each, (c) draining balances, (d) calling `prune()` repeatedly and asserting the allowance count remains unchanged, and (e) asserting `ALLOWANCES_MEMORY` size grows monotonically.
