Audit Report

## Title
Missing Memory Size Bounds Validation on Sandbox-Provided `ExecutionFinishedRequest` Allows Memory Accounting Corruption — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

## Summary
A compromised canister sandbox process can forge an `ExecutionFinishedRequest` with an arbitrarily large `wasm_memory.size` or `stable_memory.size`. The replica controller's `update_execution_state` assigns these values directly to the canister's execution state without validating them against the canister's allocated memory limit. The only guard (`verify_size()`) checks the wrong direction and passes for inflated values. Developers explicitly acknowledge this gap in a TODO comment at the exact location, and the analogous instructions path has a clamp that memory lacks.

## Finding Description
In `update_execution_state` (`sandboxed_execution_controller.rs:1779, 1800`), the replica unconditionally assigns:

```rust
wasm_memory.size = execution_state_modifications.wasm_memory.size;
// ...
stable_memory.size = execution_state_modifications.stable_memory.size;
```

The TODO at lines 1772–1774 explicitly acknowledges: *"If a canister has broken out of wasm then it might have allocated more wasm or stable memory then allowed. We should add an additional check here..."*

The subsequent `verify_size()` call (`execution_state.rs:192–202`) only asserts `page_map_bytes <= memory_bytes`. An inflated `size` trivially satisfies this — the page map is smaller than the inflated declared size — so the error path is never triggered and the inflated `Memory` struct is returned in `CanisterStateChanges`.

`apply_canister_state_changes` (`common.rs:588–589`) then writes the inflated `wasm_memory` and `stable_memory` directly into `execution_state` with no post-hoc bounds check.

The IPC path is: sandbox process → `ControllerServiceImpl::execution_finished` (`controller_service_impl.rs:48–80`, which passes `exec_output` to the registered completion closure without inspection) → `process_completion` → `update_execution_state` → `apply_canister_state_changes`.

Contrast with the instructions path (`sandboxed_execution_controller.rs:1713–1724`), which explicitly comments *"If sandbox is compromised this value could be larger than the initial limit"* and clamps the value. No equivalent clamp exists for memory size.

`wasm_memory_usage()` (`execution_state.rs:426–429`) derives the canister's reported memory usage directly from `self.wasm_memory.size`, so the inflated value propagates into storage cost accounting, `update_on_low_wasm_memory_hook_status` (`common.rs:630`), and subnet-level memory tracking.

## Impact Explanation
This matches the **High** impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Memory corruption or execution integrity loss constrained to specific canisters, subnet conditions, or pre-existing properties."*

A forged `wasm_memory.size = NumWasmPages::MAX` causes the replica to record that the canister has consumed the maximum possible memory. Subnet-level memory invariants that depend on per-canister memory sizes become inconsistent, storage cost accounting is corrupted, and the subnet can be made to believe all available execution memory is exhausted — starving all other canisters of memory allocation for the remainder of the round or until state is corrected.

## Likelihood Explanation
The precondition is a code execution primitive inside a canister sandbox process, requiring exploitation of a Wasmtime vulnerability. This is a non-trivial prerequisite. However, the IC sandbox is explicitly designed as a security boundary against exactly this scenario — the developer TODO and the instructions-guard comment both confirm this threat model is intended to be defended against. The asymmetry (instructions are guarded, memory is not) is a concrete, auditable gap, not a theoretical one. Once a sandbox escape is achieved, the memory inflation is trivially reproducible and deterministic.

## Recommendation
Apply the same pattern used for instructions at `sandboxed_execution_controller.rs:1713–1724`. After receiving `execution_state_modifications`, clamp `wasm_memory.size` and `stable_memory.size` to the pre-execution values (or the maximum allowed by the canister's memory allocation). Replace the TODO comment with an actual bounds check:

```rust
// Guard against compromised sandbox inflating memory size.
if execution_state_modifications.wasm_memory.size > execution_state.wasm_memory.size {
    // log critical error and clamp or reject
    execution_state_modifications.wasm_memory.size = execution_state.wasm_memory.size;
}
```

A stricter check would validate against the canister's configured memory allocation limit. The metric `sandboxed_execution_critical_error_invalid_memory_size` already exists and should be incremented on violation.

## Proof of Concept
1. Deploy a canister that exploits a Wasmtime sandbox escape to gain arbitrary code execution within the sandbox process.
2. From within the sandbox process, forge an `ExecutionFinishedRequest` with `exec_output.state.execution_state_modifications.wasm_memory.size = NumWasmPages::MAX`.
3. Send it over the controller IPC socket using the existing `ControllerClientStub::execution_finished` path.
4. `ControllerServiceImpl::execution_finished` (`controller_service_impl.rs:74–76`) passes it to the registered completion closure without inspection.
5. `update_execution_state` assigns the inflated size at line 1779; `verify_size()` passes because `page_map_bytes < inflated memory_bytes`.
6. `apply_canister_state_changes` writes the inflated `wasm_memory` into `execution_state` at `common.rs:588`.
7. Assert `execution_state.wasm_memory.size == NumWasmPages::MAX` — memory accounting is now corrupted.
8. Observe that subsequent calls to `wasm_memory_usage()` return a value derived from `NumWasmPages::MAX`, and that subnet available memory tracking reflects the inflated usage, blocking memory allocation for other canisters.

A deterministic integration test can be written without a real sandbox escape by directly constructing a forged `CanisterStateChanges` with `wasm_memory: Memory::new(PageMap::new_for_testing(), NumWasmPages::MAX)` and passing it to `apply_canister_state_changes`, asserting the resulting `execution_state.wasm_memory.size` equals `NumWasmPages::MAX`.
