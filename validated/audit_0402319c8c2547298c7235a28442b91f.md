### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then calling the exit precompile via inline assembly. The return value of the assembly `call` is captured in a local variable (`res`) but is **never checked**. If the precompile call fails for any reason, the EVM sub-call silently returns 0, the burn is committed, and no cross-chain transfer occurs. The tokens are permanently destroyed with no corresponding credit on NEAR or Ethereum.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — silent failure
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum` in both contracts: [2](#0-1) [3](#0-2) [4](#0-3) 

The exit precompile (`ExitToNear` at `0xe9217bc7…`, `ExitToEthereum` at `0xb0bd02f6…`) can return a hard failure (`ExitError`) in several reachable conditions. In `native.rs`, `exit_erc20_token_to_near` calls `get_nep141_from_erc20`, which returns `ERR_TARGET_TOKEN_NOT_FOUND` if the ERC-20 address is not registered in the NEP-141 map:

```rust
fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    ...
}
``` [5](#0-4) 

Additional failure paths include: amount exceeding `u128::MAX` (`ERR_INVALID_AMOUNT`), malformed recipient account ID (`ERR_INVALID_RECEIVER_ACCOUNT_ID`), and input size violations (`ERR_INVALID_INPUT`). [6](#0-5) 

In standard EVM semantics, a failed sub-`call` returns 0 in the success flag but does **not** revert the calling frame. Because `res` is never inspected, the `_burn` that already executed is committed to state. The user's tokens are gone; no NEAR-side `ft_transfer` or Ethereum-side `withdraw` promise is ever created.

---

### Impact Explanation

**Permanent freezing / destruction of user funds (Critical).**

Any user who calls `withdrawToNear` or `withdrawToEthereum` when the precompile fails loses their tokens permanently. The burn is irreversible; there is no refund path because the failure is invisible to the Solidity layer.

---

### Likelihood Explanation

**Medium.** The most realistic trigger is a user interacting with an ERC-20 whose address has not yet been (or has been incorrectly) registered in the NEP-141 map. A user supplying an invalid NEAR account ID as `recipient` (e.g., an empty string, an account ID that is too long, or one containing illegal characters) also triggers the failure path. Both scenarios are reachable by any unprivileged EVM user with no special access.

---

### Recommendation

Check the return value of every assembly `call` to the exit precompiles and revert on failure, so the burn is rolled back atomically:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy an `EvmErc20` whose address is **not** registered in the engine's NEP-141 map (or use a valid registered ERC-20 but pass an invalid NEAR account ID as `recipient`).
2. Call `withdrawToNear(invalidRecipient, amount)`.
3. Observe: `_burn` succeeds (token balance decreases), the assembly `call` returns 0 (precompile error), but no revert occurs.
4. The user's `amount` tokens are permanently destroyed; no NEAR-side transfer is scheduled.

The structural analog to the reported issue is exact: in the Morpho finding, `_asset.approve(morphoBlue)` succeeds while `loanToken.transferFrom` later fails — two operations that must be consistent are decoupled. Here, `_burn` succeeds while the exit precompile call fails — the token destruction and the cross-chain credit are decoupled, with no atomicity guarantee enforced by the Solidity code.
