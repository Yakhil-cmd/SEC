### Title
SNS `neuron_minimum_dissolve_delay_to_vote_seconds` Not Enforced to Exceed Maximum Voting Period - (File: `rs/sns/governance/src/types.rs`)

### Summary
The SNS governance canister's `validate_neuron_minimum_dissolve_delay_to_vote_seconds` function does not enforce that `neuron_minimum_dissolve_delay_to_vote_seconds` exceeds the maximum possible proposal lifetime (`initial_voting_period_seconds + 2 * wait_for_quiet_deadline_increase_seconds`). This allows an SNS to be legally configured such that a neuron holder can start dissolving, fully exit their stake, and disburse tokens before any governance proposal targeting them can complete — a direct analog to the Audius `decreaseStakeLockupDuration ≤ votingPeriod` bypass.

### Finding Description

In `rs/sns/governance/src/types.rs`, the `validate_neuron_minimum_dissolve_delay_to_vote_seconds` function enforces only one constraint: the minimum dissolve delay must not exceed `max_dissolve_delay_seconds`. [1](#0-0) 

There is no check that `neuron_minimum_dissolve_delay_to_vote_seconds > initial_voting_period_seconds + 2 * wait_for_quiet_deadline_increase_seconds`.

The SNS parameter bounds allow:
- `initial_voting_period_seconds` floor = 1 day, ceiling = 30 days
- `wait_for_quiet_deadline_increase_seconds` floor = 1 second, ceiling = 30 days (constrained to ≤ half the initial period)
- `neuron_minimum_dissolve_delay_to_vote_seconds` has no explicit floor in the SNS validator [2](#0-1) 

The default values show `initial_voting_period_seconds = 4 days` and `neuron_minimum_dissolve_delay_to_vote_seconds = 6 months`, which are safe. But these are independently configurable via `ManageNervousSystemParameters` proposals, and the validator does not cross-check them. [3](#0-2) 

The NNS governance code explicitly acknowledges this invariant must hold: [4](#0-3) 

The NNS enforces it implicitly by bounding `neuron_minimum_dissolve_delay_to_vote_seconds` to a minimum of 14 days, which exceeds the NNS `wait_for_quiet_threshold_seconds` of 4 days. The SNS has no equivalent cross-parameter enforcement.

The `validate` function in `NervousSystemParameters` calls each validator independently with no cross-field check between dissolve delay and voting period: [5](#0-4) 

### Impact Explanation

An SNS configured with `neuron_minimum_dissolve_delay_to_vote_seconds ≤ initial_voting_period_seconds + 2 * wait_for_quiet_deadline_increase_seconds` enables the following attack:

1. A neuron holder with the minimum dissolve delay starts dissolving their neuron.
2. They submit a proposal (e.g., `TransferSnsTreasuryFunds` or a `GenericNervousSystemFunction` that penalizes them or transfers funds to themselves).
3. They cast their vote at proposal creation time (ballots are fixed at creation).
4. Their neuron fully dissolves before the proposal voting deadline.
5. They disburse their tokens and exit the SNS entirely.
6. The proposal may still pass or fail — but the actor has already exited with no stake at risk.

This bypasses the economic security assumption of the SNS: that neuron holders bear the consequences of governance outcomes they participate in. The SNS treasury (`TransferSnsTreasuryFunds`) is a concrete, high-value target reachable via this path. [1](#0-0) 

### Likelihood Explanation

Any SNS launched via `CreateServiceNervousSystem` or subsequently updated via `ManageNervousSystemParameters` can reach this misconfiguration. The SNS init validation (`rs/sns/init/src/lib.rs`) also only checks `neuron_minimum_dissolve_delay_to_vote_seconds ≤ max_dissolve_delay_seconds`, not against the voting period. [6](#0-5) 

An SNS developer who intentionally or accidentally sets a short minimum dissolve delay (e.g., 1 day) alongside a longer voting period (e.g., 4 days default) creates this window. The NNS proposal that creates the SNS passes through `validate_post_execution`, which calls the same insufficient validator. [7](#0-6) 

### Recommendation

Add a cross-field invariant check in `validate_neuron_minimum_dissolve_delay_to_vote_seconds` (and the equivalent in `rs/sns/init/src/lib.rs`):

```rust
let max_voting_period = initial_voting_period_seconds
    .saturating_add(wait_for_quiet_deadline_increase_seconds.saturating_mul(2));
if neuron_minimum_dissolve_delay_to_vote_seconds <= max_voting_period {
    return Err(format!(
        "neuron_minimum_dissolve_delay_to_vote_seconds ({}) must exceed the maximum \
         possible voting period ({} + 2*{} = {})",
        neuron_minimum_dissolve_delay_to_vote_seconds,
        initial_voting_period_seconds,
        wait_for_quiet_deadline_increase_seconds,
        max_voting_period,
    ));
}
```

This mirrors the fix applied in Audius PR #657, which enforced `decreaseStakeLockupDuration > votingPeriod + executionDelay`.

### Proof of Concept

Configure an SNS with:
- `neuron_minimum_dissolve_delay_to_vote_seconds` = 86400 (1 day)
- `initial_voting_period_seconds` = 86400 (1 day, the floor)
- `wait_for_quiet_deadline_increase_seconds` = 43200 (0.5 days, ≤ half of initial)

This passes all current SNS validation. Maximum voting period = 1 + 2×0.5 = 2 days.

Attack sequence:
1. Attacker holds a neuron with 1-day dissolve delay and significant voting power.
2. Attacker calls `StartDissolving` — neuron will be fully dissolved in 1 day.
3. Attacker submits a `TransferSnsTreasuryFunds` proposal to drain the treasury to their address.
4. Attacker votes `Yes` (ballot is locked at proposal creation with their full voting power).
5. After 1 day, neuron is dissolved; attacker calls `Disburse` and exits with their tokens.
6. The proposal is still open for up to 1 more day. If it passes (attacker had majority), treasury is drained. The attacker has no remaining stake to be penalized. [2](#0-1) [1](#0-0)

    /// Validates that the nervous system parameter
    /// neuron_minimum_dissolve_delay_to_vote_seconds is well-formed.
    fn validate_neuron_minimum_dissolve_delay_to_vote_seconds(&self) -> Result<(), String> {
        let max_dissolve_delay_seconds = self.validate_max_dissolve_delay_seconds()?;

        let neuron_minimum_dissolve_delay_to_vote_seconds = self
            .neuron_minimum_dissolve_delay_to_vote_seconds
            .ok_or_else(|| {
                "NervousSystemParameters.neuron_minimum_dissolve_delay_to_vote_seconds must be set"
                    .to_string()
            })?;

        if neuron_minimum_dissolve_delay_to_vote_seconds > max_dissolve_delay_seconds {
            Err(format!(
                "The minimum dissolve delay to vote ({neuron_minimum_dissolve_delay_to_vote_seconds}) cannot be greater than the max \
                dissolve delay ({max_dissolve_delay_seconds})"
            ))
        } else {
            Ok(())
        }
    }
