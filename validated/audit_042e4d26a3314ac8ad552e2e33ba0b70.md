### Title
Permanent ETH Freeze When `exit_to_near` NEAR-Side Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is absent, ETH transferred into the `exit_to_near` precompile address during an EVM exit operation is permanently locked if the downstream NEAR-side `ft_transfer` promise fails. There is no fallback debt-recovery or refund path, creating an irrecoverable insolvency between the EVM ETH balance and the NEP-141 connector supply.

---

### Finding Description

The `ExitToNear` precompile's `run()` method constructs a `callback_args` struct that conditionally includes a `refund` field:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the base-ETH exit path (non-wNEAR), `transfer_near_args` is also `None`, so `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The code then selects a bare `PromiseArgs::Create` with **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback scheduled
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

During EVM execution the caller's ETH is transferred to `exit_to_near::ADDRESS` (the precompile's own address) via `context.apparent_value`. If the NEAR-side `ft_transfer` promise subsequently fails (e.g., recipient account not registered, insufficient attached gas, connector balance drained), there is no scheduled callback to invoke `exit_to_near_precompile_callback`, and therefore no path to call `refund_on_error`:

```rust
} else {
    // ETH exit; transfer ETH back from precompile address
    let exit_address = exit_to_near::ADDRESS;
    ...
    engine.call(&exit_address, &refund_address, amount, ...)
}
``` [3](#0-2) 

The ETH remains permanently locked inside `exit_to_near::ADDRESS` with no on-chain mechanism to recover it. The `accounting.rs` layer explicitly permits `Net::Lost` silently:

```rust
// Net loss is possible if `SELFDESTRUCT(self)` calls are made.
accounting::Net::Zero | accounting::Net::Lost(_) => (),
``` [4](#0-3) 

This means the EVM-side ETH supply silently decreases while the NEAR connector's NEP-141 total supply remains unchanged — a permanent accounting divergence (insolvency).

The same issue applies to the wNEAR unwrap path: a callback IS scheduled, but when the `near_withdraw` promise fails, the callback reaches the `else if let Some(args) = args.refund` branch, finds `refund = None` (without `error_refund`), and silently returns `None` with no refund:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op; ETH stays locked
};
``` [5](#0-4) 

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH transferred to `exit_to_near::ADDRESS` during a failed exit is irrecoverable. No privileged function, admin call, or user-initiated transaction can drain that address once the callback window has passed. The EVM ETH supply and the NEAR connector NEP-141 supply diverge permanently, constituting connector insolvency. Any user who exits ETH to a NEAR account that is not registered with the connector (or any other failure condition) loses their ETH forever.

---

### Likelihood Explanation

**Medium-High.** The trigger condition — a failing NEAR-side `ft_transfer` — is reachable by any unprivileged EVM user:

- Sending to a NEAR account not registered with the NEP-141 connector (no storage deposit).
- Sending to a non-existent NEAR account.
- Attaching insufficient gas (the `FT_TRANSFER_GAS` constant is a fixed estimate, not guaranteed).
- Any transient NEAR runtime failure.

The `test_exit_to_near_eth_refund` test demonstrates the scenario by draining the Aurora connector balance before the exit call, causing the `ft_transfer` to fail. [7](#0-6) 

---

### Recommendation

1. **Enable `error_refund` unconditionally in production builds**, or promote it to a default feature in `engine-precompiles/Cargo.toml`.
2. If the feature must remain optional, add a compile-time `#[cfg(not(feature = "error_refund"))]` assertion or documentation warning that deployments without it permanently lose user ETH on NEAR-side failures.
3. Consider adding a privileged `recover_stuck_eth(address)` admin function that can drain `exit_to_near::ADDRESS` back to a specified recipient as a last-resort debt-recovery mechanism, analogous to the `repayDebt()` discussed in the original report.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Fund EVM address `A` with 1 ETH.
3. Call the `exit_to_near` precompile from `A` with 1 ETH, targeting a NEAR account `unregistered.near` that has no storage deposit with the NEP-141 connector.
4. The EVM transfers 1 ETH to `exit_to_near::ADDRESS`; a bare `PromiseArgs::Create(ft_transfer)` is scheduled with no callback.
5. The NEAR-side `ft_transfer` fails (recipient not registered).
6. No callback fires; `exit_to_near::ADDRESS` now holds 1 ETH permanently.
7. `A`'s EVM balance is 0; the NEP-141 connector total supply is unchanged; the 1 ETH is unrecoverable.

Relevant code path: `ExitToNear::run` → `PromiseArgs::Create` (no callback) → NEAR promise failure → no `refund_on_error` invoked. [8](#0-7) [9](#0-8) [10](#0-9)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
