### Title
Unchecked Return Value of Exit Precompile Call After Irreversible Token Burn - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary
Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens before making a low-level `call()` to the Aurora exit precompile. The return value of that `call()` is captured in a local assembly variable (`res`) but is never inspected. If the precompile call fails for any reason, the burn is not reverted and the user's tokens are permanently destroyed with no corresponding NEP-141 release on the NEAR side.

### Finding Description
In `withdrawToNear` and `withdrawToEthereum` of both `EvmErc20.sol` and `EvmErc20V2.sol`, the sequence is:

1. `_burn(_msgSender(), amount)` — irreversibly destroys the caller's ERC-20 balance.
2. A raw assembly `call()` is made to the `ExitToNear` or `ExitToEthereum` precompile address.
3. The return value `res` is stored in a local assembly variable and then silently discarded; no `require(res == 1)` or equivalent check is present.

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // <-- irreversible

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
```

The same pattern appears in `withdrawToEthereum` (lines 65-76) and in both functions of `EvmErc20V2.sol` (lines 53-77).

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can legitimately fail and return a non-success exit reason in several cases: the ERC-20 contract is not registered in the NEP-141 mapping (`get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND`), the recipient account ID is malformed, the amount exceeds `u128::MAX`, or the call runs out of gas. When the precompile returns failure, the EVM subcall returns `0` in `res`, but because `res` is never checked, execution continues and the transaction succeeds from the caller's perspective — with the tokens already burned.

### Impact Explanation
The user's ERC-20 tokens are permanently destroyed on Aurora (the `_burn` is committed as part of the successful outer transaction), while the corresponding NEP-141 tokens on NEAR are never transferred to the recipient. The bridged asset is lost with no recovery path. This constitutes **permanent freezing / destruction of user funds** (Critical).

### Likelihood Explanation
Any token holder can trigger this by calling `withdrawToNear` or `withdrawToEthereum` with a recipient string that causes the precompile to fail (e.g., an invalid NEAR account ID that passes Solidity-side length checks but fails the Rust-side `AccountId::try_from` parse inside `parse_recipient`). No special privileges are required. The call succeeds at the EVM level (burn committed), while the precompile silently fails.

### Recommendation
After the assembly `call()`, revert if `res` is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to all four affected assembly blocks in both `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions so the precompile call is made before the burn, and only burn on success.

### Proof of Concept

1. Deploy `EvmErc20` with a registered NEP-141 mapping.
2. Mint tokens to `alice`.
3. Alice calls `withdrawToNear` with a recipient string that is syntactically valid bytes but not a valid NEAR `AccountId` (e.g., `"INVALID!!ACCOUNT"`).
4. Inside the EVM: `_burn(alice, amount)` executes and commits.
5. The assembly `call()` to the `ExitToNear` precompile returns `0` because `parse_recipient` fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID`.
6. `res` is discarded; the Solidity function returns normally.
7. Alice's ERC-20 balance is zero; no NEP-141 transfer was scheduled; tokens are gone.

```rust
fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
```
