Audit Report

## Title
Rate Limiter Permanently Incremented Before Async Delivery in `ensure_balance` — (`rs/nns/cmc/src/main.rs`)

## Summary
In the Cycles Minting Canister, `ensure_balance` permanently records cycles in the sliding-window rate limiter and increments `total_cycles_minted` before the async delivery call is made. If the subsequent delivery fails (e.g., cycles ledger rejects the deposit, or all subnet canister-creation attempts fail), the limiter is not rolled back. An unprivileged user can exploit this during a cycles ledger upgrade window to exhaust the 150 T cycles/hour base minting budget with minimal cost (ICP is refunded minus ledger fees), causing a temporary DoS of all CMC minting operations for up to one hour.

## Finding Description
`ensure_balance` at `rs/nns/cmc/src/main.rs:2306–2325` performs three irreversible mutations synchronously:
1. `limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)` — permanently records `cycles_to_mint` in the sliding-window limiter (`rs/nns/cmc/src/limiter.rs:54`).
2. `state.total_cycles_minted += cycles_to_mint` — inflates the on-chain accounting counter.
3. `ic0_mint_cycles128(cycles_to_mint)` — mints cycles into the CMC's own balance.

`ensure_balance` is called at line 2151 inside `do_mint_cycles` and at line 2249 inside `do_create_canister`, both before the `.await` on the delivery call. In `do_mint_cycles`, the delivery is `call_with_payment128(cycles_ledger, "deposit", …).await` at line 2158. When this call fails, `process_mint_cycles` (lines 1975–1981) calls only `refund_icp` — there is no rollback of the limiter or `total_cycles_minted`. The same pattern applies in `deposit_cycles` (line 2117) used by `process_top_up`.

The integration test at `rs/nns/integration_tests/src/cycles_minting_canister.rs:1337–1350` confirms the delivery-failure path is reachable: sending an amount below the cycles ledger fee (`CYCLES_LEDGER_FEE = 100_000_000` cycles) causes a `NotifyError::Refunded` response, proving `ensure_balance` ran and the limiter was updated before the ledger rejected the deposit.

## Impact Explanation
The base minting limit is `CYCLES_MINTING_LIMIT = 150e15` cycles per hour. At current ICP/cycles conversion rates (~100 T cycles per ICP), an attacker needs approximately 1.5 ICP worth of failed `notify_mint_cycles` calls to exhaust the hourly budget. Once exhausted, every subsequent legitimate call to `notify_mint_cycles`, `notify_top_up`, and `notify_create_canister` is rejected with a rate-limit error for up to one hour. This constitutes an application/platform-level DoS of a core NNS financial integration canister, matching the High impact class: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The primary realistic trigger is the cycles ledger upgrade window: during a canister upgrade, the cycles ledger temporarily rejects calls. Any user who calls `notify_mint_cycles` during this window causes `ensure_balance` to run and permanently consume limiter budget. The cycles ledger is upgraded periodically via NNS proposals, making this window predictable and publicly observable. The attacker's ICP is refunded (minus ~0.0001 ICP ledger fee per call), so the net cost to exhaust the 150 T/hr limit is approximately 0.0002 ICP in fees. No special privileges are required.

## Recommendation
Move `ensure_balance` to execute only after the async delivery call succeeds, mirroring the existing `burn_and_log` pattern which is already called only on success. In `do_mint_cycles`, restructure so the `call_with_payment128` deposit call is attempted first, and `ensure_balance` (or an equivalent record-only function) is called only on `Ok`. In `do_create_canister`, call `ensure_balance` only after a subnet successfully returns a canister ID. If pre-minting into the CMC balance is architecturally required, add an explicit rollback path that decrements the limiter's `total_count` and `state.total_cycles_minted` when delivery fails.

## Proof of Concept
**Deterministic integration test plan (safe, local state machine):**

1. Configure a local NNS state machine with CMC and cycles ledger.
2. Record the initial limiter count via a query.
3. Stop the cycles ledger canister (simulating an upgrade window).
4. Call `notify_mint_cycles` with 1 ICP; observe `NotifyError::Refunded` response and ICP refund.
5. Query the limiter count — assert it increased by ~100 T cycles despite no cycles being delivered.
6. Call `notify_mint_cycles` with another 1 ICP; observe second refund.
7. Restart the cycles ledger canister.
8. Call `notify_mint_cycles` with a legitimate 1 ICP — assert it fails with the rate-limit error ("More than 150000000000000000 cycles have been minted in the last … seconds, please try again later"), confirming the budget is exhausted and legitimate users are blocked.
