### Title
Permanent Fund Freeze When `ExitToNear` Promise Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is absent, the `ExitToNear` precompile schedules an `ft_transfer` or `ft_transfer_call` promise to the eth-connector with **no failure callback**. If that promise fails, the user's ETH (held at the precompile address) or burned ERC-20 tokens are permanently unrecoverable — a direct structural analog to the TON `raw_reserve`-before-gas-check pattern.

### Finding Description

The `ExitToNear::run` method in `engine-precompiles/src/native.rs` constructs `callback_args` conditionally on the `error_refund` feature flag:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the ETH base-token exit path, `transfer_near_args` is also `None`, so `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The subsequent branch:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

emits a bare `PromiseArgs::Create` — no `exit_to_near_precompile_callback` is chained. If the `ft_transfer` promise to the eth-connector fails (e.g., recipient account not registered, connector balance insufficient), there is no handler to execute `engine::refund_on_error`.

The same applies to the ERC-20 exit path: the ERC-20 tokens are burned inside the EVM before the promise is dispatched, and without a callback, a failed `ft_transfer` leaves the supply permanently destroyed.

The `exit_to_near_precompile_callback` function does contain the refund logic, but it is only reachable when a callback was attached:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [3](#0-2) 

Without `error_refund`, `args.refund` is always `None`, so this branch is dead code even when a callback is present (wNEAR unwrap case).

The engine-tests explicitly document the loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

### Impact Explanation

**Permanent freezing of funds.** For ETH exits: the ETH is transferred from the user's EVM balance to the `exit_to_near::ADDRESS` precompile address at call time; if the subsequent `ft_transfer` fails with no callback, that ETH is stranded at the precompile address with no recovery path. For ERC-20 exits: the tokens are burned in the EVM before the promise is dispatched; a failed `ft_transfer` means the supply is destroyed on the EVM side and never credited on the NEAR side.

### Likelihood Explanation

The `ft_transfer` promise can fail for reasons entirely within a normal user's control or outside anyone's control: sending to an unregistered NEAR account, the eth-connector contract having insufficient NEP-141 balance, or transient NEAR runtime errors. Any EVM user invoking the `ExitToNear` precompile is exposed. The feature flag `error_refund` is a compile-time opt-in, not a default, making deployments without it the vulnerable configuration.

### Recommendation

- **Short term:** Enable the `error_refund` feature in all production builds, or unconditionally populate `callback_args.refund` so a failure callback is always attached when funds are at risk.
- **Long term:** Remove the `error_refund` feature gate entirely and make the refund callback mandatory for all exit paths. The callback overhead is negligible compared to the risk of permanent fund loss.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Alice holds 10 ETH in her EVM account on Aurora.
3. Alice calls the `ExitToNear` precompile (flag `0x0`) with `value = 10 ETH`, targeting `"unregistered.near"` as the NEAR recipient.
4. The EVM deducts 10 ETH from Alice's balance and transfers it to `exit_to_near::ADDRESS`.
5. The precompile emits `PromiseArgs::Create(ft_transfer(...))` — no callback.
6. The `ft_transfer` call to the eth-connector fails because `"unregistered.near"` is not registered.
7. No `exit_to_near_precompile_callback` fires. Alice's 10 ETH remains at the precompile address permanently. Alice's EVM balance shows 0; the NEAR recipient received nothing. [6](#0-5) [7](#0-6)

```rust
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
