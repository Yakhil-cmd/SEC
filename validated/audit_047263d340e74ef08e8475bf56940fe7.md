Audit Report

## Title
Unsolicited Pong Triggers Unconditional Disconnect-Reconnect Loop, Enabling ckBTC Block Ingestion DoS — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary
`process_pong_message` calls `conn.disconnect()` whenever a `Pong` arrives while `PingState::Idle`, regardless of whether the pong was solicited. Because `disconnect()` (not `discard()`) is used, `reap_disconnected` returns the peer's address to `known_addresses` rather than removing it, and `manage_connections` immediately reconnects to the same peer. A malicious Bitcoin node that repeatedly sends unsolicited `Pong` messages can permanently occupy one or more connection slots in a tight disconnect-reconnect cycle, keeping the adapter below `min_connections` and stalling ckBTC block delivery.

## Finding Description
In `process_pong_message`, the `PingState::Idle` branch unconditionally sets `valid_pong = false`: [1](#0-0) 

This causes `conn.disconnect()` to be called, which sets the connection state to `NodeDisconnected` and aborts the stream task: [2](#0-1) 

In `reap_disconnected`, a `NodeDisconnected` state triggers `address_book.remove_from_active()`: [3](#0-2) 

`remove_from_active` moves the address from `active_addresses` back into `known_addresses` (for `AddressEntry::Discovered` entries), making it immediately eligible for reconnection: [4](#0-3) 

This is the critical difference from `discard()`, which removes the address entirely when seeds are available: [5](#0-4) 

After reaping, `manage_connections` refills connections up to `max_connections`: [6](#0-5) 

`make_connection` calls `address_book.pop()` for discovered addresses, which randomly selects from `known_addresses` — including the just-returned malicious peer address. With 2+ malicious peers in the address book, the adapter can be kept permanently below `min_connections = 2`: [7](#0-6) 

## Impact Explanation
This is a **High** severity finding matching the "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS" impact class. The ckBTC minter relies on the Bitcoin adapter to ingest blocks for UTXO tracking. Keeping the adapter below `min_connections` stalls block delivery, halting ckBTC minting and redemption operations. The disruption is persistent and self-sustaining as long as the malicious peers remain in the address book.

## Likelihood Explanation
The attacker must control Bitcoin nodes that the adapter connects to. This is realistic: the adapter discovers peers via DNS seeds and `addr` messages from connected nodes. A malicious operator can run standard Bitcoin nodes that complete the version handshake normally and then send a `Pong` message. The Bitcoin P2P protocol places no restriction on sending `Pong` at any time. No privileged access is required — only the ability to run a Bitcoin node and have its address propagated into the adapter's address book. The attack is repeatable indefinitely with zero cost per iteration after initial setup.

## Recommendation
Replace `conn.disconnect()` in the `PingState::Idle` branch with a no-op. An unsolicited `Pong` is not a protocol violation; only a `Pong` with a mismatched nonce when one was expected (`PingState::ExpectingPong`) indicates misbehavior warranting disconnection:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong — not a protocol violation; ignore.
    trace!(self.logger, "Received unsolicited pong from {}", address);
    return Ok(());
}
```

Alternatively, restructure the logic so `conn.disconnect()` is only called when `PingState::ExpectingPong` and the nonce does not match.

## Proof of Concept
1. Run a Bitcoin node that completes the version handshake with the adapter (sends `Version` + `Verack`), reaching `ConnectionState::HandshakeComplete` with `PingState::Idle`.
2. From the peer side, send `NetworkMessage::Pong(42)` (any nonce) without having received a `Ping`.
3. `process_pong_message` is called → `PingState::Idle` branch → `valid_pong = false` → `conn.disconnect()` → state becomes `NodeDisconnected`.
4. On the next `manage_connections` tick: `reap_disconnected` calls `remove_from_active`, returning the peer address to `known_addresses`; then `make_connection` reconnects to the same peer.
5. Repeat step 2 indefinitely. The connection slot cycles in a tight disconnect-reconnect loop.
6. With 2 such peers, the adapter stays below `min_connections = 2`, stalling ckBTC block ingestion.

A deterministic unit test can be written against `ConnectionManager` using the existing test infrastructure in `connectionmanager.rs` by injecting a `NetworkMessage::Pong` into a connection in `HandshakeComplete`/`Idle` state and asserting that `address_book.known_addresses` contains the peer address after `reap_disconnected` runs.
