### Title
Persistent `DownloadChunkError::Overloaded` (HTTP 429) Never Evicts Peer, Enabling Indefinite State Sync Stall — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

---

### Summary

A Byzantine peer that persistently returns HTTP 429 (`TOO_MANY_REQUESTS`) is never removed from `active_downloads`. Because the loop's only self-termination condition is `active_downloads.is_empty()`, a lagging replica whose sole connected peer behaves this way will spin indefinitely — re-queuing the same chunk, re-dispatching it to the same peer, with no backoff and no eviction — violating the liveness invariant that state sync must eventually complete or terminate.

---

### Finding Description

**HTTP 429 → `Overloaded` mapping** is explicit in `parse_chunk_handler_response`: [1](#0-0) 

**`Overloaded` handling in `handle_downloaded_chunk_result`** re-queues the chunk but, unlike `NoContent` and `RequestError`, does **not** call `active_downloads.remove(&peer_id)`: [2](#0-1) 

Compare with the eviction that **does** happen for `NoContent` and `RequestError`: [3](#0-2) 

**`download_failed` simply pushes the chunk back onto the queue** with no counter, no cooldown, no deduplication: [4](#0-3) 

**The loop's only self-exit path** is `active_downloads.is_empty()`: [5](#0-4) 

Because `Overloaded` never removes the peer, `active_downloads` never empties, and the loop never self-terminates. There is no wall-clock timeout on the overall sync session anywhere in `ongoing.rs`.

**The per-chunk `CHUNK_DOWNLOAD_TIMEOUT` of 10 s** exists: [6](#0-5) 

but a `Timeout` result is handled identically to `Overloaded` — same arm, same no-eviction path — so a peer that simply never responds also triggers the same infinite loop.

The existing test `test_cancel_if_running` in `ongoing.rs` already uses a `MockTransport` that always returns 429 and confirms the sync does **not** self-terminate; it only exits when `shutdown.shutdown()` is called externally: [7](#0-6) 

---

### Impact Explanation

A lagging replica that connects to a Byzantine peer before any honest peer has advertised the same state will have that Byzantine peer as the sole entry in `active_downloads`. The event loop then spins at full speed (one 10-second timeout per chunk, 10 parallel slots) re-dispatching the same chunk to the same peer forever. The replica cannot catch up to the current certified state, permanently degrading its participation in consensus. Even when honest peers later advertise and are added, the Byzantine peer remains in the pool and continues to consume download slots, slowing progress proportionally.

---

### Likelihood Explanation

The `ADVERT_BROADCAST_INTERVAL` is 5 seconds: [8](#0-7) 

In normal operation, honest peers will advertise within seconds and be added to `active_downloads`, breaking the sole-peer condition. However, the window is real and exploitable: a Byzantine peer that is the first to advertise (triggering `maybe_start_state_sync`) and that operates in a degraded network where honest peers are slow to connect can hold the sole-peer position for an extended or indefinite period. No privileged access is required — any subnet peer can send adverts and control its own HTTP response codes.

---

### Recommendation

1. **Evict on repeated `Overloaded`**: Track a consecutive-failure counter per peer; after a configurable threshold (e.g., 5 consecutive `Overloaded` or `Timeout` results), call `active_downloads.remove(&peer_id)` and reduce `allowed_downloads`.
2. **Exponential backoff**: Instead of immediately re-queuing via `download_failed`, place the chunk in a time-delayed retry queue so the loop does not spin at full CPU.
3. **Overall sync timeout**: Add a wall-clock deadline inside `run` (e.g., 10 minutes) after which the sync is cancelled and the `Chunkable` object is dropped, allowing a fresh sync attempt with a new peer set.
