Audit Report

## Title
`AmountTooLow` branch permanently burns ckBTC without reimbursement when UTXO pool is fragmented — (`rs/bitcoin/ckbtc/minter/src/lib.rs`, `rs/bitcoin/ckbtc/minter/src/fees/mod.rs`)

## Summary

`fee_based_minimum_withdrawal_amount` computes the minimum safe withdrawal amount using a hardcoded single-input vsize bound (`PER_REQUEST_VSIZE_BOUND = 221`). When the minter's UTXO pool is fragmented into many small UTXOs, the greedy selection algorithm picks far more inputs, causing `evaluate_transaction_fee` (which uses the real vsize) to return a fee exceeding the withdrawal amount. The resulting `BuildTxError::AmountTooLow` path in `submit_pending_requests` finalizes the request without calling `reimburse_canceled_requests`, permanently burning the user's ckBTC with no BTC sent and no refund.

## Finding Description

**Root cause — single-input vsize assumption:**
`fee_based_minimum_withdrawal_amount` in `rs/bitcoin/ckbtc/minter/src/fees/mod.rs` (L133–144) uses `PER_REQUEST_VSIZE_BOUND = 221` vbytes, modeling exactly one P2WPKH input. Each additional input adds ~68 vbytes to the signed transaction.

**Burn before transaction build:**
In `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` (L209–210), `burn_ckbtcs` is called and the ckBTC ledger burn is committed before the request is queued. The minimum-amount guard at L169–171 only checks against the single-input-based minimum.

**Greedy UTXO selection with fragmented pool:**
`greedy` in `rs/bitcoin/ckbtc/minter/src/lib.rs` (L1070–1099) accumulates UTXOs until the target is met, with no cap on input count (the `UTXOS_COUNT_THRESHOLD` consolidation path at L1049–1057 only adds extra UTXOs when the pool exceeds the threshold, it does not limit greedy selection). With N small UTXOs each worth slightly less than the withdrawal amount, all N are selected.

**Real vsize fee check:**
`evaluate_transaction_fee` at `rs/bitcoin/ckbtc/minter/src/fees/mod.rs` (L149–152) calls `fake_sign(tx).vsize()` — the actual vsize with all N inputs. At L1304–1308 of `lib.rs`, if `fee + minter_fee > amount`, `BuildTxError::AmountTooLow` is returned.

**No reimbursement on `AmountTooLow`:**
In `submit_pending_requests` (L412–434 of `lib.rs`), the `AmountTooLow` branch calls only `state::audit::remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. That function (L67–84 of `audit.rs`) records an event and pushes to `finalized_requests` — no reimbursement is scheduled. By contrast, the `InvalidTransaction` branch at L400–410 explicitly calls `reimburse_canceled_requests`. The `FinalizedStatus` enum (L259–267 of `state.rs`) has no reimbursement variant for `AmountTooLow`, confirming it is a terminal dead-end state.

## Impact Explanation

A user who submits a withdrawal at exactly `fee_based_retrieve_btc_min_amount` loses their ckBTC permanently when the minter's UTXO pool is fragmented: the ckBTC ledger burn is irreversible, no BTC transaction is ever sent, and no reimbursement is scheduled. This constitutes a concrete, unrecoverable loss of user funds in the ckBTC system, matching the allowed impact: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.**

## Likelihood Explanation

Any unprivileged user can fragment the minter's UTXO pool by depositing BTC in small amounts (each ≥ `deposit_btc_min_amount` ≈ 2,001 sats on mainnet). The consolidation mechanism only activates when `available_utxos.len() > UTXOS_COUNT_THRESHOLD` (typically 1,000+), so ~100 small UTXOs (well below the threshold) are sufficient to trigger the bug. At `fee_rate = 343,000` millisat/vbyte: 100 UTXOs × 2,001 sats → greedy selects all 100; vsize ≈ 6,905 vbytes; `bitcoin_fee` ≈ 2,368,415 sats >> 200,000-sat withdrawal. The attack cost (depositing ~200,100 sats + on-chain fees) is comparable to the victim's loss, making this a repeatable griefing vector. The condition is reachable without any special privileges.

## Recommendation

1. **Reimbursement parity (minimal fix):** The `AmountTooLow` branch in `submit_pending_requests` should call `reimburse_canceled_requests` (minus a small processing fee), matching the behavior of the `InvalidTransaction` branch at L400–410 of `lib.rs`.
2. **Multi-input-aware minimum:** `fee_based_minimum_withdrawal_amount` should account for the worst-case number of inputs the greedy algorithm might select for the given amount, or apply a conservative upper bound on input count.
3. **Lower consolidation threshold:** Reduce `UTXOS_COUNT_THRESHOLD` so UTXO fragmentation is addressed before it can affect minimum-amount withdrawals.

## Proof of Concept

```rust
// Construct a fragmented UTXO pool: 100 UTXOs of 2_001 sats each
let mut utxos: UtxoSet = (0..100u32)
    .map(|i| Utxo {
        outpoint: OutPoint { txid: [i as u8; 32].into(), vout: 0 },
        value: 2_001,
        height: 10,
    })
    .collect();

let fee_rate = FeeRate::from_millis_per_byte(343_000);
let fee_estimator = BitcoinFeeEstimator::new(Network::Mainnet, 100_000, 2_000);

// Minimum amount at this fee rate = 200_000 sats (single-input assumption)
let min_amount = fee_estimator.fee_based_minimum_withdrawal_amount(fee_rate);

// Build transaction at exactly the minimum — should succeed per the invariant
let result = build_unsigned_transaction(
    &mut utxos,
    vec![(BitcoinAddress::P2wpkhV0([1; 20]), min_amount)],
    &BitcoinAddress::P2wpkhV0([0; 20]),
    DEFAULT_MAX_NUM_INPUTS_IN_TRANSACTION,
    fee_rate,
    &fee_estimator,
);

// FAILS: returns Err(BuildTxError::AmountTooLow)
// 100 inputs × ~68 vbytes = ~6905 vbytes → fee ≈ 2.37M sats >> 200_000 sats
assert!(result.is_ok(), "invariant violated: {:?}", result);
```

A deterministic integration test can verify the full path: deposit 100 UTXOs of 2,001 sats each to the minter, submit a `retrieve_btc` request at `fee_based_retrieve_btc_min_amount`, advance the minter's heartbeat, and assert that `RetrieveBtcStatusV2` is `AmountTooLow` while the user's ckBTC balance has decreased with no BTC sent and no reimbursement minted.
