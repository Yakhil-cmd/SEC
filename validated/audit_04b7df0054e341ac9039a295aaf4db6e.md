### Title
Truncating Floor Division in `native_per_pubdata` Calculation Allows Pubdata to Be Underpriced to Zero - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` rate used to charge native resources for pubdata is computed with floor (truncating) integer division. When `pubdata_price < native_price`, the result truncates to zero, making pubdata completely free in native resource terms. Any unprivileged L2 transaction sender can exploit this to generate the maximum block pubdata without paying the operator-configured pubdata cost, bypassing the native resource accounting for pubdata entirely.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This is floor (truncating) division. By contrast, `native_per_gas` is computed with ceiling division on the very next lines:

```rust
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [2](#0-1) 

The same asymmetry is present in the public API helper `validate_l2_tx_intrinsic_native_resources`:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata == 0`, two downstream charging sites both produce zero cost:

1. **Intrinsic pubdata pre-charge** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [4](#0-3) 

2. **Execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ``` [5](#0-4) 

With `native_per_pubdata == 0`, the entire native budget is available for computation, and pubdata generation costs zero native resources regardless of how many bytes are written to state.

---

### Impact Explanation

When `pubdata_price < native_price` (a valid operator configuration, e.g. `pubdata_price = 5`, `native_price = 10`), `native_per_pubdata` truncates to `0`. An attacker submitting L2 transactions can:

- Generate up to the full block pubdata limit of state diffs, L2→L1 logs, and messages without paying any native resource cost for pubdata.
- Exhaust the block pubdata limit (`BlockPubdataLimitReached`) at zero marginal native cost, forcing the block to close early and denying inclusion to legitimate transactions.
- Cause state bloat and increased proving costs without proper compensation to the operator.

The only remaining protection is the block-level pubdata cap enforced in `check_for_block_limits`:

```rust
} else if !cfg!(feature = "resources_for_tester") && pubdata_used > system.get_pubdata_limit() {
    Err(InvalidTransaction::BlockPubdataLimitReached)
``` [6](#0-5) 

This cap limits the total damage per block but does not prevent the attack from being repeated across every block.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a normal operational state. The operator sets both values independently via block metadata. Any time `pubdata_price` is set below `native_price` — which is expected when pubdata is cheap relative to computation — `native_per_pubdata` truncates to zero. Any unprivileged L2 transaction sender can then generate pubdata at zero native cost by submitting transactions that write to storage (e.g., a simple contract loop writing to a mapping). No special privileges, leaked keys, or governance access are required.

---

### Recommendation

Replace the floor division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — can produce 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — always charges at least 1 native per pubdata byte when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427 for consistency between the API pre-validation and the bootloader execution path. [7](#0-6) 

---

### Proof of Concept

Set block context with `pubdata_price = 5`, `native_price = 10`, `eip1559_basefee = 1`. Then:

```
native_per_pubdata = 5.wrapping_div(10) = 0
```

Submit an L2 transaction that writes to 100 storage slots (generating ~3,300 bytes of pubdata). The `get_resources_to_charge_for_pubdata` call computes `3300 * 0 = 0` native charged for pubdata. The transaction succeeds with its full native budget intact, having generated pubdata at zero native cost. Repeat across every block to continuously fill pubdata capacity without paying the operator-configured pubdata price. [8](#0-7)
