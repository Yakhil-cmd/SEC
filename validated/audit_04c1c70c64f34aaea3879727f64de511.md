### Title
Pre-EIP-155 Legacy Transactions Bypass Chain ID Validation, Enabling Cross-Chain Replay - (File: engine/src/engine.rs)

### Summary
The chain ID check in `submit_with_alt_modexp` is guarded by `if let Some(chain_id)`, meaning it only fires when a chain ID is present in the transaction signature. Pre-EIP-155 legacy transactions (with `v = 27` or `v = 28`) produce `chain_id = None` and silently bypass the check. A transaction signed on any other EVM chain without EIP-155 protection can be replayed on Aurora by any unprivileged actor who can observe it on-chain.

### Finding Description
In `submit_with_alt_modexp` in `engine/src/engine.rs`, the chain ID validation reads:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

The guard `if let Some(chain_id)` means the rejection only triggers when a chain ID is embedded in the signature. When `chain_id` is `None`, the entire block is skipped and the transaction proceeds unconditionally.

`NormalizedEthTransaction.chain_id` is set from `LegacyEthSignedTransaction::chain_id()`:

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [2](#0-1) 

Specifically, `v = 27` and `v = 28` are accepted as valid (pre-EIP-155) by `sender()`:

```rust
27..=28 => (
    None,
    u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
),
``` [3](#0-2) 

These produce `chain_id = None` in `NormalizedEthTransaction`:

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for v=27/28
    ...
``` [4](#0-3) 

The `CHANGES.md` explicitly acknowledged this as a security issue and claimed it was fixed in v2.4.0:

> "Security improvement: Engine will no longer accept EVM transactions without a chain ID as part of their signature" [5](#0-4) 

The current production code does not enforce this: the `if let Some` pattern is a regression that re-opens the pre-EIP-155 replay window.

### Impact Explanation
An attacker who observes any pre-EIP-155 legacy transaction on Ethereum mainnet (or any other EVM chain) — all of which are publicly visible — can submit the identical raw bytes to Aurora's `submit` endpoint. If the transaction sender holds ETH on Aurora at the same address and their Aurora nonce matches the transaction nonce, the transaction executes on Aurora exactly as it did on the origin chain. Because the transaction payload is fixed (destination, value, calldata are all signed), the attacker can replay any pre-EIP-155 ETH transfer or contract call that moves funds to an address they control, resulting in direct theft of user funds.

**Impact: Critical — direct theft of user funds.**

### Likelihood Explanation
Pre-EIP-155 transactions are rare in modern tooling but are not impossible: old hardware wallets, legacy scripts, and certain DeFi protocols have historically produced them. All such transactions are permanently visible on public chains. The attacker requires no special privilege — only the ability to call the public `submit` endpoint (callable by any NEAR account). The nonce-matching constraint limits the attack to specific moments, but an attacker can monitor both chains and act opportunistically. The CHANGES.md confirms the project itself previously treated this as a security issue worth fixing.

### Recommendation
Replace the conditional chain ID check with an unconditional rejection of transactions that carry no chain ID:

```rust
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This restores the intent documented in `CHANGES.md` v2.4.0 and matches the comment on the `submit` export: "Must match `CHAIN_ID` to make sure it's signed for given chain vs replayed from another chain." [6](#0-5) 

### Proof of Concept
1. On Ethereum mainnet, locate any pre-EIP-155 transaction (v=27 or v=28) whose sender holds ETH on Aurora at the same address and whose nonce on Aurora equals the transaction nonce. Such transactions are indexed by any block explorer.
2. Extract the raw RLP-encoded transaction bytes.
3. Call Aurora's `submit` NEAR method with those bytes as input from any NEAR account.
4. The engine parses the transaction, recovers the sender via `ecrecover`, finds `chain_id = None`, skips the chain ID check at lines 1054–1059, and executes the transaction — transferring the sender's Aurora ETH to the destination encoded in the original Ethereum transaction.
