Audit Report

## Title
Unbounded Iteration Over Expired Allowances in `get_allowances` Violates `MAX_TAKE_ALLOWANCES` Cost Bound — (File: `rs/ledger_suite/icrc1/ledger/src/lib.rs`, `rs/ledger_suite/icp/ledger/src/lib.rs`)

## Summary
In `get_allowances` and `get_allowances_list`, the `max_results` break guard fires before the expiry `continue`, so expired allowances consume loop iterations without contributing to `result.len()`. An attacker who pre-populates N expired allowances for a single account forces every subsequent `icrc103_get_allowances` query for that account to scan all N entries, with iteration cost growing linearly in N and unbounded by `MAX_TAKE_ALLOWANCES = 500`. The same ordering flaw exists in both the ICRC1 and ICP ledger implementations.

## Finding Description
In `get_allowances` (`lib.rs:1194–1222`), the loop body executes in this order:

1. `continue` if this is the pagination start entry (lines 1198–1199)
2. **`break` if `result.len() >= max_results`** (lines 1201–1202)
3. `break` if past the `from` account owner (lines 1204–1205)
4. **`continue` if expired** (lines 1207–1210) — does NOT increment `result.len()`
5. `result.push(...)` for valid entries (line 1212)

Because step 4 fires after step 2, expired entries are skipped without being counted toward the result limit. The `max_results` break at step 2 can only fire once `result.len()` reaches the cap, which requires accumulating that many non-expired entries. With N expired entries preceding M non-expired entries (M ≥ max_results), the loop performs N + max_results iterations instead of the intended max_results iterations.

The identical ordering flaw exists in the ICP ledger's `get_allowances_list` (`rs/ledger_suite/icp/ledger/src/lib.rs:657–680`): the `result.len() >= max_results` break at line 664 precedes the expiry `continue` at line 670.

The `prune` function (`approvals.rs:372–399`) only removes expired entries with a bounded `limit` during `icrc2_approve` calls and does not eagerly clean up all expired entries for an account. The `approve` function rejects allowances already expired at submission time (`approvals.rs:247–248`), so the attacker must set `expires_at` slightly in the future and wait — a trivial constraint.

## Impact Explanation
`icrc103_get_allowances` is a `#[query]` endpoint callable by any principal without authentication. IC query calls are bounded by a 5-billion-instruction limit. Each stable-memory B-tree range iteration costs on the order of thousands of instructions. An attacker who accumulates hundreds of thousands of expired allowances for a single account can force every query call for that account to consume a disproportionate instruction budget, degrading or trapping legitimate queries for that account. This is a non-volumetric, persistent query exhaustion: a single well-crafted ledger state causes repeated query degradation without ongoing attacker participation. This matches the **High ($2,000–$10,000)** impact class: application/platform-level DoS on an in-scope financial ledger canister, not based on raw volumetric DDoS.

## Likelihood Explanation
The attack requires N distinct spender principals and paying a ledger fee per allowance — economically bounded but not prohibitive at scale. Once the expired allowances are written to stable memory, the attacker needs no further action. Pruning only occurs during `icrc2_approve` calls with a bounded limit, so expired entries persist indefinitely unless the account owner actively prunes them. The attack is repeatable and affects any caller querying the targeted account.

## Recommendation
Move the expiry check before the `max_results` check, or introduce a total-iterations counter that counts expired entries toward the loop budget:

```rust
// Option A: move expiry check before max_results check
if let Some(expires_at) = storable_allowance.expires_at
    && expires_at.as_nanos_since_unix_epoch() <= now
{
    continue;
}
if result.len() >= max_results as usize {
    break;
}
```

```rust
// Option B: add a total scan limit independent of result count
let mut scanned = 0usize;
const MAX_SCAN: usize = MAX_TAKE_ALLOWANCES as usize * 4;
for (...) {
    if scanned >= MAX_SCAN { break; }
    scanned += 1;
    ...
}
```

Option A is the simplest correct fix and should be applied to both `get_allowances` and `get_allowances_list`.

## Proof of Concept
1. Fund account `A` with sufficient tokens.
2. Create N allowances from `A` to N distinct spender accounts, each with `expires_at = now + 1_000_000_000` (1 second).
3. Wait 1 second for all allowances to expire.
4. Repeatedly call `icrc103_get_allowances({ from_account: A, prev_spender: None, take: None })`.
5. Measure instruction count via `ic_cdk::api::performance_counter(0)` or replica metrics.
6. Assert instruction count grows linearly with N, far exceeding the `MAX_TAKE_ALLOWANCES = 500` bound.
7. Compare against a baseline with 0 expired allowances and 500 non-expired ones — baseline uses ~500 iterations; attack case uses N + 500 iterations. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    /// Prunes allowances that are expired, removes at most `limit` allowances.
    pub fn prune(&mut self, now: TimeStamp, limit: usize) -> usize {
        self.with_postconditions_check(|table| {
            let mut pruned = 0;
            for _ in 0..limit {
                match table.allowances_data.first_expiry() {
                    Some((ts, _key)) => {
                        if ts > now {
                            return pruned;
                        }
                    }
                    None => {
                        return pruned;
                    }
                }
                if let Some((_, (account, spender))) = table.allowances_data.pop_first_expiry() {
                    let key = (account, spender);
                    if let Some(allowance) = table.allowances_data.get_allowance(&key)
                        && allowance.expires_at.unwrap_or_else(remote_future) <= now
                    {
                        table.allowances_data.remove_allowance(&key);
                        pruned += 1;
                    }
                }
            }
            pruned
        })
    }
