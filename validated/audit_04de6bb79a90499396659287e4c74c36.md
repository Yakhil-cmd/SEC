Audit Report

## Title
Unbounded Permanent Allowance Accumulation in `ALLOWANCES_MEMORY` Stable Storage — (`rs/ledger_suite/common/ledger_core/src/approvals.rs`, `rs/ledger_suite/icp/ledger/src/lib.rs`, `rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `AllowanceTable::approve()` function inserts an expiry queue entry only when `expires_at` is `Some`; allowances with `expires_at: None` are stored permanently with no corresponding expiry entry. The `prune()` function operates exclusively on the expiry queue and returns immediately when the queue is empty, making it structurally incapable of reclaiming permanent allowances. Both the ICP and ICRC-1 ledger canisters back the allowance table with an uncapped `StableBTreeMap` (`ALLOWANCES_MEMORY`), so an unprivileged caller can inflate this map without bound by repeatedly calling `icrc2_approve` with no expiration, eventually degrading or halting ledger approval functionality.

## Finding Description
In `rs/ledger_suite/common/ledger_core/src/approvals.rs` at L265–267, the expiry queue insertion is gated on `expires_at` being `Some`:

```rust
if let Some(expires_at) = expires_at {
    table.allowances_data.insert_expiry(expires_at, key.clone());
}
```

When `expires_at` is `None`, the allowance is written to `ALLOWANCES_MEMORY` at L268–275 with no corresponding expiry record. The `prune()` function at L372–399 iterates only the expiry queue: when `first_expiry()` returns `None` (because the queue is empty or contains only permanent allowances), it returns `0` immediately at L383–385 — it cannot touch any permanent entry. Both storage maps are initialized as unbounded `StableBTreeMap`s: ICP ledger at L136–137 of `rs/ledger_suite/icp/ledger/src/lib.rs`, and ICRC-1 ledger at L526–527 of `rs/ledger_suite/icrc1/ledger/src/lib.rs`. A grep across the entire ledger suite confirms no `MAX_ALLOWANCES`, per-account cap, or global allowance count limit exists anywhere. The only allowance-count constant, `MAX_TAKE_ALLOWANCES = 500` at L74 of `rs/ledger_suite/icrc1/ledger/src/lib.rs`, governs query result pagination, not storage.

Exploit path:
1. Attacker funds many subaccounts of a single principal with enough tokens to pay the approval fee (10,000 e8s each for ICP).
2. For each subaccount, attacker calls `icrc2_approve` with `expires_at: None` and a distinct spender.
3. Each call passes the fee check at L1369–1373 of `rs/ledger_suite/icp/ledger/src/main.rs`, burns the fee, and inserts a permanent entry into `ALLOWANCES_MEMORY`.
4. Attacker drains token balances via transfer, leaving zero-balance accounts with permanent allowances that `prune()` can never remove.
5. `ALLOWANCES_MEMORY` grows monotonically; as the stable BTreeMap fills the canister's stable memory allocation, new insertions fail, causing `icrc2_approve` and `icrc2_transfer_from` calls to trap.

## Impact Explanation
This is an application/platform-level DoS against the ICP and ICRC-1 ledger canisters — critical in-scope financial infrastructure. When `ALLOWANCES_MEMORY` exhausts the canister's stable memory allocation, all operations that attempt to write to stable memory (new approvals, balance updates on `transfer_from`) will fail. The ICP ledger is a system canister on the NNS subnet and does not pay cycles for storage in the standard way, removing the cycles-depletion mitigating factor. This matches the allowed High impact class: "Application/platform-level DoS … not based on raw volumetric DDoS."

## Likelihood Explanation
The attack requires paying the approval fee per entry. At 10,000 e8s (~$0.001 at $10/ICP), creating 1 million permanent entries costs ~$1,000 and consumes ~200 MB of stable memory. Full exhaustion of a 400 GB allocation would cost on the order of millions of dollars, making complete exhaustion economically prohibitive for most attackers. However, meaningful performance degradation and partial DoS (inability to create new approvals) are achievable at far lower cost. The attack is silent, gradual, requires no special privileges, and is not rate-limited. A well-funded adversary targeting critical ICP financial infrastructure has realistic motivation. Likelihood is Medium.

## Recommendation
1. **Enforce a per-account cap**: Reject `icrc2_approve` if the approver already has N active allowances (e.g., N = 100–1000). This directly bounds `ALLOWANCES_MEMORY` growth per principal.
2. **Require expiration or charge a higher fee for permanent allowances**: Making `expires_at: None` cost significantly more (e.g., 10× the standard fee) raises the economic barrier proportionally.
3. **Add a global allowance count limit**: A configurable `max_allowances` parameter analogous to `MAX_TRANSACTIONS_IN_WINDOW` would provide a hard ceiling on `ALLOWANCES_MEMORY` size.
4. **Implement a zero-balance cleanup hook**: A callable function that removes allowances whose approver balance is zero (since a zero-balance account cannot pay fees and is unlikely to need active allowances) would reclaim entries left by the drain-and-abandon pattern in the PoC.

## Proof of Concept
Minimal deterministic integration test plan using PocketIC or the state-machine test framework:

```rust
// 1. Initialize ledger with minting account M and transfer fee 10_000 e8s.
// 2. Mint 20_000 e8s to each of N subaccounts of principal P.
// 3. For each subaccount sub_i, call icrc2_approve({
//        from_subaccount: Some(sub_i),
//        spender: { owner: spender_principal, subaccount: Some([i; 32]) },
//        amount: 1,
//        expires_at: None,   // permanent
//        fee: None,
//        ...
//    }) — assert Ok.
// 4. Transfer remaining balance out of each sub_i (leaving 0 balance).
// 5. Call ledger.approvals().get_num_approvals() — assert == N.
// 6. Call prune(now + 1_year, N) — assert returns 0 (cannot prune permanent entries).
// 7. Assert ALLOWANCES_MEMORY.len() == N (entries persist indefinitely).
// 8. Repeat steps 2–4 until stable memory allocation is exhausted;
//    assert subsequent icrc2_approve calls return Err or trap.
```

This test is reproducible locally without mainnet access, uses only unprivileged ingress calls, and directly demonstrates the unbounded accumulation and prune ineffectiveness.
