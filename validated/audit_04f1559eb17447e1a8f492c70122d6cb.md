### Title
ICRC-107 Fee Collector Cannot Be Set at Runtime — Missing `icrc107_set_fee_collector` Endpoint - (File: `rs/ledger_suite/icrc1/ledger/ledger.did`)

---

### Summary

The ICRC-107 standard defines a dedicated `107set_fee_collector` endpoint transaction type that allows the fee collector to be changed at runtime by an authorized caller. The index-ng canister fully implements tracking of this block type. However, the ICRC-1 ledger canister exposes no `icrc107_set_fee_collector` update endpoint, making it impossible for any caller to produce a `107feecol` endpoint-type block at runtime. The ICRC-107 fee collector feature is structurally inaccessible.

---

### Finding Description

The ICRC-107 schema defines two distinct transaction subtypes within a `107feecol` block:

1. **Non-endpoint transactions** — fee-collector metadata emitted alongside other operations (optional `mthd`, optional `ts`, optional `caller`).
2. **Endpoint transactions** — produced when a caller explicitly invokes the `icrc107_set_fee_collector` endpoint; requires `mthd = "107set_fee_collector"`, a mandatory `ts`, and a mandatory `caller`. [1](#0-0) 

The index-ng canister tracks the ICRC-107 fee collector state in `fee_collector_107`, updated whenever it encounters a `107feecol` block with `mthd = "107set_fee_collector"`: [2](#0-1) 

However, the ledger canister's Candid interface exposes **no** `icrc107_set_fee_collector` update method. The complete service definition is: [3](#0-2) 

The ledger's `main.rs` implements `icrc152_mint` and `icrc152_burn` but contains no `icrc107_set_fee_collector` function. The only runtime path to change the fee collector is via `UpgradeArgs.change_fee_collector`, which requires a canister upgrade (controller-only privileged operation): [4](#0-3) 

All tests that exercise ICRC-107 fee collector blocks bypass the ledger endpoint entirely, using a test-only `add_block` helper that directly injects raw blocks into the ledger's ICRC-3 log — this path does not exist in production: [5](#0-4) 

---

### Impact Explanation

Any canister or user that attempts to call `icrc107_set_fee_collector` on the production ICRC-1 ledger will receive a "method not found" rejection. The ICRC-107 endpoint transaction block type (`mthd = "107set_fee_collector"`) can never be produced by the ledger through normal ingress. As a result:

- The index-ng's `fee_collector_107` state is never populated in production; the ICRC-107 fee collector tracking in the index is dead code.
- The fee collector can only be changed via a canister upgrade, which is a privileged controller operation — not the runtime, permissioned delegation the ICRC-107 standard is designed to provide.
- Any integrator relying on the ICRC-107 endpoint to change the fee collector at runtime without a full upgrade is blocked.

**Impact: Low** (fee collector can still be changed via upgrade; no funds are at risk).

---

### Likelihood Explanation

This is a structural omission — the endpoint is simply absent from the canister interface. Every attempt to use the ICRC-107 runtime fee-collector-setting feature will fail deterministically. **Likelihood: High**.

---

### Recommendation

Add an `icrc107_set_fee_collector` update endpoint to the ICRC-1 ledger canister (`rs/ledger_suite/icrc1/ledger/src/main.rs`) that:

1. Checks the caller is authorized (e.g., a controller or a designated fee-collector admin).
2. Records the caller principal and current timestamp.
3. Applies the new fee collector to ledger state.
4. Emits a `107feecol` block with `mthd = "107set_fee_collector"`, the required `ts`, and `caller` fields, conforming to the ICRC-107 schema.

Expose the new method in `rs/ledger_suite/icrc1/ledger/ledger.did` and add it to `icrc10_supported_standards` / `icrc3_supported_block_types` responses.

---

### Proof of Concept

1. Deploy the production ICRC-1 ledger canister.
2. Attempt to call `icrc107_set_fee_collector` from any principal (controller or otherwise).
3. The IC management layer returns a `CanisterMethodNotFound` rejection — the method does not exist in the service interface.
4. Confirm via `icrc3_supported_block_types`: the `107feecol` block type may be advertised, but no endpoint exists to produce endpoint-type `107feecol` blocks.
5. Inspect the index-ng's `fee_collector_107` state — it remains `None` indefinitely in production, confirming the ICRC-107 fee collector tracking is never activated. [6](#0-5) [7](#0-6)

/// Validate if a block is compatible with the ICRC-107 schema.
pub fn validate(block: &Value) -> Result<(), ValuePredicateFailures> {
    use ItemRequirement::*;

    let is_timestamp = is_more_or_equal_to(0);
    let is_icrc107_nonendpoint_transaction = and(vec![
        is_map(),
        item(
            "mthd",
            Optional,
            and(vec![is_text(), is_not(Value::text(SET_FEE_COL_107))]),
        ),
        item("fee_collector", Optional, is_account()),
        item("ts", Optional, is_timestamp.clone()),
        item("caller", Optional, is_principal()),
    ]);
    let is_icrc107_endpoint_transaction = and(vec![
        is_map(),
        item("mthd", Required, is(Value::text(SET_FEE_COL_107))),
        item("fee_collector", Optional, is_account()),
        item("ts", Required, is_timestamp.clone()),
        item("caller", Required, is_principal()),
