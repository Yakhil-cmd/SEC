### Title
Boundary Node Generic Rate Limiter Autoscale Integer Division Only Guards Zero, Silently Multiplying Effective Limit When `n < scale` - (File: `rs/boundary_node/ic_boundary/src/rate_limiting/generic.rs`)

### Summary

The `GenericLimiter::process_rules()` function in the IC boundary node autoscale path performs integer division of the configured per-rule limit `n` by the number of API boundary nodes `scale`, then applies `.max(1)` to prevent the result from reaching zero. This is structurally identical to the reported `TradingLimits::update()` bug: it only corrects the case where truncation produces 0, but silently allows the effective aggregate limit to exceed the operator-configured value whenever `n < scale`. An unprivileged external user who routes requests across multiple boundary nodes can bypass the intended rate limit by a factor of up to `scale`.

---

### Finding Description

In `rs/boundary_node/ic_boundary/src/rate_limiting/generic.rs`, the `process_rules` function scales each rule's limit down by the number of API boundary nodes so that the aggregate across all nodes equals the configured global limit: [1](#0-0) 

```rust
if let Action::Limit(n, d) = rule.limit {
    // Make sure the limit doesn't go below 1
    let limit = (n / scale).max(1);
    rule.limit = Action::Limit(limit, d);
}
```

`n` is a `u32` (the configured count from the rule, e.g. `10` in `10/1h`) and `scale` is a `u32` equal to `v.api_bns.len().max(1)`: [2](#0-1) 

The division is Rust integer (floor) division. When `n < scale`, `n / scale` evaluates to `0`, and `.max(1)` silently promotes it to `1`. Each of the `scale` boundary nodes then independently enforces a limit of `1` per period, so the aggregate effective limit becomes `scale` — which can be many times larger than the intended `n`.

The `Action` type is defined in `rs/boundary_node/rate_limits/api/src/schema_versions/v1.rs`: [3](#0-2) 

The parser explicitly rejects `count == 0`, so there is no `Action::Limit(0, d)` variant to use as a "block" sentinel — making the `.max(1)` guard appear necessary, but it is the root cause of the bypass.

The autoscale feature is enabled in production: [4](#0-3) 

The rules are fetched from the rate-limit canister `u637p-5aaaa-aaaaq-qaaca-cai`: [5](#0-4) 

---

### Impact Explanation

**Vulnerability class:** boundary/API validation bypass — rate limit bypass due to integer division rounding error.

When a rule is configured with a low count (e.g., `3/1h` to tightly restrict a specific canister or subnet) and the number of API boundary nodes `scale` exceeds that count (production deployments have ~10–20 BNs), the per-BN limit is silently set to `1` instead of `0`. Each BN independently allows `1` request per hour. An attacker who distributes requests across `scale` BNs can send `scale` requests per hour instead of the intended `3` — a bypass factor of `scale / n`, which can be 5–20× or more.

For rules intended to tightly throttle abuse of a specific canister (e.g., `1/1h` or `2/1h`), the bypass factor equals the full number of BNs, completely defeating the rate limit.

---

### Likelihood Explanation

- The autoscale feature is enabled in production (`RATE_LIMIT_GENERIC_AUTOSCALE=true`).
- Rules with low counts relative to the number of BNs are a natural operational choice when operators want to tightly restrict access to a specific canister or subnet.
- BN IP addresses are publicly discoverable from the IC registry, so an attacker can trivially route requests to distinct BNs.
- No privileged access, key material, or governance majority is required — only the ability to send HTTP requests to different BN endpoints.
- The `Limiter::Single` path (non-sharded rules) uses one `Ratelimiter` per BN per rule, so each BN's counter is fully independent. [6](#0-5) 

---

### Recommendation

Replace the `.max(1)` guard with a branch that converts to `Action::Block` when the scaled limit would be zero, preserving the intent of the configured rule:

```rust
if let Action::Limit(n, d) = rule.limit {
    let limit = n / scale;
    rule.limit = if limit == 0 {
        Action::Block
    } else {
        Action::Limit(limit, d)
    };
}
```

Alternatively, if the intent is to allow at least `n` requests in aggregate (ceiling semantics), use:

```rust
let limit = n.div_ceil(scale); // always >= 1 when n >= 1
rule.limit = Action::Limit(limit, d);
```

The chosen fix should match the documented intent: "the total ratelimit of all API BNs would be `n`." [7](#0-6) 

---

### Proof of Concept

**Setup:** A rule is configured as `3/1h` for a specific canister. The IC registry reports 10 API boundary nodes, so `scale = 10`.

**Execution in `process_rules`:**
```
n = 3, scale = 10
n / scale = 3 / 10 = 0   (integer division)
(0).max(1) = 1
→ each BN enforces limit = 1/1h
```

**Attacker action:** Send 1 request per hour to each of the 10 BN endpoints (IPs are public via the IC registry). Each BN independently passes the request (its own counter is at 1/1, within limit). Total requests processed: **10 per hour** instead of the intended **3 per hour** — a 3.3× bypass.

For `n = 1, scale = 20`: effective limit becomes **20/period** instead of **1/period** — a 20× bypass. [8](#0-7)

    fn process_rules(
        &self,
        rules: Vec<RateLimitRule>,
        old: &Arc<Vec<Bucket>>,
        scale: u32,
    ) -> Vec<Bucket> {
        rules
            .into_iter()
            .enumerate()
            .map(|(idx, mut rule)| {
                // Scale the rule limit accordingly
                if let Action::Limit(n, d) = rule.limit {
                    // Make sure the limit doesn't go below 1
                    let limit = (n / scale).max(1);
                    rule.limit = Action::Limit(limit, d);
                }

                // Check if the same rule exists in the same position.
                // If yes, then copy over the old limiter to avoid resetting it.
                if let Some(v) = old.get(idx)
                    && v.rule == rule
                {
                    return v.clone();
                }

                let limiter = if let Action::Limit(limit, duration) = &rule.limit {
                    Some(if let Some(v) = &rule.ip_prefix_group {
                        Limiter::Sharded(
                            Arc::new(ShardedRatelimiter::new(
                                *limit,
                                *limit,
                                *duration,
                                self.opts.tti,
                                self.opts.max_shards,
                            )),
                            *v,
                        )
                    } else {
                        Limiter::Single(Arc::new(create_ratelimiter(*limit, *limit, *duration)))
                    })
                } else {
                    None
                };

                Bucket { rule, limiter }
            })
            .collect()
    }
