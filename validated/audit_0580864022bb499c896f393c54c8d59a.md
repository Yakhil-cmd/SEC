Audit Report

## Title
`already_tried_to_auto_finalize` Set Irrevocably Before Finalization Completes, Permanently Blocking Auto-Finalization Retry on Partial Failure - (File: rs/sns/swap/src/swap.rs)

## Summary
In `try_auto_finalize`, `self.already_tried_to_auto_finalize` is set to `Some(true)` at line 719 before the async `finalize` call at line 722. If `finalize_inner` returns early due to any step failure (e.g., `sweep_icp` error), the flag is never reset. `can_auto_finalize` then permanently rejects all subsequent auto-finalization attempts, the periodic task timer is cleared, and the swap is left in a partially-finalized state with no automatic recovery path. Manual invocation of the `finalize_swap` endpoint remains possible but requires active external intervention.

## Finding Description
`try_auto_finalize` unconditionally sets the flag before attempting finalization:

```rust
// rs/sns/swap/src/swap.rs line 719-722
self.already_tried_to_auto_finalize = Some(true);
let auto_finalize_swap_response = self.finalize(now_fn, environment).await;
```

`finalize_inner` is a sequential pipeline that returns early on any step error:

```rust
// lines 1557-1561
finalize_swap_response
    .set_sweep_icp_result(self.sweep_icp(now_fn, environment.icp_ledger()).await);
if finalize_swap_response.has_error_message() {
    return finalize_swap_response;
}
```

The same early-return pattern repeats for `settle_neurons_fund_participation`, `create_sns_neuron_recipes`, `sweep_sns`, and `claim_swap_neurons`. On any such early return, `already_tried_to_auto_finalize` remains `true` and is never reset.

`can_auto_finalize` then permanently blocks retry:

```rust
// lines 2935-2941
if self.already_tried_to_auto_finalize.unwrap_or(true) {
    return Err(format!(
        "self.already_tried_to_auto_finalize is {:?}, indicating that an attempt has already been made..."
    ));
}
```

`requires_periodic_tasks` returns `false` once the lifecycle is terminal and the flag is `true`:

```rust
// lines 669-674
pub fn requires_periodic_tasks(&self) -> bool {
    !self.lifecycle_is_terminal() || !self.already_tried_to_auto_finalize.unwrap_or(true)
}
```

The canister-level `run_periodic_tasks` (in `rs/sns/swap/canister/canister.rs` lines 293–302) clears the IC timer when `requires_periodic_tasks()` returns `false`, permanently stopping the heartbeat-driven retry loop. The log message at line 716 explicitly acknowledges this design: *"Will not automatically attempt again even if this fails."*

## Impact Explanation
When auto-finalization partially fails, ICP buyer funds remain locked in Swap canister subaccounts, Neurons' Fund maturity reserved for the SNS remains unreturned to NNS neurons, and SNS tokens remain undistributed. The `error_refund_icp` path is also blocked until finalization completes. This constitutes a concrete, significant SNS-framework impact with direct user and protocol harm — matching the High bounty category: *"Significant... SNS... security impact with concrete user or protocol harm."* Manual finalization via the public `finalize_swap` endpoint is still callable, but requires active external monitoring and intervention; for unmonitored swaps the state is effectively stuck indefinitely.

## Likelihood Explanation
`finalize_inner` makes cross-canister calls to the ICP ledger, NNS governance, SNS ledger, SNS governance, and SNS root. Any transient error from any of these — canister temporarily busy, message timeout, or a single failed transfer — triggers the early return. Transient cross-canister failures are a realistic and documented occurrence on the IC. No attacker action is required; the condition arises from ordinary operational failures. The trigger is fully reachable without any special privileges.

## Recommendation
After `finalize` returns, check whether the response contains an error and reset the flag if so:

```rust
self.already_tried_to_auto_finalize = Some(true);
let auto_finalize_swap_response = self.finalize(now_fn, environment).await;
if auto_finalize_swap_response.has_error_message() {
    self.already_tried_to_auto_finalize = Some(false); // allow retry
}
```

Alternatively, set the flag to `Some(true)` only after a fully successful finalization response. The log message should be updated to reflect the corrected retry behavior.

## Proof of Concept
1. Deploy an SNS swap configured with `should_auto_finalize = true`.
2. Advance the swap to `Lifecycle::Committed`.
3. Mock the ICP ledger to return a transient error on the first `transfer_funds` call in `sweep_icp`.
4. Allow the periodic task to fire, invoking `run_periodic_tasks` → `try_auto_finalize`.
5. Observe: `already_tried_to_auto_finalize` is set to `Some(true)` at line 719; `sweep_icp` fails; `finalize_inner` returns early with `error_message` set; the flag is not reset.
6. Observe: `requires_periodic_tasks()` now returns `false`; the IC timer is cleared.
7. Restore the ICP ledger mock to succeed. Observe that no further automatic finalization occurs — `can_auto_finalize()` returns `Err` on every subsequent call.
8. Confirm buyer ICP remains in the Swap subaccount and SNS tokens are undistributed until `finalize_swap` is called manually.

This is directly reproducible as a unit test in `rs/sns/swap/tests/swap.rs` by injecting a failing `SpyLedger` reply for `sweep_icp` and asserting the post-condition state.
