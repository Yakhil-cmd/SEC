### Title
Block Proposer Can Manipulate Ingress Message Ordering via Pool Insertion Timestamp to Front-Run SNS Swap Participants - (File: `rs/ingress_manager/src/ingress_selector.rs`)

---

### Summary

The ingress selector in `get_ingress_payload` sorts messages by their pool insertion timestamp to prevent users from gaming ordering via crafted expiry times. However, the pool insertion timestamp is set by the artifact processor at the moment it drains its input channel — a moment fully controlled by the block-proposing node. A malicious block proposer can delay draining the channel for victim messages (giving them a later timestamp) while immediately processing its own message (giving it an earlier timestamp). Because the selector processes oldest-first, the proposer's message executes first in the block. For any canister with order-dependent state — most concretely the SNS swap canister near its ICP participation cap — this allows the block proposer to front-run other participants and capture the remaining allocation.

---

### Finding Description

**Root cause — timestamp assignment in the artifact processor**

In `rs/p2p/artifact_manager/src/lib.rs`, `IngressProcessor::process_changes` stamps every ingress message with the node's local clock at the instant the artifact processor loop drains its channel:

```rust
UnvalidatedArtifactMutation::Insert((message, peer_id)) => {
    let unvalidated_artifact = UnvalidatedArtifact {
        message,
        peer_id,
        timestamp: time_source.get_relative_time(),   // ← set by the node, not the sender
    };
    ingress_pool.insert(unvalidated_artifact);
}
``` [1](#0-0) 

The timestamp is therefore the node's wall-clock time when it chose to process the message, not when the user submitted it.

**Root cause — ordering in the ingress selector**

`get_ingress_payload` in `rs/ingress_manager/src/ingress_selector.rs` explicitly sorts per-canister queues by this timestamp (descending), then `pop()`s from the back, so the **oldest** (smallest timestamp) message is selected first:

```rust
// In order to prevent malicious users from putting their messages ahead of others
// by carefully crafting the expiry times, we sort the ingress messages by the time
// they were delivered to the pool.
// NOTE: We sort in reverse order, because messages are pop()-ed from the back.
for v in canister_queues.values_mut() {
    v.msgs.sort_unstable_by_key(|artifact| {
        std::cmp::Reverse(artifact.timestamp.as_nanos_since_unix_epoch())
    });
}
``` [2](#0-1) 

The comment acknowledges the ordering-fairness intent, but the fix is incomplete: it moves the manipulation surface from the user-controlled expiry field to the node-controlled insertion timestamp.

**Exploitation path**

A block proposer that is also a node operator can:
1. Receive victim's `refresh_buyer_tokens` ingress message via the HTTP endpoint; deliberately delay draining the artifact-processor channel, assigning it timestamp `T_victim` (later).
2. Submit its own `refresh_buyer_tokens` call immediately; the artifact processor drains it at timestamp `T_self < T_victim`.
3. Call `get_ingress_payload`; the selector places the proposer's message at the back of the sorted queue (oldest = smallest timestamp) and pops it first.
4. The proposer's message executes first inside the block.

**Impact on the SNS swap canister**

`refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` computes the available headroom at call time:

```rust
let max_increment_e8s = self.available_direct_participation_e8s();
``` [3](#0-2) 

It then caps the accepted amount to whatever headroom remains:

```rust
let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);
``` [4](#0-3) 

When the swap is within one `max_participant_icp_e8s` of the `max_direct_participation_icp_e8s` ceiling, the first call consumes all remaining headroom. Subsequent calls in the same block receive zero increment and may be rejected:

```rust
if new_balance_e8s < params.min_participant_icp_e8s {
    return Err(format!(
        "Rejecting participation of effective amount {}; minimum required to participate: {}",
        new_balance_e8s, params.min_participant_icp_e8s
    ));
}
``` [5](#0-4) 

The block proposer therefore captures the last allocation slot at the direct expense of the victim whose message arrived first but was timestamped later.

The swap's own code acknowledges that message re-ordering is a known concern:

```rust
if old_amount_icp_e8s >= e8s {
    // Already up-to-date. Strict inequality can happen if messages are re-ordered.
``` [6](#0-5) 

---

### Impact Explanation

A malicious block proposer can guarantee that its own `refresh_buyer_tokens` call executes before any competing call in the same block. When the SNS swap is near its ICP cap, this lets the proposer capture the last participation slot, causing the victim's call — which was submitted earlier — to be rejected or receive a reduced allocation. The victim loses their intended SNS token allocation; the proposer gains it. The same ordering primitive applies to any other canister that has first-come-first-served semantics (e.g., a canister-based auction, a limited-supply NFT mint, or any rate-limited resource).

---

### Likelihood Explanation

Every subnet node is eligible to be the block proposer in any given round. A node operator running a node on the subnet hosting the SNS swap canister (an application subnet) can monitor the swap's participation level via query calls, wait until the cap is within reach, and execute the attack in the round where they are selected as proposer. Block proposer selection is deterministic and publicly observable in advance (via the random beacon), so the attacker can prepare. No threshold corruption, governance majority, or privileged key is required — only operating a single honest-appearing replica node on the relevant subnet.

---

### Recommendation

1. **Remove the timestamp-based ordering dependency.** The pool insertion timestamp is node-local and unverifiable by other replicas. Ordering within a block should not rely on it for fairness.
2. **Use a verifiable, sender-controlled ordering key.** The ingress expiry time is user-set and signed; a bounded window (e.g., round to the nearest second) prevents fine-grained manipulation while still providing a canonical order that all replicas can independently verify.
3. **Alternatively, randomize intra-block ordering** using the round's random tape, which is unpredictable to the proposer at payload-construction time.
4. **For the SNS swap specifically**, consider making `refresh_buyer_tokens` idempotent with respect to ordering by recording the ICP ledger balance at a certified height rather than accepting the first-writer-wins race.

---

### Proof of Concept

1. Deploy an SNS swap canister on an application subnet with `max_direct_participation_icp_e8s = 100 ICP` and `max_participant_icp_e8s = 10 ICP`.
2. Fill the swap to 91 ICP (9 ICP remaining headroom, below `max_participant_icp_e8s`).
3. Victim sends `refresh_buyer_tokens` with 9 ICP deposited; the message enters the proposer node's artifact-processor channel.
4. The proposer node delays draining the channel (e.g., by 200 ms), assigning the victim's message timestamp `T+200ms`.
5. The proposer immediately submits its own `refresh_buyer_tokens` with 9 ICP; the artifact processor assigns timestamp `T`.
6. The proposer calls `get_ingress_payload`; the selector pops the proposer's message first (timestamp `T < T+200ms`).
7. The proposer's call executes first: `available_direct_participation_e8s()` returns 9 ICP, the proposer receives the full 9 ICP allocation, swap reaches cap.
8. The victim's call executes second: `available_direct_participation_e8s()` returns 0, the call is rejected with "Rejecting participation of effective amount 0".
9. The victim's ICP sits in the swap subaccount and must be reclaimed via `error_refund_icp` after the swap closes.
