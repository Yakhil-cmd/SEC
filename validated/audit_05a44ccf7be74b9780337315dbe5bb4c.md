Audit Report

## Title
Byzantine Peer Exploiting `DownloadChunkError::Overloaded` to Stall State Sync — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
A Byzantine subnet node that always responds with HTTP 429 is never evicted from `active_downloads` because the `Overloaded` error branch only re-queues the chunk without removing the peer. The `run` loop's sole non-cancellation exit condition is `active_downloads.is_empty()`, which is never satisfied, so the downloading replica loops indefinitely re-assigning the same chunks to the same Byzantine peer with no retry limit, no backoff, and no peer-score penalty. If the Byzantine node is the only (or last remaining) advertiser of the target state height, state sync stalls permanently and the replica cannot rejoin consensus.

## Finding Description
**Root cause — asymmetric peer-removal logic in `handle_downloaded_chunk_result`:**

`NoContent` (L194-199) and `RequestError` (L201-209) both call `active_downloads.remove(&peer_id)` and decrement `allowed_downloads`. The `Overloaded | Timeout | Cancelled` arm (L211-222) does neither — it only calls `self.chunks_to_download.download_failed(chunk_id)`.

`download_failed` (chunks_to_download.rs L26-28) is a bare `self.chunks.push(chunk_id)` — no counter, no backoff, no deduplication.

`parse_chunk_handler_response` (chunk.rs L129) maps `StatusCode::TOO_MANY_REQUESTS` directly to `DownloadChunkError::Overloaded`.

**Exploit flow:**
1. Victim replica falls behind and calls `maybe_start_state_sync`.
2. Byzantine node is first (or only) to send an advert for the target height; it is inserted into `active_downloads` with budget `PARALLEL_CHUNK_DOWNLOADS = 10`.
3. Every chunk request returns 429 → `Overloaded` → chunk re-queued, peer retained.
4. `spawn_chunk_downloads` immediately re-assigns up to 10 tasks to the same peer.
5. `active_downloads` never empties; the loop exit at L173-176 is never reached.
6. The only escape is an external `cancellation` signal (operator shutdown or `cancel_if_running` returning true), neither of which is guaranteed.

**Why existing checks are insufficient:**
- The `new_peers_rx` self-healing path (lib.rs L154) uses `try_send` (can drop) and only fires if an honest peer happens to advertise the *same* artifact ID while the sync is running. If honest nodes have already pruned that state height, no honest advert ever arrives.
- There is no overall state-sync deadline independent of per-chunk timeouts.
- The comment at L390-391 explicitly labels `Overloaded` as "transient," making the missing eviction a deliberate design choice that a Byzantine peer can exploit without bound.

## Impact Explanation
A replica stalled in state sync cannot apply new blocks, cannot participate in consensus, and is effectively partitioned from the subnet for the duration of the attack. This constitutes a **limited subnet availability impact** caused by a single Byzantine protocol peer operating below the consensus fault threshold — matching the Medium impact tier: *"node/boundary-node control … but still with meaningful security impact."* In subnets already at the Byzantine fault boundary (f honest nodes remaining), removing one more honest participant from consensus can degrade finality guarantees.

## Likelihood Explanation
- Requires control of exactly one subnet node — a Byzantine protocol peer below the consensus fault threshold, which is explicitly in scope.
- No privileged key, governance majority, or subnet-majority corruption is needed.
- The attack is most effective when the Byzantine node is the first to advertise a state height to a lagging replica (e.g., immediately after a restart or network partition), before honest peers re-advertise.
- The `Overloaded` path is explicitly documented as "transient," so the missing eviction is a stable, exploitable property of the current design, not a race condition.
- The existing test `test_cancel_if_running` (ongoing.rs L429-460) uses a mock that always returns 429 and only terminates because `shutdown()` is called manually — confirming the loop does not self-terminate under this condition.

## Recommendation
1. **Track consecutive `Overloaded` responses per peer.** After a configurable threshold (e.g., 20 consecutive 429s with no successful chunk in between), treat the peer identically to `RequestError` and remove it from `active_downloads`.
2. **Introduce exponential backoff** before re-assigning chunks to a peer that returned `Overloaded`, preventing the tight re-queue loop from spinning at wire speed.
3. **Add an overall state-sync deadline** (e.g., a configurable wall-clock timeout on the entire `OngoingStateSync::run` future) so a stalled sync is eventually abandoned and restarted, potentially discovering a different peer set.

## Proof of Concept
Extend `test_cancel_if_running` to omit the manual `shutdown()` call and add a `tokio::time::timeout` around the block:

```rust
rt.block_on(async move {
    ongoing.sender.send(NODE_1).await.unwrap();
    // Without shutdown(), this times out — confirming the permanent stall:
    let result = tokio::time::timeout(
        Duration::from_secs(5),
        ongoing.shutdown.shutdown(),
    ).await;
    assert!(result.is_err(), "loop should have stalled");
});
```

The test will reliably time out, demonstrating that a single peer returning 429 indefinitely prevents the `run` loop from exiting without external cancellation.
