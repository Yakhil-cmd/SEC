### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Over-Minting and Bridge Insolvency - (File: `engine/src/engine.rs`)

### Summary
When a NEP-141 token with a transfer fee is bridged into Aurora via `ft_transfer_call`, the engine mints ERC-20 tokens equal to the *nominal* `args.amount` reported by the NEP-141 contract, not the *net* amount actually credited to Aurora's NEP-141 balance. This permanently inflates the ERC-20 mirror supply beyond Aurora's actual NEP-141 reserves, making the bridge insolvent for that token.

### Finding Description
The NEP-141 `ft_transfer_call` flow works as follows:
1. A user calls `ft_transfer_call(receiver_id=aurora, amount=X, msg=<evm_address>)` on a fee-on-transfer NEP-141 contract.
2. The NEP-141 contract deducts a fee `f`, crediting Aurora with only `X - f` tokens in its NEP-141 balance.
3. The NEP-141 contract then calls `ft_on_transfer(sender_id, amount=X, msg)` on Aurora — passing the *original* `X`, not the net `X - f`.
4. Aurora's `ft_on_transfer` handler dispatches to `receive_erc20_tokens`: [1](#0-0) 

Inside `receive_erc20_tokens`, the mint amount is taken directly from `args.amount` without any balance-delta verification: [2](#0-1) 

This value is then passed to `setup_receive_erc20_tokens_input`, which encodes a `mint(recipient, amount)` call to the ERC-20 mirror contract: [3](#0-2) 

The ERC-20 `mint` function on the mirror contract mints exactly `amount` tokens with no further checks: [4](#0-3) 

The same pattern exists in `receive_base_tokens` for the ETH connector path: [5](#0-4) 

After each fee-bearing deposit, the invariant `erc20_total_supply == aurora_nep141_balance` is broken by `f` units. The discrepancy accumulates with every deposit.

### Impact Explanation
The ERC-20 mirror token becomes over-minted relative to Aurora's actual NEP-141 reserves. When users later call `withdrawToNear` (which burns ERC-20 tokens and triggers an `ft_transfer` of NEP-141 tokens from Aurora to the recipient), the last users to withdraw will find Aurora's NEP-141 balance insufficient to cover the burn amount. Their `ft_transfer` call will fail, permanently freezing their ERC-20 tokens with no redemption path. This is bridge insolvency: the sum of all ERC-20 claims exceeds the backing NEP-141 reserve.

**Impact: Critical — Insolvency / Permanent freezing of funds.**

### Likelihood Explanation
Any NEP-141 token that implements a transfer fee and is registered as an ERC-20 mirror on Aurora triggers this automatically on every `ft_transfer_call`. No privileged access is required. The NEAR ecosystem has no protocol-level prohibition on fee-on-transfer NEP-141 tokens. Any token deployer or user who bridges such a token into Aurora triggers the accounting divergence. The vulnerability is reachable by any unprivileged token holder.

### Recommendation
In `receive_erc20_tokens` (and `receive_base_tokens`), replace the use of `args.amount` as the mint quantity with the actual balance delta observed on Aurora's side. Concretely:
- Before processing `ft_on_transfer`, read Aurora's current NEP-141 balance for the token (`predecessor_account_id`).
- After the transfer has been credited by the NEP-141 contract (which has already happened before `ft_on_transfer` is called), read the new balance.
- Mint only `new_balance - old_balance` ERC-20 tokens.

Alternatively, enforce at the `deploy_erc20_token` registration step that the NEP-141 token does not implement transfer fees, and reject `ft_on_transfer` calls from tokens that are not explicitly whitelisted.

### Proof of Concept
1. Deploy a NEP-141 token `fee_token.near` that deducts a 1% fee on every transfer (i.e., a transfer of `1000` credits the receiver with `990`).
2. Call `deploy_erc20_token` on Aurora to register `fee_token.near` → ERC-20 mirror at address `0xMIRROR`.
3. Call `ft_transfer_call(receiver_id="aurora", amount=1000, msg=<alice_evm_address>)` on `fee_token.near`.
   - Aurora's NEP-141 balance for `fee_token.near` increases by **990**.
   - `ft_on_transfer` is called with `args.amount = 1000`.
   - `receive_erc20_tokens` mints **1000** ERC-20 tokens to Alice.
4. Repeat step 3 for Bob with `amount=1000`. Aurora holds **1980** NEP-141, but **2000** ERC-20 tokens exist.
5. Alice calls `withdrawToNear(alice_near_account, 1000)` — succeeds; Aurora now holds **980** NEP-141.
6. Bob calls `withdrawToNear(bob_near_account, 1000)` — Aurora only holds **980** NEP-141 but must transfer **1000**. The `ft_transfer` fails. Bob's 1000 ERC-20 tokens are burned but he receives nothing. **Bob's funds are permanently lost.**
