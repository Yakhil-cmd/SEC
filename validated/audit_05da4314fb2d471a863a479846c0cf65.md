### Title
`new()` Initialization Can Be Front-Run, Enabling Attacker to Seize Ownership and Redirect Bridge Funds - (`engine/src/contract_methods/admin.rs`)

---

### Summary

The `new()` function that initializes the Aurora Engine contract state contains no access control. In NEAR Protocol, contract deployment (uploading WASM) and initialization (`new()`) are separate transactions. An attacker who observes the deployment can call `new()` before the legitimate deployer, supplying their own `owner_id`. This gives the attacker full ownership of the engine, from which they can set a malicious ETH connector account via `set_eth_connector_contract_account()`, redirecting all bridge withdrawals to an attacker-controlled contract.

---

### Finding Description

`new()` in `engine/src/contract_methods/admin.rs` performs only a single guard:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // No check on env.predecessor_account_id() or env.current_account_id()
    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    let state: EngineState = args.into();
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

There is no check that `env.predecessor_account_id()` equals `env.current_account_id()` (the contract account itself) or any other trusted deployer. Any NEAR account can call `new()` as long as the state has not yet been written.

The deployment helper confirms that deployment and initialization are two distinct async transactions:

```rust
let contract = account.deploy(&self.code...).await?;
// ...
engine.new(self.chain_id, self.owner_id, self.upgrade_delay_blocks).transact().await?;
``` [2](#0-1) 

Between these two transactions there is an observable on-chain window. An attacker who monitors NEAR for the deployment receipt can submit their own `new()` call with `owner_id` set to their account before the deployer's `new()` lands.

Once the attacker is owner, `set_eth_connector_contract_account()` — which is gated by `require_owner_only` — becomes callable by the attacker:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);
        ...
    })
}
``` [3](#0-2)

---

### Impact Explanation

**Critical — Direct theft of user funds in motion.**

Any user who calls the `ExitToNear` or `ExitToEthereum` precompiles after the attacker has poisoned the connector account will have their bridged ETH or ERC-20 tokens sent to the attacker's malicious NEAR contract. The attacker can drain all bridge withdrawals indefinitely until the legitimate owner redeploys the contract.

---

### Likelihood Explanation

**High.** NEAR Protocol transactions are publicly observable. The window between WASM deployment and `new()` initialization spans at least one block (and in practice, multiple blocks if the deployer uses separate scripts or manual steps, as shown in the workspace helper). An attacker running a NEAR indexer or monitoring the RPC can detect the deployment and submit a competing `new()` call in the same or next block. No special privileges or leaked keys are required — only the ability to submit a NEAR transaction.

---

### Recommendation

Add a predecessor-account check inside `new()` to ensure only the contract account itself (i.e., a self-call from a batch deploy action) can initialize the state:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Require that the caller is the contract itself (batch deploy+init pattern)
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    ...
}
```

Alternatively, deploy and initialize atomically in a single NEAR batch action so no inter-transaction window exists.

---

### Proof of Concept

1. Attacker monitors NEAR for a transaction that deploys Aurora Engine WASM to account `aurora.near`.
2. Before the deployer's `new()` lands, attacker submits:
   ```
   aurora.near::new({ chain_id: ..., owner_id: "attacker.near", upgrade_delay_blocks: 1 })
   ```
3. Deployer's `new()` reverts with `ERR_ALREADY_INITIALIZED`.
4. Attacker calls:
   ```
   aurora.near::set_eth_connector_contract_account({ account: "evil-connector.attacker.near", ... })
   ```
   This succeeds because `attacker.near` is now `owner_id`.
5. Any user who calls the `ExitToNear` precompile (e.g., bridging ETH back to NEAR) triggers a `ft_transfer_call` to `evil-connector.attacker.near`, which steals the tokens. [1](#0-0) [6](#0-5) [7](#0-6)

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}
