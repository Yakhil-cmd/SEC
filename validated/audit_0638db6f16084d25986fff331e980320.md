### Title
`native_price` and `pubdata_price` Absent from ZK Public Inputs Enables Prover to Diverge Fee Accounting from Forward Execution - (`basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/mod.rs`)

---

### Summary

`native_price` and `pubdata_price` are operator-supplied block-level parameters that directly control whether transactions succeed or fail and how much fee is charged. They are consumed during both forward execution and proving, but are **never committed to in the ZK proof public inputs**. A malicious prover can therefore substitute arbitrary values for these parameters when generating a proof, producing a proven state transition that diverges from the one actually executed by the sequencer — causing valid transactions to appear reverted, or failed transactions to appear successful, with direct financial impact on users.

---

### Finding Description

`BlockMetadataFromOracle` carries `native_price` and `pubdata_price` alongside the standard Ethereum block fields. These two ZKsync-specific pricing knobs are read from the oracle at block start and used throughout every transaction:

- `native_per_gas = ceil(gas_price / native_price)` — the native-resource budget per unit of EVM gas.
- `native_per_pubdata = pubdata_price / native_price` — the native-resource cost per byte of pubdata. [1](#0-0) 

These ratios determine whether a transaction runs out of native resources (causing a revert and full gas consumption), and feed the `delta_gas` adjustment that inflates `gas_used` when native consumption exceeds EVM gas consumption. [2](#0-1) 

At block finalization, `form_block_header` explicitly omits both values with a `TODO` comment:

```rust
// TODO: add pubdata price and native price
``` [3](#0-2) 

The ZK public inputs (`ChainStateCommitment`, `BatchOutput`, `BatchPublicInput`) commit to: state root, block number, block hashes, timestamps, chain ID, DA commitment, L1 tx hashes, and L2 log roots. Neither `native_price` nor `pubdata_price` appear anywhere in these structures. [4](#0-3) 

The project documentation explicitly acknowledges this gap:

> "The block header should determine the block fully, i.e. include all the inputs needed to execute the block. Currently it misses `gas_per_pubdata` and `native_price`, but we already working on design and implementation to solve this issue." [5](#0-4) 

---

### Impact Explanation

Because `native_price` and `pubdata_price` are unconstrained by the proof, a malicious prover can substitute any values when generating the RISC-V execution trace submitted to the settlement layer. Two concrete attack paths:

**Path A — Revert-inflation (user fund loss):** The prover uses `native_price' ≪ native_price_sequencer`. This inflates `native_per_gas`, exhausting the native budget of transactions that succeeded during forward execution. Those transactions appear as reverted in the proven state. Users lose their full gas fee (the bootloader charges `gas_limit * gas_price` on revert) even though their transactions actually executed successfully.

**Path B — Revert-suppression (invalid state acceptance):** The prover uses `native_price' ≫ native_price_sequencer`. This deflates `native_per_gas` to near zero, granting effectively unlimited native resources. Transactions that ran out of native resources during forward execution (and were reverted) appear as successful in the proven state, committing invalid state changes to the settlement layer.

In both cases the settlement layer accepts the proof because the manipulated parameters are not part of the public input preimage it verifies. [6](#0-5) 

---

### Likelihood Explanation

The attacker must control the prover. In ZKsync's architecture the sequencer and prover are typically operated by the same entity (Matter Labs or a delegated operator). A compromised or malicious operator can exploit this without any on-chain signal. The attack is silent: the settlement layer sees a valid proof and a valid state root transition; the discrepancy is only visible by comparing the forward-execution receipts against the proven state, which requires off-chain monitoring. Likelihood is **medium** given the privileged-but-realistic attacker model explicitly included in the Immunefi scope ("prover/forward execution input").

---

### Recommendation

Include `native_price` and `pubdata_price` in the ZK public inputs so the settlement layer can verify that the prover used the same pricing parameters as the sequencer. Concretely:

1. Add both fields to `BatchOutput` (or `ChainStateCommitment`) and include them in the respective `hash()` computation.
2. Alternatively, encode them in the `extra_data` field of the block header (already noted as the intended location in the bootloader docs) and commit the block header hash into the public inputs.
3. Until the fix is deployed, the sequencer should publish `native_price` and `pubdata_price` alongside each batch so that watchers can detect divergence between forward receipts and the proven state. [7](#0-6) 

---

### Proof of Concept

**Setup:** Sequencer executes a block with `native_price = 1_000_000` and `pubdata_price = 10_000_000_000`. A user submits a transaction with `max_fee_per_gas = 2_000_000`, `gas_limit = 100_000`. The sequencer computes `native_per_gas = ceil(2_000_000 / 1_000_000) = 2`, giving a native budget of `200_000` native units. The transaction executes successfully and consumes 150_000 native units.

**Attack:** The malicious prover re-runs the RISC-V execution with `native_price = 1` (instead of `1_000_000`). Now `native_per_gas = 2_000_000`, native budget = `200_000_000_000`. The transaction still succeeds (budget is now enormous), but the prover could equally set `native_price = 2_000_001`, making `native_per_gas = 1`, native budget = `100_000`. The transaction now exhausts its native budget mid-execution and is reverted. The prover submits this proof.

**Result:** The settlement layer verifies the proof against `(state_before, state_after, batch_output_hash)`. None of these commit to `native_price`. The proof is accepted. The user's transaction is recorded as reverted; the user loses `100_000 * 2_000_000 = 200_000_000_000` wei in gas fees for a transaction that actually succeeded. [8](#0-7) [3](#0-2)

pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }
}
