Audit Report

## Title
Node Provider Monthly Reward Silently Dropped Instead of Capped When Exceeding `maximum_node_provider_rewards_e8s` - (File: rs/nns/governance/src/governance.rs)

## Summary
`reward_node_provider_helper` returns a hard `Err` when a node provider's computed reward exceeds `maximum_node_provider_rewards_e8s` rather than capping the payment at the maximum. Because `mint_monthly_node_provider_rewards` discards the error result with `let _` and then unconditionally advances the monthly reward timestamp via `update_most_recent_monthly_node_provider_rewards`, the affected node provider receives zero ICP for that period and the reward is permanently unrecoverable.

## Finding Description
The reward flow is:

1. `get_node_provider_reward` (L8248–8271) computes `amount_e8s` via raw XDR→ICP division with no cap applied.
2. The resulting `RewardNodeProvider` structs are passed to `reward_node_providers` (L3987–4006), which calls `reward_node_provider_helper` per provider.
3. `reward_node_provider_helper` (L3953–3960) checks `reward.amount_e8s > maximum_node_provider_rewards_e8s` and returns `Err(PreconditionFailed)` — no ICP is minted, not even the capped amount.
4. `reward_node_providers` logs the error and continues (L3995–4001); the aggregate `Err` is returned.
5. `mint_monthly_node_provider_rewards` (L4073–4076) discards the result with `let _` and calls `update_most_recent_monthly_node_provider_rewards` unconditionally, advancing the timestamp.

The timestamp advance means the next cycle starts fresh; the skipped reward is never retried.

## Impact Explanation
This is a permanent, silent loss of ICP that should have been minted to a legitimate node provider. The correct behavior — capping at `maximum_node_provider_rewards_e8s` — is never executed. This matches the **High** impact category: "Significant NNS security impact with concrete user or protocol harm," specifically financial harm to node providers whose rewards are silently zeroed rather than capped.

## Likelihood Explanation
No privileged action is required. The trigger is purely market-driven: a sustained drop in ICP price causes the same XDR reward to convert to a larger e8s amount. Large node providers with many nodes in high-reward regions are most exposed. The periodic task `mint_monthly_node_provider_rewards` fires automatically each month. If `maximum_node_provider_rewards_e8s` is ever lowered via a `ManageNetworkEconomics` governance proposal (a routine governance action), the threshold becomes easier to breach. The condition is deterministic and repeatable every reward cycle as long as the price condition persists.

## Recommendation
In `reward_node_provider_helper`, replace the hard rejection with a cap:

```rust
let amount_to_mint = reward.amount_e8s.min(maximum_node_provider_rewards_e8s);
```

Use `amount_to_mint` when constructing the minting call instead of `reward.amount_e8s`. Additionally, the `let _` discard in `mint_monthly_node_provider_rewards` should be replaced with proper error handling so that a failed reward either prevents the timestamp from advancing for the affected provider or triggers a retry.

## Proof of Concept
1. Deploy a local replica with NNS governance.
2. Register node provider NP-A with enough nodes that their monthly XDR reward is large.
3. Set `maximum_node_provider_rewards_e8s` to a low value via `ManageNetworkEconomics` (e.g., `10_000_000_000_000` e8s = 100,000 ICP).
4. Set the ICP/XDR conversion rate so that NP-A's XDR reward converts to more than 100,000 ICP.
5. Trigger `mint_monthly_node_provider_rewards`.
6. Observe: NP-A's ledger account receives 0 ICP; the monthly reward timestamp advances; no retry occurs next cycle.
7. A unit test can directly call `reward_node_provider_helper` with `amount_e8s > maximum_node_provider_rewards_e8s` and assert the `Err` return, then verify `mint_monthly_node_provider_rewards` still calls `update_most_recent_monthly_node_provider_rewards` despite the error.
