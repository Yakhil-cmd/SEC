Audit Report

## Title
Predictable Subaccount DoS on `spawn_neuron` with Nonce via Pre-emptive Neuron Claim - (File: `rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with an explicit `nonce`, the child neuron's subaccount is computed using `compute_neuron_staking_subaccount(child_controller, nonce_val)` — the same public formula used by the open `ClaimOrRefresh::MemoAndController` ingress path. Any unprivileged actor who knows the intended `child_controller` and `nonce` can pre-emptively stake ICP to that subaccount and claim a neuron there, causing `ensure_subaccount_available` to return `SubaccountAlreadyExists` and permanently blocking that `(child_controller, nonce)` pair for `spawn_neuron`.

## Finding Description
In `rs/nns/governance/src/governance.rs` at L2678–2688, `spawn_neuron` branches on whether a nonce is supplied:

```rust
let to_subaccount = match spawn.nonce {
    None => self.neuron_store.new_neuron_subaccount(&mut *self.randomness)?,
    Some(nonce_val) => {
        let to_subaccount =
            ledger::compute_neuron_staking_subaccount(child_controller, nonce_val);
        self.neuron_store
            .ensure_subaccount_available(to_subaccount)?
    }
};
```

`ensure_subaccount_available` in `rs/nns/governance/src/neuron_store.rs` at L488–496 fails immediately with no retry:

```rust
pub fn ensure_subaccount_available(&self, subaccount: Subaccount) -> Result<Subaccount, NeuronStoreError> {
    if self.has_neuron_with_subaccount(subaccount) {
        return Err(NeuronStoreError::SubaccountAlreadyExists { subaccount });
    }
    Ok(subaccount)
}
```

The `ClaimOrRefresh::MemoAndController` ingress path in `claim_or_refresh_neuron_by_memo_and_controller` at L5858–5860 uses the identical formula:

```rust
let controller = memo_and_controller.controller.unwrap_or(*caller);
let memo = memo_and_controller.memo;
let subaccount = ledger::compute_neuron_staking_subaccount(controller, memo);
```

Crucially, `controller` is taken from the message payload — not from the caller — so any unprivileged sender can specify an arbitrary `controller`. This is explicitly tested and supported behavior (see `test_claim_neuron_memo_and_controller_by_proxy`). The subaccount formula uses domain `"neuron-stake"` (confirmed in `rs/nervous_system/common/src/ledger.rs` at L6–7), which is shared between both paths with no namespace separation.

**Attack path:**
1. Attacker computes `target = SHA256(len("neuron-stake") || "neuron-stake" || child_controller_bytes || nonce_u64_be)`.
2. Attacker transfers ≥ `neuron_minimum_stake_e8s` ICP to `governance_canister[target_subaccount]`.
3. Attacker submits `manage_neuron { ClaimOrRefresh { MemoAndController { controller: child_controller, memo: nonce } } }` — a standard unprivileged ingress call.
4. A neuron is created at `target_subaccount` owned by `child_controller`.
5. Victim calls `spawn_neuron` with the same `child_controller` and `nonce`; `ensure_subaccount_available` returns `SubaccountAlreadyExists` and the spawn fails.

No existing guard prevents this: `claim_or_refresh_neuron_by_memo_and_controller` performs no caller-authorization check when a `controller` is explicitly provided, and `ensure_subaccount_available` has no fallback or retry path for deterministic subaccounts.

## Impact Explanation
This is an application-level DoS on the `spawn_neuron` operation for any `(child_controller, nonce)` pair the attacker can predict. Automated or protocol-driven spawning workflows that rely on deterministic nonces (e.g., derived from neuron IDs, timestamps, or sequential counters) are permanently blocked for those nonce values. The victim's maturity is not lost (the spawn simply fails), but the intended nonce is permanently unusable. This matches the **High** impact category: *Application/platform-level DoS with concrete user or protocol harm* on NNS Governance, a core in-scope system.

## Likelihood Explanation
The attack requires no special privileges — only a standard ingress message to the NNS Governance canister. The cost is ≥ 1 ICP per blocked nonce (the attacker's ICP is locked in a neuron they control, not destroyed). The attacker must know the victim's intended `child_controller` and `nonce` in advance, which is feasible when: (a) the victim uses predictable nonces (sequential integers, timestamps, neuron IDs), (b) the victim's tooling is open-source with known nonce derivation, or (c) the victim announces spawn parameters. The attack is repeatable for each new nonce the victim tries if the attacker can observe intent.

## Recommendation
Introduce a spawn-specific domain separator so spawn-derived subaccounts are disjoint from the staking namespace. Replace:

```rust
ledger::compute_neuron_staking_subaccount(child_controller, nonce_val)
```

with a new function in `rs/nervous_system/common/src/ledger.rs`:

```rust
pub fn compute_neuron_spawn_subaccount_bytes(controller: PrincipalId, nonce: u64) -> [u8; 32] {
    compute_neuron_domain_subaccount_bytes(controller, b"neuron-spawn", nonce)
}
```

This is consistent with the existing pattern: `compute_neuron_disburse_subaccount_bytes` uses domain `"neuron-split"` and `compute_neuron_split_subaccount_bytes` uses domain `"split-neuron"` (L22–33 of `rs/nervous_system/common/src/ledger.rs`). A `"neuron-spawn"` domain makes the spawn subaccount namespace unreachable via `ClaimOrRefresh`.
