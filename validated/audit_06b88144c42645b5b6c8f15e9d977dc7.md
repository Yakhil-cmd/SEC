### Title
Permanent Fund Loss in `ExitToEthereum` Precompile Due to Missing Refund Callback — (`engine-precompiles/src/native.rs`)

### Summary
The `ExitToEthereum` precompile burns ERC-20 tokens (or deducts ETH) inside the EVM and then schedules a NEAR promise to call `withdraw` on the eth-connector contract. No error-handling callback is attached to that promise. If the NEAR-side `withdraw` promise fails after the EVM state has already been committed, the user's tokens are permanently destroyed with no recovery path. This is the Aurora analog of the Atlas "simulation success ≠ on-chain success" class: an EVM-level simulation of the transaction succeeds, but the asynchronous NEAR promise that completes the bridge operation can silently fail.

### Finding Description

**Root cause — `ExitToEthereum::run`** in `engine-precompiles/src/native.rs` (lines 977–990):

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

The promise is wrapped as `PromiseArgs::Create` — a fire-and-forget call with **no callback**. [1](#0-0) 

Contrast this with `ExitToNear::run`, which (when the `error_refund` feature is compiled in) wraps the transfer promise in a `PromiseArgs::Callback` that targets `exit_to_near_precompile_callback` on the engine itself:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [2](#0-1) 

The `exit_to_near_precompile_callback` handler inspects the promise result and, on failure, calls `engine::refund_on_error` to re-mint the burned tokens: [3](#0-2) 

`refund_on_error` re-mints ERC-20 tokens (or transfers ETH back from the precompile address) to the original sender: [4](#0-3) 

`ExitToEthereum` has **no equivalent callback and no equivalent refund path**.

The two-phase execution model is:
1. **EVM phase** (synchronous, simulated by relayers): ERC-20 tokens are burned / ETH is deducted. State is committed.
2. **NEAR promise phase** (asynchronous, NOT covered by EVM simulation): `withdraw` is called on the eth-connector. If this fails, the EVM state is already final.

Because relayers simulate only the EVM phase, a simulation that shows success gives no guarantee that the NEAR promise will succeed.

### Impact Explanation

If the eth-connector's `withdraw` promise fails after the EVM burn has been committed:
- The user's ERC-20 tokens (or ETH) are permanently destroyed on Aurora.
- No NEP-141 tokens are released on the NEAR side.
- No Ethereum withdrawal is initiated.
- There is no callback, no re-mint, and no recovery path in the contract.

This constitutes **permanent freezing / destruction of user funds** — Critical impact under the allowed scope.

### Likelihood Explanation

The NEAR `withdraw` promise can fail under several realistic conditions:
- The eth-connector contract is **paused** for maintenance while a user's transaction is in-flight (the EVM execution commits before the pause is observed by the user).
- The eth-connector account is **migrated or replaced** and the old account no longer accepts `withdraw` calls.
- The eth-connector has **insufficient attached NEAR** for storage or gas in an edge-case invocation.
- A bug in the eth-connector causes a panic on certain inputs.

None of these require the attacker to compromise a private key; they are operational conditions that can arise in the normal lifecycle of the Aurora bridge. The likelihood is **Low-Medium**: individually each scenario is uncommon, but the absence of any safety net means any single failure mode results in irreversible loss.

### Recommendation

Add an error-handling callback to `ExitToEthereum` analogous to the `exit_to_near_precompile_callback` pattern already used by `ExitToNear`:

1. Attach a `PromiseArgs::Callback` whose `callback` targets a new `exit_to_ethereum_precompile_callback` method on the engine.
2. In that callback, inspect `promise_result(0)`. On failure, call `refund_on_error` (or an equivalent) to re-mint the burned ERC-20 tokens / restore the ETH balance to the original sender.
3. Ensure the refund address is encoded in the precompile input (as `ExitToNear` does with its `refund_address` field under `error_refund`).

### Proof of Concept

1. User holds 100 ERC-20 tokens on Aurora that represent a bridged NEP-141 asset.
2. User calls the `ExitToEthereum` precompile (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) with flag `0x01`, amount = 100, and their Ethereum recipient address.
3. The EVM execution burns 100 ERC-20 tokens from the user's balance. State is committed. EVM simulation reports success.
4. Aurora schedules `PromiseArgs::Create(withdraw_promise)` targeting the eth-connector's `withdraw` method. [1](#0-0) 
5. The eth-connector is paused between step 3 and step 4's execution (or fails for any other reason). The `withdraw` promise panics/fails.
6. No callback exists. The engine never learns of the failure.
7. The user has lost 100 tokens permanently: they are gone from Aurora's EVM state and were never released on the NEAR/Ethereum side.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
