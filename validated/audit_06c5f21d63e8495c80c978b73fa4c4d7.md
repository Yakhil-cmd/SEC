Audit Report

## Title
`estimate_withdrawal_fee` Calls `mutate_state` Inside a `#[query]` Endpoint, Permanently Draining `available_utxos` When Invoked via Inter-Canister Call - (`rs/bitcoin/ckbtc/minter/src/main.rs`)

## Summary

`estimate_withdrawal_fee` is annotated `#[query]` but calls `mutate_state`, which invokes `greedy` to destructively remove UTXOs from `available_utxos` without re-inserting them. The inline comment claiming "any change will be discarded" is incorrect for inter-canister calls: on the IC, when a canister calls another canister's query endpoint via an inter-canister call, execution runs in replicated mode and state mutations are committed. An unprivileged attacker canister can therefore permanently drain the minter's `available_utxos` set, causing all legitimate `retrieve_btc` calls to fail with `NotEnoughFunds`.

## Finding Description

In `rs/bitcoin/ckbtc/minter/src/main.rs` at line 219, `estimate_withdrawal_fee` is declared `#[query]` but calls `mutate_state` at line 223, passing `&mut s.available_utxos` directly to `ic_ckbtc_minter::estimate_retrieve_btc_fee` at line 226–233. [1](#0-0) 

`estimate_retrieve_btc_fee` delegates to `queries::estimate_withdrawal_fee` in `rs/bitcoin/ckbtc/minter/src/queries.rs`, which calls `utxos_selection(withdrawal_amount, available_utxos, 1)` at line 57, passing the live mutable UTXO set. [2](#0-1) 

`utxos_selection` calls `greedy`, which at line 1083 of `rs/bitcoin/ckbtc/minter/src/lib.rs` calls `available_utxos.remove(&utxo)` for each selected UTXO. When sufficient UTXOs exist, `greedy` returns them in `solution` and they are **never re-inserted** into `available_utxos`. [3](#0-2) 

The comment at line 221–222 asserts state changes are always discarded, but this is only true for direct user-initiated query calls (non-replicated mode). Per the IC specification, when a canister invokes another canister's query method via an inter-canister call, the callee executes in **replicated mode** and state mutations are committed to the replicated state. The `#[query]` CDK macro does not inject any rollback mechanism; rollback only occurs at the system level for non-replicated calls. The ckDoge test harness confirms the correct pattern: its equivalent `estimate_withdrawal_fee` is called via `update_call`, not `query_call`. [4](#0-3) 

Exploit path:
1. Attacker deploys a canister and calls `estimate_withdrawal_fee` on the ckBTC minter via inter-canister call (update context).
2. The minter executes in replicated mode; `greedy` removes UTXOs from `available_utxos` and they are not returned.
3. State is committed. Repeating this loop depletes `available_utxos`.
4. Subsequent `retrieve_btc` / `retrieve_btc_with_approval` calls fail with `NotEnoughFunds` even though the minter's Bitcoin UTXOs on-chain are intact.
5. The minter's internal accounting diverges from the actual on-chain UTXO set.

No existing guard prevents this: there is no `check_anonymous_caller`, no rate limit, and no privilege check on `estimate_withdrawal_fee`.

## Impact Explanation

This is a High-severity application-level DoS with concrete user harm: all ckBTC withdrawal requests fail for all users until a manual canister upgrade re-scans and re-populates the UTXO set. The minter's conservation invariant (internal UTXO accounting matches on-chain holdings) is broken. This matches the allowed impact: **"Significant Chain Fusion, ck-token... security impact with concrete user or protocol harm"** and **"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS."**

## Likelihood Explanation

The attack requires only deploying a canister and calling a public, unauthenticated endpoint in a loop via standard inter-canister calls. No privileged access, key material, governance majority, or special conditions are needed. The cost is only cycles. The endpoint is callable by any principal including anonymous.

## Recommendation

Replace `mutate_state` with `read_state` and operate on a cloned UTXO set so no live state is mutated:

```rust
#[query]
fn estimate_withdrawal_fee(arg: EstimateFeeArg) -> WithdrawalFee {
    read_state(|s| {
        let mut utxos_clone = s.available_utxos.clone();
        let fee_estimator = IC_CANISTER_RUNTIME.fee_estimator(s);
        let withdrawal_amount = arg.amount.unwrap_or(s.fee_based_retrieve_btc_min_amount);
        ic_ckbtc_minter::estimate_retrieve_btc_fee(
            &mut utxos_clone,
            withdrawal_amount,
            s.last_median_fee_per_vbyte.expect("..."),
            s.max_num_inputs_in_transaction,
            &fee_estimator,
        )
    })
    // ... match arms unchanged
}
```

Alternatively, change the annotation to `#[update]` consistent with the ckDoge minter's design, though this increases latency and cost for callers.

## Proof of Concept

```rust
// Attacker canister (deployed on IC mainnet or local replica)
use ic_cdk::api::call::call;

#[update]
async fn drain_ckbtc_utxos(minter: Principal) {
    for _ in 0..500 {
        let _: (WithdrawalFee,) = call(
            minter,
            "estimate_withdrawal_fee",
            (EstimateFeeArg { amount: Some(100_000) },),
        ).await.unwrap();
        // Each call removes UTXOs from minter's available_utxos
        // because inter-canister query calls run in replicated mode
    }
    // minter.available_utxos is now depleted;
    // all subsequent retrieve_btc calls fail with NotEnoughFunds
}
```

A deterministic integration test using PocketIC can confirm this by: (1) setting up the minter with a known UTXO set, (2) calling `estimate_withdrawal_fee` via `update_call` from a second canister, (3) asserting that `available_utxos` is smaller after the call, and (4) verifying that `retrieve_btc` subsequently returns `NotEnoughFunds`.
