Audit Report

## Title
`Command::Spawn` with `new_controller` Unguarded in `validate_manage_neuron_proposal`, Enabling NeuronManagement Followee to Redirect Victim Maturity to Attacker-Controlled Neuron — (`rs/nns/governance/src/governance.rs`)

## Summary

`validate_manage_neuron_proposal` explicitly blocks `Command::Disburse` and `Command::DisburseToNeuron` for non-NFP neurons in the proposal execution path, but `Command::Spawn` falls through the catch-all `_ => {}` branch. When a ManageNeuron proposal embedding `Spawn { new_controller: Some(attacker_principal), .. }` is adopted, `perform_action` re-invokes `manage_neuron` using the victim neuron's own controller as `caller`, satisfying `spawn_neuron`'s `is_controlled_by` guard. The `new_controller` field is then accepted verbatim, creating a new neuron owned by the attacker funded entirely by the victim's maturity.

## Finding Description

**Root cause — missing `Command::Spawn` arm in the non-NFP guard:**

`validate_manage_neuron_proposal` (lines 4686–4703) matches only `Disburse` and `DisburseToNeuron` for non-NFP neurons; `Spawn` falls through `_ => {}` and the function returns `Ok(())`. [1](#0-0) 

**Execution re-uses victim's controller as `caller`:**

`perform_action` fetches the managed neuron's controller and passes it directly as the `caller` argument to `manage_neuron`. [2](#0-1) 

**`spawn_neuron` authorization check is trivially satisfied:**

Because `caller` is set to the victim neuron's own controller, `is_controlled_by(caller)` returns `true` and execution proceeds. [3](#0-2) 

**`new_controller` accepted without further authorization:**

The attacker-supplied principal is used directly as `child_controller` with no check that the entity who initiated the proposal is authorized to redirect ownership to a third party. [4](#0-3) 

The full exploit chain: attacker (NeuronManagement followee) → `MakeProposal(ManageNeuron { Spawn { new_controller: attacker } })` → proposal adopted → `perform_action` → `manage_neuron(victim_controller, Spawn)` → `spawn_neuron` passes auth → new neuron created with `controller = attacker`, funded by victim's maturity.

## Impact Explanation

The spawned neuron is created with `child_controller = attacker_principal` and is funded by the victim neuron's maturity (up to 100%). Maturity represents accumulated staking rewards convertible to ICP stake. This is functionally equivalent to `DisburseToNeuron`, which is explicitly blocked for non-NFP neurons in the proposal path, but achieved via the unguarded `Spawn` path. This constitutes unauthorized access to and theft of governance assets (neuron maturity) — matching the **High** bounty impact: *"Unauthorized access to neurons, governance assets… where exploitation requires meaningful per-target work or other constraints."* The constraint is the NeuronManagement followee relationship.

## Likelihood Explanation

The precondition — attacker's neuron being a NeuronManagement followee of the victim — is a real-world configuration used for governance delegation. Once that relationship exists, the attack is a single `manage_neuron` call (MakeProposal) followed by automatic vote adoption (if the attacker is the sole or majority followee), with no further privileges required. The attack is repeatable against any victim neuron for which the attacker holds followee status.

## Recommendation

Add `Command::Spawn(_)` to the non-NFP block inside `validate_manage_neuron_proposal`, mirroring the treatment of `Disburse` and `DisburseToNeuron`:

```rust
Command::Spawn(_) => {
    return Err(GovernanceError::new_with_message(
        ErrorType::NotAuthorized,
        "Cannot issue a spawn command through a proposal",
    ));
}
```

Alternatively, if spawning via proposal is intentional, restrict `new_controller` to `None` when the command arrives through the proposal execution path, forcing the spawned neuron to inherit the parent's controller.

## Proof of Concept

State-machine test outline:

1. Create victim neuron with sufficient `maturity_e8s_equivalent` and controller `victim_principal`.
2. Create attacker neuron with controller `attacker_principal`.
3. Set attacker neuron as the sole NeuronManagement followee of victim neuron.
4. Attacker calls `manage_neuron` with `MakeProposal` wrapping `ManageNeuron { neuron_id: victim_id, command: Spawn { new_controller: Some(attacker_principal), percentage_to_spawn: Some(100), nonce: None } }`.
5. Assert proposal is accepted (no validation error from `validate_manage_neuron_proposal`).
6. Tick governance to process the adopted proposal via `perform_action`.
7. Assert a new neuron exists with `controller == attacker_principal` and `maturity_e8s_equivalent` equal to the victim's original maturity.
8. Assert victim neuron's `maturity_e8s_equivalent` is now 0.
