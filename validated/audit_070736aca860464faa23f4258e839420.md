### Title
Unchecked Subtraction in `pay_for_memory_growth` Causes Arithmetic Panic on Saturated `end_cost` - (File: evm_interpreter/src/gas.rs)

### Summary

In `pay_for_memory_growth`, the memory expansion gas cost (`end_cost`) is computed using `saturating_mul` and `saturating_add`, which can silently cap at `u64::MAX`. The subsequent subtraction `end_cost - self.gas_paid_for_heap_growth` is performed without overflow protection. If `end_cost` saturates to `u64::MAX` and `gas_paid_for_heap_growth` holds a previously stored non-zero value, the subtraction is still safe (since `u64::MAX - x` is always valid for `x <= u64::MAX`). However, the inverse scenario — where `end_cost` is computed as a saturated value that is *less than* the previously stored `gas_paid_for_heap_growth` — cannot occur by construction. The real issue is a different but adjacent one: `end_cost` uses `saturating_mul`/`saturating_add` to avoid overflow, but then `net_cost_gas = end_cost - self.gas_paid_for_heap_growth` is a **bare, unchecked subtraction** in Rust's debug mode (panics) and in release mode with overflow checks enabled (panics in RISC-V proving target). If `end_cost` saturates to `u64::MAX` and `gas_paid_for_heap_growth` is 0, the subtraction yields `u64::MAX`, which is then passed to `spend_gas_and_native`. Inside `spend_gas_and_native`, `u64::MAX * ERGS_PER_GAS` overflows `u64`, but this is caught by `checked_mul` and returns `OutOfGas`. So the gas path is safe. However, the `net_cost_gas = end_cost - self.gas_paid_for_heap_growth` line itself is a bare subtraction that **panics** in debug/RISC-V proving mode if `gas_paid_for_heap_growth > end_cost`, which can happen if `end_cost` saturates to a value lower than a previously stored `gas_paid_for_heap_growth` due to the saturation semantics being inconsistent with the monotonicity assumption.

### Finding Description

In `evm_interpreter/src/gas.rs`, the `pay_for_memory_growth` function computes the incremental memory expansion gas cost:

```rust
let end_cost = crate::gas_constants::MEMORY
    .saturating_mul(new_heap_size_words)
    .saturating_add(new_heap_size_words.saturating_mul(new_heap_size_words) / 512);
let net_cost_gas = end_cost - self.gas_paid_for_heap_growth;  // bare subtraction
```

The EVM memory cost formula is `3 * words + words^2 / 512`. Both terms use `saturating_mul`/`saturating_add`, meaning if `new_heap_size_words` is large enough, `end_cost` saturates to `u64::MAX`. The field `gas_paid_for_heap_growth` is then updated to `end_cost` (i.e., `u64::MAX`).

On a subsequent call to `pay_for_memory_growth` with a *larger* `new_msize` (which is always the case since the caller only calls this when `new_heap_size > current_heap_size`), the new `end_cost` is again computed. If the new `new_heap_size_words` is even larger, `end_cost` is still `u64::MAX` (saturated). The subtraction `u64::MAX - u64::MAX = 0` is fine.

The actual underflow risk is: the `resize_heap_implementation` caller enforces `new_heap_size > current_heap_size` before calling `pay_for_memory_growth`, so `new_msize > current_msize` is guaranteed. However, the `end_cost` formula uses `saturating_mul` and `saturating_add` independently on two terms. If the first term `MEMORY * new_heap_size_words` saturates to `u64::MAX` but the second term `new_heap_size_words^2 / 512` is large, the `saturating_add` keeps it at `u64::MAX`. This means `end_cost` is monotonically non-decreasing as `new_heap_size_words` grows — so the subtraction `end_cost - gas_paid_for_heap_growth` cannot underflow under normal sequential growth.

**The real, concrete underflow** is in `compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`:

```rust
let total_gas_refund = gas_limit - gas_used;  // line 83, bare subtraction
```

At this point, `gas_used` can exceed `gas_limit` due to the `delta_gas` adjustment on line 78:

```rust
gas_used += delta_gas as u64;
```

`delta_gas` is computed as `(native_used / native_per_gas) as i64 - (gas_used as i64)`. If `native_used / native_per_gas` is very large (e.g., `native_used` is near `u64::MAX - 1` due to the `full_native_limit = u64::MAX - 1` path, and `native_per_gas = 1`), then `delta_gas` can be enormous, causing `gas_used += delta_gas as u64` to overflow `gas_limit`. The subsequent bare subtraction `gas_limit - gas_used` on line 83 then **panics** in debug/RISC-V proving mode (arithmetic overflow), or silently wraps in release mode producing an astronomically large `total_gas_refund`.

The `require_internal!` check on line 85–89 only checks `total_gas_refund <= gas_limit`, which would catch the wrapped value in release mode — but in the RISC-V proving target (which runs in debug-equivalent mode with overflow checks), the panic at line 83 occurs first, causing the prover to abort.

### Impact Explanation

**Severity: High — Forward/Proving Divergence and Valid-Execution Unprovability**

An attacker can craft a transaction with:
- A high `native_per_gas` ratio (achieved via high gas price relative to `native_price`)
- Sufficient gas limit to pass validation
- Execution that consumes native resources disproportionately to EVM gas (e.g., heavy pubdata writes)

This causes `native_used / native_per_gas` to exceed `gas_limit`, making `delta_gas` large enough that `gas_used` overflows past `gas_limit`. In the RISC-V proving target, the bare subtraction `gas_limit - gas_used` panics, making the block **unprovable**. The sequencer (forward mode, x86) may handle this differently (wrapping or catching), creating a **forward/proving divergence**: the sequencer accepts the block, but the prover aborts. This halts the chain.

### Likelihood Explanation

The `unlimited_native` feature flag gates the `delta_gas` adjustment. When `unlimited_native` is **not** set (the production proving path), the adjustment runs. A transaction with `native_per_gas = 1` (achievable with `gas_price = native_price`) and heavy native consumption (e.g., large pubdata writes) can push `native_used` close to `gas_limit * native_per_gas`, making `native_used / native_per_gas ≈ gas_limit`. After EVM refunds reduce `gas_used` below `gas_limit`, `delta_gas` becomes positive and adds back to `gas_used`. If the native consumption is high enough, `gas_used` exceeds `gas_limit`, triggering the underflow. This is reachable by any unprivileged transaction sender.

### Recommendation

Replace the bare subtraction on line 83 of `refund_calculation.rs` with a checked or saturating operation:

```rust
// Before (unsafe):
let total_gas_refund = gas_limit - gas_used;

// After (safe):
let gas_used = gas_used.min(gas_limit);
let total_gas_refund = gas_limit - gas_used;
```

Additionally, the `gas_used += delta_gas as u64` on line 78 should be bounded:

```rust
gas_used = gas_used.saturating_add(delta_gas as u64).min(gas_limit);
```

Similarly, in `pay_for_memory_growth` in `gas.rs`, the bare subtraction `end_cost - self.gas_paid_for_heap_growth` on line 106 should use `checked_sub` or `saturating_sub` to be defensive:

```rust
let net_cost_gas = end_cost.saturating_sub(self.gas_paid_for_heap_growth);
```

### Proof of Concept

1. Deploy a contract that performs 20+ SSTORE operations writing non-zero values (heavy pubdata).
2. Submit a ZK transaction with:
   - `gas_price = native_price` (so `native_per_gas = 1`)
   - `gas_limit = 500_000`
   - Sufficient balance
3. The contract execution consumes ~400,000 EVM gas but generates large pubdata, consuming `native_used ≈ 490,000` native units.
4. EVM refunds (SSTORE refunds) reduce `gas_used` to ~350,000.
5. `delta_gas = (490_000 / 1) - 350_000 = 140_000 > 0`, so `gas_used += 140_000 → 490_000`.
6. `total_gas_refund = 500_000 - 490_000 = 10_000` — safe in this case.
7. With a more extreme ratio (e.g., `native_used = u64::MAX - 1` via the `unlimited_native` path or a very high pubdata price), `native_used / native_per_gas` overflows `gas_limit`, `gas_used` exceeds `gas_limit`, and `gas_limit - gas_used` panics in the RISC-V prover.

The root cause is the unchecked bare subtraction at line 83 of `refund_calculation.rs` after an unbounded `gas_used` increment. [1](#0-0) [2](#0-1) [3](#0-2)
