Audit Report

## Title
Unbounded `ingress_expiries` Allocation via Attacker-Controlled `ingress_start`/`ingress_end` Causes Rosetta Node OOM — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `construction_payloads` handler accepts user-supplied `ingress_start` and `ingress_end` nanosecond timestamps with no server-side cap on their span. A while-loop iterates `(ingress_end - ingress_start) / interval` times, pushing one `u64` per iteration into an unbounded `Vec`. With `ingress_start=0` and `ingress_end=u64::MAX`, the loop attempts ~153 billion iterations requiring ~1.23 TB of heap, crashing the Rosetta process via OOM before completion.

## Finding Description

**Interval computation:** `interval = MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s = 300s - 60s - 120s = 120s = 120_000_000_000 ns`. [1](#0-0) [2](#0-1) 

**Unbounded loop:** `ingress_start` and `ingress_end` are taken directly from attacker-controlled metadata with no validation of their span. [3](#0-2) 

The loop at lines 99–107 pushes one `u64` per iteration with no bound check: [4](#0-3) 

With `ingress_start=0`, `ingress_end=u64::MAX`: iterations ≈ `u64::MAX / 120_000_000_000 ≈ 153,722,867,280`; memory for `ingress_expiries` alone ≈ `153.7B × 8 bytes ≈ 1.23 TB`.

**Overflow / infinite-loop risk:** `Time::add` and `AddAssign` use `Duration::from_nanos(self.0) + dur`, then cast the resulting `u128` back to `u64` via a truncating `as u64` cast in `from_duration`. When `now` approaches `u64::MAX`, the addition overflows into `u128` territory and the truncating cast wraps `now` back to a small value, which remains `< ingress_end`, making the loop condition permanently true — an infinite loop if the process somehow survived the OOM. [5](#0-4) [6](#0-5) 

**No existing guards in the ICP handler:** The ICP Rosetta handler performs zero validation on the ingress span. By contrast, the ICRC1 handler at least checks `ingress_start < ingress_end` and `ingress_end >= now + interval` (though it also lacks a span cap): [7](#0-6) 

**`add_payloads` multiplier:** For each entry in `ingress_expiries`, `add_payloads` pushes **two** `SigningPayload` objects into `payloads` (one for the call, one for `read_state`), multiplying memory usage by the number of transactions in the request: [8](#0-7) 

## Impact Explanation

The Rosetta node process crashes with OOM. Any exchange, wallet, or integration relying on the ICP Rosetta API for transaction construction is denied service. This matches the allowed bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS**, specifically targeting the in-scope Rosetta API financial integration component.

## Likelihood Explanation

The `/construction/payloads` HTTP endpoint is publicly reachable by design and requires no authentication. The malicious payload is a small JSON object. A single unauthenticated HTTP POST is sufficient to trigger the crash. The attack is trivially repeatable to prevent recovery after restart.

## Recommendation

Add a span cap immediately before the loop in `construction_payloads`:

```rust
let max_span = interval * MAX_ALLOWED_INTERVALS; // e.g., 48 * 60 * 60 seconds
if ingress_end.saturating_duration_since(ingress_start) > max_span {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed span",
    ));
}
```

Also add the same basic ordering check present in the ICRC1 handler (`ingress_start < ingress_end`). Additionally, fix `Time::add` / `AddAssign` in `rs/types/types/src/time.rs` to use saturating or checked arithmetic instead of the truncating `as u64` cast to eliminate the infinite-loop risk. [6](#0-5) 

## Proof of Concept

```
POST /construction/payloads HTTP/1.1
Host: <rosetta-node>:8080
Content-Type: application/json

{
  "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
  "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<valid_account>"},"amount":{"value":"-100000000","currency":{"symbol":"ICP","decimals":8}}}],
  "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

The Rosetta process will attempt to allocate ~1.23 TB for `ingress_expiries`, exhaust available memory, and crash (SIGKILL / OOM). A unit test can confirm the behavior by calling `construction_payloads` with these metadata values and asserting that it returns an error rather than attempting allocation.
