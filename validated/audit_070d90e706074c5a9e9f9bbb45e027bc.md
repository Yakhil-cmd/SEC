### Title
Non-Atomic `transfer_from_treasury` Leaves Tokens Permanently Unaccounted After Post-Execution Mint Failure — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`mint_base_token` first notifies the `L2AssetTracker` (committing its state to the parent frame), then calls `transfer_from_treasury`, which performs two separate, non-atomic balance mutations: debit treasury (Step 1), credit recipient (Step 2). If Step 1 succeeds but Step 2 fails for a **post-execution mint** (operator fee or refund), the treasury is permanently debited without the recipient being credited. The `tx_loop` error handler commits — not rolls back — the frame on `TxError::Internal`, making the token loss and asset-tracker inconsistency permanent. The developers themselves flagged this exact commit-vs-rollback decision with a `TODO`.

---

### Finding Description

**`mint_base_token` call sequence** (`process_l1_transaction.rs:757-768`):

```
notify_l2_asset_tracker(...)   // 1. asset tracker records deposit — frame committed
transfer_from_treasury(...)    // 2. debit treasury, then credit recipient
```

**`transfer_from_treasury` is non-atomic** (`process_l1_transaction.rs:793-831`):

```rust
// Step 1 — debit treasury
system.io.update_account_nominal_token_balance(treasury_address, amount, true, ...)?;
// Step 2 — credit recipient
system.io.update_account_nominal_token_balance(to, amount, false, ...)?;
``` [1](#0-0) 

If Step 1 succeeds but Step 2 fails (`MintingBalanceOverflow`), the treasury has been debited but the recipient has not been credited. Tokens are in limbo — neither in the treasury nor in the recipient's account.

**Post-execution mints have no rollback path.** The operator-fee mint (line 290–309) and refund mint (line 338–359) both run on `FORMAL_INFINITE` resources after the main execution frame has already been committed or rolled back: [2](#0-1) [3](#0-2) 

When either of these fails, the error propagates up through `process_transaction` as `TxError::Internal`. In `tx_loop.rs`, the handler **commits** the frame:

```rust
Err(TxError::Internal(err)) => {
    // Finish the frame opened before processing the tx
    system.finish_global_frame(None)?; // TODO should we use pre_tx_rollback_handle here?
    return Err(err);
}
``` [4](#0-3) 

`finish_global_frame(None)` = **commit**, not rollback. The `TODO` comment is the developers' own acknowledgment that this is the wrong choice. The partial state — treasury debited, recipient not credited, asset tracker already updated — is permanently committed to the chain state.

**The `notify_l2_asset_tracker` ordering compounds the issue.** The asset tracker is notified *before* the balance transfer: [5](#0-4) 

If `transfer_from_treasury` fails at Step 1 (treasury insufficient), the asset tracker has already recorded a deposit that never completed. For post-execution mints, this inconsistency is also permanently committed.

**The refund recipient is attacker-controlled.** `transaction.reserved[1]` is set by the L1 transaction sender: [6](#0-5) 

An attacker can point the refund recipient at any address, including one with a balance near `U256::MAX`.

---

### Impact Explanation

- **Token loss**: Treasury is permanently debited without the recipient being credited. The total circulating supply decreases by the refund or operator-fee amount.
- **Asset tracker inconsistency**: `L2AssetTracker.totalSuccessfulDepositsFromL1` records a deposit that did not fully materialize, corrupting the bridge accounting used for asset migration decisions (`_needToForceSetAssetMigrationOnL2`).
- **Block halt**: The fatal internal error propagates out of `tx_loop`, halting block processing.

This is the direct analog to the OmoVault bug: funds are in an intermediate state (debited from treasury, not yet credited to recipient) that is not captured by any accounting function, corrupting the system's token supply invariant.

---

### Likelihood Explanation

**Low.** The `MintingBalanceOverflow` path (Step 2 failure) requires the recipient's balance to be near `U256::MAX`, which is practically impossible under normal economic conditions. The `TreasuryTransferFailed` path (Step 1 failure) requires the treasury to be underfunded relative to the post-execution mint amounts — possible only if the treasury was not correctly pre-funded or if a separate bug depleted it. The `TODO` comment in `tx_loop.rs:111` confirms the developers are aware the commit-on-internal-error behavior is questionable, raising the probability that this path is reachable under edge conditions not yet fully analyzed.

---

### Recommendation

1. **Fix the frame commit on internal error** in `tx_loop.rs:111`: replace `system.finish_global_frame(None)` with `system.finish_global_frame(Some(&pre_tx_rollback_handle))` to roll back all partial state on fatal errors, consistent with the `TODO` comment already present.

2. **Make `transfer_from_treasury` atomic**: wrap both balance mutations in a sub-frame so that if Step 2 fails, Step 1 is automatically rolled back.

3. **Reorder `mint_base_token`**: call `transfer_from_treasury` *before* `notify_l2_asset_tracker` so the asset tracker is only notified after the balance transfer has succeeded, eliminating the window where the asset tracker records a deposit that did not complete.

---

### Proof of Concept

1. Attacker accumulates balance `B ≈ U256::MAX` at address `R`.
2. Attacker submits an L1→L2 transaction with `reserved[1] = R` (refund recipient = `R`).
3. Pre-execution mint succeeds; main tx body executes and succeeds.
4. Post-execution refund mint begins:
   - `notify_l2_asset_tracker(refund_amount)` → asset tracker records `refund_amount` as deposited; state committed to parent frame.
   - `transfer_from_treasury` Step 1: treasury debited by `refund_amount` → succeeds.
   - `transfer_from_treasury` Step 2: `R.balance + refund_amount` overflows → `MintingBalanceOverflow` returned.
5. Error propagates as `TxError::Internal` to `tx_loop.rs:108`.
6. `system.finish_global_frame(None)` commits the frame.
7. **Result**: treasury permanently short by `refund_amount`; `R` receives nothing; asset tracker records a deposit that never completed; block processing halts.
