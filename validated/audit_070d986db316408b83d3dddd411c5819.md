Audit Report

## Title
Single Failed `get_transaction_receipt` Call Aborts Entire Finalization Batch — (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

In `finalize_transactions_batch`, when any single `get_transaction_receipt` call returns `Err`, the function immediately `return`s at line 442, discarding all successfully fetched receipts for other pending withdrawals in the same batch. Because `NoReduction` returns `Err(InconsistentResults)` whenever any EVM RPC provider disagrees with the others — a realistic, non-adversarial condition — every ckETH and ckERC20 withdrawal whose nonce is below the on-chain finalized count is blocked from completing in that timer tick. A secondary `assert_eq!` at lines 447–450 can panic the canister if all transaction hashes for a finalized nonce return `Ok(None)`, causing a full message rollback.

## Finding Description

`finalize_transactions_batch` (line 386) collects all transaction hashes for nonces below the on-chain finalized count via `sent_transactions_to_finalize`, fires all `get_transaction_receipt` calls in parallel via `join_all`, then iterates over results. The `Err` arm at line 437–443 calls `return`, exiting the entire function:

```rust
Err(e) => {
    log!(INFO, "Failed to get transaction receipt for {hash} ...: {e:?}. Will retry later");
    return;   // line 442 — aborts the entire batch
}
```

The `NoReduction` strategy (lines 177–182) returns `Err(MultiCallError::InconsistentResults(...))` whenever any provider's response differs from the others — not only when all providers fail. `eth_getTransactionReceipt` is particularly susceptible to this because providers may have slightly different chain views at any moment (one provider lagging, returning `null` while others return the receipt). This is a normal, non-adversarial network condition.

By contrast, `send_transactions_batch` (line 340) uses `AnyOf` (succeeds if at least one provider agrees) and handles errors per-item without aborting the loop (lines 363–383), demonstrating the intended resilient pattern.

The secondary risk: after the loop, `assert_eq!(expected_finalized_withdrawal_ids, actual_finalized_withdrawal_ids, ...)` at lines 447–450 panics if any nonce below the finalized count has all its transaction hashes return `Ok(None)`. On the IC, a panic in a timer callback traps the message and rolls back all state, preventing any finalization until the next successful invocation.

## Impact Explanation

Every ckETH and ckERC20 withdrawal in the `sent_tx` state with a nonce below the on-chain finalized count is blocked from completing in the affected timer tick. Users experience delayed withdrawals. If the provider inconsistency persists across ticks (e.g., a provider is consistently behind), the delay extends across multiple timer cycles. This constitutes a concrete, repeated availability impact on Chain Fusion ck-token withdrawals — matching the allowed ICP bounty impact: **High ($2,000–$10,000): Significant Chain Fusion, ck-token, ledger security impact with concrete user or protocol harm.** In the assert-panic path, the canister traps and rolls back state, compounding the delay.

## Likelihood Explanation

No attacker action is required. The failure is triggered by natural EVM RPC provider disagreement on `eth_getTransactionReceipt`, which occurs during normal mainnet operation whenever one of the four configured providers is slightly behind the chain tip. The `NoReduction` strategy makes this particularly fragile: a single provider returning a different result (including `null` vs. a receipt) is sufficient to produce `Err(InconsistentResults)` and abort the entire batch. With 4 providers queried per call and multiple calls per batch, the probability of at least one inconsistency per batch is non-trivial during periods of chain activity or provider latency.

## Recommendation

1. **Per-item error handling**: Replace `return` at line 442 with `continue`, so a single failed receipt fetch does not abort processing of all other withdrawals. Log the failure and retry only the failed item on the next tick.
2. **Relax the reduction strategy**: Use a majority-based strategy (e.g., `StrictMajorityByKey`) for `get_transaction_receipt` instead of `NoReduction`, consistent with how `send_raw_transaction` uses `AnyOf`.
3. **Guard the assert**: Replace the `assert_eq!` at lines 447–450 with a graceful error log and early return to prevent a canister trap in the unexpected-state scenario.

## Proof of Concept

1. Two ckETH withdrawals are queued: burn index 1 (nonce 5), burn index 2 (nonce 6). Both nonces advance below the on-chain finalized transaction count.
2. `finalize_transactions_batch` is invoked by the timer.
3. `sent_transactions_to_finalize` returns transaction hashes for both nonces (including any resubmissions).
4. `join_all` fires all `get_transaction_receipt` calls in parallel.
5. One provider returns a different result for nonce 5's hash (e.g., `null` vs. a receipt from the other three providers). `NoReduction` returns `Err(InconsistentResults)`.
6. The loop hits line 442 and `return`s immediately.
7. The receipt for nonce 6 (burn index 2), successfully fetched, is silently discarded.
8. Neither withdrawal is finalized. Both users wait for the next timer tick.
9. A deterministic integration test can reproduce this by mocking one EVM RPC provider to return `null` for a specific transaction hash while others return a valid receipt, then asserting that `finalize_transactions_batch` finalizes zero withdrawals despite valid receipts being available for other nonces.
