Audit Report

## Title
Unsupported ERC-20 Tokens Deposited via `depositErc20` Are Permanently Irrecoverable at the ckETH Minter Address - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol, rs/ethereum/cketh/minter/src/eth_logs/scraping.rs)

## Summary
The `depositErc20` function in `DepositHelperWithSubaccount.sol` accepts any ERC-20 token address without enforcing a supported-token whitelist, unconditionally transferring tokens to the minter's Ethereum address. The ckETH minter's log scraping restricts `eth_getLogs` topic filters to only currently supported `ckerc20_tokens`, so any `ReceivedEthOrErc20` event for an unsupported token is never fetched, never processed, and no ckERC20 is minted. No recovery or rescue endpoint exists in the minter canister, making the loss permanent.

## Finding Description
**Deposit path — no whitelist:**

`depositErc20` in `DepositHelperWithSubaccount.sol` performs only a zero-address check before calling `safeTransferFrom` to the minter address and emitting `ReceivedEthOrErc20`: [1](#0-0) 

There is no check against any supported-token list at the Solidity level.

**Log scraping path — strict whitelist:**

`ReceivedErc20LogScraping::next_scrape` builds its `eth_getLogs` topic filter using only `erc20_smart_contracts_addresses_as_topics(state)`, which iterates `state.ckerc20_tokens`: [2](#0-1) 

`ReceivedEthOrErc20LogScraping::next_scrape` (the subaccount-aware scraper) applies the same restriction, prepending only the zero address (for ETH) to the supported ERC-20 addresses: [3](#0-2) 

The helper function confirms the filter is derived exclusively from `state.ckerc20_tokens`: [4](#0-3) 

Any `ReceivedEthOrErc20` event whose token address is not in `ckerc20_tokens` is never returned by the RPC provider query and is therefore never processed.

**No recovery path:**

The `withdraw_erc20` endpoint enforces `find_ck_erc20_token_by_ledger_id` and returns `TokenNotSupported` for any unrecognized ledger ID: [5](#0-4) 

No admin, governance, or rescue endpoint exists in `main.rs` to transfer stuck ERC-20 tokens out of the minter's Ethereum address.

## Impact Explanation
This is a concrete, permanent loss of ERC-20 assets held at the minter's Ethereum address. Real token value enters the system via `safeTransferFrom` but can never exit: no ckERC20 is minted, no refund is issued, and no recovery path exists. This constitutes a direct ledger conservation violation in the ckERC20/Chain Fusion financial integration, which is explicitly in scope. The impact maps to **High ($2,000–$10,000): Significant Chain Fusion / ck-token security impact with concrete user or protocol harm**, and could reach Critical if aggregate losses across users exceed $1M.

## Likelihood Explanation
The exploit is reachable by any unprivileged Ethereum user with no special access. The most realistic and non-user-error trigger is a token that was previously in `ckerc20_tokens` and was subsequently removed from the supported list: a user who deposited correctly before removal may deposit again after removal without knowing the token was delisted, and the contract silently accepts and permanently locks the tokens. Additional triggers include depositing a token not yet added to the supported list or using an incorrect ERC-20 address. The contract provides no on-chain feedback to the depositor that the token is unsupported.

## Recommendation
Add a supported-token whitelist check inside `depositErc20` in `DepositHelperWithSubaccount.sol`, reverting if `erc20Address` is not in the minter's supported list (queryable via a view function or passed as a constructor-set mapping). Alternatively, add a governance-controlled rescue function in the minter canister that can issue an Ethereum transaction to transfer stuck ERC-20 tokens to a designated recovery address. The fix must close the asymmetry: deposit must be at least as restrictive as retrieval.

## Proof of Concept
1. Query `get_minter_info` on the ckETH minter to obtain `deposit_with_subaccount_helper_contract_address` and `supported_ckerc20_tokens`.
2. Select any ERC-20 token address **not** present in `supported_ckerc20_tokens` (e.g., a previously supported token that was removed, or any arbitrary ERC-20).
3. Call `approve(helper_contract_address, amount)` on that ERC-20 contract from a test account.
4. Call `depositErc20(unsupported_erc20_address, amount, principal, subaccount)` on the `CkDeposit` helper. `safeTransferFrom` succeeds; tokens move to `minterAddress`; `ReceivedEthOrErc20` is emitted.
5. Observe that `ReceivedEthOrErc20LogScraping::next_scrape` constructs a topic filter containing only supported token addresses. The RPC `eth_getLogs` call never returns the event for `unsupported_erc20_address`.
6. Confirm no ckERC20 is minted. Confirm `withdraw_erc20` returns `TokenNotSupported`. Confirm no other minter endpoint can recover the tokens. The tokens remain permanently locked at the minter's Ethereum address.

    function depositErc20(
        address erc20Address,
        uint256 amount,
        bytes32 principal,
        bytes32 subaccount
    ) public {
        require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
        IERC20 erc20Token = IERC20(erc20Address);
        erc20Token.safeTransferFrom(
            msg.sender,
            minterAddress,
            amount
        );

        emit ReceivedEthOrErc20(
            erc20Address,
            msg.sender,
            amount,
            principal,
            subaccount
        );
    }
