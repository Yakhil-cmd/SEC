### Title
Non-expiring ICRC-2 Approvals Bypass `prune()` Causing Unbounded AllowanceTable Growth — (`rs/ledger_suite/common/ledger_core/src/approvals.rs`)

### Summary
`AllowanceTable::prune()` only removes entries present in the `expiration_queue`. Non-expiring approvals (`expires_at = None`) are never inserted into `expiration_queue`, so they are never pruned. An unprivileged attacker controlling many funded accounts can issue `icrc2_approve` calls with no expiry, causing the `allowances` BTreeMap to grow without bound until the ledger canister exhausts its heap memory.

### Finding Description
In `approve()`, the `insert_expiry()` call is guarded by `if let Some(expires_at) = expires_at`, so non-expiring approvals are stored only in `allowances` and never in `expiration_queue`. In `prune()`, the loop immediately returns when `first_expiry()` returns `None` (empty queue). Since `apply_transaction` calls `prune(now, APPROVE_PRUNE_LIMIT)` on every transaction, and this call is a no-op when all approvals are non-expiring, the `allowances` BTreeMap accumulates entries permanently. There is no total-allowance cap, no per-account approval limit, and no alternative cleanup path for non-expiring entries (they are only removed when explicitly revoked via `amount = 0` or fully consumed via `use_allowance`).

### Impact Explanation
Each unique `(account, spender)` pair with a non-expiring approval occupies a permanent heap entry. With enough entries, the ledger canister's heap is exhausted, causing allocation failures and canister traps. This renders the ICP or ICRC-1/2 ledger canister unavailable — a critical DoS on a core protocol component.

### Likelihood Explanation
The economic barrier is real: each approval requires a transaction fee and a funded account. However, the throttle mechanism (`max_transactions_in_window`) limits rate, not total volume. A patient, well-funded attacker can accumulate entries over time. The IC heap limit (~4 GB) and ~100–200 bytes per entry means ~20–40 million entries are needed for exhaustion — expensive but not infeasible for a motivated attacker targeting a high-value ledger.

### Recommendation
1. Introduce a maximum total allowance count (e.g., reject `icrc2_approve` when `len_allowances()` exceeds a configured cap).
2. Alternatively, track non-expiring approvals in a separate bounded structure (e.g., an arrival queue) and prune oldest non-expiring approvals when the table exceeds a threshold.
3. Consider charging a higher fee for non-expiring approvals or requiring a deposit held until the approval is revoked.

    pub fn prune(&mut self, now: TimeStamp, limit: usize) -> usize {
        self.with_postconditions_check(|table| {
            let mut pruned = 0;
            for _ in 0..limit {
                match table.allowances_data.first_expiry() {
                    Some((ts, _key)) => {
                        if ts > now {
                            return pruned;
                        }
                    }
                    None => {
                        return pruned;
                    }
                }
                if let Some((_, (account, spender))) = table.allowances_data.pop_first_expiry() {
                    let key = (account, spender);
                    if let Some(allowance) = table.allowances_data.get_allowance(&key)
                        && allowance.expires_at.unwrap_or_else(remote_future) <= now
                    {
                        table.allowances_data.remove_allowance(&key);
                        pruned += 1;
                    }
                }
            }
            pruned
        })
    }
