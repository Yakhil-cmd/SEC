Audit Report

## Title
Missing `is_canonical()` Check Allows Identity-Element TLS Certificate Registration and Node Impersonation at Boundary Nodes — (`rs/crypto/node_key_validation/tls_cert_validation/src/lib.rs`)

## Summary

`verify_ed25519_public_key` checks only `is_torsion_free()`, not `is_canonical()`. The three non-canonical encodings of the Ed25519 identity element are torsion-free (confirmed by the codebase's own test), so they pass this check. Because `ic_ed25519::PublicKey::verify_signature` implements ZIP215, a certificate whose public key is the identity element can be self-signed with a trivially forgeable signature `(R=[S]B, S)`. This allows a registered node operator to store such a certificate in the registry, after which any party can impersonate that node to boundary nodes by presenting a forged certificate that passes the boundary node's `verify_signature` call.

## Finding Description

**Root cause — missing canonicality check:**

`verify_ed25519_public_key` in `rs/crypto/node_key_validation/tls_cert_validation/src/lib.rs` only calls `is_torsion_free()`: [1](#0-0) 

The `ic_ed25519` crate explicitly separates `is_torsion_free` and `is_canonical` as distinct properties: [2](#0-1) 

The codebase's own test confirms that the three non-canonical encodings of the identity element are torsion-free but not canonical: [3](#0-2) 

**Self-signature forgery under ZIP215:**

`verify_ed25519_signature` calls `public_key.verify_signature`, which is documented and implemented as ZIP215: [4](#0-3) 

The verification equation is `(recomputed_r - signature.r()).mul_by_cofactor().is_identity()`. With `A = identity`, `minus_a = identity`, so `recomputed_r = [S]B + [k]·identity = [S]B`. The check reduces to `([S]B − R)·8 == identity`, satisfied for any message by choosing any scalar `S` and setting `R = [S]B`. The attacker constructs a TBSCertificate with the identity-element public key, picks `S=1`, sets `R=B`, and the self-signature passes `verify_ed25519_signature` regardless of TBSCertificate content.

**Registration path:**

`ValidTlsCertificate::try_from` calls `verify_ed25519_public_key` (only `is_torsion_free()`) then `verify_ed25519_signature` (ZIP215, forgeable): [5](#0-4) 

The crafted certificate passes all checks and is stored in the registry via `make_add_node_registry_mutations`.

**Downstream impact — boundary node `TlsVerifier`:**

`verify_server_cert` in `rs/boundary_node/ic_boundary/src/tls_verify.rs` extracts the public key from the registry certificate and calls `verify_signature` (ZIP215) on the presented certificate: [6](#0-5) 

With the identity element as the registry public key, any presented certificate whose self-signature satisfies `R = [S]B` passes this check. An attacker presents a certificate with the identity-element public key and a forged self-signature, and the boundary node accepts it as the registered node.

**Downstream impact — node-to-node TLS verifier:**

`verify_tls13_signature` delegates to `rustls::crypto::ring::default_provider()`: [7](#0-6) 

Whether ring rejects the identity element as a small-order point before reaching the verification equation is not confirmed from available source. If ring does not reject it, the same forgery applies to node-to-node TLS. If ring does reject it, the node-to-node path is blocked, but the boundary node impersonation path remains fully exploitable.

The same gap exists in `ValidNodeSigningPublicKey::try_from`: [8](#0-7) 

## Impact Explanation

A registered node operator can register a node whose TLS certificate carries the identity-element public key. The boundary node `TlsVerifier` (in scope as boundary node infrastructure) uses `verify_signature` (ZIP215) to authenticate the certificate's self-signature. With the identity element, this accepts any forged signature, allowing any party to impersonate the registered node to boundary nodes without possessing a private key. This constitutes a significant boundary node/infrastructure security impact with concrete protocol harm — unauthorized node impersonation at the boundary layer. Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation

The prerequisite is being a registered node operator (requires governance approval), which is a meaningful but not prohibitive constraint — node operators are a relatively broad class of IC participants. The cryptographic forgery is deterministic, requires no brute force, and is a direct consequence of the identity element's algebraic properties. The three non-canonical encodings are publicly documented. The attack is locally reproducible as described in the proof of concept.

## Recommendation

Add an `is_canonical()` check to `verify_ed25519_public_key` in `rs/crypto/node_key_validation/tls_cert_validation/src/lib.rs`:

```rust
fn verify_ed25519_public_key(
    public_key: &ic_ed25519::PublicKey,
) -> Result<(), TlsCertValidationError> {
    if !public_key.is_torsion_free() || !public_key.is_canonical() {
        return Err(invalid_tls_certificate_error(
            "public key verification failed",
        ));
    }
    Ok(())
}
```

Apply the same fix to `ValidNodeSigningPublicKey::try_from` in `rs/crypto/node_key_validation/src/lib.rs` at line 236.

## Proof of Concept

```rust
// 1. Build a TBSCertificate with subject_public_key =
//    0x0100000000000000000000000000000000000000000000000000000000000080
//    (non-canonical identity element, confirmed torsion-free by existing test)
//
// 2. Forge self-signature: S=1, R=B (base point).
//    ZIP215 check: ([1]B - B).mul_by_cofactor() == identity  ✓
//    (independent of TBSCertificate content)
//
// 3. Call ValidTlsCertificate::try_from((cert, node_id, time))
//    → assert Ok(_)   // registration succeeds
//
// 4. Confirm universal forgery:
//    ic_ed25519::PublicKey::deserialize_raw(&IDENTITY_NONCANONICAL)
//        .unwrap()
//        .verify_signature(arbitrary_msg, &forged_sig_with_R_equals_SB)
//    → assert Ok(())
//
// 5. Confirm boundary node accepts forged certificate:
//    Present the identity-element certificate with forged self-signature
//    to TlsVerifier::verify_server_cert — passes verify_signature (ZIP215)
```

The ZIP215 test data already in the repository (`packages/ic-ed25519/tests/data/zip215.txt`) confirms that the identity element key `0x0000...0080` accepts multiple distinct signatures for the same message, directly confirming the universal forgery property.

    /// Return true if and only if the key is contained within the prime
    /// order subgroup
    pub fn is_torsion_free(&self) -> bool {
        // We don't need to call is_weak here since that is subsumed by the
        // test that the point is torsion free - is_weak just checks if the
        // point is within the size-8 cofactor group.
        self.pk.to_edwards().is_torsion_free()
    }

    /// Return true if and only if the public key uses a canonical encoding
    pub fn is_canonical(&self) -> bool {
        self.pk.to_bytes() == self.pk.to_edwards().compress().0
    }

    pub fn verify_signature(&self, msg: &[u8], signature: &[u8]) -> Result<(), SignatureError> {
        let signature = Signature::from_slice(signature)?;

        let k = Self::compute_challenge(&signature, self, msg);
        let minus_a = -self.pk.to_edwards();
        let recomputed_r =
            EdwardsPoint::vartime_double_scalar_mul_basepoint(&k, &minus_a, signature.s());

        use curve25519_dalek::traits::IsIdentity;

        if (recomputed_r - signature.r())
            .mul_by_cofactor()
            .is_identity()
        {
            Ok(())
        } else {
            Err(SignatureError::InvalidSignature)
        }
    }
