### Title
Precision Loss in `native_per_pubdata` Calculation Silently Zeros Pubdata Native Cost - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio used to charge native resources for pubdata is computed with floor (truncating) integer division. When `pubdata_price < native_price`, the result truncates to zero, causing all pubdata to be free from the native resource perspective. Any user transacting in a block with this price configuration can generate unlimited pubdata without being charged native resources for it.

---

### Finding Description

In `validation_impl.rs`, the `native_per_pubdata` ratio is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is floor (truncating) integer division. When `pubdata_price < native_price`, the result is `0`.

By contrast, `native_per_gas` is computed with ceiling division:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

The same floor-division bug exists in the off-chain helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata == 0`, the downstream effects are:

1. **Intrinsic pubdata overhead becomes zero** — `intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` evaluates to `0`, so no native is deducted for intrinsic pubdata at resource creation time. [4](#0-3) 

2. **Post-execution pubdata check always passes** — `get_resources_to_charge_for_pubdata` returns `0` native cost, so `check_enough_resources_for_pubdata` always returns `enough = true` regardless of how much pubdata the transaction generated. [5](#0-4) 

3. **`compute_gas_refund` charges zero native for pubdata** — the `to_charge_for_pubdata` resource is zero, so `resources.charge_unchecked(&to_charge_for_pubdata)` is a no-op. [6](#0-5) 

The `delta_gas` adjustment in `compute_gas_refund` does not compensate for this, because it accounts for computational native (RISC-V cycles) but not pubdata native: [7](#0-6) 

---

### Impact Explanation

When `pubdata_price < native_price`, every transaction in the block pays **zero native resources for pubdata**. A user can write many storage slots (maximizing pubdata) without being charged native resources for the L1 data availability cost. The operator bears the L1 DA cost without receiving native resource compensation. This is a resource accounting bug: the pubdata fee mechanism is silently bypassed due to precision loss, analogous to the Derby vault's `rewardPerLockedToken` truncating to zero.

---

### Likelihood Explanation

The operator sets `pubdata_price` and `native_price` independently as block context parameters. A configuration where `pubdata_price < native_price` is realistic — for example, during periods of low L1 congestion where pubdata is cheap relative to computation cost. Any user submitting a transaction in such a block can exploit this without any special privileges. The attacker only needs to submit a transaction that generates significant pubdata (e.g., multiple storage writes).

---

### Recommendation

Use ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — can produce 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — minimum 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, consider adding a validation that rejects transactions (or blocks) where `native_per_pubdata == 0` and `pubdata_price > 0`, to make the invariant explicit.

---

### Proof of Concept

**Setup:**
- `native_price = 100`
- `pubdata_price = 50` (less than `native_price`)

**Computation:**
- `native_per_pubdata = 50.wrapping_div(100) = 0`

**Effect:**
- `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0` → no native deducted for intrinsic pubdata
- `get_resources_to_charge_for_pubdata(pubdata_bytes=640) = 640 * 0 = 0` → pubdata check always passes
- A transaction writing 20 storage slots (~640 bytes pubdata) pays **0 native** for pubdata instead of `640 * ceil(50/100) = 640 * 1 = 640` native units

The user's transaction succeeds with full pubdata generation, paying only computational native (for EVM execution), while the operator absorbs the L1 DA cost uncompensated.
