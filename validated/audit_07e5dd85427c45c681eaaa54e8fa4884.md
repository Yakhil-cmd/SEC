Audit Report

## Title
DTS Long-Execution Starvation via `executed_rounds` Accumulation in `abort_paused_executions_above_limit` — (`rs/execution_environment/src/scheduler.rs`)

## Summary

The `CanisterRoundState::Ord` implementation uses `executed_rounds.max(1)` to equalize aborted canisters (`executed_rounds = 0`) with newly started long executions (`executed_rounds = 1`). However, once K attacker-controlled canisters accumulate `executed_rounds >= 2` (after a single uncontested round), the victim canister — perpetually reset to `executed_rounds = 0` — is always ranked last and aborted every round indefinitely. The victim's DTS long execution never completes, permanently blocking its ingress messages.

## Finding Description

**Sort order** (`rs/execution_environment/src/scheduler/round_schedule.rs`, L119–125): Among long executions, `CanisterRoundState::Ord` sorts by `executed_rounds` descending (canisters with more rounds come first). The `.max(1)` clamp only equalizes `0` with `1`; it provides no protection once any canister reaches `executed_rounds >= 2`.

**Abort logic** (`rs/execution_environment/src/scheduler.rs`, L1068–1082): `abort_paused_executions_above_limit` sorts all paused-execution canisters and aborts those beyond index `max_paused_executions` (production default: `K = 4`, `rs/config/src/subnet_config.rs` L125). The K canisters with the highest `executed_rounds` are kept; the rest are aborted.

**`executed_rounds` increment** (`rs/execution_environment/src/scheduler/round_schedule.rs`, L498–503, L534–537): When a canister completes a DTS slice (`NextExecution::ContinueLong`), it is added to `fully_executed_canisters`, and `finish_round` increments its `executed_rounds`. This applies to both attacker and victim canisters each round they execute.

**Call order** (`rs/execution_environment/src/scheduler.rs`, L1526, L1530): `round_schedule.finish_round()` (increments `executed_rounds`) runs before `self.finish_round()` (calls `abort_paused_executions_above_limit`). The abort decision therefore sees the already-incremented values.

**Reset on abort** (`rs/execution_environment/src/scheduler.rs`, L2203–2207): When a canister is aborted, only `executed_rounds` is reset to `0`; `accumulated_priority` and `long_execution_start_round` are preserved.

**Round-by-round trace** with `K = 4` attacker canisters, 1 victim:
- **Round 1**: All 5 execute a slice → all `executed_rounds = 1`. Abort runs: all `.max(1) = 1`, AP tiebreaker decides. Victim (lower AP) aborted → reset to `0`.
- **Round 2**: Attackers execute → `executed_rounds = 2`. Victim re-executes one slice → `executed_rounds = 1`. Abort: `2 > 1`, victim aborted → reset to `0`.
- **Round N**: Attackers have `executed_rounds = N`. Victim always has `executed_rounds = 1` (then reset to `0`). Gap grows unboundedly. Victim perpetually aborted.

The existing `respect_max_paused_executions` proptest (`rs/execution_environment/src/scheduler/tests/dts.rs`, L268–321) does not cover this case because it uses equal compute allocations and no pre-accumulated `executed_rounds` advantage.

## Impact Explanation

The victim canister's long DTS execution never completes. Its ingress messages remain in `Processing` state indefinitely — a complete liveness denial for the targeted canister. This matches the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS." The attack is targeted at a specific canister but requires no subnet-wide disruption and is self-perpetuating once established.

## Likelihood Explanation

An unprivileged attacker needs only 4 canisters sending long messages (many slices up to `max_instructions_per_message`). No privileged access, governance majority, or threshold corruption is required. The attack is stable: after round 1, it self-perpetuates with no further attacker intervention. The attacker pays cycles only for their own K long executions; the victim pays nothing for aborted slices. The barrier is low and the attack is deterministic.

## Recommendation

Replace the `executed_rounds.max(1)` clamp with a mechanism that gives aborted executions a monotonically increasing priority boost. Options include: (1) introducing a separate `abort_count` field that is never reset and factors into the sort order; (2) when a canister is aborted by `abort_paused_executions_above_limit`, setting its `executed_rounds` to `max(executed_rounds among all currently paused canisters) + 1`; or (3) sorting aborted executions (`executed_rounds == 0`) before all other long executions rather than equal to `executed_rounds = 1`.

## Proof of Concept

Extend the existing DTS test suite at `rs/execution_environment/src/scheduler/tests/dts.rs` with an adversarial scenario:

```rust
// K=4 attacker canisters with high compute allocation, long messages (1000 slices each)
// 1 victim canister with CA=0, long message (10 slices)
// max_paused_executions = 4

// After round 1: attacker executed_rounds=1, victim executed_rounds=1
//   → victim aborted (lower AP), executed_rounds reset to 0

// After round 2: attacker executed_rounds=2, victim executed_rounds=1
//   → victim aborted (2 > 1), executed_rounds reset to 0

// After round N: attacker executed_rounds=N, victim executed_rounds=1
//   → victim always aborted

for round in 0..10000 {
    test.execute_round(OrdinaryRound);
}
// Assert victim message never completed despite requiring only 10 uninterrupted rounds
assert!(victim_message_still_processing);
```
