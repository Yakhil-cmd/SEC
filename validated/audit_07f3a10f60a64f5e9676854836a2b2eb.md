### Title
Missing Reimbursement on `BuildTxError::AmountTooLow` After Fee Spike Burns User ckBTC — (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

---

### Summary

A user who passes the `AmountTooLow` ingress check in `retrieve_btc` can have their ckBTC permanently burned with no reimbursement if Bitcoin network fees spike between request acceptance and batch processing. The `submit_pending_requests` function re-fetches the live fee rate via `estimate_fee_per_vbyte`, then calls `build_unsigned_transaction` with that new rate. If the batch fails with `BuildTxError::AmountTooLow`, the requests are silently finalized as `FinalizedStatus::AmountTooLow` — with no call to `reimburse_canceled_requests`, unlike every other failure mode.

---

### Finding Description

**Step 1 — Ingress check reads a stale minimum.**

In `retrieve_btc`, the minimum is read once before any async operations:

```rust
let (min_retrieve_amount, btc_network) =
    read_state(|s| (s.fee_based_retrieve_btc_min_amount, s.btc_network));
if args.amount < min_retrieve_amount {
    return Err(RetrieveBtcError::AmountTooLow(min_retrieve_amount));
}
``` [1](#0-0) 

After this check, the function awaits `balance_of(caller)`, then `check_address(...)`, then `burn_ckbtcs(...)`. Each `.await` is a yield point where the IC scheduler can interleave other messages. [2](#0-1) 

**Step 2 — `submit_pending_requests` re-fetches the live fee rate.**

Every time the batch timer fires, `submit_pending_requests` calls `estimate_fee_per_vbyte`, which atomically writes the new fee rate and a new `fee_based_retrieve_btc_min_amount` into state:

```rust
let fee_millisatoshi_per_vbyte = match estimate_fee_per_vbyte(runtime).await {
    Some(fee) => fee,
    None => return,
};
``` [3](#0-2) 

```rust
mutate_state(|s| {
    s.last_fee_per_vbyte = fees;
    s.last_median_fee_per_vbyte = Some(median_fee);
    s.fee_based_retrieve_btc_min_amount = fee_based_retrieve_btc_min_amount;
});
``` [4](#0-3) 

The batch is then built and `build_unsigned_transaction` is called with this freshly-fetched, potentially much higher fee rate. [5](#0-4) 

**Step 3 — `BuildTxError::AmountTooLow` silently drops requests with no reimbursement.**

When `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, the handler calls only `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. There is no call to `reimburse_canceled_requests`:

```rust
Err(BuildTxError::AmountTooLow) => {
    for request in batch {
        state::audit::remove_retrieve_btc_request(
            s,
            request,
            state::FinalizedStatus::AmountTooLow,
            runtime,
        );
    }
    None
}
``` [6](#0-5) 

Contrast this with `BuildTxError::InvalidTransaction`, which **does** reimburse:

```rust
Err(BuildTxError::InvalidTransaction(err)) => {
    let reason = reimbursement::WithdrawalReimbursementReason::InvalidTransaction(err);
    let reimbursement_fee = fee_estimator
        .reimbursement_fee_for_pending_withdrawal_requests(batch.len() as u64);
    reimburse_canceled_requests(s, batch, reason, reimbursement_fee, runtime);
    None
}
``` [7](#0-6) 

**Step 4 — `FinalizedStatus::AmountTooLow` is a terminal dead-end.**

`remove_retrieve_btc_request` records a `RemovedRetrieveBtcRequest` event and pushes to `finalized_requests`. No reimbursement task is ever scheduled. [8](#0-7) 

The event log replay also confirms this: `RemovedRetrieveBtcRequest` unconditionally sets `FinalizedStatus::AmountTooLow` with no reimbursement side-effect. [9](#0-8) 

`FinalizedStatus` has only two variants — `AmountTooLow` and `Confirmed` — neither of which triggers a reimbursement mint. [10](#0-9) 

**Step 5 — The fee formula's fixed buffer is insufficient for large fee spikes.**

`fee_based_minimum_withdrawal_amount` uses a fixed `PER_REQUEST_RBF_BOUND` of 22,100 sats as a buffer. If the fee rate more than doubles from the time of acceptance to the time of batch processing, the buffer is exhausted and the accepted amount can fall below what `build_unsigned_transaction` requires at the new rate. [11](#0-10) 

---

### Impact Explanation

A user who submits `retrieve_btc` with an amount exactly at or just above the current `fee_based_retrieve_btc_min_amount` will have their ckBTC burned on the ledger. If Bitcoin network fees spike before the batch is processed, `build_unsigned_transaction` fails with `AmountTooLow`, and the request is finalized with `FinalizedStatus::AmountTooLow` — permanently. The user's ckBTC is gone with no reimbursement and no recovery path. The `RetrieveBtcStatusV2::WillReimburse` / `Reimbursed` states are never reached for this failure mode.

---

### Likelihood Explanation

Bitcoin fees are highly volatile. Spikes of 5–10× within a single batch window (`max_time_in_queue_nanos`) are historically documented (e.g., Ordinals inscription waves, halving periods). Any user submitting at the minimum during a low-fee period is at risk if fees spike before their batch fires. No attacker action is required — normal fee market behavior is sufficient.

---

### Recommendation

In the `Err(BuildTxError::AmountTooLow)` branch of `submit_pending_requests`, call `reimburse_canceled_requests` (or an equivalent reimbursement path) instead of silently finalizing with `FinalizedStatus::AmountTooLow`. The `InvalidTransaction` branch already demonstrates the correct pattern. A new `WithdrawalReimbursementReason::AmountTooLow` variant can be added to distinguish this case in the event log and status API.

---

### Proof of Concept

State-machine test outline:

1. Initialize minter with low fee percentiles → `fee_based_retrieve_btc_min_amount` = 100,000 sats.
2. User calls `retrieve_btc(amount = 100_000)` → passes `AmountTooLow` check, ckBTC burned, request enters `pending_retrieve_btc_requests`.
3. Before the batch timer fires, inject high fee percentiles (e.g., 10 sat/vbyte) so the next `estimate_fee_per_vbyte` call returns a rate that makes 100,000 sats insufficient.
4. Advance time past `max_time_in_queue_nanos` to trigger `submit_pending_requests`.
5. Assert: `retrieve_btc_status_v2(block_index)` == `AmountTooLow` (not `WillReimburse` or `Reimbursed`).
6. Assert: user's ckBTC balance has not been restored.

This directly demonstrates that a request accepted past the `AmountTooLow` check can be permanently dropped with no reimbursement, violating the invariant that accepted-and-burned requests are either fulfilled or reimbursed.

    fn fee_based_minimum_withdrawal_amount(&self, median_fee_rate: FeeRate) -> Satoshi {
        match self.network {

                const PER_REQUEST_RBF_BOUND: u64 = 22_100;
