### Title
Precision Loss in `native_per_pubdata` Floor Division Truncates to Zero, Bypassing Native Resource Charging for Pubdata — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, the `native_per_pubdata` ratio is computed with floor (truncating) integer division, while the analogous `native_per_gas` ratio uses ceiling division. When `pubdata_price < native_price`, `native_per_pubdata` silently truncates to `0`. Every downstream function that uses this value then charges **zero** native resources for pubdata, regardless of how many bytes are published. The `deltaGas` adjustment mechanism consequently adds no extra gas for pubdata costs, so the user's final gas payment does not cover the prover's pubdata overhead.

---

### Finding Description

**Root cause — asymmetric rounding in two sibling calculations:**

```rust
// validation_impl.rs line 135 — uses div_ceil (rounds UP, user pays more)
u256_try_to_u64(&gas_price.div_ceil(native_price))

// validation_impl.rs line 142 — uses wrapping_div (floor, rounds DOWN)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same floor-division pattern is replicated verbatim in the public API helper:

```rust
// api/src/helpers.rs line 426-427
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

**Propagation through the resource pipeline:**

When `native_per_pubdata == 0`, every call to `get_resources_to_charge_for_pubdata` returns a zero-native `Resources` object:

```rust
// gas_helpers.rs line 430-431
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

`check_enough_resources_for_pubdata` therefore always returns `true`, and `resources.charge_unchecked(&to_charge_for_pubdata)` deducts nothing from the native budget.

**`deltaGas` adjustment is silently suppressed:**

The documentation defines:
> `deltaGas := (nativeUsed / nativePerGas) - gasUsed` [4](#0-3) 

Implemented in `compute_gas_refund`:

```rust
// refund_calculation.rs line 69-73
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [5](#0-4) 

Because `native_per_pubdata == 0` means pubdata never consumed any native units, `native_used` is smaller than it should be. `delta_gas` is therefore under-estimated, and the user's `gas_used` is not bumped to cover pubdata proving costs.

**Intrinsic pubdata overhead is also zeroed:**

```rust
// gas_helpers.rs line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
``` [6](#0-5) 

The intrinsic pubdata cost is also silently dropped from the native limit.

---

### Impact Explanation

When `pubdata_price < native_price` (a realistic operator configuration — e.g., cheap L1 DA with expensive proving), every L2 transaction:

1. Passes the pubdata-sufficiency check unconditionally, regardless of how many storage slots it writes.
2. Consumes zero native resources for pubdata, leaving the full native budget available for computation.
3. Receives a gas refund that does not account for pubdata proving costs, so the user's ETH payment is less than the true cost.
4. The intrinsic pubdata overhead (charged before execution) is also zeroed, further reducing the effective native limit deduction.

The ETH shortfall per transaction is approximately `pubdata_bytes × pubdata_price` — the full pubdata fee the operator intended to collect — silently waived due to integer truncation.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is not exotic. `native_price` reflects RISC-V proving cycles (can be large), while `pubdata_price` reflects L1 calldata/blob cost (can be small relative to proving). Any block where the operator sets, for example, `native_price = 1000` and `pubdata_price = 500` triggers `native_per_pubdata = 0`. No privileged access is required by the transaction sender; they simply submit any transaction that writes storage. The existing test suite already exercises `pubdata_price = 2, native_price = 100` (line 1236–1237 of `tests/instances/transactions/src/lib.rs`), which silently produces `native_per_pubdata = 0`. [7](#0-6) 

---

### Recommendation

Replace the floor division for `native_per_pubdata` with ceiling division, consistent with `native_per_gas`:

```diff
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
      .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures `native_per_pubdata >= 1` whenever `pubdata_price > 0`, mirroring the conservative (protocol-protective) rounding already applied to `native_per_gas`.

---

### Proof of Concept

Given:
- `native_price = 10`, `pubdata_price = 5`
- `native_per_pubdata = 5 / 10 = 0` (floor)
- Transaction writes 100 storage slots → ~3200 bytes pubdata

With `native_per_pubdata = 0`:
- `get_resources_to_charge_for_pubdata` → `(3200 * 0) = 0` native charged
- `check_enough_resources_for_pubdata` → always `true`
- `delta_gas` → `native_used` excludes pubdata cost → no extra gas added
- User's gas refund is inflated by `3200 * 5 / 10 / native_per_gas` gas units

With the fix (`div_ceil`, `native_per_pubdata = 1`):
- `get_resources_to_charge_for_pubdata` → `3200 * 1 = 3200` native charged
- `delta_gas` correctly adds `3200 / native_per_gas` extra gas
- User pays the intended pubdata cost
