Audit Report

## Title
`fee_based_minimum_withdrawal_amount` uses a 1000× underestimated minter fee bound, enabling `AmountTooLow` finalization without reimbursement on fragmented UTXO sets — (`rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

## Summary

`fee_based_minimum_withdrawal_amount` computes the safe minimum withdrawal amount using `PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu, which is 1000× smaller than the actual 2-input minter fee of 326,000,000 koinu. When the minter's UTXO set is fragmented, the actual `minter_fee` (which scales at 146,000,000 koinu per input) can exceed the withdrawal amount, causing `build_unsigned_transaction` to return `BuildTxError::AmountTooLow`. This path finalizes the request with `FinalizedStatus::AmountTooLow` without any reimbursement, permanently burning the user's ckDOGE.

## Finding Description

**Root cause — underestimated minter fee bound:**

`fee_based_minimum_withdrawal_amount` uses `PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu. [1](#0-0) 

The actual minter fee for 2 inputs is `2×146_000_000 + 2×4_000_000 + 26_000_000 = 326_000_000` koinu — exactly 1000× larger than the bound. The minimum withdrawal amount returned is therefore approximately `retrieve_doge_min_amount` (e.g., 50 DOGE), with the minter fee contribution being negligible. [2](#0-1) 

**Actual minter fee scales linearly with inputs:**

`evaluate_minter_fee` charges `MINTER_FEE_PER_INPUT = 146_000_000` koinu per input. With 50 inputs of 1 DOGE each selected to cover a 50 DOGE withdrawal: `50×146_000_000 + 2×4_000_000 + 26_000_000 = 7_334_000_000` koinu = **73.34 DOGE**, which exceeds the 50 DOGE withdrawal amount. [3](#0-2) 

**`AmountTooLow` check in `build_unsigned_transaction`:**

The check `if fee + minter_fee > amount` fires and returns `BuildTxError::AmountTooLow`. [4](#0-3) 

**`TooManyInputs` guard is insufficient:**

The `TooManyInputs` guard (which does trigger reimbursement via `InvalidTransaction`) only fires at `num_inputs > 500`. The dangerous zone of 35–499 inputs is enough to push `minter_fee > amount` but never reaches the reimbursed path. [5](#0-4) [6](#0-5) 

**`AmountTooLow` finalizes without reimbursement:**

The `AmountTooLow` handler calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` and enqueues no reimbursement task. By contrast, `BuildTxError::InvalidTransaction` (which covers `TooManyInputs`) calls `reimburse_canceled_requests`. [7](#0-6) [8](#0-7) 

`FinalizedStatus::AmountTooLow` is a terminal state with no recovery path. [9](#0-8) 

## Impact Explanation

A user who submits a withdrawal at exactly `fee_based_minimum_withdrawal_amount` (e.g., 50 DOGE) when the minter's UTXO set is fragmented will have their ckDOGE permanently burned with no reimbursement. The ledger burn is irreversible. This constitutes a concrete, permanent loss of in-scope chain-key/ledger assets (ckDOGE) for affected users. This matches the **High ($2,000–$10,000)** impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

The minter naturally accumulates small UTXOs from user deposits. Any user depositing amounts below ~1.43 DOGE contributes to fragmentation. An adversary can deliberately deposit many small UTXOs (e.g., 60 deposits of 1 DOGE each) to fragment the UTXO set, then trigger the bug for themselves or other users withdrawing at the published minimum. The threshold is reachable with as few as 35 inputs of <1.43 DOGE each. No special privileges are required — any unprivileged user can both cause and trigger the condition. The `fee_based_minimum_withdrawal_amount` is the value advertised to users as the safe minimum, so victims have no way to detect the risk.

## Recommendation

1. **Fix `PER_REQUEST_MINTER_FEE_BOUND`**: Replace `326_000` with a conservative upper-bound based on a realistic worst-case input count (e.g., `DOGECOIN_MAX_NUM_INPUTS_IN_TRANSACTION × MINTER_FEE_PER_INPUT`), or dynamically compute it from the actual UTXO set composition.
2. **Add reimbursement to the `AmountTooLow` handler**: The user submitted a valid amount per the published minimum; the failure is due to minter UTXO state, not user error. Reimburse the user minus a small processing fee, consistent with the `InvalidTransaction` handler.
3. **Alternatively**: Before finalizing with `AmountTooLow`, attempt UTXO consolidation or defer the request rather than burning it.

## Proof of Concept

```
1. Minter state: 60 UTXOs of 1 DOGE each (60 DOGE total).
2. fee_based_minimum_withdrawal_amount(median_fee) returns ~50 DOGE
   (PER_REQUEST_MINTER_FEE_BOUND=326_000 koinu is negligible).
3. User calls retrieve_doge_with_approval(50_DOGE). Passes ingress check.
4. submit_pending_requests → build_unsigned_transaction:
   - UTXO selection picks 50 UTXOs of 1 DOGE to cover 50 DOGE.
   - minter_fee = 50×146_000_000 + 2×4_000_000 + 26_000_000
               = 7_334_000_000 koinu = 73.34 DOGE
   - fee + minter_fee ≈ 73.34 DOGE > 50 DOGE → AmountTooLow
5. remove_retrieve_btc_request(request, FinalizedStatus::AmountTooLow)
   → no reimbursement, ckDOGE burned.

Fuzz/invariant test: generate a UtxoSet with N UTXOs of value V < 1.43 DOGE,
set withdrawal_amount = fee_based_minimum_withdrawal_amount(fee_rate),
assert build_unsigned_transaction does not return AmountTooLow.
This assertion fails for N ≥ 35.
```

        )
