Audit Report

## Title
Unchecked Index Access on Empty `Vec<EnvelopePair>` Causes Panic in `/construction/parse` — (`rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

## Summary
Any unauthenticated HTTP client can POST a crafted CBOR-hex blob to `/construction/parse` with `signed=true` that encodes a `SignedTransaction` whose `requests` vector contains a `(RequestType, Vec<EnvelopePair>)` entry with an empty inner `Vec`. The handler unconditionally accesses `updates[0]` without a length guard, triggering a Rust index-out-of-bounds panic. The Rosetta server runs on `actix-web` with no explicit panic-catching middleware, so the panic propagates through the async handler, causing actix-web to abort the worker and return a 500 response. Repeated requests constitute a persistent denial-of-service against the Rosetta API.

## Finding Description
`ParsedTransaction::try_from` at `models.rs:185` deserializes the `transaction` field via `serde_cbor::from_slice` into `SignedTransaction { requests: Vec<(RequestType, Vec<EnvelopePair>)> }`. `serde_cbor` accepts any structurally valid CBOR, including entries where the inner `Vec<EnvelopePair>` is empty — no post-deserialization invariant enforces `len >= 1`.

Immediately after, `construction_parse.rs:42` maps over `requests` and blindly indexes:

```rust
|(request_type, updates)| match updates[0].update.content.clone() {
    HttpCallContent::Call { update } => (request_type.clone(), update),
},
```

When `updates` is empty this is an unconditional Rust panic: `index out of bounds: the len is 0 but the index is 0`. The `construction_parse` endpoint handler in `rosetta_server.rs:115-122` is a plain `async fn` with no `catch_unwind` wrapper; the panic propagates through the actix-web worker thread, which actix-web terminates and replaces, returning HTTP 500 to the caller. No existing check in the deserialization path or the handler guards against this condition.

## Impact Explanation
The Rosetta API is explicitly listed as an in-scope financial integration target. Any unauthenticated caller can repeatedly trigger this panic to exhaust actix-web worker threads or cause persistent 500 responses on the `/construction/parse` endpoint, rendering the Rosetta API unavailable. This constitutes a significant Rosetta API availability impact with concrete harm to users and integrators relying on the construction flow. This matches the High impact class: "Significant... Rosetta... security impact with concrete user or protocol harm."

## Likelihood Explanation
The exploit requires no authentication, no privileged role, and no special knowledge beyond the CBOR encoding of `SignedTransaction`. The `transaction` field is an arbitrary hex string accepted from any HTTP client. The crafted payload is a handful of bytes. The path is fully reproducible locally without any mainnet interaction.

## Recommendation
Add an explicit length guard before the index access in `construction_parse.rs`:

```rust
.map(|(request_type, updates)| {
    if updates.is_empty() {
        return Err(ApiError::invalid_request(
            "SignedTransaction contains a request with no envelope pairs".to_string()
        ));
    }
    match updates[0].update.content.clone() {
        HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
    }
})
.collect::<Result<Vec<_>, _>>()?
```

Alternatively, enforce the non-empty invariant during deserialization by replacing `Vec<EnvelopePair>` with a `NonEmpty<EnvelopePair>` newtype that rejects empty sequences at the `serde_cbor` boundary.

## Proof of Concept
Construct a CBOR blob encoding `SignedTransaction { requests: [(RequestType::Send, [])] }` and POST it:

```python
import requests, cbor2, json

# CBOR: {"requests": [["Send", []]]}
payload = cbor2.dumps({"requests": [["Send", []]]})
tx_hex = payload.hex()

body = {
    "network_identifier": {"blockchain": "Internet Computer", "network": "<network_id>"},
    "signed": True,
    "transaction": tx_hex
}
r = requests.post("http://<rosetta-host>/construction/parse", json=body)
assert r.status_code == 500, f"Expected panic/500, got {r.status_code}"
```

A unit test can reproduce this without a running server by calling `RosettaRequestHandler::construction_parse` directly with a `ConstructionParseRequest` whose `transaction` field is the hex of the above CBOR blob and `signed=true`, and asserting that it returns an `Err(ApiError)` rather than panicking.
