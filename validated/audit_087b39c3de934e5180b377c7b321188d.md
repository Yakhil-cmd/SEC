Audit Report

## Title
Third-Party Bypass of `confirmation_text` Consent Gate Forces Unauthorized SNS Swap Participation — (File: `rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` endpoint in the SNS Swap canister accepts an arbitrary `buyer` principal without verifying that the caller equals the specified buyer. Because the `confirmation_text` consent gate is validated only by string equality against a value stored in publicly-readable canister state, any unprivileged third party can call this endpoint on behalf of any buyer, supplying the correct confirmation text and recording the buyer's participation without their explicit action. A buyer who deposited ICP but had not yet confirmed can have their ICP committed to the swap and swept to the SNS governance canister without ever calling the endpoint themselves.

## Finding Description

In `rs/sns/swap/canister/canister.rs` (L127–143), the handler resolves the target buyer from the request argument rather than from the authenticated caller:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no check: caller == p
};
```

There is no assertion that `caller_principal_id() == p`. The `Init` proto field `confirmation_text` (swap.proto L325) is described as "An optional text that swap participants should confirm before they may participate in the swap." The `validate_confirmation_text` function in `rs/sns/swap/src/swap.rs` (L363–384) performs only a string equality check against `self.init_or_panic().confirmation_text` — it does not check the caller's identity. The `Init` struct, including `confirmation_text`, is returned by the public `get_init` query endpoint, making the expected text trivially obtainable by any caller.

The exploit path is:
1. Alice deposits ICP to the swap canister's subaccount derived from her principal (her own deliberate action), but has not yet called `refresh_buyer_tokens`.
2. Bob queries `get_init` (a public query) to retrieve the `confirmation_text`.
3. Bob submits an ingress update: `refresh_buyer_tokens({ buyer: "<Alice's principal>", confirmation_text: "<exact text>" })`.
4. `refresh_buyer_token_e8s` reads Alice's subaccount balance, passes `validate_confirmation_text`, and records Alice's participation in `self.buyers`.
5. When the swap commits and `sweep_icp` runs, Alice's ICP is transferred to the SNS governance canister and she receives SNS tokens — without ever having called the endpoint herself.

Without the attack, Alice could have retrieved her ICP via `error_refund_icp` after the swap closed (as confirmed by the lifecycle test at `rs/nervous_system/integration_tests/tests/sns_lifecycle.rs` L139). With the attack, her ICP is committed and swept, and `error_refund_icp` returns a precondition error because her buyer state shows ICP in escrow (swap.rs L1950–1959).

The design intent of allowing third-party calls is confirmed by a system test (`rs/tests/nns/sns/lib/src/sns_deployment.rs` L919–926) that explicitly calls `refresh_buyer_tokens` for `wealthy_user_identity` from a `default_sns_agent`. However, that test passes `None` for `confirmation_text`, meaning it does not exercise the consent-bypass scenario. The confirmation_text feature was added specifically to require explicit participant consent; allowing any caller to satisfy it with a publicly-readable string nullifies that purpose entirely.

## Impact Explanation

This is a **High** severity finding. A buyer who deposited ICP into the swap subaccount but had not yet confirmed participation can have their ICP permanently committed to the swap and exchanged for SNS tokens without their explicit consent. The `confirmation_text` field exists precisely to gate participation on an affirmative on-chain action by the participant; this vulnerability renders that gate ineffective. The concrete harm is unauthorized disposition of a user's ICP funds — the ICP is swept to the SNS governance canister and cannot be recovered. This matches the allowed impact: "Significant SNS security impact with concrete user or protocol harm" and "Unauthorized access to canister-controlled funds."

## Likelihood Explanation

**Medium-High.** The attacker requires only: (1) the target buyer's principal, derivable from the public `get_buyers_total` or `get_buyer_state` query endpoints or from the ICP ledger subaccount structure; (2) the confirmation text, directly readable from the public `get_init` query; and (3) the buyer to have deposited ICP into their swap subaccount. No privileged access, key material, or subnet-majority corruption is needed. The attack is a single ingress update call, repeatable for any buyer who has deposited but not yet confirmed.

## Recommendation

Add a caller-equals-buyer check in the canister endpoint when a non-empty `buyer` is specified:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, buyer,
            "Caller {} is not authorized to refresh tokens for buyer {}", caller, buyer);
        buyer
    };
    // ... rest unchanged
}
```

If third-party notification is a desired composability feature (as evidenced by the system test), a separate permissioned path restricted to a known aggregator canister or NNS governance should be provided, rather than leaving the endpoint fully open to any ingress caller.

## Proof of Concept

1. Deploy an SNS swap canister initialized with `confirmation_text = "I agree to the SNS swap terms"`.
2. Alice transfers 100 ICP to the swap canister's subaccount derived from her principal (using `principal_to_subaccount`), but does not call `refresh_buyer_tokens`.
3. Bob calls `swap.get_init()` (public query) and reads `confirmation_text`.
4. Bob submits: `refresh_buyer_tokens({ buyer: "<Alice's principal>", confirmation_text: Some("I agree to the SNS swap terms") })`.
5. Assert: `swap.get_buyer_state({ principal_id: Some(<Alice's principal>) })` now returns a non-zero `icp_e8s` — Alice is recorded as a confirmed participant.
6. Advance time past `swap_due_timestamp_seconds` so the swap commits; call `finalize_swap`.
7. Assert: Alice's ICP subaccount balance is 0 (swept to SNS governance); `error_refund_icp` for Alice returns a precondition error ("has X ICP e8s in escrow").
8. Alice received SNS neuron(s) without ever calling `refresh_buyer_tokens` herself.

This can be reproduced as a PocketIC integration test following the pattern in `rs/nervous_system/integration_tests/tests/sns_lifecycle.rs`, adding a step where a separate agent calls `refresh_buyer_tokens` for a participant before that participant has confirmed.
