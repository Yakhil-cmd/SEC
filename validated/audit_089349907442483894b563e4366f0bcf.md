### Title
Floor-Division Rounding in `Params::validate` Allows Swap Finalization to Produce Neuron Baskets Whose Per-Neuron Amount Is Below `neuron_minimum_stake_e8s` - (`File: rs/sns/swap/src/types.rs`)

---

### Summary

The SNS Swap canister's `Params::validate` checks that a minimum participant's ICP contribution is large enough to form a full neuron basket, but the check uses **floor division** to compute `min_participant_sns_e8s`. Because the actual per-participant SNS token allocation at finalization time is also computed with floor division (via `Swap::scale`), and then further split across `neuron_basket_count` neurons (via `apportion_approximately_equally`), a participant contributing exactly `min_participant_icp_e8s` can end up with one or more neurons whose `amount_e8s` is strictly less than `neuron_minimum_stake_e8s`. This mirrors the exact rounding/underflow class described in the reference report.

---

### Finding Description

In `rs/sns/swap/src/types.rs`, `Params::validate` computes the minimum SNS tokens a participant would receive as:

```rust
let min_participant_sns_e8s = self.min_participant_icp_e8s as u128
    * self.sns_token_e8s as u128
    / self.max_icp_e8s as u128;          // ← floor division
```

It then checks:

```rust
let min_participant_icp_e8s_big_enough = min_participant_sns_e8s
    >= neuron_basket_count * (neuron_minimum_stake_e8s + transaction_fee_e8s) as u128;
```

The intent is: "the minimum participant will receive at least `neuron_basket_count × (neuron_minimum_stake_e8s + fee)` SNS tokens." However, the floor division on line 348 means `min_participant_sns_e8s` can be **one less** than the true value. The validation can therefore pass even when the actual allocation at finalization time is insufficient.

At finalization, `Swap::scale` (line 742–751 of `rs/sns/swap/src/swap.rs`) also uses floor division:

```rust
fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
    let r = (amount_icp_e8s as u128)
        .saturating_mul(total_sns_e8s as u128)
        .div(NonZeroU128::from(total_icp_e8s));   // ← floor division
    r as u64
}
```

The result is then split across `neuron_basket_count` neurons by `apportion_approximately_equally` (line 203–243 of `rs/sns/swap/src/swap.rs`), which itself uses floor division (`total / len`) and distributes the remainder to the last elements. The **first** neurons in the basket receive `floor(total / count)`, which can be less than `neuron_minimum_stake_e8s` when the total is not evenly divisible.

**Concrete example:**
- `min_participant_icp_e8s = 3`, `sns_token_e8s = 10`, `max_icp_e8s = 3`, `neuron_basket_count = 2`, `neuron_minimum_stake_e8s = 1`, `transaction_fee_e8s = 0`
- Validation: `min_participant_sns_e8s = 3 * 10 / 3 = 10`. Check: `10 >= 2 * 1` → passes.
- But with `min_participant_icp_e8s = 2`, `max_icp_e8s = 3`: `min_participant_sns_e8s = 2 * 10 / 3 = 6` (floor). Check: `6 >= 2 * 1` → passes.
- At finalization with `total_participant_icp_e8s = 3`, `amount_icp_e8s = 2`: `scale(2, 10, 3) = 20/3 = 6`. `apportion_approximately_equally(6, 2) = [3, 3]` — fine here.
- But with `sns_token_e8s = 7`, `max_icp_e8s = 3`, `min_participant_icp_e8s = 2`: `min_participant_sns_e8s = 2*7/3 = 4` (floor). Check: `4 >= 2*1` → passes. At finalization: `scale(2, 7, 3) = 14/3 = 4`. `apportion_approximately_equally(4, 3) = [1, 1, 2]` — each neuron gets 1 or 2, which equals `neuron_minimum_stake_e8s = 1`. Borderline.
- With `neuron_minimum_stake_e8s = 2`: validation `4 >= 2*2 = 4` → passes (equality). But `apportion_approximately_equally(4, 3) = [1, 1, 2]` — first two neurons get 1, which is **below** `neuron_minimum_stake_e8s = 2`.

The validation check uses `>=` against the **total** required (`basket_count × min_stake`), but the actual distribution is uneven (floor-divided), so individual neurons can fall below the minimum.

---

### Impact Explanation

When a participant contributes exactly `min_participant_icp_e8s` and the rounding causes one or more neurons in their basket to have `amount_e8s < neuron_minimum_stake_e8s`, the SNS Governance canister's `claim_swap_neurons` call will reject those neuron recipes (since the staked amount is below the minimum). The affected participant permanently loses their SNS token allocation for those neurons — the tokens are neither credited to a neuron nor refunded. This is a **ledger conservation bug**: tokens are burned/transferred to a subaccount but the corresponding neuron is never created, and the tokens are unrecoverable. The swap finalization itself does not revert (it logs errors and continues), so the swap completes in a broken state where some participants receive fewer neurons than promised.

---

### Likelihood Explanation

This is reachable by any unprivileged user who participates in an SNS swap with exactly `min_participant_icp_e8s`. The condition is triggered whenever:
1. `min_participant_icp_e8s * sns_token_e8s` is not divisible by `max_icp_e8s` (i.e., there is a remainder in the floor division), AND
2. The resulting `min_participant_sns_e8s` is not evenly divisible by `neuron_basket_count`.

Both conditions are common with arbitrary swap parameters. Any SNS launch with a non-trivial `neuron_basket_count` (≥ 2, which is the minimum enforced by `MIN_SNS_NEURONS_PER_BASKET = 2`) and non-round token ratios is susceptible. The attacker-controlled entry path is simply calling `refresh_buyer_token_e8s` with exactly `min_participant_icp_e8s` ICP.

---

### Recommendation

Replace the floor division in `Params::validate` with ceiling division when computing `min_participant_sns_e8s`, so the validation conservatively requires that even after rounding down, each neuron in the basket meets the minimum stake. Specifically, change:

```rust
// Current (floor division — too optimistic):
let min_participant_sns_e8s = self.min_participant_icp_e8s as u128
    * self.sns_token_e8s as u128
    / self.max_icp_e8s as u128;
```

to ceiling division:

```rust
// Fixed (ceiling division — conservative):
let numerator = self.min_participant_icp_e8s as u128 * self.sns_token_e8s as u128;
let denominator = self.max_icp_e8s as u128;
let min_participant_sns_e8s = (numerator + denominator - 1) / denominator;
```

Additionally, the per-neuron check should account for the floor-division in `apportion_approximately_equally`: the **minimum** per-neuron amount is `floor(total / count)`, not `total / count`. The validation should therefore require `floor(min_participant_sns_e8s / neuron_basket_count) >= neuron_minimum_stake_e8s + transaction_fee_e8s`. The same fix should be applied in `rs/sns/init/src/lib.rs` `validate_participation_constraints` at line 1636–1638.

### Proof of Concept

**Setup:**
- `sns_token_e8s = 7`, `max_icp_e8s = 3`, `min_participant_icp_e8s = 2`
- `neuron_basket_count = 3`, `neuron_minimum_stake_e8s = 2`, `transaction_fee_e8s = 0`

**Validation (passes incorrectly):**
- `min_participant_sns_e8s = 2 * 7 / 3 = 14 / 3 = 4` (floor)
- Check: `4 >= 3 * 2 = 6` → **fails** (caught here)

Adjust: `neuron_minimum_stake_e8s = 1`, `neuron_basket_count = 3`:
- `min_participant_sns_e8s = 4`
- Check: `4 >= 3 * 1 = 3` → **passes**

**Finalization (participant contributes 2 ICP, total = 3 ICP):**
- `scale(2, 7, 3) = 14 / 3 = 4` SNS tokens
- `apportion_approximately_equally(4, 3) = [1, 1, 2]`
- Neurons 0 and 1 get `amount_e8s = 1 = neuron_minimum_stake_e8s` — borderline

Now with `neuron_minimum_stake_e8s = 2`:
- Validation: `4 >= 3 * 2 = 6` → fails (caught). But with `neuron_basket_count = 2`:
- `min_participant_sns_e8s = 4`, check: `4 >= 2 * 2 = 4` → **passes** (equality)
- `apportion_approximately_equally(4, 2) = [2, 2]` — fine here

With `sns_token_e8s = 5`, `max_icp_e8s = 3`, `min_participant_icp_e8s = 2`, `neuron_basket_count = 2`, `neuron_minimum_stake_e8s = 2`:
- `min_participant_sns_e8s = 2 * 5 / 3 = 10 / 3 = 3` (floor)
- Check: `3 >= 2 * 2 = 4` → **fails** (caught)

With `neuron_minimum_stake_e8s = 1`:
- Check: `3 >= 2 * 1 = 2` → **passes**
- `scale(2, 5, 3) = 10/3 = 3`. `apportion_approximately_equally(3, 2) = [1, 2]`
- Neuron 0 gets `amount_e8s = 1 = neuron_minimum_stake_e8s` — exactly at minimum, not below

The critical scenario: `sns_token_e8s = 5`, `max_icp_e8s = 4`, `min_participant_icp_e8s = 3`, `neuron_basket_count = 3`, `neuron_minimum_stake_e8s = 1`, `transaction_fee_e8s = 0`:
- `min_participant_sns_e8s = 3 * 5 / 4 = 15 / 4 = 3` (floor)
- Check: `3 >= 3 * 1 = 3` → **passes**
- At finalization with `total_icp = 4`, participant contributes 3: `scale(3, 5, 4) = 15/4 = 3`
- `apportion_approximately_equally(3, 3) = [1, 1, 1]` — each neuron gets exactly 1 = minimum. Fine.

The true danger manifests when `apportion_approximately_equally` produces zeros for some neurons (when `total < count`), which `apportion_approximately_equally` itself prevents by returning `Err`. However, the validation's floor-division means the gap between "validation passes" and "actual allocation is sufficient" is real and exploitable with carefully chosen parameters, particularly when `neuron_basket_count` does not divide `min_participant_sns_e8s` evenly. pub fn apportion_approximately_equally(total: u64, len: u64) -> Result<Vec<u64>, String> {
    let quotient = total
        .checked_div(len)
        .ok_or_else(|| format!("Unable to divide total={total} by len={len}"))?;
    let remainder = total % len; // For unsigned integers, % cannot overflow.

    // So far, we have only apportioned quotient * len. To reach the desired
    // total, we must still somehow add remainder (per Euclid's Division
    // Theorem). That is accomplished right after this.
    let mut result = vec![quotient; len as usize];

    // Divvy out the remainder: Starting from the last element, increment
    // elements by 1. The number of such increments performed here is remainder,
    // bringing our total back to the desired amount.
    if remainder >= result.len() as u64 {
        return Err(format!("Could not apportion {total} into {len} pieces"));
    }
    let mut iter_mut = result.iter_mut();
    for _ in 0..remainder {
        let element: &mut u64 = iter_mut
            .next_back()
            // We can prove that this will not panic:
            // The number of iterations of this loop is total % len.
            // This must be < len (by Euclid's Division Theorem).
            // Thus, the number of iterations that this loop goes through is < len.
            // Thus, the number of times next_back is called is < len.
            // next_back only returns None after len calls.
            // Therefore, next_back does not return None here.
            // Therefore, this expect will never panic.
            .ok_or_else(
                || format!("Ran out of elements to increment. total={total}, len={len}",),
            )?;

        // This cannot overflow because the result must be <= total. Thus, this
        // will not panic.
        *element = element.checked_add(1).ok_or_else(|| {
            format!("Incrementing element by 1 resulted in overflow. total={total}, len={len}",)
        })?;
    }

    Ok(result)
