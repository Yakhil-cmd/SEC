Audit Report

## Title
VetKD Share Verification Omits Cryptographic Validity Check, Enabling Byzantine Node to Force O(n) Pairing Fallback Per Request — (`rs/crypto/src/vetkd/mod.rs`)

## Summary

`verify_encrypted_key_share_internal` only verifies the node's basic signature over the raw share bytes and never checks whether the share is cryptographically valid. A single Byzantine subnet node can therefore submit a share with a valid node signature but garbage elliptic-curve content. Because all `n` VetKD shares are admitted to the validated pool (not just threshold+1), the poisoned share reaches `combine_encrypted_key_shares_internal`, causes `combine_all` to return `InvalidShares`, and triggers a fallback that performs up to O(n × 5) BLS12-381 pairing operations instead of the normal 5. The attack is repeatable on every VetKD request.

## Finding Description

**Root cause — share verification skips cryptographic validity.**

`verify_encrypted_key_share_internal` calls only `BasicSigVerifierInternal::verify_basic_sig` over `key_share.encrypted_key_share`. There is no call to `EncryptedKeyShare::is_valid` or `check_validity`. A share whose `c1`/`c2`/`c3` points are structurally deserializable but cryptographically inconsistent passes this check. [1](#0-0) 

**All n shares are admitted for VetKD.**

The consensus signer collects `n` shares (not `get_faults_tolerated(n) + 1`) before calling `combine_encrypted_key_shares`, so the invalid share is guaranteed to be present in the combination input. [2](#0-1) 

**`combine_all` interpolates over all submitted shares; one invalid share poisons the result.**

`combine_unchecked` performs Lagrange interpolation over the entire `nodes` map. If any share is invalid, the interpolated `c1`/`c2`/`c3` values will not satisfy the pairing equations, `is_valid` returns `false`, and `combine_all` returns `InvalidShares`. [3](#0-2) 

**`InvalidShares` fallback iterates all n nodes with O(n) pairing calls.**

On `InvalidShares`, `combine_encrypted_key_shares_internal` maps over the full `clib_shares` vector (all n nodes), calls `lazily_calculated_public_key_from_store` for each, then passes all n `(G2Affine, EncryptedKeyShare)` pairs to `combine_valid_shares`. [4](#0-3) 

`combine_valid_shares` then calls `node_eks.is_valid` (which invokes `check_validity`) for each node in BTreeMap order until `reconstruction_threshold` valid shares are collected. With one invalid share at a low `NodeIndex`, this means `(threshold + 1)` calls to `is_valid` in the loop, plus one final `is_valid` on the combined key. [5](#0-4) 

Each `check_validity` call performs two `Gt::multipairing` invocations totalling 5 BLS12-381 pairings. [6](#0-5) 

**Existing test confirms the fallback path is reachable in production.**

`should_succeed_if_reconstruction_threshold_many_shares_are_valid` explicitly corrupts shares by swapping `c1`/`c3` (structurally valid, cryptographically invalid) and asserts the fallback log message is emitted, confirming the path is exercised under realistic conditions. [7](#0-6) 

## Impact Explanation

For a 34-node subnet (threshold=23), the normal path costs 5 pairings. With one invalid share at NodeIndex 0, the fallback costs `(23 + 1) × 5 + 5 = 125` pairings — a ~25× amplification. BLS12-381 pairings cost ~1–2 ms each on modern hardware, so each VetKD request goes from ~5 ms to ~125–250 ms of pairing work on the combiner's crypto thread. Because the attack is repeatable on every request and scales with subnet size, it constitutes a sustained, per-request throughput degradation of the VetKD subsystem. This matches the allowed impact: **High — Application/platform-level DoS / subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

The attack requires only a single Byzantine subnet node operating below the BFT fault threshold — a realistic adversary. The technique (producing a structurally valid but cryptographically invalid share with a valid node signature) is explicitly demonstrated in the existing test suite (`swap_share_c1c3`). No special key material, coordination, or external access is needed beyond normal subnet membership. The attack is repeatable indefinitely at zero marginal cost per request.

## Recommendation

Inside `verify_encrypted_key_share_internal`, after the basic signature check passes, also verify the cryptographic validity of the share. Concretely:

1. Retrieve the signer's individual public key via `lazily_calculated_public_key_from_store`.
2. Deserialize the share into an `EncryptedKeyShare`.
3. Call `EncryptedKeyShare::is_valid` with the node's individual public key, the master public key, derivation context, input, and transport public key.
4. Return an error if `is_valid` returns `false`.

This ensures only cryptographically valid shares enter the validated pool, making the `InvalidShares` fallback unreachable via Byzantine input. If per-share pairing verification at admission time is considered too expensive, the fallback path should at minimum track and exclude nodes that have previously submitted invalid shares within the same epoch.

## Proof of Concept

```
1. Set up a subnet with n=34 nodes, VetKD threshold=23.
2. Byzantine node creates a valid VetKdEncryptedKeyShare normally.
3. Swap the first 48 bytes (c1) with the last 48 bytes (c3) in encrypted_key_share.0.
4. Re-sign the modified bytes with the node's signing key → valid node_signature.
5. Submit the share. verify_encrypted_key_share passes (basic sig check only).
6. All 34 shares (including the invalid one) are admitted to the validated pool.
7. combine_encrypted_key_shares is called with all 34 shares.
8. combine_all fails with InvalidShares (interpolation over all 34 shares produces invalid key).
9. Fallback: lazily_calculated_public_key_from_store called for all 34 nodes.
10. combine_valid_shares iterates shares calling is_valid (5 pairings each) until 23 valid ones found.
11. Assert: combiner log contains "falling back to EncryptedKey::combine_valid_shares".
12. Assert: ~125 pairing operations performed vs. 5 in the normal path.
13. Repeat for every VetKD request → sustained per-request throughput degradation.

This is directly reproducible by extending the existing test
`should_succeed_if_reconstruction_threshold_many_shares_are_valid`
to corrupt exactly 1 share (not n-threshold shares) and measuring
pairing call counts via instrumentation.
```

```rust
fn check_validity(
    c1: &G1Affine,
    c2: &G2Affine,
    c3: &G1Affine,
    tpk: &TransportPublicKey,
    verification_pk: &G2Affine,
    msg: &G1Affine,
) -> bool {
    let neg_g2_g = G2Prepared::neg_generator();
    let c2_prepared = G2Prepared::from(c2);

    // check e(c1,g2) == e(g1, c2)
    let c1_c2 = Gt::multipairing(&[(c1, neg_g2_g), (G1Affine::generator(), &c2_prepared)]);
    if !c1_c2.is_identity() {
        return false;
    }

    let verification_key_prepared = G2Prepared::from(verification_pk);

    // check e(c3, g2) == e(tpk, c2) * e(msg, dpki)
    let c3_c2_msg = Gt::multipairing(&[
        (c3, neg_g2_g),
        (tpk.point(), &c2_prepared),
        (msg, &verification_key_prepared),
    ]);

    if !c3_c2_msg.is_identity() {
        return false;
    }

    true
}
```
