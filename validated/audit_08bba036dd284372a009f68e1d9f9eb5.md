### Title
Balance Overflow in `process_withdrawals_list` Causes Unconditional Block-Level Panic - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/withdrawals.rs`)

---

### Summary

`process_withdrawals_list` iterates over all EIP-4895 beacon-chain withdrawal entries and calls `update_account_nominal_token_balance` to credit each recipient. The result is unwrapped with `.expect("must not fail")`. If any recipient's balance would overflow `U256::MAX`, the `BalanceError::Overflow` returned by the storage layer causes an unconditional panic, halting the entire block-finalization step. Because the withdrawal list is oracle-supplied (sequencer-controlled) and the recipient address is attacker-controlled (any address whose balance is near `U256::MAX`), this is a reachable, externally-triggerable denial-of-service path that prevents the block from being finalized.

---

### Finding Description

In `process_withdrawals_list` (the function that applies EIP-4895 beacon-chain withdrawals at block finalization), each withdrawal entry is applied by calling `update_account_nominal_token_balance` with `should_subtract = false` (i.e., an addition). The underlying implementation in `io_subsystem.rs` uses `checked_add` and returns `BalanceError::Overflow` if the result would exceed `U256::MAX`.

The call site in `withdrawals.rs` does not handle this error — it calls `.expect("must not fail")`:

```rust
resources
    .with_infinite_ergs(|resources| {
        system.io.update_account_nominal_token_balance(
            ExecutionEnvironmentType::NoEE,
            resources,
            &el.address,
            &amount,
            false,
            false,
        )
    })
    .expect("must not fail");   // <-- panics on BalanceError::Overflow
```

The `update_account_nominal_token_balance` implementation explicitly returns `BalanceError::Overflow` when `checked_add` overflows:

```rust
old_value
    .checked_add(*diff)
    .ok_or(interface_error! {BalanceError::Overflow})
```

`process_withdrawals_list` is called unconditionally during block post-processing in both the sequencing path (`post_tx_op_sequencing.rs`, line 72) and the proving path (`post_tx_op_proving.rs`, line 190), both of which also `.expect("must process withdrawals list")` on the result. A panic inside `process_withdrawals_list` propagates upward and terminates block finalization entirely.

The withdrawal list is provided by the oracle (sequencer-supplied input). The `el.address` field is decoded directly from the oracle-provided RLP bytes with no balance pre-check. An attacker who can influence the oracle input (or who is the sequencer) can include a withdrawal targeting an address whose balance is `U256::MAX - amount + 1`, causing the addition to overflow.

Even without a malicious sequencer, a legitimate scenario exists: if the same address appears in multiple withdrawal entries within a single block, the cumulative additions could overflow `U256::MAX` even if each individual entry is valid.

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug / valid-execution unprovability / state-transition bug.

A panic in `process_withdrawals_list` during block finalization means:
- The block cannot be finalized (no state commitment, no L2→L1 logs, no pubdata output).
- In the proving path, the RISC-V binary aborts, making the block unprovable.
- In the sequencing path, the sequencer process crashes or the block is dropped.
- All transactions already executed in the block have their effects lost (or the block is never committed), effectively freezing the chain at that block height.

This is a **chain-halting denial of service**: a single crafted block with a withdrawal targeting a near-`U256::MAX` balance address causes permanent block finalization failure for that block.

---

### Likelihood Explanation

The withdrawal list is oracle-supplied. In the ZKsync OS architecture, the oracle provides all external data including the withdrawals buffer. The sequencer controls what goes into the oracle. A malicious or compromised sequencer can trivially craft a withdrawal targeting an address with balance near `U256::MAX`. Additionally, even without malice, if a contract or EOA accumulates a balance near `U256::MAX` through normal operation and then receives a beacon withdrawal, the overflow occurs naturally. The `u64::MAX` GWei maximum per withdrawal entry translates to approximately `1.8 × 10^19` wei, which is well within the range that could overflow a near-`U256::MAX` balance.

---

### Recommendation

Replace the `.expect("must not fail")` with proper error handling. The function signature already returns `Result<Bytes32, InternalError>`, so the error can be propagated:

```rust
resources
    .with_infinite_ergs(|resources| {
        system.io.update_account_nominal_token_balance(
            ExecutionEnvironmentType::NoEE,
            resources,
            &el.address,
            &amount,
            false,
            false,
        )
    })
    .map_err(|_| internal_error!("withdrawal balance overflow"))?;
```

This converts the panic into a recoverable `InternalError`, which the callers in `post_tx_op_sequencing.rs` and `post_tx_op_proving.rs` can then handle (e.g., by skipping the overflowing entry or by treating it as a fatal-but-graceful block error rather than a process abort).

---

### Proof of Concept

1. Craft a block whose oracle-supplied withdrawals buffer contains a `WithdrawalRequest` with:
   - `address` = any address `A` whose current balance is `U256::MAX - amount + 1` (e.g., set via a prior transaction or genesis state).
   - `value_in_gwei` = any nonzero value such that `amount = 1_000_000_000 * value_in_gwei` causes `balance + amount > U256::MAX`.

2. Submit the block for execution.

3. All transactions in the block execute normally. At block finalization, `process_withdrawals_list` is called.

4. For the crafted entry, `update_account_nominal_token_balance` returns `Err(BalanceError::Overflow)`.

5. `.expect("must not fail")` panics, aborting block finalization.

6. The block is never committed; the chain is halted at this height.
