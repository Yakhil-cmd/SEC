### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Users to Overpay When Native Resource Consumption Is Below EVM Gas Consumption - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` correction is applied **only when positive** (increasing `gas_used` to cover native resource costs), but is **never applied when negative** (which would decrease `gas_used` to refund users for under-consumed native resources). A `TODO` comment in the code explicitly acknowledges this gap. The result is a one-directional resource accounting asymmetry: users are penalized when native > gas, but receive no benefit when native < gas — directly analogous to the Beranames `renew`/`register` discount inconsistency.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
``` [1](#0-0) 

When `delta_gas > 0`, `gas_used` is increased so the user pays enough to cover native resource consumption:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [2](#0-1) 

When `delta_gas < 0` — meaning `native_used / native_per_gas < gas_used`, i.e., the user's actual native resource consumption is lower than their EVM gas consumption implies — `gas_used` is **not decreased**. The `TODO` comment at line 80 explicitly acknowledges this missing branch.

This function is called by both the ZK transaction flow (`ZkTransactionFlow::before_refund`) and the Ethereum transaction flow (`EthereumTransactionFlow::before_refund`): [3](#0-2) [4](#0-3) 

The `delta_gas` adjustment is documented in `docs/double_resource_accounting.md`:

> If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. [5](#0-4) 

The documentation only describes the `deltaGas > 0` case. The `deltaGas < 0` case — where the user should receive a gas refund for under-consumed native resources — is entirely absent from both the documentation and the implementation.

The final `gas_used` value is then used to compute the operator payment and the user refund:

```rust
let token_to_refund = context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
let token_to_pay_operator = U256::from(context.gas_used).checked_mul(gas_price_for_operator)?;
``` [6](#0-5) [7](#0-6) 

Because `gas_used` is not reduced when `delta_gas < 0`, the operator receives excess fees and the user receives a smaller refund than warranted.

---

### Impact Explanation

When `delta_gas < 0`, the user overpays by `|delta_gas| * gas_price` tokens per transaction. The operator receives this excess. The user's balance is reduced more than their actual resource consumption justifies. This is a direct, quantifiable financial loss for any transaction sender whose native resource consumption falls below `gas_used * native_per_gas`.

---

### Likelihood Explanation

`delta_gas < 0` occurs when `native_used / native_per_gas < gas_used`. Since `native_per_gas = ceil(gas_price / native_price)`, this condition is most easily triggered when:

1. **High gas price**: A user paying a high gas price (e.g., to prioritize inclusion) causes `native_per_gas` to be large, making `native_used / native_per_gas` small relative to `gas_used`.
2. **EVM-gas-heavy but native-light transactions**: Transactions that consume significant EVM gas through operations with low native cost (e.g., many cheap arithmetic opcodes, JUMP-heavy code) produce a `gas_used` that exceeds `native_used / native_per_gas`.

Both scenarios are reachable by any unprivileged transaction sender without any special access.

---

### Recommendation

Apply the `delta_gas` adjustment symmetrically. When `delta_gas < 0`, decrease `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so users receive a gas refund for under-consumed native resources:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Refund gas for under-consumed native resources
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction);
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

The `TODO` comment at line 80 should be resolved with this fix.

### Proof of Concept

Consider a transaction with:
- `gas_limit = 100_000`, `gas_price = 1_000_000`, `native_price = 1`
- Derived: `native_per_gas = 1_000_000`
- EVM execution consumes: `gas_used = 50_000` (after EVM refunds)
- Actual native resources consumed: `native_used = 10_000_000_000` (= 10,000 gas-equivalent)

In `compute_gas_refund`:
```
delta_gas = (10_000_000_000 / 1_000_000) - 50_000
          = 10_000 - 50_000
          = -40_000
```

Since `delta_gas < 0`, no adjustment is made. `gas_used` remains `50_000`.

- **Actual fee charged**: `50_000 * 1_000_000 = 50_000_000_000` tokens
- **Correct fee** (based on native consumption): `10_000 * 1_000_000 = 10_000_000_000` tokens
- **Overpayment**: `40_000_000_000` tokens transferred to the operator in excess [1](#0-0)
