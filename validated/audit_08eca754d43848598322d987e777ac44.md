The external report describes a Solidity smart contract vulnerability in the `remora-dynamic-tokens` project — specifically a two-contract architecture (`PaymentSettler` + `CentralToken`) where an admin mistake in calling `enableBurning` on the wrong contract causes `totalBurnPayout` to be overwritten, leading to stuck funds. The vulnerability requires admin-level actions to trigger.

Checking sei-chain for an analogous pattern requires finding:
1. A burn enable/disable flag that can be set from two separate code paths
2. Accumulated fund accounting that gets overwritten on re-enable
3. An unprivileged attacker path
