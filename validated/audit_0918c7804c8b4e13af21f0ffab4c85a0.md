### Title
Whitelisted-Address Removal Permanently Freezes ERC-20 Token Balances With No Admin Recovery Path - (`engine/src/contract_methods/silo/mod.rs`, `engine/src/engine.rs`)

---

### Summary

In Aurora Engine Silo mode, a user whose EVM `Address` or NEAR `AccountId` is removed from the active whitelist (or whose entry was never added after the whitelist was enabled) loses all ability to exit ERC-20 tokens they already hold inside the EVM. The sole ERC-20 exit path — the `ExitToNear` precompile — is only reachable by submitting an EVM transaction via `submit` or `call`, both of which are gated by `is_allow_submit`. No admin-controlled force-exit or recovery function exists. The tokens are permanently frozen.

---

### Finding Description

Silo mode enforces two independent whitelist dimensions:

- **`WhitelistKind::Address`** — restricts which EVM `Address` (derived from the transaction signature) may execute transactions.
- **`WhitelistKind::Account`** — restricts which NEAR `AccountId` may call `submit` or `call`.

Both are checked together in `is_allow_submit`: [1](#0-0) 

The internal helpers enforce the check: [2](#0-1) 

This gate is evaluated inside `engine::submit` (called from both `submit` and `submit_with_args`) and inside `Engine::call_with_args` (called from `call`). All three NEAR entry points that can execute EVM bytecode pass through this check. [3](#0-2) 

The **only** mechanism to exit ERC-20 tokens from the Aurora EVM back to NEAR is the `ExitToNear` precompile at address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`. It is invoked by an ERC-20 contract's burn/withdraw function, which itself must be triggered by an EVM transaction submitted via `submit` or `call`. [4](#0-3) 

The `withdraw` connector function handles only the ETH base token and does not check the whitelist, but it has no effect on ERC-20 balances: [5](#0-4) 

There is no `force_exit`, `admin_withdraw`, or equivalent function that allows the owner or any privileged role to move ERC-20 tokens on behalf of a non-whitelisted address. The whitelist management functions (`remove_entry_from_whitelist`, `set_whitelist_status`) only modify access control state; they do not provide a recovery path for already-held balances. [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 token balance held by a user inside the Aurora Silo EVM becomes permanently inaccessible if:
1. The `WhitelistKind::Address` or `WhitelistKind::Account` whitelist is enabled, **and**
2. The user's EVM address or NEAR account is absent from the respective whitelist.

The tokens exist in EVM storage but cannot be moved, burned, or exited by anyone — including the contract owner. The owner can re-add the user to the whitelist as a workaround, but there is no direct admin recovery path that bypasses the whitelist to recover funds.

---

### Likelihood Explanation

**Medium.** Silo mode is an explicitly supported operational mode. Whitelist entries are expected to be managed over time (compliance changes, user offboarding, key rotation). Any operational action that removes or fails to include a user's address while that user holds ERC-20 tokens triggers the freeze. The whitelist can also be enabled retroactively via `set_whitelist_status`, which would simultaneously freeze all users not yet added. [7](#0-6) 

---

### Recommendation

Introduce an owner-only `force_exit_erc20` (or equivalent) function that can execute an `ExitToNear` precompile call on behalf of a specified EVM address, bypassing the `is_allow_submit` whitelist check. This mirrors the fix suggested in the reference report: privileged roles should be able to recover locked funds regardless of the access-control status of the affected address. Alternatively, the `ExitToNear` precompile path should be exempted from the whitelist check when the caller is acting solely to withdraw their own balance to their own NEAR account.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode with `WhitelistKind::Address` enabled.
2. Add `user_address` to the whitelist. User calls `submit` with a signed EVM transaction that deposits ERC-20 tokens (via `ft_on_transfer` → `receive_erc20_tokens`). `is_allow_receive_erc20_tokens` passes; tokens are minted to `user_address` in EVM state. [8](#0-7) 
3. Owner calls `remove_entry_from_whitelist` for `user_address`.
4. User attempts to call `submit` with a transaction invoking the ERC-20 `withdraw` function (which calls `ExitToNear`). `is_allow_submit` returns `false` → `EngineErrorKind::NotAllowed` → transaction rejected. [9](#0-8) 
5. Owner has no function to force-exit the tokens. ERC-20 balance is permanently frozen.

}

impl<I: IO> Precompile for ExitToNear<I> {
    fn required_gas(_input: &[u8]) -> Result<EthGas, ExitError> {
        Ok(costs::EXIT_TO_NEAR_GAS)
    }

    #[allow(clippy::too_many_lines)]
    fn run(
        &self,
        input: &[u8],
        target_gas: Option<EthGas>,
        context: &Context,
        is_static: bool,
    ) -> EvmPrecompileResult {
        // ETH (base) transfer input format: (85 bytes)
        //  - flag (1 byte)
        //  - refund_address (20 bytes), present if the feature "error_refund" is enabled
        //  - recipient_account_id (max MAX_INPUT_SIZE - 20 - 1 bytes)
        // ERC-20 transfer input format: (124 bytes)
        //  - flag (1 byte)
        //  - refund_address (20 bytes), present if the feature "error_refund" is enabled.
        //  - amount (32 bytes)
        //  - recipient_account_id (max MAX_INPUT_SIZE - 1 - (20) - 32 bytes)
        //  - `:unwrap` suffix in a case of wNEAR (7 bytes)
        let required_gas = Self::required_gas(input)?;

        if let Some(target_gas) = target_gas
            && required_gas > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        // It's not allowed to call exit precompiles in static mode
        if is_static {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_STATIC")));
        } else if context.address != exit_to_near::ADDRESS.raw() {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_DELEGATE")));
        }

        let exit_to_near_params = ExitToNearParams::try_from(input)?;

        let (nep141_address, args, exit_event, method, transfer_near_args) =
            match exit_to_near_params {
                // ETH(base) token transfer
                //
                // Input slice format:
                //  recipient_account_id (bytes) - the NEAR recipient account which will receive
                //  NEP-141 (base) tokens, or also can contain the `:unwrap` suffix in case of
                //  withdrawing wNEAR, or another message of JSON in case of OMNI, or address of
                //  receiver in case of transfer tokens to another engine contract.
                ExitToNearParams::BaseToken(ref exit_params) => {
                    let eth_connector_account_id = self.get_eth_connector_contract_account()?;
                    exit_base_token_to_near(eth_connector_account_id, context, exit_params)?
                }
                // ERC-20 token transfer
                //
                // This precompile branch is expected to be called from the ERC-20 burn function.
                //
                // Input slice format:
                //  amount (U256 big-endian bytes) - the amount that was burned
                //  recipient_account_id (bytes) - the NEAR recipient account which will receive
                //  NEP-141 tokens, or also can contain the `:unwrap` suffix in case of withdrawing
                //  wNEAR, or another message of JSON in case of OMNI, or address of receiver in case
                //  of transfer tokens to another engine contract.
                ExitToNearParams::Erc20TokenParams(ref exit_params) => {
                    exit_erc20_token_to_near(context, exit_params, &self.io)?
                }
            };

        let callback_args = ExitToNearPrecompileCallbackArgs {
            #[cfg(feature = "error_refund")]
            refund: refund_call_args(&exit_to_near_params, &exit_event),
            #[cfg(not(feature = "error_refund"))]
            refund: None,
            transfer_near: transfer_near_args,
        };
        let attached_gas = if method == "ft_transfer_call" {
            costs::FT_TRANSFER_CALL_GAS
        } else {
            costs::FT_TRANSFER_GAS
        };

        let transfer_promise = PromiseCreateArgs {
            target_account_id: nep141_address,
            method,
            args: args.into_bytes(),
            attached_balance: Yocto::new(1),
            attached_gas,
        };

        let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
            PromiseArgs::Create(transfer_promise)
        } else {
            PromiseArgs::Callback(PromiseWithCallbackArgs {
                base: transfer_promise,
                callback: PromiseCreateArgs {
                    target_account_id: self.current_account_id.clone(),
                    method: "exit_to_near_precompile_callback".to_string(),
                    args: borsh::to_vec(&callback_args).unwrap(),
                    attached_balance: Yocto::new(0),
                    attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,
                },
            })
        };
        let promise_log = Log {
            address: exit_to_near::ADDRESS.raw(),
            topics: Vec::new(),
            data: borsh::to_vec(&promise).unwrap(),
        };
        let ethabi::RawLog { topics, data } = exit_event.encode();
        let exit_event_log = Log {
            address: exit_to_near::ADDRESS.raw(),
            topics: topics.into_iter().map(|h| H256::from(h.0)).collect(),
            data,
        };

        Ok(PrecompileOutput {
            logs: vec![promise_log, exit_event_log],
            cost: required_gas,
            output: Vec::new(),
        })
    }
