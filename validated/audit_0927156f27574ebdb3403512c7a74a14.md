### Title
Empty `state_migration()` Leaves Upgrade-Induced State Corruption Unmitigated - (File: `engine/src/lib.rs`)

---

### Summary

The Aurora Engine contract is upgradeable. Both upgrade paths (`upgrade()` and `deploy_upgrade()`) schedule `state_migration()` as a mandatory post-deploy callback. However, `state_migration()` is permanently a no-op. Only `EngineState` has versioned Borsh deserialization with lazy migration; all other persistent state namespaces (connector, hashchain, silo, whitelist) are stored as raw, unversioned Borsh bytes with no migration logic. Any future upgrade that changes the serialized layout of those structs will leave on-chain bytes in the old format, causing every affected contract method to panic and permanently freezing all bridged ETH and ERC-20 assets.

---

### Finding Description

**Upgrade flow**

Both upgrade paths converge on calling `state_migration()` immediately after the new WASM binary is live:

- `upgrade()` in `engine/src/contract_methods/admin.rs` creates a NEAR promise batch that deploys the new code and then calls `state_migration`: [1](#0-0) 

- `self_deploy()` in `engine-sdk/src/near_runtime.rs` does the same for the `deploy_upgrade` path: [2](#0-1) 

**`state_migration()` is a permanent no-op**

The callback that is supposed to transform on-chain bytes to match the new code does nothing: [3](#0-2) 

**Only `EngineState` is protected**

`EngineState` uses a versioned Borsh enum (`BorshableEngineState` V1/V2/V3) with lazy in-place migration in `try_from_slice`: [4](#0-3) 

Every other persistent namespace has no such protection. The `KeyPrefix` enum exposes at least six additional storage namespaces that hold Borsh-serialized structs: [5](#0-4) 

Concrete examples of unversioned persistent structs:

- `WithdrawSerializeType` / connector account ID stored under `KeyPrefix::EthConnector`: [6](#0-5) 

- Hashchain data under `KeyPrefix::Hashchain`, silo parameters under `KeyPrefix::Silo`, whitelist entries under `KeyPrefix::Whitelist` — none have versioned deserialization or migration logic.

**Historical precedent for schema evolution**

`EngineState` itself has been extended three times (V1 → V2 → V3), proving that state schema changes are a recurring pattern in this codebase: [7](#0-6) 

The `state_migration()` TODO comment acknowledges that migrations will eventually be required.

---

### Impact Explanation

If a new contract version adds or reorders a field in any unversioned persistent struct (e.g., connector account, hashchain header, silo params, whitelist entry), the stored bytes remain in the old layout. The new code calls `BorshDeserialize::try_from_slice` on those bytes, fails, and panics. Because state is read on every invocation of `submit`, `call`, `ft_transfer`, `withdraw`, `ft_on_transfer`, and all other user-facing methods, the entire contract becomes permanently bricked. All bridged ETH (held as NEP-141 tokens) and all ERC-20 mirror tokens are permanently frozen — **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

- The contract has an explicit, production-used upgrade mechanism and has shipped multiple versions (CHANGES.md documents version history from 3.x onward).
- The `// TODO: currently we don't have migrations` comment signals developer awareness that this gap exists.
- `EngineState` has already been extended three times; the same evolution is expected for connector and silo state as features grow.
- There is no compile-time or CI guard preventing a developer from changing a persistent struct without updating `state_migration()`.

---

### Recommendation

1. Add versioning (enum-based, mirroring `BorshableEngineState`) to every persistent struct stored under `KeyPrefix::EthConnector`, `KeyPrefix::Hashchain`, `KeyPrefix::Silo`, and `KeyPrefix::Whitelist`.
2. Implement `state_migration()` with concrete migration logic whenever any persistent struct layout changes.
3. Add a CI check or integration test that asserts `state_migration()` is non-empty before any release that modifies persistent state layouts.

---

### Proof of Concept

1. Current `state_migration()` no-op: [3](#0-2) 

2. `self_deploy()` always schedules `state_migration` as a post-deploy callback: [8](#0-7) 

3. `upgrade()` creates the same batch action: [9](#0-8) 

4. Only `EngineState` has versioned migration; all other namespaces do not: [4](#0-3) 

5. Unversioned connector state written directly as raw Borsh: [10](#0-9) 

A concrete scenario: a developer adds a `fee_recipient: Option<AccountId>` field to the connector account struct in a new release. The stored bytes at `KeyPrefix::EthConnector | EthConnectorStorageId::EthConnectorAccount` remain in the old format. After `deploy_upgrade()` completes, every call to `ft_transfer`, `withdraw`, or `ft_on_transfer` reads that key, fails Borsh deserialization, and panics — permanently bricking the bridge and freezing all user funds.

    pub fn try_from_slice<I: IO + Copy>(bytes: &[u8], io: &I) -> Result<Self, EngineStateError> {
        let Ok(borshable) = BorshableEngineState::try_from_slice(bytes) else {
            let legacy = BorshableEngineStateV1::try_from_slice(bytes)

            return Ok(result);
