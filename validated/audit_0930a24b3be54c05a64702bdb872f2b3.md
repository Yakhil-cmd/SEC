Audit Report

## Title
Missing Neuron Lock Before Inter-Canister Ledger Call in `transfer_gtc_neuron` â€” (File: rs/nns/governance/src/governance.rs)

## Summary
`transfer_gtc_neuron` in the NNS Governance canister makes an inter-canister ledger call without first acquiring a per-neuron lock or modifying any state on the donor neuron. Because `forward_whitelisted_unclaimed_accounts` on the GTC canister is publicly callable with no access-control guard, an attacker can trigger two concurrent executions of this path for the same donor neuron, causing the second governance call to pass all checks and attempt a duplicate ledger transfer. The result is corruption of the GTC's `failed_transferred_neurons` audit log and a potential `cached_neuron_stake_e8s` overcounting on the recipient neuron.

## Finding Description
`transfer_gtc_neuron` at `rs/nns/governance/src/governance.rs` lines 1863â€“1921 performs the following sequence:

1. **Checks** (lines 1869â€“1894): verifies caller is GTC, reads donor state.
2. **Interaction** (lines 1902â€“1911): calls `self.ledger.transfer_funds(...).await` â€” no lock held, donor neuron still present in store.
3. **Effects** (lines 1913â€“1918): removes donor neuron, increments recipient `cached_neuron_stake_e8s`.

Every other governance function that makes a ledger call acquires a lock first:
- `disburse_neuron` acquires `_neuron_lock` at line 2031 before any `await`.
- `split_neuron` acquires `_parent_lock` at line 2233 before the ledger call.
- `disburse_to_neuron` acquires `_parent_lock` at line 2987 before the ledger call.

`transfer_gtc_neuron` omits this entirely.

The entry path is `forward_whitelisted_unclaimed_accounts` on the GTC canister (`rs/nns/gtc/canister/canister.rs` lines 174â€“182), which carries **no caller check** â€” the comment explicitly states "This method may be called by anyone 6 months after the IC Genesis." The only guard is the time-based `assert_forward_whitelisted_unclaimed_accounts_can_be_called` check (`rs/nns/gtc/src/lib.rs` lines 160â€“168), which has been satisfied since 188 days after IC genesis (May 2021).

Concurrent execution path:
1. Attacker sends ingress #1 â†’ GTC enters `forward_whitelisted_unclaimed_accounts`, iterates to account X, calls `account.transfer()` â†’ calls `GovernanceCanister::transfer_gtc_neuron(neuron_A, custodian)` â†’ GTC suspends at `await`.
2. Attacker sends ingress #2 â†’ GTC processes it while suspended; `account.has_forwarded` is still `false` (set only after `account.transfer()` returns in `forward_whitelisted_unclaimed_accounts` line 119) â†’ GTC enters `account.transfer()` again for account X â†’ calls `GovernanceCanister::transfer_gtc_neuron(neuron_A, custodian)` â†’ GTC suspends.
3. Governance processes msg #1: donor neuron exists, no lock â†’ calls ledger â†’ awaits.
4. Governance processes msg #2: donor neuron still exists (not yet removed), no lock â†’ calls ledger â†’ awaits.
5. Ledger responds to msg #1: success â†’ governance removes donor neuron, adds `transfer_amount_doms` to recipient `cached_neuron_stake_e8s`.

Additionally, if a concurrent `disburse_neuron` on the recipient neuron executes between steps 5 and the additive update at line 1917, `cached_neuron_stake_e8s` is incremented on top of an already-modified value, producing a balance that does not match the on-chain ledger.

## Impact Explanation
**NNS governance state corruption with concrete protocol harm (Medium).** The GTC is an NNS canister. The `failed_transferred_neurons` list is part of certified GTC state; a spurious entry records a successfully transferred neuron as failed, permanently corrupting the GTC's audit log. If any downstream tooling or governance process relies on this list to determine which neurons require remediation, it will act on incorrect data. The secondary impact â€” `cached_neuron_stake_e8s` overcounting on the recipient neuron â€” would cause the governance canister to attribute more voting power to that neuron than its actual on-chain balance, a concrete NNS governance integrity violation. This matches the allowed impact: "Significant NNS security impact with concrete user or protocol harm."

## Likelihood Explanation
`forward_whitelisted_unclaimed_accounts` is publicly callable by any unprivileged principal after 188 days post-genesis, a condition that has been met on mainnet since late 2021. An attacker needs only to send two ingress messages in rapid succession to the GTC canister. No special privileges, leaked keys, or social engineering are required. The window is constrained by the inter-canister round-trip latency, but this is a deterministic property of the IC execution model and is reliably exploitable. Likelihood is low-to-medium given the GTC is a legacy canister and many accounts may already be forwarded, but the code path is unconditionally vulnerable for any remaining unforwarded whitelisted accounts.

## Recommendation
1. Before calling `self.ledger.transfer_funds`, acquire a lock on the donor neuron using `lock_neuron_for_command` (mirroring `disburse_neuron` at line 2031).
2. Optionally acquire a lock on the recipient neuron to prevent concurrent stake mutations during the `await`.
3. Move `self.remove_neuron(donor_neuron)` to before the ledger call (set `cached_neuron_stake_e8s = 0` as a tombstone first), mirroring the pattern in `disburse_to_neuron` (lines 3006â€“3023) where the child neuron is added with zero stake before the transfer.
