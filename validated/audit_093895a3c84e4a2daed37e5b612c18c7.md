Audit Report

## Title
VIP Caller Starvation via Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` rejects all callers — including VIP NNS canisters — when `available_slot_count == 0`, despite VIPs being designed to consume zero slots. An unprivileged attacker can exhaust all 167 slots via NNS Root's public `canister_status` endpoint, causing NNS Governance's calls to `update_canister_settings`, `stop_canister`, and related governance operations to receive `SysTransient` errors for as long as the flood is sustained.

## Finding Description
In `try_borrow_slot` (lines 264–287 of `rs/nervous_system/clients/src/management_canister_client.rs`), the intent is that VIP callers (`is_caller_vip = true`) consume zero slots (`used_slot_count = 0`) and are therefore never blocked. However, the early-exit guard at line 269 fires unconditionally:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {   // ← no VIP exemption
                return Err((RejectCode::SysTransient as i32,
                            "Unavailable. Maybe, try again later?".to_string()));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

When `available_slot_count` reaches 0, VIP callers are rejected before ever reaching the `saturating_sub(0)` line that would have been a no-op. The VIP path is dead code under exhaustion.

**Attack surface:** `canister_status` at lines 88–98 of `rs/nns/handlers/root/impl/canister/canister.rs` is a public `#[update]` with no access control. Any principal can call it. Each call invokes `new_management_canister_client()` (lines 53–67), which creates a `LimitedOutstandingCallsManagementCanisterClient` with `is_caller_vip = false` for non-NNS callers, consuming one slot per outstanding call.

**Slot pool:** `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is initialized to 167 (line 50). All governance operations — `update_canister_settings` (line 221), `stop_or_start_nns_canister` (line 175), `change_canister_controllers` (line 207), `take_canister_snapshot` (line 245), `load_canister_snapshot` (line 259) — all call `new_management_canister_client()` and share this same thread-local counter.

**Exploit flow:**
1. Attacker deploys a canister and sends 167 concurrent inter-canister calls to NNS Root's `canister_status`.
2. All 167 slots are consumed; `available_slot_count` reaches 0.
3. NNS Governance (a VIP) calls `update_canister_settings` → `new_management_canister_client()` → `try_borrow_slot()` → hits `== 0` → returns `Err((SysTransient, "Unavailable..."))`.
4. The governance operation fails. Attacker repeats each round (~1–2 seconds) to maintain exhaustion.

## Impact Explanation
This is an application/platform-level DoS against NNS governance operations. NNS proposal execution for canister upgrades, controller changes, and emergency stops is stalled for the duration of the attack. This matches the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS... not based on raw volumetric DDoS." The attack exploits a logic error in the rate-limiting mechanism, not raw packet flooding.

## Likelihood Explanation
The attacker needs only a canister with sufficient cycles to sustain 167 concurrent calls per round. The management canister responds within 1–2 rounds, so the attacker must re-flood each round. This is cycle-expensive but automatable and repeatable. No special privileges, leaked keys, or social engineering are required — any principal on the IC can execute this.

## Recommendation
Add a VIP exemption to the `== 0` guard in `try_borrow_slot`:

```rust
if *available_slot_count == 0 && !self.is_caller_vip {
    return Err((RejectCode::SysTransient as i32,
                "Unavailable. Maybe, try again later?".to_string()));
}
```

This ensures VIP callers always pass through regardless of slot counter state, while non-VIP callers remain rate-limited.

## Proof of Concept
State-machine or PocketIC integration test:
1. Initialize NNS Root with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Deploy an attacker canister (non-NNS principal).
3. From the attacker canister, send 167 concurrent calls to NNS Root's `canister_status` targeting any NNS-controlled canister; hold them in-flight (e.g., target a canister that delays response).
4. Before any response returns, send `update_canister_settings` from NNS Governance (VIP caller).
5. Assert the Governance call returns `Err((SysTransient, "Unavailable..."))`.
6. Allow the 167 attacker calls to complete; assert `available_slot_count` recovers to 167 and a subsequent Governance call succeeds.
