Audit Report

## Title
SNS Swap `refresh_buyer_token_e8s` Permanently Locks Remaining Capacity When All Participants Are at Per-Participant Maximum - (File: rs/sns/swap/src/swap.rs)

## Summary

When `available_direct_participation_e8s()` drops below `min_participant_icp_e8s` and every existing buyer is already at `max_participant_icp_e8s`, no participant can consume the remaining capacity. New participants are rejected by Guard 2, and existing participants at their cap have their `new_balance_e8s` silently capped back to `max_participant_icp_e8s` (a no-op write), leaving the remainder permanently unconsumed. The swap can never reach `max_direct_participation_icp_e8s` under these conditions.

## Finding Description

**Root cause — two interacting caps in `refresh_buyer_token_e8s`:**

`max_increment_e8s` is set to `available_direct_participation_e8s()` at the top of the function. [1](#0-0) 

Guard 2 rejects any call whose effective `new_balance_e8s` (after capping at `max_increment_e8s`) is below `min_participant_icp_e8s`: [2](#0-1) 

The per-participant cap is applied *before* Guard 2: [3](#0-2) 

**Exploit trace (concrete example: max=100 ICP, min_participant=10 ICP, max_participant=33 ICP):**

1. Users A, B, C each participate with 33 ICP → `available_direct_participation_e8s() = 1 ICP`.
2. **New participant** transfers 10 ICP (≥ min), calls `refresh_buyer_tokens`:
   - `actual_increment = min(1, 10) = 1`; `new_balance = 1 < 10` → Guard 2 rejects.
3. **Existing participant at max** transfers 1 extra ICP, calls `refresh_buyer_tokens`:
   - `requested_increment = 1`; `actual_increment = min(1, 1) = 1`; `new_balance = 34`
   - Line 1237 caps: `new_balance = min(34, 33) = 33` (same as `old_amount`)
   - Guard 2 passes (`33 ≥ 10`), but `set_amount_icp_e8s(33)` is a no-op. [4](#0-3) 
4. `available_direct_participation_e8s()` remains 1 ICP forever; the swap expires at 99 ICP.

**Existing validation is insufficient:**

`validate_participation_constraints` enforces `max_participant_icp_e8s ≤ max_direct_participation_icp_e8s` and `max_direct ≥ min_participants × min_participant`, but does not require that `max_direct_participation_icp_e8s % max_participant_icp_e8s == 0` or that the remainder is `≥ min_participant_icp_e8s`. [5](#0-4) 

**Existing test gap:**

`test_swap_cannot_finalize_via_new_participation_if_remaining_lt_minimal_participation_amount` only covers the case where an existing participant is *below* `max_participant_icp_e8s` and can increase to fill the gap. It uses `max_participant_icp_e8s = 500_000 * E8` (equal to `max_direct`), so the "all participants at cap" scenario is never exercised. [6](#0-5) 

## Impact Explanation

This is a **High** severity SNS security impact. An SNS project's `max_direct_participation_icp_e8s` becomes permanently unreachable whenever the remainder condition is triggered. The swap commits (if `min_direct_participation` was met) but raises less ICP than the configured maximum, directly harming the project's tokenomics (token price, neuron basket sizing). The locked amount is bounded by `min_participant_icp_e8s − 1` e8s per occurrence, but for swaps with large minimum participation amounts this can represent a material shortfall. This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation

Triggerable by any unprivileged principal via the public `#[update]` endpoint `refresh_buyer_tokens`. [7](#0-6) 

The condition arises naturally whenever `max_direct_participation_icp_e8s % max_participant_icp_e8s ≠ 0` and the remainder is `< min_participant_icp_e8s`. A malicious actor can deliberately trigger it by being the last participant to fill to exactly `max_participant_icp_e8s`, pushing the remainder below the minimum. No special privileges, governance majority, or key material are required.

## Recommendation

In `refresh_buyer_token_e8s`, waive Guard 2 when the participant is consuming all remaining swap capacity (i.e., `actual_increment_e8s == max_increment_e8s`):

```rust
let is_filling_remaining_capacity = actual_increment_e8s == max_increment_e8s;
if new_balance_e8s < params.min_participant_icp_e8s && !is_filling_remaining_capacity {
    return Err(format!(
        "Rejecting participation of effective amount {}; minimum required to participate: {}",
        new_balance_e8s, params.min_participant_icp_e8s
    ));
}
```

Additionally, add a check in `validate_participation_constraints` requiring that `max_direct_participation_icp_e8s % max_participant_icp_e8s == 0` or `max_direct_participation_icp_e8s % max_participant_icp_e8s >= min_participant_icp_e8s`, preventing the problematic parameter combination at configuration time. [5](#0-4) 

## Proof of Concept

Unit test plan (extend `rs/sns/swap/tests/swap.rs`):

1. Configure swap: `max_direct = 100 * E8`, `min_participant = 10 * E8`, `max_participant = 33 * E8`, `min_participants = 1`.
2. Three distinct principals each call `refresh_buyer_tokens` with ledger balance `33 * E8`. Assert each succeeds and `available_direct_participation_e8s() == 1 * E8`.
3. A fourth principal transfers `10 * E8` (≥ min) and calls `refresh_buyer_tokens`. Assert the call returns `Err("Rejecting participation of effective amount 1...")`.
4. One of the first three principals transfers `1 * E8` more (ledger = `34 * E8`) and calls `refresh_buyer_tokens`. Assert the call returns `Ok` but `current_direct_participation_e8s()` is still `99 * E8` and `available_direct_participation_e8s()` is still `1 * E8`.
5. Assert the swap cannot reach `max_direct_participation_icp_e8s` before the deadline.

#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    log!(INFO, "refresh_buyer_tokens");
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()
    };
    let icp_ledger = create_real_icp_ledger(swap().init_or_panic().icp_ledger_or_panic());
    match swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
    {
        Ok(r) => r,
        Err(msg) => panic!("{}", msg),
    }
}
