Audit Report

## Title
Node Provider Monthly Reward Entirely Rejected Instead of Capped When Exceeding `maximum_node_provider_rewards_e8s`, Causing Permanent ICP Loss - (File: rs/nns/governance/src/governance.rs)

## Summary
In `reward_node_provider_helper`, when a node provider's computed ICP reward exceeds `maximum_node_provider_rewards_e8s`, the function returns a hard `Err` rather than minting the capped maximum. Because `mint_monthly_node_provider_rewards` discards this error with `let _` and unconditionally advances the monthly reward timestamp via `update_most_recent_monthly_node_provider_rewards`, the affected node provider receives zero ICP for that period and the reward is permanently unrecoverable.

## Finding Description
The reward flow is:

1. `get_node_provider_reward` (L8248‚Äì8271) computes `amount_e8s` as a raw XDR‚ÜíICP conversion with no cap applied.
2. `reward_node_provider_helper` (L3953‚Äì3960) checks `reward.amount_e8s > maximum_node_provider_rewards_e8s` and returns `Err(PreconditionFailed)` ‚Äî no ICP is minted, not even the capped amount.
3. `reward_node_providers` (L3993‚Äì4003) logs the per-provider error and continues; the aggregate error propagates upward.
4. `mint_monthly_node_provider_rewards` (L4073‚Äì4076) discards the aggregate result with `let _` and unconditionally calls `update_most_recent_monthly_node_provider_rewards`, advancing the monthly timestamp.
5. `maximum_node_provider_rewards_e8s` is fetched at L7678 and stored in `MonthlyNodeProviderRewards` for auditing only ‚Äî it is never used to clamp the reward at calculation time.

The result is that any node provider whose XDR reward converts to more e8s than the cap receives zero ICP, and because the timestamp advances, no retry is possible.

## Impact Explanation
This is a permanent, silent loss of ICP rewards for legitimate node providers ‚Äî a financial loss bug in NNS governance, which is explicitly in-scope. The affected amounts can be substantial: at low ICP prices, large node providers can accumulate XDR rewards that convert to hundreds of thousands of ICP, all of which are silently discarded. This matches the High impact class: "Significant NNS security impact with concrete user or protocol harm." If the lost amount exceeds $1M in value (plausible during ICP price drops for large node providers), it escalates to Critical: "Permanent loss involving exorbitant ICP."

## Likelihood Explanation
No privileged action is required. The trigger is purely market-driven: a sustained drop in ICP price causes the same XDR reward to convert to a larger e8s amount. The periodic task `mint_monthly_node_provider_rewards` fires automatically. Large node providers with many nodes in high-reward regions are at risk whenever the ICP/XDR rate falls sufficiently. No governance vote, admin key, or attacker action is needed ‚Äî the bug fires on its own.

## Recommendation
In `reward_node_provider_helper`, replace the hard rejection with a cap:

```rust
let amount_to_mint = reward.amount_e8s.min(maximum_node_provider_rewards_e8s);
// construct a capped RewardNodeProvider and pass it to mint_reward_to_neuron_or_account
```

Additionally, the `let _` discard in `mint_monthly_node_provider_rewards` should be removed. At minimum, a failed reward should prevent the monthly timestamp from advancing for the affected provider so the reward can be retried in the next cycle.

## Proof of Concept
1. Node provider NP-A has 500 type-0 nodes; monthly XDR reward = 5,000,000 XDR permyriad.
2. 30-day average ICP/XDR rate drops to `xdr_permyriad_per_icp = 10_000` (1 XDR per ICP).
3. `get_node_provider_reward` computes: `(5_000_000 * 100_000_000) / 10_000 = 50_000_000_000_000 e8s` = 500,000 ICP.
4. `maximum_node_provider_rewards_e8s` = `10_000_000_000_000` (100,000 ICP, lowered via `ManageNetworkEconomics`).
5. `reward_node_provider_helper` sees `500,000 ICP > 100,000 ICP`, returns `Err(PreconditionFailed)`.
6. `reward_node_providers` logs the error; `mint_monthly_node_provider_rewards` discards it with `let _`.
7. `update_most_recent_monthly_node_provider_rewards` advances the timestamp unconditionally.
8. NP-A receives 0 ICP; the 500,000 ICP reward is permanently lost.

A deterministic integration test can reproduce this by: setting `maximum_node_provider_rewards_e8s` to a low value via `ManageNetworkEconomics`, registering a node provider with a reward that exceeds the cap, triggering `mint_monthly_node_provider_rewards`, and asserting that the node provider's account balance is zero and the monthly timestamp has advanced.
