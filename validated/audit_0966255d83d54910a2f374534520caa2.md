### Title
Missing Staleness Check on CMC Exchange Rate in SNS Treasury Valuation — (File: `rs/sns/governance/token_valuation/src/lib.rs`)

### Summary
The SNS governance canister fetches the ICP/XDR exchange rate from the Cycles Minting Canister (CMC) to compute treasury valuations that gate `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals, but never validates the `timestamp_seconds` of the returned rate. If the CMC's stored rate is stale (due to XRC unavailability), the SNS governance uses an outdated rate to compute treasury transfer limits, potentially allowing a complete treasury drain for a treasury that should be subject to a fractional weekly limit.

### Finding Description

In `rs/sns/governance/token_valuation/src/lib.rs`, the inner struct `CmcBased30DayMovingAverageXdrsPerIcpClient` implements `XdrsPerIcpClient::get()` by calling `get_average_icp_xdr_conversion_rate` on the CMC:

The `IcpXdrConversionRateCertifiedResponse` carries a `data.timestamp_seconds` field indicating when the rate was last updated, but this field is never read or compared against the current time. The code proceeds directly to use `response.data.xdr_permyriad_per_icp` without any staleness guard.

The upstream `validate_exchange_rate` function — shared between the CMC heartbeat and the NNS Governance's XRC client — only validates source counts, not the rate's timestamp:

```rust

The CMC's `do_set_icp_xdr_conversion_rate` only enforces that a new rate's timestamp is strictly greater than the currently stored rate — it does not check that the rate is recent relative to `now`:

The CMC heartbeat refreshes the rate every 5 minutes (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`), but if the XRC is unavailable for an extended period, the stored `average_icp_xdr_conversion_rate` becomes stale with no mechanism to signal this to callers. [4](#0-3) 

The stale rate flows into `treasury_valuation_if_proposal_amount_is_small_enough_or_err`, which computes the 7-day transfer limit for `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals: [5](#0-4) 

### Impact Explanation

The 7-day treasury transfer limit has a threshold: treasuries valued below 100,000 XDR are treated as "small" and the limit is 100% of the treasury (all tokens may be transferred in one proposal). Treasuries above this threshold are subject to a fractional weekly cap.

If the CMC's stored rate is stale and **deflated** (ICP price has risen significantly since the last successful XRC update), a treasury that is actually large (e.g., worth 500,000 XDR at the real price) will appear small (e.g., worth 80,000 XDR at the stale price). The SNS governance will then classify it as a "small" treasury and allow 100% of the tokens to be transferred in a single proposal — bypassing the fractional weekly limit entirely. An SNS governance participant with sufficient voting power could drain the entire treasury in one proposal window.

### Likelihood Explanation

Low. The CMC updates its rate every 5 minutes via heartbeat. For the rate to become significantly stale, the XRC canister must be unavailable for an extended period (e.g., during an upgrade or a bug). An unprivileged attacker cannot directly cause XRC unavailability. However, during any such operational gap, an SNS governance participant who holds sufficient voting power can submit and pass a treasury-draining proposal before the rate is refreshed. The window is bounded by the XRC downtime duration, which has historically occurred during planned upgrades.

### Recommendation

In `CmcBased30DayMovingAverageXdrsPerIcpClient::get()`, after receiving the response, compare `response.data.timestamp_seconds` against the current canister time (`ic_cdk::api::time() / 1_000_000_000`). If the rate is older than an acceptable staleness threshold (e.g., 24–48 hours), return a `ValuationError::External` rather than using the stale rate. This mirrors the pattern already used in the NNS Governance's `fetch_and_validate_rate`, which rejects rates whose timestamps do not match the requested day. [6](#0-5) 

### Proof of Concept

1. The XRC canister becomes unavailable for 48 hours (e.g., during an upgrade). The CMC's heartbeat fails to update the rate; `average_icp_xdr_conversion_rate.timestamp_seconds` is now 48 hours old.
2. During this period, ICP price doubles (e.g., from 5 XDR/ICP to 10 XDR/ICP). The stale CMC rate still shows 5 XDR/ICP.
3. An SNS treasury holds 25,000 ICP tokens. At the real price, the treasury is worth 250,000 XDR (large treasury, fractional weekly limit applies). At the stale price, it appears worth 125,000 XDR — still large, but if the price had tripled, it would appear below 100,000 XDR.
4. An SNS governance participant with sufficient voting power submits a `TransferSnsTreasuryFunds` proposal to transfer all 25,000 ICP tokens.
5. `treasury_valuation_if_proposal_amount_is_small_enough_or_err` calls `CmcBased30DayMovingAverageXdrsPerIcpClient::get()`, receives the stale rate, computes an undervalued treasury, classifies it as "small", and sets the limit to 100% of the treasury.
6. The proposal passes validation and, upon execution, drains the entire treasury — bypassing the fractional weekly limit that should have applied.

async fn treasury_valuation_if_proposal_amount_is_small_enough_or_err<MyTokenProposalAction>(
    env: &dyn Environment,
    sns_ledger_canister_id: CanisterId,
    swap_canister_id: CanisterId,
    proposals: impl Iterator<Item = &ProposalData>,
    action: &MyTokenProposalAction,
) -> Result<Valuation, String>
where
    MyTokenProposalAction: TokenProposalAction,
{
    let spent_tokens = action.recent_amount_total_tokens(proposals, env.now())?;

    // Get valuation of the tokens in the treasury.
    let token = action.token()?;
    let valuation = assess_treasury_balance(
        token,
        env.canister_id(),
        sns_ledger_canister_id,
        swap_canister_id,
    )
    .await?;

    // From valuation, determine limit on the total from the past 7 days.
    let max_tokens = MyTokenProposalAction::recent_amount_total_upper_bound_tokens(&valuation)
        // Err is most likely a bug.
        .map_err(|treasury_limit_error| {
            format!("Unable to validate amount: {treasury_limit_error:?}",)
        })?;

    // Finally, inspect the proposal's amount: it must not exceed max - spent (remainder). Or if
    // you prefer, equivalently, amount + spent must be <= max.
    let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| {
        format!("Arithmetic error while performing {max_tokens} - {spent_tokens}",)
    })?;
    let proposal_amount_tokens = action.proposal_amount_tokens()?;
    if proposal_amount_tokens > allowance_remainder_tokens {
        // Although it might not be obvious to the user, their proposal is invalid, and we
        // consider it to be "their fault".
        return Err(format!(
            "Amount is too large. Within the past 7 days, a total of {spent_tokens} tokens has already \
             been executed in like proposals. Whereas, at most {max_tokens} is allowed. An additional \
             {proposal_amount_tokens} tokens from this proposal would cause that upper bound to be exceeded. \
             Maybe, try again in a few days?"
        ));
    }

    Ok(valuation)
