Audit Report

## Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_start`/`ingress_end` Causes OOM DoS in ICP Rosetta `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `construction_payloads` handler accepts `ingress_start` and `ingress_end` nanosecond timestamps from unauthenticated request metadata and feeds them directly into an unbounded `while` loop with no window-size validation. An attacker can supply a maximally large `ingress_end − ingress_start` value, causing the server to allocate tens of millions of `u64` entries in `ingress_expiries` and then tens of millions of heap-allocated `SigningPayload` structs in `add_payloads`, exhausting process memory and crashing the single Rosetta process.

## Finding Description

The step size is fixed at compile time: [1](#0-0) 

`MAX_INGRESS_TTL = 300s`, `PERMITTED_DRIFT = 60s` → `interval = 120s = 1.2×10¹¹ ns`. [2](#0-1) 

`ingress_start` and `ingress_end` are taken verbatim from the JSON metadata with no range or window-size check: [3](#0-2) 

The loop then runs `(ingress_end − ingress_start) / interval` iterations with no cap: [4](#0-3) 

With `ingress_end − ingress_start ≈ 9.2×10¹⁸ ns` (half of `u64::MAX`), this yields ~77 million iterations and ~616 MB for `ingress_expiries` alone. Every entry in `ingress_expiries` is then iterated in `add_payloads`, which pushes **two** heap-allocated `SigningPayload` structs per expiry per transaction: [5](#0-4) 

For a single-transaction request this produces ~154 million `SigningPayload` objects, each containing heap-allocated `String` fields, totalling tens of gigabytes — a guaranteed OOM crash.

The ICRC1 counterpart has at minimum an ordering guard (`ingress_start >= ingress_end`) that the ICP version lacks entirely: [6](#0-5) 

The ICP handler has no equivalent guard anywhere between metadata parsing and the loop.

## Impact Explanation

The ICP Rosetta API runs as a single OS process with no per-request memory quota and no timeout on the allocation loop. A single malformed POST to `/construction/payloads` causes the process to OOM-crash, taking down the entire Rosetta service until manually restarted and losing all in-flight signing sessions. This matches the **High** bounty impact: *Application/platform-level DoS, crash, or significant Rosetta/financial-integration security impact with concrete user or protocol harm.*

## Likelihood Explanation

The endpoint requires no authentication. The attacker only needs to craft a valid JSON body with two large integer fields (`ingress_start`, `ingress_end`). No key material, no privileged access, no coordination, and no prior state is required. The attack is reproducible with a single HTTP request and is trivially repeatable after each restart.

## Recommendation

Add a window-size cap immediately after parsing `ingress_start`/`ingress_end`, before the allocation loop:

```rust
let max_window = interval * MAX_REASONABLE_INTERVALS; // e.g., 1440 for a 48-hour window
if ingress_end > ingress_start + max_window {
    return Err(ApiError::invalid_request(
        "ingress_end − ingress_start exceeds the maximum allowed window",
    ));
}
```

Also add the basic ordering guard already present in the ICRC1 implementation:

```rust
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request("ingress_start must be before ingress_end"));
}
```

## Proof of Concept

```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"STAKE",
      "account":{"address":"<valid-account>"},
      "metadata":{"neuron_index":0}}],
    "public_keys": [{"hex_bytes":"<valid-pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 1000000000,
      "ingress_end":   9223372036854775807
    }
  }'
```

Expected: the server process OOMs and crashes before returning a response. The `ingress_expiries` loop at lines 99–107 executes ~77 million iterations (~616 MB), followed by `add_payloads` allocating ~154 million `SigningPayload` objects (tens of GB). A local integration test can assert that the handler returns an error for any window exceeding a reasonable bound (e.g., 48 hours) without allocating.
