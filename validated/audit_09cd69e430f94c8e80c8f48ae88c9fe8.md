Audit Report

## Title
Global Cycles Minting Rate Limiter Has No Per-Principal Tracking, Allowing Single Actor to DoS All Other Users - (File: rs/nns/cmc/src/main.rs)

## Summary
The Cycles Minting Canister (CMC) maintains a single global `base_limiter` rolling-window counter with no per-principal sub-limits. Any unprivileged user who can acquire sufficient ICP can exhaust the entire hourly minting quota in one or a few transactions, causing every subsequent minting call from any other principal to be rejected with a refund for up to one hour. The behavior is confirmed by an existing integration test.

## Finding Description
The CMC state holds one shared `Limiter` instance and a single `base_cycles_limit` (150 P cycles/hour):

```
pub base_cycles_limit: Cycles,
pub base_limiter: limiter::Limiter,
``` [1](#0-0) 

Both public minting paths converge on `ensure_balance`, which atomically checks and increments this global counter before any cycles are minted: [2](#0-1) 

`deposit_cycles` (called by `notify_top_up` via `process_top_up`) calls `ensure_balance` when `mint_cycles` is true: [3](#0-2) 

`do_mint_cycles` (called by `notify_mint_cycles` via `process_mint_cycles`) also calls `ensure_balance` unconditionally: [4](#0-3) 

`Limiter::check_and_add_cycles` rejects any request that would push the rolling total above the limit, with no per-caller tracking: [5](#0-4) 

There is no per-principal accounting anywhere in the `Limiter` struct or its callers. The `CyclesMintingLimiterSelector` only distinguishes between `BaseLimit` and `SubnetRentalLimit` — it does not introduce per-caller granularity.

## Impact Explanation
This is an **application/platform-level DoS** of the CMC, a core NNS canister. When the global limiter is saturated, `notify_top_up` and `notify_mint_cycles` return `NotifyError::Refunded { reason: "…please try again later" }` to every caller for up to one hour. Canister creation funded via CMC, cycle top-ups, and cycles-ledger deposits all fail during this window. The attack is repeatable every hour. This matches the allowed High impact: *"Application/platform-level DoS … not based on raw volumetric DDoS."*

## Likelihood Explanation
The attack requires no privileged access, no key material, and no governance majority. It requires acquiring ~1,500 ICP on the open market (a real but bounded and predictable economic cost), transferring it to the CMC subaccount for a controlled canister, and calling `notify_top_up`. The ICP is burned, so the attacker does not recover it, but the cost is fixed and the attack can be repeated each hour. On-chain ICP liquidity is public. Likelihood is **medium**: the economic barrier is non-trivial but the attack is fully reachable by an unprivileged ingress sender.

## Recommendation
1. **Per-principal sub-limits**: Track minting volume per `caller()` principal within the existing rolling window. Reject only when the per-principal quota is exceeded, not the global one.
2. **Reserve-commit pattern**: Apply the pattern already implemented in `rs/nervous_system/rate_limits/src/lib.rs` so the global counter is only committed after the downstream management-canister or cycles-ledger call succeeds, preventing failed operations from consuming quota. [6](#0-5) 

## Proof of Concept
The existing integration test `cmc_notify_top_up_rate_limited` directly confirms the behavior: [7](#0-6) 

Steps:
1. Attacker acquires ≥ 1,500 ICP.
2. Attacker transfers 1,500 ICP to the CMC subaccount for their own canister and calls `notify_top_up`, minting ≈ 150 P cycles and saturating `base_limiter`.
3. All subsequent `notify_top_up` or `notify_mint_cycles` calls from any other principal within the same rolling hour window hit the `count + cycles_to_mint > limit` check in `Limiter::check_and_add_cycles` and receive `NotifyError::Refunded { reason: "…try again later" }`.
4. The attacker repeats at the start of each new rolling window.

fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;
```

**File:** rs/nns/cmc/src/limiter.rs (L34-56)
```rust
    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }

//! Rate limits library for nervous system components.
//!
//! This library uses a [Leaky Bucket Algorithm](https://en.wikipedia.org/wiki/Leaky_bucket) to
//! enforce rate limites, which allows for a configurable amount of spikiness around the average
//! limit desired.
//!
//! A reserve-commit pattern is used so that any failure would not affect the rate limit (otherwise
//! a DoS would be possible by deliberately triggering operations that would fail).
//!
//! Example:
//!
//! ```
//! let reservation = get_rate_limiter().try_reserve(now, key, requested_capacity)?;
//! do_something()?; // if this fails, the reservation is dropped without affecting the rate limit
//! get_rate_limiter().commit(now, reservation).expect("Failed to commit reservation"); // Or handle the error by logging. It should not fail unless there the resevation is invalid.
//! ```
//!
//! This pattern is also safe to use in an asynchronous context - if the reservation happens at the
//! first message of the async call, then the entire async call is protected by the rate limiting.
//! Although, in those cases, the user should consider whether the reservation should be committed
//! if the async call fails, depending on whether the rate limiting is trying to protect against the
//! inter-canister call (in which case, the reservation should be committed) or the some other
//! mutations inside the canister (in which case, the reservation should not be committed).
```

#[test]
fn cmc_notify_top_up_rate_limited() {
    let state_machine = state_machine_builder_for_nns_tests().build();

    let account = AccountIdentifier::new(*TEST_USER1_PRINCIPAL, None);
    // The only requirement here is to have sufficient funds. Other than that,
    // the precise number here does not matter.
    let balance = Tokens::new(1e6 as u64, 0).unwrap();
    let nns_init_payloads = NnsInitPayloadsBuilder::new()
        .with_test_neurons()
        .with_ledger_account(account, balance)
        .build();
    setup_nns_canisters(&state_machine, nns_init_payloads);

    // First top-up should succeed since it's 90P - less than the 150P/hr limit.
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(900, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(90e15 as u128));

    // Second top-up should also succeed after 1 hour.
    state_machine.advance_time(Duration::from_secs(4000));
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(900, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(90e15 as u128));

    // Third top-up should fail since the rate limit is 150e15 cycles per hour,
    // and less than an hour has passed.
    state_machine.advance_time(Duration::from_secs(3000));
    let error = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(900, 0).unwrap(),
    )
    .unwrap_err();
    assert_matches!(error, NotifyError::Refunded { reason, .. } if reason.contains("try again later"));
}
