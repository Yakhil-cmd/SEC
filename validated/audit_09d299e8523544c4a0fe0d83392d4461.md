### Title
Stale ICP/XDR Exchange Rate Used in Cycles Minting Without Freshness Validation - (File: `rs/nns/cmc/src/main.rs`)

---

### Summary

The Cycles Minting Canister (CMC) converts ICP to cycles using a cached `icp_xdr_conversion_rate` in `tokens_to_cycles`. This function checks only whether the rate exists, never whether it is fresh. If the Exchange Rate Canister (XRC) becomes unavailable for an extended period, the CMC silently continues minting cycles at an arbitrarily stale rate, causing users to receive incorrect cycle amounts.

---

### Finding Description

The `tokens_to_cycles` function in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` and uses only `rate.xdr_permyriad_per_icp`, completely ignoring `rate.timestamp_seconds`:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);  // timestamp never checked
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...),
        }
    })
}
``` [1](#0-0) 

The rate is refreshed via the canister heartbeat, which calls `update_exchange_rate()` targeting the XRC every `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes): [2](#0-1) [3](#0-2) 

However, `REFRESH_RATE_INTERVAL_SECONDS` is only used to schedule the next XRC call — it is **never used** to gate whether the stored rate is fresh enough to be used in a conversion. If the XRC is unavailable (returns errors), the CMC retries every minute but keeps the old rate in state indefinitely. The `do_set_icp_xdr_conversion_rate` function only rejects a new rate if its timestamp is not strictly greater than the current stored one — it does not enforce any maximum age on the stored rate itself: [4](#0-3) 

`tokens_to_cycles` is called by all three public ICP-to-cycles conversion paths:
- `process_top_up` → `notify_top_up`
- `process_create_canister` → `notify_create_canister`
- `process_mint_cycles` → `notify_mint_cycles` [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Cycles/resource accounting bug.** ICP price is volatile. If the XRC is unavailable for hours or days, the CMC continues minting cycles at the last known rate. A user who observes that the ICP market price has dropped significantly while the CMC still holds a stale high rate can send ICP and receive more cycles than the current market rate warrants — effectively extracting value from the protocol. Conversely, users during a price spike receive fewer cycles than they should. Because cycles are the fundamental compute resource on the IC and are minted by burning ICP, incorrect cycle amounts represent a direct ledger conservation violation: ICP is burned at the wrong rate relative to cycles issued.

---

### Likelihood Explanation

The XRC is an external canister dependency. Any XRC outage (canister upgrade, subnet issue, or sustained error responses) causes the CMC to accumulate rate staleness silently. The CMC's own integration test documents this behavior — when the XRC returns an error, the rate timestamp stays at the previous value and the CMC continues operating: [8](#0-7) 

Any unprivileged principal can call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` at any time. No special role is required. An attacker only needs to observe that the CMC's published `icp_xdr_conversion_rate` timestamp is stale (readable via the public `get_icp_xdr_conversion_rate` query) and that the current ICP market price has diverged favorably. [9](#0-8) 

---

### Recommendation

In `tokens_to_cycles`, compare `rate.timestamp_seconds` against the current canister time. If the rate is older than a defined maximum staleness threshold (e.g., `REFRESH_RATE_INTERVAL_SECONDS` × some multiplier, or a fixed bound such as 1 hour), return an error rather than proceeding with the conversion. This mirrors the existing `REFRESH_RATE_INTERVAL_SECONDS` intent but enforces it at the point of use, not just at the point of refresh scheduling.

---

### Proof of Concept

1. Observe the CMC's current `icp_xdr_conversion_rate` via `get_icp_xdr_conversion_rate` (public query). Note `timestamp_seconds = T` and `xdr_permyriad_per_icp = R_old`.
2. Wait for or observe an XRC outage (e.g., XRC returns errors). The CMC heartbeat will fail to update the rate; `timestamp_seconds` stays at `T`.
3. Observe that the real ICP market price has dropped to `P_new < P_old` (where `R_old` reflects `P_old`).
4. Call `notify_top_up` or `notify_mint_cycles` with ICP. The CMC calls `tokens_to_cycles`, which reads `R_old` without checking `T`, and mints cycles at the stale high rate.
5. The caller receives `cycles = ICP_amount × R_old × cycles_per_xdr / 10_000` — more cycles than the current market rate justifies. [10](#0-9)
