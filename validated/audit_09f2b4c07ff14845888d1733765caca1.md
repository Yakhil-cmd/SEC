Audit Report

## Title
Unbounded Header Count Causes Multi-Layer Memory Amplification Per Stream, Enabling OOM Crash via Concurrent Streams — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary

`read_request` in `rs/p2p/quic_transport/src/request_handler.rs` processes incoming QUIC streams with no limit on the number of HTTP headers in a `pb::HttpRequest`. A single Byzantine subnet peer can craft messages with tens of thousands of headers whose aggregate wire size stays below `MAX_MESSAGE_SIZE_BYTES` (128 MB), causing three simultaneous heap allocations per stream: raw wire bytes, decoded `Vec<HttpHeader>`, and `http::HeaderMap` entries. With `MAX_CONCURRENT_BIDI_STREAMS = 1,000` and no backpressure in the stream acceptor, a single Byzantine peer can drive the victim replica into OOM.

## Finding Description

**Root cause:** No header-count guard exists anywhere in the `read_request` path.

**Step 1 — raw wire buffer:**
`recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)` at [1](#0-0)  buffers the entire stream into a `Vec<u8>` of up to 128 MB. `STREAM_RECEIVE_WINDOW = 4 MB` paces delivery but does not cap total application-layer buffering; `read_to_end` advances the window continuously until the stream closes or the 128 MB cap is hit.

**Step 2 — protobuf decode:**
`pb::HttpRequest::decode(request_bytes.as_slice())` at [2](#0-1)  allocates a `Vec<HttpHeader>` on the heap. Each element holds a `String` key and `Vec<u8>` value — a second O(N × header_size) allocation. Critically, `request_bytes` remains live in scope while `request_proto` is being processed, so both allocations coexist.

**Step 3 — HeaderMap construction:**
The unbounded loop at [3](#0-2)  calls `request_builder.header(key, value)` for every entry, inserting into `http::HeaderMap` (a hash table). This is a third allocation proportional to header count. During the loop, all three allocations — raw bytes, remaining `Vec<HttpHeader>` elements, and the growing `HeaderMap` — coexist on the heap simultaneously.

**No backpressure in stream acceptor:**
`start_stream_acceptor` spawns a new task for every accepted stream unconditionally, [4](#0-3)  with the code itself acknowledging the unbounded buffering risk at [5](#0-4) .

**Configuration constants confirming the attack surface:**
- `MAX_CONCURRENT_BIDI_STREAMS = 1_000` and `STREAM_RECEIVE_WINDOW = 4_000_000` at [6](#0-5) 
- `MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024` at [7](#0-6) 

## Impact Explanation

**Per-stream memory (concrete example):** 50,000 headers × (1 KB key + 1 KB value) = ~100 MB wire size (within 128 MB limit).

| Layer | Peak size |
|---|---|
| `read_to_end` raw buffer | ~100 MB |
| `Vec<HttpHeader>` from decode | ~100 MB |
| `http::HeaderMap` insertions | ~50–100 MB |
| **Total per stream** | **~250–300 MB** |

**Across 1,000 concurrent streams:** 1,000 × ~250 MB = ~250 GB heap pressure → OOM kill of the replica process.

This is a non-volumetric, application/platform-level crash of a replica, matching the **High** impact category: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

- The attacker must be a Byzantine subnet member — its TLS certificate is in the registry and a QUIC connection is already established. This is within IC's explicit Byzantine fault model (up to f nodes).
- No threshold corruption is required; a single Byzantine peer suffices.
- The attack requires only crafting a valid protobuf message with many small headers — trivially constructable.
- No special timing, race condition, or external dependency is needed.
- The attack is repeatable and deterministic. A single Byzantine peer connected to all subnet members could sequentially or simultaneously target multiple replicas.

## Recommendation

1. **Add a header count limit** in `read_request` immediately after decode, before the header loop at [3](#0-2) :
   ```rust
   const MAX_HEADERS_PER_REQUEST: usize = 100;
   if request_proto.headers.len() > MAX_HEADERS_PER_REQUEST {
       return Err(P2PError::from("too many headers".to_string()));
   }
   ```
2. **Add per-header size limits** on key and value byte lengths.
3. **Add a concurrent-stream semaphore** in `start_stream_acceptor` to bound total in-flight memory across all streams from a single peer, rather than accepting up to `MAX_CONCURRENT_BIDI_STREAMS` unconditionally, as acknowledged in the existing comment at [5](#0-4) .

## Proof of Concept

```rust
// Byzantine peer sends 1000 concurrent streams, each with 50_000 headers × 2KB
for _ in 0..1000 {
    let (mut send, _recv) = conn.open_bi().await.unwrap();
    let headers: Vec<pb::HttpHeader> = (0..50_000)
        .map(|_| pb::HttpHeader {
            key: "x".repeat(1024),
            value: vec![0u8; 1024],
        })
        .collect();
    let req = pb::HttpRequest {
        uri: "/".to_string(),
        headers,
        method: pb::HttpMethod::Get as i32,
        body: vec![],
    };
    let bytes = req.encode_to_vec(); // ~100 MB, within 128 MB limit
    send.write_all(&bytes).await.unwrap();
    send.finish().unwrap();
    // Do NOT await response — keep all streams open concurrently
}
// Victim replica allocates ~250 GB → OOM kill
```

A deterministic integration test can be written using `PocketIC` or a local two-node setup: open 1,000 bidirectional QUIC streams from a test Byzantine peer, send the crafted protobuf on each, and observe RSS growth via `/proc/self/status` or `jemalloc` stats before the process is killed.
