Audit Report

## Title
`inspect_message` Guard Bypassed via `#[query]` Annotation, Leaking Non-Disclosed Rule and Incident Metadata - (File: `rs/boundary_node/rate_limits/canister/canister.rs`)

## Summary
The rate-limits canister uses an `#[inspect_message]` hook to restrict ingress access to `get_config` (authorized principal or API boundary node only) and to reject all ingress callers for `get_rule_by_id` and `get_rules_by_incident_id`. Because all three methods are annotated `#[query]`, the IC runtime never invokes `inspect_message` for them; any caller — including anonymous — can reach them as plain query calls. The `RestrictedRead` path inside each method redacts `rule_raw` and `description` for non-disclosed rules but returns their `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` verbatim, leaking structural metadata the canister's design treats as confidential until explicit disclosure.

## Finding Description
**Root cause — `inspect_message` does not fire for query calls.**

The hook at `canister.rs` L34–68 guards three logical paths:
- `get_config` (`REPLICATED_QUERY_METHOD`): accepted only for `authorized_principal` or an API boundary node.
- `add_config` / `disclose_rules`: accepted only for `authorized_principal`.
- Everything else (including `get_rule_by_id`, `get_rules_by_incident_id`): unconditionally trapped. [1](#0-0) [2](#0-1) 

All three read methods are `#[query]`: [3](#0-2) 

The IC protocol invokes `inspect_message` only for **ingress update messages**. Query calls bypass it entirely. Therefore:
- Any caller can invoke `get_config` as a query call; the guard that restricts it to authorized principals and API boundary nodes is never evaluated.
- `get_rule_by_id` and `get_rules_by_incident_id` are reachable as query calls even though `inspect_message` would trap every ingress update caller for them.

**What `RestrictedRead` actually returns.**

`AccessLevelResolver::get_access_level()` returns `AccessLevel::RestrictedRead` for any caller that is neither the authorized principal nor a registered API boundary node: [4](#0-3) 

`ConfigConfidentialityFormatter` nulls out `rule_raw` and `description` for non-disclosed rules but leaves `rule_id` and `incident_id` intact: [5](#0-4) 

`RuleConfidentialityFormatter` does the same, leaving `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` intact: [6](#0-5) 

The unit test for `RestrictedRead` on `get_rule_by_id` explicitly asserts that `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` are returned for a non-disclosed rule: [7](#0-6) 

The `OutputRuleMetadata` wire type confirms all four fields are present in the Candid response: [8](#0-7) 

**Exploit flow.**
1. Anonymous caller issues a query call to `get_config(null)`.
2. `inspect_message` is never invoked; execution proceeds.
3. `AccessLevelResolver` returns `RestrictedRead`.
4. `ConfigConfidentialityFormatter` is applied; response contains every rule's `rule_id` and `incident_id`, including non-disclosed ones.
5. Caller iterates the returned `rule_id` list and calls `get_rule_by_id("<uuid>")` for each, obtaining `added_in_version` and `removed_in_version` for every non-disclosed rule.
6. Caller calls `get_rules_by_incident_id("<incident_uuid>")` to enumerate all rules under a given incident.

## Impact Explanation
The rate-limits canister explicitly models confidentiality: rules have a `disclosed_at` field, responses carry an `is_redacted` flag, and the `inspect_message` hook was written to prevent unauthorized principals from reading `get_config` at all. The bypass allows any anonymous principal to enumerate the full set of active, non-disclosed rate-limit rules — their UUIDs, associated incident UUIDs, and the config versions in which they were added or removed. This constitutes unauthorized disclosure of active security-incident metadata on the boundary node infrastructure, matching the allowed impact: **significant boundary/API infrastructure security impact with concrete protocol harm** (High, $2,000–$10,000).

## Likelihood Explanation
No privilege, key material, or inter-canister coordination is required. The canister is publicly reachable; its Candid interface is published. Any IC identity, including the anonymous principal, can issue a query call. The attack is deterministic, repeatable, and trivially scriptable with `dfx` or any IC agent library.

## Recommendation
1. **Change `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` from `#[query]` to `#[update]`** so that `inspect_message` is actually invoked for every ingress call. This matches the developer intent expressed by `REPLICATED_QUERY_METHOD` and the unconditional trap for the other two methods.
2. **Add an explicit caller check inside each method** (defense-in-depth) using `AccessLevelResolver`, so that inter-canister callers are also subject to the same access control independent of `inspect_message`.
3. If `#[query]` semantics are required for latency, remove the `inspect_message` entries for these methods and enforce access control solely through `AccessLevelResolver`, but ensure `RestrictedRead` returns an empty rule list (or an error) rather than structural metadata for non-disclosed rules.

#[inspect_message]
fn inspect_message() {
    // In order for this hook to succeed, accept_message() must be invoked.
    let caller_id: Principal = ic_cdk::api::caller();
    let called_method = ic_cdk::api::call::method_name();

    let (has_full_access, has_full_read_access) = with_canister_state(|state| {
        let authorized_principal = state.get_authorized_principal();
        (
            Some(caller_id) == authorized_principal,
            state.is_api_boundary_node_principal(&caller_id),
        )
    });

    if called_method == REPLICATED_QUERY_METHOD {
        if has_full_access || has_full_read_access {
            ic_cdk::api::call::accept_message();
        } else {
            ic_cdk::api::trap(
                "message_inspection_failed: method call is prohibited in the current context",
            );
        }
    } else if UPDATE_METHODS.contains(&called_method.as_str()) {
        if has_full_access {
            ic_cdk::api::call::accept_message();
        } else {
            ic_cdk::api::trap("message_inspection_failed: unauthorized caller");
        }
    } else {
        // All others calls are rejected
        ic_cdk::api::trap(
            "message_inspection_failed: method call is prohibited in the current context",
        );
    }
}
