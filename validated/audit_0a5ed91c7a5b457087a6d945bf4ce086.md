Audit Report

## Title
Registry Canister DoS via Unconditional Panic in `extract_chip_id_from_payload` on Oversized `node_signing_pk` — (`rs/registry/canister/src/mutations/node_management/do_add_node.rs`)

## Summary
In `extract_chip_id_from_payload`, when `node_registration_attestation` is `Some(...)`, the code calls `OctetStringRef::new(&payload.node_signing_pk).expect(...)`. The `der` crate's `OctetStringRef::new` returns `Err` for any byte slice exceeding 65535 bytes, and the `.expect()` unconditionally panics. Because `valid_keys_from_payload` only validates the decoded protobuf struct fields (not the raw byte length), a `PublicKey` with a valid 32-byte Ed25519 `key_value` and a large `proof_data` blob passes all crypto checks while its raw encoding exceeds 65535 bytes, triggering the panic and trapping the registry canister.

## Finding Description
The vulnerable `.expect()` is at:

```rust
let expected_custom_data = NodeRegistrationAttestationCustomData {
    node_signing_pk: OctetStringRef::new(&payload.node_signing_pk)
        .expect("node_signing_pk must be valid"),
};
``` [1](#0-0) 

This is reached only when `node_registration_attestation` is `Some(...)`: [2](#0-1) 

The preceding validation at `valid_keys_from_payload` decodes the raw bytes into a `PublicKey` proto and validates only the decoded `key_value` field: [3](#0-2) 

`ValidNodeSigningPublicKey::try_from` checks algorithm ID, `key_value` length (32 bytes), and torsion-freeness — it never inspects `proof_data` or the raw byte length of the protobuf encoding: [4](#0-3) 

A `PublicKey` proto with a valid 32-byte Ed25519 `key_value` and a `proof_data` field padded to ≥65536 bytes passes all crypto validation. Its `encode_to_vec()` output exceeds 65535 bytes. `OctetStringRef::new` then returns `Err` (the `der` crate enforces a `u16::MAX` length limit on `OctetStringRef`), and `.expect()` converts that into an unconditional panic, trapping the canister.

The rate-limit reservation is made before the panic: [5](#0-4) 

Because the trap rolls back all state changes including the reservation, the attacker can resubmit the same malicious payload indefinitely without consuming rate-limit capacity.

## Impact Explanation
The registry canister is an NNS canister. Repeated trapping of the registry canister denies service to all legitimate registry operations — node additions, subnet changes, key updates, etc. — for as long as the attacker continues submitting the malicious payload. This matches the allowed High impact: **Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The attacker must be a registered node operator, a role obtainable through NNS governance without admin privileges. The crafted payload is trivial to construct: take any valid node registration payload, set `proof_data` to a ≥65536-byte blob in the `node_signing_pk` protobuf, and set `node_registration_attestation` to any non-`None` value (the panic occurs before attestation verification). The 65536-byte payload is well within the IC's 2 MB ingress limit. The attack is deterministic, locally reproducible, and indefinitely repeatable due to the rate-limit rollback.

## Recommendation
Replace the `.expect()` with a proper error return:

```rust
let node_signing_pk_ref = OctetStringRef::new(&payload.node_signing_pk)
    .map_err(|e| format!("{LOG_PREFIX}do_add_node: node_signing_pk too large for DER encoding: {e}"))?;
let expected_custom_data = NodeRegistrationAttestationCustomData {
    node_signing_pk: node_signing_pk_ref,
};
```

Additionally, add an explicit size check in `valid_keys_from_payload` to reject `node_signing_pk` raw bytes exceeding a reasonable bound (e.g., 512 bytes) before any further processing.

## Proof of Concept

```rust
use prost::Message;
use ic_protobuf::registry::crypto::v1::PublicKey;

// Start with a valid Ed25519 node signing key proto
let mut pk = valid_node_signing_public_key(); // key_value is 32 bytes, passes ValidNodeSigningPublicKey
// Pad proof_data to force raw encoding > 65535 bytes
pk.proof_data = Some(vec![0u8; 65536]);
let node_signing_pk = pk.encode_to_vec();
assert!(node_signing_pk.len() > 65535);

// valid_keys_from_payload succeeds: only key_value (32 bytes) is validated
// extract_chip_id_from_payload panics: OctetStringRef::new(&node_signing_pk) returns Err

let payload = AddNodePayload {
    node_signing_pk,
    node_registration_attestation: Some(SevAttestationPackage { /* any value */ }),
    // ... other valid fields
};
// registry.do_add_node_(payload, node_operator_id, now) → PANIC → canister trap
// Rate-limit state is rolled back → attack is indefinitely repeatable
```
