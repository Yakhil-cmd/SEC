Audit Report

## Title
Ed25519 Small-Order Public Key Bypasses Ingress Signature Verification via ZIP215 Cofactor Check — (`rs/crypto/standalone-sig-verifier/src/lib.rs`, `packages/ic-ed25519/src/lib.rs`)

## Summary
The ingress signature verification path calls `PublicKey::deserialize_raw` followed by `verify_signature`, neither of which performs a subgroup (torsion-free) check. For any Ed25519 public key whose order divides 8 (e.g., `[0u8; 32]`), the ZIP215 cofactor verification equation `(recomputed_r − R)·8 = identity` is satisfied for **any** message when the forged signature is `(R = identity, S = 0)`, because `8·A = identity` for such keys. An unprivileged attacker can derive the self-authenticating principal for such a key and submit ingress messages that pass all validation as that principal without possessing any private key.

## Finding Description

**Root cause — missing torsion check in the ingress path:**

`verify_basic_sig_by_public_key` in `rs/crypto/standalone-sig-verifier/src/lib.rs` (L21–43) calls `PublicKey::deserialize_raw` then `pk.verify_signature`. `deserialize_raw` (L619–631 of `packages/ic-ed25519/src/lib.rs`) only calls `VerifyingKey::from_bytes`, which checks the point is on the curve but explicitly does **not** check torsion-freeness (the docstring at L614–618 confirms this). The separate `verify_public_key` function at `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs` (L97–103) does call `pk.is_torsion_free()`, but it is never invoked in the ingress path.

**Why `verify_signature` accepts the forgery:**

`verify_signature` (L709–727 of `packages/ic-ed25519/src/lib.rs`) implements the ZIP215 check:

```
(recomputed_r − R) · 8 == identity
```

With `A` of order dividing 8, forged `R = identity`, `S = 0`:
- `recomputed_r = S·B + k·(−A) = k·(−A)`
- `(k·(−A) − identity) · 8 = k · (−8A) − 8·identity = k · identity − identity = identity − identity = identity ✓`

This holds for **any** challenge scalar `k`, i.e., for any message.

**`Signature::from_slice` accepts `(R=identity, S=0)`:**

`from_slice` (L449–471) decompresses `R` via `CompressedEdwardsY::decompress` (succeeds for `[1,0,...,0]`) and parses `S` via `Scalar::from_canonical_bytes` (succeeds for `[0u8;32]`). No check rejects either value.

**The ingress validation chain has no guard:**

`validate_user_id_and_signature` (L842–876 of `rs/validator/src/ingress_validation.rs`) → `validate_user_id` (checks sender = `SHA224(DER(pk)) ‖ 0x02`) → `validate_signature` → `validate_signature_plain` (L705–714) → `verify_basic_sig_by_public_key`. No step in this chain calls `is_torsion_free()` or `verify_public_key`.

The test at `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api/tests.rs` (L342–355) explicitly confirms `[0u8; 32]` is a small-order point (order 8) that passes `deserialize_raw` but fails `verify_public_key`. Node key validation (`rs/crypto/node_key_validation/src/lib.rs` L233–240) correctly applies the torsion check, demonstrating the fix is known and available but not applied in the ingress path.

## Impact Explanation

An attacker can forge valid ingress signatures for the deterministic self-authenticating principal derived from any small-order Ed25519 key (e.g., `SHA224(DER([0u8;32])) ‖ 0x02`). This constitutes unauthorized access to any canister-controlled resources, governance assets, neurons, or funds that have been granted to that specific principal. The principal is publicly computable, so any canister that has granted it permissions (or that an attacker can cause to grant it permissions) is fully compromised. This matches the **High** bounty impact: unauthorized access to identities, wallets, or canister-controlled funds under constrained conditions (the constraint being that the target principal must have been granted access).

## Likelihood Explanation

No privileged access, no key material, and no victim interaction is required. The attack is fully constructible offline. The forged principal is deterministic and publicly known, which limits the attack surface to principals that have been explicitly granted permissions — but the authentication bypass itself is unconditional and trivially reproducible.

## Recommendation

In `verify_basic_sig_by_public_key` (`rs/crypto/standalone-sig-verifier/src/lib.rs`, after `deserialize_raw` succeeds), add the same torsion check already present in `verify_public_key` and node key validation:

```rust
if !pk.is_torsion_free() {
    return Err(CryptoError::MalformedPublicKey {
        algorithm: AlgorithmId::Ed25519,
        key_bytes: Some(pk_bytes.to_vec()),
        internal_error: "Public key has torsion component".to_string(),
    });
}
```

Alternatively, enforce this check inside `PublicKey::deserialize_raw` itself (with a documented behavior change), so all callers are protected by default.

## Proof of Concept

```rust
// 1. Small-order public key (order 8), confirmed by existing test
let pk_raw = [0u8; 32];

// 2. DER-encode and derive self-authenticating principal
let pk_der = ic_ed25519::PublicKey::deserialize_raw(&pk_raw)
    .unwrap()
    .serialize_rfc8410_der();
let principal = PrincipalId::new_self_authenticating(&pk_der);

// 3. Forged signature: R = identity encoding [1,0,...,0], S = 0
let mut forged_sig = [0u8; 64];
forged_sig[0] = 1; // canonical encoding of the identity point

// 4. Verify directly — must return Ok(())
let pk = ic_ed25519::PublicKey::deserialize_raw(&pk_raw).unwrap();
assert!(pk.verify_signature(b"any message at all", &forged_sig).is_ok());

// 5. verify_basic_sig_by_public_key(Ed25519, any_msg, &forged_sig, &pk_raw) → Ok(())
// 6. Submit ingress with sender=principal, signer_pubkey=pk_der, signature=forged_sig → accepted
```
