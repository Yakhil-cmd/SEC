### Title
Unsanitized `predecessor_account_id` in `ft_on_transfer` Enables Minting of Unbacked ERC-20 Tokens — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` entrypoint in Aurora Engine accepts any NEAR account as the calling token contract (`predecessor_account_id`) without validating it against a whitelist of registered NEP-141 tokens. Because `deploy_erc20_token` is permissionless, an attacker can register a malicious NEP-141 contract, then call `ft_on_transfer` directly — without ever transferring real tokens — causing Aurora to mint ERC-20 tokens backed by nothing.

---

### Finding Description

In `engine/src/contract_methods/connector.rs`, `ft_on_transfer` dispatches on `predecessor_account_id`:

```rust
// engine/src/contract_methods/connector.rs  lines 81-90
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,   // ← user-controlled, unvalidated
        &args,
        &current_account_id,
        handler,
    )
};
```

The only guard present is `require_running` (line 68). There is no whitelist check confirming that `predecessor_account_id` is a legitimate, registered NEP-141 token that has actually transferred tokens to Aurora.

`receive_erc20_tokens` uses `predecessor_account_id` to look up the ERC-20 address in the NEP-141↔ERC-20 registry. If a mapping exists, it mints ERC-20 tokens for the `amount` field supplied in `args` — a value the caller controls entirely.

The registry is populated by `deploy_erc20_token`, which carries no owner or access-control check:

```rust
// engine/src/contract_methods/connector.rs  lines 112-158
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // only liveness check
        ...
    })
}
```

**Attack path:**

1. Attacker deploys a malicious NEAR contract at `attacker-token.near` that exposes a minimal NEP-141 interface.
2. Attacker calls `deploy_erc20_token` on Aurora with `attacker-token.near` as the NEP-141 argument. Aurora deploys an ERC-20 mirror and writes the `attacker-token.near ↔ erc20_addr` mapping into storage.
3. Attacker's contract calls `ft_on_transfer` on Aurora directly (no `ft_transfer_call` on any real token contract), supplying `amount = N` and `sender = attacker_evm_address`.
4. `predecessor_account_id` is `attacker-token.near`; the registry lookup succeeds; `receive_erc20_tokens` mints `N` ERC-20 tokens to the attacker's EVM address.
5. No NEP-141 tokens were ever transferred to Aurora. The minted ERC-20 tokens are entirely unbacked.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The attacker holds ERC-20 tokens that are supposed to be 1:1 backed by NEP-141 tokens held in Aurora's custody. With unbacked tokens the attacker can:

- Swap them for legitimate ERC-20 tokens in any Aurora DEX, draining liquidity providers.
- Use them as collateral in lending protocols, borrowing real assets.
- Repeat the mint arbitrarily (no per-call cost beyond NEAR gas), scaling the theft.

Every legitimate holder of the same ERC-20 token suffers insolvency: the total ERC-20 supply exceeds the NEP-141 balance Aurora actually holds, so not all holders can exit.

---

### Likelihood Explanation

**High.**

- `ft_on_transfer` is a public NEAR method; any account can call it.
- `deploy_evm20_token` is permissionless; no admin approval is required to register a NEP-141 account.
- The attacker needs only standard NEAR gas and the ability to deploy a trivial contract.
- No leaked keys, governance capture, or oracle manipulation is required.

---

### Recommendation

1. **Whitelist validation in `ft_on_transfer`**: Before calling `receive_erc20_tokens`, confirm that `predecessor_account_id` is already present in the NEP-141↔ERC-20 registry *and* that the call originates from the canonical token contract (not an arbitrary account).

2. **Access-control on `deploy_erc20_token`**: Restrict registration of new NEP-141↔ERC-20 pairs to the contract owner or a governance-approved list, preventing attackers from self-registering malicious tokens.

3. **Checks-effects pattern**: Record the expected NEP-141 balance increase before minting ERC-20 tokens, and verify it in a callback, analogous to the re-entrancy guard recommended in the original report.

---

### Proof of Concept

```
# 1. Deploy minimal NEP-141 stub at attacker-token.near
#    (implements ft_on_transfer, ft_metadata; no real token supply)

# 2. Register the stub as an ERC-20 on Aurora (permissionless)
near call aurora.near deploy_erc20_token \
  '{"nep141": "attacker-token.near"}' \
  --accountId attacker.near

# 3. From attacker-token.near, call ft_on_transfer directly
#    predecessor_account_id will be "attacker-token.near"
near call aurora.near ft_on_transfer \
  '{"sender_id":"attacker.near","amount":"1000000000000000000","msg":"<attacker_evm_hex>"}' \
  --accountId attacker-token.near   # ← predecessor = registered NEP-141

# Result: Aurora mints 1 000 000 000 000 000 000 ERC-20 tokens
# to the attacker's EVM address with zero NEP-141 backing.
```

The root cause — `predecessor_account_id` flowing into `receive_erc20_tokens` without whitelist validation — is directly analogous to the unsanitized `farmToken` address in the original report, and the impact (unbacked token minting → fund theft / insolvency) falls squarely within the allowed critical impact scope.
