Audit Report

## Title
Byzantine `content_size` Inflation in `ResponsesTooLarge` Proof Allows False Impossibility Claim — (`rs/https_outcalls/consensus/src/payload_builder.rs`)

## Summary
The `ResponsesTooLarge` validation branch in `validate_canister_http_payload_impl` uses the `content_size` field from `CanisterHttpResponseShare` objects directly in its size-sum calculation without bounding it against `MAX_CANISTER_HTTP_RESPONSE_BYTES`. A Byzantine committee member can sign a receipt with an inflated `content_size` (e.g., 3 MiB), and a colluding block proposer can include only that single share while omitting all honest shares. For canister-configured `min_responses = total_requests` (M = C), this yields `min_known_ok_needed = 1`, and the inflated sum exceeds `MAX_CANISTER_HTTP_PAYLOAD_SIZE`, causing the validator to accept a forged `ResponsesTooLarge` proof. The finalized error permanently blocks the canister's HTTP callback and burns its cycles.

## Finding Description
In `payload_builder.rs` at L786–818, the `ResponsesTooLarge` branch computes:

```rust
let num_unseen = flex_committee.len().saturating_sub(all_seen_shares.len());
let min_known_ok_needed = min_responses.saturating_sub(num_unseen);
```

It then maps each non-reject share to an estimated byte size using `share.content.content_size() as usize` directly:

```rust
FlexibleCanisterHttpResponseWithProof::count_bytes_from_parts(
    &context.request.sender,
    share.content.content_size() as usize,  // ← unbounded
    share,
)
```

The sum of the `min_known_ok_needed` smallest sizes is compared against `MAX_CANISTER_HTTP_PAYLOAD_SIZE` (2 MiB). `MAX_CANISTER_HTTP_RESPONSE_BYTES` is **never referenced** in `payload_builder.rs` (confirmed by grep). The `validate_response_share` helper at `utils.rs` L208–251 checks committee membership, callback ID, duplicate signers, and registry version — but not `content_size` bounds.

The pool manager does validate `content_size` against the actual response at `pool_manager.rs` L544–549, but a Byzantine block proposer can embed shares directly into a block, bypassing the pool manager entirely. A Byzantine node controls its own signing key and can sign any `CanisterHttpResponseReceipt` with any `content_size` value; the payload validator only verifies the cryptographic signature, not the semantic validity of `content_size`.

**Concrete attack (C = 13, M = 13, f = 1):**

| Variable | Value |
|---|---|
| `all_seen_shares.len()` | 1 (Byzantine share only) |
| `num_unseen` | 12 |
| `min_known_ok_needed` | `13 − 12 = 1` |
| Byzantine `content_size` | 3 MiB (signed, valid sig) |
| `smallest_sum` | 3 MiB > 2 MiB = MAX |
| Validator decision | **ACCEPTED** (forged proof) |

The 12 honest nodes' small responses are omitted by the proposer and treated as "unseen." The validator cannot distinguish this from a legitimate scenario where 12 nodes have not yet responded, and it accepts the proof because `min_known_ok_needed = 1` is satisfied by the single inflated Byzantine share.

The existing test `flexible_error_responses_too_large_invalid_when_committee_members_omitted` at `tests.rs` L4006–4043 uses `min_responses = 2` with `committee = 4`, yielding `min_known_ok_needed = 0` when 2 shares are omitted — it never tests a single inflated Byzantine share against an M = C configuration and does not cover this case.

## Impact Explanation
Once the forged `ResponsesTooLarge` block is finalized, the execution environment delivers a permanent error to the canister's HTTP callback. The callback is never retried, the canister's cycles are burned, and any downstream logic depending on the HTTP response is permanently blocked. This is a concrete, irreversible application-level DoS against canisters using high-M flexible HTTP outcall configurations, matching the allowed impact: **High — Application/platform-level DoS, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The attack requires: (1) one Byzantine node that is a member of the flexible committee for the targeted callback, and (2) that Byzantine node colluding with the block proposer for that round (or being the proposer itself). With C = 13 and fault threshold = 4, a single Byzantine node is well within the protocol's fault tolerance. The canister must use `min_responses = total_requests` or another high-M configuration, which is a valid and documented option per `canister_http.rs` L608–636. The Byzantine share is cryptographically indistinguishable from a legitimate share to the validator. No external infrastructure compromise, governance majority, or threshold key access is required.

## Recommendation
In the `ResponsesTooLarge` validation branch, add an explicit upper-bound check on each share's `content_size` before using it in the size calculation:

```rust
let raw_content_size = share.content.content_size() as usize;
if raw_content_size > MAX_CANISTER_HTTP_RESPONSE_BYTES as usize {
    return invalid_artifact(
        InvalidCanisterHttpPayloadReason::FlexibleContentSizeExceedsMax {
            callback_id,
            signer: share.signature.signer,
            content_size: raw_content_size,
        },
    );
}
```

This mirrors the bound enforced by the pool manager on incoming artifacts and closes the gap between pool-level and payload-level validation.

## Proof of Concept
```rust
// Committee C=4, M=4, fault_threshold=1, f=1 Byzantine node
// Byzantine node signs share with content_size = 3 MiB (> MAX = 2 MiB)
let huge: u32 = (MAX_CANISTER_HTTP_PAYLOAD_SIZE as u32) + 1_000_000;
let byzantine_share = metadata_share_with_content_size(callback_id.get(), 0, huge);

// Proposer omits the 3 honest shares; all_seen_shares has only 1 entry
let payload = CanisterHttpPayload {
    flexible_errors: vec![FlexibleCanisterHttpError::ResponsesTooLarge {
        callback_id,
        all_seen_shares: vec![byzantine_share],
        total_requests: 4,   // matches flex_committee.len()
        min_responses: 4,    // M = C
    }],
    ..Default::default()
};

// num_unseen = 4 - 1 = 3
// min_known_ok_needed = 4 - 3 = 1
// ok_entry_sizes = [3 MiB + overhead]
// smallest_sum = 3 MiB > MAX → validator accepts the forged proof
let result = pb.validate_payload(Height::new(1), &ctx, &payload_bytes, &[]);
assert_matches!(result, Ok(())); // false proof accepted
```

This test can be added to `rs/https_outcalls/consensus/src/payload_builder/tests.rs` using the existing `setup_test_with_flexible_context` and `metadata_share_with_content_size` helpers, with `min_responses = 4` and `total_requests = 4` (M = C = 4).
