Audit Report

## Title
Unauthenticated Third Party Can Commit Victim's ICP to SNS Swap by Supplying Public Confirmation Text on Their Behalf - (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update method resolves the buyer principal from the request body rather than from the authenticated caller, with no authorization check enforcing that the caller equals the specified buyer. Because the `confirmation_text` consent gate is validated only by comparing it against the publicly readable SNS init payload, any unprivileged ingress sender can call `refresh_buyer_tokens` with an arbitrary victim's principal and the known public confirmation text, committing the victim's pre-deposited ICP to the swap without the victim's explicit consent. If the swap commits, the victim's ICP is swept to SNS governance and they receive SNS tokens they never agreed to purchase.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at L130–134, the buyer principal is resolved from `arg.buyer` with no check that the caller matches:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // arbitrary principal, no auth check
};
```

The `RefreshBuyerTokensRequest` proto at `rs/sns/swap/src/gen/ic_sns_swap.pb.v1.rs` L1117–1126 documents the `buyer` field as optional ("If not specified, the caller is used") and the `confirmation_text` field as the participation consent gate ("a participant should send the confirmation text"), but imposes no restriction on who may supply a non-empty `buyer`.

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` L1149–1150), the only consent check is:

```rust
self.validate_confirmation_text(confirmation_text)?;
```

`validate_confirmation_text` (L363–384) simply compares the supplied string against `self.init_or_panic().confirmation_text` — the public SNS init payload. There is no check that the entity supplying the text is the buyer. Since the init payload is publicly readable by anyone, the confirmation text is not a secret.

After this check passes, the function reads the ICP balance from the victim's subaccount (L1152–1163) and records it as committed participation (L1285–1291). Once the victim is inserted into `self.buyers`, `error_refund_icp` (`rs/sns/swap/src/swap.rs` L1950–1959) blocks any refund while the ICP is in escrow:

```rust
if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
    if let Some(transfer) = &buyer_state.icp
        && transfer.transfer_success_timestamp_seconds == 0
    {
        return ErrorRefundIcpResponse::new_precondition_error(format!(
            "ICP cannot be refunded as principal {} has {} ICP (e8s) in escrow", ...
        ));
    }
```

If the swap commits, `sweep_icp` transfers the victim's ICP to SNS governance and the victim receives SNS tokens they never consented to.

## Impact Explanation
This is a **High** severity finding. An unprivileged attacker can force any user who has pre-deposited ICP into the swap canister's subaccount to become a committed swap participant, bypassing the `confirmation_text` consent mechanism entirely. If the swap commits, the victim's ICP is permanently transferred to SNS governance in exchange for SNS tokens the victim never agreed to receive. This constitutes unauthorized disposition of user funds through a broken consent/authentication assumption in a core SNS canister, matching: *"Significant SNS security impact with concrete user or protocol harm"* and *"Unauthorized access to … canister-controlled funds."*

## Likelihood Explanation
Medium-to-High. The precondition — that the victim has already transferred ICP to their swap subaccount — is a normal, expected step in the SNS participation flow, so many users will be in this state during any open swap. The attacker requires only the victim's principal (publicly observable on-chain) and the confirmation text (public in the SNS init payload). No privileged access, key material, or special network position is required. Any unprivileged ingress sender can execute this against any number of victims during the open swap window.

## Recommendation
**Short term:** In `refresh_buyer_tokens` (`rs/sns/swap/canister/canister.rs` L130–134), enforce that when a non-empty `buyer` is supplied, the caller must equal the resolved buyer principal:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() {
        panic!("Caller is not authorized to refresh tokens on behalf of another buyer.");
    }
    specified
};
```

**Long term:** Remove the `buyer` field entirely and always derive the buyer from the authenticated caller. If third-party notification is a required use case (e.g., for automated bots), introduce an explicit allowlist of authorized notifier principals in the SNS init payload rather than permitting any caller to act on behalf of any buyer.

## Proof of Concept
1. Alice transfers 10 ICP to `swap_canister_subaccount(Alice)` on the ICP ledger during an open SNS swap that has `confirmation_text = "I agree to the terms"`.
2. Alice reads the terms and decides not to participate. She does **not** call `refresh_buyer_tokens`.
3. Eve (any unprivileged principal) submits an ingress call to `refresh_buyer_tokens` on the swap canister:
   ```
   RefreshBuyerTokensRequest {
       buyer: Alice.to_string(),
       confirmation_text: Some("I agree to the terms".to_string()),
   }
   ```
4. The canister resolves `p = Alice` (no auth check), calls `validate_confirmation_text` (passes, text matches public init payload), reads Alice's subaccount balance (10 ICP), and records Alice in `self.buyers` with 10 ICP committed.
5. Alice's ICP is now in escrow. `error_refund_icp` returns a precondition error ("ICP cannot be refunded … has ICP in escrow"). If the swap commits, `sweep_icp` transfers Alice's ICP to SNS governance and Alice receives SNS tokens she never consented to purchase.

A deterministic unit test can reproduce this by calling `swap.refresh_buyer_token_e8s(alice_principal, Some("I agree to the terms".to_string()), swap_canister_id, &mock_ledger)` from a different caller principal and asserting that Alice appears in `swap.buyers` with the committed amount.
