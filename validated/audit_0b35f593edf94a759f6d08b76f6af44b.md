Audit Report

## Title
SNS Token Supply Deflation via Unrestricted Burn Inflates Treasury Valuation, Restricting Governance Treasury Transfers - (File: rs/sns/governance/token_valuation/src/lib.rs)

## Summary
`IcpsPerSnsTokenClient::fetch_icps_per_sns_token()` computes the current SNS token price as `initial_icps_per_sns_token / (current_supply / initial_supply)`. Any unprivileged SNS token holder can burn their own tokens via a standard ICRC-1 transfer to the minting account, deflating `current_supply` and inflating the computed price without bound. Because there is no upper clamp on `icps_per_token`, this inflated price can push the treasury's XDR valuation into a more restrictive 7-day transfer regime, blocking or throttling `TransferSnsTreasuryFunds` proposals at submission time.

## Finding Description

**Root cause — unbounded price inflation via supply deflation:**

`fetch_icps_per_sns_token` in `rs/sns/governance/token_valuation/src/lib.rs` (L397–414) computes:

```
total_inflation = current_supply_e8s / initial_supply_e8s
result          = initial_icps_per_sns_token / total_inflation
```

When `current_supply < initial_supply` (deflation via burns), `total_inflation < 1`, so `result > initial_icps_per_sns_token`. There is no cap on this value.

**No upper clamp on `icps_per_token`:**

`clamp_xdrs_per_icp` in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` (L137–140) only enforces `MIN_XDRS_PER_ICP = 1`. The code explicitly documents the absence of a `MAX_XDRS_PER_ICP` (L60–64). Critically, this clamp applies only to `xdrs_per_icp` (the ICP/XDR exchange rate from CMC), not to `icps_per_token` (the SNS token price), which is the value inflated by the burn. There is no clamp on `icps_per_token` anywhere in the pipeline.

**Regime boundaries:**

`from_valuation_xdr` in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` (L116–135) classifies the treasury:
- ≤ 100,000 XDR → `NoLimit`
- ≤ 1,200,000 XDR → `Fraction(0.25)` (25% per 7 days)
- \> 1,200,000 XDR → `Xdr(300_000)` (fixed 300,000 XDR cap per 7 days)

**Unrestricted burn path:**

The ICRC-1 ledger (`rs/ledger_suite/icrc1/ledger/src/main.rs`, L606–635) allows any account holder to burn by transferring to the minting account, with only a `min_burn_amount = min(transfer_fee, balance)` check. No governance approval is required.

**Attack path (submission-time DoS):**

1. Treasury is in "small" regime (valuation < 100,000 XDR, `NoLimit`).
2. Attacker burns a large fraction of circulating SNS token supply.
3. `current_supply` drops, `total_inflation` drops below 1, `icps_per_token` inflates proportionally.
4. When a `TransferSnsTreasuryFunds` proposal is submitted, `treasury_valuation_if_proposal_amount_is_small_enough_or_err` (`rs/sns/governance/src/proposal.rs`, L770–816) fetches a fresh valuation using the now-inflated price.
5. The inflated XDR valuation places the treasury in "medium" or "large" regime.
6. The proposal amount, valid under the prior "small" regime, now exceeds the new limit and is rejected at submission time.

**Clarification on execution-time behavior:**

The execution-time check (`transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, `rs/sns/governance/src/proposal.rs`, L2600–2658) uses the **stored** valuation captured at submission time (retrieved via `get_action_auxiliary`, `rs/sns/governance/src/governance.rs`, L2204–2208). Burns occurring *after* proposal submission do not affect the execution-time check. The primary attack surface is therefore at **proposal submission time**.

## Impact Explanation

An unprivileged SNS token holder can unilaterally impose a 25%-per-7-day cap (or a fixed 300,000 XDR cap) on `TransferSnsTreasuryFunds` proposals for an SNS whose treasury was previously uncapped. This constitutes a targeted governance denial-of-service: legitimate treasury management proposals are blocked at submission, preventing the SNS community from executing authorized fund transfers. The impact is concrete and directly harms SNS governance integrity. This matches: **High — Significant SNS security impact with concrete user or protocol harm.**

## Likelihood Explanation

Likelihood is **low-to-medium**. The attacker must own and burn a substantial fraction of the circulating SNS token supply, which is economically self-destructive. The attack is most feasible for SNS instances with a small circulating supply and a treasury valuation near the small/medium boundary (100,000 XDR). The effect is partially self-correcting as governance mints new tokens via voting rewards. The attack is repeatable as long as the attacker retains sufficient tokens to burn.

## Recommendation

1. **Clamp `icps_per_token` symmetrically:** Apply a `MAX_ICPS_PER_TOKEN` cap in `fetch_icps_per_sns_token` or in `ProposalsAmountTotalUpperBound::in_tokens`, analogous to `MIN_XDRS_PER_ICP`, to bound the effective price used in treasury limit calculations.
2. **Floor `total_inflation`:** Cap `total_inflation` from below at a minimum value (e.g., `0.5`) in `fetch_icps_per_sns_token` so that deflation cannot inflate the computed price beyond 2× the initial price.
3. **Time-weighted average supply:** Use a time-weighted average of `icrc1_total_supply` rather than the instantaneous value to resist short-term supply manipulation.

## Proof of Concept

**Setup:**
- `initial_supply = 10,000,000 e8s` (100 tokens)
- `initial_sns_tokens_per_icp = 10` → `initial_icps_per_sns_token = 0.1`
- `xdrs_per_icp = 10`
- Treasury holds 50,000 SNS tokens → initial valuation = `50,000 × 0.1 × 10 = 50,000 XDR` (small, no limit)

**Attack:**
- Attacker burns 95% of circulating supply (9,500,000 e8s), leaving `current_supply = 500,000 e8s`
- `total_inflation = 500,000 / 10,000,000 = 0.05`
- `current_price = 0.1 / 0.05 = 2.0 ICP per token`
- Treasury valuation = `50,000 × 2.0 × 10 = 1,000,000 XDR` → **medium regime** (25% cap)

**Result:** Any `TransferSnsTreasuryFunds` proposal attempting to transfer more than 12,500 SNS tokens (25% of 50,000) is now rejected at submission via `treasury_valuation_if_proposal_amount_is_small_enough_or_err`. Previously, the full treasury was transferable in a single proposal.

**Reproducible test plan:** Write a unit test in `rs/sns/governance/proposals_amount_total_limit/src/tests.rs` that constructs a `Valuation` with `icps_per_token` set to `initial_icps_per_sns_token / 0.05` (simulating 95% burn), verifies `from_valuation_xdr` returns `Fraction(0.25)` instead of `NoLimit`, and confirms `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` returns 25% of the treasury balance rather than the full balance.
