### Title
Unrestricted `refund_on_error` Entry Point Allows Arbitrary ERC-20 Minting and ETH Theft - (File: engine/src/engine.rs, engine/src/lib.rs)

---

### Summary

The `refund_on_error` function is exposed as a public NEAR contract entry point with no caller restriction. It is designed to be invoked only as an internal callback from `exit_to_near_precompile_callback`, but any unprivileged NEAR account can call it directly with attacker-controlled arguments, minting arbitrary amounts of any bridged ERC-20 token or draining ETH from the exit precompile address.

---

### Finding Description

`refund_on_error` in `engine/src/engine.rs` performs two privileged EVM operations depending on its arguments:

1. **ERC-20 branch** (`erc20_address = Some(...)`): It constructs a `mint(recipient, amount)` call to the target ERC-20 contract, executing it with the engine's own EVM address as the caller — which is the admin/minter of every bridged ERC-20 deployed on Aurora.

2. **ETH branch** (`erc20_address = None`): It transfers `amount` of ETH from `exit_to_near::ADDRESS` to `recipient_address`. [1](#0-0) 

The function contains **no access control check** — no `assert_private_call()`, no `require_owner_only()`, no predecessor validation of any kind.

The function is registered as a callable NEAR contract method. The workspace operation registry confirms the method name `"refund_on_error"` is a live contract entry point: [2](#0-1) 

The workspace contract wrapper further confirms it is callable by any account with arbitrary arguments: [3](#0-2) 

The intended call path is exclusively through `exit_to_near_precompile_callback`, which is itself protected by `assert_private_call()`: [4](#0-3) 

The protection on the callback wrapper is bypassed entirely when `refund_on_error` is called directly as a top-level NEAR transaction.

For comparison, all other sensitive callback-only functions in the codebase correctly enforce `assert_private_call()`: [5](#0-4) [6](#0-5) [7](#0-6) 

`refund_on_error` is the sole state-mutating entry point missing this guard.

---

### Impact Explanation

**Critical — Direct theft of any user funds.**

In the ERC-20 branch, the engine executes `mint(attacker_address, attacker_amount)` on any registered bridged ERC-20 token, acting as the EVM-level admin: [8](#0-7) 

Every ERC-20 deployed by Aurora uses `onlyAdmin` for minting, and the admin is the engine's own EVM address (`current_address(&current_account_id)`). An attacker can mint an unbounded quantity of any bridged token (e.g., USDC, USDT, wBTC mirrors) to their own address, then call `withdrawToNear` on the ERC-20 to redeem the minted tokens for the real underlying NEP-141 assets held in the bridge, draining the bridge reserve. [9](#0-8) 

In the ETH branch, any ETH balance held at `exit_to_near::ADDRESS` (accumulated during in-flight exits) can be redirected to an arbitrary address.

---

### Likelihood Explanation

**High.** The entry point is publicly callable by any NEAR account with no preconditions (no deposit, no special key, no whitelist). The arguments are fully attacker-controlled borsh-encoded inputs. The attack requires a single NEAR transaction. Any registered bridged ERC-20 is a valid target. The attacker needs only to know the ERC-20 address of a bridged token (publicly discoverable via `get_erc20_from_nep141`) and their own EVM address.

---

### Recommendation

Add `assert_private_call()` (or equivalent `require_owner_only`) to the `refund_on_error` NEAR entry point, consistent with every other callback-only function in the codebase:

```rust
pub extern "C" fn refund_on_error() {
    let io = Runtime;
    let env = Runtime;
    // ADD THIS:
    env.assert_private_call().sdk_unwrap();
    // ... rest of handler
}
```

Alternatively, remove the public entry point entirely and only invoke `engine::refund_on_error` from within `exit_to_near_precompile_callback`, which is already private-call-gated.

---

### Proof of Concept

1. Attacker identifies a registered bridged ERC-20 (e.g., USDC mirror) by calling `get_erc20_from_nep141("usdc.token.near")` → returns `erc20_address`.
2. Attacker calls the Aurora Engine NEAR contract method `refund_on_error` with borsh-encoded args:
   - `recipient_address` = attacker's EVM address
   - `erc20_address` = `Some(usdc_erc20_address)`
   - `amount` = `[0u8; 31] ++ [desired_amount_byte]` (e.g., 1,000,000 USDC units)
3. The engine executes `mint(attacker_evm_address, 1_000_000)` on the USDC ERC-20 as admin — no prior deposit required.
4. Attacker calls `withdrawToNear(attacker_near_account, 1_000_000)` on the ERC-20.
5. The exit precompile triggers `ft_transfer` of 1,000,000 USDC NEP-141 tokens from Aurora's bridge reserve to the attacker's NEAR account.
6. Attacker repeats until the bridge reserve is drained. [10](#0-9) [11](#0-10)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
