All four code claims are confirmed by the actual source. Let me verify the `RequestContext` struct definition to ensure `request_type` is indeed a field that exists but is excluded from `Hash`/`Eq`.

The code confirms:

1. `RequestContext.request_type` exists as a field ( [1](#0-0) ) but is absent from both `Hash` ( [2](#0-1) ) and `PartialEq` ( [3](#0-2) ).

2. `RequestType::is_query()` returns `true` for `QueryV2`, `QueryV3`, and `QuerySubnetV3`. ( [4](#0-3) )

3. `BypasserIC::bypass` only bypasses when `!is_query()`, so both V2 and V3 enter the cache. ( [5](#0-4) )

4. `KeyExtractorContext::extract` uses `Arc<RequestContext>` as the key, whose `Hash`/`Eq` omit `request_type`. ( [6](#0-5) )

The bug is real. However, the severity must be assessed carefully. The IC security model does **not** trust the boundary node for certified-query integrity — the guarantee comes from the replica's node signature embedded in the QueryV3 CBOR response. A correctly implemented QueryV3 client (e.g., the `ic-agent` library) validates the certificate in the response body; if the certificate field is absent (as in a cached QueryV2 response), the client rejects the response with a verification error. This means the primary concrete outcome is a **functional failure** (client error), not a silent security bypass, unless the client is itself buggy in skipping certificate validation.

The "silently stripping certification" scenario requires a client that treats a missing certificate as acceptable — which is a client-side defect, not proven by the boundary node code alone. The report does not demonstrate that any in-scope IC client library actually accepts a missing certificate without error.

The impact therefore fits at most the Medium class: "forged or stale certified response accepted only under constrained conditions." The conditions here are: caching enabled, a prior QueryV2 request with identical parameters, **and** a client that does not enforce certificate presence.

Audit Report

## Title
Cross-Version Cache Key Collision Between QueryV2 and QueryV3 — (`rs/boundary_node/ic_boundary/src/routes.rs`, `rs/boundary_node/ic_boundary/src/http/middleware/cache.rs`)

## Summary
`RequestContext`'s `Hash` and `PartialEq` implementations omit the `request_type` field, causing QueryV2 and QueryV3 requests with otherwise identical parameters to share the same cache entry. Because `RequestType::is_query()` returns `true` for both variants and `BypasserIC` only bypasses non-query types, both versions enter the same cache namespace. A QueryV2 response (no node certificate) can be stored and subsequently served to a QueryV3 client that expects a certified response, or vice versa.

## Finding Description
In `routes.rs` lines 94–125, `Hash` and `PartialEq` for `RequestContext` cover `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg`/`http_request` — but not `request_type`. In `http/mod.rs` line 62–64, `is_query()` matches `QueryV2 | QueryV3 | QuerySubnetV3`. In `cache.rs` line 74, `BypasserIC::bypass` returns `None` (i.e., do not bypass) for all query types. `KeyExtractorContext::extract` returns `Arc<RequestContext>` as the cache key, so the key hash and equality are governed by the incomplete `Hash`/`Eq` impls. Two `RequestContext` values differing only in `request_type` (V2 vs V3) are treated as identical keys. A QueryV2 request populates the cache; a subsequent QueryV3 request with the same `(canister_id, sender, method_name, ingress_expiry, arg)` receives the cached QueryV2 CBOR response, which lacks the node certificate that QueryV3 clients expect.

## Impact Explanation
Medium ($200–$2,000). The boundary node serves a structurally incorrect response format across API versions. The most security-relevant direction is QueryV2 populating the cache and a QueryV3 client receiving the uncertified response. A QueryV3 client that enforces certificate presence will reject the response (functional failure). A QueryV3 client that does not enforce certificate presence will silently accept an uncertified result as if it were certified, breaking the certified-response integrity guarantee QueryV3 was designed to provide. This matches the allowed impact class: "forged or stale certified response accepted only under constrained conditions."

## Likelihood Explanation
The attacker is unprivileged. The only preconditions are: (1) the boundary node has query caching enabled (`cache_size` configured), and (2) the attacker can issue both a QueryV2 and a QueryV3 request with identical parameters. No keys, admin access, or network-level position is required. The collision is deterministic and repeatable. The constraint is that actual security impact (vs. a client error) depends on whether the QueryV3 client enforces certificate validation.

## Recommendation
Add `request_type` to both `Hash` and `PartialEq` for `RequestContext` in `rs/boundary_node/ic_boundary/src/routes.rs`:

```rust
// In Hash impl:
self.request_type.hash(state);

// In PartialEq impl:
&& self.request_type == other.request_type
```

Alternatively, add a version discriminant in `KeyExtractorContext::extract` in `cache.rs`, or split the cache into per-version namespaces.
