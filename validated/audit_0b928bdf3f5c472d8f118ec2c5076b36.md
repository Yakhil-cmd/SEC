### Title
Incorrect Modular Inverse Computation in secp256k1 Field Arithmetic Corrupts ecrecover Output - (File: `crypto/src/secp256k1/field/mod_inv64.rs`)

---

### Summary

The `update_de62` function in `crypto/src/secp256k1/field/mod_inv64.rs` contains multiple copy-paste errors where the Bezout coefficient `e` accumulator (`ce`) incorrectly uses the correction factor `md` (intended for `d`) instead of `me` for limbs 1, 2, and 3 of the modulus. An additional condition checks `modulus.0[2]` instead of `modulus.0[3]` for the third correction block. This corrupts the secp256k1 field element modular inverse, which is called in `Jacobian::to_affine()` during ecrecover, producing wrong public key recovery results on 64-bit forward execution.

---

### Finding Description

In `crypto/src/secp256k1/field/mod_inv64.rs`, the `update_de62` function (adapted from bitcoin-core's `modinv64_impl.h`) computes the update `[d_out, e_out] = 2^-62 * T * [d, e] mod modulus`. The algorithm requires separate modulus correction factors `md` (for `cd`) and `me` (for `ce`) to ensure each accumulator is divisible by `2^62` before the right-shift.

The correct pattern is visible at lines 197–198 and 248–249:
```rust
cd += mod_info.modulus.0[0] as i128 * md as i128;  // correct
ce += mod_info.modulus.0[0] as i128 * me as i128;  // correct
// ...
cd += ... + mod_info.modulus.0[4] as i128 * md as i128;  // correct
ce += ... + mod_info.modulus.0[4] as i128 * me as i128;  // correct
```

But for limbs 1, 2, and 3, `ce` incorrectly uses `md`: [1](#0-0) 

```rust
if mod_info.modulus.0[1] != 0 {
    cd += mod_info.modulus.0[1] as i128 * md as i128;
    ce += mod_info.modulus.0[1] as i128 * md as i128;  // BUG: should be `me`
}
``` [2](#0-1) 

```rust
if mod_info.modulus.0[2] != 0 {
    cd += mod_info.modulus.0[2] as i128 * md as i128;
    ce += mod_info.modulus.0[2] as i128 * md as i128;  // BUG: should be `me`
}
``` [3](#0-2) 

```rust
if mod_info.modulus.0[2] != 0 {  // BUG: should check modulus.0[3]
    cd += mod_info.modulus.0[3] as i128 * md as i128;
    ce += mod_info.modulus.0[3] as i128 * md as i128;  // BUG: should be `me`
}
```

The secp256k1 field prime `p = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEFFFFFC2F` has non-zero values at all five 62-bit limbs (limbs 1–3 are all `0x3FFFFFFFFFFFFFFF`), so all three buggy branches are unconditionally executed on every call.

`FieldElement5x52::invert_in_place()` calls `modinv64()` which calls `update_de62()` in a loop: [4](#0-3) 

This is called from `Jacobian::to_affine()` to compute `1/z` when converting the recovered point from Jacobian to affine coordinates: [5](#0-4) 

Which is called in the ecrecover recovery path: [6](#0-5) 

The `FieldElement5x52` / `mod_inv64` path is selected on 64-bit targets (forward/sequencer mode): [7](#0-6) 

On 32-bit RISC-V (proving mode), `FieldElement10x26` / `mod_inv32` is used instead: [8](#0-7) 

The existing `test_invert` proptest in `field/mod.rs` does **not** exercise `FieldElement5x52` during normal `cargo test` runs because the `debug_assertions` cfg branch selects `field_impl` instead: [9](#0-8) 

---

### Impact Explanation

A wrong field inversion in `Jacobian::to_affine()` produces an incorrect recovered public key from ecrecover. This has two consequences:

1. **Valid transactions rejected (DoS):** The sequencer (64-bit) computes the wrong signer address, causing valid transactions to fail signature verification. The prover (32-bit RISC-V) would compute the correct address. This creates a forward/proving divergence where the sequencer silently drops valid user transactions.

2. **Invalid signature accepted (authentication bypass):** If the corrupted field inversion happens to produce an address matching the expected signer, an invalid signature could pass verification on the sequencer. The prover would then reject it, causing an unprovable block — a liveness failure.

The ecrecover precompile is the root of all EOA transaction authentication: [10](#0-9) 

---

### Likelihood Explanation

- The bug is triggered on every call to `FieldElement::invert_in_place()` on a 64-bit host, because all three buggy branches execute unconditionally (secp256k1 prime limbs 1–3 are all non-zero).
- Every ecrecover call (i.e., every EOA transaction) goes through `Jacobian::to_affine()` → `invert_in_place()`.
- The bug is not caught by existing tests because `cargo test` on 64-bit uses `field_impl` (debug mode), not `FieldElement5x52`.
- An unprivileged user submitting any standard EOA transaction triggers this path.

---

### Recommendation

Fix `update_de62` in `crypto/src/secp256k1/field/mod_inv64.rs` to use `me` (not `md`) for all `ce` accumulator corrections, and fix the condition at line 237 to check `modulus.0[3]` instead of `modulus.0[2]`. The corrected blocks should be:

```rust
if mod_info.modulus.0[1] != 0 {
    cd += mod_info.modulus.0[1] as i128 * md as i128;
    ce += mod_info.modulus.0[1] as i128 * me as i128;  // fix: me
}
// ...
if mod_info.modulus.0[2] != 0 {
    cd += mod_info.modulus.0[2] as i128 * md as i128;
    ce += mod_info.modulus.0[2] as i128 * me as i128;  // fix: me
}
// ...
if mod_info.modulus.0[3] != 0 {  // fix: check index 3
    cd += mod_info.modulus.0[3] as i128 * md as i128;
    ce += mod_info.modulus.0[3] as i128 * me as i128;  // fix: me
}
```

Add a dedicated proptest for `FieldElement5x52::invert_in_place()` that runs without `debug_assertions` or with an explicit feature flag to ensure the 64-bit path is covered.

---

### Proof of Concept

The bug is structurally identical to the roulette report's analog: a copy-paste error causes the wrong variable (`md` instead of `me`) to be used in a repeated calculation block, producing asymmetric/incorrect results for one of two tracked values (`e` vs `d`). The secp256k1 prime's non-zero limbs 1–3 guarantee all three buggy branches execute on every inversion call.

Call chain for any EOA transaction:
1. Bootloader calls `ecrecover` precompile [11](#0-10) 
2. `ecrecover_inner` → `crypto::secp256k1::recover()` → `recover_with_context()` [12](#0-11) 
3. `ecmult(...).to_affine()` → `zi.invert_in_place()` → `FieldElement5x52::invert_in_place()` → `modinv64()` → `update_de62()` (buggy) [13](#0-12) 
4. Wrong `e_out` from `update_de62` propagates through all divsteps iterations, producing incorrect `d` (the modular inverse), yielding wrong affine coordinates and wrong recovered address. [14](#0-13)

    pub(super) fn modinv64(&self, mod_info: &ModInfo) -> Self {
        let mut f = mod_info.modulus;
        let mut g = *self;
        let mut eta = -1;
        let mut d = Self::ZERO;
        let mut e = Self::ONE;

        let mut len = 5;

        let mut i = 0;

        loop {
            // Compute transition matrix and new eta
            let t = TransitionMatrix::divsteps62(
                &mut eta,
                Wrapping(f.0[0] as u64),
                Wrapping(g.0[0] as u64),
            );

            (d, e) = t.update_de62(&d, &e, mod_info);

            debug_assert!(f > -mod_info.modulus);
            debug_assert!(f <= mod_info.modulus);
            debug_assert!(g > -mod_info.modulus);
            debug_assert!(g < mod_info.modulus);

            (f, g) = t.updaet_fg62(len, &f, &g);

            if g.is_zero() {
                break;
            }

            let fi = f.0[len - 1];
            let gi = g.0[len - 1];
            let mut cond = (len as i64 - 2) >> 63;
            cond |= fi ^ (fi >> 63);
            cond |= gi ^ (gi >> 63);

            if cond == 0 {
                f.0[len - 2] |= fi << 62;
                g.0[len - 2] |= gi << 62;

                len -= 1;
            }

            // we should never need more than 12 * 62 = 744 division steps
            debug_assert!({
                i += 1;
                i < 12
            });

            debug_assert!(f > -mod_info.modulus);
            debug_assert!(f <= mod_info.modulus);
            debug_assert!(g > -mod_info.modulus);
            debug_assert!(g < mod_info.modulus);
        }

        // at this point g is zero and f is +/-1 (i.e. gcd(self, modulus)) and d is +/- the modular inverse
        debug_assert!(g == Self::ZERO);
        debug_assert!(
            (f == Self::ONE || f == -Self::ONE)
                || (*self == Self::ZERO
                    && d == Self::ZERO
                    && (f == mod_info.modulus || f == -mod_info.modulus))
        );

        d.normalize(f.0[len - 1], mod_info)
    }

pub fn recover_with_context(
    message: &crate::k256::Scalar,
    signature: &crate::k256::ecdsa::Signature,
    recovery_id: &crate::k256::ecdsa::RecoveryId,
    context: &ECMultContext,
) -> Result<Affine, Secp256k1Err> {
    let (mut sigr, mut sigs) = Scalar::from_signature(signature);
    let message = Scalar::from_k256_scalar(*message);

    // We go through bytes because it's mod GROUP_ORDER and later we need mod BASE FIELD
    let mut brx = sigr.to_repr();

    if recovery_id.is_x_reduced() {
        match <U256 as FieldBytesEncoding<Secp256k1>>::decode_field_bytes(&brx)
            .checked_add(&Secp256k1::ORDER)
            .into_option()
        {
            Some(restored) => {
                brx = <U256 as FieldBytesEncoding<Secp256k1>>::encode_field_bytes(&restored);
            }
            None => return Err(Secp256k1Err::OperationOverflow),
        }
    }

    let is_odd = recovery_id.is_y_odd();
    let x = Affine::decompress(&brx, is_odd).ok_or(Secp256k1Err::InvalidParams)?;

    let xj = x.to_jacobian();

    sigr.invert_in_place();
    sigs *= sigr;

    sigr *= message;
    sigr.negate_in_place();

    let mut pk = ecmult(&xj, &sigs, &sigr, context).to_affine();
    pk.normalize_in_place();

    if pk.is_infinity() {
        return Err(Secp256k1Err::RecoveredInfinity);
    }

    Ok(pk)
}

pub fn ecrecover_inner(
    digest: &[u8; 32],
    r: &[u8; 32],
    s: &[u8; 32],
    rec_id: u8,
) -> Result<crypto::k256::EncodedPoint, ()> {
    use crypto::k256::{
        ecdsa::{hazmat::bits2field, RecoveryId, Signature},
        elliptic_curve::ops::Reduce,
        Scalar,
    };

    let signature = Signature::from_scalars(*r, *s).map_err(|_| ())?;
    let recovery_id = RecoveryId::try_from(rec_id).map_err(|_| ())?;

    let message = <Scalar as Reduce<crypto::k256::U256>>::reduce_bytes(
        &bits2field::<crypto::k256::Secp256k1>(digest).map_err(|_| ())?,
    );

    let Ok(pk) = crypto::secp256k1::recover(&message, &signature, &recovery_id) else {
        return Err(());
    };

    // represent as bytes, and we do not need compression
    let encoded = pk.to_encoded_point(false);

    Ok(encoded)
}
