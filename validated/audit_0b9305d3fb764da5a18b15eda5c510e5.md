### Title
ERC-20 Mirror Over-Minting via Fee-on-Transfer NEP-141 Tokens — (`engine/src/engine.rs`)

---

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` unconditionally trusts `args.amount` from the NEP-141 `ft_on_transfer` callback to mint Aurora ERC-20 mirror tokens. For fee-on-transfer NEP-141 tokens, the actual amount credited to Aurora is less than `args.amount`, causing Aurora to mint more ERC-20 tokens than it holds NEP-141 backing for. This creates bridge insolvency: the last users to exit cannot redeem their ERC-20 tokens for NEP-141, permanently freezing their funds.

---

### Finding Description

When a NEP-141 token is bridged into Aurora, the flow is:

1. A user calls `ft_transfer_call` on the NEP-141 token contract, specifying Aurora as the receiver.
2. The NEP-141 contract transfers tokens and calls `ft_on_transfer` on Aurora with an `amount` argument.
3. Aurora's `ft_on_transfer` entrypoint (in `engine/src/contract_methods/connector.rs`) dispatches to `engine.receive_erc20_tokens()`.
4. `receive_erc20_tokens` reads `args.amount` directly and mints that many ERC-20 mirror tokens.

The vulnerable line is:

```rust
// engine/src/engine.rs
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount),
``` [1](#0-0) 

`args.amount` is the value reported by the NEP-141 token contract in the callback — it is not verified against an actual balance snapshot of Aurora's NEP-141 holdings before and after the transfer. For a fee-on-transfer NEP-141 token, the contract deducts `amount` from the sender but only credits `amount - fee` to Aurora, then calls `ft_on_transfer` with the gross `amount`. Aurora mints `amount` ERC-20 tokens but only holds `amount - fee` NEP-141 tokens.

The same pattern exists in `receive_base_tokens` for the ETH connector path:

```rust
let amount = Wei::new_u128(args.amount.as_u128());
// ...
set_balance(&mut self.io, &receipient, &new_balance);
``` [2](#0-1) 

The `ft_on_transfer` entrypoint is publicly reachable — any NEP-141 token contract can call it as part of the standard `ft_transfer_call` flow: [3](#0-2) 

The ERC-20 mirror token's `mint` function is called with the unverified `amount`: [4](#0-3) 

And the exit path burns ERC-20 tokens and calls the exit precompile to transfer NEP-141 back: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds / Bridge insolvency.**

Each deposit of a fee-on-transfer NEP-141 token mints more ERC-20 tokens than Aurora holds NEP-141 backing for. The deficit accumulates with every deposit. When users attempt to exit by burning ERC-20 tokens via `withdrawToNear`, Aurora's `ft_transfer` call to the NEP-141 contract will fail for the last users because Aurora's NEP-141 balance is insufficient. The ERC-20 tokens are burned in the EVM execution before the NEAR promise is dispatched; if the NEP-141 transfer fails, there is no rollback of the burn, resulting in permanent loss of user funds.

---

### Likelihood Explanation

**Medium.** Any NEP-141 token can be registered with Aurora via `deploy_erc20_token`. The NEP-141 standard does not prohibit fee-on-transfer behavior. An attacker or an unsuspecting user who bridges a fee-on-transfer NEP-141 token triggers the accounting discrepancy without any special privileges. The entry path requires only calling `ft_transfer_call` on a fee-on-transfer NEP-141 token with Aurora as the receiver — a standard, unprivileged operation.

---

### Recommendation

Before minting ERC-20 tokens, record Aurora's NEP-141 balance before the transfer and compare it to the balance after. Use the actual balance delta (not `args.amount`) as the mint amount. Concretely, in `receive_erc20_tokens`, schedule a `ft_balance_of` view call on the NEP-141 contract before and after the transfer, and mint only the difference. Alternatively, document and enforce that only non-fee-on-transfer NEP-141 tokens may be registered via `deploy_erc20_token`, and add an on-chain check or allowlist to enforce this invariant.

---

### Proof of Concept

1. Deploy a fee-on-transfer NEP-141 token `fee_token.near` that deducts 10% on every transfer.
2. Register it with Aurora via `deploy_erc20_token`, creating an ERC-20 mirror at address `0xMIRROR`.
3. Call `ft_transfer_call` on `fee_token.near` with `amount = 1000`, `receiver_id = aurora`, `msg = <recipient_evm_address>`.
4. `fee_token.near` credits Aurora with 900 tokens (after 10% fee) and calls `ft_on_transfer` with `amount = 1000`.
5. Aurora's `receive_erc20_tokens` reads `args.amount = 1000` and mints 1000 ERC-20 tokens to the recipient.
6. Aurora holds 900 NEP-141 tokens but has 1000 ERC-20 tokens outstanding — a 100-token deficit.
7. Repeat step 3–6 ten times: Aurora holds 9000 NEP-141 tokens but has 10000 ERC-20 tokens outstanding.
8. The first 9 users call `withdrawToNear(1000)` successfully (burning 1000 ERC-20, receiving 1000 NEP-141 each — Aurora's balance goes to 0).
9. The 10th user calls `withdrawToNear(1000)`: the ERC-20 burn succeeds in the EVM, but the subsequent `ft_transfer` on `fee_token.near` fails because Aurora holds 0 tokens. The user's 1000 ERC-20 tokens are permanently destroyed with no NEP-141 received. [6](#0-5) [7](#0-6)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
