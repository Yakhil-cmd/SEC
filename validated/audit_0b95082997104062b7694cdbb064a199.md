Audit Report

## Title
Metrics Endpoint Exposes Confidential Rule and Incident Counts to Anonymous Callers — (`rs/boundary_node/rate_limits/canister/metrics.rs`, `canister.rs`)

## Summary
The `/metrics` HTTP endpoint of the rate-limit canister is reachable by any anonymous caller with no authorization check. It recomputes and exposes `active_rules_count` and `stored_incidents_count` directly from raw canister state, counting all rules and incidents regardless of their disclosure status. This bypasses the access control enforced by `inspect_message` that explicitly blocks anonymous callers from calling `get_config`, leaking the existence and count of confidential (undisclosed) rate-limit rules and incidents to the public.

## Finding Description

**Step 1 — `http_request` routes `/metrics` with no identity check.**

The `http_request` query handler in `canister.rs` routes the `/metrics` path directly to `export_metrics_as_http_response` without inspecting the caller principal. [1](#0-0) 

The `inspect_message` hook only fires for update methods and replicated queries. `http_request` is a non-replicated query, so `inspect_message` never runs for it, and any anonymous HTTP caller reaches this handler unconditionally. [2](#0-1) 

**Step 2 — `recompute_metrics` reads raw state with no confidentiality filter.**

`export_metrics_as_http_response` calls `recompute_metrics`, which sets gauges directly from the raw `CanisterApi`: [3](#0-2) 

**Step 3 — `active_rules_count` counts all rule IDs in the latest config, disclosed or not.**

`active_rules_count()` returns `value.rule_ids.len()` from the last config entry with no filter on `disclosed_at`: [4](#0-3) 

**Step 4 — `incidents_count` counts all incidents, disclosed or not.**

`incidents_count()` returns the raw length of the incidents map: [5](#0-4) 

**Step 5 — The confidentiality layer has no effect on the metrics path.**

`ConfigConfidentialityFormatter` only redacts `description` and `rule_raw` fields when serving `get_config` responses. It is never invoked on the metrics path: [6](#0-5) 

**Step 6 — `get_config` is explicitly gated to authorized callers, confirming the design intent.**

`inspect_message` traps any caller that is neither the authorized principal nor an API boundary node when calling `get_config` (`REPLICATED_QUERY_METHOD`). This confirms that even the count of rules is intended to be hidden from anonymous callers: [7](#0-6) 

The metrics endpoint circumvents this gate entirely.

## Impact Explanation

An anonymous caller polling `GET /metrics` observes `active_rules_count`, `stored_incidents_count`, `active_config_version`, and `stored_configs_count` — all of which change immediately when a confidential rule or incident is added via `add_config`. This directly violates the stated confidentiality invariant enforced by `inspect_message`. An attacker conducting an abuse campaign can monitor these metrics and detect the moment a new blocking rule is deployed in response to their activity, allowing them to pause, rotate infrastructure, or time their attack to precede rule activation. This constitutes a meaningful boundary/API infrastructure security impact: the operational effectiveness of the rate-limiting system protecting IC boundary nodes is undermined. This maps to the **Medium** impact tier — a meaningful security impact against boundary node infrastructure, constrained in that only aggregate counts (not rule content) are exposed.

## Likelihood Explanation

The exploit requires only an unauthenticated HTTP GET to a publicly accessible canister endpoint. No special privileges, keys, or network position are needed. The endpoint is intentionally designed to be scraped by Prometheus, so it is reachable from the public internet. The attack is trivially repeatable with zero cost.

## Recommendation

Apply the same confidentiality filtering to the metrics that is applied to `get_config`:
- `active_rules_count` should count only rules where `disclosed_at.is_some()`
- `incidents_count` should count only incidents where `is_disclosed == true` (the `StorableIncident.is_disclosed` field already tracks this)
- Alternatively, remove these sensitive gauges from the public metrics endpoint and expose them only to privileged scrapers via a separate authenticated path

## Proof of Concept

State-machine or PocketIC test:
1. Deploy the canister with an authorized principal.
2. Call `add_config` (as the authorized principal) with one rule whose `incident_id` has no subsequent `disclose_rules` call — the rule remains confidential.
3. As the anonymous principal, send `http_request { method="GET"; url="/metrics"; ... }`.
4. Assert that the response body contains `active_rules_count 1` and `stored_incidents_count 1`.
5. Confirm that calling `get_config` as the anonymous principal is rejected by `inspect_message`, proving the metrics endpoint bypasses the access control gate.

fn inspect_message() {
    // In order for this hook to succeed, accept_message() must be invoked.

    let called_method = ic_cdk::api::call::method_name();

    let (has_full_access, has_full_read_access) = with_canister_state(|state| {
        let authorized_principal = state.get_authorized_principal();
        (
            Some(caller_id) == authorized_principal,
            state.is_api_boundary_node_principal(&caller_id),
        )
    });

    if called_method == REPLICATED_QUERY_METHOD {
        if has_full_access || has_full_read_access {
            ic_cdk::api::call::accept_message();
        } else {
            ic_cdk::api::trap(
                "message_inspection_failed: method call is prohibited in the current context",
            );
        }
    } else if UPDATE_METHODS.contains(&called_method.as_str()) {
        if has_full_access {
            ic_cdk::api::call::accept_message();
        } else {
            ic_cdk::api::trap("message_inspection_failed: unauthorized caller");
        }

        // All others calls are rejected
        ic_cdk::api::trap(
