### Title
Uncapped Cross-Block Log Accumulation in Multiblock Batch Proving Panics on `ArrayVec` Overflow — (`zk_ee/src/common_structs/logs_storage.rs`)

### Summary

In the multiblock batch proving path, `LogsStorage::apply_to_array_vec` accumulates L2→L1 log hashes from each block into a single `ArrayVec<Bytes32, 16384>`. The per-block limit allows up to 16,384 logs per block, but there is no batch-level cap on the total logs across all blocks. When the cumulative log count across blocks in a batch exceeds 16,384, `ArrayVec::push` panics, crashing the prover.

### Finding Description

`MAX_NUMBER_OF_LOGS` is defined as 16,384 — matching the fixed-size Merkle tree used for L2→L1 logs: [1](#0-0) 

The per-block limit is enforced in `check_for_block_limits` using a strict `>` comparison, meaning exactly 16,384 logs per block are permitted: [2](#0-1) 

In the multiblock batch proving post-op, `apply_to_array_vec` is called once per block, pushing that block's log hashes into the shared `batch_data.logs_storage`: [3](#0-2) 

The target is an `ArrayVec<Bytes32, 16384>` with a hard capacity of 16,384: [4](#0-3) 

`apply_to_array_vec` calls `array_vec.push(log.hash())` unconditionally for every log in the block's storage. `ArrayVec::push` **panics** when the array is full. There is no pre-check, no graceful error return, and no batch-level log count cap.

The `l2_logs_root` function in `batch_data.rs` also operates on an `ArrayVec<Bytes32, 16384>`: [5](#0-4) 

The `push_message` path that feeds logs into `LogsStorage` has no count guard of its own — it only accumulates pubdata metadata: [6](#0-5) 

### Impact Explanation

**Vulnerability class**: Resource accounting bug / uncapped accumulation exceeding a fixed downstream capacity.

When a multiblock batch contains:
- Block A: exactly 16,384 L2→L1 logs (at the per-block limit)
- Block B: ≥ 1 L2→L1 log

…the prover panics during `apply_to_array_vec` for Block B, because the `ArrayVec` is already full after Block A. This crashes proof generation for the entire batch. No valid ZK proof can be produced for that batch, permanently blocking L1 finalization of those blocks.

**Impact**: Denial of Service on the ZKsync OS proving system. Affected blocks cannot be finalized on L1 until the batch is recomposed, which may require operator intervention and causes chain liveness failure.

### Likelihood Explanation

**Likelihood**: Medium.

An unprivileged user can emit L2→L1 logs by calling the L1 Messenger system contract (`0x8008`) via `sendToL1(bytes)`. Each call emits one log. With sufficient gas budget (the block gas limit is the only constraint), an attacker can fill a block with 16,384 logs. Any subsequent block in the same multiblock batch that contains even a single L2→L1 log (including L1→L2 tx result logs emitted automatically by the bootloader) triggers the panic.

L1→L2 priority transactions automatically emit a result log via `emit_l1_l2_tx_log`: [7](#0-6) 

This means the attacker does not even need to control Block B — any block containing a priority transaction suffices.

### Recommendation

1. **Add a batch-level log count cap**: Before calling `apply_to_array_vec`, assert or check that `batch_data.logs_storage.len() + io.logs_storage.len() <= 16384` and return a graceful error if exceeded.
2. **Replace `ArrayVec::push` with `try_push`**: Use `ArrayVec::try_push` and propagate the error instead of panicking.
3. **Enforce a per-batch log limit at the block-sealing level**: Reduce the per-block log limit when multiple blocks are batched together, so the total never exceeds 16,384.

### Proof of Concept

1. Attacker deploys a contract that calls `L1Messenger.sendToL1(bytes)` in a loop.
2. Attacker submits a transaction with enough gas to emit exactly 16,384 L2→L1 messages in Block N. The block-level check (`logs_used > 16384`) passes since `16384 > 16384` is false.
3. Block N+1 is included in the same multiblock batch and contains any L1→L2 priority transaction (which automatically emits one result log via `push_l1_l2_tx_log`).
4. The prover calls `apply_to_array_vec` for Block N (fills the `ArrayVec` to capacity 16,384), then calls it for Block N+1 (attempts `push` on a full `ArrayVec`) → **panic**.
5. Proof generation for the batch fails. The blocks cannot be finalized on L1. [4](#0-3) [2](#0-1) [3](#0-2)
