Audit Report

## Title
Illegal SNS Governance Token Over-Minting via `end_timestamp_seconds.unwrap_or_default()` Baseline Collapse in `distribute_rewards` - (File: `rs/sns/governance/src/governance.rs`)

## Summary
In `distribute_rewards`, the reward baseline timestamp is derived from `latest_reward_event().end_timestamp_seconds.unwrap_or_default()`. When `end_timestamp_seconds` is `None` — which occurs for any SNS whose stored `RewardEvent` predates the introduction of this optional field — the baseline silently collapses to Unix epoch 0. This causes `new_rounds_count` to equal `now / round_duration_seconds` (~19,675 for a 1-day SNS), and `seconds_since_genesis` to saturate to 0 via u64 underflow for nearly all iterations, applying the maximum genesis-era reward rate across all phantom rounds. The result is a one-time, automatic, massive over-distribution of governance tokens on the first periodic task execution after upgrade.

## Finding Description

**Root cause — `unwrap_or_default()` on an optional timestamp field:**

In `rs/sns/governance/src/governance.rs` at L5808–5811, `reward_start_timestamp_seconds` is set to 0 when `end_timestamp_seconds` is `None`:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();   // returns 0 if None
``` [1](#0-0) 

**Phantom round inflation:**

`new_rounds_count` is then computed as `(now - 0) / round_duration_seconds`:

```rust
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [2](#0-1) 

For a 1-day SNS at Unix time ~1,700,000,000, this yields ~19,675 phantom rounds instead of 1.

**u64 underflow in `seconds_since_genesis`:**

Inside the reward purse loop:

```rust
let seconds_since_genesis = round_duration_seconds
    .saturating_mul(i)
    .saturating_add(reward_start_timestamp_seconds)
    .saturating_sub(self.proto.genesis_timestamp_seconds);
``` [3](#0-2) 

With `reward_start_timestamp_seconds = 0` and `genesis_timestamp_seconds ≈ 1,700,000,000`, the expression `round_duration_seconds * i + 0 - genesis_timestamp_seconds` underflows (u64 `saturating_sub`) to 0 for all ~19,675 iterations where `i ≤ 19,675`.

**Maximum genesis-era rate applied to all phantom rounds:**

`reward_rate_at` is called with `seconds_since_genesis = 0`:

```rust
let current_reward_rate = voting_rewards_parameters.reward_rate_at(
    crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
);
``` [4](#0-3) 

In `rs/sns/governance/src/reward.rs`, `reward_rate_at` explicitly clamps pre-genesis times to genesis (L207–211), returning the maximum initial reward rate for every phantom round. [5](#0-4) 

**`should_distribute_rewards` has the same flaw:**

```rust
let seconds_since_last_reward_event = now.saturating_sub(
    self.latest_reward_event()
        .end_timestamp_seconds
        .unwrap_or_default(),
);
``` [6](#0-5) 

When `end_timestamp_seconds` is `None`, this returns `now`, which is always greater than `round_duration_seconds`, so `should_distribute_rewards` unconditionally returns `true`, triggering `distribute_rewards`.

**Initialization guard does not cover existing `RewardEvent` records:**

The guard at L726 only fires when `latest_reward_event.is_none()`. If an SNS was upgraded from a version that stored a `RewardEvent` without `end_timestamp_seconds`, the existing event is preserved with `end_timestamp_seconds: None`, and this guard is skipped entirely. [7](#0-6) 

**Proto confirms the field is optional:**

```proto
optional uint64 end_timestamp_seconds = 5;
``` [8](#0-7) 

**Overflow protection does not save the SNS:**

The u64 overflow guard at L5878–5889 returns early only if `rewards_purse_e8s` exceeds u64 max (~1.84 × 10^19 e8s). For a 1B-token SNS at 10%/year with 19,675 phantom rounds, the inflated purse is ~5.39 × 10^17 e8s — well within u64 range. The function proceeds to distribute. [9](#0-8) 

## Impact Explanation

On the first periodic task execution after an SNS upgrade where `end_timestamp_seconds` is `None`:

1. ~19,675 phantom rounds are computed.
2. All iterations use the maximum genesis reward rate (due to u64 underflow to 0).
3. The rewards purse is inflated by a factor of ~19,675 relative to a normal single-round distribution.
4. For a 1B-token SNS at 10%/year initial rate: normal per-round reward ≈ 274 tokens; inflated reward ≈ 5,390,000 tokens — a ~19,675× over-mint of governance tokens distributed as neuron maturity.
5. After this single event, `end_timestamp_seconds` is correctly set and subsequent rounds proceed normally. The damage is permanent and irreversible.

This constitutes **illegal minting of SNS governance tokens**, permanently inflating the token supply and diluting all existing holders. For any SNS with significant token value, the over-minted tokens could easily exceed $1M in value. This matches the Critical impact category: **Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets, especially over $1M.**

## Likelihood Explanation

The `end_timestamp_seconds` field (proto field 5) replaced the deprecated `round` field. Any SNS canister deployed before this field was introduced and subsequently upgraded to a version that reads it will have `end_timestamp_seconds: None` in its stored `latest_reward_event`. The periodic task (`run_periodic_tasks`) is called automatically by the IC timer — no external attacker action is required. The bug fires automatically on the first timer tick after upgrade. No special privileges, no user interaction, no external dependencies.

## Recommendation

Replace `unwrap_or_default()` with a fallback that reconstructs the correct timestamp from the deprecated `round` field and `genesis_timestamp_seconds` when `end_timestamp_seconds` is absent:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_else(|| {
        let round = self.latest_reward_event().round;
        round
            .saturating_mul(round_duration_seconds)
            .saturating_add(self.proto.genesis_timestamp_seconds)
    });
```

Apply the same fix to `should_distribute_rewards` at L5735–5738. Alternatively, add a one-time migration in `post_upgrade` (or in `Governance::new`) that populates `end_timestamp_seconds` from `round` for any stored `RewardEvent` that lacks it, analogous to the existing `latest_reward_event.is_none()` guard.

## Proof of Concept

**State**: SNS canister with stored state:
- `genesis_timestamp_seconds = 1_700_000_000`
- `latest_reward_event = RewardEvent { round: 100, end_timestamp_seconds: None, ... }`
- `round_duration_seconds = 86_400` (1 day)
- `now = 1_700_086_400` (one day after genesis)

**Execution of `distribute_rewards`**:
1. `reward_start_timestamp_seconds = None.unwrap_or_default() = 0`
2. `new_rounds_count = (1_700_086_400 - 0) / 86_400 = 19_676`
3. Loop `i = 1..=19_676`:
   - `seconds_since_genesis = 86_400 * i + 0 - 1_700_000_000`
   - For `i ≤ 19_675`: result is negative → saturates to `0`
   - `reward_rate_at(Instant::from_seconds_since_genesis(0))` = maximum genesis rate
4. `rewards_purse_e8s ≈ 19_676 × max_rate × 86_400 × supply`
5. For supply = 10^17 e8s (1B tokens): purse ≈ 5.39 × 10^17 e8s (5.39M tokens) instead of ~274 tokens
6. All neurons receive ~19,676× their expected maturity increment
7. `latest_reward_event.end_timestamp_seconds` is set to `19_676 × 86_400 ≈ now`
8. Subsequent calls to `should_distribute_rewards` correctly return `false` until the next round

A deterministic unit test can reproduce this by constructing a `Governance` instance with a `latest_reward_event` having `end_timestamp_seconds: None` and `round: 100`, setting `now` to `genesis + 86_400`, calling `distribute_rewards`, and asserting that `rewards_purse_e8s` is ~19,676× the expected single-round value.
