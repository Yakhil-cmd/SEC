### Title
Permanent Fund Loss When `ExitToNear` NEP-141 Promise Fails Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary
The `ExitToNear` precompile burns ERC-20 tokens on Aurora before the NEAR-side `ft_transfer` promise is executed. A compile-time feature flag (`error_refund`) controls whether a refund is issued if that promise fails. Because `error_refund` is **not** in the default feature set, any standard build of Aurora Engine permanently loses user funds whenever the NEP-141 `ft_transfer` fails — including when the NEP-141 contract is deprecated or migrated to a new account.

---

### Finding Description

**Step 1 — Tokens are burned before the NEAR promise is confirmed.**

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` method constructs the callback arguments that will govern what happens if the NEAR-side promise fails: [1](#0-0) 

When the `error_refund` feature is absent, `refund` is unconditionally set to `None`. The ERC-20 burn has already occurred at this point; the promise is merely scheduled.

**Step 2 — The callback silently swallows promise failures.**

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` checks the promise result: [2](#0-1) 

When `args.refund` is `None` (the default-build path), the `else if` branch is never reachable. A failed promise falls through to the final `else { None }` arm — no refund, no error, no recovery.

**Step 3 — `error_refund` is not a default feature.** [3](#0-2) 

`error_refund` is absent from `default = ["std"]`. Every standard build ships without the refund path.

**Step 4 — The existing test explicitly documents the fund loss.** [4](#0-3) 

The comment reads: *"If the refund feature is not enabled then there is no refund in the EVM."* The balance is permanently reduced by `FT_EXIT_AMOUNT`.

---

### Impact Explanation
**Critical — Permanent freezing/loss of funds.** Any user who calls `ExitToNear` (directly or via an ERC-20 `withdrawToNear`) in a default build permanently loses their tokens if the NEAR-side `ft_transfer` fails. The ERC-20 burn on Aurora is irreversible, and without the refund path there is no recovery mechanism.

---

### Likelihood Explanation
**High.** The `error_refund` feature is absent from the default build, so every standard deployment is affected. The NEP-141 `ft_transfer` can fail for multiple realistic, externally-triggered reasons that mirror the Balancer oracle deprecation scenario:

- The NEP-141 contract is **migrated or deprecated** by the token issuer (direct analog: Balancer migrating away from deprecated oracle vaults).
- The recipient account is not registered with the NEP-141 contract.
- The NEP-141 contract is paused.
- The Aurora engine account has insufficient NEP-141 balance.

Any unprivileged EVM user can trigger this by calling the `ExitToNear` precompile at address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`. [5](#0-4) 

---

### Recommendation
1. **Short term:** Add `error_refund` to the `default` features in both `engine/Cargo.toml` and `engine-precompiles/Cargo.toml` so the refund path is always compiled in.
2. **Long term:** Remove the compile-time gate entirely and make the refund path unconditional, so any failure of the NEAR-side promise always restores the burned ERC-20 tokens to the caller.

---

### Proof of Concept
1. Build Aurora Engine without explicitly enabling `error_refund` (the default).
2. Bridge a NEP-141 token to Aurora, receiving an ERC-20 mirror token.
3. Migrate or deprecate the NEP-141 contract on NEAR (or call `ExitToNear` with an unregistered recipient — as done in `test_exit_to_near_refund`).
4. Call `withdrawToNear` on the ERC-20 mirror token.
5. Observe: ERC-20 tokens are burned on Aurora; the `ft_transfer` promise fails on NEAR; `exit_to_near_precompile_callback` returns `None` without refunding; tokens are permanently lost.

The existing test `test_exit_to_near_refund` at: [6](#0-5) 

explicitly demonstrates this: with `error_refund` disabled, the ERC-20 balance after a failed exit is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` — the exited tokens are gone with no recovery path.

    #[tokio::test]
    async fn test_exit_to_near_refund() {
        // Deploy Aurora; deploy NEP-141; bridge NEP-141 to ERC-20 on Aurora
        let TestExitToNearContext {
            ft_owner,
            ft_owner_address,
            nep_141,
            erc20,
            aurora,
            ..
        } = test_exit_to_near_common().await.unwrap();

        // Call exit on ERC-20; ft_transfer promise fails; expect refund on Aurora;
        exit_to_near(
            &ft_owner,
            // The ft_transfer will fail because this account is not registered with the NEP-141
            "unregistered.near",
            FT_EXIT_AMOUNT,
            &erc20,
            &aurora,
        )
        .await
        .unwrap();

        assert_eq!(
            nep_141_balance_of(&nep_141, &ft_owner.id()).await,
            FT_TOTAL_SUPPLY - FT_TRANSFER_AMOUNT
        );
        assert_eq!(
            nep_141_balance_of(&nep_141, &aurora.id()).await,
            FT_TRANSFER_AMOUNT
        );

        #[cfg(feature = "error_refund")]
        let balance = FT_TRANSFER_AMOUNT.into();
        // If the refund feature is not enabled then there is no refund in the EVM
        #[cfg(not(feature = "error_refund"))]
        let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();

        assert_eq!(
            erc20_balance(&erc20, ft_owner_address, &aurora).await,
            balance
        );
    }
