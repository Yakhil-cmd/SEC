### Title
Arithmetic Overflow in XCC Precompile `required_near` Calculation Allows Draining Engine's NEAR Balance - (File: `engine-precompiles/src/xcc.rs`)

### Summary
An arithmetic overflow in the `required_near` computation inside the `CrossContractCall` precompile's `run_with_handle` function allows an unprivileged EVM user to trigger XCC router deployments without paying the required storage-staking cost (`STORAGE_AMOUNT = 2 NEAR`). The engine pays the 2 NEAR from its own balance and never receives a refund, enabling an attacker to drain the engine's NEAR balance by repeating the exploit across many EVM addresses.

### Finding Description
In `engine-precompiles/src/xcc.rs`, when a new XCC router must be deployed for a sender address that has never used XCC, the precompile computes the total NEAR required from the user as:

```rust
let required_near =
    match state::get_code_version_of_address(&self.io, &Address::new(sender)) {
        None => attached_near + state::STORAGE_AMOUNT,
        Some(_) => attached_near,
    };
``` [1](#0-0) 

`attached_near` is derived directly from user-controlled Borsh-deserialized input (`call.total_near()`), and `STORAGE_AMOUNT` is `Yocto::new(2_000_000_000_000_000_000_000_000)`. The `+` operator on `Yocto` (a `u128` newtype) wraps on overflow in Rust release/WASM builds. An attacker who supplies `attached_near = u128::MAX - STORAGE_AMOUNT + 1` causes `required_near` to wrap to `0`.

Because `required_near == ZERO_YOCTO`, the guard at line 184 skips the `transferFrom` call on the wNEAR contract entirely — the attacker pays nothing:

```rust
if required_near != ZERO_YOCTO {
    // transferFrom wNEAR from sender to engine
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
}
``` [2](#0-1) 

The precompile then emits a log recording `required_near = 0` in the topics:

```rust
let topics = vec![
    cross_contract_call::AMOUNT_TOPIC,
    H256(aurora_engine_types::types::u256_to_arr(&U256::from(
        required_near.as_u128(),
    ))),
];
``` [3](#0-2) 

The engine then calls `handle_precompile_promise` with `required_near = 0`. Inside that function, because `create_needed = true` (first-time router), the engine unconditionally transfers `STORAGE_AMOUNT` from its own NEAR balance to the new router account:

```rust
if *create_needed {
    promise_actions.push(PromiseAction::CreateAccount);
    promise_actions.push(PromiseAction::Transfer {
        amount: STORAGE_AMOUNT,
    });
``` [4](#0-3) 

The design intent (documented in the comment at lines 283–288) is that the engine is reimbursed via a wNEAR withdrawal followed by a `send_refund` callback from the router. However, because `required_near = 0`, the withdrawal branch is skipped:

```rust
let withdraw_id = if required_near == ZERO_YOCTO {
    setup_id
} else {
    // withdraw wNEAR to router ...
``` [5](#0-4) 

No wNEAR is unwrapped, the router has no NEAR to `send_refund`, and the engine permanently loses `STORAGE_AMOUNT` (2 NEAR) per exploit iteration.

### Impact Explanation
Each exploit iteration costs the engine 2 NEAR with zero cost to the attacker (no wNEAR balance required). An attacker generating N fresh EVM addresses can drain N × 2 NEAR from the engine's NEAR balance. As the engine's NEAR balance is depleted, it becomes unable to pay for NEAR storage and cross-contract operations, causing insolvency and permanent freezing of all engine functionality for all users. This is a **Critical** impact: insolvency and permanent fund freeze.

### Likelihood Explanation
The attack requires only:
1. An EVM address that has never used XCC (trivially satisfied with any new key pair).
2. Calling the XCC precompile with a crafted `attached_near` value (`u128::MAX - STORAGE_AMOUNT + 1`).
3. No wNEAR balance, no special permissions, no admin access.

EVM key pairs are free to generate. The exploit is fully permissionless and repeatable. Likelihood is **High**.

### Recommendation
Replace the plain `+` with a checked or saturating addition and return an error on overflow:

```rust
let required_near = match state::get_code_version_of_address(&self.io, &Address::new(sender)) {
    None => attached_near
        .as_u128()
        .checked_add(state::STORAGE_AMOUNT.as_u128())
        .map(Yocto::new)
        .ok_or_else(|| revert_with_message("ERR_NEAR_AMOUNT_OVERFLOW"))?,
    Some(_) => attached_near,
};
```

Additionally, add an explicit upper-bound check on `attached_near` before the computation to reject unreasonably large values early.

### Proof of Concept
1. Attacker generates a fresh EVM key pair with address `A` (never used XCC).
2. Attacker submits an EVM transaction calling the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372` with a `CrossContractCallArgs::Eager` payload where `total_near()` returns `u128::MAX - 2_000_000_000_000_000_000_000_000 + 1`.
3. Inside `run_with_handle`: `required_near = (u128::MAX - STORAGE_AMOUNT + 1) + STORAGE_AMOUNT = 0` (wrapping overflow).
4. The `if required_near != ZERO_YOCTO` guard is false; no `transferFrom` is executed; attacker pays 0 wNEAR.
5. The precompile emits a log with `required_near = 0`.
6. The engine calls `handle_precompile_promise(..., required_near = 0, ...)`.
7. `create_needed = true` (address `A` has no router); engine issues `PromiseAction::Transfer { amount: STORAGE_AMOUNT }` — 2 NEAR leaves the engine's balance.
8. `required_near == ZERO_YOCTO` → no wNEAR withdrawal, no `send_refund` — engine is never reimbursed.
9. Repeat with fresh addresses to drain the engine's NEAR balance.
