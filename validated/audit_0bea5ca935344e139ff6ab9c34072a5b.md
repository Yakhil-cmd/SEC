Audit Report

## Title
Unconditional index-out-of-bounds panic via empty `EnvelopePair` list in `/construction/parse` — (`rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

## Summary

The signed-transaction branch of `construction_parse` unconditionally accesses `updates[0]` on a `Vec<EnvelopePair>` that is never validated to be non-empty. An unauthenticated caller can POST a CBOR-encoded `SignedTransaction` containing a `(RequestType, vec![])` tuple, which deserializes successfully and then panics at the index access, crashing or disrupting the Rosetta server process.

## Finding Description

In `construction_parse.rs` at line 42, the signed-transaction branch maps over `signed_transaction.requests` and unconditionally indexes `updates[0]`:

```rust
ParsedTransaction::Signed(signed_transaction) => signed_transaction
    .requests
    .iter()
    .map(
        |(request_type, updates)| match updates[0].update.content.clone() {
            HttpCallContent::Call { update } => (request_type.clone(), update),
        },
    )
    .collect(),
```

The `Request` type alias is `(RequestType, Vec<EnvelopePair>)`, meaning the inner vector can be empty. The `ParsedTransaction::try_from` implementation in `models.rs` (lines 180–196) performs only CBOR deserialization with no structural validation — it does not check that any inner `Vec<EnvelopePair>` is non-empty. CBOR encoding of an empty vector is valid and deserializes without error. When the iterator reaches a tuple whose `Vec<EnvelopePair>` is empty, `updates[0]` panics unconditionally.

The `SignedTransaction` struct (`models.rs` line 33) and `Request` type alias (`models.rs` line 58) confirm the structure accepts empty inner vectors. The `construction_combine` path (`construction_combine.rs` line 172) always produces non-empty vectors in normal operation, but there is no enforcement at the parse boundary.

## Impact Explanation

This is a High-severity finding matching the allowed impact: **"Application/platform-level DoS, crash, or Rosetta/boundary/API security impact with concrete user or protocol harm."** The Rosetta API is an in-scope financial integration component. A Rust index-out-of-bounds panic unwinds the handler thread. Under actix-web (confirmed by the `actix_web::ResponseError` impl on `Error` in `models.rs` lines 307–316), panics in request handlers may be caught and return HTTP 500, but repeated triggering renders the Rosetta API non-functional for all clients depending on it for ICP transaction submission and parsing.

## Likelihood Explanation

The attack requires no authentication, no key material, and no privileged access. The attacker needs only network access to the Rosetta HTTP endpoint (publicly exposed by design), a CBOR library to encode `SignedTransaction { requests: [(RequestType::Send, [])] }`, and hex-encoding of the result. The payload is trivially constructable and the attack is repeatable at will.

## Recommendation

Replace the unconditional `updates[0]` index with a bounds-checked access that returns an `ApiError` on empty input:

```rust
ParsedTransaction::Signed(signed_transaction) => signed_transaction
    .requests
    .iter()
    .map(|(request_type, updates)| {
        let first = updates.first().ok_or_else(|| {
            ApiError::invalid_request(
                "SignedTransaction request has no envelope pairs"
            )
        })?;
        match first.update.content.clone() {
            HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
        }
    })
    .collect::<Result<Vec<_>, _>>()?,
```

Alternatively, add a validation step inside `ParsedTransaction::try_from` that rejects any `SignedTransaction` containing an empty `Vec<EnvelopePair>`.

## Proof of Concept

```rust
// Construct the minimal crashing payload
let signed_tx = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],  // empty EnvelopePair list
};
let cbor_bytes = serde_cbor::to_vec(&signed_tx).unwrap();
let hex_tx = hex::encode(cbor_bytes);

// POST to /construction/parse
let body = serde_json::json!({
    "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
    "signed": true,
    "transaction": hex_tx
});
// Server panics at updates[0] in construction_parse.rs:42
```
