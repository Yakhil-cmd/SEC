### Title
Silent Discard of EIP-7002/7251/6110 System-Call Errors Creates Forward/Proving Divergence — (`basic_bootloader/src/bootloader/block_flow/ethereum/post_tx_op_sequencing.rs`)

---

### Summary

In the non-proving (forward/sequencer) post-block operation path, the results of three critical system calls — `eip6110_events_parser`, `eip7002_system_part`, and `eip7251_system_part` — are silently discarded with `let _ =`. In the proving path, the identical calls are guarded with `.expect(...)`, which panics on any error. This asymmetry creates a forward/proving divergence: a block that the sequencer accepts (because errors are swallowed) may be impossible to prove (because the prover panics on the same error), causing a chain halt.

---

### Finding Description

In `post_tx_op_sequencing.rs`, the non-proving `PostTxLoopOp` implementation discards all errors from the three EIP system-part functions:

```rust
// Environment may have no such contracts predeployed for tests or sequencing purposes
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);
``` [1](#0-0) 

In contrast, the proving path in `post_tx_op_proving.rs` calls the same functions with `.expect(...)`, which panics on any error:

```rust
if eip7002_system_part(system, &mut intermediate_hasher)
    .expect("withdrawal requests must be processed")
{ ... }
if eip7251_system_part(system, &mut intermediate_hasher)
    .expect("consolidation requests must be processed")
{ ... }
``` [2](#0-1) 

`eip7002_system_part` returns `Err(SystemError::LeafDefect(...))` when the EIP-7002 withdrawal predeploy contract is absent or when any storage read/write fails. The function also performs storage mutations (updating queue head pointer, resetting excess count) before it may return an error: [3](#0-2) [4](#0-3) 

When `eip7002_system_part` (or `eip7251_system_part`) fails mid-execution in the forward path, the error is silently swallowed, the `requests_hasher` is not updated with the correct withdrawal/consolidation data, and any partial storage mutations are left in place. The block is then sealed and accepted by the sequencer. When the prover later attempts to reproduce the same block, it calls the same function under `.expect()`, which panics — making the block unprovable.

---

### Impact Explanation

**Forward/proving divergence** — the sequencer accepts a sealed block that the prover cannot reproduce. This can halt the ZKsync chain because the sequencer's committed state cannot be proven. Additionally, if partial storage mutations were applied before the error (e.g., the queue head was updated but the excess count was not reset), the on-chain state becomes inconsistent with what the prover expects, compounding the divergence.

---

### Likelihood Explanation

The divergence is triggered whenever the EIP-7002 or EIP-7251 predeploy contracts are absent from the state (e.g., during a protocol upgrade window, a fresh chain deployment, or any environment where these contracts have not yet been deployed). The comment in the code explicitly acknowledges this scenario: *"Environment may have no such contracts predeployed for tests or sequencing purposes."* Because the non-proving path silently accepts this condition while the proving path panics on it, any block produced in such an environment is unprovable. The condition is not attacker-controlled in the traditional sense, but it is a realistic operational state that can arise during normal chain lifecycle events.

---

### Recommendation

Replace the `let _ =` pattern with explicit error handling that mirrors the proving path's strictness, or introduce a shared helper that distinguishes "contract not deployed" (acceptable skip) from all other errors (fatal):

```rust
// Non-proving path should propagate unexpected errors
eip6110_events_parser(&system, &mut requests_hasher)
    .or_else(|e| if is_not_deployed_error(&e) { Ok(false) } else { Err(e) })?;
eip7002_system_part(&mut system, &mut requests_hasher)
    .or_else(|e| if is_not_deployed_error(&e) { Ok(false) } else { Err(e) })?;
eip7251_system_part(&mut system, &mut requests_hasher)
    .or_else(|e| if is_not_deployed_error(&e) { Ok(false) } else { Err(e) })?;
```

This ensures that only the expected "contract not deployed" case is silently skipped, while all other errors are propagated and halt block sealing — matching the proving path's behavior.

---

### Proof of Concept

1. Deploy a ZKsync OS chain without the EIP-7002 withdrawal predeploy contract at `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS`.
2. Submit any valid transaction and seal a block. The sequencer calls `post_op` in `EthereumPostOp<VC, false>`. `eip7002_system_part` returns `Err(SystemError::LeafDefect("EIP-7002 withdrawal contract is not deployed"))`. The `let _ =` discards the error; the block is sealed and committed.
3. The prover attempts to reproduce the block. It calls `eip7002_system_part(...).expect("withdrawal requests must be processed")`. The same error is returned, causing a panic. The block cannot be proven.
4. The chain halts because the sequencer has committed a block that the prover cannot verify. [5](#0-4) [6](#0-5) [3](#0-2)
