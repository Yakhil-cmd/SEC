Audit Report

## Title
Stale `API_BOUNDARY_NODE_PRINCIPALS` Heap Cache Allows Removed Boundary Nodes Unauthorized Access and Causes Post-Upgrade Access Denial - (`rs/boundary_node/rate_limits/canister/canister.rs`, `rs/boundary_node/salt_sharing/canister/canister.rs`)

## Summary

Both the rate-limit canister and salt-sharing canister store authorized API boundary node principals in a `thread_local! RefCell<HashSet<Principal>>` backed by Wasm heap memory, not stable memory. This cache is populated only by a periodic timer and is never immediately refreshed. Two concrete security windows result: (1) a removed boundary node retains access for up to one polling interval (~60 s), and (2) after every canister upgrade the cache is empty until the first timer tick, denying all legitimate boundary nodes access to certified rate-limit rules.

## Finding Description

**Root cause â€” non-stable heap storage:**

In the rate-limit canister, `API_BOUNDARY_NODE_PRINCIPALS` is declared as a plain `thread_local!` `RefCell<HashSet<Principal>>` with no stable-memory backing: [1](#0-0) 

In the salt-sharing canister, the same pattern is used: [2](#0-1) 

Because neither uses `ic_stable_structures`, both caches are reset to an empty `HashSet` on every canister upgrade.

**Rate-limit canister â€” no immediate poll on init/post_upgrade:**

`periodically_poll_api_boundary_nodes` uses `set_timer_interval`, which fires only after the first full interval has elapsed, never immediately: [3](#0-2) 

`post_upgrade` simply calls `init`, which calls `periodically_poll_api_boundary_nodes`: [4](#0-3) [5](#0-4) 

**Salt-sharing canister â€” same gap in `init_async`:**

Although `init` fires `init_async` via `set_timer(Duration::ZERO, ...)`, `init_async` itself only registers a `set_timer_interval` for boundary node polling â€” it does not call `poll_api_boundary_nodes` immediately: [6](#0-5) 

**Authorization checks read from the stale/empty cache:**

Rate-limit canister `inspect_message` reads `is_api_boundary_node_principal` from the heap cache: [7](#0-6) 

Salt-sharing canister `inspect_message` and `get_salt` both read from the same heap cache: [8](#0-7) [9](#0-8) 

**Instance 1 â€” Stale access after node removal:**
When an API boundary node is removed from the NNS registry, the registry is updated immediately, but `API_BOUNDARY_NODE_PRINCIPALS` in both canisters is only refreshed on the next timer tick. The removed node's principal remains in the cache for up to `registry_polling_period_secs` seconds. During this window, `is_api_boundary_node_principal()` returns `true` for the removed node, allowing it to call `get_config` (reading confidential, not-yet-disclosed rate-limit rules) and `get_salt` (reading the shared privacy salt). [10](#0-9) 

**Instance 2 â€” Post-upgrade empty cache:**
After every canister upgrade, `API_BOUNDARY_NODE_PRINCIPALS` is empty. The first timer tick fires only after the full polling interval (~60 s). During this window, `is_api_boundary_node_principal()` returns `false` for every caller, so `inspect_message` rejects all `get_config` update calls (replicated queries) from legitimate boundary nodes, preventing them from obtaining certified rate-limit rules. [11](#0-10) 

## Impact Explanation

**Instance 1** maps to: *Medium â€” Difficult attack requiring node/boundary-node control, but with meaningful security impact.* A decommissioned or compromised API boundary node (operated by an independent party, removed via NNS governance) can read confidential rate-limit rules and the shared privacy salt for up to 60 seconds. The salt exposure enables cross-node user deanonymization, undermining the privacy guarantees of the rate-limiting system.

**Instance 2** maps to: *High â€” Application/platform-level DoS on boundary/API infrastructure with concrete protocol harm.* Every canister upgrade (a routine governance operation) triggers a guaranteed ~60-second window during which no legitimate API boundary node can obtain certified rate-limit rules via replicated query. This disrupts rate-limit enforcement immediately after upgrades.

## Likelihood Explanation

**Instance 1**: Realistic. API boundary node removal via NNS proposal is a normal operational event (e.g., decommissioning, compromise response). The 60-second window is sufficient for a targeted call. No subnet majority or threshold key is required â€” only control of the removed node's principal.

**Instance 2**: Certain. Every canister upgrade resets the heap cache. Upgrades are routine NNS governance operations. The window is deterministic and repeatable.

## Recommendation

1. **Immediate poll on init/post_upgrade (rate-limit canister):** Replace `set_timer_interval` with an initial `set_timer(Duration::ZERO, ...)` that performs one immediate poll and then re-registers the interval, so `API_BOUNDARY_NODE_PRINCIPALS` is populated before the first ingress message is processed.

2. **Immediate poll in `init_async` (salt-sharing canister):** Call `poll_api_boundary_nodes().await` directly inside `init_async` before registering the interval timer, so the cache is populated at zero-delay init time.

3. **Persist to stable memory:** Store `API_BOUNDARY_NODE_PRINCIPALS` in `ic_stable_structures` (e.g., a `StableBTreeMap` or `StableCell`) so it survives upgrades without requiring a fresh poll, eliminating the post-upgrade empty-cache window entirely.

4. **Push-based invalidation:** Expose an authenticated endpoint callable only by the registry canister or NNS governance to immediately remove a principal from `API_BOUNDARY_NODE_PRINCIPALS` upon node removal, eliminating the polling-window stale-access problem.

## Proof of Concept

**Instance 1 (stale access):**
1. API boundary node `BN_A` is registered. Both canisters poll and add `BN_A` to `API_BOUNDARY_NODE_PRINCIPALS`.
2. NNS governance calls `remove_api_boundary_nodes({node_ids: [BN_A]})` on the registry. Registry is updated immediately.
3. Within 60 seconds (before the next timer tick), `BN_A` calls `get_config` on the rate-limit canister as an update call. `inspect_message` checks `is_api_boundary_node_principal(BN_A)` â€” returns `true` (stale cache). Call is accepted; `BN_A` reads confidential rate-limit rules.
4. `BN_A` calls `get_salt` on the salt-sharing canister. Same stale-cache check passes. `BN_A` reads the shared privacy salt.

**Instance 2 (post-upgrade denial):**
1. Rate-limit canister is upgraded via NNS proposal. `post_upgrade` â†’ `init` â†’ `periodically_poll_api_boundary_nodes(60, ...)`. `API_BOUNDARY_NODE_PRINCIPALS` is now empty.
2. Within the next 60 seconds, legitimate boundary node `BN_B` calls `get_config` as an update call. `inspect_message` checks `is_api_boundary_node_principal(BN_B)` â€” returns `false` (empty cache). Call is rejected with `"message_inspection_failed: method call is prohibited in the current context"`.
3. After 60 seconds, the timer fires, polls the registry, repopulates the cache. `BN_B` can now call `get_config` successfully.

A deterministic integration test using PocketIC can reproduce both instances: (1) register a boundary node, poll, remove it from the registry mock, advance time by less than the polling interval, assert `get_config`/`get_salt` still succeeds from the removed node; (2) upgrade the canister, immediately call `get_config` from a registered boundary node, assert rejection, advance time past the polling interval, assert success.
