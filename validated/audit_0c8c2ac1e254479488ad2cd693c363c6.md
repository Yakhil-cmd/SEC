Audit Report

## Title
Silently Ignored ICP Burn Result Allows Cycles Minting Without ICP Destruction — (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows ICP ledger burn failures after cycles have already been irreversibly minted or deposited. When the ICP ledger returns a transient error, cycles are issued to the caller while the corresponding ICP remains unburned in the CMC's subaccount, and the notification is immediately finalized in a terminal state that prevents any retry. This breaks the ICP/cycles conservation invariant by creating cycles supply without the required ICP supply reduction.

## Finding Description

All three callers invoke `burn_and_log` only after the service-side operation has already succeeded:

- `process_top_up`: calls `deposit_cycles` → on `Ok`, calls `burn_and_log(sub, amount).await` [2](#0-1) 
- `process_create_canister`: calls `do_create_canister` → on `Ok`, calls `burn_and_log(sub, amount).await` [3](#0-2) 
- `process_mint_cycles`: calls `do_mint_cycles` → on `Ok`, calls `burn_and_log(sub, amount).await` [4](#0-3) 

After `burn_and_log` returns (regardless of whether the burn succeeded), the caller inserts a terminal `NotifiedTopUp`/`NotifiedMint`/`NotifiedCreateCanister` entry into `blocks_notified`. The `is_transient_error` removal guard only applies to errors returned from the `process_*` functions themselves — since `burn_and_log` returns `()` unconditionally, the terminal state is always committed. [5](#0-4) 

There is no pending-burn queue, no heartbeat retry, and no stable-state record of failed burns anywhere in the file.

## Impact Explanation
When `burn_and_log` fails silently: cycles are irreversibly credited to the target canister or cycles ledger account; the ICP is not burned from the CMC's subaccount; the `blocks_notified` entry is set to a terminal processed state preventing re-notification; and the ICP sits permanently orphaned in the CMC subaccount with no automated recovery path. The net protocol effect is cycles minted without corresponding ICP destruction — a direct violation of the ICP/cycles conservation invariant constituting illegal minting. This matches the Critical/High allowed impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving ICP/Cycles or in-scope chain-key/ledger assets."* Each occurrence permanently inflates cycles supply relative to ICP supply with no recovery mechanism.

## Likelihood Explanation
The trigger requires the ICP ledger to return a transient inter-canister call error (`SysTransient` or call rejection) during the window between `deposit_cycles`/`do_create_canister`/`do_mint_cycles` succeeding and `burn_and_log` completing. This window opens on every subnet upgrade of the NNS subnet (which hosts the ICP ledger), on replica restarts, and on any transient inter-canister call rejection. No attacker capability beyond submitting a normal ingress `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` call is required. The timing is opportunistic but the window recurs with every NNS upgrade cycle, making this a realistic low-to-medium likelihood event with permanent per-occurrence impact.

## Recommendation
Two safe remediation paths exist:

1. **Async retry via timer/heartbeat**: On burn failure, persist the `(subaccount, amount)` pair in stable state and retry the burn asynchronously via a heartbeat or timer, consistent with how the CMC already handles other deferred operations. This separates the double-mint guard (the `Processing` entry) from the burn-completion guarantee.

2. **Do not finalize the notification on burn failure**: Have `burn_and_log` return a `Result`, and in the callers, only insert the terminal `NotifiedTopUp`/`NotifiedMint`/`NotifiedCreateCanister` entry when the burn succeeds. On burn failure, remove the `Processing` entry (as `is_transient_error` already does for other transient failures), allowing the user to re-notify and trigger a fresh burn attempt. This is the minimal correct fix.

The comment "we don't want to reject the transaction notification because then it could be retried" conflates preventing double-minting (already solved by the `Processing` guard) with ensuring the burn completes (currently unguarded). These concerns are separable.

## Proof of Concept
1. User transfers ICP to the CMC's canister-derived subaccount on the ICP ledger.
2. User calls `notify_top_up` with the block index during a window when the ICP ledger is transiently unavailable (e.g., NNS subnet upgrade in progress).
3. CMC calls `fetch_transaction`, verifies the payment, sets `blocks_notified[block_index] = Processing`.
4. CMC calls `deposit_cycles` — succeeds; cycles are credited to the target canister.
5. CMC calls `burn_and_log(sub, amount).await` — `call_protobuf` to the ledger returns `Err((SysTransient, ...))`.
6. `burn_and_log` logs the error and returns `()`.
7. CMC sets `blocks_notified[block_index] = NotifiedTopUp(Ok(cycles))` — notification is permanently finalized.
8. **Result**: target canister received cycles; ICP was not burned; notification is closed; ICP sits unburned in CMC subaccount indefinitely.

A deterministic integration test can reproduce this by deploying a mock ICP ledger canister that returns `SysTransient` on `send_pb` calls, then calling `notify_top_up` and asserting that (a) cycles were deposited, (b) the CMC subaccount balance is unchanged, and (c) a second `notify_top_up` with the same block index returns `InvalidTransaction` (confirming the terminal state was committed).

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
