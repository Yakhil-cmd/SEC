Audit Report

## Title
Global `ManageNeuron` Proposal Queue Exhaustion via Cheap Non-Refundable Fee Enables NNS-Wide DoS - (File: rs/nns/governance/src/governance.rs)

## Summary
The NNS governance canister enforces a single global cap on open `ManageNeuron` proposals (`MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS`) with no per-neuron sub-limit. The submission fee (`neuron_management_fee_per_proposal_e8s`, 0.01 ICP) is non-refundable and small enough that any neuron holder can fill the entire global queue at minimal cost, permanently blocking all `ManageNeuron` proposals from every NNS participant until the attacker's proposals expire.

## Finding Description
`make_proposal` computes the submission fee via `proposal_submission_fee`, which returns `neuron_management_fee_per_proposal_e8s` (0.01 ICP) for `ManageNeuron` actions. [1](#0-0) 

The companion `reject_cost_e8s` explicitly returns **zero** for `ManageNeuron` proposals, meaning the fee is never reimbursed regardless of outcome. [2](#0-1) 

The fee is charged unconditionally before the proposal is stored. [3](#0-2) 

The only guard against queue exhaustion is a global scan of all open `ManageNeuron` proposals with no per-neuron sub-limit. [4](#0-3) 

An attacker submits `ManageNeuron` proposals (e.g., `Command::Follow` on `Topic::NeuronManagement` targeting their own neuron) without voting on them. Each proposal is accepted, deducts 0.01 ICP from `neuron_fees_e8s`, and increments the global counter. Once `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS` is reached, every subsequent `ManageNeuron` proposal from **any** principal returns `ResourceExhausted`. The code comment at L5352–5355 acknowledges the fee is intended to prevent this, but the fee is too small relative to the global cap to be an effective deterrent. [5](#0-4) 

## Impact Explanation
When the global cap is exhausted, no NNS participant can submit a `ManageNeuron` proposal. This blocks all neuron-management operations (changing followees, adding/removing hot keys, merging neurons, splitting neurons) for the entire NNS for the duration of the attack. This is a concrete application/platform-level DoS on NNS governance, matching the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user and protocol harm on NNS.**

## Likelihood Explanation
Any principal holding a neuron with stake ≥ `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS × 0.01 ICP` can execute this attack. No privileged access is required; no dissolve delay is required for `ManageNeuron` proposals (L5215–5216). The attack is cheap, fully automated, and can be sustained indefinitely by topping up the neuron after each voting period expires and re-filling the queue. [6](#0-5) 

## Recommendation
1. **Introduce a per-neuron open-proposal sub-limit** for `ManageNeuron` proposals (e.g., cap each proposer neuron at N concurrent open proposals) so a single actor cannot exhaust the global quota.
2. **Increase `neuron_management_fee_per_proposal_e8s`** to a level that makes filling the global queue economically prohibitive relative to `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS`.
3. **Add a per-neuron cooldown** between consecutive `ManageNeuron` proposal submissions to rate-limit burst filling.

## Proof of Concept
1. Create or obtain a neuron with stake ≥ `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS × 0.01 ICP` (no dissolve delay required).
2. In a loop, call `make_proposal` with `Action::ManageNeuron` (e.g., `Command::Follow` on `Topic::NeuronManagement` targeting the attacker's own neuron). Do not vote on any submitted proposal.
3. Each call succeeds, deducts 0.01 ICP via `neuron_fees_e8s`, and increments the global open-`ManageNeuron` counter.
4. After `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS` submissions, every `make_proposal` call from any principal with a `ManageNeuron` action returns `ErrorType::ResourceExhausted`.
5. After the voting period expires, the attacker immediately re-fills the queue, sustaining the DoS indefinitely.
6. A deterministic PocketIC integration test can reproduce this by submitting proposals in a loop until `ResourceExhausted` is returned, then verifying that a separate test neuron's `ManageNeuron` proposal is also rejected.
