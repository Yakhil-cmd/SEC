Audit Report

## Title
Byzantine Node Bypasses BLS12-381 Share Validation to Force O(n·pairing) Fallback on Every vetKD Combine — (`rs/crypto/src/vetkd/mod.rs`)

## Summary

`verify_encrypted_key_share_internal` performs only an Ed25519 basic-signature check over the raw share bytes and never calls `EncryptedKeyShare::is_valid(...)`. A single Byzantine subnet member can craft a share with valid curve-point encodings, a wrong cryptographic relationship, and a valid Ed25519 signature, causing the share to pass admission and persistently force the expensive `combine_valid_shares` fallback on every `vetkd_derive_key` request for the duration of the node's participation.

## Finding Description

`verify_encrypted_key_share_internal` (lines 272–302 of `rs/crypto/src/vetkd/mod.rs`) reads the registry version from the threshold-sig data store and then performs only:

```rust
let signature = BasicSigOf::new(BasicSig(key_share.node_signature.clone()));
BasicSigVerifierInternal::verify_basic_sig(
    csp_signer, registry, &signature,
    &key_share.encrypted_key_share, signer, registry_version_from_store,
)
.map_err(VetKdKeyShareVerificationError::VerificationError)
```

It never calls `EncryptedKeyShare::is_valid(master_pk, node_pk, context, input, tpk)`. A Byzantine node can therefore produce a share whose three curve points `(c1, c2, c3)` satisfy `e(c1, g2) == e(g1, c2)` (first pairing check passes) but violate `e(c3, g2) == e(tpk, c2) * e(msg, dpk_i)` (second check fails), sign the 192 serialized bytes with its registered Ed25519 key, and broadcast the `VetKdKeyShare`. Every honest node's call to `verify_encrypted_key_share` succeeds, and the share enters the validated pool.

`combine_encrypted_key_shares_internal` (lines 373–430) then:
1. **Fast path** — calls `EncryptedKey::combine_all`, which interpolates all shares and calls `is_valid` once on the result. One poisoned share makes the interpolated result invalid → `InvalidShares`.
2. **Slow fallback** — on `InvalidShares`, falls back to `EncryptedKey::combine_valid_shares`, which calls `node_eks.is_valid(...)` (two multi-pairings per node) for every node until `reconstruction_threshold` valid shares are accumulated.

The fallback is confirmed by the existing test at `rs/crypto/tests/vetkd.rs` lines 396–399, which asserts the log message `"EncryptedKey::combine_all failed with InvalidShares, falling back to EncryptedKey::combine_valid_shares"` is emitted whenever any share is cryptographically invalid but structurally well-formed.

The `swap_share_c1c3` helper in the test suite (lines 289–294) demonstrates exactly this class of share: swapping two valid G1 points produces a deserializable but cryptographically invalid share. A Byzantine node does the same but re-signs with its own Ed25519 key, making the share indistinguishable from a valid one at the admission layer.

## Impact Explanation

For every `vetkd_derive_key` request while the Byzantine node is active, the combiner performs O(n) BLS12-381 multi-pairing operations (two pairings per node) instead of O(1). On a 13-node subnet (threshold 9), this is approximately 26 pairing operations (~1–2 ms each) per request instead of ~4, a roughly 6× slowdown on the combine step. Because `combine_encrypted_key_shares` is called in the consensus hot path for every vetKD response, this constitutes a persistent, targeted degradation of subnet-wide vetKD throughput — a limited subnet availability impact matching the **Medium** bounty tier ($200–$2,000).

## Likelihood Explanation

The precondition is one legitimate subnet member behaving maliciously — below the Byzantine fault threshold. No privileged access beyond being a registered subnet node is required. The crafted share is indistinguishable from a valid share at the admission layer. The attack is persistent and repeatable across all concurrent vetKD requests for the duration of the Byzantine node's participation.

## Recommendation

`verify_encrypted_key_share_internal` should also validate the cryptographic content of the share by calling `EncryptedKeyShare::is_valid(master_pk, node_pk, context, input, tpk)`. The individual node public key is already derivable via `lazily_calculated_public_key_from_store` (used in `combine_encrypted_key_shares_internal`), and the master public key and context are derivable from the NI-DKG transcript data in `lockable_threshold_sig_data_store`. Adding this check to the admission layer ensures cryptographically invalid shares are rejected before entering the pool and never reach `combine_encrypted_key_shares`.

## Proof of Concept

1. Byzantine node generates three valid BLS12-381 curve points `(c1, c2, c3)` satisfying `e(c1, g2) == e(g1, c2)` but violating `e(c3, g2) == e(tpk, c2) * e(msg, dpk_i)` (e.g., by swapping c1 and c3 as in `swap_share_c1c3`).
2. Node serializes these 192 bytes and signs with its registered Ed25519 node key.
3. Every honest node calls `verify_encrypted_key_share` → passes (Ed25519 only).
4. Share enters the validated pool.
5. `combine_encrypted_key_shares_internal` calls `combine_all` → `InvalidShares`.
6. Fallback to `combine_valid_shares` runs O(n) pairing checks.
7. **Deterministic integration test**: adapt `should_succeed_if_reconstruction_threshold_many_shares_are_valid` in `rs/crypto/tests/vetkd.rs` — corrupt exactly one share using `swap_share_c1c3` and re-sign it with the corresponding node's Ed25519 key; assert the fallback log message is emitted and measure latency increase proportional to n·pairing\_cost repeating on every request.
