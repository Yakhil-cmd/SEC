### Title
Uninitialized `end_timestamp_seconds` in SNS `RewardEvent` Inflates Rewards Purse on Upgrade - (`rs/sns/governance/src/governance.rs`)

---

### Summary

SNS governance's `distribute_rewards` uses `end_timestamp_seconds.unwrap_or_default()` (defaulting to 0) as the reward-period start anchor. `Governance::new()` only seeds a proper `end_timestamp_seconds: Some(now)` when `latest_reward_event` is entirely `None`. Any SNS whose stored proto already contains a `RewardEvent` with `end_timestamp_seconds: None` (i.e., deployed before that field was introduced) survives the upgrade untouched. On the very next periodic-task heartbeat, `distribute_rewards` treats Unix epoch 0 as the last reward boundary, computes ~20 000 elapsed rounds, and distributes a rewards purse that is several multiples of the entire token supply.

---

### Finding Description

**Root cause — `Governance::new()` does not back-fill `end_timestamp_seconds`** [1](#0-0) 

The guard is `if proto.latest_reward_event.is_none()`. When the deserialized proto already holds `Some(RewardEvent { end_timestamp_seconds: None, … })` — the state of every SNS deployed before `end_timestamp_seconds` was added — the block is skipped entirely and the `None` persists.

**`canister_post_upgrade` calls `canister_init_()` → `Governance::new()`** [2](#0-1) 

The restored proto is passed straight through `canister_init_()` into `Governance::new()`, so the same non-fix applies on every upgrade.

**`should_distribute_rewards` fires immediately** [3](#0-2) 

With `end_timestamp_seconds = None`, `unwrap_or_default()` yields 0, so `seconds_since_last_reward_event ≈ now ≈ 1 750 000 000 s`, which is far larger than any `round_duration_seconds`. The function returns `true` on the first heartbeat after upgrade.

**`distribute_rewards` computes ~20 000 rounds** [4](#0-3) 

`reward_start_timestamp_seconds = 0`, so:

```
new_rounds_count = now / round_duration_seconds
                 ≈ 1_750_000_000 / 86_400
                 ≈ 20 254   (for a daily-round SNS)
```

**Rewards purse loop accumulates 20 000+ rounds** [5](#0-4) 

For rounds where `round_duration_seconds * i < genesis_timestamp_seconds`, `saturating_sub` clamps `seconds_since_genesis` to 0, so `reward_rate_at(0)` returns the maximum initial rate. Every one of the ~20 254 iterations adds a full round's reward to the purse.

Approximate purse with a 10 %/year initial rate and daily rounds:

```
20 254 rounds × (10% / 365.25) × supply ≈ 5.55 × supply
```

The overflow guard only fires if the purse exceeds `u64::MAX`; for any SNS with a supply below ~33 billion tokens (in e8s) the purse fits in a `u64` and is distributed in full. [6](#0-5) 

---

### Impact Explanation

All neurons that voted on any settled proposal in the inflated reward round receive maturity proportional to the bloated purse. Maturity can be converted to governance tokens via the spawn-neuron path, permanently inflating the SNS token supply by up to several multiples of its original size. This breaks ledger conservation and destroys the economic model of every affected SNS.

---

### Likelihood Explanation

`end_timestamp_seconds` was introduced as an `optional` field (tag 5) to replace the deprecated `round` field (tag 1) in `RewardEvent`. Any SNS launched before that proto change has a persisted `latest_reward_event` with `end_timestamp_seconds: None`. Because `canister_post_upgrade` calls `Governance::new()` with the raw deserialized proto, and `Governance::new()` only patches the event when it is entirely absent, the first upgrade of any such SNS to the current codebase silently leaves the `None` in place. The periodic-task timer then fires automatically — no user action required — and triggers the inflated distribution.

---

### Recommendation

In `Governance::new()`, extend the initialization guard to also back-fill `end_timestamp_seconds` when it is `None` inside an existing `latest_reward_event`:

```rust
if proto.latest_reward_event.is_none() {
    proto.latest_reward_event = Some(RewardEvent {
        end_timestamp_seconds: Some(now),
        ...
    });
} else if let Some(ref mut event) = proto.latest_reward_event {
    if event.end_timestamp_seconds.is_none() {
        // Back-fill for SNS instances deployed before this field existed.
        // Use genesis_timestamp_seconds as a safe lower bound so that
        // new_rounds_count is computed from genesis, not from Unix epoch 0.
        event.end_timestamp_seconds = Some(proto.genesis_timestamp_seconds);
    }
}
```

Alternatively, replace every `end_timestamp_seconds.unwrap_or_default()` call with `end_timestamp_seconds.unwrap_or(self.proto.genesis_timestamp_seconds)` so that a missing value is treated as genesis rather than epoch 0.

---

### Proof of Concept

1. Deploy an SNS with a `latest_reward_event` whose `end_timestamp_seconds` is `None` (any SNS predating the field addition, or constructed directly with `RewardEvent { round: 1, end_timestamp_seconds: None, … }`).
2. Upgrade the SNS governance canister to the current code. `canister_post_upgrade` deserializes the proto and calls `Governance::new()`; the `is_none()` guard is `false`, so `end_timestamp_seconds` stays `None`.
3. Wait for (or manually trigger in a test) `run_periodic_tasks`. `should_distribute_rewards` returns `true` because `now - 0 >> round_duration_seconds`.
4. `distribute_rewards` computes `new_rounds_count ≈ now / round_duration_seconds ≈ 20 254` and loops that many times, accumulating a rewards purse ≈ 5.55 × total_supply.
5. All voting neurons receive maturity proportional to this inflated purse; spawning those neurons mints tokens, permanently inflating the supply. [1](#0-0) [4](#0-3) [2](#0-1)
