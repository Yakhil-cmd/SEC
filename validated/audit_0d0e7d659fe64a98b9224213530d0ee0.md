### Title
ERC-20 Tokens Permanently Burned When `ft_transfer` Fails Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary
When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile schedules a bare `PromiseArgs::Create` (no callback) for the NEP-141 `ft_transfer`. Because `EvmErc20.withdrawToNear()` burns the caller's ERC-20 tokens **before** the promise is dispatched, a failed `ft_transfer` (e.g., unregistered recipient) results in the ERC-20 tokens being permanently destroyed with no corresponding NEP-141 tokens ever received. There is no refund path.

### Finding Description

**Step 1 – ERC-20 burn is unconditional and immediate.**

`EvmErc20.withdrawToNear()` burns the caller's tokens first, then calls the precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // <-- irreversible burn
    // ... calls ExitToNear precompile
}
``` [1](#0-0) 

**Step 2 – Refund args are gated behind `error_refund` feature.**

Inside `ExitToNear::run()`, the `refund` field of the callback args is set to `None` when the feature is absent:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

**Step 3 – Without refund args, no callback is attached.**

For the standard ERC-20 `ft_transfer` path, `transfer_near` is also `None`, so `callback_args == ExitToNearPrecompileCallbackArgs::default()`. The branch at line 470 then creates a bare `PromiseArgs::Create` with **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback, no refund
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

**Step 4 – The callback that would refund is never scheduled.**

`exit_to_near_precompile_callback` is the only place `refund_on_error` is called. Without a callback promise, this function is never invoked, so no re-mint of burned ERC-20 tokens occurs: [4](#0-3) 

**Step 5 – The test explicitly confirms permanent loss.**

The integration test acknowledges that without `error_refund`, the user's tokens are gone:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

The same pattern applies to ETH exits: [6](#0-5) 

### Impact Explanation

**Impact: Critical – Permanent freezing / theft of user funds.**

When `error_refund` is not compiled in and a user calls `withdrawToNear()` targeting an unregistered NEAR account (or any account that causes `ft_transfer` to fail), their ERC-20 tokens are burned on Aurora with no NEP-141 tokens ever credited on NEAR. The tokens are permanently destroyed. This is a direct, irreversible loss of user funds reachable by any unprivileged EVM user.

### Likelihood Explanation

**Likelihood: Medium.**

Any user can trigger this by specifying a NEAR account that is not registered with the NEP-141 token contract as the `recipient`. This is a common mistake (e.g., a freshly created account that has not called `storage_deposit`). The condition is easily reachable without any special privileges. The severity is conditional on the `error_refund` feature being absent from the production binary.

### Recommendation

1. Always compile production binaries with the `error_refund` feature enabled, and enforce this in CI/CD.
2. Migrate all deployed ERC-20 contracts to `EvmErc20V2`, which encodes the `sender` address in the precompile calldata so the refund address is always available.
3. Consider making the refund path unconditional (not feature-gated) so it cannot be accidentally omitted.

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to an ERC-20 on Aurora.
2. Call `erc20.withdrawToNear("unregistered.near", amount)` from an EVM account that holds tokens.
3. The ERC-20 tokens are burned immediately (confirmed by `_burn` in `EvmErc20.sol` line 54).
4. The `ft_transfer` promise to the NEP-141 contract fails because `unregistered.near` has no storage deposit.
5. Without `error_refund`, no callback fires, no re-mint occurs.
6. Result: ERC-20 balance decremented, NEP-141 balance of `unregistered.near` unchanged, tokens permanently lost.

This is directly confirmed by the existing test `test_exit_to_near_refund` which asserts `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` (i.e., tokens are gone) when the feature is absent. [7](#0-6) [1](#0-0) [8](#0-7)
