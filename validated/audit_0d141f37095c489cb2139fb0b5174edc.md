Audit Report

## Title
Stale ICP/XDR Exchange Rate Consumed Without Freshness Check in Cycles Minting Canister — (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles` function in the CMC reads the cached `icp_xdr_conversion_rate` and uses it for all cycle minting operations without ever checking whether the rate's `timestamp_seconds` is recent relative to the current canister time. When the Exchange Rate Canister (XRC) is unavailable, the CMC silently continues minting cycles at the last successfully cached rate indefinitely. An unprivileged attacker who observes an XRC outage coinciding with a drop in ICP market price can exploit the stale (higher) rate to receive more cycles per ICP than the current market rate warrants, constituting illegal minting at the protocol level.

## Finding Description
`tokens_to_cycles` (lines 1900–1922 of `rs/nns/cmc/src/main.rs`) extracts only `xdr_permyriad_per_icp` from the cached rate option, discarding `timestamp_seconds` entirely:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);   // timestamp_seconds never read
```

The setter `do_set_icp_xdr_conversion_rate` (lines 1022–1030) enforces only monotonicity (new timestamp > stored timestamp) and a non-zero value; it does not enforce that the rate is recent relative to wall-clock time. There is no `MAX_RATE_AGE` constant or equivalent guard anywhere in the CMC codebase.

The heartbeat calls `update_exchange_rate` every five minutes (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`, `exchange_rate_canister.rs` line 16). On XRC failure, the guard schedules a retry one minute later (`exchange_rate_canister.rs` lines 153–160) but leaves the stale rate in place. All three minting entry points — `process_create_canister` (line 1932), `process_mint_cycles` (line 1965), and `process_top_up` (line 1991) — call `tokens_to_cycles` unconditionally, so every minting operation during an XRC outage uses the stale rate.

**Exploit flow:**
1. XRC becomes unavailable (its HTTPS outcalls to external forex/crypto endpoints fail with e.g. `CryptoBaseAssetNotFound` or `StablecoinRateTooFewRates`).
2. CMC heartbeat fails to update `icp_xdr_conversion_rate`; the cached rate's `timestamp_seconds` stops advancing.
3. ICP market price falls below the stale cached rate (or the attacker times the attack during a known price drop coinciding with the outage).
4. Attacker calls `notify_top_up` or `notify_mint_cycles` with an ICP ledger block burning ICP at the current lower market price.
5. `tokens_to_cycles` reads the stale (higher) `xdr_permyriad_per_icp` and mints more cycles than the burned ICP is worth at current market rates.
6. Attacker receives excess cycles; the ICP burned is worth less than the cycles received.

No privileged access is required; `notify_top_up` and `notify_mint_cycles` are open to any principal.

## Impact Explanation
This is an illegal minting vulnerability: an unprivileged user can obtain cycles in excess of the ICP value they burned, breaking the protocol's economic invariant that 1 ICP always converts to cycles at the current market rate. The magnitude scales with the size of the price drop and the volume of ICP converted during the outage window. This matches the allowed High impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm" and potentially the Critical class ("Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles") depending on the price drop magnitude and attacker capital.

## Likelihood Explanation
The XRC is a system canister that performs HTTPS outcalls to external exchange and forex APIs. Outages of those external sources (exchange API downtime, broad network disruption) are realistic and historically observed. The attacker does not need to cause the outage — they only need to observe it (publicly visible by calling the XRC directly or monitoring replica logs) and act while ICP price is below the stale cached rate. The attack requires no special privileges, is repeatable for the duration of the outage, and the only constraint is having ICP to convert.

## Recommendation
Add a maximum-age check in `tokens_to_cycles` before using the cached rate. If `now - rate.timestamp_seconds` exceeds a defined threshold (e.g., 30–60 minutes, giving the heartbeat several retry windows), return an error rather than use the stale rate:

```rust
const MAX_RATE_AGE_SECONDS: u64 = 30 * 60; // 30 minutes

fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let now = ic_cdk::api::time() / 1_000_000_000;
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) if now.saturating_sub(rate.timestamp_seconds) <= MAX_RATE_AGE_SECONDS => {
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            Some(_) => Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "ICP/XDR conversion rate is stale; minting suspended".to_string(),
            }),
            None => Err(/* existing None error */),
        }
    })
}
```

## Proof of Concept
1. Deploy a local replica with the CMC and a mock XRC that returns `CryptoBaseAssetNotFound`.
2. Advance canister time past `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes) to trigger a heartbeat that fails to update the rate.
3. Confirm via `get_icp_xdr_conversion_rate` that the cached rate's `timestamp_seconds` is no longer advancing.
4. Submit an ICP ledger block burning ICP and call `notify_top_up` on the CMC.
5. Observe that `tokens_to_cycles` uses the stale rate and mints cycles at the pre-outage (higher) `xdr_permyriad_per_icp` value.
6. Compare minted cycles against what the current (lower) market rate would have produced to quantify the excess.
