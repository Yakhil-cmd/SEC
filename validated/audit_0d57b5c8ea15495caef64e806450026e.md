### Title
SNS Governance `refresh_neuron` Missing Neuron Lock Before Async Ledger Call Enables Cached-Stake Inflation via Concurrent Disburse — (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `refresh_neuron` function does not acquire a neuron lock before making an async inter-canister call to the ledger to query the account balance. During the await, another `manage_neuron` message (e.g., `Disburse`) can be processed by the canister, modifying the neuron's actual ledger balance. When `refresh_neuron` resumes, it uses the stale pre-disbursal balance to overwrite `cached_neuron_stake_e8s`, creating a detached state where the neuron's cached stake is higher than the actual ledger balance — inflating voting power without corresponding ICP backing.

This is the direct IC analog of the ENS NameWrapper reentrancy: in ENS, a transfer callback fires between the transfer and the fuse/owner update, allowing state to be manipulated so the ERC1155 token is detached from actual ENS ownership. In IC, the inter-canister `await` is the callback window, and the detachment is between `cached_neuron_stake_e8s` and the real ledger balance.

---

### Finding Description

In `rs/sns/governance/src/governance.rs`, `refresh_neuron` executes as follows: [1](#0-0) 

```rust
async fn refresh_neuron(&mut self, nid: &NeuronId) -> Result<(), GovernanceError> {
    // ... checks neurons_fund_controlled (no lock acquired) ...
    
    // AWAIT POINT — neuron is NOT in neurons_with_ongoing_operations
    let balance = self.ledger.account_balance(account).await?;

    // After await — stale balance used to update cached stake
    let neuron = self.get_neuron_result_mut(nid)?;
    neuron.update_stake(balance.get_e8s(), now);
```

No neuron lock is acquired before the `await`. This directly contrasts with the NNS governance `refresh_neuron`, which explicitly acquires `lock_neuron_for_command` before the ledger call, with the comment: [2](#0-1) 

> "We need to lock the neuron to make sure it doesn't undergo concurrent changes while we're checking the balance and refreshing the stake."

The SNS `claim_neuron` also documents the same rationale for locking: [3](#0-2) 

> "We need to create the neuron before checking the balance so that we record the neuron and add it to the set of neurons with ongoing operations. This avoids a race where a user calls this method a second time before the first time responds."

`claim_neuron` calls `add_neuron` which places the neuron into `neurons_with_ongoing_operations`, blocking concurrent mutations. `refresh_neuron` does neither — it does not call `add_neuron` and does not call `lock_neuron_for_command`. [4](#0-3) 

The call chain is: [5](#0-4) 

`manage_neuron` → `claim_or_refresh_neuron_by_memo_and_controller` → `refresh_neuron` (no lock at any level).

---

### Impact Explanation

The IC execution model allows a canister to process other incoming update messages while suspended at an `await` point. If `refresh_neuron` is awaiting `ledger.account_balance`, the SNS governance canister can concurrently process a `Disburse` command for the same neuron. Because the neuron is absent from `neurons_with_ongoing_operations`, `Disburse` proceeds, transfers ICP out of the neuron's subaccount, and reduces the actual ledger balance.

When `refresh_neuron` resumes, it reads the pre-disbursal balance and writes it into `cached_neuron_stake_e8s`. The neuron now has:
- `cached_neuron_stake_e8s` = 100 ICP (stale)
- Actual ledger subaccount balance = ~0 ICP (post-disbursal)

`cached_neuron_stake_e8s` drives voting power in SNS governance proposals. An attacker can thus obtain inflated voting power without ICP backing, potentially manipulating SNS governance outcomes (treasury withdrawals, parameter changes, upgrade proposals). [6](#0-5) 

---

### Likelihood Explanation

The attacker controls both calls and can submit them in the same ingress batch. The IC scheduler will process `refresh_neuron` first (suspending at the ledger `await`), then process `Disburse` during the suspension window, then resume `refresh_neuron` with the stale balance. No privileged access, no threshold corruption, no social engineering is required — only two ordinary `manage_neuron` update calls from the neuron's controller.

---

### Recommendation

Acquire a neuron lock in SNS governance `refresh_neuron` before the ledger `await`, mirroring the NNS governance implementation:

```rust
async fn refresh_neuron(&mut self, nid: &NeuronId) -> Result<(), GovernanceError> {
    // Add neuron to neurons_with_ongoing_operations before the await
    let _neuron_lock = self.add_neuron_to_ongoing_operations(nid, ...)?;
    
    let balance = self.ledger.account_balance(account).await?;
    // ... rest of function
}
```

Alternatively, re-read and validate the neuron's state after the `await` to detect concurrent mutations (e.g., verify the neuron still exists and its dissolve state is consistent before applying the balance update).

### Proof of Concept

1. Attacker controls SNS neuron N with 100 ICP staked (`cached_neuron_stake_e8s = 100e8`).
2. Attacker sends `manage_neuron { ClaimOrRefresh { by: NeuronIdOrSubaccount } }` → triggers `refresh_neuron` → canister suspends at `ledger.account_balance(account).await`.
3. During the suspension, attacker sends `manage_neuron { Disburse { amount: 100 ICP, to: attacker_account } }` → neuron is NOT in `neurons_with_ongoing_operations` → `Disburse` proceeds → 100 ICP transferred out → ledger subaccount balance ≈ 0.
4. `refresh_neuron` resumes with the pre-disbursal balance of 100e8 → calls `neuron.update_stake(100e8, now)`.
5. Result: neuron has `cached_neuron_stake_e8s = 100e8` but actual ledger balance ≈ 0. Attacker holds 100 ICP in their account AND retains a neuron with 100 ICP of voting power.
6. Attacker uses inflated voting power to pass SNS proposals (e.g., treasury drain, parameter manipulation). [7](#0-6) [8](#0-7)
