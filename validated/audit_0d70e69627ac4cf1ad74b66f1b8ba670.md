Audit Report

## Title
Byzantine Subnet Peer Memory Exhaustion via Unbounded `read_to_end` Across Concurrent QUIC Streams — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
The `start_stream_acceptor` event loop in `request_handler.rs` spawns an unbounded `JoinSet` task for every accepted bidirectional QUIC stream, with each task calling `read_to_end(MAX_MESSAGE_SIZE_BYTES)` (128 MiB cap). Because QUIC flow-control windows pace delivery but do not cap total heap accumulation — the application advancing the window as it reads — a TLS-authenticated Byzantine subnet peer can open up to 1,000 concurrent streams and stream 128 MiB on each, forcing the victim replica to accumulate up to 128 GiB of heap, triggering an OOM kill and dropping the replica from consensus.

## Finding Description

**Confirmed constants** (`connection_manager.rs:71-74`):
- `RECEIVE_WINDOW` = 200 MB (connection-level)
- `STREAM_RECEIVE_WINDOW` = 4 MB (per-stream)
- `MAX_CONCURRENT_BIDI_STREAMS` = 1,000
- `MAX_MESSAGE_SIZE_BYTES` = 128 MiB (`lib.rs:74`)

**Root cause — unbounded task spawning** (`request_handler.rs:50`):

```rust
let mut inflight_requests: JoinSet<Result<(), P2PError>> = tokio::task::JoinSet::new();
```

The event loop at `request_handler.rs:72-90` calls `conn_handle.conn().accept_bi()` and immediately spawns into `inflight_requests` with no concurrency cap. The developers acknowledge this in the comment at lines 52-56:

> "The extreme result of a slow handler is that the stream limit will be reached, hence having buffered up to the stream limit number of messages/requests."

However, the comment understates the impact: the concern is not just the stream count but the per-stream allocation ceiling.

**Per-task allocation** (`request_handler.rs:173-175`):

```rust
let request_bytes = recv_stream
    .read_to_end(MAX_MESSAGE_SIZE_BYTES)
    .await
```

`quinn::RecvStream::read_to_end` allocates a `Vec<u8>` that grows continuously as data arrives. As bytes are consumed from the QUIC receive buffer into the `Vec`, quinn automatically advances the flow-control window, granting the peer permission to send more. The `STREAM_RECEIVE_WINDOW` (4 MB) and `RECEIVE_WINDOW` (200 MB) therefore pace delivery but do not cap the total bytes accumulated across all concurrently live `Vec<u8>` buffers.

**Ceiling calculation:**
```
1,000 streams × 128 MiB/stream = 128 GiB
```

**Why existing guards are insufficient:**
- `MAX_CONCURRENT_BIDI_STREAMS = 1,000` is a QUIC transport ceiling, not an application-level memory budget.
- `STREAM_RECEIVE_WINDOW = 4 MB` limits instantaneous in-flight bytes per stream, but the window advances as `read_to_end` drains it, allowing the full 128 MiB to be delivered per stream over time.
- `RECEIVE_WINDOW = 200 MB` similarly paces the connection but does not bound total heap across all streams.
- There is no semaphore, permit pool, or per-connection memory budget guarding `inflight_requests`.

**Secondary outbound path** (`connection_handle.rs:115-117`): The same `read_to_end(MAX_MESSAGE_SIZE_BYTES)` pattern exists in `ConnectionHandle::rpc()`. A Byzantine peer can delay response bodies on all open RPC streams and flush 128 MiB simultaneously, though this path requires the victim's upper-layer protocols to have ~1,000 concurrent RPCs outstanding to the same peer.

## Impact Explanation

A single compromised subnet replica (below the `f < n/3` Byzantine fault threshold) can crash the victim replica process via OOM. The victim drops out of consensus; if the attack is repeated against additional replicas, subnet finalization stalls. This matches the **High** severity impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attacker must hold a valid TLS certificate for a subnet node — i.e., be a compromised subnet replica. This is a meaningful prerequisite but is explicitly within the "protocol peer behavior below the consensus fault threshold" scope. The attack requires no special timing, race conditions, or victim interaction. The 200 MB connection window paces delivery but does not prevent the attack; the attacker simply streams data continuously across all 1,000 streams. The attack is repeatable and requires no governance action after the initial node compromise.

## Recommendation

1. **Cap concurrent in-flight stream tasks.** Replace the unbounded `JoinSet` with a `tokio::sync::Semaphore`-guarded pool. Only call `accept_bi()` when a permit is available, so QUIC back-pressure naturally limits the Byzantine peer without requiring application-level changes to handlers.

2. **Enforce a per-connection memory budget.** Track total bytes currently held in `read_to_end` buffers for a given peer connection and close the connection if the budget is exceeded (e.g., 2× `RECEIVE_WINDOW` = 400 MB).

3. **Read a length-prefix before allocating the full buffer.** Reject streams that declare an oversized body before allocating the full `Vec<u8>`, rather than discovering the size limit only after buffering.

4. **Decouple `MAX_CONCURRENT_BIDI_STREAMS` from application concurrency.** The QUIC limit (1,000) should be a transport-layer ceiling; the application-level concurrency limit should be set independently and much lower.

## Proof of Concept

A turmoil-based state-machine test:

1. Spin up a victim `QuicTransport` endpoint with the production constants from `connection_manager.rs`.
2. Create a mock Byzantine peer that completes mutual TLS with a valid (test) certificate accepted by the victim's `SomeOrAllNodes` server config.
3. From the Byzantine peer, open 1,000 bidirectional streams via `connection.open_bi()`.
4. On each stream, write exactly 128 MiB of arbitrary bytes and call `finish()`.
5. Poll the victim process RSS (via `/proc/self/status` or `jemalloc` stats) after all streams are accepted by the victim's `start_stream_acceptor` loop.
6. Assert RSS remains below a safe bound (e.g., 2 GiB).

Expected result without a fix: RSS grows toward the OS OOM threshold and the victim process is killed before the assertion is reached, demonstrating the vulnerability.
