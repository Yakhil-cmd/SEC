### Title
Missing Slippage Protection in SNS Treasury Manager Deposit/Withdraw API - (File: `rs/sns/treasury_manager/treasury_manager.did`, `rs/sns/governance/src/extensions.rs`)

### Summary
The SNS Treasury Manager API (`deposit` and `withdraw`) does not include a `min_amount_out` or equivalent slippage guard. The `DepositRequest` carries only `allowances` (how much to spend), and `WithdrawRequest` carries only optional `withdraw_accounts`. Neither carries a minimum acceptable return. The codebase itself acknowledges this risk in a comment, yet the API does not enforce any protection. When a Treasury Manager implementation deposits SNS/ICP tokens into a DEX liquidity pool, a malicious actor can front-run the governance-approved transaction to manipulate the pool price, causing the SNS treasury to receive far fewer LP shares than expected.

### Finding Description

The `TreasuryManager` API is defined in `rs/sns/treasury_manager/treasury_manager.did` and `rs/sns/treasury_manager/src/lib.rs`. The `DepositRequest` type contains only `allowances`:

```
type DepositRequest = record {
  allowances : vec Allowance;
};
```

and `WithdrawRequest` contains only optional destination accounts:

```
type WithdrawRequest = record {
  withdraw_accounts : opt vec record { principal; Account };
};
```

Neither type includes a `min_amount_out`, `min_lp_tokens`, or any slippage bound.

The file `rs/sns/treasury_manager/treasury_manager.did` lines 35–40 explicitly acknowledges the risk:

> "Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."

The same warning is reproduced verbatim in the governance proposal rendering at `rs/sns/governance/src/proposal.rs` lines 1540–1545, shown to voters when a `RegisterExtension` proposal is submitted. However, the warning is only informational — no enforcement exists in the API contract itself.

The execution path is:
1. An SNS governance proposal of type `ExecuteExtensionOperation` with `operation_name = "deposit"` is submitted and passes.
2. `execute_treasury_manager_deposit` in `rs/sns/governance/src/extensions.rs` (lines 1546–1610) calls `governance.approve_treasury_manager(...)` to grant ICRC-2 allowances, then calls `extension_canister_id.deposit(arg_blob)`.
3. The Treasury Manager implementation (e.g., a KongSwap adaptor) uses those allowances to deposit into a DEX.
4. No minimum LP token return is checked anywhere in the governance layer or the API contract.

For withdrawals, `execute_treasury_manager_withdraw` (lines 1612–1661) similarly calls `extension_canister_id.withdraw(arg_blob)` with no minimum return amount enforced.

The validation step `validate_deposit_operation_impl` (lines 276–321) only checks that the requested amount does not exceed 50% of the current treasury balance — it does not check what the treasury will receive in return.

### Impact Explanation

An attacker who observes a pending governance-approved deposit proposal can front-run the `deposit` call (which is a public update call on the IC) by manipulating the DEX pool state (e.g., large swap to skew the price). The SNS treasury then deposits at a severely unfavorable ratio, receiving far fewer LP shares than the governance voters expected. The lost value is permanently transferred to the attacker via arbitrage. For withdrawals, the same front-running can cause the treasury to receive far fewer tokens than expected when redeeming LP shares. Since SNS treasury funds are DAO-owned assets, this directly harms all SNS token holders.

**Impact: Medium** — treasury funds are partially lost (not fully drained), but the loss can be significant depending on pool liquidity and deposit size.

### Likelihood Explanation

**Likelihood: Medium** — The IC does not have a public mempool in the traditional sense, but governance proposal execution is publicly observable (proposals are on-chain and their execution timing is predictable). An attacker monitoring the SNS governance canister can observe when a deposit proposal is adopted and about to execute, then submit a front-running transaction to the DEX canister in the same or prior round. The IC's deterministic execution order within a subnet means that inter-canister calls from governance to the Treasury Manager to the DEX are sequenced, and a sufficiently motivated attacker with knowledge of the execution schedule can exploit this window.

### Recommendation

1. Add a `min_amount_out` (or `min_lp_tokens_out`) field to `DepositRequest` in `rs/sns/treasury_manager/treasury_manager.did` and `rs/sns/treasury_manager/src/lib.rs`:
   ```
   type DepositRequest = record {
     allowances : vec Allowance;
     min_lp_tokens_out : opt nat;  // revert if LP tokens received < this
   };
   ```
2. Add a `min_tokens_out` field to `WithdrawRequest` similarly.
3. Require Treasury Manager implementations to enforce these bounds when calling the underlying DEX, reverting if the DEX returns fewer tokens than the specified minimum.
4. Require the SNS governance proposal validation (`validate_deposit_operation_impl`) to mandate that `min_lp_tokens_out` is set to a non-zero value, so voters can see and approve the slippage tolerance.

### Proof of Concept

1. An SNS passes a `ExecuteExtensionOperation` proposal to deposit 1000 ICP + 1000 SNS tokens into a KongSwap pool, expecting ~500 LP tokens at current prices.
2. An attacker observes the proposal is adopted and about to execute.
3. The attacker submits a large swap on KongSwap (e.g., dumps SNS tokens) to skew the pool ratio, making SNS tokens worth far less relative to ICP.
4. The governance canister executes `execute_treasury_manager_deposit` → calls `deposit` on the Treasury Manager → Treasury Manager calls KongSwap with the approved allowances.
5. KongSwap accepts the deposit at the manipulated ratio, minting only ~50 LP tokens instead of ~500.
6. The attacker reverses their swap, profiting from the arbitrage.
7. The SNS treasury has permanently lost ~90% of the expected LP value with no revert or error, because neither `DepositRequest` nor the governance execution layer enforces any minimum return.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
