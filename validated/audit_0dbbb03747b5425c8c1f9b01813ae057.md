Audit Report

## Title
Unbounded Body Buffering in `RosettaMetricsMiddleware` Allows Unauthenticated OOM Crash of ICRC-1 Rosetta Process — (File: `rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
`extract_canister_id` in the ICRC-1 Rosetta metrics middleware calls `to_bytes(body, usize::MAX)`, buffering the entire HTTP request body into heap memory with no size limit. This middleware runs unconditionally on every non-`/metrics` request, and the ICRC-1 Axum router applies no upstream body-size guard. A single unauthenticated HTTP POST with an arbitrarily large body can exhaust the process heap and OOM-kill the Rosetta instance.

## Finding Description
In `rs/rosetta-api/common/rosetta_core/src/metrics.rs` at line 389, `extract_canister_id` reads the full request body with no bound:

```rust
let bytes = to_bytes(body, usize::MAX).await?;
```

This is called unconditionally from `RosettaMetricsMiddleware::call` (lines 331–339) for every request that is not `GET /metrics`, before dispatch to any route handler. In `rs/rosetta-api/icrc1/src/main.rs` (lines 361–391), the Axum router applies `metrics_layer` as the outermost layer with no `DefaultBodyLimit`, `RequestBodyLimitLayer`, or any `Content-Length` rejection anywhere in the stack — confirmed by grep returning zero matches for any body-limit construct in the `rs/rosetta-api/icrc1/` tree. By contrast, the ICP Rosetta (Actix-web) enforces a 4 MB `JsonConfig` limit at `rs/rosetta-api/icp/src/rosetta_server.rs` lines 296–303. The ICRC-1 path has no equivalent guard.

## Impact Explanation
This matches **High ($2,000–$10,000): Application/platform-level DoS, crash** — specifically a single-request crash of the ICRC-1 Rosetta process. The Rosetta service is a single-process replica; OOM-killing it takes down the entire ICRC-1 Rosetta instance until an operator restarts it. This is not raw volumetric DDoS — it is one request exploiting an unbounded allocation, which is a distinct and in-scope impact class.

## Likelihood Explanation
The attack requires only a TCP connection to the Rosetta HTTP port and the ability to stream bytes. No credentials, tokens, or prior knowledge are needed beyond the open port. A single POST to any listed endpoint (e.g., `/construction/submit`) with a multi-gigabyte body is sufficient. The Rosetta HTTP port is typically exposed to clients, making this trivially reachable by any unauthenticated external user.

## Recommendation
Add an explicit body-size limit as the outermost layer in the Axum router in `rs/rosetta-api/icrc1/src/main.rs`, before `metrics_layer`:

```rust
use axum::extract::DefaultBodyLimit;

let app = Router::new()
    // ... routes ...
    .layer(metrics_layer)
    .layer(add_request_span())
    .layer(RequestIdLayer)
    .layer(DefaultBodyLimit::max(4 * 1024 * 1024)) // 4 MB, consistent with ICP Rosetta
    .with_state(token_app_states.clone());
```

Alternatively, replace `usize::MAX` with a bounded constant directly in `extract_canister_id` at `rs/rosetta-api/common/rosetta_core/src/metrics.rs` line 389, so the middleware is safe regardless of router configuration.

## Proof of Concept
```bash
# Stream a 2 GB body to any Rosetta endpoint; observe OOM kill before any 413 is returned
dd if=/dev/zero bs=1M count=2048 | \
  curl -X POST http://<rosetta-host>:8080/construction/submit \
       -H "Content-Type: application/json" \
       --data-binary @- \
       -v
# Confirm OOM kill:
dmesg | grep -i oom
```

A deterministic integration test can also be written: spin up the ICRC-1 Rosetta server locally, POST a body of e.g. 500 MB, and assert the process exits with an OOM signal before returning any HTTP response.
