Audit Report

## Title
Unauthorized SNS Swap Participation Registration via Arbitrary `buyer` Parameter Bypasses Confirmation-Text Consent Mechanism - (File: `rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` update endpoint in the SNS Swap canister resolves the effective buyer principal from the request payload rather than from the authenticated caller. When `arg.buyer` is non-empty, no check is performed that the caller equals the supplied principal. An unprivileged attacker can read the public `confirmation_text` via `get_init`, then call `refresh_buyer_tokens` with a victim's principal and the correct confirmation text, registering the victim as a committed swap participant without their explicit consent and bypassing the only consent gate the SNS provides.

## Finding Description

In `rs/sns/swap/canister/canister.rs` at L130–134, the handler unconditionally trusts the `buyer` field when it is non-empty:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
};
``` [1](#0-0) 

The resolved principal `p` and the caller-supplied `confirmation_text` are then forwarded directly to `refresh_buyer_token_e8s`: [2](#0-1) 

Inside `refresh_buyer_token_e8s`, `validate_confirmation_text` only checks that the supplied text matches the SNS init config — it performs no check that the caller is the buyer: [3](#0-2) 

The function then reads the ICP balance from the subaccount derived from the attacker-supplied `buyer` principal and records that principal as a committed participant: [4](#0-3) 

The `confirmation_text` is the sole consent gate for SNS swaps that configure one. Because it is publicly readable via the `get_init` query endpoint and is validated only for string equality (not for caller identity), any unprivileged ingress sender can supply it on behalf of any victim.

## Impact Explanation

This matches the **High** impact category: *Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.*

When a victim has transferred ICP to the swap subaccount (the normal first step of participation) but has not yet called `refresh_buyer_tokens`, an attacker can register the victim as a committed buyer. If the swap commits, the victim's ICP is swept to the SNS treasury and the victim receives SNS tokens they never explicitly agreed to purchase. The victim cannot reverse this — they must wait for a swap abort (which may never occur) to reclaim ICP via `error_refund_icp`. The `confirmation_text` mechanism, which is the explicit per-user consent gate, is entirely defeated.

## Likelihood Explanation

The precondition — that the victim has already transferred ICP to the swap subaccount — is the normal first step of the participation flow and creates a routine attack window. The attacker requires only: (1) the victim's principal, which is publicly observable on-chain, and (2) the `confirmation_text`, which is readable by anyone via the public `get_init` query. No privileged access, key material, or threshold corruption is required. Any unprivileged ingress sender can execute this attack against any victim who has transferred ICP but not yet called `refresh_buyer_tokens`.

## Recommendation

Enforce that when `arg.buyer` is non-empty, the caller must equal the resolved principal, or alternatively reject any call that supplies both a non-empty `buyer` and a `confirmation_text` when the caller is not the buyer:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let p = PrincipalId::from_str(&arg.buyer).unwrap();
    if arg.confirmation_text.is_some() && p != caller_principal_id() {
        panic!("confirmation_text can only be submitted by the buyer themselves");
    }
    p
};
```

Alternatively, remove the ability to specify an arbitrary `buyer` entirely and always use `caller_principal_id()`, since the third-party notification use case does not require accepting a `confirmation_text` on the victim's behalf.

## Proof of Concept

1. Alice transfers 100 ICP to the swap canister subaccount for her principal (`principal_to_subaccount(alice_principal)`).
2. Alice intends to review the SNS terms before confirming participation.
3. Attacker queries `get_init` on the swap canister (public query) and reads `confirmation_text = "I agree to the SNS token sale terms"`.
4. Attacker sends an ingress message to `refresh_buyer_tokens` with `buyer = alice_principal.to_string()` and `confirmation_text = Some("I agree to the SNS token sale terms")`.
5. The swap canister resolves `p = alice_principal` at L133, passes `validate_confirmation_text` at L1150 (text matches), reads Alice's subaccount balance (100 ICP) at L1153–1163, and records Alice as a committed buyer.
6. The swap commits. Alice's 100 ICP is swept to the SNS treasury; Alice receives SNS tokens she never explicitly agreed to purchase.

A deterministic integration test or PocketIC test can reproduce this by: initializing a swap with a `confirmation_text`, having a test identity transfer ICP to the swap subaccount, then calling `refresh_buyer_tokens` from a *different* identity with the victim's principal and the correct text, and asserting the victim is recorded as a buyer.     pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
