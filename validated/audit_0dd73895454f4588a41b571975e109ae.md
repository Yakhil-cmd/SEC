Audit Report

## Title
Coinbase Transaction Input Causes Permanent Retriable Error Loop in ckBTC Checker — (`rs/bitcoin/checker/src/state.rs`, `rs/bitcoin/checker/src/fetch.rs`)

## Summary
The ckBTC checker canister has no coinbase-awareness. When `check_transaction` is called with the txid of a coinbase transaction (which occurs when a miner directs their coinbase output directly to a ckBTC deposit address), `TransactionCheckData::from_transaction` stores the coinbase sentinel input `PreviousOutput{txid:[0;32], vout:0xFFFFFFFF}` verbatim. `check_fetched` then unconditionally attempts to fetch that phantom txid from all three external providers. Every provider returns a non-200 response (404), which is classified as a retriable `TransientInternalError`, creating a permanent retry loop that blocks the deposit and drains minter cycles.

## Finding Description

**Step 1 — No coinbase guard in `from_transaction`**

`TransactionCheckData::from_transaction` maps all inputs verbatim to `PreviousOutput` with no coinbase sentinel check. A grep for "coinbase" across all files under `rs/bitcoin/checker/` returns zero matches.

```rust
// rs/bitcoin/checker/src/state.rs L92-98
let inputs = tx
    .input
    .iter()
    .map(|input| PreviousOutput {
        txid: Txid::from(*(input.previous_output.txid.as_ref() as &[u8; 32])),
        vout: input.previous_output.vout,
    })
    .collect();
```

For a coinbase transaction, `input.previous_output.txid` is `[0;32]` and `vout` is `0xFFFFFFFF`. These are stored without any guard.

**Step 2 — `check_fetched` unconditionally fetches every input with a missing address**

For every input whose `input_addresses[index]` is `None`, `try_fetch_tx(input.txid)` is called. For a coinbase transaction, `input.txid` is `[0;32]`.

```rust
// rs/bitcoin/checker/src/fetch.rs L196-203

    if fetched.input_addresses[index].is_none() {
        match self.try_fetch_tx(input.txid) {
            ToFetch(do_fetch) => { ... }
```

**Step 3 — 404 from providers maps to a retriable error**

In `main.rs`, any non-200 HTTP status (including 404 for the non-existent txid `[0;32]`) becomes `HttpGetTxError::Rejected`:

```rust
// rs/bitcoin/checker/src/main.rs L426-431
if response.status != 200_u32 {
    return Err(HttpGetTxError::Rejected {
        code: 2, //SYS_TRANSIENT
        message: format!("HTTP call {} received code {}", url, response.status),
    });
```

`Rejected` maps to `CheckTransactionRetriable::TransientInternalError` — a retriable, not irrecoverable, response:

```rust
// rs/bitcoin/checker/src/fetch.rs L25-27
HttpGetTxError::Rejected { message, .. } => {
    CheckTransactionRetriable::TransientInternalError(message).into()
}
```

**Step 4 — Error state cycles through all three providers indefinitely**

When `try_fetch_tx` sees `FetchTxStatus::Error`, it advances to the next provider and retries. With three providers (Btcscan → Blockstream → MempoolSpace → Btcscan), all of which return 404 for txid `[0;32]`, the cycle never terminates:

```rust
// rs/bitcoin/checker/src/fetch.rs L85-90
Some(FetchTxStatus::Error(err)) => (
    err.provider.next(),
    err.max_response_bytes,
),
```

Provider rotation is confirmed in `providers.rs` L72-74: `Btcscan → Blockstream → MempoolSpace → Btcscan`.

**Clarification on the precise attack path:**

The claim's PoC states that spending a mature coinbase UTXO to a ckBTC deposit address triggers the bug. This is incorrect. When a spending transaction (which spends a coinbase output) is checked, `check_fetched` fetches the coinbase transaction as an input, reads its output address at the relevant `vout`, and resolves the input address successfully — the coinbase transaction's own `[0;32]` input is never recursively fetched.

The bug is triggered only when `check_transaction` is called with the coinbase txid itself, which happens when a **miner directs their coinbase output directly to a ckBTC deposit address**. The minter detects the resulting UTXO and calls `check_transaction(coinbase_txid)`, triggering the loop.

## Impact Explanation

This is a **High** severity issue. A miner who directs their coinbase reward to a ckBTC deposit address causes the checker to enter a permanent `TransientInternalError` loop for that deposit. The deposit never resolves to `Passed` or an irrecoverable `Error`. The ckBTC minter retries `check_transaction` indefinitely, paying `CHECK_TRANSACTION_CYCLES_SERVICE_FEE` on every call, draining minter cycles over time. This constitutes a concrete, sustained DoS on ckBTC deposit processing and a financial drain on the minter canister — a significant Chain Fusion / ck-token security impact with concrete protocol harm.

## Likelihood Explanation

The attack requires mining capability (directing the coinbase output to a ckBTC deposit address). Any miner — including solo miners and pool operators — can perform this action once per block they mine. It requires no special privileges beyond normal Bitcoin mining participation. The loop is self-sustaining once triggered and persists across canister upgrades (the minter will resume retrying). A miner can repeat this on every block they mine, compounding the cycles drain.

## Recommendation

In `TransactionCheckData::from_transaction` (`rs/bitcoin/checker/src/state.rs`), detect coinbase inputs using the Bitcoin protocol sentinel values (`txid == [0;32]` and `vout == 0xFFFFFFFF`) and return an irrecoverable error immediately:

```rust
pub fn from_transaction(...) -> Result<Self, ...> {
    // Reject coinbase transactions: their inputs reference the null txid
    // sentinel and cannot be fetched from any provider.
    if tx.input.iter().any(|i| {
        i.previous_output.txid.as_ref() == &[0u8; 32]
        && i.previous_output.vout == 0xFFFF_FFFF
    }) {
        return Err(/* coinbase sentinel error */);
    }
    ...
}
```

Alternatively, surface this as `CheckTransactionIrrecoverableError::InvalidTransaction("coinbase transactions cannot be checked")` so the minter can apply a defined policy rather than looping indefinitely.

## Proof of Concept

1. Mine a Bitcoin block where the coinbase output address is set to a valid ckBTC deposit address (any address monitored by the ckBTC minter).
2. Wait for the UTXO to be detected by the ckBTC minter (it will call `check_transaction(coinbase_txid)`).
3. The checker fetches the coinbase transaction successfully (it exists on-chain with a real txid).
4. `check_fetched` calls `try_fetch_tx([0;32])`.
5. All three providers return 404 → `FetchTxStatus::Error` → `TransientInternalError`.
6. On every subsequent `check_transaction` call, `try_fetch_tx` sees `FetchTxStatus::Error` for `[0;32]`, advances to the next provider, fetches again, gets 404 again.
7. The response is always `TransientInternalError`; the deposit never resolves; cycles drain on every retry.

**Deterministic integration test:** Using the existing `MockEnv` framework in `rs/bitcoin/checker/src/fetch/tests.rs`, construct a mock coinbase transaction (one input with `txid=[0;32]`, `vout=0xFFFFFFFF`), inject it as the reply to `http_get_tx`, call `check_fetched`, and assert the response is `CheckTransactionIrrecoverableError` rather than `TransientInternalError`. A second call should confirm the error does not cycle to a new provider.
