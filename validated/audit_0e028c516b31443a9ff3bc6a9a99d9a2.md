### Title
EVM Precompile Hooks Silently Accept Non-Zero `nominal_token_value`, Permanently Trapping ETH at Precompile Addresses — (`File: system_hooks/src/call_hooks/precompiles.rs`)

---

### Summary

The generic EVM precompile hook handler `pure_system_function_hook_impl` ignores the `nominal_token_value` field of the call request entirely. Unlike every other system hook in the codebase, it performs no check for non-zero value and never reverts. Because the bootloader unconditionally transfers value to the callee address *before* invoking the hook, and commits that transfer when the hook returns success, any ETH sent with a call to a precompile address (ecrecover `0x1`, sha256 `0x2`, ripemd-160 `0x3`, identity `0x4`, modexp `0x5`, ecadd `0x6`, ecmul `0x7`, ecpairing `0x8`, P256 `0x100`) is permanently trapped. There is no code at these addresses and no mechanism to withdraw the funds.

---

### Finding Description

**Root cause — `pure_system_function_hook_impl` ignores `nominal_token_value`:**

```rust
// system_hooks/src/call_hooks/precompiles.rs  lines 52-57
let ExternalCallRequest {
    available_resources,
    input: calldata,
    modifier,
    ..          // ← nominal_token_value is silently discarded
} = request;
``` [1](#0-0) 

There is no subsequent `if nominal_token_value != U256::ZERO { return error; }` guard anywhere in this function or in the `IdentityPrecompile` implementation. [2](#0-1) 

**Contrast with every other system hook**, which all explicitly reject non-zero value:

```rust
// l1_messenger.rs line 62
error |= nominal_token_value != U256::ZERO;

// mint_base_token.rs line 50
error |= nominal_token_value != U256::ZERO;

// set_bytecode_on_address.rs line 56
let mut error = nominal_token_value != U256::ZERO;

// contract_deployer_temp.rs line 52
let mut error = nominal_token_value != U256::ZERO;
``` [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

**The bootloader commits the value transfer before the hook can object:**

In `execute_call` (`basic_bootloader/src/bootloader/runner.rs`):

1. `perform_transfer_if_required` is called — it calls `system.io.transfer_nominal_token_value(caller → callee_precompile_address)` and succeeds if the caller has sufficient balance.
2. Only then is `call_to_special_address_execute_callee_frame` invoked to run the hook.
3. Because `pure_system_function_hook_impl` returns `CallResult::Successful`, the bootloader calls `finish_global_frame(None)` — **no rollback** — permanently committing the transfer. [7](#0-6) [8](#0-7) [9](#0-8) 

The affected precompiles are registered via `add_precompile` / `add_precompile_ext`, all of which route through `pure_system_function_hook_impl`: [10](#0-9) [11](#0-10) 

---

### Impact Explanation

Any EVM contract or EOA that calls a precompile address with a non-zero `msg.value` (e.g., `call{value: 1 ether}(address(1), ...)`) will have that ETH permanently transferred to the precompile address. Precompile addresses are system hook addresses with no deployed code and no withdrawal mechanism. The ETH is irrecoverably lost. This is a direct public funds-loss path reachable by any unprivileged transaction sender.

---

### Likelihood Explanation

Medium. Contracts that perform signature verification or cryptographic operations sometimes forward `msg.value` to sub-calls inadvertently (e.g., a proxy that forwards both calldata and value). A user or contract that calls `ecrecover` while accidentally including value loses those funds with no error or revert to warn them. The precompile call succeeds normally, making the loss silent.

---

### Recommendation

Add a non-zero value guard at the top of `pure_system_function_hook_impl`, consistent with all other system hooks:

```rust
// In pure_system_function_hook_impl, after destructuring:
let ExternalCallRequest {
    available_resources,
    input: calldata,
    modifier,
    nominal_token_value,  // ← bind instead of ignoring
    ..
} = request;

if nominal_token_value != U256::ZERO {
    return Ok((make_error_return_state(available_resources), return_memory));
}
```

This mirrors the pattern already used in `l1_messenger_hook`, `mint_base_token_hook`, `set_bytecode_on_address_hook`, and `contract_deployer_temp_hook`.

---

### Proof of Concept

1. Deploy a contract on ZKsync OS that calls `ecrecover` (address `0x0000…0001`) with `value: 1 ether`:
   ```solidity
   (bool ok, ) = address(1).call{value: 1 ether}(
       abi.encode(hash, v, r, s)
   );
   // ok == true, but 1 ETH is now at address(1) forever
   ```
2. The bootloader's `perform_transfer_if_required` transfers 1 ETH from the caller to `0x1`.
3. `pure_system_function_hook_impl` runs `ecrecover`, ignores the value, and returns `CallResult::Successful`.
4. `finish_global_frame(None)` commits the transfer.
5. `address(1).balance == 1 ether`; no code exists there; funds are permanently trapped. [1](#0-0) [12](#0-11) [9](#0-8)

pub fn pure_system_function_hook_impl<'a, F, E, S>(
    request: ExternalCallRequest<S>,
    _caller_ee: u8,
    system: &mut System<S>,
    return_memory: &'a mut [MaybeUninit<u8>],
) -> Result<(CompletedExecution<'a, S>, &'a mut [MaybeUninit<u8>]), SystemError>
where
    F: SystemFunctionInvocation<S, E>,
    S: EthereumLikeTypes,
    S::IO: IOSubsystemExt,
    E: Subsystem,
{
    let ExternalCallRequest {
        available_resources,
        input: calldata,
        modifier,
        ..
    } = request;

    // We allow static calls as we are "pure" hook
    if modifier == CallModifier::Constructor {
        return Err(internal_error!("precompile called with constructor modifier").into());
    }

    let mut resources = available_resources;

    let allocator = system.get_allocator();

    let mut return_vec = SliceVec::new(return_memory);
    let mut logger = system.get_logger();
    let result = F::invoke(
        system.io.oracle(),
        &mut logger,
        &calldata,
        &mut return_vec,
        &mut resources,
        allocator,
    );

    match result {
        Ok(()) => {
            let (returndata, rest) = return_vec.destruct();
            Ok((
                make_return_state_from_returndata_region(resources, returndata),
                rest,
            ))
        }
        Err(e) => match e.root_cause() {
            // Following EVM precompiles, we burn all gas on out-of-gas or invalid inputs
            RootCause::Runtime(RuntimeError::OutOfErgs(_)) | RootCause::Usage(_) => {
                system_log!(system, "Out of gas during system hook\nError:{e:?}");
                resources.exhaust_ergs();
                let (_, rest) = return_vec.destruct();
                Ok((make_error_return_state(resources), rest))
            }
            // Internal error means something is fatally wrong inside the hook, so we propagate it
            RootCause::Internal(e) => Err(e.clone_or_copy().into()),
            // On fatal runtime error (e.g., out of return memory or native resources) we also propagate the error
            RootCause::Runtime(e @ RuntimeError::FatalRuntimeError(_)) => {
                Err(e.clone_or_copy().into())
            }
        },
    }
}

pub fn add_precompile<S: EthereumLikeTypes, A: Allocator + Clone, P, E>(
    hooks: &mut HooksStorage<S, A>,
    address_low: u16,
) -> Result<(), InternalError>
where
    S::IO: IOSubsystemExt,
    P: SystemFunction<S::Resources, E>,
    E: Subsystem,
{
    // Sanity check to ensure that the address being added is indeed in the precompile addresses list
    if !PRECOMPILE_ADDRESSES_LOWS.contains(&address_low) {
        return Err(internal_error!(
            "Attempted to add a precompile that is not in the precompile addresses list"
        ));
    }
    hooks.add_call_hook(
        address_low,
        SystemCallHook::new(
            pure_system_function_hook_impl::<SystemFunctionInvocationUser<S, E, P>, E, S>,
        ),
    )
}
