### Title
Missing `ValidateNonPayable` in `rewards` Method Allows Silent Fund Lock at Distribution Precompile - (File: `precompiles/distribution/distribution.go`)

---

### Summary

The `rewards` method in the distribution precompile does not validate that no value is attached to the call, and the `value *big.Int` parameter is not even forwarded to the function. If a user calls `rewards` with a non-zero `msg.value`, the EVM transfers the funds to the distribution precompile address, the call succeeds without reverting, and the funds are permanently locked with no recovery path.

---

### Finding Description

In `precompiles/distribution/distribution.go`, the top-level `Execute` dispatcher handles all distribution precompile methods. Every other method either calls `pcommon.ValidateNonPayable(value)` directly or passes `value` to a sub-function that does so. The `RewardsMethod` case is the sole exception:

```go
case RewardsMethod:
    return p.rewards(ctx, method, args)   // value is silently dropped
``` [1](#0-0) 

The `rewards` function signature does not accept `value` at all:

```go
func (p PrecompileExecutor) rewards(ctx sdk.Context, method *abi.Method, args []interface{}) (ret []byte, remainingGas uint64, rerr error) {
``` [2](#0-1) 

Compare with every other non-payable method, which calls `ValidateNonPayable`:

```go
func (p PrecompileExecutor) validateInput(value *big.Int, args []interface{}, expectedArgsLength int) error {
    if err := pcommon.ValidateNonPayable(value); err != nil {
        return err
    }
    ...
}
``` [3](#0-2) 

`ValidateNonPayable` is defined as:

```go
func ValidateNonPayable(value *big.Int) error {
    if value != nil && value.Sign() != 0 {
        return errors.New("sending funds to a non-payable function")
    }
    return nil
}
``` [4](#0-3) 

The distribution precompile address (`0x0000000000000000000000000000000000001007`) is **not** in the `payablePrecompiles` registry, confirming it is not intended to receive value:

```go
var payablePrecompiles = map[string]struct{}{
    bank.BankAddress:       {},
    staking.StakingAddress: {},
    gov.GovAddress:         {},
    wasmd.WasmdAddress:     {},
}
``` [5](#0-4) 

---

### Impact Explanation

In go-ethereum's EVM `Call` path, value is transferred from caller to callee **before** the precompile executes. If the precompile returns success (no error), the transfer is committed. Because `rewards` always succeeds regardless of `value`, any usei/wei attached to the call is credited to the distribution precompile address and committed to state. The distribution precompile has no `withdraw` or sweep function, so those funds are permanently locked. The loss is bounded only by what the caller sends — there is no on-chain cap.

---

### Likelihood Explanation

Likelihood is low-to-medium. A user or contract that mistakenly attaches value when calling the `rewards` query (e.g., copy-paste error, wallet UI bug, or a contract that forwards `msg.value` indiscriminately to a precompile call) will silently lose funds. The call does not revert, so there is no on-chain signal that anything went wrong.

---

### Recommendation

Add `ValidateNonPayable` at the top of the `rewards` function (or pass `value` to it and check), consistent with every other non-payable method in the same precompile:

```go
func (p PrecompileExecutor) rewards(ctx sdk.Context, method *abi.Method, args []interface{}, value *big.Int) (ret []byte, remainingGas uint64, rerr error) {
    if err := pcommon.ValidateNonPayable(value); err != nil {
        rerr = err
        return
    }
    ...
}
```

And update the call site in `Execute`:

```go
case RewardsMethod:
    return p.rewards(ctx, method, args, value)
```

---

### Proof of Concept

1. Deploy or use any EVM wallet/contract.
2. Call the distribution precompile at `0x0000000000000000000000000000000000001007` with selector for `rewards(address)`, passing any valid associated address, and attach `msg.value = 1 ether` (or any non-zero amount).
3. Observe: the call succeeds (returns valid reward data), the caller's balance is reduced by the sent value, and the distribution precompile address balance increases by that amount.
4. Attempt to recover the funds — no function exists on the distribution precompile to do so. Funds are permanently locked. [6](#0-5)
