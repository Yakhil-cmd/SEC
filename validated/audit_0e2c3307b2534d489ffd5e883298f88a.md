### Title
Silo Whitelist Bypass via `call` and `deploy_code` — (`engine/src/contract_methods/evm_transactions.rs`)

### Summary

In Silo mode, Aurora Engine enforces a whitelist (`WhitelistKind::Account`, `WhitelistKind::Address`, `WhitelistKind::Admin`, `WhitelistKind::EvmAdmin`) to restrict which NEAR accounts and EVM addresses may submit transactions or deploy code. This check is implemented in `assert_access` and is called only inside `submit_with_alt_modexp`. The `call` and `deploy_code` NEAR entrypoints execute EVM code without ever invoking `assert_access`, so any NEAR account — regardless of whitelist status — can bypass the Silo access control entirely.

### Finding Description

`assert_access` enforces the Silo whitelist: [1](#0-0) 

It is called only inside `submit_with_alt_modexp`: [2](#0-1) 

which is reached only through `submit` and `submit_with_args`: [3](#0-2) 

The `call` function, however, constructs an `Engine` and calls `engine.call_with_args` directly, with no call to `assert_access`: [4](#0-3) 

Likewise, `deploy_code` constructs an `Engine` and calls `engine.deploy_code_with_input` directly, also without `assert_access`: [5](#0-4) 

Neither the NEAR-level entrypoints for `call` nor `deploy_code` in `lib.rs` add any whitelist guard: [6](#0-5) 

The whitelist helpers confirm that `is_allow_submit` and `is_allow_deploy` are the intended gatekeepers: [7](#0-6) 

### Impact Explanation

In Silo mode, the whitelist is the primary access-control boundary. An attacker who is not whitelisted can call the `call` entrypoint to execute arbitrary EVM transactions — including ETH transfers and interactions with any deployed EVM contract — using their implicit EVM address (`predecessor_address`). They can also call `deploy_code` to deploy arbitrary EVM contracts. This completely undermines the Silo whitelist security model, enabling unauthorized EVM execution and potential theft of any ETH or ERC-20 tokens held at the attacker's implicit EVM address, as well as unauthorized interaction with any EVM contract in the Silo.

**Impact: High — unauthorized EVM execution bypassing the Silo whitelist; direct theft of funds held at the attacker's implicit EVM address and unauthorized contract deployment.**

### Likelihood Explanation

The `call` and `deploy_code` NEAR functions are publicly callable by any NEAR account. No key, signature, or special privilege is required. Any NEAR account that is intentionally excluded from the Silo whitelist can trivially bypass the restriction by routing through `call` instead of `submit`. Likelihood is **High**.

### Recommendation

Apply `assert_access` (or an equivalent Silo whitelist check) inside `call` and `deploy_code` before executing EVM logic. For `call`, the EVM sender is `predecessor_address(&predecessor_account_id)` and the target is taken from `CallArgs`; both should be checked against the `Account` and `Address` whitelists. For `deploy_code`, the `Admin` and `EvmAdmin` whitelists should be checked. Alternatively, factor the whitelist guard into a shared pre-execution hook applied to all EVM-executing entrypoints.

### Proof of Concept

1. Deploy Aurora Engine in Silo mode; enable all four whitelists via `set_whitelists_statuses`.
2. Do **not** add attacker's NEAR account (`attacker.near`) or its implicit EVM address to any whitelist.
3. Attempt `submit` with a signed Ethereum transaction from `attacker.near` → fails with `ERR_NOT_ALLOWED` (whitelist enforced via `assert_access`).
4. Call `call` from `attacker.near` with `CallArgs { contract: <target>, value: <amount>, input: <calldata> }` → **succeeds**; EVM execution proceeds with `msg.sender = predecessor_address("attacker.near")`, whitelist never checked.
5. Call `deploy_code` from `attacker.near` with arbitrary EVM bytecode → **succeeds**; contract deployed, whitelist never checked.

The root cause is the absence of `assert_access` in `call` (`evm_transactions.rs:45–71`) and `deploy_code` (`evm_transactions.rs:20–43`), while `submit_with_alt_modexp` (`engine.rs:1052`) correctly enforces it.

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

/// Check if a user has the right to submit transactions.
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}

/// Check if a user has the right to receive erc20 tokens.
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}

fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Admin);
    !list.is_enabled() || list.is_exist(account_id)
}

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::EvmAdmin);
    !list.is_enabled() || list.is_exist(address)
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
