Audit Report

## Title
Hot Keys Not Cleared When Spawning NNS Neuron with a New Controller - (File: `rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with a `new_controller` principal different from the parent neuron's controller, the child neuron is unconditionally constructed with the parent's full hot key list cloned onto it. The new controller never authorized these hot keys, yet they retain the ability to vote and follow on behalf of the new controller's neuron. This constitutes unauthorized access to governance rights of a neuron transferred to a new principal.

## Finding Description
In `spawn_neuron`, `child_controller` is resolved from `spawn.new_controller` if provided, otherwise falls back to `parent_neuron.controller()`: [1](#0-0) 

The child neuron is then built with `child_controller` as its controller, but `.with_hot_keys(parent_neuron.hot_keys.clone())` is called unconditionally — there is no branch that skips or clears hot keys when `new_controller` differs from the parent's controller: [2](#0-1) 

The authorization check `is_authorized_to_vote` passes for any principal that is either the controller **or** a hot key: [3](#0-2) 

The `ManageNeuron` proto confirms that `Follow` and `RegisterVote` are available to registered hot keys: [4](#0-3) 

No existing guard clears or conditionally omits hot keys based on whether `new_controller` differs from the parent's controller. The result is that principals the new controller never authorized retain delegated governance authority over the child neuron.

## Impact Explanation
This is unauthorized access to a neuron's governance rights. The parent's hot key holders can call `manage_neuron` with `Follow` or `RegisterVote` on the child neuron and pass the `is_authorized_to_vote` check, exercising the new controller's voting power in NNS governance without their knowledge or consent. This matches the High ($2,000–$10,000) bounty impact: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The new controller's NNS voting power is silently exercised by third parties, potentially influencing governance outcomes. The new controller can remediate by removing hot keys, but may never discover the issue.

## Likelihood Explanation
Medium-to-High. The preconditions are realistic and common: (1) the parent neuron has at least one hot key — standard operational practice for cold-key security — and (2) the parent controller spawns a child neuron to a different `new_controller`, which occurs naturally when neurons are gifted, used as rewards, or transferred. The `Spawn` command with `new_controller` is a documented, publicly accessible ingress call requiring only the parent controller's signature. No special privileges beyond owning the parent neuron are required to trigger the bug.

## Recommendation
In `spawn_neuron`, conditionally copy hot keys only when `new_controller` is `None` (i.e., the child controller equals the parent controller). When `spawn.new_controller` is `Some(...)` and resolves to a principal different from the parent's controller, initialize the child neuron with an empty hot key list, allowing the new controller to add their own trusted hot keys:

```rust
let hot_keys = if spawn.new_controller.is_some() {
    vec![]
} else {
    parent_neuron.hot_keys.clone()
};
// ...
.with_hot_keys(hot_keys)
``` [5](#0-4) 

## Proof of Concept
1. Alice creates NNS neuron N with hot key `H`.
2. Neuron N accumulates maturity.
3. Alice calls `manage_neuron` → `Spawn { new_controller: Some(Bob), percentage_to_spawn: 100, nonce: None }`.
4. `spawn_neuron` executes: child neuron C is created with `controller = Bob` and `hot_keys = [H]` (cloned from Alice's neuron).
5. `H` calls `manage_neuron` on neuron C with `RegisterVote { proposal: <id>, vote: Yes }` or `Follow { topic: ..., followees: [...] }`.
6. The call passes `is_authorized_to_vote` because `H ∈ C.hot_keys`.
7. Bob's neuron votes or follows according to `H`'s instructions without Bob's knowledge.

A deterministic integration test can reproduce this by: creating a parent neuron with a hot key, spawning to a new controller, then asserting that the hot key can successfully call `RegisterVote` or `Follow` on the child neuron — and separately asserting that the child neuron's `hot_keys` field is non-empty when it should be empty.
