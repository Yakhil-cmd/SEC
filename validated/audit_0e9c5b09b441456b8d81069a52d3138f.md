### Title
L1→L2 Deposit Block Halt via `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Revert - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

The `notify_l2_asset_tracker` function in the ZKsync OS bootloader calls the external EVM contract `L2AssetTracker` during every L1→L2 deposit transaction and unconditionally treats any revert from that call as a **fatal error that halts all block processing**. Because `L2AssetTracker` is a regular EVM contract whose storage state can be read and potentially manipulated, any condition that causes `handleFinalizeBaseTokenBridgingOnL2` to revert — whether through crafted transaction parameters or state manipulation — permanently blocks the entire L1→L2 deposit pipeline for the chain.

### Finding Description

During L1→L2 transaction processing, the bootloader calls `notify_l2_asset_tracker` up to three times (once for value mint, once for operator fee, once for refund). Each call invokes `handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)` on the `L2AssetTracker` contract at `L2_ASSET_TRACKER_ADDRESS`. [1](#0-0) 

The critical design decision is at lines 901–911: if the call returns a failure result, the function returns an `internal_error!` — a fatal `BootloaderSubsystemError` that propagates up and halts block processing entirely, not just the individual transaction. [2](#0-1) 

The `L2AssetTracker` is a regular EVM contract (not a system hook with hardcoded Rust logic). Its bytecode is loaded from a hex predeploy and its storage is initialized with specific slots: [3](#0-2) 

The `handleFinalizeBaseTokenBridgingOnL2` function in the `L2AssetTracker` contract checks internal state including `isAssetRegistered`, `assetMigrationNumber`, and chain ID matching. The calldata passed to it includes the `l1_chain_id` (read from `L2AssetTracker` storage slot 154) and the `amount` (derived from the L1 transaction's `to_mint` field, which is fully attacker-controlled): [4](#0-3) 

The `mint_base_token` function calls `notify_l2_asset_tracker` before modifying balances, meaning the external call happens unconditionally for every non-zero deposit: [5](#0-4) 

This is called for operator fee payment and refund as well: [6](#0-5) 

### Impact Explanation

If `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` reverts for any reason, the `internal_error!` propagates as a fatal `BootloaderSubsystemError`. This does not merely fail the individual L1→L2 transaction — it halts the entire block processing pipeline. Every subsequent L1→L2 deposit in the same block (and potentially all future blocks, depending on how the sequencer handles fatal errors) is blocked. Deposited funds that were committed on L1 cannot be finalized on L2, resulting in a chain liveness failure and potential permanent loss of bridged funds.

This is a strictly more severe impact than the original report (which blocked a single offer): here, a single crafted L1→L2 transaction can halt the entire chain's deposit processing.

### Likelihood Explanation

The `L2AssetTracker` is a real EVM contract with internal state checks. The `amount` parameter passed to `handleFinalizeBaseTokenBridgingOnL2` is derived directly from the attacker-controlled `to_mint` field of the L1 transaction. An attacker can:

1. Submit an L1→L2 transaction with a `to_mint` value crafted to trigger an arithmetic overflow or boundary condition inside `handleFinalizeBaseTokenBridgingOnL2` (e.g., `type(uint256).max` causing overflow in the contract's accounting).
2. Call public functions on `L2AssetTracker` directly (it is a regular EVM contract accessible to any L2 user) to manipulate storage state — such as deregistering the base token asset or altering migration numbers — so that subsequent bootloader calls to `handleFinalizeBaseTokenBridgingOnL2` revert.

The `L2AssetTracker` is not a system hook with hardcoded Rust access controls; it is an EVM contract whose behavior depends on its mutable storage state. [7](#0-6) 

### Recommendation

**Short term:** Change `notify_l2_asset_tracker` to treat a revert from `L2AssetTracker` as a non-fatal condition. Log the failure and continue block processing (skipping the accounting notification) rather than halting the entire block. The token balance transfers should not be gated on the success of the asset tracker notification.

**Long term:** Replace the external EVM contract call with a system hook (hardcoded Rust logic) for the `handleFinalizeBaseTokenBridgingOnL2` accounting, removing the dependency on a mutable EVM contract's behavior for block liveness. Alternatively, implement a circuit-breaker that allows the chain to continue processing L1→L2 deposits even if the asset tracker is in a bad state, with a separate recovery mechanism.

### Proof of Concept

1. Attacker calls a public function on `L2AssetTracker` (e.g., a function that modifies `isAssetRegistered` or `assetMigrationNumber` for the base token asset ID) to put the contract in a state where `handleFinalizeBaseTokenBridgingOnL2` will revert.
2. Attacker (or any user) submits an L1→L2 deposit transaction with `to_mint > 0`.
3. The bootloader calls `notify_l2_asset_tracker` → `run_single_interaction` → `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2`.
4. The call reverts (due to the manipulated state).
5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
6. This fatal error propagates up through `mint_base_token` → `process_l1_transaction` → the block processing loop.
7. Block processing halts. All L1→L2 deposits are permanently blocked. [8](#0-7)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
}
