### Title
Silent Burn Without Exit on Empty Recipient in `withdrawToNear` — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

An unprivileged EVM user can call `withdrawToNear` with an empty `recipient` bytes argument. The function burns the caller's ERC-20 tokens first, then calls the `ExitToNear` precompile with a 33-byte input (flag + 32-byte amount, zero recipient bytes). The precompile fails internally because `parse_recipient` rejects an empty account ID, but the Solidity assembly block never checks the return value of `call`, so the outer transaction succeeds. Without the `error_refund` feature (which is **not** a default feature in production), no refund callback is scheduled. The burned ERC-20 tokens are permanently destroyed and the corresponding NEP-141 tokens remain locked in Aurora's account with no recovery path for the user.

---

### Finding Description

**Step 1 — Entry point (no privilege required)**

`withdrawToNear` is `external` with no access control: [1](#0-0) 

The function burns tokens before calling the precompile, and the assembly `call` result `res` is captured but **never checked**. There is no `require(res != 0)`.

**Step 2 — Input construction with empty recipient**

When `recipient = bytes('')` (length 0):
- `input_size = 1 + 32 + 0 = 33`
- `input = [0x01] ++ amount_b (32 bytes)` [2](#0-1) 

**Step 3 — `MIN_INPUT_SIZE` check passes**

Without the `error_refund` feature, `MIN_INPUT_SIZE = 3`. A 33-byte input passes `validate_input_size` trivially: [3](#0-2) 

`parse_input` (non-`error_refund` variant) strips the flag byte and returns the remaining 32 bytes: [4](#0-3) 

**Step 4 — `parse_recipient` fails on empty slice**

For flag `0x1`, `parse_amount` consumes the 32 bytes successfully, then `parse_recipient` is called with `&input[32..]` = `&[]` (empty slice): [5](#0-4) 

`parse_recipient(b"")` converts the empty bytes to `""` (valid UTF-8), then attempts `"".parse::<AccountId>()`, which fails because an empty string is not a valid NEAR account ID: [6](#0-5) 

**Step 5 — Precompile `run` propagates the error**

The `?` at line 419 causes `run` to return `Err(ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID"))`: [7](#0-6) 

The EVM sub-call returns 0 (failure). The Solidity code never checks `res`, so the outer transaction succeeds with tokens already burned.

**Step 6 — No refund path without `error_refund`**

`error_refund` is not a default feature in either the engine or precompiles crate: [8](#0-7) [9](#0-8) 

Without it, `refund: None` is always set, and the existing test explicitly acknowledges that no EVM refund occurs in this configuration: [10](#0-9) 

---

### Impact Explanation

The user's ERC-20 tokens are permanently burned. The corresponding NEP-141 tokens remain locked in Aurora's account with no user-accessible recovery path. This constitutes at minimum **High: Temporary freezing of funds** (if an admin can manually recover the NEP-141 side), and arguably **Critical: Permanent freezing of funds** from the user's perspective, since the ERC-20 tokens are irrecoverably destroyed.

---

### Likelihood Explanation

- No privilege is required; any EVM account can call `withdrawToNear`.
- The `error_refund` feature is absent from the default feature set, making this the standard production configuration.
- The Solidity contract performs no validation on the `recipient` argument length.
- The attack is a single transaction with a trivially constructed argument.

---

### Recommendation

1. **Solidity-side**: Add a length check on `recipient` before burning:
   ```solidity
   require(recipient.length > 0, "ERR_EMPTY_RECIPIENT");
   ```
   And check the precompile call result:
   ```solidity
   require(res != 0, "ERR_EXIT_PRECOMPILE_FAILED");
   ``` [1](#0-0) 

2. **Rust-side**: Enforce `MIN_INPUT_SIZE` to be large enough to guarantee a non-empty recipient even without `error_refund`. For ERC-20 exits (flag `0x1`), the minimum meaningful input is `1 (flag) + 32 (amount) + 1 (at least one recipient byte)` = 34 bytes.

3. **Enable `error_refund`** as a default feature so that failed exits always schedule a re-mint callback.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.0;

// Precondition: engine compiled WITHOUT error_refund feature
// 1. Deploy EvmErc20, mint 100 tokens to attacker
// 2. Call withdrawToNear with empty recipient
function exploit(EvmErc20 token) external {
    uint256 balanceBefore = token.balanceOf(address(this));
    // recipient = bytes('') — length zero
    token.withdrawToNear(bytes(''), 100);
    uint256 balanceAfter = token.balanceOf(address(this));
    // balanceBefore - balanceAfter == 100 (tokens burned)
    // No ExitToNear event emitted
    // No NEP-141 transfer occurred
    // Tokens are permanently lost
    assert(balanceBefore - balanceAfter == 100);
}
```

The transaction succeeds (no revert), the balance decreases by 100, and no `ExitToNear` log is emitted, confirming the invariant violation.
