### Title
Asymmetric Authorization Between `pause_precompiles` and `resume_precompiles` Allows Future ACL Members to Permanently Freeze Bridge Withdrawals - (File: `engine/src/contract_methods/admin.rs`)

### Summary

`pause_precompiles` uses a broader ACL-based `EngineAuthorizer` (explicitly designed to be extended to non-owner accounts per a `TODO` comment in `get_authorizer()`), while `resume_precompiles` enforces strict owner-only access. When the ACL is extended as planned, any ACL member can pause the `ExitToNear`/`ExitToEthereum` bridge precompiles—blocking all user withdrawals from Aurora—but cannot resume them, leaving funds frozen until the owner acts.

---

### Finding Description

**`pause_precompiles`** uses an ACL-based authorizer:

```rust
// engine/src/contract_methods/admin.rs:226-241
pub fn pause_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        require_running(&state::get_state(&io)?)?;
        let authorizer: EngineAuthorizer = engine::get_authorizer(&io);

        if !authorizer.is_authorized(&env.predecessor_account_id()) {
            return Err(b"ERR_UNAUTHORIZED".into());
        }
        // ... pauses precompiles
    })
}
```

**`resume_precompiles`** uses strict owner-only access:

```rust
// engine/src/contract_methods/admin.rs:208-223
pub fn resume_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &predecessor_account_id)?;
        // ... resumes precompiles
    })
}
```

The `get_authorizer()` function contains an explicit `TODO` acknowledging this is temporary and will be extended:

```rust
// engine/src/engine.rs:1254-1260
pub fn get_authorizer<I: IO + Copy>(io: &I) -> EngineAuthorizer {
    // TODO: a temporary use the owner account only until the engine adapts std with near-plugins
    state::get_state(io)
        .map(|state| EngineAuthorizer::from_accounts(once(state.owner_id)))
        .unwrap_or_default()
}
```

The `EngineAuthorizer` struct is explicitly designed to hold a set of multiple accounts:

```rust
// engine/src/pausables.rs:86-100
pub struct EngineAuthorizer {
    /// List of AccountIds with the permission to pause precompiles.
    pub acl: BTreeSet<AccountId>,
}
```

The two precompiles that can be paused are the **only** bridge exit mechanisms:

- `PrecompileFlags::EXIT_TO_NEAR = 0b01` — exits funds from Aurora to NEAR
- `PrecompileFlags::EXIT_TO_ETHEREUM = 0b10` — exits funds from Aurora to Ethereum

When either is paused, the precompile set returns a fatal `ERR_PAUSED` error, blocking all user withdrawals:

```rust
// engine-precompiles/src/lib.rs:140-144
if self.is_paused(&address) {
    return Some(Err(PrecompileFailure::Fatal {
        exit_status: ExitFatal::Other(prelude::Cow::Borrowed("ERR_PAUSED")),
    }));
}
```

---

### Impact Explanation

**Impact: High — Temporary freezing of funds.**

When the ACL is extended to include a non-owner account (as the `TODO` comment explicitly plans), that account can call `pause_precompiles` with mask `0b11` to pause both `ExitToNear` and `ExitToEthereum`. All Aurora users holding ETH or bridged ERC-20 tokens are then unable to withdraw to NEAR or Ethereum. Since `resume_precompiles` is owner-only, the freeze persists until the owner acts. If the owner is slow to respond or unavailable, this constitutes a prolonged temporary fund freeze.

---

### Likelihood Explanation

**Likelihood: Medium.**

The `TODO` comment in `get_authorizer()` explicitly states the ACL will be extended to non-owner accounts when `near-plugins` is integrated. The `EngineAuthorizer` struct and its `acl: BTreeSet<AccountId>` field are already in place for this purpose. Once extended, any ACL member (a semi-trusted operator, relayer manager, or similar role) who acts maliciously or is compromised can trigger the freeze. This mirrors the exact scenario in the external report: a semi-trusted role with pause power but no resume power.

---

### Recommendation

Enforce the same authorization level for both `pause_precompiles` and `resume_precompiles`. Either:

1. **Symmetric ACL**: Allow the same ACL members who can pause to also resume. This prevents a one-sided freeze.
2. **Owner-only for both**: Keep both functions owner-only until a proper role-separation design is implemented.
3. **Time-bounded pauses**: Introduce a pause expiry so that pauses automatically lift after a fixed duration, preventing indefinite freezes by non-owner ACL members.

---

### Proof of Concept

1. Owner extends the ACL (once `near-plugins` integration lands) to include `operator.near`.
2. `operator.near` calls `pause_precompiles` with `paused_mask = 0b11` (both `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM`).
3. All EVM transactions calling either precompile address receive a fatal `ERR_PAUSED` error — no user can withdraw ETH or ERC-20 tokens from Aurora.
4. `operator.near` attempts to call `resume_precompiles` — rejected with `ERR_NOT_ALLOWED` because `require_owner_only` is enforced.
5. Funds remain frozen until the owner calls `resume_precompiles`.

pub fn resume_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
