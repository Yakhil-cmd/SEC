Audit Report

## Title
`migrations_disabled` Flag Not Checked in Timer-Driven Processing Pipeline — (`rs/migration_canister/src/processing.rs`)

## Summary
The migration canister exposes a `disable_api()` privileged endpoint that sets a persistent `DISABLED` flag in stable memory, intended to halt all migration activity in an emergency. The flag is correctly checked in the `migrate_canister` ingress handler to block new requests, but none of the seven timer-driven processing stages (`process_accepted`, `process_controllers_changed`, `process_stopped`, `process_renamed`, `process_updated`, `process_routing_table`, `process_migrated_canister_deleted`) consult `migrations_disabled()` before advancing in-flight requests. As a result, calling `disable_api()` does not stop already-accepted migrations; they proceed through every irreversible stage to completion.

## Finding Description
The `DISABLED` flag is stored in stable memory and read via `migrations_disabled()`:

- `canister_state.rs` L27–28: `static DISABLED: RefCell<Cell<bool, Memory>>` initialized to `false`.
- `canister_state.rs` L56–58: `pub fn migrations_disabled() -> bool { DISABLED.with_borrow(|x| *x.get()) }`.

`disable_api()` in `privileged.rs` L34–38 sets this flag to `true` after verifying the caller is a controller or the NNS governance canister.

The flag is checked at the ingress entry point in `migration_canister.rs` L63–65:
```rust
if migrations_disabled() {
    return Err(Some(ValidationError::MigrationsDisabled(Reserved)));
}
```

However, `process_all_by_predicate` in `processing.rs` L35–71 — the single dispatcher called by all seven timer intervals in `lib.rs` L462–517 — contains no call to `migrations_disabled()`. Its only guard is the `MethodGuard` re-entrancy lock (L43–45). Each individual processor (`process_accepted` at L75, `process_controllers_changed` at L111, `process_stopped` at L199, `process_renamed` at L231, `process_updated` at L253, `process_routing_table` at L287, `process_migrated_canister_deleted` at L309) also begins executing without any disabled-flag check.

The exploit path is straightforward:
1. Caller A (controller of both `canister_X` and `canister_Y`) calls `migrate_canister`; the `migrations_disabled()` check passes and the request enters `RequestState::Accepted`.
2. A controller or governance canister calls `disable_api()`, setting `DISABLED = true`.
3. The timer fires every second. `process_all_by_predicate("accepted", ...)` runs, does not call `migrations_disabled()`, and invokes `process_accepted`, which calls `set_exclusive_controller` on both canisters — stripping their original controllers.
4. Subsequent timer ticks advance the request through `process_stopped` (calls `rename_canister`), `process_renamed` (calls the registry `migrate_canister` to permanently reassign the subnet), `process_routing_table` (calls `delete_canister` on the original subnet), and finally `process_migrated_canister_deleted` (restores controllers after a mandatory wait).
5. The migration completes in full despite `DISABLED = true`.

## Impact Explanation
This constitutes a **High** severity finding matching: *"Significant NNS/SNS or infrastructure security impact with concrete user or protocol harm."*

The migration pipeline performs a sequence of irreversible, high-impact operations on the Internet Computer: permanent subnet reassignment via registry routing-table update (`process_renamed`), deletion of the migrated canister from its original subnet (`process_routing_table`), and stripping of original controllers from both canisters for the entire duration of the migration (`process_accepted`). When `disable_api()` is invoked as an emergency stop — the exact scenario it exists for — none of these operations are halted. The routing-table update and canister deletion are not reversible. Controllers of both canisters remain stripped until the very end of the pipeline, leaving canisters in an uncontrolled state for the full duration of a migration that was supposed to be stopped.

## Likelihood Explanation
The precondition for the initial request is that the caller controls both the migrated and replaced canisters — a legitimate, unprivileged ingress call available to any canister owner. The `disable_api()` call requires controller or governance access, but that is precisely the party expected to invoke it in an emergency (e.g., after discovering a registry inconsistency or a bug in the migration logic). The timer fires every second, so in-flight requests advance continuously and rapidly regardless of the flag. The scenario — accept a migration, discover a problem, call `disable_api()`, observe that the migration completes anyway — is the primary use case for the emergency stop and is fully reproducible.

## Recommendation
Add a `migrations_disabled()` guard at the top of `process_all_by_predicate` in `rs/migration_canister/src/processing.rs`, immediately after the `MethodGuard` acquisition:

```rust
pub async fn process_all_by_predicate<F>(
    tag: &str,
    predicate: impl Fn(&RequestState) -> bool,
    processor: impl Fn(RequestState) -> F,
) where
    F: Future<Output = ProcessingResult<RequestState, RequestState>>,
{
    let Ok(_guard) = MethodGuard::new(tag) else {
        return;
    };
    if crate::canister_state::migrations_disabled() {
        return;
    }
    // ... rest of function
}
```

This single insertion covers all seven processing stages without requiring changes to each individual processor. Alternatively, the same guard can be added inside each `process_*` function for finer-grained control.

## Proof of Concept
A PocketIC integration test can reproduce this deterministically:

1. Deploy the migration canister with a test allowlist containing principal A.
2. Principal A (controller of both `canister_X` and `canister_Y`) calls `migrate_canister({ migrated_canister_id: canister_X, replaced_canister_id: canister_Y })`. Assert `Ok(())` is returned and the request is in `RequestState::Accepted`.
3. A controller of the migration canister calls `disable_api()`. Assert `Ok(())`.
4. Advance the PocketIC clock by 1 second to trigger the timer. Observe that `process_all_by_predicate("accepted", ...)` runs and `set_exclusive_controller` is called on both canisters (verifiable via `canister_status` showing the migration canister as sole controller).
5. Advance the clock through all subsequent stages. Assert that the routing-table update and canister deletion complete despite `DISABLED = true`.
6. Assert that `migration_status` returns `MigrationStatus::Succeeded`, confirming the migration completed in full while the disabled flag was set.
