### Title
`receive_erc20_tokens` trusts caller-supplied `ft_on_transfer` amount without verifying actual NEP-141 balance change, enabling ERC-20 over-minting - (File: `engine/src/engine.rs`)

---

### Summary

`Engine::receive_erc20_tokens` mints ERC-20 mirror tokens on Aurora using the `amount` field supplied by the calling NEP-141 contract in the `ft_on_transfer` callback. It never checks the actual change in Aurora's NEP-141 balance. For any fee-on-transfer NEP-141 token registered as an ERC-20 mirror, the NEP-141 contract transfers `amount - fee` to Aurora's account but reports `amount` in the callback. Aurora mints the full `amount`, creating unbacked ERC-20 supply and eventual bridge insolvency.

---

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` dispatches to `Engine::receive_erc20_tokens` when the predecessor is any registered NEP-141 token (i.e., not the eth-connector): [1](#0-0) 

Inside `receive_erc20_tokens`, the mint amount is taken verbatim from the callback argument: [2](#0-1) 

This `amount` is then passed directly to the ERC-20 `mint` call: [3](#0-2) 

There is no balance-before / balance-after check on Aurora's NEP-141 holdings. The engine unconditionally trusts the value the NEP-141 contract chose to pass in the callback.

For a fee-on-transfer NEP-141 token, the standard `ft_transfer_call` flow works as follows:

1. NEP-141 deducts `amount` from the sender.
2. NEP-141 credits Aurora's account with `amount - fee` (fee retained by the token contract).
3. NEP-141 calls `ft_on_transfer` on Aurora with `amount` (the pre-fee figure).
4. Aurora mints `amount` ERC-20 tokens to the recipient.

Aurora now holds `amount - fee` NEP-141 tokens but has issued `amount` ERC-20 tokens — a deficit of `fee` per deposit. Each subsequent deposit compounds the deficit.

When a user later calls `withdrawToNear` on the `EvmErc20` contract, it burns the ERC-20 and calls the `ExitToNear` precompile: [4](#0-3) 

The precompile issues an `ft_transfer` to the NEP-141 contract for the full burned amount. Once cumulative over-minting exceeds Aurora's NEP-141 balance, those `ft_transfer` calls fail, permanently freezing the corresponding ERC-20 tokens on Aurora.

---

### Impact Explanation

**Insolvency / Permanent fund freeze.**

The ERC-20 mirror supply on Aurora grows faster than the NEP-141 backing. The last users to attempt `withdrawToNear` find the NEP-141 contract has insufficient balance to honour the transfer. Their ERC-20 tokens are burned but no NEP-141 is received — funds are permanently lost. The deficit is proportional to the fee rate and the total bridged volume, and it accumulates with every deposit.

---

### Likelihood Explanation

**Medium.**

Exploitation requires a fee-on-transfer NEP-141 token to be registered via `deploy_erc20_token` (an admin action). However, fee-on-transfer tokens are common in the NEAR ecosystem (reflection tokens, burn-on-transfer tokens, protocol-fee tokens). Once any such token is registered — even inadvertently — every ordinary `ft_transfer_call` by any token holder silently over-mints. No special attacker capability is needed beyond holding the token.

---

### Recommendation

In `receive_erc20_tokens`, replace the caller-supplied `args.amount` with the actual balance delta of Aurora's NEP-141 account. Concretely:

1. Read Aurora's NEP-141 balance **before** the `ft_transfer_call` completes (this requires a cross-contract read or a storage slot updated by the NEP-141 callback).
2. Alternatively, restructure the deposit flow so that Aurora queries its own NEP-141 balance before and after the transfer and mints only the observed difference — analogous to the "balance-before / balance-after" pattern recommended in the original report.
3. As a defence-in-depth measure, document that fee-on-transfer NEP-141 tokens must not be registered as ERC-20 mirrors, and add a runtime guard (e.g., a dry-run balance check during `deploy_erc20_token`) to detect such tokens before registration.

---

### Proof of Concept

```
1. Admin registers a fee-on-transfer NEP-141 token T (fee = 1%) via deploy_erc20_token.
   → ERC-20 mirror M is deployed on Aurora.

2. Alice calls T.ft_transfer_call(receiver_id=aurora, amount=1000, msg=<alice_evm_addr>).
   → T transfers 990 tokens to Aurora's account (10 fee retained).
   → T calls aurora.ft_on_transfer(sender=alice, amount=1000, msg=<alice_evm_addr>).

3. ft_on_transfer dispatches to receive_erc20_tokens.
   → amount = args.amount.as_u128() = 1000   ← pre-fee figure, line 803
   → Aurora mints 1000 M tokens to Alice.

4. Aurora holds 990 T tokens but has issued 1000 M tokens (deficit: 10).

5. Repeat step 2 N times. Deficit grows to 10*N.

6. Alice calls M.withdrawToNear(recipient=alice.near, amount=1000).
   → EvmErc20.withdrawToNear burns 1000 M tokens.
   → ExitToNear precompile issues T.ft_transfer(receiver=alice.near, amount=1000).
   → T.ft_transfer fails: Aurora's balance < 1000 after enough deposits.
   → Alice's ERC-20 tokens are burned; no NEP-141 received. Funds permanently frozen.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());
