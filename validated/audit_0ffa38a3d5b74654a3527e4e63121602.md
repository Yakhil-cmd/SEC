Audit Report

## Title
Unbounded Neuron Iteration in `compute_ballots_for_new_proposal` Exhausts Instruction Limit - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS Governance canister's `compute_ballots_for_new_proposal` function performs a synchronous, unbounded linear scan over all heap-resident neurons in `self.proto.neurons` with no instruction-limit guard. Any neuron holder can trigger this path via `make_proposal`. If the neuron population is large enough, the per-message instruction budget is exhausted, permanently blocking all proposal submission and rendering the SNS canister ungovernable.

## Finding Description
`compute_ballots_for_new_proposal` iterates over every entry in `self.proto.neurons` (a heap `BTreeMap<String, Neuron>`) without any instruction-limit check: [1](#0-0) 

This function is called synchronously inside `make_proposal` before any state mutation: [2](#0-1) 

The only population guard, `check_neuron_population_can_grow`, prevents adding neurons beyond `max_number_of_neurons` but provides zero protection against the iteration cost of `compute_ballots_for_new_proposal`: [3](#0-2) 

A grep for `noop_self_call_if_over_instructions` in `rs/sns/**` returns **zero matches**. By contrast, the NNS Governance canister uses this guard in `rs/nns/governance/src/voting.rs` and benchmarks its ballot computation against `MAX_NUMBER_OF_NEURONS` with a 25B instruction budget: [4](#0-3) 

The NNS benchmark shows 100 neurons consume ~2.45M instructions. Scaling linearly to the IC's 5B per-message limit, approximately 200,000 neurons exhaust the budget. The SNS `max_number_of_neurons` is configurable up to `MAX_NUMBER_OF_NEURONS_CEILING` (defined in `rs/sns/governance/src/types.rs`), and no equivalent benchmark or projection exists for the SNS path. [5](#0-4) 

## Impact Explanation
Once the neuron population reaches the instruction-exhaustion threshold, every call to `make_proposal` traps with `CanisterInstructionLimitExceeded`. Because SNS upgrades, parameter changes, and all governance actions require proposals, the canister becomes permanently unupgradeable and ungovernable — a complete governance DoS. This matches the **High ($2,000–$10,000)** impact class: *Application/platform-level DoS with concrete user and protocol harm*.

## Likelihood Explanation
Any principal holding SNS tokens can stake neurons; no privileged role is required. The attacker entry point is the public `manage_neuron` update call with `Command::MakeProposal`. Filling the neuron population to the ceiling requires acquiring enough tokens to stake many neurons (an economic cost), but organic growth of a popular SNS can also reach the threshold without any deliberate attack. Once the threshold is crossed, the DoS is permanent and affects all users.

## Recommendation
1. Add an instruction-limit guard to `compute_ballots_for_new_proposal` analogous to the NNS Governance's `noop_self_call_if_over_instructions` pattern in `rs/nns/governance/src/voting.rs`, or move ballot computation to a timer-based async task.
2. Add a canbench benchmark for `compute_ballots_for_new_proposal` that projects against the actual `MAX_NUMBER_OF_NEURONS_CEILING` for SNS (mirroring the NNS `check_projected_instructions` pattern) and enforce the projected instruction count stays well below 5B.
3. Consider migrating SNS neuron storage to stable memory (as NNS did) to reduce per-neuron iteration cost.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons` set to a large value (e.g., the ceiling defined in `rs/sns/governance/src/types.rs`).
2. Stake SNS tokens across many principals until `self.proto.neurons.len()` approaches `max_number_of_neurons`.
3. Call `manage_neuron` with `Command::MakeProposal(...)` from any neuron holder.
4. `make_proposal` calls `compute_ballots_for_new_proposal` at line 3557–3559, iterating all neurons synchronously.
5. The message traps with `CanisterInstructionLimitExceeded`; no proposal can ever be submitted again.
6. A deterministic integration test using PocketIC can reproduce this by pre-populating neurons to the threshold and asserting that `make_proposal` returns a trap error.
