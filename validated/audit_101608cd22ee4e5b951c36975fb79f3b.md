Audit Report

## Title
Unbounded `ingress_messages` Deserialization in `TryFrom<pb::StrippedBlockProposal>` Enables Byzantine Peer to Exhaust Replica Memory and Saturate Tokio Task Scheduler — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

## Summary

`TryFrom<pb::StrippedBlockProposal>` performs no upper-bound check on the count of `ingress_messages` entries before collecting them into a `Vec`. A Byzantine subnet peer can serve a crafted `StrippedBlockProposal` carrying an arbitrarily large `ingress_messages` repeated field under a valid, currently-wanted block proposal hash. `BlockProposalAssembler::new` allocates a `Vec` proportional to that count, and `assemble_message` then spawns one unbounded `get_or_fetch` Tokio task per entry, each of which loops indefinitely with no elapsed-time cap. This allows a single below-threshold Byzantine peer to exhaust replica memory and saturate the Tokio task scheduler.

## Finding Description

**Root cause — no length bound at deserialization.**

`TryFrom<pb::StrippedBlockProposal>` in `stripped.rs` validates that `pruned_block_proposal` is present, that the pruned block's ingress payload is empty, and that `unstripped_consensus_message_id` is a `BlockProposal` hash. It does not check `ingress_messages.len()` before the `collect()` call: [1](#0-0) 

**Proportional allocation in `BlockProposalAssembler::new`.**

Every deserialized `SignedIngressId` is mapped into a `Vec<(SignedIngressId, Option<SignedIngress>)>` with no cap. The developer comment at line 440 reads "We can have at most 1000 elements in the vector" — this is an unenforced assumption, not a runtime check: [2](#0-1) [3](#0-2) 

**One Tokio task spawned per entry.**

`assemble_message` calls `assembler.missing_stripped_messages()` (which returns all N entries since none are in the local pool) and spawns a `get_or_fetch` task for each without any concurrency cap: [4](#0-3) 

**Each spawned task loops indefinitely.**

`download_stripped_message` uses `with_max_elapsed_time(None)` and an unconditional `loop`, meaning each task retries forever until the `JoinSet` is dropped: [5](#0-4) [6](#0-5) 

**ID check does not constrain ingress count.**

`MaybeStrippedConsensusMessage::id()` derives the artifact ID solely from `unstripped_consensus_message_id`, not from the `ingress_messages` field. A Byzantine peer can set this field to any valid, currently-wanted block proposal hash while inflating `ingress_messages` to an arbitrary count — the ID check passes: [7](#0-6) 

**Bouncer has no visibility into ingress count.**

`BouncerFactoryWrapper` delegates to the consensus pool bouncer, which evaluates only the `ConsensusMessageId` (height + hash). It has no visibility into the number of ingress message IDs embedded in the stripped payload: [8](#0-7) 

**The `try_assemble` ID mismatch check is not a mitigation.** It fires only after all N tasks have already been spawned and have been running. The `JoinSet` is only dropped when `assemble_message` returns, by which point the memory and scheduler damage has already occurred.

## Impact Explanation

A single Byzantine peer can craft a `StrippedBlockProposal` proto with up to ~55,000 `ingress_messages` entries within a 4 MB QUIC message (each `pb::StrippedIngressMessage` is ~72 bytes). This causes the victim replica to:

1. Allocate a `Vec` of ~55,000 `(SignedIngressId, Option<SignedIngress>)` entries.
2. Spawn ~55,000 Tokio tasks, each holding a stack frame and looping indefinitely.

With up to `LOOK_AHEAD = 10` concurrent block proposals in flight, the total unbounded task count multiplies. This constitutes a **replica-level DoS**: memory exhaustion and Tokio thread-pool saturation starve legitimate consensus, message routing, and execution work on the targeted node.

This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

The attacker must be a subnet peer (below the Byzantine fault threshold) and must observe a block proposal ID that the victim is currently downloading — both are trivially achievable in normal subnet operation via the public P2P advert mechanism. No privileged access, key material, or majority corruption is required. The crafted message passes all existing validation gates (bouncer, ID check, proto field validation) before any resource allocation occurs. The attack is repeatable for every new block proposal height.

## Recommendation

Add an upper-bound check on `ingress_messages.len()` (and `stripped_idkg_dealings.len()`) inside `TryFrom<pb::StrippedBlockProposal>`, immediately before the `collect()` calls. The bound should be the protocol-enforced maximum: `max_ingress_messages_per_block` (registry parameter, default 1000) for ingress messages, and the per-block IDKG dealing limit for dealings. Return `ProxyDecodeError::Other(...)` if either count exceeds its respective limit. This enforces the invariant already assumed by the comment at line 440 of `assembler.rs`.

## Proof of Concept

```rust
// Craft a StrippedBlockProposal proto with N fake ingress_messages entries.
let mut proto = pb::StrippedBlockProposal {
    pruned_block_proposal: Some(pb::BlockProposal::default()),
    unstripped_consensus_message_id: Some(/* valid, currently-wanted block proposal ID */),
    ingress_messages: (0..55_000)
        .map(|i| pb::StrippedIngressMessage {
            stripped: Some(pb::IngressMessageId {
                expiry: 0,
                message_id: vec![i as u8 % 256; 32],
            }),
            ingress_bytes_hash: vec![0u8; 32],
        })
        .collect(),
    stripped_idkg_dealings: vec![],
};

// Serve this proto in response to the victim's RPC for the wanted block proposal ID.
// Expected observations:
// 1. TryFrom<pb::StrippedBlockProposal> succeeds (no length check).
// 2. BlockProposalAssembler::new allocates a Vec of 55,000 entries.
// 3. assemble_message spawns 55,000 get_or_fetch tasks.
// 4. Each task enters download_stripped_message's infinite loop.
// 5. Memory usage and spawned task count are unbounded until the bouncer fires (~3s later).
// Invariant to assert: ingress_messages.len() <= max_ingress_messages_per_block (1000)
//                      must be checked before collect() in TryFrom.
```

A deterministic integration test can verify this by constructing the above proto, calling `StrippedBlockProposal::try_from(proto)`, and asserting that the result is `Err(ProxyDecodeError::Other(...))` when `ingress_messages.len() > 1000`.
