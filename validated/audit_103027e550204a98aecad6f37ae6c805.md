Audit Report

## Title
Unsolicited Pong Message Causes Forced Disconnection from Bitcoin Peer — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary
`process_pong_message` unconditionally calls `conn.disconnect()` when a `Pong` is received while the connection is in `PingState::Idle`, which is the default state between 120-second ping intervals. Any Bitcoin P2P peer that completes the version handshake can immediately send an unsolicited `Pong` to force the adapter to drop the connection. This deviates from BIP-31 / Bitcoin Core behavior, which requires unsolicited `Pong` messages to be silently ignored.

## Finding Description
In `process_pong_message` ( [1](#0-0) ), the `valid_pong` flag is set to `false` for `PingState::Idle { last_pong_at: _ }`, and the `else` branch unconditionally calls `conn.disconnect()`.

`PingState::Idle` is the initial state of every `Connection` ( [2](#0-1) ) and remains the active state for the full 120-second window between pings ( [3](#0-2) ).

Incoming `NetworkMessage::Pong` is routed directly to `process_pong_message` with no pre-filtering ( [4](#0-3) ). There are no guards that check whether the connection is in `HandshakeComplete` state or rate-limit incoming `Pong` messages before the disconnect path is reached.

## Impact Explanation
This is a **High** severity finding. An attacker operating Bitcoin P2P nodes can keep the adapter below `min_connections = 2` ( [5](#0-4) ), causing `send_to_all` to silently drop all outbound messages ( [6](#0-5) ). This disrupts the Bitcoin adapter's ability to relay transactions and discover blocks, directly impacting ckBTC and Chain Fusion operations — a concrete platform-level DoS against in-scope Chain Fusion infrastructure.

## Likelihood Explanation
- Entry point is the public Bitcoin P2P network — no privileged access required.
- The attack window is the full 120-second idle interval, making it trivially easy to trigger.
- After disconnection, `reap_disconnected` removes the connection ( [7](#0-6) ) and the adapter reconnects, but an attacker controlling or Sybil-attacking the address book can repeatedly reconnect to malicious peers and repeat the attack indefinitely.
- No cryptographic or consensus-level protection prevents a peer from sending arbitrary P2P messages after handshake.

## Recommendation
Replace the `conn.disconnect()` call in the `PingState::Idle` branch with a silent no-op, consistent with BIP-31:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong — ignore per Bitcoin protocol spec (BIP-31)
}
```

Only disconnect on `PingState::ExpectingPong` with a nonce mismatch.

## Proof of Concept
Deterministic unit test (no fuzzing required):
1. Create a `Connection` in `HandshakeComplete` state; `ping_state` defaults to `PingState::Idle`.
2. Call `process_pong_message` with any nonce (e.g., `42`).
3. Assert `conn.state()` is `ConnectionState::NodeDisconnected`.

This is directly provable from the code path at [1](#0-0)  without any probabilistic assumptions.
