## Verdict

### Title
Engine Pause Mid-Flight Strands NEAR Tokens and Blocks EVM Refund in ExitToNear Callback — (`engine/src/contract_methods/connector.rs`)

### Summary
When the engine is paused between the execution of an ExitToNear EVM transaction and its NEAR callback receipt, `exit_to_near_precompile_callback` fails at `require_running` before it can either complete the NEAR-side transfer (wNEAR unwrap) or issue an EVM-side refund (`error_refund` path). The EVM-side debit is already committed and irrecoverable via the normal flow.

### Finding Description
The `ExitToNear` precompile schedules a two-step NEAR receipt chain:

1. **Base promise**: `near_withdraw` (wNEAR unwrap) or `ft_transfer`/`ft_transfer_call` (ERC-20 exit). This executes and commits atomically with the EVM transaction.
2. **Callback**: `exit_to_near_precompile_callback`, which either (a) transfers the unwrapped NEAR to the user, or (b) calls `refund_on_error` if the base promise failed.

The callback opens with `require_running(&state)?` at line 203. If `pause_contract` is called by the owner between receipt 1 and receipt 2 (which is possible because NEAR processes receipts asynchronously across blocks), the callback panics. On NEAR, a panicking callback does **not** revert the already-committed base promise. The result:

- **wNEAR unwrap case**: `near_withdraw` has already moved NEAR tokens into Aurora's account. The callback that should forward them to the user never runs. The user's wNEAR ERC-20 balance was burned in the EVM. The NEAR tokens are stranded in Aurora's account with no automatic recovery path.
- **`error_refund` case**: `ft_transfer` failed on NEAR (tokens not sent), but the EVM-side burn already committed. The callback that should call `refund_on_error` never runs. The user loses their ERC-20 tokens with no refund.

### Impact Explanation
**High — Temporary (potentially permanent) freezing of funds.** For the wNEAR unwrap case, the stranded NEAR tokens can only be recovered by an admin manually intervening (no on-chain recovery function is visible in the codebase). For the `error_refund` case, the EVM-side tokens are permanently lost unless the owner manually mints a replacement. In both cases, the user's funds are frozen for an indefinite period.

### Likelihood Explanation
**Low.** The precondition requires the owner to call `pause_contract` in the narrow window between the base promise receipt and the callback receipt. This is not a deliberate attack — it is an inadvertent consequence of a legitimate operational action (e.g., pausing for a security incident or upgrade). The window is short (typically one NEAR block), but any in-flight ExitToNear-with-callback transaction at the moment of pause is affected. The wNEAR unwrap path is a production-used flow.

### Recommendation
Remove `require_running` from `exit_to_near_precompile_callback`, or replace it with a weaker guard that still allows the callback to complete its refund/transfer logic even when the engine is paused. The callback is a self-call (`assert_private_call` already enforces this) and its only actions are to forward already-committed NEAR tokens or issue an EVM refund — neither of which represents new user-initiated activity that the pause is intended to block.

Alternatively, before committing the EVM state changes in `submit`, verify that the engine will remain running for the duration of the promise chain — though this is not feasible in a decentralized setting.

### Proof of Concept
1. Deploy Aurora engine locally with wNEAR configured.
2. User submits an EVM transaction calling the ExitToNear precompile to unwrap wNEAR (sets `transfer_near` args, schedules callback).
3. After the `submit` receipt commits (EVM balance debited, `near_withdraw` receipt queued), call `pause_contract` as the owner before the callback receipt executes.
4. Observe: `exit_to_near_precompile_callback` panics with `ERR_PAUSED`.
5. Assert: user's wNEAR ERC-20 balance is zero (burned in EVM), Aurora's NEAR balance increased by the unwrapped amount, user received nothing.
6. Resume the engine: no automatic retry occurs; funds remain stranded. [9](#0-8) [10](#0-9) [2](#0-1)
