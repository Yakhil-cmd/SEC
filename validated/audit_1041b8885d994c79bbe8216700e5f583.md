Audit Report

## Title
`u64` Multiplication Overflow in `disburse_maturity` Traps SNS Governance Canister - (File: `rs/sns/governance/src/governance.rs`)

## Summary
`Governance::disburse_maturity` multiplies `maturity_e8s_equivalent` (u64) by `percentage_to_disburse` (cast to u64) using `checked_mul`, then calls `.expect()` on the result. When the product exceeds `u64::MAX`, `checked_mul` returns `None` and `.expect()` panics, trapping the update call. Any neuron controller whose `maturity_e8s_equivalent` meets or exceeds `u64::MAX / percentage_to_disburse` is permanently unable to disburse maturity via this path. The NNS governance already fixed the identical pattern by widening to `u128` first; the SNS governance was never updated.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 1643–1649:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent                                        // u64
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)  // u64 × u64
    .expect("Overflow while processing maturity to disburse.")      // panics on None
    .checked_div(100)
    .expect("Error when processing maturity to disburse.")
    as u128;
``` [1](#0-0) 

Both operands are `u64`. `checked_mul` returns `None` when the product exceeds `u64::MAX ≈ 1.844 × 10¹⁹`. The `.expect()` call then panics, which in the IC execution model causes the update call to **trap** and roll back all state changes. The percentage validation at lines 1633–1640 only checks that `percentage_to_disburse` is between 1 and 100 — it does not guard against arithmetic overflow. [2](#0-1) 

The NNS governance already fixed the identical pattern in `rs/nns/governance/src/governance/disburse_maturity.rs` at lines 225–244 by widening both operands to `u128` before multiplying:

```rust
(total_maturity_e8s as u128)
    .checked_mul(percentage_to_disburse as u128)
    ...
``` [3](#0-2) 

The SNS governance `disburse_maturity` was never updated to apply the same widening.

## Impact Explanation
A neuron controller whose `maturity_e8s_equivalent ≥ u64::MAX / percentage_to_disburse` cannot disburse any portion of their maturity. Every call to `manage_neuron { DisburseMaturity { ... } }` traps the SNS governance canister update call. The maturity remains locked in the neuron but is permanently inaccessible via this path — the neuron controller loses the ability to convert accumulated maturity into liquid SNS tokens. This constitutes a **significant SNS governance security impact with concrete user financial harm**, matching the High ($2,000–$10,000) impact tier: "Significant SNS or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The overflow threshold at 100% disbursement is `u64::MAX / 100 ≈ 1.844 × 10¹⁷ e8s ≈ 1.844 billion tokens` of maturity. SNS tokens can have very large total supplies; a neuron holding a large stake over a long staking period can accumulate maturity in this range for large SNS DAOs. The entry path is a standard unprivileged `manage_neuron` ingress call — no special role or key is required beyond controlling the neuron. The condition is self-inflicted (the neuron controller's own maturity triggers it) but is not a user mistake — it is a code defect that silently becomes a permanent trap once the threshold is crossed.

## Recommendation
Widen `maturity_e8s_equivalent` to `u128` before multiplying, matching the NNS governance fix:

```rust
// After (safe):
let maturity_to_deduct = (neuron.maturity_e8s_equivalent as u128)
    .checked_mul(disburse_maturity.percentage_to_disburse as u128)
    .and_then(|v| v.checked_div(100))
    .expect("Overflow while processing maturity to disburse.")
    as u64;
```

This mirrors the pattern already used in `rs/nns/governance/src/governance/disburse_maturity.rs`. [4](#0-3) 

## Proof of Concept
1. Deploy an SNS with a token that has a large total supply.
2. Create a neuron and accumulate `maturity_e8s_equivalent ≥ u64::MAX / 100 ≈ 1.844 × 10¹⁷` (achievable for large SNS DAOs with billions of tokens in supply).
3. Call `manage_neuron` with `DisburseMaturity { percentage_to_disburse: 100, to_account: None }` as the neuron controller.
4. The SNS governance canister traps with `"Overflow while processing maturity to disburse."` — the call returns an error and no maturity is disbursed.
5. Repeat with any `percentage_to_disburse` value; the call always traps as long as `maturity_e8s_equivalent * percentage_to_disburse > u64::MAX`.
6. A deterministic unit test can reproduce this by constructing a neuron with `maturity_e8s_equivalent = u64::MAX / 50 + 1` and calling `disburse_maturity` with `percentage_to_disburse = 50`, asserting the call panics.

fn percentage_of_maturity(
    total_maturity_e8s: u64,
    percentage_to_disburse: u32,
) -> Result<u64, InitiateMaturityDisbursementError> {
    (total_maturity_e8s as u128)
        .checked_mul(percentage_to_disburse as u128)
        .and_then(|result| result.checked_div(100))
        .and_then(|result| {
