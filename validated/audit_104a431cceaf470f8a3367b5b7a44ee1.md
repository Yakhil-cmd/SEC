Audit Report

## Title
SNS Governance Distributes Voting Rewards During `PreInitializationSwap` Mode Without Mode Guard - (File: rs/sns/governance/src/governance.rs)

## Summary
`run_periodic_tasks()` calls `should_distribute_rewards()` and then `distribute_rewards()` without checking whether the SNS governance canister is in `PreInitializationSwap` mode. Because `MakeProposal` and `RegisterVote` are explicitly permitted in that mode, developer neurons can submit and vote on proposals, causing the periodic reward distribution to accrue maturity to those neurons during the swap period — before any swap participant receives a neuron.

## Finding Description
`run_periodic_tasks()` (L5471–5534) unconditionally calls `self.should_distribute_rewards()` (L5503) and, if true, `self.distribute_rewards(supply)` (L5513). Neither function contains a mode check.

`should_distribute_rewards()` (L5725–5753) returns `true` solely based on whether `voting_rewards_parameters` is set and whether enough time has elapsed since the last reward event. It never inspects `self.mode()`.

`distribute_rewards()` (L5763+) similarly performs no mode check and directly increments `neuron.maturity_e8s_equivalent` (L5994) for every neuron that voted on settled proposals.

Meanwhile, `manage_neuron_command_is_allowed_in_pre_initialization_swap_or_err()` in `rs/sns/governance/src/types.rs` (L182–211) explicitly permits `C::MakeProposal` and `C::RegisterVote` in `PreInitializationSwap` mode. This means developer neurons can submit `Motion` proposals and vote on them during the swap period, creating settled ballots that `distribute_rewards()` will process.

The test `test_disallow_enabling_voting_rewards_while_in_pre_initialization_swap` (L815–871 of `assorted_governance_tests.rs`) confirms the design intent: voting rewards must not be activatable during `PreInitializationSwap` mode. However, this guard only blocks enabling rewards via a `ManageNervousSystemParameters` proposal. If `voting_rewards_parameters` is already set at genesis — the standard SNS deployment pattern — the periodic task distributes rewards regardless of mode.

Exploit path:
1. SNS is deployed with `voting_rewards_parameters` set (e.g., `round_duration_seconds = 86400`) and enters `PreInitializationSwap` mode.
2. Developer neuron controller calls `manage_neuron` with `Command::MakeProposal(Motion {...})` — succeeds because `MakeProposal` is explicitly allowed.
3. Developer neuron votes via `Command::RegisterVote` — also explicitly allowed.
4. After `round_duration_seconds` elapses, `run_periodic_tasks()` fires; `should_distribute_rewards()` returns `true`; `distribute_rewards()` accrues maturity to the developer neuron.
5. After swap finalizes and mode becomes `Normal`, the developer neuron controller calls `DisburseMaturity`, extracting tokens accrued during the swap period — an advantage unavailable to swap participants who only receive neurons after finalization.

## Impact Explanation
Developer neurons accumulate maturity (convertible to tokens) during the swap period while swap participants cannot. This distorts the economic fairness of the initial SNS token distribution. The maturity amount scales with the configured reward rate, swap duration, and developer neurons' voting power share — all of which are non-trivial in production SNS deployments. This constitutes a concrete, measurable economic harm to swap participants and falls under the High impact category: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
All required conditions are routinely met in production SNS deployments: `voting_rewards_parameters` is set at genesis, the swap period spans at least one reward round (typically one day), and developer neurons can submit `Motion` proposals without any special privileges. The exploit requires only an unprivileged ingress call to `manage_neuron` from a developer neuron controller. No governance majority, subnet compromise, or leaked keys are needed.

## Recommendation
Add a mode guard inside `should_distribute_rewards()`:

```rust
fn should_distribute_rewards(&self) -> bool {
    if self.mode() == governance::Mode::PreInitializationSwap {
        return false;
    }
    // ... existing logic
}
```

This mirrors the design intent already expressed in `test_disallow_enabling_voting_rewards_while_in_pre_initialization_swap` and closes the gap between the proposal-path guard and the periodic-task path.

## Proof of Concept
Write an integration test using `PocketIC` or the existing `NativeEnvironment` harness:
1. Construct a `GovernanceProto` with `mode = Mode::PreInitializationSwap`, `voting_rewards_parameters` set with `round_duration_seconds = 1`, and a developer neuron with voting power.
2. Call `make_proposal` with a `Motion` action — assert it succeeds.
3. Call `register_vote` for the developer neuron — assert it succeeds.
4. Advance the mock clock by 2 seconds (past one reward round).
5. Call `run_periodic_tasks()`.
6. Assert that the developer neuron's `maturity_e8s_equivalent` has increased — demonstrating rewards were distributed during `PreInitializationSwap` mode.
7. Apply the fix (mode guard in `should_distribute_rewards`) and re-run; assert maturity remains zero.
