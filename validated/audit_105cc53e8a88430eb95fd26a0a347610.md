### Title
Unbounded `fixed_gas` in Silo Mode Allows Owner to Freeze All User Transactions - (File: `engine/src/contract_methods/silo/mod.rs`)

---

### Summary
In Silo mode, `set_fixed_gas` and `set_silo_params` allow the owner to set `fixed_gas` to any `EthGas` (`u64`) value with no upper or lower bounds validation. Setting it to an extreme value (e.g., `u64::MAX`) causes every user transaction to fail, permanently freezing all user funds in the Silo deployment.

---

### Finding Description

`set_fixed_gas` in `engine/src/contract_methods/silo/mod.rs` writes the caller-supplied value directly to storage with zero validation: [1](#0-0) 

`set_silo_params` delegates to `set_fixed_gas` without any bounds check either: [2](#0-1) 

Both are gated only by `require_owner_and_running`: [3](#0-2) 

When `fixed_gas = u64::MAX` is stored, two independent failure paths block every subsequent user transaction:

**Path 1 — `FixedGasOverflow`:** Before `charge_gas` is called, the engine checks: [4](#0-3) 

Any transaction whose `gas_limit < u64::MAX` is rejected here. Standard transactions use gas limits in the tens-of-thousands range, so every normal transaction fails immediately.

**Path 2 — `EthAmountOverflow`:** If a user crafts a transaction with `gas_limit = u64::MAX` to bypass Path 1, the `charge_gas` multiplication overflows: [5](#0-4) 

`u64::MAX * effective_gas_price` overflows `U256` for any `effective_gas_price > 1`, returning `GasPaymentError::EthAmountOverflow`. Even at `effective_gas_price = 1`, the required prepayment is `u64::MAX` Wei ≈ 18.4 ETH — unaffordable for virtually all users.

The `EthGas` type itself imposes no ceiling: [6](#0-5) 

---

### Impact Explanation

All user transactions in the affected Silo deployment fail. Users cannot transfer funds, interact with EVM contracts, or withdraw their assets. This constitutes **High (temporary) to Critical (permanent) freezing of funds**, depending on whether the owner ever corrects the value. Because the Silo owner controls the parameter unilaterally and there is no on-chain time-lock or governance veto for users, the freeze can be made permanent.

---

### Likelihood Explanation

The Silo mode is explicitly designed for third-party operators who deploy their own Aurora instances and set their own gas policies. The `set_fixed_gas` / `set_silo_params` call path is a normal operational action. There is no protocol-enforced ceiling, no multi-sig requirement, and no delay before the new value takes effect. A single owner transaction is sufficient to freeze all user funds with no recourse.

---

### Recommendation

Add an explicit maximum bound for `fixed_gas` inside `set_fixed_gas` (and transitively `set_silo_params`). Define a protocol constant (e.g., `MAX_FIXED_GAS: EthGas = EthGas::new(30_000_000)`) and reject any value exceeding it with a descriptive error. This mirrors the recommendation in the original report: define immutable upper/lower limits and enforce them at the setter.

---

### Proof of Concept

1. Owner calls `set_silo_params` (or `set_fixed_gas`) with `fixed_gas = u64::MAX`.
2. User submits a standard ETH transfer with `gas_limit = 30_000`, `gas_price = 1`.
3. Engine evaluates `u64::MAX > 30_000` → `true` → returns `EngineErrorKind::FixedGasOverflow`.
4. User raises `gas_limit` to `u64::MAX` to attempt bypass; with any non-zero `gas_price > 1`, `checked_mul` returns `None` → `GasPaymentError::EthAmountOverflow`.
5. At `gas_price = 1`, required prepayment = 18.4 ETH — effectively unaffordable.
6. All user transactions in the Silo are frozen. Funds held in EVM accounts are inaccessible.

/// Set an amount of gas per transaction.
pub fn set_fixed_gas<I: IO>(io: &mut I, gas: Option<EthGas>) {
    let key = fixed_gas_key();

    if let Some(gas) = gas {
        io.write_borsh(&key, &gas);
    } else {
        io.remove_storage(&key);
    }
}
