Audit Report

## Title
Silent permanent ckBTC loss via UTXO-count-dependent vsize exceeding `PER_REQUEST_VSIZE_BOUND` in `AmountTooLow` path — (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary
`fee_based_minimum_withdrawal_amount` computes the minimum withdrawal threshold using a fixed 221-vbyte estimate. When the minter's UTXO set is fragmented into many small UTXOs, the actual transaction vsize can far exceed this bound, causing `evaluate_transaction_fee` to return a fee larger than the withdrawal amount. The resulting `BuildTxError::AmountTooLow` branch in `submit_pending_requests` finalizes the request with `FinalizedStatus::AmountTooLow` and issues **no reimbursement**, permanently destroying the user's burned ckBTC.

## Finding Description

**Root cause — fixed vsize bound in minimum amount calculation:**

`fee_based_minimum_withdrawal_amount` at [1](#0-0)  uses `PER_REQUEST_VSIZE_BOUND = 221` vbytes unconditionally. This corresponds to a ~1-input P2WPKH transaction and does not account for multi-input transactions.

**Acceptance gate burns ckBTC before transaction construction:**

`retrieve_btc` at [2](#0-1)  gates on `fee_based_retrieve_btc_min_amount`. If the amount passes, ckBTC is burned unconditionally at [3](#0-2)  before any transaction is built.

**Actual fee uses real vsize:**

`evaluate_transaction_fee` at [4](#0-3)  calls `fake_sign(tx).vsize()` on the actual constructed transaction. The check at [5](#0-4)  returns `BuildTxError::AmountTooLow` if `fee + minter_fee > amount`. With 10 P2WPKH inputs, vsize ≈ 750 vbytes — 3.4× the 221-vbyte bound.

**`AmountTooLow` path silently drops the request with no reimbursement:**

The `BuildTxError::AmountTooLow` branch at [6](#0-5)  calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` and returns `None`. No reimbursement is issued.

This is in direct contrast to the `BuildTxError::InvalidTransaction` branch at [7](#0-6) , which calls `reimburse_canceled_requests`. The `FinalizedStatus` enum at [8](#0-7)  has no reimbursement variant for `AmountTooLow`.

`remove_retrieve_btc_request` at [9](#0-8)  only records the event and pushes to finalized requests — no mint-back or reimbursement task is enqueued.

**UTXO fragmentation path:**

`utxos_selection` at [10](#0-9)  uses `greedy` to select UTXOs. If the minter holds only small UTXOs, many inputs are required to cover the withdrawal amount, linearly increasing vsize.

## Impact Explanation

Permanent loss of user ckBTC. Burned tokens are not reimbursed and cannot be recovered. This is a direct financial loss in the ckBTC system, an in-scope Chain Fusion / ck-token financial integration. Severity: **High** — significant ck-token security impact with concrete, irreversible user fund loss. The per-event loss is bounded by the minimum withdrawal amount, but the condition is repeatable.

## Likelihood Explanation

Triggerable by an unprivileged user without special privileges. An attacker can deliberately fragment the minter's UTXO set by making many small BTC deposits, then submit a withdrawal at exactly the computed minimum amount during elevated fee rates. The condition also arises naturally: any user submitting a minimum-amount withdrawal when the minter's UTXO set is organically fragmented and fees are elevated will lose funds with no action required from an attacker. No governance majority, subnet compromise, or key leakage is required.

## Recommendation

In the `BuildTxError::AmountTooLow` branch of `submit_pending_requests` (lib.rs ~L412), call `reimburse_canceled_requests` with an appropriate `WithdrawalReimbursementReason` instead of silently finalizing with `FinalizedStatus::AmountTooLow`, consistent with the `InvalidTransaction` branch. Additionally, replace the fixed `PER_REQUEST_VSIZE_BOUND = 221` in `fee_based_minimum_withdrawal_amount` with a more conservative bound that accounts for multi-input transactions (e.g., parameterized by expected UTXO count or using a worst-case bound).

## Proof of Concept

State-machine integration test:
1. Initialize minter with 10 UTXOs of 10,000 satoshi each (total 100,000 sat).
2. Set fee percentiles to yield 200,000 millisat/vbyte; trigger fee update so `fee_based_retrieve_btc_min_amount` computes to 100,000 satoshi.
3. Call `retrieve_btc` for exactly 100,000 satoshi — accepted, ckBTC burned.
4. Advance time past `max_time_in_queue_nanos`.
5. Trigger `submit_pending_requests`.
6. Assert `retrieve_btc_status(block_index) != AmountTooLow` — it should be `Submitted` or `WillReimburse`. The assertion fails, demonstrating that ckBTC was burned and the request was silently dropped with no reimbursement.
