Audit Report

## Title
Third-Party Caller Can Force-Register Victim's ICP Participation, Bypassing `confirmation_text` Consent - (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` endpoint in the SNS Swap canister resolves the buyer principal from caller-supplied input without verifying it matches the actual message caller. Because the `confirmation_text` consent check only validates the string value (which is publicly readable from `get_state`/`get_init`) and not the caller's identity, any unprivileged ingress sender can force-register an arbitrary victim's ICP participation. Once registered, the victim's ICP is locked in escrow and swept to SNS governance upon swap commitment — without the victim ever having consented.

## Finding Description

**Root cause — no caller == buyer check in `canister.rs`:**

In `rs/sns/swap/canister/canister.rs` at L130–134, when `arg.buyer` is non-empty the resolved principal is used directly with no assertion that it equals `caller_principal_id()`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller check
};
``` [1](#0-0) 

This `p` is passed directly to `refresh_buyer_token_e8s`. [2](#0-1) 

**`confirmation_text` check is caller-agnostic and the text is public:**

`validate_confirmation_text` at `rs/sns/swap/src/swap.rs` L363–384 only compares the submitted string against the SNS-configured value — it does not check who submitted it: [3](#0-2) 

The `confirmation_text` is stored in `Init` and returned verbatim by both the `get_state` query (L2964–2975) and the `get_init` query — both are unauthenticated public endpoints: [4](#0-3) [5](#0-4) 

The proto definition confirms `confirmation_text` is part of the publicly-readable `Init` message: [6](#0-5) 

**Ticket system does not block a third-party call:**

The ticket check at `rs/sns/swap/src/swap.rs` L1250–1272 only applies if a ticket already exists for the buyer. The code explicitly states: *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket"* — so an attacker calling without a ticket for the victim proceeds unimpeded: [7](#0-6) 

**Victim's participation is registered:**

On success, the victim is inserted into `self.buyers`: [8](#0-7) 

**`error_refund_icp` escape hatch is closed:**

During `Open`, `error_refund_icp` is unconditionally rejected: [9](#0-8) 

After the swap closes, the victim is now in `self.buyers` with `transfer_success_timestamp_seconds == 0` (not yet disbursed), so the escrow guard blocks the refund: [10](#0-9) 

The `error_refund_icp` escape hatch is explicitly designed for principals who *never* called `refresh_buyer_tokens`. Once the attacker force-registers the victim, that path is closed until the normal sweep completes — at which point the ICP is already gone.

## Impact Explanation

This is a **High** severity finding matching: *"Unauthorized access to… canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

If the swap commits, the victim's ICP is swept to SNS governance and they receive SNS tokens in exchange — without ever having consented. The victim suffers a direct, irreversible financial loss (ICP exchanged for potentially lower-value SNS tokens). The `confirmation_text` field is documented as a participant consent mechanism; this attack renders it meaningless for any victim who has ICP sitting in the swap subaccount. The constraint (victim must have ICP in the subaccount) is realistic: sending ICP to the subaccount is the normal first step of participation, and users may reconsider or send funds accidentally.

## Likelihood Explanation

- **Attacker capability**: Any unprivileged ingress sender. No special role or key required.
- **Required information**: Victim's principal ID (observable on-chain) and the `confirmation_text` (publicly readable via `get_state` or `get_init`).
- **Required precondition**: Victim has ICP in their swap subaccount — a common state during the `Open` lifecycle.
- **Attack window**: The entire `Open` lifecycle, which can span days to weeks.
- **Execution cost**: A single `update` call.

Likelihood is **medium-high**: the precondition is realistic and the attack is trivially executable once met.

## Recommendation

1. **Enforce `caller == buyer`**: In `canister.rs`, when `arg.buyer` is non-empty, assert `PrincipalId::from_str(&arg.buyer).unwrap() == caller_principal_id()`, or reject non-empty `buyer` fields from non-privileged callers entirely.
2. **Alternatively, remove the `buyer` override field** and always derive the buyer from `caller_principal_id()`. Third-party notification of participation is not a necessary feature and introduces this consent bypass.
3. If third-party notification is intentionally required (e.g., for automation), the `confirmation_text` requirement must be tied to the **buyer's** explicit submission — for example, by requiring the buyer to pre-register consent via a separate signed call before a third party can trigger `refresh_buyer_tokens` on their behalf.

## Proof of Concept

1. Deploy SNS swap in `Open` state with `confirmation_text = "I agree to participate"`.
2. Victim sends 10 ICP to `principal_to_subaccount(victim_principal)` on the ICP ledger targeting the swap canister, then reconsiders.
3. Attacker queries `get_init` (unauthenticated) to read `confirmation_text = "I agree to participate"`.
4. Attacker submits ingress as any principal:
   ```
   refresh_buyer_tokens({
     buyer: "<victim_principal_text>",
     confirmation_text: Some("I agree to participate")
   })
   ```
5. Canister resolves `p = victim_principal`, `validate_confirmation_text` passes (text matches), no ticket exists for victim so ticket check is skipped, ledger balance query returns 10 ICP, victim is inserted into `self.buyers` with `amount_icp_e8s = 10_ICP`.
6. Victim calls `error_refund_icp` — rejected: *"Error refunds can only be performed when the swap is ABORTED or COMMITTED"* (swap still `Open`).
7. Swap commits. Victim's 10 ICP is swept to SNS governance; victim receives SNS tokens having never consented.
8. Victim calls `error_refund_icp` post-commit — rejected: *"ICP cannot be refunded as principal X has Y ICP (e8s) in escrow"* (sweep not yet marked complete), or if sweep completed, the subaccount is empty and nothing is returned.

A deterministic integration test can reproduce this using `PocketIC` or `StateMachine` by calling `refresh_buyer_tokens` with `sender = attacker_principal` but `buyer = victim_principal_text`, verifying the victim appears in `get_buyer_state`, and confirming `error_refund_icp` is blocked.
