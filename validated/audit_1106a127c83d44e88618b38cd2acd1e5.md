Audit Report

## Title
Missing Pairing-Equation Check at Share Admission Allows Single Byzantine Node to Force O(n) Fallback on Every VetKD Request — (`rs/crypto/src/vetkd/mod.rs`, `rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs`)

## Summary

`verify_encrypted_key_share_internal` validates only the Ed25519 node signature over the raw share bytes; it never verifies the BLS12-381 pairing relationships of the three curve points inside the share. A single Byzantine subnet node can therefore submit a share whose curve points individually pass deserialization but whose pairing equations are wrong. This causes `EncryptedKey::combine_all` to return `InvalidShares` on every request, unconditionally triggering the `combine_valid_shares` fallback, which performs up to O(n) individual pairing checks instead of the single check in the fast path — a ~34× increase in pairing work per request on a 34-node subnet.

## Finding Description

**Pool admission gate — Ed25519 only.**
`verify_encrypted_key_share_internal` (called from `signer.rs` L513–523) performs only a `BasicSigVerifierInternal::verify_basic_sig` over the raw bytes of `encrypted_key_share`. There is no call to `EncryptedKeyShare::is_valid`. The pairing relationships of `(c1 ∈ G1, c2 ∈ G2, c3 ∈ G1)` are never checked at admission time. [1](#0-0) 

**Deserialization accepts any valid curve points.**
`EncryptedKeyShare::deserialize` checks only that each component is a valid curve point; it does not verify the pairing equations. A Byzantine node can craft a 192-byte blob of three valid-but-unrelated G1/G2/G1 points, sign it with its Ed25519 key, and have it admitted to the validated pool. [2](#0-1) 

**`combine_all` fails with `InvalidShares` when any share is wrong.**
`combine_all` performs Lagrange interpolation over all shares (cheap), then calls `is_valid` on the combined result. One bad share corrupts the interpolated result, so `is_valid` fails and `InvalidShares` is returned. [3](#0-2) 

**`InvalidShares` unconditionally triggers the expensive fallback.**
`combine_encrypted_key_shares_internal` catches `InvalidShares` and falls back to `combine_valid_shares`, which requires computing individual node public keys from the NI-DKG transcript for every share. [4](#0-3) 

**`combine_valid_shares` does O(n) pairing checks.**
`combine_valid_shares` iterates through every share and calls `node_eks.is_valid(...)` on each, which internally calls `check_validity` with two `Gt::multipairing` operations per share. [5](#0-4) [6](#0-5) 

The existing guards are insufficient: the Ed25519 check at admission time only authenticates the bytes as coming from the claimed node; it provides no assurance that the curve points satisfy the required pairing equations.

## Impact Explanation

Every `vetkd_derive_key` request processed by an honest combiner incurs up to O(n) pairing checks (up to 68 `Gt::multipairing` calls for a 34-node subnet) instead of the 2 calls in the fast path — a ~34× increase in cryptographic work per request. Under concurrent load this degrades vetkd throughput for the entire subnet. The system remains correct (the final key is valid, the Byzantine share is excluded), but availability is persistently degraded for all vetkd operations on the subnet. This matches the **High** impact class: application/platform-level subnet availability impact not based on raw volumetric DDoS.

## Likelihood Explanation

The attack requires exactly one Byzantine node in the subnet committee — well below the fault threshold. The node needs only to generate three random valid BLS12-381 curve points, concatenate them into a 192-byte blob, sign with its Ed25519 key, and broadcast. No special capability beyond subnet membership is required. The attack is persistent across all concurrent requests as long as the node remains in the committee, and it is trivially repeatable.

## Recommendation

Add a cryptographic validity check inside `verify_encrypted_key_share_internal`. After the Ed25519 check passes, deserialize the share and call `EncryptedKeyShare::is_valid` with the signer's individual node public key (derivable from the NI-DKG transcript via `lazily_calculated_public_key_from_store`). Shares that fail this check should be rejected at pool admission with a `VetKdKeyShareVerificationError`, preventing them from ever reaching `combine_encrypted_key_shares_internal`. This makes the fast `combine_all` path the normal case even in the presence of Byzantine nodes.

## Proof of Concept

```
1. In a 13-node test committee (threshold = 9):
   - Generate 12 honest shares via EncryptedKeyShare::create
   - Generate 1 Byzantine share: pick three random valid G1/G2/G1 points,
     concatenate (192 bytes), sign with the Byzantine node's Ed25519 key
   - Call verify_encrypted_key_share for the Byzantine share → passes (Ed25519 OK)
   - Call combine_encrypted_key_shares with all 13 shares

2. Observe:
   - combine_all returns InvalidShares (combined result fails is_valid)
   - combine_valid_shares is invoked, performing 13 is_valid calls
   - Each is_valid call does 2 Gt::multipairing operations (26 total vs. 2 baseline)

3. The existing benchmark at
   rs/crypto/internal/crypto_lib/bls12_381/vetkd/benches/vetkd.rs
   already benchmarks combine_valid_shares for n=13 and n=34.
   Add a companion benchmark for the combine_all fast path (0 bad shares)
   and the attack path (1 bad share) to directly measure the latency ratio.
   Assert ratio ≈ n (bounded only by committee size, not threshold).
```

fn check_validity(
    c1: &G1Affine,
    c2: &G2Affine,
    c3: &G1Affine,
    tpk: &TransportPublicKey,
    verification_pk: &G2Affine,
    msg: &G1Affine,
) -> bool {
    let neg_g2_g = G2Prepared::neg_generator();
    let c2_prepared = G2Prepared::from(c2);

    // check e(c1,g2) == e(g1, c2)
    let c1_c2 = Gt::multipairing(&[(c1, neg_g2_g), (G1Affine::generator(), &c2_prepared)]);
    if !c1_c2.is_identity() {
        return false;
    }

    let verification_key_prepared = G2Prepared::from(verification_pk);

    // check e(c3, g2) == e(tpk, c2) * e(msg, dpki)
    let c3_c2_msg = Gt::multipairing(&[
        (c3, neg_g2_g),
        (tpk.point(), &c2_prepared),
        (msg, &verification_key_prepared),
    ]);

    if !c3_c2_msg.is_identity() {
        return false;
    }

    true
