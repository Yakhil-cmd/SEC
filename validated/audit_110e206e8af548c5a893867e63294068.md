Audit Report

## Title
Unbounded Synchronous Iteration in SNS `distribute_rewards` Causes Instruction-Limit Trap, Permanently Blocking Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary

SNS governance's `distribute_rewards` function iterates over all ballots of all considered proposals and all voting neurons in a single synchronous heartbeat execution with no instruction-limit guard. The IC's 5-billion-instruction message limit can be exceeded when neuron and proposal counts approach their configured ceilings. Because the heartbeat is the only mechanism that calls `distribute_rewards` and no batched-timer fallback exists, a trap permanently halts reward distribution for all SNS participants. NNS governance explicitly fixed this identical class of bug by moving reward distribution to a batched timer with an `is_over_instructions_limit` guard; SNS governance has no equivalent protection.

## Finding Description

**Root cause — unbounded double loop in `distribute_rewards`:**

`distribute_rewards` at `rs/sns/governance/src/governance.rs` L5892–5997 performs two unbounded loops:

1. An outer loop over all `considered_proposals` (up to `max_number_of_proposals_with_ballots` proposals, ceiling 700), with an inner loop over every ballot in each proposal (up to `max_number_of_neurons` entries, ceiling 200,000), performing `NeuronId::from_str`, `HashMap` entry operations, and `Decimal` arithmetic per entry.
2. A second loop over every neuron that voted, performing a mutable neuron lookup and maturity update per neuron. [1](#0-0) 

There is no call to any instruction-counter guard, no checkpoint, and no timer-based continuation anywhere in this function or in its caller.

**Synchronous call path — `run_periodic_tasks`:**

`distribute_rewards` is called directly and synchronously inside `run_periodic_tasks` at L5509–5520, which is the SNS heartbeat handler. If `distribute_rewards` traps on instruction exhaustion, the entire heartbeat state is rolled back. The proposals remain in `ReadyToSettle` state, so the next heartbeat invocation will attempt the same computation, trap again, and so on indefinitely. [2](#0-1) 

**No instruction guard exists in SNS governance:**

A grep for `is_message_over_threshold`, `is_over_instructions`, and related patterns across all SNS governance Rust sources returns zero matches. The guard is entirely absent.

**Confirmed bounds:**

`MAX_NUMBER_OF_NEURONS_CEILING` = 200,000 and `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING` = 700 are confirmed governance-parameter upper bounds. [3](#0-2) 

Worst-case ballot entries: 700 × 200,000 = 140 million. The NNS canbench benchmark for `distribute_rewards_with_stable_neurons` records ~2,867,879 instructions for a small neuron set (≈100 neurons, 1 proposal), implying roughly 28,000+ instructions per neuron-proposal pair. Extrapolating to even a modest 10,000 neurons × 100 proposals = 1 million entries yields ~28 billion instructions — far exceeding the 5-billion-instruction application-subnet limit. [4](#0-3) 

**Contrast with NNS governance (fixed):**

NNS governance's `continue_processing` in `rs/nns/governance/src/reward/distribution.rs` L154–188 uses a `while let` loop that calls `is_over_instructions_limit()` after each neuron update and breaks to resume in the next timer invocation. The NNS CHANGELOG explicitly documents this fix. [5](#0-4) 

**Secondary surface — `compute_ballots_for_new_proposal`:**

The same unbounded iteration over all neurons occurs synchronously during proposal submission at L5255–5280. At 200,000 neurons × ~24,500 instructions/neuron (per NNS benchmark) ≈ 4.9 billion instructions, approaching the limit for a single update call. [6](#0-5) 

## Impact Explanation

When the SNS neuron population and accumulated proposals are large enough to push `distribute_rewards` past the 5-billion-instruction limit, the heartbeat traps on every invocation. Because state is rolled back on trap, proposals remain in `ReadyToSettle` indefinitely. No rewards are ever distributed. All SNS participants who voted on proposals permanently lose their accrued maturity increments for the affected period. Secondarily, if `compute_ballots_for_new_proposal` also traps, no new proposals can be submitted, freezing governance entirely.

This matches the **High** impact category: *Significant SNS infrastructure security impact with concrete user or protocol harm* — specifically, permanent application-level DoS of SNS reward distribution triggered without any special privilege.

## Likelihood Explanation

The triggering conditions require no special privilege:
- Setting `max_number_of_neurons` to a large value is a governance decision made at SNS launch or via proposal; the ceiling of 200,000 is explicitly permitted.
- Staking SNS tokens to claim neurons is a standard, unprivileged user action (`manage_neuron` → `ClaimOrRefresh`).
- Proposal accumulation occurs through normal governance activity.

The condition worsens monotonically as the SNS matures. No attacker coordination is required; organic growth of a successful SNS with active governance is sufficient. The threshold at which the trap occurs is well below the theoretical ceiling — even a few thousand neurons with hundreds of proposals can exceed the limit given the per-entry instruction cost observed in NNS benchmarks.

## Recommendation

1. Move SNS `distribute_rewards` out of the synchronous `run_periodic_tasks` heartbeat and into a dedicated periodic timer task, mirroring the NNS governance approach.
2. Add an instruction-counter guard (equivalent to `is_over_instructions_limit`) inside the reward-distribution loop so that work is checkpointed and resumed across multiple messages.
3. Add a similar guard or snapshot-based approach to `compute_ballots_for_new_proposal`, as NNS governance does via `compute_voting_power_snapshot_for_standard_proposal` with `with_active_neurons_iter_sections`.

## Proof of Concept

**Deterministic reasoning (no mainnet execution required):**

1. Deploy an SNS with `max_number_of_neurons` = 200,000 (within the allowed ceiling per `MAX_NUMBER_OF_NEURONS_CEILING`).
2. Have users stake SNS tokens and call `manage_neuron` → `ClaimOrRefresh` until the neuron count is large (e.g., 10,000+). This is an unprivileged operation.
3. Submit and allow to expire enough proposals so that `ready_to_be_settled_proposal_ids()` returns a large set (e.g., 100+). Each proposal records a ballot for every eligible neuron.
4. When the reward round ends, `run_periodic_tasks` calls `distribute_rewards`. The function iterates over N_proposals × N_neurons ballot entries. At 10,000 neurons × 100 proposals = 1 million entries × ~28,000 instructions/entry ≈ 28 billion instructions — exceeding the 5-billion-instruction limit.
5. The heartbeat traps. State rolls back. Proposals remain in `ReadyToSettle`. Every subsequent heartbeat repeats steps 4–5. Reward distribution is permanently blocked.

A local PocketIC or replica integration test can reproduce this deterministically by populating the SNS governance state with the required neuron and proposal counts and invoking the heartbeat, then asserting that no `RewardEvent` is ever created and that the latest reward event timestamp does not advance.
