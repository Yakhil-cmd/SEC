### Title
Unchecked NEAR Promise Result in `ExitToEthereum` Precompile Leads to Permanent Fund Loss - (File: engine-precompiles/src/native.rs)

### Summary
The `ExitToEthereum` precompile schedules a NEAR `withdraw` promise on the eth connector contract but never attaches a callback to verify the promise succeeded. If the promise fails, the EVM-side token burn is already committed and irrevocable, resulting in permanent loss of user funds. This is the direct Aurora Engine analog of the unchecked return value / silent failure pattern described in the external report.

### Finding Description

When a user exits ERC-20 tokens to Ethereum via the `ExitToEthereum` precompile, the flow is:

1. The ERC-20 contract burns the user's tokens (EVM state change, committed atomically).
2. The ERC-20 contract calls the `ExitToEthereum` precompile, which constructs a NEAR promise to call `withdraw` on the eth connector.
3. The promise is emitted as a log and later scheduled by the engine as a real NEAR cross-contract call.

The critical issue is in the promise construction at the end of `ExitToEthereum::run`:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

The promise is always `PromiseArgs::Create` — a fire-and-forget call with **no callback attached**. There is no `PromiseArgs::Callback` wrapping, no `exit_to_ethereum_precompile_callback`, and no `error_refund` feature guard (unlike `ExitToNear`, which has a partial refund path for the wNEAR unwrap case).

Contrast this with `ExitToNear`, which conditionally attaches a callback when `transfer_near_args` or `refund` is non-default:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

`ExitToEthereum` has no equivalent conditional — it is unconditionally `PromiseArgs::Create`.

The same pattern applies to the ETH (base token) exit path in `ExitToNear` for the legacy `ft_transfer` case (no `error_refund` feature, no `transfer_near` args), where the promise is also `PromiseArgs::Create` with no callback. [3](#0-2) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

If the NEAR `withdraw` promise fails (e.g., the eth connector contract panics, runs out of gas, or is paused), the EVM-side burn is already finalized in the previous NEAR receipt. NEAR's asynchronous execution model means the EVM state change and the NEAR promise are in separate receipts; a failed promise does not roll back the prior receipt's EVM state. The user's ERC-20 tokens are burned with no corresponding withdrawal on the NEAR side, and no refund mechanism exists to restore them.

### Likelihood Explanation

**Low-Medium.** Under normal operation the eth connector's `withdraw` method succeeds. However, the risk is non-zero because:
- The eth connector can be independently paused (a separate contract with its own pausability).
- Gas exhaustion in the promise chain is possible under congestion.
- Any future bug in the eth connector's `withdraw` logic would silently destroy user funds with no recovery path.

The absence of a callback means there is zero tolerance for failure — any single failure permanently destroys the funds.

### Recommendation

Attach a callback to the `withdraw` promise in `ExitToEthereum::run`, analogous to the `exit_to_near_precompile_callback` pattern already used in `ExitToNear`. The callback should check `handler.promise_result(0)` and, on failure, re-mint the burned ERC-20 tokens (or ETH balance) to the original sender. The `error_refund` feature already implements this pattern for `ExitToNear`; the same mechanism should be extended to `ExitToEthereum`. [4](#0-3) 

### Proof of Concept

1. Alice holds 100 units of a bridged ERC-20 token on Aurora.
2. Alice calls `withdraw(amount=100, eth_recipient=0xAlice)` on the ERC-20 contract, which burns 100 tokens and calls the `ExitToEthereum` precompile.
3. The precompile emits a `PromiseArgs::Create` log targeting the eth connector's `withdraw` method. No callback is registered.
4. The EVM receipt completes: Alice's ERC-20 balance is now 0.
5. The NEAR runtime processes the scheduled promise. Suppose the eth connector is paused at this moment (or runs out of gas). The promise fails.
6. Because there is no callback, the engine never learns of the failure. No refund is issued.
7. Alice has lost 100 tokens permanently: burned on Aurora, never withdrawn on NEAR/Ethereum. [5](#0-4)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
