### Title
Division-Before-Multiplication Precision Loss in Native Resource Accounting — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` and `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

Two integer division-before-multiplication patterns in ZKsync OS's native resource accounting cause systematic precision loss. The first truncates `native_per_pubdata` before multiplying by pubdata bytes, undercharging users for pubdata in native resources. The second truncates `native_used / native_per_gas` before computing `delta_gas`, causing `gas_used` to be underestimated and users to receive a larger-than-correct gas refund.

---

### Finding Description

**Instance 1 — `native_per_pubdata` truncation**

In `validation_impl.rs` line 142:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This computes `floor(pubdata_price / native_price)` and stores it as `native_per_pubdata`. This truncated ratio is then used in two places:

- **Intrinsic pubdata overhead** (`gas_helpers.rs` line 352):
  ```rust
  let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
  ```
- **Post-execution pubdata charge** (`gas_helpers.rs` line 431):
  ```rust
  let native = current_pubdata_spent.checked_mul(native_per_pubdata)...;
  ```

The mathematically correct charge for `N` pubdata bytes is:
```
floor(N * pubdata_price / native_price)
```
But the code computes:
```
N * floor(pubdata_price / native_price)
```

The undercharge per byte is up to `(native_price - 1) / native_price` native units — nearly 1 native unit per pubdata byte. For a transaction publishing 100,000 bytes of pubdata with `native_price = 1000`, the undercharge can reach ~99,900 native units.

An extreme edge case: if `pubdata_price < native_price`, then `native_per_pubdata = 0`, making pubdata completely free in native resource terms. Any user can then publish pubdata up to the block's `pubdata_limit` without consuming any native budget for it.

**Instance 2 — `delta_gas` truncation**

In `refund_calculation.rs` line 72:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
```

The division `native_used / native_per_gas` truncates. The correct value is `native_used / native_per_gas` (exact rational). The truncation means `delta_gas` is underestimated by up to 1 gas unit, so `gas_used` is underestimated by up to 1 gas unit, and the user receives a slightly larger ETH refund than they should.

---

### Impact Explanation

**Instance 1**: Users are systematically undercharged native resources for pubdata. The undercharge accumulates with pubdata volume. In the degenerate case (`pubdata_price < native_price`), pubdata costs zero native resources, allowing a user to exhaust the block's pubdata limit without paying the expected native cost. This is a **resource accounting bug** — the protocol bears more proving/DA cost than the user paid for.

**Instance 2**: Every transaction where `native_used mod native_per_gas != 0` receives a 1-gas-unit excess refund. The financial impact per transaction is at most `gas_price` wei — negligible individually but systematic across all transactions.

---

### Likelihood Explanation

**Instance 1**: Triggered on every transaction where `pubdata_price mod native_price != 0`. Since `pubdata_price` and `native_price` are independent operator-set values, this condition holds for virtually all real-world block configurations. Likelihood is **high**.

**Instance 2**: Triggered on every transaction where `native_used mod native_per_gas != 0`. This is the common case. Likelihood is **high**.

---

### Recommendation

**Instance 1**: Avoid pre-dividing. Instead of storing `native_per_pubdata = floor(pubdata_price / native_price)` and later multiplying by pubdata bytes, carry `pubdata_price` and `native_price` separately and compute the native charge as:
```rust
let native = pubdata_bytes
    .checked_mul(pubdata_price_u64)
    .and_then(|v| v.checked_div(native_price_u64))
    ...;
```
This performs multiplication before division, preserving precision. Alternatively, use `div_ceil` for `native_per_pubdata` to overestimate (charging users slightly more) rather than underestimate.

**Instance 2**: Use `div_ceil` for the `native_used / native_per_gas` step, or compute `delta_gas` using the same multiply-then-divide pattern:
```rust
let implied_gas = native_used.div_ceil(native_per_gas);
let delta_gas = implied_gas as i64 - gas_used as i64;
```

### Proof of Concept

**Instance 1 — `native_per_pubdata = 0` edge case:**

Set block context:
- `pubdata_price = 999`
- `native_price = 1000`

Then `native_per_pubdata = floor(999/1000) = 0`.

Submit a transaction that writes 20 storage slots (generating ~640 bytes of pubdata). The native charge for pubdata is `640 * 0 = 0` native units. The user publishes 640 bytes of pubdata at zero native cost, while the correct charge should be `floor(640 * 999 / 1000) = 639` native units.

**Instance 1 — accumulation with large pubdata:**

Set block context:
- `pubdata_price = 1001`
- `native_price = 1000`
- `native_per_pubdata = floor(1001/1000) = 1`

A transaction publishing 100,000 bytes of pubdata is charged `100,000 * 1 = 100,000` native units. The correct charge is `floor(100,000 * 1001 / 1000) = 100,100` native units. The user underpays by 100 native units.

**Instance 2 — `delta_gas` underestimation:**

- `native_used = 999`, `native_per_gas = 1000`, `gas_used = 0`
- `delta_gas = (999 / 1000) - 0 = 0 - 0 = 0`
- Correct: `delta_gas = ceil(999/1000) - 0 = 1`
