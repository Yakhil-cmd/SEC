### Title
Unvalidated Pool Purge Bypass via Injected CatchUpPackageShare at Unfinalized Height — (`rs/consensus/src/consensus/purger.rs`)

---

### Summary

A single Byzantine P2P peer can permanently suppress unvalidated-pool purging on an honest replica by continuously injecting `CatchUpPackageShare` artifacts at heights that have no finalized block yet. The validator leaves such shares in the unvalidated pool indefinitely (returning `None`), and the purger's skip-purge guard fires on every round, preventing `PurgeUnvalidatedBelow` from ever being emitted. The unvalidated pool grows without bound until the replica runs out of memory.

---

### Finding Description

**Step 1 — The skip-purge guard in the Purger**

`purge_unvalidated_pool_by_expected_batch_height` skips purging when the unvalidated pool contains a `CatchUpPackageShare` whose height range satisfies two predicates simultaneously:

```
below_range_max(catch_up_height, range)  →  catch_up_height < range.max
above_range_min(expected_batch_height, range)  →  expected_batch_height > range.min
``` [1](#0-0) 

`height_range()` returns the overall `(min, max)` of all shares in the unvalidated pool. A single share at height H where `catch_up_height < H < expected_batch_height` satisfies both predicates (`range.min = range.max = H`), causing an immediate `return` with no purge action emitted and `prev_expected_batch_height` left unchanged. [2](#0-1) 

**Step 2 — `check_integrity()` for `CatchUpPackageShare` is trivially satisfiable**

The only integrity check for a share verifies that the embedded random beacon's hash field matches `crypto_hash(random_beacon)`: [3](#0-2) 

An attacker can craft a share with any random beacon value and compute the matching hash. No subnet membership, no threshold key, no valid block reference is required to pass this check.

**Step 3 — The validator leaves shares at unfinalized heights in the pool forever**

`validate_catch_up_package_shares` calls `validate_catch_up_share_content` before ever touching the signature. That function immediately returns `Err(ValidationFailed::FinalizedBlockNotFound(height))` when no finalized block exists at the share's height: [4](#0-3) 

The `ValidationFailed` arm of the match returns `None` — no `HandleInvalid`, no `RemoveFromUnvalidated`: [5](#0-4) 

The signature is **never verified** in this code path. The share remains in the unvalidated pool across all future rounds until a finalized block appears at that height — which never happens for a fabricated height.

**Step 4 — Continuous injection sustains the bypass**

As `catch_up_height` advances, the attacker injects a fresh share at the new `catch_up_height + 1`. Because `prev_expected_batch_height` is never updated when purging is skipped, the purger re-enters the guard on every `expected_batch_height` increment and is blocked again. The unvalidated pool accumulates shares from every round.

---

### Impact Explanation

The unvalidated pool is in-memory only (explicitly documented as volatile). Unbounded accumulation of injected shares causes heap exhaustion and eventual OOM-kill of the replica process. This takes a single honest replica offline without requiring any threshold of corrupted nodes. [6](#0-5) 

---

### Likelihood Explanation

The attacker needs only:
- P2P connectivity to the target replica (standard subnet peer relationship)
- The ability to craft a `CatchUpPackageShare` with a self-consistent random beacon hash (trivial arithmetic, no cryptographic secret)
- Knowledge of the current `catch_up_height` (publicly observable from the pool)

No threshold key, no subnet majority, no admin access is required. A single Byzantine peer below the fault threshold suffices.

---

### Recommendation

1. **Remove the share from the pool when content validation fails with `FinalizedBlockNotFound` after a timeout** — the current `unvalidated_for_too_long` path only logs; it should also emit `RemoveFromUnvalidated`.
2. **Verify the share's signature before consulting the unvalidated pool's height range in the purger**, or alternatively move the purge-skip check to only fire after the validator has had a chance to remove invalid artifacts.
3. **Bound the unvalidated pool size** per artifact type to cap the worst-case memory impact regardless of injection rate.

### Proof of Concept

```
Setup:
  catch_up_height = 10
  expected_batch_height = 12
  finalized_height = 11

Attacker action (each consensus round):
  Craft CatchUpPackageShare at height 11:
    - random_beacon = RandomBeacon::fake(height=11, ...)
    - content.random_beacon.hash = crypto_hash(&random_beacon)   // passes check_integrity()
    - signature = ThresholdSignatureShare::fake(attacker_node_id) // never checked
  Inject via P2P into target replica's unvalidated pool.

Purger round:
  unvalidated_catch_up_share_range = Some(HeightRange { min: 11, max: 11 })
  below_range_max(10, range) = 10 < 11 = true
  above_range_min(12, range) = 12 > 11 = true
  → return (no purge, prev_expected_batch_height unchanged)

Validator round:
  validate_catch_up_share_content(height=11):
    get_finalized_block(11) = None  // no finalized block at 11 yet
    → Err(ValidationFailed::FinalizedBlockNotFound(11))
  → None (share stays in pool)

Next round: expected_batch_height = 13, catch_up_height = 11
  Attacker injects share at height 12.
  Same bypass fires. Pool grows by one share per round.
```
