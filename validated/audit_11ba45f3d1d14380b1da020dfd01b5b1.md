### Title
`upgrade()` Bypasses `upgrade_delay_blocks` Timelock, Enabling Immediate Malicious Code Deployment - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine exposes two distinct upgrade paths. The `stage_upgrade()` + `deploy_upgrade()` path enforces the `upgrade_delay_blocks` timelock. However, the `upgrade()` function deploys new WASM contract code **immediately** with no delay check, completely nullifying the timelock protection that was explicitly designed to give users time to exit before a malicious upgrade takes effect.

---

### Finding Description

The `upgrade_delay_blocks` field in `EngineState` was introduced specifically to protect users:

> "How many blocks after staging upgrade can deploy it." [1](#0-0) 

The `stage_upgrade()` function correctly computes and stores the earliest allowed deployment block:

```rust
let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
``` [2](#0-1) 

And `deploy_upgrade()` enforces this delay:

```rust
if io.block_height() <= index {
    sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
}
``` [3](#0-2) 

However, the `upgrade()` function — a **separate, parallel upgrade path** — performs only two checks (`require_running` and `require_owner_only`) and then immediately issues `PromiseAction::DeployContract { code }` with **zero delay**:

```rust
pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;
    // ... no timelock check ...
    actions: vec![
        PromiseAction::DeployContract { code },
``` [4](#0-3) 

This is exposed as a public NEAR contract entrypoint: [5](#0-4) 

The `upgrade_delay_blocks` timelock is therefore entirely ineffective as a user protection mechanism — the owner can always bypass it by calling `upgrade()` directly instead of `stage_upgrade()` + `deploy_upgrade()`.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The Aurora Engine holds bridged ETH and ERC-20 token balances for all users of the Aurora EVM. A malicious or compromised owner can call `upgrade()` to atomically replace the entire WASM contract with malicious code in a single transaction, with no advance notice. The malicious code can implement arbitrary logic to drain all ETH and token balances held by the engine. The `upgrade_delay_blocks` parameter — the only on-chain mechanism intended to give users time to withdraw — is rendered meaningless.

---

### Likelihood Explanation

The `upgrade()` entrypoint is callable by the `owner_id` account. The owner is a privileged NEAR account. The bypass requires no special conditions beyond owner key access — no additional governance steps, no multi-sig threshold beyond what already controls the owner key. The existence of a parallel, timelock-free upgrade path means the timelock provides no real security guarantee.

---

### Recommendation

Remove the `upgrade()` function entirely, or enforce the same `upgrade_delay_blocks` check inside it. All code deployments to the Aurora Engine contract should be required to go through the `stage_upgrade()` + `deploy_upgrade()` path so the timelock is unconditionally enforced. Additionally, `set_upgrade_delay_blocks()` should itself be subject to a timelock or a minimum floor value to prevent the owner from atomically setting the delay to zero and then staging an upgrade in the same block. [6](#0-5) 

---

### Proof of Concept

1. Aurora Engine is deployed with `upgrade_delay_blocks = 10000` (signaling a ~14-hour user protection window).
2. Users deposit ETH and ERC-20 tokens into the engine, trusting the timelock.
3. Owner calls `upgrade()` directly with malicious WASM bytecode containing a `drain_all` function.
4. `upgrade()` checks only `require_running` and `require_owner_only` — both pass.
5. `PromiseAction::DeployContract { code: malicious_wasm }` executes in the same transaction, replacing the engine contract immediately.
6. Owner calls `drain_all` on the newly deployed malicious contract.
7. All user ETH and ERC-20 balances are stolen. The `upgrade_delay_blocks = 10000` setting provided zero protection. [4](#0-3)

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}
```
