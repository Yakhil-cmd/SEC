### Title
Integer Division Truncation in `native_per_pubdata` Calculation Silently Zeroes Pubdata Charging — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio used to charge native resources for pubdata is computed via floor (truncating) division. When `pubdata_price < native_price`, the result truncates to `0`, making all pubdata generation completely free for the transaction sender. This is the direct analog of the reported pattern: a round-down division produces `0`, and that `0` is silently propagated into a charging function that then charges nothing instead of reverting or enforcing a minimum.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (L2 transaction validation path):

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `0`. No error is returned — `0` is a valid `u64`.

This `0` value is then passed as `native_per_pubdata_byte` into `create_resources_for_tx`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0, no native deducted for intrinsic pubdata
```

And later into `get_resources_to_charge_for_pubdata`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, line 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
// → current_pubdata_spent * 0 = 0, no native charged for execution pubdata
```

`check_enough_resources_for_pubdata` then always returns `enough = true` because the required resources are `0`. The entire pubdata charging mechanism is bypassed.

The identical pattern also appears in the off-chain API helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

Note the asymmetry: `native_per_gas` uses `div_ceil` to guarantee a non-zero result when `gas_price > 0`, but `native_per_pubdata` uses `wrapping_div` (floor), which can silently produce `0`.

---

### Impact Explanation

When `0 < pubdata_price < native_price`, every transaction pays zero native resources for pubdata regardless of how many storage slots it writes. An attacker can:

1. Submit transactions that write to many storage slots (maximising pubdata output).
2. Pay only EVM gas for those writes — zero native resource is charged for the resulting pubdata.
3. Force the operator/sequencer to bear the full L1 publication cost of that pubdata.

This is a **resource accounting bug** with a **public funds-loss path**: the operator subsidises pubdata that should have been charged to the sender. At scale it enables a low-cost DoS against the sequencer's L1 publication budget.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is operator-set but is a realistic operating state. The default test configuration already uses `pubdata_price = 0, native_price = 10`. During periods of low L1 gas prices the operator may legitimately set a small `pubdata_price` that falls below `native_price`. Any unprivileged user transacting during such a window exploits the bug automatically — no special calldata or contract is required, only normal storage-writing transactions.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
      .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that any non-zero `pubdata_price` always produces at least `native_per_pubdata = 1`, preventing silent free-pubdata generation.

---

### Proof of Concept

**Setup**: operator sets `native_price = 10`, `pubdata_price = 9` (a realistic low-L1-gas scenario).

**Step 1** — `native_per_pubdata` truncates to 0:
```
native_per_pubdata = 9 / 10 = 0   (floor division)
```

**Step 2** — intrinsic pubdata overhead is zeroed:
```
intrinsic_pubdata_overhead = 0 * L2_TX_INTRINSIC_PUBDATA = 0
```
`native_limit` is not reduced; the transaction passes resource creation.

**Step 3** — attacker submits a transaction writing to 100 storage slots (generating ~3200 bytes of pubdata):
```
current_pubdata_spent = 3200
native_charged = 3200 * 0 = 0
```
`check_enough_resources_for_pubdata` returns `enough = true` with zero cost.

**Step 4** — the attacker pays only EVM gas (SSTORE costs) while the operator pays the full L1 cost to publish 3200 bytes of pubdata. Repeating this across many transactions drains the operator's L1 budget at negligible cost to the attacker. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
