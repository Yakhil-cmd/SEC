Audit Report

## Title
No Staleness Check on ICP/XDR Conversion Rate Used for Cycle Minting - (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles()` function in the Cycles Minting Canister reads the cached `icp_xdr_conversion_rate` and uses its `xdr_permyriad_per_icp` field directly to compute cycles without ever consulting the rate's `timestamp_seconds`. If the Exchange Rate Canister becomes temporarily unavailable, the CMC silently continues minting cycles at the last known — potentially arbitrarily stale — ICP/XDR price, allowing any unprivileged user to receive more cycles per ICP than the current market rate justifies.

## Finding Description
`tokens_to_cycles()` at lines 1900–1923 of `rs/nns/cmc/src/main.rs` extracts only `xdr_permyriad_per_icp` from the stored rate, discarding `timestamp_seconds` entirely:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);  // timestamp_seconds never read
```

This function is the sole conversion entry point for all three minting paths: `process_top_up` (L1991), `process_create_canister` (L1932), and `process_mint_cycles` (L1965).

The rate is refreshed by `update_exchange_rate()` in `rs/nns/cmc/src/exchange_rate_canister.rs`, called from the CMC heartbeat every `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS` (L16). When the XRC call fails, `UpdateExchangeRateError::FailedToRetrieveRate` is returned (L271–274), the next attempt is rescheduled one minute later (L153–160), and the old rate is silently retained in state — there is no clearing or invalidation of the stale value.

`do_set_icp_xdr_conversion_rate()` (L1009–1040) only enforces that a new rate has a strictly greater timestamp than the current one; it imposes no upper bound on how old the stored rate may be. `validate_exchange_rate()` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` (L110–129) checks only source counts (minimum 4 ICP, 4 CXDR sources), not rate age.

**Exploit flow:**
1. The XRC canister becomes temporarily unavailable (e.g., it is stopped via governance proposal, or is being upgraded independently of the CMC — both canisters are on the NNS subnet but can be upgraded separately).
2. The CMC heartbeat calls `update_exchange_rate()`, receives `FailedToRetrieveRate`, and retains the last cached rate. This repeats every minute with no staleness invalidation.
3. During the outage, the ICP market price drops materially (e.g., from 10 XDR/ICP to 5 XDR/ICP).
4. An unprivileged user sends ICP to the CMC's subaccount and calls `notify_top_up`. `tokens_to_cycles()` reads the stale `xdr_permyriad_per_icp` (reflecting 10 XDR/ICP) and mints cycles at the old rate — twice as many cycles as the current market rate justifies.
5. The `timestamp_seconds` field of the stored rate is never consulted. The user successfully receives cycles at the stale price.

## Impact Explanation
Cycles are minted at an economically incorrect rate, directly undermining the ICP/cycles peg that the CMC is designed to maintain. The `base_cycles_limit` (`DEFAULT_CYCLES_LIMIT = 150e15` cycles/hour, L83) partially bounds the damage per hour but does not eliminate it: at a 2× price discrepancy, an attacker extracts up to 75 ICP worth of excess cycles per hour, and the attack is repeatable across hours for as long as the outage persists. This constitutes illegal minting with concrete protocol economic harm, matching the **High** impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The XRC is a system canister on the NNS subnet and can be stopped or upgraded independently of the CMC via governance proposal — a routine operational event. The CMC retries every minute on failure but imposes no upper bound on how long the stale rate is used. The exploit requires only that (a) the XRC be unreachable for long enough for the ICP price to move materially, and (b) an unprivileged user with a valid ICP ledger block calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles`. No special privileges are required on the attacker's side. The condition is realistic during planned NNS canister upgrades.

## Recommendation
In `tokens_to_cycles()`, check the age of `icp_xdr_conversion_rate.timestamp_seconds` against the current canister time before using the rate. If the rate is older than a defined maximum staleness threshold (e.g., 30 minutes), return an error rather than minting at a potentially stale price:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref()
            .ok_or_else(|| NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "No conversion rate found in CMC".to_string(),
            })?;

        let now = now_seconds();
        let age_seconds = now.saturating_sub(rate.timestamp_seconds);
        if age_seconds > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: format!("ICP/XDR rate is stale ({age_seconds}s old)"),
            });
        }

        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

A reasonable `MAX_RATE_AGE_SECONDS` would be 30–60 minutes, giving the heartbeat retry mechanism sufficient time to recover from transient XRC failures while bounding the staleness window.

## Proof of Concept
A deterministic PocketIC or StateMachine integration test can reproduce this:

1. Initialize the NNS with the CMC and a mock XRC canister.
2. Set the CMC's stored `icp_xdr_conversion_rate` to a rate with `xdr_permyriad_per_icp = 100_000` (10 XDR/ICP) and `timestamp_seconds = T`.
3. Stop the mock XRC canister so all subsequent `get_icp_to_xdr_exchange_rate` calls fail.
4. Advance the state machine clock by 2 hours (well past `REFRESH_RATE_INTERVAL_SECONDS`).
5. Simulate a market price drop by noting the expected rate should now be 50_000 (5 XDR/ICP).
6. Call `notify_top_up` with 1 ICP from an unprivileged test principal.
7. Assert that the returned cycles equal `100_000 / 10_000 * cycles_per_xdr` (computed at the stale 10 XDR/ICP rate) rather than the correct 5 XDR/ICP rate — confirming cycles are minted at 2× the correct amount.
