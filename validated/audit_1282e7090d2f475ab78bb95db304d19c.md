Audit Report

## Title
`spawn_neuron` Unconditionally Copies Parent Hot Keys to Child Neuron Assigned to a Different Controller — (`rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with `new_controller` set to a principal different from the parent neuron's controller, the child neuron is constructed with the parent's full `hot_keys` list cloned onto it unconditionally. The new controller receives a neuron with pre-authorized hot key principals they did not choose, giving those principals the ability to vote, follow, and modify Neuron Fund membership on the new controller's neuron without their knowledge or consent.

## Finding Description
In `spawn_neuron` (`rs/nns/governance/src/governance.rs`), the `child_controller` is resolved from the optional `spawn.new_controller` field:

```rust
// L2651–2655
let child_controller = if let Some(child_controller) = &spawn.new_controller {
    *child_controller
} else {
    parent_neuron.controller()
};
```

The child neuron is then built with `child_controller` as its controller, but the parent's hot keys are cloned unconditionally:

```rust
// L2704–2718
let child_neuron = NeuronBuilder::new(child_nid, to_subaccount, child_controller, ...)
    .with_spawn_at_timestamp_seconds(...)
    .with_hot_keys(parent_neuron.hot_keys.clone())  // ← no guard on child_controller == parent_controller
    ...
    .build();
```

There is no conditional check to clear or filter `hot_keys` when `child_controller != parent_neuron.controller()`. The inherited hot keys remain valid because `is_authorized_to_vote` delegates to `is_hotkey_or_controller`:

```rust
// L243–255 (neuron/types.rs)
pub(crate) fn is_authorized_to_vote(&self, principal: &PrincipalId) -> bool {
    self.is_hotkey_or_controller(principal)
}
fn is_hotkey_or_controller(&self, principal: &PrincipalId) -> bool {
    self.is_controlled_by(principal) || self.hot_keys.contains(principal)
}
```

Hot keys are also authorized to join/leave the Neuron Fund via `is_authorized_to_configure_or_err` (`neuron/types.rs` L780–781). The `split_neuron` path (L2241–2257) always uses `*caller` as the child controller, so the same controller can clean up; in `spawn_neuron` the controller is a third party with no awareness of the inherited keys.

## Impact Explanation
This constitutes unauthorized access to a neuron's governance capabilities. A hot key holder (Bob) on the parent neuron retains the ability to: (1) cast `RegisterVote` on NNS proposals on behalf of Charlie's neuron; (2) set `Follow` relationships, causing Charlie's neuron to auto-vote in ways Charlie did not choose; (3) call `JoinCommunityFund`/`LeaveCommunityFund` on Charlie's neuron, affecting maturity exposure. This matches the allowed High impact: *"Unauthorized access to neurons, governance assets … where exploitation requires meaningful per-target work or other constraints."* Direct fund theft is not possible via hot keys (disburse/split/spawn require controller), but persistent unauthorized governance control over a third party's neuron is a concrete, non-hypothetical harm.

## Likelihood Explanation
Both preconditions are normal, documented operations: Alice adds Bob as a hot key (routine), and Alice spawns a neuron to Charlie via `new_controller` (the explicit purpose of that field, documented in the proto). No special privileges, no victim mistakes, and no social engineering are required beyond Bob having been a hot key at any point before the spawn. Charlie has no UI-level warning and no reason to inspect hot keys on a freshly received neuron. The attack is repeatable for any neuron spawned to a different controller.

## Recommendation
In `spawn_neuron`, only copy `hot_keys` when the child controller is the same as the parent controller. When `child_controller` differs, initialize the child with an empty hot key list:

```rust
let hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// then: .with_hot_keys(hot_keys)
```

This is a localized, one-line-equivalent change to the `NeuronBuilder` call at `rs/nns/governance/src/governance.rs` L2714.

## Proof of Concept
1. Alice controls neuron N with hot key Bob.
2. Alice calls `manage_neuron` → `Spawn { new_controller: Some(Charlie), nonce: Some(42), percentage_to_spawn: Some(100) }`.
3. `spawn_neuron` sets `child_controller = Charlie` (L2651–2655) and builds the child neuron with `.with_hot_keys(parent_neuron.hot_keys.clone())` (L2714), storing `controller = Charlie, hot_keys = [Bob]`.
4. Bob sends `manage_neuron` → `RegisterVote { proposal: P, vote: Yes }` authenticated as Bob, targeting the child neuron ID.
5. Governance calls `is_authorized_to_vote` → `is_hotkey_or_controller` → `hot_keys.contains(Bob)` → `true` → vote is cast on Charlie's neuron.

A deterministic integration test can be written using `PocketIC` or the existing NNS integration test harness (`rs/nns/integration_tests/`) by: creating a neuron with a hot key, spawning it to a different controller, then asserting that the original hot key principal can successfully call `RegisterVote` on the child neuron.
