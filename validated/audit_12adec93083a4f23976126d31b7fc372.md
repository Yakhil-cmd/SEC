### Title
ERC-20 Tokens Permanently Burned With No Refund Path When `ft_transfer` Fails to Unregistered NEAR Recipient — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` burns the caller's ERC-20 tokens before scheduling a NEAR-side `ft_transfer`. When the engine is compiled without the `error_refund` feature (the production build configuration), no callback is attached to the promise. If `ft_transfer` fails — e.g., because the recipient has not registered a storage deposit with the NEP-141 contract — the burned tokens are never re-minted and the user's funds are permanently lost.

---

### Finding Description

**Step 1 — Entrypoint: `EvmErc20.withdrawToNear`**

Any EVM user can call `withdrawToNear` with an arbitrary `recipient` byte string. The function immediately burns the caller's tokens, then encodes the precompile input as `\x01 | amount_b | recipient` and calls the `ExitToNear` precompile: [1](#0-0) 

There is no `refund_address` field in this encoding. This matches the non-`error_refund` wire format.

**Step 2 — Precompile: `refund` is hardcoded to `None` without `error_refund`**

Inside `ExitToNear::run`, the `callback_args` struct is built with `refund: None` when the feature is absent: [2](#0-1) 

For the legacy ERC-20 path (no wNEAR unwrap, no Omni message), `transfer_near_args` is also `None`: [3](#0-2) 

**Step 3 — No callback is scheduled**

Because both fields of `callback_args` are `None`, it equals `ExitToNearPrecompileCallbackArgs::default()`, and the promise is created as a bare `PromiseArgs::Create` — no callback to `exit_to_near_precompile_callback` is attached: [4](#0-3) 

**Step 4 — `ft_transfer` fails silently**

The NEAR-side `ft_transfer` to an unregistered account fails with a storage deposit error. Since there is no callback, `exit_to_near_precompile_callback` is never invoked, and the `refund_on_error` path (which would re-mint the ERC-20 tokens) is never reached: [5](#0-4) 

**Step 5 — Production build omits `error_refund`**

The production build target is `CARGO_FEATURES_BUILD = "contract"`, which does not include `error_refund`: [6](#0-5) 

`error_refund` is a non-default, opt-in feature: [7](#0-6) 

---

### Impact Explanation

The user's ERC-20 tokens are burned on the EVM side. The corresponding NEP-141 tokens remain locked in Aurora's account on NEAR (the `ft_transfer` failed, so Aurora's NEP-141 balance is unchanged), but there is no user-accessible path to recover them. The ERC-20 tokens are gone with no re-mint. This constitutes at minimum **High — Temporary freezing of funds** (NEP-141 tokens stuck in Aurora's account), and in practice is a permanent loss of the user's ERC-20 position.

---

### Likelihood Explanation

The trigger condition — calling `withdrawToNear` with a NEAR account ID that is syntactically valid but not registered with the NEP-141 contract — is trivially reachable by any EVM user. No special privilege is required. The NEP-141 storage registration requirement is a standard NEAR FT mechanic that many accounts will not have satisfied. The existing test suite explicitly documents this loss as the expected behavior without `error_refund`: [8](#0-7) 

---

### Recommendation

Enable the `error_refund` feature in the production build by adding it to `CARGO_FEATURES_BUILD` in `Makefile.toml`. This causes the precompile to embed a `refund_address` in the callback args and attach a `exit_to_near_precompile_callback` promise that re-mints the ERC-20 tokens if `ft_transfer` fails.

Alternatively, restructure `EvmErc20.withdrawToNear` to perform the burn only inside the callback (after confirming NEAR-side success), though this requires a more significant architectural change.

---

### Proof of Concept

1. Deploy a NEP-141 token; bridge it to an ERC-20 on Aurora (standard flow).
2. As an EVM user holding ERC-20 tokens, call `withdrawToNear(recipient, amount)` where `recipient` is a syntactically valid NEAR account ID that has **not** registered a storage deposit with the NEP-141 contract.
3. Observe: `_burn` executes, ERC-20 balance drops to zero.
4. The `ExitToNear` precompile schedules a bare `ft_transfer` promise with no callback (confirmed by `callback_args == default()`).
5. NEAR-side `ft_transfer` fails with storage deposit error.
6. No `exit_to_near_precompile_callback` is invoked; no re-mint occurs.
7. Assert: ERC-20 balance remains zero; NEP-141 balance of the recipient remains zero; Aurora's NEP-141 balance is unchanged (tokens locked).

This is directly confirmed by the existing test `test_exit_to_near_refund` which asserts `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` (i.e., no refund) when compiled without `error_refund`. [9](#0-8)
