### Title
Unhandled `panic!` on Invalid EIP-6110 Deposit Event Data Causes Node Abort - (`File: basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

In `eip6110_events_parser`, when a deposit event emitted from `DEPOSIT_CONTRACT_ADDRESS` fails structural validation, the code calls `panic!("invalid deposit event structure")` instead of returning a typed error. Because the deposit contract address is not guarded by a pre-deployment check before event processing, an unprivileged attacker who deploys a malicious contract at that address can emit a structurally invalid deposit event and trigger an unconditional abort of the node process.

---

### Finding Description

`eip6110_events_parser` iterates over all block events, filters those whose `address` equals `DEPOSIT_CONTRACT_ADDRESS` and whose first topic equals `DEPOSIT_EVENT_SIGNATURE_HASH`, and then calls `validate_and_write_event_data`. If that function returns `Err(())` — which it does for any event whose data length is not exactly 576 bytes, or whose SSZ offset words fail the `validate_u16_at_most` checks — the code panics unconditionally:

```rust
let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
    panic!("invalid deposit event structure");
};
``` [1](#0-0) 

`validate_and_write_event_data` returns `Err(())` on the very first check if `data.len() != 576`:

```rust
if data.len() != 576 {
    return Err(());
}
``` [2](#0-1) 

This `panic!` is reached in **both** execution paths:

- **Proving path** (`post_tx_op_proving.rs`): `eip6110_events_parser` is called and its `Result` is `.expect()`'d, so the inner panic propagates directly. [3](#0-2) 

- **Sequencing/forward path** (`post_tx_op_sequencing.rs`): the return value is discarded with `let _`, but the `panic!` inside the function still unwinds the stack and aborts the process. [4](#0-3) 

The project's own panic policy explicitly forbids this pattern:

> "Do not introduce `todo!`, `unimplemented!`, or `unreachable!` on paths reachable from external input."
> "Panic paths reachable from untrusted input. ZKsync OS should not panic in production." [5](#0-4) 

The `panic_handler` in the RISC-V binary calls `rust_abort()`, making the abort unrecoverable in the proving environment: [6](#0-5) 

---

### Impact Explanation

An attacker who successfully triggers the panic causes an unconditional process abort in both the forward (sequencing) and proving execution paths. In the proving path this halts proof generation for the block; in the sequencing path it halts the sequencer. Either outcome constitutes a liveness failure: the chain stops producing or proving blocks until the node is restarted and the offending transaction/block is handled differently. Because the panic fires during post-block processing (after all transactions have been executed), the entire block's work is discarded.

---

### Likelihood Explanation

The `DEPOSIT_CONTRACT_ADDRESS` (`0x00000000219ab540356cbb839cbe05303d7705fa`) is referenced only as a constant filter address; no code in the repository pre-deploys a contract there or enforces that the address is occupied before event processing begins. If the address is empty at chain genesis (or in any test/staging environment), any unprivileged sender can:

1. Deploy a contract at `DEPOSIT_CONTRACT_ADDRESS` via a standard `CREATE` transaction (the address is not protected by any system-level guard visible in the codebase).
2. Call that contract to emit a `LOG1` with `topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH` and `data` of any length other than 576 bytes (or with invalid SSZ offset words).
3. Submit the transaction; when the block is sealed, `eip6110_events_parser` processes the event and panics.

Even in environments where the real deposit contract is pre-deployed, any future upgrade or misconfiguration that causes the contract to emit a non-conforming event would trigger the same abort.

---

### Recommendation

Replace the `panic!` with a graceful error return that propagates up to the block-processing boundary, where it can be logged and the block rejected without aborting the process:

```rust
// In eip6110_events_parser — change return type to Result<bool, SystemError>
validate_and_write_event_data(event, requests_hasher, &mut logger)
    .map_err(|_| SystemError::LeafDefect(internal_error!("invalid deposit event structure")))?;
```

Additionally, add a pre-check that `DEPOSIT_CONTRACT_ADDRESS` is occupied (nonce > 0 or bytecode present) before processing its events, consistent with the guard already used in `eip7002_system_part`:

```rust
// Mirror the guard from eip_7002_withdrawal_contract/mod.rs lines 61-76
let props = system.io.read_account_properties(..., &DEPOSIT_CONTRACT_ADDRESS, ...)?;
if !is_contract(props) {
    return Err(SystemError::LeafDefect(internal_error!(
        "EIP-6110 deposit contract is not deployed"
    )));
}
``` [7](#0-6)

### Proof of Concept

1. Start ZKsync OS in Ethereum-compatible mode with `DEPOSIT_CONTRACT_ADDRESS` unoccupied.
2. Submit a transaction that deploys the following minimal contract at `0x00000000219ab540356cbb839cbe05303d7705fa`:
   ```solidity
   // Emits a deposit-signature event with 32 bytes of data (not 576)
   contract MaliciousDeposit {
       bytes32 constant SIG =
           0x649bbc62d0e31342afea4e5cd82d4049e7e1ee912fc0889aa790803be39038c5;
       function trigger() external {
           assembly {
               mstore(0, 0xdeadbeef)
               log1(0, 32, SIG)   // data length = 32, not 576
           }
       }
   }
   ```
3. Submit a second transaction calling `trigger()`.
4. Observe that when the block is sealed, `eip6110_events_parser` reaches line 46, `validate_and_write_event_data` returns `Err(())` (because `data.len() == 32 != 576`), and `panic!("invalid deposit event structure")` aborts the node. [8](#0-7)
