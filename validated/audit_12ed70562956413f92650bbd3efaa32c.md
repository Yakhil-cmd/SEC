### Title
Unchecked Assembly `call` Return Value in `withdrawToNear`/`withdrawToEthereum` Silently Burns Tokens on Precompile Failure — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` implement token withdrawal via undocumented assembly `call` blocks that invoke hardcoded Aurora precompile addresses. In both `withdrawToNear` and `withdrawToEthereum`, the user's ERC-20 tokens are burned **before** the assembly `call` is made, and the return value of that `call` is captured in a local variable `res` that is **never checked**. If the precompile call fails, execution continues silently, the tokens are permanently destroyed, and no withdrawal is initiated.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` (lines 53–63) and `withdrawToEthereum` (lines 65–76) follow this pattern:

1. `_burn(_msgSender(), amount)` — tokens are irreversibly destroyed.
2. Input bytes are constructed.
3. An undocumented assembly block issues a low-level `call` to a hardcoded precompile address.
4. The return value `res` is assigned but never inspected; no `require` or `if` branch acts on it.

```solidity
// withdrawToNear — EvmErc20.sol lines 60-62
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
}
``` [1](#0-0) 

```solidity
// withdrawToEthereum — EvmErc20.sol lines 73-75
assembly {
    let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
}
``` [2](#0-1) 

The same pattern is replicated verbatim in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

Neither assembly block carries any comment explaining the precompile address, the expected input encoding, or the required success condition. The use of inline assembly bypasses Solidity's automatic revert-on-failure behavior that a high-level `call` would provide.

The burn is unconditional and precedes the assembly call: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

If the low-level `call` to the precompile returns `0` (failure), the EVM does not revert the transaction. The tokens have already been burned from the user's balance and from total supply. There is no mechanism to recover them. The user permanently loses their bridged assets with no corresponding credit on NEAR or Ethereum.

---

### Likelihood Explanation

Any unprivileged EVM user holding a balance of the affected ERC-20 token can trigger this path by calling `withdrawToNear` or `withdrawToEthereum`. The call fails silently if:

- The precompile address is not active in the current Aurora deployment context.
- The encoded input does not match the precompile's expected format (e.g., a malformed `recipient` argument supplied by the caller).
- The precompile itself reverts internally.

Because the functions are `external` with no access control, the attack surface is every token holder. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **Check the return value** of the assembly `call` and revert on failure:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Document every line** of each assembly block: the precompile address, the input encoding, and the expected success condition.
3. **Reorder operations** (checks-effects-interactions): perform the precompile call first and only burn tokens after confirming success, or use a try/catch pattern.

---

### Proof of Concept

1. User holds 100 tokens of an `EvmErc20`-based mirror token on Aurora.
2. User calls `withdrawToNear(recipient_bytes, 100e18)`.
3. `_burn(msg.sender, 100e18)` executes — balance drops to 0, supply decreases.
4. The assembly `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` returns `0` (precompile unavailable or input rejected).
5. `res` is never checked; the function returns normally.
6. User's 100 tokens are permanently destroyed. No NEAR-side credit is issued. Funds are unrecoverable.
