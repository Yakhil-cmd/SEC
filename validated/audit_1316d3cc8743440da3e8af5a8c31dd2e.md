### Title
Permanent Fund Freeze via `ExitToNear` Precompile When `error_refund` Feature Is Disabled — (File: `engine-precompiles/src/native.rs`)

### Summary
When the `error_refund` compile-time feature is not enabled (it is absent from the default feature set), the `ExitToNear` precompile burns a user's ERC-20 tokens in the EVM and issues a `ft_transfer` promise to the NEP-141 contract with no callback. If that `ft_transfer` fails — for example because the recipient NEAR account is not registered in the NEP-141 contract — the burned ERC-20 tokens are never re-minted and the corresponding NEP-141 tokens remain permanently stranded in Aurora's account. The result is an irreversible, user-triggered permanent fund freeze.

### Finding Description
The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles ERC-20 withdrawals in two steps:

**Step 1 — Burn.** The ERC-20 contract calls the precompile, which records the burn inside the EVM state. At this point the user's ERC-20 balance is already zero.

**Step 2 — Transfer.** The precompile constructs a `ft_transfer` (or `ft_transfer_call`) promise targeting the NEP-141 contract and, conditionally, a callback promise (`exit_to_near_precompile_callback`) that can re-mint tokens on failure.

The callback is only attached when `callback_args != ExitToNearPrecompileCallbackArgs::default()`:

```rust
// engine-precompiles/src/native.rs  lines 449-483
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
// ...
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [1](#0-0) 

For a regular ERC-20 exit (flag `0x1`, no wNEAR unwrap), `transfer_near` is also `None`, so `callback_args` equals the default value and **no callback is ever scheduled**. If the downstream `ft_transfer` promise fails, the engine receives no notification and performs no re-mint.

The `error_refund` feature is explicitly absent from the default feature set of both the engine and the precompiles crate:

### Impact Explanation
When `error_refund` is not compiled in:

- The ERC-20 tokens are burned from the user's EVM balance (irreversible within the EVM).
- The NEP-141 tokens remain in Aurora's contract account with no corresponding ERC-20 claim, making them permanently inaccessible to the user.
- This constitutes a **permanent freezing of funds** matching the Critical impact tier.

### Likelihood Explanation
The trigger condition — specifying a NEAR recipient account that has not performed a `storage_deposit` on the NEP-141 contract — is easy to reach accidentally (e.g., a user who has never interacted with the NEP-141 contract directly, or a typo in the account ID). Any EVM token holder can reach this path by calling the ERC-20 contract's burn/withdraw function, which internally calls the `ExitToNear` precompile at address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`. No special privileges are required. [7](#0-6) 

### Recommendation
Enable the `error_refund` feature in the production build, or unconditionally attach a failure-callback to every `ExitToNear` promise that re-mints the burned tokens to the original sender when the downstream `ft_transfer` fails. The `refund_on_error` function in `engine/src/engine.rs` already implements the correct re-mint logic and should be invoked regardless of the compile-time feature flag. [8](#0-7) 

### Proof of Concept
1. Alice holds 100 units of a bridged ERC-20 token on Aurora (backed by 100 NEP-141 tokens held by the Aurora contract).
2. Alice calls the ERC-20 contract's `withdrawToNear` function specifying `"alice-new.near"` as the recipient — an account that has never called `storage_deposit` on the NEP-141 contract.
3. The ERC-20 contract calls the `ExitToNear` precompile (flag `0x1`). The precompile burns Alice's 100 ERC-20 tokens and schedules a bare `ft_transfer` promise (no callback, because `error_refund` is off and `transfer_near` is `None`).
4. The NEAR runtime executes `ft_transfer("alice-new.near", 100)` on the NEP-141 contract. The call panics with "account not registered."
5. Because no callback was attached, Aurora receives no failure notification. Alice's ERC-20 balance remains zero; the 100 NEP-141 tokens remain in Aurora's account with no owner. The funds are permanently frozen.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
