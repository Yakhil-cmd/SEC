Audit Report

## Title
Unbounded Ballot Iteration in SNS Governance `distribute_rewards` Exhausts Instruction Limit, Permanently Halting Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary

The SNS governance `distribute_rewards` function performs two unbounded synchronous loops — one over all ballots of all `ReadyToSettle` proposals and one over all voting neurons — with no instruction-limit guard. At ceiling parameters (`MAX_NUMBER_OF_NEURONS_CEILING = 200,000` × `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700`), the ballot aggregation loop alone can reach 140 million iterations, far exceeding the IC's 5-billion-instruction per-message limit. When the timer callback traps, state is rolled back and the condition persists, permanently halting reward distribution for the SNS. The NNS governance already received an explicit fix for this exact class of bug; the SNS governance has not.

## Finding Description

`distribute_rewards` in `rs/sns/governance/src/governance.rs` is called synchronously from `run_periodic_tasks` with no instruction-limit check anywhere in the function body.

**Loop 1 — ballot aggregation (lines 5894–5930):** [1](#0-0) 

`considered_proposals` is populated from `ready_to_be_settled_proposal_ids()` and can hold up to `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700` entries. [2](#0-1) 

Each proposal's ballot map can hold up to `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` entries. [3](#0-2) 

Worst-case: 700 × 200,000 = 140 million iterations, each performing `NeuronId::from_str` (string parsing), a `HashMap::entry` operation, and `Decimal` arithmetic — conservatively thousands of instructions each, yielding hundreds of billions of instructions total, far above the 5B limit.

**Loop 2 — maturity distribution (lines 5954–5997):** [4](#0-3) 

Up to 200,000 neurons, each requiring a mutable neuron lookup and `Decimal` arithmetic, with no instruction-limit break.

**Additional amplifier — `new_rounds_count` loop (lines 5861–5872):** [5](#0-4) 

When `latest_reward_event().end_timestamp_seconds` is `None`, `unwrap_or_default()` returns `0`, so `new_rounds_count = now / round_duration_seconds`. With the minimum `round_duration_seconds = 86400` and a Unix timestamp of ~1.7×10⁹, this yields ~19,675 iterations on first invocation.

**Invocation path:** [6](#0-5) 

`distribute_rewards` is called synchronously from `run_periodic_tasks`, which is scheduled as a repeating timer: [7](#0-6) 

**No instruction-limit guard exists in SNS governance.** A grep for `is_message_over_threshold`, `DISTRIBUTION_MESSAGE_LIMIT`, or any instruction counter in `rs/sns/governance/**` returns zero matches.

**The NNS governance already fixed this exact class of bug** using `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` to break the distribution loop across multiple timer invocations: [8](#0-7) 

The SNS governance has received no equivalent fix.

## Impact Explanation

When the instruction limit is exceeded inside a timer callback on the IC, the execution traps and all state mutations are rolled back. Because the condition that caused the overflow (large neuron count + large ballot count) is persistent state, every subsequent timer invocation reproduces the same trap. The SNS governance canister enters a permanent liveness failure: voting rewards are never distributed, neuron maturity never increases, and `latest_reward_event` is never updated. This freezes the economic incentive layer of the SNS, degrading governance participation and token holder trust. This matches the allowed impact: **High — Significant SNS security impact with concrete user or protocol harm / Application-level DoS with concrete governance harm.**

## Likelihood Explanation

Any SNS that configures `max_number_of_neurons` close to `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` and `max_number_of_proposals_with_ballots` close to `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700` is at risk. These are governance-controlled parameters that an SNS community may raise to support growth. An unprivileged participant who creates neurons and votes on proposals — normal, incentivized behavior — organically drives the system toward the failure threshold. No special privilege or coordination is required beyond normal participation at scale. The failure is self-sustaining once triggered.

## Recommendation

Apply the same pattern already used in NNS governance:

1. Move the per-neuron maturity increment out of `distribute_rewards` into a separate batched timer task analogous to `RewardsDistributionStateMachine` in `rs/nns/governance/src/reward/distribution.rs`.
2. Store the pending per-neuron reward map in stable memory after `distribute_rewards` computes it.
3. Process the map in chunks across multiple timer invocations, checking `is_message_over_threshold` after each neuron update and breaking when the limit is approached.
4. Cap `new_rounds_count` to a safe maximum per invocation, or initialize `reward_start_timestamp_seconds` from `genesis_timestamp_seconds` rather than defaulting to Unix epoch 0.

## Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_number_of_proposals_with_ballots = 700`.
2. Create 200,000 neurons (achievable by any token holder staking the minimum stake).
3. Submit 700 proposals and have all neurons vote on each (via following, this requires only the root neuron to vote explicitly).
4. Wait for `run_periodic_tasks` to fire (every `RUN_PERIODIC_TASKS_INTERVAL`).
5. `should_distribute_rewards` returns `true`; `distribute_rewards` is called.
6. The ballot aggregation loop at line 5894 iterates 700 × 200,000 = 140 million times, each calling `NeuronId::from_str` and a `HashMap::entry` operation.
7. The IC instruction counter exceeds 5 billion; the timer callback traps; state is rolled back.
8. `latest_reward_event` is never updated; `should_distribute_rewards` returns `true` again on the next tick.
9. The SNS governance canister is permanently unable to distribute voting rewards.

A deterministic integration test using PocketIC can reproduce this by instantiating an SNS canister with ceiling parameters, populating neurons and proposals via direct state injection, and asserting that `latest_reward_event` is never updated after repeated `run_periodic_tasks` invocations.
