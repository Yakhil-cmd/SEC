Audit Report

## Title
Attacker-Controlled `unsigned_transaction` Triggers Unconditional `panic!` in `handle_change_auto_stake_maturity`, Causing Rosetta Node Worker Crash — (`rs/rosetta-api/icp/src/ledger_client/handle_change_auto_stake_maturity.rs`)

## Summary

`handle_change_auto_stake_maturity` contains an unconditional `panic!` wildcard arm that fires on any `ManageNeuronResponse::Command` variant other than `Configure` or `Error`. Because `UnsignedTransaction` is fully attacker-controlled and the `RequestType` tag is never validated for consistency with the Candid-encoded argument, an attacker who owns any ICP neuron can craft a transaction that causes the governance canister to reply with `Command::Disburse(...)`, which is dispatched to `handle_change_auto_stake_maturity`, triggering the panic. The panic crashes the actix-web worker thread handling the request, terminating all in-flight requests on that thread; the attack is repeatable and can exhaust all worker threads, rendering the Rosetta node unavailable.

## Finding Description

**The panic (confirmed):**

`handle_change_auto_stake_maturity` matches only `Configure` and `Error`, panicking on all other variants:

```rust
_ => panic!(
    "Unexpected change auto staking maturity result: {:?}",
    response.command
),
``` [1](#0-0) 

**No panic catching exists anywhere in the ICP Rosetta API codebase** — a search for `catch_unwind`, `panic::set_hook`, and `std::panic` returns zero results in `rs/rosetta-api/icp/**/*.rs`. The main entry point uses `#[actix_web::main]` with no unwind guard around the request dispatch path. [2](#0-1) 

**Attacker-controlled input with no cross-validation (confirmed):**

`UnsignedTransaction` is deserialized directly from the caller-supplied CBOR+hex string. The `RequestType` tag and the raw Candid `arg` bytes inside `HttpCanisterUpdate` are both part of this blob and are never cross-validated against each other: [3](#0-2) [4](#0-3) 

**Dispatch solely on the attacker-supplied `RequestType` tag (confirmed):**

After the governance canister replies, `handle_reply` dispatches on the `RequestType` from the (attacker-supplied) `unsigned_transaction`, not on the actual Candid argument that was sent: [5](#0-4) 

The reply bytes are passed directly to `handle_change_auto_stake_maturity` regardless of what the governance canister actually returned: [6](#0-5) 

**Exploit chain:**

1. Attacker crafts `UnsignedTransaction` with `RequestType::ChangeAutoStakeMaturity` but with a Candid-encoded `Disburse` command as the `arg`, signed with their own key (they control the neuron).
2. `/construction/combine` accepts it — signature verification only checks that the signature covers `update.id()`, which the attacker can compute and sign correctly.
3. `/construction/submit` forwards the update to the governance canister.
4. Governance canister executes `Disburse` and returns `Command::Disburse(DisburseResponse { transfer_block_height })`.
5. `handle_reply` dispatches to `handle_change_auto_stake_maturity(bytes)`.
6. The `_` arm fires → `panic!` → worker thread terminates.

The same `panic!` wildcard pattern exists in `handle_disburse` and other sibling handlers, broadening the attack surface: [7](#0-6) 

## Impact Explanation

The panic terminates the actix-web worker thread processing the request. All concurrent requests on that thread are lost. Because the attack is repeatable at negligible cost (one dissolved neuron, public HTTP API), an attacker can continuously trigger panics across all worker threads, making the Rosetta node unable to process `/construction/submit` requests. Any exchange or custodian relying on the ICP Rosetta node for transaction submission loses that capability until the process is manually restarted or the worker pool is exhausted. This matches: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user and protocol harm to a Rosetta financial integration component.**

## Likelihood Explanation

Preconditions are minimal: owning any ICP neuron (small ICP stake, publicly available), moderate Candid/CBOR encoding skill, and access to the public Rosetta HTTP API. No admin keys, governance majority, or network-level access is required. The attack is repeatable indefinitely at low cost.

## Recommendation

Replace the `panic!` wildcard arm in `handle_change_auto_stake_maturity` with a graceful error return:

```rust
_ => Err(format!(
    "Unexpected change auto staking maturity result: {:?}",
    response.command
)),
```

Apply the identical fix to all sibling handlers containing the same pattern (`handle_disburse`, `handle_spawn`, etc.).

Additionally, add validation in `construction_combine` or `construction_submit` to verify that the Candid-encoded `arg` in `HttpCanisterUpdate` is consistent with the declared `RequestType`, rejecting mismatched payloads before they reach the governance canister.

## Proof of Concept

```
1. Attacker owns a dissolved NNS neuron (neuron_id = X, controller = attacker_principal).

2. Craft malicious UnsignedTransaction (CBOR+hex):
   updates = [(RequestType::ChangeAutoStakeMaturity { neuron_index: 0 },
               HttpCanisterUpdate {
                 canister_id: GOVERNANCE_CANISTER_ID,
                 method_name: "manage_neuron",
                 arg: candid::encode_one(ManageNeuron {
                   id: Some(NeuronId { id: X }),
                   command: Some(Command::Disburse(Disburse {
                     amount: None,
                     to_account: Some(attacker_account),
                   })),
                   neuron_id_or_subaccount: None,
                 }).unwrap(),
                 sender: attacker_principal_bytes,
                 ingress_expiry: <valid>,
                 nonce: None,
               })]
   ingress_expiries = [<valid>]

3. POST /construction/combine
   { unsigned_transaction: <malicious_cbor_hex>,
     signatures: [<valid Ed25519 sig over update.id()>] }

4. POST /construction/submit with the returned signed_transaction.

5. Governance canister executes Disburse, returns
   ManageNeuronResponse { command: Some(Command::Disburse(...)) }.

6. handle_reply dispatches to handle_change_auto_stake_maturity(bytes).

7. `_` arm fires → panic! → worker thread terminates.

8. Repeat to exhaust all worker threads.
```

A deterministic integration test using PocketIC can reproduce this by wiring a mock governance canister that returns `Command::Disburse` and asserting that the Rosetta node returns a 500 error (rather than panicking) after the fix is applied.
