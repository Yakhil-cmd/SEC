### Title
Scheduled XCC Promises Bypass Engine Pause State Check — (`etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router implements a two-phase promise execution pattern: `schedule` (called by the Aurora Engine parent, with full precondition checks) and `execute_scheduled` (callable by **anyone**, with **no** precondition checks). If the Aurora Engine is paused after a promise is scheduled, any external caller can still invoke `execute_scheduled` to execute the stored NEAR cross-contract call, bypassing the engine's pause mechanism entirely.

---

### Finding Description

The XCC router contract stores promises submitted by the Aurora Engine parent in `scheduled_promises` and allows deferred execution via `execute_scheduled`.

**Phase 1 — `schedule`**: Requires `assert_preconditions()`, which enforces that the caller is the parent (Aurora Engine) and no prior promises have failed. [1](#0-0) 

**Phase 2 — `execute_scheduled`**: Explicitly open to any caller, with zero state validation: [2](#0-1) 

The developer comment reads: *"It is intentional that this function can be called by anyone (not just the parent). There is no security risk to allowing this function to be open because it can only act on promises that were created via `schedule`."*

This reasoning is incomplete. It ignores the case where the Aurora Engine is paused **after** a promise is scheduled. The engine's `pause_contract` sets `state.is_paused = true`: [3](#0-2) 

All engine entrypoints that mutate state call `require_running(&state)` at the start of execution: [4](#0-3) 

However, `execute_scheduled` lives in the **XCC router** — a separate NEAR contract — and makes no call back to the engine to verify its pause state before dispatching the stored NEAR cross-contract call: [2](#0-1) 

The `assert_preconditions` helper that *would* block execution is only called in `execute` and `schedule`, not in `execute_scheduled`: [5](#0-4) 

---

### Impact Explanation

**High — Temporary freezing of funds bypass.**

The Aurora Engine's `pause_contract` is an emergency mechanism intended to halt all state-changing operations, including XCC-driven token movements (wNEAR withdrawals, NEP-141 transfers, arbitrary NEAR contract calls). Because `execute_scheduled` is callable by anyone and performs no pause check, any NEAR cross-contract call that was scheduled before the pause can still be dispatched after the pause. In a scenario where the pause was triggered precisely because a bug allowed unauthorized or malicious XCC scheduling, an attacker can execute those stored promises post-pause, moving funds out of the system in direct contradiction of the operator's intent.

---

### Likelihood Explanation

**Medium.** The engine pause is an emergency-only action. For this to be exploited, the engine must be paused while scheduled promises remain unexecuted. Given that XCC scheduling and execution are asynchronous by design (the `schedule`/`execute_scheduled` split exists specifically to allow deferred execution), there is a realistic window between scheduling and execution during which a pause could be issued. Any user or bot monitoring the XCC router's state can call `execute_scheduled` at any time.

---

### Recommendation

1. Add a pause flag to the XCC router itself, settable only by the parent (Aurora Engine), and check it in `execute_scheduled`.
2. Alternatively, require `execute_scheduled` to be called only by the parent, removing the open-caller design for this sensitive path.
3. At minimum, document that `pause_contract` on the Aurora Engine does **not** halt already-scheduled XCC promises, so operators know to manually cancel them (e.g., by having the engine call a `cancel_scheduled` method) during an incident.

---

### Proof of Concept

1. User submits an EVM transaction that triggers the XCC precompile (e.g., to transfer wNEAR to an external NEAR account). The Aurora Engine calls `schedule` on the XCC router after passing whitelist and pause checks. The promise is stored at `nonce = N`.
2. An operator discovers a critical vulnerability and calls `pause_contract` on the Aurora Engine. All engine entrypoints now reject with `ERR_PAUSED`.
3. An attacker (or the original user) calls `execute_scheduled(N)` **directly on the XCC router contract** — bypassing the Aurora Engine entirely.
4. `execute_scheduled` removes the promise from storage and dispatches the NEAR cross-contract call (e.g., a wNEAR `ft_transfer` to an attacker-controlled account).
5. Funds move despite the engine being paused, defeating the operator's emergency response.

The root cause mirrors the original report exactly: state conditions (pause, whitelist) are checked at scheduling time (Phase 1) but not rechecked at execution time (Phase 2), and Phase 2 is reachable by an unprivileged external caller. [6](#0-5)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
