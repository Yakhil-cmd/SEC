Audit Report

## Title
Unbounded Synchronous Iteration in SNS Governance `distribute_rewards` Can Permanently Break Voting Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards` function performs three sequential unbounded iterations over all considered proposals, all ballots per proposal, and all voting neurons in a single synchronous message execution with no instruction-limit guard. If the total work exceeds the IC per-message instruction limit (~40 billion instructions), the periodic timer callback traps, and because no partial state is checkpointed, every subsequent timer firing also traps, permanently halting SNS voting reward distribution.

## Finding Description

`distribute_rewards` at line 5763 of `rs/sns/governance/src/governance.rs` executes three unbounded loops atomically:

**Phase 1** (lines 5894–5931): iterates over every `considered_proposals` entry and for each proposal over every `ballot` (one per eligible neuron at proposal creation time), building `neuron_id_to_reward_shares`. [1](#0-0) 

**Phase 2** (lines 5954–5997): iterates over every entry in `neuron_id_to_reward_shares`, calling `get_neuron_result_mut` for each neuron and updating maturity. [2](#0-1) 

**Phase 3** (lines 6013–6081): iterates over `considered_proposals` again, calling `process_proposal` for each and clearing ballots. [3](#0-2) 

A grep search for `is_message_over_threshold`, `is_over_instructions_limit`, `instruction_limit`, and `instructions_limit` in `rs/sns/governance/src/governance.rs` returns **zero matches** — there is no instruction-limit guard anywhere in this function.

`distribute_rewards` is called synchronously from `run_periodic_tasks` at line 5513, which is itself invoked from the repeating timer callback `run_periodic_tasks()` in `rs/sns/governance/canister/canister.rs`. [4](#0-3) [5](#0-4) 

**Contrast with NNS governance**, which was explicitly fixed for this exact problem. The NNS `RewardsDistributionStateMachine::continue_processing` in `rs/nns/governance/src/reward/distribution.rs` processes one neuron at a time and calls `is_over_instructions_limit()` after each, breaking out and resuming in the next timer tick. [6](#0-5) 

The NNS CHANGELOG at lines 655–670 explicitly documents this fix: *"Distribute rewards is moved to timer, and has a mechanism to distribute in batches in multiple messages."* The SNS governance has received no equivalent fix. [7](#0-6) 

When the timer callback traps due to instruction-limit exhaustion, the `latest_reward_event` is never updated and no neuron's `maturity_e8s_equivalent` is incremented. Because the state is not checkpointed mid-loop, every subsequent timer firing encounters the same oversized distribution and traps again. Reward distribution is permanently broken.

## Impact Explanation

This is a **High** severity application/platform-level DoS. Once triggered, no neuron in the affected SNS ever receives voting maturity again, and the reward purse rolls over indefinitely without being paid out. This constitutes a permanent, irreversible loss of governance incentives for all participants in the SNS, matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact"* and *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation

SNS neuron creation is permissionless — any user can stake SNS tokens and create a neuron. No adversarial action is required; organic growth of a popular SNS to tens or hundreds of thousands of participants is sufficient. The `max_number_of_neurons` governance parameter provides a configurable upper bound, but it is not a hard-coded safety limit and can be set to arbitrarily large values by the SNS DAO. With 100,000+ neurons and even a modest number of proposals in a reward period, the O(proposals × neurons) work in Phases 1–2 combined with `HashMap` lookups and `get_neuron_result_mut` calls can plausibly exhaust the 40-billion-instruction budget in a single message.

## Recommendation

Apply the same fix used in NNS governance:

1. Introduce a `RewardsDistributionStateMachine` equivalent for SNS that persists partially-completed distributions in stable state across messages.
2. Add an `is_over_instructions_limit` check after each neuron processed in Phase 2, breaking out and re-scheduling the remainder for the next timer tick.
3. Separate Phase 1 (ballot accumulation) from Phase 2 (maturity crediting) so each phase can be independently interrupted and resumed.
4. Consider adding a hard upper bound on `max_number_of_neurons` that is calibrated against the instruction budget as a defense-in-depth measure.

## Proof of Concept

1. Deploy an SNS with `voting_rewards_parameters` enabled and `max_number_of_neurons` set to a large value (e.g., 200,000).
2. Have many users stake SNS tokens to create N neurons (N large enough to exhaust the instruction budget).
3. Submit and pass a governance proposal so `considered_proposals` is non-empty.
4. Wait for `should_distribute_rewards()` to return `true`.
5. Observe the next `run_periodic_tasks` timer invocation traps with an instruction-limit exceeded error.
6. Observe that `latest_reward_event` is never updated and no neuron's `maturity_e8s_equivalent` increases across subsequent timer firings, confirming permanent reward distribution failure.

A deterministic integration test can reproduce this by constructing a `Governance` instance with a large synthetic neuron set, calling `distribute_rewards` directly, and asserting that the IC instruction counter exceeds the per-message limit before the function returns.

```rust
    fn continue_processing(
        &mut self,
        neuron_store: &mut NeuronStore,
        is_over_instructions_limit: fn() -> bool,
    ) {
        while let Some((id, reward_e8s)) = self.rewards.pop_first() {
            match neuron_store.with_neuron_mut(&id, |neuron| {
                let auto_stake = neuron.auto_stake_maturity.unwrap_or(false);
                if auto_stake {
                    neuron.staked_maturity_e8s_equivalent = Some(
                        neuron
                            .staked_maturity_e8s_equivalent
                            .unwrap_or_default()
                            .saturating_add(reward_e8s),
                    );
                } else {
                    neuron.maturity_e8s_equivalent =
                        neuron.maturity_e8s_equivalent.saturating_add(reward_e8s);
                }

                Ok(_) => {}
                Err(e) => {
                    println!(
                        "{}Error rewarding neuron {:?} during reward_distribution.\
                    This should not be possible as neuron existence is checked when \
                    rewards are calculated: {}",
                        LOG_PREFIX, id, e
                    );
                }
            };
            if is_over_instructions_limit() {
                break;
            }
        }
```

```markdown
## Changed

* Voting Rewards will be scheduled by a timer instead of by heartbeats.
* Unstaking maturity task will be processing up to 100 neurons in a single message, to avoid
  exceeding the instruction limit in a single execution.
* Voting Rewards will be distributed asynchronously in the background after being calculated.
    * This will allow rewards to be compatible with neurons being stored in Stable Memory.
```
