## Title
Unconditional Neuron ID Removal on Failed Transfer Permanently Locks GTC Account Funds - (File: rs/nns/gtc/src/lib.rs)

## Summary
`AccountState::transfer` in `rs/nns/gtc/src/lib.rs` unconditionally removes each neuron ID from `self.neuron_ids` before inspecting the result of `GovernanceCanister::transfer_gtc_neuron`, and always returns `Ok(())`. Both callers then permanently mark the account as donated or forwarded. Any neuron whose inter-canister transfer failed is silently orphaned: the GTC has lost its only reference to it, the account is locked, and the ICP staked in that neuron becomes permanently inaccessible.

## Finding Description
In `AccountState::transfer` (lines 188–210), the loop body unconditionally executes `self.neuron_ids.retain(|id| id != &neuron_id)` at line 192 before the `match result` block at line 200. Whether `GovernanceCanister::transfer_gtc_neuron` returns `Ok` or `Err`, the neuron ID is removed from `self.neuron_ids`. The function then falls through to `Ok(())` at line 209 in all cases.

`donate_account` (lines 89–90) calls `account.transfer(...).await?` and immediately sets `account.has_donated = true`. Because `transfer` never returns `Err`, the `?` never short-circuits, and `has_donated` is always set. `forward_whitelisted_unclaimed_accounts` (lines 118–119) sets `account.has_forwarded = true` on `Ok(_)`, which is every return.

On the Governance side, `transfer_gtc_neuron` (lines 1902–1911) performs a real ledger transfer with `.await?`; if the ledger call fails, the function returns early with an error and the donor neuron is **not** deleted. The donor neuron therefore still exists in Governance, still controlled by the GTC, but the GTC has already removed its ID from `neuron_ids` and locked the account.

Existing guards (`has_claimed`, `has_donated`, `has_forwarded` checks at lines 175–183) only prevent re-entry; they do not protect against the state corruption caused by a failed transfer within a single invocation.

## Impact Explanation
When one or more `transfer_gtc_neuron` calls fail, the affected neuron IDs are erased from `self.neuron_ids`, the account is permanently locked, and the ICP staked in those neurons is frozen with no recovery path. Neither the original account owner nor any other party can claim, donate, forward, or disburse those neurons. This constitutes permanent loss of ICP staked in NNS governance neurons — matching the allowed High impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds"* and *"significant NNS security impact with concrete user or protocol harm."*

## Likelihood Explanation
`forward_whitelisted_unclaimed_accounts` is callable by **any unprivileged principal** after 188 days from genesis (line 27). The failure condition — a transient error from the Governance or ICP ledger canister — arises naturally during routine canister upgrades, which are frequent NNS operations. An attacker who can observe the upgrade schedule (publicly visible via NNS proposals) can time a call to `forward_whitelisted_unclaimed_accounts` to coincide with a ledger or governance upgrade window, reliably triggering the failure path for whitelisted accounts. Even without a deliberate attacker, routine maintenance creates the same risk for `donate_account` callers.

## Recommendation
1. Move `self.neuron_ids.retain(...)` inside the `Ok(_)` arm so the neuron ID is only removed on confirmed success:
```rust
match result {
    Ok(_) => {
        self.neuron_ids.retain(|id| id != &neuron_id);
        self.successfully_transferred_neurons.push(donated_neuron);
    }
    Err(e) => {
        donated_neuron.error = Some(e.to_string());
        self.failed_transferred_neurons.push(donated_neuron);
    }
}
```
2. Return `Err` from `transfer` if any neuron transfer failed, so callers do not set `has_donated`/`has_forwarded` on a partial or fully failed operation.
3. Add a retry endpoint that re-attempts transfers for neurons in `failed_transferred_neurons`, gating the permanent `has_donated`/`has_forwarded` flag only after all neurons are successfully transferred.

## Proof of Concept
1. Alice holds a GTC account with neurons `N1` and `N2`.
2. Alice calls `donate_account` (or any principal calls `forward_whitelisted_unclaimed_accounts` after 188 days).
3. `AccountState::transfer` is entered. For `N1`, `transfer_gtc_neuron` succeeds; `N1` is removed from `neuron_ids` and added to `successfully_transferred_neurons`.
4. For `N2`, the ICP ledger canister is stopped for upgrade at this moment; `transfer_gtc_neuron` returns `Err(...)`.
5. Line 192 (`self.neuron_ids.retain(...)`) still removes `N2` from `neuron_ids`.
6. `N2` is added to `failed_transferred_neurons` with an error string.
7. The loop ends; `transfer` returns `Ok(())`.
8. The caller sets `has_donated = true` (or `has_forwarded = true`).
9. Alice's account is permanently locked. `N2` still exists in Governance controlled by the GTC, but the GTC holds no reference to it. The ICP staked in `N2` is permanently inaccessible.

A deterministic integration test can reproduce this by mocking `GovernanceCanister::transfer_gtc_neuron` to return `Err` for the second neuron ID and asserting that after the call: (a) `neuron_ids` is empty, (b) `has_donated` is `true`, and (c) `failed_transferred_neurons` is non-empty — confirming the orphaned neuron state.
