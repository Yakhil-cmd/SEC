### Title
Permanent Fund Freeze When NEAR Transfer Fails in `ExitToNear` Precompile Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile bridges ETH (base token) or ERC-20 tokens from the Aurora EVM to NEAR. When the downstream NEAR `ft_transfer` or `ft_transfer_call` promise fails, and the engine is compiled **without** the `error_refund` feature flag, there is no callback to refund the user. The EVM-side funds (ETH deducted from balance, or ERC-20 tokens burned) are permanently destroyed with no corresponding NEAR-side credit.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` function constructs a `ExitToNearPrecompileCallbackArgs` struct. The `refund` field is unconditionally set to `None` when the `error_refund` feature is absent:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
```

When both `refund` and `transfer_near` are `None`, `callback_args` equals the default value, so the code takes the simpler branch and schedules **only** the base transfer promise with no callback attached:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no failure callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

If the NEAR `ft_transfer` promise fails (e.g., recipient account not registered with the NEP-141 contract, or Aurora's NEP-141 balance is insufficient), there is no `exit_to_near_precompile_callback` invocation to call `refund_on_error`. The EVM-side deduction (ETH balance subtracted, or ERC-20 tokens burned) has already been committed, and the NEAR-side transfer never completes. The funds are permanently lost.

The test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

and for ETH:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

- For the **ETH (base token) exit path**: the user's EVM ETH balance is decremented by `context.apparent_value` before the NEAR promise is dispatched. If the promise fails without `error_refund`, that ETH is gone from the EVM and never arrives on NEAR.
- For the **ERC-20 exit path**: the ERC-20 tokens are burned from the user's EVM balance before the NEAR `ft_transfer` is dispatched. If the promise fails without `error_refund`, the tokens are destroyed with no NEAR-side credit.

In both cases the loss is permanent and irrecoverable.

---

### Likelihood Explanation

**Medium.** Any user who calls the `ExitToNear` precompile targeting a NEAR account that is not registered with the relevant NEP-141 contract will trigger the failure path. This is a realistic user error (e.g., sending to a freshly created NEAR account that has not called `storage_deposit`). Additionally, if Aurora's own NEP-141 balance is drained (as demonstrated in `test_exit_to_near_eth_refund`), any ETH exit will silently destroy funds. The trigger requires no special privilege — any EVM user can reach this path.

---

### Recommendation

Ensure the `error_refund` feature is always enabled in production builds of `aurora-engine` and `aurora-engine-precompiles`. Alternatively, remove the feature flag entirely and make the refund callback unconditional, so that a failed NEAR transfer always triggers `exit_to_near_precompile_callback` and calls `refund_on_error` to restore the user's EVM balance.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
