Audit Report

## Title
Stale ICP/XDR Spot Rate in `tokens_to_cycles` Enables Bounded Cycle Arbitrage During Rapid Price Drops - (`rs/nns/cmc/src/main.rs`, `rs/nns/cmc/src/exchange_rate_canister.rs`)

## Summary
The Cycles Minting Canister refreshes its `icp_xdr_conversion_rate` at most once every 5 minutes via the Exchange Rate Canister. The `tokens_to_cycles` function reads this cached spot rate with no staleness guard. During a rapid ICP price decline, any unprivileged user can buy ICP cheaply on the open market and convert it to cycles at the stale higher rate, extracting value from the protocol up to the global hourly minting cap.

## Finding Description
`REFRESH_RATE_INTERVAL_SECONDS` is set to `5 * ONE_MINUTE_SECONDS` (300 s) in `rs/nns/cmc/src/exchange_rate_canister.rs:16`. On a successful XRC call the next attempt is scheduled at the next 5-minute boundary; on failure it is deferred by only 1 minute, but the stale rate remains in use for all minting operations in the interim (`exchange_rate_canister.rs:149-161`).

`tokens_to_cycles` (`main.rs:1900-1923`) reads `state.icp_xdr_conversion_rate` directly:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);
```

There is no check on `rate.timestamp_seconds`. The only guard is a `None` check; a present-but-stale rate is used unconditionally. All three public minting entry points (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`) call this function. The state already holds a 30-day `average_icp_xdr_conversion_rate` (`main.rs:224`) that would be far more resistant to short-term swings, but it is not used here.

The global `base_limiter` (`main.rs:239`, `main.rs:358-382`) enforces `DEFAULT_CYCLES_LIMIT = 150e15 cycles` over a rolling 1-hour window (`max_age = Duration::from_secs(60 * 60)`). This limiter is a single shared instance in `State`; it is not per-principal. The claim's assertion that "multiple accounts can each hit the limit independently" is incorrect — all callers share the same 150e15 cycles/hour ceiling.

Exploit flow:
1. Attacker monitors `get_icp_xdr_conversion_rate()` (certified query, free) and an external ICP spot price feed.
2. ICP drops ≥10% on external markets; CMC rate has not yet refreshed (within the 5-minute window).
3. Attacker buys N ICP at the new lower price, transfers to the CMC subaccount with `MEMO_TOP_UP_CANISTER`, calls `notify_top_up`.
4. `process_top_up` → `tokens_to_cycles` uses the stale `xdr_permyriad_per_icp`, minting proportionally more cycles than the ICP is currently worth.
5. Attacker repeats until the global hourly cap is hit or the rate refreshes.

## Impact Explanation
This constitutes illegal minting: cycles are issued at a rate that does not reflect the current ICP value, extracting value from the protocol. The global rate cap bounds the worst-case exposure to ~12.5e15 cycles per 5-minute window (1/12 of the hourly limit). At realistic ICP/XDR values and a 15% price discrepancy, this corresponds to hundreds of ICP-equivalent of protocol loss per window — concrete, repeatable economic harm to the NNS/CMC financial system. This matches the allowed impact class: **High** — significant NNS/CMC infrastructure security impact with concrete protocol harm, requiring no privileged access.

## Likelihood Explanation
ICP price moves of 10–20% within a 5-minute window are historically plausible during market stress. The attack requires no privileged access: any principal can transfer ICP to a CMC subaccount and call `notify_top_up` or `notify_mint_cycles`. The rate discrepancy is observable via a certified query at zero cost. The entire attack is automatable. The window is short (≤5 minutes) but the steps are deterministic and repeatable each refresh cycle.

## Recommendation
1. **Add a staleness guard in `tokens_to_cycles`**: Reject minting if `icp_xdr_conversion_rate.timestamp_seconds` is older than a configurable threshold (e.g., 10 minutes), returning a retriable `NotifyError` so callers can retry after the rate refreshes.
2. **Use `average_icp_xdr_conversion_rate` for minting**: The CMC already maintains a 30-day TWAP (`main.rs:224`, `main.rs:891-912`). Switching `tokens_to_cycles` to use this rate would make cycle pricing far more resistant to short-term price swings.
3. **Reduce `REFRESH_RATE_INTERVAL_SECONDS`**: Shortening the interval narrows the exploitable window at the cost of more frequent XRC calls.

## Proof of Concept
Deterministic integration test plan using `StateMachine` / PocketIC:

1. Set up NNS with a mock XRC returning rate R.
2. Advance time past one refresh cycle so CMC caches rate R.
3. Update the mock XRC to return rate R × 0.85 (15% drop) but do **not** advance time enough to trigger a CMC refresh.
4. Fund a test principal with N ICP at the new lower price.
5. Call `notify_top_up` for that principal.
6. Assert that cycles minted = `tokens_to_cycles(N ICP, rate=R)` (stale rate), not `tokens_to_cycles(N ICP, rate=R×0.85)` (current rate).
7. Confirm the delta represents ~17.6% excess cycles minted relative to current market value.

This test is directly expressible using the existing `state_machine_builder_for_nns_tests` and `setup_mock_exchange_rate_canister` infrastructure already present in `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs`.
