Audit Report

## Title
Missing Reentrancy Guard in `perform_mint_sns_tokens` Allows Double-Minting of SNS Tokens - (File: rs/sns/governance/src/governance.rs)

## Summary
`perform_mint_sns_tokens` calls `self.ledger.transfer_funds(...).await` without any concurrency guard. Because the proposal's execution status is only written after the function returns, a second heartbeat processed during the `await` checkpoint sees the proposal still in `Adopted` state and re-executes it, minting SNS tokens a second time. The identical pattern in `perform_transfer_sns_treasury_funds` is explicitly protected by an `IN_PROGRESS_PROPOSAL_ID` lock; `perform_mint_sns_tokens` has no such protection.

## Finding Description
`perform_mint_sns_tokens` (lines 3062–3088) issues an inter-canister call with no guard:

```rust
self.ledger
    .transfer_funds(amount_e8s, 0, None, to, mint.memo())
    .await?;   // no lock acquired before this point
```

`set_proposal_execution_status` is called only after the function returns (line 2242). During the `await`, the proposal remains `Adopted`.

The IC execution model allows other messages — including heartbeats — to be processed at every `await` point. If `run_periodic_tasks` fires a second heartbeat while the first `transfer_funds` call is in-flight, it finds the proposal still `Adopted` and spawns a second execution of `perform_mint_sns_tokens`, issuing a second `transfer_funds` to the ledger.

A canister upgrade during the `await` window compounds this: the in-flight callback is dropped, the proposal state is restored from stable memory still as `Adopted`, and the next heartbeat re-executes the proposal.

The directly adjacent function `perform_transfer_sns_treasury_funds` (lines 2980–3060) demonstrates the correct pattern:

```rust
thread_local! {
    static IN_PROGRESS_PROPOSAL_ID: RefCell<Option<u64>> = const { RefCell::new(None) };
}
let release_on_drop = acquire(&IN_PROGRESS_PROPOSAL_ID, proposal_id);
if let Err(already_in_progress_proposal_id) = release_on_drop {
    return Err(...);
}
// ... then await
```

`perform_mint_sns_tokens` has no equivalent guard.

## Impact Explanation
An adopted `MintSnsTokens` proposal can be executed more than once, minting SNS tokens to the target account multiple times beyond what governance approved. The minted tokens are immediately spendable. This constitutes unauthorized token issuance against an SNS ledger — a concrete financial harm matching the allowed impact: *Significant SNS security impact with concrete user or protocol harm* (High, $2,000–$10,000), or Critical if the minted amount is large.

## Likelihood Explanation
No special privileges beyond sufficient SNS voting power are required. Two concurrently adopted proposals (`MintSnsTokens` + any proposal that triggers a heartbeat overlap) are a normal operational scenario. An adversary controlling enough voting power can deliberately time proposals to maximize the overlap window. The upgrade-drop path additionally requires only that an `UpgradeSnsToNextVersion` or NNS-initiated upgrade coincides with an in-flight mint, which is routine for SNS canisters.

## Recommendation
Before the first `await` in `perform_mint_sns_tokens`, acquire a `thread_local` lock keyed by proposal ID, identical to the pattern already used in `perform_transfer_sns_treasury_funds`:

```rust
thread_local! {
    static IN_PROGRESS_PROPOSAL_ID: RefCell<Option<u64>> = const { RefCell::new(None) };
}
let release_on_drop = acquire(&IN_PROGRESS_PROPOSAL_ID, proposal_id);
if let Err(id) = release_on_drop {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!("MintSnsTokens proposal {id} is already in progress."),
    ));
}
```

The function signature must also accept `proposal_id` (as `perform_transfer_sns_treasury_funds` does) so the guard can be keyed correctly.

## Proof of Concept
1. Submit and pass a `MintSnsTokens` proposal targeting an attacker-controlled account.
2. In a PocketIC or local replica test, intercept the `transfer_funds` call to the ledger so it is in-flight (not yet responded).
3. While the call is in-flight, trigger a second heartbeat (or simulate a canister upgrade followed by a heartbeat).
4. Observe that `perform_mint_sns_tokens` is entered a second time and a second `transfer_funds` is issued.
5. Confirm the target account balance equals `2 × amount_e8s` while the proposal was only approved once.

A deterministic integration test using PocketIC's `advance_time` + manual heartbeat injection can reproduce this without mainnet access.
