### Title
L1→L2 Transaction Sender Can Set `gas_per_pubdata_limit = 0` to Avoid All Pubdata Native-Resource Charges — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

For L1→L2 (priority) transactions, the `gas_per_pubdata_limit` field is read directly from the user-supplied transaction and used without any lower-bound validation to derive `native_per_pubdata` — the native-resource cost per byte of pubdata. Setting this field to `0` causes `native_per_pubdata = 0`, which makes every pubdata-resource check trivially pass and charges the sender zero native resources for all pubdata generated by the transaction.

---

### Finding Description

In `process_l1_transaction.rs`, the bootloader reads the user-supplied `gas_per_pubdata_limit` field verbatim:

```rust
// For L1->L2 transactions we always use the pubdata price provided by the transaction.
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
``` [1](#0-0) 

This value is then multiplied by `native_per_gas` to produce `native_per_pubdata`:

```rust
let native_per_pubdata = (gas_per_pubdata as u64)
    .checked_mul(native_per_gas)
    .unwrap_or_else(|| { ... u64::MAX });
``` [2](#0-1) 

When `gas_per_pubdata = 0`, `native_per_pubdata = 0`. This zero value propagates into two critical places:

**1. Intrinsic pubdata overhead in `create_resources_for_tx`:**

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0  → no native deducted for intrinsic pubdata
``` [3](#0-2) 

**2. Post-execution pubdata charge in `get_resources_to_charge_for_pubdata`:**

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // = pubdata_used * 0 = 0
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

The `check_enough_resources_for_pubdata` call that guards execution will always return `enough = true` because zero native is required regardless of how much pubdata the transaction generates. [5](#0-4) 

By contrast, for L2 (EIP-712 / ZK) transactions, `native_per_pubdata` is derived entirely from operator-set `pubdata_price` and `native_price` — not from any user field — so this path is not exploitable for L2 transactions. [6](#0-5) 

The deposited-amount check only verifies `total_deposited >= gas_price * gas_limit`; it does not account for pubdata costs at all, so there is no compensating control: [7](#0-6) 

---

### Impact Explanation

A user who submits an L1→L2 priority transaction with `gas_per_pubdata_limit = 0` pays **zero native resources** for every byte of pubdata their transaction generates. Pubdata costs (state-diff publication, L1 messaging) are real off-chain costs borne by the operator/protocol. The attacker can write to many storage slots, emit large L1 messages, or deploy large contracts — all without paying the native-resource cost that would normally cover proving and data-availability expenses. This is a direct, quantifiable financial loss for the protocol proportional to the pubdata generated.

---

### Likelihood Explanation

Any externally-owned account that can submit an L1→L2 transaction (a standard, permissionless operation on any ZKsync chain) can set `gas_per_pubdata_limit = 0`. No privileged role, leaked key, or oracle manipulation is required. The ZKsync OS bootloader performs no lower-bound check on this field. The only potential mitigation would be an L1-side contract check, but that is outside the ZKsync OS scope and is explicitly noted in the code as an invariant "enforced on L1" — meaning the bootloader itself does not enforce it.

---

### Recommendation

Add a minimum-value check on `gas_per_pubdata` for L1→L2 transactions in `prepare_and_check_resources` (or immediately after reading the field). A reasonable minimum is `1`, or a protocol-defined constant analogous to `MIN_GAS_PER_PUBDATA_BYTE` used in prior ZKsync versions. For example:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
// Enforce a minimum to prevent zero-cost pubdata generation
if gas_per_pubdata == 0 {
    // For L1 txs: saturate to a safe minimum rather than reject
    gas_per_pubdata = MIN_GAS_PER_PUBDATA_BYTE;
}
```

Alternatively, derive `native_per_pubdata` from the operator's `pubdata_price / native_price` as a floor, taking the maximum of the user-supplied value and the operator floor.

---

### Proof of Concept

1. Attacker constructs an L1→L2 transaction with:
   - `gas_per_pubdata_limit = 0`
   - `gas_limit` large enough to execute the body
   - `gas_price` sufficient to pass the `total_deposited >= gas_price * gas_limit` check
   - Calldata that writes to many storage slots (e.g., 100 SSTORE operations)

2. The bootloader reads `gas_per_pubdata = 0` at line 80 of `process_l1_transaction.rs`.

3. `native_per_pubdata = 0 * native_per_gas = 0` at line 481.

4. `intrinsic_pubdata_overhead = 0 * L1_TX_INTRINSIC_PUBDATA = 0` — no native deducted upfront.

5. After execution, `check_enough_resources_for_pubdata` computes `native_needed = pubdata_used * 0 = 0` — the check passes unconditionally.

6. The attacker's 100 SSTORE operations generate ~3,200 bytes of pubdata at zero native cost. A normal transaction paying the operator's `pubdata_price / native_price` rate would have been charged `3200 * native_per_pubdata` native units for the same pubdata.
