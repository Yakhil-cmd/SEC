Audit Report

## Title
VIP Callers Blocked by Zero-Slot Guard Before VIP Check in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary

`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` evaluates the `available_slot_count == 0` early-return guard unconditionally for all callers, including VIP callers (NNS canisters), before the VIP path (`used_slot_count = 0`) can take effect. An unprivileged attacker who floods Root's public `canister_status` endpoint with 167 concurrent calls exhausts all slots and causes subsequent VIP calls from NNS Governance to be rejected with `SysTransient`, blocking governance-triggered canister management operations for as long as the flood is sustained.

## Finding Description

In `try_borrow_slot` (lines 264–287 of `rs/nervous_system/clients/src/management_canister_client.rs`):

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };  // VIP uses 0 slots

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {          // ← fires for ALL callers, including VIP
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));         // ← VIP never reaches the sub below
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The intent is that VIP callers (`is_caller_vip = true`) consume zero slots and should always pass through. However, the `== 0` guard at line 269 is evaluated **before** the subtraction, so when `available_slot_count` reaches 0, every caller — VIP or not — receives `Err(SysTransient)`.

The public `canister_status` endpoint on Root (lines 88–98 of `rs/nns/handlers/root/impl/canister/canister.rs`) has no authorization check. Any principal can call it. `new_management_canister_client()` (lines 53–67) sets `is_caller_vip` based on whether the caller is in `ALL_NNS_CANISTER_IDS`, and all calls share the single thread-local `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` initialized to 167 (line 50).

The existing test (lines 173–255 of `rs/nervous_system/clients/src/management_canister_client/tests.rs`) validates VIP behavior only when slots are **not** exhausted — VIP calls start at `t=0ms`, before pleb calls at `t=50ms`. It does not cover the case where a VIP call arrives when `available_slot_count == 0`.

## Impact Explanation

When all 167 slots are occupied by attacker-controlled `canister_status` calls, the following governance-triggered endpoints return `Err(SysTransient)` and fail:

- `update_canister_settings` (NNS Governance only, lines 220–230)
- `change_canister_controllers` (SNS-W only, lines 207–216)
- `take_canister_snapshot` (NNS Governance only, lines 244–254)
- `load_canister_snapshot` (NNS Governance only, lines 258–268)

This constitutes an application/platform-level DoS of NNS governance canister management operations, matching the **High** impact category: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

Note: `change_nns_canister` (the main canister upgrade path, lines 135–165) calls `change_canister` directly and does not go through `new_management_canister_client()`, so it is not affected.

## Likelihood Explanation

The `canister_status` endpoint is public and requires no authorization. Any principal can send 167 concurrent update calls to Root. Because Root suspends on each inter-canister call to the management canister, all 167 slots can be simultaneously occupied within a single consensus round. The `SlotLoan` RAII guard releases slots on drop, so the DoS window is bounded by the round-trip time of the management canister's response (~1–2 consensus rounds). The attacker must sustain the flood to maintain the condition, but the attack is repeatable, requires no privileged access, leaked keys, or subnet-majority corruption, and can be automated trivially.

## Recommendation

Fix the ordering in `try_borrow_slot` so that VIP callers bypass the zero-slot guard entirely:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            // VIP callers consume 0 slots and must never be throttled.
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

Additionally, add a test that verifies a VIP call succeeds when `available_slot_count == 0` (i.e., after all non-VIP slots are occupied).

## Proof of Concept

Using the existing test harness with `SLOTS = 2`:

1. Spawn 2 non-VIP `canister_status` futures with a long `inner_duration` (e.g., 500ms) — both borrow a slot and suspend, driving `available_slot_count` to 0.
2. While both are in-flight (`available_slot_count == 0`), call `try_borrow_slot` with `is_caller_vip = true`.
3. **Observed**: `Err((SysTransient, "Unavailable. Maybe, try again later?"))` — VIP call rejected.
4. **Expected**: `Ok(SlotLoan { used_slot_count: 0 })` — VIP call passes through.

The existing test at lines 173–255 of `tests.rs` shows VIP calls succeed only because they start at `t=0ms`, before pleb calls exhaust the slots at `t=50ms` — it does not cover the failure case described above.
