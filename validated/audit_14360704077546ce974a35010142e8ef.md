Audit Report

## Title
Stale ICP/XDR Conversion Rate Used Without Staleness Check in CMC `tokens_to_cycles` — (File: rs/nns/cmc/src/main.rs)

## Summary

The Cycles Minting Canister stores an ICP/XDR conversion rate that is refreshed every 5 minutes via the exchange rate canister. When the exchange rate canister returns an error, the CMC silently retains the last-known rate with no expiry. The `tokens_to_cycles` function, called by every ICP-to-cycles minting path, reads this rate without any staleness check. During a sustained exchange rate canister outage coinciding with an ICP price drop, any unprivileged caller can mint cycles at an inflated rate, extracting value from the cycles economy.

## Finding Description

`tokens_to_cycles` at `rs/nns/cmc/src/main.rs:1900–1922` reads `state.icp_xdr_conversion_rate` and extracts only `xdr_permyriad_per_icp`, discarding the `timestamp_seconds` field entirely:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);   // timestamp_seconds never read
        ...
    })
}
```

The rate is refreshed by a heartbeat-driven call to the exchange rate canister every `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes, `rs/nns/cmc/src/exchange_rate_canister.rs:16`). When the XRC call fails, the error path at `rs/nns/cmc/src/exchange_rate_canister.rs:270–274` returns `UpdateExchangeRateError::FailedToRetrieveRate` without touching `state.icp_xdr_conversion_rate`, leaving the stale rate in place indefinitely.

`do_set_icp_xdr_conversion_rate` at `rs/nns/cmc/src/main.rs:1009–1040` only enforces that a new rate's timestamp is strictly greater than the current one — it imposes no maximum age on the stored rate at consumption time.

The integration test at `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs:162–185` explicitly exercises and asserts this behavior: after the XRC is reinstalled to return `StablecoinRateTooFewRates`, the CMC's published rate retains the previous timestamp and value.

Exploit path:
1. Exchange rate canister begins returning errors (e.g., `StablecoinRateTooFewRates`, canister trap, or network partition).
2. ICP market price drops below the stale stored rate.
3. Attacker calls `notify_top_up` or `notify_mint_cycles` with ICP tokens.
4. CMC calls `tokens_to_cycles`, which uses the stale (inflated) `xdr_permyriad_per_icp` without any age check.
5. Attacker receives more cycles than the current market rate warrants.
6. Attacker repeats up to `base_cycles_limit` per hour for as long as the outage persists.

## Impact Explanation

This constitutes illegal minting of cycles at below-market rates, directly harming the cycles economy. Cycles are the protocol's unit of computation cost; minting them at an inflated ICP/XDR ratio dilutes their value and allows extraction of subsidized computation. The `base_cycles_limit` caps per-hour damage but does not bound total damage across a multi-hour outage. This matches the allowed impact: **High — Significant XRC/NNS infrastructure security impact with concrete protocol harm**, and potentially **Critical — illegal minting of exorbitant cycles** depending on the duration of the outage and the magnitude of the price divergence.

## Likelihood Explanation

The exchange rate canister is a NNS system canister that queries external data sources for stablecoin rates. `StablecoinRateTooFewRates` is a documented, tested error condition that can arise from external data source unavailability. The CMC's own integration test suite explicitly exercises and asserts the stale-rate-retention behavior. ICP price is volatile. A window of minutes to hours where the XRC is degraded and ICP price drops is a realistic, monitorable condition. An attacker watching on-chain CMC state can detect the divergence between the published `icp_xdr_conversion_rate` timestamp and wall-clock time and act within that window. No special privileges are required — `notify_top_up` and `notify_mint_cycles` are open to any principal.

## Recommendation

Add a staleness guard in `tokens_to_cycles` (or equivalently in `process_top_up` / `process_mint_cycles`) that rejects minting when the stored rate's `timestamp_seconds` is older than a configurable threshold (e.g., 30 minutes):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref()
            .ok_or_else(|| NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "No conversion rate found in CMC".to_string(),
            })?;

        let age_seconds = now_seconds().saturating_sub(rate.timestamp_seconds);
        if age_seconds > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: format!(
                    "ICP/XDR conversion rate is stale ({age_seconds}s old); minting suspended"
                ),
            });
        }

        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

`MAX_RATE_AGE_SECONDS` should be set to a value meaningfully larger than `REFRESH_RATE_INTERVAL_SECONDS` (5 min) to tolerate transient XRC failures, but small enough to bound exposure (e.g., 30 minutes).

## Proof of Concept

A deterministic integration test using the existing `state_machine` / `MockExchangeRateCanister` infrastructure:

1. Initialize CMC with a known rate (e.g., 100,000 `xdr_permyriad_per_icp`).
2. Reinstall the mock XRC to return `ExchangeRateError::StablecoinRateTooFewRates` (as done in the existing test at `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs:162`).
3. Advance time by 2+ hours (`state_machine.advance_time`); trigger heartbeats.
4. Assert that `get_icp_xdr_conversion_rate` still returns the original timestamp (stale rate retained).
5. Transfer ICP to the CMC's top-up subaccount for a test canister.
6. Call `notify_top_up`; assert that cycles minted correspond to the stale 100,000 rate, not a lower current-market rate.
7. Confirm no staleness error is returned — demonstrating the missing guard.
