### Title
ICRC-2 Allowance Revocation Race Condition via Missing `expected_allowance` Guard in `remove_approval` - (File: rs/ledger_suite/icp/ledger/src/main.rs)

---

### Summary

The ICP ledger's `remove_approval` endpoint unconditionally sets `expected_allowance: None` when revoking a spender's allowance. Because the core `approve` logic only enforces the current-allowance check when `expected_allowance` is `Some(...)`, a malicious spender who observes Alice's pending revocation can race a `icrc2_transfer_from` call and drain the full original allowance before the revocation is applied.

---

### Finding Description

The `ApproveArgs` struct exposes `expected_allowance` as an optional field: [1](#0-0) 

The core allowance table's `approve` function only enforces the current-allowance check when `expected_allowance` is `Some`: [2](#0-1) 

When `expected_allowance` is `None`, the function falls through and unconditionally overwrites the stored allowance: [3](#0-2) 

The ICP ledger's dedicated `remove_approval` endpoint hard-codes `expected_allowance: None`: [4](#0-3) 

This means the revocation path never asserts what the current allowance is before zeroing it, leaving a window for a spender to drain the full original allowance between Alice's revocation submission and its execution.

The ICRC-1 ledger's `icrc2_approve` endpoint also passes `expected_allowance` through as-is, so any caller who omits it (the default) is equally unprotected: [5](#0-4) 

---

### Impact Explanation

A spender who has been granted an allowance and who can submit a `icrc2_transfer_from` call concurrently with the account owner's revocation can drain the full original allowance before the revocation is applied. On the IC, multiple ingress messages can reside in the pool simultaneously; a malicious spender canister that monitors ledger state can react within the same consensus round. After the race, the spender has drained the original allowance and the owner's revocation still succeeds — but the tokens are already gone.

---

### Likelihood Explanation

Low. The IC does not expose a public mempool in the same way Ethereum does, making opportunistic front-running harder for external observers. However, a malicious spender **canister** can poll `icrc2_allowance` and submit `icrc2_transfer_from` the moment it detects a pending revocation, exploiting the inter-round ordering window. The `remove_approval` code path concretely omits the only available guard (`expected_allowance`), making the revocation path structurally unprotected.

---

### Recommendation

**Short term:** Document that `remove_approval` and any `icrc2_approve` call that omits `expected_allowance` is subject to a race condition. Users and integrators should always supply `expected_allowance` equal to the current allowance when reducing or revoking an approval.

**Long term:** Change `remove_approval` to populate `expected_allowance` with the current on-chain allowance before submitting the zero-amount approval, so the revocation atomically fails if the spender has already drained the allowance in the same round. Alternatively, enforce that `expected_allowance` is required (non-optional) for any approval that reduces an existing non-zero allowance.

---

### Proof of Concept

1. Alice calls `icrc2_approve(SpenderCanister, 100_000)` — allowance is now 100,000 tokens.
2. Alice decides to revoke and calls `remove_approval(SpenderCanister)`.
3. `remove_approval` constructs `ApproveArgs { amount: 0, expected_allowance: None, ... }` and submits it.
4. SpenderCanister, running concurrently, calls `icrc2_transfer_from(Alice → Attacker, 100_000)`.
5. If SpenderCanister's message is ordered before Alice's revocation in the same consensus round, `use_allowance` succeeds and 100,000 tokens are transferred.
6. Alice's revocation then executes against an already-zero allowance — it succeeds silently, but the tokens are gone.
