Audit Report

## Title
Unbounded Sequential Cross-Canister Calls in `maybe_finalize_disburse_maturity` Enables SNS Governance Liveness Degradation — (`rs/sns/governance/src/governance.rs`)

## Summary
`maybe_finalize_disburse_maturity` in the SNS governance canister collects all neurons with matured disbursements into an unbounded `Vec` and then issues a sequential `.await` cross-canister `transfer_funds` call for each entry within a single heartbeat invocation. An unprivileged attacker holding sufficient SNS tokens can force this loop to make an arbitrarily large number of sequential ledger round-trips, holding the `is_finalizing_disburse_maturity` re-entry flag for the entire duration and delaying all subsequent periodic governance tasks. The NNS governance canister has already fixed this exact pattern by processing exactly one disbursement per timer invocation.

## Finding Description
At `rs/sns/governance/src/governance.rs` lines 4938–4975, the function collects every neuron whose first pending disbursement has passed its `finalize_disbursement_timestamp_seconds` into an unbounded `Vec`:

```rust
let neuron_id_and_disbursements: Vec<(NeuronId, DisburseMaturityInProgress)> = self
    .proto.neurons.values()
    .filter_map(|neuron| { ... Some((id.clone(), first_disbursement.clone())) })
    .collect();
```

At lines 4976–5082, it iterates over the entire collected set and issues a blocking cross-canister call for each entry:

```rust
for (neuron_id, disbursement) in neuron_id_and_disbursements.into_iter() {
    ...
    let transfer_result = self.ledger.transfer_funds(...).await;  // sequential, unbounded
    ...
}
self.proto.is_finalizing_disburse_maturity = None;  // flag released only after all N calls
```

The `is_finalizing_disburse_maturity` flag is set at line 4935 and cleared only at line 5082, after all N calls complete. The only guard, `can_finalize_disburse_maturity()`, prevents re-entry but does nothing to bound the work per invocation. There is no cap on the number of SNS neurons.

This function is called unconditionally from `run_periodic_tasks` at line 5527:
```rust
self.maybe_finalize_disburse_maturity().await;
```

The tasks that follow in the same heartbeat invocation (`maybe_move_staked_maturity`, `compute_cached_metrics`, `maybe_gc`) are all delayed until the N-call loop completes. Subsequent heartbeat invocations that attempt disbursement finalization return immediately due to the held flag.

By contrast, the NNS governance canister processes exactly **one** disbursement per timer invocation via `try_finalize_maturity_disbursement` (`rs/nns/governance/src/governance/disburse_maturity.rs`, lines 462–469), implemented as a `RecurringAsyncTask` that self-schedules based on the next pending disbursement timestamp (`rs/nns/governance/src/timer_tasks/finalize_maturity_disbursements.rs`, lines 21–33). The SNS canister has no equivalent bound.

## Impact Explanation
An attacker controlling N SNS neurons with simultaneously matured disbursements causes `maybe_finalize_disburse_maturity` to issue N sequential cross-canister calls to the SNS ledger in a single heartbeat invocation. Each call requires at least one consensus round-trip. During this period: (1) the `is_finalizing_disburse_maturity` flag blocks all further disbursement finalization from subsequent heartbeats; (2) periodic tasks scheduled after `maybe_finalize_disburse_maturity` in `run_periodic_tasks` (reward distribution, upgrade checks, metrics) are delayed proportionally to N; (3) the governance canister pays cycles for every ledger call, constituting an attacker-directed cycles drain funded by the SNS treasury. This constitutes an application/platform-level DoS on SNS governance with concrete, scalable harm — mapping to **High ($2,000–$10,000)**.

## Likelihood Explanation
The attack is permissionless for any party holding sufficient SNS tokens. The steps require only standard SNS user operations: create N neurons, call `DisburseMaturity` on each, and wait 7 days for `MATURITY_DISBURSEMENT_DELAY_SECONDS` to elapse. No privileged access, no social engineering, and no external dependency is required. The incubation period blends with normal user behavior. The attacker's cost is the opportunity cost of locking SNS tokens, which is recoverable after dissolving. The attack is repeatable as long as the attacker retains the neurons.

## Recommendation
Apply the same fix already used in the NNS governance canister: replace the unbounded `for` loop in `maybe_finalize_disburse_maturity` with a single-disbursement pattern. Process exactly one maturity disbursement per heartbeat/timer invocation, analogous to `try_finalize_maturity_disbursement` in `rs/nns/governance/src/governance/disburse_maturity.rs`. Convert the SNS periodic task to a `RecurringAsyncTask` (as in `rs/nns/governance/src/timer_tasks/finalize_maturity_disbursements.rs`) that self-schedules based on the next pending disbursement timestamp, eliminating both the unbounded loop and the need for the `is_finalizing_disburse_maturity` re-entry flag.

## Proof of Concept
1. Deploy an SNS instance on a local PocketIC or test network.
2. Create N neurons (e.g., N = 500) each holding the minimum stake, all controlled by the attacker principal.
3. Call `ManageNeuron { DisburseMaturity { percentage_to_disburse: 100, ... } }` on each neuron.
4. Advance the test clock by `MATURITY_DISBURSEMENT_DELAY_SECONDS + 1` so all disbursements mature simultaneously.
5. Trigger a heartbeat. Observe via canister logs that `maybe_finalize_disburse_maturity` issues N sequential `transfer_funds` calls to the SNS ledger.
6. Measure the round-count delay before `maybe_move_staked_maturity`, `compute_cached_metrics`, and `maybe_gc` execute in the same heartbeat invocation — the delay scales linearly with N.
7. Confirm that subsequent heartbeats return immediately from `can_finalize_disburse_maturity()` while the N-call loop is in progress, blocking all disbursement finalization.
