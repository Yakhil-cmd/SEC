### Title
Partial NEP-141 `ft_transfer_call` Refunds Not Re-minted as ERC-20 Tokens in `ExitToNear` Precompile — (`engine-precompiles/src/native.rs`)

---

### Summary

When a user exits ERC-20 tokens (or the base ETH token) to NEAR via the `ExitToNear` precompile using the **Omni message path**, the precompile calls `ft_transfer_call` on the NEP-141 contract. Under the NEP-141 standard, the receiving contract may return a partial refund (unused tokens). These refunded tokens are silently credited back to Aurora's NEP-141 account balance, but no corresponding ERC-20 tokens are re-minted for the user. The user's ERC-20 tokens were already burned before the cross-chain call, so the refunded amount is permanently frozen in Aurora's NEP-141 account with no ERC-20 representation and no recovery path.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles two exit paths for ERC-20 tokens:

1. **Legacy path** (`ft_transfer`): a simple one-way transfer with no receiver callback.
2. **Omni path** (`ft_transfer_call`): the receiver's `ft_on_transfer` is invoked, and the receiver may return an unused amount (partial refund) per the NEP-141 standard.

In the Omni path, the precompile constructs a `ft_transfer_call` promise: [1](#0-0) 

After the promise is built, the precompile decides whether to attach a callback: [2](#0-1) 

The `callback_args` struct is: [3](#0-2) 

For the Omni path, `transfer_near_args` is `None` (only set for wNEAR unwrap), and `refund` is only populated when the `error_refund` feature is compiled in. Without `error_refund`, `callback_args` equals `default()`, so **no callback is attached at all**. Even when `error_refund` is enabled, the callback (`exit_to_near_precompile_callback`) only handles the case where the entire `ft_transfer_call` fails — it does not inspect the return value of `ft_on_transfer` to detect a partial refund.

The NEP-141 `ft_transfer_call` flow is:
1. Tokens transferred from Aurora → receiver.
2. `ft_on_transfer` called on receiver; receiver returns unused amount.
3. NEP-141 contract automatically refunds unused amount back to Aurora's NEP-141 balance.

Step 3 happens entirely within the NEP-141 contract's `ft_resolve_transfer` — Aurora receives no callback and has no hook to detect or act on the refund. The refunded NEP-141 tokens silently accumulate in Aurora's account.

Meanwhile, the ERC-20 burn already occurred on the EVM side before the precompile was invoked: [4](#0-3) 

The comment explicitly states this precompile branch is called from the ERC-20 burn function. The burn is irreversible. The refunded NEP-141 tokens have no ERC-20 counterpart and cannot be recovered by the user.

The same issue applies to the base ETH token Omni path: [5](#0-4) 

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

A user who calls `exitToNear` with an Omni message to any NEP-141 receiver that returns a partial refund (standard, legitimate NEP-141 behavior — e.g., a DEX that only consumes part of the input) will have their ERC-20 tokens burned with no recovery. The refunded NEP-141 tokens are credited to Aurora's own NEP-141 account balance, where they are permanently inaccessible: there is no ERC-20 representation, no re-mint mechanism, and no admin recovery path exposed in the precompile or its callback.

---

### Likelihood Explanation

**Medium.** The Omni message path is an intentional, documented feature of the `ExitToNear` precompile. Any NEP-141 contract that implements `ft_on_transfer` and returns a non-zero refund (which is standard and expected behavior for DEXes, lending protocols, etc.) will trigger this bug. No special attacker capability is required — any unprivileged EVM user who calls `exitToNear` with an Omni message to such a receiver will lose funds.

---

### Recommendation

Attach a callback after `ft_transfer_call` that reads the promise result (the return value of `ft_on_transfer`, which is the refunded amount). If the refunded amount is non-zero, re-mint the corresponding ERC-20 tokens (or re-credit ETH) to the original caller's EVM address. This is analogous to the existing `error_refund` mechanism but must also cover the partial-refund case, not only full failures.

Concretely:
- Extend `ExitToNearPrecompileCallbackArgs` with a field for the partial-refund case.
- Always attach a callback for the `ft_transfer_call` (Omni) path.
- In `exit_to_near_precompile_callback`, read the promise result and re-mint ERC-20 tokens equal to the returned refund amount.

---

### Proof of Concept

1. Deploy a NEP-141 receiver contract on NEAR that implements `ft_on_transfer` and returns `amount` (i.e., refunds 100% of tokens).
2. From an Aurora EVM address, hold 100 units of an ERC-20 token backed by that NEP-141.
3. Call the ERC-20's `withdrawToNear` function with an Omni message targeting the malicious receiver. This burns 100 ERC-20 tokens and calls `ft_transfer_call` on the NEP-141 contract.
4. The receiver's `ft_on_transfer` returns `100` (full refund). The NEP-141 contract refunds 100 tokens to Aurora's account.
5. Observe: the user's ERC-20 balance is 0 (burned), Aurora's NEP-141 balance increased by 100, and no ERC-20 tokens were re-minted. The 100 tokens are permanently frozen.
