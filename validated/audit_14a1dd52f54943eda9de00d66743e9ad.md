### Title
L1→L2 Priority Queue Permanently Halted if `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Reverts - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function in the ZKsync OS bootloader unconditionally treats any revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a **fatal system error that halts block processing**. Because every L1→L2 priority transaction with a non-zero deposit triggers this call, and because L1→L2 transactions form a FIFO priority queue that cannot be skipped, any condition that causes `L2AssetTracker` to revert will permanently block the entire L1→L2 priority queue — halting the chain.

---

### Finding Description

During L1→L2 priority transaction processing, the bootloader calls `mint_base_token` up to three times (value mint, operator fee, refund). Each call internally invokes `notify_l2_asset_tracker`, which executes `handleFinalizeBaseTokenBridgingOnL2(l1_chain_id, amount)` on the `L2AssetTracker` contract at `L2_ASSET_TRACKER_ADDRESS`. [1](#0-0) 

The critical design flaw is at lines 901–911:

```rust
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    )
    .into());
}
``` [2](#0-1) 

This fatal error propagates upward through `mint_base_token` → `execute_l1_transaction_and_notify_result` → `process_l1_transaction`. The error is **not** caught as a `FatalRuntimeError` (which would be converted to a top-level revert), so it propagates as an unrecoverable `BootloaderSubsystemError::LeafDefect`, halting the entire block. [3](#0-2) 

The three call sites where `mint_base_token` is invoked (and thus `notify_l2_asset_tracker`) are:

1. **Value mint** (line ~634): minting `to_transfer` to `from` inside the execution frame
2. **Operator fee** (line ~290): minting `pay_to_operator` to `coinbase`
3. **Refund** (line ~338): minting `to_refund_recipient` to `refund_recipient` [4](#0-3) [5](#0-4) [6](#0-5) 

The `l1_chain_id` passed to `handleFinalizeBaseTokenBridgingOnL2` is read from `L2AssetTracker` storage slot 154: [7](#0-6) 

The bootloader's own comment acknowledges the fragility:

> *"Failure halts block processing — if the asset tracker reverts, the chain's token accounting would be inconsistent, so we treat it as fatal rather than silently continuing with incorrect bookkeeping."* [8](#0-7) 

The L1→L2 priority queue is explicitly documented as a chain-halting dependency: [9](#0-8) 

---

### Impact Explanation

If `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` reverts for **any reason** during processing of an L1→L2 priority transaction with a non-zero deposit, the entire block processing halts. Because L1→L2 transactions form a FIFO priority queue that cannot be skipped (invalidating them would itself halt the chain), the queue is permanently blocked. All subsequent L1→L2 deposits are frozen. Users who have already committed funds on L1 cannot receive them on L2. This is a complete chain halt / permanent DOS of the L1→L2 bridge.

---

### Likelihood Explanation

The `L2AssetTracker` is a system contract predeploy. Revert conditions include:

1. **Contract upgrade introducing a revert path** — any governance-approved upgrade that adds a revert condition (e.g., a pause mechanism, a new access check, or a bug) immediately halts the chain for all future L1→L2 deposits.
2. **Bug in `L2AssetTracker`** — the contract performs storage writes (`chainBalance`, `totalSuccessfulDepositsFromL1`). An arithmetic overflow or an uninitialized state (e.g., `baseTokenAssetId == 0`, `isAssetRegistered == false`) could cause a revert.
3. **State manipulation via L2 transactions** — if any L2-callable function on `L2AssetTracker` can be used to put the contract into a state where `handleFinalizeBaseTokenBridgingOnL2` reverts (e.g., deregistering the base token asset), an unprivileged attacker could trigger this before submitting an L1→L2 transaction.

The design is fragile by construction: a single-point-of-failure push to an external contract with no fallback, in a queue that cannot be skipped.

---

### Recommendation

1. **Do not treat `L2AssetTracker` revert as a fatal block-halting error.** Instead, log the failure and continue block processing (accepting that the asset tracker's accounting may be temporarily inconsistent, which can be reconciled later).
2. **Alternatively**, implement a retry/skip mechanism for the `notify_l2_asset_tracker` call, similar to how the bootloader handles other non-fatal errors.
3. **At minimum**, ensure that `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` can never revert under any reachable input (e.g., by making it infallible by design — using unchecked arithmetic or try/catch internally).

---

### Proof of Concept

**Trigger path:**

1. `L2AssetTracker` enters a state where `handleFinalizeBaseTokenBridgingOnL2` reverts (e.g., via upgrade, bug, or state manipulation).
2. An attacker (or any user) submits an L1→L2 priority transaction with `to_mint > gas_limit * gas_price` (non-zero `to_transfer`).
3. The bootloader calls `execute_l1_transaction_and_notify_result` → `mint_base_token` → `notify_l2_asset_tracker`.
4. `run_single_interaction` executes `handleFinalizeBaseTokenBridgingOnL2` on `L2AssetTracker`; the call reverts.
5. `asset_tracker_result.failed()` returns `true`.
6. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
7. This error propagates as `BootloaderSubsystemError::LeafDefect` — not matched by the `FatalRuntimeError` handler at line 221 — so it is returned from `execute_l1_transaction_and_notify_result` and then re-returned from `process_l1_transaction` at line 238.
8. Block processing halts. All subsequent L1→L2 priority transactions in the queue are permanently blocked. [10](#0-9) [11](#0-10)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
}

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
}
