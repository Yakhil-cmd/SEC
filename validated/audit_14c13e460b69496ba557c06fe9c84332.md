### Title
Off-by-One in `SliceVec::resize` Causes Valid EVM Heap Expansion to Fail with OutOfGas - (File: `zk_ee/src/memory/slice_vec.rs`)

### Summary
`SliceVec::resize` uses a strict `>=` boundary check instead of `>`, preventing the EVM heap from ever being resized to exactly its backing memory capacity. When the requested heap size equals the backing slice length, the resize returns `Err(())`, which `resize_heap_implementation` maps to `OutOfGas`. A valid EVM execution that requires exactly that many bytes of heap is incorrectly rejected.

### Finding Description
In `zk_ee/src/memory/slice_vec.rs`, the `resize` method guards with:

```rust
if new_length >= self.memory.len() {
    return Err(());
}
``` [1](#0-0) 

The condition should be `>`, not `>=`. As written, `new_length == self.memory.len()` is rejected even though the backing slice has exactly that many slots. The maximum reachable length is `self.memory.len() - 1`.

This error propagates directly into the EVM heap expansion path in `evm_interpreter/src/utils.rs`:

```rust
heap.resize(new_heap_size, 0)
    .map_err(|_| ExitCode::EvmError(EvmError::OutOfGas))?;
``` [2](#0-1) 

`new_heap_size` is computed as `max_offset.next_multiple_of(32)`. Because the EVM heap is always a multiple of 32 bytes, the off-by-one effectively reduces the usable heap ceiling by up to 32 bytes: if the backing slice length `M` is a multiple of 32, the largest reachable heap size is `M - 32` rather than `M`. [3](#0-2) 

The heap backing memory is a sub-slice of the total fixed memory region. As the call stack deepens, the remaining slice shrinks. An attacker who drives the call stack to a depth where the remaining heap backing memory equals the heap size they then request will hit this boundary.

### Impact Explanation
A valid EVM execution whose heap expansion lands exactly on the backing slice boundary is rejected with `OutOfGas` instead of succeeding. This is an EVM semantic mismatch: the execution environment diverges from the Ethereum specification. In a ZK proving context this also creates a forward/proving divergence — the forward runner rejects a transaction that a correct EVM would accept, making the block unprovable or forcing the operator to drop the transaction.

### Likelihood Explanation
The backing slice size is not attacker-controlled in isolation, but it is indirectly reachable: a crafted transaction can consume call-stack frames to reduce the remaining heap backing memory to a predictable value, then issue a memory-touching opcode (`MLOAD`, `CALLDATACOPY`, etc.) whose `offset + len` rounds up to exactly that value. The 32-byte granularity of heap growth makes the target a discrete, enumerable set of values.

### Recommendation
Change the boundary check from `>=` to `>`:

```rust
// Before (wrong):
if new_length >= self.memory.len() {
    return Err(());
}

// After (correct):
if new_length > self.memory.len() {
    return Err(());
}
``` [4](#0-3) 

This allows the `SliceVec` to be resized to exactly its backing capacity, matching the intended semantics described in the doc comment ("Resizes the `SliceVec` to the requested length").

### Proof of Concept
1. Deploy a contract that recursively calls itself to depth `D`, consuming call-stack frames and reducing the heap backing slice to `M` bytes (a multiple of 32).
2. In the innermost frame, execute `MLOAD` at offset `M - 32` (length 32), causing `resize_heap_implementation` to request `new_heap_size = M`.
3. `SliceVec::resize(M, 0)` evaluates `M >= M` → `true` → returns `Err(())`.
4. `resize_heap_implementation` maps this to `EvmError::OutOfGas`.
5. The transaction fails with OutOfGas even though the backing memory has exactly `M` bytes available and the EVM specification requires success. [5](#0-4) [3](#0-2)

impl<T: Clone> SliceVec<'_, T> {
    /// Resizes the `SliceVec` to the requested length.
    /// Adds copies of `padding` to the end if the size increases.
    pub fn resize(&mut self, new_length: usize, padding: T) -> Result<(), ()> {
        if new_length >= self.memory.len() {
            return Err(());
        }

        if new_length > self.length {
            for x in &mut self.memory[self.length..new_length] {
                x.write(padding.clone());
            }
        }
        if new_length < self.length {
            unsafe {
                assert_unchecked(self.length <= self.memory.len());
                ptr::drop_in_place(
                    &mut self.memory[new_length..self.length] as *mut [MaybeUninit<T>]
                        as *mut [T],
                );
            }
        }
        self.length = new_length;

        Ok(())
    }
```

    pub(crate) fn resize_heap_implementation<'a>(
        heap: &mut SliceVec<'a, u8>,
        gas: &mut Gas<S>,
        offset: usize,
        len: usize,
    ) -> Result<(), ExitCode> {
        let max_offset = offset.saturating_add(len);
        let new_heap_size = if max_offset > ((u32::MAX - 31) as usize) {
            return Err(ExitCode::EvmError(EvmError::MemoryLimitOOG));
        } else {
            max_offset.next_multiple_of(32)
        };
        let current_heap_size = heap.len();
        if new_heap_size > current_heap_size {
            gas.pay_for_memory_growth(current_heap_size, new_heap_size)?;

            heap.resize(new_heap_size, 0)
                .map_err(|_| ExitCode::EvmError(EvmError::OutOfGas))?;
        }

        Ok(())
    }
