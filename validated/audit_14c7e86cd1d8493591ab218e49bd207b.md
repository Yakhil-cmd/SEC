Audit Report

## Title
ckETH Withdrawal Permanently Stuck With No Reimbursement When Gas Fee Exceeds Withdrawal Amount — (`rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary
When a ckETH withdrawal request is accepted, the user's ckETH is burned immediately. If the Ethereum gas fee later spikes above the withdrawal amount, `create_transaction` returns `InsufficientTransactionFee` and the request is rescheduled indefinitely with no reimbursement path. The `maybe_reimburse` mechanism is only populated on the success path, never on the `InsufficientTransactionFee` error path, leaving the user's burned ckETH permanently unrecoverable if gas fees do not return below the withdrawal amount.

## Finding Description
In `create_transactions_batch` (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 281–291), the `InsufficientTransactionFee` error arm calls only `reschedule_withdrawal_request(request)` and logs a message. No reimbursement entry is created. [1](#0-0) 

The only place `maybe_reimburse.insert(withdrawal_id)` is called is inside `record_created_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, line 552), which is reached exclusively via the `Ok(transaction)` branch — never via the error branch. [2](#0-1) 

The root cause is in `create_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 1125–1133): when `request.withdrawal_amount.checked_sub(max_transaction_fee)` returns `None`, the function returns `InsufficientTransactionFee` with no side effects. [3](#0-2) 

The entry-point guard in `withdraw_eth` (`rs/ethereum/cketh/minter/src/main.rs`, lines 291–296) only rejects amounts below the static `cketh_minimum_withdrawal_amount`; it does not check the dynamic Ethereum gas fee estimate, which is only computed later at transaction-creation time. [4](#0-3) 

The `panic!` in `endpoints.rs` (lines 241–243) confirms the developers assumed the ledger fee is always lower than the withdrawal amount, but this assumption applies only to the ICRC ledger transfer fee — not to the Ethereum network gas fee, which is orders of magnitude larger and highly volatile. [5](#0-4) 

There is no staleness deadline, retry counter, or timeout on `EthWithdrawalRequest` that would trigger reimbursement after repeated `InsufficientTransactionFee` failures.

## Impact Explanation
This is a permanent, irreversible loss of user funds: ckETH is burned (supply decreases) with no corresponding ETH transfer and no refund mechanism. This is a ledger conservation violation for an in-scope chain-key token. The impact matches **High ($2,000–$10,000): Significant Chain Fusion / ck-token / ledger security impact with concrete user or protocol harm**. At scale (many users affected during a gas spike), the aggregate loss could reach the Critical threshold.

## Likelihood Explanation
Any unprivileged user calling `withdraw_eth` with an amount near `cketh_minimum_withdrawal_amount` is at risk. No special privileges, oracle manipulation, or consensus attack is required — only a post-burn Ethereum gas spike. The recent reduction of the minimum from 0.03 ETH to 0.005 ETH (proposal 139665, confirmed in `rs/ethereum/cketh/mainnet/minter_upgrade_2026_05_29.md`, line 21) significantly narrows the safety margin: at 300 gwei base fee, a standard ETH transfer costs `21,000 × 300e9 = 6.3e15 wei = 0.0063 ETH`, which already exceeds the new 0.005 ETH minimum. [6](#0-5) 

Gas spikes of this magnitude are historically common and require no attacker action.

## Recommendation
In the `InsufficientTransactionFee` error arm of `create_transactions_batch`, after a configurable number of reschedule attempts or a staleness deadline, add the withdrawal ID to `maybe_reimburse` and create a `ReimbursementRequest` (deducting a small penalty fee), mirroring the pattern used for failed ERC-20 withdrawals. Alternatively, add a `rescheduled_count` or `created_at` timestamp field to `EthWithdrawalRequest` and reimburse any request that has exceeded the deadline, regardless of current gas fees.

## Proof of Concept
1. Ethereum gas fees are low; user calls `withdraw_eth` with `amount = 5_000_000_000_000_000 wei` (the new minimum).
2. ckETH is burned; `EthWithdrawalRequest` is queued.
3. Ethereum gas spikes to 300 gwei: `max_transaction_fee = 21_000 × 300e9 = 6_300_000_000_000_000 wei > 5_000_000_000_000_000 wei`.
4. `create_transactions_batch` calls `create_transaction`; `checked_sub` returns `None`; `InsufficientTransactionFee` is returned.
5. The error arm at `withdraw.rs:290` calls `reschedule_withdrawal_request(request)`. `maybe_reimburse` is never updated.
6. Steps 4–5 repeat on every timer tick indefinitely. The user's ckETH is gone; no ETH is ever sent; no refund is ever issued.

A deterministic integration test can reproduce this by: (a) setting `cketh_minimum_withdrawal_amount = 5_000_000_000_000_000`, (b) submitting a withdrawal at that amount, (c) injecting a `GasFeeEstimate` with `max_fee_per_gas` high enough to make `max_transaction_fee > withdrawal_amount`, (d) running the timer, and (e) asserting that the request remains in the withdrawal queue forever with no reimbursement event emitted and no `maybe_reimburse` entry created.

With current Ethereum fees of ~$0.10–$1, a $10 minimum still preserves the ~10× safety margin even after several fee bumps.
