### Title
`exit_to_near_precompile_callback` Reverts When Contract Is Paused, Causing Permanent Loss of User Funds During wNEAR Unwrap - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `exit_to_near_precompile_callback` function enforces `require_running` (i.e., the contract must not be paused) at the start of its execution. This callback is the second receipt in a two-receipt chain initiated by the `ExitToNear` precompile. If the Aurora Engine contract is paused between the execution of the base promise (e.g., `near_withdraw`) and the callback receipt, the callback will revert with `ERR_PAUSED`. Because NEAR Protocol does not revert already-executed receipts, the user's wNEAR ERC-20 tokens are permanently burned while the unwrapped NEAR tokens remain permanently stuck in Aurora's account.

---

### Finding Description

The `ExitToNear` precompile supports a wNEAR-unwrap flow. When a user burns their wNEAR ERC-20 tokens and calls the precompile with the `:unwrap` suffix, the following two-receipt chain is scheduled:

1. **Base promise**: `near_withdraw` on the wNEAR NEP-141 contract — unwraps wNEAR into raw NEAR, crediting Aurora's account.
2. **Callback**: `exit_to_near_precompile_callback` on the Aurora Engine — transfers the unwrapped NEAR from Aurora's account to the user's NEAR account.

The callback is attached unconditionally for the wNEAR unwrap case because `transfer_near_args` is always `Some(...)`: [1](#0-0) 

The callback function begins with a `require_running` check: [2](#0-1) 

`require_running` returns `Err(ERR_PAUSED)` if `state.is_paused` is true: [3](#0-2) 

The contract can be paused by the owner at any time via `pause_contract`: [4](#0-3) 

In NEAR Protocol, receipts are processed in separate blocks. The base promise (`near_withdraw`) executes in one block, and the callback executes in a subsequent block. If the owner pauses the contract between these two blocks, the callback receipt will fail with `ERR_PAUSED`. NEAR does not revert already-executed receipts, so:

- The wNEAR ERC-20 tokens are permanently burned (EVM state change from the initial `submit` call is irreversible).
- The unwrapped NEAR tokens are credited to Aurora's account (the `near_withdraw` receipt is irreversible).
- The callback fails → the NEAR tokens are never transferred to the user.
- There is no retry mechanism for failed NEAR callbacks.

The same issue applies to the `error_refund` path: if the base `ft_transfer` fails and the contract is paused before the callback runs, the refund (re-minting burned ERC-20 tokens) is never executed. [5](#0-4) 

The `refund_on_error` function re-mints burned ERC-20 tokens or refunds ETH: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

For the wNEAR unwrap path (no feature flag required):
- User's wNEAR ERC-20 tokens are permanently burned.
- Unwrapped NEAR tokens are permanently stuck in Aurora's account with no recovery path.

For the `error_refund` path:
- User's ERC-20 tokens are permanently burned with no refund.
- User's ETH is permanently transferred to the `exit_to_near` precompile address with no refund.

There is no admin recovery function in the codebase to rescue stuck funds.

---

### Likelihood Explanation

The owner must pause the contract while a wNEAR unwrap (or ERC-20 exit with `error_refund`) is in-flight — specifically between the base promise receipt and the callback receipt. This is a realistic scenario during emergency pauses, which are precisely the situations where the owner is most likely to act quickly. The window is one NEAR block (~1 second), but given that multiple users may have in-flight exits at any time, the probability of at least one user being affected during an emergency pause is non-trivial.

---

### Recommendation

Remove the `require_running` check from `exit_to_near_precompile_callback`. This callback is a private self-call (enforced by `env.assert_private_call()`) that can only be triggered as the result of a previously scheduled promise chain. It should always be allowed to complete regardless of the contract's pause state, analogous to the mitigation recommended in the original report. The pause check is appropriate for user-initiated entry points, not for internal cleanup/refund callbacks. [2](#0-1) 

---

### Proof of Concept

1. User calls `submit` with an EVM transaction that burns wNEAR ERC-20 tokens and calls the `ExitToNear` precompile with `:unwrap` suffix.
2. Aurora Engine schedules a two-receipt chain: `near_withdraw` (base) → `exit_to_near_precompile_callback` (callback).
3. `near_withdraw` executes in block N+1: NEAR is unwrapped and credited to Aurora's account. wNEAR ERC-20 tokens are already burned in the EVM.
4. Owner calls `pause_contract` in block N+1 (emergency pause).
5. `exit_to_near_precompile_callback` executes in block N+2: `require_running` returns `ERR_PAUSED`, callback reverts.
6. NEAR tokens remain in Aurora's account. User's wNEAR ERC-20 tokens are permanently burned. Funds are permanently lost.

        let mut state = state::get_state(&io)?;
