Audit Report

## Title
Missing `is_canonical()` Check Allows Identity-Element TLS Certificate Registration and Node Impersonation at Boundary Nodes — (`rs/crypto/node_key_validation/tls_cert_validation/src/lib.rs`)

## Summary

`verify_ed25519_public_key` checks only `is_torsion_free()`, not `is_canonical()`. The three non-canonical encodings of the Ed25519 identity element are torsion-free (confirmed by the codebase's own tests), so they pass the public-key check and a crafted self-signed TLS certificate carrying one of these encodings is accepted by `ValidTlsCertificate::try_from` and stored in the registry. Because the boundary node's `verify_server_cert` then uses `ic_ed25519::PublicKey::verify_signature` (ZIP215) with the registry public key to authenticate the presented certificate, and the identity element degenerates that equation to `8·[S]B == 8·R`, an attacker can present any certificate with any public key they control, attach a trivially forged self-signature, and complete the TLS handshake — fully impersonating the registered node at boundary nodes.

## Finding Description

**Root cause — missing canonicality check:**

`verify_ed25519_public_key` in `rs/crypto/node_key_validation/tls_cert_validation/src/lib.rs` (lines 220–229) only calls `is_torsion_free()`:

```rust
fn verify_ed25519_public_key(
    public_key: &ic_ed25519::PublicKey,
) -> Result<(), TlsCertValidationError> {
    if !public_key.is_torsion_free() {
        return Err(invalid_tls_certificate_error(
            "public key verification failed",
        ));
    }
    Ok(())
}
```

`is_torsion_free` and `is_canonical` are explicitly documented as separate properties in `packages/ic-ed25519/src/lib.rs` (lines 546–558). The test `public_key_accepts_but_can_detect_non_canonical_keys` at `packages/ic-ed25519/tests/tests.rs` (lines 499–515) confirms that all three non-canonical encodings of the identity element satisfy `is_torsion_free() == true` and `is_canonical() == false`.

**Self-signature forgery under ZIP215:**

`verify_ed25519_signature` calls `public_key.verify_signature(msg, sig)`, which implements ZIP215 (`packages/ic-ed25519/src/lib.rs`, lines 709–727):

```
recomputed_r = [S]B + [k]·(-A)
check: (recomputed_r − R)·8 == identity
```

With `A = identity`:
- `-A = identity`
- `recomputed_r = [S]B + [k]·identity = [S]B`
- Check reduces to: `([S]B − R)·8 == identity`, satisfied for any message by choosing any `S` and setting `R = [S]B`.

The attacker picks `S=1`, `R=B` (base point), and the signature `(R, S)` passes `verify_ed25519_signature` for any TBSCertificate content.

**Registration path:**

`valid_keys_from_payload` → `ValidNodePublicKeys::try_from` → `ValidTlsCertificate::try_from` (`rs/registry/canister/src/mutations/node_management/do_add_node.rs`, lines 340–400). The crafted certificate passes all checks and is stored in the registry via `make_add_node_registry_mutations`.

**Boundary node exploitation:**

`verify_server_cert` in `rs/boundary_node/ic_boundary/src/tls_verify.rs` (lines 107–129):
1. Extracts the public key from the **registry** certificate (the identity element).
2. Calls `node_tls_pubkey_from_registry.verify_signature(provided_cert.tbs_certificate.as_ref(), &provided_cert_sig)` — ZIP215 with the identity element.
3. Does **not** check that the provided certificate's public key matches the registry certificate's public key.

The attacker presents a certificate embedding their own Ed25519 public key, attaches the trivially forged self-signature, and `verify_server_cert` returns `Ok(ServerCertVerified::assertion())`. The TLS 1.3 CertificateVerify step then uses the attacker's own public key (from the presented certificate), which the attacker can sign legitimately. The handshake completes, and the boundary node treats the attacker as the registered node.

**Secondary gap — node signing key:**

The same missing `is_canonical()` check exists in `ValidNodeSigningPublicKey::try_from` at `rs/crypto/node_key_validation/src/lib.rs` (lines 233–240).

## Impact Explanation

A malicious registered node operator can register a node whose TLS certificate carries an identity-element public key. Any third party can then impersonate that node in TLS connections to boundary nodes — presenting a certificate with an attacker-controlled public key, forging the self-signature deterministically, and completing the handshake. This constitutes unauthorized impersonation of a registered IC node at boundary node infrastructure, enabling traffic interception and forged responses to boundary node requests. This matches the allowed High impact: "Significant boundary/API infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The prerequisite is being a registered node operator (requires governance approval), which is a meaningful but not prohibitive constraint — node operators are a broad class of IC participants. Once registered, the cryptographic forgery is fully deterministic, requires no brute force, and is locally testable. The three non-canonical identity-element encodings are publicly documented. The attack is repeatable and does not degrade over time.

## Recommendation

Add an `is_canonical()` check to `verify_ed25519_public_key` in `rs/crypto/node_key_validation/tls_cert_validation/src/lib.rs`:

```rust
fn verify_ed25519_public_key(
    public_key: &ic_ed25519::PublicKey,
) -> Result<(), TlsCertValidationError> {
    if !public_key.is_torsion_free() || !public_key.is_canonical() {
        return Err(invalid_tls_certificate_error(
            "public key verification failed",
        ));
    }
    Ok(())
}
```

Apply the same fix to `ValidNodeSigningPublicKey::try_from` in `rs/crypto/node_key_validation/src/lib.rs` (line 236).

## Proof of Concept

```rust
// 1. Build a TBSCertificate with subject_public_key =
//    0x0100000000000000000000000000000000000000000000000000000000000080
//    (non-canonical identity element, confirmed torsion-free by existing test)
//
// 2. Forge self-signature: S=1, R=B (base point).
//    ZIP215 check: ([1]B − B)·8 = identity ✓  (independent of TBSCertificate content)
//
// 3. Call ValidTlsCertificate::try_from((cert, node_id, time))
//    → assert Ok(_)   // certificate accepted and stored in registry
//
// 4. Simulate boundary node verify_server_cert:
//    - registry_pubkey = identity element
//    - presented_cert  = certificate with attacker's own Ed25519 public key
//    - presented_sig   = forged (R=B, S=1)
//    - registry_pubkey.verify_signature(presented_cert.tbs, presented_sig)
//    → assert Ok(())  // ZIP215 degeneracy confirmed
//
// 5. Attacker signs TLS 1.3 CertificateVerify with their own private key
//    → handshake completes; boundary node accepts attacker as registered node
```

The ZIP215 test vectors already shipped in `packages/ic-ed25519/tests/data/zip215.txt` confirm that the canonical identity element (`0x0000...0080`) accepts multiple distinct signatures for the same message, validating the degenerate verification equation.
