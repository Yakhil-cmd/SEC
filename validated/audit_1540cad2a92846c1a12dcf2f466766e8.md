Audit Report

## Title
Unprivileged Caller Can Frontrun Victim's `update_balance` to Cause Persistent `AlreadyProcessing` DoS - (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary

The `update_balance` endpoint in the ckBTC (and ckDOGE) minter accepts an optional `owner` principal that any non-anonymous caller can set to an arbitrary victim principal. The `balance_update_guard` is keyed on the resolved account (the victim's), not the caller's. Because the guard is held across multiple async inter-canister calls, an attacker can continuously call `update_balance(owner = victim)` to keep the guard occupied, causing every legitimate call from the victim to return `AlreadyProcessing`. The victim's on-chain BTC deposit is confirmed but ckBTC minting is blocked for as long as the attacker sustains the attack.

## Finding Description

In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` L164–168, the target account is resolved from the caller-supplied `args.owner` without any authorization check binding the owner to the caller:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),  // attacker sets this to victim's principal
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;  // guard keyed on victim's account
```

The only caller restriction is `check_anonymous_caller()` in `rs/bitcoin/ckbtc/minter/src/main.rs` L117–121, which only blocks the anonymous principal — any authenticated principal may pass any `owner`. The comment at L156–160 confirms this is intentional to support third-party notification.

In `rs/bitcoin/ckbtc/minter/src/guard.rs` L45–59, `Guard::new` inserts the account into `state.update_balance_accounts` and returns `GuardError::AlreadyProcessing` if already present:

```rust
if accounts.contains(&account) {
    return Err(GuardError::AlreadyProcessing);
}
accounts.insert(account);
```

The guard is held across at least one async `get_utxos` call (L175–183) and potentially a second one (L225–236) when no UTXOs are found, plus up to `MAX_CHECK_TRANSACTION_RETRY = 10` async BTC checker calls per UTXO. The guard is only released when `_guard` is dropped at function exit (L63–66 of `guard.rs`).

The `AlreadyProcessing` error propagates directly to the victim caller via the `From<GuardError>` impl at L108–116 of `update_balance.rs`.

**Exploit flow:**
1. Attacker (any non-anonymous principal) calls `update_balance({ owner = opt principal "VICTIM"; subaccount = null })`.
2. Guard for victim's account is acquired; attacker's call enters async `get_utxos` round-trip.
3. Victim calls `update_balance({ owner = null; subaccount = null })` — resolves to same account — receives `Err(AlreadyProcessing)`.
4. Attacker's call completes and releases the guard.
5. Attacker immediately re-submits; IC message queuing allows pre-queuing to minimize the gap.
6. Victim's retries continue to fail.

The `MAX_CONCURRENT = 100` limit does not protect against a targeted single-account attack since only one attacker call per victim account is needed at any time.

## Impact Explanation

This is a targeted, sustained denial-of-service against the ckBTC (and ckDOGE) deposit/minting flow. A victim who has deposited BTC on-chain cannot mint ckBTC for as long as the attacker sustains the attack. The victim's funds are not stolen, but the minting service — a core function of the ckBTC financial integration — is rendered unavailable to the targeted user. This matches the **High** impact category: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm" and "Application/platform-level DoS... not based on raw volumetric DDoS."

## Likelihood Explanation

The attack requires only a valid non-anonymous IC principal (trivially obtained) and knowledge of the victim's principal ID (publicly derivable). No BTC, ckBTC, or special permissions are needed. The attacker pays only standard IC ingress fees per call, while the minter canister bears the cost of Bitcoin canister calls. The attack is trivially scriptable and can be sustained indefinitely against a specific target. The same vulnerability exists in the ckDOGE minter via the shared `update_balance` implementation.

## Recommendation

Restrict the `owner` field so that an unprivileged ingress caller may only trigger balance updates for their own account. The simplest fix is to require `args.owner == None || args.owner == Some(caller)` for ingress callers, mirroring the approach in `retrieve_btc_with_approval` (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` L248–262) which always uses `ic_cdk::api::msg_caller()` directly without an overridable owner field.

If third-party notification by helper canisters is a required feature, the guard should be keyed on the **caller** rather than the resolved owner, so an attacker's guard does not block the victim's own call. Alternatively, restrict third-party `owner` overrides to a whitelist of trusted canister principals.

## Proof of Concept

```
// Setup: victim has deposited BTC on-chain, awaiting confirmation.

// Attacker loop (any non-anonymous principal, tight loop):
loop {
    update_balance({ owner = opt principal "VICTIM_PRINCIPAL_ID"; subaccount = null })
    // Each call acquires balance_update_guard for victim's account,
    // holds it across async get_utxos() Bitcoin canister call,
    // then releases. Immediately re-submit.
}

// Victim's legitimate call (concurrent):
update_balance({ owner = null; subaccount = null })
// → Returns: Err(AlreadyProcessing)
// Victim cannot mint ckBTC for as long as attacker continues.
```

A deterministic integration test can be written using `PocketIC` or the `StateMachine` test framework: install the minter, submit an attacker `update_balance` call with `owner = Some(victim)` that will block on a mocked slow Bitcoin canister response, then concurrently submit the victim's own `update_balance` call and assert it returns `Err(UpdateBalanceError::AlreadyProcessing)`.
