Audit Report

## Title
Any Deployed SNS Swap Canister Can Settle Another SNS's Neurons' Fund Participation With Fabricated Data - (`File: rs/nns/governance/src/governance.rs`)

## Summary

`settle_neurons_fund_participation` in NNS Governance authorizes callers by checking only that the caller is *any* deployed SNS swap canister, never that it is the swap canister specifically associated with the `nns_proposal_id` in the request. A legitimately deployed SNS swap canister can therefore call this function targeting a different SNS's proposal, supplying a fabricated `Committed` result with an attacker-controlled `sns_governance_canister_id` as the ICP minting destination and an inflated `total_direct_participation_icp_e8s`, causing the NNS to mint Neurons' Fund ICP to an arbitrary canister and permanently corrupt the target proposal's settlement state.

## Finding Description

`settle_neurons_fund_participation` (line 6980) accepts a `caller` and a `request` containing `nns_proposal_id` and a `SwapResult`. The authorization block at lines 7018–7038 converts the caller to a `CanisterId` and calls `is_canister_id_valid_swap_canister_id`. That helper (lines 8191–8226) queries SNS-W's `list_deployed_snses` and returns `Ok(())` if the caller appears as *any* swap canister in the global list:

```rust
let is_swap = list_deployed_snses_response
    .instances
    .iter()
    .any(|sns| sns.swap_canister_id == Some(target_canister_id.into()));
```

The code itself acknowledges the gap with the comment at line 7018: *"Note that a Swap could settle each other's participation."* No subsequent check binds the caller to the specific proposal identified by `nns_proposal_id`.

After passing this check, the function extracts `total_direct_participation_icp_e8s` and `sns_governance_canister_id` directly from the caller-supplied `request.swap_result` (lines 7198–7218 and 7249–7254). The `sns_governance_canister_id` is passed verbatim to `mint_to_sns_governance` (lines 7266–7273) as the ICP minting destination. The lifecycle of the target proposal is then set to `Committed` or `Aborted` (line 7235), and the idempotency guard (line 7166) prevents any future legitimate call from correcting the corrupted state.

## Impact Explanation

A malicious SNS swap canister (Swap A) can call `settle_neurons_fund_participation` with Swap B's `nns_proposal_id` and a fabricated `Committed { sns_governance_canister_id: attacker_canister, total_direct_participation_icp_e8s: <inflated>, ... }`. The NNS will mint Neurons' Fund ICP to the attacker-controlled canister and permanently mark Swap B's proposal as settled. Alternatively, submitting a fabricated `Aborted` result causes the NNS to refund all reserved Neurons' Fund maturity before Swap B's actual swap concludes, permanently preventing Neurons' Fund participation in Swap B. This constitutes unauthorized minting of ICP to attacker-controlled accounts and/or permanent loss of Neurons' Fund participation — matching the High impact class: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."* Depending on the Neurons' Fund reserves allocated to the target SNS, the stolen amount could reach the Critical threshold (>$1M).

## Likelihood Explanation

The precondition is obtaining a legitimately deployed SNS swap canister, which requires an NNS governance proposal to pass. This is a meaningful constraint but not a governance-majority attack — the NNS community routinely approves SNS deployments for projects that appear legitimate, without auditing whether the resulting swap canister will be used to attack other SNS deployments. Once deployed, the attack requires no privileged key, no threshold corruption, and no external dependency. The attack window is any time between Swap B's proposal being created and its first successful settlement call. The code comment explicitly acknowledges the gap, confirming the developers were aware but did not close it.

## Recommendation

After the `is_canister_id_valid_swap_canister_id` check passes, retrieve the swap canister ID recorded in the proposal data for `nns_proposal_id` and assert the caller matches it:

```rust
let expected_swap_canister_id = proposal_data
    .swap_canister_id()
    .ok_or_else(|| GovernanceError::new_with_message(
        ErrorType::NotAuthorized,
        "Proposal has no associated swap canister ID".to_string(),
    ))?;
if target_canister_id != expected_swap_canister_id {
    return Err(GovernanceError::new_with_message(
        ErrorType::NotAuthorized,
        format!(
            "Caller {} is not the swap canister for proposal {:?}. \
             Expected {}.",
            caller, request.nns_proposal_id, expected_swap_canister_id,
        ),
    ));
}
```

Additionally, the `sns_governance_canister_id` supplied in the `Committed` variant should be validated against the value recorded in the proposal data rather than used as-is as the minting destination.

## Proof of Concept

1. Deploy SNS-A through a legitimate NNS `CreateServiceNervousSystem` proposal; receive `swap_canister_a`.
2. Identify an unsettled SNS-B proposal (`nns_proposal_id = B`) whose Neurons' Fund participation has been reserved.
3. From `swap_canister_a`, call `settle_neurons_fund_participation` on NNS Governance with:
   - `nns_proposal_id = B`
   - `result = Committed { sns_governance_canister_id: attacker_canister, total_direct_participation_icp_e8s: u64::MAX, total_neurons_fund_participation_icp_e8s: 0 }`
4. `is_canister_id_valid_swap_canister_id` passes because `swap_canister_a` is a valid swap canister.
5. NNS Governance mints Neurons' Fund ICP to `attacker_canister` and sets proposal B's lifecycle to `Committed`.
6. Any subsequent call by `swap_canister_b` hits the idempotency guard (line 7166) and returns the previously corrupted result without performing any new minting.

A deterministic integration test using PocketIC can reproduce this by deploying two SNS instances, calling `settle_neurons_fund_participation` from the first swap canister with the second proposal's ID, and asserting that ICP was minted to the attacker-controlled destination and that the second swap canister's subsequent call returns the corrupted snapshot.
