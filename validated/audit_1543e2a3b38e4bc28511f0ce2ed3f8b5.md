### Title
Missing Zero-Modulus Guard in `addmod` Opcode Causes Panic Instead of EVM-Specified Zero Return - (File: `evm_interpreter/src/instructions/arithmetic.rs`)

---

### Summary

The `addmod` opcode handler calls `ruint::aliases::U256::add_mod` without first checking whether the modulus operand is zero. The EVM Yellow Paper mandates that `ADDMOD(a, b, 0) = 0`. Instead, the unguarded call panics (division by zero inside ruint), causing an uncontrolled abort rather than a graceful result. The sibling `mulmod` opcode correctly guards against this case; `addmod` does not.

---

### Finding Description

In `evm_interpreter/src/instructions/arithmetic.rs`, the `addmod` handler is:

```rust
pub fn addmod(&mut self) -> InstructionResult {
    self.gas
        .spend_gas_and_native(gas_constants::MID, ADDMOD_NATIVE_COST)?;
    let ((op1, op2), op3) = self.stack.pop_2_and_peek_mut()?;

    *op3 = U256::add_mod(*op1, *op2, *op3);   // ← no zero-check on op3
    Ok(())
}
``` [1](#0-0) 

`U256` here is `ruint::aliases::U256`. Ruint's `add_mod` internally performs a modular reduction (division) and panics when the modulus is zero.

Compare with `mulmod`, which delegates to a local wrapper that explicitly returns `U256::ZERO` when the modulus is zero:

```rust
pub fn mul_mod(this: &U256, other: &U256, mut modulus: U256) -> U256 {
    if modulus == U256::ZERO {
        return U256::ZERO;
    }
    ...
}
``` [2](#0-1) 

The same asymmetry is visible in the custom `U256` implementations. The risc_v `add_mod` calls `reduce_mod` which does guard for zero:

```rust
pub fn reduce_mod(&mut self, modulus: &Self) {
    if modulus.is_zero() {
        Self::write_zero(self);
        return;
    }
    ...
}
``` [3](#0-2) 

But the `addmod` opcode bypasses these custom wrappers entirely and calls `ruint::aliases::U256::add_mod` directly, which has no such guard.

The naive `add_mod` wrapper also delegates straight to ruint without a zero check:

```rust
pub fn add_mod(a: &mut Self, b: &mut Self, modulus_or_result: &mut Self) {
    modulus_or_result.0 = ruint::aliases::U256::add_mod(a.0, b.0, modulus_or_result.0);
}
``` [4](#0-3) 

---

### Impact Explanation

**Classification:** EVM semantic mismatch + valid-execution unprovability / forward-execution abort.

1. **EVM semantic mismatch:** The Yellow Paper specifies `ADDMOD(a, b, 0) = 0`. Any contract that relies on this (e.g., uses a runtime-computed modulus that may be zero as a sentinel) will receive a panic/abort instead of the correct result `0`. This breaks EVM equivalence.

2. **Uncontrolled abort in no_std context:** ZKsync OS runs in a no_std environment. A Rust panic in the EVM interpreter does not produce a graceful EVM revert; it triggers an unrecoverable trap. This means a transaction that is *valid* per the EVM spec (ADDMOD with zero modulus should succeed and return 0) instead causes the execution to abort, making it unprovable or causing a divergence between forward execution and the prover.

3. **Denial of valid execution:** Any contract that intentionally or incidentally passes a zero modulus to `ADDMOD` will have its transaction aborted rather than executed correctly, constituting a loss of liveness for affected transactions.

---

### Likelihood Explanation

**Medium.** Any unprivileged user can deploy or call a contract that executes `ADDMOD` with a zero modulus. This requires no special privileges, no oracle manipulation, and no governance access. The trigger is a single EVM opcode with attacker-controlled stack values. Contracts that compute a modulus dynamically (e.g., from user input or a storage slot that can be zero) are directly affected.

---

### Recommendation

Add a zero-modulus guard in `addmod`, mirroring the pattern already used in `mulmod`:

```rust
pub fn addmod(&mut self) -> InstructionResult {
    self.gas
        .spend_gas_and_native(gas_constants::MID, ADDMOD_NATIVE_COST)?;
    let ((op1, op2), op3) = self.stack.pop_2_and_peek_mut()?;

    if op3.is_zero() {
        // EVM spec: ADDMOD(a, b, 0) = 0
        // op3 is already zero, nothing to do
        return Ok(());
    }
    *op3 = U256::add_mod(*op1, *op2, *op3);
    Ok(())
}
``` [1](#0-0)

### Proof of Concept

1. Deploy a contract containing the bytecode sequence:
   ```
   PUSH1 0x00   // modulus = 0
   PUSH1 0x01   // b = 1
   PUSH1 0x01   // a = 1
   ADDMOD       // should return 0 per EVM spec; instead panics
   ```
2. Submit a transaction calling this contract from any EOA.
3. The EVM interpreter reaches `arithmetic.rs` line 73, calls `ruint::aliases::U256::add_mod(1, 1, 0)`, which internally attempts division by zero and panics.
4. In the no_std ZKsync OS environment, the panic aborts execution rather than producing a graceful EVM revert with result `0`, causing a forward/proving divergence. [1](#0-0) [5](#0-4)

pub fn mul_mod(this: &U256, other: &U256, mut modulus: U256) -> U256 {
    if modulus == U256::ZERO {
        return U256::ZERO;
    }
    // Compute full product.
    // The challenge here is that Rust doesn't allow us to create a
    // `Uint<2 * BITS, _>` for the intermediate result. Otherwise,
    // we could just use a `widening_mul`. So instead we allocate from heap.
    // Alternatively we could use `alloca`, but that is blocked on
    // See <https://github.com/rust-lang/rust/issues/48055>
    let mut product = [0u64; 8];
    let overflow = ruint::algorithms::addmul(&mut product, this.as_limbs(), other.as_limbs());
    debug_assert!(!overflow);

    // Compute modulus using `div_rem`.
    // This stores the remainder in the divisor, `modulus`.
    unsafe {
        ruint::algorithms::div(&mut product, modulus.as_limbs_mut());
    }

    modulus
}
```
