Audit Report

## Title
ckERC20 Deposit Helper Contracts Emit Caller-Supplied Amount Instead of Actual Received Amount, Enabling Over-Minting for Fee-on-Transfer Tokens - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol, rs/ethereum/cketh/minter/ERC20DepositHelper.sol)

## Summary
Both ckERC20 helper contracts (`CkDeposit.depositErc20` and `CkErc20Deposit.deposit`) emit the caller-supplied `amount` parameter in their deposit events rather than the actual token balance delta received by the minter address. The IC ckETH minter canister scrapes these events and mints ckERC20 tokens equal to the event-reported `amount` with no on-chain balance verification. For any ERC-20 token that charges a transfer fee, the minter receives fewer tokens than the event records, causing the IC minter to mint unbacked ckERC20 supply and breaking the 1:1 peg.

## Finding Description
**Root cause — Solidity contracts:**

In `DepositHelperWithSubaccount.sol` (`CkDeposit.depositErc20`, L511–532), the contract calls `safeTransferFrom(msg.sender, minterAddress, amount)` and then emits `ReceivedEthOrErc20(..., amount, ...)` using the caller-supplied `amount` directly, with no balance-before/balance-after measurement. The same pattern exists in `ERC20DepositHelper.sol` (`CkErc20Deposit.deposit`, L498–503).

**Root cause — Rust minter:**

`parser.rs` (`ReceivedErc20LogParser::parse_log`, L86–97) reads the event data field directly as `Erc20Value::from_be_bytes(value_bytes)` — this is the emitted `amount`, not a verified balance delta. `deposit.rs` (`mint()`, L73–81) then calls `client.transfer(TransferArg { amount: event.value(), ... })` to mint exactly that value to the beneficiary. No step in `scrape_logs`, `register_deposit_events`, or `mint` queries the minter's actual ERC-20 balance before and after to compute the real received amount.

**Exploit flow:**
1. A fee-on-transfer ERC-20 token is whitelisted as a supported ckERC20 token (via NNS proposal, or an already-whitelisted upgradeable token re-enables its fee).
2. An unprivileged user calls `depositErc20(tokenAddr, N, principal, subaccount)`.
3. The helper calls `safeTransferFrom(user, minterAddress, N)`. Due to the transfer fee, minter receives `N - fee`.
4. The helper emits `ReceivedEthOrErc20(..., N, ...)`.
5. The IC minter scrapes the log, parses `value = N`, and mints `N` ckERC20 to the user's IC account.
6. The user holds `N` ckERC20 backed by only `N - fee` ERC-20 tokens.
7. Repeated deposits accumulate unbacked ckERC20 supply; the last withdrawers cannot redeem their tokens.

**Existing guards are insufficient:** The minter enforces a token whitelist (only scraping logs for supported ERC-20 contract addresses), but this whitelist check does not validate whether the token implements fee-on-transfer behavior. There is no balance reconciliation anywhere in the deposit pipeline.

## Impact Explanation
This is an illegal minting / protocol insolvency bug in the ckERC20 chain-fusion financial integration. Every deposit of a fee-on-transfer token causes the IC minter to mint more ckERC20 than the ERC-20 tokens it actually holds. Over time, total ckERC20 supply exceeds the minter's ERC-20 reserve. Users attempting to withdraw ckERC20 back to ERC-20 will find the minter unable to fulfill all requests — the last withdrawers receive nothing. This is a direct ledger conservation break and loss of funds for ckERC20 holders.

This matches the allowed impact: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm**, and potentially **Critical** if a high-value token (e.g., ckUSDT, whose Tether contract has a dormant fee mechanism) is involved at scale.

## Likelihood Explanation
The exploit requires a whitelisted token to have active fee-on-transfer behavior. Currently whitelisted tokens (USDC, USDT, LINK, PEPE, SHIB, UNI, WBTC, wstETH, XAUt, EURC, OCT) do not currently charge transfer fees. However:
- USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is a proxy-upgradeable contract with a dormant fee mechanism that Tether can re-enable unilaterally at any time.
- USDC is similarly upgradeable by Circle.
- A future NNS proposal could add a fee-on-transfer token without the community recognizing the protocol-level risk, since the NNS proposal process does not validate token transfer behavior.
- Once the precondition is met, the exploit requires no special privileges — any user can call `depositErc20`.

**Likelihood: Low-Medium** — requires a whitelisted token to activate fee-on-transfer, but the protocol provides no defense-in-depth.

## Recommendation
Both helper contracts should measure the actual balance delta and emit that as the deposit amount:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualAmount = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualAmount, principal, subaccount);
}
```

Apply the same pattern to `CkErc20Deposit.deposit` in `ERC20DepositHelper.sol`. This ensures the IC minter only mints ckERC20 equal to the tokens it actually received.

## Proof of Concept
**Deterministic local test plan:**

1. Deploy a mock ERC-20 token with a configurable transfer fee (e.g., 1% deducted on every `transferFrom`).
2. Register this token as a supported ckERC20 token in a local PocketIC/StateMachine test environment.
3. Call `depositErc20(mockToken, 1_000_000, principal, subaccount)` on the `CkDeposit` helper.
4. Observe: `safeTransferFrom` credits minter with `990_000` tokens; helper emits `ReceivedEthOrErc20(..., 1_000_000, ...)`.
5. Observe: IC minter scrapes the log, parses `value = 1_000_000`, mints `1_000_000` ckERC20 to the caller's IC account.
6. Assert: `mockToken.balanceOf(minterAddress) == 990_000` but ckERC20 ledger shows `1_000_000` minted — a `10_000` token discrepancy per deposit.
7. Repeat N times; assert total ckERC20 supply exceeds minter's ERC-20 reserve by `N * 10_000`.
8. Attempt withdrawal of full ckERC20 supply; assert the final `N * 10_000` ckERC20 cannot be redeemed.
