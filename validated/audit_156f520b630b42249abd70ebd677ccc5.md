### Title
Rounding Error in `native_per_pubdata` Calculation Causes Pubdata to Be Undercharged to Zero — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio is computed using floor (truncating) division. When `pubdata_price < native_price`, the result rounds down to `0`, making pubdata effectively free for every transaction in the block. Any unprivileged transaction sender can then produce pubdata up to the block limit without paying the associated L1 data-publication cost, causing the operator to bear those costs uncompensated.

---

### Finding Description

In `validation_impl.rs` line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs floor (truncating) integer division. When `pubdata_price < native_price` (e.g., `pubdata_price = 999`, `native_price = 1000`), the result is `0`.

This zero propagates through every pubdata-charging site in the transaction lifecycle:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // = 0 * intrinsic_pubdata = 0
   ```

2. **Execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)?;
   // = pubdata_bytes * 0 = 0
   ```

3. **Post-execution pubdata check** in `check_enough_resources_for_pubdata` always returns `true` because the required native is `0`.

The same `wrapping_div` pattern is replicated in the off-chain helper `api/src/helpers.rs` line 427:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

The inconsistency is notable: `native_per_gas` uses `div_ceil` (rounds **up**, protecting against undercharging), while `native_per_pubdata` uses `wrapping_div` (rounds **down**, causing undercharging):

```rust
// native_per_gas — rounds UP (line 135)
u256_try_to_u64(&gas_price.div_ceil(native_price))

// native_per_pubdata — rounds DOWN (line 142)
u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
```

---

### Impact Explanation

When `native_per_pubdata = 0`:

- Every transaction in the block pays **zero** native resource for pubdata, regardless of how many storage slots it writes.
- The operator must still publish that data to L1 and pay the associated L1 gas cost, but receives no compensation from the transaction fees.
- A transaction sender can perform the maximum number of SSTORE operations (paying only EVM gas) and fill the block's pubdata quota for free, causing the operator a direct, measurable financial loss proportional to the block's pubdata limit.
- The block-level pubdata limit (`system.get_pubdata_limit()`) is the only remaining guard, but it does not prevent the loss — it only caps its magnitude per block.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic:

- `pubdata_price` tracks L1 blob/calldata gas prices, which fluctuate with Ethereum network conditions.
- `native_price` is a constant set by the operator to reflect proving cost.
- During periods of low L1 gas prices (e.g., post-EIP-4844 blob fee drops), `pubdata_price` can easily fall below `native_price`.
- No special privilege is required: once the operator-set prices satisfy the condition, **any** unprivileged transaction sender can exploit it by submitting storage-heavy transactions.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. Alternatively, add an explicit guard: if `pubdata_price > 0` but `native_per_pubdata == 0`, reject the transaction or clamp `native_per_pubdata` to `1`.

---

### Proof of Concept

**Setup:** Operator sets `native_price = 1000`, `pubdata_price = 999` (a realistic scenario where L1 gas is slightly cheaper than proving cost).

**Calculation:** `native_per_pubdata = 999 / 1000 = 0` (floor division).

**Attack:**
1. Attacker submits a transaction calling a contract that performs 1000 SSTORE operations (each writing a new storage slot, producing ~66 bytes of pubdata each = ~66 000 bytes total).
2. During validation, `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0` — no pubdata cost deducted from native limit.
3. During execution, `get_resources_to_charge_for_pubdata` returns `(66000, 0_native)`.
4. `check_enough_resources_for_pubdata` returns `true` (0 ≤ remaining native).
5. Transaction succeeds. Attacker paid only EVM gas for SSTORE (~20 000 gas × 1000 = 20M gas). Operator must publish 66 KB to L1 at their own expense.
