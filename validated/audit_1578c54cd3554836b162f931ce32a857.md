### Title
Missing Storage-Collision Check in CREATE/CREATE2 Allows Deployment Over Accounts With Pre-existing Storage - (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter omits the EIP-684 / EIP-7610 requirement that a CREATE or CREATE2 must fail when the target address already has non-empty storage. The collision guard only checks `code_len != 0 || nonce != 0`, leaving the storage dimension entirely unchecked. An unprivileged attacker can pre-seed storage at any deterministically predictable CREATE2 address, then allow the legitimate deployer's CREATE2 to succeed — producing a deployed contract that silently inherits attacker-controlled storage state.

---

### Finding Description

In `before_executing_frame` inside `evm_interpreter/src/ee_trait_impl.rs`, the constructor collision check reads:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // … burn gas, emit CreateCollision, return false
}
```

The comment explicitly acknowledges the missing third condition. Per EIP-684 (enforced since Spurious Dragon) and EIP-7610 (Paris), a CREATE/CREATE2 must revert if the target address has **any** non-zero storage slot. ZKsync OS skips this check entirely.

The codebase's own test-index files confirm the gap: `create2collisionStorageParis.json` is disabled with `comment: We do not check storage for collision`, and the entire `eip7610_create_collision` test suite (`test_init_collision_create_opcode.json`, `test_init_collision_create_tx.json`) is disabled with `comment: We do not check the storage`.

**Attack path:**

1. Attacker computes the deterministic CREATE2 address for a victim factory contract (salt + initcode are observable from the mempool or known in advance).
2. Attacker sends a transaction that calls the target address and writes one or more storage slots (e.g., via a contract that does `SSTORE` at that address, or by deploying a temporary contract there and self-destructing to clear code/nonce while leaving storage).
3. The target address now has `code_len == 0`, `nonce == 0`, but non-zero storage.
4. The victim's CREATE2 passes the collision guard (only checks code and nonce) and the constructor executes successfully.
5. The newly deployed contract starts with attacker-controlled storage values that its constructor never wrote.

This is the direct analog of the external report: just as the DAO name check only verified `getENSAddress[name] == address(0)` without validating ownership, ZKsync OS's collision check only verifies `code_len == 0 && nonce == 0` without verifying storage emptiness.

---

### Impact Explanation

A deployed contract inheriting unexpected storage state can have its invariants broken from the first transaction. Concrete examples:

- An ERC-20 token deployed via CREATE2 whose `balances[attacker]` slot is pre-set to a large value, giving the attacker a free token balance.
- An ownership slot (`owner`) pre-set to the attacker's address, granting immediate administrative control.
- A reentrancy guard slot pre-set to `1` (locked), permanently bricking the contract.
- A proxy's implementation slot pre-set to a malicious address.

The state divergence also constitutes an **EVM semantic mismatch**: on a canonical EVM node the CREATE2 would revert (returning `0` to the caller), but on ZKsync OS it succeeds. Any cross-chain bridge, verifier, or application that relies on CREATE2 deployment determinism and EVM-equivalence guarantees is affected.

---

### Likelihood Explanation

CREATE2 is widely used for counterfactual deployments (account abstraction wallets, Uniswap pools, proxy factories). The target address is fully deterministic and publicly computable before deployment. An attacker only needs to:

1. Watch the mempool for a CREATE2 deployment transaction, or know the salt/initcode in advance (common for factory patterns).
2. Submit a storage-seeding transaction to the target address before the deployment lands.

No privileged access, leaked keys, or oracle manipulation is required. The attacker pays only normal gas costs. Likelihood is **high** for any protocol that uses CREATE2 factories with publicly known salts or initcode.

---

### Recommendation

Implement the missing storage-emptiness check in `before_executing_frame` in `evm_interpreter/src/ee_trait_impl.rs`. Before allowing the constructor to execute, query whether the target address has any non-zero storage slot. If it does, treat it as a collision: burn all gas and return `EvmError::CreateCollision`, consistent with EIP-684 / EIP-7610 semantics.

The comment in the code says "we cannot perform such a check for now" — the underlying system IO layer must expose a method to test whether an address has non-empty storage (e.g., by checking the storage root or iterating the first slot). This capability should be added to the `IOSubsystemExt` trait and called from the collision guard.

Enable the currently-disabled test cases (`create2collisionStorageParis.json`, `RevertInCreateInInitCreate2Paris.json`, `test_init_collision_create_opcode.json`, `test_init_collision_create_tx.json`) to verify the fix.

**Scenario:**

```
// Step 1: Attacker pre-seeds storage at the victim's future CREATE2 address
// target = keccak256(0xff ++ factory ++ salt ++ keccak256(initcode))[12:]
// Attacker deploys a contract at `target` (nonce=0, code=0 after selfdestruct),
// or calls a helper that does SSTORE(target, slot, value).

// Step 2: Victim factory calls CREATE2 with the same salt + initcode
// ZKsync OS collision check: code_len==0, nonce==0 → passes (BUG)
// Constructor runs successfully.

// Step 3: Deployed contract reads storage slot that attacker pre-set
// e.g., slot 0 (owner) = attacker_address  ← attacker now owns the contract
```

On a canonical EVM (post-EIP-7610), Step 2 would revert with a collision error. On ZKsync OS it succeeds, producing a state root that diverges from Ethereum equivalence and a contract with attacker-controlled initial state.
