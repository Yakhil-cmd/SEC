Audit Report

## Title
Pre-authentication unbounded heap allocation and O(N log N) sort in `sub_tree_map_from` via crafted XNet `CertifiedStreamSlice` payload — (`rs/crypto/tree_hash/src/proto.rs`, `rs/state_manager/src/lib.rs`)

## Summary

In `decode_certified_stream_slice` (`rs/state_manager/src/lib.rs`), `decode_labeled_tree` is called at line 4135 to fully decode and sort the payload tree before signature verification occurs at line 4145. A Byzantine node on a remote subnet can serve a crafted `CertifiedStreamSlice` whose `payload` encodes a `LabeledTree::SubTree` with millions of flat children, causing the receiving replica to allocate ~1–2 GB of heap and perform an O(N log N) sort before the invalid signature is detected and the slice is rejected. Repeated delivery of such slices can cause sustained memory pressure or OOM-crash of the receiving replica.

## Finding Description

`decode_certified_stream_slice` (`rs/state_manager/src/lib.rs:4094`) unconditionally decodes the payload before verifying the certification:

```
line 4135: let tree = stream_encoding::decode_labeled_tree(&certified_slice.payload)?;
...
line 4145: if !verify_recomputed_digest(...) { return Err(InvalidSignature) }
```

`decode_labeled_tree` (`rs/state_manager/src/stream_encoding.rs:193`) calls `v1::LabeledTree::proxy_decode` → `TryFrom<pb::LabeledTree>` → `sub_tree_map_from` (`rs/crypto/tree_hash/src/proto.rs:41`):

```rust
let kv: Vec<_> = subtree.children.into_iter()
    .map(|child| Ok((Label::from(child.label),
        try_from_option_field(child.node, "LabeledTree::Subtree::value")?)))
    .collect::<Result<_, ProxyDecodeError>>()?;
Ok(FlatMap::from_key_values(kv))
```

There is no child-count limit anywhere in this path. `FlatMap::from_key_values` (`rs/crypto/tree_hash/src/flat_map.rs:88`) then sorts the entire Vec:

```rust
pub fn from_key_values(mut kv: Vec<(K, V)>) -> Self {
    if kv.windows(2).any(|w| w[0].0 >= w[1].0) {
        kv.sort_unstable_by(|l, r| l.0.cmp(&r.0));
        kv.dedup_by(|l, r| l.0 == r.0);
    }
    let (keys, values) = kv.into_iter().unzip();
    Self { keys, values }
}
```

Prost's recursion depth limit (100) does not protect against wide/flat trees. The HTTP client caps the body at `5 * POOL_SLICE_BYTE_SIZE_MAX` (`rs/xnet/payload_builder/src/lib.rs:1839`), which bounds the input to ~20 MB. At ~7 bytes per minimal child (1-byte label + empty leaf), this encodes ~2.8M children. Each child materializes as a heap-allocated `Label` (`Vec<u8>`) and `LabeledTree::Leaf(Vec<u8>)`; the original Vec, sort buffer, and final `FlatMap` (two separate `Vec`s) together consume ~1–2 GB of heap — all before any authentication check.

## Impact Explanation

A single Byzantine node on a remote subnet can repeatedly trigger ~1–2 GB of heap allocation and O(N log N) CPU work per XNet pool-refill cycle on a receiving replica, before the invalid signature causes rejection. Under repeated fire, this causes sustained memory pressure or OOM-crash of the receiving replica. This matches the **High ($2,000–$10,000)** bounty impact: *Application/platform-level DoS, crash, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation

Requires controlling one node on a remote subnet, below the consensus fault threshold — a valid Byzantine peer in the ICP protocol model. The XNet pool-refill task runs periodically and fetches from a single node per subnet per cycle, so one malicious node is sufficient. The crafted payload is trivially constructable as a protobuf `LabeledTree` with a `SubTree` containing millions of minimal `Child` entries. The attack is repeatable every pool-refill cycle with no rate limiting on the decode path.

## Recommendation

1. **Move signature verification before payload decoding** in `decode_certified_stream_slice` — verify `certification` against the raw `payload` bytes before calling `decode_labeled_tree`.
2. **Add a child-count limit** in `sub_tree_map_from` (e.g., reject if `subtree.children.len() > MAX_CHILDREN`).
3. **Add a payload byte-size limit** before calling `decode_labeled_tree` (e.g., reject if `payload.len() > POOL_SLICE_BYTE_SIZE_MAX`).

## Proof of Concept

Construct a protobuf `CertifiedStreamSlice` where `payload` encodes a `LabeledTree::SubTree` with ~2.8M children (each with a 1-byte label and empty leaf), totaling ~20 MB. Serve this as the HTTP response body from a node acting as a remote XNet endpoint. The receiving replica's pool-refill task will fetch the slice, call `decode_labeled_tree` at `rs/state_manager/src/lib.rs:4135`, allocate ~1–2 GB of heap, perform an O(N log N) sort in `FlatMap::from_key_values`, and only then reject the slice at line 4145 for invalid signature. A minimal Python script to generate the payload:

```python
def encode_varint(n):
    result = []
    while n > 0x7F:
        result.append((n & 0x7F) | 0x80)
        n >>= 7
    result.append(n)
    return bytes(result)

def encode_field(field_num, wire_type, data):
    tag = (field_num << 3) | wire_type
    return encode_varint(tag) + encode_varint(len(data)) + data

leaf_node = encode_field(1, 2, b'')
child = encode_field(1, 2, b'\x00') + encode_field(2, 2, leaf_node)
subtree_children = child * 2_800_000
subtree = encode_field(1, 2, subtree_children)
labeled_tree = encode_field(2, 2, subtree)
# Use labeled_tree as the payload field of a CertifiedStreamSlice
```

A deterministic integration test can verify the fix by asserting that `decode_certified_stream_slice` rejects an oversized payload (or verifies the signature) before allocating heap proportional to child count.
