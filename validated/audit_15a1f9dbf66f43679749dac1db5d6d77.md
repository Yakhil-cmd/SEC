Audit Report

## Title
Fee-on-Transfer ERC20 Deposit Mints Unbacked ckERC20 Tokens - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

## Summary

The `depositErc20` function in `DepositHelperWithSubaccount.sol` emits the caller-supplied `amount` in the `ReceivedEthOrErc20` event rather than the actual tokens received by the minter address. The IC ckETH minter reads this event and mints exactly `amount` ckERC20 tokens. For any fee-on-transfer ERC20 token, the minter receives `amount - fee` tokens on Ethereum but mints `amount` ckERC20 on the IC, permanently breaking the 1:1 backing invariant and causing protocol insolvency that compounds with every deposit.

## Finding Description

In `DepositHelperWithSubaccount.sol` at lines 511–532, `depositErc20` calls `safeTransferFrom(msg.sender, minterAddress, amount)` and then emits `ReceivedEthOrErc20(..., amount, ...)` using the caller-supplied parameter directly, with no measurement of actual tokens received:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```

The IC minter's `ReceivedEthOrErc20LogParser::parse_log` in `src/eth_logs/parser.rs` (L149–160) reads `value_bytes` from the event data and constructs `Erc20Value::from_be_bytes(value_bytes)` — this is the emitted nominal `amount`, not the actual received amount.

The `mint()` function in `src/deposit.rs` (L73–81) passes `event.value()` directly as the `amount` in `TransferArg`, minting the nominal amount on the IC ledger. The `value()` method in `src/eth_logs/mod.rs` (L205–210) simply returns the parsed log field. The internal balance accounting in `src/state.rs` (L332–338) also records `event.value` (the emitted amount) in `erc20_balances`, causing the minter's own accounting to diverge from on-chain reality.

No existing check in the Solidity contract, the Rust log parser, or the minting logic verifies that the emitted amount matches the actual balance change at the minter address. The minter's token whitelist (enforced via `ckerc20_tokens` in state) prevents processing of unsupported tokens, but provides no protection once a fee-on-transfer token is added as supported.

## Impact Explanation

This is an illegal minting / protocol insolvency bug affecting the ckERC20 chain-fusion financial integration, which is explicitly in scope. For each deposit of `amount` tokens with a fee rate `f`:

- Minter holds `amount × (1 - f)` ERC20 tokens on Ethereum
- IC ledger has `amount` ckERC20 tokens minted
- Shortfall per deposit: `amount × f`
- After N deposits: total shortfall = `N × amount × f`, growing unboundedly

When users attempt to withdraw ckERC20 back to Ethereum, the minter will eventually be unable to fulfill withdrawals because it holds fewer ERC20 tokens than the outstanding ckERC20 supply. This matches the allowed impact: **Theft, permanent loss, illegal minting, or protocol insolvency involving in-scope chain-key/ledger assets**. Severity is **High** given the precondition that a fee-on-transfer token must be supported, though the shortfall could exceed $1M at scale.

## Likelihood Explanation

The exploit requires a fee-on-transfer ERC20 token to be a supported ckERC20 token. Two realistic paths exist:

1. **Future governance**: Any NNS proposal to add a new ckERC20 token could inadvertently add a fee-on-transfer token. The `add_ckerc20_token` endpoint in `src/main.rs` (L563–573) accepts any ERC20 address from the orchestrator without fee-on-transfer validation. The NNS governance process documented in `rs/ethereum/ledger-suite-orchestrator/README.adoc` specifies no requirement to verify the absence of transfer fees.

2. **Existing supported token**: USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is already a supported ckERC20 token (added via NNS proposal, documented in `rs/ethereum/cketh/mainnet/orchestrator_upgrade_2024_07_22_ckusdt.md`). The USDT contract contains an owner-controlled fee mechanism (currently set to 0 bps) that Tether can enable at any time. If enabled, every ckUSDT deposit would immediately mint unbacked tokens.

Once the precondition is met, exploitation requires no special privileges — any user calling `depositErc20` triggers the invariant violation.

## Recommendation

In `depositErc20`, measure the actual received amount by comparing the minter's balance before and after the transfer, and emit that value:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 receivedAmount = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, receivedAmount, principal, subaccount);
```

Additionally, the NNS governance process for adding new ckERC20 tokens should explicitly verify that the ERC20 token does not implement transfer fees, and the `add_ckerc20_token` endpoint should document this as a required precondition.

## Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FeeToken` with a 1% fee on every transfer, or use the existing USDT contract after Tether enables its fee mechanism.
2. Add `FeeToken` as a supported ckERC20 token via NNS governance proposal targeting the ledger suite orchestrator.
3. Alice calls `depositErc20(FeeToken, 1_000_000, alicePrincipal, subaccount)` on the helper contract at `DepositHelperWithSubaccount.sol`.
4. `safeTransferFrom` transfers 1,000,000 tokens from Alice; the ERC20 contract deducts 1% fee, so `minterAddress` receives 990,000 tokens.
5. The helper contract emits `ReceivedEthOrErc20(FeeToken, Alice, 1_000_000, alicePrincipal, subaccount)` — the nominal amount.
6. The IC minter scrapes the log; `ReceivedEthOrErc20LogParser::parse_log` parses `value = 1_000_000`.
7. `mint()` calls `client.transfer(TransferArg { amount: 1_000_000, to: aliceAccount, ... })`, minting 1,000,000 ckFeeToken to Alice.
8. The minter holds 990,000 FeeToken on Ethereum but has issued 1,000,000 ckFeeToken on IC — a 10,000 token shortfall per deposit.
9. After 100 such deposits: minter holds 99,000,000 FeeToken, has issued 100,000,000 ckFeeToken — 1,000,000 token shortfall.
10. Verify via a local PocketIC integration test: mock a fee-on-transfer ERC20, simulate `depositErc20` log emission with nominal amount, confirm `mint()` mints the nominal amount, and assert `erc20_balances` diverges from the mock ERC20 `balanceOf(minterAddress)`.
