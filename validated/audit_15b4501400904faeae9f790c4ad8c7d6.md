Audit Report

## Title
SNS Governance `distribute_rewards` Unbounded Neuron Iteration Causes Permanent Reward Liveness Failure - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS Governance canister's `distribute_rewards` function iterates over all voting neurons in a single synchronous message with no instruction-limit guard. If the neuron count grows large enough to exhaust the IC per-message instruction limit (~40B instructions), the message traps and rolls back all state. Because `latest_reward_event` is only written after the loop completes, `should_distribute_rewards()` returns `true` on every subsequent timer invocation, re-triggering the same unbounded loop indefinitely, permanently halting voting reward distribution.

## Finding Description
`run_periodic_tasks` calls `distribute_rewards` synchronously after fetching total supply from the ledger: [1](#0-0) 

Inside `distribute_rewards`, `neuron_id_to_reward_shares` is built from all proposal ballots across all `considered_proposals`, then iterated without any instruction-limit check: [2](#0-1) 

Only after this loop completes does the function settle proposals and write `latest_reward_event`: [3](#0-2) 

If the loop traps due to instruction exhaustion, all state changes roll back. `latest_reward_event` is unchanged, `considered_proposals` remain in `ReadyToSettle`, and `should_distribute_rewards()` returns `true` again on the next timer tick — re-entering the identical (or larger) loop. There is no instruction-limit guard anywhere in the SNS reward distribution path; a grep for `is_message_over_threshold` in `rs/sns/**` returns zero matches.

The NNS Governance canister received an explicit fix for this identical bug (Proposal 135702) via a `RewardsDistributionStateMachine` stored in stable memory and a `continue_processing` loop that breaks on `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)`: [4](#0-3) 

The NNS fix schedules remaining work via a recurring 2-second timer task: [5](#0-4) 

The SNS canister received no equivalent remediation.

## Impact Explanation
Once the neuron-iteration loop exceeds the IC instruction limit, every invocation of `run_periodic_tasks` traps at the same point. Voting rewards are permanently halted: neuron maturity stops accruing, breaking the economic incentive for governance participation in the affected SNS DAO. This is a concrete, permanent application-level liveness failure for an in-scope SNS governance canister — matching the allowed impact class: *"Significant SNS infrastructure security impact with concrete user or protocol harm"* and *"Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS."* Severity: **High**.

## Likelihood Explanation
This is an organic, non-adversarial failure requiring no special privileges. Any SNS DAO that accumulates a sufficiently large number of active voting neurons will hit this. The NNS governance team's own changelog explicitly confirms this was a real production bug requiring an emergency fix: [6](#0-5) 

The SNS canister did not receive the same remediation, leaving any sufficiently popular SNS DAO at risk.

## Recommendation
Apply the same batched-distribution pattern already implemented in NNS Governance:
1. After computing `neuron_id_to_reward_shares`, store the pending distribution in stable memory (analogous to `RewardsDistributionStateMachine`).
2. Advance `latest_reward_event` and settle proposals immediately so they are not re-processed on the next timer tick.
3. Distribute maturity to neurons in a separate recurring timer task that calls `is_message_over_threshold` after each neuron and resumes in the next message if the limit is reached.

## Proof of Concept
1. Deploy an SNS DAO and accumulate a large number of neurons (e.g., 50,000+), all of which vote on proposals.
2. Wait for the recurring timer to fire `run_periodic_tasks` → `distribute_rewards`.
3. The loop at line 5954 iterates over all `neuron_id_to_reward_shares` entries; after enough neurons, the IC instruction counter exceeds the per-message limit and the message traps.
4. All state changes roll back. `latest_reward_event` is unchanged. `considered_proposals` remain `ReadyToSettle`.
5. The next timer fires; `should_distribute_rewards()` returns `true` again. The identical loop runs and traps again.
6. Voting rewards are permanently halted.

A deterministic integration test can reproduce this by mocking `env.now()` to advance past the reward round boundary, populating a large `neuron_id_to_reward_shares` map, and asserting that `latest_reward_event` is never advanced when the instruction limit is simulated as exceeded mid-loop.
