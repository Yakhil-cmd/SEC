Audit Report

## Title
CMC `tokens_to_cycles` Uses Unbounded XRC Rate for Cycle Minting Without Governance-Defined Ceiling - (`rs/nns/cmc/src/main.rs`)

## Summary
The Cycles Minting Canister stores and uses any ICP/XDR rate returned by the Exchange Rate Canister that is strictly greater than zero, with no floor or ceiling derived from governance-defined parameters. Every other rate-consuming path in the codebase applies explicit bounds, but the cycle minting path (`tokens_to_cycles`) does not. If the XRC returns a transiently inflated rate, any user calling `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during that window receives proportionally more cycles per ICP than the protocol intends.

## Finding Description
`validate_exchange_rate` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` (L111–129) validates only that enough data sources responded; it performs no check on the numeric value of the returned rate. [1](#0-0) 

`do_set_icp_xdr_conversion_rate` in `rs/nns/cmc/src/main.rs` (L1018–1020) rejects only a rate of exactly zero and stores any positive value directly into state. [2](#0-1) 

`tokens_to_cycles` (L1900–1923) reads this stored rate and passes it directly to `TokensToCycles::to_cycles`, which computes cycles linearly in `xdr_permyriad_per_icp` with no floor or ceiling applied. [3](#0-2) [4](#0-3) 

By contrast, the node-provider-rewards path applies `max(avg_rate, minimum_xdr_permyriad_per_icp)`: [5](#0-4) 

And the Neurons' Fund path clamps to `[minimum_icp_xdr_rate, maximum_icp_xdr_rate]`: [6](#0-5) 

The governance-defined `minimum_icp_xdr_rate` default is 100 (= 1 XDR per ICP), and `NeuronsFundEconomics` defines a `maximum_icp_xdr_rate` that is validated but never consulted by the CMC minting path. [7](#0-6) [8](#0-7) 

The CMC does have a `base_cycles_limit` (default `150e15` cycles per hour) enforced via `ensure_balance` → `check_and_add_cycles`. This limits total minting volume per hour but does not prevent the per-ICP excess: a 5× inflated rate still yields 5× cycles per ICP for every transaction that fits within the hourly cap. [9](#0-8) [10](#0-9) 

The heartbeat calls `update_exchange_rate` every five minutes and immediately stores whatever the XRC returns (provided source counts pass), with no smoothing and no comparison against the previous stored rate beyond a timestamp check. [11](#0-10) [12](#0-11) 

## Impact Explanation
Any user who submits an ICP transfer and calls `notify_top_up` / `notify_create_canister` / `notify_mint_cycles` while an inflated rate is stored receives proportionally more cycles per ICP than the protocol intends. Because cycles are the resource that funds computation on the IC, minting excess cycles at a below-market ICP cost is a direct, permanent resource accounting loss for the network. The impact matches the allowed bounty class: **illegal minting of cycles / protocol resource insolvency** (High, $2,000–$10,000), constrained by the 5-minute exposure window per anomalous rate and the hourly minting cap. [4](#0-3) 

## Likelihood Explanation
The XRC queries multiple centralised exchanges and computes a median, which provides resistance to single-exchange anomalies. However, correlated thin-order-book flash spikes across a subset of exchanges (a well-documented phenomenon in crypto markets) can transiently push the median above the true market rate without any exchange being "compromised." No special privileges are required: any unprivileged user can call `notify_top_up` with a standard ICP transfer. The exploit window is up to five minutes per anomalous rate update. The attacker needs only to monitor the CMC's publicly readable `icp_xdr_conversion_rate` certified state and submit a transfer during the elevated-rate window. [12](#0-11) 

## Recommendation
1. In `do_set_icp_xdr_conversion_rate`, query the governance canister (or read a cached copy of `NetworkEconomics`) for `minimum_icp_xdr_rate` and `NeuronsFundEconomics::maximum_icp_xdr_rate`, and reject (or clamp and log) any proposed rate outside that window — mirroring the clamp already present in `try_derive_neurons_fund_participation_limits_impl`.
2. Alternatively, apply the floor inside `tokens_to_cycles` itself, consistent with how `get_monthly_node_provider_rewards` applies `max(avg_rate, minimum_rate)`.
3. Add a numeric sanity check inside `validate_exchange_rate` that rejects rates outside a reasonable absolute range, not only source-count checks. [6](#0-5) [5](#0-4) 

## Proof of Concept
1. The XRC transiently returns `xdr_permyriad_per_icp = 500_000` (50 XDR/ICP, ~5× real rate) due to a correlated flash spike across multiple exchanges pushing the median up.
2. `validate_exchange_rate` passes: source counts meet `MINIMUM_ICP_SOURCES` and `MINIMUM_CXDR_SOURCES`.
3. `do_set_icp_xdr_conversion_rate` passes: `500_000 > 0`.
4. CMC state now holds `icp_xdr_conversion_rate.xdr_permyriad_per_icp = 500_000`.
5. An attacker calls `notify_top_up` with 1 ICP (1e8 e8s).
6. `TokensToCycles::to_cycles` computes: `1e8 * 500_000 * cycles_per_xdr / (1e8 * 10_000)` = `50 * cycles_per_xdr` instead of the intended `~10 * cycles_per_xdr`.
7. The attacker receives ~5× the intended cycles with no revert or bounds check.

A deterministic integration test can reproduce this using `state_machine_builder_for_nns_tests`, injecting a mock XRC that returns `xdr_permyriad_per_icp = 500_000` with valid source counts, then calling `notify_top_up` and asserting the returned `Cycles` value exceeds the governance-intended ceiling. [3](#0-2) [4](#0-3)

pub fn validate_exchange_rate(
    exchange_rate: &ExchangeRate,
) -> Result<(), ValidateExchangeRateError> {
    if exchange_rate.metadata.base_asset_num_received_rates < MINIMUM_ICP_SOURCES {
        return Err(ValidateExchangeRateError::NotEnoughIcpSources {
            received: exchange_rate.metadata.base_asset_num_received_rates,
            queried: exchange_rate.metadata.base_asset_num_queried_sources,
        });
    }

    if exchange_rate.metadata.quote_asset_num_received_rates < MINIMUM_CXDR_SOURCES {
        return Err(ValidateExchangeRateError::NotEnoughCxdrSources {
            received: exchange_rate.metadata.quote_asset_num_received_rates,
            queried: exchange_rate.metadata.quote_asset_num_queried_sources,
        });
    }

    Ok(())
}

fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles {
                xdr_permyriad_per_icp,
                cycles_per_xdr: state.cycles_per_xdr,
            }
            .to_cycles(amount)),
            None => {
                let error_message =
                    "No conversion rate found in CMC, notification aborted".to_string();
                print(&error_message);
                Err(NotifyError::Other {
                    error_code: NotifyErrorCode::Internal as u64,
                    error_message,
                })
            }
        }
    })
}
