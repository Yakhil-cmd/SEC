### Title
`set_eth_connector_contract_account` Replaces Connector Without Balance Verification, Permanently Freezing User Funds - (File: engine/src/contract_methods/connector.rs)

---

### Summary

The `set_eth_connector_contract_account` function allows the engine owner to atomically replace the stored eth-connector account ID with no check that the new connector holds the NEP-141 token supply that backs all existing EVM ETH balances. After the swap, every user `withdraw` call is routed to the new connector, which holds zero NEP-141 balance, making it impossible to redeem EVM ETH for real ETH. The funds are permanently frozen unless the owner manually reverts the change.

---

### Finding Description

`set_eth_connector_contract_account` unconditionally overwrites the stored connector account:

```rust
// engine/src/contract_methods/connector.rs  lines 417-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);          // ← no balance check
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}

The stored account ID is the sole routing key used by every outbound connector call. The `withdraw` function (and `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_withdraw`, `ft_metadata`) all reach the connector exclusively through `return_promise`, which reads `get_connector_account_id` at call time:

```rust

The `withdraw` entry point builds the `engine_withdraw` promise and delegates entirely to `return_promise`:

    return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)

``` [3](#0-2) 

The engine never deducts the user's EVM ETH balance before creating the promise; the deduction is performed inside the connector's `engine_withdraw` handler. If the connector account has been replaced with a contract that holds no NEP-141 balance, `engine_withdraw` on the new connector will fail (no balance to burn), the promise reverts, and the user's EVM ETH is left intact but permanently unwithdrawable.

The `ft_on_transfer` path has a symmetric problem: the old connector's pending `ft_on_transfer` calls are no longer recognized as the trusted connector, so they fall into the `receive_erc20_tokens` branch and fail:

```rust

---

### Impact Explanation

**Critical – Permanent freezing of funds.**

All EVM ETH balances that were minted by the old connector are backed by NEP-141 tokens held in that old connector. After the connector account is replaced, the `withdraw` → `engine_withdraw` promise is routed to the new connector, which holds no NEP-141 balance. Every withdrawal attempt fails. Users retain their EVM ETH balance on Aurora but have no mechanism to redeem it for real ETH on Ethereum. The total locked value equals the entire NEP-141 supply of the old connector at the time of the switch.

---

### Likelihood Explanation

The function is callable by the engine owner at any time with no timelock or migration guard:

```rust
// engine/src/contract_methods/connector.rs  lines 425-429
let is_private = env.assert_private_call();
if is_private.is_err() {
    require_owner_only(&state, &env.predecessor_account_id())?;
}
``` [5](#0-4) 

A legitimate connector upgrade (e.g., deploying a new version of the eth-connector contract) is a realistic operational event. The function provides no guardrails to prevent the owner from pointing to a new connector before migrating the NEP-141 balances, making the freeze reachable through ordinary maintenance operations without any malicious intent.

---

### Recommendation

Before overwriting the connector account ID, verify that the new connector holds at least as much NEP-141 total supply as the old connector. If a cross-contract balance query is not feasible atomically, consider:

1. Making the connector account immutable after initialization (preferred if migration is not required).
2. Requiring a two-step migration: first register the new connector, then migrate balances, then finalize the switch.
3. At minimum, emitting a warning and requiring an explicit acknowledgment that balances have been migrated before the switch takes effect.

---

### Proof of Concept

1. Alice deposits 1,000,000 ETH through `old_connector`. `old_connector` calls `ft_on_transfer` on Aurora; `predecessor_account_id == get_connector_account_id()` is `true`; `receive_base_tokens` mints 1,000,000 EVM ETH for Alice.
2. Owner deploys `new_connector` (NEP-141 balance = 0) and calls `set_eth_connector_contract_account` with `new_connector`.
3. Alice calls `withdraw` on Aurora engine. `return_promise` routes `engine_withdraw` to `new_connector`.
4. `new_connector` has no NEP-141 balance for Alice; `engine_withdraw` fails; the promise reverts.
5. Alice's 1,000,000 EVM ETH remains in the Aurora EVM state but is permanently unwithdrawable. Alice suffers a total loss of 1,000,000 ETH.
