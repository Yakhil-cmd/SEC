### Title
Idempotent IPv4 Re-Apply Causes Registry Canister Panic Due to Self-Inclusive Duplicate Check — (`rs/registry/canister/src/mutations/node_management/do_update_node_ipv4_config_directly.rs`)

---

### Summary

`node_exists_with_ipv4` scans **all** registry nodes without excluding the target node. When a node operator re-submits the same IPv4 address that is already set on their own node, the check finds the node itself and panics with `"There is already at least one other node with the same IPv4 address"`. The comment at the call site explicitly says *"any other node"*, but the implementation does not enforce that exclusion.

---

### Finding Description

`validate_update_node_ipv4_config_directly_payload` calls `node_exists_with_ipv4` at line 92 before any mutation is applied (mutation happens at line 72): [1](#0-0) 

The comment reads *"Ensure that the IPv4 address is not used by any other node"*, but `node_exists_with_ipv4` performs no such exclusion: [2](#0-1) 

It iterates over every `NodeRecord` in the registry and returns `true` the moment any node — including the target node itself — has a matching `ip_addr`. Because validation runs against the **pre-mutation** state (line 52 precedes line 72), the target node's existing IPv4 address is still present in the registry when the scan executes: [3](#0-2) 

**Concrete trigger sequence:**
1. Node operator calls `do_update_node_ipv4_config_directly` with node A and IPv4 `X` → succeeds, registry now stores `X` for node A.
2. Node operator calls `do_update_node_ipv4_config_directly` again with the same node A and same IPv4 `X` (idempotent re-apply).
3. `node_exists_with_ipv4(self, "X")` finds node A's own record, returns `true`.
4. `panic!("There is already at least one other node with the same IPv4 address")` fires → registry canister traps.

The existing test suite does not cover this case. The test `should_succeed_updating_ipv4_config_two_times` only tests set-then-clear-to-`None`, not set-then-re-set-same-value: [4](#0-3) 

---

### Impact Explanation

Every call to `do_update_node_ipv4_config_directly` with an already-assigned IPv4 address traps the registry canister for that message. Any orchestrator or node management component that retries the same config (e.g., after a transient failure) will be permanently blocked from succeeding. The node operator cannot update any other field of their IPv4 config (gateway, prefix length) without first clearing the address to `None` in a separate call, which is an undocumented and non-obvious workaround.

---

### Likelihood Explanation

This is reachable by any legitimate node operator via a direct ingress call to the registry canister — no privileged access required. Orchestrator retry logic that re-applies the same config is a standard operational pattern, making accidental triggering likely in production.

---

### Recommendation

Pass the target `node_id` into `node_exists_with_ipv4` (or a new variant) and exclude it from the scan:

```rust
pub fn node_exists_with_ipv4_excluding(
    registry: &Registry,
    ipv4_addr: &str,
    exclude_node_id: NodeId,
) -> bool {
    get_key_family::<NodeRecord>(registry, NODE_RECORD_KEY_PREFIX)
        .into_iter()
        .find_map(|(k, v)| {
            let nid = NodeId::from(PrincipalId::from_str(&k).unwrap());
            if nid == exclude_node_id { return None; }
            v.public_ipv4_config.and_then(|config| {
                (config.ip_addr == ipv4_addr).then_some(nid)
            })
        })
        .is_some()
}
```

Then replace the call at line 92 with `node_exists_with_ipv4_excluding(self, ipv4_config.ip_addr(), payload.node_id)`.

/// Checks if there is a node with the provided IPv4 address
pub fn node_exists_with_ipv4(registry: &Registry, ipv4_addr: &str) -> bool {
    get_key_family::<NodeRecord>(registry, NODE_RECORD_KEY_PREFIX)
        .into_iter()
        .find_map(|(k, v)| {
            v.public_ipv4_config.and_then(|config| {
                (config.ip_addr == ipv4_addr)
                    .then(|| NodeId::from(PrincipalId::from_str(&k).unwrap()))
            })
        })
        .is_some()
}
