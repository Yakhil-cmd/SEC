### Title
Asymmetric Rounding in `native_per_pubdata` vs `native_per_gas` Allows Systematic Underpayment for Pubdata — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

ZKsync OS computes the two key resource-pricing ratios with **opposite rounding directions**: `native_per_gas` uses ceiling division (`div_ceil`) while `native_per_pubdata` uses floor/truncating division (`wrapping_div`). This is the direct analog of the ERC4626 bug where `mint` was patched but `deposit` was left unchanged — one path was tightened, the other was not. The result is that any L2 transaction sender can systematically underpay for pubdata native resources. In the degenerate case where `pubdata_price < native_price`, the floor truncates to zero and pubdata costs the sender **nothing** in native resources.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` the two ratios are computed as:

```rust
// native_per_gas — ceiling division (rounds UP, user gets more native budget)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::NativeResourcesAreTooExpensive))?;

// native_per_pubdata — floor/truncating division (rounds DOWN, user pays less for pubdata)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same floor-division pattern is replicated in the public API helper:

```rust
// api/src/helpers.rs
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

`native_per_pubdata` is then used in two places that directly determine how much native resource a sender must hold:

1. **Intrinsic pubdata overhead** deducted from the native limit at tx setup:
   `intrinsic_pubdata_overhead = native_per_pubdata * intrinsic_pubdata` [3](#0-2) 

2. **Post-execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   `native = current_pubdata_spent * native_per_pubdata` [4](#0-3) 

The documentation itself states the intended formula as a plain ratio with no specified rounding direction:
> `nativePerGas := gasPrice/nativePrice` [5](#0-4) 

The asymmetry is therefore unintentional: the gas-side ratio was hardened with ceiling (protecting the protocol), but the pubdata-side ratio was left with floor (benefiting the sender).

---

### Impact Explanation

**Bounded underpayment (general case):** For every pubdata byte written, the sender underpays by up to `(native_price − 1)` native units. With `native_price = 10` (the L1 constant) and a transaction writing the maximum allowed pubdata, the cumulative underpayment can reach hundreds of thousands of native units — real proving cost absorbed by the protocol without compensation.

**Zero-cost pubdata (degenerate case):** When `pubdata_price < native_price`, `wrapping_div` truncates to `0`. Both the intrinsic pubdata overhead and the per-byte pubdata charge become zero. A sender can write to arbitrarily many storage slots and pay **zero native resources** for the resulting pubdata, while the prover still bears the full cost of publishing those state diffs. This is the direct analog of the ERC4626 "1 wei → 1 wad shares" path: a specific numeric condition (here `pubdata_price < native_price`, there `totalSupply == 0`) collapses the intended price to zero.

The `delta_gas` adjustment in `compute_gas_refund` also uses floor division:

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
``` [6](#0-5) 

This compounds the asymmetry: the sender is charged fewer extra gas units for native consumption than the ceiling-based `native_per_gas` would imply, further reducing effective payment.

---

### Likelihood Explanation

The degenerate (`native_per_pubdata = 0`) case requires `pubdata_price < native_price`. Both values are oracle-supplied block metadata. Any block where the operator or an oracle-data-influencing caller sets `pubdata_price` below `native_price` — whether by misconfiguration, a market condition where L1 data costs drop, or deliberate oracle manipulation — immediately enables zero-cost pubdata for every L2 transaction in that block. The bounded-underpayment case is active in every block where `pubdata_price` is not an exact multiple of `native_price`, which is the common case for arbitrary price feeds.

---

### Recommendation

Replace the floor division for `native_per_pubdata` with ceiling division to match `native_per_gas`:

```rust
// Before (floor):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. Additionally, add an explicit guard that rejects (or treats as zero-pubdata-budget) any transaction submitted in a block where `native_per_pubdata == 0` but `pubdata_price > 0`, mirroring the ERC4626 recommendation to revert when the initial deposit would produce a zero price.

---

### Proof of Concept

**Setup:** Deploy a contract that writes to 20 distinct storage slots (generating ~640 bytes of pubdata). Set block context: `native_price = 10`, `pubdata_price = 9` (one unit below `native_price`).

**Step 1 — Compute `native_per_pubdata`:**
```
native_per_pubdata = floor(9 / 10) = 0
```

**Step 2 — Intrinsic pubdata overhead:**
```
intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0
```

**Step 3 — Post-execution pubdata charge:**
```
native_charge = 640 bytes * 0 = 0
```

**Result:** The sender pays zero native resources for 640 bytes of pubdata. The prover must still prove and publish all 640 bytes of state diffs. The protocol absorbs the full proving cost.

**Contrast with correct ceiling behavior:**
```
native_per_pubdata = ceil(9 / 10) = 1
native_charge = 640 bytes * 1 = 640 native units
```

The attacker-controlled entry path is a standard L2 EIP-1559 transaction submitted to a block with `pubdata_price < native_price`. No privileged access is required; the sender only needs to craft a transaction that touches storage.
