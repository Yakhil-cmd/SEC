All six steps of the claimed call chain are confirmed in the actual production code. Let me verify the one nuance about `initial_address_discovery` before writing the report.

`initial_address_discovery` is set at construction and only cleared to `false` in `process_addr_message` — it is never reset to `true` after depletion. So the `MAX_CONNECTIONS_DURING_ADDRESS_DISCOVERY` path does not re-engage post-depletion. However, the core impact still holds: when the address book is depleted, `make_connection` falls back to `resolve_next_seed()`, which returns `AddressEntry::Seed` entries, and `available_connections()` explicitly filters out all seed connections — leaving zero connections available to serve Bitcoin data to the IC during re-discovery.

Audit Report

## Title
Single `NetworkMessage::Unknown` Permanently Removes Peer Address, Enabling Address Book Depletion — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary

`process_unknown_message` unconditionally calls `internal_discard` on any peer that sends a single unknown Bitcoin P2P message, with no rate limiting, counter, or grace period. When DNS seeds are configured (normal production), `address_book.discard()` permanently removes the peer's address from both `active_addresses` and `known_addresses`. An attacker controlling enough Bitcoin-protocol-speaking nodes can deplete the adapter's address book, forcing it into a continuous seed re-discovery loop during which no connections are available to serve Bitcoin data to the IC, disrupting ckBTC and canister Bitcoin API calls.

## Finding Description

**Step 1 — Router dispatch**: `router.rs` L96–99 passes every received `NetworkMessage::Unknown` to `connection_manager.process_bitcoin_network_message`. Because `process_unknown_message` returns `Ok(())`, the router's outer `discard` guard does not fire; the discard happens entirely inside the handler. [1](#0-0) 

**Step 2 — Message dispatch**: `process_bitcoin_network_message` matches `NetworkMessage::Unknown` and calls `process_unknown_message` with no prior state check — it fires regardless of whether the connection is in `Initializing`, `Connected`, `HandshakeComplete`, or any other state. [2](#0-1) 

**Step 3 — Unconditional discard**: `process_unknown_message` logs the message and immediately calls `self.internal_discard(address)`. There is no counter, threshold, time window, or connection-state guard. [3](#0-2) 

**Step 4 — State transition**: `internal_discard` calls `conn.discard()`, which sets `state = ConnectionState::AdapterDiscarded` and aborts the stream task. [4](#0-3) [5](#0-4) 

**Step 5 — Reap on tick**: `reap_disconnected`, called every 100 ms via `tick()`, matches `ConnectionState::AdapterDiscarded` and calls `self.address_book.discard(conn.address_entry())`. [6](#0-5) 

**Step 6 — Permanent removal**: `address_book.discard()` removes the address from both `active_addresses` and `known_addresses` when `has_seeds()` is true (i.e., `dns_seeds` is non-empty, which is always the case in production mainnet/testnet deployments). The address is gone permanently — it is not returned to `known_addresses` as `remove_from_active` would do. [7](#0-6) 

**Step 7 — Fallback to seeds, no available connections**: Once `known_addresses` drops below `min_addresses`, `make_connection` calls `resolve_next_seed()` instead of `pop()`, producing `AddressEntry::Seed` connections. `available_connections()` explicitly filters out all seed entries, so the adapter has zero connections available to serve Bitcoin data during re-discovery. [8](#0-7) [9](#0-8) 

**Existing guards are insufficient**: The only guard in `internal_discard` is a `get_connection` lookup — it does not check connection state. No misbehavior counter, no minimum address-book floor, and no rate limit exist anywhere in the path.

## Impact Explanation

During the re-discovery phase the adapter has no `available_connections`, so `send_to_all` is a no-op and all Bitcoin data requests from the IC (block fetches, transaction broadcasts, `bitcoin_get_utxos`, `bitcoin_send_transaction`) go unanswered. This directly disrupts ckBTC minting/redemption and all canister Bitcoin API calls for as long as the attacker sustains the campaign. This matches the allowed High impact: **Application/platform-level DoS with concrete user and protocol harm to a significant Chain Fusion / ck-token integration**.

## Likelihood Explanation

Running Bitcoin-protocol-speaking nodes is unprivileged and low-cost. The attacker must get their node addresses into the adapter's `known_addresses`, which happens via normal Bitcoin `addr` message propagation from any peer the adapter is already connected to. Once an attacker-controlled address is popped and connected, a single `Unknown` message permanently removes it. With `max_connections = 5` and a small address book (`min_addresses` is small per `connection_limits`), an attacker needs only a handful of nodes cycling through the book to sustain continuous disruption. No admin access, key material, governance majority, BGP/DNS hijack, or victim mistake is required. [10](#0-9) 

## Recommendation

1. **Add a per-address misbehavior counter**: Only call `internal_discard` after N unknown messages within a sliding time window, not on the first occurrence.
2. **Soft-disconnect instead of hard-discard**: Treat `Unknown` as `conn.disconnect()` → `NodeDisconnected` → `remove_from_active`, which returns the address to `known_addresses` rather than permanently deleting it.
3. **Address-book floor**: In `address_book.discard()`, if removing the address would drop `known_addresses` below `min_addresses`, demote to `NodeDisconnected` instead of permanently discarding.

## Proof of Concept

State-machine test (no network required):

1. Construct a `ConnectionManager` with DNS seeds configured (so `has_seeds()` is true).
2. Insert 10 `Connection` objects with `AddressEntry::Discovered` and `ConnectionState::HandshakeComplete` into `manager.connections`; insert their addresses into `address_book.active_addresses`.
3. For each of the 10 addresses, call `manager.process_bitcoin_network_message(addr_i, &NetworkMessage::Unknown { command: "exploit".parse().unwrap(), payload: vec![] })`.
4. Call `manager.reap_disconnected()`.
5. Assert `manager.address_book.size() == 0` — all addresses have been permanently removed from both `active_addresses` and `known_addresses`.
6. Assert `manager.address_book.has_enough_addresses() == false` — the adapter will fall back to seed re-discovery on the next `tick()`.
