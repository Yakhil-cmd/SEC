Audit Report

## Title
Single Canister Can Monopolize the Subnet-Wide Threshold Signature Queue, Denying Service to All Other Canisters - (File: `rs/execution_environment/src/execution_environment.rs`)

## Summary
The `sign_with_threshold` function enforces only a subnet-wide total queue limit for threshold signature requests (ECDSA, Schnorr, VetKD) with no per-sender cap. A single canister can fill all available queue slots, blocking every other canister on the subnet from obtaining threshold signatures until the attacker's requests are processed. Canisters on the NNS subnet can perform this attack at zero cycle cost.

## Finding Description
In `sign_with_threshold()` at `rs/execution_environment/src/execution_environment.rs` lines 3844–3858, the only admission control before enqueuing a new `SignWithThresholdContext` is a check against the total number of in-flight requests for the requested key:

```rust
if state
    .metadata
    .subnet_call_context_manager
    .sign_with_threshold_contexts_count(&threshold_key)
    >= dynamic_queue_size
{ return Err(...); }
```

`sign_with_threshold_contexts_count` at `rs/replicated_state/src/metadata_state/subnet_call_context_manager.rs` lines 437–453 counts **all** pending requests for the key across **all** senders — it does not filter by `request.sender`. There is no per-canister quota anywhere in the admission path.

`dynamic_queue_size` is bounded by `MAX_PAIRED_PRE_SIGNATURES = 100` (`rs/limits/src/lib.rs` line 102), with a registry default of 20. A single canister can submit exactly `dynamic_queue_size` requests in rapid succession, filling every slot. Subsequent requests from any other canister are rejected with `"request queue for key … is full."` until the attacker's requests are processed.

Additionally, at lines 3783–3818, requests from canisters on the NNS subnet are **exempt from the signature fee entirely**, allowing a NNS-deployed canister to flood the queue at zero cost indefinitely.

The existing test `test_sign_with_threshold_key_queue_fills_up` at `rs/execution_environment/tests/threshold_signatures.rs` lines 848–905 directly confirms this behavior: a single canister fills all 20 slots and the next request is rejected.

## Impact Explanation
This is a concrete **application/platform-level DoS** on threshold signature availability — a High-severity in-scope ICP bounty impact. Any canister on the subnet can block all other canisters from obtaining ECDSA, Schnorr, or VetKD signatures for the duration its requests are in-flight. Since the attacker can immediately re-fill the queue once its requests are processed, the DoS can be sustained continuously. Threshold signatures are a critical cross-chain and authentication primitive; sustained unavailability directly harms all dapps and Chain Fusion integrations relying on them.

## Likelihood Explanation
Any unprivileged canister on any application subnet can perform this attack by paying `dynamic_queue_size × signature_fee` cycles (e.g., 20 × 10B = 200B cycles for ECDSA — trivially affordable). Any canister on the NNS subnet can perform this attack for **free** with no cycle cost at all. The attack requires no special privileges, leaked keys, or social engineering, and is fully deterministic and repeatable.

## Recommendation
Introduce a per-sender cap within `sign_with_threshold`. Before enqueuing, count how many in-flight requests `request.sender` already has for the given key and reject if it exceeds a configured per-canister limit (e.g., a fixed small constant, or `max_queue_size / N`). This mirrors the per-canister callback quota already used elsewhere in the execution environment (`subnet_callback_soft_limit` / `canister_guaranteed_callback_quota` at `rs/config/src/execution_environment.rs` lines 266–276). A corresponding per-sender count method should be added to `SubnetCallContextManager`.

## Proof of Concept
The existing test `test_sign_with_threshold_key_queue_fills_up` (`rs/execution_environment/tests/threshold_signatures.rs` lines 848–905) already demonstrates the core behavior with a single canister. To demonstrate cross-canister DoS:

1. Extend the test to create two canisters (`A` and `B`) on the same subnet.
2. From canister `A`, submit `max_queue_size` (20) threshold signature requests — all are accepted.
3. From canister `B`, submit one threshold signature request — it is rejected with `"request queue for key … is full."`.
4. For the zero-cost variant: configure the subnet as NNS (`with_subnet_id(nns_subnet).with_nns_subnet_id(nns_subnet)`) and repeat — canister `A`'s requests consume zero cycles, confirming free indefinite DoS.
