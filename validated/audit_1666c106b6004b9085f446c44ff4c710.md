### Title
`ft_on_transfer` Mints ERC-20 Tokens Without Verifying Actual NEP-141 Balance Increase - (File: `engine/src/contract_methods/connector.rs` and `engine/src/engine.rs`)

---

### Summary

The `ft_on_transfer` handler in Aurora Engine mints ERC-20 tokens based solely on the `amount` field reported by the calling NEP-141 contract, without verifying that Aurora's actual NEP-141 token balance increased by that amount. This is the direct analog of Ajna M-13: a token that reports a successful transfer but withholds internal accounting updates can cause Aurora to mint unbacked ERC-20 tokens, leading to bridge insolvency for that token.

---

### Finding Description

When a NEP-141 contract calls `ft_on_transfer` on Aurora (as part of `ft_transfer_call`), the engine executes the following sequence:

**Step 1** — `ft_on_transfer` in `engine/src/contract_methods/connector.rs` reads `args.amount` from the call arguments (provided entirely by the calling NEP-141 contract) and dispatches to `receive_erc20_tokens`:

```rust
let args: FtOnTransferArgs = read_json_args(&io)?;
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [1](#0-0) 

**Step 2** — `receive_erc20_tokens` in `engine/src/engine.rs` mints exactly `args.amount` ERC-20 tokens via an EVM `mint()` call, with no balance verification:

```rust
let amount = args.amount.as_u128();
// ...
let result = self
    .call(
        &erc20_admin_address,
        &erc20_token,
        Wei::zero(),
        setup_receive_erc20_tokens_input(&recipient, amount),
        u64::MAX,
        Vec::new(),
        Vec::new(),
        handler,
    )
    .and_then(submit_result_or_err)?;
``` [2](#0-1) 

**Step 3** — `setup_receive_erc20_tokens_input` encodes a `mint(recipient, amount)` call to the ERC-20 contract, using the caller-supplied `amount` directly:

```rust
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);
    [selector, tail.as_slice()].concat()
}
``` [3](#0-2) 

**The missing invariant**: At no point does Aurora verify that its actual NEP-141 balance (i.e., the balance of the Aurora engine account on the NEP-141 contract) increased by `args.amount`. The engine trusts the reported amount entirely.

This is structurally identical to the Ajna M-13 root cause: "Flash loan safety is now controlled by `safeTransfer()` only, which internal mechanics can vary between ERC-20 tokens." Here, Aurora's bridge safety is controlled by the `amount` field in `FtOnTransferArgs` only, which is provided by the NEP-141 contract and is never cross-checked against an actual balance query.

**Direct attacker-controlled entry path**: Any NEAR account registered as a NEP-141 token in Aurora can call `ft_on_transfer` directly on Aurora with an arbitrary `amount`, without actually transferring any tokens. Aurora only checks `predecessor_account_id` against the registered NEP-141 token list:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,  // only check: is this a known NEP-141?
        &args,
        ...
    )
};
``` [4](#0-3) 

Registering a NEP-141 token in Aurora is permissionless via `deploy_erc20_token`. The `EvmErc20.sol` and `EvmErc20V2.sol` contracts confirm that the ERC-20 `mint` function is restricted to the admin (Aurora engine address), so only Aurora itself can mint — but Aurora mints based on the unverified reported amount. [5](#0-4)
