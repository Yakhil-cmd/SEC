Audit Report

## Title
Unbounded ingress-expiry loop in ICP Rosetta `construction_payloads` allows unauthenticated OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `/construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` values from JSON metadata with no range validation. Setting `ingress_start=0` and `ingress_end=u64::MAX` causes the `while` loop at lines 99–107 to execute approximately 153 billion iterations, each pushing a `u64` onto `ingress_expiries`, exhausting process memory and crashing the Rosetta server. The ICRC1 Rosetta implementation has an explicit guard for this case; the ICP path does not.

## Finding Description
`ConstructionPayloadsRequestMetadata` stores `ingress_start` and `ingress_end` as plain `Option<u64>` with no range constraints: [1](#0-0) 

These values are deserialized directly from the JSON request body and used without validation. The handler computes `interval = MAX_INGRESS_TTL(300s) − PERMITTED_DRIFT(60s) − 120s = 120s = 120,000,000,000 ns`: [2](#0-1) 

It then enters an unbounded loop with no cap on the number of iterations: [3](#0-2) 

With `ingress_start=0` and `ingress_end=u64::MAX (18446744073709551615)`, the loop runs `18446744073709551615 / 120000000000 ≈ 153,722,867,280` iterations, each pushing a `u64` (8 bytes) onto `ingress_expiries`, totaling ~1.2 TB of heap allocation before OOM kills the process.

By contrast, the ICRC1 Rosetta implementation explicitly validates the range before entering its equivalent loop: [4](#0-3) 

The ICP path has no such guard.

## Impact Explanation
Any unauthenticated HTTP client can crash the ICP Rosetta process with a single small JSON POST. Rosetta is the standard exchange-integration gateway for ICP; its crash denies service to all users of that instance (exchange deposits/withdrawals, wallet integrations) until the process is restarted. The attack is immediately repeatable after restart. This matches the **High** bounty impact: *"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant... Rosetta... security impact with concrete user or protocol harm."*

## Likelihood Explanation
The endpoint is public and requires no authentication. The exploit is a single well-formed JSON POST with two integer fields. No special knowledge beyond the Rosetta API spec is required. The `ingress_start` and `ingress_end` fields are documented in the metadata struct. The attack is trivially repeatable.

## Recommendation
Add an explicit upper bound on the computed range before entering the loop, mirroring the ICRC1 guard pattern at `rs/rosetta-api/icrc1/src/construction_api/services.rs` lines 148–158:

```rust
let max_expiries = 1000usize;
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    / interval.as_nanos() as u64 > max_expiries as u64 {
    return Err(ApiError::invalid_request("ingress window too large"));
}
```

Alternatively, cap `ingress_end` to `ingress_start + N * interval` for some small constant `N`.

## Proof of Concept
```
POST /construction/payloads
{
  "network_identifier": { ... },
  "operations": [ <any valid transfer op> ],
  "public_keys": [ <valid pk> ],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```
The Rosetta process will allocate unbounded memory in the `ingress_expiries` vec and OOM-crash before returning a response. A local integration test can confirm this by calling `construction_payloads` with these metadata values and observing process termination or an OOM error.
