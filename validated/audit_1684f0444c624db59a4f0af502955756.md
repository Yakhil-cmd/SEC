### Title
Unbounded `ingress_expiries` Loop in ICP Rosetta `construction_payloads` Allows Unauthenticated OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The `construction_payloads` handler in the ICP Rosetta node accepts user-controlled `ingress_start` and `ingress_end` values from request metadata without any bounds validation. These values drive a `while now < ingress_end` loop that pushes one entry per `interval` into an unbounded `Vec`. An unauthenticated attacker can set `ingress_start = 0` and `ingress_end = u64::MAX` to trigger ~154 billion loop iterations, exhausting process memory and crashing the Rosetta node.

---

### Finding Description

In `construction_payloads`, `ingress_start` and `ingress_end` are deserialized directly from the request metadata as raw `u64` nanosecond values with no range check: [1](#0-0) 

They are then used to drive this loop: [2](#0-1) 

The step size is:

```
interval = MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s
         = 300s - 60s - 120s = 120s = 120_000_000_000 ns
``` [3](#0-2) 

With `ingress_start = 0` and `ingress_end = u64::MAX = 18_446_744_073_709_551_615`:

```
iterations ≈ 18_446_744_073_709_551_615 / 120_000_000_000 ≈ 153,722,867,280
```

Each iteration pushes a `u64` (8 bytes) into `ingress_expiries`. That is ~1.2 TB of allocation before the loop even completes. The process OOMs and crashes well before that.

Additionally, the `Time::add_assign` implementation uses `as u64` truncation on overflow: [4](#0-3) [5](#0-4) 

This means when `now` wraps past `u64::MAX`, it resets to a small value still less than `ingress_end`, making the loop **infinite** rather than merely very long — the process will OOM regardless.

The `ingress_expiries` vec is then passed into `add_payloads`, which iterates it again, multiplying the allocation: [6](#0-5) 

**No validation exists in the ICP Rosetta handler.** The ICRC1 Rosetta implementation of the same function has the correct guards: [7](#0-6) 

The ICP version is missing equivalent checks entirely.

---

### Impact Explanation

An unauthenticated attacker sends a single HTTP POST to `/construction/payloads` with `metadata.ingress_start = 0` and `metadata.ingress_end = 18446744073709551615`. The Rosetta process enters an effectively infinite loop allocating memory until the OS OOM-killer terminates it. This takes down the ICP Rosetta node, denying service to all users (exchanges, wallets, custodians) relying on it for ICP transfers and neuron management.

---

### Likelihood Explanation

- No authentication or rate limiting is required to reach this endpoint.
- The payload is a small, valid JSON body.
- The attack is repeatable: the node restarts and can be immediately crashed again.
- The root cause is a missing two-line bounds check that the ICRC1 sibling already has.

---

### Recommendation

Add an upper bound on the ingress window before the loop, mirroring the ICRC1 implementation:

```rust
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request("ingress_start must be before ingress_end"));
}
let max_window = Duration::from_secs(24 * 3600); // or a tighter bound
if ingress_end.saturating_duration_since(ingress_start) > max_window {
    return Err(ApiError::invalid_request("ingress window too large"));
}
```

Alternatively, cap `ingress_expiries` to a fixed maximum count (e.g., 100 entries) and return an error if the window would exceed it.

---

### Proof of Concept

```bash
curl -X POST http://<rosetta-node>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<any>"},"amount":{"value":"-1","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<any valid pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

The Rosetta process will exhaust memory and crash. Fuzz assertion: `assert!(response.payloads.len() <= MAX_EXPECTED_PAYLOADS)` will fail for any `ingress_end - ingress_start > interval`.
