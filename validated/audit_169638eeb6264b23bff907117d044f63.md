### Title
Stale Maturity Modulation Rate Applied to Neuron Spawning and Disbursement Without Staleness Guard - (File: rs/nns/governance/src/governance.rs)

### Summary
`maybe_spawn_neurons` in NNS Governance reads `heap_data.maturity_modulation.current_value_permyriad` and applies it to mint ICP for spawned neurons, but never checks the companion `updated_at_days_since_epoch` field. If the `UpdateIcpXdrRateRelatedData` daily timer task fails to refresh the rate for an extended period, an arbitrarily stale modulation factor is silently applied to every neuron that matures during that window, causing users to receive more or fewer ICP than the current market price warrants.

### Finding Description

Proposal 141779 (2026-05-17) switched neuron spawning and maturity disbursement from the CMC-polled `cached_daily_maturity_modulation_basis_points` to a locally computed `MaturityModulation` struct that is populated by the `UpdateIcpXdrRateRelatedData` recurring async task.

The struct carries two fields:

```
pub struct MaturityModulation {
    pub current_value_permyriad: Option<i32>,   // the rate
    pub updated_at_days_since_epoch: Option<u64>, // when it was last computed
}
``` [1](#0-0) 

`maybe_spawn_neurons` reads only `current_value_permyriad` and performs a range check, but never inspects `updated_at_days_since_epoch`:

```rust
let maturity_modulation = match self
    .heap_data
    .maturity_modulation
    .as_ref()
    .and_then(|m| m.current_value_permyriad)
{
    None => return,
    Some(value) => value,
};

if !VALID_MATURITY_MODULATION_BASIS_POINTS_RANGE.contains(&maturity_modulation) {
    // log and return
    return;
}
``` [2](#0-1) 

The same pattern applies in `finalize_maturity_disbursement`, which receives the modulation value from the same field and applies it via `apply_maturity_modulation`: [3](#0-2) 

The `UpdateIcpXdrRateRelatedData` task is the sole writer of this field. It fetches one day's ICP/XDR rate from the Exchange Rate Canister (XRC) per timer tick, with a 60-second retry on failure: [4](#0-3) 

If XRC is persistently unavailable (network partition, canister freeze, or a sustained `GetExchangeRateError`), the task returns `ERROR_RETRY_INTERVAL_SECONDS` (60 s) and loops without updating the buffer. `update_maturity_modulation` is only called once the buffer is considered complete for the day: [5](#0-4) 

Because `maybe_spawn_neurons` never checks `updated_at_days_since_epoch`, it will apply a modulation value that may be days or weeks old to every neuron that reaches its `spawn_at_timestamp_seconds` during the outage window.

The initial seed value is `current_value_permyriad: Some(0)` with `updated_at_days_since_epoch: None`: [6](#0-5) 

This means a freshly upgraded canister whose XRC backfill has not yet completed will spawn all pending neurons at a neutral 0-permyriad modulation regardless of actual market conditions.

### Impact Explanation

Every neuron whose `spawn_at_timestamp_seconds` falls within the staleness window is minted with the wrong ICP amount. The modulation range is bounded at −10 % / +2 % (`MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70` = −1 000, `MATURITY_MODULATION_MAX_PERMYRIAD_MISSION_70` = 200): [7](#0-6) 

In the worst case a stale +2 % modulation is applied during a period when the true modulation should be −10 %, causing a 12-percentage-point over-mint of ICP per spawned neuron. Because `apply_maturity_modulation` mints real ICP via the ledger, this is a ledger conservation issue: ICP is created in excess of what the protocol intends, diluting all holders. The inverse (under-mint) harms the neuron owner. Neither outcome is detectable at spawn time because no staleness guard exists.

### Likelihood Explanation

The XRC is an external canister dependency. Sustained XRC unavailability (canister freeze, subnet stall, or a persistent `GetExchangeRateError`) lasting more than one day is sufficient to leave `updated_at_days_since_epoch` behind `current_day`. The retry interval is only 60 seconds, so a multi-day gap requires a multi-day outage, which is uncommon but not impossible. Additionally, on a fresh canister upgrade the 365-day backfill takes ~30 minutes; any neurons that mature during that window are spawned at the seed value of 0 regardless of market conditions. Any unprivileged user who has a neuron with `spawn_at_timestamp_seconds` set can trigger this path simply by waiting.

### Recommendation

Before applying the modulation in `maybe_spawn_neurons` (and in the disbursement path), verify that `updated_at_days_since_epoch` is no more than one day behind `current_day`. If the value is stale, defer spawning until the timer task has produced a fresh value, or fall back to 0 permyriad explicitly with a log entry. Concretely:

```rust
let current_day = now_seconds / ONE_DAY_SECONDS;
let modulation_day = self.heap_data.maturity_modulation
    .as_ref()
    .and_then(|m| m.updated_at_days_since_epoch)
    .unwrap_or(0);

if current_day.saturating_sub(modulation_day) > ACCEPTABLE_STALENESS_DAYS {
    println!("{}Maturity modulation is stale (day {}), deferring spawn.", LOG_PREFIX, modulation_day);
    return;
}
```

### Proof of Concept

1. XRC becomes persistently unavailable (e.g., canister frozen on the XRC subnet).
2. `UpdateIcpXdrRateRelatedData::execute` loops every 60 s, each time returning `ERROR_RETRY_INTERVAL_SECONDS` without updating `icp_price_history` or `maturity_modulation`.
3. `maturity_modulation.updated_at_days_since_epoch` remains at day D while `current_day` advances to D+N.
4. An unprivileged user whose neuron has `spawn_at_timestamp_seconds ≤ now` calls no special endpoint; the governance periodic task `maybe_spawn_neurons` fires automatically.
5. `maybe_spawn_neurons` reads `current_value_permyriad` (the stale value from day D), passes the range check, and calls `apply_maturity_modulation(original_maturity, stale_value)`.
6. The ledger mints `original_maturity * (1 + stale_value / 10_000)` ICP — potentially 12 % more or less than the protocol intends — with no error or revert. [8](#0-7) [9](#0-8)

```rust
pub fn apply_maturity_modulation(
    amount_maturity_e8s: u64,
    maturity_modulation_basis_points: i32,
) -> Result<u64, String> {
    let amount_e8s = u128::from(amount_maturity_e8s);

    let adjusted_maturity_modulation_basis_points = saturating_add_or_subtract_u128_i32(
        BASIS_POINTS_PER_UNITY,
        maturity_modulation_basis_points,
    );

    let modulated_amount_e8s: u128 = amount_e8s
        .checked_mul(adjusted_maturity_modulation_basis_points)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?
        .checked_div(BASIS_POINTS_PER_UNITY)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?;

    u64::try_from(modulated_amount_e8s).map_err(|err| err.to_string())
}
