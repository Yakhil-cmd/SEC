Looking at the actual code, I need to trace each component of this claim carefully.

**1. `check_transaction` access control**

The `check_transaction` update method has no caller restriction — only a cycles check: [1](#0-0) [2](#0-1) 

Any principal can call it as long as they attach sufficient cycles.

**2. FetchGuard slot exhaustion**

`FetchGuard::new` decrements `OUTCALL_CAPACITY` and returns `NoCapacity` when it reaches 0: [3](#0-2) 

The guard is held for the entire duration of the HTTP outcall (dropped only when `fetch_tx` completes). An attacker sending 50 concurrent calls with uncached txids can hold all slots simultaneously.

**3. HighLoad path in the checker**

When `new_fetch_guard` fails, `try_fetch_tx` returns `TryFetchResult::HighLoad`: [4](#0-3) 

This propagates to `CheckTransactionRetriable::HighLoad` in `check_transaction_inputs`: [5](#0-4) 

**4. Minter's handling of HighLoad — the critical path**

In `check_utxo`, the retry loop only retries for `NotEnoughCycles`. For `Retriable` (which includes `HighLoad`), it **immediately returns an error** without retrying: [6](#0-5) 

This means a `HighLoad` response from the checker causes `update_balance` to return `UpdateBalanceError::TemporarilyUnavailable` to the user immediately.

**5. The "targeted" claim is inaccurate**

The attacker cannot selectively block specific txids. Exhausting all 50 slots causes general high load affecting all callers equally — the minter and all other users are affected indiscriminately. There is no mechanism to target a specific user's deposit txid.

### Title
Unprivileged Cycle-Funded DoS via OUTCALL_CAPACITY Exhaustion Causes Minter to Return TemporarilyUnavailable — (`rs/bitcoin/checker/src/state.rs`, `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

### Summary
An unprivileged attacker can exhaust all 50 `OUTCALL_CAPACITY` slots in the BTC checker canister by sending 50 concurrent `check_transaction` calls with uncached txids and sufficient cycles. While slots are occupied, the minter's `check_utxo` receives `HighLoad` and immediately returns `UpdateBalanceError::TemporarilyUnavailable` to users without retrying, temporarily blocking ckBTC minting for all users.

### Finding Description
`check_transaction` is a public update method with no caller restriction beyond a cycles payment. `FetchGuard::new` decrements a shared `OUTCALL_CAPACITY` counter; when it reaches zero, all subsequent calls return `HighLoad`. HTTP outcalls can take several seconds, so an attacker holding 50 guards simultaneously blocks the checker for the duration. The minter's `check_utxo` loop retries only on `NotEnoughCycles`; on `Retriable(HighLoad)` it immediately propagates `TemporarilyUnavailable` to the `update_balance` caller. [7](#0-6) [8](#0-7) 

### Impact Explanation
All users calling `update_balance` during the attack window receive `TemporarilyUnavailable` and must retry manually. The attack is **not targeted** — it is a general DoS against the checker canister. It is also **temporary**: once the attacker's HTTP outcalls complete (success or failure), `FetchGuard::drop` restores capacity and normal operation resumes. The attacker must continuously send new calls to sustain the DoS, incurring ongoing cycle costs. [9](#0-8) 

### Likelihood Explanation
The attack requires sustained cycle expenditure (50 × `CHECK_TRANSACTION_CYCLES_REQUIRED` per HTTP outcall round-trip, repeated continuously). This is an economic deterrent but not a hard barrier. The checker canister is publicly accessible on mainnet. The attack is self-healing and does not cause permanent state corruption.

### Recommendation
1. Add a retry on `Retriable(HighLoad)` in `check_utxo` (with bounded retries and backoff), mirroring the `NotEnoughCycles` retry logic.
2. Consider per-caller rate limiting or a minimum cycle cost that makes sustained slot exhaustion economically prohibitive.
3. Increase `OUTCALL_CAPACITY` or implement per-caller slot quotas to reduce the blast radius.

### Proof of Concept
A state-machine test can:
1. Acquire 50 `FetchGuard`s simultaneously (one per distinct uncached txid).
2. Assert that `FetchGuard::new` returns `Err(FetchGuardError::NoCapacity)` on the 51st attempt.
3. Simulate a minter `check_utxo` call receiving `CheckTransactionRetriable::HighLoad` and assert it returns `UpdateBalanceError::TemporarilyUnavailable` without retrying. [10](#0-9) [6](#0-5)
