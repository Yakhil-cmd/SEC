### Title
Stale Storage Persists Across CREATE2 Redeployment at Colliding Address — (`basic_system/src/system_implementation/flat_storage_model/account_cache.rs`)

---

### Summary

When a contract is selfdestructed in a transaction where it was **not** created (post-EIP-6780 semantics), ZKsync OS correctly skips clearing its storage. However, when a **new** contract is subsequently deployed at the same address via CREATE2, ZKsync OS also fails to clear the old storage. The new contract silently inherits all storage slots written by the previous deployment. This is an EVM semantic mismatch: the EVM specification requires that a new contract creation at a colliding address begins with empty storage. The project itself acknowledges this gap by disabling the relevant EVM test suite entries.

---

### Finding Description

The `deploy_code` function in `flat_storage_model/account_cache.rs` updates account properties (bytecode hash, versioning data, etc.) but **never clears the storage** of the target address: [1](#0-0) 

Storage clearing only occurs inside `finish_tx`, and only when `is_marked_for_deconstruction` is `true`: [2](#0-1) 

`is_marked_for_deconstruction` is only set to `true` inside `mark_for_deconstruction` when `should_be_deconstructed` is `true`, which requires the contract to have been **deployed in the current transaction** (`deployed_in_tx == Some(cur_tx)`) or to be executing inside a constructor: [3](#0-2) 

For a **pre-existing** contract (deployed in a prior transaction), `should_be_deconstructed` is `false`, so `is_marked_for_deconstruction` is never set, and the storage is never cleared — neither at selfdestruct time nor at the subsequent CREATE2 redeployment.

The same pattern exists in the Ethereum storage model: [4](#0-3) [5](#0-4) 

The project explicitly acknowledges this gap by disabling the relevant EVM conformance tests: [6](#0-5) [7](#0-6) 

The same entries are disabled in the stable test index: [8](#0-7) 

---

### Impact Explanation

A new contract deployed via CREATE2 at an address that previously held a selfdestructed contract inherits all of the old contract's storage slots. The new contract's logic operates on stale, attacker-pre-seeded data rather than a clean slate. Concrete consequences:

1. **EVM semantic mismatch / state divergence**: ZKsync OS execution diverges from canonical EVM. A prover or verifier using standard EVM semantics will compute a different post-state root, making valid ZKsync OS blocks unprovable or causing proof rejection.
2. **Incorrect contract execution**: A new contract reading slot `k` expects zero (fresh deployment) but receives the value written by the previous occupant. This can corrupt token balances, access-control mappings, counters, or any other state the new contract relies on being zero-initialized.
3. **Attacker-controlled state injection**: An attacker who controls the deployer factory can pre-populate storage at a deterministic CREATE2 address, selfdestruct, and then cause a victim factory to redeploy at the same address, giving the attacker influence over the new contract's initial state.

---

### Likelihood Explanation

The attack requires:
1. Knowledge of a CREATE2 salt and deployer address (trivially achievable when the attacker controls the deployer, or when the deployer is a public factory).
2. Two transactions: one to deploy-and-selfdestruct, one to redeploy.

No privileged role, governance key, or oracle manipulation is required. Any unprivileged user who can call a CREATE2 factory can trigger this. The scenario is realistic in DeFi deployments where factory contracts redeploy at deterministic addresses after upgrades or migrations.

---

### Recommendation

In `deploy_code` (and the equivalent in the Ethereum storage model), before writing new bytecode metadata to an address, explicitly clear all storage slots at that address if the account already has a non-trivial storage root. Concretely, call `storage.clear_state_impl(at_address)` at the start of `deploy_code` when the account being deployed to is not freshly created (i.e., `cache_appearance.is_new_element()` is `false`). This mirrors the behavior already implemented for same-transaction selfdestruct in `finish_tx`.

### Proof of Concept

```
Block 1, Tx 1:
  - Attacker calls factory.deploy(salt=S)
    → CREATE2 deploys ContractA at address X
    → ContractA constructor writes slot[0] = 0xdeadbeef
    → ContractA calls SELFDESTRUCT(beneficiary=attacker)
    → Post-EIP-6780: balance transferred, storage NOT cleared (pre-existing? No — same tx, so storage IS cleared here)

Revised scenario (cross-transaction):
Block 1, Tx 1:
  - Attacker deploys ContractA at address X via CREATE2(salt=S)
  - ContractA writes slot[0] = 0xdeadbeef (SSTORE)
  - Tx 1 ends; storage slot[0]=0xdeadbeef is committed to state

Block 1, Tx 2:
  - Attacker calls ContractA.selfdestruct(beneficiary=attacker)
  - deployed_in_tx != cur_tx → should_be_deconstructed = false
  - Balance transferred; storage NOT cleared (EIP-6780 + ZKsync OS behavior)
  - Tx 2 ends; slot[0]=0xdeadbeef still in state tree

Block 2, Tx 1:
  - Victim calls factory.deploy(salt=S) → CREATE2 deploys ContractB at address X
  - deploy_code() writes new bytecode hash, does NOT call clear_state_impl
  - ContractB reads slot[0] → returns 0xdeadbeef (stale attacker value)
  - ContractB logic is corrupted by attacker-controlled initial state
```

The disabled test `create2collisionStorageParis.json` in both `develop-state-tests.yaml` and `stable-state-tests.yaml` directly exercises this exact scenario and confirms the divergence is present and unresolved. [6](#0-5) [3](#0-2) [2](#0-1) [1](#0-0)
