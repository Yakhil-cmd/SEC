Audit Report

## Title
Fee-on-Transfer ERC20 Deposit Mints Unbacked ckERC20 Tokens - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol)

## Summary
The `depositErc20` function in `DepositHelperWithSubaccount.sol` emits the caller-supplied `amount` in the `ReceivedEthOrErc20` event rather than the actual tokens received by the minter address. The IC ckETH minter parses this event value and mints exactly `amount` ckERC20 tokens on the IC. For any fee-on-transfer ERC20 token added as a supported ckERC20 token, the minter receives `amount - fee` tokens on Ethereum but mints `amount` ckERC20 on the IC, permanently breaking the 1:1 backing invariant and causing unbacked token issuance that compounds with every deposit.

## Finding Description
In `DepositHelperWithSubaccount.sol` lines 511–532, `depositErc20` calls `erc20Token.safeTransferFrom(msg.sender, minterAddress, amount)` and then emits `ReceivedEthOrErc20(..., amount, ...)` using the caller-supplied `amount` directly — not the actual balance delta at `minterAddress`. For a fee-on-transfer token, the minter receives `amount - fee` but the event records `amount`.

The IC minter's `ReceivedEthOrErc20LogParser::parse_log()` in `src/eth_logs/parser.rs` lines 127–149 reads `value_bytes` from the event data and constructs `Erc20Value::from_be_bytes(value_bytes)` — this is the emitted nominal value, with no on-chain verification of actual receipt.

In `src/deposit.rs` lines 73–81, the minter calls `client.transfer(TransferArg { amount: event.value(), ... })`, minting exactly the emitted value. The `value()` method in `src/eth_logs/mod.rs` lines 205–210 returns `evt.value` directly from the parsed log. The internal balance accounting in `src/state.rs` lines 332–338 also records `event.value` (the emitted amount), so the minter's own `erc20_balances` state diverges from on-chain reality by the cumulative fee amount.

No existing guard in the Solidity contract, the log parser, or the minting logic checks whether the actual received amount matches the emitted amount. The minter's token whitelist is enforced by NNS governance, not by any technical fee-on-transfer detection.

## Impact Explanation
This is a significant ck-token / Chain Fusion security impact with concrete protocol harm: illegal minting of ckERC20 tokens breaks the 1:1 backing invariant. The minter's Ethereum address holds fewer ERC20 tokens than the outstanding ckERC20 supply, growing with every deposit. Eventually the minter cannot fulfill withdrawal requests, causing loss of user funds. This matches the allowed High impact: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation
Exploitation requires a fee-on-transfer ERC20 token to be added as a supported ckERC20 token via NNS governance — a meaningful precondition. However, this is a normal operational action (adding a new token), not an adversarial attack. Once such a token is added, any unprivileged user can trigger the discrepancy by calling `depositErc20`. Several real tokens (USDT has dormant fee parameters in its contract; PAXG, STA, and others charge active fees) could be proposed. The NNS governance process has no technical check for fee-on-transfer behavior, relying entirely on human review. No special privileges are required beyond the governance decision.

## Recommendation
In `depositErc20`, measure the actual received amount by comparing the minter's ERC20 balance before and after the transfer, and emit that value:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 receivedAmount = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, receivedAmount, principal, subaccount);
```

Additionally, the NNS governance checklist for adding new ckERC20 tokens should explicitly require verification that the ERC20 contract does not implement transfer fees.

## Proof of Concept
1. Deploy or identify a fee-on-transfer ERC20 token `FeeToken` (e.g., 1% fee per transfer).
2. Submit and pass an NNS proposal to add `FeeToken` as a supported ckERC20 token.
3. Alice approves the helper contract for `1_000_000` `FeeToken` units.
4. Alice calls `depositErc20(FeeToken, 1_000_000, alicePrincipal, subaccount)`.
5. `safeTransferFrom` executes; `FeeToken` deducts 1% fee; minter receives `990_000` tokens.
6. Helper emits `ReceivedEthOrErc20(FeeToken, Alice, 1_000_000, alicePrincipal, subaccount)`.
7. IC minter parses `value = 1_000_000` from the log via `ReceivedEthOrErc20LogParser::parse_log`.
8. Minter calls `client.transfer(TransferArg { amount: 1_000_000, ... })`, minting `1_000_000` ckFeeToken to Alice.
9. Minter holds `990_000` FeeToken on Ethereum; `1_000_000` ckFeeToken exist on IC — `10_000` token shortfall per deposit.
10. After 100 deposits: minter holds `99_000_000` FeeToken, `100_000_000` ckFeeToken outstanding — `1_000_000` token shortfall. Withdrawals begin failing.

A deterministic integration test can reproduce this using PocketIC with a mock ERC20 that deducts a fee in `transferFrom`, verifying that `erc20_balances` in minter state exceeds the mock contract's recorded balance for the minter address after a deposit.
