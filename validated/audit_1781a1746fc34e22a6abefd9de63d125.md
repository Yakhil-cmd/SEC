### Title
Owner Key Loss While Paused Causes Permanent Freezing of All EVM User Funds - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The Aurora Engine's `resume_contract` function is exclusively gated behind `require_owner_only`. Simultaneously, `set_owner` is gated behind both `require_owner_only` and `require_running`. This creates an irrecoverable deadlock: if the contract is paused and the owner loses access to their key, there is no mechanism for any party to unpause the engine or transfer ownership. All EVM execution entry points (`submit`, `submit_with_args`, `call`, `deploy_code`) check `require_running` and will permanently revert, freezing all user funds held in the Aurora EVM.

`resume_contract` requires `require_owner_only` at line 266. There is no time-lock, no multisig fallback, and no alternative unpausing path.

Critically, `set_owner` — the only way to change the owner — also requires `require_running`: [3](#0-2)

### Impact Explanation

**Critical — Permanent freezing of funds.**

If the contract is paused and the owner's key becomes inaccessible (loss, incapacitation, compromise, or destruction), every user's ETH and ERC-20 token balance held inside the Aurora EVM is permanently frozen. No EVM transaction can execute. No withdrawal, transfer, or bridge-out is possible. The `set_owner` recovery path is itself blocked by `require_running`, closing the only administrative escape route.

### Likelihood Explanation

The owner pausing the contract is an intended operational action (e.g., emergency response to an exploit). The scenario requires only that the owner's key becomes inaccessible after a pause — a realistic operational risk for any long-running production system. The Aurora engine is deployed at the `aurora` account on NEAR Mainnet, making this a production-grade risk affecting real user funds.

### Recommendation

Remove the `require_running` guard from `set_owner` so that ownership can be transferred even while the contract is paused. This breaks the deadlock and allows a new owner to call `resume_contract`:

```rust
// In set_owner: remove require_running(&state)?;
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        // require_running(&state)?;  <-- remove this line
        require_owner_only(&state, &env.predecessor_account_id())?;
        ...
    })
}
```

Alternatively, add a time-based fallback to `resume_contract` (analogous to the Fantom recommendation) that allows any caller to unpause after a sufficiently long delay has elapsed since the pause was set, ensuring the system cannot be permanently locked.

### Proof of Concept

1. Owner calls `pause_contract` → `state.is_paused = true` is written to storage. [6](#0-5) 

2. Owner's NEAR key becomes inaccessible.

3. Any user calls `submit` with a valid EVM transaction → `require_running(&state)?` returns `Err(ERR_PAUSED)` → transaction reverts. [7](#0-6) 

4. Any party attempts `set_owner` to install a new owner → `require_running(&state)?` at line 108 returns `Err(ERR_PAUSED)` → reverts. [8](#0-7) 

5. Any party attempts `resume_contract` → `require_owner_only` at line 266 returns `Err(ERR_NOT_ALLOWED)` → reverts. [9](#0-8) 

6. No further state transition is possible. All user funds in the Aurora EVM are permanently inaccessible.
