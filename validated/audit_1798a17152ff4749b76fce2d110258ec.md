Audit Report

## Title
Unbounded Unvalidated Pool Growth via Deferred Remote DKG Dealings with No Size Cap — (`rs/consensus/dkg/src/lib.rs`)

## Summary

In `validate_dealings_for_dealer`, remote DKG dealings whose `NiDkgId` has no matching config are silently deferred by returning `Mutations::new()` — no removal, no invalidation, no signature check. Combined with `DkgPoolImpl::insert()` having no size guard and the DKG channel being registered with `SLOT_TABLE_NO_LIMIT = usize::MAX`, a single Byzantine subnet member can flood the unvalidated pool with arbitrarily many distinct remote DKG dealings within one DKG interval, causing unbounded memory growth and potential OOM on honest replicas.

## Finding Description

**Deferred path returns without removal or verification:**

In `validate_dealings_for_dealer`, when `configs.get(message_dkg_id)` returns `None` and `message_dkg_id.target_subnet.is_remote()` is `true`, the function immediately returns `Mutations::new()`: [1](#0-0) 

No signature verification, no dealer eligibility check, no `HandleInvalid`, no `RemoveFromUnvalidated` is issued. The dealing remains in the unvalidated pool indefinitely until the next DKG interval purge.

**Purge only fires at interval boundary:**

The purge only triggers when `start_height > dkg_pool.get_current_start_height()`, i.e., only when a new DKG summary block is produced (~500 blocks later): [2](#0-1) 

**No size guard in `DkgPoolImpl::insert()`:**

The pool insert is unconditional — it inserts directly into a `PoolSection` (a `BTreeMap`) with no capacity check: [3](#0-2) 

**`DkgBouncer` only filters by height:**

The bouncer accepts any dealing at the current `start_height` regardless of content, `target_subnet` type, or sender: [4](#0-3) 

**DKG channel registered with `SLOT_TABLE_NO_LIMIT`:**

The DKG artifact channel is registered with `SLOT_TABLE_NO_LIMIT = usize::MAX`, meaning there is no per-peer cap on how many distinct artifact IDs a peer can advertise: [5](#0-4) [6](#0-5) 

The slot table enforces the limit at `peer_slot_table_len < self.slot_limit` — with `usize::MAX`, this is never triggered: [7](#0-6) 

**`DkgMessageId` is keyed by `crypto_hash(message)`:**

Each message with a distinct `NiDkgTargetId` (a 32-byte field) produces a unique pool entry: [8](#0-7) 

**`NiDkgTargetId` is an unconstrained 32-byte array:** [9](#0-8) 

**Existing test explicitly confirms deferred dealings remain in the pool:**

The test `test_remote_dealing_validation_is_deferred_until_context_exists` confirms that deferred remote dealings remain in the unvalidated pool and are only cleared at the next DKG interval: [10](#0-9) 

**Exploit flow:**

1. Byzantine subnet member (single node, below fault threshold) crafts N distinct DKG messages with `target_subnet = NiDkgTargetSubnet::Remote(distinct_id_i)` for i=1..N, each with the current `start_block_height`. No valid signature or dealing bytes are required since the deferred path never reaches `crypto_validate_dealing`.
2. Advertises all N artifacts via P2P slot table (no limit for DKG).
3. Each message passes the `DkgBouncer` height check and is inserted into the unvalidated pool.
4. `validate_dealings_for_dealer` returns `Mutations::new()` for each — no removal occurs.
5. Pool grows without bound until the next DKG interval purge (~500 blocks).

## Impact Explanation

A single Byzantine subnet member can exhaust heap memory on honest replicas within one DKG interval by injecting millions of distinct remote DKG dealings. Each `NiDkgDealing` contains BLS12-381 crypto material (several KB). Sustained injection causes OOM kills, crashing honest replicas and stalling the subnet. This matches the allowed impact: **Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS** — **High ($2,000–$10,000)**.

## Likelihood Explanation

- **Attacker prerequisite**: Must be a registered subnet member (Byzantine node below fault threshold). IC P2P uses mutual TLS with node certificates. A single compromised or malicious node suffices.
- **Ease of exploit**: Trivial — vary the 32-byte `NiDkgTargetId`. No valid crypto material required since signature and dealing verification are never reached for the deferred path.
- **No rate limiting**: `SLOT_TABLE_NO_LIMIT` for DKG means no per-peer cap exists.
- **Bounded window**: Pool is purged at each DKG interval (~500 blocks, ~500 seconds at 1s block time). This is a partial mitigation but does not prevent within-interval exhaustion.

## Recommendation

1. **Add a pool size cap** in `DkgPoolImpl::insert()` or in `on_state_change` before processing unvalidated dealings, rejecting new entries once a configurable limit is reached.
2. **Apply a slot table limit for DKG** analogous to `SLOT_TABLE_LIMIT_INGRESS` (50,000) rather than `SLOT_TABLE_NO_LIMIT`.
3. **Reject deferred remote dealings after a configurable count** per `(dealer_id, start_height)` tuple, or cap the number of distinct unknown remote target IDs.
4. **Consider issuing `HandleInvalid`** for remote dealings whose `target_subnet` remote ID is not plausibly associated with any pending `SetupInitialDKG` or `ReshareChainKey` context visible in the certified state, rather than deferring unconditionally.

## Proof of Concept

```rust
// Byzantine subnet member crafts N distinct remote DKG dealings
let start_height = dkg_pool.get_current_start_height();
for i in 0u64..1_000_000 {
    let mut id_bytes = [0u8; 32];
    id_bytes[..8].copy_from_slice(&i.to_le_bytes());
    let target_id = NiDkgTargetId::new(id_bytes); // distinct IDs
    let dkg_id = NiDkgId {
        start_block_height: start_height,
        dealer_subnet: attacker_subnet_id,
        dkg_tag: NiDkgTag::LowThreshold,
        target_subnet: NiDkgTargetSubnet::Remote(target_id), // unknown config
    };
    // No valid signature needed — deferred path never calls crypto_validate_dealing
    let msg = craft_dealing_with_arbitrary_bytes(dkg_id);
    dkg_pool.insert(UnvalidatedArtifact {
        message: msg,
        peer_id: attacker_node_id,
        timestamp: now,
    });
}

// Each on_state_change call returns Mutations::new() for all deferred messages
for _ in 0..100 {
    let changeset = dkg_impl.on_state_change(&dkg_pool);
    // No HandleInvalid, no RemoveFromUnvalidated for deferred remote dealings
    assert!(changeset.iter().all(|a| !matches!(a, ChangeAction::HandleInvalid(..))));
}
// Pool size remains at 1_000_000 — unbounded accumulation confirmed
assert_eq!(dkg_pool.get_unvalidated().count(), 1_000_000);
```

A deterministic integration test can be written extending `test_remote_dealing_validation_is_deferred_until_context_exists` to insert N distinct `NiDkgTargetId` values and assert the pool count grows proportionally with no removals across multiple `on_state_change` calls.
