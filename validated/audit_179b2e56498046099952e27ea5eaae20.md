### Title
Truncation in `native_per_pubdata` Calculation Allows Zero-Cost Pubdata Consumption - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When `pubdata_price < native_price`, the integer floor division used to compute `native_per_pubdata` truncates to zero. This means any L2 transaction can consume arbitrarily large amounts of pubdata without being charged any native resource for it, while the protocol still bears the full L1 data-availability cost. The root cause is an asymmetric rounding policy: `native_per_gas` uses `div_ceil` (always ≥ 1 when price > 0), but `native_per_pubdata` uses `wrapping_div` (floor), which silently rounds to 0.

---

### Finding Description

In `validation_impl.rs`, the per-pubdata-byte native cost is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

By contrast, `native_per_gas` is computed with ceiling division:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), `wrapping_div` yields `native_per_pubdata = 0`.

This zero propagates into two critical paths:

**1. Intrinsic pubdata overhead in `create_resources_for_tx`:**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
``` [3](#0-2) 

With `native_per_pubdata_byte = 0`, `intrinsic_pubdata_overhead = 0`, so no native is deducted for intrinsic pubdata.

**2. Execution pubdata charging in `get_resources_to_charge_for_pubdata`:**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

With `native_per_pubdata = 0`, `native = 0` regardless of how many pubdata bytes were written. `check_enough_resources_for_pubdata` then always returns `true`, and the transaction is never reverted for pubdata costs.

The same truncation pattern exists for L1 transactions in `process_l1_transaction.rs` via `prepare_and_check_resources`, and is also mirrored in the public API helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [5](#0-4) 

---

### Impact Explanation

When `native_per_pubdata = 0`:

- The transaction's native resource budget is derived solely from gas (`native_per_gas * gas_limit`), with zero contribution from pubdata.
- `check_enough_resources_for_pubdata` always passes, so the post-execution pubdata revert path is never triggered.
- A transaction can write to as many storage slots as its EVM gas allows (e.g., ~500 SSTOREs at 20,000 gas each with a 10M gas limit = ~16 KB of pubdata) and pay **zero** native resource for any of it.
- The protocol must still publish this pubdata to L1, absorbing the full data-availability cost without compensation.
- This enables systematic pubdata spam: an attacker submits storage-write-heavy transactions whenever `pubdata_price < native_price`, forcing the protocol to subsidize L1 costs.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are independent operator-set block parameters. `native_price` reflects the cost per RISC-V proving cycle (typically 10 in tests and documentation). `pubdata_price` reflects the cost per byte of L1 calldata, which fluctuates with L1 gas prices. During periods of low L1 gas prices, `pubdata_price` can legitimately fall below `native_price`, making this condition reachable without any privileged access. An unprivileged attacker only needs to monitor block metadata and submit pubdata-heavy transactions when the condition holds.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — can produce 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — matches native_per_gas policy):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `process_l1_transaction.rs` (`prepare_and_check_resources`) and `api/src/helpers.rs`. Alternatively, add an explicit guard: if `pubdata_price > 0` and `native_per_pubdata == 0`, reject the transaction or clamp to 1.

---

### Proof of Concept

**Setup**: Block context with `native_price = 10`, `pubdata_price = 5` (i.e., `pubdata_price < native_price`).

**Step 1**: Attacker submits an L2 EIP-1559 transaction calling a contract that performs 500 `SSTORE` operations (each writing a new non-zero value to a distinct slot).

**Step 2**: During validation in `validate_and_compute_fee_for_transaction`:
- `native_per_gas = ceil(gas_price / 10)` → non-zero, normal native limit allocated.
- `native_per_pubdata = floor(5 / 10) = 0`.

**Step 3**: During `create_resources_for_tx`:
- `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0` → no native deducted for intrinsic pubdata.

**Step 4**: After execution, `get_resources_to_charge_for_pubdata` computes:
- `native = 500_slots * 32_bytes * 0 = 0`.
- `check_enough_resources_for_pubdata` returns `true`.

**Step 5**: Transaction succeeds. ~16 KB of pubdata is published to L1. Attacker paid only EVM gas (~10M gas × gas_price), zero native for pubdata. Protocol absorbs the full L1 data-availability cost.
