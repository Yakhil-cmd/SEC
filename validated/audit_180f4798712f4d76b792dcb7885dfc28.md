### Title
Canister HTTP Outcall with `PayAsYouGo` Pricing Causes Permanent Cycles Loss for Calling Canister - (File: `rs/https_outcalls/client/src/client.rs`)

### Summary

A canister that submits an HTTP outcall using the `FlexibleHttpRequest` management API (which forces `PayAsYouGo` pricing) will have its entire payment taken upfront by the execution layer, but the HTTPS outcalls adapter unconditionally rejects every such request with `SysFatal`. The calling canister loses the base fee permanently with no recourse, analogous to the PuttyV2 M-03 pattern where a parameter set at one stage causes a later stage to always fail, consuming the user's upfront payment for nothing.

### Finding Description

The `FlexibleHttpRequest` management canister endpoint always assigns `PricingVersion::PayAsYouGo` to the resulting `CanisterHttpRequestContext`. Under `PayAsYouGo` pricing, the execution environment takes the **entire payment** out of the request upfront at the time the context is created, storing only a `refundable_cycles` amount to be returned later.

However, in `rs/https_outcalls/client/src/client.rs`, the HTTPS outcalls adapter client unconditionally checks whether the request uses `PayAsYouGo` pricing and, if so, immediately rejects it with `RejectCode::SysFatal` and the message `"Canister HTTP request with PayAsYouGo pricing is not supported"`. The `budget.create_payment_receipt()` it sends back claims zero refund (the default `CanisterHttpPaymentReceipt`), so the refundable cycles that were set aside are never returned to the canister.

The execution layer accepted the request and deducted the base fee (non-refundable) plus set aside the refundable portion. The adapter then rejects the request without honoring the refund, causing the calling canister to lose the refundable cycles permanently.

**Step-by-step root cause:**

1. A canister calls `FlexibleHttpRequest` with a payment of `P` cycles.
2. In `try_add_http_context_to_replicated_state` (`rs/execution_environment/src/execution_environment.rs`), the `PayAsYouGo` branch takes the entire payment out: `canister_http_request_context.request.payment.take()`. The `refund_status.refundable_cycles` is set to `P - base_fee`.
3. The context is stored in `subnet_call_context_manager.canister_http_request_contexts`.
4. The adapter client (`rs/https_outcalls/client/src/client.rs`, line 155–178) receives the request, detects `PayAsYouGo`, logs a warning, and sends back a `SysFatal` reject with `budget.create_payment_receipt()` — which produces a zero-refund receipt.
5. The consensus/payload layer processes this reject response. The refundable cycles are never returned because the payment receipt claims nothing was refunded.
6. The calling canister receives a `SysFatal` reject and has permanently lost `P - base_fee` cycles.

### Impact Explanation

Any canister that calls `FlexibleHttpRequest` (the new flexible HTTP outcalls API) loses cycles equal to `payment - base_fee` on every call. The `base_fee` itself is also non-refundable. Since `FlexibleHttpRequest` is the only way to use non-replicated or flexible-committee HTTP outcalls, any canister developer using this API loses cycles on every invocation. The loss is proportional to the payment attached. This is a **cycles/resource accounting bug** with direct financial impact on canister developers.

### Likelihood Explanation

The `FlexibleHttpRequest` endpoint is a production API endpoint reachable by any canister. Any canister developer who reads the documentation and uses `FlexibleHttpRequest` will trigger this bug on every call. The adapter-side rejection is unconditional — there is no code path that allows a `PayAsYouGo` request to succeed. The likelihood is **high** for any user of the `FlexibleHttpRequest` API.

### Recommendation

Either:
1. Implement proper `PayAsYouGo` support in the adapter client so that the refund receipt correctly reports the refundable cycles, allowing the consensus layer to return them to the calling canister; or
2. Reject `PayAsYouGo` requests **before** the payment is deducted in `try_add_http_context_to_replicated_state`, so the canister never loses cycles for an unsupported feature.

The `ALLOWED_HTTP_OUTCALLS_PRICING_VERSIONS` constant already excludes `PayAsYouGo` from the legacy `HttpRequest` endpoint (only `PRICING_VERSION_LEGACY` is allowed), but the `FlexibleHttpRequest` endpoint bypasses this gate and always sets `PayAsYouGo`.

    pub fn generate_from_args(
        time: Time,
        request: &Request,
        args: CanisterHttpRequestArgs,
        node_ids: &BTreeSet<NodeId>,
        rng: &mut dyn RngCore,
    ) -> Result<Self, CanisterHttpRequestContextError> {
        validate_transform_principal(&args.transform, request.sender.get())?;
        validate_url_length(&args.url)?;
        validate_http_headers_and_body(args.headers.get(), args.body.as_ref().unwrap_or(&vec![]))?;
        if node_ids.is_empty() {
            return Err(CanisterHttpRequestContextError::NoNodesAvailableForDelegation);
        }

        let max_response_bytes = match args.max_response_bytes {
            Some(max_response_bytes) => {
                if max_response_bytes > MAX_CANISTER_HTTP_RESPONSE_BYTES {
                    Err(CanisterHttpRequestContextError::MaxResponseBytes(
                        InvalidMaxResponseBytes {
                            min: 0,
                            max: MAX_CANISTER_HTTP_RESPONSE_BYTES,
                            given: max_response_bytes,
                        },
                    ))
                } else {
                    Ok(Some(NumBytes::from(max_response_bytes)))
                }
            }
            None => Ok(None),
        }?;

        // Allow PUT, DELETE, and PATCH only in non-replicated mode to avoid
        // confusing race conditions that may occur.
        // For example, if first a DELETE outcall for resource R is made,
        // directly followed by a PUT, PATCH, or POST outcall for R, in
        // replicated mode it may happen that R is actually deleted after the
        // PUT/PATCH/POST outcall has finished, because the IC does not
        // necessarily wait for all outcalls to complete before a result is
        // delivered back to the canister: The IC only waits for sufficient
        // calls to complete to reach consensus on the result.
        if matches!(
            args.method,
            HttpMethod::PUT | HttpMethod::DELETE | HttpMethod::PATCH
        ) && args.is_replicated != Some(false)
        {
            return Err(CanisterHttpRequestContextError::DeterministicResponseCountRequired);
        }

        let replication = match args.is_replicated {
            Some(false) => {
                let delegated_node_id = node_ids
                    .iter()
                    .copied()
                    .choose(rng)
                    .ok_or(CanisterHttpRequestContextError::NoNodesAvailableForDelegation)?;

                Replication::NonReplicated(delegated_node_id)
            }
            _ => Replication::FullyReplicated,
        };

        Ok(CanisterHttpRequestContext {
            request: request.clone(),
            url: args.url,
            max_response_bytes,
            headers: args.headers.get().iter().cloned().map(From::from).collect(),
            body: args.body,
            http_method: args.method.into(),
            transform: args.transform.map(From::from),
            time,
            replication,
            pricing_version: {
                let final_version_u32 = args
                    .pricing_version
                    .filter(|v| ALLOWED_HTTP_OUTCALLS_PRICING_VERSIONS.contains(v))
                    .unwrap_or(DEFAULT_HTTP_OUTCALLS_PRICING_VERSION);
                PricingVersion::from_repr(final_version_u32).unwrap_or(PricingVersion::Legacy)
            },
            // The refund status is populated in `try_add_http_context_to_replicated_state`
            // based on the request's payment and the base fee.
            refund_status: RefundStatus::default(),
            // TODO: populate with the actual registry version this request is processed at.
            registry_version: RegistryVersion::from(0),
        })
