Audit Report

## Title
Unbounded Tokio Task Spawning via Attacker-Controlled `stripped_idkg_dealings` Count in `assemble_message` — (File: `rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary
A Byzantine block maker below the consensus fault threshold can craft a `StrippedBlockProposal` with an arbitrarily large `stripped_idkg_dealings` list. Because `assemble_message` spawns one unbounded tokio task per entry with no concurrency cap, and each task enters an infinite retry loop, the attacker can transiently exhaust the tokio task pool and QUIC transport connection pool for the duration of the consensus round, degrading subnet availability.

## Finding Description

**Root cause — No count guard at deserialization.**

`TryFrom<pb::StrippedBlockProposal> for StrippedBlockProposal` in `rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs` (lines 110–128) iterates over `value.stripped_idkg_dealings` and collects into a `Vec` with no upper-bound check:

```rust
stripped_idkg_dealings: StrippedIDkgDealings {
    stripped_dealings: value
        .stripped_idkg_dealings
        .into_iter()
        .map(|dealing| { ... })
        .collect::<Result<Vec<_>, ProxyDecodeError>>()?,
},
```

The protobuf schema (`rs/protobuf/def/types/v1/consensus.proto`, line 263) declares `repeated StrippedIDkgDealing stripped_idkg_dealings = 4;` with no field-count constraint. The NiDKG `TooManyDealings` guard in `rs/consensus/dkg/src/payload_validator.rs` (lines 164–170) and `DKG_DEALINGS_PER_BLOCK = 1` in `rs/limits/src/lib.rs` (line 93) apply exclusively to NiDKG, not to IDKG chain-key dealings.

**Unbounded JoinSet spawn.**

`assemble_message` in `rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs` (lines 211–226) spawns one task per entry returned by `missing_stripped_messages()` with no semaphore or concurrency cap:

```rust
for stripped_message_id in stripped_message_ids {
    join_set.spawn(get_or_fetch(...));
}
```

**Infinite retry loop per task.**

`download_stripped_message` in `rs/p2p/artifact_downloader/src/fetch_stripped_artifact/download.rs` (lines 301–305) configures the backoff with `with_max_elapsed_time(None)` and loops forever until a successful response. For fabricated dealing IDs that no peer holds, the loop never exits.

**Bouncer is the only cleanup path.**

Tasks are aborted only when `bouncer.wait_for(|bouncer| matches!(bouncer(&id), BouncerValue::Unwanted))` fires (lines 233–242 of `assembler.rs`). The bouncer refresh period is 3 seconds (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`, line 101). A `BlockProposal` remains `Wants` while `finalized_height < height <= notarized_height + LOOK_AHEAD(10)` (`rs/consensus/src/consensus/priority.rs`, lines 83–96), meaning the attack window spans multiple seconds to tens of seconds per round.

**ID-mismatch check fires too late.**

`try_assemble()` (lines 620–649 of `assembler.rs`) checks `assembled_id != claimed_id` only after all tasks complete — but the tasks loop forever for non-existent dealings, so this check is never reached until the bouncer fires. The resource exhaustion has already occurred.

## Impact Explanation

During the window between advertisement of the Byzantine `StrippedBlockProposal` and finalization of its height, N tasks — one per crafted `stripped_idkg_dealings` entry — each hold a tokio task slot and issue repeated QUIC transport RPCs. With N in the thousands (protobuf imposes no per-field count limit), this can saturate the QUIC connection pool and spike tokio task count, degrading the replica's ability to process other P2P messages, consensus artifacts, and ingress. The attack is repeatable every consensus round.

This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

The attacker must be a valid block maker for the current round — a Byzantine node below the fault threshold. Such a node can produce a legitimate `BlockProposal` (so `unstripped_consensus_message_id` passes the bouncer), then craft a `StrippedBlockProposal` with the correct block hash but an inflated `stripped_idkg_dealings` list, and advertise it to victim peers. No victim mistake or external compromise is required. The attack is repeatable each round the Byzantine node is selected as block maker.

## Recommendation

Add a maximum-count guard during deserialization of `StrippedBlockProposal`, analogous to the NiDKG `TooManyDealings` check. A reasonable bound is the subnet size (number of nodes), since each transcript can have at most one dealing per dealer. Reject any `StrippedBlockProposal` whose `stripped_idkg_dealings.len()` exceeds this bound before any tasks are spawned. As defense-in-depth, cap the `JoinSet` concurrency in `assemble_message` with a semaphore.

## Proof of Concept

Minimal deterministic integration test plan:

1. Instantiate a `FetchStrippedConsensusArtifact` with a mock transport and mock bouncer.
2. Craft a `pb::StrippedBlockProposal` with a valid `unstripped_consensus_message_id` (pointing to a real block hash at the current height) and 1,000 entries in `stripped_idkg_dealings`, each with a syntactically valid but non-existent `IDkgArtifactId::Dealing`.
3. Deserialize via `StrippedBlockProposal::try_from` — confirm it succeeds with no error.
4. Call `assemble_message` with this artifact.
5. Assert that `tokio::runtime::Handle::current().metrics().num_alive_tasks()` spikes to ≥1,000 immediately after the call.
6. Confirm tasks are only cleaned up after the bouncer fires (mock bouncer returns `Unwanted` after a delay).
7. Confirm that with a count guard of `subnet_size` the deserialization returns an error before any tasks are spawned.
