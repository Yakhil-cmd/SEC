## Analysis

The inconsistency is real and confirmed. However, the critical question for impact is: **does the EVM executor (SputnikVM) actually credit the value to the precompile address's balance when the precompile call succeeds?**

Yes — in SputnikVM's `MemoryStackState`, a `transact_call` with non-zero value to a precompile address performs the value transfer at the executor level (debit sender, credit callee) before dispatching to the precompile. If the precompile returns `Ok(...)`, the executor commits those balance changes. `Engine::apply` then writes them to Aurora's NEAR storage. The ETH lands at `0x0000...0004` with no code capable of sending it back, and no admin recovery path exists.

The `validate_no_value_attached_to_precompile` guard in the Aurora-specific precompiles exists precisely to revert the transaction (and thus roll back the value transfer) before it is committed. `Identity` lacks this guard, so the value transfer is committed on success.

**Important scope note:** The same missing check applies to all other standard Ethereum precompiles registered as `AllPrecompiles::Generic` (ECRecover `0x01`, SHA256 `0x02`, RIPEMD160 `0x03`, ModExp `0x05`, Bn256Add `0x06`, Bn256Mul `0x07`, Bn256Pair `0x08`, Blake2F `0x09`). [11](#0-10) 

On standard Ethereum this is accepted behavior (ETH sent to precompile addresses is effectively burned). On Aurora, ETH is backed by bridged real ETH, so permanent freezing at an unrecoverable address is a real loss.

### Title
Missing `validate_no_value_attached_to_precompile` in `Identity::run` Allows Permanent ETH Freezing — (`engine-precompiles/src/identity.rs`)

### Summary
`Identity::run` ignores `context.apparent_value`, allowing any EVM user to send ETH to `0x0000000000000000000000000000000000000004`. The EVM executor credits the value to that address, the precompile succeeds, and `Engine::apply` commits the balance change. The ETH is permanently frozen with no recovery path.

### Finding Description
Six Aurora-specific precompiles (`PrepaidGas`, `RandomSeed`, `CurrentAccount`, `PredecessorAccount`, `PromiseResult`, `CrossContractCall`) call `utils::validate_no_value_attached_to_precompile(context.apparent_value)` at the top of their `run` methods, returning `Err(ExitError::Other("ATTACHED_VALUE_ERROR"))` when `apparent_value > 0`. This causes the EVM executor to revert the call, rolling back the value transfer.

`Identity::run` takes `_context: &Context` (underscore-prefixed, intentionally unused) and performs no such check. When called with `value > 0`, it returns `Ok(PrecompileOutput::without_logs(...))`. The executor then commits the balance delta: sender loses `value`, address `0x0000...0004` gains `value`. Since the Identity precompile has no EVM code capable of initiating a transfer out, and no admin/operator function exists to drain precompile addresses, the ETH is irrecoverable.

### Impact Explanation
Any ETH sent to the Identity precompile address is permanently frozen. On Aurora, ETH balances are backed by bridged ETH; frozen ETH represents a real, irreversible loss for the sender. The impact is **Critical — Permanent freezing of funds**.

### Likelihood Explanation
The attack requires only a standard EVM transaction from any unprivileged user with an ETH balance. No special privileges, no governance capture, no external dependencies. The path is: sign a transaction with `to = 0x0000...0004`, `value > 0` → submit via Aurora's `submit` endpoint → `Engine::call` → `executor.transact_call` → `Precompiles::execute` → `AllPrecompiles::Generic(Identity)` → `process_precompile` → `Identity::run` (succeeds) → `Engine::apply` commits balance change.

### Recommendation
Add the value guard at the top of `Identity::run`, consistent with all Aurora-specific precompiles:

```rust
fn run(
    &self,
    input: &[u8],
    target_gas: Option<EthGas>,
    context: &Context,   // remove underscore prefix
    _is_static: bool,
) -> EvmPrecompileResult {
    utils::validate_no_value_attached_to_precompile(context.apparent_value)?;
    // ... existing logic
}
```

Apply the same fix to all other standard Ethereum precompiles registered as `AllPrecompiles::Generic` (ECRecover, SHA256, RIPEMD160, ModExp, Bn256Add, Bn256Mul, Bn256Pair, Blake2F, BLS12-381 variants, Secp256r1).
