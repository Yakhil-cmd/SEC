Audit Report

## Title
Off-by-One Epoch in SNS `distribute_rewards` Causes Systematic Under-Rewarding of Neuron Holders - (File: rs/sns/governance/src/governance.rs)

## Summary
The `distribute_rewards` function in SNS governance computes the reward rate for each round using the timestamp at the **end** of the round (`round_duration_seconds * i`) rather than the **beginning** (`round_duration_seconds * (i-1)`). Because `reward_rate_at` returns a monotonically decreasing rate during the transition period, every round's contribution to `rewards_purse_e8s` is understated. The shortfall is permanent — the under-minted maturity is never recovered — and accumulates across every reward round for the full `reward_rate_transition_duration_seconds` window.

## Finding Description
In `rs/sns/governance/src/governance.rs`, `distribute_rewards` builds the reward purse with:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = round_duration_seconds
        .saturating_mul(i)                    // ← end of round i, not beginning
        .saturating_add(reward_start_timestamp_seconds)
        .saturating_sub(self.proto.genesis_timestamp_seconds);

    let current_reward_rate = voting_rewards_parameters.reward_rate_at(
        crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
    );
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [1](#0-0) 

`reward_start_timestamp_seconds` is the `end_timestamp_seconds` of the latest reward event, i.e., the **start** of the current reward window. [2](#0-1) 

For round `i = 1`, the code feeds `reward_start_timestamp_seconds + 1 × round_duration_seconds − genesis` to `reward_rate_at`. That is the **end** of round 1. The correct value is `reward_start_timestamp_seconds + 0 × round_duration_seconds − genesis` (the **beginning** of round 1).

`reward_rate_at` returns a quadratically decreasing rate from `initial_reward_rate` toward `final_reward_rate` over `reward_rate_transition_duration_seconds`: [3](#0-2) 

The rate is strictly decreasing during the transition, as confirmed by the `reward_is_bounded_during_transition` and `reward_is_convex_and_decreasing` tests: [4](#0-3) 

Therefore `rate(end_of_round_i) < rate(start_of_round_i)` for every round during the transition, and every round's purse contribution is understated.

The NNS governance explicitly documents the correct design intent in its parallel implementation:

> "Despite the rate being arguable a continuous function of time, we don't integrate the rate here. Instead we multiply the **rate at the beginning of that day** with the considered duration." [5](#0-4) 

The accumulated purse is then distributed proportionally to neuron voting shares with no correction applied: [6](#0-5) 

## Impact Explanation
Every SNS neuron holder who participates in governance voting receives permanently less maturity than the protocol intends to award. The total shortfall over the full transition window approximates `(initial_rate − final_rate) × round_duration × token_supply`. For a large SNS (e.g., 100 million tokens, 2 %→1 % transition over 42 weeks with weekly rounds), this is on the order of ~19,000 tokens per SNS instance — a concrete, irreversible financial loss to all voting participants. The lost maturity is simply never minted. This constitutes a **significant SNS impact with concrete user financial harm**, matching the High bounty tier.

## Likelihood Explanation
The code path executes automatically via `run_periodic_tasks` on every SNS governance canister that has `voting_rewards_parameters` configured with a non-zero `reward_rate_transition_duration_seconds`. No privileged action is required. Any neuron holder who votes causes proposals to enter `ReadyToSettle`, which triggers `distribute_rewards`. The bug is active for the entire duration of the reward rate transition on every affected SNS instance.

## Recommendation
Change the loop to use `i - 1` (or `i.saturating_sub(1)`) when computing `seconds_since_genesis`, so the rate at the **beginning** of each round is used:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = round_duration_seconds
        .saturating_mul(i - 1)   // rate at the START of round i
        .saturating_add(reward_start_timestamp_seconds)
        .saturating_sub(self.proto.genesis_timestamp_seconds);

    let current_reward_rate = voting_rewards_parameters.reward_rate_at(
        crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
    );
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
```

This aligns the SNS implementation with the NNS design intent documented in `rs/nns/governance/src/reward/calculation.rs`.

## Proof of Concept
A deterministic unit test suffices. Using the existing `VOTING_REWARDS_PARAMETERS` fixture in `rs/sns/governance/src/reward.rs` (200 bp initial, 100 bp final, 42-week transition, 1-week rounds):

1. Simulate `distribute_rewards` for `new_rounds_count = 1` with `reward_start_timestamp_seconds = genesis_timestamp_seconds` (first ever round).
2. **Current code** feeds `seconds_since_genesis = round_duration_seconds` (end of round 1) → rate ≈ 1.976 %/yr.
3. **Fixed code** feeds `seconds_since_genesis = 0` (beginning of round 1) → rate = 2.000 %/yr exactly (confirmed by `reward_for_first_day` test at line 547).
4. Assert that the purse computed by the fixed code exceeds the purse computed by the current code.
5. Repeat for all 42 rounds and assert the cumulative shortfall equals approximately `(initial_rate − final_rate) × round_duration × supply`.
