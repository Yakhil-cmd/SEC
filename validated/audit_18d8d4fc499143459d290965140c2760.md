### Title
Unimplemented `todo!()` in MPT Insertion Path Causes Runtime Panic on Reachable State Transition - (File: `basic_system/src/system_implementation/ethereum_storage_model/mpt/updates/mod.rs`)

### Summary

The `find_insertion_strategy` function in the Ethereum MPT implementation contains two `todo!()` stubs in reachable match arms. When a transaction triggers MPT insertion along a proof-traversal path that returns `AppendPath::PathDiverged` or `AppendPath::EmptyBranchTaken`, the process unconditionally panics. This is a direct analog to the reported `force_unreserve` pattern: a code path is registered and reachable through normal execution but its implementation body is a `todo!()` stub.

### Finding Description

Inside `find_insertion_strategy`, after `descend_through_existing_nodes` returns `UnreferencedPathEncountered` (meaning the trie has partially-loaded proof nodes), the function enters a second loop that calls `descend_through_proof`. Two of the five match arms in that loop are stubs:

```rust
AppendPath::PathDiverged { allocated_node } => {
    // ...
    self.link_if_needed(current_node, parent_branch_index, allocated_node)?;
    todo!();   // ← line 201
}
AppendPath::EmptyBranchTaken { allocated_node, .. } => {
    // ...
    self.link_if_needed(current_node, parent_branch_index, allocated_node)?;
    todo!();   // ← line 207
}
``` [1](#0-0) 

The function is called from `insert_lazy_value` → `insert`, both of which are `pub` methods on `EthereumMPT`. [2](#0-1) 

`EthereumMPT` is a production module (not test/config/build-only) under `basic_system/src/system_implementation/ethereum_storage_model/`.



### Impact Explanation

When the Ethereum storage model STF processes a transaction that writes to a new storage slot in an account whose existing storage trie contains unreferenced (partially-loaded) proof nodes, `find_insertion_strategy` enters the proof-traversal loop. If the new key diverges from the existing proof path (`PathDiverged`) or lands on an empty branch slot (`EmptyBranchTaken`), the `todo!()` macro fires, causing an unconditional `panic!`. This crashes the running process (sequencer / forward-execution node), halting block production and preventing the chain from advancing — identical in impact to the reported `force_unreserve` DoS.

### Likelihood Explanation

An unprivileged transaction sender can trigger this by:
1. Targeting any account that already has non-trivial storage (non-empty MPT with unreferenced nodes from a partial proof).
2. Writing to a **new** storage key whose path diverges from the existing proof path or lands on an empty branch.

This is a normal EVM `SSTORE` to a fresh slot in an existing contract — no special privilege is required. The `UnreferencedPathEncountered` branch is explicitly handled in `find_insertion_strategy` (lines 171–178), confirming the runtime expects to reach it. [3](#0-2) 

### Recommendation

- **Short term**: Replace both `todo!()` stubs with a proper `return Err(())` (or the correct insertion logic) so that the unimplemented case is handled gracefully rather than panicking.
- **Long term**: Audit all `todo!()` / `unimplemented!()` occurrences in production (non-test) code paths and either implement them or gate them behind a compile-time error so they cannot be reached at runtime.

### Proof of Concept

1. Deploy a contract that writes to at least one storage slot (creates a non-trivial MPT).
2. In a subsequent transaction, call the same contract with a new `SSTORE` to a key whose nibble path diverges from the existing proof path inside the partially-loaded MPT.
3. The Ethereum storage model calls `EthereumMPT::insert` → `insert_lazy_value` → `find_insertion_strategy`.
4. `descend_through_existing_nodes` returns `UnreferencedPathEncountered`; the proof-traversal loop begins.
5. `descend_through_proof` returns `PathDiverged` or `EmptyBranchTaken`.
6. `todo!()` fires at line 201 or 207, panicking the process. [4](#0-3)
