### Title
Unverified `native_price` and `pubdata_price` from Oracle Enable Malicious Prover to Manipulate Native Resource Accounting - (File: `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

### Summary

`BlockMetadataFromOracle` is read directly from the oracle at block start and its `native_price` and `pubdata_price` fields are used verbatim to derive every transaction's native resource limit and pubdata cost. Neither field is committed to in the ZK public input (`BatchOutput` / `BatchPublicInput`). A malicious prover/forward-execution input can therefore supply arbitrary values for these fields, causing the proven state transition to reflect fee accounting that was never agreed upon by the settlement layer, without any proof-level detection.

### Finding Description

In `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`, the entire `BlockMetadataFromOracle` struct is fetched from the oracle in a single call:

```rust
let block_level_metadata: BlockMetadataFromOracle =
    oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)?;
``` [1](#0-0) 

The struct contains `native_price` and `pubdata_price`:

```rust
impl ZkSpecificPricingMetadata for BlockMetadataFromOracle {
    fn get_pubdata_price(&self) -> U256 { self.pubdata_price }
    fn native_price(&self) -> U256 { self.native_price }
``` [2](#0-1) 

These two values drive the critical `native_per_gas` and `native_per_pubdata` ratios used in every L2 transaction:

```rust
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [3](#0-2) 

The `BatchOutput` that is hashed into the public input contains `chain_id`, `first_block_timestamp`, `last_block_timestamp`, `da_commitment_scheme`, `pubdata_commitment`, L1/L2 tx counts, `priority_operations_hash`, `l2_logs_tree_root`, `upgrade_tx_hash`, `interop_roots_rolling_hash`, and `settlement_layer_chain_id` — but **not** `native_price`, `pubdata_price`, `eip1559_basefee`, `coinbase`, `gas_limit`, `pubdata_limit`, or `mix_hash`. [4](#0-3) 

The `ChainStateCommitment` (the other half of the public input) commits only to `state_root`, `next_free_slot`, `block_number`, `last_256_block_hashes_blake`, and `last_block_timestamp`. [5](#0-4) 

No other code path re-derives or constrains `native_price` / `pubdata_price` from the committed state. The oracle documentation acknowledges that "Oracle responses are non-deterministic and MUST be treated as untrusted input" and that "All responses should be validated against additional constraints during deserialization or usage," but no such validation exists for these two pricing fields. [6](#0-5) 

### Impact Explanation

A malicious prover (or a crafted forward-execution input) can set `native_price = 1` and `pubdata_price = 0` in the oracle response for `BLOCK_METADATA_QUERY_ID`. The resulting execution:

1. **`native_price = 1`** → `native_per_gas = gas_price / 1 = gas_price`. Every transaction receives a native resource limit equal to its full gas price, far exceeding what the user actually paid for in terms of proving cost. Transactions that should have been rejected for insufficient native resources are accepted.

2. **`pubdata_price = 0`** → `native_per_pubdata = 0`. All pubdata is effectively free. Transactions that write large amounts of state (and would normally be reverted post-execution for insufficient native resources to cover pubdata) succeed unconditionally.

The resulting state root — with incorrect balances, incorrect revert/success outcomes, and incorrect fee transfers to the coinbase — is committed to the settlement layer as a valid proven state. The settlement layer has no mechanism to detect the manipulation because neither field appears in the public input.

Conversely, setting `native_price = U256::MAX` causes `native_per_gas ≈ 0`, which collapses the native limit to near zero and reverts every user transaction in the block, constituting a provable-but-incorrect DoS against all L2 users in that batch.

### Likelihood Explanation

The attacker must control the oracle data fed to the proving run — i.e., must be the prover or must supply the forward-execution input. This is explicitly listed as an in-scope attacker class ("prover/forward execution input"). The manipulation requires no special cryptographic capability: it is a single field substitution in the oracle response before the RISC-V run begins. There is no runtime check, no Merkle proof, and no public-input binding that would cause the proof to fail.

### Recommendation

Commit `native_price` and `pubdata_price` (and the other unbound fields: `eip1559_basefee`, `coinbase`, `gas_limit`, `pubdata_limit`, `mix_hash`) into the `BatchOutput` hash so that the settlement layer can verify them. At minimum, `native_price` and `pubdata_price` must be included because they directly determine the native resource accounting that governs which transactions succeed or fail and how much ETH is transferred between users and the operator.

### Proof of Concept

1. Construct a proving run where the `BlockMetadataResponder` returns `native_price = U256::from(1)` and `pubdata_price = U256::ZERO` for `BLOCK_METADATA_QUERY_ID`.
2. Include a transaction that writes to 20 storage slots (generating significant pubdata). Under correct pricing this transaction would be reverted post-execution for insufficient native resources to cover pubdata.
3. Run the RISC-V prover. The transaction succeeds; the resulting state root reflects the storage writes.
4. Inspect `BatchPublicInput::batch_output` — it contains no field that encodes `native_price` or `pubdata_price`.
5. The proof verifies on the settlement layer, committing an incorrect state root that reflects free pubdata and inflated native limits. [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9)

        oracle: &mut impl IOOracle,

        let block_level_metadata: BlockMetadataFromOracle =

            block_level: block_level_metadata,
            _marker: core::marker::PhantomData,

        if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT

        }

        Ok(metadata)
    }

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }

    pub number_of_layer_1_txs: U256,
    /// Number of processed L2 txs in the batch.
    pub number_of_layer_2_txs: U256,

    pub priority_operations_hash: Bytes32,
    /// L2 logs tree root.
    /// Note that it's full root, it's keccak256 of:

    /// - multichain root - commitment to logs emitted on chains that settle on the current.
    pub l2_logs_tree_root: Bytes32,
    /// Protocol upgrade tx hash (0 if there wasn't)
    pub upgrade_tx_hash: Bytes32,
    /// Linear keccak256 hash of interop roots

    /// Settlement layer chain id.
    pub settlement_layer_chain_id: U256,
}

impl BatchOutput {
    ///
    /// Calculate keccak256 hash of public input
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = Keccak256::new();
        hasher.update(self.chain_id.to_be_bytes::<32>());
        hasher.update(&self.first_block_timestamp.to_be_bytes());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
