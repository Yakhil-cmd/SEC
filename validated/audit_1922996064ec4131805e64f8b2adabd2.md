Audit Report

## Title
Unescaped UTF-8 Memo Enables Markdown Injection into ICRC-21 `GenericDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_memo` function in `responses.rs` interpolates raw, caller-supplied UTF-8 memo bytes directly into a Markdown-formatted `GenericDisplayMessage` string without escaping backticks or newlines. A malicious dapp can craft a `TransferArg` whose memo breaks out of the intended inline code span and appends fake bold fields (e.g., `**Amount:**`, `**To:**`) to the consent message rendered by wallets and hardware signing devices, undermining the trust guarantee that ICRC-21 is designed to provide.

## Finding Description
In `packages/icrc-ledger-types/src/icrc21/responses.rs`, `add_memo` at line 288 decodes memo bytes as UTF-8 with no sanitization, and at line 294 interpolates the result directly into a Markdown string:

```rust
// L288-291
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),  // safe path
};
// L293-294
ConsentMessage::GenericDisplayMessage(message) => {
    message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
}
```

The hex fallback is safe (only `[0-9a-f]`), but the UTF-8 path is not. A memo beginning with a backtick followed by `\n\n**Amount:** \`999999 ICP` produces:

```
\n\n**Memo:**\n``\n\n**Amount:** `999999 ICP`
```

Under CommonMark, the two leading backticks open a double-backtick code span that is never closed, so they render as literal characters. The subsequent `\n\n**Amount:**` is parsed as a normal bold field, and `` `999999 ICP` `` becomes a code span — producing a visually convincing fake field appended after the real ones.

The call path is direct and requires no privilege:
1. Caller submits `icrc21_canister_call_consent_message` with a `TransferArg` whose `memo` contains the crafted bytes.
2. `build_icrc21_consent_info` at `lib.rs` L392–394 wraps the memo as `GenericMemo::Icrc1Memo(memo.0)` and passes it to the builder.
3. `build()` calls `message.add_memo(memo)` at `lib.rs` L290–291 as the final step, appending the unescaped string to the already-complete message.
4. The `GenericDisplayMessage` returned to the wallet contains the injected Markdown.

No existing guard sanitizes or escapes the UTF-8 memo content before interpolation.

## Impact Explanation
The `GenericDisplayMessage` variant is a Markdown string explicitly intended to be rendered by wallets and hardware signing devices per the ICRC-21 specification. The entire purpose of ICRC-21 is to give users a trustworthy, ledger-generated view of what they are signing, even when the initiating dapp is untrusted. A crafted memo can append fake `**Amount:**` or `**To:**` fields after the real ones. On a hardware wallet with a small scrollable display (one field visible at a time), a user reviewing the last visible field may be misled into believing the injected value is authoritative and approve a transaction transferring a much larger amount or to a different recipient than intended. This constitutes a moderate user-funds/security impact matching the Medium bounty tier.

## Likelihood Explanation
- No privilege is required; any caller can invoke `icrc21_canister_call_consent_message`.
- The ICRC-1 memo field is a free-form byte blob up to 32 bytes; crafting a valid UTF-8 payload with a leading backtick and embedded newlines is trivial.
- The exploit is deterministic and locally reproducible with a unit test.
- The only precondition is that the wallet renders `GenericDisplayMessage` as Markdown, which is the intended and specified behavior of ICRC-21.
- Practical impact is constrained to hardware wallets or wallets with limited display area where the injected field appears as the last visible item; desktop wallets showing the full message may expose the duplication.

## Recommendation
Escape Markdown control characters in `memo_str` before interpolation. At minimum, replace backtick, newline, and carriage-return characters:

```rust
let safe_memo_str = memo_str
    .replace('`', "\\`")
    .replace('\n', " ")
    .replace('\r', " ");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo_str}`"));
```

Alternatively, hex-encode the memo unconditionally for the `GenericDisplayMessage` path, reserving UTF-8 display only for the `FieldsDisplayMessage` path where the value is a structured `Value::Text` field and not embedded in a Markdown string.

## Proof of Concept
```rust
let memo_bytes = b"`\n\n**Amount:** `999999 ICP".to_vec();
let transfer_arg = TransferArg {
    memo: Some(Memo(memo_bytes)),
    amount: Nat::from(1u64),
    to: real_receiver,
    fee: None,
    from_subaccount: None,
    created_at_time: None,
};
let request = ConsentMessageRequest {
    method: "icrc1_transfer".to_string(),
    arg: Encode!(&transfer_arg).unwrap(),
    user_preferences: ConsentMessageSpec {
        metadata: ConsentMessageMetadata {
            language: "en".to_string(),
            utc_offset_minutes: None,
        },
        device_spec: Some(DisplayMessageType::GenericDisplay),
    },
};
let info = build_icrc21_consent_info(/* ... */).unwrap();
if let ConsentMessage::GenericDisplayMessage(msg) = info.consent_message {
    // Injection confirmed: fake bold Amount field appears after real fields
    assert!(msg.contains("**Amount:** `999999 ICP`"));
}
```
