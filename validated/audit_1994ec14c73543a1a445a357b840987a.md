### Title
Cross-Chain Signature Replay via Accepted Pre-EIP-155 Legacy Transactions - (`engine-transactions/src/legacy.rs`, `engine/src/engine.rs`)

### Summary
Aurora Engine's `submit` entrypoint accepts pre-EIP-155 legacy Ethereum transactions (those with `v = 27` or `v = 28`) without enforcing any chain ID binding. Because the signing hash for such transactions omits the chain ID entirely, a transaction signed on Ethereum mainnet (or any other EVM chain) is cryptographically identical to one signed for Aurora. An unprivileged attacker who observes such a transaction on-chain or in the mempool can replay it verbatim on Aurora Engine, executing it against the victim's Aurora account.

### Finding Description

**Step 1 — Hash construction omits chain ID for pre-EIP-155 transactions.**

In `TransactionLegacy::rlp_append_unsigned`, when `chain_id` is `None`, only 6 fields are RLP-encoded (nonce, gas_price, gas_limit, to, value, data). No chain ID, no padding zeros are appended: [1](#0-0) 

**Step 2 — `LegacyEthSignedTransaction::sender()` sets `chain_id = None` for v ∈ {27, 28}.**

When `v` is 27 or 28, the sender-recovery path passes `chain_id = None` to `rlp_append_unsigned`, producing a 6-field hash with no chain binding. The recovered address is then used as the transaction sender: [2](#0-1) 

**Step 3 — `NormalizedEthTransaction` propagates `chain_id: None` for legacy transactions.**

When converting a `Legacy` variant, `chain_id` is taken directly from `tx.chain_id()`, which returns `None` for pre-EIP-155 transactions: [3](#0-2) 

**Step 4 — `submit_with_alt_modexp` skips chain ID validation when `chain_id` is `None`.**

The guard uses `if let Some(chain_id) = transaction.chain_id`. When `chain_id` is `None`, the entire check is bypassed and execution continues unconditionally: [4](#0-3) 

The `submit` public entrypoint delegates directly to `submit_with_alt_modexp`: [5](#0-4) 

### Impact Explanation

An attacker who observes a pre-EIP-155 legacy transaction (v=27 or v=28) sent by a victim on Ethereum mainnet can submit the identical raw bytes to Aurora Engine's `submit` endpoint. If the victim holds ETH on Aurora at the same address and the nonce matches, the transaction executes on Aurora as if the victim signed it. This enables:

- **Direct theft of ETH**: A pre-EIP-155 ETH transfer transaction replayed on Aurora drains the victim's Aurora ETH balance.
- **Unauthorized contract interaction**: A pre-EIP-155 contract call (e.g., ERC-20 `approve`, `transfer`) replayed on Aurora executes against the victim's Aurora token holdings.

Impact: **Critical — direct theft of user funds at rest.**

### Likelihood Explanation

Pre-EIP-155 transactions remain in active use. Hardware wallets in certain legacy modes, older Ethereum tooling, and some DeFi protocols still produce v=27/28 transactions. The Ethereum mainnet mempool and block history contain millions of such transactions. An attacker needs only to:

1. Monitor the Ethereum mempool or block history for a pre-EIP-155 transaction from an address that also holds funds on Aurora.
2. Submit the same raw bytes to Aurora's `submit` endpoint.

No special privileges, leaked keys, or social engineering are required. The attack is fully permissionless and can be automated.

### Recommendation

Reject pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`) at the `submit` entrypoint. The chain ID check in `submit_with_alt_modexp` should be changed from an optional guard to a mandatory one:

```rust
// Current (vulnerable):
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}

// Fixed: require chain_id to be present and correct
let chain_id = transaction.chain_id.ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

Alternatively, reject any `LegacyEthSignedTransaction` with `v ∈ {27, 28}` during parsing in `EthTransactionKind::try_from`.

### Proof of Concept

1. On Ethereum mainnet, a user sends a pre-EIP-155 ETH transfer: `nonce=5, to=0xBob, value=1 ETH, v=27`. The raw RLP bytes are `TX_BYTES`.
2. The same user holds 10 ETH on Aurora at the same address, also at nonce 5.
3. Attacker submits `TX_BYTES` to Aurora Engine's `submit` endpoint (callable by any NEAR account).
4. `LegacyEthSignedTransaction::sender()` recovers the victim's address (hash is identical — no chain ID in either case).
5. `transaction.chain_id` is `None`; the chain ID guard at line 1055 of `engine.rs` is skipped.
6. `check_nonce` passes (nonce 5 matches).
7. Aurora Engine transfers 1 ETH from the victim's Aurora account to `0xBob` — funds stolen without the victim's consent. [6](#0-5) [4](#0-3)

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);
        let message_hash = sdk::keccak(rlp_stream.as_raw());
        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
            .map_err(|_| Error::EcRecover)
    }
