### Title
XCC Precompile TOCTOU Race Condition Permanently Freezes User wNEAR — (File: `engine-precompiles/src/xcc.rs`)

---

### Summary

The XCC precompile reads `get_code_version_of_address` to decide whether to charge `STORAGE_AMOUNT` (2 NEAR) from the user's wNEAR balance. The state update that marks the router as deployed (`factory_update_address_version`) executes in a later NEAR receipt, not atomically with the EVM execution. A second XCC call arriving in this window sees stale state, charges `STORAGE_AMOUNT` again via an EVM-committed `transferFrom`, then has its entire NEAR promise chain fail — permanently stranding the user's wNEAR in the engine's implicit EVM address with no refund path.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, `run_with_handle` decides how much wNEAR to charge:

```rust
let required_near =
    match state::get_code_version_of_address(&self.io, &Address::new(sender)) {
        None => attached_near + state::STORAGE_AMOUNT,
        Some(_) => attached_near,
    };
if required_near != ZERO_YOCTO {
    // EVM transferFrom: moves wNEAR from user → engine implicit address
    let (exit_reason, return_value) =
        handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
    ...
}
``` [1](#0-0) 

This `transferFrom` is committed as part of the EVM state transition in the current NEAR receipt. The wNEAR is now in the engine's implicit address.

The state that `get_code_version_of_address` reads is only updated by the `factory_update_address_version` callback, which runs in a **subsequent NEAR receipt**:

```rust
pub fn factory_update_address_version<I: IO + Copy, E: Env, H: PromiseHandler>(
    ...
) -> Result<(), ContractError> {
    ...
    let args: xcc::AddressVersionUpdateArgs = io.read_input_borsh()?;
    xcc::set_code_version_of_address(&mut io, &args.address, args.version);
    Ok(())
}
``` [2](#0-1) 

The promise chain in `handle_precompile_promise` is:

```
[deploy batch] → [factory_update_address_version] → [withdraw_wnear_to_router] → [send_refund] → [user call]
``` [3](#0-2) 

Between the first call's EVM execution and the `factory_update_address_version` receipt, `get_code_version_of_address` still returns `None`. A second XCC call in this window:

1. Sees `None` → charges `STORAGE_AMOUNT` again via `transferFrom` (committed to EVM state)
2. Creates a second deploy batch with `CreateAccount` — which **fails** because the router account already exists from the first call
3. `factory_update_address_version` receives `Some(false)` and returns `ERR_ROUTER_DEPLOY_FAILED`
4. `withdraw_wnear_to_router` checks `promise_result_check()`, gets `Some(false)`, returns `ERR_CALLBACK_OF_FAILED_PROMISE` [4](#0-3) 

5. `send_refund` calls `require_no_failed_promises()` and panics [5](#0-4) 

The entire second promise chain fails, but the wNEAR `transferFrom` already executed in the EVM. There is no refund path for the user's wNEAR. It remains permanently in the engine's implicit EVM address.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

`STORAGE_AMOUNT` is 2 NEAR:

```rust
pub const STORAGE_AMOUNT: Yocto = Yocto::new(2_000_000_000_000_000_000_000_000);
``` [6](#0-5) 

The wNEAR is moved to the engine's implicit EVM address. No automatic refund mechanism exists in the XCC promise chain for the case where the deploy batch fails. The funds are permanently inaccessible to the user without manual admin intervention.

---

### Likelihood Explanation

**Medium.** Two realistic trigger paths exist:

**Path 1 — Self-inflicted (no attacker needed):** A user's application sends two XCC transactions in quick succession (e.g., double-click, retry logic, or concurrent requests). Both land in the same NEAR block before `factory_update_address_version` runs. The second call sees stale state and loses 2 NEAR.

**Path 2 — Griefing attack via `fund_xcc_sub_account`:** `fund_xcc_sub_account` is callable by **anyone** for any target address when `wnear_account_id` is `None` (no access control):

```rust
if args.wnear_account_id.is_some() {
    require_owner_only(&state, &env.predecessor_account_id())?;
}
xcc::fund_xcc_sub_account(&io, handler, env, args)?;
``` [7](#0-6) 

An attacker observing a victim's first XCC call in the mempool can front-run it with `fund_xcc_sub_account` targeting the victim's address. Both calls land in the same block. The attacker's deploy batch runs first (router created). The victim's XCC call's deploy batch fails. The victim loses `STORAGE_AMOUNT` worth of wNEAR.

---

### Recommendation

1. **Add a failure refund path in the XCC promise chain.** Attach a callback after the deploy batch that, on failure, calls back into the engine to refund the user's wNEAR from the engine's implicit address.
2. **Alternatively, check on-chain account existence** before charging `STORAGE_AMOUNT`, rather than relying solely on the engine's internal `get_code_version_of_address` storage key, which lags behind actual on-chain state.
3. **Add access control to `fund_xcc_sub_account`** or rate-limit it per address to prevent griefing.

---

### Proof of Concept

```
Block N:
  Receipt 1: Victim's EVM tx calls XCC precompile.
             get_code_version_of_address(victim) → None
             transferFrom: 2 NEAR wNEAR moved from victim → engine implicit addr (COMMITTED)
             Promise chain created: [deploy] → [factory_update_address_version] → [withdraw] → [send_refund] → [victim call]

  Receipt 2: Attacker calls fund_xcc_sub_account(target=victim, wnear_account_id=None)
             (OR: victim's second EVM tx calls XCC precompile again)
             get_code_version_of_address(victim) → None  ← STALE STATE
             transferFrom: 2 NEAR wNEAR moved from victim → engine implicit addr (COMMITTED)
             Second promise chain created: [deploy] → [factory_update_address_version] → [withdraw] → [send_refund] → [victim call]

Block N+1:
  Attacker's deploy batch: CreateAccount(victim.aurora) → SUCCESS. Router created.
  Attacker's factory_update_address_version: sets version. SUCCESS.

  Victim's deploy batch: CreateAccount(victim.aurora) → FAIL (account exists).
  Victim's factory_update_address_version: promise_result_check() = Some(false) → ERR_ROUTER_DEPLOY_FAILED
  Victim's withdraw_wnear_to_router: promise_result_check() = Some(false) → ERR_CALLBACK_OF_FAILED_PROMISE
  Victim's send_refund: require_no_failed_promises() → panic
  Victim's actual call: never executes.

Result: Victim permanently loses 2 NEAR worth of wNEAR.
        Funds sit in engine's implicit EVM address with no recovery path.
```

pub fn factory_update_address_version<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &H,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // The function is only set to be private, otherwise callback error will happen.
        env.assert_private_call()?;
        let check_deploy: Result<(), &[u8]> = match handler.promise_result_check() {
            Some(true) => Ok(()),
            Some(false) => Err(b"ERR_ROUTER_DEPLOY_FAILED"),
            None => Err(b"ERR_ROUTER_UPDATE_NOT_CALLBACK"),
        };
        check_deploy?;
        let args: xcc::AddressVersionUpdateArgs = io.read_input_borsh()?;
        xcc::set_code_version_of_address(&mut io, &args.address, args.version);
        Ok(())
    })

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
