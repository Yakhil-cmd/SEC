Audit Report

## Title
Arbitrary `buyer` Field in `refresh_buyer_tokens` Enables Confirmation-Text Consent Bypass and Forced Participation — (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The SNS Swap canister's `refresh_buyer_tokens` update endpoint resolves the effective buyer from a caller-controlled payload field rather than from the authenticated ingress sender. Because `validate_confirmation_text` only checks that the supplied text matches the swap's configured text without verifying caller-vs-buyer identity, any unprivileged principal can register an arbitrary third-party principal as a swap participant and record that principal as having accepted the mandatory `confirmation_text` — without that principal ever sending a message to the swap canister. This renders the informed-consent gate ineffective and enables forced participation and swap-outcome manipulation.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 128–143, the handler resolves the effective buyer from the request payload:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // caller-controlled
};
swap_mut()
    .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
    .await
```

No check is performed to verify that the supplied `buyer` matches `caller_principal_id()`.

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` lines 1134–1310), the only consent-related check is `validate_confirmation_text` (lines 363–384), which compares the supplied text against the swap's configured text:

```rust
(Some(expected_text), Some(text)) => {
    if &text != expected_text {
        Err("...does not match...".to_string())
    } else {
        Ok(())
    }
}
```

This check is purely textual — it does not bind the confirmation to the authenticated caller. After passing this check, the function queries the ICP ledger balance of `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(&buyer) }` (lines 1153–1163), where `buyer` is the attacker-supplied value. If the balance meets `min_participant_icp_e8s`, the function inserts `buyer` into `self.buyers` (lines 1285–1288) and calls `update_total_participation_amounts` (line 1291). The `buyers` map is keyed by `buyer.to_string()`, and `min_participation_reached` counts `self.buyers.len()` (line 2817), so artificially inserted entries directly affect swap-commitment logic.

**Attack Path A — zero-cost forced registration:**
1. Victim Bob transfers `min_participant_icp_e8s` ICP to the swap's subaccount derived from his principal (the normal first step of participation).
2. Before Bob calls `refresh_buyer_tokens`, attacker Alice calls it with `buyer = Bob` and `confirmation_text = <swap's required text>`.
3. Bob is registered as a participant; his ICP is locked; the swap records Bob as having accepted the confirmation text — without Bob ever sending a message to the swap canister.

**Attack Path B — attacker-funded manipulation:**
1. Alice transfers `min_participant_icp_e8s` ICP to the swap's subaccount for any target principal T.
2. Alice calls `refresh_buyer_tokens` with `buyer = T` and the correct `confirmation_text`.
3. T is registered as a participant. Repeating this across many principals can push the swap to `max_direct_participation_icp_e8s`, triggering early commitment and excluding legitimate participants who arrive after the cap is hit.

No existing guard prevents either path. The `validate_lifecycle_is_open` and `validate_possibility_of_direct_participation` checks are unrelated to caller-vs-buyer identity.

## Impact Explanation
This is a **High** severity finding. The `confirmation_text` field is the SNS's explicit informed-consent gate, documented as requiring each participant to confirm they have read the swap's terms before their ICP is committed. The vulnerability renders this gate completely ineffective: any caller who knows the public confirmation text (which is stored in the swap's init payload and readable by anyone) can satisfy it on behalf of any principal. Additionally, Attack Path A enables forced participation — a victim's ICP is locked in the swap without the victim ever interacting with the swap canister. Attack Path B enables swap-outcome manipulation by artificially inflating participant counts and pushing the swap to early commitment, directly harming legitimate participants who are excluded after the cap is reached. This constitutes a significant SNS security impact with concrete user and protocol harm, matching the High ($2,000–$10,000) impact class for SNS framework security issues.

## Likelihood Explanation
`refresh_buyer_tokens` is a public, unauthenticated update call reachable by any ingress sender with no role, key, or privileged access required. Attack Path A costs the attacker nothing beyond a single update call and requires only that the victim has already transferred ICP to their subaccount — which is the standard, publicly observable first step of participation. Attack Path B requires the attacker to spend `min_participant_icp_e8s` ICP per target, but the attacker receives SNS neurons in return on finalization, making the net cost only the opportunity cost of locked ICP. The confirmation text is publicly readable from the swap's init state. The attack is repeatable, requires no victim interaction, and is trivially scriptable.

## Recommendation
Enforce that the effective buyer is always the authenticated ingress sender, or — if third-party notification is intentionally supported — prohibit the caller from supplying `confirmation_text` on behalf of a different principal:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() && arg.confirmation_text.is_some() {
        ic_cdk::trap("confirmation_text may not be submitted on behalf of another principal");
    }
    specified
};
```

Alternatively, require that `buyer`, when non-empty, must equal the caller, fully eliminating the third-party registration path.

## Proof of Concept
Minimal unit test plan against `refresh_buyer_token_e8s` directly (no canister deployment required):

```rust
#[test]
fn test_third_party_confirmation_bypass() {
    let confirmation_text = "I confirm I have read the terms.".to_string();
    let victim = PrincipalId::new_user_test_id(1);
    let attacker = PrincipalId::new_user_test_id(2);
    let amount = 101 * E8;

    let mut swap = SwapBuilder::new()
        .with_lifecycle(Open)
        .with_confirmation_text(confirmation_text.clone())
        .build();

    // Attacker calls refresh_buyer_token_e8s with buyer=victim and correct confirmation_text.
    // The ledger mock returns a balance for victim's subaccount (simulating victim's prior transfer).
    let result = swap.refresh_buyer_token_e8s(
        victim,                          // buyer = victim, not attacker
        Some(confirmation_text.clone()), // confirmation_text supplied by attacker
        SWAP_CANISTER_ID,
        &mock_stub(vec![LedgerExpect::AccountBalance(
            Account {
                owner: SWAP_CANISTER_ID.get().into(),
                subaccount: Some(principal_to_subaccount(&victim)),
            },
            Ok(Tokens::from_e8s(amount)),
        )]),
    )
    .now_or_never()
    .unwrap();

    // Victim is registered as a participant without ever calling the swap canister.
    assert!(result.is_ok());
    assert!(swap.buyers.contains_key(&victim.to_string()));
}
```

This test will pass against the current codebase, confirming the bypass.
