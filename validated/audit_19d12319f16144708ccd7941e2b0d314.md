Audit Report

## Title
Permanent Slot Leak via `.unwrap()` Trap in Public `canister_status` Endpoint — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary
The public `canister_status` endpoint in NNS Root calls `.unwrap()` on the management canister response after the inter-canister call returns. When the management canister rejects the call (e.g., NNS Root is not a controller of the target canister), the callback execution traps. Because the IC's execution model rolls back all heap state changes made during a callback that traps, the `SlotLoan` restoration — which increments `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` — is permanently undone. An unprivileged attacker can drain all 167 slots with 167 sequential calls, after which the zero-slot guard blocks all management canister operations routed through `new_management_canister_client()` in NNS Root.

## Finding Description

**Entrypoint — no caller restriction:**
`canister_status` is a public `#[update]` with no `check_caller_is_*` guard: [1](#0-0) 

**Slot accounting — decrement happens before the `.await`:**
`try_borrow_slot()` decrements `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` synchronously before the inter-canister call is sent. This decrement is committed as part of Round 1: [2](#0-1) [3](#0-2) 

**`SlotLoan` Drop — restores the slot on normal completion:** [4](#0-3) 

**The two-round atomicity problem:**

- **Round 1:** `try_borrow_slot()` runs, slot count decrements (167→166), the inter-canister call is enqueued. Round 1 commits; the canister's persisted state now has slot count = 166.
- **Round 2 (callback):** The management canister rejects (NNS Root is not a controller of the target). In Rust-compiled-to-Wasm, `panic!()` / `.unwrap()` on `Err` compiles to a Wasm `unreachable` trap. Wasm traps do **not** unwind the stack, so `SlotLoan::drop` is never called. The trap causes the IC to roll back all Round 2 state changes. The committed slot count remains 166.

Even if Drop were called before the trap (e.g., in a future runtime that does unwind), the IC rollback of Round 2 would still revert the restoration, yielding the same net result: slot count = 166.

**Zero-slot guard blocks VIPs too:**
When `available_slot_count == 0`, the guard returns `Err` for all callers before the VIP-aware subtraction is reached: [5](#0-4) 

VIPs have `used_slot_count = 0` but the `if *available_slot_count == 0` branch fires unconditionally, blocking them regardless.

**Initial slot count:** [6](#0-5) 

**Public interface confirmation:** [7](#0-6) 

## Impact Explanation

Once all 167 slots are drained, every call to `new_management_canister_client()` inside NNS Root returns `Err(SysTransient, "Unavailable")`. This permanently blocks the following NNS Root endpoints until the canister is upgraded:

- `change_canister_controllers` (SNS-W → NNS Root → management canister)
- `update_canister_settings` (NNS Governance → NNS Root → management canister)
- `take_canister_snapshot` / `load_canister_snapshot` (NNS Governance → NNS Root → management canister)

Note: `change_nns_canister` uses `ic_cdk::api::call::call` directly (bypassing `new_management_canister_client()`), so the upgrade/recovery path via a governance proposal remains available. This constrains the impact to a persistent but recoverable DoS of the affected NNS Root operations.

This matches: **High ($2,000–$10,000) — Significant NNS security impact with concrete user or protocol harm**, specifically the disruption of NNS Governance proposals that depend on the blocked management canister operations, with no self-recovery and requiring an out-of-band governance upgrade to remediate.

## Likelihood Explanation

The attack requires no privileges, no special cycles beyond ingress fees, and no coordination. Any principal (including the anonymous principal) can call `canister_status` with an arbitrary `CanisterIdRecord`. Supplying any canister ID not controlled by NNS Root causes the management canister to reject, triggering the trap. 167 sequential ingress calls suffice; concurrency is not required. The `canister_status` endpoint is publicly documented in `root.did`. [7](#0-6) 

## Recommendation

Replace `.unwrap()` with explicit error propagation so the function returns an error to the caller without trapping. This ensures `SlotLoan::drop` executes and the slot is restored within the same Round 2 execution, which the IC commits normally:

```rust
#[update]
async fn canister_status(canister_id_record: CanisterIdRecord) -> Result<CanisterStatusResult, String> {
    let client = new_management_canister_client();
    client
        .canister_status(canister_id_record)
        .await
        .map(CanisterStatusResult::from)
        .map_err(|(code, msg)| format!("Management canister error {code}: {msg}"))
}
```

This eliminates the trap path entirely: on `Err`, `_loan` drops (slot restored), the function returns `Err`, and Round 2 commits with the slot count incremented back.

## Proof of Concept

State-machine test outline using PocketIC or the IC state-machine testing framework:

1. Deploy NNS Root canister with its standard initialization.
2. Create a canister `C` whose controller is **not** NNS Root.
3. From the anonymous principal, call `canister_status` on NNS Root with `C`'s ID.
4. Assert the call returns a replica-level rejection (trap propagated as rejection).
5. Read `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` via the `/metrics` HTTP endpoint or a test hook; assert it equals 166.
6. Repeat steps 3–5 a total of 167 times; assert the count reaches 0.
7. From NNS Governance, call `update_canister_settings` on NNS Root; assert it fails with `SysTransient / "Unavailable"`.
8. Confirm `change_nns_canister` (the recovery path) still succeeds, demonstrating the DoS is recoverable via upgrade.
