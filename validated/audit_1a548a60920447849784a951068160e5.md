Audit Report

## Title
SNS Governance Reward Purse Permanently Destroyed When `total_reward_shares` Is Zero ÔÇö (File: `rs/sns/governance/src/governance.rs`)

## Summary
In `distribute_rewards`, when all ballots on `ReadyToSettle` proposals carry `voting_power = 0` (or all votes are `Unspecified`), `total_reward_shares` equals zero, maturity distribution is skipped, yet the proposals are still written into `settled_proposals` of the new `RewardEvent`. Because `rewards_rolled_over()` gates on `settled_proposals.is_empty()`, the entire `rewards_purse_e8s` ÔÇö including any accumulated rollover from prior rounds ÔÇö is silently discarded with no recovery path. The developers confirmed this exact outcome in the `zero_total_reward_shares` integration test.

## Finding Description
**Root cause ÔÇö `distribute_rewards` in `rs/sns/governance/src/governance.rs`:**

`rewards_purse_e8s` is computed at lines 5854ÔÇô5875, incorporating any rolled-over amount from the previous event plus the current round's inflation-derived maturity. At lines 5892ÔÇô5934, `total_reward_shares` is the sum of `ballot.voting_power` for every `Yes`/`No` ballot across all `ReadyToSettle` proposals. When every such ballot has `voting_power = 0`, `total_reward_shares == dec!(0)`.

At lines 5946ÔÇô5953, the distribution branch is skipped entirely, leaving `distributed_e8s_equivalent = 0`. Regardless, at lines 6083ÔÇô6092, the function unconditionally writes a new `RewardEvent` with `settled_proposals: considered_proposals` (non-empty) and `total_available_e8s_equivalent` set to the full purse value.

**Rollover gate ÔÇö `rs/sns/governance/src/types.rs` lines 2054ÔÇô2067:**

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty()
}
pub(crate) fn e8s_equivalent_to_be_rolled_over(&self) -> u64 {
    if self.rewards_rolled_over() { self.total_available_e8s_equivalent.unwrap_or_default() }
    else { 0 }
}
```

Because `settled_proposals` is non-empty, `e8s_equivalent_to_be_rolled_over()` returns `0`. On the next call to `distribute_rewards`, `rewards_purse_e8s` starts from zero rollover, and the prior round's purse is gone permanently. There is no admin escape hatch, no governance proposal to recover it, and no retry path.

**Developer-confirmed behavior ÔÇö `rs/sns/integration_tests/src/neuron.rs:1167`:**

The `zero_total_reward_shares` test constructs a neuron with `cached_neuron_stake_e8s == neuron_fees_e8s` (net stake = 0, voting power = 0), has it vote `Yes`, and asserts `distributed_e8s_equivalent == 0` with `settled_proposals = [proposal_id]`. The test was written to verify no divide-by-zero panic, not to verify reward preservation, and it implicitly accepts the purse loss.

**Two realistic trigger paths:**
1. All voting neurons have `neuron_fees_e8s >= cached_neuron_stake_e8s` (depleted neurons, possible in early-stage or fee-heavy SNS instances).
2. A proposal's voting period expires with zero `Yes`/`No` votes (all ballots remain `Unspecified`); this can occur naturally in low-activity SNS instances and requires no attacker action.

## Impact Explanation
The entire `rewards_purse_e8s` for the affected round ÔÇö which includes the current round's inflation-derived maturity plus any accumulated rollover from prior rounds where no proposals were settled ÔÇö is permanently destroyed. Neurons that should have received maturity receive nothing, and the shortfall does not carry forward. For an SNS with a large accumulated rollover (e.g., many consecutive rounds with no proposals), a single zero-voting-power round can wipe out a substantial maturity pool. This constitutes a **significant SNS security impact with concrete user and protocol harm**, matching the High ($2,000ÔÇô$10,000) bounty tier: "SignificantÔÇª SNSÔÇª security impact with concrete user or protocol harm."

## Likelihood Explanation
The trigger condition fires automatically via the `run_periodic_tasks` heartbeat ÔÇö no attacker action is required once the degenerate state exists. The "universal abstention" path (no neuron votes before the deadline) is reachable on any low-activity SNS without any special privileges. The "depleted neurons" path requires a specific neuron state but is also unprivileged. The condition is not common on healthy, active SNS instances, placing likelihood at **Low**, consistent with the claim.

## Recommendation
When `total_reward_shares == dec!(0)` and `considered_proposals` is non-empty, treat the round as a rollover rather than a settled event. Concretely, either:
- Omit the non-empty `considered_proposals` from `settled_proposals` in the `RewardEvent` (leave it empty so `rewards_rolled_over()` returns `true` and the purse is carried forward), or
- Explicitly preserve `total_available_e8s_equivalent` in the rollover path so `e8s_equivalent_to_be_rolled_over()` returns the full purse to the next round.

This mirrors the existing rollover logic that already handles the "no proposals to settle" case correctly.

## Proof of Concept
The `zero_total_reward_shares` test at `rs/sns/integration_tests/src/neuron.rs:1167` is a direct, self-contained proof:
1. Neuron with `cached_neuron_stake_e8s = neuron_fees_e8s = 1_000_000_000` ÔåÆ `voting_power = 0`.
2. Proposal created; neuron votes `Yes` with `voting_power = 0`.
3. `run_periodic_tasks()` called.
4. Result: `settled_proposals = [proposal_id]`, `distributed_e8s_equivalent = 0`, `total_available_e8s_equivalent > 0`.
5. On the next `run_periodic_tasks()` call, `e8s_equivalent_to_be_rolled_over()` returns `0` for the previous event (because `settled_proposals` was non-empty), confirming the purse is permanently lost.
