Audit Report

## Title
Unbounded Loop in `distribute_rewards` Due to Insufficient Minimum `round_duration_seconds` Enforcement Causes Instruction-Limit DoS - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's `distribute_rewards` function computes `new_rounds_count = (now - reward_start_timestamp_seconds) / round_duration_seconds` and iterates over every missed round with no upper bound. Because `round_duration_seconds` is validated only to be ≥ 1 second, and a fresh SNS has `reward_start_timestamp_seconds = 0`, setting `round_duration_seconds = 1` at deployment yields `new_rounds_count ≈ 1.75 billion`, exhausting the IC per-message instruction budget and permanently trapping the canister on every subsequent timer invocation.

## Finding Description
In `rs/sns/governance/src/governance.rs` lines 5808–5814, `reward_start_timestamp_seconds` is taken from `latest_reward_event().end_timestamp_seconds.unwrap_or_default()`, which resolves to `0` for a freshly deployed SNS whose genesis `RewardEvent` has `end_timestamp_seconds = None`. `new_rounds_count` is then `now / round_duration_seconds` with no cap.

The only guard (lines 5815–5820) is a zero-check:
```rust
if new_rounds_count == 0 { return; }
```

The loop at lines 5861–5872 iterates unconditionally over all `new_rounds_count` rounds, performing floating-point arithmetic per iteration, with no `is_over_instructions_limit()` check or early-exit mechanism.

The validation in `rs/sns/governance/src/reward.rs` lines 254–260 enforces only `1..=*MAX_REWARD_ROUND_DURATION_SECONDS` (1 s to ~31.5 M s). A value of `1` passes this check. With `round_duration_seconds = 1` and `reward_start_timestamp_seconds = 0`, `new_rounds_count ≈ 1_750_000_000`, which exhausts the ~5 billion instruction budget. Because `latest_reward_event` is updated only on successful completion of `distribute_rewards`, the trap is permanent: every subsequent timer invocation recomputes the same enormous `new_rounds_count` and traps again.

## Impact Explanation
This matches the High bounty impact: **Application/platform-level DoS with concrete user and protocol harm**. The SNS governance canister enters a permanent DoS state: voting rewards are never distributed, and the entire `run_periodic_tasks` periodic execution path is broken. Token holders who have staked neurons in the SNS are permanently denied rewards, and any other periodic governance work sharing the same execution path is also blocked. The impact is scoped to the affected SNS instance but is irreversible without an upgrade.

## Likelihood Explanation
The exploit is triggerable at SNS deployment time by any deployer who sets `voting_rewards_parameters.round_duration_seconds = 1` in the `CreateServiceNervousSystem` NNS proposal — a value that passes all current validation checks. No special cryptographic privilege, governance majority, or subnet-level compromise is required. A malicious deployer could attract users to stake tokens before the first timer fires, then the permanent DoS traps those staked assets. Alternatively, any SNS governance majority can pass a `ManageNervousSystemParameters` proposal to reduce `round_duration_seconds` to 1 post-deployment, triggering the same effect.

## Recommendation
1. **Enforce a meaningful minimum for `round_duration_seconds`** in `round_duration_seconds_defects` — e.g., 3 600 s (1 hour) — so that `new_rounds_count` is bounded to a value that cannot exhaust the instruction limit within a single message execution.
2. **Cap `new_rounds_count` inside `distribute_rewards`** — add an explicit upper bound (e.g., `new_rounds_count.min(MAX_ROUNDS_PER_DISTRIBUTION)`) and emit a warning log when the cap is hit, analogous to the existing `new_rounds_count > 1` log at line 5828.
3. **Add an instruction-limit check inside the loop** — mirror the pattern used in `rs/nns/governance/src/reward/distribution.rs`'s `continue_processing`, which calls `is_over_instructions_limit()` after each iteration and breaks early.

## Proof of Concept
1. Deploy an SNS via `CreateServiceNervousSystem` with `voting_rewards_parameters.round_duration_seconds = 1`. This passes the current `validate()` check since `1 ≥ 1`.
2. The genesis `RewardEvent` has `end_timestamp_seconds = None`, so `reward_start_timestamp_seconds = 0` (via `.unwrap_or_default()`).
3. On the first timer firing, `should_distribute_rewards` returns `true` because `now > 1`.
4. Inside `distribute_rewards`, `new_rounds_count = now / 1 ≈ 1_750_000_000`.
5. The `for i in 1..=new_rounds_count` loop attempts ~1.75 billion iterations of floating-point arithmetic, exhausting the ~5 billion instruction budget and causing `InstructionLimitExceeded`.
6. The canister traps; `latest_reward_event` is not updated; the next timer invocation repeats identically — permanent DoS.

A deterministic integration test using PocketIC can reproduce this by: deploying an SNS with `round_duration_seconds = 1`, advancing the mock clock by 1 second, triggering the heartbeat/timer, and asserting that `latest_reward_event().end_timestamp_seconds` remains `None` (i.e., the distribution never completed).
