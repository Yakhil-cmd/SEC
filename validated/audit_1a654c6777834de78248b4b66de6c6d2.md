### Title
Single-Step Ownership Transfer Allows Permanent Loss of Contract Administrative Control - (File: engine/src/contract_methods/admin.rs)

---

### Summary

The `set_owner` function in Aurora Engine performs an immediate, irreversible single-step ownership transfer with no confirmation from the new owner and no on-chain existence check for the target account. If the current owner accidentally supplies a wrong or non-existent NEAR account ID, all administrative control over the contract is permanently lost with no recovery path.

---

### Finding Description

The `set_owner` function at `engine/src/contract_methods/admin.rs` immediately overwrites `state.owner_id` with the caller-supplied `args.new_owner` in a single atomic step: [1](#0-0) 

The only guards present are:
1. `require_running` — contract must not be paused.
2. `require_owner_only` — caller must be the current owner.
3. A same-owner check (`ERR_SAME_OWNER`). [2](#0-1) 

There is no pending-owner pattern, no confirmation call from the new owner, and no on-chain existence check for the new account ID. The `SetOwnerArgs` struct contains only a single field: [3](#0-2) 

The `AccountId` type validates NEAR account ID format (length, character set) but does **not** verify that the account exists on-chain. A syntactically valid but non-existent or inaccessible account ID will be accepted and committed immediately. The public NEAR entry point is exposed unconditionally: [4](#0-3) 

The test `test_set_owner` in the test suite confirms the ownership change takes effect immediately with no confirmation step: [5](#0-4) 

---

### Impact Explanation

The owner is the **sole** account authorized to call the following critical administrative functions:

- `pause_contract` / `resume_contract` — emergency circuit breaker for all EVM transaction processing. [6](#0-5) 
- `pause_precompiles` / `resume_precompiles` — emergency circuit breaker for Aurora-native precompiles (bridge exits, XCC). [7](#0-6) 
- `set_eth_connector_contract_account` — redirects the ETH bridge connector used for all deposit/withdrawal accounting. [8](#0-7) 
- `set_upgrade_delay_blocks`, `set_key_manager`, `attach_full_access_key` — contract upgrade and key management. [9](#0-8) 

If ownership is transferred to a wrong or non-existent NEAR account, all of these capabilities are permanently lost. The contract cannot be paused in an emergency, cannot be upgraded to patch critical bugs, and cannot have its bridge connector corrected. If a critical vulnerability is subsequently discovered and actively exploited, there is no mechanism to halt the exploit. User funds held in the Aurora bridge (bridged ETH and ERC-20 tokens) can be permanently frozen or drained with no recourse.

**Impact: High — Temporary freezing of funds** (loss of emergency pause capability means an active exploit cannot be stopped, effectively freezing or draining user funds until the exploit completes). Escalates to **Critical — Permanent freezing of funds** if the exploit results in an irrecoverable state.

---

### Likelihood Explanation

The owner must manually call `set_owner` with a Borsh-encoded NEAR account ID string. A typo, copy-paste error, or use of a not-yet-created sub-account ID would silently succeed — the transaction returns `Ok(())` with no warning. This is a realistic operational mistake during key rotations, governance transitions, or multisig migrations. The operation is irreversible once committed to state.

---

### Recommendation

Implement a two-step ownership transfer pattern:

1. The current owner calls `propose_new_owner(new_owner: AccountId)`, which stores a `pending_owner` field in `EngineState` without changing the active `owner_id`.
2. Only the account matching `pending_owner` can call `accept_ownership()`, which atomically moves `pending_owner` into `owner_id` and clears the pending field.

This guarantees the new owner account exists and is accessible before control is relinquished, and allows the current owner to cancel a mistaken proposal before it is accepted.

---

### Proof of Concept

1. Current owner calls `set_owner` with `new_owner = "typo-acccount.near"` (syntactically valid NEAR account ID, but non-existent on-chain).
2. The call succeeds: `state.owner_id` is immediately and permanently set to `"typo-acccount.near"`. [10](#0-9) 
3. The original owner attempts to call `pause_contract` — receives `ERR_NOT_ALLOWED` because `require_owner_only` now checks against `"typo-acccount.near"`. [11](#0-10) 
4. `"typo-acccount.near"` does not exist and cannot sign any NEAR transaction.
5. All owner-gated administrative controls are permanently inaccessible.
6. A subsequently discovered critical precompile or bridge bug cannot be paused, and user funds are frozen or stolen with no recovery path.

#[named]
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;

        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;
        state::set_state(&mut io, &state)?;

        Ok(())
    })
}

#[test]
fn test_set_owner() {
    let mut runner = utils::deploy_runner();
    let aurora_account_id = runner.aurora_account_id.clone();

    // set owner args
    let set_owner_args = SetOwnerArgs {
        new_owner: str_to_account_id("new_owner.near"),
    };

    let result = runner.call(
        "set_owner",
        &aurora_account_id,
        borsh::to_vec(&set_owner_args).unwrap(),
    );

    // setting owner from the owner with same owner id should succeed
    assert!(result.is_ok());

    // get owner to see if the owner_id property has changed
    let outcome = runner
        .one_shot()
        .call("get_owner", &aurora_account_id, vec![])
        .unwrap();

    // check if the owner_id property has changed to new_owner.near
    assert_eq!(
        b"new_owner.near",
        outcome.return_data.as_value().unwrap().as_slice()
    );
}
