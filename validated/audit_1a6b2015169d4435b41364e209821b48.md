### Title
Accumulated `total_unspent_tx_fees` ETH Permanently Locked in ckETH Minter with No Recovery Mechanism - (File: `rs/ethereum/cketh/minter/src/state.rs`)

---

### Summary

The ckETH minter charges users a conservative `max_tx_fee_estimate` upfront for every `withdraw_eth` and `withdraw_erc20` call. After each Ethereum transaction finalizes, the difference between the charged estimate and the actual gas cost (`unspent_tx_fee = max_tx_fee_estimate - actual_tx_fee`) accumulates in the minter's Ethereum address. The minter tracks this in `EthBalance::total_unspent_tx_fees` but provides no endpoint — neither user-facing nor governance-controlled — to recover or redistribute this ETH. It is permanently locked.

---

### Finding Description

In `update_balance_upon_withdrawal`, after each finalized Ethereum transaction the minter computes:

```rust
let unspent_tx_fee = charged_tx_fee.checked_sub(tx_fee)...;
self.eth_balance.eth_balance_sub(debited_amount);   // only debits tx_amount + actual_tx_fee
self.eth_balance.total_effective_tx_fees_add(tx_fee);
self.eth_balance.total_unspent_tx_fees_add(unspent_tx_fee);
``` [1](#0-0) 

For **ckETH withdrawals**: the user burns `withdraw_amount` ckETH; the minter sends `withdraw_amount - max_tx_fee_estimate` ETH to the destination and pays `actual_tx_fee` in gas. The remainder `max_tx_fee_estimate - actual_tx_fee` stays on the minter's Ethereum address, with no corresponding ckETH minted.

For **ckERC20 withdrawals**: the user burns `max_transaction_fee` worth of ckETH upfront; the minter pays only `actual_tx_fee` in gas. The documentation explicitly states: *"Overcharged transaction fees are not reimbursed."* [2](#0-1) 

The `total_unspent_tx_fees` field is only ever incremented, never decremented: [3](#0-2) 

The minter's full Candid interface exposes no endpoint to withdraw ETH without burning ckETH, and no admin/governance endpoint to recover accumulated unspent fees: [4](#0-3) 

The dashboard tracks and displays `total_unspent_tx_fees` as a metric, confirming it is a known, growing quantity: [5](#0-4) 

---

### Impact Explanation

Every finalized ckETH or ckERC20 withdrawal contributes a non-zero `unspent_tx_fee` to the minter's Ethereum address. Because:

1. `withdraw_eth` requires burning ckETH — but no ckETH was ever minted for the unspent fee portion, so it can never be withdrawn via the normal user flow.
2. There is no privileged or governance endpoint to issue an Ethereum transaction from the minter's address without a corresponding ckETH burn.

The ETH is permanently locked on the minter's Ethereum address. Over the protocol's lifetime, this represents a growing, irrecoverable loss of ETH value — directly analogous to ERC-20 tokens locked in a Treasury contract with no withdrawal function.

**Vulnerability class:** Ledger conservation bug — funds accumulate in a protocol-controlled address with no recovery path.

---

### Likelihood Explanation

**High.** Every single `withdraw_eth` and `withdraw_erc20` call contributes to `total_unspent_tx_fees`. The minter uses a deliberately conservative fee estimate (with a ≥10% safety margin for resubmissions), so the unspent portion is structurally non-zero on every finalized transaction. No special attacker action is required; normal protocol usage by any unprivileged user continuously increases the locked amount. [6](#0-5) 

---

### Recommendation

- **Short term:** Add a governance-gated endpoint (callable only by the NNS or the ledger suite orchestrator) that allows issuing an Ethereum transaction from the minter's address to a designated fee-collector address, without requiring a ckETH burn, specifically for recovering accumulated `total_unspent_tx_fees`.
- **Long term:** Consider reimbursing users for the overcharged portion after each transaction finalizes (mint the `unspent_tx_fee` back as ckETH to the original withdrawer), or document and enforce an explicit protocol-level policy for how these funds are managed.

---

### Proof of Concept

1. User calls `withdraw_eth` with `withdraw_amount = 39_998_000_000_000_000` wei (as documented).
2. Minter charges `max_tx_fee_estimate = 1_823_126_598_888_000` wei; actual fee is `899_399_014_248_000` wei.
3. `unspent_tx_fee = 923_727_584_640_000` wei (~0.00092 ETH) is added to `total_unspent_tx_fees` and remains on the minter's Ethereum address.
4. No ckETH was minted for this amount; `withdraw_eth` cannot be called to retrieve it (no ckETH to burn).
5. Inspecting `cketh_minter.did` confirms no endpoint exists to recover this ETH without a corresponding ckETH burn.
6. After N withdrawals, `total_unspent_tx_fees` = Σ(unspent fees) — permanently locked, observable via the dashboard metric `cketh_minter_total_unspent_tx_fees`. [7](#0-6) [8](#0-7)

```text
. Estimate the maximum current cost of a transaction on Ethereum, say `max_tx_fee_estimate`. This `max_tx_fee_estimate` is expected to be large enough to be valid for the few next blocks.
. Issue an Ethereum transaction (via threshold ECDSA) with the value `withdraw_amount - max_tx_fee_estimate`. This requires of course that `withdraw_amount >= max_tx_fee_estimate` and that's why we currently have a conservative minimum value for withdrawals of `30_000_000_000_000_000` wei. This ensures that the minter can always send the transaction to Ethereum if one or several resubmissions are needed if the Ethereum network is congested and fees are increasing rapidly (each resubmission requires an increase of at least 10% of the transaction fee).
. When the transaction is mined, the destination of the transaction will receive `withdraw_amount - max_tx_fee_estimate`. Since on Ethereum transactions are paid by the sender, the minter’s account will be charged with
+
----
(withdraw_amount - max_tx_fee_estimate) + actual_tx_fee == withdrawal_amount - (max_tx_fee_estimate - actual_tx_fee),
----
where `actual_tx_fee` represents the actual transaction fee (can be retrieved from the transaction receipt) and by construction `max_tx_fee_estimate - actual_tx_fee > 0`.
```
