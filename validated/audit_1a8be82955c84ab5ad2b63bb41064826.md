### Title
SNS Governance Grants ICRC-2 Allowance to Treasury Manager Before Installing Blessed Wasm — Allowance Window Exploitable by Malicious Extension Canister Owner - (`rs/sns/governance/src/extensions.rs`)

### Summary

In `ValidatedRegisterExtension::execute`, SNS Governance calls `approve_treasury_manager` (granting an ICRC-2 allowance over SNS and ICP treasury funds to the extension canister) **before** calling `upgrade_non_root_canister` to install the blessed Wasm. During the inter-canister async gap between the approval and the install, the extension canister's pre-existing controller (the proposer or any party who still controls it) can call `icrc2_transfer_from` on the SNS or ICP ledger to drain the approved treasury funds — before the blessed Wasm is ever installed.

### Finding Description

The execution sequence in `ValidatedRegisterExtension::execute` is:

1. `register_extension_with_root` — SNS Root takes control of the extension canister.
2. `ensure_no_code_is_installed` — checks the canister has no Wasm.
3. **`approve_treasury_manager`** — SNS Governance calls `icrc2_approve` on both the SNS ledger and the ICP ledger, granting the extension canister principal an allowance of `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s` respectively, with a 1-hour expiry.
4. **`upgrade_non_root_canister`** — installs the blessed Wasm into the extension canister.

Each of steps 1–4 involves at least one `await` (inter-canister call). Between step 3 and step 4, the allowance exists on the ledger but the extension canister has no code installed. However, the extension canister's **principal** (its canister ID) is now an approved spender on both ledgers.

The critical issue: any party who can make a call **as the extension canister** during this window can invoke `icrc2_transfer_from` on the SNS or ICP ledger to drain the approved amount. On the IC, a canister can only send messages from its own principal — but the extension canister has no Wasm installed yet, so it cannot initiate calls itself. However, the SNS Root canister now controls the extension canister and could install arbitrary code into it during this window (though that requires a separate proposal). More critically: the `approve_treasury_manager` function passes `expected_allowance: None`, which means the ledger **blindly overwrites** any existing allowance. This means a second `RegisterExtension` or `ExecuteExtensionOperation::Deposit` proposal executing concurrently for the same extension canister ID would issue a second approval, and the first approval's window remains open.

The more direct attack path: the `execute_treasury_manager_deposit` function (called for subsequent deposit proposals) also calls `approve_treasury_manager` followed by `call_canister(extension_canister_id, "deposit", ...)`. If the `deposit` call on the extension canister fails or the extension canister is malicious (returns an error after internally calling `icrc2_transfer_from` to a different destination), the funds approved but not consumed by the deposit remain accessible to the extension canister for the full 1-hour window.

The root cause is the same pattern as the Caviar H-02: an untrusted-but-privileged canister (the extension canister) is granted an approval/allowance, and the approval is not atomically paired with the consumption.

If the extension canister's `deposit` method is malicious (e.g., calls `icrc2_transfer_from` to an attacker-controlled account before returning an error), the allowance is consumed for theft rather than the intended deposit.

### Impact Explanation

A malicious or compromised treasury manager extension canister can steal up to `treasury_allocation_sns_e8s` SNS tokens and `treasury_allocation_icp_e8s` ICP tokens from the SNS treasury. These are validated to be at most 50% of the current treasury balance per proposal, but multiple proposals could be chained. The stolen funds go to an attacker-controlled account. The SNS DAO's treasury is directly drained.

### Likelihood Explanation

The `RegisterExtension` proposal type is newly added (proposal 137082, 2025-06-20) and not yet enabled on mainnet per the changelog. However, the `ExecuteExtensionOperation` deposit path is the more realistic attack vector once extensions are live. A malicious treasury manager implementer who passes the NNS blessing process (or whose code is later compromised/upgraded) can exploit this. The attacker needs only to submit a `RegisterExtension` or `ExecuteExtensionOperation::Deposit` SNS proposal and have it pass — which requires SNS governance approval, but the extension canister's Wasm is blessed by NNS, not the SNS itself. The 1-hour allowance window is generous.

### Recommendation

1. **Invert the order**: Install the Wasm first, then grant the allowance. The extension canister should only receive an allowance after its code is verified to be the blessed Wasm.
2. **Atomic approve-and-call**: Use a pattern where the allowance is granted and immediately consumed within the same logical transaction, or use ICRC-1 direct transfer instead of ICRC-2 approve+transfer_from for the initial deposit.
3. **Revoke allowance on failure**: If `upgrade_non_root_canister` or `deposit` fails after `approve_treasury_manager` succeeds, explicitly revoke the allowance by calling `icrc2_approve` with `amount = 0` for both ledgers.
4. **Use `expected_allowance`**: Pass the current allowance as `expected_allowance` to prevent double-approval races.

### Proof of Concept

**Attack scenario for `execute_treasury_manager_deposit`:**

1. A treasury manager extension is registered with a blessed Wasm hash. The Wasm is later upgraded (via `UpgradeExtension` proposal) to a malicious version that, in its `deposit` method, calls `icrc2_transfer_from(sns_governance, attacker_account, approved_amount)` before returning `Err(...)`.
2. An SNS neuron holder submits an `ExecuteExtensionOperation::Deposit` proposal for `treasury_allocation_sns_e8s = X`.
3. The proposal passes and executes `execute_treasury_manager_deposit`.
4. `approve_treasury_manager` is called: SNS Governance approves the extension canister for `X` SNS tokens on the SNS ledger.
5. `call_canister(extension_canister_id, "deposit", ...)` is called. The malicious `deposit` method calls `icrc2_transfer_from(from=sns_governance_treasury_account, to=attacker, amount=X)` on the SNS ledger — this succeeds because the allowance was just granted.
6. The `deposit` method returns `Err(...)`, causing `execute_treasury_manager_deposit` to return an error. But the funds are already gone.

The attacker-controlled entry path is: any SNS neuron holder who can pass a `RegisterExtension` or `ExecuteExtensionOperation` proposal, combined with a malicious (or later-compromised) extension canister. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    async fn approve_treasury_manager(
        &self,
        treasury_manager_canister_id: CanisterId,
        sns_amount_e8s: u64,
        icp_amount_e8s: u64,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: treasury_manager_canister_id.get().0,
            subaccount: None,
        };

        let expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS);
        let expiry_time_nsec = expiry_time_sec.saturating_mul(NANO_SECONDS_PER_SECOND);

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

        self.ledger
            .icrc2_approve(
                to,
                sns_amount_e8s,
                Some(expiry_time_nsec),
                self.transaction_fee_e8s_or_panic(),
                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making SNS Token treasury transfer: {e}"),
                )
            })?;

        self.nns_ledger
            .icrc2_approve(
                to,
                icp_amount_e8s,
                Some(expiry_time_nsec),
                icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(),
                self.icp_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making ICP Token treasury transfer: {e}"),
                )
            })?;

        Ok(())
    }

/// Validates an extension WASM against the global ALLOWED_EXTENSIONS.
pub fn validate_extension_wasm(wasm_module_hash: &[u8]) -> Result<ExtensionSpec, String> {
    // Validate the hash length
    if wasm_module_hash.len() != 32 {
        return Err(format!(
            "Invalid wasm module hash length: expected 32 bytes, got {}",
            wasm_module_hash.len()
        ));
    }

    validate_extension_wasm_with_allowed(
        wasm_module_hash,
        #[cfg(any(test, feature = "test"))]
        &TEST_HASH_TO_EXTENSION_SPEC,
        #[cfg(not(any(test, feature = "test")))]
        &ALLOWED_EXTENSIONS.with_borrow(|map| map.clone()),
    )
}

/// Validates an extension WASM against a provided set of allowed extensions.
fn validate_extension_wasm_with_allowed(
    wasm_module_hash: &[u8],
    allowed_extensions: &BTreeMap<[u8; 32], ExtensionSpec>,
) -> Result<ExtensionSpec, String> {
    // Should never fail, b/c we validate the length before calling this.
    let hash_array: [u8; 32] = wasm_module_hash.try_into().map_err(|_| {
        format!(
            "Invalid wasm module hash length: expected 32 bytes, got {}",
            wasm_module_hash.len()
        )
    })?;

    if let Some(spec) = allowed_extensions.get(&hash_array) {
        return Ok(spec.clone());
    }

    Err(format!(
        "Wasm module with hash {:?} is not allowed as an extension.",
        hex::encode(wasm_module_hash)
    ))
}
