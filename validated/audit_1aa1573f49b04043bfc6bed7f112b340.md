### Summary

`Bls12381G1AdditionPrecompile` and `Bls12381G2AdditionPrecompile` accept curve points that are on the curve but outside the prime-order subgroup (torsion points). This directly contradicts EIP-2537, which mandates subgroup membership for all inputs, and is inconsistent with the MSM and Pairing precompiles in the same codebase, which do enforce the check.

### Finding Description

`parse_g1_bytes` constructs a point with `G1Affine::new_unchecked` and only calls `is_on_curve()`: [1](#0-0) 

`parse_g1` (used by G1ADD) wraps this with no additional validation: [2](#0-1) 

`Bls12381G1AdditionPrecompile` calls `parse_g1` for both inputs â€” no subgroup check at any point: [3](#0-2) 

By contrast, `parse_g1_with_subgroup_check` exists and explicitly calls `is_in_correct_subgroup_assuming_on_curve()`: [4](#0-3) 

MSM uses `parse_g1_with_subgroup_check`: [5](#0-4) 

Pairing uses `parse_g1_with_subgroup_check`: [6](#0-5) 

The same pattern applies to G2: `parse_g2_bytes` only checks `is_on_curve()` [7](#0-6) , and `Bls12381G2AdditionPrecompile` uses `parse_g2` without a subgroup check. [8](#0-7) 

### Impact Explanation

BLS12-381 G1 has a non-trivial cofactor (`h â‰ˆ 2^126`), so torsion points exist. An attacker can:

1. Construct a point `T` on the BLS12-381 curve that is **not** in the prime-order subgroup (e.g., multiply a valid G1 generator by the cofactor to get a point of order `h`, or use a known torsion point).
2. Call the G1ADD precompile with `T` as one or both inputs.
3. Receive a successful return (no error) with an output that is also outside the prime-order subgroup.

This has two concrete consequences:

- **EIP-2537 non-compliance / Ethereum mainnet divergence**: Ethereum mainnet rejects torsion-point inputs to G1ADD with an error. ZKsync OS silently accepts them and returns a result. Smart contracts deployed on both chains that use G1ADD will observe different behavior, breaking cross-chain protocol assumptions.
- **Downstream cryptographic protocol breakage**: Any smart contract that feeds G1ADD output into a subsequent pairing or MSM call will get a `PointNotInSubgroup` error from those precompiles, since they do enforce the check. A contract that does not re-validate the G1ADD output before using it in a pairing could be manipulated into unexpected states.

### Likelihood Explanation

The attack requires no privileges. Any EOA or contract can call the G1ADD precompile (address `BLS12_G1ADD`) with attacker-controlled input. Constructing a torsion point is straightforward: multiply the G1 generator by the cofactor `h`. The cost is one G1ADD call at 375 gas.

### Recommendation

Replace `parse_g1` with `parse_g1_with_subgroup_check` in `Bls12381G1AdditionPrecompile::execute` and `parse_g2` with `parse_g2_with_subgroup_check` in `Bls12381G2AdditionPrecompile::execute`, mirroring what MSM and Pairing already do.

### Proof of Concept

```
1. Compute T = cofactor * G1_generator  (T is on the curve, not in the prime-order subgroup)
2. Encode T as 128 bytes per EIP-2537 format
3. Call BLS12_G1ADD precompile with input = T || T  (256 bytes)
4. Observe: call succeeds (no revert), output R is returned
5. Assert: R.is_on_curve() == true, R.is_in_correct_subgroup_assuming_on_curve() == false
6. On Ethereum mainnet the same call returns an error
```

fn parse_g1_with_subgroup_check(
    input: &[u8; G1_SERIALIZATION_LEN],
) -> Result<G1Affine, Bls12PrecompileSubsystemError> {
    let point = parse_g1(input)?;
    if point.is_zero() || point.is_in_correct_subgroup_assuming_on_curve() {
        Ok(point)
    } else {
        Err(interface_error!(
            Bls12PrecompileInterfaceError::PointNotInSubgroup
        ))
    }
}
