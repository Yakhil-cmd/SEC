### Title
EIP-7702 Authorization Signature Replay Across ZKsync OS Chains via `chain_id = 0` - (File: `basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

---

### Summary

The `validate_and_apply_delegation` function in ZKsync OS accepts EIP-7702 authorization entries with `chain_id = 0` as valid on any chain. Because the signed message hash commits to the literal value `0` (not the actual chain ID), a single authorization signature produced with `chain_id = 0` can be replayed on every ZKsync OS chain (mainnet, testnet, any future chain) that shares the same authority nonce value. An unprivileged attacker who observes such a signature on one chain can submit it on any other chain to forcibly re-delegate the victim's account code.

---

### Finding Description

In `validate_and_apply_delegation`, step 1 explicitly permits `chain_id = 0`:

```rust
// 1. Check chain id
if !auth_chain_id.is_zero() && auth_chain_id != &U256::from(chain_id) {
    return Ok(false);
}
``` [1](#0-0) 

The signed message hash is computed in `compute_auth_message_signed_hash`, which encodes `auth_chain_id` verbatim — so when `auth_chain_id = 0`, the hash commits to `0`, not to the actual chain ID of the executing chain:

```rust
rlp::apply_number_encoding_to_hash(&auth_chain_id.to_be_bytes::<32>(), hasher);
``` [2](#0-1) 

This is the same structural flaw as the CoreWallet co-signer replay: a signed authorization is not bound to a specific chain, so the same `(authority, nonce, delegation_address, sig)` tuple is valid on every chain that accepts `chain_id = 0`.

The nonce check at step 6 does prevent replay on the *same* chain after the nonce is incremented:

```rust
if account_properties.nonce.0 != auth_nonce {
    return Ok(false);
}
``` [3](#0-2) 

But it provides no protection across chains, because each chain maintains its own independent nonce state. If the authority's nonce on chain B happens to equal the `auth_nonce` embedded in the signature (e.g., both chains start at nonce 0, or the attacker waits until the nonce matches), the replay succeeds.

The `chain_id = 0` behavior is intentional per EIP-7702 (it is meant to allow "chain-agnostic" authorizations), but it creates a concrete replay surface in a multi-chain deployment like ZKsync OS, where the same EOA key controls accounts on multiple chains.

---

### Impact Explanation

An attacker who observes a `chain_id = 0` EIP-7702 authorization signature (e.g., from a public mempool, a cross-chain bridge UI, or a prior transaction on another chain) can:

1. Replay it on any ZKsync OS chain where the authority's nonce matches `auth_nonce`.
2. Force the authority's account code to be set to `0xef0100 || delegation_address` — i.e., redirect all calls to the authority's address to an arbitrary contract chosen by the original signer (or the attacker if they control the `delegation_address` field in the replayed transaction).
3. If the `delegation_address` is a malicious contract, all ETH/token transfers and calls to the victim's address will execute attacker-controlled logic, enabling theft of funds.

The authority's nonce is incremented after each successful delegation, so the replay window is bounded to moments when the nonce matches — but on a freshly deployed chain or after a nonce reset scenario, this is trivially satisfied (nonce = 0).

---

### Likelihood Explanation

- ZKsync OS is a multi-chain system (mainnet, testnet, future chains). Any user who signs a `chain_id = 0` authorization on one chain exposes themselves on all others.
- The `chain_id = 0` path is explicitly accepted by the code and is a documented EIP-7702 feature, so users and wallets may generate such signatures intentionally for "portability."
- The attacker entry path is fully unprivileged: any transaction sender can submit an EIP-7702 type-4 transaction containing a replayed authorization entry. No special role is required.
- The nonce constraint means the attacker must wait for the nonce to match, but on new chains or accounts this is immediate (nonce = 0).

---

### Recommendation

When `auth_chain_id = 0`, the signed message hash should substitute the actual current chain ID rather than encoding `0`. This binds the signature to the executing chain even when the user intended a "chain-agnostic" authorization:

```rust
// In compute_auth_message_signed_hash:
let effective_chain_id = if auth_chain_id.is_zero() {
    U256::from(system.get_chain_id())
} else {
    *auth_chain_id
};
rlp::apply_number_encoding_to_hash(&effective_chain_id.to_be_bytes::<32>(), hasher);
```

Alternatively, reject `chain_id = 0` entirely in `validate_and_apply_delegation` if ZKsync OS does not intend to support cross-chain authorizations.

### Proof of Concept

**Setup:** Two ZKsync OS chains, chain A (id=37) and chain B (id=300). Alice controls EOA `0xAlice` with nonce 0 on both chains.

**Step 1 — Alice signs a chain-agnostic authorization on chain A:**
```
Authorization { chain_id: 0, address: 0xMalicious, nonce: 0 }
sig = Alice.sign(keccak(0x05 || rlp([0, 0xMalicious, 0])))
```
Alice submits this in an EIP-7702 tx on chain A to delegate her account to `0xMalicious`. The delegation succeeds; Alice's nonce on chain A becomes 1.

**Step 2 — Attacker observes the signature from chain A's public state.**

**Step 3 — Attacker replays on chain B:**
```
EIP7702Tx {
  chain_id: 300,  // chain B's tx chain_id (outer tx must match)
  authorization_list: [{
    chain_id: 0,          // inner auth chain_id = 0 → accepted on any chain
    address: 0xMalicious,
    nonce: 0,             // Alice's nonce on chain B is still 0
    sig: <Alice's original sig>
  }]
}
```

The `validate_and_apply_delegation` call on chain B:
- Step 1: `auth_chain_id = 0` → `is_zero()` is true → check passes.
- Step 3: `compute_auth_message_signed_hash` hashes `0` as chain_id → same hash as on chain A → `ecrecover` returns `0xAlice`.
- Step 6: Alice's nonce on chain B = 0 = `auth_nonce` → check passes.
- Step 8: Alice's code on chain B is set to `0xef0100 || 0xMalicious`.
- Step 9: Alice's nonce on chain B becomes 1.

Alice's account on chain B is now delegated to `0xMalicious` without her consent. Any ETH sent to `0xAlice` on chain B will execute `0xMalicious`'s code, enabling fund theft. [4](#0-3) [5](#0-4) [6](#0-5)

fn compute_auth_message_signed_hash<S: EthereumLikeTypes>(
    resources: &mut S::Resources,
    auth_chain_id: &U256,
    auth_nonce: u64,
    delegation_address: &[u8; 20],
    hasher: &mut crypto::sha3::Keccak256,
) -> Result<[u8; 32], TxError> {
    use crate::bootloader::rlp;

    let list_payload_len = rlp::estimate_number_encoding_len(&auth_chain_id.to_be_bytes::<32>())
        + rlp::ADDRESS_ENCODING_LEN
        + rlp::estimate_number_encoding_len(&auth_nonce.to_be_bytes());
    let total_list_len = rlp::estimate_length_encoding_len(list_payload_len) + list_payload_len;
    let encoding_len = 1 + total_list_len;
    crate::bootloader::transaction::charge_keccak(encoding_len, resources)?;
    hasher.update([EIP7702_MAGIC]);
    rlp::apply_list_length_encoding_to_hash(list_payload_len, hasher);
    rlp::apply_number_encoding_to_hash(&auth_chain_id.to_be_bytes::<32>(), hasher);
    rlp::apply_bytes_encoding_to_hash(delegation_address, hasher);
    rlp::apply_number_encoding_to_hash(&auth_nonce.to_be_bytes(), hasher);

    Ok(hasher.finalize_reset())
}
