Audit Report

## Title
SNS Swap `finalize_swap_in_progress` Lock Not Released on Panic — Permanent Token Lock - (File: rs/sns/swap/src/swap.rs)

## Summary

The `finalize` function in `rs/sns/swap/src/swap.rs` acquires a boolean lock (`finalize_swap_in_progress`) before delegating to `finalize_inner`, which spans multiple inter-canister await points. The lock is released only via an explicit `unlock_finalize_swap()` call at the end of `finalize`. Because the IC execution model commits state at each await boundary, a panic in any message after the first await permanently leaves the lock set to `true`, blocking all future `finalize_swap` calls and locking all ICP and SNS tokens in the swap canister.

## Finding Description

**Lock acquisition and release** (lines 1506–1531): `finalize` acquires the lock via `lock_finalize_swap()` (sets `finalize_swap_in_progress = Some(true)`, committed to stable state), then calls `finalize_inner(...).await`. The only release is `self.unlock_finalize_swap()` at line 1531. The code comment at lines 1528–1530 explicitly acknowledges: *"Note, if there is a panic, the lock will not be released."* The `finalize_inner` docstring at lines 1538–1543 repeats this warning.

**Await chain in `finalize_inner`** (lines 1556–1621): The function crosses at least five inter-canister await boundaries — `sweep_icp`, `settle_neurons_fund_participation`, `sweep_sns`, `claim_swap_neurons`, and `set_sns_governance_to_normal_mode`. After each await, the IC commits state. The lock (`Some(true)`) is committed after the very first message boundary.

**Panic path in `claim_swap_neurons`** (lines 1680–1683): After three prior await points have committed the lock to stable state, `claim_swap_neurons` executes synchronously (before its own internal await at line 1706) and calls:

```rust
let neuron_id = neuron_recipe.neuron_id.clone().expect(
    "NeuronRecipe.neuron_id is always set by \
    SnsNeuronRecipe::to_neuron_recipe",
);
```

This `.expect()` panics if `to_neuron_recipe` returns `Ok(neuron_recipe)` with `neuron_id = None`. The panic rolls back only the current message's state changes; the lock committed in the first message is **not** rolled back.

**Additional panic paths** (lines 1661–1662): `init.nns_governance_or_panic()` and `init.transaction_fee_e8s_or_panic()` are called after `init_and_validate()` succeeds, but they are still `_or_panic()` methods that could panic if the validated `Init` is missing those fields in an edge case.

**Lock persistence**: Once stuck at `Some(true)`, every subsequent call to `finalize` returns immediately with `"The Swap canister has finalize_swap call already in progress"` (line 1446). The proto field and its comment (lines 232–240 of `swap.proto`) confirm the only documented recovery is a canister upgrade with a post-upgrade hook.

**Existing guards are insufficient**: The `has_error_message()` checks after each sub-step handle graceful errors but do not protect against panics. There is no `Drop`-based guard or `catch_unwind` equivalent.

## Impact Explanation

If the lock becomes permanently stuck: all ICP contributed by buyers cannot be swept to the SNS governance treasury or refunded; all SNS tokens pre-loaded into the swap canister cannot be distributed; SNS governance remains in `PreInitializationSwap` mode indefinitely. Recovery requires an SNS Swap canister upgrade approved by SNS governance — but governance in `PreInitializationSwap` mode has restricted operations, potentially creating a circular dependency that blocks recovery. This constitutes a significant SNS framework security impact with concrete, permanent user fund harm, matching the High impact category for SNS security impact with concrete user or protocol harm.

## Likelihood Explanation

The `finalize_swap` endpoint is an unprivileged `#[update]` callable by any user. The panic at line 1680 is reachable if `to_neuron_recipe` returns `Ok` with `neuron_id = None` — a condition that could arise from malformed neuron recipe state, a future code regression, or an unexpected edge case in the `investor` variant handling. The structural absence of RAII means any future `.unwrap()` or `.expect()` added anywhere inside `finalize_inner` or its callees (across any of the five await boundaries) would silently introduce the same lock-stuck risk. The developers themselves acknowledge the risk in two separate code comments, confirming awareness but no structural mitigation.

## Recommendation

Replace the manual lock/unlock pattern with a Rust RAII guard:

```rust
struct FinalizeLockGuard<'a>(&'a mut Swap);
impl Drop for FinalizeLockGuard<'_> {
    fn drop(&mut self) {
        self.0.finalize_swap_in_progress = Some(false);
    }
}
```

Acquire the guard immediately after `lock_finalize_swap()` succeeds and let it drop naturally at the end of `finalize`. This ensures the lock is released regardless of panic or normal return, eliminating the entire class of lock-stuck-on-panic bugs.

Additionally, audit all `.expect()` and `_or_panic()` calls inside `finalize_inner` and its callees and replace them with graceful error returns that set `error_message` and return early, consistent with the existing pattern used in `sweep_icp` and `settle_neurons_fund_participation`.

## Proof of Concept

1. Deploy an SNS swap canister with `neuron_recipes` containing at least one recipe whose `to_neuron_recipe()` produces a `NeuronRecipe` with `neuron_id = None` (e.g., by crafting a recipe whose `investor` variant produces a missing `NeuronId`).
2. Advance the swap to `Lifecycle::Committed`.
3. Any user calls `finalize_swap({})`.
4. `finalize` acquires the lock: `finalize_swap_in_progress = Some(true)` — state committed at first message boundary.
5. `sweep_icp` completes (await #1, state committed with lock=true).
6. `settle_neurons_fund_participation` completes (await #2, state committed).
7. `sweep_sns` completes (await #3, state committed).
8. `claim_swap_neurons` is entered; the loop at line 1675 reaches the recipe with `neuron_id = None`; `.expect(...)` at line 1680 fires — **panic**.
9. The IC rolls back state changes from the current message, but the lock committed in step 4 is **not** rolled back.
10. All subsequent calls to `finalize_swap` return `"The Swap canister has finalize_swap call already in progress"`.
11. All ICP and SNS tokens are permanently locked; recovery requires an SNS governance upgrade proposal that may itself be blocked by `PreInitializationSwap` mode.

A deterministic integration test using PocketIC can reproduce this by injecting a malformed `SnsNeuronRecipe` into the swap state before calling `finalize_swap`, then asserting that a second `finalize_swap` call returns the lock-held error message. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    pub async fn finalize(
        &mut self,
        now_fn: fn(bool) -> u64,
        environment: &mut impl CanisterEnvironment,
    ) -> FinalizeSwapResponse {
        // Acquire the lock or return a FinalizeSwapResponse with an error message.
        if let Err(error_message) = self.lock_finalize_swap() {
            return FinalizeSwapResponse::with_error(error_message);
        }

        // The lock is now acquired and asynchronous calls to finalize are blocked.
        // Perform all subactions.
        let finalize_swap_response = self.finalize_inner(now_fn, environment).await;

        if finalize_swap_response.has_error_message() {
            log!(
                ERROR,
                "The swap did not finalize successfully. \n\
                finalize_swap_response: {finalize_swap_response:?}"
            );
        } else {
            log!(
                INFO,
                "The swap finalized successfully. \n\
                finalize_swap_response: {finalize_swap_response:?}"
            );
        }

        // Release the lock. Note, if there is a panic, the lock will
        // not be released. In that case, the Swap canister will need
        // to be upgraded to release the lock.
        self.unlock_finalize_swap();

        finalize_swap_response
    }
