Audit Report

## Title
Bitcoin Adapter Response Duplicate Delivery Validation Bypass — (`File: rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary
`BitcoinPayloadBuilder` computes past-delivered callback IDs in `validate_payload` but discards them before calling `validate_self_validating_payload_impl`, and `validate_self_validating_payload` silently ignores its `_past_payloads` parameter entirely. A Byzantine block proposer below the consensus fault threshold can craft a `SelfValidatingPayload` re-including a `callback_id` already present in a past payload; every honest validator will accept the block, enabling duplicate Bitcoin adapter response delivery to the execution layer.

## Finding Description

**Root cause — asymmetry between build and validate paths:**

`build_payload` (line 341–343) correctly passes `delivered_ids` into `get_self_validating_payload_impl`, which filters at line 129: [1](#0-0) 

`validate_payload` (line 369) computes the same `delivered_ids` set but then calls `validate_self_validating_payload_impl` without forwarding it (line 379–382): [2](#0-1) 

`validate_self_validating_payload` (line 294–301) receives `_past_payloads` with an underscore prefix and passes nothing to the impl: [3](#0-2) 

`validate_self_validating_payload_impl` (line 234–251) performs no duplicate check at all — it only checks for an empty payload and returns the byte size: [4](#0-3) 

The file-level `#![allow(dead_code, unused_variables)]` at line 1 suppresses the Rust compiler warning that would otherwise flag `delivered_ids` as unused: [5](#0-4) 

The existing test suite (`includes_only_responses_for_callback_ids_not_seen_in_past_payloads`, line 291) only exercises `build_payload` with `past_payloads`; there is no test asserting that `validate_payload` rejects a payload whose `callback_id` appears in `past_payloads`: [6](#0-5) 

**Exploit flow:**
1. Block H is finalized containing a `SelfValidatingPayload` with `callback_id = 42`. It becomes a `PastPayload` for subsequent blocks in the same batch window.
2. A malicious proposer at height H+1 crafts a `SelfValidatingPayload` that again includes `callback_id = 42`.
3. Every honest validator calls `validate_payload`; `delivered_ids` is populated with `{42}` but is never forwarded — validation returns `Ok(())`.
4. Block H+1 is finalized. In the IC's pipelined execution model, both blocks can be finalized before execution catches up, so the execution layer receives two `BitcoinAdapterResponse` entries for `callback_id = 42`.

## Impact Explanation

Duplicate delivery of `BitcoinAdapterResponse` messages to Bitcoin-integrated canisters (including the ckBTC minter) can cause the same Bitcoin block data (`GetSuccessors`) or transaction confirmation (`SendTransaction`) to be processed twice. For ckBTC, replaying a `GetSuccessors` response can trigger double-minting of ckBTC tokens, constituting illegal minting of chain-key assets. This maps to: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.**

## Likelihood Explanation

The attacker must be a block proposer on a Bitcoin-enabled subnet. Block proposership rotates among all subnet nodes, so any single compromised or malicious node can exploit this during its proposer turn without requiring a majority or any special cryptographic material. The duplicate `callback_id` is trivially observable from public chain history. The attack is repeatable every time the malicious node is scheduled as proposer.

## Recommendation

1. Extend `validate_self_validating_payload_impl` to accept a `past_callback_ids: &BTreeSet<u64>` parameter and return `InvalidArtifact` if any response's `callback_id` is already in the set.
2. In `BatchPayloadBuilder::validate_payload`, pass the already-computed `delivered_ids` to that function instead of discarding it.
3. In `SelfValidatingPayloadBuilder::validate_self_validating_payload`, derive past callback IDs from `past_payloads` (mirroring the logic in `get_self_validating_payload` at lines 264–268) and pass them to the same function instead of ignoring `_past_payloads`.
4. Remove or narrow the `#![allow(dead_code, unused_variables)]` pragma so the compiler can flag future regressions.
5. Add a test asserting that `validate_payload` returns an error when the payload contains a `callback_id` already present in `past_payloads`.

## Proof of Concept

1. In `tests.rs`, construct a `past_payload` containing a `BitcoinAdapterResponse` with `callback_id = 0` (as done in `includes_only_responses_for_callback_ids_not_seen_in_past_payloads`).
2. Manually craft a raw payload bytes containing a `BitcoinAdapterResponse` with the same `callback_id = 0`.
3. Call `bitcoin_payload_builder.validate_payload(Height::new(1), &proposal_context, &duplicate_payload_bytes, &past_payloads)`.
4. Assert the result is `Err(...)` — it will instead return `Ok(())`, demonstrating that the duplicate check is absent in the validation path.

    fn validate_self_validating_payload_impl(
        &self,
        payload: &SelfValidatingPayload,
        validation_context: &ValidationContext,
    ) -> Result<NumBytes, SelfValidatingPayloadValidationError> {
        let since = Instant::now();

        // An empty block is always valid.
        if *payload == SelfValidatingPayload::default() {
            return Ok(0.into());
        }

        self.metrics
            .observe_validate_duration(VALIDATION_STATUS_VALID, since);
        let size = NumBytes::new(payload.count_bytes() as u64);

        Ok(size)
    }
