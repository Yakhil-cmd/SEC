### Title
Arithmetic Underflow in `block_number - 1` Produces Incorrect Public Input for Genesis Block — (`File: basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/post_tx_op_proving_singleblock_batch.rs` and `post_tx_op_proving_multiblock_batch.rs`)

---

### Summary

Both proving post-tx-op implementations compute `chain_state_commitment_before` using an unchecked `metadata.block_number() - 1` on a `u64`. When the current block number is `0` (genesis), this underflows to `u64::MAX` in release/RISC-V mode, producing a corrupted `chain_state_commitment_before` hash. The same codebase explicitly guards against this in `form_block_header` (with an `if block_number == 0` branch), but the public-input path has no such guard, creating an inconsistency that makes the genesis block unprovable.

---

### Finding Description

In both proving post-tx-op implementations, the `chain_state_commitment_before` is constructed as:

```rust
let chain_state_commitment_before = ChainStateCommitment {
    state_root: state_commitment.root,
    next_free_slot: state_commitment.next_free_slot,
    block_number: metadata.block_number() - 1,   // ← unchecked u64 subtraction
    last_256_block_hashes_blake: blocks_hasher.finalize().into(),
    last_block_timestamp,
};
``` [1](#0-0) [2](#0-1) 

`block_number` is a `u64` field of `ChainStateCommitment`: [3](#0-2) 

In Rust release mode (the RISC-V proving target), `0u64 - 1` wraps to `u64::MAX` without a panic. The resulting `chain_state_commitment_before` hash is therefore incorrect, and the `BatchPublicInput` / block public input derived from it is invalid.

By contrast, the `form_block_header` helper — called in the same `post_op` function just before the public-input calculation — explicitly handles block 0:

```rust
let previous_block_hash = if block_number == 0 {
    Bytes32::ZERO
} else {
    system.get_blockhash(block_number - 1)?
};
``` [4](#0-3) 

This inconsistency confirms that block 0 is a recognized valid scenario in the codebase, yet the public-input path is unguarded.

---

### Impact Explanation

**Vulnerability class:** Valid-execution unprovability / state-transition bug.

When the prover attempts to prove block 0 (genesis), `chain_state_commitment_before.block_number` is set to `u64::MAX` instead of the semantically correct value (e.g., `0` or a sentinel). The `blake2s` hash of this commitment is committed into the proof's public input: [5](#0-4) 

The on-chain verifier holds the correct pre-state commitment. Because the proof commits to a corrupted `state_before`, the verifier rejects it. The genesis block is permanently unprovable, halting the chain's ability to bootstrap from block 0.

---

### Likelihood Explanation

The ZKsync OS chain must prove from its initial state. If the initial block number is 0 (which is the natural starting point), this underflow is triggered on the very first proof. The test rig explicitly prevents `block_number = 0` via a `checked_sub` guard, indicating the developers are aware of the edge case but did not propagate the guard into the proving path. [6](#0-5) 

---

### Recommendation

Mirror the `form_block_header` guard in both proving post-op implementations:

```rust
block_number: metadata.block_number().saturating_sub(1),
// or:
block_number: if metadata.block_number() == 0 { 0 } else { metadata.block_number() - 1 },
```

Apply the fix to both:
- `post_tx_op_proving_singleblock_batch.rs` line 143
- `post_tx_op_proving_multiblock_batch.rs` line 138

---

### Proof of Concept

1. Configure the ZKsync OS prover to prove a block with `block_number = 0` (genesis).
2. The oracle provides `metadata.block_number() = 0`.
3. In `post_op`, `metadata.block_number() - 1` on `u64` wraps to `18446744073709551615` (`u64::MAX`) in release mode.
4. `chain_state_commitment_before` is hashed with `block_number = u64::MAX`.
5. The resulting `BatchPublicInput.state_before` hash does not match the on-chain commitment (which encodes `block_number = 0`).
6. The verifier rejects the proof; the genesis block is unprovable.

The same underflow occurs in the multiblock-batch path at the identical line: [7](#0-6)
