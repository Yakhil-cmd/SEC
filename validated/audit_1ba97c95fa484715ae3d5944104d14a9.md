### Title
Cross-Chain Signature Replay via Accepted Unprotected Legacy Transactions (No Chain ID in Signing Hash) - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

### Summary

ZKsync OS explicitly accepts pre-EIP-155 ("unprotected") legacy transactions whose signing hash contains **no chain ID**. Any such transaction signed on Ethereum mainnet or any other EVM chain can be replayed verbatim on ZKsync OS, draining the sender's funds on the ZKsync chain without their knowledge or consent.

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, when the signature's `v` value equals `27` or `28` (i.e., `is_eip155()` returns `false`), the signing hash is computed over only the six transaction fields `[nonce, gasPrice, gasLimit, to, value, data]` — with no chain ID included:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
``` [1](#0-0) 

The `is_eip155()` predicate is simply:

```rust
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
``` [2](#0-1) 

The parsed transaction is then stored as `Self::Legacy(tx, sig_data)` and proceeds through the full validation and execution pipeline without any chain-ID check:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    Self::Legacy(tx, sig_data)
};
Ok((tx, sig_hash))
``` [3](#0-2) 

During validation, `transaction.signed_hash()` returns this chain-ID-free hash, which is fed directly into `secp256k1_ec_recover`. If the recovered address matches the oracle-supplied `from`, the transaction is accepted and executed: [4](#0-3) 

Because the signing hash is identical on every EVM chain for the same `[nonce, gasPrice, gasLimit, to, value, data]` tuple, a valid signature on one chain is a valid signature on ZKsync OS.

### Impact Explanation

An attacker who observes any unprotected legacy transaction (v=27/28) broadcast on Ethereum mainnet, a testnet, or any other EVM chain can submit the identical raw bytes to ZKsync OS. If the originating address holds a non-zero balance on ZKsync OS and the nonce matches, the bootloader will:

1. Verify the signature as valid (same hash, same sig).
2. Execute the transaction (transfer value, call contract, deploy code).

This constitutes a **direct, unprivileged loss of user funds** on ZKsync OS triggered by a transaction the user never intended for this chain. The attacker needs no special access — only the ability to submit a transaction to the ZKsync OS sequencer.

### Likelihood Explanation

Pre-EIP-155 transactions are still produced by legacy tooling, hardware wallets in compatibility mode, and some DeFi protocols. Any user who has ever broadcast an unprotected legacy transaction on any EVM chain and who also holds funds on ZKsync OS is at risk. The attacker's cost is a single RPC call.

### Recommendation

**Short term**: Reject unprotected legacy transactions (v=27 or v=28) at the bootloader parsing layer. Add an explicit check in `parse_and_compute_signed_hash` that returns `Err(InvalidTransaction::InvalidChainId)` when `!sig_data.is_eip155()`:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    return Err(InvalidTransaction::InvalidChainId.into()); // reject unprotected
};
```

**Long term**: Document the chain-ID binding requirements for all supported transaction types and add a protocol-level invariant that every accepted transaction must commit to the ZKsync OS chain ID in its signing hash.

### Proof of Concept

1. On Ethereum mainnet, Alice broadcasts an unprotected legacy transaction (v=27):
   - `nonce=5, gasPrice=10gwei, gasLimit=21000, to=Bob, value=1ETH, data=[]`
   - Signed hash: `keccak256(rlp([5, 10gwei, 21000, Bob, 1ETH, []]))` — **no chain ID**

2. Alice also holds 1 ETH on ZKsync OS at the same address, with nonce=5.

3. Attacker copies the raw RLP bytes of Alice's Ethereum transaction and submits them to the ZKsync OS sequencer.

4. `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` computes the identical chain-ID-free hash. [1](#0-0) 

5. `secp256k1_ec_recover` recovers Alice's address — signature is valid. [5](#0-4) 

6. Nonce matches; transaction executes; Alice's 1 ETH on ZKsync OS is transferred to Bob (or the attacker if they craft the `to` field).
