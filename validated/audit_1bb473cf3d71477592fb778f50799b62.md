Audit Report

## Title
`withdraw_eth` Panics on `LedgerBurnError::AmountTooLow` Due to Missing Graceful Error Path — (`rs/ethereum/cketh/minter/src/endpoints.rs`)

## Summary
The `From<LedgerBurnError> for WithdrawalError` implementation in `endpoints.rs` panics (traps the canister message) when the `AmountTooLow` variant is encountered, instead of returning the already-defined `WithdrawalError::AmountTooLow` variant. Because `validate_config` in `state.rs` guards against this condition using a hardcoded fee constant rather than the live ledger fee, any NNS governance action that raises the ckETH ledger `transfer_fee` above `cketh_minimum_withdrawal_amount` — or lowers `minimum_withdrawal_amount` below the actual fee — silently bypasses the guard and causes every subsequent `withdraw_eth` call in the affected range to trap, breaking the core ckETH withdrawal endpoint for all users.

## Finding Description

**Root cause 1 — Hardcoded fee in `validate_config`:**
`State::validate_config` in `rs/ethereum/cketh/minter/src/state.rs` compares `cketh_minimum_withdrawal_amount` against a hardcoded constant (`2_000_000_000_000` wei for Mainnet), not the live ledger fee. [1](#0-0) 

If the actual ckETH ledger `transfer_fee` is raised above `cketh_minimum_withdrawal_amount` via NNS governance, this check still passes because it only compares against the stale hardcoded value.

**Root cause 2 — Panic in `From<LedgerBurnError> for WithdrawalError`:**
The conversion impl in `endpoints.rs` handles every `LedgerBurnError` variant gracefully except `AmountTooLow`, which unconditionally panics: [2](#0-1) 

The `WithdrawalError` enum already defines an `AmountTooLow { min_withdrawal_amount: Nat }` variant at line 217 that is never populated by this code path. [3](#0-2) 

**Exploit chain:**
1. `ledger_client.rs` correctly converts `TransferFromError::BadBurn` → `LedgerBurnError::AmountTooLow`. [4](#0-3) 
2. `main.rs` calls `Err(WithdrawalError::from(e))` on the burn error. [5](#0-4) 
3. `WithdrawalError::from(LedgerBurnError::AmountTooLow { .. })` hits the `panic!` branch, trapping the canister message.

**Inconsistency:** `withdraw_erc20` handles `LedgerBurnError::AmountTooLow` gracefully by returning a structured error; `withdraw_eth` does not.

## Impact Explanation
Once the ledger fee exceeds `minimum_withdrawal_amount`, every `withdraw_eth` call for amounts in the range `[minimum_withdrawal_amount, actual_fee)` traps. The caller receives a reject response with no actionable error. This is a complete, persistent DoS on the core ckETH withdrawal endpoint affecting all users until the minter is upgraded. This matches the allowed impact: **High — Application/platform-level DoS on a ck-token financial integration with concrete user harm.**

## Likelihood Explanation
The trigger requires one or two NNS governance proposals: raising the ckETH ledger `transfer_fee` and/or lowering `cketh_minimum_withdrawal_amount`. Both are routine governance actions with precedent (NNS proposals 126309 and 139665). The governance action is not an attack — it is a legitimate operational adjustment. Once the precondition is met, any unprivileged user calling `withdraw_eth` with an amount in the affected range triggers the panic. The condition persists until the minter is upgraded.

## Recommendation
1. Replace the `panic!` in `From<LedgerBurnError> for WithdrawalError` with a graceful return using the existing variant:
```rust
LedgerBurnError::AmountTooLow { minimum_burn_amount, .. } => {
    Self::AmountTooLow { min_withdrawal_amount: minimum_burn_amount.into() }
}
```
2. Remove the hardcoded fee from `validate_config` or supplement it with a live query to the ledger's `icrc1_fee` endpoint during minter initialization and upgrade, so the guard reflects the actual constraint.
3. Align `withdraw_eth` error handling with `withdraw_erc20`, which already propagates `AmountTooLow` as a structured error.

## Proof of Concept
1. Deploy a local ckETH minter + ICRC-1 ledger (PocketIC or local replica).
2. Set ledger `transfer_fee` to `X` and minter `cketh_minimum_withdrawal_amount` to `Y` where `Y < X` but `Y >= 2_000_000_000_000` (hardcoded Mainnet constant).
3. Call `withdraw_eth` with `amount = Y` (passes the `amount < minimum_withdrawal_amount` guard at line 292).
4. The ledger returns `BadBurn { min_burn_amount: X }` because `Y < X`.
5. `burn_from` returns `LedgerBurnError::AmountTooLow`.
6. `WithdrawalError::from(e)` hits the `panic!` branch; the canister message is trapped.
7. Observe the caller receives a reject response instead of `WithdrawalError::AmountTooLow`.
