### Title
Unchecked `native_price` / `pubdata_price` in Oracle-Provided Block Metadata Enable Forward/Proving Divergence — (`basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

---

### Summary

`BlockMetadataFromOracle` carries `native_price` and `pubdata_price` that are read from the oracle and used in every transaction's fee and native-resource calculation. Neither field is validated for bounds in `metadata_op.rs`, and — as the project's own documentation explicitly acknowledges — neither field is committed to in the batch public inputs. A prover supplying crafted oracle data can therefore re-execute a sequencer block with extreme pricing values, produce a cryptographically valid proof for a materially different state transition, and have that proof accepted on L1.

---

### Finding Description

**Root cause — no bounds check at metadata initialisation**

`metadata_op.rs` reads the full `BlockMetadataFromOracle` struct from the oracle and applies exactly two checks:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs  lines 27-31
if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT
    || metadata.individual_tx_gas_limit() > MAX_TX_GAS_LIMIT
{
    return Err(internal_error!("block or tx gas limit is too high"));
}
``` [1](#0-0) 

`native_price`, `pubdata_price`, `eip1559_basefee`, and `blob_fee` are accepted without any upper- or lower-bound validation.

**The struct fields that are unchecked**

```rust
// zk_ee/src/system/metadata/zk_metadata.rs  lines 122-131
pub eip1559_basefee: U256,
pub pubdata_price:   U256,
pub native_price:    U256,
...
pub blob_fee:        U256,
``` [2](#0-1) 

Deserialization (`from_iter`) accepts any bit-pattern for every `U256` field with no range enforcement: [3](#0-2) 

**`native_price` and `pubdata_price` are absent from the public inputs**

The project documentation explicitly states the gap:

> "The block header should determine the block fully, i.e. include all the inputs needed to execute the block. Currently it misses `gas_per_pubdata` and `native_price`, but we are already working on design and implementation to solve this issue." [4](#0-3) 

Inspecting the actual `BatchOutput` hash confirms this: the hash covers `chain_id`, timestamps, DA commitment, L1-tx count, logs root, upgrade-tx hash, and settlement-layer chain id — but **not** `native_price` or `pubdata_price`: [5](#0-4) 

The final `BatchPublicInput` that the L1 verifier checks is only a hash of `state_before`, `state_after`, and `batch_output`: [6](#0-5) 

**How the unchecked value propagates into fee/resource accounting**

`native_price` is the divisor used to convert gas price into native resource units for every L2 transaction:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs  lines 121-138
let native_per_gas = {
    if native_price.is_zero() {
        return Err(internal_error!("Native price cannot be 0").into());
    }
    ...
    u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(
        TxError::Validation(InvalidTransaction::NativeResourcesAreTooExpensive),
    )?
};
``` [7](#0-6) 

When `native_price = U256::MAX`, `gas_price.div_ceil(U256::MAX) = 0` for any realistic gas price, so `native_per_gas = 0`. The resource-creation helper then treats `native_per_gas == 0` as the "free native" signal:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs  lines 345-349
let native_limit = if cfg!(feature = "unlimited_native") || free_native {
    u64::MAX - 1
} else {
    native_prepaid_from_gas
};
``` [8](#0-7) 

Every transaction in the block therefore executes with an effectively unlimited native resource budget, regardless of what the sequencer computed.

Conversely, setting `native_price = 1` makes `native_per_gas = gas_price` (a full U256 value), which overflows `u256_try_to_u64` and causes every L2 transaction to be rejected with `NativeResourcesAreTooExpensive`.

`pubdata_price` follows the same pattern: an extreme value drives `native_per_pubdata` to overflow or to zero, silently altering which transactions succeed and how much pubdata cost is charged. [9](#0-8) 

---

### Impact Explanation

Because `native_price` and `pubdata_price` are absent from the public inputs, the L1 verifier cannot distinguish a proof produced with the sequencer's correct values from one produced with attacker-chosen values. A malicious prover can:

1. **Grant unlimited native resources** (`native_price = U256::MAX`): transactions that the sequencer rejected for exceeding native limits now succeed, producing a `state_after` that differs from the sequencer's output. The proof is valid; the L1 contract updates to the manipulated state.
2. **Reject all L2 transactions** (`native_price = 1`): every transaction fails with `NativeResourcesAreTooExpensive`, producing an empty-execution state. Users lose gas fees; the operator loses revenue.
3. **Manipulate pubdata cost** (`pubdata_price = U256::MAX`): any transaction that writes storage is charged an astronomically large native cost and reverts post-execution, silently changing execution outcomes.

In all cases the resulting `state_after` is committed to in the public inputs and accepted by L1, making the divergence permanent and undetectable on-chain.

---

### Likelihood Explanation

The prover is explicitly listed as a valid attacker profile in the scope ("prover/forward execution input"). In the proving mode the prover supplies all oracle data, including the `BLOCK_METADATA_QUERY_ID` response. No additional privilege beyond running the prover is required. The project documentation already acknowledges the missing commitment, confirming the gap is real and reachable today.

---

### Recommendation

1. **Add bounds validation in `metadata_op.rs`**: enforce a minimum (`> 0`) and a maximum (e.g., a protocol-defined `MAX_NATIVE_PRICE`) on `native_price` and `pubdata_price` immediately after deserialization, analogous to the existing gas-limit checks. [10](#0-9) 

2. **Commit `native_price` and `pubdata_price` to the public inputs**: include them in `BatchOutput::hash()` (or the block-level equivalent) so the L1 verifier can enforce that the prover used the same pricing parameters as the sequencer. The project already tracks this as a known gap. [11](#0-10) 

---

### Proof of Concept

**Setup**: proving mode; attacker controls the oracle response for `BLOCK_METADATA_QUERY_ID`.

**Step 1** — Sequencer executes block B with `native_price = 10`, producing state S₁ where all transactions succeed.

**Step 2** — Prover re-executes block B with `native_price = U256::MAX` (supplied via the oracle):

```
native_per_gas = gas_price.div_ceil(U256::MAX) = 0
free_native    = (native_per_gas == 0) = true
native_limit   = u64::MAX - 1   // unlimited
```

Transactions that the sequencer would have bounded by native limits now execute without restriction. The resulting state S₂ ≠ S₁.

**Step 3** — Prover generates a valid ZK proof for the transition `S₀ → S₂`.

**Step 4** — Prover submits the proof to L1. `BatchPublicInput` contains only `state_before = hash(S₀)`, `state_after = hash(S₂)`, and `batch_output` (which does not include `native_price`). The L1 verifier accepts the proof and updates the canonical state to S₂.

The divergence between the sequencer's S₁ and the on-chain S₂ is permanent and undetectable by the L1 contract.

impl BatchOutput {
    ///
    /// Calculate keccak256 hash of public input
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = Keccak256::new();
        hasher.update(self.chain_id.to_be_bytes::<32>());
        hasher.update(&self.first_block_timestamp.to_be_bytes());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
