Audit Report

## Title
Precision Loss in ICRC-21 Consent Message Token Amount Display - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `convert_tokens_to_string_representation` function converts a `Nat` token amount to `f64` before formatting it for display in ICRC-21 `GenericDisplayMessage` consent messages. Because `f64` has only 53 bits of mantissa, any integer above 2^53 (≈ 9.007 × 10^15) is silently rounded, causing the consent message shown to a user to display a different amount than what they are actually signing. This directly undermines the core security guarantee of ICRC-21, which is the trusted display layer used by hardware wallets and signing devices.

## Finding Description
At `packages/icrc-ledger-types/src/icrc21/responses.rs` lines 318–327, the function `convert_tokens_to_string_representation` calls `tokens.0.to_f64()` on a `Nat` (arbitrary-precision integer), then divides by `10_f64.pow(decimals)` and formats the result:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```

This function is called from four methods on `ConsentMessage`, all exclusively for the `GenericDisplayMessage` path:
- `add_amount` (L113)
- `add_fee` (L141)
- `add_allowance` (L186)
- `add_existing_allowance` (L212)

The `GenericDisplayMessage` path is the **default** when no `device_spec` is specified by the caller (`lib.rs` L173–175). Any `Nat` value above 2^53 will be silently rounded by the `to_f64()` conversion. The rounded value is then formatted and embedded in the consent message string. There are no guards or checks that detect or reject amounts above 2^53.

The exploit flow is:
1. A malicious dApp constructs a `TransferArg` with `amount = 9007199254740993` (2^53 + 1 wei for ckETH).
2. It calls `icrc21_consent_message` on the ckETH ledger with `device_spec = None` (GenericDisplay).
3. The ledger's `convert_tokens_to_string_representation` converts `9007199254740993` to `f64`, which rounds to `9007199254740992.0`.
4. The returned `GenericDisplayMessage` displays `**Amount:** \`9007199254740992 wei\``.
5. The user's hardware wallet shows this rounded amount and the user approves.
6. The actual on-chain transfer executes for `9007199254740993 wei` — 1 wei more than displayed.

## Impact Explanation
ICRC-21 consent messages are the security-critical display layer that hardware wallets and signing devices use to show users exactly what they are signing. A discrepancy between the displayed amount and the actual signed amount directly undermines this guarantee. For ckETH (18 decimals), the precision loss threshold is ~9,007 ETH (2^53 wei), a realistic high-value transfer. The discrepancy at the threshold is 1 wei, growing for larger amounts (e.g., at 2^54 + 1, both `2^54` and `2^54 + 1` display identically). This constitutes a concrete security impact on ck-token consent message integrity, mapping to the High impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation
Any unprivileged caller can invoke `icrc21_consent_message` on any ICRC-1/ICRC-2 ledger canister with an arbitrary `Nat` amount. The `Nat` type imposes no upper bound. The `GenericDisplayMessage` path is the default (no `device_spec` required). Amounts above 2^53 wei are realistic for ckETH transfers (>~9,007 ETH). The code path is exercised on every consent message request for the default display type, making this reliably triggerable without any special privileges.

## Recommendation
Replace the `f64`-based conversion with exact integer arithmetic using `BigUint` division and modulo to separate whole and fractional parts:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let divisor = num_bigint::BigUint::from(10u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let frac = &tokens.0 % &divisor;
    if frac.is_zero() {
        Ok(whole.to_string())
    } else {
        Ok(format!("{}.{:0>width$}", whole, frac, width = decimals as usize)
            .trim_end_matches('0')
            .to_string())
    }
}
```

This avoids any floating-point conversion and preserves exact precision for all `Nat` values, consistent with the `DisplayAmount` pattern used in `rs/bitcoin/ckbtc/minter/src/tx.rs`.
