### Title
Inconsistent Gas-Limit Overflow Handling Between Ethereum STF and ZK STF Validation Paths — (`File: basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs` vs `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

Both the Ethereum STF and ZK STF validation paths contain an identical guard checking that `tx_gas_limit * ERGS_PER_GAS` does not overflow a `u64`. However, the two paths classify the failure differently and apply different upstream guards, creating an inconsistency directly analogous to the external report: one path treats the condition as a recoverable per-transaction validation error, while the other treats it as a block-terminating internal error.

---

### Finding Description

**ZK STF path** (`zk/validation_impl.rs`):

```rust
// lines 69-73
require!(
    tx_gas_limit.saturating_mul(ERGS_PER_GAS) < u64::MAX,
    InvalidTransaction::CallerGasLimitTooHigh,   // ← TxError::Validation
    system
)?;
```

Before this check, the ZK STF also enforces a block-level ceiling:

```rust
// lines 84-93
require!(
    block_gas_limit <= MAX_BLOCK_GAS_LIMIT,       // MAX_BLOCK_GAS_LIMIT = u64::MAX / ERGS_PER_GAS
    InvalidTransaction::BlockGasLimitTooHigh,
    system
)?;
require!(
    tx_gas_limit <= block_gas_limit,
    InvalidTransaction::CallerGasLimitMoreThanBlock,
    system
)?;
```

Because `MAX_BLOCK_GAS_LIMIT = u64::MAX / ERGS_PER_GAS`, the `block_gas_limit` check guarantees that any `tx_gas_limit ≤ block_gas_limit` will never overflow `u64` when multiplied by `ERGS_PER_GAS`. The overflow guard is therefore unreachable in the ZK STF under normal operation, and even if somehow reached, it returns a **per-transaction validation error** (`TxError::Validation`), which only rejects the transaction and allows the block to continue.

**Ethereum STF path** (`ethereum/validation_impl.rs`):

```rust
// lines 116-120
require!(
    tx_gas_limit.saturating_mul(ERGS_PER_GAS) < u64::MAX,
    internal_error!("TX gas limit overflows ergs counter"),  // ← TxError::Internal
    system
)?;
```

The Ethereum STF has **no** `MAX_BLOCK_GAS_LIMIT` guard. Its only upstream check is:

```rust
// lines 128-133
let tx_limit = system.metadata.individual_tx_gas_limit();
require!(
    tx_gas_limit <= tx_limit,
    InvalidTransaction::CallerGasLimitMoreThanTxLimit,
    system
)?;
```

`individual_tx_gas_limit()` for the Ethereum STF returns `block_gas_limit()` directly (the raw `header.gas_limit` field), with no ceiling enforced:

```rust
// block_header.rs line 176-178
fn individual_tx_gas_limit(&self) -> u64 {
    self.block_gas_limit()
}
```

If the block header carries a `gas_limit` value large enough that `gas_limit * ERGS_PER_GAS ≥ u64::MAX`, a transaction with `tx_gas_limit = block_gas_limit` passes the per-tx check and then hits the overflow guard, which returns `TxError::Internal`. Per the `TxError` enum documentation, `TxError::Internal` is a block-terminating error, unlike `TxError::Validation` which "shouldn't terminate the block execution." ---

### Impact Explanation

When the Ethereum STF processes a transaction whose `gas_limit * ERGS_PER_GAS ≥ u64::MAX`, the block fails entirely with an internal error rather than simply rejecting the offending transaction. This is a **state-transition bug**: the block cannot be sealed, no transactions in the block are committed, and the sequencer must discard the entire block. In the ZK STF the identical arithmetic condition is a recoverable per-transaction rejection. The divergence means the two execution modes produce different outcomes for the same input, breaking the equivalence required for correct forward/proving operation. [5](#0-4) 

---

### Likelihood Explanation

The Ethereum STF's `individual_tx_gas_limit()` returns the raw `header.gas_limit` with no ceiling. If the sequencer (or a malicious block producer in a permissioned-sequencer model) sets `header.gas_limit` to a value where `gas_limit * ERGS_PER_GAS ≥ u64::MAX`, any transaction that fills the block gas limit will trigger the internal error. Even without sequencer cooperation, if `ERGS_PER_GAS > 1` the threshold is `u64::MAX / ERGS_PER_GAS`, which is a finite and potentially reachable value for a high-gas-limit block. The ZK STF explicitly prevents this via `MAX_BLOCK_GAS_LIMIT = u64::MAX / ERGS_PER_GAS`; the Ethereum STF has no equivalent guard. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **Add a `MAX_BLOCK_GAS_LIMIT` guard to the Ethereum STF** at metadata initialization (analogous to `metadata_op.rs` for the ZK STF) so that a block with an overflowing gas limit is rejected before any transaction is processed.

2. **Change the overflow check in `ethereum/validation_impl.rs`** to return `InvalidTransaction::CallerGasLimitTooHigh` (a `TxError::Validation`) instead of `internal_error!()`, matching the ZK STF behavior:

```rust
// ethereum/validation_impl.rs
require!(
    tx_gas_limit.saturating_mul(ERGS_PER_GAS) < u64::MAX,
    InvalidTransaction::CallerGasLimitTooHigh,  // consistent with ZK STF
    system
)?;
``` [8](#0-7) 

---

### Proof of Concept

1. Construct an Ethereum-STF block with `header.gas_limit = u64::MAX / ERGS_PER_GAS + 1` (one above the ZK STF ceiling).
2. Submit a transaction with `gas_limit = header.gas_limit`.
3. The Ethereum STF check `tx_gas_limit <= individual_tx_gas_limit()` passes (equal values).
4. `tx_gas_limit.saturating_mul(ERGS_PER_GAS)` saturates to `u64::MAX`, failing `< u64::MAX`.
5. `internal_error!("TX gas limit overflows ergs counter")` is returned as `TxError::Internal`, terminating block execution.
6. The same transaction submitted to the ZK STF would be caught by `block_gas_limit <= MAX_BLOCK_GAS_LIMIT` at metadata initialization and rejected cleanly as `BlockGasLimitTooHigh`, never reaching the overflow guard. [9](#0-8) [10](#0-9)
