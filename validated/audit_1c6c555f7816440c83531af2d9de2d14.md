Audit Report

## Title
Unguarded `.unwrap()` in `apply_balance_changes` Allows Compromised Sandbox to Panic Replica — (`rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs`)

## Summary

`SystemStateModifications::apply_balance_changes` calls `state.reserve_cycles(self.reserved_cycles).unwrap()` at line 600. The upstream guard, `validate_cycle_change`, only checks internal accounting consistency and does not verify that `reserved_cycles` respects the canister's `reserved_balance_limit`. A compromised sandbox process can craft a `SystemStateModifications` payload that passes `validate_cycle_change` but causes `reserve_cycles` to return `Err(ReservationError::ReservedLimitExceed)`, triggering the `.unwrap()` panic and crashing the replica. Because all replicas on a subnet execute messages deterministically, every replica crashes on the same message, causing a subnet-wide availability failure.

## Finding Description

`apply_changes` calls `validate_cycle_change` then unconditionally calls `apply_balance_changes`: [1](#0-0) 

`apply_balance_changes` returns `()` and contains a panicking `.unwrap()`: [2](#0-1) 

and a panicking `assert_eq!`: [3](#0-2) 

`validate_cycle_change` only checks that `cycles_balance_change` equals the sum of all outgoing payments plus `reserved_cycles`. It does not check whether `reserved_cycles` exceeds `reserved_balance_limit - current_reserved_balance`: [4](#0-3) 

`reserve_cycles` delegates to `can_reserve_cycles`, which enforces the limit: [5](#0-4) 

The legitimate path through `reserve_storage_cycles` in `SandboxSafeSystemState` does check the limit before setting `reserved_cycles`: [6](#0-5) 

However, `SystemStateModifications` is `Serialize + Deserialize` and is returned from the sandbox process over IPC: [7](#0-6) 

A compromised sandbox process can craft arbitrary bytes in the IPC response, deserializing into a `SystemStateModifications` with any `reserved_cycles` value, bypassing the check in `reserve_storage_cycles` entirely. The `validate_cycle_change` guard is insufficient because it only verifies `cycles_balance_change == Removed(reserved_cycles)` — a constraint the attacker trivially satisfies by setting both fields consistently.

## Impact Explanation

A replica panic in the execution path crashes the replica process. Because all replicas on a subnet process the same messages deterministically, every replica hits the same panic on the same message, causing a subnet-wide crash loop. This is a complete availability break for the affected subnet, matching the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation

Exploitation requires a prior sandbox escape — arbitrary code execution within the canister sandbox process. Sandbox escapes are non-trivial but are a known attack class against Wasm runtimes. The IC's architecture explicitly treats the sandbox as a security boundary: `SandboxSafeSystemState` exists precisely to validate sandbox-supplied data before applying it to replica state. The comment at line 597–599 ("This must succeed because the cycle changes were validated") confirms the developer intent was for `validate_cycle_change` to prevent this panic — but the validation is incomplete. Once sandbox code execution is achieved, crafting the triggering payload is straightforward and deterministic. Any canister with a `reserved_balance_limit` set (a standard configuration) is a viable target.

## Recommendation

- **Short term:** Change `apply_balance_changes` to return `HypervisorResult<()>`. Replace `state.reserve_cycles(self.reserved_cycles).unwrap()` with `state.reserve_cycles(self.reserved_cycles).map_err(|e| Self::error(format!("Failed to reserve cycles: {e:?}")))?`. Replace the `assert_eq!` balance invariant with `debug_assert_eq!`. Propagate the result through `apply_changes`.
- **Long term:** Add an explicit check in `validate_cycle_change` (or a new `validate_reserved_cycles` step) that verifies `initial_reserved_balance + reserved_cycles <= reserved_balance_limit` before calling `apply_balance_changes`. Audit all code reachable from `apply_changes` for remaining `unwrap`, `expect`, `assert`, or `unreachable` calls operating on sandbox-supplied fields.

## Proof of Concept

A compromised sandbox process crafts and returns the following `SystemStateModifications` over IPC:

```
reserved_cycles          = reserved_balance_limit - current_reserved_balance + 1  (call this V)
cycles_balance_change    = CyclesBalanceChange::Removed(V)
consumed_cycles_by_use_case = default (zero)
call_context_balance_taken  = None
requests                 = []
```

Execution path on the replica:

1. `apply_changes` calls `validate_cycle_change`: `expected_change = Removed(V)`, `cycles_balance_change = Removed(V)` → equal → `Ok(())`
2. `apply_changes` calls `apply_balance_changes`
3. `adjusted_balance_change = Removed(V) + Added(V) = zero` → no net main balance change
4. `state.reserve_cycles(V)` → `can_reserve_cycles` checks `reserved_balance + V > reserved_balance_limit` → `Err(ReservedLimitExceed)`
5. `.unwrap()` panics → replica process crashes → all replicas on the subnet crash on the same message → subnet enters crash loop

A deterministic integration test can reproduce this by constructing a `SystemStateModifications` directly with the above fields and calling `apply_balance_changes` on a `SystemState` whose `reserved_balance_limit` is set such that `current_reserved_balance + V > limit`.

    fn validate_cycle_change(&self, is_cmc_canister: bool) -> HypervisorResult<()> {
        let mut expected_change = CyclesBalanceChange::zero();

        if let Some((_, call_context_balance_taken)) = self.call_context_balance_taken {
            expected_change =
                expected_change + CyclesBalanceChange::added(call_context_balance_taken);
        }

        for req in self.requests.iter() {
            expected_change = expected_change + CyclesBalanceChange::removed(req.payment);
        }

        for amount in self.consumed_cycles_by_use_case.iter_over_real() {
            expected_change = expected_change + CyclesBalanceChange::removed(amount);
        }

        expected_change = expected_change + CyclesBalanceChange::removed(self.reserved_cycles);

        // If the canister is not the cycles minting canister, then the balance
        // change coming from the Wasm execution must match the expected balance
        // change that we just computed.
        if is_cmc_canister || self.cycles_balance_change == expected_change {
            Ok(())
        } else {
            Err(HypervisorError::WasmEngineError(
                WasmEngineError::FailedToApplySystemChanges(format!(
                    "Invalid cycle change: expected {:?}, got {:?}",
                    expected_change, self.cycles_balance_change
                )),
            ))
        }
    }
