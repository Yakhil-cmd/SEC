Audit Report

## Title
ICRC-21 Consent Message Fails for u256-Token Amounts Exceeding u64::MAX — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `Value::TokenAmount` struct stores `amount: u64`, and `nat_to_u64` is called unconditionally when building `FieldsDisplay` consent messages. For any ckETH or ckERC20 ledger (deployed as the u256 variant), a transfer or approval amount exceeding `u64::MAX` (≈ 18.4 ETH in wei) causes `nat_to_u64` to return `Icrc21Error::GenericError`, aborting consent message generation entirely. The `GenericDisplay` path additionally converts via `to_f64()`, silently losing precision for amounts above ~9×10¹⁵ wei.

## Finding Description
`Value::TokenAmount` is defined with `amount: u64` at [1](#0-0)  The helper `nat_to_u64` at [2](#0-1)  calls `BigUint::to_u64()`, which returns `None` for any value requiring more than one 64-bit limb, propagating as `Icrc21Error::GenericError`. This is called unconditionally in `add_amount` [3](#0-2) , `add_fee` [4](#0-3) , `add_allowance` [5](#0-4) , and `add_existing_allowance` [6](#0-5)  for the `FieldsDisplay` branch. The `GenericDisplay` path uses `convert_tokens_to_string_representation`, which calls `to_f64()` [7](#0-6) ; `BigUint::to_f64()` succeeds for all u256 values (u256::MAX ≈ 1.16×10⁷⁷ << f64::MAX ≈ 1.8×10³⁰⁸) but silently rounds amounts above ~9×10¹⁵. The single public entry point `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints` [8](#0-7)  is used by all ICRC-1/ICRC-2 ledgers. ckETH and all ckERC20 ledgers are deployed as the u256 variant (`ic-icrc1-ledger-u256.wasm.gz`) [9](#0-8) , whose `Tokens::MAX` exceeds `u64::MAX` as confirmed by the test suite [10](#0-9) . ckETH amounts are denominated in wei (18 decimals), so u64::MAX corresponds to only ~18.4 ETH [11](#0-10) .

## Impact Explanation
Any ckETH or ckERC20 user whose transfer/approval amount exceeds `u64::MAX` receives `Icrc21Error::GenericError` from the `icrc21_canister_call_consent_message` endpoint when requesting `FieldsDisplay`. ICRC-21-compliant wallets (including hardware wallets) that require a consent message before signing are unable to render a consent screen for those users, blocking them from signing transactions through compliant wallet software. This is a concrete, application-level DoS on the ck-token ledger consent infrastructure with direct user harm, matching **High ($2,000–$10,000): Significant Chain Fusion / ck-token / ledger security impact with concrete user or protocol harm**. The `GenericDisplay` precision-loss path is a secondary integrity issue (silent misleading display for large approvals such as `u256::MAX` infinite approvals).

## Likelihood Explanation
The exploit requires no special privileges: any unprivileged user calls the standard `icrc21_canister_call_consent_message` update endpoint with a `FieldsDisplay` device spec and an amount > `u64::MAX`. At current ETH prices, balances exceeding 18.4 ETH are common for retail and institutional users. The condition is deterministic and repeatable with zero cost beyond a normal canister call.

## Recommendation
1. Replace `Value::TokenAmount { amount: u64 }` with a `Nat` or `String` field to support arbitrary-precision amounts, or add a `Text` fallback for amounts exceeding `u64::MAX`.
2. In `convert_tokens_to_string_representation`, replace `to_f64()` with arbitrary-precision decimal arithmetic (e.g., using `BigUint` division with remainder) to avoid silent precision loss for large amounts.
3. Add test coverage in the ICRC-21 consent message test suite for amounts exceeding `u64::MAX` on u256-token ledgers, covering both `FieldsDisplay` and `GenericDisplay` paths.

## Proof of Concept
1. Deploy the u256-variant ICRC-1 ledger (`ic-icrc1-ledger-u256.wasm.gz`).
2. Mint `u64::MAX + 1` tokens to a test account.
3. Call `icrc21_canister_call_consent_message` with `method = "icrc1_transfer"`, `amount = u64::MAX + 1`, `device_spec = FieldsDisplay`.
4. Observe `Icrc21Error::GenericError { description: "Failed to convert tokens to u64" }` returned instead of a valid consent message.

This can be implemented as a deterministic PocketIC integration test using the existing u256-token ledger test infrastructure in `rs/ledger_suite/icrc1/index-ng/tests/tests.rs`.
