Audit Report

## Title
Rate-Limit Bucket Charged to Node's Registered Operator Instead of Caller in `do_remove_node_directly_` — (`rs/registry/canister/src/mutations/node_management/do_remove_node_directly.rs`)

## Summary
In `do_remove_node_directly_`, the rate-limit reservation is charged against the node's registered operator (`node_operator_id`) rather than the actual ingress caller (`caller_id`). Because `make_remove_or_replace_node_mutations` permits a different operator (B) to remove nodes belonging to operator A when they share the same DC and node provider, operator B can silently exhaust operator A's weekly rate-limit bucket (up to 140 operations) while leaving its own bucket completely untouched. This locks operator A out of all rate-limited node operations for up to 7 days.

## Finding Description
In `do_remove_node_directly_` (lines 52–54 of `do_remove_node_directly.rs`), the rate-limit identity is taken from the node record, not from the ingress caller:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
```

`caller_id` is only passed to `make_remove_or_replace_node_mutations`, which performs the DC + provider fallback authorization check (lines 89–119). That check explicitly allows a caller whose `caller_id != node_operator_id` to proceed if both operators share the same `dc_id` and the same node provider — a documented and supported redeployment scenario (lines 83–88).

`try_reserve_capacity_for_node_operator_operation` in `rate_limits.rs` (lines 114–117) keys the reservation on `node_operator_key(node_operator_id)`, confirming the charge goes to the node's operator identity, not the caller's. The per-operator bucket is capped at `NODE_OPERATOR_MAX_SPIKE = 20 × 7 = 140` (lines 24–25 of `rate_limits.rs`).

The existing test `should_succeed_remove_node_compare_dc_and_node_provider` (lines 482–554 of `do_remove_node_directly.rs`) explicitly confirms that operator B can remove operator A's nodes when they share the same DC and provider, and that operator B's `node_allowance` is not decremented — demonstrating the cross-operator removal path is reachable and exercised.

## Impact Explanation
Operator B can call `do_remove_node_directly` on each of operator A's nodes (up to 140 within the spike window). Each call: (1) passes authorization (same DC + same provider), (2) deducts 1 from operator A's 140-op weekly bucket, (3) deducts 1 from the shared node-provider bucket, and (4) leaves operator B's own operator bucket untouched. Once operator A's bucket reaches 0, `try_reserve_capacity_for_node_operator_operation` returns `NotEnoughCapacity` for any call keyed to operator A — including operator A's own `do_add_node` and `do_remove_node_directly` calls — for up to 7 days. The shared node-provider bucket depletion can cascade to other operators under the same provider.

This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user harm**, specifically the inability of a legitimate node operator to manage their own nodes for up to 7 days without any governance vote, admin key, or threshold corruption.

## Likelihood Explanation
The precondition — two node operators sharing the same DC and node provider — is an explicitly supported and documented redeployment scenario (lines 83–88 of `do_remove_node_directly.rs`). No special privileges beyond being a registered node operator in the same DC under the same provider are required. This is a normal on-chain state. The attack is a sequence of ordinary ingress update calls to the registry canister, requiring no victim mistakes, social engineering, or external compromise.

## Recommendation
Charge the rate limit against `caller_id` (the actual caller), not `node_operator_id` (the node's registered operator). In `do_remove_node_directly_`, replace:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
```

with:

```rust
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, caller_id, 1)?;
```

This ensures the cost of the operation is always borne by the entity initiating it, regardless of which node is being removed.

## Proof of Concept
Extend the existing test `should_succeed_remove_node_compare_dc_and_node_provider` in `do_remove_node_directly.rs`:

```rust
#[test]
fn test_operator_b_drains_operator_a_rate_limit() {
    let mut registry = invariant_compliant_registry(0);
    let operator_a = PrincipalId::new_user_test_id(2000);
    let operator_b = PrincipalId::new_user_test_id(2001);
    let provider   = PrincipalId::new_user_test_id(3000);

    // Register both operators in same DC under same provider
    // Add N nodes owned by operator_a (N = NODE_OPERATOR_MAX_SPIKE = 140)
    // operator_b calls do_remove_node_directly_ on each of operator_a's nodes

    for node_id in operator_a_nodes {
        registry.do_remove_node_directly_(
            RemoveNodeDirectlyPayload { node_id },
            operator_b,
            now,
        ).unwrap();
    }

    // operator_a's bucket is now 0; operator_b's bucket is untouched
    let result = registry.do_remove_node_directly_(
        RemoveNodeDirectlyPayload { node_id: some_new_node_of_a },
        operator_a,
        now,
    );
    assert_eq!(result.unwrap_err(),
        "Rate Limit Capacity exceeded. Please wait and try again later.");

    let b_capacity = registry.get_available_node_operator_op_capacity(operator_b, now);
    assert_eq!(b_capacity, 140); // operator_b untouched
}
```
