### Title
Permanent Fund Freeze via Malicious NEP-141 Contract Registered Through Permissionless `deploy_erc20_token` — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

The `deploy_erc20_token` entrypoint is permissionless — any NEAR account can register an arbitrary NEP-141 contract as an ERC-20 mirror on Aurora. When a user later burns their ERC-20 tokens to exit to NEAR (via `withdrawToNear`), the engine schedules an `ft_transfer` promise to the registered NEP-141 contract. In the production build (features = `"contract"`), the `error_refund` feature is **not** compiled in, so no refund callback is attached to the promise. If the NEP-141 contract reverts on `ft_transfer`, the burned ERC-20 tokens are permanently unrecoverable — a complete fund freeze with no on-chain recovery path.

---

### Finding Description

**Step 1 — Permissionless NEP-141 registration.**

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` performs only a liveness check (`require_running`) and no caller authorization: [1](#0-0) 

Any NEAR account can call this public entrypoint with an arbitrary NEP-141 account ID, causing Aurora to deploy an ERC-20 mirror and permanently bind it to that NEP-141 via `engine::register_token`.

**Step 2 — ERC-20 exit burns tokens before the NEAR promise resolves.**

When a user calls `withdrawToNear` on the ERC-20 contract (`EvmErc20.sol`), the ERC-20 tokens are burned first, then the `ExitToNear` precompile is invoked: [2](#0-1) 

The precompile resolves the NEP-141 account from the ERC-20 address and schedules an `ft_transfer` promise to it: [3](#0-2) 

**Step 3 — `error_refund` is absent in the production build.**

The production build is compiled with `CARGO_FEATURES_BUILD = "contract"`: [4](#0-3) 

The `error_refund` feature is a separate opt-in: [5](#0-4) 

Without it, the `refund` field in the callback args is hardcoded to `None`: [6](#0-5) 

**Step 4 — No callback is scheduled for regular ERC-20 exits.**

For a standard ERC-20 exit (non-wNEAR), `transfer_near_args` is also `None`. This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, so the code takes the `PromiseArgs::Create` branch — **no callback is attached at all**: [7](#0-6) 

**Step 5 — Callback does nothing on failure even when it exists.**

For the wNEAR unwrap path (where a callback is attached), the callback handler in `exit_to_near_precompile_callback` falls through to `else { None }` when the promise fails and `args.refund` is `None`: [8](#0-7) 

In both cases, a failed `ft_transfer` leaves the ERC-20 tokens burned and the NEP-141 tokens locked in Aurora's account with no recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who burns ERC-20 tokens to exit to NEAR loses those tokens permanently if the underlying NEP-141 contract reverts on `ft_transfer`. The ERC-20 tokens are already destroyed; there is no re-mint, no refund, and no admin escape hatch for this state. The NEP-141 tokens remain in Aurora's account, inaccessible to the user.

---

### Likelihood Explanation

**Medium.**

`deploy_erc20_token` is callable by any NEAR account with no access control. An attacker can:
1. Deploy a NEP-141 contract that accepts `ft_transfer_call` (so deposits work) but always panics on `ft_transfer` (so withdrawals fail).
2. Register it on Aurora via `deploy_erc20_token`.
3. Lure users into bridging tokens to Aurora (e.g., by presenting a fake token with a legitimate-looking name/symbol).
4. Once users hold ERC-20 balances and attempt to exit, their funds are frozen.

Alternatively, a legitimate NEP-141 contract that is later upgraded (if it has an upgradeable admin) to block `ft_transfer` would trigger the same freeze for all existing ERC-20 holders.

---

### Recommendation

**Short term:** Enable the `error_refund` feature in the production build (`CARGO_FEATURES_BUILD = "contract,error_refund"`). This causes the precompile to attach a callback that re-mints burned ERC-20 tokens when `ft_transfer` fails, restoring the user's balance.

**Long term:** Add access control to `deploy_erc20_token` (e.g., require the caller to be the contract owner or a governance-approved address), preventing arbitrary NEP-141 contracts from being registered as ERC-20 mirrors. This is the direct analog to the external report's whitelist recommendation.

---

### Proof of Concept

1. Attacker deploys `MaliciousNep141` on NEAR: `ft_transfer_call` succeeds (returns `"0"`), `ft_transfer` always panics.
2. Attacker calls `deploy_erc20_token` on Aurora with `MaliciousNep141`'s account ID. Aurora deploys `MaliciousErc20` and registers the mapping.
3. Victim calls `ft_transfer_call` on `MaliciousNep141` → Aurora's `ft_on_transfer` mints ERC-20 tokens for the victim. Victim now holds `MaliciousErc20` balance.
4. Victim calls `MaliciousErc20.withdrawToNear(recipient, amount)`:
   - `_burn(victim, amount)` executes — ERC-20 tokens destroyed.
   - `ExitToNear` precompile fires; `exit_erc20_token_to_near` resolves `MaliciousNep141` from the ERC-20 address.
   - `callback_args = { refund: None, transfer_near: None }` → equals `default()` → `PromiseArgs::Create(ft_transfer)` with **no callback**.
5. `MaliciousNep141.ft_transfer` panics. NEAR runtime marks the receipt as failed.
6. No callback fires. Victim's ERC-20 tokens are gone; NEP-141 tokens remain in Aurora's account. Funds are permanently frozen.
