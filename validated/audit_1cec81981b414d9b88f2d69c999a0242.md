### Title
Single NEAR Account (`owner_id`) Controls All Critical Engine Functions Including Upgrade and Full Access Key Attachment — (File: `engine/src/state.rs`, `engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine stores a single `owner_id: AccountId` in `EngineState`. This single NEAR account is the sole gatekeeper for every critical privileged operation: deploying arbitrary new WASM code (`upgrade`/`stage_upgrade`), pausing the entire engine (`pause_contract`), attaching a full-access key to the engine account (`attach_full_access_key`), redirecting the ETH connector to an arbitrary contract (`set_eth_connector_contract_account`), and transferring ownership (`set_owner`). There is no enforcement that `owner_id` must be a multisig or DAO; it can be any NEAR account, including a plain single-key EOA. Compromise of that one key gives an attacker complete, irreversible control over all user funds held by the engine.

---

### Impact Explanation

**Critical — Direct theft of all user funds and/or permanent fund freeze.**

The Aurora Engine contract is the custodian of all ETH and ERC-20 token balances for every EVM user on Aurora. If the single `owner_id` key is compromised:

- **`upgrade()`**: The attacker deploys a malicious WASM binary that transfers all ETH balances and ERC-20 holdings to an attacker-controlled address. This is a complete, irreversible drain of all funds at rest.
- **`attach_full_access_key()`**: The attacker adds a full-access key to the Aurora engine NEAR account, giving them permanent, unrestricted control over the account and all its assets, independent of the contract logic.
- **`set_eth_connector_contract_account()`**: The attacker redirects the ETH connector pointer to a malicious contract, causing all subsequent deposits and withdrawals to be routed to the attacker.
- **`pause_contract()`**: The attacker permanently freezes the engine, preventing all user withdrawals (permanent fund freeze).

---

### Likelihood Explanation

**High.** The `owner_id` is a plain NEAR `AccountId` with no multisig or threshold-signature enforcement at the protocol level. In practice, if the owner is a single-key NEAR account (the common deployment pattern, as shown in `engine-workspace/src/lib.rs` where `owner_id` is set to a simple account string), the entire system's security reduces to the security of one private key. This is a well-known, high-value attack target. The `upgrade_delay_blocks` parameter provides a time window, but with `upgrade_delay_blocks = 0` (or `= 1` as in the default workspace setup) the delay is negligible. [9](#0-8) 

---

### Recommendation

- **Short term**: Change `owner_id` from a plain single-key NEAR account to a NEAR multisig contract (e.g., `multisig.near` or a DAO contract) that requires M-of-N key holders to approve any privileged transaction before it executes.
- **Short term**: Set `upgrade_delay_blocks` to a meaningful value (e.g., several thousand blocks) to provide a time window for the community to detect and respond to a malicious upgrade.
- **Long term**: Document all privileged roles (`owner_id`, `key_manager`) and their associated risks. Implement an incident response plan for key compromise. Consider a timelocked governance model for the most dangerous operations (`upgrade`, `attach_full_access_key`).

---

### Proof of Concept

1. Aurora Engine is deployed with `owner_id = "alice.near"` (a single-key NEAR account).
2. Eve compromises Alice's NEAR private key.
3. Eve calls `upgrade()` on the Aurora Engine contract, passing a malicious WASM binary as input. The `require_owner_only` check passes because Eve signs with Alice's key.
4. The engine deploys the malicious WASM via `PromiseAction::DeployContract`. The new code contains a function that transfers all ETH balances stored in the engine's NEAR storage to Eve's address.
5. Alternatively, Eve calls `attach_full_access_key()` with her own public key, gaining permanent full-access to the Aurora engine NEAR account, bypassing all contract-level access controls entirely.
6. All user funds (ETH and ERC-20 tokens) held by the Aurora Engine are drained. No recovery is possible without a NEAR protocol-level intervention. [3](#0-2) [4](#0-3) [1](#0-0)

```rust
pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}
```

```rust
pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
```
