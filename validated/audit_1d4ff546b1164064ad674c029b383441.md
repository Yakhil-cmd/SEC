### Title
Fee-on-Transfer ERC20 Deposit Mints Unbacked ckERC20 Tokens - (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

The `depositErc20` function in the ckETH/ckERC20 helper smart contract emits the caller-supplied `amount` as the deposit value in the `ReceivedEthOrErc20` event, regardless of how many tokens the minter address actually received. The IC ckETH minter reads this event and mints exactly `amount` ckERC20 tokens. For fee-on-transfer ERC20 tokens, the minter receives `amount - fee` tokens on Ethereum but mints `amount` ckERC20 on the IC, permanently breaking the 1:1 backing invariant.

---

### Finding Description

In `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, the `depositErc20` function performs a `safeTransferFrom` and then emits the event using the caller-supplied `amount` parameter directly:

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);  // actual received = amount - fee

    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        amount,          // ← emits requested amount, not actual received amount
        principal,
        subaccount
    );
}
``` [1](#0-0) 

The IC ckETH minter scrapes these Ethereum logs via `ReceivedEthOrErc20LogParser::parse_log()`, which reads the `value` field directly from the event data bytes:

```rust

### Impact Explanation

For any fee-on-transfer ERC20 token that is added as a supported ckERC20 token:

- The minter's Ethereum address holds `amount - fee` tokens.
- The IC ckERC20 ledger has `amount` tokens minted.
- The ckERC20 supply is unbacked by the difference `fee` per deposit.
- Repeated deposits compound the discrepancy.
- When users attempt to withdraw ckERC20 back to Ethereum, the minter will eventually be unable to fulfill withdrawals because it holds fewer ERC20 tokens than the outstanding ckERC20 supply.
- This is a **ledger conservation bug** / **chain-fusion mint/burn/replay bug**: the 1:1 backing invariant of ckERC20 is permanently broken.

The `erc20_balances` state tracked by the minter will also be inflated by the same amount, causing the minter's own accounting to diverge from on-chain reality.

---

### Likelihood Explanation

The likelihood depends on whether a fee-on-transfer ERC20 token is ever added as a supported ckERC20 token via NNS governance. Several widely-used tokens (e.g., USDT historically, STA, PAXG, and many DeFi tokens) implement transfer fees. The helper contract enforces no whitelist — any ERC20 address can be passed to `depositErc20`. The minter enforces a whitelist of supported tokens, but the whitelist is governed by NNS proposals. If a fee-on-transfer token is ever proposed and accepted, the vulnerability is immediately exploitable by any user making a deposit. No privileged access is required beyond the governance decision to add the token.

---

### Recommendation

In `depositErc20`, measure the actual received amount by comparing the minter's balance before and after the transfer, and emit that value in the event:

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 receivedAmount = erc20Token.balanceOf(minterAddress) - balanceBefore;

    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        receivedAmount,   // ← actual received amount
        principal,
        subaccount
    );
}
```

Alternatively, the NNS governance process for adding new ckERC20 tokens should explicitly verify that the ERC20 token does not implement transfer fees before approving it.

### Proof of Concept

1. A fee-on-transfer ERC20 token `FeeToken` (1% fee on every transfer) is added as a supported ckERC20 token via NNS governance.
2. Alice calls `depositErc20(FeeToken, 1_000_000, alicePrincipal, subaccount)` on the helper contract.
3. `safeTransferFrom` transfers 1,000,000 tokens from Alice; the ERC20 contract deducts 1% fee, so the minter address receives 990,000 tokens.
4. The helper contract emits `ReceivedEthOrErc20(FeeToken, Alice, 1_000_000, alicePrincipal, subaccount)`.
5. The IC ckETH minter scrapes the log, parses `value = 1_000_000` from the event data in `ReceivedEthOrErc20LogParser::parse_log`.
6. `mint()` calls `client.transfer(TransferArg { amount: 1_000_000, to: aliceAccount, ... })`, minting 1,000,000 ckFeeToken to Alice.
7. The minter holds 990,000 FeeToken on Ethereum but has issued 1,000,000 ckFeeToken on IC.
8. After 100 such deposits, the minter holds 99,000,000 FeeToken but has issued 100,000,000 ckFeeToken — a 1,000,000 token shortfall that grows with every deposit. [1](#0-0) [6](#0-5) [7](#0-6)

impl LogParser for ReceivedEthOrErc20LogParser {
    fn parse_log(entry: LogEntry) -> Result<ReceivedEvent, ReceivedEventError> {
        let (block_number, event_source) = ensure_not_pending(&entry)?;
        ensure_not_removed(&entry, event_source)?;

        ensure_topics(
            &entry,
            |topics| {
                topics.len() == 4
                    && topics.first()
                        == Some(&Hex32::from(
                            RECEIVED_ETH_OR_ERC20_WITH_SUBACCOUNT_EVENT_TOPIC,
                        ))
            },
            event_source,
        )?;
        let erc20_contract_address = parse_address(&entry.topics[1], event_source)?;
        let from_address = parse_address(&entry.topics[2], event_source)?;
        let principal = parse_principal(&entry.topics[3], event_source)?;
        let [value_bytes, subaccount_bytes] =
            parse_hex_into_32_byte_words(entry.data, event_source)?;
        let subaccount = LedgerSubaccount::from_bytes(subaccount_bytes);
        let EventSource {
            transaction_hash,
            log_index,
        } = event_source;

        if erc20_contract_address == Address::ZERO {
            let value = Wei::from_be_bytes(value_bytes);
            return Ok(ReceivedEthEvent {
                transaction_hash,
                block_number,
                log_index,
                from_address,
                value,
                principal,
                subaccount,
            }
            .into());
        }

        let value = Erc20Value::from_be_bytes(value_bytes);
        Ok(ReceivedErc20Event {
            transaction_hash,
            block_number,
            log_index,
            from_address,
            value,
            principal,
            erc20_contract_address,
            subaccount,
        }
        .into())
    }
