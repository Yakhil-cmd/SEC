### Title
Zero EVM Gas Cost on Exit Precompiles with No Minimum Amount Enforcement Enables Bridge DoS and Temporary Fund Freeze — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` and `ExitToEthereum` precompiles in Aurora Engine charge **zero EVM gas** for execution and enforce **no minimum withdrawal amount**. An unprivileged EVM user can deploy a contract that calls these precompiles in a tight loop with dust amounts (e.g., 1 wei), generating a flood of NEAR cross-contract promise calls to the eth-connector bridge. This overwhelms the off-chain bridge infrastructure, delays or blocks legitimate withdrawals (temporary fund freeze), and drains the NEAR gas budget of the bridge operator's account.

---

The `parse_amount` helper used in both precompile `run` paths only rejects amounts exceeding `u128::MAX`; it does **not** enforce any minimum:

```rust
fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    Ok(amount)
}
``` [4](#0-3) 

Neither `ExitToNear::run` nor `ExitToEthereum::run` adds any minimum-amount guard after parsing. For the ERC-20 exit path in `ExitToEthereum::run`, the amount is parsed and immediately used to build the bridge promise with no floor check: [5](#0-4) 

For the ERC-20 exit path in `ExitToNear::run`, the amount flows through `exit_erc20_token_to_near` with no minimum check: [6](#0-5) 

Each successful precompile invocation emits a NEAR `PromiseArgs::Create` log targeting the eth-connector's `withdraw` (or `ft_transfer` / `ft_transfer_call`) method, which the Aurora runtime schedules as a real cross-contract call: [7](#0-6) 

The `EvmErc20.sol` and `EvmErc20V2.sol` contracts expose `withdrawToNear` and `withdrawToEthereum` as public entry points with no minimum amount guard, making the precompile reachable by any token holder: [8](#0-7) 

The on-chain `withdraw` method in the connector also has no minimum amount check: [9](#0-8) 

---

### Impact Explanation

Because the EVM gas cost is zero, an attacker can write a Solidity contract that calls the exit precompile in a loop thousands of times within a single EVM transaction, each iteration with `amount = 1`. Every iteration schedules a real NEAR cross-contract call to the eth-connector. The NEAR promise queue is flooded with dust withdrawal requests. The off-chain bridge relayer must process each promise, consuming its NEAR gas budget and delaying or blocking all legitimate user withdrawals. This constitutes **temporary freezing of funds** for honest users and **drainage of the bridge operator's NEAR gas balance**. [1](#0-0) 

---

### Likelihood Explanation

The attack requires only a small token balance (1 wei of ETH or 1 unit of any bridged ERC-20) and the ability to deploy a Solidity contract on Aurora — both are trivially achievable by any unprivileged user. The zero EVM gas cost means the attacker pays only NEAR gas for the outer `submit` transaction, making the cost-per-promise negligible. No special permissions, admin keys, or governance actions are required. [2](#0-1) 

---

### Recommendation

1. **Set a non-zero EVM gas cost** for `EXIT_TO_NEAR_GAS` and `EXIT_TO_ETHEREUM_GAS`. The TODO comments at lines 46–49 acknowledge this is unresolved; it must be resolved before mainnet exposure.
2. **Enforce a minimum withdrawal amount** inside both `ExitToNear::run` and `ExitToEthereum::run` immediately after `parse_amount`, rejecting any amount below a protocol-defined floor (e.g., 1000 wei).
3. **Add the same minimum check** to the `withdraw` function in `engine/src/contract_methods/connector.rs`. [4](#0-3) 

---

### Proof of Concept

```solidity
// Attacker contract deployed on Aurora
interface IEvmErc20 {
    function withdrawToEthereum(address recipient, uint256 amount) external;
}

contract FloodBridge {
    function attack(address token, address recipient) external {
        IEvmErc20 erc20 = IEvmErc20(token);
        // EVM gas cost per precompile call = 0, so this loop is essentially free
        for (uint256 i = 0; i < 10_000; i++) {
            erc20.withdrawToEthereum(recipient, 1); // 1 wei each
        }
    }
}
```

Each iteration of the loop calls `withdrawToEthereum` → `_burn(sender, 1)` → exit precompile → schedules a NEAR `withdraw` promise on the eth-connector. After 10,000 iterations, the NEAR promise queue contains 10,000 dust withdrawal requests. The off-chain bridge relayer must process all of them before legitimate withdrawals can proceed, effectively freezing honest users' funds in transit. [10](#0-9)

fn exit_erc20_token_to_near<I: IO>(
    context: &Context,
    exit_params: &Erc20TokenParams,
    io: &I,
) -> Result<
    (
        AccountId,
        String,
        events::ExitToNear,
        String,
        Option<TransferNearArgs>,
    ),
    ExitError,
> {
    // In case of withdrawing ERC-20 tokens, the `apparent_value` should be zero. In opposite way
    // the funds will be locked in the address of the precompile without any possibility
    // to withdraw them in the future. So, in case if the `apparent_value` is not zero, the error
    // will be returned to prevent that.
    if context.apparent_value != U256::zero() {
        return Err(ExitError::Other(Cow::from(
            "ERR_ETH_ATTACHED_FOR_ERC20_EXIT",
        )));
    }

    let erc20_address = context.caller; // because ERC-20 contract calls the precompile.
    let nep141_account_id = get_nep141_from_erc20(erc20_address.as_bytes(), io)?;

    let (nep141_account_id, args, method, transfer_near_args, event) = match exit_params.message {
        // wNEAR address should be set via the `factory_set_wnear_address` transaction first.
        Some(Message::UnwrapWnear) if erc20_address == get_wnear_address(io).raw() =>
        // The flow is following here:
        // 1. We call `near_withdraw` on wNEAR account id on `aurora` behalf.
        // In such way we unwrap wNEAR to NEAR.
        // 2. After that, we call callback `exit_to_near_precompile_callback` on the `aurora`
        // in which make transfer of unwrapped NEAR to the `target_account_id`.
        {
            (
                nep141_account_id,
                format!(r#"{{"amount":"{}"}}"#, exit_params.amount.as_u128()),
                "near_withdraw",
                Some(TransferNearArgs {
                    target_account_id: exit_params.receiver_account_id.clone(),
                    amount: exit_params.amount.as_u128(),
                }),
                events::ExitToNear::Legacy(ExitToNearLegacy {
                    sender: Address::new(erc20_address),
                    erc20_address: Address::new(erc20_address),
                    dest: exit_params.receiver_account_id.to_string(),
                    amount: exit_params.amount,
                }),
            )
        }
        // In this flow, we're just forwarding the `msg` to the `ft_transfer_call` transaction.
        Some(Message::Omni(msg)) => (
            nep141_account_id,
            ft_transfer_call_args(&exit_params.receiver_account_id, exit_params.amount, msg)?,
            "ft_transfer_call",
            None,
            events::ExitToNear::Omni(ExitToNearOmni {
                sender: Address::new(erc20_address),
                erc20_address: Address::new(erc20_address),
                dest: exit_params.receiver_account_id.to_string(),
                amount: exit_params.amount,
                msg: msg.to_string(),
            }),
        ),
        // The legacy flow. Just withdraw the tokens to the NEAR account id.
        // P.S. We use underscore here instead of `None` to handle the case when a user
        // could add the `unwrap` suffix for non wNEAR ERC-20 token by mistake.
        _ => {
            // There is no way to inject json, given the encoding of both arguments
            // as decimal and valid account id respectively.
            (
                nep141_account_id,
                format!(
                    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
                    exit_params.receiver_account_id,
                    exit_params.amount.as_u128()
                ),
                "ft_transfer",
                None,
                events::ExitToNear::Legacy(ExitToNearLegacy {
                    sender: Address::new(erc20_address),
                    erc20_address: Address::new(erc20_address),
                    dest: exit_params.receiver_account_id.to_string(),
                    amount: exit_params.amount,
                }),
            )
        }
    };

    Ok((
        nep141_account_id,
        args,
        event,
        method.to_string(),
        transfer_near_args,
    ))
}
