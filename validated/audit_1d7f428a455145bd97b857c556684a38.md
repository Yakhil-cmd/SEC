Audit Report

## Title
Arbitrary `buyer` Parameter in `refresh_buyer_tokens` Allows Forced Swap Participation and Confirmation Text Bypass - (File: `rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` update handler in the SNS Swap canister accepts an arbitrary `buyer` principal in the request body and uses it as the acting identity without verifying it matches the authenticated caller. Any unprivileged ingress sender can call this method with a victim's principal ID, forcing that victim's ICP (already in their swap subaccount) into the swap and supplying the `confirmation_text` consent field on the victim's behalf.

## Finding Description

In `rs/sns/swap/canister/canister.rs` lines 130–134, the handler unconditionally trusts the `buyer` field from the request body when it is non-empty:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // no caller == buyer check
};
```

There is no guard of the form `assert!(arg.buyer.is_empty() || arg.buyer == caller_principal_id().to_string())`. The resolved principal `p` is then passed directly to `refresh_buyer_token_e8s`, which:

1. Queries the ICP ledger balance of `p`'s subaccount on the swap canister (`rs/sns/swap/src/swap.rs` lines 1153–1163).
2. Validates the `confirmation_text` supplied by the **attacker** as if it were supplied by `p` (`rs/sns/swap/src/swap.rs` line 1150, `validate_confirmation_text` at lines 363–384).
3. Registers `p` as a swap participant and locks their ICP (`rs/sns/swap/src/swap.rs` lines 1285–1288).

The `confirmation_text` check only compares the submitted string against the SNS-configured expected text; it does not verify that the submitter is the token holder. The expected text is publicly readable via the `get_init` query endpoint (`rs/sns/swap/canister/canister.rs` lines 213–215).

The integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 directly demonstrates the exploit path: the **default agent** (a different identity) successfully calls `refresh_buyer_tokens` with `wealthy_user_identity.principal_id` as the `buyer` field after ICP has been transferred to the wealthy user's subaccount, and the call succeeds (`unwrap()`). The earlier failure at line 818 is solely because no ICP had yet been deposited (balance below `min_participant_icp_e8s`), not due to any caller-identity check.

## Impact Explanation

This is a **High** severity finding matching: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

A victim who has transferred ICP to their swap subaccount but has not yet called `refresh_buyer_tokens` (e.g., they changed their mind) can be force-registered as a participant. Their ICP is locked until the swap closes. If the swap commits, they receive SNS tokens they did not intend to purchase and cannot recover their ICP. If the swap aborts, they must call `error_refund_icp` to recover funds. Additionally, the `confirmation_text` consent mechanism — the only explicit user-consent gate in the participation flow — is bypassed, with the attacker's submission recorded as the victim's consent.

## Likelihood Explanation

The attack requires no special privileges, no key material, and no trusted role. The attacker only needs to observe the public ICP ledger for transfers to swap subaccounts and read the `confirmation_text` from the public `get_init` endpoint. A single ingress call per victim is sufficient. The attack is fully automatable and permissionless, making it highly likely to be exploited in practice.

## Recommendation

**Short term**: In `refresh_buyer_tokens`, when the call arrives via ingress, enforce that `arg.buyer` is either empty or equals `caller_principal_id()`. If canister-to-canister notification is needed (e.g., for the Neurons' Fund), gate the `buyer` override on the caller being a known trusted canister (e.g., NNS governance canister ID stored in `Init`).

**Long term**: Add an invariant/integration test asserting that calling `refresh_buyer_tokens` with `buyer = B` from a caller `A ≠ B` via ingress is rejected. Audit all other SNS Swap endpoints for similar caller-identity substitution patterns.

## Proof of Concept

The existing integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 already demonstrates the exploit: after ICP is deposited to the wealthy user's subaccount, the default agent (a different principal) calls `refresh_buyer_tokens` with `buyer = wealthy_user_identity.principal_id` and the call succeeds, registering the wealthy user as a participant without their direct invocation.

A minimal standalone PoC:

```
// 1. Alice transfers ICP to her swap subaccount but does not call refresh_buyer_tokens.
// 2. Mallory reads confirmation_text from get_init (public query).
// 3. Mallory sends:
dfx canister call <swap_canister_id> refresh_buyer_tokens '(
  record {
    buyer = "<alice_principal_text>";
    confirmation_text = opt "<confirmation_text_from_get_init>"
  }
)'
// Result: Alice is registered as a swap participant with ICP locked,
// and the swap records Alice as having consented — without Alice ever calling this method.
```

A deterministic integration test can be written using `PocketIC` or `StateMachine`: create two principals A and B, transfer ICP from B to B's swap subaccount, then call `refresh_buyer_tokens` as A with `buyer = B.to_string()`, and assert the call succeeds and B appears in `buyers`.
