Audit Report

## Title
Unbounded Proposal Iteration in `maybe_gc` Causes Instruction-Limit DoS in SNS Governance - (File: `rs/sns/governance/src/governance.rs`)

## Summary
`maybe_gc` in the SNS governance canister iterates over the entire `self.proto.proposals` map with no instruction-limit guard. If the map grows large enough, the message traps at the instruction limit. Because `latest_gc_timestamp_seconds` is written before the loop and is rolled back on trap (no `await` separates it from the loop), the GC condition is satisfied again on every subsequent periodic-task invocation, creating a permanent retry loop that prevents garbage collection and allows proposals to accumulate without bound.

## Finding Description
`run_periodic_tasks` calls `self.maybe_gc()` unconditionally as its final statement, after `self.compute_cached_metrics().await` at line 5531. [1](#0-0) 

Inside `maybe_gc`, `latest_gc_timestamp_seconds` is updated at line 5406 before any iteration begins. [2](#0-1) 

The function then performs two unbounded passes over `self.proto.proposals`: first to build `action_to_proposals` (line 5436), then to purge old entries (lines 5445–5465). There is no call to `is_message_over_threshold`, no batch limit, and no early-exit mechanism. [3](#0-2) 

Because `maybe_gc` contains no `await` point, any trap caused by exceeding the 40 B-instruction per-message limit rolls back all state changes made since the preceding `compute_cached_metrics().await`. This includes the `latest_gc_timestamp_seconds` write at line 5406, so the GC trigger condition (`now > latest_gc_timestamp_seconds + 86400`) is immediately satisfied again on the next heartbeat.

By contrast, the `ic_nervous_system_long_message` crate provides `is_message_over_threshold` and `noop_self_call_if_over_instructions`, which are used in NNS governance's reward distribution and voting paths to avoid exactly this class of trap. [4](#0-3) 

No equivalent guard has been applied to SNS governance's `maybe_gc`.

## Impact Explanation
This is an application/platform-level DoS on an in-scope SNS governance canister. Once the proposals map is large enough to exhaust the instruction budget, every periodic-task execution traps at `maybe_gc`, permanently disabling garbage collection. Proposals accumulate without bound, growing heap memory until the canister approaches its memory limit or becomes too slow to service user calls. `compute_cached_metrics` state changes are also rolled back on every heartbeat, degrading observability. No funds are lost, but the governance canister's liveness and correctness are materially impaired. This matches the allowed High impact: "Application/platform-level DoS … not based on raw volumetric DDoS." Given the cost required to bootstrap the attack (submitting enough proposals to exhaust 40 B instructions), Medium severity is the more conservative and appropriate classification.

## Likelihood Explanation
Any SNS neuron holder with sufficient stake to submit proposals can trigger this. No privileged role is required. The `max_proposals_to_keep_per_action` limit is a soft cap enforced only by GC itself; once GC is broken the cap is unenforced. The attack is self-reinforcing: each failed GC run leaves more proposals in place, making the next run more likely to trap. The initial cost depends on the SNS's minimum proposal stake and the per-proposal instruction cost of iterating the map, but the attack requires no special technical capability beyond repeated proposal submission.

## Recommendation
1. Add an instruction-limit guard inside `maybe_gc` using `is_message_over_threshold` from `ic_nervous_system_long_message`, mirroring the pattern already used in NNS governance's reward distribution. [4](#0-3) 
2. Persist a cursor across invocations (similar to `groom_some_neurons` / `prune_some_following`) so that GC resumes where it left off rather than restarting from the beginning each time.
3. Enforce a hard cap on the total number of live proposals at submission time so the map cannot grow without bound regardless of GC health.

## Proof of Concept
1. Acquire SNS governance tokens and submit a large number of proposals of the same action type (e.g., `Motion`) so that `self.proto.proposals` grows to a size where iterating it exhausts the 40 B-instruction per-message limit.
2. Wait for `run_periodic_tasks` to satisfy the GC condition (`now > latest_gc_timestamp_seconds + 86400` or `proposals.len() > latest_gc_num_proposals + 100`).
3. Observe that `maybe_gc` iterates the full map at line 5436, exceeds the instruction limit, and the message traps.
4. Confirm via canister logs or state inspection that `latest_gc_timestamp_seconds` was not durably updated (rolled back to its pre-call value), so the GC condition is satisfied again immediately.
5. Observe that every subsequent periodic-task invocation repeats the trap, proposals continue to accumulate, and heap memory grows without bound.
A deterministic integration test using `StateMachine` with a reduced instruction limit (as demonstrated in `rs/nervous_system/long_message/tests/test_message_breaks.rs`) can reproduce the trap without mainnet deployment. [5](#0-4)
