### Title
Permanent ERC-20/ETH Token Loss When `ExitToNear` NEAR-Side Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` Cargo feature is not compiled in, the `ExitToNear` precompile schedules no refund callback. If the NEAR-side `ft_transfer` or `ft_transfer_call` promise fails after EVM-side tokens have already been burned/deducted, those tokens are permanently destroyed with no recovery path available to the user.

---

### Finding Description

**Root cause — `engine-precompiles/src/native.rs`, `ExitToNear::run()`:**

The precompile constructs `ExitToNearPrecompileCallbackArgs` with `refund` hardcoded to `None` when the `error_refund` feature is absent:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,                    // ← no refund address recorded
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When both `refund` and `transfer_near` are `None`, `callback_args` equals the default value, so the code takes the branch that attaches **no callback at all** to the outgoing promise:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

**Root cause — `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback()`:**

Even when a callback *is* attached (e.g., the wNEAR-unwrap path where `transfer_near` is `Some`), the callback function contains an explicit silent no-op branch reached when the promise fails and `refund` is `None`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← promise failed, refund is None → tokens permanently lost
};
``` [3](#0-2) 

**Execution sequence that causes permanent loss:**

1. User calls an ERC-20 contract's burn function, which internally calls the `ExitToNear` precompile at address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`.
2. The precompile burns the ERC-20 tokens from the user's EVM balance (or deducts ETH via `context.apparent_value`).
3. A NEAR-side `ft_transfer` (or `ft_transfer_call`) promise is dispatched to the NEP-141 contract.
4. The promise fails — e.g., the recipient NEAR account is not registered with the NEP-141 token (has not called `storage_deposit`), or the NEP-141 contract is paused.
5. Without `error_refund`: no callback fires, no re-mint occurs, tokens are gone. [4](#0-3) 

---

### Impact Explanation

**Permanent freezing of funds (Critical).** ERC-20 tokens are burned from the EVM state before the NEAR promise is dispatched. If the promise fails, the tokens no longer exist anywhere — they are not in the EVM (burned) and never arrived on NEAR. The user has no mechanism to recover them. The same applies to ETH sent via the base-token exit path: ETH is deducted from the sender's EVM balance before the promise fires.

The test suite explicitly confirms this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

---

### Likelihood Explanation

**Medium–High.** The failure condition is easily triggered by any unprivileged EVM user:

- Specifying a recipient NEAR account that has not called `storage_deposit` on the NEP-141 contract (the most common mistake for new accounts).
- Specifying a recipient that is not a valid NEAR account ID.
- The NEP-141 contract being paused at the time the promise executes.

No admin action, key compromise, or special privilege is required. Any EVM address that holds ERC-20 tokens can trigger this by calling the exit precompile with a failing recipient. [6](#0-5) 

---

### Recommendation

1. **Make `error_refund` a default Cargo feature** so the refund path is always compiled in for production builds.
2. Alternatively, **always include the refund callback** regardless of the feature flag, using `context.caller` as the default refund address when no explicit refund address is provided.
3. As a belt-and-suspenders measure, **validate that the recipient NEAR account is registered** with the NEP-141 contract before burning EVM-side tokens, reverting the EVM call if not.

---

### Proof of Concept

The existing integration test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` directly demonstrates the vulnerability:

```
1. Bridge NEP-141 tokens to ERC-20 on Aurora.
2. Call ExitToNear targeting "unregistered.near" (not registered with NEP-141).
3. ft_transfer promise fails.
4. Without error_refund: ERC-20 balance = FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT
   (FT_EXIT_AMOUNT tokens are permanently destroyed).
5. With error_refund: ERC-20 balance = FT_TRANSFER_AMOUNT (tokens refunded).
``` [7](#0-6) 

The same pattern applies to the ETH base-token exit path, confirmed by `test_exit_to_near_eth_refund`:

```rust
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [8](#0-7)
