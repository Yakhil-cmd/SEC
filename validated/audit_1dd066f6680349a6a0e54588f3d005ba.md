### Title
Unvalidated NEP-141 Account ID in `deploy_erc20_token` Enables Unbacked ERC-20 Token Minting - (File: `engine/src/contract_methods/connector.rs`)

### Summary

`deploy_erc20_token` carries no access control and performs no validation that the caller-supplied `nep141` account ID is a legitimate NEP-141 token contract. Because `ft_on_transfer` unconditionally trusts `predecessor_account_id` as the token identity, any NEAR account that has been registered via `deploy_erc20_token` can call `ft_on_transfer` directly and instruct Aurora to mint an arbitrary quantity of ERC-20 tokens without ever depositing real NEP-141 tokens.

### Finding Description

`deploy_erc20_token` is a public NEAR entry point with no caller restriction beyond `require_running`: [1](#0-0) 

The `Legacy` variant immediately deploys an ERC-20 and writes the `nep141 → erc20_address` mapping with no cross-contract verification: [2](#0-1) 

`register_token` only rejects an already-registered account ID; it never verifies that the account actually implements NEP-141: [3](#0-2) 

`ft_on_transfer` is also a public entry point. When the caller is not the ETH connector it routes directly to `receive_erc20_tokens`, passing `predecessor_account_id` as the token: [4](#0-3) 

`receive_erc20_tokens` looks up the ERC-20 address for the predecessor, then has the Aurora Engine (acting as admin) call the ERC-20 contract's mint function with the attacker-supplied `amount` and `recipient`: [5](#0-4) 

The `WithMetadata` variant calls `ft_metadata` on the supplied account, but an attacker can deploy a fake contract that responds to `ft_metadata`, so this path is equally exploitable. [6](#0-5) 

The comment inside `WithMetadata` even acknowledges the intent that only the owner should call this, yet no enforcement exists: [7](#0-6) 

The `mirror_erc20_token` function (the analogous silo-mirroring path) does enforce `require_owner_only`, confirming the developers know this pattern is needed but omitted it from `deploy_erc20_token`: [8](#0-7) 

### Impact Explanation

An attacker can mint an unbounded quantity of ERC-20 tokens on Aurora that are backed by zero real NEP-141 tokens. If those tokens are paired in a DEX liquidity pool with real assets (ETH, USDC, etc.) — whether by the attacker luring liquidity providers or by front-running a legitimate token registration — the attacker can swap the minted tokens for the real assets and drain the pool. This constitutes direct theft of user funds held in Aurora-side DeFi protocols.

### Likelihood Explanation

Both entry points (`deploy_erc20_token` and `ft_on_transfer`) are fully public NEAR functions callable by any account with no deposit or permission requirement. The attacker needs only a NEAR account and the ability to call two contract methods. Attracting liquidity to a fake pool requires social engineering, but the technical barrier is zero.

### Recommendation

1. **Add access control to `deploy_erc20_token`**: Restrict it to the contract owner or a governed whitelist, consistent with how `mirror_erc20_token` is already protected.
2. **Validate the NEP-141 account ID**: Before registering a mapping, verify on-chain (via a cross-contract call) that the account actually implements the NEP-141 standard (e.g., `ft_total_supply` succeeds), and reject accounts that do not.
3. **Track registered tokens in a canonical registry**: Only allow `ft_on_transfer` to mint ERC-20 tokens for accounts that were registered through the validated path, analogous to how Escher should track editions deployed from the legitimate factory.

### Proof of Concept

```
1. Attacker creates NEAR account `attacker.near`.

2. Attacker calls `deploy_erc20_token` on Aurora Engine
   with args = DeployErc20TokenArgs::Legacy("attacker.near").
   → No access control check fires.
   → Aurora deploys an ERC-20 at address ERC20_A and writes
     nep141_erc20_map["attacker.near"] = ERC20_A.

3. Attacker (optionally) seeds a DEX pool on Aurora pairing
   ERC20_A with ETH, attracting other liquidity providers.

4. Attacker calls `ft_on_transfer` on Aurora Engine
   from `attacker.near` with:
     sender_id = "attacker.near"
     amount    = 1_000_000_000_000_000_000   (arbitrary)
     msg       = hex(attacker_evm_address)

5. ft_on_transfer sees predecessor ≠ ETH connector,
   calls receive_erc20_tokens("attacker.near", args, …).
   get_erc20_from_nep141 returns ERC20_A (registered in step 2).
   Aurora Engine mints 1e18 ERC20_A tokens to attacker_evm_address.
   No real NEP-141 tokens were ever transferred.

6. Attacker swaps ERC20_A for ETH in the DEX pool,
   draining real funds from other liquidity providers.
```

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
