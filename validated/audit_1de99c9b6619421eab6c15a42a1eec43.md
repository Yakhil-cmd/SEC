### Title
Silent Success on Empty-Account Call to L2AssetTracker Allows Token Minting Without Accounting Update — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`notify_l2_asset_tracker` determines success solely by checking `result.failed()` (i.e., whether the call reverted). By EVM convention, a call to an address with no deployed code returns `CallResult::Successful` with empty returndata — it never reverts. If `L2AssetTracker` is absent at `0x1000f`, the notification silently "succeeds," the bootloader proceeds to mint tokens via `transfer_from_treasury`, and the asset tracker's deposit accounting is never updated. The code itself acknowledges this exact behavior in a comment but treats it as an acceptable assumption rather than a guarded invariant.

---

### Finding Description

In `notify_l2_asset_tracker`, the bootloader calls `run_single_interaction` targeting `L2_ASSET_TRACKER_ADDRESS` and evaluates the result with:

```rust
Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
``` [1](#0-0) 

`failed()` returns `true` only when the call reverts. It returns `false` for a successful call — including a call to an empty account that executed no code.

This is enforced by `call_execute_callee_frame` in the runner:

```rust
// By convention, calls to empty accounts succeed without any return data
if next_ee_type == ExecutionEnvironmentType::NoEE {
    ...
    return Ok((
        ...,
        CallResult::Successful { return_values: ReturnValues::empty() },
    ));
}
``` [2](#0-1) 

`L2_ASSET_TRACKER_ADDRESS` is defined as `B160::from_limbs([0x1000f, 0, 0])` — a regular (non-hook) address. [3](#0-2) 

The code's own comment documents the silent-success path:

> "If no contract is deployed at L2AssetTracker, the call succeeds silently (a call to an empty address returns success with no returndata in EVM). However, we are certain that L2AssetTracker is available after the upgrade." [4](#0-3) 

Because `failed` is `false`, the guard at line 901 is not triggered, and `mint_base_token` proceeds to call `transfer_from_treasury`, which subtracts from the treasury and credits the recipient: [5](#0-4) 

The asset tracker's `totalSuccessfulDepositsFromL1` storage slot is never written. The token supply on L2 increases, but the bridge's accounting record does not.

---

### Impact Explanation

Every L1→L2 deposit transaction with `total_deposited > 0` that executes while `L2AssetTracker` is absent at `0x1000f` will:

1. Mint base tokens to the recipient (treasury balance decreases, recipient balance increases — real state change).
2. Skip the `handleFinalizeBaseTokenBridgingOnL2` accounting update entirely.

This permanently diverges the actual L2 token supply from the amount recorded in `interopInfo[BASE_TOKEN_ASSET_ID].totalSuccessfulDepositsFromL1`. Any protocol logic that relies on this accounting for cross-chain interoperability, migration tracking, or bridge invariants will operate on stale/incorrect data. The divergence is irreversible without a manual correction upgrade.

---

### Likelihood Explanation

The trigger condition — `L2AssetTracker` not deployed at `0x1000f` — can occur during the upgrade window between when the bootloader starts calling `notify_l2_asset_tracker` and when the upgrade transaction that deploys `L2AssetTracker` is finalized. Any L1→L2 deposit processed in that window silently skips accounting. Additionally, if a future upgrade inadvertently removes or replaces the contract at that address, the same silent-success path reactivates. The entry point (L1→L2 deposit transaction) is fully unprivileged — any user can submit one from L1.

---

### Recommendation

Replace the revert-only check with a positive verification that actual code was executed. Options:

1. **Verify returndata is non-empty** if `handleFinalizeBaseTokenBridgingOnL2` is modified to return a value (e.g., `bool success`), and treat empty returndata as failure.
2. **Check that `L2AssetTracker` has deployed code** before calling it (read `unpadded_code_len` from account properties), and treat a zero-code account as a fatal error rather than a silent success.
3. **Assert the invariant at startup** — add a block-initialization check that `L2AssetTracker` is deployed before processing any L1 transactions, halting block production if the contract is absent.

The current design relies on an unguarded assumption ("we are certain that L2AssetTracker is available after the upgrade") that is not enforced in code. The assumption should be converted into an explicit, runtime-checked invariant.

---

### Proof of Concept

1. Deploy ZKsync OS without `L2AssetTracker` at `0x1000f` (pre-upgrade state or after a failed upgrade).
2. Submit an L1→L2 transaction with `total_deposited > 0` (e.g., `to_mint = 1_000_000`).
3. `process_l1_transaction` calls `mint_base_token` → `notify_l2_asset_tracker`.
4. `run_single_interaction` targets `0x1000f`; since no code is deployed there, `call_execute_callee_frame` returns `CallResult::Successful` with empty returndata.
5. `asset_tracker_result.failed()` returns `false`; the `if failed { return Err(...) }` guard is not triggered.
6. `transfer_from_treasury` executes: treasury balance decreases by `1_000_000`, recipient balance increases by `1_000_000`.
7. `L2AssetTracker.interopInfo[BASE_TOKEN_ASSET_ID].totalSuccessfulDepositsFromL1` remains `0`.
8. Token supply on L2 is `1_000_000` but the bridge's accounting records `0` — permanent divergence. [6](#0-5) [7](#0-6)
