### Title
Lack of Output Safeguards in SNS Treasury Manager Deposit and Withdraw Operations Enables Sandwich Attacks on DEX Interactions - (File: rs/sns/governance/src/extensions.rs)

### Summary
The SNS Treasury Manager extension framework, which allows SNS DAOs to deposit and withdraw treasury assets (SNS tokens + ICP) into external DEX liquidity pools, provides no mechanism to enforce minimum acceptable return values (slippage protection) for either deposit or withdraw operations. Because SNS governance proposals are public and have multi-day voting periods before execution, a malicious actor can observe a pending deposit proposal and front-run it by manipulating the DEX pool composition, causing the SNS treasury to receive far fewer LP shares than expected. The codebase itself acknowledges this risk in two separate locations but does not implement a fix.

### Finding Description
The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs` approves the treasury manager canister to spend `treasury_allocation_sns_e8s` and `treasury_allocation_icp_e8s` from the SNS treasury, then calls `deposit` on the extension canister with no postcondition check on the LP tokens or shares received in return. [1](#0-0) 

The `execute_treasury_manager_withdraw` function similarly calls `withdraw` on the extension canister and only logs the returned `Balances` struct without verifying that the tokens returned to the treasury meet any minimum threshold. [2](#0-1) 

The `DepositRequest` type in the Treasury Manager interface contains only `allowances` (how much to deposit) with no `min_lp_tokens_out` or equivalent slippage guard field. The `WithdrawRequest` type similarly contains only destination accounts with no minimum return amount. [3](#0-2) 

The `validate_deposit_operation_impl` function only validates that the requested deposit does not exceed 50% of the current treasury balance. It performs no check on the expected output from the DEX. [4](#0-3)

The `validate_and_render_register_extension` function in `rs/sns/governance/src/proposal.rs` renders a warning to voters: [6](#0-5) 

Despite acknowledging the risk, neither the `DepositRequest`/`WithdrawRequest` types nor the governance execution layer enforce any minimum acceptable return value.

### Impact Explanation
An SNS DAO that votes to deposit treasury funds into a DEX liquidity pool (e.g., via the KongSwap adaptor) can have those funds extracted at an unfavorable rate. A malicious actor who front-runs the deposit by manipulating the DEX pool composition forces the treasury to receive far fewer LP shares than the deposited value warrants. On withdrawal, the same attack can be applied in reverse: the attacker manipulates the pool so the treasury receives fewer tokens back than the LP shares represent. In both cases, the SNS treasury (and by extension all SNS token holders) suffers a direct, irreversible financial loss. The magnitude scales with the deposit/withdrawal size, which can be up to 50% of the treasury balance per proposal. [7](#0-6) 

### Likelihood Explanation
SNS governance proposals are fully public and have multi-day voting periods before execution. This gives any observer ample time to prepare a front-running transaction against the target DEX canister. On the Internet Computer, messages are processed in rounds; a malicious actor can submit a DEX manipulation call in the same or an adjacent round before the governance proposal execution call is processed. The attack does not require any privileged access, leaked keys, or subnet-majority corruption — only the ability to call the DEX canister as an unprivileged principal, which is open to any canister caller. The KongSwap adaptor integration test confirms this is a live, production-targeted flow. [8](#0-7) 

### Recommendation
1. **Add a `min_lp_tokens_out` field to `DepositRequest`** and a `min_tokens_out` field to `WithdrawRequest` in `rs/sns/treasury_manager/treasury_manager.did`. These fields should be mandatory for DEX-backed implementations.
2. **Enforce the minimum in `execute_treasury_manager_deposit` and `execute_treasury_manager_withdraw`** in `rs/sns/governance/src/extensions.rs` by inspecting the returned `Balances` struct and reverting (or logging a governance failure) if the actual output falls below the proposal-specified minimum.
3. **Require SNS deposit proposals to include a `min_lp_tokens_out` parameter** in `ValidatedDepositOperationArg`, so voters can explicitly approve the slippage tolerance at proposal creation time.
4. **Add a test** that verifies a deposit proposal fails (or is flagged) when the DEX returns fewer LP tokens than the specified minimum, analogous to the fix validation that was missing in the referenced external report.

### Proof of Concept
1. An SNS DAO submits a `ExecuteExtensionOperation` proposal with `operation_name = "deposit"`, `treasury_allocation_sns_e8s = X`, `treasury_allocation_icp_e8s = Y` targeting a KongSwap adaptor canister.
2. The proposal enters the voting period (publicly visible on-chain).
3. A malicious actor calls the KongSwap DEX canister directly, depositing a large amount of one token to skew the pool ratio significantly before the governance proposal executes.
4. The governance proposal executes: `execute_treasury_manager_deposit` calls `approve_treasury_manager` then `deposit`. The treasury manager forwards the tokens to KongSwap at the now-skewed ratio.
5. The SNS treasury receives LP shares worth significantly less than `X` SNS + `Y` ICP.
6. The attacker withdraws their position from KongSwap, restoring the pool ratio and pocketing the difference.
7. The `execute_treasury_manager_deposit` function returns `Ok(())` with no error, because no minimum output check exists. [1](#0-0) [3](#0-2)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager withdraw operation
async fn execute_treasury_manager_withdraw(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedWithdrawOperationArg,
) -> Result<(), GovernanceError> {
    let arg_blob = construct_treasury_manager_withdraw_payload(arg.original).map_err(|err| {
        GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!("Failed to construct treasury manager withdraw payload: {err}"),
        )
    })?;

    let balances = governance
        .env
        .call_canister(extension_canister_id, "withdraw", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.withdraw failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!(
                        "Error decoding TreasuryManager.withdraw response: {err:?}"
                    ),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.withdraw failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.withdraw succeeded with response: {:?}",
        balances
    );

    Ok(())
}
