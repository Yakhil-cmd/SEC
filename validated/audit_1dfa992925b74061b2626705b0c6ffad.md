Audit Report

## Title
SNS Swap Participant Slot Exhaustion via Sybil Participation — (File: rs/sns/swap/src/swap.rs)

## Summary
The `refresh_buyer_token_e8s` function enforces a hard ceiling on direct participants by checking `self.buyers.len()` against `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`. Once a principal is inserted into `self.buyers`, no code path removes it while the swap is in the `Open` lifecycle. An adversary controlling many sybil principals can fill all available slots, permanently blocking new legitimate participants from joining the swap during its active window.

## Finding Description
The participant ceiling check at `rs/sns/swap/src/swap.rs` L1180–1197 computes `num_direct_participants = self.buyers.len()` and rejects any new participant when `(num_direct_participants + 1) * num_sns_neurons_per_basket > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`. [1](#0-0) 

When the check passes, the buyer is unconditionally inserted at L1285–1288 via `self.buyers.entry(buyer.to_string()).or_insert_with(...)`, and `insert_buyer_into_buyers_list_index` is called at L1277 to persist the entry in stable memory. [2](#0-1) 

A grep for `buyers.remove` in `swap.rs` returns no matches, confirming there is no removal path during the `Open` state. The only ICP-return path, `error_refund_icp`, is gated behind `Lifecycle::Aborted || Lifecycle::Committed` at L1932–1936 and does not remove the entry from `self.buyers` even when it executes. [3](#0-2) 

The root cause is that `self.buyers.len()` counts all registered principals regardless of their actual ICP balance, and there is no eviction mechanism during the `Open` phase.

## Impact Explanation
Once `self.buyers` is saturated, any new principal calling `refresh_buyer_token_e8s` receives the error "The swap has reached the maximum number of direct participants." If the swap's `min_participants` threshold cannot be reached because all slots are held by sybil principals, the swap aborts. After abort, the adversary recovers all deposited ICP via `sweep_icp` or `error_refund_icp`, making the net cost only transfer fees. The SNS project loses its decentralization swap entirely. This matches the allowed High impact: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
Creating IC principals is free. The adversary must lock `floor(MAX_NEURONS_FOR_DIRECT_PARTICIPANTS / basket_size) × min_participant_icp_e8s` ICP for the swap duration (days to weeks), but this capital is fully recoverable after abort. With a typical basket size of 3–5, the required number of sybil principals is in the tens of thousands. Transfer fees for that many transactions are the only unrecoverable cost. A motivated adversary — a competitor or someone who shorted the SNS token — has clear economic incentive. The attack requires no special privileges and is triggerable by any unprivileged user.

## Recommendation
Count only buyers with a non-zero `amount_icp_e8s` toward the participant ceiling, or remove a buyer from `self.buyers` and `BUYERS_LIST_INDEX` when their balance drops to zero (detectable during `refresh_buyer_token_e8s` via the ledger query already performed). Alternatively, introduce a withdrawal mechanism during the `Open` phase that removes the buyer entry upon full withdrawal, or require a meaningful minimum deposit that makes the attack economically prohibitive even before recovery.

## Proof of Concept
1. Adversary generates `N = floor(MAX_NEURONS_FOR_DIRECT_PARTICIPANTS / basket_size)` sybil principals.
2. For each sybil principal `P_i`, adversary transfers `min_participant_icp_e8s` ICP to `subaccount(swap_canister, P_i)` on the ICP ledger.
3. Adversary calls `refresh_buyer_token_e8s` for each `P_i`. Each call passes the check at L1187 (since `(i+1) * basket_size ≤ MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`) and inserts `P_i` into `self.buyers`.
4. After step 3, `self.buyers.len() == N`. Any new legitimate principal `P_legit` calling `refresh_buyer_token_e8s` hits the check at L1187: `(N+1) * basket_size > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` → rejected.
5. The swap cannot reach `min_participants` from legitimate users → swap aborts at `swap_due_timestamp_seconds`.
6. Adversary calls `sweep_icp` or `error_refund_icp` for each `P_i` → recovers all ICP minus transfer fees.

This is directly reproducible as a unit test following the pattern of `test_refresh_buyer_tokens_no_sns_neuron_baskets_available` in `rs/sns/swap/tests/swap.rs` L4986–5036, which already demonstrates slot exhaustion with a large basket size. [4](#0-3)

fn test_refresh_buyer_tokens_no_sns_neuron_baskets_available() {
    let user1 = PrincipalId::new_user_test_id(1);
    let user2 = PrincipalId::new_user_test_id(2);
    let user3 = PrincipalId::new_user_test_id(3);
    let user4 = PrincipalId::new_user_test_id(4);

    let mut swap = SwapBuilder::new()
        .with_sns_governance_canister_id(SNS_GOVERNANCE_CANISTER_ID)
        .with_lifecycle(Open)
        .with_swap_start_due(Some(START_TIMESTAMP_SECONDS), Some(END_TIMESTAMP_SECONDS))
        .with_min_participants(1)
        .with_min_max_participant_icp(2 * E8, 40 * E8)
        .with_min_max_direct_participation(5 * E8, 100 * E8)
        .with_sns_tokens(100_000 * E8)
        // An extremely large basket size, so we can reach MAX_NEURONS_FOR_DIRECT_PARTICIPANTS with
        // a relatively small number of participants.
        .with_neuron_basket_count(33_000)
        .with_neurons_fund_participation()
        .build();

    let params = swap.params.unwrap();

    let amount_user1_0 = 5 * E8;
    let amount_user2_0 = 40 * E8;
    let amount_user3_0 = 40 * E8;
    let amount_user4_0 = 99 * E8 - (amount_user2_0 + amount_user3_0);

    // All tokens but one should be already bought up by users 2 to 4 --> 99 Tokens were bought
    buy_token_ok(&mut swap, &user2, &amount_user2_0, &amount_user2_0);
    buy_token_ok(&mut swap, &user3, &amount_user3_0, &amount_user3_0);
    buy_token_ok(&mut swap, &user4, &amount_user4_0, &amount_user4_0);

    // Make sure the 99 tokens were registered
    assert_eq!(
        swap.get_buyers_total().buyers_total,
        amount_user2_0 + amount_user3_0 + amount_user4_0
    );

    // Make sure that only an amount smaller than the minimum amount to be bought per user is available
    assert!(
        params.max_direct_participation_icp_e8s.unwrap() - swap.get_buyers_total().buyers_total
            < params.min_participant_icp_e8s
    );

    // No user that has not participated in the swap yet can buy this one token left
    buy_token_err(
        &mut swap,
        &user1,
        &amount_user1_0,
        "The swap has reached the maximum number of direct participants",
    );
