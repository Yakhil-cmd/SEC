Audit Report

## Title
Unchecked Index Panic in `construction_hash` on Empty `EnvelopePair` Vec — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

## Summary
The `construction_hash` handler directly indexes `envelope_pairs[0]` after matching a `RequestType::Send` entry via `.find(|(rt, _)| rt.is_transfer())`, with no guard against an empty `Vec<EnvelopePair>`. Because `SignedTransaction::from_str` performs only hex-decode and CBOR deserialization with no structural invariant enforcing a non-empty envelope vec, an unauthenticated attacker can POST a crafted payload containing `(RequestType::Send, vec![])` to the `/construction/hash` endpoint and trigger an index-out-of-bounds panic, causing a DoS of the Rosetta service.

## Finding Description
In `construction_hash.rs` lines 18–28, after deserializing the signed transaction, the handler iterates in reverse to find the first transfer request:

```rust
if let Some((request_type, envelope_pairs)) =
    signed_transaction
        .requests
        .iter()
        .rev()
        .find(|(rt, _)| rt.is_transfer())
{
    TransactionIdentifier::try_from_envelope(
        request_type.clone(),
        &envelope_pairs[0].update,   // panics if envelope_pairs is empty
    )
``` [1](#0-0) 

The `Request` type is defined as `(RequestType, Vec<EnvelopePair>)`, meaning the vec can legally be empty: [2](#0-1) 

`SignedTransaction::from_str` performs only hex-decode and CBOR deserialization — no invariant enforces that the `Vec<EnvelopePair>` is non-empty: [3](#0-2) 

`is_transfer()` returns `true` for `RequestType::Send`: [4](#0-3) 

The fix exists in `TryFrom<&models::Request> for Request`, which correctly guards with `.first().ok_or_else(...)`: [5](#0-4) 

But `construction_hash` never goes through that conversion — it directly indexes the raw `Vec<EnvelopePair>` from the deserialized `SignedTransaction`. The endpoint is registered with no authentication: [6](#0-5) 

## Impact Explanation
This is a **High** severity DoS. With `panic=unwind` (default), actix-web catches the panic, aborts the worker thread, and restarts it — the in-flight request is dropped with a 500. With `panic=abort` (common in production IC builds for determinism), the entire Rosetta process terminates. In either case, the attacker can loop this indefinitely with a single crafted HTTP POST, causing continuous worker restarts or a persistent crash loop. This denies service to all clients depending on this Rosetta instance for ICP transaction hashing and submission. This matches the allowed impact: **"Significant Rosetta/boundary/API security impact with concrete user or protocol harm"** (High, $2,000–$10,000).

## Likelihood Explanation
The `/construction/hash` endpoint requires no credentials. The CBOR payload is trivially constructable: serialize `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` to CBOR, hex-encode it, and POST as the `signed_transaction` field. No prior knowledge of system state, ledger accounts, or cryptographic material is needed. The attack is repeatable with zero cost per request.

## Recommendation
Add an emptiness check before indexing, mirroring the guard already present in `TryFrom<&models::Request> for Request` at `request.rs:237–239`:

```rust
if envelope_pairs.is_empty() {
    return Err(ApiError::invalid_request(
        "No envelope pairs provided for Send request.",
    ));
}
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pairs[0].update,
)
```

Alternatively, replace `envelope_pairs[0]` with `envelope_pairs.first().ok_or_else(|| ApiError::invalid_request(...))` to return a proper error instead of panicking.

## Proof of Concept
```rust
#[test]
fn test_construction_hash_empty_envelope_pairs_panics() {
    use std::str::FromStr;
    use crate::models::{SignedTransaction, Request};
    use crate::request_types::RequestType;

    // Craft a SignedTransaction with a Send request but empty envelope pairs
    let crafted = SignedTransaction {
        requests: vec![(RequestType::Send, vec![])],
    };
    let hex_cbor = crafted.to_string(); // hex(cbor(crafted))

    // Simulate what construction_hash does after from_str succeeds
    let parsed = SignedTransaction::from_str(&hex_cbor).unwrap();
    let result = parsed
        .requests
        .iter()
        .rev()
        .find(|(rt, _)| rt.is_transfer());

    // This panics at envelope_pairs[0] in the current code
    if let Some((_rt, envelope_pairs)) = result {
        let _ = &envelope_pairs[0]; // index-out-of-bounds panic
    }
}
```

Running `cargo test` with `RUSTFLAGS="-C panic=abort"` crashes the process. Under default settings the panic unwinds and actix-web restarts the worker, but the attacker can loop this indefinitely via repeated HTTP POSTs to `/construction/hash`.
