### Title
Predictable Genesis Random Beacon Enables Deterministic Block Maker Ranking and Committee Selection at Subnet Start - (File: rs/consensus/utils/src/membership.rs)

### Summary
The genesis (and recovery) CatchUpPackage random beacon is constructed with a fully deterministic, empty-bytes signature. Because `get_shuffled_nodes` derives all block-maker rankings and notarization-committee memberships from this beacon, every role assignment at height 1 of any subnet is publicly computable before the subnet produces its first block. A malicious node that pre-computes its rank-0 assignment can exploit this window to manipulate block content or selectively withhold its block proposal.

### Finding Description
The genesis random beacon is built in `make_registry_cup_from_cup_contents` with a hard-coded empty signature:

```rust
let random_beacon = Signed {
    content: RandomBeaconContent {
        version: replica_version,
        height: cup_height,
        parent: Id::from(CryptoHash(Vec::new())),   // fixed zero hash
    },
    signature: ThresholdSignature {
        signer: low_dkg_id,
        signature: CombinedThresholdSigOf::new(CombinedThresholdSig(vec![])), // empty bytes
    },
};
``` [1](#0-0) 

This beacon is stored in the CUP and seeded into the consensus pool as the height-0 beacon. The code that derives block-maker rankings and committee memberships for height 1 uses this beacon directly:

```rust
let mut rng = Csprng::from_random_beacon_and_purpose(previous_beacon, purpose);
node_ids.shuffle(&mut rng);
``` [2](#0-1) 

The code itself documents the consequence:

> "One consequence of this design, is that the randomness derived from the genesis random beacon at height 1 is predictable, because the beacon from the genesis height is preconstructed and known in advance." [3](#0-2) 

Because `Csprng::from_random_beacon_and_purpose` is a pure, deterministic function of the beacon bytes and a domain-separator string, and because the genesis beacon bytes are entirely fixed (`vec![]` signature, `CryptoHash(Vec::new())` parent), any external observer can reproduce the exact ChaCha20 RNG state and compute the full shuffled node list — and therefore every block-maker rank and every notarization-committee membership — for height 1 before the subnet starts. [4](#0-3) 

The same construction is used for recovery CUPs, so the predictability window re-opens after every subnet recovery. [5](#0-4) 

### Impact Explanation
A node operator who pre-computes that their node will hold rank 0 at height 1 can:

1. **Selectively include or exclude ingress messages** in the first data block, since rank-0 nodes propose first and honest nodes will notarize the first valid proposal they see.
2. **Withhold the rank-0 proposal** to force a higher-ranked (potentially colluding) node to propose instead, biasing the first block's content.
3. **Pre-compute the notarization committee** for height 1 and coordinate with colluding committee members to notarize a preferred block.

Because the first DKG interval's execution begins at height 1, any bias in the first block's content (e.g., which canister calls are processed first) can have downstream effects on subnet state. The impact is bounded to height 1 (and post-recovery height), but within that window the randomness provides no unpredictability guarantee.

### Likelihood Explanation
The genesis CUP is published in the NNS registry before the subnet starts. Any party — including a node operator who applied to join the subnet — can download the CUP, extract the beacon bytes, and run the same deterministic shuffle offline. The computation is trivial (a single ChaCha20 key schedule). The only constraint is that the attacker must be an elected subnet member to act as block maker; however, node operators are public participants in the IC ecosystem and the registry lists all node IDs on the subnet, so the full ranking is computable by anyone with registry read access.

### Recommendation
1. **Incorporate unpredictable entropy into the genesis beacon.** For example, include a hash of the NiDKG transcript's combined public key or a commitment from the DKG dealing phase as the beacon's parent field, so the beacon value is not known until after DKG completes.
2. **Alternatively**, accept the current design trade-off but document it explicitly as a security assumption (the first block's proposer is publicly known) and ensure that no security-sensitive state transitions depend on the randomness of height-1 role assignments.

### Proof of Concept
Given a genesis CUP from the registry, an observer can reproduce the height-1 block-maker ranking offline:

```rust
// Pseudocode — all inputs are public registry data
let genesis_beacon: RandomBeacon = cup.content.random_beacon.as_ref().clone();
// genesis_beacon.signature.signature == CombinedThresholdSig(vec![])
// genesis_beacon.content.parent      == CryptoHash(Vec::new())

let randomness = randomness_from_crypto_hashable(&genesis_beacon); // deterministic hash
let seed = Seed::from_bytes(&randomness.get());
let seed_for_purpose = seed.derive("ic-crypto-prng-blockmaker-ranking"); // fixed domain sep
let mut rng = ChaCha20Rng::from_seed(seed_for_purpose.value);

let mut node_ids = registry.get_node_ids_on_subnet(subnet_id, version);
node_ids.sort_unstable(); // deterministic sort
node_ids.shuffle(&mut rng); // deterministic shuffle

// node_ids[0] is the rank-0 block maker at height 1 — known before the subnet starts
```

The `Csprng` and `Seed` implementations confirm this is a pure function of the beacon bytes with no external entropy. [6](#0-5) [7](#0-6)

impl Csprng {
    /// Creates a CSPRNG from the given random beacon for the given purpose.
    pub fn from_random_beacon_and_purpose(
        random_beacon: &RandomBeacon,
        purpose: &RandomnessPurpose,
    ) -> Self {
        let randomness = randomness_from_crypto_hashable(random_beacon);
        Self::from_randomness_and_purpose(&randomness, purpose)
    }

    /// Creates a CSPRNG from the Randomness value for the given purpose.
    pub fn from_randomness_and_purpose(seed: &Randomness, purpose: &RandomnessPurpose) -> Self {
        let seed = Seed::from_bytes(&seed.get());
        let seed_for_purpose = seed.derive(&purpose.domain_separator());
        Csprng::from_seed(seed_for_purpose)
    }

    /// Creates a CSPRNG from the given seed.
    fn from_seed(seed: ic_crypto_internal_seed::Seed) -> Self {
        Csprng {
            rng: seed.into_rng(),
        }
    }

impl Seed {
    fn new(input: &[u8], domain_separator: &str) -> Self {
        let value = xmd::<SEED_LEN, ic_crypto_sha2::Sha256>(input, domain_separator.as_bytes());
        Self { value }
    }

    /// Create a Seed from some input string
    ///
    /// If the Seed is intended to be random the input should be at least 256
    /// bits long.
    pub fn from_bytes(value: &[u8]) -> Self {
        Self::new(value, "ic-crypto-seed-from-bytes")
    }

    /// Create a Seed from a random number generator
    ///
    /// The security of the Seed depends on the security of the RNG
    pub fn from_rng<R: CryptoRng + RngCore>(rng: &mut R) -> Self {
        let mut rng_output = [0_u8; SEED_LEN];
        rng.fill_bytes(&mut rng_output);
        Self::new(&rng_output, "ic-crypto-seed-from-rng")
    }

    /// Derive a new Seed from self
    ///
    /// The domain_separator should be distinct for every derivation
    pub fn derive(&self, domain_separator: &str) -> Self {
        Self::new(&self.value, domain_separator)
    }

    /// Convert a Seed into a random number generator
    ///
    /// The Seed is consumed by this operation
    pub fn into_rng(self) -> rand_chacha::ChaCha20Rng {
        rand_chacha::ChaCha20Rng::from_seed(self.value)
    }
