Audit Report

## Title
Self-Withdrawal Guard Bypass via Bech32 Case Normalization in `retrieve_btc` and `retrieve_btc_with_approval` — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary
The guard preventing withdrawals to the minter's own deposit address in both `retrieve_btc` and `retrieve_btc_with_approval` uses a plain byte-for-byte string equality check against a lowercase bech32 string. Because `bech32::decode` is case-insensitive and `BitcoinAddress::parse` accepts both `'b'` and `'B'` as routing characters, an attacker can supply the minter's address in uppercase or mixed case, bypass the string check, and have ckBTC burned while BTC is sent to the minter's own address — permanently destroying the caller's ckBTC without a corresponding BTC redemption.

## Finding Description
`derive_minter_address_str` (lib.rs L1809–1811) calls `display(state.btc_network)` → `encode_bech32` → `bech32::encode(hrp, data, Variant::Bech32)`, which always produces an all-lowercase string (e.g., `bc1q...`).

The guard at retrieve_btc.rs L158–160 and L256–258 is:
```rust
if args.address == main_address_str {
    ic_cdk::trap("illegal retrieve_btc target");
}
```
This is a byte-for-byte string comparison. Supplying `BC1Q...` (uppercase) makes the comparison false, so the guard does not trap.

Immediately after, `BitcoinAddress::parse` (address.rs L80–84) routes both `'b'` and `'B'` to `parse_bip173_address`. Inside `parse_bip173_address` (address.rs L340–349), `bech32::decode(address)` is case-insensitive, and the HRP check uses `found_hrp.to_lowercase() != expected_hrp`, so `BC1Q...` decodes to the identical `P2wpkhV0(minter_pkhash)` as `bc1q...`. The parsed address equals the minter's own address, but the guard has already been passed.

The ckBTC burn then executes, and a `RetrieveBtcRequest` targeting the minter's own P2WPKH address is enqueued. When the minter's processing logic runs, it sends BTC to its own address. The BTC is absorbed into the minter's UTXO pool, but the caller's ckBTC has been permanently destroyed with no redemption.

## Impact Explanation
This matches **High ($2,000–$10,000): Significant ck-token / ledger security impact with concrete user or protocol harm.** Any ckBTC holder can permanently destroy their own ckBTC balance without receiving BTC in return. The ckBTC supply decreases while the minter's BTC UTXO pool grows unexpectedly, creating an accounting discrepancy. The BTC sent to the minter's own address may also interfere with change-output tracking and reimbursement logic. The loss is irreversible: the ledger burn is final and the BTC is not redeemable by the caller.

## Likelihood Explanation
No privileges are required. The minter's main address is publicly queryable via `get_btc_address`. The transformation (lowercase → uppercase) is trivial. Both `retrieve_btc` and `retrieve_btc_with_approval` are affected. The only prerequisite is holding a nonzero ckBTC balance. The attack is repeatable and deterministic.

## Recommendation
Replace the string comparison with a structural comparison after parsing both addresses:
```rust
let main_address = state::read_state(|s| runtime.derive_minter_address(s));
let parsed_address = BitcoinAddress::parse(&args.address, btc_network)?;
if parsed_address == main_address {
    ic_cdk::trap("illegal retrieve_btc target");
}
```
`BitcoinAddress` implements `Eq`, so this compares decoded enum values rather than string representations, making the check encoding- and case-agnostic. Apply the same fix in both `retrieve_btc` and `retrieve_btc_with_approval`.

## Proof of Concept
1. Call `get_btc_address` (public query) → receive `bc1qabcdef...` (the minter's main address).
2. Convert to uppercase: `BC1QABCDEF...`.
3. Call `retrieve_btc { address: "BC1QABCDEF...", amount: <min_amount> }`.
4. retrieve_btc.rs L158: `"BC1QABCDEF..." == "bc1qabcdef..."` → `false` → guard does **not** trap.
5. retrieve_btc.rs L173: `BitcoinAddress::parse("BC1QABCDEF...", network)` → `Ok(P2wpkhV0(minter_pkhash))` → succeeds.
6. ckBTC burn executes; `RetrieveBtcRequest { address: P2wpkhV0(minter_pkhash), ... }` is enqueued.
7. Minter's processing task sends BTC to its own address; caller's ckBTC is permanently destroyed.

A deterministic unit test can be written against the existing `retrieve_btc` test harness by constructing a `RetrieveBtcArgs` with the minter's address uppercased and asserting that the call does **not** return an error (demonstrating the guard bypass) while also asserting that `parsed_address == minter_address`.
