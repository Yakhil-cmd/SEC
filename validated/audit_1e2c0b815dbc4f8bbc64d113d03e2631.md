### Title
Unauthenticated `execute_scheduled` Allows Any Caller to Prematurely Execute and Permanently Discard Another User's Scheduled XCC Promise - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC router contract has no access control and accepts an attacker-controlled `nonce` parameter. Any NEAR account can trigger execution of another user's scheduled promise at an unintended time, permanently removing it from storage. If the promise fails due to missing preconditions at the time of premature execution, the user's intended cross-contract operation is permanently lost and any associated tokens may be temporarily frozen in the router contract.

---

### Finding Description

The `execute_scheduled` function is explicitly designed to be callable by anyone: [1](#0-0) 

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

Three properties make this dangerous in combination:

1. **No caller check** — unlike `execute` and `schedule`, which call `self.assert_preconditions()` to enforce that only the parent Aurora engine can call them, `execute_scheduled` has no such guard. [2](#0-1) 

2. **Attacker-controlled nonce** — the `nonce: U64` parameter is fully attacker-supplied. Nonces are sequential starting from 0 and are logged on-chain (`"Promise scheduled at nonce {}"`) making them trivially discoverable. [3](#0-2) 

3. **Irreversible removal before execution** — `self.scheduled_promises.remove(&nonce.0)` permanently deletes the promise from storage *before* the promise is dispatched. If the dispatched promise fails (e.g., the target contract rejects the call because required state has not been set up yet), the promise is gone and cannot be retried. [4](#0-3) 

The developers' comment that "there is no security risk" assumes all scheduled promises are safe to execute at any time. This is incorrect: a `CrossContractCallArgs::Delayed` promise is explicitly a *deferred* operation the user intends to execute at a chosen moment. Premature execution can cause the promise to fail (e.g., a DEX swap before an approval is set, a token transfer before storage registration, a governance vote before a proposal is live). Once removed from `scheduled_promises`, the operation is permanently cancelled.

The `schedule` entry-point that populates `scheduled_promises` is correctly gated: [5](#0-4) 

But the corresponding execution path is not.

The router account ID is deterministic (`{hex(address)}.{aurora_engine_account}`), and the nonce is logged publicly, so an attacker can target any user's router contract with zero privileged access. [6](#0-5) 

---

### Impact Explanation

An attacker can:

1. Observe a victim's `schedule` call on-chain (nonce is emitted in a log).
2. Call `execute_scheduled(nonce=<victim_nonce>)` on the victim's router contract before the victim does.
3. The promise is permanently removed from `scheduled_promises` and dispatched immediately.
4. If the promise fails at the target contract (wrong timing, missing registration, etc.), the victim's intended operation is permanently cancelled.
5. Any NEP-141 tokens the router contract holds on behalf of the user (e.g., wNEAR pre-positioned for the XCC call) remain locked in the router contract. Recovery requires the parent Aurora engine to call `send_refund`, which itself requires `require_no_failed_promises` to pass — a condition that may not hold after the forced failure. [7](#0-6) 

This constitutes **High — Temporary (potentially permanent) freezing of funds** and **High — Theft of unclaimed yield** when the cancelled promise was a reward-claim or token-withdrawal operation.

---

### Likelihood Explanation

- No special privileges are required; any NEAR account can call `execute_scheduled` on any router sub-account.
- Router account IDs are deterministic and publicly derivable from the EVM address.
- Nonces are sequential from 0 and are emitted as on-chain logs, making them trivially observable.
- The attack costs only gas; no NEAR deposit is required.
- The attack can be automated to front-run every `schedule` call across the entire protocol.

---

### Recommendation

Add the same `assert_preconditions()` guard used by `execute` and `schedule` to `execute_scheduled`, restricting callers to the parent Aurora engine account:

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    self.assert_preconditions(); // add this line
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

If open execution by third parties is desired for liveness reasons, the removal should only happen *after* confirming the promise succeeded (i.e., in a callback), so a failed execution does not permanently destroy the scheduled entry.

---

### Proof of Concept

1. **Setup**: User A submits an EVM transaction using the XCC precompile with `CrossContractCallArgs::Delayed(promise)`. The engine calls `schedule` on User A's router contract (`{hex(A)}.aurora`), storing the promise at nonce 0 and emitting `"Promise scheduled at nonce 0"`. [8](#0-7) 

2. **Attack**: Attacker observes the log, then calls:
   ```
   {hex(A)}.aurora → execute_scheduled({"nonce": "0"})
   ```
   from any NEAR account, before User A does. [4](#0-3) 

3. **Result**: The promise is removed from `scheduled_promises` and dispatched immediately. If the target contract rejects the call (e.g., User A had not yet completed a prerequisite step), the promise fails. The entry at nonce 0 is gone; User A cannot retry.

4. **Fund freeze**: Any wNEAR or NEP-141 tokens held by the router contract for this operation are now stranded. The `send_refund` recovery path requires the parent engine to call it and `require_no_failed_promises` to pass, which may not hold after the forced failure. [7](#0-6) 

5. **Scale**: The attacker can repeat this for every user's router contract across the entire Aurora deployment, systematically cancelling all pending delayed XCC operations protocol-wide.
