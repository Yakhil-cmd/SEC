### Title
Unbounded Ingress Window Loop in ICP Rosetta `construction_payloads` Causes OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from request metadata and feeds them directly into an unbounded `while now < ingress_end` loop with no window-size validation. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes ~153 billion loop iterations, each pushing a `u64` onto a `Vec`, exhausting process memory and crashing the Rosetta node.

---

### Finding Description

In `construction_payloads`, `ingress_start` and `ingress_end` are read directly from user-supplied metadata with no bounds check: [1](#0-0) 

The loop step size (`interval`) is fixed at compile time: [2](#0-1) 

`interval = MAX_INGRESS_TTL − PERMITTED_DRIFT − 120 s = 300 s − 60 s − 120 s = 120 s = 120,000,000,000 ns`.

The loop then runs without any guard: [3](#0-2) 

With `ingress_start = 0` and `ingress_end = u64::MAX = 18,446,744,073,709,551,615`:

```
iterations ≈ 18_446_744_073_709_551_615 / 120_000_000_000 ≈ 153,722,867,280
memory     ≈ 153 × 10⁹ × 8 bytes ≈ 1.2 TB
```

The process OOMs and is killed long before the loop completes.

The ICRC1 counterpart explicitly rejects this class of input before entering its equivalent loop: [4](#0-3) 

The ICP handler has no analogous guard.

---

### Impact Explanation

Any unauthenticated HTTP client can send a single `POST /construction/payloads` request to crash the Rosetta node process. This denies service to all legitimate clients (exchanges, wallets, tooling) that depend on the node for transaction construction and submission.

---

### Likelihood Explanation

The endpoint is public, requires no credentials, and the malicious payload is a trivial two-field JSON object. The crash is deterministic and reproducible with a single request. No special knowledge beyond the Rosetta API spec is required.

---

### Recommendation

Add window-size validation before the loop, mirroring the ICRC1 implementation:

```rust
if ingress_start >= ingress_end {
    return Err(ApiError::InvalidRequest(...));
}
// Cap the window to a reasonable maximum, e.g. 24 hours
let max_window = Duration::from_secs(86_400);
if ingress_end - ingress_start > max_window.as_nanos() as u64 {
    return Err(ApiError::InvalidRequest(...));
}
```
