### Title
Missing Refund on Failed ERC-20 Exit Permanently Freezes User Funds — (`engine-precompiles/src/native.rs` / `engine/src/contract_methods/connector.rs`)

### Summary

When the `error_refund` compile feature is absent (the default production build), the `ExitToNear` precompile hardcodes `refund: None` in the callback arguments. If the downstream `ft_transfer` / `near_withdraw` NEAR promise fails, `exit_to_near_precompile_callback` reaches an `else { None }` branch that performs no recovery. The ERC-20 tokens were already burned inside the EVM, so the user's funds are permanently frozen with no on-chain path to recover them.

---

### Finding Description

**Step 1 – Tokens are burned before the cross-chain promise is sent.**

In `ExitToNear::run` (`engine-precompiles/src/native.rs`), the precompile is called from inside the ERC-20 `withdrawToNear` function, which burns the caller's tokens before the precompile returns. The precompile then schedules a NEAR `ft_transfer` (or `near_withdraw` for wNEAR) promise.

**Step 2 – The callback args carry `refund: None` in the default build.** [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in the default build
    transfer_near: transfer_near_args,
};
```

`error_refund` is not listed in the `default` feature set of either `aurora-engine` or `aurora-engine-precompiles`. [2](#0-1) [3](#0-2) 

**Step 3 – The callback silently drops the failure.** [4](#0-3) 

```rust
let maybe_result =
    if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
        // success path – optionally transfer unwrapped NEAR
        ...
        None
    } else if let Some(args) = args.refund {   // ← None in default build
        let refund_result = engine::refund_on_error(...)?;
        ...
    } else {
        None   // ← always reached on failure; no refund, no error
    };
```

Because `args.refund` is `None`, the `else if` arm is never entered. The callback returns `Ok(None)`, the transaction succeeds from NEAR's perspective, and the burned ERC-20 tokens are gone forever.

**Step 4 – The wNEAR unwrap path is equally affected.**

For the wNEAR `:unwrap` flow, `transfer_near` is `Some`, so a callback *is* always scheduled (even without `error_refund`). If `near_withdraw` fails, the same `else { None }` branch is reached and the burned wNEAR ERC-20 tokens are unrecoverable. [5](#0-4) 

---

### Impact Explanation

Any user who calls `withdrawToNear` (ERC-20 exit) or the wNEAR unwrap path and whose downstream NEAR promise fails will have their ERC-20 tokens permanently burned with no NEP-141 tokens received and no on-chain refund path. This is a **permanent freezing of funds** (Critical).

The existing test suite explicitly acknowledges this outcome: [6](#0-5) 

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

---

### Likelihood Explanation

The failure condition is reachable by any unprivileged EVM user:

- Sending `withdrawToNear` with a NEAR recipient that is not registered with the NEP-141 contract causes `ft_transfer` to fail.
- Sending the wNEAR `:unwrap` suffix when the wNEAR contract's `near_withdraw` reverts (e.g., insufficient balance edge cases, contract paused) causes the same outcome.

No admin action or special privilege is required. The attacker-controlled entry path is the standard ERC-20 `withdrawToNear` call available to every token holder.

---

### Recommendation

Enable the `error_refund` feature in the production WASM build, or unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` so that `exit_to_near_precompile_callback` always has the information needed to re-mint burned ERC-20 tokens (or return ETH) when the downstream NEAR promise fails. [7](#0-6) 

---

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to an ERC-20 on Aurora.
2. As a token holder, call `withdrawToNear("unregistered.near", amount)` on the ERC-20 (flag `0x01`).
3. The ERC-20 contract burns `amount` tokens and calls the `ExitToNear` precompile.
4. The precompile schedules `ft_transfer` to `unregistered.near`; because that account is not registered, the promise fails.
5. `exit_to_near_precompile_callback` is invoked (or skipped entirely for non-wNEAR paths); `args.refund` is `None`; the `else { None }` branch executes.
6. The transaction returns success. The user's ERC-20 balance is zero; the NEP-141 balance of `unregistered.near` is zero; the NEP-141 balance of Aurora is unchanged. Funds are permanently frozen. [8](#0-7) [9](#0-8)

#[cfg(feature = "error_refund")]
#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}
