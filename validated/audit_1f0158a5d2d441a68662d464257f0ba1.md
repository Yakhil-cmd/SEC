### Title
Hardcoded `L1_TX_NATIVE_PRICE = 10` Is an Unvalidated Placeholder That Miscalibrates Native Resource Budgets for L1→L2 Transactions — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded to `10` with an explicit `TODO` acknowledging it is not a calibrated value. This constant is the sole divisor used to convert an L1→L2 transaction's gas price into its native (proving) resource budget. Because L1 transactions cannot be invalidated, a miscalibrated `L1_TX_NATIVE_PRICE` that is too low allows any unprivileged L1 sender to obtain a disproportionately large native resource budget per transaction at minimal cost, consuming the block's proving capacity without paying the corresponding fee.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant governing the native resource price for all L1→L2 transactions is:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

This constant is used in `prepare_and_check_resources` inside `process_l1_transaction.rs` to compute `nativePerGas` for every priority L1 transaction:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The resulting `nativePerGas` is then multiplied by `gasLimit` to produce `native_prepaid_from_gas`, which becomes the transaction's native resource budget (capped at `MAX_NATIVE_COMPUTATIONAL = 2^35`):

```rust
let native_prepaid_from_gas = native_per_gas.checked_mul(gas_limit)...;
``` [3](#0-2) 

The cap itself is defined as:

```rust
pub const MAX_NATIVE_COMPUTATIONAL: u64 = 1 << 35;  // ~34 billion
``` [4](#0-3) 

For L2 transactions, the operator dynamically sets `native_price` to reflect actual proving costs. For L1 transactions, the operator has **no ability to adjust** this constant — it is baked into the binary. The docs confirm this design choice:

> "Note: for L1→L2 transactions we use a code constant instead of one provided by operator." [5](#0-4) 

---

### Impact Explanation

**Vulnerability class: Resource accounting bug (hardcoded value miscalibration)**

With `L1_TX_NATIVE_PRICE = 10`:

- An attacker submitting an L1 transaction with `gas_price = 10` gets `nativePerGas = ceil(10/10) = 1`.
- With `gasLimit` set to any large value, `native_prepaid_from_gas = gasLimit * 1`.
- The budget is capped at `MAX_NATIVE_COMPUTATIONAL = 2^35 ≈ 34 billion` native units.
- The attacker's on-chain L1 cost is `gas_price * gas_used` — a minimal ETH amount.

The native resource represents RISC-V proving cycles. Consuming `2^35` native units per transaction means the attacker forces the prover to perform the maximum allowed proving work per block at a cost calibrated against a placeholder constant that the codebase itself acknowledges is wrong. The block-level native limit check (`computational_native_used > MAX_NATIVE_COMPUTATIONAL`) will trigger `BlockNativeLimitReached` for subsequent transactions after the attacker's transaction exhausts the budget: [6](#0-5) 

Since L1 transactions cannot be invalidated (doing so would halt the chain), the bootloader must process them regardless:

> "L1 transactions cannot be invalidated. Therefore, the following function makes sure L1 transactions are processable even when some checks that should be performed by the L1 don't hold." [7](#0-6) 

**Impact**: An unprivileged attacker can cheaply exhaust the block's native (proving) resource budget via L1→L2 transactions, causing legitimate L2 transactions to be excluded from the block with `BlockNativeLimitReached`, and forcing the prover to perform maximum-cost proving work without adequate fee compensation.

---

### Likelihood Explanation

- **Entry path is fully permissionless**: any Ethereum address can submit a priority transaction to the L1 mailbox contract.
- **No privileged access required**: the attacker only needs to pay the L1 gas cost of submitting the transaction.
- **The miscalibration is self-documented**: the `TODO (EVM-1157)` comment confirms the value `10` is a placeholder, not a calibrated figure. The operator cannot correct it at runtime.
- **Reproducible**: tests in the codebase already hard-code `native_price = 10` as the known L1 constant, confirming it is the live production value. [8](#0-7) 

---

### Recommendation

1. **Calibrate `L1_TX_NATIVE_PRICE`** against the actual RISC-V proving cost per native unit before production deployment. Resolve `TODO (EVM-1157)`.
2. **Add a minimum gas price check for L1 transactions** on the L1 side (in the mailbox contract) that enforces `gas_price >= L1_TX_NATIVE_PRICE * minimum_native_per_gas`, analogous to how L2 transactions enforce `base_fee <= max_fee_per_gas`.
3. Consider making `L1_TX_NATIVE_PRICE` operator-configurable via a governance-controlled parameter rather than a compile-time constant, so it can be updated as proving costs change without a binary upgrade.

---

### Proof of Concept

1. Attacker submits an L1→L2 priority transaction with:
   - `gas_price = 10` (equals `L1_TX_NATIVE_PRICE`)
   - `gas_limit = MAX_BLOCK_GAS_LIMIT`
   - Calldata designed to consume maximum native resources (e.g., repeated SSTORE/SLOAD operations)

2. In `prepare_and_check_resources`:
   - `nativePerGas = ceil(10 / 10) = 1`
   - `native_prepaid_from_gas = MAX_BLOCK_GAS_LIMIT * 1`
   - After capping: `native_limit = MAX_NATIVE_COMPUTATIONAL = 2^35`

3. The transaction executes with a native budget of `2^35`, consuming the entire block's native resource allowance.

4. All subsequent L2 transactions in the same block receive `BlockNativeLimitReached` and are excluded.

5. The prover must prove a block with `2^35` native cycles of work. The attacker's L1 cost was `10 * actual_gas_used` — a trivially small amount relative to the proving cost imposed. [1](#0-0) [4](#0-3) [9](#0-8)
