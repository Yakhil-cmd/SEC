### Title
Predictable XCC Router Account ID Allows Attacker to Pre-Deploy Router with Malicious `wnear_account`, Blocking Victim's Cross-Contract Calls - (File: engine/src/xcc.rs)

### Summary

The `fund_xcc_sub_account` function in `engine/src/xcc.rs` is callable by any unprivileged NEAR account and accepts a caller-supplied `wnear_account_id`. Because the XCC router's NEAR sub-account ID is fully deterministic (`{evm_address_hex}.{aurora_engine_account}`), an attacker can front-run a victim's first XCC interaction by calling `fund_xcc_sub_account` for the victim's EVM address with an attacker-controlled `wnear_account_id`. This deploys the router and records the version in the engine's storage, so the engine subsequently treats the router as up-to-date and never re-deploys it. The router is initialized with the wrong wNEAR account, causing all subsequent XCC calls for that address that require wNEAR unwrapping to fail.

### Finding Description

The XCC router sub-account ID for any EVM address is constructed deterministically: [1](#0-0) 

This produces `{evm_address_hex}.aurora`, which is fully predictable from the victim's public EVM address.

`fund_xcc_sub_account` is a public NEAR method with no caller-identity check. It accepts an optional `wnear_account_id` from the caller: [2](#0-1) 

When `create_needed` is true (first-time deployment), the function issues a NEAR batch that creates the account, deploys the router code, and calls `initialize` with the attacker-supplied `wnear_account`: [3](#0-2) 

After the batch succeeds, a `factory_update_address_version` callback records the latest code version for the victim's address in the engine's storage: [4](#0-3) 

From this point on, `AddressVersionStatus::new` returns `UpToDate` for the victim's address: [5](#0-4) 

The engine therefore skips re-deployment and calls the already-deployed (poisoned) router directly. The router's `wnear_account` field is set to the attacker's account, not the real wNEAR contract. Because the real wNEAR contract never received a `storage_deposit` for the router, the router is not registered with wNEAR. Any XCC call that requires wNEAR unwrapping (i.e., the `withdraw_wnear_to_router` step) will fail because the wNEAR contract rejects transfers to unregistered accounts.

The XCC router's `initialize` function in the router contract sets `wnear_account` from the argument passed by the engine: [6](#0-5) 

### Impact Explanation

Any EVM address that has not yet used XCC can have its router pre-deployed with a wrong wNEAR account. All subsequent XCC calls for that address that involve wNEAR unwrapping will revert. The victim's wNEAR remains locked in the EVM state (not directly stolen), but the XCC mechanism is non-functional for that address until an engine-level router upgrade is performed. This constitutes a **temporary freezing of funds** in motion (wNEAR intended for cross-contract calls cannot be unwrapped or forwarded).

### Likelihood Explanation

The attack requires only:
1. Knowledge of the victim's EVM address (public information).
2. Calling `fund_xcc_sub_account` with the victim's address and an attacker-controlled `wnear_account_id`, attaching ≥ 2 NEAR (`STORAGE_AMOUNT`).

No special privileges are required. The attacker can monitor the mempool or simply target any address that has not yet used XCC. The cost to the attacker is the 2 NEAR deposit (which is transferred to the victim's router sub-account, not lost by the attacker). This is a low-cost, high-impact griefing attack.

### Recommendation

- **Short term:** Add an access control check in `fund_xcc_sub_account` so that only the owner of the target EVM address (i.e., the NEAR account whose predecessor maps to the EVM address) or the engine owner can call it for a given target. Alternatively, ignore the caller-supplied `wnear_account_id` entirely and always derive it from the engine's internal registry.
- **Long term:** When `wnear_account_id` is supplied externally, validate it against the engine's canonical wNEAR address registry before using it in the `initialize` call.

### Proof of Concept

1. Victim has EVM address `0xAAAA...AAAA` and has never used XCC.
2. Attacker calls `fund_xcc_sub_account` on the Aurora engine with:
   - `target = 0xAAAA...AAAA`
   - `wnear_account_id = Some("attacker.near")`
   - Attached deposit ≥ 2 NEAR
3. The engine creates `aaaa...aaaa.aurora`, deploys the router, and calls `initialize(wnear_account = "attacker.near", must_register = true)`.
4. The router calls `storage_deposit` on `attacker.near` (not the real wNEAR contract). The real wNEAR contract has no registration for `aaaa...aaaa.aurora`.
5. `factory_update_address_version` records the current version for `0xAAAA...AAAA` in the engine's storage.
6. Victim submits an EVM transaction that triggers the XCC precompile. `AddressVersionStatus::new` returns `UpToDate`, so no re-deployment occurs.
7. The engine calls `withdraw_wnear_to_router` to send wNEAR to the router. The wNEAR contract rejects the transfer because `aaaa...aaaa.aurora` is not registered.
8. The victim's XCC call fails. The victim's wNEAR remains in the EVM but cannot be used for cross-contract calls until the router is re-initialized via an engine upgrade.

impl AddressVersionStatus {
    fn new<I: IO>(
        io: &I,
        latest_code_version: CodeVersion,
        target_code_version: Option<CodeVersion>,
    ) -> Self {
        let first_upgradable_version =
            get_first_upgradable_version(io).unwrap_or(CodeVersion::ZERO);
        match target_code_version {
            None => Self::DeployNeeded {
                create_needed: true,
            },
            Some(version) if version < first_upgradable_version => {
                // It is impossible to upgrade the initial XCC routers because
                // they lack the upgrade method.
                Self::UpToDate
            }
            Some(version) if version < latest_code_version => Self::DeployNeeded {
                create_needed: false,
            },
            Some(_version) => Self::UpToDate,
        }
    }
