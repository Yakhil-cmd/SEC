Audit Report

## Title
Provider-Level Rate Limit Starvation: Sibling Node Operator Can Exhaust Shared Provider Bucket, Blocking All Co-Provider Operators — (`rs/registry/canister/src/rate_limits.rs`)

## Summary

`NODE_PROVIDER_MAX_SPIKE` and `NODE_OPERATOR_MAX_SPIKE` are both set to `20 * 7 = 140`. Because `try_reserve_capacity_for_node_operator_operation` debits both the caller's per-operator bucket and the shared per-provider bucket on every committed operation, a single operator O1 can fully drain the provider bucket within their own operator budget. After 140 committed operations, every sibling operator O2 under the same provider fails at the provider-bucket reservation step, blocking all four node-management entry points (`add_node`, `remove_node_directly`, `update_node_domain_directly`, `update_node_ipv4_config_directly`) for at least one refill interval (72 minutes), and indefinitely if O1 re-exhausts the one unit that refills each interval.

## Finding Description

**Constants** — both buckets share the same `max_capacity`: [1](#0-0) 

**Bucket configuration** — provider bucket `max_capacity = NODE_PROVIDER_MAX_SPIKE = 140`: [2](#0-1) 

**Double debit** — every call to `try_reserve_capacity_for_node_operator_operation` reserves from both the per-operator bucket and the shared per-provider bucket: [3](#0-2) 

**Commit path** — `do_update_node_ipv4_config_directly_` calls `try_reserve` before the mutation and `commit_used_capacity_for_node_operator_operation` after, so every successful call permanently consumes one unit from both buckets: [4](#0-3) 

**No-op clear path** — when `ipv4_config: None`, `validate_update_node_ipv4_config_directly_payload` skips the uniqueness check and only asserts the node exists, so O1 can call this 140 times on a single owned node without any validation failure: [5](#0-4) 

**`get_available_capacity` logic** — after 140 committed operations by O1, `committed_capacity` for the provider key equals `max_capacity = 140`, so `get_available_capacity` returns 0 for any key under that provider: [6](#0-5) 

**`try_reserve` check** — O2's reservation fails with `RateLimiterError::NotEnoughCapacity` at the provider-bucket step, and the already-acquired operator-bucket reservation is dropped (not committed) via the `Drop` impl: [7](#0-6) [8](#0-7) 

**Missing test coverage** — `test_combined_rate_limiting` only tests partial depletion (5 and 3 units) and never asserts that full exhaustion by O1 blocks O2: [9](#0-8) 

## Impact Explanation

All four node-management operations route through `try_reserve_capacity_for_node_operator_operation` and fail at the provider-bucket step once it is exhausted. O2 is completely unable to add nodes, remove nodes, update node domain, or update node IPv4 config for the duration of the lockout. With continuous re-exhaustion (one call per 72-minute refill interval), the lockout is indefinite. This is an application/platform-level DoS on the registry canister's node-management surface — a High-severity impact under the ICP bounty criteria ("Application/platform-level DoS … not based on raw volumetric DDoS").

## Likelihood Explanation

The attacker must be a registered node operator under the same provider as the victim — a realistic relationship in the IC ecosystem where one provider can have multiple operators. No privileged access, governance majority, or key material is required. The initial attack costs 140 ingress calls to a public endpoint; maintenance costs one call per 72 minutes. The attack is fully repeatable and requires no victim interaction.

## Recommendation

Size the provider-level bucket to accommodate all operators under it fairly. Concrete options:

1. **Increase `NODE_PROVIDER_MAX_SPIKE`** to `N × NODE_OPERATOR_MAX_SPIKE` where N is the expected maximum number of operators per provider, so no single operator can exhaust the full provider budget.
2. **Add a per-operator cap on provider-bucket consumption** — after reserving from the operator bucket, limit the provider-bucket debit to `provider_max / max_operators_per_provider`.
3. **Add a cross-operator starvation regression test** that sets up two operators under one provider, exhausts the provider bucket via O1 with committed operations, and asserts O2's subsequent call returns a rate-limit error.

## Proof of Concept

```
State: O1 and O2 both registered under provider P.
       Provider bucket: 140/140. O1 operator bucket: 140/140. O2 operator bucket: 140/140.

Step 1: O1 calls do_update_node_ipv4_config_directly_(node_id=N1, ipv4_config=None) × 140
        → each call: operator[O1] -= 1 (committed), provider[P] -= 1 (committed)
        → after 140 calls: operator[O1].capacity_used = 140, provider[P].capacity_used = 140

Step 2: O2 calls do_update_node_ipv4_config_directly_(node_id=N2, ipv4_config=None)
        → try_reserve: operator[O2] reserved (succeeds, 139 available)
        → try_reserve: provider[P] → get_available_capacity returns 0 → NotEnoughCapacity
        → operator reservation dropped (not committed)
        → returns Err("Rate Limit Capacity exceeded. Please wait and try again later.")

Step 3: O1 waits 72 minutes (one refill interval), provider[P].capacity_used decrements to 139.
        O1 calls once more → provider[P].capacity_used = 140 again.
        O2 remains blocked indefinitely.
```

Minimal unit test to prove it (extend `rs/registry/canister/src/rate_limits.rs` tests):
- Register O1 and O2 under shared provider P.
- Loop 140 times: call `try_reserve_capacity_for_node_operator_operation(now, O1, 1)` and immediately `commit_used_capacity_for_node_operator_operation`.
- Assert `try_reserve_capacity_for_node_operator_operation(now, O2, 1)` returns `Err(RateLimiterError::NotEnoughCapacity)`.

    pub fn get_available_capacity(&mut self, key: K, now: SystemTime) -> u64 {
        let committed_capacity =
            self.with_capacity_usage_record(key.clone(), now, |usage| usage.capacity_used);

        let reservations = self.reservations.lock().unwrap();

        // Get all reservations for this key to calculate current usage
        let reserved_capacity: u64 = reservations
            .range((key.clone(), 0)..=(key.clone(), u64::MAX))
            .map(|(_, data)| data.capacity)
            .sum();

        self.config
            .max_capacity
            .saturating_sub(reserved_capacity)
            .saturating_sub(committed_capacity)
    }

impl<K: Clone + Ord> Drop for Reservation<K> {
    fn drop(&mut self) {
        #[allow(clippy::collapsible_if)]
        if let Some(reservations_arc) = self.reservations_map.upgrade() {
            if let Ok(mut reservations) = reservations_arc.lock() {
                reservations.remove(&(self.key.clone(), self.index));
            }
        }
    }
}
