Audit Report

## Title
Blocklist Enforcement Bypassed via Proxy Contract ‚Äî `msg.sender` in Helper Reflects Proxy, Not Sanctioned Originator - (`rs/ethereum/cketh/minter/src/deposit.rs`, `rs/ethereum/cketh/minter/src/blocklist.rs`)

## Summary
The ckETH/ckERC20 minter's blocklist check in `register_deposit_events` operates exclusively on `event.from_address()`, which is derived from the `owner` field (`msg.sender`) emitted by the helper contract. When a sanctioned address routes a deposit through any unlisted proxy contract, `msg.sender` inside the helper resolves to the proxy address rather than the sanctioned originator. Because the proxy is not on `ETH_ADDRESS_BLOCKLIST`, `is_blocked` returns `false` and minting proceeds, violating the stated invariant that blocked addresses cannot deposit.

## Finding Description
**Root cause:** `DepositHelperWithSubaccount.sol` unconditionally emits `msg.sender` as the `owner` field in both `depositEth` (line 504) and `depositErc20` (line 527). All three Rust log parsers (`ReceivedEthLogParser`, `ReceivedErc20LogParser`, `ReceivedEthOrErc20LogParser`) extract `from_address` from this field ‚Äî `topics[1]` for ETH events and `topics[2]` for ERC20/combined events. `register_deposit_events` in `deposit.rs` (line 323) calls `crate::blocklist::is_blocked(&event.from_address())` and only this address. `is_blocked` performs a binary search on the static `ETH_ADDRESS_BLOCKLIST` array. There is no check on `tx.origin` or any other field that would identify the ultimate transaction signer. A proxy contract address not present in the list returns `false`, and `event.into_deposit()` is called, triggering minting.

**Exploit flow:**
1. Sanctioned address `S` (present in `ETH_ADDRESS_BLOCKLIST`) deploys proxy contract `P` (not on the list).
2. `S` calls `P.forwardDeposit{value: V}(helperAddress, encodedPrincipal, subaccount)`.
3. `P` calls `CkDeposit(helper).depositEth{value: V}(encodedPrincipal, subaccount)` ‚Äî inside the helper, `msg.sender == address(P)`.
4. Helper emits `ReceivedEthOrErc20(address(0), address(P), V, encodedPrincipal, subaccount)`.
5. Minter scrapes the log; parser sets `from_address = address(P)`.
6. `is_blocked(address(P))` ‚Üí `false`; minting proceeds to `S`'s IC principal.

**Existing guards are insufficient:** The blocklist check at `deposit.rs:323` is the sole enforcement point. No secondary check on `tx.origin`, transaction sender, or any other field exists anywhere in the ckETH codebase.

## Impact Explanation
This is a **High** severity finding. It constitutes a significant ck-token security impact with concrete protocol harm: the protocol's stated sanctions-enforcement invariant ("ETH is not accepted from nor sent to addresses on this list") is fully defeated. Sanctioned entities can freely receive ckETH and ckERC20 tokens, exposing the ICP/DFINITY ecosystem to regulatory and legal consequences and undermining the integrity of the Chain Fusion financial integration. This maps to the allowed impact: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."*

## Likelihood Explanation
The attack requires no privileged access, no key compromise, and no victim interaction. Deploying a forwarding proxy on Ethereum costs a single transaction and a small amount of gas. The technique is well-established and widely understood. The sanctioned entity does not need to interact with the IC directly; the IC principal encoded in the deposit can be any account they control. The attack is repeatable indefinitely.

## Recommendation
1. **Emit and check `tx.origin`:** Modify `depositEth` and `depositErc20` in `DepositHelperWithSubaccount.sol` to emit `tx.origin` as an additional indexed field. Update the Rust log parsers to extract this field and have `register_deposit_events` call `is_blocked` on both `from_address` (the `msg.sender` proxy) and the `tx.origin` field. Block the deposit if either is on the list.
2. **New helper deployment required:** A new helper contract must be deployed and the minter upgraded to parse the additional event field.
3. **Interim documentation:** Until the fix is deployed, document explicitly that proxy-based bypasses are a known limitation.

## Proof of Concept
```solidity
// Proxy contract (not on blocklist)
contract Proxy {
    function forwardDeposit(
        address helper,
        bytes32 principal,
        bytes32 subaccount
    ) external payable {
        CkDeposit(helper).depositEth{value: msg.value}(principal, subaccount);
    }
}
```
1. Deploy `Proxy` from `SAMPLE_BLOCKED_ADDRESS` (`0x0330070FD38Ec3bB94F58FA55D40368271E9e54A`).
2. Call `proxy.forwardDeposit{value: 1 ether}(helperAddress, encodedPrincipal, bytes32(0))` from `SAMPLE_BLOCKED_ADDRESS`.
3. Helper emits `ReceivedEthOrErc20(address(0), proxy_address, 1 ether, encodedPrincipal, bytes32(0))`.
4. Minter parses log: `from_address = proxy_address`.
5. Assert: `is_blocked(SAMPLE_BLOCKED_ADDRESS) == true`, `is_blocked(proxy_address) == false`.
6. `register_deposit_events` calls `event.into_deposit()` ‚Üí ckETH minted to sanctioned entity's IC principal.

A deterministic integration test can be written using a local Ethereum fork: deploy the helper and proxy, call `forwardDeposit` from `SAMPLE_BLOCKED_ADDRESS`, scrape the log through the minter's `ReceivedEthOrErc20LogParser`, and assert that `is_blocked` on the parsed `from_address` returns `false` while `is_blocked(SAMPLE_BLOCKED_ADDRESS)` returns `true`.
