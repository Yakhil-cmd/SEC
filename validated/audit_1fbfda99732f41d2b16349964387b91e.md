### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Silently Overcharges Users When Native Resource Consumption Is Below EVM Gas Consumption - (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the native-vs-gas reconciliation only adjusts `gas_used` **upward** when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`). When the opposite is true (`delta_gas < 0`, i.e., native consumption is less than EVM gas consumption), `gas_used` is never reduced. The user is therefore charged for more gas than native resources warrant, and the excess tokens are never returned. A `TODO` comment in the code explicitly acknowledges this gap.

---

### Finding Description

In `compute_gas_refund` (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`), after computing EVM `gas_used` and `native_used`, the code reconciles the two resource dimensions:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native consumption > gas consumption → charge more gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

When `delta_gas < 0` (native resource consumption is less than EVM gas consumption), the branch is skipped entirely. `gas_used` remains at the full EVM gas figure. The refund calculation in `refund_and_commit_fee` then computes:

```
token_to_refund = gas_price * (tx_gas_limit - gas_used)
``` [2](#0-1) 

Because `gas_used` was not reduced by `|delta_gas|`, the user receives `gas_price * |delta_gas|` fewer tokens in their refund than they are owed. The excess is silently transferred to the operator (or partially burned when the `burn_base_fee` feature is active).

The same `compute_gas_refund` function is called for both ZK L2 transactions and Ethereum-type L2 transactions: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

For every transaction where `native_used / native_per_gas < gas_used` (i.e., the transaction is compute-heavy but generates little pubdata), the user is overcharged by:

```
overcharge = gas_price * (gas_used - native_used / native_per_gas)
```

These tokens are deducted from the user's pre-charged balance but never returned. When `burn_base_fee` is enabled, the portion `|delta_gas| * base_fee` is permanently burned (truly lost), while `|delta_gas| * priority_fee` is incorrectly paid to the operator. When `burn_base_fee` is disabled, the full `|delta_gas| * gas_price` is incorrectly paid to the operator.

**Impact: High** — user funds are permanently lost or incorrectly redistributed on every qualifying transaction.

---

### Likelihood Explanation

The condition `delta_gas < 0` arises whenever a transaction consumes more EVM gas than native resources warrant. This is a common scenario for:
- Compute-heavy transactions with minimal storage writes (low pubdata)
- Transactions where `native_per_gas` is set relatively high (e.g., when `native_price` is low relative to `gas_price`)

The `TODO` comment at line 80 confirms the developers are aware this case is unhandled. Any unprivileged user submitting a standard ZK or Ethereum-type L2 transaction can trigger this path without any special access.

**Likelihood: Medium** — occurs in normal operation for a well-defined class of transactions.

---

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` to reflect the lower native resource consumption, symmetrically with the upward adjustment:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // Native consumption is less than gas consumption; refund the difference
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Ensure gas_used does not fall below minimal_gas_used (already enforced above)
}
```

This ensures the user is only charged for the resource dimension that was actually more expensive, symmetrically in both directions.

---

### Proof of Concept

1. Submit a ZK L2 transaction with:
   - `gas_limit = 100_000`
   - `gas_price = 1000`
   - Transaction body: compute-heavy EVM execution (e.g., many arithmetic ops) with zero storage writes → `native_used` is small, `gas_used` is large
   - Block context: `native_price` set such that `native_per_gas` is, say, 1

2. Suppose after execution: `gas_used = 80_000`, `native_used = 40_000`, `native_per_gas = 1`.
   - `delta_gas = (40_000 / 1) - 80_000 = -40_000` → branch skipped, `gas_used` stays at `80_000`
   - Refund = `1000 * (100_000 - 80_000)` = `20_000_000` tokens
   - Correct refund should be = `1000 * (100_000 - 40_000)` = `60_000_000` tokens
   - **User loses `40_000_000` tokens** that are incorrectly paid to the operator

The `TODO` comment at line 80 of `refund_calculation.rs` is a direct acknowledgment of this unimplemented case. [5](#0-4)
