Audit Report

## Title
Unprivileged Caller Can Exhaust the Global `pending_retrieve_btc_requests` Queue, Causing Temporary DoS of ckBTC Withdrawals - (File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs)

## Summary
The ckBTC minter enforces a global cap of 5,000 incomplete withdrawal requests (`MAX_CONCURRENT_PENDING_REQUESTS`) but imposes no per-account limit on how many entries a single principal may accumulate in `pending_retrieve_btc_requests`. An attacker holding sufficient ckBTC can sequentially (or in parallel via subaccounts) submit minimum-amount withdrawal requests until the global cap is reached, after which every legitimate user's withdrawal attempt returns `TemporarilyUnavailable`. The attacker recovers their BTC minus fees, making the attack economically repeatable.

## Finding Description
`MAX_CONCURRENT_PENDING_REQUESTS` is set to 5,000 at the module level. [1](#0-0) 

Both `retrieve_btc` and `retrieve_btc_with_approval` check the global queue size before accepting a new request: [2](#0-1) [3](#0-2) 

The `retrieve_btc_guard` (backed by `retrieve_btc_accounts: BTreeSet<Account>`) only prevents a single account from having more than one concurrent in-flight async call, with a global ceiling of `MAX_CONCURRENT = 100` simultaneous in-flight calls. [4](#0-3) 

The guard is dropped as soon as the async function returns (via `Drop`), immediately making the same account eligible to submit another request. [5](#0-4) 

`accept_retrieve_btc_request` unconditionally pushes to `pending_retrieve_btc_requests` with no per-account check: [6](#0-5) 

`retrieve_btc_account_to_block_indices` records which block indices belong to each account for status queries only and is never consulted to limit how many pending requests an account may hold. [7](#0-6) 

`count_incomplete_retrieve_btc_requests()` sums pending + in-flight + submitted requests across all accounts with no per-account breakdown: [8](#0-7) 

**Exploit flow:**
1. Attacker calls `retrieve_btc_with_approval` with up to 100 distinct subaccounts in parallel (bypassing the per-account guard), each with `amount = fee_based_retrieve_btc_min_amount`.
2. Each call passes the guard check (distinct subaccount), passes the queue-size check (count < 5,000), burns ckBTC, and appends a `RetrieveBtcRequest` to `pending_retrieve_btc_requests`.
3. After 5,000 accepted requests, `count_incomplete_retrieve_btc_requests() == 5000`.
4. Every subsequent legitimate withdrawal call hits the cap check and receives `TemporarilyUnavailable("too many pending retrieve_btc requests")`.
5. The minter's timer processes the attacker's requests via `build_batch`, eventually draining the queue. The attacker receives BTC and can repeat. [9](#0-8) 

## Impact Explanation
Once the global queue reaches 5,000 entries, all ckBTC withdrawal functionality is unavailable to every user until the minter's timer loop drains the queue. This is a concrete application/platform-level DoS of a core ckBTC financial integration component, matching the allowed High impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The DoS is not raw volumetric — it exploits a logic flaw (absent per-account limit) to exhaust a bounded global resource using valid, processed transactions. [2](#0-1) 

## Likelihood Explanation
The minimum withdrawal amount is `fee_based_retrieve_btc_min_amount` (~100,000 satoshis on mainnet). Filling the queue requires 5,000 × 0.001 BTC = 5 BTC of working capital. The attacker recovers the principal; the only sunk cost is the minter fee (~305 satoshis/request) plus Bitcoin transaction fees — on the order of a few hundred USD total to sustain the attack indefinitely. No privileged access, leaked keys, or governance majority is required. Using `retrieve_btc_with_approval` with up to 100 distinct subaccounts allows parallelizing the flood to 100 concurrent requests per round, significantly accelerating queue saturation. [10](#0-9) [11](#0-10) 

## Recommendation
Introduce a per-account cap on the number of accepted (pending + in-flight + submitted) withdrawal requests. Before accepting a new request, check how many entries the calling account already has in `retrieve_btc_account_to_block_indices` and reject with `TemporarilyUnavailable` if the count exceeds a reasonable threshold (e.g., 10). This preserves the global cap as a secondary safety valve while preventing any single account from monopolising the queue. [7](#0-6) 

## Proof of Concept
1. Acquire 5 BTC worth of ckBTC across one principal using 100 subaccounts (50,000 satoshis each, or distribute as needed).
2. In a local PocketIC or replica test, call `retrieve_btc_with_approval` 100 times in parallel using distinct subaccounts (each call passes the guard since accounts differ), each with `amount = fee_based_retrieve_btc_min_amount` and a valid Bitcoin address.
3. Repeat in batches of 100 until 5,000 requests are accepted (50 rounds).
4. Assert `count_incomplete_retrieve_btc_requests() == 5000`.
5. Call `retrieve_btc` or `retrieve_btc_with_approval` from a fresh, unrelated principal with sufficient ckBTC balance.
6. Assert the call returns `Err(TemporarilyUnavailable("too many pending retrieve_btc requests"))`.
7. Confirm the minter's timer eventually drains the queue and the attacker receives BTC at their addresses, demonstrating repeatability. [12](#0-11)

impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }

pub fn accept_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    runtime: &R,
) {
    record_event(
        EventType::AcceptedRetrieveBtcRequest(request.clone()),
        runtime,
    );
    state.pending_retrieve_btc_requests.push(request.clone());
    if let Some(account) = request.reimbursement_account {
        state
            .retrieve_btc_account_to_block_indices
            .entry(account)
            .and_modify(|entry| entry.push(request.block_index))
            .or_insert(vec![request.block_index]);
    }
    if let Some(kyt_provider) = request.kyt_provider {
        *state.owed_kyt_amount.entry(kyt_provider).or_insert(0) += state.check_fee;
    }
}

    /// Forms a batch of retrieve_btc requests that the minter can fulfill.
    pub fn build_batch(&mut self, max_size: usize) -> BTreeSet<RetrieveBtcRequest> {
        let available_utxos_value = self.available_utxos.iter().map(|u| u.value).sum::<u64>();
        let mut batch = BTreeSet::new();
        let mut tx_amount = 0;
        for req in std::mem::take(&mut self.pending_retrieve_btc_requests) {
            if available_utxos_value < req.amount + tx_amount || batch.len() >= max_size {
                // Put this request back to the queue until we have enough liquid UTXOs.
                self.pending_retrieve_btc_requests.push(req);
            } else {
                tx_amount += req.amount;
                batch.insert(req);
            }
        }

        batch
    }
