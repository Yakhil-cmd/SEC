Audit Report

## Title
`SenderInfoContent` Lacks Request-Context Binding, Enabling Cross-Canister Attestation Replay — (`rs/types/types/src/messages/http.rs`, `rs/validator/src/ingress_validation.rs`)

## Summary

`SenderInfoContent` signs only the raw `info` blob, with no binding to `canister_id`, `sender`, or `ingress_expiry`. The validator `verify_sender_info_canister_sig` performs no certificate-timestamp or expiry check. An attacker who obtains a valid `sender_info.sig` for a given `info` blob (e.g., `role:admin`) can replay it verbatim against any target canister, because the inner canister signature is not scoped to any specific request context.

## Finding Description

`SenderInfoContent` is confirmed at [1](#0-0)  to sign only `self.0` (the raw `info` bytes). No `canister_id`, `sender`, or `ingress_expiry` is included in the signed payload.

`verify_sender_info_canister_sig` at [2](#0-1)  constructs `SenderInfoContent(&sender_info.info)` and verifies the canister signature over it. The only structural check performed is that the canister ID encoded in `sender_pubkey` matches `sender_info.signer` [3](#0-2) . No certificate timestamp, no `canister_id`, no `sender` binding.

`validate_sender_info` at [4](#0-3)  performs no expiration check on the canister-signature certificate before delegating to `verify_sender_info_canister_sig`.

The `MessageId` hash at [5](#0-4)  does include `canister_id`, `sender`, `ingress_expiry`, and the full `sender_info` blob (including `sig`). This means the envelope-level `sender_sig` is bound to the specific `sender_info.sig` value and the target canister. However, the inner `sender_info.sig` canister signature itself is not — it verifies independently of any request context. An attacker constructs a fresh `MessageId` for a new target canister, signs it with their own key (producing a valid envelope-level signature), and reuses the original `sender_info.sig` unchanged. `validate_sender_info` passes because `SenderInfoContent` only checks `info` bytes.

## Impact Explanation

Any canister that calls `msg_caller_info_data()` and makes access-control decisions on the returned blob is affected. An attacker who has obtained a `sender_info.sig` for `info_bytes = b"role:admin"` from signing canister II for one target can replay it against any other canister that trusts the same signing canister. This constitutes unauthorized access to canister-controlled resources and identities — matching the **High** impact tier: unauthorized access to canister-controlled assets where exploitation requires meaningful prior work (obtaining the initial `sender_info.sig`) but is then trivially repeatable.

## Likelihood Explanation

The attacker must first obtain a valid `sender_info.sig` for the desired `info` blob. This requires a prior legitimate interaction with the signing canister (e.g., Internet Identity), which is a realistic precondition for any user of the system. Once obtained, the replay requires no special privileges: the attacker constructs a new `HttpCanisterUpdate` targeting a different canister, reuses the `sender_info.sig` verbatim, and signs the new `MessageId` with their own key. The `sender_info` feature is present in production validator code with integration tests, making this a realistic attack surface.

## Recommendation

1. **Bind `SenderInfoContent` to the request context.** Extend the struct to carry `canister_id` and `sender`, and include them in `write_signed_bytes_without_domain_separator` before the `info` bytes, so the canister signature is scoped to a specific target and principal. [1](#0-0) 

2. **Enforce certificate freshness in `validate_sender_info`.** Extract the `time` field from the canister-signature certificate and reject it if it is older than `MAX_INGRESS_TTL` relative to the current replica time. [4](#0-3) 

## Proof of Concept

```
1. Call the signing canister (e.g., Internet Identity) to obtain a valid
   sender_info.sig over info_bytes = b"role:admin" for a request to canister X.

2. Construct a new HttpCanisterUpdate targeting canister Y:
   {
     canister_id: canister_Y,          // different target
     method_name: "privileged_action",
     sender: user_A,
     ingress_expiry: <fresh expiry>,
     sender_info: Some(RawSignedSenderInfo {
       info: b"role:admin",
       signer: II_canister_id,
       sig: <reused sender_info.sig from step 1>,  // unchanged
     }),
   }

3. Compute the new MessageId (includes canister_Y, fresh expiry, reused sig).
   Sign it with user_A's key — this envelope-level signature is fresh and valid.

4. Submit to the replica. validate_sender_info passes:
   - sender_pubkey encodes (II_canister_id, seed_A) ✓
   - pubkey_canister_id == sender_info.signer ✓
   - SenderInfoContent(b"role:admin") verifies against the reused certificate ✓
   - No canister_id or expiry check performed ✓

5. Canister Y's msg_caller_info_data() returns b"role:admin"; access granted.
```

A deterministic integration test can be written using PocketIC: issue a `sender_info.sig` targeting canister X, then submit an otherwise-identical request to canister Y with the same `sender_info.sig` and verify that `validate_request` accepts it.

fn validate_sender_info<C: HttpRequestContent, R: RootOfTrustProvider>(
    request: &HttpRequest<C>,
    ingress_signature_verifier: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    let Some(sender_info) = request.sender_info() else {
        return Ok(());
    };

    // Per the spec, the sender_info signature must verify using the
    // envelope-level sender_pubkey as a canister signature public key.
    let sender_pubkey = match request.authentication() {
        Authentication::Authenticated(sig) => &sig.signer_pubkey,
        Authentication::Anonymous => {
            return Err(InvalidSenderInfo(
                "sender_info requires an authenticated request with sender_pubkey".to_string(),
            ));
        }
    };

    verify_sender_info_canister_sig(
        sender_info,
        sender_pubkey,
        ingress_signature_verifier,
        root_of_trust_provider,
    )
}
