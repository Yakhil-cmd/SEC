### Title
Permanent wNEAR Freeze in Engine's Implicit EVM Address When XCC Router Deployment Fails After wNEAR Transfer Is Committed - (`engine-precompiles/src/xcc.rs`, `engine/src/xcc.rs`, `engine/src/contract_methods/xcc.rs`)

### Summary

When a user makes their first XCC cross-contract call with wNEAR attached, the XCC precompile atomically deducts wNEAR from the user's EVM balance and transfers it to the engine's implicit EVM address — an EVM state change committed in the current NEAR receipt. A subsequent NEAR promise chain is then scheduled to deploy the router contract and move the wNEAR to the router. If the router deployment fails (e.g., because an attacker pre-created the deterministic router account on NEAR), the `withdraw_wnear_to_router` callback fails, and the wNEAR is permanently stranded in the engine's implicit EVM address with no automated recovery path.

### Finding Description

**Step 1 — wNEAR deducted from user's EVM balance (committed, irreversible)**

In `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the precompile executes an EVM `transferFrom` call that moves wNEAR from the user's EVM address to the engine's implicit EVM address. This EVM state change is committed as part of the current NEAR receipt. [1](#0-0) 

**Step 2 — NEAR promise chain scheduled (separate receipts)**

In `engine/src/xcc.rs`, `handle_precompile_promise` schedules a multi-step promise chain:
1. Batch: `CreateAccount` + `Transfer(STORAGE_AMOUNT)` + `DeployContract` + `FunctionCall(initialize)`
2. Callback: `factory_update_address_version`
3. Callback: `withdraw_wnear_to_router`
4. Callback: `send_refund` (if `create_needed`)
5. Callback: `router.execute` or `router.schedule` [2](#0-1) 

**Step 3 — Router deployment fails, cascading callback failures**

If the router account was pre-created by an attacker, the `CreateAccount` action in the batch fails, causing the entire batch to fail. `factory_update_address_version` then receives a failed promise result and returns `ERR_ROUTER_DEPLOY_FAILED`. [3](#0-2) 

**Step 4 — `withdraw_wnear_to_router` fails, wNEAR stranded**

`withdraw_wnear_to_router` is called as a callback of the failed `factory_update_address_version`. It checks `promise_result_check()`, receives `Some(false)`, and returns `ERR_CALLBACK_OF_FAILED_PROMISE`. The NEAR runtime panics and reverts any state changes from this receipt. However, the wNEAR transfer from Step 1 (a prior, already-committed receipt) is **not** reverted. The wNEAR is now permanently stranded in the engine's implicit EVM address. [4](#0-3) 

**No automated recovery mechanism exists.** The engine's implicit EVM address is derived deterministically from the engine's NEAR account ID. There is no engine function that allows recovering wNEAR from this address. The `send_refund` function in the router only sends NEAR back to the engine and requires `require_no_failed_promises()`, so it cannot be used in the failure path. [5](#0-4) 

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

Any wNEAR attached to a user's first XCC call (covering `STORAGE_AMOUNT` = 2 NEAR for router deployment plus any user-specified attached NEAR) is permanently stranded in the engine's implicit EVM address when the router deployment fails. There is no automated refund path. The engine's implicit address is not directly controllable by any user, and no admin function exists to drain it. [6](#0-5) 

### Likelihood Explanation

**Likelihood: High.**

The router account ID is fully deterministic: `{hex(evm_address)}.aurora`. An attacker can:
1. Observe any EVM address that has not yet made an XCC call (visible from on-chain activity).
2. Pre-create `{hex(victim_address)}.aurora` on NEAR (a permissionless operation costing only storage).
3. Wait for the victim to make their first XCC call with any wNEAR attached.
4. The deploy batch fails, the callback chain fails, and the victim's wNEAR is frozen.

The attacker's cost is minimal (a small NEAR storage deposit). The victim loses all wNEAR attached to the XCC call (at minimum `STORAGE_AMOUNT` = 2 NEAR, plus any user-specified attached NEAR). [7](#0-6) [6](#0-5) 

### Recommendation

Implement a refund mechanism for the wNEAR transferred to the engine's implicit EVM address when the XCC promise chain fails. Specifically:

1. Attach a top-level error-handling callback to the entire XCC promise chain that, on failure, calls a new engine method (e.g., `xcc_refund_on_error`) to re-mint or return the wNEAR to the original sender's EVM address.
2. Alternatively, defer the wNEAR EVM transfer until after the router deployment is confirmed successful (i.e., move the EVM deduction into the `withdraw_wnear_to_router` callback rather than in the precompile itself).
3. At minimum, add an admin recovery function that can transfer wNEAR from the engine's implicit EVM address back to a specified recipient.

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker identifies victim EVM address `0xVICTIM` that has never made an XCC call (engine storage has no router version for this address → `create_needed = true`).
2. Attacker calls NEAR to create account `{hex(0xVICTIM)}.aurora` (permissionless, costs ~0.1 NEAR storage).
3. Victim submits an EVM transaction calling the XCC precompile at `0x516cded1d16af10cad51a19d` with `CrossContractCallArgs::Eager(call)` where `call.total_near() > 0`.
4. XCC precompile executes: wNEAR `transferFrom(victim, engine_implicit, required_near)` succeeds in EVM → state committed.
5. NEAR promise chain fires: batch `CreateAccount` on `{hex(0xVICTIM)}.aurora` → **fails** (account already exists).
6. `factory_update_address_version` callback fires → receives `Some(false)` → returns `ERR_ROUTER_DEPLOY_FAILED`.
7. `withdraw_wnear_to_router` callback fires → receives `Some(false)` → panics with `ERR_CALLBACK_OF_FAILED_PROMISE`.
8. All subsequent callbacks are skipped.
9. Victim's wNEAR (`required_near` = `attached_near + STORAGE_AMOUNT`) is permanently stranded in the engine's implicit EVM address. [8](#0-7) [1](#0-0)
