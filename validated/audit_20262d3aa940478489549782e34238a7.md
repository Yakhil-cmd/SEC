## Title
Neuron Split Resets `voting_power_refreshed_timestamp_seconds`, Bypassing Deciding Voting Power Inactivity Penalty - (`File: rs/nns/governance/src/governance.rs`)

## Summary
The `split_neuron` operation in the NNS governance canister constructs the child neuron via `NeuronBuilder::new` with `created_timestamp_seconds = now`, which is also used as the default for `voting_power_refreshed_timestamp_seconds`. No `.with_voting_power_refreshed_timestamp_seconds(...)` call is made to inherit the parent's timestamp. As a result, any neuron controller whose deciding voting power has been reduced or zeroed due to inactivity can call `split_neuron` to produce a child neuron with a fully refreshed deciding voting power, bypassing the NNS inactivity penalty mechanism entirely.

## Finding Description
In `rs/nns/governance/src/governance.rs` at line 2207, `created_timestamp_seconds` is set to `self.env.now()`. The child neuron is then built at lines 2241–2257 using `NeuronBuilder::new(..., created_timestamp_seconds)` with no subsequent `.with_voting_power_refreshed_timestamp_seconds(...)` call.

In `NeuronBuilder::new` (`rs/nns/governance/src/neuron/types.rs`, line 1759), the field `voting_power_refreshed_timestamp_seconds` is initialized to `created_timestamp_seconds`. Therefore the child neuron always receives `voting_power_refreshed_timestamp_seconds = now`, regardless of how stale the parent is.

The deciding voting power calculation in `potential_and_deciding_voting_power` (`rs/nns/governance/src/neuron/types.rs`, lines 390–396) computes `time_since_last_refreshed = now - voting_power_refreshed_timestamp_seconds` and applies a linear penalty that reaches zero after 7 months of inactivity. A parent neuron inactive for ≥7 months has `deciding_voting_power = 0`.

The ballot creation path in `compute_voting_power_snapshot_for_standard_proposal` (`rs/nns/governance/src/neuron_store/voting_power.rs`, line 159) inserts `deciding_voting_power` — not potential voting power — into the `voting_power_map` used to construct proposal ballots. The child neuron, having `voting_power_refreshed_timestamp_seconds = now`, will have `deciding_voting_power = potential_voting_power` (full), and this full value will appear in its ballot at the next proposal.

The existing test at `rs/nns/governance/tests/governance.rs` lines 5602–5616 explicitly asserts and documents this behavior: `child_neuron.voting_power_refreshed_timestamp_seconds == Some(driver.now())` and that it is strictly greater than the parent's stale timestamp. This confirms the root cause is present in production code, not a test artifact.

No existing guard in `split_neuron` checks or propagates the parent's `voting_power_refreshed_timestamp_seconds`. The only checks present are minimum stake, transaction fee, and caller authorization.

## Impact Explanation
This matches the allowed impact: **High ($2,000–$10,000) — Significant NNS governance security impact with concrete user or protocol harm.**

The deciding voting power mechanism (NNS proposal 132411) is a deliberate governance security feature designed to reduce the influence of inactive participants. Bypassing it allows a coalition of large inactive neuron holders to recover their full deciding voting power on demand immediately before a critical governance vote, without ever casting a vote or calling `refresh_voting_power`. This directly undermines governance accounting integrity: the total deciding voting power used in ballot tallies can be artificially inflated by inactive actors, distorting quorum calculations and vote outcomes. The parent neuron's deciding VP is not restored, but the child's is, so the net effect is a partial-to-full recovery proportional to the split fraction.

## Likelihood Explanation
Any NNS neuron controller can call `manage_neuron` with a `Split` command via ingress message — no privileged access, no social engineering, and no threshold corruption is required. The operation costs one ledger transfer fee. It can be executed immediately before a governance proposal is created. The minimum stake requirement (e.g., 1 ICP) bounds the number of splits but does not prevent the bypass: a 10,000 ICP neuron can produce up to ~9,999 child neurons each with full deciding VP. The attack is repeatable and deterministic.

## Recommendation
In `split_neuron` (`rs/nns/governance/src/governance.rs`, lines 2241–2257), inherit the parent's `voting_power_refreshed_timestamp_seconds` when constructing the child neuron:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    *caller,
    parent_neuron.dissolve_state_and_age(),
    created_timestamp_seconds,
)
.with_voting_power_refreshed_timestamp_seconds(
    parent_neuron.voting_power_refreshed_timestamp_seconds()
)
.with_hot_keys(parent_neuron.hot_keys.clone())
// ... rest unchanged
.build();
```

This ensures the child carries the same staleness as the parent. The child can then legitimately refresh by voting or calling `refresh_voting_power`. The test at lines 5602–5616 of `rs/nns/governance/tests/governance.rs` must also be updated to assert inheritance rather than reset.

## Proof of Concept
1. Create NNS neuron N with stake 1000 ICP and dissolve delay ≥ 6 months.
2. Advance time by 7+ months without voting, setting following, or calling `refresh_voting_power`. Neuron N now has `deciding_voting_power = 0`.
3. Call `manage_neuron` with `Split { amount_e8s: 500_000_000_000, memo: None }`.
4. Retrieve child neuron C: assert `voting_power_refreshed_timestamp_seconds = now` and `deciding_voting_power = potential_voting_power` (full, ~500 ICP × bonuses).
5. Create a governance proposal. Observe that C's ballot carries full deciding VP while N's carries 0.
6. The existing unit test at `rs/nns/governance/tests/governance.rs` lines 5602–5616 already reproduces and asserts steps 3–4 deterministically, confirming the behavior in the current codebase without any additional test infrastructure.
