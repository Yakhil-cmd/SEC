### Title
Treasury (`BASE_TOKEN_HOLDER_ADDRESS`) Has No On-Chain Replenishment Mechanism, Causing Fatal Chain Halt When Depleted — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

Every L1→L2 deposit in ZKsync OS transfers base tokens from a treasury account (`BASE_TOKEN_HOLDER_ADDRESS`) to the deposit recipient. If the treasury has insufficient balance, `transfer_from_treasury` returns `TreasuryTransferFailed` — a `BootloaderInterfaceError` that is **not** caught by the graceful error-handling path and propagates as a fatal internal error, halting block processing entirely. There is no system hook or bootloader function to replenish `BASE_TOKEN_HOLDER_ADDRESS` on-chain. The existing `mint_base_token_hook` (0x7100) only mints to `L2_BASE_TOKEN_ADDRESS` (0x800a), not to the treasury. This is the direct analog to the Golom finding: a missing replenishment function causes a critical system operation to fail fatally with no recovery path.

---

### Finding Description

**Step 1 — Treasury depletion path.**

Every L1→L2 transaction with `total_deposited > 0` calls `mint_base_token` → `transfer_from_treasury`, which subtracts from `BASE_TOKEN_HOLDER_ADDRESS`: [1](#0-0) 

If the treasury balance is insufficient, the function returns `BootloaderInterfaceError::TreasuryTransferFailed`: [2](#0-1) 

**Step 2 — Error is not caught gracefully.**

In `process_l1_transaction`, the error handling only converts `RuntimeError::FatalRuntimeError` to a graceful revert. `TreasuryTransferFailed` has root cause `RootCause::Interface(...)`, so it falls through the `_ =>` arm and is returned as a fatal error: [3](#0-2) 

The same unhandled propagation occurs in the post-execution operator-fee and refund minting calls: [4](#0-3) [5](#0-4) 

**Step 3 — Fatal error halts block processing.**

In the ZK transaction loop, any `TxError::Internal` causes `return Err(err)`, halting the entire block: [6](#0-5) 

**Step 4 — No on-chain replenishment mechanism exists.**

The `mint_base_token_hook` (0x7100) only allows `L2_BASE_TOKEN_ADDRESS` to mint tokens to itself — it does not target `BASE_TOKEN_HOLDER_ADDRESS`: [7](#0-6) [8](#0-7) 

There is no system hook, precompile, or bootloader function that adds balance to `BASE_TOKEN_HOLDER_ADDRESS`. The treasury can only be initialized off-chain (e.g., at genesis). The `mint_base_token` function in the bootloader is named misleadingly — it calls `transfer_from_treasury`, which is a one-way drain: [9](#0-8) 

**Step 5 — L1 transactions cannot be invalidated.**

The codebase explicitly documents that L1 transactions from the priority queue cannot be rejected, because doing so would halt the chain: [10](#0-9) 

This means once the treasury is depleted, every subsequent L1→L2 deposit causes a fatal block halt with no recovery path within ZKsync OS.

---

### Impact Explanation

When `BASE_TOKEN_HOLDER_ADDRESS` has insufficient balance for an L1→L2 deposit, `TreasuryTransferFailed` propagates as a fatal internal error that halts block processing. Since L1 transactions cannot be invalidated (they come from the priority queue), the chain is permanently halted. There is no on-chain mechanism to replenish the treasury and resume operation. This is a complete DoS of the L2 chain.

The existing test confirms the fatal behavior: [11](#0-10) 

---

### Likelihood Explanation

The treasury is initialized with `2^128 - 1` tokens in the test framework: [12](#0-11) 

For a production deployment, the treasury would be funded with the total bridgeable supply. Depletion through normal operation is unlikely. However:

1. An improper genesis initialization (e.g., treasury funded with a small balance due to a deployment error) immediately triggers the halt on the first L1→L2 deposit.
2. Any future protocol change that increases deposit volumes without a corresponding treasury top-up mechanism has no recovery path.
3. The missing replenishment hook is a structural gap — unlike the `mint_base_token_hook` which exists for `L2_BASE_TOKEN_ADDRESS`, no equivalent exists for `BASE_TOKEN_HOLDER_ADDRESS`.

---

### Recommendation

1. **Add a treasury replenishment system hook** analogous to `mint_base_token_hook` (0x7100), callable only by an authorized address (e.g., `L2_BASE_TOKEN_ADDRESS` or a governance contract), that adds balance to `BASE_TOKEN_HOLDER_ADDRESS`.

2. **Handle `TreasuryTransferFailed` gracefully** in `process_l1_transaction` — treat it as a transaction-level revert rather than a fatal block-level error, so the chain can continue processing subsequent transactions even if one deposit cannot be fulfilled.

3. **Ensure withdrawals (L2→L1) replenish the treasury** so the treasury is self-sustaining rather than a one-way drain.

---

### Proof of Concept

The existing test `test_treasury_insufficient_balance_failure` directly demonstrates the fatal chain halt: [11](#0-10) 

The treasury is set to 1000 wei, a deposit of 500,000 wei is submitted, and the block execution fails with `TreasuryTransferFailed` — a fatal error that halts block processing. Since L1 transactions cannot be invalidated, this is a permanent chain halt with no on-chain recovery path.

/// Test treasury transfer failure when treasury has insufficient balance
#[test]
fn test_treasury_insufficient_balance_failure() {
    use rig::system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    // Manually set very low treasury balance instead of using default
    let low_treasury_balance = U256::from(1000u64);
    let mut tester = TestingFramework::new()
        .with_balance(BASE_TOKEN_HOLDER_ADDRESS.into_alloy(), low_treasury_balance);

    // Create L1→L2 transaction that requires more tokens than treasury has
    let l1_sender = address!("1234000000000000000000000000000000000000");
    let l1_recipient = address!("5678000000000000000000000000000000000000");

    let gas_price = 1000u64;
    let gas_limit = 100_000u64;
    let value_to_transfer = U256::from(500_000u64); // More than treasury can cover

    let l1_tx: ZKsyncTxEnvelope = L1TxBuilder::new()
        .from(l1_sender)
        .to(l1_recipient)
        .gas_price(gas_price.into())
        .gas_limit(gas_limit.into())
        .value(value_to_transfer)
        .build()
        .into();

    // Ensure we rely on treasury balance, not auto-minting.
    tester = tester.without_minting_tokens_to_treasury();

    // This should fail due to insufficient treasury balance
    let result = tester.execute_block_no_panic(vec![l1_tx]);

    // Verify transaction fails due to treasury insufficient balance
    assert!(
        result.is_err(),
        "L1→L2 transaction should fail when treasury has insufficient balance"
    );

    // Verify the specific error is treasury transfer failed
    let error_debug = format!("{:?}", result.unwrap_err());
    assert!(
        error_debug.contains("TreasuryTransferFailed"),
        "Error should indicate treasury transfer failed, got: {}",
        error_debug
    );
}
