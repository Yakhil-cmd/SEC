### Title
Native Resource Exhaustion in Callee Bypasses EVM CALL Failure Semantics, Causing Uncatchable Transaction-Level Revert - (`basic_bootloader/src/bootloader/runner.rs`)

### Summary

ZKsync OS's double resource accounting model passes native resources fully from caller to callee with no per-call limit. When a callee exhausts native resources, a `FatalRuntimeError::OutOfNativeResources` propagates as a Rust `Err` through the call stack, bypassing the EVM's normal CALL return-value mechanism. The caller never receives a `0` (failure) on its stack; instead, the entire transaction is reverted. This is an EVM semantic mismatch exploitable by any unprivileged contract caller.

### Finding Description

ZKsync OS implements a **double resource accounting** model: EVM gas (ergs) and native resources (RISC-V proving cycles). The two resources have fundamentally different propagation rules:

- **Ergs**: The 63/64 rule applies; a caller can limit how many ergs it passes to a callee.
- **Native resources**: Passed **fully** from frame to frame; a caller cannot limit the callee's native consumption.

This is explicitly documented:

> "The native resources are passed fully from frame to frame, a call cannot set a limit on how much of it the callee can spend." [1](#0-0) 

The transfer is implemented unconditionally in `read_callee_and_prepare_frame_state`:

```rust
// Give native resource to the callee.
resources_in_caller_frame.give_native_to(&mut callee_resources);
``` [2](#0-1) 

When the callee exhausts native resources, `FatalRuntimeError::OutOfNativeResources` is raised: [3](#0-2) 

This error propagates as a Rust `Err` via the `?` operator at the call-dispatch site:

```rust
let (resources_returned_from_callee, call_result) = callee_frame_execution_result?;
``` [4](#0-3) 

The `?` means the error **never reaches** `continue_after_preemption` in the EVM interpreter. That function only handles the three `CallResult` variants (`PreparationStepFailed`, `Failed`, `Successful`): [5](#0-4) 

The `FatalRuntimeError` instead propagates all the way up to the transaction-level handler, which converts it into a full transaction revert with all gas exhausted:

```rust
Err(e) => match e.root_cause() {
    RootCause::Runtime(e @ RuntimeError::FatalRuntimeError(_)) => {
        context.resources.main_resources.exhaust_ergs();
        system.finish_global_frame(Some(&main_body_rollback_handle))?;
        ExecutionResult::Revert { output: &[] }
    }
``` [6](#0-5) 

The same pattern exists for ZK-style transactions: [7](#0-6) 

### Impact Explanation

In standard EVM, when a callee runs out of gas, the `CALL` opcode returns `0` (failure) on the caller's stack. The caller retains its remaining gas (minus what was forwarded) and continues execution. Solidity `try/catch` blocks, low-level `(bool success, ) = addr.call{gas: N}(...)` patterns, and any contract that checks the CALL return value all rely on this guarantee.

In ZKsync OS, if the callee exhausts **native** resources (not ergs), the entire transaction reverts unconditionally. The caller never gets control back. This means:

1. **Locked funds**: A contract that calls an external address and falls back to a safe path on failure (e.g., a vault that tries to call a hook and refunds on failure) will instead have its entire transaction reverted, potentially locking deposited funds.
2. **DoS via griefing**: An attacker-controlled callee can exhaust native resources to force a transaction-level revert on any caller that invokes it, even if the caller allocated only a small gas stipend.
3. **EVM equivalence violation**: ZKsync OS claims EVM equivalence, but this behavior diverges from the EVM specification for any transaction where a callee's native cost exceeds the remaining native budget.

### Likelihood Explanation

The native resource budget is derived from `gasPrice / nativePrice * gasLimit`. The `nativePrice` is an operator-set constant. If the operator sets a low `nativePrice` (cheap native resources), the native budget is large and exhaustion is unlikely. However:

- The `nativePrice` can change between blocks.
- A transaction estimated at one `nativePrice` may run out of native resources at a higher `nativePrice`.
- An attacker can craft a callee that performs many native-resource-intensive operations (e.g., storage writes, keccak rounds, bytecode preprocessing) to exhaust the budget.
- Any contract that uses `try/catch` or checks CALL return values to handle callee failures is vulnerable if the callee is attacker-controlled or if native costs are underestimated.

### Recommendation

1. **Intercept `FatalRuntimeError::OutOfNativeResources` at the call boundary** in `handle_requested_external_call` and convert it to a `CallResult::Failed` (with all ergs in the callee frame burned), analogous to how `OutOfErgs` is handled. This restores EVM-compatible CALL failure semantics.

2. **Alternatively**, document this as a known deviation from EVM semantics and ensure that `eth_estimateGas` always provides sufficient native budget so that native exhaustion cannot occur mid-call in practice.

3. **Per-call native limits**: Allow callers to cap the native resources passed to a callee (analogous to the gas parameter in `CALL`), so that a callee cannot exhaust the caller's entire native budget.

### Proof of Concept

```
Contract A (victim):
  function safeCall(address target) external {
      (bool ok, ) = target.call{gas: 50000}("");
      if (!ok) {
          // expected fallback path
          emit CallFailed();
      }
  }

Contract B (attacker):
  fallback() external {
      // Perform 1000 cold storage writes to exhaust native resources
      for (uint i = 0; i < 1000; i++) {
          assembly { sstore(i, 1) }
      }
  }
```

On standard EVM: `A.safeCall(B)` returns with `ok = false`; `CallFailed` is emitted; transaction succeeds.

On ZKsync OS: When `B`'s storage writes exhaust native resources, `FatalRuntimeError::OutOfNativeResources` propagates past `runner.rs:219` via `?`, bypasses `continue_after_preemption`, reaches the transaction-level handler at `ethereum/mod.rs:441`, and the entire transaction reverts. `A` never executes its fallback path. All gas is burned. [8](#0-7) [9](#0-8) [10](#0-9)
