Audit Report

## Title
Rate-Limit Quota Permanently Consumed by Failed `notify_top_up` Calls, Enabling CMC Minting DoS - (`File: rs/nns/cmc/src/main.rs`)

## Summary

In `deposit_cycles`, `ensure_balance` is called before the management-canister `deposit_cycles` call, charging the rate limiter and physically minting cycles into the CMC's balance. If the subsequent management-canister call fails (e.g., non-existent target canister), the ICP is refunded to the caller but the rate-limit quota is permanently consumed with no rollback path. An unprivileged attacker can exploit this to exhaust the 150 P/hour global minting cap at the cost of only ledger transfer fees, blocking all CMC minting operations for up to one hour per attack window.

## Finding Description

In `deposit_cycles` (`rs/nns/cmc/src/main.rs`, lines 2110–2138), `ensure_balance` is called unconditionally when `mint_cycles = true`, before the management-canister call:

```rust
if mint_cycles {
    ensure_balance(cycles, limiter_to_use)?;   // rate limit charged here
}
let res = ic_cdk::api::call::call_with_payment128(...).await;  // may fail
```

`ensure_balance` (lines 2306–2324) calls `limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)` and increments `state.total_cycles_minted` before the management-canister call is even attempted. In `Limiter::check_and_add_cycles` (`rs/nns/cmc/src/limiter.rs`, lines 34–56), once `self.add(now, cycles_to_mint)` is called at line 54, there is no rollback mechanism — the window count is permanently incremented.

When the management-canister call fails, `deposit_cycles` returns `Err`, and `process_top_up` (lines 1985–2011) issues a full ICP refund via `refund_icp`. The rate-limit quota is not restored. The global base limit is `DEFAULT_CYCLES_LIMIT = 150e15` (150 P/hour, line 83).

The integration test `cmc_notify_top_up_invalid` (lines 1618–1641 of `rs/nns/integration_tests/src/cycles_minting_canister.rs`) explicitly confirms that 100 T cycles are recorded as minted even when the top-up fails and ICP is refunded. The test `cmc_notify_top_up_not_rate_limited_by_invalid_top_up` (lines 1689–1743) models the attack at 149.04 P/hour (just below the 150 P limit) and confirms that failed top-ups do consume the rate-limit quota; it only passes because the attacker deliberately stays below the cap.

An attacker using the iterative approach (reusing the same ICP since it is refunded each time) can push the total to exactly 150 P within one hour, at which point all subsequent `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` calls are rejected with the rate-limit error for the remainder of the hour.

## Impact Explanation

Exhausting the 150 P/hour base limit blocks all CMC minting operations for up to one hour. Any canister that runs out of cycles during this window will be frozen or deleted. The attack is repeatable every hour indefinitely. This constitutes an application/platform-level DoS on a core NNS infrastructure canister (CMC), matching the **High** impact category: "Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

`notify_top_up` is a public, permissionless `#[update]` endpoint callable by any principal with no staking, governance role, or privileged key required. The attacker's principal ICP is refunded on each failed call; the only cost is the ICP ledger transfer fee (~0.0001 ICP per block) plus `TOP_UP_CANISTER_REFUND_FEE`. Starting with 1 ICP, an attacker can sustain the attack across many hourly windows by reusing the same ICP. The attack requires no special knowledge beyond the existence of a non-existent canister ID.

## Recommendation

Move the `ensure_balance` / rate-limiter charge to **after** a confirmed successful management-canister `deposit_cycles` call. Alternatively, if the deposit call fails and cycles are returned to the CMC's balance, subtract the previously charged amount from the limiter's window count before returning the error. A third option is to verify that the target canister exists and is running (via a management-canister `canister_status` call) before charging the limiter.

## Proof of Concept

1. Attacker holds ~1 ICP (enough to cover fees across many iterations).
2. For each iteration (repeating ~720 times over one hour, advancing 5 seconds between calls):
   - Transfer ~0.69 ICP to the CMC subaccount for a non-existing `canister_id` (e.g., `CanisterId::from_u64(999_999_999)`), obtaining `block_index`.
   - Call `notify_top_up { block_index, canister_id }`.
   - CMC calls `ensure_balance(~207T cycles)` → rate limiter records ~207T, cycles minted into CMC balance.
   - CMC calls management canister `deposit_cycles` for the non-existing canister → fails, cycles returned to CMC balance.
   - CMC refunds ~0.69 ICP minus fees to attacker.
3. After ~720 iterations, the `base_limiter` shows ≥150 P consumed within the sliding hour window.
4. All subsequent `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` calls are rejected with `"More than 150000000000000000 cycles have been minted in the last 3600 seconds, please try again later."` for up to one hour.
5. The attack is repeatable each hour. Total cost: ~0.072 ICP in transfer fees per hour.

This is directly reproducible by modifying the existing integration test `cmc_notify_top_up_not_rate_limited_by_invalid_top_up` to use amounts that sum to ≥150 P within the sliding window and asserting that the final valid top-up fails with the rate-limit error. fn ensure_balance(
```rust
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    // unused because of check above
    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
```

```rust
    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }
```

```rust
#[test]
fn cmc_notify_top_up_not_rate_limited_by_invalid_top_up() {
    let good_account = AccountIdentifier::new(*TEST_USER1_PRINCIPAL, None);
    let good_account_icpts = Tokens::new(10, 0).unwrap();
    let bad_account = AccountIdentifier::new(*TEST_USER2_PRINCIPAL, None);
    let bad_account_icpts = Tokens::new(1, 0).unwrap();

    let state_machine = state_machine_builder_for_nns_tests().build();
    let nns_init_payloads = NnsInitPayloadsBuilder::new()
        .with_test_neurons()
        .with_ledger_accounts(vec![
            (good_account, good_account_icpts),
            (bad_account, bad_account_icpts),
        ])
        .build();
    setup_nns_canisters(&state_machine, nns_init_payloads);
    let non_existing_canister_id = CanisterId::from_u64(123_456_789);

    // First make sure topping up 400T cycles on a valid canister works.
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(4, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(400_000_000_000_000_u128));

    // Advance time by 1 hour to make sure the rate limit is reset.
    state_machine.advance_time(Duration::from_secs(60 * 60));

    // Now the attack begins - the bad account sends 0.69 tokens per 5 seconds to the non-existing
    // canister, which makes it 207T * 12 * 60 = 149.04P cycles per hour, close to the 150P limit,
    // while getting the 0.69 tokens back each time (the account only has 1 token in the
    // beginning).
    for _ in 0..(12 * 60) {
        let error = notify_top_up(
            &state_machine,
            non_existing_canister_id,
            Tokens::from_e8s(207_000_000),
        )
        .unwrap_err();
        assert_matches!(error, NotifyError::Refunded { .. });
        state_machine.advance_time(Duration::from_secs(5));
    }

    // Now the good account tries to top up 400T cycles on the governance canister again, which
    // should still succeed.
    let cycles = notify_top_up(
        &state_machine,
        GOVERNANCE_CANISTER_ID,
        Tokens::new(4, 0).unwrap(),
    )
    .unwrap();
    assert_eq!(cycles, Cycles::new(400_000_000_000_000_u128));
}
```
