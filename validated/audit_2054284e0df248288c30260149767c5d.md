### Title
Fee-on-Transfer ERC-20 Tokens Cause ckERC20 Over-Minting via Deposit Helper Contracts - (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol` and `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

Both ckERC20 deposit helper contracts emit the caller-supplied `amount` parameter in the deposit event rather than the actual token balance received by the minter address. The IC ckETH minter reads this emitted value and mints an equal quantity of ckERC20 tokens. For any ERC-20 token that deducts a fee on transfer (fee-on-transfer tokens), the minter receives fewer tokens than the event reports, causing it to mint more ckERC20 than it holds in ERC-20 backing тАФ a direct conservation violation.

---

### Finding Description

**Helper contract `CkDeposit.depositErc20()` (`DepositHelperWithSubaccount.sol`):**

Both contracts emit the `amount` argument directly. For a fee-on-transfer ERC-20, `safeTransferFrom` succeeds but the minter address receives `amount - fee`, while the event records `amount`.

**IC minter log parser reads the emitted value verbatim:**

The `ReceivedErc20LogParser` and `ReceivedEthOrErc20LogParser` parse the `data` field of the event log тАФ which contains the emitted `amount` тАФ and store it as `value` in `ReceivedErc20Event`: [3](#0-2) 

**IC minter mints exactly `event.value()` ckERC20 tokens:**

```rust

        ...

    .await
``` [4](#0-3) 

**Minter internal ERC-20 balance tracking is also inflated:**

The minter's `erc20_balances` state is updated with `event.value` (the emitted amount), not the actual on-chain balance: [5](#0-4) 

---

### Impact Explanation

For every deposit of a fee-on-transfer ERC-20 token, the minter mints `amount` ckERC20 but only holds `amount - fee` ERC-20 tokens. The 1:1 backing invariant is broken:

- **ckERC20 total supply > actual ERC-20 held by minter**
- The minter's internal `erc20_balances` counter is inflated by the fee amount per deposit
- When users later withdraw (ckERC20 тЖТ ERC-20), the minter will eventually run out of ERC-20 tokens before all ckERC20 can be redeemed тАФ the last withdrawers receive nothing
- An attacker can exploit this by repeatedly depositing a fee-on-transfer token, accumulating over-minted ckERC20, and withdrawing real ERC-20 tokens, draining the minter's reserves

The `erc20_sub` function will panic with an underflow when the minter tries to debit more ERC-20 than it actually holds: [6](#0-5) 

---

### Likelihood Explanation

The ckERC20 system is designed to support arbitrary ERC-20 tokens added via NNS proposal. The helper contract itself explicitly states it does not enforce a whitelist: [7](#0-6) 

Fee-on-transfer tokens are a well-known ERC-20 pattern (e.g., USDT on some chains, STA, REFLECT-based tokens, various DeFi tokens). If any such token is ever added as a supported ckERC20 token via NNS governance, every deposit will silently over-mint. Additionally, existing tokens (e.g., USDT) could enable their fee mechanism via an upgrade тАФ USDT's contract already has fee logic that is currently disabled.

---

### Recommendation

Both helper contracts should measure the actual balance change rather than trusting the `amount` parameter:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

This mirrors the mitigation recommended in the original M-08 report. The IC minter would then mint only the actual received amount.

Alternatively, the minter canister could validate that the emitted `value` does not exceed the actual ERC-20 balance increase by cross-checking on-chain balances, though this is more complex in the IC's async RPC model.

---

### Proof of Concept

1. A fee-on-transfer ERC-20 token `FeeToken` (1% fee) is added as a supported ckERC20 token via NNS proposal.
2. Alice calls `depositErc20(FeeToken, 1000, alicePrincipal, subaccount)` on the helper contract.
3. `safeTransferFrom(Alice, minter, 1000)` executes: Alice loses 1000 tokens, minter receives 990 (10 taken as fee).
4. The helper emits `ReceivedEthOrErc20(FeeToken, Alice, 1000, alicePrincipal, subaccount)`.
5. The IC minter scrapes the log, parses `value = 1000`, and mints 1000 ckFeeToken to Alice.
6. Minter holds 990 FeeToken but has issued 1000 ckFeeToken тАФ 10 ckFeeToken are unbacked.
7. Alice calls `withdraw_erc20` for 1000 ckFeeToken, receiving 1000 FeeToken from the minter's reserves (which only had 990 from her deposit, supplemented by other users' deposits).
8. Repeated over many deposits, the minter's ERC-20 reserves are drained relative to outstanding ckERC20 supply. [1](#0-0) [8](#0-7) [4](#0-3)
