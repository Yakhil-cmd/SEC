Audit Report

## Title
SNS Governance Neuron Slot Exhaustion via Minimum-Stake Claim and Immediate Disburse - (File: rs/sns/governance/src/governance.rs)

## Summary
`disburse_neuron` transfers a neuron's stake back to the caller but never calls `remove_neuron`, leaving a zero-stake "zombie" entry permanently in `self.proto.neurons`. Because `check_neuron_population_can_grow` counts all entries in that map, an attacker can exhaust all `max_number_of_neurons` slots at a net cost of only transaction fees, permanently preventing any new neuron creation in the SNS.

## Finding Description

**Root cause — `disburse_neuron` does not remove the neuron:**
`disburse_neuron` (L1119–1237) performs two ledger transfers and then only decrements `cached_neuron_stake_e8s` via `saturating_sub`. It never calls `remove_neuron`. The neuron entry remains in `self.proto.neurons` with a zero stake indefinitely. [1](#0-0) 

**Slot counter counts all entries including zombies:**
`check_neuron_population_can_grow` (L6363–6379) checks `self.proto.neurons.len()`, which includes every entry regardless of stake. [2](#0-1) 

**`claim_neuron` creates an immediately-dissolved neuron:**
`claim_neuron` (L4319–4400) initialises every new neuron with `dissolve_state: Some(DissolveState::DissolveDelaySeconds(0))`, placing it in the `Dissolved` state immediately, so `disburse_neuron`'s dissolved-state check passes right away. [3](#0-2) 

**Exploit flow:**
1. Attacker transfers `neuron_minimum_stake_e8s` tokens to the governance subaccount for `(principal_i, memo_i)`.
2. Calls `manage_neuron(ClaimOrRefresh)` → `claim_neuron` creates the neuron with `DissolveDelaySeconds(0)` and increments the slot count.
3. Calls `manage_neuron(Disburse)` → `disburse_neuron` returns the stake but leaves the neuron in `proto.neurons`.
4. Repeat until `proto.neurons.len() == max_number_of_neurons`.

After the loop, every subsequent `ClaimOrRefresh` call returns `"Cannot add neuron. Max number of neurons reached."`.

**No SNS rate limiter exists:**
NNS governance enforces `MAX_SUSTAINED_NEURONS_PER_HOUR = 15` and `MAX_NEURON_CREATION_SPIKE = 300` to make this attack impractical. [4](#0-3) 

SNS governance has no equivalent protection — a search for any rate-limiting construct in `rs/sns/governance/src/governance.rs` returns no results.

**Hard cap:**
`MAX_NUMBER_OF_NEURONS_CEILING` is 200,000. [5](#0-4) 

## Impact Explanation

Once all 200,000 slots are filled, no new principal can create a neuron in the SNS. New token holders are permanently locked out of voting, proposal submission, and reward accrual. This is a concrete, application-level DoS of SNS governance with direct user and protocol harm, matching the **High ($2,000–$10,000)** impact class: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation

Net attacker cost is `200,000 × 2 × 10,000 e8s = 40 governance tokens` in transaction fees; all staked principal is recovered. The attack requires no special privileges — any principal holding ~40 governance tokens can execute it via standard `manage_neuron` ingress calls. It is fully automatable and non-reversible without a governance proposal to raise `max_number_of_neurons`, which itself may be impaired if the SNS has few existing neuron holders.

## Recommendation

1. **Remove neurons on full disbursal**: In `disburse_neuron`, after reducing `cached_neuron_stake_e8s` to zero (or below `transaction_fee_e8s`), call `remove_neuron` to free the slot.
2. **Add a rate limiter to SNS `claim_neuron`**: Mirror the NNS `MAX_SUSTAINED_NEURONS_PER_HOUR` / `MAX_NEURON_CREATION_SPIKE` token-bucket rate limiter.
3. **Enforce a minimum dissolve delay**: Require a non-zero dissolve delay at claim time so that stake cannot be immediately recovered, raising the economic cost of the attack.

## Proof of Concept

```
for i in 1..=max_number_of_neurons:
  1. Transfer neuron_minimum_stake_e8s tokens to governance subaccount(principal_i, memo_i)
  2. manage_neuron(ClaimOrRefresh { memo: memo_i, controller: principal_i })
     → claim_neuron creates neuron with DissolveDelaySeconds(0) → Dissolved immediately
     → add_neuron increments proto.neurons count
  3. manage_neuron(Disburse { amount: None, to_account: None })
     → disburse_neuron transfers stake back; proto.neurons entry stays with stake = 0

After loop:
  manage_neuron(ClaimOrRefresh { ... }) by any new user returns:
  GovernanceError { PreconditionFailed, "Cannot add neuron. Max number of neurons reached." }
```

A deterministic integration test using PocketIC can reproduce this by iterating 200,000 claim+disburse cycles against a local SNS canister and asserting that the 200,001st `ClaimOrRefresh` returns `PreconditionFailed`.
