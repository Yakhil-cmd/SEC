Audit Report

## Title
Unpermissioned `ClaimOrRefresh` Neuron Operation Enables Transient DoS on Neuron Ledger Operations - (File: rs/nns/governance/src/governance.rs)

## Summary
Any IC principal can invoke `ClaimOrRefresh` with `By::NeuronIdOrSubaccount` on any existing NNS neuron without authorization. This operation acquires a per-neuron `in_flight_commands` lock before an async ledger `account_balance` call. While the lock is held across the async boundary, all concurrent balance-modifying operations (`Disburse`, `Split`, `DisburseToNeuron`, `Merge`) on the same neuron are rejected with `LedgerUpdateOngoing`. An attacker sustaining a stream of such messages can transiently block a victim neuron owner from executing time-sensitive operations.

## Finding Description
In `manage_neuron_internal` (`rs/nns/governance/src/governance.rs`, L6104–6148), `ClaimOrRefresh` is dispatched before any authorization check and returns early. The `By::NeuronIdOrSubaccount` variant calls `refresh_neuron_by_id_or_subaccount` with no caller validation.

Inside `refresh_neuron` (L5900–5923), `lock_neuron_for_command` is called to insert the neuron ID into `heap_data.in_flight_commands` before the `await` on `self.ledger.account_balance(account)`. Because the IC canister model yields execution at `await` points and can process other incoming messages while waiting for the inter-canister response, the lock is held across message boundaries.

`lock_neuron_for_command` (`rs/nns/governance/src/neuron_lock.rs`, L219–238) returns `ErrorType::LedgerUpdateOngoing` for any concurrent attempt to acquire the same lock. This means any `Disburse`, `Split`, or `Merge` call arriving while Eve's `ClaimOrRefresh` is in-flight will be rejected.

The test suite at `rs/nns/governance/tests/governance.rs` (L5014–5029) explicitly documents: *"Tests that a neuron can be refreshed by subaccount, and that anyone can do it."* — confirming the permissionless design.

In SNS Governance (`rs/sns/governance/src/governance.rs`, L4786–4793), the lock is acquired at the top of `manage_neuron_internal` for all commands including `ClaimOrRefresh`, before dispatching to `refresh_neuron` (L4237–4256) which makes the async ledger call. This extends the lock window further.

The `NeuronAsyncLock` Drop implementation (`rs/nns/governance/src/neuron_lock.rs`, L43–60) intentionally retains the lock if `is_recovering_from_trap()` is true, meaning a trap during a `ClaimOrRefresh` callback permanently freezes the neuron — though this secondary scenario requires a trap, which is not directly attacker-controlled.

## Impact Explanation
This matches the allowed High impact: *"Application/platform-level DoS … not based on raw volumetric DDoS."* A victim neuron owner is transiently blocked from executing `Disburse`, `Split`, `DisburseToNeuron`, `Merge`, or `Spawn` for the duration of the attack. For NNS neurons with time-sensitive dissolve windows or pending disbursements, this constitutes concrete user harm. The attack targets governance infrastructure (NNS/SNS), which is explicitly in-scope.

## Likelihood Explanation
No special privileges are required — any IC principal can submit ingress messages to the NNS governance canister. The attacker must sustain message throughput sufficient to keep the lock occupied across ledger round-trips (~2 seconds each on mainnet). IC ingress rate limits and per-message cycle costs constrain indefinite blocking, but transient blocking (seconds to minutes) is feasible with modest throughput. The SNS case is slightly easier because the lock window begins at the top of `manage_neuron_internal` rather than inside `refresh_neuron`.

## Recommendation
- Restrict `ClaimOrRefresh` with `By::NeuronIdOrSubaccount` to callers who hold at least one permission on the target neuron (e.g., the neuron controller or a hotkey), consistent with how other neuron commands are gated.
- Alternatively, do not acquire the neuron lock for `ClaimOrRefresh` operations that only read the ledger balance and update `cached_neuron_stake_e8s` upward; the lock is only strictly necessary to prevent concurrent stake-modifying operations.
- Add a per-neuron rate limit on `ClaimOrRefresh` invocations to bound the DoS surface even if the operation remains permissionless.

## Proof of Concept
1. Alice owns NNS neuron `N` and wishes to call `Disburse`.
2. Eve submits a stream of `manage_neuron` ingress messages with `Command::ClaimOrRefresh { by: By::NeuronIdOrSubaccount(Empty{}) }` targeting neuron `N`.
3. Each message causes `refresh_neuron` to acquire the `in_flight_commands` lock for `N` and issue an async `account_balance` query to the ICP ledger canister.
4. While any one of Eve's messages is in-flight (lock held), Alice's `Disburse` call hits `lock_neuron_for_command`, finds `N` in `in_flight_commands`, and returns `ErrorType::LedgerUpdateOngoing`.
5. Eve maintains the stream; Alice's disbursement is continuously rejected for the duration of the attack.
6. A deterministic integration test using PocketIC can reproduce this by: (a) creating a neuron, (b) submitting a `ClaimOrRefresh` message and pausing before the ledger callback, (c) submitting a `Disburse` message and asserting it returns `LedgerUpdateOngoing`.
