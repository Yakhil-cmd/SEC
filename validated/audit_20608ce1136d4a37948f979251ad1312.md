### Title
Permanent Fund Freeze in `ExitToNear` Precompile When `ft_transfer` Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When a user calls the `exit_to_near` precompile to bridge ERC-20 tokens (or ETH) to NEAR, the tokens are burned/deducted from the EVM balance **before** the NEAR-side `ft_transfer` promise executes. If that promise fails (e.g., recipient not registered with the NEP-141 contract) and the `error_refund` compile-time feature is absent, the `refund` field in `ExitToNearPrecompileCallbackArgs` is hardcoded to `None`. The callback therefore takes no refund action, leaving the burned ERC-20 tokens permanently unrecoverable and the NEP-141 tokens permanently locked in Aurora's contract balance.

---

### Finding Description

**Root cause — `engine-precompiles/src/native.rs`, lines 449–455:**

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not compiled in, `refund` is unconditionally `None` regardless of whether the downstream `ft_transfer` promise succeeds or fails.

**Consequence — `engine/src/contract_methods/connector.rs`, lines 214–242:**

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {
    // refund path — only reached if refund is Some(...)
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← taken when refund == None; no recovery action
};
``` [2](#0-1) 

When `ft_transfer` fails **and** `refund` is `None`, the `else` branch executes and returns `None` — no re-mint, no ETH credit, no recovery of any kind.

**The two-step accounting mismatch (analogous to the commission escrow):**

1. **ERC-20 exit path**: The ERC-20 contract burns the user's tokens (deducting from EVM state) and calls the precompile. The NEP-141 `ft_transfer` is then attempted. If it fails, the burned ERC-20 tokens are gone and the NEP-141 tokens remain in Aurora's balance — permanently frozen.

2. **ETH exit path**: ETH is transferred from the user's EVM address to the `exit_to_near` precompile address before the `ft_transfer` promise runs. If `ft_transfer` fails without `error_refund`, the ETH is never returned from the precompile address.

Both cases are explicitly documented in the test suite:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [3](#0-2) 

```rust

``` [4](#0-3) 

The code comment in `exit_erc20_token_to_near` itself acknowledges the locking risk for the ETH-attached case:

> "the funds will be locked in the address of the precompile without any possibility to withdraw them in the future" [5](#0-4) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

- ERC-20 tokens are burned from the user's EVM balance with no corresponding NEP-141 credit to the recipient.
- NEP-141 tokens remain in Aurora's contract balance with no mechanism to recover them.
- ETH transferred to the `exit_to_near` precompile address is similarly unrecoverable.
- There is no admin function, no retry path, and no sweep mechanism to return these funds.

---

### Likelihood Explanation

**Likelihood: Medium.**

The failure condition (`ft_transfer` failing) is realistic and user-triggerable:
- Specifying a NEAR account not registered with the NEP-141 contract (the exact scenario in the test).
- Specifying a NEAR account that has been deleted.
- Any transient NEP-141 contract failure.

Any EVM user who calls `withdrawToNear` on a bridged ERC-20 token with a failing recipient triggers this path. The user does not need any special privileges.

---

### Recommendation

**Short term:** Ensure the `error_refund` feature is always enabled in production builds, or remove the conditional compilation entirely and unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs`.

**Long term:** Add invariant tests asserting that the total NEP-141 supply held by Aurora plus all ERC-20 mirror supplies remains constant across all exit operations, including failed ones.

---

### Proof of Concept

1. Alice has 100,000 units of a bridged ERC-20 token on Aurora (backed by NEP-141 tokens held by Aurora's contract).
2. Alice calls `withdrawToNear("unregistered.near", 100000)` on the ERC-20 contract.
3. The ERC-20 contract burns 100,000 tokens from Alice's EVM balance and calls the `exit_to_near` precompile.
4. The precompile constructs `ExitToNearPrecompileCallbackArgs { refund: None, ... }` (because `error_refund` is not compiled in).
5. A `ft_transfer` promise is scheduled targeting the NEP-141 contract with recipient `unregistered.near`.
6. The `ft_transfer` fails because `unregistered.near` has no storage deposit with the NEP-141 contract.
7. `exit_to_near_precompile_callback` is invoked; `args.refund` is `None`; the `else { None }` branch executes.
8. Alice's 100,000 ERC-20 tokens are permanently burned. The 100,000 NEP-141 tokens remain in Aurora's balance. No recovery is possible. [6](#0-5) [7](#0-6)
