Audit Report

## Title
`sender_info.sig` Included in `MessageId` Computation Enables Deduplication Bypass via Non-Deterministic Canister Signatures — (File: `rs/types/types/src/messages/http.rs`)

## Summary

The `representation_independent_hash_call_or_query` function includes the full `sender_info` nested map — including the `sig` field — when computing the `MessageId`. Because IC canister signatures are non-deterministic across rounds (the same `(canister_id, seed, msg)` triple produces distinct valid `sig` byte strings using different IC certificates), an attacker who controls the signer canister can submit two logically identical ingress messages with different `sender_info.sig` values, yielding two distinct `MessageId`s. Both pass all validation, are not deduplicated against each other, and are both executed.

## Finding Description

**Root cause — `sig` in the hash:**

In `rs/types/types/src/messages/http.rs` lines 68–77, `representation_independent_hash_call_or_query` inserts the full `sender_info` nested map including `sig` into the hash:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, sig }) = sender_info {
    map.insert(
        "sender_info",
        Map(btreemap! {
            "info" => Bytes(info),
            "signer" => Bytes(signer),
            "sig" => Bytes(sig),   // ← signature bytes affect MessageId
        }),
    );
}
``` [1](#0-0) 

This is consumed by `HttpCanisterUpdate::id()` and `SignedIngressContent::id()`: [2](#0-1) [3](#0-2) 

**Deduplication is keyed on `MessageId`:**

`ValidSetRuleImpl::is_duplicate` checks `state.get_ingress_status(&msg.content().id())`. Two messages with different `sig` bytes produce different `id()` values and are treated as distinct: [4](#0-3) 

`IngressHistorySet::contains` in the ingress selector does the same: [5](#0-4) 

`validate_ingress_pool_object` in the ingress handler also checks `ingress_message_status(&ingress_object.message_id)`, keyed on `MessageId`: [6](#0-5) 

The ingress pool itself stores artifacts keyed by `IngressMessageId` (derived from `MessageId`), so two messages with different `sig` bytes occupy distinct pool slots: [7](#0-6) 

**Canister signatures are non-deterministic across rounds:**

`rs/crypto/iccsa/src/api.rs` shows that a canister signature is valid if the IC certificate covers the canister's `certified_data` and the witness tree contains an empty leaf at `["sig", SHA256(seed), SHA256(msg)]`. The certificate itself (embedded in the `sig` bytes) changes every IC round. Two certificates issued at different rounds for the same `certified_data` both pass `verify_certified_data`, producing two distinct but valid `sig` byte strings for the same `(seed, msg)` pair: [8](#0-7) [9](#0-8) 

**`sender_info.sig` is verified against `sender_pubkey`:**

`verify_sender_info_canister_sig` verifies `sender_info.sig` using the envelope-level `sender_pubkey` (which encodes `(canister_id, seed)`). Both `sig_info_1` and `sig_info_2` pass this check independently: [10](#0-9) 

**No feature flag gates `sender_info`:**

There is no feature flag gating the `sender_info` code path. The feature is unconditionally active in the production validation path and is covered by integration tests: [11](#0-10) 

**Exploit flow:**

1. Attacker deploys canister `C`. `sender_pubkey` encodes `(C, seed)`. `sender` principal = `SHA224(DER(sender_pubkey)) || 0x02`.
2. `C` sets `certified_data` = root hash of a tree containing leaf at `["sig", SHA256(seed), SHA256(info_blob)]`.
3. At round T1, attacker fetches IC certificate `cert1` → constructs `sig_info_1 = CBOR{cert1, tree}`.
4. At round T2 (~1 second later), attacker fetches IC certificate `cert2` → constructs `sig_info_2 = CBOR{cert2, tree}`. Both pass `verify_sender_info_canister_sig`.
5. `MessageId1 = hash(..., {info, C, sig_info_1})` and `MessageId2 = hash(..., {info, C, sig_info_2})` are distinct.
6. `C` certifies `["sig", SHA256(seed), SHA256(MessageId1)]` and `["sig", SHA256(seed), SHA256(MessageId2)]` → valid envelope signatures for both.
7. Both messages are submitted. Both pass `validate_request_content`. Both have distinct `MessageId`s. Neither is in ingress history when the other is checked. Both are inducted and executed.

## Impact Explanation

Any application canister that relies on the IC's `MessageId`-based deduplication to prevent double-execution of a state-mutating call is vulnerable. An attacker who controls a signer canister can execute the same logical ingress call twice against any target canister that does not implement its own application-level deduplication. This can result in double-minting, double-crediting, double-claiming, or other double-execution of financial or state-mutating operations. This matches the allowed impact: **High ($2,000–$10,000) — unauthorized access to canister-controlled funds or significant platform-level security impact with concrete user harm**.

## Likelihood Explanation

The attacker only needs to deploy a canister on the IC (unprivileged, costs cycles) and wait approximately one IC round (~1 second) between obtaining the two certificates. No privileged access, key compromise, or threshold attack is required. The `sender_info` feature is unconditionally active in the production code path with no feature flag. The attack is repeatable for any target canister that lacks its own deduplication.

## Recommendation

Remove `sig` from the `MessageId` computation. Only `info` and `signer` carry logical message identity; `sig` is a proof of `info` and must not affect the hash. Change the nested map in `representation_independent_hash_call_or_query` to:

```rust
Map(btreemap! {
    "info"   => Bytes(info),
    "signer" => Bytes(signer),
    // sig intentionally excluded from MessageId
})
``` [1](#0-0) 

## Proof of Concept

```
1. Deploy canister C on the IC.
2. Have C set certified_data = root_hash(tree) where tree contains
   leaf at ["sig", SHA256(seed), SHA256(info_blob)] = "".
3. Fetch IC certificate cert1 at round T1 → build sig_info_1 = CBOR{cert1, tree}.
4. Wait one round (~1s). Fetch IC certificate cert2 at round T2 → build sig_info_2.
5. Assert sig_info_1 != sig_info_2 (different cert timestamps/BLS signatures).
6. Assert both pass ic_crypto_iccsa::verify(info_blob, sig_info_1, pk, root_pk) == Ok(())
   and ic_crypto_iccsa::verify(info_blob, sig_info_2, pk, root_pk) == Ok(()).
7. Compute MessageId1 and MessageId2 using representation_independent_hash_call_or_query
   with identical (canister_id, method, arg, expiry, sender) but different sig fields.
   Assert MessageId1 != MessageId2.
8. Have C certify ["sig", SHA256(seed), SHA256(MessageId1)] and
   ["sig", SHA256(seed), SHA256(MessageId2)] → valid envelope signatures for both.
9. Submit both messages to a target canister with a state-mutating method
   (e.g., a counter increment or token transfer without app-level dedup).
   Observe both are executed (counter incremented twice, tokens transferred twice).
```

fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
