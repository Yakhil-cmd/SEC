Audit Report

## Title
Ingress Pool Admission Checks Canister Cycles Per-Message, Not Cumulatively, Enabling Ingress Pool Exhaustion DoS - (File: rs/execution_environment/src/execution_environment.rs)

## Summary
The `should_accept_ingress_message` function performs a non-cumulative, per-message cycles balance check at ingress admission time. An unprivileged attacker can submit many messages to a canister whose balance covers only a single message; all pass the admission check and enter the pool, while the block-building selector rejects all but the first. The pool fills with unexecutable messages, and once the per-node quota is exhausted, the node returns HTTP 503 to all subsequent ingress submissions until the messages expire.

## Finding Description
`should_accept_ingress_message` in `rs/execution_environment/src/execution_environment.rs` (L3340–3373) explicitly acknowledges it is a "first-pass check" and calls `can_withdraw_cycles_with_threshold` with only `cost` — the cost of the single incoming message — against the canister's current balance snapshot from the latest certified state. It does not consult the ingress pool for pending cumulative cost already admitted for the same canister.

The "more rigorous check" deferred to block-building time is `validate_ingress` in `rs/ingress_manager/src/ingress_selector.rs` (L566–584). That function maintains a `cycles_needed: BTreeMap<CanisterId, Cycles>` accumulator and checks `*cumulative_ingress_cost + ingress_cost` before including any message in a block. Messages that pass the first check but fail the cumulative check are never included in any block yet remain in the pool until `MAX_INGRESS_TTL` (5 minutes, `rs/limits/src/lib.rs` L17).

The ingress throttler in `rs/http_endpoints/public/src/call.rs` (L229–236) calls `exceeds_threshold()` before the ingress filter runs. `exceeds_threshold` in `rs/artifact_pool/src/ingress_pool.rs` (L362–371) checks `exceeds_limit(&self.node_id)` — a per-originating-node count/byte limit. Once the local node's quota is exhausted, all subsequent ingress submissions to that node receive HTTP 503.

The certified state used by `should_accept_ingress_message` is obtained via `get_latest_certified_state()` (`rs/execution_environment/src/ingress_filter.rs` L60), which is updated once per consensus round (~1 second). All messages submitted within a single round see the same stale balance snapshot and all pass.

The default pool limits in `rs/config/src/artifact_pool.rs` (L38–39) are `usize::MAX`, meaning in default configuration the throttler never fires. In production deployments where finite limits are configured, the attack directly triggers HTTP 503. Even with unbounded limits, the pool accumulates unexecutable messages consuming memory and degrading performance.

## Impact Explanation
This is an application/platform-level DoS against individual replica nodes. An attacker can cause a targeted replica to return HTTP 503 to all ingress submissions for up to 5 minutes per attack wave, with the attack trivially repeatable. This matches the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS." The attack requires no special privileges, no victim interaction, and no governance or subnet-majority compromise.

## Likelihood Explanation
Any unprivileged user with access to the public HTTP endpoint can execute this attack. The attacker needs only to identify a canister whose balance satisfies `cost_per_message ≤ B < N × cost_per_message` — a condition observable via `canister_status`. Submitting N messages within one consensus round (~1 second) is trivially achievable with a simple script. The attack is repeatable every 5 minutes as the pool drains. No leaked keys, social engineering, or third-party compromise is required.

## Recommendation
`should_accept_ingress_message` should query the current ingress pool to compute the total pending induction cost already admitted for the paying canister and check `pending_cost + new_cost` against the canister's balance, mirroring the accumulator logic already present in `validate_ingress`. Alternatively, the ingress pool should enforce a per-canister message count or cost cap at admission time, or the admission check should apply a conservative headroom multiplier (e.g., reject if `balance < k × cost` for some small `k`). Production deployments should also configure finite `ingress_pool_max_count` and `ingress_pool_max_bytes` values rather than relying on the `usize::MAX` defaults.

## Proof of Concept
1. Identify canister `C` with balance `B` where `cost_per_message ≤ B < N × cost_per_message` for `N > 1` (observable via `canister_status`).
2. Within a single consensus round (~1 second), submit `N` distinct ingress messages (distinct nonces/expiry times) to `C` via `POST /api/v2/canister/{id}/call` on a single replica node.
3. Each message is evaluated by `should_accept_ingress_message` against the same certified state snapshot; each sees `B ≥ cost_per_message` and is admitted.
4. All `N` messages enter the ingress pool and are gossiped.
5. At block-building time, `validate_ingress` admits only the first message; messages 2..N remain in the pool until expiry (`MAX_INGRESS_TTL` = 5 minutes).
6. Once the pool's per-node count/byte limit is reached, `exceeds_threshold()` returns `true` and all subsequent ingress submissions to that node receive HTTP 503 until the pool drains.
7. A deterministic integration test can be written using `PocketIC` or the `StateMachine` test harness: configure a canister with minimal cycles, submit a burst of messages in one round, advance the state machine, and assert that `exceeds_threshold()` returns `true` and that only one message appears in the next block payload.
