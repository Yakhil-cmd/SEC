### Title
Negative `delta_gas` Not Applied to `gas_used` Causes User Overpayment of Gas Fees — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, ZKsync OS implements a dual-resource accounting model where both EVM gas (ergs) and a native proving resource are tracked. When the native resource cost exceeds the EVM gas cost (`delta_gas > 0`), extra gas is charged to the user. However, when the native resource cost is **lower** than the EVM gas cost (`delta_gas < 0`), the negative delta is silently discarded and the user is not refunded the difference. The codebase itself acknowledges this with a `// TODO: return delta_gas to gas_used?` comment. The excess payment flows to the operator rather than being returned to the user.

---

### Finding Description

`compute_gas_refund` in `refund_calculation.rs` computes `delta_gas` as:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native cost > EVM gas cost: charge extra gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The `delta_gas` value represents the difference between the gas-equivalent of native resource consumption (`native_used / native_per_gas`) and the EVM gas consumed (`gas_used`).

- When `delta_gas > 0`: the user is charged extra gas to cover native proving costs. ✓
- When `delta_gas < 0`: the native cost is lower than the EVM gas cost. The code does **nothing** — `gas_used` is not reduced, and the user is not refunded the difference.

The final refund to the user is `gas_price * (gas_limit - gas_used)`. When `delta_gas < 0`, `gas_used` is inflated relative to what the native resource cost would justify, so the user receives a smaller refund than they should. The operator receives the excess via `gas_price * gas_used`.

The `// TODO: return delta_gas to gas_used?` comment at line 80 is an explicit acknowledgment by the developers that this case is unresolved.

This entire block is gated by `#[cfg(not(feature = "unlimited_native"))]`, meaning it is active in production deployments where `unlimited_native` is not set.

---

### Impact Explanation

Users overpay gas fees by `gas_price * |delta_gas|` tokens per transaction where `delta_gas < 0`. The excess is transferred to the operator (coinbase). This is a direct, quantifiable financial loss for users. The loss scales with:
- The magnitude of `|delta_gas|`
- The gas price of the transaction

For a transaction with `gas_price = P`, `gas_used = G`, and `native_used / native_per_gas = N` where `N < G`, the user overpays by `P * (G - N)` tokens.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever `native_used / native_per_gas < gas_used`, i.e., when the native resource consumption is lower than the EVM gas consumption. This is a common scenario:

1. **High priority fee transactions**: A user who sets a high `max_priority_fee_per_gas` to ensure inclusion gets a high `native_per_gas = gas_price / native_price`. Their native resource limit is large, but if the transaction doesn't consume much native resource, `native_used` is small, making `native_used / native_per_gas` much smaller than `gas_used`.

2. **EVM-heavy, proving-light transactions**: Transactions that execute many EVM opcodes (high `gas_used`) but require relatively little proving work (low `native_used`).

Both scenarios are routine in normal operation. The `unlimited_native` feature flag is not set in production, so the vulnerable code path is active.

---

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` to refund the user for the excess payment:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native cost is lower than EVM gas cost; refund the difference
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Ensure gas_used does not fall below minimal_gas_used (already applied above)
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
// Remove the TODO comment
```

This ensures the user pays `max(minimal_gas_used, max(gas_used_from_ergs, native_used/native_per_gas))` rather than always paying the full EVM gas consumed when native costs are lower.

### Proof of Concept

Consider:
- `gas_limit = 100_000`
- `gas_price = 5_000` (high priority fee)
- `native_price = 100`
- `native_per_gas = gas_price / native_price = 50`
- Transaction consumes `gas_used = 50_000` EVM gas
- Transaction consumes `native_used = 500_000` native units

Then:
- `native_used / native_per_gas = 500_000 / 50 = 10_000`
- `delta_gas = 10_000 - 50_000 = -40_000`

Current behavior: `gas_used` stays at `50_000`. User refund = `5_000 * (100_000 - 50_000) = 250_000_000` tokens.

Correct behavior: `gas_used` should be reduced to `10_000`. User refund = `5_000 * (100_000 - 10_000) = 450_000_000` tokens.
