### Title
Unbounded ingress-expiry loop in ICP Rosetta `construction_payloads` allows unauthenticated OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

### Summary
The `construction_payloads` handler in ICP Rosetta iterates from `ingress_start` to `ingress_end` in steps of ~120 seconds (in nanoseconds), pushing one `u64` per step into a `Vec`. Because `ingress_start` and `ingress_end` are taken directly from attacker-controlled request metadata with no validation, setting `ingress_start=0` and `ingress_end=u64::MAX` causes ~153 billion iterations and ~1.2 TB of heap allocation, crashing the Rosetta process with OOM.

### Finding Description
In `construction_payloads.rs` lines 99–107, the loop:

```rust
let mut now = ingress_start;   // attacker-controlled
while now < ingress_end {      // attacker-controlled
    ingress_expiries.push(...);
    now += interval;           // interval = 120_000_000_000 ns
}
```

has no upper-bound check on `(ingress_end - ingress_start) / interval`. The `ConstructionPayloadsRequestMetadata` struct accepts any `u64` for both fields and the `TryFrom` conversion performs only JSON deserialization, not semantic validation. [6](#0-5) 

### Impact Explanation
OOM crash of the Rosetta process. All users of that Rosetta instance lose access to ICP transaction construction (transfers, neuron management, etc.) until the process is restarted. A single HTTP request is sufficient; the attack can be repeated immediately after restart.

### Likelihood Explanation
The endpoint is public, requires no authentication, and the malicious values are trivially expressible in JSON. The ICRC1 Rosetta sibling already has the fix pattern; the ICP Rosetta path was simply never hardened.

### Recommendation
Before the loop, validate that the window is within a sane maximum, e.g.:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // 24h
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    > MAX_INGRESS_WINDOW.as_nanos() as u64
{
    return Err(ApiError::invalid_request("ingress window too large"));
}
```

Mirror the guard already present in the ICRC1 Rosetta implementation. [5](#0-4)
