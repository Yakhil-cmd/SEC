### Title
Unchecked Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Fund Freeze — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` burns a user's ERC-20 tokens **before** calling the `ExitToNear` or `ExitToEthereum` precompile, and never checks whether that precompile call succeeded. If the precompile returns a failure (e.g., due to an invalid NEAR recipient account ID), the ERC-20 tokens are permanently destroyed while no NEP-141 or ETH transfer is ever initiated, freezing the user's funds forever.

---

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` follow the same unsafe pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure is ignored
    }
}
``` [1](#0-0) 

The `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) validates the recipient as a NEAR account ID and returns `ExitError` for invalid inputs (e.g., non-UTF-8 bytes, account IDs exceeding 64 characters, or illegal characters). [2](#0-1) 

When the precompile fails, the EVM-level `call` returns `0` into `res`, but the Solidity function does not revert. The `_burn` has already executed, so the ERC-20 supply is permanently reduced with no corresponding NEP-141 transfer ever created. [1](#0-0) 

The identical unchecked pattern exists in `withdrawToEthereum`, calling the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`. [3](#0-2) 

The `ExitToNear` precompile explicitly rejects invalid NEAR account IDs with `ERR_INVALID_RECEIVER_ACCOUNT_ID` and other errors, all of which propagate as `ExitError` and cause the EVM call to return `0`. [2](#0-1) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 tokens are burned and gone from Aurora's EVM state.
- No NEAR promise is created, so no NEP-141 tokens are ever transferred to the recipient.
- The NEP-141 tokens remain locked in the ETH connector contract indefinitely with no recovery path.
- There is no refund mechanism in `EvmErc20.sol` itself; the `error_refund` feature in the precompile only handles failures of the *downstream NEAR promise*, not failures of the precompile call itself. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The `recipient` parameter in `withdrawToNear` is a raw `bytes` value supplied by the caller. Any caller who passes bytes that do not form a valid NEAR account ID (e.g., bytes containing `@`, exceeding 64 characters, or non-ASCII bytes) will trigger the failure. This can occur through:

1. Direct user error when calling `withdrawToNear` with a malformed recipient.
2. A smart contract on Aurora that constructs the recipient programmatically with a bug.
3. An attacker who deliberately crafts an invalid recipient to grief another user if the function is called on their behalf through a wrapper contract.

The `withdrawToEthereum` path is lower likelihood because Ethereum addresses are always exactly 20 bytes and structurally valid, but the same unchecked pattern exists. [3](#0-2) 

---

### Recommendation

Validate the precompile return value and revert if it indicates failure. The fix should be applied to both `withdrawToNear` and `withdrawToEthereum`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, validate the `recipient` bytes as a well-formed NEAR account ID in Solidity before burning, so the burn is never reached with an invalid recipient.

---

### Proof of Concept

1. Deploy `EvmErc20` (as Aurora does for any bridged NEP-141 token).
2. Mint tokens to `alice` via the admin (simulating a normal bridge-in).
3. Alice calls `withdrawToNear(invalidRecipient, amount)` where `invalidRecipient` contains bytes that fail NEAR account ID validation (e.g., `bytes("invalid@account!!!")`).
4. `_burn(alice, amount)` executes — Alice's ERC-20 balance drops to zero.
5. The precompile call returns `0` (failure) because `parse_recipient` rejects the invalid account ID.
6. The function returns normally without reverting.
7. Alice has zero ERC-20 tokens and receives zero NEP-141 tokens. Funds are permanently frozen. [1](#0-0) [2](#0-1)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
