### Title
Unbounded `active_assembles` / `artifact_processor_tasks` growth via `SLOT_TABLE_NO_LIMIT` allows single Byzantine subnet peer to OOM a replica — (`rs/p2p/consensus_manager/src/receiver.rs` + `rs/replica/setup_ic_network/src/lib.rs`)

### Summary
A single authenticated subnet peer advertising N distinct artifact IDs across N distinct slot numbers for any artifact type configured with `SLOT_TABLE_NO_LIMIT` (`usize::MAX`) causes `active_assembles` and `artifact_processor_tasks` to grow without bound, exhausting heap memory and crashing the replica process.

### Finding Description
`SLOT_TABLE_NO_LIMIT` is defined as `usize::MAX` and is assigned as the `slot_limit` for consensus, certification, DKG, IDKG, and HTTPS-outcalls artifact managers. The only per-peer admission guard in `handle_slot_update_receive` is:

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { ... }
```

With `slot_limit = usize::MAX`, this guard never fires. For every new `(slot_number, artifact_id)` pair where the artifact ID has not been seen before, the code:
1. Inserts a new `SlotEntry` into `slot_table[peer_id]`
2. Inserts a new `watch::Sender` into `active_assembles`
3. Spawns a new async task into `artifact_processor_tasks`

There is no secondary cap on `active_assembles.len()` or `artifact_processor_tasks.len()`. The `handle_topology_update` path only cleans up entries when a peer *leaves* the subnet topology — a still-connected Byzantine peer is never pruned.

### Impact Explanation
Each spawned `process_slot_update` task allocates stack, a `watch` channel pair, and holds a clone of the `artifact_assembler`. Sending O(millions) of slot updates from one peer exhausts heap memory, causing an OOM kill of the replica process. A crashed replica drops out of consensus participation. If an attacker controls `f` nodes (below the Byzantine fault threshold `f < n/3`), they can crash all honest replicas simultaneously, halting subnet liveness.

### Likelihood Explanation
The attacker must be a valid subnet member (TLS-authenticated via QUIC transport). A single compromised node — well below the fault threshold — is sufficient. The attack requires only sending crafted P2P slot-update messages in a tight loop, which is trivially scriptable. No cryptographic material beyond the node's own TLS key is needed.

### Recommendation
Replace `SLOT_TABLE_NO_LIMIT` with a concrete, protocol-justified per-peer cap for each artifact type (analogous to `SLOT_TABLE_LIMIT_INGRESS = 50_000`). The correct bound is the maximum number of in-flight artifacts a single honest peer would legitimately advertise simultaneously for each artifact type (e.g., bounded by the consensus round window, DKG transcript count, etc.). Alternatively, add a global cap on `active_assembles.len()` with back-pressure or eviction.

### Proof of Concept
```rust
// Unit test sketch (no external infra needed):
let (mut mgr, _) = ReceiverManagerBuilder::new()
    .with_slot_limit(usize::MAX)  // matches SLOT_TABLE_NO_LIMIT
    .build();
let cancellation = CancellationToken::new();
let n = 1_000_000usize;
for i in 0..n {
    mgr.handle_slot_update_receive(
        SlotUpdate {
            slot_number: SlotNumber::from(i as u64),
            commit_id: CommitId::from(1),
            update: Update::Id(i as u64),  // distinct ID each time
        },
        NODE_1,
        ConnId::from(1),
        cancellation.clone(),
    );
}
// Both grow to n with no bound:
assert_eq!(mgr.active_assembles.len(), n);
assert_eq!(mgr.artifact_processor_tasks.len(), n);
// In practice, process OOMs long before n reaches usize::MAX.
```
