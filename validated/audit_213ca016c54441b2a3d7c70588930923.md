### Title
Unprotected Event Hook Triggering Allows Arbitrary Settlement Layer Chain ID Manipulation and Interop Root Injection - (File: `system_hooks/src/event_hooks/system_context.rs`, `system_hooks/src/event_hooks/interop_root_reporter.rs`)

### Summary
ZKsync OS event hooks (`system_context_event_hook` and `interop_root_reporter_event_hook`) perform critical system state mutations — updating the settlement layer chain ID and injecting interop roots — based solely on the topic signature of an emitted event, with no validation of who triggered the event emission. Unlike call hooks which explicitly check the caller address, event hooks accept fully attacker-controlled event parameters and apply them to system state without any origin check.

### Finding Description

ZKsync OS has two categories of system hooks:

**Call hooks** (e.g., `l1_messenger_hook`, `set_bytecode_on_address_hook`, `mint_base_token_hook`) all perform explicit caller validation:

```rust
// l1_messenger_hook
if caller != L1_MESSENGER_ADDRESS {
    // Pretend to be an empty account
    return Ok((make_return_state_from_returndata_region(available_resources, &[]), return_memory));
}
``` [1](#0-0) 

**Event hooks**, by contrast, have no such protection. The `system_context_event_hook` registered at `SYSTEM_CONTEXT_ADDRESS_LOW` (`0x800b`) and `interop_root_reporter_event_hook` registered at `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW` (`0x10008`) both ignore the `caller_ee` parameter and perform no caller address check:

### Impact Explanation

**Settlement layer chain ID manipulation**: Any unprivileged transaction that causes the SystemContext contract (`0x800b`) to emit `SettlementLayerChainIdUpdated(uint256)` with an attacker-chosen value will cause `update_settlement_layer_chain_id` to be called with that value. The settlement layer chain ID is a critical batch output field used in proving and cross-chain settlement verification. [7](#0-6) 

**Interop root injection**: Any unprivileged transaction that causes the InteropRootStorage contract (`0x10008`) to emit `InteropRootAdded(uint256,uint256,bytes32[])` with attacker-chosen `chain_id`, `block_or_batch_number`, and `root` will inject a fake interop root into the system's interop root storage. This root is included in the batch's `interop_roots_rolling_hash` commitment, potentially enabling fraudulent cross-chain message acceptance. [8](#0-7) 

### Likelihood Explanation

The attack requires the SystemContext (`0x800b`) or InteropRootStorage (`0x10008`) contracts to expose a publicly callable function that emits the relevant event. The event hooks are registered unconditionally at those addresses: [9](#0-8) 

Unlike call hooks which enforce defense-in-depth at the hook level regardless of the underlying contract's access control, event hooks provide zero additional protection. Any access control gap in the underlying contracts — including functions callable by any EOA or contract — directly translates to a system-level state manipulation. The `interop_root_reporter_event_hook` is particularly exposed because it accepts a fully arbitrary `InteropRoot` struct with no validation of the `root`, `chain_id`, or `block_or_batch_number` values beyond structural checks.

### Recommendation

1. Add a caller address parameter to the `SystemEventHook` function signature so event hooks can validate the origin of the event emission, mirroring the defense-in-depth pattern used by call hooks.
2. In `system_context_event_hook`, validate that the emitting call originated from a trusted source (e.g., the bootloader or a governance-controlled address).
3. In `interop_root_reporter_event_hook`, validate that the emitting call originated from the authorized interop center contract (`L2_INTEROP_CENTER_ADDRESS` at `0x1000d`).
4. Add validation of `chain_id` and `root` values in `interop_root_reporter_event_hook` (e.g., reject zero root, reject zero chain_id).

### Proof of Concept

1. Deploy or use the existing contract at `0x10008` (InteropRootStorage).
2. If the contract exposes any publicly callable function that emits `InteropRootAdded(uint256,uint256,bytes32[])`, call it with:
   - `topics[1]` = attacker-chosen `chain_id`
   - `topics[2]` = attacker-chosen `block_or_batch_number`
   - `data[64:96]` = attacker-chosen `root`
3. The `interop_root_reporter_event_hook` fires (no caller check), parses the attacker-controlled values, and calls `system.io.add_interop_root(...)` with the fake root.
4. The fake interop root is included in the batch's `interop_roots_rolling_hash`, causing the prover to commit to a batch containing fraudulent cross-chain state.

The same pattern applies to `system_context_event_hook` at `0x800b` for settlement layer chain ID manipulation.

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())

    pub fn emit_event(
        &mut self,
        hooks: &mut HooksStorage<S, S::Allocator>,
        ee_type: ExecutionEnvironmentType,
        resources: &mut S::Resources,
        address: &<S::IOTypes as SystemIOTypesConfig>::Address,

        data: &[u8],

        // First, emit the event using io subsystem

            .emit_event(ee_type, resources, address, topics, data)?;

        // If successful, intercept event hook, if any
        if let Some(address_low) = address.try_into_low() {
            let _ = hooks.try_intercept_event(
                address_low,
                topics,
                data,
                ee_type as u8,
                self,
                resources,
            )?;
        }
        Ok(())
    }

/// Reads SL chain id from the SystemContext(0x800b) contract.
///
pub fn read_settlement_layer_chain_id<IO: IOSubsystem>(io: &mut IO) -> U256
where
    IO::IOTypes: SystemIOTypesConfig<Address = B160, StorageKey = Bytes32, StorageValue = Bytes32>,
{
    // This helper is intentionally generic over the IO subsystem so it can be
    // reused from bootloader transaction flow code that only has access to
    // `System<S>::io`.
    const SL_CHAIN_ID_STORAGE_SLOT: Bytes32 = Bytes32::ZERO;
    let mut inf_resources = IO::Resources::FORMAL_INFINITE;
    let chain_id = io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &SYSTEM_CONTEXT_ADDRESS,
            &SL_CHAIN_ID_STORAGE_SLOT,
        )
        .expect("must read SystemContext SL chain id");
    U256::from_be_bytes(chain_id.as_u8_array())
}
