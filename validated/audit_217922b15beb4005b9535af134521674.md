### Title
Missing `y_parity ∈ {0,1}` validation in `AuthorizationEntry::decode_list_body` allows silent delegation skip — (`basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/eip_7702_tx.rs`)

### Summary
`AuthorizationEntry::decode_list_body` parses `y_parity` with `r.u8()`, which accepts any value `0..=255`. Values `2..=255` pass the RLP decoder, reach `recover_authority`, which returns `Ok(None)` for `parity > 1`, causing `validate_and_apply_delegation` to silently return `Ok(false)`. The transaction sender (who controls the raw RLP bytes of the authorization list) can flip any authority's `y_parity` from `0` or `1` to `2`, causing that delegation to be skipped without any error.

### Finding Description
The `y_parity` field in an EIP-7702 authorization entry is the ECDSA recovery parameter and must be `0` or `1`. The outer transaction signature uses `r.bool()` (which enforces this), but `AuthorizationEntry` uses `r.u8()` (which does not). The check `parity > 1 → return Ok(None)` in `recover_authority` is a runtime guard, not a parse-time rejection. Because `validate_and_apply_delegation` treats `Ok(false)` as a soft skip (not an error), and the caller loop has a no-op `if !success {}` branch, the transaction completes successfully with the delegation omitted.

The `y_parity` field is NOT covered by the authority's signature — the signed message is `keccak(0x05 || rlp([chain_id, address, nonce]))`. [7](#0-6)  The transaction sender can freely mutate `y_parity` in the submitted RLP without invalidating the authority's signature.

### Impact Explanation
An unprivileged transaction sender who includes a third party's signed authorization in an EIP-7702 transaction can change `y_parity` from `0`/`1` to `2` in the raw RLP. The delegation is silently skipped: the authority's code is not updated, but the authority's nonce is also not incremented (since the delegation was skipped before step 9). The circuit proves this execution as valid because the execution path (`recover_authority → None → Ok(false) → skip`) is a legitimate code path. The attacker can repeatedly grief any pending delegation.

### Likelihood Explanation
EIP-7702 transactions are permissionless. Any transaction sender can include any signed authorization in their transaction. Modifying a single byte of `y_parity` in the RLP requires no special knowledge beyond the raw transaction bytes. The attack is trivially repeatable.

### Recommendation
Replace `r.u8()?` with `r.bool()?` in `AuthorizationEntry::decode_list_body`:

```rust
// Before:
let y_parity = r.u8()?;

// After:
let y_parity = r.bool()? as u8;
```

This mirrors the validation already applied to the outer transaction signature in `EIP2718SignatureData::decode_list_body` and rejects `y_parity > 1` at parse time with `InvalidTransaction::InvalidStructure`, consistent with the rest of the RLP validation layer.

### Proof of Concept
Craft an `AuthorizationEntry` RLP with `y_parity = 0x02` (raw byte `0x02` in the RLP stream, which `r.u8()` accepts as `Ok(2)`). Submit an EIP-7702 transaction containing this entry alongside a valid authority signature for `y_parity = 0`. Observe that `decode_list_body` succeeds, `recover_authority` returns `Ok(None)`, `validate_and_apply_delegation` returns `Ok(false)`, and the delegation is not applied. Assert that `AuthorizationEntry::decode_list_body` should return `Err(InvalidTransaction::InvalidStructure)` for any `y_parity` byte outside `{0x80 (→0), 0x01 (→1)}`.
