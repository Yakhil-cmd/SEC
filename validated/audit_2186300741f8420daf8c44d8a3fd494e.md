### Title
Panic in `canister_status` via `.unwrap()` on Management Canister Rejection Permanently Leaks Slot Count, Enabling DoS of NNS Root — (`rs/nns/handlers/root/impl/canister/canister.rs`)

---

### Summary

The public `canister_status` update method on the NNS root canister calls `.unwrap()` on the result of a management canister call. When the management canister rejects (e.g., the target canister does not exist or root is not a controller), the `.unwrap()` panics, trapping the callback. The IC execution model rolls back all heap changes made during the callback, including the `SlotLoan::drop` that was supposed to return the borrowed slot. This permanently leaks one slot per failed call. After 167 such calls the shared `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` reaches zero, at which point `try_borrow_slot` rejects **all** callers — including privileged NNS governance — blocking every management-canister-dependent operation on root.

### Finding Description

**Entrypoint — no access control:**

```rust

        .await

    canister_status_response.unwrap()   // ← panics on Err

``` [1](#0-0) 

Any principal may call this method; there is no `check_caller_is_*` guard.

**Slot borrow — decrements heap-resident counter before the await:**

```rust
async fn canister_status(&self, ...) -> Result<...> {
    let _loan = self.try_borrow_slot()?;          // decrements AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT
    self.inner.canister_status(canister_id_record).await
}
``` [2](#0-1) 

The decrement happens in the **first** message execution (before the inter-canister call). The IC commits this heap state at the `await` suspension point.

**Slot return — happens in the callback, before the panic, but is rolled back:**

`SlotLoan::drop` increments the counter:

```rust
impl Drop for SlotLoan {
    fn drop(&mut self) {
        self.available_slot_count.with_borrow_mut(|c| {
            *c = c.saturating_add(self.used_slot_count);
        });
    }
}
``` [3](#0-2) 

This drop runs in the **callback** message execution (after the management canister responds). The outer `canister_status` then calls `.unwrap()` on the `Err`, which panics. The IC traps the callback and **rolls back all heap changes made during that callback**, including the `SlotLoan::drop` increment. The net result is the counter stays at its decremented value.

**Zero-slot guard blocks VIP callers too:**

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };
    self.available_slot_count.with_borrow_mut(|available_slot_count| {
        if *available_slot_count == 0 {          // ← checked before VIP branch
            return Err((code, message));
        }
        *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
        Ok(())
    })?;
``` [4](#0-3) 

The `== 0` guard fires regardless of `is_caller_vip`. Once the counter reaches zero, NNS governance calls to `change_nns_canister`, `add_nns_canister`, `stop_or_start_nns_canister`, `update_canister_settings`, `create_canister_and_install_code`, `take_canister_snapshot`, and `load_canister_snapshot` all fail with a `SysTransient` rejection.

### Impact Explanation

An attacker making 167 ingress calls to `canister_status` with a non-existent canister ID exhausts the slot counter. After that:

- All management-canister-dependent NNS root operations fail for every caller, including NNS governance.
- NNS cannot upgrade canisters, change controllers, update settings, or perform any operation that routes through root's management canister client.
- The counter resets only on canister upgrade (which itself requires governance to function, creating a potential deadlock depending on upgrade path).

---

### Likelihood Explanation

The attack requires exactly 167 cheap ingress update calls to a public, unauthenticated endpoint. No privileged access, no key material, no social engineering. The management canister reliably rejects `canister_status` for non-existent canister IDs with a deterministic error, making the trigger 100% reproducible.

---

### Recommendation

Replace the `.unwrap()` with graceful error propagation. Change the return type to `Result<CanisterStatusResult, String>` (or a Candid-compatible error variant) and return the error to the caller:

```rust
#[update]
async fn canister_status(canister_id_record: CanisterIdRecord) -> Result<CanisterStatusResult, String> {
    let client = new_management_canister_client();
    client
        .canister_status(canister_id_record)
        .await
        .map(CanisterStatusResult::from)
        .map_err(|(_, msg)| msg)
}
```

This ensures the callback returns normally (no trap), so the IC does **not** roll back the `SlotLoan::drop`, and the slot counter is correctly restored on every code path.

Additionally, the `try_borrow_slot` zero-check should be conditioned on `!self.is_caller_vip` so that NNS-internal callers are never blocked even if the counter is exhausted by external abuse.

### Proof of Concept

State-machine test outline:

1. Install the NNS root canister.
2. Generate a canister ID that does not exist on the subnet.
3. Call `canister_status` on root 167 times with that ID from an anonymous principal; assert each call traps (reject response).
4. Now call `canister_status` with a valid NNS canister ID that root controls; assert the call is rejected with "Unavailable" (slot count = 0).
5. Optionally: call `change_nns_canister` from the governance principal; assert it also fails with `SysTransient`, confirming governance operations are blocked.
