Audit Report

## Title
Unprivileged Caller Can Force Participation Registration for Any Buyer via Missing Caller-Equals-Buyer Check - (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update method in the SNS Swap canister accepts an arbitrary `buyer` principal from the request body without verifying that the caller is that buyer. Any unprivileged ingress sender can trigger participation registration for a victim who has already transferred ICP and holds an open ticket, deleting the ticket and permanently preventing the victim from canceling their participation via `notify_payment_failure`.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 127–143, the `refresh_buyer_tokens` handler resolves the target principal from `arg.buyer` without any `caller == buyer` check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no authorization check
};
```

This dispatches to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` (lines 1134–1312) with the attacker-supplied principal. Inside that function, at lines 1248–1272, if the victim's subaccount balance satisfies the open ticket amount, the ticket is unconditionally deleted and participation is registered:

```rust
memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow_mut().remove(&principal));
```

The `validate_confirmation_text` check at line 1150 provides no protection because the confirmation text is a public value set at SNS initialization and readable by anyone.

The only mechanism for a user to voluntarily cancel their pending participation is `notify_payment_failure` (lines 1320–1343), which operates on `caller_principal_id()`. Once the attacker's call deletes the victim's ticket, `notify_payment_failure` returns `None` and cancellation is impossible.

## Impact Explanation
This matches the allowed High impact: "Significant SNS security impact with concrete user or protocol harm." A victim who has transferred ICP and holds an open ticket has their participation irrevocably registered without consent. If the swap commits, the victim's ICP is swept to SNS governance and they receive unwanted SNS tokens with no recovery path via `error_refund_icp` (which only refunds unregistered ICP). The harm is conditional on swap commitment, but the loss of the cancellation right is unconditional and immediate.

## Likelihood Explanation
Preconditions are routinely met during any active SNS swap: the victim's principal is public on-chain, the victim transfers ICP before calling `refresh_buyer_tokens` as part of the normal participation flow, and the confirmation text (if any) is publicly visible in the SNS init config. No cycles, tokens, or special permissions are required. The attack window is the entire OPEN phase and does not require front-running.

## Recommendation
Add a caller-equals-buyer authorization check in the canister endpoint before dispatching:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Only the buyer can refresh their own tokens");
        }
        buyer
    };
    ...
}
```

Alternatively, remove the `buyer` field entirely and always use `caller_principal_id()`.

## Proof of Concept
1. User A calls `new_sale_ticket` to create a ticket for amount X ICP.
2. User A transfers X ICP to their swap subaccount on the ICP ledger.
3. User A intends to cancel by calling `notify_payment_failure`.
4. Attacker calls `refresh_buyer_tokens` with `buyer = <User_A_principal>` (and correct `confirmation_text` if required, which is public).
5. `refresh_buyer_token_e8s` reads User A's subaccount balance = X ≥ ticket amount X → ticket deleted at line 1270, participation registered at line 1288.
6. User A calls `notify_payment_failure` → returns `NotifyPaymentFailureResponse { ticket: None }` (ticket already gone); cancellation is impossible.
7. Swap commits → User A's ICP is swept to SNS governance; User A receives unwanted SNS tokens and cannot recover ICP via `error_refund_icp`.

A deterministic integration test using PocketIC can reproduce this by: (a) opening a swap, (b) having a test identity transfer ICP and create a ticket, (c) calling `refresh_buyer_tokens` from a different identity with the first identity's principal as `buyer`, and (d) asserting that `notify_payment_failure` returns `None` for the first identity.
