### Title
Missing Caller Access Control on `deploy_erc20_token` Allows Unauthorized ERC-20 Token Deployment — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `deploy_erc20_token` function in `engine/src/contract_methods/connector.rs` performs no caller authorization check. Any NEAR account can invoke it to register an ERC-20 token mapped to an arbitrary NEP-141 token. The code's own comment explicitly states the function "could be executed by the owner of the contract only," but no such restriction is enforced — a direct analog to the keychain missing-access-control class.

---

### Finding Description

`deploy_erc20_token` only calls `require_running` (a pause-state check) and then proceeds to deploy an ERC-20 contract and write a NEP-141↔ERC-20 mapping into the engine's storage: [1](#0-0) 

The function contains no `require_owner_only`, no `assert_private_call`, and no whitelist check. The comment at line 148–149 explicitly acknowledges the intended restriction:

```
// Safe because these promises are read-only calls to the main engine contract
// and this transaction could be executed by the owner of the contract only.
```

but the code does not enforce it. [2](#0-1) 

Compare with the analogous `mirror_erc20_token` function, which correctly enforces `require_owner_only` before performing a similar registry write: [3](#0-2) 

The `require_owner_only` helper is defined and used consistently across all other sensitive admin operations: [4](#0-3) 

The public WASM entrypoint exposes `deploy_erc20_token` with no additional guard: [5](#0-4) 

---

### Impact Explanation

The engine's `ft_on_transfer` callback routes incoming NEP-141 tokens to the ERC-20 address stored in the registry for that NEP-141: [6](#0-5) 

An attacker who calls `deploy_erc20_token` for a NEP-141 token that has not yet been registered can pre-populate the registry mapping before the authorized operator does (front-running). If the engine's internal `deploy_erc20_token` logic permits overwriting an existing mapping (i.e., does not revert on a duplicate NEP-141 key), an attacker can replace the ERC-20 address for an already-live token. All subsequent `ft_on_transfer` calls for that NEP-141 would mint into the new ERC-20 contract, while existing holders of the original ERC-20 tokens would be unable to bridge back — a **permanent fund freeze** for those holders.

Even in the non-overwrite case, unauthorized pre-registration prevents the legitimate operator from ever deploying the canonical ERC-20 for that NEP-141 (the slot is already taken), which also constitutes a permanent freeze of the bridging path for that token.

**Impact**: Permanent freezing of bridged funds (High–Critical).

---

### Likelihood Explanation

The attack requires no special privilege — any NEAR account can call `deploy_erc20_token` directly. The attacker only needs to know the `AccountId` of a NEP-141 token that has not yet been registered (or is about to be registered). NEP-141 token launches are public on-chain events, making front-running straightforward. Likelihood: **Medium–High**.

---

### Recommendation

Add `require_owner_only` (or an equivalent authorized-caller check) at the top of `deploy_erc20_token`, consistent with the comment's stated intent and with the pattern used in `mirror_erc20_token` and all other sensitive admin functions:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
+       require_owner_only(&state, &env.predecessor_account_id())?;
        ...
    })
}
```

---

### Proof of Concept

1. Attacker observes that NEP-141 token `token.example.near` has not yet been registered in the Aurora engine.
2. Attacker calls the Aurora engine's `deploy_erc20_token` WASM entrypoint with `token.example.near` as the argument — no deposit, no special key required.
3. The engine executes `deploy_erc20_token` without any caller check, writes the NEP-141→ERC-20 mapping, and deploys the ERC-20 contract.
4. The legitimate operator's subsequent call to `deploy_erc20_token` for the same NEP-141 either fails (slot taken) or overwrites the mapping.
5. In the overwrite case: existing ERC-20 holders cannot bridge back; their tokens are permanently frozen.

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
