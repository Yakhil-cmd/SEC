Audit Report

## Title
Minter Canister ID Accepted as Deposit Beneficiary Causes Permanent ETH Lock — (`rs/ethereum/cketh/minter/src/eth_logs/mod.rs`, `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

`parse_principal_from_slice` only rejects the management canister and the anonymous principal, allowing any opaque canister ID — including the ckETH minter's own principal — to be encoded as a deposit beneficiary. When the minter attempts to mint ckETH to itself, the ICRC1 ledger interprets the call as a burn from the minting account (which has zero balance) and returns `InsufficientFunds`. The minter defuses the quarantine guard on this error path, leaving the event permanently in `events_to_mint` and retrying forever. The deposited ETH is already in the minter's Ethereum address and cannot be returned.

## Finding Description

**Step 1 — `parse_principal_from_slice` accepts canister IDs.**
The function at `rs/ethereum/cketh/minter/src/eth_logs/mod.rs:260-292` only rejects `num_bytes == 0` (management canister) and `principal_bytes == [4]` (anonymous). Any other valid principal, including the 10-byte opaque canister ID of the minter itself (`sv3dd-oaaaa-aaaar-qacoa-cai`), passes all checks and is returned as `Ok(Principal)`.

**Step 2 — No beneficiary validation in `register_deposit_events`.**
`rs/ethereum/cketh/minter/src/deposit.rs:311-345` only checks whether the Ethereum *sender* is on the blocklist. There is no check that the decoded beneficiary is not the minter canister or the ledger canister. A valid deposit event with `beneficiary = minter_id` is unconditionally passed to `process_event` and placed in `events_to_mint`.

**Step 3 — `mint()` calls the ledger with `to = minting_account`.**
At `rs/ethereum/cketh/minter/src/deposit.rs:73-81`, the minter calls `icrc1_transfer { to: event.beneficiary(), ... }`. When the beneficiary is `Account { owner: minter_id, subaccount: None }`, this equals the ckETH ledger's `minting_account`.

**Step 4 — ICRC1 ledger takes the burn path and fails.**
`rs/ledger_suite/icrc1/ledger/src/main.rs:606` checks `if &to == ledger.minting_account()` first. Since `from_account` is also the minting account (the minter canister is the caller), the ledger attempts a burn from the minting account. The minting account has no balance entry in the store. `rs/ledger_suite/common/ledger_core/src/balances.rs:165-172` returns `Err(InsufficientFunds { balance: 0 })` when `prev == None`. The pre-flight check at `rs/ledger_suite/icrc1/ledger/src/main.rs:612-621` does not catch this because `min_burn_amount = transfer_fee.min(0) = 0` and `amount > 0`, so both guards pass. The error surfaces only at `apply_transaction` time (`rs/ledger_suite/icrc1/ledger/src/main.rs:669`), returning `CoreTransferError::InsufficientFunds`.

**Step 5 — The event is never quarantined; the minter retries forever.**
At `rs/ethereum/cketh/minter/src/deposit.rs:43-52`, a `scopeguard` is set up to quarantine the event *only if the function panics*. On `Ok(Err(err))` (line 85-90), the guard is explicitly defused with `ScopeGuard::into_inner`, the event remains in `events_to_mint`, and `error_count` is incremented. At line 131-136, a non-zero `error_count` causes the minter to reschedule `mint()` after `MINT_RETRY_DELAY`. Because the ledger will always reject this transfer, the event is permanently stuck and the ETH is permanently locked.

## Impact Explanation

This is a **High** severity finding. An unprivileged attacker can permanently lock any amount of ETH deposited to the ckETH minter by encoding the minter canister ID as the `bytes32` beneficiary in the Ethereum helper contract call. The deposited ETH is irrecoverable without a canister upgrade that explicitly quarantines or handles this case. This constitutes a concrete, permanent loss of in-scope chain-key/ck-token assets and a significant Chain Fusion security impact with direct user harm.

## Likelihood Explanation

The minter canister ID (`sv3dd-oaaaa-aaaar-qacoa-cai`) is public. Encoding it as a `bytes32` principal follows the documented scheme: `[10, b0..b9, 0..0]`. The attack requires only a standard Ethereum transaction to the helper contract — no privileged access, no special tooling. It is trivially repeatable and can be triggered by either a deliberate attacker or an innocent user who accidentally encodes the wrong principal.

## Recommendation

1. In `parse_principal_from_slice` (`rs/ethereum/cketh/minter/src/eth_logs/mod.rs:260`), add explicit rejection of the minter canister ID. Since the minter ID is known at runtime via `ic_cdk::id()`, the check can be passed as a parameter or closure.
2. Alternatively, add a guard in `mint()` or `register_deposit_events()` that checks `event.beneficiary().owner != ic_cdk::id()` and quarantines or marks the event as `InvalidDeposit` before attempting the ledger call.
3. Add a unit test asserting that `parse_principal_from_slice` rejects the minter canister ID bytes.

## Proof of Concept

1. Obtain the raw bytes of `sv3dd-oaaaa-aaaar-qacoa-cai` (10-byte opaque canister principal).
2. Encode as `bytes32`: `[0x0A, b0, b1, ..., b9, 0x00, ..., 0x00]`.
3. Call `depositEth(encoded_principal, 0x0)` on the Ethereum helper contract with any ETH amount.
4. The minter scrapes the log; `parse_principal_from_slice` returns `Ok(minter_principal)`.
5. `register_deposit_events` records the event in `events_to_mint`.
6. `mint()` calls `icrc1_transfer { to: Account { owner: minter_id, subaccount: None }, amount: X }`.
7. The ICRC1 ledger at `rs/ledger_suite/icrc1/ledger/src/main.rs:606` takes the burn path; `apply_transaction` at line 669 calls `balances.burn(minting_account, X)`, which calls `debit`, which finds no balance entry and returns `InsufficientFunds { balance: 0 }`.
8. The minter's `Ok(Err(err))` branch at `rs/ethereum/cketh/minter/src/deposit.rs:85` defuses the quarantine guard and reschedules. The ETH is permanently locked.

A deterministic integration test using PocketIC can reproduce this by: initializing the ckETH minter and ledger, submitting a mock Ethereum log with the minter canister ID encoded as the beneficiary, triggering `mint()`, and asserting that the event remains in `events_to_mint` after repeated retries.
