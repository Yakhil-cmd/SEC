### Title
`#[payable]` Functions Never Consume `attached_deposit`, Permanently Freezing NEAR on XCC Router - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

Two functions in the XCC router contract — `execute_scheduled` and `deploy_upgrade` — are annotated `#[payable]`, allowing callers to attach NEAR tokens. Neither function ever reads or uses `env::attached_deposit()`. Any NEAR attached to a successful call is silently absorbed into the router contract's balance with no recovery path, permanently freezing those funds.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, both `execute_scheduled` (line 149) and `deploy_upgrade` (line 160) carry the `#[payable]` attribute: [1](#0-0) [2](#0-1) 

A grep for `attached_deposit` or `env::attached` across the entire file returns **zero matches**, confirming that neither function — nor any helper it calls — ever reads the attached deposit value. In NEAR Protocol, when a `#[payable]` function completes successfully without consuming `env::attached_deposit()`, the attached NEAR is not automatically refunded; it remains on the contract's balance.

The only fund-movement function in the contract is `send_refund`, which unconditionally transfers a hardcoded constant `REFUND_AMOUNT` (2 NEAR) to the parent: [3](#0-2) 

This provides no mechanism to recover arbitrary amounts of stuck NEAR.

---

### Impact Explanation

Any NEAR attached to a successful call to `execute_scheduled` or `deploy_upgrade` is permanently locked in the XCC router contract. There is no admin withdrawal function, no sweep mechanism, and `send_refund` only moves a fixed 2 NEAR regardless of the actual balance. The funds are irrecoverable.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

`execute_scheduled` is explicitly designed to be callable by **any** unprivileged user: [4](#0-3) 

A user executing a scheduled cross-contract call may reasonably believe they need to attach NEAR to fund the outgoing promise (since the scheduled promise itself carries an `attached_balance` field). The `#[payable]` annotation reinforces this false expectation. The entry path is direct, requires no special privilege, and the mistake is intuitive.

`deploy_upgrade` is restricted to the parent (Aurora Engine) via `assert_preconditions`, making accidental NEAR attachment less likely there, but the root cause is identical.

---

### Recommendation

1. **`execute_scheduled`**: Remove the `#[payable]` attribute entirely. The function has no legitimate reason to accept NEAR; the promise's own `attached_balance` field governs funds forwarded to the target. Removing `#[payable]` causes the NEAR runtime to automatically reject any call that attaches NEAR, preventing the freeze.

2. **`deploy_upgrade`**: Same — remove `#[payable]`. The `initialize` call inside the upgrade batch is made with `NearToken::default()` (zero), so no deposit is needed.

3. If either function genuinely needs to forward NEAR in some future scenario, add an explicit assertion: `assert_eq!(env::attached_deposit(), NearToken::from_yoctonear(0), "ERR_UNEXPECTED_DEPOSIT")` until that logic is implemented.

---

### Proof of Concept

1. Deploy the XCC router contract (as Aurora Engine does for each user address).
2. The parent (Aurora Engine) calls `schedule` with a `PromiseArgs` payload, storing a promise at nonce `N`.
3. Any external account calls `execute_scheduled` with `nonce = N` and attaches, e.g., 1 NEAR.
4. The function succeeds: the promise is removed from storage and dispatched.
5. The 1 NEAR attached by the caller is now on the router contract's balance.
6. No function in the contract can return this 1 NEAR to the caller. `send_refund` only transfers the fixed `REFUND_AMOUNT` (2 NEAR) to the parent, not to the original depositor, and only covers the pre-configured storage refund — not arbitrary stuck deposits.
7. The NEAR is permanently frozen. [5](#0-4) [1](#0-0)

    #[payable]

            env::panic_str("ERR_PROMISE_NOT_FOUND")
