### Title
Unchecked `args.amount` in `ft_on_transfer` Enables ERC-20 Over-Minting with Fee-on-Transfer NEP-141 Tokens — (File: `engine/src/engine.rs`)

---

### Summary

Aurora Engine's `ft_on_transfer` entry point blindly trusts the `amount` field supplied by the calling NEP-141 contract and mints exactly that many ERC-20 tokens on Aurora, without verifying the actual balance change in the NEP-141 contract. A fee-on-transfer NEP-141 token will cause Aurora to mint more ERC-20 tokens than the NEP-141 tokens it actually received, creating an insolvent bridge.

---

### Finding Description

When a NEP-141 token is bridged into Aurora, the flow is:

1. A user calls `ft_transfer_call` on the NEP-141 contract, naming Aurora Engine as the receiver.
2. The NEP-141 contract transfers tokens (possibly deducting a fee) and then calls `ft_on_transfer` on Aurora Engine, passing `amount` = the nominal transfer amount.
3. Aurora Engine's `ft_on_transfer` handler in `engine/src/contract_methods/connector.rs` reads `args.amount` from the JSON input and passes it directly to `receive_erc20_tokens`. [1](#0-0) 

4. `receive_erc20_tokens` in `engine/src/engine.rs` takes `args.amount.as_u128()` verbatim and calls `setup_receive_erc20_tokens_input(&recipient, amount)`, which encodes a `mint(recipient, amount)` call to the ERC-20 contract. [2](#0-1) 

5. `setup_receive_erc20_tokens_input` encodes the ERC-20 `mint` selector with the unverified `amount`. [3](#0-2) 

At no point does Aurora Engine query its own NEP-141 balance before and after the transfer to confirm the actual amount received. The engine unconditionally mints `args.amount` ERC-20 tokens regardless of how many NEP-141 tokens it actually holds.

---

### Impact Explanation

**Impact: Critical — Insolvency.**

If a fee-on-transfer NEP-141 token is bridged, Aurora Engine's NEP-141 balance grows by `amount - fee`, but it mints `amount` ERC-20 tokens. The ERC-20 total supply on Aurora exceeds the NEP-141 collateral held by the engine. When users later burn ERC-20 tokens to withdraw NEP-141 (via `exit_to_near`), the last withdrawers will find insufficient NEP-141 balance in Aurora Engine's account, permanently freezing their funds. The discrepancy compounds with every deposit, making the bridge insolvent.

---

### Likelihood Explanation

**Likelihood: Medium.**

Any unprivileged NEAR account can deploy a custom NEP-141 token with fee-on-transfer logic. The `deploy_erc20_token` function in `engine/src/contract_methods/connector.rs` does not restrict which NEP-141 accounts can be registered. [4](#0-3) 

Once registered, any user can call `ft_transfer_call` on the malicious NEP-141 to trigger the over-minting. No admin compromise or privileged access is required.

---

### Recommendation

Before minting ERC-20 tokens, Aurora Engine should record its NEP-141 balance before the transfer and compare it to the balance after. The actual minted amount should be `balance_after - balance_before`, not `args.amount`. This requires a cross-contract call to the NEP-141 contract's `ft_balance_of` view before and after, or alternatively, the engine should only accept NEP-141 tokens from a curated allowlist of non-fee-on-transfer tokens.

---

### Proof of Concept

1. Attacker deploys a NEP-141 token `evil.near` that deducts a 10% fee on every transfer (i.e., transferring 100 tokens results in the receiver getting 90).
2. Attacker calls `deploy_erc20_token` on Aurora Engine to register `evil.near`, creating a corresponding ERC-20 on Aurora.
3. Attacker calls `ft_transfer_call` on `evil.near` with `amount = 1000`, receiver = Aurora Engine, msg = attacker's EVM address.
4. `evil.near` deducts 100 tokens as fee, credits Aurora Engine's NEP-141 balance with 900, then calls `ft_on_transfer` on Aurora Engine with `amount = 1000`.
5. Aurora Engine's `receive_erc20_tokens` reads `args.amount = 1000` and mints 1000 ERC-20 tokens to the attacker. [5](#0-4) 

6. Attacker now holds 1000 ERC-20 tokens backed by only 900 NEP-141 tokens. The bridge is undercollateralized by 100 tokens per cycle.
7. Repeating this drains the bridge's NEP-141 reserves, causing insolvency for legitimate users who cannot withdraw their NEP-141 tokens.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
