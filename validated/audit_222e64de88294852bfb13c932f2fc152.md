Audit Report

## Title
`add_expiration` Silently Truncates `expires_at > i64::MAX` to Epoch 0 in `FieldsDisplayMessage`, Enabling Hardware-Wallet Consent Deception — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

In `ConsentMessage::add_expiration`, the `FieldsDisplayMessage` arm casts a `u64` nanosecond timestamp to `i64` before dividing. For any `expires_at > i64::MAX` (e.g., `u64::MAX`), the cast wraps to a negative value; Rust's truncation-toward-zero integer division then yields `0`; the subsequent `as u64` cast produces `0`. The resulting `Value::TimestampSeconds { amount: 0 }` encodes Unix epoch (Jan 1 1970) in the structured consent message, while the on-chain approval stores the original far-future timestamp unchanged. This breaks the core security guarantee of ICRC-21 `FieldsDisplay`, which exists precisely to give constrained hardware wallets accurate, machine-readable transaction data.

## Finding Description

**Root cause — `responses.rs` lines 264–270:**

```rust
Some(expires_at) => {
    let seconds = (expires_at as i64) / 10_i64.pow(9);   // line 265
    fields_display.fields.push((
        "Approval expiration".to_string(),
        Value::TimestampSeconds { amount: seconds as u64 }, // line 269
    ))
}
```

Arithmetic for `expires_at = u64::MAX`:
- `u64::MAX as i64` → `-1` (wraps; defined behavior in Rust)
- `-1_i64 / 1_000_000_000_i64` → `0` (truncation toward zero)
- `0_i64 as u64` → `0`

The `FieldsDisplayMessage` stores `TimestampSeconds { amount: 0 }` — epoch 0.

The correct value is `u64::MAX / 1_000_000_000_u64 = 18_446_744_073` (year ~2554).

**Why the `GenericDisplayMessage` arm does not silently corrupt:** Lines 232–233 apply the same cast, but the subsequent `from_unix_timestamp(0)` succeeds and `replace_nanosecond(4294967295)` fails, returning the visible string `"Invalid nanosecond: 4294967295"` — a detectable error, not a silent one.

**Why the ledger accepts `u64::MAX`:** `approvals.rs` line 247:
```rust
if expires_at.unwrap_or_else(remote_future) <= now { ... }
```
`remote_future()` returns `TimeStamp::from_nanos_since_unix_epoch(u64::MAX)` (line 411). `u64::MAX <= now` is always false for any realistic `now`, so the approval is accepted and stored with the original `u64::MAX` expiration intact.

**Exploit flow:**
1. Attacker (unprivileged dApp) constructs `icrc2_approve` args with `expires_at = u64::MAX`.
2. Attacker calls `icrc21_canister_call_consent_message` (public query, no privilege required) requesting `FieldsDisplay` format.
3. Ledger returns `FieldsDisplayMessage` containing `TimestampSeconds { amount: 0 }`.
4. Hardware wallet (Ledger, Keystone, etc.) implementing ICRC-21 renders "Approval expiration: Jan 1 1970 00:00:00 UTC" or equivalent — a past/zero timestamp. The device has no independent means to detect the discrepancy; it trusts the structured consent data.
5. User signs. On-chain approval carries `expires_at = u64::MAX`, granting the spender a ~permanent allowance.

## Impact Explanation

The ICRC-21 `FieldsDisplay` path is the designated security boundary for hardware wallets that cannot parse free-form text. Corrupting the expiration field in this path directly undermines the mechanism that is supposed to protect users from malicious dApps. A successfully deceived user grants a near-permanent ICRC-2 allowance, enabling the spender to drain the account at any future time. This constitutes **significant ledger security impact with concrete user-funds harm**, matching the High bounty tier: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

- Entry point is a public query call — no privilege, no key, no governance action required.
- Any dApp can trigger this for any user who requests a `FieldsDisplay` consent message.
- The `FieldsDisplay` path is specifically designed for hardware wallets, which are the devices least able to independently validate structured numeric fields.
- The hardware wallet cannot distinguish `TimestampSeconds { amount: 0 }` (corrupted) from a legitimately zero-valued field; the deception is invisible at the device layer.

## Recommendation

Replace the lossy `as i64` cast with pure `u64` arithmetic in both arms of `add_expiration`:

```rust
// FieldsDisplayMessage arm (line 265)
let seconds = expires_at / 1_000_000_000_u64;
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds },
));

// GenericDisplayMessage arm (lines 232–233)
let seconds = expires_at / 1_000_000_000_u64;
let nanos = (expires_at % 1_000_000_000_u64) as u32;
```

No cast to `i64` is needed; `time::OffsetDateTime::from_unix_timestamp` accepts `i64`, so pass `seconds as i64` only after bounding (or use `i64::try_from(seconds).unwrap_or(i64::MAX)`).

## Proof of Concept

```rust
#[test]
fn test_add_expiration_fields_display_u64_max() {
    let expires_at: u64 = u64::MAX;

    // Current (buggy) behavior
    let seconds_buggy = (expires_at as i64) / 10_i64.pow(9);
    let displayed = seconds_buggy as u64;
    assert_eq!(displayed, 0); // epoch 0 — WRONG

    // Correct behavior
    let seconds_correct = expires_at / 1_000_000_000_u64;
    assert_eq!(seconds_correct, 18_446_744_073_u64); // year ~2554

    // Invariant violated
    assert_ne!(displayed, seconds_correct);
}
```
