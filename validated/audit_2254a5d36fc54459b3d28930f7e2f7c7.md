Audit Report

## Title
Ingress Validation Accepts Torsion-Component Ed25519 Keys Without Subgroup Check, Enabling Signature Forgery for Torsion-Derived Principals — (`rs/crypto/standalone-sig-verifier/src/lib.rs`)

## Summary
`verify_basic_sig_by_public_key` deserializes Ed25519 public keys via `deserialize_raw` (explicitly documented as skipping torsion/subgroup checks) and immediately calls `verify_signature` (ZIP215, cofactor-based). For any of the 18 known pure torsion-order points, the ZIP215 cofactor equation collapses to the identity for any `(R = S*B, S)` pair regardless of message content, allowing anyone to forge a valid signature for a torsion-key-derived principal without possessing any private key. The `verify_public_key` function that calls `is_torsion_free` exists but is never invoked in the ingress validation path.

## Finding Description
**Call chain:**

`validate_user_id_and_signature` → `validate_signature` → `validate_signature_plain` → `validator.verify_basic_sig_by_public_key` → `ic_crypto_standalone_sig_verifier::verify_basic_sig_by_public_key`

In `rs/crypto/standalone-sig-verifier/src/lib.rs` lines 21–43, the Ed25519 branch calls `ic_ed25519::PublicKey::deserialize_raw`, which is explicitly documented at `packages/ic-ed25519/src/lib.rs` lines 614–618 as not verifying prime-order subgroup membership. It then calls `pk.verify_signature` (lines 709–727), which implements the ZIP215 check: `(k*(-A) + S*B - R).mul_by_cofactor().is_identity()`.

**Why the check collapses for torsion points:** For a pure torsion point `A` with order dividing 8 (`8*A = 0`), construct the forged signature `(R = S*B, S)` for any scalar `S` and any message `M`:
- Challenge: `k = H(R || A || M)` (arbitrary)
- `recomputed_r - R = k*(-A) + S*B - S*B = k*(-A)`
- `(k*(-A)).mul_by_cofactor() = k*(8*(-A)) = k*0 = 0` ✓

This holds for **any** `S`, any `M`, and any of the 18 confirmed torsion keys in the test suite (`packages/ic-ed25519/tests/tests.rs` lines 519–544), all of which pass `deserialize_raw` and fail `is_torsion_free`.

**Existing check is unreachable:** `verify_public_key` at `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs` lines 97–103 calls `is_torsion_free()` but is confirmed absent from the entire `rs/validator/` directory. Node key validation at `rs/crypto/node_key_validation/src/lib.rs` lines 236–240 correctly applies `is_torsion_free`, demonstrating the fix is known and applied elsewhere.

## Impact Explanation
An unprivileged attacker can forge ingress messages from any of the 18 torsion-key-derived principals. The authentication invariant of `validate_user_id_and_signature` is broken for these principals: the system accepts signatures without any private key. If any canister has granted permissions or holds assets at a torsion-key-derived principal, those are fully accessible to any attacker. This constitutes unauthorized access to canister-controlled identities and funds — matching the **High** bounty impact: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The constraint here is that only torsion-key-derived principals are affected; no legitimate user would hold such a principal through normal key generation, but the authentication invariant is concretely broken.

## Likelihood Explanation
The attack requires no privileges, no key material, no network position, and no victim interaction. It is fully constructible offline using the 18 publicly known torsion points. The forged signature is deterministic and submittable via the standard `/api/v2/canister/{id}/call` HTTP endpoint. The only constraint is that the target principal must be derived from one of the 18 known torsion keys.

## Recommendation
In `rs/crypto/standalone-sig-verifier/src/lib.rs`, after `deserialize_raw` succeeds for `AlgorithmId::Ed25519`, add a torsion-free check before proceeding to signature verification:

```rust
if !pk.is_torsion_free() {
    return Err(CryptoError::MalformedPublicKey {
        algorithm: AlgorithmId::Ed25519,
        key_bytes: Some(pk_bytes.to_vec()),
        internal_error: "public key has torsion component".to_string(),
    });
}
```

This mirrors the check already present in `verify_public_key` (`rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs` lines 97–103) and in node key validation (`rs/crypto/node_key_validation/src/lib.rs` lines 236–240).
