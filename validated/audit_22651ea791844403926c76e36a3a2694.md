Audit Report

## Title
Unbounded Single-Message Reward Distribution Loop in SNS Governance Can Permanently DOS Periodic Tasks - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance `distribute_rewards` function executes three unbounded loops synchronously in a single message with no instruction-limit guard. At the parameter ceilings of 200,000 neurons and 700 proposals, the ballot-aggregation loop alone can process up to 140 million entries, exceeding the IC per-message instruction limit. Because `latest_reward_event` is only written at the very end of the function, a trap causes a full state rollback, and every subsequent timer invocation re-enters and re-traps, permanently halting reward distribution for the affected SNS.

## Finding Description
`distribute_rewards` is a synchronous `fn` (no `async`, no `await`) called directly from `run_periodic_tasks` at line 5513 of `rs/sns/governance/src/governance.rs`. The function contains three unbounded loops executed in a single message:

**Loop 1 â€” ballot aggregation** (lines 5894â€“5931): nested iteration over `considered_proposals` (up to `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700`) and each proposal's `ballots` map (up to `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` entries per proposal). At ceiling values this is 700 Ă— 200,000 = 140 million iterations, each performing `NeuronId::from_str` parsing and `Decimal` arithmetic.

**Loop 2 â€” neuron maturity update** (lines 5954â€“5997): iterates over `neuron_id_to_reward_shares`, up to 200,000 entries, performing mutable neuron lookups and arithmetic.

**Loop 3 â€” proposal settlement** (lines 6013â€“6081): iterates over `considered_proposals` (up to 700), calling `self.process_proposal(pid.id)` for each, which may itself do non-trivial work.

`latest_reward_event` is written only at lines 6084â€“6092, after all three loops complete. If the function traps at any earlier point, the IC rolls back all state changes. `should_distribute_rewards` (lines 5725â€“5753) checks `latest_reward_event.end_timestamp_seconds`; since the rollback leaves it unchanged, the condition remains `true` on every subsequent timer invocation, causing the same trap to repeat indefinitely.

The NNS governance canister has already addressed this exact class of bug: `distribute_pending_rewards` in `rs/nns/governance/src/reward/distribution.rs` (lines 42â€“52) uses `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` and a `RewardsDistributionStateMachine` backed by stable storage to spread work across multiple messages. SNS governance has no equivalent mechanism.

The ceiling constants are confirmed in `rs/sns/governance/src/types.rs` (lines 386 and 390): `MAX_NUMBER_OF_NEURONS_CEILING = 200_000` and `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700`.

## Impact Explanation
Reward distribution is permanently halted for the affected SNS instance. Neurons that voted on proposals never receive maturity rewards. The `run_periodic_tasks` timer continues to fire but always traps inside `distribute_rewards`, consuming subnet resources and blocking all subsequent work in the same timer callback (maturity finalization, metrics, GC). The SNS governance canister cannot self-recover without an upgrade that either batches the work or manually resets `latest_reward_event`. This constitutes an application/platform-level DoS with concrete, lasting harm to SNS participants â€” matching the **High ($2,000â€“$10,000)** impact class for "Application/platform-level DoS â€¦ or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
No privileged access or governance majority is required. Any unprivileged participant can stake tokens to create neurons (up to the SNS-configured `max_number_of_neurons`) and submit proposals (up to `max_number_of_proposals_with_ballots`). The cost is proportional to `neuron_minimum_stake_e8s` and `reject_cost_e8s`, which are SNS-configurable and often low. For SNS instances with many organic participants and high proposal throughput, the triggering condition can arise without any adversarial intent. The attack is repeatable and self-sustaining once triggered.

## Recommendation
Apply the same batching pattern already present in NNS governance:

1. After computing `neuron_id_to_reward_shares` and writing `latest_reward_event` (or an intermediate "distribution in progress" marker), persist the pending per-neuron distributions to stable storage, analogous to `RewardsDistributionStateMachine`.
2. Process neuron maturity updates in a separate periodic timer task that checks `is_message_over_threshold` after each neuron and resumes in the next message if the limit is reached.
3. Add an instruction-limit guard to the ballot-aggregation loop (Loop 1), or cap `considered_proposals` to a safe batch size per invocation, consistent with the NNS `DISTRIBUTION_MESSAGE_LIMIT = 1_000_000_000` instructions threshold.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_number_of_proposals_with_ballots = 700`.
2. Create 200,000 neurons (each staking the minimum).
3. Submit 700 proposals and have all 200,000 neurons vote on each (all ballots recorded in proposal state).
4. Wait for `should_distribute_rewards()` to return `true` (one reward round elapses).
5. Observe that the next `run_periodic_tasks` timer invocation traps (instruction limit exceeded inside `distribute_rewards`).
6. Observe that `latest_reward_event` is unchanged (state rolled back due to trap).
7. Observe that every subsequent timer invocation also traps â€” reward distribution is permanently halted.

A deterministic integration test can reproduce this using PocketIC by constructing a `GovernanceProto` with 200,000 neurons and 700 `ReadyToSettle` proposals (each with 200,000 ballots), advancing the clock past one reward round, and asserting that `run_periodic_tasks` traps and that `latest_reward_event` remains unchanged across repeated invocations.
