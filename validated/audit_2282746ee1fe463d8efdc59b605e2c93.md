### Title
EIP-7623 `floor_gas` enforced only as gas-limit gate but not in fee accounting, allowing sender to reclaim fees that should accrue to the relayer — (`engine-transactions/src/lib.rs`, `engine/src/engine.rs`)

### Summary

`floor_gas` is computed and used to gate the transaction (line 1079), but it is never passed into `refund_unused_gas`. The refund is computed solely from `gas_used`, so when `floor_gas > gas_used` the sender is refunded `(floor_gas − gas_used) × effective_gas_price` and the relayer receives only `gas_used × priority_fee_per_gas` instead of `floor_gas × priority_fee_per_gas`.

### Finding Description

**`floor_gas` formula** (lib.rs:229-251):

```
tokens_in_calldata = num_non_zero_bytes × 4 + num_zero_bytes
floor_gas           = tokens_in_calldata × total_cost_floor_per_token + gas_transaction_call
``` [1](#0-0) 

**`intrinsic_gas` formula** (lib.rs:164-226) charges `gas_transaction_non_zero_data` (16 gas) per non-zero byte, while `floor_gas` charges `4 × total_cost_floor_per_token` (= 4 × 10 = 40 gas) per non-zero byte under Prague. For 1 000 non-zero bytes:

| Quantity | Value |
|---|---|
| `intrinsic_gas` | 21 000 + 1 000 × 16 = **37 000** |
| `floor_gas` | 21 000 + 1 000 × 4 × 10 = **61 000** |

The gate at engine.rs:1079 correctly requires `gas_limit ≥ max(37 000, 61 000) = 61 000`: [2](#0-1) 

However, `floor_gas` is **never forwarded** to `refund_unused_gas`: [3](#0-2) 

`refund_unused_gas` receives only `gas_used`, `prepaid_amount`, `relayer_address`, and `fixed_gas`. `floor_gas` is a local variable that is dropped after the gate check. The refund is therefore `(gas_limit − gas_used) × effective_gas_price`, and the relayer receives `gas_used × priority_fee_per_gas`.

### Impact Explanation

For the example above with `gas_limit = floor_gas = 61 000` and a simple ETH transfer (`gas_used = 21 000`):

- Sender is refunded `(61 000 − 21 000) × effective_gas_price` — they recover the floor premium.
- Relayer receives `21 000 × priority_fee_per_gas` instead of `61 000 × priority_fee_per_gas`.
- Lost relayer yield per transaction: `40 000 × priority_fee_per_gas`.

EIP-7623 requires that when `floor_gas > gas_used`, the sender is charged as if `floor_gas` units were consumed, with the surplus accruing to the block proposer/relayer. The Aurora implementation satisfies the gate but not the accounting invariant.

**Impact: High — Theft of unclaimed yield** (relayer loses `(floor_gas − gas_used) × priority_fee_per_gas` on every Prague transaction where calldata is heavy but execution is cheap).

### Likelihood Explanation

- Requires Prague config (`has_floor_gas = true`) to be active.
- Any sender can craft a transaction with many non-zero calldata bytes targeting a cheap operation (simple ETH transfer, no-op call). No special privilege is needed.
- The attacker is the sender themselves recovering fees they should not recover; the victim is the relayer.

### Recommendation

Pass `floor_gas` into `refund_unused_gas` (or compute it inside that function) and use `max(gas_used, floor_gas)` as the effective consumed gas for fee-accounting purposes:

```rust
let effective_gas_used = core::cmp::max(gas_used, floor_gas);
refund_unused_gas(
    &mut io,
    &sender,
    effective_gas_used,   // ← was: gas_used
    &prepaid_amount,
    &relayer_address,
    fixed_gas,
)?;
```

This matches the EIP-7623 specification: the sender is only refunded `gas_limit − max(gas_used, floor_gas)`, and the relayer receives `max(gas_used, floor_gas) × priority_fee_per_gas`.

### Proof of Concept

1. Enable Prague config (`has_floor_gas = true`, `total_cost_floor_per_token = 10`).
2. Craft a Type-2 EIP-1559 transaction: `to = some_EOA`, `value = 0`, `data = [0xFF; 1000]` (1 000 non-zero bytes), `max_priority_fee_per_gas = P`.
3. Compute `floor_gas = 21 000 + 1 000 × 4 × 10 = 61 000`. Set `gas_limit = 61 000`.
4. Submit transaction. EVM executes a simple call; `gas_used = 21 000`.
5. Assert: `relayer_balance_delta == 21_000 × P` (actual) vs expected `61_000 × P`.
6. Assert: `sender_balance_delta == -(21_000 × effective_gas_price)` (actual) vs expected `-(61_000 × effective_gas_price)`.

The delta `40 000 × P` is returned to the sender instead of the relayer, confirming the invariant break. [4](#0-3) [5](#0-4)

```rust
    pub fn floor_gas(&self, config: &aurora_evm::Config) -> Result<u64, Error> {
        if config.has_floor_gas {
            let num_zero_bytes = u64::try_from(self.data.iter().filter(|b| **b == 0).count())
                .map_err(|_e| Error::IntegerConversion)?;
            let data_len = u64::try_from(self.data.len()).map_err(|_e| Error::IntegerConversion)?;
            let num_non_zero_bytes = data_len
                .checked_sub(num_zero_bytes)
                .ok_or(Error::GasOverflow)?;

            let base_gas = config.gas_transaction_call;
            let tokens_in_calldata = num_non_zero_bytes
                .checked_mul(4)
                .and_then(|gas| gas.checked_add(num_zero_bytes))
                .ok_or(Error::GasOverflow)?;

            tokens_in_calldata
                .checked_mul(config.total_cost_floor_per_token)
                .and_then(|gas| gas.checked_add(base_gas))
                .ok_or(Error::GasOverflow)
        } else {
            Ok(0)
        }
    }
```
