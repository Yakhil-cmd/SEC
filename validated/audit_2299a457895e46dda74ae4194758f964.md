### Title
RSA Keys Accepted as Delegation Trust Anchors Despite Being Rejected as Direct Signers — (`rs/validator/src/ingress_validation.rs`)

### Summary

`validate_delegation` and `validate_signature` apply inconsistent algorithm policies for bare `RsaSha256PublicKeyDer` keys. `validate_signature` explicitly rejects bare RSA with an error, but `validate_delegation` silently accepts it in the same branch as Ed25519/ECDSA. An attacker with an RSA keypair can bypass the RSA rejection by placing the RSA key as the `signer_pubkey` and using it to sign a delegation to any other key type, which then signs the actual request.

### Finding Description

In `validate_signature`, bare RSA is explicitly rejected: [1](#0-0) 

In `validate_delegation`, bare `RsaSha256PublicKeyDer` is grouped with Ed25519/ECDSA in the `basic_sig` branch and accepted without restriction: [2](#0-1) 

The call chain in `validate_signature` is:

1. `validate_delegations` is called with `signature.signer_pubkey` (the RSA key) as the initial trust anchor. [3](#0-2) 
2. Inside `validate_delegations`, `validate_delegation` is called with the RSA key to verify the RSA-signed delegation — this succeeds. [4](#0-3) 
3. After the loop, `pubkey` is updated to the delegated key (e.g., Ed25519). [5](#0-4) 
4. Back in `validate_signature`, `public_key_from_bytes` is called on the **final** key (Ed25519), not the original RSA key — so the RSA rejection branch is never reached. [6](#0-5) 

The `validate_user_id` check binds the `sender` principal to `signer_pubkey` (the RSA key), so the request arrives at the canister as the RSA-derived self-authenticating principal. [7](#0-6) 

### Impact Explanation

Any canister that grants access to a principal derived from an RSA public key (e.g., via `PrincipalId::new_self_authenticating(rsa_der)`) can be reached by an attacker who holds the corresponding RSA private key, despite the explicit policy that bare RSA is not a valid signing algorithm for ingress. The security invariant — "RSA signatures are not allowed except in webauthn context" — is fully bypassable via a one-hop delegation chain.

### Likelihood Explanation

The exploit requires only a standard RSA-2048+ keypair and the ability to craft an HTTP ingress message with a `sender_delegation` field. No privileged access, no threshold corruption, no social engineering. The path is entirely local and testable.

### Recommendation

`validate_delegation` must apply the same algorithm policy as `validate_signature`. Bare `RsaSha256PublicKeyDer` should be rejected in `validate_delegation` with the same error used in `validate_signature` (lines 694–700). The two match arms must be kept in sync, or better, extracted into a shared `is_allowed_signing_algorithm` predicate used by both functions.

### Proof of Concept

```
1. Generate RSA-2048 keypair (rsa_sk, rsa_pk_der).
2. Generate Ed25519 keypair (ed_sk, ed_pk_der).
3. Construct Delegation { pubkey: ed_pk_der, expiration: far_future, targets: None }.
4. Sign the delegation bytes with rsa_sk → rsa_delegation_sig.
5. Build UserSignature {
       signer_pubkey: rsa_pk_der,
       sender_delegation: Some([SignedDelegation { delegation, signature: rsa_delegation_sig }]),
       signature: ed25519_sign(message_id, ed_sk),
   }.
6. Set sender = PrincipalId::new_self_authenticating(rsa_pk_der).
7. Call validate_request(...) → expect Ok(CanisterIdSet::all()).
```

The RSA-signed delegation passes `validate_delegation` (line 813), the final key is Ed25519, and `validate_signature` takes the Ed25519 branch (line 671) — the RSA rejection at lines 694–700 is never evaluated.
