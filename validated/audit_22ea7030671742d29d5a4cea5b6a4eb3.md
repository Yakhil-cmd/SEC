Audit Report

## Title
Permanent Fund Lock in ckERC20 Withdrawal When Gas Fees Exceed Immutable `max_transaction_fee` Cap — (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

When Ethereum gas fees spike above the `max_transaction_fee` cap locked in at `withdraw_erc20` call time, the ckETH minter's resubmission loop silently stalls: `create_resubmit_transactions` returns an `InsufficientTransactionFee` error that `resubmit_transactions_batch` only logs, triggering no state change, no reimbursement, and no cancellation. Because `record_finalized_transaction` only reimburses on an on-chain `TransactionStatus::Failure` receipt, a transaction that is never mined produces no receipt and no reimbursement. Both the ckETH burned for gas and the ckERC20 burned for the withdrawal remain locked with no user-accessible or automated recovery path.

## Finding Description

**Root cause — immutable fee cap with no escape hatch:**

At `withdraw_erc20` time, `estimate_erc20_transaction_fee()` sets `max_transaction_fee`, which is stored as `allowed_max_transaction_fee` in the `ResubmissionStrategy` and never updated. In `rs/ethereum/cketh/minter/src/tx.rs` at L169–173, every resubmission attempt checks:

```rust
if new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee() {
    return Err(ResubmitTransactionError::InsufficientTransactionFee { ... });
}
```

**Nonce-cascade hard stop:**

In `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` at L618–630, `create_resubmit_transactions` pushes the error and immediately `return`s, blocking all higher-nonce transactions as well (documented in the comment at L589–592).

**Error is silently swallowed:**

In `rs/ethereum/cketh/minter/src/withdraw.rs` at L242–244, `resubmit_transactions_batch` only logs the error:

```rust
Err(e) => {
    log!(INFO, "Failed to resubmit transaction: {e:?}");
}
```

No `mutate_state`, no `process_event`, no reimbursement scheduling.

**Finalization path requires an on-chain receipt:**

`record_finalized_transaction` (L680–748 of `transactions/mod.rs`) only calls `record_reimbursement_request` when `receipt.status == TransactionStatus::Failure`. A transaction stuck in the mempool (never mined) never produces a receipt, so this path is never reached.

**No public recovery endpoint:**

A search of `cketh_minter.did` and the minter source confirms there is no `cancel_withdrawal`, `force_reimburse`, or equivalent endpoint. The DID exposes `withdraw_erc20` and query endpoints only; no privileged cancellation path exists.

## Impact Explanation

This is a **High** severity finding matching: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

Any user who calls `withdraw_erc20` during a period of normal gas prices and then experiences a sustained gas spike will have both their ckETH (gas reserve) and ckERC20 (withdrawal amount) locked in the minter with no callable recovery path. The lock persists until either (a) Ethereum base fees fall back below the original estimate, or (b) a governance upgrade is deployed — the same emergency-upgrade pattern already required for ckBTC stuck withdrawals (`rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`). If fees remain elevated indefinitely, the loss is permanent.

## Likelihood Explanation

- Any unprivileged user calling `withdraw_erc20` can trigger this; no special knowledge or privilege is required.
- Ethereum gas fee spikes of 2–10× within minutes are historically common during network congestion events.
- The minter's 10% resubmission headroom is insufficient against sustained spikes.
- The ckBTC minter's June 2025 stuck-withdrawal incident (documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`) confirms this class of failure is reachable in production and required emergency governance intervention to resolve.

## Recommendation

1. **Automated timeout reimbursement**: Track the timestamp when a withdrawal enters `sent_tx`. If it remains there beyond a configurable threshold (e.g., 7 days) and the last resubmission attempt returned `InsufficientTransactionFee`, automatically move it to `reimbursement_requests` so both ckETH and ckERC20 are returned to the user.
2. **Governance cancel endpoint**: Add a privileged endpoint (callable by NNS governance or the minter controller) to forcibly cancel a specific stuck withdrawal by `ledger_burn_index` and schedule reimbursement, mirroring the emergency upgrade pattern already used for ckBTC.
3. **Fee cap with user-supplied headroom**: Allow users to supply a higher `max_transaction_fee` at withdrawal time, or allow the minter to automatically increase the cap up to a configurable maximum before triggering reimbursement.

## Proof of Concept

1. Call `withdraw_erc20`; minter estimates `max_transaction_fee = F` and burns ckETH + ckERC20.
2. Transaction with nonce N is placed in `sent_tx` with `allowed_max_transaction_fee = F`.
3. Ethereum base fee spikes; required fee for resubmission = `1.5 × F > F`.
4. On the next timer tick, `create_resubmit_transactions` (L618–630 of `transactions/mod.rs`) hits `InsufficientTransactionFee` for nonce N and returns immediately — nonces N+1, N+2, … are also skipped.
5. `resubmit_transactions_batch` (L242–244 of `withdraw.rs`) logs the error and exits; no state change.
6. `send_transactions_batch` re-broadcasts the original low-fee transaction, which Ethereum nodes reject as underpriced.
7. The transaction is never mined; `finalize_transactions_batch` never sees a receipt; `record_finalized_transaction` is never called; no reimbursement is triggered.
8. The user's ckETH and ckERC20 remain locked indefinitely with no callable recovery path.

A deterministic integration test can reproduce this by: (a) submitting a `withdraw_erc20` request, (b) advancing the mock gas oracle to return a fee 2× the original estimate, (c) running the timer loop, and (d) asserting that `reimbursement_requests` remains empty and the withdrawal status remains `TxSent` indefinitely.
