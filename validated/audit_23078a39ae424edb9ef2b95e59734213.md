### Title
Hardcoded Test-Placeholder `L1_TX_NATIVE_PRICE = 10` Causes Incorrect Native Resource Allocation for L1→L2 Transactions — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded to `10` with an explicit TODO comment acknowledging it is not a production-ready value. This constant directly controls how many native computational resources are allocated per gas unit for every L1→L2 priority transaction. Because the value matches the test-only `native_price` in `BlockMetadataFromOracle::new_for_test()` and is orders of magnitude smaller than any realistic production `native_price`, L1 transactions receive a disproportionately large native resource budget, breaking the native resource accounting invariant that underpins ZKsync OS's DoS protection and fee correctness.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs` line 66:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

This constant is consumed in `process_l1_transaction.rs` inside `prepare_and_check_resources` to compute `native_per_gas` for every priority (L1→L2) transaction:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

`native_per_gas` then drives two downstream quantities:

1. `native_prepaid_from_gas = native_per_gas × gas_limit` — the total native resource budget granted to the transaction.
2. `native_per_pubdata = gas_per_pubdata × native_per_gas` — the native cost per pubdata byte. [3](#0-2) 

For L2 transactions the equivalent formula uses the block-level `native_price` supplied by the operator via `BlockMetadataFromOracle`. In `new_for_test()` that field is also `10`:

```rust
native_price: U256::from(10),
``` [4](#0-3) 

This confirms `L1_TX_NATIVE_PRICE = 10` was copied from the test fixture and never updated for production. In a real deployment the operator's `native_price` would be calibrated to the actual proving cost — typically several orders of magnitude larger (e.g. `1_000_000` or more). The mismatch means:

| Scenario | `native_per_gas` formula | Result (gas_price = 1 gwei) |
|---|---|---|
| L2 tx (production `native_price = 1_000_000`) | `1e9 / 1_000_000` | **1 000** |
| L1 tx (hardcoded `L1_TX_NATIVE_PRICE = 10`) | `1e9 / 10` | **100 000 000** |

L1 transactions receive **~100 000×** more native resources per gas unit than L2 transactions paying the same gas price.

The refund calculation in `compute_gas_refund` also uses `native_per_gas` to convert native consumption back into gas:

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
``` [5](#0-4) 

With an inflated `native_per_gas`, `native_used / native_per_gas` is tiny, so `delta_gas` is negative or near-zero. The user is never charged extra gas even when their transaction consumed far more native resources than the gas price implies. This breaks the invariant that gas cost ≥ native resource cost.

---

### Impact Explanation

**Resource accounting bug / block-level DoS vector.**

1. **Unbounded native resource consumption per L1 tx.** Each priority transaction is granted a native budget of `native_per_gas × gas_limit`. With `L1_TX_NATIVE_PRICE = 10` and a realistic gas price, this budget is ~100 000× larger than for an equivalent L2 transaction. A single L1 transaction with a large gas limit can consume the entire block's native resource quota.

2. **Block native limit exhaustion at low cost.** The block-level check in `check_for_block_limits` enforces a cap on `block_computational_native_used`. Because each L1 tx consumes native resources at a rate 100 000× higher than priced, an attacker can exhaust the block's native budget with far fewer (and cheaper) L1 transactions than the protocol designers intended, starving L2 transactions from being included.

3. **Incorrect fee accounting.** The refund path under-charges L1 transactions for native resource consumption, meaning the protocol absorbs the cost difference — a direct economic loss to the protocol/operator.

---

### Likelihood Explanation

**Medium.** Submitting L1→L2 priority transactions requires depositing ETH on L1, so the attack is not free. However, because the native resource allocation is ~100 000× inflated, the cost to exhaust a block's native budget is ~100 000× lower than intended. Any party with modest ETH capital can mount a sustained block-stuffing DoS. The bug is also triggered by every legitimate L1→L2 transaction, meaning incorrect accounting is the default state, not an edge case.

---

### Recommendation

Replace the placeholder with a value derived from the actual production `native_price`, or compute it dynamically from the block metadata with a floor to prevent operator censorship:

```rust
// Before (placeholder):
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);

// After (example — calibrate to actual proving cost):
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([1_000_000, 0, 0, 0]);
```

Alternatively, resolve EVM-1157 by deriving `native_per_gas` for L1 transactions from `max(block_native_price, CENSORSHIP_FLOOR)` so the value tracks production costs while still preventing operator censorship.

### Proof of Concept

**Setup:**
- Production block `native_price = 1_000_000` (operator-set, realistic value).
- Attacker submits a priority (L1→L2) transaction: `gas_price = 1 gwei`, `gas_limit = 30_000_000`.

**L2 transaction (correct accounting):**
```
native_per_gas = 1_000_000_000 / 1_000_000 = 1_000
native_prepaid  = 1_000 × 30_000_000 = 30_000_000_000
```

**L1 transaction (broken accounting via `L1_TX_NATIVE_PRICE = 10`):**
```
native_per_gas = 1_000_000_000 / 10 = 100_000_000
native_prepaid  = 100_000_000 × 30_000_000 = 3_000_000_000_000_000
```

The L1 transaction receives a native budget **100 000× larger** than the equivalent L2 transaction for the same gas payment. If the block's native resource limit is, say, `60_000_000_000`, a single such L1 transaction's budget already exceeds it by 50 000×, and the transaction can consume the entire block native quota in one shot. The refund path (`compute_gas_refund`) will not charge the attacker extra gas for this excess native consumption because `native_used / native_per_gas` rounds down to near zero.

The root cause is the hardcoded test-placeholder `L1_TX_NATIVE_PRICE = 10` at: [1](#0-0) 

consumed unconditionally in production at: [6](#0-5)
