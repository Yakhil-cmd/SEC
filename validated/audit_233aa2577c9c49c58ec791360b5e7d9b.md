Audit Report

## Title
Unchecked Integer Overflow in `disburse_maturity` Traps SNS Governance Canister for Neurons with Large Maturity — (File: rs/sns/governance/src/governance.rs)

## Summary

The `disburse_maturity` function in SNS governance calls `.expect()` on a `u64::checked_mul` result when computing `maturity_e8s_equivalent * percentage_to_disburse`. When `maturity_e8s_equivalent` exceeds `u64::MAX / 100 ≈ 1.84 × 10¹⁷ e8s`, the multiplication overflows, `checked_mul` returns `None`, and `.expect()` panics — trapping the canister and permanently blocking that neuron owner from disbursing their maturity. The reward distribution loop accumulates maturity via bare `+=`, allowing `maturity_e8s_equivalent` to silently grow past the overflow threshold in release mode.

## Finding Description

In `rs/sns/governance/src/governance.rs`, `disburse_maturity` validates only that `percentage_to_disburse ∈ [1, 100]`: [1](#0-0) 

It then performs the multiplication in `u64` with `.expect()`: [2](#0-1) 

No constraint is placed on `maturity_e8s_equivalent`. When `maturity_e8s_equivalent > u64::MAX / 100`, `checked_mul` returns `None` and `.expect()` panics, trapping the canister. The reward distribution loop that feeds `maturity_e8s_equivalent` uses a bare `+=`: [3](#0-2) 

In Wasm release mode (production IC), this `+=` wraps silently on overflow, meaning `maturity_e8s_equivalent` can grow monotonically past `u64::MAX / 100` without any error. Once it crosses that threshold, every subsequent `disburse_maturity` call by that neuron owner traps. The NNS governance has the same `.expect()` pattern: [4](#0-3) 

but NNS is protected in practice because the total ICP supply (~5 × 10¹⁶ e8s) is below the overflow threshold. SNS token supplies are unconstrained, making the threshold reachable.

## Impact Explanation

A neuron owner whose `maturity_e8s_equivalent` exceeds `u64::MAX / 100` is permanently unable to disburse their maturity through the normal path. Every `manage_neuron` → `DisburseMaturity` call traps; the maturity remains locked at the large value with no recovery path. This constitutes a concrete, permanent loss of access to earned SNS token rewards for the affected neuron owner — a significant SNS governance security impact with direct user financial harm. This matches the **High ($2,000–$10,000)** impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation

An SNS with a large token supply (e.g., 10¹⁰ tokens = 10¹⁸ e8s), a high initial reward rate (10–100% per year), and a neuron holding a large fraction of the stake can accumulate `maturity_e8s_equivalent > 1.84 × 10¹⁷ e8s` within 1–20 years of normal operation. No attacker action is required — the condition arises from ordinary reward distribution. The neuron owner cannot avoid it; the only trigger is calling `DisburseMaturity` after the threshold is crossed, which is a normal user action requiring no special privileges. The affected window spans from when maturity first exceeds `u64::MAX / 100` until it wraps at `u64::MAX` — potentially years.

## Recommendation

Replace the `.expect()` calls with proper error propagation returning a `GovernanceError` instead of panicking:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)
    .and_then(|v| v.checked_div(100))
    .ok_or_else(|| GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Arithmetic overflow computing maturity to disburse.",
    ))?;
```

Additionally, replace the bare `+=` in the reward distribution loop with `saturating_add` or `checked_add` with error logging to prevent silent maturity wrap-around: [3](#0-2) 

## Proof of Concept

1. Deploy an SNS with total token supply ≥ 10¹⁰ tokens (10¹⁸ e8s) and a high reward rate.
2. Create a neuron holding a large fraction of the stake; allow rewards to accumulate via normal voting until `maturity_e8s_equivalent > 184_467_440_737_095_516` (i.e., `u64::MAX / 100 + 1`).
3. Call `manage_neuron` with `DisburseMaturity { percentage_to_disburse: 100, to_account: ... }`.
4. The canister traps at the `.expect("Overflow while processing maturity to disburse.")` at line 1646; the call fails and every subsequent attempt also traps.

Minimal triggering state (unit test): set `neuron.maturity_e8s_equivalent = 184_467_440_737_095_517`, call `disburse_maturity` with `percentage_to_disburse = 100`. The multiplication `184_467_440_737_095_517 * 100 = 18_446_744_073_709_551_700 > u64::MAX`, so `checked_mul` returns `None` and `.expect()` panics. [2](#0-1)
