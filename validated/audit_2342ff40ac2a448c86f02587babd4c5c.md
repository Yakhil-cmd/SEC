Audit Report

## Title
Missing Duplicate `callback_id` Check in `validate_self_validating_payload_impl` Enables Double-Delivery of Bitcoin Adapter Responses — (`rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary
`validate_self_validating_payload_impl` performs no content validation beyond an emptiness check, and `validate_payload` computes `delivered_ids` from past payloads but never uses it during validation. A Byzantine block proposer can include two `BitcoinAdapterResponse` entries with the same `callback_id` in a `SelfValidatingPayload`. This passes all honest-validator checks, gets finalized, and causes `push_response` to enqueue two `ConsensusResponse` entries for the same callback into `consensus_queue`, violating the one-response-per-request invariant and enabling double-delivery to the bitcoin system canister.

## Finding Description
`validate_self_validating_payload_impl` (L234–251 of `rs/bitcoin/consensus/src/payload_builder.rs`) only checks `if *payload == SelfValidatingPayload::default()` and returns `Ok(size)` for any non-empty payload. There is no iteration over responses, no `callback_id` uniqueness check, and no cross-reference against in-flight requests.

`validate_payload` (L357–397) computes `delivered_ids` at L369 from `past_payloads` but passes it nowhere — the call to `validate_self_validating_payload_impl` at L379 does not receive it. No intra-payload deduplication is performed.

In `push_response` (`rs/replicated_state/src/bitcoin.rs`), the `GetSuccessorsResponse` arm (L32–39) uses `get_mut` on `bitcoin_get_successors_contexts`. The context is not consumed after the first call, so a second call with the same `callback_id` also finds the context and pushes a second `ConsensusResponse` to `consensus_queue`. For `SendTransactionResponse` (L64–75), `GetSuccessorsReject` (L77–88), and `SendTransactionReject` (L90–102), there is no lookup at all — both duplicate entries push unconditionally.

Exploit flow:
1. Byzantine block proposer constructs a `SelfValidatingPayload` containing two `BitcoinAdapterResponse` entries with identical `callback_id`.
2. Honest validators call `validate_payload`: decode succeeds, size check passes, `validate_self_validating_payload_impl` returns `Ok`. Block is accepted and finalized.
3. Execution calls `push_response` for each entry. Both calls succeed and push to `consensus_queue`.
4. Execution environment delivers two responses to the bitcoin system canister for a single request.

## Impact Explanation
Two `ConsensusResponse` entries for the same `callback_id` are placed in `consensus_queue`. The execution environment processes both, delivering two responses to the bitcoin system canister for a single request. In the context of ckBTC/ckDOGE chain fusion, this can cause the canister to process the same block data twice, potentially corrupting chain-fusion state or enabling double-crediting of wrapped assets. This matches the allowed High impact: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
Requires a single Byzantine block proposer — one compromised subnet node acting within its normal role. No subnet majority, no key compromise, no external dependency. The crafted payload is deterministically accepted by all honest validators because the validation gap exists on every node. The attack is repeatable as long as the node retains its block-proposer role.

## Recommendation
1. In `validate_payload` (or `validate_self_validating_payload_impl`), collect all `callback_id` values from the decoded payload into a `HashSet` and return `InvalidArtifact(InvalidSelfValidatingPayloadReason::...)` if any duplicate is detected.
2. Cross-check each `callback_id` against in-flight requests in the replicated state (as `get_self_validating_payload_impl` already does via `past_callback_ids`), rejecting any `callback_id` not present in the current context manager.
3. In `push_response` for `GetSuccessorsResponse`, replace `get_mut` with `remove` so the context is consumed on first delivery, making a second call return `BitcoinNonMatchingResponse` and preventing the second enqueue.
