### Title
Truncating Division in `native_per_pubdata` Calculation Rounds to Zero, Making Pubdata Free - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When `pubdata_price < native_price`, the truncating integer division used to compute `native_per_pubdata` rounds down to **0**. This silently eliminates all native-resource charges for pubdata, allowing any user to write up to the block's pubdata limit for free. The operator bears the full L1 data-publication cost without compensation.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142, `native_per_pubdata` is computed with a truncating (`wrapping_div`) division:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` performs standard truncating integer division (the "wrapping" only affects division-by-zero, which is already guarded). When `pubdata_price < native_price` — for example `pubdata_price = 5`, `native_price = 10` — the result is `5 / 10 = 0`.

This is inconsistent with the `native_per_gas` calculation on line 135, which correctly uses `div_ceil` to avoid rounding to zero:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

With `native_per_pubdata = 0`, two downstream accounting paths are broken:

**1. Intrinsic pubdata overhead becomes zero** in `create_resources_for_tx`:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
``` [3](#0-2) 

**2. Post-execution pubdata charging becomes zero** in `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
// = current_pubdata_spent * 0 = 0
``` [4](#0-3) 

The same truncating division is also present in the public API helper `validate_l2_tx_intrinsic_native_resources` in `api/src/helpers.rs` line 427, which mirrors bootloader behavior for off-chain validation:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [5](#0-4) 

---

### Impact Explanation

When `pubdata_price < native_price`, `native_per_pubdata` silently becomes 0. Every storage write, event log, and other pubdata-generating operation costs **zero native resources**. A user can fill the entire block pubdata quota (up to `get_pubdata_limit()`) without paying any native fee for it. The operator must still publish this data to L1 and bears the full cost, receiving no compensation. This is a direct, quantifiable loss of funds for the operator proportional to the pubdata limit per block. [6](#0-5) 

---

### Likelihood Explanation

`pubdata_price` and `native_price` are independent operator-set block-level parameters. `native_price` reflects the cost of a single RISC-V proving cycle; `pubdata_price` reflects the cost per byte of L1 data. There is no protocol-level invariant enforcing `pubdata_price >= native_price`. A configuration such as `pubdata_price = 1 wei/byte`, `native_price = 10 wei/cycle` is operationally plausible (pubdata is cheap relative to computation). Any unprivileged transaction sender can exploit this condition once it exists, requiring no special access. [7](#0-6) 

---

### Recommendation

Replace the truncating `wrapping_div` with `div_ceil` for `native_per_pubdata`, matching the treatment of `native_per_gas`:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. Optionally, add a protocol-level assertion or validation that `native_per_pubdata > 0` when `pubdata_price > 0`.

---

### Proof of Concept

**Setup:** Operator configures block with `pubdata_price = 5`, `native_price = 10`.

**Step 1 — Rounding:**
```
native_per_pubdata = pubdata_price.wrapping_div(native_price) = 5 / 10 = 0
```

**Step 2 — Intrinsic pubdata overhead:**
```
intrinsic_pubdata_overhead = 0 * L2_TX_INTRINSIC_PUBDATA = 0
```
No native is deducted for intrinsic pubdata during resource materialization.

**Step 3 — Attacker submits a transaction** that performs 20 SSTORE operations (each writing a new slot), generating ~20 × 64 = 1280 bytes of pubdata.

**Step 4 — Post-execution pubdata charge:**
```
native_charged_for_pubdata = 1280 * native_per_pubdata = 1280 * 0 = 0
```

**Step 5 — Result:** The attacker pays only EVM gas for the SSTORE opcodes. Zero native resources are charged for 1280 bytes of pubdata. The operator publishes 1280 bytes to L1 at their own expense.

**Contrast with correct behavior:** With `div_ceil`, `native_per_pubdata = ceil(5/10) = 1`, and the attacker would be charged `1280 * 1 = 1280` native resource units for pubdata. [8](#0-7) [9](#0-8)

    native_per_pubdata: u64,
