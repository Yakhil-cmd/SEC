## Title
SNS Governance `validate_neuron_minimum_dissolve_delay_to_vote_seconds` Off-by-One Allows Permanent Electoral Roll Erasure - (`rs/sns/governance/src/types.rs`, `rs/sns/governance/src/governance.rs`)

## Summary
The validation for `neuron_minimum_dissolve_delay_to_vote_seconds` uses a strict `>` comparison against `max_dissolve_delay_seconds`, permitting the two values to be set equal. Because `max_dissolve_delay_seconds` is the hard ceiling on any neuron's dissolve delay, setting the minimum-to-vote equal to the maximum causes `compute_ballots_for_new_proposal` to produce an empty electoral roll for any SNS where no neuron holds exactly the maximum dissolve delay. The resulting `"No eligible voters."` error on every subsequent proposal submission permanently bricks the SNS governance canister with no on-chain recovery path.

If no neuron holds exactly `max_dissolve_delay_seconds`, `electoral_roll` is empty and the function returns `Err("No eligible voters.")`. Every subsequent `make_proposal` call fails at this point, making it impossible to submit a corrective `ManageNervousSystemParameters` proposal.

**Why existing checks are insufficient:** The only guard is the `>` boundary check in `validate_neuron_minimum_dissolve_delay_to_vote_seconds`. There is no live-population check, no floor relative to `max_dissolve_delay_seconds`, and no recovery mechanism (e.g., NNS-level override for SNS instances) that would allow the parameter to be reset after governance is bricked.

## Impact Explanation
Permanent, irreversible loss of governance control over an SNS instance: no proposals can be submitted, no votes cast, no parameters changed, no dapp canisters returned to fallback controllers, and no SNS treasury funds disbursed through governance. This matches the allowed High impact: *"Significant SNS security impact with concrete user or protocol harm."* It does not reach Critical because exploitation requires passing a governance proposal (not a direct unprivileged call), constraining the attacker to cases where they hold or can assemble sufficient voting power.

## Likelihood Explanation
The entry path is a standard `ManageNeuron â†’ MakeProposal` ingress call available to any neuron holder with `SubmitProposal` permission. Passing the proposal requires the normal SNS threshold (typically 3 % of total voting power, 50 % of exercised). SNS instances with concentrated token distributions, low participation, or a single founding team controlling majority stake are directly at risk from a malicious insider or from an accidental misconfiguration. The off-by-one is invisible to a non-expert reviewer because `neuron_minimum_dissolve_delay_to_vote_seconds = max_dissolve_delay_seconds` looks like a reasonable "maximum lockup required to vote" policy.

## Recommendation
1. **Tighten the boundary check** in `validate_neuron_minimum_dissolve_delay_to_vote_seconds` (`rs/sns/governance/src/types.rs` ~L764): change `>` to `>=` so that equality is also rejected.
2. **Add a live-population guard** in `perform_manage_nervous_system_parameters` (`rs/sns/governance/src/governance.rs` ~L2587): before writing new parameters, verify that at least one existing neuron satisfies the proposed `neuron_minimum_dissolve_delay_to_vote_seconds` and can afford `reject_cost_e8s`.
3. **Classify `ManageNervousSystemParameters` as a critical-threshold proposal** to raise the bar for passing governance-bricking parameter changes.
