### Title
Integer Underflow in Staleness Filter Causes Panic or Incorrect Snapshot Inclusion in Voting Power Spike Detection - (File: rs/nns/governance/src/governance/voting_power_snapshots.rs)

### Summary

The `totals_entry_with_minimum_total_potential_voting_power_if_voting_power_spiked` function in `VotingPowerSnapshots` performs an unchecked `u64` subtraction `now_seconds - created_at` when filtering snapshots for staleness. If any stored snapshot has a `created_at` timestamp greater than `now_seconds` (which can happen due to time going slightly backwards between canister timer invocations, or due to a snapshot being stored with a future timestamp), this subtraction will **panic** in debug builds or **wrap around** to a huge value in release builds, causing all snapshots to be incorrectly treated as non-stale and included in the minimum-finding logic. This corrupts the spike detection mechanism that protects NNS governance proposal ballots from artificially inflated voting power.

### Finding Description

In `rs/nns/governance/src/governance/voting_power_snapshots.rs`, the staleness filter at line 134 is:

```rust
.filter(|(created_at, _)| {
    let age = now_seconds - created_at;   // <-- bare u64 subtraction, no underflow guard
    age <= MAXIMUM_STALENESS_SECONDS
})
```

`now_seconds` and `created_at` are both `u64` (`TimestampSeconds`). If `created_at > now_seconds`, the subtraction underflows. In Rust, integer overflow/underflow in release mode wraps (for primitive `u64` arithmetic), producing a huge value (e.g., `u64::MAX - delta`). Since `MAXIMUM_STALENESS_SECONDS` is `ONE_MONTH_SECONDS * 3` (approximately 7,776,000), the wrapped value will always be far larger, so the filter predicate evaluates to `false` — the snapshot is incorrectly excluded from the staleness window. Conversely, in debug mode this panics, trapping the canister.

The analogous vulnerability in the external report is `Checkpoints#getAtBlock()` returning the wrong historical value when two checkpoints exist at the same block — both are cases where a historical/snapshot lookup returns an incorrect value due to a boundary condition in the time/block comparison logic.

The `SnapshotVotingPowerTask` records snapshots using `governance.env.now()` as the timestamp. On the Internet Computer, `ic_cdk::api::time()` returns the consensus time of the current round. While IC consensus time is monotonically non-decreasing within a single subnet, the governance canister's timer task reads `now()` twice in `execute()` — once for the spike check and once for recording — and these could theoretically differ. More critically, if the canister is upgraded or state is restored from a snapshot with a slightly different time reference, a stored `created_at` could exceed the `now_seconds` passed to the filter.

The root cause is the bare subtraction `now_seconds - created_at` without using `saturating_sub` or `checked_sub`, which is inconsistent with the rest of the codebase (e.g., `now_seconds.saturating_sub(latest_snapshot_timestamp_seconds)` used correctly in `rs/nns/governance/src/lib.rs` line 681).

### Impact Explanation

The `previous_ballots_if_voting_power_spike_detected` function is called from `compute_ballots_for_standard_proposal` in `rs/nns/governance/src/governance.rs` every time a standard NNS proposal is submitted. If the underflow causes a panic, the governance canister traps on every proposal submission, effectively halting NNS governance. If the underflow causes wrap-around, the affected snapshot is silently excluded from the staleness window, potentially causing the spike detection to use an incorrect minimum baseline — either failing to detect a real spike (allowing inflated voting power ballots to be used) or incorrectly detecting a spike (using stale ballots for a legitimate proposal). Both outcomes corrupt the integrity of NNS governance voting.

### Likelihood Explanation

The IC consensus time is monotonically non-decreasing within a subnet's normal operation, so `created_at > now_seconds` is unlikely in steady state. However, it can occur if:
1. The governance canister is restored from a checkpoint taken at a later time (e.g., during subnet recovery or state sync).
2. A canister upgrade resets or adjusts the time source.
3. The `SnapshotVotingPowerTask` records a snapshot at time T, and a subsequent call to `is_latest_snapshot_a_spike` or `previous_ballots_if_voting_power_spike_detected` is made with a `now_seconds` derived from a slightly earlier consensus round (possible if the timer fires in a different round than expected).

The likelihood is low in normal operation but non-zero in edge cases involving subnet recovery or canister upgrades, which are privileged but routine operations on the IC.

### Recommendation

Replace the bare subtraction with `saturating_sub` or `checked_sub`, consistent with the rest of the codebase:

```rust
.filter(|(created_at, _)| {
    let age = now_seconds.saturating_sub(*created_at);
    age <= MAXIMUM_STALENESS_SECONDS
})
```

This ensures that if `created_at > now_seconds`, the age is treated as 0 (i.e., the snapshot is considered fresh, not stale), which is the safe and correct behavior.
