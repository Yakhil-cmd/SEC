### Title
User-Controlled U256 Overflow in `validate_and_compute_fee_for_transaction` Returns `TxError::Internal`, Halting Block Execution — (`basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`)

---

### Summary

An unprivileged L2 transaction sender can craft a transaction with `value = U256::MAX` (or any value large enough to overflow when added to a nonzero fee) and any nonzero `max_fee_per_gas`/`gas_limit`. The `checked_add` at lines 386–388 overflows U256, maps to `internal_error!(...)`, and returns `TxError::Internal`. The transaction loop treats `TxError::Internal` as a fatal block-halting condition rather than a per-transaction rejection, terminating block execution for all subsequent transactions.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the balance sufficiency check computes:

```rust
let tx_value = transaction.value();

let mut total_required_balance = tx_value
    .checked_add(worst_case_fee_amount)
    .ok_or(internal_error!("transaction amount + fee"))?;
``` [1](#0-0) 

`tx_value` is a raw U256 read directly from the transaction with no prior bounds check. `worst_case_fee_amount` is `max_fee_per_gas * gas_limit`, which is nonzero for any valid fee-paying transaction. When `tx_value` is near U256::MAX, the addition overflows and `checked_add` returns `None`. The `.ok_or(internal_error!(...))` maps this to `TxError::Internal(InternalError(...))`.

The `InternalError` type is documented as "should not be triggered by user input": [2](#0-1) 

In the transaction processing loop, `TxError::Internal` is handled as a fatal error:

```rust
Err(TxError::Internal(err)) => {
    system.finish_global_frame(None)?; // no rollback
    return Err(err);                   // HALTS THE BLOCK
}
``` [3](#0-2) 

This `return Err(err)` propagates out of `loop_op`, terminating the entire block's transaction loop. The balance check that would normally reject this transaction (line 392–399) is never reached because the overflow occurs first. [4](#0-3) 

A secondary overflow path exists at lines 355–362, where `u256_mul_by_word(max_fee_per_gas, gas_limit)` overflow also returns `internal_error!`: [5](#0-4) 

Both paths share the same root cause: user-supplied arithmetic overflow is classified as an internal (block-fatal) error rather than a validation (per-tx) error.

---

### Impact Explanation

- The block's transaction loop terminates immediately upon encountering the malicious transaction.
- All subsequent transactions in the batch are unprocessable.
- State changes and fees from prior committed transactions in the same batch may be irrecoverable depending on batch finalization semantics.
- The invariant that user-supplied arithmetic overflow must never halt block execution is violated.

---

### Likelihood Explanation

The attack requires only a valid signed Ethereum transaction with `value = U256::MAX` (or any value ≥ `U256::MAX - worst_case_fee_amount + 1`) and any nonzero fee. No privileged access, leaked keys, or external oracle cooperation is needed. The transaction passes all prior checks (gas limit, signature, nonce) before reaching the vulnerable arithmetic. This is trivially constructable and locally testable.

---

### Recommendation

Replace `ok_or(internal_error!(...))` with `ok_or(TxError::Validation(InvalidTransaction::OverflowPaymentInTransaction))` (or an equivalent validation error variant) at all three `checked_add`/`checked_mul` sites that operate on user-supplied values:

- Line 358: `u256_mul_by_word` overflow on `max_fee_per_gas * gas_limit`
- Line 388: `checked_add` overflow on `tx_value + worst_case_fee_amount`
- Line 391: `checked_add` overflow on `total_required_balance + fee_for_blob_gas` [6](#0-5) 

The general principle: any arithmetic on attacker-controlled fields (`value`, `max_fee_per_gas`, `gas_limit`, blob counts) must produce `TxError::Validation`, never `TxError::Internal`.

---

### Proof of Concept

```
1. Craft an EIP-1559 transaction:
     value          = U256::MAX
     max_fee_per_gas = 1
     gas_limit       = 21000
     nonce           = current sender nonce
   Sign it with any valid secp256k1 key.

2. Submit to the ZKsync OS bootloader.

3. Execution path:
   - tx_gas_limit check passes (21000 is valid)
   - get_gas_prices succeeds (max_fee_per_gas >= base_fee assumed)
   - signature validates
   - nonce increments (committed, no rollback on Internal error)
   - worst_case_fee_amount = 1 * 21000 = 21000 (no mul overflow)
   - checked_add(U256::MAX, 21000) → None
   - ok_or(internal_error!(...)) → TxError::Internal

4. tx_loop matches Err(TxError::Internal(err)):
   - finish_global_frame(None)  // no state rollback
   - return Err(err)            // block loop terminates

5. All subsequent transactions in the batch are not processed.
```

Fuzz assertion: for all `(value, max_fee_per_gas, gas_limit)` where `value + max_fee_per_gas * gas_limit` overflows U256, the result must be `TxError::Validation`, not `TxError::Internal`.
