Audit Report

## Title
Unchecked u64 Underflow in `metrics_by_subnet()` Corrupts Node Provider Reward Calculations - (File: rs/node_rewards/canister/src/metrics.rs)

## Summary
`MetricsManager::metrics_by_subnet()` computes daily block activity by subtracting a stored baseline from the current cumulative totals using plain `u64` arithmetic. The `BlockmakerMetricsTimeSeries` explicitly documents and implements counter resets for pruned nodes, meaning the current total can be less than the stored baseline. The resulting wrapping underflow in Wasm produces a value near `u64::MAX`, permanently corrupting failure-rate and reward calculations for the affected node provider and all other node providers in the same subnet.

## Finding Description

**Root cause — plain u64 subtraction with no underflow guard:**

In `rs/node_rewards/canister/src/metrics.rs` lines 236–241, `metrics_by_subnet()` computes daily block counts by subtracting the previous day's stored baseline from the current cumulative total:

```rust
NodeMetricsDailyRaw {
    node_id: NodeId::from(node.node_id.unwrap()),
    num_blocks_proposed: node.num_blocks_proposed_total
        - initial_proposed_total,   // plain u64 subtraction
    num_blocks_failed: node.num_blocks_failed_total - initial_failed_total,
}
``` [1](#0-0) 

**Precondition — documented counter reset on node re-entry:**

`BlockmakerMetricsTimeSeries` explicitly documents and implements pruning of nodes whose stats are unchanged between two consecutive daily snapshots. The documentation at lines 1819–1822 states: *"If such a node reappears later on, it will be added as a new node with restarted stats."* [2](#0-1) 

The pruning logic at lines 1887–1894 retains only nodes with changed stats: [3](#0-2) 

When the pruned node reappears, `or_default()` restarts its counters from zero: [4](#0-3) 

**Exploit flow:**
1. Node A proposes 100 blocks on Day 1. `MetricsManager` stores `initial_proposed_total = 100` at timestamp T1.
2. Node A is offline on Day 2. Its stats remain at (100, 0). At the Day 3 boundary, `BlockmakerMetricsTimeSeries` prunes Node A because its stats equal the Day 1 snapshot.
3. Node A comes back online on Day 3 and proposes 1 block. `node_metrics_history` now reports `num_blocks_proposed_total = 1` (restarted from 0).
4. `metrics_by_subnet()` computes: `1u64 - 100u64 = 18446744073709551517` (wrapping underflow in Wasm — no trap).

**Why existing checks are insufficient:**

The `initial_total_metrics` map uses `unwrap_or_default()` (line 233–235) which correctly handles the case where no baseline exists (returns 0), but provides no protection when the baseline *does* exist and is larger than the current value. [5](#0-4) 

The corrupted value is then fed into `calculate_daily_node_failure_rate`: [6](#0-5) 

With `proposed ≈ u64::MAX` and `failed = 0`, the failure rate computes to `0 / u64::MAX = 0.0` (perfect performance), causing the node to appear as a perfect performer. Additionally, at line 267, `metric.num_blocks_proposed + metric.num_blocks_failed` overflows if `failed > 0`, producing a nonsensical deduplication result. [7](#0-6) 

The stored baseline in `last_timestamp_per_subnet` advances past the pruning event (lines 112–117), so subsequent days continue computing deltas from the wrong baseline — the corruption is permanent. [8](#0-7) 

## Impact Explanation

The corrupted `num_blocks_proposed ≈ u64::MAX` causes the affected node to appear as a perfect performer (failure rate = 0). This distorts the subnet-level 75th-percentile failure rate used to compute `adjusted_rewards_xdr_permyriad` for **all** node providers in the subnet. The affected node provider receives inflated rewards (better than deserved), and the subnet percentile distortion can cause all other node providers in the subnet to also receive incorrect rewards. The corruption persists across all subsequent reward periods until manually corrected. This constitutes a **High** severity impact: significant NNS/governance financial impact with concrete, permanent harm to the ICP node provider reward system.

## Likelihood Explanation

A node operator can deterministically trigger this by taking their node offline for exactly one full day (causing stats to be unchanged between two consecutive daily snapshots, satisfying the pruning condition), then bringing it back online. This is a routine operational scenario (maintenance, upgrades, network issues). No privileged access beyond operating a node is required. The pruning behavior is deterministic and documented. The attack is repeatable and low-cost.

## Recommendation

Replace the plain `u64` subtraction with saturating subtraction, treating a decrease in cumulative totals as a counter reset (daily delta = current total):

```rust
num_blocks_proposed: node.num_blocks_proposed_total
    .saturating_sub(initial_proposed_total),
num_blocks_failed: node.num_blocks_failed_total
    .saturating_sub(initial_failed_total),
```

Alternatively, detect the decrease explicitly and use the full current-day total as the daily delta when `current < baseline`. Also fix the `proposed + failed` addition in `calculate_daily_node_failure_rate` to use `saturating_add` to prevent secondary overflow.

## Proof of Concept

A deterministic unit test in `rs/node_rewards/canister/src/metrics.rs` (using the existing `management_canister_client_test` mock infrastructure):

1. Inject a stored baseline of `initial_proposed_total = 100` for Node A at timestamp T1.
2. Inject a current-day record with `num_blocks_proposed_total = 1` for Node A at timestamp T3 (simulating counter restart after pruning).
3. Call `metrics_by_subnet()` for Day 3.
4. Assert that `num_blocks_proposed` equals `18446744073709551517` (wrapping underflow), confirming the bug.
5. Pass the result to `calculate_daily_node_failure_rate(18446744073709551517, 0)` and assert the failure rate is `0` (incorrect perfect performance).
