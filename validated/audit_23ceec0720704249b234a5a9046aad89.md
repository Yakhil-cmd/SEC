Audit Report

## Title
Unsolicited Pong Disconnects Any Post-Handshake Connection — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary
`process_pong_message` unconditionally calls `conn.disconnect()` when the connection's `ping_state` is `PingState::Idle`. Because every post-handshake connection rests in `PingState::Idle`, any Bitcoin peer that completes the version handshake can immediately tear down its own connection by sending a single unsolicited `Pong` message. An attacker operating or controlling malicious Bitcoin nodes can exploit this repeatedly to hold the adapter at zero active connections.

## Finding Description
In `connectionmanager.rs`, `process_pong_message` (L517–531) matches on `conn.ping_state()`. The `PingState::Idle` arm evaluates `valid_pong = false`, which falls through to `conn.disconnect()` with no further guard:

```rust
PingState::Idle { last_pong_at: _ } => false,
// ...
} else {
    conn.disconnect();  // L530
};
```

`conn.disconnect()` (`connection.rs` L189–192) sets `state = ConnectionState::NodeDisconnected` and aborts the async task. `reap_disconnected()` (L279–299) then removes the entry from `self.connections` and calls `address_book.remove_from_active()`.

The dispatch path in `process_bitcoin_network_message` (L694) routes `NetworkMessage::Pong` directly to `process_pong_message` with no prior check on connection state. There is no guard requiring the connection to be in `PingState::ExpectingPong` before the disconnect branch is reached.

The normal resting `ping_state` for every post-handshake connection is `PingState::Idle`, set at construction (`connection.rs` L116–118) and reset after every valid pong (`connection.rs` L162–165). Therefore the `Idle` branch is the steady-state for all live connections, and any nonce value triggers the disconnect.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *Application/platform-level DoS, subnet availability impact not based on raw volumetric DDoS.* An attacker who operates or can intercept traffic to Bitcoin nodes can complete the public, unauthenticated Bitcoin version handshake, then immediately send a `Pong` message to disconnect the adapter's connection. By repeating this across every new connection the adapter establishes, the attacker can hold the Bitcoin adapter at zero active connections indefinitely. This prevents the IC's Bitcoin integration (ckBTC / Chain Fusion) from relaying Bitcoin data to the subnet — a concrete, sustained availability impact on an in-scope financial integration component.

## Likelihood Explanation
The precondition is realistic and low-cost: completing the Bitcoin version handshake requires no privileged access, key material, or majority corruption. Any party operating a Bitcoin node (or performing BGP-level interception of Bitcoin traffic) can complete the handshake and immediately send `Pong`. The attack is repeatable: each time the adapter reconnects, the attacker completes a new handshake and sends another `Pong`. No victim mistake or social engineering is required.

## Recommendation
In `process_pong_message`, treat an unsolicited `Pong` (i.e., `PingState::Idle`) as a no-op rather than a disconnect trigger:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong — ignore silently.
    return Ok(());
}
```

Only disconnect when the nonce does not match an outstanding `ExpectingPong` state. Optionally, log a warning for observability. If penalizing misbehaving peers is desired, `conn.discard()` (which blacklists the address) is more appropriate than `conn.disconnect()`, but the primary fix is to not disconnect on an unsolicited `Pong`.

## Proof of Concept
State-machine unit test (no network required):

1. Construct a `ConnectionManager` and inject a connection in `ConnectionState::HandshakeComplete` with `PingState::Idle`.
2. Call `process_bitcoin_network_message(addr, &NetworkMessage::Pong(any_nonce))`.
3. Assert `conn.state() == ConnectionState::NodeDisconnected` — confirmed by `process_pong_message` L522 → L530.
4. Call `reap_disconnected()`.
5. Assert `self.connections` is empty and `address_book.remove_from_active` was called.

    fn reap_disconnected(&mut self) {
        let mut disconnects = vec![];
        for (addr, conn) in self.connections.iter() {
            match conn.state() {
                ConnectionState::AdapterDiscarded => {
                    self.address_book.discard(conn.address_entry());
                }
                ConnectionState::NodeDisconnected => {
                    self.address_book.remove_from_active(conn.address_entry());
                }
                _ => {}
            }

            if conn.is_disconnected() {
                disconnects.push(*addr);
            }
        }

        for addr in disconnects {
            self.connections.remove(&addr);
        }
