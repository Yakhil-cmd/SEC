Audit Report

## Title
VIP Callers Blocked by Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` unconditionally rejects all callers when `available_slot_count == 0`, including VIP (NNS canister) callers that are explicitly designed to consume zero slots. An unprivileged attacker can exhaust all 167 shared slots via the public `canister_status` endpoint on NNS Root, causing governance-triggered `take_canister_snapshot` and `load_canister_snapshot` calls to fail. When the target is the Governance canister, the failure is silent: the caller receives a false `Ok` while the actual operation fails in the background.

## Finding Description
In `try_borrow_slot` at `rs/nervous_system/clients/src/management_canister_client.rs` lines 264–287, `used_slot_count` is correctly set to `0` for VIP callers, but the early-exit guard `if *available_slot_count == 0` fires unconditionally before the subtraction, blocking VIPs just as it blocks non-VIPs:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };
    self.available_slot_count.with_borrow_mut(|available_slot_count| {
        if *available_slot_count == 0 {   // ← fires for VIPs too
            return Err((RejectCode::SysTransient as i32, "Unavailable...".to_string()));
        }
        *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
        Ok(())
    })?;
    ...
}
``` [1](#0-0) 

The shared slot pool is a single `thread_local` initialized to 167, used by all callers to NNS Root: [2](#0-1) 

VIP status (NNS canister caller) is determined at call time in `new_management_canister_client()`, and both `take_canister_snapshot` and `load_canister_snapshot` call this function, sharing the same pool: [3](#0-2) [4](#0-3) 

The `canister_status` endpoint has no access control, allowing any principal to exhaust slots: [5](#0-4) 

The exploit path through `load_canister_snapshot` / `take_canister_snapshot` is:
1. `perform_offline_canister_maintenance` stops the target canister (`stop_before = true`)
2. `do_the_real_work` calls `management_canister_client.load_canister_snapshot(args)` → `try_borrow_slot` → `Err(SysTransient)` when slots are exhausted
3. `restart_after_main_operation` restarts the canister, but the snapshot operation itself has failed [6](#0-5) [7](#0-6) 

For the special case where the target is the Governance canister, the entire operation is spawned in the background and an optimistic `Ok` is returned immediately to the caller, meaning the failure is completely silent: [8](#0-7) 

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant NNS … security impact with concrete user or protocol harm."*

An unprivileged attacker can permanently block NNS governance's ability to take or restore canister snapshots — operations that are the primary disaster recovery mechanism for NNS canisters. The silent false-`Ok` path for Governance-as-target is especially dangerous: during a disaster recovery scenario, Governance would believe the restore succeeded while the canister remains in its pre-restore state, potentially causing governance to proceed on incorrect assumptions.

## Likelihood Explanation
The attack requires no privileged access, no key material, and no consensus corruption. The `canister_status` endpoint is a public `#[update]` with no rate limiting or per-caller accounting. Sustaining 167 concurrent in-flight calls is achievable by any attacker controlling one or more canisters: each call suspends at the management canister await, allowing NNS Root to accept the next message in the same round. The attacker must continuously replenish calls as slots free up, making this a low-cost sustained attack. The attack is repeatable and deterministic.

## Recommendation
Fix the guard in `try_borrow_slot` to skip the zero-check for VIP callers:

```rust
if *available_slot_count == 0 && !self.is_caller_vip {
    let code = RejectCode::SysTransient as i32;
    let message = "Unavailable. Maybe, try again later?".to_string();
    return Err((code, message));
}
``` [9](#0-8) 

## Proof of Concept
The existing test infrastructure in `rs/nervous_system/clients/src/management_canister_client/tests.rs` already demonstrates the slot exhaustion pattern and can be extended directly: [10](#0-9) 

Minimal test extension:
1. Set slot capacity to 2 (as in the existing test).
2. Launch 2 non-VIP `canister_status` calls that hold their slots for 500 ms (slots now at 0).
3. While those are in flight, launch a VIP (`is_caller_vip = true`) `canister_status` call.
4. Assert the VIP call returns `Err((SysTransient, _))` — demonstrating the bug.
5. Apply the fix and assert the VIP call returns `Ok`.

For the full NNS integration scenario, a state-machine test can: deploy an attacker canister that sends 167 concurrent calls to NNS Root's `canister_status`, assert `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT == 0`, then have Governance call `load_canister_snapshot` on NNS Root and assert the response is `LoadCanisterSnapshotResponse::Err` with code `SysTransient`.
