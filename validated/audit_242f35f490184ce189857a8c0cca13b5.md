Audit Report

## Title
Unbounded CPU Cost in Delegation Chain Validation via COSE RSA-8192 Keys — (`rs/validator/src/ingress_validation.rs`, `rs/crypto/internal/crypto_lib/basic_sig/rsa_pkcs1/src/lib.rs`)

## Summary
An unprivileged ingress sender can construct a delegation chain of up to 20 entries where every signing key is a COSE-wrapped RSA-8192 public key. Each hop forces the replica to perform a full RSA-8192 PKCS#1 v1.5 modular exponentiation in `verify_pkcs1_sha256`. No per-chain CPU budget exists; the only guard is a chain-length cap of 20 and a byte-size check, neither of which limits computational cost. Flooding with such requests saturates the `spawn_blocking` thread pool and degrades subnet ingress validation throughput.

## Finding Description
The call chain is confirmed by the code:

1. `validate_signature` calls `validate_sender_delegation_length` (caps at 20, no cost check) then `validate_delegations`, which loops over all `signed_delegations` calling `validate_delegation` per hop. [1](#0-0) [2](#0-1) 

2. Inside `validate_delegation`, a pubkey typed `RsaSha256PublicKeyDerWrappedCose` is routed to `validate_webauthn_sig`, which calls `verifier.verify_basic_sig_by_public_key`, ultimately reaching `verify_pkcs1_sha256`. [3](#0-2) 

3. `verify_pkcs1_sha256` performs an unconditional RSA modular exponentiation with no budget check. [4](#0-3) 

4. RSA-8192 is explicitly the permitted maximum key size. [5](#0-4) 

5. The COSE branch in `user_public_key_from_bytes` parses the inner RSA key via `RsaPublicKey::from_der_spki`. [6](#0-5) 

6. Validation runs in `spawn_blocking` with no concurrency cap at the call site. [7](#0-6) 

7. The only pre-validation guard is a byte-size check, not a computational-cost check. [8](#0-7) 

The existing guards are insufficient: `MAXIMUM_NUMBER_OF_DELEGATIONS = 20` limits chain length but not per-hop cost; the 2 MB size limit is not reached (~44 KB for 20 RSA-8192 keys + signatures); and `spawn_blocking` has no bounded concurrency at this call site.

## Impact Explanation
RSA-8192 verification is roughly 16× more expensive than RSA-2048 and orders of magnitude more expensive than Ed25519. With 20 delegations per request and no CPU budget, each ingress message forces ~20 RSA-8192 modular exponentiations. Concurrent flooding saturates the `spawn_blocking` thread pool, starving legitimate ingress validation and degrading subnet throughput. This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
No privileged access is required; any principal can submit ingress messages with delegations. RSA-8192 key generation and signing is expensive but done once offline; the resulting request is replayable within TTL or re-submitted with a fresh expiry. The attack payload (~44 KB) is well within the 2 MB ingress limit. No rate-limiting on per-request cryptographic cost exists in the validation path.

## Recommendation
1. **Enforce a per-chain CPU budget**: Assign weights (e.g., Ed25519 = 1, ECDSA-P256 = 2, RSA-2048 = 50, RSA-8192 = 200) and reject chains exceeding a threshold before performing any verification.
2. **Restrict RSA key sizes in delegations**: Cap delegation keys at RSA-2048 or RSA-4096, or disallow RSA COSE keys in delegation chains entirely (they are a WebAuthn-specific feature primarily needed for the final signing key, not intermediate delegation keys).
3. **Add concurrency limiting** on the `spawn_blocking` ingress validation pool with back-pressure to prevent CPU saturation.

## Proof of Concept
```rust
// Build a delegation chain of `n` COSE RSA-8192 keys and submit as ingress.
// Each validate_delegation call invokes verify_pkcs1_sha256 with an 8192-bit modulus.
fn build_rsa8192_delegation_chain(n: usize) -> Vec<SignedDelegation> {
    let mut keys: Vec<RsaKeyPair8192> = (0..=n).map(|_| RsaKeyPair8192::generate()).collect();
    let mut delegations = Vec::new();
    for i in 0..n {
        let delegation = Delegation {
            pubkey: keys[i+1].cose_der_wrapped_pubkey(),
            expiry: far_future(),
        };
        let sig = keys[i].sign_delegation(&delegation);
        delegations.push(SignedDelegation { delegation, signature: sig });
    }
    delegations
}
// Submit ingress with sender_pubkey = keys[0].cose_der_wrapped_pubkey()
// and sender_delegation = build_rsa8192_delegation_chain(20).
// Benchmark wall-clock time vs an equivalent Ed25519 chain;
// ratio should exceed 100x, confirming the computational asymmetry.
// Flood with concurrent submissions to demonstrate spawn_blocking pool saturation.
```

    pub fn verify_pkcs1_sha256(&self, message: &[u8], signature: &[u8]) -> CryptoResult<()> {
        let digest = Sha256::hash(message);

        match &self
            .key
            .verify(Pkcs1v15Sign::new::<sha2::Sha256>(), &digest, signature)
        {
            Ok(_) => Ok(()),
            Err(e) => Err(CryptoError::SignatureVerification {
                algorithm: AlgorithmId::RsaSha256,
                public_key_bytes: self.as_der().to_vec(),
                sig_bytes: signature.to_vec(),
                internal_error: format!("{e:?}"),
            }),
        }
    }
