Audit Report

## Title
Pre-Bounds-Check Heap Allocation in RSA COSE Key Parsing via `spki_from_components` — (`rs/crypto/internal/crypto_lib/basic_sig/rsa_pkcs1/src/lib.rs`)

## Summary

`RsaPublicKey::from_components` calls `spki_from_components`, which allocates a `BigInt` and two DER-encoded `Vec<u8>` buffers each proportional to the raw `n` byte slice, before the modulus size check in `from_der_spki` is ever reached. An unauthenticated sender can embed an oversized RSA modulus in a COSE-encoded public key inside an ingress message and trigger this allocation path on every replica that validates the request. The per-request heap amplification is bounded by the 2 MB ingress cap, making this a resource-amplification issue rather than a direct OOM crash.

## Finding Description

**Root cause — allocation before bounds check:**

`from_components` unconditionally calls `spki_from_components` and only then calls `from_der_spki`, which is where the modulus size guard lives: [1](#0-0) 

Inside `spki_from_components`, three heap allocations proportional to `|n|` occur before any length guard: [2](#0-1) 

The size check (`MAXIMUM_RSA_KEY_SIZE = 8192` bits) only fires inside `from_der_spki` after `spki_from_components` has already returned: [3](#0-2) 

**Reachable unauthenticated call chain:**

The COSE parser extracts `n` as a raw `Vec<u8>` from the CBOR map and passes it directly to `from_components` with no prior length check: [4](#0-3) 

`parse_cose_public_key` is reached from `user_public_key_from_bytes` in the COSE branch: [5](#0-4) 

Which is called by `public_key_from_bytes` in the ingress validator: [6](#0-5) 

Which is called from `validate_signature` during `validate_user_id_and_signature` — the unauthenticated signature validation step: [7](#0-6) 

The ingress size check at the HTTP handler fires before `validate_request` is called, but only caps the total message size: [8](#0-7) 

**Why existing guards are insufficient:**

The ingress size cap (`MAX_INGRESS_BYTES_PER_MESSAGE_APP_SUBNET = 2 MB`) bounds the maximum attacker-controlled `|n|` to roughly 2 MB, but does not prevent the allocation — it only limits its magnitude per request: [9](#0-8) 

The `MAXIMUM_RSA_KEY_SIZE` constant (8192 bits = 1024 bytes) is checked only after the three proportional allocations in `spki_from_components` have already occurred: [10](#0-9) 

## Impact Explanation

Each request with a ~2 MB `n` causes approximately 3× `|n|` (~6 MB) of transient heap allocation in `spki_from_components` before the `MalformedPublicKey` error is returned. The validation runs in a `spawn_blocking` thread pool and is stateless, so an attacker can pipeline concurrent requests. At 100 concurrent requests each carrying a ~2 MB modulus, the transient heap overhead is ~600 MB. This constitutes a **limited subnet availability impact** — sustained pipelining can degrade replica memory headroom and slow processing, but the 2 MB ingress cap prevents a single-request OOM crash.

This matches: **Medium ($200–$2,000) — limited subnet availability impact**.

## Likelihood Explanation

The path requires no authentication, no privileged role, and no victim interaction. A single crafted HTTP POST to `/api/v2/canister/{id}/call` with a COSE RSA key embedding a ~2 MB modulus is sufficient to trigger the allocation. The attack is fully reproducible and repeatable. The only constraint is the 2 MB ingress cap, which bounds per-request impact but does not prevent pipelining.

## Recommendation

Add an explicit byte-length guard in `from_components` before calling `spki_from_components`:

```rust
pub fn from_components(e: &[u8], n: &[u8]) -> CryptoResult<Self> {
    // MAXIMUM_RSA_KEY_SIZE is in bits; n is big-endian bytes
    if n.len() > Self::MAXIMUM_RSA_KEY_SIZE / 8 + 1 {
        return Err(CryptoError::MalformedPublicKey {
            algorithm: AlgorithmId::RsaSha256,
            key_bytes: None,
            internal_error: "RSA modulus too large".to_string(),
        });
    }
    let der = Self::spki_from_components(e, n)?;
    Self::from_der_spki(&der)
}
```

This makes the check O(1) and eliminates all proportional allocation for oversized inputs before any heap-intensive work begins. [1](#0-0)
