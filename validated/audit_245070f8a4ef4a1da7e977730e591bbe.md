### Title
EvmErc20V2.withdrawToNear Permanently Burns Tokens Without Scheduling NEAR Transfer When Compiled Without `error_refund` - (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

`EvmErc20V2.withdrawToNear` prepends the 20-byte `sender` address before the 32-byte `amount` in the precompile input. When the engine is compiled without the `error_refund` feature, the `ExitToNear` precompile's `parse_input` strips only the 1-byte flag, leaving `[sender_20_bytes][amount_32_bytes][recipient]`. The precompile then reads `input[..32]` — which is `sender_20_bytes || amount_high_12_bytes` — as the amount. For virtually every real Ethereum address this value exceeds `u128::MAX`, causing `parse_amount` to return `ERR_INVALID_AMOUNT`. The precompile call fails silently (the assembly `call` return value is unchecked), while `_burn` has already committed. All burned tokens are permanently lost.

### Finding Description

**V1 (`EvmErc20.sol`) input layout:**
```
[0x01][amount_32_bytes][recipient_bytes]
``` [1](#0-0) 

**V2 (`EvmErc20V2.sol`) input layout:**
```
[0x01][sender_20_bytes][amount_32_bytes][recipient_bytes]
``` [2](#0-1) 

**Precompile parsing — without `error_refund`:**

`parse_input` strips only the flag byte and returns `input[1..]`: [3](#0-2) 

For flag `0x1`, the precompile then reads `input[..32]` as the amount: [4](#0-3) 

With V2's layout, `input[..32]` = `sender_20_bytes || amount_high_12_bytes`. Interpreted as a big-endian U256 this equals `sender_as_uint160 × 2^96 + (amount >> 160)`. For any address where `sender_as_uint160 > 2^32` (i.e., virtually every real address), this exceeds `u128::MAX`: [5](#0-4) 

The precompile returns `ERR_INVALID_AMOUNT`. The EVM `call` opcode returns `0` (failure), but `withdrawToNear` never checks `res`: [6](#0-5) 

`_burn` has already executed at line 55, so the tokens are gone with no NEAR-side transfer scheduled. [7](#0-6) 

**Why `error_refund` fixes it:**

With `error_refund` enabled, `parse_input` reads `input[1..21]` as the `refund_address` (which happens to be the sender — correct semantics) and returns `input[21..]` = `[amount_32_bytes][recipient_bytes]`. The amount is then parsed correctly from `input[..32]`. [8](#0-7) 

### Impact Explanation

Every call to `EvmErc20V2.withdrawToNear` on an engine compiled without `error_refund` results in the complete and permanent loss of the withdrawn amount. The tokens are burned on the EVM side but no `ft_transfer` promise is ever scheduled on the NEAR side. There is no recovery path: no refund mechanism exists without `error_refund`, and the burn is irreversible. This is **Critical — Permanent freezing of funds**.

### Likelihood Explanation

- `EvmErc20V2` is a production contract in the repository, distinct from `EvmErc20` (V1).
- Any token holder with a balance > 0 can trigger this by calling `withdrawToNear` — no special privileges required.
- The `error_refund` feature is a compile-time flag; deployments without it are a supported configuration (the codebase explicitly handles `#[cfg(not(feature = "error_refund"))]`).
- The failure is silent: the transaction succeeds from the caller's perspective (no revert), making it easy to miss.

### Recommendation

In `EvmErc20V2.withdrawToNear`, the input must match the format the precompile actually expects for the active feature configuration:

- **Without `error_refund`**: use V1's layout — `[0x01][amount_32_bytes][recipient]` (drop the `sender` prefix entirely).
- **With `error_refund`**: the current V2 layout `[0x01][sender_20_bytes][amount_32_bytes][recipient]` is correct, as `sender` is consumed as the `refund_address`.

Additionally, the assembly `call` return value should be checked and the function should revert on failure to prevent silent token loss.

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }

    Ok(amount)
}
