## Analysis

The code behavior described is real. Let me trace it precisely.

### Loop Logic in `get_allowances` [1](#0-0) 

The iteration order is:
1. `continue` if this is the pagination start entry (line 1198–1199)
2. **`break` if `result.len() >= max_results`** (line 1201–1202)
3. `break` if past the `from` account owner (line 1204–1205)
4. **`continue` if expired** (line 1207–1210) — does NOT increment `result.len()`
5. `result.push(...)` for valid entries

The expiry `continue` at step 4 fires **after** the `max_results` break at step 2. Expired entries are skipped without contributing to `result.len()`, so the `max_results` guard never fires on them. The loop iterates through every expired entry for the account before collecting `max_results` non-expired ones.

### Attacker Precondition

The `approve` function rejects allowances already expired at submission time: [2](#0-1) 

So the attacker must set `expires_at = now + ε`, wait for expiry, then query. Each `(account, spender)` pair holds exactly one allowance, so N distinct spender accounts are needed. Each approval requires paying a ledger fee.

### No Per-Account Allowance Cap

There is no hard cap on the number of allowances a single account can hold. `MAX_TAKE_ALLOWANCES` (500) only limits the number of **results returned**, not the number of entries **iterated**. [3](#0-2) 

The `prune` function exists but is called with a bounded `limit` during `icrc2_approve` operations — it does not eagerly clean up all expired entries. [4](#0-3)

### Title
Unbounded Iteration Over Expired Allowances in `get_allowances` Violates `MAX_TAKE_ALLOWANCES` Cost Bound — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

### Summary
The `get_allowances` function iterates through all expired allowances for a given `from_account` before collecting `max_results` non-expired ones. Because the expiry `continue` (line 1210) fires after the `max_results` break check (line 1201), expired entries consume loop iterations without being counted toward the result limit. An attacker who pre-populates N expired allowances for a single account forces every subsequent `icrc103_get_allowances` query for that account to scan all N entries, with cost growing linearly in N and unbounded by `MAX_TAKE_ALLOWANCES`.

### Finding Description
In `get_allowances` (`lib.rs:1174–1222`), the loop body checks `result.len() >= max_results` before checking expiry. Expired entries hit the `continue` at line 1210 and are skipped, but they do not increment `result.len()`. Therefore the `max_results` break never fires on expired entries — the loop must exhaust all expired entries for the account before it can accumulate enough non-expired entries to break. With N expired entries and M non-expired entries (M ≥ max_results), the loop performs N + max_results iterations instead of the intended max_results iterations.

The same flaw exists in the ICP ledger's `get_allowances_list` (`rs/ledger_suite/icp/ledger/src/lib.rs:657–680`).

### Impact Explanation
`icrc103_get_allowances` is a `#[query]` endpoint callable by any principal without authentication. IC query calls are bounded by a 5-billion-instruction limit. Each stable-memory B-tree range iteration costs on the order of thousands of instructions. An attacker who accumulates hundreds of thousands of expired allowances for a single account can force every query call for that account to consume a disproportionate instruction budget, degrading or trapping legitimate queries for that account. This is a non-volumetric canister query exhaustion: a single well-crafted state causes repeated query degradation without ongoing attacker participation.

### Likelihood Explanation
The attack requires:
- N distinct spender accounts (achievable with N principals)
- Paying a ledger fee per allowance (economically bounded but not prohibitive at scale)
- Waiting for allowances to expire (requires setting `expires_at` slightly in the future)

Once the expired allowances are in stable memory, the attacker needs no further action — every query call for that account pays the cost. Pruning only occurs during `icrc2_approve` calls with a bounded `limit`, so expired entries persist indefinitely unless the account owner actively prunes them.

### Recommendation
Move the expiry check **before** the `max_results` check, or introduce a separate total-iterations counter that counts expired entries toward the loop budget:

```rust
// Option A: count expired entries toward the iteration budget
let mut iterations = 0usize;
for (...) in allowances.range(...) {
    if iterations >= max_results as usize * EXPIRED_MULTIPLIER_CAP {
        break;
    }
    iterations += 1;
    if result.len() >= max_results as usize { break; }
    if account_spender.account.owner != from.owner { break; }
    if expired { continue; }
    result.push(...);
}

// Option B: eagerly prune expired entries during the scan
```

The simplest correct fix is to add a total-scan-limit (e.g., `2 * max_results` or a fixed constant) that breaks the loop regardless of whether entries are expired.

### Proof of Concept
1. Fund account `A` with sufficient tokens.
2. Create N allowances from `A` to N distinct spender accounts, each with `expires_at = now + 1_000_000_000` (1 second).
3. Wait 1 second for all allowances to expire.
4. Repeatedly call `icrc103_get_allowances({ from_account: A, prev_spender: None, take: None })`.
5. Measure instruction count via `ic_cdk::api::performance_counter(0)` inside the canister or via replica metrics.
6. Assert instruction count grows linearly with N, far exceeding the `MAX_TAKE_ALLOWANCES = 500` bound.
7. Compare against a baseline with 0 expired allowances and 500 non-expired ones — baseline should use ~500 iterations; attack case uses N + 500 iterations.

    pub fn prune(&mut self, now: TimeStamp, limit: usize) -> usize {
        self.with_postconditions_check(|table| {
            let mut pruned = 0;
            for _ in 0..limit {
                match table.allowances_data.first_expiry() {
                    Some((ts, _key)) => {
                        if ts > now {
                            return pruned;
                        }
                    }
                    None => {
                        return pruned;
                    }
                }
                if let Some((_, (account, spender))) = table.allowances_data.pop_first_expiry() {
                    let key = (account, spender);
                    if let Some(allowance) = table.allowances_data.get_allowance(&key)
                        && allowance.expires_at.unwrap_or_else(remote_future) <= now
                    {
                        table.allowances_data.remove_allowance(&key);
                        pruned += 1;
                    }
                }
            }
            pruned
        })
    }
