### Title
Missing EIP-7610 Storage Collision Check Allows CREATE/CREATE2 Deployment at Addresses with Non-Empty Storage — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's `before_executing_frame` function checks for deployment collisions using only `deployee_code_len != 0 || deployee_nonce != 0`, explicitly omitting the EIP-7610 requirement to also reject deployments at addresses with non-empty storage. An unprivileged attacker can pre-populate storage at a deterministic CREATE2 address (by deploying a contract that writes storage and self-destructs in the same transaction), then re-deploy to that address. ZKsync OS allows the re-deployment to succeed, while Ethereum would reject it. The re-deployed contract inherits the attacker-planted storage, causing an EVM semantic mismatch and enabling unexpected contract behavior.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` function performs the deployment collision check:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    tracer.evm_tracer().on_call_error(&EvmError::CreateCollision);
    return Ok(false);
}
```

The comment explicitly acknowledges the missing check. EIP-7610 (standardized in Paris/Cancun) requires that CREATE and CREATE2 must also fail if the target address has non-empty storage. ZKsync OS omits this check entirely.

The attack path exploits EIP-6780 (Cancun SELFDESTRUCT semantics): when SELFDESTRUCT is called in the same transaction as contract creation, the contract's code, nonce, and balance are cleared, but **storage is not cleared**. This leaves an address with zero code, zero nonce, but non-zero storage — exactly the state that EIP-7610 is designed to guard against.

The disabled test cases confirm this is a real, reproducible divergence:

- `create2collisionStorageParis.json` — `enabled: false`, comment: `"We do not check storage for collision"`
- `RevertInCreateInInitCreate2Paris.json` — `enabled: false`, comment: `"We do not check for storage collisions"`
- `test_init_collision_create_opcode.json` (EIP-7610 suite) — `enabled: false`, comment: `"We do not check the storage"`
- `test_init_collision_create_tx.json` (EIP-7610 suite) — `enabled: false`, comment: `"We do not check the storage"`

---

### Impact Explanation

**EVM semantic mismatch / state-transition bug.** An attacker can:

1. Deploy a contract at a deterministic CREATE2 address (attacker-controlled salt), write attacker-chosen values to storage slots, then call SELFDESTRUCT in the same transaction. After the transaction, the address has zero code, zero nonce, but non-zero storage.
2. Re-deploy to the same address via CREATE2 with the same salt.
3. ZKsync OS allows the re-deployment (code and nonce are zero, so no collision is detected). The new contract's constructor and all subsequent calls observe the attacker-planted storage values.

A victim protocol that uses CREATE2 for deterministic deployment (e.g., factory patterns, counterfactual wallets, proxy deployers) can have its freshly deployed contract start with corrupted storage state. Depending on the contract logic, this can lead to:

- Incorrect initialization (e.g., `initialized` flag already set, bypassing `initializer` guards)
- Unexpected token balances or allowances pre-set in storage
- Ownership or access-control slots pre-populated to attacker-controlled addresses
- Divergence between ZKsync OS state and Ethereum state for the same transaction sequence

---

### Likelihood Explanation

The attack is fully unprivileged and deterministic. Any user can:
- Compute a CREATE2 address for any deployer/salt/initcode combination
- Execute the deploy-write-selfdestruct sequence in a single transaction
- Re-deploy to the same address in a subsequent transaction

The only constraint is that the victim must use CREATE2 with a salt/initcode combination the attacker can predict or influence. Factory patterns and counterfactual wallet deployments are common targets. The attack requires no special privileges, no oracle manipulation, and no governance access.

---

### Recommendation

Add a storage non-emptiness check to the deployment collision guard in `before_executing_frame`. Before allowing a constructor frame to execute, verify that the target address has no non-zero storage slots. If the storage model cannot enumerate all slots efficiently, a per-address "has-ever-had-storage" flag (set on first SSTORE, cleared only on full account deletion) would be sufficient to implement the EIP-7610 guard without a full storage scan.

The check in `evm_interpreter/src/ee_trait_impl.rs` should be extended from:

```rust
if deployee_code_len != 0 || deployee_nonce != 0 { ... }
```

to also include a storage-non-empty predicate, consistent with EIP-7610.

---

### Proof of Concept

**Transaction 1 (attacker):**
```solidity
// Deployed via CREATE2(deployer, salt, initcode)
contract Poisoner {
    constructor() {
        assembly {
            sstore(0, 0xdeadbeef)   // plant storage slot 0
            selfdestruct(caller())  // clear code+nonce, storage persists (EIP-6780)
        }
    }
}
```
After tx 1: `address(CREATE2(deployer, salt, keccak(initcode)))` has `code = 0`, `nonce = 0`, `storage[0] = 0xdeadbeef`.

**Transaction 2 (victim, or attacker targeting victim's factory):**
```solidity
// Same deployer, same salt, same initcode → same address
contract Victim {
    uint256 public initialized = 0; // expects 0 on fresh deploy
    constructor() {
        require(initialized == 0, "already initialized"); // FAILS on Ethereum, PASSES on ZKsync OS
        initialized = 1;
    }
}
```

On Ethereum (EIP-7610): tx 2 reverts — CREATE2 collision due to non-empty storage.
On ZKsync OS: tx 2 succeeds — `initialized` slot reads `0xdeadbeef` (not 0), `require` fails inside constructor, but the constructor itself was entered. More critically, any contract that reads storage before writing it will observe the attacker-planted values.
