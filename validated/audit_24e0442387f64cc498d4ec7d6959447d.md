### Title
Missing Callback for Partial `ft_on_transfer` Return in Base ETH Omni Exit Path — (`engine-precompiles/src/native.rs`)

### Summary

When an EVM caller uses `ExitToNear` with `flag=0x0` (base ETH) and an Omni message (e.g., `receiver.near:some_msg`), the precompile issues a `ft_transfer_call` on the eth connector NEP-141 contract but schedules **no callback**. If the Omni receiver's `ft_on_transfer` returns a non-zero unused amount, the eth connector's `ft_resolve_transfer` silently refunds those tokens back to Aurora's own eth connector NEP-141 balance. The EVM caller's ETH was already debited in full, and there is no mechanism to credit the unused amount back. The refunded tokens are permanently inaccessible to the original caller.

### Finding Description

**Step 1 — Entry point.** Any EVM transaction can call the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with:
- `input[0] = 0x00` (base ETH flag)
- `input[1..] = "receiver.near:some_omni_msg"` (Omni path)
- `context.apparent_value > 0` (ETH attached) [1](#0-0) 

**Step 2 — `exit_base_token_to_near` Omni branch.** The function matches `Some(Message::Omni(msg))` and returns `transfer_near_args = None` and method `"ft_transfer_call"`. [2](#0-1) 

**Step 3 — `callback_args` evaluates to `default()`.** Back in `run`, with `error_refund` disabled, `refund = None` is hardcoded. Combined with `transfer_near = None`, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. [3](#0-2) 

`ExitToNearPrecompileCallbackArgs` derives `Default` with both fields `None`: [4](#0-3) 

**Step 4 — No callback is scheduled.** The guard at line 470 fires the `PromiseArgs::Create` branch (no callback), so Aurora never receives the `ft_transfer_call` result. [5](#0-4) 

**Step 5 — NEP-141 `ft_transfer_call` partial return.** The eth connector calls `receiver.near.ft_on_transfer(sender=aurora, amount, msg)`. If `ft_on_transfer` returns `unused_amount > 0`, the eth connector's internal `ft_resolve_transfer` refunds `unused_amount` back to Aurora's eth connector NEP-141 balance. The promise result is still `Successful`. Aurora is never notified.

**Step 6 — Accounting invariant broken.** The EVM caller's ETH was debited by the full `context.apparent_value`. Only `amount - unused_amount` reached the receiver. The `unused_amount` now sits in Aurora's eth connector NEP-141 balance with no corresponding EVM credit. It is permanently inaccessible to the original caller.

**Note on `error_refund` enabled:** Even when the feature is enabled, the callback only handles full failure (`refund`) or wNEAR unwrap (`transfer_near`). A partial success (non-zero unused amount) still results in `PromiseResult::Successful`, and the callback does nothing: [6](#0-5) 

The partial return case is unhandled in both compile configurations.

### Impact Explanation

The EVM caller permanently loses `unused_amount` ETH. The tokens accumulate in Aurora's eth connector NEP-141 balance, inflating it above the sum of all EVM ETH balances. The original caller has no on-chain path to reclaim them. This constitutes **theft of unclaimed yield** (High): the refund the caller is entitled to under the NEP-141 `ft_transfer_call` protocol is silently absorbed by the protocol.

### Likelihood Explanation

The Omni path is the production path for cross-chain transfers via the Omni bridge. Any Omni receiver contract that partially rejects a transfer (e.g., due to fees, slippage limits, or routing failures) will return a non-zero unused amount. This is a normal, expected behavior of the NEP-141 `ft_transfer_call` protocol. Any user bridging base ETH via the Omni path is at risk.

### Recommendation

For the base ETH Omni path, always schedule a callback regardless of whether `error_refund` is enabled. The callback should inspect the `ft_transfer_call` promise result and, if `unused_amount > 0` (detectable from the return value of `ft_resolve_transfer`), credit the unused amount back to the EVM caller's address. Concretely:

- In `exit_base_token_to_near`, when `Message::Omni` is matched, populate `refund` (or a new dedicated field) with the caller's EVM address and the full amount, so `callback_args != default()` and a callback is always scheduled.
- In `exit_to_near_precompile_callback`, on `PromiseResult::Successful`, parse the returned transferred amount and refund the difference to the original EVM caller.

   - caller's EVM ETH balance is NOT credited back 50.
