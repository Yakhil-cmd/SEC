### Title
Pre-EIP-155 Legacy Transaction Replay Across Aurora Chain Deployments - (`engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

### Summary

The Aurora Engine's `submit` entry point accepts pre-EIP-155 legacy Ethereum transactions (those with `v = 27` or `v = 28`). For such transactions, the chain ID is `None`, and the engine's chain ID validation is gated on `Some(chain_id)`, meaning it is silently skipped. Because Aurora is deployed on multiple chains (mainnet chain ID `1313161554`, testnet `1313161555`, local `1313161556`), a valid pre-EIP-155 transaction signed on one deployment can be replayed verbatim on any other deployment by an unprivileged attacker, provided the sender's nonce matches.

### Finding Description

**Root cause — step 1: `chain_id` extraction in `LegacyEthSignedTransaction::sender()`**

`engine-transactions/src/legacy.rs` lines 68–78 decode the `v` field of a legacy signed transaction:

```rust
let (chain_id, rec_id) = match self.v {
    0..=26 | 29..=34 => return Err(Error::InvalidV),
    27..=28 => (None, ...),   // pre-EIP-155: chain_id is None
    _ => (Some((self.v - 35) / 2), ...),
};
```

When `v ∈ {27, 28}`, `chain_id` is `None`. This value propagates through `NormalizedEthTransaction::chain_id` (set via `tx.chain_id()` in `engine-transactions/src/lib.rs` line 108).

**Root cause — step 2: conditional chain ID check in `submit_with_alt_modexp()`**

`engine/src/engine.rs` lines 1054–1059:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

The guard `if let Some(chain_id)` means the entire check is **skipped** when `chain_id` is `None`. A pre-EIP-155 transaction passes this gate unconditionally and proceeds to nonce check and execution.

### Impact Explanation

An attacker who observes a valid pre-EIP-155 transaction on Aurora testnet (or any other Aurora deployment) can submit the identical raw bytes to Aurora mainnet. If the victim's mainnet nonce equals the testnet nonce at the time of replay, the transaction executes on mainnet without the user's consent. This can result in:

- Unauthorized ETH (ether) transfers on mainnet from the victim's account.
- Unauthorized ERC-20 token transfers or arbitrary contract calls on mainnet.

This constitutes **direct theft of user funds** (Critical impact).

### Likelihood Explanation

Aurora is explicitly deployed on three distinct chains (mainnet, testnet, local) as documented in `README.md`. Pre-EIP-155 transactions remain valid in the Ethereum ecosystem and are producible by older tooling, hardware wallets in legacy mode, or any client that omits EIP-155 replay protection. New accounts or accounts with identical transaction histories across deployments will have matching nonces, making the replay immediately exploitable. The attacker requires no privileged access — only the ability to observe and resubmit a raw transaction.

### Recommendation

Reject transactions whose `chain_id` is `None` (pre-EIP-155 legacy transactions) at the `submit` entry point. The chain ID check should be unconditional:

```rust
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This forces all submitted transactions to carry an explicit chain binding, eliminating the cross-chain replay surface.

### Proof of Concept

1. On Aurora testnet (chain ID `1313161555`), a user signs a transfer with `v = 27` (pre-EIP-155, no chain binding). The raw RLP-encoded transaction bytes are broadcast.
2. An attacker copies those bytes and calls `submit` on Aurora mainnet (chain ID `1313161554`).
3. In `LegacyEthSignedTransaction::sender()` [1](#0-0) , `v = 27` yields `chain_id = None`.
4. `NormalizedEthTransaction::chain_id` is `None` [2](#0-1) .
5. In `submit_with_alt_modexp`, the `if let Some(chain_id)` guard at line 1055 is not entered [3](#0-2) , so no chain ID mismatch error is raised.
6. `check_nonce` passes if the victim's mainnet nonce matches [4](#0-3) .
7. The transaction executes on mainnet, transferring the victim's funds without consent.

Aurora's multi-chain deployment context (mainnet `1313161554`, testnet `1313161555`, local `1313161556`) is confirmed in `README.md`, making this replay path realistic and directly analogous to the cross-chain `domainSeparator` replay described in the reference report. [5](#0-4)
