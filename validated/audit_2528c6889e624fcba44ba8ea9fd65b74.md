Audit Report

## Title
Notification Map Flood Permanently Locks Victim ICP via `purge_old_notifications` Advancing `last_purged_notification` Past Victim's Block Index — (`rs/nns/cmc/src/main.rs`)

## Summary

The `purge_old_notifications` function in the CMC canister advances `last_purged_notification` to the highest block index evicted from `blocks_notified`. An attacker who submits 1,000,001 real ledger transfers at block indices above a victim's unnotified block index can trigger this purge, advancing `last_purged_notification` past the victim's block. Every subsequent notify call for the victim's block then returns the terminal, non-retriable error `TransactionTooOld`, permanently locking the victim's ICP in the CMC subaccount with no recovery path.

## Finding Description

**Constants** confirmed at [1](#0-0) :
- `MAX_NOTIFY_HISTORY = 1_000_000`
- `MAX_NOTIFY_PURGE = 100_000`

**`purge_old_notifications`** at [2](#0-1)  removes the `MAX_NOTIFY_PURGE` smallest block indices from `blocks_notified` whenever the map exceeds `MAX_NOTIFY_HISTORY`, then sets `last_purged_notification` to the **highest** evicted index (not the lowest minus one). This is the root cause: the watermark is advanced to the top of the purged range, not the bottom.

**All three notify entry points** (`notify_top_up`, `notify_mint_cycles`, `notify_create_canister`) share the same pattern:
1. Execute `fetch_transaction` (async ledger call) — verifying the block exists on-chain. [3](#0-2) 
2. Call `purge_old_notifications(MAX_NOTIFY_HISTORY)`.
3. Hard-reject with `TransactionTooOld` if `block_index <= last_purged_notification`. [4](#0-3) [5](#0-4) [6](#0-5) 

**Attack sequence:**
1. Victim transfers ICP to a CMC subaccount at ledger block index **N** but has not yet called notify.
2. Attacker submits 1,000,001 real ICP transfers to CMC subaccounts at block indices **N+1 … N+1,000,001** and calls notify for each, inserting 1,000,001 entries into `blocks_notified`. (Attacker recovers ICP value as cycles, net cost ≈ fees only.)
3. On the 1,000,001st notify call, `purge_old_notifications` fires: it removes the 100,000 smallest entries (N+1 … N+100,000) and sets `last_purged_notification = N+100,000`.
4. Victim calls `notify_top_up(block_index = N)`. Check: `N <= N+100,000` → **true** → `TransactionTooOld` returned.
5. Victim's ICP is permanently locked; `TransactionTooOld` is non-retriable and the block index will never become "new" again.

The state struct comment at [7](#0-6)  acknowledges the ambiguity ("The status of blocks not newer than this is ambiguous") but provides no recovery mechanism.

## Impact Explanation

The victim's ICP is permanently locked in the CMC subaccount. `TransactionTooOld` is a terminal, non-retriable error. There is no admin refund endpoint, no heartbeat-driven recovery for meaningful-memo transactions, and no way for the victim to re-submit the notification. This constitutes **permanent, targeted loss of user funds** on an in-scope NNS/financial integration canister. This maps to **High ($2,000–$10,000)**: unauthorized permanent loss of ledger assets requiring meaningful per-target work (~100 ICP in fees, 1M+ transactions). If the victim holds a large ICP position (>$1M equivalent), the impact escalates to **Critical**.

## Likelihood Explanation

The attack requires approximately 100 ICP in ledger fees (1,000,001 transfers × 0.0001 ICP), which is economically non-trivial but feasible for a targeted attack. The victim's block index is publicly visible on the ICP ledger. The attacker must act after the victim's transfer and before the victim calls notify — a race condition that is exploitable if the victim delays notification (common in manual or scripted workflows). No privileged access, key material, or consensus corruption is required; only standard ledger write access available to any ICP holder.

## Recommendation

1. **Fix the watermark advancement**: In `purge_old_notifications`, set `last_purged_notification` to `min(purged_indices) - 1` (i.e., one below the lowest purged block), not the highest purged block. This ensures only blocks that were actually evicted from the map are considered "too old."
2. **Use a time-based cutoff**: Reject notifications only for blocks older than a fixed wall-clock window (e.g., 24 hours), independent of map eviction state.
3. **Add a refund path**: Provide an admin or governance-gated endpoint to refund ICP from CMC subaccounts whose notification block index has been purged past.
4. **Rate-limit notify calls per principal**: Make flooding economically impractical by capping the number of notify calls per principal per time window.

## Proof of Concept

```rust
// State-machine unit test (no mainnet interaction required)
let mut state = State::default();

// Attacker inserts 1_000_001 entries at indices above victim's block 0
for i in 1u64..=1_000_001 {
    state.blocks_notified.insert(i, NotificationStatus::NotifiedTopUp(Ok(Cycles::zero())));
}

// Trigger purge (simulates the 1_000_001st notify call)
state.purge_old_notifications(MAX_NOTIFY_HISTORY);

// last_purged_notification is now 100_000
assert_eq!(state.last_purged_notification, 100_000);

// Victim's block index 0 is now permanently rejected
let victim_block: BlockIndex = 0;
assert!(victim_block <= state.last_purged_notification);
// → notify_top_up(block_index=0) returns TransactionTooOld(100_001)
```

This test requires no network access and can be run as a standard Rust unit test against the CMC state module.

```rust
    fn purge_old_notifications(&mut self, max_history: usize) {
        let mut last_purged = 0;
        let mut cnt = 0;
        // Remove elements from the beginning of self.blocks_notified until either
        // it is small enough, or MAX_NOTIFY_PURGE entries have been removed.
        while self.blocks_notified.len() > max_history && cnt < MAX_NOTIFY_PURGE {
            // pop_first is nightly only
            let block_height = *self.blocks_notified.iter().next().unwrap().0;
            self.blocks_notified.remove(&block_height);
            last_purged = block_height;
            cnt += 1;
        }
        // make sure this grows monotonically (a delayed callback might have added older status)
        self.last_purged_notification = last_purged.max(self.last_purged_notification);
    }
```
