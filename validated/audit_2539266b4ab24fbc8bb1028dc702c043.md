### Title
Missing BN254 G2 Subgroup Membership Check Causes EVM Semantic Mismatch in alt_bn128 Pairing Precompile - (File: `basic_system/src/system_functions/bn254_pairing_check.rs`)

### Summary

The BN254 pairing precompile (EVM address `0x8`) in ZKsync OS rejects G2 input points that lie on the curve but outside the prime-order subgroup, returning a failed call. Ethereum's reference implementation (go-ethereum / EIP-197) does not perform this subgroup check — it accepts such points, computes the pairing, and returns `0` (false). This is a concrete EVM semantic mismatch: a transaction that calls the pairing precompile with a crafted out-of-subgroup G2 point will produce a different execution outcome on ZKsync OS versus Ethereum mainnet.

### Finding Description

In `bn254_pairing_check_inner`, each G2 point is validated with:

```rust
let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
g2_point.check().map_err(|_| ())?;
``` [1](#0-0) 

The `check()` method from `ark_serialize::Valid` for `Affine<SWCurveConfig>` calls both `is_on_curve()` **and** `is_in_correct_subgroup_assuming_on_curve()`. For BN254 G2, the subgroup check is:

```rust
fn is_in_correct_subgroup_assuming_on_curve(point: &G2Affine) -> bool {
    let x_times_point = point.mul_bigint(SIX_X_SQUARED);
    let p_times_point = p_power_endomorphism(point);
    x_times_point.eq(&p_times_point)
}
``` [2](#0-1) 

BN254 G2 has a large cofactor (≈ 2.19 × 10^76): [3](#0-2) 

This means E'(Fq2) contains many points that satisfy the curve equation but are **not** in the prime-order subgroup G2. When `check()` encounters such a point, it returns `Err`, which propagates as `InvalidPoint`:

```rust
bn254_pairing_check_inner::<A>(num_pairs, src, allocator)
    .map_err(|_| interface_error!(Bn254PairingCheckInterfaceError::InvalidPoint))?
``` [4](#0-3) 

This causes the precompile call to **fail** (empty returndata, call success = false). On Ethereum, EIP-197 only requires that points be on the curve — a G2 point outside the subgroup is not malformed per the spec. Go-ethereum computes the pairing and returns `0x00...01` or `0x00...00` without error.

By contrast, the BLS12-381 precompile (EIP-2537) correctly and intentionally performs a subgroup check via `parse_g2_with_subgroup_check`, which is required by EIP-2537: [5](#0-4) 

The BN254 G1 subgroup check is a no-op (cofactor = 1, so all curve points are in G1): [6](#0-5) 

The G2 cofactor is not 1, so the subgroup check is non-trivial and actively rejects valid-per-EIP-197 inputs.

### Impact Explanation

**EVM semantic mismatch / state-transition divergence.** A smart contract calling the alt_bn128 pairing precompile (`0x8`) with a G2 point that is on the curve but outside the prime-order subgroup will:

- **Ethereum mainnet**: precompile call succeeds, returns `0x00...00` (false). The calling contract receives `success = true` from the low-level call, with returndata `= 0`.
- **ZKsync OS**: precompile call fails (returns `InvalidPoint` error). The calling contract receives `success = false` from the low-level call, with empty returndata.

Any contract that distinguishes between "precompile call failed" and "precompile returned false" will execute different code paths, producing divergent state. This includes contracts using low-level `call` with explicit success-flag checks, or contracts that use `try/catch` around precompile calls. The divergence can affect ZK proof verifiers, signature aggregation contracts, and any protocol relying on the pairing precompile.

### Likelihood Explanation

Constructing a BN254 G2 point that is on the curve but outside the prime-order subgroup is straightforward: take any generator of E'(Fq2), multiply it by a scalar that is not a multiple of the group order `r`, and the result is a valid curve point outside G2. An unprivileged transaction sender can craft such a point and include it in a call to any contract that forwards it to the pairing precompile. No privileged access is required.

### Recommendation

Remove the implicit subgroup check from the BN254 G2 point validation in `bn254_pairing_check_inner`. Replace `g2_point.check()` with a check that only verifies the point is on the curve (i.e., call `is_on_curve()` directly, not `check()`), matching EIP-197 semantics. The BLS12-381 precompile's explicit subgroup check is correct for EIP-2537 and should remain unchanged.

### Proof of Concept

1. Construct a BN254 G2 point `P_bad` that satisfies the curve equation `y^2 = x^3 + b` over Fq2 but is not in the prime-order subgroup (e.g., take the standard G2 generator and multiply by `2`, then add a point of order equal to the cofactor — or simply take any point in E'(Fq2) \ G2).
2. Encode a pairing input: `[G1_generator || P_bad]` (192 bytes).
3. Call the precompile at address `0x8` with this input on both Ethereum mainnet and ZKsync OS.
4. **Ethereum**: call returns `success=true`, returndata = `0x00...00` (false, pairing ≠ 1).
5. **ZKsync OS**: call returns `success=false`, empty returndata — because `g2_point.check()` rejects `P_bad` at line 144 of `bn254_pairing_check.rs`.

The divergence is observable at the EVM call-return level and will cause any contract that checks the call success flag to behave differently across the two environments.
