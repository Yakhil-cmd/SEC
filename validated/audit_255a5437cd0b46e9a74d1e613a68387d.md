Audit Report

## Title
Unbounded Prometheus Label Cardinality via Unsanitized HTTP Path and Request Body — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
`RosettaMetricsMiddleware` records the raw HTTP request URI path and the attacker-supplied `network_identifier.network` JSON field as Prometheus label values with no sanitization, allowlist, or cardinality cap. Any unauthenticated HTTP client can drive unbounded heap growth in the Rosetta process by sending requests with unique paths and unique network identifiers, eventually causing OOM termination.

## Finding Description

**Vector 1 — `endpoint` label (raw URI path):**

At line 326 of `metrics.rs`, the raw URI path is captured before dispatch:
```rust
let path = req.uri().path().to_owned();
```
At lines 363–376, this path is passed verbatim to `observe_request_duration` for every response, including 404s, because Axum returns `Ok(Response { status: 404 })` for unregistered routes — the `Ok(response)` branch fires unconditionally:
```rust
Ok(response) => {
    metrics.observe_request_duration(&path, &method, &status, duration);
}
```
The router in `main.rs` (lines 361–383) defines a fixed set of routes and applies `metrics_layer` to the whole router. Any path outside that set (e.g., `/block/1`, `/block/2`, …) yields a 404 `Ok` response, and the unique path is recorded as a new `endpoint` label value.

**Vector 2 — `token_display_name` label (attacker-controlled body field):**

`extract_canister_id` (lines 397–403) reads `network_identifier.network` from the JSON body with no validation. The extracted string is passed to `get_display_name_from_canister_id` (lines 253–258), which returns the raw attacker-supplied string when no mapping exists:
```rust
.unwrap_or_else(|| canister_id.to_string())
```
This raw string is then used as the `token_display_name` label (lines 342–348). Additionally, `RosettaMetrics::new` (lines 119–120) inserts every new `(canister_id, display_name)` pair into the global `CANISTER_DISPLAY_NAMES` HashMap, compounding the memory leak.

**Combined effect:**

`rosetta_http_request_duration_seconds` carries four labels: `[token_display_name, endpoint, method, status]` (lines 92–97). Each unique tuple causes Prometheus to allocate a new histogram object with ~15 bucket counters. With N unique paths × M unique network identifiers, the registry grows to O(N×M) histogram objects. No cardinality guard, label allowlist, or rate limit exists anywhere in the middleware.

## Impact Explanation

Unbounded heap growth leads to OOM termination of the Rosetta process. The Rosetta API is the operational bridge for ICRC-1 token operations including ckBTC/ckETH withdrawals. Process termination stalls all in-flight operations until restart. This matches: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or availability impact not based on raw volumetric DDoS**, as the attack exploits a specific code-level resource exhaustion flaw, not traffic volume.

## Likelihood Explanation

The Rosetta API is publicly reachable with no authentication required. The attack requires only standard HTTP POST requests with crafted JSON bodies and unique URL paths — achievable with `curl` or a trivial script. Prometheus never evicts label combinations, so the memory growth is monotonic across the process lifetime. A moderate rate (e.g., 10K requests/minute) is sufficient to exhaust memory.

## Recommendation

1. **Normalize the `endpoint` label**: Map the raw path to a fixed allowlist of known route templates (e.g., `/block`, `/account/balance`). Replace any unrecognized path with a sentinel value such as `"unknown"` before passing it to `observe_request_duration`.
2. **Validate `token_display_name`**: In `extract_canister_id` / the middleware, reject or replace any `network_identifier.network` value not present in `CANISTER_DISPLAY_NAMES` at startup with a sentinel like `"unknown"` instead of using the raw attacker-supplied string.
3. **Add a cardinality guard**: Before calling `with_label_values`, check the current number of label-set combinations in the `HistogramVec` and refuse to create new ones beyond a configured threshold (e.g., 1000).
