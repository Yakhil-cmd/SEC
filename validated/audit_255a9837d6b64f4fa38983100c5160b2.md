### Title
Zero `native_per_pubdata` from Integer Truncation Causes Pubdata to Be Uncharged in Native Resources - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the operator-set `pubdata_price` is less than `native_price`, the integer division `pubdata_price / native_price` truncates to zero. The resulting `native_per_pubdata = 0` causes all pubdata consumption to be charged zero native resources, allowing any user to submit transactions that generate large amounts of pubdata without paying the corresponding native resource cost. The operator bears the L1 data publication cost without compensation.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the per-pubdata-byte native cost is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This is a truncating integer division. Whenever `pubdata_price < native_price`, the result is `0`. The same truncation exists in the public API helper:

```rust
// api/src/helpers.rs:427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

`native_per_pubdata` is subsequently used in three places, all of which produce zero cost when it is zero:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   // gas_helpers.rs:352
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // = 0
   ```
   No native is deducted from the transaction's native limit for intrinsic pubdata.

2. **Execution pubdata charging** in `get_resources_to_charge_for_pubdata`:
   ```rust
   // gas_helpers.rs:430-432
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)  // = 0
       .ok_or(out_of_native_resources!())?;
   ```
   Zero native resources are charged regardless of how many pubdata bytes the transaction writes.

3. **Post-execution pubdata check** in `check_enough_resources_for_pubdata`:
   The check always passes because the required native is zero.

The condition `pubdata_price < native_price` is operationally reachable. For example, if the operator sets `native_price = 1000` (reflecting high proving cost) and `pubdata_price = 500` (reflecting cheap L1 calldata), the intended per-byte native cost is `0.5`, but integer truncation yields `0`. The operator may not realize that this boundary causes complete suppression of pubdata charging.

---

### Impact Explanation

When `native_per_pubdata = 0`:

- Every transaction can write up to the block's pubdata limit (`get_pubdata_limit()`) without consuming any native resources for it.
- The `check_enough_resources_for_pubdata` check always passes, so no transaction is ever reverted for pubdata cost.
- The operator must still publish all written pubdata to L1 and pay the corresponding L1 gas, but receives zero native-resource compensation from users for that cost.
- A malicious user can craft transactions that maximally fill the pubdata budget (e.g., many `SSTORE` operations writing new storage slots) at no native resource cost, forcing the operator to absorb the L1 publication cost.

This is a **resource accounting bug** causing incorrect fee charging and potential loss of funds for the operator/protocol.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a valid and reachable operational state. The operator sets both values independently via block metadata. A reasonable configuration such as `native_price = 10_000` (high proving cost) and `pubdata_price = 9_999` (slightly cheaper pubdata) silently produces `native_per_pubdata = 0`. No error is raised; the transaction proceeds normally. Any unprivileged user observing the block context (both values are public) can immediately exploit this by submitting pubdata-heavy transactions.

---

### Recommendation

Replace the truncating division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// validation_impl.rs:142 — use div_ceil instead of wrapping_div
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs:427`. This ensures that even when `pubdata_price < native_price`, the per-byte native cost is at least `1`, preventing pubdata from being silently free.

---

### Proof of Concept

**Setup:**
- Block context: `native_price = 1000`, `pubdata_price = 999`
- `native_per_pubdata = 999 / 1000 = 0` (integer truncation)

**Attacker transaction:**
- Submit an EIP-1559 transaction calling a contract that writes 1,000 new storage slots (each slot write generates ~65 bytes of pubdata → ~65,000 bytes total)
- The transaction pays gas for EVM execution (ergs) but zero native resources for pubdata

**Trace through the code:**

1. `validate_and_compute_fee_for_transaction` computes `native_per_pubdata = 0` [1](#0-0) 

2. `create_resources_for_tx` deducts `0 * intrinsic_pubdata = 0` from the native limit [2](#0-1) 

3. After execution, `get_resources_to_charge_for_pubdata` computes `65000 * 0 = 0` native cost [3](#0-2) 

4. `check_enough_resources_for_pubdata` passes (0 ≤ remaining native), transaction succeeds [4](#0-3) 

5. Operator publishes 65,000 bytes to L1 at their own cost, receiving zero native-resource compensation from the user.

The same truncation is present in the public API helper used for off-chain fee estimation: [5](#0-4)
