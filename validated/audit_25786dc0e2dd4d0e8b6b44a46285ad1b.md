Audit Report

## Title
Silent Clamping of Over-u64 Approve Amount to `Tokens::MAX` in ICRC-1 Ledger — (File: rs/ledger_suite/icrc1/ledger/src/main.rs)

## Summary
In `icrc2_approve_not_async`, a caller-supplied `Nat` amount that exceeds `u64::MAX` is silently converted to `Tokens::max_value()` (`u64::MAX`) instead of returning an error. Any spender named in such an approval receives an allowance of `u64::MAX` tokens — far exceeding what the approver intended — and can subsequently drain the approver's entire balance via `icrc2_transfer_from`.

## Finding Description
At `rs/ledger_suite/icrc1/ledger/src/main.rs:840`:

```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```

`TryFrom<Nat> for Tokens` at `rs/ledger_suite/common/ledger_core/src/tokens.rs:291–296` returns `Err` for any `Nat` whose value does not fit in `u64`. The `unwrap_or_else` at the call site discards that `Err` and substitutes `Tokens { e8s: u64::MAX }` (confirmed at `tokens.rs:142`: `pub const MAX: Self = Tokens { e8s: u64::MAX }`).

The resulting `Transaction::Approve` with `amount = u64::MAX` is passed directly to `apply_transaction` and committed to the ledger. No subsequent guard in `apply_transaction` (`rs/ledger_suite/common/ledger_canister_core/src/ledger.rs:214–310`) or in the ICRC-1 `Transaction::apply` path (`rs/ledger_suite/icrc1/src/lib.rs:529–557`) checks whether the stored allowance amount is reasonable.

The correct pattern exists immediately below at lines 841–854, where an out-of-range `expected_allowance` is converted to `ApproveError::AllowanceChanged` and returned to the caller. The `amount` path does not follow this pattern.

At `icrc2_transfer_from` time, `rs/ledger_suite/icrc1/src/lib.rs:478–496` checks `allowance.amount < used_allowance`. With `allowance.amount = u64::MAX`, this check passes for any transfer amount up to the approver's actual balance, allowing the spender to drain the account.

## Impact Explanation
A spender named in a silently-clamped approval holds an allowance of `u64::MAX` tokens. A subsequent `icrc2_transfer_from` call can transfer the approver's entire balance to any destination. This constitutes unauthorized token transfer and theft of user funds from an in-scope ICRC-1/ICRC-2 ledger — matching the **High** impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds."

## Likelihood Explanation
The precondition is that the approver submits `icrc2_approve` with `amount > u64::MAX`. This occurs in practice when: (a) a dapp uses arbitrary-precision arithmetic and produces an out-of-range value without validating before submission; (b) a canister acting as an approver has an arithmetic bug. In either case the approver expects a protocol-compliant rejection per the ICRC-2 standard; instead the ledger silently records `u64::MAX`. The spender then needs only one `icrc2_transfer_from` call. No privileged access, key material, or governance majority is required.

## Recommendation
Replace the silent fallback with an explicit error return, consistent with the `expected_allowance` handling immediately below:

```rust
let amount = Tokens::try_from(arg.amount)
    .map_err(|_| ApproveError::GenericError {
        error_code: Nat::from(0u64),
        message: "amount exceeds maximum token value".to_string(),
    })?;
```

## Proof of Concept
1. Deploy any ICRC-1/ICRC-2 ledger canister with a u64 token type and fund account A.
2. From account A, call `icrc2_approve` with `ApproveArgs { amount: Nat(2^64), spender: B, ... }`.
3. Call `icrc2_allowance` for `(A, B)` — the returned allowance is `u64::MAX`, not a rejection.
4. From account B, call `icrc2_transfer_from` with `from: A, amount: balance_of(A) - fee` — the transfer succeeds, draining A's balance.
5. A deterministic integration test using PocketIC or `StateMachine` can encode `Nat` from `BigUint::from(u64::MAX) + 1u64` in the Candid `ApproveArgs` and assert that `icrc2_allowance` returns `u64::MAX` and that a subsequent `icrc2_transfer_from` succeeds.
