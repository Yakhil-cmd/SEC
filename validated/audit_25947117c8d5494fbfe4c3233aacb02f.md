Audit Report

## Title
Incomplete Cycle Reservation Validation Allows Compromised Sandbox to Panic Replica via `.unwrap()` in `apply_balance_changes` — (`rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs`)

## Summary

`SystemStateModifications::apply_balance_changes` calls `state.reserve_cycles(self.reserved_cycles).unwrap()` at line 600. The only upstream guard, `validate_cycle_change`, checks only internal accounting consistency (no cycles created from thin air) but does not verify that `reserved_cycles` respects the canister's `reserved_balance_limit`. A compromised sandbox process can craft a `SystemStateModifications` payload that passes `validate_cycle_change` but causes `reserve_cycles` to return `Err(ReservationError::ReservedLimitExceed)`, triggering the `.unwrap()` panic. Because all replicas process messages deterministically, every replica on the subnet crashes on the same message, causing a subnet-wide availability failure.

## Finding Description

`apply_changes` calls `validate_cycle_change` then unconditionally calls `apply_balance_changes` at line 409:

```rust
self.validate_cycle_change(system_state.canister_id() == CYCLES_MINTING_CANISTER_ID)?;
self.apply_balance_changes(system_state);
```

`apply_balance_changes` returns `()` and contains a panicking `.unwrap()` at line 600:

```rust
// Apply the reserved cycles. This must succeed because the cycle
// changes were validated. If it doesn't succeed then, it is better to
// crash here to avoid making the cycle balance incorrect.
state.reserve_cycles(self.reserved_cycles).unwrap();
```

`reserve_cycles` in `rs/replicated_state/src/canister_state/system_state.rs` delegates to `can_reserve_cycles`, which returns `Err(ReservationError::ReservedLimitExceed)` when `self.reserved_balance + amount > reserved_balance_limit` (lines 2078–2082). This error path is entirely absent from `validate_cycle_change`.

`validate_cycle_change` (lines 160–191) only verifies that `cycles_balance_change == call_context_balance_taken - request_payments - consumed_cycles - reserved_cycles`. It does not inspect `reserved_balance_limit` at all. A compromised sandbox can set `reserved_cycles = V` and `cycles_balance_change = Removed(V)` — this satisfies the accounting identity — while choosing `V > reserved_balance_limit - current_reserved_balance`, which causes `reserve_cycles(V)` to return `Err` and the `.unwrap()` to panic.

The code comment itself acknowledges the intentional panic ("it is better to crash here") but incorrectly assumes the prior validation is sufficient to prevent the error path.

## Impact Explanation

A replica panic in the execution path crashes the replica process. Because all replicas on a subnet execute the same messages in the same order deterministically, every replica hits the same panic on the same message. This causes a subnet-wide crash loop requiring manual subnet recovery procedures. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

Exploitation requires a prior sandbox escape — arbitrary code execution within the canister sandbox process — to directly craft a `SystemStateModifications` struct with an out-of-bounds `reserved_cycles` value. Sandbox escapes against Wasm runtimes are non-trivial but are a known attack class, and the IC's threat model explicitly treats the sandbox as an untrusted component that the replica must defend against. The IC's sandbox isolation layer exists precisely to contain compromised canister execution; the replica is expected to validate all sandbox-supplied data. Once sandbox code execution is achieved, constructing the triggering payload is straightforward: set `reserved_cycles` to any value exceeding `reserved_balance_limit - current_reserved_balance` and set `cycles_balance_change = Removed(reserved_cycles)`. Any canister with a `reserved_balance_limit` configured is a viable target.

## Recommendation

- **Short term:** Change `apply_balance_changes` to return `HypervisorResult<()>`. Replace `state.reserve_cycles(self.reserved_cycles).unwrap()` with `state.reserve_cycles(self.reserved_cycles).map_err(|e| Self::error(format!("Failed to reserve cycles: {e:?}")))?`. Replace the `assert_eq!` balance invariant at line 608 with `debug_assert_eq!`. Propagate the result through `apply_changes`.
- **Long term:** Extend `validate_cycle_change` to also verify that `current_reserved_balance + reserved_cycles <= reserved_balance_limit` before calling `apply_balance_changes`. Audit all code reachable from `apply_changes` for remaining `unwrap`, `expect`, `assert`, or `unreachable` calls operating on sandbox-supplied fields.

## Proof of Concept

A compromised sandbox process returns the following `SystemStateModifications`:

```
reserved_cycles       = reserved_balance_limit - current_reserved_balance + 1
cycles_balance_change = Removed(reserved_cycles)   // satisfies validate_cycle_change
consumed_cycles_by_use_case = default (zero)
call_context_balance_taken  = None
requests              = []
```

Execution path on the replica:

1. `apply_changes` calls `validate_cycle_change` → passes (accounting identity holds: `cycles_balance_change == Removed(reserved_cycles)`)
2. `apply_changes` calls `apply_balance_changes`
3. `adjusted_balance_change = Removed(reserved_cycles) + Added(reserved_cycles) = zero`; no net balance change applied
4. `state.reserve_cycles(reserved_cycles)` → `can_reserve_cycles` checks `reserved_balance + reserved_cycles > reserved_balance_limit` → returns `Err(ReservationError::ReservedLimitExceed)`
5. `.unwrap()` panics → replica process crashes → all replicas on the subnet crash on the same message → subnet enters crash loop

A deterministic integration test or PocketIC test can reproduce this by constructing a `SystemStateModifications` with the above values and calling `apply_changes` against a `SystemState` whose `reserved_balance_limit` is set, verifying the panic.
