### Title
`delta_gas` Calculation Rounds Down in `compute_gas_refund`, Allowing Users to Underpay Native Resource Costs - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the conversion from native resource consumption to EVM gas units uses integer floor division (`native_used / native_per_gas`). Because this truncates the remainder, the computed `delta_gas` is systematically underestimated, causing users to be refunded slightly more gas than they should be. The operator receives less fee than the actual native resource cost warrants on every transaction where `native_used % native_per_gas != 0`.

### Finding Description

The ZKsync OS double-resource accounting model derives a `delta_gas` value to reconcile native resource consumption with EVM gas consumption. The documentation specifies:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

This is implemented at line 72 of `refund_calculation.rs`:

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [1](#0-0) 

The division `native_used / native_per_gas` is Rust integer division, which truncates toward zero (floor division). When `native_used` is not an exact multiple of `native_per_gas`, the fractional native unit is silently discarded. This means `delta_gas` is computed as one unit less than the true ceiling value, and the user is charged one fewer gas unit than the native resource actually consumed.

The resulting `gas_used` is then used to compute the operator payment and user refund:

```rust
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
    ...
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
``` [2](#0-1) [3](#0-2) 

Because `gas_used` is one unit too low, the user receives one extra gas unit as a refund, and the operator is paid one gas unit less than the native resource cost warrants.

### Impact Explanation

**Impact: Low.** On each transaction where `native_used % native_per_gas != 0`, the operator is underpaid by up to `(native_per_gas - 1) * gas_price` wei. This is a dust amount per transaction, but it accumulates systematically across all transactions. The operator bears the cost of native resource consumption that is not fully recovered through gas fees.

### Likelihood Explanation

**Likelihood: High.** The condition `native_used % native_per_gas != 0` holds for virtually every transaction, since `native_used` is the sum of many independent native charges (EVM opcodes, precompiles, pubdata) and `native_per_gas` is an arbitrary ratio derived from `gasPrice / nativePrice`. The probability of exact divisibility on any given transaction is negligible. [4](#0-3) 

### Recommendation

Replace the floor division with ceiling division when computing `delta_gas`:

```rust
// Before (rounds down, undercharges user):
(native_used / native_per_gas) as i64 - (gas_used as i64)

// After (rounds up, correctly charges user for all native consumed):
(native_used.div_ceil(native_per_gas)) as i64 - (gas_used as i64)
```

This ensures that any fractional native unit consumed is rounded up to a full gas unit, so the operator is always compensated for the full native resource cost.

### Proof of Concept

**Setup:**
- `native_per_gas = 100` (i.e., `gas_price = 100 * native_price`)
- `native_used = 101` (transaction consumed 101 native units)
- `gas_used = 0` (EVM used no gas, hypothetical)

**Current behavior (floor division):**
```
delta_gas = (101 / 100) as i64 - 0 = 1
gas_used = 0 + 1 = 1
```
User is charged 1 gas unit. But 101 native units at 100 native/gas requires `ceil(101/100) = 2` gas units.

**Correct behavior (ceiling division):**
```
delta_gas = ceil(101 / 100) as i64 - 0 = 2
gas_used = 0 + 2 = 2
```

The user underpays by 1 gas unit × `gas_price` per such transaction. At scale, this represents a systematic operator loss proportional to transaction volume. [5](#0-4) [6](#0-5)
