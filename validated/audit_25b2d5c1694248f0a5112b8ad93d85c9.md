Audit Report

## Title
Floating-Point Precision Loss Silently Truncates Token Amounts in ICRC-21 Consent Messages — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

`convert_tokens_to_string_representation` converts an arbitrary-precision `Nat` token amount to `f64` before formatting it for `GenericDisplayMessage` consent messages. Because `f64` has only a 53-bit mantissa, any token amount (in the ledger's smallest unit) exceeding 2^53 = 9,007,199,254,740,992 is silently rounded to the nearest representable value. The resulting displayed amount in the ICRC-21 consent message — the transaction confirmation dialog used by hardware wallets and browser extensions — can differ from the amount the ledger will actually debit, with no error or warning surfaced to the user.

## Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs` at lines 318–327, the private helper `convert_tokens_to_string_representation` performs:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```

`BigUint::to_f64()` returns `None` only for infinity/NaN — it never returns `None` for precision loss. For any `Nat` value above 2^53, the conversion silently rounds to the nearest representable `f64`, and the rounded value is then divided by `10^decimals` and formatted as the displayed amount.

This helper is called by every amount-display method on `ConsentMessage` for the `GenericDisplayMessage` path:
- `add_amount` (lines 112–114) — used for `icrc1_transfer`, `icrc2_transfer_from`, and ICP legacy `transfer`
- `add_fee` (lines 140–141) — used for all transaction types
- `add_allowance` (lines 185–186) — used for `icrc2_approve`
- `add_existing_allowance` (lines 210–212) — used for `icrc2_approve`

`GenericDisplayMessage` is the default path: `ConsentMessageBuilder::build()` at lines 173–180 of `lib.rs` selects it when `device_spec` is `None` or `GenericDisplay`, making it the most common path for hardware wallets and browser extensions.

The `FieldsDisplayMessage` path avoids `f64` but uses `nat_to_u64` (lines 329–334), which hard-errors for amounts exceeding `u64::MAX` rather than silently rounding.

The `icrc21_canister_call_consent_message` endpoint is a public `#[update]` method on both the ICRC-1 ledger (`rs/ledger_suite/icrc1/ledger/src/main.rs`, lines 1189–1207) and the ICP ledger (`rs/ledger_suite/icp/ledger/src/main.rs`, lines 1478–1541), callable by any unprivileged principal with no authentication requirement.

## Impact Explanation

The ICRC-21 consent message is the security-critical transaction confirmation mechanism for the Internet Computer — its entire purpose is to show users the exact parameters of what they are authorizing before signing. Displaying an incorrect token amount in this dialog directly undermines that security guarantee. A user relying on the consent message to verify a large transfer before signing could approve a transaction for a materially different amount than shown. For ICP (8 decimal places), the precision boundary is ~90,071,992 ICP; above this threshold, increments of 1 e8 (0.00000001 ICP) may be silently dropped from the display while the ledger debits the exact original `Nat`. This constitutes a significant ledger/infrastructure security impact with concrete user harm, matching the High bounty impact class: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The 2^53 e8s threshold (~90 million ICP) is above typical retail transfer sizes, but institutional accounts, the minting account, and large SNS treasury operations routinely operate in this range. No privileged access is required: any principal can call `icrc21_canister_call_consent_message` with a crafted `ConsentMessageRequest` containing an amount above the threshold and `device_spec: Some(GenericDisplay)` (or omit `device_spec` entirely). The rounding is deterministic and reproducible. The attack is passive — the attacker does not need to intercept or modify anything; the ledger itself returns the rounded display value in response to a standard API call.

## Recommendation

Replace the `f64`-based conversion with an integer-only implementation using the `Nat`'s underlying `BigUint` arithmetic:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    use num_bigint::BigUint;
    use num_traits::Zero;

    let divisor = BigUint::from(10u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let frac  = &tokens.0 % &divisor;
    if frac.is_zero() {
        Ok(format!("{whole}"))
    } else {
        let frac_str = format!("{frac:0>width$}", width = decimals as usize);
        Ok(format!("{}.{}", whole, frac_str.trim_end_matches('0')))
    }
}
```

This eliminates the `f64` conversion entirely and produces an exact decimal string for any `Nat` value. The `num_bigint` crate is already a transitive dependency via `candid`.

## Proof of Concept

Call `icrc21_canister_call_consent_message` on any ICRC-1 or ICP ledger canister with:

```
method: "icrc1_transfer"
arg: TransferArg {
    amount: Nat(9_007_199_254_740_993),  // 2^53 + 1 e8s
    to: <any valid account>,
    fee: None, from_subaccount: None,
    created_at_time: None, memo: None,
}
user_preferences: {
    device_spec: Some(GenericDisplay),
    metadata: { language: "en", utc_offset_minutes: None }
}
```

The returned `GenericDisplayMessage` will display `**Amount:** \`90071992.5474099 TOKEN\`` (f64-rounded), while the actual ledger debit will be `90071992.54740993 TOKEN` — a discrepancy of 1 e8 that the signing user cannot detect from the consent message alone. A unit test can confirm this by calling `convert_tokens_to_string_representation(Nat::from(9_007_199_254_740_993u64), 8)` and asserting the result equals `"90071992.54740993"` — the current implementation returns a rounded value instead.
