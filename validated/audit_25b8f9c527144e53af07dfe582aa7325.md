Audit Report

## Title
Missing Signer Membership Pre-Check in IDKG Signature Share Validation Enables Sustained CPU Exhaustion by Malicious Subnet Node - (File: rs/consensus/idkg/src/signer.rs)

## Summary
`validate_signature_share` in `rs/consensus/idkg/src/signer.rs` (lines 249–326) dispatches directly to `crypto_verify_sig_share` without first verifying that the claimed signer is a member of the receiver set embedded in `ThresholdSigInputs`. The crypto layer (`rs/crypto/src/sign/canister_threshold_sig/ecdsa.rs` and `schnorr.rs`) deserializes multiple large transcripts before performing the membership check, meaning a malicious subnet node can force all honest nodes to perform repeated transcript deserialization work for every crafted share it injects. The IDKG P2P layer uses `SLOT_TABLE_NO_LIMIT` (usize::MAX), providing no per-peer rate limiting, enabling sustained attack.

## Finding Description
In `validate_signature_share` (`rs/consensus/idkg/src/signer.rs`, lines 249–326), after three cheap pre-checks (cache dedup, threshold saturation, pool dedup), the function calls `crypto_verify_sig_share` with no membership gate:

```rust
let signer = share.signer();  // attacker-controlled NodeId
// ... cache / dedup checks only ...
match self.crypto_verify_sig_share(inputs, share, idkg_pool.stats()) {
```

Inside `crypto_verify_sig_share` (`rs/consensus/idkg/src/signer.rs`, lines 480–530), for the ECDSA path, `ThresholdEcdsaSigVerifier::verify_sig_share` is called, which in `rs/crypto/src/sign/canister_threshold_sig/ecdsa.rs` (lines 100–124) first deserializes five transcripts (`kappa_unmasked`, `lambda_masked`, `kappa_times_lambda`, `key_times_lambda`, `key`) via `IDkgTranscriptInternal::try_from()` before reaching the membership check `inputs.key_transcript().index_for_signer_id(signer).ok_or(InvalidArgumentMissingSignerInTranscript)`. For Schnorr (`rs/crypto/src/sign/canister_threshold_sig/schnorr.rs`, lines 103–121), two transcripts are deserialized before the same membership check. `InvalidArgumentMissingSignerInTranscript` is classified as `is_reproducible() = true` (`rs/interfaces/src/crypto/errors.rs`, lines 316, 383), so the share is marked `HandleInvalid` and removed — but the CPU cost is already incurred.

By contrast, `validate_dealing_support` in `rs/consensus/idkg/src/pre_signer.rs` (lines 449–457) explicitly gates on receiver membership before any crypto work:

```rust
if !transcript_params_ref.receivers.contains(&signer) {
    self.metrics.pre_sign_errors_inc("unexpected_support");
    return Some(IDkgChangeAction::HandleInvalid(...));
}
```

The `ThresholdSigInputs` passed to `validate_signature_share` exposes `receivers()` via `key_transcript.receivers` (confirmed in `rs/types/types/src/crypto/canister_threshold_sig.rs`, lines 624–627 for Schnorr; analogous for ECDSA), making the same early-exit pattern directly feasible but absent.

The IDKG P2P channel is configured with `SLOT_TABLE_NO_LIMIT` (usize::MAX) at `rs/replica/setup_ic_network/src/lib.rs` line 291, meaning a malicious subnet node faces no per-peer slot cap when advertising fake shares. The `validated_sig_share_signers` cache only tracks validated signers; each distinct fake `NodeId` bypasses it and reaches the crypto layer. The parallel dispatch via `into_par_iter()` (`rs/consensus/idkg/src/signer.rs`, lines 230–245) means a flood of crafted shares saturates the thread pool simultaneously.

## Impact Explanation
A malicious subnet node can sustain CPU exhaustion on all honest nodes by continuously injecting `EcdsaSigShare`/`SchnorrSigShare`/`VetKdKeyShare` messages with arbitrary fake `signer_id` values. Each crafted share forces transcript deserialization work (5 transcripts for ECDSA, 2 for Schnorr) before being rejected. With `SLOT_TABLE_NO_LIMIT`, the attacker can advertise an unbounded number of distinct fake shares per `on_state_change` tick, saturating the parallel thread pool and delaying or preventing legitimate threshold signing operations from completing within their timeout windows. This constitutes a **subnet availability impact on threshold signing (tECDSA, tSchnorr, vetKD)**, matching the allowed High impact category: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The attacker must be a compromised subnet node operating below the Byzantine fault threshold `f = floor((n-1)/3)`. Any such node can gossip IDKG artifacts to all peers. The attack is indefinitely repeatable: after each `HandleInvalid` removal, the attacker re-injects new fake shares with fresh `NodeId` values. There is no NNS-level automatic detection, slashing, or rate-limiting mechanism for nodes that repeatedly send invalid IDKG messages. The only mitigation is manual governance removal of the malicious node, leaving an open window for sustained disruption.

## Recommendation
Add an explicit signer membership pre-check in `validate_signature_share` immediately after the existing dedup checks, mirroring the pattern in `validate_dealing_support`. For ECDSA and Schnorr inputs, call `inputs.receivers().contains(signer)` (the `receivers()` method is already available on `ThresholdEcdsaSigInputs` and `ThresholdSchnorrSigInputs`); for VetKD inputs, apply the analogous check. A non-member signer should be returned as `HandleInvalid` without invoking `crypto_verify_sig_share`. Additionally, consider tracking per-node invalid-share counts in metrics to enable detection and governance-level response.

## Proof of Concept
1. Malicious subnet node M identifies an active `request_id` from the subnet's certified state.
2. M constructs `EcdsaSigShare { signer_id: fake_node_id_N, request_id, sig_share_raw: <arbitrary bytes> }` for N = 1, 2, 3, … (distinct fake `NodeId` values not in the subnet's receiver set).
3. M advertises these to all honest nodes via the P2P layer using distinct slot numbers (no slot limit applies to IDKG); they enter each node's unvalidated IDKG pool.
4. On the next `on_state_change` tick, `validate_signature_shares` collects all unvalidated shares and dispatches them in parallel via `into_par_iter()`.
5. For each crafted share, `validate_signature_share` passes the cache/dedup checks (new fake signer ID each time) and calls `crypto_verify_sig_share`, which deserializes 5 ECDSA transcripts before returning `InvalidArgumentMissingSignerInTranscript`.
6. The share is marked `HandleInvalid` and removed — but the transcript deserialization CPU cost is already incurred across all thread-pool workers.
7. M repeats continuously, sustaining thread-pool saturation and delaying legitimate threshold signing operations.
8. A deterministic integration test can confirm this by injecting N crafted shares with fake signer IDs into the unvalidated pool, calling `on_state_change`, and observing that all N shares trigger `crypto_verify_sig_share` (via metrics or mock instrumentation) before being rejected, while a pre-check would have short-circuited at step 4.
