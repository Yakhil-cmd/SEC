Audit Report

## Title
Cross-Operator Rate-Limit Drain via Sibling Operator Node Removal — (`rs/registry/canister/src/mutations/node_management/do_remove_node_directly.rs`)

## Summary

`do_remove_node_directly_` charges the rate-limit cost against the **node's registered operator** (`node_operator_id`) rather than the **caller** (`caller_id`). Because the same function also permits a caller from a different operator record to remove nodes when DC and node-provider match, a sibling operator (operator2, same DC + same provider) can remove all of operator1's unassigned nodes while every removal is billed to operator1's rate-limit bucket. Once the bucket is exhausted (max 140 ops), operator1 is blocked from both `add_node` and `remove_node_directly` for up to 7 days.

## Finding Description

**Root cause — rate limit charged against node owner, not caller.**

In `do_remove_node_directly_` (lines 51–54), the rate-limit reservation is made against `node_operator_id` (the node's registered owner), not `caller_id`:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
``` [1](#0-0) 

The authorization check (`caller_id` vs `node_operator_id`) happens only afterwards, inside `make_remove_or_replace_node_mutations` (lines 89–119). If `caller_id != node_operator_id`, the code falls back to comparing DC and node-provider IDs. If both match, the removal is permitted: [2](#0-1) 

This means the rate-limit reservation is already committed against operator1's bucket before the caller's identity is even validated.

**Sibling operator authorization is intentional and confirmed by test.**

The test `should_succeed_remove_node_compare_dc_and_node_provider` (lines 482–554) explicitly proves that operator2 (same DC + same provider as operator1) can successfully call `do_remove_node_directly_` on a node owned by operator1: [3](#0-2) 

**Rate-limit parameters.**

The per-operator bucket has a max spike of `NODE_OPERATOR_MAX_SPIKE = 140` operations (20 ops/day × 7 days), stored in stable memory: [4](#0-3) 

`try_reserve_capacity_for_node_operator_operation` charges both the operator bucket (keyed on `node_operator_id`) and the shared provider bucket: [5](#0-4) 

**`add_node` uses the same per-operator bucket, keyed on `caller_id`.**

`do_add_node_` (lines 60–61) calls `try_reserve_capacity_for_node_operator_operation(now, caller_id, 1)`: [6](#0-5) 

So once operator1's bucket is at zero, operator1's own `add_node` calls also fail with `"Rate Limit Capacity exceeded. Please wait and try again later."` — confirmed by `test_do_remove_node_directly_fails_when_rate_limits_exceeded`: [7](#0-6) 

**Existing guards are insufficient.** The authorization check in `make_remove_or_replace_node_mutations` correctly restricts *which* nodes a caller can remove, but it does not fix the accounting error — the rate-limit reservation is already made against the wrong principal before that check runs.

## Impact Explanation

operator2 submits up to 140 ingress messages to the registry canister, each removing one of operator1's unassigned nodes. Every removal is billed to operator1's rate-limit bucket. After the burst, operator1 receives `"Rate Limit Capacity exceeded. Please wait and try again later."` on every `add_node` and `remove_node_directly` call for up to 7 days. This prevents operator1 from re-registering replacement hardware during the lockout window. This matches the allowed impact: **High — Application/platform-level DoS with concrete user harm**, specifically a 7-day lockout of a legitimate node operator's ability to manage their infrastructure.

## Likelihood Explanation

The precondition — operator2 sharing DC and node-provider with operator1 — is a normal production configuration explicitly supported by the codebase (the sibling-operator removal path exists for redeployment scenarios). The `remove_node_directly` endpoint is callable by any authenticated principal with a valid operator record; no governance proposal is required. The attack requires only that operator2 submit N sequential ingress messages to the registry canister, one per node owned by operator1. No special tooling, leaked keys, or social engineering is needed.

## Recommendation

Charge the rate-limit cost against the **caller** (`caller_id`), not the node's registered operator. In `do_remove_node_directly_`, replace:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
```

with:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, caller_id, 1)?;
```

This ensures that operator2's actions consume operator2's budget, leaving operator1's budget unaffected. The fix is a one-line change and is consistent with how `do_add_node_` already handles rate limiting (against `caller_id`).

## Proof of Concept

The two existing unit tests, combined, prove the full attack path without any new code:

1. `should_succeed_remove_node_compare_dc_and_node_provider` — proves operator2 (same DC + same provider) can remove operator1's node via `do_remove_node_directly_`.
2. `test_do_remove_node_directly_fails_when_rate_limits_exceeded` — proves that exhausting the rate-limit bucket blocks further operations with the expected error.

A combined test to demonstrate the cross-operator drain:

```rust
// Setup: operator1 owns 140 nodes; operator2 shares DC and provider
// Attack: operator2 calls do_remove_node_directly_ 140 times
for node_id in operator1_nodes {
    registry.do_remove_node_directly_(
        RemoveNodeDirectlyPayload { node_id },
        operator2_id,
        now,
    ).unwrap(); // each call drains operator1's bucket by 1
}
// operator1 is now locked out
let err = registry
    .do_remove_node_directly_(any_payload, operator1_id, now)
    .unwrap_err();
assert_eq!(err, "Rate Limit Capacity exceeded. Please wait and try again later.");
```

```rust
    pub fn try_reserve_capacity_for_node_operator_operation(
        &self,
        now: SystemTime,
        node_operator_id: PrincipalId,
        requested_capacity: u64,
    ) -> Result<RateLimitReservation, RateLimiterError> {
        // Find the associated node provider ID for this node operator
        let node_provider_id = get_node_provider_id_for_operator_id(self, node_operator_id)
            .map_err(RateLimiterError::InvalidArguments)?;

        // First reserve from node operator rate limiter (primary)
        let operator_reservation = with_node_operator_rate_limiter(|rate_limiter| {
            rate_limiter.try_reserve(now, node_operator_key(node_operator_id), requested_capacity)
        })?;

        // Then reserve from node provider rate limiter (secondary)
        let provider_reservation = with_node_provider_rate_limiter(|rate_limiter| {
            rate_limiter.try_reserve(now, node_provider_key(node_provider_id), requested_capacity)
        })?;

        Ok(RateLimitReservation {
            operator_reservation,
            provider_reservation,
        })
    }
