Audit Report

## Title
ckBTC Minter Permanently Burns User ckBTC Without Reimbursement on `BuildTxError::AmountTooLow` - (File: rs/bitcoin/ckbtc/minter/src/lib.rs)

## Summary
When `submit_pending_requests` encounters `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput`, it finalizes the withdrawal request with `FinalizedStatus::AmountTooLow` and no reimbursement. Because the user's ckBTC was already burned from the ledger during the earlier `retrieve_btc` call, the user permanently loses their funds — no BTC is sent and no ckBTC is minted back. The analogous `BuildTxError::InvalidTransaction` branch correctly calls `reimburse_canceled_requests`, demonstrating the fix is known and already applied to a parallel code path.

## Finding Description
**Burn before enqueue:** In `retrieve_btc` and `retrieve_btc_with_approval`, `burn_ckbtcs` / `burn_ckbtcs_icrc2` is called at [1](#0-0)  before the `RetrieveBtcRequest` is enqueued. The request is created with a `reimbursement_account` field set to the caller at [2](#0-1) .

**Missing reimbursement in `AmountTooLow` branch:** In `submit_pending_requests`, when `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, the code iterates over the batch and calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` for each request, with no reimbursement scheduled: [3](#0-2) 

**Correct handling exists for `InvalidTransaction`:** The immediately preceding match arm calls `reimburse_canceled_requests`, which schedules a mint-back to the user's `reimbursement_account`: [4](#0-3) 

**`DustOutput` also drops without reimbursement:** The `DustOutput` branch finalizes the affected request with `FinalizedStatus::AmountTooLow` and no reimbursement: [5](#0-4) 

**`remove_retrieve_btc_request` has no reimbursement logic:** It only records the event and pushes a `FinalizedBtcRequest`; no reimbursement is scheduled: [6](#0-5) 

**Terminal state:** `retrieve_btc_status` maps `FinalizedStatus::AmountTooLow` to `RetrieveBtcStatus::AmountTooLow` with no recovery path: [7](#0-6) 

**Event log replay hardcodes `AmountTooLow` with no reimbursement:** On state replay, `RemovedRetrieveBtcRequest` always produces `FinalizedStatus::AmountTooLow`, meaning even after an upgrade the reimbursement cannot be reconstructed: [8](#0-7) 

**Root cause:** The `fee_based_retrieve_btc_min_amount` is updated dynamically at the start of `submit_pending_requests` via `estimate_fee_per_vbyte` at [9](#0-8) , but requests already accepted (and burned) at a lower fee threshold are not re-validated against the new minimum before being batched. When fees spike, `build_unsigned_transaction` returns `AmountTooLow` for requests that passed the admission check at submission time.

## Impact Explanation
This is a **High** severity finding matching: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

The ckBTC ledger supply decreases (burn) without a corresponding BTC output or ckBTC mint-back. This is a ledger conservation violation: `tokens_burned` increases at [10](#0-9)  but neither a BTC UTXO is spent nor a reimbursement mint occurs. Affected users permanently lose their ckBTC. The `FinalizedStatus` enum has no `Reimbursed` variant for this path: [11](#0-10) 

## Likelihood Explanation
Bitcoin network fees are volatile. Any user who submits a `retrieve_btc` request near the current `fee_based_retrieve_btc_min_amount` during a low-fee period and whose request is processed during a high-fee period is affected. No privileged access is required — only a standard `retrieve_btc` call. The `max_time_in_queue_nanos` parameter means requests can sit in the queue long enough for fees to spike significantly. The `fee_based_minimum_withdrawal_amount` formula at [12](#0-11)  shows the minimum can increase by 50,000 sat increments, meaning a request accepted at one fee tier can easily fall below the threshold at the next tier.

## Recommendation
The `BuildTxError::AmountTooLow` and `BuildTxError::DustOutput` branches in `submit_pending_requests` should call `reimburse_canceled_requests` (or an equivalent) instead of silently finalizing with `FinalizedStatus::AmountTooLow`. A new `WithdrawalReimbursementReason::AmountTooLow` variant should be added to `WithdrawalReimbursementReason` in `rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`. The event log replay in `eventlog.rs` must also be updated to emit a `ScheduleWithdrawalReimbursement` event instead of `RemovedRetrieveBtcRequest` for this case, so that reimbursements survive upgrades. The `reimbursement_account` field already exists on `RetrieveBtcRequest` for exactly this purpose.

## Proof of Concept
Using the existing `CkBtcSetup` integration test framework in `rs/bitcoin/ckbtc/minter/tests/tests.rs`:

1. Initialize minter with low fee percentiles so `fee_based_retrieve_btc_min_amount` = 100,000 sat.
2. User calls `retrieve_btc(amount = 100_000)` — ckBTC is burned, `RetrieveBtcRequest` is enqueued with `reimbursement_account = Some(caller)`.
3. Before the minter processes the batch, call `set_fee_percentiles` with high fees (e.g., `FeeRate::from_millis_per_byte(343_000)`) so `fee_based_retrieve_btc_min_amount` jumps to 200,000 sat.
4. Tick the minter timer to trigger `submit_pending_requests`.
5. Assert: `retrieve_btc_status(block_index)` returns `RetrieveBtcStatus::AmountTooLow`.
6. Assert: user's ckBTC balance is unchanged from before step 2 (i.e., the burn was not reversed) — **this assertion will fail**, proving the funds are lost.
7. Assert: no `ScheduleWithdrawalReimbursement` event appears in the minter event log — **this assertion will pass**, confirming no reimbursement was scheduled.

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
