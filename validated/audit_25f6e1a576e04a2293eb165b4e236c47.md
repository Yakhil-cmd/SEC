### Title
Race Condition in `LauncherChildWatch`: Unknown PID Defaults `should_panic = true`, Crashing the Launcher — (`rs/canister_sandbox/src/launcher.rs`)

### Summary

A TOCTOU race in `LauncherServer::new` / `launch_compiler` allows the `LauncherChildWatch` watcher thread to reap a compiler child process whose PID has not yet been inserted into `pid_to_process_info`. Because the `unwrap_or(true)` default at line 146 treats any unknown PID as a panic-worthy event, the launcher panics and the replica crashes.

### Finding Description

`launch_compiler` spawns a child process and then, in a separate step, acquires the map lock and inserts the PID: [1](#0-0) 

The window between `spawn_socketed_process` returning (line 231) and `self.pid_to_process_info.lock()` being called (line 235) — widened slightly by the `drop(UnixStream::from_raw_fd(socket))` call at line 233 — is the race window.

The `LauncherChildWatch` thread only blocks on the condvar when the map is **empty**: [2](#0-1) 

Once at least one child is already registered (e.g., a sandbox process), the watcher is already past the condvar and is blocking inside `wait()`. If the newly spawned compiler process exits with a non-zero status or signal during the race window, `wait()` returns that PID before `launch_compiler` has inserted it. The watcher then:

1. Acquires the lock (succeeds — `launch_compiler` hasn't locked it yet).
2. Calls `info_map.remove(&pid)` → returns `None`.
3. Evaluates `should_panic`: [3](#0-2) 

`unwrap_or(true)` makes the default for any **unknown** PID a panic, even though compiler processes are intentionally registered with `panic_on_failure: false`. The panic at line 156 kills the launcher process, which the replica detects and re-panics via `panic_due_to_exit`. [4](#0-3) 

### Impact Explanation

The launcher panics → the replica's watcher thread calls `panic_due_to_exit` → the replica process crashes. A single replica crash on a subnet is a medium-scope denial of service; the subnet continues with the remaining replicas but the affected node is taken offline until it restarts.

### Likelihood Explanation

Three conditions must coincide:

1. At least one sandbox process is already running (map is non-empty) — this is the normal steady state during canister execution.
2. A `launch_compiler` call is in flight — triggered by any canister install/upgrade.
3. The compiler exits with non-zero status or a signal before `info_map.lock()` is called — achievable with a malformed Wasm module that causes the compiler to crash immediately on startup.

The race window is narrow (a few instructions between `spawn_socketed_process` returning and the mutex lock), but it is non-zero and repeatable under load. An attacker who can reliably crash the compiler (e.g., via a crafted Wasm binary) can attempt the race repeatedly.

### Recommendation

Change the `unwrap_or` default from `true` to `false` for unknown PIDs:

```rust
// rs/canister_sandbox/src/launcher.rs, line 146
let should_panic = process_info
    .as_ref()
    .map(|x| x.panic_on_failure)
    .unwrap_or(false);  // unknown PID → do not panic
```

This matches the intent for compiler processes (`panic_on_failure: false`) and eliminates the crash path for the race. Separately, the PID should ideally be inserted into the map **before** the process is spawned, or the spawn and insert should be performed atomically under the lock, to close the race entirely.

### Proof of Concept

1. Deploy a canister to ensure at least one sandbox process is running (map non-empty, watcher past condvar).
2. Craft a Wasm module that causes the compiler binary to exit with a non-zero code immediately on startup (e.g., by passing an invalid `--embedder-config` argument or a Wasm binary that triggers an immediate compiler panic).
3. Trigger `launch_compiler` repeatedly. On the iteration where the compiler exits during the race window (between `spawn_socketed_process` return and `info_map.lock()`), the watcher reaps the PID, finds `None`, sets `should_panic = true`, and calls `panic!("Launcher detected sandbox exit")` at line 156, crashing the replica. [5](#0-4)
