### Title
Unconditional `panic!` in `From<LedgerBurnError> for WithdrawalError` Reachable by Unprivileged Caller â€” (`rs/ethereum/cketh/minter/src/endpoints.rs`)

### Summary

The `From<LedgerBurnError> for WithdrawalError` conversion in `endpoints.rs` unconditionally panics on the `AmountTooLow` variant. This variant is reachable by an unprivileged user when the ckETH ledger's `transfer_fee` exceeds the minter's `cketh_minimum_withdrawal_amount` â€” a realistic condition after a ledger upgrade. The panic causes a canister trap in the `withdraw_eth` update handler and, because the trap fires after an `await` point, the per-caller withdrawal guard acquired before the await is never released.

---

### Finding Description

**Entrypoint:** `withdraw_eth` (`main.rs:266`) â€” a public `#[update]` method callable by any non-anonymous principal.

**Guard check (minter-side):** [1](#0-0) 

The minter checks `amount >= s.cketh_minimum_withdrawal_amount`. If the ledger's `transfer_fee` has been raised above this value (e.g., via a ledger upgrade), an amount in the range `[cketh_minimum_withdrawal_amount, transfer_fee)` passes the minter's guard but is rejected by the ledger.

**Ledger-side rejection path:**

The ICRC1 ledger computes `min_burn_amount = min(transfer_fee, balance)` and returns `TransferFromError::BadBurn` when `amount < min_burn_amount`: [2](#0-1) 

**Conversion to `LedgerBurnError`:**

`ledger_client.rs` converts `TransferFromError::BadBurn` to `LedgerBurnError::AmountTooLow` without panicking: [3](#0-2) 

**The panic:**

`main.rs:338` calls `WithdrawalError::from(e)`, which hits the unconditional `panic!`: [4](#0-3) 

The comment in the panic message asserts this case "should always" be impossible, but that invariant is only maintained if the minter's minimum is kept in sync with the ledger fee â€” which is not enforced atomically.

**Guard leak:**

The per-caller guard is acquired before the `await`: [5](#0-4) 

Because IC canisters compile with `panic = "abort"`, Rust's drop glue does not run on panic. The guard acquisition (committed in the pre-`await` message) is never undone, permanently occupying that caller's guard slot until the canister is upgraded or restarted.

**Contrast with `withdraw_erc20`:**

The ERC-20 withdrawal path correctly converts `LedgerBurnError::AmountTooLow` to a returnable `LedgerError::AmountTooLow` without panicking: [6](#0-5) 

The ckETH path is the only one that panics.

---

### Impact Explanation

1. **Trapped update call:** The caller receives a canister trap instead of a graceful `WithdrawalError::AmountTooLow`. Their ckETH is not burned (the burn call failed), but they get no actionable error.
2. **Guard slot permanently leaked:** The caller's withdrawal guard slot is never released, permanently blocking that principal from submitting future `withdraw_eth` calls until the minter canister is upgraded.

---

### Likelihood Explanation

The condition requires `ledger_transfer_fee > cketh_minimum_withdrawal_amount`. This is realistic:
- The ckETH ledger fee is a governance-controlled parameter.
- Any ledger upgrade that raises the fee creates a window (until the minter's minimum is also updated) where this panic is triggerable.
- The attacker only needs to know the current ledger fee and submit an amount in the gap â€” no privileged access required.

---

### Recommendation

Replace the `panic!` in the `LedgerBurnError::AmountTooLow` arm with a graceful return of `WithdrawalError::AmountTooLow`, mirroring the ERC-20 path:

```rust
LedgerBurnError::AmountTooLow { minimum_burn_amount, .. } => {
    Self::AmountTooLow {
        min_withdrawal_amount: minimum_burn_amount,
    }
}
```

Additionally, add a pre-flight check against the ledger's actual fee (or keep the minter minimum strictly in sync with the ledger fee via governance) to prevent the invariant from being violated in the first place.

---

### Proof of Concept

A state-machine test can:
1. Configure the minter with `cketh_minimum_withdrawal_amount = X`.
2. Stub the ckETH ledger to return `TransferFromError::BadBurn { min_burn_amount: X + 1 }` for any `transfer_from` call.
3. Call `withdraw_eth` with `amount = X` (passes the minter's guard).
4. Assert the update call traps (panics) rather than returning `Err(WithdrawalError::AmountTooLow { ... })`.
5. Assert the caller's guard slot is still occupied after the trap (blocking a second `withdraw_eth` call from the same principal).
