### Title
Missing Default Expiry on Chain Key Signing Requests Allows Indefinite Queue Accumulation - (`rs/consensus/chain_key/src/lib.rs`)

### Summary
The `signature_request_timeout_ns` field in `ChainKeyConfig` is `Option<u64>` and defaults to `None`. When `None`, the `reject_if_invalid` function in the chain key payload builder unconditionally skips the time-based expiry check, meaning `SignWithThresholdContext` entries (ECDSA, Schnorr, VetKD signing requests) remain pending in replicated state indefinitely with no cancellation mechanism. This is the IC analog of the `encodeTransactionData()` missing-deadline finding: signed data stays valid as long as it has not been executed, and there is no way to invalidate a pending signing request.

---

### Finding Description

`SignWithThresholdContext` is the replicated-state record for a pending threshold signing request. It stores the request, the signing arguments (message hash for ECDSA, full plaintext message for Schnorr), derivation path, and `batch_time` (submission time). It has **no `deadline` or `expires_at` field**: [1](#0-0) 

The only expiry mechanism is the subnet-level `signature_request_timeout_ns` in `ChainKeyConfig`: [2](#0-1) 

This field is explicitly documented as optional with no default:

> "The number of nanoseconds that a chain key signature request will time out. **If none is specified, no request will time out.**" [3](#0-2) [4](#0-3) 

In `get_enabled_keys_and_expiry`, when `signature_request_timeout_ns` is `None`, `request_expiry_time` is `None`: [5](#0-4) 

In `reject_if_invalid`, the time-based expiry check is guarded by `is_some_and(...)`, so when `request_expiry.time` is `None`, the check is **entirely skipped**: [6](#0-5) 

The `SignWithThresholdContext` is pushed into `subnet_call_context_manager` with no deadline: [7](#0-6) 

Numerous production subnet configurations set `signature_request_timeout_ns: None`: [8](#0-7) 

---

### Impact Explanation

1. **No cancellation path**: Once a canister submits `sign_with_ecdsa` or `sign_with_schnorr`, the `SignWithThresholdContext` entry persists in replicated state indefinitely. There is no management-canister API to cancel a pending signing request. If the canister is upgraded, the context changes, or the signed data becomes stale, the signature will still be produced and delivered.

2. **Queue exhaustion**: Each key has a bounded `max_queue_size`. Stale, never-expiring requests occupy queue slots permanently until processed. A canister (paying cycles) that submits requests and stops consuming responses can fill the queue, causing `sign_with_threshold contexts count >= dynamic_queue_size` rejections for all other canisters on that subnet: [9](#0-8) 

3. **Schnorr full-message exposure**: For `sign_with_schnorr`, the full plaintext `message` (not a hash) is stored in `SchnorrArguments` in replicated state with no expiry: [10](#0-9) 

---

### Likelihood Explanation

Any canister on a chain-key-enabled subnet can call `sign_with_ecdsa` or `sign_with_schnorr`. The majority of production subnet configurations leave `signature_request_timeout_ns` as `None`. The attacker path is direct: submit signing requests, pay cycles, and abandon them. No privileged access, governance majority, or key compromise is required. The queue-exhaustion scenario requires only that the attacker can afford cycles to fill `max_queue_size` slots.

---

### Recommendation

1. Set a non-`None` default for `signature_request_timeout_ns` in `ChainKeyConfig` (e.g., matching the existing `MAX_INGRESS_TTL` pattern used for ingress messages).
2. Add a `deadline` or `expires_at` field directly to `SignWithThresholdContext` so individual requests carry their own validity window, independent of subnet-level configuration.
3. Expose a management-canister API to cancel a pending signing request by callback ID, analogous to the nonce-invalidation function added in the PayrollManager fix.

---

### Proof of Concept

1. Deploy a canister on a chain-key subnet where `signature_request_timeout_ns = None`.
2. Call `sign_with_ecdsa` (or `sign_with_schnorr`) `max_queue_size` times, paying the required cycles each time, but never await the responses (or trap after submission).
3. Observe that `sign_with_threshold_contexts_count >= dynamic_queue_size` is now true: [11](#0-10) 

4. Any subsequent `sign_with_ecdsa` call from any canister on that subnet for that key is rejected with `"request queue for key â€¦ is full"`.
5. Because `signature_request_timeout_ns` is `None`, `reject_if_invalid` never evicts the stale contexts: [12](#0-11) 

The queue remains blocked until the subnet processes all stale requests, which may take an unbounded amount of time under load.
