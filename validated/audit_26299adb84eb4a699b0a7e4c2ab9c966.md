### Title
Permissionless `deploy_erc20_token` Combined with Unauthenticated `ft_on_transfer` Enables Unbacked ERC-20 Mirror Token Minting - (File: `engine/src/contract_methods/connector.rs`)

### Summary
`ft_on_transfer` is a public NEAR entry point that mints ERC-20 mirror tokens based solely on `predecessor_account_id` as the NEP-141 token identifier. Because `deploy_erc20_token` has no access control, any unprivileged NEAR account can register itself as a NEP-141 token and then call `ft_on_transfer` directly — without ever transferring any real NEP-141 tokens — to mint an arbitrary quantity of the corresponding ERC-20 mirror token. The attacker can then swap those unbacked tokens for legitimately-backed ERC-20 tokens on Aurora's EVM and withdraw the backed tokens via the exit precompile, draining real NEP-141 assets held by the engine.

### Finding Description

**Step 1 — Permissionless NEP-141 registration.**

`deploy_erc20_token` enforces only `require_running`; there is no caller whitelist or ownership check: [1](#0-0) 

Any NEAR account can call this with its own account ID as the `nep141` argument, causing the engine to deploy an ERC-20 contract and write the `nep141 → erc20` mapping into storage.

**Step 2 — `ft_on_transfer` trusts `predecessor_account_id` unconditionally.**

The public entry point `ft_on_transfer` reads `predecessor_account_id` from the NEAR runtime and uses it as the NEP-141 token identifier: [2](#0-1) 

The only branch check is whether the predecessor equals the stored connector account (for base-token minting). For every other predecessor it falls through to `receive_erc20_tokens`, passing the predecessor as the `token` argument: [3](#0-2) 

**Step 3 — `receive_erc20_tokens` mints without verifying a real transfer occurred.**

`receive_erc20_tokens` looks up the ERC-20 address for the supplied `token` account and calls the ERC-20 `mint` function with the attacker-controlled `amount`: [4](#0-3) 

There is no check that the predecessor actually executed an `ft_transfer_call` on a real NEP-141 contract, nor that any tokens moved into the engine's custody. The `amount` field comes entirely from the JSON body of the call, which the attacker controls.

**Step 4 — Public WASM entry points expose both functions.**

Both `ft_on_transfer` and `deploy_erc20_token` are exported as `no_mangle` WASM functions callable by any NEAR account: [5](#0-4) 

### Impact Explanation

An attacker mints an unbounded quantity of ERC-20 mirror tokens (e.g., a mirror of their own dummy NEP-141 account) at zero cost. On Aurora's EVM those tokens are indistinguishable from legitimately-bridged tokens. The attacker swaps them via any on-chain DEX for ERC-20 tokens that *are* backed by real NEP-141 deposits, then withdraws those backed tokens through the `ExitToNear` precompile. The real NEP-141 tokens held by the engine are transferred to the attacker. Remaining legitimate users cannot fully redeem their ERC-20 holdings — the bridge is insolvent.

**Impact class**: Critical — direct theft of user funds (NEP-141 tokens at rest in the engine) and insolvency of the ERC-20 mirror accounting.

### Likelihood Explanation

The attack requires only two permissionless NEAR transactions (one `deploy_erc20_token`, one `ft_on_transfer`) and a standard EVM swap. No privileged keys, governance capture, or oracle manipulation is needed. Any NEAR account holder can execute it.

### Recommendation

1. **Add access control to `deploy_erc20_token`**: restrict callers to the engine owner or a governance-approved whitelist, analogous to how `mirror_erc20_token` enforces `require_owner_only`.
2. **Validate the caller of `ft_on_transfer`**: maintain an allowlist of registered NEP-141 contract account IDs and revert if `predecessor_account_id` is not on it, or require that the predecessor is the same account ID that was registered via the controlled `deploy_erc20_token` path.
3. **Enforce the NEP-141 `ft_transfer_call` invariant**: `ft_on_transfer` should only be reachable as a callback from a legitimate `ft_transfer_call`; consider asserting `promise_results_count() >= 1` or checking a NEAR-level predecessor constraint.

### Proof of Concept

```
# 1. Attacker registers their own account as a NEP-141 token (no access control)
near call aurora deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# 2. Attacker calls ft_on_transfer directly — predecessor is automatically attacker.near
#    amount is attacker-chosen; msg is attacker's EVM address
near call aurora ft_on_transfer \
  '{"sender_id":"attacker.near","amount":"1000000000000000000000000","msg":"<attacker_evm_hex>"}' \
  --accountId attacker.near

# 3. On Aurora EVM: swap the freshly minted (unbacked) ERC-20 tokens
#    for a legitimately-backed ERC-20 token (e.g., USDC mirror) via any DEX.

# 4. Call withdrawToNear on the backed ERC-20 — exit precompile triggers
#    ft_transfer on the real USDC NEP-141 contract, sending real USDC to attacker.
```

At step 2, `ft_on_transfer` finds `attacker.near` in the `nep141→erc20` map (written in step 1), calls `receive_erc20_tokens`, and mints the requested amount to the attacker's EVM address with no NEP-141 tokens ever deposited. [6](#0-5)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
