### Title
Fatal Block-Processing Halt via Unguarded `L2AssetTracker` Revert in `notify_l2_asset_tracker` — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function, called for every L1→L2 deposit, makes a push-based EVM call to the `L2_ASSET_TRACKER_ADDRESS` contract. If that call reverts for any reason, the bootloader immediately escalates to a **fatal internal error that halts all block processing**. Because the `amount` argument is derived from attacker-controlled L1 transaction data, an adversary who can craft an L1→L2 deposit that causes `handleFinalizeBaseTokenBridgingOnL2` to revert can permanently freeze the L2 chain's ability to process any further L1→L2 deposits.

---

### Finding Description

In `process_l1_transaction.rs`, after the main transaction body executes, the bootloader calls `mint_base_token` up to three times (value mint, operator fee, refund). Each call internally invokes `notify_l2_asset_tracker`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs
fn notify_l2_asset_tracker<...>(
    ...
    amount: U256,
    l1_chain_id: U256,
    ...
) -> Result<(), BootloaderSubsystemError>
{
    if amount > U256::ZERO || Config::SIMULATION {
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]); // handleFinalizeBaseTokenBridgingOnL2
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution { result: asset_tracker_result, .. } =
                BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                    system, system_functions, memories, &calldata,
                    &L2_BASE_TOKEN_ADDRESS,
                    &L2_ASSET_TRACKER_ADDRESS,
                    ...
                )?;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            ).into());
        }
    }
    Ok(())
}

The code comment itself acknowledges the design choice:

> "Failure halts block processing — if the asset tracker reverts, the chain's token accounting would be inconsistent, so we treat it as fatal rather than silently continuing with incorrect bookkeeping." [2](#0-1) 

The `amount` passed to `handleFinalizeBaseTokenBridgingOnL2` originates directly from the L1→L2 transaction's `total_deposited` field, which is set by the L1 sender. The `l1_chain_id` is read from storage slot 154 of `L2_ASSET_TRACKER_ADDRESS`: [3](#0-2) 

The `L2AssetTracker` is a real EVM contract (source: `era-contracts/L2AssetTracker.sol`) deployed at a fixed address. Its `handleFinalizeBaseTokenBridgingOnL2` function tracks cumulative chain balances and migration state. Any arithmetic overflow in the contract's internal accounting (e.g., `chainBalance[fromChainId][assetId] + amount` overflowing `uint256`), or any other revert condition reachable via attacker-controlled `amount`, will propagate as a fatal error to the bootloader.

The `notify_l2_asset_tracker` is called three times per deposit: [4](#0-3) [5](#0-4) 

The predeployed contract setup confirms `L2AssetTracker` is always present after the upgrade: [6](#0-5) 

---

### Impact Explanation

If an attacker can craft an L1→L2 deposit whose `amount` causes `handleFinalizeBaseTokenBridgingOnL2` to revert (e.g., by overflowing the `L2AssetTracker`'s internal chain-balance accumulator), the bootloader escalates to a fatal internal error. This halts processing of the entire block. Because L1→L2 transactions cannot be skipped or invalidated by the sequencer, the chain is permanently unable to process any further L1→L2 deposits until a protocol upgrade fixes the `L2AssetTracker` state or the bootloader logic. This is a complete DoS of the L1→L2 bridge deposit path.

---

### Likelihood Explanation

The `amount` field in an L1→L2 transaction is fully attacker-controlled: any L1 user can submit a deposit with an arbitrary `total_deposited` value. The `L2AssetTracker` accumulates a running `chainBalance` per `(fromChainId, assetId)` pair. Since this balance is publicly readable on-chain, an attacker can compute the exact `amount` needed to overflow it (`type(uint256).max - currentBalance + 1`) and submit a single L1→L2 deposit to trigger the revert. The attack requires no privileged access, no governance majority, and no external oracle manipulation — only the ability to submit an L1→L2 transaction, which is permissionless.

---

### Recommendation

1. **Degrade gracefully instead of halting**: Wrap the `notify_l2_asset_tracker` call in a recoverable error path. If `handleFinalizeBaseTokenBridgingOnL2` reverts, record the failure (e.g., emit a log or store a flag) and continue block processing rather than treating it as a fatal error. The asset-tracker accounting inconsistency can be reconciled in a subsequent upgrade.

2. **Bound the `amount` passed to `L2AssetTracker`**: Before calling `notify_l2_asset_tracker`, validate that the cumulative chain balance in `L2AssetTracker` will not overflow. If it would, cap or reject the notification rather than propagating a revert.

3. **Separate the fatal-error boundary**: The current design conflates "asset tracker accounting is wrong" with "block processing must stop". These should be decoupled: accounting inconsistency is a recoverable protocol-level concern, not a reason to halt the entire chain.

---

### Proof of Concept

1. Read the current `chainBalance[L1_CHAIN_ID][BASE_TOKEN_ASSET_ID]` from `L2_ASSET_TRACKER_ADDRESS` storage.
2. Compute `attack_amount = type(uint256).max - chainBalance + 1`.
3. Submit an L1→L2 priority transaction with `total_deposited = attack_amount` (and matching `gas_limit * gas_price` covered by the deposit).
4. The bootloader calls `notify_l2_asset_tracker(attack_amount, l1_chain_id, ...)`.
5. Inside `handleFinalizeBaseTokenBridgingOnL2`, the addition `chainBalance + attack_amount` overflows, causing a revert.
6. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
7. Block processing halts. All subsequent L1→L2 deposits in the same and future blocks are blocked until a protocol upgrade. [7](#0-6)

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
```
