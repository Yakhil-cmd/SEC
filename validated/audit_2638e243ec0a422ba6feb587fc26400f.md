Audit Report

## Title
SNS Governance Computes Proposal Ballots from Live Neuron State Without Historical Snapshot Protection, Enabling Voting Power Inflation - (File: `rs/sns/governance/src/governance.rs`)

## Summary
`compute_ballots_for_new_proposal` in SNS governance iterates directly over live neuron state and records current voting power into proposal ballots with no spike-detection guard. NNS governance has an explicit `VotingPowerSnapshots` mechanism that falls back to a historical snapshot when current total voting power exceeds 1.5× the historical minimum; SNS has no equivalent. An attacker holding sufficient liquid SNS tokens can stake them immediately before submitting a proposal, receive a ballot reflecting the full inflated stake, auto-vote Yes as proposer, and pass a malicious proposal in a single atomic sequence.

## Finding Description
`compute_ballots_for_new_proposal` (rs/sns/governance/src/governance.rs L5255–5279) iterates `self.proto.neurons` and calls `v.voting_power(now_seconds, ...)` for every eligible neuron, writing the result directly into the ballot map with no historical comparison:

```rust
for (k, v) in self.proto.neurons.iter() {
    let voting_power = v.voting_power(now_seconds, ...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { vote: Vote::Unspecified as i32, voting_power, ... });
}
```

`voting_power` resolves to `voting_power_stake_e8s()` which reflects the neuron's current `cached_neuron_stake_e8s` at call time (rs/sns/governance/src/neuron.rs L196–205). There is no timer task, no snapshot store, and no spike-detection logic anywhere under `rs/sns/` — confirmed by a complete absence of `VotingPowerSnapshot`, `VOTING_POWER_SNAPSHOTS`, or any equivalent symbol in that subtree.

By contrast, NNS governance's `compute_ballots_for_standard_proposal` (rs/nns/governance/src/governance.rs L5497–5524) computes a current snapshot and then checks `VOTING_POWER_SNAPSHOTS.previous_ballots_if_voting_power_spike_detected(...)`. If the current total exceeds 1.5× the historical minimum, it substitutes the historical snapshot for ballot creation. The daily `SnapshotVotingPowerTask` (rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs) populates this store and explicitly skips recording a new snapshot when the latest is already a spike, preventing the attacker from poisoning the snapshot store.

SNS has no analogous task, store, or guard. The exploit path is:
1. Attacker holds large liquid SNS token balance.
2. Attacker calls `manage_neuron::ClaimOrRefresh` to stake into a neuron with dissolve delay ≥ `neuron_minimum_dissolve_delay_to_vote_seconds`.
3. Attacker immediately calls `make_proposal` (e.g., `TransferSnsTreasuryFunds` or `UpgradeSnsControlledCanister`).
4. `compute_ballots_for_new_proposal` captures the full inflated voting power into the ballot.
5. The proposer's neuron auto-votes Yes at creation time.
6. If the attacker's stake meets `minimum_yes_proportion_of_total` of the current total, the proposal is decided immediately in favor.
7. Treasury funds are transferred to the attacker, or a malicious WASM is installed.

## Impact Explanation
The impact is unauthorized control of SNS-governed assets and canisters: treasury drain via `TransferSnsTreasuryFunds`, arbitrary canister upgrade via `UpgradeSnsControlledCanister`, or governance parameter manipulation. This matches the High–Critical allowed impact: "Unauthorized access to governance assets, wallets, or canister-controlled funds" and potentially "Theft or permanent loss of in-scope assets." For SNS DAOs with low total staked supply or concentrated token distribution, the attacker's required stake to meet `minimum_yes_proportion_of_total` is achievable, making the impact Critical. For DAOs with high staking participation the barrier is higher, placing it at High.

## Likelihood Explanation
No privileged role, admin key, or social engineering is required. The only precondition is holding enough liquid SNS tokens to represent the `minimum_yes_proportion_of_total` fraction of total voting power at the moment of proposal creation. For SNS DAOs in early stages or with low staking participation this threshold is reachable by a single large token holder. The attack is atomic (stake → propose in the same session), repeatable, and leaves no window for defenders to react before the proposal is decided.

## Recommendation
Port the NNS `VotingPowerSnapshots` spike-detection mechanism to SNS governance:
- Add a periodic timer task (analogous to `SnapshotVotingPowerTask`) that records daily snapshots of total voting power per neuron into a stable store.
- In `compute_ballots_for_new_proposal`, compare the current total voting power against the historical minimum from the snapshot store; if it exceeds the spike threshold (e.g., 1.5×), substitute the historical snapshot for ballot creation.
- Guard the snapshot task against recording a new snapshot when the latest is already a spike (mirroring `is_latest_snapshot_a_spike` in NNS).
- Until the snapshot store is populated (first N days of an SNS), apply a conservative fallback such as rejecting proposals whose total voting power exceeds a configurable ceiling, or requiring a minimum staking age before a neuron's voting power is counted in ballots.

## Proof of Concept
**Minimal deterministic test plan (PocketIC or SNS integration test):**

1. Initialize an SNS with total token supply S; stake S/4 into existing neuron A (controlled by honest party), leaving S/2 liquid with attacker.
2. Attacker calls `manage_neuron::ClaimOrRefresh` to stake S/2 into neuron B with dissolve delay = `neuron_minimum_dissolve_delay_to_vote_seconds`.
3. In the same round, attacker calls `make_proposal` with action `TransferSnsTreasuryFunds { amount: treasury_balance, to: attacker_account }`.
4. Assert: `compute_ballots_for_new_proposal` returns a ballot for neuron B with voting power proportional to S/2.
5. Assert: neuron B auto-votes Yes; total Yes voting power = S/2 > `minimum_yes_proportion_of_total` × (S/4 + S/2); proposal status = Adopted.
6. Assert: treasury transfer executes.

The absence of any spike-detection branch in `compute_ballots_for_new_proposal` means step 4–6 succeed unconditionally, confirming the vulnerability. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
