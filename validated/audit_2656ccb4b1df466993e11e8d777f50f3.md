Audit Report

## Title
Unprivileged Caller Can Force Another Principal Into SNS Swap Participation, Bypassing Confirmation Text Consent — (File: rs/sns/swap/canister/canister.rs)

## Summary
`refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` resolves the effective buyer from the request argument rather than from the authenticated caller, with no authorization check that `buyer == caller`. Any unprivileged ingress sender can commit another principal's pre-funded ICP into an SNS swap by supplying the publicly readable `confirmation_text` on their behalf, recording participation as if the victim explicitly agreed to the swap terms. The `confirmation_text` consent mechanism is rendered meaningless.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 130–134, the effective buyer principal is resolved from the caller-supplied argument with no identity check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
};
```

This `p` is passed directly into `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` lines 1134–1163), which also receives the caller-supplied `confirmation_text`. The inner function has no caller parameter and performs no authorization check that the entity accepting the confirmation text is the same principal whose funds are being committed. `validate_confirmation_text` (`rs/sns/swap/src/swap.rs` lines 363–384) only compares the supplied string against the SNS init payload value — it does not verify that the caller is the buyer:

```rust
self.validate_confirmation_text(confirmation_text)?;
// ...
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),  // victim's subaccount
};
```

The `confirmation_text` is stored in the public `Init` struct and is readable by anyone via `get_init`. The doc comment on `refresh_buyer_token_e8s` (line 1123) states "the caller of this function must accept this confirmation" — but the code does not enforce that the caller is the buyer.

Critically, the existing test suite in `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 explicitly demonstrates and validates this behavior: after the wealthy user transfers ICP, the `default_sns_agent` (a completely different identity) successfully calls `refresh_buyer_tokens` with `buyer = wealthy_user_identity.principal_id` and the call succeeds (`.unwrap()` does not panic). The same call before the transfer fails (lines 807–818) only because the balance is 0 (below `min_participant_icp_e8s`) — not due to any authorization check. The `participate_in_swap` helper in `rs/sns/test_utils/src/state_test_helpers.rs` lines 291–301 also calls `refresh_buyer_tokens` with `buyer: participant_principal_id.to_string()` using `execute_ingress` (anonymous caller), confirming the pattern is exercised throughout the codebase.

The full exploit path: attacker reads `confirmation_text` via `get_init` → monitors ICP ledger for victim's transfer to swap subaccount → calls `refresh_buyer_tokens({ buyer: victim, confirmation_text: <public_text> })` → `BuyerState` for victim is created with the full subaccount balance → when swap commits, victim's ICP is swept and SNS tokens are minted to victim irreversibly.

## Impact Explanation
This is a **High** impact finding matching: *Significant SNS security impact with concrete user or protocol harm*. The victim's ICP is committed to the swap and, upon `COMMITTED` lifecycle, irreversibly swept — without the victim ever calling `refresh_buyer_tokens` themselves or explicitly agreeing to the terms. The `confirmation_text` mechanism, designed to enforce legal/terms agreement, is completely bypassed since the text is public and any third party can supply it. The victim receives SNS tokens in exchange but is forced into an irreversible financial transaction without consent, at a time and price they did not choose.

## Likelihood Explanation
No privileged access is required — any ingress sender can call `refresh_buyer_tokens`. The victim precondition (pre-funded swap subaccount) is publicly observable on the ICP ledger. The `confirmation_text` is publicly readable via `get_init`. Users commonly pre-fund their swap subaccounts before calling `refresh_buyer_tokens` themselves, creating a reliable exploitation window. The attack is repeatable against any pre-funded principal. Likelihood: **Medium-High**.

## Recommendation
Add a caller-identity check in `refresh_buyer_tokens` before accepting a third-party `buyer` value. If the specified `buyer` differs from the caller, reject the call outright (or require explicit authorization from the buyer principal):

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() {
        panic!("caller is not authorized to refresh tokens on behalf of {}", specified);
    }
    specified
};
```

If third-party notification is a required use case (e.g., helper services), it should be gated on a separate privileged path or require a signed delegation from the buyer principal.

## Proof of Concept
The existing integration test in `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–926 already constitutes a reproducible proof: after `wealthy_user_identity` transfers ICP to the swap subaccount, `default_sns_agent` (a different identity) calls `refresh_buyer_tokens(Some(wealthy_user_identity.principal_id), None)` and the call succeeds. A targeted unit test can be constructed as follows:

1. Set up a swap in `Open` state with a `confirmation_text = "I agree"` in the SNS init payload.
2. Transfer ICP from principal `V` to `subaccount(swap_canister, V)` on the ICP ledger.
3. From a completely different principal `A` (attacker), call `refresh_buyer_tokens({ buyer: V.to_string(), confirmation_text: Some("I agree") })`.
4. Assert the call succeeds and `get_buyer_state(V)` returns a non-empty `BuyerState` with the transferred amount.
5. Assert `V` never called `refresh_buyer_tokens` themselves.

This is directly reproducible using `StateMachine` tests with `execute_ingress_as(attacker_principal, swap_canister_id, "refresh_buyer_tokens", ...)` supplying `buyer = victim_principal`.
