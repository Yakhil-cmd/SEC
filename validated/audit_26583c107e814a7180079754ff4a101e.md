### Title
Weak Time-Seeded PRNG Provides Fully Predictable `RuleId` UUID Generation in Rate-Limit Canister — (File: `rs/boundary_node/rate_limits/canister/random.rs`)

---

### Summary

The rate-limit canister registers a custom `getrandom` implementation seeded with only 8 bytes of IC consensus time (`ic_cdk::api::time()`) padded with 24 constant bytes (`0x2A`/`42`). Every `RuleId` UUID ever generated by the canister is therefore deterministically computable by any observer who knows the canister's initialization block time — including a malicious block maker who controls that timestamp.

---

### Finding Description

In `rs/boundary_node/rate_limits/canister/random.rs`, the thread-local ChaCha20 RNG is initialized once at canister start:

```rust
thread_local! {
  static RNG: RefCell<ChaCha20Rng> = {
    let mut seed = [42; 32];          // 24 bytes are the constant 0x2A
    seed[..8].copy_from_slice(&time().to_le_bytes()); // only 8 bytes vary
    RefCell::new(ChaCha20Rng::from_seed(seed))
  };
}
``` [1](#0-0) 

This RNG is registered as the system-wide `getrandom` provider:

```rust
getrandom::register_custom_getrandom!(custom_getrandom_bytes_impl);
``` [2](#0-1) 

Every new rate-limit rule added via `add_config` calls `generate_random_uuid()`, which draws from this RNG via `getrandom::getrandom`:

```rust
fn generate_random_uuid() -> Result<Uuid, anyhow::Error> {
    let mut buf = [0_u8; 16];
    getrandom::getrandom(&mut buf)...
    let uuid = Uuid::from_slice(&buf)...
    Ok(uuid)
}
``` [3](#0-2) 

The resulting UUID becomes the `RuleId` stored in stable memory and returned to callers:

```rust
let rule_id = RuleId(generate_random_uuid()?);
if self.canister_api.get_rule(&rule_id).is_some() {
    return Err(AddConfigError::Internal(...));
}
``` [4](#0-3) 

The effective seed entropy is only 64 bits (the nanosecond timestamp), with the remaining 192 bits fixed at `0x2A`. Because `time()` on the IC returns the consensus time set by the block maker, a subnet node acting as block maker can choose this value within the protocol-allowed drift window, making the entire future UUID sequence computable before the canister is initialized.

---

### Impact Explanation

1. **Rule-ID enumeration without authorization.** `get_rule_by_id` and `get_rules_by_incident_id` are public query endpoints. An attacker who predicts the initialization timestamp can pre-compute every `RuleId` ever issued and enumerate all rules — including confidential, not-yet-disclosed ones — by querying those IDs directly, bypassing the confidentiality access-control layer that is otherwise enforced by `AccessLevelResolver`. [5](#0-4) 

2. **Targeted DoS via UUID pre-collision.** An authorized principal (or a compromised one) who knows the next UUID can pre-insert a rule with that ID, causing `add_config` to return `AddConfigError::Internal` for the legitimate submission, silently blocking new rate-limit rule deployment. [4](#0-3) 

The confidentiality model of the rate-limit canister explicitly distinguishes between disclosed and undisclosed rules; undisclosed rules are meant to be visible only to API boundary nodes and the authorized principal. Predictable UUIDs collapse that boundary for any external observer.

---

### Likelihood Explanation

The IC consensus time is observable on-chain (it appears in every certified block). Any party that can read the initialization block — which is public — can reconstruct the exact 32-byte seed and replay the ChaCha20 stream offline. A malicious block maker can additionally *choose* the timestamp within the allowed drift, giving them full control over the seed. No threshold corruption or privileged access is required; a single subnet node acting as block maker suffices.

---

### Recommendation

Replace the time-based seed with a call to the IC management canister's `raw_rand` endpoint during `init`/`post_upgrade`. Because `raw_rand` is asynchronous, the canister should defer UUID generation (or store the seed) until the async callback completes, or use a one-time timer that seeds the RNG immediately after initialization. The NNS governance canister already demonstrates this pattern correctly via its `SeedingTask`:

```rust
env.call_canister_method(IC_00, "raw_rand", ...).await
``` [6](#0-5) 

---

### Proof of Concept

1. Read the initialization block of the rate-limit canister from the public IC state tree; extract the `time` field (nanoseconds since epoch).
2. Reconstruct the seed: bytes `[0..8]` = `time.to_le_bytes()`, bytes `[8..32]` = `[0x2A; 24]`.
3. Instantiate `ChaCha20Rng::from_seed(seed)` locally.
4. For each rule addition expected (observable from `add_config` update calls on-chain), draw 16 bytes from the RNG and construct a `Uuid` — this matches the canister's `RuleId` exactly.
5. Call `get_rule_by_id(predicted_uuid)` as an anonymous caller; the canister returns the full rule metadata (subject to the confidentiality formatter), exposing undisclosed rule content that should be restricted to API boundary nodes only.
