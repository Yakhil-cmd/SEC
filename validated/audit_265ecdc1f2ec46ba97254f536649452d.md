### Title
Permanent Token Freeze in `ExitToNear` Precompile When NEAR-Side Transfer Fails Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary
When a user bridges ERC-20 tokens from Aurora EVM back to NEAR via the `ExitToNear` precompile, the tokens are burned on the EVM side before the NEAR-side `ft_transfer` is attempted. If the NEAR-side transfer fails for any reason (e.g., recipient not registered with the NEP-141 contract, or a NEP-141 token with a blacklist rejects the transfer), and the `error_refund` compile-time feature is not enabled, no refund mechanism exists. The burned tokens are permanently lost.

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) executes a two-step cross-chain exit:

**Step 1 – EVM burn (irreversible):** The ERC-20 contract calls the precompile, which records the burn. The EVM state change is committed.

**Step 2 – NEAR promise:** A `ft_transfer` (or `ft_transfer_call`) promise is scheduled on the NEP-141 contract to deliver tokens to the NEAR recipient.

The callback args are constructed as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← no refund path compiled in
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent, `refund` is always `None`. The callback `exit_to_near_precompile_callback` then handles a failed promise as:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op: tokens are gone
};
``` [2](#0-1) 

The `refund_on_error` function, which would re-mint ERC-20 tokens to the caller's address, is never reached: [3](#0-2) 

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

### Impact Explanation

Any failure of the NEAR-side `ft_transfer` — including a NEP-141 token's built-in blacklist rejecting the recipient, the recipient account not being registered with the NEP-141 contract, or insufficient storage deposit — causes the ERC-20 tokens to be permanently destroyed with no recovery path. The EVM burn is already committed; there is no compensating mint. This is a **permanent freeze of user funds**.

### Likelihood Explanation

The trigger is realistic and unprivileged:
- NEP-141 tokens (including NEAR-native USDC equivalents) can implement blacklists or require explicit `storage_deposit` registration before receiving tokens.
- Any user who calls `ExitToNear` targeting an unregistered or blacklisted NEAR account will silently lose their tokens.
- No admin action is required; the attacker only needs to be the recipient (or cause the recipient to be unregistered/blacklisted).
- The `error_refund` feature is not a default Cargo feature — both `EvmErc20.bin` (without refund) and `EvmErc20V2.bin` (with refund) are present in the repository, indicating the non-refund path is a live production code path. [5](#0-4) 

### Recommendation

1. **Enable `error_refund` unconditionally** or promote it to a default feature so that `refund_call_args` is always populated and the re-mint path is always available on exit failure.
2. Alternatively, treat a failed `ft_transfer` promise as a hard error that reverts the EVM burn (not possible in NEAR's async model without pre-commitment), or use a pull-based pattern: record the burned amount in storage and allow the user to reclaim it if the NEAR-side transfer fails.
3. At minimum, document clearly that exits to unregistered or blacklisted NEAR accounts result in permanent token loss, and add a pre-flight check in the ERC-20 contract's `withdraw` function.

### Proof of Concept

1. Deploy a NEP-141 token that requires `storage_deposit` before receiving (standard NEP-141 behavior).
2. Bridge some of that token into Aurora EVM (mint ERC-20 mirror).
3. Call the ERC-20's `withdraw` function targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
4. The ERC-20 tokens are burned on the EVM side.
5. The `ft_transfer` promise fails because the recipient is not registered.
6. `exit_to_near_precompile_callback` is invoked; `args.refund` is `None`; the `else { None }` branch executes.
7. The user's ERC-20 balance is zero; the NEP-141 balance of the recipient is zero; the tokens are gone permanently.

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) already demonstrates this exact scenario and confirms the token loss when `error_refund` is absent. [6](#0-5)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
