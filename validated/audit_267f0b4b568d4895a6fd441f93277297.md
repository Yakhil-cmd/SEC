### Title
`execute_scheduled` Accepts NEAR Deposits That Are Permanently Frozen — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC router contract is annotated `#[payable]` but never reads or uses `env::attached_deposit()`. Any NEAR tokens a caller attaches to this call are deposited into the router sub-account and permanently frozen, because the router provides no mechanism to recover arbitrary NEAR amounts.

---

### Finding Description

`execute_scheduled` is intentionally open to any caller (not just the parent Aurora Engine), as documented in the code comment. It is also marked `#[payable]`: [1](#0-0) 

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The function body never calls `env::attached_deposit()`. The promise it creates uses the `attached_balance` field already embedded in the stored `PromiseArgs` at scheduling time: [2](#0-1) 

```rust
fn base_promise_create(promise: &PromiseCreateArgs) -> PromiseIndex {
    env::promise_create(
        ...
        NearToken::from_yoctonear(promise.attached_balance.as_u128()),
        ...
    )
}
```

The caller's attached deposit is therefore silently accepted and deposited into the router contract's account, but never forwarded or returned.

The router contract's only outbound NEAR transfer is `send_refund`, which sends a hardcoded fixed amount (`REFUND_AMOUNT = 2 NEAR`) exclusively to the parent account: [3](#0-2) 

There is no general-purpose withdrawal or refund function. Any NEAR attached to `execute_scheduled` beyond what `send_refund` covers is permanently unrecoverable by the caller.

---

### Impact Explanation

Any NEAR attached to `execute_scheduled` is permanently frozen in the router sub-account. The only recovery path requires the Aurora Engine owner to deploy a new router contract version containing a recovery function — an admin action unavailable to the affected user. This constitutes a **permanent freezing of user funds**.

---

### Likelihood Explanation

`execute_scheduled` is explicitly designed to be callable by any unprivileged account. A user who observes a scheduled promise and wants to trigger its execution may reasonably attach NEAR, believing they must fund the promise (especially since the function is `#[payable]` and the promise may itself carry a non-zero `attached_balance`). This is a realistic and foreseeable mistake. No special access or knowledge is required — only the nonce of a scheduled promise, which is logged on-chain via `near_sdk::log!`: [4](#0-3) 

---

### Recommendation

Remove the `#[payable]` annotation from `execute_scheduled`. Since the function never uses `env::attached_deposit()`, there is no legitimate reason for it to accept deposits. Without `#[payable]`, the NEAR runtime will automatically reject any call that attaches a non-zero deposit, preventing fund loss entirely.

---

### Proof of Concept

1. The Aurora Engine (parent) calls `schedule(promise)` where `promise.attached_balance = 5 NEAR`. The promise is stored at nonce `N`.
2. An unprivileged user sees the log `"Promise scheduled at nonce N"` and calls `execute_scheduled(N)` attaching 5 NEAR, believing they must fund the promise.
3. The 5 NEAR is deposited into the router sub-account (e.g., `{address}.aurora`).
4. `execute_scheduled` creates the promise using the stored `PromiseArgs.attached_balance` (the original 5 NEAR from the engine's wNEAR balance), completely ignoring the caller's 5 NEAR deposit.
5. The caller's 5 NEAR is permanently frozen in the router contract. `send_refund` only returns a fixed 2 NEAR to the parent; the caller has no recovery path. [1](#0-0) [3](#0-2)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
