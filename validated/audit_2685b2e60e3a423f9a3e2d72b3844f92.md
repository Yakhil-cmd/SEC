### Title
One-Sided `delta_gas` Adjustment in `compute_gas_refund` Systematically Overcharges Users When EVM Gas Exceeds Native Resource Cost - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` reconciliation between EVM gas consumption and native resource consumption is applied **asymmetrically**: it only ever increases `gas_used` (when native cost exceeds EVM gas), but never decreases it (when EVM gas exceeds native cost). This means users whose transactions consume more EVM gas than native resources are systematically overcharged — they pay for EVM gas that exceeds the actual proving cost, with no refund for the difference. A developer `// TODO` comment in the code explicitly acknowledges this open question.

---

### Finding Description

ZKsync OS implements a double resource accounting model: **Ergs** (EVM gas) and **native resources** (proving/pubdata cost). After execution, `compute_gas_refund` reconciles the two by computing `delta_gas`:

```
delta_gas = (native_used / native_per_gas) - gas_used
```

The documented intent (from `docs/double_resource_accounting.md`) is:

> If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee.

The code in `refund_calculation.rs` implements only the positive branch:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` — meaning EVM gas consumption exceeds native resource consumption — `gas_used` is **not reduced**. The user pays for the full EVM gas consumed, even though the actual proving cost (native resource) was lower. The `// TODO: return delta_gas to gas_used?` comment is a developer acknowledgment that this negative case is unhandled.

The final `gas_used` determines the fee paid and the refund returned:

- **Fee paid to operator**: `gas_used * gas_price`
- **Refund to sender**: `(gas_limit - gas_used) * gas_price`

When `delta_gas < 0` is silently dropped, `gas_used` is inflated above what the native resource model would justify, the refund is reduced, and the user overpays.

---

### Impact Explanation

Users who submit **compute-heavy but storage-light** transactions (high EVM gas, low native resource consumption) are systematically overcharged. Concretely:

- A transaction that consumes 100,000 EVM gas but only 50,000 native-equivalent gas pays for 100,000 gas.
- A transaction that consumes 50,000 EVM gas but 100,000 native-equivalent gas also pays for 100,000 gas (via the positive delta_gas branch).

Both users pay `max(gas_used_evm, native_used / native_per_gas)`. The first user overpays relative to the native resource model. The excess fee flows to the operator, representing a direct, quantifiable loss of funds for the transaction sender.

This is a resource accounting bug: the fee base includes EVM gas that exceeds the actual proving cost, analogous to the external report's inclusion of `marginFrom` (user-provided, non-borrowed funds) in the fee base.

---

### Likelihood Explanation

**High.** The condition `delta_gas < 0` (EVM gas > native cost) is triggered by any transaction where EVM computation is relatively expensive but pubdata/proving overhead is low — a common pattern for compute-heavy contracts (e.g., arithmetic-heavy logic, loops, in-memory operations) that do not write much storage. Every such transaction processed by the bootloader passes through `compute_gas_refund`, making this a systematic, always-reachable overcharge for a broad class of normal user transactions. No special privileges or unusual conditions are required.

---

### Recommendation

Apply the negative `delta_gas` symmetrically: when `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so that users are charged `max(minimal_gas_used, max(gas_used_evm - evm_refund, native_used / native_per_gas))`. This mirrors the existing positive branch and resolves the asymmetry flagged by the `// TODO` comment.

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native cost is less than EVM gas cost; refund the difference,
    // but never go below the minimal_gas_used floor.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

**Concrete scenario:**

1. User submits a ZK transaction with `gas_limit = 200,000`, `gas_price = 1000`, `native_per_gas = 50`.
2. Transaction executes: EVM consumes 150,000 gas (`gas_used = 150,000`); native resources consumed = 4,000,000 units.
3. `native_used / native_per_gas = 4,000,000 / 50 = 80,000`.
4. `delta_gas = 80,000 - 150,000 = -70,000` (negative).
5. Current code: `delta_gas < 0` → no adjustment → `gas_used` stays at 150,000.
6. User pays `150,000 * 1000 = 150,000,000` wei; refund = `(200,000 - 150,000) * 1000 = 50,000,000` wei.
7. Correct behavior: `gas_used` should be reduced toward `80,000`; user should pay `80,000,000` wei and receive `120,000,000` wei refund.
8. **Overcharge: 70,000,000 wei** transferred to operator instead of refunded to user.
