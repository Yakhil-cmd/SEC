Audit Report

## Title
Unvalidated `EquivocationProof` artifacts at heights ≤ `finalized_height` accumulate unboundedly in the unvalidated pool — (`rs/consensus/src/consensus/validator.rs`)

## Summary

`validate_equivocation_proofs` constructs its iteration range starting at `finalized_height.increment()`, permanently skipping any `EquivocationProof` in the unvalidated pool at or below `finalized_height`. The purger's equivocation-proof cleanup only targets the validated pool. The sole mechanism that cleans the unvalidated pool — `PurgeUnvalidatedBelow` — is gated on `expected_batch_height` advancing. A Byzantine P2P peer can exploit this gap to grow the unvalidated pool without bound, causing OOM and replica crash.

## Finding Description

**Validator skips low-height proofs entirely.**
`validate_equivocation_proofs` sets:

```rust
let range_to_validate = HeightRange::new(finalized_height.increment(), range.max);
``` [1](#0-0) 

Both the `existing_proofs` set and the subsequent `get_by_height_range` call use `range_to_validate`, so any `EquivocationProof` at height ≤ `finalized_height` is never iterated, never validated, never emitted as `HandleInvalid` or `RemoveFromUnvalidated`. It stays in the unvalidated pool indefinitely.

**Purger only cleans the validated pool for equivocation proofs.**
`purge_equivocation_proofs_by_finalized_height` emits only `PurgeValidatedOfTypeBelow`:

```rust
changeset.push(ChangeAction::PurgeValidatedOfTypeBelow(
    PurgeableArtifactType::EquivocationProof,
    finalized_height.increment(),
));
``` [2](#0-1) 

There is no corresponding `PurgeUnvalidatedOfTypeBelow` for equivocation proofs.

**The only unvalidated-pool cleanup is gated on `expected_batch_height` advancing.**
`purge_unvalidated_pool_by_expected_batch_height` only fires when `*prev_expected_batch_height < expected_batch_height`: [3](#0-2) 

If execution stalls (state sync, slow execution), `expected_batch_height` does not advance, this block never fires, and unvalidated equivocation proofs at low heights are never removed. Even on a healthy subnet, proofs at heights ≤ `finalized_height` accumulate between purge cycles since the validator never touches them.

**No per-peer quota or admission control found** for equivocation proofs in the artifact pool or P2P layer — searches for quota, per-peer limits, and unvalidated pool size bounds returned no results for this artifact type.

## Impact Explanation

A Byzantine P2P peer can continuously inject `EquivocationProof` artifacts (cryptographically invalid ones suffice — the validator never reaches signature-check code for these heights) at heights ≤ `finalized_height`. Memory grows without bound until the replica OOMs and crashes. This matches the allowed ICP bounty impact: **High — Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000). On a stalled subnet the accumulation is indefinite; on a healthy subnet the window between `PurgeUnvalidatedBelow` firings still allows significant accumulation.

## Likelihood Explanation

- Attacker only needs to be a P2P peer — a single Byzantine node below the fault threshold suffices.
- No valid cryptographic material is required; invalid proofs accumulate identically since the validator never reaches signature verification for these heights.
- The stalled-subnet condition (state sync, slow execution) amplifies the attack to indefinite accumulation but is not strictly required.
- The attack is repeatable and low-cost.

## Recommendation

In `validate_equivocation_proofs`, explicitly emit `RemoveFromUnvalidated` for any proof whose height is ≤ `finalized_height`:

```rust
if proof.height <= finalized_height {
    return Some(ChangeAction::RemoveFromUnvalidated(proof.into_message()));
}
```

Alternatively, add `PurgeUnvalidatedOfTypeBelow(EquivocationProof, finalized_height.increment())` in `purge_equivocation_proofs_by_finalized_height` alongside the existing validated-pool purge. A per-peer quota on unvalidated equivocation proofs would provide defense-in-depth.

## Proof of Concept

State-machine test:
1. Advance the pool to finalized height H.
2. Freeze `expected_batch_height` (simulate execution stall).
3. Insert N `EquivocationProof` artifacts at heights 1..H into the unvalidated pool via the P2P path (invalid signatures are fine).
4. Run `validate_equivocation_proofs` and `purger.on_state_change` in a loop.
5. Assert: unvalidated pool equivocation proof count remains N (none removed).
6. Assert: memory usage grows linearly with N.

Root cause confirmed at:
- [4](#0-3)  — range excludes `finalized_height` and below
- [5](#0-4)  — only validated pool is purged for equivocation proofs
- [3](#0-2)  — unvalidated pool purge gated on `expected_batch_height` advancing
