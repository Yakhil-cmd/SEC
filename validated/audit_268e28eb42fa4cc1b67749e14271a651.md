Audit Report

## Title
Cross-Subaccount Allowance Disclosure in `get_allowances` Due to Owner-Only Break Condition — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function iterates a `StableBTreeMap` ordered by `(owner, effective_subaccount, spender_owner, spender_subaccount)` but terminates the scan only when the `owner` field changes, not when the full `Account` (owner + subaccount) changes. Any unauthenticated caller can supply an arbitrary `from_account` and receive allowance entries belonging to other subaccounts of the same owner.

## Finding Description
`get_allowances` builds a range starting at `(from_account, min_spender)` and iterates forward through `ALLOWANCES_MEMORY`. The sole account-scope termination guard is:

```rust
// rs/ledger_suite/icrc1/ledger/src/lib.rs line 1204
if account_spender.account.owner != from.owner {
    break;
}
```

Because `Account::cmp` orders by `(owner, effective_subaccount())` (confirmed at `packages/icrc-ledger-types/src/icrc1/account.rs` lines 54–60), entries for `{owner: X, subaccount: [1;32]}` sort immediately before entries for `{owner: X, subaccount: [2;32]}`. After exhausting all spenders for the requested subaccount, the iterator advances into the next subaccount's entries. The owner check remains false, so those entries are pushed into the result. The missing guard is a full account equality check: `account_spender.account != from`.

The public query endpoint (`rs/ledger_suite/icrc1/ledger/src/main.rs` lines 1214–1231) applies no caller restriction and accepts any `from_account` argument, making the scan reachable by any principal.

## Impact Explanation
An unprivileged attacker can enumerate all allowances for every subaccount of any target owner with a single call using `from_account = {owner: X, subaccount: None}` (effective subaccount `[0;32]`, the BTree minimum). This discloses which subaccounts hold active approvals, which spenders are authorized, allowance amounts, and expiry times — data scoped by the ICRC-103 standard to the exact account specified. This constitutes unauthorized information disclosure from the ICRC ledger with concrete user harm, matching the **High** impact tier: *Significant ledger security impact with concrete user or protocol harm*.

## Likelihood Explanation
The endpoint is a `#[query]` call with no authentication, no rate-limiting beyond `max_take`, and no caller restriction. The only precondition is knowing a target principal, which is public on-chain. The exploit is a single canister call, trivially repeatable, and requires no special privileges or technical skill beyond basic IC interaction.

## Recommendation
Replace the owner-only break condition with a full account equality check:

```rust
// Before (lib.rs line 1204):
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```

`Account::eq` already normalizes `None` and `Some([0u8;32])` via `effective_subaccount()`, so the canonical subaccount equivalence is preserved.

## Proof of Concept

```rust
// Setup: owner X approves on two subaccounts
let sub1 = Some([1u8; 32]);
let sub2 = Some([2u8; 32]);
approve(X, sub1, spender_a, 1000);
approve(X, sub2, spender_b, 2000);

// Attacker queries only sub1
let result = icrc103_get_allowances(GetAllowancesArgs {
    from_account: Some(Account { owner: X, subaccount: sub1 }),
    prev_spender: None,
    take: None,
});

// Bug: result contains entries for BOTH sub1 AND sub2
assert!(result.iter().any(|a| a.from_account.subaccount == sub2)); // passes — should fail
```

A deterministic state-machine or PocketIC integration test can reproduce this by creating two approvals on distinct subaccounts of the same owner and asserting that a query scoped to one subaccount does not return entries for the other.
