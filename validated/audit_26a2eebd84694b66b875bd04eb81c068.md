### Title
Unchecked `NEP141Wei` Subtraction Panics Under `overflow-checks = true`, Aborting Bridge Accounting Transactions — (File: `engine-types/src/types/wei.rs`)

---

### Summary

The `NEP141Wei` type, which represents ETH-connector bridge balances, implements the `Sub` and `SubAssign` traits using plain `u128` arithmetic. The workspace `Cargo.toml` sets `overflow-checks = true` in both the `[profile.release]` and `[profile.dev]` sections. Any underflow in `NEP141Wei` subtraction will therefore cause a Rust panic/abort, aborting the NEAR transaction. This is the direct Rust analog of the Solidity `FeeLib` bug: arithmetic that silently wrapped in an older runtime now aborts in a newer one.

---

### Finding Description

`NEP141Wei` is defined as a `u128` newtype representing Wei amounts bridged to NEAR via the ETH connector:

```rust
pub struct NEP141Wei(u128);
```

Its `Sub` and `SubAssign` trait implementations use plain, unchecked `u128` arithmetic:

```rust
impl Sub for NEP141Wei {
    type Output = Self;
    fn sub(self, rhs: Self) -> Self::Output {
        Self(self.0 - rhs.0)   // plain u128 subtraction — panics on underflow
    }
}

impl SubAssign for NEP141Wei {
    fn sub_assign(&mut self, rhs: Self) {
        *self = *self - rhs;   // delegates to unchecked Sub
    }
}
``` [1](#0-0) 

The workspace release profile explicitly enables overflow checking:

```toml
[profile.release]
overflow-checks = true
``` [2](#0-1) 

The same flag is set in `[profile.dev]`: [3](#0-2) 

`checked_sub` and `checked_add` methods exist on `NEP141Wei` and are the safe alternatives, but the plain operator implementations are also exported and callable from any crate in the workspace. Any call site that uses the `-` or `-=` operator on `NEP141Wei` with a right-hand side larger than the left-hand side will trigger a panic, aborting the NEAR transaction. [4](#0-3) 

The `Fee` type, which wraps `NEP141Wei`, inherits the same risk through its `Add` implementation:

```rust
impl Add for Fee {
    type Output = Self;
    fn add(self, rhs: Self) -> Self::Output {
        Self(self.0 + rhs.0)   // delegates to NEP141Wei::add — panics on overflow
    }
}
``` [5](#0-4) 

---

### Impact Explanation

**Impact: High — Temporary freezing of funds.**

`NEP141Wei` is the canonical type for ETH-connector bridge balances (amounts of ETH bridged to NEAR). Any production code path in the connector that uses the plain `-` or `-=` operator on `NEP141Wei` — rather than `checked_sub` — will abort the NEAR transaction when an underflow occurs. Because NEAR transactions are atomic, an abort rolls back all state changes, leaving bridged funds inaccessible for the duration of the condition. If the underflow is reproducible (e.g., triggered by a specific sequence of bridge operations), it constitutes a reliable denial-of-service against the affected accounts' bridge withdrawals. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: High.**

`overflow-checks = true` is unconditionally set in both dev and release profiles, so the panic path is always active — there is no build configuration that avoids it. The `Sub` and `SubAssign` operators are standard Rust idioms that any developer would reach for naturally. The `NEP141Wei` type is used in the ETH connector's balance accounting, which is exercised on every bridge deposit and withdrawal. The CHANGES.md already records prior fixes for "subtraction underflow in the xcc module" and "balance and gas overflows in the xcc module," confirming that this class of bug has been found and fixed in adjacent code, making residual instances in `NEP141Wei` itself realistic. [7](#0-6) 

---

### Recommendation

Replace the plain `Sub`, `SubAssign`, and `Add` operator implementations on `NEP141Wei` and `Fee` with implementations that use `checked_sub` / `checked_add` and either return `Option`/`Result` or explicitly `saturating_*` / `wrapping_*` variants, depending on the intended semantics. Alternatively, remove the plain operator `impl` blocks entirely and force all call sites to use the already-present `checked_sub` / `checked_add` methods, making overflow handling explicit at every call site. [1](#0-0) 

---

### Proof of Concept

1. `overflow-checks = true` is set in `Cargo.toml` for both `[profile.release]` and `[profile.dev]`.
2. `NEP141Wei::sub` performs `self.0 - rhs.0` where both operands are `u128`.
3. Construct any scenario where a connector balance operation subtracts a larger `NEP141Wei` from a smaller one using the `-` operator (e.g., a race condition or a crafted sequence of `ft_transfer_call` / `ft_resolve_transfer` callbacks that leaves the stored balance smaller than the amount being deducted).
4. With `overflow-checks = true`, the Rust runtime will call `core::panicking::panic` on the underflow, which under `panic = "abort"` (also set in both profiles) immediately aborts the WASM execution.
5. The NEAR runtime rolls back the transaction, leaving the user's bridged funds inaccessible. [8](#0-7) [9](#0-8)
