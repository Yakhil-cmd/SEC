### Title
Floor Division in `native_per_pubdata` Calculation Causes Systematic Underpayment for Pubdata Native Resources - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` — the number of native resource units charged per byte of pubdata — is computed using truncating (floor) division, while the analogous `native_per_gas` uses ceiling division. This asymmetry causes every transaction that generates pubdata to systematically underpay for its native resource cost, with the magnitude of underpayment scaling with the number of pubdata bytes written.

---

### Finding Description

In `validation_impl.rs`, the two key resource-rate conversions are computed as:

```rust
// native_per_gas: ceiling division — rounds UP (protocol-favorable)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata: floor/truncating division — rounds DOWN (user-favorable)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [1](#0-0) 

`native_per_pubdata` is then used directly to charge native resources for every pubdata byte consumed:

```rust
let native = current_pubdata_spent.checked_mul(native_per_pubdata)...
``` [2](#0-1) 

The total native charged for pubdata is therefore `pubdata_bytes × floor(pubdata_price / native_price)`, whereas the economically correct amount is `pubdata_bytes × ceil(pubdata_price / native_price)`. The per-byte shortfall is at most `native_price − 1` native units, but it accumulates linearly with pubdata volume.

A secondary rounding issue exists in `compute_gas_refund`: the `deltaGas` adjustment also uses floor division:

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [3](#0-2) 

This underestimates `deltaGas` by up to 1 gas unit, giving the user a marginally larger gas refund than warranted. The documentation confirms the intended formula is `deltaGas := (nativeUsed / nativePerGas) − gasUsed` without specifying rounding direction. [4](#0-3) 

The same floor-division pattern appears in the external API helper, confirming it is not an isolated typo:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [5](#0-4) 

---

### Impact Explanation

**Resource accounting bug / public funds-loss path.**

Every transaction that writes storage (or otherwise generates pubdata) pays fewer native resources than the true cost ratio requires. The shortfall per pubdata byte is `pubdata_price mod native_price` native units. When `pubdata_price` is just below a multiple of `native_price` (e.g., `pubdata_price = 19`, `native_price = 10`), `native_per_pubdata = 1` instead of `2` — a 50% undercharge per byte. For a transaction writing 1,000 storage slots (≈32,000 pubdata bytes), the underpayment is 32,000 native units, which translates directly to operator fee loss. The operator/sequencer absorbs the fractional cost that users do not pay.

---

### Likelihood Explanation

**High.** The condition `pubdata_price % native_price != 0` is the normal operating state; it holds for virtually all real block configurations. Any unprivileged user submitting a transaction that writes to storage (EVM `SSTORE`, contract deployment, token transfer) triggers this path. No special privileges, oracle manipulation, or governance access are required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (rounds down — user-favorable):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...

// After (rounds up — protocol-favorable):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...
```

For `deltaGas` in `compute_gas_refund`, use ceiling division so that the gas equivalent of native consumption is never underestimated:

```rust
// Before:
(native_used / native_per_gas) as i64 - (gas_used as i64)

// After:
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

### Proof of Concept

**Setup:** `pubdata_price = 19`, `native_price = 10`, `gas_price = 10`, `native_price = 10` → `native_per_gas = 1`.

**Current behavior:**
- `native_per_pubdata = floor(19 / 10) = 1`
- Transaction writes 100 storage slots → ~3,200 pubdata bytes
- Native charged for pubdata: `3,200 × 1 = 3,200`

**Correct behavior:**
- `native_per_pubdata = ceil(19 / 10) = 2`
- Native charged for pubdata: `3,200 × 2 = 6,400`

**Shortfall:** 3,200 native units per such transaction. At scale (many transactions per block), the operator consistently receives less fee than the pubdata cost warrants. An attacker maximizing pubdata output (e.g., spamming `SSTORE` with a high `gas_per_pubdata` limit) amplifies this shortfall per block.
