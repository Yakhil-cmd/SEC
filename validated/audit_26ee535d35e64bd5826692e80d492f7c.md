### Title
ETH Permanently Frozen When `ExitToNear` Transfer Fails Without `error_refund` Feature - (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is disabled, a user's ETH is permanently and irrecoverably lost if the NEAR-side `ft_transfer` promise fails during an `ExitToNear` precompile call. The ETH is deducted from the user's EVM balance at EVM execution time, but no refund callback is registered, leaving the funds with no recovery path.

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` constructs a `ExitToNearPrecompileCallbackArgs` struct that conditionally includes refund information based on a compile-time feature flag:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the ETH base-token exit path, `transfer_near_args` is always `None`. This means that without the `error_refund` feature, `callback_args` equals the default value `{ refund: None, transfer_near: None }`, triggering the branch that creates a bare promise with **no callback attached**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)  // No callback registered
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

When the `ft_transfer` promise to the eth-connector fails (e.g., connector paused, insufficient NEP-141 balance), there is no `exit_to_near_precompile_callback` invocation to restore the ETH. Even if a callback were attached, the callback handler explicitly does nothing when `args.refund` is `None` and the promise failed:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // No refund, funds are lost
};
``` [3](#0-2) 

The `refund_on_error` function, which would restore the ETH balance, is only reachable when `args.refund` is `Some(...)`: [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls the `ExitToNear` precompile to bridge ETH out of Aurora, the ETH is immediately deducted from their EVM balance during EVM execution. The NEAR-side `ft_transfer` is an asynchronous promise. If that promise fails and the `error_refund` feature is not compiled in, the ETH is gone: it is no longer in the user's EVM balance, it never arrived at the NEAR destination, and there is no callback or recovery mechanism. The funds are permanently frozen.

The test suite explicitly acknowledges this behavior:

```rust
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

### Likelihood Explanation

The `ft_transfer` to the eth-connector can fail under realistic conditions reachable by any user:

1. The eth-connector's `engine_withdraw` feature is paused (demonstrated in `test_withdraw_from_near_pausability`).
2. The eth-connector NEP-141 balance for the Aurora engine account is insufficient.
3. The eth-connector contract is misconfigured or temporarily unavailable.

Any user who calls `ExitToNear` during such a window loses their ETH permanently. The attacker-controlled entry path is simply submitting an EVM transaction that calls the `ExitToNear` precompile address with ETH value attached — a standard user action. [6](#0-5) 

### Recommendation

The `error_refund` feature should be unconditionally enabled in production builds, or the refund logic should be made unconditional (not gated behind a feature flag). At minimum, the `ExitToNear` precompile should always register a callback when ETH is deducted, so that a failed NEAR-side transfer always triggers a refund of the user's EVM balance. The conditional compilation of `refund: None` should be removed.

### Proof of Concept

1. User submits an EVM transaction calling the `ExitToNear` precompile with `ETH_EXIT_AMOUNT` wei.
2. The EVM deducts `ETH_EXIT_AMOUNT` from the user's EVM balance.
3. A bare `ft_transfer` promise is created to the eth-connector (no callback, because `error_refund` is disabled and `callback_args == default()`).
4. The eth-connector's `engine_withdraw` feature is paused (or any other failure condition), causing the `ft_transfer` to fail.
5. No callback fires. The user's EVM balance remains reduced by `ETH_EXIT_AMOUNT`. The NEAR destination receives nothing.
6. The ETH is permanently lost with no recovery path.

This is confirmed by the existing test `test_exit_to_near_eth_refund` which explicitly asserts `expected_balance = INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT` when the feature is disabled. [7](#0-6)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }

#[tokio::test]
async fn test_withdraw_from_near_pausability() -> anyhow::Result<()> {
    let contract = TestContract::new_with_owner("owner").await?;
    let user_acc = contract
        .create_sub_account(DEPOSITED_RECIPIENT_NAME)
        .await?;
    let res = contract
        .deposit_eth_to_near(user_acc.id(), DEPOSITED_AMOUNT.into())
        .await?;
    assert!(res.is_success(), "{res:#?}");
    let res = contract
        .deposit_eth_to_near(
            contract.owner.as_ref().unwrap().id(),
            DEPOSITED_AMOUNT.into(),
        )
        .await?;
    assert!(res.is_success(), "{res:#?}");

    let pause_args = json!({"key": "engine_withdraw"});

    let withdraw_amount = NEP141Wei::new(100);
    // 1st withdraw - should succeed
    let res = user_acc
        .call(contract.engine_contract.id(), "withdraw")
        .args_borsh((*RECIPIENT_ADDRESS, withdraw_amount))
        .max_gas()
        .deposit(ONE_YOCTO)
        .transact()
        .await?;
