Audit Report

## Title
Unchecked Application-Level Return Value in SNS Generic Nervous System Function Execution Marks Failed Proposals as Executed - (File: rs/sns/governance/src/canister_control.rs)

## Summary
`perform_execute_generic_nervous_system_function_call` dispatches a canister call for an `ExecuteGenericNervousSystemFunction` proposal and treats any transport-level `Ok` reply as a successful execution, discarding the reply bytes entirely. A target canister method that returns an encoded `Err(...)` value instead of trapping will cause the SNS proposal to be permanently marked as executed even though the intended action had no effect. The proposal is consumed and cannot be retried.

## Finding Description
In `rs/sns/governance/src/canister_control.rs`, `perform_execute_generic_nervous_system_function_call` (lines 278–315) calls `env.call_canister` and matches on the result. The `Ok(_reply)` arm at line 302 discards the reply bytes and returns `Ok(())`, with a TODO comment explicitly acknowledging that application-level errors in the reply are not detected.

The validator path `perform_execute_generic_nervous_system_function_validate_and_render_call` (lines 262–274) applies the correct discipline: it decodes the reply as `Result<String, String>` and propagates `Err` variants as failures. The execution path does not.

The `Ok(())` propagates through `perform_execute_generic_nervous_system_function` (governance.rs lines 2531–2555) → `perform_action` (governance.rs line 2172–2175) → `set_proposal_execution_status` (governance.rs lines 1716–1731), which sets `proposal.executed_timestamp_seconds = self.env.now()` on any `Ok(_)` result, permanently marking the proposal as executed.

Existing guards are insufficient: the only guard is the transport-level `Err` arm (lines 297–300), which catches IC-level rejections/traps but not application-level `Err(...)` returns encoded in the reply bytes.

## Impact Explanation
This is a significant SNS governance integrity issue. An `ExecuteGenericNervousSystemFunction` proposal targeting a canister method that returns `Result<_, _>` (a standard Candid pattern) and returns `Err(...)` under any condition — precondition failure, transient state, misconfiguration — will be permanently marked as `executed_timestamp_seconds > 0`. The proposal is consumed and cannot be re-executed. SNS governance participants and monitoring tools observe the proposal as successfully executed, creating a false belief that the system state was updated. This matches the allowed impact: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm**, as it silently corrupts the governance execution record and prevents corrective re-execution of failed actions (e.g., treasury operations, parameter changes, dapp configuration updates).

## Likelihood Explanation
The target canister and method are registered by prior governance action, so the registered canisters are legitimate SNS-governed canisters. Many real-world canister methods return `Result<_, _>` and may return `Err(...)` due to precondition failures or transient state rather than trapping. Any adopted `ExecuteGenericNervousSystemFunction` proposal whose target method returns an encoded error triggers the bug. No privileged access beyond normal SNS governance participation is required; the bug fires during ordinary legitimate proposal execution.

## Recommendation
**Short term:** Document that any reply from the target canister — including encoded application-level errors — is treated as a successful execution. SNS operators should ensure target canister methods trap/panic rather than return `Err(...)` to signal failure.

**Long term:** Decode the reply bytes before marking the proposal as executed. The code already contains the exact approach in the TODO comment:
```rust
Ok(reply) => {
    match candid::Decode!(&reply, Result<String, String>) {
        Ok(Err(e)) => Err(GovernanceError::new_with_message(ErrorType::External, e)),
        _ => Ok(()),
    }
}
```
Establish and enforce a convention (e.g., requiring target methods to return `Result<_, String>`) at proposal registration time, mirroring the discipline already applied in the validator path.

## Proof of Concept
1. Deploy a canister with a method `fn execute(payload: Vec<u8>) -> Result<(), String>` that returns `Err("precondition failed".to_string())` when a specific condition is active.
2. Register it as a `NervousSystemFunction` via SNS governance.
3. Activate the error condition on the target canister.
4. Submit and adopt an `ExecuteGenericNervousSystemFunction` proposal targeting that method.
5. Observe that `perform_execute_generic_nervous_system_function_call` receives `Ok(encoded_err_bytes)` from `env.call_canister`.
6. The `Ok(_reply)` arm at line 302 discards the bytes and returns `Ok(())`.
7. `set_proposal_execution_status` sets `proposal.executed_timestamp_seconds = self.env.now()`.
8. The proposal is permanently marked as executed; the intended action had no effect; no retry is possible.

This is directly reproducible as a PocketIC integration test or unit test using a mock `Environment` that returns `Ok(candid::Encode!(&Err::<(), String>("precondition failed".to_string())).unwrap())` from `call_canister`.
