### Title
Attacker Can Permanently Lose Legitimate ckETH Deposits by Flooding a Block with Zero-Value Deposits — (`rs/ethereum/cketh/minter/src/deposit.rs`, `rs/ethereum/cketh/minter/EthDepositHelper.sol`)

---

### Summary

An unprivileged Ethereum address can permanently cause the ckETH minter to skip any block — and thus permanently lose all legitimate deposits in that block — by flooding it with zero-value calls to the helper contract's `deposit()` function. The helper contract imposes no minimum `msg.value`, so each call costs only gas (~3,300 gas/event). A batch contract can emit ~3,500 matching `ReceivedEth` events in a single transaction (~11.5M gas, ~$350 at 10 gwei / $3,000 ETH), pushing the `eth_getLogs` JSON response over the IC's 2 MB HTTP outcall limit. When `scrape_block_range` binary-searches down to a singleton `[N, N]` and the response is still too large, it unconditionally emits `SkippedBlockForContract` and advances `last_scraped_block_number` past that block — with no retry and no recovery path.

---

### Finding Description

**Root cause 1 — helper contract accepts zero-value deposits:** [1](#0-0) 

`deposit()` is `public payable` with no `require(msg.value > 0)`. A call with `msg.value == 0` succeeds, transfers 0 ETH to the minter address, and emits a fully valid `ReceivedEth(from, 0, principal)` event. This event carries the exact topic (`0x257e057bb61920d8d0ed2cb7b720ac7f9c513cd1110bc9fa543079154f45f435`) that the minter's `eth_getLogs` filter matches. [2](#0-1) 

**Root cause 2 — singleton-block skip is unconditional and permanent:** [3](#0-2) 

When `from_block == to_block` and `is_response_too_large` is true, the minter emits `SkippedBlockForContract` and calls `update_last_scraped_block_number(s, to_block)`. There is no retry, no governance hook, and no flag that would cause the block to be re-examined. The block is permanently skipped.

**Attack mechanics:**

The `eth_getLogs` filter is scoped to the helper contract address and the `ReceivedEth` topic: [4](#0-3) [5](#0-4) 

An attacker deploys a batch contract that calls `CkEthDeposit.deposit{value: 0}(principal)` approximately 3,500 times in a single transaction. Each call emits one `ReceivedEth` log (~600 bytes in JSON). At ~3,300 gas per call, 3,500 calls consume ~11.55 M gas — well within Ethereum's ~30 M block gas limit. The resulting `eth_getLogs` JSON response exceeds 2 MB (confirmed by the existing test: "around 600 bytes per log, we need at least 3334 logs to reach the 2MB limit"): [6](#0-5) 

The minter then binary-searches: `[N, N+k]` → too large → halved → … → `[N, N]` → still too large → `SkippedBlockForContract(N)`. Any legitimate deposit in block N is permanently lost.

**No infinite loop:** The question's "infinite loop" framing is inaccurate — the `while` loop terminates correctly. The actual harm is the unconditional permanent skip.

---

### Impact Explanation

- Any ETH deposited to the helper contract in block N is locked forever: the minter never mints ckETH for it, and the ETH cannot be recovered from the helper contract (it was already forwarded to the minter's Ethereum address).
- The `SkippedBlockForContract` event is recorded in the audit log but there is no on-chain or canister-level mechanism to re-process the block without a governance upgrade that manually rewinds `last_scraped_block_number`.
- The `DepositHelperWithSubaccount.sol` contract (`depositEth()`) has the identical missing guard and is equally exploitable. [7](#0-6) 

---

### Likelihood Explanation

- **Attacker capability:** Any Ethereum EOA. No privileged access, no key material, no governance majority.
- **Cost:** ~$350 in gas per targeted block (11.55 M gas × 10 gwei × $3,000/ETH). A single batch transaction suffices.
- **Timing:** The attacker monitors the public Ethereum mempool for large deposits, then front-runs or same-block-floods the target block. Alternatively, the attacker can flood every block continuously for ~$350/block × ~7,200 blocks/day = ~$2.5 M/day — expensive but within reach of a motivated adversary targeting a high-TVL bridge.
- **No detection before damage:** The minter has no pre-skip validation that distinguishes zero-value spam from legitimate deposits.

---

### Recommendation

1. **Helper contracts:** Add `require(msg.value > 0, "zero deposit")` to `deposit()` in `EthDepositHelper.sol` and `depositEth()` / `depositErc20()` in `DepositHelperWithSubaccount.sol`. Since these contracts are immutable on-chain, a new version must be deployed and the minter upgraded to point to the new address.

2. **Minter — filter zero-value events before counting response size:** The minter already parses logs; it could pre-filter events with `value == 0` before the response-size path is triggered. However, the response size check happens at the HTTP layer before parsing, so this alone does not fix the root cause.

3. **Minter — do not permanently skip blocks without governance confirmation:** Replace the unconditional `SkippedBlockForContract` + `update_last_scraped_block_number` path with a quarantine that requires an explicit governance proposal to advance past the block, giving operators a chance to investigate.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface ICkEthDeposit {
    function deposit(bytes32 principal) external payable;
}

contract FloodBlock {
    // Call this in the same block as the victim's deposit.
    // ~3500 calls × ~3300 gas = ~11.55M gas; fits in one Ethereum block.
    function flood(address helper, bytes32 principal, uint256 count) external {
        for (uint256 i = 0; i < count; i++) {
            ICkEthDeposit(helper).deposit{value: 0}(principal);
        }
    }
}
```
