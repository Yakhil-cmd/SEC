### Title
Neuron Holder Can Irrevocably Renounce All Permissions, Permanently Locking Staked SNS Tokens — (`rs/sns/governance/src/governance.rs`)

---

### Summary

`remove_neuron_permissions` in SNS Governance contains no guard preventing a neuron holder from removing every permission from their own neuron, including `ManagePrincipals` and `Disburse`. Once all permissions are stripped, the neuron becomes permanently orphaned: no principal can disburse the staked tokens, vote, submit proposals, or restore access. The code itself acknowledges this is dangerous but imposes no enforcement.

---

### Finding Description

`remove_neuron_permissions` (the handler for `ManageNeuron::RemoveNeuronPermissions`) performs only two checks before mutating the neuron's ACL:

1. The caller holds `ManagePrincipals` (or `ManageVotingPermission` for voting-only changes).
2. The target principal actually holds the permissions being removed.

There is no check that at least one principal with at least one critical permission (`ManagePrincipals`, `Disburse`) will remain after the operation. [1](#0-0) 

The authorization check `check_principal_authorized_to_change_permissions` only verifies the caller has the right to make the change — it does not verify the post-state is safe: [2](#0-1) 

The low-level `remove_permissions_for_principal` removes the `NeuronPermission` entry entirely when all permission types are stripped, with no last-principal guard: [3](#0-2) 

The proto definition and the Rust doc both acknowledge this is dangerous but provide no enforcement: [4](#0-3) [5](#0-4) 

The integration test `test_neuron_remove_all_permissions_of_self` explicitly confirms the behavior succeeds and leaves `neuron.permissions.len() == 0`: [6](#0-5) 

---

### Impact Explanation

Once all permissions are removed from a neuron:

- **`Disburse` is permanently blocked** — no principal holds the permission to call `disburse` on the neuron, so the staked SNS tokens are locked in the governance canister's ledger subaccount forever.
- **`DisburseMaturity` / `StakeMaturity` are permanently blocked** — accumulated maturity is also inaccessible.
- **`ManagePrincipals` is gone** — no one can re-add permissions; the neuron cannot be recovered.
- **Voting power is orphaned** — the neuron continues to exist in governance state, consuming a slot in the neuron store, but can never vote, follow, or submit proposals.

The neuron is not deleted; it persists as a permanently frozen object. The staked tokens remain in the neuron's ledger subaccount but are unreachable by any principal.

---

### Likelihood Explanation

The entry path is a standard ingress call to the public `manage_neuron` endpoint of the SNS Governance canister, available to any neuron holder. No privileged access, no governance majority, and no social engineering is required. The caller only needs to hold `ManagePrincipals` on their own neuron — which every neuron claimer receives by default via `neuron_claimer_permissions`: [7](#0-6) 

The scenario is realistic for:
- A user who mistakenly calls `RemoveNeuronPermissions` with `NeuronPermissionList::all()` targeting themselves.
- A malicious third party who holds `ManagePrincipals` on a shared neuron and strips the owner's permissions.

---

### Recommendation

Add a post-removal guard in `remove_neuron_permissions` that rejects the operation if it would leave the neuron with zero principals holding `ManagePrincipals` (or at minimum, zero principals total). Concretely, after computing the would-be post-state, check:

```rust
// Reject if no principal would retain ManagePrincipals after this removal
let manage_principals_would_remain = neuron.permissions.iter().any(|p| {
    p.principal != Some(principal_id)
        && p.permission_type.contains(&(NeuronPermissionType::ManagePrincipals as i32))
}) || (principal_id != caller_being_stripped
    && remaining_permission_types.contains(&(NeuronPermissionType::ManagePrincipals as i32)));

if !manage_principals_would_remain {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Cannot remove permissions: at least one principal must retain ManagePrincipals",
    ));
}
```

Alternatively, mirror the Connext fix: review whether `RemoveNeuronPermissions` should be permitted to remove `ManagePrincipals` from the last remaining principal at all, and block that specific case unconditionally.

---

### Proof of Concept

1. User stakes tokens and claims a neuron, receiving all permissions via `neuron_claimer_permissions`.
2. User calls `manage_neuron` on the SNS Governance canister with:
   ```
   ManageNeuron {
     subaccount: <neuron_subaccount>,
     command: RemoveNeuronPermissions {
       principal_id: <user_principal>,
       permissions_to_remove: NeuronPermissionList::all(),
     }
   }
   ```
3. `remove_neuron_permissions` passes `check_principal_authorized_to_change_permissions` (caller has `ManagePrincipals`).
4. `remove_permissions_for_principal` removes the entry; `neuron.permissions` is now empty.
5. Any subsequent call to `disburse`, `add_neuron_permissions`, or any other neuron operation fails with `NotAuthorized` — permanently.

The integration test at `rs/sns/integration_tests/src/neuron.rs:2213` (`test_neuron_remove_all_permissions_of_self`) confirms step 4 succeeds and `neuron.permissions.len() == 0` is the resulting state. [8](#0-7)

    pub(crate) fn check_principal_authorized_to_change_permissions(
        &self,
        caller: &PrincipalId,
        permissions_to_change: NeuronPermissionList,
    ) -> Result<(), GovernanceError> {
        // If the permissions to change are exclusively voting related,
        // ManagePrincipals or ManageVotingPermission is sufficient.
        // Otherwise, only ManagePrincipals is sufficient.
        let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
            vec![
                NeuronPermissionType::ManagePrincipals,
                NeuronPermissionType::ManageVotingPermission,
            ]
        } else {
            vec![NeuronPermissionType::ManagePrincipals]
        };

        // The caller is authorized if they have any of the sufficient permissions
        let caller_authorized = sufficient_permissions
            .iter()
            .any(|sufficient_permission| self.is_authorized(caller, *sufficient_permission));

        if caller_authorized {
            Ok(())
        } else {
            let caller_permissions = self.permissions_for_principal(caller);
            Err(GovernanceError::new_with_message(
                ErrorType::NotAuthorized,
                format!(
                    "Caller '{caller:?}' is not authorized to modify permissions {permissions_to_change} for neuron '{}' as it does not have any of {sufficient_permissions:?}. (Caller's permissions are {caller_permissions})",
                    self.id.as_ref().expect("Neuron must have a NeuronId"),
                ),
            ))
        }
    }
