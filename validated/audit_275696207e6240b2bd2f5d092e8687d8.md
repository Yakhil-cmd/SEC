Audit Report

## Title
Ingress Pool Admission DoS via Per-Message vs. Cumulative Cycles Check Mismatch — (`rs/execution_environment/src/execution_environment.rs`, `rs/ingress_manager/src/ingress_selector.rs`)

## Summary
The ingress admission gate checks only the single-message induction cost against a canister's balance, while the proposal-time gate checks the cumulative cost of all messages from that canister in the current block. Messages that pass admission but fail the cumulative check at proposal time are never evicted from the validated pool — they remain until TTL expiry. An attacker with a minimally-funded canister can flood the validated pool, causing the HTTP endpoint to return `503 SERVICE_UNAVAILABLE` to all legitimate users for up to 5 minutes per wave.

## Finding Description
**Admission-time check** (`should_accept_ingress_message`, `rs/execution_environment/src/execution_environment.rs` L3340–3373): The code explicitly labels this a "first-pass" check and passes only the single-message `cost` to `can_withdraw_cycles_with_threshold`. Multiple concurrent submissions from the same canister each see the same unmodified state and all pass independently. [1](#0-0) 

**Pool validation** (`validate_ingress_pool_object`, `rs/ingress_manager/src/ingress_handler.rs` L167–206): When messages move from unvalidated to validated, only size, known-status, and signature are checked. No cycles check is performed. [2](#0-1) 

**Proposal-time check** (`validate_ingress`, `rs/ingress_manager/src/ingress_selector.rs` L566–584): At block-building time the check is cumulative — `*cumulative_ingress_cost + ingress_cost` — so only the first message from a canister with minimal balance passes; all subsequent messages fail with `InsufficientCycles`. [3](#0-2) 

**No eviction of failing messages** (`rs/ingress_manager/src/ingress_selector.rs` L204–207): When `validate_ingress` returns any error other than `IngressPayloadTooManyMessages`, the message is only popped from the local in-memory queue for that round via `queue.msgs.pop()`. No `RemoveFromValidated` change action is emitted, so the message remains in the validated pool. [4](#0-3) 

**Validated pool cleanup paths** (`rs/ingress_manager/src/ingress_handler.rs` L112–137): The only eviction paths for validated messages are `IngressHistoryReader` returning a non-`Unknown` status (requires execution) or explicit `messages_to_purge` entries (requires finalization). Messages that fail cycles validation at proposal time are never executed and never finalized, so neither path applies — they sit in the pool until `MAX_INGRESS_TTL` expires. [5](#0-4) 

**Pool capacity enforcement** (`rs/artifact_pool/src/ingress_pool.rs` L226–232): `exceeds_limit` counts both unvalidated and validated messages against `ingress_pool_max_count` and `ingress_pool_max_bytes` per peer. [6](#0-5) 

**HTTP endpoint load-shedding** (`rs/http_endpoints/public/src/call.rs` L229–236): Once `exceeds_threshold()` returns `true`, all new ingress submissions are rejected with `503 SERVICE_UNAVAILABLE`. [7](#0-6) 

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The attacker can deny ingress service to all users of a subnet for repeating 5-minute windows. The attack is not volumetric — it exploits a structural logic mismatch in the admission/validation pipeline. Legitimate users receive `503` for all canister calls until the attacker's zombie messages expire.

## Likelihood Explanation
The attack requires only an unprivileged user with a canister holding cycles sufficient for exactly one induction cost (proportional to message size; a few hundred bytes costs a negligible amount). No governance majority, subnet-majority, threshold corruption, or privileged access is needed. The attacker submits messages via the standard HTTPS `/api/v2/canister/{id}/call` endpoint. The attack is repeatable every `MAX_INGRESS_TTL` (~5 minutes) indefinitely. Using multiple canisters and boundary nodes amplifies the effect across all subnet nodes simultaneously.

## Recommendation
1. **Evict messages that fail cycles validation**: When `validate_ingress` returns `InsufficientCycles` during `get_ingress_payload`, emit a `RemoveFromValidated` change action so the message is purged from the validated pool rather than re-selected every round.
2. **Tighten admission-time gate**: Track per-canister cumulative ingress cost at admission time, or check whether the canister's balance exceeds the freeze threshold by more than a configurable multiple of a single message cost before admitting additional messages from the same canister.
3. **Per-canister admission rate limiting**: Limit the number of messages from a single canister that can be admitted to the pool within a TTL window, proportional to the canister's available cycles.

## Proof of Concept
1. Create canister `C` with cycles balance = `freeze_threshold + ingress_induction_cost(msg)` (sufficient for exactly one message).
2. Submit `N` identical messages targeting `C` via `/api/v2/canister/{id}/call` in rapid succession. Each invokes `should_accept_ingress_message`, which calls `can_withdraw_cycles_with_threshold(cost_of_one_message)` against the same unmodified state — all N pass.
3. All N messages enter the unvalidated pool and are moved to validated by `validate_ingress_pool_object` (no cycles check performed).
4. On the next `get_ingress_payload` call, `validate_ingress` checks cumulative cost: message 1 passes (`cumulative = cost ≤ balance − threshold`), message 2 fails (`cumulative = 2×cost > balance − threshold`), messages 3–N fail similarly. Each failing message is popped from the local queue (`queue.msgs.pop()`) but remains in the validated pool.
5. Repeat with additional minimally-funded canisters until `ingress_pool.exceeds_threshold()` returns `true`.
6. All subsequent legitimate ingress submissions receive `503 SERVICE_UNAVAILABLE` until the attacker's messages expire after `MAX_INGRESS_TTL`.
7. A deterministic integration test can reproduce this by: constructing an `IngressManager` with a canister at minimal balance, inserting N messages into the pool, calling `get_ingress_payload`, and asserting that messages 2–N remain in the validated pool after the call.

        &self,

        settings: &IngressMessageSettings,
        ingress_message_status: impl Fn(&MessageId) -> IngressStatus,
        consensus_time: Time,
        registry_version: RegistryVersion,
    ) -> Result<(), IngressMessageValidationError> {
        // If the message is too large, consider the ingress message invalid
        let size = ingress_object.count_bytes();
        if size > settings.max_ingress_bytes_per_message {
            return Err(IngressMessageValidationError::IngressMessageTooLarge {
                max: settings.max_ingress_bytes_per_message,
                actual: size,
            });
        }

        match ingress_message_status(&ingress_object.message_id) {
            IngressStatus::Known { .. } => {
                return Err(IngressMessageValidationError::IngressMessageAlreadyKnown);
            }
            IngressStatus::Unknown => {}
        }

        // Check signatures, remove from unvalidated if they can't be
        // verified, add to validated otherwise.

            ingress_object.signed_ingress.as_ref(),
            consensus_time,
            &self.registry_root_of_trust_provider(registry_version),
        ) {
            return Err(IngressMessageValidationError::InvalidRequest(err));
        }

        Ok(())
