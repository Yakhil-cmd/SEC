Audit Report

## Title
Unprivileged Caller Can Exhaust the `update_balance` Concurrent-Request Guard, Blocking All ckBTC Deposits — (`rs/bitcoin/ckbtc/minter/src/guard.rs`, `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
The `update_balance` endpoint allows any caller to specify an arbitrary `owner` principal, and the concurrent-request guard is keyed on the resolved account rather than the actual caller. Because the global guard set is capped at `MAX_CONCURRENT = 100` and the guard is held across an inter-canister `get_utxos` call, an unprivileged attacker can fill all 100 slots with attacker-chosen accounts, blocking every legitimate ckBTC deposit until the attacker stops re-submitting.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` at lines 164–168, the target account is constructed from the caller-supplied `args.owner`:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

The only restriction before this point (lines 149–151) is that the resolved owner must not equal the minter's own canister ID — there is no restriction preventing a caller from specifying any other principal as `owner`.

In `rs/bitcoin/ckbtc/minter/src/guard.rs` at lines 51–54, `Guard::new` rejects new requests once the global `BTreeSet<Account>` reaches `MAX_CONCURRENT = 100`:

```rust
if accounts.len() >= MAX_CONCURRENT {
    return Err(GuardError::TooManyConcurrentRequests);
}
accounts.insert(account);
```

The guard `_guard` is held across the async `get_utxos(...)` call at lines 175–183, which is an inter-canister call to the Bitcoin canister spanning multiple IC execution rounds. An attacker submits 100 ingress messages each with a distinct `owner = Some(principal_i)`. Each call:
1. Resolves `caller_account` to the attacker-chosen principal.
2. Acquires a guard slot in `update_balance_accounts`.
3. Suspends at `get_utxos(...).await` while awaiting the Bitcoin canister response.

During this window `update_balance_accounts.len() == 100`, so every legitimate `update_balance` call returns `GuardError::TooManyConcurrentRequests`, surfaced as `UpdateBalanceError::TemporarilyUnavailable("too many concurrent requests")`. The attacker re-submits as guards are released, sustaining the DoS indefinitely.

The existing comment at line 219 acknowledges a related memory-exhaustion risk from the `owner` field but does not address the concurrent-guard exhaustion vector.

## Impact Explanation
All ckBTC deposits are blocked for the duration of the attack. Users who have sent BTC to their minter-derived address cannot call `update_balance` to mint ckBTC. This constitutes an application/platform-level DoS against a core ck-token financial integration (ckBTC), with concrete and direct harm to users. This matches the allowed High impact: *"Application/platform-level DoS … not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token … security impact with concrete user or protocol harm."*

## Likelihood Explanation
The attack requires no ckBTC, no BTC, and no privileged role — only the ability to send ingress messages to the ckBTC minter, which is open to any principal. The cost is 100 concurrent ingress calls at standard IC ingress fees. The Bitcoin canister's `get_utxos` response latency (typically several seconds per round-trip) gives the attacker a comfortable window to maintain the guard set at capacity. The attack is trivially repeatable and can be sustained indefinitely.

## Recommendation
1. **Restrict `owner` to the caller's own principal.** Remove the ability for a caller to specify an arbitrary `owner`, or gate it behind an explicit allowlist.
2. **Alternatively, key the guard on the caller principal, not the resolved account.** This prevents one caller from occupying slots for accounts they do not control.
3. **Implement a per-caller concurrent request limit** (e.g., 5 per caller) so no single attacker can exhaust the global pool.

## Proof of Concept
```rust
// Attacker submits 100 ingress messages to the ckBTC minter:
for i in 0..100 {
    update_balance(UpdateBalanceArgs {
        owner: Some(principal_from_u64(i)),  // 100 distinct victim accounts
        subaccount: None,
    });
    // Each call acquires a guard slot and suspends at get_utxos().await
}

// Now update_balance_accounts.len() == 100.
// Any legitimate user calling update_balance() receives:
//   Err(UpdateBalanceError::TemporarilyUnavailable("too many concurrent requests"))
// Attacker re-submits as guards expire, sustaining the DoS at negligible cost.
```

A deterministic integration test using PocketIC can verify this by: (1) initializing the minter, (2) submitting 100 concurrent `update_balance` calls with distinct `owner` principals via `call_async`, (3) confirming a 101st call from a legitimate user returns `TemporarilyUnavailable`.
