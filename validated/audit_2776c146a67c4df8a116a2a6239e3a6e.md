### Title
CALLCODE with Non-Zero Value in Static Context Is Not Rejected — EVM Semantic Mismatch (`evm_interpreter/src/instructions/host.rs`, `zk_ee/src/system/execution_environment/environment_state.rs`)

---

### Summary

The EVM interpreter in ZKsync OS does not reject a `CALLCODE` instruction that carries a non-zero value when executing inside a static context. The EVM specification (EIP-214) requires that any state-modifying operation — including a value-bearing `CALLCODE` — must revert inside a static frame. ZKsync OS instead permits the call to proceed, committing a self-transfer and returning success. This is a direct EVM semantic mismatch analogous to the Vyper bug: just as Vyper allowed a `payable` interface function to be implemented as `nonpayable` without error, ZKsync OS allows a value-bearing call variant that should be rejected in a restricted context to silently succeed.

---

### Finding Description

**Root cause 1 — missing `is_static` guard for `CALLCODE` value in `call_impl`**

In `evm_interpreter/src/instructions/host.rs`, the `call_impl` function pops the value argument for `CALLCODE` without checking whether the current frame is static:

```rust
CallScheme::CallCode => {
    let value = self.stack.pop_1()?;
    *value          // ← no is_static guard
}
CallScheme::Call => {
    let value = self.stack.pop_1()?;
    if self.is_static && *value != U256::ZERO {
        return Err(EvmError::CallNotAllowedInsideStatic.into());
    }
    *value
}
```

`CALL` correctly rejects a non-zero value in a static frame; `CALLCODE` does not. [1](#0-0) 

**Root cause 2 — `is_transfer_allowed` explicitly permits `EVMCallcodeStatic`**

In `zk_ee/src/system/execution_environment/environment_state.rs`, the transfer-permission predicate lists `EVMCallcodeStatic` as an allowed modifier:

```rust
pub fn is_transfer_allowed(&self) -> bool {
    self.modifier == CallModifier::NoModifier
    || self.modifier == CallModifier::Constructor
    || self.modifier == CallModifier::ZKVMSystem
    || self.modifier == CallModifier::EVMCallcode
    // Positive-value callcode calls are allowed in static context,
    // as the transfer is a self-transfer.
    || self.modifier == CallModifier::EVMCallcodeStatic
}
```

The comment rationalises this as a "self-transfer" (the transfer target for `EVMCallcodeStatic` is `call_request.caller`, not `callee`). However, the EVM specification does not carve out self-transfers from the static-context prohibition; any value-bearing call in a static frame must revert. [2](#0-1) 

**Execution path**

When `CALLCODE` with value is issued inside a static frame:

1. `call_impl` sets `call_modifier = CallModifier::EVMCallcodeStatic` and emits an `ExternalCall` preemption without reverting.
2. The bootloader runner calls `perform_transfer_if_required`; `is_transfer_allowed()` returns `true` for `EVMCallcodeStatic`, so the self-transfer is executed.
3. The callee frame executes normally and returns success.

On Ethereum the same sequence reverts at step 1 because `CALLCODE` with value in a static context is a prohibited state-modifying operation. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Any contract that issues `CALLCODE` with a non-zero value inside a `STATICCALL` chain will:

- **Revert on Ethereum** (correct EVM behaviour).
- **Succeed on ZKsync OS** (incorrect behaviour).

Contracts that rely on this revert as a guard — for example, to detect whether they are being called in a read-only context, or to enforce that a particular code path cannot execute during a view call — will have that invariant silently bypassed on ZKsync OS. This is an EVM-equivalence violation: the state-transition function produces a different outcome than Ethereum for the same bytecode and inputs, which can break cross-chain logic, off-chain simulations, and any contract that assumes EVM-equivalent static-context enforcement.

---

### Likelihood Explanation

`CALLCODE` is a deprecated opcode, so the population of contracts that use it is small. However:

- Legacy contracts compiled with older Solidity versions do use `CALLCODE`.
- An attacker who controls a callee contract can deliberately craft bytecode that uses `CALLCODE` with value inside a static frame to bypass a caller's read-only assumption.
- The entry path is fully unprivileged: any user transaction can deploy such a contract and trigger the divergent behaviour.

---

### Recommendation

1. In `call_impl` (`evm_interpreter/src/instructions/host.rs`), add the same `is_static` guard for `CallScheme::CallCode` that already exists for `CallScheme::Call`:

```rust
CallScheme::CallCode => {
    let value = self.stack.pop_1()?;
    if self.is_static && *value != U256::ZERO {
        return Err(EvmError::CallNotAllowedInsideStatic.into());
    }
    *value
}
```

2. Remove `CallModifier::EVMCallcodeStatic` from the `is_transfer_allowed` predicate in `zk_ee/src/system/execution_environment/environment_state.rs`, or add a guard that rejects the transfer when the value is non-zero and the context is static.

---

### Proof of Concept

```
// Attacker deploys Contract B with this bytecode:
//   PUSH1 1        ; value = 1 wei
//   PUSH20 <addr>  ; target (any address)
//   PUSH1 0        ; argsOffset
//   PUSH1 0        ; argsLength
//   PUSH1 0        ; retOffset
//   PUSH1 0        ; retLength
//   CALLCODE       ; CALLCODE with value=1 inside static frame
//   ...

// Contract A calls Contract B via STATICCALL.
// On Ethereum:  CALLCODE reverts → STATICCALL returns 0 (failure).
// On ZKsync OS: CALLCODE succeeds → STATICCALL returns 1 (success).
// The divergence is observable by Contract A and any off-chain observer.
```

The divergence is triggered by any unprivileged user who deploys Contract B and causes Contract A to `STATICCALL` into it. No privileged access, oracle manipulation, or governance action is required.
