### Title
BLS12-381 EIP-2537 Precompiles Charge Zero Native Resource, Bypassing Proving-Cost Accounting - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

All seven BLS12-381 (EIP-2537) precompiles hardcode `cost_native = 0`, meaning they consume **zero native resource** regardless of input size or computational complexity. The native resource is ZKsync OS's mechanism for bounding the RISC-V proving cost of a transaction. In contrast, the BN254 pairing precompile correctly charges per-pair native costs. An unprivileged EVM caller can craft transactions that repeatedly invoke BLS12-381 precompiles, consuming significant proving resources without depleting the native budget, breaking the core resource accounting invariant.

---

### Finding Description

ZKsync OS implements a **double resource accounting model**: EVM gas (ergs) and native resource (proving cost). The native resource limit is derived from `gasLimit × nativePerGas`, ensuring users pay for the RISC-V proving cost of their transactions. If a transaction runs out of native resources, it is reverted.

Every BLS12-381 precompile hardcodes `cost_native = 0`:

**`addition.rs`** — G1Add and G2Add:
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs,
    <R::Native as Computational>::from_computational(cost_native)))?;
```

**`msm.rs`** — G1MSM and G2MSM:
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs,
    <R::Native as Computational>::from_computational(cost_native)))?;
```

**`pairing.rs`** — PairingCheck:
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs,
    <R::Native as Computational>::from_computational(cost_native)))?;
```

**`mappings.rs`** — MapFpToG1 and MapFp2ToG2:
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs,
    <R::Native as Computational>::from_computational(cost_native)))?;
```

In direct contrast, the BN254 pairing precompile in `basic_system/src/system_functions/bn254_pairing_check.rs` correctly charges:
```rust
let native_cost = (num_pairs as u64) * BN254_PAIRING_PER_PAIR_NATIVE_COST
    + BN254_PAIRING_BASE_NATIVE_COST;
resources.charge(&R::from_ergs_and_native(ergs_cost,
    <R::Native as Computational>::from_computational(native_cost)))?;
```

BLS12-381 operates over a larger field than BN254 and is computationally more expensive per operation, yet charges **zero** native resource while BN254 charges a non-zero, input-proportional native cost. This is an inconsistency that constitutes a resource accounting bug: the native resource counter is not decremented by BLS12-381 calls, so the native budget does not accurately reflect the proving cost of the transaction.

The ZK-flow (`proof_running_system`) enforces a real native limit derived from `gasLimit × nativePerGas` via `create_resources_for_tx`. These resources flow into the EVM interpreter and are passed to precompile calls. Because BLS12-381 charges 0 native, the native counter is never decremented by these calls, allowing a transaction to perform arbitrarily many BLS12-381 operations within its gas limit while consuming far more RISC-V proving cycles than the native budget accounts for.

---

### Impact Explanation

An attacker deploys a contract that calls the BLS12-381 pairing check or MSM precompile in a tight loop. Each call correctly deducts EVM gas (bounding the number of calls per transaction), but charges **zero** native resource. The native budget — which is supposed to bound the RISC-V proving cost — is never depleted by these calls.

The result is a **resource accounting divergence**: the forward system processes the transaction normally (native budget appears sufficient), but the proving system must execute the full BLS12-381 computation for every call, consuming RISC-V cycles that were never paid for. This breaks the invariant stated in the documentation: *"the native resource limit is derived from the transaction's gas limit and fee"* to ensure users pay for proving cost.

Concrete impacts:
1. **Proving system overload / DoS**: A block filled with transactions making maximum-gas BLS12-381 calls forces the prover to do far more work than the native budgets of those transactions imply.
2. **Forward/proving divergence**: The forward sequencer sees a transaction as within budget; the prover may exhaust resources or produce incorrect accounting.
3. **Underpriced proving**: Users pay only EVM gas for BLS12-381 work; the protocol absorbs the excess proving cost.

---

### Likelihood Explanation

The entry path is trivially reachable: any EVM contract can `CALL` address `0x0f` (BLS12-381 pairing check) or `0x0c`/`0x0e` (G1/G2 MSM). No privileged role, governance action, or external oracle is required. A single malicious contract calling the pairing precompile in a loop is sufficient to trigger the issue. The `eip-2537` feature flag enables these precompiles in production builds, as confirmed by the test index files.

---

### Recommendation

Add proportional native resource charges to all BLS12-381 precompiles, mirroring the BN254 pattern. For example, for the pairing check:

```rust
// Before (broken):
let cost_native = 0;

// After (fixed):
let cost_native = BLS12_381_PAIRING_BASE_NATIVE_COST
    + (num_pairs as u64) * BLS12_381_PAIRING_PER_PAIR_NATIVE_COST;
```

Similarly, G1/G2 MSM should charge native proportional to `num_pairs`, and G1/G2 Add and field-mapping operations should charge a fixed per-call native cost. The constants should be calibrated against actual RISC-V cycle counts for each operation.

### Proof of Concept

1. Deploy an EVM contract:
   ```solidity
   contract BLSBomb {
       address constant PAIRING = address(0x0f);
       function attack(bytes calldata input) external {
           for (uint i = 0; i < 100; i++) {
               PAIRING.call(input); // charges EVM gas, 0 native
           }
       }
   }
   ```
2. Call `attack` with a valid BLS12-381 pairing input (e.g., two infinity points). Each iteration charges `37700 + 32600 = 70300` EVM gas but **0 native resource**.
3. With a gas limit of ~7,100,000, the loop runs ~100 times. The native budget is untouched throughout.
4. The proving system must execute 100 full BLS12-381 multi-pairings — work that was never paid for in native resource — while the forward system reports the transaction as within budget. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
