### Title
SNS Treasury Manager `DepositRequest` Lacks Slippage Protection, Enabling Front-Running of DAO Treasury Deposits into DEX Pools - (File: rs/sns/treasury_manager/treasury_manager.did)

### Summary

The `TreasuryManager` interface's `DepositRequest` type contains no slippage-protection field (no `min_amount_out`, `max_slippage`, or `price_limit`). When an SNS governance proposal executes a treasury deposit into a DEX liquidity pool, an attacker who observes the public proposal can manipulate the DEX pool price before the deposit executes, causing the SNS treasury to receive far fewer LP tokens than expected. The codebase itself acknowledges this risk in two separate locations but provides no structural mitigation.

### Finding Description

The `DepositRequest` struct defined in the Treasury Manager interface carries only `allowances`:

```
// rs/sns/treasury_manager/treasury_manager.did  lines 84-86
type DepositRequest = record {
  allowances : vec Allowance;
};
```

```rust
// rs/sns/treasury_manager/src/lib.rs  lines 284-287
pub struct DepositRequest {
    pub allowances: Vec<Allowance>,
}
```

There is no field for a minimum LP-token output, a maximum acceptable price ratio, or any other slippage bound. Any Treasury Manager implementation that deposits into a DEX (e.g., KongSwap) must therefore call the DEX with no price floor, exactly mirroring the `amountOutMinimum: 0` pattern in the original report.

The execution path is:

1. `execute_treasury_manager_deposit` in `rs/sns/governance/src/extensions.rs` (lines 1566–1578) first calls `approve_treasury_manager`, which issues ICRC-2 approvals with a **1-hour expiry** to the Treasury Manager canister.
2. It then calls `deposit` on the Treasury Manager canister.
3. The Treasury Manager pulls the approved tokens via `icrc2_transfer_from` and deposits them into the DEX — with no slippage bound because `DepositRequest` provides no way to express one.

The codebase acknowledges this structural gap in two places:

- `rs/sns/treasury_manager/treasury_manager.did` lines 35–40 states: *"Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."*
- `rs/sns/governance/src/proposal.rs` lines 1540–1545 warns: *"Some Decentralized Exchanges lack slippage protection during deposits … making them vulnerable to front-running or sandwich attacks."*

Neither location enforces a mitigation; the warning is informational only.

### Impact Explanation

An attacker can drain a significant fraction of the SNS treasury's deposited value:

1. The attacker observes a passing `RegisterExtension` or `ExecuteExtensionOperation` (deposit) proposal — all SNS proposals are public.
2. Before the governance canister executes the deposit, the attacker submits transactions to the target DEX canister to skew the pool price (e.g., buy one side of the SNS-token/ICP pool heavily).
3. The governance canister's `deposit` call executes at the manipulated price; the SNS treasury receives far fewer LP tokens than the proposal specified.
4. The attacker reverses their position and profits from the price difference.

The SNS treasury loses real value (SNS tokens + ICP) proportional to the price impact the attacker can create. For large treasuries or low-liquidity pools, this can be substantial.

### Likelihood Explanation

- SNS governance proposals are fully public and have a multi-day voting period, giving attackers ample time to prepare.
- The 1-hour ICRC-2 approval window (`expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS)`) creates a predictable execution window.
- On the Internet Computer, there is no traditional mempool, but an attacker can submit DEX manipulation calls in rounds immediately preceding the governance execution round, achieving the same front-running effect.
- No special privilege is required; any unprivileged canister caller can interact with the DEX.

### Recommendation

1. **Add a slippage parameter to `DepositRequest`**: introduce a `min_lp_tokens_out` or `max_price_deviation_bps` field so that Treasury Manager implementations can pass a price floor to the DEX.
2. **Enforce the parameter in the governance validation step** (`validate_deposit_operation_impl` in `rs/sns/governance/src/extensions.rs`): reject proposals that do not specify a non-zero slippage bound.
3. **Use a time-weighted average price (TWAP) oracle** at execution time to verify the current pool price has not deviated beyond the proposal's stated tolerance before calling `deposit`.

### Proof of Concept

**Attacker-controlled entry path** (no privileged role required):

1. Attacker observes a public SNS governance proposal to deposit `X` SNS tokens + `Y` ICP into a KongSwap pool.
2. Proposal passes; governance canister calls `approve_treasury_manager` → ICRC-2 allowance granted to Treasury Manager with 1-hour expiry.
3. Attacker submits a large buy of SNS tokens on the KongSwap pool in the round before governance executes `deposit`, moving the price significantly.
4. Governance canister calls `deposit` on the Treasury Manager; Treasury Manager calls KongSwap with the full `allowances` amounts and no `min_lp_tokens_out` (because `DepositRequest` has no such field).
5. KongSwap executes the deposit at the manipulated price; SNS treasury receives far fewer LP tokens.
6. Attacker sells their SNS tokens back at the now-restored price, pocketing the difference.
