Audit Report

## Title
Subnet-Wide Canister HTTP Outcall Pool Exhaustible by Single Canister with No Per-Canister Quota - (`rs/execution_environment/src/execution_environment.rs`)

## Summary
The subnet-wide in-flight HTTP outcall pool (`canister_http_request_contexts`) is bounded by `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT = 3000` but enforces no per-canister sub-limit. A single canister with sufficient cycles can occupy all 3000 slots, causing every other canister on the same subnet to receive `CanisterRejectedMessage` for any `ic00::HttpRequest` call for as long as the attack is sustained. The existing test `http_request_bound_holds` already demonstrates that a single caller can fill the entire pool.

## Finding Description
In `try_add_http_context_to_replicated_state` (`rs/execution_environment/src/execution_environment.rs`, L2148–2162), the sole admission guard is a subnet-wide length check on `canister_http_request_contexts`:

```rust
if state
    .metadata
    .subnet_call_context_manager
    .canister_http_request_contexts
    .len()
    >= self.config.max_canister_http_requests_in_flight
{
    return Err(UserError::new(
        ErrorCode::CanisterRejectedMessage,
        format!(
            "max number ({}) of http requests in-flight reached.",
            self.config.max_canister_http_requests_in_flight
        ),
    ));
}
```

`canister_http_request_contexts` is a flat `BTreeMap<CallbackId, CanisterHttpRequestContext>` (`rs/replicated_state/src/metadata_state/subnet_call_context_manager.rs`, L219) with no per-sender index or quota. The constant is 3000 (`rs/config/src/execution_environment.rs`, L206). Requests time out after `CANISTER_HTTP_TIMEOUT_INTERVAL = 60s` (`rs/types/types/src/canister_http.rs`, L79). An attacker who continuously re-submits 3000 requests every 60 seconds holds the pool full indefinitely.

The `canister_guaranteed_callback_quota` / `subnet_callback_soft_limit` fairness mechanism (`rs/config/src/execution_environment.rs`, L266–275) exists for general inter-canister callbacks but is entirely absent for HTTP outcalls.

The test `http_request_bound_holds` (`rs/execution_environment/src/execution_environment/tests.rs`, L2319–2381) uses a single `caller_canister` to fill the pool to the configured maximum of 10, confirming no per-canister distinction is enforced.

## Impact Explanation
This matches the allowed High impact: **"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS."** All canisters on the targeted subnet that call `ic00::HttpRequest` are blocked for the duration of the attack. Any protocol relying on canister HTTP outcalls — oracle price feeds, cross-chain bridges, HTTPS API integrations — is completely unavailable. This is a sustained, repeatable availability failure of a core IC feature for an entire subnet.

## Likelihood Explanation
Any unprivileged canister deployed on the target subnet can execute this attack. No governance majority, threshold corruption, privileged access, or victim interaction is required. The only cost is the non-refundable base fee per request multiplied by 3000, renewed every 60 seconds. For a motivated attacker targeting a high-value subnet (e.g., one running DeFi protocols dependent on HTTP-based price oracles), the economic cost is bounded and predictable. The attack is trivially repeatable.

## Recommendation
Introduce a per-canister limit on simultaneously in-flight HTTP outcall requests. Track the count of in-flight requests per `sender` canister ID within `canister_http_request_contexts` and reject new requests from a canister that has already reached its individual quota (e.g., `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT / expected_active_canisters`). This mirrors the `canister_guaranteed_callback_quota` pattern already used for general inter-canister callbacks (`rs/config/src/execution_environment.rs`, L266–275) and would prevent any single canister from monopolizing the pool.

## Proof of Concept
1. Deploy a malicious canister `attacker` on any application subnet with HTTP outcalls enabled.
2. In a loop, have `attacker` call `ic00::HttpRequest` with a URL pointing to a slow or non-existent endpoint, attaching the minimum required cycles.
3. After 3000 such calls are in flight, any other canister on the same subnet calling `ic00::HttpRequest` receives `CanisterRejectedMessage: "max number (3000) of http requests in-flight reached."`.
4. After 60 seconds the attacker's requests time out; the attacker immediately re-submits 3000 new requests, maintaining the DoS.
5. The existing test `http_request_bound_holds` (`rs/execution_environment/src/execution_environment/tests.rs`, L2319–2381) already demonstrates this: a single `caller_canister` fills the entire pool to the configured maximum with no per-canister distinction enforced.
