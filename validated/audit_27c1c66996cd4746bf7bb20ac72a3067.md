Audit Report

## Title
Missing `ingress_bytes_hash` Length Validation in `TryFrom<pb::StrippedIngressMessage> for SignedIngressId` Enables Byzantine Peer to Stall Block Assembly — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types.rs`)

## Summary

`TryFrom<pb::StrippedIngressMessage> for SignedIngressId` accepts any `Vec<u8>` length for `ingress_bytes_hash` without enforcing the 32-byte SHA-256 invariant. A Byzantine subnet peer can broadcast a `StrippedBlockProposal` with arbitrarily-lengthed `ingress_bytes_hash` fields. Because every subsequent comparison derives a real 32-byte hash from actual ingress bytes, the malformed ID never matches, causing the assembler to stall until the bouncer fires (~3 s), at which point the proposal is marked `Unwanted` and the victim node does not vote on it. The attack is repeatable on every block proposal.

## Finding Description

**Root cause — `types.rs` line 47:** No length check before constructing the `CryptoHashOf`:

```rust
let ingress_bytes_hash = CryptoHashOf::from(CryptoHash(value.ingress_bytes_hash));
```

`CryptoHash` is a plain `struct CryptoHash(pub Vec<u8>)` newtype; `CryptoHashOf::from` is a zero-cost wrapper. Any length is accepted verbatim. [1](#0-0) 

The legitimate construction path always produces a 32-byte SHA-256 hash via `ic_types::crypto::crypto_hash`. [2](#0-1) 

**Ingestion path — `stripped.rs` lines 103–107:** `StrippedBlockProposal::try_from` iterates over all `ingress_messages` in the protobuf and calls `SignedIngressId::try_from` on each, propagating the malformed hash into the in-memory `StrippedIngressPayload` without rejection. [3](#0-2) 

**Local pool fast-path failure — `download.rs` line 101 and line 132:** Both pool-lookup guards compare `SignedIngressId::from(&ingress_message)` (always 32 bytes) against the attacker-controlled `signed_ingress_id` (malformed length). The comparison always fails, so the fast path is never taken. [4](#0-3) [5](#0-4) 

**Peer-download validation failure — `download.rs` lines 390–391:** After fetching the real ingress bytes from a peer, `parse_response` recomputes `derived_ingress_id = SignedIngressId::from(&ingress)` (32 bytes) and checks equality against the malformed `ingress_id`. This always returns `ParseResponseError::MessageIdMismatch`, so the download loop never exits successfully. [6](#0-5) 

**Infinite retry loop — `download.rs` lines 301–304:** `download_stripped_message` uses `with_max_elapsed_time(None)`, meaning it retries indefinitely with exponential backoff up to 120 s per interval. [7](#0-6) 

**Assembler stall — `assembler.rs` lines 235–242:** `assemble_message` spawns one `get_or_fetch` task per stripped message ID and waits for all to complete. With malformed IDs, none ever complete. The only exit is the `tokio::select!` bouncer arm firing when the proposal becomes `Unwanted`. [8](#0-7) 

**Bouncer refresh period — `assembler.rs` line 101:** The bouncer refresh period is 3 seconds, bounding the stall per proposal. [9](#0-8) 

The identical unvalidated pattern also exists in `rpc.rs` line 34, affecting the server-side ingress lookup path. [10](#0-9) 

## Impact Explanation

A Byzantine subnet node (below the consensus fault threshold) can send a `StrippedBlockProposal` with malformed `ingress_bytes_hash` fields for every block proposal it receives. Each targeted proposal stalls assembly on the victim node for ~3 seconds, after which the proposal is marked `Unwanted` and the victim node does not vote on it. If the Byzantine node targets every proposal, the victim node is effectively excluded from consensus voting, constituting **consensus blocking / subnet availability degradation**. This matches the allowed High impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attacker requires only subnet membership (below the fault threshold). No key material, governance majority, or external infrastructure is needed. The `ingress_bytes_hash` field is a plain protobuf `bytes` field with no schema-level length constraint, making payload crafting trivial. The attack is repeatable on every block proposal the Byzantine node receives, and each instance requires only a single malformed message.

## Recommendation

Add an explicit 32-byte length check in `TryFrom<pb::StrippedIngressMessage> for SignedIngressId` before constructing the `CryptoHashOf`:

```rust
const CRYPTO_HASH_LEN: usize = 32;
if value.ingress_bytes_hash.len() != CRYPTO_HASH_LEN {
    return Err(ProxyDecodeError::Other(format!(
        "ingress_bytes_hash must be {} bytes, got {}",
        CRYPTO_HASH_LEN,
        value.ingress_bytes_hash.len()
    )));
}
let ingress_bytes_hash = CryptoHashOf::from(CryptoHash(value.ingress_bytes_hash));
```

Apply the same guard to `GetIngressMessageInBlockRequest::try_from` in `rpc.rs` line 34, which has the identical pattern.

## Proof of Concept

```rust
#[test]
fn malformed_ingress_bytes_hash_stalls_assembly() {
    use ic_protobuf::types::v1 as pb;
    use crate::fetch_stripped_artifact::types::SignedIngressId;
    use ic_test_utilities_types::messages::SignedIngressBuilder;

    let ingress = SignedIngressBuilder::new().nonce(1).build();
    let real_id = SignedIngressId::from(&ingress);

    for bad_len in [0usize, 1, 31, 33, 1024] {
        let proto = pb::StrippedIngressMessage {
            stripped: Some(real_id.ingress_message_id.clone().into()),
            ingress_bytes_hash: vec![0xAB; bad_len],
        };
        // Currently succeeds — should return Err after the fix
        let parsed = SignedIngressId::try_from(proto).unwrap();
        // The parsed ID never equals the legitimately-computed one,
        // so every pool lookup and peer-download validation fails.
        assert_ne!(parsed, real_id,
            "bad_len={bad_len}: malformed hash must not match real hash");
    }
}
```

A full integration test can be constructed by crafting a `pb::StrippedConsensusMessage` with malformed `ingress_bytes_hash` entries, injecting it via the P2P mock transport into `FetchStrippedConsensusArtifact::assemble_message`, and asserting that the result is `AssembleResult::Unwanted` within the bouncer refresh window rather than `AssembleResult::Done`.
