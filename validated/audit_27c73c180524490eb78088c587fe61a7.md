### Title
Zero Native Resource Charge for BLS12-381 Precompiles Enables Unprovable Block Construction - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/pairing.rs`, `msm.rs`, `mappings.rs`)

---

### Summary

All EIP-2537 (BLS12-381) precompile implementations hard-code `cost_native = 0`, meaning they consume **zero native (proving) resources** regardless of input size. Because native resources model actual RISC-V proving cycles, an unprivileged caller can craft transactions that pass all resource checks in forward execution but require an unbounded number of RISC-V cycles to prove, making the block unprovable or causing a forward/proving divergence.

---

### Finding Description

ZKsync OS uses a dual-resource model: EVM gas (ergs) and a "native" resource that models the actual RISC-V proving cost. The native resource is the critical guard against blocks that are too expensive to prove. The block-level cap `MAX_NATIVE_COMPUTATIONAL = 1 << 35` enforces this.

In every EIP-2537 precompile, the native cost is explicitly set to zero:

**`pairing.rs`** — BLS12-381 pairing check:
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(cost_native),
))?;
``` [1](#0-0) 

**`msm.rs`** — BLS12-381 G1 MSM and G2 MSM:
```rust
let cost_native = 0;
``` [2](#0-1) [3](#0-2) 

**`mappings.rs`** — BLS12-381 G1/G2 field-to-curve mappings: [4](#0-3) 

This is in direct contrast to the BN254 pairing precompile, which correctly charges native resources proportional to the number of pairs:

```rust
let native_cost = (num_pairs as u64) * BN254_PAIRING_PER_PAIR_NATIVE_COST
    + BN254_PAIRING_BASE_NATIVE_COST;
resources.charge(&R::from_ergs_and_native(ergs_cost, ...native_cost...))?;
``` [5](#0-4) 

The `MAX_NATIVE_COMPUTATIONAL` constant is the block-level proving budget: [6](#0-5) 

The block-level check against this limit only fires if native is actually consumed: [7](#0-6) 

Because BLS12-381 operations charge `cost_native = 0`, the block-level native limit is never triggered by these precompiles, no matter how many pairs are processed.

---

### Impact Explanation

BLS12-381 pairing and MSM are among the most computationally intensive operations in the EVM. Each pairing pair involves a full Miller loop and final exponentiation over Fq12, implemented in pure Rust running on RISC-V. With `cost_native = 0`:

1. A transaction can call the BLS12-381 pairing precompile with the maximum number of pairs permitted by the EVM gas limit (e.g., with 30M block gas: ~430 pairs per call at 37700 + 32600×n gas), consuming zero native resources.
2. Multiple such transactions can fill a block, each consuming zero native but enormous actual RISC-V cycles.
3. The forward system records native = 0 for these operations and accepts the block as valid.
4. The prover must actually execute the BLS12-381 arithmetic in RISC-V, consuming cycles far exceeding `MAX_NATIVE_COMPUTATIONAL`.
5. Result: the block is valid in forward execution but **unprovable**, or the prover diverges from the forward system's resource accounting.

This is a **resource accounting bug** leading to **valid-execution unprovability** — a block that the sequencer accepts but the prover cannot process within its cycle budget.

---

### Likelihood Explanation

- The BLS12-381 precompiles are live EIP-2537 precompiles reachable by any EVM contract call.
- No privileged access is required; any unprivileged transaction sender can call these precompiles via a contract.
- The attacker only needs to submit a transaction with sufficient EVM gas to invoke the precompile with many pairs.
- The EVM gas cost is real and paid, but the native cost is zero, so the attack is economically feasible at standard gas prices.
- The attack is deterministic and reproducible.

---

### Recommendation

Assign a non-zero `cost_native` to all EIP-2537 precompiles, proportional to the actual RISC-V proving cost per operation. Follow the pattern established by `Bn254PairingCheckImpl`, which charges `BN254_PAIRING_PER_PAIR_NATIVE_COST` per pair and `BN254_PAIRING_BASE_NATIVE_COST` as a fixed overhead.

For BLS12-381 pairing, the native cost per pair should be benchmarked against actual RISC-V cycle counts for the `multi_miller_loop` and `final_exponentiation` paths in `crypto/src/bls12_381/curves/pairing_impl.rs`. [8](#0-7) 

Similarly, G1 MSM and G2 MSM should charge native proportional to `num_pairs` and the window-based Pippenger loop cost in `msm.rs`. [9](#0-8) 

---

### Proof of Concept

1. Deploy a contract with bytecode that calls precompile address `0x0f` (BLS12-381 pairing check, `BLS12_PAIRING_CHECK`) with a large input of valid G1/G2 pairs.
2. Submit a transaction with gas limit = block gas limit (e.g., 30,000,000).
3. The transaction charges EVM gas proportional to `37700 + 32600 * num_pairs` but charges **zero native resources**.
4. The forward system accepts the transaction; `computational_native_used` remains 0 for these operations.
5. The block-level native check at `check_for_block_limits` does not trigger.
6. The prover must execute the full BLS12-381 Miller loop and final exponentiation for each pair in RISC-V, consuming actual cycles orders of magnitude beyond `MAX_NATIVE_COMPUTATIONAL = 2^35`.
7. The block cannot be proven. [10](#0-9) [11](#0-10)
