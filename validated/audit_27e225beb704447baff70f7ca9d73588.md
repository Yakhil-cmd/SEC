### Title
Unchecked Exit Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens **before** calling the `ExitToNear` or `ExitToEthereum` precompile via inline assembly. The return value of the precompile call is captured in a local variable `res` but is **never checked**. If the precompile call fails at the EVM level, the tokens are permanently destroyed with no corresponding NEAR-side transfer scheduled, resulting in irreversible loss of user funds.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear()` executes the following sequence:

1. `_burn(_msgSender(), amount)` — irreversibly destroys the caller's ERC-20 tokens.
2. Constructs the precompile input.
3. Calls the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` via inline assembly.
4. The return value `res` is assigned but **never inspected or acted upon**.

```solidity
// EvmErc20.sol lines 53–63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here — irreversible

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — silent failure
    }
}
```

The same pattern appears in `withdrawToEthereum()` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return `Err(ExitError::...)` in multiple reachable conditions:

- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — recipient bytes do not parse as a valid NEAR account ID.
- `ERR_TARGET_TOKEN_NOT_FOUND` — the ERC-20 address is not registered in the NEP-141 mapping.
- `ERR_INVALID_IN_STATIC` — called in a static context.
- `ERR_INVALID_IN_DELEGATE` — called via `delegatecall`.
- `ERR_MISSING_FLAG` / `ERR_INVALID_FLAG` — malformed input.

When the precompile returns an error, the EVM `call()` opcode returns `0` in `res`. Under standard EVM semantics, only the sub-call's state is reverted; the calling contract's prior state changes (the `_burn`) are **not** automatically reverted. Because `res` is never checked and the function never reverts, the transaction succeeds with the burn committed but no NEAR promise scheduled.

The `error_refund` feature in the precompile only handles the case where a NEAR promise is successfully scheduled but subsequently fails on the NEAR side. It does not cover the case where the precompile itself fails at the EVM level and no promise is ever created. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

When the precompile call fails silently:

- The user's ERC-20 tokens are burned (EVM supply permanently reduced).
- No NEAR promise is scheduled, so no NEP-141 tokens are transferred to the recipient.
- There is no refund path for this failure mode.
- The funds are **permanently and irrecoverably lost**.

This constitutes **permanent freezing of funds**, which is a Critical impact under the allowed scope. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The most directly reachable trigger is supplying an invalid NEAR account ID as `recipient`. The `recipient` parameter is raw `bytes` with no on-chain validation before the burn. A user who mistypes a recipient (e.g., an account ID containing uppercase letters, spaces, or exceeding 64 characters) will have their tokens burned with no recovery. The `ERR_TARGET_TOKEN_NOT_FOUND` path is also reachable if the ERC-20 contract's NEP-141 mapping entry is absent or corrupted. Likelihood is **Medium** — accidental loss is plausible for any user of the bridge withdrawal flow. [7](#0-6) [8](#0-7) 

---

### Recommendation

Reverse the operation order: validate the recipient and call the precompile **before** burning tokens. Alternatively, check `res` after the assembly call and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Because the burn is irreversible once committed, the safest fix is to perform all validation and the precompile call first, and only burn on confirmed success. [1](#0-0) [2](#0-1) 

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora (or use an existing bridged token).
2. Mint tokens to address `A`.
3. From address `A`, call `withdrawToNear(invalidRecipient, amount)` where `invalidRecipient` is, e.g., `"INVALID ACCOUNT!!!"` (contains spaces/uppercase — not a valid NEAR account ID).
4. The `ExitToNear` precompile parses the recipient, fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID`, and returns `0` to the assembly block.
5. `res` is `0` but is never checked; the function returns normally.
6. The EVM transaction succeeds; `amount` tokens are burned from `A`'s balance.
7. No NEAR promise is scheduled; no NEP-141 tokens are transferred.
8. `A`'s tokens are permanently destroyed with no recourse. [8](#0-7) [1](#0-0)
