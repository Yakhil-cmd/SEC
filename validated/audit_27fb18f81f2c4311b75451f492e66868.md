Audit Report

## Title
Incomplete Redaction in `RuleConfidentialityFormatter::format` Leaks `incident_id` and Version Metadata for Undisclosed Rules to Any Caller — (`rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`, `getter.rs`)

## Summary

`RuleConfidentialityFormatter::format` only nullifies `rule_raw` and `description` for undisclosed rules, leaving `incident_id`, `added_in_version`, and `removed_in_version` in the response. Because `get_rule_by_id` and `get_rules_by_incident_id` are `#[query]` methods, the `inspect_message` hook never fires for them, so any anonymous caller can invoke them directly. Rule IDs and incident IDs needed to probe these endpoints are already obtainable from `get_config` (also a `#[query]`). The result is a concrete, reachable metadata disclosure for every undisclosed rule.

## Finding Description

**Root cause — `RuleConfidentialityFormatter::format` (`confidentiality_formatting.rs` L34–42):**
The formatter only sets `description = None` and `rule_raw = None` when `disclosed_at.is_none()`. It does not touch `incident_id`, `added_in_version`, or `removed_in_version`. These fields are fully populated before the formatter is called (`getter.rs` L224–232 for `RuleGetter`, L182–190 for `IncidentGetter`) and are returned verbatim to the caller.

**`inspect_message` does not protect query calls (`canister.rs` L34–67):**
`inspect_message` is an ingress-message pre-consensus hook; it fires only for update calls and replicated queries. `get_rule_by_id` (L123) and `get_rules_by_incident_id` (L136) are `#[query]` methods. Non-replicated query calls to these endpoints never pass through `inspect_message`, so the hook's blanket rejection of unlisted methods (L62–67) is entirely ineffective against them.

**`get_config` exposes all rule IDs and incident IDs to anonymous callers (`confidentiality_formatting.rs` L17–28, `canister.rs` L110–120):**
`ConfigConfidentialityFormatter::format` also only redacts `rule_raw` and `description`, leaving `rule_id` and `incident_id` in every `OutputRule` entry. `get_config` is a `#[query]` method callable by any anonymous principal as a non-replicated query, providing the full set of IDs needed to probe the other two endpoints.

**`AccessLevelResolver` assigns `RestrictedRead` to anonymous callers (`access_control.rs` L38–55):**
Any caller that is neither the authorized principal nor a registered API boundary node receives `RestrictedRead`, which routes through the formatter — confirming the redaction path is reached and the leak is in the formatter itself.

**Test explicitly confirms the leak (`getter.rs` L433–446):**
The existing unit test asserts that a `RestrictedRead` caller receives `incident_id`, `added_in_version: 1`, and `removed_in_version: Some(3)` for an undisclosed rule, with only `rule_raw` and `description` nulled. This is the intended behavior per the current implementation, but it is the bug.

## Impact Explanation

Any anonymous caller can enumerate the full set of undisclosed rule IDs and incident IDs via `get_config`, then call `get_rule_by_id` per rule ID to obtain `added_in_version` and `removed_in_version`, and call `get_rules_by_incident_id` per incident UUID to enumerate all rules grouped by incident with full version history. This reveals: (a) the existence of active undisclosed security incidents, (b) which rules belong to the same incident, and (c) exactly which config versions introduced or removed each rule — the precise information the confidentiality mechanism is designed to protect. This constitutes a significant boundary/API infrastructure security impact: the confidentiality invariant of the rate-limit canister is broken for all undisclosed rules, with no authentication required.

## Likelihood Explanation

Exploitability is trivial. No authentication, no special tooling, no timing dependency, and no privileged access are required. Any caller with network access to the canister can execute the full call sequence using standard IC query calls (`dfx canister call --query` or any IC agent). The attack is repeatable and deterministic.

## Recommendation

`RuleConfidentialityFormatter::format` must also redact `incident_id`, `added_in_version`, and `removed_in_version` for undisclosed rules. The correct invariant is: if `disclosed_at.is_none()` and the caller is not `FullAccess`/`FullRead`, return only `rule_id` with all other fields set to `None` (or return a `NotFound`-equivalent opaque error). The same fix must be applied to `ConfigConfidentialityFormatter::format` for the `incident_id` field in `OutputRule`. Additionally, consider whether `get_rule_by_id` and `get_rules_by_incident_id` should be replicated queries or have in-method authorization checks, since `inspect_message` cannot protect non-replicated query calls.

## Proof of Concept

```bash
# Step 1: Obtain all rule_ids and incident_ids as anonymous caller (non-replicated query)
dfx canister call <rate_limit_canister> get_config '(null)' --query
# Returns: all rule_id and incident_id values, including for undisclosed rules

# Step 2: For each rule_id, retrieve version metadata
dfx canister call <rate_limit_canister> get_rule_by_id '("<rule_id>")' --query
# Response for undisclosed rule:
# { rule_id = "<uuid>"; incident_id = "<uuid>"; rule_raw = null; description = null;
#   disclosed_at = null; added_in_version = 1; removed_in_version = opt 3 }

# Step 3: Enumerate all rules per incident
dfx canister call <rate_limit_canister> get_rules_by_incident_id '("<incident_id>")' --query
# Returns all rules for the incident with full version metadata
```

The existing unit test at `getter.rs` L433–446 already proves the leak: it asserts `incident_id`, `added_in_version: 1`, and `removed_in_version: Some(3)` are present in the `RestrictedRead` response. A regression test for the fix should assert that all of `incident_id`, `added_in_version`, and `removed_in_version` are `None` (or the call returns `NotFound`) for any `RestrictedRead` caller querying an undisclosed rule.
