### Title
Silo Whitelist Bypass in `receive_base_tokens` Allows ETH Minting to Non-Whitelisted Addresses, Freezing Funds - (File: engine/src/engine.rs)

### Summary
In Silo mode, `receive_erc20_tokens` enforces the Address whitelist and redirects tokens to a configured fallback address when the recipient is not whitelisted. The analogous `receive_base_tokens` function — which handles ETH (base token) bridging — performs **no whitelist check at all**. Any user can bridge ETH to a non-whitelisted EVM address, bypassing the Silo access control. Because non-whitelisted addresses are blocked from submitting transactions by `assert_access`, the minted ETH becomes permanently inaccessible to the recipient.

### Finding Description

`engine/src/engine.rs` contains two token-receipt paths, both reachable through `ft_on_transfer`:

**ERC-20 path (whitelist enforced):** [1](#0-0) 

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

**ETH path (no whitelist check):** [2](#0-1) 

```rust
pub fn receive_base_tokens(&mut self, args: &FtOnTransferArgs) -> ... {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let receipient = message_data.recipient;
    // ← no whitelist check here
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

The dispatch between the two paths is determined solely by whether the NEAR predecessor is the ETH connector account: [3](#0-2) 

The `is_allow_receive_erc20_tokens` helper checks the `Address` whitelist — the same list that `assert_access` uses to gate transaction submission: [4](#0-3) 

And `assert_access` blocks non-whitelisted addresses from submitting any transaction: [5](#0-4) 

The result is a structural inconsistency: ERC-20 tokens are redirected to the fallback address when the recipient is not whitelisted, but ETH is minted directly to the recipient with no redirect and no rejection.

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

When a user bridges ETH to a non-whitelisted EVM address in Silo mode:
1. `receive_base_tokens` mints the ETH to that address unconditionally.
2. The address cannot submit any EVM transaction because `assert_access` rejects it.
3. The ETH is trapped: the owner cannot move it, and no fallback redirect occurs (unlike ERC-20).

Recovery requires the Silo operator to explicitly whitelist the address. If the operator declines or is unaware, the funds are permanently frozen. The Silo operator's stated intent — enforced for ERC-20 — is that non-whitelisted addresses should not receive tokens; ETH silently violates this invariant.

### Likelihood Explanation

**Medium.** The precondition is that Silo mode is active with the `Address` whitelist enabled and a fallback address configured. This is the normal production configuration for any Silo deployment. Once those conditions hold, any user who calls `ft_transfer_call` on the ETH connector with a non-whitelisted recipient address triggers the bug. No elevated privileges are required; the attacker-controlled input is the `msg` field of the NEP-141 transfer, which specifies the EVM recipient.

### Recommendation

Add the same whitelist guard to `receive_base_tokens` that exists in `receive_erc20_tokens`. If `get_erc20_fallback_address` returns `Some` and `is_allow_receive_erc20_tokens` (or an equivalent `is_allow_receive_base_tokens`) returns `false`, redirect the ETH to the fallback address instead of the requested recipient. This makes the two token-receipt paths consistent and closes the bypass.

### Proof of Concept

1. Silo operator calls `set_silo_params` with a non-zero `erc20_fallback_address` and calls `set_whitelist_status` to enable `WhitelistKind::Address`. [6](#0-5) 

2. Attacker calls `ft_transfer_call` on the ETH connector NEP-141 contract, specifying Aurora as `receiver_id` and a **non-whitelisted** EVM address in `msg`.

3. The ETH connector calls `ft_on_transfer` on Aurora. Because `predecessor_account_id == get_connector_account_id`, the engine dispatches to `receive_base_tokens`. [3](#0-2) 

4. `receive_base_tokens` parses the recipient from `args.msg` and calls `set_balance` with no whitelist check. [7](#0-6) 

5. ETH is now credited to the non-whitelisted address. Any subsequent EVM transaction from that address is rejected by `assert_access`: [8](#0-7) 

6. The ETH is frozen. The ERC-20 fallback redirect that would have protected the user never fires, because `receive_base_tokens` never consults the whitelist.

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
