Audit Report

## Title
Silent Neuron Deduplication via Incomplete `Ord` in `NeuronsFundSnapshotPb::validate()` - (`rs/nns/governance/src/neurons_fund.rs`)

## Summary

`NeuronsFundNeuronPortion` derives `PartialEq` over all fields (including `id`) but implements `Ord` using only `controller` and `maturity_equivalent_icp_e8s`. `NeuronsFundSnapshotPb::validate()` collects validated portions into a `BTreeSet`, which uses `Ord` exclusively for membership decisions. Two distinct neurons owned by the same principal with identical maturity amounts compare `Ordering::Equal`, causing `BTreeSet` to silently discard one. The resulting `NeuronsFundSnapshot` permanently omits the dropped neuron, and its reserved maturity is never refunded after swap settlement.

## Finding Description

`NeuronsFundNeuronPortion` is declared at line 335–352 with `#[derive(Clone, Eq, PartialEq, Debug)]`, making `PartialEq` compare all six fields including `id`. The manual `Ord` implementation at lines 356–365 consults only `controller` and `maturity_equivalent_icp_e8s`:

```rust
impl Ord for NeuronsFundNeuronPortion {
    fn cmp(&self, other: &Self) -> Ordering {
        match self.controller.cmp(&other.controller) {
            Ordering::Equal => self
                .maturity_equivalent_icp_e8s
                .cmp(&other.maturity_equivalent_icp_e8s),
            ordering => ordering,
        }
    }
}
```

This violates Rust's requirement that `a == b` implies `a.cmp(b) == Ordering::Equal`. Two neurons `N1` and `N2` with the same `controller` and `maturity_equivalent_icp_e8s` but different `id` values satisfy `N1 != N2` (by `PartialEq`) yet `N1.cmp(&N2) == Ordering::Equal` (by `Ord`).

`NeuronsFundSnapshotPb::validate()` at lines 712–724 collects the deserialized portions into a `BTreeSet<NeuronsFundNeuronPortion>`:

```rust
let neurons_fund = self
    .neurons_fund_neuron_portions
    .iter()
    .enumerate()
    .map(|(i, n)| { n.validate().map_err(...) })
    .collect::<Result<BTreeSet<_>, _>>()?;
Ok(NeuronsFundSnapshot::new(neurons_fund))
```

`BTreeSet::insert` uses `Ord` exclusively. When the second of two `Ord`-equal neurons is inserted, the set silently retains the first and discards the second — no error, no warning. The surviving set is passed to `NeuronsFundSnapshot::new` (lines 500–506), which builds a `BTreeMap<NeuronId, NeuronsFundNeuronPortion>` keyed by `n.id`. The dropped neuron's `NeuronId` never appears in this map.

All downstream operations — `diff()` (line 547), `refund_maturity_to_neurons_fund` (line 469), and SNS neuron creation — operate against this corrupted map. `diff()` iterates `self.neurons` and looks up each `id` in the other snapshot; the missing `id` is simply absent, so no refund entry is produced for it.

## Impact Explanation

The dropped neuron's reserved maturity is permanently lost: it was drawn from the neuron at swap open time (via `draw_maturity_from_neurons_fund`) but is never returned because the refund snapshot computed by `diff(initial, final)` contains no entry for the missing `NeuronId`. This constitutes a permanent, irreversible loss of ICP-equivalent maturity for the neuron's controller. The affected asset is NNS neuron maturity, a governance asset directly convertible to ICP. This matches the **High** impact class: significant NNS/SNS security impact with concrete, permanent user-fund harm.

## Likelihood Explanation

A single NNS principal controlling multiple neurons is common and requires no special privileges. Two neurons under the same controller accumulating identical `maturity_e8s_equivalent` values is plausible through normal staking and maturity accrual over time. Both neurons must have joined the Neurons' Fund and been included in the same `NeuronsFundSnapshot`. The silent drop is triggered by any NNS Governance canister upgrade — a routine, regularly scheduled operation — while an SNS swap using those neurons is open or settling. No attacker action is required beyond ordinary neuron ownership; the bug manifests automatically on upgrade.

## Recommendation

Add `id` as a final tie-breaker in `NeuronsFundNeuronPortion::Ord` to make it consistent with `PartialEq`:

```rust
impl Ord for NeuronsFundNeuronPortion {
    fn cmp(&self, other: &Self) -> Ordering {
        self.controller
            .cmp(&other.controller)
            .then_with(|| self.maturity_equivalent_icp_e8s.cmp(&other.maturity_equivalent_icp_e8s))
            .then_with(|| self.id.cmp(&other.id))
    }
}
```

Alternatively, replace the `BTreeSet` in `validate()` with a `Vec`, then explicitly check for duplicate `NeuronId`s and return a `NeuronsFundSnapshotValidationError` rather than silently dropping entries. The latter approach is more defensive and surfaces data corruption at deserialization time.

## Proof of Concept

1. Create two `NeuronsFundNeuronPortionPb` entries with distinct `nns_neuron_id` values (`N1`, `N2`), the same `controller` principal `P`, and the same `maturity_equivalent_icp_e8s = 5_000_000_000`, each with `amount_icp_e8s = 2_500_000_000` and `is_capped = false`.
2. Wrap both in a `NeuronsFundSnapshotPb` and call `validate()`.
3. Assert that the resulting `NeuronsFundSnapshot` contains **two** entries. Under the current code it contains only one — the second insertion into `BTreeSet` is silently dropped.
4. Call `initial_snapshot.diff(&final_snapshot)` where `final_snapshot` contains both neurons with `amount_icp_e8s = 0`; assert the refund snapshot contains entries for both `N1` and `N2`. Under the current code, only one refund entry is produced.

This can be implemented as a unit test in `rs/nns/governance/src/neurons_fund.rs` with no external dependencies.
