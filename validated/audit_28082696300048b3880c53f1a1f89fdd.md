### Title
Stale `erc20_balances` Entry Causes Minter Canister Panic When a ckERC20 Token's ERC-20 Contract Address Persists in Balance Map After Token Removal — (File: `rs/ethereum/cketh/minter/src/state.rs`)

### Summary

The ckETH minter canister maintains two independent data structures: `ckerc20_tokens` (the registry of supported ckERC20 tokens) and `erc20_balances` (a balance map keyed by ERC-20 contract address). The `erc20_balances_by_token_symbol()` function, called during Prometheus metrics scraping, assumes every address in `erc20_balances` has a corresponding entry in `ckerc20_tokens`. If a token's ERC-20 contract address remains in `erc20_balances` while the token is absent from `ckerc20_tokens` — a state that can arise from a minter upgrade that removes a token from the registry while in-flight or pending withdrawals still hold a non-zero balance — the minter will panic on every metrics scrape, effectively bricking the canister's HTTP endpoint and halting observability. Additionally, the `withdrawal_status` query and the dashboard HTTP handler both call `.expect("BUG: unknown ERC-20 token")` / `.unwrap()` on the same lookup, causing those endpoints to trap for any pending ckERC20 withdrawal whose token has been removed from the registry.

### Finding Description

The ckETH minter tracks ERC-20 balances in `State::erc20_balances` (a `BTreeMap<Address, Erc20Value>`) separately from the token registry `State::ckerc20_tokens` (a `DedupMultiKeyMap<Principal, Address, CkTokenSymbol>`).

**Balance is added** when a deposit event is accepted: [1](#0-0) 

**Balance is subtracted** only when a withdrawal transaction is finalized with `TransactionStatus::Success`: [2](#0-1) 

This means that while a ckERC20 withdrawal is pending (created but not yet finalized on Ethereum), the ERC-20 contract address has a non-zero entry in `erc20_balances` but the corresponding Ethereum tokens have not yet left the minter's address.

The `erc20_balances_by_token_symbol()` function, called during Prometheus metrics scraping, iterates over every address in `erc20_balances` and performs a mandatory lookup into `ckerc20_tokens`:

```rust
pub fn erc20_balances_by_token_symbol(&self) -> BTreeMap<&CkTokenSymbol, &Erc20Value> {
    self.erc20_balances
        .balance_by_erc20_contract
        .iter()
        .map(|(erc20_contract, balance)| {
            let symbol = self
                .ckerc20_tokens
                .get_alt(erc20_contract)
                .unwrap_or_else(|| {
                    panic!("BUG: missing symbol for ERC-20 contract {erc20_contract}")
                });
            (symbol, balance)
        })
        .collect()
}
``` [3](#0-2) 

This panic is triggered from the metrics endpoint: [4](#0-3) 

Similarly, the `withdrawal_status` query endpoint calls `.unwrap()` on the same lookup for any pending ckERC20 withdrawal: [5](#0-4) 

And the dashboard HTTP handler calls `.expect("BUG: unknown ERC-20 token")`: [6](#0-5) 

The token registry only supports **adding** tokens via `record_add_ckerc20_token()`: [7](#0-6) 

There is no `remove_ckerc20_token` function. However, the minter is an upgradeable canister. A future upgrade that reconstructs state without a previously-supported token (e.g., to deprecate a token) while `erc20_balances` still holds a non-zero entry for that token's contract address would create the invariant violation. The `erc20_balances` map is never cleared of entries for removed tokens because the subtraction only happens on finalized successful withdrawals.

The root cause is the **absence of a cross-map invariant check**: `erc20_balances` can contain addresses not present in `ckerc20_tokens`, but multiple code paths assume this is impossible and panic unconditionally.

### Impact Explanation

If the invariant is violated (an address exists in `erc20_balances` but not in `ckerc20_tokens`):

1. **Metrics endpoint panics** on every call to `erc20_balances_by_token_symbol()`, causing the Prometheus scrape HTTP handler to trap. This disables all minter observability.
2. **`withdrawal_status` query traps** for any pending ckERC20 withdrawal of the removed token, denying users the ability to check their withdrawal status.
3. **Dashboard HTTP handler traps** when rendering pending withdrawal requests or pending deposits for the removed token.
4. The minter's timer-driven withdrawal processing continues to function (it does not call these paths), but the accounting divergence from the original report applies: the `erc20_balances` entry for the removed token is no longer reflected in any user-visible balance reporting, yet the ckERC20 tokens burned to initiate those withdrawals are already gone from the ledger.

### Likelihood Explanation

The ckERC20 token registry is append-only in the current code — there is no `remove_ckerc20_token` governance path. However:

- The minter is an upgradeable canister controlled by NNS governance. A future upgrade proposal could reconstruct state without a deprecated token while pending withdrawals remain.
- The `erc20_balances` map is populated on deposit and drained only on finalized successful withdrawal. Any window between deposit and finalization leaves a non-zero balance entry.
- The design explicitly separates the two maps with no enforced cross-map invariant, making this a latent structural bug that becomes exploitable the moment any token removal path is introduced.

The likelihood is **medium**: it requires a governance upgrade action, but the code provides no guard against it and the pattern is directly analogous to the Renzo finding.

### Recommendation

1. **Enforce the cross-map invariant at state mutation boundaries**: when `erc20_balances` is updated, assert that the contract address is present in `ckerc20_tokens`.
2. **Replace panics with graceful degradation** in `erc20_balances_by_token_symbol()`, `withdrawal_status`, and the dashboard handler — use `unwrap_or` or skip unknown addresses rather than panicking.
3. **If token removal is ever introduced**, require that `erc20_balances.balance_of(contract_address) == 0` before allowing removal, analogous to the Renzo mitigation of checking `claimReserve[token] == 0`.
4. **Add a state invariant check** (similar to `is_equivalent_to`) that verifies every key in `erc20_balances` has a corresponding entry in `ckerc20_tokens`.

### Proof of Concept

1. A user deposits 1000 ckUSDC. `erc20_balances["0xA0b8..."]` becomes `1000`.
2. The user calls `withdraw_erc20`, burning 1000 ckUSDC. The withdrawal request is queued. `erc20_balances["0xA0b8..."]` remains `1000` (subtraction only happens on finalized success).
3. An NNS upgrade proposal removes ckUSDC from the minter's `ckerc20_tokens` registry (e.g., to deprecate the token).
4. The Ethereum transaction is still pending. `erc20_balances["0xA0b8..."]` = `1000`, but `ckerc20_tokens` no longer contains `"0xA0b8..."`.
5. Any call to the Prometheus metrics endpoint triggers `erc20_balances_by_token_symbol()` → `panic!("BUG: missing symbol for ERC-20 contract 0xA0b8...")`. [8](#0-7) 
6. Any call to `withdrawal_status` for the pending withdrawal triggers `.unwrap()` → trap. [5](#0-4) 
7. The minter's HTTP dashboard endpoint traps when rendering the pending withdrawal. [6](#0-5)
