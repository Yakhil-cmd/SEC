### Title
Unrecoverable wNEAR Burn When Native NEAR Transfer Targets Non-Existent Account — (File: `engine/src/contract_methods/connector.rs`)

### Summary

In the wNEAR-unwrap exit flow, `exit_to_near_precompile_callback` issues a `PromiseAction::Transfer` of unwrapped NEAR to a user-supplied `target_account_id` with no error-recovery callback. If the target NEAR account does not exist, the NEAR runtime returns the tokens to the Aurora engine contract, but the user's wNEAR ERC-20 tokens were already burned with no re-mint path. The result is a permanent, irrecoverable fund freeze.

---

### Finding Description

The wNEAR unwrap path is triggered when a user calls `withdrawToNear(recipient + ":unwrap", amount)` on the wNEAR ERC-20 contract. The `ExitToNear` precompile detects the `Message::UnwrapWnear` suffix and constructs a two-step promise chain:

**Step 1 — base promise:** call `near_withdraw` on the wNEAR NEP-141 contract to convert wNEAR → native NEAR.

**Step 2 — callback:** `exit_to_near_precompile_callback` is invoked. If the base promise succeeded, it issues a `PromiseAction::Transfer` of the unwrapped NEAR to the user-supplied `target_account_id`. [1](#0-0) 

The critical flaw is on line 226–227: `handler.promise_create_batch(&promise)` followed by `handler.promise_return(promise_id)`. The transfer promise is returned as the function's result, but **no further callback is attached to it**. There is no handler for the case where this transfer fails. [2](#0-1) 

In NEAR Protocol, a `Transfer` action in a batch to a **non-existent account** fails. When it fails, the NEAR runtime returns the tokens to the predecessor (the Aurora engine contract), not to the user. The Aurora engine has no mechanism to credit these returned tokens back to any EVM address.

The `error_refund` feature only guards the **base promise** (`near_withdraw`). If `near_withdraw` fails, the ERC-20 tokens are re-minted via `refund_on_error`. But if `near_withdraw` **succeeds** and the subsequent `Transfer` fails, there is no analogous recovery path. [3](#0-2) 

The ERC-20 burn happens inside the Solidity `withdrawToNear` function before the precompile is called: [4](#0-3) 

Once the burn executes and `near_withdraw` succeeds, the ERC-20 tokens are gone. If the NEAR transfer then fails, the user has lost both their ERC-20 tokens and the corresponding NEAR.

### Impact Explanation

**Critical — Permanent freezing of funds.**

- The user's wNEAR ERC-20 tokens are burned (irreversible at EVM level).
- The unwrapped NEAR tokens are returned to the Aurora engine contract by the NEAR runtime after the failed transfer.
- The Aurora engine has no accounting entry to credit these returned tokens to any EVM address.
- The NEAR tokens are permanently stranded in the Aurora engine contract with no withdrawal path for the user.

---

### Likelihood Explanation

**Medium.** Any user who calls `withdrawToNear` with the `:unwrap` suffix and supplies a NEAR account ID that does not yet exist on-chain triggers this path. This includes:
- Typos in the account ID.
- Accounts that have been deleted.
- Accounts that have not yet been created (e.g., a freshly generated key pair whose account has not been funded).

The call is fully permissionless — any EVM address holding wNEAR ERC-20 tokens can trigger it.

---

### Recommendation

Attach an error-recovery callback to the `PromiseAction::Transfer` promise in `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-mint the burned wNEAR ERC-20 tokens to the original sender's EVM address, mirroring the existing `refund_on_error` logic used for the `near_withdraw` failure case. [6](#0-5) 

Specifically, the `PromiseBatchAction` for the transfer should be wrapped in a `PromiseWithCallbackArgs` whose callback invokes `exit_to_near_precompile_callback` (or a dedicated handler) with a `RefundCallArgs` pointing to the wNEAR ERC-20 address and the original sender's EVM address.

---

### Proof of Concept

1. Alice holds 100 wNEAR ERC-20 tokens on Aurora.
2. Alice calls `withdrawToNear("alice-new-account.near:unwrap", 100)` on the wNEAR ERC-20 contract. The account `alice-new-account.near` does not yet exist on NEAR.
3. The ERC-20 contract burns 100 tokens from Alice's EVM balance and calls the `ExitToNear` precompile.
4. The precompile schedules: `near_withdraw(100)` → callback `exit_to_near_precompile_callback`.
5. `near_withdraw` succeeds; the Aurora engine now holds 100 NEAR (unwrapped from wNEAR).
6. `exit_to_near_precompile_callback` fires, issues `PromiseAction::Transfer { amount: 100 NEAR }` to `alice-new-account.near`, and returns the promise.
7. The NEAR runtime executes the transfer; it fails because `alice-new-account.near` does not exist.
8. The 100 NEAR is returned to the Aurora engine contract by the NEAR runtime.
9. Alice's 100 wNEAR ERC-20 tokens are permanently burned. The 100 NEAR is stranded in the Aurora engine with no recovery path. [1](#0-0) [7](#0-6)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
}
