Audit Report

## Title
Single Byzantine Subnet Peer Can Repeatedly Abort Ongoing State Sync via `NO_CONTENT` Responses — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
A registered Byzantine subnet peer can permanently stall a victim replica's state sync by returning `StatusCode::NO_CONTENT` for every chunk request. Each `NO_CONTENT` response unconditionally evicts the peer from `active_downloads`; when it is the only peer, the `active_downloads.is_empty()` guard breaks the run loop, drops the `Chunkable` tracker (losing all progress), and stalls the node until the next advert broadcast cycle (~5 s). Because the Byzantine peer re-advertises every broadcast interval and can again be the first peer added to the fresh sync, it can repeat this cycle indefinitely, preventing the victim from ever completing state sync.

## Finding Description

**Root cause chain — four confirmed code facts:**

**Fact 1 — Server returns `NO_CONTENT` when chunk is unavailable.** [1](#0-0) 

The client maps that status code directly to `DownloadChunkError::NoContent`: [2](#0-1) 

**Fact 2 — `NoContent` unconditionally evicts the peer from `active_downloads`.**
Unlike `Overloaded`, `Timeout`, and `Cancelled` (which only re-queue the chunk), `NoContent` removes the peer and decrements `allowed_downloads` by `PARALLEL_CHUNK_DOWNLOADS`: [3](#0-2) 

**Fact 3 — Empty `active_downloads` breaks the run loop and drops the tracker.**
After every download result, the loop unconditionally checks: [4](#0-3) 

When the loop breaks, `run()` returns, dropping the `tracker: Arc<Mutex<Box<dyn Chunkable<T>>>>` — all downloaded-chunk progress is lost.

**Fact 4 — Recovery requires a new advert; in-flight `try_send` calls are silently dropped.**
After the loop breaks, `new_peers_rx.close()` is called: [5](#0-4) 

In `handle_advert`, honest peers' adverts use non-blocking `try_send` which silently discards the send if the channel is closed: [6](#0-5) 

Recovery only happens when the *next* advert arrives and `shutdown.completed()` is true, at which point `ongoing_state_sync` is cleared and `maybe_start_state_sync` is called from scratch: [7](#0-6) 

The advert broadcast interval is 5 seconds: [8](#0-7) 

**Exploit flow:**
1. Byzantine peer sends a valid advert for height H → `handle_advert` calls `maybe_start_state_sync` → `start_ongoing_state_sync` → Byzantine peer is the sole entry in `active_downloads`.
2. Byzantine peer returns `StatusCode::NO_CONTENT` for the first chunk request (no delay needed).
3. `handle_downloaded_chunk_result` removes the peer → `active_downloads.is_empty()` → loop breaks → tracker dropped.
4. `shutdown.completed()` becomes true. The next advert (from any peer, including the Byzantine one) triggers cleanup and a brand-new sync from zero.
5. Byzantine peer re-advertises within 5 s and can again be the first peer added to the new sync. Steps 2–5 repeat indefinitely.

**Why existing guards are insufficient:**
- The `try_send` guard at L154 is evaluated *before* `shutdown.completed()` at L156, but the channel is already closed after the loop breaks, so honest peers' in-flight adverts are silently dropped.
- There is no threshold, cooldown, or retry count before a peer is permanently evicted for `NoContent`.
- There is no mechanism to preserve the `Chunkable` tracker across peer-set exhaustion.

## Impact Explanation
A replica that cannot complete state sync falls behind consensus, cannot execute blocks, and — if it is a subnet member — degrades subnet liveness. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.** The attack is repeatable and causes indefinite stalling of the victim replica, not merely a one-time crash.

## Likelihood Explanation
The attacker must be a registered subnet node (NNS-approved), which is a meaningful barrier. However, the attack requires no cryptographic compromise — only returning a valid HTTP `204 No Content` response. It is most effective when the Byzantine peer is the only node advertising the target checkpoint (e.g., a lagging subnet where only one or two nodes have the checkpoint), or when it wins the 5-second advert timing race. Both conditions are achievable in practice. The attack is fully repeatable with no per-iteration cost.

## Recommendation
1. **Treat `NoContent` as transient, not permanent.** Re-queue the chunk (as done for `Overloaded`/`Timeout`) and only evict a peer after a configurable threshold of consecutive `NoContent` responses.
2. **Do not break the run loop solely on empty `active_downloads`.** If there are still in-flight downloads or the `new_peers_rx` channel has pending messages, wait for the next peer message before giving up.
3. **Preserve the `Chunkable` tracker across peer-set exhaustion.** Drop the tracker only on successful completion or explicit cancellation, not on transient peer loss.
