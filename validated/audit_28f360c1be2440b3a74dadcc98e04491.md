### Title
Fee-on-Transfer NEP-141 Token Accounting Discrepancy in `receive_erc20_tokens` - (File: `engine/src/engine.rs`)

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` blindly trusts the `amount` field from the `FtOnTransferArgs` callback to mint ERC-20 tokens on the Aurora EVM side, without verifying the actual NEP-141 balance change. A fee-on-transfer NEP-141 token can cause Aurora to mint more ERC-20 tokens than the NEP-141 tokens actually deposited, creating a bridge insolvency condition.

### Finding Description

When a NEP-141 token is bridged into Aurora via `ft_transfer_call`, the NEP-141 contract invokes `ft_on_transfer` on Aurora. The Aurora engine's `ft_on_transfer` entrypoint routes to `engine.receive_erc20_tokens`, which reads `args.amount` directly from the callback arguments and mints exactly that many ERC-20 tokens on the EVM side. [1](#0-0) [2](#0-1) 

The critical line is:

```rust
let amount = args.amount.as_u128();
``` [3](#0-2) 

This `amount` is then passed directly to `setup_receive_erc20_tokens_input`, which encodes it as the argument to the EVM-side `mint(address, uint256)` call: [4](#0-3) 

There is **no balance-before / balance-after check** on the NEP-141 side. The engine never verifies that it actually received `args.amount` NEP-141 tokens. If the NEP-141 token contract deducts a transfer fee from the receiver's credited amount but still calls `ft_on_transfer` with the original (pre-fee) `amount`, Aurora will mint more ERC-20 tokens than the NEP-141 tokens it holds, breaking the 1:1 backing invariant.

The NEP-141 standard does not prohibit fee-on-transfer behavior. A token contract is free to implement `ft_transfer_call` such that it deducts a fee from the amount credited to the receiver (Aurora) while reporting the original amount in the `ft_on_transfer` callback. Aurora has no mechanism to detect or reject this.

### Impact Explanation

**Critical — Bridge Insolvency / Permanent Fund Freeze.**

The ERC-20 mirror token on Aurora is backed 1:1 by NEP-141 tokens held by the Aurora contract. If an attacker bridges `N` NEP-141 tokens with a 10% fee, Aurora receives `0.9N` NEP-141 tokens but mints `N` ERC-20 tokens. The attacker can then exit (burn ERC-20, receive NEP-141) for the full `N`, draining `0.1N` tokens from other users' deposits. Repeated exploitation leads to insolvency: the bridge cannot fulfill all legitimate withdrawal requests, permanently freezing funds of honest users.

### Likelihood Explanation

Any unprivileged user can deploy a custom NEP-141 token with fee-on-transfer behavior, register it with Aurora via `deploy_erc20_token`, and immediately exploit the bridge. No admin access, governance capture, or key compromise is required. The entry path is fully permissionless:

1. Deploy a fee-on-transfer NEP-141 token on NEAR.
2. Call `deploy_erc20_token` on Aurora to register it.
3. Call `ft_transfer_call` on the NEP-141 token targeting Aurora with a crafted amount.
4. The NEP-141 contract calls `ft_on_transfer` on Aurora with the pre-fee amount.
5. Aurora mints the full pre-fee amount of ERC-20 tokens despite receiving fewer NEP-141 tokens.
6. Attacker exits via `withdrawToNear` / `ExitToNear` precompile, draining the surplus.

### Recommendation

In `receive_erc20_tokens`, record the NEP-141 balance of the Aurora contract before and after the transfer and use the actual delta as the mint amount, rejecting the transfer if the delta does not match `args.amount`. Since Aurora operates on NEAR and cannot directly query NEP-141 balances synchronously within `ft_on_transfer`, the practical fix is to **reject fee-on-transfer tokens at registration time** or to restructure the bridge flow to use a two-step deposit with a balance-check callback (similar to the recommendation in the external report for `L1NativeTokenVault`).

At minimum, add a guard in `ft_on_transfer` that returns the full `args.amount` (signaling refusal) whenever the predecessor NEP-141 token is not on an explicit allowlist of verified non-fee tokens, preventing unverified tokens from being silently over-minted.

### Proof of Concept

1. Deploy a NEP-141 token on NEAR that implements `ft_transfer_call` to deduct 10% from the receiver's credited balance but passes the original `amount` to `ft_on_transfer`.
2. Call `deploy_erc20_token` on Aurora for this token.
3. Mint 1000 units to attacker's NEAR account.
4. Call `ft_transfer_call` with `amount = 1000`, targeting Aurora.
5. Aurora receives 900 NEP-141 tokens but mints 1000 ERC-20 tokens.
6. Attacker calls `withdrawToNear` for 1000 ERC-20 tokens.
7. Aurora attempts to send 1000 NEP-141 tokens but only holds 900 — the 100-token deficit is taken from other depositors' funds.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
