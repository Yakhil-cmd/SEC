### Title
EvmErc20 V1 `withdrawToNear` Burns Tokens Without Scheduling NEAR Transfer When Engine Uses `error_refund` Feature — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20` V1's `withdrawToNear` encodes its precompile input as `[0x01 | amount_b(32) | recipient]`. When the engine is compiled with the `error_refund` feature, the `ExitToNear` precompile's `parse_input` consumes bytes `[1..21]` as a `refund_address`, shifting the remaining slice to `[amount_b[20..32] | recipient]`. The subsequent `parse_amount(&input[..32])` either panics (short recipient) or parses a garbled value that exceeds `u128::MAX` and returns `Err`. The precompile call returns `0` to the EVM, but the Solidity assembly block never checks `res`, so the transaction commits with `_burn` already executed and no NEAR transfer promise scheduled. The user's ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer.

---

### Finding Description

**Step 1 — V1 input encoding**

`EvmErc20.withdrawToNear` constructs:

```solidity
bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
// layout: [flag=0x01][amount_b[0..32]][recipient bytes]
``` [1](#0-0) 

**Step 2 — `error_refund` shifts the parse window**

With the feature enabled, `MIN_INPUT_SIZE = 21` and `parse_input` reads bytes `[1..21]` as `refund_address`, then returns `input[21..]` as the remaining slice:

```rust
#[cfg(feature = "error_refund")]
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);          // consumes first 20 bytes of amount_b
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))              // remaining = amount_b[20..32] + recipient
}
``` [2](#0-1) 

The remaining slice is now `amount_b[20..32]` (12 bytes) concatenated with `recipient`.

**Step 3 — `parse_amount` receives garbled data**

In the `0x1` branch:

```rust
0x1 => {
    let amount = parse_amount(&input[..32])?;   // input = [amount_b[20..32] | recipient[0..20]]
    let Recipient { receiver_account_id, .. } = parse_recipient(&input[32..])?;
``` [3](#0-2) 

- **Short recipient (< 20 bytes):** `input[..32]` panics (out-of-bounds), causing a WASM trap and the entire NEAR transaction to abort.
- **Recipient ≥ 20 bytes (common for real NEAR accounts):** `parse_amount` receives `[amount_b[20..32] | recipient[0..20]]`. The 20 ASCII bytes of the account ID make the resulting U256 far exceed `u128::MAX`, so `parse_amount` returns `Err(ExitError::Other("ERR_INVALID_AMOUNT"))`.

**Step 4 — Solidity silently ignores the failure**

The precompile error causes the EVM `call` to return `0`, but the assembly block never checks `res`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    // res is captured but never tested — no revert on failure
}
``` [4](#0-3) 

`_burn` already committed. No promise is scheduled. The transaction succeeds.

**Step 5 — No `error_refund` callback saves the user**

The `error_refund` callback (`exit_to_near_precompile_callback`) only fires when a promise was scheduled and then failed. Because the precompile itself failed before scheduling any promise, no callback is ever registered and no refund occurs. [5](#0-4) 

---

### Impact Explanation

The user's ERC-20 tokens are burned (removed from EVM state). The corresponding NEP-141 tokens remain locked inside the Aurora contract with no ERC-20 balance to back them. There is no automatic recovery path. This constitutes **permanent freezing of funds** (Critical), or at minimum **temporary freezing** (High) pending manual admin intervention to re-mint ERC-20 tokens.

---

### Likelihood Explanation

- Any V1 `EvmErc20` contract deployed before the `error_refund` feature was introduced is affected whenever the engine is upgraded to include that feature.
- The attack requires no privilege — any token holder can trigger it by calling `withdrawToNear` with a recipient ≥ 20 characters (the normal case for NEAR account IDs).
- The `error_refund` feature is present in both `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, indicating it is a production feature flag. [6](#0-5) 

---

### Recommendation

1. **EvmErc20 V1 Solidity fix:** Check the return value of the precompile `call` and revert on failure:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Precompile versioning:** The `error_refund` feature changes the wire format. The precompile should either (a) maintain backward compatibility with the V1 format, or (b) enforce a migration of all V1 contracts before enabling the feature.
3. **Bounds check before slice:** Add an explicit length check before `input[..32]` in the `0x1` branch to return a clean `Err` instead of panicking.

### Proof of Concept

Differential test (pseudo-code):

```rust
// Input as produced by EvmErc20 V1 withdrawToNear
let amount = U256::from(1_000_000u64);
let recipient = b"alice.near"; // 10 bytes — short path → panic/trap

let recipient_long = b"e523efec9b66c4c8f6e708f1cf56be1399181e5b7c1e35f845670429faf143c2"; // 64 bytes

let input_v1 = [&[0x01u8], amount.to_big_endian().as_slice(), recipient_long].concat();

// Without error_refund: parse_input returns &input[1..], amount parsed correctly → Ok
// With error_refund:    parse_input returns &input[21..] = [amount_b[20..32] | recipient_long]
//                       parse_amount(&input[..32]) parses garbled bytes → ERR_INVALID_AMOUNT → Err
//
// In both cases the Solidity assembly ignores res=0, but only with error_refund
// does _burn succeed while no promise is scheduled.

#[cfg(feature = "error_refund")]
assert!(ExitToNearParams::try_from(input_v1.as_slice()).is_err());

#[cfg(not(feature = "error_refund"))]
assert!(ExitToNearParams::try_from(input_v1.as_slice()).is_ok());
``` [7](#0-6)

        // The first byte of the input is a flag, selecting the behavior to be triggered:
        // 0x00 -> Eth(base) token withdrawal
        // 0x01 -> ERC-20 token withdrawal
        let flag = input
            .first()
            .copied()
            .ok_or_else(|| ExitError::Other(Cow::from("ERR_MISSING_FLAG")))?;

        #[cfg(feature = "error_refund")]
        let (refund_address, input) = parse_input(input)?;
        #[cfg(not(feature = "error_refund"))]
        let input = parse_input(input)?;

        match flag {
            0x0 => {
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(input)?;

                Ok(Self::BaseToken(BaseTokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    message,
                }))
            }
            0x1 => {
                let amount = parse_amount(&input[..32])?;
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(&input[32..])?;

                Ok(Self::Erc20TokenParams(Erc20TokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    amount,
                    message,
                }))
            }
            _ => Err(ExitError::Other(Cow::from("ERR_INVALID_FLAG"))),
        }
