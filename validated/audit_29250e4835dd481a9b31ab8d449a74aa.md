### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn user tokens **before** calling the exit precompile via inline assembly, but never check whether the precompile call succeeded. If the precompile reverts for any reason (invalid recipient, input too large, unregistered token, etc.), the ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer to NEAR or ETH withdrawal — a silent, unrecoverable fund loss.

---

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` follow the same pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure if precompile returns 0
    }
}
``` [1](#0-0) [2](#0-1) 

The same unchecked pattern appears in `EvmErc20V2.sol`: [3](#0-2) 

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) has multiple failure paths that return `ExitError`, causing the EVM `call` opcode to return `0`:

- `ERR_INVALID_INPUT` — input too small or too large (enforced by `validate_input_size`)
- `ERR_TARGET_TOKEN_NOT_FOUND` — ERC-20 not registered in the NEP-141 map
- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — malformed NEAR account ID in `recipient`
- `ERR_KEY_NOT_FOUND` — ETH connector account not set in storage [4](#0-3) 

When any of these errors occur, the `call` opcode returns `0` into `res`, but `res` is never tested. Solidity does not automatically revert on a failed low-level `call`. The function returns successfully, the burn is committed, and the user's tokens are gone.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

A user who calls `withdrawToNear` with a recipient string that is too long (> `MAX_INPUT_SIZE = 1024` bytes after encoding), contains invalid UTF-8, or is not a valid NEAR account ID will have their ERC-20 tokens burned with no corresponding NEP-141 credit on NEAR. The loss is irreversible: there is no refund path, no callback, and no admin recovery mechanism in the contract.

The same applies to `withdrawToEthereum` via the `ExitToEthereum` precompile at `0xb0bd02f6...`. [5](#0-4) 

---

### Likelihood Explanation

**High.** Any unprivileged EVM user can trigger this by calling `withdrawToNear` with a recipient byte string that fails precompile validation (e.g., a string longer than `MAX_INPUT_SIZE`, or one that does not parse as a valid NEAR account ID). No special permissions are required — `withdrawToNear` is `external` with no access control. The `_burn` executes unconditionally before the precompile call, so the loss occurs on the very first failed attempt. [6](#0-5) 

---

### Recommendation

Check the return value of the assembly `call` and revert on failure. Because Solidity reverts unwind all state changes in the same transaction, the `_burn` will also be rolled back:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora (as the engine does for any bridged NEP-141 token).
2. Mint tokens to an attacker-controlled address.
3. Call `withdrawToNear(recipient, amount)` where `recipient` is a byte string of length > 1024 (exceeding `MAX_INPUT_SIZE`).
4. The `ExitToNear` precompile returns `ERR_INVALID_INPUT`; the EVM `call` returns `0`.
5. `res` is never checked; the transaction succeeds.
6. The user's ERC-20 balance is zero; no NEP-141 tokens arrive on NEAR.
7. Tokens are permanently lost. [7](#0-6) [1](#0-0)

fn validate_input_size(input: &[u8], min: usize, max: usize) -> Result<(), ExitError> {
    if input.len() < min || input.len() > max {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_INPUT")));
    }
    Ok(())
}

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
