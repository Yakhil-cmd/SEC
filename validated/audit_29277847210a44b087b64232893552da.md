### Title
Unchecked Index Access on Empty `Vec<EnvelopePair>` Causes Panic in `/construction/parse` — (`rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

### Summary
An unprivileged HTTP client can POST a crafted CBOR-hex blob to `/construction/parse` with `signed=true`, encoding a `SignedTransaction` whose `requests` vector contains a `(RequestType, Vec<EnvelopePair>)` entry with an **empty** `Vec<EnvelopePair>`. The handler unconditionally accesses `updates[0]` without a length guard, triggering a Rust index-out-of-bounds panic.

### Finding Description

The deserialization path in `TryFrom<ConstructionParseRequest> for ParsedTransaction` accepts any structurally valid CBOR blob:

```rust
serde_cbor::from_slice(&from_hex(&value.transaction)?).map_err(|e| {
    ApiError::invalid_request(format!("Could not decode signed transaction: {e}"))
})?
``` [4](#0-3) 

`serde_cbor` will happily deserialize `{ "requests": [["Send", []]] }` into `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` — the empty inner vec is structurally valid CBOR. No post-deserialization invariant check enforces `updates.len() >= 1`.

Immediately after, the handler maps over `requests` and blindly indexes:

```rust
|(request_type, updates)| match updates[0].update.content.clone() {
``` [5](#0-4) 

When `updates` is empty, this is an unconditional Rust panic (`index out of bounds: the len is 0 but the index is 0`).

### Impact Explanation

The Rosetta server is a standalone HTTP process separate from the IC replica. A panic in a request handler thread will, depending on the HTTP framework's panic-catching configuration, either:
- Return a 500 and restart the worker thread (if `catch_unwind` is active), or
- Crash the Rosetta server process entirely (if not).

Either way, the invariant that all public HTTP endpoints must handle malformed input without panicking is violated. Repeated requests can cause persistent worker exhaustion or process termination, constituting a denial-of-service against the Rosetta API.

### Likelihood Explanation

The exploit requires no authentication, no privileged role, and no special knowledge beyond the CBOR encoding of `SignedTransaction`. The `transaction` field is an arbitrary hex string accepted from any HTTP client. The path is fully local-testable.

### Recommendation

Add an explicit length guard before the index access:

```rust
.map(|(request_type, updates)| {
    if updates.is_empty() {
        return Err(ApiError::invalid_request(
            "SignedTransaction contains a request with no envelope pairs".to_string()
        ));
    }
    match updates[0].update.content.clone() {
        HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
    }
})
.collect::<Result<Vec<_>, _>>()?
```
