### Title
Silo Whitelist Enforcement Bypass via `call` and `deploy_code` Entrypoints — (File: `engine/src/contract_methods/evm_transactions.rs`)

### Summary
Aurora Engine's Silo mode enforces address and account whitelists to restrict who may submit EVM transactions and deploy EVM contracts. This enforcement is applied inside `engine::submit` via `assert_access`, but the parallel `call` and `deploy_code` NEAR-level entrypoints invoke EVM execution directly without ever calling `assert_access`. Any NEAR account — including one explicitly excluded from the Silo whitelist — can bypass the whitelist entirely by using `call` instead of `submit`.

### Finding Description

The Silo whitelist check lives in `assert_access`:

```rust
// engine/src/engine.rs  lines 1756-1775
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}

This function is only reachable through the `submit` / `submit_with_args` path (which parses a signed Ethereum transaction and then calls `assert_access` before execution).

The `call` entrypoint, however, does none of this:

### Impact Explanation

**High — Temporary (and potentially permanent) freezing of funds / theft of unclaimed yield; in the worst case, direct fund theft.**

In Silo mode the operator deploys Aurora as a permissioned EVM where only whitelisted NEAR accounts and EVM addresses may transact. The whitelist is the primary access-control boundary. By calling `call` directly, a non-whitelisted NEAR account can:

1. **Execute arbitrary EVM calls** against any deployed contract — including ERC-20 `transfer`, `approve`, or any other state-mutating function — on behalf of the EVM address derived from their NEAR account ID (`predecessor_address`). This can drain ERC-20 balances held by the engine.
2. **Deploy arbitrary EVM bytecode** via `deploy_code`, bypassing the `Admin`/`EvmAdmin` whitelist entirely.
3. **Interact with the bridge connector** (e.g., trigger `exit_to_near` or `exit_to_ethereum` precompiles) to move funds out of the EVM without being whitelisted.

The impact is **direct theft of user funds** (ERC-20 tokens held in the EVM) and **unauthorized contract deployment**, both of which are in scope.

### Likelihood Explanation

**High.** The `call` entrypoint is a standard, documented public NEAR function. Any NEAR account can call it with no special privileges. The only precondition is knowing the address of a target ERC-20 contract and the ABI encoding for the desired call. This is trivially discoverable from on-chain data. No leaked keys, no governance capture, and no admin access are required.

### Recommendation

Apply the same silo access check inside `call` and `deploy_code` before dispatching to the EVM executor. Specifically:

- In `call`: derive the EVM sender address from `predecessor_account_id`, extract the target contract from `CallArgs`, and call `silo::is_allow_submit` (or an equivalent check) before `engine.call_with_args`.
- In `deploy_code`: call `silo::is_allow_deploy` before `engine.deploy_code_with_input`.

The check should mirror the logic already present in `assert_access` so that all EVM-execution entrypoints enforce the same policy.

### Proof of Concept

**Setup:** Silo mode is active; `Address` and `Account` whitelists are enabled; attacker's NEAR account `attacker.near` and its derived EVM address are **not** whitelisted.

**Step 1 — Confirm `submit` is blocked:**
```
near call aurora submit <signed_tx> --accountId attacker.near
# → ERR_NOT_ALLOWED  (whitelist enforced)
```

**Step 2 — Bypass via `call`:**
```
# ABI-encode: ERC20.transfer(victim_address, amount)
near call aurora call \
  '{"contract": "<erc20_address>", "value": "0x00...", "input": "<transfer_calldata>"}' \
  --accountId attacker.near
# → TransactionStatus::Succeed  (whitelist NOT enforced)
```

The ERC-20 `transfer` executes with `msg.sender` = `predecessor_address("attacker.near")`. If the attacker holds any ERC-20 balance at that derived address, they can transfer it to any recipient. More critically, if the attacker can front-run a deposit (where tokens are minted to a predictable address), they can immediately drain those tokens before the legitimate owner acts — all while being blocked from `submit`.
