Audit Report

## Title
Unbounded `ingress_expiries` Loop in ICP Rosetta `construction_payloads` Causes OOM Process Crash — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from the JSON request body with no range validation, then iterates from `ingress_start` to `ingress_end` in ~120-second steps, pushing a `u64` into a heap-allocated `Vec` on every iteration. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes the process to attempt allocating ~1.2 TB of memory and be killed by the OS OOM killer. The ICRC1 Rosetta handler has ordering and staleness guards before its equivalent loop; the ICP handler has none.

## Finding Description
**Root cause:** `ingress_start` and `ingress_end` are deserialized directly from the client-supplied JSON metadata with no upper-bound or range check:

```rust
// rs/rosetta-api/icp/src/request_handler/construction_payloads.rs L74-84
let ingress_start = meta.as_ref()
    .and_then(|meta| meta.ingress_start)
    .map(ic_types::time::Time::from_nanos_since_unix_epoch)
    .unwrap_or_else(ic_types::time::current_time);

let ingress_end = meta.as_ref()
    .and_then(|meta| meta.ingress_end)
    .map(ic_types::time::Time::from_nanos_since_unix_epoch)
    .unwrap_or_else(|| ingress_start + interval);
``` [1](#0-0) 

`ConstructionPayloadsRequestMetadata::try_from` is a plain `serde_json::from_value` with no numeric bounds checking: [2](#0-1) 

The step size `interval` is fixed at `MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s = 120_000_000_000 ns`: [3](#0-2) 

The unbounded loop then runs without any guard:

```rust
// L99-107
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    let ingress_expiry = (now
        + ic_limits::MAX_INGRESS_TTL.saturating_sub(ic_limits::PERMITTED_DRIFT))
    .as_nanos_since_unix_epoch();
    ingress_expiries.push(ingress_expiry);
    now += interval;
}
``` [4](#0-3) 

**Why existing checks fail:** There are none. The ICRC1 handler explicitly rejects invalid windows before entering its loop: [5](#0-4) 

The ICP handler has no analogous guard anywhere between metadata parsing and the loop entry.

**Exploit path:**
1. Attacker sends a single unauthenticated HTTP POST to `/construction/payloads` with `metadata.ingress_start=0` and `metadata.ingress_end=18446744073709551615`.
2. `ingress_start` and `ingress_end` are accepted as-is.
3. The `while now < ingress_end` loop runs ≈153 billion iterations, each pushing 8 bytes onto `ingress_expiries`.
4. The process attempts to allocate ~1.2 TB; the OS OOM killer terminates it before a response is returned.

## Impact Explanation
This is a single-request, non-volumetric, unauthenticated denial of service that crashes the ICP Rosetta process. The Rosetta API is an in-scope financial integration component. A crashed Rosetta node is unavailable for all users until restarted, and the attack is trivially repeatable to prevent recovery. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS / significant Rosetta infrastructure security impact with concrete user harm.**

## Likelihood Explanation
The `/construction/payloads` endpoint is a public, unauthenticated HTTP POST endpoint. No credentials, API keys, or privileged access are required. The attacker needs only a single crafted JSON request. The exploit is deterministic, reproducible, and requires no special tooling beyond `curl`. Any internet-accessible ICP Rosetta deployment is affected.

## Recommendation
Add an upper-bound guard on the ingress window before entering the loop, mirroring the ICRC1 handler pattern. For example:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // 24 hours
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end exceeds maximum allowed ingress window",
    ));
}
```

Alternatively, cap `ingress_expiries` to a fixed maximum count (e.g., 720 entries for a 24-hour window at 2-minute intervals) and return an error if the requested window would exceed it. Also add `ingress_start >= ingress_end` ordering validation consistent with the ICRC1 handler.

      {"operation_identifier":{"index":1},"type":"TRANSACTION","account":{"address":"<valid_dst>"},"amount":{"value":"100000000","currency":{"symbol":"ICP","decimals":8}}}
    ],
    "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],

      "ingress_start": 0,

    }
  }'
```

Expected: the Rosetta process enters the loop at line 101, allocates memory unboundedly, and is terminated by OOM before returning a response. A unit test can confirm the missing guard by calling `construction_payloads` directly with `ingress_start=0` and `ingress_end=u64::MAX` and asserting that an `Err` is returned — currently it will not return at all.
