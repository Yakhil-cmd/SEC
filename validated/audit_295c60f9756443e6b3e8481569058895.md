Audit Report

## Title
Unbounded `ingress_expiries` Loop via Attacker-Controlled `ingress_end` Causes OOM Process Crash — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `construction_payloads` handler accepts caller-supplied `ingress_start` and `ingress_end` values from request metadata with no bounds or window-size validation. It then enters an unbounded `while now < ingress_end` loop, pushing one `u64` per ~120-second interval into a heap-allocated `Vec`. An unauthenticated attacker can supply `ingress_start=0` and `ingress_end=u64::MAX` to force the Rosetta process to attempt allocating ~1.2 TB of memory, triggering OOM and process termination via a single HTTP POST.

## Finding Description
The interval is fixed at approximately 120 seconds of nanoseconds: [1](#0-0) 

`ingress_start` and `ingress_end` are taken directly from the caller-supplied metadata with no range, sanity, or maximum-window-size validation: [2](#0-1) 

The loop then runs without any bound: [3](#0-2) 

With `ingress_start=0` and `ingress_end=u64::MAX` (18446744073709551615 ns), the iteration count is `u64::MAX / (120 × 10⁹) ≈ 153,722,867,280`. Each iteration pushes an 8-byte `u64` onto the `Vec`, totaling ~1.2 TB of allocation before the OOM killer terminates the process.

The `ConstructionPayloadsRequestMetadata::try_from` path is pure JSON deserialization with no semantic validation: [4](#0-3) 

By contrast, the ICRC1 handler has pre-loop guards that at minimum reject `ingress_start >= ingress_end` and windows entirely in the past: [5](#0-4) 

The ICP handler has no equivalent guards whatsoever — not even the `ingress_start >= ingress_end` sanity check.

## Impact Explanation
A single unauthenticated HTTP POST to `/construction/payloads` causes the ICP Rosetta process to exhaust system memory and be killed by the OOM killer. This takes the Rosetta API entirely offline until manually restarted. The Rosetta API is the primary programmatic interface for exchanges and custodians integrating ICP, making its unavailability a concrete, significant harm to users and protocol participants. This matches the allowed High impact: **"Significant... Rosetta... security impact with concrete user or protocol harm."**

## Likelihood Explanation
The endpoint requires no authentication. The attack is a single standard JSON POST with two integer fields set to extreme values. No prior knowledge beyond the public Rosetta API spec is required. The attack is trivially reproducible, repeatable after each restart, and requires no special tooling. There is no rate limiting, request-size guard, or window-size cap visible anywhere in the handler path.

## Recommendation
Add a maximum ingress window size check before the loop in the ICP handler, mirroring the ICRC1 pattern. For example:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // 24 hours

if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(
        "ingress_start must be before ingress_end",
    ));
}
if ingress_end.as_nanos_since_unix_epoch().saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    > MAX_INGRESS_WINDOW.as_nanos() as u64
{
    return Err(ApiError::invalid_request(
        "ingress_end exceeds maximum allowed ingress window of 24 hours",
    ));
}
```

This caps the loop to at most `24 * 3600 / 120 = 720` iterations, which is safe.

## Proof of Concept
```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<valid_account>"},"amount":{"value":"-100000000","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

Expected: Rosetta process RSS grows unboundedly (~1.2 TB attempted allocation) and is killed by the OOM killer. A unit test can confirm the behavior by calling `construction_payloads` directly with these metadata values and observing either OOM or, after the fix, an `ApiError::invalid_request` response.
