Audit Report

## Title
Unprivileged Semaphore Exhaustion via Concurrent `check_transaction` Calls Causes Sustained HighLoad DoS Against ckBTC Minter — (`rs/bitcoin/checker/src/state.rs`)

## Summary
The ckBTC checker canister uses a single global `OUTCALL_CAPACITY` semaphore (capacity = 50) to gate HTTP outcalls in `check_transaction`. There is no per-caller rate limiting and no reserved capacity for the ckBTC minter principal. An unprivileged attacker can submit 50 concurrent `check_transaction` calls with distinct TXIDs, exhausting the semaphore while all 50 HTTP outcalls are in-flight, causing every subsequent call — including those from the ckBTC minter — to return `CheckTransactionRetriable::HighLoad` and halting ckBTC deposit processing for the duration of the attack.

## Finding Description
`OUTCALL_CAPACITY` is a `thread_local! RefCell<u32>` initialized to `MAX_CONCURRENT = 50` in `state.rs` (L116–133). `FetchGuard::new` (L244–258) decrements this counter for any caller that passes the cycles check, with no check on `ic_cdk::api::caller()` and no per-principal in-flight counter. The guard is only released in `FetchGuard::Drop` (L260–272) when the HTTP outcall completes or the guard is dropped.

In `try_fetch_tx` (`fetch.rs` L93–96), when `new_fetch_guard` returns `Err(FetchGuardError::NoCapacity)`, the function immediately returns `TryFetchResult::HighLoad` — no retry, no queue. `check_transaction_inputs` (`main.rs` L543–544) maps this directly to `CheckTransactionRetriable::HighLoad` on the wire.

`check_transaction_with` (`main.rs` L144–168) performs no caller identity check and no per-caller rate limiting — only a cycles balance check. The `check_transaction` update method is public and callable by any principal.

**Exploit path:**
1. Attacker submits 50 `check_transaction` calls, each with a distinct valid Bitcoin TXID and ≥ 40 B cycles attached.
2. Each call enters `check_transaction_inputs` → `try_fetch_tx` → `FetchGuard::new`, decrementing `OUTCALL_CAPACITY` from 50 → 0 and setting each TXID's status to `PendingOutcall`.
3. Each call then awaits `http_get_tx`. On the IC, awaiting an HTTP outcall yields execution, so all 50 calls are simultaneously suspended mid-execution with their `FetchGuard`s live.
4. `OUTCALL_CAPACITY` = 0. Any new call to `try_fetch_tx` for a TXID not already in `PendingOutcall` hits `Err(FetchGuardError::NoCapacity)` → `TryFetchResult::HighLoad`.
5. The ckBTC minter's `check_transaction` call returns `CheckTransactionRetriable::HighLoad`, causing `update_balance` to treat the check as temporarily unavailable and halt deposit processing.
6. HTTP outcalls on the IC have a timeout (~30 s). The attacker submits a fresh batch of 50 calls before the previous batch times out, sustaining the DoS indefinitely.

The TXID deduplication in `try_fetch_tx` (L84) returns `Pending` — not `HighLoad` — for a TXID already in `PendingOutcall`, so the attacker must use 50 distinct TXIDs. This is trivially achievable given Bitcoin's transaction history.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

The ckBTC minter's `update_balance` flow calls `check_transaction` on the checker canister. While `OUTCALL_CAPACITY` = 0, every such call returns `HighLoad`, and the minter cannot complete deposit checks for any user. This is a targeted resource exhaustion attack exploiting a specific design flaw (no per-caller rate limiting, no reserved capacity for the minter), not a raw volumetric DDoS. The ckBTC checker is a production financial integration component explicitly in scope.

## Likelihood Explanation
- Requires no privileged access — `check_transaction` is a public update method callable by any principal.
- Requires only 50 concurrent calls with valid TXIDs and ≥ 40 B cycles each, achievable from a single principal.
- No per-caller rate limiting, no reserved capacity for the ckBTC minter, no caller allowlist.
- The IC's message queue per canister accommodates 50 concurrent in-flight update calls.
- Cost to attacker: 50 × 40 B = 2 T cycles per ~30 s window (~$2–3 USD/30 s at current prices). Non-trivial but not prohibitive for a motivated attacker with financial incentive to disrupt ckBTC deposits.
- Likelihood: **Medium-High**.

## Recommendation
1. **Reserve capacity for the ckBTC minter**: Check `ic_cdk::api::caller()` in `check_transaction_with` and reserve a portion of `OUTCALL_CAPACITY` exclusively for the known ckBTC minter principal, so it can never be starved by unprivileged callers.
2. **Per-caller rate limiting**: Track in-flight outcall count per caller principal in a `BTreeMap<Principal, u32>` stored in `thread_local!` state and reject calls that exceed a per-caller cap (e.g., 5 concurrent calls per principal).
3. **Reduce per-caller share of `MAX_CONCURRENT`**: Even without identity checks, a simple per-principal counter would prevent a single caller from consuming all 50 slots.

impl FetchGuard {
    pub fn new(txid: Txid) -> Result<Self, FetchGuardError> {
        let guard = OUTCALL_CAPACITY.with(|capacity| {

            if *capacity > 0 {

                Ok(FetchGuard(txid))
