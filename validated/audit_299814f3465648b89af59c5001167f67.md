Audit Report

## Title
Unbounded Loop in `construction_payloads` via Attacker-Controlled `ingress_end` Causes OOM/CPU Exhaustion — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC-1 Rosetta API accepts attacker-controlled `ingress_start` and `ingress_end` values from the unauthenticated POST `/construction/payloads` endpoint with no upper-bound cap on `ingress_end`. Setting `ingress_end = u64::MAX` causes the loop to execute approximately 139 million iterations, allocating ~1.1 GB of heap memory and exhausting CPU, crashing or hanging the Rosetta process.

## Finding Description
In `rs/rosetta-api/icrc1/src/construction_api/services.rs` at lines 162–167, the loop that builds `ingress_expiries` is:

```rust
let mut ingress_expiries = vec![];
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
```

The step per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP`. From confirmed constants:
- `MAX_INGRESS_TTL = Duration::from_secs(5 * 60)` (`rs/limits/src/lib.rs`, line 17)
- `PERMITTED_DRIFT = Duration::from_secs(60)` (`rs/limits/src/lib.rs`, line 21)
- `ingress_interval = (300s − 60s) = 240s = 240,000,000,000 ns` (`services.rs`, lines 120–121)
- `INGRESS_INTERVAL_OVERLAP = Duration::from_secs(120) = 120,000,000,000 ns` (`constants.rs`, line 19)
- **Effective step = 120,000,000,000 ns per iteration**

The only guards before the loop (lines 148–158) are:
1. `if ingress_start >= ingress_end` — trivially bypassed by setting `ingress_start = now` (< `u64::MAX`)
2. `if ingress_end < now + ingress_interval` — trivially bypassed by setting `ingress_end = u64::MAX`

With `ingress_end = u64::MAX ≈ 1.84 × 10^19` and `now ≈ 1.7 × 10^18 ns`:
- Range ≈ 1.67 × 10^19 ns
- Iterations ≈ 1.67 × 10^19 / 1.2 × 10^11 ≈ **~139 million**
- Memory: 139M × 8 bytes = **~1.1 GB**

The endpoint handler in `rs/rosetta-api/icrc1/src/construction_api/endpoints.rs` (lines 90–107) calls `services::construction_payloads(...)` synchronously with no `spawn_blocking` or async yield points, blocking the Axum worker thread for the duration of the loop. The route is registered at `main.rs` line 380 with no authentication middleware.

## Impact Explanation
A single unauthenticated HTTP POST to `/construction/payloads` with `ingress_end` near `u64::MAX` causes OOM or CPU exhaustion in the Rosetta process, crashing or hanging it. This halts block synchronization and transaction submission for all ICRC-1 tokens served by that Rosetta instance. This matches the **High ($2,000–$10,000)** impact: "Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS" and "Significant Rosetta security impact with concrete user or protocol harm."

## Likelihood Explanation
The endpoint is public, requires no credentials, and the malicious field is a plain JSON integer. The attack requires a single HTTP request from any network-reachable client. No special privileges, victim interaction, or external dependencies are required. The attack is trivially repeatable.

## Recommendation
Add an explicit cap on the number of generated expiries before the loop:

```rust
const MAX_INGRESS_EXPIRIES: usize = 100; // ~200 minutes of coverage
let range = ingress_end.saturating_sub(ingress_start);
let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
if step > 0 && (range / step) as usize > MAX_INGRESS_EXPIRIES {
    return Err(Error::processing_construction_failed(
        &"ingress window too large"
    ));
}
```

Alternatively, validate `ingress_end <= now + MAX_ALLOWED_WINDOW` before entering the loop, where `MAX_ALLOWED_WINDOW` is a reasonable constant (e.g., a few hours in nanoseconds).
