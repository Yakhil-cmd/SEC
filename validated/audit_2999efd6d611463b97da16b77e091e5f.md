### Title
Unchecked Precompile Call Return Value Causes Silent Token Burn Without Bridge Transfer - (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` burn the caller's ERC-20 tokens **before** calling the Aurora exit precompile via inline assembly. The `res` value returned by the EVM `call` opcode is captured but **never checked**. If the precompile call fails (returns 0), the burn is not reverted and the user permanently loses their tokens with no corresponding NEP-141 transfer on NEAR.

---

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER CHECKED — no `if iszero(res) { revert(0,0) }`
    }
}
```

The same unchecked pattern appears in `EvmErc20V2.sol` for both `withdrawToNear` and `withdrawToEthereum`.

The EVM `call` opcode returns `0` on failure. The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` (`exitToNear`) and `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` (`exitToEthereum`) can fail for several reasons:

1. **The precompile is paused** — Aurora's precompile set checks `is_paused` and returns `PrecompileFailure::Fatal` when paused, which causes the EVM `call` to return 0.
2. **Out of gas** — the precompile returns `ExitError::OutOfGas`.
3. **Invalid input** — e.g., a malformed or too-long recipient account ID causes the precompile to return an `ExitError`.

In all these cases, the EVM `call` returns `res = 0`, but since `res` is never checked and there is no `if iszero(res) { revert(0,0) }`, the function returns successfully. The `_burn` that already executed is **not rolled back**, because it occurred in the caller's own execution frame before the `call` opcode.

The `error_refund` feature in the engine handles the case where the downstream NEAR-side promise (e.g., `ft_transfer`) fails after the precompile succeeds — but it provides **no protection** when the precompile call itself fails at the EVM level, because in that case no promise is ever scheduled. ---

### Impact Explanation

A user calling `withdrawToNear` or `withdrawToEthereum` on a deployed `EvmErc20` or `EvmErc20V2` token contract will have their ERC-20 tokens permanently burned with no corresponding NEP-141 tokens received on NEAR. The tokens are destroyed on the Aurora EVM side and never unlocked on the NEAR side. This constitutes **permanent loss of user funds** (Critical).

---

### Likelihood Explanation

The most realistic trigger is the **pause mechanism**: Aurora's precompile set can pause individual precompiles. If `exitToNear` or `exitToEthereum` is paused (e.g., during an incident response or upgrade), any user calling `withdrawToNear` or `withdrawToEthereum` during that window will silently lose their tokens. This is an externally reachable path requiring no special privileges from the attacker — any token holder can trigger it simply by calling the withdraw function while the precompile is paused. Out-of-gas conditions during congestion are a secondary trigger.

---

### Recommendation

Add a revert on failure inside the assembly block in both `EvmErc20.sol` and `EvmErc20V2.sol`, for all four affected functions:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile call fails for any reason, the entire transaction reverts, rolling back the `_burn` and preserving the user's token balance.

---

### Proof of Concept

**Step 1:** Aurora admin pauses the `exitToNear` precompile (or the precompile runs out of gas due to a tight gas limit).

**Step 2:** User holds 1000 units of a bridged ERC-20 token (`EvmErc20`) and calls `withdrawToNear("user.near", 1000)`.

**Step 3:** `_burn(msg.sender, 1000)` executes — 1000 ERC-20 tokens are destroyed from the user's balance.

**Step 4:** The `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` fails because the precompile is paused; the EVM `call` returns `res = 0`.

**Step 5:** Since `res` is never checked, no revert occurs. The function returns successfully.

**Step 6:** The user's ERC-20 balance is now 0. No NEP-141 transfer promise was ever scheduled. The corresponding NEP-141 tokens remain locked in the Aurora engine account on NEAR with no mechanism to recover them. The user has permanently lost their funds. [6](#0-5) [7](#0-6)
