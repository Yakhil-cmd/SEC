### Title
`get_authorizer` Always Constructs Single-Owner ACL, Permanently Disabling Multi-Pauser Emergency Response — (`engine/src/engine.rs`)

---

### Summary

`get_authorizer` unconditionally builds an `EngineAuthorizer` containing only `state.owner_id`, ignoring the `acl: BTreeSet<AccountId>` field that `EngineAuthorizer` is designed to hold. No function in the codebase writes secondary pausers to storage, and `get_authorizer` never reads from storage for ACL data. The result is that `pause_precompiles` is permanently restricted to the owner account, making the multi-pauser ACL model non-functional.

---

### Finding Description

`EngineAuthorizer` is declared with a `pub acl: BTreeSet<AccountId>` field explicitly documented as "List of `AccountId`s with the permission to pause precompiles." [1](#0-0) 

`get_authorizer` ignores this storage-backed ACL entirely. It always constructs a fresh `EngineAuthorizer` from a single-element iterator over `state.owner_id`. The inline TODO comment acknowledges this is an incomplete placeholder:

```rust
// TODO: a temporary use the owner account only until the engine adapts std with near-plugins
state::get_state(io)
    .map(|state| EngineAuthorizer::from_accounts(once(state.owner_id)))
    .unwrap_or_default()
``` [2](#0-1) 

`pause_precompiles` calls `engine::get_authorizer` and rejects any caller not in the returned ACL with `ERR_UNAUTHORIZED`: [3](#0-2) 

A grep across the entire repository confirms there is **no** function named `add_pauser`, `set_authorizer`, `update_authorizer`, or any variant that writes secondary accounts into an ACL storage key. The `acl` field is populated only by `from_accounts`, which is only ever called by `get_authorizer` with `once(owner_id)`. The `BTreeSet` is structurally dead for any account other than the owner.

`is_authorized` simply checks `self.acl.contains(account)`: [4](#0-3) 

---

### Impact Explanation

Any operator or protocol design that relies on a secondary pauser account (e.g., a dedicated incident-response key, a multisig guardian, or an automated circuit-breaker) to call `pause_precompiles` in an emergency will receive `ERR_UNAUTHORIZED`. The only account that can pause precompiles is the engine owner. If the owner key is unavailable — due to multisig quorum delay, key rotation, or operational unavailability — precompile exploits (e.g., a bridge exit vulnerability) cannot be halted, allowing continued fund extraction until the owner acts. This directly maps to **High: Temporary freezing of funds** (the protective freeze cannot be triggered by authorized responders, extending the window of active exploitation).

---

### Likelihood Explanation

The likelihood is high in any deployment that has configured or documented secondary pausers. The TODO comment in production code confirms the multi-pauser path was intentionally deferred but never completed. Any operator who reads `EngineAuthorizer.acl`'s documentation and assumes secondary pausers are functional will be wrong. The flaw requires no attacker action — it is a passive, always-present restriction.

---

### Recommendation

Implement a storage-backed ACL for `EngineAuthorizer`:

1. Define a storage key (e.g., under `KeyPrefix::Config`) to persist the authorized pauser set.
2. Add an admin-only function (callable by the owner) to add/remove accounts from the ACL.
3. Rewrite `get_authorizer` to read the persisted ACL from storage instead of constructing it from `once(owner_id)`.
4. Ensure `pause_precompiles` continues to use `get_authorizer` — no changes needed there once the underlying function is fixed.
5. Remove the TODO comment and add an integration test verifying a non-owner account in the ACL can successfully call `pause_precompiles`.

---

### Proof of Concept

```rust
// Demonstrates that a non-owner account in the intended ACL is rejected by pause_precompiles.
// Run against unmodified engine code.

#[test]
fn secondary_pauser_is_always_rejected() {
    // Setup: owner = "owner.near", secondary pauser = "pauser.near"
    let owner_id: AccountId = "owner.near".parse().unwrap();
    let secondary_pauser: AccountId = "pauser.near".parse().unwrap();

    // get_authorizer always returns EngineAuthorizer with only owner_id
    // Simulate what get_authorizer does:
    let authorizer = EngineAuthorizer::from_accounts(once(owner_id.clone()));

    // secondary_pauser is NOT in the ACL even though the struct supports it
    assert!(
        !authorizer.is_authorized(&secondary_pauser),
        "secondary pauser should be authorized but is not — ACL is owner-only"
    );

    // owner IS authorized
    assert!(authorizer.is_authorized(&owner_id));

    // Attempting to add secondary_pauser to ACL via storage is impossible:
    // no function exists to persist additional pausers, and get_authorizer
    // never reads from storage for ACL data.
}
```

This test passes on unmodified code, confirming the invariant break: `EngineAuthorizer.acl` can never contain more than `owner_id` in production, regardless of any off-chain configuration. [5](#0-4) [6](#0-5)

#[named]
pub fn pause_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        require_running(&state::get_state(&io)?)?;
        let authorizer: EngineAuthorizer = engine::get_authorizer(&io);

        if !authorizer.is_authorized(&env.predecessor_account_id()) {
            return Err(b"ERR_UNAUTHORIZED".into());
        }

        let args: PausePrecompilesCallArgs = io.read_input_borsh()?;
        let flags = PrecompileFlags::from_bits_truncate(args.paused_mask);
        let mut pauser = EnginePrecompilesPauser::from_io(io);
        pauser.pause_precompiles(flags);
        Ok(())
    })
}
