Looking at the `compute_gas_refund` function in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`, I can identify a direct analog to the external report's double-subtraction accounting error.

### Title
EVM Refund Negated by Native-Resource Delta Adjustment When Native Dominates — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the EIP-3529 EVM storage refund (`evm_refund`) is subtracted from `gas_used` **before** the native-resource delta adjustment (`delta_gas`) is computed. When native resource consumption dominates (`delta_gas > 0`), the delta is computed against the already-reduced `gas_used`, making it larger by exactly `evm_refund`. The subsequent addition of `delta_gas` silently cancels the EVM refund, leaving the user overcharged by up to `evm_refund × gas_price` tokens.

---

### Finding Description

`compute_gas_refund` executes the following sequence:

```
// Step 1 – gas consumed from ergs
gas_used = gas_limit - remaining_ergs / ERGS_PER_GAS          // G

// Step 2 – EIP-3529 refund (capped at G/5)
evm_refund = min(full_refund_gas, gas_used / 5)               // R

// Step 3 – subtract refund from gas_used  ← BUG ORIGIN
gas_used -= evm_refund                                         // G - R

// Step 4 – enforce intrinsic minimum
gas_used = max(gas_used, minimal_gas_used)

// Step 5 – native adjustment (only when !unlimited_native)
delta_gas = (native_used / native_per_gas) - gas_used         // N - (G-R)
if delta_gas > 0:
    gas_used += delta_gas                                      // → N
``` [1](#0-0) [2](#0-1) 

When `N > G - R` (native dominates), `delta_gas = N - (G - R) = N - G + R`. Adding it to `gas_used` yields:

```
final gas_used = (G - R) + (N - G + R) = N
```

The `evm_refund R` cancels out entirely. The user is charged for `N` gas instead of the correct `N - R`.

**Correct order** (apply delta before refund):

```
delta_gas = N - G          // compare native against pre-refund gas_used
if delta_gas > 0: gas_used = N
gas_used -= evm_refund     // R is always honoured → N - R
```

**Concrete example:**

| Variable | Value |
|---|---|
| G (gas from ergs) | 50,000 |
| R (evm_refund, SSTORE clears) | 10,000 |
| N (native_used / native_per_gas) | 60,000 |

- **Current code:** `gas_used = 60,000` — EVM refund lost, user overcharged 10,000 gas × gas_price
- **Correct code:** `gas_used = 50,000` — EVM refund honoured

The `evm_refund` is still reported in `RefundInfo.evm_refund` and surfaced as `gas_refunded` in the transaction result, creating a discrepancy between the reported refund and the actual tokens returned. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Any L2 (ZK) or L1→L2 transaction that:
1. Performs SSTORE operations that clear storage slots (generating EVM refunds via `refund_for_storage_write`), and
2. Has native resource consumption exceeding EVM gas consumption (e.g., pubdata-heavy writes)

will have its EVM refund silently zeroed. The user is overcharged by up to `gas_used / 5 × gas_price` tokens per transaction. The overcharge is not returned to the user and is instead retained by the operator (or burned under `burn_base_fee`). This is a direct, quantifiable loss of user funds. [5](#0-4) 

The `#[cfg(not(feature = "unlimited_native"))]` guard confirms this path is active in production (the `unlimited_native` feature is a test-only override). [6](#0-5) 

---

### Likelihood Explanation

The condition `delta_gas > 0` (native dominates) is reached whenever pubdata costs or proving costs push native consumption above EVM gas consumption. This is a normal operating condition on ZKsync, where pubdata pricing is a first-class resource. DeFi contracts that clear storage slots (e.g., DEX position closures, token approvals reset to zero, vault withdrawals) routinely generate EVM refunds. No privileged access is required; any unprivileged transaction sender can trigger this path.

---

### Recommendation

Move the `evm_refund` subtraction to **after** the native delta adjustment, and re-apply the intrinsic minimum floor afterward:

```rust
// Step 1: gas from ergs
let mut gas_used = gas_limit - remaining_ergs / ERGS_PER_GAS;

// Step 2: compute evm_refund against pre-adjustment gas_used
let evm_refund = core::cmp::min(full_refund_gas, gas_used / 5);

// Step 3: enforce intrinsic minimum (before native adjustment)
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

// Step 4: native adjustment (compare against pre-refund gas_used)
let delta_gas = (native_used / native_per_gas) as i64 - gas_used as i64;
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}

// Step 5: apply EVM refund AFTER native adjustment
gas_used = gas_used.saturating_sub(evm_refund);

// Step 6: re-enforce intrinsic minimum (refund cannot go below floor)
let gas_used = core::cmp::max(gas_used, minimal_gas_used);
``` [7](#0-6)

### Proof of Concept

1. Deploy a contract on ZKsync OS that:
   - Writes several non-zero storage slots during setup.
   - In a single call, clears those slots (SSTORE to zero), generating an EVM refund up to `gas_used / 5`.
   - Also emits enough pubdata (e.g., many storage writes) so that `native_used / native_per_gas > gas_used - evm_refund`.

2. Execute the transaction and observe:
   - `tx_result.gas_refunded` reports a non-zero `evm_refund` (e.g., 10,000 gas).
   - The actual token refund returned to the sender equals `gas_price × (gas_limit - gas_used)` where `gas_used = N` (native equivalent), not `N - evm_refund`.
   - The sender receives `evm_refund × gas_price` fewer tokens than entitled.

3. Repeat with a transaction that has the same SSTORE pattern but low pubdata (so `delta_gas ≤ 0`). Observe that the EVM refund is correctly applied in this case, confirming the bug is conditional on native dominance. [8](#0-7)
