### Title
Division Before Multiplication in `native_per_pubdata` Calculation Truncates to Zero, Disabling Pubdata Native Resource Charging - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the `native_per_pubdata` rate — which controls how much native resource is charged per byte of pubdata generated — is computed via a truncating integer division of `pubdata_price / native_price`. When `pubdata_price < native_price`, this division truncates to zero, completely disabling native resource charging for pubdata. Any unprivileged transaction sender can then generate pubdata (e.g., by writing to storage slots) without consuming any native resource budget, while the operator bears the actual pubdata publication cost uncompensated.

---

### Finding Description

In `validation_impl.rs`, the ZK transaction validation path computes:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This is a truncating integer division. The result `native_per_pubdata` is then used in two downstream multiplications:

1. **Intrinsic pubdata overhead** (charged at transaction start):
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
``` [2](#0-1) 

2. **Execution pubdata charging** (charged after execution):
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

The correct formula — to avoid precision loss — should multiply first and divide last:
```
native_charged = pubdata_price * pubdata_bytes / native_price
```

Instead, the division is performed first:
```
native_per_pubdata = pubdata_price / native_price   ← truncates
native_charged     = native_per_pubdata * pubdata_bytes
```

When `pubdata_price < native_price`, `native_per_pubdata` truncates to `0`. Both downstream multiplications then produce `0`, meaning **no native resource is charged for any pubdata generated by the transaction**, regardless of how many storage writes occur.

The same truncating division is also present in the off-chain pre-validation helper `validate_l2_tx_intrinsic_native_resources`:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

This means the off-chain check also passes without error when `native_per_pubdata = 0`, so transactions that will generate unlimited pubdata for free are not rejected at the API layer either.

---

### Impact Explanation

When `pubdata_price < native_price`:

- `native_per_pubdata = 0`
- `intrinsic_pubdata_overhead = 0` → no native deducted from the native limit for intrinsic pubdata
- `get_resources_to_charge_for_pubdata` returns `0` native → no native deducted for execution pubdata
- The transaction's full native limit remains available for computation, regardless of how much pubdata is generated

A user can write to many storage slots (each write generates pubdata) and exhaust the block's pubdata budget without spending any native resource. The operator must still publish this pubdata to L1 and bears the cost without receiving native resource compensation. This is a **resource accounting bug** that breaks the economic invariant that pubdata costs are borne by the transaction sender.

Even when `pubdata_price >= native_price`, the truncation still causes systematic underpayment. For example, with `pubdata_price = 15` and `native_price = 10`, `native_per_pubdata = 1` instead of the correct `1.5`, meaning users pay 33% less native for pubdata than they should.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is not enforced anywhere in the codebase. The only check is that `native_price != 0`:

```rust
if native_price.is_zero() {
    return Err(internal_error!("Native price cannot be 0").into());
}
``` [5](#0-4) 

Both `pubdata_price` and `native_price` are operator-supplied block-level parameters read from the oracle: [6](#0-5) 

If the operator sets `pubdata_price` to a small value (e.g., to subsidize pubdata costs) while `native_price` is larger, or if these values are set incorrectly due to a misconfiguration or oracle manipulation, any unprivileged transaction sender can exploit the zero `native_per_pubdata` to generate pubdata for free. The attacker's entry path is simply submitting a transaction that writes to storage slots.

---

### Recommendation

Defer the division to after the multiplication, computing the native charge per pubdata byte at the point of charging rather than pre-dividing:

```rust
// Instead of:
let native_per_pubdata = pubdata_price / native_price;
let native_charged = native_per_pubdata * pubdata_bytes;

// Use:
let native_charged = pubdata_price * pubdata_bytes / native_price;
```

This requires propagating `pubdata_price` and `native_price` separately into `create_resources_for_tx` and `get_resources_to_charge_for_pubdata` instead of the pre-divided `native_per_pubdata`. Additionally, add a validation check that `pubdata_price >= native_price` (or at minimum that `native_per_pubdata > 0`) to reject transactions when the configuration would result in free pubdata.

---

### Proof of Concept

**Setup**: Operator sets `pubdata_price = U256::from(5)`, `native_price = U256::from(10)`.

**Step 1** — `native_per_pubdata` computation in `validation_impl.rs`:
```
native_per_pubdata = 5 / 10 = 0  (integer truncation)
```

**Step 2** — `create_resources_for_tx` in `gas_helpers.rs`:
```
intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0
native_limit unchanged after pubdata deduction
```

**Step 3** — User submits a transaction writing to 100 storage slots (generating ~3200 bytes of pubdata).

**Step 4** — `get_resources_to_charge_for_pubdata`:
```
native = 3200 * 0 = 0
```

**Result**: The transaction consumes 3200 bytes of pubdata, the operator must publish this to L1, but the user's native resource budget is not reduced at all. The user pays only EVM gas for the SSTORE opcodes, not the native resource cost of pubdata publication. Repeating this across many transactions within the block's pubdata limit exhausts the operator's pubdata budget without compensation.
