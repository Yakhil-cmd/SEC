### Title
Silo Address Whitelist Bypass via `refund_on_error` Minting to Non-Whitelisted EVM Address - (File: engine/src/engine.rs)

---

### Summary

When Silo mode is active with the `Address` whitelist enabled, the `receive_erc20_tokens` function correctly redirects incoming ERC-20 tokens away from non-whitelisted EVM addresses to a configured fallback address. However, the `refund_on_error` function — invoked via `exit_to_near_precompile_callback` when a NEAR-side transfer fails — mints ERC-20 tokens directly to a user-controlled `refund_address` **without performing any whitelist check**. This allows a whitelisted caller to route ERC-20 tokens to any arbitrary EVM address, bypassing the Silo access control restriction entirely.

---

### Finding Description

**Root cause — `refund_on_error` skips the whitelist:**

In `engine/src/engine.rs`, `receive_erc20_tokens` correctly gates token receipt behind `is_allow_receive_erc20_tokens`:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which checks the `WhitelistKind::Address` list:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [2](#0-1) 

However, `refund_on_error` — called from `exit_to_near_precompile_callback` when the NEAR-side `ft_transfer` fails — mints ERC-20 tokens directly to `args.recipient_address` with **no whitelist check**:

```rust
pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I, env: &E, state: EngineState, args: &RefundCallArgs, handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);
        let refund_address = args.recipient_address;  // ← user-controlled, no whitelist check
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);
        engine.call(&erc20_admin_address, &erc20_address, Wei::zero(), input, u64::MAX, ...)
    }
``` [3](#0-2) 

**Root cause — `refund_address` is attacker-controlled calldata:**

When the `error_refund` feature is compiled in, the `ExitToNear` precompile parses bytes `[1..21]` of the caller's input as the `refund_address`:

```rust
#[cfg(feature = "error_refund")]
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))
}
``` [4](#0-3) 

This `refund_address` is embedded verbatim into `ExitToNearPrecompileCallbackArgs` and later passed to `refund_on_error` as `args.recipient_address`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    ...
};
``` [5](#0-4) 

The callback dispatch in `exit_to_near_precompile_callback` calls `refund_on_error` unconditionally on failure:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [6](#0-5) 

---

### Impact Explanation

**Impact: High — Temporary Freezing of Funds**

When the Silo `Address` whitelist is active, the invariant is that only whitelisted EVM addresses may hold ERC-20 tokens (enforced at the `receive_erc20_tokens` entry point). The `refund_on_error` path breaks this invariant: ERC-20 tokens are minted to an arbitrary non-whitelisted address `A`. Because `submit` also enforces the whitelist, address `A` cannot submit any EVM transaction to move or exit those tokens. The tokens are therefore frozen at `A` until the Silo operator explicitly whitelists `A` — constituting a temporary (or permanent, if the operator never acts) freezing of funds.

---

### Likelihood Explanation

**Likelihood: Medium**

Preconditions:
1. The Aurora Engine is compiled with the `error_refund` feature enabled (a production feature, not test-only).
2. Silo mode is active with the `Address` whitelist enabled.
3. A whitelisted EVM address calls the `ExitToNear` precompile, supplying a non-whitelisted address as `refund_address` in calldata bytes `[1..21]`.
4. The NEAR-side `ft_transfer` or `ft_transfer_call` fails (e.g., the target NEAR account lacks a storage deposit).

Steps 3 and 4 are fully attacker-controlled: the caller chooses the `refund_address` freely, and can target a NEAR account known to reject the transfer. No admin compromise is required.

---

### Recommendation

In `refund_on_error`, before minting ERC-20 tokens to `refund_address`, apply the same whitelist gate used in `receive_erc20_tokens`:

```rust
let mut refund_address = args.recipient_address;
if let Some(fallback) = silo::get_erc20_fallback_address(&io)
    && !silo::is_allow_receive_erc20_tokens(&io, &refund_address)
{
    refund_address = fallback;
}
```

This mirrors the existing pattern in `receive_erc20_tokens` and ensures the Silo whitelist is consistently enforced across all ERC-20 minting paths. [7](#0-6)

### Proof of Concept

**Setup:** Silo mode active; `Address` whitelist enabled; `error_refund` feature compiled in.

1. **Attacker** is whitelisted EVM address `W` holding 1000 units of ERC-20 token `T`.
2. **Target** is non-whitelisted EVM address `A` (attacker-controlled).
3. `W` calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with calldata:
   - byte 0: `0x01` (ERC-20 exit flag)
   - bytes 1–20: `A` (non-whitelisted `refund_address`)
   - bytes 21–52: amount = 1000 (big-endian U256)
   - bytes 53+: NEAR account ID of a recipient with no storage deposit (guaranteed to fail)
4. The ERC-20 contract burns 1000 tokens from `W`. The NEAR `ft_transfer` call fails.
5. `exit_to_near_precompile_callback` fires; `args.refund` is `Some(RefundCallArgs { recipient_address: A, erc20_address: Some(T), amount: 1000 })`.
6. `refund_on_error` calls `engine.call` to invoke `ERC20_MINT_SELECTOR` on contract `T`, minting 1000 tokens to `A`.
7. **Result:** `A` holds 1000 tokens of `T` despite never being whitelisted. `A` cannot submit transactions (whitelist blocks `submit`), so the tokens are frozen — bypassing the Silo access control model. [8](#0-7) [4](#0-3) [9](#0-8)
