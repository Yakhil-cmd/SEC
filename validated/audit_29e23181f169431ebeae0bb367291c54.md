### Title
`is_paused` Flag Blocks In-Flight Bridge Refund Callback, Permanently Freezing Burned ERC-20 Tokens - (File: `engine/src/contract_methods/connector.rs`)

### Summary

The Aurora Engine's `pause_contract` mechanism sets `state.is_paused = true`, which causes `require_running` to block **all** connector callbacks including `exit_to_near_precompile_callback`. This callback is the sole mechanism for refunding ERC-20 tokens when an `ExitToNear` bridge transfer fails mid-flight. If the contract is paused between the ERC-20 burn and the callback execution, the refund path is permanently blocked, causing the user's ERC-20 tokens to be irreversibly destroyed with no compensation.

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) implements a two-step cross-chain exit:

1. ERC-20 tokens are burned inside the EVM during the user's `submit` call.
2. A NEAR promise is scheduled to call `ft_transfer` on the NEP-141 contract, with `exit_to_near_precompile_callback` as the callback.

The callback (`engine/src/contract_methods/connector.rs`, line 196) is the only place where a failed `ft_transfer` triggers a refund of the burned ERC-20 tokens (when the `error_refund` feature is enabled):

```rust
pub fn exit_to_near_precompile_callback<...>(...) -> Result<...> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // <-- BLOCKS ENTIRE CALLBACK IF PAUSED
        env.assert_private_call()?;
        ...
        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            // handle wNEAR unwrap
        } else if let Some(args) = args.refund {
            // REFUND PATH — never reached if paused
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
            ...
        };
    })
}
```

The `require_running` guard at line 203 is evaluated **before** the promise result is inspected. If `state.is_paused == true`, the function returns `Err(ERR_PAUSED)` immediately, skipping both the success path and the refund path.

`pause_contract` is an owner-controlled operation that sets `is_paused = true`:

```rust
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        ...
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

Because NEAR receipts execute asynchronously across blocks, a `pause_contract` call can land between the ERC-20 burn receipt and the `exit_to_near_precompile_callback` receipt. The burn is already committed and irreversible; the callback failure leaves no refund path open.

The same `require_running` guard also blocks `ft_on_transfer` (line 68), which is the NEAR→Aurora deposit callback. However, in that direction the NEP-141 standard's `ft_resolve_transfer` mechanism returns the tokens to the sender when `ft_on_transfer` signals failure, so the impact is a temporary freeze rather than permanent loss.

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the contract is paused while an ERC-20 exit is in-flight and the `ft_transfer` to NEAR fails:
- The ERC-20 tokens are already burned (state committed in a prior receipt).
- The `exit_to_near_precompile_callback` fails at `require_running` before reaching the refund branch.
- No other code path re-mints or refunds the burned ERC-20 tokens.
- The user permanently loses their tokens.

### Likelihood Explanation

The owner legitimately pauses the contract for emergency maintenance, upgrades, or security incidents — exactly the intended use of `pause_contract`. NEAR's asynchronous receipt model means there is always a window between the ERC-20 burn and the callback execution. Any pause during that window triggers the freeze. The window is narrow per individual transaction but the risk accumulates across all concurrent bridge exits whenever a pause occurs.

### Recommendation

Separate the pause granularity so that in-flight bridge callbacks (`exit_to_near_precompile_callback`, `ft_on_transfer`) are exempt from the global `is_paused` flag, or at minimum allow the refund branch of `exit_to_near_precompile_callback` to execute even when the contract is paused. The `ExitToNear` and `ExitToEthereum` precompiles already have their own independent pause mechanism (`PrecompileFlags`); the same separation should apply to their associated callbacks.

### Proof of Concept

1. User calls `submit` with a transaction that invokes the `ExitToNear` precompile (e.g., `withdrawToNear` on an ERC-20 contract). ERC-20 tokens are burned; a NEAR promise is scheduled.
2. Before the next NEAR block finalizes the callback receipt, the owner calls `pause_contract`.
3. The `ft_transfer` promise executes and fails (e.g., recipient account not registered with the NEP-141 contract).
4. `exit_to_near_precompile_callback` is invoked. At line 203, `require_running` returns `Err(ERR_PAUSED)`.
5. The refund branch (`engine::refund_on_error`) is never reached.
6. The user's ERC-20 tokens are permanently destroyed; no NEP-141 tokens are received.
