### Title
TOCTOU in XCC Router Deployment Version Check Allows Double-Charging of Storage Staking Amount — (File: `engine-precompiles/src/xcc.rs`)

---

### Summary

The XCC precompile reads `get_code_version_of_address` to decide whether to charge the caller `STORAGE_AMOUNT` (2 NEAR worth of wNEAR) for router account creation. The version record is only written by the `factory_update_address_version` callback, which executes asynchronously in a later block. A second XCC call from the same EVM address submitted before that callback settles will observe stale state (`None`), charge `STORAGE_AMOUNT` a second time, and schedule a second deployment that will fail because the account already exists. The user's wNEAR is left stranded in the engine's implicit address.

---

### Finding Description

**Step 1 — Precompile charges based on stale version check.**

In `engine-precompiles/src/xcc.rs`, `run_with_handle` reads the router version for the calling address: [1](#0-0) 

```rust
let required_near =
    match state::get_code_version_of_address(&self.io, &Address::new(sender)) {
        None => attached_near + state::STORAGE_AMOUNT,
        Some(_) => attached_near,
    };
```

When the result is `None`, `STORAGE_AMOUNT` (2 NEAR) is added to `required_near`, and a `transferFrom` is immediately executed inside the EVM to move that wNEAR from the caller to the engine's implicit address: [2](#0-1) 

This EVM state change is committed atomically with the NEAR transaction.

**Step 2 — Deployment and version update are asynchronous.**

In `engine/src/xcc.rs`, `handle_precompile_promise` schedules a batch action (CreateAccount → Transfer → DeployContract → Initialize) and attaches `factory_update_address_version` as a callback: [3](#0-2) 

The callback that writes the new version to storage only executes in a subsequent block, after the deployment promise resolves: [4](#0-3) 

**Step 3 — Second transaction observes stale `None` before callback settles.**

Between block N (TX1 submitted) and block N+k (callback runs), any second NEAR transaction that executes the XCC precompile for the same EVM address will call `get_code_version_of_address` and receive `None` again, because `factory_update_address_version` has not yet written the updated version: [1](#0-0) 

TX2 therefore charges `STORAGE_AMOUNT` a second time and schedules a second deployment batch.

**Step 4 — Second deployment fails; wNEAR is stranded.**

When TX2's deployment batch executes, `CreateAccount` fails because the sub-account already exists (created by TX1). The `send_refund` callback is attached to the router contract: [5](#0-4) 

Because the router account does not exist at the time `send_refund` is invoked (the deployment failed), the refund call also fails. The `STORAGE_AMOUNT` worth of wNEAR transferred from the user in TX2 remains as an ERC-20 balance in the engine's implicit address with no recovery path exposed to the user.

---

### Impact Explanation

Each occurrence strands `STORAGE_AMOUNT` = 2 NEAR (≈ `2_000_000_000_000_000_000_000_000` yoctoNEAR) worth of wNEAR in the engine's implicit address. The user cannot recover these funds through any standard Aurora interface. This constitutes **temporary (potentially permanent) freezing of user funds** — matching the "High. Temporary freezing of funds" impact tier.

---

### Likelihood Explanation

Any EVM user or contract that issues two XCC calls from the same address within the window between TX1's submission and the settlement of its `factory_update_address_version` callback (typically 1–3 blocks on NEAR) triggers this condition. This window is wide enough to be hit accidentally (e.g., a contract that issues XCC calls in a loop, or a user retrying a failed transaction). No special privilege is required; the XCC precompile is reachable by any EVM caller.

---

### Recommendation

Apply the same fix class as the referenced report: serialize the version check and the deployment so that the version record is written before any subsequent XCC call can observe stale state. Concretely:

1. **Option A (simplest):** Before scheduling the deployment batch, write a sentinel version value (e.g., `CodeVersion(u32::MAX)`) to storage for the address. Any concurrent TX2 will then see `Some(_)` and skip the `STORAGE_AMOUNT` charge. The real version is written by the callback as today.
2. **Option B:** Introduce a "pending deployment" flag in storage that is set synchronously during the precompile call and cleared by the callback, causing subsequent calls to skip the storage-staking charge while a deployment is in flight.

---

### Proof of Concept

1. EVM address `0xALICE` calls the XCC precompile (TX1, block N).
   - `get_code_version_of_address(ALICE)` → `None`.
   - `transferFrom(ALICE, engine_implicit, attached_near + STORAGE_AMOUNT)` executes in EVM.
   - Deployment batch + `factory_update_address_version` callback scheduled.

2. Before block N+k (callback settlement), `0xALICE` submits TX2 (block N+1).
   - `get_code_version_of_address(ALICE)` → `None` (callback not yet run).
   - `transferFrom(ALICE, engine_implicit, attached_near + STORAGE_AMOUNT)` executes again.
   - Second deployment batch + callback scheduled.

3. TX1's deployment succeeds (block N+1); callback writes `version = 1` (block N+2).

4. TX2's deployment batch runs (block N+2): `CreateAccount` fails — account exists.
   - `withdraw_wnear_to_router` runs as callback but targets a non-existent router → fails.
   - `send_refund` targets the non-existent router → fails.
   - `STORAGE_AMOUNT` wNEAR remains in engine's implicit ERC-20 balance; ALICE has no recovery path.

Net loss to ALICE: `STORAGE_AMOUNT` = 2 NEAR per such window, repeatable across any future router upgrade cycle that resets `create_needed = true`.
