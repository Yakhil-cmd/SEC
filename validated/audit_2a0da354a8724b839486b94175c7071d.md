### Title
Unchecked Return Value of Low-Level Precompile `call` in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20` and `EvmErc20V2` burn a user's ERC-20 tokens before making a low-level `call` to the `ExitToNear` or `ExitToEthereum` precompile. The return value of that `call` is captured in a local assembly variable (`res`) but is **never inspected**. If the precompile call fails for any reason (e.g., the precompile is paused, gas is insufficient, or input is rejected), the burn is already committed and the tokens are permanently destroyed with no corresponding NEAR or Ethereum-side transfer.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear` and `withdrawToEthereum` functions follow this pattern:

1. `_burn(msg.sender, amount)` — irreversibly destroys the caller's tokens.
2. An inline-assembly `call` is made to the hardcoded precompile address.
3. The return value `res` is assigned but **never checked**.

`EvmErc20.sol` `withdrawToNear` (lines 53–63):
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
```

`EvmErc20.sol` `withdrawToEthereum` (lines 65–76):
```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
```

The identical pattern is present in `EvmErc20V2.sol` at lines 61–63 and 74–76.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can legitimately return failure (`res = 0`) in multiple code paths: the precompile is paused via `EnginePrecompilesPauser`, the call is made in static context, the input is malformed, or gas is exhausted. When any of these conditions hold, the EVM `call` opcode returns `0`, but because `res` is never tested, the Solidity function returns normally — after the burn has already been committed.

This is the direct structural analog of the DSExec/DSProxy bug: a low-level call whose success indicator is silently discarded, causing the caller to believe the operation succeeded when it did not.

---

### Impact Explanation

**Critical — Permanent freezing / direct theft of user funds.**

When the precompile call returns `0` (failure), the ERC-20 tokens have already been burned. There is no revert, no refund path, and no retry mechanism in the contract. The tokens are gone. The corresponding NEAR-side or Ethereum-side credit is never issued. The user suffers a total, unrecoverable loss of the bridged asset.

---

### Likelihood Explanation

**Medium-High.** The `ExitToNear` and `ExitToEthereum` precompiles are pausable via `EnginePrecompilesPauser`. Any period during which a precompile is paused — even briefly for an upgrade or incident response — will silently burn tokens for every user who calls `withdrawToNear` or `withdrawToEthereum` during that window. Additionally, any future change to the precompile addresses (the addresses are hardcoded as literals in the Solidity) would make every call silently fail. The entry path requires no special privilege: any ERC-20 token holder can trigger the loss.

---

### Recommendation

Revert the entire transaction if the precompile call fails. Replace the unchecked assembly block with a checked call:

```solidity
(bool success, ) = address(0xe9217bc70b7ed1f598ddd3199e80b093fa71124f)
    .call{gas: gasleft()}(input);
require(success, "ExitToNear precompile call failed");
```

Apply the same fix to `withdrawToEthereum` and to both `EvmErc20.sol` and `EvmErc20V2.sol`. The burn must either be moved **after** a successful precompile call, or the entire function must revert atomically on failure.

---

### Proof of Concept

1. Aurora operator pauses the `ExitToNear` precompile (a supported operational action via `EnginePrecompilesPauser`).
2. Alice holds 1000 bridged USDC (an `EvmErc20` token) on Aurora.
3. Alice calls `withdrawToNear("alice.near", 1000)`.
4. `_burn(alice, 1000)` executes — Alice's ERC-20 balance drops to 0.
5. The `call` to `0xe921...124f` returns `0` because the precompile is paused.
6. `res` is never checked; the function returns without reverting.
7. Alice's 1000 USDC are permanently destroyed. No NEP-141 tokens are transferred to `alice.near`.
8. Alice has lost her funds with no recourse. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
