Audit Report

## Title
No Staleness Guard on Stored ICP/XDR Rate at Point of Cycles Minting - (File: rs/nns/cmc/src/main.rs)

## Summary
The Cycles Minting Canister stores an `icp_xdr_conversion_rate` that is consumed by `tokens_to_cycles` without any maximum-age enforcement. When the Exchange Rate Canister's HTTPS outcalls fail to gather sufficient sources, the CMC retains the last accepted rate indefinitely and continues minting cycles at that stale price. Any unprivileged caller of `notify_top_up` or `notify_mint_cycles` during such a period receives cycles computed from a potentially arbitrarily stale rate, enabling over-minting if the real ICP price has fallen.

## Finding Description

**Root cause — `tokens_to_cycles` (`rs/nns/cmc/src/main.rs`, L1900–1923)**

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp); // timestamp_seconds never read
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...)
        }
    })
}
```

The function extracts only `xdr_permyriad_per_icp`; `rate.timestamp_seconds` is never compared against the current time. There is no maximum-age threshold that would cause the function to refuse conversion when the rate is stale.

**Validation gap — `validate_exchange_rate` (`rs/nervous_system/clients/src/exchange_rate_canister_client.rs`, L110–129)**

`validate_exchange_rate` checks only that at least `MINIMUM_ICP_SOURCES` (4) and `MINIMUM_CXDR_SOURCES` (4) responded. It performs no check on `exchange_rate.timestamp` relative to the current time. A rate that is hours or days old passes validation as long as it was originally collected from enough sources.

**Rate-setting guard — `do_set_icp_xdr_conversion_rate` (`rs/nns/cmc/src/main.rs`, L1009–1040)**

This function only enforces that the incoming rate has a strictly greater timestamp than the currently stored one. It does not check whether the incoming rate is fresh relative to wall-clock time, and it does not impose any upper bound on how old the stored rate may become.

**Staleness accumulation path — `rs/nns/cmc/src/exchange_rate_canister.rs`, L149–161**

On `FailedToRetrieveRate`, `FailedToSetRate`, or `InvalidRate`, `schedule_next_attempt` schedules a retry one minute later but leaves `state.icp_xdr_conversion_rate` unchanged. There is no circuit-breaker that halts minting after a configurable maximum age without a fresh rate.

**Attacker-controlled entry path**

Any principal can call `notify_top_up` or `notify_mint_cycles`. Both call `process_top_up` / `process_mint_cycles`, which call `tokens_to_cycles(amount)` directly with no privilege check on the rate's age.

## Impact Explanation

If the ICP/XDR rate stored in the CMC is stale and the real ICP price has fallen, every call to `notify_top_up` or `notify_mint_cycles` mints more cycles than the deposited ICP is worth at the current market price. Cycles are the IC's unit of computation cost; over-minting cycles at a stale high price is a direct conservation violation — the ICP burned is worth less than the cycles created. At scale this drains the economic backing of the cycles supply. This matches the allowed impact class: **illegal minting of in-scope protocol assets (Cycles/ICP) with concrete protocol harm**, qualifying as **High ($2,000–$10,000)** given the conditions required, with potential escalation to Critical if the XRC outage is prolonged and significant ICP volume is processed.

## Likelihood Explanation

The XRC fetches ICP and CXDR rates via HTTPS outcalls to external exchanges. The minimum-source threshold is 4 for each asset class. During periods of exchange downtime, rate-limiting, or network partitions affecting multiple exchanges simultaneously, the XRC returns `CryptoBaseAssetNotFound`, `StablecoinRateTooFewRates`, or similar errors. The CMC then retries every minute but keeps the old rate. Such conditions are realistic: exchange APIs are regularly rate-limited during high-volatility periods — exactly when price staleness is most dangerous. The window of exposure is unbounded; there is no circuit-breaker that halts minting after, say, 30 minutes without a fresh rate. The attacker requires only ICP tokens and knowledge of the outage, both of which are readily available.

## Recommendation

1. Add a maximum-age check inside `tokens_to_cycles` before using the stored rate:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref().ok_or_else(|| {
            NotifyError::Other { error_code: NotifyErrorCode::Internal as u64,
                error_message: "No conversion rate found in CMC".to_string() }
        })?;
        let age = now_seconds().saturating_sub(rate.timestamp_seconds);
        if age > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: format!("ICP/XDR rate is stale ({age}s old); minting suspended"),
            });
        }
        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

2. Extend `validate_exchange_rate` to reject rates whose `exchange_rate.timestamp` is older than a configurable threshold relative to the current time.

3. Consider adding a circuit-breaker in the heartbeat that disables minting (sets `update_exchange_rate_canister_state` to a `MintingSuspended` variant) after a configurable number of consecutive failed refresh attempts.

## Proof of Concept

1. Deploy CMC with a stored rate of 50,000 XDR/ICP (ICP = $10).
2. Simulate XRC HTTPS outcall failures (fewer than 4 sources respond) — the CMC logs `FailedToRetrieveRate` every minute but keeps the old rate.
3. Advance simulated time so the stored rate is, e.g., 2 hours old; ICP market price is now $5 (25,000 XDR/ICP).
4. As an unprivileged principal, transfer 1 ICP to the CMC subaccount and call `notify_top_up`.
5. `tokens_to_cycles` reads `xdr_permyriad_per_icp = 50,000` from the stale stored rate — no timestamp check occurs.
6. The caller receives cycles equivalent to $10 worth of ICP while only depositing $5 worth — a 2× over-mint.
7. The ICP is burned; the excess cycles are permanently in circulation.

A deterministic integration test using `StateMachine` / PocketIC can reproduce this by: (a) installing CMC with a known rate, (b) mocking the XRC client to return `CryptoBaseAssetNotFound` for all subsequent calls, (c) advancing time beyond any reasonable freshness window, and (d) asserting that `notify_top_up` still succeeds and mints cycles at the stale rate.
