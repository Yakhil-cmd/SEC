Audit Report

## Title
Unbounded `raw_rand_contexts` Queue Drained Without Instruction Limit Enables Execution Liveness DoS - (File: `rs/execution_environment/src/scheduler.rs`)

## Summary
The IC scheduler drains the entire `raw_rand_contexts` queue every round in an unbounded `while` loop that explicitly bypasses the per-round instruction limit. The underlying `VecDeque<RawRandContext>` in `SubnetCallContextManager` has no enforced maximum size, and `push_raw_rand_request` appends unconditionally. Because each iteration performs real work (CSPRNG invocation, response serialization, output queue enqueue) and the loop executes before `inner_round`, a sufficiently large queue starves all canister message execution for that round, causing subnet-wide execution liveness denial.

## Finding Description
**Root cause 1 — unbounded queue with no size cap:**
`SubnetCallContextManager.raw_rand_contexts` is declared as `pub raw_rand_contexts: VecDeque<RawRandContext>` with no capacity field or enforcement. `push_raw_rand_request` calls `self.raw_rand_contexts.push_back(...)` unconditionally with no size check.

**Root cause 2 — unbounded drain loop bypassing instruction limit:**
In `execute_round` (scheduler.rs lines 1379–1402), the loop:
```rust
while let Some(raw_rand_context) = state
    .metadata
    .subnet_call_context_manager
    .raw_rand_contexts
    .pop_front()
{
    let (new_state, _) = self.execute_subnet_message(...);
    state = new_state;
}
```
carries no iteration cap and no instruction budget check. The comment at line 1381 explicitly states the rationale: "Raw rand is not consuming instructions, so all existing raw_rand requests will be processed." The test at `scheduler/tests/subnet_messages.rs` lines 239–246 confirms zero instructions are recorded (`sum: 0.0`) per raw_rand execution, meaning the `subnet_round_limits` instruction counter is never decremented and the loop never self-terminates via the normal instruction budget path.

**Root cause 3 — loop precedes `inner_round`:**
The raw_rand drain loop at lines 1370–1404 executes before the `inner_round` call at line 1435. Any wall-clock time consumed by the drain loop directly reduces the time available for canister message execution.

**Attack flow:**
1. Attacker deploys one or more canisters on the target subnet.
2. Each round, each canister issues `ic00::RawRand` calls up to the per-canister ic00 output queue capacity (`DEFAULT_QUEUE_CAPACITY = 500`).
3. During `drain_subnet_queues`, these requests are deferred to `raw_rand_contexts` (with `execution_round_id = current_round`; the `debug_assert_lt` at line 1389 enforces they are only processed in a subsequent round).
4. The following round's `execute_round` drains the entire accumulated queue in the unbounded loop before `inner_round` is ever called.
5. With enough canisters contributing requests, the loop's wall-clock cost exhausts the round budget, and `inner_round` executes with no remaining time, starving all canister ingress, inter-canister messages, heartbeats, and timers.
6. The attacker repeats each round, maintaining the backlog.

**Why existing checks are insufficient:**
- The `subnet_round_limits.subnet_instructions` budget is initialized at `max_instructions_per_round / SUBNET_MESSAGES_LIMIT_FRACTION` but is never decremented by raw_rand processing (0 instructions consumed), so it never triggers a loop exit.
- The per-canister output queue capacity (500 slots to ic00) limits a single canister's contribution per round but does not bound the aggregate across many canisters.
- There is no operator mechanism to flush or cap `raw_rand_contexts` without a replica upgrade.

## Impact Explanation
This is a **High ($2,000–$10,000)** severity finding matching the allowed impact: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS." The effect is subnet-wide: all canisters on the targeted subnet lose execution liveness for every round in which the queue is large enough to exhaust the round's wall-clock budget. The attack is persistent — the attacker can continuously refill the queue each round. It does not require any special privilege beyond deploying a canister and holding cycles.

## Likelihood Explanation
Any canister deployed on the subnet can call `ic0.raw_rand` (a standard management canister method). The per-canister ic00 queue capacity (500) provides a natural per-round contribution limit per canister, but an attacker controlling or coordinating multiple canisters can aggregate contributions. The cycles cost per call is the standard management canister call fee, making sustained flooding economically feasible for a motivated attacker. The attack is repeatable every round and requires no victim interaction, no social engineering, and no special subnet privileges.

## Recommendation
1. **Enforce a maximum `raw_rand_contexts` queue size**: In `push_raw_rand_request`, reject new requests when `self.raw_rand_contexts.len()` exceeds a configurable bound (e.g., 500–1000), returning a reject response to the caller.
2. **Process `raw_rand` contexts in bounded batches per round**: Replace the unbounded `while` loop with a loop that processes at most `N` entries per round (e.g., `N = DEFAULT_QUEUE_CAPACITY`), carrying the remainder to the next round.
3. **Add a per-call fee premium or rate limit** to make flooding economically infeasible at scale.

## Proof of Concept
The following existing test infrastructure confirms all components of the vulnerability:

- **Unbounded queue, no size cap**: `push_raw_rand_request` at `rs/replicated_state/src/metadata_state/subnet_call_context_manager.rs` lines 407–418 appends unconditionally; `raw_rand_contexts` declared as `VecDeque<RawRandContext>` at line 228 with no capacity.
- **Unbounded drain loop**: `rs/execution_environment/src/scheduler.rs` lines 1383–1402.
- **Zero instructions consumed**: `rs/execution_environment/src/scheduler/tests/subnet_messages.rs` lines 239–246 asserts `sum: 0.0` for the raw_rand phase.
- **Deterministic reproduction**: Extend `scheduler_executes_postponed_raw_rand_requests` (subnet_messages.rs line 185) by injecting N×500 raw_rand contexts (simulating N canisters each at queue capacity), executing one round, and asserting that `inner_round` processed zero canister messages.
