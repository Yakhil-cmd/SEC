Audit Report

## Title
Merkle Root Guard Short-Circuits on Empty `txdata`, Allowing Cache Poisoning by Malicious Bitcoin Peer — (File: `rs/bitcoin/adapter/src/blockchainstate.rs`)

## Summary
The `add_block` guard in `BlockchainState` uses a compound condition that short-circuits when `txdata` is empty, bypassing merkle root validation entirely. A malicious Bitcoin peer can respond to a solicited `getdata` request with a structurally stripped block (valid header, empty `txdata`) to permanently poison the adapter's block cache. Because the `getdata_request_info` entry is consumed on first receipt and `enqueue_new_blocks_to_download` skips hashes already in the cache, the correct block is never re-fetched, causing a persistent stall in Bitcoin block ingestion and halting ckBTC operations.

## Finding Description
The guard at `blockchainstate.rs` line 237 is:

```rust
if block.compute_merkle_root().is_some() && !block.check_merkle_root() {
    return Err(AddBlockError::InvalidMerkleRoot(block_hash));
}
```

`bitcoin::Block::compute_merkle_root()` returns `Option<TxMerkleNode>` and returns `None` when `txdata` is empty. When `txdata: vec![]`, `compute_merkle_root().is_some()` evaluates to `false`, short-circuiting the `&&`. The block is never rejected regardless of what the header's `merkle_root` field contains. The block then proceeds unconditionally to be serialized and inserted into `block_cache` at lines 259–262.

The `received_block_message` handler in `blockchainmanager.rs` at line 427 removes the `getdata_request_info` entry on first receipt:

```rust
let request = match self.getdata_request_info.remove(&block_hash) {
    Some(request) => request,
    None => return Err(ReceivedBlockMessageError::UnknownBlock),
};
```

Because the block hash is derived from the header (not from `txdata`), a stripped block with the same header has the same hash. Once the malicious peer's response is accepted, the entry is consumed. `enqueue_new_blocks_to_download` at lines 724–730 will not re-queue the block because `get_block(&hash)` now returns `Some(...)`:

```rust
if self.blockchain.get_block(&hash).is_none()
    && !self.block_sync_queue.contains(&hash)
    && !self.getdata_request_info.contains_key(&hash)
{
    self.block_sync_queue.insert(hash);
}
```

The existing `test_sorted_tip` test at lines 859–873 confirms the guard permits empty-`txdata` blocks through — it calls `add_block` with `txdata: Vec::new()` and asserts `Ok(())`, demonstrating the bypass is the current production behavior.

The upstream `bitcoin::Block::check_merkle_root()` implementation returns `false` when `txdata` is empty (it matches on `compute_merkle_root()` returning `None` and returns `false`), meaning the compound guard is strictly weaker than a direct call to `!block.check_merkle_root()`.

## Impact Explanation
The Bitcoin adapter is the sole block-fetching component for the ckBTC / Chain Fusion Bitcoin integration. A single malicious peer can permanently stall block ingestion at any target height by poisoning the block cache with a stripped block. The adapter will never re-fetch the correct block, halting all ckBTC deposit/withdrawal processing that depends on confirmed Bitcoin blocks. This constitutes a significant Chain Fusion / ck-token infrastructure security impact with concrete and lasting user harm (ckBTC operations frozen at the targeted height). This matches the High impact category: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The adapter connects to external Bitcoin peers by design. Any peer in the adapter's peer list can trigger this with a single crafted `block` P2P message in response to a legitimate `getdata`. No PoW mining is required — the attacker reuses a header the adapter already accepted and strips the transactions. The attack is repeatable at any block height and requires no special privileges beyond being a connected peer.

## Recommendation
Replace the compound guard with a direct call to `check_merkle_root()`, which already handles the `None` case by returning `false`:

```rust
if !block.check_merkle_root() {
    return Err(AddBlockError::InvalidMerkleRoot(block_hash));
}
```

This correctly rejects both: (a) non-empty `txdata` with a mismatched merkle root, and (b) empty `txdata` with any merkle root value in the header. The `test_sorted_tip` test and any other tests using `txdata: Vec::new()` with a non-matching `merkle_root` in the header will need to be updated to use structurally valid blocks.

## Proof of Concept
Unit test against `blockchainstate::add_block`:

```rust
#[tokio::test]
async fn test_empty_txdata_with_nonzero_merkle_root_is_rejected() {
    let state = BlockchainState::new(
        Network::Regtest,
        None,
        &MetricsRegistry::default(),
        no_op_logger(),
    );
    let genesis = state.genesis();
    let mut header = generate_header(genesis.block_hash(), genesis.time, 0);
    // Set a non-zero merkle root that cannot match an empty txdata
    header.merkle_root = TxMerkleNode::from_raw_hash(
        bitcoin::hashes::sha256d::Hash::from_byte_array([0xff; 32])
    );
    let bad_block = Block { header, txdata: vec![] };
    let result = state.add_block(bad_block).await;
    // Under current code this returns Ok(()) — the assertion below fails,
    // confirming the bypass.
    assert!(
        matches!(result, Err(AddBlockError::InvalidMerkleRoot(_))),
        "Expected rejection, got: {:?}", result
    );
}
```

Under the current code this assertion fails — `add_block` returns `Ok(())` and caches the invalid block, confirming the guard bypass.
