Audit Report

## Title
`validate_execute_extension_operation` Permanently Blocks Asset Recovery After `ALLOWED_EXTENSIONS` Cleared — (`rs/sns/governance/src/extensions.rs`)

## Summary
`ALLOWED_EXTENSIONS` is intentionally empty in production as of April 6, 2026. `get_extension_spec_and_update_cache`, called unconditionally from `validate_execute_extension_operation`, always re-validates the installed WASM hash against this empty map and returns an `InvalidProposal` error on failure without falling back to the stable-memory cache. Any SNS that had a KongSwap extension registered with treasury assets deposited can never pass proposal validation for a withdraw or deposit operation, permanently locking those assets with no on-chain governance recovery path.

## Finding Description
`ALLOWED_EXTENSIONS` is confirmed empty in production: [1](#0-0) 

In non-test builds, `validate_extension_wasm` passes this empty map to `validate_extension_wasm_with_allowed`: [2](#0-1) 

`validate_extension_wasm_with_allowed` returns `Err` for any hash not present in the map: [3](#0-2) 

`get_extension_spec_and_update_cache` calls `validate_extension_wasm` against the live WASM hash fetched from `canister_module_hash`. On failure it returns the error directly; the stable-memory cache (`get_registered_extension_from_cache`) is never consulted as a fallback. The cache is only written on success: [4](#0-3) 

`validate_execute_extension_operation` calls `get_extension_spec_and_update_cache` unconditionally and propagates any error immediately via `?`, before any execution logic is reached: [5](#0-4) 

The stable-memory cache (`REGISTERED_EXTENSIONS`) exists and `get_registered_extension_from_cache` is available in `storage.rs`, but it is never used as a fallback in `get_extension_spec_and_update_cache`: [6](#0-5) 

The exploit path is: (1) SNS has KongSwap extension registered with treasury assets deposited before April 6, 2026; (2) governance canister is upgraded, clearing `ALLOWED_EXTENSIONS`; (3) any governance participant submits an `ExecuteExtensionOperation` withdraw proposal; (4) `validate_execute_extension_operation` calls `get_extension_spec_and_update_cache`, which calls `validate_extension_wasm` against the empty map, returns `Err(InvalidProposal, "Wasm module with hash X is not allowed as an extension.")`; (5) the proposal is rejected at validation time; (6) assets remain permanently locked with no alternative proposal type capable of calling `withdraw` on the extension canister.

## Impact Explanation
The impact is permanent, irrecoverable loss of SNS treasury assets (SNS tokens and ICP) held in the extension canister. There is no alternative `ExecuteExtensionOperation`-equivalent proposal type that bypasses the WASM-hash check, and no emergency withdrawal path in the governance canister. This matches the allowed impact class: **permanent loss of in-scope ledger/governance assets**. Severity is High to Critical depending on the total value of assets locked across affected SNS instances; if aggregate locked value exceeds $1M the Critical threshold applies.

## Likelihood Explanation
No attacker action is required. The precondition — KongSwap extension registered with deposited assets — is already satisfied on mainnet for any SNS that used KongSwap before April 6, 2026. The governance canister upgrade that cleared `ALLOWED_EXTENSIONS` has already been deployed. The condition is triggered by any legitimate governance participant submitting a withdraw proposal, which is a normal, unprivileged governance action. The failure is deterministic and repeatable on every such proposal submission.

## Recommendation
`get_extension_spec_and_update_cache` should fall back to `get_registered_extension_from_cache` when `validate_extension_wasm` fails, returning the cached `ExtensionSpec` for already-registered extensions rather than propagating the error. Alternatively, a separate `EmergencyWithdraw` or `UnregisterExtension` proposal type should be introduced that bypasses the WASM-hash check for extensions already present in the stable-memory cache, ensuring that registered extensions with deposited assets always have a governance-accessible withdrawal path independent of the current state of `ALLOWED_EXTENSIONS`.

## Proof of Concept
State-machine or PocketIC test:
1. Start an SNS governance canister with a KongSwap extension registered (WASM hash stored in `REGISTERED_EXTENSIONS` stable-memory cache via `cache_registered_extension`).
2. Simulate the production state by ensuring `ALLOWED_EXTENSIONS` is empty (already the case in non-test builds; in test builds, use a build without the `test` feature or override `TEST_HASH_TO_EXTENSION_SPEC` to exclude the KongSwap hash).
3. Submit an `ExecuteExtensionOperation` proposal with `operation_name = "withdraw"` targeting the registered extension canister.
4. Assert that `validate_execute_extension_operation` returns `Err(GovernanceError { error_type: InvalidProposal, error_message: "... Wasm module with hash ... is not allowed as an extension." })`.
5. Assert that no `withdraw` call is ever dispatched to the extension canister.
6. Assert that the treasury balance in the extension canister is unchanged.
