### Title
Silent `u32`→`u16` Truncation of `tx_number` in `L2ToL1Log` Corrupts Log Attribution for Blocks with >65535 Transactions - (`zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

The internal per-block transaction counter (`tx_number`) is stored as `u32` throughout the IO subsystem, but is silently narrowed to `u16` when serializing `L2ToL1Log` entries. Any block that includes more than 65 535 valid transactions will produce L2→L1 logs whose `tx_number_in_block` field wraps around to a previously-used value, corrupting the Merkle root that is committed to L1 and mis-attributing logs to the wrong transactions.

---

### Finding Description

**Root cause — silent narrowing cast**

`GenericLogContent` stores the originating transaction index as `u32`:

```rust
// zk_ee/src/common_structs/logs_storage.rs  line 73
pub struct GenericLogContent<IOTypes: SystemIOTypesConfig, A: Allocator = Global> {
    pub tx_number: u32,
    ...
}
```

The `L2ToL1Log` wire format declares the same field as `u16`:

```rust
// zk_ee/src/common_structs/logs_storage.rs  line 47
pub tx_number_in_block: u16,
```

The conversion between the two types is an unchecked `as` cast:

```rust
// zk_ee/src/common_structs/logs_storage.rs  line 519
tx_number_in_block: m.tx_number as u16,
```

In Rust, `as` on integer types is a **wrapping / truncating** cast; no panic, no error, no saturation. Transaction 65 536 silently becomes transaction 0, transaction 65 537 becomes transaction 1, and so on.

**How `tx_number` grows**

`tx_number` lives in `FullIO` as a `u32` (initialised to 0) and is incremented unconditionally for every successfully committed transaction:

```rust
// basic_system/src/system_implementation/system/io_subsystem.rs  line 530-533
fn finish_tx(&mut self) -> Result<(), InternalError> {
    self.storage.finish_tx()?;
    self.tx_number += 1;
    Ok(())
}
```

There is no guard that caps `tx_number` at `u16::MAX` before it is used in log construction. The ZK block-limit checks (`check_for_block_limits`) enforce gas, native resources, pubdata, and L2→L1 log count, but **not** a transaction-count ceiling of 65 535.

**Where the corrupted value propagates**

The `tx_number_in_block` field is:

1. Serialised into the 88-byte wire encoding of every L2→L1 log (bytes 2–3):
   ```rust
   // logs_storage.rs  line 453
   buffer[2..4].copy_from_slice(&self.tx_number_in_block.to_be_bytes());
   ```
2. Hashed into each leaf of the L2→L1 Merkle tree whose root is committed to L1:
   ```rust
   // logs_storage.rs  line 418-420
   self.list.iter().for_each(|el| {
       let log: L2ToL1Log = el.into();
       elements.push(log.hash())
   });
   ```
3. Forwarded to the interface layer and ultimately to L1 verifiers:
   ```rust
   // forward_system/src/run/convert.rs  line 169
   tx_number_in_block: self.tx_number_in_block,
   ```

---

### Impact Explanation

When `tx_number` wraps past 65 535:

* **Merkle root corruption** — every L2→L1 log emitted by transactions ≥ 65 536 carries a wrong `tx_number_in_block`. Because this field is part of the leaf hash, the resulting Merkle root differs from the one a correct implementation would produce. The committed root is therefore incorrect.
* **Log mis-attribution** — a withdrawal or cross-chain message sent in transaction 65 536 will appear on L1 as if it originated from transaction 0. Depending on how L1 contracts consume this field, this can allow replaying or suppressing messages.
* **Forward/proving divergence** — the forward execution and the ZK circuit must agree on the Merkle root. If the circuit independently recomputes `tx_number_in_block` from the correct `u32` value while the forward execution truncates it, the proof will fail to verify, halting the chain. Conversely, if the circuit mirrors the truncation, the L1 receives a provably-wrong root.

---

### Likelihood Explanation

A ZKsync OS block is bounded by gas, native resources, pubdata, and log count — not by raw transaction count. A block filled with minimal-cost transactions (e.g., simple ETH transfers at ~21 000 gas each) could theoretically contain tens of thousands of transactions. If the block gas limit is set above ~1.37 billion gas (65 536 × 21 000), the threshold is reachable. An attacker who can flood the sequencer with cheap transactions and who controls or predicts when a victim's L2→L1 log will land at index ≥ 65 536 can trigger the corruption. The attack requires no privileged access — only the ability to submit transactions.

---

### Recommendation

1. **Widen `tx_number_in_block` to `u32`** in `L2ToL1Log` and update the wire encoding accordingly, or
2. **Add a checked cast** that returns an error if `tx_number > u16::MAX`:
   ```rust
   tx_number_in_block: u16::try_from(m.tx_number)
       .map_err(|_| internal_error!("tx_number exceeds u16::MAX"))?,
   ```
3. **Enforce a hard block-level cap** of 65 535 valid transactions in `check_for_block_limits` as a defence-in-depth measure.

---

### Proof of Concept

1. Construct a block containing 65 537 transactions: the first 65 535 are simple ETH transfers (no L2→L1 logs), transaction 65 536 is a no-op, and transaction 65 537 calls the L1 Messenger to emit a withdrawal log.
2. After `finish_tx()` is called 65 537 times, `self.tx_number == 65537`.
3. When `push_message` is called for the withdrawal log, it stores `tx_number = 65537`.
4. In `From<&LogContent<A>> for L2ToL1Log` (line 519), `65537u32 as u16 == 1`.
5. The serialised log carries `tx_number_in_block = 1` instead of `65537`.
6. The Merkle root computed from this log differs from the correct root, and the withdrawal is attributed to transaction 1 on L1.
