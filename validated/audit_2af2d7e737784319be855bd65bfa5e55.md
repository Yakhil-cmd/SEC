### Title
Permissionless `deploy_erc20_token` Allows Any Caller to Permanently Block Silo's `mirror_erc20_token` - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` has no caller access control — any NEAR account can invoke it. It registers a NEP-141 → ERC-20 mapping via `register_token`. The `mirror_erc20_token_callback` path also calls `register_token` for the same NEP-141. Because `register_token` hard-reverts on a duplicate NEP-141 key, an attacker who pre-registers a NEP-141 on a silo permanently prevents the silo owner from ever mirroring the legitimate ERC-20 for that token.

---

### Finding Description

**Step 1 — Permissionless entry point.**

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` performs only a liveness check; there is no `require_owner_only` or any other caller restriction:

```rust

    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // ← only check
        ...
        let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;
``` [1](#0-0) 

**Step 2 — Registration is written unconditionally on first call.**

`engine::deploy_erc20_token` calls `engine.register_token(address, nep141)`, which inserts `nep141 → erc20_address` into `nep141_erc20_map`:

```rust
nep141_erc20_map(self.io).insert(&nep141_token, &erc20_token);
``` [2](#0-1) 

**Step 3 — `register_token` hard-fails on duplicate NEP-141.**

```rust
Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered),
``` [3](#0-2) 

**Step 4 — `mirror_erc20_token_callback` hits the same guard.**

The owner-only `mirror_erc20_token` flow resolves through `mirror_erc20_token_callback`, which calls `engine::mirror_erc20_token`, which also calls `register_token` for the same NEP-141:

```rust
engine
    .register_token(address, args.nep141)
    .map_err(DeployErc20Error::Register)?;
``` [4](#0-3) 

The callback itself propagates the error with `?`, so the entire `mirror_erc20_token` transaction fails: [5](#0-4) 

**Step 5 — No recovery path exists.**

There is no `unregister_token` or equivalent function anywhere in the engine. The mapping written by the attacker is permanent.

---

### Impact Explanation

Once an attacker pre-registers a NEP-141 on a silo via `deploy_erc20_token`, the silo owner can never successfully call `mirror_erc20_token` for that NEP-141. The silo is permanently unable to mirror the legitimate ERC-20 from the main Aurora contract for that token. Any subsequent `ft_on_transfer` calls for that NEP-141 will route to the attacker's spurious ERC-20 contract rather than the legitimate mirrored one, misdirecting bridged user funds. This constitutes **permanent freezing / misdirection of bridged funds** for the targeted token on the silo.

---

### Likelihood Explanation

`deploy_erc20_token` is a public NEAR contract method callable by any account with no deposit or permission requirement beyond the contract being live. An attacker only needs to know which NEP-141 tokens the silo owner intends to mirror (observable from the main Aurora contract's `nep141_erc20_map`) and front-run the `mirror_erc20_token` call. This is straightforward and requires no special privileges.

---

### Recommendation

Add a caller access control check to `deploy_erc20_token` analogous to the one already present in `mirror_erc20_token`:

```rust
require_owner_only(&state::get_state(&io)?, &env.predecessor_account_id())?;
```

Alternatively, restrict `deploy_erc20_token` to be callable only by the contract itself (private call) or by an explicit allowlist. This mirrors the fix applied in the Maia remediation: removing the redundant blocking check, or in this case, adding the missing access guard on the entry point that populates the map.

---

### Proof of Concept

1. Main Aurora contract has NEP-141 `usdt.near` bridged to ERC-20 at address `0xAAA...`.
2. Silo owner intends to call `mirror_erc20_token({ contract_id: "aurora", nep141: "usdt.near" })`.
3. **Before** the owner's transaction, attacker calls `deploy_erc20_token` on the silo with input `"usdt.near"`.
   - Silo deploys a new ERC-20 at address `0xBBB...` and writes `nep141_erc20_map["usdt.near"] = 0xBBB`.
4. Owner's `mirror_erc20_token` executes: callback fetches `0xAAA` from main contract, deploys code at `0xAAA` on silo, then calls `register_token(0xAAA, "usdt.near")`.
5. `register_token` finds `"usdt.near"` already mapped → returns `TokenAlreadyRegistered` → callback panics → transaction reverts.
6. The silo can never mirror `usdt.near`. All future `ft_on_transfer` calls for `usdt.near` on the silo mint tokens in the attacker's `0xBBB` ERC-20, not the legitimate `0xAAA`.
