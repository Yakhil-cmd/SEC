Audit Report

## Title
SNS Governance Voting Power Snapshot at Proposal Submission Time Allows Stake Manipulation Before Ballot Freeze - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister computes ballots using live neuron state at the exact moment `make_proposal` is called, with no periodic pre-committed snapshot baseline. An attacker who holds an SNS neuron can increase their staked tokens immediately before submitting a proposal, inflating their ballot voting power relative to what a pre-committed snapshot would have assigned. The NNS governance canister has an explicit mitigation (`SnapshotVotingPowerTask` + spike detection) that the SNS canister entirely lacks.

## Finding Description
`compute_ballots_for_new_proposal` in `rs/sns/governance/src/governance.rs` is called directly inside `make_proposal` and iterates over live neuron state at `now_seconds`: [1](#0-0) 

Both `cached_neuron_stake_e8s` (via `claim_or_refresh_neuron`) and `dissolve_delay_seconds` (via `IncreaseDissolveDelay`) are mutable by the neuron owner at any time. Because the IC processes ingress messages sequentially, an attacker can submit a `claim_or_refresh_neuron` call followed immediately by `make_proposal`; the ballot snapshot will reflect the inflated stake.

The NNS governance canister guards against this with a daily `SnapshotVotingPowerTask`: [2](#0-1) 

At proposal time, `compute_ballots_for_standard_proposal` compares current total voting power against the historical baseline and falls back to the pre-committed per-neuron snapshot if a spike is detected: [3](#0-2) 

A grep for `SnapshotVotingPower`, `snapshot_voting_power`, and `VotingPowerSnapshot` across all of `rs/sns/governance/src/` returns zero matches, confirming no equivalent mechanism exists in the SNS path.

## Impact Explanation
An attacker controlling an SNS neuron can transiently acquire a disproportionate share of voting power on a specific proposal by staking additional tokens immediately before submission, then dissolving after the voting period. For SNS DAOs governing treasury funds, canister upgrades, or parameter changes, this enables a per-proposal governance takeover without requiring a persistent majority stake. This matches the allowed impact: **High ($2,000–$10,000) — Significant SNS governance security impact with concrete user or protocol harm.**

## Likelihood Explanation
The attack requires no privileged access, no key compromise, and no subnet-majority corruption. The entry path is two sequential unprivileged ingress calls (`claim_or_refresh_neuron` then `make_proposal`). The attacker must hold or acquire SNS tokens and accept the dissolve delay lock-up, which is a capital and timing constraint but not a technical barrier. The attack is repeatable on any SNS with liquid token markets.

## Recommendation
Introduce a periodic voting power snapshot task for SNS governance analogous to `SnapshotVotingPowerTask` in `rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`. In `compute_ballots_for_new_proposal`, compare the current total potential voting power against the historical snapshot baseline. If a spike is detected (e.g., current > 1.5× historical minimum), use the historical per-neuron voting power for ballot creation instead of live state. The `VotingPowerSnapshots` structure at `rs/nns/governance/src/governance/voting_power_snapshots.rs` lines 35–38 can serve as a direct template. [4](#0-3) 

## Proof of Concept
1. Deploy a test SNS with a neuron `N` holding 100 tokens and a dissolve delay above the minimum voting threshold.
2. Transfer 900 additional SNS tokens to neuron `N`'s subaccount and call `claim_or_refresh_neuron`; confirm `cached_neuron_stake_e8s` updates to 1000 tokens.
3. Immediately call `make_proposal` on the SNS governance canister.
4. Inspect the created `ProposalData.ballots` for neuron `N`; confirm `voting_power` reflects the 1000-token stake rather than the pre-inflation 100-token stake.
5. Vote YES with neuron `N`; verify the proposal passes even if all other neurons (collectively holding fewer than the inflated share) vote NO.
6. A deterministic PocketIC integration test can encode steps 1–5 in a single test, controlling message ordering to guarantee the stake update precedes ballot creation.

impl RecurringSyncTask for SnapshotVotingPowerTask {
    fn execute(self) -> (Duration, Self) {
        let now_seconds = self
            .governance
            .with_borrow(|governance| governance.env.now());
        if self
            .snapshots
            .with_borrow(|snapshots| snapshots.is_latest_snapshot_a_spike(now_seconds))
        {
            return (VOTING_POWER_SNAPSHOT_INTERVAL, self);
        }

        let voting_power_snapshot = self.governance.with_borrow_mut(|governance| {
            let voting_power_economics = governance.voting_power_economics();
            governance
                .neuron_store
                .compute_voting_power_snapshot_for_standard_proposal(
                    voting_power_economics,
                    now_seconds,
                )
                .expect("Voting power snapshot failed")
        });

        self.snapshots.with_borrow_mut(|snapshots| {
            snapshots.record_voting_power_snapshot(now_seconds, voting_power_snapshot);
        });
