### Title
Floor Division in `delta_gas` Calculation Systematically Under-Charges Users for Native Resource Consumption — (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` adjustment that compensates the protocol when native resource consumption exceeds EVM-gas consumption uses Rust's default truncating (floor) integer division. This rounds the implied gas from native consumption **down**, allowing users to receive a larger gas refund than the protocol is entitled to. The rounding error is up to 1 gas unit per transaction, and the direction of the error consistently favors the user over the protocol.

---

### Finding Description

The double-resource accounting model documented in `docs/double_resource_accounting.md` defines:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
```

If `deltaGas > 0`, the extra gas is added to `gas_used` so the user pays for the native work performed. The implementation in `compute_gas_refund` is:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs, line 69-79
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)   // ← floor division
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

`native_used / native_per_gas` is Rust integer division, which truncates toward zero (floor). For any `native_used = k * native_per_gas + r` with `0 < r < native_per_gas`, the floor result is `k` while the mathematically correct value is `k + r/native_per_gas`. The discarded fractional part `r/native_per_gas` is always in `(0, 1)`, so `delta_gas` is computed as **at most 1 less** than the true value.

Concretely, when the true `delta_gas` is in the range `(N, N+1)` for any non-negative integer N, the computed value is N. The user pays N extra gas instead of the correct ~N+1, receiving up to 1 gas unit of computation for free.

The rounding direction of every other division in the same function is correct:

- `gas_used` computation: `resources.ergs().0.div_floor(ERGS_PER_GAS)` — remaining gas rounded **down** → `gas_used` rounded **up** → favors protocol. [1](#0-0) 
- `evm_refund` cap: `gas_used / 5` — floor division → user gets slightly **less** refund → favors protocol. [2](#0-1) 
- `full_refund_gas`: `full_refund_ergs.0.div_floor(ERGS_PER_GAS)` — rounds **down** → favors protocol. [3](#0-2) 

Only the `delta_gas` division rounds in the user's favor. [4](#0-3) 

The `native_per_gas` value itself is computed with `div_ceil` (correct direction) during validation: [5](#0-4) 

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

So the conversion rate is correctly rounded up, but the back-conversion at refund time is incorrectly rounded down, creating an asymmetry that always benefits the user.

The documentation explicitly acknowledges the formula without specifying rounding direction, and a `// TODO: return delta_gas to gas_used?` comment in the same block signals unresolved uncertainty about the accounting: [6](#0-5) 

---

### Impact Explanation

For every transaction where native resource consumption dominates EVM gas consumption (`delta_gas > 0`), the user receives a gas refund of up to 1 extra gas unit. The monetary value of this discount is up to `gas_price` wei per transaction.

- At `gas_price = 1 Gwei`: up to 1 Gwei per transaction.
- At `gas_price = 100 Gwei`: up to 100 Gwei per transaction.

The protocol (operator/sequencer) bears this shortfall on every such transaction. While the per-transaction loss is bounded, the rounding error is **systematic and deterministic**: an attacker who can craft transactions so that `native_used mod native_per_gas = native_per_gas - 1` maximizes the discount on every submission. The attacker does not profit net (transaction fees exceed the discount), but the protocol consistently under-collects native-resource compensation, which is a resource accounting invariant violation.

---

### Likelihood Explanation

The condition `delta_gas > 0` is reached whenever native resource consumption is the binding constraint rather than EVM gas — a normal operating condition for transactions that perform many storage writes, precompile calls, or generate significant pubdata. Any unprivileged user submitting a standard EVM transaction can trigger this path. No special access or privileged role is required. The entry path is: submit any EVM or ZK transaction → `compute_gas_refund` is called unconditionally for every processed transaction. [7](#0-6) 

---

### Recommendation

Replace the floor division with ceiling division in the `delta_gas` computation:

```rust
// Before (floor — favors user):
(native_used / native_per_gas) as i64 - (gas_used as i64)

// After (ceiling — favors protocol, consistent with nativePerGas derivation):
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

This is consistent with the `div_ceil` already used when computing `native_per_gas` from `gas_price / native_price` during validation, and ensures the protocol is never under-compensated for native resource consumption. Document the expected rounding direction for each arithmetic operation in the resource accounting pipeline.

### Proof of Concept

**Setup:**
- `gas_price = 10^9` (1 Gwei), `native_price = 1`
- `native_per_gas = ceil(10^9 / 1) = 10^9` (computed correctly with `div_ceil`)
- `gas_limit = 100_000`, `gas_used (ergs-based) = 50_000`

**Crafted native consumption:**
- Alice's transaction consumes `native_used = 50_000 * 10^9 + (10^9 - 1)` native units
  (i.e., exactly `native_per_gas - 1` remainder)

**Computed `delta_gas`:**
```
floor(native_used / native_per_gas)
  = floor((50_000 * 10^9 + 999_999_999) / 10^9)
  = 50_000          ← floor drops the remainder
```
`delta_gas = 50_000 - 50_000 = 0` → no extra gas charged.

**True `delta_gas`:**
```
(50_000 * 10^9 + 999_999_999) / 10^9 = 50_000.999999999
delta_gas_true ≈ 0.999999999
```
→ Alice should pay ~1 extra gas unit (worth 1 Gwei) but pays 0.

Alice receives a 1-gas-unit discount (1 Gwei) on this transaction. Repeated across many transactions, the protocol systematically under-collects native-resource compensation. [8](#0-7)
