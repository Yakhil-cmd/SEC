### Title
Block-Level L2-to-L1 Log Limit Exhaustion Enables Per-Block DoS of L2-to-L1 Messaging - (`zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

ZKsync OS enforces a hard-coded per-block cap of `MAX_NUMBER_OF_LOGS = 16_384` on L2-to-L1 logs. An unprivileged attacker can exhaust this limit within a single block by submitting transactions that each emit L2-to-L1 messages via the L1 Messenger system hook. Once the cap is reached, every subsequent transaction in that block that attempts to emit an L2-to-L1 log is rejected with `BlockL2ToL1LogsLimitReached`. This blocks all L2-to-L1 messaging — including user withdrawal proofs — for the duration of that block, and the attack can be repeated every block.

---

### Finding Description

**Root cause — hard-coded, non-priced per-block log cap:**

`MAX_NUMBER_OF_LOGS` is defined as a compile-time constant derived from the fixed-size (16 384-leaf) Merkle tree used for the L2-to-L1 log commitment:

```rust
pub const MAX_NUMBER_OF_LOGS: u64 = 16_384;
// Taken from the size of the Merkle tree.
``` [1](#0-0) 

After every successfully-executed transaction, `check_for_block_limits` compares the running log count against this constant:

```rust
} else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
    Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
}
``` [2](#0-1) 

`logs_used` is read directly from the live log storage (`system.io.logs_len()`), which counts every committed log in the block so far: [3](#0-2) 

When the limit is exceeded the offending transaction is rolled back and the block loop continues — but the limit remains saturated, so every subsequent transaction that would add even one log is also rejected: [4](#0-3) 

**Attacker entry path — unprivileged `sendToL1` calls:**

Any EOA or contract can call the L1 Messenger system contract (`0x8008`), which in turn calls the L1 Messenger hook (`0x7001`). The hook records the message as an L2-to-L1 log:

```rust
system.io.emit_l1_message(
    ExecutionEnvironmentType::NoEE,
    resources,
    &address_sender,
    message,
)?;
``` [5](#0-4) 

Each such call costs approximately 9 202 EVM gas (charged by the L1Messenger EVM bytecode; the hook itself charges 0 ergs): [6](#0-5) 

**Why the limit is disproportionately cheap to exhaust:**

ZKsync OS is designed for high throughput. The block gas limit is operator-set and can be orders of magnitude larger than standard Ethereum (the test default is `u64::MAX / 256 ≈ 72 × 10¹⁵`): [7](#0-6) 

Filling all 16 384 log slots costs only `16 384 × 9 202 ≈ 150 M gas`. If the block gas limit is set to even 1 billion gas (a conservative high-throughput value), the attacker consumes only ~15 % of the block gas budget to permanently saturate the log limit for that block. The remaining 85 % of gas capacity is irrelevant to L2-to-L1 messaging — those slots are simply gone.

Unlike the Linea rate-limiter attack (where the attacker can bridge ETH back to recover funds), the attacker here pays gas that is not refunded. However, the attack is still economically viable for a well-funded adversary because the cost-to-damage ratio is highly asymmetric: the attacker spends ~150 M gas to deny all other users' L2-to-L1 messaging for an entire block, and can repeat this every block.

---

### Impact Explanation

- **L2-to-L1 messaging DoS**: Any transaction that calls `sendToL1` after the cap is reached is rejected with `BlockL2ToL1LogsLimitReached`. This includes user withdrawal proofs, cross-chain messages, and any protocol-level L2-to-L1 communication.
- **Repeatable per-block**: The limit resets each block, so the attacker must re-execute the attack each block, but there is no protocol-level mechanism to prevent this.
- **Selective resource exhaustion**: Because the log limit is independent of the gas limit, the attacker can exhaust it while leaving most of the block gas budget unused, making the attack cheaper than full block stuffing.

---

### Likelihood Explanation

- No privileged access is required; any EOA can call the L1 Messenger.
- The attack requires sustained gas expenditure (~150 M gas/block), making it costly but feasible for a well-funded adversary targeting a specific window (e.g., during a large withdrawal event).
- The attack is deterministic and repeatable every block.

---

### Recommendation

1. **Price L2-to-L1 logs proportionally to their share of the block log budget**, similar to how EIP-4844 blob gas is priced independently of execution gas. This makes exhausting the log limit as expensive as exhausting the gas limit.
2. **Introduce a per-transaction log cap** so that no single transaction can consume a disproportionate share of the block log budget.
3. **Consider dynamic log pricing** (analogous to EIP-1559 base fee) that increases as the block log budget fills, creating an economic deterrent against log-stuffing.

---

### Proof of Concept

1. Attacker deploys a contract with a loop that calls `sendToL1(bytes)` on `0x8008` repeatedly.
2. Attacker submits transactions invoking this contract until the cumulative log count in the block reaches 16 384.
3. All subsequent transactions in the same block that attempt to emit an L2-to-L1 log are rejected by `check_for_block_limits` with `BlockL2ToL1LogsLimitReached`.
4. Legitimate users attempting withdrawals or cross-chain messages in that block are denied.
5. The attacker repeats this at the start of each new block.
