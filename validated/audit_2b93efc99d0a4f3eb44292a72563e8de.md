After a thorough investigation of the sei-chain codebase, I found a directly analogous vulnerability to the external report.

---

### Title
Missing Inbound/Outbound Enabled Guard on IBC Handshake Completion Steps - (`sei-ibc-go/modules/core/keeper/msg_server.go`)

### Summary
Sei's IBC module implements a circuit-breaker via `InboundEnabled` / `OutboundEnabled` params. The Init and Try steps of connection/channel handshakes are correctly gated, but the Ack and Confirm completion steps are not, allowing any unprivileged relayer to finalize IBC connections and channels even when the operator has disabled IBC.

### Finding Description

The IBC keeper in `sei-ibc-go/modules/core/keeper/msg_server.go` applies the circuit-breaker inconsistently across the four-step handshake:

| Handler | Check present? |
|---|---|
| `ConnectionOpenInit` | `IsOutboundEnabled` ✓ |
| `ConnectionOpenTry` | `IsInboundEnabled` ✓ |
| `ConnectionOpenAck` | **none** |
| `ConnectionOpenConfirm` | **none** |
| `ChannelOpenInit` | `IsOutboundEnabled` ✓ |
| `ChannelOpenTry` | `IsInboundEnabled` ✓ |
| `ChannelOpenAck` | **none** |
| `ChannelOpenConfirm` | **none** |
| `RecvPacket` | `IsInboundEnabled` ✓ |

`ConnectionOpenInit` and `ChannelOpenInit` are gated: [1](#0-0) [2](#0-1) 

`ConnectionOpenAck` (lines 141–158) and `ChannelOpenAck` (lines 266–297) have **no** enabled check: [3](#0-2) [4](#0-3) 

`ConnectionOpenConfirm` (lines 160–171) and `ChannelOpenConfirm` (lines 299–331) are also ungated: [5](#0-4) [6](#0-5) 

The same asymmetry exists at the lower-level keeper layer in `handshake.go` — `ChanOpenInit` and `ChanOpenTry` check the flag, but `ChanOpenAck` and `ChanOpenConfirm` do not: [7](#0-6) [8](#0-7) 

The params keeper exposes the flags: [9](#0-8) 

The intentional exemptions (settlement operations) are explicitly tested: [10](#0-9) 

No equivalent tests exist for `ConnectionOpenAck`, `ConnectionOpenConfirm`, `ChannelOpenAck`, or `ChannelOpenConfirm` being allowed when disabled — unlike `Timeout`/`Acknowledgement`, these are not documented as intentional exemptions.

### Impact Explanation

An unprivileged relayer can complete an in-progress IBC connection or channel handshake even after the operator has set `InboundEnabled=false` or `OutboundEnabled=false`. This establishes live connections/channels in chain state that the operator intended to block. **Direct fund movement is still prevented** because `SendPacket` and `RecvPacket` retain their own enabled checks, so this does not directly cause fund loss. The harm is state inconsistency: the chain records open connections/channels that should not exist under the disabled state, which could be exploited if packet-level guards are later relaxed or bypassed.

### Likelihood Explanation

Requires a handshake to be in the `INIT` or `TRYOPEN` state at the moment the operator disables IBC. Any relayer (no special privilege needed) can then submit the Ack/Confirm message to complete it. This is a realistic scenario during emergency response, where operators disable IBC while handshakes are in flight.

### Recommendation

Add the same enabled checks to the Ack and Confirm steps:

- `ConnectionOpenAck` → check `IsOutboundEnabled` (chain A is completing its outbound handshake)
- `ConnectionOpenConfirm` → check `IsInboundEnabled` (chain B is confirming an inbound connection)
- `ChannelOpenAck` → check `IsOutboundEnabled`
- `ChannelOpenConfirm` → check `IsInboundEnabled`

If completing in-progress handshakes is intentionally allowed (analogous to Timeout/Acknowledgement), add explicit tests documenting that intent, matching the pattern in `inbound_test.go`.

### Proof of Concept

1. Chain A calls `ConnectionOpenInit` → connection enters `INIT` state.
2. Chain B calls `ConnectionOpenTry` → connection enters `TRYOPEN` state.
3. Operator sets `OutboundEnabled=false` on chain A (emergency).
4. Relayer calls `ConnectionOpenAck` on chain A — **no enabled check**, succeeds.
5. Relayer calls `ConnectionOpenConfirm` on chain B — **no enabled check**, succeeds.
6. A fully `OPEN` IBC connection now exists despite the circuit-breaker being active.

The same four-step path applies to channel handshakes via `ChannelOpenInit` → `ChannelOpenTry` → `ChannelOpenAck` → `ChannelOpenConfirm`.

func (k Keeper) ChannelOpenAck(goCtx context.Context, msg *channeltypes.MsgChannelOpenAck) (*channeltypes.MsgChannelOpenAckResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)

	// Lookup module by channel capability
	module, cap, err := k.ChannelKeeper.LookupModuleByChannel(ctx, msg.PortId, msg.ChannelId)
	if err != nil {
		return nil, sdkerrors.Wrap(err, "could not retrieve module from port-id")
	}

	// Retrieve application callbacks from router
	cbs, ok := k.Router.GetRoute(module)
	if !ok {
		return nil, sdkerrors.Wrapf(porttypes.ErrInvalidRoute, "route not found to module: %s", module)
	}

	// Perform 04-channel verification
	if err = k.ChannelKeeper.ChanOpenAck(
		ctx, msg.PortId, msg.ChannelId, cap, msg.CounterpartyVersion, msg.CounterpartyChannelId, msg.ProofTry, msg.ProofHeight,
	); err != nil {
		return nil, sdkerrors.Wrap(err, "channel handshake open ack failed")
	}

	// Perform application logic callback
	if err = cbs.OnChanOpenAck(ctx, msg.PortId, msg.ChannelId, msg.CounterpartyChannelId, msg.CounterpartyVersion); err != nil {
		return nil, sdkerrors.Wrap(err, "channel open ack callback failed")
	}

	// Write channel into state
	k.ChannelKeeper.WriteOpenAckChannel(ctx, msg.PortId, msg.ChannelId, msg.CounterpartyVersion, msg.CounterpartyChannelId)

	return &channeltypes.MsgChannelOpenAckResponse{}, nil
}

	ctx := sdk.UnwrapSDKContext(goCtx)

	if err != nil {

	// Retrieve application callbacks from router
	cbs, ok := k.Router.GetRoute(module)
	if !ok {
		return nil, sdkerrors.Wrapf(porttypes.ErrInvalidRoute, "route not found to module: %s", module)
	}

	// Perform 04-channel verification
	if err = k.ChannelKeeper.ChanOpenConfirm(ctx, msg.PortId, msg.ChannelId, cap, msg.ProofAck, msg.ProofHeight); err != nil {
		return nil, sdkerrors.Wrap(err, "channel handshake open confirm failed")

	k.ChannelKeeper.WriteOpenConfirmChannel(ctx, msg.PortId, msg.ChannelId)

	return &channeltypes.MsgChannelOpenConfirmResponse{}, nil

// TestAcknowledgementAllowedWhenOutboundDisabled verifies that MsgAcknowledgement
// succeeds even when OutboundEnabled == false (settlement must be allowed).
func (suite *KeeperTestSuite) TestAcknowledgementAllowedWhenOutboundDisabled() {
	suite.SetupTest()

	path := ibctesting.NewPath(suite.chainA, suite.chainB)
	suite.coordinator.Setup(path)

	// send packet from A -> B
	timeoutHeight := suite.chainB.GetTimeoutHeight()
	packet := channeltypes.NewPacket(ibctesting.MockPacketData, 1,
		path.EndpointA.ChannelConfig.PortID, path.EndpointA.ChannelID,
		path.EndpointB.ChannelConfig.PortID, path.EndpointB.ChannelID,
		timeoutHeight, 0,
	)
	err := path.EndpointA.SendPacket(packet)
	suite.Require().NoError(err)

	// receive packet on B (creates ack)
	err = path.EndpointB.RecvPacket(packet)
	suite.Require().NoError(err)

	// disable outbound on chainA
	ibcKeeperA := suite.chainA.App.GetIBCKeeper()
	ibcKeeperA.SetOutboundEnabled(suite.chainA.GetContext(), false)
	suite.Require().False(ibcKeeperA.IsOutboundEnabled(suite.chainA.GetContext()))

	// ack should still succeed
	err = path.EndpointA.AcknowledgePacket(packet, ibctesting.MockAcknowledgement)
	suite.Require().NoError(err, "MsgAcknowledgement should succeed when outbound is disabled")
}
