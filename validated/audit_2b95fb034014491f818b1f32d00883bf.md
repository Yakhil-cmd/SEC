### Title
Inconsistent `MAX_BLOBS_PER_BLOCK` Constant Causes Wrong Per-Transaction Blob Limit in ZK/L2 Transaction Validation - (File: `zk_ee/src/system/constants.rs` vs `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs`)

---

### Summary

There are **two different definitions** of `MAX_BLOBS_PER_BLOCK` in the codebase with conflicting values: `6` in `zk_ee/src/system/constants.rs` and `9` in `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` (and `block_header.rs`). The ZK/L2 transaction validation path imports the value `6` from `zk_ee`, while the Ethereum block header path uses `9`. This means the ZK STF enforces a tighter per-transaction blob limit (6) than the Ethereum block header allows (9), creating a divergence between what the Ethereum block header accepts and what the ZK transaction validator permits.

---

### Finding Description

**Two conflicting constant definitions:**

1. `zk_ee/src/system/constants.rs` line 31:
   ```rust
   pub const MAX_BLOBS_PER_BLOCK: usize = 6;
   ```
   This is the value used by the ZK/L2 transaction validation path and the ZK metadata.

2. `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` line 9:
   ```rust
   pub const MAX_BLOBS_PER_BLOCK: usize = 9;
   ```
   And `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs` line 30:
   ```rust
   const MAX_BLOBS_PER_BLOCK: usize = 9;
   ```
   These are used by the Ethereum block header parsing and blob gas limit computation.

**How the ZK/L2 validation path uses the wrong (smaller) constant:**

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` line 35:
```rust
use zk_ee::system::{GAS_PER_BLOB, MAX_BLOBS_PER_BLOCK};
```
This imports `MAX_BLOBS_PER_BLOCK = 6` from `zk_ee`.

Then at line 411:
```rust
match parse_blobs_list::<MAX_BLOBS_PER_BLOCK>(blobs_list) {
```

The `parse_blobs_list` function in `basic_bootloader/src/bootloader/transaction/blobs.rs` line 11:
```rust
if blobs_list.count > MAX_BLOBS_IN_TX {
    return Err(TxError::Validation(InvalidTransaction::BlobListTooLong));
}
```

So a transaction with 7, 8, or 9 blobs тАФ which is valid per the Ethereum block header's `MAX_BLOBS_PER_BLOCK = 9` тАФ will be **rejected** by the ZK/L2 transaction validator with `BlobListTooLong`.

Similarly, the `TxLevelMetadata` in `zk_ee/src/system/metadata/zk_metadata.rs` line 28:
```rust
pub blobs: arrayvec::ArrayVec<Bytes32, { MAX_BLOBS_PER_BLOCK }>,
```
This array is sized to only 6 blobs, so even if the check were bypassed, the data structure would overflow.

**The Ethereum validation path uses the correct value:**

In `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs` line 334:
```rust
match parse_blobs_list::<MAX_BLOBS_PER_BLOCK>(blobs_list) {
```
Here `MAX_BLOBS_PER_BLOCK` is imported from the local `ethereum` module (value = 9), so the Ethereum path correctly allows up to 9 blobs.

---

### Impact Explanation

**State-transition divergence / EVM semantic mismatch:**

- A transaction carrying 7тАУ9 blobs is valid under Ethereum consensus rules (post-Pectra, `MAX_BLOBS_PER_BLOCK = 9`).
- The Ethereum block header path in ZKsync OS accepts such a block (uses `9`).
- The ZK/L2 transaction validation path rejects such a transaction with `BlobListTooLong` (uses `6`).
- This creates a **forward/proving divergence**: the forward (sequencer) system and the proving system may disagree on whether a transaction is valid, depending on which constant is in scope.
- Additionally, the `EthereumBlockMetadata` type is parameterized with `MAX_BLOBS_PER_BLOCK` from `zk_ee` (= 6), meaning the metadata struct can only hold 6 blob hashes even for blocks that the header parser accepts as having up to 9.
- An attacker who can submit a transaction with 7тАУ9 blobs to a ZKsync OS node running the Ethereum block flow (which accepts 9) would have that transaction accepted at the block level but rejected at the ZK proving level, causing a proof failure or forced block invalidation.

---

### Likelihood Explanation

- EIP-4844 blob transactions with 7тАУ9 blobs are valid on Ethereum post-Pectra (fork block 22431084 is hardcoded in the codebase).
- Any unprivileged user can submit such a transaction.
- The divergence is triggered automatically whenever a blob transaction with more than 6 blobs is included in a block that ZKsync OS processes in Ethereum-runner mode.
- The inconsistency is structural (two separate constant definitions with different values), not conditional on any configuration flag.

---

### Recommendation

Unify `MAX_BLOBS_PER_BLOCK` to a single authoritative constant. Since the Ethereum block header parser and blob gas accounting already use `9` (post-Pectra), the `zk_ee/src/system/constants.rs` value should be updated to match:

```rust
// zk_ee/src/system/constants.rs
pub const MAX_BLOBS_PER_BLOCK: usize = 9;
```

All downstream uses тАФ `TxLevelMetadata`, `parse_blobs_list`, `EthereumBlockMetadata` тАФ will then consistently allow up to 9 blobs per block/transaction, matching the Ethereum consensus rule.

---

### Proof of Concept

1. Construct an EIP-4844 transaction with 7 blob versioned hashes (all valid `0x01`-prefixed KZG versioned hashes).
2. Submit it to a ZKsync OS node running in Ethereum block flow mode.
3. The Ethereum block header parser accepts the block (uses `MAX_BLOBS_PER_BLOCK = 9`).
4. The ZK/L2 transaction validator calls `parse_blobs_list::<6>(blobs_list)` and returns `InvalidTransaction::BlobListTooLong` because `7 > 6`.
5. The transaction is rejected at the ZK proving level despite being valid at the block header level, causing a forward/proving divergence.
