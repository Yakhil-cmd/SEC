### Title
SNS Swap Token Distribution Dust Accumulation via Integer Division Truncation in `Swap::scale` - (File: `rs/sns/swap/src/swap.rs`)

### Summary

The SNS Swap canister's `create_sns_neuron_recipes` function distributes SNS tokens to participants by calling `Swap::scale`, which performs integer division `(amount_icp_e8s * total_sns_e8s) / total_icp_e8s`. Because this is floor division applied independently per participant, the sum of all per-participant allocations is strictly less than or equal to `sns_token_e8s`. The difference — up to `(N-1)` e8s for N participants — is permanently stranded in the Swap canister's SNS ledger account after finalization, with no recovery path.

### Finding Description

`Swap::scale` computes each participant's SNS token allocation as:

```
amount_sns_e8s = (buyer_icp_e8s * total_sns_e8s) / total_icp_e8s
```

using 128-bit integer arithmetic that truncates (floors) the result. [1](#0-0) 

This is called once per direct participant and once per Neurons' Fund neuron inside `create_sns_neuron_recipes`: [2](#0-1) [3](#0-2) 

The code itself logs the discrepancy as `"change"` but takes no action to recover it:

```rust
sns_being_offered_e8s - total_sns_tokens_sold_e8s
``` [4](#0-3) 

After `sweep_sns` transfers each recipe's tokens to the SNS governance neuron subaccounts, the Swap canister's own SNS ledger balance is expected to reach zero in the happy path. However, the rounding remainder is never transferred anywhere — it stays in the Swap canister's SNS ledger account permanently.

The integration test at `rs/nervous_system/integration_tests/tests/sns_lifecycle.rs` asserts `swap_canister_balance_sns_e8s == 0` after a committed swap, but this assertion only holds when the ICP amounts divide evenly into the SNS token pool. With arbitrary participant amounts, the remainder is non-zero. [5](#0-4) 

The analog to the WLSETH bug is exact: just as WLSETH's `burn(_value)` takes ETH as the unit and cannot recover the exact deposited lsETH shares due to rounding, the SNS Swap's `scale` function takes ICP as the unit and cannot distribute the exact offered SNS tokens due to per-participant floor division.

### Impact Explanation

- **Token conservation violation**: SNS tokens offered for the swap are permanently locked in the Swap canister's account after finalization. The Swap canister has no `withdraw` or `reclaim` endpoint for its SNS ledger balance post-finalization.
- **Magnitude**: With N participants, up to `N - 1` e8s (i.e., up to `N - 1` × 10⁻⁸ SNS tokens) are stranded per swap. For a swap with 10,000 participants, up to 9,999 e8s (~0.0001 SNS tokens) are permanently lost. For high-value SNS tokens this is a real, if small, financial loss to the SNS project.
- **Accumulation**: Every SNS swap that does not have perfectly divisible ICP contributions leaves dust. This is a systemic, deterministic loss across all SNS launches.
- **No recovery**: The Swap canister's `finalize` flow has no step to return the remainder to the SNS treasury or governance canister.

### Likelihood Explanation

- **Certainty**: This occurs in virtually every real-world SNS swap. ICP contributions from N independent participants will almost never sum to a value that divides `sns_token_e8s` evenly for every participant simultaneously.
- **Reachable by unprivileged users**: Any user who participates in an SNS swap triggers this path. No special role is needed.
- **Deterministic**: The rounding loss is a mathematical certainty given the integer arithmetic used.

### Recommendation

After computing all per-participant `amount_sns_e8s` values, calculate the remainder:

```rust
let remainder_e8s = sns_being_offered_e8s - total_sns_tokens_sold_e8s;
```

Then transfer `remainder_e8s` to the SNS governance treasury account (or the largest participant's allocation) as part of `sweep_sns`, ensuring full conservation of the offered token pool. Alternatively, assign the remainder to the last participant processed (similar to how `apportion_approximately_equally` handles remainders). [6](#0-5) 

### Proof of Concept

**Setup**: SNS swap with `sns_token_e8s = 10` and two participants:
- Participant A: `amount_icp_e8s = 3`, `total_icp_e8s = 10`
- Participant B: `amount_icp_e8s = 7`, `total_icp_e8s = 10`

**Computation**:
```
A gets: (3 * 10) / 10 = 3 SNS e8s
B gets: (7 * 10) / 10 = 7 SNS e8s
Total distributed: 10 SNS e8s  ← exact in this case
```

**Non-trivial case**: `sns_token_e8s = 10`, three participants each with `amount_icp_e8s = 1`, `total_icp_e8s = 3`:
```
Each gets: (1 * 10) / 3 = 3 SNS e8s  (floor of 3.333...)
Total distributed: 9 SNS e8s
Remainder stranded in Swap canister: 1 SNS e8s  ← permanently locked
```

The `create_sns_neuron_recipes` log line confirms this is known but unhandled: [7](#0-6) 

The `apportion_approximately_equally` function used for neuron basket splitting correctly handles remainders by distributing them to the last elements, but `Swap::scale` — used for the cross-participant distribution — does not apply any equivalent correction. [8](#0-7)

/// Chops up `total` in to `len` pieces.
///
/// More precisely, result.len() == len. result.sum() == total. Each element of
/// result is approximately equal to the others. However, unless len divides
/// total evenly, the elements of result will inevitably be not equal.
///
/// There are two ways that Err can be returned:
///
///   1. Caller mistake: len == 0
///
///   2. This has a bug. See implementation comments for why we know of know way
///      this can happen, but can detect if it does.
pub fn apportion_approximately_equally(total: u64, len: u64) -> Result<Vec<u64>, String> {
    let quotient = total
        .checked_div(len)
        .ok_or_else(|| format!("Unable to divide total={total} by len={len}"))?;
    let remainder = total % len; // For unsigned integers, % cannot overflow.

    // So far, we have only apportioned quotient * len. To reach the desired
    // total, we must still somehow add remainder (per Euclid's Division
    // Theorem). That is accomplished right after this.
    let mut result = vec![quotient; len as usize];

    // Divvy out the remainder: Starting from the last element, increment
    // elements by 1. The number of such increments performed here is remainder,
    // bringing our total back to the desired amount.
    if remainder >= result.len() as u64 {
        return Err(format!("Could not apportion {total} into {len} pieces"));
    }
    let mut iter_mut = result.iter_mut();
    for _ in 0..remainder {
        let element: &mut u64 = iter_mut
            .next_back()
            // We can prove that this will not panic:
            // The number of iterations of this loop is total % len.
            // This must be < len (by Euclid's Division Theorem).
            // Thus, the number of iterations that this loop goes through is < len.
            // Thus, the number of times next_back is called is < len.
            // next_back only returns None after len calls.
            // Therefore, next_back does not return None here.
            // Therefore, this expect will never panic.
            .ok_or_else(
                || format!("Ran out of elements to increment. total={total}, len={len}",),
            )?;

        // This cannot overflow because the result must be <= total. Thus, this
        // will not panic.
        *element = element.checked_add(1).ok_or_else(|| {
            format!("Incrementing element by 1 resulted in overflow. total={total}, len={len}",)
        })?;
    }

    Ok(result)
}
