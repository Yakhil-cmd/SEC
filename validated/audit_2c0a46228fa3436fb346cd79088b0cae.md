Audit Report

## Title
SNS Swap `refresh_buyer_tokens` Accepts SNS Governance Canister as Buyer, Enabling SNS Token Dilution and Governance Inflation - (File: rs/sns/swap/canister/canister.rs, rs/sns/swap/src/swap.rs)

## Summary
The `refresh_buyer_tokens` endpoint accepts any arbitrary principal as the `buyer` field, including the SNS governance canister, without any identity-conflict guard. An unprivileged attacker can deposit ICP to the SNS governance canister's subaccount on the swap canister and register it as a direct participant, causing proportional SNS token dilution for legitimate buyers and creating unintended neuron baskets controlled by the SNS governance canister.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 128–134, the handler parses `arg.buyer` as any `PrincipalId` without restriction and passes it directly to `refresh_buyer_token_e8s`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
```

In `rs/sns/swap/src/swap.rs` lines 1134–1312, `refresh_buyer_token_e8s` performs only these checks before inserting the principal into `self.buyers`:
- Lifecycle is `Open`
- Direct participation is possible
- Confirmation text matches
- Subaccount balance meets `min_participant_icp_e8s`
- Maximum participant count not exceeded

There is no guard of the form `buyer != sns_governance_canister_id`. An attacker transfers `min_participant_icp_e8s` ICP to `Account { owner: SWAP_CANISTER_ID, subaccount: Some(principal_to_subaccount(&sns_governance_canister_id)) }` on the ICP ledger, then calls `refresh_buyer_tokens` with `buyer = sns_governance_canister_id`. The balance check passes, and the SNS governance canister is inserted into `self.buyers` at lines 1285–1291.

During `sweep_icp` (lines 2046–2154) on a COMMITTED swap, the destination for every buyer's ICP is hardcoded to `Account { owner: sns_governance, subaccount: None }`. When the buyer is `sns_governance`, the source is `Account { owner: swap, subaccount: principal_to_subaccount(sns_governance) }` and the destination is `Account { owner: sns_governance, subaccount: None }` — these are distinct accounts, so the ICP ledger transfer succeeds and increments `sweep_result.success`.

During `sweep_sns` (lines 2165–2300), SNS tokens are distributed proportionally across all entries in `self.neuron_recipes`, which includes recipes generated for the SNS governance canister as a `DirectInvestment` buyer. The SNS governance canister receives SNS tokens staked into neuron baskets at `Account { owner: sns_governance, subaccount: Some(compute_neuron_staking_subaccount_bytes(sns_governance_principal, memo)) }`. Legitimate buyers' proportional share is reduced accordingly.

## Impact Explanation
This matches **High ($2,000–$10,000): Significant SNS security impact with concrete user or protocol harm.**

- **SNS token dilution**: Legitimate buyers receive fewer SNS tokens than their ICP contribution entitles them to. The dilution is directly proportional to the attacker's deposited ICP relative to total swap participation — a concrete, measurable financial loss.
- **Governance inflation**: Neuron baskets are created for the SNS governance canister principal. These neurons carry voting weight within the SNS, granting the governance canister unintended influence. Because the SNS governance canister is itself the subject of governance proposals, this creates a structurally circular voting weight.
- **Premature swap commitment**: The attacker's deposit increments `direct_participation_icp_e8s`. If it pushes the total past `max_direct_participation_icp_e8s`, the swap commits immediately, closing participation before the intended window ends.

## Likelihood Explanation
Both steps are permissionless and reachable by any unprivileged ingress sender during the `Open` lifecycle. The attacker's ICP is not stolen — it is swept to the SNS governance canister's treasury (the same destination as all other buyers' ICP in a committed swap), making this a griefing/dilution attack. The cost to the attacker is the ICP deposited plus transfer fees. Likelihood is **low** for large-scale attacks (requires significant ICP for meaningful dilution) but **medium** for targeted disruption of a specific SNS launch, particularly to trigger premature commitment or to inflate governance canister voting weight.

## Recommendation
Add identity-conflict guards at the top of `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`, before the async ICP ledger call, to reject reserved system principals:

```rust
let init = self.init_or_panic();
let sns_governance_id = PrincipalId::from_str(&init.sns_governance_canister_id).unwrap();
let nns_governance_id = PrincipalId::from_str(&init.nns_governance_canister_id).unwrap();
if buyer == sns_governance_id
    || buyer == nns_governance_id
    || buyer == this_canister.get()
{
    return Err(format!(
        "Principal {} is a system canister and cannot participate as a buyer.",
        buyer
    ));
}
```

This mirrors the standard role-collision guard pattern and should be placed before the `icp_ledger.account_balance` await to avoid unnecessary async calls.

## Proof of Concept
1. Read `sns_governance_canister_id` from `swap.get_state()` (public query).
2. Transfer `min_participant_icp_e8s` ICP to `Account { owner: SWAP_CANISTER_ID, subaccount: Some(principal_to_subaccount(&sns_governance_canister_id)) }` on the ICP ledger.
3. Call `refresh_buyer_tokens({ buyer: sns_governance_canister_id.to_text(), confirmation_text: None })` on the swap canister. The call succeeds because the subaccount balance meets the minimum.
4. Assert `swap.buyers` contains an entry for `sns_governance_canister_id` with `amount_icp_e8s = min_participant_icp_e8s` and `direct_participation_icp_e8s` is incremented.
5. Commit the swap (or wait for auto-commit). Run `finalize_swap`:
   - `sweep_icp`: transfer from `Account { owner: swap, subaccount: principal_to_subaccount(sns_governance) }` → `Account { owner: sns_governance, subaccount: None }` succeeds; `sweep_result.success` increments.
   - `sweep_sns`: SNS tokens are distributed to neuron recipes including the `DirectInvestment` recipe for `sns_governance_canister_id`; legitimate buyers receive proportionally fewer tokens.
   - `claim_swap_neurons`: neuron baskets are created for `sns_governance_canister_id`.

A deterministic unit test in `rs/sns/swap/tests/swap.rs` can reproduce this by calling `refresh_buyer_token_e8s` with `buyer = SNS_GOVERNANCE_CANISTER_ID` and a mock ledger returning a sufficient balance, then asserting the entry exists in `swap.buyers` and verifying diluted token distribution in `sweep_sns`.
