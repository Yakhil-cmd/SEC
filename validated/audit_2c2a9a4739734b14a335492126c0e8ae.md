### Title
Unprivileged Caller Can Prematurely Fail SNS Upgrade Proposals via `fail_stuck_upgrade_in_progress` - (File: rs/sns/governance/src/governance.rs)

### Summary
Any unprivileged ingress sender can call the public `fail_stuck_upgrade_in_progress` update method on an SNS Governance canister to forcibly mark an in-progress SNS upgrade proposal as failed once the 5-minute `mark_failed_at_seconds` deadline has elapsed, even if the upgrade would have succeeded. This is a direct governance authorization bug: a state-mutating function that should be restricted to privileged callers (or removed from the public interface) is callable by anyone.

### Finding Description

When an SNS `UpgradeSnsToNextVersion` proposal is adopted and executed, the governance canister kicks off the upgrade asynchronously and records a `PendingVersion` struct with a `mark_failed_at_seconds` field set to `now + 5 * 60` (5 minutes): [1](#0-0) 

The periodic task `check_upgrade_status` is supposed to poll the running canister versions and, if the deadline passes without the expected versions being observed, call `complete_sns_upgrade_to_next_version` to mark the proposal as failed.

However, `fail_stuck_upgrade_in_progress` is also exposed as a **public update method** in the SNS governance canister with **no caller authorization check**: [2](#0-1) 

The function checks only whether `now > pending_version.mark_failed_at_seconds`. If that condition is true, it calls `complete_sns_upgrade_to_next_version`, which:
1. Calls `set_proposal_execution_status` with a failure result, permanently marking the proposal as `Failed`
2. Sets `self.proto.pending_version = None`, clearing the in-progress upgrade state [3](#0-2) 

The method is registered as a public canister endpoint (confirmed by its presence in `rs/sns/governance/canister/governance.did`) and imported in the canister entrypoint: [4](#0-3) 

There is no `check_caller_is_*` guard anywhere in the call path, unlike other sensitive governance endpoints such as `call_canister` on the NNS Root canister: [5](#0-4) 

### Impact Explanation

An attacker who observes that an SNS upgrade proposal has been adopted and that 5 minutes have elapsed since the upgrade was kicked off can call `fail_stuck_upgrade_in_progress` to:
- Permanently mark the upgrade proposal as `Failed` (irreversible for that proposal ID)
- Clear `pending_version`, which unblocks future proposals but destroys the current one

The SNS community must then re-submit, re-vote, and re-execute the upgrade proposal. In a time-sensitive upgrade (e.g., a security patch), this constitutes a meaningful governance DoS. The attacker pays only the ingress fee for a single canister call.

### Likelihood Explanation

The 5-minute window (`mark_failed_at_seconds = now + 5 * 60`) is short but observable on-chain: `pending_version` is part of the canister state readable via `get_running_sns_version`. Any party monitoring SNS governance state can detect when the deadline has passed and race to call `fail_stuck_upgrade_in_progress` before the periodic task does. The attack requires no privileged access, no key material, and no coordination — only a single ingress message after the deadline.

### Recommendation

Add a caller authorization check to `fail_stuck_upgrade_in_progress` so that it can only be invoked by the SNS governance canister itself (i.e., via a self-call from a proposal action or a privileged neuron), or restrict it to a known set of privileged principals. Alternatively, remove the public endpoint entirely and rely solely on the internal periodic task (`check_upgrade_status`) to handle the timeout case.

### Proof of Concept

1. An SNS `UpgradeSnsToNextVersion` proposal is adopted and executed. Governance sets:
   ```
   pending_version = PendingVersion {
       mark_failed_at_seconds: now + 300,
       proposal_id: Some(42),
       ...
   }
   ``` [1](#0-0) 

2. Attacker monitors `get_running_sns_version` and waits until `now > mark_failed_at_seconds` (5 minutes).

3. Attacker sends an ingress call to the SNS governance canister:
   ```
   dfx canister call <sns-governance> fail_stuck_upgrade_in_progress '(record {})'
   ```

4. Inside `fail_stuck_upgrade_in_progress`, the condition `now > pending_version.mark_failed_at_seconds` is true, so `complete_sns_upgrade_to_next_version` is called with `proposal_id = Some(42)` and a failure status: [6](#0-5) 

5. `set_proposal_execution_status(42, Err(...))` is called, permanently marking proposal 42 as `Failed`: [7](#0-6) 

6. `pending_version` is cleared. The upgrade proposal is stuck in `Failed` state. The SNS community must re-submit and re-vote.

        &mut self,
        proposal_id: Option<u64>,
        status: upgrade_journal_entry::upgrade_outcome::Status,
        message: String,
        deployed_version: Option<Version>,
    ) {
        use upgrade_journal_entry::upgrade_outcome::Status;

        let result = match &status {
            Status::Success(_) => Ok(()),
            Status::InvalidState(_) => Err(GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                message.to_string(),
            )),
            Status::ExternalFailure(_) | Status::Timeout(_) => Err(
                GovernanceError::new_with_message(ErrorType::External, message.to_string()),
            ),
        };

        self.push_to_upgrade_journal(upgrade_journal_entry::UpgradeOutcome {
            human_readable: Some(message),
            status: Some(status),
        });

        if let Some(proposal_id) = proposal_id {
            self.set_proposal_execution_status(proposal_id, result);
        }

        self.proto.pending_version = None;

        if let Some(deployed_version) = deployed_version {
            self.proto.deployed_version.replace(deployed_version);
        }

    pub fn fail_stuck_upgrade_in_progress(
        &mut self,
        _: FailStuckUpgradeInProgressRequest,
    ) -> FailStuckUpgradeInProgressResponse {
        let pending_version = match self.proto.pending_version.as_ref() {
            None => return FailStuckUpgradeInProgressResponse {},
            Some(pending_version) => pending_version,
        };

        // Maybe, we should look at the checking_upgrade_lock field and only
        // proceed if it is false, or the request has force set to true.

        let now = self.env.now();

        if now > pending_version.mark_failed_at_seconds {
            let message = format!(
                "Upgrade marked as failed at {}. \
                Governance upgrade was manually aborted by calling fail_stuck_upgrade_in_progress \
                after mark_failed_at_seconds ({}). Setting upgrade to failed to unblock retry.",
                format_timestamp_for_humans(now),
                pending_version.mark_failed_at_seconds,
            );
            let status = upgrade_journal_entry::upgrade_outcome::Status::ExternalFailure(Empty {});

            self.complete_sns_upgrade_to_next_version(
                pending_version.proposal_id,
                status,
                message,
                None,
            );
        }

        FailStuckUpgradeInProgressResponse {}
    }
