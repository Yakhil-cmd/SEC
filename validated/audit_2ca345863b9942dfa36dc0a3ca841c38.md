Audit Report

## Title
Ingress Payload Validation Uses Raw `balance()` While Induction Uses `debited_balance()` During Paused DTS Execution, Enabling Zero-Cost Block-Space Exhaustion - (File: `rs/ingress_manager/src/ingress_selector.rs`)

## Summary

When a canister has a paused Deterministic Time Slicing (DTS) execution, the ingress payload validator and builder check cycle affordability against `system_state.balance()` (raw `cycles_balance`), while the actual induction path checks against `system_state.debited_balance()` (`cycles_balance − ingress_induction_cycles_debit`). Messages that pass validation against the inflated raw balance are included in finalized blocks but immediately rejected during induction with `CanisterOutOfCycles`, consuming block space without delivering any work. Because failed inductions do not charge the canister, the attacker bears zero marginal cost per wasted message.

## Finding Description

**Validation path** — `validate_ingress` in `rs/ingress_manager/src/ingress_selector.rs` (lines 566–584) calls `can_withdraw_cycles_with_threshold`, passing `&canister.system_state`:

```rust
// ingress_selector.rs L568-578
if let Err(err) = self
    .cycles_account_manager
    .can_withdraw_cycles_with_threshold(
        &canister.system_state,          // SystemState only, no paused-execution info
        *cumulative_ingress_cost + ingress_cost,
        ...
    )
```

`can_withdraw_cycles_with_threshold` (`rs/cycles_account_manager/src/cycles_account_manager.rs` line 903) evaluates:

```rust
if threshold + requested > system_state.balance() { ... }  // raw cycles_balance
```

Because the function receives only `&SystemState` (not `&CanisterState`), it cannot call `has_paused_execution_or_install_code()` and always uses the raw balance.

**Induction path** — `charge_ingress_induction_cost` (`cycles_account_manager.rs` lines 335–347) takes the DTS branch:

```rust
if canister.has_paused_execution_or_install_code() {
    if canister.system_state.debited_balance() < cycles + threshold {
        return Err(CanisterOutOfCyclesError { ... });  // no charge applied
    }
    canister.system_state.add_postponed_charge_to_ingress_induction_cycles_debit(cycles);
    Ok(())
}
```

`debited_balance()` (`system_state.rs` lines 947–950) returns `cycles_balance − ingress_induction_cycles_debit`. The debit field accumulates charges during a paused multi-round execution and is only flushed to `cycles_balance` when the execution completes via `apply_ingress_induction_cycles_debit`.

**Gap**: A canister with `cycles_balance = B` and `ingress_induction_cycles_debit = D > 0` has effective spendable balance `B − D`. The validator sees `B`; the inductor sees `B − D`. Any ingress message with cost `c` where `B − D < c ≤ B` passes validation but fails induction. Grep confirms `debited_balance` and `ingress_induction_cycles_debit` are never referenced anywhere in `rs/ingress_manager/`.

**No charge on failed induction**: The `Err` return in the DTS branch of `charge_ingress_induction_cost` exits before `add_postponed_charge_to_ingress_induction_cycles_debit` is called, so the canister is not charged for the rejected message. Marginal attacker cost per wasted slot is zero.

## Impact Explanation

This maps to **High — Application/platform-level DoS / subnet availability impact not based on raw volumetric DDoS**. An unprivileged actor can continuously fill finalized blocks with ingress messages that are guaranteed to fail induction, degrading effective subnet throughput for all users. The per-canister round-robin quota bounds the fraction of a single block that one canister can waste, but the attack is repeatable every block for the entire duration of the DTS execution (which can span many rounds for large `install_code` or compute-heavy updates), and can be amplified by deploying multiple canisters each with a paused execution.

## Likelihood Explanation

- DTS is a production feature triggered by any sufficiently large update or `install_code` call; no special privileges are required to initiate one.
- Any user can submit ingress messages to any canister.
- The certified state read by the ingress selector already contains the current `ingress_induction_cycles_debit` value, so the correct data is present but unused.
- No key material, governance majority, or node compromise is required.
- Likelihood: **Medium** — requires deliberate setup (deploying a canister and triggering a long DTS execution) but is fully reachable by an unprivileged actor at modest one-time cycle cost, with zero marginal cost per attack message thereafter.

## Recommendation

In `validate_ingress`, after resolving the `payer` canister, check whether `canister.has_paused_execution_or_install_code()` is true. If so, use `canister.system_state.debited_balance()` as the effective available balance instead of delegating to `can_withdraw_cycles_with_threshold` (which only sees `SystemState` and always uses `balance()`). The cleanest approach is to add a dedicated helper in `CyclesAccountManager` that mirrors the exact branch logic of `charge_ingress_induction_cost` — accepting `&CanisterState` rather than `&SystemState` — so that validation and induction are structurally guaranteed to stay in sync. The `cycles_needed` accumulator already correctly tracks cumulative costs across messages in the same block; only the balance baseline it is checked against needs to change.

## Proof of Concept

```
1. Deploy canister C. Fund it with exactly `threshold + cost` cycles,
   where `cost` is the ingress induction cost for one message.

2. Send C a large update that triggers a multi-round DTS execution.
   After the first slice, `charge_ingress_induction_cost` records
   `ingress_induction_cycles_debit = cost`, so:
     balance()         = threshold + cost   (unchanged by DTS design)
     debited_balance() = threshold          (= 0 spendable above threshold)

3. While C's execution is paused, submit two ingress messages M1, M2,
   each with induction cost `cost/2`.

4. The ingress selector validates both:
     cumulative check for M1: cost/2 <= balance() - threshold = cost  ✓
     cumulative check for M2: cost   <= balance() - threshold = cost  ✓
   Both are included in the next finalized block.

5. During induction, charge_ingress_induction_cost takes the DTS branch:
     debited_balance() = threshold < cost/2 + threshold  → Err for M1
     debited_balance() = threshold < cost/2 + threshold  → Err for M2
   Both messages are rejected; neither is executed; C is not charged.

6. Block space is consumed; the attacker paid no induction fee.
   Repeat from step 3 for every block until the DTS execution completes.
```

A deterministic integration test using `PocketIC` or the existing `ingress_manager` test harness can reproduce this by constructing a `ReplicatedState` with a canister whose `ingress_induction_cycles_debit` is set to a non-zero value and `task_queue` contains a paused execution task, then asserting that `get_ingress_payload` / `validate_ingress_payload` accept messages that `charge_ingress_induction_cost` subsequently rejects.
