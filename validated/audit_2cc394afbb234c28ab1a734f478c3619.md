Audit Report

## Title
Non-Disclosed Rule and Incident UUIDs Exposed to Unprivileged Callers via Redacted Query Responses - (`rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`, `rs/boundary_node/rate_limits/canister/getter.rs`, `rs/boundary_node/rate_limits/canister/canister.rs`)

## Summary
The `ConfigConfidentialityFormatter` and `RuleConfidentialityFormatter` only redact `rule_raw` and `description` for non-disclosed rules, leaving `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` intact in responses to `RestrictedRead` callers. Because `inspect_message` is only invoked by the IC runtime for ingress update messages and never for non-replicated query calls, any anonymous principal can invoke `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` as queries, bypassing the `inspect_message` gate entirely and harvesting all rule/incident UUIDs and version metadata across all historical config versions.

## Finding Description

**Root cause 1 — Formatter does not redact identifiers.**

`ConfigConfidentialityFormatter::format` iterates over rules and, for non-disclosed ones (`disclosed_at.is_none()`), sets only `rule_raw = None` and `description = None`. The fields `rule_id` and `incident_id` are never cleared:

```
// confidentiality_formatting.rs L21-26
config.rules.iter_mut().for_each(|rule| {
    if rule.disclosed_at.is_none() {
        rule.description = None;
        rule.rule_raw = None;
        // rule.id and rule.incident_id are NOT cleared
    }
});
```

The same applies to `RuleConfidentialityFormatter::format` (lines 37-40), which additionally leaves `added_in_version` and `removed_in_version` intact.

**Root cause 2 — `inspect_message` structurally cannot protect `#[query]` methods.**

The `inspect_message` hook in `canister.rs` (lines 48-67) attempts to gate `get_config` to `FullAccess`/`FullRead` callers and trap all other methods (including `get_rule_by_id` and `get_rules_by_incident_id`). However, `inspect_message` is only invoked by the IC runtime for ingress update messages. All three getter methods are annotated `#[query]` (lines 110, 123, 136), so `inspect_message` is never triggered for them. The intended access control is completely bypassed for non-replicated query calls.

**Exploit flow:**

1. Anonymous caller issues a non-replicated query call to `get_config(None)` — `inspect_message` is not invoked; `AccessLevelResolver` resolves to `RestrictedRead`; `ConfigGetter::get` builds `OutputRule` with `id: *rule_id` and `incident_id: rule.incident_id` always populated (getter.rs L113-115), then calls `formatter.format(config)` which only clears `rule_raw`/`description`.
2. Response contains all `rule_id` and `incident_id` values for every rule in every version, including non-disclosed ones.
3. Caller iterates harvested `incident_id` values and calls `get_rules_by_incident_id` (also `#[query]`, also bypasses `inspect_message`), obtaining the full set of rule UUIDs plus `added_in_version`/`removed_in_version` for every historical and active rule.
4. Caller calls `get_rule_by_id` with any harvested UUID — again bypasses `inspect_message`, receives `added_in_version`/`removed_in_version` with `rule_raw`/`description` redacted.

The existing unit test in `getter.rs` (lines 368-375) already confirms the behavior: for a `RestrictedRead` caller, the redacted response for a non-disclosed rule contains non-null `rule_id` and `incident_id`.

## Impact Explanation
This is a boundary node infrastructure component. The confidentiality mechanism exists specifically to prevent unauthorized callers from learning the structure and existence of non-disclosed rate-limit rules. The `inspect_message` bypass means the intended access control for all three query methods is entirely non-functional. An anonymous caller can enumerate the complete structural topology of all rate-limit rules and incidents — including non-disclosed ones — across all historical versions. This constitutes a significant boundary/API security impact: the confidentiality invariant of the disclosure mechanism is violated, enabling targeted probing (polling `get_rule_by_id` with known UUIDs to detect disclosure events) and structural inference about active security rules. This fits the **Medium–High** boundary of the allowed ICP bounty impact class for significant boundary/API security impact with concrete protocol harm.

## Likelihood Explanation
The attack requires no special privileges, no keys, and no social engineering. Any anonymous principal can issue non-replicated query calls from any IC client. The `inspect_message` bypass is a structural property of the IC runtime, not a configuration issue. The formatter gap is confirmed by the existing unit test. Likelihood is **high**.

## Recommendation
1. **Redact `rule_id` and `incident_id` for non-disclosed rules** in both `ConfigConfidentialityFormatter` (lines 21-26 of `confidentiality_formatting.rs`) and `RuleConfidentialityFormatter` (lines 37-40), replacing them with opaque placeholders or empty strings, so that identifiers of non-disclosed rules are never returned to `RestrictedRead` callers.
2. **Add an explicit in-method access-level check** at the top of `get_rule_by_id` and `get_rules_by_incident_id` in `canister.rs`, returning an authorization error for `RestrictedRead` callers, since `inspect_message` cannot enforce this for query calls.
3. **Add an explicit in-method access-level check** in `get_config` for the `RestrictedRead` path, or accept that anonymous callers receive a redacted response and ensure the formatter fully redacts all identifying fields.

## Proof of Concept
The existing test `test_get_config_success` in `getter.rs` (lines 358-384) already demonstrates the leak: for a `RestrictedRead` getter, the response for a non-disclosed rule (`rule_id_1`, `disclosed_at: None`) contains non-null `rule_id` and `incident_id`. To reproduce the full exploit:

```rust
// Mirrors existing test structure in getter.rs
let canister_state = CanisterState::from_static();
let rule_id = RuleId(Uuid::new_v4());
let incident_id = IncidentId(Uuid::new_v4());

canister_state.upsert_rule(rule_id, StorableRule {
    incident_id,
    rule_raw: b"{\"secret\": true}".to_vec(),
    description: "secret rule".to_string(),
    disclosed_at: None,  // NOT disclosed
    added_in_version: 1,
    removed_in_version: None,
});
canister_state.add_config(1, StorableConfig {
    schema_version: 1, active_since: 0, rule_ids: vec![rule_id],
});

let getter = ConfigGetter::new(
    canister_state,
    ConfigConfidentialityFormatter,
    create_mock_access_resolver(AccessLevel::RestrictedRead),
);
let response = getter.get(&Some(1)).unwrap();
let rule = &response.config.rules[0];

assert!(!rule.rule_id.is_empty());     // PASSES — rule_id is leaked
assert!(!rule.incident_id.is_empty()); // PASSES — incident_id is leaked
assert!(rule.rule_raw.is_none());      // PASSES — content is redacted
assert!(rule.description.is_none());   // PASSES — description is redacted
```
