### Title
Unbounded RSA Public Exponent Accepted in WebAuthn COSE Key Parsing Enables Unmetered CPU Exhaustion — (`rs/crypto/internal/crypto_lib/basic_sig/rsa_pkcs1/src/lib.rs`)

---

### Summary

`RsaPublicKey::from_der_spki` enforces that `e` is odd and that `n` is between 2048 and 8192 bits, but imposes **no upper bound on the bit-length of `e`**. An attacker can register a WebAuthn identity whose COSE RSA key carries a 2047-bit odd exponent. Every subsequent ingress message signed with that identity forces the replica to compute `s^e mod n` with a ~120× larger exponent than the standard `e = 65537`, in the **unmetered** ingress-validation path.

---

### Finding Description

The validation in `from_der_spki` is: [1](#0-0) 

Three checks are performed:
1. `e` is odd (line 100)
2. `n` is odd (line 109)
3. `2048 ≤ n.bits() ≤ 8192` (lines 120–134)

There is **no check on `e.bits()`**. A 2047-bit odd value for `e` passes all three guards.

The COSE parsing path that feeds this function is:

`parse_cose_public_key` → `CosePublicKey::from_cbor` → `parse_rsa_pkcs1_sha256` → `RsaPublicKey::from_components` → `spki_from_components` (encodes arbitrary-length `e` into ASN.1) → `from_der_spki` [2](#0-1) 

Once stored, every call to `verify_pkcs1_sha256` invokes the RustCrypto `rsa` crate's `verify`, which performs the full `s^e mod n` modular exponentiation: [3](#0-2) 

The modular exponentiation cost is O(log e) multiplications mod n. With `e` at 2047 bits vs. 17 bits (65537), this is ~120× more work per verification call.

---

### Impact Explanation

Ingress signature verification is performed in `validate_signature` inside `rs/validator/src/ingress_validation.rs`, which is called at the HTTP handler / consensus-admission layer — **outside of the Wasm instruction-metered execution environment**: [4](#0-3) 

The `RsaSha256PublicKeyDerWrappedCose` branch calls `validate_webauthn_sig` → `verify_basic_sig_by_public_key` → `verify_pkcs1_sha256`. No instruction counter is decremented. A single replica node processing many ingress messages signed with a huge-exponent WebAuthn key will spend proportionally more CPU time in this path, degrading throughput for all users sharing that replica.

---

### Likelihood Explanation

The attack is fully self-service:
1. Generate an RSA key pair where `e` is a 2047-bit odd prime and `n` is a valid 2048-bit modulus.
2. Encode the key as a COSE RSA key (CBOR map with `kty=3, alg=-257, n=..., e=<2047-bit bytes>`), DER-wrap it with OID `1.3.6.1.4.1.56387.1.1`.
3. Derive a self-authenticating principal from this key and submit signed ingress calls.
4. Each call forces the replica to compute `s^e mod n` with a 2047-bit exponent.

No privileged access, governance vote, or subnet-majority corruption is required. The attacker only needs to be able to submit HTTP ingress messages, which is open to any unprivileged user.

---

### Recommendation

Add an upper-bound check on `e.bits()` immediately after the existing oddness check in `from_der_spki`. The standard safe range for RSA public exponents is `3 ≤ e ≤ 2^32 − 1` (NIST SP 800-56B Rev. 2 §6.2). A concrete fix:

```rust
// After the existing oddness check:
const MAXIMUM_RSA_EXPONENT_SIZE: u64 = 33; // bits; covers e=65537 (17 bits) with margin
if parsed.e().bits() > MAXIMUM_RSA_EXPONENT_SIZE {
    return Err(CryptoError::MalformedPublicKey {
        algorithm: AlgorithmId::RsaSha256,
        key_bytes: Some(bytes.to_vec()),
        internal_error: "RSA public exponent is too large".to_string(),
    });
}
``` [5](#0-4)

### Proof of Concept

```rust
use ic_crypto_internal_basic_sig_rsa_pkcs1::RsaPublicKey;
use num_bigint::BigUint;

// Valid 2048-bit odd modulus (any real RSA modulus works)
let n: Vec<u8> = /* 256-byte big-endian odd number with top bit set */;

// Huge odd exponent: 2^2047 + 1 (2047-bit, odd)
let mut e_bytes = vec![0u8; 256];
e_bytes[0] = 0x80; // set top bit → 2047-bit number
e_bytes[255] = 0x01; // set bottom bit → odd

// This must succeed (no e-size check exists):
let key = RsaPublicKey::from_components(&e_bytes, &n)
    .expect("large exponent accepted — no upper bound enforced");

// Measure verification time with a 256-byte all-zero signature:
let sig = vec![0u8; 256];
let start = std::time::Instant::now();
let _ = key.verify_pkcs1_sha256(b"message", &sig);
let elapsed_large_e = start.elapsed();

// Compare with standard e=65537:
let e_standard = [0x01, 0x00, 0x01u8];
let key_std = RsaPublicKey::from_components(&e_standard, &n).unwrap();
let start = std::time::Instant::now();
let _ = key_std.verify_pkcs1_sha256(b"message", &sig);
let elapsed_std = start.elapsed();

// Assert ratio is bounded (it will not be — expect ~120x):
assert!(elapsed_large_e < elapsed_std * 10, "ratio={:?}", elapsed_large_e / elapsed_std);
