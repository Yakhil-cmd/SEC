Audit Report

## Title
Unbounded `decisions` DashMap in `Bouncer` Enables Memory Exhaustion via Unique-IP Flooding ‚Äî (File: rs/boundary_node/ic_boundary/src/bouncer/mod.rs)

## Summary

The `Bouncer` struct maintains two per-IP maps: `shards` (rate-limiter buckets), bounded via `moka::sync::Cache` with `max_capacity`, and `decisions` (the ban list), created as `DashMap::new()` with no capacity limit. Every IP that exhausts its burst is inserted into `decisions` and remains there for the full `ban_time` (default 10 minutes) with no eviction policy. An attacker rotating through unique source IPs can grow `decisions` without bound, exhausting process memory and OOM-killing the boundary node.

## Finding Description

`shards` is bounded at construction: [1](#0-0) 

`decisions` is unbounded: [2](#0-1) 

In `acquire_token`, every IP that exhausts its burst is unconditionally inserted into `decisions` with no size check: [3](#0-2) 

Entries are only removed by `process_releases`, which filters by elapsed `ban_time` and runs on the `apply_interval` tick (default 1s): [4](#0-3) 

The `shards` cap does not protect `decisions`. When the shard cache is full (20,000 entries), `moka` evicts old entries to admit new IPs. Each new unique IP gets a fresh full-burst shard, exhausts it in `burst_size + 1` requests, and is inserted into the unbounded `decisions`. The check order in `acquire_token` is: `decisions` first (line 120), then `shards` (line 124). A new IP not yet in either map always creates a new shard, exhausts its burst, and lands in `decisions`. [5](#0-4) 

The CLI documents the `shards` bound but has no equivalent for `decisions`: [6](#0-5) 

## Impact Explanation

This is a **High** severity finding matching: *Application/platform-level DoS, crash, or boundary/API infrastructure security impact with concrete user or protocol harm.*

Each `Decision` entry is ~40 bytes of struct data plus DashMap overhead (~100‚Äì200 bytes per entry). At 1 million unique IPs: ~100‚Äì200 MB. At 10 million: ~1‚Äì2 GB. The boundary node process is OOM-killed, disabling all ingress protection ‚Äî rate limiting, IP banning, and request routing ‚Äî giving unrestricted access to all IC API endpoints. This is not raw volumetric DDoS: it exploits a specific software defect (unbounded data structure), is asymmetric (601 requests per IP causes disproportionate memory growth), and can be executed at moderate sustained rates without high bandwidth.

## Likelihood Explanation

An attacker with a single machine and an IPv6 /48 prefix (commonly assigned by ISPs) has 2^80 unique source addresses available natively via Linux source address selection. The attack requires only `burst_size + 1 = 601` requests per unique IP ‚Äî not high bandwidth. At a sustained rate of 100 new unique IPs per second (60,100 req/s), the `decisions` map accumulates ~360,000 entries per hour. Over 10‚Äì24 hours this reaches multi-GB RSS and triggers OOM. With a botnet using IPv4, the same effect is achievable. The attack is repeatable, requires no privileges, and no victim interaction.

## Recommendation

Add a capacity cap to `decisions` consistent with how `shards` is bounded. The preferred fix is to replace `DashMap::new()` with a `moka::sync::Cache` configured with `max_capacity` tied to a new CLI parameter (e.g., `bouncer_max_decisions`, defaulting to a value like 1,000,000), using time-to-live equal to `ban_time` for automatic expiry. This eliminates both the unbounded growth and the need for the manual `process_releases` scan. Alternatively, enforce a hard cap in `acquire_token` by checking `self.decisions.len() < max_decisions` before inserting, refusing to ban new IPs when the cap is reached (fail-open for new IPs rather than OOM-killing the process).

## Proof of Concept

```rust
#[test]
fn test_decisions_unbounded_growth() {
    use std::net::{IpAddr, Ipv6Addr};
    use dashmap::DashSet;

    let fw = Arc::new(MockFirewall(DashSet::new()));
    // burst_size=1 so each IP is banned after 2 requests
    let bouncer = Bouncer::new(1, 1, Duration::from_secs(600), 20000,
                               Duration::from_secs(30), fw, &Registry::new()).unwrap();

    for i in 0u128..1_000_000 {
        let ip = IpAddr::V6(Ipv6Addr::from(i));
        bouncer.acquire_token(ip); // allowed (fresh burst)
        bouncer.acquire_token(ip); // denied + inserted into decisions
    }

    assert_eq!(bouncer.decisions.len(), 1_000_000);
    // Process RSS will show ~200MB+ growth with no eviction
}
```

This test uses the existing `MockFirewall` from the test module and requires no network access. It directly confirms that `decisions` grows to exactly N entries with no eviction, matching the claimed vulnerability. Scaling to 10M entries produces multi-GB RSS growth and eventual OOM on a standard boundary node host.

    fn acquire_token(&self, ip: IpAddr) -> bool {
        // Check if the IP is already banned
        if self.decisions.contains_key(&ip) {
            return false;
        }

        if self.shards.acquire(ip) {
            return true;
        }

        warn!("Bouncer: banning {ip}");

        self.decisions.insert(
            ip,
            Decision {
                ip,
                when: Instant::now(),
                length: self.ban_time,
            },
        );

        self.mark_update();
        false
    }
