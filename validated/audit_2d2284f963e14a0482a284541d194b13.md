### Title
Hardcoded `L1_TX_NATIVE_PRICE` Constant Creates Incorrect Native Resource Accounting for L1→L2 Transactions - (File: `basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

The `L1_TX_NATIVE_PRICE` constant is hardcoded at `10` and used as the sole divisor when computing native resource limits for all L1→L2 priority transactions. Unlike L2 transactions — which use the operator-supplied `native_price` from block metadata — L1 transactions always use this fixed value. When actual proving costs diverge from `10`, the native resource budget allocated to L1 transactions is systematically miscalibrated, enabling an attacker to obtain a disproportionately large proving-resource budget at a fraction of the true cost.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant is defined as:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

In `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`, this constant is unconditionally substituted for the operator-set `native_price` when computing `native_per_gas` for priority L1→L2 transactions:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
let native_per_gas = if is_priority_op {
    ...
    u256_try_to_u64(&gas_price.div_ceil(native_price)).unwrap_or_else(|| u64::MAX)
    ...
``` [2](#0-1) 

The resulting `native_per_gas` is then used to compute the total native resource budget:

```rust
let native_prepaid_from_gas = native_per_gas.checked_mul(gas_limit).unwrap_or_else(|| u64::MAX);
``` [3](#0-2) 

By contrast, L2 transactions derive `native_per_gas` from the operator-supplied `native_price` in block metadata:

```rust
let native_price = system.get_native_price();
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [4](#0-3) 

The hardcoded value `10` is explicitly flagged as a placeholder with a TODO, confirming it is not derived from any real-world proving cost model.

---

### Impact Explanation

If the actual operator-set `native_price` (reflecting true proving cost) is significantly higher than `10` — for example `1000` — then an L1 transaction with `gas_price = 10000` receives:

- **Actual allocation**: `native_per_gas = 10000 / 10 = 1000`
- **Correct allocation**: `native_per_gas = 10000 / 1000 = 10`

The attacker obtains **100× more native (proving) resources** than the fee paid on L1 justifies. Since native resources directly model RISC-V proving cycles, this allows the attacker to force the prover to execute computationally expensive work without adequate economic compensation, degrading prover economics and potentially causing prover overload. The block-level native resource cap bounds the per-block damage, but the per-transaction subsidy is unbounded relative to actual cost.

---

### Likelihood Explanation

Medium. The hardcoded value `10` is acknowledged as a placeholder (TODO comment). As the network matures and actual proving costs are measured, the divergence between `L1_TX_NATIVE_PRICE = 10` and the real `native_price` will grow. Any L1 transaction sender — an unprivileged role — can trigger this path by submitting a priority L1→L2 transaction with an elevated `gas_price`.

---

### Recommendation

Replace the hardcoded `L1_TX_NATIVE_PRICE` with a configurable parameter supplied via block metadata or governance, analogous to how `native_price` is already provided for L2 transactions. If censorship-resistance requires a floor, enforce a minimum rather than a fixed value, so the constant can be updated as proving costs evolve without a code change.

---

### Proof of Concept

1. Operator sets `native_price = 1000` in block metadata (reflecting actual proving cost).
2. Attacker submits an L1→L2 priority transaction with `gas_price = 100_000` and `gas_limit = 1_000_000`.
3. Bootloader computes: `native_per_gas = 100_000 / 10 = 10_000` (using hardcoded `L1_TX_NATIVE_PRICE`).
4. Native budget allocated: `10_000 * 1_000_000 = 10^10` native units.
5. Correct budget (using actual `native_price = 1000`): `100_000 / 1000 * 1_000_000 = 10^8` native units.
6. The attacker receives **100× the proving resources** their L1 fee justifies, forcing the prover to execute and prove a transaction at a 100× subsidy, with no recourse for the operator. [1](#0-0) [5](#0-4)
