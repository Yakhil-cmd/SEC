Audit Report

## Title
Page-Allocator RSS Excluded from Eviction Accounting Suppresses RSS-Based Sandbox Eviction for Canisters with Large Stable Memory — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

## Summary

In `monitor_and_evict_sandbox_processes`, only the anonymous RSS (`get_anon_rss`) is pushed into `sandbox_processes_rss` and subsequently stored in `stats.rss` per backend. The page-allocator RSS (`get_page_allocator_rss`), which accounts for memfd and file-backed stable memory regions, is collected solely for metrics and never included in the per-canister RSS used for eviction decisions. As a result, `total_sandboxes_rss` — the value gating RSS-based eviction in `trigger_sandbox_eviction` — systematically undercounts actual sandbox memory consumption for any canister holding significant stable memory, permanently suppressing the RSS-based eviction branch even under real node memory pressure.

## Finding Description

In `monitor_and_evict_sandbox_processes` (Linux path), for each sandbox process the code collects two RSS values:

- `get_anon_rss(pid)` → pushed to `sandbox_processes_rss` at line 1405
- `get_page_allocator_rss(pid)` → added to `process_rss` for the per-process metric only; never pushed to `sandbox_processes_rss` (lines 1409–1414)

`update_sandbox_processes_rss` then writes only the anon-RSS entries into `stats.rss` for each backend (lines 2013–2015). `total_sandboxes_rss` sums only `stats.rss` across active backends (lines 2022–2029), so it reflects anon RSS only.

`trigger_sandbox_eviction` uses a conjunctive gate (lines 1542–1545):

```
total_sandboxes_rss > self.max_sandboxes_rss()   // anon-only, undercounted
AND
available_memory() < DEFAULT_MIN_MEM_AVAILABLE_TO_EVICT_SANDBOXES
```

For a canister with large stable memory (stored in memfd or file-backed `.mem (deleted)` regions):
- Condition 1 is **always FALSE** — anon RSS is near zero
- Condition 2 is **TRUE** — real physical memory is consumed by page-allocator regions
- AND(FALSE, TRUE) = **FALSE** → RSS-based eviction never fires

Additionally, the monitor thread's own eviction call (lines 1482–1490) passes `max_sandboxes_rss = u64::MAX`, so it independently performs no RSS-based eviction regardless.

`get_page_allocator_rss` correctly reads both `/memfd:` prefixed and `.mem (deleted)` suffixed VMA regions from `/proc/<pid>/smaps` (lines 224–265 of `process_os_metrics.rs`), confirming the data is available but simply not wired into the eviction path.

## Impact Explanation

A canister with large stable memory (up to the IC protocol limit) causes its sandbox process to have near-zero anon RSS but substantial page-allocator RSS. `total_sandboxes_rss` never exceeds `max_sandboxes_rss`, so the RSS-based eviction branch in `trigger_sandbox_eviction` is permanently suppressed. The node accumulates physical memory pressure from memfd/file-backed regions without the RSS eviction mechanism engaging. Count-based and idle-time eviction paths remain functional but are insensitive to per-canister memory magnitude. Under sustained stable-memory load from one or more active canisters, the node can exhaust physical RAM, causing replica crashes and loss of subnet availability. This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation

Any unprivileged canister developer can deploy a canister that allocates stable memory via the public IC stable memory API (`stable_grow`, `stable_write`). No privileged access, governance majority, or infrastructure compromise is required. The condition is deterministic: as long as the canister keeps its Wasm heap allocation minimal and stable memory large, the anon RSS remains near zero and the eviction gate condition never fires. The effect is reproducible in a local replica environment and scales linearly with stable memory size.

## Recommendation

Include page-allocator RSS in the per-canister RSS pushed to `sandbox_processes_rss`. In `monitor_and_evict_sandbox_processes`, accumulate both values before pushing:

```rust
let anon_kib = process_os_metrics::get_anon_rss(pid).unwrap_or(0);
let page_alloc_kib = process_os_metrics::get_page_allocator_rss(pid).unwrap_or(0);
let total_kib = anon_kib + page_alloc_kib;
sandbox_processes_rss.push((*canister_id, NumBytes::new(total_kib * 1024)));
```

This ensures `stats.rss` and `total_sandboxes_rss` reflect actual sandbox memory consumption, allowing the eviction gate condition to fire correctly when the node is under real memory pressure.

## Proof of Concept

1. Start a local replica with `max_sandboxes_rss` configured to a small value (e.g., 100 MiB).
2. Deploy a canister that calls `stable_grow` to allocate ~200 MiB of stable memory and writes to it via `stable_write`, keeping Wasm heap allocation near zero.
3. Inspect `/proc/<sandbox_pid>/smaps` — confirm ~200 MiB of memfd or `.mem (deleted)` RSS and near-zero `RssAnon`.
4. Observe via metrics that `sandboxed_execution_subprocess_anon_rss` is near zero while `sandboxed_execution_subprocess_memfd_rss` is ~200 MiB.
5. Confirm `stats.rss` for the backend is near zero (only anon RSS stored).
6. Confirm `total_sandboxes_rss` is below `max_sandboxes_rss`.
7. Simulate low available memory (condition 2 = TRUE, e.g., by consuming RAM on the test host).
8. Call `trigger_sandbox_eviction` and observe no RSS-based eviction occurs despite actual memory exceeding the configured limit.
