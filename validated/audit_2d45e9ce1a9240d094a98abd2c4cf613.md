### Title
Truncated-to-Zero `native_per_pubdata` Allows Free Pubdata Writing When `pubdata_price < native_price` — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio is computed using floor (truncating) integer division. When `pubdata_price < native_price`, the result truncates to zero, making all pubdata completely free in terms of native resources. Any unprivileged transaction sender can then write maximum pubdata without paying native costs, while the operator must still publish that data on L1.

---

### Finding Description

In `validation_impl.rs` at line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs floor (truncating) integer division. When `pubdata_price < native_price` — for example `pubdata_price = 9`, `native_price = 10` — the result is `9 / 10 = 0`. [1](#0-0) 

This zero propagates into every downstream pubdata charging call. In `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

And in `check_enough_resources_for_pubdata`, which always returns `true` (enough resources) when the cost is zero: [3](#0-2) 

The same truncation is also present in the public API helper `validate_l2_tx_intrinsic_native_resources`:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

There is a critical asymmetry: `native_per_gas` is computed with `div_ceil` (rounds **up**, ensuring the user always pays enough for native computation), while `native_per_pubdata` uses `wrapping_div` (rounds **down**, potentially making pubdata completely free):

```rust
// native_per_gas — rounds UP (correct)
u256_try_to_u64(&gas_price.div_ceil(native_price))

// native_per_pubdata — rounds DOWN (bug)
u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
``` [5](#0-4) 

The intrinsic pubdata overhead is also zeroed out:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
``` [6](#0-5) 

---

### Impact Explanation

When `pubdata_price < native_price`, any unprivileged transaction sender can:

1. Submit transactions that write maximum pubdata (many `SSTORE` operations to new slots, up to `get_pubdata_limit()`).
2. Pay only EVM gas for the computation — zero native resources are charged for pubdata.
3. Force the operator to publish that data on L1, incurring real L1 DA costs without compensation.

This is a direct resource accounting bug: the operator intends to charge `pubdata_price / native_price` native units per pubdata byte, but the truncation silently makes it zero. The operator loses the native-resource revenue that should cover L1 publication costs. Repeated across many transactions, this drains operator economics without any on-chain enforcement.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic production configuration. `native_price` reflects the cost of a single RISC-V proving cycle (typically high), while `pubdata_price` reflects the cost of one byte of L1 data availability (which can be lower, especially post-EIP-4844). Any block where `pubdata_price` is set to a non-zero value smaller than `native_price` silently enables this exploit for all transactions in that block. The attacker needs no privileged access — only the ability to submit standard L2 transactions.

---

### Recommendation

Use `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed. This ensures that when `pubdata_price > 0`, at least 1 native unit is charged per pubdata byte:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [7](#0-6)

### Proof of Concept

**Setup**: Block context with `pubdata_price = 9`, `native_price = 10`.

**Step 1**: `native_per_pubdata = 9 / 10 = 0` (truncated by `wrapping_div`).

**Step 2**: Submit a transaction calling a contract that writes 20 new storage slots (generating ~640 bytes of pubdata):

```solidity
for (uint256 i = 0; i < 20; i++) {
    s[i] = 0xffffffffffffffff;
}
```

**Step 3**: `get_resources_to_charge_for_pubdata` returns `(640, 0_native)`. `check_enough_resources_for_pubdata` returns `true`. Transaction succeeds.

**Step 4**: Observe `native_used` reflects only computational native (no pubdata component). The operator publishes 640 bytes to L1 with zero native compensation.

**Step 5**: Repeat across 100 transactions. The operator publishes 64,000 bytes of pubdata, paying real L1 DA costs, while the attacker paid only EVM gas.

The same pattern from the Hedge Vault report applies: the division `pubdata_price / native_price` produces a decimal that is rounded down to zero, and the attacker can repeatedly trigger pubdata writes without increasing their native liability.
