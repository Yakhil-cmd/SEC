### Title
Missing Deadline Check in XCC Router `execute_scheduled` Allows Stale Scheduled Promise Execution - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `execute_scheduled` function has no deadline or expiry mechanism. Promises stored via `CrossContractCallArgs::Delayed` can be executed by any caller at any future time. When a user schedules a time-sensitive NEAR cross-contract call (e.g., a token swap on a NEAR DEX with encoded slippage parameters), an attacker can delay or front-run the execution until market conditions are unfavorable, causing the user to receive fewer tokens than intended.

---

### Finding Description

The Aurora Engine XCC (Cross-Contract Call) precompile supports two execution modes:

- `CrossContractCallArgs::Eager` — executes the NEAR promise immediately within the same transaction.
- `CrossContractCallArgs::Delayed` — stores the promise in the XCC router contract for later execution. [1](#0-0) 

When `Delayed` is used, the precompile calls the router's `schedule` method, which stores the `PromiseArgs` (including raw call arguments such as swap amounts and slippage bounds) in a `LookupMap` keyed by a monotonically increasing nonce: [2](#0-1) 

The stored promise is later executed by calling `execute_scheduled`: [3](#0-2) 

The comment explicitly states this function is intentionally open to any caller. However, neither `schedule` nor `execute_scheduled` accepts or enforces a deadline. The `PromiseArgs` stored in `scheduled_promises` contain the raw encoded call arguments — including any slippage bounds — that were valid at the time of scheduling but may be stale by the time of execution. [4](#0-3) 

The `PromiseCreateArgs` struct carries `args: Vec<u8>` — the raw bytes passed to the target NEAR contract — which are fixed at scheduling time and cannot be updated: [5](#0-4) 

---

### Impact Explanation

**High — Theft of user funds in motion.**

A user who schedules a token swap on a NEAR DEX (e.g., via `ft_transfer_call` with a minimum-output field encoded in `args`) sets slippage bounds based on the price at scheduling time. Because `execute_scheduled` has no deadline, an attacker can:

1. Observe the scheduled promise on-chain (all NEAR state is public).
2. Manipulate the price on the target DEX (sandwich the execution).
3. Call `execute_scheduled(nonce)` at the manipulated price.

The user's swap executes with stale slippage parameters, and the attacker extracts the price difference. The user receives fewer tokens than intended, constituting direct theft of funds in motion.

---

### Likelihood Explanation

**Medium.** The `Delayed` path is explicitly documented as the mechanism for expensive EVM calls that need a fresh 300 Tgas budget for the NEAR cross-contract call. Any Aurora user who schedules a time-sensitive NEAR call (swap, liquidity provision, etc.) is exposed. The attacker entry point — `execute_scheduled` — requires no special permissions and is callable by any NEAR account. [3](#0-2) 

---

### Recommendation

Add an optional `deadline` field (NEAR block timestamp or block height) to the `PromiseArgs` or as a separate parameter to `schedule`. In `execute_scheduled`, check `env::block_timestamp() <= deadline` before executing and panic with `ERR_PROMISE_EXPIRED` if the deadline has passed. This mirrors the standard pattern used by Uniswap v3 and other DEX routers.

---

### Proof of Concept

1. Alice calls the XCC precompile from her Aurora EVM contract with:
   ```
   CrossContractCallArgs::Delayed(PromiseArgs::Create(PromiseCreateArgs {
       target_account_id: "dex.near",
       method: "ft_transfer_call",
       args: b"{\"receiver_id\":\"pool.near\",\"amount\":\"1000000\",\"msg\":\"{\\\"min_amount_out\\\":\\\"950000\\\"}\"}",
       attached_balance: Yocto::new(1),
       attached_gas: NearGas::new(50_000_000_000_000),
   }))
   ```
2. The router stores this promise at nonce `0` with `min_amount_out = 950000` encoded in `args`.
3. Bob (attacker) reads the scheduled promise from the router's on-chain state.
4. Bob manipulates the DEX pool price so that 1,000,000 input tokens yield only 951,000 output tokens (just above Alice's slippage floor).
5. Bob calls `router.execute_scheduled({"nonce": "0"})` — succeeds because there is no deadline check.
6. Alice's swap executes at the manipulated price; Bob profits from the price impact he induced. [6](#0-5) [4](#0-3)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
