Looking at the exact code paths involved:

### Title
Unbounded `iter_metadata()` in `/dashboard` HTTP handler causes permanent query-instruction-limit trap — (`rs/cross-chain/blob_store/src/dashboard/mod.rs`)

### Summary

The `http_request` query handler unconditionally calls `dashboard::dashboard()`, which iterates every entry in the `StableBTreeMap` metadata store without pagination or a size guard. Once a controller has inserted enough blobs through normal canister use, any unprivileged HTTP client can permanently trap the dashboard endpoint by issuing a single `GET /dashboard` request.

---

### Finding Description

**Entrypoint** — `http_request` is a `#[ic_cdk::query(hidden = true)]` endpoint reachable by any caller with no authentication: [1](#0-0) 

For the `/dashboard` path it calls `dashboard::dashboard()` with no guard: [2](#0-1) 

**Unbounded iteration** — `dashboard()` collects every entry from the metadata store into a `Vec<DashboardBlob>` in a single pass: [3](#0-2) 

`iter_metadata()` is a thin wrapper over `StableBTreeMap::iter()` with no limit: [4](#0-3) 

**No cap on blob count or tag count** — `insert` enforces only that the caller is a controller and that each individual tag is ≤ 100 bytes; there is no limit on the total number of blobs or the number of tags per blob: [5](#0-4) [6](#0-5) 

Each `StableBTreeMap` iteration involves stable-memory page reads and CBOR deserialization of `Metadata` (principal + timestamp + size + `BTreeSet<Tag>`). At IC query instruction limits (~5 billion instructions), iterating O(10 000) entries with O(10) max-length tags each, followed by Askama HTML template rendering, is sufficient to exhaust the budget and trap.

---

### Impact Explanation

Once the instruction limit is exceeded, `http_request` traps on every subsequent `/dashboard` call regardless of who calls it. The dashboard is permanently unavailable as long as the blob count remains high. Because there is no `delete` endpoint, the state cannot be reduced without an upgrade. If operators rely on the dashboard to cross-check Wasm hashes before approving NNS upgrade proposals for cross-chain canisters, its unavailability removes a verification step.

---

### Likelihood Explanation

The precondition — a controller inserting many blobs — is the **intended operational use** of the canister, not a malicious action. The canister is designed to store Wasm blobs for cross-chain canisters; accumulating hundreds or thousands of versioned blobs over time is expected. No adversarial controller behavior is required. Once the threshold is crossed, any anonymous HTTP client can trigger the trap with a single request.

---

### Recommendation

Add a pagination parameter (`offset`/`limit`) to the `/dashboard` route and cap the number of entries rendered per page (e.g., 100). Alternatively, maintain a pre-aggregated summary in a separate stable cell and render only the summary plus the most-recent N entries. The `iter_metadata()` call must never be unbounded inside a query handler.

---

### Proof of Concept

A state-machine test would:
1. Install the canister.
2. Call `insert` 10 000 times as a controller, each with 10 tags of 100 bytes each.
3. Issue an `http_request` query for `/dashboard`.
4. Assert the response is a valid 200 OK — the test would instead observe an instruction-limit trap, confirming the invariant violation. [7](#0-6)

```rust
pub fn dashboard() -> DashboardTemplate {
    read_blob_store(|store| {
        let mut total_size_bytes = 0_u64;
        let blobs: Vec<DashboardBlob> = store
            .iter_metadata()
            .map(|(hash, metadata)| {
                total_size_bytes += metadata.size;
                DashboardBlob {
                    hash: hash.to_string(),
                    uploader: metadata.uploader.to_string(),
                    size: metadata.size,
                    inserted_at_ns: metadata.inserted_at_ns,
                    tags: metadata.tags.into_iter().map(|t| t.to_string()).collect(),
                }
            })
            .collect();
        DashboardTemplate {
            total_blobs: store.len(),
            total_size_bytes,
            blobs,
        }
    })
```
