### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` burns a user's ERC-20 tokens **before** calling the exit precompile via inline assembly, and **never checks the return value** of that call. If the precompile call fails for any reason (e.g., the precompile is paused via `pause_precompiles`, or the call runs out of gas), the burn is not reverted and the user's tokens are permanently destroyed with no corresponding credit on NEAR or Ethereum.

---

### Finding Description

`EvmErc20.withdrawToNear` and `EvmErc20.withdrawToEthereum` follow this pattern:

1. Call `_burn(_msgSender(), amount)` — irreversibly destroys the caller's ERC-20 balance.
2. Encode calldata and invoke the exit precompile via a raw `assembly { let res := call(...) }` block.
3. **Never inspect `res`** — a return value of `0` (failure) is silently ignored and the function returns normally. [1](#0-0) 

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // ← tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc7..., 0, ...)    // ← result never checked
    }
}
``` [2](#0-1) 

The same pattern appears in `withdrawToEthereum`.

The Aurora Engine exposes a `pause_precompiles` administrative function backed by `PrecompileFlags`, which can pause exactly these two exit precompile addresses: [3](#0-2) 

When `EXIT_TO_NEAR` or `EXIT_TO_ETHEREUM` is paused, the precompile's `run` method returns an `ExitError`. In the EVM, a low-level assembly `call()` to a precompile that returns an error sets `res = 0` but does **not** propagate a revert to the caller. Because `_burn` executed before the assembly block, the state change is committed and the user's tokens are gone.

The `ExitToNear` precompile itself enforces the static-call and delegate-call guards but has no mechanism to undo a burn that already occurred in the calling ERC-20 contract: [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

A user calling `withdrawToNear` or `withdrawToEthereum` while the corresponding precompile is paused will have their ERC-20 tokens burned with no corresponding NEP-141 or ETH credit issued. The tokens are unrecoverable: the ERC-20 supply is reduced, the NEAR-side balance is unchanged, and there is no refund path.

---

### Likelihood Explanation

**Medium.** The pause mechanism is an intentional operational feature. During any incident response or upgrade window where `EXIT_TO_NEAR` or `EXIT_TO_ETHEREUM` is paused, every user who calls `withdrawToNear`/`withdrawToEthereum` on any `EvmErc20` token will silently lose their funds. Additionally, a sufficiently gas-constrained call (e.g., a relayer forwarding a transaction with a tight gas limit) can trigger the same failure path without any admin action.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }   // ← add this
    }
}
```

Apply the same fix to `withdrawToEthereum`. This ensures that if the precompile call fails for any reason, the entire transaction reverts (including the `_burn`), leaving the user's balance intact.

---

### Proof of Concept

1. Admin calls `pause_precompiles(EXIT_TO_NEAR)` on the Aurora Engine contract.
2. User holds 100 units of a bridged `EvmErc20` token on Aurora.
3. User calls `token.withdrawToNear(recipientBytes, 100)`.
4. `_burn(user, 100)` executes — user's ERC-20 balance drops to 0.
5. The assembly `call` to `0xe9217bc7...` returns `0` (precompile is paused, `ExitError` returned).
6. `res` is never checked; the function returns normally.
7. User's 100 tokens are permanently destroyed. No NEP-141 tokens are credited on NEAR. [1](#0-0) [5](#0-4)
