### Title
Operator-Controlled `native_price`/`pubdata_price` Absent from Public Input Commitment Enables Asymmetric Fee Drain — (`basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/mod.rs`, `zk_ee/src/system/metadata/zk_metadata.rs`)

---

### Summary

`native_price` and `pubdata_price` are operator-supplied block-context parameters that directly govern how much native resource a user's transaction receives per unit of gas. Both values are consumed from the oracle at block start and used throughout transaction execution, but neither is committed to in the block header or the ZK public input. A malicious operator/prover can therefore supply a `native_price` that is far higher during actual execution than during simulation, causing user transactions to exhaust their native resource budget mid-execution, revert, and forfeit the full gas fee — while the operator collects that fee without delivering execution. This is the direct ZKsync OS analog of M-02: one party (the user) locks resources (gas fees) while the other party (the operator) bears no symmetric commitment and can manipulate the shared parameter to drain those resources.

---

### Finding Description

**Step 1 — `native_price` and `pubdata_price` come from the oracle, not from a committed source.**

`BlockMetadataFromOracle` carries both fields: [1](#0-0) 

They are deserialized directly from the oracle response with no range check or commitment verification: [2](#0-1) 

**Step 2 — These values are used to compute the user's native resource budget.**

For L2 transactions, `native_per_gas = ceil(gas_price / native_price)` and `native_limit = gas_limit × native_per_gas`. A high `native_price` yields a tiny `native_limit`: [3](#0-2) 

**Step 3 — Running out of native during execution exhausts all remaining ergs (full gas consumed).**

When a transaction runs out of native resources during execution, `exhaust_ergs()` is called, meaning `gas_used = gas_limit` and the user forfeits the entire prepaid fee: [4](#0-3) 

**Step 4 — `native_price` and `pubdata_price` are NOT included in the block header.**

`form_block_header` explicitly omits them with a TODO: [5](#0-4) 

The project documentation confirms this gap: [6](#0-5) 

**Step 5 — The public input commitment does not cover `native_price` or `pubdata_price`.**

`ChainStateCommitment` hashes only `state_root`, `next_free_slot`, `block_number`, `last_256_block_hashes_blake`, and `last_block_timestamp`: [7](#0-6) 

`blocks_output_hash` covers chain ID, timestamps, pubdata hash, L1 tx hashes, L2→L1 log hashes, and upgrade tx hash — but not `native_price` or `pubdata_price`: [8](#0-7) 

Because neither value appears in the public input, the ZK proof does not constrain them. The prover can supply any `native_price` and still produce a valid proof.

---

### Impact Explanation

**Asymmetric resource drain (direct analog of M-02):**

1. A user calls `eth_estimateGas` / simulates the transaction. The simulation uses the current `native_price = P_sim`. The user receives a gas estimate `G` and submits a transaction with `gas_limit = G` and prepays `G × gas_price` in fees.
2. The operator builds the actual block with `native_price = P_exec >> P_sim`. The user's `native_limit = G × ceil(gas_price / P_exec)` is now far smaller than the native resources the transaction actually needs.
3. The transaction passes validation (intrinsic native is covered at the inflated price) but exhausts native resources during execution body. `exhaust_ergs()` fires; `gas_used = gas_limit = G`.
4. The operator collects `G × gas_price` in fees. The user's state changes are reverted. The user receives nothing.

The same attack applies via `pubdata_price`: inflating it between simulation and execution causes post-execution pubdata charging to fail, again consuming the full gas limit: [9](#0-8) 

**Severity**: Direct, irreversible loss of user gas fees. The ZK proof — the primary trust guarantee of the rollup — does not prevent this because the manipulated parameter is outside the proof's constraint system.

---

### Likelihood Explanation

The attack requires the operator/sequencer to act maliciously. However:

- The entire purpose of a ZK rollup is to constrain what the operator can do via cryptographic proofs. The absence of `native_price` from the public input is an explicit, acknowledged gap in that constraint system.
- The operator controls the oracle that supplies `BlockMetadataFromOracle`, including `native_price` and `pubdata_price`, with no on-chain enforcement of their values.
- The docs explicitly acknowledge the issue is known and unresolved: *"Currently it misses `gas_per_pubdata` and `native_price`, but we already working on design and implementation to solve this issue."*
- Any block the operator produces with an inflated `native_price` will be accepted by the verifier because the proof is valid regardless of that value.

Likelihood is **medium-high** given the acknowledged gap and the operator's direct ability to exploit it.

---

### Recommendation

1. **Include `native_price` and `pubdata_price` in the block header** (`form_block_header`) so they become part of the block hash that feeds into the rolling transaction hash and ultimately the public input.
2. **Commit `native_price` and `pubdata_price` into `blocks_output_hash`** (or an equivalent field in `ChainStateCommitment`) so the ZK proof constrains these values to match what was announced on-chain.
3. Until the above is implemented, enforce an on-chain upper bound on `native_price` and `pubdata_price` in the L1 contracts so the operator cannot set them to arbitrary values.

---

### Proof of Concept

```
1. Deploy ZKsync OS with a malicious operator.
2. User simulates a transaction with native_price = 10 (default).
   eth_estimateGas returns gas_limit = 100_000.
   User submits tx with gas_limit=100_000, max_fee_per_gas=1000.
   Prepaid fee = 100_000 × 1000 = 100_000_000 tokens.

3. Operator builds the block with native_price = 10_000_000
   (set in BlockMetadataFromOracle via oracle).
   native_per_gas = ceil(1000 / 10_000_000) = 1
   native_limit = 100_000 × 1 = 100_000 native units.
   L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_COST >> 100_000,
   so the transaction exhausts native during execution.

4. exhaust_ergs() fires → gas_used = 100_000 = gas_limit.
   Operator receives 100_000_000 tokens.
   User's execution is reverted; user receives nothing.

5. The ZK proof is valid because native_price is not in the
   public input — the verifier cannot detect the manipulation.
``` [10](#0-9) [11](#0-10)

/// Helper method to create block header.
fn form_block_header<S: EthereumLikeTypes>(
    system: &System<S>,
    tx_rolling_hash: Bytes32,
    block_gas_used: u64,
) -> Result<BlockHeader, BootloaderSubsystemError> {
    let block_number = system.get_block_number();
    let previous_block_hash = if block_number == 0 {
        Bytes32::ZERO
    } else {
        system.get_blockhash(block_number - 1)?
    };
    let beneficiary = system.get_coinbase();
    let gas_limit = system.get_gas_limit();
    let timestamp = system.get_timestamp();
    let consensus_random = system.get_mix_hash()?;
    let base_fee_per_gas = system.get_eip1559_basefee();
    // TODO: add pubdata price and native price
    let base_fee_per_gas = base_fee_per_gas
        .try_into()
        .map_err(|_| internal_error!("base_fee_per_gas exceeds max u64"))?;

    Ok(BlockHeader::new(
        previous_block_hash,
        beneficiary,
        tx_rolling_hash,
        block_number,
        gas_limit,
        block_gas_used,
        timestamp,
        consensus_random,
        base_fee_per_gas,
    ))
}

#[derive(Debug)]
pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }
