Audit Report

## Title
TOCTOU Race in `register_dapp_canister` Allows Exceeding `DAPP_AND_EXTENSION_CANISTER_REGISTRATION_LIMIT` — (`rs/sns/root/src/lib.rs`)

## Summary

`register_dapp_canister` reads the registered-canister count and checks it against the 100-entry cap in a synchronous section, then suspends at up to three `await` points before writing the new entry. Because the IC canister scheduler may interleave a second incoming message at any `await` boundary, two concurrent `RegisterDappCanisters` proposal executions can both pass the limit check before either writes, pushing `dapp_canister_ids` beyond the cap. No concurrency guard exists for this action type, unlike upgrade proposals.

## Finding Description

`register_dapp_canister` at [1](#0-0)  reads state and checks the limit synchronously. It then suspends at the first `await` at [2](#0-1) , with up to two additional `await` points at [3](#0-2) . The write to state occurs only after all awaits at [4](#0-3) .

The IC execution model allows the canister to process the next queued message when a message suspends at an `await`. Two concurrent calls to `register_dapp_canisters` on the root canister therefore interleave:

```
Call A: reads dapps.len()=99 → passes check (99 < 100)
Call A: awaits canister_status(X)          ← suspends
Call B: reads dapps.len()=99 → passes check (99 < 100)
Call B: awaits canister_status(Y)          ← suspends
Call A: resumes → pushes X → dapps.len()=100
Call B: resumes → pushes Y → dapps.len()=101  ← EXCEEDS LIMIT
```

On the governance side, `start_proposal_execution` uses `spawn_in_canister_env` at [5](#0-4)  to spawn proposal futures concurrently. `RegisterDappCanisters` is dispatched at [6](#0-5)  with no concurrency guard, unlike upgrade proposals which use `upgrade_proposals_in_progress`. The constant being violated is defined at [7](#0-6) .

## Impact Explanation

The `dapp_canister_ids` vector exceeds the 100-entry cap, violating the SNS root canister's registration invariant. This constitutes a significant SNS security impact: `get_sns_canisters_summary` and `set_dapp_controllers` iterate over all registered dapps and can be affected by unbounded state growth. The violation is persistent and cannot be self-corrected by the canister. This matches the allowed High impact: *Significant SNS security impact with concrete user or protocol harm*.

## Likelihood Explanation

Requires two `RegisterDappCanisters` proposals to be adopted before either fully executes — a realistic scenario when an SNS has approximately 99 registered dapps and two community members independently propose registering different dapps within the same voting window. No malicious governance majority is required; normal community voting suffices. The interleaving is deterministic given the IC's cooperative async scheduling at `await` boundaries.

## Recommendation

Re-check the limit inside the final `with_borrow_mut` write, atomically with the push, so the check and write are in the same synchronous (non-`await`) critical section:

```rust
self_ref.with_borrow_mut(|state| {
    let count = state.dapp_canister_ids.len() + state.extensions.len();
    if count >= DAPP_AND_EXTENSION_CANISTER_REGISTRATION_LIMIT {
        return Err("limit reached".to_string());
    }
    state.dapp_canister_ids.push(PrincipalId::from(canister_to_register));
    Ok(())
})?;
```

Since `with_borrow_mut` is synchronous and non-`await`, no interleaving can occur between the check and the write.

## Proof of Concept

A state-machine or PocketIC test with two `RegisterDappCanisters` proposals both adopted in the same round, with the SNS root pre-loaded with 99 dapps, would demonstrate that after both proposals execute, `dapp_canister_ids.len()` equals 101, violating the invariant. The existing test `register_dapp_canisters_fails_at_limit_number` at [8](#0-7)  only tests sequential execution and does not cover concurrent interleaving.

    async fn register_dapp_canisters_fails_at_limit_number() {
        // Step 1: Prepare the world.
        thread_local! {
            static SNS_ROOT_CANISTER: RefCell<SnsRootCanister> = RefCell::new(SnsRootCanister {
                governance_canister_id: Some(PrincipalId::new_user_test_id(1)),
                ledger_canister_id: Some(PrincipalId::new_user_test_id(2)),
                swap_canister_id: Some(PrincipalId::new_user_test_id(99)),

            (100..200).map(|id| CanisterId::from(id).get()).collect();

        let mut management_canister_client = MockManagementCanisterClient::new(vec![]);

        canister_ids.iter().for_each(|_| {
            management_canister_client.push_reply(
                MockManagementCanisterClientReply::CanisterStatus(Ok(
                    CanisterStatusResultFromManagementCanister::dummy_with_controllers(vec![
                        sns_root_canister_id,
                        PrincipalId::new_user_test_id(9999),
                    ]),
                )),
            );
            management_canister_client
                .push_reply(MockManagementCanisterClientReply::UpdateSettings(Ok(())));
            management_canister_client.push_reply(
                MockManagementCanisterClientReply::CanisterStatus(Ok(
                    CanisterStatusResultFromManagementCanister::dummy_with_controllers(vec![
                        sns_root_canister_id,
                    ]),
                )),
            );
        });

        // Step 2: Max out the registered dapps, and confirm they were registered.
        SnsRootCanister::register_dapp_canisters(
            &SNS_ROOT_CANISTER,
            &management_canister_client,
            sns_root_canister_id,
            RegisterDappCanistersRequest {
                canister_ids: canister_ids.clone(),
            },
        )
        .await;

        SNS_ROOT_CANISTER.with(|r| {
            assert_eq!(
                *r.borrow().dapp_canister_ids,
                // Check that root became aware that it controls the dapp canister.
                canister_ids
            );
        });

        // Step 3: Attempt to register another dapp, which should trigger panic
        SnsRootCanister::register_dapp_canisters(
            &SNS_ROOT_CANISTER,
            &management_canister_client,
            sns_root_canister_id,
            RegisterDappCanistersRequest {
                canister_ids: vec![CanisterId::from(201).get()],
            },
        )
        .await;
    }
