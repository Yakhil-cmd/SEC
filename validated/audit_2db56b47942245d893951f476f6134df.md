### Title
SNS `MintSnsTokens` Proposal 7-Day Minting Rate Limit Is Disabled, Allowing Unbounded Token Minting - (`rs/sns/governance/src/proposal.rs`)

### Summary

The SNS governance `MintSnsTokens` proposal action has its 7-day rolling minting upper bound intentionally disabled in production code. The `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` returns `Decimal::MAX` (effectively infinity) instead of the treasury-valuation-based limit, allowing any SNS community with sufficient voting power to mint unlimited SNS tokens through successive governance proposals within any 7-day window.

### Finding Description

The SNS governance system supports a `MintSnsTokens` proposal action that mints new SNS tokens to an arbitrary recipient. The system was designed to enforce a 7-day rolling cap on minting, computed from treasury valuation via `mint_sns_tokens_7_day_total_upper_bound_tokens`. This enforcement is wired through the `TokenProposalAction` trait's `recent_amount_total_upper_bound_tokens` method, which is called during proposal validation in `treasury_valuation_if_proposal_amount_is_small_enough_or_err`.

However, the actual enforcement is commented out and replaced with a `Decimal::MAX` placeholder:

```rust
// TODO(NNS1-2982): Delete.
fn recent_amount_total_upper_bound_tokens(_valuation: &Valuation) -> Result<Decimal, String> {
    // Ideally, we'd return infinity, but Decimal does not have that. This is the next best
    // thing, and should be good enough, because we have already planned the obselences of this
    // code (see tickets NNS1-298(1|2)).
    Ok(Decimal::MAX)
}
``` [1](#0-0) 

The real enforcement is commented out:

```rust
/* TODO(NNS1-2982): Uncomment.
fn recent_amount_total_upper_bound_tokens(valuation: &Valuation) -> Result<Decimal, String> {
    mint_sns_tokens_7_day_total_upper_bound_tokens(*valuation)
        ...
}
*/
``` [2](#0-1) 

The validation path in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` computes `max_tokens` from this method and compares the proposal amount against it. Since `max_tokens` is always `Decimal::MAX`, the check `proposal_amount_tokens > allowance_remainder_tokens` is never true, and any amount passes. [3](#0-2) 

Once validated, `perform_mint_sns_tokens` executes unconditionally, calling `self.ledger.transfer_funds` from the minting account with no supply cap check: [4](#0-3) 

The integration test `sns_can_mint_funds_via_proposals` explicitly confirms this: a second minting proposal that should be rejected (because it exceeds the 7-day limit) is instead accepted, and the test has a `TODO(NNS1-2982): Delete this line` comment acknowledging the bypass: [5](#0-4) 

### Impact Explanation

Any SNS neuron holder with sufficient voting power can submit and pass multiple `MintSnsTokens` proposals within a 7-day window, minting an unbounded number of SNS tokens. This causes:

- **Unlimited token dilution** for all existing SNS token holders and stakers
- **Loss of value** for users who staked SNS tokens expecting a bounded emission schedule
- **Drain of SNS treasury value** through inflation, since the SNS token price is tied to treasury backing

This is a direct analog to the LybraFinance M-09 finding: a minting caller (SNS governance) does not enforce the supply/rate cap that the token system was designed to have, allowing emissions to exceed the intended maximum.

### Likelihood Explanation

- The entry path requires only normal SNS governance participation (submitting and voting on proposals) — no privileged keys, admin access, or external oracle manipulation needed.
- The disabled check is in production code, not a test stub.
- The `Decimal::MAX` placeholder is explicitly acknowledged as temporary but is currently live.
- The integration test confirms the bypass is reachable and functional today.

### Recommendation

Uncomment the real `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` (as indicated by `TODO(NNS1-2982): Uncomment`) and delete the `Decimal::MAX` placeholder. The correct implementation already exists in `mint_sns_tokens_7_day_total_upper_bound_tokens`: [6](#0-5) 

Additionally, update the integration test to assert that the second minting proposal is rejected (uncomment the `TODO(NNS1-2982): Uncomment` block in `sns_can_mint_funds_via_proposals`).

### Proof of Concept

1. Deploy an SNS with a treasury of SNS tokens worth ~500,000 XDR (a "medium" treasury). The 7-day minting cap should be 25% of treasury tokens.
2. Submit `MintSnsTokens` proposal #1 for 25% of treasury tokens. It passes and executes.
3. Immediately submit `MintSnsTokens` proposal #2 for another 25% of treasury tokens (within the same 7-day window).
4. Proposal #2 passes validation because `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`, so `proposal_amount_tokens > Decimal::MAX` is always false.
5. Proposal #2 executes via `perform_mint_sns_tokens`, minting another 25% — totaling 50% minted in one 7-day window instead of the intended 25%.
6. Repeat every few days to mint unlimited tokens.

This is confirmed by the existing integration test at line 966 where `doomed_make_proposal_result.unwrap()` succeeds (the second proposal is accepted) when the commented-out assertion block shows it should fail with `InvalidProposal / amount too large`. [7](#0-6)

    async fn perform_mint_sns_tokens(
        &mut self,
        mint: MintSnsTokens,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: mint
                .to_principal
                .ok_or(GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    "Expected mint to have a target principal",
                ))?
                .0,
            subaccount: mint
                .to_subaccount
                .as_ref()
                .map(|s| bytes_to_subaccount(&s.subaccount[..]))
                .transpose()?,
        };
        let amount_e8s = mint.amount_e8s.ok_or(GovernanceError::new_with_message(
            ErrorType::InvalidProposal,
            "Expected MintSnsTokens to have an an amount_e8s",
        ))?;
        self.ledger
            .transfer_funds(amount_e8s, 0, None, to, mint.memo())
            .await?;
        Ok(())
    }
