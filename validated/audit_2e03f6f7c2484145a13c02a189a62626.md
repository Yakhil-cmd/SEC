Audit Report

## Title
Wrong Field Assigned to `max_neurons_fund_participation_icp_e8s` in `From` Conversion - (File: rs/sns/swap/src/neurons_fund.rs)

## Summary
In the `From<ValidatedNeuronsFundParticipationConstraints<F>> for NeuronsFundParticipationConstraintsPb` implementation at lines 502–527 of `rs/sns/swap/src/neurons_fund.rs`, `max_neurons_fund_participation_icp_e8s` is populated with `value.min_direct_participation_threshold_icp_e8s` instead of `value.max_neurons_fund_participation_icp_e8s`. This is a copy-paste error that causes the serialized protobuf to carry a corrupted cap on Neurons' Fund participation whenever a `ValidatedNeuronsFundParticipationConstraints` is converted back to its protobuf form.

## Finding Description
At line 512–514 of `rs/sns/swap/src/neurons_fund.rs`, the `From` impl reads:

```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.min_direct_participation_threshold_icp_e8s,  // BUG
),
```

instead of:

```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.max_neurons_fund_participation_icp_e8s,
),
```

The two fields have entirely different semantics: `min_direct_participation_threshold_icp_e8s` is the floor below which the Neurons' Fund does not participate at all, while `max_neurons_fund_participation_icp_e8s` is the absolute cap on Neurons' Fund contributions. The `TryFrom` in the reverse direction (lines 536–553) correctly reads both fields separately, confirming they are distinct. This `From` conversion is the only serialization path from `ValidatedNeuronsFundParticipationConstraints` back to `NeuronsFundParticipationConstraintsPb` in the Swap canister. Any code path that round-trips through this conversion will produce a corrupted protobuf where the participation cap equals the participation threshold.

## Impact Explanation
This falls under **High** impact: significant SNS security impact with concrete user or protocol harm. Two concrete failure modes exist:

1. **Under-participation / swap abort**: In typical configurations, `min_direct_participation_threshold_icp_e8s` (e.g., 50 ICP) is far smaller than `max_neurons_fund_participation_icp_e8s` (e.g., 200,000 ICP). After the buggy conversion, the Swap canister believes the Neurons' Fund cap is 50 ICP. Any swap requiring more than 50 ICP of Neurons' Fund matching will be silently under-matched, potentially failing to reach `min_participants` or `min_direct_participation_icp_e8s`, causing the swap to abort and all participants to be refunded — blocking the SNS launch.

2. **Over-participation / governance invariant violation**: If `min_direct_participation_threshold_icp_e8s` exceeds the actual `max_neurons_fund_participation_icp_e8s`, the Neurons' Fund is permitted to contribute more than the NNS Governance-approved maximum, violating the conservation invariant enforced by governance.

## Likelihood Explanation
Every SNS swap that uses Neurons' Fund matched funding and exercises the `From` conversion path is affected. No special privileges are required — any SNS launch with Neurons' Fund participation enabled triggers this code. The bug is deterministic and reproducible on every affected swap. The `From` conversion is the sole serialization path from the validated struct back to protobuf in the Swap canister.

## Recommendation
Fix the copy-paste error in `rs/sns/swap/src/neurons_fund.rs` at line 513:

```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.max_neurons_fund_participation_icp_e8s,  // was: value.min_direct_participation_threshold_icp_e8s
),
```

Add a unit test that constructs a `ValidatedNeuronsFundParticipationConstraints` with distinct values for both fields, round-trips it through `From`, and asserts both output fields are correct.

## Proof of Concept
Given:
- `min_direct_participation_threshold_icp_e8s = 50 * E8`
- `max_neurons_fund_participation_icp_e8s = 200_000 * E8`

After the `From` conversion at lines 502–527:
- `output.min_direct_participation_threshold_icp_e8s = 50 * E8` ✓
- `output.max_neurons_fund_participation_icp_e8s = 50 * E8` ✗ (should be `200_000 * E8`)

A unit test constructing this struct and asserting `output.max_neurons_fund_participation_icp_e8s == 200_000 * E8` would fail deterministically, confirming the bug.
