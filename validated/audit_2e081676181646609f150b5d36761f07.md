Audit Report

## Title
Unbounded `join_all` Over All Managed Principals in `maybe_top_up` Can Permanently Stall the Ledger Suite Orchestrator's Cycle Top-Up Task - (File: rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs)

## Summary
`maybe_top_up` collects every managed principal (ledger, index, and all archive canisters for every ERC-20 token) into an unbounded `BTreeSet` and fans out a single `future::join_all` of concurrent `canister_cycles` inter-canister calls over the entire set. Because archive canisters are appended to the managed set by ordinary user transactions with no cap and no removal path, the set grows without bound. Once it is large enough to exhaust the IC's per-canister outstanding-callback budget, the task fails on every invocation and the orchestrator permanently stops topping up all managed canisters, eventually causing them to freeze.

## Finding Description
**Root cause — unbounded concurrent dispatch:**

`maybe_top_up` at lines 713–758 of `rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs` reads the full set of managed principals and dispatches one `canister_cycles` call per principal inside a single `future::join_all`:

```rust
let managed_principals: BTreeSet<_> =
    read_state(|s| s.all_managed_principals().cloned().collect());
...
let results = future::join_all(
    managed_principals.iter().map(|p| runtime.canister_cycles(*p)),
).await;
```

`all_managed_principals` (lines 553–556 of `src/state/mod.rs`) yields the ledger, index, and every archive for every managed ERC-20 token with no bound.

**Unbounded growth path:**

`discover_archives` (lines 1150–1192 of `scheduler/mod.rs`) calls `mutate_state(|s| s.record_archives(...))` to append newly discovered archive canisters to the managed set. The archive trigger threshold is 2 000 transactions (`trigger_threshold: 2_000` at line 958). Any user can submit ICRC-1 transfers to trigger archive creation; no privileged action is required after a token is live.

**No removal path:**

A grep across the entire LSO source confirms there is no `remove_archive`, `unmanage_canister`, or equivalent operation. The `State` struct (lines 518–530 of `state/mod.rs`) has no removal path for `managed_canisters`.

**Failure mode:**

The IC enforces a hard limit on the number of outstanding inter-canister callbacks a canister may hold simultaneously (500 per canister as documented in the IC specification). When `join_all` attempts to open more concurrent `canister_cycles` calls than this limit, the excess calls are rejected. The task returns an error, is rescheduled by the `run_task` retry logic (line 197: `schedule_after(RETRY_FREQUENCY, task_type, &runtime)`), and re-enters the same failing state on every subsequent invocation. The periodic `MaybeTopUp` task (rescheduled every hour at line 190) never succeeds again.

**Existing guards are absent:**

No chunking, pagination, semaphore, or per-batch cap exists anywhere in `maybe_top_up` or `discover_archives`. The comment at line 413 ("We expect usually 0 or 1 archive") acknowledges the expected scale but imposes no enforcement.

## Impact Explanation
When the task enters a permanent failure loop, the orchestrator stops sending top-up cycles to all managed ledger, index, and archive canisters. Those canisters drain their cycle balances, become frozen, and stop serving user requests — halting transfers, balance queries, and archive reads for every ckERC20 token managed by the LSO. This is a concrete, sustained DoS of the ckERC20 Chain Fusion financial infrastructure with direct user harm, matching the allowed impact class: **High — Significant Chain Fusion / ck-token infrastructure security impact with concrete user or protocol harm** and **High — Application/platform-level DoS not based on raw volumetric DDoS**.

## Likelihood Explanation
Adding a new ERC-20 token requires an NNS governance proposal (privileged), but once a token is live, **any unprivileged user** can trigger archive creation by submitting ICRC-1 transfers. With multiple ckERC20 tokens on mainnet and active trading, archive counts grow naturally without adversarial intent. Reaching the IC's 500-callback limit requires roughly 500 managed principals total across all tokens and their archives — achievable with tens of tokens each accumulating tens of thousands of transactions, which is realistic over the operational lifetime of the system.

## Recommendation
1. **Paginate `maybe_top_up`**: process managed principals in bounded batches (e.g., ≤ 100 at a time), persisting a cursor in state so each timer invocation advances the cursor rather than restarting from the full set.
2. **Cap concurrent calls**: replace the unbounded `future::join_all` with a chunked variant that dispatches at most a fixed number of calls per message execution.
3. **Add an `untrack_archive` operation**: allow decommissioned or excess archives to be removed from the managed set via a governance-gated call.
4. **Apply the same fix to `discover_archives`** (lines 1173–1178), which has an analogous unbounded `join_all` over all ledger principals.

## Proof of Concept
1. Deploy the LSO on a local replica or PocketIC test environment.
2. Add N ERC-20 tokens (N ≥ 20) via simulated NNS proposals.
3. For each token, submit enough ICRC-1 transfers to trigger archive creation (2 000 transfers per archive canister, targeting ≥ 25 archives per token to approach the 500-callback limit).
4. Wait for `Task::DiscoverArchives` to record the new archives in state via `record_archives`.
5. Trigger `Task::MaybeTopUp` and observe that `future::join_all` either returns callback-budget errors for the excess calls or traps on the instruction limit.
6. Confirm that subsequent `MaybeTopUp` invocations continue to fail and that managed canister cycle balances are no longer replenished.
7. An integration test can assert this invariant: after inserting > 500 principals into the managed set via `mutate_state`, calling `maybe_top_up` with a mock runtime that counts concurrent in-flight calls must observe that the call count exceeds the IC limit and the task returns an error.
