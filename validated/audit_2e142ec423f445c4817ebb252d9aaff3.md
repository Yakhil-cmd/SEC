### Title
Truncating Division in `native_per_pubdata` Calculation Causes Systematic Pubdata Fee Undercharging - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the `native_per_pubdata` rate used to charge users for pubdata consumption is computed with a truncating (floor) integer division: `pubdata_price / native_price`. Because integer division discards the remainder, the protocol systematically undercharges every transaction that generates pubdata. The magnitude of the loss scales with the ratio `(pubdata_price % native_price) / native_price` per pubdata byte, and can approach 100% of the per-byte fee when `pubdata_price` is just below a multiple of `native_price`. This is the direct analog of the M-08 "division in the midst of calculation" precision-loss class.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (L2 transaction validation path), `native_per_pubdata` is derived as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's standard truncating integer division — it discards the remainder. The truncated `native_per_pubdata` is then stored in `TxContextForPreAndPostProcessing` and used in every subsequent pubdata charge:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, line 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
```

So the actual native charged for pubdata is:

```
charged = pubdata_used * floor(pubdata_price / native_price)
```

whereas the correct amount should be:

```
correct = pubdata_used * pubdata_price / native_price
```

The difference — `pubdata_used * (pubdata_price % native_price) / native_price` — is silently lost every transaction.

**Contrast with `native_per_gas`**, which correctly uses ceiling division to protect the protocol:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

The same truncating pattern is also present in the public API helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;

### Impact Explanation

Every L2 transaction that writes to storage (generating pubdata) pays less than the true cost of that pubdata in native tokens. The operator/protocol receives:

```
pubdata_used * floor(pubdata_price / native_price)
```

instead of the correct:

```
pubdata_used * pubdata_price / native_price
```

**Concrete example:** If `pubdata_price = 101` and `native_price = 100`, then `native_per_pubdata = 1` (truncated from 1.01). For a transaction generating 1000 pubdata bytes, the protocol charges `1000` native instead of `1010` — a 1% loss. If `pubdata_price = 199` and `native_price = 100`, `native_per_pubdata = 1` (truncated from 1.99), and the loss approaches 50%. The loss is bounded by `(native_price - 1) / native_price` per pubdata byte, which can be close to 100% when `pubdata_price < native_price`.

This is a resource accounting bug causing continuous, protocol-wide value leakage on every transaction with pubdata. The operator is the direct victim. [5](#0-4) 

---

### Likelihood Explanation

Every L2 transaction that performs a storage write generates pubdata. In normal ZKsync OS operation, `pubdata_price` and `native_price` are independent oracle-supplied values and will rarely be exact multiples of each other. The truncation loss is therefore present in virtually every block. Any unprivileged user submitting a standard ERC-20 transfer or any contract interaction that touches storage triggers this path. No special privileges or attack setup are required. [6](#0-5) 

---

### Recommendation

Replace the truncating `wrapping_div` with ceiling division (`div_ceil`) for `native_per_pubdata`, consistent with how `native_per_gas` is already computed:

```rust
// Before (truncates, undercharges):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling, correct):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. This ensures all multiplication is effectively deferred and the remainder is not silently discarded, matching the pattern already used for `native_per_gas`. [1](#0-0) [4](#0-3)

### Proof of Concept

**Parameters:**
- `pubdata_price = 150`, `native_price = 100`
- Transaction generates `pubdata_used = 1000` bytes

**Current behavior (truncating division):**
```
native_per_pubdata = floor(150 / 100) = 1
charged_native     = 1000 * 1 = 1000
```

**Correct behavior (ceiling division):**
```
native_per_pubdata = ceil(150 / 100) = 2
charged_native     = 1000 * 2 = 2000
```

The protocol loses 1000 native tokens (50% of the correct pubdata fee) on this single transaction. Across all transactions in a block, this loss accumulates continuously. The entry path is any unprivileged L2 transaction that writes to storage, which flows through `validate_and_compute_fee_for_transaction` → `native_per_pubdata` truncation → `get_resources_to_charge_for_pubdata` → undercharged native. pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
