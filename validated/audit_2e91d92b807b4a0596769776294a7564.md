### Title
Rebasing NEP-141 Token Yield Permanently Frozen in Aurora Bridge — (`engine/src/engine.rs`, `engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

Aurora's NEP-141 ↔ ERC-20 bridge uses a fixed-amount accounting model that is incompatible with rebasing NEP-141 tokens (e.g., stNEAR). When a rebasing NEP-141 token is bridged to Aurora, the ERC-20 mirror mints exactly the deposited amount. As the underlying NEP-141 token rebases (accruing staking yield), Aurora's actual NEP-141 balance grows, but the ERC-20 mirror total supply does not. Because the only exit path requires burning ERC-20 tokens in a 1:1 ratio, the rebased yield accumulates in Aurora's NEP-141 account with no mechanism to recover it — permanently frozen.

### Finding Description

**Step 1 — Deposit (NEP-141 → ERC-20)**

A user calls `ft_transfer_call` on a rebasing NEP-141 token (e.g., stNEAR), transferring `N` tokens to Aurora. The NEP-141 contract calls Aurora's `ft_on_transfer`, which routes to `receive_erc20_tokens`:

```rust
// engine/src/engine.rs:803
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount),
```

Aurora mints exactly `N` ERC-20 tokens to the recipient. Aurora now holds `N` NEP-141 tokens.

**Step 2 — Rebase occurs**

The NEP-141 token rebases (daily staking rewards). Aurora's actual NEP-141 balance becomes `N + R` (where `R` is the rebased yield). The ERC-20 mirror total supply remains `N`. There is no mechanism in the bridge to observe or account for this balance increase.

**Step 3 — Withdrawal (ERC-20 → NEP-141)**

The user calls `withdrawToNear(amount)` on `EvmErc20.sol`:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol:53-62
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    // calls ExitToNear precompile at 0xe9217bc7...
}
```

This burns `amount` ERC-20 tokens and calls the `ExitToNear` precompile with that same `amount`. The precompile then constructs an `ft_transfer` call to the NEP-141 contract:

```rust
// engine-precompiles/src/native.rs:630-636
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()  // only the burned amount, not N+R
),
```

The user receives exactly `amount` NEP-141 tokens back. The rebased yield `R` remains in Aurora's NEP-141 account permanently, because:
- The ERC-20 total supply is exhausted at `N` (no tokens left to burn)
- The only exit path (`ExitToNear` precompile) requires burning ERC-20 tokens
- There is no admin recovery function or alternative exit path for the surplus NEP-141 balance

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

All staking yield (rebased amount `R`) accrued by any rebasing NEP-141 token held in Aurora's bridge account is permanently inaccessible. The ERC-20 mirror total supply is bounded by the originally deposited amount; once all ERC-20 tokens are burned and exited, Aurora still holds `R` NEP-141 tokens with no exit path. This is a structural, protocol-level loss of yield for every user who bridges a rebasing NEP-141 token.

### Likelihood Explanation

**Medium.** stNEAR (Metapool liquid staking token) is a rebasing NEP-141 token actively used on NEAR mainnet. Any user who bridges stNEAR to Aurora via the standard `ft_transfer_call` → `ft_on_transfer` flow triggers this condition. No special permissions or attacker actions are required — the yield loss is automatic and continuous from the moment of deposit.

### Recommendation

Two mitigations mirror those suggested in the external report:

1. **Use share-based accounting**: Instead of recording the raw token amount, record the number of shares (internal units) at deposit time and redeem by shares at withdrawal. This is analogous to using `wstETH` instead of `stETH`.
2. **Only support non-rebasing wrappers**: For known rebasing NEP-141 tokens, require users to wrap them into a non-rebasing equivalent before bridging (e.g., require wstNEAR instead of stNEAR).

### Proof of Concept

The bridge flow is confirmed by the existing test `test_exit_to_near` in `engine-tests/src/tests/erc20_connector.rs`, which verifies the 1:1 accounting invariant:

```
nep_141_balance_of(&nep_141, &aurora.id()) == FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT
erc20_balance(&erc20, ft_owner_address, &aurora) == (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)
```

If the NEP-141 token rebases between deposit and exit, Aurora's actual NEP-141 balance exceeds `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT`, but the ERC-20 balance does not. The surplus is unrecoverable.

**Concrete scenario**:
1. Alice bridges 100 stNEAR → receives 100 ERC-20 stNEAR on Aurora. Aurora holds 100 stNEAR.
2. stNEAR rebases +3%. Aurora's stNEAR balance becomes 103. ERC-20 total supply stays 100.
3. Alice burns all 100 ERC-20 tokens → receives 100 stNEAR back.
4. 3 stNEAR remain in Aurora's account on the stNEAR contract. No ERC-20 tokens exist to burn for exit. The 3 stNEAR are permanently frozen.
