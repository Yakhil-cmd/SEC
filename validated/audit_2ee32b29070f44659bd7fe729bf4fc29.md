### Title
Silent Return of Zero Gas in `calculate_attached_gas` Causes Token Bridge Operations to Succeed While Underlying NEAR Promises Fail — (File: `engine/src/contract_methods/connector.rs`)

### Summary
`calculate_attached_gas` silently returns `NearGas::new(0)` when prepaid gas is exhausted, instead of propagating an error. The `return_promise` function does not validate this return value and proceeds to schedule a NEAR promise with zero gas attached, then unconditionally returns `Ok(())`. This causes `withdraw`, `ft_transfer`, and `ft_transfer_call` to report success to callers while the underlying cross-contract promise will fail at runtime — an exact structural analog to the reported SwapLibrary silent-return-zero pattern.

### Finding Description

In `engine/src/contract_methods/connector.rs`, `calculate_attached_gas` (lines 587–596) returns `NearGas::new(0)` on the failure path instead of an error:

```rust
// TODO: Return `Result` with an error about lacking of gas instead.
fn calculate_attached_gas<E: Env>(env: &E) -> NearGas {
    let required_gas = env.used_gas().saturating_add(GAS_FOR_PROMISE_CREATION);
    if required_gas >= env.prepaid_gas() {
        NearGas::new(0)          // ← silent zero, not an Err
    } else {
        env.prepaid_gas() - required_gas
    }
}
``` [1](#0-0) 

The caller `return_promise` (lines 598–616) never inspects the returned gas value and always returns `Ok(())`:

```rust
fn return_promise<I: IO + PromiseHandler, E: Env>(
    mut io: I, env: &E, method: &str, args: Vec<u8>, deposit: Yocto,
) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
        ...
        attached_gas: calculate_attached_gas(env),  // ← unchecked zero
    };
    let promise_id = io.promise_create_call(&promise_args);
    io.promise_return(promise_id);
    Ok(())   // ← always succeeds regardless of gas
}
``` [2](#0-1) 

Every public token-bridge entry point delegates through `return_promise`:

- `withdraw` — bridges ETH from Aurora to Ethereum [3](#0-2) 
- `ft_transfer` — NEP-141 fungible token transfer [4](#0-3) 
- `ft_transfer_call` — NEP-141 transfer with receiver callback [5](#0-4) 

The structural parallel to the reported bug is exact:
- `_swapTokenExactOutput` → `calculate_attached_gas` (returns zero on failure instead of reverting/erroring)
- `swapTokensExactOutput` → `return_promise` (does not check the zero return, proceeds, reports success)

### Impact Explanation

**`withdraw` path (High — Temporary Freezing of Funds):** A user submits a `withdraw` call to bridge ETH from Aurora to Ethereum. If prepaid gas is insufficient, `calculate_attached_gas` returns 0, the promise to `engine_withdraw` on the connector is scheduled with 0 gas, and the NEAR runtime rejects its execution. The parent transaction returns `Ok(())`. The user's ETH is not burned and not bridged — it remains locked on Aurora — but the user observes a successful transaction receipt and believes the bridge operation completed. The user may then attempt to spend ETH on Ethereum that was never delivered, or fail to retry the withdrawal, leaving funds in a limbo state.

**`ft_transfer` / `ft_transfer_call` path (High — Temporary Freezing of Funds):** A user or contract calls `ft_transfer_call` to deposit NEP-141 tokens into a DeFi protocol. The promise to `engine_ft_transfer_call` on the connector is scheduled with 0 gas and fails. The parent transaction returns `Ok(())`. The receiving contract's `ft_on_transfer` is never invoked. Any downstream logic that acts on the apparent success (e.g., releasing collateral, crediting a position) operates on a false premise, creating conditions for fund loss.

### Likelihood Explanation

The failure condition — `used_gas + GAS_FOR_PROMISE_CREATION >= prepaid_gas` — is reachable by any unprivileged caller who attaches less than the required gas. The TODO comment in the source code (`// TODO: Return Result with an error about lacking of gas instead.`) is an explicit developer acknowledgment that this is a known defect, not an edge case. Users who rely on default gas estimates or who call these methods from within other contracts (where gas budgets are split) are particularly exposed. No special privilege is required; any token holder or bridge user can trigger this path.

### Recommendation

Change `calculate_attached_gas` to return `Result<NearGas, ContractError>` and propagate the error through `return_promise` to all callers. Each public entry point (`withdraw`, `ft_transfer`, `ft_transfer_call`, etc.) should surface the error rather than silently scheduling a zero-gas promise. Alternatively, add an explicit guard in `return_promise` that returns an error when the computed gas is zero, preventing the promise from being created at all.

### Proof of Concept

1. User calls `ft_transfer` attaching only 5 TGas (below the threshold).
2. Inside `return_promise`, `calculate_attached_gas` computes `used_gas + 2 TGas >= 5 TGas` → returns `NearGas::new(0)`.
3. `PromiseCreateArgs` is constructed with `attached_gas: NearGas::new(0)`.
4. `io.promise_create_call` schedules the promise; `io.promise_return` marks it as the return value.
5. `return_promise` returns `Ok(())` — the NEAR transaction is recorded as successful.
6. The NEAR runtime attempts to execute the `engine_ft_transfer` promise with 0 gas; execution fails.
7. No tokens are transferred. The sender's balance is unchanged. The recipient receives nothing.
8. The user's transaction receipt shows success; any contract or off-chain logic that branched on this success is now operating on a false state.
