Audit Report

## Title
`get_pending_proposals` Iterates All Open Proposals Without a Result Limit, Enabling Resource-Exhaustion DoS - (File: rs/nns/governance/src/governance.rs)

## Summary
`get_pending_proposals` collects every open proposal into an unbounded `Vec<ProposalInfo>` with no pagination cap, while `list_proposals` enforces `MAX_LIST_PROPOSAL_RESULTS = 100`. Because `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS = 10,000`, an attacker controlling a single neuron can drive the open-proposal count to the ceiling and cause every call to `get_pending_proposals` to exhaust the IC query instruction budget or exceed the per-message response size limit, making the endpoint and the Rosetta API integration that depends on it unavailable.

## Finding Description
`get_pending_proposals` at `rs/nns/governance/src/governance.rs` lines 3479–3494 performs an unbounded `.collect()` over all open proposals:

```rust
self.heap_data
    .proposals
    .values()
    .filter(|data| data.status() == ProposalStatus::Open)
    .map(|data| proposal_data_to_info(...))
    .collect()   // no limit
```

`list_proposals` at line 3623 caps output at `MAX_LIST_PROPOSAL_RESULTS = 100`; `get_pending_proposals` has no equivalent guard.

`ProposalDisplayOptions::for_get_pending_proposals` (lines 29–35 of `rs/nns/governance/src/pb/proposal_conversions.rs`) sets `omit_large_fields_requested: false`. The `multi_query: true` flag only strips large `ExecuteNnsFunction` payloads via `omit_large_execute_nns_function_payload()`; it does **not** strip ballot maps or other per-proposal data. Manage-neuron proposals carry smaller electoral rolls than regular proposals, but at 10,000 open proposals the aggregate ballot and metadata serialization is still unbounded.

The manage-neuron proposal guard at lines 5233–5246 blocks submission only once the count reaches `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS = 10,000`, confirming the ceiling an attacker can reach. Regular proposals are capped at `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 200` (line 252), so the combined worst-case is ~10,200 open proposals returned in a single response.

The endpoint is a `#[query]` callable by any principal (canister.rs lines 373–377). The Rosetta API `pending_proposals()` method (ledger_client.rs lines 375–396) calls this endpoint directly and propagates any failure as an `ApiError`.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS"* and *"Significant… Rosetta… infrastructure security impact with concrete user or protocol harm."*

When 10,000 manage-neuron proposals are open, every invocation of `get_pending_proposals` must iterate the full proposal BTreeMap, call `proposal_data_to_info` 10,000 times, and serialize the result. This either exhausts the IC query instruction budget (causing a trap) or produces a response exceeding the IC per-message size limit. Either outcome makes the endpoint unavailable for the duration of the attack, breaking the Rosetta API `/call?method=get_pending_proposals` path for all callers.

## Likelihood Explanation
An attacker needs only a single neuron (minimum 1 ICP stake) and the ability to submit manage-neuron proposals targeting their own neuron. Proposals remain open for their full voting period; the attacker can pipeline submissions to maintain the 10,000-proposal ceiling continuously. Rejection fees are charged only upon settlement, so the per-proposal cost is deferred and bounded by the voting period cadence. The attack requires no special privileges, no victim interaction, and no governance majority.

## Recommendation
Apply the same result cap used by `list_proposals` inside `get_pending_proposals`, or convert it to a paginated API:

```rust
self.heap_data
    .proposals
    .values()
    .filter(|data| data.status() == ProposalStatus::Open)
    .take(MAX_LIST_PROPOSAL_RESULTS as usize)   // add this
    .map(|data| proposal_data_to_info(...))
    .collect()
```

Callers needing the full list should use `list_proposals` with `include_status = [Open]` and paginate via `before_proposal`.

## Proof of Concept
1. Stake 1 ICP to create neuron `N` with sufficient dissolve delay.
2. In a loop, submit 10,000 `ManageNeuron` proposals targeting `N` (e.g., `IncreaseDissolveDelay` by 1 second). Each is accepted until the count reaches `MAX_NUMBER_OF_OPEN_MANAGE_NEURON_PROPOSALS`.
3. From any principal (including anonymous), call `get_pending_proposals` on the NNS governance canister.
4. Observe the query traps (instruction limit exceeded) or the IC rejects the response (size limit exceeded).
5. Confirm the Rosetta API `pending_proposals()` path returns an error for every request.

A deterministic integration test can replicate this using PocketIC: inject 10,000 synthetic open manage-neuron proposals into `heap_data.proposals`, then invoke `get_pending_proposals` and assert the query fails or the response byte count exceeds the IC limit.
