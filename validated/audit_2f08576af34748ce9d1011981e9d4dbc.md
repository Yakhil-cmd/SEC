### Title
ETH Permanently Locked at SHA256/RIPEMD160 Precompile Addresses Due to Missing Value Validation — (`engine-precompiles/src/hash.rs`)

---

### Summary

`SHA256::run` and `RIPEMD160::run` do not call `utils::validate_no_value_attached_to_precompile`, unlike every Aurora-specific precompile that handles this correctly. An unprivileged EVM user can CALL either precompile address with `value > 0`, the call succeeds, and the EVM executor credits the value to the precompile address. Because those addresses have no private key and no code, the ETH is permanently irrecoverable, causing insolvency.

---

### Finding Description

`SHA256::run` ignores `_context` entirely: [1](#0-0) 

`RIPEMD160::run` has the same pattern: [2](#0-1) 

Both functions accept `_context: &Context` (underscore-prefixed, unused) and never inspect `context.apparent_value`. Contrast with every Aurora-specific precompile: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

The guard itself is trivial — it returns `ExitError` when `value > 0`: [8](#0-7) 

When a CALL with `value = V` reaches SHA256 or RIPEMD160, SputnikVM's `MemoryStackState` transfers `V` from the caller to the precompile address before invoking `run`. Because `run` succeeds unconditionally (given sufficient gas), the transfer is committed. `Engine::apply` then processes the resulting `Apply::Modify` entries: [9](#0-8) 

The accounting sees sender balance decrease by `V` and precompile balance increase by `V`, netting to `Net::Zero`, which passes the invariant check: [10](#0-9) 

The ETH is now recorded at `0x0000...0002` or `0x0000...0003`. There is no private key for these addresses, no deployed code, and no engine-level recovery path (unlike `exit_to_near::ADDRESS`, which has a dedicated `refund_on_error` path): [11](#0-10) 

The Aurora developers themselves documented this exact risk in `native.rs` for the ERC-20 exit path: [12](#0-11) 

---

### Impact Explanation

Every wei sent to SHA256 or RIPEMD160 with `value > 0` is permanently unrecoverable. The total bridged ETH tracked by the connector exceeds the total ETH that can ever be withdrawn, growing with each such call. This is **Insolvency** / **Permanent Freezing of Funds**.

---

### Likelihood Explanation

The attack requires only a funded EOA and a standard EVM transaction. No privilege, no admin access, no social engineering. Any user — including a malicious one deliberately draining the pool — can execute this. The precompile addresses are well-known (`0x0000...0002`, `0x0000...0003`). The call succeeds silently and returns a valid hash output, giving no indication that funds were lost.

---

### Recommendation

Add the value guard at the top of both `run` implementations, mirroring every other Aurora precompile:

```rust
// In SHA256::run and RIPEMD160::run, before any other logic:
utils::validate_no_value_attached_to_precompile(context.apparent_value)?;
```

Also audit `ECRecover`, `Identity`, `ModExp`, `Blake2F`, `Bn256Add`, `Bn256Mul`, `Bn256Pair`, and `Secp256r1` for the same omission, as none of them currently call this guard either.

---

### Proof of Concept

**Unit-level (no network required):**

```rust
// engine-precompiles/src/hash.rs (add to #[cfg(test)] mod tests)
#[test]
fn test_sha256_rejects_nonzero_value() {
    use aurora_engine_types::U256;
    let mut ctx = new_context();
    ctx.apparent_value = U256::from(1u64); // 1 wei attached
    // Currently PASSES (returns Ok) — should return Err
    let result = SHA256.run(b"hello", Some(EthGas::new(60)), &ctx, false);
    assert!(result.is_err(), "SHA256 must reject nonzero value");
}
```

**Integration-level (engine-tests runner):**

1. Fund attacker EOA with 1 ETH via `receive_base_tokens`.
2. Submit `engine::call` with `origin = attacker`, `contract = SHA256::ADDRESS`, `value = Wei::new_u64(1)`, `input = b""`.
3. Assert `SubmitResult` is `Succeed`.
4. Assert `get_balance(&io, &SHA256::ADDRESS) == Wei::new_u64(1)`.
5. Assert no transaction exists that can move those funds out.
6. Repeat N times; assert `get_balance(&io, &SHA256::ADDRESS) == Wei::new_u64(N)` while total connector supply is unchanged — proving insolvency grows monotonically.

    fn run(
        &self,
        input: &[u8],
        target_gas: Option<EthGas>,
        _context: &Context,
        _is_static: bool,
    ) -> EvmPrecompileResult {
        let cost = Self::required_gas(input)?;
        if let Some(target_gas) = target_gas
            && cost > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        let hash = sdk::ripemd160(input);
        // The result needs to be padded with leading zeros because it is only 20 bytes, but
        // the evm works with 32-byte words.
        let mut output = vec![0u8; 32];
        output[12..].copy_from_slice(&hash);
        Ok(PrecompileOutput::without_logs(cost, output))
    }
