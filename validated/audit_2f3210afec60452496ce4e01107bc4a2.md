Audit Report

## Title
`SenderInfoContent` Signature Replay Across Target Canisters — (`File: rs/types/types/src/messages/http.rs`, `rs/validator/src/ingress_validation.rs`)

## Summary
The `SenderInfoContent` signed payload contains only the raw `info` bytes under a static `"ic-sender-info"` domain separator, with no binding to the target canister ID, method name, or any other request-scoping field. A user who obtains a valid `{info, signer, sig}` triple from a signing canister (e.g., Internet Identity) can replay it verbatim in requests to any other canister indefinitely, because the replica's `verify_sender_info_canister_sig` reconstructs and verifies the identical byte sequence regardless of the target. Any canister relying on `msg_caller_info_data()` for authorization cannot distinguish a freshly issued attestation from a cross-canister replay.

## Finding Description
`SenderInfoContent` is defined as a thin wrapper over raw bytes with no request context:

```rust
// rs/types/types/src/messages/http.rs L342-348
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);  // only raw info bytes
    }
}
```

Its `SignatureDomain` uses a static string with no canister ID:

```rust
// rs/types/types/src/crypto/sign.rs L161-165
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
```

The full signed bytes are therefore: `"\x0Eic-sender-info" || info_bytes` — identical for any target canister.

The validator reconstructs exactly this object:

```rust
// rs/validator/src/ingress_validation.rs L529-543
let sender_info_content = SenderInfoContent(&sender_info.info);
let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));
verify_canister_sig_with_fallback!(validator, &canister_sig, &sender_info_content, ...);
```

No field from the enclosing request — not `canister_id`, not `method_name`, not `ingress_expiry`, not the `MessageId` — is mixed into the signed payload. The outer `sender_sig` (over the `MessageId`) is controlled by the user themselves, so they can freely compose any previously obtained `sender_info` blob with a new request to any target canister. There is no feature flag gating this code path; `validate_sender_info` is called unconditionally from `validate_request_content` for all request types (`SignedIngressContent`, `Query`, `ReadState`).

The only existing check — that `pubkey_canister_id == sender_info.signer` — verifies the signing canister's identity, not the target canister. It is insufficient to prevent replay.

## Impact Explanation
This is a **High** severity finding. Any canister that reads `msg_caller_info_data()` and makes authorization decisions based on attested attributes (KYC status, age verification, role membership, etc.) cannot distinguish a freshly issued attestation from a replayed one obtained for a different service. An attacker can obtain a single valid `sender_info` attestation from Internet Identity and replay it to any number of target canisters, bypassing per-canister authorization gates. This constitutes unauthorized access to canister-controlled resources and directly undermines the Internet Identity attestation model, which is explicitly in scope.

## Likelihood Explanation
The attack requires only that the user has previously obtained a valid `sender_info` from a signing canister and that at least one target canister uses `msg_caller_info_data()` for authorization — both of which are the intended use-case for the feature. No privileged access, key compromise, or network-level attack is required. Any user of the system can perform it by reusing the same `{info, signer, sig}` triple across multiple requests with freshly signed outer `MessageId`s. The `sender_info.sig` carries no expiry of its own; only the outer `ingress_expiry` limits the request window, but the user can keep issuing fresh outer-signed requests with the same stale `sender_info`.

## Recommendation
Bind the `sender_info` signature to the target canister ID by including it in `SenderInfoContent`. The minimal fix is to extend the struct to carry the target canister ID and include it before the info bytes:

```rust
pub struct SenderInfoContent<'a> {
    pub target_canister_id: &'a CanisterId,
    pub info: &'a [u8],
}

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.target_canister_id.get().as_slice());
        bytes.extend_from_slice(self.info);
    }
}
```

Alternatively, include the full `MessageId` (which already commits to `canister_id`, `method_name`, `arg`, `sender`, and `ingress_expiry`) to make the attestation single-use per request. The signing canister (Internet Identity) must be updated to sign the correspondingly extended payload.

## Proof of Concept
1. User U (seed `S`, signing canister = Internet Identity `II`) calls canister A with a freshly issued `sender_info = {info: b"kyc_verified", signer: II, sig: σ}`. The replica accepts; `verify_sender_info_canister_sig` verifies σ over `"\x0Eic-sender-info" || b"kyc_verified"`.
2. U constructs a second request to canister B (a DeFi dapp gating withdrawals on `kyc_verified`), reusing the identical `sender_info` triple. U signs the new `MessageId` (which commits to canister B's ID) with their own key.
3. The replica again accepts: `verify_sender_info_canister_sig` reconstructs `SenderInfoContent(b"kyc_verified")` and verifies σ — the same bytes, the same signature, a different target canister. Canister B reads `msg_caller_info_data()`, sees `kyc_verified`, and grants the withdrawal.

The signed bytes at steps 1 and 3 are identical (`"\x0Eic-sender-info" || b"kyc_verified"`), so the same σ satisfies both verifications. A deterministic unit test can confirm this by constructing two `SenderInfoContent` instances with different (or no) canister IDs and asserting that `Signable::as_signed_bytes()` produces the same output for both. fn validate_sender_info<C: HttpRequestContent, R: RootOfTrustProvider>(
    request: &HttpRequest<C>,
    ingress_signature_verifier: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    let Some(sender_info) = request.sender_info() else {
        return Ok(());
    };

    // Per the spec, the sender_info signature must verify using the
    // envelope-level sender_pubkey as a canister signature public key.
    let sender_pubkey = match request.authentication() {
        Authentication::Authenticated(sig) => &sig.signer_pubkey,
        Authentication::Anonymous => {
            return Err(InvalidSenderInfo(
                "sender_info requires an authenticated request with sender_pubkey".to_string(),
            ));
        }
    };

    verify_sender_info_canister_sig(
        sender_info,
        sender_pubkey,
        ingress_signature_verifier,
        root_of_trust_provider,
    )
}
