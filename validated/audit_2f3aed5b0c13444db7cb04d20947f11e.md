### Title
Pre-EIP155 Legacy Transaction Cross-Chain Replay Enables Direct Fund Theft - (File: `engine-transactions/src/legacy.rs`, `engine/src/engine.rs`)

### Summary
The Aurora Engine accepts pre-EIP155 legacy transactions (v=27 or v=28) whose signatures do not commit to any chain ID. The chain ID validation in `submit_with_alt_modexp` is conditional and is entirely skipped when `transaction.chain_id` is `None`. An unprivileged attacker who observes a pre-EIP155 transaction on any EVM chain can replay it verbatim against Aurora's public `submit` endpoint, draining the victim's Aurora ETH balance, provided the victim's nonce on Aurora matches the replayed transaction's nonce.

### Finding Description

**Root cause — no chain ID in the signed hash for v=27/28:**

In `LegacyEthSignedTransaction::sender()`, when `v` is 27 or 28, `chain_id` is set to `None` and `rlp_append_unsigned` is called without a chain ID, so the signed message hash covers only `(nonce, gas_price, gas_limit, to, value, data)` — identical across every EVM chain. [1](#0-0) 

**Root cause — chain ID validation is conditional and skipped when `None`:**

In `submit_with_alt_modexp`, the chain ID check is guarded by `if let Some(chain_id) = transaction.chain_id`. When the legacy transaction carries `chain_id: None`, the entire guard is false and the check is never executed. The only remaining protection is the nonce check, which prevents same-chain replay but not cross-chain replay. [2](#0-1) 

**How `chain_id: None` propagates from the transaction type:** [3](#0-2) 

**Public entry point — no access restriction:** [4](#0-3) 

The acceptance of pre-EIP155 transactions was deliberately re-enabled (after being removed in v2.4.0) to support EIP-1820 deployment. The CHANGES.md entry confirms this: [5](#0-4) 

However, the re-enablement introduced no compensating control to prevent cross-chain replay.

### Impact Explanation
An attacker who observes any pre-EIP155 transaction (v=27/28) on Ethereum mainnet or any other EVM chain can submit the identical RLP-encoded bytes to Aurora's `submit` method. If the signing address has the same nonce on Aurora and holds ETH there, the transaction executes and transfers the victim's Aurora ETH to the transaction's `to` address. This is direct, irreversible theft of at-rest user funds — **Critical** under the allowed impact scope.

### Likelihood Explanation
Pre-EIP155 transactions are uncommon in modern tooling but are still produced by: (a) older hardware/software wallets that predate EIP-155, (b) the canonical EIP-1820 deployer transaction (nonce=0, v=27), and (c) any protocol that deliberately uses chain-agnostic signatures. The attack is most viable against new Aurora users whose nonce is 0, matching the nonce of a replayed transaction. No privileged access, leaked keys, or social engineering is required — only the ability to call the public `submit` NEAR method.

### Recommendation
1. **Reject pre-EIP155 transactions globally** (restore the v2.4.0 behavior). For EIP-1820 support, deploy the contract through a privileged admin path or a one-time allowlist keyed on the exact transaction hash rather than accepting all v=27/28 transactions.
2. If pre-EIP155 transactions must remain accepted, add an explicit allowlist of known safe transaction hashes (e.g., the EIP-1820 deployer tx hash) and reject all other v=27/28 transactions.
3. At minimum, document the cross-chain replay risk prominently so integrators are aware.

### Proof of Concept

1. Alice holds 5 ETH on Aurora mainnet; her Aurora nonce is 0.
2. Alice signs a pre-EIP155 transfer on Ethereum mainnet (v=27, nonce=0) sending 5 ETH to Bob.
3. Attacker observes the transaction on Ethereum and extracts the RLP bytes.
4. Attacker calls Aurora's `submit` NEAR method with those identical bytes.
5. `LegacyEthSignedTransaction::sender()` recovers Alice's address (the hash is chain-agnostic).
6. `submit_with_alt_modexp` skips the chain ID check (`transaction.chain_id` is `None`).
7. `check_nonce` passes (Alice's Aurora nonce is 0, matching the transaction).
8. The EVM executes the transfer; Alice loses 5 ETH on Aurora. [6](#0-5) [7](#0-6)

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);
        let message_hash = sdk::keccak(rlp_stream.as_raw());
        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
            .map_err(|_| Error::EcRecover)
    }

#[named]
pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let tx_data = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let args = SubmitArgs {
            tx_data,
            ..Default::default()
        };
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}
