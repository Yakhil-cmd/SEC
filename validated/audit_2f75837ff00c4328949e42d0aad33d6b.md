### Title
Unchecked Precompile Call Return Value in `withdrawToNear()` and `withdrawToEthereum()` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary
In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear()` and `withdrawToEthereum()` burn the caller's tokens first via `_burn()`, then invoke the Aurora precompile via a low-level `call()` in inline assembly. The return value of that `call()` is captured in `res` but **never checked**. If the precompile call fails for any reason (e.g., the precompile is paused, the recipient is invalid, or any other revert condition inside the precompile), the tokens are permanently destroyed with no withdrawal occurring and no revert to protect the user.

### Finding Description
Both production ERC-20 bridge contracts follow the same pattern:

```solidity
// EvmErc20.sol – withdrawToNear (lines 53-63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert if call fails
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum()`:

The same flaw is present in `EvmErc20V2.sol`: [3](#0-2) 

The `exitToNear` precompile (`engine-precompiles/src/native.rs`) can return errors for multiple reasons — invalid recipient account ID, paused precompile flag, amount overflow, missing wNEAR address configuration, etc. When the precompile returns an error, the EVM sets `res = 0` but does **not** revert the calling frame. Because the `EvmErc20` contracts never inspect `res`, the outer call succeeds, the `_burn()` is committed, and the user's tokens are gone. [4](#0-3) 

The engine deploys `EvmErc20.bin` or `EvmErc20V2.bin` as the canonical ERC-20 mirror for every bridged NEP-141 token: [5](#0-4) 

### Impact Explanation
Any user who calls `withdrawToNear()` or `withdrawToEthereum()` on a bridged ERC-20 token while the precompile is in a failing state (paused, misconfigured, or given a malformed recipient) will have their tokens burned with no corresponding NEP-141 or Ethereum transfer. The tokens are permanently destroyed. This constitutes **permanent freezing/loss of user funds** — Critical impact.

### Likelihood Explanation
The `exitToNear` and `exitToEthereum` precompiles are pausable via `EnginePrecompilesPauser`. During any pause window — even a brief maintenance pause — users who submit `withdrawToNear()` transactions will silently lose their tokens. Additionally, a user who supplies a malformed NEAR account ID as `recipient` (e.g., exceeding 64 bytes, containing invalid characters) will trigger a precompile error and lose their tokens. Both scenarios are reachable by an ordinary unprivileged EVM user with no special access.

### Recommendation
Check the return value of the low-level `call()` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Applying this fix to both `withdrawToNear()` and `withdrawToEthereum()` in both `EvmErc20.sol` and `EvmErc20V2.sol` ensures that if the precompile call fails, the entire transaction reverts, the `_burn()` is rolled back, and the user retains their tokens.

### Proof of Concept
1. Admin pauses the `exitToNear` precompile (legitimate maintenance action).
2. User calls `withdrawToNear(validRecipient, 1000)` on a bridged ERC-20.
3. `_burn(msg.sender, 1000)` executes — 1000 tokens destroyed.
4. The inline `call()` to the paused precompile returns `res = 0`.
5. `res` is never checked; the function returns without reverting.
6. Transaction succeeds on-chain; user has lost 1000 tokens permanently with no NEP-141 credit issued.
