### Title
Unpermissioned `deploy_upgrade` Allows Any Caller to Force Deployment of Staged Contract Code - (File: engine/src/lib.rs)

### Summary
The `deploy_upgrade` function in the Aurora Engine NEAR contract is callable by any NEAR account. While `stage_upgrade` correctly enforces `require_owner_only`, the companion `deploy_upgrade` function that actually executes the self-deployment omits this check entirely. Any unprivileged NEAR account can trigger a staged upgrade deployment once the block-height delay has elapsed, including racing against an owner who is attempting to cancel a staged upgrade by pausing the contract.

### Finding Description
`stage_upgrade` stores new contract bytecode and records a future block height after which deployment is permitted. It correctly gates access with `require_owner_only`: [1](#0-0) 

`deploy_upgrade`, however, only checks that the contract is running and that the current block height exceeds the staged index. There is no caller identity check: [2](#0-1) 

The exported NEAR entry point passes control directly to this unguarded logic: [3](#0-2) 

Compare with `require_owner_only`, which is used consistently on every other sensitive administrative function: [4](#0-3) 

The only defensive path available to the owner is to pause the contract (which blocks `deploy_upgrade` via `require_running`), but this creates a race condition: an attacker monitoring the chain can call `deploy_upgrade` in the same block or before the owner's pause transaction is included.

### Impact Explanation
If the owner stages an upgrade and subsequently discovers a critical bug in the staged bytecode, the intended remediation is to pause the contract and stage a corrected version. An attacker can front-run the pause by calling `deploy_upgrade` the moment the delay expires, forcing deployment of the vulnerable bytecode. A successfully deployed buggy contract version can lead to permanent freezing of all user funds held in the Aurora Engine or direct theft of those funds, depending on the nature of the bug in the staged code. The Aurora Engine holds bridged ETH and NEP-141 token balances for all users, making the blast radius protocol-wide.

**Impact: High — Temporary or permanent freezing of funds; escalates to Critical if the forced-deployed code contains a fund-draining bug.**

### Likelihood Explanation
The delay window (`upgrade_delay_blocks`) is set by the owner and may be short. Any NEAR account — including a bot watching the chain — can call `deploy_upgrade` with no special privileges, no deposit, and no prior relationship with the contract. The attacker only needs to observe a staged upgrade and submit a transaction after the delay block is reached. This is straightforward to automate and requires no leaked keys or social engineering.

### Recommendation
Add `require_owner_only` to `deploy_upgrade`, mirroring the pattern used in `stage_upgrade`:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)
        .map_err(ContractError::msg)
        .sdk_unwrap();
    // ADD THIS:
    require_owner_only(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

Additionally, consider adding a `cancel_upgrade` function (owner-only) that removes the staged code and the delay index from storage, giving the owner a clean abort path that does not require pausing the entire contract.

### Proof of Concept

1. Owner calls `stage_upgrade` with new bytecode `B` and a delay of `D` blocks. This succeeds because `stage_upgrade` enforces `require_owner_only`.
2. Owner discovers a critical vulnerability in `B` and prepares a pause transaction.
3. Attacker monitors the chain. At block `current_block > stage_block + D`, the attacker submits a call to `deploy_upgrade` with no special arguments.
4. `deploy_upgrade` passes both checks (`require_running` → contract is running; block height check → delay elapsed) and calls `Runtime::self_deploy`, replacing the live contract with the vulnerable bytecode `B`.
5. If the attacker's transaction is included before the owner's pause transaction (a realistic race on NEAR), the vulnerable code is now live.
6. The attacker (or anyone) can now exploit the vulnerability in `B` to steal or freeze user funds.

The entry path is fully unprivileged: any NEAR account ID, no deposit required, no prior state needed beyond a staged upgrade existing. [2](#0-1) [1](#0-0)

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;

        io.write_storage(

        );
        Ok(())
    })
}

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
