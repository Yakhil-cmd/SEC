Audit Report

## Title
Missing Voting Power Snapshot Mechanism in SNS Governance Allows Immediate Governance Takeover - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's `compute_ballots_for_new_proposal` function assigns ballots using live voting power at proposal creation time with no historical snapshot or spike-detection guard. The NNS governance canister has an explicit `VotingPowerSnapshots` mechanism that detects sudden increases in total voting power and falls back to a prior snapshot when creating ballots. An attacker who acquires a dominant stake in SNS tokens can immediately submit and pass proposals with no reaction window for other participants.

## Finding Description
`rs/sns/governance/src/governance.rs` lines 5225–5294: `compute_ballots_for_new_proposal` iterates `self.proto.neurons.iter()` and calls `v.voting_power(now_seconds, ...)` for every neuron, inserting the result directly into the ballot map. There is no reference to any snapshot store, no spike detection, and no fallback to a historical baseline.

By contrast, `rs/nns/governance/src/governance.rs` lines 5504–5524 calls `VOTING_POWER_SNAPSHOTS.with_borrow(|snapshots| snapshots.previous_ballots_if_voting_power_spike_detected(...))` before creating ballots. The snapshot store (`rs/nns/governance/src/governance/voting_power_snapshots.rs`) keeps up to 7 daily snapshots and triggers spike detection when current total voting power exceeds 1.5× the minimum of those snapshots, using the historical snapshot for ballot creation in that case.

The SNS has no equivalent timer task, no stable-structure snapshot store, and no spike-detection path. The proposal submission flow in `make_proposal` (lines 3448–3674) proceeds directly from `compute_ballots_for_new_proposal` to `insert_proposal` with no interposed guard.

Proposal criticality is topic-based. `UpgradeSnsControlledCanister` falls under `DappCanisterManagement` (`is_critical: false`), requiring only >50% of total voting power for early adoption. `TransferSnsTreasuryFunds` and `ManageNervousSystemParameters` fall under `TreasuryAssetManagement` and `DaoCommunitySettings` respectively (both `is_critical: true`), requiring >67% of total voting power for early adoption. An attacker with >50% of total voting power can immediately pass normal-criticality proposals; an attacker with >67% can immediately pass critical ones.

## Impact Explanation
This matches the allowed High impact: "Significant SNS … security impact with concrete user or protocol harm." An attacker who acquires a sufficient stake can immediately execute `UpgradeSnsControlledCanister` (replacing dapp code with malicious code) with >50% of total voting power, or `TransferSnsTreasuryFunds` / `ManageNervousSystemParameters` with >67%. The attack requires no privileged access, no admin keys, and no social engineering — only token acquisition followed by standard `manage_neuron` and `make_proposal` ingress calls. For SNS DAOs with low staking participation or concentrated early-stage token distribution, these thresholds are reachable. The NNS explicitly deployed the snapshot mechanism to address this exact risk; the SNS has not received the same protection.

## Likelihood Explanation
The attack is economically constrained: the attacker must acquire >50% (or >67% for critical proposals) of the circulating staked voting power. For early-stage SNS projects with low staking participation or thin DEX liquidity, this threshold may be reachable. The attack is repeatable, requires no special protocol knowledge beyond standard SNS ingress calls, and leaves no reaction window for defenders because ballot creation uses the live state at the moment of proposal submission.

## Recommendation
Implement a voting power snapshot mechanism in the SNS governance canister analogous to `rs/nns/governance/src/governance/voting_power_snapshots.rs`:
1. Add a recurring timer task (analogous to `SnapshotVotingPowerTask` in `rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`) that records total voting power periodically.
2. In `compute_ballots_for_new_proposal`, compare the current total voting power against the historical minimum. If a spike is detected (e.g., current > 1.5× minimum), use the previous snapshot's voting power distribution for ballot creation.
3. Define a staleness bound so that very old snapshots are not used.

## Proof of Concept
1. Deploy an SNS with a small number of neurons holding, say, 40% of total staked voting power.
2. As an attacker, acquire tokens representing >50% of total supply, stake them in a neuron with dissolve delay ≥ `neuron_minimum_dissolve_delay_to_vote_seconds` (default 6 months, but configurable per SNS).
3. Immediately call `manage_neuron` → `MakeProposal` with action `UpgradeSnsControlledCanister` (normal criticality, threshold >50% of total).
4. `compute_ballots_for_new_proposal` reads live state, assigns the attacker's full newly-staked voting power, and the proposer's automatic yes-vote triggers `early_decision` returning `Vote::Yes` because the attacker's ballot exceeds 50% of total.
5. The proposal is adopted and executed in the same round with no opportunity for other participants to react.
6. For `TransferSnsTreasuryFunds` (critical, threshold >67%), repeat with >67% of total voting power.

A deterministic integration test using `PocketIC` or `StateMachine` can reproduce this by: setting up an SNS with existing neurons holding <50% of total voting power, minting tokens to an attacker account, staking a neuron, and asserting that `make_proposal` with `UpgradeSnsControlledCanister` returns an executed proposal in the same block — with no snapshot check intervening.
