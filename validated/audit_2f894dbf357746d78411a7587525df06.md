### Title
SNS Governance `perform_manage_nervous_system_parameters` Changes `VotingRewardsParameters` Without First Distributing Pending Rewards - (File: `rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS governance canister's `perform_manage_nervous_system_parameters` function applies new `VotingRewardsParameters` (reward rates, round duration) directly to state without first flushing the current reward period via `distribute_rewards`. Because `distribute_rewards` reads reward-rate parameters from the **current** live state at the moment it runs, any parameter change that reduces reward rates retroactively reprices all rewards that accrued since the last settled reward event, causing neuron holders to receive fewer tokens than they earned under the parameters that were in effect when they voted.

---

### Impact Explanation

**Governance authorization / reward accounting bug.**

When an SNS governance proposal lowers `initial_reward_rate_basis_points` (or `final_reward_rate_basis_points`) and the proposal is executed before the next `distribute_rewards` call, all rewards that accrued since the last settled reward event are repriced at the new lower rate. Neurons that voted in good faith during that window receive materially fewer maturity tokens than they were entitled to under the parameters in effect when they cast their votes. The shortfall is permanent — the reward event records the lower amount and the difference is never recovered.

A second variant: if `round_duration_seconds` is increased to a value larger than the elapsed time since the last reward event, `new_rounds_count` evaluates to 0 and `distribute_rewards` returns early without settling any proposals or advancing `latest_reward_event.end_timestamp_seconds`. Proposals already in `ReadyToSettle` state are not settled, and the reward purse for that period is not credited, delaying (and potentially distorting) neuron maturity accrual.

---

### Likelihood Explanation

Any SNS community that passes a `ManageNervousSystemParameters` proposal touching `voting_rewards_parameters` triggers this condition. No malicious actor is required — a routine, well-intentioned governance action (e.g., reducing inflation) is sufficient. The `distribute_rewards` call is driven by a periodic timer task and is not atomically coupled to proposal execution, so the window always exists. The integration test `test_change_voting_rewards_round_duration` in `rs/sns/integration_tests/src/proposals.rs` (line 1659) demonstrates that changing `round_duration_seconds` mid-flight is an expected, supported operation, confirming the code path is live and reachable. [6](#0-5) 

---

### Recommendation

Inside `perform_manage_nervous_system_parameters`, before overwriting `self.proto.parameters`, check whether the incoming proposal modifies `voting_rewards_parameters`. If it does, call `distribute_rewards` (or the equivalent settlement logic) first to flush all accrued-but-unsettled rewards under the old parameters. This mirrors the `updatePool()` pattern recommended in the reference report and ensures that reward accounting is always consistent with the parameters that were in effect during each round.

---

### Proof of Concept

1. SNS is live with `initial_reward_rate_basis_points = 1000` and `round_duration_seconds = 86400` (1 day).
2. Neurons vote on several proposals over 10 days; those proposals enter `ReadyToSettle` state. No reward event has fired yet for this window.
3. An SNS governance proposal (`ManageNervousSystemParameters`) is submitted and adopted, setting `initial_reward_rate_basis_points = 500` (halving the initial rate).
4. `perform_manage_nervous_system_parameters` executes at line 2597: `self.proto.parameters = Some(new_params)`. No `distribute_rewards` call precedes this.
5. The periodic timer fires and calls `distribute_rewards`. It reads `initial_reward_rate_basis_points = 500` from the now-updated parameters and prices all 10 elapsed rounds at the new lower rate.
6. Neurons receive ≈50 % of the maturity they earned under the original rate for those 10 days. The shortfall is not recoverable. [7](#0-6) [8](#0-7)

    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }

pub struct VotingRewardsParameters {
    /// The amount of time between reward events.
    ///
    /// Must be > 0.
    ///
    /// During such periods, proposals enter the ReadyToSettle state. Once the round is over, voting
    /// for those proposals entitle voters to voting rewards. Such rewards are calculated in
    /// the governance canister's run_periodic_tasks function.
    ///
    /// This is a nominal amount. That is, the actual time between reward
    /// calculations and distribution cannot be guaranteed to be perfectly
    /// periodic, but actual inter-reward periods are generally expected to be
    /// within a few seconds of this.
    ///
    /// This supersedes super.reward_distribution_period_seconds.
    pub round_duration_seconds: Option<u64>,
    /// The amount of time that the growth rate changes (presumably, decreases)
    /// from the initial growth rate to the final growth rate. (See the two
    /// *_reward_rate_basis_points fields bellow.) The transition is quadratic, and
    /// levels out at the end of the growth rate transition period.
    pub reward_rate_transition_duration_seconds: Option<u64>,
    /// The amount of rewards is proportional to token_supply * current_rate. In
    /// turn, current_rate is somewhere between `initial_reward_rate_basis_points`
    /// and `final_reward_rate_basis_points`. In the first reward period, it is the
    /// initial growth rate, and after the growth rate transition period has elapsed,
    /// the growth rate becomes the final growth rate, and remains at that value for
    /// the rest of time. The transition between the initial and final growth rates is
    /// quadratic, and levels out at the end of the growth rate transition period.
    ///
    /// (A basis point is one in ten thousand.)
    pub initial_reward_rate_basis_points: Option<u64>,
    pub final_reward_rate_basis_points: Option<u64>,
}
