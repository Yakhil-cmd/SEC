### Title
`EvmErc20V2.sol` `withdrawToNear` Encodes `sender` Into Precompile Input, Causing Burned ERC-20 Amount to Diverge From NEP-141 Transfer Amount When `error_refund` Feature Is Disabled — (`File: etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.sol::withdrawToNear` prepends the caller's address (`sender`, 20 bytes) into the raw byte payload it sends to the `ExitToNear` precompile. The `ExitToNear` precompile in `engine-precompiles/src/native.rs` only consumes those 20 bytes as a `refund_address` when compiled with the `error_refund` Cargo feature. When compiled **without** that feature (the default), the precompile skips only the 1-byte flag and immediately reads the next 32 bytes as the ERC-20 amount — which are actually `[sender_20_bytes ‖ amount_first_12_bytes]`, not the real amount. The ERC-20 burn (`_burn`) uses the correct `amount`, but the NEP-141 `ft_transfer` call uses a completely different, astronomically large value that fails the `u128::MAX` guard, causing the precompile call to silently fail. Because the Solidity assembly block does not check the `call` return value, the transaction succeeds, ERC-20 tokens are permanently destroyed, and no NEP-141 tokens are released.

### Finding Description

**Root cause — `EvmErc20V2.sol` (lines 53–63):**

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // burns `amount` ERC-20 tokens

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    //                                              ^^^^^^ 20 extra bytes inserted
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // `res` is never checked — silent failure is possible
    }
}

`parse_amount` rejects any value exceeding `u128::MAX`:

```rust
fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    Ok(amount)
}
``` [4](#0-3) 

A typical 20-byte Ethereum address has non-zero leading bytes (e.g., `0x1234…`). Prepended to the 32-byte amount slot, the resulting 32-byte value is orders of magnitude larger than `u128::MAX`, so `parse_amount` always returns `Err`. The EVM `call` to the precompile returns `0` (failure), but the Solidity assembly never checks `res`, so the outer transaction does not revert.
