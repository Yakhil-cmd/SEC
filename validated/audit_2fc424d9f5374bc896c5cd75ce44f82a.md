### Title
Double-charge of `STORAGE_AMOUNT` wNEAR on concurrent XCC transactions for a new router with no refund path — (File: `engine-precompiles/src/xcc.rs`, `engine/src/xcc.rs`)

---

### Summary

Two EVM transactions from the same EVM address submitted in the same NEAR block can both observe `AddressVersionStatus::DeployNeeded{create_needed:true}` and both charge `STORAGE_AMOUNT` wNEAR from the sender. Only the first deploy batch succeeds; the second batch fails with "account already exists." The second transaction's `STORAGE_AMOUNT` wNEAR is moved to the engine's implicit address but never burned or refunded, because `withdraw_wnear_to_router` aborts when its predecessor callback (`factory_update_address_version`) reports failure. There is no recovery path for the stranded wNEAR.

---

### Finding Description

**Step 1 — EVM precompile charges wNEAR (engine-precompiles/src/xcc.rs)**

When the XCC precompile runs, it reads `get_code_version_of_address` from NEAR storage. If the result is `None` (router not yet deployed), it adds `STORAGE_AMOUNT` to `required_near` and immediately executes a `transferFrom` inside the EVM, moving `required_near` wNEAR from the sender to the engine's implicit address. [1](#0-0) 

This EVM state change is committed as part of the current NEAR transaction. The NEAR storage key for `get_code_version_of_address` is **not** updated here — it is only updated later, in a future block, when the `factory_update_address_version` callback fires.

**Step 2 — Both Tx1 and Tx2 in the same block see `None`**

NEAR processes transactions in a block sequentially. After Tx1 completes, the EVM state (sender's wNEAR balance) is updated, but `get_code_version_of_address` still returns `None` because `factory_update_address_version` has not yet fired. When Tx2 runs in the same block, it reads `None` again and charges another `STORAGE_AMOUNT`. [2](#0-1) 

**Step 3 — Both transactions schedule a deploy batch for the same sub-account**

`filter_promises_from_logs` in `engine.rs` reads the XCC log and calls `handle_precompile_promise` for each transaction. [3](#0-2) 

`handle_precompile_promise` also reads `get_code_version_of_address` and, finding `None`, schedules `CreateAccount + Transfer(STORAGE_AMOUNT) + DeployContract + Initialize` for the same router sub-account, followed by a `factory_update_address_version` callback, followed by `withdraw_wnear_to_router`. [4](#0-3) 

**Step 4 — Second deploy batch fails; `withdraw_wnear_to_router` aborts**

In a future block, Tx1's batch succeeds and `factory_update_address_version` writes the version. Tx2's batch fails because `CreateAccount` panics on an already-existing account. `factory_update_address_version` is still called as a callback but returns `ERR_ROUTER_DEPLOY_FAILED`. [5](#0-4) 

`withdraw_wnear_to_router` is chained after `factory_update_address_version`. It checks `promise_result_check()` and aborts when the predecessor failed: [6](#0-5) 

Because `withdraw_wnear_to_router` aborts, the wNEAR that was moved to the engine's implicit address in Step 1 is **never burned**. It remains stranded in the engine's implicit address — an EVM address derived from the engine's NEAR account ID for which no private key exists and no recovery method is implemented.

---

### Impact Explanation

The sender loses `STORAGE_AMOUNT` (2 × 10²⁴ yoctoNEAR = 2 NEAR) per duplicate XCC transaction targeting a not-yet-deployed router. The wNEAR is irrecoverably stranded in the engine's implicit EVM address. No refund path exists in the codebase. This satisfies **High — Theft of unclaimed yield**: the sender paid for router storage staking but received neither the deployed router nor a refund.

---

### Likelihood Explanation

The precondition — two XCC transactions from the same EVM address in the same NEAR block — is reachable without any privileged access:

- A user retrying a stuck transaction before the first one is confirmed lands both in the same block.
- A smart contract that issues two XCC calls in the same EVM transaction (via two precompile invocations) produces exactly this scenario.
- A relayer batching multiple user-signed EVM transactions into the same NEAR block triggers it for any user whose router does not yet exist.

No admin compromise, key leak, or governance capture is required.

---

### Recommendation

1. **Idempotent deploy guard**: In `handle_precompile_promise`, before scheduling the deploy batch, write a "deploy-in-progress" sentinel to NEAR storage for the sender's address. On entry, if the sentinel is present, skip the deploy batch and treat the router as already being created. Clear the sentinel in `factory_update_address_version`.

2. **Refund on deploy failure**: Add a dedicated refund callback chained after the deploy batch that, on failure, returns the `STORAGE_AMOUNT` wNEAR from the engine's implicit address back to the original sender's EVM address.

3. **Alternatively**, move the wNEAR charge out of the synchronous EVM precompile and into the `withdraw_wnear_to_router` NEAR callback, so the charge only occurs after the deploy batch result is known.

---

### Proof of Concept

```
1. Deploy Aurora engine locally (sandbox).
2. Fund EVM address A with ≥ 2 × STORAGE_AMOUNT wNEAR.
3. In a single NEAR block, submit two `submit` calls to the engine,
   each containing a signed EVM transaction from address A invoking
   the XCC precompile (CrossContractCallArgs::Eager with any valid target).
4. Wait for all promise callbacks to resolve.
5. Assert:
   - Router sub-account exists (created by Tx1's batch).
   - Address A's wNEAR balance decreased by 2 × STORAGE_AMOUNT.
   - Engine's implicit EVM address holds STORAGE_AMOUNT wNEAR
     (the stranded amount from Tx2).
   - Total NEAR delivered to router == STORAGE_AMOUNT (only Tx1's).
   - Invariant violated: 2 × STORAGE_AMOUNT charged, only 1 × delivered.
```

The integration test in `engine-tests/src/tests/xcc.rs` already exercises the XCC path and can be extended to submit two transactions in the same block and assert the wNEAR accounting invariant.
