### Title
SNS TreasuryManager `WithdrawRequest` Lacks Slippage Protection, Exposing DAO Treasury to Front-Running Loss - (File: `rs/sns/treasury_manager/treasury_manager.did`, `rs/sns/treasury_manager/src/lib.rs`, `rs/sns/governance/src/extensions.rs`)

### Summary

The SNS TreasuryManager framework provides a uniform abstraction over external custodians (DEXs such as KongSwap). The `WithdrawRequest` API type carries no slippage-protection field, and the governance validation layer actively rejects any arguments to the withdraw operation. The `Known Security Risks` section in the DID specification acknowledges slippage risk only for deposits. The same risk exists for withdrawals from DEX liquidity pools, but no mechanism exists in the IC production code to bound the minimum amount received. An unprivileged canister or user who is a liquidity provider in the same pool can observe the pending governance proposal and front-run the withdrawal, causing the SNS treasury to receive fewer tokens than expected with no on-chain recourse.

### Finding Description

The `TreasuryManager` trait in `rs/sns/treasury_manager/src/lib.rs` defines a uniform interface over DEX-backed external custodians:

```rust
pub trait TreasuryManager {
    fn withdraw(&mut self, request: WithdrawRequest) -> impl Future<Output = TreasuryManagerResult> + Send;
    ...
}
```

The `WithdrawRequest` struct contains only an optional map of destination accounts:

```rust
pub struct WithdrawRequest {
    pub withdraw_accounts: Option<BTreeMap<Principal, Account>>,
}
```

There is no `min_amount_out`, `slippage_tolerance`, or equivalent field. [1](#0-0) 

The DID specification explicitly acknowledges slippage risk, but **only for deposits**:

```
// Known Security Risks:
// ====================
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
``` [2](#0-1) 

The same risk applies to withdrawals from AMM liquidity pools: the amount of tokens returned depends on the pool ratio at execution time, which can differ from the ratio at proposal approval time.

The governance validation layer in `rs/sns/governance/src/extensions.rs` makes this worse by **actively rejecting any arguments** to the withdraw operation:

```rust
fn validate_withdraw_operation(...) -> BoxFuture<'_, Result<ValidatedOperationArg, String>> {
    Box::pin(async move {
        let ExtensionOperationArg { value } = arg;
        ValidatedWithdrawOperationArg::try_from(value)
            .map(ValidatedOperationArg::TreasuryManagerWithdraw)
    })
}
``` [3](#0-2) 

And `ValidatedWithdrawOperationArg::try_from` enforces empty arguments:

```rust
if original.value.is_some() {
    return Err("Withdraw operation does not accept arguments at this time".to_string());
}
``` [4](#0-3) 

This means it is structurally impossible for an SNS DAO to pass a `min_amount_out` or any slippage bound through the governance proposal, even if a TreasuryManager implementer wanted to support it.

The execution path in `execute_treasury_manager_withdraw` passes no slippage constraints to the extension canister: [5](#0-4) 

By contrast, the proposal rendering for `RegisterExtension` does warn about deposit slippage:

```
## WARNING
Some Decentralized Exchanges lack slippage protection during deposits. Consequently, 
deposited asset ratios may deviate from those specified in the proposal.
``` [6](#0-5) 

No equivalent warning exists for the `ExecuteExtensionOperation` withdraw proposal rendering. [7](#0-6) 

### Impact Explanation

When an SNS DAO executes a `TreasuryManagerWithdraw` governance proposal against a DEX-backed extension (e.g., the KongSwap adaptor), the extension canister withdraws liquidity from the DEX pool at the current pool ratio. If the pool ratio has moved adversarially between proposal approval and execution, the SNS treasury receives fewer tokens than expected. There is no on-chain bound on the minimum acceptable return. The loss is permanent and directly reduces the SNS treasury balance. The `BalanceBook` invariant `managed_assets[k] == managed_assets[k-1] + payers[k] - payees[k] - fee_collector[k]` can be violated in the attacker's favor. [8](#0-7) 

### Likelihood Explanation

IC governance proposal execution timing is public and predictable (proposals execute after a fixed voting period). Any canister or user who is a liquidity provider in the same DEX pool can observe the pending `TreasuryManagerWithdraw` proposal and manipulate the pool ratio before execution. The KongSwap adaptor is the reference implementation and is already integrated into the SNS framework. The attack requires no privileged access — only participation as a liquidity provider in the target pool, which is permissionless. [9](#0-8) 

### Recommendation

**Short term:** Add a slippage warning to the `ExecuteExtensionOperation` withdraw proposal rendering, analogous to the existing deposit warning in `validate_and_render_register_extension`. Expand the `Known Security Risks` section in `treasury_manager.did` to cover withdrawals. [2](#0-1) 

**Long term:** Add a `min_amounts_out` field to `WithdrawRequest` (mapping asset ledger canister ID to minimum acceptable amount) and remove the restriction in `ValidatedWithdrawOperationArg::try_from` that rejects all arguments. The governance validation layer should enforce that the `min_amounts_out` values are reasonable relative to the current cached balances. The `execute_treasury_manager_withdraw` function should pass these bounds to the extension canister, which must enforce them against the DEX response before completing the withdrawal. [1](#0-0) 

### Proof of Concept

1. An SNS DAO holds liquidity in a KongSwap pool via the KongSwap adaptor (TreasuryManager extension). The pool contains 1000 SNS tokens and 1000 ICP.
2. The DAO submits a `TreasuryManagerWithdraw` governance proposal. The proposal is public and its execution time is predictable.
3. An attacker (unprivileged LP in the same pool) observes the pending proposal. Before the proposal executes, the attacker performs a large swap in the pool, shifting the ratio to 500 SNS / 2000 ICP.
4. The governance canister calls `execute_treasury_manager_withdraw`, which calls `withdraw` on the KongSwap adaptor with an empty `WithdrawRequest` (no `min_amount_out`). [10](#0-9) 
5. The adaptor withdraws at the manipulated ratio. The SNS treasury receives significantly fewer SNS tokens than it deposited, with no on-chain check to reject the unfavorable outcome.
6. The attacker reverses their swap, profiting from the price impact at the DAO's expense.
7. The `TreasuryManagerResult` returned is `Ok(Balances {...})` — the operation succeeds with no error, and the loss is silently accepted. [11](#0-10)

/// Execute a treasury manager withdraw operation
async fn execute_treasury_manager_withdraw(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedWithdrawOperationArg,
) -> Result<(), GovernanceError> {
    let arg_blob = construct_treasury_manager_withdraw_payload(arg.original).map_err(|err| {
        GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!("Failed to construct treasury manager withdraw payload: {err}"),
        )
    })?;

    let balances = governance
        .env
        .call_canister(extension_canister_id, "withdraw", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.withdraw failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!(
                        "Error decoding TreasuryManager.withdraw response: {err:?}"
                    ),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.withdraw failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.withdraw succeeded with response: {:?}",
        balances
    );

    Ok(())
}
