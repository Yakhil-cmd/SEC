### Title
Incorrect `gas_used` Baseline in `delta_gas` Computation Causes User Gas Overcharge — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` adjustment — which ensures native (proving) resource costs are reflected in the final gas charge — is computed against a `gas_used` value that has **already had the EVM SSTORE refund subtracted**. This is inconsistent with the documented intent and with the value used to compute the EVM refund cap itself. The result is that users who both generate SSTORE refunds and have moderate native consumption are overcharged for gas.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` performs the following sequence:

1. Compute raw `gas_used` from remaining ergs after pubdata charge.
2. Compute `evm_refund` capped at `gas_used / 5` (EIP-3529).
3. **Subtract `evm_refund` from `gas_used`** → `gas_used` is now the post-refund value.
4. Apply `minimal_gas_used` floor.
5. Compute `delta_gas = (native_used / native_per_gas) as i64 - gas_used as i64`.
6. If `delta_gas > 0`, add it to `gas_used`. [1](#0-0) 

The critical inconsistency: the EVM refund cap (`max_refund = gas_used / 5`) is computed using the **pre-refund** `gas_used`, but `delta_gas` is computed using the **post-refund** `gas_used`. These are two uses of the same conceptual quantity — "gas consumed by execution" — but with different values.

The documentation in `docs/double_resource_accounting.md` states:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. [2](#0-1) 

The docs describe `gasUsed` as the gas consumed by EEs — the pre-refund value. The EVM refund is a separate incentive mechanism for clearing storage, independent of native resource consumption. Computing `delta_gas` against the post-refund `gas_used` conflates these two independent mechanisms.

**Concrete overcharge scenario:**

| Quantity | Value |
|---|---|
| `gas_limit` | 100,000 |
| EVM gas consumed (from ergs) | 80,000 |
| `evm_refund` (= 80,000/5) | 16,000 |
| `gas_used` after refund subtraction | 64,000 |
| `native_used / native_per_gas` | 70,000 |

With current code: `delta_gas = 70,000 − 64,000 = 6,000 > 0` → user pays 70,000 gas.

Correct behavior (delta_gas against pre-refund `gas_used`): `delta_gas = 70,000 − 80,000 = −10,000 ≤ 0` → no adjustment → user pays 64,000 gas.

**Overcharge: 6,000 gas units.** The native consumption (70,000 gas-equivalent) is already fully covered by the pre-refund gas consumption (80,000), so no delta adjustment is warranted.

The `// TODO: return delta_gas to gas_used?` comment at line 80 indicates the developers were aware of the asymmetry in the delta_gas logic but did not address the refund interaction. [3](#0-2) 

---

### Impact Explanation

Users who perform transactions that both:
- Generate EVM SSTORE refunds (e.g., clearing storage slots in a contract), **and**
- Have native resource consumption (in gas-equivalent units) that falls between `gas_used − evm_refund` and `gas_used`

…are overcharged for gas. The overcharge is at most `evm_refund` gas units, which is bounded by `gas_used / 5` (EIP-3529 cap). For a transaction with `gas_limit = 1,000,000` at 1 gwei gas price, the maximum overcharge is 200,000 gas × 1 gwei = 0.0002 ETH per transaction. The excess fee flows to the operator/coinbase. [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged L2 transaction sender can trigger this by deploying or calling a contract that:
1. Clears previously-set storage slots (generating SSTORE refunds via `refund_for_storage_write`), and
2. Performs enough computation to push native consumption into the vulnerable range. [5](#0-4) 

This is a realistic pattern for any contract that manages state (e.g., DEX, lending protocol, token contract). No privileged access, oracle manipulation, or governance attack is required.

---

### Recommendation

Capture the pre-refund `gas_used` before subtracting `evm_refund`, and use that value as the baseline for the `delta_gas` computation:

```rust
let gas_used_before_refund = gas_used; // save pre-refund value

let evm_refund = {
    let full_refund_ergs = system.io.get_refund_counter().ergs();
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};

gas_used -= evm_refund;
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

// Use pre-refund gas_used as the baseline, consistent with docs
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used_before_refund as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

This aligns with the documented formula `deltaGas := (nativeUsed / nativePerGas) - gasUsed` where `gasUsed` is the raw EE gas consumption, independent of the EVM refund mechanism.

---

### Proof of Concept

The overcharge is triggered when all of the following hold simultaneously:

```
gas_used_after_refund < native_used / native_per_gas ≤ gas_used_before_refund
```

i.e., `gas_used - evm_refund < native_used / native_per_gas ≤ gas_used`.

**Step-by-step:**

1. Deploy a contract that writes to N storage slots (setting them to non-zero), then in a second call clears them (setting to zero). The second call generates `4800 * N` gas of SSTORE refunds.
2. Ensure the second call also performs enough computation to push native consumption into the range `(gas_used − evm_refund, gas_used]`.
3. Submit the second call as an L2 transaction with sufficient `gas_limit`.
4. Observe that `gas_used` reported by the transaction result equals `native_used / native_per_gas` rather than `gas_used − evm_refund`, confirming the overcharge.

The `refund_for_storage_write` function in `basic_system/src/system_implementation/system/mod.rs` confirms that clearing a non-zero slot adds `4800 * ERGS_PER_GAS` to the refund counter, which is the mechanism that populates `evm_refund` in `compute_gas_refund`. [6](#0-5) [7](#0-6)
