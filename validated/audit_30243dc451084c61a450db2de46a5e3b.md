### Title
ERC-20 Mirror Over-Minting via Fee-on-Transfer NEP-141 Tokens in `receive_erc20_tokens` - (File: `engine/src/engine.rs`)

### Summary
`receive_erc20_tokens` blindly trusts the `amount` field supplied in the `ft_on_transfer` callback arguments to mint ERC-20 tokens, without verifying the actual NEP-141 balance change in Aurora Engine's account. For fee-on-transfer or deflationary NEP-141 tokens, the amount credited in the ERC-20 mirror exceeds the amount of NEP-141 tokens actually held, causing bridge insolvency and permanent fund freeze for the last withdrawers.

### Finding Description

When a NEP-141 token is bridged to Aurora Engine, the flow is:

1. A user calls `ft_transfer_call` on the NEP-141 contract, specifying Aurora Engine as the receiver.
2. The NEP-141 contract transfers tokens to Aurora Engine and invokes `ft_on_transfer` on Aurora Engine.
3. Aurora Engine's `ft_on_transfer` entry point (exposed in `engine/src/lib.rs`) dispatches to `contract_methods::connector::ft_on_transfer`.
4. That function reads `args.amount` from the callback JSON and calls `engine.receive_erc20_tokens()`.

Inside `receive_erc20_tokens` in `engine/src/engine.rs`:

```rust
pub fn receive_erc20_tokens<P: PromiseHandler>(
    &mut self,
    token: &AccountId,
    args: &FtOnTransferArgs,
    ...
) -> Result<Option<SubmitResult>, ContractError> {
    let amount = args.amount.as_u128();   // <-- trusts caller-supplied amount
    ...
    setup_receive_erc20_tokens_input(&recipient, amount),  // mints exactly `amount`
```

`args.amount` is the amount the *sender* declared, not the amount Aurora Engine *actually received*. For a fee-on-transfer NEP-141 token that deducts a percentage fee during transfer, Aurora Engine receives `amount - fee` NEP-141 tokens but mints `amount` ERC-20 tokens. No before/after balance check is performed.

The `ft_on_transfer` callback in `engine/src/contract_methods/connector.rs` also returns `args.amount.as_u128()` as the refund amount on error, but on success it returns `0`, meaning the full declared amount is accepted regardless of what was actually deposited.

### Impact Explanation

**Impact: Critical — Bridge Insolvency / Permanent Fund Freeze**

Each deposit of a fee-on-transfer NEP-141 token inflates the ERC-20 supply beyond the NEP-141 backing held by Aurora Engine. When users later call `withdrawToNear` (burning ERC-20 tokens), Aurora Engine issues `ft_transfer` promises for the full ERC-20 amount. The last users to exit will find Aurora Engine's NEP-141 balance insufficient to cover their withdrawal, permanently freezing their funds. The discrepancy grows with every deposit, making the insolvency cumulative and irreversible without admin intervention.

### Likelihood Explanation

**Likelihood: Medium**

The `deploy_erc20_token` function accepts any NEP-141 account ID without restricting token type. Any NEP-141 token with a transfer fee (a common pattern for deflationary or tax tokens) that gets bridged to Aurora triggers this bug. An attacker can deploy such a token themselves and bridge it, or the bug can be triggered inadvertently by users bridging legitimate fee-on-transfer tokens. No special privileges are required.

### Recommendation

In `receive_erc20_tokens`, record the NEP-141 balance of Aurora Engine before and after the `ft_transfer_call` completes, and mint only the actual difference. Since `ft_on_transfer` is a synchronous callback (the transfer has already occurred when it is called), the balance can be queried at callback time. Alternatively, return `args.amount - actual_received` from `ft_on_transfer` to refund the fee portion back to the sender, ensuring the minted ERC-20 amount equals the actual NEP-141 tokens retained.

### Proof of Concept

1. Deploy a NEP-141 token `fee_token.near` that deducts a 10% fee on every transfer (i.e., transferring 100 tokens results in the recipient receiving 90 tokens).
2. Call `deploy_erc20_token` on Aurora Engine to create an ERC-20 mirror for `fee_token.near`.
3. Call `ft_transfer_call` on `fee_token.near` with `receiver_id = aurora`, `amount = 1000`, `msg = <evm_address>`.
4. The NEP-141 contract transfers 900 tokens to Aurora Engine (after 10% fee) and calls `ft_on_transfer` with `amount = 1000`.
5. Aurora Engine's `receive_erc20_tokens` reads `args.amount = 1000` and mints 1000 ERC-20 tokens to `<evm_address>`.
6. Aurora Engine holds 900 NEP-141 tokens but has 1000 ERC-20 tokens in circulation.
7. Repeat: after 10 such deposits, Aurora holds 9000 NEP-141 but 10000 ERC-20 tokens exist.
8. The first 9 users to call `withdrawToNear` for 1000 tokens each succeed; the 10th user's withdrawal fails because Aurora Engine has 0 NEP-141 tokens remaining — their 1000 ERC-20 tokens are permanently frozen.

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
