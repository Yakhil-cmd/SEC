### Title
Precision Loss in `native_per_pubdata` Truncates to Zero, Allowing Pubdata Generation Without Native Resource Payment — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, the `native_per_pubdata` ratio is computed using floor (truncating) integer division. When `pubdata_price < native_price`, the result truncates to zero. This causes every subsequent native-resource check for pubdata to pass unconditionally, allowing any unprivileged transaction sender to generate pubdata up to the block limit without paying the corresponding native resource cost. The operator bears the L1 publication cost without compensation.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs floor (truncating) integer division. When `pubdata_price < native_price` — a plausible production scenario where L1 calldata is cheap relative to proving cost — the result is `0`.

By contrast, the analogous `native_per_gas` calculation on line 135 uses ceiling division to avoid this truncation:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

The same floor-division pattern is replicated in `api/src/helpers.rs` at line 427:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

When `native_per_pubdata == 0`, the downstream effects are:

1. **`create_resources_for_tx`** (`gas_helpers.rs` line 352): `intrinsic_pubdata_overhead = 0.saturating_mul(intrinsic_pubdata) = 0`, so no native resource is withheld for intrinsic pubdata.

2. **`get_resources_to_charge_for_pubdata`** (`gas_helpers.rs` line 430–431): `native = current_pubdata_spent.checked_mul(0) = 0`, so zero native resource is charged for any amount of pubdata generated during execution.

3. **`check_enough_resources_for_pubdata`** (`gas_helpers.rs` line 454): always returns `has_enough = true` because the required resource is zero.

4. **`compute_gas_refund`** (`refund_calculation.rs` line 69–72): `delta_gas = (native_used / 0)` is guarded by `native_per_gas == 0` check, but the pubdata native cost is already zero, so no gas adjustment occurs for pubdata.

The result is that pubdata is completely free from the native resource perspective, regardless of how much pubdata the transaction actually generates. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

---

### Impact Explanation

When `pubdata_price < native_price`, any unprivileged L2 transaction sender can:

- Submit transactions that write to many storage slots (generating pubdata) without paying any native resource for those writes.
- The native resource check for pubdata (`check_enough_resources_for_pubdata`) always passes, so execution is never reverted for pubdata cost reasons.
- The operator must still publish all generated pubdata to L1, bearing the full L1 calldata cost without receiving compensation through the native resource mechanism.
- Up to the block-level `pubdata_limit`, an attacker can fill every block with pubdata at zero native resource cost, causing systematic operator loss and potential throughput degradation.

This is a resource accounting bug: the operator's intended pubdata pricing is silently discarded due to integer truncation, and the economic invariant that "users pay for the pubdata they generate" is violated. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic production scenario. `native_price` reflects the cost of a single RISC-V proving cycle; `pubdata_price` reflects the cost of publishing one byte of pubdata to L1. During periods of low L1 gas prices (cheap calldata) combined with high proving demand, `pubdata_price` can be set below `native_price` by the operator without any intent to make pubdata free. For example, `native_price = 1000`, `pubdata_price = 500` yields `native_per_pubdata = 0`. The operator may intend pubdata to cost 0.5 native units per byte but due to truncation it costs zero. No privileged access or oracle manipulation is required — any user submitting a storage-writing transaction exploits this automatically. [8](#0-7) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, add a guard analogous to the `native_per_gas == 0` check: if `native_per_pubdata == 0` and `pubdata_price != 0`, treat it as a minimum of 1 to prevent unintended free pubdata. [1](#0-0) [3](#0-2) 

---

### Proof of Concept

Set block metadata with `native_price = 1000` and `pubdata_price = 500`. Submit a transaction that writes to 100 distinct storage slots (generating ~3200 bytes of pubdata). Observe:

1. `native_per_pubdata = 500 / 1000 = 0` (truncated).
2. `get_resources_to_charge_for_pubdata` returns `(3200, Resources::from_native(0))`.
3. `check_enough_resources_for_pubdata` returns `has_enough = true` regardless of remaining native resources.
4. The transaction succeeds and all pubdata is published to L1 at zero native resource cost to the sender.

Compare with `native_price = 1000`, `pubdata_price = 1001`: `native_per_pubdata = 1`, and the same transaction correctly charges `3200` native units for pubdata. The discrepancy demonstrates the precision loss. [9](#0-8) [10](#0-9)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))

///
pub fn check_enough_resources_for_pubdata<S: EthereumLikeTypes>(
