Audit Report

## Title
SNS Swap Participant Slot Exhaustion via Sybil Participation — (File: `rs/sns/swap/src/swap.rs`)

## Summary
The SNS Swap canister enforces a hard ceiling on direct participants via `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS`, checked against `self.buyers.len()` in `refresh_buyer_token_e8s`. Because there is no mechanism to remove a buyer from `self.buyers` during the `Open` lifecycle, an adversary controlling many sybil principals can permanently fill all participant slots, preventing legitimate users from joining. After the swap aborts due to insufficient participants, the adversary recovers all deposited ICP, making the attack nearly cost-free.

## Finding Description
In `refresh_buyer_token_e8s`, the participant ceiling is enforced at `rs/sns/swap/src/swap.rs` L1180–1197:

```rust
let num_direct_participants = self.buyers.len() as u64;
if (num_direct_participants + 1) * num_sns_neurons_per_basket
    > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(...)
}
``` [1](#0-0) 

The check counts every entry in `self.buyers` regardless of ICP amount. A search for `buyers.remove` in `swap.rs` returns no matches, confirming there is no removal path during the `Open` state. Once a principal is inserted into `self.buyers`, it occupies a slot permanently until the swap ends.

The only ICP-return path, `error_refund_icp`, is gated behind `Aborted` or `Committed` lifecycle: [2](#0-1) 

Even when it executes post-abort, it only transfers ICP back to the caller; it never removes the entry from `self.buyers`: [3](#0-2) 

The exploit path is:
1. Adversary generates `N = floor(MAX_NEURONS_FOR_DIRECT_PARTICIPANTS / basket_size)` sybil principals.
2. For each sybil principal `P_i`, adversary transfers `min_participant_icp_e8s` ICP to `subaccount(swap_canister, P_i)`.
3. Adversary calls `refresh_buyer_token_e8s(P_i)` for each `P_i`. Each call passes the ceiling check and inserts `P_i` into `self.buyers`.
4. `self.buyers.len() == N`. Any new legitimate principal hits the ceiling check and is rejected.
5. The swap cannot reach `min_participants` → aborts at `swap_due_timestamp_seconds`.
6. Adversary calls `sweep_icp` or `error_refund_icp` for each `P_i` → recovers all ICP minus transfer fees.

The behavior is explicitly tested in `rs/sns/swap/tests/swap.rs` L4986–5036, confirming the ceiling is enforced and new participants are rejected once it is reached: [4](#0-3) 

## Impact Explanation
This is a **High** severity finding. An adversary can unilaterally abort any SNS decentralization swap by exhausting participant slots, causing the SNS project to lose its swap entirely. This constitutes a significant SNS security impact with concrete, irreversible harm to the SNS project and its legitimate participants. It maps to the allowed impact: *"Significant SNS or infrastructure security impact with concrete user or protocol harm"* ($2,000–$10,000).

## Likelihood Explanation
Creating sybil principals on the IC is free. The only cost is locking `N × min_participant_icp_e8s` ICP for the swap duration (days to weeks), which is fully recoverable after abort minus transfer fees. For typical basket sizes of 3–5, `N` is in the tens of thousands. A motivated adversary (competitor, short-seller of the SNS token) has clear economic incentive. No special privileges are required; any unprivileged user can call `refresh_buyer_token_e8s`.

## Recommendation
Count only buyers with a non-zero `amount_icp_e8s` toward the participant ceiling, or remove a buyer from `self.buyers` and `BUYERS_LIST_INDEX` when their ICP balance drops to zero (e.g., after `sweep_icp` successfully transfers their ICP out). Alternatively, introduce a withdrawal mechanism during the `Open` phase that removes the buyer entry upon full withdrawal.

## Proof of Concept
The existing test `test_refresh_buyer_tokens_no_sns_neuron_baskets_available` in `rs/sns/swap/tests/swap.rs` L4986–5036 already demonstrates the ceiling enforcement. A minimal extension of this test would:
1. Fill `self.buyers` to the ceiling using sybil principals each contributing `min_participant_icp_e8s`.
2. Assert that a legitimate new participant is rejected with the "maximum number of direct participants" error.
3. Advance time past `swap_due_timestamp_seconds`, trigger `finalize_swap`, and assert the swap aborts due to `min_participants` not being met.
4. Call `sweep_icp` for each sybil principal and assert full ICP recovery. [5](#0-4)

fn test_refresh_buyer_tokens_no_sns_neuron_baskets_available() {
    let user1 = PrincipalId::new_user_test_id(1);
    let user2 = PrincipalId::new_user_test_id(2);
    let user3 = PrincipalId::new_user_test_id(3);
    let user4 = PrincipalId::new_user_test_id(4);

    let mut swap = SwapBuilder::new()
        .with_sns_governance_canister_id(SNS_GOVERNANCE_CANISTER_ID)
        .with_lifecycle(Open)
        .with_swap_start_due(Some(START_TIMESTAMP_SECONDS), Some(END_TIMESTAMP_SECONDS))
        .with_min_participants(1)
        .with_min_max_participant_icp(2 * E8, 40 * E8)
        .with_min_max_direct_participation(5 * E8, 100 * E8)
        .with_sns_tokens(100_000 * E8)
        // An extremely large basket size, so we can reach MAX_NEURONS_FOR_DIRECT_PARTICIPANTS with
        // a relatively small number of participants.
        .with_neuron_basket_count(33_000)
        .with_neurons_fund_participation()
        .build();

    let params = swap.params.unwrap();

    let amount_user1_0 = 5 * E8;
    let amount_user2_0 = 40 * E8;
    let amount_user3_0 = 40 * E8;
    let amount_user4_0 = 99 * E8 - (amount_user2_0 + amount_user3_0);

    // All tokens but one should be already bought up by users 2 to 4 --> 99 Tokens were bought
    buy_token_ok(&mut swap, &user2, &amount_user2_0, &amount_user2_0);
    buy_token_ok(&mut swap, &user3, &amount_user3_0, &amount_user3_0);
    buy_token_ok(&mut swap, &user4, &amount_user4_0, &amount_user4_0);

    // Make sure the 99 tokens were registered
    assert_eq!(
        swap.get_buyers_total().buyers_total,
        amount_user2_0 + amount_user3_0 + amount_user4_0
    );

    // Make sure that only an amount smaller than the minimum amount to be bought per user is available
    assert!(
        params.max_direct_participation_icp_e8s.unwrap() - swap.get_buyers_total().buyers_total
            < params.min_participant_icp_e8s
    );

    // No user that has not participated in the swap yet can buy this one token left
    buy_token_err(
        &mut swap,
        &user1,
        &amount_user1_0,
        "The swap has reached the maximum number of direct participants",
    );
