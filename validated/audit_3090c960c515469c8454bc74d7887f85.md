Based on my code analysis, this is a real vulnerability. Here is the full finding:

---

### Title
`ZKBatchDataKeeper.logs_storage` ArrayVec Overflow in Multiblock Proving Run Causes Valid Batch to Be Unprovable — (`basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`, `zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

In multiblock batch proving mode, each block's L2→L1 log hashes are appended to a fixed-capacity `ArrayVec<Bytes32, 16384>` in `ZKBatchDataKeeper`. The per-block log limit is also `16384`. Because `apply_to_array_vec` uses `ArrayVec::push` (which panics on overflow), any multiblock batch whose total log count across all blocks exceeds 16384 will panic in the proving run while the forward run (which uses a dynamically-sized `HistoryList`) succeeds. This creates a forward/proving consistency mismatch: a valid, successfully-executed batch becomes permanently unprovable.

---

### Finding Description

**`MAX_NUMBER_OF_LOGS` and `logs_storage` capacity are both 16384, but they operate at different scopes:**

`MAX_NUMBER_OF_LOGS` is defined as a **per-block** limit: [1](#0-0) 

`ZKBatchDataKeeper.logs_storage` is a **batch-level** accumulator with the same capacity: [2](#0-1) 

The per-block check in `check_for_block_limits` enforces `logs_used > MAX_NUMBER_OF_LOGS` (strictly greater), meaning a single block may legally contain exactly 16384 logs: [3](#0-2) 

In the multiblock proving path, after each block is processed, `apply_to_array_vec` is called to append that block's log hashes into the batch-level `ArrayVec`: [4](#0-3) 

`apply_to_array_vec` uses `array_vec.push(log.hash())` with no bounds check: [5](#0-4) 

`arrayvec::ArrayVec::push` panics when the vector is at capacity. There is no `try_push` or capacity guard anywhere in this path.

**The forward run is unaffected** because it uses `io.logs_storage` (a `HistoryList`-backed `LogsStorage` with dynamic allocation) and never calls `apply_to_array_vec`. The proving run is the only path that accumulates into the fixed-capacity `ArrayVec`. [6](#0-5) 

---

### Impact Explanation

A valid multiblock batch that was successfully executed in the forward run becomes permanently unprovable: the proving run panics on `ArrayVec::push` overflow. This is a **circuit/proving consistency break** — the batch cannot be finalized on L1. The impact is denial of batch finalization for any multiblock batch whose total L2→L1 log count exceeds 16384.

---

### Likelihood Explanation

Any unprivileged transaction sender can emit L2→L1 logs via the L1 messenger system hook. In a multiblock batch:
- Block 1 emits exactly 16384 logs (at the per-block limit) → `logs_storage` is full.
- Block 2 emits even 1 log → `apply_to_array_vec` panics.

The attacker does not need any privileged access. The only precondition is that the operator is running in multiblock batch mode, which is a supported production configuration (`ZKHeaderStructurePostTxOpProvingMultiblockBatch`). [7](#0-6) 

---

### Recommendation

1. **Enforce a batch-level log limit** that accounts for the number of blocks per batch: the invariant `MAX_NUMBER_OF_LOGS_PER_BLOCK * max_blocks_per_batch <= 16384` must hold, or the `ArrayVec` capacity must be sized to `MAX_NUMBER_OF_LOGS * max_blocks_per_batch`.
2. **Replace `push` with `try_push`** in `apply_to_array_vec` and propagate the error as a `BootloaderSubsystemError` rather than panicking.
3. **Add a batch-level log counter** in `ZKBatchDataKeeper` and reject blocks in `check_for_block_limits` when the running batch total would exceed the `ArrayVec` capacity.

---

### Proof of Concept

**Invariant assertion (compile-time):**
```rust
// This assertion fails for any multiblock batch with max_blocks > 1
const _: () = assert!(
    MAX_NUMBER_OF_LOGS as usize * MAX_BLOCKS_PER_BATCH <= 16384,
    "logs_storage ArrayVec too small for multiblock batch"
);
```

**Runtime path:**
1. Multiblock batch begins; `ZKBatchDataKeeper::new()` initializes `logs_storage: ArrayVec::new()` (capacity 16384, len 0). [8](#0-7) 
2. Block 1 executes with 16384 L2→L1 logs (within per-block limit). `apply_to_array_vec` appends 16384 hashes → `logs_storage.len() == 16384` (full). [5](#0-4) 
3. Block 2 executes with ≥1 L2→L1 log. `apply_to_array_vec` calls `array_vec.push(...)` on a full `ArrayVec` → **panic**.
4. Forward run never calls `apply_to_array_vec`; it completes successfully. [4](#0-3) 
5. Result: valid batch is unprovable; batch finalization on L1 is permanently blocked.
