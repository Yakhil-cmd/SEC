### Title
Users Can Lose Access to ckBTC Funds When Dynamic Minimum Withdrawal Amount Exceeds Remaining Balance - (`File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

---

### Summary

The ckBTC minter enforces a **dynamically computed** minimum withdrawal amount (`fee_based_retrieve_btc_min_amount`) that rises automatically with Bitcoin network fees. There is no exception for users who wish to withdraw their entire remaining balance. A user who makes a partial withdrawal can end up with a ckBTC balance that is permanently below the current minimum, making those funds inaccessible until either Bitcoin fees drop or the user deposits additional ckBTC to bring their total above the threshold.

---

### Finding Description

The ckBTC minter maintains two related fields in `CkBtcMinterState`:

- `retrieve_btc_min_amount`: the governance-configured floor
- `fee_based_retrieve_btc_min_amount`: a dynamically computed value that equals `retrieve_btc_min_amount` plus a fee-coverage buffer that scales with the current Bitcoin median fee rate [1](#0-0) 

The dynamic minimum is recomputed every time `estimate_fee_per_vbyte` runs (a periodic task): [2](#0-1) 

The formula adds up to `100,000` satoshi on top of the configured floor when fees are high: [3](#0-2) 

Both `retrieve_btc` and `retrieve_btc_with_approval` unconditionally reject any withdrawal below this dynamic minimum, with no exception for a user withdrawing their full remaining balance: [4](#0-3) [5](#0-4) 

The concrete scenario:

1. Bitcoin fees are low; `fee_based_retrieve_btc_min_amount` = 100,000 sat.
2. Alice deposits 200,000 sat of ckBTC.
3. Alice withdraws 150,000 sat — accepted, because 150,000 ≥ 100,000.
4. Alice now holds 50,000 sat of ckBTC on the ledger.
5. Bitcoin fees spike; `fee_based_retrieve_btc_min_amount` rises to 200,000 sat (confirmed by the test suite: at `343,000` millis/byte the default minimum jumps to 200,000 sat). [6](#0-5) 

6. Alice calls `retrieve_btc` with `amount = 50,000`. The minter returns `AmountTooLow(200,000)`.
7. Alice's 50,000 sat of ckBTC are inaccessible. She must either wait for fees to drop or deposit an additional ≥150,000 sat and pay the check fee again.

---

### Impact Explanation

Users can have ckBTC balances permanently stranded in the ledger with no way to retrieve them through the minter's normal withdrawal path. The funds are not lost (they remain on the ckBTC ledger), but they are inaccessible for an indefinite period determined by external Bitcoin network conditions outside the user's control. This is a direct availability impact on user funds — the chain-fusion bridge accepts the deposit but cannot guarantee the corresponding withdrawal path remains open.

---

### Likelihood Explanation

Bitcoin network fees are highly volatile. The minter's own test suite demonstrates that the fee-based minimum can jump from 100,000 sat to 200,000 sat in a single fee-percentile update. Any user who makes a partial withdrawal when fees are low and then faces a fee spike is at risk. This is not a contrived scenario — Bitcoin fee spikes are routine (e.g., during inscription/ordinal activity). The condition is reachable by any unprivileged user calling `retrieve_btc` or `retrieve_btc_with_approval` via ingress.

---

### Recommendation

Add an exception to the minimum-amount check: if the requested withdrawal amount equals the caller's entire ckBTC balance in the withdrawal subaccount, allow the withdrawal regardless of whether it meets the dynamic minimum. This mirrors the mitigation recommended in the original report and is safe because the minter already validates that the amount covers the ledger transfer fee separately (the `BadBurn` trap path).

Concretely, in both `retrieve_btc` and `retrieve_btc_with_approval`, after fetching the user's balance, allow the withdrawal if `args.amount == balance` even when `args.amount < min_retrieve_amount`.

---

### Proof of Concept

```
1. Configure minter with retrieve_btc_min_amount = 100_000 sat.
2. Set fee percentiles to 0 → fee_based_retrieve_btc_min_amount = 100_000.
3. User deposits 200_000 sat → receives 200_000 - check_fee ckBTC.
4. User calls retrieve_btc(amount = 150_000) → succeeds.
5. User's remaining balance = ~50_000 ckBTC sat.
6. Set fee percentiles to FeeRate::from_millis_per_byte(343_000).
7. Minter runs estimate_fee_per_vbyte → fee_based_retrieve_btc_min_amount = 200_000.
8. User calls retrieve_btc(amount = 50_000) → Err(AmountTooLow(200_000)).
9. User's 50_000 sat of ckBTC are inaccessible.
```

The fee-spike behavior is confirmed by the existing test: [7](#0-6) 

The unconditional minimum check with no full-balance exception is the root cause: [4](#0-3)

    fn fee_based_minimum_withdrawal_amount(&self, median_fee_rate: FeeRate) -> Satoshi {
        match self.network {
            Network::Mainnet | Network::Testnet => {
                const PER_REQUEST_RBF_BOUND: u64 = 22_100;
                const PER_REQUEST_VSIZE_BOUND: u64 = 221;
                const PER_REQUEST_MINTER_FEE_BOUND: u64 = 305;

                ((PER_REQUEST_RBF_BOUND
                    + median_fee_rate.fee_ceil(PER_REQUEST_VSIZE_BOUND)
                    + PER_REQUEST_MINTER_FEE_BOUND
                    + self.check_fee)
                    / 50_000) //TODO DEFI-2187: adjust increment of minimum withdrawal amount to be a multiple of retrieve_btc_min_amount/2
                    * 50_000
                    + self.retrieve_btc_min_amount
            }
            Network::Regtest => self.retrieve_btc_min_amount,
        }
    }

#[test]
fn test_min_retrieval_amount_default() {
    let ckbtc = CkBtcSetup::new();

    ckbtc.refresh_fee_percentiles();
    let retrieve_btc_min_amount = ckbtc.get_minter_info().retrieve_btc_min_amount;
    assert_eq!(retrieve_btc_min_amount, 100_000);

    // The numbers used in this test have been re-computed using a python script using integers.
    ckbtc.set_fee_percentiles(&vec![FeeRate::from_millis_per_byte(0); 100]);
    ckbtc.refresh_fee_percentiles();
    let retrieve_btc_min_amount = ckbtc.get_minter_info().retrieve_btc_min_amount;
    assert_eq!(retrieve_btc_min_amount, 100_000);

    ckbtc.set_fee_percentiles(&vec![FeeRate::from_millis_per_byte(116_000); 100]);
    ckbtc.refresh_fee_percentiles();
    let retrieve_btc_min_amount = ckbtc.get_minter_info().retrieve_btc_min_amount;
    assert_eq!(retrieve_btc_min_amount, 150_000);

    ckbtc.set_fee_percentiles(&vec![FeeRate::from_millis_per_byte(342_000); 100]);
    ckbtc.refresh_fee_percentiles();
    let retrieve_btc_min_amount = ckbtc.get_minter_info().retrieve_btc_min_amount;
    assert_eq!(retrieve_btc_min_amount, 150_000);

    ckbtc.set_fee_percentiles(&vec![FeeRate::from_millis_per_byte(343_000); 100]);
    ckbtc.refresh_fee_percentiles();
    let retrieve_btc_min_amount = ckbtc.get_minter_info().retrieve_btc_min_amount;
    assert_eq!(retrieve_btc_min_amount, 200_000);
}
