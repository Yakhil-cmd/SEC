Audit Report

## Title
Missing Slippage Protection in SNS Treasury Manager Withdraw Operation - (File: `rs/sns/governance/src/extensions.rs`)

## Summary

The SNS Treasury Manager's `withdraw` governance operation provides no mechanism for the DAO to specify minimum expected token amounts. `ValidatedWithdrawOperationArg::try_from` structurally rejects any non-empty argument, making it impossible to pass slippage bounds through a governance proposal. After a multi-day voting period, any DEX participant can manipulate the pool price so that `execute_treasury_manager_withdraw` returns `Ok(())` while the SNS treasury receives significantly fewer tokens than the DAO voted to recover.

## Finding Description

`WithdrawRequest` in `rs/sns/treasury_manager/treasury_manager.did` (L88–93) carries only an optional destination account map and no minimum-amount fields. The Rust validation in `ValidatedWithdrawOperationArg::try_from` (`rs/sns/governance/src/extensions.rs`, L1733–1746) actively rejects any non-empty argument with `"Withdraw operation does not accept arguments at this time"`, making it structurally impossible to encode slippage bounds in a governance proposal.

`execute_treasury_manager_withdraw` (`rs/sns/governance/src/extensions.rs`, L1612–1661) calls `withdraw` on the extension canister, decodes the returned `TreasuryManagerResult`, propagates any `Err` variant as a `GovernanceError`, but on `Ok(balances)` simply logs the `Balances` struct and returns `Ok(())`. The `treasury_owner` field of the returned `BalanceBook` — which contains the actual amounts credited to the SNS treasury — is never inspected or compared against any minimum.

The codebase acknowledges the analogous slippage risk for deposits in `treasury_manager.did` (L35–40) and in the proposal rendering code (`rs/sns/governance/src/proposal.rs`, L1542–1545), but neither warning nor enforcement is applied to the withdraw path. The deposit path and withdraw path are therefore treated asymmetrically: the deposit path at least surfaces a human-readable sandwich-attack warning to voters, while the withdraw path surfaces nothing.

Exploit flow:
1. SNS DAO approves a `TreasuryManagerWithdraw` proposal targeting a KongSwap pool.
2. During the multi-day voting period, an attacker executes large swaps on the same pool, skewing the position to single-sided (e.g., 2000 SNS + 0 ICP).
3. `execute_treasury_manager_withdraw` fires with an empty `WithdrawRequest`; the Treasury Manager withdraws at the manipulated price and credits 2000 SNS + 0 ICP to the SNS treasury.
4. The call returns `Ok(())` with no error; the attacker reverses their swap on the now-rebalanced pool, profiting at the DAO's expense.
5. The SNS treasury receives 0 ICP instead of the ~1000 ICP the DAO expected, with no on-chain revert.

No existing guard prevents this: the only check in `execute_treasury_manager_withdraw` is whether the canister call itself succeeded and whether the `TreasuryManagerResult` is `Ok` — not whether the amounts are acceptable.

## Impact Explanation

This matches the High bounty impact: "Significant SNS security impact with concrete user or protocol harm" and "Unauthorized access to / loss of canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The SNS treasury (DAO-controlled funds) can suffer a material, irreversible loss of one or both deposited token types with no on-chain error or revert. The loss is permanent because the attacker profits from the arbitrage and the governance proposal cannot be re-executed.

## Likelihood Explanation

The SNS governance voting period creates a mandatory multi-day window between proposal approval and execution. Any unprivileged DEX participant can execute large swaps against the pool during this window — no privileged access, no key compromise, and no consensus-level corruption is required. The attacker needs only sufficient capital to move the pool price, which is a well-known and well-resourced MEV strategy. The `TreasuryManager` API is explicitly designed to interact with DEXes such as KongSwap (confirmed by integration tests at `rs/nervous_system/integration_tests/tests/sns_extension_test.rs`, L399–420). The attack is repeatable for every future `TreasuryManagerWithdraw` proposal until the gap is closed.

## Recommendation

1. Add `min_token_amounts` fields to `WithdrawRequest` in `rs/sns/treasury_manager/treasury_manager.did` (analogous to how `DepositRequest` carries `allowances`):
   ```candid
   type WithdrawRequest = record {
     withdraw_accounts : opt vec record { principal; Account };
     min_token_amounts : opt vec record { principal; nat };
   };
   ```
2. Remove the blanket rejection in `ValidatedWithdrawOperationArg::try_from` (`rs/sns/governance/src/extensions.rs`, L1739–1743) and instead parse and validate minimum-amount fields from the governance proposal arg, storing them in `ValidatedWithdrawOperationArg`.
3. In `execute_treasury_manager_withdraw` (`rs/sns/governance/src/extensions.rs`, L1654–1660), after the `withdraw` call succeeds, iterate over `balances.asset_to_balances`, extract each `treasury_owner` balance, and compare against the minimums from the proposal. Return a `GovernanceError` if any minimum is not met.
4. Add a sandwich-attack warning to the withdraw proposal rendering (analogous to the deposit warning at `rs/sns/governance/src/proposal.rs`, L1542–1545).

## Proof of Concept

A deterministic PocketIC integration test can prove this without mainnet interaction:

1. Deploy an SNS with a KongSwap-backed Treasury Manager extension (as done in `rs/nervous_system/integration_tests/tests/sns_extension_test.rs`).
2. Execute a deposit proposal to seed the pool with 1000 SNS + 1000 ICP.
3. Simulate a large swap on the KongSwap mock that moves the pool to 2000 SNS + 0 ICP.
4. Submit and execute a `TreasuryManagerWithdraw` proposal with `operation_arg: Some(ExtensionOperationArg { value: None })`.
5. Assert that the proposal returns `Ok` (no failure reason) and that the SNS treasury's ICP balance is 0 — confirming the loss with no on-chain error.
