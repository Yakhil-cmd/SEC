### Title
Zero `max_gas_price` in `SubmitArgs` Bypasses All Silo Fees When `fixed_gas` Is Set — (`engine/src/engine.rs`)

---

### Summary

In silo mode with `fixed_gas` configured, an attacker who controls the NEAR-side relayer call (including a self-relaying EVM user) can pass `max_gas_price = Some(0)` in `SubmitArgs` to `submit_with_args`. This drives `effective_gas_price` to zero, making `prepaid_amount = 0`, so the sender's EVM balance is never debited and the relayer receives no reward. The transaction executes for free.

---

### Finding Description

**Entry point:** `submit_with_args` — a public NEAR contract method exposed at `engine/src/lib.rs:287`. [1](#0-0) 

It reads a `SubmitArgs` struct from input with no validation on `max_gas_price`: [2](#0-1) 

The value is passed directly into `charge_gas` at line 1100–1101: [3](#0-2) 

Inside `charge_gas`, the early-exit guard that would return a zero-cost result requires **all three** conditions: `max_fee_per_gas.is_zero() && fixed_gas.is_none() && block_base_fee_per_gas.is_zero()`. When `fixed_gas = Some(N)`, `fixed_gas.is_none()` is `false`, so the guard never fires even though the effective price will be zero: [4](#0-3) 

`block_base_fee_per_gas` is hardcoded to `U256::zero()` unconditionally: [5](#0-4) 

The `max_gas_price` cap is applied as a plain `min()` with no lower-bound check: [6](#0-5) 

With `max_gas_price = Some(0)`: `priority_fee_per_gas = 0.min(gas_price) = 0`, `effective_gas_price = 0 + 0 = 0`.

The prepaid amount is then: [7](#0-6) 

`fixed_gas_N * 0 = 0`. The `checked_sub(0)` on the sender's balance succeeds trivially — **no ETH is deducted**.

In `refund_unused_gas`, the zero-price early return means the relayer also receives nothing: [8](#0-7) 

---

### Impact Explanation

- The EVM sender's balance is never reduced (`prepaid_amount = 0`, `new_balance = old_balance`).
- The relayer receives zero reward.
- The silo operator collects zero fee revenue for the execution.
- Any number of transactions can be executed this way, draining the silo's economic model entirely. This constitutes **insolvency** of the fee/reward accounting.

---

### Likelihood Explanation

`submit_with_args` is callable by any NEAR account that holds a registered relayer function-call access key (or by anyone when no key manager is configured). A self-relaying EVM user — one who registers their own NEAR account as a relayer — satisfies this precondition without any privileged access or admin compromise. The `max_gas_price` field accepts `Some(0)` with no validation anywhere in the call path. The precondition (silo mode + `fixed_gas` set) is the normal operating state of a silo deployment.

---

### Recommendation

Add a lower-bound guard in `charge_gas`: when `fixed_gas.is_some()` and `effective_gas_price` resolves to zero while the transaction's `max_fee_per_gas` is non-zero, return `Err(GasPaymentError::MaxFeePerGasLessThanBaseFee)` (or a new dedicated error). Alternatively, treat `max_gas_price = Some(0)` as equivalent to `None` (no cap) when `fixed_gas` is active, or reject `SubmitArgs` where `max_gas_price == Some(0)` outright at the entry point.

---

### Proof of Concept

Call sequence on unmodified code:

```
1. set_silo_params(fixed_gas = 1_000_000, erc20_fallback_address = ...)
2. submit_with_args(SubmitArgs {
       tx_data: sign(TransactionLegacy { gas_price: 1000, gas_limit: 2_000_000, ... }),
       max_gas_price: Some(0),
       gas_token_address: None,
   })
```

Trace through `charge_gas`:
- `block_base_fee_per_gas = 0` (hardcoded)
- Early return skipped because `fixed_gas.is_none() == false`
- `priority_fee_per_gas = min(1000, 1000 − 0) = 1000`
- After cap: `min(0, 1000) = 0`
- `effective_gas_price = 0`
- `prepaid_amount = 1_000_000 * 0 = 0`
- `new_balance = sender_balance − 0 = sender_balance` ← **no deduction**

In `refund_unused_gas`: `effective_gas_price.is_zero()` → immediate `Ok(())`, relayer reward = 0.

Assert: `sender_balance_before == sender_balance_after` despite a successful EVM execution — the invariant `sender_balance_delta < 0` is broken.
