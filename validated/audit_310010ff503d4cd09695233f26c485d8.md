### Title
Silo Account Whitelist Bypassed via `call` and `deploy_code` Entry Points - (File: `engine/src/contract_methods/evm_transactions.rs`)

### Summary
In Silo mode, the `submit` entry point enforces the Account whitelist (restricting which NEAR accounts may interact with the EVM), but the `call` and `deploy_code` entry points omit this check entirely. Any unprivileged NEAR account can bypass the Silo whitelist by calling `call()` or `deploy_code()` directly, executing arbitrary EVM transactions without authorization.

---

### Finding Description

Aurora Silo is a restricted deployment mode that enforces a multi-layered whitelist. The Account whitelist is intended to ensure only approved NEAR accounts can submit EVM transactions. According to the wiki and the code, this enforcement is implemented inside `engine::submit()`.

The `submit` and `submit_with_args` entry points both route through `engine::submit()`: [1](#0-0) 

Inside `engine::submit()`, the silo Account whitelist check is performed against `predecessor_account_id` before any EVM execution proceeds.

However, the `call` entry point takes a completely different code path: [2](#0-1) 

After `require_running`, it immediately constructs an `Engine` and calls `engine.call_with_args()`. There is no silo Account whitelist check. The same omission exists in `deploy_code`: [3](#0-2) 

Both functions only call `require_running`, which checks the global pause flag, not the Silo whitelist: [4](#0-3) 

The Silo whitelist enforcement is documented as occurring specifically during `Engine::submit` internal validation, not in `call_with_args` or `deploy_code_with_input`. [5](#0-4) 

---

### Impact Explanation

**High — Whitelist bypass enabling unauthorized EVM execution.**

In a Silo deployment where the Account whitelist is enabled, the operator has explicitly restricted which NEAR accounts may interact with the EVM. By calling `call()` or `deploy_code()` instead of `submit()`, any NEAR account — regardless of whitelist status — can:

1. Execute arbitrary EVM calls against any deployed contract (e.g., ERC-20 `transfer`, DeFi interactions).
2. Deploy new EVM contracts into the Silo.
3. Interact with contracts that rely on the Silo's access control as a security boundary.

The EVM `msg.sender` for `call()` is derived from the attacker's own NEAR account ID via `predecessor_address()`, so the attacker acts as their own EVM address. This means they can move their own EVM-held assets and interact with any contract that does not have independent EVM-level access control — which in a Silo context is the expected design, since the whitelist is the intended boundary. This constitutes a **High** impact: unauthorized fund movement and unauthorized state changes within the Silo.

---

### Likelihood Explanation

**High.** The `call` and `deploy_code` NEAR contract methods are standard public entry points exposed by the Aurora Engine contract. Any NEAR account can call them with no special privileges. No key compromise, governance capture, or social engineering is required. The bypass is trivially reachable by any external actor.

---

### Recommendation

Add the Silo Account whitelist check to `call` and `deploy_code` in `engine/src/contract_methods/evm_transactions.rs`, mirroring the check performed inside `engine::submit()`. Specifically, before constructing the `Engine` and executing EVM logic, verify that `env.predecessor_account_id()` is present in the Account whitelist when that whitelist is enabled. A shared helper (analogous to `require_running`) should be introduced to avoid duplicating this logic across all EVM-executing entry points.

---

### Proof of Concept

1. Operator deploys Aurora Engine in Silo mode and enables the Account whitelist, adding only `trusted.near` to it.
2. Attacker controls `attacker.near`, which is **not** on the whitelist.
3. Attacker calls `submit()` → rejected: `engine::submit()` checks the Account whitelist and returns `ERR_NOT_ALLOWED`.
4. Attacker instead calls `call()` with a `CallArgs` payload targeting an ERC-20 contract's `transfer` function.
5. `call()` in `evm_transactions.rs` calls only `require_running` (passes, contract is not paused), then immediately executes the EVM call via `engine.call_with_args()`.
6. The EVM executes with `msg.sender = predecessor_address("attacker.near")`. The attacker successfully transfers their EVM-held tokens or interacts with any Silo contract — bypassing the Account whitelist entirely. [2](#0-1) [3](#0-2)

pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let tx_data = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let args = SubmitArgs {
            tx_data,
            ..Default::default()
        };
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}
