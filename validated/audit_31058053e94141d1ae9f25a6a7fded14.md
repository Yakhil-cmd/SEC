### Title
Missing Upper Bound on `transfer_fee` in `ManageLedgerParameters` Proposal Validation - (`File: rs/sns/governance/src/proposal.rs`)

### Summary

The `validate_and_render_manage_ledger_parameters` function in the SNS governance canister performs no upper-bound or relational validation on the `transfer_fee` field of a `ManageLedgerParameters` proposal. An SNS community that passes such a proposal with an arbitrarily large `transfer_fee` (up to `u64::MAX`) will render the SNS ledger permanently unusable: every transfer will fail because no user can hold enough tokens to pay the fee, and neuron operations (split, disburse, claim) that depend on `transaction_fee_e8s` will also break.

### Finding Description

`validate_and_render_manage_ledger_parameters` in `rs/sns/governance/src/proposal.rs` (lines 1761–1799) accepts any `transfer_fee: Option<u64>` value without checking:

1. That the new fee does not exceed the current `neuron_minimum_stake_e8s` (which would violate the invariant `neuron_minimum_stake_e8s > transaction_fee_e8s`).
2. That the new fee does not exceed the total token supply (which would make every transfer impossible).
3. That the new fee is non-zero (a zero fee is also semantically problematic but is a separate concern).

```rust
// rs/sns/governance/src/proposal.rs:1773-1776
if let Some(transfer_fee) = transfer_fee {
    render += &format!("# Set token transfer fee: {transfer_fee} token-quantums. \n",);
    change = true;
}
```

No bounds check is performed on `transfer_fee`. The value is accepted as-is and forwarded directly to the ledger upgrade path.

Upon successful proposal execution, `perform_manage_ledger_parameters` in `rs/sns/governance/src/governance.rs` (lines 3090–3212) upgrades the SNS ledger canister with the new fee via `UpgradeArgs`, and then writes the new fee back into `NervousSystemParameters.transaction_fee_e8s`:

```rust
// rs/sns/governance/src/governance.rs:3191-3194
if let Some(nervous_system_parameters) = self.proto.parameters.as_mut()
    && let Some(transfer_fee) = manage_ledger_parameters.transfer_fee
{
    nervous_system_parameters.transaction_fee_e8s = Some(transfer_fee);
}
```

This overwrites the governance-side `transaction_fee_e8s` with the unbounded value. The invariant `neuron_minimum_stake_e8s > transaction_fee_e8s` enforced at SNS init time (in `rs/sns/governance/src/types.rs` lines 610–617) is **not re-checked** when the fee is changed via `ManageLedgerParameters`.

The `ManageLedgerParameters` proposal type is classified as a **non-critical** action (id = 13) in `rs/sns/governance/src/types.rs`, meaning it passes with a simple majority rather than a supermajority, lowering the governance bar for exploitation.

### Impact Explanation

If `transfer_fee` is set to a value ≥ `neuron_minimum_stake_e8s` (or simply to any value larger than any user's token balance):

- **All ICRC-1 transfers fail** with `InsufficientFunds`, because `amount + fee > balance` for every account.
- **Neuron split fails**: `split.amount_e8s < min_stake + transaction_fee_e8s` check in `rs/sns/governance/src/governance.rs` line 1318 will always be true for any realistic split amount.
- **Neuron disburse fails**: same arithmetic check at line 1156 in the same file.
- **Neuron claiming fails**: the ledger transfer to stake a neuron will fail.
- **The SNS ledger and governance become permanently non-functional** — there is no recovery path because any corrective `ManageLedgerParameters` proposal to lower the fee also requires a governance vote, which itself requires neuron operations that are now broken.

This is a **ledger conservation / governance authorization bug** that permanently bricks an SNS instance.

### Likelihood Explanation

The `ManageLedgerParameters` action is a non-critical proposal type, requiring only a simple majority of voting power. Any SNS community member who can accumulate sufficient voting power (or coordinate with others) can submit and pass such a proposal. The attack is realistic for:

- A malicious actor who acquires a majority stake in a small or newly launched SNS.
- An accidental misconfiguration by a well-intentioned but mistaken SNS community.

The entry path is fully unprivileged from the IC protocol perspective: any principal with a neuron can submit a `ManageLedgerParameters` proposal via the SNS governance canister's public `manage_neuron` ingress endpoint.

### Recommendation

In `validate_and_render_manage_ledger_parameters` (`rs/sns/governance/src/proposal.rs`), add validation for `transfer_fee` that:

1. Rejects `transfer_fee == 0`.
2. Rejects `transfer_fee >= neuron_minimum_stake_e8s` (read from current `NervousSystemParameters`). This requires passing `current_parameters` into the function, analogous to how `validate_and_render_manage_nervous_system_parameters` already receives it.
3. Optionally, define an absolute upper-bound constant (e.g., `MAX_TRANSFER_FEE_E8S`) to prevent absurdly large values regardless of `neuron_minimum_stake_e8s`.

The existing `validate_neuron_minimum_stake_e8s` logic in `rs/sns/governance/src/types.rs` (lines 602–618) already encodes the correct invariant and should be reused or mirrored.

### Proof of Concept

1. An SNS is deployed with `neuron_minimum_stake_e8s = 100_000_000` (1 token) and `transaction_fee_e8s = 10_000`.

2. A neuron holder submits a `ManageLedgerParameters` proposal with `transfer_fee = u64::MAX` (18_446_744_073_709_551_615).

3. `validate_and_render_manage_ledger_parameters` accepts it — the only check is that at least one field is `Some`: [1](#0-0) 

4. The proposal passes governance vote (non-critical, simple majority).

5. `perform_manage_ledger_parameters` upgrades the ledger with `transfer_fee = u64::MAX` and writes it back to `NervousSystemParameters.transaction_fee_e8s`: [2](#0-1) 

6. From this point, every `icrc1_transfer` call fails with `InsufficientFunds` because no account can hold `u64::MAX` tokens. The SNS ledger is bricked.

7. Neuron split also fails because `split.amount_e8s < min_stake + u64::MAX` is always true (integer overflow aside, the check at line 1318 uses the now-corrupted `transaction_fee_e8s`): [3](#0-2) 

8. The SNS is permanently non-functional. No corrective governance action is possible because all neuron operations require ledger transfers that now fail.

The root cause is the absence of any upper-bound or relational validation on `transfer_fee` in: [4](#0-3) 

Contrast with the existing invariant enforcement at SNS init time, which correctly rejects `neuron_minimum_stake_e8s <= transaction_fee_e8s`: [5](#0-4) 

This invariant is not re-enforced when the fee is changed post-deployment via `ManageLedgerParameters`.

fn validate_and_render_manage_ledger_parameters(
    manage_ledger_parameters: &ManageLedgerParameters,
) -> Result<String, String> {
    let mut change = false;
    let mut render = "# Proposal to change ledger parameters:\n".to_string();
    let ManageLedgerParameters {
        transfer_fee,
        token_name,
        token_symbol,
        token_logo,
    } = manage_ledger_parameters;

    if let Some(transfer_fee) = transfer_fee {
        render += &format!("# Set token transfer fee: {transfer_fee} token-quantums. \n",);
        change = true;
    }
    if let Some(token_name) = token_name {
        ledger_validation::validate_token_name(token_name)?;
        render += &format!("# Set token name: {token_name}. \n",);
        change = true;
    }
    if let Some(token_symbol) = token_symbol {
        ledger_validation::validate_token_symbol(token_symbol)?;
        render += &format!("# Set token symbol: {token_symbol}. \n",);
        change = true;
    }
    if let Some(token_logo) = token_logo {
        ledger_validation::validate_token_logo(token_logo)?;
        render += &format!("# Set token logo: {token_logo}. \n",);
        change = true;
    }
    if !change {
        Err(String::from(
            "ManageLedgerParameters must change at least one value, all values are None",
        ))
    } else {
        Ok(render)
    }
}
