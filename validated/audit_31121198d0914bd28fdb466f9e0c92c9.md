### Title
SNS Swap Canister: Persistent `sweep_icp` Failure Permanently Blocks SNS Finalization - (`rs/sns/swap/src/swap.rs`)

### Summary

The SNS Swap canister's `finalize_swap` flow uses a sequential, all-or-nothing gate: if `sweep_icp` reports any `failure` or `invalid` count, `finalize_inner` immediately halts and returns without proceeding to neuron creation or SNS governance setup. A single buyer whose ICP transfer permanently fails (e.g., due to a ledger-side rejection that will never resolve) can block the entire SNS from completing finalization indefinitely. The analog to the Plaza Finance push-refund DoS is that a single participant can weaponize a permanently-failing ICP transfer to prevent all other participants from receiving their SNS neurons and prevent the SNS from transitioning to normal governance mode.

### Finding Description

In `rs/sns/swap/src/swap.rs`, `finalize_inner` calls `sweep_icp` and immediately checks for errors:

```rust
finalize_swap_response
    .set_sweep_icp_result(self.sweep_icp(now_fn, environment.icp_ledger()).await);
if finalize_swap_response.has_error_message() {
    return finalize_swap_response;
}
```

`set_sweep_icp_result` in `rs/sns/swap/src/types.rs` sets an error message if `failure > 0 || invalid > 0 || global_failures > 0`:

```rust
pub fn set_sweep_icp_result(&mut self, sweep_icp_result: SweepResult) {
    if !sweep_icp_result.is_successful_sweep() {
        self.set_error_message(
            "Transferring ICP did not complete fully, some transfers were invalid or failed. Halting swap finalization".to_string()
        );
    }
    ...
}
```

`is_successful_sweep` requires `failure == 0 && invalid == 0 && global_failures == 0`.

Inside `sweep_icp`, each buyer's transfer is attempted via `transfer_helper`. On failure, `transfer_start_timestamp_seconds` is reset to `0`, meaning the buyer is **retried on every subsequent `finalize` call** — but if the ICP ledger permanently rejects the transfer (e.g., `InsufficientFunds` because the buyer's subaccount was drained, or any other permanent ledger error), every retry will also fail, and `sweep_icp` will always return `failure >= 1`, permanently blocking finalization.

The `invalid` path (e.g., `AmountTooSmall`) is also permanent — the code comments explicitly state "This will require a manual intervention via an upgrade to correct."

The entire downstream finalization pipeline — `settle_neurons_fund_participation`, `create_sns_neuron_recipes`, `sweep_sns`, `claim_swap_neurons`, `set_sns_governance_to_normal_mode` — is gated behind the `sweep_icp` success check and will never execute.

### Impact Explanation

**For a COMMITTED swap:** All legitimate participants who contributed ICP and are waiting for their SNS neurons will never receive them. The SNS governance canister remains in `PreInitializationSwap` mode indefinitely (it cannot transition to normal mode because `set_sns_governance_to_normal_mode` is never reached). The SNS is effectively bricked — it cannot govern itself, and participants cannot claim their tokens.

**For an ABORTED swap:** If one buyer's ICP refund permanently fails, all other buyers' ICP refunds are also blocked (since `sweep_icp` halts finalization on any failure). Their ICP remains locked in the swap canister's subaccounts.

The `error_refund_icp` escape hatch exists but only works for buyers whose `transfer_success_timestamp_seconds == 0` after the normal sweep has already succeeded for them — it does not help the other buyers whose sweep was never attempted because finalization halted.

### Likelihood Explanation

This can be triggered by any single swap participant whose ICP subaccount balance becomes insufficient to cover the transfer fee between the time they participated and the time `sweep_icp` runs. This can happen:

1. **Accidentally**: If the ICP ledger fee changes, or if a participant's subaccount was somehow drained (e.g., via a ledger bug or race condition).
2. **Deliberately**: An attacker participates in a swap with the minimum amount, then somehow causes their subaccount balance to be zero at finalization time (e.g., by exploiting a race condition during the open phase). The `AmountTooSmall` path is marked `invalid` and is explicitly documented as requiring "manual intervention via an upgrade."

The `invalid` case is particularly dangerous because it is **permanent** — no amount of retrying `finalize` will resolve it without a canister upgrade.

### Recommendation

1. **Skip permanently-failing buyers**: In `is_successful_sweep`, treat `invalid` counts separately from `failure` counts. `invalid` entries should not block finalization since they are documented as unrecoverable without manual intervention.
2. **Decouple sweep failures from finalization gating**: Allow finalization to proceed past `sweep_icp` even if some transfers fail, recording failures for later retry, rather than halting the entire pipeline.
3. **Provide an administrative escape hatch**: Allow governance to mark specific buyer entries as permanently failed and skip them, unblocking finalization.

### Proof of Concept

The root cause is directly demonstrated by the existing test `test_finalization_halts_when_sweep_icp_fails` in `rs/sns/swap/tests/swap.rs`:

```rust
// A buyer with amount < DEFAULT_TRANSFER_FEE → AmountTooSmall → invalid
// A buyer whose ledger transfer fails → failure
// Result: finalize halts, no neurons created, no governance mode set
assert_eq!(result.error_message, Some(String::from(
    "Transferring ICP did not complete fully, some transfers were invalid or failed. Halting swap finalization"
)));
assert!(result.settle_neurons_fund_participation_result.is_none());
assert!(result.sweep_sns_result.is_none());
assert!(result.claim_neuron_result.is_none());
assert!(result.set_mode_call_result.is_none());
```

The `invalid` case is explicitly documented in `sweep_icp` as requiring "manual intervention via an upgrade to correct":

```rust
TransferResult::AmountTooSmall => {
    // ... In the case of a bug due to programmer error,
    // increment the invalid field. This will require a manual intervention
    // via an upgrade to correct
    sweep_result.invalid += 1;
}
```

The `is_successful_sweep` function treats both `failure` and `invalid` as blocking:

```rust
fn is_successful_sweep(&self) -> bool {
    *failure == 0 && *invalid == 0 && *global_failures == 0
}
```

This means a single `invalid` buyer — which by definition cannot be fixed by retrying — permanently prevents all other participants from receiving their SNS neurons.

#[tokio::test]
async fn test_finalization_halts_when_sweep_icp_fails() {
    // Step 1: Prepare the world

    // Setup the necessary buyers for the test
    let mut swap = Swap {
        lifecycle: Committed as i32,
        init: Some(init()),
        params: Some(params()),
        buyers: btreemap! {
            // This Buyer is `Invalid` because the amount committed is less than the
            // DEFAULT_TRANSFER_FEE of the ICP Ledger. This should never be possible
            // in production, but sweep_icp must handle this case.
            i2principal_id_string(1000) => BuyerState {
                icp: Some(TransferableAmount {
                    amount_e8s: DEFAULT_TRANSFER_FEE.get_e8s() - 1,
                    ..Default::default()
                }),
                has_created_neuron_recipes: Some(false),
            },
            // This buyer's state is valid, but a mock call to the ledger will fail the transfer,
            // which should result in a failure field increment.
            i2principal_id_string(1003) => BuyerState {
                icp: Some(TransferableAmount {
                    amount_e8s: 10 * E8,
                    ..Default::default()
                }),
                has_created_neuron_recipes: Some(false),
            },
        },
        ..Default::default()
    };

    let mut clients = CanisterClients {
        icp_ledger: SpyLedger::new(vec![
            // This mocked reply should produce a successful transfer in SweepResult
            LedgerReply::TransferFunds(Err(NervousSystemError::new_with_message(
                "Error when transferring funds",
            ))),
        ]),
        ..spy_clients()
    };

    // Step 2: Call sweep_icp
    let result = swap.finalize(now_fn, &mut clients).await;

    assert_eq!(
        result.sweep_icp_result,
        Some(SweepResult {
            success: 0,
            skipped: 0,
            failure: 1, // Single failed transfer
            invalid: 1, // Single invalid buyer
            global_failures: 0,
        })
    );

    assert_eq!(
        result.error_message,
        Some(String::from(
            "Transferring ICP did not complete fully, some transfers were invalid or failed. Halting swap finalization"
        ))
    );

    // Assert that all other fields are set to None because finalization was halted.
    assert!(result.settle_neurons_fund_participation_result.is_none());
    assert!(result.set_dapp_controllers_call_result.is_none());
    assert!(result.create_sns_neuron_recipes_result.is_none());
    assert!(result.sweep_sns_result.is_none());
    assert!(result.set_mode_call_result.is_none());
    assert!(result.claim_neuron_result.is_none());
