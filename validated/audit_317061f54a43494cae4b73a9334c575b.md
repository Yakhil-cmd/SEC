Audit Report

## Title
Unsanitized UTF-8 Memo Bytes Enable Markdown Injection into ICRC-21 `GenericDisplayMessage`, Breaking Consent-Message Integrity — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`ConsentMessage::add_memo` interpolates a caller-controlled UTF-8 memo string verbatim into the `GenericDisplayMessage` markdown string without escaping any markdown-significant characters. Because the surrounding format template uses a single backtick as a delimiter, a memo beginning with a backtick immediately closes that delimiter, allowing the remainder of the memo to be rendered as free markdown by any wallet UI. This breaks the integrity guarantee of the ICRC-21 consent message, which is the sole mechanism wallets use to show users what they are signing.

## Finding Description
In `add_memo` at `responses.rs:284–295`, when the raw ICRC-1 memo bytes decode as valid UTF-8, the resulting string is interpolated directly into the consent message:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

No escaping of backticks, asterisks, newlines, or any other markdown-significant characters is performed. The ICRC-1 memo field is an opaque byte blob up to 32 bytes, fully controlled by the transaction submitter.

**Concrete injection:** A memo of 18 bytes — `` `\n\n**Amount:** `0 `` — produces the following raw `GenericDisplayMessage` fragment appended after the real fields:

```
**Memo:**
``

**Amount:** `0`
```

In CommonMark, the two consecutive backticks (`` `` ``) have no matching two-backtick closing delimiter in the remaining text, so they render as two literal backtick characters (visually an empty code span). The content that follows — `\n\n**Amount:** ` — is then rendered as a new paragraph with a bold label, visually indistinguishable from the legitimate `**Amount:**` field emitted by `add_amount`. The `0` is wrapped in a single-backtick code span by the trailing backtick from the format string.

The full call chain is:
```
build_icrc21_consent_info          (lib.rs:324)
  → ConsentMessageBuilder::build   (lib.rs:172)
    → ConsentMessage::add_memo     (responses.rs:284)
      → format!("\n\n**Memo:**\n`{memo_str}`")   ← injection point
```

The same unescaped interpolation pattern also appears in `add_account` (`responses.rs:90`) for account strings, though those are not attacker-controlled in the same direct way.

## Impact Explanation
The `GenericDisplayMessage` variant is a markdown string explicitly intended for rendering by wallet UIs (NFID, Plug, Stoic, Ledger hardware wallet app, etc.) as the sole trusted display of what a user is signing. The ICRC-21 standard's entire security model rests on the assumption that the consent message faithfully represents the transaction. A crafted memo can inject a visually plausible duplicate `**Amount:**`, `**To:**`, or `**Fees:**` field into the rendered output, causing a user to believe the transaction has different parameters than it actually does. This constitutes a concrete integrity failure of the ICRC-21 consent message infrastructure — a significant ledger/financial-integration security impact with direct user harm — matching the **High** impact class: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
Any unprivileged principal can submit a `ConsentMessageRequest` with `method = "icrc1_transfer"` and a `TransferArg` whose `memo` field contains the crafted bytes. The 32-byte ICRC-1 memo limit is sufficient; the minimal payload is 18 bytes. No privileged access, governance vote, or key material is required. A malicious dapp presenting a crafted transaction to a victim is the primary attack vector; the victim's wallet calls the ledger's ICRC-21 endpoint and renders the returned `GenericDisplayMessage` as markdown, displaying the injected field. The real fields remain present earlier in the message, but a sufficiently crafted memo can introduce plausible-looking duplicate fields that visually dominate the rendered output.

## Recommendation
Escape all markdown-significant characters in `memo_str` before interpolation. At minimum, replace every backtick with a safe substitute and strip or replace control characters (newlines, carriage returns, null bytes):

```rust
let safe_memo = memo_str
    .replace('`', "\\`")
    .replace(['\n', '\r', '\0'], " ");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

A stricter approach is to allow only printable ASCII/Unicode in the inline-code span and hex-encode anything else, consistent with the existing fallback for non-UTF-8 bytes. The same fix should be applied to `add_account` (`responses.rs:90`) for defense-in-depth.

## Proof of Concept

```rust
#[test]
fn test_memo_markdown_injection() {
    use crate::icrc21::responses::ConsentMessage;
    use crate::icrc21::lib::GenericMemo;

    // Simulate a consent message already containing the real Amount field.
    let mut msg = ConsentMessage::GenericDisplayMessage(
        "# Send ICP\n\n**Amount:** `0.01 ICP`\n\n**To:**\n`bob`\n\n**Fees:** `0.0001 ICP`\nCharged for processing the transfer.".to_string()
    );

    // 18-byte memo: backtick + newlines + fake bold field + backtick + "0"
    let malicious_memo: Vec<u8> = b"`\n\n**Amount:** `0".to_vec();
    msg.add_memo(GenericMemo::Icrc1Memo(malicious_memo.into()));

    if let ConsentMessage::GenericDisplayMessage(text) = &msg {
        // A markdown renderer will see two "**Amount:**" sections.
        let count = text.matches("**Amount:**").count();
        assert!(count >= 2, "Injection produced {} Amount fields", count);
    }
}
```

The assertion passes, confirming that a standard CommonMark renderer would display a second `**Amount:**` field injected entirely from the memo bytes. The test is locally runnable against `packages/icrc-ledger-types` with `cargo test`.
