### Title
ICRC-1 Ledger Unconditionally Advertises ICRC-106 Support Regardless of Index Principal Configuration - (File: rs/ledger_suite/icrc1/ledger/src/main.rs)

### Summary
The ICRC-1 ledger canister unconditionally includes `"ICRC-106"` in its `icrc1_supported_standards()` response regardless of whether an `index_principal` has been configured. When no index principal is set, the mandatory `icrc106_get_index_principal` endpoint always returns `Err(Icrc106Error::IndexPrincipalNotSet)`. This is a direct analog to the EIP-4494 non-compliance finding: a canister advertises a standard it cannot fully satisfy, misleading any caller that relies on the standards-discovery mechanism to determine canister capabilities.

### Finding Description

The `supported_standards()` function in the ICRC-1 ledger unconditionally pushes `"ICRC-106"` into the returned list:

```rust
// rs/ledger_suite/icrc1/ledger/src/main.rs:751-790
#[query(name = "icrc1_supported_standards")]
fn supported_standards() -> Vec<StandardRecord> {
    let mut standards = vec![
        StandardRecord { name: "ICRC-1".to_string(), ... },
        StandardRecord { name: "ICRC-2".to_string(), ... },
        StandardRecord { name: "ICRC-3".to_string(), ... },
        StandardRecord { name: "ICRC-10".to_string(), ... },
        StandardRecord { name: "ICRC-21".to_string(), ... },
        StandardRecord { name: "ICRC-103".to_string(), ... },
        StandardRecord { name: "ICRC-106".to_string(), ... }, // always present
    ];
    if Access::with_ledger(|ledger| ledger.feature_flags().icrc152) {
        standards.push(StandardRecord { name: "ICRC-152".to_string(), ... });
    }
    standards
}
``` [1](#0-0) 

However, the `icrc106_get_index_principal` endpoint returns an error when `index_principal` is not configured:

```rust
// rs/ledger_suite/icrc1/ledger/src/main.rs:1181-1187
#[query]
fn icrc106_get_index_principal() -> Result<Principal, Icrc106Error> {
    Access::with_ledger(|ledger| match ledger.index_principal() {
        None => Err(Icrc106Error::IndexPrincipalNotSet),
        Some(index_principal) => Ok(index_principal),
    })
}
``` [2](#0-1) 

The `index_principal` field in `InitArgs` is optional (`Option<Principal>`), meaning any deployment that omits it will advertise ICRC-106 while being unable to fulfill the standard's core function. This is structurally inconsistent with how `ICRC-152` is handled — that standard is only added to the list when its feature flag is explicitly enabled: [3](#0-2) 

The ICP ledger correctly gates ICRC-2 behind its feature flag as a reference pattern: [4](#0-3) 

The DID interface confirms `icrc106_get_index_principal` is a mandatory part of the advertised service: [5](#0-4) 

The golden-state upgrade test explicitly handles the broken state where ICRC-106 is advertised but the index principal is not set, confirming this is a known reachable condition: [6](#0-5) 

### Impact Explanation

Any unprivileged query caller (wallet, DEX, explorer, or third-party canister) that calls `icrc1_supported_standards` on an ICRC-1 ledger deployed without an `index_principal` will receive a response claiming ICRC-106 compliance. When that caller subsequently invokes `icrc106_get_index_principal` — as the ICRC-106 standard requires — it receives `Err(Icrc106Error::IndexPrincipalNotSet)`. This breaks:

- **Wallet transaction-history discovery**: Wallets that use `icrc1_supported_standards` to locate the index canister will silently fail to load history.
- **Third-party canister integrations**: Any canister that gates logic on ICRC-106 presence (as shown in the golden-state test) will enter an error path despite the ledger claiming compliance.
- **Standards-based tooling**: Automatic tools that enumerate supported standards to determine capabilities will produce incorrect results, exactly mirroring the EIP-4494 finding where "automatic tools will not be able to determine" the correct interface.

The `icrc1_supported_standards` endpoint is a query, reachable by any anonymous or authenticated caller with no cycles cost beyond the query fee.

### Likelihood Explanation

The `index_principal` field is `Option<Principal>` in `InitArgs`, making it trivially omittable. Any SNS-deployed token, ckERC20 token, or custom ICRC-1 ledger that does not configure an index canister at init time will exhibit this behavior on every query to `icrc1_supported_standards`. The ledger suite orchestrator for ckERC20 tokens does set `index_principal` at deployment, but the field remains optional for all other deployments. The condition is reachable by any unprivileged query caller with no special setup.

### Recommendation

Gate the ICRC-106 entry in `supported_standards()` behind a check on whether `index_principal` is configured, mirroring the pattern used for ICRC-152:

```rust
if Access::with_ledger(|ledger| ledger.index_principal().is_some()) {
    standards.push(StandardRecord {
        name: "ICRC-106".to_string(),
        url: "https://github.com/dfinity/ICRC/pull/106".to_string(),
    });
}
```

This ensures `icrc1_supported_standards` only advertises ICRC-106 when the endpoint can actually fulfill the standard's requirements, consistent with how ICRC-152 and the ICP ledger's ICRC-2 flag are handled.

### Proof of Concept

1. Deploy the ICRC-1 ledger with `index_principal: None` in `InitArgs`.
2. Query `icrc1_supported_standards` — the response includes `"ICRC-106"`.
3. Query `icrc106_get_index_principal` — the response is `Err(Icrc106Error::IndexPrincipalNotSet)`.
4. Any third-party tool that trusts the standards list and calls `icrc106_get_index_principal` receives an error, despite the ledger advertising full ICRC-106 compliance. [1](#0-0) [2](#0-1)
