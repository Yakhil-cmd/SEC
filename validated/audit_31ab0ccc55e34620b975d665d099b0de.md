### Title
Unchecked `args.amount` in `ft_on_transfer` Enables Unbacked ERC-20 Minting — (`engine/src/contract_methods/connector.rs` / `engine/src/engine.rs`)

---

### Summary

`ft_on_transfer` in Aurora Engine trusts the caller-supplied `args.amount` field directly to mint ERC-20 tokens, without verifying that the corresponding NEP-141 tokens were actually received. Because `deploy_erc20_token` is permissionless, any NEAR account can register a malicious NEP-141 contract and then call `ft_on_transfer` directly on Aurora with an arbitrary `amount`, minting unbacked ERC-20 tokens and causing bridge insolvency.

---

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` reads the `amount` field from the JSON input args and passes it directly to `receive_erc20_tokens`:

```rust
// engine/src/contract_methods/connector.rs:80-90
let args: FtOnTransferArgs = read_json_args(&io)?;
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [1](#0-0) 

Inside `receive_erc20_tokens`, `args.amount` is used verbatim to mint ERC-20 tokens via an internal `call` to the ERC-20 contract's `mint` function:

```rust
// engine/src/engine.rs:803, 831
let amount = args.amount.as_u128();
...
setup_receive_erc20_tokens_input(&recipient, amount),
``` [2](#0-1) 

There is no balance-before/balance-after check to confirm that the Aurora contract actually received `amount` NEP-141 tokens from the predecessor. The only guard is that `predecessor_account_id` must have a registered ERC-20 via `get_erc20_from_nep141`: [3](#0-2) 

However, `deploy_erc20_token` has no access-control check — it contains only `require_running` and no `require_owner_only`:

```rust

``` [4](#0-3) 

This means any NEAR account can register itself as a NEP-141 token on Aurora, then call `ft_on_transfer` directly (from a malicious contract so that `predecessor_account_id` matches the registered token) with any `amount` in the JSON args.

---

### Impact Explanation

**Critical — Insolvency / Direct theft of user funds.**

An attacker mints ERC-20 tokens on Aurora that have no NEP-141 backing. These tokens are indistinguishable from legitimately bridged tokens within the Aurora EVM. The attacker can then call `withdrawToNear` (burning the ERC-20) to claim NEP-141 tokens that were legitimately deposited by other users, draining the bridge's NEP-141 reserves and leaving honest depositors unable to withdraw. [5](#0-4) 

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a NEAR contract (trivial, costs a small amount of NEAR).
2. Calling `deploy_erc20_token` on Aurora (permissionless).
3. Calling `ft_on_transfer` from the malicious contract with an inflated `amount`.

No privileged access, leaked keys, or governance capture is required. The entry path is fully reachable by any unprivileged NEAR account.

---

### Recommendation

In `receive_erc20_tokens`, record the ERC-20 token's total supply (or the recipient's balance) before the mint call and assert that the increase matches `args.amount`. More robustly, Aurora should maintain an internal ledger of NEP-141 tokens it has actually received (via a trusted callback from the NEP-141 contract's `ft_transfer_call` resolution) and reject `ft_on_transfer` calls whose claimed `amount` exceeds the actual inbound transfer recorded by the NEAR runtime.

Alternatively, restrict `deploy_erc20_token` to owner/admin only, so that only vetted NEP-141 contracts can be registered, reducing the attack surface to trusted counterparties.

---

### Proof of Concept

1. Attacker deploys `attacker_nep141.near` — a NEAR contract that does nothing except call `ft_on_transfer` on Aurora.
2. Attacker calls `deploy_erc20_token` on Aurora with `nep141 = "attacker_nep141.near"`. Aurora deploys an ERC-20 at address `0xABC...`.
3. Attacker's contract calls Aurora's `ft_on_transfer` with:
   ```json
   { "sender_id": "attacker.near", "amount": "1000000000000000000000", "msg": "<attacker_evm_address>" }
   ```
   Because `predecessor_account_id = "attacker_nep141.near"` is registered, `get_erc20_from_nep141` succeeds and `receive_erc20_tokens` mints `1e21` ERC-20 tokens to the attacker's EVM address — with zero NEP-141 tokens ever transferred to Aurora.
4. Attacker calls `withdrawToNear` on the ERC-20, burning the tokens and triggering Aurora to send NEP-141 tokens from its reserves, draining funds deposited by legitimate users. [6](#0-5) [7](#0-6)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
