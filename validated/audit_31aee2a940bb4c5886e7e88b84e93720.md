### Title
ERC-20 Tokens Permanently Burned Without Refund When `ft_transfer` Promise Fails in `ExitToNear` Precompile — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is absent (it is **not** in the default feature set), the `ExitToNear` precompile burns a user's ERC-20 tokens and schedules a NEP-141 `ft_transfer` promise, but attaches **no failure-recovery callback**. If that promise fails for any on-chain reason (e.g., recipient account not registered for storage), the burned ERC-20 tokens are irrecoverable — a permanent fund freeze with no remediation path.

---

### Finding Description

**Step 1 — ERC-20 tokens are burned before the promise executes.**

When a user calls `withdrawToNear` on a bridged ERC-20 contract, the contract burns the tokens and invokes the `ExitToNear` precompile. At this point the user's ERC-20 balance is already zero.

**Step 2 — The precompile constructs `callback_args` with `refund: None` when `error_refund` is disabled.** [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in the default build
    transfer_near: transfer_near_args,
};
```

**Step 3 — When both `refund` and `transfer_near` are `None`, no callback is attached at all.** [2](#0-1) 

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← bare promise, no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { … })
};
```

For every regular ERC-20 exit (non-wNEAR), `transfer_near` is `None`, so `callback_args` equals the default, and the promise is created **without** a callback.

**Step 4 — If the `ft_transfer` promise fails, `exit_to_near_precompile_callback` is never invoked.**

Even in the wNEAR-unwrap path where a callback *is* attached, the callback silently does nothing on failure when `error_refund` is disabled: [3](#0-2) 

```rust
} else if let Some(args) = args.refund {
    // refund path — only reachable when error_refund feature is on
    let refund_result = engine::refund_on_error(…)?;
    …
} else {
    None   // ← silent no-op; burned tokens are gone forever
};
```

**Step 5 — `error_refund` is not a default feature.** [4](#0-3) [5](#0-4) 

Neither `engine-precompiles` nor `engine` lists `error_refund` under `default = [...]`. The production WASM contract compiled without explicitly enabling this feature will exhibit the bug.

---

### Impact Explanation

**Permanent freezing / loss of funds (Critical).** A user who calls `exitToNear` for any bridged ERC-20 token loses their tokens permanently if the downstream NEP-141 `ft_transfer` call fails. There is no recovery path: the ERC-20 tokens are already burned, no callback re-mints them, and no administrative function can restore the balance. This matches the "permanent freezing of funds" impact class.

---

### Likelihood Explanation

**Medium.** The `ft_transfer` call fails whenever:
- The recipient NEAR account has not registered storage for the NEP-141 token (common for new accounts or tokens).
- The NEP-141 contract is paused.
- The recipient account does not exist on NEAR.

Any user who specifies a recipient that has not pre-registered storage will trigger the failure path. This is a realistic, user-reachable condition requiring no privileged access.

---

### Recommendation

1. **Enable `error_refund` in the production build** so that `refund_call_args` is populated and the callback can re-mint burned ERC-20 tokens on failure.
2. Alternatively, **always attach a refund callback** regardless of the feature flag, making the safe path unconditional.
3. Consider validating recipient storage registration before burning ERC-20 tokens, or performing the burn only after the NEP-141 transfer succeeds (commit-after-confirm pattern).

---

### Proof of Concept

```
1. Alice holds 100 USDC.e (bridged ERC-20) on Aurora.
2. Alice calls USDC.e.withdrawToNear("bob.near", 100).
3. USDC.e contract burns 100 tokens from Alice's EVM balance.
4. ExitToNear precompile is invoked; it schedules:
       ft_transfer(receiver_id="bob.near", amount=100) on usdc.near
   with NO callback (error_refund disabled, transfer_near=None).
5. bob.near has never called storage_deposit on usdc.near.
6. usdc.near rejects ft_transfer → promise fails.
7. No callback fires. Alice's ERC-20 balance is 0.
   usdc.near balance of aurora is unchanged (tokens not transferred).
8. Alice has permanently lost 100 USDC.e with no recourse.
```
