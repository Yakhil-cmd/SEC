### Title
Excess Cycles Consumed Without Corresponding Work in Bitcoin Checker Canister — (`File: rs/bitcoin/checker/src/fetch.rs`)

---

### Summary

The Bitcoin checker canister's `try_fetch_tx` function calls `cycles_accept(cycle_cost)` to deduct cycles before each HTTPS outcall. When the remaining available cycles are less than `cycle_cost`, the IC system API `msg_cycles_accept` silently accepts **all** remaining cycles (not just `cycle_cost`), then the function returns `NotEnoughCycles`. Those consumed cycles are never refunded, mirroring the Solidity pattern where excess value above the required fee is locked in the contract.

---

### Finding Description

In `rs/bitcoin/checker/src/fetch.rs`, the `try_fetch_tx` method (part of the `FetchEnv` trait) performs the following check before each HTTPS outcall:

```rust
let cycle_cost = get_tx_cycle_cost(max_response_bytes, num_subnet_nodes);
if self.cycles_accept(cycle_cost) < cycle_cost {
    TryFetchResult::NotEnoughCycles
} else {
    TryFetchResult::ToFetch(self.fetch_tx(guard, provider, txid, max_response_bytes))
}
``` [1](#0-0) 

`cycles_accept` delegates to `ic_cdk::api::msg_cycles_accept(cycle_cost)`: [2](#0-1) 

The IC system call `msg_cycles_accept(max_amount)` accepts `min(available, max_amount)` cycles. When `available < cycle_cost`, it accepts **all** available cycles and returns `available`. The condition `available < cycle_cost` is true, so `NotEnoughCycles` is returned — but the cycles have already been irrevocably consumed by the canister. No HTTP outcall was made, yet the caller's cycles are gone.

This is confirmed explicitly by the test suite comment: [3](#0-2) 

```rust
// Check available cycles: we deduct all remaining cycles even when they are not enough
assert_eq!(env.cycles_available(), 0);
```

And again for the `Pending` case (transaction with multiple inputs, cycles run out mid-way): [4](#0-3) 

```rust
// Check remaining cycle: we deduct all remaining cycles when they are not enough
assert_eq!(env.cycles_available(), 0);
```

The canister's public documentation explicitly promises the opposite: [5](#0-4) 

> "The actual cycle cost may be well less than `CHECK_TRANSACTION_CYCLES_REQUIRED`, and unspent cycles will be refunded back to the caller, minus a `CHECK_TRANSACTION_CYCLES_SERVICE_FEE`."

The same flaw exists in `check_fetched`, which calls `try_fetch_tx` for each input transaction of the checked transaction.

---

### Impact Explanation

A caller who sends the minimum required `CHECK_TRANSACTION_CYCLES_REQUIRED` (40 billion cycles) for a transaction with many inputs may have their cycles exhausted mid-processing. For each subsequent input fetch attempt where remaining cycles < `cycle_cost`, **all remaining cycles are consumed** by the canister even though no HTTP outcall is performed and no useful work is done. The caller receives a `Pending` response and must retry with fresh cycles, having lost the unconsumed remainder from the first call. This is a **cycles/resource accounting bug**: the canister gains cycles it did not earn through actual work, and callers lose funds beyond the documented service fee. [6](#0-5) 

---

### Likelihood Explanation

Any canister that calls `check_transaction` on a Bitcoin transaction with multiple inputs (common for real-world transactions) and attaches exactly `CHECK_TRANSACTION_CYCLES_REQUIRED` (40B) cycles is affected. The `get_tx_cycle_cost` per outcall scales with subnet size and response buffer, so for a 13-node subnet with `INITIAL_MAX_RESPONSE_BYTES` (4 KiB), a single outcall costs roughly 3–4 billion cycles. A transaction with 10+ inputs would exhaust 40B cycles before all inputs are fetched, triggering the bug on every retry attempt. This is a realistic scenario for ckBTC minter calls to the checker canister. [7](#0-6) 

---

### Recommendation

Replace the greedy `cycles_accept(cycle_cost)` pattern with a guard that checks availability **before** accepting:

```rust
let available = self.cycles_available();
if available < cycle_cost {
    // Do NOT accept cycles — they will be automatically refunded
    return TryFetchResult::NotEnoughCycles;
}
self.cycles_accept(cycle_cost); // Accept only the exact cost
TryFetchResult::ToFetch(self.fetch_tx(guard, provider, txid, max_response_bytes))
```

This ensures that when there are insufficient cycles for an outcall, no cycles are consumed and the full remainder is refunded to the caller by the IC runtime.

---

### Proof of Concept

**Entry path**: Any unprivileged canister calls `check_transaction` (an `#[ic_cdk::update]` endpoint) with `CHECK_TRANSACTION_CYCLES_REQUIRED` cycles attached, passing a Bitcoin transaction ID whose transaction has ≥ 5 inputs.

**Execution trace**:
1. `check_transaction` → `check_transaction_with`: accepts `CHECK_TRANSACTION_CYCLES_SERVICE_FEE` (100M). [8](#0-7) 
2. Remaining cycles check passes (≥ 39.9B remaining). [9](#0-8) 
3. `check_transaction_inputs` → `try_fetch_tx`: accepts `cycle_cost` (~4B) for the main transaction fetch. Succeeds.
4. `check_fetched` iterates over inputs, calling `try_fetch_tx` for each. Each call accepts ~4B cycles.
5. After ~9 successful outcalls (~36B consumed), remaining ≈ 3.9B < `cycle_cost` (4B).
6. `cycles_accept(4B)` accepts all 3.9B remaining, returns 3.9B < 4B → `NotEnoughCycles`. [10](#0-9) 
7. Caller receives `Pending`, has lost all 40B cycles. Must retry with another 40B.

The test mock confirms `cycles_available() == 0` after this scenario, proving the excess cycles are consumed rather than refunded. [11](#0-10)
