### Title
Wrong Rounding Direction in `native_per_pubdata` Calculation Causes Systematic Underpayment for Pubdata - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed using truncating (floor) integer division via `wrapping_div`, when it should use ceiling division. This causes every transaction to pay slightly fewer native resources per pubdata byte than the protocol requires, systematically favoring the caller over the protocol. The same inconsistency exists in `api/src/helpers.rs`. Notably, the analogous `native_per_gas` calculation correctly uses `div_ceil`, making the rounding asymmetry a clear bug.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the `native_per_pubdata` ratio is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs truncating (floor) integer division. This value is then used in two critical places:

1. **Intrinsic pubdata overhead** — `native_per_pubdata * intrinsic_pubdata` is subtracted from the user's native budget in `create_resources_for_tx`.
2. **Post-execution pubdata charge** — `current_pubdata_spent * native_per_pubdata` is charged in `get_resources_to_charge_for_pubdata`.

Both computations use the rounded-down rate, so the user pays less native resource for pubdata than the true cost.

By contrast, `native_per_gas` is computed with ceiling division:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

The comment in `api/src/helpers.rs` even documents the asymmetry explicitly:

```
// native_per_gas = ceil(gas_price / native_price)   ← correct, favors protocol
// native_per_pubdata = pubdata_price / native_price  ← floor, favors caller
```

The same floor-division bug is present in `api/src/helpers.rs` line 427:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug — wrong rounding direction in fee calculation.

When `pubdata_price % native_price != 0`, the user is charged `floor(pubdata_price / native_price)` native units per pubdata byte instead of `ceil(pubdata_price / native_price)`. The shortfall per byte is up to `native_price - 1` native units.

For a transaction generating `N` pubdata bytes, the underpayment is up to `N * (native_price - 1)` native units. The operator/protocol absorbs this difference on every such transaction. Because native resources represent proving cost (RISC-V cycles), this is a direct, systematic underpayment of proving fees by every transaction sender.

**Impact: Low** — the per-transaction shortfall is bounded by `(native_price - 1)` native units per pubdata byte, which is small in absolute terms. However, it is systematic and affects every L2 transaction.

---

### Likelihood Explanation

**Likelihood: High** — this triggers on every L2 transaction where `pubdata_price % native_price != 0`. The operator sets both `pubdata_price` and `native_price` independently; there is no enforcement that they are perfectly divisible. In practice, arbitrary operator-set values will almost always produce a non-zero remainder.

The attacker-controlled entry path is simply submitting any standard L2 transaction. No special privileges are required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// In validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427.

---

### Proof of Concept

**Setup:** `pubdata_price = 10`, `native_price = 3`.

- **Current (floor):** `native_per_pubdata = 10 / 3 = 3`
- **Correct (ceil):** `native_per_pubdata = ceil(10 / 3) = 4`

For a transaction generating 1000 pubdata bytes:
- **Current charge:** `1000 * 3 = 3000` native units
- **Correct charge:** `1000 * 4 = 4000` native units
- **Underpayment:** 1000 native units per transaction
