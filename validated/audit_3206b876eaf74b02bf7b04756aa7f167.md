Audit Report

## Title
Global Pending Withdrawal Queue Exhaustion Enables DoS of All ckETH/ckERC20 Withdrawals — (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`)

## Summary

The ckETH minter enforces a global cap of `MAX_PENDING = 100` on the `pending_withdrawal_requests` queue. An unprivileged attacker controlling 100 IC principals, each holding the minimum ckETH withdrawal amount, can fill this queue and cause every subsequent `withdraw_eth` and `withdraw_erc20` call from any legitimate user to trap via `ic_cdk::trap`, blocking all ckETH and ckERC20 withdrawals for as long as the attacker replenishes consumed slots.

## Finding Description

**Global cap check in `Guard::new`** (`rs/ethereum/cketh/minter/src/guard/mod.rs`, lines 50–54):

```rust
fn new(principal: Principal) -> Result<Self, GuardError> {
    mutate_state(|s| {
        if PR::pending_requests_count(s) >= MAX_PENDING {   // global cap = 100
            return Err(GuardError::TooManyPendingRequests);
        }
```

`pending_requests_count` for `PendingWithdrawalRequests` delegates to `state.eth_transactions.withdrawal_requests_len()`, which returns the length of the `pending_withdrawal_requests` VecDeque — the FIFO queue of requests not yet assigned an Ethereum nonce.

**Guard lifecycle mismatch**: The guard is acquired at the start of `withdraw_eth`/`withdraw_erc20`, held across the async `burn_from` call, and dropped when the function returns. On success, `record_withdrawal_request` pushes the request into `pending_withdrawal_requests`. When the function returns, the guard drops and the principal is removed from `pending_withdrawal_principals` — but the request **remains** in the queue, permanently occupying one of the 100 slots until the minter's timer processes it.

**Trap on rejection** (`rs/ethereum/cketh/minter/src/main.rs`, lines 274–278 and 401–405):

```rust
let _guard = retrieve_withdraw_guard(caller).unwrap_or_else(|e| {
    ic_cdk::trap(format!(
        "Failed retrieving guard for principal {caller}: {e:?}"
    ))
});
```

`ic_cdk::trap` rejects the call with a system error rather than returning a graceful error variant, preventing callers from distinguishing a transient capacity limit from a protocol error.

**Slow drain**: The timer-driven processing loop dequeues at most `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5` requests per cycle (`rs/ethereum/cketh/minter/src/withdraw.rs`, line 39 and lines 249–253), so the attacker only needs to submit 5 new requests per processing cycle to maintain the queue at capacity.

Both `withdraw_eth` and `withdraw_erc20` share the same guard and the same `MAX_PENDING = 100` limit, so flooding via either endpoint blocks both.

## Impact Explanation

An attacker who fills all 100 slots causes every `withdraw_eth` and `withdraw_erc20` call from any legitimate user to trap. Users holding ckETH or ckERC20 tokens cannot convert them back to ETH/ERC20 for as long as the attacker maintains the queue at capacity. This constitutes a **platform-level DoS on ckETH/ckERC20 withdrawals**, a significant Chain Fusion / ck-token financial integration impact with concrete user harm. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS not based on raw volumetric DDoS** and **Significant Chain Fusion, ck-token security impact with concrete user or protocol harm**.

## Likelihood Explanation

- **Capital required**: 100 principals × 0.005 ETH minimum = 0.5 ETH ≈ $1,000.
- **Ongoing cost**: Only Ethereum gas fees (cents per transaction) to cycle ETH → ckETH → ETH.
- **Maintenance**: The minter drains 5 slots per processing cycle; the attacker submits 5 new requests per cycle to maintain the DoS.
- **No privileged access required**: Any unprivileged IC principal can call `withdraw_eth`.
- **Attacker recovers capital**: Burned ckETH is redeemed as ETH on Ethereum, so the attack is economically sustainable indefinitely.

Likelihood is **medium-high** — the capital barrier is low, the attack is sustainable, and no special privileges are needed.

## Recommendation

1. **Raise `MAX_PENDING`** to a value reflecting realistic legitimate demand (e.g., 1,000–10,000), consistent with the ckBTC minter's `MAX_CONCURRENT_PENDING_REQUESTS = 5000`.
2. **Add per-principal pending-request limits** so a single principal or small set cannot monopolize the queue.
3. **Return a graceful error** instead of `ic_cdk::trap` for `TooManyPendingRequests`, so callers can distinguish a transient capacity limit from a protocol error and retry.
4. **Consider a minimum fee or deposit** for queuing a withdrawal request to raise the economic cost of flooding.

## Proof of Concept

1. Attacker creates 100 IC principals, each funded with ≥ 0.005 ETH worth of ckETH.
2. Each principal calls `icrc2_approve` on the ckETH ledger, approving the minter for the minimum amount.
3. Each principal calls `withdraw_eth` with `amount = minimum_withdrawal_amount` and any valid Ethereum recipient address.
4. All 100 calls succeed; each burns ckETH and enqueues a `WithdrawalRequest`. After all 100 complete, `pending_withdrawal_requests.len() == 100 == MAX_PENDING`.
5. Any legitimate user now calling `withdraw_eth` or `withdraw_erc20` receives a trap: `"Failed retrieving guard for principal X: TooManyPendingRequests"`.
6. The minter processes 5 requests per cycle. The attacker monitors the queue (via `get_minter_info` or event logs) and submits 5 new requests each cycle, maintaining the DoS.
7. The attacker receives ETH on Ethereum for each processed request, recovering capital minus gas fees.

A deterministic integration test using PocketIC can reproduce this by: (a) submitting 100 withdrawal requests from distinct principals, (b) asserting `withdrawal_requests_len() == 100`, and (c) asserting that a 101st `withdraw_eth` call traps with `TooManyPendingRequests`.
