### Title
Unbounded `move_to_block` Loop in Hashchain Causes Permanent Freezing of All Aurora Engine Functions - (File: engine-hashchain/src/hashchain.rs)

### Summary

Every hashchain-enabled Aurora Engine method (`submit`, `ft_on_transfer`, `deploy_erc20_token`, `register_relayer`, etc.) calls `load_hashchain`, which invokes `Hashchain::move_to_block` if the stored hashchain block height lags behind the current NEAR block height. `move_to_block` contains an unbounded `while` loop that iterates once per skipped block, performing a keccak hash computation per iteration. If the gap between the stored hashchain height and the current block height is large enough, this loop exhausts the NEAR transaction gas limit (300 Tgas), causing every hashchain-wrapped call to revert permanently — freezing all user funds in the engine.

### Finding Description

**Root cause — `move_to_block` unbounded loop:**

In `engine-hashchain/src/hashchain.rs` lines 75–85, `move_to_block` iterates one step per block:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(...);
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
```

Each iteration calls `compute_block_hashchain` (a keccak over ~100+ bytes) and `clear_txs` (resets a Bloom filter and a Merkle tree). The number of iterations equals `current_block_height - stored_hashchain_height`.

**How the gap is created:**

`load_hashchain` in `engine/src/hashchain.rs` lines 88–96 calls `move_to_block` unconditionally whenever the current block height exceeds the stored one:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
```

The hashchain's `current_block_height` is only advanced when a hashchain-wrapped transaction is successfully processed. If no Aurora Engine transaction is submitted for a long period (e.g., the engine is paused, or simply idle), the stored height falls far behind the NEAR chain's block height. NEAR produces approximately 1 block/second, so ~24 hours of inactivity creates a gap of ~86,400 blocks.

**Attacker-controlled entry path:**

An unprivileged attacker can deliberately cause or exploit this gap:

1. The engine is paused (e.g., for an upgrade via `start_hashchain`, which requires `require_paused`). During the pause, no hashchain-wrapped calls succeed, so the stored height freezes while NEAR blocks keep advancing.
2. After the pause ends (or after any sufficiently long idle period), the attacker submits any valid Aurora transaction (e.g., a `submit` call with a minimal ETH transfer). This triggers `load_hashchain` → `move_to_block` with a gap of N blocks.
3. If N is large enough (empirically, tens of thousands of blocks at ~keccak cost per iteration), the NEAR 300 Tgas limit is exceeded and the call reverts.
4. Because the hashchain state is never updated on revert, every subsequent call also hits the same gap and also reverts. The engine is permanently bricked.

Even without a pause, a natural idle period (e.g., low-traffic deployment, weekend downtime, or a network incident) of sufficient length creates the same condition. The attacker does not need to cause the idle period — they only need to be the first to call after it.

**All affected entry points** wrap through `with_hashchain` or `with_logs_hashchain`:
- `submit` / `submit_with_args` (EVM transaction submission)
- `ft_on_transfer` (bridge deposits)
- `deploy_erc20_token`
- `register_relayer`
- `ft_transfer_call`, `withdraw`, and other connector methods

### Impact Explanation

**Permanent freezing of funds (Critical).**

Once the gap is large enough that `move_to_block` exhausts 300 Tgas, every hashchain-wrapped method reverts. This includes `submit` (the primary EVM execution path) and `ft_on_transfer` (the bridge deposit path). Users cannot withdraw ETH or ERC-20 tokens, cannot execute EVM transactions, and cannot bridge assets out. All funds held in the Aurora Engine contract are permanently inaccessible through normal protocol flows.

The only recovery path would be an emergency admin action (`start_hashchain`) — but `start_hashchain` itself calls `move_to_block` with the same gap (lines 447–448 of `engine/src/contract_methods/admin.rs`), so it also reverts if the gap is too large.

### Likelihood Explanation

**High.** NEAR produces ~1 block/second. A gap of ~50,000–100,000 blocks (14–28 hours of inactivity) is sufficient to exhaust 300 Tgas given the per-iteration cost of keccak + Bloom + Merkle operations. Aurora Engine deployments (especially Silo instances) routinely experience low-traffic periods. The engine is also explicitly designed to be paused for upgrades, and the pause duration can easily exceed the threshold. No special attacker capability is required — any user submitting the first transaction after a long idle period triggers the condition.

### Recommendation

Replace the per-block iteration in `move_to_block` with a direct jump: since empty blocks contribute a deterministic hash (empty Merkle root + zero Bloom), the intermediate hashes for a run of N empty blocks can be computed in O(log N) using repeated squaring / exponentiation of the hash function, or simply by storing only the last non-empty block's hash and computing the gap lazily. At minimum, cap the number of iterations per call and allow the gap to be closed across multiple transactions, or store the block height directly without iterating through empty blocks.

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled.
2. Record the stored `current_block_height` H₀ from `get_latest_hashchain`.
3. Wait (or fast-forward in a sandbox) until NEAR block height reaches H₀ + N where N ≈ 50,000.
4. Submit any valid Aurora transaction (e.g., a minimal ETH transfer via `submit`).
5. Observe: `load_hashchain` → `move_to_block(H₀ + N)` iterates N times → NEAR gas exhausted → transaction reverts.
6. Repeat step 4: every subsequent call also reverts because the stored height was never updated.
7. All user funds are now permanently frozen.
