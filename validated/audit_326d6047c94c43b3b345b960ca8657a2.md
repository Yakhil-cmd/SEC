Audit Report

## Title
Unbounded `environment_variables` in `CanisterSettingsArgs` Enables Pre-Validation Resource Exhaustion via `create_canister`/`update_settings` - (File: rs/execution_environment/src/canister_settings.rs)

## Summary
`CanisterSettingsArgs.environment_variables` is typed as `Option<Vec<EnvironmentVariable>>` with no deserialization-time bound, unlike `controllers` which uses `BoundedControllers` (max 10 enforced at deserialization). An unprivileged caller can submit a `create_canister` or `update_settings` ingress message containing up to ~100,000–200,000 `EnvironmentVariable` entries within the 2 MB ingress limit, forcing the replica to deserialize the full list and perform an O(n log n) `BTreeMap` insertion pass in native Rust — outside Wasm instruction metering — before the call is rejected. This can be repeated at the ingress rate limit to exhaust replica CPU on every node of a subnet.

## Finding Description
In `rs/types/management_canister_types/src/lib.rs`, `CanisterSettingsArgs` declares `environment_variables: Option<Vec<EnvironmentVariable>>` (line 2382) as a plain `Vec`, while `controllers: Option<BoundedControllers>` (line 2372) uses `BoundedVec<10, UNBOUNDED, UNBOUNDED, PrincipalId>` whose custom `Deserialize` implementation rejects sequences exceeding 10 elements immediately during Candid decoding (bounded_vec.rs lines 108–113).

In `rs/execution_environment/src/canister_settings.rs`, the `TryFrom<CanisterSettingsArgs> for CanisterSettings` conversion (lines 177–190) calls `env_vars.len()` and `env_vars.into_iter().map(...).collect::<BTreeMap<String, String>>()` directly on the raw, unbounded `Vec` before any count check. The count limit (`MAX_ENVIRONMENT_VARIABLES = 32`, execution_environment.rs line 216) is only enforced later inside `validate_environment_variables()` in `canister_manager.rs` (lines 290–315), which is called from `validate_and_update_canister_settings()`. The sequence is:

1. Candid deserialization allocates heap memory for all `n` entries (no bound).
2. `TryFrom` iterates all `n` entries and inserts each `(name, value)` pair into a `BTreeMap` — O(n log n) in native Rust, outside Wasm instruction metering.
3. Only then is `len > 32` checked and the call rejected.

Existing guards are insufficient: the 2 MB ingress message size limit is the only upstream constraint, permitting ~100,000–200,000 minimal entries per message.

## Impact Explanation
This matches the allowed ICP bounty impact: **Application/platform-level DoS, subnet availability impact not based on raw volumetric DDoS** (High, $2,000–$10,000). An attacker can saturate replica CPU on every node of a subnet by repeatedly submitting maximally-packed `update_settings` or `create_canister` messages. The O(n log n) BTreeMap work occurs in native Rust outside Wasm metering, creating a cost asymmetry: the attacker pays only ingress submission cost while the replica performs unbounded computation. Sustained at the ingress rate limit, this degrades or halts execution of legitimate canister calls across the subnet.

## Likelihood Explanation
`create_canister` and `update_settings` are publicly callable management canister methods reachable by any principal with an identity. No cycles, canister deployment, or privileged access are required. The malicious payload is trivially constructable with any Candid encoder. The attack is fully automatable and repeatable at the ingress rate limit.

## Recommendation
Apply the same `BoundedVec` pattern already used for `controllers`. Define a bounded type in `rs/types/management_canister_types/src/lib.rs`:

```rust
const MAX_ALLOWED_ENV_VARS_COUNT: usize = 32;
pub type BoundedEnvironmentVariables =
    BoundedVec<MAX_ALLOWED_ENV_VARS_COUNT, UNBOUNDED, UNBOUNDED, EnvironmentVariable>;
```

Replace `Option<Vec<EnvironmentVariable>>` with `Option<BoundedEnvironmentVariables>` in `CanisterSettingsArgs`. This enforces the limit at Candid deserialization time, before any heap allocation or `BTreeMap` construction, consistent with how `controllers` and `allowed_viewers` are already protected.

## Proof of Concept
Using any Candid encoder (e.g., `didc`, `ic-agent`, or the Python `ic` library):

```python
# Construct ~100,000 minimal EnvironmentVariable records
env_vars = [{"name": f"K{i}", "value": ""} for i in range(100_000)]
settings = {"environment_variables": env_vars}

# Send as update_settings to any canister the attacker controls
agent.update_call(
    canister_id="aaaaa-aa",
    method="update_settings",
    arg=candid_encode({"canister_id": attacker_canister_id, "settings": settings}),
)
# Replica deserializes 100,000 entries, builds a BTreeMap, then rejects:
# "Too many environment variables: 100000 (max: 32)"
# Repeat at high frequency to exhaust replica CPU.
```

A deterministic integration test can confirm the behavior by submitting an `UpdateSettingsArgs` with >32 `EnvironmentVariable` entries via `test.subnet_message(Method::UpdateSettings, ...)` and verifying the call is not rejected at deserialization time (i.e., it reaches the `validate_environment_variables` check), demonstrating the missing early bound.
