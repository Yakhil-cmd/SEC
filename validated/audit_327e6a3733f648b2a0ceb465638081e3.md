### Title
Incomplete `PrecompileFlags` Hardcoded List Prevents Pausing of `CrossContractCall` and Other Aurora-Specific Precompiles - (`engine/src/pausables.rs`)

### Summary
`PrecompileFlags` in `engine/src/pausables.rs` is a hardcoded bitmask covering only `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM`. Four other Aurora-specific precompiles registered in the live precompile set — `CrossContractCall`, `PredecessorAccount`, `PrepaidGas`, and `PromiseResult` — have no corresponding flag and can never be paused via `pause_precompiles`, regardless of what an authorized caller submits. This is a direct analog to the RocketPool inconsistent-blacklist bug: a hardcoded list that cannot be extended when new components are added, leaving critical fund-moving precompiles permanently outside the emergency-pause perimeter.

### Finding Description

`PrecompileFlags` defines only two bits:

```rust
pub struct PrecompileFlags: u32 {
    const EXIT_TO_NEAR     = 0b01;
    const EXIT_TO_ETHEREUM = 0b10;
}
``` [1](#0-0) 

`from_address` maps only those two addresses:

```rust
pub fn from_address(address: &Address) -> Option<Self> {
    Some(if address == &exit_to_ethereum::ADDRESS {
        Self::EXIT_TO_ETHEREUM
    } else if address == &exit_to_near::ADDRESS {
        Self::EXIT_TO_NEAR
    } else {
        return None;   // all other precompiles fall here
    })
}
``` [2](#0-1) 

`apply_pause_flags_to_precompiles` uses `is_paused_by_address` to build the runtime `paused_precompiles` set:

```rust
fn apply_pause_flags_to_precompiles(...) -> Precompiles<...> {
    Precompiles {
        paused_precompiles: precompiles
            .all_precompiles
            .keys()
            .filter(|address| pause_flags.is_paused_by_address(address))
            .copied()
            .collect(),
        ...
    }
}
``` [3](#0-2) 

Because `is_paused_by_address` returns `None` (and therefore `false`) for every address that is not `exit_to_near::ADDRESS` or `exit_to_ethereum::ADDRESS`, the following precompiles are **permanently excluded** from the `paused_precompiles` set no matter what bitmask an admin submits:

- `CrossContractCall` (`cross_contract_call::ADDRESS`)
- `PredecessorAccount` (`predecessor_account::ADDRESS`)
- `PrepaidGas` (`prepaid_gas::ADDRESS`)
- `PromiseResult` (`promise_result::ADDRESS`) [4](#0-3) 

All six Aurora-specific precompiles are unconditionally inserted into `all_precompiles` on every execution, but only two of them can ever appear in `paused_precompiles`. [5](#0-4) 

### Impact Explanation

`CrossContractCall` is the most security-critical of the unpaused precompiles. It orchestrates wNEAR movements into XCC sub-accounts and schedules arbitrary NEAR cross-contract calls on behalf of EVM callers. [6](#0-5) 

If a vulnerability is discovered in `CrossContractCall` that allows an attacker to drain wNEAR from XCC sub-accounts or redirect scheduled promises, the authorized admin **cannot** invoke `pause_precompiles` to stop the attack. The bitmask submitted will be silently truncated by `from_bits_truncate` and the filter will produce an empty set for that precompile's address. The only remediation path is a full contract upgrade, which requires staging, a delay block window, and deployment — during which the exploit remains live.

**Impact: High — Temporary (potentially permanent) freezing of user funds / theft of unclaimed yield held in XCC sub-accounts**, conditional on a bug in `CrossContractCall` that cannot be mitigated by the existing pause mechanism.

### Likelihood Explanation

The hardcoded list is a structural gap, not a latent bug. Any future bug found in `CrossContractCall` immediately becomes unmitigatable via the pause mechanism. The `CrossContractCall` precompile is complex (it manages wNEAR accounting, promise scheduling, and sub-account routing), making it a realistic target for future vulnerabilities. The gap is permanent until a code upgrade is deployed.

### Recommendation

1. Add flag bits for every Aurora-specific precompile that can affect funds or NEAR-side state:
   ```rust
   pub struct PrecompileFlags: u32 {
       const EXIT_TO_NEAR          = 0b000001;
       const EXIT_TO_ETHEREUM      = 0b000010;
       const CROSS_CONTRACT_CALL   = 0b000100;
       const PREDECESSOR_ACCOUNT   = 0b001000;
       const PREPAID_GAS           = 0b010000;
       const PROMISE_RESULT        = 0b100000;
   }
   ```
2. Extend `from_address` to cover all registered Aurora-specific precompile addresses.
3. Add a compile-time or test-time assertion that every address inserted in `with_generic_precompiles` has a corresponding `PrecompileFlags` bit, so the list cannot silently fall out of sync when new precompiles are added.

### Proof of Concept

1. Admin calls `pause_precompiles` with `paused_mask = 0xFFFFFFFF` (all bits set).
2. `PrecompileFlags::from_bits_truncate(0xFFFFFFFF)` produces `0b11` (only the two defined bits survive).
3. `apply_pause_flags_to_precompiles` iterates all precompile addresses; `is_paused_by_address` returns `false` for `cross_contract_call::ADDRESS`.
4. `paused_precompiles` does not contain `cross_contract_call::ADDRESS`.
5. An EVM transaction calling `CrossContractCall` proceeds normally — the precompile is not paused.
6. `get_paused_precompiles` returns `0b11`, giving the admin false confidence that all desired precompiles are paused. [7](#0-6) [8](#0-7)
