Audit Report

## Title
Unpermissioned `update_balance` Allows Any Caller to Exhaust Per-Account Concurrent Guard and DoS ckBTC Minting - (File: rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs)

## Summary
The `update_balance` endpoint accepts an optional `owner` field with no check that the caller matches the specified owner. The concurrency guard (`balance_update_guard`) is keyed on the target account rather than the actual ingress caller, and is held across multiple async inter-canister calls. An attacker can continuously occupy the guard slot for any victim's account, causing the victim's own calls to return `AlreadyProcessing`. With 100 distinct accounts, an attacker can fill all `MAX_CONCURRENT` slots and halt ckBTC minting globally.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` at lines 164–168, the target account is resolved as:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

There is no assertion that `caller == args.owner`. The guard is keyed on `caller_account` (the victim's account). The only existing access check (line 159) validates `s.mode.is_deposit_available_for(&caller)` — it checks the ingress caller's whitelist status, not whether the caller is authorized to act on behalf of `args.owner`. The comment at lines 156–158 explicitly acknowledges this design: "we also want those principals to mint at any desired address. Therefore, the check below is on 'caller'."

The guard is held across at least one `get_utxos` async call (line 175) and up to `MAX_CHECK_TRANSACTION_RETRY` (10) calls to the Bitcoin checker canister (line 400). During this entire window, any call to `update_balance` for the same victim account — including the victim's own — hits the occupied guard and returns `AlreadyProcessing` (mapped from `GuardError::AlreadyProcessing` at line 111).

In `rs/bitcoin/ckbtc/minter/src/guard.rs`, `MAX_CONCURRENT = 100` (line 6). The guard's `new()` method (lines 45–60) enforces one slot per account and a global cap of 100. An attacker with 100 distinct principals or subaccounts can fill all slots with arbitrary victim accounts, causing every new `update_balance` call to return `TooManyConcurrentRequests` (mapped to `TemporarilyUnavailable` at line 113 of `update_balance.rs`), halting ckBTC minting for all users.

## Impact Explanation
This is a **High** severity application-level DoS. An unprivileged attacker can prevent any specific victim from converting confirmed BTC deposits into ckBTC (per-victim DoS), or halt ckBTC minting entirely for all users by filling all 100 guard slots (global DoS). No funds are stolen, but users cannot access their ckBTC for the duration of the attack. This matches the allowed impact: "Application/platform-level DoS... not based on raw volumetric DDoS" and "Significant Chain Fusion, ck-token... security impact with concrete user or protocol harm."

## Likelihood Explanation
The endpoint is publicly reachable by any IC principal with no special privileges. The `owner` field is explicitly documented as optional and intended for third-party callers. The attack requires only IC ingress fees (no BTC or ckBTC). The attacker can generate 100 distinct principals trivially. The guard window spans multiple async inter-canister calls, giving a large time window per attack iteration. The attack is repeatable indefinitely.

## Recommendation
Key the guard on the **caller** principal rather than the target account, so an attacker can only block their own slot:

```rust
let guard_account = Account {
    owner: caller,
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(guard_account)?;
```

Alternatively, require `caller == args.owner` when `owner` is explicitly set, rejecting unauthorized cross-account calls. The minting itself should still use `caller_account` (with the specified owner) for the actual UTXO lookup and ledger mint.

## Proof of Concept
1. Victim `V` deposits BTC and waits for confirmations.
2. Attacker `A` (any IC principal) calls `update_balance({ owner: Some(V), subaccount: null })`. This acquires `balance_update_guard` for `{owner: V, subaccount: null}` and holds it across the async `get_utxos` call.
3. Victim `V` calls `update_balance({})`. Guard is occupied → returns `AlreadyProcessing`.
4. Attacker re-submits step 2 in a loop. Victim cannot mint.
5. For global DoS: attacker uses 100 different principals targeting 100 different victim accounts, filling all `MAX_CONCURRENT = 100` slots. All users receive `TemporarilyUnavailable("too many concurrent requests")`.

A deterministic PocketIC integration test can reproduce this: initialize the minter, spawn two async tasks — one holding the guard for a victim account via a spoofed `owner` field, one attempting the victim's own call — and assert the victim's call returns `AlreadyProcessing`.     pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
