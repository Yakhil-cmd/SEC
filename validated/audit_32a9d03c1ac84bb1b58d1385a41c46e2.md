### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Systematically Overcharges Users When EVM Gas Exceeds Native Resource Consumption - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

`compute_gas_refund` in the ZKsync OS bootloader applies a one-sided correction to `gas_used` based on the difference between native resource consumption and EVM gas consumption (`delta_gas`). The adjustment is only applied when `delta_gas > 0` (native heavier than gas), but is silently skipped when `delta_gas < 0` (gas heavier than native). This mirrors the structural flaw in the `WithdrawProxy` report: a formula that only handles one direction of a signed discrepancy, leaving the opposite case uncompensated. The result is that users are systematically overcharged whenever their EVM gas consumption exceeds their native resource consumption, with the surplus flowing to the operator.

### Finding Description

In `compute_gas_refund`, after computing `gas_used` from remaining ergs and applying the EIP-3529 refund, the function computes:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native consumption > gas consumption → charge more gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The design intent, documented explicitly, is:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed`… [2](#0-1) 

When `delta_gas < 0` — meaning `native_used / native_per_gas < gas_used`, i.e., the transaction consumed more EVM gas than its native resource footprint warrants — the code does nothing. The `gas_used` value remains at the higher EVM-derived figure. The `TODO` comment on line 80 is the developers' own acknowledgment that this branch is unhandled.

The final `gas_used` directly determines:
- **Refund to sender**: `(gas_limit - gas_used) * gas_price` — reduced by the uncompensated `|delta_gas|`
- **Payment to operator**: `gas_used * gas_price` — inflated by the same amount [3](#0-2) [4](#0-3) 

The symmetric correct formula would be:

```
gas_used = max(minimal_gas_used, evm_gas_used_after_refund, native_used / native_per_gas)
```

or equivalently, apply `delta_gas` unconditionally (clamped so `gas_used` does not go below `minimal_gas_used`).

### Impact Explanation

Every L2 and L1→L2 transaction where EVM opcode gas consumption exceeds the native-resource-equivalent cost triggers this path. The user's refund is reduced by `|delta_gas| * gas_price` tokens, and the operator receives that surplus. For a transaction with `gas_limit = 1,000,000`, `gas_price = 1,000 gwei`, and `|delta_gas| = 50,000`, the overcharge is 0.05 ETH-equivalent per transaction. Because `gas_used` feeds directly into `token_to_refund` and `token_to_pay_operator`, this is a direct, deterministic loss of user funds with no revert path. [5](#0-4) 

### Likelihood Explanation

`delta_gas < 0` occurs whenever `remaining_ergs / ERGS_PER_GAS < remaining_native / native_per_gas` at the end of execution — i.e., the user exhausted EVM gas faster than native resources. This is the common case for EVM-heavy, storage-light transactions (e.g., pure computation, large calldata processing, many arithmetic opcodes). Any unprivileged sender submitting a standard L2 transaction can reach this code path without any special privileges. The `#[cfg(not(feature = "unlimited_native"))]` guard means the block is active in all production builds. [6](#0-5) 

### Recommendation

Apply `delta_gas` symmetrically, clamped to preserve the `minimal_gas_used` floor:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // delta_gas < 0: native consumption is less than gas consumption.
    // Return the surplus to the user, but never go below minimal_gas_used.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This matches the symmetric intent described in the documentation and eliminates the systematic overcharge.

### Proof of Concept

Consider an L2 transaction with:
- `gas_limit = 100_000`, `gas_price = 1_000`, `native_per_gas = 10`
- After execution: `remaining_ergs = 20_000 * ERGS_PER_GAS` → `gas_used = 80_000`
- `native_used = 500_000` → `native_used / native_per_gas = 50_000`
- `delta_gas = 50_000 - 80_000 = -30_000`

Current behavior: `gas_used` stays at `80_000`. Refund = `20_000 * 1_000 = 20,000,000` units.

Correct behavior: `gas_used` should be reduced to `50_000`. Refund = `50_000 * 1_000 = 50,000,000` units.

The user loses `30,000,000` token units per transaction. The operator gains the same amount. This is reachable by any unprivileged sender submitting a transaction whose EVM opcode cost exceeds its native proving cost — a routine scenario for compute-heavy, storage-light contracts. [7](#0-6)
