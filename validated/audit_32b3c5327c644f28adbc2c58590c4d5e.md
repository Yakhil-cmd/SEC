Audit Report

## Title
VIP Callers Blocked by Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
The `available_slot_count == 0` guard in `try_borrow_slot` is evaluated unconditionally before the VIP/non-VIP distinction takes effect. An unprivileged caller who exhausts all 167 slots causes every subsequent call — including VIP calls from NNS governance, SNS-W, or other NNS canisters — to receive `SysTransient`, blocking management canister operations routed through the NNS root canister for the duration of the attack.

## Finding Description
In `LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` (`rs/nervous_system/clients/src/management_canister_client.rs`, lines 264–287):

```rust

    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };  // line 265

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {          // line 269 — fires for VIP too
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The guard at line 269 returns `Err` for any caller when `available_slot_count == 0`, regardless of `is_caller_vip`. The VIP path (`used_slot_count = 0`, no slot consumed) is only reached at line 278, which is never executed when the guard fires. The `SlotLoan::drop` implementation restores `used_slot_count` slots on completion, but for VIP callers `used_slot_count = 0`, so the counter is never restored either — the design intent is that VIP callers do not consume or restore slots at all, but the guard prevents them from proceeding.

The public `canister_status` endpoint (`rs/nns/handlers/root/impl/canister/canister.rs`, lines 88–98) has no access control. `new_management_canister_client()` (lines 53–67) sets `is_caller_vip = false` for any caller that is not an NNS canister principal. The shared counter `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is initialized to 167 (line 50) and is shared across all callers.

Exploit flow:
1. Attacker sends 167 concurrent update calls to the public `canister_status` endpoint on the NNS root canister.
2. Each call passes through `try_borrow_slot`, decrements the counter by 1, and then awaits the management canister inter-canister call (yielding execution).
3. With all 167 slots occupied, `available_slot_count == 0`.
4. Any new call — including NNS governance calling `update_canister_settings`, `take_canister_snapshot`, or `load_canister_snapshot` — hits the guard at line 269 and receives `Err((SysTransient, ...))`.
5. The attacker continuously replaces completing calls with new ones to sustain the condition.

Affected governance-routed operations: `update_canister_settings` (line 221), `change_canister_controllers` (line 207), `take_canister_snapshot` (line 245), `load_canister_snapshot` (line 259), and the public `canister_status` (line 89). The public `canister_status` endpoint additionally calls `.unwrap()` on the result (line 97), meaning a trapped VIP call through that path causes the root canister to trap rather than return a graceful error.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

An unprivileged attacker can block NNS governance from executing management canister operations through the root canister — specifically `update_canister_settings`, `take_canister_snapshot`, `load_canister_snapshot`, and `change_canister_controllers` — for the entire duration of the sustained attack. These are legitimate NNS governance proposal execution paths. The attack requires no key material, governance majority, or operator access.

## Likelihood Explanation
- The `canister_status` endpoint is intentionally public with no per-caller rate limiting.
- The IC permits a single user to have many concurrent in-flight update calls; 167 is well within the per-canister queue limit of 500.
- The attacker only needs to sustain the flood, not perform any privileged action.
- No special privileges, leaked keys, or social engineering are required.
- The attack is repeatable and can be sustained indefinitely at low cost.

## Recommendation
Move the `available_slot_count == 0` guard inside the non-VIP branch so VIP callers bypass it entirely:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;

    let available_slot_count = self.available_slot_count;
    Ok(SlotLoan {
        available_slot_count,
        used_slot_count,
    })
}
```

This ensures VIP callers always proceed regardless of how many non-VIP slots are in use.
