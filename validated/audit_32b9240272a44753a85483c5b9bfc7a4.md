Audit Report

## Title
`get_allowances` leaks allowances across all subaccounts of a principal due to missing subaccount equality check — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
`icrc103_get_allowances` iterates a BTreeMap range anchored at the queried `from_account` but the loop break guard only checks `owner` inequality, never `subaccount` inequality. Because `AccountSpender` ordering is `(owner, effective_subaccount, spender)` and `[0u8;32]` is the minimum subaccount, querying with `{owner: X, subaccount: None}` returns every stored allowance for every subaccount of X. Any unprivileged caller can enumerate all active approvals across all subaccounts of any target principal.

## Finding Description
In `get_allowances` (`rs/ledger_suite/icrc1/ledger/src/lib.rs`), the BTreeMap range is opened at `start_account_spender` which is constructed with `account: from` and the minimum possible spender principal. [1](#0-0) 

The iteration loop contains only one structural break guard: [2](#0-1) 

There is no corresponding check on `account_spender.account != from`. `AccountSpender::cmp` orders by `account` first (which uses `effective_subaccount()`), then by `spender`: [3](#0-2) 

`Account::cmp` uses `effective_subaccount()`, and `Account::PartialEq` also uses `effective_subaccount()`, so `{owner: X, subaccount: None}` is identical to `{owner: X, subaccount: Some([0u8;32])}` — the minimum possible subaccount: [4](#0-3) [5](#0-4) 

Therefore a range starting at `(X, [0u8;32], min_spender)` includes every stored entry `(X, S, spender)` for any `S >= [0u8;32]`, which is every possible subaccount. The loop never breaks on subaccount mismatch, so all of them are collected and returned.

The endpoint is a public `#[query]` with no access control, callable by any principal: [6](#0-5) 

## Impact Explanation
An unprivileged attacker can call `icrc103_get_allowances({from_account: {owner: X, subaccount: None}, prev_spender: None, take: None})` and receive `Allowance103` records whose `from_account` fields point to subaccounts of X that were never queried. This discloses the full set of active approvals — including spender identities, amounts, and expiry times — for every subaccount of any target principal. This constitutes unauthorized disclosure of ledger state beyond what the queried account scope permits, enabling targeted discovery of high-value DeFi positions and approvals held in specific subaccounts. This matches the **High** bounty impact: "Significant ledger security impact with concrete user or protocol harm."

## Likelihood Explanation
The attack requires a single public query call with a crafted `from_account`. No tokens, approvals, privileged role, or prior interaction with the target is needed. The endpoint is callable by any principal on any ICRC-1 ledger deploying this code. The attack is trivially repeatable with zero cost.

## Recommendation
Add a subaccount equality guard immediately after the owner check in the loop body in `rs/ledger_suite/icrc1/ledger/src/lib.rs`:

```rust
if account_spender.account.owner != from.owner {
    break;
}
// Add this:
if account_spender.account != from {
    break;
}
``` [2](#0-1) 

## Proof of Concept
1. Owner X calls `icrc2_approve(from={owner:X, subaccount:Some([0xFF;32])}, spender:A, amount:1_000_000)`. This stores `(X, [0xFF;32], A)` in `ALLOWANCES_MEMORY`.
2. Attacker (any principal) calls `icrc103_get_allowances({from_account: Some({owner:X, subaccount:None}), prev_spender:None, take:None})`.
3. `start_account_spender` is `(X, [0u8;32], min_principal)`. The stored entry `(X, [0xFF;32], A)` satisfies `>= start` and `owner == X`, so the loop does not break.
4. The response contains `Allowance103 { from_account: {owner:X, subaccount:[0xFF;32]}, to_spender:A, allowance:1_000_000 }` — a subaccount the caller never queried.
5. Assert `∀ a ∈ response: a.from_account == queried_from_account` fails, confirming the invariant violation.

A deterministic integration test using PocketIC can reproduce this by: (a) approving from a non-default subaccount, (b) calling `icrc103_get_allowances` with the default subaccount from a different principal, and (c) asserting the response is non-empty and contains the non-default subaccount's allowance.
