### Title
Permanent Loss of ERC-20/ETH Funds When `exit_to_near` Promise Fails Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary

When a user bridges ERC-20 tokens (or ETH) from Aurora to NEAR via the `exit_to_near` precompile, the tokens are burned/debited on the Aurora side before the NEAR-side `ft_transfer` promise is dispatched. If that promise fails and the `error_refund` compile-time feature is not enabled (which is the production default), no callback is registered to re-mint the burned tokens. The funds are permanently destroyed with no recovery path.

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles bridging of ERC-20 tokens and ETH from Aurora to NEAR. The flow is:

1. ERC-20 tokens are burned on the Aurora side (or ETH is debited from the precompile address).
2. A NEAR `ft_transfer` (or `ft_transfer_call`) promise is dispatched to the NEP-141 contract.
3. If the promise fails, a callback (`exit_to_near_precompile_callback`) is supposed to re-mint the burned tokens.

However, the refund callback argument is gated behind the `error_refund` compile-time feature:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // <-- always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a standard ERC-20 exit, `transfer_near_args` is also `None`, so `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The subsequent branch then creates the promise **without any callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `error_refund` feature is absent from both the `default` and `contract` feature sets in `engine/Cargo.toml`:

```toml
[features]
default = ["std"]
contract = ["log", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [3](#0-2) 

The production WASM is built with `--features contract`, which does not include `error_refund`. Therefore, in production, any failed `ft_transfer` promise leaves the user's tokens permanently burned with no refund.

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The same applies to ETH exits:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

The callback handler in `engine/src/contract_methods/connector.rs` correctly handles the refund when `args.refund` is `Some`, but since `refund` is always `None` in production, this branch is never reached:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing/destruction of funds.**

When the `ft_transfer` promise fails (e.g., recipient NEAR account not registered with the NEP-141 token, storage deposit missing, contract paused, or invalid account ID), the ERC-20 tokens have already been burned on the Aurora side. Without the `error_refund` feature, no re-mint callback is scheduled. The tokens are permanently destroyed: gone from Aurora's EVM state and never credited on NEAR. This is an irreversible loss of user funds.

### Likelihood Explanation

**High.** NEAR NEP-141 tokens require recipients to have a storage deposit registered before they can receive tokens. A user who specifies an unregistered NEAR account ID as the exit destination will silently lose their tokens. This is a common mistake. Additionally, any transient failure of the NEP-141 contract (e.g., paused state, out-of-gas on the NEAR side) triggers the same permanent loss. The `error_refund` feature is not enabled in the production build, making every failed exit a permanent fund loss.

### Recommendation

Enable the `error_refund` feature in the production `contract` feature set:

```toml
contract = ["log", "error_refund", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
```

Alternatively, unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` and always attach the `exit_to_near_precompile_callback` callback to the `ft_transfer` promise, removing the compile-time gate entirely. This ensures that any failed NEAR-side transfer always triggers a re-mint of the burned tokens on the Aurora side.

### Proof of Concept

1. User holds 100 units of a bridged ERC-20 token on Aurora (backed by a NEP-141 on NEAR).
2. User calls the ERC-20 contract's burn/exit function, which internally calls the `exit_to_near` precompile specifying `unregistered.near` as the recipient.
3. The precompile burns 100 ERC-20 tokens on Aurora and dispatches `ft_transfer(receiver_id: "unregistered.near", amount: "100")` to the NEP-141 contract.
4. The `ft_transfer` call fails because `unregistered.near` has no storage deposit.
5. Because `error_refund` is not enabled, `callback_args.refund == None`, so no callback was attached to the promise.
6. Result: 100 ERC-20 tokens are permanently burned on Aurora; `unregistered.near` receives nothing. The user has lost 100 tokens with no recovery path.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–666) and `test_exit_to_near_eth_refund` (lines 717–781) confirm this exact behavior under `#[cfg(not(feature = "error_refund"))]`. [7](#0-6) [8](#0-7)
