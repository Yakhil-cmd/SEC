Audit Report

## Title
Byzantine `content_size` Inflation Forges `ResponsesTooLarge` Proof in Flexible HTTP Outcall Validator — (`rs/https_outcalls/consensus/src/payload_builder.rs`)

## Summary

The `ResponsesTooLarge` branch of `validate_canister_http_payload_impl` uses `share.content.content_size()` directly as a size estimate without bounding it against `MAX_CANISTER_HTTP_RESPONSE_BYTES`. A Byzantine node controls its own signing key and can sign a `CanisterHttpResponseReceipt` with an arbitrarily large `content_size`. When the canister uses a high-`min_responses` configuration (e.g., M = C), a single such inflated share is sufficient to satisfy `min_known_ok_needed = 1`, causing the validator to accept a forged impossibility proof, permanently delivering an error to the canister's callback and burning its cycles.

## Finding Description

**Arithmetic at L786–787:**

```rust
let num_unseen = flex_committee.len().saturating_sub(all_seen_shares.len());
let min_known_ok_needed = min_responses.saturating_sub(num_unseen);
```

With C=4, M=4, and only 1 share included: `num_unseen = 3`, `min_known_ok_needed = 1`.

**Size estimate at L789–799 uses `content_size` without bounds check:**

```rust
.map(|share| {
    FlexibleCanisterHttpResponseWithProof::count_bytes_from_parts(
        &context.request.sender,
        share.content.content_size() as usize,  // ← no upper-bound check
        share,
    )
})
```

**Acceptance condition at L810–818:**

```rust
let smallest_sum: usize = ok_entry_sizes.iter().take(min_known_ok_needed).sum();
if smallest_sum <= MAX_CANISTER_HTTP_PAYLOAD_SIZE {
    return invalid_artifact(...FlexibleResponsesNotTooLarge...);
}
```

If `smallest_sum > MAX` (2 MiB), the proof is accepted. A Byzantine share with `content_size = 3 MiB` satisfies this with `min_known_ok_needed = 1`.

**`validate_response_share` (L208–251) does not check `content_size`:** it only validates callback-id, duplicate signers, committee membership, registry version, and refund allowance. Cryptographic signature validity is checked via batch verification at L824–830, but a Byzantine node signs with its own legitimate key — the signature is valid.

**Pool manager check at L544–549 is bypassable:** the pool manager validates `content_size` against the actual response only for artifacts arriving via gossip. A Byzantine block proposer can include a crafted share directly in the block payload, bypassing the pool entirely.

**`content_size` is a plain `u32` field** in `CanisterHttpResponseMetadata` (L1042) with no protocol-level constraint enforced at signing time.

**Attack path (C=4, M=4, f=1):**
1. Byzantine node signs a `CanisterHttpResponseReceipt` with `content_size = 3 MiB`.
2. Byzantine block proposer includes only this 1 share in `all_seen_shares`; the 3 honest shares are omitted.
3. `num_unseen = 3`, `min_known_ok_needed = 1`, `smallest_sum = ~3 MiB > 2 MiB = MAX`.
4. Validator accepts the forged `ResponsesTooLarge` proof.
5. Execution delivers a permanent error to the canister's callback; cycles are burned; no retry occurs.

**Default configuration is not vulnerable:** with `min_responses = floor(2n/3) + 1` (L642), `min_known_ok_needed` requires `f > fault_threshold` to satisfy with inflated shares. Only canister-specified configurations where `M > C - fault_threshold(C)` are vulnerable.

## Impact Explanation

A Byzantine block proposer can permanently block any canister that uses a high-`min_responses` flexible HTTP outcall. The finalized error is irreversible: the callback receives a permanent failure, cycles are burned, and downstream logic is permanently blocked. This is a targeted, repeatable application/platform-level DoS with irreversible financial impact on affected canisters, matching the **High** impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

Requires one Byzantine node that is both in the flexible committee and acts as (or colludes with) the block proposer for the targeted round. With f=1 (within the fault threshold for C=4), this is achievable without violating the protocol's fault assumption. The canister must use M > C − fault_threshold(C), which is a valid and documented configuration option. The inflated-`content_size` share is cryptographically indistinguishable from a legitimate share to the validator. No external infrastructure compromise, governance majority, or threshold key access is needed.

## Recommendation

In the `ResponsesTooLarge` validation branch, add an explicit upper-bound check on each share's `content_size` before using it in the size calculation:

```rust
let raw_content_size = share.content.content_size() as usize;
if raw_content_size > MAX_CANISTER_HTTP_RESPONSE_BYTES as usize {
    return invalid_artifact(
        InvalidCanisterHttpPayloadReason::FlexibleContentSizeExceedsMax { ... }
    );
}
```

This mirrors the bound enforced by the pool manager on incoming artifacts and closes the gap between pool-level and payload-level validation.

## Proof of Concept

The existing test `flexible_error_responses_too_large_invalid_when_committee_members_omitted` (L4006–4043) uses `min_responses=2, committee=4`, yielding `min_known_ok_needed=0` — it never exercises the M=C case. A new test:

```rust
// C=4, M=4, f=1 Byzantine node
let huge: u32 = (MAX_CANISTER_HTTP_PAYLOAD_SIZE as u32) + 1_000_000; // 3 MiB
let byzantine_share = metadata_share_with_content_size(callback_id.get(), 0, huge);

// Proposer omits the 3 honest shares
let payload = CanisterHttpPayload {
    flexible_errors: vec![FlexibleCanisterHttpError::ResponsesTooLarge {
        callback_id,
        all_seen_shares: vec![byzantine_share],
        total_requests: 4,
        min_responses: 4,  // M = C
    }],
    ..Default::default()
};

// num_unseen=3, min_known_ok_needed=1, smallest_sum=~3MiB > MAX → accepted
let result = pb.validate_payload(Height::new(1), &ctx, &payload_bytes, &[]);
assert_matches!(result, Ok(())); // forged proof accepted — demonstrates the bug
```
