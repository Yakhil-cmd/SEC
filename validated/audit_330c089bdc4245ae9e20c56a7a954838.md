Audit Report

## Title
Gzip Footer Manipulation Exhausts Subnet Message Instruction Budget via Uncapped `compilation_cost` in `create_execution_state` - (`rs/execution_environment/src/hypervisor.rs`, `rs/embedders/src/wasm_utils/decoding.rs`)

## Summary
`wasm_encoding_and_size` reads the RFC-1952 ISIZE footer field as a raw `u32` cast to `usize` with no cap, returning an attacker-controlled value up to 4,294,967,295. `Hypervisor::create_execution_state` multiplies this value by 6,000 to produce a `compilation_cost` of ~25.7 trillion instructions, then deducts that amount from `round_limits.instructions` in the `Err` path after `decode_wasm` rejects the oversized module. This exhausts the subnet message instruction budget for the round, blocking all subsequent `InstallCode`, `InstallChunkedCode`, `UpdateSettings`, `UploadChunk`, `TakeCanisterSnapshot`, and other `counts_toward_round_limit` management operations for that round.

## Finding Description

**Step 1 — Attacker-controlled size, no cap.**

`wasm_encoding_and_size` reads the last 4 bytes of the gzip stream as a raw `u32` and casts directly to `usize`:

```rust
let uncompressed_size = u32::from_le_bytes(isize_bytes) as usize;
return Ok((WasmEncoding::Gzip, uncompressed_size));
```

The function's own doc-comment explicitly warns this value is untrusted. [1](#0-0) 

**Step 2 — Uncapped size drives `compilation_cost`.**

`Hypervisor::create_execution_state` calls `decoded_wasm_size`, takes the result at face value, and multiplies by `cost_to_compile_wasm_instruction` (default 6,000):

```rust
let wasm_size_result = decoded_wasm_size(canister_module.as_slice());
let wasm_size = match wasm_size_result {
    Ok(size) => std::cmp::max(size, canister_module.len()),
    Err(_) => canister_module.len(),
};
let compilation_cost = self.cost_to_compile_wasm_instruction * wasm_size as u64;
```

With `wasm_size = 4,294,967,295`: `compilation_cost = 6,000 × 4,294,967,295 = 25,769,803,770,000`. No overflow (fits in `u64`), no cap. [2](#0-1) [3](#0-2) 

**Step 3 — `decode_wasm` rejects the module, but `compilation_cost` is already computed.**

Since `wasm_size_result` is `Ok`, the early-return guard at lines 161–167 is **not** taken. Execution proceeds to `wasm_executor.create_execution_state`, which calls `decode_wasm` with `wasm_max_size`. Because `4,294,967,295 > 100 MiB`, `decode_wasm` returns `Err(ModuleTooLarge)`:

```rust
if uncompressed_size as u64 > max_size.get() {
    return Err(WasmValidationError::ModuleTooLarge { ... });
}
``` [4](#0-3) [5](#0-4) 

**Step 4 — The `Err` branch deducts the inflated cost from `round_limits.instructions`.**

Back in `create_execution_state`, the `Err` arm uses the pre-computed, attacker-inflated `compilation_cost`:

```rust
Err(err) => {
    let total_cost = self.create_execution_state_base_cost + compilation_cost;
    round_limits.instructions -= as_round_instructions(total_cost);
    (total_cost, Err(err))
}
``` [6](#0-5) 

`RoundInstructions` is a signed `i64`-backed counter. `as_round_instructions` converts `NumInstructions(u64)` to `i64` via `i64::try_from(n.get()).unwrap_or(i64::MAX)`. The value 25,769,803,770,000 fits in `i64`, so it converts cleanly and drives `round_limits.instructions` deeply negative. [7](#0-6) 

**Step 5 — Exhausted budget blocks all subsequent `counts_toward_round_limit` subnet messages.**

`install_code` is processed by `drain_subnet_queues` using `subnet_round_limits` (budget = `max_instructions_per_round / 16`). After the attack, `subnet_instructions` is deeply negative. `can_execute_subnet_msg` checks `round_limits.instructions_reached()` before executing any message with `counts_toward_round_limit: true`. `InstallCode` and `InstallChunkedCode` both have `counts_toward_round_limit: true`, so all such messages in the same round are skipped. [8](#0-7) [9](#0-8) [10](#0-9) 

The `SchedulerRoundLimits` maintains `subnet_instructions` and `instructions` as separate counters; the main canister execution budget (`instructions`) is unaffected, so regular canister ingress messages continue to run. The DoS is scoped to subnet management operations. [11](#0-10) 

**No upstream guard exists.** `execute_install` calls `create_execution_state` directly after assembling the module, with no pre-check on the decoded size. [12](#0-11) 

## Impact Explanation

Every execution round in which the attacker submits this crafted `install_code` message has its subnet message instruction budget exhausted. All subsequent `InstallCode`, `InstallChunkedCode`, `UpdateSettings`, `FetchCanisterLogs`, `UploadChunk`, `TakeCanisterSnapshot`, and `LoadCanisterSnapshot` calls queued for that round are skipped. The attack is repeatable every round at the cost of one failing `install_code` call. This constitutes a sustained, low-cost DoS on subnet management and canister lifecycle operations — matching the allowed impact class **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"** at **High** severity ($2,000–$10,000).

## Likelihood Explanation

Requires only: a canister ID (create one with minimal cycles), a ~20-byte crafted gzip blob with ISIZE=`0xFFFFFFFF` in the footer, and an `install_code` ingress call. No privileged access, no key material, no governance majority. The crafted blob is trivially constructable and the attack is locally reproducible. It is repeatable every round.

## Recommendation

Cap `wasm_size` to `wasm_max_size` before computing `compilation_cost` in `Hypervisor::create_execution_state`. Since `decoded_wasm_size` is explicitly documented as untrusted for gzip, the cost estimate must be bounded:

```rust
let wasm_size = wasm_size.min(self.wasm_max_size.get() as usize);
let compilation_cost = self.cost_to_compile_wasm_instruction * wasm_size as u64;
```

Alternatively, move the `decode_wasm` size check to occur **before** `compilation_cost` is computed, so an oversized footer causes an early `Err` return that charges only the base cost.

## Proof of Concept

```python
# Minimal valid gzip: 10-byte header + 2-byte empty deflate + 8-byte footer
# Footer: CRC32=0x00000000, ISIZE=0xFFFFFFFF
payload  = b'\x1f\x8b\x08\x00\x00\x00\x00\x00\x00\x03'  # gzip header
payload += b'\x03\x00'                                    # empty deflate block
payload += b'\x00\x00\x00\x00'                           # CRC32
payload += b'\xff\xff\xff\xff'                           # ISIZE = u32::MAX

# Submit via install_code to any canister you control.
# Expected: round_limits.instructions for subnet messages drops by ~25.7 trillion.
# All subsequent InstallCode/InstallChunkedCode/UpdateSettings/UploadChunk
# messages in the same round are skipped by can_execute_subnet_msg().
```

A deterministic integration test can be written using `SchedulerTestBuilder` with a small `max_instructions_per_round`, injecting the crafted blob as an `install_code` call followed by a second `install_code` call, and asserting the second call remains unexecuted in the same round.
