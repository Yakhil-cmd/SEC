### Title
ckBTC Minter `upgrade()` Silently Ignores `min_confirmations` Increases, Permanently Locking Security Parameter at Minimum — (File: `rs/bitcoin/ckbtc/minter/src/state.rs`)

---

### Summary

The `CkBtcMinterState::upgrade()` function contains inverted logic that **only applies decreases** to `min_confirmations` and silently discards any governance-approved increase. With the current live value at 4 confirmations (reduced from 72 → 12 → 6 → 4 via successive NNS proposals), the minter cannot be hardened via the standard upgrade path if Bitcoin reorg risk materializes. Any user can trigger `update_balance` to mint ckBTC after 4 Bitcoin confirmations, and a miner with sufficient hash power can reverse a 4-deep Bitcoin chain to double-spend into the minter.

---

### Finding Description

In `rs/bitcoin/ckbtc/minter/src/state.rs`, the `upgrade()` method applies `min_confirmations` only when the new value is **strictly less than** the current value:

```rust
if let Some(min_conf) = min_confirmations {
    if min_conf < self.min_confirmations {   // only decreases are applied
        self.min_confirmations = min_conf;
    } else {
        log!(
            Priority::Info,
            "Didn't increase min_confirmations to {} (current value: {})",
            min_conf,
            self.min_confirmations
        );
    }
}
``` [1](#0-0) 

The condition `min_conf < self.min_confirmations` means: if the requested value is lower, apply it (decrease security); if higher, silently drop it (block security hardening). The log message confirms this: it fires when an operator tries to raise the threshold, and the raise is ignored.

The default value is `DEFAULT_MIN_CONFIRMATIONS = 6`: [2](#0-1) 

The live mainnet value has been reduced to **4** via the 2026-01-23 NNS upgrade proposal: [3](#0-2) 

The `reinit()` path (used for full re-initialization) does allow setting any value unconditionally, but it is not the standard governance upgrade path and would require wiping and re-installing the canister: [4](#0-3) 

The `update_balance` endpoint, callable by any unprivileged user, mints ckBTC once a UTXO reaches `min_confirmations`: [5](#0-4) 

---

### Impact Explanation

**Chain-fusion mint/burn bug.** A Bitcoin miner (or pool) controlling enough hash rate to reorganize 4 blocks can:

1. Send BTC to their ckBTC deposit address.
2. Wait for 4 confirmations; call `update_balance` to mint ckBTC.
3. Reorganize the Bitcoin chain to depth ≥ 4, reversing the deposit transaction.
4. Retain the minted ckBTC while the backing BTC no longer exists on-chain.

This breaks the 1:1 BTC backing invariant of ckBTC and constitutes an unbacked mint. The secondary impact is that even if NNS governance passes a proposal to raise `min_confirmations` back to 6 or higher in response to observed reorg activity, the `upgrade()` function will silently discard the increase, leaving the minter permanently exposed until a full re-initialization (which is operationally disruptive and not the standard upgrade path).

---

### Likelihood Explanation

Bitcoin reorgs at depth 4 are rare but not theoretical — they have occurred historically on mainnet. The more immediate concern is the governance bypass: the NNS has already reduced `min_confirmations` from 72 → 12 → 6 → 4 via successive proposals, and the code now makes it impossible to reverse that trend through the normal upgrade mechanism. Any user can call `update_balance` to trigger minting, making the entry path fully unprivileged and externally reachable.

---

### Recommendation

Fix the comparison direction in `upgrade()` so that **both** increases and decreases are applied, or at minimum allow increases while blocking decreases:

```rust
if let Some(min_conf) = min_confirmations {
    // Allow any governance-approved change, including increases for security hardening
    self.min_confirmations = min_conf;
}
```

If the intent is to prevent accidental weakening, the guard should block **decreases** (not increases):

```rust
if let Some(min_conf) = min_confirmations {
    if min_conf >= self.min_confirmations {
        self.min_confirmations = min_conf;
    } else {
        log!(Priority::Info, "Refused to decrease min_confirmations ...");
    }
}
```

---

### Proof of Concept

1. Deploy ckBTC minter with `min_confirmations = 4` (current mainnet state per the 2026-01-23 proposal).
2. Submit an NNS upgrade proposal with `UpgradeArgs { min_confirmations: Some(6), .. }` to harden the minter.
3. Observe that after the upgrade executes, `get_minter_info` still returns `min_confirmations = 4` — the increase was silently dropped by the `min_conf < self.min_confirmations` guard.
4. Confirm via the `ckbtc_minter_min_confirmations` Prometheus metric (line 280–282 of `rs/bitcoin/ckbtc/minter/src/metrics.rs`) that the value remains 4. [6](#0-5)
