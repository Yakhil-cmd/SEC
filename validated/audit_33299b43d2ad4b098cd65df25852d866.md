Audit Report

## Title
Unbounded `ingress_expiries` Loop in ICP Rosetta `construction_payloads` Allows OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` u64 nanosecond timestamps from JSON metadata with no server-side validation of their span. The while-loop at lines 99–107 iterates `(ingress_end − ingress_start) / 120_000_000_000` times with no bound, pushing heap objects per iteration. With `ingress_start=0` and `ingress_end=u64::MAX`, this produces ~153 million iterations and tens of gigabytes of heap allocation in a single synchronous request, crashing the Rosetta process via OOM.

## Finding Description
The interval is computed at line 59–60 as `MAX_INGRESS_TTL − PERMITTED_DRIFT − 120s = 300s − 60s − 120s = 120s = 120,000,000,000 ns`. [1](#0-0) 

The unbounded loop at lines 99–107 pushes one `u64` per iteration into `ingress_expiries` with no cap on the span: [2](#0-1) 

`ingress_start` and `ingress_end` are deserialized directly from JSON as bare `Option<u64>` with no range or span validation: [3](#0-2) 

For each entry in `ingress_expiries`, `add_payloads` pushes two heap-allocated `SigningPayload` structs (each containing a hex-encoded string, a cloned `AccountIdentifier`, and a `SignatureType`): [4](#0-3) 

With `u64::MAX / 120_000_000_000 ≈ 153,722,867` iterations: `ingress_expiries` alone consumes ~1.2 GB; the `payloads` vec with two `SigningPayload` structs per entry (each ~200 bytes) adds tens of gigabytes. The actix-web server applies only a 4 MB JSON body limit and no per-request timeout — the attack payload is a tiny JSON object well under 4 MB: [5](#0-4) 

The ICRC1 handler performs at least basic ordering validation (`ingress_start >= ingress_end`) before its loop, but the ICP handler has no equivalent guard whatsoever: [6](#0-5) 

## Impact Explanation
The Rosetta API is an in-scope financial integration component used by exchanges, wallets, and staking UIs for ICP ledger queries and transaction construction. A single malformed POST to `/construction/payloads` causes synchronous gigabyte-scale heap allocation on the request-handling thread, triggering the OS OOM killer or an allocator panic and denying service to all legitimate users until the process is manually restarted. This matches the allowed High impact: *"Application/platform-level DoS, crash … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … Rosetta … security impact with concrete user or protocol harm."*

**Severity: High ($2,000–$10,000)**

## Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication per the Rosetta spec and is publicly reachable. The attack requires a single HTTP POST with a two-field JSON metadata object. No special privileges, cryptographic keys, or network position are needed. The exploit is deterministic and trivially reproducible locally. It can be repeated indefinitely after each process restart.

## Recommendation
Add a span cap before the loop, mirroring the intent of the ICRC1 handler. For example, cap the number of ingress windows to a small constant (e.g., 10, matching typical use):

```rust
let max_ingress_windows: u64 = 10;
let max_span = interval * max_ingress_windows;
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    > max_span.as_nanos() as u64
{
    return Err(ApiError::invalid_request(
        "ingress_end − ingress_start exceeds the maximum allowed span",
    ));
}
```

Additionally, add a basic ordering check (`ingress_start >= ingress_end → error`) consistent with the ICRC1 handler. [2](#0-1) 

## Proof of Concept
```bash
curl -s -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<ledger-canister-id>"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION",
      "account":{"address":"<addr>"},"amount":{"value":"-1","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid-pubkey-hex>","curve_type":"edwards25519"}],
    "metadata": {"ingress_start": 0, "ingress_end": 18446744073709551615}
  }'
```

The loop executes `18446744073709551615 / 120000000000 ≈ 153,722,867` iterations. Memory usage can be verified locally by running the Rosetta binary under `valgrind --tool=massif` or by observing RSS growth via `/proc/<pid>/status` before the OOM kill. A unit test asserting that the handler returns `ApiError` for spans exceeding `interval * 10` would also confirm the fix.
