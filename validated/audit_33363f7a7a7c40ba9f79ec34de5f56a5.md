### Title
Unprotected (Pre-EIP-155) Legacy Transactions Accepted Without Chain-ID Binding, Enabling Cross-Chain Replay - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS explicitly accepts legacy (type-0) Ethereum transactions whose `v` value is `27` or `28` — the pre-EIP-155 "unprotected" format — without any chain-ID binding in the signing hash. A signature produced on any other EVM-compatible chain (Ethereum mainnet, another ZKsync chain, any L2) with the same nonce, `to`, `value`, and `data` fields is valid on ZKsync OS as well, because the signing hash contains no chain-specific domain separator.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155-protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: chain_id is mixed into the hash
    ...
}
```

`is_eip155()` returns `false` when `v == 27 || v == 28`. [1](#0-0) [2](#0-1) 

When the unprotected branch is taken, the signing hash is `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the pre-EIP-155 Ethereum standard. No chain ID, no contract address, no domain separator is included.

The parsed transaction is stored as the `Legacy` variant (not `LegacyWithEIP155`):

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    Self::Legacy(tx, sig_data)
};
``` [3](#0-2) 

The `chain_id()` accessor returns `None` for the `Legacy` variant, confirming no chain binding is enforced: [4](#0-3) 

The signature is then verified in the validation flow using this chain-ID-free hash: [5](#0-4) 

---

### Impact Explanation

An attacker who observes a pre-EIP-155 legacy transaction sent by victim Alice on Ethereum mainnet (or any other EVM chain) can replay the exact same raw transaction bytes on ZKsync OS. Because the signing hash contains only `[nonce, gasPrice, gasLimit, to, value, data]`, the signature is equally valid on ZKsync OS. If Alice holds the same nonce on ZKsync OS and has sufficient balance, the attacker can cause Alice's funds to be transferred to the same `to` address on ZKsync OS without Alice's consent or knowledge.

**Impact**: Unauthorized ETH/token transfer from victim accounts — direct theft of user funds.

---

### Likelihood Explanation

Pre-EIP-155 transactions are still broadcast on Ethereum mainnet by legacy tooling, hardware wallets in compatibility mode, and some DeFi protocols. Any such transaction is immediately replayable on ZKsync OS. The attacker needs only to:
1. Monitor the Ethereum mempool or historical blocks for unprotected legacy transactions.
2. Submit the identical raw bytes to ZKsync OS.

No privileged access, no leaked keys, no governance majority is required. The attack is fully permissionless and automatable.

---

### Recommendation

Reject unprotected (pre-EIP-155) legacy transactions entirely. The `Legacy` variant (v = 27/28) should be treated as `InvalidChainId` or `InvalidEncoding` at parse time, analogous to how typed transactions (EIP-2930, EIP-1559, EIP-7702) enforce `tx.chain_id != expected_chain_id`: [6](#0-5) 

Concretely, in `parse_and_compute_signed_hash`, after the legacy path, add:

```rust
if !sig_data.is_eip155() {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

This is consistent with how modern EVM chains (e.g., Ethereum post-Spurious Dragon) handle replay protection.

---

### Proof of Concept

1. Alice sends a pre-EIP-155 legacy transaction on Ethereum mainnet:
   - `nonce = 5`, `to = 0xDEAD...`, `value = 1 ETH`, `v = 27`
2. Attacker extracts the raw RLP bytes from the Ethereum mempool or a confirmed block.
3. Attacker submits the identical raw bytes to ZKsync OS via the sequencer RPC.
4. ZKsync OS parses the transaction, detects `v == 27`, takes the unprotected branch, computes `keccak256(rlp([5, gasPrice, gasLimit, 0xDEAD..., 1e18, ""]))`.
5. `ecrecover` recovers Alice's address — the signature is valid.
6. ZKsync OS checks Alice's nonce on ZKsync OS. If it is also `5`, the transaction executes, transferring 1 ETH from Alice's ZKsync OS account to `0xDEAD...` without Alice's authorization. [7](#0-6) [8](#0-7)
