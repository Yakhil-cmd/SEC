### Title
Lack of NEP-141 Token Pause-Capability Check Enables Permanent ERC-20 Fund Freeze on Exit - (`engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens inside the EVM before the downstream `ft_transfer` promise on the NEP-141 contract is executed. When the `error_refund` feature is not compiled in, there is no callback to re-mint the burned tokens if `ft_transfer` fails. Any NEP-141 token with a pause/freeze mechanism (e.g., USDC, USDT on NEAR) that becomes paused after being bridged to Aurora will cause users' ERC-20 tokens to be permanently destroyed while the corresponding NEP-141 tokens remain locked in Aurora's account — a permanent fund freeze.

### Finding Description

**Bridge deposit path (no issue here):**
A user calls `ft_transfer_call` on a NEP-141 contract, which triggers `ft_on_transfer` on Aurora. Inside `receive_erc20_tokens`, the engine mints ERC-20 tokens to the recipient.

**Bridge exit path (vulnerable):**
A user calls `exitToNear` on the ERC-20 contract. The ERC-20 contract burns the user's tokens and calls the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`. The precompile constructs a NEAR promise to call `ft_transfer` on the NEP-141 contract. The EVM execution (including the burn) completes atomically. The `ft_transfer` promise is executed in a subsequent NEAR receipt.

The critical conditional in `ExitToNear::run()`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
// ...
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← NO callback registered
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

For a standard ERC-20 exit (not wNEAR unwrap), `transfer_near_args` is `None`. Without `error_refund`, `callback_args` equals the default value, so **no callback promise is created**. If `ft_transfer` on the NEP-141 contract fails (e.g., because the token is paused), there is no `exit_to_near_precompile_callback` invocation and therefore no call to `refund_on_error` to re-mint the burned ERC-20 tokens.

The test suite explicitly documents this behavior:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

The `deploy_erc20_token` function accepts any NEP-141 `AccountId` without inspecting whether the token contract implements a pause or freeze mechanism:


### Impact Explanation

**Permanent freezing of funds.** When a bridged NEP-141 token is paused:

1. ERC-20 tokens are burned inside the EVM (irreversible within the EVM).
2. `ft_transfer` on the NEP-141 contract fails silently (no callback without `error_refund`).
3. The NEP-141 tokens remain in Aurora's account with no mechanism to recover them.
4. Users lose both their ERC-20 tokens and their underlying NEP-141 tokens permanently.

Even with `error_refund` enabled, if the NEP-141 token is indefinitely paused, users can never bridge back to NEAR — their value is permanently trapped on Aurora.

### Likelihood Explanation

Many production NEP-141 tokens implement pause functionality (USDC, USDT, and other regulated stablecoins). A regulatory freeze, exploit response, or upgrade migration pause on any such token that has been bridged to Aurora would trigger this condition for all Aurora users holding the corresponding ERC-20. The original Lavarage judge noted that even rare events causing innocent-user losses justify Medium severity; the same reasoning applies here.

### Recommendation

1. **Always enable `error_refund` in production builds** so that a failed `ft_transfer` triggers `exit_to_near_precompile_callback` → `refund_on_error`, re-minting burned ERC-20 tokens.
2. **Add a pause-capability check** in `deploy_erc20_token`: query the NEP-141 contract for a known pause interface (e.g., `is_paused`) before registering the token mapping, and reject tokens that expose a pause authority.
3. **Implement an admin recovery path** that can transfer stuck NEP-141 tokens out of Aurora's account to affected users when a bridged token becomes permanently frozen.

### Proof of Concept

1. Deploy a NEP-141 token contract that implements `ft_transfer` with a pause guard.
2. Bridge it to Aurora via `deploy_erc20_token`.
3. Deposit NEP-141 tokens into Aurora via `ft_transfer_call` → `ft_on_transfer` → `receive_erc20_tokens`; user receives ERC-20 tokens.
4. Pause the NEP-141 contract (token issuer action).
5. User calls `exitToNear` on the ERC-20 contract with flag `0x01` (ERC-20 exit).
6. EVM burns the ERC-20 tokens; `ExitToNear` precompile creates a bare `PromiseArgs::Create` (no callback) targeting `ft_transfer` on the paused NEP-141.
7. `ft_transfer` fails; no callback fires; no `refund_on_error` is called.
8. Result: user's ERC-20 balance is zero, NEP-141 balance in Aurora's account is unchanged, user has no recovery path.

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
```

```rust
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
```
