### Title
Truncating Division-Before-Multiplication in `native_per_pubdata` Calculation Causes Systematic Underpayment for Pubdata — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validation_impl.rs`, `native_per_pubdata` is computed as `floor(pubdata_price / native_price)` via truncating integer division. This truncated per-byte rate is then multiplied by `current_pubdata_spent` to determine the native resource charge for pubdata. The division-before-multiplication ordering causes systematic underpayment: users are charged less native resource per pubdata byte than the operator intended, allowing them to publish more data to L1 than they paid for.

---

### Finding Description

**Root cause — `validation_impl.rs` line 142:**

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is truncating (floor) integer division. The result `native_per_pubdata = ⌊pubdata_price / native_price⌋` is stored and later used in two places in `gas_helpers.rs`:

**Intrinsic pubdata overhead (line 352):**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
```

**Post-execution pubdata charge (lines 430–432):**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
```

Both computations perform:

```
actual_charge = current_pubdata_spent × ⌊pubdata_price / native_price⌋
```

The mathematically correct charge is:

```
correct_charge = ⌊current_pubdata_spent × pubdata_price / native_price⌋
```

By the distributive property of floor division, `actual_charge ≤ correct_charge` always. The undercharge per transaction is:

```
undercharge = ⌊current_pubdata_spent × (pubdata_price mod native_price) / native_price⌋
```

This can reach up to `current_pubdata_spent − 1` native units.

**Concrete example:**

| Parameter | Value |
|---|---|
| `pubdata_price` | 1500 |
| `native_price` | 1000 |
| `current_pubdata_spent` | 100 bytes |

- `native_per_pubdata = ⌊1500/1000⌋ = 1`
- `actual_charge = 100 × 1 = 100` native units
- `correct_charge = ⌊100 × 1500 / 1000⌋ = 150` native units
- **Undercharge: 50 native units (33%)**

The same truncation appears in the off-chain helper at `api/src/helpers.rs` line 427:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

Note the asymmetry: `native_per_gas` uses `div_ceil` (rounding up, favouring the operator), while `native_per_pubdata` uses `wrapping_div` (rounding down, favouring the user).

---

### Impact Explanation

The native resource limit is `gas_limit × native_per_gas`. Pubdata consumes `current_pubdata_spent × native_per_pubdata` native units. When `pubdata_price` is not an exact multiple of `native_price`, the truncation means each pubdata byte costs fewer native units than intended. A user whose transaction is native-resource-bound (common when pubdata is expensive relative to gas) can publish more bytes to L1 than they paid for. In the extreme case where `pubdata_price < native_price`, `native_per_pubdata = 0` and the user pays **zero** native resource for all pubdata, regardless of how much they generate. The operator bears the full L1 data-availability cost for the excess pubdata without receiving corresponding fee compensation.

---

### Likelihood Explanation

Any unprivileged L2 transaction sender can trigger this path. The condition `pubdata_price mod native_price ≠ 0` is likely in any real deployment where the operator sets these two independent price parameters without ensuring exact divisibility. The worst-case scenario (`pubdata_price < native_price`) is reachable whenever the operator sets a low pubdata price relative to the native price. No special privileges, leaked keys, or oracle manipulation are required.

---

### Recommendation

Replace the truncating division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is already computed:

```rust
// Before (truncates, favours user):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Alternatively, defer the division to the point of use and compute `⌊current_pubdata_spent × pubdata_price / native_price⌋` directly using 128-bit or wider arithmetic to avoid overflow.

The same fix should be applied to `api/src/helpers.rs` line 427.

---

### Proof of Concept

The following values demonstrate the undercharge:

```
pubdata_price  = 1500
native_price   = 1000
pubdata_bytes  = 10_000

native_per_pubdata (current)  = floor(1500 / 1000) = 1
native_charged (current)      = 10_000 × 1         = 10_000

native_per_pubdata (correct)  = ceil(1500 / 1000)  = 2
native_charged (correct)      = 10_000 × 2         = 20_000

Undercharge = 10_000 native units (50%)
```

A user whose native resource budget is 10,000 units can generate 10,000 pubdata bytes under the current code, but should only be able to generate 5,000 bytes.
