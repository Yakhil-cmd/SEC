### Title
`RequestContext` Hash/Eq Missing `request_type` Field Enables Cross-Version Cache Poisoning - (`rs/boundary_node/ic_boundary/src/routes.rs`)

---

### Summary

The custom `Hash` and `PartialEq` implementations for `RequestContext` in the IC boundary node omit the `request_type` field. Because `request_type` is excluded from the cache key, a `QueryV2` request and a `QueryV3` request with identical parameters (`canister_id`, `sender`, `method_name`, `ingress_expiry`, `arg`) are treated as the same cache entry. An unprivileged attacker can deliberately populate the cache with a `QueryV2` response, causing all subsequent `QueryV3` requests with the same parameters to receive an uncertified response — silently stripping the node-signature certificate that `QueryV3` is designed to provide.

---

### Finding Description

`RequestContext` is defined in `rs/boundary_node/ic_boundary/src/routes.rs` and carries all per-request metadata extracted from the CBOR envelope:

```rust
pub struct RequestContext {
    pub request_type: RequestType,   // ← NOT in Hash / PartialEq
    pub request_size: u32,
    pub canister_id: Option<Principal>,
    pub sender: Option<Principal>,
    pub method_name: Option<String>,
    pub nonce: Option<Vec<u8>>,
    pub ingress_expiry: Option<u64>,
    pub arg: Option<Vec<u8>>,
    pub http_request: Option<HttpRequest>,
}
``` [1](#0-0) 

The hand-written `Hash` implementation hashes only `canister_id`, `sender`, `method_name`, `ingress_expiry`, and either `http_request` or `arg`. `request_type` is never fed into the hasher:

```rust
impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.canister_id.hash(state);
        self.sender.hash(state);
        self.method_name.hash(state);
        self.ingress_expiry.hash(state);
        // request_type is absent
        if self.http_request.is_some() {
            self.http_request.hash(state);
        } else {
            self.arg.hash(state);
        }
    }
}
``` [2](#0-1) 

The `PartialEq` implementation has the same omission: [3](#0-2) 

`RequestContext` is used directly as the cache key (`Arc<RequestContext>`) by the boundary node's response cache: [4](#0-3) 

The cache bypasser allows three query variants through:

```rust
pub const fn is_query(&self) -> bool {
    matches!(self, Self::QueryV2 | Self::QueryV3 | Self::QuerySubnetV3)
}
``` [5](#0-4) 

Because `request_type` is absent from the hash and equality check, `QueryV2` and `QueryV3` requests that share the same `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg` produce **identical cache keys** and are considered **equal** by the cache.

---

### Impact Explanation

`QueryV2` (`/api/v2/canister/{id}/query`) returns a plain response with no node signature. `QueryV3` (`/api/v3/canister/{id}/query`) returns a certified response that includes a node-level BLS signature, allowing clients to cryptographically verify the result.

When the cache serves a `QueryV2` response for a `QueryV3` request:

1. The client receives an uncertified response body with no certificate.
2. The client's certification check either fails (if it enforces the certificate) or silently passes a forged/stale value (if it falls back to the uncertified body).
3. The attacker can control the cached value by being the first to issue the `QueryV2` request, then all `QueryV3` callers for the same canister/method/arg receive the attacker-chosen uncertified response for the duration of the cache TTL.

This breaks the core security guarantee of `QueryV3`: that query responses are node-signed and verifiable.

---

### Likelihood Explanation

The attack requires no privileges. Any anonymous user can:

1. Send a `POST /api/v2/canister/{canister_id}/query` with chosen parameters to populate the cache.
2. All subsequent `POST /api/v3/canister/{canister_id}/query` requests with the same parameters (same `sender`, `method_name`, `arg`, `ingress_expiry`) will receive the cached `QueryV2` response.

The nonce-bypass check (`ctx.nonce.is_some()`) is irrelevant here because the attacker's poisoning request does not need a nonce, and the victim's `QueryV3` request also typically has no nonce. [6](#0-5) 

---

### Recommendation

Add `request_type` to both the `Hash` and `PartialEq` implementations of `RequestContext`:

```rust
impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.request_type.hash(state);   // ← add this
        self.canister_id.hash(state);
        self.sender.hash(state);
        self.method_name.hash(state);
        self.ingress_expiry.hash(state);
        if self.http_request.is_some() {
            self.http_request.hash(state);
        } else {
            self.arg.hash(state);
        }
    }
}

impl PartialEq for RequestContext {
    fn eq(&self, other: &Self) -> bool {
        let r = self.request_type == other.request_type   // ← add this
            && self.canister_id == other.canister_id
            && self.sender == other.sender
            && self.method_name == other.method_name
            && self.ingress_expiry == other.ingress_expiry;
        if self.http_request.is_some() {
            r && self.http_request == other.http_request
        } else {
            r && self.arg == other.arg
        }
    }
}
```

`RequestType` already derives `Hash`, `PartialEq`, and `Eq`, so no additional changes are needed to that type. [7](#0-6)

### Proof of Concept

```
# Step 1 – attacker populates cache with a QueryV2 response
POST /api/v2/canister/ryjl3-tyaaa-aaaaa-aaaba-cai/query
Content-Type: application/cbor
<CBOR envelope: sender=anonymous, method_name=icrc1_balance_of,
                arg=<victim_account>, ingress_expiry=T, nonce=absent>

# Boundary node forwards to replica, receives plain (uncertified) response,
# stores it under key hash(canister_id, sender, method_name, ingress_expiry, arg).

# Step 2 – victim issues a QueryV3 request with identical parameters
POST /api/v3/canister/ryjl3-tyaaa-aaaaa-aaaba-cai/query
Content-Type: application/cbor
<identical CBOR envelope>

# Boundary node computes the same cache key (request_type excluded),
# finds a hit, and returns the QueryV2 response — no certificate included.
# Victim's client receives an uncertified balance value.
```

The cache key collision is confirmed by the `Hash` implementation at lines 94–108 of `rs/boundary_node/ic_boundary/src/routes.rs`, which never hashes `self.request_type`, and the bypasser at lines 74–86 of `rs/boundary_node/ic_boundary/src/http/middleware/cache.rs`, which allows both `QueryV2` and `QueryV3` into the cache.
