### Title
Empty `state_migration()` Stub Combined with No Reserved Enum Gap in `BorshableEngineState` Enables Permanent Fund Freeze on Upgrade - (File: `engine/src/lib.rs`, `engine/src/state.rs`)

### Summary

Aurora Engine's upgrade mechanism always calls `state_migration()` after deploying new WASM code, but the production `state_migration()` is an empty no-op stub. Simultaneously, the `BorshableEngineState` Borsh enum has no reserved gap variants. If a developer adds a new enum variant in a non-appended position during an upgrade, the Borsh discriminant mapping shifts, making all existing on-chain state unreadable. Because `get_state()` is called at the entry point of every contract method, this permanently freezes all user funds with no recovery path.

### Finding Description

**Root cause 1 — Empty `state_migration()` stub:** [1](#0-0) 

The `state_migration()` exported function is intentionally empty (`// TODO: currently we don't have migrations`). Both upgrade paths — `upgrade()` (which schedules `state_migration` as a batch promise action) and `deploy_upgrade()` (which calls `Runtime::self_deploy`, which also schedules `state_migration`) — always invoke this function after deploying new code. [2](#0-1) [3](#0-2) 

**Root cause 2 — No reserved gap in `BorshableEngineState`:** [4](#0-3) 

Borsh serializes Rust enums using a leading discriminant byte: `V1 = 0x00`, `V2 = 0x01`, `V3 = 0x02`. There are no reserved gap variants. If a developer inserts a new variant between existing ones (e.g., between V2 and V3), the discriminant for V3 shifts from `0x02` to `0x03`. All existing on-chain state was written with `0x02` as the V3 discriminant. The new code would attempt to deserialize `0x02` as the newly inserted variant, producing garbage or a deserialization error.

**Root cause 3 — Fallback deserialization only handles pre-enum legacy format:** [5](#0-4) 

The `try_from_slice` fallback path only handles the pre-versioned legacy `BorshableEngineStateV1` format (the format that predates the enum wrapper). It does **not** handle the case where the discriminant mapping has shifted due to a new variant insertion. There is no recovery path.

**Root cause 4 — `get_state()` is called in every contract method:** [6](#0-5) 

Every contract entrypoint calls `get_state()`. A deserialization failure returns `EngineStateError::DeserializationFailed`, which propagates as a contract panic, making every method — including `submit`, `ft_transfer`, `withdraw`, `deposit` — permanently unreachable.

### Impact Explanation

**Critical — Permanent freezing of funds.**

If `BorshableEngineState` discriminants shift due to a non-appended variant insertion during an upgrade, `get_state()` fails on every call. All EVM transactions, ETH deposits, NEP-141 withdrawals, and FT transfers become permanently impossible. All ETH and ERC-20 balances held by the Aurora contract are frozen with no on-chain recovery mechanism, since even the `upgrade()` method itself calls `get_state()` first: [7](#0-6) 

### Likelihood Explanation

**Medium.** The upgrade mechanism is owner-gated, but the structural flaw makes an honest developer mistake catastrophic. The history of the codebase shows that new variants have been added to `BorshableEngineState` before (V1→V2→V3), and the `state_migration()` stub with its `TODO` comment explicitly signals that future migrations are anticipated. The `CHANGES.md` records multiple past state-format changes. The absence of any reserved gap variants and the empty migration stub create a latent trap for the next upgrade cycle. [8](#0-7) 

### Recommendation

1. **Short term:** Add reserved gap variants to `BorshableEngineState` (e.g., `_Reserved4` through `_Reserved20`) so that future variants can be appended without shifting existing discriminants. Enforce append-only ordering with a comment analogous to the one already present in `BorshableTransactionKind`: [9](#0-8) 

2. **Short term:** Implement `state_migration()` as a real function that re-serializes existing state into the new format whenever a new `BorshableEngineState` variant is introduced.

3. **Long term:** Add a compile-time or test-time assertion that verifies the discriminant of the current "latest" variant matches the expected constant, so that accidental reordering is caught before deployment.

### Proof of Concept

1. Current on-chain state is stored as `[0x02, chain_id_bytes, owner_id_bytes, ...]` (V3 discriminant = 2).
2. Developer adds `V2_5(BorshableEngineStateV2_5)` between `V2` and `V3` in `BorshableEngineState`.
3. New code is compiled: V3 now has discriminant `0x03`.
4. Owner calls `upgrade()` with the new WASM. `state_migration()` runs but does nothing (empty stub).
5. Any subsequent call to any contract method invokes `get_state()`, which reads `[0x02, ...]` from storage and attempts to deserialize it as `V2_5`. Deserialization fails.
6. `get_state()` returns `EngineStateError::DeserializationFailed` → contract panics with `ERR_STATE_CORRUPTED`.
7. All user funds (ETH, ERC-20 tokens) are permanently frozen. Even `upgrade()` is blocked because it calls `get_state()` at line 174 of `admin.rs`. [4](#0-3) [5](#0-4) [10](#0-9)
