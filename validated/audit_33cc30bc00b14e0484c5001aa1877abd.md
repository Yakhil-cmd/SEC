### Title
Unchecked `u32` Accumulation in `pubdata_used_by_tx()` Enables Pubdata Undercharging via Integer Overflow — (File: `basic_system/src/system_implementation/flat_storage_model/storage_cache.rs`)

---

### Summary

The `calculate_pubdata_used_by_tx()` functions in both `storage_cache.rs` and `account_cache.rs` accumulate pubdata byte counts into a `u32` variable using plain `+=` without overflow guards. The combined result is returned by `pubdata_used_by_tx()` via an unchecked `u32 + u32` addition. This value feeds directly into the native-resource charging path (`net_pubdata_used()` → `get_resources_to_charge_for_pubdata()`). If a transaction generates more than `u32::MAX` (~4.3 GB) of pubdata, the counter wraps silently in release mode, causing the protocol to undercharge for pubdata. In debug builds or with `overflow-checks = true`, the same path panics, halting block processing entirely.

The `net_pubdata_used()` call ultimately reads the `u32` value from `pubdata_used_by_tx()` and widens it to `u64`. If the `u32` has already wrapped, the `u64` carries the wrapped (smaller) value, and the native-resource charge is computed against that smaller number.

A secondary truncation exists in `apply_storage_diffs_pubdata`:

```rust
let encoded_state_diffs_count =
    (self.storage_cache.net_diffs_iter().count() as u32).to_be_bytes();
``` [5](#0-4) 

A `usize → u32` truncating cast here would corrupt the pubdata encoding header if the diff count exceeds `u32::MAX`, making the block unprovable.

---

### Impact Explanation

| Build mode | Behavior | Protocol effect |
|---|---|---|
| Release (no overflow-checks) | `u32` wraps silently | `pubdata_used_by_tx()` returns a value far smaller than actual pubdata; native resource charge is proportionally reduced; attacker publishes gigabytes of state-diff pubdata while paying for only the wrapped remainder — **resource accounting bug / state-transition correctness violation** |
| Debug / `overflow-checks = true` | Rust panics at the `+=` site | Block processing aborts; no block can be finalized while the offending transaction is in the mempool — **DoS / valid-execution unprovability** |

Both paths are reachable from an unprivileged transaction sender with no privileged role required.

---

### Likelihood Explanation

Each SSTORE writes ~65 bytes of pubdata and costs ≥ 20 000 EVM gas. Overflowing `u32::MAX ≈ 4.3 × 10⁹` bytes requires ≈ 6.6 × 10⁷ SSTOREs and ≈ 1.3 × 10¹² gas. The theoretical `MAX_BLOCK_GAS_LIMIT = u64::MAX / ERGS_PER_GAS ≈ 7.2 × 10¹⁶` gas is far above this threshold. [6](#0-5) 

Likelihood is **low under typical operator-configured gas limits** (e.g., 30 M gas) but becomes **medium-to-high if the operator sets a permissive block gas limit** in the range of trillions of gas, which the codebase explicitly allows up to `MAX_BLOCK_GAS_LIMIT`. No external oracle, governance, or privileged key is required — only a sufficiently funded transaction sender and a high block gas limit.

---

### Recommendation

1. Change the return type of both `calculate_pubdata_used_by_tx()` functions and `pubdata_used_by_tx()` from `u32` to `u64`.
2. Replace all `pubdata_used += …` additions with `pubdata_used = pubdata_used.saturating_add(…)` or `checked_add(…).ok_or(InternalError)`.
3. Replace the `net_diffs_iter().count() as u32` cast with a checked conversion (`u32::try_from(…).map_err(…)?`) and propagate an error if the count exceeds `u32::MAX`.
4. Add a block-level invariant asserting that total pubdata per block fits in the chosen type before finalizing.

---

### Proof of Concept

1. Configure a ZKsync OS node with a block gas limit near `MAX_BLOCK_GAS_LIMIT` (e.g., 2 × 10¹² gas).
2. Deploy a contract whose fallback writes to `N` distinct storage slots (e.g., `N = 7 × 10⁷`).
3. Submit a single transaction calling that contract with enough gas. Each SSTORE produces ~65 bytes of pubdata; total pubdata ≈ 4.55 × 10⁹ bytes > `u32::MAX`.
4. In a release build: `calculate_pubdata_used_by_tx()` wraps to a small value (e.g., ~2.6 × 10⁸ bytes). `get_resources_to_charge_for_pubdata()` charges native resources for only ~2.6 × 10⁸ bytes instead of ~4.55 × 10⁹ bytes — a ~17× undercharge.
5. In a debug build or with `overflow-checks = true`: the `pubdata_used += 32` line panics, aborting block finalization.

    pub fn calculate_pubdata_used_by_tx(&self) -> u32 {
        let mut visited_elements = BTreeSet::new_in(self.0.alloc.clone());

        let mut pubdata_used = 0u32;
        for element_history in self.0.cache.iter_altered_since_commit() {
            // Elements are sorted chronologically

            let element_key = element_history.key();

            // we publish preimages for account details, so no need to publish hash
            if element_key.address == ACCOUNT_PROPERTIES_STORAGE_ADDRESS {
                continue;
            }

            // Skip if already calculated pubdata for this element
            if visited_elements.contains(element_key) {
                continue;
            }
            visited_elements.insert(element_key);

            let current_value = element_history.current().value();
            let initial_value = element_history.initial().value();
            let at_tx_start_value = element_history.committed().value();

            // If the current value is resetting to the initial one,
            // we don't consider this diff in the pubdata charging.
            // This change will be optimized away, so it's actually reducing
            // pubdata.
            if current_value == initial_value {
                continue;
            }

            if at_tx_start_value != current_value {
                // TODO(EVM-1074): use tree index instead of key for repeated writes
                pubdata_used += 32; // key
                pubdata_used += ValueDiffCompressionStrategy::optimal_compression_length(
                    at_tx_start_value,
                    current_value,
                ) as u32;
            }
        }

        pubdata_used
    }

        let mut pubdata_used = 0u32;
        for element_history in self.cache.iter_altered_since_commit() {
            // Elements are sorted chronologically

            // Skip if already calculated pubdata for this element

                continue;
            }
            visited_elements.insert(element_key);

            let current = element_history.current();
            let initial = element_history.initial();

            // we don't consider this diff in the pubdata charging.

            // pubdata.

                    at_tx_start.value(),
                    current.value(),
                    current.metadata().not_publish_bytecode,
                    current.metadata().not_compress_balance,
                )

            }
        }
