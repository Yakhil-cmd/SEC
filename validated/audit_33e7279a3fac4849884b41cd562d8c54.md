Audit Report

## Title
ckDOGE Minter Never Schedules `ConsolidateUtxos` Task, Enabling UTXO Fragmentation DoS on Large Withdrawals — (File: `rs/dogecoin/ckdoge/minter/src/main.rs`)

## Summary
ckDOGE's `setup_tasks()` omits `schedule_now(TaskType::ConsolidateUtxos, ...)` that ckBTC includes, meaning the hourly UTXO consolidation loop is never bootstrapped. Because the `TASKS` queue is a thread-local that starts empty and is only populated via explicit `schedule_now`/`schedule_after` calls, `consolidate_utxos()` is never invoked at runtime. As the minter's `available_utxos` set grows monotonically from deposits, large withdrawals will eventually fail with `BuildTxError::InvalidTransaction(TooManyInputs)` once the UTXO set requires more than `max_num_inputs_in_transaction` (500) inputs to cover a withdrawal amount.

## Finding Description
`rs/dogecoin/ckdoge/minter/src/main.rs` `setup_tasks()` (lines 46–49) schedules only `ProcessLogic` and `RefreshFeePercentiles`:

```rust
fn setup_tasks() {
    schedule_now(TaskType::ProcessLogic, &DOGECOIN_CANISTER_RUNTIME);
    schedule_now(TaskType::RefreshFeePercentiles, &DOGECOIN_CANISTER_RUNTIME);
}
```

The ckBTC equivalent at `rs/bitcoin/ckbtc/minter/src/main.rs` (lines 54–58) additionally calls `schedule_now(TaskType::ConsolidateUtxos, &IC_CANISTER_RUNTIME)`, which bootstraps the hourly self-rescheduling consolidation loop.

The `TASKS` queue is defined as a `thread_local! { static TASKS: RefCell<TaskQueue> = RefCell::default(); }` in `rs/bitcoin/ckbtc/minter/src/tasks.rs` (lines 13–16). It starts empty and is only populated via `schedule_now`/`schedule_after`. Without the initial `schedule_now`, `TaskType::ConsolidateUtxos` is never inserted, so `consolidate_utxos()` in `rs/bitcoin/ckbtc/minter/src/lib.rs` (lines 1431–1548) is never called. The self-rescheduling guard in `run_task` (lines 167–189 of `tasks.rs`) only fires if the task is first enqueued.

Every `update_balance` call for a new deposit inserts a UTXO into `available_utxos`. With no consolidation, this set grows monotonically. When `retrieve_doge_with_approval` is called, `build_unsigned_transaction` enforces `max_num_inputs_in_transaction = 500` (defined at `rs/dogecoin/ckdoge/minter/src/lib.rs` line 49). If covering the requested withdrawal amount requires more than 500 inputs, the transaction build fails with `TooManyInputs`. The `utxo_consolidation_threshold` init parameter and the full consolidation infrastructure (`ConsolidateUtxosRequest`, `CreatedConsolidateUtxosRequest`, `select_utxos_to_consolidate`) are present but unreachable.

## Impact Explanation
Large withdrawals (ckDOGE → DOGE) fail with `TooManyInputs` when the minter's UTXO set is sufficiently fragmented. Users whose withdrawal requests fail are reimbursed, so funds are not permanently lost. However, the minter becomes unable to service large withdrawals until a canister upgrade adds the missing `schedule_now` call. This matches the **High** bounty impact: "Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS" and "Significant Chain Fusion, ck-token... security impact with concrete user or protocol harm."

## Likelihood Explanation
No attacker is required. Organic usage alone (many small legitimate deposits) will naturally fragment the UTXO set over time, exactly as happened with ckBTC in production (60k UTXOs documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_12_12.md` lines 19–23). A deliberate attacker can accelerate fragmentation by depositing many small amounts above `deposit_doge_min_amount`. The condition is inevitable under normal growth and requires no special privileges to trigger.

## Recommendation
Add `schedule_now(TaskType::ConsolidateUtxos, &DOGECOIN_CANISTER_RUNTIME);` to ckDOGE's `setup_tasks()` function in `rs/dogecoin/ckdoge/minter/src/main.rs`, mirroring the ckBTC implementation. This will bootstrap the hourly consolidation loop and make the already-configured `utxo_consolidation_threshold` parameter effective.

## Proof of Concept
A state-machine test can:
1. Initialize the ckDOGE minter with `utxo_consolidation_threshold = 10_000`.
2. Deposit 501+ UTXOs each worth exactly `deposit_doge_min_amount` via `update_balance`.
3. Assert `available_utxos.len() > max_num_inputs_in_transaction` (500) and that no `ConsolidateUtxos` task is present in the `TASKS` thread-local queue.
4. Call `retrieve_doge_with_approval` for an amount requiring all UTXOs.
5. Assert the result is `Err(BuildTxError::InvalidTransaction(TooManyInputs { .. }))`.
