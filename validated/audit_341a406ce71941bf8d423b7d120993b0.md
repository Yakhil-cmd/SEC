### Title
Missing Cumulative Block Gas Limit Enforcement in Ethereum Block Flow - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/loop_op.rs`)

### Summary
The Ethereum block flow in ZKsync OS validates each transaction's gas limit individually against the block gas limit, but never checks whether the running total of gas consumed across all transactions in a block exceeds the block gas limit. A malicious oracle/prover input can supply multiple transactions that each individually pass validation yet collectively push `block_gas_used` far above `block_gas_limit`, producing an invalid state transition that the system accepts without error.

### Finding Description

In `generic_loop_op` (`loop_op.rs`), after a transaction succeeds, the result is handed directly to `result_keeper.tx_processed(Ok(...))` and `system.finish_valid_tx()` with no post-execution block-level gas check: [1](#0-0) 

The only gas-related validation occurs at transaction validation time in `validate_and_compute_fee_for_transaction`, which checks that the individual transaction's gas limit does not exceed the block gas limit: [2](#0-1) 

After execution, `EthereumBasicTransactionDataKeeper::record_transaction_results` simply accumulates gas with no limit guard: [3](#0-2) 

**Contrast with the ZK block flow**, which correctly computes the post-addition total and passes it to `check_for_block_limits` before committing the transaction: [4](#0-3) 

`check_for_block_limits` enforces `gas_used > system.get_gas_limit()` as the rejection condition: [5](#0-4) 

The Ethereum path has no equivalent post-execution accumulation check.

### Impact Explanation

A malicious oracle/prover can supply a sequence of transactions where each individual `tx_gas_limit ≤ block_gas_limit` (passing per-transaction validation), but the sum of their actual `gas_used` values exceeds `block_gas_limit`. The Ethereum block flow accepts all of them, producing a block header with `gas_used > gas_limit`. This is an invalid Ethereum block by consensus rules. The resulting state transition is incorrect, and any proof generated over it attests to an invalid execution.

### Likelihood Explanation

In ZKsync OS proving mode the oracle is a non-determinism source supplied by the prover. A malicious prover controlling oracle input can craft a block with two or more transactions each claiming `gas_limit = block_gas_limit` and each consuming that full limit. Both pass the per-transaction check; neither triggers a block-level rejection. No privileged key or governance majority is required — only control over the oracle data fed into the RISC-V execution.

### Recommendation

After each transaction completes in `generic_loop_op`, compute the prospective cumulative gas and reject (or roll back) the transaction if it would exceed the block gas limit, mirroring the ZK path:

```rust
let next_block_gas_used = block_data_keeper.block_gas_used()
    .checked_add(tx_gas_used)
    .ok_or(/* overflow error */)?;
if next_block_gas_used > system.get_gas_limit() {
    // revert and report BlockGasLimitReached
}
block_data_keeper.commit_gas(next_block_gas_used);
```

### Proof of Concept

1. Set `block_gas_limit = 30_000_000`.
2. Provide oracle data containing two transactions, each with `gas_limit = 30_000_000` and each consuming `30_000_000` gas.
3. **Tx 1 validation**: `30_000_000 ≤ 30_000_000` → passes. Execution uses `30_000_000` gas. `block_gas_used = 30_000_000`.
4. **Tx 2 validation**: `30_000_000 ≤ 30_000_000` → passes. Execution uses `30_000_000` gas. `block_gas_used = 60_000_000`.
5. No check fires. The block header is sealed with `gas_used = 60_000_000 > gas_limit = 30_000_000`.
6. ZKsync OS produces a proof over this invalid state transition; the block header fails Ethereum consensus verification.

fn check_for_block_limits<S: EthereumLikeTypes>(
    system: &mut System<S>,
    gas_used: u64,
    computational_native_used: u64,
    pubdata_used: u64,
    logs_used: u64,
    blob_gas_used: u64,
) -> Result<(), InvalidTransaction>
where
    S::IO: IOSubsystemExt,
    <S as SystemTypes>::Metadata: ZkSpecificPricingMetadata,
{
    if gas_used > system.get_gas_limit() {
        system_log!(
            system,
            "Block gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockGasLimitReached)
    } else if blob_gas_used > system.get_blob_gas_limit() {
        system_log!(
            system,
            "Block blob gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockBlobGasLimitReached)
    } else if !cfg!(feature = "resources_for_tester")
        && computational_native_used > MAX_NATIVE_COMPUTATIONAL
    {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block native limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockNativeLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && pubdata_used > system.get_pubdata_limit() {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block pubdata limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockPubdataLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block logs limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
    } else {
        Ok(())
    }
}
