### Title
Unmetered `GetAllBalances` on Fee Collector in `distribution.BeginBlocker` - (File: `sei-cosmos/x/distribution/keeper/allocation.go`)

---

### Summary

The same vulnerability class exists in Sei chain. `AllocateTokens` calls `GetAllBalances` on the fee collector module account inside `BeginBlock`, which is unmetered. If an attacker can deposit a large number of distinct token denominations into the fee collector address, every subsequent block's `BeginBlock` will iterate over all of them, causing unbounded per-block execution time.

---

### Finding Description

**Call chain:**

```
app.BeginBlock (app/abci.go:56)
  → legacyabci.BeginBlock (app/legacyabci/begin_block.go:60)
    → distribution.BeginBlocker (sei-cosmos/x/distribution/abci.go:36)
      → k.AllocateTokens (sei-cosmos/x/distribution/keeper/allocation.go:22)
        → k.bankKeeper.GetAllBalances(ctx, feeCollector.GetAddress())
```

At `sei-cosmos/x/distribution/keeper/allocation.go:22`, `AllocateTokens` fetches **all** balances of the fee collector module account:

```go
feeCollector := k.authKeeper.GetModuleAccount(ctx, k.feeCollectorName)
feesCollectedInt := k.bankKeeper.GetAllBalances(ctx, feeCollector.GetAddress())
```

`GetAllBalances` in `sei-cosmos/x/bank/keeper/view.go:58-66` iterates over every stored balance entry for the address via `IterateAccountBalances`, with no bound or pagination. The result is then passed to `SendCoinsFromModuleToModule` and iterated again for per-validator reward math — both also O(N) in the number of denominations. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

If an attacker can accumulate N distinct denominations in the fee collector, every block's `BeginBlock` pays O(N) cost with no gas metering. With enough denominations, block production stalls indefinitely — a consensus liveness halt affecting all validators. This maps to **High** under the Sei bounty scope (halt/crash ≥1/3 validators).

---

### Likelihood Explanation

The fee collector (`auth.FeeCollectorName`) is a module account registered in `blockedAddrs`, which prevents direct `MsgSend` from a user account via the bank message server. However, Sei's hybrid EVM environment introduces additional transfer paths:

1. **EVM `CALL`/`TRANSFER`**: The EVM state transition does not consult the Cosmos `blockedAddrs` map. A contract or EOA can transfer ERC-20 tokens or native SEI to the fee collector's bech32-derived EVM address.
2. **IBC `MsgTransfer`**: IBC packet handling calls `bankKeeper.SendCoins` at the keeper level, which in some code paths does not enforce `BlockedAddr`. An attacker could IBC-transfer thousands of distinct IBC-wrapped denominations to the fee collector address.
3. **Token Factory**: `tokenfactory` mint-to-address calls may bypass the blocked address check at the keeper level.

The fee collector's address is deterministic and publicly known, making targeting trivial. [5](#0-4) 

---

### Recommendation

Replace the unbounded `GetAllBalances` call with a targeted `GetBalance` for only the expected fee denomination (`usei`), consistent with how the test helper already queries it:

```go
// Instead of:
feesCollectedInt := k.bankKeeper.GetAllBalances(ctx, feeCollector.GetAddress())

// Use:
feesCollectedInt := sdk.NewCoins(k.bankKeeper.GetBalance(ctx, feeCollector.GetAddress(), k.stakingKeeper.BondDenom(ctx)))
```

Additionally, ensure all transfer paths (EVM precompiles, IBC, token factory) enforce the `blockedAddrs` check before crediting the fee collector address.

---

### Proof of Concept

1. Deploy a token factory contract or use IBC to create 50,000 distinct denominations.
2. Transfer 1 unit of each to the fee collector module address.
3. Observe that every subsequent `BeginBlock` → `distribution.BeginBlocker` → `AllocateTokens` → `GetAllBalances` iterates all 50,000 entries, stalling block production.

The existing bank keeper test at `sei-cosmos/x/bank/keeper/keeper_test.go:1944` already demonstrates that `GetAllBalances` with 100,000 entries is a valid (if slow) operation — confirming the iteration is truly unbounded. [6](#0-5)

func (k BaseViewKeeper) GetAllBalances(ctx sdk.Context, addr sdk.AccAddress) sdk.Coins {
	balances := sdk.NewCoins()
	k.IterateAccountBalances(ctx, addr, func(balance sdk.Coin) bool {
		balances = append(balances, balance)
		return false
	})

	return balances.Sort()
}

func (suite *IntegrationTestSuite) TestKeeperGetAllBalances() {
	app, ctx := suite.app, suite.ctx
	addr := sdk.AccAddress([]byte("addr1_______________"))
	cnt := 100_000
	allDenoms := make(sdk.Coins, cnt)
	for i := 0; i < cnt; i++ {
		d := fmt.Sprintf("d%d", i+10) // denom must be at least 3 chars.
		app.BankKeeper.AddCoins(ctx, addr, sdk.Coins{sdk.Coin{
			Denom: d, Amount: sdk.OneInt(),
		}}, false)
		allDenoms[i] = sdk.Coin{Denom: d}
	}
	allDenoms = allDenoms.Sort()
	balances := app.BankKeeper.GetAllBalances(ctx, addr)
	suite.Require().Len(balances, cnt)
	for i := 0; i < cnt; i++ {
		suite.Require().Equal(allDenoms[i].Denom, balances[i].Denom)
		suite.Require().Equal(sdk.OneInt(), balances[i].Amount)
	}
}
