### Title
Attacker-Controlled `refund_address` Can Permanently Freeze ETH in `exit_to_near` Precompile - (File: `engine-precompiles/src/native.rs`, `engine/src/engine.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` feature is enabled, the `exit_to_near` precompile accepts a user-supplied `refund_address` directly from calldata. If the NEAR-side `ft_transfer` promise fails, Aurora attempts to refund ETH to that address via an EVM call. If `refund_address` is a contract that reverts on ETH receipt, the refund EVM call returns a non-ok `TransactionStatus`, causing `exit_to_near_precompile_callback` to return `ERR_REFUND_FAILURE`. The ETH, already debited from the user's EVM balance and held at the `exit_to_near` precompile address, is permanently frozen there with no recovery path.

---

### Finding Description

**Step 1 — User-controlled `refund_address` parsed from raw calldata.**

In `engine-precompiles/src/native.rs`, `parse_input` extracts bytes `[1..21]` of the calldata as the `refund_address` with no validation that the address can receive ETH:

```rust
// engine-precompiles/src/native.rs:779-784
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))
}
```

This `refund_address` is stored in `BaseTokenParams` / `Erc20TokenParams` and forwarded into `RefundCallArgs::recipient_address`. [1](#0-0) 

**Step 2 — ETH is transferred to the precompile address during the original EVM execution.**

When a user calls the `exit_to_near` precompile with ETH (`apparent_value > 0`), the EVM transfers that ETH from the caller's balance to the `exit_to_near::ADDRESS`. A NEAR `ft_transfer` promise is then scheduled, with `exit_to_near_precompile_callback` as the callback. [2](#0-1) 

**Step 3 — On `ft_transfer` failure, `refund_on_error` pushes ETH to `refund_address` via EVM call.**

In `engine/src/engine.rs`, `refund_on_error` constructs a new EVM engine and calls `engine.call()` targeting `refund_address` with the full ETH amount:

```rust
// engine/src/engine.rs:1204-1223
} else {
    // ETH exit; transfer ETH back from precompile address
    let exit_address = exit_to_near::ADDRESS;
    ...
    engine.call(
        &exit_address,
        &refund_address,   // ← user-controlled
        amount,
        Vec::new(),
        ...
    )
}
```

If `refund_address` is a contract whose `receive()` / `fallback()` reverts, the EVM call returns `TransactionStatus::Revert`. [3](#0-2) 

**Step 4 — A non-ok refund status causes `ERR_REFUND_FAILURE`, permanently freezing the ETH.**

In `exit_to_near_precompile_callback`:

```rust
// engine/src/contract_methods/connector.rs:231-237
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

    if !refund_result.status.is_ok() {
        return Err(errors::ERR_REFUND_FAILURE.into());
    }
    ...
}
```

`TransactionStatus::Revert` is not ok, so `ERR_REFUND_FAILURE` is returned. The NEAR callback panics. NEAR callback failure rolls back only the callback's own state changes — it does **not** roll back the original `submit` call's EVM state (where ETH was debited from the user and credited to the precompile address). The ETH remains permanently at `exit_to_near::ADDRESS` with no recovery mechanism. [4](#0-3) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

ETH sent through the `exit_to_near` precompile is irrecoverably locked at the precompile address whenever the refund EVM call fails. There is no admin function, no fallback path, and no second-chance mechanism to drain the precompile address. Any ETH frozen there is permanently removed from circulation within the Aurora EVM. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium.**

The attacker must:
1. Deploy a contract that can toggle between accepting and rejecting ETH (trivial — a single storage flag controls a `receive()` revert).
2. Call `exit_to_near` with ETH, supplying the togglable contract as `refund_address` (bytes 1–20 of calldata).
3. Specify an unregistered or invalid NEAR account as the recipient so that `ft_transfer` fails (straightforward).
4. Flip the contract to revert before the NEAR callback executes.

All four steps are within reach of any unprivileged EVM user. The `error_refund` feature must be compiled in (it is the production configuration, as evidenced by the `MIN_INPUT_SIZE` constant being 21 bytes when the feature is active). [6](#0-5) 

---

### Recommendation

Replace the push-payment refund with a pull-payment pattern: instead of calling `engine.call()` to push ETH to `refund_address`, credit the ETH balance back to `refund_address` directly in the EVM state (i.e., add to the address's balance without executing its code). This is equivalent to wrapping the refund amount and crediting the EVM balance entry, which eliminates any dependency on the recipient's `receive()` / `fallback()` behavior. Alternatively, validate at precompile entry time that `refund_address` has no deployed code (EOA-only restriction), though this is a weaker mitigation.

---

### Proof of Concept

```
1. Attacker deploys MaliciousRefund on Aurora EVM:
   - bool public revert_on_receive = false;
   - receive() external payable { require(!revert_on_receive); }
   - function flip() external { revert_on_receive = true; }

2. Attacker constructs exit_to_near calldata (error_refund feature active):
   input = 0x00                          // flag: ETH exit
         | MaliciousRefund_address (20B) // refund_address
         | "unregistered.near"           // NEAR recipient (not registered → ft_transfer will fail)

3. Attacker calls Aurora's `submit` with an EVM tx that calls exit_to_near::ADDRESS
   with value = V ETH and the above input.
   → EVM debits V ETH from attacker, credits exit_to_near::ADDRESS.
   → NEAR ft_transfer promise is scheduled.

4. Before the NEAR callback fires, attacker calls MaliciousRefund.flip().

5. ft_transfer fails (unregistered account).
   exit_to_near_precompile_callback fires:
   → refund_on_error calls engine.call(exit_to_near::ADDRESS → MaliciousRefund, V ETH)
   → MaliciousRefund.receive() reverts → TransactionStatus::Revert
   → !refund_result.status.is_ok() == true
   → ERR_REFUND_FAILURE returned, callback panics.

6. V ETH is permanently frozen at exit_to_near::ADDRESS.
   Attacker's EVM balance is permanently reduced by V ETH.

#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}

) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =

            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
}
