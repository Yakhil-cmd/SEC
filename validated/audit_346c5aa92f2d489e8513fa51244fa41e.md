Audit Report

## Title
Unprivileged Caller Can Permanently Censor Victim's ckBTC Deposits via `update_balance` Owner Front-Running - (File: rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs)

## Summary
The `update_balance` endpoint accepts an `owner: Option<Principal>` argument and constructs the concurrency guard key from the caller-supplied owner rather than the authenticated caller. Any unprivileged principal can continuously call `update_balance` with a victim's principal as `owner`, holding the victim's per-account guard for the full async round-trip to the Bitcoin canister and returning `AlreadyProcessing` to every legitimate call the victim makes. The victim's BTC deposits become permanently inaccessible as ckBTC.

## Finding Description
`UpdateBalanceArgs` exposes `owner` as an optional public Candid field with no access restriction:

```rust
pub struct UpdateBalanceArgs {
    pub owner: Option<Principal>,   // any caller can set this
    pub subaccount: Option<Subaccount>,
}
``` [1](#0-0) 

Inside `update_balance`, the account used to acquire the concurrency guard is derived from the attacker-controlled `args.owner`:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
``` [2](#0-1) 

The guard inserts the account into the global `update_balance_accounts` `BTreeSet` and is held for the entire async `get_utxos` round-trip: [3](#0-2) 

While the guard is held, any concurrent call for the same account fails immediately:

```rust
if accounts.contains(&account) {
    return Err(GuardError::AlreadyProcessing);
}
``` [4](#0-3) 

The guard is only released on `Drop`: [5](#0-4) 

The sole existing check only prevents the minter from updating its own balance — it does not prevent a caller from specifying any other arbitrary principal: [6](#0-5) 

`MAX_CONCURRENT` is 100, meaning an attacker can simultaneously lock up to 100 distinct victim accounts: [7](#0-6) 

## Impact Explanation
**High ($2,000–$10,000).** A victim who has sent BTC to their ckBTC deposit address cannot call `update_balance` to mint ckBTC. Their on-chain Bitcoin funds are permanently inaccessible as ckBTC tokens for as long as the attacker sustains the attack. This constitutes a concrete, targeted application-level DoS against ck-token financial infrastructure with direct user fund impact, matching the allowed impact class: *"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."*

## Likelihood Explanation
**Medium.** The attack requires no special privileges — any ingress sender or canister can call `update_balance` with an arbitrary `owner`. The victim's principal is publicly observable from their ckBTC ledger history or prior on-chain activity. The cost is only the cycles for repeated inter-canister calls to the Bitcoin canister. The attacker simply re-calls immediately after each guard release. Large depositors or protocols are natural high-value targets.

## Recommendation
Restrict the effective owner to the authenticated caller, removing the ability to acquire a guard on behalf of another principal:

```rust
let caller_account = Account {
    owner: caller,  // always use caller, ignore args.owner
    subaccount: args.subaccount,
};
```

If third-party deposit notification is a required feature (e.g., a smart contract depositing on behalf of a user), introduce a separate non-locking notification path or require the owner to pre-authorize the caller via an on-chain allowlist.

## Proof of Concept
1. Victim `V` sends BTC to their ckBTC deposit address and waits for confirmations.
2. Attacker `A` (any unprivileged principal) calls `update_balance({ owner: Some(V), subaccount: None })`.
3. The minter constructs `caller_account = { owner: V, subaccount: None }` and inserts it into `update_balance_accounts` via `balance_update_guard`.
4. The minter makes an async `get_utxos` call to the Bitcoin canister for V's derived address. The guard is held during this round-trip.
5. Victim `V` calls `update_balance({ owner: None, subaccount: None })`. The minter constructs the same `caller_account = { owner: V, subaccount: None }`, finds it already in `update_balance_accounts`, and returns `UpdateBalanceError::AlreadyProcessing`.
6. Attacker `A`'s call completes (with `NoNewUtxos` since A has no UTXOs at V's address). The guard is released.
7. Attacker `A` immediately re-calls step 2. Steps 3–7 repeat indefinitely.
8. Victim `V` can never successfully call `update_balance`; their BTC deposit is permanently inaccessible as ckBTC.

A deterministic integration test using PocketIC can reproduce this by: (a) initializing the minter, (b) submitting an attacker call with `owner: Some(victim)` and suspending it mid-async, (c) submitting a victim call and asserting it returns `AlreadyProcessing`.
