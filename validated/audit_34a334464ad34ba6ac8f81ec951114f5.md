### Title
ckERC20 Minter Mints Unbacked Tokens for Fee-on-Transfer ERC20 Deposits - (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol` / `DepositHelperWithSubaccount.sol`)

### Summary

The ckERC20 deposit helper smart contracts emit a deposit event using the caller-supplied `amount` argument rather than the actual tokens received by the minter address. For fee-on-transfer ERC20 tokens, the minter receives `amount - fee` on-chain but the IC minter canister reads `amount` from the log and mints that full amount of ckERC20. This permanently inflates the ckERC20 supply beyond the ERC20 collateral held, breaking the 1:1 backing invariant.

### Finding Description

Both deposit helper contracts perform the same two-step pattern:

**`ERC20DepositHelper.sol` (`deposit`):** [1](#0-0) 

**`DepositHelperWithSubaccount.sol` (`depositErc20`):** [2](#0-1) 

Step 1: `safeTransferFrom(msg.sender, minterAddress, amount)` — for a fee-on-transfer ERC20, the minter address receives `amount - fee`, not `amount`.

Step 2: The event is emitted with the original caller-supplied `amount`, not the actual balance delta.

The IC minter canister scrapes these logs and parses the `value` field directly from the event data: [3](#0-2) 

It then mints exactly `event.value()` ckERC20 tokens to the user: [4](#0-3) 

The `value` field in `ReceivedErc20Event` is populated directly from the log data, which is the pre-fee `amount`: [5](#0-4) 

### Impact Explanation

For every deposit of a fee-on-transfer ERC20 token, the minter mints `fee` more ckERC20 than the ERC20 collateral it actually holds. The ckERC20 supply grows unbacked by exactly the cumulative transfer fees. When users later withdraw ckERC20 back to ERC20, the minter's Ethereum address will eventually lack sufficient ERC20 balance to fulfill all withdrawals. The last withdrawers lose funds proportional to the total accumulated fee inflation. This is a direct ledger conservation break: the total ckERC20 supply exceeds the ERC20 held by the minter.

### Likelihood Explanation

Any fee-on-transfer ERC20 token added to the supported ckERC20 token list triggers this. The attacker path requires only a normal unprivileged deposit call — no special role, no key compromise, no governance majority. The user simply calls `depositErc20` with a supported fee-on-transfer token. PAXG is a known real-world fee-on-transfer token. The vulnerability is triggered automatically on every deposit of such a token, with no additional attacker action required beyond the deposit itself.

### Recommendation

In both helper contracts, measure the minter's actual balance change around the `transferFrom` call and emit the event with the measured delta instead of the caller-supplied `amount`:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

Apply the same pattern to `ERC20DepositHelper.sol`'s `deposit` function.

### Proof of Concept

1. A fee-on-transfer ERC20 token (e.g., PAXG, which charges a 0.02% fee on every transfer) is added to the supported ckERC20 tokens list via NNS proposal.
2. User calls `depositErc20(paxgAddress, 1_000_000, principal, subaccount)` on `DepositHelperWithSubaccount`.
3. `safeTransferFrom` executes: minter receives `999_800` PAXG (after 200 token fee).
4. Event emitted: `ReceivedEthOrErc20(paxgAddress, user, 1_000_000, principal, subaccount)` — using the original `1_000_000`.
5. IC minter scrapes the log, reads `value = 1_000_000` from event data at [3](#0-2) , and mints `1_000_000` ckPAXG to the user.
6. Minter holds only `999_800` PAXG on Ethereum but has issued `1_000_000` ckPAXG.
7. After N such deposits, the ckPAXG supply exceeds the PAXG collateral by `N × 200`. The last `N × 200` ckPAXG units are unbacked and the corresponding withdrawal requests will fail.
