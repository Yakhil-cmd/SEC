### Title
Silo Whitelist Completely Bypassed via `deploy_code` and `call` Entrypoints - (File: engine/src/contract_methods/evm_transactions.rs)

### Summary
Aurora Engine's Silo mode exposes four whitelist kinds (`Admin`, `EvmAdmin`, `Account`, `Address`) to restrict who may deploy EVM code and submit transactions. However, the `deploy_code` and `call` NEAR-level entrypoints perform **no whitelist check**, allowing any unprivileged NEAR account to deploy arbitrary EVM contracts and call any EVM contract in the Silo, completely bypassing the access-control model.

### Finding Description

The Silo whitelist is enforced only inside `engine::submit()` via `assert_access()`: [1](#0-0) 

`assert_access` dispatches to `silo::is_allow_deploy` (when `transaction.to` is `None`) or `silo::is_allow_submit` (when it is `Some`): [2](#0-1) 

Each of those checks the relevant `Whitelist` storage entry: [3](#0-2) 

The `submit` path correctly calls `assert_access` before executing: [4](#0-3) 

However, the `deploy_code` entrypoint in `evm_transactions.rs` only calls `require_running` and then immediately invokes `engine.deploy_code_with_input` — **no whitelist check is present**: [5](#0-4) 

Likewise, the `call` entrypoint performs no whitelist check: [6](#0-5) 

Both are exposed as public NEAR contract methods: [7](#0-6) 

### Impact Explanation

In Silo mode, the operator enables the `Admin`/`EvmAdmin` whitelists to restrict EVM code deployment to trusted parties, and the `Account`/`Address` whitelists to restrict transaction submission. Because `deploy_code` and `call` bypass all four whitelists, any NEAR account — regardless of whitelist status — can:

1. Deploy arbitrary EVM contracts into the Silo (bypassing `Admin`/`EvmAdmin` whitelists).
2. Call any EVM contract in the Silo (bypassing `Account`/`Address` whitelists).

This renders the entire Silo access-control model ineffective. An attacker can deploy a malicious EVM contract and interact with existing Silo contracts (e.g., ERC-20 tokens, DeFi protocols) as their NEAR-derived EVM address, enabling unauthorized fund movement and state manipulation. The impact is **High**: unauthorized access to a permissioned EVM environment, with a realistic path to theft of user funds held in Silo EVM contracts.

### Likelihood Explanation

Likelihood is **High**. The `deploy_code` and `call` methods are standard, publicly documented NEAR contract entrypoints. Any NEAR account can call them directly without any special knowledge or privilege. No signature, key, or whitelist membership is required. The bypass requires only a single NEAR transaction.

### Recommendation

Add a Silo whitelist check at the start of both `deploy_code` and `call` in `engine/src/contract_methods/evm_transactions.rs`, mirroring the `assert_access` logic already used in `engine::submit`. Specifically:

- In `deploy_code`: call `silo::is_allow_deploy(io, predecessor_account_id, predecessor_address)` and return `ERR_NOT_ALLOWED` if it returns `false`.
- In `call`: call `silo::is_allow_submit(io, predecessor_account_id, predecessor_address)` and return `ERR_NOT_ALLOWED` if it returns `false`.

### Proof of Concept

1. Deploy Aurora Engine in Silo mode with all four whitelists enabled (Admin, EvmAdmin, Account, Address).
2. Do **not** add the attacker's NEAR account or EVM address to any whitelist.
3. Confirm that calling `submit` with a deployment transaction from the attacker's account fails with `NotAllowed`.
4. Now call `deploy_code` directly from the same attacker NEAR account, passing arbitrary EVM bytecode as input.
5. Observe that the deployment succeeds — the whitelist is not consulted.
6. Similarly, call `call` directly from the attacker's NEAR account targeting any existing Silo EVM contract.
7. Observe that the call succeeds — the whitelist is not consulted.

The root cause is confirmed at: [8](#0-7) 
where `require_running` is the only guard, and no analog to `assert_access` exists.

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

    /// Check if the whitelist is enabled.
    pub fn is_enabled(&self) -> bool {
        // White list is disabled by default. So return `false` if the key doesn't exist.
        let key = self.key(STATUS);
        self.io
            .read_storage(&key)
            .is_some_and(|value| value.to_vec() == [1])
    }

    fn key(&self, value: &[u8]) -> Vec<u8> {
        let mut bytes = Vec::with_capacity(1 + value.len());

        bytes.push(u8::from(self.kind));
        bytes.extend_from_slice(value);
        bytes_to_key(KeyPrefix::Whitelist, &bytes)
    }

    /// Add a new element to the whitelist.
    pub fn add<A: AsBytes + ?Sized>(&mut self, element: &A) {
        let key = self.key(element.as_bytes());
        self.io.write_storage(&key, &[]);
    }

    /// Remove a new element from the whitelist.
    pub fn remove<A: AsBytes + ?Sized>(&mut self, element: &A) {
        let key = self.key(element.as_bytes());
        self.io.remove_storage(&key);
    }

    /// Check if the element is present in the whitelist.
    pub fn is_exist<A: AsBytes + ?Sized>(&self, element: &A) -> bool {
        let key = self.key(element.as_bytes());
        self.io.storage_has_key(&key)
    }

#[named]
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}
