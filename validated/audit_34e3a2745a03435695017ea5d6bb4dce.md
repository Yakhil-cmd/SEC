### Title
wNEAR Permanently Frozen in Engine Implicit Address When XCC Router Deployment Fails - (`engine/src/xcc.rs`, `engine-precompiles/src/xcc.rs`)

### Summary

The XCC precompile irrevocably transfers wNEAR from the user's EVM balance to the engine's implicit address before the NEAR promise chain executes. If the subsequent router deployment batch fails, the promise chain aborts without refunding the wNEAR, leaving it permanently frozen with no recovery path.

### Finding Description

The XCC precompile flow in `handle_precompile_promise` (`engine/src/xcc.rs`) constructs a multi-step NEAR promise chain:

1. **Deploy batch** (if `create_needed`): `CreateAccount` + `Transfer(STORAGE_AMOUNT)` + `DeployContract` + `FunctionCall("initialize")`
2. **Callback**: `factory_update_address_version` — updates the engine's version record
3. **Callback**: `withdraw_wnear_to_router` — burns wNEAR from the engine's implicit address and sends NEAR to the router
4. **Callback**: `send_refund` (if new account) — returns storage staking NEAR to the engine
5. **Callback**: User's actual XCC call

**Before this chain is even scheduled**, the XCC precompile (`engine-precompiles/src/xcc.rs` lines 184–216) executes an EVM `transferFrom` call that moves `required_near` worth of wNEAR from the user's EVM address to the engine's implicit address. This EVM state change is committed as part of the EVM transaction and is **not rolled back** if the subsequent NEAR promise chain fails. [1](#0-0) 

When `create_needed = true`, `required_near = attached_near + STORAGE_AMOUNT` (at minimum 2 NEAR worth of wNEAR). [2](#0-1) 

If the deploy batch fails (e.g., `initialize` panics because the wNEAR account is not registered, or gas is exhausted), `factory_update_address_version` runs and returns `ERR_ROUTER_DEPLOY_FAILED`: [3](#0-2) 

`withdraw_wnear_to_router` then runs as a callback and immediately aborts because `promise_result_check()` returns `Some(false)`: [4](#0-3) 

The wNEAR that was moved to the engine's implicit address is never withdrawn. There is no refund path in the XCC precompile or in `handle_precompile_promise` to return the wNEAR to the user. [5](#0-4) 

The engine's implicit address (`near_account_to_evm_address(engine_account_id)`) has no private key and cannot be controlled by any user. The only code path that can spend wNEAR from this address is `withdraw_wnear_to_router`, which is a private self-call only reachable as a callback in the XCC promise chain. No admin recovery function exists. [6](#0-5) 

### Impact Explanation

User wNEAR equal to `attached_near + STORAGE_AMOUNT` (minimum ~2 NEAR) is permanently frozen in the engine's implicit EVM address with no recovery mechanism. This is a **permanent freezing of user funds**.

### Likelihood Explanation

Any first-time XCC user (router not yet deployed) who attaches NEAR to their XCC call is exposed. The deploy batch can fail due to:
- The `initialize` function panicking (e.g., wNEAR account not registered with the NEP-141 storage standard)
- Gas exhaustion in the deploy batch (the batch includes `CreateAccount`, `Transfer`, `DeployContract`, and `FunctionCall("initialize")` — all in one batch with fixed gas `INITIALIZE_GAS = 15 TGas`) [7](#0-6) 

The `initialize` function itself creates an additional promise (`storage_deposit` on the wNEAR account), which can fail independently. [8](#0-7) 

### Recommendation

Before committing the EVM `transferFrom` state change, the XCC precompile should not move wNEAR until the router deployment is confirmed successful. Alternatively, attach a failure-path callback to the deploy batch that refunds the wNEAR back to the user's EVM address (by calling the wNEAR ERC-20 `transfer` back to the sender) when `factory_update_address_version` detects `Some(false)` from `promise_result_check()`.

### Proof of Concept

1. User Alice has wNEAR in her EVM address and approves the XCC precompile to spend it.
2. Alice calls the XCC precompile for the first time (no router deployed), attaching 1 NEAR to her XCC call. `required_near = 1 NEAR + 2 NEAR (STORAGE_AMOUNT) = 3 NEAR`.
3. The XCC precompile executes `transferFrom(alice, engine_implicit, 3_NEAR_in_wNEAR)` — EVM state committed, Alice's wNEAR balance decreases by 3 NEAR.
4. The deploy batch is scheduled. The `initialize` function panics (e.g., wNEAR account not registered). The batch is rolled back atomically — no NEAR account is created.
5. `factory_update_address_version` runs, sees `Some(false)`, returns `ERR_ROUTER_DEPLOY_FAILED`.
6. `withdraw_wnear_to_router` runs, sees `Some(false)`, returns `ERR_CALLBACK_OF_FAILED_PROMISE`. The wNEAR in the engine's implicit address is never burned/withdrawn.
7. Alice's 3 NEAR worth of wNEAR is permanently frozen in the engine's implicit address. Alice has no recovery path. [9](#0-8) [10](#0-9)

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;
        if matches!(handler.promise_result_check(), Some(false)) {
            return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
        }
        let args: WithdrawWnearToRouterArgs = io.read_input_borsh()?;
        let current_account_id = env.current_account_id();
        let recipient = AccountId::try_from(format!(
            "{}.{}",
            args.target.encode(),
            current_account_id.as_ref()
        ))?;
        let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&current_account_id),
            current_account_id,
            io,
            env,
        );
        let (result, ids) = xcc::withdraw_wnear_to_router(
            &recipient,
            args.amount,
            wnear_address,
            &mut engine,
            handler,
        )?;
        if !result.status.is_ok() {
            return Err(b"ERR_WITHDRAW_FAILED".into());
        }
        let id = ids.last().ok_or(b"ERR_NO_PROMISE_CREATED")?;
        handler.promise_return(*id);
        Ok(result)
    })
}

    //    and/or attached NEAR to the call the user wants to make), then we need to have the
    //    engine withdraw that amount of wNEAR to the router account and then have the router
    //    unwrap it into actual NEAR. In the case of storage staking, the engine contract
    //    covered the cost initially (see setup batch above), so the unwrapping also sends
    //    a refund back to the engine.
    let withdraw_id = if required_near == ZERO_YOCTO {
        setup_id
    } else {
        let withdraw_call_args = WithdrawWnearToRouterArgs {
            target: sender,
            amount: required_near,
        };
        let withdraw_call = PromiseCreateArgs {
            target_account_id: current_account_id.clone(),
            method: "withdraw_wnear_to_router".into(),
            args: borsh::to_vec(&withdraw_call_args).unwrap(),
            attached_balance: ZERO_YOCTO,
            attached_gas: WITHDRAW_GAS,
        };
        // Safety: This promise is safe. Even though this is a call from the engine account to
        // itself invoking the `call` method (which could be dangerous), the argument to `call`
        // is controlled entirely by us (not any user). This call will only execute the wnear
        // exit precompile, and only for the necessary amount. Note that this amount will always
        // be present, otherwise the user's call to the xcc precompile would have failed.
        let id = match setup_id {
            None => handler.promise_create_call(&withdraw_call),
            Some(setup_id) => handler.promise_attach_callback(setup_id, &withdraw_call),
        };
        let refund_needed = match deploy_needed {
            AddressVersionStatus::DeployNeeded { create_needed } => create_needed,
            AddressVersionStatus::UpToDate => false,
        };
        if refund_needed {
            let refund_call = PromiseCreateArgs {
                target_account_id: promise.target_account_id.clone(),
                method: "send_refund".into(),
                args: Vec::new(),
                attached_balance: ZERO_YOCTO,
                attached_gas: REFUND_GAS,
            };
            // Safety: This call is safe because the router's `send_refund` method
            // does not violate any security invariants. It only sends NEAR back to this contract.
            Some(handler.promise_attach_callback(id, &refund_call))
        } else {
            Some(id)
        }
    };
    // 3. Finally we can do the call the user wanted to do.

    // Safety: this call is safe because the promise comes from the XCC precompile, not the
    // user directly. The XCC precompile will only construct promises that target the `execute`
    // and `schedule` methods of the user's router contract. Therefore, the user cannot have
    // the engine make arbitrary calls.
    match withdraw_id {
        None => handler.promise_create_call(promise),
        Some(withdraw_id) => handler.promise_attach_callback(withdraw_id, promise),

/// Private enum used for bookkeeping what actions are needed in the call to the router contract.
enum AddressVersionStatus {
    UpToDate,
    DeployNeeded { create_needed: bool },
}

impl AddressVersionStatus {
    fn new<I: IO>(
        io: &I,
        latest_code_version: CodeVersion,
        target_code_version: Option<CodeVersion>,
    ) -> Self {
        let first_upgradable_version =
            get_first_upgradable_version(io).unwrap_or(CodeVersion::ZERO);
        match target_code_version {
            None => Self::DeployNeeded {
                create_needed: true,
            },
            Some(version) if version < first_upgradable_version => {
                // It is impossible to upgrade the initial XCC routers because
                // they lack the upgrade method.
                Self::UpToDate
            }
            Some(version) if version < latest_code_version => Self::DeployNeeded {
                create_needed: false,
            },
            Some(_version) => Self::UpToDate,
        }
    }
}
