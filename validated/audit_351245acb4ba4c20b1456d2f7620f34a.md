### Title
`ExitToEthereum` Precompile Burns User Tokens Without a Failure-Refund Callback, Causing Permanent Fund Loss - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns a user's EVM-side tokens (ETH or ERC-20) during EVM execution and then schedules a NEAR promise to call `withdraw` on the connector contract. Unlike `ExitToNear`, which attaches a `exit_to_near_precompile_callback` to refund tokens on promise failure, `ExitToEthereum` creates only a bare `PromiseArgs::Create` with no callback. If the connector's `withdraw` promise fails for any reason, the EVM-side burn is already committed and irreversible, permanently destroying the user's funds.

---

### Finding Description

In `ExitToEthereum::run()`, after the EVM execution burns the caller's ETH or ERC-20 tokens, the precompile constructs a NEAR promise to call `withdraw` on the connector: [1](#0-0) 

This promise is wrapped as `PromiseArgs::Create` — a fire-and-forget call with no attached callback: [2](#0-1) 

Contrast this with `ExitToNear::run()`, which conditionally wraps the transfer promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`: [3](#0-2) 

The `exit_to_near_precompile_callback` in the engine contract explicitly handles the failure branch by calling `engine::refund_on_error`: [4](#0-3) 

`ExitToEthereum` has no equivalent. The `error_refund` feature flag, which gates the refund path for `ExitToNear`, is never applied to `ExitToEthereum` at all: [5](#0-4) 

The EVM state commit (token burn) and the NEAR promise execution are separate receipts in the NEAR runtime. The burn is finalized when the EVM execution succeeds; the `withdraw` promise runs in a subsequent receipt. If that receipt fails, there is no mechanism to restore the burned tokens.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

When the connector's `withdraw` call fails, the user's EVM-side tokens (ETH balance deducted or ERC-20 burned) are permanently destroyed with no recovery path. The funds are neither on the EVM side (burned) nor on the Ethereum side (withdrawal failed). This matches the "permanent freezing of funds" impact class.

---

### Likelihood Explanation

**Moderate.** The connector is a separate NEAR contract. Realistic failure scenarios include:

1. The connector contract is independently paused (e.g., for a security incident or upgrade), while Aurora's `EXIT_TO_ETHEREUM` precompile remains unpaused. Any user who calls `ExitToEthereum` during this window loses funds permanently.
2. The owner calls `set_eth_connector_contract_account` to point to a new connector that does not yet implement `withdraw`, or implements it differently.
3. Gas exhaustion in the promise receipt (the `WITHDRAWAL_GAS` constant is set to 100 TGas but is marked with a TODO noting it has not been formally determined). [6](#0-5) 

The `pause_precompiles` admin function only pauses the Aurora-side precompile entry point; it does not prevent the connector from being independently paused on its own side. [7](#0-6) 

---

### Recommendation

Add a callback to `ExitToEthereum` analogous to `exit_to_near_precompile_callback`. The callback should:

1. Check the result of the `withdraw` promise.
2. On failure, call `engine::refund_on_error` to restore the burned ETH or re-mint the burned ERC-20 tokens to the original caller.

The promise construction in `ExitToEthereum::run()` should be changed from `PromiseArgs::Create(withdraw_promise)` to `PromiseArgs::Callback(PromiseWithCallbackArgs { base: withdraw_promise, callback: <refund_callback> })`, mirroring the pattern already used in `ExitToNear`.

---

### Proof of Concept

1. User holds 1 ETH on Aurora and calls the `ExitToEthereum` precompile (flag `0x00`, 20-byte Ethereum recipient address).
2. The EVM deducts 1 ETH from the caller's Aurora balance. The EVM execution succeeds and the state is committed.
3. Aurora schedules a NEAR promise: `withdraw` on the connector contract with `attached_gas = 100 TGas`.
4. Between step 2 and step 3's execution, the connector contract is paused by its own admin (independent of Aurora's `pause_precompiles`).
5. The `withdraw` receipt fails. Because `ExitToEthereum` registered no callback, no refund logic runs.
6. The user's 1 ETH is permanently destroyed: gone from the EVM state, never delivered on Ethereum. [8](#0-7)

pub fn pause_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        require_running(&state::get_state(&io)?)?;
        let authorizer: EngineAuthorizer = engine::get_authorizer(&io);

        if !authorizer.is_authorized(&env.predecessor_account_id()) {
            return Err(b"ERR_UNAUTHORIZED".into());
        }

        let args: PausePrecompilesCallArgs = io.read_input_borsh()?;
        let flags = PrecompileFlags::from_bits_truncate(args.paused_mask);
        let mut pauser = EnginePrecompilesPauser::from_io(io);
        pauser.pause_precompiles(flags);
        Ok(())
    })
}
