### Title
`native_per_pubdata` Truncates to Zero via Floor Division, Bypassing Pubdata Native Resource Charges — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed with `wrapping_div` (floor/truncating integer division). When `pubdata_price < native_price`, the result is zero. A zero `native_per_pubdata` means the bootloader charges **no native resources** for any amount of pubdata written during a transaction, allowing any unprivileged sender to write unlimited state data at zero native cost.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the per-pubdata-byte native resource rate is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142-143
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is floor (truncating) integer division. When `pubdata_price < native_price`, the result is `0`.

The same truncation appears in the off-chain helper:

```rust
// api/src/helpers.rs:427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

Contrast this with `native_per_gas`, which correctly uses `div_ceil` to prevent truncation to zero:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:135
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

When `native_per_pubdata == 0`, every downstream pubdata charge becomes zero:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   // basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:352
   let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata); // = 0
   ```

2. **Post-execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   // basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:430-432
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)  // = 0 for any pubdata_spent
       .ok_or(out_of_native_resources!())?;
   ```

3. **Pubdata sufficiency check** in `check_enough_resources_for_pubdata` always returns `true` (0 native needed).

The `native_per_pubdata` value is stored in `TxContextForPreAndPostProcessing` and used throughout the transaction lifecycle for all pubdata charging decisions.

---

### Impact Explanation

When `pubdata_price < native_price` (a legitimate operator configuration, e.g., when computation is expensive relative to data posting), `native_per_pubdata` truncates to zero. Any unprivileged user can then:

- Write unlimited storage slots, emit events, and generate state diffs without paying any native resource cost for the resulting pubdata.
- Bloat the L2 state and force the operator to absorb L1 data-posting costs without compensation.
- Exhaust block pubdata capacity at minimal cost, effectively denying service to other users.

The operator's intended pricing model (pubdata costs something, just less than one native unit per byte) is completely bypassed. The correct behavior would be to charge at least 1 native unit per pubdata byte whenever `pubdata_price > 0`, mirroring the `div_ceil` used for `native_per_gas`.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic operator configuration. For example, if `native_price = 1000` (wei per native cycle) and `pubdata_price = 500` (wei per pubdata byte), `wrapping_div` yields 0. Any user who observes these block-level parameters (they are public) can immediately exploit the zero `native_per_pubdata` by submitting transactions that write many storage slots. No special privileges, leaked keys, or oracle manipulation are required — only a standard L2 transaction.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142-143
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs:427`. This ensures that whenever `pubdata_price > 0`, at least 1 native unit is charged per pubdata byte, preventing the complete bypass of pubdata costs.

---

### Proof of Concept

**Setup:** Operator sets `native_price = 1000`, `pubdata_price = 999`, `eip1559_basefee = 1000`.

**Step 1 — `native_per_pubdata` truncates to zero:**
```
native_per_pubdata = pubdata_price.wrapping_div(native_price)
                   = 999 / 1000 = 0  (floor division)
```

**Step 2 — Attacker submits a transaction writing 20 storage slots:**
```rust
// Contract writes s[0..19] = 0xfff...
// Each slot write = 32 bytes pubdata → 640 bytes total pubdata
```

**Step 3 — Pubdata charge is zero:**
```
native_for_pubdata = 640 * native_per_pubdata = 640 * 0 = 0
```
`check_enough_resources_for_pubdata` returns `true` with 0 native charged.

**Step 4 — Transaction succeeds with full state writes at zero pubdata cost.**

With `div_ceil`, the correct charge would be:
```
native_per_pubdata = ceil(999 / 1000) = 1
native_for_pubdata = 640 * 1 = 640 native units
```

The attacker would need to provide sufficient native resources (via gas limit × `native_per_gas`) to cover this cost, or the transaction would revert. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
