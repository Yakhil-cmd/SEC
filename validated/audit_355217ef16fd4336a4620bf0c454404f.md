### Title
Owner Can Permanently Lose Admin Control by Setting `owner_id` to Empty String - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The `set_owner` function in Aurora Engine accepts an empty string as the new owner account ID. This is possible because the `BorshDeserialize` implementation for `AccountId` contains a backward-compatibility exception that silently accepts empty strings and returns `AccountId::default()` (an empty string) instead of an error. Setting `owner_id` to `""` permanently locks all owner-only administrative functions, since no valid NEAR account can have an empty ID.

### Finding Description
The `BorshDeserialize` implementation for `AccountId` in `engine-types/src/account_id.rs` contains a special backward-compatibility case:

```rust
impl BorshDeserialize for AccountId {
    fn deserialize_reader<R: io::Read>(reader: &mut R) -> io::Result<Self> {
        let account: String = BorshDeserialize::deserialize_reader(reader)?;

        // It's for saving backward compatibility.
        if account.is_empty() {
            return Ok(Self::default());
        }

        Self::try_from(account)
            .map_err(|e| io::Error::new(io::ErrorKind::InvalidData, e.to_string()))
    }
}
``` [1](#0-0) 

When the deserialized string is empty, it returns `Ok(Self::default())` — an `AccountId` wrapping an empty `Box<str>` — instead of returning an error. All other construction paths (`AccountId::new`, `TryFrom<String>`, `FromStr`) call `validate()` which rejects strings shorter than `MIN_ACCOUNT_ID_LEN = 2`. [2](#0-1) 

The `set_owner` function reads the new owner via `io.read_input_borsh()` and applies only two checks: that the contract is running, and that the new owner is not identical to the current owner. There is no check that `args.new_owner` is non-empty:

```rust
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }
        state.owner_id = args.new_owner;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [3](#0-2) 

`SetOwnerArgs` holds `new_owner: AccountId`, so the Borsh deserialization path above is the one used: [4](#0-3) 

`EngineState.owner_id` is typed as `AccountId` with no invariant enforcing non-emptiness at the state level: [5](#0-4) 

Once `owner_id` is stored as `""`, every subsequent call to any owner-gated function will fail because `require_owner_only` compares `predecessor_account_id` against `state.owner_id`, and no real NEAR account can have an empty ID. The affected functions include `set_owner`, `set_upgrade_delay_blocks`, `stage_upgrade`, `upgrade`, `resume_precompiles`, `pause_contract`, `resume_contract`, `set_key_manager`, and `attach_full_access_key`. [6](#0-5) [7](#0-6) 

### Impact Explanation
Permanent loss of owner-level administrative control over the Aurora Engine contract. The owner can no longer upgrade the contract, pause it, change the key manager, or perform any other owner-only operation. If a critical bug is discovered in the contract, it cannot be patched. Funds locked in the bridge or held in EVM accounts become permanently inaccessible if the contract requires an upgrade to recover them. This maps to **Critical — Permanent freezing of funds**.

### Likelihood Explanation
Low-to-medium. The current owner must call `set_owner` with a Borsh-encoded empty string as `new_owner`. This could occur due to a bug in owner tooling that serializes an uninitialized `AccountId`, a misconfiguration, or a deliberate but mistaken action. The backward-compatibility exception in `BorshDeserialize` makes this a silent success rather than a rejected transaction, increasing the risk of accidental triggering.

### Recommendation
Add an explicit non-empty check inside `set_owner` before accepting the new owner:

```rust
let args: SetOwnerArgs = io.read_input_borsh()?;
if args.new_owner.as_bytes().is_empty() {
    return Err(b"ERR_INVALID_OWNER".into());
}
if state.owner_id == args.new_owner {
    return Err(errors::ERR_SAME_OWNER.into());
}
```

Additionally, consider removing or narrowing the backward-compatibility exception in `AccountId::BorshDeserialize`. If empty `AccountId` values must be supported for legacy state migration, the exception should be scoped only to state deserialization paths, not to user-supplied input deserialization.

### Proof of Concept
1. The current owner constructs `SetOwnerArgs { new_owner: AccountId("") }` and Borsh-serializes it (the Borsh encoding of an empty string is `[0, 0, 0, 0]` — a 4-byte little-endian length of 0 followed by zero bytes).
2. The owner calls `set_owner` on the Aurora Engine contract with this payload.
3. Inside `set_owner`, `io.read_input_borsh::<SetOwnerArgs>()` invokes `AccountId::BorshDeserialize`, which hits the `if account.is_empty()` branch and returns `Ok(AccountId(""))` without error.
4. The same-owner check `state.owner_id == args.new_owner` evaluates to `false` (current owner is a real account ID, not `""`), so execution continues.
5. `state.owner_id` is written as `""` to persistent storage.
6. All subsequent calls to owner-only functions (e.g., `set_upgrade_delay_blocks`, `upgrade`, `pause_contract`) fail with `ERR_NOT_ALLOWED` because `require_owner_only` can never match `predecessor_account_id` against `""`. [8](#0-7) [9](#0-8)
