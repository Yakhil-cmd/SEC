### Title
ckETH Burned With No Reimbursement When `erc20_tx_fee ≤ CKETH_LEDGER_TRANSACTION_FEE` and ckERC20 Burn Fails - (File: rs/ethereum/cketh/minter/src/main.rs)

---

### Summary

In `withdraw_erc20`, the ckETH minter burns the caller's ckETH as a gas-fee deposit before attempting to burn the ckERC20 tokens. If the ckERC20 burn fails with a user-attributable error (`InsufficientFunds`, `AmountTooLow`, or `InsufficientAllowance`), the minter computes a `reimbursed_amount` by subtracting a penalty (`CKETH_LEDGER_TRANSACTION_FEE = 2_000_000_000_000 wei`). When the estimated gas fee is at or below that penalty, `checked_sub` returns `None`, which is silently coerced to `Wei::ZERO` via `unwrap_or`. The subsequent guard `if reimbursed_amount > Wei::ZERO` then skips recording any reimbursement request entirely. The already-burned ckETH is permanently lost with no recovery path.

---

### Finding Description

`withdraw_erc20` executes a two-step burn:

1. **Step 1** – burn `erc20_tx_fee` ckETH from the caller's account (gas-fee deposit).
2. **Step 2** – burn `ckerc20_withdrawal_amount` ckERC20 tokens from the caller's account.

If Step 2 fails with a user-attributable error, the fallback path at lines 506–535 computes the reimbursable ckETH amount:

```rust
// rs/ethereum/cketh/minter/src/main.rs  lines 507–514
let reimbursed_amount = match &ckerc20_burn_error {
    LedgerBurnError::TemporarilyUnavailable { .. } => erc20_tx_fee,
    LedgerBurnError::InsufficientFunds { .. }
    | LedgerBurnError::AmountTooLow { .. }
    | LedgerBurnError::InsufficientAllowance { .. } => erc20_tx_fee
        .checked_sub(CKETH_LEDGER_TRANSACTION_FEE)
        .unwrap_or(Wei::ZERO),   // ← silently becomes ZERO when fee ≤ penalty
};
if reimbursed_amount > Wei::ZERO {   // ← guard skips reimbursement entirely
    // record FailedErc20WithdrawalRequest ...
}
``` [1](#0-0) 

`CKETH_LEDGER_TRANSACTION_FEE` is a compile-time constant of `2_000_000_000_000 wei`: [2](#0-1) 

`erc20_tx_fee` is derived from the live Ethereum gas-fee estimate multiplied by the ERC-20 gas limit (`65_000`): [3](#0-2) 

When `erc20_tx_fee ≤ 2_000_000_000_000 wei` (i.e., `max_fee_per_gas ≤ ~30.8 gwei / 65_000 ≈ 0.031 gwei`), `checked_sub` returns `None`, `unwrap_or(Wei::ZERO)` yields `Wei::ZERO`, and the `if` guard suppresses the `FailedErc20WithdrawalRequest` event entirely. No entry is added to `reimbursement_requests`, so `process_reimbursement` never mints back any ckETH to the user. [4](#0-3) [5](#0-4) 

The ckERC20 burn never succeeded, so no ERC-20 tokens leave the chain. Only the ckETH gas-fee deposit is destroyed, with no on-chain record and no reimbursement path.

---

### Impact Explanation

Any user whose `erc20_tx_fee` estimate falls at or below `CKETH_LEDGER_TRANSACTION_FEE` and whose ckERC20 burn fails with a user-attributable error permanently loses their ckETH gas-fee deposit. The amount lost equals `erc20_tx_fee` (up to `2_000_000_000_000 wei = 0.000002 ETH` per call). There is no admin escape hatch, no retry mechanism, and no event emitted that would allow off-chain tooling to detect or recover the loss. The ckETH ledger supply decreases without a corresponding Ethereum-side event, violating the conservation invariant of the chain-fusion bridge.

---

### Likelihood Explanation

The trigger condition requires Ethereum `max_fee_per_gas` to be at or below approximately `0.031 gwei` at the moment the gas estimate is fetched. Ethereum has historically reached sub-1-gwei gas prices during periods of very low network activity (e.g., early 2020, late 2022). The condition is therefore rare but not theoretical. An unprivileged caller can deliberately time the call to a low-gas-price window and submit a `withdraw_erc20` call with an insufficient ckERC20 allowance (`icrc2_approve` for 0), reliably triggering the bug with no special privileges. The entry path is fully externally reachable via the public `withdraw_erc20` update endpoint.

---

### Recommendation

Replace the silent `unwrap_or(Wei::ZERO)` with an explicit branch that reimburses the full `erc20_tx_fee` when the penalty would exceed it (analogous to the `TemporarilyUnavailable` branch), or remove the `if reimbursed_amount > Wei::ZERO` guard and always record a reimbursement request (even for zero-amount, which the ledger will reject harmlessly). For example:

```rust
LedgerBurnError::InsufficientFunds { .. }
| LedgerBurnError::AmountTooLow { .. }
| LedgerBurnError::InsufficientAllowance { .. } => {
    erc20_tx_fee
        .checked_sub(CKETH_LEDGER_TRANSACTION_FEE)
        .unwrap_or(Wei::ZERO)  // if fee ≤ penalty, reimburse nothing (penalty absorbs all)
    // BUT: if erc20_tx_fee itself is zero, there is nothing to reimburse anyway.
    // The fix: always reimburse at least Wei::ZERO only when erc20_tx_fee > 0,
    // OR cap the penalty to min(CKETH_LEDGER_TRANSACTION_FEE, erc20_tx_fee)
    // so the user always gets back erc20_tx_fee - min(penalty, erc20_tx_fee) = 0,
    // but the event is still emitted for auditability.
}
```

The safest fix is:

```rust
let penalty = CKETH_LEDGER_TRANSACTION_FEE.min(erc20_tx_fee);
let reimbursed_amount = erc20_tx_fee.checked_sub(penalty).unwrap_or(Wei::ZERO);
// Always emit the event so the burn is traceable, even if reimbursed_amount == 0.
```

### Proof of Concept

1. Wait for (or simulate) an Ethereum gas-price environment where `max_fee_per_gas ≤ 0.031 gwei` so that `estimate_erc20_transaction_fee()` returns a value `≤ 2_000_000_000_000 wei`.
2. As an unprivileged principal, call `icrc2_approve` on the ckETH ledger to approve the minter for `erc20_tx_fee` ckETH (enough to pass the ckETH burn).
3. Call `icrc2_approve` on the ckERC20 ledger for `0` (or simply hold no ckERC20 balance), ensuring the ckERC20 burn will fail with `InsufficientAllowance` or `InsufficientFunds`.
4. Call `withdraw_erc20` on the minter.
5. Observe: the ckETH ledger records a burn of `erc20_tx_fee` from the caller's account; the minter emits **no** `FailedErc20WithdrawalRequest` event; `process_reimbursement` never mints ckETH back; the caller's ckETH balance is permanently reduced. [6](#0-5)
