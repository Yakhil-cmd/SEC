### Title
Reward Rate Change Applied Retroactively to Elapsed Rounds Without Prior Settlement - (`rs/sns/governance/src/governance.rs`)

### Summary

In the SNS Governance canister, when a `ManageNervousSystemParameters` proposal changes `VotingRewardsParameters` (e.g., `initial_reward_rate_basis_points` or `final_reward_rate_basis_points`), the new rate is applied retroactively to all reward rounds elapsed since the last `RewardEvent`, including rounds that occurred before the rate change. This is because `perform_manage_nervous_system_parameters` updates `self.proto.parameters` immediately without first triggering a reward distribution to settle the current period under the old rate.

### Finding Description

`run_periodic_tasks` in the SNS Governance canister calls `process_proposals()` before `distribute_rewards()`: [1](#0-0) 

`process_proposals()` can execute a `ManageNervousSystemParameters` proposal, which calls `perform_manage_nervous_system_parameters`: [2](#0-1) 

This directly overwrites `self.proto.parameters` with the new `VotingRewardsParameters` — no prior reward settlement occurs. Subsequently, when `distribute_rewards` runs in the same `run_periodic_tasks` invocation, it reads the **already-updated** parameters: [3](#0-2) 

It then computes the rewards purse by looping over **all** elapsed rounds since the last `RewardEvent`, applying the new rate to every round: [4](#0-3) 

The `reward_rate_at` function computes the rate from `initial_reward_rate_basis_points`, `final_reward_rate_basis_points`, and `reward_rate_transition_duration_seconds`: [5](#0-4) 

If any of these basis-point fields changed via the proposal, the new rate is applied to all elapsed rounds — including those that occurred before the parameter change — rather than only to rounds after the change.

The proto comment itself acknowledges this is problematic: [6](#0-5) 

### Impact Explanation

**Reward accounting is corrupted for the entire elapsed period since the last `RewardEvent`:**

- **Rate lowered** (e.g., `initial_reward_rate_basis_points` reduced): The lower rate is applied to all elapsed rounds, including those that occurred before the change. Stakers lose rewards they were entitled to under the old rate for the pre-change period.
- **Rate raised**: The higher rate is applied retroactively to pre-change rounds. Stakers receive more rewards than they were entitled to, inflating token supply beyond the intended schedule.

The magnitude of the error is proportional to `(new_rate - old_rate) × elapsed_rounds_before_change × token_supply`. With a round duration of one day and a rate change occurring mid-period, up to one full round's worth of rewards can be miscalculated. Over multiple such proposals, the cumulative drift in reward accounting can be significant.

### Likelihood Explanation

Any SNS token holder with sufficient stake can submit a `ManageNervousSystemParameters` proposal to change `voting_rewards_parameters`. This is a standard, documented governance operation. The bug is triggered automatically every time such a proposal is executed and there is any elapsed time since the last `RewardEvent` (which is virtually always the case, since reward events occur at most once per `round_duration_seconds`). No special attacker capability is required beyond normal governance participation.

### Recommendation

Before executing a `ManageNervousSystemParameters` proposal that modifies `voting_rewards_parameters`, the governance canister should first settle the current reward period under the old parameters. Concretely, `perform_manage_nervous_system_parameters` should check whether `voting_rewards_parameters` is being changed and, if so, trigger `distribute_rewards` (or an equivalent settlement) using the old parameters before overwriting `self.proto.parameters`. Alternatively, `distribute_rewards` should record the timestamp of each rate change and use the appropriate rate for each round based on which rate was active during that round.

### Proof of Concept

1. An SNS is deployed with `initial_reward_rate_basis_points = 200` (2% annual) and `round_duration_seconds = 86400` (1 day).
2. A reward event occurs at `T=0`.
3. At `T = 0.5 days`, a `ManageNervousSystemParameters` proposal passes, changing `initial_reward_rate_basis_points` to `100` (1% annual). `perform_manage_nervous_system_parameters` updates `self.proto.parameters` immediately.
4. At `T = 1 day`, `run_periodic_tasks` fires. `process_proposals` has nothing new to execute. `should_distribute_rewards()` returns `true`.
5. `distribute_rewards` reads the new `voting_rewards_parameters` (1% rate) and computes the reward purse for the 1 elapsed round using the 1% rate.
6. **Expected**: The first 0.5 days should have been rewarded at 2%, the second 0.5 days at 1%. The actual purse uses 1% for the entire day, causing stakers to lose approximately 0.5 days × (2% − 1%) × token_supply in rewards. [7](#0-6) [8](#0-7)

    /// Executes a ManageNervousSystemParameters proposal by updating Governance's
    /// NervousSystemParameters
    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
    }

    pub fn reward_rate_at(&self, now: Instant) -> RewardRate {
        let reward_rate_transition_duration_seconds = self
            .reward_rate_transition_duration_seconds
            .expect("reward_rate_transition_duration_seconds unset");

        let time_since_genesis = {
            let result = now - *GENESIS;
            // For the purposes of determining reward rate, treat times before
            // genesis the same as at genesis. This is not expected to occur in
            // practice. This code is just being extra defensive.
            if result.as_secs() < i2d(0) {
                Duration { days: i2d(0) }
            } else {
                result
            }
        };
        if reward_rate_transition_duration_seconds == 0
            || time_since_genesis.as_secs() >= i2d(reward_rate_transition_duration_seconds)
        {
            return self.final_reward_rate();
        }

        // s linearly varies from 1 -> 0 as seconds_since_genesis varies from 0
        // to reward_rate_transition_duration_seconds.
        let transition = LinearMap::new(
            dec!(0)..i2d(reward_rate_transition_duration_seconds),
            dec!(1)..dec!(0),
        );
        let s = transition.apply(time_since_genesis.as_secs());
        // s2 varies quadratically from 1 -> 0 (again, as seconds_since_genesis
        // varies from 0 to reward_rate_transition_duration_seconds), and
        // flattens out as seconds_since_genesis approaches
        // reward_rate_transition_duration_seconds.
        let s2 = s * s;

        // This looks backwards, but we think of variable rate as being added to
        // final growth rate, not initial, and the amount to add is up to
        // initial - final (where initial is thought of as being greater than
        // final).
        let dr = self.initial_reward_rate() - self.final_reward_rate();
        // variable_reward_rate varies from dr to 0 as round varies from
        // 1 to transition_round_count.
        let variable_reward_rate = s2 * dr;

        self.final_reward_rate() + variable_reward_rate
