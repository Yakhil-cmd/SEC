Audit Report

## Title
Incorrect Status Mapping Permanently Blocks Already-Claimed SNS Neurons from Retry — (`rs/sns/swap/src/types.rs`)

## Summary
In `rs/sns/swap/src/types.rs`, the `From<ClaimedSwapNeuronStatus> for ClaimedStatus` conversion maps `ClaimedSwapNeuronStatus::AlreadyExists` to `ClaimedStatus::Invalid` instead of `ClaimedStatus::Success`. When SNS Governance returns `AlreadyExists` for a neuron that was already successfully created in a prior `finalize_swap` invocation, the Swap canister permanently marks that neuron recipe as `Invalid`, causing all subsequent `finalize_swap` calls to skip it with `ConversionError::Invalid` and count it as an invalid sweep entry rather than a success.

## Finding Description
The root cause is in `rs/sns/swap/src/types.rs` at lines 1024–1035:

```rust
impl From<ClaimedSwapNeuronStatus> for ClaimedStatus {
    fn from(claimed_swap_neuron_status: ClaimedSwapNeuronStatus) -> Self {
        match claimed_swap_neuron_status {
            ClaimedSwapNeuronStatus::Success => ClaimedStatus::Success,
            ClaimedSwapNeuronStatus::Unspecified => ClaimedStatus::Failed,
            ClaimedSwapNeuronStatus::MemoryExhausted => ClaimedStatus::Failed,
            ClaimedSwapNeuronStatus::Invalid => ClaimedStatus::Invalid,
            ClaimedSwapNeuronStatus::AlreadyExists => ClaimedStatus::Invalid,  // BUG
        }
    }
}
``` [1](#0-0) 

SNS Governance explicitly returns `ClaimedSwapNeuronStatus::AlreadyExists` when a neuron was already created in a prior call, as confirmed in `rs/sns/governance/src/governance.rs` lines 4498–4504: [2](#0-1) 

The converted `ClaimedStatus::Invalid` is then written back to `recipe.claimed_status` at `rs/sns/swap/src/swap.rs` line 1889: [3](#0-2) 

On any subsequent `finalize_swap` call, `SnsNeuronRecipe::to_neuron_recipe` checks the stored `claimed_status` and, finding `ClaimedStatus::Invalid`, returns `ConversionError::Invalid` — permanently skipping the recipe without retrying: [4](#0-3) 

The existing guard for `ClaimedStatus::Success` (which correctly skips already-processed neurons via `ConversionError::AlreadyProcessed`) is never reached for `AlreadyExists`-affected recipes because they are stored as `Invalid`, not `Success`. The test at lines 3983–4071 explicitly asserts the buggy behavior as if it were correct, confirming the mapping was intentional but semantically wrong: [5](#0-4) 

## Impact Explanation
This matches the allowed High impact: **Significant SNS security impact with concrete user or protocol harm**. When `finalize_swap` is retried after a partial failure (a supported and expected scenario), any neuron recipe that received `AlreadyExists` from SNS Governance is permanently marked `Invalid`. The `SweepResult` counts these as `invalid` instead of `success`, meaning the finalization sweep never reaches full completion for affected participants. The SNS launch process can stall or report incorrect outcomes. While the neurons do exist in SNS Governance, the Swap canister's state is permanently corrupted for those recipes, and no unprivileged recovery path exists — the comment in `to_neuron_recipe` states "intervention is needed to make valid again" by resetting to `ClaimedStatus::Pending`, which requires canister-level state manipulation. [6](#0-5) 

## Likelihood Explanation
Triggered whenever `finalize_swap` is called more than once after a swap is committed and at least one neuron was successfully created in a prior partial run. This is an explicitly supported scenario: the `finalize_swap_in_progress` lock and `already_tried_to_auto_finalize` flag both exist because finalization can be retried, and manual calls to `finalize_swap` are always permitted regardless of `already_tried_to_auto_finalize`. Any SNS swap experiencing a partial failure mid-finalization (e.g., canister upgrade, timeout, or memory exhaustion on a subset of neurons) followed by a retry will hit this bug. No special privileges are required to call `finalize_swap`. [7](#0-6) 

## Recommendation
Map `ClaimedSwapNeuronStatus::AlreadyExists` to `ClaimedStatus::Success` in `rs/sns/swap/src/types.rs`:

```diff
-            ClaimedSwapNeuronStatus::AlreadyExists => ClaimedStatus::Invalid,
+            ClaimedSwapNeuronStatus::AlreadyExists => ClaimedStatus::Success,
```

Update the corresponding test at `rs/sns/swap/src/swap.rs` lines 4024–4070 to assert `success: 1` and `ClaimedStatus::Success` for the `AlreadyExists` case. [8](#0-7) 

## Proof of Concept
1. Deploy a committed SNS swap with at least two participants.
2. Call `finalize_swap`. During `claim_swap_neurons`, SNS Governance creates neuron N and returns `ClaimedSwapNeuronStatus::Success`. The Swap canister writes `ClaimedStatus::Success` to recipe N.
3. Simulate a partial-failure scenario: reset recipe N's `claimed_status` back to `ClaimedStatus::Pending` (or `Failed`) to represent a state where the first call created the neuron in Governance but the Swap canister's state update was not persisted (e.g., due to a mid-execution upgrade).
4. Call `finalize_swap` again. `to_neuron_recipe` sees `Pending`, includes recipe N in the batch. SNS Governance returns `ClaimedSwapNeuronStatus::AlreadyExists` for neuron N (because it already exists). The conversion maps this to `ClaimedStatus::Invalid` and writes it to recipe N.
5. Call `finalize_swap` a third time. `to_neuron_recipe` sees `Invalid`, returns `ConversionError::Invalid`, and increments `sweep_result.invalid`. Recipe N is permanently skipped. The finalization sweep never counts N as `success`.

A deterministic unit test can reproduce this by constructing a `SnsNeuronRecipe` with `claimed_status: Pending`, calling `process_swap_neuron` with `ClaimedSwapNeuronStatus::AlreadyExists`, and asserting the resulting `claimed_status` is `Invalid` — which the existing test at lines 4059–4070 already does, confirming the bug. [9](#0-8)
