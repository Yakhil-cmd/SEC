### Title
Unchecked Return Values of EIP-7685 Request Processors Corrupt `requests_hash` in Block Header — (`basic_bootloader/src/bootloader/block_flow/ethereum/post_tx_op_sequencing.rs`)

---

### Summary

In `post_tx_op_sequencing.rs`, the return values of `eip6110_events_parser`, `eip7002_system_part`, and `eip7251_system_part` are all silently discarded with `let _ =`. Each of these functions returns `Result<bool, SystemError>` and mutates a shared `requests_hasher` before potentially returning an error. If any function fails **after** it has already written data into the hasher, the `requests_hash` committed to the block header will be computed from a partially-updated, corrupted hasher state — producing an incorrect block header.

---

### Finding Description

In `post_op` inside `post_tx_op_sequencing.rs`:

```rust
// Environment may have no such contracts predeployed for tests or sequencing purposes
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);

let requests_hash = Bytes32::from_array(requests_hasher.finalize().into());
```

All three results are discarded. The comment acknowledges this is intentional for environments where the contracts are not deployed — but the `let _ =` pattern also silently swallows errors that occur **after** the hasher has been partially mutated.

Tracing `eip7002_system_part` in `eip_7002_withdrawal_contract/mod.rs`:

1. The function first checks whether the contract is deployed (lines 71–76). If not deployed, it returns `Err` **before** touching the hasher — safe.
2. If there are requests in the queue (`num_dequeued > 0`), it writes the EIP-7685 type byte to the hasher **unconditionally** at line 177:
   ```rust
   requests_hasher.update([WITHDRAWAL_REQUEST_EIP_7685_TYPE]);
   ```
3. It then enters a loop reading three storage slots per request (lines 181–235), each using `?` to propagate errors.
4. If any storage read inside the loop returns an error, the function returns `Err` — but the type byte (and any previously hashed request data) is already in the hasher.
5. Because the caller discards the error with `let _ =`, execution continues to `requests_hasher.finalize()`, producing a hash over partial/corrupted data.

The same pattern applies to `eip7251_system_part` in `eip_7251_consolidation_contract/mod.rs` (line 105: `requests_hasher.update([CONSOLIDATION_REQUEST_EIP_7685_TYPE])` before the loop).

---

### Impact Explanation

The `requests_hash` is a consensus-critical field in the block header. It is the SHA-256 hash over all EIP-7685 request data (deposits, withdrawals, consolidations) for the block. If it is computed from a partially-updated hasher, the resulting block header will contain an incorrect `requests_hash`. This is a **state-transition bug**: the ZKsync OS state transition function produces a block header that diverges from what a correct implementation would produce, causing block-level invalidity and potential consensus failure.

---

### Likelihood Explanation

The trigger requires that:
1. The EIP-7002 or EIP-7251 predeploy contract is deployed (so the early-exit guard does not fire).
2. There is at least one request in the queue (so the type byte is written to the hasher).
3. A storage read inside the processing loop returns an error.

Conditions 1 and 2 are normal in production. Condition 3 is the limiting factor — storage reads with `with_infinite_ergs` and `u64::MAX` native resources should not fail under normal operation. However, the silent discard means any future storage-layer error path, or any edge case in the storage model, would corrupt the hash without any observable signal. The structural defect (discarding `Result` after partial hasher mutation) is present and reachable today.

---

### Recommendation

Do not use `let _ =` to discard the results of functions that mutate shared state (the hasher) before potentially returning an error. Either:

1. **Propagate errors**: Use `?` so that a failure aborts `post_op` entirely.
2. **Restructure to avoid partial mutation**: Collect request data into a local buffer first; only feed it into the hasher if the collection succeeds.
3. **At minimum**, if silent failure on missing contracts is intentional, restructure each function so that the hasher is only updated after all data is successfully collected — never before a fallible operation.
