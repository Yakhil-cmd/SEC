Audit Report

## Title
Canister HTTP Outcall Queue Monopolization via Absent Per-Canister Slot Fairness - (File: `rs/execution_environment/src/execution_environment.rs`)

## Summary
The subnet-wide HTTP outcall in-flight queue enforces only a global cap of 3000 slots with no per-canister sub-limit. A single canister with sufficient cycles can occupy all 3000 slots by targeting unreachable URLs, blocking every other canister on the subnet from issuing HTTP outcalls for up to 60 seconds per saturation cycle. The attacker pays the upfront legacy fee per slot with no refund on timeout, making this a cycles-funded, sustained, protocol-level DoS against the subnet's HTTP outcall capacity.

## Finding Description
`try_add_http_context_to_replicated_state` at `rs/execution_environment/src/execution_environment.rs:2148–2162` enforces a single global gate:

```rust
if state
    .metadata
    .subnet_call_context_manager
    .canister_http_request_contexts
    .len()
    >= self.config.max_canister_http_requests_in_flight
{
    return Err(UserError::new(
        ErrorCode::CanisterRejectedMessage,
        format!("max number ({}) of http requests in-flight reached.", ...),
    ));
}
```

`MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT` is set to 3000 at `rs/config/src/execution_environment.rs:206`. There is no per-canister sub-limit, quota, or fairness scheduler anywhere in the codebase (confirmed by exhaustive grep). The consensus-layer timeout that governs how long each slot is held is `CANISTER_HTTP_TIMEOUT_INTERVAL = 60s` at `rs/types/types/src/canister_http.rs:79`.

Under legacy pricing (the default `PricingVersion`), the full legacy fee is deducted at submission time at `rs/execution_environment/src/execution_environment.rs:2208–2211`, and `LegacyTracker::create_payment_receipt` at `rs/https_outcalls/pricing/src/legacy.rs:48–52` explicitly documents that no cycles are ever refunded on timeout.

**Attack path:**
1. Deploy a canister on the target application subnet.
2. Call `ic_00::HttpRequest` 3000 times targeting an unreachable address (e.g., `https://[40d:40d:40d:40d:40d:40d:40d:40d]:28992`), attaching the minimum legacy fee per call.
3. All 3000 slots in `canister_http_request_contexts` are occupied.
4. Every subsequent `HttpRequest` from any canister on the subnet is rejected with `CanisterRejectedMessage` until the 60-second timeout expires.
5. As slots expire, the attacker resubmits at ~50 req/s to maintain saturation indefinitely.

The existing test `http_request_bound_holds` at `rs/execution_environment/src/execution_environment/tests.rs:2320–2381` directly confirms that a single `caller_canister` can fill the entire cap and that all subsequent requests from any canister are rejected once the cap is reached.

## Impact Explanation
All canisters on the targeted subnet lose the ability to make HTTP outcalls for the duration of the attack. Any application relying on HTTPS outcalls — price feeds, cross-chain bridges, oracle integrations — is completely disrupted. This is a concrete subnet availability impact not based on raw volumetric DDoS, matching the **High ($2,000–$10,000)** bounty tier: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The attack requires only a deployed canister and enough cycles to pay 3000 legacy fees. On a 13-node application subnet the fee per request is on the order of tens of billions of cycles; filling 3000 slots costs on the order of tens of trillions of cycles — non-trivial but achievable for a motivated attacker. No privileged access, governance majority, or external dependency is required. The attacker-controlled entry point is the standard `ic_00::HttpRequest` management canister method, reachable by any canister on the subnet. The attack is repeatable and can be maintained indefinitely.

## Recommendation
1. **Introduce a per-canister in-flight limit** (e.g., `max_canister_http_requests_per_canister`) enforced inside `try_add_http_context_to_replicated_state`, analogous to the existing `canister_guaranteed_callback_quota` pattern used for inter-canister callbacks.
2. **Charge a non-refundable base fee proportional to slot hold time** on timeout to raise the economic cost of slot-squatting beyond the current fixed legacy fee.
3. Consider a **per-canister sliding-window rate limit** on HTTP outcall submissions to prevent burst saturation.

## Proof of Concept
The existing unit test `http_request_bound_holds` at `rs/execution_environment/src/execution_environment/tests.rs:2320–2381` already proves the mechanism: a single `caller_canister` fills all configured slots and subsequent requests are rejected. To reproduce the full attack scenario, extend the test with two distinct canister IDs — fill the cap with canister A, then assert that canister B's `HttpRequest` is rejected with `CanisterRejectedMessage`. No mainnet testing is required; the deterministic execution test harness (`ExecutionTestBuilder`) is sufficient.
