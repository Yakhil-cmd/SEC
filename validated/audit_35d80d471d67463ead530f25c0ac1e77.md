Audit Report

## Title
ICRC-21 `add_expiration` Wrapping Cast Silently Emits Epoch 0 for `expires_at = u64::MAX` in `FieldsDisplayMessage` Path — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
In `add_expiration`, the `FieldsDisplayMessage` branch casts `expires_at: u64` to `i64` before dividing by `10^9`. For `expires_at = u64::MAX`, this wrapping cast produces `-1`, integer division yields `0`, and the re-cast to `u64` stores `0`, emitting `Value::TimestampSeconds { amount: 0 }` (Unix epoch, Jan 1 1970). The ledger's approval logic independently accepts `u64::MAX` as a valid future expiration (~year 2554), so the on-chain allowance is stored and enforceable while the consent message shown to a hardware wallet user displays an already-expired date.

## Finding Description
**Root cause** — `packages/icrc-ledger-types/src/icrc21/responses.rs`, lines 265–269:

```rust
ConsentMessage::FieldsDisplayMessage(fields_display) => {
    match expires_at {
        Some(expires_at) => {
            let seconds = (expires_at as i64) / 10_i64.pow(9);  // wrapping cast
            fields_display.fields.push((
                "Approval expiration".to_string(),
                Value::TimestampSeconds {
                    amount: seconds as u64,                       // 0 for u64::MAX
                },
            ))
``` [1](#0-0) 

Arithmetic for `expires_at = u64::MAX = 18446744073709551615`:

| Step | Expression | Result |
|------|-----------|--------|
| 1 | `u64::MAX as i64` | `-1` (defined wrapping cast in Rust) |
| 2 | `-1_i64 / 1_000_000_000_i64` | `0` (rounds toward zero) |
| 3 | `0_i64 as u64` | `0` |

The ledger's approval check at `rs/ledger_suite/common/ledger_core/src/approvals.rs` line 247 only rejects `expires_at <= now`. Since `u64::MAX` nanoseconds is ~year 2554, the approval is accepted and stored with the full `u64::MAX` expiration intact. [2](#0-1) 

The `GenericDisplayMessage` branch (lines 232–233) has the same wrapping cast but the subsequent `replace_nanosecond` call fails on the corrupted `nanos` value and returns an explicit error string, making the anomaly visible. The `FieldsDisplayMessage` branch has no such guard and silently emits epoch 0. [3](#0-2) 

**Exploit flow:**
1. Attacker-controlled dApp constructs `ApproveArgs { amount: large, expires_at: Some(u64::MAX), ... }`.
2. Hardware wallet calls `icrc21_canister_call_consent_message` requesting `FieldsDisplay` (the structured path designed for constrained devices).
3. The ledger's `add_expiration` emits `TimestampSeconds { amount: 0 }`.
4. Hardware wallet renders "Thu, 01 Jan 1970 00:00:00 +0000" — an already-expired date.
5. User concludes the approval is harmless and signs.
6. The on-chain `icrc2_approve` call succeeds; the spender holds a valid allowance until ~year 2554 and immediately drains it.

No existing guard in the consent message path validates that the computed `seconds` value is plausible or non-zero before emitting it.

## Impact Explanation
The bug undermines the core security guarantee of ICRC-21: that hardware wallets can display a trusted, ledger-generated consent message to protect users from malicious dApps. A user following the ICRC-21 protocol correctly (trusting the ledger's own consent message) can be deceived into granting a near-permanent large allowance, leading to direct theft of ledger assets. This matches the allowed High impact: "Significant ledger security impact with concrete user or protocol harm" and "Unauthorized access to ledger or canister-controlled funds."

## Likelihood Explanation
The attack requires only an unprivileged malicious dApp — no key compromise, no governance majority, no node control. Hardware wallets implementing `FieldsDisplay` are the primary ICRC-21 target and are explicitly supported by the ledger's DID. The value `u64::MAX` is a natural sentinel for "no expiration" that a dApp could plausibly pass. The attack is repeatable against any user of any ICRC-2 ledger that uses this shared `add_expiration` implementation.

## Recommendation
Replace the wrapping `i64` cast with pure `u64` arithmetic in both branches of `add_expiration`:

```rust
// FieldsDisplay branch — fix:
let seconds = expires_at / 1_000_000_000_u64;
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds },
))

// GenericDisplay branch — fix:
let seconds = expires_at / 1_000_000_000_u64;
let nanos = (expires_at % 1_000_000_000_u64) as u32;
```

Add a unit test asserting that for `expires_at = u64::MAX`, `TimestampSeconds.amount == u64::MAX / 1_000_000_000` (i.e., `18446744073`), and that the `GenericDisplay` path produces a valid formatted timestamp rather than an error string.

## Proof of Concept

```rust
// Reproduces the bug — no canister needed
let expires_at: u64 = u64::MAX;

// Buggy path (current code):
let seconds_buggy = (expires_at as i64) / 10_i64.pow(9);
let amount_buggy = seconds_buggy as u64;
assert_eq!(amount_buggy, 0); // emits Jan 1 1970

// Correct path:
let seconds_correct = expires_at / 1_000_000_000_u64;
assert_eq!(seconds_correct, 18_446_744_073_u64); // ~year 2554

// Ledger accepts u64::MAX: approvals.rs line 247
// u64::MAX > current_time → approval stored, valid until year ~2554
```
