### Title
Integer Truncation in `native_per_pubdata` Calculation Causes Systematic Pubdata Underpayment - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary
The `native_per_pubdata` rate used to charge L2 transaction senders for pubdata consumption is computed with floor (truncating) integer division, while the analogous `native_per_gas` rate uses ceiling division. This asymmetry causes every L2 transaction that generates pubdata to systematically underpay the protocol by up to `(native_price − 1)` native units per pubdata byte. An unprivileged sender can amplify the discount by maximising pubdata output (e.g., many storage writes) within a single transaction.

### Finding Description

In `validate_and_compute_fee_for_transaction`, the two key per-unit rates are computed as follows:

```rust
// native_per_gas: ceiling division — rounds UP, protecting the protocol
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))…;

// native_per_pubdata: floor division — rounds DOWN, benefiting the sender
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))…;
``` [1](#0-0) 

`native_per_pubdata` is then used verbatim to charge native resources for every pubdata byte the transaction writes:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

Because `native_per_pubdata = ⌊pubdata_price / native_price⌋`, the charged rate is always ≤ the true rate. The per-byte underpayment in native units is:

```
underpayment_per_byte = (pubdata_price mod native_price) / native_price  [native units]
```

In wei this equals `(pubdata_price mod native_price)` wei per byte — up to `native_price − 1` wei per byte.

The same truncated value is also used to compute the intrinsic pubdata overhead deducted from the native limit at transaction setup:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
``` [3](#0-2) 

So the undercharge propagates through both the intrinsic and execution-time pubdata charging paths.

The `api/src/helpers.rs` off-chain validation helper mirrors the same floor division:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

confirming this is a consistent, protocol-wide choice rather than an isolated typo.

A secondary, smaller truncation exists in `compute_gas_refund`:

```rust
let delta_gas = if native_per_gas == 0 { 0 } else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [5](#0-4) 

Here `native_used / native_per_gas` truncates, so the extra gas charged to cover native consumption is always rounded down by up to 1 gas unit per transaction. This is the direct code implementation of the documented formula `deltaGas := (nativeUsed / nativePerGas) − gasUsed`. [6](#0-5) 

### Impact Explanation

**Resource accounting bug — systematic pubdata underpayment.**

For a concrete example: if `pubdata_price = 1,500 wei` and `native_price = 1,000 wei`, then `native_per_pubdata = 1` (true value is 1.5). A transaction writing 100,000 pubdata bytes is charged for 100,000 native units instead of 150,000 — a 33% discount on pubdata costs. The operator/protocol absorbs the 50,000-native-unit shortfall.

In the worst-case configuration (`pubdata_price = 2 × native_price − 1`), `native_per_pubdata` rounds from ~2 down to 1, giving senders a ~50% discount on all pubdata costs. Any sender who maximises pubdata output (storage writes, deployments) within a single transaction maximises this discount. The discount is fixed per byte and does not compound with transaction splitting, but it is permanent and affects every L2 transaction that generates pubdata.

### Likelihood Explanation

**High.** The truncation fires on every L2 transaction that generates pubdata whenever `pubdata_price` is not an exact multiple of `native_price`. These are operator-set parameters that are unlikely to be kept in exact ratio in practice. Any unprivileged sender submitting a storage-writing transaction triggers the undercharge automatically, with no special setup required.

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, matching the rounding direction already used for `native_per_gas`:

```rust
// Before
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))…;

// After
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))…;
```

Apply the same fix in `api/src/helpers.rs`. For the `delta_gas` truncation in `refund_calculation.rs`, consider using `div_ceil` on `native_used / native_per_gas` so that the protocol always recovers at least the full native cost in gas terms.

### Proof of Concept

1. Operator sets `pubdata_price = 1_500` and `native_price = 1_000`.
2. `native_per_pubdata = floor(1500 / 1000) = 1` (true value: 1.5).
3. Sender submits a transaction that writes 20 distinct storage slots (≈ 20 × 33 = 660 pubdata bytes).
4. Charged native for pubdata: `660 × 1 = 660` native units.
5. True cost: `660 × 1.5 = 990` native units.
6. Underpayment: 330 native units = 330,000 wei at `native_price = 1,000`.
7. Scaled to a transaction at the pubdata limit (e.g., 100,000 bytes): underpayment = 50,000,000 wei ≈ 0.00005 ETH per transaction, accumulating linearly with block throughput.
