### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Fund Loss - (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` — the production bridged-token contracts deployed on Aurora mainnet — burn the caller's ERC-20 tokens **before** calling the exit precompile, and then **never check the return value** of that precompile call. If the precompile call fails silently (returns `0` without reverting), the tokens are permanently destroyed on Aurora while no NEAR-side withdrawal promise is ever scheduled, resulting in a permanent loss of user funds.

---

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, irreversibly

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — if call returns 0, execution continues silently
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum`: [2](#0-1) 

And in both functions of `EvmErc20V2`: [3](#0-2) 

The `res` variable in the `assembly` block captures the EVM `call` opcode return value (`1` = success, `0` = failure), but it is never read or acted upon. The Solidity function returns normally regardless of whether the precompile succeeded or failed.

These contracts are the production bridged-token implementation, confirmed as Security Tier 1 and deployed at `aurora/mainnet/0x8bec47865ade3b172a928df8f990bc7f2a3b9f79`: [4](#0-3) 

They are deployed for every NEP-141 token bridged to Aurora via `deploy_erc20_token`, which embeds either `EvmErc20.bin` or `EvmErc20V2.bin`: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The sequence is:
1. `_burn(msg.sender, amount)` executes and succeeds — the user's ERC-20 balance is permanently reduced to zero on Aurora.
2. The `call` to the exit precompile fails (returns `0`).
3. Because `res` is never checked, no revert occurs — the EVM transaction succeeds.
4. No `ExitToNear` or `ExitToEthereum` log is emitted, so no NEAR-side `ft_transfer`/`withdraw` promise is ever scheduled.
5. The tokens are gone from Aurora and never arrive on NEAR or Ethereum. They are permanently destroyed.

This affects every bridged NEP-141 token on Aurora (USDC, USDT, wBTC, etc.) that uses these contracts.

---

### Likelihood Explanation

The precompile call can fail silently in multiple realistic scenarios:

1. **Paused precompile**: The engine has a `pause_precompiles` mechanism. If `ExitToNear` or `ExitToEthereum` is paused at the moment of the call, the precompile returns failure. The burn has already happened.
2. **Invalid recipient account ID**: A user who provides a malformed NEAR account ID string will cause the precompile to return an error. The precompile validates the account ID and returns `ExitError`, which translates to a `call` return value of `0`. The burn is already done.
3. **Out-of-gas in the sub-call**: If insufficient gas is forwarded to the precompile, it can fail.

Scenario 2 is directly user-triggerable with no special conditions. Any user who makes a typo in the recipient account ID will permanently lose their tokens.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero. The fix must be applied **before** the burn, or the check must revert the entire transaction on failure:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }   // ← add this check
    }
}
```

This ensures that if the precompile call fails, the entire transaction reverts (including the `_burn`), so no tokens are lost.

---

### Proof of Concept

1. Deploy `EvmErc20` (or use any existing bridged token on Aurora).
2. Mint tokens to `user`.
3. Call `withdrawToNear("invalid!!account@@id", amount)` from `user`.
4. The `ExitToNear` precompile rejects the invalid account ID and returns `0`.
5. The assembly block ignores `res = 0` and the function returns normally.
6. `user`'s ERC-20 balance is now `0` (burned), but no NEAR-side `ft_transfer` was ever scheduled.
7. The tokens are permanently destroyed.
