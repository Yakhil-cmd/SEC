Audit Report

## Title
ckDOGE Minter Missing Minimum Fee-Rate Floor in `DogecoinFeeEstimator::estimate_nth_fee` Enables Stuck Withdrawal Transactions — (`rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

## Summary
`DogecoinFeeEstimator::estimate_nth_fee` returns raw fee-percentile values with no minimum floor, unlike `BitcoinFeeEstimator::estimate_nth_fee` which was patched after a confirmed mainnet incident (2025-06-21) to clamp results via `f.max(self.minimum_fee_per_vbyte())`. The ckDOGE minter shares the same `estimate_fee_per_vbyte` / `build_unsigned_transaction` pipeline from `ic_ckbtc_minter` but was not given the equivalent protection. If the Dogecoin canister returns anomalously low fee percentiles — exactly as the Bitcoin canister did — withdrawal transactions will be built at a fee rate too low to relay or mine, locking user funds.

## Finding Description

`DogecoinFeeEstimator::estimate_nth_fee` returns the raw percentile value with no floor:

```rust
// rs/dogecoin/ckdoge/minter/src/fees/mod.rs lines 53-66
fn estimate_nth_fee(&self, fee_percentiles: &[FeeRate], nth: usize) -> Option<FeeRate> {
    match &self.network {
        Network::Mainnet => {
            if fee_percentiles.len() < 100 || nth >= 100 {
                return None;
            }
            Some(fee_percentiles[nth])  // ← no floor
        }
        Network::Regtest => Some(DEFAULT_REGTEST_FEE),
    }
}
``` [1](#0-0) 

`BitcoinFeeEstimator::estimate_nth_fee` was patched to apply a hard floor after the 2025-06-21 incident:

```rust
// rs/bitcoin/ckbtc/minter/src/fees/mod.rs line 112
median_fee.map(|f| f.max(self.minimum_fee_per_vbyte()))
``` [2](#0-1) 

The `minimum_fee_per_vbyte()` function returns 1 500 ms/vbyte for Mainnet and 1 000 ms/vbyte for Testnet, with an explicit comment documenting the incident: [3](#0-2) 

The ckDOGE minter's `DogeCanisterRuntime` implements `CanisterRuntime` with `type Estimator = DogecoinFeeEstimator`, meaning the shared `estimate_fee_per_vbyte` function in `rs/bitcoin/ckbtc/minter/src/lib.rs` dispatches to `DogecoinFeeEstimator::estimate_nth_fee` for ckDOGE: [4](#0-3) 

The shared pipeline stores the unguarded result directly into `last_median_fee_per_vbyte`, which is subsequently used verbatim when building every withdrawal transaction: [5](#0-4) 

The upgrade notes confirm the Bitcoin incident caused three real ckBTC withdrawals to get stuck and required an emergency canister upgrade: [6](#0-5) 

## Impact Explanation

When the Dogecoin canister returns anomalously low fee percentiles: (1) `estimate_nth_fee` returns `Some(<very_low_rate>)` — it does not return `None`, so the minter does not skip the update; (2) `last_median_fee_per_vbyte` is overwritten with the low rate; (3) every subsequent `retrieve_doge` call builds a transaction at that rate; (4) submitted transactions are below the Dogecoin relay threshold and are never mined; (5) UTXOs are locked in `submitted_transactions` and cannot be reused; (6) users' ckDOGE is already burned on the ledger while their DOGE is not delivered — funds are frozen until an emergency upgrade. This matches the **High** impact category: significant ck-token / Chain Fusion security impact with concrete, demonstrable user funds loss.

## Likelihood Explanation

No attacker or privileged access is required. The trigger is the Dogecoin canister returning anomalously low fee percentiles — exactly the same class of event that occurred with the Bitcoin canister on 2025-06-21 (fee 142 ms/vbyte vs. expected ~2 000 ms/vbyte). The root cause of that incident is still described as unknown in the upgrade notes, meaning the same bug could exist in the Dogecoin canister. The ckBTC fix was applied only to `BitcoinFeeEstimator`; `DogecoinFeeEstimator` was not updated.

## Recommendation

Add a `minimum_fee_per_koinu_per_byte()` constant to `DogecoinFeeEstimator` (analogous to `BitcoinFeeEstimator::minimum_fee_per_vbyte`) and apply it in `estimate_nth_fee`:

```rust
const fn minimum_fee_per_koinu_per_byte(&self) -> FeeRate {
    let rate = match &self.network {
        Network::Mainnet => 1_000_000, // 1 koinu/byte = 1_000_000 millikoinus/byte
        Network::Regtest => 0,
    };
    FeeRate::from_millis_per_byte(rate)
}

fn estimate_nth_fee(&self, fee_percentiles: &[FeeRate], nth: usize) -> Option<FeeRate> {
    match &self.network {
        Network::Mainnet => {
            if fee_percentiles.len() < 100 || nth >= 100 {
                return None;
            }
            Some(fee_percentiles[nth].max(self.minimum_fee_per_koinu_per_byte()))
        }
        Network::Regtest => Some(DEFAULT_REGTEST_FEE),
    }
}
```

The minimum value should be calibrated to the Dogecoin relay policy (currently 1 koinu/byte).

## Proof of Concept

1. Configure a test where the Dogecoin canister returns 100 fee-percentile entries all set to `FeeRate::from_millis_per_byte(100)` (well below the relay threshold).
2. Call `estimate_fee_per_vbyte` — it calls `DogecoinFeeEstimator::estimate_median_fee(&fees)`, which calls `estimate_nth_fee(&fees, 50)`.
3. `estimate_nth_fee` returns `Some(FeeRate::from_millis_per_byte(100))` — no floor is applied.
4. `last_median_fee_per_vbyte` is set to 100 millikoinus/byte.
5. A user calls `retrieve_doge`; the minter builds a transaction using `fee_millisatoshi_per_vbyte = 100`.
6. The transaction is broadcast but rejected by Dogecoin nodes as below the minimum relay fee.
7. UTXOs are locked in `submitted_transactions`; the user's ckDOGE has already been burned; DOGE is not delivered.

Contrast: with `BitcoinFeeEstimator`, step 3 returns `Some(FeeRate::from_millis_per_byte(1_500))` due to the `f.max(self.minimum_fee_per_vbyte())` clamp at line 112 of `rs/bitcoin/ckbtc/minter/src/fees/mod.rs`, preventing the stuck-transaction scenario.
