### Title
Predictable `PREVRANDAO` Derived Solely from Block Timestamp Enables Randomness Manipulation in EVM Contracts - (`x/evm/keeper/keeper.go`, `giga/deps/xevm/keeper/keeper.go`)

### Summary
Sei's EVM module populates the `PREVRANDAO` opcode value with `keccak256(block.timestamp)`. Because the block timestamp is deterministic and observable before a transaction executes, any unprivileged user can pre-compute the `PREVRANDAO` value for the current block and exploit EVM smart contracts that rely on it for randomness (raffles, lotteries, NFT reveals, etc.).

### Finding Description
In `GetVMBlockContext`, both the main EVM keeper and the Giga EVM keeper compute the `Random` field (which the EVM exposes via the `PREVRANDAO` opcode) as follows:

```go
// Use hash of block timestamp as info for PREVRANDAO
r, err := ctx.BlockHeader().Time.MarshalBinary()
rh := crypto.Keccak256Hash(r)
...
Random: &rh,
``` [1](#0-0) [2](#0-1) 

The `PREVRANDAO` opcode (EIP-4399, post-merge Ethereum) is the standard mechanism Solidity contracts use for on-chain randomness. Sei maps it to `keccak256(block.timestamp)`, which is:

1. **Predictable by any observer**: The block timestamp is part of the committed block header. All transactions in a block share the same timestamp, so any user who observes the timestamp (e.g., via a prior read or by being in the same block) can compute `keccak256(timestamp)` and know the exact `PREVRANDAO` value before their transaction executes.
2. **Influenceable by the block proposer**: In Sei's BFT-time, the proposer submits a timestamp that becomes the weighted median of precommit timestamps. A proposer (or a coalition controlling >50% voting power) can bias the timestamp within the valid range to steer the `PREVRANDAO` output toward a desired value.

This is the direct Sei-chain-layer analog of the `SqwidMarketplace.sol` vulnerability: instead of `block.difficulty` / `block.timestamp` in Solidity, Sei's chain code itself provides a weak `PREVRANDAO` seed to the EVM.

### Impact Explanation
Any EVM smart contract deployed on Sei that uses `block.prevrandao` (Solidity) or the `PREVRANDAO` opcode for randomness — including raffles, lotteries, NFT mints, gaming contracts — is exploitable. An unprivileged attacker can:
- Pre-compute the winning outcome before submitting their entry transaction.
- Repeatedly simulate outcomes off-chain and only submit when the result is favorable.

This constitutes **unintended contract execution from a network-layer bug** (the chain provides a broken randomness primitive). Depending on the contract, this can lead to direct fund loss for users of those contracts.

**Scope classification: Medium** — unintended contract execution from network bug; fund loss potential depends on deployed contracts.

### Likelihood Explanation
- No special privileges required; any user can read the block timestamp and compute `keccak256(timestamp)`.
- The attack is trivially executable: observe timestamp → compute hash → submit exploit transaction in the same block.
- Sei is EVM-compatible and actively hosts DeFi/gaming contracts; the attack surface is live.

### Recommendation
Replace the timestamp-derived `PREVRANDAO` with a cryptographically unpredictable source. Options include:
1. **VRF-based randomness**: Use a verifiable random function seeded from validator signatures (e.g., the block's last-commit hash or a dedicated VRF beacon).
2. **Commit-reveal scheme**: Accumulate entropy across multiple blocks before revealing.
3. **Use the block hash of a past block**: `keccak256(previousBlockHash ++ height)` is harder to predict, though still proposer-influenceable.

At minimum, the comment `// Use hash of block timestamp as info for PREVRANDAO` should be treated as a known weakness and documented so contract developers are warned not to rely on `block.prevrandao` for security-sensitive randomness on Sei.

### Proof of Concept
1. Deploy a raffle contract on Sei EVM that picks a winner via `uint256(block.prevrandao) % numEntrants`.
2. At the start of a block, read `block.timestamp` from the block header (available via RPC before the block is finalized).
3. Compute `keccak256(abi.encode(timestamp))` off-chain.
4. Determine which entry index wins: `keccak256(timestamp) % numEntrants`.
5. Submit the entry transaction for that index in the same block.
6. The raffle contract selects the pre-computed winner every time. [3](#0-2) [4](#0-3)

func (k *Keeper) GetVMBlockContext(ctx sdk.Context, gp core.GasPool) (*vm.BlockContext, error) {
	if k.EthBlockTestConfig.Enabled {
		return k.getBlockTestBlockCtx(ctx)
	}
	if k.EthReplayConfig.Enabled {
		return k.getReplayBlockCtx(ctx)
	}
	coinbase, err := k.GetFeeCollectorAddress(ctx)
	if err != nil {
		return nil, err
	}

	// Use hash of block timestamp as info for PREVRANDAO
	r, err := ctx.BlockHeader().Time.MarshalBinary()
	if err != nil {
		return nil, err
	}
	rh := crypto.Keccak256Hash(r)

	txfer := func(db vm.StateDB, sender, recipient common.Address, amount *uint256.Int) {
		if IsPayablePrecompile(&recipient) {
			state.TransferWithoutEvents(db, sender, recipient, amount)
		} else {
			core.Transfer(db, sender, recipient, amount)
		}
	}
	var baseFee *big.Int
	if ctx.ChainID() == Pacific1ChainID && ctx.BlockHeight() < 114945913 {
		baseFee = k.GetBaseFeePerGas(ctx).TruncateInt().BigInt()
	} else {
		baseFee = k.GetNextBaseFeePerGas(ctx).TruncateInt().BigInt()
	}

	return &vm.BlockContext{
		CanTransfer: core.CanTransfer,
		Transfer:    txfer,
		GetHash:     k.GetHashFn(ctx),
		Coinbase:    coinbase,
		GasLimit: func() uint64 {
			if ctx.ConsensusParams() != nil && ctx.ConsensusParams().Block != nil {
				return uint64(ctx.ConsensusParams().Block.MaxGas) //nolint:gosec
			}
			return DefaultBlockGasLimit
		}(),
		BlockNumber: big.NewInt(ctx.BlockHeight()),
		Time:        uint64(ctx.BlockHeader().Time.Unix()), //nolint:gosec
		Difficulty:  utils.Big0,                            // only needed for PoW
		BaseFee:     baseFee,
		BlobBaseFee: utils.Big1, // Cancun not enabled
		Random:      &rh,
	}, nil
}
