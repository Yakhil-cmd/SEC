### Title
Missing `L2AssetTracker` Notification on L2→L1 Withdrawals Causes Inflated `chainBalance` Accounting - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary
The ZKsync OS bootloader explicitly notifies the `L2AssetTracker` contract about every L1→L2 deposit via `handleFinalizeBaseTokenBridgingOnL2`, but there is no corresponding notification for L2→L1 withdrawals. As a result, the `chainBalance` mapping in `L2AssetTracker` is never decremented, permanently inflating the tracked on-chain token supply relative to the actual circulating supply.

### Finding Description
The bootloader defines `notify_l2_asset_tracker` and calls it inside `mint_base_token` for every L1→L2 deposit: [1](#0-0) 

This call reaches `handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)` on the `L2AssetTracker` contract, which increments both `totalSuccessfulDepositsFromL1` and the `chainBalance` mapping for the base token asset: [2](#0-1) 

When a user performs an L2→L1 withdrawal by calling `withdraw(address)` (selector `0x51cff8d9`) or `withdrawWithMessage(address,bytes)` (selector `0x84bc3eb0`) on `L2_BASE_TOKEN_ADDRESS` (`0x800a`), the bootloader executes the EVM contract normally. The contract burns the user's balance and emits an L1 message via the L1 messenger hook: [3](#0-2) 

However, **no call is made to `L2AssetTracker` to decrement `chainBalance`**. The bootloader has no post-withdrawal hook analogous to `notify_l2_asset_tracker`. The `L2AssetTracker` storage layout comment in the code confirms `chainBalance` lives at slot 151 and is intended to track the net on-chain balance: [4](#0-3) 

Every successful withdrawal leaves `chainBalance` unchanged (too high), while every deposit increases it. After N deposit/withdraw cycles, `chainBalance` overstates the actual L2 supply by the total withdrawn amount.

### Impact Explanation
The `L2AssetTracker`'s `chainBalance` is the authoritative on-chain record of how many base tokens are locked on L2. Interop operations (cross-chain asset transfers, settlement-layer proofs) that read `chainBalance` to validate the available supply will see an inflated value. This allows the chain to assert it holds more base tokens than it actually does, which can be exploited to claim excess tokens from the L1 bridge — a direct path to fund loss from the L1 bridge contract. The accounting divergence grows monotonically with every withdrawal and is never corrected.

### Likelihood Explanation
Every L2→L1 base token withdrawal triggers this bug. Any unprivileged user who calls `withdraw` or `withdrawWithMessage` on `0x800a` contributes to the inflation. No special access is required. The entry path is the standard user-facing withdrawal flow, making this highly likely to be triggered in normal operation.

### Recommendation
After a successful L2→L1 withdrawal is processed by the L2 base token contract, the bootloader (or a dedicated post-execution hook) must call the `L2AssetTracker` with the withdrawn amount to decrement `chainBalance`. This mirrors the existing `notify_l2_asset_tracker` pattern used for deposits. A new function analogous to `notify_l2_asset_tracker` should be introduced that calls the appropriate `L2AssetTracker` method (e.g., `handleFinalizeWithdrawal` or equivalent) with the withdrawal amount and chain ID, and this function must be invoked after every confirmed L2→L1 withdrawal.

### Proof of Concept
1. Deploy a fresh chain with `L2AssetTracker` predeploy (as done in `install_default_predeployed_contracts`).
2. Submit an L1→L2 deposit of `X` tokens. Observe `chainBalance` increases by `X` (confirmed by `test_asset_tracker_called_on_deposit`).
3. Submit an L2→L1 withdrawal of `X` tokens via `withdraw(address)` on `0x800a`. Transaction succeeds (confirmed by `test_l2_base_token_withdraw_events`).
4. Read `chainBalance` from `L2AssetTracker` slot 151. It still equals `X` — it was never decremented.
5. Repeat steps 2–4 N times. `chainBalance` = `N * X` while actual circulating supply = 0.
6. An interop operation or L1 bridge proof using `chainBalance` as the available supply will accept `N * X` tokens as claimable, enabling extraction of `(N-1) * X` tokens beyond what was legitimately deposited. [5](#0-4) [6](#0-5)

pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;

    Ok(Ok(()))
