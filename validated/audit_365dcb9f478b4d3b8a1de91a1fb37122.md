### Title
Silo Mode Address Whitelist Removal Permanently Freezes User ERC-20 Token Balances - (File: engine/src/contract_methods/silo/mod.rs)

### Summary
In Aurora Engine's Silo mode, the `is_allow_submit` check gates **all** EVM transactions without distinguishing between new deposits and withdrawals of existing funds. If the owner enables the Address whitelist after users have already received ERC-20 tokens, or removes a user's address from the whitelist, those users' ERC-20 token balances are permanently frozen because every path to move or exit those tokens requires submitting an EVM transaction.

### Finding Description
The `is_allow_submit` function enforces that both the NEAR account and the EVM address must be present in their respective whitelists before any EVM transaction is accepted:

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [1](#0-0) 

`is_address_allowed` returns `false` whenever the Address whitelist is enabled and the address is absent:

```rust
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [2](#0-1) 

The `set_whitelist_status` and `remove_entry_from_whitelist` admin functions perform no validation against existing ERC-20 token balances held by affected addresses: [3](#0-2) 

ERC-20 tokens minted via `receive_erc20_tokens` are credited to the recipient's EVM address inside the Aurora EVM: [4](#0-3) 

The only ways to move or exit those ERC-20 tokens are EVM-level calls (`transfer`, `withdrawToNear`, `withdrawToEthereum` on the ERC-20 contract), all of which require passing the `is_allow_submit` whitelist gate. There is no NEAR-level bypass for ERC-20 balances (unlike ETH, which has the NEAR-level `withdraw` function). [5](#0-4) 

The `set_whitelist_status` and `remove_entry_from_whitelist` entrypoints are owner-only but perform no input validation against current user balances: [6](#0-5) 

### Impact Explanation
Any user whose EVM address is absent from an enabled Address whitelist cannot submit any EVM transaction. Their ERC-20 token balances (bridged NEP-141 tokens) are frozen with no self-service recovery path. The user cannot call `withdrawToNear`, `withdrawToEthereum`, or `transfer` on any ERC-20 contract. Recovery requires the owner to re-add the address, which may not happen (compliance removal, key loss, etc.).

**Impact: High — Temporary freezing of funds (potentially permanent without owner action).**

### Likelihood Explanation
Silo mode is a production deployment configuration. The owner legitimately enables the Address whitelist to restrict who can interact with the EVM. A realistic scenario: the whitelist is enabled after users have already received bridged ERC-20 tokens, and those users are not added to the whitelist. The owner may not realize this freezes existing balances rather than merely blocking new interactions.

**Likelihood: 2** (requires Silo mode deployment and a specific admin action, but the action is routine and the side-effect is non-obvious).

### Recommendation
The `is_allow_submit` check (and the underlying whitelist gate in the `submit` entrypoint) should distinguish between transactions that **withdraw existing funds** and transactions that **add new state**. Specifically:

- Calls to the exit precompiles (`exitToNear`, `exitToEthereum`) should bypass the Address whitelist check, analogous to how the original StakingVault fix allowed negative `wad` (unstake) regardless of the limit.
- Alternatively, `remove_entry_from_whitelist` should verify the target address holds no ERC-20 balances, or a grace-period withdrawal window should be enforced before the removal takes effect.

### Proof of Concept

```
1. Deploy Aurora Engine in Silo mode with set_silo_params.
2. Enable the Address whitelist:
       set_whitelist_status({ kind: Address, active: true })
3. Add user_address to the whitelist:
       add_entry_to_whitelist(WhitelistAddressArgs { kind: Address, address: user_address })
4. User bridges 1000 NEP-141 tokens → receives 1000 ERC-20 tokens at user_address.
5. Owner removes user_address from the whitelist:
       remove_entry_from_whitelist(WhitelistAddressArgs { kind: Address, address: user_address })
6. User submits EVM tx calling erc20.withdrawToNear(recipient, 1000):
       → EngineErrorKind::NotAllowed  (whitelist check fails)
7. User submits EVM tx calling erc20.transfer(other_address, 1000):
       → EngineErrorKind::NotAllowed
8. User's 1000 ERC-20 tokens are frozen with no self-service recovery path.
``` [7](#0-6)

fn test_submit_with_removing_entries() {

    let sender = utils::address_from_secret_key(&signer.secret_key);
    let caller: AccountId = CALLER_ACCOUNT_ID.parse().unwrap();
    let transaction = utils::transfer_with_price(

    set_silo_params(&mut runner, Some(SILO_PARAMS_ARGS));
    enable_all_whitelists(&mut runner);

    // Allow submitting transactions.
    add_account_to_whitelist(&mut runner, caller.clone());
    add_address_to_whitelist(&mut runner, sender);

    validate_address_balance_and_nonce(&runner, sender, INITIAL_BALANCE, INITIAL_NONCE.into())
        .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, ZERO_BALANCE, INITIAL_NONCE.into())
        .unwrap();

    // perform transfer
    let result = runner
        .submit_transaction(&signer.secret_key, transaction.clone())
        .unwrap();
    assert!(matches!(result.status, TransactionStatus::Succeed(_)));

    // validate post-state
    validate_address_balance_and_nonce(
        &runner,
        sender,
        INITIAL_BALANCE - TRANSFER_AMOUNT - FIXED_GAS * ONE_GAS_PRICE,
        (INITIAL_NONCE + 1).into(),

    .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, TRANSFER_AMOUNT, INITIAL_NONCE.into())

    remove_account_from_whitelist(&mut runner, caller);
    remove_address_from_whitelist(&mut runner, sender);

    let err = runner
