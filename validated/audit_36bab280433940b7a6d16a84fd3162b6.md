Audit Report

## Title
Ed25519 Small-Order Public Key Accepted in Ingress Verification Enables Signature Forgery for Derived Principal — (`rs/crypto/standalone-sig-verifier/src/lib.rs`, `packages/ic-ed25519/src/lib.rs`)

## Summary

`verify_basic_sig_by_public_key` calls `PublicKey::deserialize_raw` (which explicitly omits any subgroup/torsion check) followed by `verify_signature` (which uses a cofactor-based ZIP215 check). For a small-order public key such as `[0u8; 32]` (order 8) paired with the forged signature `(R=identity, S=0)`, the cofactor check is algebraically satisfied for any message, returning `Ok(())`. The ingress validation chain never calls `is_torsion_free()`, so an attacker can submit ingress messages that pass all authentication as the self-authenticating principal derived from the small-order key.

## Finding Description

**Root cause — `deserialize_raw` explicitly skips the subgroup check.**
The docstring on `PublicKey::deserialize_raw` warns: *"This does not verify that the key is within the prime order subgroup."* It calls only `VerifyingKey::from_bytes`, which checks the point is on the curve but not torsion-free. [1](#0-0) 

**`[0u8;32]` is a valid small-order point (order 8), confirmed by an existing test.** [2](#0-1) 

**`Signature::from_slice` accepts `(R=identity, S=0)`.** `CompressedEdwardsY([1,0,...,0])` decompresses to the identity point without error; `Scalar::from_canonical_bytes([0u8;32])` succeeds since 0 is a valid canonical scalar. No check rejects either value. [3](#0-2) 

**`verify_signature` passes for any message when `A` has order dividing 8 and `(R=identity, S=0)`.** The check is `(recomputed_r − R)·8 == identity`. With `S=0`, `recomputed_r = k·(−A)`. Since `8·A = identity` (order divides 8): `(k·(−A) − identity)·8 = k·8·(−A) − 8·identity = k·identity − identity = identity`. The check passes unconditionally for any `k` (i.e., any message). [4](#0-3) 

**The ingress path uses `verify_basic_sig_by_public_key`, which calls `deserialize_raw` then `verify_signature` with no torsion check in between.** [5](#0-4) 

**The ingress validation chain (`validate_user_id_and_signature` → `validate_user_id` → `validate_signature` → `validate_signature_plain` → `verify_basic_sig_by_public_key`) contains no call to `is_torsion_free()` at any step.** [6](#0-5) [7](#0-6) 

**The fix exists elsewhere but is not applied here.** `verify_public_key` and node key validation both call `is_torsion_free()` and would reject this key, but neither is invoked in the ingress path. [8](#0-7) [9](#0-8) 

## Impact Explanation

An attacker can forge valid ingress signatures for the self-authenticating principal derived from `SHA224(DER([0u8;32])) || 0x02`. This is a complete authentication bypass for that specific principal: any canister that has granted permissions (e.g., controller, allowlist, ownership) to this deterministic principal is fully compromised with zero key material. The principal is publicly computable, so the attack is repeatable and requires no per-target work beyond identifying whether the target canister has granted it access. This maps to **High ($2,000–$10,000): Unauthorized access to canister-controlled funds or identities where exploitation requires meaningful per-target constraints** — the constraint being that the target canister must have granted permissions to this specific principal.

## Likelihood Explanation

No privileged access, no key material, no network position, and no victim interaction are required. The attack is fully constructible offline. The forged principal is deterministic and publicly known. The practical blast radius is limited to canisters that have explicitly granted permissions to this principal; in practice this is unlikely but not impossible (e.g., a developer testing with a known-bad key, a canister initialized with a hardcoded principal derived from this key, or a governance proposal that inadvertently grants it access). The attack is repeatable and costless once identified.

## Recommendation

In `verify_basic_sig_by_public_key` (or in `PublicKey::deserialize_raw` when used for signature verification contexts), add a torsion check immediately after deserialization, mirroring the existing guard in `verify_public_key` and node key validation:

```rust
AlgorithmId::Ed25519 => {
    let pk = ic_ed25519::PublicKey::deserialize_raw(pk_bytes).map_err(|e| { ... })?;

    // Add this guard:
    if !pk.is_torsion_free() {
        return Err(CryptoError::MalformedPublicKey {
            algorithm: AlgorithmId::Ed25519,
            key_bytes: Some(pk_bytes.to_vec()),
            internal_error: "Public key has torsion component".to_string(),
        });
    }
    // ...
}
```

Alternatively, enforce the check inside `deserialize_raw` itself (or a separate `deserialize_raw_for_verification` variant) so the invariant is impossible to violate at call sites.

## Proof of Concept

use ic_standalone_sig_verifier::verify_basic_sig_by_public_key;
use ic_types::crypto::AlgorithmId;

// Step 1: small-order public key (order 8), passes deserialize_raw
let pk_raw = [0u8; 32];

// Step 2: forged signature — R = identity encoding, S = 0
let mut forged_sig = [0u8; 64];
forged_sig[0] = 1; // CompressedEdwardsY([1,0,...,0]) = identity point

// Step 3: verify_basic_sig_by_public_key returns Ok(()) for any message
let msg = b"transfer 1000 ICP to attacker";
assert!(verify_basic_sig_by_public_key(
    AlgorithmId::Ed25519,
    msg,
    &forged_sig,
    &pk_raw,
).is_ok());

    ingress_signature_verifier: &dyn IngressSigVerifier,
    sender: &UserId,
    message_id: &MessageId,
    signature: Option<&UserSignature>,
    current_time: Time,
    root_of_trust_provider: &R,
) -> Result<CanisterIdSet, RequestValidationError>
where
    R::Error: std::error::Error,
{
    match signature {
        None => {
            if sender.get().is_anonymous() {
                return Ok(CanisterIdSet::all());
            }
            Err(MissingSignature(*sender))
        }
        Some(signature) => {
            if sender.get().is_anonymous() {
                Err(AnonymousSignatureNotAllowed)
            } else {
                let sender_pubkey = &signature.signer_pubkey;
                validate_user_id(sender_pubkey, sender).and_then(|()| {
                    validate_signature(
                        ingress_signature_verifier,
                        message_id,
                        signature,
                        current_time,
                        root_of_trust_provider,
                    )
                })
            }
        }
    }
}
