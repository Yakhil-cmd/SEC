### Title
Unchecked `Yocto` Addition Overflow in XCC Precompile Allows Draining Engine's NEAR Balance - (File: engine-precompiles/src/xcc.rs)

### Summary

The `Yocto` type's `Add` implementation performs unchecked `u128` addition. In the XCC precompile, `attached_near + state::STORAGE_AMOUNT` can silently wrap to zero when `attached_near` is crafted near `u128::MAX`. This causes the engine to create a router sub-account and transfer `STORAGE_AMOUNT` (2 NEAR) from its own balance without collecting any payment from the caller, enabling a griefing attacker to drain the engine's NEAR balance.

---

### Finding Description

**Root cause — unchecked `Add` on `Yocto`:** [1](#0-0) 

The `Add` impl delegates directly to `self.0 + rhs.0` on the inner `u128`. In Rust release builds (including WASM), integer overflow wraps silently.

**Vulnerable call site in the XCC precompile:** [2](#0-1) 

`attached_near` is derived directly from user-supplied calldata via `call.total_near()`: [3](#0-2) 

`STORAGE_AMOUNT` is the fixed constant `Yocto::new(2_000_000_000_000_000_000_000_000)`: [4](#0-3) 

If an attacker supplies `attached_near = u128::MAX − STORAGE_AMOUNT + 1`, then:

```
required_near = (u128::MAX − STORAGE_AMOUNT + 1) + STORAGE_AMOUNT
              = u128::MAX + 1  →  wraps to 0
```

**Consequence — payment gate is bypassed:** [5](#0-4) 

Because `required_near == ZERO_YOCTO`, the `transferFrom` call that would collect wNEAR from the caller is skipped entirely. The log emits `required_near = 0`.

**Engine still pays `STORAGE_AMOUNT` for router creation:**

In `engine/src/xcc.rs`, `handle_precompile_promise` reads `required_near = 0` from the log and skips the `withdraw_wnear_to_router` step (and therefore the `send_refund` callback): [6](#0-5) 

But the setup batch — which transfers `STORAGE_AMOUNT` from the engine's own NEAR balance to the new router sub-account — is still executed unconditionally: [7](#0-6) 

Because `required_near == ZERO_YOCTO`, the `refund_needed` branch is never reached, so the router never sends `STORAGE_AMOUNT` back to the engine. The engine's NEAR balance is permanently reduced by 2 NEAR per attack iteration.

---

### Impact Explanation

Each attack iteration costs the engine `STORAGE_AMOUNT = 2_000_000_000_000_000_000_000_000 yoctoNEAR` (2 NEAR). The attacker can repeat the attack using different EVM addresses (each gets its own router sub-account). NEAR transaction gas costs are fractions of a cent, making this economically viable. Sustained repetition drains the engine's operational NEAR balance, eventually preventing it from paying for storage or cross-contract calls, which would freeze user funds.

**Impact class:** High — Temporary (escalating to permanent) freezing of funds.

---

### Likelihood Explanation

Any unprivileged EVM user can call the XCC precompile. No wNEAR balance is required because the overflow causes the `transferFrom` to be skipped. The only requirement is crafting a `CrossContractCallArgs::Eager` payload where `total_near()` returns a value satisfying `u128::MAX − STORAGE_AMOUNT < total_near ≤ u128::MAX`. This is trivially achievable by setting a single promise's attached NEAR to that value in the Borsh-encoded calldata.

---

### Recommendation

Replace the unchecked `Add` impl on `Yocto` with a checked variant, and use it at the call site:

```rust
// engine-types/src/types/balance.rs
impl Yocto {
    pub fn checked_add(self, rhs: Self) -> Option<Self> {
        self.0.checked_add(rhs.0).map(Self)
    }
}
```

In `engine-precompiles/src/xcc.rs`, replace:

```rust
None => attached_near + state::STORAGE_AMOUNT,
```

with:

```rust
None => attached_near
    .checked_add(state::STORAGE_AMOUNT)
    .ok_or_else(|| ExitError::Other(Cow::from("ERR_XCC_NEAR_OVERFLOW")))?,
```

Similarly audit and harden the unchecked `Sub` impl on `Yocto` and both unchecked `Add`/`Sub` impls on `NEP141Wei`: [8](#0-7) [9](#0-8) 

---

### Proof of Concept

1. Attacker computes `attack_near = u128::MAX − 2_000_000_000_000_000_000_000_000 + 1`.
2. Attacker constructs a `CrossContractCallArgs::Eager` Borsh payload where `total_near()` returns `attack_near` (e.g., a single `PromiseCreateArgs` with `attached_balance = Yocto::new(attack_near)`).
3. Attacker calls the XCC precompile address `0x516cded1d16af10cad47d6d49128e2eb7d27b372` with this payload from a fresh EVM address that has no existing router.
4. Inside the precompile: `required_near = attack_near + STORAGE_AMOUNT` wraps to `0`.
5. The `if required_near != ZERO_YOCTO` gate is false → no `transferFrom` is executed.
6. `handle_precompile_promise` sees `required_near = 0`, skips `withdraw_wnear_to_router` and `send_refund`.
7. The setup batch executes: engine creates the router sub-account and transfers 2 NEAR from its own balance.
8. Engine's NEAR balance decreases by 2 NEAR; attacker paid nothing.
9. Repeat with a different EVM address to drain further.
