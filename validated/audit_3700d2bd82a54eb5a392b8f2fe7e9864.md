### Title
Wrong Alignment Constant in `native_cost` for modexp Underestimates Native Resource Charge - (File: `basic_system/src/system_functions/modexp/mod.rs`)

### Summary
The `native_cost` function in the modexp precompile implementation uses `mod_size.next_multiple_of(32)` to round up the modulus size for native resource calculation, while the inline comment explicitly states the intent is `next_multiple_of(256)`. This discrepancy causes the native resource cost to be systematically underestimated for any modulus whose byte-length is not already a multiple of 256, allowing an unprivileged caller to consume more prover cycles than were charged.

### Finding Description
The `native_cost` function computes the native (proving) resource cost for the modexp precompile by re-invoking `ergs_cost` with the modulus size rounded up to the next alignment boundary. The comment reads:

```rust
// Use ergs for native calculation but with the next multiple of 256 for modulus,
// since we use bigint delegations.
let ergs = ergs_cost(
    base_size,
    exp_size,
    mod_size.next_multiple_of(32),   // ← uses 32, comment says 256
    exp_highp,
)?;
```

The stated rationale — "since we use bigint delegations" — refers to the fact that the bigint delegation subsystem processes data in fixed-size chunks; the comment documents that the correct chunk boundary for this purpose is 256 bytes. The code, however, passes `mod_size.next_multiple_of(32)` (32-byte alignment) instead of `mod_size.next_multiple_of(256)` (256-byte alignment).

Because `ergs_cost` computes `multiplication_complexity = ceil(max(base_size, mod_size) / 8)²`, the cost scales **quadratically** with the rounded modulus size. Rounding to 32 bytes instead of 256 bytes produces a much smaller effective modulus for the cost formula:

| Actual `mod_size` | `next_multiple_of(32)` | `next_multiple_of(256)` | Ratio of `words²` |
|---|---|---|---|
| 33 bytes | 64 | 256 | ~16× undercharge |
| 64 bytes | 64 | 256 | ~16× undercharge |
| 128 bytes | 128 | 256 | ~4× undercharge |
| 200 bytes | 224 | 256 | ~1.3× undercharge |
| 256 bytes | 256 | 256 | correct | [1](#0-0) 

The ergs (EVM gas) cost path is unaffected — it calls `ergs_cost` directly with the raw `mod_size` — so the EVM gas charge is correct. Only the native resource charge is wrong. [2](#0-1) 

### Impact Explanation
The native resource models the off-chain proving cost ("RISC-V cycles to prove a computation"). It is charged from the transaction's native budget (`gas_limit × native_per_gas`). If the native cost for a modexp call is underestimated, the call succeeds while consuming more prover cycles than were budgeted. An attacker can craft repeated modexp calls with moduli in the 33–255 byte range to exhaust the block's native resource budget at a fraction of the true cost, potentially causing the block to be unproducible (prover runs out of native budget mid-block) or allowing a denial-of-service against the proving pipeline without paying the commensurate fee. [3](#0-2) 

### Likelihood Explanation
The modexp precompile (address `0x0000…0005`) is a standard EVM precompile callable by any unprivileged transaction. Moduli of 33–255 bytes are common in RSA and other cryptographic protocols. No special privilege, governance access, or oracle manipulation is required. The attacker only needs to send a transaction calling the precompile with a suitably sized modulus. [4](#0-3) 

### Recommendation
Change the alignment constant in `native_cost` from `32` to `256` to match the documented intent:

```rust
// Before
mod_size.next_multiple_of(32)

// After
mod_size.next_multiple_of(256)
```

This ensures the native resource charge accounts for the actual bigint delegation chunk size used by the prover. [5](#0-4) 

### Proof of Concept
1. Construct a modexp call: `base_size = 1`, `exp_size = 1`, `mod_size = 33`, `exp = 0x01` (so `exp_highp` has 1 bit set, `iteration_count = 1`).
2. **Actual native cost** (per comment intent, `mod_size → 256`):
   - `words = ceil(256/8) = 32`; `multiplication_complexity = 32² = 1024`
   - `iteration_count = 1`; `computed_gas = 1024/3 ≈ 341`; `gas = max(200, 341) = 341`
   - `native = 341 × MODEXP_WORST_CASE_NATIVE_PER_GAS (300) = 102,300`
3. **Charged native cost** (current code, `mod_size → 64`):
   - `words = ceil(64/8) = 8`; `multiplication_complexity = 8² = 64`
   - `iteration_count = 1`; `computed_gas = 64/3 ≈ 21`; `gas = max(200, 21) = 200`
   - `native = 200 × 300 = 60,000`
4. The prover is charged **60,000** native units but must perform work worth **102,300** native units — a **~41% undercharge** for this input alone. With a 33-byte modulus the discrepancy is ~16× in `multiplication_complexity`, yielding up to a ~16× undercharge in the native cost before the `max(200, …)` floor kicks in. [6](#0-5) [7](#0-6)

/// Computes the ergs cost for modexp.
/// Returns an OOG error if there's an arithmetic overflow.
pub fn ergs_cost(
    base_size: u64,
    exp_size: u64,
    mod_size: u64,
    exp_highp: &U256,
) -> Result<Ergs, SystemError> {
    let multiplication_complexity = {
        let max_length = core::cmp::max(base_size, mod_size);
        let words = max_length.div_ceil(8);
        words.checked_mul(words).ok_or(out_of_ergs_error!())?
    };
    let iteration_count = {
        let ic = if exp_size <= 32 && exp_highp.is_zero() {
            0
        } else if exp_size <= 32 {
            exp_highp.bit_len() as u64 - 1
        } else {
            8u64.checked_mul(exp_size - 32)
                .ok_or(out_of_ergs_error!())?
                .checked_add(core::cmp::max(1, exp_highp.bit_len() as u64) - 1)
                .ok_or(out_of_ergs_error!())?
        };
        core::cmp::max(1, ic)
    };
    let computed_gas = multiplication_complexity
        .checked_mul(iteration_count)
        .ok_or(out_of_ergs_error!())?
        .checked_div(3)
        .ok_or(out_of_ergs_error!())?;
    let gas = core::cmp::max(200, computed_gas);
    let ergs = gas.checked_mul(ERGS_PER_GAS).ok_or(out_of_ergs_error!())?;
    Ok(Ergs(ergs))
}
