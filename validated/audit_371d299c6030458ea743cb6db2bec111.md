### Title
Controller Can Drain Any Token Holder via `icrc152_burn` + `icrc152_mint` Without Consent — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

---

### Summary

The ICRC-152 extension of the ICRC-1 ledger canister exposes `icrc152_burn` and `icrc152_mint` endpoints that are gated solely by `ic_cdk::api::is_controller(&caller)`. A malicious or compromised controller can burn tokens from **any** account without the account holder's consent and simultaneously mint tokens to themselves — effectively stealing value from all token holders. No ICRC-2 allowance or user approval is required. This is a structural analog to the BribeVault `DEFAULT_ADMIN_ROLE` rug vector.

---

### Finding Description

The `icrc152_burn_not_async` function in `rs/ledger_suite/icrc1/ledger/src/main.rs` accepts a caller-supplied `args.from` account and burns tokens from it using the `AuthorizedBurn` operation, which explicitly bypasses the ICRC-2 allowance mechanism:

```rust
fn icrc152_burn_not_async(caller: Principal, args: Icrc152BurnArgs) -> Result<u64, Icrc152BurnError> {
    let block_idx = Access::with_ledger_mut(|ledger| {
        if !ic_cdk::api::is_controller(&caller) {          // only guard
            return Err(Icrc152BurnError::Unauthorized(...));
        }
        // ... no allowance check, no account-holder consent ...
        let tx = Transaction {
            operation: Operation::AuthorizedBurn {
                from: args.from,   // arbitrary victim account
                amount,
                ...
            },
        };
``` [1](#0-0) 

The `AuthorizedBurn` operation is explicitly documented as bypassing the transfer API and consuming no allowance: [2](#0-1) 

The symmetric `icrc152_mint_not_async` function is identically gated — only by `is_controller` — and mints tokens to any caller-supplied `args.to` account: [3](#0-2) 

Both endpoints are exposed as public update methods on the ledger canister: [4](#0-3) 

---

### Impact Explanation

A malicious or compromised controller executes a two-step rug:

1. **`icrc152_burn({ from: victim_account, amount: victim_balance, ... })`** — destroys the victim's entire token balance without any consent, allowance, or notification.
2. **`icrc152_mint({ to: attacker_account, amount: victim_balance, ... })`** — mints an equivalent amount to the attacker.

Net effect: the attacker transfers value from the victim to themselves. Unlike the BribeVault case (which required the victim to have pre-approved the contract), ICRC-152 requires **no prior action from the victim** — every token holder on every ICRC-152-enabled ledger is at risk. The `AuthorizedBurn` operation leaves no allowance trail and does not deduct from any existing ICRC-2 approval: [5](#0-4) 

---

### Likelihood Explanation

The exploit requires a malicious or compromised controller of an ICRC-152-enabled ICRC-1 ledger. This is the same trust assumption as the BribeVault `DEFAULT_ADMIN_ROLE` finding (rated Medium by the judge). On the IC, canister controllers are typically a small set of principals (often a governance canister, a developer key, or a multi-sig). A compromised developer key or a malicious SNS/NNS proposal that adds a rogue controller is a realistic threat. The ICRC-152 feature flag is opt-in but, once enabled, the rug vector is unconditional for all token holders.

---

### Recommendation

1. **Require ICRC-2 allowance for `icrc152_burn`**: The `from` account should have explicitly approved the controller (or a designated burner canister) before tokens can be burned from it. This mirrors the ICRC-2 `transfer_from` model.
2. **Separate the burn authority from the controller role**: Introduce a dedicated, narrowly-scoped `authorized_burner` principal (e.g., set at init time) rather than granting all controllers unrestricted burn power.
3. **Use a governance-controlled multi-sig for the controller role** of any ICRC-152-enabled ledger, so that no single key can unilaterally drain accounts.
4. **Emit a pre-burn notification or time-lock** so that token holders can react before their balance is destroyed.

---

### Proof of Concept

**Setup**: An ICRC-152-enabled ICRC-1 ledger is deployed. Alice holds 1,000,000 tokens. The attacker controls one of the ledger's controller principals.

**Step 1** — Burn Alice's tokens (no approval needed):
```
icrc152_burn({
    from: { owner: alice_principal, subaccount: null },
    amount: 1_000_000,
    created_at_time: <now>,
    reason: null
})
```
Alice's balance drops to 0. No allowance is checked or consumed.

**Step 2** — Mint equivalent tokens to the attacker:
```
icrc152_mint({
    to: { owner: attacker_principal, subaccount: null },
    amount: 1_000_000,
    created_at_time: <now + 1>,
    reason: null
})
```
Attacker's balance increases by 1,000,000.

The existing test suite confirms that `AuthorizedBurn` does not consume any allowance even when one exists: [6](#0-5) 

And the unauthorized-caller test confirms the only guard is `is_controller`: [7](#0-6)
