Audit Report

## Title
ckERC20 Minter Over-Mints Tokens for Fee-on-Transfer ERC-20 Deposits Due to Trusting Event-Emitted Amount - (File: `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

Both Ethereum-side helper contracts emit the caller-supplied `amount` parameter in their deposit events rather than the actual tokens received by the minter address. The IC minter scrapes these events and mints exactly `event.value()` of ckERC20 without any balance-before/after verification. For fee-on-transfer ERC-20 tokens, the minter receives `amount - fee` but mints `amount`, permanently undercollateralizing the ckERC20 supply.

## Finding Description

**Root cause — Solidity contracts emit requested amount, not received amount:**

`ERC20DepositHelper.sol` lines 498–503:
```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
```
`DepositHelperWithSubaccount.sol` lines 511–532:
```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```
Both emit the caller-supplied `amount`, not `balanceAfter - balanceBefore`.

**Log parsing — value taken verbatim from event data:**

`parser.rs` `ReceivedErc20LogParser::parse_log` (lines 86–97) reads `value_bytes` from `entry.data` and stores it as `Erc20Value::from_be_bytes(value_bytes)` into `ReceivedErc20Event.value`. No cross-check against on-chain balances is performed.

**Minting — trusts the log-emitted value unconditionally:**

`deposit.rs` lines 73–81:
```rust
let block_index = match client
    .transfer(TransferArg {
        ...
        amount: event.value(),  // directly from the log
    })
    .await
```
There is no guard that verifies the minter's actual ERC-20 balance increased by `event.value()` before minting. The `prevent_double_minting_guard` only prevents re-processing the same event; it does not validate the amount.

**Exploit flow:**
1. A fee-on-transfer ERC-20 token (e.g., 1% fee) is whitelisted via NNS proposal.
2. Attacker calls `depositErc20(feeToken, 1_000_000, principal, subaccount)`.
3. Helper calls `safeTransferFrom(attacker, minter, 1_000_000)` — minter receives `990_000`.
4. Helper emits `ReceivedEthOrErc20(feeToken, attacker, 1_000_000, ...)`.
5. IC minter scrapes log, reads `value = 1_000_000`, mints `1_000_000` ckFeeToken.
6. Minter holds `990_000` ERC-20 but has issued `1_000_000` ckERC20 — 10,000 tokens unbacked per deposit.
7. Repeated deposits compound the deficit; eventual withdrawers cannot redeem their ckERC20 1:1.

**Existing checks are insufficient:** The whitelist check in `deposit.rs` lines 55–67 only validates that the ERC-20 contract address is a supported token; it does not validate the deposited amount against actual on-chain balances.

## Impact Explanation

This is illegal minting of in-scope chain-key/ck-token assets (ckERC20), directly causing protocol insolvency for the affected ckERC20 token. The ckERC20 supply exceeds the ERC-20 collateral held by the minter, breaking the 1:1 peg guarantee. Users who withdraw later receive less ERC-20 than their ckERC20 represents, or withdrawals fail entirely — constituting direct loss of user funds. This matches the **High** impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

The attack requires a fee-on-transfer ERC-20 token to be whitelisted. Currently deployed tokens (USDC, USDT, LINK) do not have active transfer fees. However: (1) USDT contains a dormant fee mechanism activatable by the USDT issuer at any time; (2) any future NNS proposal adding a fee-on-transfer token silently triggers the bug; (3) once a qualifying token is whitelisted, the exploit path is fully unprivileged — any user calling `depositErc20` on the helper contract triggers over-minting. **Likelihood: Medium.**

## Recommendation

1. **Solidity contracts**: Record the minter's ERC-20 balance before and after `safeTransferFrom` and emit `balanceAfter - balanceBefore` as the amount in the event instead of the caller-supplied `amount`.
2. **IC minter**: As defense-in-depth, when processing a deposit event, query the minter's on-chain ERC-20 balance and verify it is consistent with the expected collateral before minting. Alternatively, enforce at token-addition governance proposal validation that only non-fee-on-transfer tokens may be added as supported ckERC20 tokens.

## Proof of Concept

Deploy a local ERC-20 token with a 1% fee-on-transfer. Whitelist it as a ckERC20 token in a local replica. Call `depositErc20(feeToken, 1_000_000, principal, subaccount)` on the helper contract. Observe that the minter's ERC-20 balance increases by `990_000` while the ckERC20 ledger records a mint of `1_000_000`. Repeat N times and verify that total ckERC20 supply exceeds total ERC-20 held by the minter by `N * 10_000`. Attempt a full withdrawal of all ckERC20 and observe that the last withdrawal fails due to insufficient ERC-20 balance.
