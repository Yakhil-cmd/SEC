Audit Report

## Title
SNS Swap Neuron ID Pre-Registration Can Block Finalization for Direct Participants - (File: `rs/sns/governance/src/governance.rs`, `rs/sns/swap/src/swap.rs`)

## Summary
SNS neuron IDs for direct swap participants are computed deterministically from `(buyer_principal, memo)`. The swap reserves memos in `[NEURON_BASKET_MEMO_RANGE_START, SALE_NEURON_MEMO_RANGE_END]` = `[1_000_000, 10_000_000]` for swap-issued neurons. The SNS governance `claim_or_refresh_neuron_by_memo_and_controller` endpoint â€” callable by any user â€” performs no check against this reserved range, allowing an attacker to pre-register a neuron for any principal using a memo in the reserved range. When swap finalization calls `claim_swap_neurons`, the pre-registered neuron ID collides, causing the affected participant's entire basket to fail and their swap allocation to become permanently inaccessible.

## Finding Description

**Constants confirmed in source:**

`NEURON_BASKET_MEMO_RANGE_START = 1_000_000` and `SALE_NEURON_MEMO_RANGE_END = 10_000_000` are defined at `rs/sns/swap/src/swap.rs` lines 89â€“90.

**Memo assignment confirmed:**

`create_sns_neuron_basket_for_direct_participant` is called with `NEURON_BASKET_MEMO_RANGE_START` as `memo_offset`, and each basket neuron receives `memo = memo_offset + i` (`rs/sns/swap/src/swap.rs` line 3321). The first neuron in every direct participant's basket always uses `memo = 1_000_000`.

**Missing guard confirmed:**

`claim_or_refresh_neuron_by_memo_and_controller` (`rs/sns/governance/src/governance.rs` lines 4210â€“4227) accepts any memo value from the caller without range validation. The `controller` field is taken directly from the request (`memo_and_controller.controller.unwrap_or(*caller)`) with no requirement that `controller == caller`, meaning an attacker can specify any victim principal as the controller.

**Developer neuron protection does not cover this path:**

The range check at `rs/sns/init/src/distributions.rs` lines 219â€“228 only applies to developer neurons at SNS init time. No equivalent guard exists in the user-facing `manage_neuron` â†’ `ClaimOrRefreshNeuron` path.

**`new_neuron_id` only checks existence:**

`rs/sns/governance/src/governance.rs` lines 835â€“848 confirms `new_neuron_id` only rejects already-used IDs; it does not reject memos in the reserved range.

**Attack path (targeting another participant):**

1. Attacker identifies victim principal `P` who will participate in the swap.
2. Attacker transfers the minimum neuron stake to ledger subaccount `compute_neuron_staking_subaccount_bytes(P, 1_000_000)` of the SNS governance canister (any principal can send to any ledger account).
3. Attacker calls `manage_neuron` â†’ `ClaimOrRefreshNeuron { controller: P, memo: 1_000_000 }` on SNS governance. Since `controller` is not required to equal `caller`, this succeeds and creates a neuron owned by `P` with `NeuronId = hash(P, 1_000_000)`.
4. Swap finalizes; `create_sns_neuron_basket_for_direct_participant` generates a recipe for `P` with `memo = NEURON_BASKET_MEMO_RANGE_START = 1_000_000`, producing the identical `NeuronId`.
5. `claim_swap_neurons` in SNS governance encounters the pre-existing neuron ID and returns a failure for that recipe.
6. The swap records `sweep_result.failure` for the affected recipes (`rs/sns/swap/src/swap.rs` lines 872â€“881). The SNS tokens allocated to `P`'s basket are permanently inaccessible; the swap has no recovery path for this case.

**Critical precondition:** The exploit requires `ClaimOrRefreshNeuron` to be callable during the SNS governance mode active while the swap is open (likely `PreInitializationSwap`). The claim acknowledges this is plausible â€” developer and airdrop neurons must be claimable during this phase â€” but this specific mode restriction was not fully confirmed from the available code. If `PreInitializationSwap` mode blocks `ClaimOrRefreshNeuron`, the window narrows to any mode transition that permits it before finalization.

## Impact Explanation

An unprivileged attacker can permanently destroy any direct swap participant's SNS token allocation at the cost of a minimum neuron stake (a small, fixed amount). The victim loses their entire swap allocation â€” ICP already contributed is collected by the swap, and the SNS tokens that should have been distributed as neurons become permanently stuck. This constitutes a concrete, targeted permanent loss of user funds in the SNS framework, matching the High impact class: **"Significant SNS security impact with concrete user or protocol harm"** ($2,000â€“$10,000).

## Likelihood Explanation

The attacker is any unprivileged external user. The target neuron ID is fully deterministic and publicly computable from the victim's principal and the constant `NEURON_BASKET_MEMO_RANGE_START`. The cost to execute is the minimum neuron stake plus ledger fees. The only constraint is the SNS governance mode restriction during the swap period, which the claim argues is likely permissive for neuron claiming. The attack is repeatable against any participant whose principal is known in advance.

## Recommendation

Add a memo range guard in `claim_or_refresh_neuron_by_memo_and_controller` (and `new_neuron_id`) in `rs/sns/governance/src/governance.rs` that rejects memos in `[NEURON_BASKET_MEMO_RANGE_START, SALE_NEURON_MEMO_RANGE_END]`, mirroring the existing check in `rs/sns/init/src/distributions.rs` lines 219â€“228. Alternatively, `claim_swap_neurons` in SNS governance should treat `AlreadyExists` as a recoverable condition by verifying the existing neuron belongs to the correct swap and merging the stake, rather than treating it as a hard failure.

## Proof of Concept

1. Deploy a local SNS with a swap configured for direct participation.
2. Register principal `P` as a swap participant with a sufficient ICP contribution.
3. Before swap finalization, transfer the minimum neuron stake to `compute_neuron_staking_subaccount_bytes(P, 1_000_000)` of the SNS governance ledger from any principal.
4. Call `manage_neuron` on SNS governance with `ClaimOrRefreshNeuron { controller: Some(P), memo: 1_000_000 }` from any caller.
5. Confirm a neuron with `NeuronId = hash(P, 1_000_000)` now exists in SNS governance.
6. Trigger swap finalization.
7. Observe that `claim_swap_neurons` returns a failure for `P`'s first basket neuron and `sweep_result.failure` is incremented.
8. Confirm `P`'s SNS token allocation is permanently inaccessible with no recovery path in the swap canister.
