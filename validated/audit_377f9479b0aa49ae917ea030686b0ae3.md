Audit Report

## Title
Unchecked ICP Burn Return Value in Cycles Minting Canister Allows ICP/Cycles Conservation Violation - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally returns `()` regardless of whether the ICP ledger burn succeeds or fails. Both `process_top_up` and `process_mint_cycles` call `burn_and_log` after a successful cycle deposit and immediately return `Ok(...)`, causing the block to be permanently marked as fully processed even when the burn failed. If the ICP ledger is transiently unavailable (e.g., during a routine NNS ledger upgrade), cycles are minted and deposited while the corresponding ICP is never burned, violating the ICP/cycles conservation invariant.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` (lines 2017–2049) makes a protobuf call to the ICP ledger (`send_pb`) and on failure only logs the error, returning `()` in all cases:

```rust
match res {
    Ok(block) => print(format!("{msg} done in block {block}.")),
    Err((code, err)) => {
        let code = code as i32;
        print(format!("{msg} failed with code {code}: {err:?}"))
    }
}
```

The function signature `async fn burn_and_log(...) -> ()` propagates no error to callers.

In `process_top_up` (lines 1999–2003) and `process_mint_cycles` (lines 1966–1974), after a successful cycle deposit, `burn_and_log` is awaited and its `()` return is discarded; both functions then return `Ok(...)` unconditionally:

```rust
// process_top_up
Ok(()) => {
    burn_and_log(sub, amount).await;
    Ok(cycles)
}
// process_mint_cycles
Ok(deposit_result) => {
    burn_and_log(sub, amount).await;
    Ok(NotifyMintCyclesSuccess { ... })
}
```

After `process_top_up` returns `Ok(cycles)`, `notify_top_up` writes `NotificationStatus::NotifiedTopUp(Ok(cycles))` to `blocks_notified` (lines 1214–1222). The `is_transient_error` guard at line 1125–1130 only removes the entry when the result is an `Err` variant — since the result is `Ok`, the block is permanently finalized. Any subsequent call for the same block index returns the cached `Ok` result (line 1187), making the burn permanently unretriable through the normal notification path.

The comment at lines 2014–2016 acknowledges the intentional design: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* This reasoning prevents double-minting but simultaneously prevents burn recovery, leaving ICP permanently unburned in the CMC subaccount.

## Impact Explanation
This is a ledger conservation bug. When triggered, cycles are created without corresponding ICP being burned. The ICP remains stranded in the CMC's subaccount (inaccessible to any user), while the total circulating cycles supply exceeds what the burned ICP supply should support. Over repeated occurrences across ledger upgrades, the ICP/cycles conservation invariant — a core protocol property — is cumulatively violated. This constitutes illegal minting (cycles created without burned ICP) and a significant NNS/ledger security impact with concrete protocol harm, qualifying as **High** severity.

## Likelihood Explanation
The ICP ledger undergoes routine upgrades via NNS governance proposals, which are public and observable on-chain. During an upgrade, the ledger canister briefly stops accepting calls. Any `notify_top_up` or `notify_mint_cycles` call that reaches the `burn_and_log` step during this window will silently fail the burn. An unprivileged user who has pre-positioned ICP in the CMC subaccount and submits a notification call timed to the upgrade window can trigger this condition. The window is short (seconds) but the condition is deterministic and repeatable across every ledger upgrade cycle.

## Recommendation
`burn_and_log` should propagate burn failures to its callers. The recommended fix is to introduce a `BurnPending` or equivalent intermediate `NotificationStatus` that is written before the burn is attempted. If the burn fails, the block remains in this intermediate state rather than being finalized, allowing the burn to be retried via a heartbeat/timer or a subsequent user notification call. At minimum, the block status must distinguish "cycles deposited, burn pending" from "fully finalized" to allow recovery without enabling double-minting.

## Proof of Concept
1. User transfers ICP to `AccountIdentifier::new(CYCLES_MINTING_CANISTER_ID, Some(Subaccount::from(&target_canister_id)))` with `MEMO_TOP_UP_CANISTER`.
2. User monitors NNS governance proposals for an upcoming ICP ledger upgrade.
3. User calls `notify_top_up { block_index, canister_id }` on the CMC, timed to arrive at the `burn_and_log` step while the ledger is mid-upgrade.
4. CMC verifies the transaction via `fetch_transaction` — succeeds (ledger is readable at this point).
5. CMC sets block status to `Processing`.
6. CMC calls `process_top_up` → `deposit_cycles(canister_id, cycles, ...)` — **succeeds**, cycles deposited.
7. CMC calls `burn_and_log(sub, amount).await` — ledger rejects the call (mid-upgrade).
8. `burn_and_log` logs the failure and returns `()`.
9. `process_top_up` returns `Ok(cycles)`.
10. CMC writes `blocks_notified[block_index] = NotifiedTopUp(Ok(cycles))` — **permanently**.
11. CMC returns `Ok(cycles)` to the user.

**Result**: Target canister received cycles. ICP was not burned. Block is permanently marked as processed. ICP remains stranded in CMC subaccount. Conservation invariant violated. A deterministic integration test using PocketIC can reproduce this by pausing the mock ledger canister between the `deposit_cycles` call and the `burn_and_log` call, then verifying that the CMC subaccount retains the ICP while the target canister's cycle balance increased.
