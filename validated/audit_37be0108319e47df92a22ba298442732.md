Audit Report

## Title
Logo-Omission Guard Bypassed in `convert_proposal` When `return_self_describing_action=true` — (File: `rs/nns/governance/src/pb/proposal_conversions.rs`)

## Summary
In `convert_proposal`, `is_create_service_nervous_system_proposal` is derived from the already-conditionally-suppressed `action` field rather than from the original `pb::Proposal` action. When `return_self_describing_action=true`, `show_action` is `false`, so `action` is always `None`, making `is_create_service_nervous_system_proposal` always `false`. This permanently disables the logo-omission guard for `self_describing_action`, causing full logo bytes to be returned even when `omit_large_fields=Some(true)` is explicitly requested via `list_proposals`.

## Finding Description
`ProposalDisplayOptions::for_list_proposals` sets `show_action = !return_self_describing_action` (line 24). When `return_self_describing_action=true`, `show_action=false`.

In `convert_proposal` (lines 495–499), `action` is set to `None` when `show_action()` is false:
```rust
let action = if display_options.show_action() {
    action.as_ref().map(|x| convert_action(x, display_options))
} else {
    None
};
```

Then at lines 500–502, `is_create_service_nervous_system_proposal` is derived from this already-`None` value:
```rust
let is_create_service_nervous_system_proposal = action.as_ref().is_some_and(|action| {
    matches!(action, api::proposal::Action::CreateServiceNervousSystem(_))
});
```

Since `action` is always `None` in this branch, `is_create_service_nervous_system_proposal` is always `false`. The guard passed to `convert_self_describing_action` at lines 509–510 evaluates to `false && true = false`:
```rust
is_create_service_nervous_system_proposal
    && display_options.omit_create_service_nervous_system_large_fields()
```

Inside `convert_self_describing_action` (lines 317–328), when `omit_create_service_nervous_system_logos=false`, `paths_to_omit` is an empty `vec![]`, so `logo` and `token_logo` fields pass through unfiltered. `omit_create_service_nervous_system_large_fields()` (lines 59–61) correctly returns `true` when `omit_large_fields_requested=true && multi_query=true`, but is never reached due to the short-circuit.

The existing test `test_self_describing_value_omit_logos` (lines 745–829) only tests `convert_self_describing_action` in isolation with a hardcoded boolean and does not exercise the `convert_proposal` integration path.

## Impact Explanation
This is an application-level DoS on the NNS governance canister's `list_proposals` query. The `omit_large_fields` guard exists specifically to keep `list_proposals` responses within the IC's 2 MB query response limit. A `CreateServiceNervousSystem` proposal logo can be up to ~256 KiB (base64). With multiple such proposals in a single `list_proposals` response window, the combined payload can exceed 2 MB, causing response serialization to fail and the query to trap. This matches the allowed High impact: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
`list_proposals` is a public, unauthenticated query call — any caller can set both `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)`. `CreateServiceNervousSystem` proposals are persistent in governance state; the NNS mainnet has had multiple real such proposals, each potentially carrying logo blobs. A handful of proposals with large logos in the same response window is sufficient to trigger the oversized response. No special privileges are required.

## Recommendation
Derive `is_create_service_nervous_system_proposal` from the original `pb::Proposal` action field, not from the already-conditionally-suppressed converted `action`. In `convert_proposal`, replace lines 500–502 with:

```rust
let is_create_service_nervous_system_proposal = item.action.as_ref().is_some_and(|action| {
    matches!(action, pb::proposal::Action::CreateServiceNervousSystem(_))
});
```

This decouples type-detection from display-suppression logic. Additionally, add an integration-level test for `convert_proposal` that exercises the `show_action=false, show_self_describing_action=true, omit_large_fields=true` path and asserts that logos are `Null` in the returned `self_describing_action`.

## Proof of Concept
1. Submit a `CreateServiceNervousSystem` proposal with a 256 KiB base64-encoded logo to NNS governance.
2. Call `list_proposals` with `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)`.
3. Observe that the returned `self_describing_action.value` map contains the full logo bytes under the `"logo"` key (not `Null`), confirming the guard is bypassed.
4. With enough such proposals in the response window, the total response size exceeds 2 MB and the query traps.

For a unit-level proof: construct a `pb::Proposal` with a `CreateServiceNervousSystem` action containing a large logo, call `convert_proposal` with `ProposalDisplayOptions::for_list_proposals(true, true)`, and assert that `self_describing_action.value["logo"]` is `api::SelfDescribingValue::Null` — this assertion will fail against the current code, confirming the bug.
