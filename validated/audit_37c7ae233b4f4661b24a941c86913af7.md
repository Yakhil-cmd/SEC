Audit Report

## Title
ckERC20 Minter Mints Requested Amount Instead of Actual Received Amount for Fee-on-Transfer ERC-20 Tokens, Causing Protocol Insolvency - (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

Both ckERC20 deposit helper contracts emit the caller-supplied `amount` in their log events rather than the actual tokens received by the minter address after `safeTransferFrom`. For fee-on-transfer ERC-20 tokens, the minter address receives `amount - fee` but the event records `amount`. The IC minter reads the emitted value and mints that many ckERC20 tokens, creating a permanent deficit: total ckERC20 supply exceeds the ERC-20 balance held at the minter's Ethereum address. USDT — already a supported ckERC20 token — contains a dormant fee mechanism (`basisPointsRate`, `maximumFee`) that Tether can enable at any time without any protocol-side warning or guard.

## Finding Description

**Step 1 — Helper contracts emit requested amount, not actual received amount.**

In `ERC20DepositHelper.sol`, `deposit()` calls `safeTransferFrom(msg.sender, cketh_minter_main_address, amount)` and then emits `ReceivedErc20(..., amount, ...)` using the same `amount` parameter with no balance-before/balance-after check:

```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
```

The identical pattern exists in `DepositHelperWithSubaccount.sol` `depositErc20()`:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```

For a fee-on-transfer token, `minterAddress` receives `amount - fee` but the event records `amount`.

**Step 2 — The IC minter parses the emitted value and mints that exact amount.**

`ReceivedErc20LogParser::parse_log` in `parser.rs` reads `value_bytes` directly from the event data field and constructs `ReceivedErc20Event { value: Erc20Value::from_be_bytes(value_bytes), ... }`. No cross-check against on-chain balance is performed.

In `deposit.rs`, `mint()` calls `client.transfer(TransferArg { amount: event.value(), ... })`, minting the emitted amount verbatim to the beneficiary.

**Step 3 — No guard exists at token registration.**

`add_ckerc20_token` in `main.rs` delegates to `erc20::CkErc20Token::try_from`, which validates only chain ID, address format, and token symbol format. `record_add_ckerc20_token` in `state.rs` checks only for network match, duplicate symbol, duplicate ledger ID, and duplicate ERC-20 address. No check for fee-on-transfer properties exists anywhere in the registration path.

**Step 4 — USDT is already a supported ckERC20 token.**

The documentation at `rs/ethereum/cketh/docs/ckerc20.adoc` lists USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) as a supported mainnet ckERC20 token. The USDT contract contains `basisPointsRate` and `maximumFee` storage variables that are currently set to zero but are settable by the Tether owner at any time. If Tether enables the fee, every subsequent deposit via either helper contract will mint more ckUSDT than the minter holds in USDT, with the deficit accumulating permanently.

## Impact Explanation

This is a **High** impact finding matching: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."* When the fee is active, the ckUSDT total supply will exceed the USDT balance at the minter's Ethereum address by the cumulative fee amount across all deposits. Users who attempt to withdraw ckUSDT last will find insufficient USDT at the minter address and will be permanently unable to redeem. The deficit is irreversible because the minter has no mechanism to detect or correct the accounting discrepancy after the fact. At scale (e.g., 1% fee on $100M deposited), the unrecoverable loss exceeds $1M, which would qualify as Critical.

## Likelihood Explanation

USDT is already a supported ckERC20 token, so no governance action is required to reach the vulnerable state. The only precondition is Tether enabling their own contract's built-in fee mechanism — a legitimate action by the token issuer, not a compromise. Once the fee is non-zero, any ordinary user calling `depositErc20` on either helper contract triggers the accounting discrepancy. No privileged access, no special conditions, and no victim mistakes are required. The exploit is repeatable on every deposit.

## Recommendation

1. **In both Solidity helper contracts** (`ERC20DepositHelper.sol` L498–503, `DepositHelperWithSubaccount.sol` L511–532): Replace the direct `amount` emission with a balance-before/balance-after pattern:

```solidity
uint256 balanceBefore = IERC20(erc20Address).balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = IERC20(erc20Address).balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

2. **In `add_ckerc20_token` / `record_add_ckerc20_token`**: Add a governance-level check or documentation requirement that fee-on-transfer tokens are not supported, and reject tokens whose current transfer fee is non-zero at registration time.

## Proof of Concept

1. USDT fee is enabled by Tether (sets `basisPointsRate = 10`, i.e., 0.1% fee). USDT is already a supported ckERC20 token — no additional governance step needed.
2. Alice calls `depositErc20(USDT_ADDRESS, 1_000_000, alicePrincipal, subaccount)` on the `CkDeposit` helper contract.
3. `safeTransferFrom(Alice, minterAddress, 1_000_000)` executes; due to the 0.1% fee, `minterAddress` receives 999,000 USDT (assuming `maximumFee` is not binding).
4. The helper emits `ReceivedEthOrErc20(USDT_ADDRESS, Alice, 1_000_000, ...)` — recording 1,000,000, not 999,000.
5. The IC minter scrapes the log, reads `value = 1_000_000`, and mints 1,000,000 ckUSDT to Alice.
6. After N deposits totaling 1,000,000,000 USDT requested, the minter holds 999,000,000 USDT but has minted 1,000,000,000 ckUSDT — a 1,000,000 USDT deficit.
7. The last depositors cannot redeem their ckUSDT; the minter's Ethereum address is insolvent for USDT.

A deterministic integration test can be written using the existing `CkErc20Setup` state-machine test harness in `rs/ethereum/cketh/minter/tests/ckerc20.rs` by deploying a mock ERC-20 token that deducts a fee in `transferFrom`, depositing via the helper, and asserting that the minted ckERC20 amount exceeds the mock token balance at the minter address.
