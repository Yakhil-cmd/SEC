### Title
ckERC20 Minter Over-Mints When Deflationary ERC-20 Token Is Deposited — (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

Both ckERC20 helper smart contracts emit a deposit event carrying the **caller-supplied `amount`** parameter rather than the **actual tokens received** by the minter address. The IC ckETH minter canister reads that event value verbatim and mints the same quantity of ckERC20 tokens on the IC ledger. For any ERC-20 token that silently transfers less than the requested amount (deflationary / fee-on-transfer tokens), the minter mints more ckERC20 than the ERC-20 reserve it actually holds, permanently breaking the 1:1 backing invariant.

---

### Finding Description

**Step 1 — Helper contracts emit the requested amount, not the received amount.**

`ERC20DepositHelper.sol` (`deposit`):
```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);   // ← `amount`, not actual received
``` [1](#0-0) 

`DepositHelperWithSubaccount.sol` (`depositErc20`):
```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);  // ← `amount`
``` [2](#0-1) 

`safeTransferFrom` only verifies the boolean return value; it does **not** verify that the minter's balance increased by exactly `amount`. For a deflationary token that deducts a fee during transfer, the call succeeds but the minter receives `amount − fee`.

**Step 2 — The IC minter mints exactly `event.value()` ckERC20 tokens.**

In `rs/ethereum/cketh/minter/src/deposit.rs`, the `mint()` function issues an ICRC-1 mint for `event.value()`, which is the `amount` field parsed directly from the Ethereum log:

```rust
let block_index = match client
    .transfer(TransferArg {
        ...
        amount: event.value(),   // ← taken verbatim from the log
    })
    .await
``` [3](#0-2) 

The `ReceivedErc20Event.value` field is populated from the `uint256 amount` argument of the Ethereum log, which the helper contract set to the caller-supplied value, not the balance delta. [4](#0-3) 

**Step 3 — Invariant break.**

The ckERC20 ledger total supply grows by `amount`, but the minter's Ethereum address only holds `amount − fee` of the underlying ERC-20. Every such deposit inflates the ckERC20 supply beyond the actual reserve. When the last depositors attempt to withdraw, the minter cannot satisfy all redemptions at 1:1.

---

### Impact Explanation

**Ledger conservation bug / chain-fusion mint over-issuance.** The ckERC20 total supply on the IC ledger exceeds the ERC-20 balance held by the minter's Ethereum address. Users who deposited at full face value cannot all redeem at 1:1; the shortfall is socialised across all ckERC20 holders. The severity scales with the token's fee rate and total deposit volume.

---

### Likelihood Explanation

The minter only processes events for NNS-governance-whitelisted ERC-20 tokens, so exploitation requires either:

1. A currently-supported token (e.g. USDC, which has an admin-controlled fee switch) enabling transfer fees, or
2. A future NNS proposal adding a fee-on-transfer token to the supported list.

Neither scenario requires an attacker to hold any privileged key; any ordinary depositor calling `depositErc20` with such a token triggers the over-mint automatically. The NNS governance gate lowers the probability but does not eliminate it, particularly for tokens whose fee mechanism is dormant at listing time.

---

### Recommendation

In both helper contracts, measure the **actual balance delta** and emit that value instead of the caller-supplied `amount`:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

Apply the same pattern to `ERC20DepositHelper.sol`'s `deposit` function. Additionally, the NNS governance process for adding new ckERC20 tokens should explicitly screen for fee-on-transfer behaviour.

---

### Proof of Concept

1. NNS governance adds a supported ckERC20 token whose ERC-20 contract charges a 1% transfer fee.
2. Alice calls `depositErc20(tokenAddr, 1_000_000, alicePrincipal, subaccount)` on `DepositHelperWithSubaccount`.
3. The ERC-20 contract transfers 990,000 tokens to the minter and burns/redirects 10,000 as a fee. `safeTransferFrom` returns `true`.
4. The contract emits `ReceivedEthOrErc20(tokenAddr, alice, 1_000_000, alicePrincipal, subaccount)`.
5. The IC minter scrapes the log, reads `value = 1_000_000`, and calls `icrc1_transfer` on the ckERC20 ledger to mint 1,000,000 ckERC20 to Alice.
6. The minter's Ethereum address holds only 990,000 ERC-20 tokens, but 1,000,000 ckERC20 are in circulation.
7. Repeated deposits compound the shortfall. When Alice and other holders attempt to withdraw, the last 1% of ckERC20 supply cannot be redeemed.
