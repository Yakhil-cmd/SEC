### Title
Sequential Integer Division Precision Loss in Per-Resource Canister Cycles Charging — (`rs/cycles_account_manager/src/cycles_account_manager.rs`)

---

### Summary

The IC's `CyclesAccountManager` applies **three sequential integer divisions** when computing the actual cycles charged to a canister for each resource type (memory, message memory, compute allocation). Because `charge_canister_for_resource_allocation_and_usage` calls `charge_canister_for_single_resource` independently for each resource, each resource incurs its own truncating division by `SECONDS_PER_DAY`. Combined with prior truncating divisions inside `memory_cost` (by `one_gib`) and `scale_cost` (by `reference_subnet_size`), the protocol systematically undercharges canisters for resource usage, constituting a cycles/resource accounting bug.

---

### Finding Description

The charge pipeline for canister storage fees proceeds through three sequential integer divisions:

**Division 1 — inside `memory_cost`** (line 654–657):
```rust
let cycles = Cycles::from(
    (bytes.get() as u128
        * self.config.gib_storage_per_second_fee.get()
        * duration.as_secs() as u128)   // duration = DAY = 86400s
        / one_gib,                       // ← first truncating division
);
``` [1](#0-0) 

**Division 2 — inside `scale_cost`** (line 116):
```rust
let real =
    (cycles * subnet_cycles_config.subnet_size) / self.config.reference_subnet_size.max(1);
//                                                ← second truncating division
``` [2](#0-1) 

**Division 3 — inside `charge_canister_for_single_resource`** (line 1157):
```rust
let cycles = rate * duration_since_last_charge.as_secs() / SECONDS_PER_DAY;
//                                                          ← third truncating division
``` [3](#0-2) 

This third division is applied **independently** to each of the three resources in `charge_canister_for_resource_allocation_and_usage`: [4](#0-3) 

The rates are computed per-resource in `idle_cycles_burned_rate_by_resource`: [5](#0-4) 

The mathematically correct formula for the actual charge would consolidate all multiplications first:
```
(bytes × fee_per_second × actual_seconds × subnet_size) / (one_gib × reference_subnet_size)
```

Instead, the code computes:
```
(((bytes × fee_per_second × 86400) / one_gib) × subnet_size / reference_subnet_size) × actual_seconds / 86400
```

Each `/` is an integer division that discards the remainder. The same sequential-division pattern also affects `freeze_threshold_cycles`, which divides the already-truncated `idle_cycles_burned_rate` by `SECONDS_PER_DAY` a second time: [6](#0-5) 

---

### Impact Explanation

**Vulnerability class: cycles/resource accounting bug.**

Each charge event (triggered every `duration_between_allocation_charges` = 10 seconds by default) can undercharge by up to 1 cycle per truncating division per resource. With three resources and three divisions each, the maximum undercharge per event is up to **9 cycles**. Across all canisters on all subnets over time, this constitutes a systematic, guaranteed undercharge — the protocol never overcharges, only undercharges. The cumulative effect is a slow but steady leak of cycles revenue from the IC's fee mechanism. The `freeze_threshold_cycles` underestimation is a secondary impact: a canister's freeze threshold is computed slightly lower than the correct value, meaning a canister may be permitted to execute messages when it should be frozen.

---

### Likelihood Explanation

**High.** This code path executes on every checkpoint round for every canister with non-zero memory or compute allocation on every application subnet. The undercharge is deterministic and reproducible. No special attacker action is required — any canister developer deploying a canister with memory usage passively benefits from the undercharging. The entry path is a standard ingress message or canister deployment by an unprivileged principal.

---

### Recommendation

Consolidate all multiplications before performing a single division. Instead of computing per-resource rates with `duration = DAY` and then re-scaling by `actual_seconds / SECONDS_PER_DAY`, pass `duration_since_last_charge` directly into `memory_cost` and `compute_allocation_cost`, eliminating the intermediate per-day rate and the final division in `charge_canister_for_single_resource`. This mirrors the recommendation in the external report: perform all multiplications first, then divide once.

---

### Proof of Concept

Consider a canister with `bytes = 1_048_577` (1 MiB + 1 byte), `gib_storage_per_second_fee = 317_500`, `subnet_size = reference_subnet_size = 13`, charged over `actual_seconds = 10`:

**Actual (three-division) path:**
1. `(1_048_577 × 317_500 × 86_400) / 1_073_741_824 = 28_755_790_632_000 / 1_073_741_824 = 26_786` (truncated from 26_786.16…)
2. `(26_786 × 13) / 13 = 26_786` (no truncation here)
3. `26_786 × 10 / 86_400 = 267_860 / 86_400 = 3` (truncated from 3.100…)

**Correct (single-division) path:**
`(1_048_577 × 317_500 × 10 × 13) / (1_073_741_824 × 13) = 3_328_003_175 / 1_073_741_824 = 3` (truncated from 3.100…)

In this case the results match, but the truncation at step 1 discards `0.16` cycles/day of rate. Over 8_640 charge events per day, the compounded loss from the step-1 truncation alone is `0.16 × 10 / 86_400 × 8_640 ≈ 0.16` cycles/day — guaranteed to be zero or negative, never positive. Across three resources and millions of canisters, this is a systematic, one-directional undercharge consistent with the reported vulnerability class.
