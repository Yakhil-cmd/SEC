Audit Report

## Title
Bitcoin Adapter Response Duplicate Delivery Validation Bypass — (`File: rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary
`BitcoinPayloadBuilder` computes past-delivered callback IDs during payload validation but discards them before any duplicate check, allowing a malicious block proposer to include already-delivered `BitcoinAdapterResponse` entries in a new payload. Because `validate_payload` and `validate_self_validating_payload` both skip the duplicate guard, every honest validator accepts the block, and the execution layer enqueues duplicate `ConsensusResponse` messages for Bitcoin-integrated canisters.

## Finding Description

**Root cause — three confirmed code paths:**

**Path 1 — `BatchPayloadBuilder::validate_payload` (lines 357–397):**
`delivered_ids` is computed from `past_payloads` at line 369 but is never forwarded to `validate_self_validating_payload_impl` at line 379. The variable is silently unused. [1](#0-0) 

**Path 2 — `SelfValidatingPayloadBuilder::validate_self_validating_payload` (lines 294–301):**
`_past_payloads` is explicitly discarded (underscore prefix); the impl call receives no past-callback information. [2](#0-1) 

**Path 3 — `validate_self_validating_payload_impl` (lines 234–251):**
Performs zero duplicate checking — it only short-circuits on an empty payload and returns the byte size. [3](#0-2) 

**Contrast with the build path**, which correctly passes `delivered_ids` into `get_self_validating_payload_impl` and filters at line 129: [4](#0-3) 

**Compiler warning suppression:** `#![allow(dead_code, unused_variables)]` at line 1 silences the warning that would otherwise flag `delivered_ids` as unused. [5](#0-4) 

**Execution-layer amplification — `push_response` in `rs/replicated_state/src/bitcoin.rs`:**

- `GetSuccessorsResponse` path uses `get_mut` (not `remove`) on `bitcoin_get_successors_contexts`. The context is **not removed** by `push_response`, so a second call with the same `callback_id` within the same batch succeeds and enqueues a second `ConsensusResponse`. [6](#0-5) 

- `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` paths perform **no context lookup at all** — they unconditionally push a `ConsensusResponse` for any `callback_id`, including replayed ones. [7](#0-6) 

**Error suppression in demux:** `push_response_bitcoin` errors are swallowed at `debug` level, so any downstream failure is invisible. [8](#0-7) 

## Impact Explanation

A malicious block proposer can craft a `SelfValidatingPayload` containing a `BitcoinAdapterResponse` whose `callback_id` was already present in an earlier payload within the same batch. Because validation skips the duplicate check, all honest validators accept the block. The execution layer then enqueues two `ConsensusResponse` messages for the same callback: for `SendTransactionResponse` and reject variants this is unconditional; for `GetSuccessorsResponse` the context is still present during `push_response` (it is only removed later by `retrieve_context` during execution). The ckBTC minter or any Bitcoin-integrated canister receives the same response twice, corrupting its internal state — e.g., double-counting a confirmed transaction or replaying block data. This is a **Significant Chain Fusion / ck-token security impact with concrete user and protocol harm**, matching the High ($2,000–$10,000) impact tier.

## Likelihood Explanation

Block proposership rotates among all subnet nodes. Any single compromised or malicious node can exploit this during its proposer turn without requiring a majority or any special cryptographic material. The attacker only needs to observe a `callback_id` from the current batch's past payloads (available in-protocol) and re-include it. The exploit is repeatable every time the attacker holds the proposer role, requires no victim interaction, and is undetectable by honest validators given the current validation logic.

## Recommendation

1. Extend `validate_self_validating_payload_impl` to accept a `past_callback_ids: &BTreeSet<u64>` parameter and return `InvalidArtifact` if any response's `callback_id` is already in the set.
2. In `BatchPayloadBuilder::validate_payload`, pass the already-computed `delivered_ids` to that function instead of discarding it.
3. In `SelfValidatingPayloadBuilder::validate_self_validating_payload`, derive past callback IDs from `past_payloads` (mirroring `get_self_validating_payload`) and pass them to the same function instead of ignoring `_past_payloads`.
4. Remove or tightly scope the `#![allow(dead_code, unused_variables)]` pragma so the compiler can flag future regressions.
5. Consider making `push_response` for `SendTransactionResponse` and reject variants also validate that the `callback_id` exists in the call context manager before enqueuing, as a defense-in-depth measure.

## Proof of Concept

1. On a Bitcoin-enabled subnet, observe a batch at height `H` containing a `SelfValidatingPayload` with a `BitcoinAdapterResponse` for `callback_id = 42`. Record the serialized payload bytes.
2. As block proposer at height `H+1` (same batch round), craft a new `SelfValidatingPayload` that includes a second `BitcoinAdapterResponse` with `callback_id = 42` alongside any other valid response.
3. Call `validate_payload` on the crafted payload with the height-`H` payload in `past_payloads`. Observe it returns `Ok(())` — `delivered_ids` is computed but unused, no duplicate rejection occurs.
4. Finalize the block. `DemuxImpl::process_payload` calls `push_response_bitcoin` twice for `callback_id = 42`. For `GetSuccessorsResponse`, both calls succeed because `get_mut` finds the context still present; for `SendTransactionResponse`/rejects, both calls push unconditionally.
5. The execution layer receives two `ConsensusResponse` messages for `callback_id = 42`, causing the Bitcoin-integrated canister to process the same response twice.

A deterministic integration test can reproduce this by constructing a `BitcoinPayloadBuilder` with a mock state manager, building a payload at height `H`, then calling `validate_payload` at height `H+1` with the height-`H` payload in `past_payloads` and a duplicate `callback_id` in the new payload, asserting `Ok(())` is returned.

        &self,
        payload: &SelfValidatingPayload,
        validation_context: &ValidationContext,
    ) -> Result<NumBytes, SelfValidatingPayloadValidationError> {
        let since = Instant::now();

        // An empty block is always valid.
        if *payload == SelfValidatingPayload::default() {
            return Ok(0.into());
        }

        self.metrics
            .observe_validate_duration(VALIDATION_STATUS_VALID, since);
        let size = NumBytes::new(payload.count_bytes() as u64);

        Ok(size)
    }
