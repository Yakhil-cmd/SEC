Audit Report

## Title
Global Neuron Creation Rate Limiter Exhaustion via Unrestricted `ClaimOrRefresh` with `MemoAndController` - (File: rs/nns/governance/src/governance.rs)

## Summary
The NNS Governance canister uses a single global rate limiter keyed on `"ADD_NEURON"` with a burst capacity of 300 slots. Any unprivileged ingress caller can exhaust this capacity by pre-funding 300 subaccounts with the minimum neuron stake and repeatedly calling `manage_neuron` with `ClaimOrRefresh { by: MemoAndController { controller: None, memo: i } }`, after which all users are throttled to at most 1 new neuron per 240 seconds until capacity replenishes.

## Finding Description
The rate limiter is configured with `add_capacity_amount: 1`, `add_capacity_interval: 240s`, and `max_capacity: 300` (`MAX_NEURON_CREATION_SPIKE = MAX_SUSTAINED_NEURONS_PER_HOUR * 20 = 300`). [1](#0-0) 

A single shared key `NEURON_RATE_LIMITER_KEY = "ADD_NEURON"` is used for all neuron-creation paths. [2](#0-1) 

The `claim_neuron` function calls `try_reserve` against this key before any caller-identity or ownership check. [3](#0-2) 

The reservation is only committed (capacity permanently consumed) after the ledger balance check passes. If the balance is insufficient, the neuron is removed and the reservation is dropped without consuming capacity. [4](#0-3) 

Therefore, the attacker **must** actually fund the subaccounts to exhaust the limiter. `claim_or_refresh_neuron_by_memo_and_controller` allows any caller to claim a neuron on behalf of any controller (defaulting to the caller when `controller: None`), with no ownership check before the rate-limiter path is reached. [5](#0-4) 

The same global key is also consumed by `split_neuron` and `create_neuron`, meaning the attacker's exhaustion affects all creation paths simultaneously. [6](#0-5) [7](#0-6) 

## Impact Explanation
After exhausting all 300 burst slots, every user — including legitimate ones — receives `GovernanceError { Unavailable, "Reached maximum number of neurons that can be created in this hour. Please wait and try again later." }` until capacity replenishes at 1 slot per 240 seconds (15/hour). This is a meaningful application-level DoS on NNS neuron creation, not a raw volumetric DDoS. It matches the allowed impact: **Medium ($200–$2,000)** — difficult attack with significant cost and constraints, but with concrete user and protocol harm (NNS governance participation throttled during high-demand periods). [8](#0-7) 

## Likelihood Explanation
The attacker must pre-fund 300 subaccounts with the NNS minimum neuron stake (~1 ICP each ≈ 300 ICP total). The ICP is not destroyed — it is locked in neurons and recoverable after the dissolve delay — but the opportunity cost is non-trivial. The attack is repeatable indefinitely by dissolving and re-staking. A well-funded adversary can execute this repeatedly to sustain the throttling effect.

## Recommendation
1. **Per-principal sub-limit**: Track neuron-creation counts per caller within the rate-limiter window and reject requests exceeding a per-principal quota before consuming global capacity.
2. **Restrict proxy claiming**: Require that `ClaimOrRefresh` with `MemoAndController` only succeeds when `controller == caller`, or impose a separate, lower per-caller burst limit for proxy claims.
3. **Increase minimum stake**: A higher minimum stake raises the capital cost of the attack proportionally.

## Proof of Concept
The existing test `test_rate_limiting_neuron_creation` in `rs/nns/governance/tests/governance.rs` already demonstrates the mechanism: it exhausts `MAX_NEURON_CREATION_SPIKE` slots via `split_neuron` and one `claim_neuron`, then confirms the next `claim_neuron` call returns exactly the throttle error. The attacker path replaces splits with funded `ClaimOrRefresh` calls from a single ingress principal: [9](#0-8) 

```
// Attacker (any ingress principal P) pre-funds 300 subaccounts:
for memo in 0..300:
    transfer(1_ICP, governance_subaccount(P, memo))

// Attacker exhausts the global rate limiter:
for memo in 0..300:
    manage_neuron(ClaimOrRefresh {
        by: MemoAndController { memo, controller: None }
    })
// → 300 reservations committed; global capacity = 0

// Any other user calling manage_neuron(ClaimOrRefresh{...}) now receives:
// GovernanceError { Unavailable, "Reached maximum number of neurons that can
//   be created in this hour. Please wait and try again later." }
// until capacity replenishes at 1 slot per 240 s.
```

    fn from(value: RateLimiterError) -> Self {

            }

                committed because rate limiter has no record of it."
                .to_string(),
        };

        GovernanceError::new_with_message(ErrorType::Unavailable, message)
    }

    async fn claim_or_refresh_neuron_by_memo_and_controller(
        &mut self,
        caller: &PrincipalId,
        memo_and_controller: MemoAndController,
        claim_or_refresh: &ClaimOrRefresh,
    ) -> Result<NeuronId, GovernanceError> {
        let controller = memo_and_controller.controller.unwrap_or(*caller);
        let memo = memo_and_controller.memo;
        let subaccount = ledger::compute_neuron_staking_subaccount(controller, memo);
        match self.neuron_store.get_neuron_id_for_subaccount(subaccount) {
            Some(neuron_id) => {
                self.refresh_neuron(neuron_id, subaccount, claim_or_refresh)
                    .await
            }
            None => {
                self.claim_neuron(subaccount, controller, claim_or_refresh)
                    .await
            }
        }

fn test_rate_limiting_neuron_creation() {
    let current_peak = MAX_NEURON_CREATION_SPIKE;
    let minimum_wait_for_capacity_increase = MINIMUM_SECONDS_BETWEEN_ALLOWANCE_INCREASE;

    // Some neurons with maturity and stake so we can spawn and split
    let staked_neurons = (1..=(current_peak - 1))
        .map(|i| {
            let controller = PrincipalId::new_user_test_id(i);
            let nonce = 1234_u64;
            api::Neuron {
                id: Some(NeuronId::from_u64(i)),
                account: ledger::compute_neuron_staking_subaccount(controller, nonce).into(),
                controller: Some(controller),
                cached_neuron_stake_e8s: 10 * E8,
                neuron_fees_e8s: 0,
                created_timestamp_seconds: 0,
                aging_since_timestamp_seconds: 0,
                maturity_e8s_equivalent: 10 * E8,
                staked_maturity_e8s_equivalent: Some(10 * E8),
                dissolve_state: Some(api::neuron::DissolveState::DissolveDelaySeconds(
                    VotingPowerEconomics::DEFAULT_NEURON_MINIMUM_DISSOLVE_DELAY_TO_VOTE_SECONDS,
                )),
                ..Default::default()
            }
        })
        .collect::<Vec<_>>();

    let (driver, mut gov) = governance_with_neurons(&staked_neurons);

    for i in 1..=(current_peak - 1) {
        let neuron_id = NeuronId::from_u64(i);
        // Split
        let controller = gov
            .neuron_store
            .with_neuron(&neuron_id, |neuron| neuron.controller())
            .unwrap();
        gov.split_neuron(
            &neuron_id,
            &controller,
            &Split {
                amount_e8s: 5 * E8,
                memo: None,
            },
        )
        .now_or_never()
        .unwrap()
        .unwrap();

        // spawn should not be rate limited
        let controller = gov
            .neuron_store
            .with_neuron(&neuron_id, |neuron| neuron.controller())
            .unwrap();

                new_controller: Some(PrincipalId::new_user_test_id(i * 2)),
                nonce: None,
                percentage_to_spawn: Some(40),

            "We unexpectedly hit the rate limit, which means spawn is not exempt from the limits. \
                This is an error.",
