Audit Report

## Title
Boundary Node DoS via CRLF in `method_name` Causing Panic in `postprocess_response` — (`rs/boundary_node/ic_boundary/src/http/middleware/process.rs`)

## Summary

An unprivileged attacker can send a CBOR-encoded IC API request with a `method_name` containing CRLF bytes. The only validation applied is a length check. In `postprocess_response`, the unsanitized `method_name` is passed to `HeaderValue::from_maybe_shared(...).unwrap()`, which panics when the `http` crate rejects the CRLF bytes, aborting the Tokio connection handler task. No panic-catching layer exists anywhere in the middleware stack.

## Finding Description

**Root cause — `check_method_name_length` validates length only:** [1](#0-0) 

The deserializer rejects names longer than 20,000 bytes but performs zero content sanitization. A `method_name` of `"AAAAAAAAAA\r\nX-Evil: injected"` (30 bytes) passes unconditionally and is stored verbatim in `RequestContext`.

**Unsanitized value inserted into response header with `.unwrap()`:** [2](#0-1) 

`Bytes::from(truncated.to_string())` produces a fresh heap-allocated `Bytes`. `HeaderValue::from_maybe_shared` validates every byte for non-static inputs; bytes `\r` (0x0D) and `\n` (0x0A) are explicitly invalid per the HTTP/1.1 header value grammar enforced by the `http` crate. The function returns `Err(InvalidHeaderValue)` and `.unwrap()` panics.

**Middleware ordering confirms the path is reachable:** [3](#0-2) 

In `ServiceBuilder`, layers are applied outermost-first. `postprocess_response` (line 1011) wraps `preprocess_request` (line 1012). The flow is: `postprocess_response` calls `next.run(request)` → `preprocess_request` parses the CBOR, stores the CRLF `method_name` in `RequestContext`, and injects it into the response extensions → `postprocess_response` reads the context and panics attempting to insert the header.

**No panic-catching layer exists:**

A grep for `CatchPanic` across the entire repository returns zero matches. The `common_layers` `ServiceBuilder` chain contains no `CatchPanicLayer` or equivalent. A panic in `postprocess_response` propagates to the Tokio connection handler task, aborting it and dropping the connection.

## Impact Explanation

Each crafted request causes a task-level panic and connection drop on the boundary node. An attacker can send a continuous stream of such requests — no authentication, no rate limiting on `method_name` content — to cause repeated panics across connection handler tasks, degrading or denying service to legitimate users. This matches the allowed High impact: **Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

Header injection is **not** achievable because `from_maybe_shared` rejects CRLF before any injection can occur; the actual impact is exclusively DoS via panic.

## Likelihood Explanation

- Requires no credentials, no privileged role, no key material.
- Entry point is any public IC API endpoint (`/api/v2/canister/.../call`, `/query`, `/read_state`).
- The CBOR body is trivially constructable; the CRLF must appear within the first `MAX_LOGGING_METHOD_NAME_LENGTH` bytes of `method_name` (the PoC uses a 30-byte name with CRLF at position 10, well within any reasonable truncation limit).
- No existing guard sanitizes or rejects non-printable bytes in `method_name` before header insertion.
- Reproducible with a single crafted request.

## Recommendation

Replace the bare `.unwrap()` at line 267 with explicit error handling. Either:

1. **Sanitize before insertion**: strip or percent-encode non-printable bytes from `method_name` before constructing the `HeaderValue`.
2. **Handle the error gracefully**: use `if let Ok(v) = HeaderValue::from_maybe_shared(...) { response.headers_mut().insert(..., v); }` to silently skip insertion on invalid values rather than panicking.

The same `.unwrap()` pattern appears for `X_IC_ERROR_CAUSE` (line 203), `X_IC_CACHE_STATUS` (line 211), and `X_IC_CACHE_BYPASS_REASON` (line 217); those values derive from internal `enum`-derived strings and are safe in practice, but the `method_name` case is attacker-controlled and must be treated differently.

## Proof of Concept

```python
import cbor2, requests

method_name = "AAAAAAAAAA\r\nX-Evil: injected"  # 30 bytes, CRLF at position 10

envelope = {
    "content": {
        "request_type": "call",
        "sender": bytes(1),
        "canister_id": bytes(b'\x00' * 10),
        "method_name": method_name,
        "arg": b'',
        "ingress_expiry": 9999999999999999999,
        "nonce": b'\x01',
    }
}

body = cbor2.dumps(envelope)
resp = requests.post(
    "https://<boundary-node>/api/v2/canister/aaaaa-aa/call",
    data=body,
    headers={"Content-Type": "application/cbor"},
)
# Expected: connection reset / 500 due to panic in postprocess_response
print(resp.status_code)
```

The `method_name` passes `check_method_name_length` (30 < 20,000), is stored in `RequestContext`, and when `postprocess_response` calls `HeaderValue::from_maybe_shared(Bytes::from("AAAAAAAAAA\r\nX-Evil: injected")).unwrap()`, the `http` crate rejects the CRLF and the `.unwrap()` panics. A local integration test using `axum::test` or a PocketIC harness can reproduce this deterministically without touching mainnet.
