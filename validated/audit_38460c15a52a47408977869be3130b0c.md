Audit Report

## Title
Overly Broad Self-Approval Guard Blocks Legitimate Cross-Subaccount `icrc2_approve` Operations - (File: `rs/ledger_suite/icrc1/ledger/src/main.rs` and `rs/ledger_suite/icp/ledger/src/main.rs`)

## Summary

Both the ICRC-1 ledger and the ICP ledger compare only the `owner` Principal field — not the full ICRC-1 account identity (owner + subaccount) — when guarding against self-approval in `icrc2_approve`. This causes an unconditional, unrecoverable `trap` for any call where the caller's principal equals the spender's principal, even when the two accounts are distinct by subaccount. The lower-level `approvals.approve()` function correctly uses full-account equality, making the upper-level guard strictly more restrictive than the standard requires and permanently blocking a class of valid ICRC-2 operations.

## Finding Description

In `rs/ledger_suite/icrc1/ledger/src/main.rs`, `icrc2_approve_not_async` constructs `from_account` from the caller and `arg.from_subaccount`, then checks:

```rust
if from_account.owner == arg.spender.owner {
    ic_cdk::trap("self approval is not allowed")
}
``` [1](#0-0) 

The identical pattern appears in `rs/ledger_suite/icp/ledger/src/main.rs`:

```rust
if from_account.owner == arg.spender.owner {
    trap("self approval is not allowed");
}
``` [2](#0-1) 

Both checks compare only the `Principal` (the `owner` field), not the full `Account { owner, subaccount }`. Under ICRC-1, two accounts sharing the same `owner` but differing in `subaccount` are semantically distinct accounts. The guard therefore traps any call where `caller == spender.owner`, regardless of whether `from_subaccount != spender.subaccount`.

The lower-level `approvals.approve()` in `rs/ledger_suite/common/ledger_core/src/approvals.rs` correctly uses full-account equality:

```rust
if account == spender {
    return Err(ApproveError::SelfApproval);
}
``` [3](#0-2) 

The upper-level guard is more restrictive than the lower-level guard and prevents legitimate cross-subaccount approvals from ever reaching the correct check. Because `ic_cdk::trap` aborts the call entirely rather than returning a structured `ApproveError`, the rejection is unrecoverable and indistinguishable from a protocol error.

## Impact Explanation

This constitutes a significant ledger-level functional DoS: a well-defined, standard-compliant class of ICRC-2 operations — cross-subaccount approvals between two distinct accounts sharing the same principal — is permanently and unconditionally blocked on both the ICRC-1 ledger and the ICP ledger. Downstream `icrc2_transfer_from` flows that depend on such allowances are also blocked. Canisters managing per-user or per-pool subaccounts under a single principal cannot use the ICRC-2 delegation mechanism at all. This matches the High impact category: "Significant... ledger... security impact with concrete user or protocol harm."

## Likelihood Explanation

The exploit path requires no special privileges. Any unprivileged user or canister that controls a principal with tokens in two different subaccounts and calls `icrc2_approve` with `from_subaccount = Some(A)` and `spender = { owner: same_principal, subaccount: Some(B) }` will trigger the trap. The precondition (same principal, different subaccounts) is a standard and realistic pattern for canister-managed treasury or per-user accounts. The trigger is a direct ingress call with no governance action, key material, or social engineering required. The bug is deterministic and 100% reproducible.

## Recommendation

Replace the owner-only comparison with a full-account comparison in both ledgers.

**ICRC-1 ledger** (`rs/ledger_suite/icrc1/ledger/src/main.rs`, line 828):
```rust
// Before:
if from_account.owner == arg.spender.owner {
// After:
if from_account == arg.spender {
```

**ICP ledger** (`rs/ledger_suite/icp/ledger/src/main.rs`, line 1328):
```rust
// Before:
if from_account.owner == arg.spender.owner {
// After:
if from_account == arg.spender {
```

This aligns the upper-level guard with the lower-level `approvals.approve()` check and with the ICRC-2 standard's intent that "self" means the same full account, not the same principal.

## Proof of Concept

Deploy either ledger locally (PocketIC or local replica). Create a principal `alice`. Mint tokens into `{ owner: alice, subaccount: Some([0x01; 32]) }`. Call `icrc2_approve` as `alice` with:

```
from_subaccount: Some([0x01; 32])
spender: { owner: alice, subaccount: Some([0x02; 32]) }
amount: 1_000_000
```

Expected (per ICRC-2 standard): approval succeeds, since `from_account != spender`.
Actual: the ledger evaluates `alice == alice → true` and calls `ic_cdk::trap("self approval is not allowed")`, aborting the call. The allowance is never recorded. Any subsequent `icrc2_transfer_from` using this allowance also fails. The test in `rs/ledger_suite/tests/sm-tests/src/lib.rs` covers only same-subaccount self-approval and does not exercise the cross-subaccount case, confirming the gap. [4](#0-3) [5](#0-4)
