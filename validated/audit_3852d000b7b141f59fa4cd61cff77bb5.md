### Title
Permanent Fund Freeze in `ExitToEthereum` Precompile — No Failure Callback for Cross-Chain Withdrawal (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns ERC-20 tokens (or deducts ETH) from the EVM state and then creates a NEAR promise to call `withdraw` on the ETH connector contract. Unlike the `ExitToNear` precompile (which has a conditional `error_refund` callback path), `ExitToEthereum` **never attaches a failure callback**. If the NEAR-side `withdraw` call fails for any reason, the already-burned EVM tokens are permanently unrecoverable.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToEthereum` precompile's `run` method constructs a NEAR promise to call `withdraw` on the ETH connector: [1](#0-0) 

The promise is always wrapped as `PromiseArgs::Create(withdraw_promise)` — a fire-and-forget call with no callback: [2](#0-1) 

This is structurally different from `ExitToNear`, which conditionally attaches a `exit_to_near_precompile_callback` to handle failure and remint tokens: [3](#0-2) 

The callback handler in `connector.rs` that performs the refund on failure only exists for the `ExitToNear` path: [4](#0-3) 

No equivalent callback or recovery path exists for `ExitToEthereum`. The EVM-side burn is committed atomically within the EVM execution, but the NEAR-side `withdraw` is asynchronous and can fail independently.

---

### Impact Explanation

When the NEAR-side `withdraw` call fails:

- **ERC-20 path (flag `0x1`)**: The ERC-20 contract has already burned the user's tokens before calling the precompile. The NEP-141 tokens held by the Aurora engine contract are never released. The user's ERC-20 tokens are permanently destroyed with no corresponding NEP-141 credit.
- **ETH path (flag `0x0`)**: The ETH was already deducted from the user's EVM balance via `context.apparent_value`. The ETH connector never receives the withdrawal. The ETH is permanently lost.

In both cases, there is no admin recovery function, no re-entry point, and no on-chain mechanism to remint or refund the burned assets.

**Impact**: Permanent freezing/destruction of user funds.

---

### Likelihood Explanation

The NEAR-side `withdraw` can fail under realistic conditions:

1. **Connector account replacement**: `set_eth_connector_contract_account` can be called by the owner or via a private call. If the connector account is changed while a withdrawal is in flight, the `withdraw` call targets the old account which may reject it.
2. **Connector contract paused or broken**: If the ETH connector contract is paused, has a storage issue, or is misconfigured, the `withdraw` call fails.
3. **Insufficient attached gas**: `WITHDRAWAL_GAS` is set to 100 TGas. If the connector's `withdraw` requires more gas in a future upgrade, the call fails silently. [5](#0-4) 

Likelihood is **low** (matching the original report's assessment), as it requires the connector to be broken or reconfigured during an in-flight withdrawal. However, the window is non-zero and the impact is irreversible.

---

### Recommendation

Attach a failure callback to the `ExitToEthereum` promise, analogous to the `exit_to_near_precompile_callback` pattern. On failure, the callback should remint the burned ERC-20 tokens or restore the ETH balance to the original sender. The `RefundCallArgs` type already exists and is used by the `ExitToNear` path: [6](#0-5) 

A new `exit_to_ethereum_precompile_callback` entrypoint should be added to `engine/src/contract_methods/connector.rs` that calls `engine::refund_on_error` on failure, mirroring the existing `exit_to_near_precompile_callback`.

---

### Proof of Concept

**Scenario: ERC-20 exit with connector failure**

1. User holds ERC-20 tokens on Aurora representing a bridged NEP-141 token (e.g., `usdc.near`).
2. User calls the ERC-20 contract's `withdraw(amount, eth_recipient)` function.
3. The ERC-20 contract burns `amount` tokens and calls the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with flag `0x1`.
4. The precompile creates a NEAR promise: `usdc.near.withdraw({"amount": X, "recipient": "0x..."})`.
5. Between step 3 and step 4 executing on NEAR, the owner calls `set_eth_connector_contract_account` to point to a new connector, or the `usdc.near` connector is paused.
6. The `withdraw` promise fails on NEAR.
7. No callback fires. The ERC-20 tokens are permanently burned. The user receives nothing. [7](#0-6) 

The test suite for `ExitToNear` explicitly acknowledges the analogous problem exists when `error_refund` is disabled, confirming the team is aware of this class of issue for `ExitToNear` but has not addressed it for `ExitToEthereum`: [8](#0-7)

impl<I: IO> Precompile for ExitToEthereum<I> {
    fn required_gas(_input: &[u8]) -> Result<EthGas, ExitError> {
        Ok(costs::EXIT_TO_ETHEREUM_GAS)
    }

    #[allow(clippy::too_many_lines)]
    fn run(
        &self,
        input: &[u8],
        target_gas: Option<EthGas>,
        context: &Context,
        is_static: bool,
    ) -> EvmPrecompileResult {
        // ETH (Base token) transfer input format (min size 21 bytes)
        //  - flag (1 byte)
        //  - eth_recipient (20 bytes)
        // ERC-20 transfer input format: max 53 bytes
        //  - flag (1 byte)
        //  - amount (32 bytes)
        //  - eth_recipient (20 bytes)
        validate_input_size(input, 21, 53)?;

        let required_gas = Self::required_gas(input)?;

        if let Some(target_gas) = target_gas
            && required_gas > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        // It's not allowed to call exit precompiles in static mode
        if is_static {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_STATIC")));
        } else if context.address != exit_to_ethereum::ADDRESS.raw() {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_DELEGATE")));
        }

        // The first byte of the input is a flag, selecting the behavior to be triggered:
        //  0x00 -> ETH (Base token) token transfer
        //  0x01 -> ERC-20 transfer
        let mut input = input;
        let flag = input[0];
        input = &input[1..];

        let (nep141_address, serialized_args, exit_event) = match flag {
            0x0 => {
                // ETH (base) transfer
                //
                // Input slice format:
                //  eth_recipient (20 bytes) - the address of recipient which will receive ETH on Ethereum
                let recipient_address: Address = input
                    .try_into()
                    .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECIPIENT_ADDRESS")))?;
                let serialize_fn = match get_withdraw_serialize_type(&self.io)? {
                    WithdrawSerializeType::Json => json_args,
                    WithdrawSerializeType::Borsh => borsh_args,
                };
                let eth_connector_account_id = self.get_eth_connector_contract_account()?;

                (
                    eth_connector_account_id,
                    // There is no way to inject json, given the encoding of both arguments
                    // as decimal and hexadecimal respectively.
                    serialize_fn(recipient_address, context.apparent_value)?,
                    events::ExitToEth {
                        sender: Address::new(context.caller),
                        erc20_address: events::ETH_ADDRESS,
                        dest: recipient_address,
                        amount: context.apparent_value,
                    },
                )
            }
            0x1 => {
                // ERC-20 transfer
                //
                // This precompile branch is expected to be called from the ERC20 withdraw function
                // (or burn function with some flag provided that this is expected to be withdrawn)
                //
                // Input slice format:
                //  amount (U256 big-endian bytes) - the amount that was burned
                //  eth_recipient (20 bytes) - the address of recipient which will receive ETH on Ethereum

                if context.apparent_value != U256::from(0) {
                    return Err(ExitError::Other(Cow::from(
                        "ERR_ETH_ATTACHED_FOR_ERC20_EXIT",
                    )));
                }

                let erc20_address = context.caller;
                let nep141_address = get_nep141_from_erc20(erc20_address.as_bytes(), &self.io)?;
                let amount = parse_amount(&input[..32])?;

                input = &input[32..];

                if input.len() == 20 {
                    // Parse ethereum address in hex
                    let mut buffer = [0; 40];
                    hex::encode_to_slice(input, &mut buffer).unwrap();
                    let recipient_in_hex = str::from_utf8(&buffer).map_err(|_| {
                        ExitError::Other(Cow::from("ERR_INVALID_RECIPIENT_ADDRESS"))
                    })?;
                    // unwrap cannot fail since we checked the length already
                    let recipient_address = Address::try_from_slice(input)
                        .map_err(|_| ExitError::Other(Cow::from("ERR_WRONG_ADDRESS")))?;

                    (
                        nep141_address,
                        // There is no way to inject json, given the encoding of both arguments
                        // as decimal and hexadecimal respectively.
                        format!(
                            r#"{{"amount": "{}", "recipient": "{}"}}"#,
                            amount.as_u128(),
                            recipient_in_hex
                        )
                        .into_bytes(),
                        events::ExitToEth {
                            sender: Address::new(erc20_address),
                            erc20_address: Address::new(erc20_address),
                            dest: recipient_address,
                            amount,
                        },
                    )
                } else {
                    return Err(ExitError::Other(Cow::from("ERR_INVALID_RECIPIENT_ADDRESS")));
                }
            }
            _ => {
                return Err(ExitError::Other(Cow::from(
                    "ERR_INVALID_RECEIVER_ACCOUNT_ID",
                )));
            }
        };

        let withdraw_promise = PromiseCreateArgs {
            target_account_id: nep141_address,
            method: "withdraw".to_string(),
            args: serialized_args,
            attached_balance: Yocto::new(1),
            attached_gas: costs::WITHDRAWAL_GAS,
        };

        let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
        let promise_log = Log {
            address: exit_to_ethereum::ADDRESS.raw(),
            topics: Vec::new(),
            data: promise,
        };
        let ethabi::RawLog { topics, data } = exit_event.encode();
        let exit_event_log = Log {
            address: exit_to_ethereum::ADDRESS.raw(),
            topics: topics.into_iter().map(|h| H256::from(h.0)).collect(),
            data,
        };

        Ok(PrecompileOutput {
            logs: vec![promise_log, exit_event_log],
            cost: required_gas,
            output: Vec::new(),
        })
    }
