### Title
ERC-20 Tokens and ETH Permanently Lost on Failed `exit_to_near` When `error_refund` Feature Is Disabled - (File: `engine-precompiles/src/native.rs`)

### Summary

When the Aurora Engine production binary is compiled without the `error_refund` feature flag, the `ExitToNear` precompile never schedules a refund callback after a failed `ft_transfer`. ERC-20 tokens are burned from the EVM side before the NEAR-side transfer is attempted; if that transfer fails (e.g., unregistered recipient account), the tokens are permanently destroyed with no recovery path. This is the direct analog of H-5: value is irrevocably consumed on one side of the bridge while the corresponding credit on the other side never materialises.

### Finding Description

**Root cause — compile-time erasure of the refund path.**

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` method builds the callback argument struct:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

The promise is then chosen based on whether `callback_args` equals its default (both fields `None`):

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback at all
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { base: transfer_promise, callback: … })
};
``` [2](#0-1) 

For every non-wNEAR exit path (regular ERC-20 exit, ETH/base-token exit) when `error_refund` is absent, `transfer_near` is also `None`, so `callback_args == default()` and **no callback is ever scheduled**. The `ft_transfer` promise fires and, if it fails, the engine never learns about it.

**The refund handler that would fix this:**

`exit_to_near_precompile_callback` contains the recovery logic, but it is only reachable when a callback was scheduled:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    …
} else {
    None   // ← reached when refund is None; nothing happens
};
``` [3](#0-2) 

**Attacker-controlled entry path (no privilege required):**

1. Any EVM user calls an ERC-20's `withdrawToNear` (or sends ETH to the exit precompile address).
2. The ERC-20 contract burns the tokens from the caller's EVM balance and calls the `ExitToNear` precompile.
3. The precompile schedules a bare `ft_transfer` promise to the NEP-141 contract targeting the supplied NEAR account ID.
4. If that account is not registered with the NEP-141 token (e.g., a typo, a non-existent account, or a deliberate griefing target), `ft_transfer` fails.
5. Because no callback was scheduled, the engine never executes `refund_on_error`; the burned ERC-20 tokens (or deducted ETH) are gone forever.

The test suite explicitly documents this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The same acknowledgement appears for the ETH exit path:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any EVM user who calls `exit_to_near` with a NEAR account that is not registered with the target NEP-141 token permanently loses the bridged assets. The EVM-side burn is irreversible; the NEAR-side credit never arrives; and no recovery function exists in the contract. This is structurally identical to H-5: value enters the system (is burned/deducted) but can never be retrieved, causing a growing discrepancy between the EVM total supply and the actual NEP-141 backing.

### Likelihood Explanation

**High.** The trigger condition — supplying an unregistered NEAR account ID — requires no privilege and is trivially reachable by any user making a simple mistake (typo, wrong account format, account that has not called `storage_deposit` on the NEP-141). The vulnerability is active in every build where `error_refund` is not compiled in, which the feature-flag structure and the dual `#[cfg]` branches in the test suite indicate is a real deployment scenario.

### Recommendation

1. Enable the `error_refund` feature in all production builds, or make it a default Cargo feature so it cannot be accidentally omitted.
2. Alternatively, unconditionally schedule the `exit_to_near_precompile_callback` for every exit path and always populate `refund` with the caller's address, removing the compile-time branch entirely.
3. Add a pre-flight check in the precompile that verifies the recipient NEAR account is registered with the NEP-141 before burning any EVM-side tokens.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora (creating an ERC-20 mirror).
3. From an EVM account holding ERC-20 tokens, call `withdrawToNear("unregistered.near", amount)`.
4. Observe: ERC-20 balance decreases by `amount`; NEP-141 balance of `"unregistered.near"` remains 0; NEP-141 balance of the Aurora contract is unchanged (the `ft_transfer` was rejected); no refund occurs in the EVM.
5. The `amount` of tokens is permanently destroyed — confirmed by the test `test_exit_to_near_refund` which asserts `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into()` under `#[cfg(not(feature = "error_refund"))]`. [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8)

```rust
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
