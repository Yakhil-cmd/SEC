### Title
Pending Voting Rewards Permanently Lost When Neuron Disappears Between Reward Calculation and Asynchronous Distribution - (File: rs/nns/governance/src/reward/distribution.rs)

### Summary
The NNS governance canister uses a two-phase, asynchronous reward distribution mechanism. Voting rewards are first calculated and recorded in a `RewardsDistribution` state machine (persisted in stable memory), then distributed to neurons in subsequent timer-driven tasks. If a neuron ceases to exist between the calculation phase and the distribution phase, its allocated reward is silently dropped with only a log message. The reward tokens are neither credited to the neuron nor rolled over into the next reward period — they are permanently lost from the system.

### Finding Description

The NNS governance reward pipeline works as follows:

1. `calculate_voting_rewards` in `rs/nns/governance/src/governance.rs` iterates over all voters in settled proposals, checks `self.neuron_store.contains(neuron_id)`, and builds a `RewardsDistribution` map of `NeuronId → reward_e8s`. The reward event records `actually_distributed_e8s_equivalent` as the sum of all rewards added to this map.

2. `distribute_voting_rewards_to_neurons` calls `schedule_pending_rewards_distribution`, which persists the `RewardsDistribution` into a `StableBTreeMap` keyed by `day_after_genesis`.

3. A periodic timer task (`DistributeRewardsTask`, interval 2 seconds) calls `distribute_pending_rewards`, which calls `continue_processing` on the next pending distribution.

4. Inside `continue_processing`, for each `(neuron_id, reward_e8s)` entry, `neuron_store.with_neuron_mut` is called. If the neuron is not found, the error branch simply logs a message and **the reward entry is consumed from the map without being credited anywhere**:


The reward is popped from `self.rewards` via `pop_first()` at the top of the loop, so even on error the entry is removed and never re-queued.

The analogous pattern exists in SNS governance's `distribute_rewards` in `rs/sns/governance/src/governance.rs`, where a missing neuron causes its reward to be silently skipped with `continue`.

The root cause is a TOCTOU (time-of-check/time-of-use) gap: neuron existence is verified at calculation time, but the actual credit happens in a later asynchronous execution. Between these two points, a neuron can be removed (e.g., via a governance upgrade that cleans up neurons, or any other path that removes a neuron from the store after the reward event is committed but before the timer fires).

### Impact Explanation

Any neuron that is present at reward calculation time but absent at distribution time loses its allocated voting reward permanently. The `RewardEvent.distributed_e8s_equivalent` field already counts the reward as distributed (it was added to `actually_distributed_e8s_equivalent` during calculation), so the reward is neither re-distributed nor rolled over. The ICP equivalent of the lost reward is simply never minted. This is a **ledger conservation bug**: the governance canister accounts for distributing tokens that are never actually credited to any account. The magnitude scales with the neuron's voting power and the size of the reward pool.

### Likelihood Explanation

The window between reward calculation and distribution is small (the timer fires every 2 seconds), but it is non-zero and persisted across canister upgrades. A canister upgrade that occurs between the two phases could, in principle, remove neurons. More concretely, the comment in the code itself says "This should not be possible as neuron existence is checked when rewards are calculated" — acknowledging the assumption but not enforcing it. Any future code path that removes neurons (e.g., neuron cleanup, governance migration bugs) during the distribution window would silently cause reward loss. The likelihood is low under normal operation but non-negligible across the lifetime of the system.

### Recommendation

When `neuron_store.with_neuron_mut` fails during `continue_processing`, the reward should not be silently dropped. Instead, one of the following mitigations should be applied:

1. **Re-queue the reward**: Re-insert the `(neuron_id, reward_e8s)` entry back into the distribution map so it can be retried, rather than consuming it on error.
2. **Roll over to next reward event**: Add the undeliverable reward amount to the `e8s_equivalent_to_be_rolled_over` of the next `RewardEvent`, so the tokens are not lost from the system.
3. **Enforce the invariant**: Prevent neuron deletion while a pending reward distribution exists for that neuron (e.g., by checking the `RewardsDistributionStateMachine` before allowing neuron removal).

### Proof of Concept

**Step 2**: `distribute_voting_rewards_to_neurons` is called. `calculate_voting_rewards` finds the neuron in `neuron_store`, adds its reward to `RewardsDistribution`, and increments `actually_distributed_e8s_equivalent`. The `RewardEvent` is committed with this amount. `schedule_pending_rewards_distribution` persists the distribution to stable memory. [1](#0-0) 

**Step 3**: Before the 2-second timer fires, the neuron is removed from `neuron_store` (e.g., via a canister upgrade or a future neuron-cleanup path).
