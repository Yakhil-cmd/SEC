### Title
Off-by-One in EVM Call Depth Check Allows Depth 1025 Instead of 1024 - (File: `evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

The EVM call depth guard in `check_depth_and_balance` uses `callstack_depth > 1024` instead of `callstack_depth >= 1024`. This allows a call frame at depth 1024 to proceed, creating a 1025-frame call stack, which violates the EVM specification's hard limit of 1024.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the function `check_depth_and_balance` is responsible for enforcing the EVM call depth limit before any new call frame (CALL, CALLCODE, DELEGATECALL, STATICCALL, CREATE, CREATE2) is opened: [1](#0-0) 

```rust
fn check_depth_and_balance<S: EthereumLikeTypes>(
    system: &mut System<S>,
    call_request: &mut ExternalCallRequest<S>,
    callstack_depth: usize,
) -> Result<Option<EvmError>, SubsystemError<EvmErrors>>
where
    S::IO: IOSubsystemExt,
{
    if callstack_depth > 1024 {          // <-- BUG: should be >= 1024
        system_log!(system, "Callstack is too deep\n",);
        return Ok(Some(EvmError::CallTooDeep));
    }
```

The `callstack_depth` parameter represents the depth of the **new frame being created** (the callee's depth), as defined in `EnvironmentParameters`: [2](#0-1) 

The EVM Yellow Paper specifies the call depth limit as 1024, meaning valid frame depths are 0–1023 (1024 total frames). A call that would create a frame at depth 1024 must be rejected. With `> 1024`, the condition is false when `callstack_depth == 1024`, so that frame is allowed to proceed — producing a 1025-frame call stack.

The same guard is reused in `constructor_pre_checks` for CREATE/CREATE2: [3](#0-2) 

Both call paths are affected.

---

### Impact Explanation

This is an **EVM semantic mismatch**. Any contract that relies on the EVM-specified behavior that a call at depth 1024 returns `0` (failure) will behave differently on ZKsync OS: the call succeeds instead of failing. This can cause:

- **Divergence between ZKsync OS forward execution and standard EVM** — the prover/verifier may accept a state transition that a correct EVM would reject, or vice versa.
- **Reentrancy-guard bypass via depth exhaustion** — contracts that use call-depth as an implicit reentrancy guard (a known pattern) may be bypassed by one extra level.
- **Incorrect state changes** — a call at depth 1024 that should have been a no-op (returning 0) instead executes and mutates state.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a transaction that recurses to depth 1024 and then makes one more call. On standard EVM this final call returns 0 (failure); on ZKsync OS it executes. The attacker only needs to deploy a contract that recurses to the boundary depth, which is straightforward and requires no privileged access.

---

### Recommendation

Change the comparison from `>` to `>=`:

```rust
// Before (incorrect):
if callstack_depth > 1024 {

// After (correct):
if callstack_depth >= 1024 {
    system_log!(system, "Callstack is too deep\n",);
    return Ok(Some(EvmError::CallTooDeep));
}
```

This matches the EVM specification: valid depths are 0–1023; depth 1024 must be rejected.

---

### Proof of Concept

1. Deploy a contract `Recurse` that, when called with depth `n`, calls itself with depth `n-1`, and at depth 0 calls a target contract `T`.
2. Call `Recurse` with `n = 1024` from an EOA (depth 0 → frame at depth 1 → … → frame at depth 1024 → call to `T`).
3. **Expected (EVM spec):** The call to `T` at depth 1024 returns 0 (failure); `T` is not executed.
4. **Actual (ZKsync OS):** The call to `T` at depth 1024 succeeds because `1024 > 1024` is `false`, so the guard does not trigger. `T` executes and mutates state.

The root cause is exclusively in `check_depth_and_balance` at: [4](#0-3)
