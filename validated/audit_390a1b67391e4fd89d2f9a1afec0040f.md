Audit Report

## Title
`get_allowances` Loop Guard Checks Only `owner`, Leaking Allowances Across Subaccounts — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary

The `get_allowances` function performs a `StableBTreeMap` range scan starting at the requested `from` account and terminates only when `account_spender.account.owner != from.owner`. Because `AccountSpender` keys are ordered by `(account.owner, account.effective_subaccount(), spender.owner, spender.effective_subaccount())`, after exhausting all spenders for `{A, subaccount_1}` the iterator advances to `{A, subaccount_2}`. The owner is still `A`, so the guard never fires and allowances belonging to a different subaccount are appended to the result. Any unprivileged caller can exploit this via the public `icrc103_get_allowances` query to enumerate allowances across all subaccounts of a target principal in a single call.

## Finding Description

`AccountSpender` ordering is defined at `rs/ledger_suite/icrc1/ledger/src/lib.rs:372-378`:

```rust
impl std::cmp::Ord for AccountSpender {
    fn cmp(&self, other: &Self) -> std::cmp::Ordering {
        self.account
            .cmp(&other.account)
            .then_with(|| self.spender.cmp(&other.spender))
    }
}
```

`Account` ordering is defined at `packages/icrc-ledger-types/src/icrc1/account.rs:54-60`:

```rust
impl std::cmp::Ord for Account {
    fn cmp(&self, other: &Self) -> std::cmp::Ordering {
        self.owner.cmp(&other.owner).then_with(|| {
            self.effective_subaccount().cmp(other.effective_subaccount())
        })
    }
}
```

So keys sort as `(owner_bytes, subaccount_bytes, spender_owner_bytes, spender_subaccount_bytes)`. After all spenders for `{A, subaccount_1}` are yielded, the next key is `{A, subaccount_2, ...}`. The termination guard at `lib.rs:1204` is:

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

This only compares the `owner` field. The subaccount field is never checked, so the loop continues into entries for `{A, subaccount_2}`, `{A, subaccount_3}`, etc., appending them to `result`. The public entrypoint at `rs/ledger_suite/icrc1/ledger/src/main.rs:1214-1232` is a `#[query]` with no caller authentication — `from_account` is caller-supplied and can be any arbitrary account.

## Impact Explanation

An unprivileged caller can enumerate spender identities, allowance amounts, and expiration timestamps for every subaccount of a target principal `A` by issuing a single `icrc103_get_allowances` call with `from_account = {A, Some([0u8;32])}`. This constitutes unauthorized disclosure of financial data (allowance relationships) across account boundaries on the ICRC-1 ledger, a production in-scope financial integration. This matches: **High — Significant ledger security impact with concrete user or protocol harm**, as it violates the account isolation guarantee of the ICRC-103 standard and exposes sensitive financial metadata (who approved whom, for how much, and until when) for accounts the caller did not request.

## Likelihood Explanation

The exploit requires no privileges, no cycles beyond a normal query call, no key material, and no consensus-level interaction. It is reproducible on any local replica or state-machine test with two `icrc2_approve` calls from different subaccounts of the same principal. The endpoint is callable by any IC principal.

## Recommendation

Replace the owner-only comparison with a full `Account` equality check:

```rust
// Before (buggy):
if account_spender.account.owner != from.owner {
    break;
}

// After (correct):
if account_spender.account != from {
    break;
}
```

`Account::eq` already compares both `owner` and `effective_subaccount()` (see `packages/icrc-ledger-types/src/icrc1/account.rs:40-44`), so this single-line change correctly terminates the scan when the iterated key's account diverges from the exact requested `from` account.

## Proof of Concept

```
Setup (state-machine test):
  principal A = test_principal()
  icrc2_approve(from={A, Some([1u8;32])}, spender=spender_X, amount=100)
  icrc2_approve(from={A, Some([2u8;32])}, spender=spender_Y, amount=200)

Call:
  icrc103_get_allowances({
    from_account: Some({owner: A, subaccount: Some([1u8;32])}),
    prev_spender: None,
    take: None,
  })

Expected result (correct behavior):
  [{ from_account: {A, Some([1u8;32])}, to_spender: spender_X, allowance: 100 }]

Actual result (buggy behavior):
  [{ from_account: {A, Some([1u8;32])}, to_spender: spender_X, allowance: 100 },
   { from_account: {A, Some([2u8;32])}, to_spender: spender_Y, allowance: 200 }]
```

The second entry is returned because the break guard at line 1204 only checks `owner`, not the full `Account` including subaccount.
