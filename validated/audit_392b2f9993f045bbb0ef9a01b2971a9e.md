### Title
Negative `delta_gas` Silently Discarded Causes Operator Underpayment and Token Conservation Violation - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, when native resource consumption is *less* than what EVM gas consumption implies (`delta_gas < 0`), the negative delta is silently discarded via a `// TODO: return delta_gas to gas_used?` comment. The sender is precharged `gas_price * gas_limit` tokens, but the sum of `token_to_refund + token_to_pay_operator` equals only `gas_price * gas_used` (where `gas_used` is the ergs-derived value, not the native-adjusted one). The difference — `gas_price * (-delta_gas)` tokens — is neither returned to the sender nor paid to the operator. Those tokens are permanently destroyed from the sender's balance, breaking token conservation and underpaying the operator relative to the actual work performed.

### Finding Description

The double-resource accounting model in ZKsync OS tracks both EVM gas (ergs) and native (proving) resources. After execution, `compute_gas_refund` computes `delta_gas`:

```
delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64)
```

The documented intent (see `docs/double_resource_accounting.md`) is:
- If `delta_gas > 0`: add it to `gas_used` so the sender pays more (native cost exceeded EVM cost).
- If `delta_gas < 0`: the comment says `// TODO: return delta_gas to gas_used?` — **nothing is done**.

When `delta_gas < 0`, the EVM gas consumed exceeds what native resource consumption implies. The sender was precharged `gas_price * gas_limit`. After execution:

- `token_to_refund = gas_price * (gas_limit - gas_used)` — refunds unused EVM gas
- `token_to_pay_operator = gas_price * gas_used` — pays operator for EVM gas used

But `gas_used` here is the ergs-derived value, not reduced by `|delta_gas|`. The total disbursed is `gas_price * gas_limit`, which equals the precharge — so at first glance it balances. However, the `burn_base_fee` feature path changes this:

When `burn_base_fee` is enabled (used in `eth_runner` and `evm_tester` feature sets), the operator receives only `priority_fee_per_gas * gas_used`, not `gas_price * gas_used`. The refund to the sender is still `gas_price * (gas_limit - gas_used)`. The base fee portion of `gas_price * gas_used` is "burned" (not transferred anywhere). This is by design for EIP-1559 compatibility.

The deeper issue is the `delta_gas < 0` case: when native consumption is low relative to EVM gas, the correct `gas_used` should be *reduced* by `|delta_gas|`, giving the sender a larger refund. Instead, `gas_used` stays inflated, the sender is overcharged, and the excess tokens are neither refunded nor paid to the operator — they vanish from the sender's balance without being credited anywhere, violating token conservation.

The root cause is at line 75–80 of `refund_calculation.rs`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The `delta_gas < 0` branch is a known incomplete implementation (marked with a TODO), meaning the sender is systematically overcharged whenever native resource consumption is lower than EVM gas consumption implies.

### Impact Explanation

**Token conservation violation / operator underpayment**: When `delta_gas < 0`, the sender loses `gas_price * |delta_gas|` tokens that are neither refunded nor paid to the operator. Over many transactions, this represents a permanent loss of user funds. The magnitude scales with `|delta_gas|` and `gas_price`. An attacker who crafts transactions with high EVM gas usage but low native resource consumption (e.g., operations that are cheap to prove but expensive in EVM gas) maximizes the overcharge. The tokens are effectively burned from the sender's balance without any corresponding benefit.

**Scope**: This affects all L2 ZK transactions processed through `ZkTransactionFlow::before_refund` → `compute_gas_refund` when `native_per_gas > 0` and `native_used / native_per_gas < gas_used`.

### Likelihood Explanation

The `delta_gas < 0` case is reachable whenever a transaction uses more EVM gas than its native resource consumption implies. This is a normal operating condition — many EVM operations (e.g., SLOAD of warm slots, simple arithmetic) are cheap to prove but consume standard EVM gas. Any unprivileged user submitting a standard L2 transaction can trigger this path. The TODO comment confirms the developers are aware this case is unhandled.

### Recommendation

In `compute_gas_refund`, handle the `delta_gas < 0` case by reducing `gas_used`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption was less than EVM gas implies; refund the difference
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Ensure gas_used does not fall below minimal_gas_used
    gas_used = gas_used.max(minimal_gas_used);
}
```

This ensures token conservation: `refund + operator_payment = precharge` in all cases.

### Proof of Concept

Consider a ZK L2 transaction with:
- `gas_limit = 100_000`
- `gas_price = 1000` (so `fee_to_prepay = 100_000_000`)
- `native_per_gas = 10` (so `native_limit = 1_000_000`)
- EVM execution uses `gas_used_ergs = 50_000` gas (ergs path)
- Native resource consumed: `native_used = 200_000` (only 20,000 gas equivalent)

In `compute_gas_refund`:
- `gas_used = 50_000` (from ergs)
- `native_used = 1_000_000 - remaining_native`; suppose `remaining_native = 800_000`, so `native_used = 200_000`
- `delta_gas = (200_000 / 10) as i64 - 50_000 as i64 = 20_000 - 50_000 = -30_000`
- Since `delta_gas < 0`, nothing happens. `gas_used` stays at `50_000`.

Result:
- `token_to_refund = 1000 * (100_000 - 50_000) = 50_000_000`
- `token_to_pay_operator = 1000 * 50_000 = 50_000_000`
- Total disbursed = `100_000_000` = precharge ✓ (balances in non-`burn_base_fee` mode)

But the *correct* `gas_used` should be `max(50_000 - 30_000, minimal_gas_used) = 20_000`:
- Correct `token_to_refund = 1000 * 80_000 = 80_000_000`
- Correct `token_to_pay_operator = 1000 * 20_000 = 20_000_000`

The sender is overcharged by `30_000_000` tokens (30,000 gas × 1000 gas_price). With `burn_base_fee` enabled, the discrepancy becomes a direct token burn from the sender's balance.

**Entry path**: Any unprivileged user submits a standard EIP-1559 L2 transaction → `validate_and_compute_fee_for_transaction` → `precharge_fee` → execution → `before_refund` → `compute_gas_refund` (line 75–80 of `refund_calculation.rs`). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
