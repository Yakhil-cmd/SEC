Audit Report

## Title
Unbounded Backoff Growth via Byzantine Peer Body-Stall Allows Indefinite CUP Fetch Delay — (`rs/orchestrator/src/catch_up_package_provider.rs`)

## Summary

`CatchUpPackageProvider::fetch_catch_up_package` uses a single shared `self.backoff` field as the body-read timeout for HTTP responses from peers. A Byzantine peer that sends HTTP 200 headers promptly but then stalls the body stream causes `self.backoff` to double on each occurrence with no upper bound. Because `self.backoff` is shared state, a single malicious peer can grow the body-read timeout to `Duration::MAX`, permanently stalling the node's CUP fetch loop until the orchestrator process is restarted.

## Finding Description

Two separate timeouts govern the HTTP exchange in `fetch_catch_up_package`:

**Header/connection timeout** — fixed at 10 seconds, not attacker-influenced: [1](#0-0) 

**Body timeout** — uses the mutable shared `self.backoff`: [2](#0-1) 

On body timeout, the backoff doubles with no cap: [3](#0-2) 

On success, the backoff resets to `initial_backoff`: [4](#0-3) 

There is no `MAX_BACKOFF` constant or `.min(...)` clamp anywhere in the file. `saturating_mul(2)` will eventually saturate at `Duration::MAX` (~584 years) after approximately 30 consecutive stalls from an initial 30-second backoff.

For an **unassigned node that already has a local CUP** (`current_node_index == None`), the peer selection logic limits the attempt to exactly **1 peer per cycle**: [5](#0-4) 

This means a Byzantine peer that is the only reachable node will be selected every cycle, causing deterministic unbounded backoff growth. Once `self.backoff` saturates, every subsequent body read — including from honest peers — will wait for `Duration::MAX` before timing out, permanently blocking the CUP fetch loop.

## Impact Explanation

Once `self.backoff` saturates, the orchestrator's CUP fetch loop blocks indefinitely on every peer's body read. The node cannot obtain a newer CUP, cannot detect a replica version change, and cannot execute a subnet upgrade. The node is effectively partitioned from its subnet until the orchestrator process is manually restarted. This constitutes a **High** impact application/platform-level DoS: it disrupts subnet availability for the affected node and prevents it from participating in coordinated subnet upgrades, which is a concrete protocol-level harm beyond a mere correctness issue.

## Likelihood Explanation

The attack is most reliable when the Byzantine peer is the only peer reachable by the victim (e.g., a network partition leaving only the attacker's node reachable). In that case, the Byzantine peer is selected every cycle and the backoff doubles deterministically. The precondition — controlling one reachable peer — is within the fault model for unassigned/joining nodes. The Byzantine peer only needs to complete the TLS handshake and send HTTP 200 headers within 10 seconds, then stall the body stream, which is straightforward to implement.

## Recommendation

Add a maximum cap when updating `self.backoff`:

```rust
const MAX_BACKOFF: Duration = Duration::from_secs(300); // e.g. 5 minutes

// on timeout:
self.backoff = old_backoff.saturating_mul(2).min(MAX_BACKOFF);
```

This ensures that even after repeated Byzantine stalls, the body-read timeout for honest peers is bounded and the node can recover within a predictable window. [6](#0-5) 

## Proof of Concept

The existing test `test_fetch_catch_up_package_body_request_times_out` already demonstrates the doubling behavior: [7](#0-6) 

An extended test with two mock servers — one stalling (`TestService::SendBodyOrStall` with `send_cup = false`), one honest — would confirm that after N stalling responses the honest server is contacted with a body timeout of `initial_backoff * 2^N`, not `initial_backoff`, violating the bounded-delay invariant. The `slow_body_service` already sends one byte then stalls for 3600 seconds, which is exactly the Byzantine behavior described. [8](#0-7)

        let url = format!("https://{server_addr}");

        let node_id = node_test_id(1);

        let initial_backoff = Duration::from_secs(5);
        let mut cup_provider =
            make_cup_provider(tmp_dir.path().to_path_buf(), node_id, initial_backoff);

        let err = cup_provider
            .fetch_catch_up_package(&node_id, url.clone(), None)
            .await
            .expect_err("Expected timeout error when fetching CUP from slow server");

        assert!(
            err.contains("Timed out while reading CUP response body")
                && err.contains("after 5 secs: Elapsed(()). Setting backoff to 10 secs")
        );

        // Verify that the backoff was increased
        assert_eq!(cup_provider.backoff, Duration::from_secs(10));

        // Allow the next request to succeed
        *send_cup.lock().unwrap() = true;

        let cup = cup_provider
            .fetch_catch_up_package(&node_id, url, None)
            .await
            .expect("Expected to fetch the CUP successfully")
            .expect("Expected non-empty CUP");

        assert_eq!(cup, fake_cup());

        // Verify that the backoff was reset after a successful request
        assert_eq!(cup_provider.backoff, initial_backoff);
