### Title
Silent Success on Call to Potentially Empty `L2_ASSET_TRACKER_ADDRESS` Skips Deposit Accounting — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`notify_l2_asset_tracker` calls `L2_ASSET_TRACKER_ADDRESS` (`0x1000f`) without verifying that a contract is deployed there. Per EVM semantics, a call to an empty account always returns `success: true` with empty returndata. The bootloader interprets this as a successful notification, proceeds to mint base tokens via `transfer_from_treasury`, and never detects that the asset tracker was never actually invoked. The code itself documents this assumption without enforcing it.

---

### Finding Description

In `notify_l2_asset_tracker`, the bootloader dispatches `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` to `L2_ASSET_TRACKER_ADDRESS` via `run_single_interaction`:

```rust
BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
    system,
    system_functions,
    memories,
    &calldata,
    &L2_BASE_TOKEN_ADDRESS,
    &L2_ASSET_TRACKER_ADDRESS,   // ← no contract-existence check
    inf_ergs.clone(),
    &U256::ZERO,
    true,
    tracer,
    validator,
)?;
``` [1](#0-0) 

The only post-call check is:

```rust
if failed {
    return Err(internal_error!(...).into());
}
``` [2](#0-1) 

The bootloader's own call dispatch path explicitly documents the silent-success behavior for empty accounts:

```rust
// By convention, calls to empty accounts succeed without any return data
if next_ee_type == ExecutionEnvironmentType::NoEE {
    ...
    return Ok((
        ...,
        CallResult::Successful { return_values: ReturnValues::empty() },
    ));
}
``` [3](#0-2) 

The code comment in `notify_l2_asset_tracker` acknowledges this assumption without enforcing it:

> "If no contract is deployed at L2AssetTracker, the call succeeds silently (a call to an empty address returns success with no returndata in EVM). However, we are certain that L2AssetTracker is available after the upgrade." [4](#0-3) 

After `notify_l2_asset_tracker` returns `Ok(())`, `transfer_from_treasury` unconditionally mints the tokens: [5](#0-4) 

`L2_ASSET_TRACKER_ADDRESS` is defined as `B160::from_limbs([0x1000f, 0, 0])`: [6](#0-5) 

---

### Impact Explanation

If `L2_ASSET_TRACKER_ADDRESS` has no deployed contract (e.g., pre-upgrade chain state, a chain that has not installed the predeploy, or any environment where `install_default_predeployed_contracts` was not called), every L1 deposit with `to_mint > 0` will:

1. Call `notify_l2_asset_tracker` → silent `success: true` (empty account EVM behavior)
2. Proceed to `transfer_from_treasury` → tokens are minted to the recipient
3. The `L2AssetTracker`'s `totalSuccessfulDepositsFromL1` is never incremented

This breaks the protocol's cross-chain token accounting invariant: tokens are minted on L2 without the asset tracker recording the deposit. Downstream logic that relies on `totalSuccessfulDepositsFromL1` (e.g., interop accounting, migration checks) will operate on stale/incorrect data, potentially enabling double-accounting or bypassing migration guards.

---

### Likelihood Explanation

The entry path is fully unprivileged: any user can submit an L1 transaction with `to_mint > 0`. The `notify_l2_asset_tracker` function is triggered for every such transaction. [7](#0-6) 

The condition is `amount > U256::ZERO || Config::SIMULATION`. In production, the predeploy is installed by `install_default_predeployed_contracts`. However, the vulnerability is live in any chain instance that has not yet run the upgrade deploying the asset tracker, or in any environment (e.g., a new chain, a fork, a simulation mode) where the predeploy is absent. The code's own comment confirms this is a known, unguarded assumption.

---

### Recommendation

Add an explicit contract-existence check before calling `L2_ASSET_TRACKER_ADDRESS`, analogous to OpenZeppelin's `Address.functionCall()` pattern. In the Rust bootloader context, read the callee's `unpadded_code_len` (or `bytecode_hash`) before dispatching the call, and treat a zero-code address as a fatal error rather than a silent success:

```rust
// Before calling run_single_interaction:
let code_len = system.io.read_account_properties(
    ExecutionEnvironmentType::NoEE,
    resources,
    &L2_ASSET_TRACKER_ADDRESS,
    AccountDataRequest::empty().with_unpadded_code_len(),
)?.unpadded_code_len.0;

if code_len == 0 {
    return Err(internal_error!(
        "L2AssetTracker not deployed at expected address"
    ).into());
}
```

This converts the silent assumption into an explicit, enforceable invariant, matching the fix pattern recommended in the external report.

---

### Proof of Concept

1. Deploy a ZKsync OS chain **without** calling `install_default_predeployed_contracts` (i.e., `L2_ASSET_TRACKER_ADDRESS` has no bytecode).
2. Submit an L1 transaction with `to_mint = 1_000_000` (any nonzero value).
3. Observe: `notify_l2_asset_tracker` calls `L2_ASSET_TRACKER_ADDRESS`, receives `CallResult::Successful { return_values: ReturnValues::empty() }` (empty-account EVM behavior).
4. `failed` is `false`; no error is raised.
5. `transfer_from_treasury` mints `1_000_000` tokens to the recipient.
6. Query `L2AssetTracker.totalSuccessfulDepositsFromL1` → returns `0` (never updated).
7. Tokens exist on-chain but are untracked by the asset tracker, breaking cross-chain accounting invariants.
