Audit Report

## Title
Unprivileged Caller Can Force Any Principal With ICP in Swap Subaccount Into SNS Token Swap Participation - (File: rs/sns/swap/canister/canister.rs)

## Summary

`refresh_buyer_tokens` in the SNS Swap canister accepts an arbitrary `buyer` principal from the caller-supplied request payload with no check that the caller matches the specified buyer. Any unprivileged ingress sender can call this method with a victim's principal, causing the victim's ICP already sitting in the swap's subaccount to be irrevocably committed as swap participation — locking the victim's funds until the swap finalizes without their consent.

## Finding Description

In `rs/sns/swap/canister/canister.rs` at L130-134, the effective buyer principal is resolved from `arg.buyer` with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // ← any principal accepted
};
``` [1](#0-0) 

This `p` is passed directly to `refresh_buyer_token_e8s`, which reads the ICP balance of `principal_to_subaccount(&buyer)` on the ICP ledger and registers the buyer as a committed participant. [2](#0-1) [3](#0-2) 

The ticket system (L1248-1272) is explicitly optional: the comment at L1271 states "If there exists no ticket for the buyer, the payment flow will simply ignore the ticket." An attacker calling on behalf of a victim who never created a ticket bypasses this check entirely. [4](#0-3) 

The `confirmation_text` field is not a meaningful mitigation: it is stored in the swap's `Init` payload and is publicly readable via the `get_init` query endpoint (no authentication required). [5](#0-4) [6](#0-5) 

Once committed, the victim's ICP cannot be reclaimed via `error_refund_icp` while the swap is `OPEN`; that endpoint is gated on `Lifecycle::Aborted || Lifecycle::Committed`. [7](#0-6) 

The public Candid interface exposes `refresh_buyer_tokens` as an open update call with no caller restriction, and the proto definition confirms `buyer` is an optional free-text field defaulting to the caller only when empty. [8](#0-7) [9](#0-8) 

## Impact Explanation

This is an unauthorized commitment of a user's ICP to a financial outcome without their consent, matching **High ($2,000–$10,000): Unauthorized access to governance assets, wallets, or canister-controlled funds**. A victim who has transferred ICP to the swap subaccount (a normal intermediate state every participant passes through) can be forcibly enrolled as a buyer. Their ICP is locked until finalization; if the swap commits, they receive SNS tokens they never agreed to purchase; if it aborts, they recover ICP minus transfer fees only after the swap closes. The attack is applicable to every open SNS swap simultaneously.

## Likelihood Explanation

The attack requires only a standard ingress call from any unprivileged principal — no special role, key, or majority is needed. The victim's precondition (ICP sitting in the swap subaccount) is a normal intermediate state that every participant passes through, creating a non-trivial exploitation window. The `confirmation_text` (when present) is publicly readable from `get_init`, providing no barrier. The attack is cheap (one update call) and repeatable across all open SNS swaps.

## Recommendation

In `refresh_buyer_tokens`, reject any request where `arg.buyer` is non-empty and does not match `caller_principal_id()`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        if requested != caller {
            panic!("caller {} is not authorized to refresh tokens on behalf of {}", caller, requested);
        }
        requested
    };
    // ...
}
```

Alternatively, remove the `buyer` field entirely and always use `caller_principal_id()`, since the only legitimate use case is a principal notifying the swap of their own transfer.

## Proof of Concept

**Setup:** SNS swap is `OPEN`. Victim `V` transfers 10 ICP to `principal_to_subaccount(V)` on the ICP ledger (the required first step of participation). Attacker `A` reads `confirmation_text` (if any) from `get_init`.

**Attack (single ingress call from attacker `A`):**

```
dfx canister call <swap_canister_id> refresh_buyer_tokens '(
  record {
    buyer = "<victim_principal_text_id>";
    confirmation_text = null
  }
)'
```

**Result:** `refresh_buyer_token_e8s` reads the balance of `principal_to_subaccount(V)`, finds 10 ICP, and registers `V` as a buyer. `V` is now a committed participant with their ICP locked until finalization. `V` cannot call `error_refund_icp` while the swap is `OPEN`. If the swap commits, `V` receives SNS tokens they never agreed to purchase.

A deterministic integration test can be written using the existing `StateMachine`/PocketIC test infrastructure: create two principals, have one transfer ICP to the swap subaccount without calling `new_sale_ticket`, then call `refresh_buyer_tokens` from the second principal with the first principal's ID as `buyer`, and assert the first principal's buyer state is now set.
