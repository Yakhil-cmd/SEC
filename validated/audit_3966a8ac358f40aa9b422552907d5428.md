## Validity Assessment

The code path is concrete and fully traceable. However, the exploit requires **two simultaneous conditions**:

1. **`base_fee_per_gas = 0`** — The block base fee must be zero. In Aurora's production deployment on NEAR mainnet, the base fee is a configurable engine state value. The `EngineState::default()` initializes it to zero, and there is no enforced minimum. Whether production Aurora currently runs with a zero base fee is a deployment question, but the code imposes no guard against it.

2. **`max_gas_price = Some(0)`** — The caller of `submit_with_args` sets this. `submit_with_args` is a public NEAR contract method. An EVM transaction sender can bypass relayers entirely and call it directly, setting `max_gas_price = 0`.

When both conditions hold:
- The sender pays **zero** ETH for a transaction that executes normally.
- The relayer receives **zero** priority-fee reward, even though the signed EVM transaction declared a nonzero `max_priority_fee_per_gas`.
- The protocol collects nothing.

The existing test `test_refund_free_effective_gas_does_nothing` explicitly validates and accepts this behavior as correct: [7](#0-6) 

This confirms the early-return is intentional design, not a bug catch — but the design does not account for the case where `effective_gas_price` reaches zero via `max_gas_price` suppression rather than a genuinely free-gas context.

## Verdict

### Title
Priority-fee zeroed via `max_gas_price=0` when `base_fee=0`, causing free execution and no relayer reward — (`engine/src/engine.rs` → `charge_gas` / `refund_unused_gas`)

### Summary
When `block_base_fee_per_gas = 0` and `SubmitArgs.max_gas_price = Some(0)`, `charge_gas` computes `effective_gas_price = 0` regardless of the transaction's `max_priority_fee_per_gas`. `refund_unused_gas` then returns early, crediting nothing to the relayer. The sender pays nothing and the transaction executes normally.

### Finding Description
In `charge_gas`, the `max_gas_price` cap is applied unconditionally to `priority_fee_per_gas`: [8](#0-7) 

```rust
let priority_fee_per_gas = max_gas_price.map_or(priority_fee_per_gas, |price| {
    price.min(priority_fee_per_gas)
});
```

When `max_gas_price = 0`, this forces `priority_fee_per_gas = 0`. With `base_fee = 0`, `effective_gas_price = 0`. The `refund_unused_gas` guard at line 1270 then short-circuits before any relayer reward is computed or paid. [5](#0-4) 

### Impact Explanation
- The EVM transaction sender pays zero ETH for execution.
- The relayer receives zero priority-fee reward despite the transaction declaring a nonzero `max_priority_fee_per_gas`.
- This permanently suppresses relayer income for any transaction submitted this way, constituting **theft of unclaimed yield** (relayer's priority-fee yield is zeroed while the sender pays nothing).

### Likelihood Explanation
Requires `base_fee = 0`, which is the default engine state and may apply in certain Aurora deployments or configurations. `submit_with_args` is a public NEAR method, so any user can call it directly with `max_gas_price = Some(0)`. No privilege is required.

### Recommendation
In `charge_gas`, add a guard: if `max_gas_price` is `Some(0)` but the transaction's computed `priority_fee_per_gas` (before capping) is nonzero, either reject the transaction or ignore the `max_gas_price = 0` cap (treating it as `None`). Alternatively, enforce that `max_gas_price`, when `Some`, must be at least `base_fee_per_gas` to prevent suppression of the effective gas price below the base fee floor.
