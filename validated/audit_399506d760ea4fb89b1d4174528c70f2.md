### Title
Missing Zero-Address Check in `AdminControlled` Constructor Permanently Locks `mint()` on ERC-20 Bridge Tokens - (`File: etc/eth-contracts/contracts/AdminControlled.sol`)

---

### Summary

`AdminControlled.sol` accepts an `address _admin` constructor argument and assigns it directly to `admin` without validating `_admin != address(0)`. The suppressed Slither warning (`// slither-disable-next-line missing-zero-check`) explicitly acknowledges the omission. Both `EvmErc20` and `EvmErc20V2` — the ERC-20 mirror contracts used by Aurora's NEP-141 bridge — inherit this constructor and pass the `admin` argument through without any zero-address guard. If `address(0)` is supplied at deployment, the `onlyAdmin` modifier is permanently unsatisfiable, locking `mint()` forever and rendering the bridge token non-functional.

---

### Finding Description

`AdminControlled.sol` sets `admin` in its constructor with no zero-address check: [1](#0-0) 

The `onlyAdmin` modifier enforces `require(msg.sender == admin)`: [2](#0-1) 

No EVM transaction can have `msg.sender == address(0)`, so if `admin` is zero the modifier can never pass.

`EvmErc20` and `EvmErc20V2` both inherit `AdminControlled` and forward the `admin` constructor argument without any guard: [3](#0-2) [4](#0-3) 

The `mint()` function — the sole mechanism by which the Aurora Engine credits users with ERC-20 tokens when they bridge NEP-141 assets — is gated by `onlyAdmin`: [5](#0-4) 

The Aurora Engine deploys `EvmErc20` / `EvmErc20V2` via `setup_deploy_erc20_input`, computing the admin address as `current_address(current_account_id)`: [6](#0-5) 

Under normal operation this address is non-zero. However, the `EvmErc20` constructor itself imposes no guard, so any deployment path — including a future engine code change, a misconfigured silo deployment, or a direct EVM deployer — that supplies `address(0)` will silently succeed and produce a permanently broken bridge token with no recovery path (there is no `setAdmin` function in `AdminControlled`).

---

### Impact Explanation

If `admin = address(0)` is set at deployment:

- `mint()` is permanently locked. The Aurora Engine calls `mint()` inside `receive_erc20_tokens` every time a user bridges a NEP-141 token to Aurora via `ft_on_transfer`.
- Every bridging attempt reverts at the EVM level. The `ft_on_transfer` handler returns the full token amount to the caller, so NEP-141 tokens are not permanently lost, but the bridge token is rendered permanently non-functional.
- `adminPause()` is also locked, so the contract can never be paused or recovered by the engine.
- There is no upgrade or admin-transfer path; the state is irrecoverable without redeploying the ERC-20 contract and migrating all state.

**Impact: High — Temporary freezing of funds** (all bridging to this ERC-20 mirror is permanently blocked; users cannot receive their Aurora-side tokens).

---

### Likelihood Explanation

The current engine deployment path (`setup_deploy_erc20_input` → `current_address`) produces a non-zero admin under normal conditions. However:

- The constructor has **no on-chain enforcement** of the non-zero invariant.
- A silo operator, a future engine refactor, or a direct EVM deployer can trivially supply `address(0)`.
- The suppressed Slither warning (`// slither-disable-next-line missing-zero-check`) shows the gap is known but unmitigated.
- The `mirror_erc20_token` path also calls `setup_deploy_erc20_input` and is subject to the same risk. [7](#0-6) 

Likelihood: **Medium** — not triggered by the current engine path, but reachable by any deployer of the contract or by a future engine change.

---

### Recommendation

Add a zero-address guard in the `AdminControlled` constructor and in both `EvmErc20` / `EvmErc20V2` constructors:

```solidity
// AdminControlled.sol
constructor(address _admin, uint flags) {
    require(_admin != address(0), "AdminControlled: admin is zero address");
    admin = _admin;
    paused = flags;
}
```

```solidity
// EvmErc20.sol / EvmErc20V2.sol
constructor (string memory metadata_name, string memory metadata_symbol, uint8 metadata_decimals, address admin)
    ERC20(metadata_name, metadata_symbol)
    AdminControlled(admin, 0)
{
    require(admin != address(0), "EvmErc20: admin is zero address");
    ...
}
```

Remove the `// slither-disable-next-line missing-zero-check` suppression comments.

---

### Proof of Concept

1. Deploy `EvmErc20` with `admin = address(0)`:
   ```solidity
   EvmErc20 token = new EvmErc20("Test", "TST", 18, address(0));
   // Deployment succeeds silently — no revert.
   ```

2. Attempt to mint tokens (as the Aurora Engine would during bridging):
   ```solidity
   token.mint(userAddress, 1000);
   // Reverts: require(msg.sender == admin) → require(msg.sender == address(0)) → always false.
   ```

3. The `mint()` function is permanently inaccessible. The bridge token is non-functional. Any NEP-141 → Aurora bridging attempt via `ft_on_transfer` will fail at the EVM `mint()` call, causing `receive_erc20_tokens` to return an error, and `ft_on_transfer` to refund the full NEP-141 amount — permanently blocking the bridge for this token with no on-chain recovery path. pub fn setup_deploy_erc20_input(
    current_account_id: &AccountId,
    erc20_metadata: Option<Erc20Metadata>,
) -> Vec<u8> {
    #[cfg(feature = "error_refund")]
    let erc20_contract = include_bytes!("../../etc/eth-contracts/res/EvmErc20V2.bin");
    #[cfg(not(feature = "error_refund"))]
    let erc20_contract = include_bytes!("../../etc/eth-contracts/res/EvmErc20.bin");

    let erc20_admin_address = current_address(current_account_id);
    let erc20_metadata = erc20_metadata.unwrap_or_default();

    let deploy_args = ethabi::encode(&[
        ethabi::Token::String(erc20_metadata.name),
        ethabi::Token::String(erc20_metadata.symbol),
        ethabi::Token::Uint(erc20_metadata.decimals.into()),
        ethabi::Token::Address(erc20_admin_address.raw().0.into()),
    ]);

    [erc20_contract, deploy_args.as_slice()].concat()
}

/// Used to mirror deployed ERC-20 contract on main contract to silo.
pub fn mirror_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    args: MirrorErc20TokenArgs,
    erc20_address: Address,
    erc20_metadata: Erc20Metadata,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, Some(erc20_metadata));
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, Some(erc20_address), handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    assert_eq!(address, erc20_address);

    sdk::log!(
        "ERC-20 on: {} at address: {} has been mirrored",
        args.contract_id.as_ref(),
        address.encode()
    );
    engine
        .register_token(address, args.nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
