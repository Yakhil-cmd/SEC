### Title
Unverified `amount` in `ft_on_transfer` Enables Unbacked ERC-20 Minting via Malicious NEP-141 Callback ÔÇö (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` function in Aurora Engine mints ERC-20 tokens based solely on the `amount` field supplied by the calling NEP-141 contract, with no on-chain verification that Aurora Engine's actual NEP-141 balance increased by that amount. Because `deploy_erc20_token` carries no access-control guard, any unprivileged NEAR account can register an arbitrary NEP-141 identity and then invoke `ft_on_transfer` directly ÔÇö or through a malicious NEP-141 contract ÔÇö to mint an unbounded quantity of ERC-20 tokens backed by zero real assets.

---

### Finding Description

**Entry point ÔÇö `ft_on_transfer`**

`ft_on_transfer` is Aurora Engine's public NEP-141 receiver callback. [1](#0-0) 

The function branches on `predecessor_account_id`: if it equals the configured connector account it calls `receive_base_tokens`; otherwise it calls `receive_erc20_tokens`, passing `predecessor_account_id` as the token identifier and `args.amount` as the mint quantity. [2](#0-1) 

**No balance verification in `receive_erc20_tokens`**

`receive_erc20_tokens` resolves the ERC-20 contract for the NEP-141 token and immediately mints `amount` tokens. There is no query of Aurora Engine's actual NEP-141 balance before or after the call to confirm the claimed transfer occurred. [3](#0-2) 

The mint input is constructed directly from the caller-supplied `amount`: [4](#0-3) 

**Permissionless NEP-141 registration via `deploy_erc20_token`**

`deploy_erc20_token` contains only a `require_running` guard ÔÇö no owner-only or whitelist check ÔÇö so any NEAR account can register an arbitrary NEP-141 identity and obtain a corresponding ERC-20 contract on Aurora. [5](#0-4) 

**Analogy to the reported vulnerability**

In the LSSVMRouter finding, a malicious `assetRecipient` exploits the `onERC721Received` callback to manipulate the NFT balance before the router finalises accounting. Here, the malicious NEP-141 contract (or a direct caller whose account is registered as a NEP-141 token) is the `assetRecipient` analog: it controls the `amount` field passed into the Aurora Engine callback, and Aurora Engine finalises the ERC-20 mint without independently verifying the actual asset transfer.

---

### Impact Explanation

**Critical ÔÇö Direct theft of funds.**

An attacker who registers a malicious NEP-141 identity can mint an arbitrary quantity of the corresponding ERC-20 token on Aurora with zero real NEP-141 backing. Those tokens are indistinguishable from legitimately bridged tokens and can be sold on Aurora DEXes, used as collateral, or redeemed through the bridge, draining real assets from honest liquidity providers and bridge users.

---

### Likelihood Explanation

**Medium.**

The attack requires two permissionless on-chain actions (registering a NEP-141 identity and calling `ft_on_transfer`), both callable by any unprivileged NEAR account. No leaked keys, governance capture, or oracle manipulation is needed. The only friction is the gas cost of the two transactions.

---

### Recommendation

1. **Verify the actual balance change.** Before minting ERC-20 tokens, record Aurora Engine's NEP-141 balance of the token, then after the transfer resolves, mint only the observed delta ÔÇö not the caller-supplied `amount`.

2. **Restrict `deploy_erc20_token` to trusted callers.** Add an owner-only or whitelist guard (analogous to `require_owner_only` used elsewhere in the codebase) so that only vetted NEP-141 contracts can be registered.

3. **Validate `ft_on_transfer` callers.** Maintain an explicit allowlist of registered NEP-141 contracts and reject `ft_on_transfer` calls whose `predecessor_account_id` is not on that list, preventing direct calls from arbitrary NEAR accounts.

---

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
