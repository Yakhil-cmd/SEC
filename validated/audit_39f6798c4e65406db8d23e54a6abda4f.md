Audit Report

## Title
Maliciously Crafted CatchUpPackageShare Heights Cause Permanent Unvalidated Pool Purge Bypass Leading to Memory Exhaustion - (File: `rs/consensus/src/consensus/purger.rs`)

## Summary
The `purge_unvalidated_pool_by_expected_batch_height` function in `rs/consensus/src/consensus/purger.rs` guards against purging by checking whether the unvalidated `CatchUpPackageShare` pool's `height_range()` spans the interval `(range.min, range.max)` relative to `catch_up_height` and `expected_batch_height`. A Byzantine peer below the fault threshold can inject two crafted shares — one with height 0 and one with height `u64::MAX` — to permanently set `height_range()` to `[0, u64::MAX]`, causing the guard to evaluate to `true` on every purger invocation and suppressing all unvalidated pool purging. The attacker can then flood the pool with additional crafted shares, exhausting node memory and halting the subnet.

## Finding Description

**Root cause 1 — Purger early-return guard (`rs/consensus/src/consensus/purger.rs`, lines 198–218):**

```rust
fn below_range_max(h: Height, range: &Option<HeightRange>) -> bool {
    range.as_ref().map(|r| h < r.max).unwrap_or(false)
}
fn above_range_min(h: Height, range: &Option<HeightRange>) -> bool {
    range.as_ref().map(|r| h > r.min).unwrap_or(false)
}
// Skip purging if we have unprocessed but needed CatchUpPackageShare
let unvalidated_catch_up_share_range =
    unvalidated_pool.catch_up_package_share().height_range();
if below_range_max(catch_up_height, &unvalidated_catch_up_share_range)
    && above_range_min(expected_batch_height, &unvalidated_catch_up_share_range)
{
    return;
}
```

The intent is to skip purging only when there are shares with heights strictly between `catch_up_height` and `expected_batch_height`. However, the implementation checks only whether the pool's overall `[min, max]` range spans that interval — not whether any share actually exists at a height in that interval. With `range = [0, u64::MAX]`, `catch_up_height < u64::MAX` is always true and `expected_batch_height > 0` is always true (batch height starts at 1), so the guard fires on every invocation. Critically, `prev_expected_batch_height` is updated at line 219 only after the guard, so it is never updated when the early return fires, meaning the outer condition `*prev_expected_batch_height < expected_batch_height` remains true on every subsequent call as the subnet advances.

**Root cause 2 — Validator excludes height-0 shares from processing (`rs/consensus/src/consensus/validator.rs`, lines 1601–1617):**

```rust
let range = HeightRange::new(catch_up_height.increment(), max_height);
let shares = pool_reader.pool().unvalidated()
    .catch_up_package_share().get_by_height_range(range);
```

A share with `height = 0` is below `catch_up_height.increment()` and is never processed by the validator. It remains in the unvalidated pool indefinitely, permanently anchoring `range.min = 0`.

**Root cause 3 — Validator leaves transient-error shares in pool (`rs/consensus/src/consensus/validator.rs`, lines 1649–1658):**

```rust
Err(ValidationError::ValidationFailed(err)) => {
    if self.unvalidated_for_too_long(pool_reader, &share.get_id()) {
        warn!(...);
    }
    None  // share stays in unvalidated pool
}
```

For a share with `height = u64::MAX`, `validate_catch_up_share_content` calls `pool_reader.get_finalized_block(u64::MAX)` → `None` → `ValidationFailure::FinalizedBlockNotFound(u64::MAX)`. This is a transient error, so the share is never removed, permanently anchoring `range.max = u64::MAX`.

**Root cause 4 — `check_integrity` does not validate height bounds (`rs/types/types/src/consensus.rs`, lines 1737–1741):**

```rust
fn check_integrity(&self) -> bool {
    let content = &self.content;
    let random_beacon_hash = content.random_beacon.get_hash();
    &crypto_hash(content.random_beacon.as_ref()) == random_beacon_hash
}
```

Only the hash self-consistency is checked. An attacker can construct a `RandomBeacon` with any height, compute its hash, and the integrity check passes. No threshold key material is required.

**Root cause 5 — No height pre-validation at pool insertion (`rs/artifact_pool/src/consensus_pool.rs`, line 700–705):**

```rust
fn insert(&mut self, unvalidated_artifact: UnvalidatedConsensusArtifact) {
    let mut ops = PoolSectionOps::new();
    self.record_instant_unvalidated(&unvalidated_artifact.message);
    ops.insert(unvalidated_artifact);
    self.apply_changes_unvalidated(ops);
}
```

Artifacts received from peers are inserted into the unvalidated pool without any height range validation. The validator is the sole gatekeeper, and as shown above, it fails to remove the anchor shares.

## Impact Explanation

Once the two anchor shares are in the unvalidated pool, the purger's `purge_unvalidated_pool_by_expected_batch_height` permanently returns early, emitting no `PurgeUnvalidatedBelow` action. The attacker can then continuously broadcast additional `CatchUpPackageShare` messages (with any height ≤ `catch_up_height` or any height for which no finalized block exists). These are accepted into the unvalidated pool, never validated, and never purged. The unvalidated pool grows without bound, exhausting node memory. This can crash all honest nodes in the subnet, halting consensus — a concrete subnet availability impact matching **High ($2,000–$10,000): Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation

A single Byzantine consensus peer below the fault threshold can execute this attack. No threshold key material, admin access, or majority corruption is required. The attacker only needs to:
1. Construct a `CatchUpPackageShare` with `random_beacon.height = 0` and compute its self-consistent hash (trivial, no cryptographic secrets needed).
2. Construct a `CatchUpPackageShare` with `random_beacon.height = u64::MAX` and compute its self-consistent hash.
3. Broadcast both via the standard P2P gossip protocol to honest nodes.

The attack is persistent: once the anchor shares are in the pool, the purger is permanently suppressed. Node restarts do not help because the shares are re-gossiped by the attacker. The attack is cheap to sustain — the attacker only needs to keep the two anchor shares alive in the network.

## Recommendation

**Short term:** In `purge_unvalidated_pool_by_expected_batch_height`, replace the raw range-span check with a check that queries whether any share exists at a height strictly between `catch_up_height` and `expected_batch_height`:

```rust
let has_needed_share = unvalidated_pool
    .catch_up_package_share()
    .get_by_height_range(HeightRange::new(
        catch_up_height.increment(),
        expected_batch_height.decrement(),
    ))
    .next()
    .is_some();
if has_needed_share { return; }
```

**Long term:** Validate the height of all incoming `CatchUpPackageShare` artifacts at ingress (before inserting into the unvalidated pool), rejecting any share whose height is not within a reasonable window of the current finalized height. Additionally, consider purging unvalidated artifacts that have been in the pool longer than a configurable bound, regardless of transient validation errors.

## Proof of Concept

**Step 1:** Craft share `S` with `random_beacon.height = 0`:
- Construct `RandomBeacon { height: Height::from(0), parent: any_hash }`.
- Compute `random_beacon_hash = crypto_hash(&random_beacon)`.
- Build `CatchUpShareContent` embedding this beacon. `check_integrity()` passes since `crypto_hash(random_beacon) == random_beacon_hash`.
- `S.height() = 0 ≤ catch_up_height`, so the validator's range `[catch_up_height.increment(), max_height]` excludes it. `S` is never processed and stays in the unvalidated pool permanently.

**Step 2:** Craft share `S'` with `random_beacon.height = u64::MAX`:
- Construct `RandomBeacon { height: Height::from(u64::MAX), parent: any_hash }`.
- `check_integrity()` passes for the same reason.
- The validator's range includes `S'`. `validate_catch_up_share_content` calls `pool_reader.get_finalized_block(u64::MAX)` → `None` → `ValidationFailure::FinalizedBlockNotFound(u64::MAX)`. The share is left in the unvalidated pool.

**Step 3:** After both shares are inserted, `unvalidated_pool.catch_up_package_share().height_range()` returns `Some(HeightRange { min: Height(0), max: Height(u64::MAX) })`.

**Step 4:** On every purger invocation:
- `below_range_max(catch_up_height, &range)` → `catch_up_height < u64::MAX` → `true`
- `above_range_min(expected_batch_height, &range)` → `expected_batch_height > 0` → `true`
- `return;` — no purge action is emitted, `prev_expected_batch_height` is never updated.

**Step 5:** The attacker floods the network with additional crafted `CatchUpPackageShare` messages. Each is accepted into the unvalidated pool, never validated, never purged. Memory grows without bound until nodes crash.

**Reproducible test plan:** Write a unit test in `rs/consensus/src/consensus/purger.rs` that inserts a height-0 and a height-`u64::MAX` `CatchUpPackageShare` into the unvalidated pool, then calls `purge_unvalidated_pool_by_expected_batch_height` repeatedly while advancing `expected_batch_height`, and asserts that no `PurgeUnvalidatedBelow` action is ever emitted and the pool size grows unboundedly.
