### Title
`receive_base_tokens` Does Not Enforce the Silo Address Whitelist, Allowing ETH Deposits to Non-Whitelisted Addresses — (`engine/src/engine.rs`)

### Summary

In Silo mode, Aurora Engine maintains an `Address` whitelist (`WhitelistKind::Address`) to restrict which EVM addresses may receive tokens. The `receive_erc20_tokens` function correctly checks this whitelist and redirects tokens to a configured fallback address when the recipient is not whitelisted. However, the sibling function `receive_base_tokens` — which handles ETH (native token) deposits via the bridge — performs no such check. Any caller can bridge ETH to any EVM address, including addresses explicitly excluded by the Silo operator, bypassing the whitelist entirely.

### Finding Description

Both `receive_base_tokens` and `receive_erc20_tokens` are invoked from `ft_on_transfer` in `engine/src/contract_methods/connector.rs` depending on whether the predecessor is the ETH connector or a NEP-141 token contract.

`receive_erc20_tokens` (lines 818–822 of `engine/src/engine.rs`) correctly enforces the whitelist:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

`receive_base_tokens` (lines 773–789 of `engine/src/engine.rs`) performs no equivalent check:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    // No whitelist check here
    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance.checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

The recipient address is attacker-controlled (parsed from `args.msg`). ETH is unconditionally credited to that address regardless of whitelist status.

The dispatch logic in `ft_on_transfer` (`engine/src/contract_methods/connector.rs`, lines 81–90) routes to `receive_base_tokens` when the predecessor is the ETH connector:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
};
```

The `assert_access` function (`engine/src/engine.rs`, lines 1756–1775) — which enforces the whitelist for `submit`/`submit_with_args` — is never consulted in this path.

### Impact Explanation

In Silo mode with the `Address` whitelist enabled, the operator intends to restrict which EVM addresses can participate. ETH deposited to a non-whitelisted address via `receive_base_tokens` is credited to that address's balance. Because `submit` and `submit_with_args` enforce the `Address` whitelist, the non-whitelisted address cannot sign transactions to move those funds. The ETH is effectively frozen at that address until the operator explicitly whitelists it or the Silo configuration changes. This constitutes **temporary freezing of funds** (High impact).

### Likelihood Explanation

The entry path requires no special privileges. Any user who can bridge ETH to Aurora (a standard, publicly accessible operation) can specify an arbitrary EVM address as the recipient in the bridge message. The `ft_on_transfer` function is callable by the ETH connector, which is a normal bridge participant. No admin compromise, key leak, or governance capture is required.

### Recommendation

Apply the same whitelist guard in `receive_base_tokens` that exists in `receive_erc20_tokens`. When a Silo fallback address is configured and the recipient is not whitelisted, redirect the ETH deposit to the fallback address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut recipient = message_data.recipient;

    // Mirror the ERC-20 whitelist check for base token deposits
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
    {
        recipient = fallback_address;
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    Ok(None)
}
```

### Proof of Concept

1. Deploy Aurora Engine in Silo mode with `Address` whitelist enabled and a fallback address configured.
2. Ensure address `0xDEAD...` is **not** in the `Address` whitelist.
3. Bridge ETH to Aurora via the standard ETH connector, setting the recipient field in the bridge message to `0xDEAD...`.
4. `ft_on_transfer` is called with `predecessor_account_id == eth_connector_account_id`, routing to `receive_base_tokens`.
5. `receive_base_tokens` credits ETH to `0xDEAD...` with no whitelist check.
6. Observe that `0xDEAD...` now holds ETH but cannot call `submit` (blocked by whitelist), freezing the funds.
7. Contrast with an ERC-20 bridge transfer to the same address: the ERC-20 path redirects to the fallback address instead.

    io: &I,
    env: &E,

    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
