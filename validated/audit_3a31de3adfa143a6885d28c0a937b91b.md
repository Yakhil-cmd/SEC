### Title
Unordered Execution of Scheduled XCC Promises Allows Any Caller to Permanently Destroy Dependent Operations - (File: `etc/xcc-router/src/lib.rs`)

### Summary

The `execute_scheduled` function in the XCC router contract is intentionally callable by anyone and accepts an arbitrary nonce. This means any external NEAR account can execute scheduled promises in any order. When a user schedules multiple dependent XCC promises (e.g., `approve` then `transfer_from`), a third party can execute them out of order. Because the promise is **removed from storage before execution**, a failed out-of-order execution permanently destroys the scheduled promise, making the user's intended operation unrecoverable.

### Finding Description

The `Router` contract in `etc/xcc-router/src/lib.rs` supports two execution modes for XCC calls:

1. **Eager** (`CrossContractCallArgs::Eager`): calls `execute` on the router immediately within the same NEAR transaction.
2. **Delayed** (`CrossContractCallArgs::Delayed`): calls `schedule` on the router, storing the promise in a `LookupMap<u64, PromiseArgs>` keyed by a monotonically incrementing nonce. [1](#0-0) 

Scheduled promises are later executed via `execute_scheduled`, which is explicitly open to any caller: [2](#0-1) 

The critical flaw is that:
1. **No caller restriction** is enforced on `execute_scheduled` — any NEAR account can call it.
2. **No sequential ordering** is enforced — the caller supplies an arbitrary `nonce: U64`, so nonce `2` can be executed before nonce `0`.
3. **The promise is removed from storage before execution** (`self.scheduled_promises.remove(&nonce.0)`) — if the execution fails due to an ordering violation, the promise is permanently gone.

The nonce is assigned sequentially by `schedule`: [3](#0-2) 

But `execute_scheduled` imposes no constraint that nonces must be consumed in order: [4](#0-3) 

### Impact Explanation

Consider an EVM contract that schedules two dependent XCC calls to a NEAR DeFi protocol:

- **Nonce 0**: `ft_transfer_call` — transfer wNEAR to a staking contract
- **Nonce 1**: `stake` — stake the transferred tokens

If a third party calls `execute_scheduled(nonce: 1)` before `execute_scheduled(nonce: 0)`, the `stake` call executes first and fails (no tokens deposited yet). The promise at nonce 1 is **permanently removed** from the router's storage. The user then executes nonce 0 (the transfer succeeds), but nonce 1 is gone — the stake operation can never be retried through the router. The transferred tokens are now stuck in the staking contract with no scheduled operation to complete the flow.

This constitutes **temporary or permanent freezing of funds** depending on whether the target NEAR contract provides an alternative recovery path.

**Impact**: High — permanent loss of a scheduled operation; funds may be frozen in an intermediate state with no recovery path through the XCC system.

### Likelihood Explanation

The `execute_scheduled` function is a public NEAR contract method callable by any NEAR account with no gas or deposit requirement beyond standard transaction fees. The router account ID is deterministic (`{evm_address}.aurora`), and the nonce values are logged on-chain via `near_sdk::log!`. Any observer of the NEAR blockchain can monitor for `schedule` calls and front-run the execution with an out-of-order nonce. This is straightforward to exploit with basic NEAR tooling. [5](#0-4) 

### Recommendation

Enforce sequential execution of scheduled promises by tracking a separate "execution cursor" (next expected nonce). In `execute_scheduled`, reject any call where `nonce != execution_cursor`, and increment the cursor only on success. This mirrors the LayerZero-style ordered delivery suggested in the original report.

Alternatively, restrict `execute_scheduled` to be callable only by the parent (Aurora Engine), and have the engine enforce ordering at the EVM transaction level.

### Proof of Concept

1. EVM user (address `0xABCD...`) calls the XCC precompile with `CrossContractCallArgs::Delayed(promise_A)` — stored at nonce 0. [6](#0-5) 

2. Same user calls the XCC precompile with `CrossContractCallArgs::Delayed(promise_B)` — stored at nonce 1, where `promise_B` depends on `promise_A` having executed first. [1](#0-0) 

3. Attacker observes the NEAR chain log `"Promise scheduled at nonce 1"` and calls `execute_scheduled(nonce: 1)` on the router account `{0xABCD...}.aurora`. [7](#0-6) 

4. `promise_B` is removed from storage and dispatched. It fails on the target contract because `promise_A` has not yet run. The promise is permanently gone.

5. User calls `execute_scheduled(nonce: 0)` — `promise_A` executes successfully, but `promise_B` no longer exists in the router. The dependent operation is permanently lost and any funds moved by `promise_A` are now in an unrecoverable intermediate state.

```rust
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
```

```rust
    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }
```
