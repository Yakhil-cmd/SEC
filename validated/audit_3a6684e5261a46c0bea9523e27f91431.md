Audit Report

## Title
Byzantine Proposer Can Deliver `EmptyBlob` to a Pending `GetSuccessors` Callback via Unchecked `SendTransactionResponse` — (`rs/replicated_state/src/bitcoin.rs`, `rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary

`validate_payload` performs no check that a `BitcoinAdapterResponse`'s response variant matches the request type registered for the given `callback_id`. The `SendTransactionResponse` branch in `push_response` also performs no lookup against `bitcoin_send_transaction_internal_contexts`, unlike the `GetSuccessorsResponse` branch which guards with a `.ok_or_else(|| StateError::BitcoinNonMatchingResponse {...})?` check. A single Byzantine block proposer below the consensus fault threshold can craft a payload pairing `SendTransactionResponse` with a `GetSuccessors` callback ID; every honest replica's `validate_payload` accepts it, the block finalizes, and `push_response` enqueues `Payload::Data(EmptyBlob)` for the `GetSuccessors` callback — corrupting the bitcoin canister's chain-sync state and stalling ckBTC operations.

## Finding Description

**Root cause 1 — `validate_self_validating_payload_impl` is a size-only check.**

`validate_self_validating_payload_impl` short-circuits on an empty payload and otherwise returns only the byte count: [1](#0-0) 

`validate_payload` calls this function and adds only a size-limit check; it never reads the replicated state to verify that each response's variant matches the request type registered for its `callback_id`: [2](#0-1) 

**Root cause 2 — `SendTransactionResponse` branch in `push_response` has no context guard.**

The `GetSuccessorsResponse` branch guards with a mandatory lookup: [3](#0-2) 

The `SendTransactionResponse` branch performs no equivalent lookup against `bitcoin_send_transaction_internal_contexts`. It unconditionally constructs `EmptyBlob` and pushes it to the consensus queue for whatever `callback_id` was supplied: [4](#0-3) 

**Exploit flow.**

1. Byzantine proposer identifies a live `GetSuccessors` callback ID `G` from the certified state (publicly visible).
2. Proposer crafts `BitcoinAdapterResponse { response: SendTransactionResponse({}), callback_id: G }` and includes it in a `SelfValidatingPayload`.
3. All honest replicas call `validate_payload` → `validate_self_validating_payload_impl`; the payload passes (only size is checked).
4. Block is notarized and finalized.
5. During execution, `push_response` is called; it enters the `SendTransactionResponse` arm, skips any context lookup, and enqueues `ConsensusResponse::new(G, Payload::Data(EmptyBlob.encode()))`.
6. The bitcoin canister's `GetSuccessors` callback for ID `G` receives `EmptyBlob` instead of block data. The `bitcoin_get_successors_contexts` entry for `G` is never removed (the `SendTransactionResponse` branch does not touch it), leaving a leaked context and a delivered-but-wrong response.

## Impact Explanation

The bitcoin canister's chain-sync state machine receives an unexpected `EmptyBlob` encoding for a `GetSuccessors` callback. The canister will trap, misinterpret, or stall on this response, halting UTXO tracking. Because the ckBTC minter depends on continuous chain-sync progress, this directly disrupts ckBTC minting and burning operations. This matches the allowed impact: **High — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm**, and also **High — Application/platform-level DoS or certified-state disruption**.

## Likelihood Explanation

Exploitation requires control of exactly one subnet node acting as block proposer — a single Byzantine node strictly below the consensus fault threshold (`f < n/3`). This is explicitly within the IC threat model. No threshold corruption, key compromise, or privileged access is needed. The attack is deterministic, locally reproducible, and can be repeated every time the attacker's node is selected as proposer.

## Recommendation

1. **In `validate_payload` / `validate_self_validating_payload_impl`**: Read the certified state and, for each response in the payload, verify that the `callback_id` exists in the appropriate context map (`bitcoin_get_successors_contexts` for `GetSuccessorsResponse`, `bitcoin_send_transaction_internal_contexts` for `SendTransactionResponse`) and that the response variant matches the registered request variant. Reject the payload with `InvalidArtifact` if any mismatch is found. [1](#0-0) 

2. **In `push_response` — `SendTransactionResponse` branch**: Add a lookup against `bitcoin_send_transaction_internal_contexts` mirroring the guard in the `GetSuccessorsResponse` branch, and return `StateError::BitcoinNonMatchingResponse` if the `callback_id` is absent or belongs to the wrong context type. [4](#0-3)
