### Title
Missing Caller Validation in `ft_on_transfer` Enables Unbacked ERC-20 Token Minting — (File: `engine/src/contract_methods/connector.rs`)

### Summary
The `ft_on_transfer` NEAR contract method has no caller validation: any NEAR account can invoke it directly. Combined with `deploy_erc20_token` also having no access control, an unprivileged attacker can register any NEAR account they control as a NEP-141 mirror and then call `ft_on_transfer` from that account with an arbitrary `amount`, causing Aurora to mint unbacked ERC-20 tokens without any actual NEP-141 token deposit.

### Finding Description

**Root cause 1 — `deploy_erc20_token` is unrestricted:**

`deploy_erc20_token` only checks `require_running`; there is no caller restriction. Any NEAR account can register any NEP-141 account ID as an ERC-20 mirror. [1](#0-0) 

**Root cause 2 — `ft_on_transfer` has no caller validation:**

`ft_on_transfer` only checks `require_running`. The sole branching logic is whether `predecessor_account_id` equals the stored connector account. If it does not, the code unconditionally calls `receive_erc20_tokens`, using `predecessor_account_id` as the NEP-141 token identifier and `args.amount` (fully attacker-controlled) as the mint amount. [2](#0-1) 

**Root cause 3 — `receive_erc20_tokens` mints without verifying a real transfer occurred:**

`receive_erc20_tokens` looks up the ERC-20 address for the caller's NEP-141 account and calls the ERC-20 `mint` function (as the engine admin) for whatever `args.amount` says. There is no on-chain proof that the caller actually transferred any NEP-141 tokens to Aurora. [3](#0-2) 

**Attack path:**

1. Attacker creates or controls any NEAR account, e.g. `attacker.near`.
2. Attacker calls `deploy_erc20_token("attacker.near")` on Aurora — succeeds with no access control, deploying an ERC-20 mirror at address `X`.
3. Attacker calls `ft_on_transfer` on Aurora **from** `attacker.near` with `amount = U128::MAX` and `msg = <attacker_evm_address_hex>`.
4. Aurora's `ft_on_transfer` sees `predecessor_account_id = "attacker.near"` ≠ connector → calls `receive_erc20_tokens("attacker.near", args, …)`.
5. `get_erc20_from_nep141("attacker.near")` returns address `X`; the engine mints `U128::MAX` ERC-20 tokens to the attacker's EVM address.
6. No NEP-141 tokens were ever deposited. The ERC-20 supply is entirely unbacked.

### Impact Explanation

The ERC-20 mirror for the attacker's NEP-141 is immediately insolvent: the ERC-20 supply exceeds the NEP-141 balance held by Aurora. If any other user has legitimately bridged the same NEP-141 token, they cannot exit — their ERC-20 tokens cannot be redeemed for NEP-141 tokens because Aurora's custody is insufficient. This constitutes **permanent freezing of funds** for legitimate holders. Additionally, if the attacker lists the unbacked ERC-20 tokens on an Aurora DEX, they can drain liquidity from other users — **direct theft of user funds**.

### Likelihood Explanation

Both entry points (`deploy_erc20_token` and `ft_on_transfer`) are public NEAR contract methods callable by any NEAR account with no special privileges. The attack requires only controlling a NEAR account and sending two transactions. No admin compromise, oracle error, or governance capture is needed.

### Recommendation

1. **`ft_on_transfer`**: Validate that `predecessor_account_id` is a known, registered NEP-141 token contract (i.e., one that has a corresponding ERC-20 entry in storage). Reject calls from accounts that are not registered NEP-141 mirrors. Alternatively, require that `ft_on_transfer` can only be called as a callback from a `ft_transfer_call` (use `assert_private_call` or a similar mechanism).
2. **`deploy_erc20_token`**: Add `require_owner_only` (or a dedicated allowlist) so that only authorized parties can register new NEP-141 ↔ ERC-20 mappings.

### Proof of Concept

```
# Step 1: Register attacker's account as a NEP-141 mirror (no access control)
near call aurora deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# Step 2: Mint unbacked ERC-20 tokens by calling ft_on_transfer directly

  --accountId attacker.near

# Result: attacker's EVM address now holds U128::MAX ERC-20 tokens
# with zero NEP-141 tokens ever deposited into Aurora.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
