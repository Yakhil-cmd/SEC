### Title
Asymmetric `delta_gas` Handling in `compute_gas_refund` Causes Users to Be Systematically Overcharged - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` adjustment is applied only when `delta_gas > 0` (native resource consumption exceeds EVM gas consumption), increasing `gas_used` and charging the user more. When `delta_gas < 0` (native resource consumption is less than EVM gas consumption), `gas_used` is **not** reduced, causing the user to be overcharged. The code itself acknowledges this with a `// TODO: return delta_gas to gas_used?` comment. This is a direct analog to the external report's "some expected rewards are not distributed" pattern: tokens that should be refunded to the user are instead silently transferred to the operator (or burned when `burn_base_fee` is enabled).

---

### Finding Description

In `compute_gas_refund`, after computing `gas_used` from ergs and applying the EIP-3529 refund cap and `minimal_gas_used` floor, the function computes `delta_gas` to reconcile EVM gas consumption with native resource consumption:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The handling is **asymmetric**:
- `delta_gas > 0`: `gas_used` is increased → user pays more, operator receives more.
- `delta_gas < 0`: `gas_used` is **not** decreased → user is overcharged by `|delta_gas| * gas_price` tokens.

The `delta_gas < 0` case occurs whenever `native_used / native_per_gas < gas_used`, i.e., when the native resource consumption is proportionally less than the EVM gas consumption. This is a normal scenario for transactions that use EVM opcodes with high EVM gas costs but low native (RISC-V proving) costs (e.g., warm SLOAD, simple arithmetic-heavy loops). Additionally, integer division in `native_used / native_per_gas` introduces a systematic rounding-down bias, making `delta_gas` slightly negative even in borderline cases.

The downstream effect is in `refund_and_commit_fee` (ZK flow):

```rust
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
``` [2](#0-1) 

```rust
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
    .ok_or(internal_error!("gu*gpfo"))?;
``` [3](#0-2) 

Because `gas_used` is inflated by `|delta_gas|`:
- The user's refund is reduced by `|delta_gas| * gas_price`.
- Without `burn_base_fee`: the operator receives `|delta_gas| * gas_price` extra tokens the user should have gotten back.
- With `burn_base_fee` enabled: `|delta_gas| * base_fee` tokens are permanently burned instead of being refunded to the user, since the operator only receives `gas_used * (gas_price - base_fee)`. [4](#0-3) 

The same `compute_gas_refund` function is used for both L2 ZK transactions and L1→L2 transactions: [5](#0-4) 

The documentation in `docs/double_resource_accounting.md` only describes the `delta_gas > 0` case and is silent on the negative case, consistent with it being an unintentional omission: [6](#0-5) 

---

### Impact Explanation

Every transaction where native resource consumption is proportionally less than EVM gas consumption results in the user being overcharged. The overcharge amount is `|delta_gas| * gas_price` tokens per transaction. With `burn_base_fee` enabled, `|delta_gas| * base_fee` tokens are permanently removed from circulation rather than returned to the user. Without `burn_base_fee`, the operator receives unearned tokens. In both cases, the user receives fewer tokens back than the actual resource consumption warrants.

---

### Likelihood Explanation

High. The `delta_gas < 0` condition is triggered by any transaction where EVM gas consumption exceeds what native resource consumption implies. This is a common scenario: warm storage reads, simple arithmetic, and many other EVM operations have high EVM gas costs relative to their RISC-V proving cost. Additionally, the integer division `native_used / native_per_gas` introduces a systematic rounding-down bias, making `delta_gas` slightly negative even when the true ratio is exactly 1:1. No special attacker action is required — any unprivileged transaction sender triggers this path through normal transaction submission.

---

### Recommendation

Apply the `delta_gas` correction symmetrically. When `delta_gas < 0`, reduce `gas_used` (subject to the `minimal_gas_used` floor):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption is less than gas consumption; return the difference
    // to the user as a larger refund, but never go below minimal_gas_used.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This ensures the user is charged only for the resources actually consumed, symmetric with the upward adjustment already applied for the `delta_gas > 0` case.

---

### Proof of Concept

Consider a ZK L2 transaction with:
- `gas_limit = 100_000`
- `gas_price = 1_000` (wei)
- `native_per_gas = 10`
- After execution: `gas_used_from_ergs = 50_000`, `evm_refund = 0`, `minimal_gas_used = 21_000`
- `gas_used` after floor: `max(50_000, 21_000) = 50_000`
- `native_used = 400_000` (native units consumed)
- `delta_gas = (400_000 / 10) - 50_000 = 40_000 - 50_000 = -10_000`

Current behavior (bug):
- `delta_gas < 0` → no adjustment → `gas_used = 50_000`
- User refund: `(100_000 - 50_000) * 1_000 = 50_000_000` tokens
- Operator fee: `50_000 * 1_000 = 50_000_000` tokens

Correct behavior:
- `delta_gas = -10_000` → `gas_used = max(50_000 - 10_000, 21_000) = 40_000`
- User refund: `(100_000 - 40_000) * 1_000 = 60_000_000` tokens
- Operator fee: `40_000 * 1_000 = 40_000_000` tokens

The user is overcharged by `10_000 * 1_000 = 10_000_000` tokens per transaction in this scenario. With `burn_base_fee`, a portion of this overcharge is permanently burned rather than going to the operator. [1](#0-0)
