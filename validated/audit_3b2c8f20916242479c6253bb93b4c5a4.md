### Title
SNS Governance Allows Concurrent Identical `ExecuteGenericNervousSystemFunction` Proposals Without Deduplication Guard - (File: `rs/sns/governance/src/governance.rs`)

### Summary
SNS governance enforces explicit concurrency guards for upgrade-class proposals (`UpgradeSnsControlledCanister`, `UpgradeSnsToNextVersion`, `ManageLedgerParameters`) via `check_no_upgrades_in_progress`, but applies **no equivalent guard** for `ExecuteGenericNervousSystemFunction` proposals. Two proposals with identical `function_id` and `payload` can be submitted, voted through, and both executed against the same external target canister. SNS developers building governance-callable functions are not warned of this behavior and may not design their functions to be idempotent, leading to unintended double-execution.

### Finding Description

SNS governance's `perform_action` dispatches adopted proposals to their respective handlers. For upgrade-type actions, `check_no_upgrades_in_progress` is called before execution, blocking a second concurrent upgrade from proceeding: [1](#0-0) 

For `ExecuteGenericNervousSystemFunction`, no analogous guard exists. The validation path (`validate_and_render_execute_nervous_system_function`) only checks that the referenced `NervousSystemFunction` exists and calls the validator canister — it does not check whether an open or adopted proposal with the same `function_id` and `payload` is already in flight: [2](#0-1) 

The `make_proposal` path in SNS governance similarly has no deduplication check for `ExecuteGenericNervousSystemFunction`: [3](#0-2) 

By contrast, NNS governance applies `select_nonfinal_proposal_ids` to block concurrent `CreateServiceNervousSystem` and `SubnetRentalRequest` proposals: [4](#0-3) [5](#0-4) 

The inconsistency is structural: upgrade-class SNS proposals are protected; generic external-call proposals are not.

### Impact Explanation

An SNS's governance-callable external function (registered via `AddGenericNervousSystemFunction`) may be non-idempotent — for example, it may transfer tokens, increment a counter, or mutate state in a way that is harmful if applied twice. Because two proposals with identical `function_id` + `payload` can both be adopted and executed, the target canister method is called twice. If the method transfers SNS treasury funds or mints tokens, the double-execution drains twice the intended amount. The SNS governance framework provides no mechanism to prevent this, and the `NervousSystemFunction` registration interface does not document this constraint. [6](#0-5) 

### Likelihood Explanation

SNS neurons can follow other neurons. When a followed neuron votes Yes on proposal A, all followers automatically cast Yes on proposal A. If an attacker submits an identical proposal B during the same voting window, the same following relationships cause followers to automatically vote Yes on B as well — without any human review of the duplicate. The attacker only needs to hold enough SNS stake to submit a proposal (the `reject_cost_e8s` threshold). No governance majority needs to be malicious; the liquid-democracy following mechanism is sufficient to pass both proposals. [7](#0-6) 

### Recommendation

1. Add a concurrency guard in SNS governance's proposal validation for `ExecuteGenericNervousSystemFunction` that checks for any open or adopted proposal with the same `function_id` (and optionally the same `payload` hash), analogous to `check_no_upgrades_in_progress`.
2. Document in the `NervousSystemFunction` registration interface and SNS developer documentation that governance-callable functions may be invoked multiple times concurrently via duplicate proposals, and that functions must be designed to be idempotent or protected by their own deduplication logic.
3. Consider extending `upgrade_proposals_in_progress` to a more general `proposals_in_progress_by_action` map so that any action type can opt into single-inflight semantics.

### Proof of Concept

1. SNS is deployed with a registered `NervousSystemFunction` (id=1000) whose `target_method` transfers 500 SNS tokens to a specified account.
2. Attacker (holding enough stake) submits **Proposal A**: `ExecuteGenericNervousSystemFunction { function_id: 1000, payload: <transfer 500 tokens to attacker> }`.
3. Legitimate voters (and their followers) vote Yes; Proposal A reaches majority and enters `Adopted` status.
4. Before Proposal A executes, attacker submits **Proposal B** with identical `function_id` and `payload`.
5. The same following relationships cause followers to automatically vote Yes on Proposal B.
6. Both proposals are adopted and `perform_action` is called twice for the same `function_id`/`payload`.
7. The target canister's transfer method executes twice, sending 1000 tokens instead of 500. [8](#0-7) [9](#0-8)

        let result = match action {
            // Execution of Motion proposals is trivial.
            Action::Motion(_) => Ok(()),

            Action::ManageNervousSystemParameters(params) => {
                self.perform_manage_nervous_system_parameters(params)
            }
            Action::UpgradeSnsControlledCanister(params) => {
                self.perform_upgrade_sns_controlled_canister(proposal_id, params)
                    .await
            }
            Action::UpgradeSnsToNextVersion(_) => {
                log!(INFO, "Executing UpgradeSnsToNextVersion action",);
                let upgrade_sns_result = self
                    .perform_upgrade_to_next_sns_version_legacy(proposal_id)
                    .await;

                // If the upgrade returned `Ok(true)` that means the upgrade completed successfully
                // and the proposal can be marked as "executed". If the upgrade returned `Ok(false)`
                // that means the upgrade has successfully been kicked-off asynchronously, but not
                // completed. Governance's run_periodic_tasks logic will continuously check
                // the status of the upgrade and mark the proposal as either executed or failed.
                // So we call `return` in the `Ok(false)` branch so that
                // `set_proposal_execution_status` doesn't get called and set the proposal status
                // prematurely. If the result is `Err`, we do want to set the proposal status,
                // and passing the value through is sufficient.
                match upgrade_sns_result {
                    Ok(true) => Ok(()),
                    Ok(false) => return,
                    Err(e) => Err(e),
                }
            }
            Action::ExecuteGenericNervousSystemFunction(call) => {
                self.perform_execute_generic_nervous_system_function(call)
                    .await
            }

    /// Used for checking that no upgrades are in progress. Also checks that there are no upgrade proposals in progress except, optionally, one that you pass in as `proposal_id`
    pub fn check_no_upgrades_in_progress(
        &self,
        proposal_id: Option<u64>,
    ) -> Result<(), GovernanceError> {
        let upgrade_proposals_in_progress = self.upgrade_proposals_in_progress();
        if !upgrade_proposals_in_progress.is_subset(&proposal_id.into_iter().collect()) {
            return Err(GovernanceError::new_with_message(
                ErrorType::ResourceExhausted,
                format!(
                    "Another upgrade is currently in progress (proposal IDs {}). \
                    Please, try again later.",
                    upgrade_proposals_in_progress
                        .into_iter()
                        .map(|id| id.to_string())
                        .collect::<Vec<String>>()
                        .join(", ")
                ),
            ));
        }

        if self.proto.pending_version.is_some() {
            return Err(GovernanceError::new_with_message(
                ErrorType::ResourceExhausted,
                format!(
                    "Upgrade lock acquired (expires at {:?}), not upgrading",
                    self.proto
                        .pending_version
                        .as_ref()
                        .map(|p| p.mark_failed_at_seconds)
                ),
            ));
        }

        Ok(())

/// Validates and renders a proposal with action ExecuteNervousSystemFunction.
/// This retrieves the nervous system function's validator method and calls it.
pub async fn validate_and_render_execute_nervous_system_function(
    env: &dyn Environment,
    execute: &ExecuteGenericNervousSystemFunction,
    existing_functions: &BTreeMap<u64, NervousSystemFunction>,
) -> Result<String, String> {
    let id = execute.function_id;
    match existing_functions.get(&execute.function_id) {
        None => Err(format!("There is no NervousSystemFunction with id: {id}")),
        Some(function) => {
            // Make sure this isn't a NervousSystemFunction which has been deleted.
            if function == &*NERVOUS_SYSTEM_FUNCTION_DELETION_MARKER {
                Err(format!("There is no NervousSystemFunction with id: {id}"))
            } else {
                // To validate the proposal we try and call the validation method,
                // which should produce a payload rendering if the proposal is valid
                // or an error if it isn't.
                let rendering =
                    perform_execute_generic_nervous_system_function_validate_and_render_call(
                        env,
                        function.clone(),
                        execute.clone(),
                    )
                    .await?;

                let payload_hash = {
                    let mut state = Sha256::new();
                    state.write(execute.payload.as_slice());
                    let sha = state.finish();
                    hex::encode(sha)
                };

                Ok(format!(
                    r"# Proposal to execute nervous system function:

## Nervous system function:

{function:#?}

## Payload sha256: 

{payload_hash}

## Payload:

{rendering}"
                ))
            }
        }
    }
