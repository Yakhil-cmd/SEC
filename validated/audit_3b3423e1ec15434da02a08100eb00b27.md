Audit Report

## Title
Unauthorized Third-Party Can Force SNS Swap Participation on Behalf of Any Buyer, Bypassing Confirmation Text Consent - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` endpoint resolves the buyer principal from the request argument rather than from the authenticated caller, and passes the caller-supplied `confirmation_text` directly to validation without verifying that the caller is the buyer. Because the `confirmation_text` is publicly readable from the swap canister's `get_init` query, any unprivileged third party can register swap participation on behalf of a victim who has already transferred ICP to their swap subaccount, recording the victim as having accepted legal/compliance terms they never explicitly agreed to and locking their ICP in the swap.

## Finding Description
In `rs/sns/swap/canister/canister.rs` (L127–143), `refresh_buyer_tokens` resolves the buyer from `arg.buyer` without any `caller == buyer` check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
};
swap_mut()
    .refresh_buyer_token_e8s(p, arg.confirmation_text, ...)
    .await
```

The `confirmation_text` from the request is passed directly to `refresh_buyer_token_e8s`, which calls `validate_confirmation_text` in `rs/sns/swap/src/swap.rs` (L363–384). That function performs only a string equality check against the stored text — it does not verify that the caller is the buyer:

```rust
(Some(expected_text), Some(text)) => {
    if &text != expected_text { Err(...) } else { Ok(()) }
}
```

The stored `confirmation_text` is set at SNS initialization and is publicly readable via the `get_init` query endpoint (it is part of the `Init` struct returned in `GetInitResponse`). The proto definition at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` (L843–851) explicitly documents the `buyer` field as optional ("If not specified, the caller is used"), confirming third-party calls were anticipated but the consent bypass was not addressed.

The exploit path is:
1. SNS swap is initialized with `confirmation_text = "I confirm I am not a US person"`.
2. Alice transfers ICP to her swap subaccount (`principal_to_subaccount(alice)` under the swap canister).
3. Alice has not yet called `refresh_buyer_tokens` — she is still deliberating.
4. Bob queries `get_init` and reads the public `confirmation_text`.
5. Bob calls `refresh_buyer_tokens({buyer: alice.to_string(), confirmation_text: Some("I confirm I am not a US person")})` from his own identity.
6. The call succeeds: Alice is registered as a participant with her ICP locked, having "accepted" the confirmation text without her knowledge.

The integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` (L919–926) directly demonstrates that `default_sns_agent` (a different identity) can successfully call `refresh_buyer_tokens` specifying `wealthy_user_identity.principal_id` as the buyer, confirming the third-party call path is reachable and succeeds.

## Impact Explanation
This is a **High** severity finding. A victim who has transferred ICP to their swap subaccount (a realistic preparatory step) can have their ICP locked in the swap and, if the swap commits, permanently converted to SNS tokens without their explicit consent. Additionally, the victim is on-chain recorded as having accepted legal/compliance terms (the `confirmation_text`) they never agreed to. This constitutes concrete user financial harm within the SNS framework, matching the allowed impact: *"Significant SNS or infrastructure security impact with concrete user or protocol harm"* ($2,000–$10,000).

## Likelihood Explanation
- No special privileges are required; any unprivileged ingress sender can call `refresh_buyer_tokens` with an arbitrary `buyer` principal.
- The `confirmation_text` is publicly readable from `get_init` — no secret knowledge is needed.
- The only precondition is that the victim has already transferred ICP to their swap subaccount, which is a realistic and common scenario (e.g., a user who transferred funds but is still deliberating, or who transferred accidentally).
- The attack is repeatable and cheap: a single canister update call suffices.

## Recommendation
In `rs/sns/swap/canister/canister.rs`, when `arg.buyer` is non-empty and differs from `caller_principal_id()`, either:

1. **Reject the call outright** — require `caller == buyer` when `arg.buyer` is specified. This is the simplest and safest fix.
2. **Accept the call but ignore `confirmation_text`** — only allow `confirmation_text` to be accepted when `caller == buyer`; for third-party notification calls, treat `confirmation_text` as `None` regardless of what was supplied.

Option 1 is preferred. If third-party notification (allowing anyone to notify the swap of a buyer's transfer) is a desired feature, the `confirmation_text` must never be accepted from a non-buyer caller.

## Proof of Concept
A minimal PocketIC or local replica test:
1. Initialize a swap canister with `confirmation_text = "I confirm I am not a US person"`.
2. As Alice, transfer 100 ICP to `Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(alice)) }` on the ICP ledger.
3. As Bob (a different identity), query `get_init` on the swap canister and read `confirmation_text`.
4. As Bob, call `refresh_buyer_tokens(RefreshBuyerTokensRequest { buyer: alice.to_string(), confirmation_text: Some("I confirm I am not a US person") })`.
5. Assert the call returns `Ok(...)` and that Alice appears in the swap's `buyers` map with her ICP registered.
6. Assert Alice never called `refresh_buyer_tokens` herself.

This directly mirrors the pattern already demonstrated in `rs/tests/nns/sns/lib/src/sns_deployment.rs` (L919–926) for the no-confirmation-text case, extended to the confirmation-text case.
