### Title
Unbounded Loop in `Hashchain::move_to_block` Triggered on Every Hashchain-Enabled Transaction - (File: engine-hashchain/src/hashchain.rs)

### Summary
The `Hashchain::move_to_block` function contains an unbounded `while` loop that iterates once per NEAR block height skipped since the last Aurora Engine transaction. If the engine is idle for a large number of NEAR blocks and then receives any transaction, the loop executes `(current_block_height - stored_block_height)` iterations of keccak hashing, each consuming NEAR gas. A sufficiently large gap causes the NEAR gas limit to be exceeded, permanently preventing any new transaction from being processed until the hashchain is reset via `start_hashchain` (which itself requires the contract to be paused by the key manager).

### Finding Description

Every hashchain-enabled contract method (e.g. `submit`, `call`, `deploy_erc20_token`, `ft_on_transfer`, etc.) calls `with_hashchain` or `with_logs_hashchain` in `engine/src/hashchain.rs`, which calls `load_hashchain`:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;   // <-- unbounded loop
    }
    Ok(maybe_hashchain)
}
```

`move_to_block` in `engine-hashchain/src/hashchain.rs` iterates one step at a time:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
```

Each iteration performs a `keccak` hash over a ~100-byte buffer. NEAR's gas limit is 300 Tgas per transaction. If the engine has been idle for N blocks, this loop runs N times. At Aurora's typical NEAR gas cost per keccak call, a gap of a few hundred thousand blocks is sufficient to exceed 300 Tgas.

NEAR produces approximately 1 block per second. A gap of ~300,000 blocks (~3.5 days of engine inactivity) would cause every subsequent transaction to exceed the NEAR gas limit, permanently bricking the engine until an admin intervenes.

### Impact Explanation

**Permanent freezing of funds.** Once the block gap is large enough, every call to any hashchain-enabled method (including `submit`, `call`, `ft_on_transfer`, `withdraw`) will exceed the NEAR gas limit and be rejected. All ETH and ERC-20 tokens held inside the Aurora Engine become inaccessible. The only recovery path is `start_hashchain`, which requires the contract to first be paused by the key manager — itself a privileged, multi-step operation that may not be immediately available.

### Likelihood Explanation

Aurora Engine is a production contract. Periods of low activity (weekends, market downturns, planned maintenance) can easily produce multi-day gaps in Aurora transactions. The NEAR block time is ~1 second, so a 3–4 day gap produces ~260,000–345,000 blocks. This is a realistic, naturally occurring condition requiring no attacker action. Any user submitting the first transaction after such a gap triggers the condition. The attacker does not need to do anything except wait and then submit a normal transaction.

### Recommendation

Replace the per-block iteration with a direct jump: instead of iterating one block at a time, compute the hashchain for the gap in O(1) by treating all skipped blocks as empty (no transactions) and computing the final hash directly. If sequential hashing is required by the protocol, add a maximum-gap guard that returns an error or caps the number of iterations per call, and require the admin to advance the hashchain in bounded batches via a dedicated method before normal operation resumes.

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled (`start_hashchain` called).
2. Submit no transactions for N NEAR blocks (e.g. N = 400,000, ~4.6 days).
3. Any user submits a normal EVM transaction via `submit(rlp_bytes)`.
4. `with_logs_hashchain` → `load_hashchain` → `hashchain.move_to_block(current_height)` enters the while loop and iterates 400,000 times.
5. Each iteration calls `compute_block_hashchain` which calls `keccak` on a ~100-byte buffer.
6. The NEAR transaction exceeds 300 Tgas and is rejected.
7. All subsequent transactions are also rejected. Funds are frozen.

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}
