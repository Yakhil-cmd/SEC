### Title
Missing wNEAR Refund When XCC Promise Chain Fails Causes Permanent Fund Freeze — (`engine-precompiles/src/xcc.rs`, `engine/src/xcc.rs`, `engine/src/contract_methods/xcc.rs`)

---

### Summary

When a user initiates a Cross-Contract Call (XCC) that requires NEAR (for storage staking of a new router or for the call's attached balance), the XCC precompile irrevocably transfers wNEAR from the user's EVM address to the engine's implicit EVM address inside the same EVM transaction. If the subsequent NEAR promise chain fails — for example, because the router deployment batch is rejected — the `withdraw_wnear_to_router` callback aborts without touching the wNEAR, leaving it permanently locked in the engine's implicit address with no recovery path.

---

### Finding Description

**Step 1 — wNEAR is transferred inside the EVM transaction (committed unconditionally)**

In `engine-precompiles/src/xcc.rs`, `run_with_handle` computes `required_near` and, when it is non-zero, executes a sub-call to the wNEAR ERC-20 contract:

```rust
let required_near =
    match state::get_code_version_of_address(&self.io, &Address::new(sender)) {
        None => attached_near + state::STORAGE_AMOUNT,   // 2 NEAR + user's attached NEAR
        Some(_) => attached_near,
    };
if required_near != ZERO_YOCTO {
    let engine_implicit_address = aurora_engine_sdk::types::near_account_to_evm_address(
        self.engine_account_id.as_bytes(),
    );
    let tx_data = transfer_from_args(
        sender.0.into(),
        engine_implicit_address.raw().0.into(),
        required_near.as_u128().into(),
    );
    // EVM sub-call: wNEAR.transferFrom(user, engine_implicit, required_near)
    let (exit_reason, return_value) =
        handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
    // ... only reverts if the sub-call itself fails
}
``` [1](#0-0) 

If the sub-call succeeds, the EVM state change is committed as part of the outer `submit` transaction. The wNEAR is now held by the engine's implicit address.

**Step 2 — The NEAR promise chain is built with `withdraw_wnear_to_router` as a dependent callback**

In `engine/src/xcc.rs`, `handle_precompile_promise` chains the promises:

```
[router deploy batch]
    → factory_update_address_version (callback)
        → withdraw_wnear_to_router (callback)
            → execute / schedule on router (callback)
``` [2](#0-1) 

**Step 3 — `withdraw_wnear_to_router` aborts when the prior promise failed, leaving wNEAR stranded**

In `engine/src/contract_methods/xcc.rs`:

```rust
if matches!(handler.promise_result_check(), Some(false)) {
    return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
}
``` [3](#0-2) 

When the router deployment batch fails (e.g., due to insufficient gas, or the NEAR runtime rejecting the batch), `factory_update_address_version` returns an error. `withdraw_wnear_to_router` is then called as a callback of that failed promise, immediately returns `ERR_CALLBACK_OF_FAILED_PROMISE`, and never calls `withdrawToNear` on the wNEAR contract. The wNEAR transferred in Step 1 remains in the engine's implicit EVM address.

**Step 4 — No recovery path exists**

The engine's implicit address is derived deterministically from the engine's NEAR account ID and has no associated private key:

```rust
let engine_implicit_address = aurora_engine_sdk::types::near_account_to_evm_address(
    self.engine_account_id.as_bytes(),
);
``` [4](#0-3) 

`withdraw_wnear_to_router` is gated by `env.assert_private_call()`, so it cannot be invoked directly by users. [5](#0-4) 

There is no public engine method to transfer EVM tokens out of the engine's implicit address on behalf of a user. The wNEAR is permanently frozen.

---

### Impact Explanation

Every failed first-time XCC call (where the router does not yet exist) causes the user to lose `STORAGE_AMOUNT` (2 NEAR) plus any `attached_near` they specified. `STORAGE_AMOUNT` is defined as 2 × 10²⁴ yoctoNEAR:

```rust
pub const STORAGE_AMOUNT: Yocto = Yocto::new(2_000_000_000_000_000_000_000_000);
``` [6](#0-5) 

These funds are permanently frozen in the engine's implicit EVM address. This matches the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The failure condition is a failed router deployment batch. This is reachable by any unprivileged user who:

1. Makes their first XCC call (router does not yet exist, so `required_near = attached_near + STORAGE_AMOUNT > 0`).
2. Attaches insufficient NEAR gas to the outer NEAR transaction, causing the deployment batch to run out of gas.
3. Or encounters any other NEAR runtime rejection of the batch.

Gas estimation for XCC is non-trivial and users can easily under-provision it. The condition is realistic and externally triggerable.

---

### Recommendation

Add a fallback/refund callback to the promise chain that fires when `withdraw_wnear_to_router` fails. This callback should transfer the wNEAR back from the engine's implicit address to the original sender's EVM address. Concretely:

1. After attaching `withdraw_wnear_to_router` as a callback, attach an additional error-handling callback (analogous to `exit_to_near_precompile_callback`'s refund branch) that calls `engine.call_with_args` to execute `wNEAR.transfer(original_sender, required_near)` from the engine's implicit address.
2. Store the original sender address and `required_near` amount in the promise args so the refund callback can reconstruct the transfer.

This mirrors the pattern already used in `exit_to_near_precompile_callback`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [7](#0-6) 

---

### Proof of Concept

1. User `U` has wNEAR balance and has never made an XCC call (no router deployed).
2. `U` calls the XCC precompile with `CrossContractCallArgs::Eager(call)` where `call.total_near() = 0` but `required_near = STORAGE_AMOUNT = 2e24 yoctoNEAR` (because no router exists).
3. The precompile executes `wNEAR.transferFrom(U, engine_implicit, 2e24)` — this succeeds and is committed.
4. The NEAR promise chain is created: `[deploy_router_batch] → [factory_update_address_version] → [withdraw_wnear_to_router] → [execute]`.
5. `U` (or the relayer) attaches insufficient gas to the NEAR transaction; the `deploy_router_batch` runs out of gas and fails.
6. `factory_update_address_version` is called with a failed promise result → returns `ERR_ROUTER_DEPLOY_FAILED` → does not update version.
7. `withdraw_wnear_to_router` is called with a failed promise result → returns `ERR_CALLBACK_OF_FAILED_PROMISE` → does not call `withdrawToNear`.
8. `execute` is never called.
9. `U`'s 2 NEAR worth of wNEAR is now in the engine's implicit EVM address with no recovery path. The engine's version record for `U`'s address is still `None`, so the next XCC attempt will repeat the same failure and lose another 2 NEAR.
