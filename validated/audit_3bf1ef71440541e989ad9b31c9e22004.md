### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Mirror Over-Minting and Protocol Insolvency - (`engine/src/engine.rs`)

---

### Summary

When a NEP-141 token with a transfer fee is bridged into Aurora via `ft_transfer_call`, the `ft_on_transfer` callback receives the **nominal** amount sent by the caller, not the **actual** amount Aurora's account received after the fee deduction. Aurora's `receive_erc20_tokens` function blindly mints ERC-20 tokens equal to the nominal amount, creating more ERC-20 supply than there is NEP-141 backing. This makes the ERC-20 mirror permanently undercollateralized, causing insolvency for the last users attempting to exit.

---

### Finding Description

The NEP-141 `ft_transfer_call` standard guarantees that `ft_on_transfer` is called with the `amount` argument equal to what the **caller specified**, not what the receiver's balance actually increased by. For a fee-on-transfer NEP-141 token, the token contract deducts a fee during the transfer, so Aurora's actual NEP-141 balance increases by `amount - fee`, but `ft_on_transfer` is invoked with `amount`.

In `engine/src/contract_methods/connector.rs`, `ft_on_transfer` dispatches to `engine.receive_erc20_tokens` passing the raw `args` struct: [1](#0-0) 

In `engine/src/engine.rs`, `receive_erc20_tokens` extracts the amount directly from the callback argument without any balance-before/balance-after verification: [2](#0-1) 

That `amount` is then passed verbatim into `setup_receive_erc20_tokens_input`, which encodes a `mint(recipient, amount)` call to the ERC-20 contract: [3](#0-2) 

`setup_receive_erc20_tokens_input` encodes the ERC-20 `mint` selector with the full nominal amount: [4](#0-3) 

The ERC-20 contract's `mint` function (called only by the admin, i.e., Aurora itself) unconditionally mints the specified amount: [5](#0-4) 

There is no check anywhere in this path that compares Aurora's actual NEP-141 balance change against the minted ERC-20 amount.

---

### Impact Explanation

**Critical — Insolvency.**

Each deposit of a fee-on-transfer NEP-141 token mints `fee` more ERC-20 tokens than Aurora holds in NEP-141 backing. After `N` deposits of `amount` each with fee `f`:

- ERC-20 total supply: `N * amount`
- Aurora's NEP-141 balance: `N * (amount - f)`
- Shortfall: `N * f`

When users exit (burn ERC-20 → receive NEP-141 via `ft_transfer`), the last `N * f / (amount - f)` worth of ERC-20 holders cannot redeem. Their ERC-20 tokens are permanently stranded because Aurora's NEP-141 balance is exhausted before all ERC-20 supply is burned. This is a direct, permanent loss of user funds.

---

### Likelihood Explanation

Any NEAR account can deploy a fee-on-transfer NEP-141 token and call `deploy_erc20_token` on Aurora to register it. Once registered, any user calling `ft_transfer_call` on that token with Aurora as receiver triggers the bug. No admin compromise or special privilege is required — the attacker-controlled entry path is the standard NEP-141 bridge deposit flow available to any token holder. [6](#0-5) 

---

### Recommendation

In `receive_erc20_tokens`, record Aurora's NEP-141 balance for the incoming token **before** the transfer is initiated (or use the balance-delta pattern). Since `ft_on_transfer` is a callback from the NEP-141 contract itself, the actual received amount can be computed as:

```
actual_amount = current_nep141_balance_of(aurora) - pre_transfer_balance
```

Mint ERC-20 tokens equal to `actual_amount` rather than `args.amount`. Alternatively, document that fee-on-transfer NEP-141 tokens are explicitly unsupported and add a registry-level guard that prevents such tokens from being registered via `deploy_erc20_token`.

---

### Proof of Concept

1. Deploy a NEP-141 token on NEAR that deducts a 1% fee on every transfer (i.e., a transfer of `1000` results in the receiver getting `990`).
2. Call `deploy_erc20_token` on Aurora to register the NEP-141 and create its ERC-20 mirror.
3. Call `ft_transfer_call` on the NEP-141 with `receiver_id = aurora`, `amount = 1000`, `msg = <evm_address_hex>`.
4. The NEP-141 contract transfers `990` tokens to Aurora and calls `ft_on_transfer(sender, "1000", msg)`.
5. Aurora's `receive_erc20_tokens` reads `args.amount = 1000` and mints `1000` ERC-20 tokens to the EVM address.
6. Aurora's actual NEP-141 balance increased by only `990`.
7. Repeat 100 times: ERC-20 supply = `100,000`; Aurora's NEP-141 balance = `99,000`.
8. The first 99 users to call `withdrawToNear(recipient, 1000)` (burning ERC-20 and triggering `ft_transfer`) succeed. The 100th user's `ft_transfer` call fails because Aurora's NEP-141 balance is `0` — their `1000` ERC-20 tokens are burned but no NEP-141 is received, resulting in permanent fund loss. pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
