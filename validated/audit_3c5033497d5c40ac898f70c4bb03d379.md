### Title
Missing Recipient Address Validation in `receive_base_tokens` Allows Permanent ETH Freeze - (File: engine/src/engine.rs)

### Summary
The `receive_base_tokens` function in `engine/src/engine.rs` accepts any EVM address as the recipient when minting bridged ETH, including the engine's own EVM address. A user bridging ETH from NEAR to Aurora via `ft_transfer_call` can specify the engine's own EVM address as the recipient, permanently freezing the ETH with no recovery path.

### Finding Description
When a user calls `ft_transfer_call` on the ETH connector contract with `receiver_id` set to the Aurora engine's NEAR account ID, the ETH connector invokes `ft_on_transfer` on the engine. The engine dispatches to `receive_base_tokens` when the predecessor is the registered ETH connector: [1](#0-0) 

`receive_base_tokens` parses the `msg` field of the transfer args to obtain the recipient EVM address and directly credits the balance with no validation: [2](#0-1) 

The `FtTransferMessageData::try_from` parser accepts any 20-byte EVM address, including the zero address and the engine's own EVM address, without any rejection: [3](#0-2) 

The engine's own EVM address is deterministically derived from its NEAR account ID: [4](#0-3) 

This same address is used as the `erc20_admin_address` for internal ERC-20 minting operations: [5](#0-4) 

There is no private key for this address and no contract method in the engine that allows spending ETH from its own EVM balance. ETH credited there is permanently inaccessible.

### Impact Explanation
**Critical — Permanent freezing of funds.**

A user who specifies the engine's own EVM address (e.g., `0x4444588443C3a91288c5002483449Aba1054192b` on mainnet, the hardcoded coinbase) as the `msg` recipient when calling `ft_transfer_call` on the ETH connector will have their ETH credited to an address with no spendable key and no engine-level recovery mechanism. The ETH is permanently frozen inside the Aurora EVM state.

### Likelihood Explanation
**Medium.** The engine's EVM address is publicly known (it is the coinbase address, documented and observable on-chain). A user could accidentally supply it when constructing a bridge transaction, or a malicious actor could trick a user into doing so. The `msg` field is a raw hex string with no UI-level guard in the protocol itself.

### Recommendation
Add a validation check in `receive_base_tokens` (and analogously in `receive_erc20_tokens`) to reject any recipient that equals the engine's own EVM address or the zero address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let recipient = message_data.recipient;

    // Reject the engine's own EVM address and the zero address as recipients
    let engine_address = current_address(&self.current_account_id);
    if recipient == engine_address || recipient == Address::zero() {
        return Err(errors::ERR_INVALID_RECIPIENT);
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    ...
}
```

### Proof of Concept
1. Alice holds ETH on NEAR as NEP-141 tokens on the ETH connector contract.
2. Alice calls `ft_transfer_call` on the ETH connector:
   - `receiver_id`: `"aurora"` (the engine's NEAR account ID)
   - `amount`: `"1000000000000000000"` (1 ETH)
   - `msg`: `"4444588443c3a91288c5002483449aba1054192b"` (the engine's EVM address, derived via `near_account_to_evm_address("aurora".as_bytes())`)
3. The ETH connector invokes `ft_on_transfer` on the engine. Since the predecessor is the registered ETH connector, `receive_base_tokens` is called.
4. `receive_base_tokens` parses the `msg` field, obtains the engine's own EVM address as the recipient, and calls `set_balance` to credit 1 ETH to that address.
5. Alice's 1 ETH is now credited to the engine's EVM address. No private key exists for this address. No engine method allows spending ETH from it. The funds are permanently frozen.
