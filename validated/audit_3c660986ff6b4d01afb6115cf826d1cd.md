### Title
Unbounded Loop in `Hashchain::move_to_block` Causes NEAR Gas Exhaustion on Every Transaction Submission After Inactivity — (File: engine-hashchain/src/hashchain.rs)

---

### Summary

The `Hashchain::move_to_block` function iterates one block at a time from the stored block height to the current NEAR block height, calling `keccak` on each iteration. This function is invoked on every EVM transaction submission (`submit`, `call`, `deploy_code`, `submit_with_args`) via `load_hashchain`. When the engine is inactive for a period long enough to accumulate a large block gap, the loop exhausts NEAR's per-transaction gas limit, causing all subsequent EVM transactions to fail and temporarily (or permanently) freezing user funds.

---

### Finding Description

**Root cause — `move_to_block` unbounded loop:** [1](#0-0) 

The loop at line 75 increments `current_block_height` by one per iteration, calling `compute_block_hashchain` each time. `compute_block_hashchain` calls `keccak(&data).0` via `aurora_engine_sdk::keccak`, which in the NEAR runtime is a host function call costing approximately 1.5–2 TGas per invocation. NEAR's per-transaction gas limit is 300 TGas, so only ~150–200 iterations can complete before gas exhaustion.

**Call chain from every EVM transaction entry point:**

`load_hashchain` unconditionally calls `move_to_block` whenever the stored block height lags behind the current NEAR block height: [2](#0-1) 

`load_hashchain` is called by both `with_hashchain` and `with_logs_hashchain`: [3](#0-2) [4](#0-3) 

Every EVM transaction entry point wraps itself in `with_logs_hashchain`: [5](#0-4) 

**Admin escape path is also blocked:**

`pause_contract` — the only way to reach `start_hashchain` for a hashchain reset — also goes through `with_hashchain`: [6](#0-5) 

`start_hashchain` requires the contract to be paused: [7](#0-6) 

If `pause_contract` fails due to gas exhaustion, `start_hashchain` can never be called, and the hashchain state cannot be reset through the normal protocol path.

---

### Impact Explanation

When the hashchain is active and the NEAR block height has advanced by more than ~150 blocks since the last Aurora transaction, every call to `submit`, `submit_with_args`, `call`, or `deploy_code` panics with NEAR gas exhaustion inside `move_to_block`. All user ETH and ERC-20 balances held in the Aurora Engine become inaccessible. Because `pause_contract` shares the same code path, the standard admin recovery procedure is also blocked. Recovery requires a contract upgrade (out-of-band governance action), making this a **High — Temporary freezing of funds** (or Critical if the upgrade path is unavailable).

---

### Likelihood Explanation

- The hashchain must be initialized via `start_hashchain` for the vulnerability to be active (opt-in, but intended for production use).
- NEAR produces approximately one block per second. A gap of ~150 blocks corresponds to ~2.5 minutes of zero Aurora transaction activity — a realistic condition during low-traffic periods, maintenance windows, or network congestion.
- No attacker action is required; the condition arises from natural inactivity.
- Once triggered, every subsequent transaction fails until a contract upgrade is deployed.

---

### Recommendation

Cap the number of iterations in `move_to_block`. If `next_block_height - current_block_height` exceeds a safe bound (e.g., 1 000 blocks), either:

1. Return an error requiring an admin-initiated hashchain reset via a path that does **not** call `move_to_block` (e.g., a dedicated `reset_hashchain` that bypasses `with_hashchain`), or
2. Compute the hashchain for a large gap in O(1) by storing the gap count and hashing it directly rather than iterating block-by-block.

Additionally, `pause_contract` should be refactored to not go through `with_hashchain`, so the admin can always pause the engine regardless of hashchain state.

---

### Proof of Concept

1. Deploy Aurora Engine and initialize the hashchain via `start_hashchain`.
2. Allow ~200 NEAR blocks to pass with no Aurora transactions (~3 minutes of inactivity).
3. Submit any EVM transaction via `submit` (e.g., a simple ETH transfer).
4. Observe: the NEAR transaction fails with gas exhaustion inside `Hashchain::move_to_block` at `engine-hashchain/src/hashchain.rs:75–85`.
5. Attempt `pause_contract` as the engine owner — it also fails for the same reason (same `load_hashchain` call path).
6. `start_hashchain` cannot be called because the contract is not paused.
7. All user funds in the Aurora Engine are frozen until a contract upgrade is deployed.

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }

fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}

pub fn start_hashchain<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    let mut state = state::get_state(&io)?;
    require_paused(&state)?;
    require_key_manager_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let args = StartHashchainArgs::try_from_slice(&input).map_err(|_| errors::ERR_SERIALIZE)?;
    let block_height = env.block_height();

    // Starting hashchain must be for an earlier block
    if block_height < args.block_height {
        return Err(errors::ERR_ARGS.into());
    }

    let mut hashchain = Hashchain::new(
        state.chain_id,
        env.current_account_id(),
        args.block_height + 1,
        args.block_hashchain,
    );

    if hashchain.get_current_block_height() < block_height {
        hashchain.move_to_block(block_height)?;
    }
