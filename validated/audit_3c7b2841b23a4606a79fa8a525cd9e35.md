### Title
Base Token Permanently Locked When Sent to System Hook Addresses by Unauthorized Callers - (File: `basic_bootloader/src/bootloader/runner.rs`, `system_hooks/src/call_hooks/mint_base_token.rs`, `system_hooks/src/call_hooks/l1_messenger.rs`, `system_hooks/src/call_hooks/set_bytecode_on_address.rs`, `system_hooks/src/call_hooks/contract_deployer_temp.rs`)

---

### Summary

System hook addresses (`0x7001`, `0x7002`, `0x7100`, `0x8006`) "pretend to be empty accounts" for unauthorized callers, returning success with empty returndata. However, the EVM value transfer (`nominal_token_value`) is executed by the runner **before** the hook body runs. Any base tokens sent to these hook addresses by an unauthorized caller are permanently transferred to the hook address with no withdrawal or recovery mechanism, locking them forever.

---

### Finding Description

In ZKsync OS, the bootloader runner performs the ETH/base-token transfer from caller to callee **before** dispatching to the hook implementation. This is done in `perform_transfer_if_required` in `basic_bootloader/src/bootloader/runner.rs`: [1](#0-0) 

The transfer unconditionally moves `nominal_token_value` from `call_request.caller` to `target` (which equals `call_request.callee` for normal calls) via `system.io.transfer_nominal_token_value`. Only after this transfer completes does the runner dispatch to the hook.

Each system hook then checks the caller identity. For unauthorized callers, all four hooks "pretend to be an empty account" and return success with empty returndata — **without reversing the already-completed token transfer**:

- `mint_base_token_hook` (0x7100): [2](#0-1) 
- `l1_messenger_hook` (0x7001): [3](#0-2) 
- `set_bytecode_on_address_hook` (0x7002): [4](#0-3) 
- `contract_deployer_temp_hook` (0x8006): [5](#0-4) 

The hook addresses are permanent system addresses defined in `addresses_constants.rs`: [6](#0-5) 

These addresses are not regular EVM contracts. No bytecode can be deployed at them, and no hook function provides a withdrawal path. Once base tokens are transferred to a hook address, they are irrecoverable.

The documentation in `docs/system_hooks.md` states that unauthorized calls are treated as "calls to an empty account: success with empty returndata and **no side effects**." This claim is incorrect: the token transfer **is** a side effect that persists after the hook returns.

---

### Impact Explanation

Any user who calls a system hook address with a non-zero `msg.value` (base token amount) — whether accidentally or intentionally — permanently loses those tokens. The tokens accumulate at the hook address (e.g., `0x7001`, `0x7002`, `0x7100`, `0x8006`) with no mechanism for recovery. This is a direct, irreversible loss of user funds at the L2 level.

---

### Likelihood Explanation

The hook addresses are publicly known system addresses. Any EVM-compatible caller can send a transaction with `value > 0` to these addresses. A user interacting with ZKsync OS who mistakenly sends ETH to one of these addresses (e.g., confusing `0x8006` with a contract deployer they intend to fund) will lose their funds. The call succeeds (returns success), giving no indication that the funds are lost. Likelihood is medium: the addresses are not commonly targeted, but the silent success return makes the loss non-obvious.

---

### Recommendation

The authorization check should occur **before** the token transfer, not after. The runner should gate the transfer on caller authorization for hook addresses, or the hooks should explicitly revert (not silently succeed) when called with non-zero value by unauthorized callers, so the EVM reverts the transfer. Alternatively, the "pretend empty account" path should explicitly return an error when `nominal_token_value != 0`, consistent with how the hooks already reject non-zero value from **authorized** callers: [7](#0-6) 

The same guard (`error |= nominal_token_value != U256::ZERO`) should apply in the unauthorized-caller early-return path, causing a revert rather than a silent success, so the runner rolls back the transfer.

---

### Proof of Concept

1. Attacker (any address ≠ `0x8008`) constructs a transaction: `to = 0x7001` (L1 messenger hook), `value = 1 ETH`, any calldata.
2. Runner calls `perform_transfer_if_required`: transfers 1 ETH from attacker to `0x7001`. Transfer succeeds.
3. Runner dispatches to `l1_messenger_hook`. Hook checks `caller != L1_MESSENGER_ADDRESS` → returns `make_return_state_from_returndata_region(available_resources, &[])` (success, empty returndata).
4. Transaction completes successfully. Attacker's 1 ETH is now at `0x7001`.
5. No function exists to withdraw from `0x7001`. The 1 ETH is permanently locked.

The same sequence applies to `0x7002`, `0x7100`, and `0x8006`.
