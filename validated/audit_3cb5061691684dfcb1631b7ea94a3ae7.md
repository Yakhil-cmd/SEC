Audit Report

## Title
SNS Swap Permanently Stuck When Remaining Capacity Falls Below `min_participant_icp_e8s` With All Participants at Per-Participant Maximum - (`rs/sns/swap/src/swap.rs`)

## Summary

The `refresh_buyer_token_e8s` function in the SNS Swap canister applies a second minimum-participation check on the post-cap effective balance (line 1241), but `validate_participation_constraints` in `rs/sns/init/src/lib.rs` does not enforce that `min_participant_icp_e8s <= max_direct_participation_icp_e8s - min_direct_participation_icp_e8s`. A valid swap configuration can therefore reach a state where remaining capacity is below `min_participant_icp_e8s`, all existing participants are at `max_participant_icp_e8s`, `min_direct_participation_icp_e8s` is unreachable, and neither `can_commit` nor `can_abort` returns true until `swap_due_timestamp_seconds`. All participant ICP is locked for up to 14 days.

## Finding Description

**Root cause — missing validation constraint:**

`validate_participation_constraints` (`rs/sns/init/src/lib.rs`, lines 1530–1625) enforces only:
- `max_direct >= min_direct`
- `max_participant >= min_participant`
- `max_participant <= max_direct`
- `max_direct >= min_participants * min_participant`

It does **not** enforce `min_participant_icp_e8s <= max_direct_participation_icp_e8s - min_direct_participation_icp_e8s`, so a configuration like `min_participant=3 ICP, min_direct=9 ICP, max_direct=10 ICP, max_participant=4 ICP` passes all checks.

**Stuck state mechanics in `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs`):**

1. **Line 1202**: Raw balance check — `e8s < params.min_participant_icp_e8s` → rejects any transfer below the minimum.
2. **Line 1224**: Cap to remaining capacity — `actual_increment_e8s = min(max_increment_e8s, requested_increment_e8s)`.
3. **Line 1237**: Cap to per-participant max — `new_balance_e8s = min(new_balance_e8s, max_participant_icp_e8s)`.
4. **Line 1241**: Second minimum check — `new_balance_e8s < params.min_participant_icp_e8s` → rejects.

When `available_direct_participation_e8s() < min_participant_icp_e8s`:
- A **new participant** gets `new_balance_e8s = actual_increment_e8s < min_participant_icp_e8s` → rejected at line 1241.
- An **existing participant already at `max_participant_icp_e8s`** gets `new_balance_e8s` capped back to `max_participant_icp_e8s` at line 1237 with no net change → no progress.

**`can_commit` and `can_abort` both blocked:**

`can_commit` (`swap.rs` line 2884) requires `sufficient_participation()` (line 2796), which requires `min_direct_participation_icp_e8s_reached()` (line 2827). With current direct participation below `min_direct`, this is false.

`can_abort` (`swap.rs` line 2906) requires `(swap_due || icp_target_reached) && !sufficient_participation`. In the stuck state, `icp_target_reached` is false and `swap_due` is false until the deadline — so abort is also blocked.

**Existing test coverage gap:**

The test `test_swap_cannot_finalize_via_new_participation_if_remaining_lt_minimal_participation_amount` (line 5707, `rs/sns/swap/tests/swap.rs`) sets `max_participant_icp_e8s = 500_000 * E8` while user1 only contributes `400_000 * E8`, leaving room for user1 to top up and rescue the swap. It does not cover the case where all participants are already at `max_participant_icp_e8s`.

## Impact Explanation

All participant ICP is locked in the swap canister for up to `MAX_SALE_DURATION_SECONDS = 14 * ONE_DAY_SECONDS` (`rs/sns/swap/src/types.rs`, line 321). Funds are not permanently lost — after expiry the swap aborts and participants can reclaim via `error_refund_icp` — but the impact is a significant, concrete, bounded DoS against SNS swap participants with real financial harm (loss of liquidity for up to 14 days). This matches the allowed High impact class: **"Significant SNS security impact with concrete user or protocol harm."**

## Likelihood Explanation

The triggering configuration requires no special privileges. Any SNS creator submitting swap parameters passes all current on-chain validations with the example configuration. The stuck state can be reached organically (two early participants each filling to `max_participant_icp_e8s`) or deliberately by a single early participant who fills the swap to the critical remaining-capacity threshold. No victim mistakes, social engineering, or external compromise is required.

## Recommendation

Add the following check to `validate_participation_constraints` in `rs/sns/init/src/lib.rs` after the existing constraint checks:

```rust
if min_participant_icp_e8s > max_direct_participation_icp_e8s.saturating_sub(min_direct_participation_icp_e8s) {
    return Err(format!(
        "Error: min_participant_icp_e8s ({min_participant_icp_e8s}) must be <= \
         max_direct_participation_icp_e8s ({max_direct_participation_icp_e8s}) - \
         min_direct_participation_icp_e8s ({min_direct_participation_icp_e8s})"
    ));
}
```

This ensures there is always at least one minimum-sized contribution of headroom between `min_direct` and `max_direct`, preventing the stuck state at the configuration level.

## Proof of Concept

**Stuck configuration (all existing validations pass):**
```
min_participant_icp_e8s          = 3 ICP (300_000_000 e8s)
min_direct_participation_icp_e8s = 9 ICP (900_000_000 e8s)
max_direct_participation_icp_e8s = 10 ICP (1_000_000_000 e8s)
max_participant_icp_e8s          = 4 ICP (400_000_000 e8s)
min_participants                 = 1
```

**Validation checks that pass:**
- `max_direct (10) >= min_direct (9)` ✓
- `max_participant (4) >= min_participant (3)` ✓
- `max_participant (4) <= max_direct (10)` ✓
- `max_direct (10) >= min_participants (1) * min_participant (3)` ✓

**Attack sequence:**
1. Participant A: `refresh_buyer_tokens` with 4 ICP → accepted (total: 4 ICP)
2. Participant B: `refresh_buyer_tokens` with 4 ICP → accepted (total: 8 ICP); both A and B at `max_participant`; `available = 2 ICP`
3. New participant C tries 3 ICP: raw check passes (3 ≥ 3), but `actual_increment = min(2,3) = 2`, `new_balance = 0+2 = 2 < 3` → **rejected at line 1241**
4. New participant C tries 2 ICP: raw check fails (2 < 3) → **rejected at line 1202**
5. Participant A tries to top up 1 ICP: `new_balance = 4 + min(2,1) = 5`, capped to 4 at line 1237 → no change
6. `current_direct = 8 < min_direct = 9` → `sufficient_participation() = false` → `can_commit() = false`
7. `icp_target_reached = false`, `swap_due = false` → `can_abort() = false`
8. Swap is stuck; all ICP locked until `swap_due_timestamp_seconds` (up to 14 days)

A unit test mirroring `test_swap_cannot_finalize_via_new_participation_if_remaining_lt_minimal_participation_amount` but with all participants set to `max_participant_icp_e8s` before the remaining-capacity condition is reached would reproduce the stuck state deterministically.
