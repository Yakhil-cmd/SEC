Audit Report

## Title
DoS of `CreateServiceNervousSystem` Proposals via Global One-at-a-Time Restriction - (`File: rs/nns/governance/src/governance.rs`)

## Summary
The NNS governance `validate_create_service_nervous_system` function enforces a global one-at-a-time restriction on `CreateServiceNervousSystem` proposals keyed only on proposal type, not on proposer identity. Any NNS neuron holder meeting minimum stake and dissolve delay requirements can submit a malicious proposal to block all legitimate SNS creation proposals for the duration of each voting period, repeating the attack indefinitely at low cost.

## Finding Description
In `rs/nns/governance/src/governance.rs`, the function `validate_create_service_nervous_system` (called from `validate_proposal_action`, called from `validate_proposal`, called from `make_proposal`) performs a global uniqueness check:

```rust
// Must be unique.
let other_proposal_ids = self.select_nonfinal_proposal_ids(|action| {
    matches!(action, Action::CreateServiceNervousSystem(_))
});

#[cfg(not(feature = "test"))]
if !other_proposal_ids.is_empty() {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!(
            "There is another open CreateServiceNervousSystem proposal: {:?}",
            other_proposal_ids,
        ),
    ));
}
``` [1](#0-0) 

The helper `select_nonfinal_proposal_ids` treats both `ProposalStatus::Open` and `ProposalStatus::Adopted` proposals as blocking, meaning even an adopted-but-not-yet-executed proposal prevents new submissions. [2](#0-1) 

The check contains no reference to the proposer's `NeuronId` or `PrincipalId`. Any neuron that successfully submits a `CreateServiceNervousSystem` proposal ‚ÄĒ regardless of its content or intent ‚ÄĒ immediately blocks all other neurons from submitting the same proposal type until the blocking proposal reaches a final state (`Rejected`, `Executed`, or `Failed`).

The `make_proposal` entry point calls `validate_proposal` as its first substantive action, before any neuron-specific checks: [3](#0-2) 

The integration test explicitly confirms and tests this blocking behavior, asserting `ErrorType::PreconditionFailed` with message containing `"another open"`: [4](#0-3) 

**Exploit flow:**
1. Attacker controls a neuron with ‚Č•6 months dissolve delay and ‚Č•`reject_cost_e8s` (10 ICP) minted stake.
2. Attacker submits a `CreateServiceNervousSystem` proposal with malicious or discouraging parameters (wrong `dapp_canisters`, unfavorable token distribution, misleading description).
3. All subsequent legitimate `CreateServiceNervousSystem` proposals from any neuron are rejected with `PreconditionFailed` for the duration of the voting period (minimum 4 days).
4. After the attacker's proposal is rejected, the attacker immediately re-submits. The cycle repeats indefinitely.

There is no per-neuron cooldown, no mechanism to cancel the blocking proposal, and no way for legitimate proposers to bypass the restriction.

## Impact Explanation
This constitutes a complete, sustained DoS of the SNS creation pathway through NNS governance. No new SNS can be launched via the NNS proposal process for as long as the attacker maintains the attack. This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** The SNS framework is explicitly in-scope as a core NNS/SNS governance component. The impact is concrete: legitimate SNS projects are permanently blocked from launching through the standard governance path, causing direct protocol and economic harm to the Internet Computer ecosystem.

## Likelihood Explanation
The attack requires only a standard `manage_neuron` ingress call ‚ÄĒ no privileged access, no key compromise, no social engineering. The minimum preconditions (‚Č•6 months dissolve delay, ‚Č•10 ICP minted stake) are met by thousands of existing NNS neurons. The cost per attack cycle is at most 10 ICP (the `reject_cost_e8s` lost on rejection), while the attacker's neuron stake is preserved and continues earning voting rewards. A motivated attacker (e.g., a competitor to a specific SNS project) has clear economic incentive and faces no increasing cost or cooldown. The attack requires only one transaction every few days.

## Recommendation
1. **Include the proposer's `NeuronId` in the uniqueness check**: allow each neuron to have at most one open `CreateServiceNervousSystem` proposal at a time, rather than enforcing a global limit. Legitimate proposals from different neurons would then proceed concurrently.
2. **Allow multiple concurrent proposals and resolve conflicts at execution time**: the first adopted proposal executes; subsequent ones fail gracefully with a refund.
3. **Add a per-neuron cooldown** after a `CreateServiceNervousSystem` proposal is rejected, to raise the cost of repeated attacks.

## Proof of Concept
The existing integration test at `rs/nns/integration_tests/src/create_service_nervous_system.rs` already demonstrates the blocking behavior end-to-end. [5](#0-4) 

To demonstrate the attack:
1. Run the existing `test_several_proposals` test ‚ÄĒ it already proves that a first proposal blocks a second with `PreconditionFailed` / `"another open"`.
2. Extend the test: after the first proposal is rejected (not executed), immediately re-submit step 1 and observe the block is re-established. Repeat to confirm indefinite repeatability.
3. To confirm the type-level (not caller-level) nature: submit the blocking proposal from `TEST_NEURON_1` and attempt the legitimate proposal from `TEST_NEURON_2` ‚ÄĒ the block applies regardless of proposer identity.

    fn select_nonfinal_proposal_ids(&self, action_predicate: impl Fn(&Action) -> bool) -> Vec<u64> {
        self.heap_data
            .proposals
            .values()
            .filter_map(|proposal_data| {
                // Disregard proposals that are in a final (or Unspecified) state.
                match proposal_data.status() {
                    ProposalStatus::Open | ProposalStatus::Adopted => (),
                    ProposalStatus::Rejected
                    | ProposalStatus::Executed
                    | ProposalStatus::Failed => {
                        return None;
                    }
                    ProposalStatus::Unspecified => {
                        println!(
                            "{}ERROR: ProposalData had Unspecified status: {:#?}",
                            LOG_PREFIX, proposal_data
                        );
                        return None;
                    }
                };
