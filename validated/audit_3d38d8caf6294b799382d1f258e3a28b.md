Audit Report

## Title
Silent Permanent Loss of Neuron Maturity Rewards via Stable/Heap Memory Divergence on Trap - (File: `rs/nns/governance/src/reward/distribution.rs`)

## Summary
`schedule_pending_rewards_distribution` silently swallows the `Err` returned by `add_rewards_distribution` when a duplicate `day_after_genesis` key is detected in the stable-memory-backed `StableBTreeMap`. Because stable memory writes survive a canister trap while heap memory is rolled back, a trap inside `distribute_voting_rewards_to_neurons` after the stable write but before `self.heap_data.latest_reward_event` is updated leaves the two stores permanently diverged. On the next invocation the duplicate-key error is again silently dropped, the newly computed distribution is discarded, and the stale distribution from the failed attempt is what gets applied — causing neurons to receive wrong (or zero) maturity increments with no recovery path.

## Finding Description
**Step 1 — stable write, no error propagation.**
`schedule_pending_rewards_distribution` (distribution.rs L25-33) calls `add_rewards_distribution`, which returns `Err` when `self.distributions.contains_key(&day_after_genesis)` (distribution.rs L96-99). The `Err` is only logged; the function signature is `-> ()` and the caller receives no signal of failure.

**Step 2 — unconditional heap mutations.**
`distribute_voting_rewards_to_neurons` (governance.rs L6797-6852) calls `schedule_pending_rewards_distribution` and then unconditionally iterates over `new_reward_event.settled_proposals`, calls `self.process_proposal(pid.id)` for each, clears ballots, and finally writes `self.heap_data.latest_reward_event = Some(new_reward_event)` at L6852. The comment at L6784-6795 explicitly acknowledges that "once any mutation has happened, we cannot exit early without the rest" — but this reasoning only covers heap-to-heap consistency, not stable-to-heap consistency across a trap.

**Step 3 — trap creates divergence.**
On the IC, stable memory writes are durable through traps; heap is rolled back. If the canister traps anywhere between L6801 and L6851 (e.g., instruction-limit exhaustion while iterating over a large `settled_proposals` set, or a panic inside `process_proposal`/`record_known_neuron_abstentions`), the stable `StableBTreeMap` retains the day-N entry while heap reverts `latest_reward_event.day_after_genesis` to N−1.

**Step 4 — silent discard on retry.**
The next `CalculateDistributableRewardsTask` execution (calculate_distributable_rewards.rs L51-75) sees `latest_reward_event` at N−1, recomputes rewards (potentially with a different `total_supply` snapshot and different proposal set), calls `schedule_pending_rewards_distribution(N, dist2)`, hits the duplicate-key guard, logs and drops the error, then settles proposals and advances `latest_reward_event` to N. `dist2` is permanently gone; `dist1` (stale) is what `distribute_pending_rewards` will process.

**Existing guards are insufficient.**
The duplicate-key check in `add_rewards_distribution` (distribution.rs L96-99) is the only guard, and it is the mechanism that causes the silent discard rather than preventing it. There is no check in `distribute_voting_rewards_to_neurons` that aborts the heap mutations when the stable write fails.

## Impact Explanation
All neurons that voted on proposals settled in the affected round permanently lose their maturity increment for that round. The `distributed_e8s_equivalent` recorded in the `RewardEvent` will reflect the stale `dist1` amounts rather than the recomputed `dist2` amounts, breaking the accounting invariant of the governance ledger. Because `latest_reward_event` is advanced and proposals are marked `Settled`, there is no retry mechanism. This constitutes a significant NNS governance security impact with concrete, permanent user harm — matching the High bounty impact class: "Significant NNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The trigger requires `distribute_voting_rewards_to_neurons` to trap after the stable write (L6801) but before L6852. Realistic triggers include: (1) instruction-limit exhaustion while iterating over a large `settled_proposals` list and calling `process_proposal` for each; (2) a panic inside `record_known_neuron_abstentions` or `process_proposal`. The NNS subnet has elevated instruction limits, making (1) unlikely under normal load, but not impossible during periods with many simultaneously settling proposals. The CHANGELOG entry at L714-719 ("Neuron migration rolled back due to issues with reward distribution") confirms that the stable/heap divergence class of failure has manifested in production and required an emergency hotfix, establishing that this is not purely theoretical.

## Recommendation
1. **Propagate the error**: Change `schedule_pending_rewards_distribution` to return `Result<(), String>` and have `distribute_voting_rewards_to_neurons` abort (skip settling proposals and skip advancing `latest_reward_event`) if the insertion fails.
2. **Idempotent retry**: If a distribution for `day_after_genesis` already exists in stable memory when `distribute_voting_rewards_to_neurons` is called for the same day, replace or merge it rather than silently dropping the new computation, so a retry after a trap produces correct results.
3. **Move `latest_reward_event` to stable memory**: Storing `latest_reward_event` in stable memory (alongside the distribution) would make both writes atomic with respect to traps, eliminating the divergence window entirely.

## Proof of Concept
1. Deploy governance canister with `latest_reward_event.day_after_genesis = N−1` on heap; stable `REWARDS_DISTRIBUTION_STATE_MACHINE` is empty.
2. Inject a panic (or use a PocketIC instruction-limit hook) that fires inside `distribute_voting_rewards_to_neurons` after `schedule_pending_rewards_distribution` returns but before `self.heap_data.latest_reward_event = Some(new_reward_event)` at governance.rs L6852.
3. Trigger `CalculateDistributableRewardsTask`. Observe: stable memory now contains entry for day N; heap `latest_reward_event.day_after_genesis` is still N−1 (rolled back).
4. Remove the injected panic. Trigger `CalculateDistributableRewardsTask` again. Observe: `add_rewards_distribution` logs "already exists" and returns; `dist2` is discarded; proposals are settled; `latest_reward_event` advances to N.
5. Trigger `DistributeRewardsTask`. Observe: neurons receive maturity from stale `dist1` (wrong supply snapshot, potentially wrong amounts). `dist2` is unrecoverable. The `distributed_e8s_equivalent` in the emitted `RewardEvent` does not match the maturity actually credited.
