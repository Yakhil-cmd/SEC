Audit Report

## Title
Inherited Hot Keys on `spawn_neuron` with New Controller Allow Unauthorized Governance Actions - (File: `rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron unconditionally inherits the parent's `hot_keys` list at L2714. The new controller never authorized these hot keys, yet they retain the ability to vote on NNS proposals, set followees, and join the Neurons' Fund on behalf of the new controller's neuron. Any NNS neuron holder with sufficient maturity can exploit this against an arbitrary victim principal.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` from `spawn.new_controller` at L2651–2655, which may be a completely different principal from the parent's controller. The child neuron is then built at L2704–2718 with `child_controller` as its controller but with `.with_hot_keys(parent_neuron.hot_keys.clone())` (L2714) and `.with_followees(parent_neuron.followees.clone())` (L2715) copied verbatim from the parent — with no conditional clearing when `child_controller != parent_neuron.controller()`.

The only authorization check in the function (L2631–2633) verifies that the *caller* controls the *parent* neuron; it does not restrict or sanitize what is inherited by the child. There is no post-spawn mechanism that notifies the new controller of the inherited hot keys.

`is_authorized_to_vote` in `rs/nns/governance/src/neuron/types.rs` at L2243–2255 delegates to `is_hotkey_or_controller`, which returns `true` if the principal is in `hot_keys` — regardless of whether the current controller ever authorized those keys. This means any principal the attacker pre-loaded as a hot key on the parent neuron retains full non-privileged governance authority over the child neuron after the spawn.

Exploit flow:
1. Attacker adds attacker-controlled principal `A'` as a hot key on parent neuron `N` via `AddHotKey`.
2. Attacker calls `Spawn { new_controller: Some(victim_principal), ... }`.
3. Governance creates child neuron `C` with `controller = victim_principal` and `hot_keys = [A']`.
4. `A'` calls `RegisterVote` on neuron `C`; `is_authorized_to_vote` passes because `A'` is in `hot_keys`.
5. `A'` calls `Follow` on neuron `C`, setting it to follow the attacker's neuron on all non-`NeuronManagement` topics.
6. `A'` calls `JoinCommunityFund` on neuron `C`, committing the victim's maturity to SNS swaps.

## Impact Explanation
This constitutes unauthorized access to a victim's governance neuron. The attacker can silently influence NNS governance outcomes using the victim's voting power, amplify their own voting power through liquid democracy by setting followees, and commit the victim's ICP maturity to SNS swaps via the Neurons' Fund — all without the victim's knowledge or consent. This matches the **High ($2,000–$10,000)** impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The per-target work here is minimal (one `AddHotKey` + one `Spawn` call), but the victim must be specifically targeted, placing it in the High tier.

## Likelihood Explanation
The attack requires only: (1) ownership of any NNS neuron with maturity above the minimum spawn threshold, and (2) knowledge of the victim's principal. Both conditions are common. The entry path is standard unprivileged `manage_neuron` ingress. The scenario is directly applicable to neuron marketplaces or OTC trades where a seller spawns a child neuron to a buyer's principal. No special privileges, social engineering, or external compromise is required.

## Recommendation
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron should be created with an empty `hot_keys` list and without `NeuronManagement`-topic followees:

```rust
let hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
```

Apply the same conditional logic to `with_followees`, filtering out or clearing `NeuronManagement`-topic entries when `child_controller != parent_neuron.controller()`. This ensures the new controller starts with a clean authorization state and can add their own hot keys after taking ownership.

## Proof of Concept
1. In a local replica or PocketIC test, create neuron `N` controlled by principal `A`.
2. Call `manage_neuron(N, Configure(AddHotKey(A')))` — `A'` is a second attacker-controlled principal.
3. Call `manage_neuron(N, Spawn { new_controller: Some(victim), percentage_to_spawn: 100, nonce: None })`.
4. Assert that the resulting child neuron `C` has `controller = victim` and `hot_keys = [A']`.
5. Call `manage_neuron(C, RegisterVote { proposal_id, vote })` signed by `A'`.
6. Assert the vote is accepted (no `NotAuthorized` error), confirming `A'` can vote on behalf of `victim`'s neuron without `victim`'s consent.
7. Call `manage_neuron(C, Follow { topic, followees: [N] })` signed by `A'` and assert success.
