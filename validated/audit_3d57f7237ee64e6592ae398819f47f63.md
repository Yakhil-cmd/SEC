Audit Report

## Title
Stale ICP/XDR Conversion Rate Used Without Freshness Validation in `tokens_to_cycles` - (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles` function in the Cycles Minting Canister reads only `xdr_permyriad_per_icp` from the stored `IcpXdrConversionRate` and never validates `timestamp_seconds`. If the Exchange Rate Canister (XRC) experiences downtime and the CMC's heartbeat fails to refresh the rate, the CMC will silently continue minting cycles at an arbitrarily stale price. An unprivileged user who observes a stale rate during an ICP price decline can call `notify_top_up` or `notify_mint_cycles` to acquire cycles at a favorable rate, representing a direct and irreversible economic loss to the IC ecosystem.

## Finding Description
`tokens_to_cycles` at `rs/nns/cmc/src/main.rs` L1900–1923 extracts only `xdr_permyriad_per_icp` from the stored rate and discards `timestamp_seconds` entirely:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp); // timestamp never consulted
        ...
    })
}
```

This function is the sole conversion path for three publicly reachable update endpoints: `process_top_up` (L1991), `process_create_canister` (L1932), and `process_mint_cycles` (L1965).

The CMC is designed to refresh the rate every 5 minutes via heartbeat (`REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS`, `rs/nns/cmc/src/exchange_rate_canister.rs` L16). On failure, it retries every minute (`get_rate_at_next_minute`, L53–60). However, no code path enforces a maximum age on the stored rate at the point of use. A grep for `MAX_RATE_AGE`, `stale.*rate`, and `timestamp.*check` across the entire CMC source returns zero matches.

`do_set_icp_xdr_conversion_rate` (L1009–1040) only enforces that a new rate must have a strictly greater timestamp than the current one — it does not enforce any upper bound on how old the stored rate may be when it is consumed. The `UpdateExchangeRateGuard` prevents concurrent refreshes but provides no protection against indefinite staleness.

**Exploit flow:**
1. XRC experiences downtime (subnet issue, canister upgrade, or external data source failure). The CMC heartbeat begins failing; on each failure, the next attempt is scheduled one minute later.
2. ICP market price drops (e.g., 30%) while the XRC is unavailable.
3. The CMC retains the last known rate with its old `timestamp_seconds` and no upper-bound check.
4. An unprivileged user calls `notify_top_up` with a valid ICP ledger block.
5. `process_top_up` calls `tokens_to_cycles`, which reads the stale (pre-drop) `xdr_permyriad_per_icp` without checking `timestamp_seconds`.
6. The user receives ~43% more cycles than the current market rate warrants.
7. The CMC burns the ICP and deposits the inflated cycle count — the transaction is irreversible.

The hourly rate limiter (`base_cycles_limit = DEFAULT_CYCLES_LIMIT = 150e15` cycles/hour, L83) bounds per-hour damage but does not eliminate it. At 1 XDR ≈ $1.30 and 1 trillion cycles = 1 XDR, 150 petacycles ≈ $195,000/hour. A 30% price discrepancy yields ~$58,500/hour of extractable value. Over multi-hour XRC downtime, cumulative losses can be substantial and are irreversible.

## Impact Explanation
This matches **High ($2,000–$10,000): Significant XRC/NNS/CMC infrastructure security impact with concrete user or protocol harm.** An unprivileged user can exploit the stale rate window to acquire cycles at a discount, draining value from the IC ecosystem. The ICP is permanently burned; the inflated cycle deposit cannot be reversed. The rate limiter caps per-hour exposure but does not prevent exploitation across multiple hours of XRC downtime.

## Likelihood Explanation
The XRC is a separate system canister on the IC. Realistic triggers include: XRC subnet degradation, inter-canister call failures, CMC upgrade interrupting the heartbeat loop, or external forex/crypto data source outages. The CMC's own code acknowledges this dependency. No special privileges are required — any user with a valid ICP ledger block can call `notify_top_up` or `notify_mint_cycles`. The attack is repeatable up to the hourly rate limit and requires only that the attacker observe the rate is stale (the `get_icp_xdr_conversion_rate` query endpoint exposes `timestamp_seconds` publicly).

## Recommendation
In `tokens_to_cycles`, validate that `icp_xdr_conversion_rate.timestamp_seconds` is within an acceptable window before using the rate. A `MAX_RATE_AGE_SECONDS` of 30–60 minutes gives the heartbeat multiple retry windows before blocking minting operations:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref();
        match rate {
            Some(rate) => {
                let age = now_seconds().saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: "ICP/XDR conversion rate is stale".to_string(),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(...)
        }
    })
}
```

## Proof of Concept
1. Deploy a local replica with the NNS canisters (CMC + XRC).
2. Set the CMC's `icp_xdr_conversion_rate` to a known value with `timestamp_seconds = now`.
3. Disable the mock XRC (stop responding to calls). The CMC heartbeat will begin failing; the stored rate remains unchanged.
4. Advance the replica clock by 2+ hours (beyond any reasonable freshness window).
5. Call `notify_top_up` with a valid ICP ledger block from an unprivileged principal.
6. Observe that `tokens_to_cycles` returns cycles computed from the stale rate without error.
7. Confirm the cycle count exceeds what the current (lower) market rate would yield.
8. Confirm the ICP is burned and the cycle deposit is irreversible.

This can be implemented as a deterministic integration test using `StateMachine` (as used in `rs/nns/integration_tests/src/cycles_minting_canister.rs`) by advancing time past the staleness threshold while suppressing XRC responses.
