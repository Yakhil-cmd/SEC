### Title
SNS `default_followees` Governance Parameter Cannot Be Set to Any Non-Empty Value Despite Being Documented as Changeable via Proposals ŌĆö (`rs/sns/governance/src/types.rs`)

---

### Summary

The `default_followees` field in `NervousSystemParameters` is documented as a parameter that "can be changed, via proposals, by each nervous system community," and is used during neuron creation to assign initial followees. However, the validation logic actively rejects any non-empty value for this field, making it permanently frozen at an empty map. No `ManageNervousSystemParameters` proposal can ever set meaningful default followees for an SNS.

---

### Finding Description

`NervousSystemParameters` is described in both the proto and the generated Rust code as:

> "The nervous system's parameters, which are parameters that can be changed, via proposals, by each nervous system community." [1](#0-0) 

The `default_followees` field is included in this struct and is described as:

> "The set of default followees that every newly created neuron will follow per function." [2](#0-1) 

However, the validation function `validate_default_followees` in `NervousSystemParameters::validate()` enforces that the followees map **must always be empty**: [3](#0-2) 

The same enforcement exists at the `GovernanceProto` level: [4](#0-3) 

Both locations carry the comment `TODO NNS1-2169: default followees are not currently supported`, confirming this is an unimplemented feature. The `validate()` method is called during `ManageNervousSystemParameters` proposal execution: [5](#0-4) 

And also during proposal submission via `validate_and_render_manage_nervous_system_parameters`: [6](#0-5) 

The field is read during `claim_neuron` to assign followees to newly created neurons, but since validation always enforces an empty map, this read always yields an empty result: [7](#0-6) 

The getter also carries the same TODO: [8](#0-7) 

---

### Impact Explanation

Any SNS community that submits a `ManageNervousSystemParameters` proposal attempting to set `default_followees` to a non-empty value will have the proposal rejected at submission time (validation fails before the proposal is even stored). The feature is part of the public Candid interface: [9](#0-8) 

SNS communities cannot configure which neurons new members automatically follow upon neuron creation, even though this is a documented and exposed governance capability. All newly created neurons will always have an empty followees map regardless of any governance proposal.

---

### Likelihood Explanation

Any SNS neuron holder who reads the `NervousSystemParameters` documentation or the Candid interface and attempts to use `ManageNervousSystemParameters` to configure `default_followees` will encounter a rejection. The entry path is fully unprivileged ŌĆö any neuron holder can submit such a proposal. The feature is permanently broken for all deployed SNS instances until `NNS1-2169` is resolved.

---

### Recommendation

Either:
1. Implement the `default_followees` feature (resolve `NNS1-2169`) so that `validate_default_followees` permits non-empty maps and `claim_neuron` correctly applies them to new neurons.
2. Or remove `default_followees` from the public `NervousSystemParameters` Candid interface and proto until the feature is implemented, to avoid misleading SNS communities.

---

### Proof of Concept

1. An SNS neuron holder submits a `ManageNervousSystemParameters` proposal with:
   ```
   default_followees = Some(DefaultFollowees { followees: { 1 => [neuron_id_X] } })
   ```
2. `validate_and_render_manage_nervous_system_parameters` calls `inherit_from(...).validate()`.
3. `validate()` calls `validate_default_followees()`.
4. `validate_default_followees` returns `Err("DefaultFollowees.default_followees must be empty, but found ...")`.
5. The proposal is rejected with `ErrorType::InvalidProposal`.
6. No SNS governance proposal can ever set a non-empty `default_followees`, contradicting the documented behavior of `NervousSystemParameters`. [3](#0-2) [10](#0-9)

```rust
    /// NervousSystemParameters
    fn perform_manage_nervous_system_parameters(
        &mut self,
        proposed_params: NervousSystemParameters,
    ) -> Result<(), GovernanceError> {
        // Only set `self.proto.parameters` if "applying" the proposed params to the
        // current params results in valid params
        let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());

        log!(
            INFO,
            "Setting Governance nervous system params to: {:?}",
            &new_params
        );

        match new_params.validate() {
            Ok(()) => {
                self.proto.parameters = Some(new_params);
                Ok(())
            }

            // Even though proposals are validated when they are first made, this is still
            // possible, because the inner value of a ManageNervousSystemParameters
            // proposal is only valid with respect to the current
            // nervous_system_parameters() at the time when the proposal was first
            // made. If nervous_system_parameters() changed (by another proposal) since
            // the current proposal was first made, the current proposal might have become
            // invalid. Basically, this might occur if there are conflicting (concurrent)
            // proposals, but we expect this to be highly unusual in practice.
            Err(msg) => Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!(
                    "Failed to perform ManageNervousSystemParameters action, proposed \
                        parameters would lead to invalid NervousSystemParameters: {msg}"
                ),
            )),
        }
```

```rust
/// Validates and renders a proposal with action ManageNervousSystemParameters.
fn validate_and_render_manage_nervous_system_parameters(
    new_parameters: &NervousSystemParameters,
    current_parameters: &NervousSystemParameters,
) -> Result<String, String> {
    if new_parameters == &NervousSystemParameters::default() {
        return Err("NervousSystemParameters: at least one field must be set.".to_string());
    }

    new_parameters.inherit_from(current_parameters).validate()?;

    Ok(format!(
        r"# Proposal to change nervous system parameters:
## Current nervous system parameters:

{:#?}

## New nervous system parameters:

{:#?}",
        &current_parameters, new_parameters
    ))
```
