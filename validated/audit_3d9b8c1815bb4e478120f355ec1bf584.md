### Title
SNS-Specific Upgrade Path Entry Cannot Be Updated After Initial Set — (`rs/nns/sns-wasm/src/sns_wasm.rs`)

### Summary

`insert_sns_specific_upgrade_path_entry` in the `UpgradePath` impl silently drops all update attempts on an already-occupied entry. The `Entry::Occupied` branch logs "is being overwritten" but never calls `occupied.insert(to)`, so the first value written is permanently frozen. Every subsequent NNS Governance proposal to correct or update that SNS-specific upgrade path returns success but has zero effect.

### Finding Description

The bug is in `insert_sns_specific_upgrade_path_entry`: [1](#0-0) 

```rust
Entry::Occupied(occupied) => {
    println!(
        "Special Entry for {} from {:?} to {:?} is being overwritten with new value {:?}",
        ...
    );
    // BUG: occupied.insert(to) is never called
}
```

Compare with the non-SNS-specific sibling `insert_upgrade_path_entry`, which correctly mutates the occupied entry: [2](#0-1) 

```rust
Entry::Occupied(mut occupied) => {
    println!(...);
    occupied.insert(to);   // ← present here, absent above
}
```

The caller `insert_upgrade_path_entries` iterates over the submitted upgrade steps and delegates to `insert_sns_specific_upgrade_path_entry` for every step when an `sns_governance_canister_id` is provided: [3](#0-2) 

The function returns `InsertUpgradePathEntriesResponse { error: None }` regardless, so the caller (NNS Governance executing a proposal) receives a success response while the state is unchanged.

### Impact Explanation

The SNS-specific upgrade path is the designated escape hatch for SNSes that cannot follow the standard upgrade path due to a bug. Once a path entry is set for a given `(governance_canister_id, current_version)` pair, it can never be corrected or superseded:

- A mistaken first proposal permanently locks the wrong `next_version`.
- A follow-up NNS Governance proposal with the correct `next_version` silently no-ops.
- The affected SNS is stuck on the wrong upgrade path with no on-chain remedy. [4](#0-3) 

### Likelihood Explanation

This is triggered through the normal NNS Governance proposal flow (`InsertUpgradePathEntries` proposal type). No privileged key or majority corruption is required. Any time an operator submits a corrective proposal for an SNS-specific path that was already set — a routine operational action — the bug fires. The log message actively misleads operators into believing the update succeeded.

### Recommendation

Add `occupied.insert(to)` in the `Entry::Occupied` branch of `insert_sns_specific_upgrade_path_entry`, mirroring the existing correct behavior in `insert_upgrade_path_entry`:

```rust
Entry::Occupied(mut occupied) => {
    println!("Special Entry for {} from {:?} to {:?} is being overwritten with new value {:?}", ...);
    occupied.insert(to);   // ← add this line
}
```

### Proof of Concept

```rust
// Call insert_sns_specific_upgrade_path_entry twice with the same `from`
// but different `to` values; assert the second value is returned by get_next_version.
let mut path = UpgradePath::default();
path.insert_sns_specific_upgrade_path_entry(from.clone(), to_v1.clone(), gov_id);
path.insert_sns_specific_upgrade_path_entry(from.clone(), to_v2.clone(), gov_id);
// Currently returns to_v1 (first-set value); should return to_v2.
assert_eq!(path.get_next_version(from, gov_id.into()), Some(to_v2));
``` [5](#0-4)

    pub fn insert_sns_specific_upgrade_path_entry(
        &mut self,
        from: SnsVersion,
        to: SnsVersion,
        sns_governance_canister_id: CanisterId,
    ) {
        match self
            .sns_specific_upgrade_path
            .entry(sns_governance_canister_id)
            .or_default()
            .entry(from)
        {
            Entry::Occupied(occupied) => {
                println!(
                    "Special Entry for {}  from {:?} to {:?} is being overwritten with new value {:?}",
                    sns_governance_canister_id,
                    occupied.key(),
                    occupied.get(),
                    to
                );
            }
            Entry::Vacant(vacant) => {
                vacant.insert(to);
            }
        };
```
