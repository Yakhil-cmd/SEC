Audit Report

## Title
Index-out-of-bounds panic via empty `EnvelopePair` vec in `construction_parse` crashes Rosetta process - (File: `rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

## Summary
The `construction_parse` handler unconditionally indexes `updates[0]` on a `Vec<EnvelopePair>` that is fully attacker-controlled. Because `SignedTransaction` deserialization performs no structural validation beyond CBOR well-formedness, an attacker can supply a signed transaction containing a `Request` with an empty envelope pair list, causing a Rust index-out-of-bounds panic that terminates the Rosetta process. No authentication or special privilege is required.

## Finding Description
`Request` is defined as `(RequestType, Vec<EnvelopePair>)`. [1](#0-0) 

`SignedTransaction::from_str` decodes the hex+CBOR payload but applies no structural constraint — it only requires the bytes to be valid CBOR (or valid legacy CBOR). [2](#0-1) 

`ParsedTransaction::try_from` similarly only checks CBOR decodability for the signed branch, with no validation that each `Vec<EnvelopePair>` is non-empty. [3](#0-2) 

In `construction_parse`, the signed branch immediately indexes `updates[0]` with no bounds check: [4](#0-3) 

If `updates` is an empty `Vec`, Rust panics with `index out of bounds: the len is 0 but the index is 0`, which is an unrecoverable panic that terminates the Rosetta process. There is no `catch_unwind` or similar guard wrapping this handler.

## Impact Explanation
The Rosetta process crashes immediately upon receiving a single crafted HTTP request. All in-flight requests are dropped and the service is unavailable until the process is restarted. The ICP Rosetta API is an explicitly in-scope financial integration component. This matches the **High ($2,000–$10,000)** impact: *"Application/platform-level DoS, crash … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … Rosetta … security impact with concrete user or protocol harm."*

## Likelihood Explanation
The `/construction/parse` endpoint is public and unauthenticated. The CBOR encoding of `SignedTransaction` with an empty `Vec<EnvelopePair>` is straightforward to construct. A single HTTP POST is sufficient to crash the process; the attack is trivially repeatable and requires no special knowledge beyond the documented Rosetta API format.

## Recommendation
Replace the unchecked `updates[0]` index with a bounds-checked access that returns an `ApiError` on empty input:

```rust
.map(|(request_type, updates)| {
    let first = updates.first().ok_or_else(|| {
        ApiError::invalid_request("Empty envelope pair list".to_string())
    })?;
    match first.update.content.clone() {
        HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
    }
})
.collect::<Result<Vec<_>, _>>()?
```

Additionally, consider adding a non-empty validation in `ParsedTransaction::try_from` or in `SignedTransaction`'s deserialization to reject malformed inputs early.

## Proof of Concept
```rust
// Construct a SignedTransaction with an empty EnvelopePair vec
let signed_tx = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],  // empty Vec<EnvelopePair>
};
let cbor_bytes = serde_cbor::to_vec(&signed_tx).unwrap();
let hex_str = hex::encode(&cbor_bytes);

// POST to /construction/parse:
// { "network_identifier": {...}, "signed": true, "transaction": "<hex_str>" }
//
// -> ParsedTransaction::try_from succeeds (valid CBOR)
// -> construction_parse iterates requests, hits updates[0] on empty vec
// -> panic: index out of bounds: the len is 0 but the index is 0
// -> Rosetta process terminates
```

A unit test can be added directly in `rs/rosetta-api/icp/src/request_handler/construction_parse.rs` alongside the existing `test_payloads_parse_identity` test, constructing the above `SignedTransaction`, CBOR-encoding it, and asserting that `construction_parse` returns an `Err(ApiError)` rather than panicking.
