### Title
Query Cache `ignore_canister_balances` Omits `canister_liquid_cycle_balance128` Counter, Serving Stale Liquid Balance Results — (`rs/execution_environment/src/query_handler/query_cache.rs`)

### Summary

`EntryValue::new` sets `ignore_canister_balances = true` by checking only `canister_cycle_balance == 0 && canister_cycle_balance128 == 0`. It never checks `canister_liquid_cycle_balance128`. Because `SystemApiCallCounters` tracks all three counters independently, a canister that exclusively calls `ic0.canister_liquid_cycle_balance128()` will have its cache entry marked as balance-change-immune, causing stale liquid balance values to be served indefinitely after the raw balance changes without a canister version bump.

### Finding Description

`SystemApiCallCounters` in `rs/interfaces/src/execution_environment.rs` tracks three distinct balance-related counters:

```rust
pub canister_cycle_balance: usize,
pub canister_cycle_balance128: usize,
pub canister_liquid_cycle_balance128: usize,   // ← tracked but never consulted
``` [1](#0-0) 

`EntryValue::new` computes the `ignore_canister_balances` flag as:

```rust
let ignore_canister_balances = system_api_call_counters.canister_cycle_balance == 0
    && system_api_call_counters.canister_cycle_balance128 == 0;
// canister_liquid_cycle_balance128 is never checked
``` [2](#0-1) 

`is_valid()` then uses this flag to bypass the raw-balance staleness check:

```rust
&& (all_canister_balances_are_valid || self.ignore_canister_balances)
``` [3](#0-2) 

The raw-balance comparison itself only checks `canister.system_state.balance()` — the gross balance, not the liquid balance. The liquid balance is `gross_balance − freeze_threshold_cycles`, so any change to the gross balance (cycles added by a third-party canister, or system deductions for compute/storage fees) changes the liquid balance without bumping the canister version. [4](#0-3) 

The system API implementation confirms `canister_liquid_cycle_balance128` increments its own counter and is entirely independent of the other two: [5](#0-4) 

### Impact Explanation

Any canister that calls **only** `ic0.canister_liquid_cycle_balance128()` (the newer, recommended API for spendable balance) and never calls the deprecated `ic0.canister_cycle_balance` or `ic0.canister_cycle_balance128` will have its query result cached with `ignore_canister_balances = true`. After the canister's gross balance changes — via cycles deposited by a third-party canister or burned by the system for fees — subsequent identical queries return the stale liquid balance until the cache entry expires (`max_expiry_time`) or the canister version is bumped by an unrelated update call.

The direct impact is **stale query reads**. Because actual cycle transfers and withdrawals go through update calls (which bypass the query cache entirely), this does not directly enable unauthorized state changes. However, any frontend, monitoring system, or composite-query logic that relies on the query result to display or gate on the spendable balance will observe incorrect values for the duration of the cache window.

### Likelihood Explanation

- Triggerable by any unprivileged canister developer deploying a canister that uses the newer `ic0.canister_liquid_cycle_balance128` API exclusively.
- Balance changes without version bumps are routine: the system deducts cycles for compute and storage every round; third-party canisters can deposit cycles at any time.
- No special privileges, governance majority, or key material required.
- The bug is locally testable with a unit test in `query_cache/tests.rs`.

### Recommendation

Add `canister_liquid_cycle_balance128` to the `ignore_canister_balances` guard in `EntryValue::new`:

```rust
let ignore_canister_balances = system_api_call_counters.canister_cycle_balance == 0
    && system_api_call_counters.canister_cycle_balance128 == 0
    && system_api_call_counters.canister_liquid_cycle_balance128 == 0;
``` [2](#0-1) 

Additionally, add a regression test in `rs/execution_environment/src/query_handler/query_cache/tests.rs` that:
1. Deploys a canister calling only `ic0.canister_liquid_cycle_balance128()`.
2. Caches the first query result.
3. Changes the canister's gross balance without an update call (simulating a third-party deposit or system fee deduction).
4. Asserts the second identical query re-executes rather than returning the cached stale value.

### Proof of Concept

```
1. Deploy canister C with query method:
      call ic0.canister_liquid_cycle_balance128 → reply with result

2. Issue query Q to C → executes, result R1 cached.
   EntryValue::new:
     canister_cycle_balance        = 0  ✓
     canister_cycle_balance128     = 0  ✓
     canister_liquid_cycle_balance128 = 1  (NOT checked)
   → ignore_canister_balances = true

3. Third-party canister deposits 1_000_000 cycles into C.
   C's gross balance increases; canister_version unchanged.

4. Issue identical query Q to C → is_valid() evaluates:
     all_canister_balances_are_valid = false  (balance changed)
     ignore_canister_balances        = true   (bug)
   → cache hit, returns stale R1 (pre-deposit liquid balance)

5. Actual liquid balance is R1 + 1_000_000 cycles; query reports R1.
