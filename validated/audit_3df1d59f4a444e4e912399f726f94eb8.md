### Title
VIP Callers Blocked by Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

---

### Summary

A logic error in `LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` causes VIP callers (NNS canisters, including Governance) to be rejected with `SysTransient` when `available_slot_count == 0`, even though VIP callers are designed to consume zero slots. An unprivileged attacker can exhaust all 167 slots by flooding NNS Root's public `canister_status` endpoint with concurrent calls, then block governance-triggered `load_canister_snapshot` and `take_canister_snapshot` operations for the duration of the flood.

---

### Finding Description

**The bug — `try_borrow_slot`:** [1](#0-0) 

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {          // ← blocks VIP too
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The intent is that VIP callers (`is_caller_vip = true`) do not consume a slot (`used_slot_count = 0`). However, the early-exit guard `if *available_slot_count == 0` fires **before** the VIP distinction is applied. When the counter reaches zero, every caller — VIP or not — receives `Err(SysTransient)`.

**The slot pool and VIP determination:**

`AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is initialized to 167 and is shared across all callers of NNS Root. [2](#0-1) 

VIP status is determined by checking whether the caller is in `ALL_NNS_CANISTER_IDS`: [3](#0-2) 

**The public attack surface:**

NNS Root's `canister_status` endpoint is explicitly public — no access control: [4](#0-3) 

Each call to `canister_status` borrows a slot and holds it for the entire duration of the in-flight management canister call. With 167 concurrent in-flight calls, `available_slot_count` reaches 0.

**The blocked VIP paths:**

`take_canister_snapshot` and `load_canister_snapshot` on NNS Root are only callable by Governance (a VIP), but both call `new_management_canister_client()` which routes through `try_borrow_slot`: [5](#0-4) 

When slots are exhausted, `try_borrow_slot` returns `Err(SysTransient)` to the VIP caller. For non-Governance target canisters (e.g., ledger), `perform_offline_canister_maintenance` propagates this synchronously: the target canister is stopped, the snapshot call fails, the target canister is restarted, and `LoadCanisterSnapshotResponse::Err(...)` is returned to Governance. [6](#0-5) 

---

### Impact Explanation

- Governance-approved `TakeCanisterSnapshot` and `LoadCanisterSnapshot` proposals fail with an error response from NNS Root.
- For non-Governance target canisters (ledger, SNS canisters, etc.), the target canister is stopped and restarted with no snapshot taken/loaded — the proposal execution is wasted.
- For the Governance canister itself, the operation is spawned in the background and returns an optimistic `Ok` immediately, but the background task silently fails.
- An attacker who continuously maintains 167 in-flight `canister_status` calls can indefinitely delay disaster recovery snapshot/restore operations.

---

### Likelihood Explanation

- The `canister_status` endpoint is publicly callable with no rate limiting in the canister code itself.
- 167 concurrent in-flight calls is well within IC queue limits (500 per queue).
- Management canister `canister_status` calls are fast (milliseconds), so the attacker must continuously re-flood, but this is straightforward to automate.
- No privileged access, no key material, no social engineering required.

---

### Recommendation

In `try_borrow_slot`, skip the `== 0` guard entirely for VIP callers, since they consume zero slots and should never be blocked:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

### Proof of Concept

State-machine test outline:

1. Initialize NNS Root with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Submit 167 concurrent ingress `canister_status` calls from an unprivileged principal, each targeting a valid NNS canister ID. Configure the mock management canister to hold responses (simulate slow responses).
3. Assert `available_slot_count == 0`.
4. Submit a `load_canister_snapshot` call from the Governance canister principal (VIP).
5. Assert the response is `LoadCanisterSnapshotResponse::Err { code: Some(SysTransient), ... }` instead of `Ok`.

This directly mirrors the existing test structure in `rs/nervous_system/clients/src/management_canister_client/tests.rs` which already validates the slot-limiting behavior, but does not test the VIP-blocked-at-zero case. [7](#0-6)
