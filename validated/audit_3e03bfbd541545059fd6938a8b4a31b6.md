### Title
Ledger Transfer DoS When Archive Canister Creation Panics Due to Insufficient Cycles - (File: rs/ledger_suite/common/ledger_canister_core/src/archive.rs)

### Summary
The ICP Ledger and ICRC-1 Ledger both call `archive_blocks` as part of every transfer. When the ledger's liquid cycle balance is insufficient to create a new archive canister node, the IC system implicitly panics, rolling back the triggering transfer and permanently blocking all subsequent transfers until the ledger is topped up with cycles. This is an exact structural analog to the AlgebraVirtualPool push-pattern DoS: a critical operation (transfer) makes an external inter-canister call (archive canister creation) that, when it fails, blocks the core operation for all users.

### Finding Description

Every transfer in both the ICP Ledger and the ICRC-1 Ledger ends with a call to `archive_blocks`, which is invoked after the block has been committed to the in-memory ledger state:

**ICP Ledger (`rs/ledger_suite/icp/ledger/src/main.rs`):** [1](#0-0) 

**ICRC-1 Ledger (`rs/ledger_suite/icrc1/ledger/src/main.rs`):** [2](#0-1) 

Inside `archive_blocks`, when the local block count exceeds the configured `trigger_threshold`, the ledger must spawn a new archive canister node. The function `create_new_archive_node` in the shared archive core calls `spawn::create_canister` to do so. The code itself documents the DoS condition explicitly: [3](#0-2) 

The comment reads verbatim:
> *"Note that this will implicitly panic if: `cycles_for_archive_creation` is enough to create a canister, but the ledger does not have enough (liquid) cycles to attach to the call. Panicking leads to the rolling back of the transaction that triggered the archiving, and **no more transactions will be processed by the ledger until it has been topped up** with enough cycles to spawn the archive canister."*

The panic is an IC system-level trap triggered when the runtime attempts to attach `cycles_for_archive_creation` cycles to the `create_canister` management call but the canister's liquid balance is insufficient. Because this trap occurs before the async call is committed, the entire message execution — including the transfer that triggered archiving — is rolled back.

The persistent DoS loop is:
1. Block count reaches `trigger_threshold`.
2. User submits a transfer → block is added → `archive_blocks` is called.
3. Ledger attempts to spawn archive canister → IC system panics (insufficient liquid cycles).
4. Entire message is rolled back; block count returns to `trigger_threshold`.
5. Next transfer repeats from step 2 indefinitely.

The `archive_blocks` function itself returns `()` and handles soft errors gracefully: [4](#0-3) 

However, the implicit IC system panic bypasses this graceful error path entirely, making the failure unrecoverable without an operator top-up.

### Impact Explanation

All `icrc1_transfer`, `icrc2_transfer_from`, `icrc2_approve`, and legacy `send` calls to the affected ledger canister are permanently rejected until an operator tops up the ledger's cycle balance. This affects every user and every protocol (DEXes, bridges, governance, chain-fusion) that depends on the ledger for token transfers. The ICP Ledger and ICRC-1 Ledger are critical system-level infrastructure; a DoS here is protocol-wide.

### Likelihood Explanation

An unprivileged ingress sender can trigger this condition:

1. Any user can submit transfers to the ledger (paying the standard transfer fee).
2. Each accepted transfer adds one block to the ledger's local blockchain.
3. When the block count reaches `trigger_threshold` (a configured value, e.g., 2,000), the next transfer triggers archive canister creation.
4. If the ledger's liquid cycle balance has been depleted — either naturally over time through prior archiving operations, or accelerated by an attacker submitting many cheap transfers — the panic fires.

The attacker's cost is bounded by `trigger_threshold × transfer_fee`. For the ICP Ledger with a 10,000 e8s fee and a threshold of 2,000 blocks, the cost is approximately 0.2 ICP to trigger the condition. The condition is also reachable without any attacker if the ledger operator simply fails to top up cycles over time.

### Recommendation

**Short term:** Wrap the `spawn::create_canister` call in a guard that checks whether the ledger's liquid cycle balance strictly exceeds `cycles_for_archive_creation` plus a safety margin *before* attempting the call, and return a `FailedToArchiveBlocks` error (not a panic) if the check fails. The existing pre-flight checks at lines 420–445 of `archive.rs` already attempt this but do not cover the edge case where the balance is exactly at the boundary or changes between check and call. [5](#0-4) 

**Long term:** Adopt the pull pattern: instead of the ledger pushing blocks to the archive canister synchronously during a transfer, have the archive canister (or a separate timer task) pull blocks from the ledger. This decouples the archiving operation from the transfer critical path, so archiving failures cannot block transfers — the same recommendation made in the original AlgebraVirtualPool report.

### Proof of Concept

1. Deploy an ICRC-1 ledger on an application subnet with `archive_options.trigger_threshold = 2000`, `archive_options.cycles_for_archive_creation = DEFAULT_CYCLES_FOR_ARCHIVE_CREATION` (10T cycles), and fund the ledger with slightly less than `DEFAULT_CYCLES_FOR_ARCHIVE_CREATION` liquid cycles.
2. Submit 2,000 `icrc1_transfer` calls from any unprivileged principal.
3. The 2,001st transfer triggers `archive_blocks` → `send_blocks_to_archive` → `node_and_capacity` → `create_new_archive_node` → `spawn::create_canister`.
4. The IC system panics when attaching cycles to the `create_canister` call.
5. The 2,001st transfer is rejected and rolled back; the block count returns to 2,000.
6. Every subsequent transfer repeats steps 3–5; the ledger is permanently DoS'd.
7. The condition is confirmed by the code comment at `rs/ledger_suite/common/ledger_canister_core/src/archive.rs:449–454` and the test `test_archiving_fails_on_app_subnet_if_ledger_does_not_have_enough_cycles` in `rs/ledger_suite/tests/sm-tests/src/lib.rs:4690`. [3](#0-2) [6](#0-5)

pub async fn archive_blocks<LA: LedgerAccess>(sink: impl Sink + Clone, max_message_size: u64) {
    use crate::archive::{ArchivingGuardError, send_blocks_to_archive};

    let archive_arc = LA::with_ledger(|ledger| ledger.blockchain().archive.clone());

    // NOTE: this guard will prevent another logical thread to start the archiving process.
    let (archiving_guard, blocks_to_archive) = match blocks_to_archive::<LA>(&sink) {
        Ok((guard, blocks)) => (guard, blocks),
        Err(ArchivingGuardError::NoArchive) => {
            return; // Archiving not enabled
        }
        Err(ArchivingGuardError::AlreadyArchiving) => {
            return; // Ledger is currently archiving, skipping archive_blocks.
        }
    };

    if blocks_to_archive.is_empty() {
        return;
    }

    let num_blocks = blocks_to_archive.len();

    let result = send_blocks_to_archive(
        sink.clone(),
        archive_arc,
        blocks_to_archive,
        max_message_size,
    )
    .await;

    if result.is_err() {
        LA::with_ledger_mut(|ledger| ledger.increment_archiving_failure_metric());
    }

    remove_archived_blocks::<LA>(archiving_guard, num_blocks, &sink, result)

    pub fn test_archiving_fails_on_app_subnet_if_ledger_does_not_have_enough_cycles<T, B>(
        ledger_wasm: Vec<u8>,
        encode_init_args: fn(InitArgs) -> T,
        get_archives: fn(&StateMachine, CanisterId) -> Vec<Principal>,
        get_blocks_fn: fn(&StateMachine, CanisterId, u64, usize) -> GenericGetBlocksResponse<B>,
    ) where
        T: CandidType,
        B: Eq + Debug,
    {
        const NUM_BLOCKS_TO_ARCHIVE: usize = 1_000;
        const NUM_INITIAL_BALANCES: u64 = 70_000;
        const TRIGGER_THRESHOLD: usize = 2_000;
        let p1 = PrincipalId::new_user_test_id(1);
        let p2 = PrincipalId::new_user_test_id(2);
        let archive_controller = PrincipalId::new_user_test_id(1_000_000);
        let mut initial_balances = vec![];
        for i in 0..NUM_INITIAL_BALANCES {
            initial_balances.push((
                Account::from(PrincipalId::new_user_test_id(i).0),
                10_000_000,
            ));
        }

        let env = StateMachineBuilder::new()
            .with_subnet_type(SubnetType::Application)
            .build();
        let args = encode_init_args(InitArgs {
            archive_options: ArchiveOptions {
                trigger_threshold: TRIGGER_THRESHOLD,
                num_blocks_to_archive: NUM_BLOCKS_TO_ARCHIVE,
                node_max_memory_size_bytes: None,
                max_message_size_bytes: None,
                controller_id: archive_controller,
                more_controller_ids: None,
                cycles_for_archive_creation: Some(
                    ic_ledger_canister_core::archive::DEFAULT_CYCLES_FOR_ARCHIVE_CREATION,
                ),
                max_transactions_per_response: None,
            },
            ..init_args(initial_balances)
        });
        let args = Encode!(&args).unwrap();
        let ledger_id = env
            .install_canister_with_cycles(
                ledger_wasm.clone(),
                args,
                None,
                Cycles::new((0.9 * DEFAULT_CYCLES_FOR_ARCHIVE_CREATION as f64) as u128),
            )
            .unwrap();

        // Assert no archives exist.
        assert!(get_archives(&env, ledger_id).is_empty());
        let get_blocks_res = get_blocks_fn(&env, ledger_id, 0, 1);
        assert_eq!(get_blocks_res.chain_length, NUM_INITIAL_BALANCES);
        // Verify that the ledger response contained no archive info.
        assert!(get_blocks_res.archived_ranges.is_empty());
        // Verify that the archiving failure metric is zero.
        assert_archiving_failure_metric(&env, ledger_id, 0_u64);

        // Send a transfer that should trigger an attempt to spawn an archive.
        send_transfer(
            &env,
            ledger_id,
            p1.0,
            &TransferArg {
                from_subaccount: None,
                to: p2.0.into(),
                fee: None,
                created_at_time: None,
                memo: None,
                amount: Nat::from(10_000_u64),
            },
        )
        .expect("transfer should succeed");

        // Assert no archives exist, as the spawning failed.
        assert!(get_archives(&env, ledger_id).is_empty());
        // Verify that no new block was created, since the transfer failed (the ledger panicked).
        let get_blocks_res = get_blocks_fn(&env, ledger_id, 0, 1);
        assert_eq!(get_blocks_res.chain_length, NUM_INITIAL_BALANCES + 1);
        // Verify that the ledger response contained no archive info.
        assert!(get_blocks_res.archived_ranges.is_empty());
        // Verify that the archiving failure metric was incremented.
        assert_archiving_failure_metric(&env, ledger_id, 1_u64);
    }
