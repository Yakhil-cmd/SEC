Audit Report

## Title
Permanent Slot Leak via `.unwrap()` Trap in Public `canister_status` Endpoint — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary

The public `canister_status` endpoint in NNS Root calls `.unwrap()` on the management canister response. When the management canister rejects the call, the callback traps. Under the IC's atomic callback execution model, the `SlotLoan` restoration that occurs before the trap is rolled back, permanently decrementing `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT`. An unprivileged attacker can drain all 167 slots with 167 sequential calls, after which the zero-slot guard blocks `update_canister_settings`, `change_canister_controllers`, `take_canister_snapshot`, and `load_canister_snapshot` through NNS Root for all callers including NNS Governance and SNS-W.

## Finding Description

**Entrypoint:** `canister_status` is a public `#[update]` with no caller restriction: [1](#0-0) 

**Slot accounting:** `new_management_canister_client()` wraps the real client in `LimitedOutstandingCallsManagementCanisterClient`, which decrements `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` via a `SlotLoan` RAII guard and restores it on `Drop`: [2](#0-1) [3](#0-2) 

**The trap/rollback sequence across two IC message rounds:**

- **Round 1** — `try_borrow_slot` decrements the count (167→166), inter-canister call sent to management canister. Round 1 commits.
- **Round 2 (callback)** — management canister returns `Err` (NNS Root is not a controller of the target). `_loan` is dropped (166→167 in memory), then `.unwrap()` traps. The IC rolls back all Round 2 heap state, including the slot restoration. Net committed state: count = 166.

Each attack call permanently decrements the count by 1. After 167 calls the count reaches 0.

**Zero-slot guard blocks all callers including VIPs:** The `if *available_slot_count == 0` check fires before the VIP-aware subtraction, blocking all callers regardless of `is_caller_vip`: [4](#0-3) 

**Initial slot count:** [5](#0-4) 

**Affected operations using `new_management_canister_client()`:** [6](#0-5) [7](#0-6) 

Note: `change_nns_canister` calls `change_canister()` from `ic_nervous_system_root`, which calls the management canister directly via the runtime and is **not** affected by the slot limit. This means NNS Root can still be upgraded via governance to reset the heap-resident `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` (since `canister_post_upgrade` does not restore it from stable memory, it re-initializes to 167). However, the attack is trivially repeatable after each upgrade. [8](#0-7) 

**Public interface confirmation:** [9](#0-8) 

## Impact Explanation

Once all 167 slots are drained, every call to `new_management_canister_client()` inside NNS Root returns `Err(SysTransient, "Unavailable")`. This permanently blocks (until upgrade):
- `update_canister_settings` — NNS Governance proposals to update canister settings
- `change_canister_controllers` — SNS-W operations to change canister controllers
- `take_canister_snapshot` / `load_canister_snapshot` — NNS Governance snapshot operations

This constitutes a **High** severity application/platform-level DoS of significant NNS infrastructure operations with concrete protocol harm. The DoS is recoverable via a `change_nns_canister` governance proposal (which bypasses the slot limit), but the attack is immediately repeatable after recovery, making it a persistent operational threat.

## Likelihood Explanation

The attack requires no privileges, no cycles beyond ingress fees, and no coordination. The `canister_status` endpoint is publicly documented in `root.did` and callable by any principal with an arbitrary `CanisterIdRecord`. 167 sequential ingress calls suffice; concurrency is not required. The attacker only needs to supply a canister ID not controlled by NNS Root (trivially any self-owned canister).

## Recommendation

Replace the `.unwrap()` with explicit error handling that returns an error to the caller without trapping:

```rust
#[update]
async fn canister_status(canister_id_record: CanisterIdRecord) -> Result<CanisterStatusResult, String> {
    let client = new_management_canister_client();
    client
        .canister_status(canister_id_record)
        .await
        .map(CanisterStatusResult::from)
        .map_err(|(code, msg)| format!("Management canister error {code}: {msg}"))
}
```

This ensures the `SlotLoan` drop and the function return occur in the same Round 2 execution without a trap, so the slot is always restored on commit. The Candid interface in `root.did` should be updated accordingly.

## Proof of Concept

State-machine test outline using `ic_state_machine_tests`:
1. Deploy NNS canisters via `setup_nns_canisters`.
2. Create a canister not controlled by NNS Root (e.g., via `set_up_universal_canister` with anonymous principal as controller).
3. Call `canister_status` on NNS Root with that canister's ID from the anonymous principal; assert the call returns a replica-level rejection (trap propagated as rejection).
4. Read `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` via metrics endpoint (`/metrics`) or a test hook; assert it is 166.
5. Repeat steps 3–4 a total of 167 times; assert count reaches 0.
6. Call `update_canister_settings` from NNS Governance; assert it fails with `SysTransient / Unavailable`.
7. Call `change_nns_canister` from NNS Governance; assert it still succeeds (confirming the bypass path is unaffected).

        let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

            .with_borrow_mut(|available_slot_count| {

                Ok(())
            })?;

/// Increments available_slot_count by used_slot_count when dropped.
struct SlotLoan {
    available_slot_count: &'static LocalKey<RefCell<u64>>,
    used_slot_count: u64,
}

impl Drop for SlotLoan {
    fn drop(&mut self) {
        self.available_slot_count
            .with_borrow_mut(|available_slot_count| {
                *available_slot_count = available_slot_count.saturating_add(self.used_slot_count);
            });
    }
}
