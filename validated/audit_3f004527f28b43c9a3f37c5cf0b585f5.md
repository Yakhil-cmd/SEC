### Title
`receive_base_tokens` Skips Silo Address-Whitelist Check, Freezing Bridged ETH in Non-Whitelisted Addresses - (File: `engine/src/engine.rs`)

### Summary

In Silo mode, the Aurora Engine enforces an `Address` whitelist to control which EVM addresses may receive tokens. `receive_erc20_tokens` correctly redirects tokens to a configured fallback address when the intended recipient is not whitelisted. However, the parallel function `receive_base_tokens`, which handles bridged ETH (native token) deposits via the ETH connector, performs **no whitelist check at all**. Any caller can bridge ETH to a non-whitelisted address, minting a balance there. Because `submit` also enforces the same whitelist, the minted ETH can never be spent, permanently freezing the bridged funds.

### Finding Description

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to one of two paths based on the predecessor account: [1](#0-0) 

When the predecessor is the ETH connector account, `receive_base_tokens` is called. That function unconditionally mints ETH to the parsed recipient address with no whitelist gate: [2](#0-1) 

The ERC-20 path, `receive_erc20_tokens`, explicitly checks the Silo `Address` whitelist and redirects to the fallback address when the recipient is not allowed: [3](#0-2) 

The whitelist check itself is: [4](#0-3) 

which delegates to `is_address_allowed`: [5](#0-4) 

`receive_base_tokens` never calls either of these helpers. The `Address` whitelist is therefore enforced for ERC-20 deposits but silently bypassed for native ETH deposits.

### Impact Explanation

Once ETH is minted to a non-whitelisted address, the `submit` / `submit_with_args` path enforces the same `Address` whitelist and rejects any transaction from that address: [6](#0-5) 

The minted ETH balance is therefore unspendable for as long as the address remains off the whitelist. If the Silo operator never whitelists the address (the normal expectation for a restricted Silo), the funds are permanently frozen. This constitutes **permanent freezing of user funds**, matching the High/Critical impact tier.

### Likelihood Explanation

The attack path is fully permissionless. Any NEAR account can call `ft_transfer_call` on the ETH connector contract, specifying any EVM address as the recipient in the message field. The ETH connector then calls `ft_on_transfer` on the Aurora Engine, which routes to `receive_base_tokens`. No special privilege is required. The condition (Silo mode with `Address` whitelist enabled) is the intended production configuration for Silo deployments, making this reachable in any live Silo instance.

### Recommendation

Apply the same fallback-address redirect logic to `receive_base_tokens` that already exists in `receive_erc20_tokens`. Before crediting the balance, check `silo::is_allow_receive_erc20_tokens` (or an equivalent `is_allow_receive_base_tokens` helper) and, if a fallback address is configured and the recipient is not whitelisted, redirect the mint to the fallback address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut recipient = message_data.recipient;

    // Mirror the whitelist redirect that receive_erc20_tokens already performs.
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
    {
        recipient = fallback_address;
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    Ok(None)
}
```

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a non-zero `fixed_gas` and a valid `erc20_fallback_address`.
2. Enable the `Address` whitelist: call `set_whitelist_status` with `WhitelistKind::Address, active: true`.
3. Do **not** add victim address `V` to the whitelist.
4. From any NEAR account, call `ft_transfer_call` on the ETH connector with `receiver_id = aurora`, `amount = X`, `msg = "<relayer>:<fee><V_hex>"`.
5. The ETH connector calls `ft_on_transfer` on Aurora; `predecessor == connector_account_id`, so `receive_base_tokens` is invoked.
6. `receive_base_tokens` mints `X` ETH to `V` with no whitelist check.
7. Attempt to call `submit` from `V`'s address — it is rejected with `NotAllowed` because `V` is not on the `Address` whitelist.
8. The `X` ETH balance at `V` is permanently frozen. [2](#0-1) [7](#0-6) [8](#0-7)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
