Audit Report

## Title
ckETH Minter Sequential Nonce Head-of-Line Blocking via Insufficient `max_transaction_fee` on Gas Spike - (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary

The ckETH minter assigns strictly sequential Ethereum nonces to withdrawal requests. When a sent transaction's `max_transaction_fee` cap is exceeded by a bumped gas estimate during resubmission, `create_resubmit_transactions` pushes an `InsufficientTransactionFee` error and immediately returns, leaving all higher nonces unprocessed. The caller `resubmit_transactions_batch` only logs the error and takes no corrective action. Because `sent_transactions_to_finalize` only considers nonces strictly below the on-chain finalized transaction count, and Ethereum enforces sequential nonce ordering, the stuck nonce permanently blocks finalization of every subsequent ckETH and ckERC20 withdrawal until gas prices fall or a canister upgrade is performed.

## Finding Description

**Root cause — early return in `create_resubmit_transactions`:**
At `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` L618–630, when `last_signed_tx.resubmit(current_gas_fee)` returns `Err(InsufficientTransactionFee)` for nonce N, the function pushes the error and immediately `return`s. The code comment at L589–592 explicitly acknowledges: *"We stop on the first error since if a transaction with nonce n could not be resubmitted… then the next transactions with nonces n+1, n+2, … are blocked anyway."* [1](#0-0) 

**Caller silently discards the error:**
`resubmit_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs` L242–244 matches the `Err(e)` arm with only a `log!` call. The stuck transaction remains in `sent_tx` with its original low fee cap indefinitely. [2](#0-1) 

**Finalization gated on sequential on-chain nonce count:**
`sent_transactions_to_finalize` at L661–664 filters `sent_tx` to only nonces strictly less than `finalized_transaction_count`. Because Ethereum will not mine nonce N+1 before nonce N, the finalized count never advances past N, so no subsequent withdrawal can ever be finalized. [3](#0-2) 

**Sequential nonce assignment from FIFO queue:**
`record_created_transaction` at L521–527 assigns `self.next_nonce` to each new transaction and increments it, consuming `pending_withdrawal_requests` front-to-back. [4](#0-3) 

**Fee cap enforcement in `SignedTransactionRequest::resubmit`:**
At `rs/ethereum/cketh/minter/src/tx.rs` L169–173, if `new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee()`, the function returns `Err(InsufficientTransactionFee)`. For ckERC20 withdrawals, `allowed_max_transaction_fee` is the user-supplied `max_transaction_fee` field, which is fixed at submission time. [5](#0-4) 

**No cancel, skip, or quarantine mechanism exists:**
A search across all minter source files finds no `cancel_withdrawal`, `skip_nonce`, `burn_nonce`, or equivalent function. Resolution requires a canister upgrade.

## Impact Explanation

A single ckERC20 withdrawal with a `max_transaction_fee` that becomes insufficient during a gas spike permanently halts finalization of all subsequent ckETH and ckERC20 withdrawals. Affected users cannot receive their funds. This is a concrete, sustained platform-level DoS on the ckETH/ckERC20 withdrawal pipeline — a significant Chain Fusion / ck-token security impact with direct user harm — matching the **High ($2,000–$10,000)** impact tier: *"Application/platform-level DoS… or significant Chain Fusion, ck-token… security impact with concrete user or protocol harm."*

## Likelihood Explanation

Any unprivileged user calling `withdraw_erc20` can trigger this condition by supplying a `max_transaction_fee` that is valid at submission time but insufficient during a subsequent gas spike. No privileged access, threshold corruption, or external oracle manipulation is required. Ethereum gas price spikes of 5–10× within hours are historically common. An adversary can deliberately set a minimal fee cap (just enough to pass the initial `create_transaction` check) and wait for natural congestion, or time the submission to coincide with a known high-activity event. The condition is repeatable and requires only a single malicious or unlucky withdrawal request.

## Recommendation

1. **Add a cancel-and-reimburse path for stuck nonces.** When `create_resubmit_transactions` returns `InsufficientTransactionFee` for nonce N, the minter should burn the nonce by sending a zero-value self-transfer at the current gas price (funded from the minter's ETH reserve), remove the entry from `sent_tx`, and reimburse the user's ckERC20 tokens. This unblocks nonces N+1, N+2, ….
2. **Expose an operator-callable `cancel_withdrawal` endpoint** that performs the same nonce-burn and reimbursement under NNS governance control.
3. **Enforce a minimum `max_transaction_fee` at submission time** (e.g., a configurable multiple of the current gas estimate) to reduce the probability of a fee cap being breached during normal volatility.

## Proof of Concept

1. Ethereum mainnet gas price is 10 gwei. User calls `withdraw_erc20` with `max_transaction_fee` = 650,000 gwei (barely covers 10 gwei × 65,000 gas limit). The minter creates a transaction with nonce N and sends it.
2. Gas price spikes to 15 gwei. The minter's timer calls `resubmit_transactions_batch` → `create_resubmit_transactions(latest_tx_count, new_gas_fee)`. For nonce N, `new_tx_price.max_transaction_fee()` = 975,000 gwei > 650,000 gwei = `allowed_max_transaction_fee`. The function pushes `Err(InsufficientTransactionFee)` and returns immediately.
3. `resubmit_transactions_batch` logs the error and does nothing. Nonce N remains in `sent_tx` with the original 10-gwei fee cap. Ethereum will not mine it at 15 gwei.
4. All subsequent withdrawal requests (nonces N+1, N+2, …) are in `sent_tx` but `sent_transactions_to_finalize` filters them out because `finalized_transaction_count` is still N.
5. Every subsequent timer tick re-runs `resubmit_transactions_batch`, hits the same error for nonce N, and exits. No subsequent withdrawal is ever finalized until gas prices drop below 10 gwei or a canister upgrade intervenes.

A deterministic integration test can reproduce this by: (a) creating two withdrawal requests, (b) sending both transactions, (c) simulating a gas fee increase beyond the first request's `max_transaction_fee`, (d) calling `create_resubmit_transactions`, and (e) asserting that `sent_transactions_to_finalize` returns an empty map for the second nonce even after the first nonce's transaction is confirmed on-chain.
