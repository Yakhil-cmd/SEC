### Title
Unbounded `move_to_block` Loop in Hashchain Causes Proportional NEAR Gas Increase on Every Transaction - (File: engine-hashchain/src/hashchain.rs)

### Summary
The `Hashchain::move_to_block` function iterates once per skipped NEAR block height. This loop is invoked unconditionally on every Aurora Engine transaction (via `load_hashchain` → `with_hashchain` / `with_logs_hashchain`). Because NEAR block height is a public, observable value and Aurora transactions are processed lazily (the hashchain only advances when a new transaction arrives), any period of inactivity causes the next transaction to pay for all skipped blocks. An unprivileged user who submits the first transaction after a long idle period bears the full cost of iterating through every skipped block, each iteration performing a keccak hash. This is an exact structural analog to the ZkTrieVerifier `walkTree` depth issue: user-observable external state (block height gap) controls the number of iterations of an unbounded loop executed inside a production transaction.

### Finding Description
`load_hashchain` in `engine/src/hashchain.rs` calls `hashchain.move_to_block(block_height)` whenever the stored hashchain's `current_block_height` is behind the current NEAR block height.

```
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;   // <-- unbounded loop
    }
    Ok(maybe_hashchain)
}
```

`move_to_block` in `engine-hashchain/src/hashchain.rs` iterates one step per skipped block:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(...);
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
```

Each iteration calls `compute_block_hashchain`, which performs a `keccak` over a concatenated buffer. The number of iterations equals `current_block_height - stored_block_height`, which is entirely determined by how long the Aurora Engine has been idle (no transactions submitted).

`load_hashchain` is called from both `with_hashchain` and `with_logs_hashchain`, which wrap every state-mutating entry point: `submit`, `call`, `register_relayer`, `set_upgrade_delay_blocks`, `stage_upgrade`, `factory_update`, and others.

### Impact Explanation
NEAR has a hard 200 Tgas per-transaction gas limit. Each keccak iteration in `move_to_block` consumes NEAR Wasm gas. If the Aurora Engine is idle for a sufficiently large number of NEAR blocks (NEAR produces ~1 block/second, so ~17 million blocks/year), the first transaction after the idle period will exhaust the 200 Tgas limit inside `load_hashchain` before any EVM execution occurs, causing the transaction to fail with an out-of-gas error. The submitter loses their NEAR gas fee and the transaction is permanently dropped. Funds that were supposed to be moved by that transaction are temporarily frozen until a subsequent transaction (which also pays the catch-up cost) succeeds. In the worst case (very long idle period), every subsequent transaction also fails, creating a prolonged freeze of all Aurora Engine functionality.

**Impact class: High — Temporary freezing of funds / transaction failure for any user who submits during or after an idle period.**

### Likelihood Explanation
NEAR block height is public and monotonically increasing. Any observer can determine the last Aurora transaction block and compute the current gap. A malicious actor can deliberately avoid submitting transactions to Aurora for an extended period (or simply wait for natural low-activity periods), then submit a transaction knowing it will fail due to the catch-up loop. The attacker does not need any privileged access; any EVM user can trigger this path via the standard `submit` entry point. Aurora mainnet has historically had periods of low activity, making this a realistic scenario.

### Recommendation
Cap the number of iterations in `move_to_block` per call, or restructure the hashchain to store only the last-seen block height and compute the gap lazily without iterating. A simple fix is to add a maximum-skip constant and return an error (or skip the hashchain update) if the gap exceeds it:

```rust
const MAX_BLOCK_SKIP: u64 = 1000; // tune to gas budget

pub fn move_to_block(&mut self, next_block_height: u64) -> Result<(), BlockchainHashchainError> {
    if next_block_height <= self.current_block_height {
        return Err(BlockchainHashchainError::BlockHeightIncorrect);
    }
    if next_block_height - self.current_block_height > MAX_BLOCK_SKIP {
        return Err(BlockchainHashchainError::BlockSkipTooLarge);
    }
    while self.current_block_height < next_block_height {
        ...
    }
    Ok(())
}
```

Alternatively, store empty-block hashes as a closed-form formula (chain of keccaks can be precomputed or skipped for empty blocks) to avoid the linear loop entirely.

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled (`start_hashchain`).
2. Submit one transaction to establish the hashchain at block height `H`.
3. Wait (or simulate) `N` NEAR blocks without submitting any Aurora transaction. `N` can be as large as desired since NEAR block height is not controlled by Aurora.
4. Submit any Aurora EVM transaction (e.g., a simple ETH transfer via `submit`).
5. Observe that `load_hashchain` calls `move_to_block(H + N)`, which iterates `N` times, each calling `compute_block_hashchain` (one keccak per iteration).
6. For sufficiently large `N` (empirically, tens of thousands of blocks at current NEAR gas pricing), the transaction exhausts the 200 Tgas limit and fails.

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}
