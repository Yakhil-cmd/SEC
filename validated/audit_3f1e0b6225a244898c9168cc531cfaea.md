Audit Report

## Title
ckBTC Minter Allows Withdrawal to User Deposit Address, Enabling Circular Re-Minting That Breaks 1:1 BTC Backing - (`File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

The `retrieve_btc` and `retrieve_btc_with_approval` endpoints only block withdrawals to the minter's own main Bitcoin address. They do not block withdrawals to user deposit addresses, which are also derived from the same minter ECDSA key. An attacker can withdraw ckBTC to their own deposit address, causing the minter to create new UTXOs at that address with new outpoints. Since `minted_outpoints` only tracks previously minted outpoints, a subsequent `update_balance` call treats these new UTXOs as fresh deposits and mints ckBTC again — for BTC that was already accounted for — breaking the 1:1 backing invariant.

## Finding Description

**Root cause — incomplete destination check in `retrieve_btc`:**

In `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` lines 155–160, the only forbidden-destination guard is:

```rust
let main_address_str = state::read_state(|s| runtime.derive_minter_address_str(s));
if args.address == main_address_str {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

The same incomplete check exists in `retrieve_btc_with_approval` at lines 253–258. `derive_minter_address_str` (in `rs/bitcoin/ckbtc/minter/src/lib.rs` lines 1797–1811) only derives the address for `Account { owner: minter_canister_id, subaccount: None }` — the minter's main account. User deposit addresses, derived for `Account { owner: user_principal, subaccount: ... }` via `account_to_p2wpkh_address_from_state`, are never checked.

**Why `minted_outpoints` does not prevent re-minting:**

`minted_outpoints` (`rs/bitcoin/ckbtc/minter/src/state.rs` line 587) is a `BTreeSet<OutPoint>` populated in `add_utxos` (line 786) when a UTXO is minted. It records the outpoints of UTXOs that were deposited and minted. When the minter executes a withdrawal transaction to address A, it creates an entirely new Bitcoin transaction with new txid and new vout values — new outpoints that are not in `minted_outpoints`.

**Why `processable_utxos_for_account` does not prevent re-minting:**

The `is_known` closure in `processable_utxos_for_account` (`state.rs` lines 1368–1378) checks `utxos_state_addresses` and `finalized_utxos`. The new UTXOs created by the withdrawal transaction were never added to either structure for the user's account, so they are classified as new processable UTXOs.

**Full exploit path:**

1. Alice calls `get_btc_address(alice_principal, None)` → deposit address A.
2. Alice sends X BTC to A; calls `update_balance` → minter mints X ckBTC, records UTXO_1 outpoint in `minted_outpoints`.
3. Alice calls `retrieve_btc(address=A, amount=X-fees)` → passes the main-address check; minter burns ckBTC and sends BTC to A, creating UTXO_2 (new txid, new vout, new outpoint).
4. Alice calls `update_balance` → minter queries UTXOs at A, finds UTXO_2; UTXO_2 is not in `minted_outpoints`, not in `utxos_state_addresses`, not in `finalized_utxos` → minter mints ckBTC again for UTXO_2.
5. Steps 3–4 repeat indefinitely, inflating ckBTC supply without any net BTC entering minter custody.

The BTC checker (`check_address` in `retrieve_btc`, `check_utxo` in `update_balance`) is designed to detect tainted/sanctioned UTXOs, not to detect the minter's own withdrawal outputs, so it does not block this path.

## Impact Explanation

Each cycle produces ckBTC without any net BTC entering the minter's custody, directly violating the protocol invariant that ckBTC supply equals BTC held by the minter. This constitutes **illegal minting** of an in-scope chain-key asset (ckBTC). The attack is unbounded and repeatable, enabling protocol insolvency well above $1M. This matches the Critical impact class: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets, especially over $1M."*

## Likelihood Explanation

Any unprivileged user can execute this attack. No special access, governance majority, threshold key compromise, or privileged role is required. The only cost is Bitcoin transaction fees (paid twice per cycle). The attacker directly controls the `address` argument to the public `retrieve_btc` ingress endpoint. The attack is repeatable with no cooldown beyond Bitcoin confirmation times.

## Recommendation

In both `retrieve_btc` and `retrieve_btc_with_approval`, after parsing the destination `BitcoinAddress`, verify that it does not correspond to any address derivable from the minter's ECDSA key. At minimum, derive the caller's own deposit address (via `account_to_p2wpkh_address_from_state`) and reject any withdrawal target that matches it. A more complete fix would reject any address whose derivation path is rooted in the minter's ECDSA key hierarchy. Alternatively, maintain a registry of all known user deposit addresses and reject withdrawals to any of them.

## Proof of Concept

```
1. Alice calls get_btc_address(alice_principal, None) → address A
2. Alice sends 1 BTC to address A on Bitcoin
3. Alice calls update_balance(alice_principal, None)
   → Minter mints ~0.9999 ckBTC; UTXO_1 outpoint added to minted_outpoints
4. Alice calls retrieve_btc(address=A, amount=0.9998)
   → Main-address check passes (A != minter main address)
   → Minter burns 0.9998 ckBTC; sends BTC to A creating UTXO_2 (new outpoint)
5. Alice calls update_balance(alice_principal, None)
   → UTXO_2 not in minted_outpoints, utxos_state_addresses, or finalized_utxos
   → Minter mints ~0.9997 ckBTC again for UTXO_2
6. Repeat steps 4–5 to inflate ckBTC supply unboundedly

Reproducible as a PocketIC integration test:
- Initialize minter + ledger + mock Bitcoin canister
- Execute steps 1–5 above
- Assert tokens_minted > tokens_burned + initial_btc_deposit after step 5
- Assert minted_outpoints contains both UTXO_1 and UTXO_2 outpoints
```
