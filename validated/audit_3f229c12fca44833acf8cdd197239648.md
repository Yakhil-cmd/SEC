### Title
Single-Principal Guard Pool Exhaustion via `retrieve_btc_with_approval` Subaccount Enumeration — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

### Summary

`retrieve_btc_with_approval` uses the caller's `from_subaccount` argument as the guard key, while `retrieve_btc` always uses `subaccount: None`. Because the shared `retrieve_btc_accounts` pool has a hard cap of `MAX_CONCURRENT = 100`, a single unprivileged principal can exhaust all 100 slots by sending 100 concurrent calls with distinct subaccounts, blocking every other principal from withdrawing via either endpoint.

### Finding Description

**The asymmetry:**

`retrieve_btc` always guards with the caller's default account: [1](#0-0) 

`retrieve_btc_with_approval` guards with the caller-supplied `from_subaccount`: [2](#0-1) 

**The guard pool:**

`MAX_CONCURRENT = 100`. Once the set reaches 100 entries, every new call — regardless of caller — returns `TooManyConcurrentRequests`: [3](#0-2) 

The guard is held for the entire async lifetime of the call and released only on `Drop`: [4](#0-3) 

**Why 100 slots are reachable from one principal:**

The guard is acquired at line 263, *before* the first inter-canister await (`check_address` at line 283). The attacker does not need any ckBTC balance. Each call suspends at `check_address(...).await`, holding its guard slot across at least one IC round-trip: [5](#0-4) 

An attacker sends 100 calls with distinct non-default subaccounts (e.g., `[1u8;32]` through `[100u8;32]`). Each call:
1. Passes the mode check (no privilege required).
2. Passes `init_ecdsa_public_key().await`.
3. Acquires a unique guard slot (distinct `Account` key per subaccount).
4. Suspends at `check_address`, holding the slot.

After 100 such in-flight calls, `retrieve_btc_accounts.len() == 100`. Every subsequent call from any principal hits the `>= MAX_CONCURRENT` branch.

**Note on `None` vs `Some([0u8;32])` normalization:** The existing guard test confirms these two values map to the same `Account` key, so the attacker uses 100 distinct non-zero subaccounts — of which there are 2^256 − 1 available: [6](#0-5) 

### Impact Explanation

All principals are blocked from initiating ckBTC → BTC withdrawals via both `retrieve_btc` and `retrieve_btc_with_approval` for the duration of the attack. The attacker can sustain the DoS by re-submitting calls as old ones complete (each new batch costs only IC ingress fees, no ckBTC). This constitutes constrained subnet-level availability loss for the ckBTC withdrawal flow.

### Likelihood Explanation

The attack requires no privileged access, no funds, and no coordination. It is executable by any IC identity with the ability to send ingress messages. The only cost is IC cycle fees for 100 concurrent ingress calls. The attack is repeatable and self-sustaining.

### Recommendation

Enforce a **per-principal** slot limit in addition to the global cap. For example, count how many entries in `retrieve_btc_accounts` share the same `owner` and reject if that count exceeds a per-principal threshold (e.g., 1 or a small constant). Alternatively, normalize the guard key for `retrieve_btc_with_approval` to `Account { owner: caller, subaccount: None }` (matching `retrieve_btc`), accepting that a caller can only have one in-flight withdrawal at a time regardless of subaccount.

### Proof of Concept

```rust
// Pseudocode unit test
fn test_guard_exhaustion_by_single_principal() {
    // Attacker holds 100 in-flight retrieve_btc_with_approval calls,
    // each with a distinct non-zero subaccount, all suspended at check_address.
    let attacker = principal(1);
    let victim   = principal(2);

    let guards: Vec<_> = (1u8..=100).map(|i| {
        retrieve_btc_guard(Account {
            owner: attacker,
            subaccount: Some([i; 32]),
        }).expect("slot should be available")
    }).collect();

    // Guard pool is now full (100/100).
    // Victim attempts retrieve_btc (subaccount: None).
    let result = retrieve_btc_guard(Account {
        owner: victim,
        subaccount: None,
    });
    assert_eq!(result.unwrap_err(), GuardError::TooManyConcurrentRequests);
    // All 100 slots consumed by one attacker principal.
}
```

This maps directly to the existing test infrastructure in `guard.rs` — the `guard_prevents_more_than_max_concurrent_accounts` test already demonstrates that 50 subaccounts of a single principal can fill half the pool: [7](#0-6)
