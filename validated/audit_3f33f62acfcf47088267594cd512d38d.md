### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens before calling the `exitToNear` / `exitToEthereum` precompile via inline assembly. The assembly `call` return value (`res`) is **never checked**. If the precompile call fails for any reason, the burn is not reverted, and the user's tokens are permanently destroyed with no corresponding NEP-141 transfer on the NEAR side.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) [2](#0-1) 

The `exitToNear` precompile (`ExitToNear::run`) returns `Err(ExitError::Other(...))` in multiple reachable conditions:

- `ERR_TARGET_TOKEN_NOT_FOUND` — the calling ERC-20 address is not registered in the NEP-141 map (`get_nep141_from_erc20` fails).
- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — the `recipient` bytes do not form a valid NEAR account ID.
- `ERR_INVALID_AMOUNT` — the amount exceeds `u128::MAX`.
- `ERR_INVALID_INPUT` — input size is out of bounds. [3](#0-2) [4](#0-3) 

When the precompile returns an error, SputnikVM sets the assembly `call` result to `0` (failure) and returns control to the Solidity caller. Because `res` is never inspected, the Solidity function does **not** revert. The `_burn` that already executed is final — the ERC-20 tokens are gone, and no NEAR-side promise is ever created.

The same pattern exists in `withdrawToEthereum` for the `exitToEthereum` precompile: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A user who calls `withdrawToNear` or `withdrawToEthereum` with any input that causes the precompile to fail (e.g., a malformed NEAR account ID in `recipient`, or an amount exceeding `u128::MAX`) will have their ERC-20 tokens permanently burned with no corresponding NEP-141 transfer. The funds are irrecoverably destroyed. This is **permanent freezing/loss of user funds** — Critical severity.

---

### Likelihood Explanation

The trigger is reachable by any unprivileged EVM user who:
1. Holds a balance of a bridged ERC-20 token.
2. Calls `withdrawToNear` with a `recipient` that is not a valid NEAR account ID (e.g., an empty byte string, a string exceeding 64 characters, or one containing invalid characters).

No special privileges are required. The `recipient` parameter is a raw `bytes` argument with no on-chain validation before the burn. A user mistake or a malicious contract composing the call is sufficient.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }
    }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy a bridged ERC-20 token via `deploy_erc20_token` so it is registered in the NEP-141 map.
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear(bytes(""), amount)` — an empty byte string is not a valid NEAR account ID.
4. Inside `withdrawToNear`: `_burn(alice, amount)` executes successfully.
5. The precompile `ExitToNear::run` is called; `parse_recipient("")` fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID`.
6. The assembly `call` returns `0`; `res` is never checked; the function returns without reverting.
7. `alice`'s ERC-20 balance is now `0`; no NEP-141 tokens were transferred; funds are permanently lost. [7](#0-6) [1](#0-0)

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
