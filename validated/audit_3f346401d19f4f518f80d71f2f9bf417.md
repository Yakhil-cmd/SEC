Audit Report

## Title
Stale Cached ICP/XDR Conversion Rate Used in Cycle Minting Without Freshness Check - (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles` function in the Cycles Minting Canister reads `state.icp_xdr_conversion_rate` and applies it to every cycle-minting operation without checking the `timestamp_seconds` field for staleness. When the Exchange Rate Canister (XRC) is unavailable, the heartbeat-driven refresh fails silently and the cached rate ages indefinitely. An unprivileged attacker who observes XRC downtime coinciding with a falling ICP market price can call `notify_top_up` repeatedly to obtain cycles at the stale, inflated rate, minting more cycles than the deposited ICP is worth at current market price.

## Finding Description
`tokens_to_cycles` at `rs/nns/cmc/src/main.rs` lines 1900–1922 reads `state.icp_xdr_conversion_rate` and branches only on `None` vs `Some`:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...)
        }
    })
}
```

The `IcpXdrConversionRate` struct carries a `timestamp_seconds` field (declared at `main.rs` line 217–218) that is never consulted here. All three public minting entry points call this function first: `process_create_canister` (line 1932), `process_mint_cycles` (line 1965), and `process_top_up` (line 1991).

The rate is refreshed by `update_exchange_rate` in `exchange_rate_canister.rs` (line 236), scheduled every `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS` (line 16). On XRC failure, `schedule_next_attempt` (line 133) reschedules for the next minute (lines 153–160), but the cached `icp_xdr_conversion_rate` is never cleared or age-bounded. The `UpdateExchangeRateGuard` (lines 86–124) only prevents concurrent calls; it imposes no maximum age on the stored rate before it is consumed by minting.

**Exploit flow:**
1. XRC becomes unavailable (subnet degradation, canister upgrade, persistent error such as `StablecoinRateTooFewRates`). The CMC heartbeat fails to update `icp_xdr_conversion_rate` but does not clear it.
2. ICP market price falls (e.g., 20%) while the XRC remains unavailable. The cached rate still reflects the pre-drop price.
3. Attacker calls `notify_top_up` with a valid ICP ledger block. `tokens_to_cycles` reads the stale rate and mints cycles at the old, higher rate.
4. Attacker receives ~20% more cycles than the current ICP value warrants. Repeat until the global hourly rate limiter (`DEFAULT_CYCLES_LIMIT = 150e15` cycles/hour, line 83) is exhausted or XRC recovers.

The existing guard — the `None` check — is insufficient because a populated but arbitrarily old rate passes it unconditionally.

## Impact Explanation
This is illegal cycle minting: cycles are created in excess of the ICP value burned, violating the protocol's conservation invariant. The global hourly rate limiter (`DEFAULT_CYCLES_LIMIT = 150e15` cycles/hour) bounds but does not prevent the attack. At a 20% stale-rate premium, an attacker can extract ~30e15 excess cycles per hour. Sustained over multiple hours of XRC downtime, the cumulative over-minting can reach significant financial value. This matches the allowed High impact: "Significant XRC... security impact with concrete user or protocol harm," and potentially Critical if the stale window is long and price volatility is high enough to push total excess minting above $1M.

## Likelihood Explanation
The XRC is a canister on a system subnet. Transient subnet degradation, canister upgrades, and persistent XRC error responses (e.g., `StablecoinRateTooFewRates`, `CryptoBaseAssetNotFound`) are documented operational scenarios. The CMC retries every minute on failure but never ages out the stale rate. Any unprivileged principal holding ICP can call `notify_top_up` — no special role is required. The attack window is bounded by XRC downtime duration and ICP price volatility during that window. Both conditions have occurred historically on the IC.

## Recommendation
Add a maximum-age check inside `tokens_to_cycles`. If `now_timestamp_seconds() - icp_xdr_conversion_rate.timestamp_seconds` exceeds a configured threshold (e.g., 30 minutes), return a retriable `NotifyError` rather than minting at the stale rate. This mirrors the pattern already used in `should_refresh_xdr_rate` in `rs/nns/governance/src/governance.rs` (lines 6336–6347), which checks `timestamp_seconds` staleness before using the XDR rate for node-provider rewards. A constant such as `MAX_RATE_AGE_SECONDS: u64 = 30 * ONE_MINUTE_SECONDS` would be appropriate, with the error variant `NotifyError::Other` carrying a message indicating the rate is temporarily unavailable.

## Proof of Concept
1. Deploy a local replica with the CMC and a mock XRC that returns `ExchangeRateError::StablecoinRateTooFewRates`.
2. Seed `icp_xdr_conversion_rate` with a timestamp 60 minutes in the past and `xdr_permyriad_per_icp = 100_000` (10 XDR/ICP).
3. Set the current "market" rate to 80_000 (8 XDR/ICP, a 20% drop) — but since the XRC mock always errors, the CMC never updates.
4. Call `notify_top_up` with 1 ICP. Observe that `tokens_to_cycles` returns cycles computed at 100_000 permyriad rather than 80_000.
5. Assert that the minted cycles exceed what 1 ICP at the current rate would produce by ~20%.
6. Confirm that no error is returned and the ICP is burned — the over-minting is permanent.

The root cause is confirmed at `rs/nns/cmc/src/main.rs` lines 1900–1922: no `timestamp_seconds` freshness check exists before the rate is consumed.
