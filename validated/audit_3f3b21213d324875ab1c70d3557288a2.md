### Title
`ExitToEthereum` Precompile Has No Error-Recovery Callback — Permanent Fund Loss if `withdraw` Promise Fails - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns user funds (ETH or ERC-20 tokens) on the Aurora/EVM side and then schedules a single NEAR promise to call `withdraw` on the eth-connector NEP-141 contract. Unlike `ExitToNear`, which attaches an `exit_to_near_precompile_callback` to refund the user when the downstream promise fails, `ExitToEthereum` schedules only a bare `PromiseArgs::Create` with no error-handling callback. If the `withdraw` promise fails for any reason, the user's funds are permanently destroyed with no recovery path.

---

### Finding Description

**Step 1 — EVM-side deduction (irreversible).**
When a user calls the `ExitToEthereum` precompile:
- For ETH exit (flag `0x00`): `context.apparent_value` (the ETH sent) is consumed from the caller's EVM balance.
- For ERC-20 exit (flag `0x01`): the ERC-20 contract has already burned the tokens before calling the precompile.

Both actions are committed to EVM state as part of the same NEAR transaction that schedules the outbound promise.

**Step 2 — NEAR-side promise (no callback).**
The precompile constructs a single `PromiseArgs::Create` targeting the eth-connector's `withdraw` method:

```rust
// engine-precompiles/src/native.rs  lines 977-990
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

There is no `.then(callback)` attached. The promise log is emitted and processed by `filter_promises_from_logs` in `engine/src/engine.rs`, which schedules it as a fire-and-forget call.

**Step 3 — Contrast with `ExitToNear`.**
`ExitToNear` conditionally wraps its outbound promise in a `PromiseArgs::Callback` that calls `exit_to_near_precompile_callback` on the engine. That callback invokes `engine::refund_on_error` to re-mint the burned tokens or restore the ETH balance when the downstream call fails:

```rust
// engine-precompiles/src/native.rs  lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
```

`ExitToEthereum` has no equivalent guard — not even behind a feature flag.

**Conditions under which `withdraw` can fail:**
- The eth-connector contract is paused or upgraded to a breaking version.
- The attached gas (`WITHDRAWAL_GAS = 100 TGas`) is insufficient for the connector's internal logic.
- The attached balance (1 yoctoNEAR) is rejected by a future connector version.
- The eth-connector account is deleted or migrated.
- Any panic inside the connector's `withdraw` implementation.

None of these conditions are under the user's control, and none of them prevent the EVM-side burn from being committed.

---

### Impact Explanation

**Critical — Permanent freezing / direct loss of user funds.**

The user's ETH or ERC-20 tokens are irreversibly destroyed on the Aurora EVM side. Because there is no callback, the engine never learns that the `withdraw` failed, and there is no on-chain mechanism for the user to reclaim their funds. The funds are gone from Aurora and never arrive on Ethereum L1.

---

### Likelihood Explanation

**Medium.** Under normal operation the eth-connector's `withdraw` succeeds. However:
- The eth-connector is a separate contract that can be paused, upgraded, or replaced independently of the engine.
- Aurora has historically separated the connector from the engine, making independent failure realistic.
- A user bridging a large amount at the moment the connector is paused (e.g., during an emergency upgrade) would silently lose all funds with no retry or refund option.
- The scenario requires no attacker — it is triggered by ordinary user bridge activity coinciding with any connector-side failure.

---

### Recommendation

Mirror the `ExitToNear` pattern for `ExitToEthereum`:

1. Add a `exit_to_ethereum_precompile_callback` method on the engine that checks the promise result.
2. If the `withdraw` promise failed, call `refund_on_error` to restore the user's EVM balance (re-mint ERC-20 or credit ETH).
3. Wrap the `withdraw_promise` in a `PromiseArgs::Callback` pointing to that new callback, exactly as `ExitToNear` does at lines 473–483 of `engine-precompiles/src/native.rs`.

---

### Proof of Concept

1. User holds 1 ETH on Aurora and calls `ExitToEthereum` (flag `0x00`, recipient = their Ethereum address).
2. The EVM deducts 1 ETH from the user's Aurora balance. The EVM transaction succeeds.
3. The engine schedules `PromiseArgs::Create { method: "withdraw", ... }` against the eth-connector.
4. The eth-connector is paused at this moment (or runs out of gas, or panics for any reason). The `withdraw` promise fails.
5. Because there is no callback, the engine receives no notification of the failure.
6. The user's Aurora balance is 0 ETH. Their Ethereum address received nothing. The 1 ETH is permanently lost.
7. There is no on-chain function the user can call to recover the funds.
