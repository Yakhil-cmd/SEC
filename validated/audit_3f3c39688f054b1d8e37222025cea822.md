Audit Report

## Title
Unprivileged Caller Can Force-Register Any Principal as a Swap Participant via Unchecked `buyer` Field in `refresh_buyer_tokens` — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` ingress endpoint accepts an arbitrary `buyer` principal without verifying it matches the caller. An attacker can deposit ICP into any victim's subaccount of the swap canister and then call `refresh_buyer_tokens` with `buyer` set to the victim's principal, causing the swap canister to credit the victim's `BuyerState` and register them as a direct participant — without the victim's knowledge or consent. The ticket system provides no secondary defense because it is explicitly bypassed when no ticket exists for the buyer.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 130–134, the handler resolves the effective buyer principal as follows:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller check
};
```

The parsed principal `p` is passed verbatim to `refresh_buyer_token_e8s`. Inside that function (`rs/sns/swap/src/swap.rs` lines 1152–1163), the ICP ledger balance is queried for `Account { owner: swap_canister, subaccount: principal_to_subaccount(&buyer) }`. Since ICP ledger transfers are unrestricted, any principal can pre-fund a victim's subaccount. The ticket check at lines 1250–1272 is explicitly non-mandatory: the comment at line 1271 states "If there exists no ticket for the buyer, the payment flow will simply ignore the ticket." If the balance meets `min_participant_icp_e8s`, the victim is unconditionally inserted into `self.buyers` at lines 1274–1288 and registered as a direct participant. The `confirmation_text` field compounds this: if the SNS configured a `confirmation_text`, the attacker supplies it in the call, satisfying the `validate_confirmation_text` gate at line 1150 and allowing participation to proceed on the victim's behalf.

## Impact Explanation
This is a **High** severity finding. The SNS swap framework is explicitly in scope. Concrete harms include: (1) forced direct participation in a token sale without the victim's consent, consuming their participation slot up to `max_participant_icp_e8s`; (2) if the swap commits, the victim's ICP is swept to SNS governance and the victim receives unwanted SNS neurons they cannot refuse; (3) an attacker registering many victim principals each at `min_participant_icp_e8s` can push `direct_participation_icp_e8s` toward the minimum ICP target, causing the swap to commit when it otherwise would have aborted — directly manipulating the swap outcome. The attacker permanently loses the deposited ICP in all scenarios (it either goes to SNS governance on commit, or is recoverable only by the victim via `error_refund_icp` on abort), making this a costly but executable attack for a motivated adversary with a stake in the swap's outcome. This matches the allowed impact: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The attack requires only a standard ICP ledger transfer (publicly available to any principal) and a single ingress call to `refresh_buyer_tokens` with a non-empty `buyer` field. No privileged access, key material, governance majority, or social engineering is required. The swap canister is a public ingress endpoint reachable by any IC principal. The ticket system, the only plausible secondary defense, is explicitly skipped when no ticket exists. The attack is repeatable across any number of victim principals during the swap's open period.

## Recommendation
In `rs/sns/swap/canister/canister.rs`, enforce that the resolved buyer principal equals the caller:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let parsed = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(parsed, caller, "buyer must match caller");
        parsed
    };
    // ...
}
```

Alternatively, remove the `buyer` field from `RefreshBuyerTokensRequest` entirely and always derive the buyer from the caller, since the field's only documented purpose is "if not specified, the caller is used."

## Proof of Concept
```
// Preconditions: swap is OPEN, victim = Principal V, attacker = Principal A
// min_participant_icp_e8s = 1_000_000

// Step 1 (attacker): Transfer ICP to victim's subaccount of the swap canister
icp_ledger.transfer({
    to: Account {
        owner: swap_canister_id,
        subaccount: principal_to_subaccount(V),
    },
    amount: 1_000_000,
    from: attacker_account,
})

// Step 2 (attacker): Call refresh_buyer_tokens with buyer = victim
swap_canister.refresh_buyer_tokens({
    buyer: V.to_text(),
    confirmation_text: Some("<exact SNS confirmation text>"),
})

// Postcondition: swap.buyers[V].amount_icp_e8s == 1_000_000
// V is a registered participant; V never called anything.
// Verifiable via: get_buyer_state({ principal_id: Some(V) }).buyer_state.is_some() == true
```

A deterministic integration test using PocketIC or the existing `SpyLedger` / `mock_stub` test infrastructure (as used in `rs/sns/swap/tests/swap.rs`) can reproduce this by calling `refresh_buyer_token_e8s` with a `buyer` principal that differs from any simulated caller and asserting the buyer appears in `swap.buyers`.
