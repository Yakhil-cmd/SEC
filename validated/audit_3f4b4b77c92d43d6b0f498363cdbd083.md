Audit Report

## Title
Missing Anonymous Principal Validation in `update_balance` Allows ckBTC Minting to Anonymous Account - (File: `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
The `update_balance` endpoint resolves the mint destination as `args.owner.unwrap_or(caller)` but only guards against the resolved owner equaling the minter's own canister ID — not `Principal::anonymous()`. Because the IC protocol permits unsigned (anonymous) ingress update calls, any ckBTC minted to the anonymous principal's account is immediately spendable by any party, making it an effectively public pool. The sibling endpoint `get_btc_address` already contains the correct guard; `update_balance` does not.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/main.rs` (L196–200), the outer handler calls `check_anonymous_caller()`, which only inspects `msg_caller()` — the actual ingress sender — not the `args.owner` field supplied in the request body.

Inside `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` (L149–151), the only guard on the resolved owner is:

```rust
if args.owner.unwrap_or(caller) == runtime.id() {
    ic_cdk::trap("cannot update minter's balance");
}
```

`Principal::anonymous()` is not `runtime.id()`, so it passes. The resolved account is then constructed at L164–167:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),   // anonymous passes through
    subaccount: args.subaccount,
};
```

By contrast, `rs/bitcoin/ckbtc/minter/src/updates/get_btc_address.rs` (L31–35) explicitly rejects this case:

```rust
assert_ne!(
    owner,
    Principal::anonymous(),
    "the owner must be non-anonymous"
);
```

The IC Interface Specification permits anonymous ingress messages (no `sender_pubkey`/`sender_sig` required), confirmed by production test code in `rs/tests/execution/general_execution_tests/input_deduplication.rs` (L61–82) where anonymous-sender update calls are accepted with HTTP 202. The ICRC-1 ledger's own test suite (`rs/ledger_suite/tests/sm-tests/src/lib.rs`, L652–686, `test_anonymous_transfers`) explicitly verifies that `icrc1_transfer` calls originating from `Principal::anonymous()` succeed.

## Impact Explanation
An attacker can cause ckBTC to be minted into `Account { owner: Principal::anonymous(), subaccount: None }`. Because the IC allows unsigned update calls from the anonymous principal, any third party can immediately call `icrc1_transfer` from that account and redirect the tokens to an attacker-controlled address. This constitutes **illegal minting to a publicly drainable account**, breaking the 1:1 BTC↔ckBTC conservation guarantee for the depositor and enabling direct theft of chain-key assets. This matches the allowed High impact: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm"* ($2,000–$10,000).

## Likelihood Explanation
The BTC deposit address for `Principal::anonymous()` is deterministically derivable off-chain from the minter's public ECDSA key (which is publicly known). The attacker needs only to fund the anonymous BTC address (their own cost) and then call `update_balance` from any non-anonymous principal — no privileged role, no key material, no social engineering. The inconsistency with `get_btc_address` makes the gap non-obvious. Likelihood is **medium**: the attacker bears an upfront BTC cost, but the attack is fully self-contained and repeatable, and the payoff scales with any BTC that accumulates at the anonymous address (including deposits from deceived users).

## Recommendation
After resolving the owner in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, add the same guard that exists in `get_btc_address`:

```rust
let resolved_owner = args.owner.unwrap_or(caller);
if resolved_owner == Principal::anonymous() {
    ic_cdk::trap("the owner must be non-anonymous");
}
```

This should be placed immediately after the existing `runtime.id()` check at L149–151, before `caller_account` is constructed at L164.

## Proof of Concept
1. Derive the BTC deposit address for `Principal::anonymous()` off-chain using the minter's public ECDSA key and the standard derivation path.
2. Send any amount of BTC to that address and wait for `min_confirmations`.
3. From any non-anonymous principal `P`, call:
   ```
   update_balance({ owner: Some(Principal::anonymous()), subaccount: None })
   ```
   `check_anonymous_caller()` passes because `msg_caller() == P`. The resolved owner is `Principal::anonymous()`, which is not `runtime.id()`, so no trap fires. The minter mints ckBTC to `Account { owner: Principal::anonymous(), subaccount: None }`.
4. Submit an unsigned (anonymous) ingress update call to the ckBTC ledger:
   ```
   icrc1_transfer({
       from_subaccount: None,
       to: Account { owner: <attacker>, subaccount: None },
       amount: <minted_amount - fee>,
       ...
   })
   ```
   The ledger accepts this call (as demonstrated by `test_anonymous_transfers`), and the ckBTC is transferred to the attacker.

A deterministic integration test using PocketIC or `StateMachine` can reproduce this end-to-end by: (a) installing the minter and ckBTC ledger, (b) crediting the anonymous BTC address with a mock UTXO, (c) calling `update_balance` with `owner: Some(Principal::anonymous())` from a test principal, and (d) asserting the anonymous ledger balance is non-zero, then calling `icrc1_transfer` as anonymous and asserting the attacker balance increases. pub fn test_anonymous_transfers<T>(ledger_wasm: Vec<u8>, encode_init_args: fn(InitArgs) -> T)
where
    T: CandidType,
{
    const INITIAL_BALANCE: u64 = 10_000_000;
    const TRANSFER_AMOUNT: u64 = 1_000_000;
    let p1 = PrincipalId::new_user_test_id(1);
    let anon = PrincipalId::new_anonymous();
    let (env, canister_id) = setup(
        ledger_wasm,
        encode_init_args,
        vec![
            (Account::from(p1.0), INITIAL_BALANCE),
            (Account::from(anon.0), INITIAL_BALANCE),
        ],
    );

    assert_eq!(INITIAL_BALANCE * 2, total_supply(&env, canister_id));
    assert_eq!(INITIAL_BALANCE, balance_of(&env, canister_id, p1.0));
    assert_eq!(INITIAL_BALANCE, balance_of(&env, canister_id, anon.0));

    // Transfer to the account of the anonymous principal
    println!("transferring to the account of the anonymous principal");
    transfer(&env, canister_id, p1.0, anon.0, TRANSFER_AMOUNT).expect("transfer failed");

    // Transfer from the account of the anonymous principal
    println!("transferring from the account of the anonymous principal");
    transfer(&env, canister_id, anon.0, p1.0, TRANSFER_AMOUNT).expect("transfer failed");

    assert_eq!(
        INITIAL_BALANCE * 2 - FEE * 2,
        total_supply(&env, canister_id)
    );
    assert_eq!(INITIAL_BALANCE - FEE, balance_of(&env, canister_id, p1.0));
    assert_eq!(INITIAL_BALANCE - FEE, balance_of(&env, canister_id, anon.0));
