### Title
Permanent ckBTC Loss Without Reimbursement When `BuildTxError::AmountTooLow` Occurs in `submit_pending_requests` — (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

---

### Summary

The ckBTC minter enforces a minimum withdrawal amount in `retrieve_btc` using a **stale** cached value (`fee_based_retrieve_btc_min_amount`), which was computed from the last periodic `estimate_fee_per_vbyte` call. When `submit_pending_requests` later runs, it fetches a **fresh** fee estimate. If Bitcoin network fees have spiked between these two points, `build_unsigned_transaction` can return `BuildTxError::AmountTooLow`, and the minter finalizes the request as `FinalizedStatus::AmountTooLow` **without issuing any reimbursement mint**. The user's ckBTC has already been burned and is permanently lost.

---

**Step 1 — Stale minimum check in `retrieve_btc`:**

`retrieve_btc` reads `s.fee_based_retrieve_btc_min_amount` from state, which was last written by `estimate_fee_per_vbyte`: [1](#0-0) 

This value is computed from the median fee at the time of the last periodic fee refresh: [2](#0-1) 

At the minimum fee floor of 1,500 millisat/byte (Mainnet), `fee_based_minimum_withdrawal_amount` returns `retrieve_btc_min_amount` (e.g., 100,000 sats): [3](#0-2) [4](#0-3) 

**Step 2 — ckBTC is burned before any transaction is built:**

After passing the stale minimum check, `retrieve_btc` burns the user's ckBTC unconditionally and enqueues the request: [5](#0-4) 

**Step 3 — Fresh fee estimate in `submit_pending_requests` can be much higher:**

`submit_pending_requests` calls `estimate_fee_per_vbyte` again for a fresh estimate before building the transaction: [6](#0-5) 

If Bitcoin fees have spiked (e.g., from 1.5 sat/vbyte to 1000 sat/vbyte — a scenario that has occurred during Ordinals/BRC-20 congestion), `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`.

**Step 4 — `AmountTooLow` path silently drops the request with NO reimbursement:** [7](#0-6) 

`remove_retrieve_btc_request` only records an event and pushes to finalized requests — it does **not** schedule any reimbursement: [8](#0-7) 

**Step 5 — Contrast with `InvalidTransaction` path which DOES reimburse:** [9](#0-8) 

The `WithdrawalReimbursementReason` enum has no `AmountTooLow` variant — reimbursement is structurally impossible for this path: [10](#0-9)

### Impact Explanation

A user who submits `retrieve_btc` at exactly `fee_based_retrieve_btc_min_amount` during a period of low fees will have their ckBTC burned. If Bitcoin network fees spike before `submit_pending_requests` processes the batch, the request is finalized as `AmountTooLow` and the burned ckBTC is **permanently lost** — no reimbursement mint is issued. The `reimbursement_account` field is populated on the request but is never used in this code path.

---

### Likelihood Explanation

- The minimum fee floor is 1,500 millisat/byte (1.5 sat/vbyte), yielding a minimum withdrawal of 100,000 sats at low fees.
- A single-input P2WPKH transaction is ~141 vbytes; fees at ~710 sat/vbyte would consume the entire 100,000 sat withdrawal.
- Bitcoin fees of 500–1,000+ sat/vbyte have occurred during peak congestion (2023 Ordinals/BRC-20 events).
- The window between `retrieve_btc` accepting the request and `submit_pending_requests` running can span multiple IC rounds.
- No special privilege is required — any user calling `retrieve_btc` at the advertised minimum is at risk.

---

### Recommendation

In `submit_pending_requests`, when `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput` is encountered, call `reimburse_canceled_requests` (or an equivalent reimbursement path) instead of silently calling `remove_retrieve_btc_request`. A new `WithdrawalReimbursementReason::AmountTooLow` variant should be added. The reimbursement fee charged should be bounded by `retrieve_btc_min_amount` (consistent with the existing assertion in `reimburse_canceled_requests`).

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,

    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
