Audit Report

## Title
Unauthenticated `buyer` Parameter in `refresh_buyer_tokens` Allows Anyone to Commit Victim's ICP Into SNS Swap Escrow Without Consent - (File: rs/sns/swap/canister/canister.rs)

## Summary

The `refresh_buyer_tokens` update endpoint on the SNS Swap canister accepts an arbitrary `buyer` principal in its request payload without verifying it matches the authenticated caller. Any unprivileged ingress sender can supply a victim's principal as the `buyer` field, causing the victim's ICP (already transferred to the swap subaccount) to be committed into the swap's escrow and their open ticket to be deleted — all without the victim's consent and bypassing any configured `confirmation_text` requirement.

## Finding Description

In `rs/sns/swap/canister/canister.rs` at lines 128–143, the `refresh_buyer_tokens` handler reads the effective buyer principal from the request argument rather than from the authenticated caller:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()          // safe path
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // ← no caller check
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

When `arg.buyer` is non-empty, the attacker-supplied principal `p` is passed directly to `refresh_buyer_token_e8s`. That function then:

1. Reads the ICP balance of `principal_to_subaccount(&buyer)` on the swap canister — the victim's subaccount. [2](#0-1) 
2. Validates the `confirmation_text` (a publicly visible string set at SNS init time). [3](#0-2) 
3. Deletes the victim's open ticket from `OPEN_TICKETS_MEMORY`. [4](#0-3) 
4. Inserts the victim into the `buyers` map with their committed ICP amount. [5](#0-4) 

Once a buyer is present in the `buyers` map with `transfer_success_timestamp_seconds == 0`, `error_refund_icp` explicitly blocks any refund while the swap is open: [6](#0-5) 

```rust
if !(self.lifecycle() == Lifecycle::Aborted || self.lifecycle() == Lifecycle::Committed) {
    return ErrorRefundIcpResponse::new_precondition_error(
        "Error refunds can only be performed when the swap is ABORTED or COMMITTED",
    );
}
```

And even after the swap closes, if the buyer has ICP in escrow with `transfer_success_timestamp_seconds == 0`, the refund is blocked: [7](#0-6) 

By contrast, `new_sale_ticket` always uses `caller_principal_id()` with no override, demonstrating the intended pattern: [8](#0-7) 

The `RefreshBuyerTokensRequest.buyer` field is documented as "If not specified, the caller is used," but no authorization check enforces this intent when a non-empty value is provided: [9](#0-8) 

## Impact Explanation

This vulnerability enables unauthorized commitment of a victim's ICP into an SNS swap escrow without the victim's consent. The attacker can force participation on behalf of any user who has transferred ICP to their swap subaccount, bypassing the `confirmation_text` requirement (a public value readable from `get_init`). If the swap commits, the victim's ICP is permanently swept to the SNS governance treasury and the victim receives SNS tokens they never agreed to purchase. This constitutes unauthorized access to and permanent loss of user funds held in canister-controlled subaccounts, matching the **High ($2,000–$10,000)** impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds." The constraint is that the victim must have already transferred ICP to the swap subaccount (the normal first step of participation), which limits the severity below Critical.

## Likelihood Explanation

The attack requires no special privileges, no governance majority, and no cryptographic material. Any IC user can submit an ingress update call to `refresh_buyer_tokens` with an arbitrary `buyer` string. The victim's principal is observable on-chain (e.g., from `new_sale_ticket` call history or `list_direct_participants`). The `confirmation_text` is readable from the public `get_init` query. The only precondition is that the victim has transferred ICP to their swap subaccount, which is the normal first step of participation and is also observable on-chain via the ICP ledger. The attack is therefore trivially executable by any observer of on-chain state, is repeatable across all active SNS swaps, and requires no victim interaction beyond the initial ICP transfer.

## Recommendation

Remove the unauthenticated `buyer` override. The effective buyer should always be derived from the authenticated caller:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p = caller_principal_id();   // always use the authenticated caller
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
}
```

If third-party notification of another user's participation is a desired feature (e.g., a helper canister calling on behalf of a user), it should be gated by an explicit allowlist or require the victim's principal to match the caller, analogous to how `new_sale_ticket` always uses `caller_principal_id()` with no override.

## Proof of Concept

1. **Setup**: SNS swap is in `Lifecycle::Open`. Alice calls `new_sale_ticket` (which correctly uses `caller_principal_id()`) to create a ticket for `amount = min_participant_icp_e8s`. Alice then transfers that ICP to `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(alice) }` on the ICP ledger.

2. **Attack**: Before Alice calls `refresh_buyer_tokens`, attacker Bob submits:
   ```
   refresh_buyer_tokens({
       buyer: alice_principal.to_text(),
       confirmation_text: Some("<public confirmation text from get_init>"),
   })
   ```

3. **Effect**: The swap canister reads Alice's subaccount balance (≥ `min_participant_icp_e8s`), validates the public confirmation text, deletes Alice's open ticket from `OPEN_TICKETS_MEMORY`, and inserts Alice into `self.buyers` with her ICP amount.

4. **Consequence**: Alice's ICP is now in escrow. If she calls `error_refund_icp` while the swap is open, it is rejected with "Error refunds can only be performed when the swap is ABORTED or COMMITTED." If the swap commits, Alice's ICP is sent to the SNS treasury and she receives SNS tokens she did not explicitly agree to purchase.

**Reproducible test**: A PocketIC integration test can be written that (a) opens an SNS swap, (b) has Alice transfer ICP to her subaccount, (c) has Bob call `refresh_buyer_tokens` with Alice's principal, and (d) asserts that Alice's ICP is committed in the buyers map and her open ticket is deleted, while Alice's own call to `error_refund_icp` is rejected.
