### Title
No Way to De-register a Bridged NEP-141 Token Enables ERC-20 Over-Minting via Fee-on-Transfer — (`engine/src/engine.rs`, `engine/src/map.rs`)

---

### Summary

Aurora Engine permanently registers NEP-141 tokens into a bijection map with no removal path. If a registered NEP-141 token later introduces a fee-on-transfer, the `ft_on_transfer` handler mints ERC-20 tokens equal to the gross `amount` argument, while Aurora's actual NEP-141 balance only increases by `amount - fee`. This creates a persistent insolvency: ERC-20 total supply exceeds the NEP-141 backing, and the last users to exit cannot redeem their tokens.

---

### Finding Description

**Root cause 1 — No de-registration path for NEP-141 tokens**

The NEP-141↔ERC-20 mapping is stored in a `BijectionMap` defined in `engine/src/map.rs`. This map exposes only `insert`, `lookup_left`, and `lookup_right` — **no `remove` method exists**. [1](#0-0) 

The only write path is `register_token` in `engine/src/engine.rs`, which inserts into this map and has no inverse: [2](#0-1) 

The `RegisterTokenError` enum only covers `TokenAlreadyRegistered`, `InvalidNep141AccountId`, and `InvalidAddress` — there is no de-registration error variant and no de-registration function anywhere in the codebase. [3](#0-2) 

The public entry point `deploy_erc20_token` calls `register_token` and there is no corresponding `undeploy` or `unregister` entry point in `engine/src/lib.rs` or `engine/src/contract_methods/connector.rs`. [4](#0-3) 

**Root cause 2 — `ft_on_transfer` trusts the caller-supplied `amount` without verification**

When a NEP-141 token calls `ft_on_transfer` on Aurora, the handler in `engine/src/contract_methods/connector.rs` dispatches to `receive_erc20_tokens`: [5](#0-4) 

Inside `receive_erc20_tokens`, the `amount` is taken verbatim from `args.amount` and passed directly to the ERC-20 mint call: [6](#0-5) 

The mint input is constructed with this unchecked `amount`: [7](#0-6) 

Aurora never checks its own NEP-141 balance before and after the transfer to verify the actual amount received. It unconditionally mints `amount` ERC-20 tokens.

---

### Impact Explanation

If a registered NEP-141 token introduces a fee-on-transfer (e.g., 1% fee), every bridge-in call mints more ERC-20 tokens than the NEP-141 backing held by Aurora. Over time, the ERC-20 total supply diverges from the NEP-141 reserve. When users attempt to exit (burn ERC-20 → receive NEP-141 via `ExitToNear`), the last users in queue cannot redeem because Aurora's NEP-141 balance is insufficient. This is **insolvency** and **permanent fund freeze** for a subset of ERC-20 holders.

---

### Likelihood Explanation

- Any NEP-141 token that was registered at zero-fee and later introduces a fee-on-transfer triggers this bug.
- The triggering action is a normal user bridge-in (`ft_transfer_call` on the NEP-141 token), requiring no special privilege.
- Aurora has no mechanism to pause or de-register the affected token to stop the damage from accumulating.
- Real-world precedent: USDT and similar stablecoins have fee-on-transfer switches that can be activated by the token issuer.

---

### Recommendation

1. Add a `remove` method to `BijectionMap` in `engine/src/map.rs` and expose an admin-only `unregister_erc20_token` entry point that removes the NEP-141↔ERC-20 mapping and prevents further `ft_on_transfer` minting for that token.
2. In `receive_erc20_tokens`, verify the actual NEP-141 balance change by reading Aurora's NEP-141 balance before and after (via a cross-contract view or by requiring the NEP-141 contract to report net received amount), and mint only the verified net amount.

---

### Proof of Concept

1. Deploy a NEP-141 token `fot.near` with 0% fee and register it with Aurora via `deploy_erc20_token`, creating ERC-20 mirror at address `E`.
2. The `fot.near` token issuer activates a 10% fee-on-transfer.
3. Aurora has no way to de-register `fot.near` — `BijectionMap` has no `remove`, and no admin entry point exists.
4. User calls `fot.near::ft_transfer_call(receiver_id: "aurora", amount: "1000", msg: "<evm_address>")`.
5. `fot.near` transfers 1000 tokens, deducts 100 as fee, credits Aurora's account with 900 NEP-141 tokens.
6. `fot.near` calls `aurora::ft_on_transfer(sender_id: user, amount: "1000", msg: "<evm_address>")`.
7. `receive_erc20_tokens` reads `args.amount = 1000` and mints 1000 ERC-20 tokens at address `E` for the user.
8. Aurora holds 900 NEP-141 but has issued 1000 ERC-20 — a 100-token deficit per bridge-in.
9. Repeated bridge-ins accumulate the deficit. Eventually, `ExitToNear` calls for the last users fail because Aurora's NEP-141 balance is exhausted, permanently freezing their funds.

impl<L: AsRef<[u8]> + TryFrom<Vec<u8>>, R: AsRef<[u8]> + TryFrom<Vec<u8>>, I: IO>
    BijectionMap<L, R, I>
{
    pub const fn new(left_prefix: KeyPrefix, right_prefix: KeyPrefix, io: I) -> Self {
        Self {
            left_prefix,
            right_prefix,
            io,
            left_phantom: PhantomData,
            right_phantom: PhantomData,
        }
    }

    pub fn insert(&mut self, left: &L, right: &R) {
        let key = self.left_key(left);
        self.io.write_storage(&key, right.as_ref());

        let key = self.right_key(right);
        self.io.write_storage(&key, left.as_ref());
    }

    pub fn lookup_left(&self, left: &L) -> Option<R> {
        let key = self.left_key(left);
        self.io
            .read_storage(&key)
            .and_then(|v| v.to_vec().try_into().ok())
    }

    pub fn lookup_right(&self, right: &R) -> Option<L> {
        let key = self.right_key(right);
        self.io
            .read_storage(&key)
            .and_then(|v| v.to_vec().try_into().ok())
    }

    fn left_key(&self, left: &L) -> Vec<u8> {
        bytes_to_key(self.left_prefix, left.as_ref())
    }

    fn right_key(&self, right: &R) -> Vec<u8> {
        bytes_to_key(self.right_prefix, right.as_ref())
    }

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

/// Used to bridge NEP-141 tokens from NEAR to Aurora. On Aurora the NEP-141 becomes an ERC-20.
pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
