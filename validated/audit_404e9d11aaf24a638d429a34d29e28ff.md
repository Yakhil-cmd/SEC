### Title
Unrestricted `deploy_erc20_token` Allows Any Caller to Register Fake NEP-141 Tokens and Mint Unbacked ERC-20 Tokens on Aurora - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` has no access control. Any NEAR account can register an arbitrary NEP-141 account ID as an ERC-20 token on Aurora. An attacker who controls a fake NEP-141 contract can then call `ft_on_transfer` directly on Aurora Engine from that contract, minting arbitrary amounts of ERC-20 tokens without depositing any real assets. These unbacked tokens can be used to drain real assets from Aurora DeFi protocols.

---

### Finding Description

**Root cause — missing authorization in `deploy_erc20_token`:**

The public entry point `deploy_erc20_token` performs only a liveness check (`require_running`) and no caller authorization: [1](#0-0) 

Compare this to `mirror_erc20_token`, which performs an identical registration-adjacent operation but correctly enforces owner-only access: [2](#0-1) 

And `set_eth_connector_contract_account`, which enforces either a private call or owner check: [3](#0-2) 

The `deploy_erc20_token` entry point is exposed as a public NEAR contract method with no restriction: [4](#0-3) 

**How `ft_on_transfer` trusts the predecessor as the token identity:**

When `ft_on_transfer` is called, it uses `predecessor_account_id` (the calling NEAR account) as the NEP-141 token identifier. If that account is registered in the ERC-20 map, it mints ERC-20 tokens for the amount specified in `args.amount`: [5](#0-4) 

The minting path in `receive_erc20_tokens` calls `get_erc20_from_nep141` with the predecessor as the token key. If the predecessor is registered, minting proceeds unconditionally: [6](#0-5) 

**Attack steps:**

1. Attacker creates a NEAR account `fake-token.near` and deploys a minimal contract on it.
2. Attacker calls `deploy_erc20_token` on Aurora Engine with `fake-token.near` as the NEP-141 account ID. No authorization check prevents this. Aurora deploys an ERC-20 contract and writes the mapping `fake-token.near → ERC-20 address` into storage.
3. The attacker's `fake-token.near` contract calls `ft_on_transfer` directly on Aurora Engine with `amount = <arbitrary large value>` and `msg = <attacker's EVM address>`. In NEAR, any contract can call any public function on any other contract; `predecessor_account_id` is set by the runtime to `fake-token.near`.
4. Inside `ft_on_transfer`, `predecessor_account_id` (`fake-token.near`) is not the ETH connector account, so `receive_erc20_tokens` is called with `token = fake-token.near`.
5. `get_erc20_from_nep141` succeeds (the mapping was written in step 2). Aurora mints the attacker-specified amount of ERC-20 tokens to the attacker's EVM address. No real NEP-141 tokens were ever transferred.
6. The attacker uses these unbacked ERC-20 tokens in Aurora DeFi protocols (AMMs, lending markets) to steal real user assets.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The attacker can mint an arbitrary quantity of any ERC-20 token on Aurora that is backed by zero real assets. These tokens are indistinguishable from legitimately bridged tokens at the EVM level. Any Aurora DeFi protocol that accepts the token can be drained. The `register_token` function prevents double-registration of the same NEP-141 ID, but the attacker controls a fresh account and faces no such collision. [7](#0-6) 

---

### Likelihood Explanation

**High.** The attack requires:
- Creating a NEAR account (trivial, costs fractions of a cent).
- Deploying a minimal NEAR contract that calls `ft_on_transfer` on Aurora (a few lines of Rust or AssemblyScript).
- Calling two public functions on Aurora Engine.

No privileged access, leaked keys, or governance capture is needed. The vulnerability is reachable by any unprivileged NEAR account.

---

### Recommendation

Add the same owner-only guard used by `mirror_erc20_token` to `deploy_erc20_token`:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        // Add this:
        require_owner_only(&state, &env.predecessor_account_id())?;
        ...
    })
}
```

Alternatively, introduce a dedicated admin role (as the `TODO` comment in `mirror_erc20_token` already suggests) to allow a controlled set of bridge operators to register tokens. [2](#0-1)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
