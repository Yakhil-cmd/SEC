### Title
`receive_base_tokens()` Bypasses Silo Address Whitelist, Allowing Non-Whitelisted Addresses to Receive Base ETH — (`engine/src/engine.rs`)

---

### Summary

In Silo mode with the `Address` whitelist enabled, `receive_base_tokens()` does not check whether the recipient EVM address is whitelisted before minting base ETH tokens. This is inconsistent with `receive_erc20_tokens()`, which correctly checks the whitelist and redirects tokens to the configured fallback address. Any caller can bridge base ETH to a non-whitelisted address, bypassing the Silo access control and the fallback mechanism entirely.

---

### Finding Description

The Silo mode in Aurora Engine includes an `Address` whitelist that controls which EVM addresses may receive ERC-20 tokens. When a non-whitelisted address is specified as recipient, `receive_erc20_tokens()` redirects the tokens to the configured `erc20_fallback_address`:

```rust
// engine/src/engine.rs lines 818-822
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

However, `receive_base_tokens()` contains no equivalent check:

```rust
// engine/src/engine.rs lines 773-790
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    // ← No whitelist check; no fallback redirect
    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
``` [2](#0-1) 

Both functions are dispatched from `ft_on_transfer()` in `engine/src/contract_methods/connector.rs`. The branch taken depends on whether the predecessor is the ETH connector account (base ETH) or a NEP-141 token account (ERC-20):

```rust
// engine/src/contract_methods/connector.rs lines 81-90
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [3](#0-2) 

The whitelist check used by `receive_erc20_tokens()` is `silo::is_allow_receive_erc20_tokens()`, which delegates to `is_address_allowed()`:

```rust
// engine/src/contract_methods/silo/mod.rs lines 140-143
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [4](#0-3) 

`is_address_allowed()` returns `false` for any address not in the `Address` whitelist when the whitelist is enabled:

```rust
// engine/src/contract_methods/silo/mod.rs lines 155-158
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [5](#0-4) 

The `erc20_fallback_address` is the Silo operator's designated recovery address for tokens that would otherwise go to non-whitelisted recipients:

```rust
// engine-types/src/parameters/silo.rs lines 19-23
/// EVM address, which is used for withdrawing ERC-20 base tokens in case
/// a recipient of the tokens is not in the silo white list.
/// Note: the logic described above works only if the fallback address
/// is set by `set_silo_params` function. In other words, in Silo mode.
pub erc20_fallback_address: Address,
``` [6](#0-5) 

Because `receive_base_tokens()` skips the whitelist check, base ETH is minted directly to the caller-specified address regardless of whitelist status, while the fallback address receives nothing.

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

When Silo mode is active with the `Address` whitelist enabled and a fallback address configured:

1. A user bridges base ETH to Aurora via `ft_transfer_call` on the ETH connector, specifying a non-whitelisted EVM address as recipient.
2. `receive_base_tokens()` mints the ETH to that non-whitelisted address without any whitelist check.
3. The fallback address receives nothing, defeating the Silo operator's recovery mechanism.
4. The non-whitelisted address cannot submit EVM transactions (blocked by `assert_access` → `is_allow_submit`), cannot call the `ExitToNear` or `ExitToEthereum` precompiles, and cannot move the funds.
5. The funds are frozen at the non-whitelisted address. Recovery requires the Silo operator to explicitly whitelist that address — an action they may never take, making the freeze permanent. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** This requires Silo mode to be active with the `Address` whitelist enabled and a fallback address configured — a deliberate operator configuration. Once that configuration is in place, any unprivileged user can trigger the bypass by calling `ft_transfer_call` on the ETH connector contract with any non-whitelisted EVM address as the `msg` recipient. No special permissions or prior state are required beyond having ETH tokens to bridge.

---

### Recommendation

Apply the same whitelist check in `receive_base_tokens()` as exists in `receive_erc20_tokens()`. If a fallback address is configured and the recipient is not whitelisted, redirect the base ETH to the fallback address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut receipient = message_data.recipient;

    // Mirror the check in receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
    {
        receipient = fallback_address;
    }

    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

### Proof of Concept

**Attacker-controlled entry path:**

1. Silo operator deploys Aurora Engine in Silo mode: `set_silo_params` with `fixed_gas` and `erc20_fallback_address = <fallback>`, and enables the `Address` whitelist via `set_whitelist_status(WhitelistKind::Address, active: true)`.
2. Attacker (non-whitelisted NEAR account) calls `ft_transfer_call` on the ETH connector contract with `receiver_id = aurora` and `msg = <attacker_evm_address>`.
3. ETH connector calls `ft_on_transfer` on Aurora engine with `predecessor_account_id == eth_connector_account_id`.
4. `ft_on_transfer` dispatches to `receive_base_tokens()`.
5. `receive_base_tokens()` mints base ETH to `<attacker_evm_address>` with no whitelist check.
6. **Observed:** `<attacker_evm_address>` holds base ETH; `<fallback>` receives nothing.
7. **Expected:** `<fallback>` should receive the ETH (matching the behavior of `receive_erc20_tokens()`).
8. Attacker cannot use the ETH (blocked by `assert_access` on all EVM transaction submission paths), and the Silo operator's fallback recovery mechanism has been bypassed. Funds are frozen. [8](#0-7) [9](#0-8)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,

    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)

        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)

            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
