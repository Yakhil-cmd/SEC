Audit Report

## Title
Unbounded `Vec<GetBlocksRequest>` Input Enables Instruction-Budget Exhaustion DoS on ICRC-1 Ledger and Archive Query Endpoints - (File: rs/ledger_suite/icrc1/ledger/src/lib.rs, rs/ledger_suite/icrc1/archive/src/main.rs)

## Summary

The `icrc3_get_blocks` query endpoint on both the ICRC-1 ledger canister and the ICRC-1 archive canister accepts a `Vec<GetBlocksRequest>` with no bound on the number of elements. The only early-exit guard fires when 100 *local* blocks have been accumulated, which is trivially bypassed by supplying zero-length or archive-only requests. An unprivileged caller can force the canister to iterate over an arbitrarily large input vector, invoking `query_blocks` / `decode_block_range` per element until the per-query instruction limit is exhausted, causing the query to trap. Because query calls on the IC are free to the caller, this can be repeated at no cost to deny service to legitimate users.

## Finding Description

**ICRC-1 Ledger — `rs/ledger_suite/icrc1/ledger/src/lib.rs` (L1107–1153)**

`icrc3_get_blocks` iterates over the caller-supplied `args: Vec<GetBlocksRequest>` with no pre-loop bound check:

```rust
pub fn icrc3_get_blocks(&self, args: Vec<GetBlocksRequest>) -> GetBlocksResult {
    const MAX_BLOCKS_PER_RESPONSE: u64 = 100;
    let mut blocks = vec![];
    for arg in args {                          // ← no bound on args.len()
        ...
        let max_length = MAX_BLOCKS_PER_RESPONSE.saturating_sub(blocks.len() as u64);
        if max_length == 0 { break; }          // ← only exits when 100 LOCAL blocks collected
        let length = max_length.min(length)...;
        let (first_index, local_blocks, archived_ranges) = self.query_blocks(start, length, ...);
        ...
        if blocks.len() as u64 >= MAX_BLOCKS_PER_RESPONSE { break; }
    }
```

When every request has `length = 0`, `query_blocks(start, 0, ...)` is called per element. Inspecting `query_blocks` (L1008–1038), it calls `block_locations(self, start, 0)` and `self.blockchain.get_blocks(local_blocks_range)` — both non-trivial operations — but returns zero local blocks. Consequently `blocks.len()` never reaches 100, the guard never fires, and the loop runs to completion over the entire input vector. The same bypass applies to requests targeting only archived ranges (no local blocks appended).

The public query handler passes the input directly with no additional validation (`rs/ledger_suite/icrc1/ledger/src/main.rs` L1171–1174):

```rust
#[query]
fn icrc3_get_blocks(args: Vec<GetBlocksRequest>) -> GetBlocksResult {
    Access::with_ledger(|ledger| ledger.icrc3_get_blocks(args))
}
```

**ICRC-1 Archive — `rs/ledger_suite/icrc1/archive/src/main.rs` (L359–390)**

The archive canister has the identical pattern. With `length = 0` per request, `decode_block_range(start, 0, decode_icrc1_block)` is invoked for every element but returns an empty iterator, so `blocks.len()` never grows and the loop never exits early.

**Root cause**: The guard condition is keyed on *local blocks collected*, not on *requests processed*. Zero-length or archive-only requests consume per-iteration work (archive index lookups, range computations, closure construction) without ever incrementing `blocks.len()`, making the guard permanently ineffective for such inputs.

## Impact Explanation

This is an application/platform-level DoS on production ICRC-1 ledger and archive canisters — explicitly in-scope financial infrastructure. A query call exhausting the IC's per-query instruction limit (~5 billion instructions) traps and fails. Repeated concurrent calls from one or more clients saturate the replica threads serving query calls for the ledger and archive canisters, causing timeouts and failures for legitimate callers (wallets, explorers, index canisters) querying block history. This matches the allowed High impact: *"Application/platform-level DoS … not based on raw volumetric DDoS."* The attack exploits a specific unbounded-iteration code path, not raw packet flooding.

## Likelihood Explanation

The endpoint is publicly callable by any anonymous or authenticated principal with no authentication requirement. The ICRC-3 standard defines the input as `vec GetBlocksRequest`, so any standard IC agent can construct the call. No special knowledge, privilege, on-chain stake, or funds are required. Query calls are free to the caller on the IC, so the attack can be repeated in a tight loop at zero cost. Archive canister IDs are publicly discoverable via `icrc3_get_archives`. Likelihood is high.

## Recommendation

1. **Enforce a hard cap on the number of request items** before entering the loop in both `rs/ledger_suite/icrc1/ledger/src/lib.rs` and `rs/ledger_suite/icrc1/archive/src/main.rs`:

```rust
const MAX_REQUESTS_PER_CALL: usize = 100;
if args.len() > MAX_REQUESTS_PER_CALL {
    ic_cdk::api::trap("Too many block ranges requested");
}
```

2. **Charge the early-exit condition against total requests processed**, not only local blocks collected, so that a flood of zero-length or archive-only requests also triggers an exit:

```rust
if blocks.len() as u64 >= MAX_BLOCKS_PER_RESPONSE || processed >= MAX_REQUESTS_PER_CALL {
    break;
}
```

## Proof of Concept

Using any IC agent, send a query call to the ICRC-1 ledger's `icrc3_get_blocks` method with a large vector of zero-length requests:

```python
# Pseudocode using ic-py or similar
requests = [{"start": 0, "length": 0}] * 500_000
agent.query(ICRC1_LEDGER_CANISTER_ID, "icrc3_get_blocks", encode(requests))
```

The canister iterates over all 500,000 elements, calling `query_blocks(0, 0, ...)` each time. The query exhausts the instruction limit and traps. A deterministic integration test using PocketIC can verify this locally: construct a ledger with a small number of blocks, call `icrc3_get_blocks` with N zero-length entries, and assert the call traps for sufficiently large N while succeeding for small N. The same call directed at any archive canister ID obtained from `icrc3_get_archives` reproduces the issue in `rs/ledger_suite/icrc1/archive/src/main.rs`.
