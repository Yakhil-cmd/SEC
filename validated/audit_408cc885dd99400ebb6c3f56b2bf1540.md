Audit Report

## Title
Unvalidated `buyer` Parameter in `refresh_buyer_tokens` Allows Third-Party Forced Participation Bypassing Confirmation Text Consent - (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The SNS Swap canister's `refresh_buyer_tokens` endpoint resolves the buyer identity from a caller-supplied string without verifying it matches the actual message caller. Any unprivileged ingress sender can specify an arbitrary victim principal in the `buyer` field, supply the publicly-readable `confirmation_text`, and force-register that victim's ICP participation — permanently locking their funds during the `Open` lifecycle and bypassing the explicit consent mechanism the confirmation text was designed to enforce.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 130–134, when `arg.buyer` is non-empty the resolved principal is used directly with no equality check against `caller_principal_id()`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller check
};
```

This `p` is passed to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`. Inside that function, `validate_confirmation_text` (lines 363–384) only checks that the supplied string matches the SNS-configured text — it is entirely caller-agnostic. The confirmation text is stored in `Init.confirmation_text` and is publicly readable via the `get_state` query endpoint.

The ticket system at lines 1249–1272 provides no protection: the comment at line 1271 explicitly states "If there exists no ticket for the buyer, the payment flow will simply ignore the ticket." A victim who transferred ICP directly to their swap subaccount without going through `new_sale_ticket` has no ticket, so the attacker's call proceeds unconditionally.

Once the attacker's call succeeds, the victim's entry is written to `self.buyers` (lines 1285–1288). The `error_refund_icp` escape hatch is unavailable during `Open` state (lines 1932–1935) and is additionally blocked for any registered buyer whose `transfer_success_timestamp_seconds == 0` (lines 1950–1959) — exactly the state a freshly force-registered buyer is in.

## Impact Explanation
This matches the High impact category: "Significant SNS security impact with concrete user or protocol harm." A victim who has sent ICP to the swap subaccount and reconsidered participation is forcibly enrolled without their consent. Their ICP is locked for the entire `Open` lifecycle (potentially days to weeks). If the swap commits, their ICP is swept to SNS governance and they receive SNS tokens having never explicitly agreed to participate — a direct, concrete financial loss. The confirmation text, which is the sole explicit consent gate for direct participation, is rendered meaningless as a protection.

## Likelihood Explanation
The entry path requires no special privilege — any unprivileged ingress sender can trigger it. The only precondition is that the victim has ICP sitting in their swap subaccount, which is the normal first step of participation and a realistic scenario for users who reconsider. The `confirmation_text` is a public string readable by anyone via `get_state`. Execution requires a single `update` call. The attack window is the entire `Open` lifecycle.

## Recommendation
1. **Enforce caller == buyer**: In `canister.rs`, when `arg.buyer` is non-empty, require `PrincipalId::from_str(&arg.buyer).unwrap() == caller_principal_id()`, or reject non-empty `buyer` fields from non-privileged callers entirely.
2. **Alternatively, remove the `buyer` override field** and always use `caller_principal_id()`. Third-party notification of participation is not a necessary feature and introduces this consent bypass.
3. If third-party notification is intentionally supported, require the buyer to pre-register consent via a separate caller-authenticated call before a third party can trigger `refresh_buyer_tokens` on their behalf.

## Proof of Concept
1. Deploy an SNS swap in `Open` state with `confirmation_text = "I agree to participate"`.
2. Victim transfers 10 ICP to `principal_to_subaccount(victim_principal)` on the ICP ledger targeting the swap canister. Victim then reconsiders and does not call `refresh_buyer_tokens`.
3. Attacker queries `get_state` to read the public `confirmation_text`.
4. Attacker submits:
   ```
   refresh_buyer_tokens({
     buyer: "<victim_principal_text>",
     confirmation_text: Some("I agree to participate")
   })
   ```
5. The swap canister resolves `p = victim_principal`, passes `validate_confirmation_text` (text matches), finds no ticket for the victim (ticket check is skipped per line 1271), queries the victim's subaccount balance (10 ICP ≥ `min_participant_icp_e8s`), and writes `BuyerState { amount_icp_e8s: 10_ICP }` for the victim.
6. Victim calls `error_refund_icp` — rejected: "Error refunds can only be performed when the swap is ABORTED or COMMITTED."
7. Swap commits; victim's 10 ICP is swept to SNS governance and they receive SNS tokens, having never explicitly consented.

This is reproducible as a deterministic integration test using the existing `StateMachine`/PocketIC infrastructure in `rs/sns/swap/tests/swap.rs` and `rs/sns/integration_tests/src/payment_flow.rs` by calling `refresh_buyer_tokens` from a different sender identity than the victim principal.
