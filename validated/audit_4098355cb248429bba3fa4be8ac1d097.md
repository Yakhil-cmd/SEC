### Title
BLAKE2F Precompile Charges Zero Native Resource for Attacker-Controlled `num_rounds` Loop — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_152/impls.rs`)

---

### Summary

The EIP-152 BLAKE2F precompile implementation charges **zero native resource** for the `num_rounds` mixing-function loop, while the loop count is a full `u32` value read directly from attacker-controlled calldata. An unprivileged transaction sender can force the prover to execute up to ~4.29 billion iterations of the BLAKE2 mixing function per call while paying only EVM gas (ergs), with no corresponding native resource deduction. This is a direct analog to CVE-2023-50658 (`jose2go` PBES2 Count → unbounded CPU), transposed to ZKsync OS's native resource accounting layer.

---

### Finding Description

In `Blake2FPrecompile::execute`, the first four bytes of the 213-byte input are decoded as a big-endian `u32` to obtain `num_rounds`:

```rust
let num_rounds = u32::from_be_bytes(input.as_chunks::<4>().0[0]);
let cost_ergs = Ergs(((num_rounds as u64) * GAS_PER_ROUND) * ERGS_PER_GAS);
let cost_native = 0;                          // ← hardcoded zero
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(cost_native),
))?;
``` [1](#0-0) 

After the charge, the actual computation loop runs unconditionally:

```rust
round_function_for_num_rounds(&mut extended_state, &message_block, num_rounds as usize);
``` [2](#0-1) 

`round_function_for_num_rounds` iterates exactly `num_rounds` times, calling the full BLAKE2 mixing function each iteration:

```rust
for i in 0..num_rounds {
    let sigma = &SIGMAS[i % 10];
    mixing_function(extended_state, message_block, sigma);
}
``` [3](#0-2) 

Each `mixing_function` call executes 8 invocations of `g_function`, each performing multiple 64-bit additions, XORs, and rotations — non-trivial RISC-V work. [4](#0-3) 

By contrast, the internal `Blake2s256` hasher used elsewhere in the system is correctly charged at `BLAKE2S_ROUND_NATIVE_COST = 340` native units per round: [5](#0-4) 

The EIP-152 precompile omits this charge entirely. `GAS_PER_ROUND = 1` means ergs are charged at 1 gas per round, so the ergs budget is exhausted at `num_rounds ≈ gas_limit`. But native resource — which models prover RISC-V cycle complexity — is never decremented. [6](#0-5) 

---

### Impact Explanation

The native resource is the primary guard against prover overload. `MAX_NATIVE_COMPUTATIONAL = 1 << 35 ≈ 34.4 billion` native units caps total prover work per execution: [7](#0-6) 

With `BLAKE2S_ROUND_NATIVE_COST = 340` as the reference cost per mixing round, a transaction spending 30 million gas on BLAKE2F rounds would consume approximately **10.2 billion native units** of actual prover work (≈ 30% of the total block budget) while charging **0 native units**. Multiple such transactions in a single block can collectively exhaust the prover's real computational capacity while the native resource counter remains near zero.

The concrete impact is a **prover denial-of-service**: the sequencer's forward system accepts the block (ergs are paid, native limit appears unexceeded), but the prover — executing the same RISC-V binary — must perform the full mixing-function loop and may exceed its proving time budget, making the block unprovable. This is a forward/proving divergence rooted in a resource accounting bug.

---

### Likelihood Explanation

The attack requires only a standard EVM transaction calling address `0x0009` (the BLAKE2F hook) with a 213-byte payload whose first four bytes encode a large `num_rounds`. No privileged access, governance, or oracle manipulation is needed. The attacker pays only EVM gas, which is economically bounded but does not compensate for the disproportionate prover work. The precompile is registered unconditionally in the Ethereum execution path via `initialize_eip_152`. [8](#0-7) 

---

### Recommendation

Charge native resources proportionally to `num_rounds`, mirroring the pattern used for other iterative precompiles. Replace the hardcoded `cost_native = 0` with:

```rust
let cost_native = (num_rounds as u64)
    .saturating_mul(BLAKE2F_ROUND_NATIVE_COST);
```

where `BLAKE2F_ROUND_NATIVE_COST` is benchmarked analogously to `BLAKE2S_ROUND_NATIVE_COST = 340` in `basic_system/src/cost_constants.rs`. [5](#0-4) 

---

### Proof of Concept

1. Construct a 213-byte BLAKE2F input with `num_rounds = 0x01C9C380` (30,000,000 — within a 30M gas block limit). Bytes 0–3: `01 C9 C3 80`. Remaining 209 bytes: arbitrary valid state (e.g., all zeros except finalization flag byte = `0x00`).

2. Submit a ZKsync OS transaction to address `0x0000000000000000000000000000000000000009` with this calldata and `gas_limit = 30_000_000`.

3. The ergs charge succeeds: `30_000_000 * 1 * ERGS_PER_GAS` is deducted.

4. `cost_native = 0` is charged — native resource counter is unchanged.

5. `round_function_for_num_rounds` executes 30 million iterations of the BLAKE2 mixing function.

6. The forward system records the transaction as successful. The prover must reproduce the same 30 million mixing rounds on RISC-V without any native budget having been consumed, potentially exceeding its time limit and failing to produce a proof.

7. Repeat with additional transactions in the same block to amplify the effect up to the block gas limit.
