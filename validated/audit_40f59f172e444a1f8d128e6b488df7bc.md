### Title
Cleared Silo Params With Active Address Whitelist Permanently Freezes Bridged ERC-20 Tokens — (`engine/src/engine.rs`, `engine/src/contract_methods/silo/mod.rs`)

### Summary

`set_silo_params(None)` atomically removes both `fixed_gas` and `erc20_fallback_address` from storage but leaves the Address whitelist untouched. The redirect guard in `receive_erc20_tokens` is a conjunctive let-chain that only fires when a fallback address is present **and** the recipient is not whitelisted. With the fallback cleared, the guard is never entered regardless of whitelist state, so ERC-20 tokens are minted directly to any caller-supplied address — including non-whitelisted ones that can never submit EVM transactions.

### Finding Description

`set_silo_params` in `engine/src/contract_methods/silo/mod.rs`:

```rust
pub fn set_silo_params<I: IO>(io: &mut I, args: Option<SiloParamsArgs>) {
    let (cost, address) = args.map_or((None, None), |params| {
        (Some(params.fixed_gas), Some(params.erc20_fallback_address))
    });
    set_fixed_gas(io, cost);
    set_erc20_fallback_address(io, address);   // ← removes key when None
}
```

When called with `None`, `set_erc20_fallback_address` calls `io.remove_storage(&key)`, erasing the fallback key. The Address whitelist is a completely separate storage structure and is not modified. [1](#0-0) 

`receive_erc20_tokens` in `engine/src/engine.rs` then performs:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

This is a Rust let-chain: the entire block is skipped when `get_erc20_fallback_address` returns `None`. The whitelist check (`is_allow_receive_erc20_tokens`) is never evaluated. `recipient` is used as-is, and the ERC-20 `mint` call proceeds unconditionally. [2](#0-1) 

`is_allow_receive_erc20_tokens` delegates entirely to the Address whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)   // returns false when whitelist enabled + address absent
}
``` [3](#0-2) 

`is_allow_submit` uses the same `is_address_allowed` check, so any address that fails the receive check also fails the submit check — it cannot initiate any EVM transaction to recover the tokens. [4](#0-3) 

### Impact Explanation

Tokens are minted to a non-whitelisted EVM address. That address cannot call `exit_to_near` or any other EVM function because `is_allow_submit` rejects it. The NEP-141 side has already debited the sender. The ERC-20 balance is live on-chain but unreachable. Unless an admin subsequently adds the specific recipient address to the whitelist (which requires knowing which addresses were affected), the tokens are permanently frozen.

Impact: **Critical — Permanent freezing of funds.**

### Likelihood Explanation

The trigger is an admin calling `set_silo_params(None)` — a documented, legitimate operation to disable silo mode — while the Address whitelist remains enabled. This is a realistic operational sequence: an operator disabling fixed-gas billing without realising the call also silently removes the fallback guard. No malicious intent is required; the API design makes the dangerous state easy to reach by omission. Any bridge transfer to a non-whitelisted address that arrives after the call is affected.

### Recommendation

Two complementary fixes:

1. **In `receive_erc20_tokens`**: decouple the whitelist check from the fallback existence check. If the Address whitelist is enabled and the recipient is not whitelisted, either redirect to the fallback (if set) **or** return an error that causes `ft_on_transfer` to refund the sender — never silently mint to the non-whitelisted address.

2. **In `set_silo_params`**: when called with `None`, also disable the Address whitelist (or at minimum emit a hard error if the whitelist is currently active, forcing the operator to disable it explicitly first).

### Proof of Concept

```rust
// 1. Enable silo mode with fallback
set_silo_params(&mut runner, Some(SiloParamsArgs {
    fixed_gas: FIXED_GAS,
    erc20_fallback_address: FALLBACK,
}));

// 2. Enable Address whitelist (recipient R is NOT added)
enable_address_whitelist(&mut runner);

// 3. Admin disables silo mode — also silently removes fallback
set_silo_params(&mut runner, None);
// get_erc20_fallback_address() now returns None
// Address whitelist is still enabled

// 4. Bridge ERC-20 tokens to non-whitelisted address R via ft_on_transfer
bridge_erc20_to(&mut runner, R, AMOUNT);

// 5. Assert: tokens minted to R, not to FALLBACK
assert_eq!(erc20_balance(R), AMOUNT);      // ← minted to non-whitelisted address
assert_eq!(erc20_balance(FALLBACK), 0);    // ← fallback never used

// 6. Assert: R cannot submit any EVM transaction
assert!(runner.submit_with_address(R, any_tx()).is_err());
// Tokens are permanently frozen at R
```

The `ft_on_transfer` error-handling path only refunds on `Err`; a successful mint to the wrong address returns `Ok(0)` to the NEP-141 contract, so no automatic refund occurs. [5](#0-4)
