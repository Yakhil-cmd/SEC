### Title
ERC-20 Mirror Over-Minting via Fee-on-Transfer NEP-141 Tokens - (`engine/src/engine.rs`)

---

### Summary

`receive_erc20_tokens` mints ERC-20 wrapper tokens based on the `amount` field supplied in the `ft_on_transfer` callback arguments, without verifying that the Aurora Engine's actual NEP-141 balance increased by that same amount. A fee-on-transfer NEP-141 token deducts a fee during `ft_transfer_call`, so Aurora's on-chain NEP-141 balance grows by `amount - fee`, yet the ERC-20 mirror mints the full `amount`. This creates a permanent accounting deficit: more ERC-20 tokens exist than can ever be redeemed for NEP-141 tokens, making the bridge insolvent for that token.

---

### Finding Description

When a NEP-141 token is bridged into Aurora, the NEAR runtime calls `ft_on_transfer` on the Aurora Engine contract. The engine routes this to `receive_erc20_tokens`: [1](#0-0) 

```rust
pub fn receive_erc20_tokens<P: PromiseHandler>(
    &mut self,
    token: &AccountId,
    args: &FtOnTransferArgs,
    ...
) -> Result<Option<SubmitResult>, ContractError> {
    let amount = args.amount.as_u128();   // ← taken verbatim from callback args
```

This `amount` is then passed directly to the ERC-20 `mint` call: [2](#0-1) 

```rust
    let result = self
        .call(
            &erc20_admin_address,
            &erc20_token,
            Wei::zero(),
            setup_receive_erc20_tokens_input(&recipient, amount),  // mints `amount`
            ...
        )
        .and_then(submit_result_or_err)?;
```

The NEP-141 standard's `ft_transfer_call` flow is:

1. Sender calls `ft_transfer_call(aurora, amount, msg)` on the NEP-141 contract.
2. NEP-141 contract credits Aurora's internal balance with `amount - fee` (for a fee-on-transfer token).
3. NEP-141 contract calls `aurora.ft_on_transfer(sender, amount, msg)` — passing the **pre-fee** amount.
4. Aurora mints `amount` ERC-20 tokens to the recipient.

Aurora never performs a before/after balance check on its NEP-141 account to confirm it actually received `amount`. It blindly trusts the callback argument.

The same pattern exists in `receive_base_tokens` for the ETH connector path: [3](#0-2) 

The entry point that dispatches to both paths is `ft_on_transfer` in `connector.rs`: [4](#0-3) 

---

### Impact Explanation

After N deposits of a fee-on-transfer NEP-141 token with fee rate `f`:

- **ERC-20 total supply** = `N × amount`
- **Aurora's NEP-141 balance** = `N × amount × (1 - f)`

The deficit grows linearly with deposit volume. When users exit via the `ExitToNear` precompile, the engine issues `ft_transfer` promises against its NEP-141 balance. The last users to exit will find the engine's NEP-141 balance exhausted; their ERC-20 tokens are permanently frozen or lost. This is **insolvency** of the ERC-20 mirror and **permanent freezing of funds** for late redeemers.

---

### Likelihood Explanation

Any party can deploy a NEP-141 token with a fee-on-transfer mechanism on NEAR — this requires no special privilege. Once the token's ERC-20 wrapper is deployed via `deploy_erc20_token`, any user can trigger the vulnerability by calling `ft_transfer_call` on the NEP-141 contract targeting Aurora. No admin compromise or governance capture is required. The attacker-controlled entry path is fully permissionless.

---

### Recommendation

Replace the blind trust in `args.amount` with an actual balance check. Before and after the NEP-141 transfer settles, query Aurora's NEP-141 balance and use the **delta** as the mint amount. Concretely, in `receive_erc20_tokens`, the amount minted to the ERC-20 contract should be derived from a cross-contract balance query rather than `args.amount.as_u128()`. Alternatively, maintain an internal ledger of expected NEP-141 balances per token and reject deposits where `args.amount` exceeds the observed balance increase.

---

### Proof of Concept

1. Deploy a NEP-141 token `fee_token.near` that charges a 5% fee on every transfer.
2. Call `aurora.deploy_erc20_token({ nep141: "fee_token.near" })` to create the ERC-20 wrapper.
3. Call `fee_token.near.ft_transfer_call({ receiver_id: "aurora", amount: "1000", msg: "<recipient_evm_address>" })`.
   - `fee_token.near` credits Aurora's balance with `950` (deducting 5% fee).
   - `fee_token.near` calls `aurora.ft_on_transfer(sender, "1000", msg)`.
   - Aurora mints **1000** ERC-20 tokens to `<recipient_evm_address>`.
4. Repeat for many depositors. Total ERC-20 supply grows to `N × 1000`; Aurora's NEP-141 balance is only `N × 950`.
5. When the first `N × 0.95` depositors exit via `ExitToNear`, they succeed. The remaining depositors find Aurora's NEP-141 balance at zero; their ERC-20 tokens cannot be redeemed — **permanent fund freeze**.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
