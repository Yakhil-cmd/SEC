Audit Report

## Title
SNS Swap Validation Uses Direct-Only ICP Cap While Distribution Divides by Total (Direct + Neurons' Fund) ICP, Causing Permanent SNS Token Loss — (`rs/sns/init/src/lib.rs`, `rs/sns/swap/src/types.rs`, `rs/sns/swap/src/swap.rs`)

## Summary
The SNS initialization and swap-open validation compute the minimum SNS tokens a participant receives using only `max_direct_participation_icp_e8s` as the denominator. At finalization, actual token distribution divides by `total_participant_icp_e8s` — which includes both direct and Neurons' Fund (NF) participation. When NF participation is non-zero, minimum-contributing direct participants receive fewer SNS tokens than the validation guaranteed. If the shortfall pushes per-neuron allocation below `neuron_minimum_stake_e8s`, SNS governance rejects the neuron claim with `ClaimedSwapNeuronStatus::Invalid`, while the SNS tokens have already been transferred to the neuron subaccount by `sweep_sns`. No recovery path exists in the swap canister, permanently stranding those tokens.

**Exploit chain:**

1. `create_sns_neuron_recipes()` calls `create_sns_neuron_basket_for_direct_participant()` with the diluted `amount_sns_e8s`. `generate_vesting_schedule()` does **not** validate minimum stake — it only checks `count == 0` — so it succeeds and produces recipes with sub-threshold per-neuron amounts. [6](#0-5) 

2. `sweep_sns()` transfers the sub-threshold SNS token amounts to the neuron subaccounts (subaccounts of the SNS governance canister keyed by `(buyer_principal, memo)`). [7](#0-6) 

3. `claim_swap_neurons()` in SNS governance calls `neuron_recipe.validate(neuron_minimum_stake_e8s, ...)`. If `stake_e8s < neuron_minimum_stake_e8s`, the recipe is rejected with `ClaimedSwapNeuronStatus::Invalid`. [8](#0-7) [9](#0-8) 

4. `ClaimedSwapNeuronStatus::Invalid` maps to `ClaimedStatus::Invalid` in the swap canister, which is a terminal state — no retry is possible without manual canister upgrade intervention. [10](#0-9) 

5. The SNS tokens already transferred to the neuron subaccount are permanently stranded: the neuron was never created, `error_refund_icp` only handles ICP, and there is no `error_refund_sns` equivalent in the swap canister.

## Impact Explanation

Direct participants who contribute exactly `min_participant_icp_e8s` ICP permanently lose their SNS token allocation when NF participation is large enough to push their per-neuron share below `neuron_minimum_stake_e8s`. Their ICP is committed to the swap treasury and their SNS tokens are locked in unclaimed neuron subaccounts with no on-chain recovery path. This constitutes a concrete, irreversible loss of user funds in the SNS framework — matching the **High** impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation

NF participation via matched funding is a standard, commonly used feature of SNS swaps. The NF contribution amount is computed dynamically and can be a substantial fraction of direct participation. SNS configurations that set `min_participant_icp_e8s` close to the neuron-minimum threshold — a common design choice to maximize participation breadth — are silently vulnerable whenever NF participation is active. No privileged access is required; the condition arises from normal, expected swap operation. The mismatch is systematic: every SNS swap with NF participation and parameters near the threshold is affected.

## Recommendation

Replace the denominator in both validation sites with the worst-case total participation, i.e., `max_direct_participation_icp_e8s + max_neurons_fund_participation_icp_e8s`. Additionally, use ceiling division so the check is conservative in the correct direction:

```rust
// In rs/sns/init/src/lib.rs and rs/sns/swap/src/types.rs:
let max_total_icp = max_direct_participation_icp_e8s
    .saturating_add(max_neurons_fund_participation_icp_e8s);
let min_participant_sns_e8s = (min_participant_icp_e8s as u128
    * initial_swap_amount_e8s as u128
    + max_total_icp as u128 - 1)
    / max_total_icp as u128;  // ceiling division
```

This ensures the validation is conservative relative to the worst-case distribution denominator at finalization time.

## Proof of Concept

**Parameters (all in e8s):**
- `min_participant_icp_e8s` = 2 × 10⁸ (2 ICP)
- `sns_token_e8s` = 1,000,000 × 10⁸
- `max_direct_participation_icp_e8s` = 100 × 10⁸ (100 ICP)
- `neuron_basket_count` = 3
- `neuron_minimum_stake_e8s` = 3,000
- `sns_transaction_fee_e8s` = 1,000
- `max_neurons_fund_participation_icp_e8s` = 100 × 10⁸ (100 ICP, matched 1:1)

**Validation check (passes):**
```
min_participant_sns_e8s = floor(2e8 × 1e14 / 100e8) = 20,000
threshold = 3 × (3,000 + 1,000) = 12,000
20,000 ≥ 12,000 → PASSES
```

**Actual distribution when NF contributes 100 ICP:**
```
total_participant_icp_e8s = 100e8 + 100e8 = 200e8
actual_sns = floor(2e8 × 1e14 / 200e8) = 10,000
per_neuron_stake = floor(10,000 / 3) - 1,000 = 2,333 - 1,000 = 1,333
1,333 < 3,000 (neuron_minimum_stake_e8s) → claim FAILS with ClaimedSwapNeuronStatus::Invalid
```

A deterministic integration test can be written using `SwapBuilder` with `with_neurons_fund_participation()`, setting parameters as above, running `finalize()` with a mock NF response contributing 100 ICP, and asserting that `claim_neuron_result.invalid > 0` while the SNS ledger subaccount for the affected buyer still holds tokens.
