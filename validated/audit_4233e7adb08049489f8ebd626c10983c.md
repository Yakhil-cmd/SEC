### Title
`notify_mint_cycles` Returns Overstated `minted` Amount Due to Cycles Ledger Fee Deduction - (File: rs/nns/cmc/src/main.rs)

### Summary
The `notify_mint_cycles` endpoint in the Cycles Minting Canister (CMC) returns a `NotifyMintCyclesSuccess` struct whose `minted` field reports the full converted cycles amount, but the user's cycles ledger account is actually credited with `cycles - CYCLES_LEDGER_FEE`. Any caller or integrating canister relying on the `minted` field for accounting will observe a systematic overstatement equal to the cycles ledger deposit fee.

### Finding Description
In `rs/nns/cmc/src/main.rs`, the function `process_mint_cycles` converts ICP to cycles and deposits them into the cycles ledger via `do_mint_cycles`. The cycles ledger's `deposit` endpoint deducts a fee (`CYCLES_LEDGER_FEE`) from the deposited amount before crediting the recipient account. However, the `minted` field in the returned `NotifyMintCyclesSuccess` is set to the pre-fee `cycles` value, not the post-fee amount actually credited:

```rust
Ok(NotifyMintCyclesSuccess {
    block_index: deposit_result.block_index,
    minted: cycles.into(),          // <-- full amount, fee NOT subtracted
    balance: deposit_result.balance, // <-- actual post-fee balance (correct)
})
```

The `balance` field correctly reflects the post-fee balance returned by the cycles ledger's `deposit` call, but `minted` overstates the credited amount by exactly `CYCLES_LEDGER_FEE`. This is confirmed by the integration test which asserts the actual balance is `100_000_000_000_000 - CYCLES_LEDGER_FEE` after a 1 ICP deposit, while `minted` would report `100_000_000_000_000`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Any canister or off-chain client that uses the `minted` field from `NotifyMintCyclesSuccess` to track cycles balances, perform accounting, or make downstream decisions will systematically overcount cycles by `CYCLES_LEDGER_FEE` per `notify_mint_cycles` call. Over many calls, this discrepancy accumulates. Canisters that gate logic on "did I receive at least X cycles?" using `minted` may proceed incorrectly when the actual credited amount is below their threshold. The `minted` field is documented as "Amount of cycles that were minted and deposited to the cycles ledger," which is factually incorrect since the deposited-to-user amount is `cycles - fee`. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
Every successful call to `notify_mint_cycles` that goes through the cycles ledger deposit path triggers this discrepancy. Any unprivileged principal can call this endpoint after sending ICP to the CMC's subaccount with `MEMO_MINT_CYCLES`. The discrepancy is deterministic and reproducible on every invocation. Integrating canisters that automate cycles management using the `minted` return value are directly affected. [6](#0-5) 

### Recommendation
Change `minted: cycles.into()` to report the actual post-fee amount credited to the user. Since `deposit_result.balance` is already available from the cycles ledger response, the CMC should either:
1. Compute `minted = cycles - cycles_ledger_fee` using the known fee, or
2. Have the cycles ledger's `deposit` endpoint return the net credited amount in `CyclesLedgerDepositResult` and use that value for `minted`.

Additionally, update the `NotifyMintCyclesSuccess` documentation to clarify whether `minted` represents the gross converted amount or the net credited amount. [7](#0-6) 

### Proof of Concept
1. A user sends 1 ICP to the CMC's subaccount with `MEMO_MINT_CYCLES`.
2. The user calls `notify_mint_cycles` with the resulting `block_index`.
3. The CMC converts 1 ICP → `100_000_000_000_000` cycles and calls the cycles ledger `deposit` endpoint with that amount.
4. The cycles ledger deducts `CYCLES_LEDGER_FEE` and credits `100_000_000_000_000 - CYCLES_LEDGER_FEE` to the user's account.
5. `notify_mint_cycles` returns `NotifyMintCyclesSuccess { minted: 100_000_000_000_000, balance: 100_000_000_000_000 - CYCLES_LEDGER_FEE, ... }`.
6. The `minted` field overstates the actual credited amount by `CYCLES_LEDGER_FEE`.

This is directly confirmed by the existing integration test:
```rust
assert_eq!(
    cycles_ledger_balance_of(&state_machine, main_account),
    100_000_000_000_000 - CYCLES_LEDGER_FEE  // actual balance
    // but minted field reports 100_000_000_000_000
);
``` [8](#0-7)

async fn do_mint_cycles(
    account: Account,
    cycles: Cycles,
    deposit_memo: Option<Vec<u8>>,
) -> Result<CyclesLedgerDepositResult, String> {
    let Some(cycles_ledger_canister_id) = with_state(|state| state.cycles_ledger_canister_id)
    else {
        return Err("No cycles ledger canister id configured.".to_string());
    };
    // Always use base cycles limit for minting cycles, since the Subnet Rental Canister
    // doesn't call endpoints using this function.
    ensure_balance(cycles, CyclesMintingLimiterSelector::BaseLimit)?;

    let arg = CyclesLedgerDepositArgs {
        to: account,
        memo: deposit_memo,
    };

    let result: CallResult<(CyclesLedgerDepositResult,)> = ic_cdk::api::call::call_with_payment128(
        cycles_ledger_canister_id.get().0,
        "deposit",
        (arg,),
        u128::from(cycles),
    )
    .await;

    result.map(|r| r.0).map_err(|(code, msg)| {
        format!(
            "Cycles ledger rejected deposit call with code {}: {:?}",
            code as i32, msg
        )
    })
}
