### Title
Forward/Proving Divergence: Production `run_block` Skips EOA Signature Validation While Prover Enforces It, Enabling Unprovable Batch Injection — (`forward_system/src/run/mod.rs`)

---

### Summary

The production sequencer forward-execution path (`run_block`) uses `BasicBootloaderForwardSimulationConfig`, which sets `VALIDATE_EOA_SIGNATURE: false`. This disables EOA signature verification entirely during sequencing. The proving system (`run_proving_inner`) uses `BasicBootloaderProvingExecutionConfig`, which sets `VALIDATE_EOA_SIGNATURE: true` and performs full ecrecover + `from`-address matching. An unprivileged attacker can submit a transaction with an invalid or forged signature. The sequencer includes it and mutates state. The prover rejects it with `InvalidTransaction::IncorrectFrom`, producing a different state. The batch becomes unprovable, halting the chain.

---

### Finding Description

**Root cause — config divergence:**

`BasicBootloaderForwardSimulationConfig` disables signature validation:

```rust
impl BasicBootloaderExecutionConfig for BasicBootloaderForwardSimulationConfig {
    const VALIDATE_EOA_SIGNATURE: bool = false;  // ← skips ecrecover
    const SIMULATION: bool = false;
}
``` [1](#0-0) 

`BasicBootloaderProvingExecutionConfig` enforces it:

```rust
impl BasicBootloaderExecutionConfig for BasicBootloaderProvingExecutionConfig {
    const SIMULATION: bool = false;
    const VALIDATE_EOA_SIGNATURE: bool = true;   // ← full ecrecover
}
``` [2](#0-1) 

**Production `run_block` (the public API) uses the no-validation config:**

```rust
run_forward::<BasicBootloaderForwardSimulationConfig>(oracle, &mut result_keeper, tracer, validator);
``` [3](#0-2) 

This is re-exported as the public API: [4](#0-3) 

**The proving binary uses the strict config:**

```rust
ProvingBootloader::<O, L>::run_prepared::<BasicBootloaderProvingExecutionConfig>(...)
``` [5](#0-4) 

**The validation branch that is skipped in forward mode:**

```rust
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // Only charge native cost — NO ecrecover, NO from-address check
    intrinsic_resources.charge(&Resources::from_native(ECRECOVER_NATIVE_COST))?;
} else {
    // Full ecrecover + from-address match
    if recovered_from != from {
        return Err(InvalidTransaction::IncorrectFrom { recovered, tx: from }.into());
    }
}
``` [6](#0-5) 

The test rig itself acknowledges the divergence with the comment:

```rust
// we use proving config here for benchmarking,
// although sequencer can have extra optimizations
run_forward_no_panic::<BasicBootloaderProvingExecutionConfig>(forward_oracle, ...)?;
``` [7](#0-6) 

The `docs/not-a-bug.md` does not list this behavior as intentional or expected. [8](#0-7) 

---

### Impact Explanation

An attacker submits a transaction with an invalid signature (e.g., `r`/`s` values that recover to an address ≠ `from`, or a zeroed signature). The sequencer's `run_block` includes it — no ecrecover is performed. The transaction's state effects (ETH transfer, contract call, nonce increment) are committed to the sequencer's state. When the prover runs `run_proving_inner` with `BasicBootloaderProvingExecutionConfig`, it performs ecrecover, finds `recovered_from != from`, and returns `Err(InvalidTransaction::IncorrectFrom)`. The transaction is excluded from the proven state. The prover's output state root diverges from the sequencer's. The batch is **unprovable**. The chain halts until the sequencer re-processes the batch, discarding the invalid transaction and rolling back its state effects. This is a **valid-execution unprovability / forward-proving divergence** with direct chain-liveness impact.

---

### Likelihood Explanation

Any unprivileged user can submit a transaction with an arbitrary signature directly to the sequencer. The bootloader's own signature check is disabled in forward mode. The external `TxValidator` hook is called only for L2 transactions and is operator-supplied — it is not guaranteed to validate signatures. The attack requires no special access, no leaked keys, and no governance majority. It is trivially reproducible by crafting any RLP or ABI-encoded transaction with a zeroed or randomized signature field.

---

### Recommendation

Change the production `run_block` function to use `BasicBootloaderForwardETHLikeConfig` (which has `VALIDATE_EOA_SIGNATURE: true`) or `BasicBootloaderProvingExecutionConfig` instead of `BasicBootloaderForwardSimulationConfig`. The sequencer and prover must use identical validation logic for all security-critical checks. If skipping signature validation is desired as a sequencer optimization, the prover must also skip it — or the sequencer must guarantee (via its `TxValidator`) that no transaction with an invalid signature ever reaches the bootloader.

---

### Proof of Concept

1. Craft a legacy EIP-155 transaction with `from = 0xAAAA...`, `nonce = 0`, `value = 1 ETH`, and a zeroed signature (`r = 0`, `s = 0`, `v = 37`).
2. Submit it to the sequencer via `run_block` (using `BasicBootloaderForwardSimulationConfig`).
3. Observe: the transaction is accepted; the sequencer's state shows `0xAAAA...` balance decremented by 1 ETH.
4. Run the prover via `run_proving_inner` (using `BasicBootloaderProvingExecutionConfig`) on the same oracle data.
5. Observe: ecrecover on the zeroed signature returns `B160::ZERO ≠ 0xAAAA...`; the prover returns `InvalidTransaction::IncorrectFrom`; the transaction is excluded; the prover's state root differs from the sequencer's.
6. The batch cannot be proven. The chain halts.

# Not a Bug

This page lists ZKsync OS behavior that is intentional, protocol-correct, or already documented, but is commonly misreported as a vulnerability.

Use this page as a triage aid, not as a substitute for checking the code. If an observed behavior differs from what is described here, investigate it normally.

## EVM / Cancun Behavior

### Cancun is the supported EVM hardfork

ZKsync OS currently targets Cancun EVM semantics. Reports that assume later hardfork behavior are not valid unless they identify an issue under the supported hardfork.

See [EVM Execution Environment](./execution_environments/evm.md).

### EIP-4844 transactions are disabled in production

EIP-4844 type `0x03` transactions are implemented behind the `basic_bootloader/eip-4844` feature, but this feature is not enabled by the production feature sets. It is enabled for test and Ethereum test-runner configurations.

See [Transaction formats](./bootloader/transaction_format.md).

### `BLOBHASH` returns `0`

This is expected in production. Since EIP-4844 transactions are disabled, transactions do not have blob versioned hashes. Cancun/EIP-4844 specifies that `BLOBHASH(index)` returns a zeroed `bytes32` when `index` is outside `tx.blob_versioned_hashes`.

### `BLOBBASEFEE` returns `1`

This is expected in production. There are no EIP-4844 transactions in production history, so `excess_blob_gas` remains `0`. Cancun/EIP-4844 defines the minimum blob base fee as `1`, and EIP-7516 makes `BLOBBASEFEE` return the current block's blob base fee.

### Opcode `0x44` returns `1`

Opcode `0x44` is `PREVRANDAO` under Cancun semantics, historically named `DIFFICULTY`. ZKsync OS uses the block `mix_hash` value for this opcode. In production, `mix_hash` is mocked to `1`.

This does not contradict the block header `difficulty` field being `0` after the Merge.

See [Bootloader block header](./bootloader/bootloader.md#block-header).

### `SUB` operand order

ZKsync OS follows the standard EVM operand order for `SUB`. The top stack item is the first operand, and the item below it is the second operand. For example, `PUSH1 0x03 PUSH1 0x05 SUB` leaves `0x02` on the stack.

This is not a reversed subtraction bug.
