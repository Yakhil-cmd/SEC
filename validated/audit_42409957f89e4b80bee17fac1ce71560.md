Audit Report

## Title
Missing Self-Address Guard in `withdraw_eth` and `withdraw_erc20` Allows Permanent User Fund Loss - (File: rs/ethereum/cketh/minter/src/address.rs, rs/ethereum/cketh/minter/src/main.rs)

## Summary
The ckETH minter's `validate_address_as_destination` function does not check whether the supplied recipient Ethereum address equals the minter's own tECDSA-derived Ethereum address. An unprivileged user who calls `withdraw_eth` or `withdraw_erc20` with the minter's own address as the recipient will have their ckETH/ckERC20 tokens permanently burned while the corresponding ETH/ERC20 is sent back to the minter's address, where it becomes permanently inaccessible and unaccounted for. The ckBTC minter contains an explicit guard against the analogous pattern; the ckETH minter does not.

## Finding Description

**Root cause — `validate_address_as_destination` (rs/ethereum/cketh/minter/src/address.rs, lines 47–56):**

```rust
pub fn validate_address_as_destination(address: &str) -> Result<Address, AddressValidationError> {
    let address =
        Address::from_str(address).map_err(|e| AddressValidationError::Invalid { error: e })?;
    if address == Address::ZERO {
        return Err(AddressValidationError::NotSupported(address));
    }
    if crate::blocklist::is_blocked(&address) {
        return Err(AddressValidationError::Blocked(address));
    }
    Ok(address)
}
```

The function rejects only the zero address and OFAC-blocklisted addresses. The minter's own Ethereum address — derived from its threshold ECDSA key and publicly queryable via `minter_address()` — is neither zero nor on the blocklist, so it passes validation unconditionally.

**Contrast with ckBTC minter (rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs, lines 156–160 and 254–258):**

```rust
let main_address_str = state::read_state(|s| runtime.derive_minter_address_str(s));
if args.address == main_address_str {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

Both `retrieve_btc` and `retrieve_btc_with_approval` explicitly trap when the destination equals the minter's own address. No equivalent guard exists anywhere in the ckETH minter.

**Exploit flow for `withdraw_eth` (rs/ethereum/cketh/minter/src/main.rs, lines 265–340):**

1. Attacker (or victim) queries `minter_address()` to obtain the minter's Ethereum address `M`.
2. Caller approves the minter on the ckETH ledger for amount `A`.
3. Caller calls `withdraw_eth({ amount: A, recipient: M })`.
4. `validate_address_as_destination(M)` passes — `M` is not zero and not blocklisted.
5. `burn_from(caller_account, A, ...)` succeeds; `A` wei of ckETH is permanently burned.
6. An `EthWithdrawalRequest { destination: M, withdrawal_amount: A }` is queued.
7. The minter signs and broadcasts an Ethereum transaction sending `A` wei to `M` (itself).
8. The ETH arrives at the minter's own address. The minter's internal `eth_balance` is decremented by `A` as if the ETH left the system.
9. The ETH is now "extra" at the minter's address: the ckETH supply is reduced by `A` but the ETH is still present and untracked. It can never be claimed because there is no ckETH to burn for it.

The same flow applies to `withdraw_erc20` (lines 389–543): ckETH fee tokens and ckERC20 tokens are both burned, and the ERC20 transfer is directed to the minter's own Ethereum address, permanently locking the ERC20 tokens.

**Why existing checks are insufficient:**
- The zero-address check does not cover the minter's address.
- The OFAC blocklist (`ETH_ADDRESS_BLOCKLIST`, rs/ethereum/cketh/minter/src/blocklist.rs) contains sanctioned third-party addresses; the minter's own address is not and should not be on it.
- There is no post-burn guard, no state-level invariant, and no test covering this case.

## Impact Explanation

Permanent, irreversible loss of user ck-token assets (ckETH or ckERC20) with no recovery path. The burned tokens are gone; the ETH/ERC20 at the minter's address is permanently orphaned because the minter's accounting treats it as withdrawn while it is actually still present. This matches the allowed impact: **"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets"** and **"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."**

Severity: **High** ($2,000–$10,000). The loss is permanent and repeatable, but each execution harms only the caller's own funds (not other users' funds), and the attacker must know the minter's Ethereum address (publicly available).

## Likelihood Explanation

The minter's Ethereum address is publicly queryable by any caller via the `minter_address()` update endpoint and is displayed on the minter's HTTP dashboard. No special privilege is required. The attack is deterministic and repeatable. A user could trigger it accidentally (e.g., copy-paste error) or deliberately (self-harm, griefing the protocol's accounting). The minimum withdrawal amount is the only economic barrier.

## Recommendation

Add a check in `validate_address_as_destination` (or at the call sites in `withdraw_eth` and `withdraw_erc20`) that rejects the minter's own Ethereum address as a destination, mirroring the ckBTC minter's guard:

```rust
// In withdraw_eth / withdraw_erc20, after validate_address_as_destination:
if let Some(own_address) = read_state(|s| s.minter_address()) {
    if destination == own_address {
        ic_cdk::trap("illegal withdrawal target: minter's own address");
    }
}
```

Alternatively, extend `AddressValidationError` with a `SelfAddress` variant and handle it in `validate_address_as_destination` by passing the minter's current address as a parameter. Add a corresponding integration test analogous to the ckBTC test at `rs/tests/ckbtc/ckbtc_minter_deposit_and_withdrawal.rs` lines 188–202.

## Proof of Concept

**Minimal PocketIC / StateMachine integration test plan:**

1. Deploy the ckETH minter and ledger using `CkEthSetup::default()` (existing test harness).
2. Perform a deposit to give the test caller a ckETH balance.
3. Call `ledger.icrc2_approve(minter_id, balance)`.
4. Query `minter_address()` to obtain `M`.
5. Call `minter.withdraw_eth({ amount: MIN_WITHDRAWAL, recipient: M })`.
6. Assert the call returns `Ok(...)` (currently it does — demonstrating the missing guard).
7. Advance the state machine to process the withdrawal.
8. Assert the caller's ckETH balance is now zero (burned).
9. Assert the minter's internal `eth_balance` is lower than the actual ETH at address `M` on the simulated chain — demonstrating the accounting discrepancy and permanent lock.
