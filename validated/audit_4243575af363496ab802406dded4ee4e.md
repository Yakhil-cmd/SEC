### Title
Unchecked `args.amount` in `ft_on_transfer` Enables ERC-20 Over-Minting for Fee-on-Transfer NEP-141 Tokens - (`engine/src/engine.rs`)

---

### Summary

The Aurora Engine's `ft_on_transfer` handler unconditionally mints ERC-20 tokens equal to `args.amount` — the nominal amount reported by the calling NEP-141 contract — without verifying the actual balance change on the NEP-141 side. Because any user can permissionlessly register any NEP-141 token via `deploy_erc20_token`, a fee-on-transfer or deflationary NEP-141 token can be bridged such that Aurora mints more ERC-20 tokens than the NEP-141 tokens it actually holds, creating a permanent insolvency condition for that token's bridge.

---

### Finding Description

**Entry point 1 — permissionless token registration:**

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` has no owner or access-control guard:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // No require_owner / whitelist check
        let bytes = io.read_input().to_vec();
        let args = DeployErc20TokenArgs::deserialize(&bytes)...;
        ...
    })
}
```

Any NEAR account can call this to register any NEP-141 account ID and create a corresponding ERC-20 mirror on Aurora.

**Entry point 2 — `ft_on_transfer` trusts caller-supplied amount:**

When a NEP-141 contract calls `ft_on_transfer` on Aurora (via `ft_transfer_call`), the handler in `engine/src/contract_methods/connector.rs` dispatches to `receive_erc20_tokens`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(&predecessor_account_id, &args, ...)
};
```

`receive_erc20_tokens` in `engine/src/engine.rs` takes `args.amount` at face value and mints that exact quantity of ERC-20 tokens:

```rust
pub fn receive_erc20_tokens<P: PromiseHandler>(...) -> ... {
    let amount = args.amount.as_u128();   // ← taken directly from callback args
    ...
    setup_receive_erc20_tokens_input(&recipient, amount)  // ← mints `amount` ERC-20
```

There is no balance-before / balance-after check to confirm that Aurora's NEP-141 balance actually increased by `amount`. The engine fully trusts the `amount` field supplied by the calling NEP-141 contract.

---

### Impact Explanation

**Impact: Critical — Insolvency.**

For a fee-on-transfer NEP-141 token (one that deducts a percentage on every `ft_transfer`), the sequence is:

1. Aurora holds 0 NEP-141 tokens.
2. User calls `ft_transfer_call` with `amount = 1000`; the token deducts 10%, so Aurora receives 900 NEP-141 tokens.
3. The NEP-141 contract calls `ft_on_transfer` on Aurora with `amount = 1000`.
4. Aurora mints 1000 ERC-20 tokens.
5. Aurora now holds 900 NEP-141 but has 1000 ERC-20 in circulation.
6. The first 900 ERC-20 holders can exit successfully; the last 100 ERC-20 holders cannot redeem — their funds are permanently frozen/lost.

This is a direct bridge insolvency: the ERC-20 supply exceeds the NEP-141 collateral held by Aurora.

---

### Likelihood Explanation

**Likelihood: High.**

- `deploy_erc20_token` is permissionless — any NEAR account can register any NEP-141 token.
- Non-standard NEP-141 tokens (fee-on-transfer, rebasing, deflationary) exist on NEAR mainnet.
- No monitoring or whitelist mechanism prevents such tokens from being registered.
- The exploit requires only two ordinary NEAR transactions (register + bridge), with no privileged access.

---

### Recommendation

1. **Short term:** Add a balance-before / balance-after check inside `receive_erc20_tokens` (analogous to `checkBalance` in the reference report): query the NEP-141 balance of the Aurora contract before and after the transfer, and mint only the actual difference.
2. **Long term:** Restrict `deploy_erc20_token` to the contract owner or a governance-approved whitelist, so only vetted NEP-141 tokens can be bridged. Alternatively, publish a token integration checklist and enforce it on-chain.

---

### Proof of Concept

1. Alice deploys a NEP-141 token `fee_token.near` that deducts 10% on every `ft_transfer`.
2. Alice calls `deploy_erc20_token` on Aurora Engine with `fee_token.near` — succeeds with no access control.
3. Alice calls `ft_transfer_call` on `fee_token.near` with `amount = 1_000_000`, `receiver_id = aurora`, `msg = <alice_evm_address>`.
4. `fee_token.near` transfers 900_000 tokens to Aurora (10% fee deducted) and calls `ft_on_transfer` on Aurora with `amount = 1_000_000`.
5. Aurora's `receive_erc20_tokens` (`engine/src/engine.rs:803`) reads `args.amount = 1_000_000` and calls `setup_receive_erc20_tokens_input(&recipient, 1_000_000)` (`engine/src/engine.rs:831`), minting 1_000_000 ERC-20 tokens.
6. Aurora holds 900_000 NEP-141 but has 1_000_000 ERC-20 in circulation.
7. Alice exits 1_000_000 ERC-20 via `withdrawToNear`; the `ft_transfer` to Alice for 1_000_000 NEP-141 fails because Aurora only holds 900_000 — the bridge is insolvent for this token. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);

    [selector, tail.as_slice()].concat()
}
