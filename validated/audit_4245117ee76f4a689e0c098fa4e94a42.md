Audit Report

## Title
IPv4-mapped IPv6 addresses bypass IPv4 CIDR rate-limit and block rules in `Bucket::evaluate` — (`rs/boundary_node/ic_boundary/src/rate_limiting/generic.rs`)

## Summary
`Bucket::evaluate` compares `rule.ip` (an `IpNet`) against `ctx.ip` (an `IpAddr`) using `IpNet::contains`, which performs no cross-family normalization. On a dual-stack Linux socket bound to `::` without `IPV6_V6ONLY`, an IPv4 client's address is presented to userspace as `IpAddr::V6(::ffff:x.x.x.x)`. The `ipnet` crate returns `false` for `IpNet::V4(10.0.0.0/24).contains(IpAddr::V6(::ffff:10.0.0.1))`, causing the rule to be silently skipped and `Decision::Pass` returned, completely bypassing any operator-configured IPv4 CIDR block or rate-limit rule.

## Finding Description
At `generic.rs` lines 140–144, the IP filter guard is:

```rust
if let Some(v) = self.rule.ip
    && !v.contains(&ctx.ip)
{
    return None;  // rule skipped → caller falls through to Decision::Pass
}
```

`ctx.ip` is populated at line 491 as `conn_info.remote_addr.ip()` with no call to `to_canonical()` or `to_ipv4_mapped()`. The `RateLimitRule.ip` field is typed `Option<IpNet>` (v1.rs line 158), so an operator-configured `10.0.0.0/24` rule is stored as `IpNet::V4(...)`. On a dual-stack Linux socket (the default when binding to `::`), an IPv4 client connecting to the boundary node is presented to userspace as `IpAddr::V6(::ffff:10.0.0.1)`. The `ipnet` crate's `IpNet::contains` performs a strict address-family check and unconditionally returns `false` when the families differ, so the rule is skipped. The `GenericLimiter::evaluate` loop at lines 403–410 falls through to `Decision::Pass` when no bucket matches.

Notably, the metrics middleware at `metrics.rs` line 473 explicitly calls `.to_canonical()` on the same `remote_addr.ip()` value for logging, demonstrating that the developers are aware of the IPv4-mapped representation but did not apply the same normalization in the rate-limiting path.

## Impact Explanation
Any operator-configured `ip`-scoped rule targeting an IPv4 CIDR is completely ineffective against clients connecting via IPv4-mapped IPv6 addresses. This allows clients from a blocked IPv4 range to bypass the block entirely, and clients from a rate-limited IPv4 range to send unlimited requests to the boundary node. This constitutes a significant boundary/API infrastructure security impact with concrete user and protocol harm — matching the High ($2,000–$10,000) impact class for "Significant boundary/API... infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The boundary node is a public-facing component accepting untrusted traffic. Dual-stack sockets without `IPV6_V6ONLY` are the Linux default. IPv4 CIDR rules are a natural operator configuration (e.g., blocking an abusive IPv4 range). No special privileges are required; any client on a dual-stack network automatically connects via IPv4-mapped IPv6 when the server socket is dual-stack. The bug is locally reproducible with a unit test. The `ip_prefix_group` sharded path at lines 155–163 is also affected: an IPv4-mapped address takes the `V6` branch and uses `prefix.v6` for sharding, further diverging from the intended IPv4 rule behavior.

## Recommendation
Normalize `ctx.ip` before comparison using `IpAddr::to_canonical()` (stabilized in Rust 1.75):

```rust
let ip = ctx.ip.to_canonical();

if let Some(v) = self.rule.ip
    && !v.contains(&ip)
{
    return None;
}
```

Apply the same normalization in the `ip_prefix_group` sharded branch at lines 155–163 so that IPv4-mapped addresses use `prefix.v4` and are sharded correctly.

## Proof of Concept

```rust
#[test]
fn test_ipv4_mapped_bypass() {
    use std::str::FromStr;
    use ipnet::IpNet;
    use std::net::IpAddr;

    let rule_ip: IpNet = "10.0.0.0/24".parse().unwrap();
    // IPv4-mapped IPv6 address as presented by a dual-stack Linux socket
    let client_ip: IpAddr = "::ffff:10.0.0.1".parse().unwrap();

    // ipnet returns false — address families differ, rule is skipped
    assert!(!rule_ip.contains(&client_ip));

    // With canonicalization the match works correctly
    assert!(rule_ip.contains(&client_ip.to_canonical()));
}
```
