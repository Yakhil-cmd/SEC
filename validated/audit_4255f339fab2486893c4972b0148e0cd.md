### Title
Unbounded Filter Map Growth via Unauthenticated `eth_newFilter` / `eth_newBlockFilter` Causes RPC Node Memory Exhaustion and Mutex Starvation — (`evmrpc/filter.go`)

---

### Summary

`NewFilter` and `NewBlockFilter` accept unauthenticated JSON-RPC calls and insert entries into the shared `a.filters` map with no cap on the total number of filters and no rate limiting. The `cleanupLoop` goroutine only evicts entries after `FilterTimeout` (default 120 s). An attacker can flood the node with filter-creation requests, exhausting process memory and causing `cleanupExpiredFilters` to hold the `filtersMu` mutex for extended periods, starving all concurrent filter operations.

---

### Finding Description

`NewFilterAPI` starts `cleanupLoop` as a background goroutine that ticks every `timeout/2` (default 60 s) and calls `cleanupExpiredFilters`. [1](#0-0) 

`cleanupLoop` itself is not attacker-reachable; the vulnerability is in the two public entry points it manages:

**`NewFilter`** (line 397) and **`NewBlockFilter`** (line 422) both:
- Acquire `filtersMu.Lock()`
- Insert a new `filter` struct into `a.filters` (an unbounded `map[ethrpc.ID]filter`)
- Return immediately with no rate-limit check and no map-size guard [2](#0-1) 

The only rate limiter in `FilterAPI` is `globalRPSLimiter`, which is checked exclusively inside `GetLogs` for queries exceeding `RPSLimitThreshold` blocks: [3](#0-2) 

It is never consulted by `NewFilter` or `NewBlockFilter`. No `maxFilters` constant or map-size check exists anywhere in the file.

Default configuration sets `FilterTimeout = 120 * time.Second`: [4](#0-3) 

So filters accumulate for up to 120 s before the cleanup tick removes them. During that window the map is unbounded.

`cleanupExpiredFilters` iterates the entire map under a read lock, then re-acquires a write lock to delete: [5](#0-4) 

With millions of entries this iteration is O(N) under the mutex, blocking every concurrent `NewFilter`, `GetFilterChanges`, `GetFilterLogs`, and `UninstallFilter` call for the duration.

---

### Impact Explanation

Two compounding effects:

1. **Memory exhaustion.** Each `filter` struct holds a `filters.FilterCriteria` that can carry large `Addresses []common.Address` and `Topics [][]common.Hash` slices. The go-ethereum RPC layer enforces a per-request body cap (`maxRequestContentLength`, ~5 MB), so each call can allocate up to ~5 MB of heap. At a modest 500 req/s sustained for 120 s, the live filter map holds 60,000 entries × up to 5 MB = up to ~300 GB of referenced heap, triggering OOM and process termination.

2. **Mutex starvation.** Even with small criteria, millions of map entries cause `cleanupExpiredFilters` to hold `filtersMu` for seconds, making the entire filter subsystem unresponsive. This crashes or hangs the RPC node from the perspective of all connected clients.

Both effects are reachable from a single unauthenticated HTTP connection on the default port 8545.

---

### Likelihood Explanation

- The endpoint is enabled by default (`HTTPEnabled: true`, port 8545, `CORSOrigins: "*"`).
- No authentication, no per-IP rate limit, no filter-count cap.
- A single attacker machine can sustain hundreds to thousands of HTTP requests per second.
- The 120-second accumulation window is long enough to build a map large enough to exhaust memory on typical validator/RPC hardware.

---

### Recommendation

1. **Cap the filter map size.** Reject `NewFilter` / `NewBlockFilter` when `len(a.filters)` exceeds a configurable `MaxFilters` (e.g., 10,000).
2. **Rate-limit filter creation.** Apply `globalRPSLimiter` (or a dedicated limiter) inside `NewFilter` and `NewBlockFilter`, not only inside `GetLogs`.
3. **Limit `FilterCriteria` size.** Reject criteria with more than N addresses or M topic sets before inserting into the map.
4. **Shorten the default `FilterTimeout`** or add a hard upper bound on how long a filter can live regardless of access.

---

### Proof of Concept

```python
import requests, threading, time

URL = "http://<rpc-node>:8545"
LARGE_ADDRS = ["0x" + ("aa" * 20)] * 5000  # ~100 KB of addresses per request

def flood():
    while True:
        requests.post(URL, json={
            "jsonrpc": "2.0", "id": 1,
            "method": "eth_newFilter",
            "params": [{"address": LARGE_ADDRS, "fromBlock": "latest"}]
        }, timeout=5)

threads = [threading.Thread(target=flood) for _ in range(50)]
for t in threads: t.start()
time.sleep(120)
# Node OOMs or filter mutex stalls all RPC responses
```

Each request inserts a new entry into `a.filters` with no eviction until the 120-second cleanup tick. Fifty concurrent threads sustaining ~500 req/s accumulate 60,000 entries × ~100 KB = ~6 GB of live heap within the first cleanup window, with no server-side rejection path. func (a *FilterAPI) cleanupExpiredFilters(timeout time.Duration) {
```go
	now := time.Now()
	toRemove := make([]ethrpc.ID, 0)

	// First pass: identify expired filters (read lock)
	a.filtersMu.RLock()
	for id, filter := range a.filters {
		if now.Sub(filter.lastAccess) > timeout {
			toRemove = append(toRemove, id)
		}
	}
	a.filtersMu.RUnlock()

	// Second pass: remove expired filters (write lock)
	if len(toRemove) > 0 {
		a.filtersMu.Lock()
		for _, id := range toRemove {
			if filter, exists := a.filters[id]; exists {
				delete(a.filters, id)
				if filter.cancelFunc != nil {
					filter.cancelFunc()
				}
			}
		}
		a.filtersMu.Unlock()
	}
}
```
