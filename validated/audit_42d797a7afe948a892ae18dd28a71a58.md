### Title
ERC-20 Mirror Bridge Does Not Account for Rebasing NEP-141 Token Balances, Permanently Freezing Accrued Yield — (`etc/eth-contracts/contracts/EvmErc20.sol`, `engine/src/engine.rs`, `engine-precompiles/src/native.rs`)

---

### Summary

The Aurora Engine's NEP-141 ↔ ERC-20 bridge mints and burns ERC-20 tokens in a strict 1:1 ratio with deposited/withdrawn NEP-141 amounts. It does not account for rebasing NEP-141 tokens (tokens whose on-chain balance increases autonomously, such as liquid staking tokens like stNEAR). When such a token is bridged into Aurora and its balance accrues rewards inside Aurora's account, the rebased excess is permanently frozen: no ERC-20 tokens are ever minted for it, and the only exit path for NEP-141 tokens requires burning ERC-20 tokens. The excess is irrecoverable.

---

### Finding Description

**Deposit path** (`ft_on_transfer` → `receive_erc20_tokens`):

When a user bridges a NEP-141 token into Aurora, `ft_on_transfer` is called on the Aurora contract. For non-base tokens, `receive_erc20_tokens` is invoked, which calls `setup_receive_erc20_tokens_input(&recipient, amount)` — minting exactly `amount` ERC-20 tokens to the recipient. The ERC-20 total supply is therefore always equal to the total NEP-141 deposited, never more. [1](#0-0) 

**Withdrawal path** (`withdrawToNear` / `withdrawToEthereum` → `exit_erc20_token_to_near`):

The only way to retrieve NEP-141 tokens from Aurora is through `EvmErc20.withdrawToNear` or `withdrawToEthereum`, which first `_burn`s the caller's ERC-20 tokens, then calls the `ExitToNear` precompile. The precompile constructs an `ft_transfer` call to the NEP-141 contract for exactly the burned amount. [2](#0-1) [3](#0-2) 

**The gap**: If the underlying NEP-141 token is rebasing (its balance held by Aurora's account grows autonomously between deposit and withdrawal), Aurora's actual NEP-141 balance exceeds the ERC-20 total supply. The excess rebased tokens:

1. Cannot be withdrawn via `withdrawToNear`/`withdrawToEthereum` — no ERC-20 tokens exist to burn for them.
2. Cannot be recovered by any admin function — no such function exists in the production contract.
3. Are permanently locked in Aurora's NEP-141 account with no exit path. [4](#0-3) 

---

### Impact Explanation

**High — Theft of unclaimed yield / Permanent freezing of rebased funds.**

Any rebasing NEP-141 token bridged to Aurora will have its autonomously accrued balance permanently frozen inside Aurora's account. The token holders who deposited the underlying NEP-141 are the rightful owners of the rebased yield, but they can never retrieve it. The principal (originally deposited amount) can still be withdrawn, but all rebased excess is irrecoverably lost to the users and locked in the contract forever. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The NEAR ecosystem has rebasing liquid staking tokens (e.g., Meta Pool's stNEAR). Any such token that is deployed as an ERC-20 mirror on Aurora via `deploy_erc20_token` and then bridged by users will trigger this condition. The rebasing happens passively and continuously — no attacker action is required. The loss accumulates silently over time proportional to the staking yield rate and the total bridged amount. [6](#0-5) 

---

### Recommendation

The bridge accounting must be made aware of rebasing. Two approaches:

1. **Snapshot-based accounting**: Record the NEP-141 balance held by Aurora at deposit time and at withdrawal time. Allow users to withdraw their proportional share of the actual current balance (not just the originally minted ERC-20 amount), adjusting ERC-20 balances accordingly.

2. **Admin sweep function**: Add a privileged function that can detect and redistribute the surplus NEP-141 balance (actual NEP-141 balance minus ERC-20 total supply) — either by minting additional ERC-20 tokens pro-rata to existing holders, or by transferring the surplus to a designated recovery address. [4](#0-3) [7](#0-6) 

---

### Proof of Concept

1. Deploy a rebasing NEP-141 token `rebase.near` on NEAR (e.g., a liquid staking token whose balance increases each epoch).
2. Call `deploy_erc20_token` on Aurora to create the ERC-20 mirror.
3. User Alice holds 1,000 `rebase.near` tokens. She calls `ft_transfer_call` on `rebase.near` to send 1,000 tokens to Aurora. Aurora's `ft_on_transfer` fires, `receive_erc20_tokens` mints 1,000 ERC-20 tokens to Alice's EVM address.
4. One epoch passes. `rebase.near` rebases: Aurora's balance of `rebase.near` increases from 1,000 to 1,050 (5% staking yield).
5. Alice calls `withdrawToNear(alice.near, 1000)` on the ERC-20 contract. This burns 1,000 ERC-20 tokens and calls `ft_transfer` for 1,000 tokens. This succeeds — Alice gets her 1,000 tokens back.
6. The remaining 50 `rebase.near` tokens sit in Aurora's account. No ERC-20 tokens exist to burn for them. No admin function can retrieve them. They are permanently frozen. [2](#0-1) [8](#0-7) [3](#0-2)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
