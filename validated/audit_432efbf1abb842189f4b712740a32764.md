### Title
Unbounded `rewards` Array in `reward_node_providers()` Iterated Without Instruction-Limit Guard, Enabling Governance Proposal Execution Trap - (File: rs/nns/governance/src/governance.rs)

### Summary

The `reward_node_providers()` function in the NNS Governance canister iterates over an unbounded `rewards: Vec<RewardNodeProvider>` slice without any instruction-limit check between iterations. Each iteration performs an async inter-canister call to the ICP ledger (`transfer_funds`). A `RewardNodeProviders` proposal with a sufficiently large `rewards` array can exhaust the canister's instruction budget mid-execution, causing the entire async message to trap. Because the trap occurs inside the spawned async future (not inside a recoverable `Result`-returning context), the proposal's execution status is never updated, and the governance canister's state is left partially mutated — some node providers may have already received ICP while the proposal is permanently stuck in a non-terminal state.

### Finding Description

`reward_node_providers_from_proposal()` is called when a `RewardNodeProviders` proposal is executed:

```rust
// rs/nns/governance/src/governance.rs:4008-4021
async fn reward_node_providers_from_proposal(
    &mut self,
    pid: u64,
    reward_nps: RewardNodeProviders,
) {
    let result = if reward_nps.use_registry_derived_rewards == Some(true) {
        self.mint_monthly_node_provider_rewards().await
    } else {
        self.reward_node_providers(&reward_nps.rewards).await  // <-- unbounded
    };
    self.set_proposal_execution_status::<()>(pid, result.map(|()| vec![]));
}
```

`reward_node_providers()` iterates over the full `rewards` slice with no bound or instruction-limit check:

```rust
// rs/nns/governance/src/governance.rs:3987-4006
async fn reward_node_providers(
    &mut self,
    rewards: &[RewardNodeProvider],
) -> Result<(), GovernanceError> {
    let mut result = Ok(());
    for reward in rewards {                                    // <-- no bound, no limit check
        let reward_result = self.reward_node_provider_helper(reward).await;
        ...
        result = result.or(reward_result);
    }
    result
}
```

Each call to `reward_node_provider_helper()` performs at least one async ledger `transfer_funds` call. The loop has no instruction-limit guard analogous to the one used in `continue_processing()` in `distribution.rs`:

```rust
// rs/nns/governance/src/reward/distribution.rs:184-186
if is_over_instructions_limit() {
    break;
}
```

The `RewardNodeProviders` protobuf field `rewards` is a `repeated RewardNodeProvider` with no enforced maximum length at the proposal submission layer. A governance proposal with a large `rewards` array (when `use_registry_derived_rewards` is `false`) passes validation and is accepted for execution.

### Impact Explanation

When the instruction limit is exceeded mid-loop, the IC runtime traps the executing message. The async future spawned by `start_proposal_execution()` is dropped without ever reaching `set_proposal_execution_status()`. The result is:

1. **Partial ICP minting**: Node providers processed before the trap have already received ICP (ledger state is committed at each `await` point). Those after the trap do not.
2. **Proposal stuck in non-terminal state**: The proposal's `executed_timestamp_seconds` and `failed_timestamp_seconds` are never set, leaving it in a permanently pending execution state. The governance canister cannot re-execute it.
3. **Ledger conservation break**: ICP is minted to a subset of intended recipients with no record of failure, and no mechanism to retry or complete the remaining transfers.

This is a **cycles/resource accounting bug** and **ledger conservation bug** — the governance canister's own execution exhausts its instruction budget, causing partial state mutation with no recovery path.

### Likelihood Explanation

The `RewardNodeProviders` proposal action is a standard NNS governance action. Any NNS neuron holder with sufficient voting power to pass a proposal (or a coalition of neurons) can submit a `RewardNodeProviders` proposal with an arbitrarily large `rewards` array. The `rewards` field is `repeated` in the protobuf with no enforced maximum. The NNS governance canister's proposal validation does not cap the length of `reward_nps.rewards`. This is reachable by any governance participant — no privileged access is required beyond the ability to submit and pass an NNS proposal.

### Recommendation

1. **Bound the `rewards` array at proposal submission time**: Enforce a maximum length on `reward_nps.rewards` during proposal validation (analogous to how `_distributeFunds()` caps `MAX_ROYALTY_RECIPIENTS_INDEX` in the referenced NFT report).
2. **Add an instruction-limit guard inside `reward_node_providers()`**: Check `is_message_over_threshold()` between iterations, as is already done in `continue_processing()` in `rs/nns/governance/src/reward/distribution.rs:184`.
3. **Use a resumable state machine**: Persist partially-completed reward distributions to stable memory (as `RewardsDistributionInProgress` already does for neuron maturity rewards) so that execution can be resumed across multiple messages if the instruction limit is hit.

### Proof of Concept

1. Construct a `RewardNodeProviders` proposal with `use_registry_derived_rewards = false` and a `rewards` vec containing hundreds of valid `RewardNodeProvider` entries (each with a valid node provider principal and `RewardToAccount` mode).
2. Submit and pass the proposal through NNS governance.
3. Observe that `reward_node_providers_from_proposal()` is invoked, which calls `reward_node_providers()` iterating over all entries without any instruction-limit check.
4. With a sufficiently large array, the canister exhausts its per-message instruction budget mid-loop, trapping the async future.
5. `set_proposal_execution_status()` is never called; the proposal remains in a non-terminal state. ICP has been minted to the first N node providers; the remaining entries receive nothing and cannot be retried.

The contrast with the already-fixed neuron reward distribution path is direct: `continue_processing()` in `rs/nns/governance/src/reward/distribution.rs` uses `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` and a resumable `StableBTreeMap`-backed state machine, while `reward_node_providers()` has neither.
