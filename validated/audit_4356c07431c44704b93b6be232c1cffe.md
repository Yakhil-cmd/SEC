Audit Report

## Title
Unprivileged Caller Can Force Any Principal With ICP in Swap Subaccount Into SNS Token Swap Participation - (File: rs/sns/swap/canister/canister.rs)

## Summary

`refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` accepts a caller-supplied `buyer` principal with no check that the caller matches the specified buyer. Any unprivileged ingress sender can invoke this method with a victim's principal, causing the victim's ICP already sitting in the swap's derived subaccount to be irrevocably committed as swap participation. The victim's ICP is then locked until the swap finalizes, and if the swap commits, the victim receives SNS tokens they never consented to purchase.

## Finding Description

In `rs/sns/swap/canister/canister.rs` at lines 127–143, `refresh_buyer_tokens` resolves the effective buyer from the caller-supplied `arg.buyer` string with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // any principal accepted
};
```

The resolved principal `p` is passed directly to `refresh_buyer_token_e8s`, which reads the ICP balance of the swap canister's subaccount derived from `buyer` (`principal_to_subaccount(&buyer)`) and, if the balance meets the minimum threshold, registers `buyer` as a committed participant in `self.buyers`.

The ticket system does not mitigate this. The ticket check at `rs/sns/swap/src/swap.rs` lines 1250–1272 is guarded by `if let Some(ticket_sns_sale_canister) = ...` — it only validates when a ticket already exists for the buyer. If no ticket exists, the code explicitly proceeds: *"If there exists no ticket for the buyer, the payment flow will simply ignore the ticket."* Furthermore, the ticketing system is documented as optional in the integration tests: *"The ticketing system is optional in the current implementation, so the participants are free to choose if they use it or not."*

The `confirmation_text` field is not a meaningful barrier: it is stored in the swap's `Init` payload and is publicly readable via the `get_init` query endpoint at `rs/sns/swap/canister/canister.rs` lines 211–216. An attacker reads it and includes it verbatim.

Once committed, the victim cannot reclaim their ICP via `error_refund_icp` while the swap is `OPEN`; that endpoint is gated on the swap being `ABORTED` or `COMMITTED` (`rs/sns/swap/src/swap.rs` lines 1931–1936).

## Impact Explanation

This is an unauthorized manipulation of user funds in the SNS framework, a core in-scope governance/financial component. A victim who has transferred ICP to the swap subaccount (a normal intermediate state every participant passes through) can be forcibly enrolled as a buyer. Their ICP is locked until finalization; if the swap commits, they receive SNS tokens without consent; if it aborts, they recover their ICP minus transfer fees only after the swap closes. This matches the **High ($2,000–$10,000)** impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds"* and *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation

The attack requires only a standard ingress call from any unprivileged principal — no special role, key, or majority is needed. The victim's precondition (ICP in the swap subaccount) is a normal intermediate state every participant passes through, creating a non-trivial exploitation window. The `confirmation_text` (when present) is publicly readable from `get_init`, providing no barrier. The attack is cheap (one update call) and can be repeated across all open SNS swaps simultaneously.

## Recommendation

In `refresh_buyer_tokens`, reject any request where `arg.buyer` is non-empty and does not match `caller_principal_id()`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        if requested != caller {
            panic!("caller {} is not authorized to refresh tokens on behalf of {}", caller, requested);
        }
        requested
    };
    // ...
}
```

Alternatively, remove the `buyer` field entirely and always use `caller_principal_id()`, since the only legitimate use case is a principal notifying the swap of their own transfer.

## Proof of Concept

**Setup:**
- SNS swap is `OPEN`, no `confirmation_text` (or attacker reads it from `get_init`).
- Victim `V` transfers 10 ICP to the swap canister's subaccount `principal_to_subaccount(V)` on the ICP ledger, intending to evaluate participation.

**Attack (single ingress call from attacker `A`):**

```
dfx canister call <swap_canister_id> refresh_buyer_tokens '(
  record {
    buyer = "<victim_principal_text_id>";
    confirmation_text = null
  }
)'
```

**Verification via integration test (local StateMachine):**

Using the existing `refresh_buyer_tokens` helper in `rs/sns/test_utils/src/state_test_helpers.rs` (lines 644–659), construct a test where:
1. Victim `V` transfers ICP to the swap subaccount without creating a ticket.
2. Attacker `A` calls `refresh_buyer_tokens` with `buyer = V.to_string()` as the sender.
3. Assert `get_buyer_state(V)` shows `amount_e8s > 0`.
4. Assert `error_refund_icp` returns a precondition error while the swap is `OPEN`.

**Result:** `refresh_buyer_token_e8s` reads the balance of `principal_to_subaccount(V)`, finds 10 ICP, and registers `V` as a buyer. `V` is now a committed participant with their ICP locked until finalization.
