### Title
Warm Storage Access Status Incorrectly Rolled Back on Sub-call Revert, Causing EVM Gas Deviation — (`basic_system/src/system_implementation/caches/generic_pubdata_aware_plain_storage.rs`)

---

### Summary

ZKsync OS rolls back the warm/cold access status (`last_touched_in_tx`) of storage slots when a sub-call frame reverts. EIP-2929 specifies the opposite: a slot accessed in a reverted sub-call **remains warm** in the parent frame. This causes the outer frame to pay the cold access surcharge (2100 gas) for a slot it should see as warm (100 gas), overcharging by 2000 gas per reverted cold access.

---

### Finding Description

**Warm status storage path**

`last_touched_in_tx` lives inside `StorageElementMetadata`, which is the metadata field of `CacheRecord<V, StorageElementMetadata>`: [1](#0-0) 

This `CacheRecord` is the **value type `V`** stored in the `HistoryMap`. It is part of the versioned history chain, not the non-rollback-able `element_properties` field.

**`ElementWithHistory` distinguishes the two kinds of data explicitly:** [2](#0-1) 

`element_properties` (type `CacheElementProperties`) persists across rollbacks. The value chain (`head`/`initial`/`committed`) does not — it is the rollback target.

**Warm-up writes into the rollback-able chain**

When a cold slot is first accessed, `materialize_element` calls `x.update(...)` to set `last_touched_in_tx`: [3](#0-2) 

`HistoryMapItemRefMut::update` creates a new history record tagged with the current snapshot ID and registers the key in `pending_updated_elements`: [4](#0-3) 

**`finish_frame_impl` rolls back the entire cache, including warm status** [5](#0-4) 

`HistoryMap::rollback` walks `pending_updated_elements` and reverts every record whose snapshot ID is newer than the rollback point: [6](#0-5) 

This unconditionally reverts `last_touched_in_tx` back to `None`, making the slot cold again in the parent frame.

**The code's own comment confirms this is intentional but wrong** [7](#0-6) 

The comment says "in case of revert it should become cold again." This is the implemented behavior, but it contradicts EIP-2929.

---

### Impact Explanation

EIP-2929 (Berlin hard fork) specifies that the access list is a **transaction-scoped** set that is never rolled back on revert. A slot touched in a reverted sub-call is warm for the remainder of the transaction. ZKsync OS instead scopes warm status to the call frame, so after a reverted inner call:

- The outer frame re-pays the cold surcharge (2100 gas instead of 100 gas) for the same slot.
- Overcharge: **2000 gas per reverted cold access**.
- Contracts that rely on EVM gas semantics (e.g., gas-stipend-sensitive patterns, try/catch patterns that access the same storage after a failed sub-call) will behave differently or run out of gas on ZKsync OS vs. Ethereum mainnet.

---

### Likelihood Explanation

Any unprivileged EVM contract can trigger this by:
1. Calling an inner contract (or itself) that SLOADs a cold slot.
2. Having the inner call revert.
3. SLOADing the same slot in the outer frame.

No special privileges, governance access, or external oracle manipulation is required. The call sequence is a standard Solidity `try/catch` or low-level `call` with a revert.

---

### Recommendation

Warm/cold status must be tracked outside the rollback-able history chain. Move `last_touched_in_tx` from `StorageElementMetadata` (which lives in the versioned `CacheRecord` value) into `CacheElementProperties` (which is stored in `element_properties` and explicitly documented as persisting across rollbacks). Since `element_properties` is never rolled back by `HistoryMap::rollback`, warm status set in an inner frame will correctly survive a revert and remain visible to the parent frame, matching EIP-2929 semantics.

The same fix applies to `BasicAccountPropertiesMetadata.last_touched_in_tx` used in the account cache, which has the same structural issue.

### Proof of Concept

```
// Differential test (pseudo-Solidity)
contract Outer {
    Inner inner;
    uint slot0; // cold at start of tx

    function test() external returns (uint gasInner, uint gasOuter) {
        // Inner call: SLOAD slot0 (cold, pays 2100), then REVERT
        try inner.sloadAndRevert(address(this), 0) {} catch {}

        // Outer frame: SLOAD slot0 again
        // EVM spec: warm → costs 100
        // ZKsync OS: rolled back to cold → costs 2100
        uint g0 = gasleft();
        assembly { sload(0) }
        gasOuter = g0 - gasleft();
        // assert gasOuter == 100 on EVM; ZKsync OS returns ~2100
    }
}
```

Run the same bytecode against REVM and ZKsync OS. The outer SLOAD gas cost will be 100 on REVM and ~2100 on ZKsync OS, confirming the 2000-gas overcharge per reverted cold access.
