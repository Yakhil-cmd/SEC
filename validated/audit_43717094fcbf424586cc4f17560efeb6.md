### Title
TOCTOU Race in `register_dapp_canister` Allows Exceeding `DAPP_AND_EXTENSION_CANISTER_REGISTRATION_LIMIT` — (`rs/sns/root/src/lib.rs`)

### Summary

The `register_dapp_canister` function performs a limit check by reading state, then suspends at multiple `await` points before writing state. Because the IC canister execution model allows interleaving of concurrent update calls at `await` boundaries, two concurrent `RegisterDappCanisters` proposal executions can both pass the limit check before either writes, pushing `dapp_canister_ids` beyond the 100-entry cap.

### Finding Description

In `rs/sns/root/src/lib.rs`, `register_dapp_canister` (the per-canister helper called by `try_register_dapp_canisters`) follows this pattern:

**Step 1 — Read state and check limit (line 669–679):** [1](#0-0) 

**Step 2 — Suspend at `await` (inter-canister call to management canister, line 703–706):** [2](#0-1) 

There are up to two additional `await` points if `update_settings` is needed (lines 718–728, 732–735). [3](#0-2) 

**Step 3 — Write state (line 741–744):** [4](#0-3) 

The constant being guarded is: [5](#0-4) 

On the IC, a canister processes one message at a time, but when a message suspends at an `await` (inter-canister call), the canister scheduler can begin processing the next queued message. Two update calls to `register_dapp_canisters` on the root canister therefore interleave:

```
Call A: reads dapps.len()=99 → passes check (99 < 100)
Call A: awaits canister_status(X)          ← suspends here
Call B: reads dapps.len()=99 → passes check (99 < 100)
Call B: awaits canister_status(Y)          ← suspends here
Call A: resumes → pushes X → dapps.len()=100
Call B: resumes → pushes Y → dapps.len()=101  ← EXCEEDS LIMIT
```

On the governance side, `start_proposal_execution` uses `spawn_in_canister_env`, which spawns proposal futures concurrently: [6](#0-5) 

Unlike upgrade proposals (which have an `upgrade_proposals_in_progress` guard), there is **no concurrency guard** for `RegisterDappCanisters` proposals. Two adopted proposals can both trigger concurrent calls to root's `register_dapp_canisters`. [7](#0-6) 

### Impact Explanation

`dapp_canister_ids.len() + extensions.len()` exceeds `DAPP_AND_EXTENSION_CANISTER_REGISTRATION_LIMIT` (100), violating the registration cap invariant. This causes unbounded state growth in the SNS root canister, potential memory pressure on the replica hosting the SNS root, and can affect `get_sns_canisters_summary` and `set_dapp_controllers` which iterate over all registered dapps. [8](#0-7) 

### Likelihood Explanation

Requires two `RegisterDappCanisters` proposals to be adopted before either executes — a realistic scenario when an SNS has ~99 registered dapps and two community members independently propose registering different dapps in the same voting window. No malicious governance majority is required; normal community voting suffices. The interleaving is deterministic given the IC's cooperative async scheduling.

### Recommendation

Re-check the limit **inside the final `with_borrow_mut` write**, atomically with the push, so the check and write are in the same synchronous (non-`await`) critical section:

```rust
self_ref.with_borrow_mut(|state| {
    let count = state.dapp_canister_ids.len() + state.extensions.len();
    if count >= DAPP_AND_EXTENSION_CANISTER_REGISTRATION_LIMIT {
        return Err("limit reached".to_string());
    }
    state.dapp_canister_ids.push(PrincipalId::from(canister_to_register));
    Ok(())
})?;
```

Since `with_borrow_mut` is synchronous and non-`await`, no interleaving can occur between the check and the write.

### Proof of Concept

A state-machine test with two `RegisterDappCanisters` proposals both adopted in the same round, with the SNS root pre-loaded with 99 dapps, would demonstrate that after both proposals execute, `dapp_canister_ids.len()` equals 101, violating the invariant. The existing test `register_dapp_canisters_fails_at_limit_number` only tests sequential execution and does not cover concurrent interleaving. [9](#0-8)
