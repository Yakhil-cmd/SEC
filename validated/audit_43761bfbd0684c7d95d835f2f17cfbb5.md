Audit Report

## Title
Unbounded Global HashMap and Prometheus Label Growth via Attacker-Controlled `network_identifier.network` — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary

The ICRC1 Rosetta server's `RosettaMetricsMiddleware` extracts `network_identifier.network` from every inbound HTTP request body as a raw, unvalidated string and unconditionally inserts it into the process-global `CANISTER_DISPLAY_NAMES: Mutex<HashMap<String, String>>`. There is no bound, no eviction, and no validation. An unauthenticated attacker sending requests with distinct strings exhausts heap memory and causes an OOM crash of the Rosetta process, halting ledger synchronization and all dependent operations.

## Finding Description

**Step 1 — Middleware is applied to all routes.**

In `rs/rosetta-api/icrc1/src/main.rs` at line 383, `metrics_layer` is applied unconditionally to every route via `.layer(metrics_layer)`, covering `/block`, `/account/balance`, `/network/status`, and all other endpoints.

**Step 2 — `extract_canister_id` accepts any string without validation.**

In `rs/rosetta-api/common/rosetta_core/src/metrics.rs` at lines 397–404, the helper reads `network_identifier.network` as a raw `String` from the JSON body with no validation that it is a valid canister ID, principal, or hex string:

```rust
let canister_id = match serde_json::from_slice::<Value>(&bytes) {
    Ok(json) => json
        .get("network_identifier")
        .and_then(|ni| ni.get("network"))
        .and_then(|n| n.as_str())
        .map(|s| s.to_string()),  // ← any string accepted
    Err(_) => None,
};
```

**Step 3 — `RosettaMetrics::new` unconditionally inserts into the global map.**

At lines 117–126, every call inserts into `CANISTER_DISPLAY_NAMES` with no capacity check:

```rust
pub fn new(token_display_name: String, canister_id: String) -> Self {
    let mut map = CANISTER_DISPLAY_NAMES.lock().unwrap();
    map.insert(canister_id.clone(), token_display_name.clone()); // ← no bound
    ...
}
```

**Step 4 — The middleware calls `RosettaMetrics::new` before the endpoint handler validates the network ID.**

At lines 342–348, the insertion happens before the request is forwarded to the handler:

```rust
let metrics = match canister_id {
    Some(id) => {
        let display_name = RosettaMetrics::get_display_name_from_canister_id(&id);
        RosettaMetrics::new(display_name, id)  // ← insert happens here
    }
    None => default_metrics,
};
// ... then inner.call(req).await — handler may reject, but map already mutated
```

The endpoint handler (`get_state_from_network_id`) rejects unknown network IDs, but the map insertion has already occurred.

**Step 5 — Secondary amplifier: Prometheus label cardinality explosion.**

`get_display_name_from_canister_id` at line 257 returns the raw attacker string when no mapping exists:

```rust
.unwrap_or_else(|| canister_id.to_string())
```

This string is then used as the `token_display_name` Prometheus label in `inc_api_status_count` (line 137) and `observe_request_duration` (line 167), causing unbounded Prometheus time-series growth in addition to the HashMap growth.

**No existing guard is sufficient.** The body size limit in `extract_canister_id` is `usize::MAX` (line 389), providing no protection. The `CANISTER_DISPLAY_NAMES` map is a `lazy_static` with no capacity limit. The Prometheus registry has no label cardinality cap.

## Impact Explanation

Each unique `network_identifier.network` value adds one entry to `CANISTER_DISPLAY_NAMES` and one or more Prometheus metric time-series. Both are process-global and never evicted. With a large number of unique strings, the combined heap growth causes OOM and process crash. This halts ledger synchronization and all asset bridge operations that depend on the Rosetta process. This matches the **High ($2,000–$10,000): Application/platform-level DoS, crash** impact category — specifically a crash of the Rosetta server (a listed in-scope financial integration component) triggered without raw volumetric DDoS.

## Likelihood Explanation

The attack requires only unauthenticated HTTP POST access to any Rosetta endpoint (e.g., `/block`). No credentials, no special knowledge, and no rate-limit bypass are needed. The ICRC1 Rosetta server is designed to be publicly reachable. The attack is trivially scriptable and repeatable.

## Recommendation

1. **Validate before inserting**: In `extract_canister_id`, attempt to parse the extracted string as a valid `CanisterId` (using `PrincipalId::try_from` + `CanisterId::try_from`, as already done in `TryFrom<&NetworkIdentifier> for CanisterId`). Only pass the value to `RosettaMetrics::new` if it parses successfully.
2. **Bound the map**: Cap `CANISTER_DISPLAY_NAMES` at the number of configured tokens (known at startup), or remove the per-request insert entirely and only populate the map at startup from the configured token list via `register_canister_display_name`.
3. **Do not use attacker-controlled strings as Prometheus label values**: Use only the pre-registered display names from the startup configuration; fall back to a static placeholder (e.g., `"unknown"`) rather than the raw input string.

## Proof of Concept

```python
import requests, threading

url = "http://<rosetta-host>/block"

def flood(n, offset):
    for i in range(n):
        requests.post(url, json={
            "network_identifier": {"network": f"deadbeef{(offset + i):08x}"},
            "block_identifier": {"index": 0}
        })

threads = [threading.Thread(target=flood, args=(100_000, i * 100_000)) for i in range(10)]
for t in threads: t.start()
for t in threads: t.join()
# After ~1M unique values, RSS grows proportionally; process OOMs
```

Each request is rejected by the handler (unknown network ID), but the map insertion and Prometheus label creation have already occurred. The process RSS grows monotonically until OOM.
