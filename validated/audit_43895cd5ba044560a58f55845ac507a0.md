Audit Report

## Title
Unbounded Loop via Integer Overflow in `construction_payloads` Causes OOM DoS of ICP Rosetta Node — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The `construction_payloads` handler in the ICP Rosetta API accepts attacker-controlled `ingress_start` and `ingress_end` values with no bounds validation. When near-`u64::MAX` values are supplied, the `now += interval` step in the ingress-expiry loop overflows via a truncating `as u64` cast in `Time::from_duration`, wrapping `now` to ~120 seconds since epoch and causing the loop to execute ~153 billion additional iterations. Each iteration pushes a `u64` onto a `Vec`, exhausting process memory and crashing the Rosetta node. A single unauthenticated HTTP POST is sufficient to trigger this.

## Finding Description

**Interval computation** (`construction_payloads.rs` L59-60):
```
interval = MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s = 300s - 60s - 120s = 120_000_000_000 ns
``` [1](#0-0) [2](#0-1) 

**The loop** (`construction_payloads.rs` L99-107) iterates from `ingress_start` to `ingress_end` in steps of `interval`, with no pre-loop validation of the range: [3](#0-2) 

**Root cause — truncating cast in `Time::add_assign`** (`time.rs` L55-58): `now += interval` calls `Time::from_duration`, which computes `Duration::as_nanos()` (returns `u128`) and silently truncates with `as u64`: [4](#0-3) [5](#0-4) 

**Overflow trace with `ingress_start = u64::MAX - 1`, `ingress_end = u64::MAX`:**
- Iteration 1: condition `u64::MAX - 1 < u64::MAX` → true, push entry, then `now += 120_000_000_000`
- `(u64::MAX - 1) + 120_000_000_000` as `u128` = `18446744193709551614`
- `as u64` truncates to `≈ 119_999_999_998` (~120 seconds since epoch)
- Iteration 2+: `119_999_999_998 < u64::MAX` → true → loop runs `≈ (u64::MAX) / 120_000_000_000 ≈ 153 billion` more iterations
- Each iteration pushes a `u64` (8 bytes) → ~1.2 TB of allocation → OOM crash

**No input validation** on `ingress_start`/`ingress_end` in the ICP handler — they are raw `Option<u64>` fields deserialized directly: [6](#0-5) 

**Contrast with ICRC1 handler**, which explicitly rejects invalid ranges before its loop: [7](#0-6) 

The ICP handler has no equivalent guard. A safe `checked_add` API already exists on `Time` but is not used in `AddAssign`: [8](#0-7) 

## Impact Explanation
This is a **High** severity finding matching: *"Significant Rosetta/boundary/API security impact with concrete user or protocol harm."* A single unauthenticated request crashes the ICP Rosetta node process via OOM, making it unavailable to exchange operators and integrators who depend on it for ICP transfers and account queries. The node cannot recover without a manual restart.

## Likelihood Explanation
The Rosetta API is a public HTTP endpoint with no authentication. The exploit requires only a single crafted JSON POST with two specific field values (`ingress_start = u64::MAX - 1`, `ingress_end = u64::MAX`). No privileged access, key material, or network-level capability is needed. The attack is trivially repeatable after each restart.

## Recommendation

1. **Add range validation before the loop** in `construction_payloads.rs`, mirroring the ICRC1 handler:
   ```rust
   if ingress_end <= ingress_start {
       return Err(ApiError::invalid_request("ingress_end must be after ingress_start"));
   }
   let max_window = ic_limits::MAX_INGRESS_TTL.as_nanos() as u64;
   if ingress_end.as_nanos_since_unix_epoch()
       .saturating_sub(ingress_start.as_nanos_since_unix_epoch()) > max_window {
       return Err(ApiError::invalid_request("ingress window exceeds maximum allowed TTL"));
   }
   ```
2. **Replace `Time::from_duration` (truncating cast) with `Time::checked_add`** in `AddAssign<Duration> for Time` to make overflow explicit and panic/return an error rather than silently wrap.
3. Optionally, add a server-side cap on `ingress_end` relative to the current wall-clock time (as the ICRC1 handler does with its `ingress_end < now + ingress_interval` check).

## Proof of Concept

```rust
// Single unauthenticated HTTP POST to /construction/payloads:
{
  "network_identifier": { "blockchain": "Internet Computer", "network": "<mainnet_id>" },
  "operations": [ /* any valid ICP transfer operation */ ],
  "public_keys": [ /* any valid Ed25519 public key */ ],
  "metadata": {
    "ingress_start": 18446744073709551614,
    "ingress_end":   18446744073709551615
  }
}
// Expected (unfixed): loop wraps after 1 iteration, runs ~153 billion iterations → OOM kill.
// Expected (fixed):   returns ApiError immediately after range validation.
```

A deterministic unit test can be written in `construction_payloads.rs` asserting that calling `construction_payloads` with these metadata values returns an `Err(ApiError)` rather than hanging or panicking.
