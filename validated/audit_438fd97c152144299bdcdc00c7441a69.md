Audit Report

## Title
Unchecked ICP Burn Result in CMC `burn_and_log` Allows Cycles Minting Without ICP Destruction - (File: rs/nns/cmc/src/main.rs)

## Summary
The Cycles Minting Canister mints cycles (or tops up a canister) before burning the backing ICP. The burn is performed by `burn_and_log`, which explicitly discards all errors and returns `()`. If the ICP ledger rejects the burn call for any transient reason, cycles are permanently issued while the corresponding ICP remains unburned in CMC's subaccount, breaking the ICP-to-cycles conservation invariant with no recovery path.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` (L2017–2049) is declared `async fn burn_and_log(...) -> ()`. The error arm of the `match` on `call_protobuf` result only calls `print(...)` and returns normally:

```rust
Err((code, err)) => {
    let code = code as i32;
    print(format!("{msg} failed with code {code}: {err:?}"))
}
``` [1](#0-0) 

In `process_mint_cycles` (L1958–1983), `do_mint_cycles` is awaited first; only on success is `burn_and_log` called. The function then returns `Ok(NotifyMintCyclesSuccess{...})` regardless of whether the burn succeeded: [2](#0-1) 

The same ordering applies in `process_top_up` (L1985–2012): `deposit_cycles` succeeds, then `burn_and_log` is called fire-and-forget: [3](#0-2) 

Because `process_mint_cycles` returns `Ok(...)` unconditionally after a successful deposit, `is_transient_error` evaluates to `false`, and the block index is permanently recorded as `NotifiedMint(Ok(...))` in `blocks_notified`: [4](#0-3) 

The design comment at L2014–2016 explicitly acknowledges this tradeoff: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* This prevents double-minting but creates a silent, unrecoverable conservation failure path. [5](#0-4) 

## Impact Explanation
When `burn_and_log` fails: (1) the user has already received cycles or a canister top-up; (2) the ICP in CMC's subaccount is **not burned** — it remains locked there permanently; (3) the block index is marked processed, so no retry is possible; (4) the total ICP supply is permanently higher than it should be relative to cycles issued, breaking the economic invariant that underpins the ICP/cycles exchange. This constitutes a **significant NNS/CMC ledger conservation impact with concrete protocol harm** — matching the High bounty tier for significant ledger/NNS security impact with concrete user or protocol harm. At scale (e.g., during a ledger upgrade window where multiple notifications complete their deposit step), the cumulative unburned ICP could be material.

## Likelihood Explanation
An unprivileged user cannot directly force the ICP ledger to be unavailable, but the condition arises naturally and predictably during ledger canister upgrades (which require a stop/upgrade/start cycle during which all in-flight calls to the ledger are rejected). NNS ledger upgrades are publicly announced via governance proposals. A user who submits `notify_mint_cycles` such that the `do_mint_cycles` deposit completes just before the ledger stops will trigger this path. The IC can also reject inter-canister calls under message-queue pressure. Likelihood is low per individual transaction but non-zero and repeatable across upgrade windows.

## Recommendation
1. **Retry queue**: On burn failure, persist `(subaccount, amount, block_index)` in CMC stable state and retry in a heartbeat/timer, analogous to ckBTC/ckETH failed-mint recovery.
2. **Reverse ordering**: Burn ICP first via `send_pb`; only if the burn succeeds, mint cycles. A burn failure returns an error to the caller (the block index is not yet marked processed, so the user can retry after the ledger recovers).
3. **At minimum, expose a metric**: Increment a stable counter on burn failure so operators can detect and manually remediate stuck ICP.

## Proof of Concept
1. User transfers `X` ICP to `cmc_subaccount(caller)` with `MEMO_MINT_CYCLES` on the ICP ledger.
2. User calls `notify_mint_cycles` on CMC. CMC enters `process_mint_cycles`; `do_mint_cycles` → cycles ledger `deposit` succeeds; cycles are credited to the user.
3. CMC calls `burn_and_log(sub, amount)`. At this moment the ICP ledger is mid-upgrade (stopped); `call_protobuf(..., "send_pb", ...)` returns `Err(RejectionCode::CanisterError, "canister is stopped")`.
4. `burn_and_log` logs the error and returns `()`. `process_mint_cycles` returns `Ok(NotifyMintCyclesSuccess{...})`.
5. CMC records `NotificationStatus::NotifiedMint(Ok(...))` for the block index; `is_transient_error` is `false`; the entry is permanent.
6. **Result**: User holds cycles. ICP in CMC's subaccount is unburned. Block index is permanently consumed. ICP supply is inflated by `X` tokens relative to cycles minted, with no on-chain recovery mechanism.

A deterministic integration test can reproduce this by: deploying a mock ICP ledger that returns an error on `send_pb`, running `notify_mint_cycles` against it, and asserting that (a) cycles were deposited and (b) the mock ledger recorded zero burn calls.
