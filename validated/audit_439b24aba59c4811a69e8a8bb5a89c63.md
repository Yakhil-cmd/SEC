Audit Report

## Title
ckERC20 Minter Mints Based on Event-Logged Amount Rather Than Actual Received Balance for Fee-on-Transfer ERC-20 Tokens - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol)

## Summary
The `depositErc20` function in the ckERC20 helper contract emits the caller-supplied `amount` argument in the `ReceivedEthOrErc20` event rather than the actual balance delta received by the minter address after `safeTransferFrom`. The IC minter canister scrapes these logs and mints ckERC20 tokens 1:1 against the event's `amount` field without any balance reconciliation. USDT — which has a configurable fee-on-transfer mechanism currently set to 0% — is already a supported ckERC20 token on Ethereum mainnet; if Tether activates its fee, every deposit will silently over-mint ckUSDT, breaking the 1:1 collateral invariant and eventually rendering the minter insolvent.

## Finding Description
**Root cause — helper contract (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, lines 511–532):**

`depositErc20` calls `erc20Token.safeTransferFrom(msg.sender, minterAddress, amount)` and then immediately emits `ReceivedEthOrErc20(..., amount, ...)` using the caller-supplied `amount` parameter verbatim. There is no balance snapshot before/after the transfer to compute the actual delta received by `minterAddress`. For a fee-on-transfer token the minter receives `amount − fee`, but the event records `amount`.

**Consumption in the IC minter (`rs/ethereum/cketh/minter/src/deposit.rs`, lines 73–81):**

The minter calls `client.transfer(TransferArg { ..., amount: event.value(), ... })`, where `event.value()` is parsed directly from the log's data word (`rs/ethereum/cketh/minter/src/eth_logs/parser.rs`, lines 149–160). There is no step that queries the minter's ERC-20 balance before and after the on-chain transfer; the minter unconditionally trusts the log's `amount` field.

**No existing guard:** The codebase contains no mention of fee-on-transfer exclusions, no balance-reconciliation check, and no token-property validation at deposit time. The minter's token whitelist (enforced by log-scraping filter) only checks that the ERC-20 contract address is a supported token — it does not screen for fee-on-transfer behavior.

**USDT is already supported:** `rs/ethereum/cketh/docs/ckerc20.adoc` lines 37–38 lists USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) as a supported ckERC20 token on Ethereum mainnet. No additional governance action is required for the precondition to be met; only Tether activating its existing fee mechanism is needed.

## Impact Explanation
If USDT (or any other supported fee-on-transfer ERC-20) activates its transfer fee, every call to `depositErc20` will cause the IC minter to mint more ckERC20 than the ERC-20 collateral it actually holds. The invariant `ckERC20 total supply ≤ ERC-20 balance of minter address` is broken on every deposit. Accumulated unbacked supply means later redeemers cannot withdraw at par — a direct, concrete loss of user funds and protocol insolvency for the affected ck-token. This matches the allowed impact: **Significant Chain Fusion / ck-token security impact with concrete user or protocol harm** (High, $2,000–$10,000), and potentially scales to Critical if the fee is activated while significant ckUSDT supply exists.

## Likelihood Explanation
USDT is already a supported token; no NNS governance vote is needed. The sole precondition is Tether enabling its on-chain fee parameter, which is a legitimate, documented feature of the USDT contract that can be activated unilaterally by Tether at any time. Once the fee is active, any unprivileged user calling `depositErc20` triggers the over-mint automatically via the minter's log-scraping timer. The exploit is repeatable on every subsequent deposit. Likelihood is **medium**: it depends on a third-party action (Tether fee activation) outside the attacker's control, but the structural flaw is present in production today and requires no privileged IC access to exploit once the precondition is met.

## Recommendation
In `depositErc20`, record the minter's ERC-20 balance before and after `safeTransferFrom` and emit the delta as the event amount:

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        actualReceived,
        principal,
        subaccount
    );
}
```

This ensures the event amount always reflects the true collateral received, regardless of the token's fee behavior.

## Proof of Concept
1. Tether activates USDT's transfer fee (e.g., 1 basis point = 0.01%).
2. Attacker calls `depositErc20(USDT, 1_000_000, principal, subaccount)` on the deployed helper contract (`0x18901044688D3756C35Ed2b36D93e6a5B8e00E68`).
3. Helper calls `USDT.transferFrom(attacker, minter, 1_000_000)`; due to the 0.01% fee, minter receives `999_900` USDT.
4. Helper emits `ReceivedEthOrErc20(USDT, attacker, 1_000_000, ...)`.
5. IC minter's log-scraping timer (`mint()` in `deposit.rs`) reads `value = 1_000_000` and mints `1_000_000` ckUSDT to the attacker.
6. Attacker holds `1_000_000` ckUSDT backed by only `999_900` USDT.
7. Repeated deposits accumulate unbacked ckUSDT; eventually the minter cannot honour all redemptions.

A deterministic integration/state-machine test can reproduce this by deploying a mock ERC-20 with a configurable fee-on-transfer, registering it as a supported token, calling `depositErc20`, and asserting that the minted ckERC20 balance exceeds the mock token balance held by the minter address.
