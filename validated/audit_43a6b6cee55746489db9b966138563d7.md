Audit Report

## Title
`validate_self_validating_payload_impl` Accepts Fabricated Bitcoin Adapter Responses Without Callback ID Validation — (`rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary
`BitcoinPayloadBuilder::validate_self_validating_payload_impl` accepts any non-empty `SelfValidatingPayload` after checking only its byte size, with no verification that each `BitcoinAdapterResponse::callback_id` corresponds to a real pending request in the certified state. A single malicious block-proposing replica (below the consensus fault threshold) can craft a `SelfValidatingPayload` containing fabricated `BitcoinAdapterResponse` entries bearing real `callback_id` values; all other replicas accept the block because their validation also only checks size. The execution environment then processes the fabricated responses as legitimate Bitcoin network data, enabling false confirmation of Bitcoin transactions or injection of fabricated Bitcoin block data into the IC's UTXO state.

## Finding Description

**Consensus-layer validation (root cause):**

`validate_self_validating_payload_impl` at `rs/bitcoin/consensus/src/payload_builder.rs` lines 234–251 performs only an emptiness check and a byte-size return. It does not load the certified state, does not iterate over `subnet_call_context_manager.bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`, and does not verify that any `callback_id` in the payload corresponds to a real pending request. The `validate_payload` implementation of `BatchPayloadBuilder` (lines 357–397) computes `delivered_ids` from past payloads but never uses it to cross-check the proposed payload's callback IDs.

**Execution-layer partial mitigation and remaining gap:**

`rs/replicated_state/src/bitcoin.rs` `push_response` (lines 23–103) is called during block execution. For `BitcoinAdapterResponseWrapper::GetSuccessorsResponse` it does look up the `callback_id` in `bitcoin_get_successors_contexts` and returns `StateError::BitcoinNonMatchingResponse` if absent (lines 31–39). However, for `SendTransactionResponse` (lines 64–75), `GetSuccessorsReject` (lines 77–88), and `SendTransactionReject` (lines 90–102), **no such lookup is performed** — the function unconditionally pushes a `ConsensusResponse` to the consensus queue for any `callback_id`. This means:

- A fabricated `SendTransactionResponse` with a real `callback_id` from a pending `SendTransactionRequest` passes both the consensus-layer size check and the execution-layer handler with no error, delivering a false "transaction broadcast" confirmation to the Bitcoin management canister.
- A fabricated `GetSuccessorsResponse` with a real `callback_id` passes the execution-layer existence check (the context is found) and delivers fabricated Bitcoin block data to the Bitcoin management canister; the management canister has no independent proof-of-work validation at this layer.

**Contrast with canister HTTP:**

`validate_canister_http_payload_impl` in `rs/https_outcalls/consensus/src/payload_builder.rs` lines 468–473 explicitly looks up each `callback_id` in `canister_http_request_contexts` and rejects the payload with `UnknownCallbackId` if not found. No equivalent check exists for Bitcoin payloads.

## Impact Explanation

This matches the allowed impact: **High–Critical — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.**

- **`SendTransactionResponse` path (no execution-layer guard):** An attacker injects a fake success response for a real pending `SendTransactionRequest`. The Bitcoin management canister marks the withdrawal as confirmed. The ckBTC minter burns ckBTC and considers the withdrawal complete, but the Bitcoin transaction was never broadcast. The user loses ckBTC with no corresponding Bitcoin received — a direct theft of chain-key assets.
- **`GetSuccessorsResponse` path (execution-layer guard bypassed with real callback_id):** An attacker uses a real `callback_id` from a pending `GetSuccessorsRequest` and supplies fabricated Bitcoin blocks. The execution layer finds the context and delivers the fabricated response to the Bitcoin management canister, which updates its UTXO set. The ckBTC minter can be induced to mint ckBTC for non-existent deposits.

## Likelihood Explanation

Every replica participates in block-proposal rotation. A single compromised or malicious node operator — well below the consensus fault threshold — will eventually be selected as block proposer. No threshold cryptography, admin key, or majority collusion is required. The attacker only needs to: (1) read the current certified state to obtain a live `callback_id`, and (2) craft a `SelfValidatingPayload` within `MAX_BITCOIN_PAYLOAD_IN_BYTES`. This is trivially achievable by any node operator who controls the replica binary. The attack is repeatable every time the malicious node wins a proposal slot.

## Recommendation

`validate_self_validating_payload_impl` must load the certified state at `validation_context.certified_height` and, for each `BitcoinAdapterResponse` in the payload:
1. Verify that `response.callback_id` exists in `subnet_call_context_manager.bitcoin_get_successors_contexts` (for `GetSuccessors*` variants) or `bitcoin_send_transaction_internal_contexts` (for `SendTransaction*` variants).
2. Verify that the response variant matches the request type stored for that `callback_id`.
3. Reject any payload containing a duplicate `callback_id` or a `callback_id` already present in `past_payloads`.

Additionally, `push_response` in `rs/replicated_state/src/bitcoin.rs` should add the same existence check for `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` that already exists for `GetSuccessorsResponse` (lines 31–39), returning `StateError::BitcoinNonMatchingResponse` for unknown callback IDs.

## Proof of Concept

1. Attacker controls one replica on the Bitcoin-enabled subnet and reads the certified state to obtain a live `callback_id` for a pending `SendTransactionRequest` (e.g., a ckBTC withdrawal in progress).
2. When the attacker's node is selected as block proposer, it constructs a `SelfValidatingPayload` containing a `BitcoinAdapterResponse { response: SendTransactionResponse(()), callback_id: <real_id> }`.
3. The block is proposed. Every other replica calls `validate_self_validating_payload_impl`; the payload passes because its byte size is within `MAX_BITCOIN_PAYLOAD_IN_BYTES` (lines 384–394 of `payload_builder.rs`).
4. The block is finalized. `push_response` is called with the fabricated response; the `SendTransactionResponse` branch (lines 64–75 of `bitcoin.rs`) pushes a success `ConsensusResponse` to the consensus queue without any callback_id lookup.
5. The Bitcoin management canister receives the success response and marks the withdrawal transaction as broadcast.
6. The ckBTC minter finalizes the withdrawal, burning the user's ckBTC. The Bitcoin transaction was never submitted to the Bitcoin network.

A deterministic integration test can reproduce this by: constructing a `BitcoinPayloadBuilder` with a mock state containing one pending `SendTransactionRequest`, calling `validate_payload` with a hand-crafted payload containing a `SendTransactionResponse` for that `callback_id` but with no actual adapter interaction, asserting `Ok(())` is returned, then calling `push_response_bitcoin` on the resulting state and asserting the consensus queue contains the fabricated success response.
