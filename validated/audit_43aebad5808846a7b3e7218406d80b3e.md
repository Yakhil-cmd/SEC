### Title
Missing Slippage Protection in SNS Treasury Manager `DepositRequest` Interface Enables Front-Running of DAO Treasury Deposits - (File: rs/sns/treasury_manager/treasury_manager.did)

---

### Summary

The `DepositRequest` type in the SNS Treasury Manager interface contains no slippage tolerance field. When an SNS governance proposal to deposit treasury funds into a DEX liquidity pool is executed, the deposit proceeds at whatever price ratio exists at execution time, with no minimum LP token output or maximum price impact constraint. This enables front-running and sandwich attacks on SNS treasury deposits. The IC codebase itself explicitly acknowledges this as a "Known Security Risk."

---

### Finding Description

The `DepositRequest` record defined in `rs/sns/treasury_manager/treasury_manager.did` contains only `allowances` (the token amounts to deposit) and no slippage protection field:

```candid
type DepositRequest = record {
  allowances : vec Allowance;
};
``` [1](#0-0) 

The `Allowance` type itself only carries `asset`, `amount_decimals`, and `owner_account` — no `min_lp_tokens_out`, no `max_price_impact`, no `deadline`. [2](#0-1) 

The production execution path in `rs/sns/governance/src/extensions.rs` — `execute_treasury_manager_deposit` — constructs the deposit payload exclusively from the `allowances` and calls `deposit` on the extension canister with no slippage constraint:

```rust
let arg_blob =
    construct_treasury_manager_deposit_payload(context, original).map_err(...)?;
// ...
governance.env.call_canister(extension_canister_id, "deposit", arg_blob).await
``` [3](#0-2) 

`construct_treasury_manager_deposit_payload` builds a `DepositRequest { allowances }` with no slippage field: [4](#0-3) 

The `ValidatedDepositOperationArg` that feeds this path carries only `treasury_allocation_sns_e8s`, `treasury_allocation_icp_e8s`, and `original` — no slippage tolerance is parsed or forwarded from the governance proposal: [5](#0-4) 

The IC codebase itself acknowledges this structural gap in two places:

1. `treasury_manager.did` lines 35–40:

> *"Known Security Risks: Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."* [6](#0-5) 

2. `proposal.rs` lines 1540–1545, in the `validate_and_render_register_extension` warning rendered to voters:

> *"Some Decentralized Exchanges lack slippage protection during deposits. Consequently, deposited asset ratios may deviate from those specified in the proposal. This can expose liquidity pool adaptors to mispricing, making them vulnerable to front-running or sandwich attacks."* [7](#0-6) 

Because the `DepositRequest` interface has no slippage field, **no Treasury Manager implementation** (including the KongSwap adaptor integrated and tested in this repo) can accept or enforce slippage bounds from the SNS governance layer. The interface structurally prevents slippage protection from being expressed.

---

### Impact Explanation

An SNS DAO's treasury funds (ICP and SNS tokens) can be extracted via a sandwich attack when a `TreasuryManagerDeposit` governance proposal executes:

1. The attacker observes the pending proposal (all IC governance state is public).
2. Before the governance heartbeat/timer fires the deposit, the attacker submits a large swap to the DEX (e.g., KongSwap) in a prior consensus round, moving the pool price adversely.
3. The governance deposit executes at the manipulated price — the SNS treasury receives far fewer LP tokens than the approved amounts imply.
4. The attacker reverses their swap, capturing the price difference at the SNS treasury's expense.

Because SNS governance voting periods are measured in days, the attacker has ample time to prepare. The IC's lack of a public mempool does not prevent this: the attacker submits their manipulation in a round that precedes the predictable governance execution round. The loss is permanent and proportional to the deposit size and the attacker's capital.

---

### Likelihood Explanation

**Medium.** The attack requires capital to move the DEX pool price, but:
- The governance execution time is publicly observable and predictable.
- The `DepositRequest` interface structurally guarantees no slippage check will occur.
- The IC codebase itself flags this as a known risk, confirming the attack surface is real and understood.
- Any SNS that uses a Treasury Manager extension to deposit into a DEX liquidity pool is exposed on every deposit proposal.

---

### Recommendation

1. **Add a slippage field to `DepositRequest`** in `treasury_manager.did`, e.g.:

```candid
type DepositRequest = record {
  allowances : vec Allowance;
  min_lp_tokens_out : opt nat;   // minimum LP tokens the SNS must receive
  deadline_ns : opt nat64;       // reject execution after this timestamp
};
```

2. **Propagate slippage parameters through the governance proposal** — add `min_lp_tokens_out` and `deadline_ns` to the `ValidatedDepositOperationArg` struct and to the `ExecuteExtensionOperation` argument schema so SNS voters can specify and inspect the slippage tolerance they are approving.

3. **Enforce the deadline in `execute_treasury_manager_deposit`** — reject execution if `env.now()` exceeds the proposal-specified deadline, preventing stale proposals from executing at arbitrarily bad prices.

---

### Proof of Concept

**Setup**: An SNS has a registered KongSwap Treasury Manager extension. A governance proposal is submitted to deposit 100 ICP + 1,000 SNS tokens into the KongSwap ICP/SNS pool. The proposal passes after a multi-day voting period.

**Attack**:

1. Attacker monitors SNS governance and sees the deposit proposal will execute at the next governance heartbeat (predictable from the proposal's `executed_timestamp_seconds` field once voting closes).

2. In the round immediately before the heartbeat, attacker calls KongSwap's swap endpoint to buy a large amount of SNS tokens with ICP, shifting the pool ratio so that ICP is now underpriced relative to SNS.

3. The governance heartbeat fires. `execute_treasury_manager_deposit` calls `approve_treasury_manager` (ICRC-2 approve for 100 ICP + 1,000 SNS), then calls `deposit` on the KongSwap adaptor with `DepositRequest { allowances: [...] }` — no slippage check. [8](#0-7) 

4. The KongSwap adaptor deposits at the manipulated ratio. The SNS treasury receives LP tokens representing a position worth significantly less than 100 ICP + 1,000 SNS at fair market price.

5. Attacker reverses their swap (sells SNS back for ICP), restoring the pool price and pocketing the spread extracted from the SNS treasury.

**Root cause**: `DepositRequest` carries no `min_lp_tokens_out` field, so step 3 cannot enforce any price bound regardless of what the Treasury Manager implementation does. [1](#0-0)

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
```
