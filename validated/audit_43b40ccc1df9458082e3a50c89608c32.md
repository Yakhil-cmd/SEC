### Title
Key Manager Can Bypass Owner-Only Emergency Pause via `start_hashchain` — (File: engine/src/contract_methods/admin.rs)

### Summary
The `start_hashchain` function is gated by `require_key_manager_only` but unconditionally sets `state.is_paused = false`, allowing the registered key manager to resume a paused contract without owner authorization. This gives a registered privileged entity (the key manager) excessive access to a critical function that is otherwise restricted to the owner only.

### Finding Description
The `resume_contract` function correctly enforces `require_owner_only` before setting `state.is_paused = false`: [1](#0-0) 

However, `start_hashchain` performs the same state mutation — `state.is_paused = false` — while only requiring `require_key_manager_only`: [2](#0-1) 

Specifically, line 429 enforces the key-manager gate, and line 460 unconditionally resumes the contract: [3](#0-2) 

The key manager is a registered entity set by the owner via `set_key_manager`, whose intended role is managing relayer function-call keys (`add_relayer_key`, `remove_relayer_key`): [4](#0-3) 

The `require_key_manager_only` guard is defined as: [5](#0-4) 

The key manager is a distinct, separately-registered role from the owner. Its access to `start_hashchain` — which both resets the hashchain with caller-supplied data and resumes the contract — is excessive relative to its intended purpose.

### Impact Explanation
**High — Temporary freezing bypass / potential fund theft.**

If the owner pauses the contract as an emergency measure (e.g., to halt an ongoing exploit draining user funds), the key manager can immediately undo this by calling `start_hashchain` with any valid `StartHashchainArgs`. The `require_paused` precondition means this path is only reachable when the owner has already paused the contract, making it a direct bypass of the owner's emergency response. If the pause was protecting in-flight user funds, resumption allows the attack to continue.

### Likelihood Explanation
**Low.** The key manager is set by the owner and is expected to be a trusted operator. Exploitation requires the key manager account to be compromised (e.g., key exfiltration) or to act maliciously. The design flaw is structural: the key manager is granted access to a contract-resumption side-effect that should be owner-only, creating an unnecessary attack surface.

### Recommendation
Decouple hashchain initialization from contract resumption. `start_hashchain` should not set `state.is_paused = false`. After the hashchain is initialized, the owner should explicitly call `resume_contract`. Alternatively, restrict `start_hashchain` to `require_owner_only`, consistent with all other contract-state-changing administrative functions.

### Proof of Concept
1. An exploit begins draining user funds from the Aurora Engine.
2. The owner calls `pause_contract` — gated by `require_owner_only` — to halt all EVM execution.
3. The key manager (compromised or malicious) calls `start_hashchain` with a valid `StartHashchainArgs` payload (any past block height and any 32-byte hashchain value).
4. `start_hashchain` passes `require_paused` (contract is paused ✓) and `require_key_manager_only` (key manager is caller ✓).
5. Line 460 executes: `state.is_paused = false` — the contract is resumed without owner consent.
6. The exploit continues; user funds are stolen.

The root cause is identical in class to the external report: a registered entity (key manager, analogous to a registered module) is granted access to a critical state-mutation function (`start_hashchain` resuming the contract, analogous to `addModule`/`resetOwners`) that should be restricted to the owner/self only.

#[named]
pub fn start_hashchain<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    let mut state = state::get_state(&io)?;
    require_paused(&state)?;
    require_key_manager_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let args = StartHashchainArgs::try_from_slice(&input).map_err(|_| errors::ERR_SERIALIZE)?;
    let block_height = env.block_height();

    // Starting hashchain must be for an earlier block
    if block_height < args.block_height {
        return Err(errors::ERR_ARGS.into());
    }

    let mut hashchain = Hashchain::new(
        state.chain_id,
        env.current_account_id(),
        args.block_height + 1,
        args.block_hashchain,
    );

    if hashchain.get_current_block_height() < block_height {
        hashchain.move_to_block(block_height)?;
    }

    hashchain.add_block_tx(
        block_height,
        function_name!(),
        &input,
        &[],
        &Bloom::default(),
    )?;
    crate::hashchain::save_hashchain(&mut io, &hashchain)?;

    state.is_paused = false;
    state::set_state(&mut io, &state)?;

    Ok(())
}

fn require_key_manager_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    let key_manager = state
        .key_manager
        .as_ref()
        .ok_or(errors::ERR_KEY_MANAGER_IS_NOT_SET)?;
    if key_manager != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
