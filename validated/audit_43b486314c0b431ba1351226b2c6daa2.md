### Title
Silent Precompile Failure on Empty Recipient Burns ERC-20 Tokens Without NEAR-Side Credit — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`withdrawToNear` burns ERC-20 tokens before calling the exit precompile. When called with a zero-length `recipient`, the precompile's `parse_recipient` rejects the empty `AccountId` and returns `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`. The EVM maps this to `PrecompileFailure::Error`, which causes the inner `CALL` to return 0 (failure) without reverting the calling context. Because the Solidity assembly block does not check `res`, the outer transaction succeeds: tokens are burned, no `ft_transfer` promise is ever scheduled, and the NEP-141 balance remains locked in the connector.

---

### Finding Description

**Step 1 — Entrypoint: `EvmErc20.sol::withdrawToNear`**

`_burn` executes unconditionally before the precompile call:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← burn happens first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;  // = 33 when recipient is empty

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [1](#0-0) 

**Step 2 — Precompile input parsing**

With `recipient = bytes('')`, the 33-byte input (`\x01` + 32-byte amount + 0-byte recipient) passes the `MIN_INPUT_SIZE` check (3 bytes for non-`error_refund`, 21 bytes for `error_refund`). `parse_input` strips the flag byte and returns the 32-byte amount slice. [2](#0-1) 

**Step 3 — `parse_recipient` rejects the empty slice**

```rust
fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)   // "" is valid UTF-8 — succeeds
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None), ...
    );
    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()   // "".parse::<AccountId>() → TooShort (len 0 < MIN=2)
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
``` [3](#0-2) 

`AccountId::validate` enforces `MIN_ACCOUNT_ID_LEN = 2`, so an empty string immediately returns `Err(ParseAccountError::TooShort)`: [4](#0-3) 

**Step 4 — Error propagation: subcall fails, caller is NOT reverted**

`process_precompile` maps `ExitError` → `PrecompileFailure::Error`:

```rust
p.run(input, gas_limit.map(EthGas::new), context, is_static)
    .map_err(|exit_status| PrecompileFailure::Error { exit_status })
``` [5](#0-4) 

Per EVM semantics, `PrecompileFailure::Error` causes the inner `CALL` to return 0 in the success slot, but the **calling frame is not reverted**. The Solidity assembly never inspects `res`, so `withdrawToNear` returns normally.

**Step 5 — No promise is ever created**

The promise log that carries the `ft_transfer` call is only emitted on the `Ok(...)` path of `ExitToNear::run`: [6](#0-5) 

Because the function returned early with `Err(...)`, no `PromiseArgs` log is written, no `ft_transfer` is scheduled, and the NEP-141 connector balance is unchanged.

---

### Impact Explanation

ERC-20 tokens are permanently burned (totalSupply decreases, caller's balance zeroed) with no corresponding NEP-141 transfer on the NEAR side. The NEP-141 tokens remain locked in the Aurora connector, creating a persistent imbalance: the connector holds more NEP-141 than the total ERC-20 supply warrants. The burned tokens are irrecoverable — the caller has no ERC-20 balance to attempt a second withdrawal, and no refund mechanism is triggered because the precompile error occurs before any promise is created.

---

### Likelihood Explanation

Any holder of a bridged ERC-20 token can trigger this by calling `withdrawToNear` with an empty `bytes` argument. No special privilege, admin access, or external dependency is required. The call is a standard EVM transaction. The condition (empty recipient) is trivially constructable and not blocked by any on-chain guard prior to the burn.

---

### Recommendation

1. **Check `res` in the assembly block.** Revert if the precompile call fails:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                       0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Validate `recipient.length > 0` (and ideally a minimum of 2 bytes) in Solidity** before calling `_burn`, so the burn never executes against an invalid recipient.
3. **Enforce `MIN_INPUT_SIZE` to include at least one recipient byte** in `validate_input_size` for the ERC-20 path (currently `MIN_INPUT_SIZE = 3` allows a 33-byte input with a 0-byte recipient).

interface IEvmErc20 {
    function withdrawToNear(bytes memory recipient, uint256 amount) external;
    function balanceOf(address account) external view returns (uint256);
    function totalSupply() external view returns (uint256);

contract PoC {
    function exploit(address erc20, uint256 amount) external {
        IEvmErc20 token = IEvmErc20(erc20);

        uint256 supplyBefore = token.totalSupply();
        uint256 balBefore    = token.balanceOf(address(this));

        // Pass empty recipient — precompile will reject it silently
        token.withdrawToNear(bytes(""), amount);

        uint256 supplyAfter = token.totalSupply();
        uint256 balAfter    = token.balanceOf(address(this));

        // Assert: burn executed (supply and balance decreased)
        assert(supplyAfter == supplyBefore - amount);
        assert(balAfter    == balBefore    - amount);

        // Assert: no ft_transfer promise was scheduled
        // (verified off-chain: NEP-141 connector balance unchanged)
    }
}

- Caller's ERC-20 balance decreases by `amount` ✓
- No `ft_transfer` NEAR promise is created ✓
- NEP-141 connector balance is unchanged ✓
- Transaction status: `Succeed` (not reverted) ✓
