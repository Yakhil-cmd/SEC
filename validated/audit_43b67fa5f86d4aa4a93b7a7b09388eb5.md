Audit Report

## Title
Unchecked `envelope_pairs[0]` Index in `construction_hash` Causes Panic on Empty `Vec<EnvelopePair>` — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

## Summary
In `construction_hash`, after finding the first `RequestType::Send` entry via `.find(|(rt, _)| rt.is_transfer())`, the code unconditionally accesses `envelope_pairs[0]` without a bounds check. Because `SignedTransaction::from_str` performs pure CBOR deserialization with no structural validation, an unauthenticated attacker can supply a hex-encoded CBOR blob encoding a `Request` with `RequestType::Send` and an empty `Vec<EnvelopePair>`, triggering an index-out-of-bounds panic. This crashes the actix-web worker thread handling the request, and repeated requests degrade Rosetta API availability.

## Finding Description
`Request` is defined as `pub type Request = (RequestType, Vec<EnvelopePair>)` at `models.rs:58`. `SignedTransaction::from_str` at `models.rs:37–48` performs only CBOR deserialization with no assertion that the inner `Vec<EnvelopePair>` is non-empty.

In `construction_hash.rs:18–28`, after the `.find(|(rt, _)| rt.is_transfer())` call matches a `RequestType::Send` entry, the code immediately executes:

```rust
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pairs[0].update,  // line 27 — unchecked index
)
```

If `envelope_pairs` is empty, this panics unconditionally. There is no `if envelope_pairs.is_empty()` guard, no `.first().ok_or_else(...)` call, and no `?`-propagated error.

The contrast is explicit: `TryFrom<&models::Request> for Request` in `request.rs:237–239` uses:
```rust
let payload: &models::EnvelopePair = calls
    .first()
    .ok_or_else(|| ApiError::invalid_request("No request payload provided."))?;
```
`construction_hash` is the only production call site that uses the bare `[0]` index on attacker-controlled data.

The HTTP handler at `rosetta_server.rs:97–104` is registered with actix-web with no authentication middleware and no `catch_unwind` wrapper. `is_transfer()` at `request_types.rs:137–139` confirms `RequestType::Send` is the sole match condition.

## Impact Explanation
A Rust index-out-of-bounds panic in an actix-web handler propagates through the async task and kills the worker thread. actix-web's `HttpServer` restarts crashed workers, but repeated requests keep workers in a crash-restart cycle, degrading throughput and availability of the ICP Rosetta API. This constitutes an application-level DoS against a Rosetta financial integration endpoint, matching the allowed High impact: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … Rosetta … infrastructure security impact with concrete user or protocol harm."* Exchanges and integrations relying on the Rosetta node for ICP transaction hash derivation and submission are directly affected.

## Likelihood Explanation
The endpoint is publicly reachable with no authentication. Crafting the exploit requires only encoding a valid CBOR blob with `RequestType::Send` and an empty `Vec<EnvelopePair>` — a trivial operation using any CBOR library, with all type definitions publicly available in the open-source codebase. A single malformed request is sufficient to trigger the panic; the attack is repeatable at negligible cost.

## Recommendation
Replace the unchecked index with a bounds-checked accessor mirroring the pattern in `request.rs`:

```rust
let first_pair = envelope_pairs
    .first()
    .ok_or_else(|| ApiError::invalid_request("No envelope pairs provided for Send request"))?;
TransactionIdentifier::try_from_envelope(request_type.clone(), &first_pair.update)
```

This returns a proper `ApiError` on empty input instead of panicking.

## Proof of Concept
```rust
// Craft the malicious payload
let malicious = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],  // empty EnvelopePair vec
};
let cbor_bytes = serde_cbor::to_vec(&malicious).unwrap();
let hex_payload = hex::encode(&cbor_bytes);

// POST to /construction/hash
let body = serde_json::json!({
    "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
    "signed_transaction": hex_payload
});
// Expected: ApiError response
// Actual: index out of bounds panic at envelope_pairs[0] → worker thread crash
```

The `.find` at `construction_hash.rs:23` matches the `RequestType::Send` entry (since `is_transfer()` returns `true` only for `RequestType::Send`), `envelope_pairs` is the empty `vec![]`, and `envelope_pairs[0]` panics. A deterministic unit test can reproduce this by calling `RosettaRequestHandler::construction_hash` directly with the crafted `SignedTransaction`.
