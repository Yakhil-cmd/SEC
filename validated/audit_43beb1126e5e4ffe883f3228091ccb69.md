### Title
Unpermissioned `deploy_erc20_token` Allows Any Caller to Pre-Register Arbitrary NEP-141 Tokens, Permanently Corrupting Bridge Accounting - (File: engine/src/contract_methods/connector.rs)

### Summary

The `deploy_erc20_token` contract method (Legacy variant) has no caller authorization check. Any unprivileged NEAR account can invoke it with an arbitrary NEP-141 account ID, deploying a fake ERC-20 mapping for a token they do not control. Once a NEP-141 is registered, subsequent legitimate registration attempts fail, and all future `ft_on_transfer` bridge deposits for that token are routed to the attacker-controlled ERC-20 address, permanently freezing or misdirecting bridged user funds.

### Finding Description

`deploy_erc20_token` is exported as a public WASM entry point with no access control beyond `require_running`:

```rust
// engine/src/contract_methods/connector.rs, lines 112-159
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // ← NO require_owner_only or any other caller check
        let bytes = io.read_input().to_vec();
        let args = DeployErc20TokenArgs::deserialize(&bytes)...
        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;
                ...
            }
```

The WASM export confirms it is reachable by any NEAR account:

```rust
// engine/src/lib.rs
#[unsafe(no_mangle)]
pub extern "C" fn deploy_erc20_token() {
    let io = Runtime;
    let env = Runtime;
    let mut handler = Runtime;
    contract_methods::connector::deploy_erc20_token(io, &env, &mut handler)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
```

Inside `engine::deploy_erc20_token`, the engine deploys an ERC-20 contract and permanently writes the `nep141 → erc20_address` mapping via `register_token`:

```rust
// engine/src/engine.rs, lines 1370-1372
engine
    .register_token(address, nep141)
    .map_err(DeployErc20Error::Register)?;
```

The `WithMetadata` variant carries a misleading comment ("this transaction could be executed by the owner of the contract only") but also enforces no such restriction at the call site.

The bridge's `ft_on_transfer` handler resolves the ERC-20 address exclusively from this stored mapping:

```rust
// engine/src/engine.rs, line 824
let erc20_token = get_erc20_from_nep141(&self.io, token)?;
```

Once an attacker pre-registers a NEP-141 (e.g., `usdc.near`) with a fake ERC-20 address, every subsequent `ft_on_transfer` call from the real `usdc.near` contract mints tokens into the attacker-registered ERC-20 contract. The legitimate owner can never re-register the same NEP-141 because `register_token` fails on a duplicate key, making the corruption permanent.

### Impact Explanation

**Critical — Permanent freezing of bridged user funds / ERC-20 mirror accounting corruption.**

An attacker pre-registers any high-value NEP-141 token (USDC, wBTC, etc.) before the legitimate operator does. All subsequent bridge deposits for that token are minted into the attacker-registered ERC-20 contract. The NEAR-side tokens are consumed (transferred to Aurora), but users receive tokens in a contract that is not the canonical ERC-20 for that asset. The legitimate ERC-20 can never be deployed for that NEP-141 because the mapping slot is already occupied. Bridged funds are permanently frozen in an unrecognized ERC-20 contract.

### Likelihood Explanation

**High.** The entry point is a public WASM export callable by any NEAR account with no deposit requirement and no whitelist. The attacker only needs to know the account ID of a NEP-141 token that has not yet been registered on Aurora (e.g., any newly announced bridge token). The attack is a single transaction with negligible cost.

### Recommendation

Add a `require_owner_only` (or equivalent authorized-caller) check at the top of `deploy_erc20_token`, consistent with the pattern used in other sensitive methods such as `factory_update`, `factory_set_wnear_address`, and `mirror_erc20_token`:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
+       require_owner_only(&state, &env.predecessor_account_id())?;
        ...
    })
}
```

If permissionless deployment is desired in the future, a separate allowlist of trusted NEP-141 contracts should gate the call.

### Proof of Concept

1. Attacker observes that `usdc.near` has not yet been registered on Aurora.
2. Attacker calls `deploy_erc20_token` on the Aurora Engine contract with `DeployErc20TokenArgs::Legacy("usdc.near")`.
3. The engine deploys an ERC-20 at a deterministic address and writes `usdc.near → attacker_erc20_address` into storage.
4. The legitimate operator later calls `deploy_erc20_token` for `usdc.near`; `register_token` fails because the mapping already exists — the call reverts.
5. A user bridges 1000 USDC via `ft_transfer_call` on `usdc.near` targeting Aurora. The `ft_on_transfer` callback fires with `predecessor = usdc.near`.
6. `get_erc20_from_nep141` returns `attacker_erc20_address`. The engine mints 1000 tokens into `attacker_erc20_address` for the user.
7. The user's 1000 NEAR-side USDC are consumed; the tokens they receive are in an unrecognized ERC-20 contract. The funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}
```

```rust
/// Used to bridge NEP-141 tokens from NEAR to Aurora. On Aurora the NEP-141 becomes an ERC-20.
pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
```
