### Title
Missing `attached_deposit` Validation in EVM Transaction Entry Points — (File: `engine/src/contract_methods/evm_transactions.rs`)

### Summary
The `deploy_code` and `call` functions (and by extension `submit`) in the Aurora Engine contract do not validate that `env.attached_deposit() == 0`. These functions process EVM transactions and require no NEAR deposit. If a user inadvertently attaches NEAR tokens to any of these calls, the NEAR is permanently absorbed into the Aurora Engine contract's balance with no automatic recovery path.

### Finding Description
The `deploy_code` function in `engine/src/contract_methods/evm_transactions.rs` reads the EVM input and executes the deployment with a hardcoded `value = Wei::zero()`: [1](#0-0) 

The internal dispatch immediately calls `deploy_code_with_input`, which hardcodes the EVM value to zero: [2](#0-1) 

Neither function reads or validates `env.attached_deposit()`. The `Env` trait exposes `attached_deposit()` as the amount of NEAR attached to the current NEAR call: [3](#0-2) 

By contrast, security-sensitive functions such as `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_unregister`, and `storage_withdraw` all call `env.assert_one_yocto()` to enforce an exact deposit amount: [4](#0-3) 

No analogous `assert_zero_deposit` guard exists for `deploy_code`, `call`, or `submit`. The Aurora Engine uses its own SDK (`engine-sdk`) rather than `near-sdk`, so the NEAR runtime's `#[payable]` attribute enforcement is absent — the runtime will silently accept any deposit on these functions: [5](#0-4) 

Any NEAR attached to `deploy_code`, `call`, or `submit` is transferred into the Aurora Engine contract's account balance and is not returned to the caller.

### Impact Explanation
A user who inadvertently attaches NEAR tokens to `deploy_code`, `call`, or `submit` permanently loses those tokens. The NEAR is absorbed into the Aurora Engine contract's balance. There is no automatic refund mechanism. The contract owner could manually redistribute the balance, but no on-chain guarantee exists. This is a direct, permanent loss of user funds in motion — matching the "Critical: direct theft of any user funds, whether at-rest or in-motion" category.

### Likelihood Explanation
Medium-low. NEAR-native users who interact with Aurora directly (e.g., via NEAR CLI or custom tooling) may attach NEAR by mistake, particularly when calling `deploy_code` or `call`, which are NEAR-facing entry points. The risk is lower for `submit` since it is typically invoked by relayers that do not attach deposits. However, the absence of any guard means any accidental deposit is silently and permanently lost.

### Recommendation
Add an explicit zero-deposit assertion at the top of `deploy_code`, `call`, and `submit` in `engine/src/contract_methods/evm_transactions.rs`, analogous to the `assert_one_yocto` pattern already used elsewhere:

```rust
if env.attached_deposit() != 0 {
    return Err(errors::ERR_WRONG_ATTACHED_DEPOSIT.into());
}
```

This mirrors the fix recommended in the external report (`require(msg.value == 0, "CL1IN: wrong msg.value")`), adapted to the NEAR/Aurora context.

### Proof of Concept
1. A user constructs a NEAR transaction calling `deploy_code` on the Aurora Engine contract with 5 NEAR attached.
2. The function executes: `require_running` passes, EVM deployment proceeds with `value = Wei::zero()` — the EVM state is updated correctly.
3. `env.attached_deposit()` is never read or validated; the 5 NEAR is transferred into the Aurora Engine contract's account at the NEAR protocol level.
4. The function returns `Ok(result)` with no error and no refund.
5. The user's 5 NEAR is permanently lost with no on-chain recovery path. [6](#0-5) [7](#0-6)
