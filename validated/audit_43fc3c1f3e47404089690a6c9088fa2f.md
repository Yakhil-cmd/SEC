Audit Report

## Title
Excess ICP Locked in Buyer Subaccount During Open Swap With No Recovery Path Until Swap Closes - (File: rs/sns/swap/src/swap.rs)

## Summary
In `rs/sns/swap/src/swap.rs`, the `refresh_buyer_token_e8s` function caps accepted ICP at the per-participant maximum (`max_participant_icp_e8s`) and the remaining global capacity (`available_direct_participation_e8s()`), but does not transfer the excess ICP back to the buyer. The excess remains locked in the buyer's subaccount of the Swap canister for the entire duration of the open swap. The only recovery mechanism, `error_refund_icp`, is explicitly gated behind the swap reaching `Aborted` or `Committed` state, which can be up to 90 days away.

## Finding Description
In `rs/sns/swap/src/swap.rs`, `refresh_buyer_token_e8s` (lines 1134–1312) reads the buyer's full ICP balance `e8s` from the ledger, then computes a capped `new_balance_e8s`:

- Line 1224: `let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);` — caps at global remaining capacity.
- Line 1237: `let new_balance_e8s = std::cmp::min(new_balance_e8s, max_participant_icp_e8s);` — caps at per-participant limit.

The function then records only `new_balance_e8s` in buyer state (line 1288) and returns `Ok(RefreshBuyerTokensResponse { icp_accepted_participation_e8s: new_balance_e8s, icp_ledger_account_balance_e8s: e8s })` (lines 1308–1311) without initiating any transfer of the excess `e8s - new_balance_e8s` back to the buyer.

The only recovery path is `error_refund_icp` (line 1925), which hard-gates on `Lifecycle::Aborted || Lifecycle::Committed` (lines 1932–1935). During the `Open` lifecycle, calling `error_refund_icp` returns a precondition error. The developer-acknowledged TODO at line 1133 (`TODO(NNS1-1682): attempt to refund ICP that cannot be accepted`) confirms this is a known unaddressed gap. The docstring on `error_refund_icp` (lines 1906–1909) also explicitly acknowledges it handles ICP "not fully accepted (by refresh_buyer_tokens)" — confirming the design relies on post-swap recovery rather than in-call refund.

The externally callable entry point is `refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` (lines 127–143), callable by any unprivileged principal.

## Impact Explanation
Any direct swap participant who transfers more ICP than `max_participant_icp_e8s` or more than `available_direct_participation_e8s()` will have the excess ICP locked in the Swap canister's subaccount for the full remaining duration of the open swap — up to 90 days per SNS swap parameters. The buyer cannot withdraw, stake, or use those tokens elsewhere during this period. This constitutes a significant, time-locked loss of liquidity for users, matching the allowed impact: **High — Significant SNS security impact with concrete user or protocol harm**. The funds are not permanently lost but are inaccessible for an extended, protocol-enforced period with no in-swap remedy.

## Likelihood Explanation
This is straightforwardly triggered by any unprivileged user during a popular SNS launch. When a swap is nearly full, any buyer sending more than `available_direct_participation_e8s()` triggers the lock. Any buyer sending more than `max_participant_icp_e8s` also triggers it. No special privileges, coordination, or attack setup is required — it is a normal participation flow. Both conditions are routine during high-demand SNS launches.

## Recommendation
Inside `refresh_buyer_token_e8s`, after computing `new_balance_e8s` and before returning, calculate `excess_e8s = e8s.saturating_sub(new_balance_e8s)`. If `excess_e8s > DEFAULT_TRANSFER_FEE`, initiate an ICP ledger transfer from `subaccount(swap_canister, buyer)` back to the buyer's principal account, following the same pattern used in `error_refund_icp` (lines 1971–2004). This directly addresses the open TODO at line 1133 (`NNS1-1682`).

## Proof of Concept
1. Deploy a local SNS swap canister with `max_participant_icp_e8s = 5_0000_0000` (5 ICP) and configure the swap so `available_direct_participation_e8s() = 3_0000_0000` (3 ICP, swap nearly full).
2. As buyer principal `P`, transfer 5 ICP to `subaccount(swap_canister, P)` on the ICP ledger.
3. Call `refresh_buyer_tokens` as `P`.
4. Observe the response: `icp_accepted_participation_e8s = 3_0000_0000`, `icp_ledger_account_balance_e8s = 5_0000_0000`. The 2 ICP excess is not returned.
5. Immediately call `error_refund_icp` with `source_principal_id = P`. Observe it returns `"Error refunds can only be performed when the swap is ABORTED or COMMITTED"`.
6. Verify via ledger query that `subaccount(swap_canister, P)` still holds 2 ICP, inaccessible until swap closes.
7. This is reproducible as a PocketIC integration test using the SNS swap canister WASM with a controlled ICP ledger mock.
