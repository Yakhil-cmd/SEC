### Title
Call Depth Limit Check Off-By-One Allows Exceeding EVM 1024-Frame Limit — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

The `check_depth_and_balance` function enforces the EVM call-stack depth limit using `callstack_depth > 1024` instead of the correct `callstack_depth >= 1024`. Because `callstack_depth` is the depth of the **callee frame being launched** (root = 0), this off-by-one allows a frame at depth 1024 to execute when the EVM specification requires it to fail. The function's name and purpose imply it correctly enforces the depth limit, but the implementation silently permits one extra level of nesting — a direct analog to the external report's pattern of a "safe" function that skips the very check its name promises.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, `check_depth_and_balance` is the single authoritative gate for the EVM call-depth limit. It is invoked from both `before_executing_frame` (all call types) and `constructor_pre_checks` (CREATE/CREATE2):

```rust
// evm_interpreter/src/ee_trait_impl.rs  line 460
fn check_depth_and_balance<S: EthereumLikeTypes>(
    system: &mut System<S>,
    call_request: &mut ExternalCallRequest<S>,
    callstack_depth: usize,
) -> Result<Option<EvmError>, SubsystemError<EvmErrors>>
{
    if callstack_depth > 1024 {          // ← BUG: should be >= 1024
        system_log!(system, "Callstack is too deep\n",);
        return Ok(Some(EvmError::CallTooDeep));
    }
    ...
}
```

`callstack_depth` is the depth of the **new frame** (callee). The root transaction frame is at depth 0, confirmed by the constructor guard:

```rust
// evm_interpreter/src/ee_trait_impl.rs  line 391
if frame_state.environment_parameters.callstack_depth > 0 {
    // Increase nonce. Ignore, if we are in the root frame …
```

The EVM specification (Yellow Paper, Cancun) states the call-stack depth limit is **1024**, meaning a frame at depth ≥ 1024 must not execute. With `> 1024`, a frame at depth exactly 1024 passes the guard and executes, violating the spec by one level.

---

### Impact Explanation

1. **EVM semantic mismatch / divergence from Ethereum**: ZKsync OS permits a 1025-frame call stack where Ethereum allows only 1024. Any contract whose security relies on the depth-1024 boundary (e.g., reentrancy guards that exploit call-depth exhaustion, or contracts that test whether a sub-call reverts due to depth) will behave differently on ZKsync OS than on Ethereum mainnet.

2. **Forward/proving consistency**: Both execution modes share the same `check_depth_and_balance` path, so the divergence is consistent internally but incorrect relative to the EVM specification. A prover verifying EVM-equivalence would accept a trace that Ethereum would reject.

3. **Concrete exploit surface**: An attacker who can control a recursive call chain can reach depth 1024 and execute code that Ethereum would have silently failed. Contracts that use call-depth as a guard (e.g., "if this sub-call fails, it must be because the stack is full") will have their assumption violated.

---

### Likelihood Explanation

Reaching depth 1024 requires significant gas (each frame consumes at minimum the `CALL` base cost). However:
- The bug is unconditional — it applies to every CALL, CALLCODE, DELEGATECALL, STATICCALL, CREATE, and CREATE2.
- No privileged access is required; any unprivileged transaction sender can craft a recursive contract.
- The off-by-one is silent: no error, no revert, no observable signal that the limit was exceeded.

Likelihood is **low-to-medium**: the gas cost is a practical barrier, but the attack is fully permissionless and deterministic.

---

### Recommendation

Change the comparison in `check_depth_and_balance`:

```rust
// Before (incorrect):
if callstack_depth > 1024 {

// After (correct):
if callstack_depth >= 1024 {
```

Apply the same fix to `constructor_pre_checks`, which delegates to `check_depth_and_balance` and inherits the same off-by-one.

---

### Proof of Concept

Deploy a contract `Recurse` that calls itself until the call fails, then checks the depth at which failure occurred:

```solidity
contract Recurse {
    uint256 public maxDepth;
    function go(uint256 depth) external {
        (bool ok,) = address(this).call(
            abi.encodeWithSelector(this.go.selector, depth + 1)
        );
        if (!ok && depth > maxDepth) maxDepth = depth;
    }
}
```

On Ethereum mainnet, `maxDepth` settles at **1023** (the deepest frame that can still make a call that fails at 1024).  
On ZKsync OS with this bug, `maxDepth` settles at **1024** — one level deeper than the EVM specification permits — demonstrating that the frame at depth 1024 executes instead of being rejected by `check_depth_and_balance`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

fn check_depth_and_balance<S: EthereumLikeTypes>(
    system: &mut System<S>,
    call_request: &mut ExternalCallRequest<S>,
    callstack_depth: usize,
) -> Result<Option<EvmError>, SubsystemError<EvmErrors>>
where
    S::IO: IOSubsystemExt,
{
    if callstack_depth > 1024 {
        system_log!(system, "Callstack is too deep\n",);
        return Ok(Some(EvmError::CallTooDeep));
    }

    // Check caller has enough balance for token transfer
    if !call_request.nominal_token_value.is_zero() && !call_request.is_delegate() {
        let caller_balance = call_request
            .available_resources
            .with_infinite_ergs(|inf_resources| {
                system.io.read_account_properties(
                    THIS_EE_TYPE,
                    inf_resources,
                    &call_request.caller,
                    AccountDataRequest::empty().with_nominal_token_balance(),
                )
            })?
            .nominal_token_balance
            .0;

        if caller_balance < call_request.nominal_token_value {
            system_log!(system, "Not enough balance for transfer\n",);
            return Ok(Some(EvmError::InsufficientBalance));
        }
    }

    Ok(None)
}

/// Checks that must pass before a CREATE/CREATE2 callee is read:
/// depth, balance, and nonce overflow (read-only check).
fn constructor_pre_checks<S: EthereumLikeTypes>(
    system: &mut System<S>,
    call_request: &mut ExternalCallRequest<S>,
    callstack_depth: usize,
) -> Result<Option<EvmError>, SubsystemError<EvmErrors>>
where
    S::IO: IOSubsystemExt,
{
    if let Some(error) = check_depth_and_balance(system, call_request, callstack_depth)? {
        return Ok(Some(error));
    }

    // Read-only nonce overflow check (actual increment happens in before_executing_frame)
    if callstack_depth > 0 {
        let caller_nonce = call_request
            .available_resources
            .with_infinite_ergs(|inf_resources| {
                system.io.read_account_properties(
                    THIS_EE_TYPE,
                    inf_resources,
                    &call_request.caller,
                    AccountDataRequest::empty().with_nonce(),
                )
            })?
            .nonce
            .0;
        if caller_nonce == u64::MAX {
            return Ok(Some(EvmError::NonceOverflow));
        }
    }

    Ok(None)
}
