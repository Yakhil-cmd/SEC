### Title
`ic0.cycles_burn128` Does Not Account for Pending `ingress_induction_cycles_debit`, Allowing Free Ingress Induction During DTS Execution - (File: `rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs`)

---

### Summary

During Deterministic Time Slicing (DTS), when a canister's execution is paused between slices, new ingress messages can arrive and their induction cost is recorded as a **postponed charge** (`ingress_induction_cycles_debit`) against the canister's balance. When the canister's Wasm code subsequently calls `ic0.cycles_burn128`, the burn calculation uses the raw `cycles_balance()` — which does not subtract the pending debit — allowing the canister to burn cycles that are already committed to cover the postponed ingress induction cost. When `apply_ingress_induction_cycles_debit` is later called, the balance is insufficient, and the remaining debit is silently **dropped**, making those ingress messages effectively free.

---

### Finding Description

The `cycles_burn128` method in `SandboxSafeSystemState` computes the available cycles to burn using `self.cycles_balance()`:

```rust
pub(super) fn cycles_burn128(
    &mut self,
    amount_to_burn: Cycles,
    canister_current_memory_usage: NumBytes,
    canister_current_message_memory_usage: MessageMemoryUsage,
) -> Cycles {
    let mut new_balance = self.cycles_balance();   // <-- raw balance, ignores pending debit
    let burned_cycles = self.cycles_account_manager.cycles_burn(
        &mut new_balance,
        amount_to_burn,
        ...
    );
    ...
    burned_cycles
}
``` [1](#0-0) 

`cycles_balance()` returns the raw balance tracked inside the sandbox, without accounting for the pending `ingress_induction_cycles_debit` stored in `SystemState`:

```rust
pub fn debited_balance(&self) -> Cycles {
    // We rely on saturating operations of `Cycles` here.
    self.cycles_balance - self.ingress_induction_cycles_debit
}
``` [2](#0-1) 

The `cycles_burn` function then computes:

```rust
let burning = min(amount_to_burn, *cycles_balance - threshold);
``` [3](#0-2) 

This allows burning up to `balance - freeze_threshold` cycles, completely ignoring the `ingress_induction_cycles_debit`. After execution completes, `apply_ingress_induction_cycles_debit` is called and discovers the balance is insufficient:

```rust
let remaining_debit = self.ingress_induction_cycles_debit - self.cycles_balance;
debug_assert_eq!(remaining_debit.get(), 0);
if remaining_debit.get() > 0 {
    // This case is unreachable and may happen only due to a bug
    charging_from_balance_error.inc();
    error!(log, "[EXC-BUG]: Debited cycles exceed the cycles balance...");
    // Continue the execution by dropping the remaining debit, which makes
    // some of the postponed charges free.
}
self.consume_cycles(CompoundCycles::<IngressInduction>::new(
    self.ingress_induction_cycles_debit,
    cost_schedule,
));
self.ingress_induction_cycles_debit = Cycles::zero();
``` [4](#0-3) 

The remaining debit is dropped. The `debug_assert_eq!` is a no-op in production release builds, so this path is taken silently.

By contrast, `charge_ingress_induction_cost` correctly uses `debited_balance()` when a paused execution is present:

```rust
if canister.has_paused_execution_or_install_code() {
    if canister.system_state.debited_balance() < cycles + threshold {
        return Err(...);
    }
    canister.system_state.add_postponed_charge_to_ingress_induction_cycles_debit(cycles);
``` [5](#0-4) 

The ingress induction path correctly accounts for the debit when *adding* a postponed charge, but `cycles_burn128` does not account for it when *burning*.

---

### Impact Explanation

A malicious canister developer can cause ingress induction costs to be dropped (paid for free). Concretely:

- The canister's balance is debited for ingress induction of messages sent during DTS pause, but those debits are silently forgiven after the canister burns the committed cycles.
- This is a **cycles/resource accounting bug**: the subnet absorbs the cost of ingress induction without receiving payment, breaking the economic invariant that ingress messages must be paid for.
- The `[EXC-BUG]` error counter is incremented and an error is logged, but execution continues normally with the debit dropped.

---

### Likelihood Explanation

- Requires a canister developer to deploy a canister with a long-running update method (to trigger DTS) that calls `ic0.cycles_burn128`.
- DTS is a standard production feature active on all subnets for long-running executions.
- The attacker must send ingress messages to the canister while it is paused between DTS slices — this is a normal, unprivileged operation.
- The canister's Wasm code must call `ic0.cycles_burn128` in a subsequent slice — this is a standard system API available to all canisters.
- No privileged access, governance majority, or threshold corruption is required.

---

### Recommendation

In `cycles_burn128`, subtract the pending `ingress_induction_cycles_debit` from the available balance before computing how many cycles can be burned. The sandbox should be initialized with or have access to the pending debit so that:

```rust
let available = self.cycles_balance() - self.ingress_induction_cycles_debit;
let burned_cycles = self.cycles_account_manager.cycles_burn(
    &mut available,
    amount_to_burn,
    ...
);
```

Alternatively, pass `ingress_induction_cycles_debit` into `SandboxSafeSystemState` at construction time (alongside `initial_cycles_balance`) and subtract it in `cycles_burn128` before calling `cycles_account_manager.cycles_burn`. This mirrors how `charge_ingress_induction_cost` already uses `debited_balance()` to guard against over-commitment.

---

### Proof of Concept

1. Deploy a canister with a Wasm update method that:
   - Runs enough instructions to trigger DTS (spans multiple rounds).
   - In a later slice, calls `ic0.cycles_burn128(balance - freeze_threshold)` to burn all available cycles.

2. Send an ingress message to trigger the long-running update. The canister begins DTS execution and is paused after the first slice.

3. While the canister is paused, send one or more additional ingress messages to the same canister. Each message's induction cost is added via `add_postponed_charge_to_ingress_induction_cycles_debit`. [6](#0-5) 

4. The canister resumes. `cycles_burn128` computes available cycles as `cycles_balance - freeze_threshold`, ignoring the pending debit, and burns all of them. [7](#0-6) 

5. After execution, `apply_ingress_induction_cycles_debit` finds `remaining_debit > 0`, logs `[EXC-BUG]`, and drops the remaining debit. [8](#0-7) 

6. The ingress induction cost of the messages sent in step 3 is never collected. The canister has effectively obtained free ingress induction.
