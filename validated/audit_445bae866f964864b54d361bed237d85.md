### Title
`EvmErc20V2.withdrawToNear` Passes Sender Address Bytes as Amount to `ExitToNear` Precompile When `error_refund` Feature Is Disabled, Causing Permanent ERC-20 Token Loss — (File: `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.sol` unconditionally prepends the caller's 20-byte address into the calldata sent to the `ExitToNear` precompile. The precompile's `parse_input` only strips those 20 bytes when compiled with the `error_refund` feature flag. Without that flag, `parse_input` strips only the 1-byte flag, leaving the 20-byte sender address at the front of the remaining slice. The subsequent `parse_amount(&input[..32])` then reads `sender_address(20 bytes) + amount_b[0..12]` as the U256 amount — a value that virtually always exceeds `u128::MAX` — causing the precompile sub-call to fail with `ERR_INVALID_AMOUNT`. Because the `_burn` call precedes the assembly `call` and the return value of the assembly `call` is never checked, the ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer to NEAR.

---

### Finding Description

**`EvmErc20V2.sol` — `withdrawToNear`** always encodes the sender address into the precompile calldata:

```solidity
// etc/eth-contracts/contracts/EvmErc20V2.sol  lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                                          // ← burn committed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    //                                             ^^^^^^ always 20 bytes
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [1](#0-0) 

The `ExitToNear` precompile's `parse_input` has two compile-time variants:

```rust
// engine-precompiles/src/native.rs  lines 778-791
#[cfg(feature = "error_refund")]
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    // strips flag byte + 20-byte refund address → remaining starts at byte 21
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))
}

#[cfg(not(feature = "error_refund"))]
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    // strips only the flag byte → remaining starts at byte 1
    Ok(&input[1..])
}
``` [2](#0-1) 

After `parse_input`, the ERC-20 branch reads the amount from the first 32 bytes of the remaining slice:

```rust
// engine-precompiles/src/native.rs  lines 758-763
0x1 => {
    let amount = parse_amount(&input[..32])?;
    ...
}
``` [3](#0-2) 

`parse_amount` rejects any value exceeding `u128::MAX`:

```rust
// engine-precompiles/src/native.rs  lines 337-345
fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    Ok(amount)
}
``` [4](#0-3) 

**Without `error_refund`**, the 32 bytes fed to `parse_amount` are:

```
input[0..32] = sender_address[0..20]  (20 bytes)
             + amount_b[0..12]         (first 12 bytes of the real amount)
```

The resulting U256 has the sender's 20-byte address in its high bits. For any non-zero Ethereum address, this value is on the order of `2^(96+)`, which is far above `u128::MAX` (`2^128 - 1`). `parse_amount` returns `ERR_INVALID_AMOUNT`, the precompile sub-call fails, and the assembly `call` returns `0`. Because the return value is never checked, the outer `withdrawToNear` function returns successfully — but the `_burn` that preceded the assembly call has already been committed to EVM state.

**Contrast with `EvmErc20.sol` (V1)**, which does not include the sender address and works correctly without `error_refund`:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol  lines 53-63
bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
//                                    no sender bytes
``` [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Every call to `EvmErc20V2.withdrawToNear` on an Aurora instance compiled without `error_refund` results in:

1. The caller's ERC-20 balance is permanently reduced by `amount` (burn committed to EVM state).
2. No NEP-141 tokens are transferred to NEAR (precompile call fails silently).
3. The tokens are irrecoverable — there is no refund path and no admin escape hatch.

The loss is proportional to the `amount` argument supplied by the caller. Any user who attempts to bridge ERC-20 tokens back to NEAR via `withdrawToNear` loses their entire withdrawal amount.

---

### Likelihood Explanation

**Medium.** The `error_refund` feature is a compile-time Cargo feature, not always enabled. It appears as an optional feature in `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, and is toggled in `Makefile.toml`. Any Aurora deployment (mainnet, testnet, or silo) that ships `EvmErc20V2.sol` as its ERC-20 mirror contract while compiling the engine without `error_refund` will exhibit this bug for every user who calls `withdrawToNear`. The entry path requires no special privileges — any token holder can trigger it. [6](#0-5) 

---

### Recommendation

1. **Check the precompile return value in `EvmErc20V2.sol`**: revert the entire transaction if the assembly `call` returns `0`, so the `_burn` is rolled back on precompile failure.
2. **Align calldata layout with the compiled feature**: either always include the sender address (and always compile with `error_refund`), or never include it (and always compile without `error_refund`). The two contracts and the two feature variants must be kept in sync.
3. **Guard deployment**: prevent `EvmErc20V2.sol` from being deployed on engine builds that lack `error_refund`, or unify the two contracts into a single version that is correct under both compilation modes.

---

### Proof of Concept

**Setup**: Deploy Aurora engine compiled **without** `error_refund`. Deploy a NEP-141 token and its `EvmErc20V2` mirror. Bridge 1000 tokens to the ERC-20 side.

**Attack**:
```solidity
// Victim calls:
evmErc20V2.withdrawToNear("victim.near", 1000);
```

**Trace**:
1. `_burn(victim, 1000)` executes — ERC-20 balance decremented by 1000.
2. `input = abi.encodePacked("\x01", victim_address, bytes32(1000), "victim.near")`
   - `victim_address` = e.g. `0xAbCd...` (20 bytes)
3. Assembly `call` → `ExitToNear` precompile.
4. `parse_input` (no `error_refund`): strips only flag byte → `input[1..]` = `victim_address(20) + bytes32(1000)(32) + "victim.near"`.
5. `parse_amount(&input[..32])` = `U256::from_big_endian(victim_address ++ bytes32(1000)[0..12])` ≈ `0xAbCd... * 2^96 + ...` >> `u128::MAX`.
6. Returns `Err("ERR_INVALID_AMOUNT")` → sub-call fails → assembly `res = 0`.
7. `res` is not checked → `withdrawToNear` returns `success`.

**Result**: Victim's 1000 ERC-20 tokens are burned. Zero NEP-141 tokens arrive on NEAR. Funds are permanently lost. [1](#0-0) [7](#0-6) [3](#0-2) [4](#0-3)
