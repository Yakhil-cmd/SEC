### Title
Unchecked Assembly `call()` Return Value After Token Burn Enables Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` use inline assembly to invoke the `ExitToNear` and `ExitToEthereum` precompiles. In both `withdrawToNear` and `withdrawToEthereum`, the ERC-20 tokens are **burned before** the assembly `call()` is made, and the return value `res` is **never checked**. If the precompile call fails (returns 0), execution continues silently, the burn is committed, and the user's tokens are permanently destroyed with no corresponding withdrawal.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` burns the caller's tokens and then invokes the `ExitToNear` precompile via assembly:

```solidity
// EvmErc20.sol L53-L63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert on failure
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`: [2](#0-1) [3](#0-2) [4](#0-3) 

The `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) can return an `ExitError` (causing the sub-call to return 0) in multiple reachable conditions:

- The ERC-20 contract is not registered in the NEP-141 map (`ERR_TARGET_TOKEN_NOT_FOUND`) [5](#0-4) 
- The ETH connector account key is missing from storage (`ERR_KEY_NOT_FOUND`) [6](#0-5) 
- The precompile is in the `paused_precompiles` set [7](#0-6) 
- The input flag or recipient is invalid [8](#0-7) 

When the EVM executes a `call()` opcode and the callee reverts/errors, the return value is 0. Because the Solidity assembly block captures `res` but never acts on it, the parent call frame does **not** revert. The `_burn` that already executed is committed to state.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

A user calling `withdrawToNear` or `withdrawToEthereum` on an ERC-20 token whose NEP-141 mapping is absent, whose connector account is unset, or whose precompile is paused will have their tokens permanently burned with zero recourse. The ERC-20 supply decreases, the NEAR-side transfer never occurs, and there is no refund path in the Solidity contract. The `error_refund` feature in the Rust precompile layer is irrelevant here because the precompile never even executes successfully to schedule a callback.

---

### Likelihood Explanation

**High.** Any unprivileged EVM user holding tokens in a deployed `EvmErc20` or `EvmErc20V2` contract can trigger this by calling `withdrawToNear` or `withdrawToEthereum`. The failure conditions (unregistered NEP-141 mapping, missing connector account, paused precompile) are all realistic operational states. No special privileges or key compromise are required — only a standard ERC-20 token withdrawal call.

---

### Recommendation

Replace the unchecked assembly `call()` with a checked low-level call or a typed interface call, and revert if the precompile returns failure. The burn should either occur after a successful precompile call, or the entire transaction should revert on precompile failure:

```solidity
// Replace assembly block with:
(bool success, ) = address(0xe9217bc70b7ed1f598ddd3199e80b093fa71124f)
    .call{gas: gasleft()}(input[32:32 + input_size]);
require(success, "ExitToNear precompile call failed");
```

This directly mirrors the recommendation in the referenced report: eliminate assembly and use proper Solidity constructs that surface errors, ensuring the burn is only committed when the withdrawal is guaranteed to proceed.

---

### Proof of Concept

1. Deploy `EvmErc20` for a token whose NEP-141 mapping has **not** been registered in the Aurora engine storage.
2. Mint tokens to `attacker` via `mint(attacker, 1000)`.
3. Call `withdrawToNear(recipientBytes, 1000)` from `attacker`.
4. The `_burn` executes, reducing `attacker`'s balance to 0.
5. The assembly `call()` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` fails (precompile returns `ERR_TARGET_TOKEN_NOT_FOUND`), returning `res = 0`.
6. No `require(res != 0)` check exists; the function returns normally.
7. `attacker` has 0 tokens. No NEAR-side transfer occurred. Tokens are permanently destroyed. [1](#0-0) [5](#0-4)

    fn try_from(input: &'a [u8]) -> Result<Self, Self::Error> {
        // The first byte of the input is a flag, selecting the behavior to be triggered:
        // 0x00 -> Eth(base) token withdrawal
        // 0x01 -> ERC-20 token withdrawal
        let flag = input
            .first()
            .copied()
            .ok_or_else(|| ExitError::Other(Cow::from("ERR_MISSING_FLAG")))?;

        #[cfg(feature = "error_refund")]
        let (refund_address, input) = parse_input(input)?;
        #[cfg(not(feature = "error_refund"))]
        let input = parse_input(input)?;

        match flag {
            0x0 => {
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(input)?;

                Ok(Self::BaseToken(BaseTokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    message,
                }))
            }
            0x1 => {
                let amount = parse_amount(&input[..32])?;
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(&input[32..])?;

                Ok(Self::Erc20TokenParams(Erc20TokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    amount,
                    message,
                }))
            }
            _ => Err(ExitError::Other(Cow::from("ERR_INVALID_FLAG"))),
        }
    }
