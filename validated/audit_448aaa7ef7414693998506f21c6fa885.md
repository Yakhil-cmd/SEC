### Title
Silent Allowance Capping and Missing Approval Event in `approve()` - (`contracts/src/CW20ERC20Pointer.sol`)

---

### Summary

`CW20ERC20Pointer.approve()` silently caps any `amount > type(uint128).max` to `type(uint128).max`, then performs a no-op (no state change, no `Approval` event) when the capped value equals the current allowance. This is the Sei analog of the reported H-02 class: a caller supplying `type(uint256).max` receives `true` but may get no state change and no event, violating ERC-20 spec and breaking integrations that rely on infinite-approval patterns.

---

### Finding Description

In `CW20ERC20Pointer.sol`, the `approve` function:

```solidity
function approve(address spender, uint256 amount) public override returns (bool) {
    // if amount is larger uint128 then set amount to uint128 max
    if (amount > type(uint128).max) {
        amount = type(uint128).max;          // ← silent cap, no revert, no event
    }
    uint256 currentAllowance = allowance(msg.sender, spender);
    if (currentAllowance > amount) {
        // decrease_allowance via CW20
    } else if (currentAllowance < amount) {
        // increase_allowance via CW20
    }
    // ← if currentAllowance == amount: nothing happens, no Approval event
    return true;
}
``` [1](#0-0) 

Two compounding defects:

1. **Silent truncation**: Any `amount > type(uint128).max` (including the canonical `type(uint256).max` infinite-approval sentinel) is silently replaced with `type(uint128).max`. The function returns `true` with no indication the requested value was not honored.

2. **No-op on equality**: When `currentAllowance == amount` after the cap (e.g., the allowance is already `type(uint128).max` and the caller re-approves with `type(uint256).max`), neither CW20 branch executes. No `increase_allowance` or `decrease_allowance` message is sent, so no synthetic `Approval` log is generated. ERC-20 (EIP-20) mandates an `Approval` event on every successful `approve` call.

The `NativeSeiTokensERC20.sol` contract does not override `approve`, so it inherits the standard OpenZeppelin implementation and is not affected. [2](#0-1) 

---

### Impact Explanation

- A DEX or vault that calls `approve(router, type(uint256).max)` to grant infinite allowance will receive `true` but the on-chain allowance is `type(uint128).max`. For tokens with high supply (e.g., 18-decimal meme tokens with billions of units), `type(uint128).max` ≈ 3.4 × 10³⁸ is still enormous, so practical fund loss is unlikely in most cases.
- The more concrete impact is the **missing `Approval` event** when `currentAllowance` is already `type(uint128).max`. Off-chain indexers, wallets, and approval-tracking tools that rely on `Approval` events will show a stale or incorrect allowance state, potentially misleading users into believing a re-approval succeeded when it did not.
- Protocols that check `allowance() == type(uint256).max` as a sentinel for "infinite approval" will always see `type(uint128).max` instead, causing them to incorrectly treat the allowance as finite and attempt unnecessary re-approvals — which themselves silently no-op.

**Severity: Low** (ERC-20 spec violation and integration breakage; no direct path to significant fund loss given the large capped value).

---

### Likelihood Explanation

The `type(uint256).max` infinite-approval pattern is the dominant pattern used by DEXes (Uniswap, Curve, etc.) and ERC-20 helper libraries. Any EVM-side integration with a CW20-backed token via this pointer will trigger the silent cap on first approval. The no-op + missing event path is triggered on every subsequent re-approval attempt, which is a routine operation.

---

### Recommendation

1. **Remove the silent cap.** If CW20 cannot represent `uint256` amounts, revert explicitly rather than silently truncating:
   ```solidity
   require(amount <= type(uint128).max, "CW20: amount exceeds uint128 max");
   ```
2. **Always emit `Approval`.** Emit the event unconditionally after any `approve` call, regardless of whether the underlying CW20 state changed:
   ```solidity
   emit Approval(msg.sender, spender, amount);
   ``` [1](#0-0) 

---

### Proof of Concept

1. Deploy a CW20 token and its `CW20ERC20Pointer`.
2. Call `pointer.approve(dex, type(uint256).max)` → allowance becomes `type(uint128).max`, not `type(uint256).max`. Function returns `true`.
3. Call `pointer.approve(dex, type(uint256).max)` again (e.g., to refresh) → `currentAllowance == type(uint128).max == capped amount`, so neither branch executes. No CW20 message sent, no `Approval` event emitted. Function returns `true`.
4. Off-chain: query `eth_getLogs` for `Approval` events — the second call produces zero logs, violating ERC-20 spec.
5. A protocol checking `allowance(owner, dex) == type(uint256).max` will always get `false`, causing it to loop on unnecessary re-approvals that silently no-op. [3](#0-2)

pragma solidity ^0.8.0;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import {IBank} from "./precompiles/IBank.sol";

contract NativeSeiTokensERC20 is ERC20 {

    address constant BANK_PRECOMPILE_ADDRESS = 0x0000000000000000000000000000000000001001;

    string public denom;
    string public nname;
    string public ssymbol;
    uint8 public ddecimals;
    IBank public BankPrecompile;

    constructor(string memory denom_, string memory name_, string memory symbol_, uint8 decimals_) ERC20("", "") {
        BankPrecompile = IBank(BANK_PRECOMPILE_ADDRESS);
        denom = denom_;
        nname = name_;
        ssymbol = symbol_;
        ddecimals = decimals_;
    }

    function name() public view override returns (string memory) {
        return nname;
    }

    function symbol() public view override returns (string memory) {
        return ssymbol;
    }

    function balanceOf(address account) public view override returns (uint256) {
        return BankPrecompile.balance(account, denom);
    }

    function decimals() public view override returns (uint8) {
        return ddecimals;
    }

    function totalSupply() public view override returns (uint256) {
        return BankPrecompile.supply(denom);
    }

    function _update(address from, address to, uint256 value) internal override {
        bool success = BankPrecompile.send(from, to, denom, value);
        require(success, "NativeSeiTokensERC20: transfer failed");
        emit Transfer(from, to, value);
    }
}
