Audit Report

## Title
Hot Keys Not Reset When Spawning a Neuron with a New Controller - (File: `rs/nns/governance/src/governance.rs`)

## Summary
The `spawn_neuron` function unconditionally copies the parent neuron's `hot_keys` to the child neuron, even when `spawn.new_controller` designates a completely different controller. Because hot keys grant voting and follow authority via `is_hotkey_or_controller`, principals authorized by the original owner retain governance power over a neuron they never owned, without the new controller's knowledge or consent.

## Finding Description
At [1](#0-0)  the child controller is resolved from `spawn.new_controller` and can be any self-authenticating principal distinct from the parent's controller.

At [2](#0-1)  the child neuron is built with `child_controller` as its controller but `.with_hot_keys(parent_neuron.hot_keys.clone())` — no conditional reset when the controller changes.

The authorization check in `register_vote` at [3](#0-2)  delegates to `is_authorized_to_vote`, which resolves to `is_hotkey_or_controller` at [4](#0-3)  — returning `true` for any inherited hot key principal. There is no guard anywhere in the vote path that checks whether the hot key was authorized by the current controller rather than a prior one. Once the spawned neuron finishes dissolving and accumulates a ballot on a new proposal, any inherited hot key holder can cast votes using the new controller's neuron voting power.

## Impact Explanation
This is unauthorized access to a governance neuron: a third party (the original hot key holder) can vote on NNS proposals using a neuron they do not own and whose new controller never authorized them. This matches the **High** bounty impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The new controller's voting power is silently hijackable, which is a concrete governance integrity harm on the NNS.

## Likelihood Explanation
The `manage_neuron` → `Spawn` endpoint is callable by any neuron controller with sufficient maturity — no privileged role is required. The attacker only needs to: (1) hold a neuron with maturity, (2) register an alternate principal as a hot key on that neuron, and (3) spawn a child neuron to a target `new_controller`. This is a realistic scenario in neuron delegation or maturity-sale arrangements. The attack is repeatable and leaves no on-chain notification to the new controller.

## Recommendation
In `spawn_neuron`, replace the unconditional hot key copy with a conditional reset:

```rust
.with_hot_keys(if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
})
```

This ensures hot keys are only inherited when the spawned neuron stays under the same controller, mirroring the principle of least privilege for governance authorization.

## Proof of Concept
1. Alice controls neuron N with maturity. Alice calls `manage_neuron` → `AddHotKey { new_hot_key: BOB }` on N.
2. Alice calls `manage_neuron` → `Spawn { new_controller: Some(CAROL), percentage_to_spawn: 100, nonce: None }`.
3. Governance executes `spawn_neuron`: `child_controller = CAROL`, but `.with_hot_keys([BOB])` is set on the child neuron C. [2](#0-1) 
4. After the neuron finishes spawning and a new NNS proposal P is submitted (creating a ballot for C), BOB calls `manage_neuron` → `RegisterVote { neuron_id: C, proposal: P, vote: Yes }`.
5. Governance checks `is_authorized_to_vote(BOB)` → `is_hotkey_or_controller(BOB)` → `hot_keys.contains(BOB)` → `true`. [4](#0-3)  BOB's vote is cast using CAROL's neuron. CAROL never authorized BOB.

A deterministic integration test can be written using PocketIC: create a neuron, add a hot key, spawn to a new controller, submit a proposal, and assert that the hot key principal can successfully call `RegisterVote` on the child neuron.
