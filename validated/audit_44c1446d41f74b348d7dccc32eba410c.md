Audit Report

## Title
Unbounded `ingress_expiries` Allocation via Attacker-Controlled `ingress_start`/`ingress_end` Causes OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `/construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` values from the JSON request body with no range validation. These values drive an unbounded `while` loop that pushes one `u64` per 120-second interval into a heap-allocated vector. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes approximately 153 billion iterations allocating ~1.23 TB of heap, OOM-killing the Rosetta process. No authentication is required and a single HTTP request suffices.

## Finding Description
**Interval computation:** `interval` is set to `MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120,000,000,000 ns`. [1](#0-0) [2](#0-1) 

**Unconstrained user input:** `ingress_start` and `ingress_end` are plain `Option<u64>` fields deserialized from JSON with no constraints, and are mapped directly to `ic_types::time::Time` values. [3](#0-2) [4](#0-3) 

**Vulnerable loop:** The loop iterates from `ingress_start` to `ingress_end` with no upper-bound guard, pushing one `u64` per iteration. [5](#0-4) 

With `ingress_start=0` and `ingress_end=u64::MAX (18,446,744,073,709,551,615)`:
- Iterations: `⌊u64::MAX / 120_000_000_000⌋ ≈ 153,722,867,280`
- Heap allocation: `153,722,867,280 × 8 bytes ≈ 1.23 TB`

**No authentication on the endpoint:** The handler is registered with no authentication middleware; only a 4 MB JSON body size limit is applied, which is irrelevant since the malicious payload is two small integers. [6](#0-5) [7](#0-6) 

**Guards absent in ICP Rosetta, present in ICRC1 Rosetta:** The ICRC1 sibling implementation rejects requests where `ingress_start >= ingress_end`, confirming the developers are aware of the pattern. The ICP Rosetta version has no equivalent guard whatsoever. [8](#0-7) 

## Impact Explanation
The Rosetta process is a single, non-replicated off-chain service. OOM-killing it takes down the entire ICP Rosetta node until it is manually restarted. Any exchange or wallet relying on this node for ICP transfers, neuron management, or balance queries loses service. This matches the allowed impact: **High ($2,000–$10,000) — Significant Rosetta/infrastructure security impact with concrete user or protocol harm.**

## Likelihood Explanation
The endpoint is publicly reachable on any deployed ICP Rosetta node with no credentials required. The attack requires no prior state, no special knowledge beyond the Rosetta API spec, and the malicious values fit in a normal JSON body well under the 4 MB limit. A single request is sufficient; the attack is trivially repeatable after each restart.

## Recommendation
Before the `while` loop in `construction_payloads`, add the following guards mirroring the ICRC1 implementation:

1. Reject if `ingress_end <= ingress_start`.
2. Reject if `(ingress_end - ingress_start)` exceeds a compile-time maximum window (e.g., 24 hours = `86_400_000_000_000 ns`), capping `ingress_expiries.len()` to at most `86_400_000_000_000 / 120_000_000_000 = 720` entries.

Reference: `rs/rosetta-api/icrc1/src/construction_api/services.rs` lines 148–158. [8](#0-7) 

## Proof of Concept
Send the following single unauthenticated HTTP request to any deployed ICP Rosetta node:

```
POST /construction/payloads HTTP/1.1
Host: <rosetta-node>:8080
Content-Type: application/json

{
  "network_identifier": { "blockchain": "Internet Computer", "network": "00000000000000020101" },
  "operations": [
    {
      "operation_identifier": { "index": 0 },
      "type": "TRANSACTION",
      "account": { "address": "<valid_account>" },
      "amount": { "value": "-100000000", "currency": { "symbol": "ICP", "decimals": 8 } }
    },
    {
      "operation_identifier": { "index": 1 },
      "type": "TRANSACTION",
      "account": { "address": "<valid_dest>" },
      "amount": { "value": "100000000", "currency": { "symbol": "ICP", "decimals": 8 } }
    }
  ],
  "public_keys": [{ "hex_bytes": "<valid_pubkey_hex>", "curve_type": "edwards25519" }],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

The server enters the `while now < ingress_end` loop at line 101 of `construction_payloads.rs`, iterates ~153 billion times, exhausts process memory, and is OOM-killed. A local integration test can confirm this by asserting that the call returns an error (after adding the fix) or observing process termination (before the fix) with a small but still oversized window such as `ingress_end - ingress_start = 10^15 ns` (~11 days, ~8,333 iterations, ~66 KB — safe to test locally).
