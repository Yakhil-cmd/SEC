### Title
No Cancel Mechanism for Scheduled XCC Promises Causes Permanent Freezing of User NEAR — (`etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router contract exposes a `schedule` function that stores a `PromiseArgs` for deferred execution. When a user schedules a cross-contract call with attached NEAR (converted from wNEAR), those funds are transferred into the router sub-account. The router provides no `cancel_scheduled` function, and no path exists to return the frozen NEAR to the user if the promise is abandoned or fails on execution.

---

### Finding Description

The XCC precompile (`engine-precompiles/src/xcc.rs`) supports two execution modes. In the `Delayed` path, the promise is routed to the router's `schedule` method rather than `execute`: [1](#0-0) 

Before the `schedule` call is dispatched, the precompile computes `required_near = call.total_near()` and, if non-zero, performs an EVM-level `transferFrom` to move the user's wNEAR to the engine's implicit address: [2](#0-1) 

The engine then converts that wNEAR to real NEAR and transfers it to the router sub-account via `withdraw_wnear_to_router`, chaining the `schedule` call as a callback: [3](#0-2) 

The router stores the promise in `scheduled_promises` and increments a nonce: [4](#0-3) 

`execute_scheduled` (callable by anyone) removes the entry and creates the promise, spending the NEAR held in the router's account: [5](#0-4) 

**The router contract exposes no `cancel_scheduled` function.** The complete public interface is `initialize`, `get_version`, `execute`, `schedule`, `execute_scheduled`, `deploy_upgrade`, and `send_refund`. None of these allow a user or the engine to cancel a pending scheduled promise and recover the NEAR that was deposited into the router. [6](#0-5) 

The only refund path that exists, `send_refund`, is hardcoded to return exactly `REFUND_AMOUNT` (2 NEAR for storage staking) to the parent engine — it is not a general-purpose withdrawal: [7](#0-6) 

Additionally, if `execute_scheduled` is called and the underlying promise fails, NEAR attached to that promise is returned by the NEAR runtime to the router account (the predecessor), where it becomes permanently unrecoverable because no withdrawal function exists.

---

### Impact Explanation

**Permanent freezing of funds.** Any NEAR that a user converts from wNEAR and deposits into a router sub-account for a delayed XCC call is irrecoverable if:

1. The user wishes to cancel the scheduled promise (e.g., a deadline has passed, market conditions changed, or the target contract no longer exists).
2. `execute_scheduled` is called and the promise fails — the NEAR is returned to the router by the NEAR runtime but cannot be forwarded back to the user.

The NEAR is held in the router sub-account (`{evm_address}.aurora`) with no mechanism for the user or the engine to withdraw it.

---

### Likelihood Explanation

Any EVM user who calls the XCC precompile with `CrossContractCallArgs::Delayed` and a non-zero `attached_near` is exposed. This is a standard, documented usage path for deferred cross-contract calls. The scenario where a scheduled promise becomes stale or fails is realistic (network congestion, target contract upgrade, time-sensitive logic). No special privileges or attacker cooperation are required — the user's own funds are frozen by the absence of a cancel path.

---

### Recommendation

Add a `cancel_scheduled` function to the router contract, callable only by the parent (Aurora Engine), that:

1. Removes the entry from `scheduled_promises` by nonce.
2. Transfers the NEAR that was deposited for that promise back to the originating EVM address (or to the engine for re-crediting as wNEAR).

A corresponding engine-level entrypoint should allow the EVM user to trigger this cancellation from the EVM side, mirroring the `execute_scheduled` flow.

---

### Proof of Concept

1. Alice calls the XCC precompile with `CrossContractCallArgs::Delayed(call)` where `call.total_near() = 5 NEAR` (e.g., to fund a call to a DeFi contract).
2. The precompile transfers 5 wNEAR from Alice's EVM balance to the engine's implicit address.
3. The engine converts 5 wNEAR → 5 NEAR and transfers it to Alice's router sub-account (`{alice_addr}.aurora`).
4. The promise is stored at nonce `N` in `scheduled_promises`.
5. Before Alice calls `execute_scheduled`, the target DeFi contract is paused or the price condition expires.
6. Alice has no `cancel_scheduled` function to call. Her 5 NEAR is permanently frozen in the router sub-account.
7. Even if a third party calls `execute_scheduled(N)` and the promise fails, the 5 NEAR is returned to the router by the NEAR runtime and remains unrecoverable. [8](#0-7) [1](#0-0)

#[near]
impl Router {
    #[init(ignore_state)]
    #[must_use]
    pub fn initialize(wnear_account: AccountId, must_register: bool) -> Self {
        // The first time this function is called there is no state and the parent is set to be
        // the predecessor account id. In subsequent calls, only the original parent is allowed to
        // call this function. The idea is that the Create, Deploy and Initialize actions are done in a single
        // NEAR batch when a new router is deployed by the engine, so the caller will be the Aurora
        // engine instance that the user's address belongs to. If we update this contract and deploy
        // a new version of it, again the Deploy and Initialize actions will be done in a single batch
        // by the engine.
        let caller = env::predecessor_account_id();
        let mut parent = LazyOption::new(StorageKey::Parent, None);
        match parent.get() {
            None => {
                parent.set(&caller);
            }
            Some(parent) => {
                // Allow self-calls to `initialize` also.
                // This happens during the upgrade flow.
                if (caller != parent) && (caller != env::current_account_id()) {
                    env::panic_str(ERR_ILLEGAL_CALLER);
                }
            }
        }

        if must_register {
            env::promise_create(
                wnear_account.clone(),
                "storage_deposit",
                b"{}",
                WNEAR_REGISTER_AMOUNT,
                WNEAR_REGISTER_GAS,
            );
        }

        let mut version = LazyOption::new(StorageKey::Version, None);
        if version.get().unwrap_or_default() != CURRENT_VERSION {
            // Future migrations would go here

            version.set(&CURRENT_VERSION);
        }

        let nonce = LazyOption::new(StorageKey::Nonce, None);
        let scheduled_promises = LookupMap::new(StorageKey::Map);
        Self {
            parent,
            version,
            nonce,
            scheduled_promises,
            wnear_account,
        }
    }

    pub fn get_version(&self) -> u32 {
        self.version.get().unwrap_or_default()
    }

    /// This function can only be called by the parent account (i.e. Aurora engine) to ensure that
    /// no one can create calls on behalf of the user this router contract is deployed for.
    /// The engine only calls this function when the special precompile in the EVM for NEAR cross
    /// contract calls is used by the address associated with the sub-account this router contract
    /// is deployed at.
    pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Similar security considerations here as for `execute`.
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Allows the parent contract to trigger an update to the logic of this contract
    /// (by deploying a new contract to this account);
    #[payable]
    pub fn deploy_upgrade(&mut self, #[serializer(borsh)] args: DeployUpgradeParams) {
        self.assert_preconditions();

        let promise_id = env::promise_batch_create(&env::current_account_id());
        env::promise_batch_action_deploy_contract(promise_id, &args.code);
        env::promise_batch_action_function_call(
            promise_id,
            INITIALIZE,
            &args.initialize_args,
            NearToken::default(),
            INITIALIZE_GAS,
        );
        env::promise_return(promise_id);
    }

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
}
