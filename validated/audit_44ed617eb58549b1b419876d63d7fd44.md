Audit Report

## Title
Unbounded `inbound_connecting` JoinSet Enables Byzantine Subnet Peer to Exhaust Replica Memory via QUIC Handshake Flooding — (`rs/p2p/quic_transport/src/connection_manager.rs`)

## Summary
`handle_inbound_conn_attemp` unconditionally spawns a Tokio task into `inbound_connecting` — a plain, unbounded `JoinSet` — for every accepted Quinn `Incoming` object, before TLS authentication occurs. There is no per-peer cap, no global cap, and no `concurrent_connections` limit on the Quinn `ServerConfig`. A single Byzantine subnet peer holding a valid TLS certificate can flood the JoinSet by rapidly initiating QUIC handshakes without completing them, causing unbounded task accumulation, memory exhaustion, and Tokio scheduler degradation on the target replica.

## Finding Description
**Unbounded JoinSet declaration (line 119–121):** `inbound_connecting` is declared as `JoinSet<Result<ConnectionWithPeerId, ConnectionEstablishError>>` with no capacity bound, unlike `outbound_connecting` which is a `JoinMap<NodeId, …>` keyed by peer (at most one task per peer).

**Unconditional spawn path (lines 354–356 → 538–587):** Every time `self.endpoint.accept()` yields an `Incoming`, `run()` immediately calls `self.handle_inbound_conn_attemp(incoming)`. Inside that function, a future is unconditionally spawned into `self.inbound_connecting` at line 587 with no size check, no per-IP counter, and no call to `incoming.refuse()`.

**Authentication deferred inside the task (lines 541–577):** The TLS handshake (`incoming.await`) and certificate extraction happen entirely inside the spawned future. The task is already resident in the JoinSet before any identity check occurs.

**No `concurrent_connections` limit (lines 229–231):** The `quinn::ServerConfig` is constructed and configured with only a `transport_config`; `concurrent_connections(…)` is never called. A grep for `concurrent_connections` across the entire `quic_transport` crate returns zero matches, confirming Quinn's internal default (effectively unlimited) is in place.

**10-second task lifetime (line 82):** `CONNECT_TIMEOUT` is 10 seconds. An attacker sending N QUIC Initial packets per second accumulates up to 10×N live tasks simultaneously before any begin to expire.

**Exploit flow:** A Byzantine subnet peer sends a stream of QUIC Initial packets (each ~1200 bytes, containing a valid TLS ClientHello with their legitimate subnet certificate). Quinn yields one `Incoming` per packet to `endpoint.accept()`. Each `Incoming` is immediately spawned into `inbound_connecting`. The attacker never completes the handshake (drops the connection after the Initial), so `incoming.await` inside each task blocks until `CONNECT_TIMEOUT` fires. The JoinSet grows without bound. The only observable effect before exhaustion is the `connecting_connections` metric (lines 373–375), which reflects the count but enforces nothing.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The vulnerability is in application-level resource management (an unbounded `JoinSet`), not in network bandwidth. The fix is application-level. At scale, heap grows proportionally to live tasks and their QUIC buffers; the Tokio scheduler must poll thousands of pending tasks per tick, starving consensus, artifact pool, and state sync work, causing the target replica to drop out of consensus and stalling the subnet.

## Likelihood Explanation
The precondition is a single Byzantine subnet node with a valid TLS certificate — a standard Byzantine fault assumption for IC (threshold f < n/3). No majority corruption, no key leakage, and no external infrastructure attack is required. QUIC Initial packets are ~1200 bytes; a single datacenter-connected node can sustain thousands per second. The attack is repeatable and requires no victim interaction.

## Recommendation
1. **Cap `inbound_connecting` globally:** Before spawning in `handle_inbound_conn_attemp`, check `self.inbound_connecting.len()` against a constant (e.g., `2 × subnet_size`) and call `incoming.refuse()` if the cap is exceeded.
2. **Per-source-IP cap before TLS:** Use `Incoming::remote_address()` to extract the source IP before spawning and reject if a per-IP counter exceeds a threshold.
3. **Set `ServerConfig::concurrent_connections`:** Call `.concurrent_connections(MAX)` on the `quinn::ServerConfig` at lines 229–231 so Quinn drops excess `Incoming` objects before they reach the application layer.
4. **Replace `JoinSet` with a bounded `JoinMap`:** Key by source address to prevent duplicate in-flight tasks from the same peer, mirroring the deduplication already present in `outbound_connecting`.

## Proof of Concept
```rust
// Local integration test using two Quinn endpoints sharing a test TLS cert
// (valid subnet certificate for a Byzantine peer)
let byzantine_endpoint = make_test_endpoint_with_valid_subnet_cert();
let target_addr = replica_quic_addr();

let mut handles = vec![];
for _ in 0..10_000 {
    let ep = byzantine_endpoint.clone();
    handles.push(tokio::spawn(async move {
        // Initiate handshake but never drive it to completion
        let _connecting = ep.connect(target_addr, "server_name").unwrap();
        tokio::time::sleep(Duration::from_secs(15)).await;
        // Hold open past CONNECT_TIMEOUT to maximize JoinSet accumulation
    }));
}
join_all(handles).await;

// Scrape the replica's Prometheus metrics
let metric = scrape_metric(target_addr, "quic_transport_connecting_connections");
// Vulnerable: metric approaches 10_000
// Fixed:      metric <= MAX_INBOUND_CAP
assert!(metric <= MAX_INBOUND_CAP, "JoinSet unbounded: got {}", metric);
```
A deterministic version can be written as a `#[tokio::test]` using `PocketIC` or a local `quinn::Endpoint` pair, asserting that `inbound_connecting.len()` never exceeds the configured cap after N rapid connection initiations.
