Audit Report

## Title
Silent Integer Overflow in `add_expiration` Produces Epoch-0 Expiration in `FieldsDisplayMessage` Consent — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

In the `FieldsDisplayMessage` arm of `add_expiration`, a `u64` nanosecond timestamp is cast to `i64` without overflow checking. For any `expires_at > i64::MAX` (e.g., `u64::MAX`), the cast wraps to a negative value, integer division truncates it to `0`, and the re-cast to `u64` produces `TimestampSeconds { amount: 0 }` — epoch 0 (1970-01-01). A malicious dApp can craft an `icrc2_approve` with `expires_at = u64::MAX` (a near-permanent approval), request the consent message with `FieldsDisplay`, and the wallet will display "Approval expiration: 1970-01-01", deceiving the user into believing the approval is already expired while the on-chain approval is effectively permanent.

## Finding Description

In `add_expiration` at line 265 of `responses.rs`, the `FieldsDisplayMessage` branch performs an unchecked `u64 → i64` cast:

```rust
let seconds = (expires_at as i64) / 10_i64.pow(9);
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds as u64 },
))
``` [1](#0-0) 

For `expires_at = u64::MAX = 18446744073709551615`:

| Step | Expression | Result |
|---|---|---|
| 1 | `u64::MAX as i64` | `-1` (two's-complement wrapping) |
| 2 | `(-1_i64) / 1_000_000_000_i64` | `0` (Rust truncates toward zero) |
| 3 | `0_i64 as u64` | `0` |
| Output | `TimestampSeconds { amount: 0 }` | epoch 0 = 1970-01-01 |

The `GenericDisplayMessage` branch at lines 232–243 performs the same `ts as i64` cast, but then calls `time::OffsetDateTime::from_unix_timestamp(seconds)` and `replace_nanosecond(nanos)`. For `u64::MAX`, the remainder `(-1_i64 % 10^9) = -1`, which cast to `u32` wraps to `4294967295`, causing `replace_nanosecond` to return the observable error string `"Invalid nanosecond: 4294967295"`. The `FieldsDisplayMessage` branch has no such guard and silently emits a plausible-looking but wrong timestamp. [2](#0-1) 

The full call chain: `build_icrc21_consent_info` → `ConsentMessageBuilder::build` → `add_expiration`. The `expires_at` field flows directly from the decoded `ApproveArgs` into `with_expires_at` with no range validation before `add_expiration` is called. [3](#0-2) [4](#0-3) 

## Impact Explanation

ICRC-21 consent messages are the canonical, canister-generated descriptions that wallets display to users before signing. A malicious dApp crafts an `icrc2_approve` with `expires_at = u64::MAX` (valid per the Candid decoder; the ICRC-2 ledger accepts it as a far-future timestamp ~year 2554), requests the consent message with `device_spec = FieldsDisplay`, and receives `TimestampSeconds { amount: 0 }`. A wallet rendering this field displays "Approval expiration: 1970-01-01", leading the user to believe the approval is already expired or trivially short-lived. The user approves; the on-chain allowance is effectively permanent. The attacker can subsequently call `icrc2_transfer_from` to drain the user's funds. This is a concrete ledger security impact with direct user harm — the security mechanism ICRC-21 is designed to provide (informed user consent) is silently subverted. This matches the **Medium** bounty impact tier: a meaningful security impact requiring a malicious dApp and user interaction, but no special privileges or technical barriers beyond crafting a single ingress message.

## Likelihood Explanation

Any unprivileged caller can trigger this with a single crafted ingress message. No special privileges, node access, or governance majority are required. The only constraint is that the victim's wallet must render `FieldsDisplay` consent messages. Current real-world timestamps (~1.7 × 10¹⁸ ns) are safely below `i64::MAX` (~9.2 × 10¹⁸ ns), so this does not trigger accidentally — it requires deliberate use of an out-of-range value. The attack is repeatable and deterministic.

## Recommendation

Replace the unchecked cast with `i64::try_from` and return a fallback display string for out-of-range values, consistent with the `GenericDisplayMessage` branch:

```rust
Some(expires_at) => {
    match i64::try_from(expires_at) {
        Ok(ts_i64) => {
            let seconds = ts_i64 / 1_000_000_000_i64;
            fields_display.fields.push((
                "Approval expiration".to_string(),
                Value::TimestampSeconds { amount: seconds as u64 },
            ));
        }
        Err(_) => fields_display.fields.push((
            "Approval expiration".to_string(),
            Value::Text { content: format!("Invalid timestamp: {expires_at}") },
        )),
    }
}
```

## Proof of Concept

```rust
// Unit test in packages/icrc-ledger-types/src/icrc21/responses.rs
#[test]
fn test_add_expiration_overflow_fields_display() {
    let mut msg = ConsentMessage::FieldsDisplayMessage(Default::default());
    msg.add_expiration(Some(u64::MAX), None);
    if let ConsentMessage::FieldsDisplayMessage(fd) = &msg {
        let (_, val) = fd.fields.iter()
            .find(|(k, _)| k == "Approval expiration")
            .unwrap();
        // BUG: amount is 0 (epoch 0), not a large future timestamp
        assert!(
            !matches!(val, Value::TimestampSeconds { amount } if *amount == 0),
            "Silent overflow: u64::MAX produces epoch-0 expiration"
        );
    }
}
```

This test can be run with `cargo test` in `packages/icrc-ledger-types` and will fail on the current code, confirming the bug. [5](#0-4)
