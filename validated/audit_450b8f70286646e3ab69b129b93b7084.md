### Title
Prover-Controlled Block Count in Multiblock Batch Allows Skipping Block Execution - (`proof_running_system/src/system/bootloader.rs`)

---

### Summary

In the `multiblock-batch` proving path, the number of blocks to process (`count`) is read directly from the prover-controlled CSR non-determinism input with no lower-bound enforcement. A malicious prover can supply `count = 0` (or any value less than the true batch size), causing the RISC-V program to skip all (or some) block executions while still producing a structurally valid, non-zero public input. This is a direct analog to the chip-ordering underconstraint: just as the Pico verifier used a prover-supplied chip list to decide which chips to open, ZKsync OS uses a prover-supplied count to decide how many blocks to execute.

---

### Finding Description

In `proof_running_system/src/system/bootloader.rs`, the `multiblock-batch` variant of `run_proving_inner` reads the block count from the CSR oracle at line 217:

```rust
I::csr_write_impl(0xdeadbeef);
I::csr_write_impl(0);
let count = I::csr_read_impl();          // ← entirely prover-supplied
let mut batch_data = ZKBatchDataKeeper::new();
for _ in 0..count {                       // ← loop bound is prover-controlled
    oracle = ProvingBootloader::<O, L>::run_prepared::<...>(
        oracle, &mut batch_data, ...
    ).expect("Tried to prove a failing batch");
    ...
}
let public_input = batch_data
    .into_public_input(L::default(), &mut oracle)
    .hash();
``` [1](#0-0) 

The CSR is the sole source of non-determinism for the RISC-V guest program; it is populated by the prover before the proof run begins. There is **no assertion** that `count >= 1`, no check that `count` equals the number of blocks committed to in the batch header, and no check that `batch_data` is non-empty before `into_public_input` is called.

If `count = 0`:
- The `for` loop body never executes.
- `batch_data` remains in its freshly-constructed, empty state.
- `into_public_input` is called on that empty keeper and hashes it, producing a non-zero 32-byte value.
- The RISC-V program terminates without a panic, so the Airbender prover generates a valid proof for this execution trace.
- The resulting public input is the hash of an empty batch — not the hash of the actual state transition.

The non-multiblock path (`#[cfg(not(feature = "multiblock-batch"))]`) does not have this issue because it calls `run_prepared` unconditionally. [2](#0-1) 

The oracle security model explicitly states that all oracle responses are untrusted and must be validated by the caller: [3](#0-2) [4](#0-3) 

The `count` value is never validated against any external commitment, violating this requirement.

---

### Impact Explanation

A malicious prover can produce a cryptographically valid ZK proof whose public input is the hash of an empty (or truncated) batch rather than the hash of the true state transition. If the L1 verifier or any downstream consumer of the public input does not independently enforce that the proven batch covers the expected set of blocks, the attacker can:

- Submit a proof that claims to have executed a batch of N blocks while actually executing 0 (or k < N) blocks.
- The resulting state root embedded in the public input will not reflect the actual transactions, enabling invalid state transitions (e.g., forged balances, skipped withdrawals) to be accepted on-chain.

This matches the target impact class: **invalid proofs of execution accepted by the verifier**.

---

### Likelihood Explanation

The `multiblock-batch` feature is a production code path compiled into the `multiblock_batch.bin` RISC-V binary (referenced in `tests/instances/multiblock_batch/src/lib.rs`). The prover is the entity that populates the CSR witness stream before the proof run; in ZKsync's architecture the prover is an external, potentially untrusted party whose correctness is supposed to be enforced entirely by the ZK proof constraints. Because the constraint is missing, any prover operating the multiblock batch path can trivially set `count = 0` with no additional effort. [5](#0-4) 

---

### Recommendation

1. **Assert a minimum count**: Add `assert!(count >= 1, "batch must contain at least one block");` immediately after reading `count` from the CSR.
2. **Commit the count in the public input**: Include `count` (or a hash that commits to it) in the data fed into `into_public_input`, so the L1 verifier can check it against the expected batch size.
3. **Validate against a pre-committed batch descriptor**: Before the loop, read a batch descriptor from the oracle that specifies the expected number of blocks, verify it against a known commitment (e.g., the L1 batch hash), and assert `count == descriptor.num_blocks`.

---

### Proof of Concept

1. Prover constructs a CSR witness stream for a multiblock batch of N real blocks.
2. Instead of writing `N` as the count, the prover writes `0`.
3. The RISC-V program executes `run_proving_inner` with `count = 0`; the `for` loop body is skipped entirely.
4. `batch_data.into_public_input(...).hash()` returns the hash of an empty `ZKBatchDataKeeper` — a deterministic, non-zero 32-byte value.
5. Airbender generates a valid proof for this execution trace (the program ran to completion without panicking).
6. The proof and its public input (empty-batch hash) are submitted to the L1 verifier. If the L1 verifier does not independently enforce that the public input encodes the correct N-block state transition, the proof is accepted, and the on-chain state root is updated to an attacker-chosen value. [6](#0-5)

#[cfg(feature = "multiblock-batch")]
pub fn run_proving_inner<
    O: IOOracle,
    I: NonDeterminismCSRSourceImplementation,
    L: Logger + Default,
>(
    mut oracle: O,
) -> [u32; 8] {
    logger_log!(L::default(), "IO implementer init is complete");

    // simulating query, just in case
    I::csr_write_impl(0xdeadbeef);
    I::csr_write_impl(0);
    let count = I::csr_read_impl();
    let mut batch_data = basic_bootloader::bootloader::block_flow::ZKBatchDataKeeper::new();
    for _ in 0..count {
        oracle = ProvingBootloader::<O, L>::run_prepared::<BasicBootloaderProvingExecutionConfig>(
            oracle,
            &mut batch_data,
            &mut NopResultKeeper::default(),
            &mut NopTracer::default(),
            &mut NopTxValidator,
        )
        .expect("Tried to prove a failing batch");
        // we do this query for consistency with block based input generation(there is empty iterator as response to this query)
        // but during proving this request shouldn't have the effect with "u32 array based" oracle
        #[allow(unused_must_use)]
        oracle
            .raw_query_with_empty_input(DISCONNECT_ORACLE_QUERY_ID)
            .expect("must disconnect an oracle before performing arbitrary CSR access");
    }

    let public_input = zk_ee::utils::Bytes32::from_array(
        batch_data
            .into_public_input(L::default(), &mut oracle)
            .hash(),
    );

    unsafe { core::mem::transmute(public_input) }
}
