Audit Report

## Title
Unbounded Neuron Scan in SNS Governance `maybe_finalize_disburse_maturity` Causes Instruction-Limit DoS and Permanent Lock-Out - (File: `rs/sns/governance/src/governance.rs`)

## Summary
`maybe_finalize_disburse_maturity` performs an unbounded O(N) scan over the entire neuron map in a single pre-`await` instruction-counting segment on every heartbeat. The `is_finalizing_disburse_maturity` guard flag is set to `Some(true)` before the scan and is only cleared at the very end of the function; if the function traps mid-execution, the flag is never reset, permanently preventing all future disbursement finalizations. The NNS governance team has already acknowledged and patched the identical class of bug in analogous tasks, but the SNS governance counterpart remains unguarded.

## Finding Description

**Unbounded pre-`await` scan:** At line 4935, `is_finalizing_disburse_maturity` is set to `Some(true)`. Immediately after, lines 4938–4975 iterate `self.proto.neurons.values()` — the full `BTreeMap<String, Neuron>` — with no upper bound, collecting every neuron with a ready disbursement into a `Vec` before any `await` suspension point. The entire scan runs in a single instruction-counting segment.

**Unbounded disbursement loop:** Lines 4976–5081 then iterate over every collected disbursement, issuing one `ledger.transfer_funds(...).await` per neuron. While each inter-`await` segment is individually bounded, the total number of iterations is uncapped, allowing the heartbeat to occupy many consecutive rounds and starve other periodic tasks.

**Flag cleared only on success:** The guard is reset to `None` only at line 5082, after the loop completes. If the function traps at any point — including an instruction-limit trap during the pre-`await` scan — the flag remains `Some(true)` permanently. `can_finalize_disburse_maturity()` will return `false` on every subsequent heartbeat, freezing all pending disbursements indefinitely without any error surfaced to users.

**Contrast with NNS governance fix:** The NNS governance equivalent (`try_finalize_maturity_disbursement` in `rs/nns/governance/src/governance/disburse_maturity.rs`) processes exactly one neuron per invocation and uses a timer-based retry loop. The NNS CHANGELOG (lines 657–668) explicitly records: *"Unstaking maturity task has a limit of 100 neurons per message, which prevents it from exceeding instruction limit."* No equivalent guard exists in the SNS path.

**Exploit path:**
1. Attacker (or organic growth) brings the SNS neuron count to a level where the pre-`await` scan exhausts the 40B-instruction heartbeat budget.
2. The heartbeat traps; `is_finalizing_disburse_maturity` remains `Some(true)`.
3. All subsequent heartbeat invocations of `run_periodic_tasks` call `can_finalize_disburse_maturity()`, which returns `false`, and skip the disbursement task entirely.
4. All users with pending `disburse_maturity_in_progress` entries are permanently frozen until a canister upgrade.

## Impact Explanation

This is an application/platform-level DoS on a core SNS governance function. Users who have initiated maturity disbursements will never receive their tokens; the frozen state is invisible to them and persists across heartbeats until a governance upgrade is executed. This matches the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact"* — High severity ($2,000–$10,000).

## Likelihood Explanation

The `NervousSystemParameters.max_number_of_neurons` cap limits total neuron count, but default ceilings for deployed SNS instances can reach tens of thousands to hundreds of thousands. At those scales the pre-`await` scan consumes a meaningful fraction of the instruction budget. An attacker with access to the minimum SNS token stake can accelerate neuron creation toward the cap. Even without adversarial intent, large SNS token distributions with many small holders each creating a neuron can organically approach the threshold. The secondary lock-out (permanent flag) is triggered by any trap, not only an instruction-limit trap, making it reachable through other unexpected failure modes as well.

## Recommendation

1. **Batch the scan**: process at most N neurons per heartbeat (analogous to the NNS limit of 100 neurons per message). Persist a cursor (last-processed neuron ID) in canister state so successive heartbeats make forward progress.
2. **Clear the lock on trap**: use a RAII guard or `scopeguard` to ensure `is_finalizing_disburse_maturity` is reset to `None` even when the function panics or traps, preventing the secondary permanent lock-out.
3. **Cap the disbursement loop**: limit the number of ledger calls issued per heartbeat invocation to bound total round-time impact.
