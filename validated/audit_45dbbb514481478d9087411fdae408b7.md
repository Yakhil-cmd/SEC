### Title
Permanent Chain Halt via Missing Recovery Mechanism When `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Reverts During L1 Deposit Processing — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The bootloader unconditionally treats a revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a fatal `internal_error!` that terminates block processing entirely. Because L1 priority transactions cannot be invalidated or skipped — doing so would itself halt the chain via the priority queue — there is no recovery path once this fatal path is triggered. Any L1 deposit transaction that causes the `L2AssetTracker` call to revert permanently halts the chain with no operator escape hatch.

---

### Finding Description

**Root cause — `notify_l2_asset_tracker` in `process_l1_transaction.rs`:**

`notify_l2_asset_tracker` is called up to three times per L1 deposit transaction: once for the value mint (inside the execution frame), once for the operator fee, and once for the refund. In all three cases, if the EVM call to `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` returns a failure result, the function immediately returns a fatal `internal_error!`:

```rust
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    ).into());
}
``` [1](#0-0) 

This error propagates as `BootloaderSubsystemError`, which is converted to `TxError::Internal` and causes the ZK tx loop to `return Err(err)`, terminating the entire block:

```rust
Err(TxError::Internal(err)) => {
    system_log!(system, "Tx execution result: Internal error = {err:?}\n",);
    system.finish_global_frame(None)?;
    return Err(err);
}
``` [2](#0-1) 

**Why there is no recovery:**

The codebase explicitly documents that L1 transactions cannot be invalidated:

> "L1 transactions cannot be invalidated (doing so would halt the chain due to the priority queue)." [3](#0-2) 

The design for L2 validation errors (e.g., gas limit below intrinsic) uses saturating arithmetic to keep processing. No analogous fallback exists for `L2AssetTracker` reverts. There is no flag, bypass, or operator-callable escape hatch to skip the `L2AssetTracker` notification and allow the block to seal.

**The three call sites that are all fatal:**

1. Value-mint notification (inside execution frame, user resources): [4](#0-3) 

2. Operator-fee notification (post-execution, `FORMAL_INFINITE` resources): [5](#0-4) 

3. Refund notification (post-execution, `FORMAL_INFINITE` resources): [6](#0-5) 

**What can cause `L2AssetTracker` to revert:**

`L2AssetTracker` (predeployed at `0x1000f`) is a real Solidity contract: [7](#0-6) 

Its `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` function (selector `0x03117c8c`) can revert if:

- `isAssetRegistered[assetId]` is false (the base token asset is not registered)
- `totalSuccessfulDepositsFromL1 += amount` overflows (Solidity 0.8+ checked arithmetic; an attacker who accumulates enough deposits across many L1 transactions can push this counter to overflow)
- Any internal invariant check in the contract fails after an upgrade or unexpected state change

The `l1_chain_id` passed to the call is read directly from `L2AssetTracker` storage slot 154: [8](#0-7) 

If this slot is zero (e.g., before proper initialization or after a storage corruption), the contract may revert on a chain-ID mismatch check.

---

### Impact Explanation

A single L1 deposit transaction that triggers a revert in `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` permanently halts the ZKsync OS chain. Block processing cannot continue, no further transactions (L1 or L2) can be processed, and there is no operator-callable recovery mechanism. All user funds locked in the L1 bridge become permanently inaccessible because the L2 chain cannot advance to process withdrawals. This is a complete, unrecoverable chain halt.

---

### Likelihood Explanation

The entry path — submitting an L1 deposit transaction — is fully attacker-controlled and requires no privileged access. The `to_mint` field is set by the depositor on L1. The `L2AssetTracker` is a real Solidity 0.8+ contract; its `totalSuccessfulDepositsFromL1` accumulator uses checked arithmetic and will revert on overflow. An attacker with sufficient capital can accumulate deposits across many transactions until the counter overflows. Alternatively, any bug introduced in a future upgrade to `L2AssetTracker` that causes `handleFinalizeBaseTokenBridgingOnL2` to revert would immediately trigger this halt on the next L1 deposit. The design comment itself acknowledges the dependency: "we are certain that L2AssetTracker is available after the upgrade" — but certainty about availability does not guarantee the call never reverts. [9](#0-8) 

---

### Recommendation

Apply the same resilience pattern already used for other L1 transaction validation errors: treat a revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a non-fatal condition rather than a chain-halting fatal error. Concretely:

1. In `notify_l2_asset_tracker`, when `failed == true`, emit a system log and return `Ok(())` (or a dedicated soft-error variant) instead of `Err(internal_error!(...))`. The block can still seal; the accounting discrepancy can be reconciled via an operator-triggered correction transaction.
2. Alternatively, add an operator-callable bypass flag (analogous to the `emergencyResolveMarket` pattern in the external report) that allows a block to be sealed even when the `L2AssetTracker` notification fails, so the priority queue does not permanently stall.
3. At minimum, document and test the overflow boundary of `totalSuccessfulDepositsFromL1` and add a guard in the bootloader that caps the amount passed to `notify_l2_asset_tracker` to prevent arithmetic overflow in the contract.

---

### Proof of Concept

**Precondition:** `L2AssetTracker` is deployed at `0x1000f` with `isAssetRegistered[BASE_TOKEN_ASSET_ID] = true` and `totalSuccessfulDepositsFromL1` near `U256::MAX`.

**Steps:**

1. Attacker submits an L1 priority transaction (type `0x7f`) with `to_mint = N` such that `totalSuccessfulDepositsFromL1 + N` overflows `uint256`.
2. The L1 contracts accept the deposit (they only validate that the depositor has locked sufficient ETH; they do not check the L2 accumulator).
3. The bootloader processes the L1 tx and calls `notify_l2_asset_tracker` with `amount = N`.
4. `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` executes `totalSuccessfulDepositsFromL1 += N`, which overflows and reverts (Solidity 0.8+ checked arithmetic).
5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
6. The ZK tx loop receives `TxError::Internal` and executes `return Err(err)`.
7. Block processing halts. The L1 transaction cannot be skipped. The chain is permanently halted.

**Expected outcome:** Chain halt with no recovery path. All subsequent L1 and L2 transactions are unprocessable.

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
```
