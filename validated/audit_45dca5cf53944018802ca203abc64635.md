### Title
Missing EIP-7610 Storage Collision Check in CREATE/CREATE2 — (`evm_interpreter/src/ee_trait_impl.rs`)

### Summary

The `before_executing_frame` function in ZKsync OS's EVM interpreter explicitly omits the EIP-7610 requirement that deployment must fail if the target address has non-empty storage. The check only tests `deployee_code_len != 0 || deployee_nonce != 0`, and the code itself contains a comment acknowledging the missing storage check.

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the CreateCollision guard reads:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
// We need to check this here (not when we actually deploy the code)
// because if this check fails the constructor shouldn't be executed.
if deployee_code_len != 0 || deployee_nonce != 0 {
    ...
    tracer.evm_tracer().on_call_error(&EvmError::CreateCollision);
    return Ok(false);
}
``` [1](#0-0) 

The condition `deployee_code_len != 0 || deployee_nonce != 0` is missing the third required predicate: `|| target_has_nonempty_storage`. EIP-7610 (active since the Paris hardfork) mandates that CREATE/CREATE2 must revert if the target address has any non-zero storage slot, even when nonce and code length are both zero.

This is further confirmed by the test index files, which explicitly disable all EIP-7610 storage collision tests: [2](#0-1) [3](#0-2) [4](#0-3) 

Additional disabled tests in the state test suite also reference this gap: [5](#0-4) [6](#0-5) 

### Impact Explanation

An unprivileged attacker can:

1. Pre-populate storage at a deterministic CREATE2 target address (e.g., deploy a contract that calls `SSTORE`, then self-destruct it, leaving dirty storage behind with nonce=0 and code_len=0).
2. Call `CREATE2` with the same salt/deployer/initcode to deploy a new contract at that address.
3. On mainnet EVM (post-EIP-7610), step 2 reverts. On ZKsync OS, it **succeeds**.

The newly deployed contract starts with pre-existing, attacker-controlled storage values. Any contract logic that assumes a freshly deployed contract has empty storage (e.g., proxy patterns, factory contracts, storage-based access control initialized in constructors) can be exploited. This is a concrete, demonstrable EVM deviation.

### Likelihood Explanation

The attack path is fully unprivileged and requires only standard EVM opcodes (`SSTORE`, `SELFDESTRUCT`/`SENDALL`, `CREATE2`). The deviation is deterministic and reproducible. The developers have explicitly acknowledged the gap in both a code comment and by disabling the relevant EIP-7610 test suite entries, confirming it is not a misread of the code.

### Recommendation

Implement the storage non-emptiness check as part of the CreateCollision guard. Before executing a constructor frame, query whether the target address has any non-zero storage slots. If so, treat it as a collision and return `Ok(false)` (burning all gas, as is done for the existing collision cases). This aligns with EIP-7610 semantics.

### Proof of Concept

State test setup:
- Pre-state: address `A` (the CREATE2 target) has `nonce=0`, `code=0x`, `storage={0x00: 0x01}`.
- Transaction: deployer contract calls `CREATE2` with salt and initcode that deterministically produces address `A`.
- Expected (mainnet EVM): CREATE2 returns 0 (failure), constructor not executed.
- Actual (ZKsync OS): CREATE2 succeeds, constructor executes, contract deployed at `A` with pre-existing storage slot `0x00 = 0x01`.

The disabled test `create2collisionStorageParis.json` (comment: *"We do not check storage for collision"*) and `test_init_collision_create_opcode.json` / `test_init_collision_create_tx.json` (comment: *"We do not check the storage"*) are exactly this proof of concept, already present in the test suite but suppressed.
