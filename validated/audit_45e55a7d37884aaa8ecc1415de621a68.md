Audit Report

## Title
Byzantine Peer Permanently Stalls State Sync via Persistent HTTP 429 Without Eviction ‚Äî (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
A Byzantine subnet peer that always responds with HTTP 429 is never removed from `active_downloads` because `DownloadChunkError::Overloaded` has no eviction path, no retry limit, and no backoff. When the Byzantine peer is the sole advertiser of the target state, the victim replica's `OngoingStateSync::run` loop spins indefinitely, preventing the replica from completing state sync and rejoining consensus.

## Finding Description
**Root cause ‚Äî no eviction on `Overloaded`.**
In `parse_chunk_handler_response` (`routes/chunk.rs`, line 129), HTTP 429 maps to `DownloadChunkError::Overloaded`. In `handle_downloaded_chunk_result` (`ongoing.rs`, lines 211‚Äì222), the `Overloaded | Timeout | Cancelled` arm only calls `self.chunks_to_download.download_failed(chunk_id)`. It never calls `self.active_downloads.remove(&peer_id)` and never decrements `self.allowed_downloads`. Contrast with `NoContent` (lines 194‚Äì199) and `RequestError` (lines 201‚Äì209), both of which remove the peer and decrement the budget.

**Infinite loop.**
The only self-termination condition in `OngoingStateSync::run` (besides external cancellation) is `self.active_downloads.is_empty()` (lines 173‚Äì176). Because `Overloaded` never removes the peer, this condition is never satisfied when the Byzantine peer is the only advertiser.

**Cycle repeats indefinitely.**
After each 429 response: (1) `active_downloads[peer]` is decremented to 0 via `saturating_sub` (lines 130‚Äì132) ‚Äî peer stays in the map; (2) `download_failed` re-queues the chunk; (3) `spawn_chunk_downloads` selects the peer again (weight = `max - 0 + 1`, highest possible); (4) a new download task is spawned for the same chunk against the same peer. There is no per-peer consecutive-failure counter, no exponential backoff, and no circuit breaker.

**No higher-level timeout.**
`StateSyncManager::run` in `lib.rs` (lines 156‚Äì164) only cleans up an ongoing sync when `ongoing.shutdown.completed()` (inner loop exited) or `cancel_if_running` returns true (triggered by a new advert for a different artifact). Neither condition is triggered by a Byzantine peer that keeps the inner loop alive.

**Test confirms the design intent.**
`test_cancel_if_running` (`ongoing.rs`, lines 429‚Äì461) uses a mock transport that always returns `StatusCode::TOO_MANY_REQUESTS` and only asserts that external cancellation works ‚Äî it does not assert peer eviction. This confirms the "transient, retry forever" design with no eviction path.

## Impact Explanation
A lagging replica performing state sync is stalled for as long as the Byzantine peer is the sole advertiser of the target state. The replica cannot complete state sync, cannot apply the certified state, and cannot rejoin consensus. This constitutes a targeted, sustained denial-of-service against a single replica's availability ‚Äî matching the High impact class: *"Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS."* The subnet's fault tolerance is reduced by one node for the duration of the attack, which is repeatable across every state sync attempt by the victim.

## Likelihood Explanation
Any node operator running a malicious replica can implement this by returning HTTP 429 for all chunk requests. No threshold corruption, key compromise, or privileged access is required. The attacker only needs to be the first peer to advertise the target state to the victim replica ‚Äî a condition that is naturally satisfied during the initial phase of state sync before other honest peers have broadcast their adverts (broadcast interval is 5 seconds, `ADVERT_BROADCAST_INTERVAL`). The attack is repeatable: every time the victim attempts state sync, the Byzantine peer can re-stall it by being the first advertiser.

## Recommendation
In the `Overloaded` arm of `handle_downloaded_chunk_result`, introduce a per-peer consecutive-overload counter stored alongside the active-download count in `active_downloads` (or a parallel `HashMap<NodeId, u32>`). After a configurable threshold (e.g., `N = PARALLEL_CHUNK_DOWNLOADS` consecutive `Overloaded` responses), treat the peer identically to `RequestError`: call `self.active_downloads.remove(&peer_id)` and decrement `self.allowed_downloads`. Alternatively, apply an exponential backoff that effectively starves the peer of download slots over time, or add a per-peer wall-clock deadline after which the peer is evicted if it has never served a successful chunk.

## Proof of Concept
Construct a state-machine unit test in `ongoing.rs`:
1. Create an `OngoingStateSync` with a single mock peer whose transport always returns `StatusCode::TOO_MANY_REQUESTS`.
2. Send `NODE_1` as the only peer via `ongoing.sender`.
3. After `N * PARALLEL_CHUNK_DOWNLOADS` download cycles (tracked via a counter in the mock), assert either: (a) `NODE_1` is absent from `active_downloads`, or (b) the sync has terminated without external cancellation.
Under the current code, neither assertion holds ‚Äî the loop runs indefinitely until `ongoing.shutdown.shutdown()` is called externally, exactly as demonstrated by the existing `test_cancel_if_running` test.
