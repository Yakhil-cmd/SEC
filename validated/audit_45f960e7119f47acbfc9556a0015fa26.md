Audit Report

## Title
Unbounded Reward-Round Loop in SNS Governance `distribute_rewards` Permanently Traps Periodic Tasks - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The `distribute_rewards` function iterates `new_rounds_count` times with no upper bound. When an SNS is initialized with `round_duration_seconds = 1` (the minimum valid value) and no prior reward event exists, `new_rounds_count` equals the current Unix timestamp (~1.7 billion). The resulting loop exhausts the IC per-message instruction limit on the first heartbeat, and because the trap rolls back state, `latest_reward_event` is never updated, causing every subsequent heartbeat to trap identically and permanently freezing all SNS governance periodic tasks.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 5808–5814, `reward_start_timestamp_seconds` is set to `0` via `unwrap_or_default()` when no reward event has been recorded:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();          // → 0 on first invocation
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
```

With `round_duration_seconds = 1`, `new_rounds_count ≈ now ≈ 1_700_000_000`. The only guard present (lines 5815–5820) checks `new_rounds_count == 0` and returns early — it does not cap the value. The loop at lines 5861–5872 then iterates unconditionally:

```rust
for i in 1..=new_rounds_count {
    ...
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
```

Each iteration performs multiple `Decimal` multiplications. A grep for any cap (`MAX_ROUNDS_PER`, `max_rounds`, `rounds_cap`, `.min(`) returns no matches in this function. The `round_duration_seconds_defects` validator in `rs/sns/governance/src/reward.rs` (lines 254–260) explicitly permits `1` as the minimum:

```rust
require_field_set_and_in_range(
    "round_duration_seconds",
    &self.round_duration_seconds,
    1..=*MAX_REWARD_ROUND_DURATION_SECONDS,
)
```

Because the trap rolls back all state changes, `latest_reward_event.end_timestamp_seconds` remains `None` on every subsequent heartbeat, and the canister re-enters the same fatal loop indefinitely.

## Impact Explanation
This matches the **High** bounty impact: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … SNS … security impact with concrete user or protocol harm."* Every periodic task sharing `run_periodic_tasks` — proposal settlement, maturity spawning, and reward distribution — is permanently blocked. No neuron holder ever receives voting rewards; the SNS governance canister's liveness is degraded for all token holders of the affected SNS.

## Likelihood Explanation
The SNS deployer controls the init payload and sets `round_duration_seconds = 1`, which passes all on-chain validation. No governance vote, privileged override, or post-deployment action is required. The condition triggers automatically on the first heartbeat after deployment. A malicious SNS launcher can exploit this to harm token holders who participate in the SNS swap, and a developer testing short reward cycles can trigger it inadvertently.

## Recommendation
Cap `new_rounds_count` to a safe maximum before entering the loop, e.g.:

```rust
let new_rounds_count = new_rounds_count.min(MAX_ROUNDS_PER_DISTRIBUTION);
```

and carry over remaining rounds to subsequent heartbeats. Alternatively, replace the per-round loop with a closed-form integral over the reward rate curve (as NNS governance does), eliminating the O(n) iteration entirely.

## Proof of Concept
```rust
let params = NervousSystemParameters {
    voting_rewards_parameters: Some(VotingRewardsParameters {
        round_duration_seconds: Some(1),   // minimum valid; passes validation
        reward_rate_transition_duration_seconds: Some(0),
        initial_reward_rate_basis_points: Some(100),
        final_reward_rate_basis_points: Some(100),
    }),
    ..NervousSystemParameters::with_default_values()
};
// After deployment:
//   latest_reward_event.end_timestamp_seconds = None → reward_start = 0
//   now ≈ 1_700_000_000
//   new_rounds_count = 1_700_000_000 / 1 = 1_700_000_000
//
// Loop `for i in 1..=1_700_000_000` exhausts the ~5B instruction limit.
// State rolls back; next heartbeat repeats identically. Canister is permanently stuck.
```

A deterministic integration test using PocketIC can set the mock clock to a realistic Unix timestamp, initialize the SNS with the above parameters, advance one heartbeat, and assert that `latest_reward_event` is never updated and that the canister traps on every `run_periodic_tasks` invocation.
