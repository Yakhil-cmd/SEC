### Title
`pause_contract` Blocks the `withdraw` Function and All EVM Exit Paths, Causing Temporary Freezing of User Funds - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The Aurora Engine's `pause_contract` mechanism sets `state.is_paused = true`, which causes the `withdraw` function (ETH connector withdrawal) and all EVM transaction submission paths (`submit`, `submit_with_args`, `call`) to fail via `require_running`. When the contract is legitimately paused for security reasons, users are completely unable to exit their funds from the protocol — both at the NEAR connector level and at the EVM precompile level — causing a temporary but complete freeze of all user funds.

---

### Finding Description

**Root cause — `withdraw` gated by `require_running`:**

In `engine/src/contract_methods/connector.rs`, the `withdraw` function is the primary NEAR-level mechanism for users to exit ETH from Aurora back to Ethereum. It unconditionally calls `require_running` before doing anything else:

```rust
pub fn withdraw<I: IO + Copy + PromiseHandler, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    require_running(&state::get_state(&io)?)?;   // ← blocks if paused
    env.assert_one_yocto()?;
    ...
}
```

`require_running` in `engine/src/contract_methods/mod.rs` returns `Err(ERR_PAUSED)` whenever `state.is_paused` is `true`:

```rust
pub fn require_running(state: &state::EngineState) -> Result<(), ContractError> {
    if state.is_paused {
        return Err(errors::ERR_PAUSED.into());
    }
    Ok(())
}
```

`pause_contract` in `engine/src/contract_methods/admin.rs` sets this flag:

```rust
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

**All EVM exit paths are also blocked:**

`submit`, `submit_with_args`, and `call` in `engine/src/contract_methods/evm_transactions.rs` all call `require_running` as their first check. This means that when the contract is paused, users also cannot submit EVM transactions that would invoke the `ExitToNear` or `ExitToEthereum` precompiles — the two EVM-level fund exit mechanisms.

**The only two pausable precompiles are both withdrawal mechanisms:**

`engine/src/pausables.rs` defines `PrecompileFlags` with exactly two flags:

```rust
pub struct PrecompileFlags: u32 {
    const EXIT_TO_NEAR     = 0b01;
    const EXIT_TO_ETHEREUM = 0b10;
}
```

There are no deposit-side precompiles in this bitmask. The `pause_precompiles` mechanism exclusively targets the exit (withdrawal) precompiles, compounding the design issue: even if `pause_contract` were fixed, `pause_precompiles` can independently block both EVM-level exit paths.

**Complete withdrawal path matrix when paused:**

| Withdrawal Path | Blocked by `pause_contract` | Blocked by `pause_precompiles` |
|---|---|---|
| NEAR-level `withdraw` (ETH connector) | Yes (`require_running`) | No |
| EVM `submit`/`call` → `ExitToNear` | Yes (`require_running`) | Yes (`ERR_PAUSED` from precompile set) |
| EVM `submit`/`call` → `ExitToEthereum` | Yes (`require_running`) | Yes (`ERR_PAUSED` from precompile set) |

---

### Impact Explanation

**High. Temporary freezing of funds.**

When `pause_contract` is invoked (for any legitimate security reason), every withdrawal path available to an unprivileged user is simultaneously blocked. Users holding ETH or ERC-20 tokens inside Aurora cannot exit to NEAR or Ethereum until the owner calls `resume_contract`. The freeze duration is bounded only by the owner's response time, but there is no mechanism that allows users to self-rescue their funds during the pause. This is a direct analog to the LpToken `whenNotPaused` issue: a security-oriented pause mechanism inadvertently applies to withdrawal operations that should remain available to users.

---

### Likelihood Explanation

**Medium.**

`pause_contract` is an owner-only emergency function that would be invoked in response to a real or perceived security incident. The scenario is realistic: a security event triggers a legitimate pause, and during the pause window (which could last hours or days), all users are unable to withdraw. The owner does not need to be malicious — the design flaw activates under normal, intended use of the pause feature.

---

### Recommendation

Separate the pause scope so that deposit-side operations (`ft_on_transfer`, `finish_deposit`, `deploy_erc20_token`) are blocked when paused, but withdrawal-side operations (`withdraw`, and EVM transactions whose sole purpose is to exit funds) remain available. Concretely:

- Remove the `require_running` guard from `withdraw` in `connector.rs`, or introduce a finer-grained pause flag (e.g., `PAUSE_DEPOSIT` vs. `PAUSE_WITHDRAW`) analogous to the `PausedMask` constants already defined in the test utilities (`PAUSE_DEPOSIT = 1 << 0`, `PAUSE_WITHDRAW = 1 << 1`).
- For `pause_precompiles`, consider whether `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` should be pausable at all, or whether only deposit-side precompiles should be pausable. If exit precompiles must remain pausable, ensure a time-bounded or governance-gated unpause path exists.

---

### Proof of Concept

1. User deposits ETH into Aurora via the ETH connector; funds are credited inside the EVM.
2. A security incident occurs; the Aurora owner calls `pause_contract`.
3. `state.is_paused` is set to `true` in engine state.
4. User calls `withdraw` on the Aurora engine contract to retrieve their ETH.
5. `withdraw` calls `require_running` → returns `Err(ERR_PAUSED)` → transaction reverts.
6. User attempts to submit an EVM transaction calling the `ExitToNear` precompile address.
7. `submit` calls `require_running` → returns `Err(ERR_PAUSED)` → transaction reverts.
8. All user funds are frozen until the owner calls `resume_contract`. No user-accessible escape hatch exists.
