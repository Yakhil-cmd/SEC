### Title
Incompatible Input Formats Between `EvmErc20` and `EvmErc20V2` Cause Silent Token Loss When Calling `ExitToNear` Precompile - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`, `engine-precompiles/src/native.rs`)

---

### Summary

`EvmErc20` and `EvmErc20V2` both implement the `IExit` interface and are deployed as ERC-20 mirror token contracts on Aurora. Their `withdrawToNear` functions call the same `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) using **structurally different input byte layouts**. The precompile's parser is compiled with a feature flag (`error_refund`) that selects which layout it expects. When the wrong contract version is paired with the wrong precompile build, `withdrawToNear` silently burns the caller's tokens without completing the bridge transfer, causing permanent fund loss.

---

### Finding Description

**`EvmErc20.withdrawToNear`** constructs its precompile input as:

```
[flag=0x01][amount: 32 bytes][recipient: variable bytes]
``` [1](#0-0) 

**`EvmErc20V2.withdrawToNear`** constructs its precompile input as:

```
[flag=0x01][sender: 20 bytes][amount: 32 bytes][recipient: variable bytes]
``` [2](#0-1) 

Both call the same precompile address via assembly, passing the exact byte count (`input_size`) with no ABI padding. The precompile's `parse_input` function is conditionally compiled:

**Without `error_refund`** (expects `EvmErc20` format):
```rust
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
``` [3](#0-2) 

**With `error_refund`** (expects `EvmErc20V2` format):
```rust
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))
}
``` [4](#0-3) 

After `parse_input`, the ERC-20 branch reads `amount` from the first 32 bytes of the returned slice:

```rust
let amount = parse_amount(&input[..32])?;
let Recipient { receiver_account_id, message } = parse_recipient(&input[32..])?;
``` [5](#0-4) 

**Mismatch scenario A** — `EvmErc20V2` used against a non-`error_refund` precompile:
- Precompile receives `[0x01][sender:20][amount:32][recipient]`
- `parse_input` strips only the flag byte, returning `[sender:20][amount:32][recipient]`
- `parse_amount` reads the first 32 bytes = `[sender:20][first 12 bytes of amount]`
- This 256-bit value has the top 160 bits set to the sender address, which always exceeds `u128::MAX`
- `parse_amount` returns `ERR_INVALID_AMOUNT`; the precompile call fails

**Mismatch scenario B** — `EvmErc20` used against an `error_refund` precompile:
- Precompile receives `[0x01][amount:32][recipient]`
- `parse_input` reads `refund_address` from `input[1..21]` = first 20 bytes of the amount field (wrong)
- Returns `input[21..]` = last 12 bytes of amount + recipient bytes
- `parse_amount` reads 32 bytes = `[last 12 bytes of amount][first 20 bytes of recipient]` — completely wrong value

In both cases, the assembly call does **not check the return value**:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
}
``` [6](#0-5) [7](#0-6) 

The `_burn` call executes unconditionally before the precompile call in both contracts: [8](#0-7) [9](#0-8) 

When the precompile fails, the EVM transaction does not revert (no `require(res != 0)`), so tokens are permanently burned with no corresponding NEP-141 transfer on NEAR.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user calling `withdrawToNear` on the mismatched contract version will have their ERC-20 tokens burned on Aurora with no corresponding NEP-141 tokens released on NEAR. The tokens are irrecoverably destroyed. There is no revert, no error surfaced to the caller, and no refund mechanism that can recover from a silent assembly call failure.

---

### Likelihood Explanation

**Medium.** Both `EvmErc20` and `EvmErc20V2` are production contracts built and deployed as Aurora ERC-20 mirror tokens (both appear in the `package.json` build script). The `error_refund` feature is a compile-time flag in the Aurora Engine WASM binary. If the engine is upgraded to enable `error_refund` while legacy `EvmErc20` (V1) contracts remain deployed, every user calling `withdrawToNear` on those V1 contracts will silently lose funds. The incompatibility is not surfaced to users or callers in any way.

---

### Recommendation

1. **Short term:** Add a `require(res != 0, "EXIT_FAILED")` check after the assembly `call` in both `EvmErc20.withdrawToNear` and `EvmErc20V2.withdrawToNear` so that a failed precompile call reverts the entire transaction (including the `_burn`), preventing silent token loss.

2. **Long term:** Unify the input format across `EvmErc20` and `EvmErc20V2` so that both contracts produce the same byte layout regardless of which precompile build is active. Document the coupling between the `error_refund` feature flag and the required ERC-20 contract version, and enforce a migration path that atomically upgrades both the engine binary and all deployed ERC-20 mirror contracts.

---

### Proof of Concept

1. Deploy `EvmErc20V2` as a mirror token on an Aurora instance compiled **without** the `error_refund` feature.
2. Alice holds 100 tokens in `EvmErc20V2`.
3. Alice calls `EvmErc20V2.withdrawToNear("alice.near", 100)`.
4. `_burn(alice, 100)` executes — Alice's balance drops to 0.
5. The precompile input is `[0x01][alice_address:20][100_as_u256:32]["alice.near"]`.
6. The precompile (without `error_refund`) reads `amount` from bytes 1–32 = `[alice_address:20][first 12 bytes of 100]` — a value far exceeding `u128::MAX`.
7. `parse_amount` returns `ERR_INVALID_AMOUNT`; the precompile call returns 0.
8. The assembly block does not revert; the EVM transaction succeeds.
9. Alice's 100 tokens are permanently destroyed. No NEP-141 tokens are released on NEAR.
