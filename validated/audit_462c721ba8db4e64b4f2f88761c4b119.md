### Title
`ManageVotingPermission` Holders Can Transitively Delegate `ManageVotingPermission` to Arbitrary Principals — (`rs/sns/governance/src/neuron.rs`)

### Summary

In SNS governance, a principal holding only `ManageVotingPermission` can call `add_neuron_permissions` to grant `ManageVotingPermission` itself to any new principal. Because `ManageVotingPermission` is included in `PERMISSIONS_RELATED_TO_VOTING`, the authorization check treats it as a voting-related permission that a `ManageVotingPermission` holder may freely grant. This creates an unbounded, neuron-owner-unaware delegation chain: each grantee can re-delegate the same permission, ultimately allowing an attacker to gain `Vote` and `SubmitProposal` rights over a neuron they were never authorized to control.

### Finding Description

`PERMISSIONS_RELATED_TO_VOTING` is defined as:

```rust
pub const PERMISSIONS_RELATED_TO_VOTING: &'static [NeuronPermissionType] = &[
    NeuronPermissionType::Vote,
    NeuronPermissionType::SubmitProposal,
    NeuronPermissionType::ManageVotingPermission,   // ← included
];
``` [1](#0-0) 

`check_principal_authorized_to_change_permissions` then allows any caller who holds `ManageVotingPermission` to add or remove any permission that `is_exclusively_voting_related()` returns `true` for:

```rust
let sufficient_permissions = if permissions_to_change.is_exclusively_voting_related() {
    vec![
        NeuronPermissionType::ManagePrincipals,
        NeuronPermissionType::ManageVotingPermission,   // ← sufficient to grant ManageVotingPermission
    ]
} else {
    vec![NeuronPermissionType::ManagePrincipals]
};
``` [2](#0-1) 

`is_exclusively_voting_related` simply checks membership in `PERMISSIONS_RELATED_TO_VOTING`:

```rust
pub fn is_exclusively_voting_related(&self) -> bool {
    let permissions_related_to_voting = Neuron::PERMISSIONS_RELATED_TO_VOTING
        .iter().map(|p| *p as i32).collect::<Vec<_>>();
    self.permissions.iter().all(|p| permissions_related_to_voting.contains(p))
}
``` [3](#0-2) 

Because `ManageVotingPermission` is in that set, a caller with only `ManageVotingPermission` can pass `[ManageVotingPermission]` as `permissions_to_add`, satisfy the check, and grant `ManageVotingPermission` to any new principal. That new principal can repeat the process indefinitely.

The `add_neuron_permissions` function enforces no additional constraint that would prevent this: [4](#0-3) 

### Impact Explanation

A principal with `ManageVotingPermission` can:
1. Grant `ManageVotingPermission` to an attacker-controlled principal.
2. That attacker can then grant `Vote` and `SubmitProposal` to themselves.
3. The attacker can now vote and submit proposals on behalf of the neuron without the neuron owner's knowledge or consent.

`SubmitProposal` is explicitly noted as "potentially a balance changing operation" because submitting proposals can reduce a neuron's stake. [5](#0-4) 

The neuron owner who granted `ManageVotingPermission` to a trusted party (e.g., a voting delegate) did not intend for that party to be able to re-delegate the management permission itself to arbitrary third parties.

### Likelihood Explanation

Any unprivileged ingress sender who has been granted `ManageVotingPermission` on a neuron can exploit this directly. The entry path is a standard `manage_neuron` → `AddNeuronPermissions` ingress call. No privileged access, key compromise, or social engineering beyond the initial `ManageVotingPermission` grant is required. The scenario where a voting delegate is phished or acts maliciously is realistic.

### Recommendation

Remove `ManageVotingPermission` from `PERMISSIONS_RELATED_TO_VOTING`. A principal with `ManageVotingPermission` should only be able to grant/revoke `Vote` and `SubmitProposal` — not the management permission itself. Granting `ManageVotingPermission` to new principals should require `ManagePrincipals`.

Concretely, change:
```rust
pub const PERMISSIONS_RELATED_TO_VOTING: &'static [NeuronPermissionType] = &[
    NeuronPermissionType::Vote,
    NeuronPermissionType::SubmitProposal,
    // ManageVotingPermission removed — only ManagePrincipals may grant it
];
```

### Proof of Concept

1. Neuron owner (Alice) holds `ManagePrincipals` and grants `ManageVotingPermission` to Bob (a trusted voting delegate) via `AddNeuronPermissions`.
2. Bob is phished. Attacker (Eve) convinces Bob to call `AddNeuronPermissions` with `principal_id = Eve`, `permissions_to_add = [ManageVotingPermission]`.
3. `check_principal_authorized_to_change_permissions` is called with `caller = Bob`, `permissions_to_change = [ManageVotingPermission]`. Since `[ManageVotingPermission]` is exclusively voting-related and Bob has `ManageVotingPermission`, the check passes.
4. Eve now holds `ManageVotingPermission`. Eve calls `AddNeuronPermissions` with `principal_id = Eve`, `permissions_to_add = [Vote, SubmitProposal]`.
5. Eve can now vote and submit proposals on behalf of Alice's neuron. Alice never authorized Eve.
