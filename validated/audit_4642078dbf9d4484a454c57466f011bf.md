### Title
Hardcoded `L1_TX_NATIVE_PRICE` Creates Permanent Native-Resource Mispricing for L1→L2 Transactions — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is a compile-time constant set to `10` that governs how many native proving resources are allocated per gas unit for every L1→L2 priority transaction. Unlike L2 transactions, which use an operator-adjustable `native_price` from block metadata, L1→L2 transactions are permanently bound to this hardcoded value. The constant carries an explicit `TODO (EVM-1157): find a reasonable value for it.` comment, acknowledging it is not properly calibrated. As actual proving costs evolve, the fixed constant creates a permanent and growing mismatch between the native resources granted to L1→L2 transactions and what the operator is economically compensated to prove.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs` line 66, the constant is defined:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

In `process_l1_transaction.rs`, this constant is unconditionally substituted for the operator-provided `native_price`:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
let native_per_gas = ...
    u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

For L2 transactions, the operator-supplied `native_price` is read from block metadata and used to compute `native_per_gas`:

```rust
let native_price = system.get_native_price();
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [3](#0-2) 

The documentation explicitly confirms the design split: *"Note: for L1→L2 transactions we use a code constant instead of one provided by operator."* [4](#0-3) 

The `native_per_gas` ratio directly controls the native resource limit granted to a transaction:

```
nativeLimit = gasLimit × native_per_gas = gasLimit × (gas_price / L1_TX_NATIVE_PRICE)
``` [5](#0-4) 

Because `L1_TX_NATIVE_PRICE = 10` is fixed, any increase in actual proving cost (reflected by the operator raising `native_price` for L2 transactions) creates a permanent divergence:

| Transaction type | gas_price | native_price | native_per_gas |
|---|---|---|---|
| L2 tx (operator sets `native_price = 100`) | 1000 | 100 | 10 |
| L1→L2 tx (hardcoded `L1_TX_NATIVE_PRICE = 10`) | 1000 | **10** | **100** |

The L1→L2 transaction receives **10× more native resources** per gas unit than an L2 transaction at the same gas price, while paying the same fee. The operator must prove the excess work without compensation.

---

### Impact Explanation

**Resource accounting bug / public funds-loss path for the operator.**

When the operator raises `native_price` to reflect real proving costs, L1→L2 transactions continue to be priced against the stale constant of `10`. Every L1→L2 priority transaction submitted at the minimum viable gas price forces the operator to perform `(operator_native_price / 10)×` more proving work than the fee covers. At scale, or when proving costs increase significantly (e.g., after a protocol upgrade that increases RISC-V cycle costs), this makes operating the L1→L2 bridge economically unviable. The operator cannot refuse to process priority transactions without breaking the bridge, so the loss is forced and unbounded.

Additionally, if the constant is ever miscalibrated in the opposite direction (too high), L1→L2 transactions will be allocated insufficient native resources and will revert, breaking the L1→L2 deposit path entirely.

---

### Likelihood Explanation

**Medium.** The TODO comment in the source code explicitly acknowledges the value `10` is a placeholder. Any protocol upgrade that changes proving costs, or any deployment where the operator sets `native_price` significantly above `10`, immediately triggers the mispricing. The test suite confirms the hardcoded value is `10` and is used in production paths: [6](#0-5) 

The entry path requires no privilege: any user can submit an L1→L2 priority transaction.

---

### Recommendation

Replace the compile-time constant `L1_TX_NATIVE_PRICE` with a value that is either:

1. **Dynamically read from a governance-controlled on-chain parameter** (analogous to how `native_price` is set for L2 transactions via block metadata), so it can be updated as proving costs evolve without a full protocol upgrade.
2. **Bounded relative to the operator's current `native_price`** — e.g., `L1_TX_NATIVE_PRICE = min(operator_native_price, CENSORSHIP_FLOOR)` — so that L1→L2 transactions cannot receive more native resources per gas unit than L2 transactions at the same price, while still preventing operator censorship via an enforced floor.

The TODO `EVM-1157` should be resolved before mainnet deployment.

---

### Proof of Concept

1. Operator sets `native_price = 1000` in block metadata to reflect actual proving costs.
2. User submits an L1→L2 priority transaction with `gas_price = 1000`, `gas_limit = 1_000_000`.
3. Bootloader computes: `native_per_gas = ceil(1000 / 10) = 100` (using `L1_TX_NATIVE_PRICE`).
4. Native limit granted: `1_000_000 × 100 = 100_000_000`.
5. An equivalent L2 transaction at the same gas price would receive: `native_per_gas = ceil(1000 / 1000) = 1`, native limit = `1_000_000`.
6. The L1→L2 transaction consumes up to `100×` more proving resources than the fee covers.
7. The operator bears the proving cost of `99_000_000` native units without compensation per such transaction.
8. A user submitting many L1→L2 transactions (which cannot be censored) can drain operator proving capacity at a fraction of the true cost. [7](#0-6) [8](#0-7) [9](#0-8)
