### Title
Front-Running `submit` Steals Relayer Priority Fee Rewards - (File: `engine/src/contract_methods/evm_transactions.rs`)

### Summary
The `submit` and `submit_with_args` functions on the Aurora Engine NEAR contract are callable by any NEAR account without restriction. The EIP-1559 priority fee (tip) embedded in an EVM transaction is credited as an EVM balance reward to whoever calls `submit` as the NEAR predecessor. An attacker monitoring the NEAR transaction pool can front-run a legitimate relayer by rebroadcasting the same EVM transaction bytes with a higher NEAR gas price, consuming the EVM nonce first and stealing the priority fee reward.

### Finding Description

In `engine/src/contract_methods/evm_transactions.rs`, the `submit` function derives the relayer address directly from the NEAR predecessor account with no access control beyond `require_running`: [1](#0-0) 

```rust
let relayer_address = predecessor_address(&env.predecessor_account_id());
```

This relayer address is passed into `engine::submit`, which eventually calls `refund_unused_gas` in `engine/src/engine.rs`: [2](#0-1) 

Inside `refund_unused_gas`, the priority fee portion of the gas payment is credited to the relayer's EVM balance: [3](#0-2) 

Specifically, `relayer_reward = gas_used * priority_fee_per_gas` (the EIP-1559 tip set by the EVM transaction sender) is added to the relayer's EVM account on Aurora. The same pattern exists in `submit_with_args`: [4](#0-3) 

Because there is no caller restriction on `submit`, any NEAR account can call it. An attacker who observes a pending `submit` call in the NEAR transaction pool can:

1. Extract the raw EVM transaction bytes from the pending call's input.
2. Rebroadcast an identical `submit` call with a higher NEAR gas price.
3. Have their transaction included first, consuming the EVM sender's nonce via `check_nonce`.
4. Receive the EIP-1559 priority fee reward in their own EVM address on Aurora.

The legitimate relayer's transaction then fails with a nonce error and receives nothing.

### Impact Explanation

The attacker steals the EIP-1559 priority fee (tip) that the EVM transaction sender explicitly set to compensate the relayer. This fee is real ETH-on-Aurora balance credited to the attacker's EVM address. This constitutes **theft of unclaimed yield** (the relayer's expected compensation), classified as **High** impact.

### Likelihood Explanation

NEAR's transaction pool is observable. The attack requires only standard mempool monitoring and submitting a competing NEAR transaction with a higher gas price — a well-understood front-running technique. No special privileges or leaked keys are required. Any unprivileged NEAR account can execute this. Likelihood is **Medium-High** whenever EVM transactions carry non-zero priority fees.

### Recommendation

Either:
- Restrict `submit` and `submit_with_args` to a set of registered/whitelisted relayer NEAR accounts (analogous to the `add_relayer_key` / key-manager system already present for access-key-based relayers), or
- Document clearly that the priority fee reward goes to whoever submits the NEAR transaction, so EVM transaction senders are aware the tip may not reach the intended relayer.

### Proof of Concept

1. Alice (legitimate relayer) broadcasts a NEAR transaction calling `submit` on `aurora` with EVM tx bytes `T` where `T` has `max_priority_fee_per_gas = 10 gwei`.
2. Attacker Bob observes Alice's pending transaction in the NEAR mempool.
3. Bob broadcasts an identical `submit` call with the same `T` bytes but a higher NEAR gas price.
4. Bob's transaction is included first. Inside `submit_with_alt_modexp`:
   - `relayer_address = predecessor_address(Bob's NEAR account)` [5](#0-4) 
   - EVM nonce for the sender is consumed.
   - `refund_unused_gas` credits `gas_used * 10 gwei` to Bob's EVM address. [6](#0-5) 
5. Alice's transaction arrives and fails: `check_nonce` rejects it because the nonce was already consumed. Alice receives no reward.
6. Bob has stolen the priority fee that Alice was entitled to for relaying the transaction.
