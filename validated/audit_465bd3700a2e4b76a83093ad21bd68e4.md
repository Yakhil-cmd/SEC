Audit Report

## Title
Missing Size Bound on `VetKdEncryptedKeyShareContent` Enables Per-Artifact Memory Exhaustion via Byzantine Node — (`rs/types/types/src/consensus/idkg.rs`, `rs/crypto/src/vetkd/mod.rs`)

## Summary
The `TryFrom<pb::VetKdKeyShare>` conversion wraps an attacker-controlled `Vec<u8>` directly into `VetKdEncryptedKeyShareContent` with no length check. During per-share verification, `verify_basic_sig` calls `message.as_signed_bytes()`, which allocates a full copy of the blob before the signature check fails. A single Byzantine subnet node can exploit this to cause OOM on an honest replica by submitting crafted oversized `VetKdKeyShare` artifacts, temporarily removing the replica from the subnet.

## Finding Description
**Root cause — no length check in `TryFrom`:**
`rs/types/types/src/consensus/idkg.rs` L1551–1562 wraps `value.encrypted_key_share` (an arbitrary `Vec<u8>`) directly into `VetKdEncryptedKeyShareContent` with no size validation. The legitimate size is `EncryptedKeyShare::BYTES` = `2 * G1Affine::BYTES + G2Affine::BYTES` (≈144 bytes for BLS12-381), defined at `rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs` L401. [1](#0-0) [2](#0-1) 

**Transport has no body-size limit:**
`rs/p2p/consensus_manager/src/receiver.rs` L44–51 explicitly calls `.layer(DefaultBodyLimit::disable())`, so a Byzantine peer can push an arbitrarily large `SlotUpdate` with no transport-level rejection. [3](#0-2) 

**Verification allocates a full copy of the blob:**
`verify_encrypted_key_share_internal` at `rs/crypto/src/vetkd/mod.rs` L292–301 calls `BasicSigVerifierInternal::verify_basic_sig` with `&key_share.encrypted_key_share` as the `Signable` message. [4](#0-3) 

`verify_basic_sig` at `rs/crypto/src/sign/basic_sig.rs` L30 calls `message.as_signed_bytes()`. [5](#0-4) 

`as_signed_bytes()` at `rs/types/types/src/crypto/sign.rs` L38–42 calls `write_signed_bytes_without_domain_separator`, which for `VetKdEncryptedKeyShareContent` at `rs/types/types/src/crypto/vetkd.rs` L49–52 does `bytes.extend_from_slice(&self.0)` — allocating a full copy of the entire blob before the Ed25519 signature check fails. [6](#0-5) [7](#0-6) 

**Existing length guard does not protect the verification path:**
`EncryptedKeyShare::deserialize` at `rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs` L446–448 checks `val.len() != Self::BYTES`, but this function is only called inside `combine_encrypted_key_shares_internal` at `rs/crypto/src/vetkd/mod.rs` L357. It is never called during `verify_encrypted_key_share_internal`. [8](#0-7) [9](#0-8) 

**Parallel validation amplifies impact:**
`validate_signature_shares` at `rs/consensus/idkg/src/signer.rs` L226–246 processes all unvalidated shares via `into_par_iter()` on a thread pool. With N active VetKd requests and thread pool width W, up to W × blob_size bytes can be allocated simultaneously. [10](#0-9) 

## Impact Explanation
A single Byzantine subnet node (below the consensus fault threshold) can cause OOM on one honest replica by sending crafted `VetKdKeyShare` artifacts with oversized `encrypted_key_share` blobs. The replica process is killed by the OS, temporarily removing it from the subnet. This matches the **High** bounty impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation
The attack requires being a registered subnet node or having compromised one — an explicitly in-scope Byzantine protocol peer below the consensus fault threshold. The exploit path is deterministic, requires no timing, and is locally testable. The VetKd feature is active on subnets processing vetKD requests. A single malicious node can repeatedly trigger the condition.

## Recommendation
Add a length check in `TryFrom<pb::VetKdKeyShare>` before constructing `VetKdEncryptedKeyShareContent` in `rs/types/types/src/consensus/idkg.rs`:

```rust
impl TryFrom<pb::VetKdKeyShare> for VetKdKeyShare {
    type Error = ProxyDecodeError;
    fn try_from(value: pb::VetKdKeyShare) -> Result<Self, Self::Error> {
        use ic_crypto_internal_bls12_381_vetkd::EncryptedKeyShare;
        if value.encrypted_key_share.len() != EncryptedKeyShare::BYTES {
            return Err(ProxyDecodeError::Other(format!(
                "VetKdKeyShare::encrypted_key_share has invalid length: {} (expected {})",
                value.encrypted_key_share.len(),
                EncryptedKeyShare::BYTES,
            )));
        }
        Ok(Self { ... })
    }
}
```

Alternatively, add the same check at the top of `verify_encrypted_key_share_internal` in `rs/crypto/src/vetkd/mod.rs` before calling `BasicSigVerifierInternal::verify_basic_sig`.

## Proof of Concept
```rust
// Construct an oversized artifact
let large_share = pb::VetKdKeyShare {
    signer_id: Some(valid_node_id_proto),
    request_id: Some(valid_request_id_proto),
    encrypted_key_share: vec![0u8; 10_000_000], // 10 MB
    node_signature: vec![0u8; 64],
};

// TryFrom succeeds — no length check
let share = VetKdKeyShare::try_from(large_share).unwrap();

// verify_encrypted_key_share_internal allocates ~10 MB in as_signed_bytes()
// before returning VerificationError (invalid signature)
let result = crypto.verify_encrypted_key_share(signer_id, &share.share, &args);
assert!(matches!(result, Err(VetKdKeyShareVerificationError::VerificationError(_))));
// Memory spike of ~10 MB occurred before the error was returned
```

A deterministic integration test can confirm the allocation by measuring process RSS before and after calling `verify_encrypted_key_share` with a 100 MB blob, asserting the spike occurs and that the fix eliminates it.
