### Title
SNS-specific upgrade path entries are permanently immutable after first insertion due to missing `occupied.insert(to)` — (`rs/nns/sns-wasm/src/sns_wasm.rs`)

### Summary
`insert_sns_specific_upgrade_path_entry` silently discards overwrite attempts on existing entries. Once a custom upgrade path entry `A -> B` is set for a specific SNS governance canister, no subsequent NNS proposal can change it to `A -> C`. The log message falsely claims the entry "is being overwritten" but the actual map is never updated.

### Finding Description
In `UpgradePath::insert_sns_specific_upgrade_path_entry` (lines 1927–1951), the `Entry::Occupied` match arm logs a message but omits the call to `occupied.insert(to)`. The `to` parameter is consumed by the log format string but never written to the map. The parallel function `insert_upgrade_path_entry` (lines 1954–1969) correctly calls `occupied.insert(to)` in the same branch. This is a copy-paste omission: the `mut` keyword is also absent from the `occupied` binding in the SNS-specific variant, which would be required to call `.insert()`.

The `insert_upgrade_path_entries` public method (lines 532–628) dispatches to `insert_sns_specific_upgrade_path_entry` for all SNS-specific requests and returns `InsertUpgradePathEntriesResponse { error: None }` — a success response — regardless of whether the overwrite actually occurred. [5](#0-4) 

### Impact Explanation
Once any custom upgrade path entry is written for a specific SNS governance canister, NNS governance permanently loses the ability to correct it. If the path points to a malicious or broken WASM, the SNS is permanently directed to that WASM on its next upgrade cycle. The SNS governance and ledger canisters would be replaced with attacker-controlled code, enabling theft of SNS tokens.

### Likelihood Explanation
The trigger requires an NNS proposal to first insert a custom path (a legitimate operational action used for emergency upgrades), and a second NNS proposal to correct it. The second proposal succeeds at the governance layer (returns no error) but silently has no effect. The scenario does not require a malicious governance majority — it only requires that any custom path was ever set and later needs correction, which is the exact operational use case this feature was designed for (emergency re-routing of a stuck SNS).

### Recommendation
In `insert_sns_specific_upgrade_path_entry`, change the `Entry::Occupied` arm to match `insert_upgrade_path_entry`:

```rust
Entry::Occupied(mut occupied) => {
    println!(...);
    occupied.insert(to);  // <-- this line is missing
}
```

### Proof of Concept
Unit test: insert custom path `A -> B` for governance canister `G`, then insert `A -> C` for the same `G`. Call `get_next_version` with `from = A` and `caller = G`. Assert result is `C`. With the current code, the result is `B`. [6](#0-5)

                    to

            }
        };
