Audit Report

## Title
Copy-Paste Error in `From<ValidatedNeuronsFundParticipationConstraints>` Sets Wrong Hard Cap for Neurons' Fund Participation — (File: `rs/sns/swap/src/neurons_fund.rs`)

## Summary
In `rs/sns/swap/src/neurons_fund.rs` lines 512–514, the `From<ValidatedNeuronsFundParticipationConstraints<F>> for NeuronsFundParticipationConstraintsPb` implementation assigns `value.min_direct_participation_threshold_icp_e8s` to the `max_neurons_fund_participation_icp_e8s` field instead of `value.max_neurons_fund_participation_icp_e8s`. Any code path that serializes a `ValidatedNeuronsFundParticipationConstraints` to protobuf via this `From` impl will produce a corrupted protobuf where the hard cap equals the minimum threshold — a value potentially orders of magnitude smaller — causing the SNS swap canister to severely underestimate and truncate Neurons' Fund participation amounts.

## Finding Description
The root cause is at `rs/sns/swap/src/neurons_fund.rs:512–514`:

```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.min_direct_participation_threshold_icp_e8s,  // ← wrong field
),
```

`ValidatedNeuronsFundParticipationConstraints` holds both fields as distinct `u64` values (`rs/nervous_system/neurons_fund/src/lib.rs:296–297`). The `From` impl is the only conversion path from the validated struct back to protobuf.

When `update_total_participation_amounts` runs in the swap canister (`rs/sns/swap/src/swap.rs:563`), it calls `constraints.try_into()` to re-validate the protobuf into a `PolynomialNeuronsFundParticipation`, then calls `MatchedParticipationFunction::apply`. Inside `apply`, `self.max_neurons_fund_participation_icp_e8s` is used as `hard_cap_icp` (`rs/nervous_system/neurons_fund/src/lib.rs:1025`). If the protobuf was produced by the buggy `From` impl, `hard_cap_icp` equals `min_direct_participation_threshold_icp_e8s` — truncating every NF participation computation that would otherwise exceed the minimum threshold. Additionally, `update_total_participation_amounts` applies a second cap at `constraints.max_neurons_fund_participation_icp_e8s` from the raw protobuf (`rs/sns/swap/src/swap.rs:587–589`), which is also corrupted.

Existing validation in `TryFrom<NeuronsFundParticipationConstraintsPb>` does not detect this corruption because `min_direct_participation_threshold_icp_e8s` is a valid `u64` value that passes all field-level checks — the semantic relationship between the two fields is not enforced.

## Impact Explanation
This matches the allowed High impact: "Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm." If the `From` impl is invoked in the NNS governance → SNS swap init path (governance creates a `ValidatedNeuronsFundParticipationConstraints` and serializes it to pass as `neurons_fund_participation_constraints`), every SNS swap using matched Neurons' Fund funding will have its NF participation hard-capped at the minimum threshold value rather than the intended maximum. The SNS treasury receives less ICP than the Neurons' Fund neurons' maturity warrants, and refund accounting is computed against the wrong effective amount, constituting a concrete financial harm to SNS projects and NF participants.

## Likelihood Explanation
The `From` impl is production code with no `#[cfg(test)]` gate. It is the standard Rust conversion mechanism and will be invoked whenever governance serializes a `ValidatedNeuronsFundParticipationConstraints` to protobuf. The NNS governance code references `ValidatedNeuronsFundParticipationConstraints` in `rs/nns/governance/src/neurons_fund.rs`. No privileged access is required beyond submitting an SNS swap proposal that passes NNS governance — a normal governance action. The bug fires deterministically on every affected swap.

## Recommendation
Fix the copy-paste error:

```rust
max_neurons_fund_participation_icp_e8s: Some(
    value.max_neurons_fund_participation_icp_e8s,  // ← correct field
),
```

Add a round-trip unit test asserting that after `NeuronsFundParticipationConstraintsPb::from(validated)`, the resulting `max_neurons_fund_participation_icp_e8s` equals the original `max_neurons_fund_participation_icp_e8s`, not `min_direct_participation_threshold_icp_e8s`.

## Proof of Concept
Construct a `ValidatedNeuronsFundParticipationConstraints` with `min_direct_participation_threshold_icp_e8s = 36_000 * E8` and `max_neurons_fund_participation_icp_e8s = 45_000 * E8` (matching existing test data in `rs/nns/governance/src/governance/test_data.rs:205–218`). Call `NeuronsFundParticipationConstraintsPb::from(constraints)`. Assert that the resulting protobuf's `max_neurons_fund_participation_icp_e8s` equals `45_000 * E8`. The assertion will fail — it will equal `36_000 * E8` — demonstrating the bug. A follow-on integration test can pass this corrupted protobuf through `update_total_participation_amounts` with `direct_participation_icp_e8s` in the `[36_000, 45_000]` ICP range and observe that NF participation is truncated to `36_000 ICP` instead of the correct higher value.
