### Title
Unchecked Precompile Return Value in `withdrawToNear` Enables Silent Token Burn Without Exit Promise — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`withdrawToNear` in `EvmErc20.sol` burns the caller's ERC-20 tokens before calling the `ExitToNear` precompile, and never checks the return value of that call. When a non-UTF-8 `recipient` byte sequence is supplied, the precompile's `parse_recipient` returns an `ExitError`, the inner EVM `call` returns `0`, and Solidity silently continues. The burn is committed but no NEAR promise is scheduled, permanently destroying the user's ERC-20 balance while the corresponding NEP-141 tokens remain locked in Aurora's account.

---

### Finding Description

**Step 1 — Entry point (no privilege required)**

Any EVM account holding a balance of a bridged ERC-20 token can call `withdrawToNear` directly. The `recipient` parameter is typed `bytes memory` with no Solidity-level validation. [1](#0-0) 

**Step 2 — Burn executes unconditionally before the precompile call**

`_burn` is called on line 54, reducing the caller's balance immediately. This state change is not rolled back if the subsequent precompile call fails. [2](#0-1) 

**Step 3 — Assembly call return value is never checked**

The `call` opcode stores its success flag in `res`, but `res` is never read after the assembly block. There is no `require(res != 0)` or equivalent guard. [3](#0-2) 

**Step 4 — Non-UTF-8 recipient causes `parse_recipient` to return `ExitError`**

Inside the precompile, `ExitToNearParams::try_from` dispatches to `parse_recipient` for the ERC-20 path (flag `0x1`). `parse_recipient` calls `str::from_utf8` and propagates an `ExitError` on failure. [4](#0-3) [5](#0-4) 

The existing unit test at line 1139 of `native.rs` explicitly confirms this failure path: [6](#0-5) 

**Step 5 — No promise log is emitted; NEP-141 tokens are stranded**

A successful precompile run emits a `promise_log` containing the serialized `PromiseArgs` that the Aurora engine uses to schedule the NEP-141 transfer. When the precompile returns `ExitError`, this log is never produced, so no cross-contract call is scheduled. [7](#0-6) 

**Step 6 — `error_refund` feature does not mitigate this**

The `error_refund` feature adds a `refund_address` field and a callback (`exit_to_near_precompile_callback`) that refunds tokens when the *downstream NEAR call* fails. It does not protect against the case where the precompile itself fails before any promise is created, because the refund callback is only registered as part of a successfully constructed `PromiseArgs`. [8](#0-7) 

The same flaw exists identically in `EvmErc20V2.sol`: [9](#0-8) 

---

### Impact Explanation

The caller's ERC-20 tokens are permanently burned (EVM supply decreases). The corresponding NEP-141 tokens remain in Aurora's account on NEAR with no user-accessible recovery path. This constitutes at minimum **High — Temporary freezing of funds** (NEP-141 tokens locked in Aurora's account) and arguably **Critical — Permanent freezing of funds** (ERC-20 tokens destroyed with no refund mechanism triggered).

---

### Likelihood Explanation

- No admin access, no special role, no governance action required.
- Any EVM account holding a balance of any bridged ERC-20 token can trigger this.
- The `recipient` parameter is `bytes memory` — arbitrary byte sequences are accepted by the ABI encoder.
- A single malformed call is sufficient; the attacker only needs to hold a non-zero balance.

---

### Recommendation

Add a revert on precompile call failure immediately after the assembly block in both `EvmErc20.sol` and `EvmErc20V2.sol`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures the entire transaction (including the preceding `_burn`) is atomically reverted if the precompile rejects the input, restoring the invariant that every burn is paired with a scheduled exit promise.
