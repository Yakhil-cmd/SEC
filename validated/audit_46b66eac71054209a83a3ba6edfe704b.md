### Title
Unbounded `while` Loop in `Hashchain::move_to_block` Can Permanently Freeze All Engine Transactions — (`engine-hashchain/src/hashchain.rs`)

---

### Summary

The `Hashchain::move_to_block` function contains an unbounded `while` loop that iterates once per skipped NEAR block. When the hashchain is enabled and a large number of NEAR blocks pass without any Aurora Engine transactions, the next call to any hashchain-wrapped engine method will trigger this loop with a number of iterations equal to the entire block-height gap. If the gap is large enough to exhaust the NEAR transaction gas limit (300 TGas), the call fails, the hashchain state is never updated, and every subsequent transaction fails identically — permanently freezing the engine.

---

### Finding Description

`Hashchain::move_to_block` in `engine-hashchain/src/hashchain.rs` (lines 67–88) iterates from the stored `current_block_height` up to `next_block_height` one step at a time:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
``` [1](#0-0) 

This function is called unconditionally from `load_hashchain` in `engine/src/hashchain.rs` whenever the stored block height lags behind the current NEAR block height:

```rust
if let Some(hashchain) = maybe_hashchain.as_mut()
    && block_height > hashchain.get_current_block_height()
{
    hashchain.move_to_block(block_height)?;
}
``` [2](#0-1) 

`load_hashchain` is called at the start of both `with_hashchain` and `with_logs_hashchain`, which wrap every state-changing engine method (e.g., `submit`): [3](#0-2) [4](#0-3) 

There is no cap on the number of iterations. Each iteration calls `compute_block_hashchain`, which performs a keccak hash over a concatenated byte array — a non-trivial amount of NEAR gas per step. The gap `next_block_height − current_block_height` grows monotonically whenever the engine is idle (no transactions submitted). Once the gap is large enough to exhaust the 300 TGas NEAR transaction gas limit, the call panics/reverts before `save_hashchain` is reached, so the stored `current_block_height` is never advanced. Every subsequent transaction faces the same (or larger) gap and fails identically.

---

### Impact Explanation

**Permanent freezing of funds — Critical.**

Once the block-height gap exceeds the gas threshold, no hashchain-wrapped engine call can succeed. Users cannot submit EVM transactions, cannot withdraw ETH or ERC-20 tokens via exit precompiles, and cannot interact with any bridged asset. All funds held in the Aurora Engine EVM state become inaccessible. Because the stored hashchain block height is never updated on a failed call, the freeze is self-reinforcing: the gap only grows larger with each passing block, making recovery impossible without an admin intervention path (e.g., disabling the hashchain or resetting its state), which is not visible in the production code paths examined.

---

### Likelihood Explanation

**Medium.**

NEAR produces approximately one block per second. The number of iterations required to exhaust 300 TGas depends on the per-iteration gas cost of `compute_block_hashchain` (one keccak + memory allocation). Even at a conservative estimate of 1 GGas per iteration, only ~300,000 blocks (~3.5 days of inactivity) are needed. For a less-active Aurora deployment, a silo instance, or any engine that experiences a multi-day maintenance window or low-traffic period, this threshold is reachable without any attacker action. A malicious actor who observes the gap approaching the threshold can be the first to submit a transaction, triggering the freeze and locking all other users out.

---

### Recommendation

Replace the unbounded `while` loop with a bounded iteration that processes at most `MAX_BLOCKS_PER_TX` steps per call, returning a resumable error or a "catch-up needed" status so callers can issue multiple transactions to advance the hashchain incrementally. Alternatively, store only the previous block hash and skip intermediate empty blocks in O(1) by hashing the gap count into the chain computation, eliminating the need to iterate at all.

---

### Proof of Concept

1. The Aurora Engine is deployed with the hashchain enabled; `HC_STATE` is written to storage at block height `B`.
2. The engine receives no transactions for `N` NEAR blocks (e.g., a maintenance window or low-traffic silo).
3. At block `B + N`, any user submits an EVM transaction. The NEAR runtime calls `with_logs_hashchain` → `load_hashchain` → `move_to_block(B + N)`.
4. The `while` loop executes `N` iterations of `compute_block_hashchain`. For sufficiently large `N`, NEAR gas is exhausted before the loop completes.
5. The NEAR transaction fails; `save_hashchain` is never reached; the stored `current_block_height` remains `B`.
6. At block `B + N + 1`, the next user's transaction faces a gap of `N + 1` and also fails.
7. The engine is permanently frozen: no EVM transaction can succeed, and all user funds are inaccessible. [5](#0-4) [6](#0-5)

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}

fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
