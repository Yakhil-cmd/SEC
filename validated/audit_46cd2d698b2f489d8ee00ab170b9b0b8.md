Audit Report

## Title
Incomplete Blocklist Enforcement Allows Blocked Ethereum Addresses to Withdraw Existing ckETH/ckERC-20 Balances - (`rs/ethereum/cketh/minter/src/main.rs`, `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

The ckETH minter's `ETH_ADDRESS_BLOCKLIST` is enforced at deposit time (blocking the `from_address` Ethereum sender) and at withdrawal time (blocking the destination address), but the `withdraw_eth` and `withdraw_erc20` endpoints perform no check on the IC caller's associated Ethereum address. A blocked Ethereum address that previously deposited ETH and holds ckETH under an IC principal can call `withdraw_eth` with any non-blocked destination address, causing the minter to burn the ckETH and send ETH to that destination — bypassing the blocklist entirely.

## Finding Description

The blocklist comment in `blocklist.rs` states: *"ETH is not accepted from nor sent to addresses on this list."* The implementation enforces only two of the three required checks:

**1. Deposit-side check** (`deposit.rs` L323): `register_deposit_events` calls `crate::blocklist::is_blocked(&event.from_address())` and records an `InvalidDeposit` if the Ethereum sender is blocked. The IC-side `principal` (beneficiary) is never checked against the blocklist. [1](#0-0) 

**2. Withdrawal destination check** (`address.rs` L53): `validate_address_as_destination` checks only the *destination* Ethereum address against the blocklist. [2](#0-1) 

**3. Missing: withdrawal caller check** — `withdraw_eth` (`main.rs` L273) calls only `validate_caller_not_anonymous()` and `validate_address_as_destination(&recipient)`. There is no check on the IC caller's associated Ethereum address. [3](#0-2) 

The same gap exists in `withdraw_erc20` (`main.rs` L400-414). [4](#0-3) 

The `State` struct contains `minted_events: BTreeMap<EventSource, MintedEvent>` where each `MintedEvent` holds the original `ReceivedEvent` (which includes both `from_address` and `principal`), but there is no reverse lookup from principal to Ethereum address, and no such lookup is performed during withdrawal. [5](#0-4) 

## Impact Explanation

This is a **High** severity finding. The `ETH_ADDRESS_BLOCKLIST` is an OFAC sanctions compliance mechanism. The gap allows a sanctioned/blocked Ethereum address that holds ckETH (acquired before being blocked, or received via ICRC-1 transfer from another user) to call `withdraw_eth` or `withdraw_erc20` with a non-blocked destination address. The minter burns the ckETH from the principal and submits an Ethereum transaction to the non-blocked destination — the blocked entity has effectively moved funds out of the ckETH system, defeating the purpose of the blocklist. This constitutes a significant ck-token security impact with concrete compliance harm, matching the High impact class: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation

The exploit path requires no privileged access. Any IC principal whose associated Ethereum address appears on `ETH_ADDRESS_BLOCKLIST` and that holds a non-zero ckETH balance can trigger this path. The only preconditions are: (1) the principal is non-anonymous (`validate_caller_not_anonymous`), and (2) the supplied destination address is not itself blocked. No admin key, governance majority, or threshold corruption is required. The attack is repeatable for any amount up to the principal's full ckETH balance. [6](#0-5) 

## Recommendation

Add a blocklist check on the caller's associated Ethereum address in both `withdraw_eth` and `withdraw_erc20`. Since the minter does not currently maintain a principal-to-Ethereum-address mapping, the fix requires building one: at deposit time, record the `from_address` → `principal` association (already available in `ReceivedEthEvent` and `ReceivedErc20Event`) in a new `BTreeMap<Principal, Address>` field in `State`. During withdrawal, look up the caller's Ethereum address and reject if blocked:

```rust
// In withdraw_eth / withdraw_erc20, after validate_caller_not_anonymous():
if let Some(eth_addr) = read_state(|s| s.eth_address_of(caller)) {
    if crate::blocklist::is_blocked(&eth_addr) {
        return Err(WithdrawalError::SenderAddressBlocked { address: eth_addr.to_string() });
    }
}
```

Note: because ckETH is an ICRC-1 token freely transferable between principals, a complete fix also requires considering that a blocked principal could transfer ckETH to an unblocked principal before withdrawing. A defense-in-depth approach (e.g., also checking the `from_address` recorded at mint time for the specific tokens being burned) would provide stronger guarantees. [7](#0-6) 

## Proof of Concept

1. Ethereum address `A` (present in `ETH_ADDRESS_BLOCKLIST`) previously deposited 1 ETH via the helper contract, specifying IC principal `P`. The minter minted 1 ckETH to `P`.
2. `A` is added to the blocklist. Future deposits from `A` are rejected by `register_deposit_events`.
3. `P` calls `withdraw_eth { amount: 1_ckETH, recipient: "<non-blocked-address-B>", from_subaccount: None }`.
4. `validate_caller_not_anonymous()` passes — `P` is non-anonymous.
5. `validate_address_as_destination("<non-blocked-address-B>")` passes — `B` is not on the blocklist.
6. The minter calls `burn_from(Account { owner: P, subaccount: None }, 1_ckETH, ...)` on the ckETH ledger — succeeds.
7. The minter queues an `AcceptedEthWithdrawalRequest` and submits an Ethereum transaction sending ETH to `B`.
8. `A` has successfully moved 1 ETH out of the ckETH system despite being on the blocklist.

A deterministic integration test can be written using the existing `CkEthSetup` test harness: deposit from `SAMPLE_BLOCKED_ADDRESS` before it is blocked (or mock a pre-existing balance), add the address to the blocklist, then call `call_minter_withdraw_eth` with a non-blocked destination and assert the withdrawal succeeds — demonstrating the gap. pub fn validate_address_as_destination(address: &str) -> Result<Address, AddressValidationError> {
    let address =
        Address::from_str(address).map_err(|e| AddressValidationError::Invalid { error: e })?;
    if address == Address::ZERO {
        return Err(AddressValidationError::NotSupported(address));
    }
    if crate::blocklist::is_blocked(&address) {
        return Err(AddressValidationError::Blocked(address));
    }
    Ok(address)
