### Title
SNS-WASM Canister Stable Memory Layout Corruption on Upgrade Due to Missing Version/Compatibility Validation — (File: `rs/nns/sns-wasm/src/stable_memory.rs`)

---

### Summary

The `SnsWasmStableMemory` struct in the SNS-WASM canister uses a raw, unversioned binary layout with hardcoded byte offsets to store critical header fields in stable memory. The `post_upgrade` hook reads this layout with no version marker, no compatibility check, and no validation that the on-disk format matches the current code's expectations. Any future upgrade that alters `RESERVED_INITIAL_STABLE_MEMORY_BYTES`, the header field types, or their order will silently misread the stored offsets, causing the canister to serve WASM binaries from wrong stable-memory locations or to reconstruct a completely wrong `SnsWasmCanister` state — directly analogous to the Celo proxy storage-layout corruption class.

---

### Finding Description

The SNS-WASM canister persists its state across upgrades using a hand-rolled binary layout described in the module comment:

```
[reserved bytes][WASMs-end offset][Canister state size][WASM 1][WASM 2]...
``` [1](#0-0) 

The two header fields are located at hardcoded offsets derived from `reserved_bytes` (always `RESERVED_INITIAL_STABLE_MEMORY_BYTES = 1_000_000`):

```rust
fn get_wasms_end_offset_offset(&self) -> u64 { self.reserved_bytes }
fn get_canister_state_size_offset(&self) -> u64 {
    self.get_wasms_end_offset_offset() + (size_of::<u64>() as u64)
}
``` [2](#0-1) 

There is an existing layout inconsistency: `wasms_end_offset` is stored and read as a **`u32`** (4 bytes), but the gap to the next field is computed as `size_of::<u64>()` = **8 bytes**, leaving 4 bytes of silent padding between the two fields. This undocumented gap is invisible to any future developer who reads the code and assumes the gap equals the actual field width. [3](#0-2) 

The `post_upgrade` hook calls `from_stable_memory()`, which unconditionally constructs a fresh `SnsWasmStableMemory::default()` (hardcoding `reserved_bytes = 1_000_000`) and reads the header fields from those fixed offsets — with no magic-byte check, no version field, and no assertion that the on-disk layout matches the current code:

```rust
pub fn from_stable_memory() -> Self {
    SnsWasmStableMemory::<M>::default()
        .read_canister_state()
        .expect("Failed to read canister state from stable memory")
        .into()
}
``` [4](#0-3) 

```rust
fn canister_post_upgrade() {
    SNS_WASM.with(|c| {
        c.replace(SnsWasmCanister::<CanisterStableMemory>::from_stable_memory());
    });
}
``` [5](#0-4) 

Contrast this with the ICP Ledger and ICRC-1 Ledger `post_upgrade` hooks, which check for the `"MGR"` magic bytes and enforce a `ledger_version` integer before proceeding: [6](#0-5) 

The SNS-WASM canister has no equivalent guard.

The `wasm_indexes` map stored in `StableCanisterState` maps each WASM hash to a `(offset, size)` pair pointing into stable memory. If `RESERVED_INITIAL_STABLE_MEMORY_BYTES` is changed in a future version, or if the header field types or order change, `read_wasms_end_offset()` and `read_canister_state_size()` will return garbage values. The canister will then attempt to read WASM bytes from a wrong offset and wrong length, silently returning corrupted or zeroed bytes as valid SNS component WASMs. [7](#0-6) 

---

### Impact Explanation

The SNS-WASM canister is the authoritative source of SNS component WASMs (governance, ledger, root, swap, archive, index) and the SNS upgrade path. If its stable memory is misread after an upgrade:

- `get_latest_version_wasms()` will read WASM bytes from wrong stable-memory offsets, returning garbage or zero-length blobs as valid WASMs.
- `deploy_new_sns()` will install those corrupted WASMs into newly created SNS instances, rendering them non-functional or potentially exploitable.
- The SNS upgrade path (`upgrade_path`) will be reconstructed from a corrupted `StableCanisterState`, causing all existing SNS instances to receive wrong upgrade targets. [8](#0-7) 

---

### Likelihood Explanation

Exploitation requires a future NNS governance proposal that upgrades the SNS-WASM canister to a version that inadvertently changes the stable memory layout (e.g., alters `RESERVED_INITIAL_STABLE_MEMORY_BYTES`, changes a header field from `u32` to `u64`, or reorders fields). The existing 4-byte silent gap between `wasms_end_offset` and `canister_state_size` (caused by using `size_of::<u64>()` as the stride for a `u32` field) is precisely the kind of latent inconsistency that a future developer could "fix" without realizing it breaks the on-disk format. Because there is no version marker, the breakage would be silent — `post_upgrade` would succeed, and the corruption would only manifest when WASMs are later read or SNSes are deployed.

---

### Recommendation

1. **Add a magic-byte and version field** at the very start of the stable memory header (before `reserved_bytes`), analogous to the `"MGR"` check in the ICP Ledger. Validate these in `post_upgrade` before reading any other field.
2. **Replace the raw binary header** with a length-prefixed protobuf or CBOR envelope for the header metadata, eliminating the fragile offset arithmetic.
3. **Document and enforce the field widths** used in offset calculations. The current code uses `size_of::<u64>()` as the stride for a `u32` field, creating a silent 4-byte gap that is not reflected in any comment or test.
4. **Add a post-upgrade invariant check** that asserts `wasms_end_offset >= expected_minimum` and `canister_state_size > 0` before accepting the deserialized state.

---

### Proof of Concept

**Current layout written by version N:**
```
offset 1_000_000: [wasms_end_offset: u32 big-endian] [4 bytes padding] [canister_state_size: u32 big-endian]
```

**Hypothetical version N+1 that "fixes" the stride to match the actual field width:**
```rust
fn get_canister_state_size_offset(&self) -> u64 {
    self.get_wasms_end_offset_offset() + (size_of::<u32>() as u64) // "fix"
}
```

After upgrade, `read_canister_state_size()` reads from offset `1_000_004` instead of `1_000_008`. It reads the 4 bytes of silent padding (all zeros) as the canister state size, yielding `0`. `read_canister_state()` then allocates a zero-length buffer and decodes an empty `StableCanisterState`, wiping the entire `wasm_indexes` map and `upgrade_path` from the canister's in-memory state. All subsequent `get_wasm` and `deploy_new_sns` calls fail or return empty results. No error is surfaced during `post_upgrade`. [9](#0-8) [10](#0-9)

//! indexes in stable memory. The SNS-WASM stable memory format looks something like:
//!
//! [reserved bytes][WASMs-end offset][Canister state size][WASM 1][WASM 2]...
//!
//! There are some initial reserved bytes that may be used in the future. There are then two
//! variables saved in stable memory: the WASMs-end offset and the canister state size. The

//! written bytes is written to "canister state size" in stable memory, so that in post-upgrade
//! SNS-WASM can use "WASMs-end offset" and "canister state size" to read canister state from
//! stable memory.

    pub fn read_canister_state(&self) -> Result<StableCanisterState, StableMemoryError> {
        let wasms_end_offset = self.read_wasms_end_offset()?;
        let canister_state_size = self.read_canister_state_size()?;
