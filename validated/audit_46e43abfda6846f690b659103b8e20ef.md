### Title
`EvmErc20V2.withdrawToNear()` Encodes Extra 20-Byte Sender Field Causing Silent Fund Loss in `ExitToNear` Precompile - (File: `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.withdrawToNear()` unconditionally encodes the caller's address (20 bytes) between the flag byte and the amount field. The `ExitToNear` precompile, when compiled **without** the `error_refund` feature, does not expect this extra field. The result is a byte-offset mismatch structurally identical to the reported vulnerability: the precompile reads `sender_bytes ++ amount_bytes[0..12]` as the 32-byte amount, which always exceeds `u128::MAX`, causing `ERR_INVALID_AMOUNT`. The precompile call fails silently (return value unchecked in assembly), but `_burn()` has already executed, permanently destroying the user's tokens with no corresponding NEP-141 credit on NEAR.

---

### Finding Description

**`EvmErc20V2.withdrawToNear()` always encodes:**

```
[0x01][sender(20 bytes)][amount_b(32 bytes)][recipient(N bytes)]
``` [1](#0-0) 

**The `ExitToNear` precompile (without `error_refund`) expects:**

```
[0x01][amount(32 bytes)][recipient(N bytes)]
``` [2](#0-1) 

After stripping the flag byte, the precompile reads `input[..32]` as the amount: [3](#0-2) 

With `EvmErc20V2`'s layout, `input[..32]` is `sender[0..20] ++ amount_bytes[0..12]`. Since any non-zero sender address sets the high bits of this 32-byte value, the parsed `U256` always exceeds `u128::MAX`, triggering: [4](#0-3) 

The precompile returns `ERR_INVALID_AMOUNT`. The assembly `call` return value is **not checked**: [5](#0-4) 

The EVM transaction completes successfully. `_burn()` at line 55 has already destroyed the user's ERC-20 tokens. No NEP-141 tokens are credited on NEAR. [6](#0-5) 

**Contrast with `EvmErc20` (V1)**, which does not include the sender field and is compatible with both engine configurations: [7](#0-6) 

The `error_refund` compile-time feature controls which layout the precompile expects. When enabled, `parse_input` extracts 20 bytes at `input[1..21]` as `refund_address` and the V2 layout is correct. When disabled (the default, `MIN_INPUT_SIZE = 3`), those 20 bytes are not consumed, corrupting all subsequent field reads: [8](#0-7) 

---

### Impact Explanation

Any user calling `withdrawToNear()` on an `EvmErc20V2`-deployed token on an engine compiled without `error_refund` will:
1. Have their ERC-20 tokens burned (irreversible).
2. Receive no NEP-141 tokens on NEAR (precompile fails silently).
3. Suffer **permanent loss of bridged funds** with no recovery path.

This matches **Critical: Permanent freezing/loss of funds**.

---

### Likelihood Explanation

`EvmErc20V2` is a production contract in the repository. Any deployment of `EvmErc20V2` on an Aurora Engine instance compiled without `error_refund` (the default `MIN_INPUT_SIZE = 3` path) exposes every user of that token to total loss on withdrawal. The trigger is a normal, unprivileged user action (`withdrawToNear`). No admin compromise or special condition is required beyond the deployment configuration mismatch.

---

### Recommendation

1. Add an explicit revert check on the precompile `call` return value in `EvmErc20V2.withdrawToNear()` so that if the precompile fails, the entire transaction reverts (protecting the burned tokens).
2. Guard `EvmErc20V2` deployment with a runtime check or constructor assertion that verifies the engine supports the V2 input format.
3. Alternatively, unify the input format: always include or always omit the sender field, and update both the Solidity contracts and the Rust precompile parser to agree unconditionally.

---

### Proof of Concept

With engine compiled **without** `error_refund`:

```
User calls EvmErc20V2.withdrawToNear("alice.near", 1000)
  sender = 0xABCD...1234 (20 bytes, non-zero)

Input sent to ExitToNear precompile:
  [0x01]                          ← flag
  [AB CD ... 12 34]               ← sender (20 bytes) — UNEXPECTED
  [00 00 ... 03 E8]               ← amount = 1000 (32 bytes)
  [61 6C 69 63 65 2E 6E 65 61 72] ← "alice.near"

Precompile (no error_refund) reads input[1..33] as amount:
  [AB CD ... 12 34 00 00 ... 00 00 00 00 00 00 00 00 00 00 00 00]
  = 0xABCD...1234_000000000000000000000000 >> u128::MAX → ERR_INVALID_AMOUNT

call() returns 0. Assembly does not check. _burn() already executed.
1000 tokens permanently destroyed. alice.near receives nothing.
```
