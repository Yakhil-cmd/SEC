Audit Report

## Title
Permanent ckBTC Loss on `AmountTooLow` Finalization Due to Fee Spike Between Request Submission and Batch Processing — (`rs/bitcoin/ckbtc/minter/src/lib.rs`, `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

The ckBTC minter irrevocably burns the full user-specified ckBTC amount at `retrieve_btc`/`retrieve_btc_with_approval` call time, but deducts the actual Bitcoin network fee only at batch-processing time using the **then-current** fee rate. When a fee spike causes `fee + minter_fee > amount` at processing time, the minter finalizes the request as `FinalizedStatus::AmountTooLow` without issuing any ckBTC reimbursement — the burned ckBTC is permanently destroyed with no BTC sent and no refund. This is a concrete, non-recoverable loss of user funds in the ckBTC chain-fusion system.

## Finding Description

**Step 1 — ckBTC burned immediately and irrevocably at request time.**

In `retrieve_btc` and `retrieve_btc_with_approval`, `burn_ckbtcs` / `burn_ckbtcs_icrc2` is called before the request is enqueued. The `RetrieveBtcRequest` stored in `pending_retrieve_btc_requests` carries only `amount`, `address`, `block_index`, `received_at`, `kyt_provider`, and `reimbursement_account` — no `max_fee` or `min_receive_amount` field exists. [1](#0-0) [2](#0-1) 

**Step 2 — Fee determined at batch-processing time using the current rate.**

`submit_pending_requests` calls `estimate_fee_per_vbyte` at the moment the batch fires, then passes that live rate to `build_unsigned_transaction`. Requests can sit in the queue until `can_form_a_batch` returns true — either when `MIN_PENDING_REQUESTS` accumulates or when `max_time_in_queue_nanos` elapses for the oldest request, creating a window of hours. [3](#0-2) [4](#0-3) 

**Step 3 — `AmountTooLow` path finalizes with no reimbursement.**

Inside `build_unsigned_transaction_from_inputs`, if `fee + minter_fee > amount` the function returns `BuildTxError::AmountTooLow`. The caller in `submit_pending_requests` handles this by calling `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` for every request in the batch — no reimbursement is scheduled. [5](#0-4) [6](#0-5) [7](#0-6) 

**Contrast with `InvalidTransaction` path — which does reimburse.**

The `BuildTxError::InvalidTransaction` branch explicitly calls `reimburse_canceled_requests`, demonstrating the infrastructure for reimbursement exists and is intentionally omitted for `AmountTooLow`. [8](#0-7) 

**No slippage parameter in the API.**

`RetrieveBtcArgs` and `RetrieveBtcWithApprovalArgs` accept only `address` and `amount`; no `max_fee` or `min_receive_amount` field is present. `estimate_withdrawal_fee` is advisory only. [9](#0-8) 

## Impact Explanation

When Bitcoin network fees spike between request submission and batch processing, any user whose withdrawal amount is near the fee-based minimum can have their entire ckBTC balance burned with zero BTC received and zero ckBTC reimbursed. The burned ckBTC is permanently removed from the ledger supply — a direct, non-recoverable loss of chain-key ledger assets. This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."** The loss per affected user is bounded by their withdrawal amount (up to the near-minimum range where `AmountTooLow` is reachable), but the mechanism is repeatable across any number of users during congestion events.

## Likelihood Explanation

Bitcoin transaction fees are well-documented to spike 10–100× during congestion events (Ordinals waves, halving periods). The `max_time_in_queue_nanos` parameter is configurable and used in production, meaning requests legitimately sit pending for hours. Any unprivileged user submitting a near-minimum `retrieve_btc` request during a low-fee period who has their request processed during a high-fee period is exposed. No special privileges, social engineering, or external compromise is required — the user simply calls a public endpoint. The scenario is realistic and recurring on Bitcoin mainnet.

## Recommendation

1. Add an optional `max_fee_per_vbyte: opt nat64` (or `min_receive_amount: opt nat64`) field to `RetrieveBtcArgs`, `RetrieveBtcWithApprovalArgs`, and `RetrieveBtcRequest`.
2. At batch-processing time in `build_unsigned_transaction`, if the computed fee exceeds the user's specified maximum, return a new `BuildTxError::FeeTooHigh` variant instead of `AmountTooLow`.
3. Handle `FeeTooHigh` by rescheduling the request to the end of the pending queue (mirroring the ckETH minter's `reschedule_withdrawal_request` pattern for `InsufficientTransactionFee`) rather than finalizing it without reimbursement.
4. For the existing `AmountTooLow` path (where the amount is structurally too small regardless of fee rate), issue a ckBTC reimbursement to `reimbursement_account` before finalizing, consistent with how `InvalidTransaction` is handled.

## Proof of Concept

1. Bitcoin mainnet fee rate is 5 sat/vbyte. User calls `retrieve_btc_with_approval(address="bc1q...", amount=100_000)`. The full 100,000 satoshi of ckBTC is burned immediately via `burn_ckbtcs_icrc2`.
2. The `RetrieveBtcRequest` enters `pending_retrieve_btc_requests` with `received_at = T`.
3. No other requests arrive. The minter waits until `max_time_in_queue_nanos` elapses (hours).
4. During this window, a Bitcoin congestion event occurs; fee rate rises to 800 sat/vbyte.
5. `submit_pending_requests` fires; `estimate_fee_per_vbyte` returns 800 sat/vbyte.
6. `build_unsigned_transaction` computes `fee ≈ 800 * ~140 vbytes = 112,000 satoshi`.
7. `112,000 + minter_fee > 100,000` → `BuildTxError::AmountTooLow` is returned.
8. `remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)` is called — no reimbursement.
9. The user has lost 100,000 satoshi of ckBTC: the ledger burn is permanent, no BTC was sent, and no ckBTC was reminted.

A deterministic integration test can reproduce this by: (a) initializing a PocketIC instance with the ckBTC minter and ledger, (b) minting ckBTC to a test principal, (c) calling `retrieve_btc_with_approval` with a near-minimum amount, (d) advancing the mock fee estimator to return a fee exceeding the withdrawal amount, (e) triggering `submit_pending_requests`, and (f) asserting that `retrieve_btc_status` returns `AmountTooLow` while the ledger total supply has decreased by the burned amount with no compensating mint.

            && self.max_time_in_queue_nanos < now.saturating_sub(req.received_at)

        {
            return true;
        }

        false
    }

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
