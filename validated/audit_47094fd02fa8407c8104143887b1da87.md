Audit Report

## Title
Unsanitized UTF-8 Memo Bytes Enable Markdown Injection in ICRC-21 `GenericDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

`add_memo` in `ConsentMessage` inserts valid UTF-8 memo bytes verbatim into the `GenericDisplayMessage` markdown string wrapped only in single backticks. A memo whose first byte is a backtick (0x60) and that does not end with a backtick causes the opening double-backtick sequence to fail to form a CommonMark code span, leaving the remainder of the memo rendered as raw markdown. A malicious dapp can exploit this to inject visually identical bold headers (`**Amount:**`, `**To:**`) into the consent message shown to a user before they sign a transaction.

## Finding Description

`add_memo` at `packages/icrc-ledger-types/src/icrc21/responses.rs` lines 284–315 performs a single UTF-8 validity check and then inserts the memo string verbatim:

```rust
// L288-291
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),   // safe path
};
// L293-294
ConsentMessage::GenericDisplayMessage(message) => {
    message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
}
```

Non-UTF-8 bytes are hex-encoded (safe). Valid UTF-8 is inserted without any markdown escaping.

**CommonMark injection mechanics.** The format string produces `` `{memo_str}` ``. If `memo_str` begins with a backtick but does **not** end with one (and contains no internal ` `` ` sequence), the resulting string starts with ` `` ` (two backticks). Under CommonMark rules, a two-backtick code-span opener that finds no matching ` `` ` closer is rendered as two literal backtick characters; everything following it is processed as markdown. For example, with:

```
memo_str = "`\n\n**Amount:** 0.00001 ICP\n\n**To:**\nlegitimate_account"
```

the formatted segment becomes:

```
``\n\n**Amount:** 0.00001 ICP\n\n**To:**\nlegitimate_account`
```

No ` `` ` closer exists, so the opener degrades to literal ` `` `, and `**Amount:**` / `**To:**` are rendered as bold markdown headers — visually identical to the legitimate fields already written by `build()`.

The memo is appended **after** the legitimate fields (amount, recipient, fee) are already written, so the full `GenericDisplayMessage` contains both the correct fields and the attacker-injected duplicate fields.

The endpoint is a public `#[update]` callable by any principal (`rs/ledger_suite/icrc1/ledger/src/main.rs` lines 1189–1207). The memo flows from `TransferArg.memo` → `GenericMemo::Icrc1Memo` → `add_memo` with no sanitization step (`packages/icrc-ledger-types/src/icrc21/lib.rs` lines 392–395, 290–292).

**Note on the submitted PoC:** The specific byte sequence in the report ends with a backtick, causing the opening ` `` ` to match the closing ` `` ` (last memo byte + format-string trailing backtick), which places the entire content inside a code span — the injection does **not** fire as described. A corrected PoC (memo without a trailing backtick) triggers the injection trivially, as shown above.

## Impact Explanation

A malicious dapp constructs a large-value transfer to the attacker's account and supplies a memo whose UTF-8 content begins with a backtick and injects false `**Amount:**` / `**To:**` sections. Any wallet that renders `GenericDisplayMessage` as markdown (the intended use per ICRC-21) will display the injected fields as visually indistinguishable from the legitimate ones. A user who skims the message or whose wallet truncates it may approve a transaction whose real parameters differ from what they believe they are signing. This constitutes concrete user-funds harm on an in-scope ICRC ledger endpoint.

Severity: **Medium–High**. The correct fields always appear before the memo section, so a careful top-to-bottom reader sees the real values first; the deception relies on the user skimming or the wallet truncating. This constrains practical severity below the highest tier but does not eliminate it, as the ICRC-21 consent message is the primary security gate for user approval.

## Likelihood Explanation

- The endpoint is publicly callable with no authentication.
- Crafting a triggering memo requires a single leading backtick byte (0x60) and no trailing backtick — trivial to construct.
- The ICRC-21 standard explicitly intends `GenericDisplayMessage` to be rendered as markdown; compliant wallets are the target.
- `FieldsDisplayMessage` is unaffected — the memo is stored as a structured `Value::Text` field and never concatenated into a markdown string.

## Recommendation

Before inserting `memo_str` into the `GenericDisplayMessage` branch, escape all markdown-significant characters. The simplest safe approach is to unconditionally hex-encode the memo in the `GenericDisplayMessage` branch (matching the existing non-UTF-8 path), or to escape backticks and other markdown metacharacters (`*`, `_`, `#`, `[`, `]`) before interpolation. The `FieldsDisplayMessage` branch requires no change.

## Proof of Concept

```rust
// Corrected PoC: memo starts with backtick, does NOT end with backtick,
// contains no internal `` sequence.
let malicious_memo = b"`\n\n**Amount:** 0.00001 ICP\n\n**To:**\nlegitimate_account";
let transfer_arg = TransferArg {
    memo: Some(Memo(ByteBuf::from(malicious_memo.to_vec()))),
    amount: Nat::from(1_000_000_000_000_u64), // 10,000 ICP
    to: attacker_account,
    fee: None,
    from_subaccount: None,
    created_at_time: None,
};
// Call icrc21_canister_call_consent_message with method="icrc1_transfer"
// and device_spec=GenericDisplay.
// The resulting GenericDisplayMessage will contain:
//   - correct fields: 10,000 ICP → attacker (rendered first)
//   - injected fields: **Amount:** 0.00001 ICP / **To:** legitimate_account
//     (rendered as bold markdown, visually identical to the legitimate fields)
```

A deterministic unit test in `packages/icrc-ledger-types` can verify this by calling `add_memo` with the above bytes on a `ConsentMessage::GenericDisplayMessage`, asserting that the resulting string contains unescaped `**Amount:**` outside any code span.
