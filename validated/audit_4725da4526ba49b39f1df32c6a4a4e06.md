### Title
`BN254_PAIRING_PER_PAIR_NATIVE_COST` Incorrectly Set Equal to `BN254_PAIRING_BASE_NATIVE_COST` - (`basic_system/src/cost_constants.rs`)

---

### Summary

`BN254_PAIRING_PER_PAIR_NATIVE_COST` is hardcoded to the same value as `BN254_PAIRING_BASE_NATIVE_COST`, making the per-pair incremental native (proving) resource cost identical to the fixed base cost. This causes multi-pair BN254 pairing calls to consume disproportionately large native resources, causing valid transactions to be rejected with out-of-native-resources errors.

---

### Finding Description

In `basic_system/src/cost_constants.rs`, the native resource costs for BN254 pairing are defined as:

```rust
pub const BN254_PAIRING_BASE_NATIVE_COST: u64 = native_with_delegations!(13_000_000, 500_000, 0);
pub const BN254_PAIRING_PER_PAIR_NATIVE_COST: u64 = BN254_PAIRING_BASE_NATIVE_COST;
``` [1](#0-0) 

The per-pair cost is not independently calibrated — it is simply aliased to the base cost. This is consumed in `bn254_pairing_check.rs`:

```rust
let native_cost = (num_pairs as u64) * BN254_PAIRING_PER_PAIR_NATIVE_COST
    + BN254_PAIRING_BASE_NATIVE_COST;
``` [2](#0-1) 

This means the total native cost for N pairs is `(N + 1) * 13_000_000` native units. Compare this to the EVM gas model, where the base cost is 45,000 gas and the per-pair cost is 34,000 gas — two distinct values. The native cost model should mirror this structure with independently calibrated constants. All other precompile pairs in the codebase (e.g., `BN254_ECADD_NATIVE_COST`, `BN254_ECMUL_NATIVE_COST`) have a single flat cost, but the pairing precompile is the only one with a two-part cost structure, making the aliasing error uniquely impactful here. [3](#0-2) 

---

### Impact Explanation

For a 2-pair pairing call, the native cost is `3 × 13_000_000 = 39_000_000` native units. For a 10-pair call, it is `11 × 13_000_000 = 143_000_000` native units. Since the native resource budget per transaction is finite, any transaction that calls the BN254 pairing precompile (address `0x0000000000000000000000000000000000000008`) with more than a small number of pairs will be rejected with out-of-native-resources even if the user provides sufficient EVM gas. This is a **resource accounting bug** causing a **state-transition divergence**: the forward system and proving system both reject transactions that Ethereum mainnet would accept, breaking EVM equivalence for a standard precompile.

---

### Likelihood Explanation

The BN254 pairing precompile is widely used by ZK proof verifiers (e.g., Groth16, PLONK verifiers) deployed on-chain. These contracts routinely call the precompile with 2–8 pairs. Any such contract deployed on ZKsync OS would fail at runtime for multi-pair inputs, making this a high-likelihood issue for any realistic DeFi or ZK-application workload.

---

### Recommendation

Define `BN254_PAIRING_PER_PAIR_NATIVE_COST` as an independently benchmarked value representing the incremental proving cost of one additional pairing, separate from the fixed base overhead:

```rust
pub const BN254_PAIRING_BASE_NATIVE_COST: u64 = native_with_delegations!(13_000_000, 500_000, 0);
// Should be independently benchmarked, not aliased to the base cost:
pub const BN254_PAIRING_PER_PAIR_NATIVE_COST: u64 = native_with_delegations!(<per_pair_cycles>, <per_pair_delegations>, 0);
``` [1](#0-0) 

---

### Proof of Concept

1. Deploy a Groth16 verifier contract on ZKsync OS that calls `ecPairing` (address `0x8`) with 2 pairs.
2. Submit a transaction calling the verifier with sufficient EVM gas (e.g., 200,000 gas, which covers `45,000 + 2×34,000 = 113,000` EVM gas).
3. Observe: the transaction fails with out-of-native-resources because the native cost charged is `3 × 13,000,000 = 39,000,000` native units — far exceeding the native budget derived from 200,000 gas.
4. On Ethereum mainnet, the same transaction succeeds.
