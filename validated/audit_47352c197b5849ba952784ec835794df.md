Audit Report

## Title
Unbounded Ingress Window Causes Uncapped Memory Allocation in `/construction/payloads` - (File: `rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function accepts attacker-controlled `ingress_start` and `ingress_end` values with no upper bound on the window size. The while loop at lines 163–167 of `services.rs` iterates once per ~150-second step across the entire window, and `handle_construction_payloads` allocates one `EnvelopeContent::Call` with a full clone of `canister_method_args` per iteration. A single unauthenticated HTTP POST with a multi-year window can exhaust server memory and crash the ICRC1 Rosetta sidecar process.

## Finding Description
In `construction_payloads` (`services.rs` lines 148–167), the only validations on the ingress window are:

- `ingress_start >= ingress_end` → reject [1](#0-0) 
- `ingress_end < now + ingress_interval` → reject (rejects windows that are too *small*, not too *large*) [2](#0-1) 

There is no maximum window size check. The while loop then runs unboundedly: [3](#0-2) 

With `ingress_interval = MAX_INGRESS_TTL − PERMITTED_DRIFT = 270 s` and `INGRESS_INTERVAL_OVERLAP = 120 s`, the step size is 150 seconds. A 10-year window produces ~2,102,400 iterations.

For each iteration, `handle_construction_payloads` allocates one `EnvelopeContent::Call` and clones `canister_method_args`: [4](#0-3) 

The resulting `UnsignedTransaction` is then CBOR-serialized and hex-encoded (doubling byte size) before being returned in the HTTP response: [5](#0-4) 

The endpoint handler passes `ingress_start`/`ingress_end` directly from the JSON body with no additional guards: [6](#0-5) 

`ConstructionPayloadsRequestMetadata` accepts both fields as plain `Option<u64>` with no domain-level constraints at deserialization: [7](#0-6) 

## Impact Explanation
An unprivileged attacker can crash the ICRC1 Rosetta sidecar process by exhausting its heap memory, taking down the Rosetta API service entirely. This matches the allowed bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or significant Rosetta infrastructure security impact with concrete user or protocol harm.** The Rosetta API is an explicitly in-scope financial integration component. The crash disrupts all Rosetta-dependent exchange integrations and transfer workflows for the affected ICRC1 ledger.

## Likelihood Explanation
The endpoint requires no authentication. The request payload is trivially small (a single JSON object). The attacker needs only a network path to the Rosetta HTTP port. The attack is repeatable: after the process restarts, a new request immediately re-triggers the crash. No special privileges, leaked keys, or victim interaction are required.

## Recommendation
Add an explicit upper bound on the ingress window before the while loop in `construction_payloads` (`services.rs`), consistent with the documented 24-hour design intent stated in `types.rs` line 44:

```rust
const MAX_INGRESS_WINDOW: u64 = 24 * 3600 * 1_000_000_000; // 24 hours in nanoseconds
if ingress_end - ingress_start > MAX_INGRESS_WINDOW {
    return Err(Error::processing_construction_failed(
        "Ingress window exceeds maximum allowed 24-hour range"
    ));
}
```

This caps the vector at ~576 entries and matches the stated design intent. [8](#0-7) 

## Proof of Concept
```bash
# Set ingress_start = current time in nanoseconds
# Set ingress_end = ingress_start + 10 years (~3.15e17 ns)
NOW_NS=$(date +%s)000000000
END_NS=$((NOW_NS + 315360000000000000))

curl -X POST http://<rosetta-host>:8081/construction/payloads \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\":\"Internet Computer\",\"network\":\"<ledger_canister_id>\"},
    \"operations\": [/* valid ICRC1 transfer ops */],
    \"public_keys\": [{\"hex_bytes\":\"<ed25519_pubkey>\",\"curve_type\":\"edwards25519\"}],
    \"metadata\": {
      \"ingress_start\": $NOW_NS,
      \"ingress_end\":   $END_NS
    }
  }"
```

With a 10-year window, the while loop runs ~2,102,400 iterations, each cloning the transfer args and constructing an `EnvelopeContent`. The CBOR+hex-encoded response will be several gigabytes, exhausting server memory before the response is written and crashing the Rosetta process.
