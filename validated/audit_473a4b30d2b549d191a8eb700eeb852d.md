Audit Report

## Title
Ingress Pool Admission DoS via Per-Message vs. Cumulative Cycles Check Mismatch — (`rs/execution_environment/src/execution_environment.rs`, `rs/ingress_manager/src/ingress_selector.rs`)

## Summary
The ingress admission gate checks only the single-message induction cost, while the proposal-time gate checks the cumulative cost across all messages from the same canister in a block. Messages that pass admission but fail the cumulative check at proposal time are never evicted from the validated pool — they are only popped from a local in-memory queue. An attacker with a minimally-funded canister can flood the validated pool with zombie messages, saturating it until TTL expiry and causing a 503 DoS for all legitimate ingress submissions.

## Finding Description
**Admission-time check** (`should_accept_ingress_message`, `rs/execution_environment/src/execution_environment.rs` L3340–3373): The code explicitly labels itself a "first-pass" check and calls `can_withdraw_cycles_with_threshold` with only the cost of the single message being submitted. The same canister state is read for every concurrent submission, so N rapid submissions each independently pass against the same unchanged balance. [1](#0-0) 

**Pool-to-validated transition** (`validate_ingress_pool_object`, `rs/ingress_manager/src/ingress_handler.rs` L167–206): When messages move from unvalidated to validated, only size, known-status, and signature are checked. No cycles check is performed. [2](#0-1) 

**Proposal-time check** (`validate_ingress`, `rs/ingress_manager/src/ingress_selector.rs` L566–584): At block-building time the check is cumulative — `*cumulative_ingress_cost + ingress_cost` is tested against the canister's balance. For a canister funded for exactly one message, message 1 passes and messages 2–N all fail with `InsufficientCycles`. [3](#0-2) 

**No eviction of failing messages** (`get_ingress_payload`, `rs/ingress_manager/src/ingress_selector.rs` L204–207): When `validate_ingress` returns any error other than `IngressPayloadTooManyMessages`, the message is only popped from the local `queue.msgs` vector for that round. No `RemoveFromValidated` change action is emitted (confirmed: zero occurrences of `RemoveFromValidated` in `ingress_selector.rs`). The message remains in the validated pool. [4](#0-3) 

**Validated pool cleanup paths** (`on_state_change`, `rs/ingress_manager/src/ingress_handler.rs` L112–137): The only eviction paths for validated messages are: `IngressHistoryReader` returning non-Unknown status, finalization-triggered purge, and TTL expiry. Messages that fail cycles validation are never executed (no non-Unknown status), never included in a finalized block, and thus persist until TTL expiry. [5](#0-4) 

**Pool capacity enforcement and HTTP 503** (`rs/artifact_pool/src/ingress_pool.rs` L226–232, `rs/http_endpoints/public/src/call.rs` L229–236): Once `exceeds_limit` returns true for a peer, `exceeds_threshold` at the HTTP endpoint causes all new ingress submissions to be rejected with `SERVICE_UNAVAILABLE`. [6](#0-5) [7](#0-6) 

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The attack saturates the validated ingress pool with structurally valid but economically unexecutable messages, causing all legitimate users on the targeted subnet to receive 503 errors for up to 5 minutes per attack wave. The attack is repeatable indefinitely and affects the entire subnet's ingress availability, not just a single user.

## Likelihood Explanation
The attack requires only an unprivileged user with a canister holding a minimal cycles balance (enough for one induction cost — a negligible amount proportional to message size). No special privileges, governance access, or threshold corruption is needed. The attacker submits messages via the standard HTTPS `/api/v2/canister/{id}/call` endpoint. The structural mismatch is present in production code with no feature flag gating. The attack can be repeated every TTL window (~5 minutes) to maintain continuous DoS, and can be amplified by using multiple canisters and multiple boundary nodes.

## Recommendation
1. **Evict on cycles failure**: In `get_ingress_payload` (`ingress_selector.rs`), when `validate_ingress` returns `InsufficientCycles`, emit a `RemoveFromValidated` change action so the message is purged from the pool rather than re-selected every round.
2. **Tighten admission gate**: At admission time, check whether the canister's available balance (above freeze threshold) exceeds the cost of at least one message before admitting additional messages from the same canister, or track per-canister in-flight message count at the HTTP layer.
3. **Per-canister pool admission cap**: Limit the number of messages from a single canister that can reside in the validated pool at any time, proportional to the canister's available cycles divided by per-message induction cost.

## Proof of Concept
1. Create canister `C` with cycles balance = `freeze_threshold + ingress_induction_cost(msg)` (sufficient for exactly one message).
2. Submit `N` messages targeting `C` via `/api/v2/canister/{id}/call` in rapid succession. Each call independently invokes `should_accept_ingress_message`, which reads the same unchanged canister state and passes the per-message cost check for all N messages.
3. All N messages enter the unvalidated pool and are promoted to validated by `validate_ingress_pool_object` (no cycles check performed).
4. On the next `get_ingress_payload` call: message 1 passes (`cumulative = cost ≤ balance − threshold`); message 2 fails (`cumulative = 2×cost > balance − threshold`); messages 3–N fail similarly. Each failing message is popped from the local queue but remains in the validated pool.
5. Repeat with additional minimally-funded canisters until `ingress_pool.exceeds_limit()` returns `true`.
6. All subsequent legitimate ingress submissions receive HTTP 503 `Service is overloaded, try again later.` until the attacker's messages expire after `MAX_INGRESS_TTL`.
7. A deterministic integration test can verify this by: (a) creating a canister with minimal cycles, (b) inserting N messages into the pool directly, (c) calling `get_ingress_payload` and asserting only 1 message is selected, (d) asserting the pool still contains N messages, and (e) asserting `exceeds_limit` returns true.
