Audit Report

## Title
ICP Burn Silently Ignored After Successful Cycles Minting, Breaking Conservation Invariant - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows all errors from the ICP ledger burn that must follow a successful cycles mint or canister creation. If the burn call fails, cycles have already been irreversibly deposited to the caller, the ICP remains unburned in the CMC's subaccount, and the notification is permanently finalized — preventing any retry or recovery. This breaks the ICP/cycles conservation invariant: cycles are created without the corresponding ICP being destroyed.

## Finding Description
Three processing functions call `burn_and_log` **after** the irreversible action has already succeeded:

- `process_top_up` [1](#0-0)  calls `deposit_cycles`, then `burn_and_log`
- `process_mint_cycles` [2](#0-1)  calls `do_mint_cycles`, then `burn_and_log`
- `process_create_canister` [3](#0-2)  calls `do_create_canister`, then `burn_and_log`

The `burn_and_log` function is explicitly annotated: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* [4](#0-3)  It catches all ledger call errors and returns `()` regardless of outcome. [5](#0-4) 

After `burn_and_log` returns, the caller stores a terminal `NotificationStatus` (`NotifiedTopUp`, `NotifiedMint`, or `NotifiedCreateCanister`), permanently closing the notification. [6](#0-5)  Any subsequent retry attempt is rejected with `InvalidTransaction`. [7](#0-6) 

The existing `is_transient_error` guard only removes the notification entry when the **cycles deposit itself** fails with a transient error — it does not apply when the deposit succeeds but the burn fails, because `burn_and_log` always returns `Ok(())` to its caller. [8](#0-7) 

The burn silently no-ops in three additional cases: `minting_account_id` is `None`, `amount < DEFAULT_TRANSFER_FEE`, or the `send_pb` call returns any `Err`. [9](#0-8) 

## Impact Explanation
When `burn_and_log` fails silently, cycles are minted without the corresponding ICP being destroyed. The ICP remains stranded in the CMC's subaccount keyed by the user's principal/canister ID, the notification is permanently closed, and no recovery path exists. This constitutes **illegal minting** of cycles — a direct violation of the ICP/cycles conservation invariant that underpins the economic model of the protocol. This matches the High impact category: *"Significant NNS/SNS or infrastructure security impact with concrete user or protocol harm"* — the CMC is a core NNS canister and the ICP↔cycles peg is a fundamental protocol invariant.

## Likelihood Explanation
The ICP ledger is a production canister on the NNS subnet. Transient inter-canister call failures (subnet reject codes, ledger mid-upgrade, message queue full) are realistic and have occurred historically on mainnet. No privileged access, key compromise, or majority attack is required. An unprivileged user who calls `notify_top_up` or `notify_mint_cycles` during a natural ledger upgrade window will receive cycles while the ICP burn is silently dropped. The attacker cannot directly force ledger unavailability, but the condition arises naturally and periodically, making this opportunistically exploitable without any special capability.

## Recommendation
1. **Do not silently discard burn failures.** If `burn_and_log` fails, the notification should remain in `Processing` state (allowing retry) or be stored with a distinct `BurnFailed` status that a privileged operator can later resolve.
2. **Separate finalization from burn outcome.** Only finalize the notification as `NotifiedTopUp`/`NotifiedMint`/`NotifiedCreateCanister` after the burn is confirmed successful.
3. **Add an invariant monitor** (e.g., in `heartbeat`) that compares the sum of all CMC subaccount balances against expected zero, alerting on any residual ICP that was not burned.

## Proof of Concept
1. User sends `N` ICP to the CMC's subaccount (keyed by their canister ID) with memo `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id }`.
3. CMC sets `blocks_notified[block_index] = Processing` and calls `process_top_up`.
4. `deposit_cycles` succeeds — the target canister receives cycles.
5. `burn_and_log` is called. The ICP ledger is transiently unavailable (e.g., mid-upgrade). The `call_protobuf` call returns `Err(...)`. `burn_and_log` logs the error and returns `()`.
6. `process_top_up` returns `Ok(cycles)`. The CMC stores `NotifiedTopUp(Ok(cycles))`.
7. Any retry of `notify_top_up` is rejected with `InvalidTransaction("already processed as a top up request")`.
8. The user's canister has received cycles. The `N` ICP remains unburned in the CMC's subaccount. The conservation invariant is broken: `N * rate` cycles exist without the corresponding `N` ICP being destroyed.

A deterministic integration test can reproduce this by deploying a mock ICP ledger that returns a transient error on `send_pb`, then calling `notify_top_up` and verifying that (a) cycles were deposited, (b) the notification is permanently finalized, and (c) the CMC subaccount balance is non-zero.
