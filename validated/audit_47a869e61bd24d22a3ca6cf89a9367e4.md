Audit Report

## Title
Single Unprivileged Participant Can Permanently Abort SNS Swap via Early-Abort Trigger Under Permissible Parameter Configuration - (File: `rs/sns/swap/src/swap.rs`)

## Summary
When an SNS swap is configured with `max_participant_icp_e8s == max_direct_participation_icp_e8s` and `min_participants > 1` — a configuration explicitly permitted by the parameter validation — a single unprivileged participant can fill the entire swap capacity in one `refresh_buyer_token_e8s` call. The next heartbeat then triggers the early-abort path because `icp_target_reached && !sufficient_participation`, permanently killing the SNS launch at negligible cost to the attacker.

## Finding Description

**Abort condition** (`can_abort`, `rs/sns/swap/src/swap.rs` L2906–2913):
```rust
(self.swap_due(now_seconds) || self.icp_target_progress().is_reached_or_exceeded())
    && !self.sufficient_participation()
```
`icp_target_progress().is_reached_or_exceeded()` becomes true when `current_direct_participation_e8s >= max_direct_participation_e8s` (L2844–2853). `sufficient_participation()` (L2796–2797) requires both `min_participation_reached()` — i.e., `buyers.len() >= min_participants` (L2817) — and `min_direct_participation_icp_e8s_reached()`. With one buyer and `min_participants = 2`, `sufficient_participation()` is `false`.

**Participant cap in `refresh_buyer_token_e8s`** (L1177, L1224, L1237):
```rust
let max_increment_e8s = self.available_direct_participation_e8s();
let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);
let new_balance_e8s = std::cmp::min(new_balance_e8s, max_participant_icp_e8s);
```
When `max_participant_icp_e8s == max_direct_participation_icp_e8s`, the per-participant cap equals the total swap capacity, so a single call can set `current_direct_participation_e8s = max_direct_participation_e8s`.

**Validation gap** (`validate_participation_constraints`, `rs/sns/init/src/lib.rs`):
- Check (4) at L1603–1607 only enforces `max_participant_icp_e8s <= max_direct_participation_icp_e8s`, explicitly allowing equality.
- Check (6) at L1618–1624 enforces `max_direct_participation_icp_e8s >= min_participants * min_participant_icp_e8s`, but this uses `min_participant_icp_e8s`, not `max_participant_icp_e8s`.
- **No check** enforces `max_participant_icp_e8s * min_participants <= max_direct_participation_icp_e8s`.

The proto field comment (`swap.proto` L596–604) explicitly documents that `max_participant_icp_e8s` "can effectively be disabled by setting it to `max_icp_e8s`," confirming this is a supported configuration.

**Existing test confirms the behavior** (`rs/sns/swap/src/swap.rs` L4672–4708): `test_try_commit_or_abort_insufficient_participation_with_max_icp` sets `min_participants=2`, `max_participant_icp_e8s=20`, `max_direct_participation_icp_e8s=20`, places one buyer at 20 e8s, and asserts `try_abort` returns `true` and lifecycle transitions to `Aborted`.

**Exploit flow:**
1. Attacker transfers `max_direct_participation_icp_e8s` ICP to their subaccount on the swap canister.
2. Attacker calls `refresh_buyer_token_e8s` — accepted because `available_direct_participation_e8s()` equals the full capacity and the per-participant cap equals the same value.
3. `current_direct_participation_e8s` now equals `max_direct_participation_e8s`; `icp_target_progress()` returns `Reached`.
4. Next heartbeat: `run_periodic_tasks` → `try_abort` → `can_abort` returns `true` (target reached, 1 buyer < `min_participants=2`) → lifecycle set to `Aborted`.
5. Attacker calls `finalize` → receives full ICP refund minus two transfer fees (~0.0002 ICP).

## Impact Explanation

The SNS launch is permanently aborted. The dapp's control is returned to `fallback_controller_principal_ids`. All legitimate participants are refunded but the SNS is never created. This constitutes a **High** severity impact: significant SNS infrastructure security impact with concrete, irreversible harm to the protocol and its users. The attacker recovers their ICP, making the attack essentially free and repeatable across any swap with the vulnerable parameter combination.

## Likelihood Explanation

The vulnerable configuration (`max_participant_icp_e8s == max_direct_participation_icp_e8s`, `min_participants > 1`) is explicitly documented as valid and is commonly used in practice to allow large ("whale") participation. `min_participants > 1` is standard for decentralization requirements. The attack requires no privileged access, no key material, no governance majority, and no coordination — only the ability to hold and transfer ICP. The swap canister is publicly callable on mainnet. The attacker's net cost is two ICP ledger transfer fees.

## Recommendation

Add a validation constraint in `validate_participation_constraints` (`rs/sns/init/src/lib.rs`) that enforces:

```rust
if max_participant_icp_e8s.saturating_mul(min_participants) > max_direct_participation_icp_e8s {
    return Err(format!(
        "Error: max_participant_icp_e8s ({max_participant_icp_e8s}) * min_participants \
         ({min_participants}) must be <= max_direct_participation_icp_e8s \
         ({max_direct_participation_icp_e8s})"
    ));
}
```

This ensures no single participant can fill the entire swap capacity when `min_participants > 1`. Alternatively, the `can_abort` logic in `rs/sns/swap/src/swap.rs` could be tightened to only trigger when it is provably impossible for the remaining capacity to accommodate `(min_participants - current_participants)` additional participants each contributing at least `min_participant_icp_e8s`.

## Proof of Concept

The existing test `test_try_commit_or_abort_insufficient_participation_with_max_icp` at `rs/sns/swap/src/swap.rs` L4672–4708 already constitutes a reproducible proof. To replicate the exact attack scenario:

```
Parameters:
  min_participants = 2
  min_participant_icp_e8s = 1_000_000_000   (10 ICP)
  max_participant_icp_e8s = 100_000_000_000  (1000 ICP)
  min_direct_participation_icp_e8s = 10_000_000_000 (100 ICP)
  max_direct_participation_icp_e8s = 100_000_000_000 (1000 ICP)

Steps:
1. Confirm validate_participation_constraints passes (all existing checks satisfied).
2. Attacker transfers 1000 ICP to swap_canister_subaccount(attacker_principal).
3. Attacker calls refresh_buyer_token_e8s → accepted, buyer balance set to 1000 ICP.
4. available_direct_participation_e8s() = 0; icp_target_progress() = Reached(100_000_000_000).
5. Call try_abort(now) → can_abort returns true (1 buyer < min_participants=2) → Aborted.
6. Attacker finalizes → receives 1000 ICP refund minus two transfer fees.
```
