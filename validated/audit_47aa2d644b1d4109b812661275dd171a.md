### Title
Unbounded Per-Round Loop in SNS Governance `distribute_rewards` Can Exhaust Instruction Limit - (File: rs/sns/governance/src/governance.rs)

### Summary
The SNS governance `distribute_rewards` function contains an unbounded `for i in 1..=new_rounds_count` loop that iterates once per missed reward round. When `round_duration_seconds` is configured to a small value and the canister experiences any period of downtime (e.g., subnet upgrade, canister stop), `new_rounds_count` can grow to millions of iterations, causing the canister to trap on instruction limit exhaustion. Once stuck, every subsequent timer-triggered attempt to distribute rewards also traps, permanently bricking the SNS reward distribution subsystem.

### Finding Description

In `rs/sns/governance/src/governance.rs`, the `distribute_rewards` function computes the number of missed reward rounds as:

```rust
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [1](#0-0) 

It then iterates over every missed round in a tight loop with no instruction-limit guard:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = round_duration_seconds
        .saturating_mul(i)
        .saturating_add(reward_start_timestamp_seconds)
        .saturating_sub(self.proto.genesis_timestamp_seconds);

    let current_reward_rate = voting_rewards_parameters.reward_rate_at(
        crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
    );

    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [2](#0-1) 

Each iteration performs multiple `rust_decimal` 128-bit arithmetic operations (`reward_rate_at`, multiplication, addition). The `round_duration_seconds` field is validated to be in the range `[1, 31_557_600]` seconds: [3](#0-2) 

With `round_duration_seconds = 1` (the minimum allowed), a single day of canister downtime produces `new_rounds_count = 86,400`. One year of downtime produces `new_rounds_count = 31,536,000`. At even a conservative ~1,000 Wasm instructions per Decimal arithmetic iteration, 31.5 million iterations consumes ~31.5 billion instructions — far exceeding the IC's 5-billion-instruction per-message limit.

The NNS governance analog (`calculate_voting_rewards`) uses a fixed `REWARD_DISTRIBUTION_PERIOD_SECONDS = 86400` (one day) and cheaper `f64` arithmetic, making it far less susceptible: [4](#0-3) 

The SNS path has no analogous protection.

### Impact Explanation

If `new_rounds_count` is large enough to exhaust the instruction limit, `distribute_rewards` traps. Because the `latest_reward_event` is only updated at the end of a successful execution, the next timer invocation recomputes the same (or larger) `new_rounds_count` and traps again. The SNS governance canister enters a permanent livelock: it can never successfully distribute voting rewards, neuron maturity stops accruing, and the SNS reward mechanism is permanently disabled. This is a governance-level denial-of-service against the SNS.

### Likelihood Explanation

The scenario requires two conditions to coincide:
1. An SNS configured with a small `round_duration_seconds` (e.g., 1 second — technically valid per the schema).
2. Any period of canister downtime (subnet upgrade, canister stop, or a bug that prevents `run_periodic_tasks` from firing).

Subnet upgrades are routine IC operations. The minimum `round_duration_seconds` is not enforced to be "reasonable" — only that it is positive and ≤ 365.25 days. An SNS deployer who sets a short round duration (even 1 hour) combined with a few weeks of downtime could trigger this. Likelihood is **medium** for SNS instances with short round durations.

### Recommendation

Add an upper bound on `new_rounds_count` before entering the loop, analogous to the recommendation in the external report. For example:

```rust
// Cap the number of rounds processed in a single message to avoid
// exceeding the instruction limit.
const MAX_ROUNDS_PER_DISTRIBUTION: u64 = 10_000;
let new_rounds_count = new_rounds_count.min(MAX_ROUNDS_PER_DISTRIBUTION);
```

Alternatively, use a closed-form integral of the reward rate over the missed interval (since `reward_rate_at` is a piecewise quadratic function) to compute the total reward purse in O(1) instead of O(n) time, eliminating the loop entirely.

### Proof of Concept

1. Deploy an SNS with `round_duration_seconds = 1` (minimum allowed).
2. Stop the SNS governance canister (or simulate subnet downtime) for 24 hours.
3. Restart the canister. The next timer tick calls `run_periodic_tasks` → `distribute_rewards`.
4. `new_rounds_count = 86,400`. The `for i in 1..=86400` loop with Decimal arithmetic executes.
5. With `round_duration_seconds = 1` and 1 year of downtime, `new_rounds_count = 31,536,000`, which exhausts the 5-billion-instruction limit, causing a trap.
6. The `latest_reward_event` is never updated. Every subsequent timer invocation recomputes the same large `new_rounds_count` and traps again. Reward distribution is permanently broken. [5](#0-4)
