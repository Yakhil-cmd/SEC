### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes User Overcharge When Native Resource Consumption Is Below Gas Consumption — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` reconciles two independent resource dimensions — EVM gas (ergs) and native resources — by computing a `delta_gas` correction. The correction is applied **only when `delta_gas > 0`** (native exceeds gas), but is silently skipped when `delta_gas < 0` (gas exceeds native). The TODO comment at line 80 explicitly acknowledges this asymmetry. As a result, any transaction whose native resource consumption is lower than its EVM gas consumption is overcharged: the user receives a smaller refund than they are owed, and the excess flows to the operator (coinbase).

---

### Finding Description

In `compute_gas_refund` (`refund_calculation.rs`, lines 66–81):

```rust
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // native consumption > gas consumption → charge more gas
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?   ← acknowledged but unimplemented
}
```

`delta_gas` is defined as:

```
delta_gas = (native_used / native_per_gas) − gas_used
```

| Condition | Meaning | Action taken |
|---|---|---|
| `delta_gas > 0` | Native consumed more than gas | `gas_used` increased ✓ |
| `delta_gas < 0` | Gas consumed more than native | `gas_used` **not** decreased ✗ |

When `delta_gas < 0`, `gas_used` remains inflated relative to actual native consumption. The downstream refund calculation uses this inflated `gas_used`:

```rust
// refund_and_commit_fee (zk/mod.rs ~line 457-458)
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
```

and the operator payment:

```rust
// zk/mod.rs ~line 514-516
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
    ...
```

The user's refund is reduced by `gas_price × |delta_gas|` and the operator receives that same excess amount.

The root cause is structurally identical to the external report: a variable (`gas_used`) is adjusted in one branch of a conditional but not in the symmetric branch, causing incorrect resource accounting. The TODO comment at line 80 is direct evidence that the developer recognized the missing branch.

---

### Impact Explanation

Every transaction where native resource consumption is lower than EVM gas consumption results in the user being overcharged. The overcharge amount is:

```
overcharge = gas_price × (gas_used − native_used / native_per_gas)
```

These tokens are transferred to the operator (coinbase) rather than refunded to the transaction sender. This is a direct, quantifiable loss of user funds on every affected transaction. The effect is permanent — once the block is finalized, the overcharge cannot be recovered.

---

### Likelihood Explanation

The condition `delta_gas < 0` is reached whenever a transaction consumes more EVM gas than native resources (normalized by `native_per_gas`). This is a common scenario for computation-heavy transactions that do not write extensively to storage (storage writes are the primary driver of native resource consumption). The `#[cfg(not(feature = "unlimited_native"))]` guard means this code is active in the standard production build. Any unprivileged user submitting a standard EVM transaction can trigger this path.

---

### Recommendation

Apply the symmetric correction when `delta_gas < 0`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption was lower than gas consumption.
    // Reduce gas_used so the user is refunded for unused native resources.
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Ensure gas_used does not fall below minimal_gas_used (enforced by the
    // max() call that follows this block).
}
```

The existing `core::cmp::max(gas_used, minimal_gas_used)` floor at line 56 already prevents `gas_used` from being reduced below the intrinsic minimum, so the fix is safe to apply without additional guards.

---

### Proof of Concept

1. Submit a transaction with a high EVM gas limit that performs complex computation (many opcodes) but minimal storage writes.
2. After execution, `gas_used` (from ergs) is large; `native_used` is small.
3. `delta_gas = (native_used / native_per_gas) − gas_used < 0`.
4. The `if delta_gas > 0` branch is not taken; `gas_used` is unchanged.
5. `token_to_refund = gas_price × (gas_limit − gas_used)` is smaller than it should be.
6. `token_to_pay_operator = gas_price × gas_used` is larger than it should be.
7. The user's balance after the transaction is lower by `gas_price × |delta_gas|` compared to correct accounting; the operator's balance is higher by the same amount. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
