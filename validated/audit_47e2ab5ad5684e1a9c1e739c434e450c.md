### Title
Single `is_paused` Flag Simultaneously Blocks Deposits and Withdrawals, Preventing Selective Emergency Withdrawal Mode - (File: engine/src/contract_methods/admin.rs)

---

### Summary

The Aurora Engine uses a single `is_paused` boolean in `EngineState` to gate both the ETH bridge withdrawal path (`withdraw`) and the deposit path (`ft_on_transfer`). When the owner pauses the contract in response to a security incident in the deposit path, all user withdrawals are also blocked. To re-enable withdrawals, the owner must call `resume_contract`, which simultaneously re-enables deposits. There is no mechanism to selectively pause only deposits while keeping withdrawals open, resulting in a temporary freeze of user funds.

---

### Finding Description

`EngineState` contains a single `is_paused` boolean field:

```rust
// engine/src/state.rs:28
pub is_paused: bool,
```

The `require_running` guard enforces this flag uniformly:

```rust
// engine/src/contract_methods/mod.rs:65-69
pub fn require_running(state: &state::EngineState) -> Result<(), ContractError> {
    if state.is_paused {
        return Err(errors::ERR_PAUSED.into());
    }
    Ok(())
}
```

Both the withdrawal path and the deposit path call this guard as their first check:

```rust
// engine/src/contract_methods/connector.rs:47
pub fn withdraw(...) -> Result<(), ContractError> {
    require_running(&state::get_state(&io)?)?;  // blocks withdrawals when paused
    ...
}

// engine/src/contract_methods/connector.rs:68
pub fn ft_on_transfer(...) -> Result<Option<SubmitResult>, ContractError> {
    require_running(&state::get_state(&io)?)?;  // blocks deposits when paused
    ...
}
```

`pause_contract` and `resume_contract` toggle this single flag:

```rust
// engine/src/contract_methods/admin.rs:256
state.is_paused = true;   // pause_contract

// engine/src/contract_methods/admin.rs:268
state.is_paused = false;  // resume_contract
```

When the owner pauses the engine (e.g., due to a discovered exploit in the `ft_on_transfer` deposit path), all user `withdraw` calls immediately begin returning `ERR_PAUSED`. To allow users to exit, the owner must call `resume_contract`, which atomically re-enables `ft_on_transfer` as well. There is no intermediate state where withdrawals are open but deposits are closed via the engine-level `is_paused` flag.

Note: the ETH connector contract has a separate, granular `pa_pause_feature` mechanism (with named keys like `"deposit"` and `"engine_withdraw"`), but the engine-level `pause_contract` / `resume_contract` path has no such granularity and is the primary emergency lever for the engine owner.

---

### Impact Explanation

**Impact: High — Temporary freezing of funds.**

When the owner pauses the engine in response to a deposit-path vulnerability, all user ETH held in the Aurora EVM becomes inaccessible via `withdraw`. The owner faces an irresolvable dilemma:

- **Keep paused**: user funds are frozen; no withdrawals possible.
- **Resume**: the vulnerable deposit path is re-enabled, potentially re-exposing the exploit.

In either case, user funds are either temporarily frozen or the attack surface is re-opened. This matches the "High. Temporary freezing of funds" impact category.

---

### Likelihood Explanation

**Likelihood: Medium.**

Emergency pausing is an infrequent but critical admin operation. The scenario requires a real security incident in the deposit path that forces the owner to pause the engine. This is not a routine operation, but it is a realistic and important one. The lack of granular pause control is a structural gap that will manifest precisely when it matters most.

---

### Recommendation

Introduce separate pause flags for deposits and withdrawals at the engine level, analogous to the ETH connector's existing `pa_pause_feature` mechanism. Concretely:

1. Replace the single `is_paused: bool` in `EngineState` with a bitmask (similar to `PrecompileFlags`) that independently controls the deposit path (`ft_on_transfer`) and the withdrawal path (`withdraw`).
2. Expose `pause_deposits` / `pause_withdrawals` admin methods so the owner can selectively disable only the vulnerable path while keeping the exit path open for users.

---

### Proof of Concept

1. Users deposit ETH into Aurora via the bridge; `ft_on_transfer` credits their EVM balances.
2. A vulnerability is discovered in the `ft_on_transfer` deposit path.
3. Owner calls `pause_contract` → `state.is_paused = true`.
4. Any user calling `withdraw` receives `ERR_PAUSED` — funds are frozen.
5. Owner calls `resume_contract` to unblock withdrawals → `state.is_paused = false`.
6. `ft_on_transfer` is now also re-enabled; the vulnerable deposit path is live again.
7. The owner has no way to reach a state where withdrawals succeed but deposits are blocked via the engine-level pause flag.
