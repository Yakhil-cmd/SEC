### Title
Pausing the Engine Mid-Flight Permanently Freezes Funds in `exit_to_near_precompile_callback` - (File: engine/src/contract_methods/connector.rs)

---

### Summary

`exit_to_near_precompile_callback` unconditionally calls `require_running` before it inspects the promise result or executes any refund/transfer logic. Because NEAR's async model executes the callback in a separate receipt from the originating `submit` receipt, a legitimate `pause_contract` call issued between those two receipts causes the callback to panic with `ERR_PAUSED`. At that point the EVM-side debit (ERC-20 burn or ETH transfer) has already been committed and the base promise (`near_withdraw` for wNEAR, or `ft_transfer` for ERC-20) has already executed, but the NEAR-side delivery to the user never occurs and no refund path is reachable. The funds are frozen with no automated recovery mechanism.

---

### Finding Description

**Execution flow for the wNEAR-unwrap path:**

1. **Receipt R1** — `submit` executes atomically:
   - `require_running` passes (engine is running).
   - EVM runs; the ERC-20 `burn` call debits the user's wNEAR ERC-20 balance.
   - `filter_promises_from_logs` schedules a `PromiseArgs::Callback` whose base is `near_withdraw` on the wNEAR contract and whose callback is `exit_to_near_precompile_callback` on Aurora. [1](#0-0) [2](#0-1) 

2. **Receipt R2** — `near_withdraw` executes on the wNEAR contract:
   - wNEAR NEP-141 tokens are burned; the equivalent NEAR yocto balance is credited to Aurora's account.

3. **Between R1 and R3** — the engine owner calls `pause_contract`:
   - `state.is_paused` is set to `true` and persisted. [3](#0-2) 

4. **Receipt R3** — `exit_to_near_precompile_callback` executes:
   - The very first substantive check is `require_running(&state)?` at line 203.
   - Because `is_paused == true`, this returns `Err(ERR_PAUSED)`.
   - `sdk_unwrap()` in `lib.rs` panics; the receipt fails.
   - The `transfer_near` branch (which would send the unwrapped NEAR to the user) is **never reached**.
   - The `refund` branch (which would re-mint burned ERC-20 tokens on failure) is **never reached**. [4](#0-3) 

The same failure mode applies to the `error_refund` feature path: if `ft_transfer` fails and the engine is paused before the callback, `refund_on_error` is never called and the burned ERC-20 tokens are permanently lost. [5](#0-4) 

The callback is only scheduled when `callback_args != default()`, which is true for:
- **wNEAR unwrap** (`transfer_near` is `Some`) — always scheduled regardless of feature flags.
- **`error_refund` feature** (`refund` is `Some`) — scheduled for all ExitToNear calls when the feature is compiled in. [6](#0-5) 

---

### Impact Explanation

For the **wNEAR-unwrap** path:
- The user's wNEAR ERC-20 balance is burned (EVM state committed in R1).
- The wNEAR NEP-141 is burned and NEAR is credited to Aurora's account (R2 committed).
- The NEAR never reaches the user; it sits in Aurora's account with no automated recovery path.
- Without explicit admin intervention (identifying affected users and manually sending NEAR), the funds are permanently frozen.

For the **`error_refund`** path:
- The user's ERC-20 tokens are burned in R1.
- `ft_transfer` fails in R2 (tokens not delivered).
- The callback panics in R3; `refund_on_error` is never called; the burned tokens are not re-minted.
- The user loses the full exit amount with no recourse.

Impact: **High — Temporary (potentially permanent) freezing of funds.**

---

### Likelihood Explanation

`pause_contract` is an owner-only operation intended for legitimate maintenance and emergency response. The window between R1 and R3 spans at least one NEAR block (~1 second). Any pause issued during active ExitToNear traffic — including emergency pauses triggered by unrelated incidents — silently destroys in-flight exits. The scenario requires no attacker: a routine operational pause is sufficient. The wNEAR-unwrap path is a documented, user-facing feature, making this a realistic production risk.

---

### Recommendation

Remove the `require_running` guard from `exit_to_near_precompile_callback`, or move it so it only gates new state-changing work rather than blocking the completion of already-committed exits. Specifically:

- The refund branch (`refund_on_error`) and the `transfer_near` branch must execute regardless of the engine's pause state, because the EVM-side debit was already committed in a prior receipt.
- If a pause check is desired for other reasons, it should be placed **after** the promise-result inspection, and only applied to paths that do not correspond to completing a previously committed exit.

---

### Proof of Concept

```
1. Deploy Aurora engine with wNEAR support.
2. Bridge wNEAR ERC-20 to a test account on Aurora.
3. Submit an EVM transaction calling the ExitToNear precompile with the `:unwrap` suffix
   (triggers the wNEAR-unwrap path, schedules near_withdraw + callback).
4. After the submit receipt is finalized (R1 done, R2 pending), call `pause_contract`
   from the owner account before R3 executes.
5. Observe:
   - User's wNEAR ERC-20 balance: 0 (burned in R1).
   - Aurora's wNEAR NEP-141 balance: decreased (burned in R2).
   - User's NEAR balance: unchanged (callback panicked in R3).
   - Aurora's NEAR balance: increased by the exit amount (NEAR stuck).
6. Assert balance invariant violation: EVM debit occurred without corresponding NEAR delivery.
```

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
