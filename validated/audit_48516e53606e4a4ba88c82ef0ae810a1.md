Audit Report

## Title
ckBTC Minter `estimate_withdrawal_fee` Traps on Oversized Amounts and `get_minter_info` Omits Maximum Withdrawal Limit - (File: `rs/bitcoin/ckbtc/minter/src/main.rs`)

## Summary
The ckBTC minter's `estimate_withdrawal_fee` query endpoint calls `panic!` for two over-limit `BuildTxError` variants (`NotEnoughFunds` and `InvalidTransaction(TooManyInputs)`), causing the IC canister to trap and reject the call with `CANISTER_ERROR` instead of returning a structured error. Simultaneously, `get_minter_info` exposes only `retrieve_btc_min_amount` with no corresponding maximum, leaving integrators with no standard mechanism to discover the effective upper bound. A user who submits `retrieve_btc_with_approval` with an oversized amount has their ckBTC burned and is reimbursed minus `BitcoinFeeEstimator::COST_OF_ONE_BILLION_CYCLES`, constituting a real financial loss with no prior warning.

## Finding Description
**Root cause 1 — panic in `estimate_withdrawal_fee`:**
At `rs/bitcoin/ckbtc/minter/src/main.rs` lines 236–248, the match arms for `BuildTxError::NotEnoughFunds` and `BuildTxError::InvalidTransaction(TooManyInputs)` both call `panic!`, causing the query to trap:

```rust
Err(BuildTxError::NotEnoughFunds) => {
    panic!("ERROR: withdrawal amount is too large for the minter")
}
Err(BuildTxError::InvalidTransaction(
    e @ InvalidTransactionError::TooManyInputs { .. },
)) => panic!("ERROR: the minter cannot currently process such a large withdrawal amount ...")
```

The ckDOGE minter (`rs/dogecoin/ckdoge/minter/src/main.rs` lines 116–127) handles the identical `BuildTxError` variants by returning `EstimateWithdrawalFeeError::AmountTooHigh` via `.map_err(...)`, demonstrating the correct pattern exists and was intentionally applied to the newer minter but not backported to ckBTC.

**Root cause 2 — missing maximum in `get_minter_info`:**
`MinterInfo` (`rs/bitcoin/ckbtc/minter/src/lib.rs` lines 83–91) has no maximum field. `get_minter_info()` (lines 251–259) returns only `retrieve_btc_min_amount`. The effective maximum is determined at runtime by `max_num_inputs_in_transaction` (`rs/bitcoin/ckbtc/minter/src/state.rs` line 484), which is operator-configurable and can change without any public signal.

**Root cause 3 — post-hoc reimbursement with fee deduction:**
The test `should_cancel_and_reimburse_large_withdrawal` (`rs/bitcoin/ckbtc/minter/tests/tests.rs` lines 3108–3173) confirms that `retrieve_btc_with_approval` accepts an oversized withdrawal, burns the ckBTC, and then reimburses `withdrawal_amount - BitcoinFeeEstimator::COST_OF_ONE_BILLION_CYCLES`. The user suffers a net loss of `COST_OF_ONE_BILLION_CYCLES` with no prior warning possible, because the only pre-validation endpoint (`estimate_withdrawal_fee`) traps instead of returning a structured error.

The exploit path is: (1) user or integrator calls `estimate_withdrawal_fee` with a large amount → receives canister trap, cannot distinguish from a bug; (2) user submits `retrieve_btc_with_approval` with the same amount → ckBTC is burned; (3) minter detects `TooManyInputs` during transaction building → reimburses minus fee → user loses `COST_OF_ONE_BILLION_CYCLES` of ckBTC.

## Impact Explanation
This constitutes a **High** severity impact under "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm." The ckBTC minter is a production mainnet canister with significant TVL. The panic causes concrete integration failures (callers cannot distinguish an over-limit condition from a canister crash). The missing maximum, combined with the trapping pre-validation endpoint, creates a scenario where users have no safe mechanism to discover the effective limit, leading to real ckBTC financial loss (reimbursement fee deduction) on every oversized withdrawal attempt.

## Likelihood Explanation
Any integrator or wallet that calls `estimate_withdrawal_fee` to pre-validate a withdrawal amount before calling `retrieve_btc_with_approval` will encounter the trap. The `max_num_inputs_in_transaction` limit is dynamic and operator-configurable, meaning the effective maximum changes over time without any public signal, increasing the probability that a previously valid amount becomes invalid. Likelihood is medium-high for integrators and wallets; low for casual users who stay within typical small amounts.

## Recommendation
1. Change `estimate_withdrawal_fee` to return `Result<WithdrawalFee, WithdrawalFeeError>` and map `BuildTxError::NotEnoughFunds` and `BuildTxError::InvalidTransaction(TooManyInputs)` to a structured `WithdrawalFeeError::AmountTooHigh`, matching the ckDOGE minter pattern.
2. Add a `retrieve_btc_max_amount: Option<u64>` field to `MinterInfo` computed from the current available UTXOs and `max_num_inputs_in_transaction`, so callers can discover the effective maximum without trial-and-error.

## Proof of Concept
**Step 1 (query trap):** Call `estimate_withdrawal_fee` with an amount exceeding the minter's UTXO capacity or requiring more than `max_num_inputs_in_transaction` inputs:
```
dfx canister call ckbtc_minter estimate_withdrawal_fee '(record { amount = opt (99_999_999_999_999 : nat64) })'
```
Expected (ckDOGE): `Err(AmountTooHigh)`. Actual (ckBTC): canister trap with `"ERROR: withdrawal amount is too large for the minter"`.

**Step 2 (financial loss):** Reproduce using the existing integration test `should_cancel_and_reimburse_large_withdrawal` in `rs/bitcoin/ckbtc/minter/tests/tests.rs` lines 3108–3173. The test deposits `MAX_INPUTS + 100` UTXOs, submits a withdrawal requiring `MAX_INPUTS + 1` inputs, and asserts the user is reimbursed `withdrawal_amount - BitcoinFeeEstimator::COST_OF_ONE_BILLION_CYCLES`, confirming the fee loss. Run with:
```
cargo test -p ic-ckbtc-minter should_cancel_and_reimburse_large_withdrawal
```
