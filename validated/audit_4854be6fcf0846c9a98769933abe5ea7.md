Audit Report

## Title
Unbounded JoinSet Task Spawning and `AlreadyInserted`-Triggered Block Discard via Crafted Duplicate `StrippedBlockProposal` Entries — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary
A Byzantine peer below the consensus fault threshold can send a `StrippedBlockProposal` containing an arbitrarily large number of duplicate `SignedIngressId` entries. Because `TryFrom<pb::StrippedBlockProposal>` applies no length cap, `BlockProposalAssembler::new` creates one slot per entry without deduplication, and `assemble_message` spawns one unbounded Tokio task per slot, this causes replica-local task and memory exhaustion. Additionally, when two tasks for the same duplicate ID both complete, the second hits `InsertionError::AlreadyInserted`, which the caller unconditionally treats as fatal, returning `AssembleResult::Unwanted` and aborting assembly of an otherwise valid block.

## Finding Description

**Root cause 1 — No size guard at deserialization:**

`TryFrom<pb::StrippedBlockProposal> for StrippedBlockProposal` in `rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs` (lines 100–108) collects `ingress_messages` from the protobuf repeated field with no length check:

```rust
ingress_messages: value
    .ingress_messages
    .into_iter()
    .map(SignedIngressId::try_from)
    .collect::<Result<Vec<_>, _>>()?,
```

The protocol constant `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` (in `rs/limits/src/lib.rs`) is enforced only during ingress payload validation, not here.

**Root cause 2 — No deduplication before task spawning:**

`BlockProposalAssembler::new` (assembler.rs lines 572–588) maps the raw, potentially duplicate-filled list verbatim into `Vec<(SignedIngressId, Option<SignedIngress>)>`. `missing_stripped_messages()` (lines 591–598) returns every entry whose slot is `None` — all N of them on first call. `assemble_message` (lines 214–226) then spawns one `get_or_fetch` task per returned ID with no cap on the `JoinSet`.

**Root cause 3 — `AlreadyInserted` treated as fatal:**

`try_insert` (lines 435–454) uses a linear `find` that always locates the *first* matching slot. After the first duplicate task fills it, every subsequent task for the same ID hits the `is_some()` branch and returns `InsertionError::AlreadyInserted`. The caller at lines 262–271 treats any `InsertionError` as a fatal bug and returns `AssembleResult::Unwanted`, aborting the entire assembly.

**Why existing checks fail:**

`download_stripped_message` (download.rs lines 332–375) runs an infinite loop with `with_max_elapsed_time(None)` exponential backoff — tasks only terminate when a successful response is received or the outer bouncer `select!` fires. With N duplicate entries, N tasks run concurrently for the full bouncer window. The bouncer check (assembler.rs lines 236–240) is the only termination mechanism, and it fires only after the block is no longer wanted by consensus.

## Impact Explanation

**Resource exhaustion (High):** A single crafted `StrippedBlockProposal` with N duplicate entries spawns N Tokio tasks. Each task that misses the local pool enters `download_stripped_message`, which loops indefinitely with exponential backoff until the bouncer fires. With N = 10,000 entries, this saturates the Tokio thread pool and heap for the duration of the bouncer window, delaying or stalling block assembly at that height for the targeted replica.

**Consensus blocking / subnet availability impact (High):** If the duplicate IDs correspond to ingress messages present in the local pool, both tasks complete immediately, the second returns `AlreadyInserted`, and `assemble_message` returns `AssembleResult::Unwanted`, discarding the assembly attempt. If the Byzantine peer is the sole advertiser of the block at that height, this contributes to finalization delay. This matches the allowed impact class: *Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation

- Requires one Byzantine peer below the fault threshold — a realistic assumption in the IC threat model.
- The peer need not be the block proposer; it only needs to advertise a `ConsensusMessageId` it observed from an honest proposer and respond with a crafted stripped body.
- No cryptographic material needs to be forged; the `unstripped_consensus_message_id` hash is taken from the honest block and the duplicate list is purely in the stripped metadata.
- The attack is fully local to the targeted replica and leaves no on-chain trace.
- Repeatable: the Byzantine peer can trigger this for every block height.

## Recommendation

1. **Enforce a size cap at deserialization:** In `TryFrom<pb::StrippedBlockProposal>` (stripped.rs), reject any `ingress_messages` list whose length exceeds `MAX_INGRESS_MESSAGES_PER_BLOCK`, and similarly cap `stripped_idkg_dealings`.
2. **Deduplicate before spawning:** In `BlockProposalAssembler::new` or `missing_stripped_messages`, deduplicate the ID list before returning it to the task-spawning loop.
3. **Treat `AlreadyInserted` as non-fatal:** In `assemble_message` (assembler.rs lines 262–271), log a warning and continue rather than returning `AssembleResult::Unwanted` on `AlreadyInserted` — a duplicate fetch result should not abort a valid assembly.
