Audit Report

## Title
Missing Reimbursement for `AmountTooLow` and `DustOutput` Withdrawal Failures Causes Permanent ckBTC Loss - (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary

When `submit_pending_requests` encounters `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput`, it finalizes the affected `RetrieveBtcRequest` entries via `remove_retrieve_btc_request` without scheduling any ckBTC reimbursement. Because the user's ckBTC was already burned at request-acceptance time, this results in a permanent, irrecoverable loss of the withdrawal principal. The `InvalidTransaction` error path in the same function correctly calls `reimburse_canceled_requests`, confirming the omission is an inconsistency rather than a design choice.

## Finding Description

**Burn at acceptance time (irreversible):**
In `retrieve_btc` (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, line 210), `burn_ckbtcs` is called before the `RetrieveBtcRequest` is enqueued. In `retrieve_btc_with_approval` (line 314), `burn_ckbtcs_icrc2` is called before enqueue. Both paths set `reimbursement_account: Some(...)` on the request (lines 218 and 327 respectively), so the reimbursement infrastructure is fully populated.

**Three error paths in `submit_pending_requests` (`rs/bitcoin/ckbtc/minter/src/lib.rs`):**

- **`BuildTxError::InvalidTransaction` (lines 400–411):** calls `reimburse_canceled_requests`, which invokes `state::audit::reimburse_withdrawal` → `state.schedule_withdrawal_reimbursement`, queuing a ckBTC mint back to the user. User is made whole.
- **`BuildTxError::AmountTooLow` (lines 412–434):** iterates the batch and calls `state::audit::remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)` for each request. No reimbursement is scheduled.
- **`BuildTxError::DustOutput` (lines 436–453):** same as above for the offending request. No reimbursement is scheduled.

**`remove_retrieve_btc_request` (`rs/bitcoin/ckbtc/minter/src/state/audit.rs`, lines 67–84):** records a `RemovedRetrieveBtcRequest` event and pushes a `FinalizedBtcRequest` with `FinalizedStatus::AmountTooLow`. It never calls `reimburse_withdrawal` or `schedule_withdrawal_reimbursement`.

**Reimbursement infrastructure is present and functional:** `state::audit::reimburse_withdrawal` (lines 240–265 of `audit.rs`) records a `ScheduleWithdrawalReimbursement` event and calls `state.schedule_withdrawal_reimbursement`. The async `reimburse_withdrawals` task (`reimbursement/mod.rs`, lines 58–116) then mints ckBTC back to the user. This entire pipeline is exercised by the `InvalidTransaction` path but is never invoked for `AmountTooLow` or `DustOutput`.

**`WithdrawalReimbursementReason` (`reimbursement/mod.rs`, lines 39–43):** currently has only one variant, `InvalidTransaction(InvalidTransactionError)`. A new variant (e.g., `FeesTooHigh`) would need to be added to support reimbursement for these paths.

## Impact Explanation

This is a **High** severity finding. Any user who calls `retrieve_btc` or `retrieve_btc_with_approval` with an amount that passes the `fee_based_retrieve_btc_min_amount` check at submission time can permanently lose their entire withdrawal amount if Bitcoin network fees spike before `submit_pending_requests` processes the batch. The ckBTC is burned, no BTC is sent, and no ckBTC is minted back. The user receives `RetrieveBtcStatusV2::AmountTooLow` with no recourse. This is a concrete, irrecoverable financial loss in an in-scope ck-token system, matching the allowed impact: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."*

## Likelihood Explanation

Bitcoin transaction fees are volatile and can spike by orders of magnitude within minutes. The minter queues requests for up to `max_time_in_queue_nanos` before processing them (`rs/bitcoin/ckbtc/minter/src/state.rs`, line 440). Any user whose request sits in the queue during a fee spike is at risk. The entry path requires only a standard unprivileged call to `retrieve_btc` or `retrieve_btc_with_approval` — no special privileges or coordination. The real-world precedent of stuck ckBTC withdrawals documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` confirms that fee-related stuck transactions are not hypothetical.

## Recommendation

Apply the same reimbursement logic used for `BuildTxError::InvalidTransaction` to the `AmountTooLow` and `DustOutput` error paths:

1. Add a new variant to `WithdrawalReimbursementReason` (e.g., `FeesTooHigh`) in `rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`.
2. In the `BuildTxError::AmountTooLow` branch (`lib.rs`, lines 412–434), replace the `remove_retrieve_btc_request` loop with a call to `reimburse_canceled_requests(s, batch, WithdrawalReimbursementReason::FeesTooHigh, 0, runtime)` (zero fee since no Bitcoin transaction was broadcast).
3. In the `BuildTxError::DustOutput` branch (`lib.rs`, lines 436–453), apply the same reimbursement call for the offending request before putting the remaining requests back.
4. Update the `retrieve_btc_status_v2` mapping in `state.rs` to handle the new reason variant.

## Proof of Concept

1. User calls `retrieve_btc_with_approval` with `amount = fee_based_retrieve_btc_min_amount` (passes the upfront check at `retrieve_btc.rs` line 266). ckBTC is burned at line 314; `reimbursement_account` is set at line 327.
2. Bitcoin network fees spike before `submit_pending_requests` runs.
3. `build_unsigned_transaction` returns `BuildTxError::AmountTooLow` because the fee now exceeds the withdrawal amount.
4. The minter executes the `AmountTooLow` branch (`lib.rs` lines 412–434): `remove_retrieve_btc_request` is called, finalizing the request with `FinalizedStatus::AmountTooLow`. No `schedule_withdrawal_reimbursement` is called.
5. User queries `retrieve_btc_status_v2` and receives `RetrieveBtcStatusV2::AmountTooLow`. Their ckBTC is permanently gone.

A deterministic integration test can reproduce this by: (a) calling `retrieve_btc_with_approval` with a minimal valid amount, (b) advancing the mock fee estimator to return a fee exceeding the withdrawal amount before the next `ProcessLogic` tick, (c) triggering `submit_pending_requests`, and (d) asserting that `retrieve_btc_status_v2` returns `AmountTooLow` while the user's ckBTC balance has not been restored — contrasted with the `InvalidTransaction` path which does restore the balance. fn reimburse_canceled_requests<R: CanisterRuntime>(
    state: &mut state::CkBtcMinterState,
    requests: BTreeSet<state::RetrieveBtcRequest>,
    reason: WithdrawalReimbursementReason,
    total_fee: u64,
    runtime: &R,
) {
    assert!(!requests.is_empty());
    let fees = distribute(total_fee, requests.len() as u64);
    // This assertion makes sure the fee is smaller than each request amount
    assert!(
        fees[0] <= state.retrieve_btc_min_amount,
        "BUG: fees {fees:?} for {} withdrawal requests are larger than `retrieve_btc_min_amount` {}",
        requests.len(),
        state.retrieve_btc_min_amount
    );
    for (request, fee) in requests.into_iter().zip(fees.into_iter()) {
        if let Some(account) = request.reimbursement_account {
            let amount = request.amount.saturating_sub(fee);
            if amount > 0 {
                state::audit::reimburse_withdrawal(
                    state,
                    request.block_index,
                    amount,
                    account,
                    reason.clone(),
                    runtime,
                );
            }
        } else {
            log!(
                Priority::Info,
                "[reimburse_canceled_requests]: account is not found for retrieve_btc request ({:?})",
                request
            );
        }
    }
}

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}

pub fn reimburse_withdrawal<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    burn_block_index: LedgerBurnIndex,
    reimbursed_amount: u64,
    reimbursement_account: Account,
    reason: WithdrawalReimbursementReason,
    runtime: &R,

    record_event(
        EventType::ScheduleWithdrawalReimbursement {
            account: reimbursement_account,
            amount: reimbursed_amount,
            reason: reason.clone(),

        runtime,
    );
    state.schedule_withdrawal_reimbursement(
        burn_block_index,
        ReimburseWithdrawalTask {
            account: reimbursement_account,
            amount: reimbursed_amount,
            reason,
        },
    )
}
