### Title
Unprivileged caller can force-register any principal as a swap buyer via `refresh_buyer_tokens`, locking their ICP until finalization — (`rs/sns/swap/canister/canister.rs`)

### Summary
The `refresh_buyer_tokens` canister endpoint accepts an arbitrary `buyer` principal in the request body and uses it without verifying it matches the caller. Any unprivileged principal can therefore register a victim as a swap participant, blocking the victim's `error_refund_icp` path and, if the swap commits, forcing the victim to receive unwanted SNS neurons in exchange for their ICP.

### Finding Description
`refresh_buyer_tokens` at `canister.rs:130–133` resolves the effective buyer principal from `arg.buyer` when non-empty, with no `caller == buyer` guard:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // ← no auth check
};
```

`refresh_buyer_token_e8s` then reads the ICP ledger balance of the subaccount derived from `p` (not from the caller). If the balance meets the minimum participation threshold, a `BuyerState` entry is created for `p` with `transfer_success_timestamp_seconds = 0`.

After the swap closes, `error_refund_icp` checks:

```rust
if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
    if let Some(transfer) = &buyer_state.icp
        && transfer.transfer_success_timestamp_seconds == 0
    {
        return ErrorRefundIcpResponse::new_precondition_error(...);
    }
}
```

Because the attacker-created `BuyerState` always starts with `transfer_success_timestamp_seconds = 0`, the victim's `error_refund_icp` call is rejected until `sweep_icp` processes the entry — at which point the ICP has already been disbursed (to SNS governance if committed, or back to the victim if aborted).

### Impact Explanation
- **Swap commits:** Victim's ICP is swept to SNS governance; victim receives SNS neurons they never consented to. The `error_refund_icp` escape hatch is unavailable.
- **Swap aborts:** Victim's ICP is eventually refunded via `sweep_icp`, but the `error_refund_icp` path is blocked in the interim, causing a denial-of-service on the refund mechanism until finalization runs.
- In both cases the victim is enrolled as a swap participant without their knowledge or consent.

### Likelihood Explanation
The preconditions are realistic: ICP accidentally sent to a swap subaccount is a documented user error (the `error_refund_icp` mechanism exists precisely for this case). The attacker needs only the victim's principal ID (public) and the confirmation text (readable from canister state). No privileged access is required.

### Recommendation
Add a caller-identity check in the canister endpoint:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let requested = PrincipalId::from_str(&arg.buyer).unwrap();
    assert_eq!(
        requested,
        caller_principal_id(),
        "buyer must match caller"
    );
    requested
};
```

Alternatively, remove the `buyer` override field entirely and always use the caller, since the only legitimate use case (a user notifying the swap on their own behalf) is already covered by the empty-buyer path.
