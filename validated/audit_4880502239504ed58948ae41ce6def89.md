Audit Report

## Title
SNS Governance `compute_ballots_for_new_proposal` Performs Unbounded Linear Neuron Scan Without Instruction-Limit Guard, Enabling Governance DoS - (File: rs/sns/governance/src/governance.rs)

## Summary

`compute_ballots_for_new_proposal` in SNS Governance iterates synchronously over every registered neuron in a heap-resident `BTreeMap` on every `make_proposal` call, with no instruction-counter check or multi-message splitting. NNS Governance has been explicitly patched with hard/soft instruction limits and `noop_self_call_if_over_instructions` for the equivalent operation; SNS Governance has received none of these mitigations. If the neuron count grows large enough, every `make_proposal` call will trap at the IC instruction limit, permanently freezing SNS governance.

## Finding Description

**Root cause — unbounded loop with no guard:**

`compute_ballots_for_new_proposal` at line 5255 iterates over the entire `self.proto.neurons` `BTreeMap` in a single synchronous pass, computing voting power and inserting into a `BTreeMap<String, Ballot>` for every eligible neuron: [1](#0-0) 

There is no call to `ic_cdk::api::instruction_counter()`, no soft/hard limit constant, and no mechanism to split work across messages. A grep over the entire `rs/sns/governance/` tree confirms zero uses of `instruction_counter`, `HARD_VOTING_INSTRUCTIONS_LIMIT`, or `SOFT_VOTING_INSTRUCTIONS_LIMIT`.

**Unconditional call from `make_proposal`:**

`make_proposal` calls `compute_ballots_for_new_proposal` unconditionally after validation, before any state mutation: [2](#0-1) 

**Contrast with NNS Governance — patched:**

NNS Governance defines explicit instruction limits and uses `noop_self_call_if_over_instructions` to split ballot computation across messages: [3](#0-2) [4](#0-3) 

SNS Governance has no equivalent.

**Neuron population ceiling:**

`check_neuron_population_can_grow` enforces `max_number_of_neurons`, a governance-configurable parameter: [5](#0-4) 

`NervousSystemParameters::MAX_NUMBER_OF_NEURONS_CEILING` is sized to accommodate all swap participants (direct + Neurons' Fund + developer neurons), as confirmed by the constraints test: [6](#0-5) 

This ceiling is designed to be large — potentially hundreds of thousands of neurons — which is the same scale at which instruction exhaustion occurs.

**Exploit path:**

1. `max_number_of_neurons` is at or near `MAX_NUMBER_OF_NEURONS_CEILING` (set by default after a large swap, or raised via a governance proposal).
2. Token holders (or a single well-funded attacker) stake tokens and call `manage_neuron { ClaimOrRefresh }` to fill the neuron table with neurons whose dissolve delay meets `neuron_minimum_dissolve_delay_to_vote_seconds`.
3. Any principal calls `make_proposal`; execution enters the O(N) loop at line 5255, exhausts the ~40B instruction limit, and traps.
4. The call rolls back; no proposal is created. All subsequent `make_proposal` calls trap identically.

## Impact Explanation

Once the instruction-exhaustion threshold is reached, no new proposals can be submitted to the SNS. Upgrades, parameter changes, treasury transfers, and dapp management are all blocked. Existing open proposals can still be voted on and executed, but once they settle, the SNS is permanently ungovernable without voluntary neuron dissolution — a coordination problem that may be infeasible in a decentralized setting. This is a concrete application/platform-level DoS on SNS governance, matching the **High** impact category: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact."

## Likelihood Explanation

Medium. The neuron table must be filled to the instruction-exhaustion threshold. This can occur naturally as an SNS grows (large swap + post-swap staking), or be forced by a well-funded attacker acquiring enough tokens. Raising `max_number_of_neurons` requires a governance proposal, but SNS communities routinely do this to accommodate growth. The NNS Governance team has already recognized and fixed this exact class of bug in their own canister, confirming it is a realistic concern and not purely theoretical.

## Recommendation

1. Add an instruction-limit check inside the loop in `compute_ballots_for_new_proposal` mirroring NNS Governance's `HARD_VOTING_INSTRUCTIONS_LIMIT` / `SOFT_VOTING_INSTRUCTIONS_LIMIT` pattern in `rs/nns/governance/src/voting.rs`.
2. Refactor `make_proposal` to use `noop_self_call_if_over_instructions` (from `ic_nervous_system_long_message`) to split ballot computation across multiple messages if needed.
3. Pre-compute and cache a voting-power snapshot (as NNS Governance now does via `compute_ballots_for_standard_proposal`) so `make_proposal` reads from a snapshot rather than iterating all neurons inline.
4. Enforce a tighter `max_number_of_neurons` ceiling calibrated against the per-neuron instruction cost to guarantee the loop always completes within the IC instruction limit.

## Proof of Concept

```
1. Deploy an SNS with max_number_of_neurons = MAX_NUMBER_OF_NEURONS_CEILING.
2. Distribute SNS tokens to N principals where N approaches the ceiling.
3. Each principal calls manage_neuron { ClaimOrRefresh } with a dissolve_delay
   >= neuron_minimum_dissolve_delay_to_vote_seconds.
4. Once N neurons are registered, call make_proposal from any eligible neuron.
5. Execution enters compute_ballots_for_new_proposal at line 5255,
   iterates over all N neurons, exhausts the ~40B instruction limit, and traps.
6. The call is rolled back; the proposal is never created.
7. All subsequent make_proposal calls trap identically — governance is frozen.

Reproducible as a PocketIC integration test: initialize SNS governance with
MAX_NUMBER_OF_NEURONS_CEILING neurons (all with sufficient dissolve delay),
then assert that make_proposal returns an instruction-limit trap rather than
a ProposalId.
```

        let max_number_of_neurons = self
            .nervous_system_parameters_or_panic()
            .max_number_of_neurons

        if (self.proto.neurons.len() as u64) + 1 > max_number_of_neurons {
