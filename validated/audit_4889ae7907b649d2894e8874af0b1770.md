### Title
Unsynchronized `transaction_fee_e8s` / `neuron_minimum_stake_e8s` in SNS Swap `Init` Never Cross-Validated Against Live SNS Ledger and Governance — (`rs/sns/swap/src/types.rs`)

---

### Summary

The SNS Swap canister's `Init` struct accepts `transaction_fee_e8s` and `neuron_minimum_stake_e8s` as caller-supplied values that are required to mirror the live SNS Ledger and SNS Governance canisters respectively. The code itself explicitly documents that these values are **never cross-validated** against the actual on-chain state of those canisters. The swap canister uses these locally-stored copies to compute neuron basket sizes during finalization. If they diverge from the real values — either at deployment time or after a post-deployment governance parameter change — finalization fails and participant ICP becomes locked in the swap canister.

---

### Finding Description

The `Init` message for the SNS Swap canister contains two fields that must be kept in sync with sibling canisters:

- `transaction_fee_e8s`: must equal the SNS Ledger's transfer fee.
- `neuron_minimum_stake_e8s`: must equal the SNS Governance's `NervousSystemParameters.neuron_minimum_stake_e8s`.

The proto definition explicitly acknowledges the risk:

> "Same as SNS ledger. Must hold the same value as SNS ledger. **Whether the values match is not checked. If they don't match things will break.**"
> "Same as SNS governance. Must hold the same value as SNS governance. **Whether the values match is not checked. If they don't match things will break.**" [1](#0-0) 

The generated Rust struct carries the same documentation: [2](#0-1) 

The `Init::validate()` function only checks that these fields are **present** (`is_none()` guard), but never queries the SNS Ledger or SNS Governance to verify the values are correct: [3](#0-2) 

`Swap::new()` calls `init.validate()` and panics on failure, but that validation does not include any cross-canister consistency check: [4](#0-3) 

The `canister_init` entry point passes the caller-supplied `Init` directly to `Swap::new()`: [5](#0-4) 

There are two concrete divergence paths:

1. **At deployment**: An NNS proposal creator submits a `CreateServiceNervousSystem` proposal with `transaction_fee_e8s` or `neuron_minimum_stake_e8s` values that differ from what the SNS Ledger / Governance will actually be initialized with. The `SnsInitPayload` validation (`validate_participation_constraints`) only checks internal self-consistency of the swap init values, not that they match the sibling canisters. [6](#0-5) 

2. **Post-deployment**: After the swap is deployed and in `PENDING`/`ADOPTED` state, an SNS governance proposal can update `NervousSystemParameters` (including `neuron_minimum_stake_e8s`) or the SNS Ledger fee. The swap canister holds a stale local copy and never re-queries the live values before finalization.

---

### Impact Explanation

During `finalize_swap`, the swap canister uses its locally-stored `transaction_fee_e8s` and `neuron_minimum_stake_e8s` to compute neuron basket sizes and validate that each participant receives enough tokens to form valid neurons. If the swap's copy is lower than the actual SNS Governance minimum stake, the swap will attempt to claim neurons that governance rejects as too small. If the swap's copy of `transaction_fee_e8s` is lower than the actual ledger fee, token transfers during neuron creation will fail. In both cases, finalization fails and participant ICP contributions are locked in the swap canister until an error-refund path is exercised — if one is available. The `SnsNeuronRecipe` entries are marked with failed `ClaimedStatus`, and participants may not be able to recover their ICP. [7](#0-6) 

---

### Likelihood Explanation

The post-deployment path is reachable by any SNS token holder who can pass a governance proposal to update `NervousSystemParameters` between swap deployment and finalization. The at-deployment path is reachable by any NNS neuron holder who can submit a `CreateServiceNervousSystem` proposal. Neither requires privileged access beyond normal governance participation. The codebase itself documents the unsynchronized state as a known risk with the comment "If they don't match things will break." [2](#0-1) 

---

### Recommendation

1. During `canister_init`, perform cross-canister calls to the SNS Ledger (`icrc1_fee`) and SNS Governance (`get_nervous_system_parameters`) to verify that the supplied `transaction_fee_e8s` and `neuron_minimum_stake_e8s` match the live values, rejecting initialization if they diverge.
2. Alternatively, remove the local copies entirely and query the live values from the sibling canisters at finalization time, eliminating the stale-cache risk.
3. At minimum, add a pre-finalization consistency check that re-reads the live ledger fee and governance minimum stake and aborts finalization with a recoverable error (rather than silently producing failed neuron recipes) if they no longer match the swap's cached values.

---

### Proof of Concept

1. Deploy an SNS via `CreateServiceNervousSystem` with `transaction_fee_e8s = 10` in the swap init, but configure the SNS Ledger with `transfer_fee = 10_000`.
2. The swap passes `Init::validate()` because `transaction_fee_e8s` is present.
3. Participants contribute ICP; the swap reaches `COMMITTED`.
4. `finalize_swap` is called. The swap computes neuron basket sizes using `transaction_fee_e8s = 10`, concluding each participant has enough tokens.
5. The actual ledger charges `10_000` per transfer; neuron-creation transfers fail.
6. Participants' ICP is locked in the swap canister; `SnsNeuronRecipe` entries are marked `Failed`.
