Audit Report

## Title
`inspect_message`-Only Authorization Bypassed via Composite-Query Inter-Canister Calls — (`rs/boundary_node/rate_limits/canister/canister.rs`)

## Summary
The rate-limits canister relies exclusively on `canister_inspect_message` to gate access to its three read-only query methods (`get_config`, `get_rule_by_id`, `get_rules_by_incident_id`). Because `inspect_message` is only invoked for ingress messages and is never called for inter-canister calls, any on-chain canister can bypass the restriction via a composite-query inter-canister call. The query methods execute unconditionally and return a redacted but structurally complete response that leaks the existence, `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` of every confidential (non-disclosed) rate-limit rule.

## Finding Description
The `inspect_message` hook in `canister.rs` (L34–68) is the sole access-control gate for `get_config` (named `REPLICATED_QUERY_METHOD`): it traps any caller that is neither the authorized principal nor an API boundary-node principal. However, the three query methods themselves contain no rejection logic. Each one constructs an `AccessLevelResolver`, resolves the caller's level, and returns data — applying only confidentiality *formatting* for `RestrictedRead` callers, not an outright error.

`AccessLevelResolver::get_access_level` (`access_control.rs` L38–55) returns `RestrictedRead` for any caller that is neither the authorized principal nor an API boundary-node principal. For such callers, `ConfigConfidentialityFormatter::format` (`confidentiality_formatting.rs` L17–28) sets `is_redacted = true` and nulls out `rule_raw` and `description` for non-disclosed rules, but leaves `rule_id` and `incident_id` intact in every `OutputRule`. Similarly, `RuleConfidentialityFormatter` (`confidentiality_formatting.rs` L34–42) leaves `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` intact in `OutputRuleMetadata`.

The IC execution environment explicitly does not invoke `inspect_message` for inter-canister calls: `execute_inspect_message` takes a `SignedIngressContent` argument, making it structurally impossible to invoke for canister-to-canister messages. The IC supports composite queries, where a query method on canister A can call a query method on canister B; in this path, `inspect_message` is never triggered on canister B. The test at `rs/execution_environment/src/execution_environment/tests.rs:2754` confirms that a canister whose `inspect_message` rejects all ingress messages is still fully reachable via inter-canister calls.

The update methods `add_config` and `disclose_rules` are safe because they use the `WithAuthorization` wrapper (`access_control.rs` L72–98), which independently checks `AccessLevel::FullAccess` at execution time and returns `Unauthorized` otherwise.

## Impact Explanation
An attacker canister issues a composite-query inter-canister call to `get_config`, `get_rule_by_id`, or `get_rules_by_incident_id`. `inspect_message` is never invoked. The query executes with `RestrictedRead` access. The response leaks the existence of every non-disclosed (confidential) rate-limit rule, along with its `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version`. The actual `rule_raw` and `description` fields are redacted, but the structural metadata of confidential security rules — which are supposed to be invisible to unauthorized parties — is exposed. This is a concrete confidentiality breach of the boundary node rate-limit system, which is an explicitly in-scope boundary/API infrastructure component. This matches the Medium-to-High allowed impact: "Significant boundary/API infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
Any canister deployed on the IC can issue a composite-query call to the rate-limits canister. No privileged role, key material, or subnet-majority corruption is required. The attacker only needs to know the canister ID of the rate-limits canister (which is public). The attack is deterministic, repeatable, and requires no victim interaction.

## Recommendation
Add explicit caller-identity checks inside the query function bodies, mirroring the `WithAuthorization` pattern already used for `add_config` and `disclose_rules`. For `RestrictedRead` callers, return an `Unauthorized` error rather than a redacted response. The `inspect_message` hook should be treated as a best-effort pre-consensus filter (to save cycles), not as the authoritative access-control boundary. Apply this pattern to all three query methods: `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id`.

## Proof of Concept
1. Deploy an attacker canister on any subnet.
2. Implement a `#[query(composite = true)]` method in the attacker canister that calls `get_config(None)` on the rate-limits canister via `ic_cdk::call`.
3. Invoke the attacker canister's composite query method from any user principal.
4. `inspect_message` is not invoked on the rate-limits canister (inter-canister path).
5. `get_config` executes; `AccessLevelResolver` resolves the attacker canister's principal to `RestrictedRead`.
6. `ConfigGetter::get` returns a `ConfigResponse` with `is_redacted: true` but with all rule entries present, each carrying `rule_id` and `incident_id` — including rules that have never been publicly disclosed.
7. Verify: the integration test at `rs/boundary_node/rate_limits/integration_tests/tests/rate_limit_canister_tests.rs` (step 5, L154–187) already confirms that a `RestrictedRead` caller receives a response with `rule_id` and `incident_id` populated for non-disclosed rules, with only `rule_raw` and `description` set to `None`.
