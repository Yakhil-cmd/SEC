Audit Report

## Title
ICRC-21 Consent Message Hard Failure for u256 Token Amounts via `nat_to_u64` - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `nat_to_u64` helper in `responses.rs` converts a `Nat` token amount to `u64` for the `FieldsDisplay` consent message path. On the u256-token ledger variant (used by ckETH and all ckERC20 tokens), any transfer amount exceeding `u64::MAX` (~18.4 ETH at 18 decimals) causes `icrc21_canister_call_consent_message` to return `GenericError { error_code: 500, description: "Failed to convert tokens to u64" }`, blocking hardware wallet signing flows entirely. A secondary issue in `convert_tokens_to_string_representation` silently loses precision via `f64` conversion for the `GenericDisplay` path.

## Finding Description
In `packages/icrc-ledger-types/src/icrc21/responses.rs`, the private helper `nat_to_u64` is:

```rust
fn nat_to_u64(tokens: Nat) -> Result<u64, Icrc21Error> {
    tokens.0.to_u64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })
}
``` [1](#0-0) 

This is called from `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` whenever the `FieldsDisplayMessage` branch is taken: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The `Value::TokenAmount` variant hard-codes `amount: u64`: [6](#0-5) 

The public `#[update]` endpoint `icrc21_canister_call_consent_message` in the ICRC-1 ledger passes the decoded transfer amount directly into this path: [7](#0-6) 

The ledger is compiled in two variants; the u256 variant is used for ckETH and all ckERC20 tokens: [8](#0-7) 

For the `GenericDisplay` path, `convert_tokens_to_string_representation` uses `to_f64()`, which never returns `None` on `BigUint` but silently rounds for values above 2^53: [9](#0-8) 

The exploit path is direct: an unprivileged caller submits a `ConsentMessageRequest` with `device_spec: Some(DisplayMessageType::FieldsDisplay)` and a `TransferArg` whose `amount` exceeds `u64::MAX`. The `Decode!` call in `build_icrc21_consent_info` succeeds (the Candid `TransferArg.amount` is typed as `Nat`, not `u64`), the amount is passed to `add_amount`, `nat_to_u64` returns `None`, and the `?` operator propagates `GenericError { error_code: 500 }` back to the caller. No existing guard prevents this — the only size check is on the raw argument byte length, not on the numeric value of the amount. [10](#0-9) 

## Impact Explanation
This is a **High** impact finding under "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm." The ICRC-21 consent message endpoint is the standard hardware wallet signing flow for ICRC-1 ledgers. Any ICRC-21-compliant hardware wallet (e.g., Ledger) that requests `FieldsDisplay` will receive a `GenericError` for any ckETH transfer exceeding ~18.4 ETH, making it impossible to produce a consent message and therefore impossible to sign the transaction via the hardware wallet. This is a concrete, demonstrable degradation of the security model for large ckETH holders: they cannot use hardware wallets for full-balance or large transfers on the production mainnet ckETH canister. The `GenericDisplay` precision loss additionally means every non-trivial ckETH transfer displays a factually incorrect amount to the user, undermining the trust guarantee ICRC-21 is designed to provide.

## Likelihood Explanation
The ckETH canister is a production mainnet canister. Transfers exceeding 18.4 ETH are routine for institutional users and DeFi protocols. The endpoint is a standard public `#[update]` method reachable by any unprivileged ingress sender with no preconditions. The `TransferArg.amount` field is typed as `Nat` in the Candid interface, so any valid large-amount transfer argument will trigger the failure. The `GenericDisplay` precision loss affects all ckETH transfers above ~0.009 ETH (~2^53 wei), which covers essentially every non-trivial transfer.

## Recommendation
1. Replace `nat_to_u64` with a `Nat`-native representation. Change `Value::TokenAmount { amount: u64 }` to `Value::TokenAmount { amount: Nat }` (or `u128` as a minimum) and update the Candid `.did` files accordingly.
2. Replace the `f64` conversion in `convert_tokens_to_string_representation` with exact integer arithmetic using `BigUint` division and modulo to compute integer and fractional parts separately, avoiding any floating-point rounding.
3. Add integration tests for `icrc21_canister_call_consent_message` on the u256 ledger with amounts exceeding `u64::MAX` for both `FieldsDisplay` and `GenericDisplay` paths.

## Proof of Concept
```rust
// On the u256 ckETH ledger canister:
let large_amount = Nat::from(u64::MAX) + Nat::from(1_u64); // 18_446_744_073_709_551_616 wei (~18.4 ETH)
let transfer_arg = TransferArg {
    from_subaccount: None,
    to: some_account,
    amount: large_amount,
    fee: None,
    memo: None,
    created_at_time: None,
};
let request = ConsentMessageRequest {
    method: "icrc1_transfer".to_string(),
    arg: Encode!(&transfer_arg).unwrap(),
    user_preferences: ConsentMessageSpec {
        metadata: ConsentMessageMetadata { language: "en".to_string(), utc_offset_minutes: None },
        device_spec: Some(DisplayMessageType::FieldsDisplay),
    },
};
// Result: Err(GenericError { error_code: 500, description: "Failed to convert tokens to u64" })
// nat_to_u64 at responses.rs:330 returns None for Nat > u64::MAX, propagated via ?
```

This can be reproduced as a PocketIC integration test against the u256 ledger WASM with the `u256-tokens` feature enabled.
