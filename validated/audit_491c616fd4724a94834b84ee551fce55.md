Audit Report

## Title
Single Malicious Callee Can Exhaust Shared Composite Query Instruction Budget, Causing Permanent DoS - (File: rs/execution_environment/src/query_handler/query_call_graph.rs)

## Summary
The composite query execution framework initializes a single shared instruction budget equal to `MAX_INSTRUCTIONS_PER_COMPOSITE_QUERY_CALL = 5,000,000,000`. The per-sub-call instruction limit is capped at `max_instructions_per_query = MAX_INSTRUCTIONS_PER_QUERY_MESSAGE = 5,000,000,000` — the same value. A single malicious callee can therefore consume the entire shared budget in one execution, causing `evaluate_query_call_graph` to abort the whole composite query with `QueryCallGraphTotalInstructionLimitExceeded` on the very next DFS iteration. Any canister that fans out composite queries to a registry of user-controlled canisters is permanently vulnerable to this denial-of-service.

## Finding Description
`QueryContext::new` initializes `round_limits.instructions` from `max_query_call_graph_instructions`, which defaults to `MAX_INSTRUCTIONS_PER_COMPOSITE_QUERY_CALL = 5_000_000_000`:

```rust
// rs/execution_environment/src/query_handler/query_context.rs L152-153
let round_limits = RoundLimits {
    instructions: as_round_instructions(max_query_call_graph_instructions),
```

Before each sub-call, `execute_query` computes the per-call instruction limit as:

```rust
// query_context.rs L425-428
let instruction_limit = self.max_instructions_per_query.min(NumInstructions::new(
    self.round_limits.instructions.get().max(0) as u64,
));
```

`max_instructions_per_query` is `MAX_INSTRUCTIONS_PER_QUERY_MESSAGE = 5 * B` (`subnet_config.rs` L41). At the start of the first sub-call, both values are 5 B, so `min(5B, 5B) = 5B` — the first callee receives the entire graph budget as its instruction allowance.

After execution, the overhead deduction occurs:

```rust
// query_context.rs L857
self.round_limits.instructions -= self.instruction_overhead_per_query_call;
```

This deduction (50 M) is applied *after* execution and cannot prevent a callee from consuming the full 5 B. Once `round_limits.instructions` reaches zero or below, the next DFS loop iteration in `evaluate_query_call_graph` hits:

```rust
// query_call_graph.rs L73-79
if query_context.instruction_limit_reached() {
    let error = UserError::new(
        ErrorCode::QueryCallGraphTotalInstructionLimitExceeded,
        "Composite query calls exceeded the instruction limit.",
    );
    return QueryResponse::UserError(error);
}
```

This terminates the *entire* composite query as a `UserError`, not just the offending sub-call. All remaining callees in the DFS graph are never reached.

The existing test `composite_query_callgraph_max_instructions_is_enforced` (`tests.rs` L362–429) explicitly confirms that exceeding the graph budget causes the whole query to fail with `QueryCallGraphTotalInstructionLimitExceeded`.

## Impact Explanation
Any canister implementing an open-registry composite query pattern (DEX aggregator, portfolio tracker, on-chain index) that dynamically builds its call graph from a list that any principal can extend is permanently vulnerable. An attacker registers one malicious canister that spins a tight Wasm loop consuming all available instructions. Every composite query that includes that canister fails with `QueryCallGraphTotalInstructionLimitExceeded`, making the victim canister's composite query endpoint non-functional for all users. This matches the allowed impact: **Application/platform-level DoS** (High, $2,000–$10,000).

## Likelihood Explanation
The attack requires only an unprivileged canister deployment followed by registration in the target's open registry — no special privileges, no governance access, no node control. The attacker pays only the one-time canister creation cost; each subsequent query failure costs the attacker nothing (instructions are charged to the query caller). The malicious canister persists indefinitely. The attack is fully repeatable and deterministic.

## Recommendation
Decouple the per-sub-call instruction limit from the total graph budget. Cap each sub-call at `max_query_call_graph_instructions / MAX_QUERY_CALL_DEPTH` (≈ 833 M with current defaults of 5 B / 6), so no single callee can exhaust the shared pool. Alternatively, introduce a dedicated `max_instructions_per_composite_query_sub_call` configuration parameter distinct from `max_instructions_per_query`, and set it to a fraction of the total graph budget.

## Proof of Concept
1. Attacker deploys canister **M** whose query method executes a tight Wasm loop until all available instructions are consumed before returning.
2. Legitimate canister **A** implements a composite query that sequentially calls canisters from a registry; attacker registers **M** in that registry.
3. User sends a composite query to **A**.
4. **A** calls **M**; `execute_query` grants **M** `min(5B, 5B) = 5B` instructions; **M** consumes all of them.
5. On the next DFS iteration in `evaluate_query_call_graph`, `instruction_limit_reached()` returns `true`; the function returns `QueryResponse::UserError(QueryCallGraphTotalInstructionLimitExceeded)`.
6. All remaining callees are never executed; the user receives an error.
7. Every subsequent composite query to **A** that includes **M** fails identically at zero marginal cost to the attacker.

A deterministic integration test can reproduce this by setting `max_query_call_graph_instructions` to `MAX_INSTRUCTIONS_PER_QUERY_MESSAGE` (the production default), deploying a universal canister that loops until instruction exhaustion, and asserting that a composite query fanning out to `[M, B]` returns `QueryCallGraphTotalInstructionLimitExceeded` while **B** is never reached — mirroring the structure of the existing `composite_query_callgraph_max_instructions_is_enforced` test.
