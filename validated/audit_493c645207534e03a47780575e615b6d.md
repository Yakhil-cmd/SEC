### Title
Unicode Homoglyph Bypass in `validate_token_symbol` Allows Visually-ICP-Identical SNS Token Symbols — (`rs/nervous_system/common/src/ledger_validation.rs`)

---

### Summary

`validate_token_symbol` uses Rust's `.len()` (byte count) for length enforcement and `.to_uppercase()` for ban-list matching. Neither check catches Unicode homoglyphs. An attacker can submit a `CreateServiceNervousSystem` NNS proposal with `token_symbol = "ІСР"` (Cyrillic І U+0406, С U+0421, Р U+0420) — visually indistinguishable from "ICP" — and it passes all validation gates, resulting in a deployed SNS token whose symbol renders identically to ICP in every standard UI.

---

### Finding Description

The full function is:

```rust
pub fn validate_token_symbol(token_symbol: &str) -> Result<(), String> {
    if token_symbol.len() > MAX_TOKEN_SYMBOL_LENGTH {   // byte count, not char count
        ...
    }
    if token_symbol.len() < MIN_TOKEN_SYMBOL_LENGTH {   // byte count
        ...
    }
    if token_symbol != token_symbol.trim() { ... }

    if BANNED_TOKEN_SYMBOLS.contains(&token_symbol.to_uppercase().as_ref()) {
        // "ІСР".to_uppercase() == "ІСР" (Cyrillic), not "ICP" (Latin)
        ...
    }
    Ok(())
}
``` [1](#0-0) 

Two independent bugs combine:

**Bug 1 — `.len()` is byte count, not character count.**
In Rust, `str::len()` returns the number of UTF-8 bytes. The comment on `MAX_TOKEN_SYMBOL_LENGTH` says "characters," but the check enforces bytes. [2](#0-1) [3](#0-2) 

**Bug 2 — `.to_uppercase()` is Unicode-aware but not homoglyph-aware.**
`"ІСР".to_uppercase()` returns `"ІСР"` (Cyrillic uppercase), which is not equal to `"ICP"` (Latin). The ban list check is a plain string equality after uppercasing, with no Unicode normalization, confusable-character detection, or ASCII-only enforcement. [4](#0-3) [5](#0-4) 

**Concrete bypass with "ІСР":**

| Character | Unicode | UTF-8 bytes | Looks like |
|---|---|---|---|
| І | U+0406 | 2 (0xD0 0x86) | Latin I |
| С | U+0421 | 2 (0xD0 0xA1) | Latin C |
| Р | U+0420 | 2 (0xD0 0xA0) | Latin P |

Total: 6 bytes. `6 < 10` → passes length check. `"ІСР".to_uppercase() == "ІСР" ≠ "ICP"` → passes ban check. All other checks (trim, min length) also pass.

### Exploit Path

1. Attacker (any NNS neuron holder) submits a `CreateServiceNervousSystem` proposal with `ledger_parameters.token_symbol = "ІСР"`.
2. At proposal submission, `validate_create_service_nervous_system` calls only `SnsInitPayload::try_from` (field conversion), which does **not** invoke `validate_token_symbol`. [6](#0-5) 
3. NNS community votes. The Cyrillic characters are visually identical to "ICP" in proposal UIs; reviewers may not notice.
4. On execution, `execute_create_service_nervous_system_proposal` calls `sns_init_payload.validate_post_execution()`, which calls `validate_token_symbol("ІСР")` — this **passes**. [7](#0-6) 
5. SNS is deployed with token symbol "ІСР", visually identical to "ICP" in all standard wallets and DEX UIs. [8](#0-7)

### Impact Explanation

The invariant encoded in `BANNED_TOKEN_SYMBOLS` — that no SNS token may use the symbol "ICP" or "DFINITY" — is violated. A deployed SNS token with symbol "ІСР" is indistinguishable from ICP in every rendering context that does not perform Unicode code-point inspection. Users can be deceived into purchasing the fraudulent token. The financial impact depends on attacker execution (swap liquidity, marketing), but the protocol-level invariant is concretely broken.

The ">$1M illegal minting" framing in the question is inaccurate — the attacker cannot mint ICP. The actual impact is fraud enablement: users lose funds by purchasing a confusable token, not by having ICP minted against them.

---

### Likelihood Explanation

- Requires submitting an NNS proposal (needs ICP stake for a neuron), but this is permissionless for any token holder.
- The Cyrillic homoglyphs are well-known and trivially constructable.
- NNS voters reviewing proposals in standard UIs would see "ICP" and likely approve without noticing the encoding difference.
- The bug is locally testable with a single unit test call.

---

### Recommendation

Replace the length and ban checks with:

1. **ASCII-only enforcement**: Reject any `token_symbol` containing non-ASCII characters (`!token_symbol.is_ascii()`). This is the simplest and most robust fix, consistent with how `CkTokenSymbol` is validated in `rs/ethereum/cketh/minter/src/erc20.rs`. [9](#0-8) 

2. **Character count, not byte count**: Replace `token_symbol.len()` with `token_symbol.chars().count()` for the length checks, to match the documented semantics ("characters"). [10](#0-9)

### Proof of Concept

```rust
#[test]
fn cyrillic_icp_homoglyph_bypasses_validation() {
    // Cyrillic І (U+0406), С (U+0421), Р (U+0420) — visually identical to "ICP"
    let fake_icp = "\u{0406}\u{0421}\u{0420}";
    assert_eq!(fake_icp.len(), 6);           // 6 bytes, passes 10-byte limit
    assert_eq!(fake_icp.chars().count(), 3); // 3 chars
    assert_ne!(fake_icp.to_uppercase(), "ICP"); // Cyrillic uppercase ≠ Latin "ICP"
    // This must return Ok(()) under the current implementation — demonstrating the bug:
    assert!(validate_token_symbol(fake_icp).is_ok());
}

pub fn validate_token_symbol(token_symbol: &str) -> Result<(), String> {
    if token_symbol.len() > MAX_TOKEN_SYMBOL_LENGTH {
        return Err(format!(
            "Error: token-symbol must be fewer than {} characters, given character count: {}",
            MAX_TOKEN_SYMBOL_LENGTH,
            token_symbol.len()
        ));
    }

    if token_symbol.len() < MIN_TOKEN_SYMBOL_LENGTH {
        return Err(format!(
            "Error: token-symbol must be greater than {} characters, given character count: {}",
            MIN_TOKEN_SYMBOL_LENGTH,
            token_symbol.len()
        ));
    }

    if token_symbol != token_symbol.trim() {
        return Err("Token symbol must not have leading or trailing whitespaces".to_string());
    }

    if BANNED_TOKEN_SYMBOLS.contains(&token_symbol.to_uppercase().as_ref()) {
        return Err("Banned token symbol, please chose another one.".to_string());
    }

    Ok(())
}
