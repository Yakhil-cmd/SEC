### Title
Fixed-Capacity `ArrayVec` Overflow in Multiblock Batch Log Accumulation Causes Proving Panic - (File: `basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`)

---

### Summary

`ZKBatchDataKeeper.logs_storage` is declared as `ArrayVec<Bytes32, 16384>` — a fixed-capacity stack-allocated array. In the multiblock batch proving path, `apply_to_array_vec` is called once per block to push all block-level L2→L1 log hashes into this single accumulator. Because the per-block log limit equals the accumulator's total capacity (both 16,384), any batch containing two or more blocks that together emit more than 16,384 logs will cause `ArrayVec::push` to panic, making the batch unprovable while the forward run succeeds.

---

### Finding Description

**Vulnerability class:** Forward/proving divergence → valid-execution unprovability.

**Root cause — `ZKBatchDataKeeper.logs_storage` fixed capacity:**

`ZKBatchDataKeeper` holds a batch-level log accumulator with a hard-coded capacity of 16,384 entries: [1](#0-0) 

In the multiblock batch proving post-tx operation, after each block is finalized, all of that block's log hashes are pushed into this accumulator: [2](#0-1) 

The `apply_to_array_vec` implementation uses bare `ArrayVec::push`, which panics when the array is full: [3](#0-2) 

**The per-block log limit is also 16,384:** [4](#0-3) 

The block-level enforcement in the ZK tx loop rejects any transaction that would push a single block past 16,384 logs: [5](#0-4) 

This means a single block can legally contain up to 16,384 logs. When a second block in the same batch contributes even one additional log, the batch-level `ArrayVec` overflows and panics.

**The forward run is unaffected** — it does not use `ZKBatchDataKeeper` or `apply_to_array_vec`. The sequencer processes each block independently and enforces only the per-block limit. The panic is exclusive to the proving path.

---

### Impact Explanation

When the prover executes the RISC-V binary for a multiblock batch where the total log count across all blocks exceeds 16,384, `ArrayVec::push` panics unconditionally (the `arrayvec` crate's `push` panics on overflow; there is no graceful error return). This terminates the proving run with an unrecoverable error.

The consequence is that a legitimately sequenced batch — one that passed all forward-run validations — cannot be proven. The chain cannot produce a ZK proof for that batch, blocking L1 settlement. If the operator cannot selectively re-batch (which requires re-executing blocks), the chain is effectively halted at the L1 settlement layer until the issue is resolved out-of-band.

---

### Likelihood Explanation

The multiblock batch mode is the production proving path for ZKsync OS rollups. Any batch containing two or more blocks where the combined log count exceeds 16,384 triggers the panic. A single block at the per-block maximum (16,384 logs) followed by any block with at least one L2→L1 log is sufficient. L2→L1 logs are emitted by:

1. Every L1→L2 (priority) transaction result — one log per L1 tx.
2. Any contract calling the L1 Messenger system hook.

An unprivileged attacker can submit many L1→L2 transactions (which are accepted by the operator) to fill one block to the log limit, then ensure the next block in the same batch has at least one log (trivially achieved by any L1→L2 tx or L1 Messenger call). This is a low-cost, externally reachable attack.

---

### Recommendation

Replace the fixed-capacity `ArrayVec<Bytes32, 16384>` in `ZKBatchDataKeeper` with a dynamically growable structure (e.g., `Vec<Bytes32, A>` using the system allocator), or enforce a batch-level log cap that accounts for the number of blocks in the batch. Additionally, change `apply_to_array_vec` to use `try_push` and propagate an error instead of panicking, so that capacity exhaustion is handled gracefully rather than aborting the prover. [6](#0-5) 

---

### Proof of Concept

1. Configure the system for multiblock batch proving (`ZKHeaderStructurePostTxOpProvingMultiblockBatch`).
2. Submit enough L1→L2 transactions to fill **Block 1** with exactly 16,384 L2→L1 logs (one per L1 tx result). The per-block check passes: `logs_used == MAX_NUMBER_OF_LOGS` is not `> MAX_NUMBER_OF_LOGS`.
3. Submit any single L1→L2 transaction in **Block 2**, generating 1 log.
4. Trigger the multiblock batch proving run.
5. In `post_tx_op_proving_multiblock_batch.rs`, after Block 1 is processed, `apply_to_array_vec` fills `batch_data.logs_storage` to capacity (16,384 entries).
6. After Block 2 is processed, `apply_to_array_vec` calls `array_vec.push(log.hash())` on a full `ArrayVec` → **panic**.
7. The forward run for the same two blocks completes successfully; only the proving run panics. [7](#0-6) [1](#0-0)
