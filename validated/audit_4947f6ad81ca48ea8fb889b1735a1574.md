### Title
Any SNS Swap Canister Can Settle Another SNS's Neurons' Fund Participation, Redirecting Minted ICP - (`rs/nns/governance/src/governance.rs`)

### Summary

The `settle_neurons_fund_participation` function in NNS Governance authorizes any registered SNS Swap canister to settle the Neurons' Fund participation for **any** NNS proposal, not just the proposal associated with the calling swap canister. Combined with the fact that the `sns_governance_canister_id` destination for minted ICP is taken directly from the caller-supplied request without cross-validation against the proposal's stored SNS governance canister, a malicious SNS Swap operator can redirect Neurons' Fund ICP reserved for a victim SNS to their own SNS governance treasury.

### Finding Description

In `settle_neurons_fund_participation`, the authorization check calls `is_canister_id_valid_swap_canister_id`, which queries SNS-W's `list_deployed_snses` and verifies only that the caller is *any* registered SNS Swap canister: [1](#0-0) 

The function then proceeds to use the caller-supplied `sns_governance_canister_id` from the `Committed` variant of the request as the ICP mint destination: [2](#0-1) 

The `is_canister_id_valid_swap_canister_id` helper confirms only that the caller appears in the global SNS-W registry — it does not verify that the caller is the swap canister *associated with the specific `nns_proposal_id`* in the request: [3](#0-2) 

The code even contains an explicit acknowledgment of this design gap in a comment: [4](#0-3) 

> "Note that a Swap could settle each other's participation."

The `CreateServiceNervousSystem` proposal that creates an SNS records the expected swap canister ID, but this stored value is never compared against the actual caller during settlement. The `sns_governance_canister_id` field in the `Committed` request is accepted verbatim: [5](#0-4) 

Once settlement completes, the proposal lifecycle is set to terminal and the idempotency guard returns the cached result to any subsequent caller: [6](#0-5) 

This means the victim SNS's legitimate `finalize_swap` call will receive a cached success response, but the ICP was already sent to the attacker's governance canister.

### Impact Explanation

A malicious SNS Swap operator (SNS-A) can:

1. Monitor NNS proposals for a victim SNS-B whose swap has committed and whose Neurons' Fund participation has not yet been settled.
2. Call `settle_neurons_fund_participation` on NNS Governance with `nns_proposal_id = P_B` (SNS-B's proposal) and `sns_governance_canister_id = A_gov` (SNS-A's own governance canister).
3. NNS Governance mints the Neurons' Fund ICP reserved for SNS-B and sends it to SNS-A's governance treasury.
4. SNS-B's subsequent `finalize_swap` call hits the idempotency path and receives a success response, but the ICP was never delivered to SNS-B's governance canister. SNS-B's Neurons' Fund participation is effectively stolen.

The maturity of NNS neurons in the Neurons' Fund is permanently consumed for the wrong SNS, constituting a ledger conservation violation and a governance authorization bypass.

### Likelihood Explanation

The attacker must control a legitimately deployed SNS Swap canister, which requires an NNS governance proposal to pass. However, many SNSes have already been deployed on mainnet, and any of their swap canister operators could exploit this. The attack requires only a single inter-canister call timed before the victim SNS's `finalize_swap` executes `settle_neurons_fund_participation`. The proposal ID of any SNS swap is public on-chain information. No key compromise, threshold attack, or social engineering is required beyond operating a deployed SNS.

### Recommendation

1. **Bind settlement authorization to the specific proposal**: After looking up the `ProposalData` for `nns_proposal_id`, extract the expected swap canister ID from the stored `CreateServiceNervousSystem` action and verify it matches `caller`. This replaces the overly broad "any SNS swap" check with a per-proposal check.

2. **Validate `sns_governance_canister_id` against proposal data**: The `sns_governance_canister_id` provided in the `Committed` request should be cross-checked against the SNS governance canister ID recorded in the `CreateServiceNervousSystem` proposal action, rejecting any mismatch.

3. **Document the trust boundary**: If cross-settlement is intentionally permitted for operational reasons (e.g., retry by a different swap), document the rationale and add explicit guards to prevent destination redirection.

/// Given a target_canister_id, is it a CanisterId of a deployed SNS recorded by
/// the SNS-W canister.
async fn is_canister_id_valid_swap_canister_id(
    target_canister_id: CanisterId,
    env: &dyn Environment,
) -> Result<(), String> {
    let list_deployed_snses_response = env
        .call_canister_method(
            SNS_WASM_CANISTER_ID,
            "list_deployed_snses",
            Encode!(&ListDeployedSnsesRequest {}).expect(""),
        )
        .await
        .map_err(|err| {
            format!(
                "Failed to call the list_deployed_snses method on sns_wasm ({SNS_WASM_CANISTER_ID}): {err:?}",
            )
        })?;

    let list_deployed_snses_response =
        Decode!(&list_deployed_snses_response, ListDeployedSnsesResponse).map_err(|err| {
            format!(
                "Unable to decode response as ListDeployedSnsesResponse: {err}. reply_bytes = {list_deployed_snses_response:#?}",
            )
        })?;

    let is_swap = list_deployed_snses_response
        .instances
        .iter()
        .any(|sns| sns.swap_canister_id == Some(target_canister_id.into()));
    if !is_swap {
        return Err(format!(
            "target_swap_canister_id is not the ID of any swap canister known to sns_wasm: {target_canister_id}"
        ));
    }

    Ok(())
