Audit Report

## Title
`AccountState::transfer()` Unconditionally Removes Neuron IDs and Returns `Ok(())` on Partial Failure, Permanently Locking Genesis Account Owners Out of `claim_neurons` - (File: rs/nns/gtc/src/lib.rs)

## Summary
`AccountState::transfer()` removes each neuron ID from `self.neuron_ids` unconditionally — before checking whether the `transfer_gtc_neuron` inter-canister call succeeded — and always returns `Ok(())` even when individual calls fail. Because `forward_whitelisted_unclaimed_accounts` (callable by any unprivileged principal after 188 days) sets `has_forwarded = true` on any `Ok(())` return, a single transient failure during the per-neuron loop permanently denies the account owner access to `claim_neurons`, while the failed neurons remain stuck under GTC control in the Governance canister with no on-chain recovery path.

## Finding Description
In `AccountState::transfer()` (`rs/nns/gtc/src/lib.rs`, lines 188–209), the loop over `neuron_ids` calls `GovernanceCanister::transfer_gtc_neuron` and then unconditionally executes `self.neuron_ids.retain(|id| id != &neuron_id)` at line 192 — before inspecting `result`. Whether the call succeeds or fails, the neuron ID is removed from `self.neuron_ids`. After the loop, `Ok(())` is returned unconditionally at line 209 regardless of how many neurons ended up in `self.failed_transferred_neurons`.

The caller `forward_whitelisted_unclaimed_accounts` (lines 118–119) treats any `Ok(_)` as complete success and sets `account.has_forwarded = true`. The `claim_neurons` entry point (lines 58–60) then permanently rejects the account owner with `"Account has previously forwarded its funds"`. Because `neuron_ids` was also emptied, even removing the `has_forwarded` guard would leave nothing to claim. The same logic flaw applies to `donate_account` (line 89–90), which uses `transfer().await?` — but since `transfer()` never returns `Err` on partial failure, `has_donated` is set even when neurons were not fully transferred.

`forward_whitelisted_unclaimed_accounts` is exposed as an unprivileged update call with no caller authentication; the only guard is the time-based `assert_forward_whitelisted_unclaimed_accounts_can_be_called` check, which passes after 188 days — a window that has long elapsed on mainnet.

## Impact Explanation
Any genesis account owner whose whitelisted account experiences even one transient `transfer_gtc_neuron` failure during forwarding permanently loses the ability to claim their neurons. The failed neurons are neither credited to the custodian nor recoverable by the owner; they remain frozen under GTC control in the Governance canister. This constitutes permanent loss of access to governance assets (genesis neurons representing ICP stake) for affected account owners, matching the High bounty impact: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds." Seed Round accounts hold up to 48 neurons each, increasing the probability of at least one inter-canister call failure per account and the magnitude of loss per affected account.

## Likelihood Explanation
Any unprivileged principal can trigger `forward_whitelisted_unclaimed_accounts` today (188-day window elapsed on mainnet). The attack requires no special role, no key material, and no victim interaction. A transient Governance canister error — message queue saturation, a canister trap, or temporary unavailability — during any one of the per-neuron `transfer_gtc_neuron` calls is sufficient to trigger the partial-failure path. Accounts with many neurons (e.g., 48 per Seed Round account) have a higher probability of encountering at least one failure. The condition is repeatable: each invocation of `forward_whitelisted_unclaimed_accounts` re-processes accounts not yet marked `has_forwarded`, so repeated calls increase cumulative exposure.

## Recommendation
Move `self.neuron_ids.retain(|id| id != &neuron_id)` inside the `Ok(_)` arm so that failed neurons remain in `self.neuron_ids` and are retryable:

```rust
match result {
    Ok(_) => {
        self.neuron_ids.retain(|id| id != &neuron_id);
        self.successfully_transferred_neurons.push(donated_neuron);
    }
    Err(e) => {
        donated_neuron.error = Some(e.to_string());
        self.failed_transferred_neurons.push(donated_neuron);
    }
}
```

Additionally, `transfer()` should return `Err` (or a partial-failure indicator) when `self.failed_transferred_neurons` is non-empty, so that `forward_whitelisted_unclaimed_accounts` does not set `has_forwarded = true` until all neurons have been successfully transferred. Alternatively, `has_forwarded` should only be set when `account.failed_transferred_neurons.is_empty()` after the call.

## Proof of Concept
1. Confirm the 188-day window has elapsed (it has on mainnet).
2. Any principal sends an ingress `forward_whitelisted_unclaimed_accounts` call to the GTC canister.
3. The GTC iterates over whitelisted, unclaimed, undonated, unforwarded accounts and calls `AccountState::transfer()` for each.
4. During the per-neuron loop, the Governance canister's `transfer_gtc_neuron` returns an error for neuron N (e.g., due to a transient queue-full or canister trap).
5. `self.neuron_ids.retain(...)` removes N unconditionally; N is pushed to `failed_transferred_neurons`.
6. `transfer()` returns `Ok(())`.
7. `has_forwarded = true` is set for the account.
8. The account owner calls `claim_neurons` → receives `"Account has previously forwarded its funds"`.
9. Neuron N remains under GTC control in the Governance canister, neither claimable by the owner nor credited to the custodian, with no on-chain recovery path.

A deterministic integration test can reproduce this by mocking `transfer_gtc_neuron` to return an error for one neuron ID in a multi-neuron account, then asserting that `has_forwarded == true`, `neuron_ids` is empty, and `failed_transferred_neurons` is non-empty after the call.
