### Title
Truncating Division in `native_per_pubdata` Calculation Allows Underpayment for Pubdata - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio used to charge native resources for pubdata consumption is computed with floor (truncating) division via `wrapping_div`, while the analogous `native_per_gas` ratio uses ceiling division (`div_ceil`). This asymmetric rounding causes users to systematically underpay native resources for pubdata. In the extreme case where `pubdata_price < native_price`, the result truncates to zero, meaning a transaction sender can generate arbitrary pubdata while paying **zero** native resources for it.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, two ratios are computed:

```rust
// native_per_gas: rounds UP (protects protocol)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata: rounds DOWN (favors sender)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [1](#0-0) 

`native_per_pubdata` is subsequently used in `get_resources_to_charge_for_pubdata` to compute the native cost of all pubdata generated by the transaction:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), `wrapping_div` yields `0`. The user is then charged `0` native resources for every byte of pubdata they generate, regardless of how much pubdata the transaction produces. Even when `pubdata_price >= native_price` but is not exactly divisible, the user is undercharged by up to `(native_price - 1)` native units per pubdata byte, accumulating across all pubdata bytes.

The same truncating division appears in `api/src/helpers.rs`: [3](#0-2) 

---

### Impact Explanation

The native resource is the ZKsync OS accounting unit for proving and pubdata publication costs. Per the double-resource accounting model, pubdata costs are charged from native resources: [4](#0-3) 

When `native_per_pubdata` is truncated to zero, the protocol publishes pubdata to L1 at its own cost while the transaction sender pays nothing for it. This is a direct financial loss to the protocol/operator proportional to the pubdata generated. A sender can craft transactions that write many storage slots (maximizing pubdata) while paying only EVM gas for the EVM execution, with zero native resource cost for the pubdata itself.

**Concrete scenario:**
- `pubdata_price = 9`, `native_price = 10`
- `native_per_pubdata = 9 / 10 = 0` (truncated)
- A transaction writing 1000 storage slots (~32,000 pubdata bytes) pays 0 native resources for pubdata
- The protocol must still publish those 32,000 bytes to L1

---

### Likelihood Explanation

The operator sets `native_price` and `pubdata_price` via block metadata. These values are independently derived from L1 gas prices and proving costs. It is realistic for `pubdata_price < native_price` to occur during normal operation, particularly when L1 calldata costs are low relative to proving costs. Any unprivileged L2 transaction sender can exploit this by submitting transactions that generate pubdata (e.g., SSTORE opcodes). No special access is required. [5](#0-4) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (rounds down, can yield 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...

// After (rounds up, protects protocol):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...
```

Apply the same fix in `api/src/helpers.rs` at line 427. This ensures the protocol is never undercompensated for pubdata costs, mirroring the ceiling-division approach already used for `native_per_gas`. [6](#0-5) 

---

### Proof of Concept

1. Operator sets block context: `native_price = 10`, `pubdata_price = 9`, `eip1559_basefee = 1`.
2. Attacker submits an EIP-1559 transaction with `max_fee_per_gas = 1`, `gas_limit = 10_000_000`, calling a contract that executes 500 `SSTORE` operations (each writing a new storage slot, generating ~32 bytes of pubdata each = ~16,000 pubdata bytes total).
3. In `validate_and_compute_fee_for_transaction`:
   - `native_per_gas = ceil(1 / 10) = 1`
   - `native_per_pubdata = floor(9 / 10) = 0`
4. In `get_resources_to_charge_for_pubdata`: `native = 16000 * 0 = 0`.
5. The transaction executes, generates 16,000 bytes of pubdata, and pays **zero** native resources for it.
6. The protocol publishes 16,000 bytes to L1 at its own cost. [7](#0-6)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
