Audit Report

## Title
Cross-Subaccount Allowance Enumeration via Incomplete Owner-Only Boundary Check in `get_allowances` — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function iterates over a `StableBTreeMap` keyed by `AccountSpender`, which sorts entries by `(owner, effective_subaccount, spender)`. The loop termination guard at line 1204 only checks `account_spender.account.owner != from.owner`, ignoring the subaccount field. As a result, a query for `from_account = (P, sub1)` will continue iterating into `(P, sub2)`, `(P, sub3)`, etc., returning allowances belonging to other subaccounts of the same principal. This allows any unprivileged caller to enumerate all allowances across every subaccount of an arbitrary principal.

## Finding Description
**Root cause:** In `rs/ledger_suite/icrc1/ledger/src/lib.rs` at line 1204:

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

The `AccountSpender::Ord` implementation (lines 372–378) sorts by the full `Account` (owner then `effective_subaccount()`), then by spender. So the map is ordered as `(owner_bytes, subaccount_bytes, spender_owner_bytes, spender_subaccount_bytes)`. When the iterator crosses from `(P, sub1, ...)` to `(P, sub2, ...)`, the owner field is still `P`, so the guard never fires. All allowances for all subaccounts of `P` that sort lexicographically after `sub1` are returned.

**Exploit path:**
1. Attacker calls `icrc103_get_allowances` (a public `#[query]` endpoint, lines 1214–1231 of `main.rs`) with `from_account = (P, [0u8; 32])` — the lexicographically smallest subaccount.
2. `get_allowances` starts iteration at `(P, [0u8;32], min_spender)` and iterates forward.
3. The guard at line 1204 only breaks on owner change, so all entries `(P, any_subaccount, any_spender)` are returned.
4. Attacker paginates via `prev_spender` to exhaust the full set.

**Why existing checks fail:** `Account::PartialEq` (lines 40–44 of `account.rs`) already normalises `None` and `Some([0u8;32])` identically via `effective_subaccount()`, but this correct equality is never used in the loop guard — only the raw `.owner` field is compared.

## Impact Explanation
This is a concrete privacy violation in the ICRC-1 ledger, an in-scope financial integration. Any caller can enumerate the complete set of allowances across all subaccounts of any principal: which subaccounts hold active approvals, the identity of each spender, and the approved amounts. This constitutes significant ledger security impact with concrete user harm — financial relationship data and subaccount usage patterns that users reasonably expect to be private are fully exposed to any observer. This fits the **High ($2,000–$10,000)** impact class: "Significant ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation
The endpoint is a public query callable by any principal with no fee, no privilege, and no on-chain transaction required. The attacker supplies an arbitrary `from_account` — there is no restriction to the caller's own principal. A single well-formed query suffices; pagination allows full enumeration. Exploitation is trivially repeatable against any target principal.

## Recommendation
Replace the owner-only check at line 1204 with a full `Account` comparison, which already normalises `None`/`Some([0u8;32])` correctly:

```rust
// Before (line 1204):
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```

`Account::eq` uses `effective_subaccount()` for both sides, so the existing behaviour for the default-subaccount equivalence (`None` == `Some([0u8;32])`) is preserved, while enumeration is correctly scoped to the exact `(owner, subaccount)` pair supplied by the caller.

## Proof of Concept
Using the existing state-machine test harness in `rs/ledger_suite/tests/sm-tests/src/lib.rs`:

```rust
let sub1 = [1u8; 32];
let sub2 = [2u8; 32];
let account1 = Account { owner: P, subaccount: Some(sub1) };
let account2 = Account { owner: P, subaccount: Some(sub2) };
let spender_a = Account { owner: SPENDER_A, subaccount: None };
let spender_b = Account { owner: SPENDER_B, subaccount: None };

// Fund both subaccounts and create approvals
send_approval(&env, ledger, P, &ApproveArgs { from_subaccount: Some(sub1), spender: spender_a, amount: 100.into(), .. });
send_approval(&env, ledger, P, &ApproveArgs { from_subaccount: Some(sub2), spender: spender_b, amount: 200.into(), .. });

// Query with from_account = (P, sub1) — should return ONLY sub1's allowance
let args = GetAllowancesArgs { from_account: Some(account1), prev_spender: None, take: None };
let allowances = list_allowances(&env, ledger, ATTACKER, args).unwrap();

// BUG: returns 2 (both sub1 and sub2 allowances)
// EXPECTED: returns 1 (only sub1's allowance)
assert_eq!(allowances.len(), 1);  // FAILS with current code
assert!(allowances.iter().all(|a| a.from_account == account1));
```

The existing test suite (lines 2985–3079 of `sm-tests/src/lib.rs`) only tests the `None`/`Some([0u8;32])` equivalence case and a single subaccount per principal — it does not cover two distinct non-default subaccounts under the same principal, which is the scenario that exposes the bug.
