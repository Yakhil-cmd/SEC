### Title
Manipulated Gzip Footer Inflates Compilation Cost Charged to Round Instruction Budget - (`rs/execution_environment/src/hypervisor.rs`)

---

### Summary

In `Hypervisor::create_execution_state`, the uncompressed size of a gzip-compressed Wasm module is read from the attacker-controlled gzip footer via `decoded_wasm_size()` and used to compute a `compilation_cost` that is charged to the shared `round_limits.instructions` budget in the compilation-failure path. Because `decoded_wasm_size()` performs no size-cap validation, an unprivileged canister developer can submit a valid gzip blob whose footer claims ~4 GiB of uncompressed data. The actual compilation then fails (the real decompressed size does not match, or exceeds `wasm_max_size`), and the error branch charges `cost_to_compile_wasm_instruction × 4 GiB` instructions against the round budget — far more than the attacker pays in cycles — starving other canisters in the same round.

---

### Finding Description

`Hypervisor::create_execution_state` in `rs/execution_environment/src/hypervisor.rs` estimates the compilation cost before invoking the Wasm executor:

```rust
let wasm_size_result = decoded_wasm_size(canister_module.as_slice());
let wasm_size = match wasm_size_result {
    Ok(size) => std::cmp::max(size, canister_module.len()),
    Err(_)   => canister_module.len(),
};
let compilation_cost = self.cost_to_compile_wasm_instruction * wasm_size as u64;
``` [1](#0-0) 

`decoded_wasm_size` is implemented in `rs/embedders/src/wasm_utils/decoding.rs`. For a gzip module it reads the last 4 bytes of the blob (the ISIZE field of the gzip footer) and returns that value as the uncompressed size — with no upper-bound check:

```rust
isize_bytes.copy_from_slice(&module_bytes[module_bytes.len() - 4..module_bytes.len()]);
let uncompressed_size = u32::from_le_bytes(isize_bytes) as usize;
return Ok((WasmEncoding::Gzip, uncompressed_size));
``` [2](#0-1) 

The file's own documentation warns explicitly that this value is untrusted:

> "If the Wasm is gzipped, then the returned size cannot be trusted. It would come from the gzip footer which could have been manipulated." [3](#0-2) 

When `wasm_executor.create_execution_state` subsequently calls `decode_wasm` (which *does* enforce `wasm_max_size`), it rejects a module whose footer claims more than the allowed maximum:

```rust
if uncompressed_size as u64 > max_size.get() {
    return Err(WasmValidationError::ModuleTooLarge { ... });
}
``` [4](#0-3) 

The executor therefore returns `Err`, and `create_execution_state` takes the error branch, charging the inflated estimate to the shared round budget:

```rust
Err(err) => {
    let total_cost = self.create_execution_state_base_cost + compilation_cost;
    round_limits.instructions -= as_round_instructions(total_cost);
    (total_cost, Err(err))
}
``` [5](#0-4) 

The success path uses the *actual* compilation cost returned by the executor, so it is not affected. Only the error path uses the untrusted estimate.

The canister's cycles are prepaid based on the *message instruction limit* (not the inflated estimate):

```rust
let prepaid_execution_cycles = self.cycles_account_manager.prepay_execution_cycles(
    ...
    original.execution_parameters.instruction_limits.message(),
    ...
)?;
``` [6](#0-5) 

This creates a direct asymmetry: the attacker pays cycles proportional to the message instruction limit (≈ 200 B instructions for `install_code`), but `round_limits.instructions` is reduced by `cost_to_compile_wasm_instruction × 4 GiB` — potentially orders of magnitude more.

---

### Impact Explanation

`round_limits.instructions` is the shared instruction budget for an entire execution round on a subnet. Reducing it by `cost_to_compile_wasm_instruction × 4 GiB` (up to ~4 B instructions with a per-byte cost of 1) against a typical round budget of ~7 B instructions can exhaust the round budget in a single malicious `install_code` call. Canisters scheduled later in the same round receive zero instructions and are skipped, constituting a targeted denial-of-service against all other canisters on the subnet for that round. The attacker's own cycles cost is bounded by the normal `install_code` prepayment, making the attack economically cheap relative to the damage caused.

---

### Likelihood Explanation

Any canister developer (unprivileged ingress sender) can craft a gzip blob with a manipulated 4-byte footer using standard tools. The attack requires only:
1. A canister with enough cycles to cover the normal `install_code` prepayment.
2. A gzip file whose footer's ISIZE field is set to a value exceeding `wasm_max_size` (e.g., `0xFFFFFFFF`).

No privileged access, no threshold corruption, and no social engineering is required. The existing test `test_decode_large_compressed_module_with_tweaked_size` in `rs/embedders/tests/misc_tests.rs` already demonstrates that tweaking the footer bytes is trivial and causes `decode_wasm` to fail. [7](#0-6) 

---

### Recommendation

In the error branch of `Hypervisor::create_execution_state`, cap the fallback `compilation_cost` to the actual compressed module length (which is bounded by the ingress message size limit) rather than the untrusted gzip footer value:

```rust
// Use the compressed size as the fallback, not the untrusted decoded size.
let safe_fallback_size = canister_module.len();
let compilation_cost = self.cost_to_compile_wasm_instruction * safe_fallback_size as u64;
```

Alternatively, apply the same `wasm_max_size` cap to the value returned by `decoded_wasm_size` before using it to compute `compilation_cost`, mirroring the check already present in `decode_wasm`.

---

### Proof of Concept

1. Take any minimal valid Wasm (e.g., `\x00asm\x01\x00\x00\x00`, 8 bytes).
2. Gzip-compress it with any standard tool.
3. Overwrite the last 4 bytes of the resulting `.gz` file with `\xFF\xFF\xFF\xFF` (little-endian `u32::MAX` ≈ 4 GiB).
4. Submit the modified blob as the `wasm_module` field of an `install_code` ingress message to any canister on the subnet.
5. `decoded_wasm_size` returns `Ok(4_294_967_295)`.
6. `compilation_cost = cost_to_compile_wasm_instruction × 4_294_967_295`.
7. `wasm_executor.create_execution_state` calls `decode_wasm`, which returns `ModuleTooLarge` (4 GiB > `wasm_max_size` = 100 MiB).
8. The error branch reduces `round_limits.instructions` by `compilation_cost` — consuming the entire round instruction budget.
9. All other canisters scheduled in the same round are starved of instructions.
