Audit Report

## Title
Pending Voting Rewards Silently Dropped for Neurons Disbursed Between Async Calculation and Distribution Phases - (`rs/nns/governance/src/reward/distribution.rs`)

## Summary
The NNS governance canister's two-phase async reward system records `distributed_e8s_equivalent` in the `RewardEvent` during Phase 1 (calculation), but actually credits maturity in Phase 2 (a periodic timer task). If a neuron is disbursed and removed from the neuron store between these two phases, `continue_processing` calls `pop_first()` to remove the reward entry before attempting to credit the neuron, and on failure silently drops the reward with no rollback, retry, or rollover. The governance accounting permanently overstates distributed maturity while the neuron controller receives nothing.

## Finding Description
**Phase 1 — `calculate_voting_rewards` (`governance.rs` L6722–6751):**
For each voting neuron that still exists, the code adds it to `reward_distribution` and increments `actually_distributed_e8s_equivalent`. The `RewardEvent` is immediately constructed with `distributed_e8s_equivalent: actually_distributed_e8s_equivalent` and the distribution is persisted to stable memory via `schedule_pending_rewards_distribution`. At this point the accounting is committed.

**Phase 2 — `continue_processing` (`distribution.rs` L159–183):**
```rust
while let Some((id, reward_e8s)) = self.rewards.pop_first() {   // entry removed first
    match neuron_store.with_neuron_mut(&id, |neuron| { ... }) {
        Ok(_) => {}
        Err(e) => {
            println!("...This should not be possible as neuron existence is checked...");
            // reward_e8s is gone; no re-insertion, no rollover
        }
    };
```
`pop_first()` unconditionally removes the `(neuron_id, reward_e8s)` pair before the neuron lookup. If the neuron has been removed from the store (e.g., via `disburse`), the `Err` branch logs and continues — the maturity is permanently destroyed.

**The `with_next_distribution` wrapper (`distribution.rs` L76–84)** only re-inserts the distribution back into stable memory if `!distribution.is_completely_finished()` (i.e., `self.rewards` is non-empty). Once `pop_first()` consumes the entry and the neuron is missing, that entry is gone from `self.rewards`, so the re-insertion guard does not help.

**Root cause:** The assumption encoded in the developer comment — "neuron existence is checked when rewards are calculated" — is violated because `disburse` can execute in a separate ingress message during the 2-second timer interval between Phase 1 and Phase 2.

## Impact Explanation
The `RewardEvent.distributed_e8s_equivalent` permanently overstates actual maturity delivered, creating an irrecoverable ledger conservation violation in the NNS governance canister. The neuron controller loses voting rewards they legitimately earned; the maturity is not redirected but destroyed. This constitutes a permanent, concrete loss of NNS governance assets with a demonstrable accounting inconsistency in an in-scope production canister. This matches the Medium impact class: meaningful security impact on NNS with concrete user harm, under strict timing conditions.

## Likelihood Explanation
The race window is the 2-second `DistributeRewardsTask` interval. Reward rounds occur daily. A neuron controller who calls `disburse` (a normal, legitimate operation) in this narrow window after a reward round closes will silently lose their pending rewards with no warning. The controller cannot observe that a distribution is in progress. This is not a targeted third-party attack — it is triggered by the neuron's own controller performing a routine operation. With many neurons and daily rounds, the probability of at least one occurrence over time is non-negligible. The condition is strict (2-second window) but requires no special privileges or technical skill.

## Recommendation
In `continue_processing`, when `neuron_store.with_neuron_mut` returns `Err`, re-insert the `(id, reward_e8s)` pair back into `self.rewards` before continuing, so the entry survives and is re-persisted by `with_next_distribution`. Alternatively, accumulate failed reward amounts into an `e8s_equivalent_to_be_rolled_over` field and include them in the next reward event's available pool. A complementary guard in the `disburse` path should check whether a `RewardsDistributionInProgress` entry exists for the neuron ID and either block disbursement or flush the pending reward first.

## Proof of Concept
1. Advance the governance canister to a reward round boundary so `distribute_voting_rewards_to_neurons` runs. Neuron N (large stake, voted) is included in `RewardsDistribution` with reward R e8s; `distributed_e8s_equivalent` is incremented by R; the distribution is written to stable memory.
2. Before the 2-second `DistributeRewardsTask` timer fires, submit a `disburse` ingress for neuron N. The neuron is removed from the neuron store.
3. Allow the timer to fire. `distribute_pending_rewards` → `continue_processing`: `pop_first()` removes neuron N's entry; `with_neuron_mut` returns `Err`; the error is logged; the loop continues.
4. Assert: neuron N's maturity was never incremented; `latest_reward_event.distributed_e8s_equivalent` includes R e8s; the `RewardsDistributionInProgress` map no longer contains neuron N's entry. The R e8s are permanently lost.

This can be reproduced as a deterministic PocketIC integration test by: (a) creating a governance canister with neuron N, (b) triggering a reward round, (c) disbursing neuron N before advancing the timer, (d) advancing the timer, and (e) asserting the maturity discrepancy.
