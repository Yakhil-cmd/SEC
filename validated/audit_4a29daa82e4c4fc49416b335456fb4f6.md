Audit Report

## Title
Hardcoded `E8` Decimal Assumption in SNS Treasury Valuation Bypasses Governance Transfer Limits - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

## Summary
The SNS treasury valuation system divides the raw `icrc1_balance_of` result by the hardcoded constant `E8` (10^8) without ever querying `icrc1_decimals` from the ledger. Because the ICRC-1 standard permits any decimal count and the SNS init code enforces no 8-decimal constraint, an SNS token with fewer than 8 decimals causes systematic treasury undervaluation, collapsing it into the "small" regime where `NoLimit` applies and the entire treasury can be drained in a single 7-day window.

## Finding Description
In `try_get_balance_valuation_factors`, the raw `Nat` returned by `icrc1_balance_of` is divided by the hardcoded `E8`:

```rust

``` [1](#0-0)

The resulting `tokens` value feeds into `ValuationFactors::to_xdr` (`tokens * icps_per_token * xdrs_per_icp`): [3](#0-2) 

That XDR total is consumed by `ProposalsAmountTotalUpperBound::from_valuation_xdr` to select the regime: [4](#0-3)

No enforcement of 8 decimals exists anywhere in `rs/sns/init/src/lib.rs` (confirmed: zero matches for "decimals" in that file). The SNS init code does not validate or constrain the decimal count of the SNS token ledger.

**Concrete arithmetic for a 6-decimal SNS token:**
- Treasury holds 1,000,000 tokens → raw balance = `1_000_000_000_000`
- Code computes: `1_000_000_000_000 / 100_000_000 = 10_000` tokens (100× too low)
- At 10 XDR/token: computed valuation = 100,000 XDR → `NoLimit` (should be 10,000,000 XDR → `Xdr(300_000)` cap)

The `fetch_icps_per_sns_token` function compounds the issue: it names variables `*_e8s` but treats them as raw `Nat` values from `icrc1_total_supply`, implying the same 8-decimal assumption throughout the price calculation path. [6](#0-5) 

## Impact Explanation
This is a **High** severity finding. An SNS launched with a non-8-decimal token (e.g., 6 decimals to match a wrapped stablecoin) will have its treasury systematically undervalued. The `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` and `mint_sns_tokens_7_day_total_upper_bound_tokens` functions both return `NoLimit` (the full balance) instead of the intended 25%/300,000 XDR cap. This enables complete treasury drainage or unbounded token minting via governance proposals within a single 7-day window, constituting unauthorized access to SNS governance assets and significant SNS security impact with concrete user harm — matching the "High ($2,000–$10,000): Significant SNS security impact with concrete user or protocol harm" bounty category. [7](#0-6) 

## Likelihood Explanation
The ICRC-1 standard explicitly supports configurable decimals via `icrc1_decimals`. The SNS framework creates ICRC-1 ledgers with no enforced decimal constraint. Any SNS launched with a token configured to fewer than 8 decimals silently triggers this miscalculation. No special privileges are required beyond the ability to submit a governance proposal on the affected SNS. The exploit is deterministic and repeatable on any such SNS.

## Recommendation
1. Extend `Icrc1Client` to include `async fn icrc1_decimals(&mut self) -> Result<u8, (i32, String)>` and query it alongside `icrc1_balance_of` in `try_get_balance_valuation_factors`.
2. Replace `/ Decimal::from(E8)` with `/ Decimal::from(10_u128.pow(decimals as u32))` using the actual on-chain decimal value.
3. Add a validation step in SNS init or governance that rejects SNS tokens whose ledger reports a decimal count outside a safe range (e.g., reject if `decimals > 18` to prevent overflow).
4. Rename `current_supply_e8s` / `initial_supply_e8s` variables in `fetch_icps_per_sns_token` to reflect that they are raw token units, not necessarily e8s, and verify the price calculation is correct for non-8-decimal tokens.

## Proof of Concept
1. Launch an SNS with a token ledger configured to 6 decimals (valid per ICRC-1 standard; no SNS init validation prevents this).
2. Fund the SNS governance treasury with 1,000,000 tokens (raw `icrc1_balance_of` = `1_000_000_000_000`).
3. Submit a `TransferSnsTreasuryFunds` proposal to transfer the entire treasury balance.
4. During proposal validation, `try_get_sns_token_balance_valuation` is called:
   - `icrc1_balance_of` returns `Nat(1_000_000_000_000)`
   - Code computes `1_000_000_000_000 / 100_000_000 = 10_000` tokens
   - At a market price of 10 XDR/token, valuation = 100,000 XDR → `from_valuation_xdr` returns `NoLimit`
   - `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` returns the full balance
5. The proposal passes the limit check and the full treasury (actual value ≈ 10,000,000 XDR) is transferred, bypassing the intended 300,000 XDR cap.

A deterministic integration test can reproduce this by mocking `Icrc1Client` to return a balance consistent with a 6-decimal token and asserting that `ProposalsAmountTotalUpperBound::in_tokens` incorrectly returns `NoLimit` instead of `Xdr(300_000)`. [8](#0-7)

async fn try_get_balance_valuation_factors(
    account: Account,
    icrc1_client: &mut dyn Icrc1Client,
    icps_per_token_client: &mut dyn IcpsPerTokenClient,
    xdrs_per_icp_client: &mut dyn XdrsPerIcpClient,
) -> Result<ValuationFactors, ValuationError> {
    // Fetch the three ingredients:
    //
    //     1. balance
    //     2. token -> ICP
    //     3. ICP -> XDR
    //
    // No await here. Instead, we use join (right after this).
    let balance_of_request = icrc1_client.icrc1_balance_of(account);
    let icps_per_token_request = icps_per_token_client.get();
    let xdrs_per_icp_request = xdrs_per_icp_client.get();

    // Make all (3) requests (concurrently).
    let (balance_of_response, icps_per_token_response, xdrs_per_icp_response) = join!(
        balance_of_request,
        icps_per_token_request,
        xdrs_per_icp_request,
    );

    // Unwrap/forward errors to the caller.
    let balance_of_response = balance_of_response.map_err(|err| {
        ValuationError::new_external(format!("Unable to obtain balance from ledger: {err:?}"))
    })?;
    let icps_per_token_response = icps_per_token_response.map_err(|err| {
        ValuationError::new_external(format!("Unable to determine ICPs per token: {err:?}"))
    })?;
    let xdrs_per_icp_response = xdrs_per_icp_response.map_err(|err| {
        ValuationError::new_external(format!("Unable to obtain XDR per ICP: {err:?}"))
    })?;

    // Extract and interpret the data we actually care about from the (Ok) responses.
    let tokens = Decimal::from(u128::try_from(balance_of_response.0).map_err(|err| {
        ValuationError::new_arithmetic(format!(
            "Balance of {account:?} does not fit in u128: {err:?}"
        ))
    })?) / Decimal::from(E8);
    let icps_per_token = icps_per_token_response;
    let xdrs_per_icp = xdrs_per_icp_response;

    // Compose the fetched/interpretted data (i.e. multiply them) to construct the final result.
    Ok(ValuationFactors {
        tokens,
        icps_per_token,
        xdrs_per_icp,
    })
}

    fn from_valuation_xdr(valuation_xdr: Decimal) -> Self {
        // Ideally, this would be checked at compile time. In principal, this should be possible,
        // since all the inputs are const, but I'm not sure how to do that. Therefore,
        // debug_assert_eq is used instead, and should be very nearly as good, because this will be
        // run during CI.
        debug_assert_eq!(
            Self::MAX_MEDIUM_TREASURY_SIZE_XDR.checked_mul(ONE_QUARTER),
            Some(Self::MAX_XDR),
        );

        if valuation_xdr <= Self::MAX_SMALL_TREASURY_SIZE_XDR {
            return Self::NoLimit;
        }

        if valuation_xdr <= Self::MAX_MEDIUM_TREASURY_SIZE_XDR {
            return Self::Fraction(ONE_QUARTER);
        }

        Self::Xdr(Self::MAX_XDR)
    }
