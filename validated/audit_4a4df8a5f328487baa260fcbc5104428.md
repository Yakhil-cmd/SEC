### Title
Missing L1-to-L2 Address Aliasing for Contract Senders Allows L1 Contracts to Impersonate L2 Contracts — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

ZKsync OS's L1→L2 transaction processing reads the `from` field directly from the ABI-encoded transaction and uses it verbatim as both `tx.origin` and `msg.sender` on L2, with no address-aliasing transformation applied. In ZKsync Era's Solidity bootloader, contract senders were aliased by adding `0x1111000000000000000000000000000000001111` before execution to prevent an L1 contract at address X from impersonating an L2 contract at the same address X. ZKsync OS omits this step entirely, creating the same cross-chain identity-mismatch class of bug described in the external report.

---

### Finding Description

In `process_l1_transaction.rs`, the `from` address is extracted from the transaction struct and forwarded without modification:

```rust
let from = transaction.from.read();   // line 184 — raw L1 sender, no aliasing
```

This raw address is then used in three security-sensitive ways:

1. **As `tx.origin`** — set in `TxLevelMetadata` passed to the EVM context: [1](#0-0) 

2. **As the recipient of the minted deposit** — base tokens are minted directly to `from`: [2](#0-1) 

3. **As the EVM `msg.sender` / caller** — passed as the `caller` argument to `run_single_interaction`: [3](#0-2) 

The `AbiEncodedTransaction` struct stores `from` as a plain `B160` with no aliasing flag or transformation: [4](#0-3) 

A codebase-wide search for `0x1111000000000000000000000000000000001111`, `L1_TO_L2_ALIAS`, `applyL1ToL2Alias`, and `l1_to_l2_alias` returns **zero matches**, confirming the aliasing step is entirely absent.

---

### Impact Explanation

An attacker who controls a contract on L1 at address `X` can submit an L1→L2 priority transaction. ZKsync OS will execute it on L2 with `msg.sender == X` and `tx.origin == X`. If any L2 contract uses `msg.sender` for access control and the privileged address happens to be `X` (e.g., an `onlyOwner` guard, a multisig, or an AA wallet deployed via CREATE2 to the same address), the L1 contract can bypass that guard and execute privileged operations — draining funds, changing ownership, or corrupting state — without ever controlling the L2 contract.

This is the direct analog of the SecuritizeBridge bug: the protocol assumes the same address on two different chains is controlled by the same entity, which is false for smart-contract wallets and CREATE2-deployed contracts.

---

### Likelihood Explanation

- **CREATE2 address collisions are deterministic**: an attacker who knows the L2 contract's salt and deployer can pre-deploy a contract on L1 at the identical address.
- **AA wallets and multisigs** (Gnosis Safe, etc.) are routinely deployed to the same address across chains via CREATE2 factories; the L1 instance is not necessarily controlled by the same key set as the L2 instance.
- Priority (L1→L2) transactions are permissionless: any L1 contract can submit one.
- The attack requires no privileged role, no leaked key, and no oracle manipulation.

Likelihood: **Medium** (requires address coincidence, but this is engineerable with CREATE2).

---

### Recommendation

Before using `from` as the L2 execution identity, check whether the L1 sender is a contract (non-zero code size on L1, conveyed via a reserved field or a separate flag). If it is, apply the standard alias offset:

```rust
const ALIAS_OFFSET: U256 = U256::from_be_bytes(/* 0x1111...1111 */);
let effective_from = if sender_is_contract {
    B160::from(U256::from(from).wrapping_add(ALIAS_OFFSET))
} else {
    from
};
```

This matches the behaviour of ZKsync Era's `AddressAliasHelper.applyL1ToL2Alias` and closes the cross-chain identity-mismatch window.

---

### Proof of Concept

1. Attacker uses CREATE2 to deploy a no-op contract on L1 at address `0xDEAD…`.
2. A legitimate L2 DeFi vault has `owner = 0xDEAD…` (same address, different controller — e.g., a Gnosis Safe deployed there by the protocol team).
3. Attacker submits a priority L1→L2 transaction from `0xDEAD…` calling `vault.withdrawAll(attacker_l2_address)`.
4. `process_l1_transaction` reads `from = 0xDEAD…` at line 184, sets `tx_origin = 0xDEAD…` and `msg.sender = 0xDEAD…` with no aliasing.
5. The vault's `onlyOwner` check passes (`msg.sender == owner`).
6. Attacker drains the vault on L2 without ever controlling the L2 owner address. [5](#0-4) [1](#0-0)
