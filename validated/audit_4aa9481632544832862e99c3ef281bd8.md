### Title
Zero Interop Root Accepted by System Hook Without Validation — (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

### Summary

The `interop_root_reporter_event_hook` system hook adds an `InteropRoot` to the system's interop root storage whenever any contract emits the `InteropRootAdded` event signature, without validating that the `root` field is non-zero. The `InteropRoot` struct itself documents that "The merkle root hash (cannot be zero for valid roots)" and "Source chain identifier (must be non-zero)", but neither the hook nor `push_root` enforces these invariants. An unprivileged user can deploy a contract that emits the event with a zero root, causing the hook to store an invalid interop root and corrupt the `interop_roots_rolling_hash` committed to in the batch public input.

### Finding Description

**Root cause — missing null check in the event hook:**

In `system_hooks/src/event_hooks/interop_root_reporter.rs`, the hook parses the event data and unconditionally calls `add_interop_root`:

```rust
let root = Bytes32::from_array(data[64..96].try_into().unwrap());
let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
system.io.add_interop_root(
    ExecutionEnvironmentType::NoEE,
    resources,
    InteropRoot { root, block_or_batch_number, chain_id },
)?;
```

There is no check that `root.is_zero()` or `chain_id.is_zero()`. The hook fires on the event signature alone — the function signature takes `_caller_ee: u8` (execution environment type) but **no caller address parameter**, meaning it fires for any contract emitting `INTEROP_ROOT_ADDED_EVENT_SIG`.

**The storage layer also has no guard:**

`InteropRootStorage::push_root` in `zk_ee/src/common_structs/interop_root_storage.rs` simply pushes the root unconditionally:

```rust
pub fn push_root(&mut self, interop_root: InteropRoot) -> Result<(), SystemError> {
    self.list.push(interop_root, ());
    Ok(())
}
```

The struct comment says "cannot be zero for valid roots" but this is never enforced in code.

**Contrast with the L2 contract path:**

The test `run_fails_if_interop_root_is_incorrect` (in `tests/instances/interop/src/lib.rs`) shows that when the *official* `L2_INTEROP_ROOT_STORAGE_ADDRESS` contract is called with a zero root, the transaction reverts — because the Solidity contract validates the root before emitting the event. However, an attacker who deploys their own contract and directly emits `InteropRootAdded` with a zero root bypasses the L2 contract's validation entirely. The system hook has no awareness of which contract emitted the event.

**Downstream impact — corrupted batch commitment:**

The stored interop roots feed directly into `calculate_interop_roots_rolling_hash`, which is committed to in the batch public input in both `post_tx_op_proving_singleblock_batch.rs` and `post_tx_op_proving_multiblock_batch.rs`:

```rust
let interop_roots_rolling_hash = calculate_interop_roots_rolling_hash(
    Bytes32::zero(),
    io.interop_root_storage.iter(),
    &mut crypto::sha3::Keccak256::new(),
);
```

A zero root injected by an attacker changes this hash, producing a batch commitment that does not match what the L1 settlement contract expects, causing batch verification to fail.

### Impact Explanation

An unprivileged user can corrupt the `interop_roots_rolling_hash` field of the batch public input by emitting the `InteropRootAdded` event from an arbitrary contract with a zero (or attacker-chosen) root value. This causes the ZK proof's public input to diverge from the L1 contract's expected value, preventing the batch from being accepted on L1. The result is a **chain-halting denial of service**: no further batches can be finalized until the corrupted state is addressed. This matches the Immunefi critical/high scope for state-transition correctness and chain liveness.

### Likelihood Explanation

The attack requires only the ability to deploy a contract and send a transaction — standard unprivileged capabilities on any EVM-compatible chain. The event signature `INTEROP_ROOT_ADDED_EVENT_SIG` is a fixed public constant. No privileged access, leaked keys, or oracle manipulation is required. The hook's lack of caller-address filtering makes this trivially reachable.

### Recommendation

Add an explicit zero-value guard in `interop_root_reporter_event_hook` before calling `add_interop_root`, mirroring the pattern used by the L2 contract:

```rust
if root.is_zero() {
    return Err(internal_error!("Interop root cannot be zero").into());
}
if chain_id.is_zero() {
    return Err(internal_error!("Interop root chain_id cannot be zero").into());
}
```

Additionally, enforce the invariant in `InteropRootStorage::push_root` so that the storage layer itself rejects invalid roots regardless of call site. Optionally, restrict the hook to fire only when the emitting contract is `L2_INTEROP_ROOT_STORAGE_ADDRESS`, providing defense-in-depth.

### Proof of Concept

1. Attacker deploys a contract with the following logic:
   ```solidity
   event InteropRootAdded(uint256 indexed chainId, uint256 indexed blockOrBatchNumber, bytes32[] roots);
   function attack() external {
       bytes32[] memory roots = new bytes32[](1);
       roots[0] = bytes32(0); // zero root
       emit InteropRootAdded(1, 42, roots);
   }
   ```
2. Attacker calls `attack()` in a normal L2 transaction.
3. The `interop_root_reporter_event_hook` fires (matching `INTEROP_ROOT_ADDED_EVENT_SIG`), parses the zero root from `data[64..96]`, and calls `system.io.add_interop_root(...)` with `root = Bytes32::ZERO`.
4. `push_root` stores the zero `InteropRoot` in the list without error.
5. At block finalization, `calculate_interop_roots_rolling_hash` includes the zero root, producing a `interop_roots_rolling_hash` that differs from what the L1 contract expects.
6. The batch proof is submitted to L1 but fails verification because the public input hash does not match, halting the chain. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
