### Title
Unchecked Return Value of Exit Precompile `call` in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without NEP-141 Release - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens before invoking the Aurora exit precompile via a low-level `call`. The return value of that `call` (stored in `res`) is never inspected. If the precompile call fails for any reason, the function returns silently: the ERC-20 tokens are permanently destroyed while no corresponding NEP-141 transfer promise is ever scheduled, freezing the underlying assets in the Aurora contract on NEAR with no recovery path.

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow the same pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // `res` is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}
```

The identical pattern appears in `withdrawToEthereum` (calling `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) and is replicated verbatim in `EvmErc20V2.sol`.

The EVM `call` opcode returns `0` on failure and `1` on success. The variable `res` captures this value but is never read. The function always returns normally regardless of whether the precompile succeeded.

The exit precompile (`ExitToNear` / `ExitToEthereum`) can fail and return `0` in several reachable conditions:
- The precompile is paused via `pause_precompiles` (the paused-precompile path returns `ERR_PAUSED` as a fatal exit, causing the inner `call` to return 0).
- The caller provides a gas limit tight enough that the `_burn` succeeds but the 63/64 gas forwarded to the precompile is insufficient for it to complete (the EVM's 63/64 rule means the outer frame retains 1/64 and continues executing after the inner call fails).
- Any other internal precompile error (e.g., invalid NEP-141 mapping, storage read failure).

In all these cases: `_burn` has already executed and is not rolled back (the outer transaction does not revert), while no promise log is emitted, so no NEP-141 transfer is ever scheduled on NEAR.

### Impact Explanation

**Permanent freezing of funds.** The user's ERC-20 mirror tokens are destroyed. The corresponding NEP-141 tokens remain locked in the Aurora engine contract's NEAR account with no mechanism to recover them, because the ERC-20 supply has already been reduced. The NEP-141 balance is permanently stranded.

### Likelihood Explanation

Any token holder who calls `withdrawToNear` or `withdrawToEthereum` is exposed. The most realistic trigger is a transaction submitted with a gas limit that is sufficient for the `_burn` but insufficient for the precompile's internal execution (the 63/64 rule guarantees the outer frame survives). A user who underestimates gas, or a contract that wraps the call with a fixed gas stipend, can silently lose funds. The paused-precompile scenario is also reachable by any user during a pause window.

### Recommendation

Check the return value of the low-level `call` and revert if it is zero. Replace the unchecked assembly block with:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures that if the precompile call fails, the entire transaction reverts, rolling back the `_burn` and preserving the user's token balance.

### Proof of Concept

1. Deploy `EvmErc20` with a bridged NEP-141 token.
2. Mint ERC-20 tokens to address `A`.
3. Submit a transaction from `A` calling `withdrawToNear(recipient, amount)` with a gas limit calibrated so that `_burn` succeeds but the forwarded 63/64 gas to the precompile is exhausted before the precompile completes.
4. The transaction succeeds (no revert). `A`'s ERC-20 balance is reduced by `amount`. No NEP-141 transfer promise is scheduled. The NEP-141 tokens remain in the Aurora contract's NEAR balance permanently.
