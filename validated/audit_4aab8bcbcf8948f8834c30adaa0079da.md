### Title
EVM Storage Refunds Silently Nullified by `delta_gas` Adjustment in Double Resource Accounting — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the EVM gas refund (earned by clearing storage slots via SSTORE) is subtracted from `gas_used` **before** the `delta_gas` native-resource adjustment is applied. When native resource consumption dominates, `delta_gas > 0` and the adjustment adds back exactly the amount the EVM refund removed, silently nullifying it. Users who clear storage slots in native-resource-heavy transactions receive no benefit from their EVM refunds and are overcharged relative to both EVM expectations and the documented double-resource-accounting model.

---

### Finding Description

`compute_gas_refund` in `refund_calculation.rs` implements the double resource accounting reconciliation. The execution order is:

**Step 1** — Compute EVM gas consumed from remaining ergs:
```
gas_used = gas_limit - remaining_ergs / ERGS_PER_GAS
```

**Step 2** — Apply EVM refund (EIP-3529, capped at 1/5 of `gas_used`):
```
evm_refund = min(refund_counter / ERGS_PER_GAS, gas_used / 5)
gas_used  -= evm_refund
```

**Step 3** — Apply `minimal_gas_used` floor.

**Step 4** — Compute native resource consumption:
```
native_used = gas_limit * native_per_gas - remaining_native
```

**Step 5** — Compute and apply `delta_gas`:
```
delta_gas = (native_used / native_per_gas) - gas_used
if delta_gas > 0:
    gas_used += delta_gas          // ← cancels out the evm_refund
``` [1](#0-0) 

**The asymmetry (algebraic proof):**

Let `G` = EVM gas consumed, `R` = EVM refund, `N` = `native_used / native_per_gas`.

- After Step 2: `gas_used = G − R`
- After Step 5 (when `N > G − R`): `gas_used = (G − R) + (N − (G − R)) = N`

The final `gas_used = N` is **independent of R**. The EVM refund is completely absorbed by `delta_gas`. The developer comment `// TODO: return delta_gas to gas_used?` at line 80 confirms awareness of this unresolved asymmetry. [2](#0-1) 

The EVM refunds themselves are generated by `refund_for_storage_write` in the storage system, which adds up to 4 800 gas per SSTORE-to-zero operation into the `refund_counter`: [3](#0-2) 

These refunds are read back in `compute_gas_refund` via `system.io.get_refund_counter().ergs()` and applied at line 48, only to be cancelled at line 78 when native resources dominate. [4](#0-3) 

The `before_refund` hook in the ZK transaction flow calls `compute_gas_refund` with the live `native_per_gas` derived from `gas_price / native_price`: [5](#0-4) 

---

### Impact Explanation

A user who clears N storage slots (earning up to `N × 4 800` gas in EVM refunds, capped at 1/5 of `gas_used`) in a transaction where native resource consumption dominates pays **exactly the same** as a user who performs identical computation without clearing any storage. The EVM refund is silently absorbed. The overcharge per transaction is:

```
overcharge = min(evm_refund, delta_gas_without_refund + evm_refund) × gas_price
           = evm_refund × gas_price   (when native dominates unconditionally)
```

For a transaction clearing 10 storage slots at a gas price of 1 000 gwei:
- EVM refund = min(48 000, gas_used/5) gas
- At 1 000 gwei, overcharge ≈ 48 000 × 1 000 gwei = 0.000048 ETH per transaction

This scales with gas price and number of cleared slots. At high gas prices or in batch-clearing contracts (e.g., DeFi settlement, token sweeps), the loss per transaction is material. The `evm_refund` value is stored in `context.gas_refunded` and reported to the user, creating a false impression that the refund was applied. [6](#0-5) 

---

### Likelihood Explanation

The condition `native_used / native_per_gas > gas_used − evm_refund` is met whenever:

1. The transaction performs non-trivial computation (native resources consumed by proving dominate EVM gas), **and**
2. The transaction also clears storage slots (generating EVM refunds).

This is a realistic combination for DeFi contracts that both compute (e.g., AMM price calculations) and clear storage (e.g., closing positions, settling orders). The `test_delta_gas` test in the test suite confirms that native-dominated transactions are an expected and tested scenario: [7](#0-6) 

The asymmetry is more pronounced at higher `native_per_gas` ratios (higher gas prices relative to `native_price`), directly mirroring the BPool finding where higher swap fees amplified the divergence between equivalent swap paths.

---

### Recommendation

Apply the `delta_gas` adjustment **before** computing and subtracting the EVM refund, so the EIP-3529 refund is always computed against the final `gas_used` (the maximum of EVM gas and native resource consumption):

```rust
// 1. Compute raw EVM gas used
let mut gas_used = gas_limit - resources.ergs().0.div_floor(ERGS_PER_GAS);

// 2. Apply delta_gas FIRST so gas_used reflects the true cost floor
let native_used = full_native_limit - resources.native().remaining().as_u64();
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}

// 3. NOW apply EVM refund against the correct gas_used
let evm_refund = core::cmp::min(full_refund_gas, gas_used / 5);
gas_used -= evm_refund;
```

This ensures EVM refunds reduce the final charge regardless of whether EVM gas or native resources dominate, consistent with EVM semantics and user expectations.

---

### Proof of Concept

**Setup**: Deploy a contract that (a) performs heavy computation (to drive up native resource consumption) and (b) clears a pre-set storage slot (to earn an EVM refund).

**Transaction A** (control): Same computation, no storage clearing.
**Transaction B** (test): Same computation + SSTORE slot to zero.

**Expected (EVM semantics)**: Transaction B should cost `gas_A − evm_refund` gas.

**Actual (ZKsync OS)**: Both transactions cost `native_used / native_per_gas` gas when native dominates. The `gas_refunded` field in the result of Transaction B will show a non-zero refund, but `gas_used` will be identical to Transaction A — confirming the refund was reported but not applied.

The `compute_gas_refund` function path is exercised for every L2 ZK transaction via `before_refund`: [8](#0-7) 

and for every L1 transaction via `process_l1_transaction`: [9](#0-8)

    fn before_refund<'a, Config: BasicBootloaderExecutionConfig>(
        system: &mut System<S>,
        transaction: &Transaction<<S as SystemTypes>::Allocator>,
        context: &mut Self::TransactionContext,
        _result: &ExecutionResult<'a, <S as SystemTypes>::IOTypes>,
        pubdata_info: Self::ExecutionBodyExtraData,
        _tracer: &mut impl Tracer<S>,
    ) -> Result<(), BootloaderSubsystemError> {
        use evm_interpreter::ERGS_PER_GAS;

        // Just used for computing native used
        context.resources_before_refund = context.resources.main_resources.clone();

        // Now we can actually reclaim resources withheld for pubdata
        context
            .resources
            .main_resources
            .reclaim_withheld(context.resources.withheld.take());

        system_log!(
            system,
            "Have {:?} resources available before refund, and need to cover {:?} pubdata\n",
            &context.resources.main_resources,
            &pubdata_info
        );

        let intrinsic_pubdata = calculate_l2_tx_intrinsic_pubdata(
            context.authorization_list_num,
            transaction.is_service(),
        );

        // Pubdata for validation has been charged already,
        // we charge for the rest now.
        let (total_pubdata_used, to_charge_for_pubdata) = match pubdata_info {
            Some(CachedPubdataInfo {
                pubdata_used,
                to_charge_for_pubdata,
            }) => (pubdata_used + intrinsic_pubdata, to_charge_for_pubdata),
            None => {
                let (execution_pubdata_spent, to_charge_for_pubdata) =
                    get_resources_to_charge_for_pubdata(
                        system,
                        context.native_per_pubdata,
                        Some(context.validation_pubdata),
                    )?;
                (
                    execution_pubdata_spent + intrinsic_pubdata,
                    to_charge_for_pubdata,
                )
            }
        };
        let min_gas_used = context.minimal_ergs_to_charge.0 / ERGS_PER_GAS;
        let refund_info = compute_gas_refund(
            system,
            to_charge_for_pubdata,
            transaction.gas_limit(),
            min_gas_used,
            context.native_per_gas,
            &mut context.resources.main_resources,
        )?;
        debug_assert_eq!(context.gas_used, 0);
        context.gas_used = refund_info.gas_used;
        context.gas_refunded = refund_info.evm_refund;
        context.total_pubdata = total_pubdata_used;
        context.native_used = refund_info.native_used;

        Ok(())
