Audit Report

## Title
Missing `http2_max_header_list_size` cap on SOCKS-proxy hyper client allows unbounded HTTP/2 header buffering — (`rs/https_outcalls/adapter/src/rpc_server.rs`)

## Summary

The direct-connection `Client` in `CanisterHttp::new` is built with a 52 KiB HTTP/2 header list cap via `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)`, but the SOCKS-proxy `Client` constructed in `create_socks_proxy_client` omits this cap entirely. A malicious IPv4-only HTTPS server — reachable only via the SOCKS fallback path — can respond with arbitrarily large HTTP/2 HEADERS frames that the h2 layer will buffer without limit, causing unbounded memory growth and potential OOM crash of the adapter process.

## Finding Description

`MAX_HEADER_LIST_SIZE` is defined as 52 KiB: [1](#0-0) 

The direct client enforces this cap: [2](#0-1) 

`create_socks_proxy_client` builds its `Client` without the cap: [3](#0-2) 

The h2 crate's client-side default for `SETTINGS_MAX_HEADER_LIST_SIZE` is effectively unlimited when not set. The setting is enforced on the receiving side: the h2 layer will buffer all header bytes the server sends before returning an error, and without the cap it will allocate whatever the server provides.

The SOCKS path is triggered whenever the direct connection fails: [4](#0-3) 

`do_https_outcall_socks_proxy` calls `get_socks_client` → `create_socks_proxy_client`, producing the uncapped client: [5](#0-4) 

The existing application-level header size checks (`headers_size_bytes`, `max_response_size_bytes`) run only after `http_resp` is returned from the client — i.e., after hyper has already fully buffered the HTTP/2 HEADERS frame(s). They do not prevent the unbounded allocation at the h2 layer.

## Impact Explanation

A malicious server can cause the HTTPS outcalls adapter process to exhaust memory and crash, disabling HTTPS outcalls on the affected replica until the process restarts. This constitutes an application/platform-level DoS on a core IC protocol component (HTTPS outcalls), achievable without raw volumetric DDoS. This maps to the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

The attacker requires no privileged access. They need only:
1. Operate any IPv4-only HTTPS server (trivially achievable).
2. Have a canister issue an HTTPS outcall to that server — either their own canister or by inducing another canister to call an attacker-controlled URL.
3. Respond with an HTTP/2 HEADERS frame (or HEADERS + CONTINUATION chain) carrying hundreds of KiB or more of header data.

The SOCKS fallback is a documented, production code path. The direct IPv6 connection to an IPv4-only server fails fast, reliably triggering the SOCKS path. The attack is repeatable: after the adapter restarts, the same request pattern re-triggers it.

## Recommendation

Add `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)` to the `Client::builder` call inside `create_socks_proxy_client`, mirroring the direct client:

```rust
// rs/https_outcalls/adapter/src/rpc_server.rs ~line 119
Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)  // add this
    .build::<_, Full<Bytes>>(
        builder
            .enable_all_versions()
            .wrap_connector(SocksConnector {
                proxy_addr,
                auth: None,
                connector: http_connector,
            }),
    )
```

## Proof of Concept

1. Spin up a mock TLS server listening on `127.0.0.1` only (IPv4) that, upon receiving any HTTP/2 request, sends a HEADERS response whose combined header bytes exceed 200 KiB (e.g., 200 headers each with a 1 KiB value). The server's `max_header_list_size` must be set large enough to send them — as done in the existing `httpbin-rs` test server at `rs/tests/httpbin-rs/src/main.rs` line 42 (`MAX_HEADER_LIST_SIZE: u32 = 1024 * 1024`). [6](#0-5) 
2. Configure a forwarding SOCKS5 proxy pointing at the mock server (as done in existing `server_test.rs` integration tests).
3. Issue a canister outcall to the mock server's address with the SOCKS proxy address set, forcing `do_https_outcall_socks_proxy`.
4. **Expected (direct path with cap):** adapter returns an error after ~52 KiB of headers are received.
5. **Actual (SOCKS path without cap):** adapter buffers all 200 KiB before returning; peak RSS of the adapter process is measurably higher, and with a sufficiently large payload the process OOMs.

    fn get_socks_client(
        &self,
        socks_proxy_uri: Uri,
    ) -> Client<HttpsConnector<SocksConnector<HttpConnector>>, OutboundRequestBody> {
        let cache_guard = self.cache.upgradable_read();

        if let Some(client) = cache_guard.get(&socks_proxy_uri.to_string()) {
            client.clone()
        } else {
            let mut cache_guard = RwLockUpgradableReadGuard::upgrade(cache_guard);
            self.metrics.socks_cache_misses.inc();
            let client = self.create_socks_proxy_client(socks_proxy_uri.clone());
            cache_guard.insert(socks_proxy_uri.to_string(), client.clone());
            self.metrics.socks_cache_size.set(cache_guard.len() as i64);
            client
        }
    }
