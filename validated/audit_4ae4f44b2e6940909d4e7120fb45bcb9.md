Audit Report

## Title
Stable Memory Dirty Pages Excluded from DTS Yield Threshold — Round Monopolization via Stable Memory Writes - (File: rs/embedders/src/wasm_executor.rs)

## Summary
The DTS yield optimization in `process()` checks only `res.wasm_dirty_pages.len()` against the 1 GiB threshold to decide whether to yield for a dedicated page-copy slice. `res.stable_memory_dirty_pages` is tracked separately and never included in this check. Because the per-message stable memory dirty page limit is 2 GiB — twice the yield threshold — a canister writing exclusively to stable memory can bypass the yield optimization entirely and force up to 2 GiB of page copying synchronously within a single execution round, delaying all other canisters scheduled in that round.

## Finding Description
In `rs/embedders/src/wasm_executor.rs` at lines 681–716, the yield guard is:

```rust
let dirty_pages = NumOsPages::from(res.wasm_dirty_pages.len() as u64);
if execution_parameters.instruction_limits.slicing_enabled()
    && dirty_pages.get() > embedder.config().max_dirty_pages_without_optimization as u64
{
    system_api.yield_for_dirty_memory_copy()...
}
``` [1](#0-0) 

`InstanceRunResult` carries two distinct dirty-page lists: [2](#0-1) 

Only `wasm_dirty_pages` feeds the threshold. After the yield check (which is never triggered for stable-only writes), the stable memory page copy proceeds unconditionally: [3](#0-2) 

The yield threshold is 1 GiB: [4](#0-3) 

The per-message stable memory dirty page limit is 2 GiB — 2× the threshold: [5](#0-4) 

The existing guard that enforces the 2 GiB limit is a hard cap on dirty pages produced, not a yield trigger. It does not spread the page-copy work across slices; it only prevents writes beyond 2 GiB. A canister writing exactly 2 GiB to stable memory will never trigger the yield optimization but will still force the replica to copy 2 GiB of dirty pages synchronously in one round.

## Impact Explanation
The yield optimization is a fairness/availability mechanism: when a canister dirties more than 1 GiB of Wasm heap pages, the replica yields and copies pages in a dedicated slice, preventing round monopolization. This protection is entirely absent for stable memory writes. An attacker can repeatably send update messages that write >1 GiB to stable memory, causing the replica to spend the entire round copying up to 2 GiB of dirty stable memory pages, delaying all other canisters scheduled in that round. This is a concrete, repeatable subnet availability impact — matching the High bounty impact class: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
Any unprivileged canister controller can deploy a canister and send an update message calling `stable_write` across >1 GiB of stable memory. No privileged access, governance majority, or threshold corruption is required. The exploit is deterministic, locally testable, and repeatable at will. The 2 GiB per-message cap bounds the maximum impact per round but does not prevent repeated exploitation across rounds.

## Recommendation
Include `stable_memory_dirty_pages` in the dirty-page count used for the yield threshold check:

```rust
let dirty_pages = NumOsPages::from(
    (res.wasm_dirty_pages.len() + res.stable_memory_dirty_pages.len()) as u64
);
```

This ensures that large stable memory writes trigger the same yield optimization as equivalent Wasm heap writes, spreading the page-copy work across a dedicated slice. [6](#0-5) 

## Proof of Concept
A differential integration test mirroring the existing `yield_for_dirty_pages_copy_works` test at: [7](#0-6) 

1. Deploy two canisters on a DTS-enabled `StateMachine` subnet:
   - Canister A: writes >1 GiB to Wasm heap (`i32.store` loop) — triggers yield → two slices.
   - Canister B: writes >1 GiB to stable memory (`stable_write`) — does **not** trigger yield → one slice.
2. Send two ingress messages to Canister B simultaneously and call `execute_payload` + `tick`.
3. Expected (broken) result: both messages complete after the first round, demonstrating that the yield was never triggered for stable memory writes — unlike the Wasm heap path where the first message remains `Processing` after round one.

fn yield_for_dirty_pages_copy_works() {
    let env = ic_state_machine_tests::StateMachineBuilder::new()
        .with_subnet_type(SubnetType::Application)
        .build();

    let wasm = wat::parse_str(WRITE_MORE_THAN_1G_WAT).unwrap();
    let canister_id = env
        .install_canister_with_cycles(wasm, vec![], None, INITIAL_CYCLES_BALANCE)
        .unwrap();

    let mut payload = ic_state_machine_tests::PayloadBuilder::new().with_nonce(0);
    // Send two ingress messages to the same canister.
    for _ in 0..2 {
        payload = payload.ingress(PrincipalId::new_anonymous(), canister_id, "write", vec![]);
    }
    let message_ids = payload.ingress_ids();
    env.execute_payload(payload);

    // Neither of messages should be completed after the first round.
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[0])),
        Some(IngressState::Processing)
    );
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[1])),
        Some(IngressState::Received)
    );

    env.tick();

    // Only the first message must be completed after two rounds.
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[0])),
        Some(IngressState::Completed(_))
    );
    assert_matches!(
        ingress_state(env.ingress_status(&message_ids[1])),
        Some(IngressState::Received)
    );
