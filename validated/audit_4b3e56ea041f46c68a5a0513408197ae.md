### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn user tokens **before** calling the `ExitToNear` / `ExitToEthereum` precompile via a low-level assembly `call`. The assembly captures the success flag in `res` but **never checks it**. When the precompile call fails (e.g., due to an invalid NEAR recipient account ID supplied by the caller), `res = 0` is silently ignored, the function returns normally, and the already-burned tokens are permanently lost with no cross-chain exit ever initiated.

---

### Finding Description

In `EvmErc20.withdrawToNear()`:

```solidity
// EvmErc20.sol L53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens burned here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — if call fails, execution continues silently
    }
}
```

The identical pattern appears in `EvmErc20V2.withdrawToNear()` and in both contracts' `withdrawToEthereum()`.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `output: Vec::new()` on success and propagates `Err(ExitError)` on failure. Failures are mapped to `PrecompileFailure::Error`, which causes the EVM to set `res = 0` for the low-level `call`. Concrete failure paths reachable with user-controlled input include:

- **`ERR_INVALID_RECEIVER_ACCOUNT_ID`** — `recipient` is `bytes memory` (fully attacker/user-controlled); if the bytes are not valid UTF-8 or not a valid NEAR account ID, `parse_recipient` returns an error.
- **`ERR_INVALID_AMOUNT`** — if `amount > u128::MAX`, `parse_amount` returns `ERR_INVALID_AMOUNT`. ERC-20 balances are `uint256`, so a token minted with an amount exceeding `u128::MAX` triggers this path.
- **`ERR_TARGET_TOKEN_NOT_FOUND`** — if the ERC-20 → NEP-141 mapping is absent in storage.

In all cases: `_burn` has already executed, `res = 0` is ignored, the function returns without reverting, and the tokens are gone.

### Impact Explanation

**Critical — Permanent freezing / destruction of funds.**

When the precompile call fails silently, the user's ERC-20 tokens are burned on Aurora with no corresponding NEP-141 release on NEAR. The tokens are irrecoverably destroyed. There is no refund path, no revert, and no event indicating failure. The same impact applies to `withdrawToEthereum()`.

---

### Likelihood Explanation

**Medium.** The `recipient` parameter is `bytes memory` — any caller can supply arbitrary bytes. Passing bytes that are not valid UTF-8 (e.g., `0xFF 0xFE ...`) is trivially achievable, either by accident (off-chain encoding bug) or by a user who misunderstands the expected format. No privileged access is required; any token holder can trigger this by calling `withdrawToNear()` directly.

---

### Recommendation

Check `res` inside the assembly block and revert on failure, **before** burning tokens (or use a checks-effects-interactions ordering where the burn is the last step):

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum()` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` with a valid NEP-141 mapping.
2. Mint 1000 tokens to `alice`.
3. `alice` calls `withdrawToNear(bytes("\xff\xfe"), 1000)` — `\xff\xfe` is invalid UTF-8.
4. `_burn(alice, 1000)` executes; alice's balance drops to 0.
5. The precompile call fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID`; `res = 0`.
6. The assembly block does not revert; `withdrawToNear` returns normally.
7. Alice's 1000 tokens are permanently destroyed. No NEP-141 tokens are released on NEAR.
