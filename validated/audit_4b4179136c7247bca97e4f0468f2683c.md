Audit Report

## Title
Unvalidated pool unbounded growth via bouncer-window artifacts never purged or validated — (`rs/https_outcalls/consensus/src/gossip.rs`, `rs/https_outcalls/consensus/src/pool_manager.rs`)

## Summary
The canister HTTP outcalls subsystem admits artifacts with `callback_id` in `[next_callback_id, next_callback_id + 345]` via the gossip bouncer, but both pool-management functions that could remove such artifacts apply a hard `id < next_callback_id` pre-filter, leaving the entire bouncer window permanently unprocessed. A single Byzantine P2P peer can exploit this structural gap to grow the unvalidated pool without bound, causing OOM and replica crash.

## Finding Description
`CanisterHttpGossipImpl::new_bouncer` in `rs/https_outcalls/consensus/src/gossip.rs` computes `highest_accepted_request_id = next_callback_id + MAX_NUMBER_OF_REQUESTS_AHEAD` (345) and returns `BouncerValue::Wants` for any artifact whose `callback_id` falls in `[next_callback_id, highest_accepted_request_id]`, subject only to a registry-version equality check — no signature verification is performed. [1](#0-0) [2](#0-1) 

Both pool-management functions that could evict such artifacts apply a hard `id < next_callback_id` pre-filter:

- `purge_shares_of_processed_requests` (line 180): only considers unvalidated artifacts with `id < next_callback_id` for removal. [3](#0-2) 

- `validate_shares` (line 484): only considers unvalidated artifacts with `id < next_callback_id` for validation or invalidation. [4](#0-3) 

Artifacts with `id >= next_callback_id` are therefore invisible to both functions and accumulate indefinitely. This behavior is even confirmed by the existing test `test_validation_of_shares_above_known_requests`, which inserts a share with `callback_id = 1` (above `next_callback_id`) and asserts that `validate_shares` returns an empty change set — the share is silently filtered out and never removed. [5](#0-4) 

Because the artifact ID is the full `CanisterHttpResponseShare` (the complete signed struct), distinct `content_hash` values produce distinct artifact IDs. A Byzantine peer can therefore produce an unlimited number of pool entries for the same `callback_id` by varying `content_hash`. [6](#0-5) 

The `CanisterHttpPoolImpl` unvalidated section has no per-peer or total-size cap. [7](#0-6) 

## Impact Explanation
A single Byzantine replica (below the fault threshold, no key compromise required) can cause unbounded heap growth in the unvalidated pool of any honest replica it is peered with, leading to OOM and replica crash. A crashed replica falls behind and must resync, degrading subnet resilience. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation
The attacker needs only one subnet membership slot and the ability to gossip crafted artifacts. The registry-version check is the only bouncer gate, and the current registry version is publicly observable. Critically, the bouncer performs no signature verification, so the Byzantine peer does not even need a valid node key to flood the pool — any artifact with the correct `callback_id` range and `registry_version` is admitted. The attack is repeatable every gossip round and requires no coordination with other nodes. [8](#0-7) 

## Recommendation
Add an explicit eviction path for unvalidated artifacts whose `callback_id` falls in the bouncer window but is not in `active_callback_ids`. Concretely, in `purge_shares_of_processed_requests`, remove the `id < next_callback_id` pre-filter on the unvalidated pool scan, or add a second pass that evicts unvalidated artifacts with `id >= next_callback_id` that are not in `active_callback_ids`. Alternatively, enforce a hard cap on the unvalidated pool size (e.g., `subnet_size * MAX_NUMBER_OF_REQUESTS_AHEAD`) and drop excess artifacts by age or peer.

## Proof of Concept
State-machine test sketch (extends the existing test harness in `rs/https_outcalls/consensus/src/pool_manager.rs`):

1. Set `next_callback_id = K`, `active_callback_ids = {K}`.
2. For each round `r` in `1..N`, insert `M` artifacts with distinct `content_hash` values and `callback_id = K + 1` (in the bouncer window) from a single Byzantine peer.
3. Call `on_state_change` each round.
4. Assert that `canister_http_pool.get_unvalidated_artifacts().count()` grows by `M` each round and is never reduced.
5. Observe pool size = `N * M` after `N` rounds, confirming unbounded growth.
