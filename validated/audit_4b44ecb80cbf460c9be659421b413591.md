### Title
All EIP-2537 BLS12-381 Precompiles Charge Zero Native Resources Despite Significant Proving Cost — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

Every one of the seven EIP-2537 BLS12-381 precompiles (G1 add, G2 add, G1 MSM, G2 MSM, pairing check, G1 field-to-curve mapping, G2 field-to-curve mapping) hard-codes `cost_native = 0`. This is the ZKsync OS analog of the external report's pattern: a single shared configuration value (`cost_native = 0`) is applied uniformly across all BLS12-381 operation types, even though each operation has a distinct and non-trivial actual proving cost. All other cryptographic precompiles in the same codebase carry carefully benchmarked non-zero native costs. The result is a resource-accounting bug: BLS12-381 operations consume real RISC-V proving cycles but are never charged against the transaction's native resource budget.

---

### Finding Description

ZKsync OS implements a **double resource accounting** model. Every operation charges two independent resources:

1. **Ergs** (EVM gas × `ERGS_PER_GAS`) — limits EE execution.
2. **Native** — models actual RISC-V proving cost; derived from `gasPrice / nativePrice` and enforced as a hard per-transaction cap (`MAX_NATIVE_COMPUTATIONAL = 2^35`).

All non-BLS12-381 cryptographic precompiles carry explicit native costs:

| Precompile | Native cost |
|---|---|
| ECRECOVER | `native_with_delegations!(350_000, 43_000, 0)` ≈ 522 k |
| BN254 ECADD | `native_with_delegations!(46_000, 1_650, 0)` ≈ 52 k |
| BN254 ECMUL | `native_with_delegations!(600_000, 41_000, 0)` ≈ 764 k |
| BN254 Pairing (base) | `native_with_delegations!(13_000_000, 500_000, 0)` ≈ 15 M |
| P256 Verify | `native_with_delegations!(500_000, 71_000, 0)` ≈ 784 k |
| KZG Point Eval | `native_with_delegations!(49_900_000, 3_301_000, 0)` ≈ 63 M | [1](#0-0) 

Every BLS12-381 precompile, by contrast, sets `cost_native = 0`:

```rust
// G1 addition
let cost_ergs = Ergs(BLS12_381_G1_ADDITION_GAS * ERGS_PER_GAS);
let cost_native = 0;   // ← shared zero for all BLS12-381 ops
resources.charge(&R::from_ergs_and_native(cost_ergs, ...cost_native...))?;
``` [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

All seven precompiles are registered unconditionally in `initialize_eip_2537`: [6](#0-5) 

The native resource budget for a transaction is:

```
nativeLimit = gasLimit × ceil(gasPrice / nativePrice)
``` [7](#0-6) 

Because BLS12-381 operations charge `cost_native = 0`, they consume **none** of this budget regardless of how many are executed. The prover must still prove every field-arithmetic and group-law step, but the transaction is never charged for it.

---

### Impact Explanation

**Resource accounting divergence / prover DoS.**

An unprivileged caller submits a transaction whose EVM gas is entirely consumed by BLS12-381 calls (e.g., repeated G2 MSM or pairing-check calls). The transaction's native budget is untouched because every BLS12-381 call charges 0 native. The prover must nonetheless execute and prove all the BLS12-381 arithmetic — work that is more expensive per gas unit than BN254 pairing (which costs ≈15 M native per base call). Across a full block, many such transactions can collectively force the prover to spend far more RISC-V cycles than the aggregate native budgets of those transactions account for. This:

- Breaks the economic invariant that `nativeLimit` upper-bounds the prover's work per transaction.
- Allows a low-cost attack (pay only EVM gas, no native premium) to inflate block proving time.
- Creates a forward/proving divergence: the sequencer's native accounting says the block is within budget; the prover's actual cycle count is not. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

**High.** Any externally-owned account can call BLS12-381 precompile addresses (0x0b–0x11 post-Pectra). No privileged role, oracle manipulation, or governance action is required. The attacker only needs to pay standard EVM gas; the native resource surcharge that would normally deter expensive proving work is entirely absent for these precompiles.

---

### Recommendation

Assign each BLS12-381 precompile a benchmarked non-zero `cost_native` value, following the same `native_with_delegations!(raw_cycles, bigint_delegations, blake_delegations)` pattern used for BN254 and other precompiles. At minimum, the pairing check and MSM operations — which are the most expensive — must be addressed first. The G1/G2 addition and field-mapping operations are cheaper but still non-trivial and should also receive proper native costs.

---

### Proof of Concept

1. Deploy or use any contract that calls the BLS12-381 pairing precompile (address `0x0f`) in a loop.
2. Submit a transaction with `gasLimit = 30_000_000` and `gasPrice` set to the minimum that passes validation.
3. Each pairing call with `k` pairs costs `37700 + 32600k` EVM gas and `0` native.
4. With 30 M gas, an attacker can execute roughly `30_000_000 / (37700 + 32600*1)` ≈ 424 single-pair pairing calls per transaction.
5. Observe that the transaction's native resource counter never decreases from BLS12-381 calls, while the prover must execute hundreds of BLS12-381 pairing computations — each more expensive than a BN254 pairing (≈15 M native) — without any native budget being consumed. [10](#0-9) [11](#0-10)
