### Title
ckERC20 Over-Minting Due to Fee-on-Transfer ERC-20 Tokens: Event Log Amount Trusted Without Actual Balance Verification - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol` / `rs/ethereum/cketh/minter/src/deposit.rs`)

---

### Summary

Both the on-chain Ethereum helper contracts (`ERC20DepositHelper.sol` and `DepositHelperWithSubaccount.sol`) and the IC-side ckETH minter (`src/deposit.rs`) share the same structural flaw as the AaveV3 report: the deposit event is emitted with the caller-supplied `amount` parameter rather than the actual balance increase of the minter address, and the IC minter mints ckERC20 tokens equal to that unverified event value. For any fee-on-transfer ERC-20 token added as a supported ckERC20 asset, the minter would mint more ckERC20 than the ERC-20 it actually holds, breaking the 1:1 collateral invariant.

---

### Finding Description

**Step 1 — Ethereum helper contract emits the wrong amount.**

In `DepositHelperWithSubaccount.sol`, `depositErc20` calls `safeTransferFrom` with the caller-supplied `amount` and then emits `ReceivedEthOrErc20` with that same `amount` field — not the actual balance increase of `minterAddress`:

```solidity
// DepositHelperWithSubaccount.sol lines 511-532
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    erc20Token.safeTransferFrom(
        msg.sender,
        minterAddress,
        amount          // <-- requested amount, not actual received amount
    );

    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        amount,         // <-- emits requested amount, not balance delta
        principal,
        subaccount
    );
}
```

The same pattern exists in the older `ERC20DepositHelper.sol` `deposit()` function (lines 498–503).

For a fee-on-transfer ERC-20 token, `minterAddress` receives `amount - fee`, but the event records `amount`.

**Step 2 — IC minter mints based on the unverified event log value.**

The IC ckETH minter's `mint()` function in `src/deposit.rs` reads events from the Ethereum log and mints ckERC20 tokens using `event.value()` — which is the `amount` field parsed directly from the emitted log:

```rust
// rs/ethereum/cketh/minter/src/deposit.rs lines 73-81
let block_index = match client
    .transfer(TransferArg {
        from_subaccount: None,
        to: event.beneficiary(),
        fee: None,
        created_at_time: None,
        memo: Some((&event).into()),
        amount: event.value(),   // <-- amount from log event, not verified balance change
    })
    .await
```

There is no step where the minter queries the actual ERC-20 balance of its Ethereum address before and after the transfer to compute the real received amount. The minter unconditionally trusts the `amount` field in the `ReceivedEthOrErc20` log event.

---

### Impact Explanation

**Vulnerability class:** Chain-fusion mint/burn/replay bug — specifically, ckERC20 over-minting leading to undercollateralization.

If any fee-on-transfer ERC-20 token is added as a supported ckERC20 asset via NNS governance:

- A user deposits `N` tokens; the ERC-20 contract deducts a fee, so the minter's Ethereum address receives `N - fee`.
- The helper contract emits `ReceivedEthOrErc20(..., amount=N, ...)`.
- The IC minter mints `N` ckERC20 tokens to the user.
- The minter holds only `N - fee` ERC-20 tokens on Ethereum.
- The ckERC20 supply exceeds the ERC-20 collateral held by the minter.
- Repeated deposits drain the collateral pool; the last withdrawers cannot redeem their ckERC20 for ERC-20.

This is a direct ledger conservation break: total ckERC20 minted > total ERC-20 held by minter.

---

### Likelihood Explanation

**Medium-Low.** The minter currently only supports tokens whitelisted via NNS governance proposals, and the currently supported tokens (USDC, USDT, etc.) are standard ERC-20 tokens without transfer fees. However:

1. The code contains no guard or assertion preventing a fee-on-transfer token from being added.
2. Future NNS proposals could add a token that has a fee-on-transfer mechanism (e.g., tokens with built-in tax, rebasing tokens, or tokens that add fees via upgrades).
3. The attack requires no privileged access beyond the normal user flow of calling `depositErc20` on the helper contract.

The entry path is fully unprivileged: any Ethereum user can call `depositErc20` on the deployed helper contract with any ERC-20 token address that has been approved by the minter.

---

### Recommendation

**In the Ethereum helper contracts**, replace the emitted `amount` with the actual balance delta of `minterAddress`:

```solidity
function depositErc20(
    address erc20Address,
    uint256 amount,
    bytes32 principal,
    bytes32 subaccount
) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;

    emit ReceivedEthOrErc20(
        erc20Address,
        msg.sender,
        actualReceived,   // <-- actual balance increase, not caller-supplied amount
        principal,
        subaccount
    );
}
```

**In the IC minter**, add a validation step or documentation constraint that explicitly rejects fee-on-transfer tokens during the NNS governance token-addition process, and consider adding an invariant check that the minter's tracked ERC-20 balance matches the on-chain balance periodically.

---

### Proof of Concept

1. An NNS proposal adds a fee-on-transfer ERC-20 token (e.g., one that deducts 1% on every transfer) as a supported ckERC20 asset.
2. A user approves the helper contract for 1,000 tokens and calls `depositErc20(tokenAddress, 1000, principal, subaccount)`.
3. The ERC-20 contract transfers 990 tokens to `minterAddress` (1% fee deducted).
4. The helper contract emits `ReceivedEthOrErc20(..., amount=1000, ...)`.
5. The IC minter scrapes the log via `scrape_logs()` → `register_deposit_events()` → `mint()`.
6. `mint()` calls `client.transfer(TransferArg { amount: event.value() })` where `event.value() = 1000`.
7. The user receives 1,000 ckERC20 tokens on the IC ledger.
8. The minter holds only 990 ERC-20 tokens on Ethereum.
9. After 100 such deposits, the minter holds 99,000 ERC-20 but has minted 100,000 ckERC20 — a 1,000-token shortfall that grows with each deposit.
