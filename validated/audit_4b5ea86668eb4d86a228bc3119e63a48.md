Audit Report

## Title
Ingress Pool Admission DoS via Per-Message vs. Cumulative Cycles Check Mismatch — (`rs/execution_environment/src/execution_environment.rs`, `rs/ingress_manager/src/ingress_selector.rs`)

## Summary
The admission-time gate checks only the cost of a single message against a canister's balance, while the proposal-time gate checks cumulative cost across all messages from the same canister in a block. Messages that pass admission but fail the cumulative check at proposal time are never evicted from the validated pool — only popped from a local in-memory queue. An attacker with a minimally-funded canister can flood the validated pool with messages that will never be included in a block, saturating the pool and causing the HTTP endpoint to return `503 SERVICE_UNAVAILABLE` to all legitimate users until TTL expiry.

## Finding Description
**Admission-time check** (`should_accept_ingress_message`, `rs/execution_environment/src/execution_environment.rs` L3340–3373): The code explicitly labels itself a "first-pass" check and passes only the single-message `cost` to `can_withdraw_cycles_with_threshold`. Multiple concurrent submissions from the same canister each see the same unmodified state, so all N submissions independently pass. [1](#0-0) 

**Pool-to-validated promotion** (`validate_ingress_pool_object`, `rs/ingress_manager/src/ingress_handler.rs` L167–206): When messages move from unvalidated to validated, only size, known-status, and signature are checked. No cycles check is performed at this stage, so all N messages enter the validated pool. [2](#0-1) 

**Proposal-time check** (`validate_ingress`, `rs/ingress_manager/src/ingress_selector.rs` L566–584): At block-building time the check is cumulative — `*cumulative_ingress_cost + ingress_cost` — so message 1 passes and messages 2–N fail with `InsufficientCycles`. [3](#0-2) 

**No pool eviction on failure** (`rs/ingress_manager/src/ingress_selector.rs` L204–207): The `_` arm of the match on `validate_ingress` result calls only `queue.msgs.pop()`, which removes the message from the local round-robin queue for that round. It does not emit a `RemoveFromValidated` change action, so the message remains in the validated pool. [4](#0-3) 

**Validated pool cleanup paths** (`rs/ingress_manager/src/ingress_handler.rs` L112–137): The only mechanisms that remove messages from the validated pool are `IngressHistoryReader` returning non-Unknown status, finalization, and TTL expiry (`PurgeBelowExpiry`). None of these apply to messages that fail cycles validation — they have never been executed, so their status remains Unknown, and they persist until `MAX_INGRESS_TTL` (5 minutes). [5](#0-4) 

**Pool capacity enforcement** (`rs/artifact_pool/src/ingress_pool.rs` L226–232): `exceeds_limit` counts both unvalidated and validated messages per peer against `ingress_pool_max_count` and `ingress_pool_max_bytes`. [6](#0-5) 

**HTTP endpoint rejection** (`rs/http_endpoints/public/src/call.rs` L229–236): When `exceeds_threshold()` returns true, all new ingress submissions are rejected with `503 SERVICE_UNAVAILABLE`. [7](#0-6) 

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The attacker can render a subnet's ingress endpoint unavailable to all legitimate users for up to 5 minutes per attack wave, with no raw volumetric component — the attack exploits a logical mismatch in the protocol's own admission and validation logic. Additionally, every round of `get_ingress_payload` wastes selector work iterating over zombie messages that will never be included.

## Likelihood Explanation
The attack requires only a canister with a minimal cycles balance (enough for one induction cost, proportional to message size — negligible cost). No privileged access, governance majority, subnet majority, leaked keys, or social engineering is required. The attacker submits messages via the standard HTTPS `/api/v2/canister/{id}/call` endpoint. The attack is repeatable every 5 minutes to maintain a sustained DoS. Using multiple canisters and boundary nodes, the attacker can saturate the pool across all subnet nodes simultaneously. The structural mismatch is present in production code and is not gated by any feature flag.

## Recommendation
1. **Evict messages that fail cycles validation**: In the `_` arm at `rs/ingress_manager/src/ingress_selector.rs` L204–207, emit a `RemoveFromValidated` change action for messages that fail with `InsufficientCycles`, so they are purged from the pool rather than re-selected every round.
2. **Tighten admission-time gate**: In `should_accept_ingress_message`, check whether the canister's available balance exceeds the freeze threshold by more than a configurable multiple of a single message cost before admitting additional messages from the same canister within a TTL window.
3. **Per-canister admission rate limiting**: Limit the number of messages from a single canister that can be admitted to the validated pool within a TTL window, proportional to the canister's available cycles divided by the per-message induction cost.

## Proof of Concept
1. Create canister `C` with cycles balance = `freeze_threshold + ingress_induction_cost(msg)` (sufficient for exactly one message).
2. Submit `N` messages targeting `C` via `/api/v2/canister/{id}/call` in rapid succession. Each invocation of `should_accept_ingress_message` reads the same unmodified state and passes `cost` (single-message) to `can_withdraw_cycles_with_threshold` — all N pass.
3. All N messages are promoted to the validated pool by `validate_ingress_pool_object` (no cycles check performed).
4. On the next `get_ingress_payload` call, `validate_ingress` checks cumulative cost: message 1 passes (`cumulative = cost ≤ balance − threshold`), message 2 fails (`cumulative = 2×cost > balance − threshold`), messages 3–N fail similarly. Each failure hits the `_ => { queue.msgs.pop(); continue; }` arm — messages remain in the validated pool.
5. Repeat with additional minimally-funded canisters until `ingress_pool.exceeds_threshold()` returns `true`.
6. All subsequent legitimate ingress submissions receive `503 SERVICE_UNAVAILABLE` until the attacker's messages expire after `MAX_INGRESS_TTL` (~5 minutes).
7. A deterministic integration test or PocketIC test can reproduce this by: (a) creating a canister with exactly one message's worth of cycles above freeze threshold, (b) submitting N > 1 ingress messages, (c) calling `get_ingress_payload` and asserting only 1 message is included, (d) asserting the validated pool still contains N messages, and (e) asserting `exceeds_threshold()` returns true after pool saturation.
