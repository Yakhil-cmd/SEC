### Title
Missing `MAX_NUMBER_OF_LOGS` Enforcement in `push_message`/`push_l1_l2_tx_log` Causes Block-Finalization Panic via `ArrayVec` Overflow — (`zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

`LogsStorage::push_message` and `push_l1_l2_tx_log` accept new L2→L1 log entries without checking the hard cap `MAX_NUMBER_OF_LOGS = 16_384`. During batch finalization, `apply_to_array_vec` writes every accumulated log into a fixed-capacity `ArrayVec<Bytes32, 16384>`. `ArrayVec::push` **panics** when capacity is exceeded. If the log count surpasses 16 384 — either because the per-transaction `BlockL2ToL1LogsLimitReached` guard is bypassed for L1 (priority) transactions, or because the guard is per-block while the Merkle tree is per-batch — the batch can never be finalized, permanently locking every L2→L1 withdrawal committed in that batch.

---

### Finding Description

`MAX_NUMBER_OF_LOGS` is declared as the Merkle-tree capacity:

```rust
// zk_ee/src/common_structs/logs_storage.rs:25
pub const MAX_NUMBER_OF_LOGS: u64 = 16_384;
```

Neither `push_message` nor `push_l1_l2_tx_log` reference this constant:

```rust
// logs_storage.rs:181-210  (push_message)
pub fn push_message(...) -> Result<(), SystemError> {
    // computes total_pubdata, then:
    self.list.push(LogContent { ... }, total_pubdata);
    Ok(())   // ← no MAX_NUMBER_OF_LOGS check
}

// logs_storage.rs:213-236  (push_l1_l2_tx_log)
pub fn push_l1_l2_tx_log(...) -> Result<(), SystemError> {
    self.list.push(LogContent { ... }, total_pubdata);
    Ok(())   // ← no MAX_NUMBER_OF_LOGS check
}
```

During batch finalization the entire log list is drained into a fixed-capacity array:

```rust
// logs_storage.rs:311-316
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.push(log.hash())   // ← panics if len > 16384
    });
}
```

The same fixed-capacity signature appears in the batch-data path:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs:178
fn l2_logs_root(mut logs: ArrayVec<Bytes32, 16384>) -> Bytes32 { ... }
```

The `BlockL2ToL1LogsLimitReached` guard exists at the transaction level:

```rust
// basic_bootloader/src/bootloader/errors.rs:103-104
/// Transaction makes the block reach the l2->l1 logs limit
BlockL2ToL1LogsLimitReached,
```

However, two gaps remain:

1. **L1 (priority) transactions cannot be invalidated.** The codebase explicitly documents this: *"invalidating an L1 transaction can halt the chain (due to the priority queue)."* If the guard is skipped for L1 transactions (to avoid a different chain-halt), a single L1 transaction that emits many `sendToL1` messages can push the block's log count past 16 384. The system-level `push_l1_l2_tx_log` call that records every L1 tx result is also unconditional, adding one more log per L1 tx regardless of the guard.

2. **Multi-block batch accumulation.** The guard is checked per-block, but `apply_to_array_vec` is called once per batch across all blocks. If each block stays just under the per-block limit, the cumulative total across blocks in a batch can still exceed 16 384, triggering the panic at batch-finalization time.

---

### Impact Explanation

When `apply_to_array_vec` panics, the batch cannot be finalized and no ZK proof can be generated. Every L2→L1 withdrawal committed in that batch (tokens already burned on L2 via the L2 base-token `withdraw` hook) becomes permanently unclaimable on L1. The chain is halted for all subsequent batches until the issue is resolved at the protocol level. This is a direct fund-loss path for all users who withdrew in the affected batch, and a complete denial-of-service for the chain.

---

### Likelihood Explanation

An attacker who controls an L1 (priority) transaction can craft a call to the L1 Messenger system hook (`0x8008 → 0x7001`) that emits thousands of `sendToL1` messages in a single transaction. Because L1 transactions cannot be rejected, and because `push_message` imposes no cap, the attacker can fill the log storage past 16 384 entries. The cost is bounded only by the L1 gas paid to submit the priority transaction, which is a one-time cost for a permanent chain halt.

---

### Recommendation

1. **Enforce the cap inside `push_message` and `push_l1_l2_tx_log`**: return a `SystemError` (or a new `LogsLimitExceeded` variant) when `self.list.len() as u64 >= MAX_NUMBER_OF_LOGS`.
2. **Apply `BlockL2ToL1LogsLimitReached` to L1 transactions**: treat the transaction body as failed (roll back its logs) while still recording the L1 tx result log, and ensure the result log itself is counted against the limit.
3. **Track the limit at the batch level, not just the block level**, so that multi-block batches cannot silently accumulate more than 16 384 logs across blocks.
4. **Replace the silent `ArrayVec::push` with a checked push** (e.g., `try_push`) and propagate the error rather than panicking.

---

### Proof of Concept

1. Attacker submits an L1 priority transaction targeting the L1 Messenger system contract (`0x8008`) with calldata that loops and calls `sendToL1` 16 385 times.
2. Because L1 transactions cannot be invalidated, the bootloader processes the transaction. Each `sendToL1` call reaches `send_to_l1_inner` → `emit_l1_message` → `push_message`, which appends unconditionally.
3. After the transaction, `logs_storage.len()` is 16 385.
4. At batch finalization, `apply_to_array_vec` iterates the list and calls `array_vec.push(...)` on the 16 385th element of an `ArrayVec<Bytes32, 16384>`, triggering a Rust panic.

pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;

    Ok(Ok(()))
