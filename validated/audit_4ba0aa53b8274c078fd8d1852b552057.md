### Title
ERC-20 Tokens Permanently Burned When `ft_transfer` Fails in `ExitToNear` Precompile Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens on Aurora before issuing a `ft_transfer` (or `ft_transfer_call`) promise to the NEP-141 contract. When the `error_refund` compile-time feature is disabled — which is the default — no callback is attached to handle promise failures. If the outbound NEP-141 transfer fails (e.g., recipient not registered, or recipient contract does not implement `ft_on_transfer`), the ERC-20 tokens are permanently destroyed with no refund path, causing a permanent freeze of user funds.

---

### Finding Description

**Vulnerability class:** Permanent fund freeze via unhandled callback failure (analog of `safeTransferFrom` requiring `ERC721TokenReceiver`).

The `ExitToNear::run()` function in `engine-precompiles/src/native.rs` constructs a `ExitToNearPrecompileCallbackArgs` struct:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is **not** enabled (the default per `engine-precompiles/Cargo.toml`), `refund` is hardcoded to `None`. [2](#0-1) [3](#0-2) 

For the **legacy ERC-20 exit** and **Omni ERC-20 exit** paths, `transfer_near_args` is also `None`:

```rust
// Omni case
Some(Message::Omni(msg)) => (
    nep141_account_id,
    ft_transfer_call_args(...),
    "ft_transfer_call",
    None,   // <-- transfer_near_args
    ...
),
// Legacy case
_ => (
    nep141_account_id,
    format!(...),
    "ft_transfer",
    None,   // <-- transfer_near_args
    ...
),
``` [4](#0-3) 

Because both fields are `None`, `callback_args == ExitToNearPrecompileCallbackArgs::default()`, and the promise is created **without any callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [5](#0-4) 

For the **wNEAR unwrap** path, a callback *is* created (because `transfer_near` is `Some`), but the callback's failure branch does nothing when `error_refund` is off:

```rust
} else if let Some(args) = args.refund {
    // refund is None without error_refund → this branch is never taken
    engine::refund_on_error(...)
} else {
    None   // silent no-op on failure
};
``` [6](#0-5) 

The test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

---

### Impact Explanation

**Impact: Permanent freezing of funds.**

When `ft_transfer` or `ft_transfer_call` fails after ERC-20 tokens have already been burned on Aurora:
- The user's ERC-20 balance is permanently destroyed.
- The NEP-141 tokens remain held by the Aurora engine contract (not transferred to the recipient).
- There is no recovery path for the user.

This matches the "permanent fund freeze" impact class.

---

### Likelihood Explanation

**Likelihood: Medium.**

Realistic failure triggers include:
1. **Recipient not registered** — NEP-141 tokens require a storage deposit (`storage_deposit`) before `ft_transfer` can succeed. Any NEAR account that has never interacted with the specific NEP-141 contract will cause `ft_transfer` to fail.
2. **Omni path / `ft_transfer_call`** — If the recipient contract does not implement `ft_on_transfer`, the NEP-141 contract reverts the transfer. This is the direct analog of the `safeTransferFrom` / `ERC721TokenReceiver` pattern in the original report.
3. **wNEAR unwrap** — If `near_withdraw` fails for any reason, the callback silently does nothing.

An unprivileged EVM user can trigger this by calling the `ExitToNear` precompile with a recipient that is not registered with the target NEP-141 contract, which is a common real-world scenario.

---

### Recommendation

Enable `error_refund` as a **default feature** in both `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, or unconditionally attach a failure-handling callback to every `ft_transfer`/`ft_transfer_call` promise issued by the `ExitToNear` precompile. The `refund_on_error` function in `engine/src/engine.rs` already implements the correct re-mint logic for ERC-20 tokens and ETH refund logic for base-token exits. [8](#0-7) 

---

### Proof of Concept

1. User holds 100 units of a bridged ERC-20 token on Aurora (backed by a NEP-141 token).
2. User calls the `ExitToNear` precompile (legacy path, flag `0x01`) specifying a recipient NEAR account that has **never registered** with the NEP-141 contract (no `storage_deposit`).
3. The precompile burns 100 ERC-20 tokens from the user's Aurora balance.
4. A `ft_transfer` promise is issued to the NEP-141 contract with no attached callback (because `error_refund` is not enabled and `transfer_near` is `None`).
5. The NEP-141 contract rejects `ft_transfer` because the recipient has no storage deposit.
6. There is no callback to detect the failure and re-mint the burned ERC-20 tokens.
7. The user has lost 100 ERC-20 tokens permanently. The NEP-141 tokens remain with the Aurora engine contract, inaccessible to the user.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
