### Title
SNS Governance `claim_neuron` Fails When `neuron_minimum_stake_e8s` Increases After Token Transfer — (`rs/sns/governance/src/governance.rs`)

### Summary
In SNS governance, a user who pre-commits tokens to the governance staking subaccount based on the current `neuron_minimum_stake_e8s` can have their `claim_neuron` call fail — and their tokens become stuck — if a `ManageNervousSystemParameters` governance proposal increases `neuron_minimum_stake_e8s` between the token transfer and the claim call. The code always validates against the **current** parameter at call time, not the value at the time of the user's pre-commitment.

### Finding Description

Staking a neuron in SNS governance is a two-step process:

1. The user transfers tokens to the governance-controlled neuron staking subaccount (a pre-commitment of funds).
2. The user calls `claim_neuron` (or `claim_or_refresh_neuron_from_account`) to register the neuron.

In `claim_neuron`, the governance canister reads `neuron_minimum_stake_e8s` from the **live** `NervousSystemParameters` at the time of the call:

```rust
let min_stake = self
    .nervous_system_parameters_or_panic()
    .neuron_minimum_stake_e8s
    .expect("NervousSystemParameters must have neuron_minimum_stake_e8s");
// ...
if balance.get_e8s() < min_stake {
    self.remove_neuron(&neuron_id, neuron)?;
    return Err(GovernanceError::new_with_message(
        ErrorType::InsufficientFunds, ...
    ));
}
```

`neuron_minimum_stake_e8s` is a mutable governance parameter that any SNS community can raise via a `ManageNervousSystemParameters` proposal, as confirmed by `perform_manage_nervous_system_parameters`:

```rust
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    // ...
    self.proto.parameters = Some(new_params);
    Ok(())
}
```

If `neuron_minimum_stake_e8s` is raised between step 1 and step 2, the user's balance — which was sufficient under the old parameter — is now below the new minimum. The claim fails, the embryo neuron is deleted, and the user's tokens remain in the governance subaccount with no automatic recovery path.

The NNS governance `claim_neuron` (`rs/nns/governance/src/governance.rs`) has the identical pattern, reading `neuron_minimum_stake_e8s` from `NetworkEconomics` at call time.

A secondary variant exists in `split_neuron`: the split amount is validated against the live `transaction_fee_e8s`, which can be changed independently via `ManageNervousSystemParameters` (without touching the actual ledger fee via `ManageLedgerParameters`). A user who calculated `split.amount_e8s = min_stake + old_fee` will have their split rejected if `transaction_fee_e8s` is raised before the call executes.

### Impact Explanation

For `claim_neuron`: the user's tokens are locked in the governance subaccount. The only recovery path is to transfer additional tokens to meet the new minimum and retry — or to accept that the stake is inaccessible until they can do so. The NNS governance code itself acknowledges this: *"We can't return the funds without more information about the source account."* A user who cannot afford additional tokens has their stake permanently inaccessible.

For `split_neuron`: the operation fails silently from the user's perspective; no tokens are lost, but the user must recalculate and resubmit.

### Likelihood Explanation

`ManageNervousSystemParameters` proposals are a routine part of SNS governance lifecycle. Raising `neuron_minimum_stake_e8s` is a legitimate governance action (e.g., to prevent dust neurons). The window between token transfer and `claim_neuron` is non-trivial: users may wait for ledger confirmation, experience network delays, or simply not monitor governance proposals continuously. Any unprivileged user who transfers tokens during an active governance vote on `neuron_minimum_stake_e8s` is at risk.

### Recommendation

- Store the `neuron_minimum_stake_e8s` value at the time of the ledger transfer (e.g., in a pending-claim record) and validate against that stored value in `claim_neuron`, not the live parameter.
- Alternatively, provide a dedicated `recover_stake` endpoint that allows a user to reclaim tokens from a failed claim subaccount without needing to meet the new minimum.
- At minimum, document that users should call `claim_neuron` immediately after the token transfer and before any governance vote on `neuron_minimum_stake_e8s` concludes.
