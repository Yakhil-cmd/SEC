Audit Report

## Title
Unguarded `wait_for` Awaits After `select!` Exit Allow Byzantine Peer to Permanently Leak Assemble Tasks — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

In `process_slot_update`, once the `assemble_artifact` branch of the outer `select!` wins, execution enters the arm body and reaches bare `peer_rx.wait_for(|p| p.is_empty()).await` calls at lines 500 and 521. These awaits have no cancellation guard. A single Byzantine subnet peer that advertises an artifact and never sends a slot-deletion will hold these awaits indefinitely, permanently leaking the Tokio task, keeping the `active_assembles` entry alive, and — on the `Done` path — preventing `UnvalidatedArtifactMutation::Remove` from ever being sent to the unvalidated pool.

## Finding Description

`process_slot_update` races three futures in a `select!`:

```
select! {
    _ = cancellation_token.cancelled() => {}   // branch A
    assemble_result = assemble_artifact => { … } // branch B
    _ = all_peers_deleted_artifact => { … }    // branch C
}
``` [1](#0-0) 

Once branch B wins, Tokio executes the arm body to completion; branch A is no longer polled. Inside that body, both the `Done` and `Unwanted` paths call:

```rust
let _ = peer_rx.wait_for(|p| p.is_empty()).await;  // lines 500, 521
// TODO: NET-1774
``` [2](#0-1) 

`peer_rx` is a `watch::Receiver<PeerCounter>`. The corresponding `watch::Sender` (`tx`) is inserted into `active_assembles` at line 405 and is only dropped inside `handle_artifact_processor_joined` when `self.active_assembles.remove(&id)` is called at line 318. [3](#0-2) [4](#0-3) 

`handle_artifact_processor_joined` is only reached when the task joins from `artifact_processor_tasks.join_next()`. The task is stuck at `wait_for`, so it never joins, the sender is never dropped, and `wait_for` never receives a `RecvError` to break out — a self-reinforcing deadlock.

The `PeerCounter` is decremented only when the event loop receives a slot-deletion or slot-overwrite from the advertising peer, via `sender.send_if_modified(|h| h.remove(peer_id))` at line 427. [5](#0-4) 

A Byzantine peer that never sends a deletion never triggers this decrement. `handle_topology_update` removes peers only if they leave the subnet topology; a Byzantine peer that remains a valid subnet member is never removed. [6](#0-5) 

For consensus artifacts, `slot_limit` is `SLOT_TABLE_NO_LIMIT = usize::MAX`, so there is no per-peer cap on how many artifact IDs a Byzantine peer can advertise. [7](#0-6) [8](#0-7) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

Concrete consequences per stuck task:

1. **Permanent task leak**: The Tokio task and its associated memory are never reclaimed.
2. **Unvalidated pool pollution** (`Done` path): `UnvalidatedArtifactMutation::Remove` is never sent, so the assembled artifact remains in the unvalidated pool indefinitely, consuming pool memory and potentially interfering with consensus processing.
3. **`active_assembles` map exhaustion**: The entry for the artifact ID is never removed. Any subsequent slot update for the same ID from an honest peer merely calls `send_if_modified` on the existing sender, which the stuck task is waiting on — no new assemble task can be started for that ID.
4. **Shutdown hang**: When the cancellation token fires during node shutdown, stuck tasks do not terminate, potentially blocking the `JoinSet` drain in `start_event_loop`. [9](#0-8) 

## Likelihood Explanation

The attacker is a single Byzantine subnet peer, well within the standard BFT fault tolerance of `f < n/3`. The attack requires no threshold corruption, no key material, and no privileged access:

1. Send one slot-update HTTP request to the victim node's P2P endpoint advertising artifact ID `X`.
2. Wait for the assembler to return `Done` or `Unwanted` (the task reaches line 500 or 521).
3. Never send a slot-deletion for slot `X`.

The `// TODO: NET-1774` comments at lines 499 and 520 confirm the developers are aware these awaits lack cancellation protection. The attack is repeatable for each new artifact ID, allowing unbounded resource exhaustion over time. [10](#0-9) [11](#0-10) 

## Recommendation

Wrap both `wait_for` calls in a `select!` that also polls `cancellation_token.cancelled()`. On the `Done` path, still send `Remove` on cancellation to avoid pool pollution:

```rust
// Done path (line 500):
select! {
    _ = cancellation_token.cancelled() => {
        // still send Remove to clean up unvalidated pool
        let _ = sender.send(UnvalidatedArtifactMutation::Remove(id)).await;
        return (peer_rx, id_c);
    }
    _ = peer_rx.wait_for(|p| p.is_empty()) => {}
}

// Unwanted path (line 521):
select! {
    _ = cancellation_token.cancelled() => { return (peer_rx, id_c); }
    _ = peer_rx.wait_for(|p| p.is_empty()) => {}
}
```

This is exactly what NET-1774 tracks.

## Proof of Concept

State-machine test (no network required, using the existing `ReceiverManagerBuilder` test harness):

1. Build a `ConsensusManagerReceiver` with a mock assembler that immediately returns `AssembleResult::Done`.
2. Call `handle_slot_update_receive` from `NODE_1` for artifact ID `0` — this spawns the assemble task and inserts the `watch::Sender` into `active_assembles`.
3. Await `Insert` on `unvalidated_artifact_receiver` — confirms the assembler returned `Done` and the task is now blocked at line 500.
4. Fire the cancellation token.
5. With a short timeout, attempt to join the assemble task from `artifact_processor_tasks.join_next()` — **the join never completes** because the task is stuck at `wait_for` with no cancellation guard.
6. Assert that `Remove` is never received on `unvalidated_artifact_receiver`.
7. Assert that `active_assembles` still contains the entry for artifact ID `0`.

The existing test `overwrite_slot_send_remove` demonstrates the normal path where a slot-overwrite triggers peer removal and unblocks `wait_for`. [12](#0-11)

    /// Notifies all running tasks about the topology update.
    fn handle_topology_update(&mut self) {
        self.metrics.topology_updates_total.inc();
        let new_topology = self.topology_watcher.borrow().clone();
        let mut nodes_leaving_topology = HashSet::new();

        self.slot_table.retain(|node_id, _| {
            if !new_topology.is_member(node_id) {
                nodes_leaving_topology.insert(*node_id);
                let _ = self
                    .metrics
                    .slot_table_new_entry_total
                    .remove_label_values(&[node_id.to_string().as_str()]);
                false
            } else {
                true
            }
        });

        for peers_sender in self.active_assembles.values() {
            peers_sender
                .send_if_modified(|set| nodes_leaving_topology.iter().any(|n| set.remove(*n)));
        }
        debug_assert!(
            self.slot_table.len() <= self.topology_watcher.borrow().iter().count(),
            "Slot table contains more nodes than nodes in subnet after pruning"
        );
    }
