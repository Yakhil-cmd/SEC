Audit Report

## Title
Guard Slot Exhaustion via Subaccount Enumeration Blocks All ckBTC Withdrawals - (File: rs/bitcoin/ckbtc/minter/src/guard.rs, rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs)

## Summary
`retrieve_btc_with_approval` keys its concurrency guard on the full `Account { owner, subaccount }` tuple, where `from_subaccount` is entirely attacker-controlled. Because the global `retrieve_btc_accounts` set is capped at `MAX_CONCURRENT = 100` and the duplicate check is an exact-`Account` membership test, a single unprivileged principal can occupy all 100 slots using 100 distinct subaccounts. The guard is held across the `check_address` inter-canister await with no funds check having occurred, so no ckBTC balance is required. While all slots are occupied, every call to `retrieve_btc` or `retrieve_btc_with_approval` by any user returns `TemporarilyUnavailable("too many concurrent requests")`, completely blocking ckBTC withdrawals.

## Finding Description

**Guard implementation** (`rs/bitcoin/ckbtc/minter/src/guard.rs`):

The global cap is a flat count of entries across all accounts: [1](#0-0) [2](#0-1) 

The duplicate check at line 48 is an exact `Account` equality test. Two calls from the same `owner` principal with different `subaccount` values are treated as two independent accounts and each consumes one slot.

**Guard acquisition in `retrieve_btc_with_approval`** (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`):

The guard key is built directly from the caller-supplied `from_subaccount`: [3](#0-2) 

The guard is acquired before any funds check and is then held across the `check_address` inter-canister await: [4](#0-3) [5](#0-4) 

`burn_ckbtcs_icrc2`, which actually requires ckBTC funds, only executes after `check_address` returns: [6](#0-5) 

**Contrast with `retrieve_btc`**: that function unconditionally passes `subaccount: None`, so a single principal can hold at most one slot there: [7](#0-6) 

**Existing test already proves the exhaustion pattern**: 50 guards from subaccounts of a single principal plus 50 from other principals fills all 100 slots: [8](#0-7) 

**Exploit flow:**
1. Attacker sends 100 concurrent calls to `retrieve_btc_with_approval`, each with a distinct `from_subaccount` value (e.g., `[i; 32]` for `i` in `0..100`), a valid Bitcoin address, and `amount >= min_retrieve_amount`.
2. Each call executes synchronously past `init_ecdsa_public_key().await` (cached after first call), acquires a guard slot at line 263, then suspends at the `check_address` inter-canister await.
3. All 100 slots are now occupied. No ckBTC balance has been checked.
4. Any subsequent call to `retrieve_btc` or `retrieve_btc_with_approval` by any user hits `accounts.len() >= MAX_CONCURRENT` and returns `TooManyConcurrentRequests`.
5. When `check_address` returns and `burn_ckbtcs_icrc2` fails (no funds), the guard is dropped — the attacker immediately re-submits 100 new calls to sustain the DoS indefinitely.

## Impact Explanation

All ckBTC → BTC withdrawals are blocked for the duration of the attack. This is a sustained, application-level DoS against the ckBTC minter's withdrawal functionality — a core Chain Fusion / ck-token financial integration. It maps to the **High ($2,000–$10,000)** impact class: *"Application/platform-level DoS... not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token... security impact with concrete user or protocol harm."*

## Likelihood Explanation

No privileged access, no ckBTC balance, and no social engineering are required. The attacker only needs a valid Bitcoin address and `amount >= min_retrieve_amount` (a public parameter). Generating 100 distinct 32-byte subaccount values is trivial. The attack is repeatable indefinitely at negligible cost, and the existing unit test in `guard.rs` already demonstrates the exact exhaustion pattern.

## Recommendation

1. **Per-principal slot accounting**: Count guard slots per `owner` principal rather than per `Account`. A single principal should not be able to hold more than one concurrent slot regardless of subaccount.
2. **Alternatively, key the guard on `owner` only** (ignoring `from_subaccount`), consistent with how `retrieve_btc` already behaves. Change line 259–263 to:
   ```rust
   let _guard = retrieve_btc_guard(Account {
       owner: caller,
       subaccount: None,
   })?;
   ```
   The actual `caller_account` (with subaccount) is still used for `burn_ckbtcs_icrc2`; only the guard key changes.
3. **Move the guard before `init_ecdsa_public_key().await`** (line 253) to eliminate the minor ordering issue where one await precedes guard acquisition.

## Proof of Concept

The existing test at `guard.rs` lines 141–162 already demonstrates the exhaustion pattern. A minimal state-machine extension:

```rust
// Acquire all 100 retrieve_btc guard slots from a single principal
let attacker = test_principal(0);
let guards: Vec<_> = (0u8..100)
    .map(|i| {
        retrieve_btc_guard(Account {
            owner: attacker,
            subaccount: Some([i; 32]),
        })
        .expect("slot should be available")
    })
    .collect();
assert_eq!(guards.len(), 100);

// Any legitimate user is now rejected
let victim = test_principal(1);
assert_eq!(
    retrieve_btc_guard(Account { owner: victim, subaccount: None }),
    Err(GuardError::TooManyConcurrentRequests)
);
```

In an integration/PocketIC test, send 100 concurrent `retrieve_btc_with_approval` calls (distinct `from_subaccount`, valid address, `amount = min_retrieve_amount`, zero ckBTC balance) and verify that a subsequent call from a different principal returns `TemporarilyUnavailable("too many concurrent requests")` before any of the 100 calls complete.

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
