### Title
Unchecked Precompile Call Return Value in `withdrawToEthereum` Allows Silent Token Burn Without Withdrawal Promise — (`etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

`withdrawToEthereum` burns ERC-20 tokens before calling the `ExitToEthereum` precompile via inline assembly. The assembly `call` return value (`res`) is never checked. When `amount > u128::MAX`, the precompile's `parse_amount` guard returns `Err(ExitError::Other("ERR_INVALID_AMOUNT"))`, which the EVM executor maps to `PrecompileFailure::Error` — a non-fatal subcall failure that returns `0` to the caller without reverting the parent context. The `_burn` has already committed, so the user's ERC-20 tokens are destroyed with no corresponding NEAR withdraw promise scheduled.

### Finding Description

**Step 1 — Entry point.**
Any EVM user holding a `uint256` balance > `u128::MAX` calls `withdrawToEthereum(recipient, amount)`.

**Step 2 — Burn executes unconditionally.** [1](#0-0) 

`_burn(_msgSender(), amount)` succeeds as long as the caller holds sufficient balance. OpenZeppelin's `_burn` reduces `totalSupply` and the caller's balance immediately.

**Step 3 — Precompile call is made with oversized amount.** [2](#0-1) 

The full `uint256` amount is packed into `amount_b` (32 bytes) and passed to the precompile. The assembly `call` result `res` is captured but **never checked**.

**Step 4 — Precompile rejects the amount.** [3](#0-2) 

`parse_amount` explicitly rejects any value exceeding `u128::MAX` with `Err(ExitError::Other("ERR_INVALID_AMOUNT"))`. This is confirmed by the existing test: [4](#0-3) 

**Step 5 — Error is mapped to a non-fatal subcall failure.** [5](#0-4) 

`process_precompile` maps `ExitError` → `PrecompileFailure::Error`. In EVM semantics, `PrecompileFailure::Error` causes the subcall to return `0` (failure) but does **not** revert the calling context. Only `PrecompileFailure::Fatal` would propagate upward and abort the transaction.

**Step 6 — `res` is never checked; no revert occurs.**

The Solidity assembly block captures `res` but performs no `iszero(res)` check and no `revert`. Execution returns normally from `withdrawToEthereum`.

**Step 7 — No promise log is produced.**

The `withdraw` promise is only constructed and emitted as a log after `parse_amount` succeeds: [6](#0-5) 

Since the precompile returned early at line 934, no `PromiseCreateArgs` log is ever written. The NEAR connector never receives a `withdraw` call, so the corresponding NEP-141 tokens remain locked in the Aurora contract with no claimant.

The same unchecked-`res` pattern exists in `withdrawToNear`: [7](#0-6) 

### Impact Explanation

The user's ERC-20 tokens are permanently burned (totalSupply decreases, balance zeroed), but the backing NEP-141 tokens on NEAR are never released. The user receives nothing on Ethereum. The NEP-141 tokens are frozen in the Aurora contract with no on-chain recovery path for the affected user. This matches **High — Temporary freezing of funds** (the NEP-141 tokens remain in the system but are inaccessible to the user without out-of-band admin intervention).

### Likelihood Explanation

The precondition is that a user holds a balance > `u128::MAX` (~3.4 × 10^38). Since `mint` is `onlyAdmin`, this requires an admin to have minted such an amount. While uncommon in practice, it is a valid and reachable code path — the ERC-20 uses `uint256` precisely to allow balances up to 2^256-1, and nothing in the contract prevents minting above `u128::MAX`. Any token with high decimal precision or large supply could reach this threshold.

### Recommendation

Check the return value of the precompile `call` and revert on failure:

```solidity
assembly {
    let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToNear`. Alternatively, add an explicit `require(amount <= type(uint128).max, "amount exceeds u128::MAX")` guard before `_burn` so the transaction fails before any state change.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IEvmErc20 {
    function withdrawToEthereum(address recipient, uint256 amount) external;
    function balanceOf(address account) external view returns (uint256);
    function totalSupply() external view returns (uint256);
}

contract PoC {
    // Precondition: admin has minted (u128::MAX + 1) tokens to this contract
    function exploit(IEvmErc20 token, address recipient) external {
        uint256 oversized = uint256(type(uint128).max) + 1;
        uint256 supplyBefore = token.totalSupply();

        // Burns tokens; precompile call silently fails; no revert
        token.withdrawToEthereum(recipient, oversized);

        uint256 supplyAfter = token.totalSupply();
        // Assert: totalSupply decreased but no promise log was produced
        assert(supplyAfter == supplyBefore - oversized);
        // No NEAR withdraw promise was scheduled → funds frozen
    }
}
```

Fuzz assertion: for any `amount` in `[u128::MAX + 1, type(uint256).max]`, post-call `totalSupply` decreases by `amount` but the Aurora engine emits zero promise logs for the `ExitToEthereum` precompile address.

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }

    Ok(amount)
}
