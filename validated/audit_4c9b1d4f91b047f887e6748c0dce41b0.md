Audit Report

## Title
Stale ICP/XDR Exchange Rate Used for Cycles Minting Without Freshness Check - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `tokens_to_cycles` function reads `state.icp_xdr_conversion_rate` with no check on `timestamp_seconds`, and the default state always initializes this field to `Some(...)` with a hardcoded 2021 rate (100 XDR/ICP, timestamp `1_620_633_600`). If the Exchange Rate Canister is unavailable or unconfigured, the CMC silently mints cycles at the stale default rate — up to ~12–30x the correct amount at current ICP market prices — for any unprivileged caller of `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles`.

## Finding Description
`tokens_to_cycles` at `rs/nns/cmc/src/main.rs:1900–1923` extracts only `xdr_permyriad_per_icp` from the cached rate and discards `timestamp_seconds` entirely. The only guard is a `None` match arm that is unreachable in practice because `State::default()` always sets `icp_xdr_conversion_rate` to `Some(initial_icp_xdr_conversion_rate)` at `rs/nns/cmc/src/main.rs:373`.

The default rate is defined in `rs/nns/cmc/src/lib.rs:33–34`:
```
DEFAULT_ICP_XDR_CONVERSION_RATE_TIMESTAMP_SECONDS = 1_620_633_600  // 10 May 2021
DEFAULT_XDR_PERMYRIAD_PER_ICP_CONVERSION_RATE     = 1_000_000      // 100 XDR/ICP
```

The heartbeat at `rs/nns/cmc/src/main.rs:2397–2402` only calls `update_exchange_rate()` when `exchange_rate_canister_id.is_some()`. The default state sets `exchange_rate_canister_id: None` (`rs/nns/cmc/src/main.rs:368`), so on a fresh deployment with no XRC configured, the rate is never refreshed. Even when the XRC is configured, `exchange_rate_canister.rs:16` shows the refresh interval is 5 minutes; if the XRC returns errors, the CMC logs the failure and continues serving minting calls at the last cached (potentially very old) rate with no user-visible error.

Exploit path:
1. Precondition: XRC is unavailable (subnet incident, canister stopped) or CMC is freshly deployed without XRC configured.
2. Attacker sends ICP to their CMC subaccount via the ICP ledger.
3. Attacker calls `notify_top_up` (or `notify_create_canister` / `notify_mint_cycles`) — no privileges required.
4. `tokens_to_cycles` reads the stale default rate (100 XDR/ICP) and mints `1 ICP × 100 XDR × 1T cycles/XDR = 100T cycles` instead of the correct ~3–8T cycles at current market rates.

## Impact Explanation
This constitutes illegal minting of cycles at a 12–30x ratio relative to the correct market rate. The `base_cycles_limit` rate limiter constrains per-user throughput but does not prevent the over-minting ratio itself; each minted cycle is backed by far less ICP than the protocol requires. During a sustained XRC outage, any number of unprivileged users can simultaneously exploit this, causing significant protocol-level economic harm. This matches the High impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
Two realistic triggers exist: (1) a fresh CMC deployment or upgrade before the first successful XRC heartbeat, and (2) a sustained XRC unavailability during a subnet incident. Neither requires special privileges — any user with ICP can call the affected endpoints. The XRC is a dependency of the CMC and its unavailability is a plausible operational condition. The window of exploitation scales with the duration of the outage.

## Recommendation
Add a staleness check in `tokens_to_cycles` that rejects conversions when the cached rate's `timestamp_seconds` is older than a configurable threshold (e.g., 1 hour):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref()
            .ok_or_else(|| NotifyError::Other { ... })?;
        let now = ic_cdk::api::time() / 1_000_000_000;
        if now.saturating_sub(rate.timestamp_seconds) > MAX_RATE_AGE_SECONDS {
            return Err(NotifyError::Other {
                error_code: NotifyErrorCode::Internal as u64,
                error_message: "ICP/XDR conversion rate is stale".to_string(),
            });
        }
        Ok(TokensToCycles {
            xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
            cycles_per_xdr: state.cycles_per_xdr,
        }.to_cycles(amount))
    })
}
```

Additionally, the default state should not pre-populate `icp_xdr_conversion_rate` with a 2021 value; it should remain `None` until the first successful XRC fetch.

## Proof of Concept
1. Instantiate `State::default()`. Confirm `icp_xdr_conversion_rate` is `Some({ timestamp_seconds: 1_620_633_600, xdr_permyriad_per_icp: 1_000_000 })` and `exchange_rate_canister_id` is `None`.
2. Do not configure an XRC canister ID (or simulate XRC returning errors). The heartbeat guard at `rs/nns/cmc/src/main.rs:2399` ensures `update_exchange_rate` is never called.
3. Call `tokens_to_cycles(Tokens::from_e8s(1_0000_0000))` (1 ICP). Observe it returns `100_000_000_000_000` cycles (100T) using the stale default rate.
4. At the correct current rate (~4 XDR/ICP), the expected result is ~4T cycles — a 25x over-mint.
