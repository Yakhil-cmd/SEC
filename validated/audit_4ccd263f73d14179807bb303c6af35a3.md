### Title
Silo Mode `erc20_fallback_address` Accepts Uncontrolled Addresses, Enabling Permanent Freeze of Bridged ERC-20 Tokens - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, when a NEP-141 token is bridged into Aurora via `ft_transfer_call` and the recipient EVM address is not whitelisted, the engine silently redirects the minted ERC-20 tokens to the configured `erc20_fallback_address`. The protocol imposes no validation on this address: it can be set to `Address::zero()` or any other address for which no private key exists. When that happens, the redirected tokens are permanently frozen inside the EVM state with no protocol-level recovery path — a direct structural analog to the FeeReceiver pattern where funds arrive but can never leave.

---

### Finding Description

**Step 1 — Tokens are silently redirected to the fallback address.**

In `engine/src/engine.rs`, `receive_erc20_tokens` checks whether the intended recipient is whitelisted. If not, it silently replaces the recipient with `erc20_fallback_address` and mints the full token amount there:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

The minting call that follows credits the full `amount` to `recipient` (now the fallback address) with no further checks.

**Step 2 — The fallback address is stored with zero validation.**

`set_erc20_fallback_address` in `engine/src/contract_methods/silo/mod.rs` writes any caller-supplied address directly to storage:

```rust
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);
    }
}
``` [2](#0-1) 

No check prevents `Address::zero()` or any other uncontrolled address from being stored.

**Step 3 — The public entrypoint enforces only ownership, not address validity.**

`engine/src/lib.rs` exposes `set_erc20_fallback_address` as a WASM export. It checks `require_owner_and_running` but performs no address-level validation before delegating to the storage write: [3](#0-2) 

**Step 4 — The test suite itself uses `Address::zero()` as the canonical fallback.**

The unit-test constant confirms this is a realistic, exercised configuration:

```rust
const ERC20_FALLBACK_ADDRESS: Address = Address::zero();
const SILO_PARAMS_ARGS: SiloParamsArgs = SiloParamsArgs {
    fixed_gas: FIXED_GAS,
    erc20_fallback_address: ERC20_FALLBACK_ADDRESS,
};
``` [4](#0-3) 

**Step 5 — No protocol-level recovery path exists.**

The only way to move ERC-20 tokens out of an EVM address is for the holder of that address's private key to call `withdrawToNear` or `withdrawToEthereum` on the ERC-20 contract. `Address::zero()` has no private key. The engine exposes no admin function to forcibly transfer or burn tokens held at the fallback address. The `SiloParamsArgs` type documents the field but provides no safety constraint: [5](#0-4) 

---

### Impact Explanation

Any ERC-20 tokens bridged into a Silo-mode Aurora instance whose recipient address is not whitelisted are permanently credited to `erc20_fallback_address`. If that address is `Address::zero()` (or any address whose key is unknown or lost), the tokens are irrecoverably frozen inside the EVM state. The NEP-141 supply on the NEAR side has already been transferred to the Aurora engine account, so the tokens are also removed from the user's NEAR balance. The result is a permanent, bilateral loss: the user loses their NEP-141 tokens and the corresponding ERC-20 balance is unspendable. This satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

- The test suite ships with `Address::zero()` as the canonical fallback value, demonstrating that the developers treat it as a valid configuration.
- A Silo operator may set the fallback to zero intentionally (to "burn" misdirected tokens) or accidentally (as a placeholder during setup).
- An operator may also set it to an address whose key is later lost or rotated away.
- The bridge entry path (`ft_transfer_call` → `ft_on_transfer` → `receive_erc20_tokens`) is the standard, documented way to bridge NEP-141 tokens into Aurora; no special attacker knowledge is required.

Likelihood: **Medium** (depends on operator configuration, but the zero-address case is explicitly exercised in tests).

---

### Recommendation

1. **Reject `Address::zero()` in `set_erc20_fallback_address`**: add a guard that panics or returns an error if the supplied address is the zero address.
2. **Optionally validate that the fallback address is whitelisted**: this ensures the address is at least known to the operator.
3. **Add an admin-only `recover_fallback_tokens` function**: allow the engine owner to forcibly transfer ERC-20 tokens from the fallback address to a specified destination, providing a last-resort recovery path analogous to the withdrawal function recommended in the original report.

---

### Proof of Concept

```
1. Silo operator calls set_silo_params / set_erc20_fallback_address
   with erc20_fallback_address = 0x0000000000000000000000000000000000000000

2. User (alice.near) calls ft_transfer_call on the NEP-141 contract:
     receiver_id = "aurora"
     amount      = 1_000_000
     msg         = hex(alice_evm_address)   // alice's EVM address

3. Aurora engine receives ft_on_transfer → receive_erc20_tokens:
     recipient = alice_evm_address
     is_allow_receive_erc20_tokens(alice_evm_address) == false  // not whitelisted
     → recipient = Address::zero()
     → ERC-20 mint(Address::zero(), 1_000_000) executes

4. Result:
     - alice.near NEP-141 balance: reduced by 1_000_000  (transferred to aurora)
     - alice EVM ERC-20 balance:   0
     - Address::zero() ERC-20 balance: 1_000_000  (no private key → unspendable)
     - No protocol function exists to move or recover these tokens.
```

The root cause is the absence of address validation in `set_erc20_fallback_address` (`engine/src/contract_methods/silo/mod.rs`) and the absence of any recovery mechanism for tokens accumulated at the fallback address, directly mirroring the FeeReceiver pattern where funds arrive but can never leave.
