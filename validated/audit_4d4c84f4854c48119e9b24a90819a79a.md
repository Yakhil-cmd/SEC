### Title
`notify_mint_cycles` Reports Inaccurate `minted` Amount in `NotifyMintCyclesSuccess`, Overstating Cycles Credited to User - (File: `rs/nns/cmc/src/main.rs`)

---

### Summary

The Cycles Minting Canister (CMC) `notify_mint_cycles` endpoint returns a `NotifyMintCyclesSuccess` struct whose `minted` field is set to the pre-computed cycles amount **before** the cycles ledger's deposit fee is deducted. The actual cycles credited to the user's cycles ledger account is `cycles - CYCLES_LEDGER_FEE`, but the CMC reports `minted: cycles` — the full pre-fee amount. Any caller (canister or user) that uses the `minted` return value for accounting will systematically overestimate the cycles they received.

---

### Finding Description

In `process_mint_cycles` (`rs/nns/cmc/src/main.rs`), after a successful call to `do_mint_cycles`, the success response is constructed as:

```rust
Ok(NotifyMintCyclesSuccess {
    block_index: deposit_result.block_index,
    minted: cycles.into(),          // pre-computed, BEFORE cycles ledger fee
    balance: deposit_result.balance, // actual post-fee balance from ledger
})
``` [1](#0-0) 

The `cycles` value is computed by `tokens_to_cycles(amount)` — the ICP-to-cycles conversion — and is passed directly to `do_mint_cycles` as the payment: [2](#0-1) 

Inside `do_mint_cycles`, this exact `cycles` amount is sent as `call_with_payment128` to the cycles ledger's `deposit` endpoint: [3](#0-2) 

The cycles ledger deducts its own fee from the deposited cycles before crediting the user. This is confirmed by the integration test, which shows the credited balance is `100_000_000_000_000 - CYCLES_LEDGER_FEE` for a 1 ICP deposit: [4](#0-3) 

The `CyclesLedgerDepositResult` returned by the cycles ledger contains the accurate post-fee `balance`, which is correctly forwarded in the response. However, the `minted` field is set to the pre-fee `cycles` value, not to the actual amount credited (`deposit_result.balance - balance_before`): [5](#0-4) 

The `minted` field is documented as "Amount of cycles that were minted and deposited to the cycles ledger" — but it overstates the actual deposited amount by exactly `CYCLES_LEDGER_FEE`.

---

### Impact Explanation

Any canister or off-chain system that calls `notify_mint_cycles` and uses `NotifyMintCyclesSuccess.minted` for internal accounting will record more cycles than were actually credited. Over many calls, this accumulates into a systematic overcount of `N * CYCLES_LEDGER_FEE` cycles. Downstream canisters that track their cycles ledger balance using the `minted` field (rather than querying the ledger directly) will believe they have more cycles than they do, potentially leading to incorrect resource allocation decisions or failed operations when the actual balance is lower than expected.

The `balance` field is accurate (it comes from `deposit_result.balance`), but the `minted` field — which is the natural value to use for "how many cycles did I just receive" — is wrong.

---

### Likelihood Explanation

This affects every caller of `notify_mint_cycles` that uses the `minted` field for accounting. The endpoint is publicly callable by any unprivileged principal. The discrepancy is deterministic and reproducible on every successful call. No special conditions are required.

---

### Recommendation

Replace `minted: cycles.into()` with the actual amount credited by the cycles ledger. The simplest approach is to query the user's balance before and after the deposit and use the difference, or to have the cycles ledger return the credited amount (net of fee) in `CyclesLedgerDepositResult`. At minimum, the `minted` field should be set to `deposit_result.balance - balance_before_deposit` to reflect the real credit:

```rust
Ok(NotifyMintCyclesSuccess {
    block_index: deposit_result.block_index,
    minted: deposit_result.balance - balance_before,  // actual credited amount
    balance: deposit_result.balance,
})
```

Alternatively, the cycles ledger's `deposit` endpoint could be extended to return the net credited amount directly in `CyclesLedgerDepositResult`. [6](#0-5) 

---

### Proof of Concept

1. User sends 1 ICP to the CMC subaccount with `MEMO_MINT_CYCLES`.
2. User calls `notify_mint_cycles` with the resulting `block_index`.
3. CMC converts 1 ICP → 100,000,000,000,000 cycles via `tokens_to_cycles`.
4. CMC calls cycles ledger `deposit` with 100T cycles as payment.
5. Cycles ledger deducts `CYCLES_LEDGER_FEE` and credits `100T - CYCLES_LEDGER_FEE` to the user.
6. CMC returns `NotifyMintCyclesSuccess { minted: 100T, balance: 100T - CYCLES_LEDGER_FEE }`.
7. The `minted` field overstates the actual credited amount by `CYCLES_LEDGER_FEE`.

The integration test at `rs/nns/integration_tests/src/cycles_minting_canister.rs` line 1315–1318 confirms the actual credited balance is `100_000_000_000_000 - CYCLES_LEDGER_FEE`, while `process_mint_cycles` at line 1971 reports `minted: cycles.into()` = `100_000_000_000_000`. [7](#0-6)
