### Title
Unbounded `DeleteDeposits`/`RefundDeposits` Iteration in `gov.EndBlocker` Enables Block-Delay DoS - (`sei-cosmos/x/gov/keeper/deposit.go`)

### Summary
An unprivileged attacker can create a large number of governance deposit records (one per Sybil address, each with a minimal 1 usei deposit) against a single proposal. When the proposal's deposit or voting period expires, `gov.EndBlocker` calls `DeleteDeposits` or `RefundDeposits`, which iterate over every deposit record with no batch limit and no gas cap. This unbounded work executes inside a single block's `EndBlock`, causing block-production delay proportional to the number of attacker-controlled deposit records.

### Finding Description

`gov.EndBlocker` runs every block and processes all proposals whose deadlines have passed: [1](#0-0) 

For each expired inactive proposal it calls `DeleteDeposits`, and for each expired active proposal it calls `DeleteDeposits` or `RefundDeposits`: [2](#0-1) 

Both functions iterate over every deposit record for the proposal with no limit: [3](#0-2) [4](#0-3) 

The underlying `IterateDeposits` is a raw KV prefix scan with no cap: [5](#0-4) 

`AddDeposit` enforces no minimum deposit amount — any positive coin value (1 usei) is accepted, and each unique depositor address creates a separate record: [6](#0-5) 

The deposit key is `(proposalID, depositorAddr)`, so each Sybil address contributes exactly one record. The deposit period is 2 days (`DefaultPeriod = 48h`): [7](#0-6) 

### Impact Explanation

`EndBlocker` is not gas-limited. All deposit records for an expired proposal are processed synchronously in a single block. Each record requires a KV read, a bank `BurnCoins`/`SendCoins` operation, and a KV delete. With tens of thousands of records, the cumulative work can push block processing time well beyond 2.5 seconds, satisfying the **Medium** impact threshold (block delay > 2.5 s) in the Sei bounty scope.

If the attacker coordinates multiple proposals expiring simultaneously (each seeded with many deposits), the delay compounds across all of them in the same `EndBlocker` call, since `IterateInactiveProposalsQueue` and `IterateActiveProposalsQueue` also have no batch limit: [8](#0-7) 

### Likelihood Explanation

The attack is cheap and permissionless:
- Submitting a proposal costs the minimum deposit (10 SEI, `DefaultMinDepositTokens`).
- Each Sybil deposit costs only the transaction fee (~2000 usei on Sei) plus 1 usei deposited.
- During the 2-day deposit window, Sei's high throughput allows an attacker to submit tens of thousands of deposit transactions at a total cost of tens of SEI.
- No privileged access, governance role, or validator key is required.

### Recommendation

1. **Enforce a minimum per-deposit amount** (e.g., a governance parameter `MinSingleDeposit`) in `AddDeposit`, making Sybil flooding economically prohibitive.
2. **Batch-limit deposit cleanup** in `EndBlocker`: process at most N deposits per block and carry over the remainder to subsequent blocks, preventing any single block from doing unbounded work.
3. **Cap the total number of depositors per proposal** as an alternative hard limit.

### Proof of Concept

1. Attacker submits a governance proposal, paying the 10 SEI minimum deposit. The proposal enters `StatusDepositPeriod` for 2 days.
2. Attacker generates 50,000 fresh addresses, funds each with ~4000 usei (1 usei deposit + ~2000 usei fee + buffer), and submits `MsgDeposit` with `amount = 1usei` from each address. Each call succeeds because `AddDeposit` has no minimum-amount guard.
3. The proposal never reaches the minimum total deposit threshold, so it remains inactive and expires after 2 days.
4. At the expiry block, `gov.EndBlocker` → `IterateInactiveProposalsQueue` → `DeleteDeposits` iterates all 50,000 records, performing 50,000 bank `BurnCoins` + KV delete operations inside a single `EndBlock`. This causes the block to take several seconds to finalize, exceeding the 2.5 s Medium threshold.

func (keeper Keeper) DeleteDeposits(ctx sdk.Context, proposalID uint64) {
	store := ctx.KVStore(keeper.storeKey)

	keeper.IterateDeposits(ctx, proposalID, func(deposit types.Deposit) bool {
		err := keeper.bankKeeper.BurnCoins(ctx, types.ModuleName, deposit.Amount)
		if err != nil {
			panic(err)
		}

		depositor := sdk.MustAccAddressFromBech32(deposit.Depositor)

		store.Delete(types.DepositKey(proposalID, depositor))
		return false
	})
}

func (keeper Keeper) IterateDeposits(ctx sdk.Context, proposalID uint64, cb func(deposit types.Deposit) (stop bool)) {
	store := ctx.KVStore(keeper.storeKey)
	iterator := sdk.KVStorePrefixIterator(store, types.DepositsKey(proposalID))

	defer func() { _ = iterator.Close() }()

	for ; iterator.Valid(); iterator.Next() {
		var deposit types.Deposit

		keeper.cdc.MustUnmarshal(iterator.Value(), &deposit)

		if cb(deposit) {
			break
		}
	}
}

// AddDeposit adds or updates a deposit of a specific depositor on a specific proposal
// Activates voting period when appropriate
func (keeper Keeper) AddDeposit(ctx sdk.Context, proposalID uint64, depositorAddr sdk.AccAddress, depositAmount sdk.Coins) (bool, error) {
	// Checks to see if proposal exists
	proposal, ok := keeper.GetProposal(ctx, proposalID)
	if !ok {
		return false, sdkerrors.Wrapf(types.ErrUnknownProposal, "%d", proposalID)
	}

	// Check if proposal is still depositable
	if (proposal.Status != types.StatusDepositPeriod) && (proposal.Status != types.StatusVotingPeriod) {
		return false, sdkerrors.Wrapf(types.ErrInactiveProposal, "%d", proposalID)
	}

	// update the governance module's account coins pool
	err := keeper.bankKeeper.SendCoinsFromAccountToModule(ctx, depositorAddr, types.ModuleName, depositAmount)
	if err != nil {
		return false, err
	}

	// Update proposal
	proposal.TotalDeposit = proposal.TotalDeposit.Add(depositAmount...)
	keeper.SetProposal(ctx, proposal)

	// Check if deposit has provided sufficient total funds to transition the proposal into the voting period
	activatedVotingPeriod := false

	if proposal.Status == types.StatusDepositPeriod && proposal.TotalDeposit.IsAllGTE(keeper.GetDepositParams(ctx).GetMinimumDeposit(proposal.IsExpedited)) {
		keeper.ActivateVotingPeriod(ctx, proposal)

		activatedVotingPeriod = true
	}

	// Add or update deposit object
	deposit, found := keeper.GetDeposit(ctx, proposalID, depositorAddr)

	if found {
		deposit.Amount = deposit.Amount.Add(depositAmount...)
	} else {
		deposit = types.NewDeposit(proposalID, depositorAddr, depositAmount)
	}

	// called when deposit has been added to a proposal, however the proposal may not be active
	keeper.AfterProposalDeposit(ctx, proposalID, depositorAddr)

	ctx.EventManager().EmitEvent(
		sdk.NewEvent(
			types.EventTypeProposalDeposit,
			sdk.NewAttribute(sdk.AttributeKeyAmount, depositAmount.String()),
			sdk.NewAttribute(types.AttributeKeyProposalID, fmt.Sprintf("%d", proposalID)),
		),
	)

	keeper.SetDeposit(ctx, deposit)

	return activatedVotingPeriod, nil
}

// IterateInactiveProposalsQueue iterates over the proposals in the inactive proposal queue
// and performs a callback function
func (keeper Keeper) IterateInactiveProposalsQueue(ctx sdk.Context, endTime time.Time, cb func(proposal types.Proposal) (stop bool)) {
	iterator := keeper.InactiveProposalQueueIterator(ctx, endTime)

	defer func() { _ = iterator.Close() }()
	for ; iterator.Valid(); iterator.Next() {
		proposalID, _ := types.SplitInactiveProposalQueueKey(iterator.Key())
		proposal, found := keeper.GetProposal(ctx, proposalID)
		if !found {
			panic(fmt.Sprintf("proposal %d does not exist", proposalID))
		}

		if cb(proposal) {
			break
		}
	}
}
