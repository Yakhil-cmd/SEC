Audit Report

## Title
Fee-on-Transfer ERC-20 Deposit Emits Unreduced Amount, Causing ckERC20 Over-Minting - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol, rs/ethereum/cketh/minter/ERC20DepositHelper.sol)

## Summary

Both ckERC20 helper smart contracts emit the deposit event using the caller-supplied `amount` parameter rather than the actual balance delta received by the minter address. The IC ckETH minter canister parses this event value and mints exactly that many ckERC20 tokens. For any fee-on-transfer or deflationary ERC-20 token added to the supported list, the minter receives fewer tokens than the event records, breaking the 1:1 backing invariant and inflating the ckERC20 supply beyond the actual ERC-20 collateral held.

## Finding Description

**Root cause — `ERC20DepositHelper.sol` `CkErc20Deposit.deposit()`:**

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal); // ← input `amount`, not balance delta
}
```

**Root cause — `DepositHelperWithSubaccount.sol` `CkDeposit.depositErc20()`:**

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount); // ← input `amount`, not balance delta
```

In both cases the emitted `amount` is the *requested* transfer amount. For a fee-on-transfer token, `minterAddress` receives `amount - fee`, but the event records `amount`.

**Minter canister trust chain:**

`ReceivedErc20LogParser::parse_log()` in `parser.rs` reads the event data field directly:
```rust
let [value_bytes] = parse_hex_into_32_byte_words(entry.data, event_source)?;
value: Erc20Value::from_be_bytes(value_bytes),  // trusts the event-logged amount
```

`mint()` in `deposit.rs` then mints exactly `event.value()` ckERC20 tokens:
```rust
amount: event.value(),  // mints the event-logged amount, not actual received
```

There is no on-chain or off-chain reconciliation step that compares the minted ckERC20 supply against the actual ERC-20 balance held by the minter address. The `erc20_balances` tracked in `State` are computed purely from audit events (which themselves derive from the emitted log value), not from on-chain `balanceOf` queries.

**Existing guards are insufficient:** The minter enforces a whitelist of supported ERC-20 tokens (checked in `deposit.rs` before minting), and currently supported tokens (USDC, USDT, LINK, PEPE, SHIB, UNI, WBTC, wstETH, EURC, XAUt, OCT) are not fee-on-transfer. However, the helper contracts themselves enforce no whitelist, and the minter's supported token list is governed by NNS proposals and can be extended to any ERC-20 token. No guard prevents a fee-on-transfer token from being added.

## Impact Explanation

For any fee-on-transfer ERC-20 token added to the minter's supported list, every deposit mints more ckERC20 than the ERC-20 tokens actually held by the minter. The ckERC20 supply becomes partially unbacked. When users later attempt to withdraw ckERC20 back to ERC-20, the minter will eventually be unable to fulfill withdrawals because it holds less ERC-20 than the outstanding ckERC20 supply. This constitutes **illegal minting and protocol insolvency** for the affected ck-token, matching the allowed ICP bounty impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving in-scope chain-key/ledger assets."* The deficit accumulates with every deposit and is unbounded. Severity is **High** given the latency constraint (requires a fee-on-transfer token to first be added to the supported list).

## Likelihood Explanation

The exploit is latent: it requires a fee-on-transfer or rebasing ERC-20 token to be added to the minter's supported list via an NNS proposal, which is a normal governance operation. Once such a token is added, any unprivileged Ethereum user calling `deposit()` or `depositErc20()` with that token triggers the over-minting — no special privileges, no front-running, no complex setup. The NNS governance process is designed to add new tokens (as evidenced by the addition of PEPE, SHIB, wstETH, XAUt, etc.), and there is no technical barrier preventing a fee-on-transfer token from being proposed and accepted. The vulnerability is fully reachable in production code.

## Recommendation

Both helper contracts should measure the actual balance delta at the minter address rather than emitting the caller-supplied `amount`:

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    uint256 balanceBefore = erc20Token.balanceOf(cketh_minter_main_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    uint256 actualReceived = erc20Token.balanceOf(cketh_minter_main_address) - balanceBefore;
    emit ReceivedErc20(erc20_address, msg.sender, actualReceived, principal);
}
```

Apply the same pattern to `CkDeposit.depositErc20()` in `DepositHelperWithSubaccount.sol`. The IC minter canister code in `deposit.rs` and `parser.rs` requires no changes — it already trusts the event value, and the fix is to make the event value accurate.

## Proof of Concept

1. Deploy a fee-on-transfer ERC-20 token (e.g., 1% fee on every transfer) on Ethereum mainnet or Sepolia.
2. Submit an NNS proposal to add this token to the ckETH minter's supported list (following the standard `AddErc20Arg` flow documented in `rs/ethereum/ledger-suite-orchestrator/README.adoc`).
3. Approve the `DepositHelperWithSubaccount` contract for `1,000,000` units and call `depositErc20(tokenAddr, 1_000_000, principal, subaccount)`.
4. The helper calls `safeTransferFrom(..., 1_000_000)`. Due to the 1% fee, `minterAddress` receives `990,000` tokens. The helper emits `ReceivedEthOrErc20(..., 1_000_000, ...)`.
5. The IC minter scrapes the log, `ReceivedErc20LogParser::parse_log()` parses `value = 1_000_000`, and `mint()` mints `1,000,000` ckERC20 tokens to the depositor.
6. The depositor holds `1,000,000` ckERC20 but the minter only holds `990,000` ERC-20 tokens — a `10,000`-unit deficit per deposit.
7. Repeating inflates the ckERC20 supply unboundedly relative to actual ERC-20 collateral, eventually making the bridge insolvent for that token.

A deterministic integration test can be written using the existing `CkErc20Setup` state-machine test harness in `rs/ethereum/cketh/minter/tests/ckerc20.rs` by deploying a mock fee-on-transfer ERC-20 and asserting that `erc20_balance_from_get_minter_info()` is less than the minted ckERC20 supply after a deposit.
