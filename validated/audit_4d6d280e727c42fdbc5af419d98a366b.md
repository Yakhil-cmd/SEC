### Title
Engine Owner Can Redirect All ETH Connector Operations to a Malicious Account, Enabling Fund Theft - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The Aurora Engine exposes a `set_eth_connector_contract_account` function callable by the engine owner that replaces the stored ETH connector account ID with any arbitrary NEAR account. Because every user-facing connector operation (`withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw`) resolves its cross-contract call target dynamically from this stored account ID at call time, a malicious or compromised owner can silently redirect all in-flight user fund operations to an attacker-controlled contract. This is a structural single point of failure that mirrors the `setImplementation()` pattern in the reported `LiquidityPoolProxy` vulnerability.

---

### Finding Description

**Root cause — `set_eth_connector_contract_account`**

`engine/src/contract_methods/connector.rs` lines 418–438:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);          // ← overwrites stored connector
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
```

The function accepts any `AccountId` and writes it unconditionally to storage. There is no validation that the new account is a legitimate ETH connector, no timelock, and no multi-sig requirement.

**Propagation — `return_promise` reads the stored account at call time**

`engine/src/contract_methods/connector.rs` lines 598–616:

```rust
fn return_promise<I: IO + PromiseHandler, E: Env>(
    mut io: I, env: &E, method: &str, args: Vec<u8>, deposit: Yocto,
) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
```

Every connector operation delegates to `return_promise`, which resolves the target account from storage at the moment of the call. The affected user-facing entry points are:

### Impact Explanation

**Critical — Direct theft of user funds.**

Once the connector account is replaced with a malicious NEAR contract, every subsequent call to `withdraw` causes the Aurora Engine to:
1. Accept the user's ETH burn (the EVM-side balance is decremented).
2. Forward the cross-contract call to the attacker's contract instead of the legitimate ETH connector.
3. The attacker's contract ignores or mishandles `engine_withdraw`, so the user receives zero NEAR tokens.

The same applies to `ft_transfer` and `ft_transfer_call` — NEP-141 token balances are debited on the engine side but the cross-contract transfer goes to the attacker. Because the EVM-side state change and the NEAR-side promise are in the same transaction batch, the EVM debit is committed while the NEAR credit is lost.

---

### Likelihood Explanation

**Medium.** The owner of the Aurora Engine is a single NEAR account (the `aurora` account or its DAO). A single compromised key, a malicious governance proposal, or a social-engineering attack on the key holder is sufficient to trigger the exploit. No on-chain timelock or multi-party approval is enforced by the contract code itself before the connector account is overwritten. The comment in `set_erc20_metadata` at line 377 (`// TODO: Define special role for this transaction. Potentially via multisig?`) shows the development team is aware that these privileged setter functions lack adequate access control decomposition, but no such control exists for `set_eth_connector_contract_account`.

---

### Recommendation

**Short term:**
- Document explicitly that `set_eth_connector_contract_account` can redirect all user fund flows and that the owner key is a critical single point of failure.
- Enforce a mandatory timelock (e.g., `upgrade_delay_blocks` already exists in `EngineState`) before a new connector account becomes active, giving users time to exit.

**Long term:**
- Decompose the owner role: separate the "connector routing" privilege from the general owner role and require multi-sig or DAO governance approval specifically for connector account changes.
- Validate that the new connector account responds to a known interface before committing the change (e.g., a view call to `ft_metadata` that must succeed).

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker gains control of the engine owner account (e.g., `aurora`).
2. Attacker deploys a malicious NEAR contract `evil.near` that implements `engine_withdraw` as a no-op (returns success but transfers no NEAR).
3. Attacker calls `set_eth_connector_contract_account` with `account = "evil.near"`.
   - `require_owner_only` passes because the attacker controls the owner account.
   - `set_connector_account_id` writes `"evil.near"` to storage.
4. Victim calls `withdraw` on the Aurora Engine to bridge 10 ETH back to NEAR.
   - `withdraw` (line 43) calls `return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)`.
   - `return_promise` (line 606) resolves `target_account_id = get_connector_account_id(&io)` → `"evil.near"`.
   - A cross-contract call to `evil.near::engine_withdraw` is scheduled.
5. `evil.near::engine_withdraw` executes, does nothing, returns success.
6. The victim's EVM ETH balance has been decremented; no NEAR tokens are received.
7. The attacker can drain `evil.near` of any NEAR that was attached to the promise.

The necessary vulnerable step is `return_promise` resolving the connector account dynamically from storage (line 606) with no integrity check, making every connector operation immediately affected by a connector account substitution. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
