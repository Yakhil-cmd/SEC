Audit Report

## Title
ckETH/ckERC20 Withdrawal Requests Permanently Stuck With No Reimbursement on Gas Fee Spike - (`rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

When Ethereum gas fees rise above a user's withdrawal amount (ckETH) or pre-burned gas budget (ckERC20), the minter's `create_transactions_batch` function returns `CreateTransactionError::InsufficientTransactionFee` and calls `reschedule_withdrawal_request`, looping the request to the back of the pending queue indefinitely. No reimbursement is ever triggered for this code path. For ckERC20 withdrawals, both the ckETH gas-fee tokens and the ckERC20 tokens are already burned and irrecoverable while the request cycles in the queue.

## Finding Description

**ckETH path:**

In `rs/ethereum/cketh/minter/src/main.rs` L301–312, the user's full `withdrawal_amount` is burned on the ckETH ledger before any gas check occurs. [1](#0-0) 

In `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` L1125–1133, `create_transaction` checks `withdrawal_amount.checked_sub(max_transaction_fee)` and returns `InsufficientTransactionFee` if the fee exceeds the amount. [2](#0-1) 

In `rs/ethereum/cketh/minter/src/withdraw.rs` L281–291, the sole handler for this error arm calls `reschedule_withdrawal_request` — no reimbursement event is emitted. [3](#0-2) 

**ckERC20 path:**

In `main.rs` L448–481, both the ckETH gas-fee (`erc20_tx_fee`) and the ckERC20 tokens are burned before the request is enqueued. The `max_transaction_fee` field is fixed at the burned amount. [4](#0-3) 

In `transactions/mod.rs` L1159–1168, the ckERC20 branch checks `actual_min_max_fee_per_gas > request_max_fee_per_gas` and returns `InsufficientTransactionFee`. [5](#0-4) 

The same `reschedule_withdrawal_request` call at `withdraw.rs` L290 handles this error — again with no reimbursement. [6](#0-5) 

**`reschedule_withdrawal_request` has no escape logic:**

The function at `transactions/mod.rs` L469–483 simply removes the request and re-appends it to the back of `pending_withdrawal_requests` with no retry counter, no expiry timestamp, and no state change. [7](#0-6) 

**The `FailedErc20WithdrawalRequest` reimbursement path does NOT cover this case:**

`FailedErc20WithdrawalRequest` is only emitted in `main.rs` L525–530 when the ckERC20 ledger burn fails at call time (i.e., during `withdraw_erc20` itself). It is never emitted from `create_transactions_batch`. [8](#0-7) 

There is no code path that transitions a pending withdrawal request from the queue to a reimbursement state when `InsufficientTransactionFee` is returned during batch processing.

## Impact Explanation

This is a **High** severity finding matching: *"Significant Chain Fusion, ck-token, ledger security impact with concrete user or protocol harm."*

- **ckETH withdrawals:** A user's ckETH is burned at call time. If fees spike above the withdrawal amount, the request loops forever. The user permanently loses their ckETH with no ETH delivered and no refund.
- **ckERC20 withdrawals:** Both the ckETH gas-fee tokens and the ckERC20 tokens are burned at call time. Both are permanently locked in the minter's pending queue with no reimbursement path while fees remain elevated.
- **Queue degradation:** Stuck requests are re-evaluated on every timer tick, wasting minter cycles and delaying legitimate withdrawals for all users.

The minimum ckETH withdrawal amount was recently reduced to `5_000_000_000_000_000` wei (~$10), which provides only a ~10× safety margin against fee spikes — a margin that historical Ethereum congestion events have exceeded. [9](#0-8) 

## Likelihood Explanation

Any unprivileged user can trigger this by calling `withdraw_eth` or `withdraw_erc20` with a near-minimum amount during a low-fee period. No special access is required. Ethereum gas fee spikes of 10–100× within hours are historically documented (NFT mints, network congestion). The recently reduced minimum withdrawal amount makes near-minimum submissions more common and the safety margin tighter. The condition is not contrived; it is a normal market event.

## Recommendation

1. **Add a retry limit or expiry timestamp** to `WithdrawalRequest`. After N failed `InsufficientTransactionFee` attempts or after a configurable deadline, emit a reimbursement event instead of rescheduling.
2. **For ckERC20 requests**, when `InsufficientTransactionFee` is returned in `create_transactions_batch`, emit a `FailedErc20WithdrawalRequest` event (mirroring the existing call-time failure path in `withdraw_erc20`) to reimburse both the burned ckETH gas fee and the ckERC20 tokens.
3. **For ckETH requests**, when the retry limit is exceeded, emit a `ReimbursedEthWithdrawal` event to return the burned ckETH to the user.
4. Consider enforcing a minimum withdrawal amount that accounts for realistic worst-case gas spikes (e.g., 100× the current estimate) rather than the current ~10× margin.

## Proof of Concept

1. Query `eip_1559_transaction_price` to confirm current `max_fee_per_gas` is low (e.g., 1 gwei).
2. Call `withdraw_eth` with `amount = 5_000_000_000_000_000` wei (current minimum). ckETH is burned immediately at `main.rs` L301–312.
3. Simulate an Ethereum fee spike (e.g., set `base_fee_per_gas` to 20 gwei in a local/PocketIC test environment).
4. Trigger the minter timer to invoke `create_transactions_batch`. Observe `create_transaction` returns `InsufficientTransactionFee` at `transactions/mod.rs` L1125–1133.
5. Observe `reschedule_withdrawal_request` is called at `withdraw.rs` L290 — no reimbursement event is emitted.
6. Trigger the timer repeatedly. Confirm the request cycles in `pending_withdrawal_requests` indefinitely with no reimbursement and no state change.
7. For ckERC20: repeat with `withdraw_erc20`. Confirm both the ckETH gas fee and ckERC20 tokens are burned at step 2 and that neither is reimbursed while the request cycles in the queue.

A deterministic integration test using PocketIC can reproduce this by controlling the gas fee estimate returned to the minter and asserting that after N timer ticks with elevated fees, no reimbursement mint appears on either ledger.

* Reduce the minimum ETH withdrawal amount by a factor of 6, from 0.03 ETH (`30_000_000_000_000_000` wei) to 0.005 ETH (`5_000_000_000_000_000` wei) — approximately $10 at current prices. The reasoning is as follows:
    * The current minimum dates back to December 2023, when the ckETH minter was installed (see proposal [126171](https://dashboard.internetcomputer.org/proposal/126171)). At that time ETH traded in a similar USD range (around $2000), but Ethereum mainnet transaction fees were averaging $5–$10 per transaction ([source](https://bitinfocharts.com/comparison/ethereum-transactionfees.html#3y)).
    * Today, Ethereum mainnet fees are in the order of cents and rarely exceed $1.
    * As explained [here](https://github.com/dfinity/ic/blob/14382b5abb14b8e7de2bd4a3fb402ba069b82861/rs/ethereum/cketh/docs/cketh.adoc?plain=1#L208), an order-of-magnitude safety margin is preserved so the minter can always submit the transaction even when the Ethereum network is congested and one or more resubmissions are needed (each resubmission requires at least a 10% fee bump). With current Ethereum fees of ~$0.10–$1, a $10 minimum still preserves the ~10× safety margin even after several fee bumps.
