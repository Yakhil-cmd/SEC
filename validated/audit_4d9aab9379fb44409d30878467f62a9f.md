### Title
Missing Zero-Address Validation in `ExitToEthereum` Precompile Allows Permanent Burning of User Funds - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile and the NEAR-side `withdraw` connector function both accept a caller-supplied Ethereum recipient address without validating it is non-zero. An unprivileged EVM user or NEAR token holder who accidentally (or deliberately) supplies `address(0)` as the recipient will permanently destroy their funds with no recovery path.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run()` function handles two exit paths:

**Flag `0x00` – ETH (base token) exit:** [1](#0-0) 

`recipient_address` is parsed directly from the 20 caller-controlled bytes following the flag byte. There is no check that `recipient_address != Address::zero()`. The address is immediately forwarded to the ETH connector's `withdraw` call.

**Flag `0x01` – ERC-20 exit:** [2](#0-1) 

Same pattern: `recipient_address` is parsed from the trailing 20 bytes of caller-controlled input with no zero-address guard.

The NEAR-side entry point `withdraw` in `engine/src/contract_methods/connector.rs` has the same omission: [3](#0-2) 

`args.recipient_address` is read from Borsh-encoded user input and forwarded to `engine_withdraw` without any zero-address check. This function is callable by any NEAR account holding ETH-on-NEAR (NEP-141) tokens.

The `WithdrawCallArgs` struct confirms `recipient_address` is a plain `Address` field with no invariant enforcement: [4](#0-3) 

---

### Impact Explanation

When `recipient_address` is `Address::zero()` (`0x0000000000000000000000000000000000000000`):

1. The ETH-on-NEAR (NEP-141) balance of the caller is burned/decremented on the NEAR side.
2. A NEAR promise is created to call `withdraw` on the ETH connector contract with the zero Ethereum address as recipient.
3. The ETH connector issues a withdrawal proof for the zero address on Ethereum.
4. The funds are permanently unrecoverable — the zero address on Ethereum is not controlled by any party.

This constitutes **permanent freezing (burning) of funds**, matching the "Critical – Permanent freezing of funds" impact tier.

---

### Likelihood Explanation

The `ExitToEthereum` precompile is reachable by any EVM contract or EOA on Aurora. The NEAR-side `withdraw` is callable by any NEAR account holding ETH-on-NEAR with a 1-yoctoNEAR deposit. Both are standard, documented user-facing interfaces. Accidental zero-address submission is a well-known class of user error (copy-paste of uninitialized variable, off-by-one in calldata encoding, etc.). No special privilege is required.

---

### Recommendation

Add a zero-address guard in both locations:

1. In `ExitToEthereum::run()` (both flag `0x00` and `0x01` branches), immediately after parsing `recipient_address`:
   ```rust
   if recipient_address == Address::zero() {
       return Err(ExitError::Other(Cow::from("ERR_ZERO_RECIPIENT_ADDRESS")));
   }
   ```

2. In `withdraw()` in `engine/src/contract_methods/connector.rs`, after deserializing `args`:
   ```rust
   if args.recipient_address == Address::zero() {
       return Err(errors::ERR_ZERO_RECIPIENT_ADDRESS.into());
   }
   ```

---

### Proof of Concept

**Via `ExitToEthereum` precompile (EVM path):**

An EVM caller constructs calldata for the precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`:
```
input = 0x00 || 0x0000000000000000000000000000000000000000
       (flag=ETH exit) (20-byte zero recipient)
``` [5](#0-4) 

The call succeeds. `recipient_address` is `Address::zero()`. A `withdraw` promise is dispatched to the ETH connector with the zero Ethereum address. The caller's ETH-on-Aurora is burned; the withdrawal proof targets `0x0000...0000` on Ethereum. Funds are permanently lost.

**Via NEAR-side `withdraw` (NEAR path):**

A NEAR account calls `withdraw` on the Aurora engine contract with Borsh-encoded `WithdrawCallArgs { recipient_address: Address::zero(), amount: X }` and 1 yoctoNEAR attached. The function passes validation, forwards to `engine_withdraw`, and the same permanent burn occurs. [6](#0-5)
