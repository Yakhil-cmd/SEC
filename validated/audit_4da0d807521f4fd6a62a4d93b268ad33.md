Audit Report

## Title
Fixed 65,000 Gas Limit for All ckERC20 Withdrawals Causes Permanent ckETH Loss on Out-of-Gas Failures — (`rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary
The ckETH minter applies a single hardcoded gas limit (`CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000`) to every ckERC20 withdrawal transaction regardless of which ERC20 token is being withdrawn. When a supported token's `transfer` call consumes more than 65,000 gas, the Ethereum transaction reverts out-of-gas, the ckERC20 tokens are reimbursed, but the ckETH fee burned upfront is permanently lost with no reimbursement path.

## Finding Description
In `rs/ethereum/cketh/minter/src/withdraw.rs` lines 43–44, two constants are defined:
```rust
pub const CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(21_000);
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);
```
`estimate_gas_limit` at lines 296–301 dispatches on withdrawal type and returns the same constant for every ckERC20 token with no per-token differentiation. `estimate_erc20_transaction_fee` in `rs/ethereum/cketh/minter/src/main.rs` lines 545–553 also hardcodes this limit when computing the ckETH fee to burn. The `withdraw_erc20` flow (lines 429–503 of `main.rs`) burns ckETH first, then burns ckERC20, then queues the Ethereum transaction. The minter documentation at `rs/ethereum/cketh/docs/ckerc20.adoc` line 275 explicitly states: "will reimburse the ckERC20 tokens in case the transaction failed. Overcharged transaction fees are not reimbursed." This confirms that on any Ethereum-level transaction failure — including out-of-gas — the ckETH fee is not returned. The supported token list at `ckerc20.adoc` lines 11–48 includes wstETH (stETH share accounting) and USDT (non-standard transfer with allowance-zeroing logic), both of which can consume gas at or above the 65,000 limit under realistic storage conditions (e.g., first transfer to a new address for USDT, or cold storage slots for wstETH).

## Impact Explanation
Any user withdrawing a ckERC20 token whose underlying `transfer` exceeds 65,000 gas suffers a permanent, unrecoverable loss of ckETH — a chain-key asset. The loss is bounded per transaction (65,000 × gas_price in ETH) but is structurally guaranteed to recur for every such withdrawal attempt. This constitutes concrete, permanent user-fund loss in the ckERC20 Chain Fusion system, matching the allowed High impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation
The exploit path is fully unprivileged: any IC principal can call `withdraw_erc20`. No attacker coordination is required — a normal user withdrawal of a borderline token is sufficient. USDT's transfer is known to cost 65,000–80,000 gas when writing to a previously-zero storage slot (first transfer to a recipient address), a common real-world condition. wstETH involves stETH share accounting across multiple storage slots. Both tokens are currently listed as supported. The user has no mechanism to query whether their specific withdrawal will succeed before committing the ckETH burn.

## Recommendation
1. **Per-token gas limits**: Store a configurable `gas_limit` field alongside each `CkErc20Token` entry in minter state, set at token-addition time via `add_ckerc20_token`, and use it in both `estimate_gas_limit` and `estimate_erc20_transaction_fee`.
2. **Reimburse ckETH on out-of-gas failure**: When the finalized transaction receipt shows status = 0 and `gas_used == gas_limit` (indicating out-of-gas), treat it as a minter-side configuration error and reimburse the ckETH fee to the user, analogous to the `TemporarilyUnavailable` reimbursement path already present for ckERC20 burn failures.
3. **Pre-flight gas estimation**: Before burning ckETH, perform an `eth_estimateGas` outcall for the specific token transfer to detect likely failures early and reject the withdrawal with no fee loss.

## Proof of Concept
1. User holds ckwstETH and sufficient ckETH on the IC.
2. User calls `icrc2_approve` on the ckETH ledger authorizing the minter for the fee amount.
3. User calls `icrc2_approve` on the ckwstETH ledger authorizing the minter for the withdrawal amount.
4. User calls `withdraw_erc20` on the minter with `ckerc20_ledger_id = wstETH_ledger_id`.
5. Minter calls `estimate_erc20_transaction_fee()` → computes fee using `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000`.
6. Minter burns user's ckETH for the estimated fee (confirmed at `main.rs` lines 448–458).
7. Minter burns user's ckwstETH (confirmed at `main.rs` lines 468–477).
8. Minter submits Ethereum transaction with `gas_limit = 65_000` calling `wstETH.transfer(destination, amount)`.
9. Ethereum transaction reverts out-of-gas when wstETH transfer exceeds 65,000 gas.
10. Minter reimburses ckwstETH to user (per `ckerc20.adoc` line 275).
11. ckETH fee is **not** reimbursed (per `ckerc20.adoc` line 275: "Overcharged transaction fees are not reimbursed") — user permanently loses ETH-denominated value.

A deterministic integration test can reproduce this by deploying a mock ERC20 contract whose `transfer` function consumes exactly 66,000 gas (e.g., via a loop writing to storage), adding it as a supported ckERC20 token on a local replica, and verifying that after a failed withdrawal the ckETH ledger balance is not restored while the ckERC20 balance is.
