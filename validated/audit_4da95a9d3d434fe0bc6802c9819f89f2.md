Audit Report

## Title
Silently Ignored ICP Burn Result in Cycles Minting Canister Allows ICP/Cycles Conservation Invariant Violation — (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister deliberately swallows ICP ledger burn errors after cycles have already been minted or deposited. If the ICP ledger returns a transient error during `burn_and_log`, cycles are irreversibly credited to the caller while the corresponding ICP remains unburned in the CMC's subaccount. The `blocks_notified` entry is then set to a terminal processed state, permanently preventing any retry of the burn, and leaving the ICP supply inflated relative to the cycles issued.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` (lines 2014–2049) explicitly discards the `CallResult<BlockIndex>` from the ICP ledger `send_pb` call:

```rust
match res {
    Ok(block) => print(format!("{msg} done in block {block}.")),
    Err((code, err)) => {
        let code = code as i32;
        print(format!("{msg} failed with code {code}: {err:?}"))
    }
}
```

The function returns `()` in both branches. All three callers — `process_top_up` (line 2001), `process_create_canister` (line 1945), and `process_mint_cycles` (line 1968) — invoke `burn_and_log` only after the service-side operation (cycle deposit, canister creation, or cycles ledger mint) has already succeeded, and they return `Ok(...)` regardless of whether the burn succeeded. The upstream notification handlers (`notify_top_up` at lines 1214–1222, `notify_create_canister` at lines 1411–1419, `notify_mint_cycles` at lines 1305–1313) then write a terminal `NotifiedTopUp`/`NotifiedCreateCanister`/`NotifiedMint` status into `blocks_notified`. The `is_transient_error` guard only removes the entry when the *process* function itself returns a transient error — since `process_top_up` returns `Ok(cycles)` even when the burn failed, the guard never fires for burn failures, and the notification is permanently finalized. There is no timer, heartbeat, or retry queue for failed burns anywhere in the codebase.

## Impact Explanation
This is a concrete violation of the ICP/cycles conservation invariant that underpins the economic security of the Internet Computer. Cycles are minted from ICP by burning the ICP; if the burn silently fails, cycles exist without the corresponding ICP having been destroyed. The unburned ICP is permanently orphaned in the CMC's subaccount (inaccessible to the user, not reclaimed by the protocol), and the ICP total supply is not reduced as the protocol requires. Repeated occurrences — each subnet upgrade cycle creates a window — accumulate unburned ICP, constituting illegal minting of cycles against an inflated ICP supply. This maps to the High impact class: significant NNS/ledger security impact with concrete protocol-level harm to the ICP/cycles accounting invariant.

## Likelihood Explanation
An unprivileged user submitting a standard `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` ingress message during any transient ICP ledger unavailability (subnet upgrade, replica restart, inter-canister call rejection) will trigger the silent failure path. No attacker capability beyond a normal ingress call is required. The user does not control the timing, but the window recurs predictably with every NNS subnet upgrade. The likelihood per individual transaction is low, but the condition is not rare across the full operational lifetime of the CMC.

## Recommendation
Separate the double-minting prevention concern from the burn-completion concern. The `Processing` guard already prevents double-minting regardless of whether the burn is retried. Two safe options:

1. **Retry via timer**: On burn failure, persist the pending `(subaccount, amount)` pair in stable state and retry asynchronously via a heartbeat or timer, consistent with how the CMC already handles other deferred operations.
2. **Do not finalize on burn failure**: Have `burn_and_log` return a `Result`, and only write the terminal `blocks_notified` entry when the burn succeeds. On burn failure, remove the `Processing` entry so the user can re-notify, triggering a fresh burn attempt without risk of double-minting (the cycles/canister already exist and the re-notification path would detect that).

## Proof of Concept
1. User transfers ICP to the CMC subaccount corresponding to their canister on the ICP ledger.
2. User calls `notify_top_up` with the block index.
3. CMC calls `fetch_transaction`, verifies the payment, sets `blocks_notified[block_index] = Processing`.
4. CMC calls `deposit_cycles` — succeeds; cycles are credited to the target canister.
5. CMC calls `burn_and_log(sub, amount).await` — the ICP ledger returns a transient `Err` (e.g., `SysTransient` during a subnet upgrade).
6. `burn_and_log` logs the error and returns `()`.
7. `process_top_up` returns `Ok(cycles)`.
8. CMC sets `blocks_notified[block_index] = NotifiedTopUp(Ok(cycles))`; `is_transient_error` does not fire.
9. Result: cycles credited, ICP not burned, notification permanently closed, ICP orphaned in CMC subaccount.

A deterministic integration test can reproduce this by injecting a mock ICP ledger that returns `RejectionCode::SysTransient` on the `send_pb` call, then asserting that (a) the target canister received cycles, (b) the CMC subaccount balance is unchanged, and (c) a subsequent `notify_top_up` with the same block index returns `NotifiedTopUp(Ok(...))` rather than allowing a retry.
