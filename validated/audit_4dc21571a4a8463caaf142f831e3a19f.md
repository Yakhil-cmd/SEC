Audit Report

## Title
`ensure_balance()` Uses Saturating-Subtracted `cycles_to_mint` for Rate-Limiter and Accounting, Enabling Pre-Balance Bypass - (File: rs/nns/cmc/src/main.rs)

## Summary
`ensure_balance()` computes `cycles_to_mint = cycles - current_balance` using saturating subtraction. When the CMC's live balance already meets or exceeds the requested `cycles` amount, `cycles_to_mint` silently becomes zero. Both the hourly minting rate limiter and the `total_cycles_minted` accounting counter are gated on this derived zero value rather than on the full distribution amount, allowing any canister to pre-load the CMC's balance and cause subsequent user conversions to bypass the rate cap entirely while leaving the canonical cycles-minted counter unchanged.

## Finding Description
`ensure_balance()` at `rs/nns/cmc/src/main.rs:2306-2325` reads the CMC's live balance and computes the amount to mint:

```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;   // saturating: yields 0 if balance >= cycles
```

`Cycles::sub` is explicitly saturating (`rs/types/cycles/src/cycles.rs:133-138`), so `cycles_to_mint` is `Cycles(0)` whenever `current_balance >= cycles`. Both downstream checks use this derived value:

```rust
limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;  // sees 0
state.total_cycles_minted += cycles_to_mint;                        // adds 0
```

`CyclesMintingLimiterSelector::check_and_add_cycles` (`rs/nns/cmc/src/main.rs:402-419`) passes `cycles_to_mint` directly to `base_limiter.check_and_add_cycles`, which enforces `count + cycles_to_mint > limit` (`rs/nns/cmc/src/limiter.rs:43`). With `cycles_to_mint == 0`, the check always passes and records nothing. The full `cycles` amount is still forwarded to the user via `call_with_payment128(..., u128::from(cycles))` at `rs/nns/cmc/src/main.rs:2120-2128`, drawing from the pre-loaded balance.

The exploit path: any canister calls the management canister's `deposit_cycles` endpoint targeting the CMC's canister ID, inflating its balance. A subsequent `notify_top_up` or `notify_create_canister` call triggers `process_top_up` → `deposit_cycles(canister_id, cycles, true, BaseLimit)` → `ensure_balance(cycles, BaseLimit)`, where `cycles_to_mint` saturates to zero, the limiter is bypassed, and `total_cycles_minted` is not incremented, while the user receives the full cycles amount.

No existing guard prevents external canisters from depositing into the CMC. The `deposit_cycles` internal function (`rs/nns/cmc/src/main.rs:2116-2117`) only calls `ensure_balance` when `mint_cycles == true`; there is no check on whether the CMC's balance was externally inflated.

## Impact Explanation
This matches **High ($2,000–$10,000): Significant NNS security impact with concrete user or protocol harm.** The CMC is a core NNS canister. The hourly minting cap (`base_cycles_limit`, default 150×10¹⁵ cycles) is a governance-set economic control. Bypassing it allows conversion of ICP to cycles at rates exceeding the governance-approved limit, undermining the economic rate-control mechanism. The `total_cycles_minted` counter exposed via the `total_cycles_minted` query (`rs/nns/cmc/src/main.rs:2327-2330`) and the `cmc_cycles_minted_total` Prometheus metric (`rs/nns/cmc/src/main.rs:2469-2473`) is used for governance and economic analysis; permanent undercount corrupts this canonical on-chain record. Both harms are concrete and irreversible per occurrence.

## Likelihood Explanation
Any canister on any subnet can call the management canister's `deposit_cycles` endpoint targeting the CMC's canister ID — no privileged role is required. The attacker permanently loses the deposited cycles (~$195,000 USD equivalent to pre-load a full 150×10¹⁵ cycle window), making casual exploitation expensive. However, a large ICP holder wishing to convert beyond the hourly cap, or an entity seeking to corrupt governance metrics, has a clear, permissionless, and repeatable path. The `total_cycles_minted` undercount is reachable even with small residual balances from normal CMC operation, requiring no deliberate attack.

## Recommendation
Decouple the rate-limiter and accounting update from `cycles_to_mint`. The limiter should be checked against the full `cycles` amount being distributed to the user, regardless of how much comes from pre-existing balance versus fresh minting. Only `total_cycles_minted` (tracking ICP-sourced minting) should remain gated on `cycles_to_mint`:

```rust
fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance; // saturating; may be 0

    with_state_mut(|state| {
        // Rate-limit against the full distribution amount, not just freshly minted
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        // Only count truly ICP-sourced minting in the accounting counter
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

## Proof of Concept
1. Deploy an attacker canister on any subnet. Call:
   ```
   ic00::deposit_cycles(CMC_CANISTER_ID, 150_000_000_000_000_000u128)
   ```
   This is a standard management canister call. The CMC's balance is now ≥ 150×10¹⁵ cycles.

2. Send 1,500 ICP to the CMC's subaccount and call `notify_top_up` (or `notify_create_canister`). At the current exchange rate this converts to ~150×10¹⁵ cycles, which would normally exhaust the hourly cap.

3. Inside `ensure_balance`:
   - `current_balance >= cycles` → `cycles_to_mint = Cycles(0)`
   - `base_limiter.check_and_add_cycles(now, Cycles(0), limit)` → passes, records 0
   - `total_cycles_minted += 0`
   - `ic0_mint_cycles128(0)` → no new cycles minted
   - `call_with_payment128(..., 150×10¹⁵)` → full cycles sent to user from pre-loaded balance

4. Verify: query `total_cycles_minted()` — unchanged. Query `base_limiter` state — records 0 for this window. User's canister balance increased by 150×10¹⁵ cycles. Repeat for subsequent hourly windows by re-loading the CMC balance.

A deterministic integration test using PocketIC can reproduce this by: (a) minting cycles into a test canister, (b) having it deposit into the CMC, (c) triggering `notify_top_up`, and (d) asserting that `total_cycles_minted` did not increment and the limiter counter remains at zero while the target canister received the full cycles amount.
