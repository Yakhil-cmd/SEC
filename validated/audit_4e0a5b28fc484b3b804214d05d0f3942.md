### Title
Single Byzantine Subnet Peer Can Exhaust Replica Heap via Unbounded Slot Table Growth for IDKG/DKG/Consensus/Certification Channels — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

---

### Summary

`SLOT_TABLE_NO_LIMIT = usize::MAX` is assigned to every artifact channel except ingress. A single Byzantine subnet peer (below the BFT fault threshold) can send an unbounded number of slot updates with distinct `slot_number` values and distinct valid artifact IDs, causing `slot_table`, `active_assembles`, and `artifact_processor_tasks` to grow without bound, exhausting heap memory and stalling consensus/IDKG progress.

---

### Finding Description

In `rs/replica/setup_ic_network/src/lib.rs`, two constants are defined: [1](#0-0) 

```rust
/// This limit is used to protect against a malicious peer advertising many ingress messages.
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
```

The comment explicitly acknowledges that `SLOT_TABLE_LIMIT_INGRESS` exists to protect against malicious peers. However, every other artifact channel — consensus, certification, DKG, IDKG, and HTTPS outcalls — is registered with `SLOT_TABLE_NO_LIMIT`: [2](#0-1) 

The slot limit is enforced in `handle_slot_update_receive` in `rs/p2p/consensus_manager/src/receiver.rs`: [3](#0-2) 

```rust
// Only insert slot update if we are below peer slot table limit.
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => {
    empty_slot.insert(new_slot_entry);
    (true, None)
}
Entry::Vacant(_) => {
    // dropped
    (false, None)
}
```

With `slot_limit = usize::MAX`, the guard `peer_slot_table_len < usize::MAX` is always true in practice. Every distinct `(slot_number, artifact_id)` pair from a peer is accepted unconditionally.

For each accepted new slot entry where the artifact ID is not already tracked, the receiver: [4](#0-3) 

1. Inserts into `active_assembles` (a `HashMap<WireArtifact::Id, watch::Sender<PeerCounter>>`)
2. Spawns a new task into `artifact_processor_tasks` (a `JoinSet`)

The `slot_number` is decoded directly from the peer-controlled protobuf field `slot_id` (a `u64`), so the attacker can supply any value: [5](#0-4) 

The `slot_table` is keyed per-peer by `SlotNumber`, so N distinct slot numbers from one peer create N entries. Each distinct artifact ID creates one entry in `active_assembles` and one spawned task.

Furthermore, when a spawned task completes (e.g., bouncer returns `Unwanted`), `handle_artifact_processor_joined` restarts it if the peer is still advertising: [6](#0-5) 

This means N tasks run continuously as long as the Byzantine peer keeps advertising, compounding CPU exhaustion alongside memory exhaustion.

---

### Impact Explanation

- **Memory exhaustion**: Each of N distinct slot updates allocates a `SlotEntry`, a `watch::Sender<PeerCounter>`, and a Tokio task. At scale (N in the millions), this exhausts the replica's heap.
- **Consensus/IDKG stall**: The Tokio runtime is saturated with N tasks waiting on `peer_rx.wait_for(|p| p.is_empty())`, starving legitimate consensus and IDKG processing tasks.
- **Sustained attack**: The Byzantine peer simply holds connections open and keeps advertising; tasks are never cleaned up until the peer disconnects or is removed from topology.

---

### Likelihood Explanation

The attacker must be a registered subnet node (authenticated via TLS/QUIC). This is a meaningful barrier — it requires a compromised or malicious node operator. However, the IC's BFT model explicitly tolerates up to `f < n/3` Byzantine nodes, and the invariant that a **single** below-threshold peer cannot cause unbounded resource consumption is violated here. The attack is:

- **Concrete**: The slot_number is a peer-controlled `u64`; artifact IDs for IDKG/DKG are complex structs but their protobuf encoding is known to any subnet participant.
- **Local-testable**: A state-machine test driving `ConsensusManagerReceiver` with `slot_limit = usize::MAX`, injecting N distinct slot updates from one peer, and asserting RSS growth proportional to N is straightforward (the test infrastructure already exists in `receiver.rs`).
- **No threshold corruption required**: A single Byzantine node suffices.

---

### Recommendation

Apply a per-peer slot table limit to all artifact channels, not just ingress. The appropriate limit should reflect the maximum number of in-flight artifacts a legitimate peer would ever advertise for each artifact type. For IDKG and DKG, this is bounded by protocol parameters (e.g., number of active transcripts × dealers). A reasonable cap (e.g., `10_000`–`100_000` per peer per channel) should be derived from protocol bounds and applied uniformly:

```rust
const SLOT_TABLE_LIMIT_IDKG: usize = 10_000;
const SLOT_TABLE_LIMIT_DKG: usize = 1_000;
const SLOT_TABLE_LIMIT_CONSENSUS: usize = 1_000;
const SLOT_TABLE_LIMIT_CERTIFICATION: usize = 1_000;
const SLOT_TABLE_LIMIT_HTTPS_OUTCALLS: usize = 1_000;
```

Additionally, consider adding a metric alert when any peer approaches its slot limit, to detect misbehaving nodes early.

---

### Proof of Concept

State-machine test (no network required):

```rust
#[tokio::test]
async fn byzantine_peer_exhausts_memory_via_idkg_slots() {
    let (mut mgr, _channels) = ReceiverManagerBuilder::new()
        // slot_limit = usize::MAX (the production default for IDKG)
        .with_slot_limit(usize::MAX)
        .build();
    let cancellation = CancellationToken::new();
    let n = 1_000_000usize;
    for i in 0..n {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(i as u64),
                commit_id: CommitId::from(1),
                update: Update::Id(i as u64), // distinct artifact ID per slot
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    // Assert unbounded growth:
    assert_eq!(mgr.slot_table.get(&NODE_1).unwrap().len(), n);
    assert_eq!(mgr.active_assembles.len(), n);
    assert_eq!(mgr.artifact_processor_tasks.len(), n);
    // RSS at this point is proportional to n — OOM on production replicas
}
```

The existing test `slot_table_limit_exceeded` at line 847 already demonstrates the correct behavior when a limit is set, confirming the guard works — it is simply not applied to IDKG/DKG/consensus/certification channels. [7](#0-6)
