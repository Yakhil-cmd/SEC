Audit Report

## Title
Single Principal Exhausts Global `retrieve_btc_accounts` Guard via Subaccount Enumeration in `retrieve_btc_with_approval` — (`rs/bitcoin/ckbtc/minter/src/guard.rs`, `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

`retrieve_btc_with_approval` keys its concurrency guard on the full ICRC-1 `Account` (owner + subaccount), while `retrieve_btc` keys on `(caller, None)`. Because `MAX_CONCURRENT = 100` and a single principal can supply 100 distinct `from_subaccount` values, one unprivileged attacker can occupy every slot in the shared `retrieve_btc_accounts` set — with no ckBTC balance or ICRC-2 approval required — blocking all ckBTC withdrawals for all users until the attacker's calls drain.

## Finding Description

**Guard implementation** (`guard.rs` L45–60): `Guard::new` inserts an `Account` into a `BTreeSet<Account>` and rejects with `TooManyConcurrentRequests` when `accounts.len() >= MAX_CONCURRENT` (100). [1](#0-0) 

Both endpoints share the same backing set via `RetrieveBtcUpdates::pending_requests` → `state.retrieve_btc_accounts`. [2](#0-1) 

**Asymmetry between endpoints:**

`retrieve_btc` always passes `subaccount: None`, so one principal can hold at most one slot: [3](#0-2) 

`retrieve_btc_with_approval` passes the caller-supplied `args.from_subaccount` directly into the guard key: [4](#0-3) 

Because `Account` equality is `(owner, subaccount)`, a single principal with 100 distinct subaccounts is treated as 100 independent accounts by the guard.

**Guard acquired before any balance or allowance check:** The guard is taken at line 263, but the `check_address` inter-canister await occurs at lines 283–307, and `burn_ckbtcs_icrc2` (which would fail with `InsufficientAllowance`) does not execute until line 314. The guard slot is held for the entire duration of the `check_address` round-trip with no funds verification having occurred. [5](#0-4) [6](#0-5) 

**Existing test confirms the mechanism:** `guard_prevents_more_than_max_concurrent_accounts` explicitly fills 50 slots with subaccounts of principal `0` and 50 slots with distinct principals, proving a single principal's subaccounts count against the global cap. [7](#0-6) 

## Impact Explanation

With all 100 slots occupied, every new caller receives `TemporarilyUnavailable("too many concurrent requests")` from both `retrieve_btc` and `retrieve_btc_with_approval`, since both share `retrieve_btc_accounts`. This constitutes a sustained, application-level DoS on ckBTC withdrawals — a financial integration explicitly in scope. This matches: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user and protocol harm** and **High — Significant ck-token security impact with concrete user harm**.

## Likelihood Explanation

- **No funds required**: guard is acquired before any balance or allowance check.
- **Single unprivileged principal sufficient**: 100 subaccounts from one principal exhaust the global limit.
- **Repeatable and sustained**: each `check_address` round-trip takes ~4 s on mainnet; the attacker re-submits as slots free, maintaining the DoS continuously.
- **No privileged access**: the endpoint is open to any ingress caller.
- **Negligible cost**: only IC ingress fees; calls that eventually fail at `InsufficientAllowance` cost only the ingress fee.

## Recommendation

Key the `retrieve_btc_with_approval` guard on `(caller, None)` to match `retrieve_btc`:

```rust
// Change in retrieve_btc_with_approval:
let _guard = retrieve_btc_guard(Account {
    owner: caller,
    subaccount: None,  // was: args.from_subaccount
})?;
```

Alternatively, add a pre-check that rejects any caller who already has any in-flight request regardless of subaccount before checking the global `MAX_CONCURRENT` limit. The `retrieve_btc` path already demonstrates the correct pattern. [3](#0-2) 

## Proof of Concept

The existing unit test infrastructure directly supports this PoC (no real IC needed):

```rust
// 1. Initialize minter state
init(init_args(), &IC_CANISTER_RUNTIME);

// 2. Attacker principal acquires all 100 slots via distinct subaccounts
let attacker_principal = test_principal(42);
let guards: Vec<_> = (0u8..100)
    .map(|i| retrieve_btc_guard(Account {
        owner: attacker_principal,
        subaccount: Some([i; 32]),
    }).expect("slot should be available"))
    .collect();
assert_eq!(guards.len(), 100);

// 3. Legitimate user is blocked
let victim = test_principal(999);
assert_eq!(
    retrieve_btc_guard(Account { owner: victim, subaccount: None }),
    Err(GuardError::TooManyConcurrentRequests)
);
```

This is structurally identical to the existing `guard_prevents_more_than_max_concurrent_accounts` test, which already proves the mechanism works exactly this way. [8](#0-7)

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }

        // is for a different principal

        init(init_args(), &IC_CANISTER_RUNTIME);

                })
            })
            .chain((MAX_CONCURRENT / 2..MAX_CONCURRENT).map(|id| {
                balance_update_guard(test_account(id as u64, None)).unwrap_or_else(|e| {
                    panic!("Could not create guard for principal num {id}: {e:#?}")
                })
            }))
            .collect();
        assert_eq!(guards.len(), MAX_CONCURRENT);
        let account = test_account(MAX_CONCURRENT as u64 + 1, None);
        let res = balance_update_guard(account).err();
