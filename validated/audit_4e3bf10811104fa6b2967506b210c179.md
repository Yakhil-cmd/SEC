### Title
Insufficient ICRC-2 Allowance in `approve_treasury_manager` Causes SNS Treasury Deposit Failure - (File: `rs/sns/governance/src/extensions.rs`)

### Summary

The `approve_treasury_manager` function in SNS Governance approves exactly the requested token amount (`sns_amount_e8s` / `icp_amount_e8s`) for the treasury manager canister. However, the ICRC-2 standard requires that the allowance cover **amount + ledger fee** for a `transfer_from` to succeed. Because the approved allowance equals only the transfer amount (without the fee), any subsequent `icrc2_transfer_from` call by the treasury manager will fail with `InsufficientAllowance`, making the entire treasury deposit operation inoperable.

### Finding Description

In `rs/sns/governance/src/extensions.rs`, `approve_treasury_manager` issues two `icrc2_approve` calls — one for the SNS token ledger and one for the ICP ledger — each approving exactly the requested deposit amount: [1](#0-0) 

The approved amounts are `sns_amount_e8s` and `icp_amount_e8s` respectively, with the ledger fee passed separately as the fee for the approval transaction itself (which is burned from the treasury balance, not added to the allowance).

The ICRC-2 ledger enforces that `allowance >= amount + fee` when processing a `transfer_from`. This is implemented in `rs/ledger_suite/icrc1/src/lib.rs`: [2](#0-1) 

And equivalently in the ICP ledger at `rs/ledger_suite/icp/src/lib.rs`: [3](#0-2) 

**Code flow:**

1. A governance proposal executes `execute_treasury_manager_deposit`, which calls `approve_treasury_manager(sns_amount_e8s, icp_amount_e8s)`. [4](#0-3) 

2. `approve_treasury_manager` sets allowance = `sns_amount_e8s` (and `icp_amount_e8s` for ICP). [5](#0-4) 

3. Governance then calls `deposit` on the treasury manager canister, passing the same amounts as `Allowance` records. [6](#0-5) 

4. The treasury manager calls `icrc2_transfer_from(from=governance_treasury, amount=sns_amount_e8s)`. The ledger requires `allowance >= sns_amount_e8s + fee`. Since `allowance == sns_amount_e8s`, the check fails and the call returns `InsufficientAllowance`.

The `Allowance` type passed to the treasury manager even includes `ledger_fee_decimals` in the asset descriptor, confirming the treasury manager is expected to account for fees when pulling funds: [7](#0-6) 

### Impact Explanation

Any SNS governance proposal that executes a `TreasuryManagerDeposit` operation will fail at the `deposit` call step after the allowance has already been set. The treasury manager cannot pull the approved funds because the allowance is short by exactly one ledger fee. The deposit is permanently broken for the full requested amount — the treasury manager receives nothing, and the governance canister's treasury retains the funds but has wasted the approval fee. This renders the core treasury deposit functionality of the SNS extension system inoperable.

### Likelihood Explanation

This is triggered by any SNS that has registered a treasury manager extension and submits a deposit proposal. The bug is in the unconditional code path of `execute_treasury_manager_deposit` — there is no configuration or edge case required. Every deposit proposal execution hits this bug. Likelihood is **high** for any SNS actively using the treasury manager feature.

### Recommendation

Approve `sns_amount_e8s + transaction_fee_e8s` (and `icp_amount_e8s + DEFAULT_TRANSFER_FEE` for ICP) so the treasury manager's `transfer_from` call has sufficient allowance to cover both the transfer amount and the ledger fee:

```rust
self.ledger
    .icrc2_approve(
        to,
        sns_amount_e8s + self.transaction_fee_e8s_or_panic(), // add fee
        Some(expiry_time_nsec),
        self.transaction_fee_e8s_or_panic(),
        self.sns_treasury_subaccount(),
        None,
    )
    .await?;

self.nns_ledger
    .icrc2_approve(
        to,
        icp_amount_e8s + icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(), // add fee
        Some(expiry_time_nsec),
        icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(),
        self.icp_treasury_subaccount(),
        None,
    )
    .await?;
```

### Proof of Concept

1. An SNS registers a treasury manager extension via a `RegisterExtension` proposal.
2. A governance proposal is submitted with `ExecuteExtensionOperation { operation_name: "deposit", treasury_allocation_sns_e8s: X, treasury_allocation_icp_e8s: Y }`.
3. The proposal passes and `execute_treasury_manager_deposit` runs.
4. `approve_treasury_manager` sets allowance = `X` on the SNS ledger and `Y` on the ICP ledger.
5. Governance calls `treasury_manager.deposit(DepositRequest { allowances: [{ amount: X, ... }, { amount: Y, ... }] })`.
6. The treasury manager calls `icrc2_transfer_from(from=governance_treasury, amount=X)`.
7. The SNS ledger checks: `X >= X + fee` → **false** → returns `InsufficientAllowance { allowance: X }`.
8. The deposit call fails. The treasury manager returns an error. The governance proposal execution fails at step 2 of `execute_treasury_manager_deposit`, after the allowance was already set (and the approval fee burned from the treasury).
