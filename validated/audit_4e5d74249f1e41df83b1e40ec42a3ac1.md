### Title
`native_per_pubdata` Truncates to Zero via Integer Division, Bypassing Pubdata Native-Resource Enforcement — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, the ratio `native_per_pubdata` is computed with a plain truncating integer division (`wrapping_div`). When `pubdata_price < native_price` — a valid operator configuration — the result is **0**. Every downstream pubdata-cost check then multiplies by 0, making pubdata effectively free for the transaction. This is the direct analog of the WooFi precision-loss bug: a division truncates to zero, and the zero value silently bypasses a resource-accounting guard.

---

### Finding Description

**Root cause — line 142 of `validation_impl.rs`:**

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is truncating integer division. Whenever `pubdata_price < native_price` (e.g., `pubdata_price = 50`, `native_price = 100`), the result is **0**. No error is returned; the zero silently propagates.

Compare with `native_per_gas`, which uses `div_ceil` to avoid the same truncation:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

The inconsistency means gas costs round up (never zero), but pubdata costs round down to zero.

**Downstream effects of `native_per_pubdata == 0`:**

1. **`create_resources_for_tx` (gas_helpers.rs line 352):**
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // → 0 * intrinsic_pubdata = 0
   ```
   No native is deducted from the transaction's native limit for intrinsic pubdata.

2. **`get_resources_to_charge_for_pubdata` (gas_helpers.rs lines 430–432):**
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)   // → 0
       .ok_or(out_of_native_resources!())?;
   ```
   Zero native resources are charged for all execution pubdata.

3. **`check_enough_resources_for_pubdata` (gas_helpers.rs line 454):**
   ```rust
   let enough = resources.has_enough(&resources_for_pubdata);
   // resources_for_pubdata is always 0 → always true
   ```
   The post-execution pubdata sufficiency check always passes, so the transaction is never reverted for insufficient native resources regardless of how much pubdata it produces.

---

### Impact Explanation

When `pubdata_price < native_price`, any unprivileged transaction sender can submit a transaction that writes to many storage slots (producing large pubdata) without being reverted for insufficient native resources. The transaction pays only for EVM gas and computational native, but zero native for pubdata. The operator/prover bears the full proving cost of the extra pubdata. This is a **resource accounting bug** that allows systematic underpayment for pubdata, enabling griefing of the operator or extraction of disproportionate state-change throughput per unit of fee paid.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a valid and realistic operator configuration: `native_price` reflects the cost of a single RISC-V proving cycle, while `pubdata_price` reflects the cost of publishing one byte to L1. These two quantities are independently set and can easily satisfy `pubdata_price < native_price` in normal operation. The default test configuration already demonstrates this: `pubdata_price = 0`, `native_price = 10`, giving `native_per_pubdata = 0` by design. Any transaction submitted under such a block context is affected.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, never silently zero when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Alternatively, add an explicit guard: if `pubdata_price > 0` but `native_per_pubdata == 0` after division, treat it as 1 (minimum charge), preventing pubdata from being silently free.

---

### Proof of Concept

**Setup:**
- `native_price = 100` (operator-set, e.g., 100 wei per RISC-V cycle)
- `pubdata_price = 50` (operator-set, e.g., 50 wei per pubdata byte)
- `native_per_pubdata = 50 / 100 = 0` (truncating division)

**Attack:**
1. Attacker submits a ZK transaction that writes to 500 distinct storage slots (producing ~16,000 bytes of pubdata).
2. `get_resources_to_charge_for_pubdata` computes `16000 * 0 = 0` native.
3. `check_enough_resources_for_pubdata` returns `has_enough = true`.
4. Transaction completes without native-resource revert.
5. Attacker paid only for EVM gas and computational native; zero native for 16,000 bytes of pubdata.
6. Operator/prover absorbs the full proving cost of the pubdata.
