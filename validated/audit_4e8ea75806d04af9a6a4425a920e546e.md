Audit Report

## Title
Metrics Endpoint Exposes Confidential Rule and Incident Counts to Anonymous Callers — (`rs/boundary_node/rate_limits/canister/metrics.rs`, `rs/boundary_node/rate_limits/canister/canister.rs`)

## Summary
The `/metrics` HTTP endpoint of the rate-limits canister is served by a non-replicated `#[query]` handler that bypasses `inspect_message` entirely, making it reachable by any anonymous caller. The endpoint recomputes and exposes `active_rules_count` and `stored_incidents_count` gauges by reading raw canister state with no filtering for confidentiality status, directly leaking the count of undisclosed rules and incidents to the public. This breaks the canister's explicit confidentiality invariant, which is designed to prevent unprivileged callers from observing the existence of undisclosed rate-limit rules.

## Finding Description
**Unauthenticated access path:**
`http_request` is a `#[query]` (non-replicated) method. The `inspect_message` hook in `canister.rs` only fires for ingress messages (replicated queries and updates); it never fires for non-replicated queries. The `/metrics` route inside `http_request` has no caller check of its own:

```rust
// canister.rs L188-192
fn http_request(request: HttpRequest) -> HttpResponse {
    match request.path() {
        "/metrics" => with_canister_state(|state| {
            with_metrics_registry(|registry| export_metrics_as_http_response(registry, state))
        }),
```

**Metrics recomputation reads raw state:**
`export_metrics_as_http_response` calls `recompute_metrics`, which sets gauges directly from `CanisterApi`:

```rust
// metrics.rs L154-157
cell.incidents_count.set(canister_api.incidents_count() as i64);
cell.active_rate_limit_rules_count.set(canister_api.active_rules_count() as i64);
```

**Both state reads are unfiltered:**
- `incidents_count()` returns `self.incidents.with(|cell| cell.borrow().len())` — the total length of the incidents map, including all undisclosed incidents (`state.rs` L139-141).
- `active_rules_count()` returns `value.rule_ids.len()` from the latest config entry, counting every rule regardless of `disclosed_at` (`state.rs` L143-149).

**Confidentiality formatters are irrelevant to this path:**
`ConfigConfidentialityFormatter` and `RuleConfidentialityFormatter` only redact `rule_raw` and `description` fields in `get_config`/`get_rule_by_id` responses (`confidentiality_formatting.rs` L14-28). They are never invoked on the metrics path and have no effect on the gauge values.

**The only gated path is `get_config`:**
`inspect_message` gates `get_config` (a replicated query sent via ingress) to `FullAccess` or `FullRead` callers only (`canister.rs` L48-55). There is no equivalent gate on `http_request`.

## Impact Explanation
This is a concrete information-disclosure vulnerability against the boundary node rate-limiting infrastructure. The rate-limits canister's confidentiality model explicitly exists to prevent attackers from learning that new rules have been deployed against them. An anonymous attacker polling `GET /metrics` can observe `active_rules_count`, `stored_incidents_count`, `active_config_version`, and `stored_configs_count` change in real time. When an operator deploys a confidential rule in response to an ongoing attack, all four gauges change simultaneously and are immediately visible. The attacker learns: (a) a new rule was deployed, (b) approximately when, and (c) the total rule count. This allows the attacker to detect that their traffic was flagged and adapt accordingly — directly undermining the purpose of the confidentiality system. This constitutes a significant boundary/API security impact with concrete harm to the rate-limiting infrastructure's effectiveness.

## Likelihood Explanation
Exploitation requires only an unauthenticated HTTP GET to a public canister endpoint — no credentials, no special access, no social engineering. The canister is deployed on IC mainnet and reachable at `https://<canister-id>.raw.ic0.app/metrics`. Any caller on the internet can poll this endpoint repeatedly at negligible cost. The attack is fully repeatable and deterministic.

## Recommendation
1. Replace `active_rules_count` with a count of only **disclosed** rules (those with `disclosed_at.is_some()`) in the active config.
2. Replace `stored_incidents_count` with a count of only **disclosed** incidents.
3. Either remove `active_config_version` and `stored_configs_count` from the public metrics or replace them with counters that only reflect disclosed state.
4. Alternatively, restrict the `/metrics` endpoint to callers with `FullAccess` or `FullRead` access level, consistent with how `get_config` is gated — though this would require adding an explicit caller check inside the `http_request` handler since `inspect_message` does not apply to non-replicated queries.

## Proof of Concept
State-machine or PocketIC test (no mainnet access required):

1. Initialize the canister with an authorized principal.
2. As the authorized principal, call `add_config` with one rule where `disclosed_at` is `None` (confidential rule).
3. As the anonymous principal, call `http_request { method: "GET", url: "/metrics", headers: [], body: vec![] }`.
4. Decode the UTF-8 response body and assert that the Prometheus text output contains `active_rules_count 1` and `stored_incidents_count 1`.
5. The assertion passes — confirming that the count of confidential, undisclosed rules is visible to anonymous callers via the metrics endpoint. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
