### Title
Unvalidated `SettlementLayerChainIdUpdated` Event Hook Bypasses Rollback, Causing Forward/Proving Divergence - (File: `system_hooks/src/event_hooks/system_context.rs`, `zk_ee/src/common_structs/new_settlement_layer_chain_id_storage.rs`)

---

### Summary

The `system_context_event_hook` intercepts `SettlementLayerChainIdUpdated` events without validating the emitting contract's address. Simultaneously, `NewSettlementLayerChainIdStorage` is intentionally excluded from the IO frame rollback snapshot. An unprivileged user can deploy a contract that emits the `SettlementLayerChainIdUpdated(uint256)` event with an attacker-controlled value. This sets `new_settlement_layer_chain_id_storage` to a value that does not correspond to any actual write to `SystemContext` storage slot 0. At block finalization in proving mode, `read_batch_context_inputs` asserts that the in-memory `new_settlement_layer_chain_id_storage` value matches the on-chain storage read — this assertion fails, causing a forward/proving divergence that prevents the batch from being proven.

---

### Finding Description

**Root Cause 1 — No caller validation in the event hook:**

`system_context_event_hook` is triggered by any emission of the `SettlementLayerChainIdUpdated(uint256)` event signature (`208daf0b...`), regardless of which contract emitted it. [1](#0-0) 

There is no check on the emitting contract's address. Any EVM contract can emit this event and trigger the hook.

**Root Cause 2 — `new_settlement_layer_chain_id_storage` excluded from IO frame rollback:**

`start_io_frame` snapshots `io`, `transient`, `messages`, `events`, and `interop_roots` — but **not** `new_settlement_layer_chain_id_storage`: [2](#0-1) 

`finish_io_frame` similarly omits rolling back `new_settlement_layer_chain_id_storage`: [3](#0-2) 

Although `NewSettlementLayerChainIdStorage` has `start_frame()` / `finish_frame()` methods, they are never called from the IO frame lifecycle: [4](#0-3) 

**Root Cause 3 — Assertion at batch finalization:**

`read_batch_context_inputs` (called in proving mode) asserts that if `new_settlement_layer_chain_id_storage` has a value, it must match the value read from `SystemContext` storage slot 0: [5](#0-4) 

If an attacker sets `new_settlement_layer_chain_id_storage` to a value that was never written to `SystemContext` storage, this `assert_eq!` panics in proving mode.

---

### Impact Explanation

An unprivileged L2 transaction sender can:

1. Deploy a contract that emits `SettlementLayerChainIdUpdated(uint256)` with an arbitrary `uint256` value (e.g., `U256::MAX`).
2. Call that contract in a regular L2 transaction. The event hook fires, calling `update_settlement_layer_chain_id(attacker_value)`, which writes to `new_settlement_layer_chain_id_storage`.
3. The transaction succeeds. The actual `SystemContext` storage slot 0 is **not** updated (only the operator's service transaction can do that).
4. At block finalization in proving mode, `read_settlement_layer_chain_id` reads the old value from storage, while `new_settlement_layer_chain_id_storage.value()` returns the attacker's value. The `assert_eq!` panics.

**Result**: The sequencer (forward mode) accepts the block, but the prover (proving mode) cannot prove it — a **forward/proving divergence**. This permanently stalls batch finalization for any block containing the attacker's transaction.

Additionally, the `update()` guard prevents a second update in the same block: [6](#0-5) 

So even a legitimate operator service transaction attempting to set the SL chain ID in the same block would fail with an internal error after the attacker's transaction has already poisoned the storage.

---

### Likelihood Explanation

- Any funded L2 account can deploy a contract and emit the event — no privileged access required.
- The event signature is public and fixed (`208daf0b...`).
- The attack requires a single successful L2 transaction.
- The block containing the attack transaction cannot be proven, permanently halting the chain's settlement until the batch is discarded or the bug is patched.

---

### Recommendation

1. **Add caller validation** in `system_context_event_hook`: only process the `SettlementLayerChainIdUpdated` event if it was emitted by the `SystemContext` system contract at address `0x800b`.

2. **Include `new_settlement_layer_chain_id_storage` in the IO frame snapshot**: add its snapshot handle to `FullIOStateSnapshot` and call `start_frame()`/`finish_frame()` from `start_io_frame()`/`finish_io_frame()`, mirroring how `interop_root_storage` is handled. [7](#0-6) [8](#0-7) 

---

### Proof of Concept

```
1. Deploy attacker contract on L2:
   contract Attacker {
       event SettlementLayerChainIdUpdated(uint256 indexed newChainId);
       function attack() external {
           emit SettlementLayerChainIdUpdated(type(uint256).max);
       }
   }

2. Submit regular L2 tx calling attacker.attack().
   - system_context_event_hook fires on the emitted event.
   - new_settlement_layer_chain_id_storage is set to U256::MAX.
   - SystemContext storage slot 0 is unchanged (still old SL chain ID).
   - Transaction succeeds.

3. Block finalization (proving mode):
   - read_settlement_layer_chain_id() returns old_sl_chain_id (from storage).
   - new_settlement_layer_chain_id_storage.value() returns Some(U256::MAX).
   - assert_eq!(U256::MAX, old_sl_chain_id) → PANIC.
   - Prover cannot generate a valid proof for this batch.
``` [9](#0-8) [10](#0-9)

fn new_sl_chain_id_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // Internal error if the data supplied isn't empty
    if !data.is_empty() {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad data").into(),
        );
    }
    // Same if there's a mismatch in expected topics
    if topics.len() != 2 {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad topics").into(),
        );
    }

    let new_sl_chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    system.io.update_settlement_layer_chain_id(
        ExecutionEnvironmentType::NoEE,
        resources,
        new_sl_chain_id,
    )?;

    Ok(())
