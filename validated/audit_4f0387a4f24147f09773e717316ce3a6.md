### Title
Asymmetric Rounding in `native_per_pubdata` Calculation Causes Systematic Pubdata Undercharge — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_gas` is computed with ceiling division (`div_ceil`) — conservative, protocol-favoring — while `native_per_pubdata` is computed with floor division (`wrapping_div`) — non-conservative, user-favoring. This asymmetry is the direct analog of the external report's `max` vs `min` issue: the protocol always presents the "best" (cheapest) pubdata rate rather than a conservative one, causing a systematic undercharge for every byte of pubdata a transaction writes.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the two key resource-pricing ratios are derived as follows:

```rust
// native_per_gas uses CEILING division — conservative
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...?;

// native_per_pubdata uses FLOOR division — non-conservative
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same asymmetry is replicated verbatim in the public API helper:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price   ← floor, no ceil
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

`native_per_pubdata` is then used in two places:

1. **Intrinsic pubdata pre-charge** at validation time (`create_resources_for_tx`): [3](#0-2) 
2. **Post-execution pubdata charge** (`get_resources_to_charge_for_pubdata`): [4](#0-3) 

Because `native_per_pubdata` is floored, the native resource consumed per pubdata byte is `floor(pubdata_price / native_price)` instead of the correct `ceil(pubdata_price / native_price)`. The `delta_gas` adjustment in `compute_gas_refund` then converts this underestimated native consumption back into gas charged:

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 { gas_used += delta_gas as u64; }
``` [5](#0-4) 

Since `native_used` is already underestimated (because `native_per_pubdata` is too low), `delta_gas` is also underestimated, and the user pays less in base tokens than the actual pubdata cost.

---

### Impact Explanation

Let `pubdata_price = P`, `native_price = N`, `pubdata_bytes = B`.

- Charged native per byte: `floor(P / N)`
- Actual cost per byte: `P / N`
- Undercharge per byte (in base tokens): `P mod N`
- Total undercharge per transaction: `B × (P mod N)` base tokens

**Concrete example**: `P = 1000`, `N = 7`, `B = 100,000` bytes of pubdata (realistic for a storage-heavy transaction):
- `native_per_pubdata = floor(1000/7) = 142` (should be `143`)
- Undercharge per byte = `1000 mod 7 = 6` base tokens
- Total undercharge = `600,000` base tokens per transaction

The protocol (sequencer/operator) systematically collects less than the actual L1 data-availability cost. Over many transactions this accumulates into a real deficit. The maximum undercharge per byte is `N − 1` base tokens, bounded by `max_pubdata_per_tx × (N − 1)`.

---

### Likelihood Explanation

**High.** Every L2 transaction that writes to storage generates pubdata. No special privileges are required. Any unprivileged user submitting a normal EIP-1559 or ZK-format transaction that touches storage slots triggers this path. The undercharge is deterministic and repeatable.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` to match the conservative rounding already applied to `native_per_gas`:

```rust
// Before (floor — non-conservative):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling — conservative, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [6](#0-5)

### Proof of Concept

**Setup**: Deploy a contract that writes to 20 distinct storage slots (generating ~640 bytes of pubdata). Set block context:
- `native_price = 7`
- `pubdata_price = 1000`
- `eip1559_basefee = 7` (so `gas_price = 7`, `native_per_gas = ceil(7/7) = 1`)

**Observed**:
- `native_per_pubdata = floor(1000/7) = 142`
- Native charged for 640 bytes = `640 × 142 = 90,880`
- Gas charged via delta = `90,880 / 1 = 90,880`
- User pays = `90,880 × 7 = 636,160` base tokens

**Expected** (with `div_ceil`):
- `native_per_pubdata = ceil(1000/7) = 143`
- Native charged = `640 × 143 = 91,520`
- User pays = `91,520 × 7 = 640,640` base tokens

**Shortfall per transaction** = `640,640 − 636,160 = 4,480` base tokens. Scaled to a block full of such transactions, the operator deficit is material.
