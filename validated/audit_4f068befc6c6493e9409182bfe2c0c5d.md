### Title
Cross-Chain Transaction Replay via Unconditional Chain ID Bypass for Pre-EIP-155 Legacy Transactions — (File: `engine/src/engine.rs`)

---

### Summary

The `submit_with_alt_modexp` function in `engine/src/engine.rs` skips chain ID validation entirely when a legacy (pre-EIP-155) transaction is submitted with `v = 27` or `v = 28`. Because `transaction.chain_id` is `None` in that case, the guard `if let Some(chain_id) = transaction.chain_id` never fires, and no chain-binding check is performed. An attacker who captures such a transaction signed on any other EVM chain can replay it verbatim on Aurora, spending the victim's Aurora funds without their consent.

---

### Finding Description

In `submit_with_alt_modexp`, the only replay-domain check is:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

The guard is a short-circuit `if let Some(...)`. When `transaction.chain_id` is `None`, the entire block is skipped and execution continues to the nonce check.

`NormalizedEthTransaction.chain_id` is populated from `LegacyEthSignedTransaction::chain_id()`:

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [2](#0-1) 

For `v = 27` or `v = 28` (pre-EIP-155 encoding), this returns `None`. The `TryFrom<EthTransactionKind>` conversion propagates that `None` directly into `NormalizedEthTransaction`:

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for pre-EIP-155
    ...
``` [3](#0-2) 

The subsequent nonce check at line 1063 prevents replay **within Aurora** but does nothing to prevent replay of a transaction that was originally signed for a different chain. [4](#0-3) 

**Analog to the external report.** The external report's root cause is that a security check (nonce uniqueness) is silently absent when a feature flag (`redis_enabled`) is `False`, and the system continues operating without error. Here, the security check (chain-domain binding) is silently absent when the transaction carries no chain ID, and the engine continues executing without error. In both cases a constant or absent value replaces a proper discriminator, and the signed artifact becomes replayable across contexts.

---

### Impact Explanation

An attacker who captures a victim's pre-EIP-155 signed transaction from any EVM chain (Ethereum mainnet, a testnet, another L2) can submit those exact bytes to Aurora's `submit` endpoint. If the victim's Aurora account nonce matches the transaction nonce and the victim holds ETH or ERC-20 tokens on Aurora, the transaction executes and transfers those assets to the `to` address encoded in the original transaction — which the attacker may control or which may be meaningless on Aurora. This constitutes **direct theft of user funds** (Critical).

---

### Likelihood Explanation

Pre-EIP-155 transactions are still produced by:
- Hardware wallets operating in legacy compatibility mode.
- Older tooling and custom scripts that omit chain ID.
- The EIP-1820 deployment flow, which Aurora explicitly re-enabled for compatibility (documented in `CHANGES.md`).

A victim who uses any such path and also holds funds on Aurora at the same address is exposed. The nonce must match, which is trivially true for fresh accounts (nonce 0) and possible at any point where both chains share the same nonce value for the sender. Likelihood is **Medium**.

---

### Recommendation

- **Short term**: Reject legacy transactions with `v ∈ {27, 28}` (no chain ID) at the `submit` entry point with a clear error, unless a specific allow-list of known-safe use cases (e.g., EIP-1820 deployer address) is matched.
- **Long term**: Require EIP-155 for all user-submitted transactions. If pre-EIP-155 support must be retained, document it explicitly as an unsafe configuration and gate it behind an operator-controlled flag that defaults to disabled in production.

---

### Proof of Concept

1. Victim signs a legacy transaction on Ethereum mainnet (`v = 27`) transferring ETH to address `X` (attacker-controlled on Aurora). Nonce is `N`.
2. Victim's Aurora account also has nonce `N` and holds ETH.
3. Attacker submits the identical RLP-encoded bytes to Aurora's `submit` endpoint.
4. `submit_with_alt_modexp` parses the transaction; `chain_id()` returns `None`; the `if let Some(chain_id)` guard at lines 1054–1059 is not entered — no chain ID rejection occurs.
5. `check_nonce` at line 1063 passes because the Aurora nonce is `N`.
6. The EVM executes the transfer; the victim's Aurora ETH moves to `X`.
7. The victim's Ethereum transaction is unaffected; only Aurora funds are stolen. [5](#0-4) [6](#0-5) [7](#0-6)

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);
        let message_hash = sdk::keccak(rlp_stream.as_raw());
        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
            .map_err(|_| Error::EcRecover)
    }

    /// Returns chain id encoded in `v` parameter of the signature if that was done, otherwise None.
