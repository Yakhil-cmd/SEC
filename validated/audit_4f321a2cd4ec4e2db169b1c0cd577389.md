Audit Report

## Title
Unenforced `ExecuteGenericNervousSystemFunction` Payload Size Limit Enables Heap Exhaustion DoS in SNS Governance - (File: `rs/sns/governance/src/types.rs`)

## Summary
The SNS governance canister declares a 70 KB payload size cap (`PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX`) for `ExecuteGenericNervousSystemFunction` proposals but marks it `#[allow(dead_code)]` and never enforces it. Any SNS neuron holder can submit proposals with payloads up to ~2 MB (the ingress limit), which are stored verbatim on the governance canister's heap. With up to 700 active proposals, this can push heap usage toward the 3.5 GiB soft limit, causing `check_heap_can_grow` to reject all further governance activity.

## Finding Description
The constant is declared but never referenced anywhere in the codebase:

```rust
// rs/sns/governance/src/types.rs:72-75
#[allow(dead_code)]
/// TODO Use to validate the size of the payload 70 KB (for executing
/// SNS functions that are not canister upgrades)
const PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX: usize = 70000;
```

`validate_and_render_execute_nervous_system_function` (`rs/sns/governance/src/proposal.rs:1433`) performs no check on `execute.payload.len()` before calling the external validator canister. The full proposal — including the raw payload bytes — is then stored in `ProposalData.proposal` on the heap (`rs/sns/governance/src/governance.rs:3611`). `get_proposal` returns the full `ProposalData` with no truncation, while `list_proposals` does apply a 1 KB cap via `EXECUTE_NERVOUS_SYSTEM_FUNCTION_PAYLOAD_LISTING_BYTES_MAX = 1000` (`rs/sns/governance/src/governance.rs:158`). The heap soft limit is 3.5 GiB (7/8 of 4 GiB, `rs/sns/governance/src/governance.rs:168-169`), and `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 700` (`rs/sns/governance/src/proposal.rs:79`). The mem-test canister itself estimates `ExecuteGenericNervousSystemFunction` payloads at 1 MB on average (`rs/sns/integration_tests/test_canisters/sns_governance_mem_test_canister.rs:280`), confirming real-world scale.

## Impact Explanation
An SNS token holder can submit up to 700 active proposals each carrying a ~2 MB payload, consuming ~1.4 GB of heap. Once the heap approaches the 3.5 GiB soft limit, `check_heap_can_grow` rejects all new proposals, blocking all SNS governance operations. Additionally, each `get_proposal` call serializes and returns the full oversized payload, consuming cycles proportional to response size (~28× the intended cap). This maps to **High** — application/platform-level DoS with concrete SNS governance availability impact.

## Likelihood Explanation
The attacker only needs a valid SNS neuron with sufficient dissolve delay to submit proposals. No privileged role, admin key, or majority corruption is required. The `reject_cost_e8s` fee is the primary economic barrier, but in many SNS deployments the minimum stake and fee are low enough to make submitting dozens of oversized proposals feasible. The `#[allow(dead_code)]` annotation and `TODO` comment confirm this is an acknowledged gap, not an intentional design choice.

## Recommendation
Enforce the limit inside `validate_and_render_execute_nervous_system_function` before the validator canister is called:

```rust
if execute.payload.len() > PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX {
    return Err(format!(
        "ExecuteGenericNervousSystemFunction payload is too large: {} bytes (max {} bytes)",
        execute.payload.len(),
        PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX,
    ));
}
```

Remove the `#[allow(dead_code)]` attribute and the `TODO` comment. Additionally, consider enforcing a size limit on `payload_text_rendering` returned by the validator canister before storing it in `ProposalData`.

## Proof of Concept
1. Deploy an SNS with a `GenericNervousSystemFunction` whose validator canister accepts any payload.
2. Stake a neuron meeting the minimum dissolve delay.
3. Submit a proposal: `ExecuteGenericNervousSystemFunction { function_id: <id>, payload: vec![0u8; 2_000_000] }`.
4. Observe `make_proposal` succeeds without error — no payload size check is performed.
5. Call `get_proposal` on the resulting proposal ID; observe the full 2 MB payload is returned.
6. Repeat up to 700 times; observe heap growing ~2 MB per proposal, eventually causing `check_heap_can_grow` to reject further proposals and block all SNS governance activity.
