### Title
Pre-EIP-155 Legacy Transactions Accepted Without Chain-ID Binding, Enabling Cross-Chain Replay — (`engine/src/engine.rs`)

### Summary
Aurora Engine's `submit` entrypoint accepts legacy (pre-EIP-155) Ethereum transactions whose `v` value is `27` or `28`. These transactions carry no chain ID in their signed payload. The engine's chain-ID guard is gated on `if let Some(chain_id) = transaction.chain_id`, so it is silently skipped for such transactions. Any pre-EIP-155 transaction broadcast on Ethereum mainnet (or any other EVM chain) can be replayed verbatim on Aurora to execute the same state change against the signer's Aurora account, and vice versa.

### Finding Description

**Root cause — `engine-transactions/src/legacy.rs`**

`LegacyEthSignedTransaction::chain_id()` returns `None` whenever `v ∈ {27, 28}`: [1](#0-0) 

`sender()` reconstructs the signing hash without a chain ID in that case, so the recovered address is valid for any chain: [2](#0-1) 

**Root cause — `engine-transactions/src/lib.rs`**

When a `Legacy` transaction is normalised, `chain_id` is propagated as-is (i.e. `None`): [3](#0-2) 

**Root cause — `engine/src/engine.rs`**

The only chain-ID guard in `submit_with_alt_modexp` is:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [4](#0-3) 

When `transaction.chain_id` is `None` the entire block is skipped. No other check in the function enforces chain binding for legacy transactions.

**Intentional design, confirmed in changelog**

The project explicitly re-enabled pre-EIP-155 transactions for EIP-1820 compatibility:

> "Original ETH transactions which do not contain a Chain ID are allowed again to allow for use of EIP-1820" [5](#0-4) 

This design decision is the direct root cause of the replay surface.

### Impact Explanation

An attacker who observes a pre-EIP-155 transaction on Ethereum mainnet (or any other EVM network) can submit the identical raw bytes to Aurora Engine's `submit` endpoint. If the victim's Aurora nonce equals the nonce in the replayed transaction, the engine will:

1. Recover the correct sender address (signature is chain-agnostic).
2. Pass the chain-ID guard (guard is skipped for `None`).
3. Pass the nonce check (nonces match by assumption).
4. Execute the transaction — transferring ETH/ERC-20 tokens or invoking arbitrary contract logic on behalf of the victim.

The reverse direction is equally valid: a pre-EIP-155 Aurora transaction can be replayed on Ethereum mainnet or any other EVM chain that still accepts such transactions.

**Impact class:** Critical — direct theft of user funds at rest.

### Likelihood Explanation

- Pre-EIP-155 transactions are still produced by: EIP-1820 registry deployments (the project's own stated reason for the allowance), older hardware-wallet firmware, certain DeFi tooling, and any user who explicitly sets `v = 27/28`.
- Nonce alignment is required but is not a strong barrier: accounts with low nonce values (0, 1, 2 …) are common, and an attacker can wait until the victim's Aurora nonce reaches the target value before replaying.
- The attack requires no privileged access — only the ability to observe public mempool/chain data and call the public `submit` NEAR method.

### Recommendation

Reject legacy transactions that carry no chain ID at the `submit` entrypoint. Replace the conditional guard with an unconditional one:

```rust
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

If EIP-1820 compatibility must be preserved, handle it as a special-cased, admin-gated one-time deployment path rather than a permanent open door for all pre-EIP-155 transactions.

### Proof of Concept

1. On Ethereum mainnet, victim signs and broadcasts a pre-EIP-155 ETH transfer (`v = 27`):
   - `nonce = 3, to = attacker_address, value = 1 ETH`
2. Victim also holds 5 ETH on Aurora at the same address, and their Aurora nonce happens to be `3`.
3. Attacker copies the raw RLP-encoded transaction bytes from the Ethereum mempool/chain.
4. Attacker calls Aurora Engine's `submit` NEAR method with those bytes as input.
5. `LegacyEthSignedTransaction::chain_id()` returns `None` → chain-ID guard at line 1055 is skipped.
6. `check_nonce` passes (Aurora nonce = 3 = transaction nonce).
7. Aurora Engine executes the transfer, sending 1 ETH from the victim's Aurora account to the attacker. [6](#0-5) [7](#0-6) [3](#0-2)
