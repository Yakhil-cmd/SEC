### Title
Vote Overwriting in Root Proposal Allows Node Operator to Flip Previously Cast Ballot - (File: `rs/nns/handlers/root/impl/src/root_proposals.rs`)

### Summary
The `vote_on_root_proposal_to_upgrade_governance_canister` function in the NNS root canister unconditionally overwrites a node operator's ballot on every call, with no guard preventing a previously cast `No` vote from being silently changed to `Yes`. This undermines the Byzantine fault-tolerance guarantees of the root proposal system and is a direct analog to the reported oracle-quorum vulnerability: just as a compromised oracle's report remains counted after role modification, a compromised node-operator key can retroactively flip a blocking `No` vote into a `Yes`, reducing the effective resistance of the quorum.

### Finding Description
`GovernanceUpgradeRootProposal` is the only mechanism to upgrade the NNS governance canister outside of normal NNS governance. It requires a Byzantine majority of NNS node operators (`N - f` yes-votes, where `f = (N-1)/3`). [1](#0-0) 

When a node operator calls `vote_on_root_proposal_to_upgrade_governance_canister`, the voting logic iterates over `node_operator_ballots` and **unconditionally overwrites** the ballot for the matching principal, regardless of whether a ballot has already been cast: [2](#0-1) 

There is no check equivalent to `if ballot == Undecided { ... }` before the assignment `*b = ballot.clone()`. A node operator who previously voted `No` can call the function again with `RootProposalBallot::Yes` and their ballot is silently overwritten. The function only checks:

1. Proposal is not expired. [3](#0-2) 
2. Registry version has not changed since submission. [4](#0-3) 
3. The caller is in the original ballot list. [5](#0-4) 

None of these checks prevent re-voting.

### Impact Explanation
The root proposal system is explicitly designed to tolerate up to `f = (N-1)/3` Byzantine faults. For the NNS subnet with N=7 nodes, `f=2` and 5 yes-votes are required. If a node operator can change their vote, the effective Byzantine threshold is broken:

- A node operator votes `No` to block a malicious governance upgrade.
- An attacker who has compromised that node operator's key (or the node operator themselves, acting maliciously) calls `vote_on_root_proposal_to_upgrade_governance_canister` again with `Yes`.
- The `No` ballot is silently overwritten to `Yes`.
- The `is_byzantine_majority_yes` tally now counts the flipped vote, potentially pushing the proposal over the threshold. [6](#0-5) 

The consequence is that a malicious governance upgrade of the NNS governance canister — the most security-critical canister on the Internet Computer — could be executed with fewer legitimate yes-votes than the Byzantine model requires. This is a **governance authorization bug** with critical impact: arbitrary code execution in the governance canister.

### Likelihood Explanation
The NNS subnet has a small, fixed set of node operators. A single compromised node-operator key is sufficient to flip one ballot. Key compromise of a node operator is a realistic threat (phishing, supply-chain attack on operator infrastructure, insider threat). The 7-day voting window (`MAX_TIME_FOR_GOVERNANCE_UPGRADE_ROOT_PROPOSAL`) gives an attacker ample time to observe a `No` vote and then flip it. [7](#0-6) 

### Recommendation
Add a guard in `vote_on_root_proposal_to_upgrade_governance_canister` that rejects a vote if the caller's ballot is already in a non-`Undecided` state:

```rust
for (p, b) in &mut proposal.node_operator_ballots {
    if p == &caller {
        // Prevent vote overwriting
        if !matches!(b, RootProposalBallot::Undecided) {
            return Err(format!(
                "{}Caller: {} has already cast a ballot.",
                LOG_PREFIX, caller
            ));
        }
        *b = ballot.clone();
        voted_on += 1;
    }
}
```

This mirrors the protection already present in the NNS governance neuron voting path, which explicitly rejects a second vote: [8](#0-7) 

### Proof of Concept
Assume N=7 NNS nodes, f=2, threshold=5. Node operators are P1…P7. P1 is the proposer (auto-votes Yes).

1. P1 submits a malicious root proposal. Tally: 1 Yes, 0 No, 6 Undecided.
2. P2, P3, P4 vote `No` to block. Tally: 1 Yes, 3 No, 3 Undecided.
3. Attacker (with P2's compromised key) calls `vote_on_root_proposal_to_upgrade_governance_canister(proposer=P1, ballot=Yes)` as P2. Tally: 2 Yes, 2 No, 3 Undecided.
4. Attacker (with P3's compromised key) flips P3. Tally: 3 Yes, 1 No, 3 Undecided.
5. P5, P6 vote `Yes` (legitimately). Tally: 5 Yes, 1 No, 1 Undecided.
6. `is_byzantine_majority_yes()` returns `true` (5 >= 7-2=5). The malicious governance upgrade executes. [9](#0-8) 

Without the vote-overwriting bug, the attacker would need 5 legitimate yes-votes. With it, they only need to compromise enough `No`-voters to flip the tally — a significantly lower bar.

impl GovernanceUpgradeRootProposal {
    /// For a root proposal to have a byzantine majority of yes, it
    /// needs to collect N - f ""yes"" votes, where N is the total number
    /// of nodes (same as the number of ballots) and f = (N - 1) / 3.
    fn is_byzantine_majority_yes(&self) -> bool {
        let num_nodes = self.node_operator_ballots.len();
        let max_faults = (num_nodes - 1) / 3;
        let votes_yes: usize = self
            .node_operator_ballots
            .iter()
            .map(|(_, b)| match b {
                RootProposalBallot::Yes => 1,
                _ => 0,
            })
            .sum();
        votes_yes >= (num_nodes - max_faults)
    }
