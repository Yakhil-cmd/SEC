### Title
Unenforced `ExecuteGenericNervousSystemFunction` Payload Size Limit Enables Heap Exhaustion and Cycles DoS in SNS Governance - (File: `rs/sns/governance/src/types.rs`)

### Summary
The SNS governance canister defines a 70 KB maximum payload size constant (`PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX`) for `ExecuteGenericNervousSystemFunction` proposals, but this constant is explicitly marked `#[allow(dead_code)]` with a `TODO` comment and is never referenced in any validation path. An SNS token holder can therefore submit proposals with payloads up to ~2 MB (the ingress message size limit), 28× the intended cap. These oversized payloads are stored verbatim in the governance canister's heap and returned in full by `get_proposal`, causing excessive cycles consumption per query and, across many proposals, progressive heap exhaustion that can degrade or block SNS governance operations.

### Finding Description

In `rs/sns/governance/src/types.rs`, the intended payload size limit is declared but never enforced:

```rust
#[allow(dead_code)]
/// TODO Use to validate the size of the payload 70 KB (for executing
/// SNS functions that are not canister upgrades)
const PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX: usize = 70000;
```

A grep across the entire repository confirms this constant appears exactly once and is never read. [1](#0-0) 

The proposal submission path in `make_proposal` calls `validate_and_render_proposal`, which in turn calls `validate_and_render_action`. For `ExecuteGenericNervousSystemFunction`, this dispatches to `validate_and_render_execute_nervous_system_function`, which calls the external validator canister but performs **no check on `execute.payload.len()`**. [2](#0-1) 

The full proposal — including the raw payload bytes — is then stored in `ProposalData.proposal` on the governance canister's heap: [3](#0-2) 

When `get_proposal` is called (a query method), the full `ProposalData` including the untruncated payload is returned to the caller with no size reduction: [4](#0-3) 

By contrast, `list_proposals` does truncate the payload to 1 KB (`EXECUTE_NERVOUS_SYSTEM_FUNCTION_PAYLOAD_LISTING_BYTES_MAX = 1000`), but `get_proposal` applies no such limit. [5](#0-4) 

The memory test canister itself acknowledges that `ExecuteGenericNervousSystemFunction` payloads are expected to be ~1 MB in practice, confirming the real-world scale of the problem: [6](#0-5) 

The maximum number of proposals with active ballots is capped at 700 (`MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS`). At 2 MB per payload, 700 such proposals consume ~1.4 GB of heap, against a soft heap limit of ~3.5 GB (7/8 of 4 GB). [7](#0-6) 

### Impact Explanation

1. **Heap exhaustion / governance DoS**: An attacker submitting many `ExecuteGenericNervousSystemFunction` proposals with ~2 MB payloads can push the SNS governance canister's heap toward its soft limit. Once the heap is saturated, `check_heap_can_grow` will reject new proposals, blocking all governance activity until proposals are purged.

2. **Cycles exhaustion on `get_proposal`**: Each `get_proposal` call must serialize and return the full `ProposalData`, including the oversized payload. A 2 MB response costs ~28× more cycles than the intended 70 KB cap, making repeated queries to the governance canister disproportionately expensive for callers and for the canister itself.

3. **Proposal execution amplification**: When an adopted proposal is executed, the full payload is forwarded to the target canister via inter-canister call. An oversized payload consumes additional cycles on both the governance canister and the target.

### Likelihood Explanation

The attacker must hold SNS tokens and stake a neuron with sufficient dissolve delay to submit proposals. The proposal submission fee (`reject_cost_e8s`) is the primary economic barrier. However, in many SNS deployments the minimum stake and fee are low enough that a determined attacker can submit dozens of oversized proposals at modest cost. The attack does not require any privileged role, admin key, or majority corruption — only a valid SNS neuron. The `#[allow(dead_code)]` annotation and the `TODO` comment confirm this is an acknowledged gap, not an intentional design choice.

### Recommendation

Enforce `PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX` inside `validate_and_render_execute_nervous_system_function` before the validator canister is called:

```rust
if execute.payload.len() > PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX {
    return Err(format!(
        "ExecuteGenericNervousSystemFunction payload is too large: {} bytes \
         (max {} bytes)",
        execute.payload.len(),
        PROPOSAL_EXECUTE_SNS_FUNCTION_PAYLOAD_BYTES_MAX,
    ));
}
```

Remove the `#[allow(dead_code)]` attribute and the `TODO` comment once the check is in place. Additionally, consider enforcing a size limit on `payload_text_rendering` returned by the validator canister before storing it in `ProposalData`.

### Proof of Concept

1. Deploy an SNS with a `GenericNervousSystemFunction` whose validator canister accepts any payload.
2. Stake a neuron meeting the minimum dissolve delay.
3. Submit a proposal:
   ```
   ExecuteGenericNervousSystemFunction {
       function_id: <registered_id>,
       payload: vec![0u8; 2_000_000],  // 2 MB — 28× the intended 70 KB limit
   }
   ```
4. Observe the proposal is accepted by `make_proposal` without error.
5. Call `get_proposal` on the resulting proposal ID; observe the full 2 MB payload is returned, consuming cycles proportional to the response size.
6. Repeat step 3 up to 700 times (the ballot-retention limit); observe the governance canister's heap growing by ~2 MB per proposal, approaching the 3.5 GB soft limit and eventually causing `check_heap_can_grow` to reject further proposals. [1](#0-0) [8](#0-7) [9](#0-8) [5](#0-4) [4](#0-3)

pub async fn validate_and_render_execute_nervous_system_function(
    env: &dyn Environment,
    execute: &ExecuteGenericNervousSystemFunction,
    existing_functions: &BTreeMap<u64, NervousSystemFunction>,
) -> Result<String, String> {
    let id = execute.function_id;
    match existing_functions.get(&execute.function_id) {
        None => Err(format!("There is no NervousSystemFunction with id: {id}")),
        Some(function) => {
            // Make sure this isn't a NervousSystemFunction which has been deleted.
            if function == &*NERVOUS_SYSTEM_FUNCTION_DELETION_MARKER {
                Err(format!("There is no NervousSystemFunction with id: {id}"))
            } else {
                // To validate the proposal we try and call the validation method,
                // which should produce a payload rendering if the proposal is valid
                // or an error if it isn't.
                let rendering =
                    perform_execute_generic_nervous_system_function_validate_and_render_call(
                        env,
                        function.clone(),
                        execute.clone(),
                    )
                    .await?;

                let payload_hash = {
                    let mut state = Sha256::new();
                    state.write(execute.payload.as_slice());
                    let sha = state.finish();
                    hex::encode(sha)
                };

                Ok(format!(
                    r"# Proposal to execute nervous system function:

## Nervous system function:

{function:#?}

## Payload sha256: 

{payload_hash}

## Payload:

{rendering}"
                ))
            }
        }
    }
}
