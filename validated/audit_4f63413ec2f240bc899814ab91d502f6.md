### Title
Missing Factory Deps Processing in L1→L2 Transactions Allows Deployment Transactions to Silently Fail Without Proper Bytecode Registration - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

A `TODO` comment in the L1 transaction processing path explicitly marks that factory dependency handling (marking factory deps) is **not implemented**. L1→L2 transactions that include `factory_deps` (bytecode hashes for contracts to be deployed) will have those deps silently ignored. This means an L1-initiated deployment transaction can execute, appear to succeed on-chain, but the required bytecode is never registered in the system — leaving the deployed contract address with no executable code, or causing the deployment to revert unexpectedly when the bytecode is needed.

---

### Finding Description

In `process_l1_transaction`, immediately after the deposited-amount check and before transaction execution, there is an explicit unimplemented stub:

```rust
// TODO: l1 transaction preparation (marking factory deps)
``` [1](#0-0) 

The `AbiEncodedTransaction` struct carries a `factory_deps` field — a list of `bytes32` hashes of bytecodes that must be published and registered before the transaction executes: [2](#0-1) 

The `validate_structure` function for `AbiEncodedTransaction` parses and validates the `factory_deps` field but performs no semantic check on its contents: [3](#0-2) 

The `reserved_dynamic` field (which carries access/authorization lists for L2 txs) is also rejected as non-empty for L1 txs, but `factory_deps` is accepted without being acted upon: [4](#0-3) 

The execution path then calls `run_single_interaction` directly, skipping any factory-dep registration: [5](#0-4) 

Additionally, the same file contains a second TODO noting that **deployment transactions** (CREATE) are not supported for L1 txs at all: [6](#0-5) 

---

### Impact Explanation

**State-transition bug / valid-execution unprovability:**

1. An L1→L2 transaction that includes `factory_deps` hashes (e.g., a deployment transaction that requires a specific bytecode to be pre-registered) will have those deps silently dropped. The transaction executes as if no factory deps were provided.

2. If the transaction body attempts to deploy a contract whose bytecode hash was supposed to be registered via `factory_deps`, the deployment will fail at runtime (bytecode not found), causing the L1 tx to revert. Since L1 txs **cannot be invalidated** (reverting halts the chain or causes incorrect accounting), this is a liveness/correctness risk.

3. More critically: the forward-running system (sequencer) and the proving system must agree on state transitions. If the sequencer silently ignores factory deps but the L1 contract on Ethereum expects them to be processed, the resulting state root will diverge from what L1 expects — a **forward/proving divergence** and **state-transition bug**.

4. Funds deposited via the L1 tx (`total_deposited`) are still minted and transferred even when the execution fails due to missing bytecode, meaning ETH is locked in a contract address with no code.

---

### Likelihood Explanation

**Medium-High.** Any L1→L2 transaction that includes `factory_deps` (a standard ZKsync feature used for contract deployments from L1) will trigger this path. The ZKsync L1 contracts allow users to submit such transactions through the priority queue. A user or protocol upgrade that submits an L1 tx with non-empty `factory_deps` will silently have those deps ignored, leading to failed deployments or incorrect state.

The `L1TxBuilder` in the test utilities even exposes a `factory_deps` field, confirming this is an expected use case: [7](#0-6) 

---

### Recommendation

Implement factory deps processing before `run_single_interaction` is called in `process_l1_transaction`. Specifically:

1. Iterate over `transaction.factory_deps` (the parsed `bytes32[]` vector of bytecode hashes).
2. For each hash, retrieve the corresponding bytecode preimage from the oracle and register it in the preimages cache / storage model so it is available for deployment.
3. If a required preimage is unavailable, handle it gracefully (log and treat as out-of-resources, consistent with the L1 tx non-invalidation policy).
4. Implement CREATE support for L1 txs (the second TODO at line 665), unifying with the EOA deployment logic.

---

### Proof of Concept

1. Submit an L1→L2 transaction via the ZKsync priority queue with:
   - `to` = `ContractDeployer` address (`0x8006`)
   - `data` = ABI-encoded `create(salt, bytecodeHash, constructorInput)`
   - `factory_deps` = `[bytecodeHash]` (the bytecode to deploy)
   - `to_mint` = sufficient ETH to cover gas

2. The bootloader receives the transaction in `process_l1_transaction`.

3. At line 139, the `// TODO: l1 transaction preparation (marking factory deps)` is skipped — `factory_deps` is never processed.

4. `run_single_interaction` is called. The call to `ContractDeployer` attempts to deploy using `bytecodeHash`, but the bytecode is not registered in the preimages cache.

5. The deployment reverts (bytecode not found). The L1 tx is marked as failed. The deposited ETH is refunded to `reserved[1]` (refund recipient), but the intended contract is never deployed.

6. The state root produced by the sequencer differs from what the L1 contract expects (it expected the bytecode to be published as part of pubdata), causing a **state-transition divergence**. [8](#0-7)
