### Title
Governance-Settable `max_ingress_bytes_per_message = 0` Causes All Ingress Messages to Be Silently Rejected at the Boundary Node - (`File: rs/ingress_manager/src/ingress_handler.rs`, `rs/http_endpoints/public/src/call.rs`)

### Summary

The `max_ingress_bytes_per_message` field in the `SubnetRecord` stored in the NNS registry is a governance-settable parameter that controls the maximum allowed size of a single ingress message. If this value is set to `0` (or an extremely small value), every ingress message submitted to the subnet will be rejected at the boundary node HTTP endpoint and at the ingress pool validation layer, because every real message will exceed the limit. This is the direct IC analog of the Morpho `maxGas = 0` bug: a misconfigured governance parameter causes a critical protocol function to silently fail for all users.

### Finding Description

The `SubnetRecord` protobuf field `max_ingress_bytes_per_message` is a `uint64` with no enforced minimum in the registry canister mutation path. It is set at subnet creation via `CreateSubnetPayload` and can be updated via `UpdateSubnetPayload` through an NNS governance proposal.

At the boundary node HTTP endpoint, every incoming ingress call is checked against this value:

```rust
// rs/http_endpoints/public/src/call.rs:302
if msg.count_bytes() > ingress_registry_settings.max_ingress_bytes_per_message {
    Err(HttpError { status: StatusCode::PAYLOAD_TOO_LARGE, ... })?;
}
```

At the ingress pool validation layer, the same check is applied:

```rust
// rs/ingress_manager/src/ingress_handler.rs:177
if size > settings.max_ingress_bytes_per_message {
    return Err(IngressMessageValidationError::IngressMessageTooLarge { ... });
}
```

And again during payload selection and validation:

```rust
// rs/ingress_manager/src/ingress_selector.rs:501
if ingress_message_size > settings.max_ingress_bytes_per_message {
    return Err(ValidationError::InvalidArtifact(
        InvalidIngressPayloadReason::IngressMessageTooBig(...),
    ));
}
```

If `max_ingress_bytes_per_message` is set to `0`, every ingress message (which always has non-zero byte size due to headers, signatures, etc.) will be rejected. There is **no floor enforcement** for this field in the registry mutation path — unlike `max_ingress_messages_per_block`, which has an explicit floor of `1` applied in `IngressManager::get_ingress_message_settings`:

```rust
// rs/ingress_manager/src/lib.rs:189-196
if settings.max_ingress_messages_per_block == 0 {
    warn!(...);
    settings.max_ingress_messages_per_block = 1;
}
```

No equivalent floor guard exists for `max_ingress_bytes_per_message`. Similarly, an excessively large value (e.g., larger than the actual block payload size) would allow oversized messages to pass the per-message check but then be rejected at the block-level size check, creating inconsistency.

### Impact Explanation

If `max_ingress_bytes_per_message` is set to `0` or a value smaller than the minimum real ingress message size (which is at least several hundred bytes due to envelope, signature, and headers), **all ingress messages to the affected subnet are permanently rejected** at the boundary node with HTTP 413. No canister on the subnet can receive any user-initiated update call. The subnet becomes effectively inaccessible to all unprivileged users for the duration the misconfigured value is in the registry. This is a subnet-wide denial of service for all ingress traffic.

### Likelihood Explanation

This requires an NNS governance proposal to pass (majority vote). However, the NNS governance system has no validation that prevents `max_ingress_bytes_per_message = 0` from being written to the registry — the registry canister's `do_update_subnet` mutation applies the value directly without a minimum check. A malicious or mistaken governance proposal could set this to `0`. Unlike `max_ingress_messages_per_block`, there is no runtime floor guard to recover from this. The risk is analogous to the Morpho `maxGas` finding: a governance-controlled parameter with no safe-minimum enforcement.

### Recommendation

1. Add a minimum floor check for `max_ingress_bytes_per_message` in `IngressManager::get_ingress_message_settings` (analogous to the existing floor for `max_ingress_messages_per_block`), so that a value of `0` is treated as the protocol default.
2. Add validation in the registry canister's `do_update_subnet` and `do_create_subnet` mutation handlers to reject `max_ingress_bytes_per_message` values below a safe minimum (e.g., `MIN_INGRESS_BYTES_PER_MESSAGE`).
3. Document the safe minimum and maximum values for all subnet configuration parameters that affect ingress processing.

### Proof of Concept

**Entry path:** An unprivileged NNS governance participant submits a proposal `UpdateSubnetPayload { max_ingress_bytes_per_message: Some(0), ... }`. The proposal passes. The registry canister writes `max_ingress_bytes_per_message = 0` to the `SubnetRecord`.

**Vulnerable step 1 — boundary node rejection:** [1](#0-0) 

Every subsequent ingress call to the subnet hits `msg.count_bytes() > 0` (always true), returning HTTP 413 to all users.

**Vulnerable step 2 — ingress pool validation:** [2](#0-1) 

Any message that somehow enters the pool is also rejected here.

**Vulnerable step 3 — payload selection:** [3](#0-2) 

No ingress message can ever be included in a block.

**Missing floor guard (contrast with `max_ingress_messages_per_block`):** [4](#0-3) 

The `max_ingress_messages_per_block` field has an explicit floor of `1` applied at runtime. `max_ingress_bytes_per_message` has no such guard.

**No registry-level validation:** [5](#0-4) 

The `do_update_subnet` handler does not validate that `max_ingress_bytes_per_message` is above any minimum value before writing it to the registry.

**Protocol default constants (safe values exist but are not enforced):** [6](#0-5)
