Audit Report

## Title
`spawn_neuron` Unconditionally Copies Parent Hot Keys to Child Neuron With a Different Controller - (File: `rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with `spawn.new_controller` set to a principal different from the parent neuron's controller, the child neuron is created with the parent's full `hot_keys` list cloned in unconditionally. Hot keys are authorized to vote and change followees on behalf of a neuron. The new controller of the child neuron never consented to these inherited hot keys, allowing the parent's hot key holders to exercise governance authority over a neuron they do not own.

## Finding Description
In `spawn_neuron`, the `child_controller` is resolved from `spawn.new_controller` and can be an entirely different principal from the parent's controller: [1](#0-0) 

Immediately after, the child neuron is built with `child_controller` as owner but with the parent's hot keys cloned in without any conditional check: [2](#0-1) 

The `is_authorized_to_vote` method grants voting rights to any principal in `hot_keys`, regardless of whether that principal has any relationship to the current controller: [3](#0-2) 

There is no guard that clears or conditionally omits hot keys when `child_controller != parent_neuron.controller()`. The exploit path is:
1. Alice creates a neuron and adds Mallory as a hot key via `AddHotKey`.
2. Alice calls `spawn_neuron` with `new_controller = Some(Bob)`.
3. The governance canister creates a child neuron with `controller = Bob` and `hot_keys = [Mallory]`.
4. After the 7-day spawn delay, Mallory calls `manage_neuron → RegisterVote` on the child neuron ID, casting a vote on an active NNS proposal using Bob's neuron without Bob's knowledge or consent.

## Impact Explanation
This constitutes unauthorized access to a governance asset (Bob's neuron). Mallory can vote on NNS proposals and change followees using Bob's neuron's voting power without Bob's authorization. This maps to the **High ($2,000–$10,000)** impact class: *"Unauthorized access to neurons, governance assets … where exploitation requires meaningful per-target work or other constraints."* The constraint is that Alice must spawn to a different controller, which is a documented and tested feature path.

## Likelihood Explanation
`spawn_neuron` with `new_controller` is a documented, reachable feature exercised in integration tests and exposed via the Rosetta API. Any neuron controller who has previously added hot keys (a common operational security practice) and then spawns a neuron for a different principal triggers this condition. Mallory needs no special privilege beyond already being a registered hot key on the parent neuron. There is no notification to Bob and no enforcement that he inspect or clear hot keys before the spawn delay expires.

## Recommendation
In `spawn_neuron`, when `child_controller != parent_neuron.controller()`, construct the child neuron with an empty hot keys list:

```rust
let hot_keys_for_child = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(hot_keys_for_child)
```

This mirrors the behavior of `disburse_to_neuron`, which does not propagate hot keys to the child neuron, and is consistent with the principle that a new controller should start with a clean authorization state.

## Proof of Concept
1. Deploy a local replica or use PocketIC.
2. Create a neuron controlled by Alice; call `AddHotKey` to add Mallory.
3. Accumulate maturity; call `spawn_neuron` with `new_controller = Some(Bob)` and any nonce.
4. Assert the returned child neuron's `hot_keys` field contains Mallory — confirming the bug.
5. Advance time past the 7-day spawn delay.
6. As Mallory, call `manage_neuron → RegisterVote` on the child neuron ID against an active proposal.
7. Confirm the vote is accepted, demonstrating Mallory exercised governance authority over Bob's neuron without Bob's consent.
