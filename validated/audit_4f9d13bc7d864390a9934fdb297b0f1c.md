Audit Report

## Title
Off-by-One in SNS Governance `register_vote` Deadline Check Allows Voting at Exact Deadline Second, Enabling Unauthorized Wait-for-Quiet Extension - (File: rs/sns/governance/src/governance.rs)

## Summary
SNS governance's `register_vote` uses `now_seconds > deadline` to reject late votes, which permits a vote when `now_seconds == deadline`. The canonical `accepts_vote` helper uses strict `<`, treating the deadline second as expired. Because `evaluate_wait_for_quiet` also uses `>` for its early-return guard, a vote cast at exactly the deadline second bypasses both guards and can trigger a wait-for-quiet deadline extension, prolonging a proposal's voting period beyond its intended close.

## Finding Description
**Root cause — `register_vote`:** [1](#0-0) 
The guard `now_seconds > deadline` is `false` when `now_seconds == deadline`, so the vote proceeds.

**Canonical deadline semantics — `accepts_vote`:** [2](#0-1) 
`accepts_vote` uses strict `<`; at `now_seconds == deadline` it returns `false` — the deadline is considered reached.

**Wait-for-quiet guard — `evaluate_wait_for_quiet`:** [3](#0-2) 
The early-return condition `now_seconds > current_deadline` is also `false` at `now_seconds == deadline`, so the function proceeds to compute and apply a deadline extension.

**Deadline extension logic:** [4](#0-3) 
`required_margin` is computed and `current_deadline_timestamp_seconds` is updated to `now_seconds + required_margin`, extending the deadline.

**Call chain from `register_vote`:** [5](#0-4) 
`cast_vote_and_cascade_follow` records the vote, then `process_proposal` is called, which invokes `recompute_tally`: [6](#0-5) 

**Contrast with NNS governance**, which correctly delegates to `accepts_vote` (strict `<`): [7](#0-6) 
NNS is consistent; SNS is not.

## Impact Explanation
An SNS neuron holder controlling enough voting power to flip the current tally (yes→no or no→yes) can submit a vote at exactly `now_seconds == deadline`. The vote is accepted, the tally flips, and `evaluate_wait_for_quiet` extends `current_deadline_timestamp_seconds` by up to `wait_for_quiet_deadline_increase_seconds` (typically 86400 seconds / 1 day for SNS defaults). The proposal cannot be finalized until the extended deadline passes. This is unauthorized mutation of SNS governance state — a concrete governance manipulation impact against an explicitly in-scope SNS canister, fitting the **High** impact tier: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The deadline is stored in public canister state readable by any observer via `get_proposal`. IC consensus produces blocks roughly every second; the attacker simply waits for the round where `env.now() == deadline` and submits their ingress message. No mempool front-running, special privileges, or social engineering is required. The only constraint is controlling a neuron with sufficient voting power to flip the current tally — a realistic condition for a large SNS token holder or coordinated group, and well below a governance-majority threshold.

## Recommendation
Change the deadline guard in `register_vote` from strict `>` to `>=`, consistent with `accepts_vote`:

```diff
- if now_seconds > deadline {
+ if now_seconds >= deadline {
      return Err(GovernanceError::new_with_message(
          ErrorType::PreconditionFailed,
          "Proposal deadline has passed.",
      ));
  }
``` [8](#0-7) 

This aligns `register_vote` with `accepts_vote` and closes the one-second window.

## Proof of Concept
**Setup:** SNS proposal with `initial_voting_period_seconds = 345600` (4 days), `wait_for_quiet_deadline_increase_seconds = 86400` (1 day). Current tally slightly favors Yes. `current_deadline_timestamp_seconds = T`.

1. Read `get_proposal` to obtain `T`.
2. Wait until the IC round where `env.now() == T`.
3. Call `manage_neuron { RegisterVote { proposal_id, vote: No } }` with a neuron large enough to flip Yes→No.
4. Execution:
   - `register_vote`: `T > T` → `false` → vote accepted.
   - `cast_vote_and_cascade_follow` records No, flipping the tally.
   - `process_proposal` → `recompute_tally` → `evaluate_wait_for_quiet`:
     - `T > T` → `false` → does not return early.
     - `vote_has_turned` → `true`.
     - `required_margin = 86400 + 172800 - elapsed/2 > 0`.
     - `new_deadline = T + required_margin > T` → deadline extended.
5. **Result:** `current_deadline_timestamp_seconds` is now `T + ~86400+`; the proposal cannot be finalized until the new deadline passes.

A deterministic unit test in `rs/sns/governance/src/` can reproduce this by constructing a `ProposalData` with a known deadline, setting `env.now() == deadline`, calling `register_vote` with a flip vote, and asserting that `current_deadline_timestamp_seconds` has increased — which it will under the current code and must not under the fix.
