Audit Report

## Title
Unbounded Ingress Window Loop in ICP Rosetta `construction_payloads` Causes OOM/Infinite-Loop DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from unauthenticated JSON request metadata and feeds them directly into an unbounded `while` loop with no window-size validation. Setting `ingress_start = 0` and `ingress_end = u64::MAX` causes the loop to attempt ~153 billion `Vec::push` calls, exhausting process memory (OOM kill), or — due to wrapping arithmetic in `Time::AddAssign` — spinning indefinitely.

## Finding Description
The interval is computed at line 59–60 of `construction_payloads.rs`:
```rust
let interval = ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT - Duration::from_secs(120);
```
With `MAX_INGRESS_TTL = 300s` and `PERMITTED_DRIFT = 60s`, `interval = 120s = 120_000_000_000 ns`.

The loop at lines 99–107 has no guard on the window size:
```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;
}
``` [1](#0-0) 

`ingress_start` and `ingress_end` are plain `Option<u64>` fields deserialized from JSON with no range or window-size check: [2](#0-1) 

The `TryFrom<ObjectMap>` conversion is a pure `serde_json` deserialize with no validation: [3](#0-2) 

The `AddAssign<Duration>` implementation for `ic_types::time::Time` uses a truncating `as u64` cast:
```rust
fn from_duration(t: Duration) -> Self {
    Time(t.as_nanos() as u64)
}
``` [4](#0-3) [5](#0-4) 

When `now` approaches `u64::MAX`, `Duration::from_nanos(now) + interval` produces a `u128` that overflows `u64` on the `as u64` cast, wrapping `now` back to a small value (≈ 120s from epoch). The loop condition `now < u64::MAX` remains true and the loop runs forever.

**Two concrete outcomes:**
1. **OOM kill**: `u64::MAX / 120_000_000_000 ≈ 153 billion` iterations × 8 bytes/entry ≈ 1.2 TB allocation → OS OOM-kills the process before wrapping occurs.
2. **Infinite loop**: If OOM does not kill the process (e.g., swap, memory limits), the wrapping arithmetic causes `now` to reset to a small value and the loop never terminates.

No existing guard prevents this. The ICP Rosetta handler has no `ingress_start >= ingress_end` check, no window-size limit, and no iteration cap. By contrast, the ICRC1 Rosetta implementation does include basic ordering checks: [6](#0-5) 

## Impact Explanation
A single malformed HTTP POST to `/construction/payloads` causes the ICP Rosetta process to be killed by the OS OOM killer or spin indefinitely, blocking all ICP ledger transfers routed through that node until it is manually restarted. This matches the allowed impact: **High ($2,000–$10,000) — Significant Rosetta/ledger infrastructure security impact with concrete user or protocol harm** (exchange and custodian ICP transfer availability).

## Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication. Any network-reachable caller can send the payload. The exploit is a single HTTP POST with two integer fields (`ingress_start: 0`, `ingress_end: 18446744073709551615`). No privileged access, key material, or social engineering is required. The attack is trivially repeatable after each restart.

## Recommendation
Add an explicit window-size guard before the loop, mirroring the pattern already present in the ICRC1 Rosetta implementation. Before line 99 in `construction_payloads.rs`, add:
```rust
let max_window = ic_limits::MAX_INGRESS_TTL * 24; // e.g., 24 hours
if ingress_end.saturating_duration_since(ingress_start) > max_window {
    return Err(ApiError::invalid_request(
        "ingress_end − ingress_start exceeds maximum allowed window",
    ));
}
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(
        "ingress_start must be before ingress_end",
    ));
}
```
This bounds `ingress_expiries` to at most `max_window / interval` entries (≤ 720 for a 24-hour window).

## Proof of Concept
```rust
#[test]
fn test_construction_payloads_unbounded_loop() {
    let metadata = ConstructionPayloadsRequestMetadata {
        ingress_start: Some(0),
        ingress_end: Some(u64::MAX),
        memo: None,
        created_at_time: None,
    };
    // Build a minimal valid ConstructionPayloadsRequest with the above metadata,
    // valid TRANSACTION+FEE operations, and a real public key.
    // Call handler.construction_payloads(request).
    // Without the fix: process OOMs or loops indefinitely.
    // With the fix: returns Err immediately.
}
```
The interval arithmetic can be verified independently: `u64::MAX / 120_000_000_000 ≈ 153_722_867_280` iterations; at 8 bytes each, `Vec` growth will trigger OOM well before completion. The wrapping behavior of `Time::from_duration` (truncating `as u64` cast) can be verified with a unit test on `ic_types::time::Time`.
