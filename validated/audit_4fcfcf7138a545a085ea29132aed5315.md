### Title
Truncating Integer Division in `native_per_pubdata` Computation Silently Zeros Pubdata Cost When `pubdata_price < native_price` - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The computation of `native_per_pubdata` in the ZK transaction validation path uses integer floor division (`wrapping_div`). When the operator-supplied `pubdata_price` is less than `native_price`, the result truncates to **zero**. A zero `native_per_pubdata` means the transaction is charged **nothing** for any amount of pubdata it writes, allowing an attacker to generate unbounded pubdata at zero native cost while still passing all validation checks.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (the L2 ZK transaction validation path), the per-pubdata-byte native cost is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This is a **floor division**. If `pubdata_price < native_price`, the result is `0`. The same pattern is replicated in the public API helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

The only guard present is a check that `native_price != 0` (line 122–124 of `validation_impl.rs`). There is **no check** that `native_per_pubdata != 0` after the division.

`native_per_pubdata` is then threaded through the entire resource accounting pipeline:

1. **Intrinsic pubdata overhead** at validation time: `native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` — evaluates to `0`, so no native is deducted for intrinsic pubdata.
2. **Post-execution pubdata charge**: `current_pubdata_spent.checked_mul(native_per_pubdata)` — evaluates to `0`, so no native is charged for any pubdata written during execution.
3. **Pubdata sufficiency check**: `resources.has_enough(&resources_for_pubdata)` — always passes because `resources_for_pubdata` is zero.

The result is that a transaction that writes an arbitrarily large amount of state-diff pubdata passes all resource checks and is accepted into a block with zero native cost for that pubdata.

The `pubdata_price` and `native_price` values come from the `BlockMetadataFromOracle` struct, which is populated from the oracle (sequencer-supplied block metadata). The oracle documentation explicitly states responses are **untrusted input**. In the forward (sequencer) execution path, these values are passed in directly by the operator without any on-chain enforcement of the relationship `pubdata_price >= native_price`.

---

### Impact Explanation

**Resource accounting bug / valid-execution unprovability divergence.**

- An attacker who can influence block metadata (e.g., a malicious or misconfigured sequencer, or a sequencer that sets `pubdata_price` to a value smaller than `native_price`) can submit transactions that write large amounts of pubdata at zero native cost.
- The transaction passes validation and executes successfully, but the actual proving cost of the pubdata is not covered by the native resource budget.
- This leads to **underpayment for pubdata**: the sequencer/prover bears the cost of publishing and proving large state diffs that were not charged to the transaction sender.
- In the worst case, a single transaction can fill the entire block's pubdata limit (`pubdata_limit`) at zero native cost, effectively performing a **free DoS** against the block's data availability capacity.
- The `check_enough_resources_for_pubdata` call at post-execution will always return `enough = true` when `native_per_pubdata == 0`, so the transaction is never reverted for pubdata overuse.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a **realistic operational scenario**:

- The `BlockMetadataFromOracle` default test value sets `pubdata_price = 0` and `native_price = 10`, which immediately triggers this condition.
- Any block where the operator sets `pubdata_price` to a value smaller than `native_price` (e.g., during low-fee periods or misconfiguration) silently enables free pubdata.
- The attacker entry path is a standard L2 transaction — no privileged access is required. The attacker only needs to submit a transaction that writes storage slots when `native_per_pubdata` is zero.

---

### Recommendation

Add an explicit check after computing `native_per_pubdata` to reject transactions (or treat pubdata as free only when explicitly intended) when the division truncates to zero but `pubdata_price` is nonzero:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// Add: reject if pubdata_price is nonzero but native_per_pubdata rounded to zero,
// OR use ceiling division to avoid silent truncation:
// let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
//     .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Using `div_ceil` (as is already done for `native_per_gas` on line 135) would be consistent and prevent the truncation-to-zero case. Alternatively, add a post-division check: if `pubdata_price > U256::ZERO && native_per_pubdata == 0`, return a validation error.

The same fix should be applied to `api/src/helpers.rs` line 427.

---

### Proof of Concept

**Trigger condition**: Set block context with `pubdata_price = 1` and `native_price = 10`.

```
native_per_pubdata = 1 / 10 = 0  (integer floor division)
```

**Effect trace**:

1. `create_resources_for_tx` is called with `native_per_pubdata_byte = 0`.
   - `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0` → no native deducted for intrinsic pubdata.
2. Transaction executes and writes N storage slots, generating N × 66 bytes of pubdata.
3. `get_resources_to_charge_for_pubdata(system, 0, None)` returns `native = 0`.
4. `check_enough_resources_for_pubdata` returns `enough = true`.
5. Transaction succeeds with zero native charged for pubdata.
