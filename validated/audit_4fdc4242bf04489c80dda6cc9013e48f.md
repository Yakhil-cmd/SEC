### Title
EIP-7623 Floor Gas Not Enforced in Refund — Sender Underpays Calldata Cost Floor — (`engine/src/engine.rs`)

### Summary
`refund_unused_gas` refunds `(gas_limit − gas_used) × price` without enforcing the EIP-7623 invariant that the sender must consume at least `floor_gas`. A sender who sets `gas_limit = floor_gas` and executes a path that uses fewer gas units than `floor_gas` receives a refund that reduces their net payment below the mandatory floor, stealing fees from the relayer.

### Finding Description

`floor_gas` is computed in `NormalizedEthTransaction::floor_gas` and used in `submit_with_alt_modexp` only to validate that `gas_limit >= max(intrinsic_gas, floor_gas)`. [1](#0-0) 

After execution, the refund path receives `gas_used` (raw EVM gas consumed) but never `floor_gas`: [2](#0-1) 

The `charge_gas` function charges `gas_limit × effective_gas_price` upfront: [3](#0-2) 

So the net cost to the sender is `gas_used × price`, not `max(gas_used, floor_gas) × price` as EIP-7623 requires.

`floor_gas` itself is computed per EIP-7623 as `base_gas + tokens_in_calldata × total_cost_floor_per_token`: [4](#0-3) 

### Impact Explanation

An attacker crafts a transaction with large calldata (e.g., 1000 non-zero bytes → `floor_gas = 61000`) targeting a contract whose execution path consumes only `intrinsic_gas = 21000`. They set `gas_limit = floor_gas = 61000`. The engine charges `61000 × price` upfront, executes, observes `gas_used = 21000`, and refunds `40000 × price` back to the sender. Net cost: `21000 × price` instead of the mandated `61000 × price`. The `40000 × price` difference is stolen from the relayer's fee income. This is repeatable at scale with any calldata-heavy transaction whose execution is cheap.

### Likelihood Explanation

Prague config is hardcoded as the active config (`const CONFIG: &Config = &Config::prague();`): [5](#0-4) 

`has_floor_gas` is therefore always `true` in production. Any user can submit a transaction through the standard `submit` entrypoint. No special privileges are required. The exploit is deterministic and requires only crafting a transaction with large calldata and a cheap execution target.

### Recommendation

Pass `floor_gas` into `refund_unused_gas` and replace `gas_used` with `max(gas_used, floor_gas)` as the minimum consumed gas when computing the refund:

```rust
let effective_gas_used = core::cmp::max(gas_used, floor_gas);
refund_unused_gas(&mut io, &sender, effective_gas_used, &prepaid_amount, &relayer_address, fixed_gas)?;
```

This ensures the sender's net payment is always at least `floor_gas × effective_gas_price`, satisfying the EIP-7623 invariant.

    pub fn floor_gas(&self, config: &aurora_evm::Config) -> Result<u64, Error> {
        if config.has_floor_gas {
            let num_zero_bytes = u64::try_from(self.data.iter().filter(|b| **b == 0).count())
                .map_err(|_e| Error::IntegerConversion)?;
            let data_len = u64::try_from(self.data.len()).map_err(|_e| Error::IntegerConversion)?;
            let num_non_zero_bytes = data_len
                .checked_sub(num_zero_bytes)
                .ok_or(Error::GasOverflow)?;

            let base_gas = config.gas_transaction_call;
            let tokens_in_calldata = num_non_zero_bytes
                .checked_mul(4)
                .and_then(|gas| gas.checked_add(num_zero_bytes))
                .ok_or(Error::GasOverflow)?;

            tokens_in_calldata
                .checked_mul(config.total_cost_floor_per_token)
                .and_then(|gas| gas.checked_add(base_gas))
                .ok_or(Error::GasOverflow)
        } else {
            Ok(0)
        }
    }
