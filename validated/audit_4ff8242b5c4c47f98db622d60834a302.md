### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Silently Overcharges Users When Native Resource Consumption Is Lower Than EVM Gas Implies - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, ZKsync OS's double-resource accounting adjusts `gas_used` upward when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), but **never adjusts it downward** when native resource consumption is lower (`delta_gas < 0`). This is the direct analog of the reported vulnerability: the user "sends" a full `gas_limit * gas_price` fee budget, only a fraction of the native resource budget is consumed, but the unspent portion is not returned — it flows to the operator instead of the user.

The code even contains an explicit `// TODO: return delta_gas to gas_used?` comment at the exact location of the missing branch, confirming the gap is known but unresolved.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements ZKsync OS's double-resource accounting reconciliation. After computing `gas_used` from ergs (EVM gas), it computes:

```
delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64)
```

The code then applies an **asymmetric** correction:

```rust
if delta_gas > 0 {
    // native consumption is more than EVM gas implies → charge extra gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

When `delta_gas < 0` — meaning the transaction's native resource consumption corresponds to **fewer** gas units than the ergs-based `gas_used` — the code does nothing. `gas_used` remains at the higher ergs-derived value.

The user's refund is then computed as:

```rust
let token_to_refund = context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
``` [2](#0-1) 

Because `gas_used` was not reduced by `|delta_gas|`, the user receives a smaller refund than the actual resource consumption justifies. The excess fee (`|delta_gas| * gas_price`) is paid to the operator:

```rust
let token_to_pay_operator = U256::from(context.gas_used).checked_mul(gas_price_for_operator)...
``` [3](#0-2) 

This same `compute_gas_refund` function is called for both L2 transactions (via `before_refund` in `ZkTransactionFlow`) and L1→L2 transactions (via `process_l1_transaction`): [4](#0-3) [5](#0-4) 

The protocol documentation explicitly states:

> "If `deltaGas > 0`, we add it to `gasUsed`… Finally, any remaining gas left is refunded as usual." [6](#0-5) 

The documentation's "refunded as usual" clause implies the negative case should also be handled, but the code omits it.

---

### Impact Explanation

Every transaction where EVM gas consumption exceeds what native resource consumption justifies results in the user being overcharged. The excess fee is transferred to the operator (or burned as base fee under EIP-1559). This is a direct, systematic financial loss for users — structurally identical to the reported vulnerability where unspent input tokens are retained by the contract rather than returned to the sender.

The magnitude per transaction is `|delta_gas| * gas_price` tokens. For transactions with complex EVM computation but low storage writes (low pubdata), `delta_gas` can be significantly negative, making the overcharge substantial.

---

### Likelihood Explanation

The `delta_gas < 0` condition is reached whenever a transaction's EVM gas consumption is higher than its native resource consumption implies. This is a common scenario for:

- Transactions with heavy EVM computation but few storage writes (low pubdata cost)
- Any block context where `native_price` is set such that `native_per_gas` is low, making `native_used / native_per_gas` small relative to `gas_used`

The condition is reachable by any unprivileged L2 or L1→L2 transaction sender without any special privileges or external dependencies. The operator's control over `native_price` can amplify the effect.

---

### Recommendation

Apply the symmetric correction: when `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (clamped to not go below `minimal_gas_used`):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native resource consumption is less than EVM gas implies.
    // Reduce gas_used so the user is refunded the difference,
    // but never below the intrinsic minimum.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This resolves the TODO at line 80 and ensures the refund correctly reflects actual resource consumption in both directions.

### Proof of Concept

**Setup:**
- `gas_limit = 100,000`, `gas_price = 1,000`
- `native_price = 100`, so `native_per_gas = gas_price / native_price = 10`
- `native_limit = gas_limit * native_per_gas = 1,000,000`

**Execution:**
- Transaction uses 50,000 EVM gas → `gas_used = 50,000` (from ergs)
- Transaction uses 200,000 native units → `native_used = 200,000`
- `native_used / native_per_gas = 200,000 / 10 = 20,000`
- `delta_gas = 20,000 - 50,000 = -30,000` (negative)

**Current behavior (buggy):**
- `delta_gas < 0` → branch not taken, `gas_used` stays at `50,000`
- User refund: `(100,000 - 50,000) * 1,000 = 50,000,000` tokens
- Operator receives: `50,000 * 1,000 = 50,000,000` tokens

**Correct behavior:**
- `gas_used` should be reduced to `max(20,000, minimal_gas_used)`
- User refund: `(100,000 - 20,000) * 1,000 = 80,000,000` tokens
- Operator receives: `20,000 * 1,000 = 20,000,000` tokens

**User loss per transaction: `30,000,000` tokens** — transferred to the operator rather than refunded.

The entry path is a standard L2 transaction submitted by any unprivileged user. No privileged access, oracle manipulation, or external dependency is required.
