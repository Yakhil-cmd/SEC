Audit Report

## Title
Silent Fire-and-Forget Execution of `ManageDappCanisterSettings` Always Returns False Success to SNS Governance — (`rs/sns/root/src/lib.rs`)

## Summary

`manage_dapp_canister_settings` in SNS root spawns the actual management canister calls as a fire-and-forget future via `CdkRuntime::spawn_future` and unconditionally returns `ManageDappCanisterSettingsResponse { failure_reason: None }` before any per-canister `update_settings` call completes. Additionally, the spawned function `call_management_canister_for_update_dapp_canister_settings` returns `()` and swallows all per-canister errors by only logging them. SNS governance therefore always marks a `ManageDappCanisterSettings` proposal as successfully executed, even when every targeted canister's settings update failed.

## Finding Description

**Root cause — two compounding defects:**

**Defect 1 (fire-and-forget):** `manage_dapp_canister_settings` at `rs/sns/root/src/lib.rs:906-912` calls `CdkRuntime::spawn_future(...)`, which resolves to `ic_cdk::spawn(future)`. The response is constructed and returned on the very next line, before the spawned future has executed a single await point. The `failure_reason: None` is therefore structurally guaranteed regardless of runtime outcome.

**Defect 2 (error swallowing):** `call_management_canister_for_update_dapp_canister_settings` at `rs/sns/root/src/lib.rs:220-248` iterates over all canister IDs and calls `update_settings` on each. Every `Err` branch only logs the error; no error accumulation, no early return, no propagation. The function signature `-> ()` makes it impossible to propagate failures even if the caller were awaiting it.

**Governance call chain:**

1. SNS governance's `perform_manage_dapp_canister_settings` (`rs/sns/governance/src/governance.rs:3214-3255`) calls root's `manage_dapp_canister_settings` via inter-canister call and awaits the response.
2. Root immediately returns `ManageDappCanisterSettingsResponse { failure_reason: None }` (line 910-912) while the real work runs in a detached future.
3. Governance decodes the response and checks `failure_reason` (line 3240-3248): `None` → `Ok(())` → proposal marked as executed successfully.

No existing guard catches this: the validation step (`ValidatedManageDappCanisterSettingsRequest::try_from`) only checks that canister IDs are registered dapp canisters, not that the management canister calls will succeed. The `failure_reason` field in the response proto (`rs/sns/root/proto/ic_sns_root/pb/v1/root.proto:179-182`) is documented as "absence of failure_reason indicates success" but is never populated for per-canister execution errors.

## Impact Explanation

A passed `ManageDappCanisterSettings` governance proposal targeting N dapp canisters where one or more canisters are frozen (or otherwise reject the management call) will:
- Partially apply settings to reachable canisters
- Silently skip unreachable canisters
- Report full success to governance

Settings that can be silently misapplied include `freezing_threshold` and `reserved_cycles_limit` — both security-critical parameters that govern canister liveness and resource protection. A governance action intended to uniformly harden all dapp canisters (e.g., raising `freezing_threshold` in response to a known attack vector) will leave a subset of canisters unprotected with no indication to token holders or the SNS framework. This constitutes **significant SNS security impact with concrete user and protocol harm** — the governance mechanism's execution integrity is broken for this proposal type, and the resulting inconsistent security posture across dapp canisters is invisible to all participants.

## Likelihood Explanation

The precondition — one targeted dapp canister being frozen — is a natural production condition requiring no attacker action. Any SNS with multiple dapp canisters where one has been drained of cycles (common in production) is permanently susceptible. Any token holder with sufficient voting power to pass a `ManageDappCanisterSettings` proposal can observe the silent failure. The scenario is fully deterministic and reproducible in a local state machine test.

## Recommendation

Restructure `manage_dapp_canister_settings` to await the management canister calls before returning. Change `call_management_canister_for_update_dapp_canister_settings` to return `Result<(), String>`, collect all per-canister errors into a single error string, and propagate it. The `manage_dapp_canister_settings` handler must await the result (making the canister method `async`) and populate `ManageDappCanisterSettingsResponse { failure_reason: Some(...) }` when any update fails. If the fire-and-forget pattern must be preserved for deadlock-avoidance reasons (as documented for `change_canister`), the spawned future must write its result back to stable state and expose it via a separate query, with governance polling for completion — but the current design of returning `failure_reason: None` unconditionally is incorrect.

## Proof of Concept

1. Deploy a local SNS with two registered dapp canisters A and B.
2. Drain canister B's cycles below its freezing threshold so it is frozen.
3. Submit and pass a `ManageDappCanisterSettings` proposal targeting both A and B with `freezing_threshold = 999999`.
4. Observe: `perform_manage_dapp_canister_settings` in governance returns `Ok(())` and the proposal is marked executed.
5. Query canister A's settings: `freezing_threshold` is updated to 999999.
6. Query canister B's settings: `freezing_threshold` is unchanged — silent partial failure, invisible to governance.

This is directly reproducible as a PocketIC or state-machine integration test by extending the existing test suite at `rs/sns/integration_tests/src/manage_dapp_canister_settings.rs`.
