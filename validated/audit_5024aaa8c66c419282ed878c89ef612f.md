Audit Report

## Title
Unbounded Synchronous BFS in `cast_vote_and_cascade_follow` Enables Persistent Governance DoS - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `cast_vote_and_cascade_follow` function performs a synchronous, unbounded BFS over the neuron follower graph with no instruction-limit guard. An unprivileged attacker who creates many neurons all following a single target neuron can cause every `register_vote` or `make_proposal` call by that target neuron to trap due to instruction exhaustion, permanently preventing it from voting and potentially blocking proposals from reaching quorum.

## Finding Description
`cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs` is a synchronous (non-`async`) function. Its core loop at line 3749 iterates over the entire follower graph with no check against the IC instruction limit:

```rust
while !induction_votes.is_empty() {
    // ... processes all followers of current tier ...
}
``` [1](#0-0) 

This function is called synchronously from both `make_proposal` (line 3658) and `register_vote` (line 3931), meaning the entire BFS must complete within a single message execution: [2](#0-1) [3](#0-2) 

A grep search for `instruction_counter`, `over_soft_message_limit`, and `is_message_over_threshold` in `rs/sns/governance/**/*.rs` returns zero matches, confirming there is no instruction-limit guard anywhere in SNS governance.

By contrast, NNS governance's `cast_vote_and_cascade_follow` in `rs/nns/governance/src/voting.rs` uses `over_soft_message_limit()` and `noop_self_call_if_over_instructions()` with a `VotingStateMachine` that persists state and defers remaining BFS work to a timer task across multiple messages: [4](#0-3) [5](#0-4) 

The existing `max_followees_per_function` guard (line 3991) only limits how many neurons a single neuron can *follow* (outgoing edges), not how many neurons can follow a single target (incoming edges). Any number of neurons up to `max_number_of_neurons` can all follow the same target, making the BFS fan-out proportional to the total neuron count: [6](#0-5) 

## Impact Explanation
When the BFS loop exhausts the per-message instruction limit (~40 billion instructions for update calls), the entire `register_vote` or `make_proposal` update call traps and all state changes are rolled back. The target neuron's vote is never recorded. If the target neuron is a high-weight voter or a followee of many other neurons, this prevents the proposal from reaching quorum, causing a persistent governance liveness failure for the affected SNS. This matches the allowed High impact: **Application/platform-level DoS** on an SNS governance canister with concrete governance harm.

## Likelihood Explanation
The attacker only needs standard unprivileged SNS operations: acquiring SNS tokens, creating neurons, and configuring following relationships. The `max_number_of_neurons` parameter is configurable per-SNS and can be set to values in the thousands. Once enough follower neurons are created to exhaust the instruction budget in a single BFS pass, the DoS is persistent — every subsequent vote attempt by the target neuron traps identically. The cost is proportional to the SNS neuron creation fee, which is a one-time investment for a permanent effect.

## Recommendation
Introduce an instruction-limit check inside the BFS loop in `cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs`, analogous to the mechanism already present in NNS governance (`rs/nns/governance/src/voting.rs`). Specifically:
1. Convert `cast_vote_and_cascade_follow` to an `async` function.
2. After each BFS tier, check `is_message_over_threshold(SOFT_VOTING_INSTRUCTIONS_LIMIT)`.
3. If over the threshold, persist the remaining `induction_votes` to stable state and schedule a timer task to continue processing in a subsequent message.
4. Alternatively, enforce a hard cap on the number of *incoming* followers per neuron (not just outgoing followees) to bound the BFS fan-out.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons = N` (e.g., 10,000).
2. As an attacker, acquire sufficient SNS tokens and create `N - 1` neurons, configuring each to follow a single target neuron (the victim) on all function IDs via the `follow` management neuron command.
3. Submit a proposal or wait for the victim neuron to call `register_vote`.
4. The SNS governance canister executes `cast_vote_and_cascade_follow` synchronously. The BFS loop must process up to `N - 1` follower neurons in a single message.
5. The message exhausts the IC instruction limit and traps. The vote is not recorded and all state changes are rolled back.
6. Every subsequent `register_vote` call by the victim neuron traps identically, permanently preventing it from voting.
7. A deterministic integration test using PocketIC can reproduce this by creating thousands of follower neurons and asserting that `register_vote` returns a trap/error after the follower count crosses the instruction-exhaustion threshold.
