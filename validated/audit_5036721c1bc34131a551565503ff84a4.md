### Title
Unbounded Batch Aggregation + Post-Serialization Response with No Size Cap Enables OOM DoS on EVM RPC — (`evmrpc/server.go`, `evmrpc/rpcstack.go`, `evmrpc/filter.go`)

---

### Summary

Sei's EVM JSON-RPC HTTP server initializes `batchItemLimit` and `batchResponseSizeLimit` as zero (unlimited) and never sets them before calling `srv.SetBatchLimits(0, 0)`. The Tendermint JSON-RPC server fully materializes all batch responses into a single `[]byte` with no response-size cap. An unauthenticated attacker can exploit both surfaces to exhaust process memory and crash the node's RPC process.

---

### Finding Description

**Surface 1 — EVM RPC (go-ethereum-based, port 8545)**

`NewEVMHTTPServer` in `evmrpc/server.go` calls `httpServer.EnableRPC(apis, HTTPConfig{...})` without ever populating `batchItemLimit` or `batchResponseSizeLimit` in the `HTTPConfig` struct:

```go
if err := httpServer.EnableRPC(apis, HTTPConfig{
    CorsAllowedOrigins: strings.Split(config.CORSOrigins, ","),
    Vhosts:             []string{"*"},
    DenyList:           config.DenyList,
    SeiLegacyAllowlist: seiLegacyAllowlist,
}); err != nil {
``` [1](#0-0) 

Both fields default to `0`. `EnableRPC` then calls `srv.SetBatchLimits(config.batchItemLimit, config.batchResponseSizeLimit)` — i.e., `SetBatchLimits(0, 0)` — which in go-ethereum means no batch item limit and no response size limit. [2](#0-1) 

The `RPCEndpointConfig` struct confirms these fields exist but are never populated for the HTTP server: [3](#0-2) 

**Amplification via `eth_getLogs`**

`GetLogs` enforces a block-range cap (`maxBlock`, default 2000) and has backpressure checks, but returns an unbounded `[]*ethtypes.Log` slice with no cap on the number of log entries or the serialized response size: [4](#0-3) 

With a batch of N `eth_getLogs` requests (N unlimited), each fetching 2000 blocks of dense logs, all N responses are materialized in memory concurrently before the batch is serialized a second time into the HTTP response body. There is no intermediate size check.

**Surface 2 — Tendermint JSON-RPC (port 26657)**

`makeJSONRPCHandler` enforces `REQUEST_BATCH_SIZE_LIMIT = 10` before executing requests, which limits amplification. However, `writeRPCResponse` fully materializes all responses into a single `[]byte` via `json.Marshal(rsps)` with no response-size cap before writing: [5](#0-4) [6](#0-5) 

The Tendermint surface is less severe due to the 10-item batch cap, but the pattern is identical.

---

### Impact Explanation

An unauthenticated attacker sends a single HTTP POST to port 8545 containing a JSON array of many `eth_getLogs` requests (e.g., 1000 items, each with `fromBlock: 0x1, toBlock: latest`). The server:
1. Executes all requests concurrently, materializing N large `[]*ethtypes.Log` slices.
2. Serializes the entire batch into a second large buffer.
3. Only then attempts to write the response.

Peak memory = (sum of all per-item response sizes) × 2 (pre-serialized + serialized). With 2000 blocks × many logs per block × N batch items, this easily reaches multiple GB, triggering OOM kill or severe swap thrash. The RPC process crashes, denying service to all users. This matches the scope criterion: **unauth RPC crash (Medium)**.

---

### Likelihood Explanation

- No authentication is required; the EVM RPC endpoint is public by default.
- The default config (`MaxBlocksForLog = 2000`, no batch item limit, no response size limit) makes this trivially exploitable.
- A single crafted HTTP request is sufficient; no chain state manipulation is needed.
- The backpressure checks in `GetLogs` (queue backlog, semaphore saturation) are runtime-load-dependent and do not prevent the first wave of concurrent requests from consuming memory. [7](#0-6) 

---

### Recommendation

1. **Set batch limits**: Populate `batchItemLimit` and `batchResponseSizeLimit` in `HTTPConfig` before calling `EnableRPC`. Expose these as configurable fields in `evmrpc/config/config.go` with safe defaults (e.g., 100 items, 25 MB).
2. **Cap log count per response**: Add a `maxLogsPerResponse` limit in `GetLogs` / `GetFilterLogs` that returns an error if the result set exceeds it, before returning the slice.
3. **Tendermint surface**: Add a response-size cap in `writeRPCResponse` before `w.Write(body)`.

---

### Proof of Concept

```bash
# Craft a batch of 500 eth_getLogs requests, each spanning the full block range
python3 -c "
import json, sys
batch = [{'jsonrpc':'2.0','id':i,'method':'eth_getLogs',
          'params':[{'fromBlock':'0x1','toBlock':'latest'}]}
         for i in range(500)]
sys.stdout.write(json.dumps(batch))
" | curl -s -X POST http://<node>:8545 \
     -H 'Content-Type: application/json' \
     --data-binary @- > /dev/null
# Node RPC process is killed by OOM killer
```

Each item triggers `GetLogs` fetching up to `MaxBlocksForLog` (2000) blocks of logs. All 500 responses are held in memory simultaneously, then re-serialized into the batch response buffer. On a chain with moderate log density, this exhausts available memory before any size check is applied.

func writeRPCResponse(w http.ResponseWriter, rsps ...rpctypes.RPCResponse) {
	var body []byte
	var err error
	if len(rsps) == 1 {
		body, err = json.Marshal(rsps[0])
	} else {
		body, err = json.Marshal(rsps)
	}
	if err != nil {
		logger.Error("Error encoding RPC response", "err", err)
		writeError(w, http.StatusInternalServerError, err)
		return
	}
	statusCode := http.StatusOK
	for _, res := range rsps {
		// If there's any error for lag is high, override the status code
		if res.Error != nil && res.Error.Code == int(rpctypes.CodeLagIsHighError) {
			statusCode = http.StatusExpectationFailed
			break
		}
	}
	w.Header().Set("Content-Type", "application/json")
	w.WriteHeader(statusCode)
	_, _ = w.Write(body)
```
