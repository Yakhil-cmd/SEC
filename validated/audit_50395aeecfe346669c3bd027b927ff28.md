### Title
Truncating Division in `native_per_pubdata` Computation Causes Systematic Pubdata Undercharging — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio — which governs how many native resource units are charged per byte of pubdata — is computed using truncating (floor) integer division (`wrapping_div`), while the analogous `native_per_gas` ratio is correctly computed using ceiling division (`div_ceil`). When `pubdata_price < native_price`, the truncating division yields `native_per_pubdata = 0`, making pubdata completely free regardless of how much the operator intended to charge. Any unprivileged transaction sender can then generate unbounded pubdata at zero native cost.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two pricing ratios are computed as follows:

**`native_per_gas`** — ceiling division (rounds up, user always pays at least the correct amount):
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

**`native_per_pubdata`** — truncating division (rounds down, user may pay nothing):
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same truncating pattern is replicated in the external API helper:
```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

The `native_per_pubdata` value flows directly into `create_resources_for_tx` and `get_resources_to_charge_for_pubdata`, which multiply it by the number of pubdata bytes to determine the native charge:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

When `native_per_pubdata = 0`, this product is always 0 — pubdata is free regardless of volume.

The `L1_TX_NATIVE_PRICE` constant used for L1→L2 transactions is hardcoded at `10`:
```rust
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [5](#0-4) 

For L1→L2 transactions, `native_per_pubdata = gas_per_pubdata * native_per_gas`. If `gas_per_pubdata` is small and `native_per_gas` is also small (e.g., low gas price), the product truncates to 0 via the `checked_mul` path. [6](#0-5) 

---

### Impact Explanation

When `pubdata_price < native_price` (e.g., `pubdata_price = 2`, `native_price = 100` — a configuration present in the existing test suite):

- `native_per_pubdata = floor(2 / 100) = 0`
- Every pubdata byte costs 0 native resources
- A transaction that writes 20 storage slots (generating ~640 bytes of pubdata) pays nothing for that pubdata
- The operator receives no native-resource compensation for the L1 data availability cost
- The chain's pubdata budget can be exhausted by any sender at no cost

The correct value should be `ceil(2 / 100) = 1` native unit per pubdata byte, consistent with how `native_per_gas` is computed. The asymmetry — ceiling for gas, floor for pubdata — is the root cause.

This is a **resource accounting bug**: the price conversion from `pubdata_price` to native units silently truncates to zero, analogous to the RAACNFT bug where a USD price is used directly as a crvUSD amount without conversion.

---

### Likelihood Explanation

High. The existing test `test_check_pubdata_encoding_version` already exercises exactly this configuration (`native_price = 100`, `pubdata_price = 2`), confirming the code path is reachable and that `native_per_pubdata = 0` is a live runtime state. [7](#0-6) 

Any operator that sets `pubdata_price < native_price` — whether intentionally (cheap pubdata period) or as a result of market conditions — exposes the chain to free pubdata spam. No privileged access is required by the attacker; a standard EIP-1559 transaction that performs SSTORE operations is sufficient.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with the treatment of `native_per_gas`:

```rust
// Before (truncates, can yield 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [3](#0-2) 

---

### Proof of Concept

1. Operator sets block context: `native_price = 100`, `pubdata_price = 2`, `eip1559_basefee = 1`.
2. `native_per_pubdata = floor(2 / 100) = 0` (line 142 of `validation_impl.rs`).
3. Attacker submits a transaction calling a contract that performs 20 SSTORE operations (generating ~640 bytes of pubdata).
4. `get_resources_to_charge_for_pubdata` computes `640 * 0 = 0` native units charged for pubdata.
5. The transaction succeeds, all pubdata is published, and the attacker paid 0 native resources for it.
6. Repeating this across many transactions exhausts the block's pubdata limit at zero cost to the attacker, while the operator bears the full L1 data availability cost.
