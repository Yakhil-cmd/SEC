### Title
Cross-Chain Signature Replay via Accepted Unprotected Legacy Transactions - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

The ZKsync OS bootloader explicitly accepts pre-EIP-155 "unprotected" legacy transactions (signature `v ∈ {27, 28}`). For these transactions, the signing hash is computed over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` with **no chain ID included**. This is the direct analog of the reported vulnerability: a signature that does not commit to a chain ID can be replayed on any other EVM-compatible chain that also accepts unprotected legacy transactions.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155 protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()   // ← no chain ID
} else {
    // EIP-155 protected: chain_id IS included
    ...
    apply_u64_encoding_to_hash(chain_id, &mut hasher);
    ...
}
``` [1](#0-0) 

`is_eip155()` returns `false` when `v == 27 || v == 28`: [2](#0-1) 

The caller in `parse_and_compute_signed_hash` then stores the transaction as `Self::Legacy` and returns it without error: [3](#0-2) 

The resulting `sig_hash` (chain-ID-free) is later used directly in `ecrecover` during EOA signature validation: [4](#0-3) 

Because the hash does not commit to any chain, a signature that is valid on Ethereum mainnet is equally valid on ZKsync OS, and vice versa.

---

### Impact Explanation

An attacker who observes a user's unprotected legacy transaction on Ethereum mainnet (or any other EVM chain) can submit the identical RLP bytes to ZKsync OS. The bootloader will:
1. Detect `v ∈ {27, 28}` → compute the same chain-ID-free hash.
2. Run `ecrecover` → recover the same `from` address.
3. Accept the transaction and execute it against the user's ZKsync OS account.

Since Ethereum and ZKsync OS share the same address space, the user's ZKsync OS funds (ETH, ERC-20 tokens) can be drained or arbitrary contract calls can be executed on their behalf. The reverse direction (ZKsync OS → Ethereum mainnet) is equally possible.

---

### Likelihood Explanation

- Pre-EIP-155 transactions are still produced by some hardware wallets in legacy mode, older signing libraries, and custom tooling.
- ZKsync OS is a new chain, so users may have lower nonces there, making nonce-collision replay more likely.
- The attacker only needs to monitor the public Ethereum mempool/chain for unprotected legacy transactions from addresses that also hold funds on ZKsync OS.

---

### Recommendation

Reject unprotected legacy transactions at the bootloader level. Only accept EIP-155-protected legacy transactions (`v = 35 + 2*chainId` or `v = 36 + 2*chainId`) and typed transactions (EIP-2930, EIP-1559, EIP-7702) that structurally include `chain_id` in their signing hash. The check should be added in `parse_and_compute_signed_hash` immediately after parsing the legacy signature:

```rust
if !sig_data.is_eip155() {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

This mirrors the enforcement already applied to all typed transaction formats. [5](#0-4) 

---

### Proof of Concept

1. User signs a legacy ETH-mainnet transfer without EIP-155: `rlp([nonce, gasPrice, gasLimit, to, value, data, 27, r, s])`.
2. Attacker observes the transaction on Ethereum (mempool or confirmed block).
3. Attacker submits the identical RLP bytes to ZKsync OS as a new transaction.
4. `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` detects `v=27`, computes `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the Ethereum signing hash.
5. `ecrecover` recovers the same `from` address; the bootloader accepts the transaction.
6. The transaction executes on ZKsync OS: ETH or tokens are transferred from the victim's ZKsync OS account to the attacker-controlled `to` address. [6](#0-5)

```rust
impl LegacyPayloadParser {
    pub(crate) fn try_parse_and_hash_for_signature_verification<'a>(
        src: &'a [u8],
        expected_chain_id: u64,
    ) -> Result<(LegacyTXInner<'a>, LegacySignatureData<'a>, Bytes32), TxError> {
        // Legacy path: input must be a single list with 9 elements total.
        let mut outer = Rlp::new(src);

        // Strip the list encoding
        let mut inner = outer.list()?;

        // Outer list must be fully consumed
        if !outer.is_empty() {
            return Err(InvalidTransaction::InvalidStructure.into());
        }

        // Capture the concatenation bytes of the first 6 fields for hashing.
        let mark = inner.mark();
        let legacy_inner: LegacyTXInner<'a> = LegacyTXInner::decode_list_body(&mut inner)?;
        let inner_slice = inner.consumed_since(mark);

        let legacy_signature = LegacySignatureData::decode_list_body(&mut inner)?;
        if !inner.is_empty() {
            return Err(InvalidTransaction::InvalidStructure.into());
        }

        let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
            // Unprotected legacy
            let mut hasher = crypto::sha3::Keccak256::new();
            apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
            hasher.update(inner_slice);
            hasher.finalize_reset().into()
        } else {
            // EIP-155 protected legacy: v must match 35 + 2*chainId (+ {0,1})
            let min_v = U256::from(35) + U256::from(expected_chain_id) * U256::from(2);
            if !(legacy_signature.v == min_v || legacy_signature.v == min_v + U256::ONE) {
                return Err(InvalidTransaction::InvalidEncoding.into());
            }

            // Compute signing hash over the 6-field payload plus chainId and two empty strings.
            let chain_id = expected_chain_id;
            let chain_id_encoding_len = u64_encoding_len(chain_id);

            let mut hasher = crypto::sha3::Keccak256::new();
            apply_list_concatenation_encoding_to_hash(
                (inner_slice.len() + chain_id_encoding_len + 2) as u32, // 0x80, 0x80 for r/s
                &mut hasher,
            );
            hasher.update(inner_slice);
            apply_u64_encoding_to_hash(chain_id, &mut hasher);
            hasher.update(&[0x80, 0x80]);
            hasher.finalize_reset().into()
        };

        Ok((legacy_inner, legacy_signature, sig_hash))
    }
