### Title
Inconsistent `MAX_BLOBS_PER_BLOCK` Threshold Causes Forward/Proving Divergence for Blob Transactions - (`File: zk_ee/src/system/constants.rs` and `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs`)

---

### Summary

`MAX_BLOBS_PER_BLOCK` is defined as **6** in the core `zk_ee` crate but as **9** in the Ethereum block-flow module. This causes `blobs_gas_limit()` to return different values in the forward execution system vs. the ZK proving system, and causes the ZK validation path to cap blob transactions at 6 blobs while the Ethereum block header reports a limit of 9. A transaction with 7–9 blobs is accepted by the forward sequencer but rejected by the prover, producing an unprovable valid execution.

---

**Definition 2 — Ethereum block-flow module (value = 9):** [2](#0-1) 

Also redeclared locally in the block-header module: [3](#0-2)

`HeaderAndHistory` (used in the forward/Ethereum execution path) returns `9 * GAS_PER_BLOB`: [4](#0-3)

`TxLevelMetadata` in the ZK path is also sized for 6: [9](#0-8)

### Impact Explanation

The forward system (sequencer) uses `HeaderAndHistory`, which reports `max_blobs() = 9` and `blobs_gas_limit() = 9 * GAS_PER_BLOB`. The ZK proving system uses `BlockMetadataFromOracle`, which reports `max_blobs() = 6` and `blobs_gas_limit() = 6 * GAS_PER_BLOB`. A block containing 7–9 blob transactions passes all forward-execution checks but fails the ZK proving path's blob-count and blob-gas-limit enforcement. This is a **valid-execution unprovability** bug: the sequencer commits to a block that the prover cannot prove, halting the chain or forcing a rollback. Additionally, the inconsistent `blobs_gas_limit()` corrupts the blob base fee calculation used for EIP-4844 fee market enforcement.

---

### Likelihood Explanation

EIP-4844 blob transactions are standard Ethereum transaction type 0x03. Any user can submit a transaction carrying 7–9 blob hashes. The Ethereum forward path accepts it (limit = 9); the ZK proving path rejects it (limit = 6). No privileged access is required. The attacker only needs to craft a valid EIP-4844 transaction with 7–9 blobs and submit it to the sequencer.

---

### Recommendation

Consolidate `MAX_BLOBS_PER_BLOCK` to a single authoritative constant. The Pectra-era Ethereum specification sets the maximum at 9 blobs per block. Either:

1. Update `zk_ee/src/system/constants.rs` to `MAX_BLOBS_PER_BLOCK = 9` and remove the duplicate definitions in `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` and `block_header.rs`; or
2. If 6 is the intended ZKsync OS limit, update `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` and `block_header.rs` to use 6 and ensure `HeaderAndHistory::max_blobs()` returns 6.

Whichever value is chosen must be used consistently across all paths: block-header metadata, transaction validation, and the `EthereumTransactionMetadata` type parameter.

---

### Proof of Concept

1. Construct an EIP-4844 transaction with 8 blob versioned hashes.
2. Submit it to the ZKsync OS forward system (sequencer mode). The forward path uses `HeaderAndHistory::max_blobs() = 9`, so the transaction passes blob-count validation.
3. The sequencer includes the transaction in a block and commits it.
4. The ZK prover attempts to prove the block. It uses `BlockMetadataFromOracle::max_blobs() = 6` and `parse_blobs_list::<6>`, which rejects the transaction with `BlobListTooLong`.
5. The prover cannot generate a valid proof for the committed block, causing a proving failure and chain halt.
