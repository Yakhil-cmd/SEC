Audit Report

## Title
BIP-341 Taproot Output Key Mismatch in Odd-Y Derived Key Case — (`rs/crypto/internal/crypto_lib/threshold_sig/canister_threshold_sig/src/signing/bip340.rs`)

## Summary
In `RerandomizedPresignature::compute`, the odd-y branch of the taproot key derivation computes `P − t·G` as the output key, but BIP-341 mandates `lift_x(P) + t·G = −P + t·G` when P has odd y. These two points differ by `2P` and are never equal for a valid non-identity key. As a result, the IC threshold signing code produces a signature valid under a completely different key than the one a standard BIP-341 library would derive from the same inputs, making any Bitcoin UTXO sent to the IC-derived taproot address permanently unspendable in the odd-y case (~50% of all derived keys).

## Finding Description
BIP-341 defines the taproot output key as:
```
Q = lift_x(P) + t·G
```
where `lift_x(P)` always yields the point with even y. When P has odd y, `lift_x(P) = −P`, so `Q = −P + t·G`.

The IC code at `bip340.rs` lines 153–167:

```rust
if rr_even {
    // P has even y: Q = P + t·G  ← CORRECT
    let (tweak_key, flip_key) = fix_to_even_y(&derived_key.add_points(&g_tweak)?)?;
    let tweak_sum = key_tweak.add(&tap_tweak)?;
    (tweak_key, flip_key, tweak_sum)
} else {
    // P has odd y: IC computes P − t·G  ← WRONG (BIP-341 requires −P + t·G)
    let (tweak_key, flip_key) = fix_to_even_y(&derived_key.sub_points(&g_tweak)?)?;
    let tweak_sum = key_tweak.sub(&tap_tweak)?;
    (tweak_key, flip_key, tweak_sum)
}
```

The even-y branch is correct. The odd-y branch subtracts `t·G` from P instead of negating P first and then adding `t·G`. The resulting point `fix_to_even_y(P − t·G)` is distinct from `fix_to_even_y(−P + t·G)` by `2P`.

The reference single-key implementation in `packages/ic-secp256k1/src/lib.rs` correctly handles this at the scalar level:

```rust
let z = if pk_y_is_even {
    self.key.to_nonzero_scalar().as_ref() + t
} else {
    self.key.to_nonzero_scalar().as_ref().negate() + t  // −x + t, so z·G = −P + t·G
};
```

The `compute_taproot_tweak` function correctly uses only the x-coordinate of the derived key (`pk.affine_x_bytes()`), so the tweak scalar `t` is identical in both the IC code and any standard BIP-341 library. The divergence is solely in how the point operation is applied.

The internal protocol test (`should_be_able_to_perform_taproot_signature`) only calls `proto.verify_signature`, which is the IC's own internal verifier that recomputes the same (buggy) output key and therefore always passes. The third-party cross-check function `verify_taproot_signature_using_third_party` exists in `test_utils/src/lib.rs` but has an early-return bypass (`if msg.len() != 32 { return true; }`) and is not called in the taproot protocol test, so the mismatch is never caught.

## Impact Explanation
A canister calls `schnorr_public_key` to obtain the derived key P (returned in SEC1 compressed form, which may have odd y). Any correct BIP-341 implementation used to compute the Bitcoin taproot address will compute `Q = lift_x(P) + t·G = −P + t·G` when P has odd y. Bitcoin is deposited to the address corresponding to Q. When the canister calls `sign_with_schnorr` with `aux: bip341 { merkle_root_hash: h }`, the IC produces a signature valid under `fix_to_even_y(P − t·G)`, a completely different key. The Bitcoin network rejects the spend. The UTXO is permanently unspendable. This constitutes direct, irreversible loss of Bitcoin chain-fusion assets — matching the Critical impact class: **Theft, permanent loss, or protocol insolvency involving in-scope chain-key/ledger assets**.

## Likelihood Explanation
The y-parity of a derived key is uniformly distributed (50/50) and is determined entirely by the key transcript and derivation path. Neither the depositor nor the canister can predict or control it. Every taproot signing operation has a ~50% probability of hitting the buggy branch. The bug is systematic and deterministic for any given (key, derivation path, taproot root) triple. No special privileges, attacker action, or unusual conditions are required — any canister using the BIP-341 taproot signing feature is affected.

## Recommendation
In the odd-y branch, negate `derived_key` before adding `g_tweak`, and negate `key_tweak` before adding `tap_tweak`, to match `lift_x(P) + t·G`:

```rust
} else {
    // lift_x(P) = -P when P has odd y, so Q = -P + t·G
    let (tweak_key, flip_key) = fix_to_even_y(
        &derived_key.negate().add_points(&g_tweak)?
    )?;
    let tweak_sum = key_tweak.negate().add(&tap_tweak)?;
    (tweak_key, flip_key, tweak_sum)
}
```

Add a cross-implementation test that generates a derived key with known odd y, computes the taproot output key using both the IC code and `verify_taproot_signature_using_third_party` (with a 32-byte message), and asserts the signature verifies under the reference implementation.

## Proof of Concept
1. In `tests/protocol.rs`, add a test that forces a derivation path known to produce a derived key with odd y (or iterate until one is found).
2. Call `Bip340SignatureProtocolExecution::new_with_taproot` with a 32-byte message and a non-empty taproot tree root.
3. Generate and combine shares to produce a signature.
4. Call `verify_taproot_signature_using_third_party` from `test_utils/src/lib.rs` with the pre-taproot SEC1 public key, the signature, the 32-byte message, and the taproot hash.
5. Observe: `verify_taproot_signature_using_third_party` returns `false` when the derived key has odd y, confirming the mismatch.
