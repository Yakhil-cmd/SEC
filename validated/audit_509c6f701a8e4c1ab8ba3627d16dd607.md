### Title
XCC Precompile Panics with `ERR_MISSING_WNEAR_ADDRESS` When `factory_set_wnear_address` Has Not Been Called - (File: engine-precompiles/src/xcc.rs)

### Summary

The Cross-Contract Call (XCC) precompile (`CrossContractCall`) unconditionally calls `state::get_wnear_address()` whenever a user submits an EVM transaction that requires NEAR payment (i.e., when `required_near != ZERO_YOCTO`). If the engine owner has not yet called `factory_set_wnear_address`, the storage key is absent and `get_wnear_address` **panics** with `ERR_MISSING_WNEAR_ADDRESS`. This panic propagates as a NEAR contract panic, reverting the entire transaction and permanently bricking any EVM transaction that needs to pay NEAR via the XCC precompile until the admin sets the address. The analog to M-27 is: just as tokens could be minted before properties were set (leaving them in an unrenderable limbo), EVM users can submit XCC transactions before the required wNEAR address is configured, causing a hard panic rather than a graceful error.

### Finding Description

The `get_wnear_address` function in `engine-precompiles/src/xcc.rs` reads the wNEAR ERC-20 address from storage. If the key is absent (i.e., `factory_set_wnear_address` was never called), it calls `panic!("{ERR_MISSING_WNEAR_ADDRESS}")`:

```rust
pub fn get_wnear_address<I: IO>(io: &I) -> Address {
    let key = storage::bytes_to_key(KeyPrefix::CrossContractCall, WNEAR_KEY);
    io.read_storage(&key).map_or_else(
        || panic!("{ERR_MISSING_WNEAR_ADDRESS}"),   // <-- unconditional panic
        |bytes| Address::try_from_slice(&bytes.to_vec()).expect(ERR_CORRUPTED_STORAGE),
    )
}

The `factory_set_wnear_address` function is the only way to set this storage key, and it is a separate admin-only call that must be made after deployment:

```rust

    env: &E,
) -> Result<(), ContractError> {

``` [5](#0-4) 

There is no guard in the XCC precompile or in `handle_precompile_promise` that checks whether the wNEAR address has been set before attempting to use it. The `factory_update` (router bytecode upload) and `factory_set_wnear_address` are two independent admin calls with no enforced ordering or co-dependency check.

### Impact Explanation

**High — Temporary freezing of funds.**

Any EVM user who holds wNEAR ERC-20 tokens and attempts to use the XCC precompile (address `0x516cded1d16af10cad47d6d49128e2eb7d27b372`) before `factory_set_wnear_address` has been called will have their EVM transaction panic-reverted. Because the panic occurs inside the NEAR contract execution (not as an EVM revert), the entire NEAR receipt fails. The user's wNEAR tokens that were approved for the precompile's `transferFrom` call are not consumed (the EVM state is rolled back), but the user's NEAR gas is burned and the XCC call cannot succeed until the admin acts. If the admin delays or forgets to call `factory_set_wnear_address` (analogous to the M-27 scenario where properties were never set), all XCC-dependent operations are blocked for all users for that duration. Any wNEAR tokens that a user has already approved to the XCC precompile address remain locked in an unusable state until the admin configures the address.

### Likelihood Explanation

**Medium.** The `factory_update` (router bytecode) and `factory_set_wnear_address` are two separate admin transactions. An operator who uploads the XCC router bytecode via `factory_update` but forgets or delays calling `factory_set_wnear_address` creates a window during which any user attempting XCC will trigger the panic. This is directly analogous to the M-27 scenario: the system accepts user interactions (XCC precompile calls) before a required precondition (wNEAR address registration) is satisfied. The test setup in `engine-tests/src/tests/xcc.rs` confirms this ordering requirement exists and must be manually respected:

```rust
let result = aurora
    .factory_set_wnear_address(wnear_erc20.0.address)
    .transact()
    .await?;
assert!(result.is_success());
``` [6](#0-5) 

### Recommendation

Add a guard in `get_wnear_address` (or at the call sites in `CrossContractCall::run_with_handle` and `handle_precompile_promise`) that returns a graceful `ExitError` or `PrecompileFailure::Revert` instead of panicking when the wNEAR address is not set. Alternatively, enforce that `factory_set_wnear_address` must be called before `factory_update` can succeed, or check the wNEAR address presence at the start of `run_with_handle` and return a descriptive revert before any state changes occur.

### Proof of Concept

1. Deploy the Aurora Engine contract.
2. Call `factory_update` with valid XCC router bytecode (this succeeds).
3. Do **not** call `factory_set_wnear_address`.
4. As any EVM user, submit a transaction to the XCC precompile address (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a `CrossContractCallArgs::Eager` payload targeting any NEAR contract.
5. Since the user has no existing router account, `required_near = attached_near + STORAGE_AMOUNT > 0`.
6. The precompile reaches `state::get_wnear_address(&self.io)` at line 193 of `engine-precompiles/src/xcc.rs`.
7. Storage key `WNEAR_KEY` is absent → `panic!("ERR_MISSING_WNEAR_ADDRESS")`.
8. The NEAR receipt fails with a contract panic. The user's gas is consumed, the XCC call is permanently blocked until the admin acts. [7](#0-6) [8](#0-7) [9](#0-8)
