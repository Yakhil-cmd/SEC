### Title
Signed/Unsigned Cast Without Range Check Corrupts `delta_gas` Accounting - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, two `u64` values are cast to `i64` without checking whether they exceed `i64::MAX`. When either value wraps to a negative `i64`, the `delta_gas` sign comparison silently produces the wrong result, causing the native-resource gas adjustment to be skipped. This is a direct structural analog to the reported `uint256(-amount)` overflow: an unsigned value is reinterpreted as signed without a guard, producing a large-magnitude value of the wrong sign.

### Finding Description

In `compute_gas_refund` (the shared refund path for both ZK and Ethereum-mode transactions), the `delta_gas` adjustment is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs, line 72
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

Both operands are `u64`. Rust's `as i64` cast is a bitwise reinterpretation: any value above `i64::MAX` (≈ 9.22 × 10¹⁸) wraps to a large negative `i64`. No range check is performed before either cast.

**Overflow path A — `native_used / native_per_gas` wraps:**

`native_used` is bounded by `full_native_limit = gas_limit.saturating_mul(native_per_gas)`, so `native_used / native_per_gas ≤ gas_limit`. If `gas_limit > i64::MAX`, the quotient exceeds `i64::MAX` and wraps to a large negative `i64`. The subtraction then yields a very negative `delta_gas`; the `delta_gas > 0` guard is false; `gas_used` is not incremented. The user avoids paying for native resource consumption that was actually incurred.

**Overflow path B — `gas_used` wraps:**

`gas_used` is also bounded by `gas_limit`. If `gas_limit > i64::MAX`, `gas_used as i64` can wrap to a negative value. The subtraction then yields a very large positive `delta_gas`; `gas_used += delta_gas as u64` overflows `u64`; the subsequent `gas_limit - gas_used` subtraction underflows, producing a huge `total_gas_refund` that fails the `require_internal!` guard and returns an `InternalError`, aborting the transaction. [1](#0-0) 

The documentation confirms the intended semantics: `deltaGas := (nativeUsed / nativePerGas) - gasUsed` is supposed to be an unsigned comparison, but the implementation uses signed arithmetic. [2](#0-1) 

### Impact Explanation

**Path A (resource accounting bug):** A user who sets `gas_limit > i64::MAX` causes `delta_gas` to wrap negative. The native-resource gas surcharge is silently skipped. The user is refunded more gas than they should be, effectively underpaying for the proving cost of their transaction. The operator and protocol absorb the shortfall.

**Path B (transaction abort / DoS):** The same large `gas_limit` causes `gas_used` to wrap, `delta_gas` to become a huge positive, `gas_used` to overflow, and the refund subtraction to underflow. The `require_internal!` check fires and the function returns `InternalError`, aborting the transaction with an internal error rather than a normal validation failure.

Both paths affect the ZKsync OS state-transition / resource-accounting layer, which is within the Immunefi scope.

### Likelihood Explanation

The trigger condition is `gas_limit > i64::MAX ≈ 9.22 × 10¹⁸`. The `gas_limit` field is a `u64` with no protocol-enforced upper bound in the validation code. [3](#0-2) 

A user must hold enough balance to prepay `gas_limit × gas_price`. At `gas_price = 1 wei`, the required balance is ≈ 9.2 ETH — feasible for a motivated attacker. No privileged role is required; any L2 transaction sender can set an arbitrary `gas_limit` up to `u64::MAX`. Likelihood is **low** due to the extreme gas limit required, but the entry path is fully unprivileged.

### Recommendation

Replace the unchecked `as i64` casts with explicit range-checked arithmetic. Compute `delta_gas` entirely in `u64` and compare unsigned:

```rust
let native_gas_equivalent = native_used / native_per_gas;
if native_gas_equivalent > gas_used {
    // saturating to avoid overflow on gas_used accumulation
    gas_used = gas_used.saturating_add(native_gas_equivalent - gas_used);
}
```

This eliminates the signed cast entirely and matches the documented semantics (`deltaGas := nativeUsed/nativePerGas − gasUsed`, applied only when positive).

### Proof of Concept

1. Construct a ZK-mode L2 transaction with `gas_limit = i64::MAX + 1` (= `0x8000_0000_0000_0000u64`) and `gas_price = 1 wei`. Fund the sender with ≥ `i64::MAX + 1` wei.
2. Submit the transaction. Execution consumes some native resource, so `native_used > 0` and `native_used / native_per_gas` is in the range `(i64::MAX, gas_limit]`.
3. At line 72 of `refund_calculation.rs`, `(native_used / native_per_gas) as i64` wraps to a large negative value.
4. `delta_gas` is very negative; `delta_gas > 0` is false.
5. `gas_used` is not incremented by the native surcharge.
6. The user receives a refund larger than correct, paying less than the actual proving cost. [4](#0-3)
