### Title
Unbounded `NearPromise::And` in `CrossContractCallArgs::Delayed` Allows Permanent Freezing of NEAR Funds via Gas Exhaustion in `execute_scheduled` - (File: `engine-precompiles/src/xcc.rs`)

---

### Summary

The `Delayed` variant of the XCC precompile attaches a fixed `ROUTER_SCHEDULE = 5 TGas` to the `schedule` call regardless of the complexity of the stored promise. A user can craft a `Delayed` XCC call with a `NearPromise::And` vector containing enough promises that `execute_scheduled` can never complete within NEAR's 300 TGas per-transaction limit. Any NEAR sent to the router account (unwrapped from the user's wNEAR) is permanently frozen.

---

### Finding Description

**Phase 1 — cheap commit (`schedule`):**

In `engine-precompiles/src/xcc.rs`, the `Eager` path correctly charges NEAR gas proportional to the number of callbacks: [1](#0-0) 

```rust
let callback_count = call.promise_count().checked_sub(1)...;
let router_exec_cost = costs::ROUTER_EXEC_BASE
    + NearGas::new(callback_count * costs::ROUTER_EXEC_PER_CALLBACK.as_u64());
```

The `Delayed` path, however, attaches a **fixed** `ROUTER_SCHEDULE = 5 TGas` regardless of promise count: [2](#0-1) 

```rust
CrossContractCallArgs::Delayed(call) => {
    let promise = PromiseCreateArgs {
        attached_gas: costs::ROUTER_SCHEDULE, // always 5 TGas
        ...
    };
}
```

The constants confirm the asymmetry: [3](#0-2) 

```
ROUTER_EXEC_BASE:         7 TGas
ROUTER_EXEC_PER_CALLBACK: 12 TGas
ROUTER_SCHEDULE:          5 TGas  (fixed, no per-promise component)
```

The `schedule` function in the router simply serialises and stores the promise: [4](#0-3) 

**Phase 2 — expensive execute (`execute_scheduled`):**

`execute_scheduled` retrieves the stored promise and calls `recursive_promise_create`, which iterates over every element of `NearPromise::And` with no upper-bound check: [5](#0-4) 

```rust
NearPromise::And(promises) => {
    let indices: Vec<PromiseIndex> = promises
        .iter()
        .map(Self::recursive_promise_create)  // one env::promise_create per element
        .collect();
    env::promise_and(&indices)
}
```

Each `env::promise_create` host call consumes NEAR gas. With `ROUTER_EXEC_PER_CALLBACK = 12 TGas` per callback as the measured upper bound, only ~25 callbacks exhaust the 300 TGas NEAR transaction limit. A minimal `PromiseCreateArgs` serialises to ~50–100 bytes; 25 such entries occupy ~2.5 KB, well within the 5 TGas storage budget of `schedule`.

**NEAR atomicity does not save the funds.** If `execute_scheduled` panics (out-of-gas), NEAR rolls back state, so the promise remains in `scheduled_promises`. But if the tree is sized to permanently exceed 300 TGas, no future call to `execute_scheduled` can ever succeed, and the NEAR already sent to the router account is irrecoverable.

**NEAR transfer path that locks funds:** [6](#0-5) 

When `total_near() > 0`, the engine creates a promise chain `withdraw_wnear_to_router → schedule`. The `withdraw_wnear_to_router` step unwraps the user's wNEAR and deposits real NEAR into the router sub
