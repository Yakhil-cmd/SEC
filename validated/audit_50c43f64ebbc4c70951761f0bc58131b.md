Audit Report

## Title
Unprivileged Caller Can Repeatedly Reset SNS Swap Periodic Task Timer, Delaying Lifecycle Transitions — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `reset_timers` `#[update]` endpoint in the SNS swap canister carries no caller authorization check. Any unprivileged ingress sender can invoke it to clear and restart the 60-second interval timer that drives all automatic swap lifecycle transitions (`try_open`, `try_commit`, `try_abort`, `try_auto_finalize`). The only protection is a 600-second cooldown, allowing an attacker to impose up to 60 seconds of delay per invocation and accumulate up to ~2.4 hours of lifecycle delay per day, keeping participants' ICP locked beyond the protocol's time guarantees.

## Finding Description
`reset_timers` at `rs/sns/swap/canister/canister.rs:348–365` is a plain `#[update]` with no `caller()` guard:

```rust
#[update]
fn reset_timers(_request: ResetTimersRequest) -> ResetTimersResponse {
    let reset_timers_cool_down_interval_seconds = RESET_TIMERS_COOL_DOWN_INTERVAL.as_secs();

    if let Some(timers) = swap_mut().timers
        && let Some(last_reset_timestamp_seconds) = timers.last_reset_timestamp_seconds
        && now_seconds().saturating_sub(last_reset_timestamp_seconds)
            < reset_timers_cool_down_interval_seconds
    {
        panic!(...);
    }

    init_timers();
    ResetTimersResponse {}
}
``` [1](#0-0) 

`RESET_TIMERS_COOL_DOWN_INTERVAL` is 600 seconds and `RUN_PERIODIC_TASKS_INTERVAL` is 60 seconds: [2](#0-1) 

`init_timers()` clears the existing interval timer via `ic_cdk_timers::clear_timer` and schedules a fresh 60-second interval timer for `run_periodic_tasks`: [3](#0-2) 

`run_periodic_tasks` is the sole driver of the swap's automatic state machine — `try_open`, `try_commit`, `try_abort`, and `try_auto_finalize`: [4](#0-3) 

By calling `reset_timers` just before the timer fires, an attacker delays the next `run_periodic_tasks` execution by up to 60 seconds. With a 600-second cooldown, this yields 144 resets/day × 60s = 8,640s (~2.4 hours) of maximum cumulative lifecycle delay per day.

The cooldown guard uses a three-way `&&` short-circuit: if `swap_mut().timers` is `None` **or** `timers.last_reset_timestamp_seconds` is `None`, the entire cooldown check is bypassed and the reset proceeds unconditionally. In practice, `init_timers()` is called at `canister_init` and `post_upgrade`, so both fields are set after initialization; however, this structural bypass remains a latent weakness. [5](#0-4) 

`can_abort` and `can_commit` both depend on `swap_due(now_seconds)`, which is evaluated at the time `run_periodic_tasks` fires: [6](#0-5) [7](#0-6) 

## Impact Explanation
A motivated attacker calling `reset_timers` every 600 seconds can delay:

- **Auto-abort**: A swap that has reached its due date without sufficient participation is not aborted on schedule. Participants' ICP remains locked in the canister beyond the protocol's time guarantees, delaying refunds by up to ~2.4 hours per day of sustained attack.
- **Auto-commit / auto-finalize**: SNS neuron creation and ICP disbursement to the SNS governance canister are delayed.
- **Auto-open**: The swap remains in `ADOPTED` state longer than intended, blocking participation.

This constitutes a **significant SNS security impact with concrete user harm**: participants' ICP is locked beyond the protocol's stated time guarantees, and the SNS decentralization process is disrupted. This matches the High bounty impact category: *"Significant SNS or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
The entry path is a plain ingress update call requiring no special role, no elevated cycles beyond the standard call fee, and no prior state. The cooldown timestamp is publicly observable via the `get_timers` query endpoint, making it trivial to time calls precisely. The attack is fully scriptable by any external principal. [8](#0-7) 

## Recommendation
Add a caller authorization check to `reset_timers` restricting it to the NNS governance canister, the swap canister's controllers, or the SNS root canister — consistent with how `open` is gated to NNS governance only. The SNS governance and root canisters expose the identical unauthenticated `reset_timers` pattern and should be reviewed and fixed similarly. [9](#0-8) [10](#0-9) 

## Proof of Concept
The existing integration test `try_reset_timers` in `rs/sns/integration_tests/src/timers.rs` already demonstrates that any arbitrary principal (no special role) can successfully call `reset_timers` on the swap canister: [11](#0-10) [12](#0-11) 

A minimal exploit test extending `test_swap_reset_timers` would:
1. Advance time past `swap_due_timestamp_seconds` so `can_abort` returns true.
2. Call `reset_timers` from an unprivileged principal every 600 seconds.
3. Assert that `try_abort` has not fired (lifecycle remains `Open`) for the duration of the attack window, confirming the delay.

fn init_timers() {
    let last_reset_timestamp_seconds = Some(now_seconds());
    let requires_periodic_tasks = swap().requires_periodic_tasks();

    swap_mut().timers.replace(Timers {
        requires_periodic_tasks: Some(requires_periodic_tasks),
        last_reset_timestamp_seconds,
        ..Default::default()
    });

    if requires_periodic_tasks {
        let new_timer_id =
            ic_cdk_timers::set_timer_interval(RUN_PERIODIC_TASKS_INTERVAL, async || {
                run_periodic_tasks().await
            });
        TIMER_ID.with(|saved_timer_id| {
            let mut saved_timer_id = saved_timer_id.borrow_mut();
            if let Some(saved_timer_id) = *saved_timer_id {
                ic_cdk_timers::clear_timer(saved_timer_id);
            }
            saved_timer_id.replace(new_timer_id);
        });
    } else {
        log!(
            INFO,
            "Periodic tasks are not required for this Swap anymore."
        );
    }
}

fn reset_timers(_request: ResetTimersRequest) -> ResetTimersResponse {
    let reset_timers_cool_down_interval_seconds = RESET_TIMERS_COOL_DOWN_INTERVAL.as_secs();

    STATE.with(|state| {
        let state = state.borrow();
        if let Some(timers) = state.timers
            && let Some(last_reset_timestamp_seconds) = timers.last_reset_timestamp_seconds {
                assert!(
                    now_seconds().saturating_sub(last_reset_timestamp_seconds)
                        >= reset_timers_cool_down_interval_seconds,
                    "Reset has already been called within the past {reset_timers_cool_down_interval_seconds:?} seconds"
                );
