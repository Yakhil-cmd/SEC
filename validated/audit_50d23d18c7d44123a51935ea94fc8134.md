### Title
O(n) `iter_skip_n` in `ListVec` Causes Uncharged Proving-Work Growth in `EventsStorage::events_in_transaction_ref_iter` — (File: `zk_ee/src/memory/stack_implementations/skip_list_stack.rs`, `zk_ee/src/common_structs/events_storage.rs`)

---

### Summary

The `ListVec`-backed skip-list stack used in proving mode lacks an optimized `iter_skip_n`, falling back to an O(n) traversal. `EventsStorage::events_in_transaction_ref_iter` calls `iter_skip_n(self.list_len_at_tx_start)` to skip all events from prior transactions. Because `list_len_at_tx_start` grows monotonically across every transaction in a block, the skip cost grows linearly with the cumulative event count — yet this traversal work is never charged to the current transaction's native (proving) resources.

---

### Finding Description

**Root cause — O(n) skip in `ListVec`:**

The `Stack` trait provides a default `iter_skip_n` that calls `self.iter().skip(n)`, which is O(n): [1](#0-0) 

`VecStack` overrides this with an O(1) slice-index: [2](#0-1) 

`ListVec` (the skip-list stack used in proving mode) does **not** override it. The codebase itself acknowledges this with a TODO: [3](#0-2) 

**Growing skip offset in `events_in_transaction_ref_iter`:**

`EventsStorage` stores all events for the entire block in a single `HistoryList`. At the start of each new transaction, `list_len_at_tx_start` is snapshotted to the current total length: [4](#0-3) 

`events_in_transaction_ref_iter` then calls `iter_skip_n(self.list_len_at_tx_start)` to yield only the current transaction's events: [5](#0-4) 

`HistoryList::iter_skip_n` delegates directly to the underlying stack's `iter_skip_n`: [6](#0-5) 

In proving mode the underlying stack is `ListVec`, so the skip is O(`list_len_at_tx_start`) = O(total events from all prior transactions in the block).

**No resource charge for the skip work:**

The skip traversal happens at the system I/O layer, outside any EVM gas or native-resource accounting. The transaction's native resource budget is not debited for the O(n) pointer-chasing work the prover must perform.

---

### Impact Explanation

In proving mode (RISC-V / Airbender), every instruction executed by the prover consumes native (proving) resources. The O(n) skip in `iter_skip_n` translates directly into O(n) RISC-V instructions. Because this cost is not reflected in the transaction's native resource consumption, the prover performs more work than the resource accounting predicts. Across a full block, the cumulative uncharged proving work is O(E × T), where E is the average events per transaction and T is the transaction count. This can cause:

1. **Proving resource exhaustion** — the prover runs out of native resources before completing the block, making the block unprovable.
2. **Forward/proving divergence** — forward execution (using `VecStack`, O(1) skip) succeeds, but proving (using `ListVec`, O(n) skip) fails or exceeds its budget, creating a valid-execution-unprovability scenario.

The block-level log limit (`block_logs_used` checked in `check_for_block_limits`) bounds the absolute maximum, but within that limit an attacker can maximize the imbalance. [7](#0-6) 

---

### Likelihood Explanation

An unprivileged transaction sender can emit events cheaply (LOG0 costs 375 gas + 8 gas/byte of data). By packing early transactions with many LOG opcodes up to the block log limit, the attacker maximizes `list_len_at_tx_start` for every subsequent transaction in the same block. No privileged access, oracle manipulation, or external dependency is required. The attacker pays only standard EVM gas; the extra proving cost is borne entirely by the prover.

---

### Recommendation

1. **Implement an optimized `iter_skip_n` for `ListVec`** that uses node-level arithmetic to jump directly to the correct node and offset in O(nodes_skipped / N) time, as the existing TODO comment already requests: [3](#0-2) 

2. **Charge native resources for the skip work** in `events_in_transaction_ref_iter`, or restructure `EventsStorage` to maintain a direct pointer/index to the start of the current transaction's events so that no skip traversal is needed at all (analogous to how `LogsStorage` uses a running `pubdata_used_by_committed_logs` counter to avoid re-iteration). [8](#0-7) 

---

### Proof of Concept

1. Attacker submits transaction T1 that calls a contract emitting the maximum allowed LOG events per block (e.g., 16 384 LOG1 events, each with 1 topic and minimal data).
2. Attacker submits transaction T2 (any valid transaction) in the same block.
3. During proving of T2, `events_in_transaction_ref_iter` is called. `list_len_at_tx_start` = 16 384. `ListVec::iter_skip_n(16384)` traverses 16 384 list nodes via the O(n) default path.
4. This traversal is not charged to T2's native resource budget.
5. The prover's actual native consumption for T2 exceeds its budgeted native cost, potentially exhausting the block's native resource limit or causing the proof to fail — while forward execution of T2 succeeds (VecStack skip is O(1)). [5](#0-4) [1](#0-0) [9](#0-8)
