### Title
Silent Precompile Failure After Burn Causes Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

`withdrawToNear` burns the caller's ERC-20 tokens **before** invoking the `ExitToNear` precompile, and **never checks the return value** of the low-level `call`. When `recipient` is empty (or otherwise invalid), the precompile rejects the input and returns failure, but the Solidity function does not revert. The burn is committed, no NEAR-side credit is issued, and the tokens are permanently destroyed.

### Finding Description

**Step 1 — Solidity entry point (`EvmErc20.sol`, lines 53–63)**

`withdrawToNear` is `external` with no access control. Any token holder can call it with `bytes('')`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← burn committed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;  // = 33 when recipient is empty

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no require(res == 1)
    }
}

**Step 5 — Precompile returns `ExitError`; EVM `call` returns 0; Solidity ignores it**

The precompile error propagates as a failed inner `call` (return value 0). Because `res` is never tested, `withdrawToNear` returns normally. The `_burn` from Step 1 is already committed and irreversible.

### Impact Explanation

The caller's ERC-20 tokens are permanently destroyed with no corresponding NEP-141 credit on the NEAR side. There is no recovery path. This constitutes **Critical — Permanent freezing (destruction) of funds**.

### Likelihood Explanation

The path requires no privilege. Any token holder can trigger it accidentally (e.g., a frontend bug passing an empty byte array) or deliberately. The function is `external` with no pause gate or recipient length guard.

### Recommendation

1. Add a recipient length guard at the Solidity level before burning:
   ```solidity
   require(recipient.length >= 2, "ERR_EMPTY_RECIPIENT");
   ```
2. Check the precompile call return value and revert on failure:
   ```solidity
   assembly {
       let res := call(...)
       if iszero(res) { revert(0, 0) }
   }
   ```
3. Apply the same fix to `EvmErc20V2.sol`, which has an identical pattern. [1](#0-0) 

### Proof of Concept

```
1. Deploy EvmErc20 (or use an existing mirror token).
2. Mint `amount` tokens to attacker address.
3. Call withdrawToNear(bytes(''), amount) from attacker address.
4. Observe: ERC-20 balance of attacker = 0 (burned).
5. Observe: No NEP-141 ft_transfer promise was scheduled (precompile rejected silently).
6. Tokens are permanently lost.
```

Fuzz extension: supply `recipient` values of length 0, 1, and non-UTF-8 bytes (e.g., `[0xc2]`). All should revert at the Solidity level; currently none do.
