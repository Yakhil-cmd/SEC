### Title
ERC-20 Tokens Permanently Burned Without Refund on Failed Cross-Chain Exit — (`engine-precompiles/src/native.rs`)

### Summary

The `ExitToEthereum` precompile burns a user's ERC-20 tokens inside the EVM before dispatching a NEAR cross-chain promise to the eth-connector's `withdraw` method. No callback is ever attached to that promise. If the `withdraw` promise fails for any reason (e.g., eth-connector paused, gas exhaustion, accounting error), the burned tokens are permanently unrecoverable. The same structural flaw exists in `ExitToNear` when the `error_refund` compile-time feature is disabled (the default build).

---

### Finding Description

**Step 1 — ERC-20 burn happens first, inside the EVM.**

`EvmErc20.withdrawToEthereum` (and `EvmErc20V2.withdrawToEthereum`) unconditionally burns the caller's tokens before invoking the precompile:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← tokens destroyed here, atomically
    // ... then calls exitToEthereum precompile
}
``` [1](#0-0) [2](#0-1) 

**Step 2 — `ExitToEthereum` precompile creates a fire-and-forget promise.**

The precompile builds a single `PromiseArgs::Create(withdraw_promise)` with **no callback attached**. There is no `error_refund` feature guard, no `exit_to_ethereum_precompile_callback`, and no refund path — unconditionally:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
// ↑ No callback. No refund. Promise result is never inspected.
``` [3](#0-2) 

**Step 3 — `ExitToNear` has the same flaw when `error_refund` is disabled.**

`ExitToNear::run()` conditionally populates a `refund` field only when the `error_refund` feature is compiled in. Without it, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the promise degrades to a bare `PromiseArgs::Create` with no callback:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← no refund data
    transfer_near: transfer_near_args,
};

let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback
} else { ... };
``` [4](#0-3) 

The `error_refund` feature is opt-in and is not part of the default build: [5](#0-4) 

**Step 4 — The callback that handles failure exists only for `ExitToNear` with the feature enabled.**

`exit_to_near_precompile_callback` re-mints burned ERC-20 tokens or refunds ETH only when the promise result is a failure **and** `args.refund` is `Some`. Without `error_refund`, `args.refund` is always `None`, so the else-branch is never taken:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent drop; no refund
};
``` [6](#0-5) 

The test suite explicitly acknowledges the permanent loss when the feature is off:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

---

### Impact Explanation

When the cross-chain `withdraw` (for `ExitToEthereum`) or `ft_transfer` (for `ExitToNear`) promise fails after the EVM burn has already committed:

- The user's ERC-20 tokens are gone from the EVM state.
- No NEP-141 tokens are transferred to the destination.
- No ETH is released on Ethereum.
- There is no callback, no re-mint, and no recovery path.

This is **permanent freezing / permanent loss of funds** — matching the Critical impact tier.

---

### Likelihood Explanation

Realistic failure triggers for the `withdraw` promise in `ExitToEthereum`:

1. **Eth-connector paused**: The eth-connector supports pausability (demonstrated by `test_withdraw_from_near_pausability`). If `withdraw` is paused between the EVM burn and the promise execution, the promise fails silently.
2. **Gas exhaustion**: `WITHDRAWAL_GAS` is set to `100_000_000_000_000` TGas with a `TODO` comment acknowledging it may be wrong. Underestimation causes the promise to fail.
3. **Accounting discrepancy**: Any state where Aurora's NEP-141 balance is less than the burn amount causes `withdraw` to revert.

For `ExitToNear` without `error_refund`, the test `test_exit_to_near_refund` demonstrates a concrete trigger: sending to an unregistered NEAR account causes `ft_transfer` to fail, and the ERC-20 tokens are permanently lost. [8](#0-7) [9](#0-8) 

---

### Recommendation

1. **`ExitToEthereum`**: Attach a callback promise (analogous to `exit_to_near_precompile_callback`) that inspects the `withdraw` promise result. On failure, re-mint the burned ERC-20 tokens to the original caller (or a caller-supplied refund address).

2. **`ExitToNear`**: Promote the `error_refund` feature to always-on, or restructure the refund path so it does not depend on a compile-time feature flag. The current design silently drops funds when the feature is absent.

3. **Gas constants**: Replace the `TODO` gas constants with empirically validated values to reduce the risk of promise failure due to gas exhaustion.

---

### Proof of Concept

The existing test suite already demonstrates the fund-loss path for `ExitToNear`:

```
test_exit_to_near_refund (engine-tests/src/tests/erc20_connector.rs)

Without error_refund feature:
  - User bridges FT_TRANSFER_AMOUNT NEP-141 tokens → ERC-20 on Aurora
  - User calls withdrawToNear("unregistered.near", FT_EXIT_AMOUNT)
  - ERC-20 burns FT_EXIT_AMOUNT tokens
  - ft_transfer promise fails (unregistered recipient)
  - No callback fires
  - ERC-20 balance = FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT  ← tokens permanently lost
  - NEP-141 balance of Aurora = FT_TRANSFER_AMOUNT        ← NEP-141 never moved
```

For `ExitToEthereum`, the analogous scenario is: pause the eth-connector's `withdraw` method, then call `withdrawToEthereum`. The ERC-20 burn succeeds inside the EVM, the `withdraw` promise fails silently, and the user loses their tokens with no recourse. [1](#0-0) [3](#0-2) [4](#0-3) [6](#0-5)
