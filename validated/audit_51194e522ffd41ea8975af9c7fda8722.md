### Title
Unverified NEAR Promise Result in `ExitToEthereum` and `ExitToNear` Precompiles Enables Permanent Fund Freeze — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile unconditionally, and the `ExitToNear` precompile conditionally (when the `error_refund` feature is not compiled in), create NEAR cross-contract call promises whose results are never verified. Because ERC-20 tokens are burned on the Aurora EVM side before the NEAR promise executes, any failure of the NEAR-side `withdraw` or `ft_transfer` call results in permanent, unrecoverable loss of user funds — a direct analog to the ignored `transferFrom` return value in Amp.sol.

---

### Finding Description

**`ExitToEthereum` — unconditional, no callback ever attached:**

In `ExitToEthereum::run`, after burning ERC-20 tokens on the EVM side, a NEAR `withdraw` promise is constructed and encoded as `PromiseArgs::Create` with no callback: [1](#0-0) 

The result of the `withdraw` call on the NEP-141 connector is never observed. There is no `PromiseArgs::Callback` wrapper, no `exit_to_ethereum_precompile_callback`, and no refund path — for any code path, unconditionally.

**`ExitToNear` — conditional on `error_refund` feature flag:**

In `ExitToNear::run`, the callback is only attached when `callback_args != ExitToNearPrecompileCallbackArgs::default()`: [2](#0-1) 

When the `error_refund` feature is not compiled in, `refund` is forced to `None`: [3](#0-2) 

For all non-wNEAR ERC-20 exits (where `transfer_near_args` is also `None`), `callback_args` equals `default()`, so the promise degrades to `PromiseArgs::Create` with no callback — the `ft_transfer` result is silently discarded.

In both cases, the ERC-20 burn is committed atomically within the EVM execution. The NEAR promise is scheduled asynchronously. If the NEAR-side call fails, the EVM state is already finalized with no rollback or refund mechanism.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

If the NEAR-side `withdraw` (for `ExitToEthereum`) or `ft_transfer` (for `ExitToNear` without `error_refund`) fails for any reason, the user's ERC-20 tokens are permanently destroyed on the Aurora EVM with no corresponding NEP-141 tokens delivered and no recovery path. The funds are irrecoverably lost.

---

### Likelihood Explanation

**Medium for `ExitToEthereum`** (unconditional): Any condition that causes the NEP-141 connector's `withdraw` to fail — including an accounting mismatch between Aurora EVM balances and the connector's NEP-141 balance, a connector contract panic, or a NEAR runtime-level failure — silently destroys user funds. This is reachable by any token holder calling the exit precompile.

**Low-to-Medium for `ExitToNear`** (feature-flag dependent): Depends on whether `error_refund` is compiled into the production binary. If it is not, any `ft_transfer` failure (e.g., recipient account not registered for the NEP-141 token, connector balance mismatch) causes permanent fund loss.

---

### Recommendation

**For `ExitToEthereum`:** Introduce a callback promise (analogous to `exit_to_near_precompile_callback`) that checks the `withdraw` result and refunds the ERC-20 tokens to the sender if the NEAR call failed.

**For `ExitToNear`:** Ensure the `error_refund` feature is always compiled into production builds, or make the callback unconditional rather than feature-gated, so that `ft_transfer` failures always trigger a refund.

The fix mirrors the Amp.sol recommendation: validate the success of the transfer before treating the operation as complete.

---

### Proof of Concept

1. User holds ERC-20 tokens on Aurora that map to a NEP-141 token on NEAR.
2. User calls the `ExitToEthereum` precompile (flag `0x1`) with a valid Ethereum recipient address and an amount equal to their full balance.
3. The EVM executes: the ERC-20 contract burns the tokens (balance set to zero). The precompile returns a log encoding `PromiseArgs::Create(withdraw_promise)` with no callback.
4. Aurora Engine schedules the NEAR `withdraw` promise on the NEP-141 connector.
5. The NEP-141 connector's `withdraw` call fails (e.g., due to a pre-existing accounting discrepancy where the connector holds fewer NEP-141 tokens than the EVM balance reflects).
6. The NEAR promise fails silently. No callback fires. No refund is issued.
7. The user's ERC-20 tokens are gone from Aurora, and no ETH withdrawal is initiated on Ethereum. Funds are permanently frozen. [4](#0-3) [5](#0-4)
