### Title
Unbounded BFS Follower Traversal in SNS Governance `cast_vote_and_cascade_follow` Exhausts Instruction Limit - (File: rs/sns/governance/src/governance.rs)

---

### Summary

The SNS governance canister's `cast_vote_and_cascade_follow` function performs a synchronous, unbounded breadth-first search (BFS) over the entire neuron follower graph with no instruction-limit guard. As the number of SNS neurons and following relationships grows without bound, a single `register_vote` or `make_proposal` call can exhaust the IC's per-message instruction limit, causing the message to be rejected and governance to stall.

---

### Finding Description

`Governance::cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs` is a **synchronous** function that traverses the follower graph via BFS:

```rust
while !induction_votes.is_empty() {
    let mut follower_neuron_ids = BTreeSet::new();
    for (current_neuron_id, current_new_vote) in &induction_votes {
        // ... fill ballot, collect followers ...
    }
    induction_votes.clear();
    for follower_neuron_id in follower_neuron_ids {
        let follower_vote = follower_neuron.vote_from_ballots_following(...);
        if follower_vote != Vote::Unspecified {
            induction_votes.insert(follower_neuron_id.to_string(), follower_vote);
        }
    }
}
```

There is **no instruction-limit check anywhere in this loop**. The loop terminates only when the follower graph is fully traversed. The number of neurons in an SNS grows without bound as more users participate, and each neuron can follow any other neuron.

This function is called synchronously from two reachable paths:

1. **`register_vote`** (line 3931) — called when any neuron holder votes on a proposal.
2. **`make_proposal`** (line 3658) — called when a neuron submits a proposal (the proposer auto-votes Yes, triggering the cascade). [1](#0-0) 

**Contrast with NNS governance**, which has an explicit soft/hard instruction limit mechanism with async self-calls to avoid exhausting the message budget:

```rust
// NNS governance (rs/nns/governance/src/voting.rs)
while !is_voting_finished {
    with_voting_state_machines_mut(|voting_state_machines| {
        self.process_machine_until_soft_limit(machine, over_soft_message_limit);
    });
    if let Err(e) = noop_self_call_if_over_instructions(
        SOFT_VOTING_INSTRUCTIONS_LIMIT,
        Some(HARD_VOTING_INSTRUCTIONS_LIMIT),
    ).await { break; }
}
``` [2](#0-1) 

The NNS version also checks `is_over_instructions_limit()` inside `continue_processing` at every iteration step: [3](#0-2) 

The SNS version has **none of these safeguards**. The `register_vote` call site is also synchronous (not `async`): [4](#0-3) 

The IC's per-message instruction limit is 40 billion instructions: [5](#0-4) 

Each BFS iteration performs `BTreeMap` lookups (O(log N)) and `BTreeSet` insertions for every follower. With a large fan-out following graph (e.g., 10,000+ neurons all following a single popular neuron), the total instruction count can exceed 40B, causing `CanisterInstructionLimitExceeded`.

---

### Impact Explanation

When the instruction limit is exceeded during `register_vote` or `make_proposal`, the entire update message is rejected by the IC execution environment. The effects are:

- **Voting is permanently blocked** for any neuron whose vote would trigger a cascade over a large follower set. The neuron cannot cast its vote, and the proposal cannot reach quorum.
- **Proposal submission is blocked** for any neuron with a large follower set, since `make_proposal` calls `cast_vote_and_cascade_follow` synchronously before returning.
- **Governance liveness is broken**: if the neuron with the largest follower set (e.g., the founding team's neuron) cannot vote, proposals may never reach the required voting threshold, permanently freezing the SNS DAO.

This is a direct analog to the YetiFinance M-04 finding: an unbounded collection (the follower graph) grows without limit, and a function that iterates over it in a single call eventually exhausts the computational resource budget.

---

### Likelihood Explanation

- Any unprivileged SNS neuron holder can trigger this by voting on a proposal or submitting one.
- The follower graph grows naturally as more users join the SNS and configure following relationships. Popular SNS DAOs (e.g., OpenChat, Kinic) already have thousands of neurons.
- No special privilege, key, or majority is required — a single `manage_neuron { RegisterVote }` call from any authorized neuron holder suffices.
- The growth is monotonic: neurons are not deleted, so the follower graph only ever grows.

---

### Recommendation

Apply the same instruction-limit-aware, async state-machine pattern used in NNS governance:

1. Make `register_vote` in SNS governance `async`.
2. Introduce a `ProposalVotingStateMachine` equivalent with soft/hard instruction limits.
3. Check `over_soft_message_limit()` (or equivalent) at each BFS iteration step and suspend remaining work to a timer job when the limit is approached.
4. Alternatively, enforce a hard cap on the maximum number of followers per neuron per topic at the `follow` call site, analogous to the `max_followees_per_function` parameter already present in SNS nervous system parameters. [6](#0-5) 

---

### Proof of Concept

1. Deploy an SNS with N neurons (N large, e.g., 50,000).
2. Have all N neurons call `follow` to follow neuron A on a given function/topic.
3. Neuron A submits a proposal (triggering `make_proposal` → `cast_vote_and_cascade_follow` with N followers in the BFS).
4. Alternatively, neuron A votes on an existing proposal (triggering `register_vote` → `cast_vote_and_cascade_follow`).
5. The BFS iterates over all N followers in a single synchronous message, consuming O(N log N) instructions.
6. With N large enough to exceed 40B instructions, the IC rejects the message with `CanisterInstructionLimitExceeded`.
7. Neuron A can never vote or submit proposals; governance is permanently impaired.

The call chain is:
- `manage_neuron` (public update endpoint) → `register_vote` → `cast_vote_and_cascade_follow` (unbounded synchronous BFS, no instruction guard). [7](#0-6) [8](#0-7)

    fn register_vote(
        &mut self,
        neuron_id: &NeuronId,
        caller: &PrincipalId,
        request: &manage_neuron::RegisterVote,
    ) -> Result<(), GovernanceError> {
        let now_seconds = self.env.now();

        let neuron = self
            .proto
            .neurons
            .get_mut(&neuron_id.to_string())
            .ok_or_else(||
                // The specified neuron is not present.
                GovernanceError::new_with_message(ErrorType::NotFound, "Neuron not found"))?;

        neuron.check_authorized(caller, NeuronPermissionType::Vote)?;
        let proposal_id = request.proposal.as_ref().ok_or_else(|| {
            GovernanceError::new_with_message(
                // InvalidCommand would probably be more apt, but that would
                // be a non-backwards compatible change.
                ErrorType::PreconditionFailed,
                "Registering of vote must include a proposal id.",
            )
        })?;

        let proposal = self.proto.proposals.get_mut(&proposal_id.id).ok_or_else(||
            // Proposal not found.
            GovernanceError::new_with_message(ErrorType::NotFound, "Can't find proposal."))?;

        let action = proposal
            .proposal
            .as_ref()
            .expect("ProposalData must have a proposal")
            .action
            .as_ref()
            .expect("Proposal must have an action");

        // Take topic-based criticality as it was defined when the proposal was made.
        let proposal_topic = proposal.topic();

        let vote = Vote::try_from(request.vote).unwrap_or(Vote::Unspecified);
        if vote == Vote::Unspecified {
            // Invalid vote specified, i.e., not yes or no.
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                "Invalid vote specified.",
            ));
        }
        let neuron_ballot = proposal.ballots.get_mut(&neuron_id.to_string()).ok_or_else(||
            // This neuron is not eligible to vote on this proposal.
            GovernanceError::new_with_message(ErrorType::NotAuthorized, "Neuron not eligible to vote on proposal."))?;
        if neuron_ballot.vote != (Vote::Unspecified as i32) {
            // Already voted.
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                "Neuron already voted on proposal.",
            ));
        }

        // Check if the proposal is still open for voting.
        let deadline = proposal.get_deadline_timestamp_seconds();
        if now_seconds > deadline {
            // Deadline has passed, so the proposal cannot be voted on
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                "Proposal deadline has passed.",
            ));
        }

        // Update ballots.
        let function_id = u64::from(action);
        let proposal_topic = if proposal_topic == Topic::Unspecified {
            None
        } else {
            Some(proposal_topic)
        };
        Governance::cast_vote_and_cascade_follow(
            proposal_id,
            neuron_id,
            vote,
            function_id,
            &self.function_followee_index,
            &self.topic_follower_index,
            &self.proto.neurons,
            now_seconds,
            &mut proposal.ballots,
            proposal_topic.unwrap_or_default(),
        );

        self.process_proposal(proposal_id.id);

        Ok(())
    }

        ballots: &mut HashMap<u64, Ballot>,
