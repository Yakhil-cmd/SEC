Audit Report

## Title
Global Pending-Queue Exhaustion by Single Unprivileged Principal — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary
`retrieve_btc_with_approval`, used directly by the ckDOGE minter's `retrieve_doge_with_approval`, enforces only a per-account in-flight concurrency guard (`MAX_CONCURRENT = 100`) that is released on function return. The global pending-queue cap (`MAX_CONCURRENT_PENDING_REQUESTS = 5000`) has no per-principal sub-limit. A single funded principal can sequentially fill all 5000 slots, causing every subsequent withdrawal from any user to return `TemporarilyUnavailable("too many pending retrieve_btc requests")`.

## Finding Description

`retrieve_doge_with_approval` delegates unconditionally to `retrieve_btc_with_approval`: [1](#0-0) 

Inside `retrieve_btc_with_approval`, Guard 1 (`retrieve_btc_guard`) is acquired per `Account` (owner + subaccount pair) and caps total simultaneous in-flight calls at `MAX_CONCURRENT = 100`: [2](#0-1) 

This guard is dropped via `Drop` as soon as the async function returns — whether the request was accepted or not: [3](#0-2) 

Guard 2 checks the global pending count against `MAX_CONCURRENT_PENDING_REQUESTS = 5000` with no per-principal accounting: [4](#0-3) [5](#0-4) 

`accept_retrieve_btc_request` simply pushes the request into `state.pending_retrieve_btc_requests` with no per-principal limit: [6](#0-5) 

The gap: once a call completes and the request is accepted, Guard 1 is released and the attacker can immediately submit another request. Guard 2 is a global counter only. Nothing prevents a single principal (using one or more subaccounts) from accumulating all 5000 slots sequentially.

By contrast, the ckETH minter's guard checks `pending_requests_count` before accepting, though even that is a global check: [7](#0-6) 

## Impact Explanation

Once the attacker holds all 5000 pending slots, every call to `retrieve_doge_with_approval` from any user returns `TemporarilyUnavailable("too many pending retrieve_btc requests")`. This is a complete withdrawal DoS on the ckDOGE chain-fusion bridge, matching the allowed ICP bounty impact: **Application/platform-level DoS with concrete user and protocol harm** — **High ($2,000–$10,000)**.

## Likelihood Explanation

- Requires no privileged access — only a funded ckDOGE account and ICRC-2 approvals.
- The attack is fully sequential: submit → wait for acceptance → repeat. No concurrency tricks needed.
- Dogecoin's slow confirmation time (~60 blocks, ~1 hour) means the pending queue drains slowly, maximizing DoS duration per attack cycle.
- Economic cost is bounded and partially recoverable (attacker receives DOGE back minus fees), making sustained attacks feasible for a motivated adversary.
- The attacker can use up to 100 subaccounts simultaneously (Guard 1 limit) to fill the queue faster.

## Recommendation

1. **Add a per-principal pending-request cap**: before accepting a new request into `pending_retrieve_btc_requests`, count how many existing pending/in-progress requests belong to `caller_account.owner` and reject if above a threshold (e.g., 10–20).
2. **Alternatively, track per-principal accepted-request counts** in `CkBtcMinterState` and enforce the limit inside `accept_retrieve_btc_request` in `rs/bitcoin/ckbtc/minter/src/state/audit.rs`.
3. The fix must account for subaccounts — the per-principal count should aggregate across all subaccounts of the same `owner` principal.

impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }

pub fn accept_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    runtime: &R,
) {
    record_event(
        EventType::AcceptedRetrieveBtcRequest(request.clone()),
        runtime,
    );
    state.pending_retrieve_btc_requests.push(request.clone());
    if let Some(account) = request.reimbursement_account {
        state
            .retrieve_btc_account_to_block_indices
            .entry(account)
            .and_modify(|entry| entry.push(request.block_index))
            .or_insert(vec![request.block_index]);
    }
    if let Some(kyt_provider) = request.kyt_provider {
        *state.owed_kyt_amount.entry(kyt_provider).or_insert(0) += state.check_fee;
    }
}
