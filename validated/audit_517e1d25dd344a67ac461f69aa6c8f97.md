### Title
Unprotected `new()` Initialization Allows Any Caller to Seize Engine Ownership Before Legitimate Deployer - (File: engine/src/contract_methods/admin.rs)

---

### Summary

The `new()` function in Aurora Engine has no caller access control. Any NEAR account can call it in the window between contract deployment and legitimate initialization. An attacker who monitors the NEAR blockchain can frontrun the initialization, setting themselves as `owner_id`. This gives the attacker full administrative control over the shared EVM execution environment, enabling permanent fund freezing and direct fund theft via malicious upgrade.

---

### Finding Description

The `new()` function at `engine/src/contract_methods/admin.rs:56-88` is the sole initialization entry point for the Aurora Engine. Its only guard is a check that the state does not already exist:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
```

There is no check on `env.predecessor_account_id()`. Any NEAR account can call this function before the legitimate deployer does.

The exported Wasm entry point in `engine/src/lib.rs:77-83` makes this callable by any external account:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn new() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::admin::new(io, &env)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
```

The `NewCallArgs` accepted by `new()` includes `owner_id`, which the attacker fully controls. Once the attacker's `new()` call succeeds, the legitimate deployer's call fails with `ERR_ALREADY_INITIALIZED`, and the attacker holds `owner_id` permanently.

The test infrastructure in `engine-tests-connector/src/utils.rs:92-168` confirms that deployment and initialization are performed as separate transactions in practice, creating a concrete frontrunning window:

```rust
let engine_contract = engine.deploy(&engine_contract_bytes).await?.into_result()?;
// ... separate transaction:
let res = engine_contract
    .call("new")
    .args_borsh((chain_id, engine_contract.id(), engine_contract.id(), 1_u64))
    .max_gas()
    .transact()
    .await?;
```

**Attacker-controlled entry path:**

1. Attacker monitors the NEAR blockchain for a new Aurora Engine deployment.
2. Attacker calls `new()` with `owner_id = attacker.near` before the legitimate deployer.
3. Legitimate deployer's `new()` fails with `ERR_ALREADY_INITIALIZED`.
4. Attacker now controls `owner_id` in `EngineState`.

With `owner_id`, the attacker can:
- Call `pause_contract()` (`admin.rs:251-259`) to set `is_paused = true`, blocking all EVM execution via `require_running()` checks on every entry point.
- Call `upgrade()` (`admin.rs:169-206`) to deploy arbitrary malicious Wasm code to the engine account, enabling direct theft of all EVM balances.
- Call `set_owner()` (`admin.rs:103-121`) to transfer ownership to a different account, making recovery impossible without a NEAR-level governance intervention.

---

### Impact Explanation

**Critical.** Aurora Engine is a shared EVM execution environment for all users. The attacker gains full administrative control over this shared state. They can:

1. **Permanently freeze all user funds** by calling `pause_contract()`, which sets `is_paused = true` and causes every subsequent `submit`, `call`, `deploy_code`, `ft_on_transfer`, and `withdraw` call to fail via `require_running()`.
2. **Steal all user funds** by calling `upgrade()` to deploy malicious contract code that drains EVM balances, since the upgrade mechanism (`admin.rs:188-205`) deploys arbitrary Wasm to the engine account with no code verification.

---

### Likelihood Explanation

**Medium.** The attack requires:
- Monitoring the NEAR blockchain for a new Aurora Engine deployment (trivially automated).
- Submitting a `new()` call before the legitimate deployer in the window between deployment and initialization.

NEAR has a ~1-second block time. The window exists whenever deployment and initialization are separate transactions, which is the pattern shown in the test infrastructure and common in practice. The attack does not require any special privileges, leaked keys, or social engineering — only the ability to submit a NEAR transaction.

---

### Recommendation

Add a predecessor check to `new()` so that only the contract's own account (or a designated deployer) can initialize it:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Add: only the contract account itself may initialize
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    // ...
}
```

Alternatively, use a NEAR batch transaction to atomically combine `DeployContract` and the `new()` `FunctionCall` action, eliminating the frontrunning window entirely.

---

### Proof of Concept

1. Observe a new Aurora Engine deployment to `aurora.near` on-chain (deployment transaction visible before `new()` is called).
2. Submit a NEAR transaction calling `aurora.near::new` with `owner_id = attacker.near` and arbitrary `chain_id`.
3. Legitimate deployer's `new()` call reverts with `ERR_ALREADY_INITIALIZED`.
4. Call `aurora.near::pause_contract` as `attacker.near` — succeeds, sets `is_paused = true`.
5. All subsequent `submit`, `call`, `ft_on_transfer`, and `withdraw` calls revert via `require_running()`, permanently freezing all user funds in the EVM.

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}
