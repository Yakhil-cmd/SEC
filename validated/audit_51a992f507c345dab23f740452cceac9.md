### Title
Unauthenticated `ft_on_transfer` Enables Arbitrary ERC-20 Minting Without Real Token Transfer — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` entrypoint in Aurora Engine's connector module accepts caller-supplied `amount` and recipient parameters without verifying that the call originated from a legitimate `ft_transfer_call` flow. Any NEAR account whose account ID is registered as a NEP-141 token in Aurora's bridge can call `ft_on_transfer` directly — bypassing the actual token transfer — and mint an arbitrary quantity of ERC-20 tokens to any EVM address. An attacker who controls a registered NEP-141 token can exploit this to drain Aurora's on-chain balance of that token, stealing funds from legitimate depositors.

---

### Finding Description

**Root cause — no caller-origin enforcement in `ft_on_transfer`:** [1](#0-0) 

The function reads `predecessor_account_id` and uses it only to decide *which* minting path to take:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)   // ETH path — gated on connector account
} else {
    engine.receive_erc20_tokens(        // ERC-20 path — no further gate
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [2](#0-1) 

For the ERC-20 path, the predecessor is used as the NEP-141 token address to look up the registered ERC-20 contract, and `args.amount` (fully attacker-controlled) is minted to `args.msg` (also attacker-controlled). There is no check that the call arrived via `ft_transfer_call` on the token contract, no check that real tokens were escrowed, and no `assert_private_call()` guard.

**Minting logic that trusts the supplied amount unconditionally:** [3](#0-2) 

`receive_erc20_tokens` calls `get_erc20_from_nep141` with the predecessor as the token key, then mints `args.amount` ERC-20 tokens to the decoded recipient address. No balance snapshot or cross-check against an actual on-chain transfer is performed.

**Public NEAR entrypoint — reachable by any NEAR account:** [4](#0-3) 

**Contrast with properly guarded callbacks** — every other sensitive callback in the codebase uses `env.assert_private_call()`: [5](#0-4) [6](#0-5) 

`ft_on_transfer` has no equivalent guard.

---

### Impact Explanation

**Impact: Critical — direct theft of user funds.**

Attack scenario:

1. Attacker deploys a NEP-141 token contract (`FAKE`) on NEAR. This is permissionless.
2. Attacker calls `deploy_erc20_token` on Aurora to register `FAKE` and deploy its ERC-20 mirror. This is also permissionless.
3. Legitimate users deposit real `FAKE` tokens to Aurora via the standard `ft_transfer_call` flow. Aurora's on-chain `FAKE` balance grows to, say, 1 000 000 tokens.
4. Attacker calls `ft_on_transfer` **directly** on Aurora from the `FAKE` token account (which the attacker controls), supplying `amount = 1_000_000` and `msg = <attacker_evm_address>`. No actual token transfer occurs; Aurora mints 1 000 000 ERC-20 `FAKE` tokens to the attacker's EVM address.
5. Attacker calls the `ExitToNear` precompile with those 1 000 000 ERC-20 tokens. Aurora calls `ft_transfer` on `FAKE`, transferring 1 000 000 real `FAKE` tokens from Aurora's custodial balance to the attacker.
6. Aurora's `FAKE` balance is now 0. All legitimate depositors' ERC-20 `FAKE` tokens are unbacked — their funds are stolen.

The attacker's only precondition is controlling a NEAR account that is registered as a NEP-141 token in Aurora's bridge, which is fully self-serviceable.

---

### Likelihood Explanation

**Likelihood: High.**

- Deploying a NEP-141 token on NEAR is permissionless and costs only storage fees.
- Registering it in Aurora via `deploy_erc20_token` is also permissionless.
- The attacker does not need any privileged key, governance vote, or admin access.
- The only social-engineering step is attracting depositors to the token; this is realistic in any DeFi context (yield farming, liquidity pools, etc.).
- The exploit itself is a single direct NEAR function call with no complex sequencing.

---

### Recommendation

Enforce that `ft_on_transfer` can only be triggered as a callback from a legitimate `ft_transfer_call` on the token contract. Two complementary mitigations:

1. **Track expected incoming transfers**: Before `ft_on_transfer` is called, record the expected `(token, amount)` in storage during the `ft_transfer_call` dispatch, and verify it inside `ft_on_transfer`. Clear the record after use.
2. **Snapshot-based accounting**: Record the Aurora contract's NEP-141 balance before the transfer and verify inside `ft_on_transfer` that the balance increased by at least `args.amount` (analogous to the `pre_balance` pattern used in the referenced external report, but applied correctly with a trusted pre-snapshot).

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
