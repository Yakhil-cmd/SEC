### Title
Missing Explicit Size Limit on L2→L1 Message Data Allows Unbounded Pubdata Growth - (`system_hooks/src/call_hooks/l1_messenger.rs`)

### Summary

The `send_to_l1_inner` function in the L1 messenger system hook accepts an arbitrarily large message payload with no hard size cap. While native resources are charged proportionally to message length, the absence of an explicit maximum size bound means a sufficiently funded caller can push unbounded data into the pubdata stream and the in-memory log store, mirroring the missing `compose_msg` size check in the original ONFT report.

### Finding Description

`send_to_l1_inner` in `system_hooks/src/call_hooks/l1_messenger.rs` extracts the message as everything after the 20-byte sender address and forwards it directly to `emit_l1_message` with no length guard:

```rust
let message = &calldata[20..];
system.io.emit_l1_message(
    ExecutionEnvironmentType::NoEE,
    resources,
    &address_sender,
    message,
)?;
``` [1](#0-0) 

`emit_l1_message` in `basic_system/src/system_implementation/system/io_subsystem.rs` charges native resources that scale with `data.len()`, but imposes no hard upper bound:

```rust
let native = hashing_native_cost
    + EVENT_STORAGE_BASE_NATIVE_COST
    + EVENT_DATA_PER_BYTE_COST * (data.len() as u64);
resources.charge(&R::from_native(R::Native::from_computational(native)))?;
// ...
self.logs_storage.push_message(self.tx_number, address, data, data_hash)?;
``` [2](#0-1) 

`push_message` in `zk_ee/src/common_structs/logs_storage.rs` stores the raw bytes and accumulates pubdata without any size check:

```rust
let total_pubdata = 4 + data.len() + L2_TO_L1_LOG_SERIALIZE_SIZE;
``` [3](#0-2) 

The constant `MAX_NUMBER_OF_LOGS: u64 = 16_384` is declared in the same file but is **never checked** inside `push_message`: [4](#0-3) 

`apply_to_array_vec`, called during multiblock-batch finalization, uses a fixed-capacity `ArrayVec<Bytes32, 16384>` and will **panic** if the log count exceeds 16,384:

```rust
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.push(log.hash())   // panics when capacity is exceeded
    });
}
``` [5](#0-4) 

This is called from `post_tx_op_proving_multiblock_batch.rs` during block finalization.

### Impact Explanation

1. **Unbounded pubdata growth**: A single `sendToL1` call with a multi-megabyte payload inflates the block's pubdata by the full message length. If the resulting pubdata exceeds the DA layer's calldata or blob limit, the block cannot be committed to L1, stalling the chain.

2. **Proving-system memory exhaustion**: In the RISC-V proving target, the custom `talc` allocator operates in a fixed memory region. Storing a very large `UsizeAlignedByteBox` for the message can exhaust the prover's heap, causing a fatal runtime error and an unprovable block.

3. **Block-finalization panic**: Because `MAX_NUMBER_OF_LOGS` is never enforced in `push_message`, a block that accumulates more than 16,384 logs (user messages + L1→L2 tx logs) will cause `apply_to_array_vec` to panic, making the block unfinalizeable.

### Likelihood Explanation

The L1 messenger hook is reachable by any unprivileged user through the deployed L1Messenger system contract at `0x8008` via `sendToL1(bytes)`. The hook enforces only that the caller is `0x8008`; the message content and length are fully attacker-controlled. Native resource charging scales linearly with message size, so a well-funded attacker can craft a single transaction with a payload large enough to push pubdata past DA limits or exhaust prover memory. The 16,384-log panic path requires many transactions in one block but is structurally unguarded.

### Recommendation

1. **Add a hard message-size cap in `send_to_l1_inner`**:

```rust
const MAX_L2_TO_L1_MSG_SIZE: usize = 100_000; // tune to DA limits
if message.len() > MAX_L2_TO_L1_MSG_SIZE {
    return Ok(Err("L1 messenger failure: message too large"));
}
```

2. **Enforce `MAX_NUMBER_OF_LOGS` in `push_message`**:

```rust
if self.list.len() as u64 >= MAX_NUMBER_OF_LOGS {
    return Err(SystemError::LeafDefect(internal_error!("Too many L2->L1 logs")));
}
```

### Proof of Concept

1. Deploy or use the existing L1Messenger system contract at `0x8008`.
2. Call `sendToL1(bytes)` with a `bytes` argument of, e.g., 2 MB.
3. The hook passes the full payload to `emit_l1_message` → `push_message` with no size rejection.
4. The block's pubdata now includes a 2 MB message blob; if the DA layer limit is ~128 KB (Ethereum calldata) or ~768 KB (blob), the block cannot be posted to L1.
5. In the proving system, allocating a 2 MB `UsizeAlignedByteBox` in the fixed RISC-V heap triggers a fatal runtime error, making the block unprovable. [6](#0-5) [7](#0-6) [8](#0-7) [3](#0-2) [5](#0-4)

pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;

    Ok(Ok(()))

    fn emit_l1_message(
        &mut self,
        _ee_type: ExecutionEnvironmentType,
        resources: &mut Self::Resources,
        address: &<Self::IOTypes as SystemIOTypesConfig>::Address,
        data: &[u8],
    ) -> Result<Bytes32, SystemError> {
        // TODO(EVM-1077): consider adding COMPUTATIONAL_PRICE_FOR_PUBDATA as in Era

        // We need to charge cost of hashing:
        // - keccak256_native_cost(L2_TO_L1_LOG_SERIALIZE_SIZE) and
        //   keccak256_native_cost(64) when reconstructing L2ToL1Log
        // - keccak256_native_cost(64) + keccak256_native_cost(data.len())
        //   when reconstructing Messages
        // - at most 1 time keccak256_native_cost(64) when building the
        //   Merkle tree (as merkle tree can contain ~2*N nodes, where the
        //   first N nodes are leaves the hash of which is calculated on the
        //   previous step).

        let hashing_native_cost =
            keccak256_native_cost::<Self::Resources>(L2_TO_L1_LOG_SERIALIZE_SIZE).as_u64()
                + 3 * keccak256_native_cost::<Self::Resources>(64).as_u64()
                + keccak256_native_cost::<Self::Resources>(data.len()).as_u64();

        // We also charge some native resource for storing the log
        let native = hashing_native_cost
            + EVENT_STORAGE_BASE_NATIVE_COST
            + EVENT_DATA_PER_BYTE_COST * (data.len() as u64);

        resources.charge(&R::from_native(R::Native::from_computational(native)))?;

        // TODO(EVM-1078): for Era backward compatibility we may need to add events for l2 to l1 log and l1 message

        // Compute data hash directly: the native cost for this keccak is already
        // pre-charged above (included in `hashing_native_cost`), and this function
        // must not charge ergs — EVM gas accounting is the caller's responsibility
        // (the L1Messenger system contract charges it before invoking the hook).
        use crypto::MiniDigest;
        let data_hash = Bytes32::from_array(crypto::sha3::Keccak256::digest(data));
        let data = UsizeAlignedByteBox::from_slice_in(data, self.allocator.clone());
        self.logs_storage
            .push_message(self.tx_number, address, data, data_hash)?;
        Ok(data_hash)

    pub fn push_message(
        &mut self,
        tx_number: u32,
        address: &B160,
        data: UsizeAlignedByteBox<A>,
        data_hash: Bytes32,
    ) -> Result<(), SystemError> {
        // We are publishing message data(4 bytes to encode length) and underlying log
        // TODO: double check that we should have 4 here
        let total_pubdata = 4 + data.len() + L2_TO_L1_LOG_SERIALIZE_SIZE;
        let total_pubdata = total_pubdata as u32;

        let total_pubdata = self
            .list
            .top()
            .map_or(total_pubdata, |(_, m)| *m + total_pubdata);

        self.list.push(
            LogContent {
                tx_number,
                data: GenericLogContentData::UserMsg(UserMsgData {
                    address: *address,
                    data,
                    data_hash,
                }),
            },
            total_pubdata,
        );

        Ok(())
    }
