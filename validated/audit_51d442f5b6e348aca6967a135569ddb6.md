Audit Report

## Title
Permanent Deposit Loss via Single-Block Log Overflow in ckETH/ckERC20 Minter — (`rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

The `scrape_block_range` function in the ckETH/ckERC20 minter permanently skips any Ethereum block whose deposit-event log response exceeds the 2 MB HTTP-outcall ceiling. Once skipped, the block is recorded in `skipped_blocks` and `last_scraped_block_number` is advanced past it with no recovery path. An attacker who floods the helper contract with enough deposit calls in a single block causes every legitimate deposit in that block to be permanently unprocessed, resulting in irreversible loss of user funds locked in the Ethereum helper contract.

## Finding Description

The initial response-size estimate is `ETH_GET_LOGS_INITIAL_RESPONSE_SIZE_ESTIMATE + HEADER_SIZE_LIMIT = 100 + 2048 = 2148 bytes`. [1](#0-0) 

The `DoubleCycles` retry strategy (with `MAX_NUM_RETRIES = 10`) doubles the estimate on each retry. The EVM-RPC canister enforces a hard 2 MB ceiling on HTTP outcalls, so after all retries are exhausted the response is still too large. [2](#0-1) 

When the range has been bisected down to a single block (`from_block == to_block`) and the response still exceeds 2 MB, the terminal branch in `scrape_block_range` emits `SkippedBlockForContract` and immediately advances `last_scraped_block_number` past the block: [3](#0-2) 

`record_skipped_block_for_contract` inserts the block into `skipped_blocks` with an `assert!` that panics the canister if the same block is ever re-inserted, making the skip permanent and any re-scrape attempt fatal: [4](#0-3) 

The only active scraping path is `EthOrErc20DepositWithSubaccount`; the other two IDs are `Deprecated`, so this single path covers all ETH and ERC-20 deposits: [5](#0-4) 

`update_last_scraped_block_number` unconditionally sets the cursor forward, and no admin endpoint, governance proposal path, or state-machine mechanism exists to reset `last_scraped_block_number` to a prior value: [6](#0-5) 

## Impact Explanation

Every deposit event emitted by the helper contract in the skipped block is permanently ignored. Depositors' ETH or ERC-20 tokens remain locked in the Ethereum helper contract with no corresponding ckETH/ckERC20 minted on the IC. There is no on-chain or off-chain recovery path. This constitutes a **High** impact: significant Chain Fusion / ck-token security impact with concrete, permanent user-fund harm, matching the allowed impact class "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

Each `ReceivedEthOrErc20WithSubaccount` log entry is approximately 600 bytes; exceeding 2 MB requires roughly 3,334 matching log entries in a single block. [7](#0-6) 

An Ethereum block's ~30 M gas limit bounds independent transactions, but a single transaction can emit many events via a batch-deposit contract or a looping contract calling the helper. The attacker's own deposits are also lost, making this a griefing/vandalism attack rather than a profitable one, which lowers likelihood. However, the permanent, irreversible nature of the impact and the absence of any recovery path make it a meaningful risk for a high-value bridge.

## Recommendation

1. **Remove the permanent-skip branch.** Instead of advancing `last_scraped_block_number` past an oversized single block, return an error and let the timer retry. Introduce a per-block retry counter; only skip after N consecutive failures, and emit an alert.
2. **Add a governance-controlled re-scrape endpoint** that allows the NNS or minter controller to reset `last_scraped_block_number` to a prior value so a previously skipped block can be re-attempted.
3. **Remove or soften the `assert!` in `record_skipped_block_for_contract`** — replace it with a no-op or a log so that a re-scrape attempt does not panic the canister. [8](#0-7) 
4. **Raise or make the 2 MB ceiling configurable** via upgrade argument so a temporary spike in log volume does not immediately trigger the skip path.

## Proof of Concept

The existing integration test `should_skip_single_block_containing_too_many_events` already demonstrates the exact mechanism end-to-end: 3,500 log entries (~2.1 MB) in a single block cause the minter to emit `SkippedBlock` and advance past the block with no further attempt to process those deposits. [9](#0-8) 

An attacker replicates this on mainnet by submitting transactions within one block that call the `ReceivedEthOrErc20WithSubaccount` helper contract enough times to produce ≥ 3,334 log entries. The next scraping cycle of the ckETH minter will:

1. Attempt `get_logs` for the block range containing the flooded block.
2. Receive a `LimitExceeded` HTTP-outcall error at every doubling of the response-size estimate up to 2 MB.
3. Bisect down to the single flooded block (`from_block == to_block`).
4. Emit `SkippedBlockForContract` and advance `last_scraped_block_number` past it — permanently discarding all deposits in that block. [10](#0-9)

#[test]
fn should_skip_single_block_containing_too_many_events() {
    let cketh = CkEthSetup::default();
    let deposit = DepositParams::default().to_log_entry();
    // around 600 bytes per log
    // we need at least 3334 logs to reach the 2MB limit
    let large_amount_of_logs = multi_logs_for_single_transaction(deposit.clone(), 3_500);
    assert!(serde_json::to_vec(&large_amount_of_logs).unwrap().len() > 2_000_000);

    cketh.env.advance_time(SCRAPING_ETH_LOGS_INTERVAL);
    MockJsonRpcProviders::when(JsonRpcMethod::EthGetBlockByNumber)
        .respond_for_all_with(block_response(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 3))
        .build()
        .expect_rpc_calls(&cketh);

    for max_response_bytes in cketh.all_eth_get_logs_response_size_estimates() {
        MockJsonRpcProviders::when(JsonRpcMethod::EthGetLogs)
            .with_request_params(json!([{
                "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "toBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 3),
                "address": [ETH_HELPER_CONTRACT_ADDRESS],
                "topics": [cketh.received_eth_event_topic()]
            }]))
            .with_max_response_bytes(max_response_bytes)
            .respond_for_all_with(large_amount_of_logs.clone())
            .build()
            .expect_rpc_calls(&cketh);
    }

    for max_response_bytes in cketh.all_eth_get_logs_response_size_estimates() {
        MockJsonRpcProviders::when(JsonRpcMethod::EthGetLogs)
            .with_request_params(json!([{
                "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "toBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 2),
                "address": [ETH_HELPER_CONTRACT_ADDRESS],
                "topics": [cketh.received_eth_event_topic()]
            }]))
            .with_max_response_bytes(max_response_bytes)
            .respond_for_all_with(large_amount_of_logs.clone())
            .build()
            .expect_rpc_calls(&cketh);
    }

    for max_response_bytes in cketh.all_eth_get_logs_response_size_estimates() {
        MockJsonRpcProviders::when(JsonRpcMethod::EthGetLogs)
            .with_request_params(json!([{
                "fromBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
                "toBlock": BlockNumber::from(LAST_SCRAPED_BLOCK_NUMBER_AT_INSTALL + 1),
