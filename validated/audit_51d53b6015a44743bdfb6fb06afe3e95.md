### Title
Uncontrolled Slippage in XCC Precompile: User Charged More wNEAR Than Specified Due to Silent `STORAGE_AMOUNT` Addition - (File: engine-precompiles/src/xcc.rs)

---

### Summary

When a user calls the Cross-Contract Call (XCC) precompile for the first time (i.e., their router sub-account does not yet exist), the precompile silently adds `STORAGE_AMOUNT` (2 NEAR = `2_000_000_000_000_000_000_000_000` yoctoNEAR) on top of the user-specified `attached_near` and withdraws the total from the user's wNEAR ERC-20 balance via `transferFrom`. There is no `max_amount` field in `CrossContractCallArgs` to bound the total wNEAR the user is willing to spend, creating an uncontrolled slippage condition directly analogous to the reported vault-purchase bug.

---

### Finding Description

The XCC precompile's `run_with_handle` function parses the user-supplied `CrossContractCallArgs` to extract `attached_near` — the amount of wNEAR the user intends to attach to the cross-contract call. [1](#0-0) 

It then computes `required_near` by checking whether the caller's router sub-account already exists on-chain: [2](#0-1) 

If the router does **not** exist (`None` branch), `STORAGE_AMOUNT` (2 NEAR) is silently added:

```
required_near = attached_near + state::STORAGE_AMOUNT   // 2_000_000_000_000_000_000_000_000 yoctoNEAR
```

The precompile then immediately calls `transferFrom` on the wNEAR ERC-20 contract to pull `required_near` from the caller's balance: [3](#0-2) 

The `STORAGE_AMOUNT` constant is defined as: [4](#0-3) 

The `CrossContractCallArgs` enum has no `max_amount` or slippage-limit field: [5](#0-4) 

There is therefore no mechanism for the user to express "I am willing to pay at most X wNEAR total." The extra 2 NEAR is deducted atomically and irrevocably within the same EVM call.

---

### Impact Explanation

A user who has approved a large wNEAR allowance (e.g., `uint256.max`, a common pattern) to the XCC precompile address `0x516cded1d16af10cad47d6d49128e2eb7d27b372` and submits their first XCC call with `attached_near = X` will have `X + 2 NEAR` of wNEAR withdrawn from their ERC-20 balance without explicit consent to the extra 2 NEAR charge. The 2 NEAR is locked as storage staking in the router sub-account, which the user did not agree to fund at that moment. This constitutes a **High** impact: theft of unclaimed yield / unexpected permanent loss of user funds (the storage-staked NEAR is not freely recoverable by the user through normal EVM interactions).

---

### Likelihood Explanation

Every first-time XCC caller is affected. The XCC feature is a core Aurora production interface reachable by any unprivileged EVM user. Users who follow the common pattern of approving `uint256.max` wNEAR (to avoid repeated approvals) are silently overcharged by exactly 2 NEAR on their first call. The condition is deterministic and requires no special timing or adversarial setup.

---

### Recommendation

Add a `max_near` field to `CrossContractCallArgs` (or as a wrapper around it) representing the maximum total wNEAR the caller is willing to spend. After computing `required_near`, compare it against `max_near` and revert with a descriptive error if `required_near > max_near`:

```rust
if required_near > max_near {
    return Err(revert_with_message("ERR_XCC_SLIPPAGE_EXCEEDED"));
}
```

This mirrors the `max_gas_price` field already present in `SubmitArgs` for the `submit_with_args` entrypoint: [6](#0-5) 

---

### Proof of Concept

1. User calls `wNEAR.approve(XCC_PRECOMPILE_ADDRESS, uint256.max)` on Aurora.
2. User constructs `CrossContractCallArgs::Eager(PromiseArgs::Create(PromiseCreateArgs { attached_balance: Yocto::new(1), ... }))` — intending to spend 1 yoctoNEAR.
3. User submits the EVM transaction targeting the XCC precompile address.
4. Inside `run_with_handle`, `attached_near = 1` yoctoNEAR.
5. `state::get_code_version_of_address` returns `None` (first call, no router deployed).
6. `required_near = 1 + 2_000_000_000_000_000_000_000_000` yoctoNEAR.
7. `transferFrom(user, engine_implicit_address, required_near)` is called on wNEAR — the user loses 2 NEAR more than they specified, with no recourse. [7](#0-6)
