### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Users to Systematically Overpay - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary
In `compute_gas_refund`, the `delta_gas` reconciliation between native resource consumption and EVM gas consumption is applied asymmetrically. When native consumption exceeds gas consumption (`delta_gas > 0`), extra gas is charged to the user. When gas consumption exceeds native consumption (`delta_gas < 0`), the negative delta is silently discarded and never returned to the user. A developer-authored `// TODO: return delta_gas to gas_used?` comment at the exact site confirms this is an unresolved accounting gap. The result is that users systematically overpay whenever their EVM gas consumption exceeds what native resources would imply.

### Finding Description

In `compute_gas_refund`:

```rust
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // native consumption > gas consumption → charge extra gas
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?   ← acknowledged gap
}
```

The design intent, documented in `docs/double_resource_accounting.md`, is:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed` … This ensures that gas estimation will include additional gas to cover for native resources.

The documentation describes only the positive branch. The negative branch — where EVM gas used exceeds the native-resource-implied gas — is never handled. The TODO comment at line 80 of `refund_calculation.rs` is the codebase's own acknowledgment that this case was left incomplete.

The `compute_gas_refund` function is called from both the ZK L2 transaction flow (`basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`, `before_refund`) and the L1 transaction flow (`process_l1_transaction.rs`). The Ethereum-mode flow passes `native_per_gas = 0`, which forces `delta_gas = 0` and is unaffected. Only ZK/L2 transactions with a non-zero `native_per_gas` are exposed.

The `gas_used` value produced by `compute_gas_refund` is then used directly to compute the user's refund:

```rust
// zk/mod.rs refund_and_commit_fee
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
```

Because `gas_used` is never reduced by a negative `delta_gas`, the refund is always smaller than it should be when native resources are cheaper than EVM gas. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

For every L2 transaction where `native_used / native_per_gas < gas_used`, the user's gas refund is understated by exactly `(gas_used - native_used / native_per_gas) * gas_price`. The user pays for EVM gas that was not backed by any native resource consumption. This is a direct, per-transaction financial loss for ordinary users — no privileged role is required to be affected.

Concrete example:
- `gas_limit = 100 000`, EVM `gas_used = 50 000`, native equivalent = `40 000` gas
- `delta_gas = 40 000 − 50 000 = −10 000` → discarded
- Actual refund: `50 000 * gas_price` instead of the correct `60 000 * gas_price`
- Overpayment: `10 000 * gas_price` per transaction [4](#0-3) [5](#0-4) 

### Likelihood Explanation

Any L2 transaction whose EVM gas consumption exceeds what native resources would imply triggers the bug. This is the common case for transactions that are EVM-gas-heavy but proving-cheap (e.g., simple ETH transfers, ERC-20 transfers, read-heavy calls). The condition `native_used / native_per_gas < gas_used` is routinely satisfied in normal operation, making this a high-likelihood, every-block occurrence. [6](#0-5) 

### Recommendation

Apply the `delta_gas` adjustment symmetrically. When `delta_gas < 0`, reduce `gas_used` by the magnitude of the negative delta (clamped to avoid underflow below `minimal_gas_used`):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction);
    // Re-apply the minimal_gas_used floor after the reduction
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

This mirrors the recommendation in the referenced external report: instead of using a hardcoded/one-sided amount, compute the actual remaining delta and apply it in both directions. [7](#0-6) 

### Proof of Concept

1. Deploy any EVM contract that consumes significant EVM gas but writes few storage slots (low pubdata / low native cost).
2. Submit an L2 transaction calling that contract with a generous `gas_limit` (e.g., 200 000) and a non-zero `gas_price` so `native_per_gas > 0`.
3. After execution, observe `tx_result.gas_used`. Compare it against `native_used / native_per_gas`.
4. When `gas_used > native_used / native_per_gas`, the user's refund is `(gas_limit - gas_used) * gas_price` instead of the correct `(gas_limit - native_used/native_per_gas) * gas_price`.
5. The difference — `(gas_used - native_used/native_per_gas) * gas_price` — is the amount silently kept from the user due to the missing negative-`delta_gas` branch.

The existing test `test_delta_gas` in `tests/instances/transactions/src/native_charging.rs` already exercises the positive-`delta_gas` path; an analogous test for the negative path would demonstrate the overpayment. [8](#0-7) [1](#0-0)
