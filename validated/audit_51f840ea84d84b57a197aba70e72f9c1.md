### Title
Floor Division in `native_per_pubdata` Causes Systematic Pubdata Undercharging - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed using floor (truncating) division (`wrapping_div`) instead of ceiling division (`div_ceil`). This mirrors the exact class of bug in the external report: a per-unit price derived by integer division rounds down, so when that unit is consumed in quantity the total collected is always less than the stated cost. Any L2 transaction that writes pubdata (e.g. via `SSTORE`) is systematically undercharged, causing the protocol to collect less native resource than the actual pubdata cost warrants.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the two resource-rate conversions are computed as follows:

```rust
// native_per_gas  — uses CEILING division
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))…;

// native_per_pubdata — uses FLOOR division
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))…;
``` [1](#0-0) 

`native_per_pubdata` is the number of native-resource units charged per byte of pubdata. It is later multiplied by the actual pubdata bytes consumed to produce the native charge:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

Because `native_per_pubdata = floor(pubdata_price / native_price)`, the ETH-equivalent charge per pubdata byte is:

```
floor(pubdata_price / native_price) * native_price
  ≤  pubdata_price
```

The shortfall per byte equals `pubdata_price mod native_price`, which can be as large as `native_price − 1`. This shortfall propagates into `native_used`, which in turn reduces `delta_gas` in the refund calculation:

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 { gas_used += delta_gas as u64; }
``` [3](#0-2) 

A smaller `native_used` yields a smaller `delta_gas`, so the user is charged fewer gas units and receives a larger refund than the actual pubdata cost justifies.

The same floor-division defect is replicated in the API helper used by the sequencer for pre-flight validation:

```rust
// native_per_pubdata = pubdata_price / native_price  (floor)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

The comment on the line above it (`// native_per_gas = ceil(gas_price / native_price)`) makes the asymmetry explicit and unambiguous.

---

### Impact Explanation

For every L2 transaction that generates pubdata the protocol collects:

```
pubdata_bytes × floor(pubdata_price / native_price) × native_price
```

instead of the intended:

```
pubdata_bytes × pubdata_price
```

The per-byte shortfall is `pubdata_price mod native_price`. For a transaction writing 1 000 storage slots (≈ 32 000 pubdata bytes) with `native_price = 1 000` and `pubdata_price = 999 999`, the shortfall is `999` wei × 32 000 bytes ≈ 32 M wei per transaction. At scale this represents a continuous, compounding revenue loss for the operator/protocol on every block that contains pubdata-generating transactions. The user pays less than the stated cost of data publication, directly analogous to the GroupBuy pool never reaching its `initialPrice`.

---

### Likelihood Explanation

Every ordinary EVM transaction that writes to storage generates pubdata. The condition `pubdata_price mod native_price ≠ 0` is the normal case whenever the operator sets prices that are not exact multiples of each other. No special transaction crafting is required; a standard ERC-20 transfer or any `SSTORE` opcode triggers the undercharge automatically.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is already computed:

```rust
// Before (floor — undercounts)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))…;

// After (ceiling — matches native_per_gas pattern)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))…;
```

Apply the same fix in `api/src/helpers.rs` line 427 so the sequencer-side pre-flight check is consistent with the on-chain bootloader behaviour. [5](#0-4) [6](#0-5)

### Proof of Concept

**Setup:**
- `native_price  = 1_000` (operator-set, 1 000 wei per native unit)
- `pubdata_price = 1_999` (operator-set, 1 999 wei per pubdata byte)
- Transaction writes 100 storage slots → 3 200 pubdata bytes

**Current behaviour (floor):**
```
native_per_pubdata = floor(1_999 / 1_000) = 1
native_charge      = 3_200 × 1 = 3_200 native units
ETH equivalent     = 3_200 × 1_000 = 3_200_000 wei
```

**Correct behaviour (ceiling):**
```
native_per_pubdata = ceil(1_999 / 1_000) = 2
native_charge      = 3_200 × 2 = 6_400 native units
ETH equivalent     = 6_400 × 1_000 = 6_400_000 wei
```

The user is charged **3 200 000 wei less** than the actual pubdata cost of **6 400 000 wei** — a 50 % undercharge — for a single ordinary transaction. The shortfall is realised as a larger gas refund via the `delta_gas` path in `compute_gas_refund`. [7](#0-6) [8](#0-7)
