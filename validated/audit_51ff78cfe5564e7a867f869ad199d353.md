Audit Report

## Title
Byzantine Block Maker Can Trigger K-Fold RPC Amplification via Duplicate `SignedIngressId` Entries in Crafted `StrippedBlockProposal` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary

When `HASHES_IN_BLOCKS_ENABLED = true` (unconditionally set in production), a Byzantine block maker can craft a `StrippedBlockProposal` protobuf with K duplicate `StrippedIngressMessage` entries for the same ingress ID. Because neither the deserialization path nor `BlockProposalAssembler::new` deduplicate the `Vec<SignedIngressId>`, the assembler spawns K concurrent `get_or_fetch` Tokio tasks, each issuing an independent RPC to a random peer before any insertion conflict is detected. The assembled block is ultimately rejected (`Unwanted`), but the K concurrent RPCs have already been issued, constituting a repeatable per-round amplification attack.

## Finding Description

**Root cause — unsigned `stripped_ingress_payload` field:**

The `StrippedBlockProposal` struct carries no signature over its `stripped_ingress_payload` field. The `pruned_block_proposal_proto` carries the block maker's signature over the pruned block content, and `unstripped_consensus_message_id` carries the hash of the full block, but neither covers the `ingress_messages` list in the stripped wrapper. [1](#0-0) 

The protobuf definition confirms `ingress_messages` is a plain `repeated` field with no MAC or signature: [2](#0-1) 

**Deserialization collects into `Vec` with no deduplication:**

`TryFrom<pb::StrippedBlockProposal>` maps the repeated field directly into a `Vec<SignedIngressId>`: [3](#0-2) 

**`BlockProposalAssembler::new` maps the Vec directly — no deduplication:** [4](#0-3) 

**`missing_stripped_messages` returns all K `None`-slots — no deduplication:**

`missing_artifacts` filters for `None` slots; with K duplicate entries all initialized to `None`, all K are returned: [5](#0-4) [6](#0-5) 

**`assemble_message` spawns one `get_or_fetch` task per returned ID — K tasks for K duplicates:** [7](#0-6) 

**`get_or_fetch` issues an RPC if the message is not in the local pool:** [8](#0-7) 

The RPC path in `download_stripped_message` issues a `transport.rpc()` call to a randomly selected peer: [9](#0-8) 

**Post-spawn failure does not prevent the amplification:**

`try_insert` uses a linear `find` that returns the first matching slot. The first task fills slot 0; the second task finds slot 0 already `Some` and returns `AlreadyInserted`, causing `assemble_message` to return `Unwanted`. But all K RPCs have already been issued concurrently: [10](#0-9) [11](#0-10) 

## Impact Explanation

Each receiving replica spawns K concurrent Tokio tasks and issues up to K independent RPCs to random peers for the same ingress message. With `max_ingress_messages_per_block ≈ 1000`, a single crafted proposal causes up to 1000× RPC amplification per receiving replica per round. The Byzantine node is selected as block maker with probability proportional to their rank in each round — this is a regular, predictable event. The assembled block is ultimately rejected (`Unwanted`), so there is no consensus safety impact, but the resource exhaustion (task spawning, network bandwidth, peer RPC handler load) is real and repeatable.

This matches the **High ($2,000–$10,000)** impact class: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation

- Requires only a single Byzantine replica acting as block maker for any round — well within the "below fault threshold" attacker model.
- No cryptographic forgery required: the `stripped_ingress_payload` is not signed.
- The crafted protobuf is trivially constructable by repeating the same `StrippedIngressMessage` entry K times.
- `HASHES_IN_BLOCKS_ENABLED = true` is the live production configuration.
- The attack is repeatable every round the Byzantine node is selected as block maker.

## Recommendation

Deduplicate `SignedIngressId` entries immediately upon deserialization in `TryFrom<pb::StrippedBlockProposal>`, or at the start of `BlockProposalAssembler::new`. A `HashSet` or sorted-dedup pass over `stripped_ingress_payload.ingress_messages` before constructing the `ingress_messages` Vec eliminates the amplification. Alternatively, reject the `StrippedBlockProposal` during deserialization if any duplicate `SignedIngressId` is detected, treating it as a malformed artifact.

## Proof of Concept

The existing `missing_stripped_messages_test` at line 911 of `assembler.rs` confirms the Vec is returned as-is with no deduplication. A minimal extension demonstrates the vulnerability:

```rust
// Construct a StrippedBlockProposal with K=100 duplicate entries
let (_, ingress_id) = fake_ingress_message_with_arg_size("dup", 64);
let duplicate_ids = vec![StrippedMessageId::Ingress(ingress_id); 100];
let stripped = fake_stripped_block_proposal_with_messages(duplicate_ids);

let assembler = BlockProposalAssembler::new(stripped);
let missing = assembler.missing_stripped_messages();
assert_eq!(missing.len(), 100); // 100 duplicate tasks will be spawned
// Each spawns a get_or_fetch task → up to 100 concurrent RPCs for the same message
``` [12](#0-11)
