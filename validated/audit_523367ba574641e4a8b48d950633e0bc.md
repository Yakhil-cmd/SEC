### Title
ckBTC Minter: TOCTOU on `fee_based_retrieve_btc_min_amount` Causes User ckBTC to Be Locked in Withdrawal Account - (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

---

### Summary

The ckBTC minter exposes a query endpoint `estimate_withdrawal_fee` and `get_minter_info` that users consult to determine the current minimum withdrawal amount before transferring ckBTC to their withdrawal account. Because `fee_based_retrieve_btc_min_amount` is updated asynchronously by the minter's timer (via `estimate_fee_per_vbyte`), the minimum can increase between the user's query and their subsequent `retrieve_btc` call. When this happens, `retrieve_btc` returns `AmountTooLow` without burning any ckBTC, but the user's ckBTC is already locked in the minter-controlled withdrawal account with no automatic refund path.

---

### Finding Description

The ckBTC withdrawal flow for the `retrieve_btc` path requires users to:
1. Call `get_withdrawal_account()` to obtain their minter-controlled withdrawal subaccount.
2. Transfer ckBTC to that account.
3. Call `retrieve_btc(amount, address)`.

Before step 2, users naturally call the query endpoints `estimate_withdrawal_fee` and `get_minter_info` to learn the current fee and minimum withdrawal amount (`fee_based_retrieve_btc_min_amount`).

Inside `retrieve_btc`, the minimum is re-read from live state:

```rust

if args.amount < min_retrieve_amount {
    return Err(RetrieveBtcError::AmountTooLow(min_retrieve_amount));
}

The withdrawal account is a subaccount of the minter canister. The user has no direct control over it and cannot transfer ckBTC back out of it — they can only consume it via `retrieve_btc`. There is no automatic refund mechanism when `retrieve_btc` returns `AmountTooLow`.

---

### Impact Explanation

A user who:
1. Queries `get_minter_info()` and sees `retrieve_btc_min_amount = 100,000` sats,
2. Transfers exactly 100,000 sats of ckBTC to their withdrawal account,
3. Then calls `retrieve_btc(100_000, address)` after the timer has updated `fee_based_retrieve_btc_min_amount` to 150,000 sats,

will receive `AmountTooLow(150_000)`. Their 100,000 sats of ckBTC is now locked in the minter-controlled withdrawal account. To proceed, the user must:
- Acquire an additional 50,000 sats of ckBTC and transfer it to the withdrawal account, then retry with `retrieve_btc(150_000)`, **or**
- Wait indefinitely for Bitcoin network fees to drop back below the threshold.

This is a direct financial impact: the user is forced to spend more ckBTC than the amount they planned to withdraw, or their funds are temporarily frozen. Unlike the `retrieve_btc_with_approval` path (where no ckBTC is moved before the call), the `retrieve_btc` path requires the user to pre-fund the withdrawal account, making this TOCTOU window exploitable by normal Bitcoin fee volatility.

---

### Likelihood Explanation

Bitcoin network fees are well-known to spike sharply and unpredictably (e.g., during inscription/ordinal activity). The minter's timer refreshes `fee_based_retrieve_btc_min_amount` periodically. Any user following the documented two-step flow (`get_withdrawal_account` → transfer → `retrieve_btc`) faces this race window. No privileged access, no malicious actor, and no threshold corruption is required — ordinary Bitcoin fee volatility is sufficient to trigger this condition. The discrete 50,000-satoshi jump in the minimum makes it easy for a user's exact transfer to fall below the new threshold.

---

### Recommendation

1. **Snapshot the minimum at transfer time**: When the user transfers ckBTC to the withdrawal account, record the `fee_based_retrieve_btc_min_amount` at that moment and honor it for a bounded time window (e.g., one timer cycle).
2. **Automatic refund on `AmountTooLow`**: If `retrieve_btc` rejects a request with `AmountTooLow`, automatically transfer the ckBTC back from the withdrawal subaccount to the user's principal account, analogous to the `Refunded` path in the CMC's `notify_top_up`.
3. **Prefer `retrieve_btc_with_approval`**: Document and encourage the ICRC-2 approval path (`retrieve_btc_with_approval`), which does not require pre-funding a withdrawal account and therefore does not expose the user to this TOCTOU window.

---

### Proof of Concept

```
1. User calls get_minter_info() → retrieve_btc_min_amount = 100_000 sats
   (fee_based_retrieve_btc_min_amount = 100_000)

2. User calls get_withdrawal_account() → gets withdrawal_account

3. User calls ckBTC ledger icrc1_transfer(to=withdrawal_account, amount=100_000)
   → ckBTC is now locked in minter-controlled subaccount

4. Bitcoin fee rate spikes; minter timer fires estimate_fee_per_vbyte():
   fee_based_retrieve_btc_min_amount updated to 150_000

5. User calls retrieve_btc(amount=100_000, address="bc1q...")
   → read_state: min_retrieve_amount = 150_000
   → 100_000 < 150_000 → return Err(AmountTooLow(150_000))
   → No burn, no refund

6. User's 100_000 sats of ckBTC is stuck in the withdrawal account.
   User must acquire 50_000 more sats of ckBTC and retry with amount=150_000,
   or wait for fees to drop.
```
