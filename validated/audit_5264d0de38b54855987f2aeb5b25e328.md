### Title
`miller_rabin_u32` Inner-Loop `continue` Bug Causes False Negatives in Hash-to-Prime Primality Test, Enabling Oracle Panic and Proving Divergence ‚Äî (`callable_oracles/src/hash_to_prime/common.rs`)

---

### Summary

The `miller_rabin_u32` function in `callable_oracles/src/hash_to_prime/common.rs` contains a concrete control-flow bug: a `continue` statement inside the inner squaring loop only continues that inner loop, not the outer base-iteration loop. As a result, the function always falls through to `return false` after the inner loop, even when the Miller-Rabin witness condition (`result == minus_one`) is satisfied. This produces false negatives ‚Äî valid primes are incorrectly classified as composite. The function is called on both the oracle (compute) side and the in-proof verifier side of the hash-to-prime construction, with no dedicated unit tests for the primitive itself.

---

### Finding Description

The Miller-Rabin primality test in `miller_rabin_u32` is structured as:

```
outer: for a in MILLER_RABIN_BASES {
    // compute a^d mod candidate (Montgomery form)
    ...
    if result == mont_r  { continue; }   // a^d ‚â° 1 ‚Äî base passes
    if result == minus_one { continue; } // a^d ‚â° -1 ‚Äî base passes

    for _r in 1..s {
        result = mont_square_u32(result, candidate, inv);
        if result == minus_one {
            continue;   // ‚Üê BUG: continues inner loop, NOT outer loop
        }
    }

    return false;   // always reached after inner loop, even if minus_one was found
}
``` [1](#0-0) 

The correct algorithm requires that when `result == minus_one` is found inside the inner loop, execution must break out of the inner loop **and** continue to the next base in the outer loop. The bare `continue` only advances the inner `for _r in 1..s` iterator. After the inner loop terminates (whether or not `minus_one` was found), control unconditionally reaches `return false`.

**Concrete falsification**: For `p = 5`, `p ‚àí 1 = 4 = 2¬≤ √ó 1`, so `s = 2`, `d = 1`. For base `a = 2`: `a^d mod 5 = 2`, which is neither `1` nor `4`. The inner loop squares once: `2¬≤ mod 5 = 4 = minus_one`, hits `continue` (inner loop ends), then falls through to `return false`. So `miller_rabin_u32(5)` returns `false` even though 5 is prime.

The same function is the **only** primality check used in the compute path: [2](#0-1) 

And in the in-proof verify path: [3](#0-2) 

The only existing test (`try_compute_hash_to_prime`) exercises the full round-trip (compute ‚Üí verify) over 1000 seeds: [4](#0-3) 

Because both sides share the same buggy implementation, the round-trip test passes ‚Äî the oracle skips primes that fail the buggy test, and the verifier also rejects them. There are **no unit tests** for `miller_rabin_u32`, `bigint_mont_mul`, `bigint_mont_square`, `mont_pow`, or `little_fermat` in isolation. [5](#0-4) 

---

### Impact Explanation

**Oracle panic / valid-execution unprovability**: `compute_from_entropy` searches up to `2^low_bits = 2^11 = 2048` consecutive candidates for the initial prime: [6](#0-5) 

If the buggy `miller_rabin_u32` rejects every prime in that window (possible for entropy inputs that land in a sparse-prime region where all primes satisfy the inner-loop condition), the oracle hits `unreachable!()` and panics. This makes the corresponding block unprovable ‚Äî a valid-execution unprovability impact.

**Forward/proving divergence**: If the oracle is ever patched (correct `miller_rabin_u32`) while the in-proof verifier retains the buggy version, the oracle will produce a `HashToPrimeData` certificate whose `first_step_n` points to a prime `p` that passes the correct test but fails the buggy verifier check at line 24 of `verify.rs`. `verify_hash_to_prime` returns `false`, the proof fails, and the block cannot be finalized ‚Äî a forward/proving divergence. [7](#0-6) 

---

### Likelihood Explanation

The bug is deterministic and reproducible for any prime `p` where at least one base `a ‚àà {2, 7, 61}` satisfies `a^d mod p ‚àâ {1, p‚àí1}` but `(a^d)^{2^r} mod p = p‚àí1` for some `r ‚àà [1, s)`. This is a large fraction of all 32-bit primes (e.g., 5, 13, 17, ‚Ä¶). The oracle panic path requires an entropy input whose derived window contains only such primes ‚Äî unlikely for random inputs but deterministically triggerable by a crafted input to the hash-to-prime oracle. The divergence path is triggered the moment the two code paths are desynchronized by any patch.

---

### Recommendation

Fix the inner loop to use a labeled `continue` targeting the outer loop:

```rust
'outer: for a in MILLER_RABIN_BASES.iter().copied() {
    ...
    for _r in 1..s {
        result = mont_square_u32(result, candidate, inv);
        if result == minus_one {
            continue 'outer;   // correct: skip to next base
        }
    }
    return false;
}
```

Add isolated unit tests for `miller_rabin_u32` (including known primes such as 5, 13, 17, and the MILLER_RABIN_BASES themselves), and for `bigint_mont_mul` / `bigint_mont_square` / `compute_mont_params` against reference big-integer arithmetic.

### Proof of Concept

Trace for `miller_rabin_u32(5)`:

- `p = 5`, `p ‚àí 1 = 4 = 2¬≤ √ó 1` ‚Üí `s = 2`, `d = 1`
- `mont_r = 2^32 mod 5 = 1`, `mont_r2 = 1`, `minus_one = 4`
- Base `a = 2` (Montgomery form = 2):
  - `bits = 32 ‚àí leading_zeros(1) ‚àí 1 = 0` ‚Üí exponentiation loop skipped, `result = 2`
  - `result ‚âÝ mont_r (1)`, `result ‚âÝ minus_one (4)`
  - Inner loop `for _r in 1..2`: `result = 2¬≤ mod 5 = 4 = minus_one` ‚Üí `continue` (inner loop ends)
  - Falls through ‚Üí **`return false`** ‚Üê incorrect; 5 is prime

The same pattern applies to primes 13, 17, 29, 37, ‚Ä¶ making the bug systematic, not a corner case.

pub fn verify_hash_to_prime(entropy: &[u8; 64], certificate: &HashToPrimeData) -> bool {
    let mut entropy_it = entropy.iter().copied();
    let step_0_prime = {
        let (entropy_bits, low_bits) = GENERATION_STEPS[0];
        let take_bytes = entropy_bits.div_ceil(8);
        let mut repr = [0u8; 4];
        write_entropy_le(
            &mut repr[..take_bytes as usize],
            &mut entropy_it,
            entropy_bits,
        );
        let high_part = u32::from_le_bytes(repr);
        let mut candidate = high_part << low_bits;
        if certificate.first_step_n >= 1u32 << low_bits {
            return false;
        }
        candidate += certificate.first_step_n;
        if candidate & 1 != 1 {
            return false;
        }
        if miller_rabin_u32(candidate) != true {
            return false;
        }

        candidate
    };
