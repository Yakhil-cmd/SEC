### Title
Unsafe Truncation of `tx_number` (`u32` → `u16`) in L2ToL1Log Construction - (`zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

In `logs_storage.rs`, the `tx_number` field (type `u32`) stored in `GenericLogContent` is silently narrowed to `u16` when constructing `L2ToL1Log`. If a block contains more than 65,535 transactions, the `tx_number_in_block` field is silently truncated, producing incorrect per-log hashes and an incorrect L2→L1 Merkle root that is published to L1.

---

### Finding Description

The `From<&LogContent<A>> for L2ToL1Log` implementation performs an unchecked narrowing cast:

```rust
// zk_ee/src/common_structs/logs_storage.rs  line 519
tx_number_in_block: m.tx_number as u16,
```

`m.tx_number` is declared as `u32` in `GenericLogContent`:

```rust
// zk_ee/src/common_structs/logs_storage.rs  line 73
pub struct GenericLogContent<IOTypes: SystemIOTypesConfig, A: Allocator = Global> {
    pub tx_number: u32,
    ...
}
```

`tx_number_in_block` in `L2ToL1Log` is declared as `u16`:

```rust
// zk_ee/src/common_structs/logs_storage.rs  line 47
pub tx_number_in_block: u16,
```

The `tx_number` counter is incremented unconditionally once per committed transaction in `FullIO::finish_tx`:

```rust
// basic_system/src/system_implementation/system/io_subsystem.rs  line 532
self.tx_number += 1;
```

No bounds check or assertion guards the `as u16` cast. When `tx_number` exceeds `u16::MAX` (65,535), the value wraps silently, producing a `tx_number_in_block` that is incorrect for every L2→L1 log emitted by transactions numbered 65,536 and above.

The `tx_number_in_block` field is included verbatim in the canonical log encoding used to compute each log's Keccak hash:

```rust
// zk_ee/src/common_structs/logs_storage.rs  lines 473-480
fn write_encoding<T: WriteBytes + ?Sized>(&self, dst: &mut T) {
    dst.write(&[self.l2_shard_id]);
    dst.write(&[if self.is_service { 1 } else { 0 }]);
    dst.write(&self.tx_number_in_block.to_be_bytes());   // ← corrupted field
    ...
}
```

Those hashes feed directly into the L2→L1 Merkle tree root that is published as pubdata to L1.

---

### Impact Explanation

A corrupted `tx_number_in_block` produces an incorrect individual log hash, which propagates into an incorrect L2→L1 Merkle root. The L1 bridge contract uses this root to verify message inclusion proofs. Consequences include:

- **Unprovable valid messages**: A legitimate L2→L1 message emitted by a transaction with `tx_number ≥ 65536` will have an incorrect hash in the Merkle tree, making its inclusion proof permanently invalid on L1. Users cannot withdraw or relay those messages.
- **State-transition / pubdata integrity bug**: The pubdata committed to L1 contains a Merkle root derived from corrupted leaf hashes, creating a divergence between the actual execution state and the committed state.

---

### Likelihood Explanation

Triggering the overflow requires more than 65,535 transactions in a single block. With a minimum EVM intrinsic gas of 21,000 per transaction, this demands a block gas limit exceeding ~1.37 billion gas — far above typical production limits. The likelihood is therefore **low** under normal operating conditions. However:

- The code contains no assertion, checked cast, or compile-time guarantee preventing this path.
- If the block gas limit is ever raised significantly, or if a future transaction type has a lower intrinsic cost, the threshold becomes reachable.
- The `tx_number` field is typed `u32` precisely to accommodate large counts; the narrowing to `u16` is an inconsistency that is one configuration change away from being exploitable.

---

### Recommendation

Replace the silent cast with a checked conversion that panics (or returns an error) on overflow:

```rust
// Option A – panic on overflow (acceptable if block limits guarantee safety)
tx_number_in_block: u16::try_from(m.tx_number)
    .expect("tx_number exceeds u16::MAX"),

// Option B – widen the field to u32 to match the counter type
pub tx_number_in_block: u32,
```

Widening `tx_number_in_block` to `u32` in `L2ToL1Log` (and updating the encoding accordingly) is the safest long-term fix, as it eliminates the type mismatch entirely.

---

### Proof of Concept

1. Construct a block containing 65,537 transactions where transaction #65,536 (0-indexed) emits an L2→L1 message via the L1Messenger hook.
2. `FullIO::finish_tx` increments `self.tx_number` to 65,536 after that transaction.
3. `push_message` stores `tx_number: 65536u32` in `GenericLogContent`.
4. When `From<&LogContent<A>> for L2ToL1Log` is called, `65536u32 as u16 == 0u16`.
5. The log is encoded with `tx_number_in_block = 0` instead of `65536`, producing a different Keccak hash.
6. The Merkle root computed from this hash is committed to L1.
7. Any attempt to prove inclusion of that message on L1 fails because the leaf hash does not match the committed root.
