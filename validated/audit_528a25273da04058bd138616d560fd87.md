Audit Report

## Title
Missing Certified-State Validation of `BitcoinAdapterResponse` Callback IDs Allows Fabricated Bitcoin Responses â€” (`rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary
`BitcoinPayloadBuilder::validate_self_validating_payload_impl` accepts any non-empty `SelfValidatingPayload` that fits within the byte limit without verifying that each `BitcoinAdapterResponse::callback_id` corresponds to a pending request in the certified state. A single malicious block-proposing node can craft a `SelfValidatingPayload` containing a `GetSuccessorsResponse` with a real, live `callback_id` and fabricated Bitcoin block data. All honest replicas accept the block because their validation also only checks size. The fabricated blocks are then delivered to the Bitcoin management canister, corrupting the UTXO set and enabling illegal ckBTC minting.

## Finding Description
`validate_self_validating_payload_impl` performs only two checks: whether the payload is the default empty value, and whether its byte size is within `MAX_BITCOIN_PAYLOAD_IN_BYTES`. It never loads the certified state or inspects any `callback_id`:

```rust
fn validate_self_validating_payload_impl(
    &self,
    payload: &SelfValidatingPayload,
    validation_context: &ValidationContext,
) -> Result<NumBytes, SelfValidatingPayloadValidationError> {
    let since = Instant::now();
    if *payload == SelfValidatingPayload::default() {
        return Ok(0.into());
    }
    self.metrics.observe_validate_duration(VALIDATION_STATUS_VALID, since);
    let size = NumBytes::new(payload.count_bytes() as u64);
    Ok(size)
}
```

The `BatchPayloadBuilder::validate_payload` implementation calls this function and adds only a raw byte-length check, with no state-based validation.

The downstream `push_response` function in `rs/replicated_state/src/bitcoin.rs` does check that a `GetSuccessorsResponse`'s `callback_id` exists in `bitcoin_get_successors_contexts`, which prevents responses with entirely fabricated callback IDs from being processed. However, this is not a complete mitigation: an attacker who reads the certified state can obtain a real, currently-pending `callback_id` and supply it with fabricated block content. The existence check passes, and the fabricated blocks are encoded and pushed to the consensus queue as legitimate data. For `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject`, there is no callback_id lookup at all â€” any value is pushed directly to the consensus queue.

By contrast, `validate_canister_http_payload_impl` in `rs/https_outcalls/consensus/src/payload_builder.rs` correctly loads the certified state and rejects any response whose `callback_id` is not found in `canister_http_request_contexts`, establishing the correct pattern that is absent here.

## Impact Explanation
A malicious block proposer can inject a `GetSuccessorsResponse` carrying fabricated Bitcoin blocks (e.g., a block containing a transaction crediting the attacker's ckBTC deposit address with a large UTXO) under a real pending `callback_id`. After finalization, the Bitcoin management canister processes the fabricated response and updates its UTXO set with the fake transaction. The ckBTC minter then observes the fake UTXO via `get_utxos` and mints ckBTC to the attacker's account with no corresponding real Bitcoin deposit. This constitutes **illegal minting of a chain-key asset (ckBTC)**, matching the Critical impact class: "Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets."

## Likelihood Explanation
Every node on the Bitcoin-enabled subnet participates in block proposal rotation via the random beacon. A single compromised or malicious node operator â€” well below the consensus fault threshold â€” will eventually be selected as block proposer. No threshold cryptography, majority collusion, or admin key is required. The attacker only needs to: (1) read the certified state to obtain a live `callback_id`, and (2) craft a size-compliant `SelfValidatingPayload` with fabricated block data. This is trivially achievable by any node operator who controls the replica binary. The attack is repeatable every time the malicious node wins a proposal slot.

## Recommendation
`validate_self_validating_payload_impl` must load the certified state at `validation_context.certified_height` and, for each `BitcoinAdapterResponse` in the payload:
1. Verify that `response.callback_id` exists in `subnet_call_context_manager.bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`.
2. Verify that the response variant matches the request type stored for that `callback_id` (a `GetSuccessorsResponse` must correspond to a `GetSuccessorsRequest`; a `SendTransactionResponse` to a `SendTransactionRequest`).
3. Reject any payload containing duplicate `callback_id` values.

This mirrors the pattern already correctly implemented in `validate_canister_http_payload_impl`, which rejects any response with an unknown `callback_id`.

## Proof of Concept
1. Attacker controls one replica node on the Bitcoin-enabled subnet.
2. Attacker reads the certified state to obtain a live `callback_id` for a pending `GetSuccessorsRequest` (e.g., `callback_id = 42`).
3. When the attacker's node is selected as block proposer, it constructs a `SelfValidatingPayload` containing a `BitcoinAdapterResponse { callback_id: 42, response: GetSuccessorsResponse { blocks: [<fake_block_with_attacker_utxo>], next: [] } }`.
4. The block is proposed. Every other replica calls `validate_self_validating_payload_impl`; the payload passes because its byte size is within `MAX_BITCOIN_PAYLOAD_IN_BYTES`.
5. The block is notarized and finalized. `push_response_bitcoin` is called; the `callback_id = 42` lookup in `bitcoin_get_successors_contexts` succeeds (it is a real pending request), and the fabricated block is encoded and pushed to the consensus queue.
6. The Bitcoin management canister processes the fabricated `GetSuccessorsResponse`, updating its UTXO set with the fake transaction.
7. The attacker calls `update_balance` on the ckBTC minter; the minter observes the fake UTXO and mints ckBTC to the attacker's account.

A deterministic integration test can reproduce this by: constructing a `ReplicatedState` with a pending `GetSuccessorsRequest`, calling `validate_payload` on a `SelfValidatingPayload` containing a `GetSuccessorsResponse` with the matching `callback_id` but fabricated block bytes, asserting `Ok(())` is returned (demonstrating the missing check), then calling `push_response_bitcoin` and asserting the fabricated data reaches the consensus queue.
