### Title
Arithmetic Overflow in Fee Validation Emits Fatal `internal_error!`, Halting Entire Block Processing - (`basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, multiple user-controllable arithmetic overflow conditions are handled with `internal_error!` instead of `TxError::Validation(InvalidTransaction::OverflowPaymentInTransaction)`. Because `TxError::Internal` causes the transaction loop to `return Err(err)` — terminating the entire block — an unprivileged sender can halt block processing by submitting a single crafted transaction.

---

### Finding Description

`validate_and_compute_fee_for_transaction` in the Ethereum transaction flow contains at least four sites where a user-reachable arithmetic overflow is treated as a fatal internal error:

**Site 1 — ergs-counter overflow check (line 116–120):** [1](#0-0) 

```rust
require!(
    tx_gas_limit.saturating_mul(ERGS_PER_GAS) < u64::MAX,
    internal_error!("TX gas limit overflows ergs counter"),
    system
)?;
```

A user can set `gas_limit = u64::MAX` in any EIP-1559/EIP-2930/EIP-4844 transaction. `saturating_mul` returns `u64::MAX`, the condition is false, and `internal_error!` fires — **before** the `CallerGasLimitMoreThanTxLimit` validation check at line 129–133.

**Site 2 — `max_fee_per_gas × gas_limit` overflow (line 355–362):** [2](#0-1) 

```rust
let worst_case_fee_amount = {
    let (value, of) = u256_mul_by_word(transaction.max_fee_per_gas(), tx_gas_limit);
    if of > 0 {
        return Err(internal_error!("max gas price by tx gas limit").into());
    }
    value
};
```

A user sets `max_fee_per_gas` to any value `> U256::MAX / tx_gas_limit` (e.g., `U256::MAX` with `gas_limit = 2`). The product overflows U256, `of > 0`, and `internal_error!` fires. The existing `OverflowPaymentInTransaction` variant in `InvalidTransaction` is never used here.

**Sites 3 & 4 — `value + fee` and `effective_gas_price × gas_limit` overflows (lines 386–413):** [3](#0-2) 

Same pattern: `checked_add`/`u256_mul_by_word` failures return `internal_error!` instead of `OverflowPaymentInTransaction`.

**Why this halts the block:**

`From<InternalError> for TxError` converts every `internal_error!` to `TxError::Internal`: [4](#0-3) 

In the Ethereum transaction loop, `TxError::Internal` causes an unconditional `return Err(err)`, terminating block processing entirely: [5](#0-4) 

The same fatal path exists in the ZK transaction loop: [6](#0-5) 

By contrast, `TxError::Validation` is handled gracefully — the transaction is rejected and the loop continues: [7](#0-6) 

---

### Impact Explanation

An attacker submits a single transaction with `gas_limit = u64::MAX` (or `max_fee_per_gas` set to overflow U256 when multiplied by any non-zero gas limit). The transaction passes signature verification and all pre-IO checks, then triggers `internal_error!` in the fee calculation. This propagates as `TxError::Internal`, causing `return Err(err)` in the loop, which aborts block processing. No further transactions in the block are executed. The sequencer's block production is DoS'd for that block.

The `OverflowPaymentInTransaction` variant already exists in `InvalidTransaction` for exactly this purpose but is unused at these sites: [8](#0-7) 

---

### Likelihood Explanation

The attack requires only a valid ECDSA-signed transaction with an extreme `gas_limit` or `max_fee_per_gas` field — both are freely settable by any EOA. No privileged access, oracle manipulation, or governance is needed. The attacker pays only the transaction submission cost (which may be zero if the block halts before fee collection). The ergs-overflow path (Site 1) is the simplest: `gas_limit = u64::MAX` is a single-field change to any standard transaction.

---

### Recommendation

Replace every `internal_error!` at these overflow sites with the appropriate validation error:

```rust
// Site 1
require!(
    tx_gas_limit.saturating_mul(ERGS_PER_GAS) < u64::MAX,
    InvalidTransaction::CallerGasLimitTooHigh,  // or CallerGasLimitMoreThanTxLimit
    system
)?;

// Sites 2, 3, 4
if of > 0 {
    return Err(TxError::Validation(InvalidTransaction::OverflowPaymentInTransaction));
}
// checked_add failures:
.ok_or(TxError::Validation(InvalidTransaction::OverflowPaymentInTransaction))?;
```

The ZK validation path already demonstrates the correct pattern — it uses `OverflowPaymentInTransaction` for the equivalent balance overflow: [9](#0-8) 

---

### Proof of Concept

Craft an EIP-1559 transaction with `gas_limit = u64::MAX` signed by any funded EOA:

```rust
let tx = TxEip1559 {
    chain_id: <chain_id>,
    nonce: 0,
    gas_limit: u64::MAX,          // triggers saturating_mul overflow at line 117
    max_fee_per_gas: 1000,
    max_priority_fee_per_gas: 0,
    to: TxKind::Call(any_address),
    value: U256::ZERO,
    ..Default::default()
};
```

Submit this as the first transaction in a block. The call to `validate_and_compute_fee_for_transaction` reaches line 117, `u64::MAX.saturating_mul(ERGS_PER_GAS) == u64::MAX`, the `require!` condition is false, `internal_error!` fires, `TxError::Internal` propagates to `generic_loop_op`, and `return Err(err)` terminates block processing before any other transaction is executed.

Alternatively, set `max_fee_per_gas = U256::MAX` with `gas_limit = 2` to trigger the overflow at line 358 — this path is reached even if the ergs check is tightened, since `U256::MAX * 2` overflows U256.
