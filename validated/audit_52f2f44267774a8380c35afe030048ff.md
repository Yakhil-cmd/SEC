Audit Report

## Title
Global Base Cycles Limiter Exhaustion via Single Large Minting Call - (`rs/nns/cmc/src/limiter.rs`, `rs/nns/cmc/src/main.rs`)

## Summary
The CMC's `base_limiter` is a single shared global rolling-window counter with no per-caller isolation. Any unprivileged principal holding sufficient ICP can exhaust the entire hourly minting capacity in a single `notify_top_up` call, blocking all subsequent minting operations platform-wide for up to one hour. The attacker incurs near-zero net cost because the ICP is converted to cycles rather than destroyed.

## Finding Description
`State` holds a single `base_limiter: limiter::Limiter` with `max_age = Duration::from_secs(60 * 60)` (one hour) and `base_cycles_limit = Cycles::from(DEFAULT_CYCLES_LIMIT)` where `DEFAULT_CYCLES_LIMIT = 150e15 as u128` (150 trillion cycles).

`check_and_add_cycles` in `limiter.rs` performs a single global check:
```rust
if count + cycles_to_mint > limit { ... return Err(...) }
self.add(now, cycles_to_mint);
```
There is no per-caller cap, no per-transaction maximum, and no sub-limiter per principal.

All non-subnet-rental callers are routed to `CyclesMintingLimiterSelector::BaseLimit` in `notify_top_up`:
```rust
let limiter_to_use =
    if caller == src_canister_principal && canister_id.get() == src_canister_principal {
        CyclesMintingLimiterSelector::SubnetRentalLimit
    } else {
        CyclesMintingLimiterSelector::BaseLimit
    };
```

The call chain is: `notify_top_up` → `process_top_up` → `deposit_cycles` → `ensure_balance` → `limiter_to_use.check_and_add_cycles`. Inside `ensure_balance`:
```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;
```
Since the CMC is a pass-through canister with a near-zero balance, `cycles_to_mint ≈ cycles`. An attacker sending ICP equivalent to 150T cycles causes `check_and_add_cycles` to accept the full limit in one call (the check `0 + 150T > 150T` is false, so it passes). After this, every subsequent call from any user hits `count + cycles_to_mint > limit` and receives a rate-limit error until `purge_old` expires the window (up to one hour). The same `BaseLimit` path is used by `notify_mint_cycles` (via `do_mint_cycles` which hardcodes `CyclesMintingLimiterSelector::BaseLimit`) and `notify_create_canister` / `create_canister` (via `do_create_canister` which also hardcodes `BaseLimit`), so all minting endpoints are simultaneously blocked.

## Impact Explanation
This is a platform-level application DoS matching the High impact class: "Application/platform-level DoS... not based on raw volumetric DDoS." All users are blocked from minting cycles via any endpoint (`notify_top_up`, `notify_mint_cycles`, `notify_create_canister`, `create_canister`) for up to one hour per attack. The CMC is a critical NNS canister and its minting service is a core platform function. The attack is repeatable every hour with near-zero net cost.

## Likelihood Explanation
At 100 XDR/ICP and 1T cycles/XDR, minting 150T cycles requires ~1500 ICP. The attacker receives cycles in return, so the net cost is only transfer fees and opportunity cost — effectively zero. No privileged access, no governance participation, and no victim interaction is required. Any principal can execute this via a standard ledger transfer followed by a `notify_top_up` call. The attack is repeatable every hour indefinitely.

## Recommendation
1. Add a **per-principal sub-limiter** alongside the global limiter, capping how many cycles any single caller can mint per rolling window.
2. Alternatively, enforce a **per-call maximum** on `cycles_to_mint` independent of the global limit (e.g., reject any single call that would consume more than X% of the global limit).
3. Consider a tiered structure where large single-call mints (above a threshold) require a separate high-value limiter or governance approval.

## Proof of Concept
1. Attacker transfers ICP equivalent to `base_cycles_limit` (150T cycles at current rate) to the CMC subaccount for their canister.
2. Attacker calls `notify_top_up { block_index: <fresh_block>, canister_id: <attacker_canister> }`.
3. `ensure_balance` computes `cycles_to_mint ≈ 150T` (CMC balance ≈ 0); `check_and_add_cycles` evaluates `0 + 150T > 150T` → false → passes and records 150T in `base_limiter`.
4. Any subsequent call from any user to any minting endpoint evaluates `150T + N > 150T` → true → returns rate-limit error.
5. All minting is blocked until `purge_old` expires the window (up to one hour).
6. Reproducible as a PocketIC integration test: initialize CMC, fund attacker with 1500 ICP, call `notify_top_up`, then assert that a second caller's `notify_top_up` returns `NotifyError` containing the rate-limit message from `limiter.rs:48-52`.
