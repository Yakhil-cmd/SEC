Audit Report

## Title
Ticket Minimum-Participation Guard Checked Against Uncapped Increment Instead of Actual Credited Increment - (File: rs/sns/swap/src/swap.rs)

## Summary
In `refresh_buyer_token_e8s`, the ticket validation at line 1262 compares `amount_ticket` against `requested_increment_e8s` (the raw ledger balance increase) rather than `actual_increment_e8s` (the increment actually credited, capped by remaining swap capacity). When the swap is nearly full, a buyer's ticket is silently consumed even though their actual credited participation is less than the ticket amount, locking the excess ICP in the swap subaccount until finalization and preventing the buyer from creating a new ticket to retry.

## Finding Description
The function computes two distinct increment values:

- Line 1223: `let requested_increment_e8s = e8s - old_amount_icp_e8s;` — raw uncapped ledger increase.
- Line 1224: `let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);` — capped by `available_direct_participation_e8s()`.
- Line 1237: `let new_balance_e8s = std::cmp::min(new_balance_e8s, max_participant_icp_e8s);` — further capped per-participant.

The ticket guard at line 1262 reads:
```rust
if amount_ticket > requested_increment_e8s {
    return Err(...);
}
```

Because `requested_increment_e8s >= actual_increment_e8s` always holds, the guard passes (ticket deleted at line 1270) even when `amount_ticket > actual_increment_e8s`. The buyer is then credited with only `new_balance_e8s` (the capped amount), while the excess ICP remains in the swap subaccount. The ticket is removed from `OPEN_TICKETS_MEMORY`, so the buyer cannot create a new ticket for the same participation attempt.

**Concrete scenario:**
1. Swap has 40 ICP remaining capacity (`max_increment_e8s = 40`).
2. Buyer holds a valid open ticket for 80 ICP.
3. Buyer deposits 80 ICP; `refresh_buyer_token_e8s` is called.
4. `requested_increment_e8s = 80`, `actual_increment_e8s = min(40, 80) = 40`.
5. Check: `80 > 80` → false → ticket deleted.
6. Buyer is credited 40 ICP; 40 ICP is stranded in the subaccount until finalization.

The async await at line 1158–1163 (ICP ledger balance query) means the swap capacity can change between ticket creation and the check — a realistic race condition in any active SNS swap near its ceiling.

## Impact Explanation
This is a concrete SNS governance accounting bug reachable by any unprivileged direct participant. The buyer's open ticket is incorrectly consumed, their ICP participation is silently under-credited, and the excess ICP is locked in the swap subaccount until the swap reaches `Aborted` or `Committed` state (at which point `error_refund_icp` or `sweep_icp` can recover it). The buyer cannot create a new ticket during the `Open` phase to correct the shortfall. This constitutes a **significant SNS security impact with concrete user harm** — matching the High ($2,000–$10,000) impact tier for SNS framework issues, though the temporary (non-permanent) nature of the fund lock and the requirement for near-capacity swap conditions constrain it toward the lower end of that range, or Medium ($200–$2,000) for moderate user-funds impact.

## Likelihood Explanation
Medium. The condition requires: (a) the swap to be in the `Open` lifecycle, (b) the buyer to have an open ticket, and (c) the remaining swap capacity (`max_increment_e8s`) to be less than the buyer's deposited increment. Condition (c) is a natural race condition that occurs as a swap fills up near `max_direct_participation_icp_e8s`. The ticketing system is the recommended payment flow for SNS swap participants, making this reachable by any ordinary participant following standard instructions.

## Recommendation
Replace the check at line 1262 with a comparison against the actual credited increment:

```rust
// Before (incorrect):
if amount_ticket > requested_increment_e8s {

// After (correct):
if amount_ticket > actual_increment_e8s {
```

This ensures the ticket is only consumed when the actual credited participation meets or exceeds the ticket's stated amount, and returns an error otherwise so the buyer can retry after conditions change.

## Proof of Concept
A deterministic unit test in `rs/sns/swap/tests/swap.rs` can reproduce this:

1. Initialize a swap with `max_direct_participation_icp_e8s = 100 * E8` and `min/max_participant_icp_e8s` allowing 80 ICP.
2. Have a first participant fill 60 ICP so that `available_direct_participation_e8s() = 40 * E8`.
3. Create an open ticket for the second buyer with `amount_icp_e8s = 80 * E8`.
4. Mock the ICP ledger to return a balance of 80 ICP for the buyer's subaccount.
5. Call `refresh_buyer_token_e8s` for the buyer.
6. Assert: the call returns `Ok` (no error), the buyer's credited balance is 40 ICP (not 80), and `OPEN_TICKETS_MEMORY` no longer contains the buyer's ticket — confirming the ticket was consumed despite the participation shortfall.
