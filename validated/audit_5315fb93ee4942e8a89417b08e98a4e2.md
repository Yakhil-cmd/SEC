### Title
Missing Rollback for `new_settlement_layer_chain_id_storage` on Frame Revert — (`basic_system/src/system_implementation/system/io_subsystem.rs`)

### Summary

`FullIO::finish_io_frame` rolls back every sub-store that was snapshotted in `start_io_frame` **except** `new_settlement_layer_chain_id_storage`. If a transaction calls `update_settlement_layer_chain_id` and then reverts (EVM revert, out-of-native, or block-limit rollback), the settlement-layer chain-ID mutation persists in the live state, producing a permanently stale/incorrect chain-ID that was never committed by a successful transaction.

### Finding Description

`FullIOStateSnapshot` captures five sub-stores:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,   // ← interop roots ARE snapshotted
    // new_settlement_layer_chain_id is ABSENT
}
``` [1](#0-0) 

`start_io_frame` takes a snapshot of `interop_root_storage` but never calls `new_settlement_layer_chain_id_storage.start_frame()`:

```rust
fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
    let io = self.storage.start_frame();
    let transient = self.transient_storage.start_frame();
    let messages = self.logs_storage.start_frame();
    let events = self.events_storage.start_frame();
    let interop_roots = self.interop_root_storage.start_frame();
    // new_settlement_layer_chain_id_storage.start_frame() is NEVER called
    Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots })
}
``` [2](#0-1) 

`finish_io_frame` rolls back all five snapshotted stores but has no corresponding rollback for `new_settlement_layer_chain_id_storage`:

```rust
fn finish_io_frame(&mut self, rollback_handle: Option<&Self::StateSnapshot>) -> Result<(), InternalError> {
    self.storage.finish_frame(rollback_handle.map(|x| &x.io))?;
    self.transient_storage.finish_frame(rollback_handle.map(|x| &x.transient))?;
    self.logs_storage.finish_frame(rollback_handle.map(|x| x.messages));
    self.events_storage.finish_frame(rollback_handle.map(|x| x.events));
    self.interop_root_storage.finish_frame(rollback_handle.map(|x| x.interop_roots));
    // new_settlement_layer_chain_id_storage.finish_frame() is NEVER called
    Ok(())
}
``` [3](#0-2) 

The storage type itself **does** expose `start_frame`/`finish_frame` (confirmed by grep: 2 matches in `new_settlement_layer_chain_id_storage.rs`), so rollback support exists but is simply never wired into `FullIO`. [4](#0-3) 

The mutation path is:

```rust
fn update_settlement_layer_chain_id(
    &mut self, ..., new_sl_chain_id: U256,
) -> Result<(), SystemError> {
    ...
    self.new_settlement_layer_chain_id_storage.update(new_sl_chain_id)
}
``` [5](#0-4) 

Compare with `interop_root_storage`, which is structurally identical in purpose (cross-chain data appended during execution) but **is** correctly snapshotted and rolled back. The omission of `new_settlement_layer_chain_id_storage` from the snapshot is the root cause.

### Impact Explanation

A reverted transaction (EVM `REVERT`, out-of-native abort, or block-limit rollback via `finish_global_frame(Some(&handle))`) permanently writes a new settlement-layer chain ID into the live state. Because the chain ID is used for cross-chain message routing and protocol-level settlement, an incorrect value can:

- Cause subsequent cross-chain messages to be routed to the wrong settlement layer.
- Produce a forward-execution / proving divergence: the sequencer's forward run sees the stale chain ID while the prover, re-executing from the canonical pre-revert state, does not — breaking proof validity.
- Allow an attacker to "poison" the chain ID with a single cheap reverted transaction, permanently disrupting settlement-layer communication without any privileged access beyond the ability to invoke the relevant system hook.

### Likelihood Explanation

`update_settlement_layer_chain_id` is part of the public `IOSubsystem` trait and is reachable through a system hook callable from EVM contracts (system-hook callers are explicitly in scope per the Immunefi rules). The attacker only needs to:

1. Craft a transaction that calls the settlement-layer chain-ID system hook.
2. Arrange for the transaction to revert after the call (e.g., by consuming remaining gas or issuing an explicit `REVERT`).

No privileged key, governance majority, or operator access is required. The revert path is fully attacker-controlled.

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and wire its `start_frame`/`finish_frame` into `start_io_frame`/`finish_io_frame`, mirroring the existing pattern for `interop_root_storage`:

```rust
// In FullIOStateSnapshot
new_sl_chain_id: usize,   // snapshot handle

// In start_io_frame
let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();

// In finish_io_frame
self.new_settlement_layer_chain_id_storage
    .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id));
```

### Proof of Concept

1. Deploy an EVM contract that calls the settlement-layer chain-ID system hook with an arbitrary `new_sl_chain_id`, then immediately executes `REVERT`.
2. Submit the transaction. The bootloader calls `finish_global_frame(Some(&rollback_handle))`, rolling back storage, events, logs, transient storage, and interop roots — but **not** `new_settlement_layer_chain_id_storage`.
3. Query the settlement-layer chain ID after the block is sealed. It reflects the value written by the reverted transaction, confirming the stale state persists.
