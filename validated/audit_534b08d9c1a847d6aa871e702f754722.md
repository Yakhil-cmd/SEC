### Title
ckBTC Minter Burns User ckBTC Without Reimbursement When Batch Processing Fails with `AmountTooLow` or `DustOutput` - (File: `rs/bitcoin/ckbtc/minter/src/lib.rs`)

---

### Summary

The ckBTC minter burns a user's ckBTC at request-submission time. If the queued withdrawal request is later dropped during batch processing due to `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput`, the burned ckBTC is permanently lost with no reimbursement. This is an asymmetric accounting gap: the `BuildTxError::InvalidTransaction` path in the same function does issue a reimbursement, but the two fee-related failure paths do not.

---

### Finding Description

**Burn happens before processing.** In `retrieve_btc_with_approval`, the ckBTC burn is executed unconditionally before the request is enqueued: [1](#0-0) 

The request is then stored as `Pending`. Processing happens asynchronously in `submit_pending_requests`.

**Asymmetric error handling in `submit_pending_requests`.** Three `build_unsigned_transaction` error arms exist:

1. `BuildTxError::InvalidTransaction` → calls `reimburse_canceled_requests`, users recover their ckBTC minus a small fee. [2](#0-1) 

2. `BuildTxError::AmountTooLow` → calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` for **every request in the batch**. No reimbursement. [3](#0-2) 

3. `BuildTxError::DustOutput` → calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` for the offending request. No reimbursement. [4](#0-3) 

**`FinalizedStatus::AmountTooLow` carries no reimbursement record.** The enum has only two variants; neither triggers a mint-back: [5](#0-4) 

**The reimbursement infrastructure exists but is not wired.** `reimburse_canceled_requests` already handles the mint-back pattern and is called for `InvalidTransaction`: [6](#0-5) 

**How `AmountTooLow` / `DustOutput` become reachable after a valid submission.** The minimum withdrawal amount is computed from the current median fee rate and includes a buffer (`PER_REQUEST_RBF_BOUND = 22_100` sats): [7](#0-6) 

A request submitted when fees are low passes this check. If Bitcoin network fees spike before the batch is processed, `build_unsigned_transaction` can return `AmountTooLow` (total fee exceeds total output value) or `DustOutput` (a single output falls below the 546-sat dust limit after fee deduction): [8](#0-7) 

---

### Impact Explanation

A user who submitted a valid `retrieve_btc_with_approval` call during a low-fee period loses their entire ckBTC amount permanently if fees spike before the batch is processed. The ckBTC ledger supply is reduced (burn recorded) but no BTC is ever sent and no ckBTC is minted back. This is a direct, unrecoverable token loss for the affected user, analogous to "bad debt" that is silently absorbed with no accounting entry. The `RetrieveBtcStatusV2::AmountTooLow` terminal state gives the user no recourse. [9](#0-8) 

---

### Likelihood Explanation

Bitcoin fee spikes of 5–20× within a single processing window are historically observed (e.g., Ordinals inscription waves). The `fee_based_retrieve_btc_min_amount` is updated only when `estimate_fee_per_vbyte` is called during `submit_pending_requests`; a request submitted just before a spike and processed just after is exposed. The `DustOutput` path is reachable even with moderate fee increases for requests near the minimum. The entry point (`retrieve_btc_with_approval`) is fully open to any unprivileged caller.

---

### Recommendation

Wire `AmountTooLow` and `DustOutput` through `reimburse_canceled_requests` exactly as `InvalidTransaction` does, charging a small reimbursement fee to cover minter work already performed. The `reimbursement_account` field is already populated for `retrieve_btc_with_approval` requests and the reimbursement infrastructure (`reimburse_withdrawal`, `ReimburseWithdrawalTask`) is already in place. [10](#0-9) 

---

### Proof of Concept

1. Bitcoin mainnet fees are low (e.g., 5 sat/vB). `fee_based_retrieve_btc_min_amount` = 50,000 sats.
2. User calls `retrieve_btc_with_approval(amount=50_000, address=<valid P2WPKH>)`. ckBTC burn is recorded on the ledger.
3. Before `submit_pending_requests` fires, a fee spike raises the median to 200 sat/vB.
4. `build_unsigned_transaction` computes `fee ≈ 200 * 141 vbytes = 28,200 sats` + `minter_fee ≈ 300 sats` = 28,500 sats. For a single-output transaction the output value after fee deduction = `50,000 − 28,500 = 21,500 sats`, which is above dust. However if the fee rate spikes further (e.g., 400 sat/vB → fee ≈ 56,400 sats > 50,000 sats), `BuildTxError::AmountTooLow` is returned.
5. `submit_pending_requests` enters the `AmountTooLow` arm, calls `remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow)` — no `reimburse_canceled_requests` call.
6. The user's 50,000 sats of ckBTC are permanently burned. `retrieve_btc_status_v2(block_index)` returns `AmountTooLow` with no `WillReimburse` or `Reimbursed` follow-up. [3](#0-2)

fn reimburse_canceled_requests<R: CanisterRuntime>(
    state: &mut state::CkBtcMinterState,
    requests: BTreeSet<state::RetrieveBtcRequest>,
    reason: WithdrawalReimbursementReason,
    total_fee: u64,
    runtime: &R,
) {
    assert!(!requests.is_empty());
    let fees = distribute(total_fee, requests.len() as u64);
    // This assertion makes sure the fee is smaller than each request amount
    assert!(
        fees[0] <= state.retrieve_btc_min_amount,
        "BUG: fees {fees:?} for {} withdrawal requests are larger than `retrieve_btc_min_amount` {}",
        requests.len(),
        state.retrieve_btc_min_amount
    );
    for (request, fee) in requests.into_iter().zip(fees.into_iter()) {
        if let Some(account) = request.reimbursement_account {
            let amount = request.amount.saturating_sub(fee);
            if amount > 0 {
                state::audit::reimburse_withdrawal(
                    state,
                    request.block_index,
                    amount,
                    account,
                    reason.clone(),
                    runtime,
                );
            }
        } else {
            log!(
                Priority::Info,
                "[reimburse_canceled_requests]: account is not found for retrieve_btc request ({:?})",
                request
            );
        }
    }
}
