### Title
VIP Caller Starvation via Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

### Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` rejects all callers — including VIP (NNS canister) callers — when `available_slot_count == 0`, even though VIP callers are designed to consume zero slots. An unprivileged attacker can exhaust all 167 slots via the public `canister_status` endpoint, blocking NNS Governance from executing `update_canister_settings`, `take_canister_snapshot`, and `load_canister_snapshot` through the NNS root canister.

### Finding Description
In `try_borrow_slot`, the zero-slot guard is evaluated before the VIP/non-VIP distinction takes effect:

```rust
let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

self.available_slot_count.with_borrow_mut(|available_slot_count| {
    if *available_slot_count == 0 {   // ← fires for VIP too
        return Err(...);
    }
    *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
    Ok(())
})?;
```

The `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` thread-local is shared across all endpoints that call `new_management_canister_client()`. The public `canister_status` endpoint (no access control) and the governance-gated endpoints (`update_canister_settings`, `take_canister_snapshot`, `load_canister_snapshot`) all draw from the same pool of 167 slots.

### Impact Explanation
While the attack is sustained, NNS Governance's calls to `update_canister_settings`, `take_canister_snapshot`, and `load_canister_snapshot` through NNS root fail with `SysTransient`. These are legitimate NNS proposal execution paths. The error is transient and retryable, but an attacker who continuously replenishes 167 concurrent `canister_status` calls can sustain the blockade indefinitely, delaying or preventing time-sensitive NNS governance operations.

### Likelihood Explanation
The `canister_status` endpoint is intentionally public. Any canister on the IC can make 167 concurrent inter-canister calls to it. The IC's message queue limit is ~500; 167 is well within reach. The management canister responds quickly, so the attacker must continuously re-flood, but this is straightforward for an automated attacker canister.

### Recommendation
Move the zero-slot guard inside the non-VIP branch:

```rust
self.available_slot_count.with_borrow_mut(|available_slot_count| {
    if !self.is_caller_vip && *available_slot_count == 0 {
        return Err((RejectCode::SysTransient as i32, "Unavailable. Maybe, try again later?".to_string()));
    }
    *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
    Ok(())
})?;
```

This ensures VIP callers are never rejected by non-VIP slot exhaustion, preserving the stated invariant.

### Proof of Concept
State-machine test:
1. Initialize `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Spawn 167 concurrent non-VIP `canister_status` calls that hold their management canister call open (e.g., via a mock that does not resolve immediately).
3. Assert `available_slot_count == 0`.
4. Issue a VIP call (`is_caller_vip = true`) to `canister_status` (or `update_canister_settings`).
5. Assert the result is `Err((SysTransient, _))` despite `is_caller_vip = true` and `used_slot_count = 0`.

The existing test suite in `management_canister_client/tests.rs` already demonstrates VIP calls succeed when slots are partially consumed, but does not test the `available_slot_count == 0` edge case for VIP callers. [7](#0-6)
