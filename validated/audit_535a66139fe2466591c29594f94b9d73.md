### Title
`require_running` Guard in `exit_to_near_precompile_callback` Permanently Freezes User Funds When Engine Is Paused — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The `exit_to_near` flow is a two-step cross-contract operation. Step 1 (EVM execution) burns the user's ERC-20 tokens and schedules an `ft_transfer_call` promise. Step 2 (`exit_to_near_precompile_callback`) handles the result — either transferring NEAR to the user or refunding the burned tokens. The callback unconditionally checks `require_running` before any recovery logic. If the engine is paused between steps 1 and 2, the callback fails entirely: neither the NEAR transfer nor the refund executes, and the user's tokens are permanently lost.

---

### Finding Description

The `exit_to_near_precompile_callback` function in `engine/src/contract_methods/connector.rs` is the completion handler for the `exit_to_near` precompile. It is called as a private NEAR callback after the connector's `ft_transfer_call` resolves. [1](#0-0) 

The function begins with:

```rust
let state = state::get_state(&io)?;
require_running(&state)?;   // line 203 — blocks ALL paths below
env.assert_private_call()?;
``` [2](#0-1) 

Below that gate, the function branches on the promise result:

- **Success path** — transfers NEAR to the user.
- **Failure path** — calls `engine::refund_on_error` to re-mint the burned ERC-20 tokens. [3](#0-2) 

Both paths are unreachable if `require_running` fails. The ERC-20 burn that happened in step 1 is already committed to the blockchain. There is no retry mechanism, no admin recovery function, and no way to re-enter the callback after the engine is unpaused — NEAR promise callbacks execute exactly once.

The asymmetry mirrors the Steadefi bug precisely:

| | Steadefi | Aurora Engine |
|---|---|---|
| Step 1 (succeeds) | Repay loan to lending vault | Burn ERC-20 tokens in EVM |
| Step 2 (blocked) | Borrow back from paused vault | Callback blocked by `require_running` |
| Result | Vault stuck in `Withdraw_Failed` | User tokens permanently burned |

The engine's pause mechanism is managed by `EnginePrecompilesPauser` and the engine's running state, both of which are legitimate operational controls: [4](#0-3) 

The callback is already protected against external callers by `env.assert_private_call()`, so `require_running` provides no additional security benefit here — it only blocks legitimate in-flight recovery.

The same structural flaw exists in `withdraw_wnear_to_router`, where `require_running` blocks the wNEAR-unwrap callback after the user's wNEAR has already been transferred to the engine's implicit EVM address: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The ERC-20 tokens burned in step 1 are committed state. If `exit_to_near_precompile_callback` fails, neither the NEAR transfer nor the ERC-20 refund executes. The user receives nothing. There is no admin function to manually complete or reverse the operation. Funds are permanently lost.

For the XCC variant (`withdraw_wnear_to_router`), the user's wNEAR is transferred to the engine's implicit EVM address — an address for which no private key exists and which has no withdrawal mechanism — making those funds equally unrecoverable.

---

### Likelihood Explanation

NEAR's asynchronous execution model guarantees a window between the EVM transaction (step 1) and the callback (step 2). These execute in separate NEAR blocks. Any pause that occurs in that window — including routine maintenance pauses — triggers the vulnerability. The engine's pause capability is an intended operational feature, making this a realistic scenario whenever maintenance coincides with in-flight `exit_to_near` or XCC operations.

---

### Recommendation

Remove `require_running` from callbacks that complete already-initiated operations (`exit_to_near_precompile_callback`, `withdraw_wnear_to_router`). These functions are already protected against external callers by `env.assert_private_call()`. Alternatively, introduce a separate "completing in-flight operations" state that is not blocked by the engine pause, analogous to how the Steadefi recommendation was to allow the rollback borrow to bypass the pause restriction.

---

### Proof of Concept

1. User submits an EVM transaction calling the `exit_to_near` precompile. Their ERC-20 tokens are burned. The `ft_transfer_call` promise to the connector is scheduled.
2. Admin pauses the engine for maintenance (legitimate operational action).
3. The connector's `ft_transfer_call` executes in the next NEAR block.
4. `exit_to_near_precompile_callback` is invoked as the callback.
5. `require_running(&state)?` at line 203 of `engine/src/contract_methods/connector.rs` returns `ERR_RUNNING`.
6. The function returns an error. Neither the NEAR transfer nor `engine::refund_on_error` is called.
7. The user's ERC-20 tokens are permanently burned. No NEAR is received. No refund is issued. [1](#0-0)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
