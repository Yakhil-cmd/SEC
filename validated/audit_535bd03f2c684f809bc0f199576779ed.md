### Title
L1→L2 Deployment Transactions Silently Execute as Call to Address Zero Instead of Deploying — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The ZKsync OS bootloader processes L1→L2 (`0x7f`) transactions but does not implement the contract deployment path. When an L1→L2 transaction carries `to = B160::ZERO` (the conventional signal for a deployment), the code silently executes it as a regular EVM call to address `0x0000…0000` instead of performing a contract deployment. The transaction is reported as successful, no contract is ever deployed, and the user's gas deposit is consumed.

---

### Finding Description

In `execute_l1_transaction_and_notify_result` the code reads `from` and `to` from the transaction and unconditionally passes them to `run_single_interaction`:

```rust

        system,
        system_functions,
        memories.reborrow(),
        calldata,
        &from,
        &to,          // ← always used as-is; B160::ZERO is never treated as "deploy"
        resources_for_tx,
        &value,
        false,
        tracer,
        validator,
    )
``` [1](#0-0) 

The `AbiEncodedTransaction` struct documents `reserved[1]` as the "EVM deployment transaction flag (`to` == null in such case)", confirming that `to = B160::ZERO` is the intended encoding for a deployment: [2](#0-1) 

`validate_structure` imposes no constraint that prevents `to = B160::ZERO` for `L1_L2_TX_TYPE` or `UPGRADE_TX_TYPE`: [3](#0-2) 

By contrast, the Ethereum STF (`EthereumTransactionFlow`) correctly branches on `transaction.to().is_some()` and calls `perform_deployment` when `to` is absent: [4](#0-3) 

The ZK transaction flow has no equivalent branch. `run_single_interaction` with `to = B160::ZERO` resolves to a call against an empty account, which the runner returns as `CallResult::Successful` with empty returndata: [5](#0-4) 

Because the call succeeds, `is_success = true`, the full post-execution accounting runs (operator fee paid, refund issued), and the transaction is committed to state — with no contract ever deployed.

---

### Impact Explanation

An L1→L2 transaction submitted with `to = B160::ZERO` and deployment calldata:

1. Has the user's L1 deposit minted to the sender address.
2. Executes a call to `0x0000…0000` (empty account) — succeeds silently.
3. Pays the operator `gas_used × gas_price` from the deposit.
4. Refunds the remainder to `reserved[1]` (refund recipient).
5. Emits an L1→L2 log marking the transaction as **successful**.

The intended contract is never deployed. The user loses the gas cost of the transaction and receives no contract at the expected address. Because the transaction is marked successful on both L1 and L2, the user has no on-chain signal that the deployment failed. Any protocol or application logic that depends on the contract being present at the derived address will malfunction silently.

---

### Likelihood Explanation

L1→L2 transactions are initiated by users calling the L1 mailbox/bridge contracts. If those contracts permit `to = 0` (deployment intent), any user can trigger this path without any privileged access. The TODO comment and the struct documentation confirm this is a planned but unimplemented feature, meaning the encoding is already defined and the L1-side ABI is expected to support it. The ZKsync OS side simply never dispatches to the deployment branch.

---

### Recommendation

In `execute_l1_transaction_and_notify_result`, add a branch that checks whether `to == B160::ZERO` (or the `reserved[1]` deployment flag) and, when true, dispatches to the same `perform_deployment` logic used by `EthereumTransactionFlow`. Until this is implemented, the L1 contracts should explicitly reject deployment-intent L1→L2 transactions (`to = 0`) to prevent silent misbehaviour.

---

### Proof of Concept

1. Craft an `AbiEncodedTransaction` with `tx_type = 0x7f`, `to = B160::ZERO`, non-empty `data` (deployment bytecode), and a valid `reserved[0]` (deposit amount).
2. Submit via the L1 mailbox so it enters the ZK transaction loop.
3. Observe in `execute_l1_transaction_and_notify_result` that `to = B160::ZERO` is passed directly to `run_single_interaction`.
4. The runner resolves the callee as an empty account (`NoEE`, zero bytecode length) and returns `CallResult::Successful`.
5. Post-execution accounting commits the transaction as successful; no contract exists at the address the user expected. [6](#0-5)
