### Title
Unchecked Return Values of EIP-7002/7251/6110 System Functions Silently Corrupt `requests_hash` Block Header Field - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/post_tx_op_sequencing.rs`)

### Summary
In `post_tx_op_sequencing.rs`, the return values of `eip6110_events_parser`, `eip7002_system_part`, and `eip7251_system_part` are all discarded with `let _ =`. Each function returns `Result<bool, SystemError>` and mutates a shared `requests_hasher`. If any function fails mid-execution (after partially writing to the hasher), the error is silently swallowed and execution continues, producing a corrupted `requests_hash` that is committed into the sealed block header.

### Finding Description

In `post_tx_op_sequencing.rs` lines 90–92, the post-block processing step calls three system functions that are responsible for computing the EIP-7685 `requests_hash`:

```rust
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);
```

All three functions share the same `requests_hasher` and write into it incrementally. Each function returns `Result<bool, SystemError>`, but the result is discarded entirely via `let _ =`.

The critical failure mode is in `eip7002_system_part` and `eip7251_system_part`. Both functions:

1. Write a type-prefix byte into the hasher **before** the processing loop (e.g., `requests_hasher.update([WITHDRAWAL_REQUEST_EIP_7685_TYPE])` at line 177 of `eip_7002_withdrawal_contract/mod.rs`).
2. Then iterate over queue entries, reading storage slots and writing request data into the hasher.
3. Also perform storage writes (queue head pointer updates, excess count resets) using `?` which propagates errors as `Err(...)`.

If a storage read or write fails mid-loop, the function returns `Err(...)`. Because the caller discards this with `let _ =`, the hasher is left in a **partially-updated state**: the type byte was written but not all request data. The final `requests_hash` computed from this corrupted hasher is then committed into the block header via `result_keeper.record_sealed_block(metadata.block_level.header)`.

Additionally, if `eip7002_system_part` or `eip7251_system_part` fails after some storage writes (e.g., queue head pointer was updated but excess count reset was not), the storage state of the predeploy contracts is left inconsistent, with no rollback. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The `requests_hash` is a consensus-critical field in the Ethereum block header (EIP-7685). If ZKsync OS computes an incorrect `requests_hash` due to a partially-updated hasher, the sealed block header will not match what the Ethereum consensus layer expects. This constitutes a **state-transition bug**: the ZKsync OS execution output diverges from the canonical Ethereum state transition, causing:

- Blocks produced by ZKsync OS to be rejected by Ethereum consensus (invalid `requests_hash`).
- Proofs generated over such blocks to be invalid or unprovable.
- Potential storage state corruption in the EIP-7002/7251 predeploy contracts (inconsistent queue head pointers and excess counts), which persists across blocks and compounds over time. [4](#0-3) 

### Likelihood Explanation

The comment on line 89 reads `"Environment may have no such contracts predeployed for tests or sequencing purposes"`, indicating the `let _ =` was introduced to tolerate missing predeploy contracts. However, in a production Ethereum-mode environment, these contracts are always deployed (EIP-7002 and EIP-7251 are Pectra fork requirements). Any storage I/O error during queue processing — which can be triggered by a user submitting withdrawal or consolidation requests that fill the queue — will silently corrupt the hasher and the resulting block header. The entry path is fully unprivileged: any user can call the EIP-7002 withdrawal request contract to enqueue a request. [5](#0-4) 

### Recommendation

Replace the `let _ =` discards with proper error propagation using `?`:

```rust
eip6110_events_parser(&system, &mut requests_hasher)?;
eip7002_system_part(&mut system, &mut requests_hasher)?;
eip7251_system_part(&mut system, &mut requests_hasher)?;
```

If tolerance for missing predeploy contracts is required in non-production environments, introduce an explicit check for the specific `SystemError::LeafDefect` variant (contract not deployed) and handle only that case gracefully, while propagating all other errors. [6](#0-5) 

### Proof of Concept

1. Deploy the EIP-7002 withdrawal request predeploy contract at `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS`.
2. Submit enough withdrawal requests to populate the queue (any unprivileged user can do this by calling the predeploy contract with ETH).
3. Trigger a block seal. During `post_op`, `eip7002_system_part` is called.
4. If a storage I/O error occurs after `requests_hasher.update([WITHDRAWAL_REQUEST_EIP_7685_TYPE])` (line 177) but before all queue entries are hashed, the function returns `Err(...)`.
5. The `let _ =` at line 91 discards the error. The hasher now contains only the type byte `0x01` without the actual request data.
6. `requests_hash` is finalized from the corrupted hasher and committed into the block header at line 105.
7. The resulting block has an incorrect `requests_hash`, causing consensus failure. [7](#0-6) [8](#0-7)
