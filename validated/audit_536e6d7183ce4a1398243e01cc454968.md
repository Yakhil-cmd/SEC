### Title
Payable `execute_scheduled` in XCC Router Silently Locks Attached NEAR Permanently - (File: `etc/xcc-router/src/lib.rs`)

### Summary
The `execute_scheduled` function in the XCC router contract is marked `#[payable]` but never reads or uses `env::attached_deposit()`. Any NEAR tokens attached to a call of this function are permanently locked in the router contract with no recovery path.

### Finding Description
In `etc/xcc-router/src/lib.rs`, the `execute_scheduled` function is decorated with `#[payable]`:

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The `#[payable]` attribute in NEAR SDK means the function accepts an attached NEAR deposit without reverting. However, the function body never calls `env::attached_deposit()` and never forwards or refunds the attached amount. Any NEAR sent with this call is silently absorbed into the router contract's balance.

The only fund-recovery mechanism in the router is `send_refund`, which:
1. Is restricted to the parent account (Aurora Engine) only via `require_caller(&parent)`.
2. Sends a hardcoded fixed amount (`REFUND_AMOUNT = NearToken::from_near(2)`) to the parent — it does not recover arbitrary locked NEAR.

There is no general withdrawal function. The router contract has no `receive` or fallback that returns excess deposits.

Critically, the comments in the code explicitly state this function is intentionally open to anyone:

> "It is intentional that this function can be called by anyone (not just the parent)."

This means any unprivileged NEAR account can call `execute_scheduled` with an arbitrary NEAR deposit, and that deposit is permanently locked.

### Impact Explanation
Any NEAR tokens attached to a call to `execute_scheduled` are permanently frozen in the XCC router contract. There is no mechanism for the depositor or any other party to recover them. This constitutes **permanent freezing of funds**.

### Likelihood Explanation
`execute_scheduled` is intentionally callable by any NEAR account with no access control. A user who wants to trigger execution of a scheduled XCC promise might reasonably attach NEAR thinking it is needed to fund the underlying promise execution (since the scheduled promise itself may carry an `attached_balance`). The open nature of the function and the absence of any revert on non-zero deposit makes accidental fund loss realistic.

### Recommendation
Either:
1. Remove the `#[payable]` attribute from `execute_scheduled` so the NEAR runtime automatically rejects any non-zero attached deposit, or
2. Add an explicit assertion at the start of the function body: `assert_eq!(env::attached_deposit(), NearToken::from_yoctonear(0), "ERR_UNEXPECTED_DEPOSIT");`

The same review should be applied to `deploy_upgrade`, which is also `#[payable]` and does not forward `env::attached_deposit()` to the `initialize` call (it passes `NearToken::default()` instead).

### Proof of Concept
1. An XCC router contract is deployed for a user's EVM address (e.g., `{address}.aurora`).
2. The Aurora Engine schedules a promise via `schedule`, storing it at nonce `0`.
3. An unprivileged attacker calls `execute_scheduled` on the router with `nonce=0` and attaches `10 NEAR`.
4. The NEAR runtime accepts the call (because `#[payable]` is present), the promise executes, and `10 NEAR` is absorbed into the router's balance.
5. The depositor has no way to recover the `10 NEAR`. The parent (Aurora Engine) can only call `send_refund` which transfers a fixed `2 NEAR` to itself — not the depositor's funds. [1](#0-0) [2](#0-1) [3](#0-2)
