### Title
SNS Treasury Manager `DepositRequest` Lacks Slippage Parameters, Enabling Sandwich Attacks on DAO Liquidity Deposits — (`rs/sns/treasury_manager/src/lib.rs`, `rs/sns/governance/src/extensions.rs`)

---

### Summary

The SNS Treasury Manager extension framework allows SNS DAOs to deposit treasury funds into external DEX liquidity pools via governance proposals. The `DepositRequest` type contains only `allowances` (amounts to send) and has no slippage parameters (no minimum LP tokens to receive, no minimum token amounts). The `execute_treasury_manager_deposit` function approves and sends funds but never validates what the treasury received in return. Because governance proposals are public and take days to pass, an attacker can observe a pending deposit proposal and execute a sandwich attack, causing the SNS treasury to receive far fewer LP tokens than expected.

---

### Finding Description

The `DepositRequest` struct in `rs/sns/treasury_manager/src/lib.rs` is defined as:

```rust
pub struct DepositRequest {
    pub allowances: Vec<Allowance>,
}
```

It carries only the amounts to be approved — no minimum received LP tokens, no minimum token ratios, no slippage bounds. [1](#0-0) 

The `execute_treasury_manager_deposit` function in `rs/sns/governance/src/extensions.rs`:
1. Calls `approve_treasury_manager` to grant ICRC-2 allowances to the extension canister.
2. Calls `deposit` on the treasury manager canister.
3. Only checks whether the call returned `Ok` — it never inspects the `Balances` response to verify that the treasury received a fair amount of LP tokens. [2](#0-1) 

The `validate_deposit_operation_impl` function only validates that the requested amounts do not exceed 50% of the current treasury balance. It performs no slippage validation whatsoever. [3](#0-2) 

The `DepositRequest` interface in the DID specification also has no slippage fields, making it structurally impossible for any treasury manager implementation to receive slippage parameters from the SNS governance canister. [4](#0-3) 

The codebase itself acknowledges this gap as a "Known Security Risk":

> *"Some liquidity pools do not implement slippage protection for deposits. In other words, the price ratio at the time of execution may differ from the ratio at the time the proposal was approved."* [5](#0-4) 

The same warning is rendered in the proposal text for `RegisterExtension`, but no on-chain enforcement exists. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a pending `ExecuteExtensionOperation` (deposit) governance proposal can execute a sandwich attack:

1. **Before proposal execution**: Attacker manipulates the DEX pool price by depositing a large imbalanced amount, inflating the price of one token relative to the other.
2. **Proposal executes**: The SNS treasury deposit goes through at the manipulated price, receiving far fewer LP tokens than the DAO expected at proposal creation time.
3. **After execution**: Attacker withdraws their position, restoring the pool price and pocketing the difference.

Because the `DepositRequest` carries no minimum LP token floor, the treasury manager canister has no protocol-level basis to reject the deposit even if the price has been severely manipulated. The SNS treasury suffers a direct, permanent loss of funds — the difference between the fair LP token value and the manipulated LP token value is extracted by the attacker.

This is a **ledger conservation bug**: SNS DAO treasury assets are transferred out but the corresponding LP token value received is not enforced, allowing value to be silently drained.

---

### Likelihood Explanation

- SNS governance proposals are fully public and have a multi-day voting period, giving attackers ample time to prepare.
- The IC's deterministic execution model means the attacker can predict the exact round in which the proposal will execute.
- The attacker needs capital to manipulate the DEX pool, but this is a realistic requirement for any well-funded actor targeting a DAO treasury deposit.
- No privileged access, admin key, or threshold corruption is required — the attacker is an unprivileged canister caller or boundary user.
- The attack is directly analogous to the Salty.IO M-28 finding: in both cases, the slippage protection mechanism is structurally absent or ineffective, and the first (or any) deposit is vulnerable to frontrunning.

---

### Recommendation

1. **Add slippage parameters to `DepositRequest`**: Extend the type with fields such as `min_lp_tokens_received: Option<Nat>` or `min_amount0_received`/`min_amount1_received`, mirroring the Uniswap V3 approach recommended in the external report.

2. **Enforce slippage in `execute_treasury_manager_deposit`**: After calling `deposit`, inspect the returned `Balances` to verify that the `external_custodian` balance meets the minimum LP token floor specified in the proposal. Revert (or log a governance failure) if the received amount is below the threshold.

3. **Validate slippage parameters in `validate_deposit_operation_impl`**: Require that any deposit proposal includes a non-zero minimum received amount, so that proposals without slippage protection are rejected at submission time.

---

### Proof of Concept

**Setup**: An SNS DAO passes an `ExecuteExtensionOperation` proposal to deposit 1,000,000 SNS tokens and 1,000,000 ICP into a KongSwap liquidity pool at an expected 1:1 ratio.

**Attack**:
1. Attacker observes the pending proposal (public on-chain).
2. Before the proposal executes, attacker calls the DEX directly and deposits 10,000,000 ICP with only 1 SNS token, severely skewing the pool to make SNS tokens appear extremely valuable relative to ICP.
3. The governance proposal executes: `approve_treasury_manager` grants allowances, then `deposit` is called. The treasury manager deposits into the now-skewed pool and receives LP tokens representing a position worth far less than 2,000,000 tokens at fair market value.
4. `execute_treasury_manager_deposit` receives `Ok(balances)` and logs success — it never checks the LP token amount received. [7](#0-6) 
5. Attacker withdraws their skewed position, restoring the pool price and extracting the value difference from the SNS treasury.

The `DepositRequest` sent to the treasury manager contains only `allowances` with no minimum LP floor, so the treasury manager has no protocol-level basis to abort. [1](#0-0)

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager deposit operation
async fn execute_treasury_manager_deposit(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedDepositOperationArg,
) -> Result<(), GovernanceError> {
    let ValidatedDepositOperationArg {
        treasury_allocation_sns_e8s,
        treasury_allocation_icp_e8s,
        original,
    } = arg;

    let context = governance.treasury_manager_deposit_context().await?;
    let arg_blob =
        construct_treasury_manager_deposit_payload(context, original).map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Failed to construct treasury manager deposit payload: {err}"),
            )
        })?;

    // 1. Transfer funds from treasury to treasury manager
    governance
        .approve_treasury_manager(
            extension_canister_id,
            treasury_allocation_sns_e8s,
            treasury_allocation_icp_e8s,
        )
        .await?;

    // 2. Call deposit on treasury manager
    let balances = governance
        .env
        .call_canister(extension_canister_id, "deposit", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.deposit failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error decoding TreasuryManager.deposit response: {err:?}"),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.deposit failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.deposit succeeded with response: {:?}",
        balances
    );

    Ok(())
}
