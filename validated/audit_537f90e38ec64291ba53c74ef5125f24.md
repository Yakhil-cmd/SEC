Audit Report

## Title
Treasury Deposit 50% Balance Check Not Re-Evaluated at Execution Time Enables Balance-Inflation Bypass - (`rs/sns/governance/src/extensions.rs`)

## Summary
`validate_deposit_operation_impl` enforces a 50% treasury balance cap at proposal validation time by querying live balances, but `execute_treasury_manager_deposit` never re-evaluates this limit before granting ICRC-2 approval and calling `deposit`. Because ICRC-1 ledgers permit any principal to transfer tokens to the deterministic SNS treasury subaccount, an unprivileged actor can inflate the treasury balance before proposal submission, causing the 50% check to pass for an amount that exceeds 50% of the treasury's genuine pre-inflation holdings. A concurrent governance proposal that reduces the treasury balance between validation and execution creates the same violation silently.

## Finding Description
`validate_deposit_operation_impl` (lines 276â€“321) fetches live SNS and ICP treasury balances and rejects any deposit proposal whose requested amounts exceed 50% of those balances:

```rust
if sns_requested > sns_balance.checked_div(2).unwrap() {
    return Err(format!(
        "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
    ));
}
``` [5](#0-4) 

This validation runs only at proposal submission time via `validate_deposit_operation` â†’ `validate_execute_extension_operation`. After the governance voting period, `ValidatedExecuteExtensionOperation::execute` dispatches to `execute_treasury_manager_deposit`, which directly calls `approve_treasury_manager` and then `deposit` on the treasury manager canister with no re-check: [6](#0-5) 

The SNS treasury subaccount is deterministic and publicly computable from the governance canister ID: [7](#0-6) 

**Balance-inflation bypass:** An attacker holding SNS tokens transfers them directly to the treasury subaccount before a deposit proposal is submitted. The inflated balance causes the 50% check to pass for an amount exceeding 50% of the real treasury. The governance majority, relying on the automated check, approves the proposal without realizing the balance was artificially inflated. The attacker's donated tokens remain in the treasury and are not deposited, but the approved deposit now represents more than 50% of the original holdings.

**TOCTOU:** The code comment itself acknowledges the race: *"The balance could have changed since the proposal was created, and we don't assume that the proposal should work."* A concurrent `TransferSnsTreasuryFunds` proposal that reduces the treasury balance between validation and execution causes the deposit to silently exceed the 50% limit at execution time. [8](#0-7) 

## Impact Explanation
The 50% cap is the only on-chain guard preventing a single governance proposal from depositing the majority of the SNS treasury into a DEX. Bypassing it allows governance to approve depositing, for example, 60% of the real treasury into a DEX. Once the ICRC-2 approval is granted and `deposit` is called, the funds are committed. Recovery requires a separate withdrawal governance proposal, during which the DEX position is exposed to price movement and front-running. This constitutes a significant SNS governance security impact with concrete harm to canister-controlled treasury funds â€” matching the High impact category: *"Significant SNS or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
The balance-inflation path requires only an ICRC-1 `icrc1_transfer` call to the treasury subaccount â€” available to any SNS token holder with no governance role. The attacker permanently donates tokens (a real cost), but does not need voting power. The governance majority is deceived because the automated 50% check passes and they have no on-chain signal that the balance was inflated. The TOCTOU path requires no attacker at all â€” it occurs naturally whenever two treasury-affecting proposals overlap, which is a realistic operational scenario.

## Recommendation
Re-evaluate the 50% limit at execution time inside `execute_treasury_manager_deposit` by querying the live treasury balance immediately before calling `approve_treasury_manager`. This mirrors the existing check pattern and closes both the inflation bypass and the TOCTOU window. Alternatively, snapshot the treasury balance at proposal creation time, store it in `ValidatedDepositOperationArg`, and reject execution if the current balance has fallen below the snapshot value or if the requested amount now exceeds 50% of the current balance.

## Proof of Concept
1. SNS treasury holds 100 SNS tokens (real balance).
2. Attacker calls `icrc1_transfer` on the SNS ledger, sending 30 SNS tokens to the treasury subaccount. Treasury balance is now 130 SNS.
3. A governance participant submits a deposit proposal with `treasury_allocation_sns_e8s = 60_000_000` (60 SNS).
4. `validate_deposit_operation_impl` fetches balance = 130 SNS; `60 â‰¤ 65 = 130/2` â†’ check passes.
5. Proposal enters the voting period and passes.
6. `execute_treasury_manager_deposit` calls `approve_treasury_manager` granting the treasury manager an ICRC-2 allowance of 60 SNS, then calls `deposit`.
7. 60 SNS tokens are deposited into the DEX â€” representing 60% of the original 100 SNS treasury, violating the intended 50% cap.
8. The attacker's 30 donated SNS tokens remain in the treasury and are not deposited.

A deterministic integration test using PocketIC can reproduce this by: (a) initializing an SNS with a known treasury balance, (b) issuing an `icrc1_transfer` to the treasury subaccount from an unprivileged principal, (c) submitting and passing a deposit proposal with an amount between 50% of the original and 50% of the inflated balance, and (d) asserting that the deposited amount exceeds 50% of the pre-inflation treasury balance.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}
