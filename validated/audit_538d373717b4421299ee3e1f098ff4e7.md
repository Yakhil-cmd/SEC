Audit Report

## Title
Missing Caller-Buyer Identity Check Allows Forced SNS Swap Participation Without Consent — (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update endpoint in the SNS Swap canister accepts an arbitrary `buyer` principal in its request without verifying that `caller() == buyer`. Any unprivileged ingress sender who knows a victim's principal and the swap's publicly readable confirmation text can register the victim's ICP deposit as swap participation, bypassing the explicit consent mechanism and locking the victim's funds until the swap finalizes.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 128–143, the `refresh_buyer_tokens` handler resolves the buyer principal from the request field without any caller equality check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no check that caller == p
};
```

The `buyer` field is documented as "If not specified, the caller is used," implying caller-as-default is a convention, not an enforced invariant. When `buyer` is non-empty, the canister proceeds to:

1. Query the ICP balance of `swap_canister[subaccount(buyer)]` on the ICP ledger (`swap.rs` lines 1153–1163).
2. Validate the caller-supplied `confirmation_text` against the SNS-configured text (`swap.rs` lines 1149–1150, 363–384). The confirmation text is stored in the `Init` struct and is publicly readable via the `get_init` query endpoint.
3. Register the balance as the buyer's participation in `self.buyers` (`swap.rs` lines 1285–1288).
4. Delete the buyer's open ticket from stable memory (`swap.rs` lines 1249–1271).

The `validate_confirmation_text` function (`swap.rs` lines 363–384) only checks that the supplied text matches the SNS-configured text — it does not verify that the caller is the buyer. The `error_refund_icp` function (`swap.rs` lines 1932–1935) explicitly rejects refund requests unless the swap lifecycle is ABORTED or COMMITTED, so the victim has no recourse during the OPEN phase.

The integration test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 807–818 calls `refresh_buyer_tokens` with a different principal than the caller and asserts an error, but this error is due to zero ICP balance (pre-transfer state), not a caller check — confirming no such check exists.

## Impact Explanation
An attacker forces a victim's ICP (already deposited to the swap subaccount) into the swap without the victim's consent. The confirmation text — the sole consent mechanism — is bypassed because it is publicly readable from the `Init` state. The victim's funds are locked for the duration of the swap (potentially days or weeks). If the swap commits, the victim receives unwanted SNS tokens instead of their ICP. If the swap aborts, the victim recovers their ICP minus the ICP ledger transfer fee, but only after `sweep_icp` runs post-finalization. The victim's open ticket is also deleted, disrupting the intended payment flow.

This matches: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm**, specifically forced participation bypassing an explicit consent mechanism with temporary fund lockup and potential unwanted asset acquisition.

## Likelihood Explanation
The attack requires no special privileges. The victim's ICP deposit is visible on the public ICP ledger. The confirmation text is readable from the swap canister's public `get_init` query. The attacker needs only the victim's principal and the swap's confirmation text. The attack is repeatable against any victim who has deposited ICP to the swap subaccount but has not yet called `refresh_buyer_tokens`. No economic cost beyond ICP transaction fees is required.

## Recommendation
Enforce that the resolved buyer principal equals the caller. The simplest fix in `rs/sns/swap/canister/canister.rs`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    if !arg.buyer.is_empty() {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, requested, "buyer must equal caller");
    }
    let p = caller;
    // ... rest unchanged
}
```

Alternatively, remove the `buyer` field entirely and always derive the buyer from `caller()`, since the field's only documented use is as a fallback to the caller.

## Proof of Concept
1. Alice sends 10 ICP to `swap_canister[subaccount(alice_principal)]` on the ICP ledger.
2. Alice changes her mind before calling `refresh_buyer_tokens`.
3. Bob reads the swap's confirmation text via `get_init` (a public query call).
4. Bob calls:
   ```
   refresh_buyer_tokens({
     buyer: "alice_principal_text",
     confirmation_text: Some("<text from get_init>")
   })
   ```
5. The swap canister queries Alice's subaccount balance, finds 10 ICP, validates the confirmation text (which matches), and registers Alice as a buyer in `self.buyers`.
6. Alice's open ticket (if any) is deleted from stable memory.
7. Alice's 10 ICP is now locked in the swap. `error_refund_icp` returns a precondition error ("Error refunds can only be performed when the swap is ABORTED or COMMITTED") for the duration of the OPEN phase.
8. If the swap commits, Alice receives unwanted SNS tokens; if it aborts, she recovers her ICP minus the transfer fee — but only after finalization.

A deterministic unit test can reproduce this by calling `swap.refresh_buyer_token_e8s(victim_principal, Some(confirmation_text), ...)` from a different caller context and asserting the victim appears in `swap.buyers`.
