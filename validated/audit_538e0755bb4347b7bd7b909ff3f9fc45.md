Audit Report

## Title
Rate-Limit Bypass via CMC Pre-Funding in `ensure_balance` - (File: `rs/nns/cmc/src/main.rs`)

## Summary
`ensure_balance` charges the `base_limiter` with `cycles_to_mint = cycles - current_balance` (the incremental mint) rather than `cycles` (the full ICP-backed value being converted). Because any canister can deposit cycles into the CMC via the management canister's `deposit_cycles`, an attacker can artificially inflate `current_balance` before a `notify_top_up` / `notify_mint_cycles` / `notify_create_canister` call, shrinking `cycles_to_mint` and causing the limiter to record far less than the true ICP-backed conversion. This allows sustained ICP burning above `base_cycles_limit` per hour.

## Finding Description

`ensure_balance` at `rs/nns/cmc/src/main.rs` lines 2306–2325:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128()); // live snapshot
    let cycles_to_mint = cycles - current_balance;                           // attacker-reducible

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;   // charged wrong amount
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

The `Cycles` type (`rs/types/cycles/src/cycles.rs` lines 133–139) implements `Sub` via `saturating_sub`. This means if `current_balance >= cycles`, `cycles_to_mint` saturates to `Cycles::zero()` — the limiter is charged 0 and 0 cycles are minted, while the assert still passes because the CMC already holds enough balance. The submitted claim's secondary "underflow wraps to `u128::MAX`" hazard is **factually incorrect** due to saturating arithmetic; the `Cycles` type explicitly uses `saturating_sub`, not wrapping subtraction. Similarly, `check_and_add_cycles` uses `saturating_add` for the `count + cycles_to_mint` comparison, so overflow saturates to `u128::MAX` and always fails the limit check — the secondary "limiter check wraps to pass" claim is also incorrect.

The **primary vulnerability is valid and unaffected by these corrections**: an attacker pre-funds the CMC with X cycles before submitting a `notify_top_up` for Z cycles (Z > `base_cycles_limit`). `ensure_balance` computes `cycles_to_mint = Z − X` (saturating, positive), charges the limiter only `Z − X`, mints `Z − X` new cycles, and the CMC deposits the full Z cycles (X pre-funded + Z−X newly minted) to the target. The limiter records only `Z − X` against the hourly budget even though Z ICP-backed value was converted.

This path is reachable from all three public minting endpoints:
- `deposit_cycles` → `process_top_up` → `notify_top_up` (line 2117)
- `do_mint_cycles` → `process_mint_cycles` → `notify_mint_cycles` (line 2151)
- `do_create_canister` → `process_create_canister` → `notify_create_canister` (line 2249)

## Impact Explanation

The `base_cycles_limit` (150 × 10¹⁵ cycles/hour) is the protocol's primary rate control on ICP burning. By pre-funding the CMC with X cycles before a Z-cycle top-up (where Z − X ≤ `base_cycles_limit` < Z), the attacker converts Z ICP-worth of value into cycles while consuming only Z − X of the hourly budget. Repeating this each hour allows sustained above-limit ICP burning, distorting the ICP/cycles exchange rate and undermining the tokenomic controls the limit is designed to enforce. This constitutes a significant NNS/CMC financial-control bypass with concrete protocol harm, matching the **High** impact tier: significant NNS/ledger/infrastructure security impact with concrete user or protocol harm.

## Likelihood Explanation

Any canister on any subnet can call the management canister's `deposit_cycles` targeting the CMC — no privileged role, governance vote, or threshold key is required. The attacker only needs to hold cycles (obtainable by minting at the normal rate over prior hours). The attack is fully deterministic: `ensure_balance` is synchronous, so pre-funding before the `notify_top_up` ingress message is sufficient. No timing precision is required.

## Recommendation

Charge the limiter based on the full `cycles` parameter (the ICP-backed value being converted), not on `cycles_to_mint` (the incremental mint). Add an explicit guard for the already-funded case:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    if current_balance >= cycles {
        return Ok(());
    }
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        // Charge the limiter for the full ICP-backed amount
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

## Proof of Concept

Assume `base_cycles_limit` = 150 × 10¹⁵ cycles/hour, ICP/cycles rate = 1 ICP = 10¹⁴ cycles.

1. **Accumulate cycles**: Over prior hours, attacker mints at the normal rate, accumulating 150 × 10¹⁵ cycles.
2. **Pre-fund CMC**: Attacker calls management canister `deposit_cycles` targeting the CMC, depositing 150 × 10¹⁵ cycles. CMC `current_balance` = 150 × 10¹⁵.
3. **Submit large top-up**: Attacker transfers 3,000 ICP (worth 300 × 10¹⁵ cycles) to the CMC's subaccount and calls `notify_top_up`.
4. **`ensure_balance` executes**: `cycles_to_mint = 300×10¹⁵ − 150×10¹⁵ = 150×10¹⁵` (saturating sub, positive). Limiter check: `0 + 150×10¹⁵ ≤ 150×10¹⁵` → passes.
5. **Result**: CMC mints 150 × 10¹⁵ new cycles, combines with 150 × 10¹⁵ pre-funded cycles, deposits 300 × 10¹⁵ cycles to target. Limiter records only 150 × 10¹⁵ this hour — 2× the intended limit was converted.

A deterministic integration test using PocketIC can reproduce this by: (a) initializing CMC state with a known `base_cycles_limit`, (b) calling `deposit_cycles` on the management canister to pre-fund the CMC, (c) submitting a `notify_top_up` for 2× the limit, and (d) asserting that `base_limiter.get_count()` equals only half the converted amount.
