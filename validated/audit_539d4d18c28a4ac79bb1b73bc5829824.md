### Title
Silo Whitelist Bypassed via `call` Entrypoint — (`engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

In Silo mode, Aurora Engine enforces a whitelist to restrict which NEAR accounts and EVM addresses may execute EVM transactions. This check is applied in `submit` and `submit_with_args` via `assert_access`, but the `call` entrypoint — which also executes arbitrary EVM logic — contains **no whitelist check at all**. Any NEAR account, regardless of whitelist status, can invoke `call` to execute EVM transactions, including transferring tokens, defeating the Silo access-control model.

---

### Finding Description

Aurora Engine's Silo mode maintains four whitelists (`Admin`, `EvmAdmin`, `Account`, `Address`) to gate who may submit EVM transactions or deploy EVM code. The enforcement point is `assert_access`, called inside `engine::submit_with_alt_modexp`: [1](#0-0) 

`assert_access` delegates to `silo::is_allow_submit`, which checks both the NEAR predecessor account and the EVM transaction signer address: [2](#0-1) [3](#0-2) 

Both `submit` and `submit_with_args` route through `engine::submit`, so they receive this check: [4](#0-3) 

However, the `call` entrypoint — which also executes arbitrary EVM logic — does **not** call `assert_access` or any whitelist function: [5](#0-4) 

The `call` entrypoint is exposed with no access control at the contract boundary: [6](#0-5) 

`call_with_args` simply executes the EVM call using the NEAR predecessor's derived EVM address as `origin`, with no whitelist gate: [7](#0-6) 

The same omission exists in `deploy_code`, which also skips `assert_access`: [8](#0-7) 

---

### Impact Explanation

The Silo whitelist is the primary mechanism for a Silo operator to restrict EVM participation to a known set of accounts and addresses. When the whitelist is enabled, the intent is that non-whitelisted addresses cannot move tokens or interact with EVM contracts.

Because `call` bypasses the whitelist entirely, any non-whitelisted NEAR account can:
1. Transfer ETH or ERC-20 tokens from their derived EVM address (`predecessor_address(&predecessor_account_id)`)
2. Call any EVM contract (e.g., invoke `transferFrom` on an ERC-20 to move tokens they have approved)
3. Deploy EVM contracts (via `deploy_code`)

This directly defeats the Silo operator's ability to freeze non-whitelisted addresses' funds. An address that the operator intended to exclude from the Silo can still move its assets by routing through `call` instead of `submit`.

**Impact**: High — Temporary (and effectively permanent, since the bypass is always available) freezing bypass; non-whitelisted users can transfer their own tokens and interact with the EVM at will.

---

### Likelihood Explanation

The `call` entrypoint is a standard, documented Aurora Engine interface. Any NEAR account can invoke it with a simple NEAR function call. No special privileges, leaked keys, or social engineering are required. The attacker only needs to know their derived EVM address holds assets and that `call` exists. Likelihood is **High**.

---

### Recommendation

Add a whitelist check inside the `call` (and `deploy_code`) entrypoint, mirroring the check in `submit`. Specifically, call `silo::is_allow_submit` (for `call`) and `silo::is_allow_deploy` (for `deploy_code`) using `env.predecessor_account_id()` and `predecessor_address(&predecessor_account_id)` before executing EVM logic:

```rust
// In evm_transactions::call, after require_running:
let predecessor_account_id = env.predecessor_account_id();
let predecessor_addr = predecessor_address(&predecessor_account_id);
if !silo::is_allow_submit(&io, &predecessor_account_id, &predecessor_addr) {
    return Err(errors::ERR_NOT_ALLOWED.into());
}
```

Apply the analogous `is_allow_deploy` guard in `deploy_code`.

---

### Proof of Concept

1. Operator enables Silo mode with all whitelists active. Address `A` (derived from NEAR account `alice.near`) is **not** added to the `Address` or `Account` whitelist.
2. `alice.near` attempts `submit` with a signed EVM transaction — this fails with `NotAllowed` because `assert_access` rejects it.
3. `alice.near` instead calls the `call` entrypoint with `CallArgs::V2 { contract: erc20_address, value: 0, input: transfer_calldata }`.
4. `call` executes with no whitelist check. The EVM runs with `origin = predecessor_address("alice.near")` = address `A`.
5. The ERC-20 `transfer` succeeds, moving tokens from address `A` to any recipient — despite `A` being non-whitelisted.

The whitelist restriction is fully bypassed. [5](#0-4) [3](#0-2) [9](#0-8)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
