### Title
Asymmetric Deposit Restriction in ckBTC Minter Allows BTC to Be Sent But Not Minted During `DepositsRestrictedTo` Mode — (File: `rs/bitcoin/ckbtc/minter/src/state.rs`)

---

### Summary

The ckBTC minter's `Mode::DepositsRestrictedTo` variant creates an asymmetric state analogous to the Scroll L2USDCGateway pause issue: the Bitcoin network remains fully operational and accepts BTC sent to the minter's derived addresses, but the minter's `update_balance` endpoint rejects non-whitelisted callers with `TemporarilyUnavailable`. Users who send BTC during this mode have their funds temporarily inaccessible with no on-chain refund path, and the `get_minter_info` query endpoint does not expose the current mode, so users cannot detect the restriction before sending.

---

### Finding Description

The `Mode` enum in `rs/bitcoin/ckbtc/minter/src/state.rs` defines four operational states: [1](#0-0) 

The `is_deposit_available_for` method blocks all non-whitelisted principals when the mode is `DepositsRestrictedTo`: [2](#0-1) 

Critically, `is_withdrawal_available_for` explicitly allows **all** principals when the mode is `DepositsRestrictedTo`: [3](#0-2) 

The `update_balance` function enforces this check on the **caller** (not the owner), meaning a non-whitelisted user calling `update_balance` for their own account is rejected even if they have already sent BTC to their derived minter address: [4](#0-3) 

The `get_minter_info` query endpoint returns a `MinterInfo` record that does **not** include the current `mode` field: [5](#0-4) 

This means users have no standard on-chain query to detect that deposits are restricted before sending BTC.

---

### Impact Explanation

When the minter is placed in `DepositsRestrictedTo` mode via an NNS upgrade:

1. The minter's per-account Bitcoin deposit addresses (derived via threshold ECDSA) remain publicly known and on-chain valid — the Bitcoin network has no knowledge of the minter's operational mode.
2. A non-whitelisted user sends BTC to their derived address.
3. The user calls `update_balance`; the minter returns `UpdateBalanceError::TemporarilyUnavailable("BTC deposits are temporarily restricted")`.
4. The user's BTC UTXO sits at the derived address, unprocessable, with no refund path. Unlike the ckBTC withdrawal flow (which has a `WillReimburse`/`Reimbursed` status), there is no reimbursement mechanism for UTXOs that arrive during a restricted period.
5. If the mode is never restored to `GeneralAvailability` (e.g., during a minter deprecation or prolonged restriction), the BTC becomes permanently inaccessible.

This is a direct analog to the Scroll finding: one side of the bridge (Bitcoin network) remains open while the other side (ckBTC minter) silently rejects finalization, with no refund mechanism for in-flight deposits.

---

### Likelihood Explanation

- `DepositsRestrictedTo` is a documented, production-reachable mode settable via NNS upgrade args (`mode : opt Mode` in `UpgradeArgs`).
- The mode has been used in practice (the ckBTC minter has been upgraded with mode changes on mainnet).
- Users monitoring the Bitcoin network or using wallets that auto-send to derived addresses have no standard way to detect the restriction; `get_minter_info` does not expose `mode`.
- The entry path requires only an unprivileged user sending a standard Bitcoin transaction — no special permissions needed. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **Expose the current mode in `get_minter_info`**: Add a `mode` field to `MinterInfo` so users and wallets can query the restriction state before sending BTC.
2. **Coordinate mode transitions**: When setting `DepositsRestrictedTo`, ensure all in-flight UTXOs (already confirmed on Bitcoin but not yet minted) are processed before the restriction takes effect, or provide a grace-period mechanism.
3. **Add a suspended-UTXO path for restricted deposits**: Similar to the existing `SuspendedUtxo` / `Checked` status for ledger-unavailable UTXOs, introduce a state that queues UTXOs received during a restricted period for automatic processing once the mode is lifted, rather than silently rejecting `update_balance`.
4. **Document the asymmetry**: Clearly document that `DepositsRestrictedTo` does not prevent BTC from being sent to derived addresses, and that users should check the minter mode before depositing.

---

### Proof of Concept

```
1. NNS governance submits an upgrade proposal setting:
   mode = DepositsRestrictedTo([whitelisted_principal])

2. User (non-whitelisted) queries get_btc_address({}) → receives their derived BTC address.
   (get_minter_info does NOT reveal the restricted mode.)

3. User sends 0.01 BTC to the derived address on the Bitcoin network.
   Transaction confirms with >= min_confirmations.

4. User calls update_balance({owner: null, subaccount: null}).

5. Minter executes:
   state::read_state(|s| s.mode.is_deposit_available_for(&caller))
   → Err("BTC deposits are temporarily restricted")
   → returns UpdateBalanceError::TemporarilyUnavailable(...)

6. User's 0.01 BTC UTXO is now at the derived address, unprocessable.
   No reimbursement event is emitted. No WillReimburse status exists.
   The UTXO remains inaccessible until governance restores GeneralAvailability.
``` [8](#0-7) [9](#0-8)

impl Mode {
    /// Returns Ok if the specified principal can convert BTC to ckBTC.
    pub fn is_deposit_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("access to the minter is temporarily restricted".to_string());
                }
                Ok(())
            }
            Self::DepositsRestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC deposits are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }

    /// Returns Ok if the specified principal can convert ckBTC to BTC.
    pub fn is_withdrawal_available_for(&self, p: &Principal) -> Result<(), String> {
        match self {
            Self::GeneralAvailability | Self::DepositsRestrictedTo(_) => Ok(()),
            Self::ReadOnly => Err("the minter is in read-only mode".to_string()),
            Self::RestrictedTo(allow_list) => {
                if !allow_list.contains(p) {
                    return Err("BTC withdrawals are temporarily restricted".to_string());
                }
                Ok(())
            }
        }
    }
}

/// Notifies the ckBTC minter to update the balance of the user subaccount.
pub async fn update_balance<R: CanisterRuntime>(
    args: UpdateBalanceArgs,
    runtime: &R,
) -> Result<Vec<UtxoStatus>, UpdateBalanceError> {
    let caller = runtime.caller();
    if args.owner.unwrap_or(caller) == runtime.id() {
        ic_cdk::trap("cannot update minter's balance");
    }

    // Record start time of method execution for metrics
    let start_time = runtime.time();

    // When the minter is in the mode using a whitelist we only want a certain
    // set of principal to be able to mint. But we also want those principals
    // to mint at any desired address. Therefore, the check below is on "caller".
    state::read_state(|s| s.mode.is_deposit_available_for(&caller))
        .map_err(UpdateBalanceError::TemporarilyUnavailable)?;
