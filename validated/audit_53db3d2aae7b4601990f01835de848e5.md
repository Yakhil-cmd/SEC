Audit Report

## Title
Unverified `buyer` Parameter in `refresh_buyer_tokens` Allows Bypassing Swap Participation Consent Mechanism - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal from the request payload without verifying it against `ic_cdk::caller()`. The `confirmation_text` field — the sole per-participant consent gate — is validated only against the swap's globally readable init text, not against the identity of the actual caller. Any unprivileged ingress sender who knows the public confirmation text can call `refresh_buyer_tokens` with a victim's principal as `buyer`, bypassing the victim's consent and forcibly committing the victim's ICP into the swap.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 128–143, the handler resolves the effective buyer from the request payload with no caller-identity check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // no check that arg.buyer == caller()
};
``` [1](#0-0) 

The resolved principal `p` is passed directly to `refresh_buyer_token_e8s`. The inline doc comment for that function explicitly states: *"the **caller** of this function must accept this confirmation"* — yet the implementation never enforces `buyer == caller`. [2](#0-1) 

`validate_confirmation_text` compares the supplied text against the swap's global `Init.confirmation_text` field, with no reference to who the caller is: [3](#0-2) 

After the text check passes, the function reads the ICP balance of the **victim's** subaccount and inserts or updates the victim's entry in `self.buyers`, locking the ICP into the swap: [4](#0-3) [5](#0-4) 

The ticket guard (lines 1249–1272) provides no protection: it only validates if a ticket already exists for the buyer; if none exists, the comment explicitly states *"the payment flow will simply ignore the ticket"* and proceeds unconditionally. [6](#0-5) 

The `confirmation_text` is part of the public `Init` state, readable by anyone via the `get_state` query. The proto definition confirms the field is a global swap-level string, not a per-user secret: [7](#0-6) 

## Impact Explanation
**High — Unauthorized commitment of a victim's ICP into an SNS swap, bypassing the explicit consent mechanism.** Once `self.buyers` is updated, the victim's ICP is held by the swap canister until finalization. If the swap commits, the victim's ICP is swept to the SNS governance canister and the victim receives SNS tokens they never agreed to purchase. The `confirmation_text` consent mechanism — the only guard that distinguishes a willing participant from an unwilling one — is rendered entirely ineffective. This maps to the allowed High impact: *"Unauthorized access to … ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."*

## Likelihood Explanation
High. The attack requires no privileged access, no key material, and no social engineering. Prerequisites are: (1) the swap has a non-empty `confirmation_text` (common for legally-sensitive SNS launches); (2) a victim has transferred ICP to the swap canister's subaccount but has not yet called `refresh_buyer_tokens` (a normal intermediate state in the participation flow); (3) the attacker reads the public `confirmation_text` from `get_state`. All three conditions are routinely satisfied during active SNS swaps. The call is a standard ingress update message callable by any anonymous or authenticated principal.

## Recommendation
Enforce that when `arg.buyer` is non-empty it must equal `caller()`, unless the caller is an explicitly whitelisted trusted intermediary:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Caller {} is not authorized to refresh tokens on behalf of buyer {}", caller, buyer);
        }
        buyer
    };
    ...
}
```

Alternatively, if third-party notification is intentionally supported, strip the `confirmation_text` requirement when the caller is not the buyer — since consent must come from the buyer, not a third party.

## Proof of Concept
1. Alice transfers 10 ICP to the SNS swap canister's subaccount for her principal (`swap_canister_subaccount(alice_principal)`), intending to review the confirmation text before committing.
2. Attacker Bob queries `get_state` on the swap canister and reads `init.confirmation_text = "I agree to the SNS swap terms"`.
3. Bob sends an ingress update from any identity:
   ```
   refresh_buyer_tokens({
     buyer: alice_principal_text,
     confirmation_text: Some("I agree to the SNS swap terms")
   })
   ```
4. The swap canister resolves `p = alice_principal`, validates the confirmation text against the global init text (passes), reads Alice's subaccount balance (10 ICP), and inserts `alice_principal → BuyerState { amount_icp_e8s: 10_0000_0000 }` into `self.buyers`.
5. Alice's ICP is now locked in the swap. She never provided her consent. When the swap finalizes, her ICP is swept to the SNS governance canister and she receives SNS tokens she never agreed to purchase.

A deterministic integration test can reproduce this using the existing `PaymentProtocolTestSetup` harness in `rs/sns/integration_tests/src/payment_flow.rs`: have user A transfer ICP, then call `refresh_buyer_tokens` as user B with user A's principal and the known confirmation text, and assert that user A's `BuyerState` is populated despite user A never calling `refresh_buyer_tokens` themselves.
