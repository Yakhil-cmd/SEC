### Title
ckETH Burned With Zero Reimbursement When `erc20_tx_fee < CKETH_LEDGER_TRANSACTION_FEE` During Failed ckERC20 Withdrawal — (`rs/ethereum/cketh/minter/src/main.rs`)

### Summary

The `withdraw_erc20` handler silently drops the reimbursement obligation when the estimated ERC-20 transaction fee is smaller than the ckETH ledger's own transfer fee. The user's ckETH is burned, the ckERC20 burn fails, and no `FailedErc20WithdrawalRequest` event is ever emitted, so no refund is scheduled.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/main.rs`, `CKETH_LEDGER_TRANSACTION_FEE` is a compile-time constant: [1](#0-0) 

```
pub const CKETH_LEDGER_TRANSACTION_FEE: Wei = Wei::new(2_000_000_000_000_u128); // 2 000 gwei
```

When the ckERC20 burn fails for a user-attributable reason, the penalty logic is: [2](#0-1) 

```rust
let reimbursed_amount = match &ckerc20_burn_error {
    LedgerBurnError::TemporarilyUnavailable { .. } => erc20_tx_fee,
    LedgerBurnError::InsufficientFunds { .. }
    | LedgerBurnError::AmountTooLow { .. }
    | LedgerBurnError::InsufficientAllowance { .. } => erc20_tx_fee
        .checked_sub(CKETH_LEDGER_TRANSACTION_FEE)
        .unwrap_or(Wei::ZERO),   // ← saturates to ZERO when fee is tiny
};
if reimbursed_amount > Wei::ZERO {          // ← gate skips event entirely
    mutate_state(|s| {
        process_event(s, EventType::FailedErc20WithdrawalRequest(...));
    });
}
```

`estimate_erc20_transaction_fee` computes: [3](#0-2) 

```
gas_fee_estimate.to_price(CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT /* 65 000 */)
                .max_transaction_fee()
```

For `erc20_tx_fee < 2 000 000 000 000 wei` the required condition is:

```
max_fee_per_gas × 65 000 < 2 000 000 000 000
max_fee_per_gas < ~30.8 gwei/gas
```

Since `max_fee_per_gas = 2 × base_fee + max_priority_fee` and `max_priority_fee ≥ 1.5 gwei` (hard floor in `estimate_transaction_fee`), this requires `base_fee_per_gas < ~14.6 gwei`. [4](#0-3) 

Ethereum mainnet base fees have repeatedly fallen below 5 gwei during low-activity periods, making this condition reachable in production.

---

### Impact Explanation

When the condition is met:

1. ckETH is burned from the user's account for `erc20_tx_fee`.
2. The ckERC20 burn fails (e.g., `InsufficientAllowance` — a common user mistake).
3. `reimbursed_amount = Wei::ZERO`.
4. The `if reimbursed_amount > Wei::ZERO` guard is false; `FailedErc20WithdrawalRequest` is never emitted.
5. `record_reimbursement_request` is never called. [5](#0-4) 

The user permanently loses `erc20_tx_fee` ckETH with no on-chain record and no refund, violating the protocol invariant documented in the ckerc20 spec:

> *"If the burn fails … the minter schedules the reimbursement of the burnt ckETH amount … minus some (small) penalty fee."* [6](#0-5) 

**Financial cap per incident:** `erc20_tx_fee < 2 000 gwei ≈ 0.000002 ETH ≈ $0.006` at current prices. The per-user loss is tiny, but the invariant break is real and the event log is permanently inconsistent (burned ckETH with no matching reimbursement or withdrawal record).

---

### Likelihood Explanation

- Ethereum base fees below 14.6 gwei occur regularly during off-peak hours.
- Any user who calls `withdraw_erc20` with an insufficient ckERC20 allowance (a common mistake) during such a period hits this path automatically — no adversarial intent required.
- The attacker surface is the public `withdraw_erc20` update call, reachable by any non-anonymous principal.

---

### Recommendation

Replace the silent `unwrap_or(Wei::ZERO)` with an explicit branch that always emits the event (even with `reimbursed_amount = 0`) so the audit log remains consistent, and consider clamping the penalty to `min(erc20_tx_fee, CKETH_LEDGER_TRANSACTION_FEE)` so the user is never left with a zero refund when the fee is already below the penalty threshold:

```rust
let penalty = erc20_tx_fee.min(CKETH_LEDGER_TRANSACTION_FEE);
let reimbursed_amount = erc20_tx_fee.checked_sub(penalty).unwrap_or(Wei::ZERO);
// Always emit the event so the burn is traceable
mutate_state(|s| {
    process_event(s, EventType::FailedErc20WithdrawalRequest(...));
});
```
