Audit Report

## Title
Unchecked Index Panic in `construction_hash` on Empty `EnvelopePair` Vec — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

## Summary
The `construction_hash` handler directly indexes `envelope_pairs[0]` after finding a `RequestType::Send` entry via `.find(|(rt, _)| rt.is_transfer())`, with no guard against an empty `Vec<EnvelopePair>`. Because `SignedTransaction::from_str` is a pure CBOR deserialization with no structural invariant enforcing a non-empty envelope pair vector, an unauthenticated attacker can POST a crafted payload containing `(RequestType::Send, vec![])` to the `/construction/hash` endpoint and trigger an index-out-of-bounds panic, causing a DoS on the Rosetta service.

## Finding Description
The vulnerable indexing occurs at: [1](#0-0) 

`Request` is defined as a tuple type where the second element is an unconstrained `Vec<EnvelopePair>`: [2](#0-1) 

`SignedTransaction::from_str` performs only hex-decode and CBOR deserialization — no invariant enforces that the `Vec<EnvelopePair>` is non-empty: [3](#0-2) 

`is_transfer()` returns `true` for `RequestType::Send`: [4](#0-3) 

The exploit path is: craft `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` → CBOR-encode → hex-encode → POST to `/construction/hash`. The `verify_network_id` check uses the public mainnet ledger canister ID and is trivially satisfied. The `.find(|(rt, _)| rt.is_transfer())` call matches the crafted tuple, then `envelope_pairs[0]` panics on the empty vec.

A correct emptiness guard already exists in the `TryFrom<&models::Request> for Request` conversion path: [5](#0-4) 

But `construction_hash` never goes through that conversion — it directly indexes the raw deserialized `Vec<EnvelopePair>`.

The endpoint is registered with no authentication: [6](#0-5) 

## Impact Explanation
In actix-web, a panic in a handler causes the worker thread to abort. With `panic=unwind` (default), the process survives but the in-flight request is dropped with a 500 and the attacker can loop this indefinitely, causing continuous worker restarts and degraded availability. With `panic=abort` (used in production IC builds for determinism), the entire Rosetta process terminates. In either case, a single unauthenticated HTTP POST causes a persistent crash loop, denying service to all clients depending on this Rosetta instance for ICP transaction hashing and submission. This matches the **High** impact category: "Significant Rosetta, ledger, or infrastructure security impact with concrete user or protocol harm" and "Application/platform-level DoS, crash... not based on raw volumetric DDoS."

## Likelihood Explanation
The `/construction/hash` endpoint requires no credentials. The CBOR payload is trivially constructable: serialize `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` to CBOR, hex-encode it, and POST with a valid `network_identifier` (the ICP mainnet ledger canister ID is public). No prior knowledge of system state, no ledger participation, and no special privileges are required. The attack is repeatable with a single HTTP request.

## Recommendation
Add an emptiness check before indexing, mirroring the guard already present in `TryFrom<&models::Request> for Request`:

```rust
if envelope_pairs.is_empty() {
    return Err(ApiError::invalid_request(
        "No envelope pairs provided for Send request.",
    ));
}
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pairs[0].update,
)
```

Alternatively, replace `&envelope_pairs[0]` with `.first().ok_or_else(|| ApiError::invalid_request(...))` to be consistent with the existing guard pattern.

## Proof of Concept
```rust
#[test]
fn test_construction_hash_empty_envelope_pairs_panics() {
    use std::str::FromStr;
    use crate::models::SignedTransaction;
    use crate::request_types::RequestType;

    // Craft a SignedTransaction with a Send request but empty envelope pairs
    let crafted = SignedTransaction {
        requests: vec![(RequestType::Send, vec![])],
    };
    let hex_cbor = crafted.to_string(); // hex(cbor(crafted))

    // Simulate what construction_hash does after from_str succeeds
    let parsed = SignedTransaction::from_str(&hex_cbor).unwrap();
    let result = parsed
        .requests
        .iter()
        .rev()
        .find(|(rt, _)| rt.is_transfer());

    // This panics at envelope_pairs[0] in the current code
    if let Some((_rt, envelope_pairs)) = result {
        let _ = &envelope_pairs[0]; // index out of bounds: the len is 0 but the index is 0
    }
}
```

Running `cargo test` in `rs/rosetta-api/icp` will reproduce the panic. Running with `RUSTFLAGS="-C panic=abort"` will crash the process entirely, simulating the production impact.
