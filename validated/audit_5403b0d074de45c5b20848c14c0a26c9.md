Audit Report

## Title
Registry Canister DoS via Oversized `node_signing_pk` Triggering Unconditional Panic in `extract_chip_id_from_payload` — (`rs/registry/canister/src/mutations/node_management/do_add_node.rs`)

## Summary

A registered node operator can submit an `AddNodePayload` with a `node_signing_pk` whose raw protobuf encoding exceeds 65535 bytes (by padding the `proof_data` field) and any non-`None` `node_registration_attestation`. The `valid_keys_from_payload` check passes because it only validates the decoded `key_value` field. Subsequently, `extract_chip_id_from_payload` calls `.expect()` on `OctetStringRef::new(&payload.node_signing_pk)`, which returns `Err` for slices exceeding `u16::MAX`, causing an unconditional panic that traps the registry canister. Because the trap rolls back all state including the rate-limit reservation, the attack is repeatable without cost.

## Finding Description

**Root cause:** In `extract_chip_id_from_payload`, when `node_registration_attestation` is `Some(...)`, the code at line 256–257 calls:

```rust
node_signing_pk: OctetStringRef::new(&payload.node_signing_pk)
    .expect("node_signing_pk must be valid"),
```

`OctetStringRef::new` (from the `der` crate) returns `Err` for any slice whose length exceeds `u16::MAX` (65535). The `.expect()` converts this into an unconditional panic, trapping the canister. [1](#0-0) 

**Why `valid_keys_from_payload` does not block this:** At line 359–360, the raw bytes are decoded into a protobuf `PublicKey` struct. The subsequent `ValidNodeSigningPublicKey::try_from` only validates:
1. Algorithm is Ed25519
2. `key_value` length is exactly 32 bytes
3. `key_value` is a torsion-free Ed25519 point

It does not check the `proof_data` field or the total size of the raw protobuf encoding. [2](#0-1) [3](#0-2) 

**Exploit flow:**
1. Attacker is a registered node operator.
2. Constructs a `PublicKey` protobuf with a valid 32-byte Ed25519 `key_value` and a `proof_data` field padded to ≥65536 bytes. The raw `encode_to_vec()` output exceeds 65535 bytes.
3. Sets `node_registration_attestation` to any non-`None` value (even structurally invalid, since the panic occurs before attestation verification at line 265–271).
4. Submits `AddNodePayload`. `valid_keys_from_payload` succeeds (line 64–65). `extract_chip_id_from_payload` is called at line 195 and panics unconditionally.
5. The canister trap rolls back all state including the rate-limit reservation made at line 60–61, so the attacker can resubmit indefinitely at no cost. [4](#0-3) [5](#0-4) 

## Impact Explanation

The registry canister traps on every replica processing the block containing the malicious message. Repeated submissions cause repeated traps, denying service to the registry canister for all legitimate operations (node additions, subnet changes, key updates, etc.). This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** The registry is a core NNS canister; its unavailability directly disrupts subnet management and governance operations across the Internet Computer.

## Likelihood Explanation

The attacker only needs to be a registered node operator — a role obtainable through governance without admin privileges. The crafted payload is trivial to construct: take any valid node registration payload, append a large `proof_data` blob to the `PublicKey` protobuf for `node_signing_pk`, and set `node_registration_attestation` to any non-`None` value. The 65536-byte `node_signing_pk` is well within the IC's 2 MB ingress message size limit. The attack is deterministic, locally testable, and repeatable without consuming rate-limit capacity due to state rollback on trap.

## Recommendation

Replace the `.expect()` with a proper error return in `extract_chip_id_from_payload`:

```rust
let node_signing_pk_ref = OctetStringRef::new(&payload.node_signing_pk)
    .map_err(|e| format!("{LOG_PREFIX}do_add_node: node_signing_pk too large for DER encoding: {e}"))?;
let expected_custom_data = NodeRegistrationAttestationCustomData {
    node_signing_pk: node_signing_pk_ref,
};
```

Additionally, add an explicit size check in `valid_keys_from_payload` to reject `node_signing_pk` raw bytes exceeding a reasonable bound (e.g., 512 bytes) before protobuf decoding, preventing oversized inputs from reaching later processing stages. [1](#0-0) 

## Proof of Concept

```rust
// Craft a PublicKey protobuf with valid Ed25519 key_value but large proof_data
let mut pk = valid_node_signing_public_key(); // 32-byte key_value, passes ValidNodeSigningPublicKey
pk.proof_data = Some(vec![0u8; 65536]); // makes raw encoding > 65535 bytes
let node_signing_pk = pk.encode_to_vec();
assert!(node_signing_pk.len() > 65535);

// valid_keys_from_payload succeeds because it only validates the decoded key_value
// extract_chip_id_from_payload panics because OctetStringRef::new(&node_signing_pk) returns Err

let payload = AddNodePayload {
    node_signing_pk,
    node_registration_attestation: Some(SevAttestationPackage { /* any value */ }),
    // ... other valid fields
};
// registry.do_add_node_(payload, node_operator_id, now) → PANIC (canister trap)
```

A unit test can be added to `do_add_node.rs` using the existing `#[cfg(test)]` infrastructure (which already sets `SevRootCertificateVerification::TestOnlySkipVerification`) to confirm the panic occurs before attestation verification, and that replacing `.expect()` with `map_err(...)?` causes the function to return `Err` instead of panicking.
