### Title
ERC-20 and ETH Tokens Permanently Lost When `exit_to_near` Promise Fails Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary

When the `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) and the subsequent cross-chain `ft_transfer` / `ft_transfer_call` NEAR promise fails, the burned tokens are **permanently lost** unless the `error_refund` Cargo feature is compiled in. When `error_refund` is disabled (the default for many builds), the `refund` field in `ExitToNearPrecompileCallbackArgs` is hardcoded to `None`, so the callback never re-mints the tokens, and the user's funds are irrecoverably destroyed.

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) implements the bridge-exit flow:

1. The ERC-20 contract calls the precompile, which **burns** the user's ERC-20 tokens (or deducts ETH from the EVM balance) atomically within the EVM execution.
2. A NEAR promise is scheduled to call `ft_transfer` (or `ft_transfer_call`) on the NEP-141 contract to deliver the tokens on the NEAR side.
3. If a callback is needed (for wNEAR unwrap or for error refund), `exit_to_near_precompile_callback` is attached.

The critical conditional compilation block is:

```rust
// engine-precompiles/src/native.rs lines 449-454
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // <-- refund is always None without the feature
    transfer_near: transfer_near_args,
};
```

When `error_refund` is **not** enabled, `refund` is always `None`. The callback logic in `exit_to_near_precompile_callback` then reaches:

```rust
// engine/src/contract_methods/connector.rs lines 231-241
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // <-- no refund happens; tokens are gone
};
```

Because `args.refund` is `None`, the `else` branch executes and no re-mint occurs. The ERC-20 tokens that were burned (or ETH that was deducted) are permanently lost.

Furthermore, when `callback_args` equals the default (both fields `None`), **no callback is attached at all**:

```rust
// engine-precompiles/src/native.rs lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback attached
} else {
    PromiseArgs::Callback(...)
};
```

This means for the plain ERC-20 `ft_transfer` path without `error_refund`, there is no callback whatsoever â€” if the `ft_transfer` promise fails, there is zero recovery mechanism.

The test suite explicitly acknowledges this behavior:

```rust
// engine-tests/src/tests/erc20_connector.rs lines 656-660
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

The same pattern applies to ETH exits:

```rust
// engine-tests/src/tests/erc20_connector.rs lines 771-775
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

### Impact Explanation

**Critical â€” Permanent freezing/destruction of user funds.**

Any user who calls `withdrawToNear` on a bridged ERC-20 token (or `withdrawEthToNear` for ETH) when the `error_refund` feature is not compiled in, and whose `ft_transfer` NEAR promise subsequently fails (e.g., because the recipient NEAR account is not registered with the NEP-141 contract, or the NEP-141 contract is paused, or any other NEAR-side failure), will have their tokens permanently burned with no recovery. The EVM state is already committed (tokens burned), and no re-mint callback is scheduled.

### Likelihood Explanation

**High.** The `ft_transfer` promise can fail for multiple realistic reasons reachable by any unprivileged user:
- Sending to a NEAR account not registered with the NEP-141 token (storage deposit not paid).
- Sending to a non-existent NEAR account.
- NEP-141 contract being paused.
- Insufficient attached gas causing the promise to fail.

The `error_refund` feature is a compile-time opt-in, meaning any deployment that does not explicitly enable it is permanently vulnerable. The test suite confirms the loss is real and expected in that configuration.

### Recommendation

1. Enable the `error_refund` feature by default in production builds, or make it unconditional (remove the feature flag and always populate `refund`).
2. If the feature flag must remain, document clearly that deployments without `error_refund` will permanently lose user funds on failed exits.
3. Consider adding a runtime check that refuses to process exits if the refund mechanism is not available.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature flag.
2. Bridge a NEP-141 token to an ERC-20 on Aurora.
3. Call `withdrawToNear("unregistered.near", amount)` on the ERC-20 contract (the recipient is not registered with the NEP-141).
4. The ERC-20 tokens are burned atomically in the EVM.
5. The `ft_transfer` NEAR promise fails because `unregistered.near` has no storage deposit.
6. No callback is attached (or the callback's `refund` field is `None`), so `refund_on_error` is never called.
7. The user's ERC-20 tokens are permanently destroyed; their NEP-141 balance on NEAR is unchanged; the Aurora NEP-141 balance decreases by the exit amount with no corresponding credit anywhere.

Confirmed by the test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` lines 623â€“665, which explicitly shows the ERC-20 balance is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (i.e., tokens are lost) when `error_refund` is not enabled.
