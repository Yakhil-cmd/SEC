### Title
Unsolicited Neuron Age Dilution via Unauthenticated `ClaimOrRefresh` (`By::NeuronIdOrSubaccount`) - (File: `rs/nns/governance/src/governance.rs`)

---

### Summary

The NNS Governance canister's `manage_neuron` endpoint accepts a `ClaimOrRefresh { by: NeuronIdOrSubaccount }` command that allows **any unprivileged ingress caller** to trigger `refresh_neuron` on any existing neuron identified by its public neuron ID or subaccount. When the neuron's ledger balance exceeds its cached stake (e.g., because the owner just topped up the staking subaccount in preparation for a refresh), the call invokes `update_stake_adjust_age`, which **dilutes the neuron's `aging_since_timestamp_seconds`** — reducing the neuron's age bonus and therefore its voting power and staking rewards — without the neuron owner's consent.

---

### Finding Description

The `manage_neuron_internal` function in `rs/nns/governance/src/governance.rs` handles `Command::ClaimOrRefresh` before any neuron-ownership check:

```rust
if let Some(Command::ClaimOrRefresh(claim_or_refresh)) = &mgmt.command {
    return match &claim_or_refresh.by {
        ...
        Some(By::NeuronIdOrSubaccount(_)) => {
            let id = mgmt.get_neuron_id_or_subaccount()?.ok_or_else(...)?;
            self.refresh_neuron_by_id_or_subaccount(id, claim_or_refresh)
                .await
                .map(ManageNeuronResponse::claim_or_refresh_neuron_response)
        }
        ...
    };
}

`update_stake_adjust_age` computes a new weighted-average age and **overwrites `aging_since_timestamp_seconds`**:

The age bonus directly affects voting power and staking rewards. The tests explicitly confirm that **any caller** (including a different principal) can successfully call `ClaimOrRefresh { by: NeuronIdOrSubaccount }` on another user's neuron:

The comment at line 5014 explicitly states "anyone can do it," confirming the design intent — but the security consequence of this when a neuron's balance has been topped up is that an adversary can front-run the owner's own refresh call and dilute the age bonus.

---

### Impact Explanation

**Governance authorization bug / unsolicited account data modification.**

- The neuron's `aging_since_timestamp_seconds` is a time-weighted field that determines the **age bonus** (up to 25% extra voting power after 4 years).
- An attacker who observes (via the public ledger) that a neuron owner has transferred ICP to their staking subaccount — in preparation for calling `ClaimOrRefresh` themselves — can **front-run** that transaction by calling `manage_neuron` with `ClaimOrRefresh { by: NeuronIdOrSubaccount }` targeting the victim's neuron ID.
- This causes `update_stake_adjust_age` to execute with the new higher balance, diluting the neuron's age. For example, a neuron with 10 ICP aged 4 years that receives a 10 ICP top-up will have its age halved to 2 years after the refresh — losing years of accumulated age bonus.
- The victim's voting power and future staking rewards are permanently reduced (the age cannot be recovered without waiting again).
- This is directly analogous to the esFLUO vulnerability: an unprivileged caller resets time-based state for another user's account.

---

### Likelihood Explanation

- The NNS ledger is fully public; any observer can watch for ICP transfers to governance staking subaccounts.
- The `manage_neuron` endpoint is open to any non-anonymous ingress caller.
- The neuron ID or subaccount is derivable from public information (controller principal + memo).
- Front-running on the IC is feasible within the same round or across consecutive rounds, since the attacker only needs to submit their ingress message before the victim's `ClaimOrRefresh` is executed.
- The attack requires no funds, no privileged access, and no special knowledge beyond the victim's neuron ID (which is public via `list_neurons`).

---

### Recommendation

Add a caller authorization check inside `refresh_neuron_by_id_or_subaccount` (or at the `By::NeuronIdOrSubaccount` dispatch site in `manage_neuron_internal`) that verifies the caller is the neuron's controller or a registered hotkey before proceeding with `refresh_neuron`. This mirrors the authorization pattern already used by `refresh_voting_power`:

```rust
fn refresh_voting_power(...) -> Result<(), GovernanceError> {
    let is_authorized =
        self.with_neuron(neuron_id, |neuron| neuron.is_authorized_to_vote(caller))?;
    if !is_authorized {
        return Err(GovernanceError::new_with_message(
            ErrorType::NotAuthorized, ...));
    }
    ...
}
``` [6](#0-5) 

The same `is_authorized_to_vote` (controller or hotkey) guard should be applied to `ClaimOrRefresh { by: NeuronIdOrSubaccount }` before calling `refresh_neuron`.

---

### Proof of Concept

1. **Victim** (owner of neuron `N` with 10 ICP staked, aged 4 years) transfers 10 ICP to their governance staking subaccount, intending to call `ClaimOrRefresh` to increase their stake.
2. **Attacker** observes the ICP transfer on the public ledger (or monitors the mempool/ingress queue).
3. **Attacker** submits an ingress message to the NNS Governance canister:
   ```
   manage_neuron({
     neuron_id_or_subaccount: NeuronId(N),
     command: ClaimOrRefresh { by: NeuronIdOrSubaccount({}) }
   })
   ```
   with any non-anonymous principal as the caller.
4. Governance calls `refresh_neuron_by_id_or_subaccount` → `refresh_neuron` → reads ledger balance (20 ICP) → calls `update_stake_adjust_age(20 ICP, now)`.
5. The neuron's age is diluted: `new_age = (10 ICP × 4 years) / 20 ICP = 2 years`. The `aging_since_timestamp_seconds` is updated to `now - 2_years`.
6. **Victim** calls `ClaimOrRefresh` — succeeds, but the balance now equals the cached stake (both 20 ICP), so `Ordering::Equal` is hit and no further age adjustment occurs. The victim's neuron now has only 2 years of age instead of 4, permanently losing the age bonus accumulated over 2 years.

    async fn refresh_neuron_by_id_or_subaccount(
        &mut self,
        id: NeuronIdOrSubaccount,
        claim_or_refresh: &ClaimOrRefresh,
    ) -> Result<NeuronId, GovernanceError> {
        let (nid, subaccount) = match id {
            NeuronIdOrSubaccount::NeuronId(neuron_id) => {
                let neuron_subaccount =
                    self.with_neuron(&neuron_id, |neuron| neuron.subaccount())?;
                (neuron_id, neuron_subaccount)
            }
            NeuronIdOrSubaccount::Subaccount(subaccount_bytes) => {
                let subaccount = Self::bytes_to_subaccount(&subaccount_bytes)?;

                    .ok_or_else(|| Self::no_neuron_for_subaccount_error(&subaccount.0))?;
                (neuron_id, subaccount)
            }
        };
        self.refresh_neuron(nid, subaccount, claim_or_refresh).await
    }
