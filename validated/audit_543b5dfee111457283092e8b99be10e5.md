### Title
Inconsistent `MAX_BLOBS_PER_BLOCK` Values Cause EIP-4844 Transactions with 7–9 Blobs to Always Fail in ZK Validation Flow — (`zk_ee/src/system/constants.rs`, `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs`)

---

### Summary

The codebase defines `MAX_BLOBS_PER_BLOCK` with two conflicting values: `6` in `zk_ee/src/system/constants.rs` and `9` in `basic_bootloader/src/bootloader/block_flow/ethereum/mod.rs` and `block_header.rs`. The ZK transaction validation path and the ZK metadata structures use the value `6`, while the Ethereum block header reports `max_blobs() = 9` and computes blob gas limits based on `9`. This mismatch means any EIP-4844 transaction carrying 7–9 blobs is unconditionally rejected with `BlobListTooLong` in the ZK flow, even though the block-level accounting permits it.

---

### Finding Description

Three separate definitions of `MAX_BLOBS_PER_BLOCK` coexist in the codebase with different values:

**Value = 6** (Cancun-era): [1](#0-0) 

**Value = 9** (Pectra-era): [2](#0-1) [3](#0-2) 

The ZK validation path imports `MAX_BLOBS_PER_BLOCK` from `zk_ee::system` (value = 6) and passes it as the const generic to `parse_blobs_list`: [4](#0-3) 

Inside `parse_blobs_list`, any blob count exceeding the const generic immediately returns a validation error: [5](#0-4) 

Meanwhile, the Ethereum block header's `max_blobs()` and `blobs_gas_limit()` use the local constant `9`: [6](#0-5) 

The ZK metadata structure `TxLevelMetadata` also uses the `zk_ee` constant (6) for its `ArrayVec` capacity: [7](#0-6) 

The `EthereumBlockMetadata` type is parameterized with `zk_ee::system::MAX_BLOBS_PER_BLOCK = 6`: [8](#0-7) 

The result is a structural contradiction: the block header advertises a blob gas limit for 9 blobs, but the ZK validation layer enforces a hard cap of 6 blobs, and the metadata array cannot hold more than 6.

---

### Impact Explanation

Any EIP-4844 transaction submitted with 7, 8, or 9 blobs through the ZK transaction flow is unconditionally rejected with `InvalidTransaction::BlobListTooLong` during validation. The transaction is treated as a validation failure and skipped. The sender cannot make such a transaction succeed regardless of gas price or gas limit, because the rejection occurs before execution. This constitutes a permanent DoS on a class of transactions that the Pectra-era block header explicitly permits. The block-level blob gas accounting (based on 9) and the per-transaction enforcement (based on 6) are irreconcilably inconsistent.

---

### Likelihood Explanation

The trigger is direct and requires no privilege: any unprivileged user who submits an EIP-4844 transaction with 7–9 blobs in ZK mode will have it rejected every time. The Pectra fork (which raised the blob limit from 6 to 9) is already reflected in `block_header.rs` and `block_flow/ethereum/mod.rs`, but the `zk_ee` constant was not updated. This is a latent bug that activates as soon as the network operates under Pectra rules with the ZK flow enabled.

---

### Recommendation

Unify `MAX_BLOBS_PER_BLOCK` to a single authoritative constant. The Pectra-correct value is `9`. The constant in `zk_ee/src/system/constants.rs` should be updated to `9` to match `block_flow/ethereum/mod.rs` and `block_header.rs`. All downstream uses — `TxLevelMetadata`, `EthereumTransactionMetadata`, and `parse_blobs_list` call sites — will then consistently enforce the correct limit. Alternatively, if the ZK flow intentionally restricts blobs to 6 for proving capacity reasons, the block header's `max_blobs()` must be overridden to return `6` in that context, and the blob gas limit must be computed accordingly, so that the block-level check and the per-transaction check agree.

---

### Proof of Concept

1. Construct an EIP-4844 transaction with 8 blob versioned hashes (all with valid `0x01` version byte).
2. Submit it through the ZK transaction flow (`validate_and_compute_fee_for_transaction` in `zk/validation_impl.rs`).
3. The call reaches `parse_blobs_list::<6>(blobs_list)` at line 411.
4. `blobs_list.count = 8 > 6 = MAX_BLOBS_IN_TX`, so the function immediately returns `Err(TxError::Validation(InvalidTransaction::BlobListTooLong))`.
5. The transaction is rejected unconditionally, despite the block header reporting `blobs_gas_limit() = 9 * GAS_PER_BLOB` and the block-level blob gas check passing.
