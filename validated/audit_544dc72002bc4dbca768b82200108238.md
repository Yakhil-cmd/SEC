### Title
Asymmetric Silo Whitelist Enforcement Allows ERC-20 Token Receipt Without Submit Permission, Causing Fund Freeze — (`engine/src/contract_methods/silo/mod.rs`)

---

### Summary

In Silo mode, the Aurora Engine enforces two independent whitelists — `Account` and `Address` — for transaction submission, but only the `Address` whitelist for ERC-20 token receipt. Because these whitelists are managed independently with no enforced consistency invariant, a user's address can be whitelisted to receive ERC-20 tokens while their NEAR account is simultaneously blocked from submitting transactions. Tokens deposited into such an address become permanently inaccessible to the holder.

---

### Finding Description

The Silo module exposes two distinct access-check functions with asymmetric whitelist requirements:

`is_allow_submit` requires **both** the `Account` whitelist and the `Address` whitelist to pass: [1](#0-0) 

`is_allow_receive_erc20_tokens` checks **only** the `Address` whitelist: [2](#0-1) 

The underlying helper functions confirm the split: [3](#0-2) 

The four whitelists (`Admin`, `EvmAdmin`, `Account`, `Address`) are each independently enabled/disabled and populated via `add_entry_to_whitelist`, `remove_entry_from_whitelist`, and `set_whitelist_status`: [4](#0-3) 

Each whitelist's enabled flag and member set are stored as separate storage keys with no cross-list consistency enforcement: [5](#0-4) 

There is no atomic operation or invariant check that ensures: *"every address in the `Address` whitelist has a corresponding NEAR account in the `Account` whitelist."* The two lists are managed entirely independently.

---

### Impact Explanation

When the `Account` whitelist is enabled and an EVM address is present in the `Address` whitelist but its corresponding NEAR account is absent from the `Account` whitelist:

1. `is_allow_receive_erc20_tokens` returns `true` → ERC-20 tokens are successfully deposited to the address via `ft_on_transfer` → `engine.receive_erc20_tokens`.
2. `is_allow_submit` returns `false` → any subsequent EVM transaction from that address (transfer, approval, swap) is rejected.

The deposited ERC-20 tokens are trapped: the holder can receive but never move them. This constitutes a **High — Temporary (potentially permanent) freezing of funds**.

---

### Likelihood Explanation

This is a realistic operational scenario in any Silo deployment:

- An operator enables the `Account` whitelist and populates it with a subset of NEAR accounts.
- The `Address` whitelist is separately managed and may contain EVM addresses whose backing NEAR accounts were never added to the `Account` whitelist, or were later removed from it.
- No tooling or on-chain guard prevents or warns about this inconsistency.
- The `set_whitelist_status` and `add/remove_entry_to_whitelist` functions operate on a single list at a time with no cross-list validation: [6](#0-5) 

Likelihood: **Medium** — requires no adversarial action; a routine administrative update to one whitelist is sufficient to create the inconsistent state.

---

### Recommendation

1. **Enforce a consistency invariant atomically**: When adding an entry to the `Address` whitelist, require (or automatically add) the corresponding NEAR account to the `Account` whitelist, and vice versa.
2. **Unify the receive and submit checks**: `is_allow_receive_erc20_tokens` should apply the same compound check as `is_allow_submit` so that tokens can only be received by addresses whose NEAR accounts are also permitted to submit.
3. **Provide a batch-update primitive** that atomically updates both the `Account` and `Address` whitelists together, preventing partial updates.

---

### Proof of Concept

**Setup (Silo mode, both whitelists enabled):**

```
Account whitelist: enabled, contains { "alice.near" }
Address whitelist: enabled, contains { 0xAlice }
```

**Step 1 — Operator removes `alice.near` from Account whitelist** (routine key rotation or access revocation):
```
remove_entry_from_whitelist({ WhitelistAccountArgs { kind: Account, account_id: "alice.near" } })
```
The `Address` whitelist is **not** updated — `0xAlice` remains.

**Step 2 — Bob sends ERC-20 tokens to `0xAlice`** via `ft_transfer_call` → `ft_on_transfer`:
- `is_allow_receive_erc20_tokens(io, &0xAlice)` → checks only `Address` whitelist → `true` ✓
- Tokens are credited to `0xAlice`'s EVM balance.

**Step 3 — Alice attempts to submit any EVM transaction** (e.g., `transfer` the ERC-20):
- `is_allow_submit(io, &"alice.near", &0xAlice)` → checks `Account` whitelist → `false` ✗
- Transaction rejected with access denied.

Alice's ERC-20 tokens are now frozen. The inconsistency between the two independently managed whitelists — `Account` and `Address` — is the direct root cause, matching the "too flexible configuration" vulnerability class exactly. [7](#0-6) [8](#0-7)

    pub fn enable(&mut self) {
        let key = self.key(STATUS);
        self.io.write_storage(&key, &[1]);
    }

    /// Disable a whitelist.
    pub fn disable(&mut self) {
        let key = self.key(STATUS);
        self.io.write_storage(&key, &[0]);
    }

    /// Check if the whitelist is enabled.
    pub fn is_enabled(&self) -> bool {
        // White list is disabled by default. So return `false` if the key doesn't exist.
        let key = self.key(STATUS);
        self.io
            .read_storage(&key)
            .is_some_and(|value| value.to_vec() == [1])
    }

    fn key(&self, value: &[u8]) -> Vec<u8> {
        let mut bytes = Vec::with_capacity(1 + value.len());

        bytes.push(u8::from(self.kind));
        bytes.extend_from_slice(value);
        bytes_to_key(KeyPrefix::Whitelist, &bytes)
    }

    /// Add a new element to the whitelist.
    pub fn add<A: AsBytes + ?Sized>(&mut self, element: &A) {
        let key = self.key(element.as_bytes());
        self.io.write_storage(&key, &[]);
    }

    /// Remove a new element from the whitelist.
    pub fn remove<A: AsBytes + ?Sized>(&mut self, element: &A) {
        let key = self.key(element.as_bytes());
        self.io.remove_storage(&key);
    }

    /// Check if the element is present in the whitelist.
    pub fn is_exist<A: AsBytes + ?Sized>(&self, element: &A) -> bool {
        let key = self.key(element.as_bytes());
        self.io.storage_has_key(&key)
    }
