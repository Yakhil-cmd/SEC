### Title
Excess ICP Permanently Locked in SNS Swap Canister When Participation Exceeds Per-Participant or Total Cap - (`rs/sns/swap/src/swap.rs`)

### Summary

The SNS Swap canister's `refresh_buyer_token_e8s` function accepts a buyer's full ICP subaccount balance into its internal accounting but silently caps the *accepted* amount at `max_participant_icp_e8s` or the remaining `available_direct_participation_e8s`. The excess ICP that was transferred to the swap subaccount but not accepted is never automatically refunded during the swap's Open phase. It remains locked in the buyer's subaccount on the ICP ledger until the swap closes (Committed or Aborted), at which point the user must manually call `error_refund_icp` to recover it. If the swap commits successfully, the excess ICP is swept to SNS governance along with the accepted portion, permanently losing the user's funds.

### Finding Description

In `rs/sns/swap/src/swap.rs`, `refresh_buyer_token_e8s` reads the full ICP balance from the buyer's subaccount on the ledger, then caps the accepted amount:

```rust
let requested_increment_e8s = e8s - old_amount_icp_e8s;
let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);
let new_balance_e8s = old_amount_icp_e8s.saturating_add(actual_increment_e8s);
// ...
let new_balance_e8s = std::cmp::min(new_balance_e8s, max_participant_icp_e8s);
```

The function records only `new_balance_e8s` (the capped amount) in `self.buyers`, but the full `e8s` (the actual ledger balance) remains in the buyer's subaccount. The excess `e8s - new_balance_e8s` is never returned to the user during the Open phase. The code even acknowledges this with a `TODO`:

```
/// TODO(NNS1-1682): attempt to refund ICP that cannot be accepted.
```

When the swap commits and `finalize_swap` is called, `sweep_icp` transfers **all** ICP from each buyer's subaccount to SNS governance — not just the accepted `new_balance_e8s`. This means the excess ICP is swept away permanently.

The analog to the external report is direct: just as `MoneyMarketHook._handleRepay` transfers tokens based on a user-provided amount but the actual repay operation uses a capped (lesser) amount, `refresh_buyer_token_e8s` accepts a full ledger balance but only records a capped portion, leaving the excess locked with no automatic recovery path during the critical Open→Committed transition.

### Impact Explanation

A user who transfers more ICP than `max_participant_icp_e8s` (or more than the remaining `available_direct_participation_e8s`) to their swap subaccount and calls `refresh_buyer_tokens` will have:

1. Their full balance sitting in the swap subaccount.
2. Only the capped amount recorded in `self.buyers`.
3. When the swap commits, `sweep_icp` transfers the full subaccount balance to SNS governance, not just the accepted amount.

The excess ICP is permanently lost to the user. This is a **ledger conservation bug** / **token accounting bug** — user funds are transferred to a canister but not fully utilized or returned.

### Likelihood Explanation

This scenario is reachable by any unprivileged ingress sender participating in an SNS swap. It is triggered whenever:
- A user sends more ICP than `max_participant_icp_e8s` to their subaccount (a common mistake given the per-participant cap is not always obvious).
- The swap's remaining capacity (`available_direct_participation_e8s`) drops below the user's transferred amount between the time of transfer and the time of `refresh_buyer_tokens` (a race condition when the swap is nearly full).

Both scenarios are realistic and require no privileged access.

### Recommendation

In `refresh_buyer_token_e8s`, after computing `new_balance_e8s`, immediately refund the excess `e8s - new_balance_e8s` back to the buyer's principal on the ICP ledger before returning. This mirrors the fix recommended in the external report: check the actual accepted amount against the transferred amount and return the difference. The existing `TODO(NNS1-1682)` comment acknowledges this gap.

### Proof of Concept

1. SNS swap is open with `max_participant_icp_e8s = 5 ICP` and `available_direct_participation_e8s = 3 ICP`.
2. User transfers 6 ICP to their swap subaccount on the ICP ledger.
3. User calls `refresh_buyer_tokens`. The function reads `e8s = 6 ICP`, computes `actual_increment_e8s = min(3, 6) = 3 ICP`, records `new_balance_e8s = 3 ICP`. The remaining 3 ICP stays in the subaccount.
4. Swap reaches its target and commits. `finalize_swap` → `sweep_icp` is called.
5. `sweep_icp` transfers the full subaccount balance (6 ICP) to SNS governance, not just the 3 ICP accepted amount.
6. User loses 3 ICP permanently.
