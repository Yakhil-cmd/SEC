### Title
ERC-20 Fallback Address Accepts `Address::zero()` Without Validation, Causing Permanent Token Loss — (File: `engine/src/contract_methods/silo/mod.rs`)

---

### Summary

In Silo mode, `set_erc20_fallback_address` stores any `Address` value — including `Address::zero()` — without validation. The `SiloParamsArgs` struct derives `Default`, which sets `erc20_fallback_address` to `Address::zero()`. When the fallback address is zero and a non-whitelisted EVM recipient receives NEP-141 tokens via `ft_on_transfer`, the tokens are permanently minted to the zero address and irrecoverably lost.

---

### Finding Description

`set_erc20_fallback_address` in `engine/src/contract_methods/silo/mod.rs` accepts and persists any `Address` value without checking for the zero address: [1](#0-0) 

The `SiloParamsArgs` struct derives `Default`, which initialises `erc20_fallback_address` to `Address::zero()`: [2](#0-1) 

The unit test suite itself uses `Address::zero()` as the fallback address, confirming this is a realistic operator configuration: [3](#0-2) 

When `receive_erc20_tokens` is called and the stored fallback address is `Some(Address::zero())`, the recipient is silently overwritten with the zero address: [4](#0-3) 

The EVM zero address has no known private key. Tokens minted there are permanently inaccessible. There is no revert, no error, and no log distinguishing this from a legitimate fallback redirect.

This is structurally identical to the FRAX finding: just as `collatDolarValue` falls back to returning the raw balance (treating each unit as $1) when `oracle == address(0)`, `receive_erc20_tokens` falls back to minting to `Address::zero()` when the fallback address is zero — in both cases the "default" value is silently incorrect and causes irreversible accounting damage.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any NEP-141 tokens transferred to a non-whitelisted EVM address while the fallback is `Address::zero()` are deducted from the sender's NEP-141 balance and minted to the uncontrolled zero address. They cannot be recovered. The loss is silent and permanent.

---

### Likelihood Explanation

**Medium.** The `SiloParamsArgs::default()` value sets `erc20_fallback_address` to `Address::zero()`. An operator may call `set_silo_params(Some(SiloParamsArgs::default()))`, or may set the fallback to zero intending it to mean "no fallback" — a common EVM convention (`address(0)` as sentinel). The codebase's own unit tests use `Address::zero()` as the fallback address, demonstrating this is a plausible and accepted configuration. No on-chain guard rejects it. [5](#0-4) 

---

### Recommendation

Add a zero-address guard in `set_erc20_fallback_address`:

```rust
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        // Reject the zero address — it has no owner and tokens sent there are unrecoverable.
        assert!(!address.is_zero(), "ERR_ZERO_FALLBACK_ADDRESS");
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);
    }
}
```

Alternatively, add a guard in `receive_erc20_tokens` to skip the redirect when the fallback address is zero, treating it as "no fallback configured."

---

### Proof of Concept

1. Operator calls `set_silo_params(Some(SiloParamsArgs { fixed_gas: X, erc20_fallback_address: Address::zero() }))`. [6](#0-5) 

2. Operator enables the `Address` whitelist so that only approved addresses may receive tokens.

3. An unprivileged user calls `ft_on_transfer` on a NEP-141 token contract, specifying a non-whitelisted EVM address as the recipient message.

4. `receive_erc20_tokens` is invoked. `get_erc20_fallback_address` returns `Some(Address::zero())`. `is_allow_receive_erc20_tokens` returns `false` for the non-whitelisted recipient. The recipient is overwritten with `Address::zero()`. [4](#0-3) 

5. The ERC-20 `mint` call succeeds, crediting the tokens to `Address::zero()`. The user's NEP-141 balance is reduced. The tokens are permanently inaccessible.
