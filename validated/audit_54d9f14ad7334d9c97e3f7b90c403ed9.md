### Title
Base Tokens Sent to System Hook Addresses Are Permanently Locked With No Recovery Path - (`basic_bootloader/src/bootloader/runner.rs`, `system_hooks/src/call_hooks/`)

---

### Summary

Any user can send a transaction with non-zero `nominal_token_value` to a system hook address (e.g., `L1_MESSENGER_ADDRESS_HOOK` `0x7001`, `SET_BYTECODE_ON_ADDRESS_HOOK` `0x7002`, `MINT_HOOK_ADDRESS` `0x7100`, `CONTRACT_DEPLOYER_ADDRESS` `0x8006`). The runner unconditionally transfers the value to the hook address before invoking the hook. When called by an unauthorized caller, every one of these hooks returns **success** (pretending to be an empty account), so the value transfer is never rolled back. The hook addresses are pure Rust implementations with no bytecode, no storage, and no withdrawal function — accumulated base tokens are permanently locked.

---

### Finding Description

**Step 1 — Value is transferred before the hook runs.**

`perform_transfer_if_required` in the runner executes `transfer_nominal_token_value` from caller to callee unconditionally whenever `nominal_token_value != 0`: [1](#0-0) 

The target of the transfer is `call_request.callee` — the hook address itself.

**Step 2 — Unauthorized callers receive a success response.**

All four system hooks follow the same pattern: when the caller is not the privileged address, they return `make_return_state_from_returndata_region(available_resources, &[])` — a **successful** empty-account response — without touching the already-transferred value:

`mint_base_token_hook` (0x7100): [2](#0-1) 

`l1_messenger_hook` (0x7001): [3](#0-2) 

`set_bytecode_on_address_hook` (0x7002): [4](#0-3) 

`contract_deployer_temp_hook` (0x8006): [5](#0-4) 

Because the hook returns **success**, the frame rollback path is never taken, and the value remains credited to the hook address.

**Step 3 — Hook addresses have no withdrawal mechanism.**

The hook addresses are registered as pure Rust call hooks with no deployed bytecode, no storage, and no outbound transfer capability: [6](#0-5) 

There is no `adminTransferETH`, no `withdraw`, and no protocol-level drain function for these addresses. The only recovery path would be a full protocol upgrade.

**Step 4 — Address constants confirm the hook addresses.** [7](#0-6) 

---

### Impact Explanation

Base tokens (the L2 native token) sent to any of the four system hook addresses (`0x7001`, `0x7002`, `0x7100`, `0x8006`) are permanently locked. There is no on-chain mechanism to recover them short of a protocol upgrade. This is a direct, irreversible loss of user funds.

---

### Likelihood Explanation

The entry path is fully unprivileged: any EOA or contract can issue a CALL with non-zero value to a hook address. This can happen accidentally (e.g., a user mistakenly targets `0x7001` instead of `0x8008`) or deliberately (griefing). The `check_depth_and_balance` guard only verifies the caller has sufficient balance — it does not block calls to hook addresses: [8](#0-7) 

No special privilege, leaked key, or oracle manipulation is required.

---

### Recommendation

Add a value-rejection guard at the runner level for system hook addresses, or inside each hook's unauthorized-caller early-return path. The simplest fix mirrors what the hooks already do for their *authorized* callers: check `nominal_token_value != U256::ZERO` and return `make_error_return_state` (failure) instead of `make_return_state_from_returndata_region` (success) when an unauthorized caller sends value. This causes the frame to roll back and the value transfer to be undone.

Alternatively, add a protocol-level drain function callable by the operator/governance to recover any accidentally accumulated balance from hook addresses.

---

### Proof of Concept

1. Deploy a funded EOA with balance `B > 0`.
2. Submit an EVM transaction: `from = EOA`, `to = 0x7001` (L1 messenger hook), `value = B`, `calldata = []`.
3. `perform_transfer_if_required` transfers `B` from EOA to `0x7001`.
4. `l1_messenger_hook` fires; caller is not `0x8008`, so it returns success with empty returndata.
5. No rollback occurs. EOA balance is now `0`; `0x7001` balance is `B`.
6. No transaction can ever move `B` out of `0x7001` — the address has no bytecode and no hook-level withdrawal path. [9](#0-8) [3](#0-2)

    /// Check if transfer is requested and try to perform it
    fn perform_transfer_if_required(
        &mut self,
        call_request: &mut ExternalCallRequest<S>,
        caller_ee_type: ExecutionEnvironmentType,
    ) -> Result<bool, BootloaderSubsystemError>
    where
        S::IO: IOSubsystemExt,
    {
        if call_request.nominal_token_value.is_zero() || call_request.is_delegate() {
            return Ok(true);
        }

        // Check transfer is allowed and determine transfer target
        if !call_request.is_transfer_allowed() {
            system_log!(
                self.system,
                "Call failed: positive value with modifier {:?}\n",
                call_request.modifier
            );
            return Err(internal_error!("Positive value with incorrect modifier").into());
        }
        // Adjust transfer target due to CALLCODE
        // TODO: in future should be moved to EE
        let target = match call_request.modifier {
            CallModifier::EVMCallcode | CallModifier::EVMCallcodeStatic => call_request.caller,
            _ => call_request.callee,
        };

        match call_request
            .available_resources
            .with_infinite_ergs(|inf_resources| {
                self.system.io.transfer_nominal_token_value(
                    ExecutionEnvironmentType::NoEE,
                    inf_resources,
                    &call_request.caller,
                    &target,
                    &call_request.nominal_token_value,
                )
            }) {
            Ok(()) => Ok(true),
            Err(e) => {
                match e {
                    SubsystemError::LeafUsage(_interface_error) => {
                        let _ = self
                            .system
                            .get_logger()
                            .write_fmt(format_args!("Insufficient balance for transfer\n"));

                        // Insufficient balance
                        match caller_ee_type {
                            ExecutionEnvironmentType::NoEE => Err(interface_error!(
                                BootloaderInterfaceError::TopLevelInsufficientBalance
                            ))
                            .with_context(|| {
                                error_ctx! {
                                     "caller" => debug_format(call_request.caller),
                                     "target" => debug_format(target),
                                }
                            }),
                            ExecutionEnvironmentType::EVM => {
                                // Following EVM, a call with insufficient balance is not a revert,
                                // but rather a normal failing call.
                                Ok(false)
                            }
                        }
                    }
                    SubsystemError::LeafDefect(_) => Err(wrap_error!(e)),
                    SubsystemError::LeafRuntime(ref runtime_error) => match runtime_error {
                        RuntimeError::FatalRuntimeError(_) => Err(wrap_error!(e)),
                        RuntimeError::OutOfErgs(_) => {
                            Err(internal_error!("Out of ergs on infinite ergs").into())
                                .with_context(|| {
                                    error_ctx! {
                                        "inner" => runtime_error,
                                    }
                                })
                        }
                    },
                    SubsystemError::Cascaded(cascaded_error) => match cascaded_error {},
                }
            }
        }
    }
