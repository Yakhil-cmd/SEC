Audit Report

## Title
SNS Governance `distribute_rewards` Unbounded Neuron Iteration Causes Permanent Reward Liveness Failure - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The `distribute_rewards` function in the SNS Governance canister iterates over all voting neurons in a single synchronous call with no instruction-limit guard. If the neuron count grows large enough to exhaust the IC per-message instruction limit, the message traps and rolls back all state. Because `latest_reward_event` is only written after the loop completes, `should_distribute_rewards` returns `true` again on every subsequent timer invocation, permanently halting voting reward distribution.

## Finding Description
`run_periodic_tasks` calls `distribute_rewards` whenever `should_distribute_rewards()` returns `true`: [1](#0-0) 

`should_distribute_rewards` returns `true` based solely on elapsed time since `latest_reward_event.end_timestamp_seconds`: [2](#0-1) 

Inside `distribute_rewards`, after computing `neuron_id_to_reward_shares` from all proposal ballots, the function enters an unbounded loop over every voting neuron: [3](#0-2) 

There is no call to `is_message_over_threshold` or any equivalent guard anywhere in `rs/sns/governance/src/`. The loop runs to completion or the message traps.

`latest_reward_event` is only written **after** the loop and proposal settlement: [4](#0-3) 

Proposal settlement (clearing ballots, setting `reward_event_round`) also occurs after the loop: [5](#0-4) 

If the loop traps due to instruction exhaustion, all state changes roll back: `latest_reward_event` is unchanged, proposals remain in `ReadyToSettle`, and the next timer invocation re-enters the identical (or larger) loop and traps again — indefinitely.

By contrast, NNS Governance uses `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` inside a resumable state machine: [6](#0-5) [7](#0-6) 

SNS Governance has no equivalent mechanism.

## Impact Explanation
Once the neuron-iteration loop exceeds the IC per-message instruction limit (~40 billion instructions), every timer invocation of `run_periodic_tasks` traps at the same point. `latest_reward_event` is never advanced, so the canister is permanently stuck: voting rewards are never distributed again, neuron maturity stops accruing, and the economic incentive for governance participation is broken. This is a **governance liveness failure** matching the allowed High impact: *"Application/platform-level DoS … or subnet availability impact"* and *"Significant … SNS … security impact with concrete user or protocol harm."*

## Likelihood Explanation
This is an organic, non-adversarial failure requiring no special privileges. Any SNS DAO that accumulates a sufficiently large number of active voting neurons will hit this threshold. The NNS governance team already acknowledged and fixed the identical bug in NNS Governance (Proposal 135702), explicitly noting: *"Distribute rewards is moved to timer, and has a mechanism to distribute in batches in multiple messages"* and *"Unstaking maturity task has a limit of 100 neurons per message, which prevents it from exceeding instruction limit."* The SNS canister did not receive the same remediation.

## Recommendation
Apply the same batched-distribution pattern already used in NNS Governance:
1. After computing `neuron_id_to_reward_shares`, store the pending distribution in stable memory (analogous to `RewardsDistributionStateMachine`).
2. Advance `latest_reward_event` and settle proposals immediately so they are not re-processed on the next invocation.
3. Distribute maturity to neurons in a separate recurring timer task that checks `is_message_over_threshold` after each neuron and resumes in the next message if the limit is reached.

## Proof of Concept
1. An SNS DAO accumulates a large number of neurons (e.g., 50,000+), all of which vote on proposals.
2. The recurring timer fires `run_periodic_tasks` → `should_distribute_rewards()` returns `true` → `distribute_rewards(supply)` is called.
3. The loop at line 5954 iterates over all `neuron_id_to_reward_shares` entries; after enough neurons, the IC instruction counter exceeds the per-message limit and the message traps.
4. All state changes roll back. `latest_reward_event` is unchanged. Proposals remain in `ReadyToSettle`.
5. The next timer fires; `should_distribute_rewards()` returns `true` again (same epoch, same proposals). The identical loop runs and traps again.
6. Voting rewards are permanently halted.

A deterministic integration test or PocketIC benchmark can reproduce this by creating 50,000+ neurons that all vote on a proposal, advancing the clock past one reward round, and invoking `run_periodic_tasks` — observing that `latest_reward_event` is never updated and the canister traps on each invocation.
