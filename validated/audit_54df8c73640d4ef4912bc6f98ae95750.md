Audit Report

## Title
Caller-Supplied `buyer` Principal in `refresh_buyer_tokens` Bypasses Confirmation-Text Consent and Enables Forced Participation - (`rs/sns/swap/canister/canister.rs`)

## Summary
The SNS Swap canister's `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal from the caller without verifying it equals `msg_caller()`. When `confirmation_text` is configured by the SNS operator as a legal/compliance gate, any unprivileged third party can satisfy that gate on behalf of a victim who has already deposited ICP into their swap subaccount, registering the victim as a committed participant without the victim ever sending the confirmation text themselves. The victim's ICP is locked in the swap until finalization.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 130–134, the buyer principal is resolved from caller-supplied input with no identity check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // arbitrary attacker-supplied value
};
```

This `p` is forwarded directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`. Inside that function, `validate_confirmation_text` (lines 363–384) checks only that the supplied string matches the SNS-configured public text — it performs no check that the caller is the same principal as `buyer`:

```rust
self.validate_confirmation_text(confirmation_text)?;
```

The ICP balance is then read from the subaccount keyed by the attacker-supplied `buyer` (line 1156), and the participation is recorded under that same key (lines 1285–1288). The confirmation text is a **public** string visible in the SNS init payload; any caller who knows it (trivially, since it is public) can supply it on behalf of any victim.

The `RefreshBuyerTokensRequest` proto explicitly documents the `buyer` override as an intentional feature ("If not specified, the caller is used"), and integration tests in `rs/tests/nns/sns/lib/src/sns_deployment.rs` (lines 749–756) confirm the design allows a different caller to notify on behalf of a buyer. However, the function's own docstring (line 1123) states "the **caller** of this function must accept this confirmation" — a direct contradiction when `buyer ≠ caller`.

The existing guards (`validate_lifecycle_is_open`, `validate_possibility_of_direct_participation`, minimum/maximum ICP checks) are all orthogonal to the caller/buyer identity mismatch and do not mitigate it.

## Impact Explanation
**Confirmation-text consent bypass (High):** The `confirmation_text` field is the SNS operator's only mechanism to enforce explicit participant consent (e.g., legal terms, KYC acknowledgment). Because the text is public and the canister never verifies that the consenting caller is the economic subject (`buyer`), an attacker trivially satisfies the gate on behalf of any victim. This constitutes a significant SNS security impact with concrete protocol harm: the SNS operator's compliance invariant is violated for every victim registered this way.

**Forced participation / ICP lock (High):** Any principal who has pre-funded their swap subaccount (via the ticket-based flow or a direct transfer) can be forcibly committed to the swap by a third party before they decide to participate. Once committed, the ICP is locked until the swap finalizes (committed or aborted). This matches the allowed impact: "Significant SNS security impact with concrete user or protocol harm."

**Participant-slot exhaustion (High):** An attacker can enumerate funded subaccounts and exhaust `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` with victim principals, preventing legitimate new participants from joining.

## Likelihood Explanation
- `refresh_buyer_tokens` is a public `#[update]` endpoint with no cycles cost, no token cost, and no access control.
- The `buyer` override is documented and trivially exercised.
- The confirmation text is a public string; no secret knowledge is required.
- Any principal using the ticket-based payment flow is vulnerable during the window between their ICP transfer and their own `refresh_buyer_tokens` call.
- No special privileges, key material, or majority corruption is required.

## Recommendation
Enforce `buyer == caller` at the canister entry point. When `arg.buyer` is non-empty, verify it equals `caller_principal_id()` before proceeding, or remove the override entirely:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    if !arg.buyer.is_empty() {
        let supplied = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, supplied, "buyer must equal caller");
    }
    swap_mut()
        .refresh_buyer_token_e8s(caller, arg.confirmation_text, ...)
        .await
}
```

If third-party notification is a required use-case, add an explicit on-chain pre-authorization step (e.g., the victim calls an `authorize_notifier` endpoint) so that the victim's consent is cryptographically established before a third party can act on their behalf.

## Proof of Concept
1. SNS is configured with `confirmation_text = "I agree to the terms"`.
2. Alice transfers 10 ICP to `swap_canister_subaccount(alice_principal)` on the ICP ledger (e.g., via the ticket-based flow).
3. Before Alice calls `refresh_buyer_tokens` herself, Mallory (any principal) calls:
   ```
   refresh_buyer_tokens({
     buyer: alice_principal.to_string(),
     confirmation_text: Some("I agree to the terms")
   })
   ```
4. The canister resolves `p = alice_principal`, reads Alice's subaccount balance (10 ICP), passes `validate_confirmation_text` with Mallory's supplied text, and records Alice as a committed participant.
5. Alice's ICP is now locked in the swap. Alice never sent the confirmation text. The SNS operator's compliance gate is bypassed.

A deterministic unit test can be written against `refresh_buyer_token_e8s` directly (as in `rs/sns/swap/tests/swap.rs`) by passing `buyer = alice` and `confirmation_text = Some(expected_text)` from a mock caller that is not Alice, then asserting `self.buyers` contains Alice's entry — which the existing test infrastructure already supports.
