### Title
Silent Partial-Transfer Failure in `AccountState::transfer` Permanently Orphans GTC Neurons After `donate_account` Sets `has_donated = true` — (`rs/nns/gtc/src/lib.rs`)

### Summary

`AccountState::transfer` iterates over all neuron IDs, calls `GovernanceCanister::transfer_gtc_neuron` for each, removes every neuron from `self.neuron_ids` regardless of outcome, and unconditionally returns `Ok(())`. When any individual call fails, the neuron is silently moved to `failed_transferred_neurons`. The caller (`donate_account`) then sets `has_donated = true`. Any subsequent call to `donate_account` reaches the `has_donated` guard inside `transfer` and returns an error, making the failed neurons permanently unrecoverable.

### Finding Description

**Entrypoint:** `canister_update donate_account` — publicly callable by any principal who owns a GTC account. [1](#0-0) 

**Root cause — `transfer` always returns `Ok(())`:** [2](#0-1) 

For every `neuron_id`, the neuron is unconditionally removed from `self.neuron_ids` at line 192 before the result is even inspected. On failure the neuron is appended to `failed_transferred_neurons` (line 204), but the function still falls through to `Ok(())` at line 209. There is no early return, no error propagation, and no rollback.

**Root cause — `donate_account` sets the permanent flag regardless:** [3](#0-2) 

Because `transfer` always returns `Ok(())`, the `?` on line 89 never short-circuits, and `has_donated = true` is set unconditionally.

**Root cause — retry is permanently blocked:** [4](#0-3) 

A second call to `donate_account` → `transfer` hits the `has_donated` guard at line 177 and returns `Err("Account has already donated its funds")`. There is no administrative escape hatch, no `retry_failed_neurons` method, and no governance proposal path to recover orphaned neurons.

**State after partial failure:**
- `self.neuron_ids` is empty (all neurons removed regardless of outcome).
- `failed_transferred_neurons` contains the neurons whose governance transfer failed.
- The neurons still exist in the Governance canister, still controlled by the GTC canister principal, but the GTC canister has no code path to re-attempt the transfer. [5](#0-4) 

### Impact Explanation

Any GTC account whose `donate_account` call experiences even a single transient `transfer_gtc_neuron` failure (governance canister busy, message-queue rejection, canister upgrade in flight, cycles exhaustion, etc.) will have those neurons permanently orphaned. The neurons' ICP stake is locked in governance under GTC canister control with no mechanism to move, dissolve, or reclaim it. Seed-round accounts hold up to tens of millions of ICP; a single partial failure destroys that stake irreversibly.

### Likelihood Explanation

Inter-canister calls on the IC can fail for transient reasons (reject codes, queue limits, canister upgrades). The `transfer` loop makes one inter-canister call per neuron; accounts with many neurons increase the probability of at least one failure. The failure condition is not attacker-controlled but is realistic under normal mainnet operation. Once triggered, the loss is permanent and unrecoverable without a canister upgrade.

### Recommendation

1. **Propagate failures:** Change `transfer` to return `Err(...)` if any individual `transfer_gtc_neuron` call fails, and do **not** remove the neuron from `self.neuron_ids` on failure, so a retry is possible.
2. **Atomic flag setting:** Only set `has_donated = true` (or `has_forwarded = true`) after confirming all neurons transferred successfully.
3. **Alternatively:** Keep failed neurons in `neuron_ids` and only move them to `successfully_transferred_neurons` on success, so a subsequent call can retry the remaining set.
