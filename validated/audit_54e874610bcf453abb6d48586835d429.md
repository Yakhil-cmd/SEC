### Title
`native_per_pubdata` Computed with Floor Division Instead of Ceiling Division, Allowing Free Pubdata When `pubdata_price < native_price` - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` — the native-resource cost per byte of pubdata — is computed using truncating (floor) integer division (`wrapping_div`), while the analogous `native_per_gas` ratio uses ceiling division (`div_ceil`). When the operator-supplied `pubdata_price` is any positive value strictly less than `native_price`, the floor division produces zero, making all pubdata effectively free. An unprivileged transaction sender can then submit pubdata-heavy transactions and force the protocol to publish data to L1 without paying the intended native-resource cost.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (ZK flow), the two key ratios are computed as:

```rust
// native_per_gas — uses ceiling division
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [1](#0-0) 

```rust
// native_per_pubdata — uses floor (truncating) division
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same asymmetry is reproduced in the public API helper:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;
// native_per_pubdata = pubdata_price / native_price  ← floor
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

`native_per_pubdata` is then forwarded into `create_resources_for_tx`, where it is multiplied by the intrinsic pubdata count to deduct from the user's native limit:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
let native_limit = match native_limit.checked_sub(intrinsic_pubdata_overhead) { ... };
``` [4](#0-3) 

When `native_per_pubdata == 0`, `intrinsic_pubdata_overhead == 0`, so the native limit is never reduced for pubdata. The same zero value propagates into the runtime pubdata charge path and into the `deltaGas` adjustment in `compute_gas_refund`:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [5](#0-4) 

Because `native_per_pubdata` is zero, pubdata native consumption is never reflected back into `gas_used`, so the user is never charged for it.

---

### Impact Explanation

When the operator sets `pubdata_price` to any positive value strictly less than `native_price` (e.g., `pubdata_price = 99`, `native_price = 100`), floor division yields `native_per_pubdata = 0`. Every transaction then produces pubdata at zero native cost. The operator must still pay to publish that data to L1, but receives no compensation from the transaction sender. A malicious sender can craft transactions that maximise pubdata output (large storage writes, deployments) and repeat them indefinitely, draining the operator's L1 publication budget.

---

### Likelihood Explanation

The operator adjusts `pubdata_price` and `native_price` dynamically in response to L1 gas costs and proving costs. Any configuration where `pubdata_price` is a non-zero value smaller than `native_price` — a common transient state during price adjustments — silently enables the attack. No special privilege is required from the attacker; a standard L2 transaction is sufficient.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that any non-zero `pubdata_price` always results in at least 1 native unit charged per pubdata byte, eliminating the precision-loss window.

---

### Proof of Concept

1. Operator sets `native_price = 100`, `pubdata_price = 50` (a realistic ratio where pubdata is cheaper than proving).
2. Bootloader computes `native_per_pubdata = 50 / 100 = 0` (floor division).
3. `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0`; native limit is not reduced.
4. Attacker submits a transaction with maximum storage writes (each write produces pubdata). The transaction's native limit is entirely consumed by computational work; zero native is charged for pubdata.
5. The operator publishes the pubdata to L1 at its own expense. The attacker repeats step 4 indefinitely.

With `div_ceil`, step 2 would yield `native_per_pubdata = ceil(50/100) = 1`, and each pubdata byte would cost 1 native unit, correctly reflecting the operator's intended pricing.
