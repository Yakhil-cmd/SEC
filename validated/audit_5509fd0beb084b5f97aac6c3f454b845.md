Audit Report

## Title
Silent ICP Burn Failure Allows Cycles Minting Without ICP Destruction, Violating Conservation Invariant - (`rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the CMC is intentionally designed to swallow all ICP burn errors after cycles have already been irreversibly minted or deposited. If the ICP ledger rejects the burn call (e.g., during a canister upgrade), cycles are permanently created while the corresponding ICP remains unburned in the CMC's subaccount. Because the notification block index is then recorded as successfully processed, no retry is possible, leaving the ICP stranded and the ICP/cycles conservation invariant violated.

## Finding Description
`burn_and_log` at `rs/nns/cmc/src/main.rs:2014-2049` explicitly silences all errors:

```rust
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    ...
    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
            // returns () — no error propagated
        }
    }
}
``` [1](#0-0) 

This function is called in three paths only after the irreversible action has already succeeded:

- `process_top_up`: `deposit_cycles` succeeds → `burn_and_log` called → `Ok(cycles)` returned
- `process_mint_cycles`: `do_mint_cycles` succeeds → `burn_and_log` called → `Ok(...)` returned
- `process_create_canister`: `do_create_canister` succeeds → `burn_and_log` called → `Ok(canister_id)` returned [2](#0-1) [3](#0-2) [4](#0-3) 

Because `burn_and_log` returns `()` on failure, the calling function returns `Ok(...)`. This `Ok` result is then stored in `blocks_notified` as `NotificationStatus::NotifiedTopUp(Ok(cycles))` (or equivalent). The `is_transient_error` guard only removes the entry for error results, not for `Ok` results, so the block is permanently marked as processed. [5](#0-4) 

Any subsequent call with the same `block_index` hits the `Entry::Occupied` branch and returns the cached `Ok` result without re-executing the burn. The ICP in the CMC's subaccount is permanently stranded with no user-accessible recovery path. [6](#0-5) 

## Impact Explanation
This is a ledger conservation violation: cycles are minted without the corresponding ICP being destroyed. The ICP total supply is not reduced as expected, inflating it relative to the cycles created. The stranded ICP can only be recovered via an NNS governance proposal — an out-of-band privileged action unavailable to the affected user. This matches the allowed impact class: **significant NNS/ledger security impact with concrete protocol harm** (High, $2,000–$10,000), and potentially **illegal minting** if accumulated across many transactions during a ledger upgrade window (Critical, $10,000–$50,000).

## Likelihood Explanation
The ICP ledger undergoes periodic upgrades on mainnet during which it is briefly unavailable and rejects inter-canister calls. Any user who calls `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` during this window will trigger the condition non-adversarially. No special privileges or attacker-controlled conditions are required — the trigger is a normal operational event. The per-transaction likelihood is low-to-medium, but the CMC processes high transaction volume, and a single ledger upgrade window could affect multiple concurrent notifications simultaneously.

## Recommendation
The burn should not be silenced. The correct fix is to attempt the burn **before** minting cycles, so a burn failure prevents the mint rather than leaving ICP stranded. If the post-mint ordering must be preserved for UX reasons, a persistent retry queue (recorded in stable state before minting) should be used to retry failed burns in a background task — the same pattern used by the ckBTC minter. At minimum, a failed burn should cause the notification entry to be removed from `blocks_notified` so the user can retry, accepting the double-notification risk as the lesser evil compared to permanent ICP loss.

## Proof of Concept
1. User sends N ICP to the CMC subaccount corresponding to their canister via the ICP ledger (block index B).
2. User calls `notify_top_up` on the CMC with block index B.
3. CMC fetches the transaction, sets `blocks_notified[B] = Processing`.
4. CMC calls `deposit_cycles` — succeeds, cycles are irreversibly deposited.
5. CMC calls `burn_and_log`. The ICP ledger is mid-upgrade and returns a reject code.
6. `burn_and_log` logs the error and returns `()`.
7. `process_top_up` returns `Ok(cycles)`.
8. CMC sets `blocks_notified[B] = NotifiedTopUp(Ok(cycles))`.
9. Any retry of `notify_top_up` with block index B returns the cached `Ok` without re-burning.
10. N ICP remain permanently in the CMC's subaccount; cycles were minted without ICP destruction.

A deterministic integration test using PocketIC can reproduce this by: (a) deploying a mock ledger that rejects `send_pb` calls, (b) calling `notify_top_up` against the CMC, (c) asserting that the CMC subaccount balance is non-zero and `blocks_notified` contains a successful `NotifiedTopUp` entry.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
