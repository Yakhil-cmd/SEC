Looking at the exact code in question:

### Title
GenericDisplayMessage Silently Rounds Token Amounts via `to_f64()` Precision Loss — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

---

### Summary

`convert_tokens_to_string_representation` converts a `Nat` amount to `f64` before formatting, losing precision for any integer > 2^53. The parallel path for `FieldsDisplayMessage` uses `nat_to_u64`, which is exact. An unprivileged caller can trigger this divergence by submitting a `ConsentMessageRequest` with an amount in `(2^53, u64::MAX]`, causing the two display modes to show different numeric values for the same transaction.

---

### Finding Description

`add_amount` dispatches to one of two code paths depending on the `ConsentMessage` variant:

- **`GenericDisplayMessage`** path calls `convert_tokens_to_string_representation`, which does:
  ```rust
  let tokens = tokens.0.to_f64().ok_or(...)?;
  Ok(format!("{}", tokens / 10_f64.pow(decimals)))
  ``` [1](#0-0) 

- **`FieldsDisplayMessage`** path calls `nat_to_u64`, which does:
  ```rust
  tokens.0.to_u64().ok_or(...)
  ``` [2](#0-1) 

IEEE 754 `f64` has a 53-bit mantissa. Any integer > 2^53 cannot be represented exactly. `num_bigint::BigUint::to_f64()` rounds to the nearest representable value. `to_u64()` is exact for all values in `[0, u64::MAX]`.

The `add_amount` dispatch: [3](#0-2) 

The display type is set directly from the caller-supplied `device_spec`: [4](#0-3) 

Any unprivileged user can call `icrc21_canister_call_consent_message` on a deployed ICRC-1 ledger: [5](#0-4)

### Impact Explanation

A malicious dApp supplies `TransferArg { amount = 2^53 + 1, ... }` to a wallet. The wallet calls `icrc21_canister_call_consent_message` with `device_spec = GenericDisplay`. The consent message string shows `2^53 / 10^decimals` (rounded down by 1 base unit). The user approves. The actual on-chain transfer executes with the full `2^53 + 1` base units.

The rounding error grows with the magnitude of the amount. For values near `u64::MAX`, the ULP of `f64` is 2048, so the displayed amount can differ from the actual amount by up to 1024 base units. For a token with 0 decimals, that is 1024 tokens silently omitted from the consent display. For tokens with 8 decimals (e.g., ICP), the maximum discrepancy is ~0.00001024 ICP — negligible in practice, but the principle holds for any token configuration.

The invariant that `GenericDisplayMessage` and `FieldsDisplayMessage` must present identical amounts to the user is concretely broken.

---

### Likelihood Explanation

- The entrypoint is public and requires no privilege.
- The triggering condition (amount > 2^53 base units) is reachable for any token with sufficient supply or low decimals.
- The wallet must use `GenericDisplay` mode, which is the default when `device_spec` is absent: [6](#0-5) 
- No existing guard in `build_icrc21_consent_info` validates that the amount fits in the exact range of `f64` integers before calling `convert_tokens_to_string_representation`. [7](#0-6) 

---

### Recommendation

Replace the `to_f64()` path in `convert_tokens_to_string_representation` with exact integer arithmetic. Use `num_bigint` division and modulo to split the amount into whole and fractional parts, then format them as a decimal string without any floating-point conversion. This is the same approach used by most token display libraries and eliminates the precision loss entirely.

---

### Proof of Concept

```rust
// Differential unit test
let amount = Nat::from(2u128.pow(53) + 1); // 9007199254740993
let decimals = 8u8;

// GenericDisplay path
let generic_str = convert_tokens_to_string_representation(amount.clone(), decimals).unwrap();
// generic_str == "90071992.54740992"  (rounded DOWN by 1 base unit)

// FieldsDisplay path
let exact_u64 = nat_to_u64(amount).unwrap();
// exact_u64 == 9007199254740993  (correct)
// human-readable: 90071992.54740993

assert_eq!(generic_str, "90071992.54740993"); // FAILS
```

The two display modes produce different numeric values for the same input, confirming the invariant violation.

fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,

    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError {
        error_code: Nat::from(500_u64),
        description: "Failed to convert tokens to u64".to_owned(),
    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
