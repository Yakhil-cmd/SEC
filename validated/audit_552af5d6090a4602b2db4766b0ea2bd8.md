Audit Report

## Title
Stale ICP/XDR Conversion Rate Used in Cycles Minting Without Freshness Check — (File: `rs/nns/cmc/src/main.rs`)

## Summary
The Cycles Minting Canister stores the ICP/XDR conversion rate as `Option<IcpXdrConversionRate>` carrying both a rate value and a `timestamp_seconds` field. The `tokens_to_cycles()` function, called by all three public minting endpoints, reads only the rate value and never inspects `timestamp_seconds`. When the Exchange Rate Canister (XRC) is unavailable, the heartbeat-driven refresh fails silently and the cached rate remains at its last successfully fetched value indefinitely, allowing any caller to mint cycles at a potentially outdated rate.

## Finding Description
`tokens_to_cycles()` at `rs/nns/cmc/src/main.rs` L1900–1923 reads `state.icp_xdr_conversion_rate` and branches only on `Some`/`None` — the `timestamp_seconds` field carried in `IcpXdrConversionRate` is extracted but immediately discarded via `.map(|rate| rate.xdr_permyriad_per_icp)`. No maximum-age check exists anywhere in the CMC source.

The rate is refreshed by the heartbeat at `rs/nns/cmc/src/main.rs` L2397–2402, which calls `update_exchange_rate()`. On any XRC call failure (`FailedToRetrieveRate`, `FailedToSetRate`, `InvalidRate`), `schedule_next_attempt()` in `rs/nns/cmc/src/exchange_rate_canister.rs` L149–162 reschedules the next attempt one minute later but leaves `state.icp_xdr_conversion_rate` unchanged. The `UpdateExchangeRateState::Disabled` path at L98–100 (triggered by a governance `DivergedRate` proposal) permanently halts XRC queries while also leaving the cached rate in place.

All three public minting paths call `tokens_to_cycles()` unconditionally before any ICP is burned:
- `process_top_up` → L1991
- `process_create_canister` → L1932
- `process_mint_cycles` → L1965

No caller authentication or rate-freshness guard precedes these calls.

## Impact Explanation
This maps to the **High** impact class: *Significant XRC/NNS/infrastructure security impact with concrete user or protocol harm.* If ICP market price falls while the XRC is unavailable, the CMC mints cycles at the stale (higher) ICP/XDR rate. Because cycles are backed by ICP at the prevailing market rate and the CMC is the sole authorized minter, this constitutes over-issuance of cycles — a direct protocol accounting violation. The inverse (stale rate too low) harms users but not the protocol. The over-issuance direction is the security-relevant case. The impact does not reach Critical because exploitation requires a pre-existing XRC outage not caused by the attacker, the window is bounded by outage duration, and NNS governance retains an out-of-band fallback to set the rate via proposal.

## Likelihood Explanation
The XRC is a separate system canister on the NNS subnet. Routine canister upgrades, transient inter-canister call timeouts, or a canister trap each produce a `FailedToRetrieveRate` error that leaves the cached rate unchanged. The `Disabled` state (governance-triggered on diverged rate) creates an unbounded staleness window. Any unprivileged principal holding a valid ICP ledger block can call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` with no additional preconditions. The attacker need only observe that the XRC is unavailable (detectable via the public `get_icp_xdr_conversion_rate` query) and that the cached rate is higher than the current market rate, then submit a normal ICP transfer and notification.

## Recommendation
In `tokens_to_cycles()` (`rs/nns/cmc/src/main.rs` L1900), after reading `state.icp_xdr_conversion_rate`, compare `rate.timestamp_seconds` against the current time obtained via `ic_cdk::api::time() / 1_000_000_000`. If the rate age exceeds a defined threshold (e.g., 30 minutes, or a governance-configurable `max_rate_age_seconds` stored in `StateV2`), return `NotifyError::Other` with a descriptive message rather than proceeding with the conversion. This mirrors the existing `None` guard and requires no changes to the refresh path.

## Proof of Concept
1. Confirm the XRC is unavailable (e.g., during a scheduled upgrade): query `get_icp_xdr_conversion_rate` on the CMC and observe the `timestamp_seconds` is not advancing.
2. Wait until the cached rate is meaningfully higher than the current ICP market price (e.g., rate set at 10 XDR/ICP, market now at 5 XDR/ICP).
3. Transfer 1 ICP to the CMC top-up subaccount for a target canister via the ICP ledger.
4. Call `notify_top_up` with the resulting block index.
5. Observe that `process_top_up` → `tokens_to_cycles` returns cycles computed from the stale 10 XDR/ICP rate, yielding ~2× the cycles justified by the current market rate.
6. A deterministic integration test can reproduce this by: initializing CMC state with a known rate and timestamp, advancing the mock clock beyond any freshness threshold, injecting a mock XRC that always returns an error, and asserting that `notify_top_up` succeeds and returns the over-issued cycle count.
