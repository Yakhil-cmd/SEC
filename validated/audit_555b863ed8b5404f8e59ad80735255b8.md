### Title
Missing Access Control on `refund_on_error` Allows Arbitrary ERC-20 Minting and ETH Theft - (File: engine/src/engine.rs)

### Summary
The Aurora Engine exposes a `refund_on_error` NEAR entrypoint that internally calls `engine::refund_on_error`, a function with no caller access control. Any unprivileged NEAR account can invoke it to mint arbitrary amounts of any registered ERC-20 (bridged NEP-141) token to any address, or to drain ETH from the `ExitToNear` precompile address.

### Finding Description

The function `refund_on_error` in `engine/src/engine.rs` performs two privileged operations with no caller check:

**ERC-20 branch** (when `args.erc20_address` is `Some`): it constructs the engine's own EVM admin address via `current_address(&current_account_id)`, then calls the `EvmErc20.mint(recipient, amount)` selector on the target ERC-20 contract acting as the admin. [1](#0-0) 

The `setup_refund_on_error_input` helper encodes the ERC-20 `mint` selector with attacker-controlled `refund_address` and `amount`: [2](#0-1) 

The `EvmErc20` and `EvmErc20V2` contracts gate `mint` with `onlyAdmin`, where admin is set to `current_address(current_account_id)` at deploy time: [3](#0-2) 

Because `refund_on_error` calls the ERC-20 contract with `erc20_admin_address` as the EVM `msg.sender`, the `onlyAdmin` check passes unconditionally.

**ETH branch** (when `args.erc20_address` is `None`): it transfers ETH from the `ExitToNear` precompile address to any attacker-specified `recipient_address`: [4](#0-3) 

The `refund_on_error` NEAR entrypoint is confirmed as a production-callable method via the workspace operation registry: [5](#0-4) [6](#0-5) 

Contrast this with the callback functions that perform similar privileged operations — `exit_to_near_precompile_callback` and `withdraw_wnear_to_router` — which both enforce `env.assert_private_call()` before executing: [7](#0-6) [8](#0-7) 

The `assert_private_call` guard requires `predecessor_account_id == current_account_id`, i.e., only the contract itself can call: [9](#0-8) 

The `refund_on_error` function has no equivalent guard.

### Impact Explanation

**Critical — Direct theft of user funds (ERC-20 minting):** Any NEAR account can call `refund_on_error` with an arbitrary registered ERC-20 address, an arbitrary recipient, and an arbitrary amount. Because the engine's EVM address is the admin of every `EvmErc20` contract it deploys, the `mint` call succeeds. The attacker receives unbacked bridged tokens redeemable for real NEP-141 assets held by the Aurora connector, draining the bridge.

**High — Theft of in-motion ETH:** Any NEAR account can call `refund_on_error` with `erc20_address = None` and an arbitrary `recipient_address` and `amount`. This transfers ETH from the `ExitToNear` precompile address to the attacker. ETH accumulates at that address during normal exit flows.

### Likelihood Explanation

The entrypoint is publicly reachable by any NEAR account with no preconditions (no attached deposit, no whitelist, no role). The `RefundCallArgs` struct is borsh-serialized and fully attacker-controlled. The attack requires a single NEAR transaction. Likelihood: **5/5**.

### Recommendation

Add `env.assert_private_call()?` (or an equivalent `require_owner_only` check) to the `refund_on_error` NEAR entrypoint in `engine/src/lib.rs`, mirroring the pattern already used in `exit_to_near_precompile_callback` and `withdraw_wnear_to_router`. The function is only ever legitimately invoked as an internal callback from `exit_to_near_precompile_callback`, so restricting it to self-calls is the correct fix.

### Proof of Concept

```
# Attacker NEAR account: attacker.near
# Target: any registered EvmErc20 at EVM address 0xABC...
# Recipient: attacker's EVM address 0xDEF...
# Amount: U256::MAX

near call aurora refund_on_error \
  "$(python3 -c "
import borsh  # borsh-encode RefundCallArgs:
# recipient_address: [0xDE, 0xF..., 20 bytes]
# erc20_address: Some([0xAB, 0xC..., 20 bytes])
# amount: [0xFF * 32 bytes]
print(borsh_encode(recipient, Some(erc20), amount))
")" \
  --accountId attacker.near
```

The engine calls `EvmErc20(0xABC).mint(0xDEF, U256::MAX)` as the admin, minting an unbounded supply of bridged tokens to the attacker. The attacker then calls `withdrawToNear` or `withdrawToEthereum` to redeem the minted tokens for real assets held by the connector.

pub fn setup_refund_on_error_input(amount: U256, refund_address: Address) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let mint_args = ethabi::encode(&[
        ethabi::Token::Address(refund_address.raw().0.into()),
        ethabi::Token::Uint(amount.to_big_endian().into()),
    ]);

    [selector, mint_args.as_slice()].concat()
}

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
