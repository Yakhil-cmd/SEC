### Title
Truncating Division in `native_per_pubdata` Computation Yields Zero, Allowing Free Pubdata Generation - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When `pubdata_price < native_price`, the truncating integer division used to compute `native_per_pubdata` yields `0`. This causes the system to charge **zero native resources** for all pubdata generated by a transaction, allowing an attacker to generate large amounts of L1 state-diff data without paying the corresponding native resource cost. The operator must still publish this data to L1 but is not compensated through the native resource mechanism.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (the L2 transaction validation path), `native_per_pubdata` is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` performs **truncating (floor) integer division**. When `pubdata_price < native_price`, the result is `0`.

By contrast, `native_per_gas` — computed just above — uses **ceiling division** (`div_ceil`):

```rust
// line 135
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

The same inconsistency exists in the public API helper `validate_l2_tx_intrinsic_native_resources`:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata = 0`, the downstream charging functions produce zero cost for any amount of pubdata:

```rust
// gas_helpers.rs, line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0
``` [4](#0-3) 

```rust
// gas_helpers.rs, line 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
// → current_pubdata_spent * 0 = 0
``` [5](#0-4) 

The `check_enough_resources_for_pubdata` check always passes (0 ≤ any remaining native), so the transaction is never reverted for pubdata costs. [6](#0-5) 

---

### Impact Explanation

When `pubdata_price < native_price` (a legitimate operational condition — these are in different units and their relative magnitude is not constrained), any L2 transaction sender can:

1. Generate large amounts of pubdata (e.g., via many `SSTORE` operations) up to the block's `pubdata_limit`.
2. Pay **zero native resources** for that pubdata, because `native_per_pubdata = 0`.
3. The `deltaGas` adjustment in `compute_gas_refund` also becomes zero for pubdata (`native_used / native_per_gas` is unaffected by pubdata native cost), so the gas refund is not reduced to compensate.

The operator must still publish the state diffs to L1 and bears the L1 data availability cost, but is not compensated through the native resource / gas mechanism. This is a direct financial loss to the operator and a resource accounting bypass for the attacker. [7](#0-6) 

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is not an edge case. `pubdata_price` is the cost per byte of L1 calldata/blob data (typically a small number of wei), while `native_price` is the cost per RISC-V proving cycle (a different unit entirely). In normal operation — for example, `native_price = 10` and `pubdata_price = 5` — the truncating division yields `0` instead of the correct ceiling value of `1`. Any unprivileged transaction sender who observes the block context can exploit this whenever the condition holds. [8](#0-7) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` computation, consistent with how `native_per_gas` is computed:

```rust
// Before (truncating — yields 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling — yields at least 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [1](#0-0) [3](#0-2) 

---

### Proof of Concept

1. Operator sets block context with `native_price = 100`, `pubdata_price = 50`, `eip1559_basefee = 100`.
2. Attacker submits an L2 EIP-1559 transaction with `max_fee_per_gas = 100`, `gas_limit = 1_000_000`.
3. In `validate_and_compute_fee_for_transaction`:
   - `gas_price = 100`
   - `native_per_gas = ceil(100 / 100) = 1`
   - `native_per_pubdata = floor(50 / 100) = 0` ← truncation yields zero
4. Transaction executes, writing to 100 storage slots (generating ~3200 bytes of pubdata).
5. `get_resources_to_charge_for_pubdata` returns `3200 * 0 = 0` native resources.
6. `check_enough_resources_for_pubdata` passes trivially.
7. Transaction succeeds; attacker paid only EVM gas for `SSTORE` operations, zero native resources for pubdata.
8. Operator publishes 3200 bytes to L1 at their own expense. [9](#0-8)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
