### Title
SNS Governance Silently Marks Failed Generic Proposal Execution as Executed - (File: rs/sns/governance/src/canister_control.rs)

### Summary
`perform_execute_generic_nervous_system_function_call` in the SNS governance canister unconditionally returns `Ok(())` for any successful inter-canister call reply, regardless of whether the target canister's response encodes an application-level error. This causes an adopted `ExecuteGenericNervousSystemFunction` proposal to be permanently marked `Executed` (success) even when the target canister explicitly signals failure, producing a state indistinguishable from a genuinely successful execution.

### Finding Description
The SNS governance proto documents the required target method signature as:

```
<method_name>(proposal_data: ProposalData) -> Result<(), String>
``` [1](#0-0) 

When `perform_execute_generic_nervous_system_function_call` receives a reply from the target canister, it discards the reply bytes entirely and returns `Ok(())` unconditionally, with an explicit TODO acknowledging that application-level errors are not detected:

```rust
Ok(_reply) => {
    // TODO: Do something with reply. E.g. store it in the proposal,
    // and/or deserialize it so that we can detect whether there was an
    // application-level error, as opposed to a communication error.
    // For now, any reply is considered a success.
    Ok(())
}
``` [2](#0-1) 

This `Ok(())` propagates back through `perform_execute_generic_nervous_system_function` → `perform_action` → `set_proposal_execution_status`, which then sets `executed_timestamp_seconds` to the current time and leaves `failed_timestamp_seconds` and `failure_reason` at their zero/None defaults: [3](#0-2) 

The result is a `ProposalDecisionStatus::Executed` proposal whose on-chain state is identical to a genuinely successful execution, even though the target canister returned `Err("…")`.

### Impact Explanation
Any `ExecuteGenericNervousSystemFunction` proposal whose target canister returns an application-level `Err` is permanently and irrevocably recorded as `Executed`. Because `set_proposal_execution_status` asserts `ProposalDecisionStatus::Adopted` before writing and never revisits a proposal once `executed_timestamp_seconds != 0`, the proposal cannot be retried or corrected: [4](#0-3) 

Downstream observers (dashboards, integrations, the SNS itself) see `executed_timestamp_seconds > 0` and `failure_reason = None`, which is the canonical success signal used throughout the codebase: [5](#0-4) 

The intended state change never occurred, but governance permanently believes it did. For proposals that gate critical SNS operations (treasury transfers, dapp configuration, upgrade coordination), this silent false-success permanently blocks the corrective re-proposal path.

### Likelihood Explanation
Every registered `GenericNervousSystemFunction` whose target method returns `Result<(), String>` is affected whenever the target canister returns `Err`. This is a normal, expected code path for any target canister that validates its inputs or encounters transient state. No special attacker capability is required beyond submitting a proposal that passes governance voting — a capability available to any neuron holder. The TODO comment confirms this is a known, unresolved gap, not an edge case.

### Recommendation
Decode the reply bytes as `Result<(), String>` (matching the documented method signature) and propagate an application-level `Err` as a `GovernanceError::External` failure, exactly as the TODO comment suggests:

```rust
Ok(reply) => {
    match Decode!(&reply, Result<(), String>) {
        Ok(Ok(())) => Ok(()),
        Ok(Err(app_err)) => Err(GovernanceError::new_with_message(
            ErrorType::External,
            format!("Target canister returned application error: {app_err}"),
        )),
        Err(decode_err) => Err(GovernanceError::new_with_message(
            ErrorType::External,
            format!("Failed to decode reply: {decode_err}"),
        )),
    }
}
``` [6](#0-5) 

### Proof of Concept
1. Register a `GenericNervousSystemFunction` (ID ≥ 1000) whose `target_method` returns `Encode!(&Err::<(), String>("intentional failure".to_string()))`.
2. Submit and pass an `ExecuteGenericNervousSystemFunction` proposal referencing that function.
3. After execution, query `get_proposal`: observe `executed_timestamp_seconds > 0`, `failed_timestamp_seconds == 0`, `failure_reason == None`.
4. The proposal is permanently `Executed`; the target canister's `Err` was silently discarded.

The `perform_action` dispatch confirms this path is live for all generic proposals: [7](#0-6)

/// Executes a generic nervous system function (i.e., a non-native SNS proposal).
pub async fn perform_execute_generic_nervous_system_function_call(
    env: &dyn Environment,
    function: NervousSystemFunction,
    call: ExecuteGenericNervousSystemFunction,
) -> Result<(), GovernanceError> {
    // Get the canister id and the method against which we execute the proposal.
    let valid_function = ValidGenericNervousSystemFunction::try_from(&function)
        .map_err(|e| GovernanceError::new_with_message(ErrorType::InvalidProposal, e))?;

    let result = env
        .call_canister(
            valid_function.target_canister_id,
            &valid_function.target_method,
            call.payload,
        )
        .await;

    // Convert result.
    match result {
        Err(err) => Err(GovernanceError::new_with_message(
            ErrorType::External,
            format!("Canister method call to execute proposal failed: {err:?}"),
        )),

        Ok(_reply) => {
            // TODO: Do something with reply. E.g. store it in the proposal,
            // and/or deserialize it so that we can detect whether there was an
            // application-level error, as opposed to a communication
            // error. Detecting application error could be done as follows:
            //
            //   candid::!Decode(&reply, Result<String, String>)
            //
            // This could then be converted into a Result<(), GovernanceError>.
            // For now, any reply is considered a success.
            Ok(())
        }
    }
}
