Audit Report

## Title
Unbounded Slot Table Growth via `SLOT_TABLE_NO_LIMIT` Enables Single Byzantine Peer to OOM Subnet Nodes — (`rs/p2p/consensus_manager/src/receiver.rs`, `rs/replica/setup_ic_network/src/lib.rs`)

## Summary
`SLOT_TABLE_NO_LIMIT = usize::MAX` is assigned as the per-peer slot table limit for all artifact types except ingress (consensus, certifier, DKG, iDKG, HTTPS-outcalls). The only guard against unbounded slot table growth in `handle_slot_update_receive` is `peer_slot_table_len < self.slot_limit`, which is trivially always true when `slot_limit = usize::MAX`. A single authenticated subnet peer can therefore send an unbounded stream of slot updates with distinct slot numbers and artifact IDs, causing unbounded growth of `slot_table`, `active_assembles`, and `artifact_processor_tasks` on every receiving node, ultimately exhausting heap memory and crashing the replica process.

## Finding Description
**Root cause — `SLOT_TABLE_NO_LIMIT` assignment:**
In `rs/replica/setup_ic_network/src/lib.rs` lines 72–75, `SLOT_TABLE_NO_LIMIT = usize::MAX` is defined alongside `SLOT_TABLE_LIMIT_INGRESS = 50_000`. The comment on line 72 explicitly acknowledges the ingress limit exists to protect against a malicious peer. Lines 237, 246, 268, 279, 291, and 303 pass `SLOT_TABLE_NO_LIMIT` to `abortable_broadcast_channel` for all non-ingress artifact types; only line 257 passes `SLOT_TABLE_LIMIT_INGRESS` for ingress.

**Broken guard in `handle_slot_update_receive`:**
In `rs/p2p/consensus_manager/src/receiver.rs` lines 372–390, the only protection against unbounded slot table growth is:
```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { ... }
Entry::Vacant(_) => { /* drop and warn */ }
```
With `self.slot_limit = usize::MAX`, the second arm (`Entry::Vacant(_)` drop branch) is unreachable for all non-ingress types. Every distinct `slot_number` sent by a malicious peer is unconditionally inserted.

**Unbounded task spawning:**
Lines 399–419 show that every new unique artifact ID seen in a slot update spawns a new `process_slot_update` Tokio task and inserts into `active_assembles`. Tasks are only removed from `active_assembles` at lines 317–318 when `peer_rx.borrow().is_empty()`, which only occurs when the advertising peer removes or overwrites its slot entries. A malicious peer that never does so keeps tasks alive indefinitely.

**Channel depth is not a mitigation:**
The inbound channel at line 43 has depth 100, which only introduces transient back-pressure; the event loop drains it continuously and there is no cap on total entries processed.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

On every honest subnet node receiving updates from the malicious peer: `slot_table[peer_id]` grows without bound (one entry per distinct slot number), `active_assembles` grows without bound (one entry per distinct artifact ID), and `artifact_processor_tasks` (a `JoinSet`) grows without bound (one Tokio task per distinct artifact ID, each holding a `watch::Receiver<PeerCounter>`, a cloned `Assembler`, a cloned `Sender`, and a `CancellationToken`). At scale this exhausts heap memory, triggering an OOM kill of the replica process. Targeting all honest nodes halts the subnet.

## Likelihood Explanation
The attacker must control one authenticated subnet node (QUIC/TLS uses node certificates; joining requires an NNS governance vote). However, only a single compromised or malicious node below the consensus fault threshold is required — no majority is needed. Once on the subnet, the exploit requires only sending crafted P2P slot-update messages with distinct slot numbers and artifact IDs, which is trivially scriptable. The attack is repeatable and persistent until the malicious node is removed from the subnet topology.

## Recommendation
Apply a finite per-peer slot limit to all artifact types, not only ingress. The bound for each type should reflect the maximum number of distinct artifacts a single honest peer would legitimately advertise per epoch (e.g., bounded by the number of consensus rounds, DKG transcripts, or HTTPS-outcall responses in flight at any given time). Alternatively, enforce a global cap on `active_assembles` and `artifact_processor_tasks` regardless of artifact type, and ensure the cap is enforced before spawning new tasks.

## Proof of Concept
1. Attacker controls one subnet node `N_evil` with a valid node TLS certificate.
2. `N_evil` opens a QUIC connection to honest node `N_victim`.
3. `N_evil` sends `POST /{consensus_uri}/update` with a valid protobuf `SlotUpdate`: `slot_id=1`, `commit_id=1`, `update=Id(<arbitrary_bytes>)`.
4. `N_victim`'s `update_handler` decodes and forwards to the event loop; `handle_slot_update_receive` evaluates `peer_slot_table_len=0 < usize::MAX` → inserts entry, spawns `process_slot_update` task for artifact ID X.
5. `N_evil` repeats step 3 with `slot_id=2,3,…,N` and distinct artifact IDs.
6. After ~10^6 iterations: `slot_table[N_evil].len() = 10^6`, `active_assembles.len() = 10^6`, `artifact_processor_tasks.len() = 10^6`.
7. `N_victim`'s heap is exhausted → OOM → replica process killed → node drops out of consensus.
8. Repeating against all honest nodes halts the subnet.

A deterministic integration test can be written using `ConsensusManagerReceiver` directly: construct a receiver with `slot_limit = usize::MAX`, feed it N slot updates with distinct slot numbers and artifact IDs from a single peer, and assert that `active_assembles.len() == N` and `artifact_processor_tasks.len() == N` after processing, confirming unbounded growth.
