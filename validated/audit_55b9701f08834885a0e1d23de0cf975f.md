### Title
ERC-20 Mirror Over-Issuance via Fee-on-Transfer NEP-141 Tokens Leads to Bridge Insolvency - (File: `engine/src/engine.rs`)

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` mints ERC-20 tokens based on the `amount` field supplied in the `ft_on_transfer` callback arguments, without verifying the actual NEP-141 balance change in Aurora's account. For fee-on-transfer (deflationary) NEP-141 tokens, the actual tokens received by Aurora are less than `args.amount`, causing Aurora to over-issue ERC-20 tokens. The resulting ERC-20 total supply exceeds Aurora's real NEP-141 reserve, making the bridge insolvent: later withdrawers cannot redeem their ERC-20 tokens for NEP-141.

---

### Finding Description

The NEP-141 standard's `ft_transfer_call` flow works as follows:

1. Caller invokes `ft_transfer_call(receiver_id: aurora, amount: X, msg: <evm_address>)` on a NEP-141 token contract.
2. The NEP-141 contract transfers tokens to Aurora and then calls `ft_on_transfer` on Aurora, passing `amount: X` in the JSON args.
3. Aurora's `ft_on_transfer` entry point (`engine/src/contract_methods/connector.rs`, line 62–109) dispatches to `engine.receive_erc20_tokens`.

Inside `receive_erc20_tokens` (`engine/src/engine.rs`, line 796–844):

```rust
let amount = args.amount.as_u128();   // line 803 — taken directly from callback args
// ...
setup_receive_erc20_tokens_input(&recipient, amount)  // line 831 — mints `amount` ERC-20
```

The function unconditionally trusts `args.amount` and mints exactly that many ERC-20 tokens. It never reads Aurora's actual NEP-141 balance before and after the transfer to compute the real received amount.

For a fee-on-transfer NEP-141 token (one that deducts a percentage fee during `ft_transfer`), the token contract transfers `X - fee` tokens to Aurora but still calls `ft_on_transfer` with `amount = X`. Aurora mints `X` ERC-20 tokens while only holding `X - fee` NEP-141 tokens. This discrepancy accumulates with every deposit.

On the withdrawal side, `EvmErc20V2.withdrawToNear` (`etc/eth-contracts/contracts/EvmErc20V2.sol`, line 53–64) burns `amount` ERC-20 tokens and calls the `ExitToNear` precompile with that same `amount`. The precompile (`engine-precompiles/src/native.rs`, line 627–646) then issues an `ft_transfer` call on the NEP-141 contract for `amount`. Because Aurora's NEP-141 balance is less than the total ERC-20 supply, the last users to withdraw will find Aurora cannot fulfill the `ft_transfer`, permanently freezing their ERC-20 tokens.

---

### Impact Explanation

**Critical — Insolvency / Permanent fund freeze.**

Every deposit of a fee-on-transfer NEP-141 token over-issues ERC-20 tokens relative to the NEP-141 reserve held by Aurora. The shortfall is permanent and grows with each deposit. When the aggregate shortfall exceeds Aurora's NEP-141 balance for that token, subsequent `withdrawToNear` calls will fail (the NEP-141 `ft_transfer` will revert because Aurora lacks sufficient balance). Without the `error_refund` feature, the burned ERC-20 tokens are not re-minted, permanently destroying the user's funds.

---

### Likelihood Explanation

**Medium.** Fee-on-transfer tokens are a well-known and widely deployed pattern in the NEP-141/ERC-20 ecosystem. Aurora accepts any NEP-141 token without a whitelist. Any unprivileged user can call `ft_transfer_call` on a fee-on-transfer NEP-141 token pointing to Aurora, triggering the over-issuance. No admin access or special privilege is required. The only prerequisite is that a fee-on-transfer NEP-141 token has been registered with Aurora via `deploy_erc20_token`.

---

### Recommendation

In `receive_erc20_tokens`, replace the use of `args.amount` with the actual balance delta. Before the transfer completes, read Aurora's NEP-141 balance; after `ft_on_transfer` is called, read it again and mint only the difference. Concretely:

```rust
// Before minting, query actual NEP-141 balance of current_account_id on `token`
// and compute: actual_received = post_balance - pre_balance
// Then use actual_received instead of args.amount.as_u128()
```

Alternatively, document that fee-on-transfer NEP-141 tokens are unsupported and add a token registry/whitelist that excludes them, analogous to the ZNSTreasury recommendation.

---

### Proof of Concept

**Step 1 — Deploy a fee-on-transfer NEP-141 token** that deducts 10% on every `ft_transfer`. Call `deploy_erc20_token` on Aurora to register it.

**Step 2 — Deposit:** Call `ft_transfer_call(receiver_id: aurora, amount: 1000, msg: <evm_address>)` on the NEP-141 contract.
- NEP-141 transfers 900 tokens to Aurora (10% fee deducted).
- NEP-141 calls `ft_on_transfer` on Aurora with `amount: 1000`.
- `receive_erc20_tokens` reads `args.amount = 1000` at [1](#0-0)  and mints 1000 ERC-20 tokens via [2](#0-1) .
- Aurora holds 900 NEP-141 but has issued 1000 ERC-20. Shortfall: 100.

**Step 3 — Withdraw full amount:** User calls `withdrawToNear(recipient, 1000)` on the ERC-20 contract.
- `EvmErc20V2.withdrawToNear` burns 1000 ERC-20 at [3](#0-2) .
- `ExitToNear` precompile issues `ft_transfer(receiver_id: recipient, amount: 1000)` on the NEP-141 contract at [4](#0-3) .
- Aurora only holds 900 NEP-141 tokens → `ft_transfer` fails.
- Without `error_refund`, the 1000 burned ERC-20 tokens are not re-minted → **permanent fund loss**.

The `ft_on_transfer` entry point that dispatches to the vulnerable function is at [5](#0-4) , and the over-issuance root cause is the unchecked trust of `args.amount` at [1](#0-0) .

```rust
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
```
