### Title
Permanent Fund Freeze in `ExitToNear` Multi-Step Promise Flow When `ft_transfer` Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ExitToNear` precompile executes a two-step cross-contract flow: (1) ERC-20 tokens are burned in the EVM (state committed), then (2) a NEAR promise calls `ft_transfer` on the NEP-141 contract. When the `error_refund` feature is disabled, the `refund` field in the callback args is hardcoded to `None`. If the `ft_transfer` promise fails, `exit_to_near_precompile_callback` takes no corrective action, leaving the EVM state permanently inconsistent: ERC-20 tokens are burned but the corresponding NEP-141 tokens are never transferred, resulting in permanent fund loss for the user.

---

### Finding Description

**Step 1 — ERC-20 burn (EVM state committed):**

In `ExitToNear::run()`, the precompile constructs callback args with a conditional `refund` field:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `refund` is always `None`. A callback is still attached when `transfer_near` is `Some` (wNEAR unwrap path) or when `callback_args != default()`.

**Step 2 — NEAR promise: `ft_transfer` or `near_withdraw`:**

The promise is created and dispatched. The EVM state (ERC-20 burn) is already committed at this point. [2](#0-1) 

**Step 3 — Callback: `exit_to_near_precompile_callback`:**

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {
    // refund path — only reached if args.refund is Some
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // ← silent no-op when error_refund is disabled
};
``` [3](#0-2) 

When `ft_transfer` fails and `args.refund` is `None`, the function returns `Ok(None)` with no refund action. The burned ERC-20 tokens are permanently lost.

**The test suite explicitly documents this behavior:**

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The same pattern applies to the `ExitToEthereum` precompile, which creates a `PromiseArgs::Create` with **no callback at all**, meaning any failure of the `withdraw` call on the ETH connector has no recovery path: [5](#0-4) 

---

### Impact Explanation

**Permanent freezing of funds (Critical).** When `ft_transfer` fails in the `ExitToNear` flow without `error_refund`:
- ERC-20 tokens are burned from the user's EVM balance (irreversible EVM state change).
- The corresponding NEP-141 tokens remain in Aurora's account, not transferred to the recipient.
- No refund is issued to the user.
- The user's ERC-20 tokens are permanently destroyed with no corresponding value received.

---

### Likelihood Explanation

**Moderate.** The failure condition is reachable by any EVM user:
- A user calls an ERC-20 contract that invokes the `ExitToNear` precompile with a recipient NEAR account that is not registered with the NEP-141 contract.
- The NEP-141 `ft_transfer` call fails (standard NEP-141 behavior for unregistered recipients).
- Without `error_refund`, no refund is issued.

This does not require any admin action, privileged access, or unusual network conditions. Any user who mistypes a recipient account ID or sends to an unregistered account triggers this path.

---

### Recommendation

1. Make the `error_refund` feature unconditionally enabled in production builds, or remove the feature flag and always populate `refund` in `ExitToNearPrecompileCallbackArgs`.
2. For `ExitToEthereum`, attach a callback to the `withdraw` promise that can re-mint burned tokens if the withdrawal fails.
3. Audit all multi-step promise flows to ensure that any EVM state committed in step N is reverted or compensated if step N+1 fails.

---

### Proof of Concept

1. User holds 100 units of ERC-20 token `T` on Aurora (backed by NEP-141 `t.near`).
2. User calls `T.exitToNear("unregistered.near", 100)` from their EVM contract.
3. The `ExitToNear` precompile fires: 100 units of `T` are burned in EVM storage (committed).
4. A NEAR promise calls `t.near::ft_transfer({"receiver_id": "unregistered.near", "amount": "100"})`.
5. `ft_transfer` panics: `"The account unregistered.near is not registered"`.
6. `exit_to_near_precompile_callback` is invoked with `PromiseResult::Failed`.
7. `args.refund` is `None` (no `error_refund` feature) → falls through to `else { None }`.
8. No re-mint, no refund. User's 100 ERC-20 tokens are permanently destroyed.
9. Aurora's NEP-141 balance of `t.near` is unchanged (transfer never happened).
10. State is permanently inconsistent: EVM shows 0 balance, NEP-141 shows Aurora still holds the tokens. [1](#0-0) [6](#0-5)
