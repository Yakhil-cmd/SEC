Audit Report

## Title
SNS Governance `approve_treasury_manager` Sets Allowances Without `expected_allowance`, Enabling Residual-Allowance Double-Spend by Malicious Treasury Manager - (File: rs/sns/governance/src/extensions.rs)

## Summary

`Governance::approve_treasury_manager` issues `icrc2_approve` with `expected_allowance: None` for both SNS and ICP token allowances. An inline comment incorrectly asserts this eliminates double-spending risk. In reality, a malicious treasury manager canister can drain a residual allowance from a prior deposit via `icrc2_transfer_from` before the governance's new `icrc2_approve` is processed by the ledger, then also consume the freshly set allowance — extracting ε + Y tokens when the governance intended only Y.

## Finding Description

In `rs/sns/governance/src/extensions.rs`, `Governance::approve_treasury_manager` (lines 777–831) issues two `icrc2_approve` calls — one to the SNS ledger (lines 794–810) and one to the ICP ledger (lines 812–828) — both with `expected_allowance: None`:

```rust
// If expected_allowance is None, the ledger *blindly* overwrites any existing
// allowance (even if non-zero). Therefore, there is no risk of double spending.
self.ledger.icrc2_approve(to, sns_amount_e8s, ..., None).await
``` [1](#0-0) 

The comment's reasoning is wrong. "Blindly overwriting" the allowance at the ledger level does not prevent the treasury manager from spending the *old* allowance in the window between when the governance canister sends the `icrc2_approve` message and when the ledger canister processes it. On the Internet Computer, messages from different canisters to the same canister are not ordered relative to each other; the treasury manager's `icrc2_transfer_from` message can arrive at the ledger before the governance's `icrc2_approve` message.

`approve_treasury_manager` is called from two sites:

1. **`ValidatedRegisterExtension::execute`** (lines 545–551) — safe, because `ensure_no_code_is_installed` (lines 523–525) guarantees the extension canister has no Wasm installed yet, so it cannot call `icrc2_transfer_from`. [2](#0-1) 

2. **`execute_treasury_manager_deposit`** (lines 1566–1573) — **vulnerable**, because the treasury manager already has code installed and may hold a residual allowance from a prior deposit round. [3](#0-2) 

The ICRC-2 `AllowanceTable::approve` in `rs/ledger_suite/common/ledger_core/src/approvals.rs` (lines 253–291) only enforces the `expected_allowance` guard when the caller supplies a non-`None` value. When `None` is passed, the allowance is unconditionally overwritten regardless of the current on-chain value. [4](#0-3) 

**Exploit flow:**

1. First deposit proposal executes: `approve_treasury_manager(tm, X_sns, X_icp)` → SNS allowance = X, ICP allowance = X.
2. Treasury manager calls `deposit` but intentionally transfers only X − ε tokens, leaving allowance = ε > 0.
3. Second deposit proposal for Y tokens is submitted and passes (proposals are public; the treasury manager observes it during the multi-day voting period).
4. Treasury manager submits `icrc2_transfer_from(treasury_sns_account → tm_account, ε)` to the SNS ledger. Because this message is from a different canister than governance, it can arrive at the ledger before governance's `icrc2_approve`. SNS allowance → 0.
5. Governance's `icrc2_approve(tm, Y, expected_allowance=None)` is processed next: SNS allowance → Y.
6. Treasury manager calls `deposit`, consuming Y tokens.
7. **Net result**: treasury manager extracted ε + Y SNS tokens; governance intended only Y.

The same sequence applies to the ICP allowance via `self.nns_ledger.icrc2_approve(…, None)`. [5](#0-4) 

## Impact Explanation

A malicious treasury manager canister can extract **ε + Y** tokens from the SNS treasury when the governance authorized only **Y** for the current deposit. Each successive deposit proposal can be exploited to drain an additional ε tokens beyond the approved amount. This constitutes unauthorized access to SNS-controlled canister funds, directly undermining the governance's ability to bound treasury outflows. This matches the **High** impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."

## Likelihood Explanation

**Medium.** The attack requires a malicious treasury manager canister — a canister installed via SNS governance proposal that either was always malicious or turned malicious post-installation. SNS governance proposals are public and have a multi-day voting period, giving the treasury manager ample time to observe an incoming deposit proposal and pre-stage an `icrc2_transfer_from` call. On the IC, inter-canister message ordering between the treasury manager and the governance canister to the ledger is not guaranteed; the treasury manager's message can arrive at the ledger before the governance's `icrc2_approve` message. The treasury manager only needs to retain a non-zero residual allowance from a prior deposit to mount the attack. The attack is repeatable across every subsequent deposit proposal.

## Recommendation

Pass the current on-chain allowance as `expected_allowance` in both `icrc2_approve` calls inside `approve_treasury_manager`. Before calling `approve_treasury_manager`, query `icrc2_allowance` for the treasury manager and supply the result as `expected_allowance`. If the ledger returns `AllowanceChanged`, the governance should first zero out the allowance (with the correct `expected_allowance`) before setting the new value. This is the standard two-step mitigation for the approve-frontrunning pattern, and the ICRC-2 `AllowanceChanged` error variant exists precisely for this purpose. [6](#0-5) 

## Proof of Concept

1. Deploy an SNS with a treasury manager extension canister.
2. Execute a first deposit proposal: `approve_treasury_manager(tm, X_sns, X_icp)` → SNS allowance = X, ICP allowance = X.
3. Treasury manager calls `deposit` but intentionally transfers only X − ε tokens, leaving allowance = ε > 0.
4. Submit and pass a second deposit proposal for Y tokens.
5. Treasury manager submits `icrc2_transfer_from(treasury_sns_account → tm_account, ε)` to the SNS ledger before the governance proposal execution begins (or races it using IC message scheduling). SNS allowance → 0.
6. Governance's `icrc2_approve(tm, Y, expected_allowance=None)` is processed: SNS allowance → Y.
7. Treasury manager calls `deposit`, consuming Y tokens.
8. Verify: treasury manager extracted ε + Y SNS tokens; governance intended only Y.

A deterministic PocketIC integration test can reproduce this by: (a) setting up the SNS with a mock malicious treasury manager, (b) executing two deposit proposals with the treasury manager retaining ε after the first, (c) submitting `icrc2_transfer_from(ε)` from the treasury manager before the second proposal's `approve_treasury_manager` call, and (d) asserting the total extracted exceeds Y. [7](#0-6)
