### Title
Missing Owner Authorization on `deploy_upgrade` Allows Any Caller to Force Contract Deployment - (File: `engine/src/lib.rs`)

### Summary

The `deploy_upgrade` entrypoint in `engine/src/lib.rs` is missing an owner-only access check. Any NEAR account can call it after the upgrade delay has elapsed, forcing deployment of whatever code the owner previously staged — even if the owner has since decided to abort the upgrade.

### Finding Description

The `deploy_upgrade` function performs only two checks before executing `Runtime::self_deploy`:

1. The engine is not paused (`require_running`).
2. The current block height exceeds the staged upgrade index.

It does **not** verify that `predecessor_account_id == state.owner_id`. [1](#0-0) 

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

Every other sensitive administrative function in the same file and in `engine/src/contract_methods/admin.rs` calls `require_owner_only` or `require_owner_and_running` before mutating state: [2](#0-1) [3](#0-2) 

`stage_upgrade` (which writes the code to be deployed) is owner-gated, but `deploy_upgrade` (which actually executes the self-deployment) is not. The asymmetry means the owner controls *what* gets staged but not *whether* it ultimately gets deployed.

### Impact Explanation

`Runtime::self_deploy` replaces the live contract bytecode with the staged code. If the owner stages an upgrade, later discovers a critical bug in it, and attempts to cancel (e.g., by staging a corrected version), an attacker can race to call `deploy_upgrade` before the cancellation lands, forcing the buggy bytecode onto the live contract. A buggy contract version could enable direct theft of user funds (ETH balances, ERC-20 mirror tokens) or permanent freezing of funds held in the engine.

**Impact: Critical** — forced deployment of a vulnerable contract version can lead to direct theft of user funds or permanent fund freeze.

### Likelihood Explanation

The precondition is that the owner has staged an upgrade (a routine operational action). Once staged and the delay elapsed, the window is open to any NEAR account. An attacker monitoring the chain for `stage_upgrade` transactions can wait for the delay and call `deploy_upgrade` at any time, including racing a cancellation transaction. No special privileges, leaked keys, or social engineering are required.

### Recommendation

Add `require_owner_only` (or `require_owner_and_running`) immediately after loading state, mirroring every other privileged entrypoint:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)
        .map_err(ContractError::msg)
        .sdk_unwrap();
    // ADD THIS:
    require_owner_only(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

### Proof of Concept

1. Owner calls `stage_upgrade` with new contract bytecode (owner-gated, succeeds).
2. `set_upgrade_delay_blocks` sets a delay of N blocks.
3. After N blocks, **any** NEAR account calls `deploy_upgrade` — no owner check is performed.
4. `Runtime::self_deploy` executes, replacing the live contract with the staged bytecode.
5. If the owner had discovered a flaw in the staged code and had not yet cancelled, the flawed code is now live and can be exploited to steal or freeze user funds.

The `require_owner_only` helper used by all other privileged functions is defined at: [2](#0-1) 

and is simply absent from `deploy_upgrade`: [1](#0-0)

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
