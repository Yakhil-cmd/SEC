Audit Report

## Title
Silent ICP Burn Failure Allows Cycles Minting Without ICP Destruction in CMC - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister intentionally swallows all burn errors to prevent notification retries. When the ICP ledger call fails after cycles have already been deposited, the CMC returns success to the caller, permanently closes the notification, and leaves the ICP unburned in its subaccount — minting cycles against ICP that was never destroyed and violating the ICP↔cycles conservation invariant.

## Finding Description
The code is confirmed to behave exactly as claimed. Three functions follow the same two-step pattern:

**`process_top_up`** (L1999–2002): calls `deposit_cycles`, and on success unconditionally calls `burn_and_log` then returns `Ok(cycles)` regardless of burn outcome.

**`process_create_canister`** (L1943–1946): calls `do_create_canister`, and on success calls `burn_and_log` then returns `Ok(canister_id)`.

**`process_mint_cycles`** (L1966–1974): calls `do_mint_cycles`, and on success calls `burn_and_log` then returns `Ok(NotifyMintCyclesSuccess{...})`.

`burn_and_log` (L2014–2049) is explicitly designed to never propagate errors — the comment at L2015–2016 states: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* On a failed `call_protobuf` to `send_pb`, it logs the error and returns normally.

After each `process_*` function returns `Ok(...)`, the caller (`notify_top_up`, `notify_mint_cycles`, `notify_create_canister`) inserts the block as permanently processed (e.g., `NotificationStatus::NotifiedTopUp(Ok(cycles))` at L1214–1218). The `is_transient_error` guard at L1219–1221 only removes the entry for transient error results — an `Ok(...)` result is never transient, so the notification is permanently closed. No retry path exists.

The root cause is the absence of any state distinguishing "cycles deposited, burn succeeded" from "cycles deposited, burn failed." Once cycles are deposited, the system has no mechanism to retry the burn or record a pending burn obligation.

## Impact Explanation
This constitutes illegal minting: cycles are created and delivered to the recipient while the corresponding ICP is never destroyed. The ICP remains locked in the CMC's subaccount keyed by `Subaccount::from(&canister_id)` or `Subaccount::from(&caller())` — inaccessible to the user (notification closed) and not automatically retried. Each occurrence inflates the effective ICP supply relative to cycles minted, breaking the protocol's economic invariant. This is a concrete, significant NNS/ledger security impact with direct protocol harm, fitting the High impact category: *"Significant... ledger... NNS... infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
The ICP ledger and CMC both reside on the NNS subnet, making co-located outages less frequent but not impossible — ledger canister upgrades, subnet stalls, or message queue exhaustion can all cause transient `send_pb` failures. Any unprivileged user calling `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` during such a window triggers this path. The CMC is one of the most frequently called NNS canisters, making at least occasional occurrences realistic over its operational lifetime. No special privileges, front-running, or victim interaction is required.

## Recommendation
**Short term:** Change `burn_and_log` to return a `Result`. On burn failure after successful cycle deposit, do not mark the notification as permanently successful — instead record a `BurnPending` state and allow the burn to be retried independently (e.g., via a heartbeat or a separate retry endpoint), consistent with how `refund_icp` failures propagate as `NotifyError::Other` and permit retry.

**Long term:** Separate the "cycles deposited" and "ICP burned" states in `blocks_notified` so that a failed burn can be retried without risking double-minting. Add an invariant check (e.g., in canister heartbeat or post-upgrade hooks) that verifies no subaccounts hold unexpected ICP balances.

## Proof of Concept
1. Alice sends 10 ICP to the CMC subaccount for canister `C` (memo `MEMO_TOP_UP_CANISTER`) at ledger block `B`.
2. Alice calls `notify_top_up(block_index=B, canister_id=C)`.
3. CMC calls `deposit_cycles(C, cycles, ...)` — succeeds; canister `C` receives cycles.
4. CMC calls `burn_and_log(Subaccount::from(&C), 10 ICP)` — the `call_protobuf` to `send_pb` returns an error (e.g., ledger is mid-upgrade).
5. `burn_and_log` logs the error and returns `()`.
6. `process_top_up` returns `Ok(cycles)`.
7. `notify_top_up` inserts `blocks_notified[B] = NotifiedTopUp(Ok(cycles))`; `is_transient_error` is false for `Ok`; entry is permanent.
8. Canister `C` holds the cycles. 10 ICP remains in `CMC[Subaccount::from(&C)]`. Block `B` is permanently closed.

**Reproducible test plan:** Write a PocketIC integration test that: (a) installs a mock ICP ledger that returns an error on the first `send_pb` call after a `notify_top_up`; (b) calls `notify_top_up`; (c) asserts the return is `Ok(cycles)`; (d) asserts `blocks_notified` contains `NotifiedTopUp(Ok(cycles))`; (e) asserts the CMC subaccount balance for the canister is still 10 ICP (burn did not occur); (f) asserts a second call to `notify_top_up` with the same block returns `NotifiedTopUp(Ok(cycles))` (no retry possible).
