### Title
ICRC-2 Allowance Does Not Snapshot Transfer Fee — Fee Change After Approval Causes Unexpected Balance Drain or Transaction Failure (`rs/ledger_suite/icrc1/src/lib.rs`)

---

### Summary

When a user calls `icrc2_approve` to grant a spender an allowance, the transfer fee at approval time is **not** stored in the allowance record. When `icrc2_transfer_from` is later executed, the **current** (potentially changed) ledger fee is used to deduct from both the `from` account's balance and the allowance. Because the SNS governance can change the transfer fee at any time via a `ManageLedgerParameters` proposal, a user who set an allowance under one fee regime may have their balance drained by more than expected, or have their intended transfer fail entirely, with no on-chain protection.

---

### Finding Description

The ICRC-1 ledger's `apply()` function for `Operation::Transfer` (which backs `icrc2_transfer_from`) computes the allowance consumed as `used_allowance = amount + effective_fee`, where `effective_fee` is the **current** ledger transfer fee at execution time: [1](#0-0) 

The `Allowance` struct stored in the approval table records only `amount`, `expires_at`, and `arrived_at` — no fee snapshot: [2](#0-1) 

The transfer fee is a mutable field on the ledger state and can be changed at any time via the `upgrade()` path: [3](#0-2) 

SNS governance exposes this as a first-class governance action via `ManageLedgerParameters`, which any SNS community can pass: [4](#0-3) 

The result is a temporal mismatch: the user commits to an allowance under fee `F1`, but the ledger enforces fee `F2` at execution time, with no mechanism for the user to express or enforce a fee expectation at approval time.

---

### Impact Explanation

**Scenario A — Fee increases after approval:**

1. User A calls `icrc2_approve(spender=B, amount=110_000)` when `transfer_fee = 10_000`. Allowance is set to `110_000`.
2. SNS governance passes `ManageLedgerParameters { transfer_fee: Some(20_000) }`.
3. Spender B calls `icrc2_transfer_from(from=A, to=C, amount=100_000)`.
4. Ledger computes `used_allowance = 100_000 + 20_000 = 120_000 > 110_000`. Transaction fails with `InsufficientAllowance`.
5. User A's funds are locked in an approval that can no longer execute the intended transfer without re-approval.

**Scenario B — Large allowance, fee increases:**

1. User A calls `icrc2_approve(spender=B, amount=u64::MAX)` when `transfer_fee = 10_000`.
2. SNS governance increases fee to `20_000`.
3. Spender B calls `icrc2_transfer_from(from=A, to=C, amount=100_000)`.
4. User A's balance is reduced by `120_000` (100,000 + 20,000) instead of the expected `110,000`. The user is charged 10,000 more than anticipated with no recourse.

**Scenario C — Fee decreases after approval:**

If the fee decreases, the allowance is consumed more slowly per transfer than the user intended, allowing the spender to execute more transfers than the user expected when setting the allowance amount.

The impact is a **ledger conservation / financial discrepancy** bug: the `from` account is debited an amount the user did not consent to at approval time, directly analogous to the SecondSwap M-07 finding where `sellFee` could be raised after listing creation. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** SNS governance fee changes via `ManageLedgerParameters` are a legitimate, expected governance action — not an attack. Any SNS that adjusts its ledger transfer fee (e.g., to reflect token price changes or operational costs) will silently affect all outstanding approvals. Users have no on-chain mechanism to express a fee expectation at approval time, and no notification when the fee changes. The entry path requires only:

1. An unprivileged user calling `icrc2_approve` (standard ICRC-2 operation).
2. A legitimate SNS governance proposal changing the fee (no malicious majority required — this is normal governance).
3. A spender calling `icrc2_transfer_from` (standard ICRC-2 operation). [6](#0-5) 

---

### Recommendation

Two mitigations are possible:

1. **Fee snapshot in approval**: Add a `fee_at_approval: Option<Tokens>` field to the `Allowance` struct. At `transfer_from` time, if `fee_at_approval` is set and the current fee exceeds it, reject the transfer with a new `FeeChanged` error, allowing the user to re-approve with updated parameters.

2. **`max_fee` parameter in `icrc2_approve`**: Add an optional `max_fee` field to `ApproveArgs`. The ledger stores this alongside the allowance and checks at `transfer_from` time that `current_fee <= max_fee`, rejecting the transfer if the fee has risen beyond what the user accepted.

Either approach gives the user explicit, on-chain protection against fee changes after approval — mirroring the SecondSwap recommendation of snapshotting `buyerFee`/`sellerFee` at listing creation time. [2](#0-1) [7](#0-6)

### Proof of Concept

```

# Allowance stored: { amount: 110_000, expires_at: None, arrived_at: T1 }

# Step 2: SNS governance passes ManageLedgerParameters { transfer_fee: 20_000 }
# Ledger state: transfer_fee = 20_000

# Step 3: Spender B calls transfer_from for 100_000 tokens
icrc2_transfer_from(from=A, to=C, amount=100_000)
# Ledger computes: used_allowance = 100_000 + 20_000 = 120_000
# Check: 110_000 (allowance) < 120_000 (used_allowance) → InsufficientAllowance

# Result: User A's intended transfer is permanently blocked unless they re-approve.
# No on-chain protection existed at approval time.
```
