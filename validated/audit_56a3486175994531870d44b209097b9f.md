### Title
Silent Discarding of EIP-7002/7251/6110 System Function Errors Produces Incorrect `requests_hash` in Block Headers - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/post_tx_op_sequencing.rs`)

---

### Summary

In `post_tx_op_sequencing.rs`, the results of three critical EIP system functions — `eip6110_events_parser`, `eip7002_system_part`, and `eip7251_system_part` — are silently discarded using `let _ = ...`. If any of these functions fail after partially modifying storage (e.g., advancing the withdrawal queue head pointer), the storage state is left inconsistent and the `requests_hash` fed into the block header is computed from an incomplete hasher state. This is a direct structural analog to the original report's `handle_payout`/`reclaim_stake` silent-failure pattern.

---

### Finding Description

In `post_tx_op_sequencing.rs` lines 90–92:

```rust
// Environment may have no such contracts predeployed for tests or sequencing purposes
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);
```

All three functions return `Result<bool, SystemError>`. Their results are unconditionally discarded. The comment justifies this for environments where the contracts are not predeployed, but the same discard applies when the contracts **are** deployed and fail mid-execution. [1](#0-0) 

`eip7002_system_part` and `eip7251_system_part` perform **storage writes** before they can fail. Specifically, `eip7002_system_part` advances the queue head pointer in storage (lines 242–271 of its module) and calls `update_excess_withdrawal_requests_and_reset_count` — both of which use `?` internally and commit writes to the in-memory cache before any potential failure. If the function returns `Err` after some writes but before completing, those writes remain in the cache while the error is discarded. [2](#0-1) 

The `requests_hasher` is then finalized at line 94 with whatever partial data was accumulated before the failure:

```rust
let requests_hash = Bytes32::from_array(requests_hasher.finalize().into());
``` [3](#0-2) 

The same pattern exists in the proving path (`post_tx_op_proving.rs`), which has 6 references to these same three functions.

---

### Impact Explanation

**State-transition bug + oracle IO mismatch:**

1. **Inconsistent queue state:** If `eip7002_system_part` or `eip7251_system_part` fail after advancing the queue head pointer but before completing the hash update, the withdrawal/consolidation queue is left in a partially-dequeued state. Validator withdrawal requests that were dequeued from storage are not reflected in `requests_hash`. Those requests are effectively lost — they will never be re-processed because the queue head has already advanced past them.

2. **Incorrect `requests_hash` in block header:** The `requests_hash` is computed from an incomplete hasher state. In the proving path, this hash is included in the sealed block header. A block with a wrong `requests_hash` violates the EIP-7685 requests commitment, causing consensus-layer rejection or proof verification failure.

3. **Forward/proving divergence risk:** If the sequencing path silently discards errors while the proving path propagates them (or vice versa), the two paths produce different `requests_hash` values for the same block, making the block unprovable.

---

### Likelihood Explanation

**Medium.** The EIP-7002 and EIP-7251 contracts are system-level predeploys. An attacker who can interact with the EIP-7002 withdrawal contract (by sending ETH with a validator pubkey) can enqueue withdrawal requests. If the attacker can craft a queue state that causes `eip7002_system_part` to fail mid-execution (e.g., by triggering a storage overflow or a `FatalRuntimeError` via resource exhaustion in the native cost accounting path), the silent discard activates. The `with_infinite_ergs` wrapper prevents gas exhaustion but does not prevent `FatalRuntimeError` from native resource exhaustion, which is a realistic trigger under adversarial load. [4](#0-3) 

---

### Recommendation

Replace the silent-discard pattern with proper error propagation:

```rust
// Before (silent discard):
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);

// After (propagate errors):
eip6110_events_parser(&system, &mut requests_hasher)?;
eip7002_system_part(&mut system, &mut requests_hasher)?;
eip7251_system_part(&mut system, &mut requests_hasher)?;
```

If the intent is to tolerate missing predeploys, distinguish that case explicitly (e.g., check deployment status before calling, or match only on the "not deployed" error variant) rather than discarding all errors. [5](#0-4) 

---

### Proof of Concept

1. Deploy the EIP-7002 withdrawal contract at `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS`.
2. Submit enough withdrawal requests to populate the queue (interact with the contract as an unprivileged user by sending ETH with a validator pubkey).
3. Craft a block where `eip7002_system_part` fails after advancing the queue head pointer (e.g., by exhausting native resources during the `update_excess_withdrawal_requests_and_reset_count` storage write).
4. Observe that `post_op` returns `Ok(())` despite the failure — the error is discarded at line 91.
5. The `requests_hash` finalized at line 94 does not include the EIP-7002 type byte or any withdrawal request data.
6. The sealed block header contains an incorrect `requests_hash`, and the withdrawal requests that were dequeued from storage are permanently lost. [6](#0-5) [7](#0-6)
