Audit Report

## Title
Unprivileged Canister Can Monopolize the Threshold Signature Queue Per Key — (`rs/execution_environment/src/execution_environment.rs`)

## Summary

The `sign_with_threshold` function enforces only a global per-key queue size limit with no per-canister sub-limit. A single canister with sufficient cycles can occupy all available slots for a given `key_id`, causing every subsequent signing request from any other canister to be rejected with a queue-full error until the attacker's requests drain. The existing state-machine test directly demonstrates this behavior.

## Finding Description

In `sign_with_threshold` (L3844–3858), the sole admission guard is a global count check:

```rust
if state
    .metadata
    .subnet_call_context_manager
    .sign_with_threshold_contexts_count(&threshold_key)
    >= dynamic_queue_size
{
    return Err(UserError::new(...queue full...));
}
```

`sign_with_threshold_contexts_count` (L437–453) iterates over all pending contexts for the key and counts them regardless of which canister submitted them. The backing store `sign_with_threshold_contexts` (L218) is a `BTreeMap<CallbackId, SignWithThresholdContext>` with no canister-level accounting. The `dynamic_queue_size` (L3838–3842, L5060–5079) is bounded below by `min(max_queue_size_registry, MAX_PAIRED_PRE_SIGNATURES)` — typically 20 for ECDSA/Schnorr — and can grow only when pre-signatures are stashed, which does not introduce per-canister fairness. No other guard in `sign_with_threshold` or `SubnetCallContextManager` limits how many pending contexts a single canister may hold. An attacker deploys one canister, sends exactly `dynamic_queue_size` signing requests (each paying the required fee), and holds all slots. Every subsequent request from any other canister is rejected until the attacker's requests are processed or time out. The `signature_request_timeout_ns` registry parameter can bound the blockade duration, but it is optional and not always configured.

## Impact Explanation

This is a targeted, protocol-level queue monopolization — not a raw volumetric DDoS. It causes an application/platform-level DoS on threshold signing for the targeted key. For subnets hosting ckBTC, ckETH, or other chain-key services, this blocks all cross-chain signing operations for the duration of the attack. The impact fits the **High** bounty tier: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

The attack requires: (1) a deployed canister on the target subnet, (2) `max_queue_size × signature_fee` cycles upfront (e.g., 20 × 10 T = 200 T cycles for ECDSA), and (3) continuous re-submission as slots free up to sustain the blockade. The cycles cost is a real economic barrier but not a technical one. The attacker receives valid signatures in return, making the attack economically self-sustaining. No special privileges, governance access, or node compromise are required.

## Recommendation

Introduce a per-canister sub-limit at the admission point in `sign_with_threshold`. Before inserting into `sign_with_threshold_contexts`, count how many existing contexts share the same `request.sender` and `key_id`, and reject if that count exceeds a configurable per-canister cap (e.g., `max(1, max_queue_size / N)` where N is a fairness divisor, or a hard cap of 1–3). Alternatively, add a `BTreeMap<CanisterId, usize>` counter inside `SubnetCallContextManager` maintained in `push_context` / `retrieve_context` to make this check O(1).

## Proof of Concept

The existing test `test_sign_with_threshold_key_queue_fills_up` at `rs/execution_environment/tests/threshold_signatures.rs` L848–905 already proves a single canister fills the queue to `max_queue_size` (20) and the 21st request is rejected. To confirm cross-canister monopolization, extend the test: create a second canister, have canister A send `max_queue_size` requests, then have canister B send one request — it will be rejected with the queue-full error, directly demonstrating the monopolization invariant violation.
