### Title
Mutable ETH Connector Account Reference Causes Bridge Accounting Corruption and Fund Freeze — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The Aurora Engine stores the ETH connector account ID in mutable storage and exposes `set_eth_connector_contract_account` to update it post-initialization. Every bridge operation — deposits via `ft_on_transfer`, withdrawals via `withdraw`, and base-token exits via the `ExitToNear` precompile — reads this reference at call time via `get_connector_account_id`. Replacing the connector account without migrating the old connector's NEP-141 token balances causes all subsequent bridge operations to route to an empty new connector, silently dropping in-flight deposits and permanently freezing all previously deposited ETH that users hold on Aurora.

---

### Finding Description

**Mutable reference:**

`set_eth_connector_contract_account` writes a new account ID into `EthConnectorStorageId::EthConnectorAccount`: [1](#0-0) 

`set_connector_account_id` overwrites the stored key unconditionally: [2](#0-1) 

**All bridge operations read the reference at call time:**

`return_promise` — used by `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_withdraw`, `ft_balance_of`, `ft_metadata` — resolves the connector account on every invocation: [3](#0-2) 

**Deposit routing is gated on the current connector account:**

`ft_on_transfer` compares `predecessor_account_id` against the live connector account to decide whether to credit base ETH or treat the caller as an ERC-20 NEP-141: [4](#0-3) 

If the connector account has been replaced, the old connector's `ft_on_transfer` call falls into `receive_erc20_tokens`, which looks up an ERC-20 address for the old connector's account ID. No such mapping exists, so the call fails and the full token amount is returned to the old connector — the user's ETH deposit is silently dropped.

**The `ExitToNear` precompile also reads the live connector account:** [5](#0-4) [6](#0-5) 

After a connector swap, base-token exits route to the new connector, which holds no NEP-141 balance, so the cross-contract call fails.

---

### Impact Explanation

**Connector/bridge accounting bug → Critical: Permanent freezing of funds.**

After `set_eth_connector_contract_account(connector_v2)` is called:

1. **Withdrawal freeze**: `withdraw` → `return_promise("engine_withdraw", …)` → `connector_v2.engine_withdraw(…)`. `connector_v2` holds zero NEP-141 tokens; the call fails. All ETH that users hold in Aurora's EVM state cannot be bridged back to NEAR.

2. **Deposit corruption**: Any `ft_on_transfer` from `connector_v1` is misclassified as an ERC-20 deposit. `receive_erc20_tokens` fails (no ERC-20 registered for `connector_v1`), and the full amount is returned. In-flight deposits are silently lost.

3. **ExitToNear failure**: Base-token exits via the precompile route to `connector_v2`, which has no balance, causing the exit promise to fail.

The ETH balances remain in Aurora's EVM state but are permanently unwithdrawable as long as `connector_v2` is the registered account. If `connector_v1` is decommissioned or its state is lost, the freeze is irreversible.

---

### Likelihood Explanation

`set_eth_connector_contract_account` is a documented, intentional owner-callable function — it is exercised in the workspace test suite and the connector test suite as part of normal deployment: [7](#0-6) [8](#0-7) 

An owner performing a legitimate connector upgrade would call this function without knowing that doing so — without first migrating all NEP-141 balances from the old connector — freezes every user's bridged ETH. No migration mechanism exists in the codebase. The function provides no warning, no balance check, and no atomicity guarantee.

---

### Recommendation

Eliminate the ability to update the connector account after initialization, mirroring the Suzaku fix. If upgradeability is required, enforce an atomic migration: the new connector must receive and acknowledge the full NEP-141 balance from the old connector before the reference is updated, and the update must be irreversible once committed.

---

### Proof of Concept

**Illustrative flow:**

1. Users deposit ETH via `connector_v1`; Aurora's EVM state credits their balances. `connector_v1` holds all NEP-141 tokens.
2. Owner calls `set_eth_connector_contract_account({ account: connector_v2, … })`.
3. `connector_v2` is freshly deployed with zero NEP-141 balance.
4. User calls `withdraw(recipient, amount)`:
   - `return_promise` resolves `get_connector_account_id` → `connector_v2`
   - `connector_v2.engine_withdraw(sender, recipient, amount)` is called
   - `connector_v2` has no NEP-141 balance → call fails
   - User's EVM balance is unchanged but permanently unwithdrawable
5. Another user initiates a deposit: `connector_v1` calls `ft_on_transfer` on Aurora:
   - `predecessor_account_id (connector_v1) != get_connector_account_id() (connector_v2)`
   - Falls into `receive_erc20_tokens(connector_v1, …)`
   - `get_erc20_from_nep141(connector_v1)` → not found → error
   - Full amount returned to `connector_v1`; deposit silently dropped [9](#0-8) [4](#0-3) [3](#0-2) [10](#0-9) [1](#0-0)

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
