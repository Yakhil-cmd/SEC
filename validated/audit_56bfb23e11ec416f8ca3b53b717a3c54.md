Audit Report

## Title
Logo-Omission Guard Permanently Disabled in `convert_proposal` When `return_self_describing_action=true` — (`rs/nns/governance/src/pb/proposal_conversions.rs`)

## Summary
In `convert_proposal`, `is_create_service_nervous_system_proposal` is derived from the already-conditionally-suppressed `action` variable, which is `None` whenever `show_action=false` (i.e., whenever `return_self_describing_action=true`). This causes the logo-omission guard passed to `convert_self_describing_action` to always evaluate to `false`, bypassing the `omit_large_fields` protection for the `self_describing_action` path. Any unprivileged caller can trigger this via `list_proposals` with `return_self_describing_action=true` and `omit_large_fields=true`, receiving full logo bytes regardless of the omission request, potentially causing the response to exceed the IC's 2 MB query response limit and trap.

## Finding Description
`ProposalDisplayOptions::for_list_proposals` sets `show_action: !return_self_describing_action` (line 24). When `return_self_describing_action=true`, `show_action=false`.

In `convert_proposal` (lines 495–499), the `action` variable is set to `None` when `show_action()` is false:
```rust
let action = if display_options.show_action() {
    action.as_ref().map(|x| convert_action(x, display_options))
} else {
    None
};
```

Immediately after (lines 500–502), `is_create_service_nervous_system_proposal` is derived from this already-`None` `action`:
```rust
let is_create_service_nervous_system_proposal = action.as_ref().is_some_and(|action| {
    matches!(action, api::proposal::Action::CreateServiceNervousSystem(_))
});
```

Since `action` is `None`, `is_create_service_nervous_system_proposal` is always `false`. The guard passed to `convert_self_describing_action` (lines 509–510) therefore evaluates to `false && display_options.omit_create_service_nervous_system_large_fields()` = `false`, regardless of whether `omit_large_fields` was requested.

Inside `convert_self_describing_action` (lines 317–328), when `omit_create_service_nervous_system_logos=false`, `paths_to_omit` is an empty `vec![]`, so `logo` and `ledger_parameters.token_logo` fields pass through unfiltered into the response.

`omit_create_service_nervous_system_large_fields()` itself correctly returns `true` when `omit_large_fields_requested=true` and `multi_query=true` (line 59–61), but this result is never reached due to the short-circuit.

The root cause is that the type-detection logic (`is_create_service_nervous_system_proposal`) is coupled to the display-suppression logic (`show_action`), using the converted output rather than the original `pb::Proposal` action field.

## Impact Explanation
This is an application/platform-level DoS on the NNS governance canister's `list_proposals` query endpoint. A `CreateServiceNervousSystem` proposal logo can be up to ~256 KiB (base64-encoded). With multiple such proposals in a single `list_proposals` response window, the combined payload can exceed the IC's 2 MB query response limit, causing response serialization to fail and the query to trap for all callers using the `return_self_describing_action=true` path. This matches the **High** impact category: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
`list_proposals` is a public, unauthenticated query call — any caller can set `omit_large_fields=true` and `return_self_describing_action=true`. The NNS mainnet has had multiple real `CreateServiceNervousSystem` proposals, each potentially carrying logo blobs. A handful of such proposals in the default response window is sufficient to trigger an oversized response. The attack requires no special privileges, no victim interaction, and is repeatable.

## Recommendation
Derive `is_create_service_nervous_system_proposal` from the original `pb::Proposal` action field (the `action` binding from the destructure at line 486), not from the already-conditionally-suppressed converted `action` variable. For example:

```rust
let is_create_service_nervous_system_proposal = item.action.as_ref().is_some_and(|action| {
    matches!(action, pb::proposal::Action::CreateServiceNervousSystem(_))
});
```

This decouples type-detection from display-suppression. Additionally, add an integration-level test for `convert_proposal` that exercises the `return_self_describing_action=true` + `omit_large_fields=true` path with a `CreateServiceNervousSystem` proposal containing a logo, asserting that the logo is `Null` in the output.

## Proof of Concept
1. Construct a `pb::Proposal` with `action = Some(pb::proposal::Action::CreateServiceNervousSystem(...))` containing a large logo, and a populated `self_describing_action` field.
2. Call `convert_proposal` with `ProposalDisplayOptions::for_list_proposals(omit_large_fields_requested=true, return_self_describing_action=true)`.
3. Observe that the returned `api::Proposal.self_describing_action.value` map contains the full logo bytes under `"logo"` (not `Null`), confirming the guard is bypassed.
4. As a unit test: assert `result.self_describing_action.unwrap().value.unwrap()` contains `api::SelfDescribingValue::Null` for the `"logo"` key — this assertion will fail against the current code, confirming the bug.
5. On mainnet: call `list_proposals` with `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)` against a governance state containing multiple `CreateServiceNervousSystem` proposals with large logos; observe the query traps due to response size exceeding 2 MB.
