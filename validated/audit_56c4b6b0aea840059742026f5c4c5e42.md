### Title
SNS Governance `distribute_rewards` Computes Inflated Reward Purse When `end_timestamp_seconds` Is `None` - (File: `rs/sns/governance/src/governance.rs`)

### Summary
The SNS governance `distribute_rewards` function uses `.unwrap_or_default()` on the optional `end_timestamp_seconds` field of `latest_reward_event` to determine the start of the current reward period. If `end_timestamp_seconds` is `None` — which occurs for any SNS instance whose persisted `RewardEvent` predates the addition of this field — the function silently treats the reward period start as Unix epoch (timestamp 0). This causes `new_rounds_count` to be computed as `now / round_duration_seconds` (potentially thousands of rounds), and the entire accumulated reward purse for that inflated span is distributed to the first voters in the next settled round, directly mirroring the ERC20Rewards `lastUpdated` stale-timestamp bug.

### Finding Description

In `rs/sns/governance/src/governance.rs`, `distribute_rewards` computes the reward window start as:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or_default();          // ← returns 0 if field is None
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [1](#0-0) 

`end_timestamp_seconds` is declared `optional` in the protobuf schema and maps to `Option<u64>` in Rust: [2](#0-1) [3](#0-2) 

When `end_timestamp_seconds` is `None`, `unwrap_or_default()` yields `0`. With `now` ≈ 1.7 × 10⁹ s and a typical `round_duration_seconds` of one week (604 800 s), `new_rounds_count` becomes ≈ 2 800. The reward purse loop then accumulates:

```rust
for i in 1..=new_rounds_count {
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [4](#0-3) 

producing a purse that is thousands of times larger than a single legitimate round's allocation. If the purse fits in `u64`, it is distributed in full to whoever voted in the first settled proposal after the upgrade. If it overflows `u64`, the function returns early without updating `latest_reward_event`, permanently stalling reward distribution.

The same `unwrap_or_default()` pattern appears in `should_distribute_rewards`, causing it to return `true` immediately after the upgrade (since `now − 0 ≫ round_duration_seconds`): [5](#0-4) 

The `Governance::new()` constructor only inserts a corrective dummy event when `latest_reward_event` is entirely absent (`is_none()`); it does **not** repair a present event whose `end_timestamp_seconds` is `None`: [6](#0-5) 

Likewise, `canister_post_upgrade` calls `canister_init_` → `Governance::new()` and performs no additional migration for this field: [7](#0-6) 

### Impact Explanation

Any neuron that votes on the first proposal settled after the upgrade receives maturity proportional to thousands of rounds of supply × reward-rate instead of one round. For a 1 billion-token supply at a 10 % annual rate and a 1-week round, the inflated purse is on the order of 5 × 10⁹ tokens-worth of maturity — five times the entire token supply — distributed to a single cohort of voters. This constitutes a direct, irreversible ledger-conservation violation: maturity is minted from thin air for time periods during which the recipients held no stake.

### Likelihood Explanation

The `end_timestamp_seconds` field was added to `RewardEvent` after the initial SNS deployment (the `round` field is explicitly marked `DEPRECATED: Use end_timestamp_seconds instead`). Any SNS instance whose stable-memory snapshot was written before that field existed will have `end_timestamp_seconds = None` in its persisted `latest_reward_event`. The vulnerability fires automatically on the very first `run_periodic_tasks` heartbeat after the canister is upgraded to a version that reads `end_timestamp_seconds`, requiring no privileged action. A sophisticated observer who monitors upgrade proposals can time their neuron's vote to be the sole beneficiary of the inflated purse.

### Recommendation

In `Governance::new()`, add a migration guard that repairs a present-but-incomplete `RewardEvent`:

```rust
if let Some(ref mut event) = proto.latest_reward_event {
    if event.end_timestamp_seconds.is_none() {
        // Anchor to genesis so no phantom rounds are counted.
        event.end_timestamp_seconds = Some(proto.genesis_timestamp_seconds);
    }
}
```

Alternatively, replace the bare `unwrap_or_default()` calls in both `distribute_rewards` and `should_distribute_rewards` with a fallback to `genesis_timestamp_seconds`:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or(self.proto.genesis_timestamp_seconds);
```

This mirrors the fix recommended in the ERC20Rewards report: always advance the "last-updated" anchor, even when no participants are present, so that no phantom reward interval can accumulate.

### Proof of Concept

1. An SNS is deployed while `RewardEvent` lacks `end_timestamp_seconds`; the canister runs normally, storing `latest_reward_event` with `end_timestamp_seconds = None`.
2. The SNS is upgraded to the current codebase version.
3. `canister_post_upgrade` → `canister_init_` → `Governance::new()`: because `latest_reward_event.is_some()`, the constructor skips the dummy-event insertion; `end_timestamp_seconds` remains `None`.
4. The next `run_periodic_tasks` heartbeat calls `should_distribute_rewards`: `now − 0 = now ≫ round_duration_seconds` → returns `true`.
5. `distribute_rewards` is called: `reward_start_timestamp_seconds = 0`, `new_rounds_count ≈ 2 800`.
6. The reward purse loop accumulates ≈ 5 × 10⁹ token-equivalents of maturity
