### Title
Unprivileged Contract Can Inject Arbitrary Interop Roots Into Batch Public Input via Unguarded Event Hook - (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` processes any EVM event matching the `InteropRootAdded` signature without verifying the emitting contract's address or the validity of the `chain_id` field. Any unprivileged user can deploy a contract that emits this event with attacker-controlled `chain_id`, `block_or_batch_number`, and `root` values, injecting fake interop roots into the batch's `interop_roots_rolling_hash` committed in the ZK proof public input.

---

### Finding Description

The system hook `interop_root_reporter_event_hook` is triggered whenever any EVM event with topic `INTEROP_ROOT_ADDED_EVENT_SIG` (`0x6b451b84...`) is emitted during block execution. The hook parses the event data and unconditionally calls `system.io.add_interop_root()` with the attacker-supplied `chain_id`, `block_or_batch_number`, and `root` values.

**Root cause — no emitter address check:**

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<...>,
    data: &[u8],
    _caller_ee: u8,          // ← ignored, no address check
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError> {
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // ... structural checks only (data length, offset, len, topics count) ...
    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot { root, block_or_batch_number, chain_id },
    )?;
``` [1](#0-0) 

The `_caller_ee` parameter is explicitly ignored (underscore prefix). No check is made on which contract address emitted the event.

**Root cause — no content validation in storage layer:**

`InteropRootStorage::push_root` performs zero validation on the `InteropRoot` fields despite the struct's own documentation stating `chain_id` must be non-zero and `root` cannot be zero:

```rust
pub fn push_root(&mut self, interop_root: InteropRoot) -> Result<(), SystemError> {
    self.list.push(interop_root, ());
    Ok(())
}
``` [2](#0-1) 

The struct comments document these invariants but they are never enforced at the system layer: [3](#0-2) 

**How the injected roots corrupt the public input:**

At batch finalization, `calculate_interop_roots_rolling_hash` iterates over all accumulated interop roots (including attacker-injected ones) and folds them into a single hash:

```rust
let interop_roots_rolling_hash = calculate_interop_roots_rolling_hash(
    Bytes32::zero(),
    io.interop_root_storage.iter(),
    &mut crypto::sha3::Keccak256::new(),
);
``` [4](#0-3) 

This hash is then committed into `BatchOutput`, which is hashed into the ZK proof's public input:

```rust
let batch_output = BatchOutput {
    ...
    interop_roots_rolling_hash,
    ...
};
let public_input = BatchPublicInput {
    ...
    batch_output: batch_output.hash().into(),
};
``` [5](#0-4) 

The `interop_roots_rolling_hash` field is explicitly included in `BatchOutput::hash()`: [6](#0-5) 

The same path exists in the multiblock batch accumulator `ZKBatchDataKeeper::apply_block`: [7](#0-6) 

---

### Impact Explanation

An attacker can forge the `interop_roots_rolling_hash` committed in the ZK proof's public input. The settlement layer uses this hash to verify cross-chain message inclusion proofs. By injecting fake interop roots (arbitrary `chain_id`, `block_or_batch_number`, `root`), the attacker can:

1. **Forge cross-chain messages**: Cause the settlement layer to accept a proof attesting to interop roots that were never legitimately produced by the claimed source chain, enabling fraudulent cross-chain asset transfers or message delivery.
2. **Corrupt batch finality**: The `interop_roots_rolling_hash` in the public input will differ from what honest verifiers expect, potentially causing proof rejection or acceptance of a forged state.

This is a direct funds-loss path via cross-chain message forgery.

---

### Likelihood Explanation

Any unprivileged L2 user can:
1. Deploy a contract that emits `InteropRootAdded(chain_id, block_or_batch_number, [root])` with arbitrary values.
2. Call that contract via a standard L2 transaction.
3. The hook fires unconditionally, injecting the fake root.

No special privileges, governance access, or cryptographic breaks are required. The attack is fully reachable from a normal user transaction.

---

### Recommendation

1. **Enforce emitter address**: In `interop_root_reporter_event_hook`, check that the event was emitted by the designated `L2_INTEROP_ROOT_STORAGE_ADDRESS` system contract. The `_caller_ee` parameter should be replaced with a caller address check.
2. **Enforce field invariants at the system layer**: In `InteropRootStorage::push_root` (or in `add_interop_root` in `io_subsystem.rs`), assert `interop_root.root != Bytes32::zero()` and `interop_root.chain_id != U256::ZERO` to match the documented invariants.
3. **Defense in depth**: Consider a whitelist of system contract addresses permitted to trigger each event hook, similar to how `NewSettlementLayerChainIdStorage` enforces single-update-per-block semantics.

---

### Proof of Concept

1. Attacker deploys `MaliciousInteropEmitter` on ZKsync OS:
   ```solidity
   contract MaliciousInteropEmitter {
       // keccak256("InteropRootAdded(uint256,uint256,bytes32[])")
       bytes32 constant SIG = 0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0;
       function inject(uint256 chainId, uint256 batchNum, bytes32 root) external {
           // Emit InteropRootAdded(chainId, batchNum, [root])
           // ABI-encoded data: offset=32, len=1, root
           bytes memory data = abi.encode(uint256(32), uint256(1), root);
           assembly {
               log3(add(data, 32), mload(data), SIG, chainId, batchNum)
           }
       }
   }
   ```
2. Attacker calls `inject(victimChainId, victimBatchNum, forgedRoot)` via a standard L2 transaction.
3. `interop_root_reporter_event_hook` fires: structural checks pass (data length = 96, offset = 32, len = 1, topics count = 3).
4. `system.io.add_interop_root(InteropRoot { root: forgedRoot, block_or_batch_number: victimBatchNum, chain_id: victimChainId })` is called with no further validation.
5. At batch finalization, `calculate_interop_roots_rolling_hash` folds the forged root into `interop_roots_rolling_hash`.
6. The ZK proof is generated with a public input containing the forged hash.
7. The settlement layer accepts the proof, treating the forged root as a legitimate cross-chain commitment from `victimChainId` at `victimBatchNum`.

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;
