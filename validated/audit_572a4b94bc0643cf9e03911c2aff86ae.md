### Title
Missing EIP-2537 (BLS12-381) and EIP-152 (Blake2F) Precompile Hook Registrations in ZK Block Post-Init - (File: `basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs`)

### Summary
The ZK block execution path's post-initialization function omits registration of BLS12-381 (EIP-2537) and Blake2F (EIP-152) precompile hooks that the Ethereum block execution path correctly registers. This mirrors the external report's pattern: a component is initialized but missing necessary capability grants, preventing it from completing its function. Calls to BLS12-381 and Blake2F precompile addresses in the ZK execution path fall through to empty-account execution, returning empty data instead of correct precompile output.

### Finding Description
`EthereumPostInitOp::post_init_op` registers three sets of hooks:

```rust
// basic_bootloader/src/bootloader/block_flow/ethereum/post_init_op.rs
add_precompiles(system_functions)?;
hooks::eip_2537::initialize_eip_2537(system_functions)?;
hooks::eip_152::initialize_eip_152(system_functions)?;
``` [1](#0-0) 

`ZKHeaderPostInitOp::post_init_op` only registers the base set and ZKsync-specific system hooks — the two EIP-specific initializers are absent:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs
system_hooks::add_precompiles(system_functions)?;
// initialize_eip_2537 — MISSING
// initialize_eip_152  — MISSING
system_hooks::add_l1_messenger(system_functions)?;
...
``` [2](#0-1) 

`initialize_eip_2537` registers seven BLS12-381 hooks (G1ADD, G2ADD, G1MSM, G2MSM, pairing check, map-fp-to-G1, map-fp2-to-G2) at addresses `0x0b`–`0x11`. [3](#0-2) 

`initialize_eip_152` registers the Blake2F hook at address `0x09`. [4](#0-3) 

When `HooksStorage::try_intercept` finds no registered hook for an address it returns `Ok(None)`, causing the call to fall through to normal (empty) account execution. [5](#0-4) 

`PRECOMPILE_ADDRESSES_LOWS` includes the BLS12-381 addresses when the `eip-2537` feature is compiled in, and `is_precompile` consults this list for gas-accounting decisions — so the EVM interpreter treats those addresses as precompiles while the hook layer silently returns nothing. [6](#0-5) 

### Impact Explanation
Any EVM contract deployed on ZKsync that calls a BLS12-381 or Blake2F precompile during a ZK-path block will receive empty returndata (`0x`) instead of the correct cryptographic output. Because the call does not revert, the contract silently proceeds with a zero/empty result. Downstream logic that relies on the precompile output (e.g., BLS signature verification, Blake2F hashing) will produce incorrect results, leading to corrupted state transitions that are provably wrong yet accepted by the sequencer.

### Likelihood Explanation
EIP-2537

    pub fn try_intercept<'a>(
        &mut self,
        address_low: u16,
        request: ExternalCallRequest<S>,
        caller_ee: u8,
        system: &mut System<S>,
        return_memory: &'a mut [MaybeUninit<u8>],
    ) -> Result<(Option<CompletedExecution<'a, S>>, &'a mut [MaybeUninit<u8>]), SystemError> {
        let Some(hook) = self.call_hooks.get(&address_low) else {
            return Ok((None, return_memory));
        };
        let (res, remaining_memory) = hook.0(request, caller_ee, system, return_memory)?;

        Ok((Some(res), remaining_memory))
    }
