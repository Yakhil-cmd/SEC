Audit Report

## Title
Unauthenticated DoS via Unconditional `todo!()` Panic in `/block` Hash-Only Request with `RosettaBlocksMode::Enabled` — (`rs/rosetta-api/icp/src/request_handler.rs`)

## Summary

When the ICP Rosetta node runs with `RosettaBlocksMode::Enabled`, any unauthenticated HTTP client can trigger an unconditional `todo!()` panic in the actix-web worker thread by sending a `/block` POST request containing only a block hash (no index). The `get_verified_block_by_hash` function returns `Err(ApiError::InvalidBlockId(true, ...))` for any unrecognized hash, which is caught by the `Err(ApiError::InvalidBlockId(_, _))` match arm at line 358 and immediately panics via `todo!()`. This crashes the worker thread and can be repeated indefinitely to deny service to all Rosetta API consumers.

## Finding Description

The exploit path is fully confirmed by the code:

**Step 1** — `rosetta_server.rs` lines 58–68: the public `/block` endpoint accepts any unauthenticated POST with a valid JSON body and delegates to `req_handler.block()`. [1](#0-0) 

**Step 2** — `request_handler.rs` lines 298–305: `block()` calls `get_block(Some(msg.block_identifier))`. A request body of `{ "block_identifier": { "hash": "<hex>" } }` produces `PartialBlockIdentifier { index: None, hash: Some(hash) }`. [2](#0-1) 

**Step 3** — `request_handler.rs` lines 349–366: the `index: None, hash: Some(hash)` arm is matched. When `is_rosetta_blocks_mode_enabled()` returns `true`, `get_verified_block_by_hash(&hash)` is called. [3](#0-2) 

**Step 4** — `request_handler.rs` lines 451–465: `get_verified_block_by_hash` calls `blocks.is_verified_by_hash(&hash)?`. For any hash absent from the SQLite DB, this returns `Ok(false)`, so `!false` is `true` and the function immediately returns `Err(ApiError::InvalidBlockId(true, Default::default()))`. [4](#0-3) 

**Step 5** — Back in `get_block()` at line 358, the returned `Err(ApiError::InvalidBlockId(true, ...))` matches `Err(ApiError::InvalidBlockId(_, _))` and `todo!()` panics unconditionally. There is no guard, fallback, or error return — the `todo!()` is the only branch. [5](#0-4) 

The `From<BlockStoreError> for ApiError` conversion in `errors.rs` lines 108–119 also maps `BlockStoreError::NotFound` to `ApiError::invalid_block_id(...)`, which is `ApiError::InvalidBlockId(false, ...)`, also matching the `todo!()` arm. [6](#0-5) 

## Impact Explanation

This is a **High** severity DoS. The ICP Rosetta API is an explicitly in-scope financial integration component. Panicking an actix-web worker thread causes the thread to be torn down and restarted; the panicking request receives no response. An attacker sending a continuous stream of hash-only `/block` requests can keep all worker threads in a crash-restart cycle, effectively denying service to all legitimate Rosetta API consumers (exchange integrations, wallets, tooling). This matches the allowed impact: *"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant... Rosetta... security impact with concrete user or protocol harm."*

## Likelihood Explanation

- No authentication is required; the `/block` endpoint is fully public.
- The triggering request is trivial: any 64-character hex string that is not a real block hash suffices.
- `RosettaBlocksMode::Enabled` is a documented, supported production configuration.
- The panic is deterministic and reproducible on every such request.
- No rate limiting is present in the handler path.

## Recommendation

Replace the `todo!()` at line 359 with a proper error return:

```rust
Err(ApiError::InvalidBlockId(false, "Fetching Rosetta Blocks by hash is not yet supported".into()))
```

Or, per the comment at lines 354–355, implement the missing Rosetta-block-by-hash lookup and return `InvalidBlockId` if neither lookup succeeds. The `todo!()` must not remain reachable from an unauthenticated request path.

## Proof of Concept

```bash
# Rosetta node running with RosettaBlocksMode::Enabled on localhost:8080
curl -s -X POST http://localhost:8080/block \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "block_identifier": {
      "hash": "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"
    }
  }'
# Worker thread panics: "not yet implemented: Fetching Rosetta Blocks by hash is not supported yet"
# Repeating the request reproduces the panic indefinitely.
```

A deterministic integration test can be written by constructing a `RosettaRequestHandler` with `RosettaBlocksMode::Enabled`, calling `block()` with a `PartialBlockIdentifier { index: None, hash: Some("<unknown hex>") }`, and asserting the call panics (via `std::panic::catch_unwind`).
