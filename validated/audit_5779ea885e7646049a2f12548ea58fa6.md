### Title
SNS Governance Reward Miscalculation When `VotingRewardsParameters` Changes Mid-Round - (File: `rs/sns/governance/src/governance.rs`)

### Summary

When a `ManageNervousSystemParameters` proposal changes `VotingRewardsParameters` (e.g., `initial_reward_rate_basis_points`), the `perform_manage_nervous_system_parameters` function immediately overwrites the live parameters without first settling the in-progress reward round. The subsequent `distribute_rewards` call then applies the **new** rate schedule retroactively to the entire elapsed period, causing neurons that voted before the change to receive materially fewer (or more) maturity than they earned.

### Finding Description

`perform_action` dispatches `Action::ManageNervousSystemParameters` directly to `perform_manage_nervous_system_parameters`:

```rust
// rs/sns/governance/src/governance.rs ~line 2144
Action::ManageNervousSystemParameters(params) => {
    self.perform_manage_nervous_system_parameters(params)
}
```

`perform_manage_nervous_system_parameters` unconditionally replaces the live parameters with no prior reward settlement:

```rust
// rs/sns/governance/src/governance.rs ~line 2597
self.proto.parameters = Some(new_params);
``` [1](#0-0) 

Later, `distribute_rewards` reads the **current** (already-changed) `voting_rewards_parameters` and uses them to price every round since the last reward event:

```rust
// rs/sns/governance/src/governance.rs ~line 5867-5871
let current_reward_rate = voting_rewards_parameters.reward_rate_at(
    crate::reward::Instant::from_seconds_since_genesis(i2d(seconds_since_genesis)),
);
result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
``` [2](#0-1) 

`voting_rewards_parameters` is fetched from `self.nervous_system_parameters_or_panic()` — the post-change state — so the entire elapsed period is priced at the new rate, not the rate that was in effect when neurons actually voted. [3](#0-2) 

The `reward_rate_at` function derives the rate from `initial_reward_rate_basis_points` and `final_reward_rate_basis_points` stored in the parameters struct: [4](#0-3) 

There is no call to `distribute_rewards` anywhere in the proposal execution path before `perform_manage_nervous_system_parameters` mutates state. `distribute_rewards` is only invoked from `run_periodic_tasks`, which runs on a heartbeat cadence independent of proposal execution.

### Impact Explanation

If a `ManageNervousSystemParameters` proposal reduces `initial_reward_rate_basis_points` (e.g., from 1000 bp / 10% to 100 bp / 1%), the reward purse for the entire current round is computed at the new 1% rate. Neurons that voted during the first portion of the round — when the rate was 10% — receive up to 10× less maturity than they earned. The converse (rate increase) silently over-distributes maturity. Because maturity is the precursor to minted ICP, this directly affects token economics for all SNS participants.

### Likelihood Explanation

`ManageNervousSystemParameters` is a standard, non-critical SNS proposal type available to any neuron holder with sufficient stake. SNSes routinely adjust reward parameters as part of tokenomics governance. No malicious intent is required: a legitimate community vote to lower the reward rate mid-round silently triggers the miscalculation. The window of exposure equals the time elapsed since the last `RewardEvent`, which can be up to one full `round_duration_seconds` (days to weeks depending on SNS configuration). [5](#0-4) 

### Recommendation

In `perform_manage_nervous_system_parameters`, call `distribute_rewards` (or an equivalent checkpoint) before overwriting `self.proto.parameters` when `voting_rewards_parameters` is being changed. This mirrors the `massUpdate()` fix recommended in the MasterChef report: settle all accrued rewards at the old rate before the slope changes.

```rust
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    // If voting_rewards_parameters is changing, settle the current round first.
    if new_params.voting_rewards_parameters
        != self.nervous_system_parameters_or_panic().voting_rewards_parameters
    {
        if let Some(supply) = self.get_cached_token_supply() {
            self.distribute_rewards(supply);
        }
    }
    match new_params.validate() {
        Ok(()) => { self.proto.parameters = Some(new_params); Ok(()) }
        Err(msg) => Err(...)
    }
}
```

### Proof of Concept

1. SNS launches with `initial_reward_rate_basis_points = 1000` (10 %), `final_reward_rate_basis_points = 500` (5 %), `round_duration_seconds = 604800` (1 week).
2. At T = 0 a reward event fires; `latest_reward_event.end_timestamp_seconds = T0`.
3. At T = 3.5 days Alice votes on Proposal A (half the round has elapsed at the old 10 % rate).
4. At T = 3.5 days a `ManageNervousSystemParameters` proposal passes, setting `initial_reward_rate_basis_points = 100` (1 %) and `final_reward_rate_basis_points = 50` (0.5 %). `perform_manage_nervous_system_parameters` overwrites parameters immediately; no reward settlement occurs.
5. At T = 7 days `distribute_rewards` fires. It reads the new parameters and computes:
   - `new_rounds_count = 1`
   - `reward_rate_at(T=7 days)` ≈ 1 % (new initial rate, transition barely started)
   - `rewards_purse ≈ 1% × 1 week × supply`
6. Alice receives ≈ 1/10 of the maturity she earned during the first 3.5 days at the old 10 % rate. The remaining ~90 % of her earned rewards are permanently lost — they are neither rolled over nor distributed. [6](#0-5) [7](#0-6)

        log!(

    pub fn reward_rate_at(&self, now: Instant) -> RewardRate {
        let reward_rate_transition_duration_seconds = self
            .reward_rate_transition_duration_seconds
            .expect("reward_rate_transition_duration_seconds unset");

        let time_since_genesis = {
            let result = now - *GENESIS;
            // For the purposes of determining reward rate, treat times before
            // genesis the same as at genesis. This is not expected to occur in
            // practice. This code is just being extra defensive.
            if result.as_secs() < i2d(0) {
                Duration { days: i2d(0) }
            } else {
                result
            }
        };
        if reward_rate_transition_duration_seconds == 0
            || time_since_genesis.as_secs() >= i2d(reward_rate_transition_duration_seconds)
        {
            return self.final_reward_rate();
        }

        // s linearly varies from 1 -> 0 as seconds_since_genesis varies from 0
        // to reward_rate_transition_duration_seconds.
        let transition = LinearMap::new(
            dec!(0)..i2d(reward_rate_transition_duration_seconds),
            dec!(1)..dec!(0),
        );
        let s = transition.apply(time_since_genesis.as_secs());
        // s2 varies quadratically from 1 -> 0 (again, as seconds_since_genesis
        // varies from 0 to reward_rate_transition_duration_seconds), and
        // flattens out as seconds_since_genesis approaches
        // reward_rate_transition_duration_seconds.
        let s2 = s * s;

        // This looks backwards, but we think of variable rate as being added to
        // final growth rate, not initial, and the amount to add is up to
        // initial - final (where initial is thought of as being greater than
        // final).
        let dr = self.initial_reward_rate() - self.final_reward_rate();
        // variable_reward_rate varies from dr to 0 as round varies from
        // 1 to transition_round_count.
        let variable_reward_rate = s2 * dr;

        self.final_reward_rate() + variable_reward_rate
