After thorough analysis of the ZKsync OS codebase, I found a valid analog vulnerability.

---

### Title
L1 Transaction Pubdata Charge Incorrectly Includes Prior-Transaction Pubdata on Revert/Failure — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

When an L1 (priority) transaction fails or reverts, the pubdata charge used in the refund calculation is computed against the **total pubdata accumulated since the start of the block** (including pubdata from all prior transactions), rather than only the pubdata generated by the failing L1 transaction itself. This is a direct resource accounting mismatch: committed pubdata from previous transactions is not excluded from the cap calculation, causing L1 transaction users to be overcharged for gas they did not consume.

### Finding Description

In `process_l1_transaction`, when the L1 transaction fails (`pubdata_info = None`), the pubdata charge is computed as:

```rust
None => get_resources_to_charge_for_pubdata(system, native_per_pubdata, None)?,
``` [1](#0-0) 

With `base_pubdata = None`, the helper computes:

```rust
let current_pubdata_spent = system
    .net_pubdata_used()?
    .saturating_sub(base_pubdata.unwrap_or(0));  // unwrap_or(0) → charges from block start
``` [2](#0-1) 

After the outer frame rollback (`system.finish_global_frame(Some(&rollback_handle))`), the pubdata counter is restored to `P_prev` — the pubdata generated by all **previous** transactions in the block. The call with `base_pubdata = None` therefore charges the failing L1 tx for `P_prev` bytes it never generated. [3](#0-2) 

The same flaw exists inside `execute_l1_transaction_and_notify_result`, where the post-execution pubdata sufficiency check also uses `base_pubdata = None`:

```rust
let (enough, to_charge_for_pubdata, pubdata_used) =
    check_enough_resources_for_pubdata(system, native_per_pubdata, resources, None)?;
```
