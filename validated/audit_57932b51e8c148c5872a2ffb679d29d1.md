### Title
`receive_base_tokens` Does Not Enforce the Address Whitelist, Allowing Non-Whitelisted Addresses to Receive ETH in Silo Mode — (`engine/src/engine.rs`)

---

### Summary

In Silo mode, Aurora enforces an `Address` whitelist that restricts which EVM addresses may receive ERC-20 tokens via the NEP-141 bridge. The `receive_erc20_tokens` function correctly checks `silo::is_allow_receive_erc20_tokens` and redirects tokens to a configured fallback address when the recipient is not whitelisted. However, the sibling function `receive_base_tokens`, which mints native ETH to a recipient via the same `ft_on_transfer` entry point, performs **no whitelist check at all**. This is a direct analog to the reported vulnerability: one operation enforces the access control, the other does not.

---

### Finding Description

Both `receive_base_tokens` and `receive_erc20_tokens` are called from the same `ft_on_transfer` contract method, branching on whether the predecessor is the ETH connector account or a NEP-141 token account. [1](#0-0) 

`receive_erc20_tokens` checks the whitelist and redirects to the fallback address if the recipient is not allowed: [2](#0-1) 

`receive_base_tokens` performs no such check — it directly credits the recipient's ETH balance unconditionally: [3](#0-2) 

The whitelist enforcement function `is_allow_receive_erc20_tokens` and the fallback address mechanism exist precisely to prevent non-whitelisted addresses from receiving tokens in Silo mode: [4](#0-3) 

The `Address` whitelist check is also enforced for all EVM transaction submissions via `assert_access`: [5](#0-4) 

---

### Impact Explanation

When the `Address` whitelist is enabled in Silo mode and a fallback address is configured, any ETH bridged via `ft_on_transfer` to a non-whitelisted EVM address is minted directly to that address instead of being redirected to the fallback. Because `assert_access` blocks all EVM transaction submissions from non-whitelisted addresses, the ETH credited to a non-whitelisted address is permanently inaccessible — the address cannot submit any transaction to move or use those funds. This constitutes **permanent freezing of funds**.

---

### Likelihood Explanation

This is reachable by any unprivileged NEAR user who holds ETH on the NEAR side and calls `ft_transfer_call` targeting the Aurora engine with a non-whitelisted EVM address as the recipient. The only precondition is that the Silo operator has enabled the `Address` whitelist and configured a fallback address (which is the intended production configuration for Silo deployments). No privileged access is required.

---

### Recommendation

Apply the same whitelist-and-fallback guard in `receive_base_tokens` that is already applied in `receive_erc20_tokens`:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut receipient = message_data.recipient;

    // Mirror the same guard used in receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
    {
        receipient = fallback_address;
    }

    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

---

### Proof of Concept

1. Silo operator enables the `Address` whitelist and sets a fallback address via `set_erc20_fallback_address`.
2. Address `0xVICTIM` is **not** added to the `Address` whitelist.
3. A NEAR user calls `ft_transfer_call` on the ETH connector NEP-141 contract, targeting the Aurora engine, with `msg` set to `0xVICTIM`'s hex address.
4. Aurora's `ft_on_transfer` is invoked with `predecessor == connector_account_id`, so it calls `engine.receive_base_tokens(&args)`.
5. `receive_base_tokens` skips all whitelist checks and credits ETH to `0xVICTIM`.
6. `0xVICTIM` now holds ETH but cannot call `submit` (blocked by `assert_access` / `is_allow_submit`), so the ETH is permanently frozen. [3](#0-2) [2](#0-1) [1](#0-0)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
