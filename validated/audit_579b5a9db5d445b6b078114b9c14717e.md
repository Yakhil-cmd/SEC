### Title
Unbounded Iteration Over All Voting Neurons in SNS Governance `distribute_rewards` Can Permanently Halt Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

### Summary
The SNS Governance canister's `distribute_rewards` function iterates over all voting neurons and all proposal ballots in a single synchronous execution segment with no instruction-limit guard. If the number of voters is large enough to exhaust the per-message instruction limit, the heartbeat traps, all state changes are rolled back, and the reward event is never recorded — causing reward distribution and proposal settlement to be permanently blocked.

### Finding Description

`distribute_rewards` in the SNS Governance canister performs three unbounded loops in a single synchronous execution (no `await` points inside the function):

1. **Reward-round backfill loop** (line 5861): iterates over every missed reward round since the last event.
2. **Ballot aggregation loop** (lines 5894–5930): for every `ReadyToSettle` proposal, iterates over every ballot (one per voting neuron).
3. **Maturity distribution loop** (lines 5954–5997): iterates over every unique voter to increment neuron maturity.

None of these loops contain an instruction-limit check or a batching mechanism. [1](#0-0) 

The function is called from `run_periodic_tasks` after a single `await` on `total_supply()`, meaning the entire body of `distribute_rewards` executes as one uninterrupted message slice: [2](#0-1) 

The NNS Governance canister faced the identical problem and was explicitly fixed: reward distribution was moved to a timer task with a `DISTRIBUTION_MESSAGE_LIMIT` guard and a `continue_processing` loop that breaks when `is_message_over_threshold` returns true. [3](#0-2) 

The NNS CHANGELOG explicitly documents this fix: [4](#0-3) 

The SNS Governance canister has no equivalent protection.

### Impact Explanation

When the instruction limit is exceeded inside a heartbeat or timer callback, the IC replica traps the execution and rolls back all state mutations for that message. Because `distribute_rewards` updates the `latest_reward_event` only at the very end of the function (after all three loops), a trap at any point in the loops means:

- The `latest_reward_event` is **not** advanced.
- Proposals remain in `ReadyToSettle` state indefinitely (ballots are never cleared).
- Neuron maturities are never incremented.
- Every subsequent heartbeat re-enters the same code path, hits the same instruction limit, and traps again — a **permanent liveness failure** for reward distribution.

Accumulated undistributed maturity represents real economic value (voting rewards) that is permanently inaccessible to neuron holders.

### Likelihood Explanation

SNS Governance allows a configurable `max_number_of_neurons`. A popular SNS DAO with many active neuron holders who vote on proposals will naturally accumulate a large voter set. The ballot aggregation loop (step 2 above) is O(proposals × voters), making it especially expensive when multiple proposals settle in the same reward round. This is a realistic operational scenario, not a contrived attack — it mirrors exactly the Protocol Guild scenario described in the external report (large participant sets in normal use). No privileged access is required; any user can create a neuron and vote.

### Recommendation

Apply the same fix used in NNS Governance:

1. Move the maturity-increment work out of `distribute_rewards` into a separate timer task.
2. Introduce a `RewardsDistributionStateMachine` equivalent for SNS that persists the pending per-neuron reward map in stable memory.
3. In each timer invocation, process neurons in a loop that checks `is_message_over_threshold` after each neuron and breaks when the limit is approached, resuming in the next timer tick.
4. Record the `latest_reward_event` and settle proposals immediately after calculation (before the per-neuron distribution loop), so that even if distribution takes multiple messages, the reward round is not re-processed. [5](#0-4) [6](#0-5) 

### Proof of Concept

The following describes the reachable execution path:

1. An SNS is deployed with a large number of neurons (e.g., 10,000 neuron holders, achievable with a liquid democracy token distribution).
2. A governance proposal reaches `ReadyToSettle` state.
3. The SNS Governance heartbeat fires and calls `run_periodic_tasks`.
4. `should_distribute_rewards` returns `true`; `total_supply()` is awaited successfully.
5. `distribute_rewards(supply)` is called synchronously.
6. The ballot aggregation loop at line 5896 iterates over all 10,000 ballots. The maturity distribution loop at line 5954 iterates over all 10,000 unique voters. Combined with `get_neuron_result_mut` stable-memory lookups per neuron, the instruction count exceeds the ~40 billion instruction per-message limit.
7. The replica traps the heartbeat; all state is rolled back.
8. `latest_reward_event` is unchanged; the proposal remains `ReadyToSettle`.
9. Every subsequent heartbeat repeats steps 5–8 indefinitely.
10. Voting rewards for all SNS neuron holders are permanently lost. [7](#0-6) [8](#0-7)

    fn continue_processing(
        &mut self,
        neuron_store: &mut NeuronStore,
        is_over_instructions_limit: fn() -> bool,
    ) {
        while let Some((id, reward_e8s)) = self.rewards.pop_first() {
            match neuron_store.with_neuron_mut(&id, |neuron| {
                let auto_stake = neuron.auto_stake_maturity.unwrap_or(false);
                if auto_stake {
                    neuron.staked_maturity_e8s_equivalent = Some(
                        neuron
                            .staked_maturity_e8s_equivalent
                            .unwrap_or_default()
                            .saturating_add(reward_e8s),
                    );
                } else {
                    neuron.maturity_e8s_equivalent =
                        neuron.maturity_e8s_equivalent.saturating_add(reward_e8s);
                }

                Ok(_) => {}

                    This should not be possible as neuron existence is checked when \
                    rewards are calculated: {}",
                        LOG_PREFIX, id, e
                    );
                }
            };
            if is_over_instructions_limit() {
                break;
            }
