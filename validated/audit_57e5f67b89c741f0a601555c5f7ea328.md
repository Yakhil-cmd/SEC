### Title
Rebasing NEP-141 Tokens Bridged to Aurora Permanently Lock Accrued Yield in the Engine Contract — (`engine/src/engine.rs`)

---

### Summary

When a rebasing NEP-141 token (one whose on-chain balance increases automatically over time, such as a liquid-staking derivative) is bridged to Aurora, the Aurora Engine contract holds the NEP-141 tokens on NEAR. Because the engine uses strict 1:1 internal accounting — minting exactly the deposited amount of ERC-20 tokens and releasing exactly the burned amount on exit — any yield that accrues to the engine's NEP-141 balance through automatic rebasing is permanently trapped. There is no mechanism in the engine to claim or redistribute this surplus.

---

### Finding Description

The bridge flow for NEP-141 → ERC-20 is:

1. A user calls `ft_transfer_call` on the NEP-141 contract, which triggers `ft_on_transfer` on the Aurora Engine.
2. `ft_on_transfer` calls `receive_erc20_tokens`, which mints exactly `args.amount` ERC-20 tokens to the recipient. [1](#0-0) 

3. The NEP-141 tokens are now custodied by the Aurora Engine contract on NEAR.

The exit flow for ERC-20 → NEP-141 is:

4. The user calls `withdrawToNear` on the `EvmErc20` contract, which burns the ERC-20 tokens and calls the `ExitToNear` precompile. [2](#0-1) 

5. `exit_erc20_token_to_near` constructs an `ft_transfer` call to the NEP-141 contract for exactly `exit_params.amount` — the burned ERC-20 amount. [3](#0-2) 

**The gap**: If the NEP-141 token is rebasing, the Aurora Engine contract's NEP-141 balance grows passively over time. For example:

- User bridges 100 stNEAR → engine mints 100 ERC-20 stNEAR.
- Staking rewards rebase the engine's NEP-141 balance to 110 stNEAR.
- ERC-20 total supply on Aurora remains 100.
- User exits: burns 100 ERC-20 → receives 100 stNEAR.
- The extra 10 stNEAR is permanently locked in the Aurora Engine contract.

There is no admin rescue function, no `ft_transfer` call that can be made by the owner to recover excess NEP-141 balance, and no mechanism to mint additional ERC-20 tokens to represent the accrued yield. The entire `connector.rs` module only exposes flows that are strictly amount-in = amount-out. [4](#0-3) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

All yield accrued by rebasing NEP-141 tokens held in the Aurora Engine contract is permanently inaccessible. No user, no admin, and no contract can retrieve it. The yield is silently destroyed from the perspective of all bridge participants. This matches the allowed impact scope of "theft of unclaimed yield."

---

### Likelihood Explanation

**Medium.**

Rebasing NEP-141 tokens exist on NEAR today (e.g., liquid staking derivatives that auto-compound staking rewards). The bridge is permissionless — any user can call `deploy_erc20_token` for any NEP-141 and then bridge tokens via `ft_transfer_call`. No special privilege is required to trigger the condition. The yield loss begins accumulating silently from the moment the first rebasing token is bridged, and grows monotonically with time and bridge TVL.

---

### Recommendation

1. **Detect rebasing tokens at registration time** and either reject them or flag them for special handling.
2. **Add a yield-claim function** (owner-gated) that reads the actual NEP-141 balance held by the engine, computes the surplus over the ERC-20 total supply, and distributes or burns the surplus appropriately.
3. Alternatively, **adopt a share-based accounting model** (similar to ERC-4626 vaults) for the ERC-20 mirror, so that the ERC-20 balance of each holder reflects their proportional claim on the engine's actual NEP-141 balance rather than a fixed nominal amount.

---

### Proof of Concept

1. Deploy a rebasing NEP-141 token `rebase.near` whose `ft_balance_of(aurora)` increases by 10% every epoch.
2. Call `deploy_erc20_token` on Aurora for `rebase.near`.
3. User A calls `ft_transfer_call` on `rebase.near` with `amount = 1000`, bridging 1000 tokens → Aurora mints 1000 ERC-20 tokens. [5](#0-4) 

4. Wait one epoch. `rebase.near`'s `ft_balance_of(aurora)` is now 1100. ERC-20 total supply is still 1000.
5. User A calls `withdrawToNear(recipient, 1000)` on the ERC-20 contract → burns 1000 ERC-20 → `ft_transfer(recipient, 1000)` is issued to `rebase.near`. [6](#0-5) 

6. User A receives 1000 tokens. The 100-token yield surplus remains in `aurora`'s NEP-141 balance with no ERC-20 counterpart and no callable path to retrieve it. [2](#0-1)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
