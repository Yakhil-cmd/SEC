Audit Report

## Title
Unauthenticated `buyer` Override in `refresh_buyer_tokens` Bypasses Confirmation-Text Consent Gate and Forces Swap Participation - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal without verifying that the caller is that principal. Any unprivileged ingress sender can supply a victim's principal as `buyer` along with the publicly readable `confirmation_text`, causing the swap canister to register the victim as a confirmed participant — bypassing the SNS-configured consent gate — and locking the victim's ICP in the swap without their explicit agreement. If the swap commits, the victim receives SNS tokens they never consented to purchase.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 130–134, the effective buyer principal is resolved as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
```

When `arg.buyer` is non-empty, the caller's identity is completely ignored and the supplied principal is used verbatim. No check that `arg.buyer == caller_principal_id()` exists. The resolved principal `p` is passed directly to `refresh_buyer_token_e8s` at line 137.

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs`, line 1150), `validate_confirmation_text(confirmation_text)` is called with the attacker-supplied text. The `confirmation_text` is publicly readable from the swap state via `get_state`, so any attacker trivially knows it. The function then reads the ICP balance of `Account { owner: swap_canister, subaccount: principal_to_subaccount(&buyer) }` (lines 1153–1163), and if it meets the minimum, inserts the victim into `self.buyers` (lines 1285–1288).

Once registered, `error_refund_icp` (`rs/sns/swap/src/swap.rs`, lines 1950–1959) blocks any error-refund path for the victim while `transfer_success_timestamp_seconds == 0`. When the swap commits, the normal `sweep_icp` path (lines 2083–2088) transfers the victim's ICP to SNS governance and the victim receives SNS tokens. There is no endpoint to cancel or withdraw participation once registered in `self.buyers`.

The proto definition (`rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto`, lines 843–851) confirms the `buyer` field is caller-supplied with no authentication requirement. The design comment at `swap.rs` line 1123 explicitly states "the caller of this function must accept this confirmation" — an invariant the canister entry point fails to enforce.

## Impact Explanation
This is a **High** severity SNS security impact. An attacker forces a victim's ICP into a committed swap without the victim's consent, causing the victim to receive SNS tokens (a potentially illiquid or unwanted asset) in exchange for their ICP. The `confirmation_text` mechanism — the only explicit user-agreement gate in the participation flow — is completely bypassed. This constitutes unauthorized commitment of user funds and a broken governance/authorization assumption in the SNS framework, matching the allowed impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
High. The only precondition is that the victim has transferred ICP to the swap canister's subaccount, which is a normal on-chain action visible on the public ICP ledger. The attacker requires no special privileges, no private keys, and no admin access. The `confirmation_text` is publicly readable from the swap state. Any ingress sender can exploit this immediately after observing the victim's ledger transfer. The attack is repeatable for every victim who transfers ICP before calling `refresh_buyer_tokens` themselves.

## Recommendation
Add a caller-identity check before accepting a non-empty `buyer` field in `rs/sns/swap/canister/canister.rs`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let supplied = PrincipalId::from_str(&arg.buyer).unwrap();
    if supplied != caller_principal_id() {
        panic!("buyer must match caller");
    }
    supplied
};
```

Alternatively, remove the `buyer` field entirely and always derive the principal from `caller_principal_id()`. The `buyer` field exists as a convenience (e.g., for tooling that passes the principal explicitly), but it must not allow impersonation.

## Proof of Concept
1. Alice transfers 5 ICP to `Account { owner: swap_canister, subaccount: principal_to_subaccount(alice) }` on the ICP ledger, intending to evaluate participation but not yet ready to commit.
2. The SNS swap has `confirmation_text = Some("I agree to the terms")` (publicly readable via `get_state`).
3. Attacker Bob calls `refresh_buyer_tokens({ buyer: alice.to_string(), confirmation_text: Some("I agree to the terms") })` from his own principal.
4. The swap canister resolves `p = alice` (line 133), reads Alice's subaccount balance (5 ICP ≥ `min_participant_icp_e8s`), validates the confirmation text (matches at line 1150), and inserts Alice into `self.buyers` with `amount_icp_e8s = 5 * E8` (lines 1285–1288).
5. Alice calls `error_refund_icp({ source_principal_id: alice })` — the canister returns `"ICP cannot be refunded as principal alice has 500000000 ICP (e8s) in escrow"` (lines 1950–1959).
6. The swap commits; `sweep_icp` transfers Alice's ICP to SNS governance (lines 2083–2088); Alice receives SNS tokens she never consented to purchase.

A deterministic integration test using `PocketIC` or `StateMachine` can reproduce this by: (a) creating a swap with `confirmation_text` set, (b) transferring ICP from Alice's principal to her swap subaccount, (c) calling `refresh_buyer_tokens` from Bob's identity with `buyer = alice`, and (d) asserting Alice appears in `get_buyers_total` and that `error_refund_icp` returns a precondition error for Alice.
