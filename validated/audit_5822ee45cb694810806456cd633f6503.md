### Title
`notify_l2_asset_tracker` Fatal-Error-on-Revert Permanently Halts Chain for All In-Flight L1 Deposits - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function, called during every L1→L2 deposit transaction, treats any revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a fatal block-level error that immediately halts chain processing. Because L1 priority-queue transactions cannot be skipped or invalidated, any state in `L2AssetTracker` that causes this call to revert permanently bricks the chain with no built-in recovery path. This is the direct ZKsync OS analog to the `whenNotPaused` guard on bridge receiver functions.

---

### Finding Description

During L1→L2 deposit processing, `process_l1_transaction` calls `mint_base_token` up to three times (value mint, operator fee, refund). Each call internally invokes `notify_l2_asset_tracker`, which executes `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2(l1_chain_id, amount)` via `run_single_interaction`. [1](#0-0) 

If the EVM call returns a failed result, `notify_l2_asset_tracker` does not degrade gracefully — it immediately returns a fatal `internal_error!`: [2](#0-1) 

The code comment itself acknowledges the severity: *"Failure halts block processing — if the asset tracker reverts, the chain's token accounting would be inconsistent, so we treat it as fatal rather than silently continuing with incorrect bookkeeping."*

This fatal error propagates as `TxError::Internal` in the ZK transaction loop: [3](#0-2) 

A `TxError::Internal` causes `return Err(err)`, terminating block processing entirely. The test suite and documentation both confirm that L1 transactions **cannot be invalidated** because doing so would halt the chain via the priority queue: [4](#0-3) 

The `L2AssetTracker` contract is an upgradeable contract (`Initializable + OwnableUpgradeable`, slots 0–100) with access-control modifiers (`onlyBaseTokenHolderOrL2BaseToken`) and state-dependent logic (`isAssetRegistered`, `assetMigrationNumber`): [5](#0-4) 

Any of the following conditions causes `handleFinalizeBaseTokenBridgingOnL2` to revert and thus permanently halts the chain:
- A protocol upgrade to `L2AssetTracker` that introduces a new revert path (e.g., a new guard, a changed modifier, or a logic bug)
- The base token asset not being registered (`isAssetRegistered[assetId] == false`)
- An incorrect `assetMigrationNumber` state
- Any other revert condition in the contract's execution path

The `notify_l2_asset_tracker` call is triggered for **every** L1 deposit with `total_deposited > 0` or during simulation: [6](#0-5) 

The three call sites (value mint, operator fee, refund) are: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Any L1 deposit transaction (submitted by any user) that triggers `notify_l2_asset_tracker` while `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` is in a reverting state causes a fatal `TxError::Internal`. This terminates block processing. Because L1 priority-queue transactions cannot be skipped, the chain is permanently halted — all subsequent blocks fail to process. All in-flight L1 deposits are stuck with no automatic recovery mechanism. This is a complete chain halt, more severe than the "funds stuck" impact in the reference report.

---

### Likelihood Explanation

The trigger condition is any revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2`. This is a realistic scenario because:
1. `L2AssetTracker` is an upgradeable contract — a protocol upgrade could introduce a new revert path
2. The predeploy initialization must correctly set `isAssetRegistered`, `assetMigrationNumber`, `L1_CHAIN_ID`, and `BASE_TOKEN_ASSET_ID`; any misconfiguration causes a revert
3. The `L2AssetTracker` contract has multiple state-dependent checks that could fail under edge cases

The ZKsync OS code is the necessary vulnerable step: without the fatal-error-on-revert design in `notify_l2_asset_tracker`, the chain would not halt.

---

### Recommendation

Mirror the mitigation from the reference report: remove the hard fatal-error treatment for `L2AssetTracker` reverts in `notify_l2_asset_tracker`. Instead:

1. Log the failure and continue block processing (treating the asset tracker notification as best-effort)
2. Or, record failed notifications and allow an admin/operator to retry them in a subsequent block
3. At minimum, add a circuit-breaker that allows the chain to skip a failing L1 deposit rather than halting entirely, consistent with the existing resilience philosophy documented in `l1_tx_resilience.rs` [9](#0-8) 

---

### Proof of Concept

1. Deploy a modified `L2AssetTracker` (or configure its storage so `isAssetRegistered[BASE_TOKEN_ASSET_ID] == false`)
2. Submit any L1→L2 deposit transaction with `total_deposited > 0`
3. The bootloader calls `notify_l2_asset_tracker` → `handleFinalizeBaseTokenBridgingOnL2` reverts
4. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`
5. The tx loop receives `TxError::Internal` and executes `return Err(err)`, halting block processing
6. All subsequent L1 transactions in the priority queue cannot be processed; the chain is permanently stuck [10](#0-9) [3](#0-2)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
