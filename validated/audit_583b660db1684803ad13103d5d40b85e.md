### Title
TLS Certificate Verification Completely Disabled in Production P2P HTTP Client — (File: rs/canister_client/src/http_client.rs)

---

### Summary

The `HttpClient` in `rs/canister_client/src/http_client.rs` installs a custom Rustls verifier named `DangerAcceptInvalidCerts` that unconditionally returns `Ok(ServerCertVerified::assertion())` for every TLS certificate presented, regardless of issuer, validity, or signature. SNI is also explicitly disabled. This client is compiled into production replica code and is used by the P2P artifact downloader (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/download.rs`). Any on-path network attacker can silently MITM HTTPS connections made by this client without needing a compromised CA at all.

---

### Finding Description

`DangerAcceptInvalidCerts` implements `rustls::client::danger::ServerCertVerifier` and accepts every certificate unconditionally: [1](#0-0) 

This verifier is wired into the production `HttpClient` via `.dangerous().with_custom_certificate_verifier(Arc::new(DangerAcceptInvalidCerts {}))`, and SNI is disabled immediately after: [2](#0-1) 

The `HttpClient` is documented as the client used to communicate with replicas: [3](#0-2) 

The P2P artifact downloader (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/download.rs`) is a production replica component that calls `HttpClient::new` to fetch stripped artifacts from peer nodes over HTTPS. Because `DangerAcceptInvalidCerts` accepts any certificate, an on-path attacker can present a self-signed or otherwise invalid certificate and the client will complete the TLS handshake and proceed.

By contrast, the HTTPS outcalls adapter — which handles canister HTTP outcalls — correctly uses the system native root CA store: [4](#0-3) 

The node-to-node TLS layer also correctly pins certificates against the registry: [5](#0-4) 

The `HttpClient` path is the outlier: it performs no verification whatsoever.

---

### Impact Explanation

An on-path network attacker (e.g., a compromised upstream router, a malicious ISP, or an attacker with access to the same data-center fabric) can intercept HTTPS connections made by the P2P artifact downloader and serve arbitrary content. Because TLS provides no authentication here, the attacker does not need to compromise any CA. The attacker can:

1. Intercept the TCP connection from a replica to a peer node's HTTPS endpoint.
2. Present any TLS certificate (self-signed, expired, wrong CN — all accepted).
3. Serve a crafted artifact payload in place of the legitimate one.

If the artifact content is not independently authenticated after download (e.g., via a separate signature check over the artifact bytes), the replica will process the attacker-controlled data. This maps to the original report's vulnerability class: **data exposure / MITM due to absent certificate pinning**, but is strictly worse because no CA compromise is required at all.

---

### Likelihood Explanation

- **Attacker capability required**: A privileged network position between two IC nodes (on-path). This does not require a compromised CA, a BGP-level hijack, or any IC-internal privilege.
- **No CA compromise needed**: Unlike the original report's scenario (compromised Indian CCA), here the attacker needs only to intercept the TCP stream; any certificate is accepted.
- **Scope**: The P2P artifact downloader runs on every replica node. The attack surface is every HTTPS connection it makes to peer nodes.
- **Mitigating factor**: If the IC independently verifies artifact integrity (e.g., via hash or signature checks after download), the impact of serving a malicious artifact body is reduced. However, the TLS bypass itself is unconditional and architectural.

---

### Recommendation

Replace `DangerAcceptInvalidCerts` with a proper certificate verifier. For connections to known IC replica nodes, pin against the registry-published TLS certificates using the same `NodeServerCertVerifier` pattern already used for node-to-node TLS: [6](#0-5) 

At minimum, use the system native root CA store (as the HTTPS outcalls adapter does) rather than accepting all certificates. Also re-enable SNI, which was explicitly disabled alongside the certificate bypass.

---

### Proof of Concept

```
1. Identify a replica node R that is downloading artifacts via HttpClient.
2. Position yourself on-path between R and a peer node P (e.g., via ARP spoofing
   on the same L2 segment, or a compromised upstream router).
3. Intercept the TCP connection from R to P's HTTPS port.
4. Complete the TLS handshake presenting any certificate (e.g., a fresh
   self-signed cert for an arbitrary CN).
5. DangerAcceptInvalidCerts::verify_server_cert() returns Ok(...) unconditionally.
6. The TLS session is established. Serve a crafted artifact body.
7. R processes the attacker-controlled artifact content.
```
