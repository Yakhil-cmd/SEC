### Title
Missing Duplicate Blob Hash Check Allows Invalid EIP-4844 Transactions to Pass Validation - (`File: basic_bootloader/src/bootloader/transaction/blobs.rs`)

---

### Summary

`parse_blobs_list` pushes each blob versioned hash into a result array without checking for duplicates. EIP-4844 mandates that all versioned hashes in a transaction be unique. ZKsync OS accepts transactions with duplicate blob hashes that Ethereum would reject, producing a state-transition divergence and incorrect blob gas accounting.

---

### Finding Description

`parse_blobs_list` in `basic_bootloader/src/bootloader/transaction/blobs.rs` iterates over the raw `BlobHashesList` from an EIP-4844 transaction and unconditionally pushes each hash into the result `ArrayVec`:

```rust
// basic_bootloader/src/bootloader/transaction/blobs.rs
for blob_hash in blobs_list.iter() {
    let blob_hash = blob_hash?;
    if blob_hash[0] != VERSIONED_HASH_VERSION_KZG {
        return Err(TxError::Validation(
            InvalidTransaction::BlobElementIsNotSupported,
        ));
    }
    // NOTE: we do NOT check that this blob hash is meaningful ...
    let blob_hash = Bytes32::from_array(*blob_hash);
    result.push(blob_hash);   // ← no duplicate check
}
``` [1](#0-0) 

The only checks performed are:
1. Total count does not exceed `MAX_BLOBS_IN_TX`
2. The first byte of each hash equals `VERSIONED_HASH_VERSION_KZG`

No check is made for whether the same hash already appears in `result`.

This function is called from both Ethereum-path and ZK-path validation: [2](#0-1) [3](#0-2) 

The returned `blobs` vec is stored directly in transaction-level metadata and used for blob gas accounting:

```rust
// ethereum/validation_impl.rs
let blob_gas_used = (blobs.len() as u64) * GAS_PER_BLOB;
``` [4](#0-3) 

And in the ZK path:

```rust
let num_blobs = system.metadata.num_blobs();
let blob_gas_used = num_blobs as u64 * GAS_PER_BLOB;
``` [5](#0-4) 

The `blobs` vec is also stored in `EthereumTransactionMetadata` and used by the `BLOBHASH` opcode via `get_blob_hash(idx)`: [6](#0-5) 

The Ethereum execution specification (EIP-4844) requires:
```python
assert len(set(tx.blob_versioned_hashes)) == len(tx.blob_versioned_hashes)
```
This uniqueness invariant is never enforced in ZKsync OS.

---

### Impact Explanation

**EVM semantic mismatch / state-transition bug:**

1. **Invalid transactions accepted**: An EIP-4844 transaction with duplicate blob hashes (e.g., `[H, H]`) is rejected by Ethereum but accepted by ZKsync OS. This is a direct state-transition divergence.

2. **Incorrect blob gas accounting**: `blob_gas_used` is computed as `blobs.len() * GAS_PER_BLOB`. With `N` duplicate hashes, the user is charged for `N` blob slots. This inflated `blob_gas_used` also propagates into `block_blob_gas_used`, potentially consuming block blob gas capacity with phantom entries: [7](#0-6) 

3. **BLOBHASH opcode returns duplicate hashes**: The `BLOBHASH` opcode at index `i` returns `blobs[i]`. With duplicates, `BLOBHASH(0)` and `BLOBHASH(1)` return the same hash, which is semantically incorrect for a valid transaction (since the transaction should have been rejected before execution).

---

### Likelihood Explanation

Any unprivileged EIP-4844 transaction sender can craft a transaction with a repeated blob versioned hash (e.g., the same `0x01...` hash twice). No special privileges, oracle manipulation, or governance access is required. The attacker simply constructs a type-3 transaction with `blob_versioned_hashes: [H, H]` and submits it. The transaction passes all current validation checks in `parse_blobs_list`.

---

### Recommendation

Add a duplicate check inside `parse_blobs_list` before pushing each hash:

```rust
pub fn parse_blobs_list<const MAX_BLOBS_IN_TX: usize>(
    blobs_list: BlobHashesList<'_>,
) -> Result<arrayvec::ArrayVec<Bytes32, MAX_BLOBS_IN_TX>, TxError> {
    let mut result = arrayvec::ArrayVec::<_, MAX_BLOBS_IN_TX>::new();
    if blobs_list.count > MAX_BLOBS_IN_TX {
        return Err(TxError::Validation(InvalidTransaction::BlobListTooLong));
    }

    for blob_hash in blobs_list.iter() {
        let blob_hash = blob_hash?;
        if blob_hash[0] != VERSIONED_HASH_VERSION_KZG {
            return Err(TxError::Validation(
                InvalidTransaction::BlobElementIsNotSupported,
            ));
        }
        let blob_hash = Bytes32::from_array(*blob_hash);
        // Enforce EIP-4844 uniqueness requirement
        if result.contains(&blob_hash) {
            return Err(TxError::Validation(InvalidTransaction::DuplicateBlobHash));
        }
        result.push(blob_hash);
    }
    // ...
}
```

A new `InvalidTransaction::DuplicateBlobHash` variant should be added to the error enum in `basic_bootloader/src/bootloader/errors.rs`.

---

### Proof of Concept

1. Craft an EIP-4844 transaction with `blob_versioned_hashes = [0x01AA...AA, 0x01AA...AA]` (same hash twice, both with valid `0x01` version byte).
2. Submit to ZKsync OS via the Ethereum transaction flow.
3. `parse_blobs_list` iterates both entries, finds no version byte error, and pushes both into `result` — returning a 2-element vec with identical hashes.
4. `blob_gas_used = 2 * GAS_PER_BLOB` is charged and `block_blob_gas_used` is incremented by `2 * GAS_PER_BLOB`.
5. The transaction executes successfully. `BLOBHASH(0)` and `BLOBHASH(1)` both return `0x01AA...AA`.
6. On Ethereum, this transaction would be rejected at the mempool/block-validation level with the uniqueness assertion failure. [8](#0-7) [9](#0-8)
