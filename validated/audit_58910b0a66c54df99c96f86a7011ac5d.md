Audit Report

## Title
Zero-Amount ETH Transaction Emitted When Gas Fee Equals Withdrawal Amount — (`rs/ethereum/cketh/minter/src/tx.rs`)

## Summary

The `resubmit` function in `tx.rs` uses a strict `>` guard instead of `>=` when comparing the new transaction fee against the allowed maximum. When `new_tx_price.max_transaction_fee() == withdrawal_amount` exactly, the guard does not fire, `checked_sub` returns `Some(0)`, and a zero-value ETH transaction is signed and broadcast. Because ckETH is burned at withdrawal time, the user permanently loses their funds while the recipient receives 0 ETH.

## Finding Description

**Root cause — `resubmit` guard (`rs/ethereum/cketh/minter/src/tx.rs`, lines 169–179):**

The guard at line 169 uses strict `>`:

```rust
if new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee() {
    return Err(ResubmitTransactionError::InsufficientTransactionFee { ... });
}
```

For `ReduceEthAmount`, `allowed_max_transaction_fee()` returns `withdrawal_amount` verbatim (line 139). When `new_fee == withdrawal_amount`, the condition is `false`, so no error is returned. Execution falls through to:

```rust
withdrawal_amount.checked_sub(new_tx_price.max_transaction_fee())
    .expect("BUG: ...")
```

`checked_sub` returns `Some(0)`, and the function returns `Ok(Some(tx_with_amount = 0))`.

**Downstream — `record_created_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 506–511):**

The only assertion on the transaction amount is:

```rust
assert!(req.withdrawal_amount > transaction.amount, ...);
```

This assertion passes when `transaction.amount == 0` because `withdrawal_amount > 0`. No further zero-amount check exists.

**Downstream — `create_transaction` path (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 1125–1134):**

The same off-by-one exists at initial transaction creation: `checked_sub` returns `Some(0)` when `max_transaction_fee == withdrawal_amount`, and the result is used as `tx_amount` with no zero-value guard.

**Downstream — `resubmit_transactions_batch` (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 225–246):**

The zero-amount transaction is recorded as a `ReplacedTransaction` event with no further validation and proceeds through signing and sending.

**Exploit path:**
1. User calls `withdraw_eth` (unprivileged public update call); ckETH is burned.
2. Gas prices spike such that `max_fee_per_gas * gas_limit == withdrawal_amount` exactly.
3. `resubmit` is called; the `>` guard does not fire; `new_amount = 0`.
4. A zero-value ETH transaction is signed and broadcast.
5. The Ethereum transaction succeeds (zero-value transfers are valid EIP-1559 transactions).
6. Recipient receives 0 ETH; user's ckETH is already burned; no reimbursement is triggered.

## Impact Explanation

Permanent, irreversible loss of user ckETH funds with no reimbursement path. This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."** The loss is bounded per-transaction (one user's withdrawal amount) rather than systemic pool drain, placing this at High rather than Critical.

## Likelihood Explanation

The condition requires `max_fee_per_gas * 21_000 == withdrawal_amount` exactly. For the current minimum withdrawal of `5_000_000_000_000_000` wei (0.005 ETH), this requires ~238 Gwei — extreme by current mainnet standards but historically reached during severe congestion. The condition is exact equality, making it a narrow edge case. However, it is triggerable by any unprivileged user who happens to submit a withdrawal at the precise moment gas prices reach this threshold, with no attacker action required beyond normal usage.

## Recommendation

Change the strict `>` to `>=` in `resubmit` (`rs/ethereum/cketh/minter/src/tx.rs`, line 169):

```rust
// Before (buggy):
if new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee() {

// After (correct):
if new_tx_price.max_transaction_fee() >= self.resubmission.allowed_max_transaction_fee() {
```

Apply the same fix to `create_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, line 1125): treat `tx_amount == Wei::ZERO` as an `InsufficientTransactionFee` error, or assert `tx_amount > Wei::ZERO` before constructing the transaction.

## Proof of Concept

```
withdrawal_amount   = 4_200_000_000_000_000 wei
gas_limit           = 21_000
max_fee_per_gas     = 200_000_000_000 wei  (200 Gwei)
max_transaction_fee = 200_000_000_000 * 21_000 = 4_200_000_000_000_000 wei

Guard (line 169): 4_200_000_000_000_000 > 4_200_000_000_000_000 → FALSE (no error returned)
new_amount = 4_200_000_000_000_000 - 4_200_000_000_000_000 = 0

→ resubmit returns Ok(Some(tx { amount: 0 }))
→ record_created_transaction assertion (line 509): withdrawal_amount(>0) > 0 → passes
→ Zero-amount transaction signed and broadcast
→ User's ckETH already burned; recipient receives 0 ETH; no reimbursement triggered
```

A deterministic unit test can be written against `SignedTransactionRequest::resubmit` with a `GasFeeEstimate` constructed so that `max_fee_per_gas * gas_limit == withdrawal_amount`, asserting that the result is `Err(InsufficientTransactionFee)` rather than `Ok(Some(tx_with_amount_zero))`.
