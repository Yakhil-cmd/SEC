Audit Report

## Title
Unbounded Correlated SQL Subquery in `get_aggregated_balance_for_principal_at_block_idx` Enables Single-Request DoS of ICRC1 Rosetta Instance — (`rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs`)

## Summary

An unauthenticated HTTP client can send a single `POST /account/balance` request with `metadata.aggregate_all_subaccounts=true` targeting a principal with many subaccounts. This triggers an unbounded correlated SQL subquery with O(N×M) complexity, no `LIMIT`, and no timeout. Because `StorageClient` wraps a single `tokio_rusqlite::Connection` that serializes all SQLite access through one background thread, the long-running query blocks every subsequent Rosetta request for its entire duration, rendering the instance completely unresponsive.

## Finding Description

**Entry point:** `account_balance_with_metadata` in `rs/rosetta-api/icrc1/src/data_api/services.rs` reads `aggregate_all_subaccounts` directly from user-supplied JSON metadata with no authentication, rate-limiting, or authorization check. [1](#0-0) 

The only guard rejects requests where a *non-default* subaccount is also specified. Any request with `subaccount=None` or `subaccount=[0u8;32]` passes through unconditionally. [2](#0-1) 

Execution then falls through to `get_aggregated_balance_for_principal_at_block_idx` with no further constraints. [3](#0-2) 

**Unbounded SQL query:** The function issues a correlated subquery that, for every row in `account_balances` matching the principal, fires a nested `SELECT MAX(block_idx)` re-scanning all rows for that `(principal, subaccount)` pair. There is no `LIMIT`, no `OFFSET`, no query timeout, and no cap on subaccounts processed. Complexity is O(N×M) where N = distinct subaccounts and M = average balance-change events per subaccount. [4](#0-3) 

**Blocking effect:** `StorageClient` holds a single `tokio_rusqlite::Connection`. The `tokio_rusqlite` crate serializes all database calls through a single background thread. A long-running query therefore blocks every other storage operation (balance lookups, block fetches, transaction searches) for its entire duration. [5](#0-4) 

No rate limiting, `busy_timeout`, or statement timeout is configured anywhere in the ICRC1 Rosetta storage layer.

## Impact Explanation

A single unauthenticated HTTP request can render the entire ICRC1 Rosetta instance unresponsive for the duration of the query. Exchanges and integrators depending on Rosetta for balance queries, transaction lookups, and construction flows are all affected. Repeated requests keep the instance permanently degraded. This matches the allowed High impact: **"Significant Rosetta... security impact with concrete user or protocol harm"** — specifically application-level DoS of a financial integration component with no raw volumetric component required.

## Likelihood Explanation

The endpoint requires no authentication. The precondition — a principal with a large number of distinct subaccounts — is realistic: DEXes, custodians, and exchanges routinely use per-user subaccounts under a single principal, and ICRC1 tokens with low or zero fees make it cheap to manufacture millions of subaccounts. An attacker does not need to create the subaccounts themselves; they only need to identify an existing high-subaccount principal and query it repeatedly. The attack is trivially repeatable with a single `curl` command.

## Recommendation

1. **Add a hard cap** on the number of subaccounts aggregated per request (e.g., 10,000), returning an error if the principal exceeds it.
2. **Rewrite the query** to use `GROUP BY (principal, subaccount)` with `MAX(block_idx)` instead of a correlated subquery, and add `LIMIT :max_subaccounts`.
3. **Configure a SQLite statement timeout** (via `progress_handler` or `interrupt`) so runaway queries are cancelled rather than blocking indefinitely.
4. **Rate-limit** the `aggregate_all_subaccounts` path per source IP or per principal at the HTTP layer.

## Proof of Concept

```bash
# 1. Populate the Rosetta SQLite DB with a principal P having N distinct subaccounts
#    (each with at least one account_balances row), e.g. via a local integration test
#    that inserts synthetic blocks.

# 2. Send a single balance request:
curl -X POST http://rosetta-host:8080/account/balance \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<ledger_id>"},
    "account_identifier": {"address":"<principal_P>"},
    "metadata": {"aggregate_all_subaccounts": true}
  }'

# 3. Concurrently send any other Rosetta request (e.g. /block) and observe it hangs
#    for the full duration of the aggregation query.
# 4. Repeat step 2 to keep the instance permanently degraded.
```

A deterministic integration test can be written by inserting synthetic `account_balances` rows for a single principal across 10^5+ subaccounts into a test `StorageClient`, then asserting that a concurrent `get_block_with_highest_block_idx` call does not complete within a reasonable deadline while the aggregation query is running.

pub fn get_aggregated_balance_for_principal_at_block_idx(
    connection: &Connection,
    principal: &PrincipalId,
    block_idx: u64,
) -> anyhow::Result<Nat> {
    // Query to get the latest balance for each subaccount of the principal at or before the given block index
    let mut stmt = connection.prepare_cached(
        "SELECT a1.subaccount, a1.amount
         FROM account_balances a1
         WHERE a1.principal = :principal
         AND a1.block_idx = (
             SELECT MAX(a2.block_idx)
             FROM account_balances a2
             WHERE a2.principal = a1.principal
             AND a2.subaccount = a1.subaccount
             AND a2.block_idx <= :block_idx
         )",
    )?;

    let rows = stmt.query_map(
        named_params! {
            ":principal": principal.as_slice(),
            ":block_idx": block_idx
        },
        |row| {
            let amount_str: String = row.get(1)?;
            Nat::from_str(&amount_str).map_err(|_| {
                rusqlite::Error::InvalidColumnType(
                    1,
                    "amount".to_string(),
                    rusqlite::types::Type::Text,
                )
            })
        },
    )?;

    let mut total_balance = Nat(BigUint::zero());
    for balance_result in rows {
        let balance = balance_result?;
        total_balance = Nat(total_balance
            .0
            .checked_add(&balance.0)
            .with_context(|| "Overflow while aggregating balances")?);
    }

    Ok(total_balance)
}
