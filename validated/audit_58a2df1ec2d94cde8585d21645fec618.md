Audit Report

## Title
SNS Governance `distribute_rewards` Settles Proposals Belonging to the Next Reward Round — (`File: rs/sns/governance/src/governance.rs`)

## Summary
`distribute_rewards` in SNS governance collects proposals to settle by calling `ready_to_be_settled_proposal_ids()`, which evaluates reward status against `now` (the live wall-clock time). The reward event it creates covers only up to `reward_event_end_timestamp_seconds`, which is always ≤ `now`. Any proposal whose voting deadline falls in the gap `(reward_event_end_timestamp_seconds, now]` is `ReadyToSettle` as of `now` but belongs to the next reward round; it is incorrectly swept into the current event, its ballots are permanently cleared, and maturity is credited to voters one full round early. The NNS governance avoids this exact issue by passing a round-boundary-truncated timestamp to its equivalent function.

## Finding Description
`ready_to_be_settled_proposal_ids` in SNS governance captures `now` internally and returns every proposal whose `reward_status(now) == ReadyToSettle`: [1](#0-0) 

`distribute_rewards` calls this function with no timestamp argument, then computes `reward_event_end_timestamp_seconds` as `reward_start + new_rounds_count * round_duration`, which is always ≤ `now`: [2](#0-1) [3](#0-2) 

The gap `now − reward_event_end_timestamp_seconds` equals `(now − reward_start) mod round_duration_seconds`, which can be anywhere from 0 to `round_duration_seconds − 1` seconds. Any proposal whose deadline falls in this gap is `ReadyToSettle` as of `now` but belongs to the next round. It is included in `considered_proposals`, settled under the current reward event, and its ballots are permanently cleared: [4](#0-3) 

The NNS governance correctly avoids this by accepting an explicit `as_of_timestamp_seconds` parameter in `ready_to_be_settled_proposal_ids`: [5](#0-4) 

And passing the round-boundary-truncated timestamp computed by `most_recent_fully_elapsed_reward_round_end_timestamp_seconds`: [6](#0-5) [7](#0-6) 

SNS governance has no equivalent guard. The `reward_status` function in SNS proposal logic takes `now_seconds` as a parameter and returns `ReadyToSettle` for any proposal whose voting period has closed: [8](#0-7) 

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Significant SNS security impact with concrete user or protocol harm."*

1. **Incorrect maturity distribution**: Neurons that voted on proposals closing just after the round boundary receive maturity one full round early, before other participants have had a chance to vote on those proposals in their proper round.
2. **Reward dilution**: The rewards purse is sized for `new_rounds_count` rounds. Settling extra next-round proposals spreads that fixed purse across more voting-power shares, reducing per-neuron maturity for legitimate current-round voters.
3. **Permanent ballot data loss**: Once settled, `p.ballots.clear()` is called irreversibly. Proposals settled one round early lose their ballot data permanently with no recourse. [9](#0-8) 

## Likelihood Explanation
This is a deterministic, structural bug that fires on every invocation of `distribute_rewards` whenever at least one proposal's voting deadline falls in the window `(reward_event_end_timestamp_seconds, now]`. Because `should_distribute_rewards` fires as soon as `seconds_since_last_reward_event > round_duration_seconds`, and proposals can close at any second, this window is non-empty in virtually every real reward distribution cycle. [10](#0-9) 

No special privilege is required. Any SNS governance participant who can submit a proposal (or observe existing proposal deadlines) can time a proposal to close in this window. The exploit is passive — it fires automatically on the periodic timer task.

## Recommendation
Mirror the NNS approach in SNS governance:

1. Add an `as_of_timestamp_seconds: u64` parameter to `ready_to_be_settled_proposal_ids` in `rs/sns/governance/src/governance.rs`.
2. Compute the round-boundary timestamp analogous to NNS `most_recent_fully_elapsed_reward_round_end_timestamp_seconds` — i.e., `reward_event_end_timestamp_seconds` (already computed at line 5837) — and pass it to `ready_to_be_settled_proposal_ids` instead of using `now` internally.
3. This ensures only proposals whose voting period closed on or before the end of the most recently completed reward round are settled in the current event.

## Proof of Concept
**Setup:**
- SNS with `round_duration_seconds = 604800` (7 days).
- Last reward event `end_timestamp_seconds = T₀`.
- Neuron A creates Proposal P with `initial_voting_period_seconds = 604801` at time `T₀ − 1`.
- Proposal P's deadline = `T₀ − 1 + 604801 = T₀ + 604800` (1 second into the next round).

**Execution:**
1. At `T₀ + 604800 + 2`, the governance timer fires `should_distribute_rewards` → `true`.
2. `distribute_rewards` runs: `new_rounds_count = floor(604802 / 604800) = 1`, `reward_event_end_timestamp_seconds = T₀ + 604800`.
3. `ready_to_be_settled_proposal_ids()` evaluates `reward_status(now = T₀ + 604800 + 2)`: Proposal P's deadline `T₀ + 604800 < now` → `accepts_vote = false` → `ReadyToSettle`.
4. Proposal P is included in `considered_proposals` and settled under the current reward event whose `end_timestamp_seconds = T₀ + 604800`.
5. Neuron A receives full proportional maturity for Proposal P in round 1, even though P closed 1 second into round 2. Ballots are cleared permanently.
6. In round 2, Proposal P is already `Settled`; no other neuron can earn rewards for it.

A deterministic integration test using PocketIC can reproduce this by: setting up an SNS, creating a proposal timed to close just after the round boundary, advancing the IC time to `reward_event_end_timestamp_seconds + 2`, triggering `distribute_rewards`, and asserting that the proposal's `reward_event_end_timestamp_seconds` equals the current round's end (not the next round's end) and that ballots have been cleared.

    pub fn ready_to_be_settled_proposal_ids(
        &self,
        as_of_timestamp_seconds: u64,
    ) -> impl Iterator<Item = ProposalId> + '_ {
        self.heap_data
            .proposals
            .iter()
            .filter(move |(_, proposal)| {
                let topic = proposal.topic();
                let voting_period_seconds = self.voting_period_seconds()(topic);
                let reward_status =
                    proposal.reward_status(as_of_timestamp_seconds, voting_period_seconds);

                reward_status == ProposalRewardStatus::ReadyToSettle
            })
            .map(|(k, _)| ProposalId { id: *k })
    }

    fn most_recent_fully_elapsed_reward_round_end_timestamp_seconds(&self) -> u64 {
        let now = self.env.now();
        let genesis_timestamp_seconds = self.heap_data.genesis_timestamp_seconds;

        if genesis_timestamp_seconds > now {
            println!(
                "{}WARNING: genesis is in the future: {} vs. now = {})",
                LOG_PREFIX, genesis_timestamp_seconds, now,
            );
            return 0;
        }

        (now - genesis_timestamp_seconds) // Duration since genesis (in seconds).
            / REWARD_DISTRIBUTION_PERIOD_SECONDS // This is where the truncation happens. Whole number of rounds.
            * REWARD_DISTRIBUTION_PERIOD_SECONDS // Convert back into seconds.
            + self.heap_data.genesis_timestamp_seconds // Convert from duration to back to instant.
    }
