### Title
Silent Success on Call to Undeployed `L2AssetTracker` Causes Permanent Deposit Accounting Divergence - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

`notify_l2_asset_tracker` calls `L2_ASSET_TRACKER_ADDRESS` (`0x1000f`) without checking whether code is deployed there. Per EVM semantics, a call to an empty account always succeeds with empty returndata. The bootloader treats this silent success as a valid notification and proceeds to transfer real base tokens from the treasury. The `L2AssetTracker` contract never records the deposit, creating a permanent divergence between actual token movements and the contract's accounting state.

### Finding Description

In `process_l1_transaction`, every L1→L2 deposit with `total_deposited > 0` triggers `mint_base_token`, which first calls `notify_l2_asset_tracker` and then `transfer_from_treasury`. [1](#0-0) 

`notify_l2_asset_tracker` dispatches `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` to `L2_ASSET_TRACKER_ADDRESS` via `run_single_interaction`. The function only checks `asset_tracker_result.failed()` — it does **not** check whether code exists at the callee address before interpreting the result. [2](#0-1) 

The code itself documents this gap:

> "If no contract is deployed at L2AssetTracker, the call succeeds silently (a call to an empty address returns success with no returndata in EVM). However, we are certain that L2AssetTracker is available after the upgrade." [3](#0-2) 

Additionally, `read_l1_chain_id` reads `L1_CHAIN_ID` from `L2_ASSET_TRACKER_ADDRESS` storage slot 154. When the contract is not deployed, this slot returns `Bytes32::ZERO`, so `l1_chain_id = 0` is passed into every `handleFinalizeBaseTokenBridgingOnL2` call during the pre-upgrade window. [4](#0-3) 

After `notify_l2_asset_tracker` returns `Ok(())` (silently, because the callee is empty), `transfer_from_treasury` unconditionally deducts from the treasury and credits the recipient — real tokens move. [5](#0-4) 

`L2_ASSET_TRACKER_ADDRESS` is defined as `B160::from_limbs([0x1000f, 0, 0])` — it is not a system hook but a regular EVM contract address that must be deployed via upgrade. [6](#0-5) 

The test rig pre-deploys it via `install_default_predeployed_contracts`, but this is a test-only guarantee; production chains have a window between genesis and the upgrade transaction where the address is empty. [7](#0-6) 

### Impact Explanation

Every L1→L2 deposit that executes before `L2AssetTracker` is deployed at `0x1000f`:

1. Transfers real base tokens from the treasury to the recipient (correct).
2. Calls `handleFinalizeBaseTokenBridgingOnL2(0, amount)` on an empty address — succeeds silently.
3. Leaves `L2AssetTracker.interopInfo[BASE_TOKEN_ASSET_ID].totalSuccessfulDepositsFromL1` permanently at zero for those deposits.

When `L2AssetTracker` is later deployed and initialized, it starts with an under-counted `totalSuccessfulDepositsFromL1`. Any bridge/interop logic that relies on this counter as a source of truth for total bridged supply (e.g., to enforce caps, compute migration numbers, or validate withdrawal proofs) will operate on incorrect data for the lifetime of the chain. This is a permanent, non-correctable state divergence between actual token movements and the contract's accounting.

### Likelihood Explanation

The vulnerable window is the period between chain genesis and the governance upgrade that deploys `L2AssetTracker`. Any unprivileged user who submits an L1→L2 priority transaction with `total_deposited > 0` during this window triggers the bug. No special access is required; submitting a standard L1→L2 deposit is the entry path. On a newly launched ZKsync OS chain, this window is realistic and may span multiple blocks or even days.

### Recommendation

1. In `notify_l2_asset_tracker`, check whether `L2_ASSET_TRACKER_ADDRESS` has deployed bytecode before treating a silent success as a valid notification. If no code is present, either skip the notification and log a warning, or revert with a clear error rather than silently proceeding.
2. In `read_l1_chain_id`, validate that the returned value is non-zero before using it. A zero chain ID indicates the contract is not yet initialized and should be treated as an error condition, not a valid chain ID.
3. Consider making `L2AssetTracker` a system hook (like `MINT_HOOK_ADDRESS`) so its presence is guaranteed at genesis rather than depending on a post-genesis upgrade.

### Proof of Concept

**Pre-conditions:** A ZKsync OS chain has been started but the governance upgrade deploying `L2AssetTracker` at `0x1000f` has not yet been executed. The treasury (`BASE_TOKEN_HOLDER_ADDRESS`) is funded.

**Steps:**

1. User submits an L1→L2 priority transaction with `total_deposited = 1_000_000` and `value = 500`.
2. `process_l1_transaction` calls `read_l1_chain_id` → reads storage slot 154 of `0x1000f` → returns `U256::ZERO` (no contract deployed).
3. `mint_base_token` is called with `amount = to_transfer`, `l1_chain_id = 0`.
4. `notify_l2_asset_tracker` encodes calldata for `handleFinalizeBaseTokenBridgingOnL2(0, amount)` and calls `0x1000f` via `run_single_interaction`.
5. `0x1000f` has no code → EVM returns success with empty returndata → `asset_tracker_result.failed()` returns `false`.
6. `notify_l2_asset_tracker` returns `Ok(())`.
7. `transfer_from_treasury` deducts `amount` from the treasury and credits the recipient. **Real tokens move.**
8. The same pattern repeats for the operator fee and refund payments.

**Result:** The recipient holds real base tokens. `L2AssetTracker` (once deployed) shows `totalSuccessfulDepositsFromL1 = 0` for all deposits that occurred before the upgrade, permanently diverging from the actual bridged token supply.

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
