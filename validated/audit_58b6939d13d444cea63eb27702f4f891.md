Audit Report

## Title
Unauthenticated `buyer` override in `refresh_buyer_tokens` allows forced swap enrollment and `error_refund_icp` DoS — (`rs/sns/swap/canister/canister.rs`)

## Summary
`refresh_buyer_tokens` accepts an arbitrary `buyer` principal in the request body and uses it without verifying it matches the caller. Any unprivileged principal can therefore register a victim as a swap participant by supplying the victim's principal in `arg.buyer`, provided the victim's swap subaccount holds sufficient ICP. This blocks the victim's `error_refund_icp` escape path and, if the swap commits, forces the victim's ICP into SNS governance in exchange for unwanted SNS neurons.

## Finding Description
In `canister.rs` lines 130–133, the effective buyer principal is resolved from `arg.buyer` when non-empty, with no `caller == buyer` guard:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // ← no auth check
};
```

`refresh_buyer_token_e8s` (`swap.rs` lines 1153–1163) then queries the ICP ledger balance of `principal_to_subaccount(&buyer)` — the subaccount derived from the supplied principal, not the caller. If the balance meets `min_participant_icp_e8s`, a `BuyerState` entry is created for that principal (`swap.rs` lines 1285–1288) with `transfer_success_timestamp_seconds = 0` (as set by `BuyerState::new`, `types.rs` lines 540–551).

After the swap closes, `error_refund_icp` (`swap.rs` lines 1950–1959) checks:

```rust
if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
    if let Some(transfer) = &buyer_state.icp
        && transfer.transfer_success_timestamp_seconds == 0
    {
        return ErrorRefundIcpResponse::new_precondition_error(...);
    }
}
```

Because the attacker-created `BuyerState` always starts with `transfer_success_timestamp_seconds = 0`, the victim's `error_refund_icp` call is rejected until `sweep_icp` processes the entry. The `confirmation_text` guard is not a barrier: it is set at SNS init time and is readable from the public `get_state` query endpoint.

## Impact Explanation
This matches the **High** bounty impact: *Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.*

- **Swap commits:** Victim's ICP is swept to SNS governance; victim receives SNS neurons they never consented to. The `error_refund_icp` escape hatch is unavailable.
- **Swap aborts:** Victim's ICP is eventually refunded via `sweep_icp`, but `error_refund_icp` is denied in the interim — a targeted denial-of-service on the refund mechanism for the duration of finalization.

In both cases the victim is enrolled as a swap participant without their knowledge or consent, and their ICP is locked until the swap finalizes.

## Likelihood Explanation
The preconditions are realistic: ICP accidentally sent to a swap subaccount is a documented user error — the `error_refund_icp` mechanism exists precisely for this case. An attacker can monitor the public ICP ledger for transfers to swap subaccounts, read the `confirmation_text` from `get_state`, and call `refresh_buyer_tokens` with the victim's principal before the victim calls `error_refund_icp`. No privileged access is required. The attack is repeatable across any SNS swap that has a non-zero `confirmation_text` or none at all.

## Recommendation
Add a caller-identity check in the canister endpoint before passing the principal to `refresh_buyer_token_e8s`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let requested = PrincipalId::from_str(&arg.buyer).unwrap();
    assert_eq!(
        requested,
        caller_principal_id(),
        "buyer must match caller"
    );
    requested
};
```

Alternatively, remove the `buyer` override field entirely and always derive the principal from the caller, since the only legitimate use case is already covered by the empty-buyer path.

## Proof of Concept
```
1. Victim accidentally sends 10 ICP to principal_to_subaccount(victim_principal)
   on the swap canister's ICP ledger subaccount.

2. Attacker reads confirmation_text from get_state() (public query).

3. Attacker calls:
   refresh_buyer_tokens({
     buyer: victim_principal.to_text(),
     confirmation_text: Some("<text from step 2>")
   })
   → swap canister queries victim's subaccount balance (10 ICP ≥ min_participant_icp_e8s),
     creates BuyerState { icp: TransferableAmount { amount_e8s: 10*E8,
       transfer_success_timestamp_seconds: 0, ... } }

4. Swap closes (COMMITTED or ABORTED).

5. Victim calls error_refund_icp({ source_principal_id: victim_principal })
   → returns precondition error:
     "ICP cannot be refunded as principal <victim> has 1000000000 ICP (e8s) in escrow"

6. If COMMITTED: sweep_icp transfers victim's ICP to SNS governance;
   victim receives unwanted SNS neurons.
```

A deterministic integration test can be written using PocketIC: open a swap, transfer ICP to a victim's subaccount from a separate account, call `refresh_buyer_tokens` as a third-party attacker principal with `buyer = victim`, commit the swap, then assert that `error_refund_icp` returns a precondition error for the victim.
