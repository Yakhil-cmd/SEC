Audit Report

## Title
ckERC20 Minter Silently Finalizes Withdrawals Without Verifying ERC20 `transfer` Return Value — (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, `rs/ethereum/cketh/minter/src/state.rs`)

## Summary
The ckERC20 minter finalizes ERC20 withdrawal transactions by checking only the Ethereum transaction-level `status` field from the receipt. It never inspects the ERC20 `transfer` return value or verifies a `Transfer` event in the logs. For non-reverting ERC20 tokens that return `false` on failure, the Ethereum transaction receipt carries `status = 1` (Success) while the token transfer silently fails. The minter then permanently burns the user's ckERC20 tokens without reimbursement and decrements its internal ERC20 balance tracking, creating a ledger conservation violation.

## Finding Description

**Root cause:** The `TransactionReceipt` struct in `rs/ethereum/cketh/minter/src/eth_rpc_client/responses.rs` contains no `logs` field and no return-data field. The `From<EvmTransactionReceipt>` conversion explicitly discards any logs from the EVM RPC response, mapping only `block_hash`, `block_number`, `effective_gas_price`, `gas_used`, `status`, and `transaction_hash`.

In `record_finalized_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 733–745), the ckERC20 reimbursement path is gated exclusively on `receipt.status == TransactionStatus::Failure`:

```rust
WithdrawalRequest::CkErc20(request) => {
    if receipt.status == TransactionStatus::Failure {
        self.record_reimbursement_request(...);
    }
}
```

In `update_balance_upon_withdrawal` (`rs/ethereum/cketh/minter/src/state.rs`, lines 377–383), the minter decrements its internal `erc20_balances` whenever `receipt.status == TransactionStatus::Success`:

```rust
if receipt.status == TransactionStatus::Success && !tx.transaction_data().is_empty() {
    let TransactionCallData::Erc20Transfer { to: _, value } = ...;
    self.erc20_balances.erc20_sub(*tx.destination(), value);
}
```

The minter constructs the withdrawal transaction using the standard `a9059cbb` (`transfer(address,uint256)`) selector. For a non-reverting ERC20 token that returns `false` on failure (e.g., due to recipient blacklisting, contract pause, or other conditions), the EVM executes the call successfully at the protocol level (`status = 1`), but the token balances do not change. The minter has no mechanism to detect this outcome.

**Exploit flow:**
1. Governance adds a non-reverting ERC20 token as a supported ckERC20 asset (the minter is explicitly designed to support any ERC20 token via governance proposal).
2. User calls `withdraw_erc20` specifying a destination address that causes the token contract to return `false` without reverting (e.g., a blacklisted address).
3. Minter burns the user's ckERC20 tokens on the IC ledger (irreversible).
4. Minter submits an EIP-1559 transaction calling `transfer(blacklisted_address, amount)`.
5. ERC20 contract executes, returns `false`, no revert. Receipt: `status = 1`.
6. `record_finalized_transaction`: `status == Success`, so the `if receipt.status == TransactionStatus::Failure` branch is not taken — no reimbursement.
7. `update_balance_upon_withdrawal`: `status == Success`, so `erc20_balances.erc20_sub` is called — internal balance decremented.
8. Withdrawal permanently marked as finalized with `TxFinalizedStatus::Success`.
9. User's ckERC20 tokens are destroyed. ERC20 tokens remain in the minter's Ethereum address. Minter's internal balance is understated relative to actual on-chain holdings.

## Impact Explanation

This is a direct ledger conservation violation: ckERC20 supply is destroyed on the IC without a corresponding ERC20 transfer on Ethereum. The user permanently loses their ckERC20 tokens with no reimbursement path. The minter's internal `erc20_balances` diverges from its actual on-chain ERC20 holdings, compounding over repeated occurrences. This matches the allowed impact: **"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm"** — High ($2,000–$10,000).

## Likelihood Explanation

The minter is explicitly designed to support any ERC20 token added via NNS governance proposal, and the documentation states "Any ERC-20 token on Ethereum can be brought to the Internet Computer." Non-reverting ERC20 tokens that return `false` on failure exist in production (e.g., BNB, certain older stablecoin implementations). Once such a token is added as a supported ckERC20 asset, any unprivileged user whose withdrawal triggers a `false` return (e.g., because the destination is blacklisted by the token contract) will lose funds. No special attacker privileges are required beyond a standard `withdraw_erc20` call.

## Recommendation

1. **Verify via transaction logs:** Extend `TransactionReceipt` and the `eth_getTransactionReceipt` RPC call to capture the `logs` array. After a `Success`-status ERC20 withdrawal, verify that a `Transfer(from, to, value)` event was emitted by the ERC20 contract with the expected parameters. If no such log is present, treat the withdrawal as failed and trigger reimbursement.

2. **Alternatively, verify via balance query:** After a `Success`-status ERC20 withdrawal, query the ERC20 balance of the minter's Ethereum address and compare it to the expected post-transfer balance. A discrepancy indicates a silent transfer failure.

3. **Token admission control:** As defense-in-depth, require that ERC20 tokens added as ckERC20 assets are known to revert on failure (standard EIP-20 compliant behavior), and document this as a governance requirement.

## Proof of Concept

1. Deploy a non-reverting ERC20 token in a local PocketIC/StateMachine test environment: implement `transfer` to return `false` (without reverting) when the recipient is a designated "blacklisted" address.
2. Add this token as a supported ckERC20 asset via the minter's `add_ckerc20_token` endpoint.
3. Deposit tokens to mint ckERC20 on the IC.
4. Call `withdraw_erc20` specifying the blacklisted Ethereum address as destination.
5. Advance the state machine to simulate transaction finalization; inject a mock receipt with `status = Success` (as the EVM would produce for a non-reverting call).
6. Assert: (a) no reimbursement request was created; (b) `erc20_balances` was decremented; (c) the user's ckERC20 balance is zero; (d) the ERC20 token balance of the minter's Ethereum address is unchanged.

The existing integration test framework in `rs/ethereum/cketh/minter/tests/ckerc20.rs` already exercises the `TransactionStatus::Success` and `TransactionStatus::Failure` paths and can be extended with a mock non-reverting ERC20 token to demonstrate this gap.
