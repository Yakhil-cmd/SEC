### Title
Integer Division Truncation in `native_used / native_per_gas` Causes Systematic Operator Underpayment Per Transaction - (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the conversion from native resource units back to EVM gas units uses integer floor division (`native_used / native_per_gas`). The remainder (`native_used % native_per_gas`) is silently discarded. This means every transaction where native resource consumption is not an exact multiple of `native_per_gas` results in the user receiving a slightly larger refund than warranted, and the operator receiving slightly less than the actual native cost incurred. The effect is structurally identical to the WLSETH dust issue: a unit-conversion rounding loss that accumulates across all transactions.

---

### Finding Description

In `compute_gas_refund` (`refund_calculation.rs`), after pubdata is charged, the code computes a `delta_gas` to reconcile native resource consumption with EVM gas consumption:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The expression `native_used / native_per_gas` is Rust integer floor division. If `native_used = k * native_per_gas + r` where `0 < r < native_per_gas`, then:

- The code charges the user `k` gas units for native consumption.
- The correct charge (to fully cover native cost) would be `k + 1` gas units (ceiling division).
- The `r` native units consumed are never charged.

The `native_per_gas` ratio is derived as `ceil(gas_price / native_price)`:

- For L2 transactions: `u256_try_to_u64(&gas_price.div_ceil(native_price))` [2](#0-1) 
- For L1 transactions: `u256_try_to_u64(&gas_price.div_ceil(native_price))` [3](#0-2) 

The maximum remainder per transaction is `native_per_gas - 1` native units, worth at most `(native_per_gas - 1) * native_price ≈ gas_price` tokens — i.e., just under 1 gas unit of value. The developer's own `TODO: return delta_gas to gas_used?` comment at line 80 acknowledges the asymmetry in this accounting path. [4](#0-3) 

The final `gas_used` feeds directly into the refund and operator payment:

- User refund: `(gas_limit - gas_used) * gas_price` [5](#0-4) 
- Operator payment: `gas_used * gas_price_for_operator` [6](#0-5) 

Because `gas_used` is floored, the user receives 1 extra gas unit of refund and the operator receives 1 fewer gas unit of payment per affected transaction.

---

### Impact Explanation

Every transaction where `native_used % native_per_gas != 0` results in:

1. The user receiving up to `gas_price` extra tokens in refund (they are undercharged for native resource consumption).
2. The operator receiving up to `gas_price_for_operator` fewer tokens than the actual native cost incurred.

Across a high-throughput chain, this accumulates into a non-trivial aggregate shortfall for the operator. The per-transaction loss is bounded by approximately 1 gas unit of value, but with millions of transactions this becomes significant. The operator cannot recover this dust because the rounding is baked into the state transition function.

---

### Likelihood Explanation

This affects every L2 and L1 transaction processed by ZKsync OS where the `#[cfg(not(feature = "unlimited_native"))]` branch is active (i.e., production mode) and `native_per_gas > 1`. Since `native_per_gas = ceil(gas_price / native_price)` and `native_price` is a small constant (e.g., `L1_TX_NATIVE_PRICE = 10`), `native_per_gas` is almost always greater than 1 at any non-trivial gas price. The probability that `native_used` is an exact multiple of `native_per_gas` is negligible. This is therefore a near-certain, per-transaction occurrence. [7](#0-6) 

---

### Recommendation

Replace the floor division with ceiling division when converting native resource consumption to gas units:

```rust
// Instead of:
(native_used / native_per_gas) as i64 - (gas_used as i64)

// Use:
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

This ensures the operator is always compensated for the full native resource consumed, consistent with how `native_per_gas` itself is computed (using `div_ceil`). The user would be charged at most 1 extra gas unit, which is the correct conservative behavior.

---

### Proof of Concept

**Setup:**
- `gas_price = 1000`, `native_price = 10` (L1_TX_NATIVE_PRICE)
- `native_per_gas = ceil(1000 / 10) = 100`
- `gas_limit = 100_000`
- After execution: `gas_used_from_ergs = 50_000`, `native_used = 5_000_099`

**Current behavior (floor division):**
```
delta_gas = (5_000_099 / 100) - 50_000
          = 50_000 - 50_000
          = 0
```
User is charged 50,000 gas. The 99 native units (`5_000_099 % 100`) are consumed but not charged.

**Correct behavior (ceiling division):**
```
delta_gas = ceil(5_000_099 / 100) - 50_000
          = 50_001 - 50_000
          = 1
```
User should be charged 50,001 gas. The operator is underpaid by `1 * gas_price = 1000` tokens per such transaction.

With `native_per_gas = 100`, the maximum underpayment is `99 * 10 = 990` tokens per transaction. At 1,000 transactions per second, this is ~990,000 tokens per second of operator shortfall — a meaningful aggregate loss over time.
