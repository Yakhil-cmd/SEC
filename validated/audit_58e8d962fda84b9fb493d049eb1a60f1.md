### Title
Permanent Token Loss When `exitToNear` Promise Fails Due to Refund Path Never Taken - (File: `engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile always sets `callback_args.refund = None`. The `exit_to_near_precompile_callback` function then checks `if let Some(args) = args.refund` to trigger a refund when the underlying NEP-141 transfer promise fails. Because `args.refund` is always `None` in this build configuration, the refund branch is structurally unreachable — directly analogous to the reported `sweep()` checking for `INSOLVENT` status that is never set. The result is permanent loss of user funds: ERC-20 tokens are burned on the EVM side but never re-minted, and ETH is transferred to the precompile address but never returned.

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear` precompile constructs the callback arguments as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `callback_args.refund` is always `None`. The promise scheduling logic then checks whether a callback is even needed:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback at all
} else {
    PromiseArgs::Callback(...)              // callback scheduled
};
``` [2](#0-1) 

For the common ERC-20 or ETH exit case where `transfer_near` is also `None`, `callback_args` equals `default()` and **no callback is scheduled at all**. If the NEP-141 promise fails, there is no handler to re-mint the burned tokens or return the ETH.

For the wNEAR-unwrap case where `transfer_near` is `Some`, a callback is scheduled, but inside `exit_to_near_precompile_callback` the refund branch is still unreachable:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {   // ← args.refund is always None
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← silent no-op on failure
};
``` [3](#0-2) 

The `ExitToNearPrecompileCallbackArgs` struct confirms `refund` is `Option<RefundCallArgs>`: [4](#0-3) 

The test suite explicitly acknowledges the fund loss when the feature is absent:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4)

### Impact Explanation

**Critical — Permanent freezing / direct theft of user funds.**

- **ERC-20 exit**: The ERC-20 tokens are burned inside the EVM before the NEP-141 `ft_transfer` promise is dispatched. If that promise fails (e.g., unregistered recipient, paused contract, insufficient balance on the NEP-141 side), the burned tokens are never re-minted. They are gone permanently.
- **ETH exit**: ETH is transferred from the user's EVM account to the `exit_to_near` precompile address before the NEP-141 promise is dispatched. On promise failure, the ETH is stranded at the precompile address with no recovery path.

In both cases the loss is irreversible: there is no admin function, no sweep, and no retry mechanism that can recover the funds.

### Likelihood Explanation

**High.** The `exitToNear` precompile is a core, publicly accessible production interface reachable by any EVM user or contract. Promise failure is a realistic, non-adversarial condition: the recipient NEAR account may not be registered with the NEP-141 token, the NEP-141 contract may be paused, or the recipient account may not exist. No special privilege is required to trigger the failure path. The `error_refund` feature is an opt-in Rust feature (not in `default`), so production builds compiled without it are affected.

### Recommendation

Enable the `error_refund` feature unconditionally in all production builds of `aurora-engine-precompiles`, or restructure the code so that `RefundCallArgs` is always populated and the refund path is always reachable when the exit promise fails, regardless of compile-time flags. The callback should never silently discard a failed promise result when user funds are at stake.

### Proof of Concept

1. Deploy Aurora Engine without the `error_refund` feature enabled.
2. Bridge a NEP-141 token to an ERC-20 on Aurora.
3. As an EVM user, call the `exitToNear` precompile specifying a NEAR recipient account that is **not registered** with the NEP-141 contract.
4. Observe: the ERC-20 tokens are burned from the user's EVM balance.
5. The NEP-141 `ft_transfer` promise fails (unregistered account).
6. Because `callback_args == ExitToNearPrecompileCallbackArgs::default()` (both `refund` and `transfer_near` are `None`), no callback is scheduled.
7. The burned ERC-20 tokens are never re-minted. The user's EVM balance is reduced and the NEP-141 balance is unchanged — funds are permanently destroyed.

The same sequence applies to ETH: replace step 3 with a call to `withdrawEthToNear` targeting a non-existent NEAR account; the ETH is stranded at `exit_to_near::ADDRESS` with no recovery path.
