### Title
Unanchored `InteropRoot.root` Accepted from Any Emitting Contract in `interop_root_reporter_event_hook` — (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

`interop_root_reporter_event_hook` captures `InteropRootAdded` events and stores the embedded `root` value into the batch public input's `interop_roots_rolling_hash` without validating that the event was emitted by the trusted `L2_INTEROP_ROOT_STORAGE_ADDRESS` contract. Any unprivileged user can deploy a contract that emits a correctly-shaped `InteropRootAdded` event with an arbitrary `root`, `chain_id`, and `block_or_batch_number`, causing the system to accept a forged interop root as legitimate.

---

### Finding Description

In `system_hooks/src/event_hooks/interop_root_reporter.rs`, the hook `interop_root_reporter_event_hook` fires on every event emitted during block execution. Its only guards are:

1. `topics[0] == INTEROP_ROOT_ADDED_EVENT_SIG` — the Keccak256 of the event signature
2. `data.len() == 96` — ABI-encoded data length
3. `offset == 32` — ABI offset field
4. `len == 1` — exactly one `sides` element
5. `topics.len() == 3` — three indexed topics [1](#0-0) 

There is **no check on the address of the contract that emitted the event**. The hook unconditionally calls `system.io.add_interop_root(...)` with the `root`, `chain_id`, and `block_or_batch_number` parsed from the event. [2](#0-1) 

`add_interop_root` performs no further validation of the root value: [3](#0-2) 

`push_root` in `InteropRootStorage` also performs no validation: [4](#0-3) 

The accepted roots are then folded into `interop_roots_rolling_hash` via `calculate_interop_roots_rolling_hash`: [5](#0-4) 

This rolling hash is committed to in `BatchOutput.interop_roots_rolling_hash`, which becomes part of the cryptographically-binding batch public input verified on the settlement layer: [6](#0-5) [7](#0-6) 

The analog to the external report is exact: just as `extraProof.root` was internally consistent but never anchored to `massExitBlock.blockRid`, here `InteropRoot.root` passes structural checks but is never anchored to the trusted `L2_INTEROP_ROOT_STORAGE_ADDRESS` emitter.

---

### Impact Explanation

An attacker who injects a forged `InteropRoot` with an arbitrary `root` for any `chain_id` and `block_or_batch_number` causes the settlement layer to accept a batch public input that asserts a false cross-chain state commitment. Any protocol logic on the settlement layer or downstream chains that trusts `interop_roots_rolling_hash` as a proof of a remote chain's state can be deceived. This enables cross-chain fraud: claiming that chain X committed to state root Y at block Z when it did not, potentially enabling false withdrawal proofs or cross-chain asset claims backed by the forged root.

---

### Likelihood Explanation

The attack requires only deploying a contract that emits `InteropRootAdded(chainId, blockNumber, [forgedRoot])` with the correct ABI encoding and calling it in a standard L2 transaction. No privileged access, leaked keys, or governance majority is needed. The event signature is public. Any unprivileged user can execute this in a single transaction.

---

### Recommendation

Add an emitting-contract address check at the top of `interop_root_reporter_event_hook`. The hook should silently return `Ok(())` (or return an error) if the event did not originate from `L2_INTEROP_ROOT_STORAGE_ADDRESS`:

```rust
// At the start of interop_root_reporter_event_hook, before any other logic:
if emitter_address != L2_INTEROP_ROOT_STORAGE_ADDRESS {
    return Ok(());
}
```

The hook's signature must be extended to receive the emitting contract address, or the dispatch mechanism must filter by address before invoking the hook.

---

### Proof of Concept

1. Attacker deploys contract `FakeInteropEmitter` on the ZKsync OS L2:
```solidity
contract FakeInteropEmitter {
    event InteropRootAdded(uint256 indexed chainId, uint256 indexed blockOrBatchNumber, bytes32[] sides);

    function forge(uint256 chainId, uint256 blockNum, bytes32 root) external {
        bytes32[] memory sides = new bytes32[](1);
        sides[0] = root;
        emit InteropRootAdded(chainId, blockNum, sides);
    }
}
```

2. Attacker calls `forge(victimChainId, victimBlockNumber, forgedRoot)` in a normal L2 transaction.

3. `interop_root_reporter_event_hook` fires, passes all structural checks (correct event sig, `data.len() == 96`, `offset == 32`, `len == 1`, `topics.len() == 3`), and calls `add_interop_root` with the forged values. [8](#0-7) 

4. The forged root is folded into `interop_roots_rolling_hash` and committed to in the batch public input. [7](#0-6) 

5. The settlement layer accepts the batch with a falsified cross-chain state commitment for `victimChainId` at `victimBlockNumber`.

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
```
