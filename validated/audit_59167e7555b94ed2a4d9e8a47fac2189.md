Audit Report

## Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_start`/`ingress_end` in ICP Rosetta `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The `construction_payloads` handler in the ICP Rosetta node accepts caller-supplied `ingress_start` and `ingress_end` nanosecond timestamps with no server-side range validation. These values directly control the iteration count of an unbounded `while` loop that pushes entries into a `Vec<u64>`. An unauthenticated attacker can supply `ingress_start=0` and `ingress_end=u64::MAX` to force approximately 153 billion loop iterations, exhausting process memory and crashing the Rosetta node. The ICRC1 Rosetta implementation of the same endpoint includes explicit range guards that the ICP Rosetta implementation lacks entirely.

## Finding Description

`ConstructionPayloadsRequestMetadata` declares both `ingress_start` and `ingress_end` as raw `Option<u64>` with no constraints: [1](#0-0) 

In `construction_payloads`, the `interval` is computed as: [2](#0-1) 

`MAX_INGRESS_TTL` (300s) − `PERMITTED_DRIFT` (60s) − 120s = **120s = 120,000,000,000 ns**. [3](#0-2) 

The caller-supplied values are consumed directly with no bounds check before the loop: [4](#0-3) 

With `ingress_start=0` and `ingress_end=u64::MAX (18_446_744_073_709_551_615)`:
- iterations ≈ 18,446,744,073,709,551,615 / 120,000,000,000 ≈ **153,722,867,280**
- memory ≈ 153 billion × 8 bytes ≈ **1.2 TB**

The process OOMs before the loop completes. Additionally, if `ic_types::time::Time` addition wraps in release mode (Rust u64 wraps on overflow in release builds), `now` wraps back to a small value and the loop becomes **infinite** since `now < u64::MAX` is always true after wrapping.

The ICRC1 Rosetta implementation of the same endpoint explicitly validates the range before entering the loop: [5](#0-4) 

The ICP Rosetta implementation has **no equivalent guard** — no check that `ingress_start < ingress_end`, no check that the window is bounded relative to current time, and no cap on the number of iterations.

## Impact Explanation

The Rosetta node is a standalone HTTP service used for ICP ledger operations (transfers, neuron management, etc.). A single malformed POST request to `/construction/payloads` causes the process to exhaust all available memory and be OOM-killed by the OS. This constitutes an **application/platform-level DoS** against a financial integration component (ICP Rosetta), with concrete user harm: all ICP ledger operations routed through the node are blocked. The attack can be repeated immediately after restart, making the node permanently unavailable. This matches the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS" and "Significant... Rosetta... infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The `/construction/payloads` endpoint is publicly reachable with no authentication. The payload is a standard JSON POST body requiring no cryptographic material, tokens, or prior on-chain state. The attack requires a single HTTP request. Any party aware of the Rosetta API specification can trigger it. The attack is immediately repeatable after node restart, making remediation without a code fix impossible.

## Recommendation

Add an explicit upper bound on `ingress_end - ingress_start` before the loop, mirroring the ICRC1 Rosetta guard. For example:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // 24h
if ingress_end > ingress_start + MAX_INGRESS_WINDOW.as_nanos() as u64 {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed window",
    ));
}
```

Also add a check that `ingress_start < ingress_end` (returning an error otherwise), and optionally cap `ingress_expiries` with a hard `Vec` size limit (e.g., 720 entries for a 24-hour window at 120s intervals) as a defense-in-depth measure.

**Unit test:**
```rust
let meta = ConstructionPayloadsRequestMetadata {
    ingress_start: Some(0),
    ingress_end: Some(u64::MAX),
    memo: None,
    created_at_time: None,
};
// interval = 120_000_000_000 ns
// expected iterations ≈ 153_722_867_280 → OOM before completion
// assert: handler returns Err(...) before allocating unbounded Vec
```

**HTTP PoC:**
```json
POST /construction/payloads
{
  "network_identifier": { "blockchain": "Internet Computer", "network": "00000000000000020101" },
  "operations": [ { "operation_identifier": { "index": 0 }, "type": "TRANSACTION", ... } ],
  "public_keys": [ { "hex_bytes": "<valid_pubkey>", "curve_type": "edwards25519" } ],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

The Rosetta process will exhaust memory and be killed by the OS before returning a response. The attack is reproducible on any deployed ICP Rosetta node without any special privileges.
