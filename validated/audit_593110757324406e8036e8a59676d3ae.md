Audit Report

## Title
Unbounded Ingress-Window Loop via Attacker-Controlled `ingress_start`/`ingress_end` Enables Single-Request OOM Crash — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The `construction_payloads()` handler in the ICP Rosetta API accepts `ingress_start` and `ingress_end` directly from caller-supplied JSON metadata and feeds them into an unbounded `while` loop that pushes one `u64` entry per 120-second interval into a `Vec`. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes approximately 1.54×10¹¹ iterations, requiring ~1.23 TB of heap allocation, crashing the Rosetta process via OOM with a single unauthenticated HTTP request.

## Finding Description

`interval` is computed as a compile-time constant:

```
interval = MAX_INGRESS_TTL(300s) − PERMITTED_DRIFT(60s) − 120s = 120s = 120,000,000,000 ns
``` [1](#0-0) [2](#0-1) 

`ingress_start` and `ingress_end` are taken verbatim from the caller's metadata with no range or window-size validation: [3](#0-2) 

The loop then runs without any cap: [4](#0-3) 

The `ConstructionPayloadsRequestMetadata` struct accepts both fields as plain `Option<u64>` with no constraints enforced at the type or deserialization level: [5](#0-4) 

With `ingress_start=0` and `ingress_end=18446744073709551615` (u64::MAX):

- iterations ≈ 18,446,744,073,709,551,615 / 120,000,000,000 ≈ **1.54 × 10¹¹**
- heap required ≈ 1.54 × 10¹¹ × 8 bytes ≈ **~1.23 TB**

The process OOMs long before the loop completes. Additionally, if `ic_types::time::Time` addition wraps on overflow in release builds, `now` wraps back below `ingress_end` and the loop becomes infinite.

The only pre-loop checks are metadata deserialization and `verify_network_id`, neither of which constrains the window size. [6](#0-5) 

## Impact Explanation

A single unauthenticated HTTP POST to `/construction/payloads` crashes the Rosetta node process. This matches the allowed High impact: **"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"** and **"Significant... Rosetta... security impact with concrete user or protocol harm."** Severity: **High ($2,000–$10,000)**. The Rosetta API is explicitly in-scope as a financial integration component. A crashed Rosetta node is unavailable for all ICP transfer construction and submission operations, directly harming users relying on it for ICP ledger access.

## Likelihood Explanation

The Rosetta API is a publicly documented, network-reachable endpoint requiring no authentication. The exploit requires no special knowledge beyond the Rosetta API spec and knowledge of the `ingress_start`/`ingress_end` metadata fields (documented in the struct comments). The vulnerable code path is exercised on every `/construction/payloads` call that supplies metadata. One request suffices; the attack is trivially repeatable after any restart.

## Recommendation

Before the loop, enforce a maximum window size. For example:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // 24 h
if ingress_end.saturating_sub(ingress_start.as_nanos_since_unix_epoch())
    > MAX_INGRESS_WINDOW.as_nanos() as u64
{
    return Err(ApiError::invalid_request(
        "ingress_end exceeds maximum allowed window of 24 hours",
    ));
}
```

Alternatively, compute the expected iteration count before allocating and return an error if it exceeds a small constant (e.g., 1440 entries for a 24-hour window at 60-second granularity).

## Proof of Concept

```rust
#[test]
fn construction_payloads_oom_with_max_ingress_window() {
    // interval = 120s = 120_000_000_000 ns
    let interval_ns: u64 = 120_000_000_000;
    let count = u64::MAX / interval_ns;
    // ~1.54e11 iterations × 8 bytes = ~1.23 TB allocation attempt
    assert!(count > 100_000_000_000, "loop count is {count}");
}
```
