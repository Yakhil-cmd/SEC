Audit Report

## Title
Unauthenticated `buyer` Field in `refresh_buyer_tokens` Enables Confirmation-Text Bypass and Forced Participation — (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts a caller-supplied `buyer` principal with no check that it matches the authenticated caller. Any unprivileged ingress sender can invoke the function on behalf of any other principal, supplying the publicly-readable `confirmation_text` verbatim to commit a victim's ICP to the swap without the victim ever having agreed to the swap terms. The victim's open ticket is also deleted as a side effect.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 130–134, the target principal is resolved from `arg.buyer` rather than from the authenticated caller:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // no caller check
};
```

This `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`. Inside that function:

1. **Confirmation text validation** (`swap.rs` lines 363–384) only checks that the supplied text matches the SNS-stored text — it does not verify that the caller is the buyer. The stored text is fully public via the `get_init` query endpoint (`canister.rs` line 213), which returns the complete `Init` struct including `confirmation_text`.

2. **Ledger balance lookup** (`swap.rs` lines 1153–1163) queries the ICP ledger balance of the victim's subaccount derived from the attacker-supplied principal.

3. **Ticket deletion** (`swap.rs` lines 1248–1270): if the victim has an open ticket with `amount_ticket <= requested_increment_e8s`, the ticket is unconditionally deleted and participation is committed.

**Concrete attack flow:**

| Step | Actor | Action |
|------|-------|--------|
| 1 | Victim | Calls `new_sale_ticket` for amount X ICP |
| 2 | Victim | Transfers X ICP to their swap subaccount |
| 3 | Griefer | Reads `confirmation_text` via `get_init` |
| 4 | Griefer | Calls `refresh_buyer_tokens { buyer: "<victim>", confirmation_text: "<text>" }` |
| 5 | Swap | Validates text (passes), reads victim balance X, deletes ticket, commits X ICP |
| 6 | Victim | Calls `refresh_buyer_tokens` — `old_amount_icp_e8s >= e8s` → returns `Ok` (participation already committed without victim's explicit agreement) |

One inaccuracy in the submitted report: step 6 does **not** return an error. The code at `swap.rs` lines 1215–1221 returns `Ok` when `old_amount_icp_e8s >= e8s`. The victim's call succeeds but the confirmation text was never supplied by the victim themselves.

The `validate_confirmation_text` function has no binding between the text supplier and the buyer principal — it is a pure string comparison against public state.

## Impact Explanation
The `confirmation_text` mechanism exists specifically to require the buyer's explicit, affirmative agreement to swap terms before their ICP is committed. A griefer can read the text from public state and supply it on behalf of any victim who has already transferred ICP, committing that victim's funds to the swap without the victim ever having agreed. This constitutes a concrete SNS security impact: the consent mechanism is rendered ineffective for any buyer whose principal is observable from `list_direct_participants` or `get_buyer_state`. This matches the **High** impact tier: "Significant SNS or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
No special privilege is required. The victim's principal is observable from public query endpoints (`list_direct_participants`, `get_buyer_state`). The confirmation text is readable from `get_init`. The attack requires only that the victim has transferred ICP to their swap subaccount (the normal participation flow). It is trivially scriptable and repeatable for every new ICP deposit the victim makes.

## Recommendation
1. If `arg.buyer` is non-empty, assert `PrincipalId::from_str(&arg.buyer) == Ok(caller_principal_id())` and reject mismatches. This ensures the confirmation text is always supplied by the actual buyer.
2. Alternatively, remove the `buyer` field entirely and always derive the target from `caller_principal_id()`, as `new_sale_ticket` already does (`canister.rs` line 234).
3. If the `buyer` field must remain for third-party notification use cases, split the function: allow third parties to trigger the ledger balance refresh but require the buyer themselves to supply the confirmation text in a separate authenticated call.

## Proof of Concept
```
# 1. Victim creates ticket and transfers ICP
dfx canister call sns_swap new_sale_ticket '(record { amount_icp_e8s = 100_000_000; subaccount = null })' --identity victim
# (victim transfers 1 ICP to their swap subaccount on ICP ledger)

# 2. Griefer reads confirmation text
CONF=$(dfx canister call sns_swap get_init '(record {})' | grep confirmation_text)

# 3. Griefer commits victim's participation without victim's agreement
dfx canister call sns_swap refresh_buyer_tokens \
  '(record { buyer = "<victim_principal_text>"; confirmation_text = opt "<conf_text>" })' \
  --identity griefer

# 4. Verify victim's ICP is committed and ticket is gone
dfx canister call sns_swap get_buyer_state '(record { principal_id = opt principal "<victim>" })'
dfx canister call sns_swap get_open_ticket '(record {})' --identity victim
# → buyer_state shows committed ICP; open ticket is null (deleted by griefer's call)
```

A deterministic integration test using `StateMachine` or PocketIC can reproduce this by calling `refresh_buyer_tokens` with a different sender than the buyer principal and verifying the ticket is deleted and participation committed without the victim's ingress call.
