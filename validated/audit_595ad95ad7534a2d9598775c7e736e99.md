### Title
Unchecked Return Value of Low-Level Assembly `call` to Exit Precompile Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` bridge token contracts burn the caller's tokens **before** making a raw assembly `call` to the exit precompile. The return value `res` of that `call` is captured in a local variable but **never checked**. If the precompile call fails for any reason, the tokens are permanently destroyed with no withdrawal occurring and no revert to restore the user's balance.

---

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
// Step 1: tokens burned â€” irreversible
_burn(_msgSender(), amount);

// Step 2: raw assembly call to exit precompile
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    // res is NEVER checked â€” no switch/if iszero/revert
}
``` [1](#0-0) [2](#0-1) 

`EvmErc20V2.sol` contains the identical unchecked pattern in both `withdrawToNear` and `withdrawToEthereum`: [3](#0-2) [4](#0-3) 

The EVM `call` opcode returns `0` on failure and `1` on success. The Solidity compiler does **not** automatically revert on a failed low-level `call` â€” the caller must explicitly check `res`. Here, `res` is assigned but the assembly block ends without any `switch res case 0 { revert(0, 0) }` guard. The outer Solidity function also has no `require` on the result. The transaction completes successfully from the EVM's perspective while the withdrawal never happened.

The two hardcoded precompile addresses are the Aurora exit-to-NEAR connector (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) and the exit-to-Ethereum connector (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`), confirmed in the engine precompile source: [5](#0-4) 

---

### Impact Explanation

**Critical â€” Permanent freezing / destruction of user funds.**

The sequence is:
1. `_burn()` permanently removes `amount` tokens from the caller's ERC-20 balance.
2. The assembly `call` to the precompile fails silently.
3. No NEAR-side or Ethereum-side transfer is initiated.
4. The transaction does not revert; the caller's tokens are gone with no recourse.

This affects every bridged NEP-141 token mirrored as an `EvmErc20` or `EvmErc20V2` on Aurora. Any user who calls `withdrawToNear` or `withdrawToEthereum` under conditions that cause the precompile to return failure loses their tokens permanently.

---

### Likelihood Explanation

**Medium-to-High.** The functions are public and callable by any EVM token holder with no access control. Failure conditions for the precompile include (but are not limited to): malformed or zero-length recipient bytes, out-of-gas at the precompile level, or any internal precompile error path. A user who passes a recipient string that the NEAR-side rejects will silently lose their tokens. The burn-before-call ordering makes every invocation a potential loss event if the precompile rejects the input.

---

### Recommendation

Add an explicit revert inside the assembly block if `res` is zero, mirroring the pattern used in the `datacopy` precompile test helper:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    switch res
    case 0 { revert(0, 0) }
}
```

Alternatively, restructure the function to call the precompile **before** burning, and only burn on confirmed success. This eliminates the burn-then-fail race entirely.

---

### Proof of Concept

1. Deploy `EvmErc20` (or use an existing bridged token on Aurora).
2. Mint tokens to an attacker-controlled address.
3. Call `withdrawToNear` with a recipient byte string that the exit precompile rejects (e.g., an empty `bytes` value or an invalid NEAR account ID format).
4. Observe: `_burn` executes and the caller's balance drops to zero. The assembly `call` returns `res = 0` (failure). No revert occurs. The transaction is marked successful. No NEAR-side transfer is initiated. Tokens are permanently destroyed. [1](#0-0) [3](#0-2)
