### Title
Integer Truncation in `native_per_pubdata` Causes Pubdata to Be Free When `pubdata_price < native_price` - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio is computed using integer floor division (`wrapping_div`). When the operator-supplied `pubdata_price` is less than `native_price`, the result truncates to zero, completely disabling pubdata charging. Any unprivileged user can then generate unlimited pubdata at zero native-resource cost, causing the operator to absorb L1 data-availability costs without collecting fees.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the key ratio is computed as:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The identical pattern appears in the public API helper:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

`wrapping_div` is unsigned integer floor division. When `pubdata_price < native_price`, the quotient is 0. This zero propagates into every downstream pubdata-charging site:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   `intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` → 0 [3](#0-2) 

2. **Post-execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   `native = current_pubdata_spent.checked_mul(native_per_pubdata)` → 0 [4](#0-3) 

3. **Pubdata sufficiency check** in `check_enough_resources_for_pubdata` always passes because 0 resources are required. [5](#0-4) 

By contrast, `native_per_gas` uses **ceiling** division (`div_ceil`), ensuring users always pay at least the true cost for computation:

```rust
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [6](#0-5) 

The asymmetry — ceiling for gas, floor for pubdata — is the root cause. The `BlockMetadataFromOracle` struct carries both prices as raw `U256` values with no enforced ordering constraint between them: [7](#0-6) 

The deserialization path (`from_iter`) performs no range or ordering validation on `pubdata_price` vs `native_price`: [8](#0-7) 

The metadata initialization in the bootloader only validates gas limits, not price ordering: [9](#0-8) 

The existing test `test_check_pubdata_encoding_version` inadvertently demonstrates the bug: it sets `pubdata_price = 2` and `native_price = 100`, so `native_per_pubdata = 0`, and the test passes without any pubdata charge: [10](#0-9) 

---

### Impact Explanation

When `pubdata_price < native_price` (a realistic condition during periods of low L1 gas prices or high proving costs):

- `native_per_pubdata = 0`
- All pubdata charges — intrinsic and post-execution — become zero
- A user submitting storage-write-heavy transactions pays only EVM gas, not native resources for pubdata
- The operator must pay for L1 data availability (calldata / blob fees) without recovering those costs from users
- The block pubdata limit (`pubdata_limit`) is the only remaining guard, but it is a block-level cap, not a per-user fee

The error magnitude is 100%: the user pays 0 instead of `pubdata_price / native_price` native units per byte. For a transaction generating 10,000 bytes of pubdata with `pubdata_price = 9`, `native_price = 10`, the user saves `9,000` native units (the entire pubdata cost).

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic:

- The default test configuration uses `pubdata_price = 0`, `native_price = 10`, confirming the system is designed to operate with `pubdata_price < native_price`.
- During low L1 gas price periods, `pubdata_price` (proportional to L1 calldata/blob cost) can fall below `native_price` (proportional to proving cost).
- No on-chain or in-protocol enforcement prevents the operator from setting these values in this order.
- An attacker only needs to observe the current block's `pubdata_price` and `native_price` (both are public block metadata) and submit pubdata-heavy transactions when the condition holds.

---

### Recommendation

Replace the floor division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — truncates to 0 when pubdata_price < native_price)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — always charges at least 1 native unit per byte when pubdata_price > 0)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, consider adding a protocol-level invariant that `pubdata_price` must be validated against `native_price` during block metadata initialization in `metadata_op.rs`.

---

### Proof of Concept

The existing test already demonstrates the zero-charge condition. With `pubdata_price = 2` and `native_price = 100`:

```
native_per_pubdata = 2 / 100 = 0  (floor division)
intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0
post_execution_pubdata_charge = pubdata_bytes * 0 = 0
```

A storage-write-heavy transaction (e.g., writing 20 storage slots as in `test_pubdata_native_calculation_overflow`) with `pubdata_price = 9`, `native_price = 10` would generate ~640 bytes of pubdata at zero native cost, while the true cost should be `640 * 0.9 = 576` native units. The operator absorbs the L1 cost with no fee recovery.
