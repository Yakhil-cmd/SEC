Audit Report

## Title
Unbounded Per-Stream Memory Allocation via `read_to_end(MAX_MESSAGE_SIZE_BYTES)` with No Application-Level In-Flight Task Cap Enables OOM by a Single Malicious Subnet Peer — (File: rs/p2p/quic_transport/src/lib.rs)

## Summary
The QUIC transport layer sets `MAX_MESSAGE_SIZE_BYTES` to 128 MiB and uses it as the ceiling for `RecvStream::read_to_end` on every accepted bidirectional stream. Because `start_stream_acceptor` spawns an unbounded `JoinSet` task per accepted stream and the QUIC layer permits up to 1,000 concurrent bidirectional streams per connection, a single authenticated malicious subnet peer can force the victim replica to accumulate up to 128 GiB of heap memory across 1,000 concurrent `read_to_end` allocations, causing OOM and crashing the replica process.

## Finding Description
**Root cause — oversized read ceiling with no application-level concurrency cap:**

`MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024` is defined in `rs/p2p/quic_transport/src/lib.rs` (line 74) and passed directly to `RecvStream::read_to_end` in two call sites:

- `read_request` in `request_handler.rs` (lines 173–178): every inbound stream spawned by `start_stream_acceptor` calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`, which allocates a `Vec<u8>` that grows as data arrives, up to 128 MiB.
- `rpc` in `connection_handle.rs` (lines 115–124): the outbound RPC response path does the same.

`start_stream_acceptor` (`request_handler.rs`, lines 43–110) maintains an unbounded `JoinSet` (`inflight_requests`) and spawns a new task for every accepted stream without any application-level cap:

```rust
let mut inflight_requests: JoinSet<Result<(), P2PError>> = tokio::task::JoinSet::new();
// ...
inflight_requests.spawn(... handle_bi_stream(...) ...);
```

The code's own comment (lines 52–56) acknowledges this: *"The extreme result of a slow handler is that the stream limit will be reached, hence having buffered up to the stream limit number of messages/requests."*

The QUIC-level ceiling is `MAX_CONCURRENT_BIDI_STREAMS = VarInt::from_u32(1_000)` (`connection_manager.rs`, line 74). The `STREAM_RECEIVE_WINDOW` of 4 MiB (`connection_manager.rs`, line 73) is only the initial flow-control window; Quinn extends it automatically as the application reads, so the effective per-stream memory ceiling is `MAX_MESSAGE_SIZE_BYTES` (128 MiB), not 4 MiB. Similarly, the connection-level `RECEIVE_WINDOW` of 200 MiB is extended as data is consumed by the concurrent `read_to_end` tasks, so it does not cap aggregate Vec allocations.

**Exploit path:**
1. Malicious peer opens 1,000 bidirectional QUIC streams to the victim (the QUIC-enforced maximum).
2. On each stream, it slowly trickles data (respecting flow control to avoid idle timeout) up to 128 MiB without closing the stream.
3. Each `handle_bi_stream` task calls `read_request` → `recv_stream.read_to_end(134_217_728)`, allocating a `Vec<u8>` that grows to 128 MiB.
4. With 1,000 concurrent tasks, total heap pressure reaches 1,000 × 128 MiB = **128 GiB**, triggering OOM and crashing the replica.

**Why existing checks are insufficient:**
- The QUIC stream/connection receive windows are extended by Quinn as data is consumed, so they do not bound the Vec size.
- There is no per-peer or global application-level limit on the number of concurrently buffered streams or total bytes buffered.
- The `IDLE_TIMEOUT` of 5 seconds applies to connection-level idleness; as long as data trickles on any stream, the connection remains alive.

## Impact Explanation
A crashed replica drops out of consensus participation, degrading subnet fault tolerance. If the attacker controls `f` nodes in an `f`-fault-tolerant subnet, it can simultaneously attack all honest nodes, halting the subnet entirely. This matches the allowed impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS" — High ($2,000–$10,000)**. The attack is not raw volumetric DDoS; it exploits a specific application-layer buffering design flaw.

## Likelihood Explanation
- **Attacker role**: A single subnet node — a Byzantine protocol peer below the consensus fault threshold. No majority corruption is required.
- **Authentication barrier**: mTLS against the registry-verified subnet topology is enforced, so only legitimate subnet peers can connect. However, a single compromised or malicious node operator suffices.
- **No rate limiting**: There is no per-peer stream rate limit or per-peer memory budget at the application layer.
- **Operational justification**: The comment at `lib.rs:72–74` explicitly states the large limit is intentional, and the comment at `request_handler.rs:52–56` acknowledges the unbounded buffering as a known design trade-off, not a security control.
- **Repeatability**: The attack can be repeated after a replica restart.

## Recommendation
1. **Short term**: Enforce an application-level cap on the number of concurrently in-flight `handle_bi_stream` tasks per peer (e.g., reject or back-pressure new streams when the `JoinSet` size exceeds a threshold). Reduce `MAX_MESSAGE_SIZE_BYTES` to the smallest value that accommodates the largest legitimate message type (consensus summary blocks are cited as >5 MiB; a limit of 10–20 MiB would be far safer than 128 MiB).
2. **Long term**: Introduce per-peer memory accounting at the transport layer. Track total bytes buffered across all in-flight streams for a given peer and close the connection if the aggregate exceeds a configurable budget. Align `MAX_MESSAGE_SIZE_BYTES` with the actual maximum payload size enforced by the consensus and state-sync protocols.

## Proof of Concept
A malicious subnet node (with a valid registry TLS certificate) executes:

```
for i in 1..=1000:
    open bidirectional QUIC stream to victim node
    slowly write 128 MiB of arbitrary bytes on stream i (respecting flow control)
    do not close the stream — keep it open to hold the allocation
```

Each open stream causes the victim's `start_stream_acceptor` → `handle_bi_stream` → `read_request` → `recv_stream.read_to_end(134_217_728)` to allocate and grow a `Vec<u8>` up to 128 MiB. With 1,000 concurrent streams, the victim's heap grows by up to 128 GiB, triggering OOM.

A deterministic integration test can be written using the existing `turmoil`-based test harness (see `connection_handle.rs` tests, lines 186–332): open 1,000 streams from a simulated peer, write data slowly on each without closing, and assert that the victim process's memory usage grows proportionally or that the connection is rejected/back-pressured before OOM.
