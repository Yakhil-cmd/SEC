Audit Report

## Title
Unchecked Index Panic on Empty `Vec<EnvelopePair>` Enables Rosetta API DoS — (`rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

## Summary

`construction_parse` unconditionally accesses `updates[0]` on a `Vec<EnvelopePair>` deserialized from attacker-controlled CBOR with no non-empty invariant enforced. A single crafted POST to the unauthenticated `/construction/parse` endpoint triggers an index-out-of-bounds panic, crashing the actix-web worker thread and causing the Rosetta API to restart that worker, dropping all in-flight requests on it. Repeated requests keep workers in a restart loop, rendering the ICP Rosetta API unavailable.

## Finding Description

In `rs/rosetta-api/icp/src/request_handler/construction_parse.rs` at line 42, the signed-transaction branch iterates over `signed_transaction.requests` and blindly indexes element zero of each `Vec<EnvelopePair>`:

```rust
.map(
    |(request_type, updates)| match updates[0].update.content.clone() {
        HttpCallContent::Call { update } => (request_type.clone(), update),
    },
)
``` [1](#0-0) 

The `Request` type alias is `(RequestType, Vec<EnvelopePair>)`, so `updates` is a plain `Vec` with no non-empty invariant. [2](#0-1) 

`ParsedTransaction::try_from` performs only CBOR deserialization — it validates hex decoding and CBOR structure, but imposes no constraint that any `Vec<EnvelopePair>` within `requests` is non-empty:

```rust
Ok(ParsedTransaction::Signed(
    serde_cbor::from_slice(&from_hex(&value.transaction)?).map_err(|e| {
        ApiError::invalid_request(format!("Could not decode signed transaction: {e}"))
    })?,
))
``` [3](#0-2) 

A `SignedTransaction` containing `requests: vec![(RequestType::Send, vec![])]` is valid CBOR and deserializes without error. When the iterator reaches the empty `Vec<EnvelopePair>`, `updates[0]` panics with an index-out-of-bounds.

The HTTP route is registered publicly with no authentication middleware: [4](#0-3) 

The actix-web `App` configuration has no panic-catching middleware; the only middleware is an HTTP metrics wrapper: [5](#0-4) 

An identical unchecked index exists in `construction_hash.rs` at line 27 (`&envelope_pairs[0].update`), reachable via the same CBOR-deserialized `SignedTransaction` path when a transfer request type is present with an empty envelope pair list: [6](#0-5) 

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

A panic in an actix-web handler propagates through the worker thread's event loop. Actix-web restarts crashed workers, but during the restart window all in-flight requests on that worker fail. A persistent attacker sending this request repeatedly can keep all worker threads in a restart loop, rendering the ICP Rosetta API unavailable. The IC protocol, consensus, and canisters are unaffected; only the Rosetta API process availability is impacted. The Rosetta API is an in-scope financial integration component used for ICP ledger interactions.

## Likelihood Explanation

The endpoint is unauthenticated and publicly reachable. Exploitation requires only the ability to POST JSON to `/construction/parse` with `signed: true` and a hex-encoded CBOR payload encoding a `SignedTransaction` with an empty `Vec<EnvelopePair>`. No privileged access, key material, or deep protocol knowledge is required. The CBOR structure is straightforward to craft using any CBOR library. The attack is non-volumetric: one request per worker thread is sufficient to trigger the panic.

## Recommendation

Add a bounds check before accessing `updates[0]` in `construction_parse.rs`. Return an `ApiError::invalid_request` if the vec is empty:

```rust
.map(|(request_type, updates)| {
    let first = updates.first().ok_or_else(|| {
        ApiError::invalid_request("EnvelopePair list is empty")
    })?;
    match first.update.content.clone() {
        HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
    }
})
.collect::<Result<Vec<_>, _>>()?
```

Apply the same fix to `construction_hash.rs` line 27: check `envelope_pairs.first()` and return `ApiError::invalid_request` if empty, rather than indexing `envelope_pairs[0]` directly.

## Proof of Concept

```rust
// Construct a SignedTransaction with an empty EnvelopePair vec
let malicious = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],  // empty Vec<EnvelopePair>
};
let cbor_bytes = serde_cbor::to_vec(&malicious).unwrap();
let hex_payload = hex::encode(&cbor_bytes);

// POST to /construction/parse:
// {
//   "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
//   "signed": true,
//   "transaction": "<hex_payload>"
// }
// => CBOR deserializes successfully (no empty-vec validation)
// => updates[0] on empty slice => index out of bounds panic
// => actix-web worker thread crashes and restarts
```

A deterministic unit test can be added to `construction_parse.rs`'s existing `#[cfg(test)]` module: construct a `ConstructionParseRequest` with `signed: true` and a hex-CBOR `SignedTransaction` containing an empty `Vec<EnvelopePair>`, call `handler.construction_parse(...)`, and assert it returns `Err(ApiError::...)` rather than panicking.
