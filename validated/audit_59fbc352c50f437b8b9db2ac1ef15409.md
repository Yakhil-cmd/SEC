### Title
ERC-20 Tokens Permanently Burned With No Refund Path When `exit_to_near` Promise Fails Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is absent from the production binary, a user who calls the `ExitToNear` precompile to bridge ERC-20 tokens back to NEAR will have their ERC-20 tokens permanently burned if the downstream `ft_transfer` promise fails. There is no refund path: the tokens are destroyed on the Aurora side and never delivered on the NEAR side.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles ERC-20 → NEP-141 exits. The flow is:

1. The ERC-20 contract burns the user's tokens and calls the precompile.
2. The precompile schedules an `ft_transfer` (or `ft_transfer_call`) promise to the NEP-141 contract.
3. A callback `exit_to_near_precompile_callback` is attached to handle success or failure.

The callback args are constructed as:

```rust
// engine-precompiles/src/native.rs
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is **not** compiled in, `refund` is unconditionally `None`. In the callback:

```rust
// engine/src/contract_methods/connector.rs
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {
    // refund path — never reached when refund is None
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op: tokens are gone, no re-mint, no error
};
``` [2](#0-1) 

The `refund_on_error` function, which re-mints the burned ERC-20 tokens to the user, is never invoked:

```rust
// engine/src/engine.rs
if let Some(erc20_address) = args.erc20_address {
    // ERC-20 exit; re-mint burned tokens
    ...
    engine.call(&erc20_admin_address, &erc20_address, Wei::zero(), input, ...)
}
``` [3](#0-2) 

The existing test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract), the ERC-20 tokens are already burned on the Aurora side. Without the `error_refund` feature, the callback silently returns `None` with no re-mint. The user's ERC-20 tokens are destroyed and the NEP-141 tokens remain locked in Aurora's account with no recovery path for the user.

---

### Likelihood Explanation

**Medium.** The `ft_transfer` promise fails whenever the recipient NEAR account is not registered with the NEP-141 token contract — a common user mistake (e.g., specifying a fresh account, a typo, or an account that has never called `storage_deposit`). Any unprivileged EVM user can trigger this by calling the ERC-20 exit function with such a recipient. The only prerequisite is that the production binary was compiled without the `error_refund` feature.

---

### Recommendation

Promote `error_refund` to a default feature in both `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, or remove the conditional compilation entirely and always include the refund logic. The refund path in `refund_on_error` is already implemented and correct; it simply needs to be unconditionally reachable. [1](#0-0) 

---

### Proof of Concept

1. Deploy Aurora with the production binary compiled **without** the `error_refund` feature.
2. Bridge a NEP-141 token into Aurora, receiving ERC-20 tokens at address `erc20`.
3. From an EVM account, call the ERC-20 exit function targeting a NEAR account that has **never** called `storage_deposit` on the NEP-141 contract (e.g., `"unregistered.near"`).
4. The ERC-20 contract burns the user's tokens and calls the `ExitToNear` precompile.
5. The precompile schedules `ft_transfer("unregistered.near", amount)` on the NEP-141 contract.
6. `ft_transfer` panics because `"unregistered.near"` is not registered.
7. `exit_to_near_precompile_callback` is invoked; `args.refund` is `None`; the callback returns `None` silently.
8. **Result:** user's ERC-20 balance is reduced by `amount`, NEP-141 balance of Aurora is unchanged, and the user has no mechanism to recover the burned tokens. [1](#0-0) [5](#0-4)
