### Title
SNS Swap Token Allocation Rounds Down, Causing Buyers to Receive Fewer SNS Tokens Than Entitled - (File: rs/sns/swap/src/swap.rs)

### Summary
The `Swap::scale` function in the SNS Swap canister uses integer division (floor) when computing each buyer's SNS token allocation. Because every buyer's allocation is independently rounded down, the sum of all allocations is strictly less than the total SNS tokens offered. The unallocated remainder is never redistributed to buyers and stays locked in the swap canister, causing a systematic, per-swap loss to all direct participants.

### Finding Description
In `create_sns_neuron_recipes`, for every direct participant the swap canister calls:

```rust
let amount_sns_e8s = Swap::scale(
    buyer_state.amount_icp_e8s(),
    sns_being_offered_e8s,
    total_participant_icp_e8s,
);
```

`Swap::scale` is implemented as:

```rust
fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
    let r = (amount_icp_e8s as u128)
        .saturating_mul(total_sns_e8s as u128)
        .div(NonZeroU128::from(total_icp_e8s));
    r as u64
}
```

`NonZeroU128::div` performs truncating (floor) integer division. For each buyer `i`, the allocated amount is `⌊icp_i × total_sns / total_icp⌋`. The sum across all N buyers satisfies:

```
Σ ⌊icp_i × total_sns / total_icp⌋  ≤  total_sns  −  (N − 1)
```

The gap (up to N−1 e8s) is never credited to any buyer. The `apportion_approximately_equally` helper used for the neuron basket split correctly handles remainders by incrementing the last bucket, but `scale` has no such correction. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Every committed SNS decentralization swap with more than one direct participant silently under-allocates SNS tokens to buyers. The unallocated dust remains in the swap canister; after finalization it is effectively transferred to the SNS treasury rather than to the buyers who paid for it. With a large number of participants (the protocol allows up to `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS / basket_count` buyers), the aggregate loss can reach hundreds of e8s per swap. Buyers have no recourse: the neuron recipes are created with the rounded-down amounts and the ledger transfers follow those recipes exactly. [4](#0-3) 

### Likelihood Explanation
Every SNS swap with more than one direct participant triggers this rounding loss — the condition is met by virtually every real-world SNS launch. No special attacker action is required; the loss occurs automatically during the normal `create_sns_neuron_recipes` → `sweep_sns` finalization flow. Any unprivileged user who calls `refresh_buyer_token_e8s` to participate in a swap is affected. [5](#0-4) 

### Recommendation
Replace the plain floor division in `Swap::scale` with ceiling division so that each buyer's allocation is rounded up:

```rust
fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
    let denom = u128::from(total_icp_e8s);
    let numer = (amount_icp_e8s as u128).saturating_mul(total_sns_e8s as u128);
    // ceiling division: (numer + denom - 1) / denom
    let r = numer.saturating_add(denom - 1) / denom;
    assert!(r <= u64::MAX as u128);
    r as u64
}
```

Alternatively, after the per-buyer loop, compute the undistributed remainder and add it to the last (or largest) buyer's allocation, mirroring the approach already used in `apportion_approximately_equally`. [6](#0-5) 

### Proof of Concept
Concrete numeric example with values that fit the swap's parameter constraints:

- `sns_being_offered_e8s = 3`
- `total_participant_icp_e8s = 2`
- Buyer A contributes 1 ICP e8s → `scale(1, 3, 2) = ⌊3/2⌋ = 1`
- Buyer B contributes 1 ICP e8s → `scale(1, 3, 2) = ⌊3/2⌋ = 1`
- Total distributed: **2** SNS e8s
- Remaining in swap canister: **1** SNS e8s (never distributed to either buyer)

Both buyers paid for 1.5 SNS tokens each but received only 1. The 1 e8s remainder stays in the swap canister. With N buyers the maximum undistributed amount is N−1 e8s per swap. [7](#0-6)

/// Chops up `total` in to `len` pieces.
///
/// More precisely, result.len() == len. result.sum() == total. Each element of
/// result is approximately equal to the others. However, unless len divides
/// total evenly, the elements of result will inevitably be not equal.
///
/// There are two ways that Err can be returned:
///
///   1. Caller mistake: len == 0
///
///   2. This has a bug. See implementation comments for why we know of know way
///      this can happen, but can detect if it does.
pub fn apportion_approximately_equally(total: u64, len: u64) -> Result<Vec<u64>, String> {
    let quotient = total
        .checked_div(len)
        .ok_or_else(|| format!("Unable to divide total={total} by len={len}"))?;
    let remainder = total % len; // For unsigned integers, % cannot overflow.

    // So far, we have only apportioned quotient * len. To reach the desired
    // total, we must still somehow add remainder (per Euclid's Division
    // Theorem). That is accomplished right after this.
    let mut result = vec![quotient; len as usize];

    // Divvy out the remainder: Starting from the last element, increment
    // elements by 1. The number of such increments performed here is remainder,
    // bringing our total back to the desired amount.
    if remainder >= result.len() as u64 {
        return Err(format!("Could not apportion {total} into {len} pieces"));
    }
    let mut iter_mut = result.iter_mut();
    for _ in 0..remainder {
        let element: &mut u64 = iter_mut
            .next_back()
            // We can prove that this will not panic:
            // The number of iterations of this loop is total % len.
            // This must be < len (by Euclid's Division Theorem).
            // Thus, the number of iterations that this loop goes through is < len.
            // Thus, the number of times next_back is called is < len.
            // next_back only returns None after len calls.
            // Therefore, next_back does not return None here.
            // Therefore, this expect will never panic.
            .ok_or_else(
                || format!("Ran out of elements to increment. total={total}, len={len}",),
            )?;

        // This cannot overflow because the result must be <= total. Thus, this
        // will not panic.
        *element = element.checked_add(1).ok_or_else(|| {
            format!("Incrementing element by 1 resulted in overflow. total={total}, len={len}",)
        })?;
    }

    Ok(result)
