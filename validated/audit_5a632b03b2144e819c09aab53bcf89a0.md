Audit Report

## Title
In-Memory Rate Limiter and Permissive Access-Control Defaults for `swap_node_in_subnet_directly` Reset on Canister Upgrade - (File: rs/registry/canister/src/mutations/do_swap_node_in_subnet_directly.rs)

## Summary
The `swap_node_in_subnet_directly` endpoint in the Registry canister stores all three of its protection layers — a global feature flag, two access-control policies, and a per-subnet/per-operator rate limiter — exclusively in `thread_local!` heap variables. Because IC canister upgrades replace the Wasm heap with fresh initial values, every upgrade resets `IS_NODE_SWAPPING_ENABLED` to `true`, both `NODE_SWAPPING_CALLERS_POLICY` and `NODE_SWAPPING_SUBNETS_POLICY` to `AccessList::allow_all()`, and `SWAP_LIMITER` to zero committed usage. Any registered node operator can immediately exploit the cleared rate limiter after any routine registry upgrade to perform swaps beyond the intended cadence, including on NNS and signing subnets, and to game node-provider reward calculations.

## Finding Description

**No governance guard at the canister entry point.** `swap_node_in_subnet_directly()` in `canister.rs` (L831–836) contains no `check_caller_is_governance_and_log` call, unlike the majority of registry mutation endpoints such as `update_node_operator_config` (L795–801).

**All protection state lives in `thread_local!` heap variables.** In `flags.rs` (L8–21), three statics are declared:
- `IS_NODE_SWAPPING_ENABLED: Cell<bool> = const { Cell::new(true) }` — defaults to `true`
- `NODE_SWAPPING_CALLERS_POLICY: RefCell<AccessList<PrincipalId>> = RefCell::new(AccessList::allow_all())` — allows all callers
- `NODE_SWAPPING_SUBNETS_POLICY: RefCell<AccessList<SubnetId>> = RefCell::new(AccessList::allow_all())` — allows all subnets

In `do_swap_node_in_subnet_directly.rs` (L106–108), the rate limiter is declared:
- `SWAP_LIMITER: RefCell<SwapRateLimiter> = RefCell::new(SwapRateLimiter::new())` — uses `InMemoryRateLimiter`, not stable-memory-backed storage

**`post_upgrade` does not restore any of these values.** `canister_post_upgrade` in `registry_lifecycle.rs` (L19–81) only deserializes registry data from stable storage and runs data migrations. It never re-applies `IS_NODE_SWAPPING_ENABLED`, `NODE_SWAPPING_CALLERS_POLICY`, `NODE_SWAPPING_SUBNETS_POLICY`, or `SWAP_LIMITER` state.

**Contrast with stable-memory rate limiters.** `NODE_PROVIDER_RATE_LIMITER` and `NODE_OPERATOR_RATE_LIMITER` in `rate_limits.rs` (L33–56) use `RateLimiter::new_stable` backed by `StableMemoryCapacityStorage`, which survives upgrades. `SWAP_LIMITER` uses `InMemoryRateLimiter::new_in_memory`, which does not.

**`allow_all()` semantics confirmed by test suite.** The integration test in `tests/swap_node_in_subnet_directly.rs` (L169–171) explicitly documents: *"In order to avoid the feature being enabled for all node operators there needs to be some other caller whitelisted."* This confirms that an empty whitelist under `allow_all()` permits every caller.

**Exploit path in `swap_nodes_inner`.** After any upgrade, `swap_nodes_inner` (L118–147) sees `is_node_swapping_enabled() == true`, `swapping_enabled_for_caller(any) == true`, `swapping_allowed_on_subnet(any) == true`, and a fresh `SWAP_LIMITER` with zero committed usage. The only remaining check is that the caller must be the node operator of both nodes — a condition the operator trivially satisfies for their own nodes.

## Impact Explanation

**High ($2,000–$10,000).** After every routine registry canister upgrade (which occurs regularly via NNS governance proposals), any registered node operator can:

1. Bypass the intended 4-hour per-subnet and 24-hour per-operator rate limits, performing swaps at an unconstrained rate until the in-memory limiter fills again.
2. Swap their nodes in and out of any subnet — including the NNS subnet and threshold-signing subnets — without governance approval, potentially degrading subnet availability or disrupting threshold-signature key availability.
3. Game node-provider reward calculations by swapping high-uptime nodes into subnets immediately before reward snapshots and swapping them back out afterward, earning rewards they would not otherwise qualify for.

This matches the allowed impact class: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm"* and *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact."*

## Likelihood Explanation

The registry canister is upgraded regularly as part of routine NNS governance. Every upgrade resets the protections. The exploit requires only that the attacker be a registered node operator (a permissioned but not rare role on the IC). The window of exposure begins immediately after each upgrade completes and persists until the rate limiter naturally refills (4 hours per subnet, 24 hours per operator per subnet). Because upgrades are frequent and the rate limiter is the sole persistent constraint, the opportunity recurs with every upgrade cycle. The financial incentive (reward gaming) makes exploitation attractive.

## Recommendation

1. **Persist `SWAP_LIMITER` in stable memory**, mirroring the pattern used by `NODE_PROVIDER_RATE_LIMITER` and `NODE_OPERATOR_RATE_LIMITER` in `rate_limits.rs` (L33–56), using `RateLimiter::new_stable` with `StableMemoryCapacityStorage`.
2. **Change the `AccessList` default from `allow_all()` to `allow_none()`** in `flags.rs` (L18–20) so that an empty whitelist denies all callers rather than permitting all callers after an upgrade.
3. **Re-apply swapping flags in `canister_post_upgrade`** in `registry_lifecycle.rs` from a stored configuration (e.g., persisted in stable memory or re-derived from the `RegistryCanisterInitPayload` pattern), ensuring `IS_NODE_SWAPPING_ENABLED`, `NODE_SWAPPING_CALLERS_POLICY`, and `NODE_SWAPPING_SUBNETS_POLICY` are restored to their pre-upgrade values rather than reverting to permissive defaults.
4. **Optionally add a `check_caller_is_governance_and_log` guard** at the canister entry point, or ensure the feature-enabled flag defaults to `false` and is explicitly re-enabled post-upgrade.

## Proof of Concept

1. NNS governance enables `swap_node_in_subnet_directly` for node operator A on subnet S with a rate limit of 1 swap per 4 hours.
2. Node operator A performs one swap, consuming the subnet rate limit.
3. NNS governance passes a routine registry canister upgrade proposal.
4. Immediately after the upgrade: `IS_NODE_SWAPPING_ENABLED == true`, `NODE_SWAPPING_CALLERS_POLICY == allow_all()`, `NODE_SWAPPING_SUBNETS_POLICY == allow_all()`, and `SWAP_LIMITER` has zero committed usage (fresh `InMemoryRateLimiter`).
5. Node operator A (and any other node operator) can now call `swap_node_in_subnet_directly` on any subnet without restriction, bypassing the intended 4-hour rate limit and the caller/subnet whitelist entirely.
6. To demonstrate reward gaming: operator A swaps their highest-uptime node into a subnet immediately before the reward snapshot, then swaps it back out afterward, claiming rewards for uptime they did not sustain during the full reward period.

A deterministic integration test can be written using PocketIC: initialize the registry canister with a whitelist and rate limit, exhaust the rate limit, simulate an upgrade (reinstall with the same stable storage), and verify that `swap_node_in_subnet_directly` succeeds immediately after the upgrade despite the rate limit having been consumed before it.
