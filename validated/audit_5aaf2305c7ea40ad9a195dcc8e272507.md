Audit Report

## Title
Unbounded O(N) Candid Decoding in `handle_construction_hash` Enables Single-Request Application-Level DoS тАФ (`rs/rosetta-api/icrc1/src/construction_api/utils.rs`)

## Summary

`handle_construction_hash` iterates every envelope in a caller-supplied `SignedTransaction` without any count guard, performing a full Candid decode and hash computation per envelope. The axum router applies no `DefaultBodyLimit`, so the body and the resulting `Vec<Envelope>` are unbounded. An unauthenticated attacker can submit a single crafted request containing arbitrarily many envelopes and drive the Rosetta process to CPU exhaustion, blocking all legitimate ICRC-1 transaction operations.

## Finding Description

**Unbounded loop with O(N) Candid decoding:**
In `rs/rosetta-api/icrc1/src/construction_api/utils.rs` (L331тАУ335), `handle_construction_hash` iterates every envelope unconditionally:

```rust
let mut tx_hashes = HashSet::new();
for envelope in signed_transaction.envelopes {
    let transaction_hash = build_transaction_hash_from_envelope_content(&envelope.content)?;
    tx_hashes.insert(transaction_hash);
}
```

Each call to `build_transaction_hash_from_envelope_content` (L194тАУ216) performs:
1. `CanisterMethodName::new_from_envelope_content` тАФ method-name parsing
2. A full Candid `Decode!` of the `arg` bytes into `TransferArg`, `ApproveArgs`, or `TransferFromArgs`
3. ICRC-1 transaction hash computation

The `tx_hashes.len() > 1` early-exit check (L338тАУ342) fires only **after** the entire vector is exhausted, providing no protection against large inputs.

**No HTTP body-size limit:**
The axum router in `rs/rosetta-api/icrc1/src/main.rs` (L361тАУ391) applies only a metrics layer, a tracing span layer, and a `RequestIdLayer`. No `DefaultBodyLimit` middleware is present anywhere in the router chain. Axum's default behavior without `DefaultBodyLimit` imposes no cap on incoming body size, so the hex-encoded CBOR `signed_transaction` field can be arbitrarily large.

**Unbounded `Vec<Envelope>` in `SignedTransaction`:**
`rs/rosetta-api/icrc1/src/construction_api/types.rs` (L45тАУ48) defines `SignedTransaction` as a plain `Vec<Envelope>` with no length constraint. The `FromStr` implementation (L56тАУ60) deserializes directly from CBOR with no size check.

**Unauthenticated, publicly reachable endpoint:**
`construction_hash` in `rs/rosetta-api/icrc1/src/construction_api/endpoints.rs` (L67тАУ76) performs only a network-identifier lookup before delegating to `services::construction_hash`. No authentication, API key, or rate-limiting is applied. The server binds to `0.0.0.0:8080` by default (main.rs L393).

**Exploit flow:**
1. Attacker constructs a `SignedTransaction` with N (e.g., 100,000) envelopes, each with a valid `icrc1_transfer` call payload.
2. Attacker hex-encodes the CBOR-serialized struct and POSTs it to `/construction/hash`.
3. The server performs N sequential Candid decodes on a single thread before returning any response.
4. Concurrent such requests multiply CPU load; the single-binary Rosetta process becomes unresponsive to all other endpoints including `/construction/submit`.

## Impact Explanation

This is an application-level DoS against the ICRC-1 Rosetta API, a production financial integration component explicitly listed in scope. CPU saturation from a single crafted request prevents legitimate `/construction/submit` calls from being processed, effectively blocking ICRC-1 transaction submission for any exchange or integrator running this Rosetta instance. This matches the allowed High impact: **"Application/platform-level DoS... not based on raw volumetric DDoS"** тАФ the attack exploits algorithmic complexity with a single request, not raw traffic volume.

**Severity: High ($2,000тАУ$10,000)**

## Likelihood Explanation

The endpoint requires no credentials. The encoding of `SignedTransaction` is fully documented and reproducible from the public `Display`/`FromStr` implementations. The server binds to `0.0.0.0` by default and is commonly reachable from internal networks or the public internet in exchange/integrator deployments. The attack is repeatable, low-cost, and requires no special privileges or victim interaction.

## Recommendation

1. **Add `DefaultBodyLimit`** to the axum router (e.g., 2 MB) to cap total request body size before deserialization.
2. **Add an envelope-count guard** at the top of `handle_construction_hash`: reject if `signed_transaction.envelopes.len() > MAX_ENVELOPES`, where `MAX_ENVELOPES` reflects the legitimate 24 h / ingress-interval ceiling (~288).
3. **Exit the loop early** once `tx_hashes.len() > 1` is detected, avoiding processing of remaining envelopes.

## Proof of Concept

```rust
// Construct N envelopes with distinct created_at_time тЖТ distinct tx hashes
let mut envelopes = vec![];
for i in 0..100_000u64 {
    let arg = Encode!(&TransferArg {
        from_subaccount: None,
        to: Account { owner: Principal::anonymous(), subaccount: None },
        fee: None,
        created_at_time: Some(i),
        memo: None,
        amount: 1u64.into(),
    }).unwrap();
    envelopes.push(Envelope {
        content: Cow::Owned(EnvelopeContent::Call {
            nonce: None, ingress_expiry: u64::MAX,
            sender: Principal::anonymous(),
            canister_id: Principal::anonymous(),
            method_name: "icrc1_transfer".to_string(),
            arg,
        }),
        sender_pubkey: None, sender_sig: None, sender_delegation: None,
    });
}
let tx = SignedTransaction { envelopes };
// POST {"network_identifier": {...}, "signed_transaction": tx.to_string()}
// to /construction/hash тЖТ server performs 100,000 Candid decodes before responding
```

The loop in `handle_construction_hash` (L332тАУ335) processes all N envelopes before the `tx_hashes.len() > 1` guard (L338) fires, confirming O(N) unbounded CPU work per unauthenticated request.
