Audit Report

## Title
SNS Governance `distribute_rewards` Unbounded Nested Loop Can Exceed Instruction Limit — (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance `distribute_rewards` function, called synchronously from `run_periodic_tasks` on every timer tick, contains three nested loops over missed reward rounds, proposal ballots, and neurons with no instruction-limit guard or batching mechanism. If the SNS accumulates many proposals in `ReadyToSettle` state with many voting neurons, a single timer execution can exceed the IC instruction limit, causing the entire execution to roll back. Because `latest_reward_event` is never updated on failure, every subsequent timer firing retries the same failing computation, permanently breaking reward distribution.

## Finding Description

**Call path:** `run_periodic_tasks` (line 5503–5513) calls `distribute_rewards` synchronously within a single message execution. There is no `is_message_over_threshold` check, no batching, and no pagination anywhere in this path.

**Loop 1 — missed rounds** (`rs/sns/governance/src/governance.rs`, lines 5861–5872):
```rust
for i in 1..=new_rounds_count {
    let current_reward_rate = voting_rewards_parameters.reward_rate_at(...);
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
```
`new_rounds_count` is computed as `(now - last_reward_event_end) / round_duration_seconds` with no cap. Any canister upgrade or subnet maintenance causes this to grow proportionally to elapsed time.

**Loop 2 — proposals × ballots** (lines 5894–5930):
```rust
for proposal_id in &considered_proposals {
    if let Some(proposal) = self.get_proposal_data(*proposal_id) {
        for (voter, ballot) in &proposal.ballots {
            *neuron_id_to_reward_shares.entry(neuron_id).or_insert_with(|| dec!(0)) += reward_shares;
        }
    }
}
```
`considered_proposals` is the full set of all proposals in `ReadyToSettle` state, collected without any cap. Each proposal's `ballots` map has one entry per neuron that voted.

**Loop 3 — neurons** (lines 5954–5997):
```rust
for (neuron_id, neuron_reward_shares) in neuron_id_to_reward_shares {
    let neuron: &mut Neuron = match self.get_neuron_result_mut(&neuron_id) { ... };
    let neuron_reward_e8s = rewards_purse_e8s * (neuron_reward_shares / total_reward_shares);
    neuron.maturity_e8s_equivalent += neuron_reward_e8s;
}
```

Total work is **O(R + P × N + N)** where R = missed rounds, P = proposals in ReadyToSettle, N = neurons. There is no `is_message_over_threshold` check, no early break, and no state persistence across messages.

**Contrast with NNS:** The NNS governance explicitly fixed this pattern in Proposal 135702. The NNS `continue_processing` in `rs/nns/governance/src/reward/distribution.rs` (lines 154–188) uses `is_over_instructions_limit()` checked after each neuron, breaking work across multiple timer firings. The SNS governance has no equivalent protection.

**Failure mode:** If the instruction limit is exceeded, the IC rolls back the entire execution. `latest_reward_event` is not updated, so the next timer firing computes the same `new_rounds_count` and retries the same failing computation. This is a permanent liveness failure until a canister upgrade is deployed.

## Impact Explanation

This is a **High** severity finding. The impact is a permanent application-level liveness failure for SNS reward distribution: voting rewards are never distributed to SNS neurons, and the SNS reward mechanism is broken until a canister upgrade is deployed. This constitutes a significant SNS infrastructure security impact with concrete user harm (neurons that voted on proposals never receive maturity rewards). This matches the allowed impact: *"Significant... NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation

The conditions are reachable through normal SNS governance participation without any privileged access:

1. **Many neurons**: SNS participants can create neurons by staking governance tokens. A popular SNS can accumulate thousands of neurons.
2. **Many proposals in ReadyToSettle**: SNS participants can submit proposals up to the configured limit. With 50 proposals each voted on by 10,000 neurons, the ballot loop runs 500,000 times.
3. **Missed rounds**: Any canister upgrade (a routine operation) causes `new_rounds_count` to increment. Even a single missed round combined with a large P×N product can push execution over the 5B instruction limit on application subnets.

The NNS governance team explicitly identified this as a real risk and fixed it (Proposal 135702). The SNS governance has not received the same fix, making this a known-class vulnerability in an unpatched component.

## Recommendation

Apply the same batching mechanism used in NNS governance:
1. Move `distribute_rewards` to a dedicated timer task that persists intermediate `neuron_id_to_reward_shares` state across messages using stable storage.
2. Check `is_message_over_threshold` after each neuron update and break early, resuming in the next timer firing.
3. Replace the `for i in 1..=new_rounds_count` loop with a closed-form sum for the rewards purse calculation to avoid O(R) per-round iteration.
4. Cap the number of proposals processed per execution or process them in batches.

## Proof of Concept

1. Deploy an SNS with `round_duration_seconds = 86400` (1 day) and `max_number_of_proposals_with_ballots_to_settle = 50`.
2. Create 10,000 neurons and have them vote on 50 proposals (500,000 ballots total).
3. Upgrade the SNS governance canister (causing 1+ missed reward rounds), then let the timer fire.
4. The timer calls `run_periodic_tasks` → `distribute_rewards` with `new_rounds_count >= 1`.
5. The function executes: O(R) Decimal-arithmetic iterations (rounds purse) + 500,000 HashMap insertions (ballots) + 10,000 neuron mutations with Decimal division.
6. On an application subnet with a 5B instruction limit, this exceeds the limit. The timer execution fails with `CanisterInstructionLimitExceeded`, state rolls back, `latest_reward_event` is unchanged.
7. Every subsequent timer firing repeats the same failing computation. Reward distribution is permanently broken.

A deterministic integration test can reproduce this using PocketIC with a mocked instruction counter, or by benchmarking `distribute_rewards` with 50 proposals × 10,000 neurons to measure instruction consumption directly.
