### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows NEAR Payment Bypass — (`engine-precompiles/src/xcc.rs`)

---

### Summary

The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect payment from the caller before scheduling a NEAR cross-contract call. The EVM-level exit reason is checked, but the ABI-encoded boolean return value of `transferFrom` is never decoded or validated. If the wNEAR contract returns `false` instead of reverting on a failed transfer, the precompile silently proceeds as though payment was received, emitting the XCC promise log and scheduling the NEAR call without the engine having collected the required wNEAR.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, `run_with_handle` computes a `required_near` amount and, when non-zero, calls `transferFrom` on the wNEAR ERC-20 contract to move that amount from the caller to the engine's implicit address:

```rust
// lines 199–216
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e) => { ... }
    aurora_evm::ExitReason::Fatal(f) => { ... }
}
``` [1](#0-0) 

When `exit_reason` is `Succeed`, the `return_value` — which contains the ABI-encoded `bool` returned by `transferFrom(address,address,uint256)` — is captured but **never decoded or inspected**. The ERC-20 standard permits a compliant token to return `false` on a failed transfer rather than reverting. In that case, the EVM sub-call exits with `ExitReason::Succeed`, the match arm does nothing, and execution falls through to emit the XCC promise log:

```rust
// lines 226–237
let promise_log = Log { ... };
Ok(PrecompileOutput {
    logs: vec![promise_log],
    ...
})
``` [2](#0-1) 

The engine's post-EVM processing interprets this log as proof that the required NEAR was collected and proceeds to schedule the actual NEAR cross-contract call with NEAR attached — without the engine having received any wNEAR from the caller.

The `transfer_from_args` helper correctly encodes the `transferFrom` selector and arguments:

```rust
// lines 292–299
fn transfer_from_args(from: ethabi::Address, to: ethabi::Address, amount: ethabi::Uint) -> Vec<u8> {
    let args = ethabi::encode(&[...]);
    [&consts::TRANSFER_FROM_SELECTOR, args.as_slice()].concat()
}
``` [3](#0-2) 

The call is correctly constructed; the defect is exclusively in the missing check of the returned boolean.

---

### Impact Explanation

**Critical — Permanent fund freeze / theft of engine NEAR balance.**

The engine contract holds a NEAR balance used to fund cross-contract calls. When a user invokes the XCC precompile with a wNEAR token that returns `false` on a failed `transferFrom` (e.g., insufficient allowance, zero balance, or a non-standard wNEAR deployment), the engine:

1. Does **not** receive the required wNEAR from the caller.
2. **Does** schedule the NEAR cross-contract call, attaching NEAR from its own balance.

Repeated exploitation drains the engine's NEAR balance, eventually making it unable to pay for any further cross-contract calls or storage staking — a permanent freeze of the XCC subsystem and loss of engine-held NEAR.

---

### Likelihood Explanation

**Medium.** The standard wNEAR contract deployed on Aurora mainnet is an OpenZeppelin-based ERC-20 that reverts (rather than returning `false`) on transfer failure, so the bug is not immediately exploitable against the canonical deployment. However:

- The wNEAR address is a configurable storage value (`state::get_wnear_address`), not a compile-time constant.
- Any future wNEAR upgrade or alternative deployment that follows the ERC-20 spec's permitted `false`-return path would make this immediately exploitable by any unprivileged EVM user.
- The code pattern is unconditionally wrong regardless of the current wNEAR implementation. [4](#0-3) 

---

### Recommendation

After the `Succeed` branch, decode `return_value` and treat a `false` result as a revert:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool returned by transferFrom
    let transferred = return_value
        .get(31)  // last byte of the 32-byte ABI bool
        .copied()
        .unwrap_or(0);
    if transferred == 0 {
        return Err(PrecompileFailure::Revert {
            exit_status: aurora_evm::ExitRevert::Reverted,
            output: b"ERR_WNEAR_TRANSFER_FAILED".to_vec(),
        });
    }
}
```

Alternatively, use a safe-transfer wrapper that treats both `false` returns and missing return data as failures, mirroring OpenZeppelin's `SafeERC20.safeTransferFrom` pattern.

---

### Proof of Concept

1. Deploy a wNEAR-compatible ERC-20 contract on Aurora whose `transferFrom` always returns `false` (does not revert).
2. Call `set_wnear_address` (or arrange for the engine to use this contract as wNEAR).
3. From any EVM account with zero wNEAR balance, invoke the XCC precompile with a `CrossContractCallArgs::Eager` payload that attaches non-zero NEAR.
4. The `transferFrom` sub-call exits with `ExitReason::Succeed` and `return_value = [0u8; 32]` (ABI `false`).
5. The match arm `Succeed(_) => ()` does nothing; the promise log is emitted.
6. The engine's NEAR-side handler reads the log, sees `required_near > 0`, and schedules the cross-contract call, debiting the engine's own NEAR balance.
7. The attacker's wNEAR balance is unchanged; the engine's NEAR balance is reduced by `required_near`. [5](#0-4)
