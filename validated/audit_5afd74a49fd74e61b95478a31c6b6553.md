Audit Report

## Title
Unbounded Slot Table and Task Growth via `SLOT_TABLE_NO_LIMIT` Enables Single-Peer OOM DoS — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

Five of six P2P artifact channels are configured with `SLOT_TABLE_NO_LIMIT = usize::MAX`, rendering the per-peer slot table guard in `handle_slot_update_receive` permanently ineffective. A single authenticated subnet peer can send an unbounded stream of slot updates with unique slot numbers and artifact IDs, causing the victim replica's `slot_table`, `active_assembles` HashMap, and `artifact_processor_tasks` JoinSet to grow without bound until OOM kills the process and halts its participation in consensus.

## Finding Description

In `rs/replica/setup_ic_network/src/lib.rs`, two constants are defined:

```rust
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
``` [1](#0-0) 

The comment on line 72 explicitly acknowledges the threat model for ingress, but the same protection is absent for consensus, certifier, dkg, idkg, and https_outcalls — all five use `SLOT_TABLE_NO_LIMIT`: [2](#0-1) 

In `ConsensusManagerReceiver`, `slot_limit` is stored as a `usize` field and passed directly from the channel configuration: [3](#0-2) 

The guard in `handle_slot_update_receive` is:

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { … (true, None) }
Entry::Vacant(_) => { /* drop */ (false, None) }
``` [4](#0-3) 

With `slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always `true` — `peer_slot_table_len` is a `usize` that cannot reach `usize::MAX` before the process is killed by OOM. The drop branch is unreachable.

When `to_add` is `true` and the artifact ID has not been seen before, the code unconditionally inserts into `active_assembles` and spawns a new Tokio task: [5](#0-4) 

Each spawned `process_slot_update` task blocks waiting for either the peer to retract the slot or the artifact to be assembled. A malicious peer that never retracts and never delivers the artifact body keeps every task alive indefinitely, holding memory and a Tokio thread-pool slot.

The three structures that grow without bound per malicious peer are `slot_table[peer_id]` (one entry per unique `slot_number`), `active_assembles` (one entry per unique artifact `id`), and `artifact_processor_tasks` (one live Tokio task per unique artifact `id`): [6](#0-5) 

## Impact Explanation

This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** A single malicious node can crash every honest replica it is connected to by exhausting their memory, halting their participation in consensus and degrading subnet availability. The attack is not volumetric DDoS — it exploits a missing logical bound in the protocol's slot management layer.

## Likelihood Explanation

The attacker must be an authenticated subnet node (QUIC/TLS mutual authentication is required). This is explicitly within the scope of "Byzantine protocol peer below the consensus fault threshold" — a single compromised or malicious node operator suffices. No governance majority is required. The attack is trivially scriptable: send a tight loop of `SlotUpdate { slot_number: i, commit_id: i, update: Update::Id(random_unique_id()) }` messages over the existing P2P connection for any of the five unprotected artifact types, never sending the artifact body and never retracting the slot.

## Recommendation

Apply a finite per-peer slot limit to all artifact channels, not just ingress. The existing `SLOT_TABLE_LIMIT_INGRESS = 50_000` pattern is the correct model. Appropriate limits for each artifact type should be derived from the maximum number of in-flight artifacts that the protocol can legitimately produce per round (e.g., consensus has a bounded number of blocks/notarizations per height). Remove or rename `SLOT_TABLE_NO_LIMIT` to prevent future misuse.

## Proof of Concept

```rust
// Attacker is a legitimate authenticated subnet node.
// For each victim replica R connected to attacker:
for i in 0u64.. {
    send_slot_update(R, artifact_type = Consensus, SlotUpdate {
        slot_number: SlotNumber(i),
        commit_id:   CommitId(i),
        update:      Update::Id(random_unique_consensus_id()),
    });
    // Never send the artifact body; never retract the slot.
    // R's slot_table[attacker], active_assembles, and artifact_processor_tasks
    // grow by one entry per iteration until OOM kills the replica process.
}
```

A deterministic integration test can verify this by constructing a `ConsensusManagerReceiver` with `slot_limit = usize::MAX`, feeding it N synthetic `SlotUpdate` messages with distinct slot numbers and distinct IDs, and asserting that `active_assembles.len() == N` and `artifact_processor_tasks.len() == N` after processing, confirming unbounded growth.
