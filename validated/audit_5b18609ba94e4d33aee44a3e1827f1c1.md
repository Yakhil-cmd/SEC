Audit Report

## Title
SNS Governance `distribute_rewards` Settles Proposals Belonging to the Next Reward Round — (`File: rs/sns/governance/src/governance.rs`)

## Summary
`distribute_rewards` in SNS governance collects proposals to settle via `ready_to_be_settled_proposal_ids()`, which evaluates reward status against `now` (the live wall-clock time). The reward event it creates covers only up to `reward_event_end_timestamp_seconds`, which is strictly ≤ `now`. Any proposal whose voting deadline falls in the gap `(reward_event_end_timestamp_seconds, now]` is incorrectly swept into the current reward event, receiving premature maturity distribution and permanent ballot erasure. The NNS governance avoids this by passing a round-boundary-truncated timestamp to its equivalent function.

## Finding Description
`ready_to_be_settled_proposal_ids` captures `now` internally and filters proposals whose `reward_status(now) == ReadyToSettle`: [1](#0-0) 

`distribute_rewards` calls this function with no timestamp argument: [2](#0-1) 

It then computes `reward_event_end_timestamp_seconds` as the end of the most recently completed round — always ≤ `now`: [3](#0-2) 

The gap `now − reward_event_end_timestamp_seconds` equals `(now − reward_start) mod round_duration_seconds`, ranging from 0 to `round_duration_seconds − 1`. Any proposal whose deadline falls in this gap is `ReadyToSettle` as of `now` but belongs to the **next** round. It is included in `considered_proposals` and fully settled — maturity credited, `reward_event_end_timestamp_seconds` set — under a reward event whose `end_timestamp_seconds` predates the proposal's close time.

The NNS governance correctly passes a truncated round-boundary timestamp: [4](#0-3) 

computed by `most_recent_fully_elapsed_reward_round_end_timestamp_seconds`, which floors `now` to the nearest completed-round boundary: [5](#0-4) 

SNS governance has no equivalent guard. Its `ready_to_be_settled_proposal_ids` accepts no `as_of_timestamp_seconds` parameter: [6](#0-5) 

## Impact Explanation
This is a **High** severity finding matching: *"Significant SNS security impact with concrete user or protocol harm."*

1. **Premature maturity distribution.** Neurons that voted on proposals closing just after the round boundary receive maturity one full round early, before other participants have had a chance to vote in the proposal's proper round.
2. **Reward dilution.** The rewards purse is sized for `new_rounds_count` completed rounds. Settling additional next-round proposals spreads that fixed purse across more voting-power shares, reducing per-neuron maturity for all legitimate current-round voters.
3. **Permanent ballot data loss.** Once `reward_event_end_timestamp_seconds` is set on a proposal (marking it `has_been_rewarded()`), its ballots are cleared and cannot be recovered. [7](#0-6) 

## Likelihood Explanation
This is a deterministic, structural bug that fires on every invocation of `distribute_rewards` whenever at least one proposal's voting deadline falls in the window `(reward_event_end_timestamp_seconds, now]`. Because `should_distribute_rewards` fires as soon as `seconds_since_last_reward_event > round_duration_seconds`, and proposals can close at any second, this window is non-empty in virtually every real reward distribution cycle. [8](#0-7) 

No special privilege is required. Any SNS governance participant who can submit a proposal (or observe existing proposal deadlines) can time a proposal to close in this window.

## Recommendation
Mirror the NNS approach in SNS governance:

1. Add an `as_of_timestamp_seconds: u64` parameter to `ready_to_be_settled_proposal_ids` in `rs/sns/governance/src/governance.rs`.
2. Before calling `ready_to_be_settled_proposal_ids`, compute the round-boundary timestamp: `reward_event_end_timestamp_seconds` (already computed at line 5837) and pass it as the `as_of_timestamp_seconds` argument.
3. This ensures only proposals whose voting period closed on or before the end of the most recently completed reward round are settled in the current event.

## Proof of Concept
**Setup:**
- SNS with `round_duration_seconds = 604800` (7 days).
- Last reward event `end_timestamp_seconds = T₀`.
- Neuron A creates Proposal P with `initial_voting_period_seconds = 604801` at time `T₀ − 1`.
- Proposal P's deadline = `T₀ − 1 + 604801 = T₀ + 604800`.

**Execution:**
1. At `T₀ + 604800 + 2`, the governance timer fires `should_distribute_rewards` → `true`.
2. `distribute_rewards` runs: `new_rounds_count = floor(604802 / 604800) = 1`, `reward_event_end_timestamp_seconds = T₀ + 604800`.
3. `ready_to_be_settled_proposal_ids()` evaluates `reward_status(now = T₀ + 604800 + 2)`: Proposal P's deadline `T₀ + 604800 < now` → `accepts_vote = false` → `ReadyToSettle`.
4. Proposal P is included in `considered_proposals` and settled under the current reward event with `end_timestamp_seconds = T₀ + 604800`, even though P closed 1 second into round 2.
5. Neuron A receives full proportional maturity for Proposal P in round 1. In round 2, Proposal P is already `Settled`; no other neuron can earn rewards for it.

A deterministic integration test using PocketIC can reproduce this by: creating an SNS, advancing time to just past a round boundary, submitting a proposal timed to close in the gap, advancing time further to trigger `distribute_rewards`, and asserting that the proposal's `reward_event_end_timestamp_seconds` exceeds the reward event's `end_timestamp_seconds`.

    fn most_recent_fully_elapsed_reward_round_end_timestamp_seconds(&self) -> u64 {
        let now = self.env.now();

    }
