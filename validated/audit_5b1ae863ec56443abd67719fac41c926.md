### Title
Missing Caller Address Validation in `interop_root_reporter_event_hook` Enables Arbitrary Interop Root Injection - (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

### Summary

The `interop_root_reporter_event_hook` function validates only the event signature (`INTEROP_ROOT_ADDED_EVENT_SIG`) but never checks which contract emitted the event. Because the hook's function signature does not include a caller address parameter at all, any EVM contract can emit a correctly-formatted `InteropRootAdded` event and inject an arbitrary `InteropRoot` into the system-level `InteropRootStorage`. The injected root is then folded into the batch commitment's `interop_roots_rolling_hash`, corrupting the proof public input and potentially enabling cross-chain message forgery.

### Finding Description

`interop_root_reporter_event_hook` is the system-level hook that captures `InteropRootAdded` events and records them in `InteropRootStorage` for inclusion in the batch commitment. Its full validation logic is:

```
1. topics[0] == INTEROP_ROOT_ADDED_EVENT_SIG
2. data.len() == 96
3. data[0..32] decoded as offset == 32
4. data[32..64] decoded as len == 1
5. topics.len() == 3
``` [1](#0-0) 

There is no check on the address of the contract that emitted the event. The function signature is:

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<...>,
    data: &[u8],
    _caller_ee: u8,          // execution environment TYPE, not address
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
``` [2](#0-1) 

The `_caller_ee` parameter is the execution environment type (EVM, EraVM, etc.), not the emitter address. No caller address is passed to the hook at all, so the hook is structurally incapable of filtering by emitter. The only authorized emitter should be `L2_INTEROP_ROOT_STORAGE_ADDRESS`, but this is never enforced at the hook level.

Once the five format checks pass, the hook unconditionally calls:

```rust
system.io.add_interop_root(ExecutionEnvironmentType::NoEE, resources, InteropRoot { root, block_or_batch_number, chain_id })?;
``` [3](#0-2) 

`add_interop_root` delegates to `InteropRootStorage::push_root`, which is a plain list append with no deduplication and no authorization check:

```rust
pub fn push_root(&mut self, interop_root: InteropRoot) -> Result<(), SystemError> {
    self.list.push(interop_root, ());
    Ok(())
}
``` [4](#0-3) 

At the end of each block, `calculate_interop_roots_rolling_hash` iterates over every entry in `InteropRootStorage` and folds them into the batch commitment:

```rust
for root in roots {
    data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
    data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
    data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
    hasher.update(data);
    hasher.update(root.root.as_u8_ref());
    rolling_hash = hasher.finalize_reset().into()
}
``` [5](#0-4) 

This `interop_roots_rolling_hash` is then committed into the batch public input via `batch_data.apply_block(...)`: [6](#0-5) 

The invariant that only `L2_INTEROP_ROOT_STORAGE_ADDRESS` may contribute interop roots to the batch commitment is enforced only by the Solidity contract's own logic, not by the system hook. An attacker who bypasses the Solidity contract (by emitting the event directly from a different contract) breaks this invariant silently.

### Impact Explanation

An unprivileged user deploys a contract that emits a well-formed `InteropRootAdded(chain_id, block_or_batch_number, [fake_root])` event. The system hook injects the attacker-chosen `InteropRoot` into `InteropRootStorage`. This fake root is folded into `interop_roots_rolling_hash` and committed into the ZK proof public input. The settlement layer accepts the proof and records the corrupted rolling hash as canonical. The attacker can then construct a Merkle proof of inclusion against the fake root to forge a cross-chain message (e.g., a withdrawal), stealing funds from cross-chain bridges. Alternatively, injecting a root that conflicts with the settlement layer's expected value causes proof verification to fail, producing a sustained DoS on batch submission.

### Likelihood Explanation

The attack requires only deploying a contract and calling it in a normal L2 transaction — no privileged keys, no operator access, no oracle manipulation. The event signature is public. The five format checks (data length, offset, array length, topic count) are trivially satisfied by any contract that ABI-encodes the event correctly. The attack is reachable in a single transaction.

### Recommendation

Pass the emitter address to event hooks and validate it inside `interop_root_reporter_event_hook`:

```rust
if emitter_address != L2_INTEROP_ROOT_STORAGE_ADDRESS {
    return Err(internal_error!("InteropRootAdded from unauthorized emitter").into());
}
```

Alternatively, restructure the event hook dispatch to only invoke `interop_root_reporter_event_hook` for events whose emitter is `L2_INTEROP_ROOT_STORAGE_ADDRESS`, enforcing the invariant at the dispatch layer rather than relying on the hook itself.

### Proof of Concept

1. Attacker deploys `MaliciousEmitter` with bytecode that emits:
   `InteropRootAdded(attacker_chain_id, attacker_batch_number, [attacker_root])`
   where `attacker_root` is the Merkle root of a fake withdrawal message tree the attacker controls.

2. Attacker submits a normal L2 transaction calling `MaliciousEmitter`.

3. The EVM emits the event; `interop_root_reporter_event_hook` is invoked, passes all five format checks, and calls `push_root(InteropRoot { root: attacker_root, block_or_batch_number: attacker_batch_number, chain_id: attacker_chain_id })`.

4. At block finalization, `calculate_interop_roots_rolling_hash` folds the fake root into `interop_roots_rolling_hash`.

5. The ZK proof is generated and submitted to the settlement layer with the corrupted rolling hash as part of the public input. The proof is valid (the prover correctly proves the computation including the fake root).

6. The settlement layer accepts the proof and records the corrupted `interop_roots_rolling_hash`.

7. Attacker presents a Merkle inclusion proof against `attacker_root` to the cross-chain bridge contract, claiming a withdrawal that was never initiated, and drains bridge funds.

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
