### Title
ERC-20 Mirror Accounting Bug: Unused Tokens from `ft_transfer_call` Omni Exit Not Re-minted — (File: engine-precompiles/src/native.rs)

---

### Summary

When a user exits ERC-20 (or base ETH) tokens to NEAR using the Omni `ft_transfer_call` path in the `ExitToNear` precompile, the ERC-20 tokens are burned in the EVM for the full requested amount before the NEAR-side promise is settled. If the receiver's `ft_on_transfer` returns a non-zero unused amount, those NEP-141 tokens are refunded back to Aurora's NEP-141 balance by the NEP-141 contract — but no corresponding ERC-20 tokens are re-minted for the user. The result is a permanent accounting discrepancy: the user's ERC-20 tokens are destroyed for the full amount, yet only the "used" portion of NEP-141 tokens reaches the intended receiver. The unused portion is permanently locked in Aurora's NEP-141 balance with no ERC-20 counterpart.

---

### Finding Description

**Root cause — wrong operation ordering in the Omni exit path:**

In `ExitToNear::run()`, when the user supplies an Omni message (a colon-delimited string that is not `unwrap`), the precompile selects `ft_transfer_call` as the NEAR-side method: [1](#0-0) 

The `transfer_near_args` field is `None` for this branch. The callback-args struct is therefore: [2](#0-1) 

**Without the `error_refund` compile feature** (lines 452-453), both `refund` and `transfer_near` are `None`, so `callback_args` equals `default()` and the promise is created with **no callback at all** — the result of `ft_transfer_call` is entirely ignored.

**With the `error_refund` feature**, `refund` is populated, so a callback (`exit_to_near_precompile_callback`) is attached. However, that callback only handles the case where the promise **failed** (re-minting ERC-20 tokens via `refund_on_error`): [3](#0-2) 

When `ft_transfer_call` **succeeds** (even with unused tokens returned), the callback takes the `Successful` branch, checks only for `transfer_near` (which is `None` in the Omni case), and returns `None` — doing nothing. The actual return value of `ft_transfer_call` (the amount used) is never inspected.

**The NEP-141 standard guarantees** that any unused amount returned by the receiver's `ft_on_transfer` is transferred back to the sender (Aurora's account). Aurora's NEP-141 balance therefore increases by the unused amount, but the ERC-20 supply was already reduced by the full amount. The unused NEP-141 tokens are permanently stranded in Aurora's balance with no corresponding ERC-20 tokens.

The same flaw applies to the base-token Omni path in `exit_base_token_to_near`: [4](#0-3) 

---

### Impact Explanation

- **Without `error_refund`**: Any `ft_transfer_call` failure or partial success (unused tokens) causes permanent loss of the user's full exit amount. Impact: **Critical — Permanent freezing of funds**.
- **With `error_refund`**: Full failures are refunded, but partial successes (unused tokens) still cause permanent loss of the unused portion. Impact: **Critical — Permanent freezing of funds** for the unused amount.

The unused NEP-141 tokens accumulate in Aurora's NEP-141 balance with no ERC-20 counterpart and no recovery path. There is no admin function or user-callable method to re-mint ERC-20 tokens against these stranded NEP-141 tokens.

---

### Likelihood Explanation

The Omni `ft_transfer_call` path is the intended mechanism for users to interact with arbitrary NEAR DeFi protocols (e.g., DEXes, lending protocols, order books) from the EVM side. Any such protocol that implements `ft_on_transfer` with partial usage (e.g., partial order fills, deposit caps, slippage limits) will return a non-zero unused amount. This is a standard and common pattern in NEP-141 integrations. No special attacker capability is required — any unprivileged EVM user calling `exitToNear` with an Omni message to such a receiver triggers the bug.

---

### Recommendation

In `exit_to_near_precompile_callback`, when the promise result is `Successful`, parse the return value of `ft_transfer_call` (which per NEP-141 is the amount actually used, serialized as a JSON string). Compute `unused = original_amount - used_amount`. If `unused > 0`, call `refund_on_error` (or an equivalent re-mint path) to restore the unused ERC-20 tokens to the original sender's EVM address.

This requires:
1. Storing the original sender's EVM address and the full exit amount in `ExitToNearPrecompileCallbackArgs` (analogous to how `RefundCallArgs` stores the refund address and amount).
2. In the `Successful` branch of `exit_to_near_precompile_callback`, reading the `ft_transfer_call` return value and re-minting ERC-20 tokens for the unused portion. [5](#0-4) 

---

### Proof of Concept

1. User holds 100 ERC-20 tokens (backed by 100 NEP-141 tokens in Aurora's NEP-141 balance).
2. User calls `exitToNear` with flag `0x01` (ERC-20), amount = 100, message = `receiver.near:{"action":"swap","min_out":"50"}` (Omni path).
3. The ERC-20 contract burns 100 tokens and calls the `ExitToNear` precompile.
4. Precompile schedules `ft_transfer_call(receiver.near, 100, msg)` on the NEP-141 contract. No callback is attached (without `error_refund`) or a failure-only callback is attached (with `error_refund`).
5. `receiver.near`'s `ft_on_transfer` processes 60 tokens and returns `40` as unused.
6. NEP-141 contract refunds 40 tokens to Aurora's NEP-141 balance.
7. **Final state**: User has 0 ERC-20 tokens. Receiver has 60 NEP-141 tokens. Aurora's NEP-141 balance has 40

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
