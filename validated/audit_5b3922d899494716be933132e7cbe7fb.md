Audit Report

## Title
Existing SNS Swap Participant Incorrectly Blocked from Increasing Contribution at Maximum Capacity - (File: rs/sns/swap/src/swap.rs)

## Summary
In `refresh_buyer_token_e8s`, the maximum-participants capacity guard at lines 1179–1198 unconditionally applies a `+1` to `self.buyers.len()` without first checking whether the caller is already a participant. When the swap is exactly at capacity, an existing buyer who transfers additional ICP and calls `refresh_buyer_token_e8s` is rejected with an error that explicitly promises "existing participants may still increase their ICP participation amount" — a guarantee the code itself breaks. The ICP transferred remains locked in the subaccount until the swap closes.

## Finding Description
`refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` enforces a hard cap on direct participants to ensure enough SNS neurons can be minted:

```rust
// lines 1179–1198
let num_direct_participants = self.buyers.len() as u64;
if (num_direct_participants + 1) * num_sns_neurons_per_basket
    > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(format!(
        "… existing participants may still increase their ICP participation amount. …"
    ));
}
```

`self.buyers.len()` counts all current buyers, including the caller if they are already a participant. The `+1` is applied unconditionally, treating every caller as a new entrant. The actual membership check only appears at line 1275:

```rust
let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
if !is_preexisting_buyer {
    insert_buyer_into_buyers_list_index(buyer)…;
}
```

Because the capacity guard returns early before this check is reached, an existing participant's top-up is rejected even though no new participant slot would be consumed. The error message is directly contradicted by the code's own behavior.

## Impact Explanation
This matches the allowed High impact category: "Significant SNS security impact with concrete user or protocol harm." Existing participants who transfer additional ICP to their swap subaccount when the swap is at capacity cannot have that ICP accepted. The ICP is locked in the subaccount until the swap closes, at which point the participant must invoke `error_refund_icp` to recover it. The participant permanently loses the opportunity to increase their stake in that swap. The error message constitutes a broken protocol guarantee that actively misleads users about the behavior of the system.

## Likelihood Explanation
The triggering condition — a swap at maximum direct-participant capacity with an existing participant attempting a top-up — is a normal, expected scenario for popular SNS launches. The call path is fully unprivileged: any principal can invoke `refresh_buyer_token_e8s`. No special privileges, social engineering, or external compromise is required. Every existing participant in a full swap is affected on every top-up attempt.

## Recommendation
Move the `is_preexisting_buyer` check before the capacity guard and skip the guard for existing participants:

```rust
{
    let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
    if !is_preexisting_buyer {
        let num_direct_participants = self.buyers.len() as u64;
        let num_sns_neurons_per_basket = params
            .neuron_basket_construction_parameters
            .as_ref()
            .expect("neuron_basket_construction_parameters must be specified")
            .count;
        if (num_direct_participants + 1) * num_sns_neurons_per_basket
            > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
        {
            return Err(format!(
                "The swap has reached the maximum number of direct participants …",
            ));
        }
    }
}
```

The `is_preexisting_buyer` variable computed here can then be reused at line 1275, eliminating the redundant `contains_key` call.

## Proof of Concept
1. Deploy an SNS swap with `neuron_basket_construction_parameters.count = 1` and `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS = N`.
2. Fill the swap with exactly `N` distinct buyers, each contributing `min_participant_icp_e8s`. At this point `(N + 1) * 1 > N` is true.
3. As buyer #1 (already in `self.buyers`), transfer an additional `min_participant_icp_e8s` to the swap subaccount.
4. Call `refresh_buyer_token_e8s` as buyer #1.
5. **Observed**: returns `Err("The swap has reached the maximum number of direct participants (N) and does not accept new participants; existing participants may still increase their ICP participation amount.")`.
6. **Expected**: call succeeds and buyer #1's balance is increased, since no new participant slot is consumed.

The ICP transferred in step 3 is locked in the subaccount until the swap closes; the buyer must call `error_refund_icp` to recover it. This can be reproduced as a deterministic PocketIC integration test against the SNS swap canister. [1](#0-0) [2](#0-1)
