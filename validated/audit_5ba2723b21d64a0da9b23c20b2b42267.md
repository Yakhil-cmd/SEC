### Title
Ingress History Memory Pressure Allows Attacker to Prematurely Evict Victim's `Completed` Status to `Done` — (`rs/replicated_state/src/metadata_state.rs`)

### Summary
`IngressHistoryState::forget_terminal_statuses` evicts `Completed`/`Failed` ingress statuses to `Done` (dropping reply payloads) in oldest-first order when memory exceeds `ingress_memory_capacity`. An unprivileged attacker who controls a canister returning large reply payloads can flood the ingress history with `Completed` entries timestamped slightly after a victim's entry, causing the victim's `Completed` status to be evicted before the victim polls `read_state`.

### Finding Description

The eviction logic in `forget_terminal_statuses` has no per-user fairness or isolation. It scans `pruning_times` from `next_terminal_time` (oldest) upward and transitions every `Completed`/`Failed` entry it encounters to `Done` until `memory_usage ≤ target_size`. [1](#0-0) 

The check `if self.memory_usage <= target_size { break; }` is evaluated **before** processing each time bucket, not after. So the oldest bucket is always processed (evicted) first, regardless of which user owns those entries. [2](#0-1) 

Attack sequence:
1. Victim submits ingress message M_v; canister completes it with a reply at time T1. `Completed` status inserted into ingress history.
2. Attacker controls canister C_a. Attacker submits N ingress messages to C_a; C_a returns large reply payloads (up to the IC's max response size). These complete at time T2 > T1.
3. As attacker's `Completed` entries are inserted via `insert`, `memory_usage` exceeds `ingress_memory_capacity`.
4. `forget_terminal_statuses` is called with `target_size = ingress_memory_capacity`.
5. Scan starts at `next_terminal_time ≤ T1`. Victim's entry at T1 is encountered first and transitioned to `Done` (payload dropped).
6. Memory drops below target; loop breaks. Attacker's entries at T2 remain `Completed`.
7. Victim polls `read_state` / `get_ingress_status` and receives `Done` — reply payload is gone. [3](#0-2) 

### Impact Explanation
A legitimate user loses their canister reply permanently. The `Done` state carries no payload. The user cannot retry a read — the data is gone from replicated state. This is a denial-of-response: the user submitted a valid ingress message, paid fees, the canister executed correctly, but the user never receives the result.

### Likelihood Explanation
The attacker only needs:
- A canister under their control that returns large reply payloads (trivially deployable on mainnet).
- Enough ingress messages to push `memory_usage` above `ingress_memory_capacity`. Given the IC's max response size (~2 MB) and a finite `ingress_memory_capacity`, a small number of messages suffices.
- The victim's message to have completed slightly before the attacker's flood — a timing window that is easy to arrange or brute-force.

No privileged access, no consensus corruption, no threshold attack is required. This is fully reachable by an unprivileged user via normal ingress submission. [4](#0-3) 

### Recommendation
1. **Per-user/per-canister memory accounting**: Track ingress history memory per originating `user_id` or `receiver`. When eviction is needed, evict from the user/canister that is consuming the most memory first (largest-contributor-first), not globally oldest-first.
2. **Evict attacker's entries first**: When memory pressure is caused by a new insertion, prefer evicting entries from the same inserting party before touching other users' entries.
3. **Minimum retention guarantee**: Guarantee that a `Completed` entry is never evicted to `Done` within a minimum window (e.g., a few seconds) after it was set, giving the originating user time to poll.
4. **Cap per-user ingress history size**: Reject or immediately evict new `Completed` entries from a user who already holds a disproportionate share of ingress history memory.
