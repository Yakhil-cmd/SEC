### Title
`pubdata_used` Double-Counting in L1→L2 Transaction Native Resource Accounting - (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

In `process_l1_transaction`, the intrinsic pubdata overhead for the value-mint-to-sender operation is pre-charged from the native limit at transaction setup, and then the same actual pubdata bytes are charged a second time via `to_charge_for_pubdata` at the refund step. This double-charges the user for the value-mint pubdata in native resources, inflating `native_used`, inflating `delta_gas`, inflating `gas_used`, and reducing the refund paid back to the L1 depositor.

### Finding Description

**Step 1 – Intrinsic pre-charge (setup)**

In `create_resources_for_tx`, the full `intrinsic_pubdata` budget (which includes `TREASURY_BALANCE_INTRINSIC_PUBDATA` = 66 bytes and `ASSET_TRACKER_INTRINSIC_PUBDATA` = 65 bytes for the value-mint-to-sender) is subtracted from the user's native limit before execution begins:

```rust
// gas_helpers.rs:351-358
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
let native_limit = match native_limit.checked_sub(intrinsic_pubdata_overhead) {
    Some(val) => val,
    ...
};
``` [1](#0-0) 

The `intrinsic_pubdata` for L1 transactions is:

```
L1_TX_INTRINSIC_PUBDATA = 88 (log)
  + COINBASE_BALANCE_INTRINSIC_PUBDATA   (66 bytes)
  + TREASURY_BALANCE_INTRINSIC_PUBDATA   (66 bytes)  ← value mint
  + REFUND_RECIPIENT_BALANCE_INTRINSIC_PUBDATA (66 bytes)
  + ASSET_TRACKER_INTRINSIC_PUBDATA      (65 bytes)  ← value mint
``` [2](#0-1) 

**Step 2 – Value mint executes inside the execution frame**

The value mint to the sender (`mint_base_token`) runs inside the global execution frame (with a rollback handle). On success, the treasury balance diff and asset-tracker SSTORE are committed and become part of `net_pubdata_used()`. [3](#0-2) 

**Step 3 – Actual pubdata charged again at refund**

After execution, `check_enough_resources_for_pubdata` is called with `base_pubdata = None`, so it returns the full `net_pubdata_used()` counter — which already includes the value-mint pubdata:

```rust
// process_l1_transaction.rs:713-714
let (enough, to_charge_for_pubdata, pubdata_used) =
    check_enough_resources_for_pubdata(system, native_per_pubdata, resources, None)?;
``` [4](#0-3) 

`compute_gas_refund` then charges `to_charge_for_pubdata` (which includes the value-mint pubdata) from the user's remaining native resources:

```rust
// refund_calculation.rs:29
resources.charge_unchecked(&to_charge_for_pubdata);
``` [5](#0-4) 

**Step 4 – `pubdata_used` reporting also double-counts**

The final `ZkTxResult` adds `intrinsic_pubdata` on top of the already-inclusive `pubdata_used`:

```rust
// process_l1_transaction.rs:410
pubdata_used: pubdata_used + intrinsic_pubdata,
``` [6](#0-5) 

**Contrast with L2 transactions (correct path)**

For L2 transactions, `check_enough_resources_for_pubdata` is called with `base_pubdata = Some(context.validation_pubdata)`, which discounts the pubdata already covered by the intrinsic pre-charge. The `intrinsic_pubdata` is then added back only for the portion not yet measured. This avoids double-counting. [7](#0-6) 

### Impact Explanation

For every **successful** L1→L2 transaction with a non-zero value transfer, the user's `native_used` is inflated by:

```
overcharge_native = actual_value_mint_pubdata_bytes × native_per_pubdata
```

The maximum overcharge is `(66 + 65) × native_per_pubdata = 131 × native_per_pubdata` native units. This propagates through `delta_gas`:

```rust
// refund_calculation.rs:72
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
``` [8](#0-7) 

Since `native_per_pubdata = gas_per_pubdata × native_per_gas`, the overcharge in EVM gas units is:

```
overcharge_gas ≈ 131 × gas_per_pubdata
```

At a typical `gas_per_pubdata` of 800 gas/byte, this is ~104,800 gas overcharged per transaction. The user receives a correspondingly smaller refund from their L1 deposit. The `pubdata_used` field in `ZkTxResult` is also inflated, which can affect downstream DA accounting.

### Likelihood Explanation

Every successful L1→L2 priority transaction that transfers a non-zero value triggers this path. The value mint is the standard mechanism for bridging base tokens from L1. This is a high-frequency, normal-operation code path reachable by any unprivileged user who submits an L1→L2 transaction with `value > 0`.

### Recommendation

In `execute_l1_transaction_and_notify_result`, record the pubdata counter **before** the value mint executes, and pass it as `base_pubdata` to `check_enough_resources_for_pubdata`. This mirrors the L2 transaction pattern where `validation_pubdata` is used to exclude already-pre-charged pubdata from the actual charge:

```rust
// Record pubdata before value mint
let pubdata_before_value_mint = system.net_pubdata_used()?;

// ... value mint and execution ...

let (enough, to_charge_for_pubdata, pubdata_used) =
    check_enough_resources_for_pubdata(
        system,
        native_per_pubdata,
        resources,
        Some(pubdata_before_value_mint), // exclude pre-charged intrinsic pubdata
    )?;
```

Alternatively, do not add `intrinsic_pubdata` to `pubdata_used` in the final `ZkTxResult` when `pubdata_used` already reflects the full `net_pubdata_used()` counter.

### Proof of Concept

Consider an L1→L2 transaction with:
- `gas_limit = 200_000`
- `gas_price = 1_000_000_000` (1 gwei)
- `gas_per_pubdata = 800`
- `native_price = 1_000_000` (so `native_per_gas = 1_000`)
- `native_per_pubdata = 800_000`
- `value = 1 ETH` (non-zero, triggers value mint)

**Setup:**
- `intrinsic_pubdata = L1_TX_INTRINSIC_PUBDATA` (includes 131 bytes for value mint)
- `intrinsic_pubdata_overhead = 131 × 800_000 = 104_800_000` native units pre-charged
- Starting native = `200_000 × 1_000 - 104_800_000 - intrinsic_computational_native`

**Execution:**
- Value mint generates ~131 bytes of actual pubdata (treasury + asset tracker diffs)
- `net_pubdata_used()` = 131 bytes (from value mint) + execution pubdata

**Refund:**
- `to_charge_for_pubdata` = `131 × 800_000 + execution_pubdata × 800_000`
- This charges the 131-byte value-mint pubdata **again** from remaining native
- `native_used` is inflated by `131 × 800_000 = 104_800_000` native units
- `delta_gas = 104_800_000 / 1_000 = 104_800` extra gas charged
- User's refund is reduced by `104_800 × 1_000_000_000 = 104_800_000_000` wei (~0.0001 ETH)

The user is systematically overcharged ~104,800 gas on every successful non-zero-value L1→L2 transaction.
