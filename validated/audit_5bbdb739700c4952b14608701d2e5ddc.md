### Title
Unchecked Exit-Precompile Return Value Burns ERC-20 Tokens Without NEAR Transfer — (`etc/eth-contracts/contracts/EvmErc20.sol`, `EvmErc20V2.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` implement the `IExit` interface by burning ERC-20 tokens and then calling the Aurora exit precompile via inline assembly. The assembly block captures the call's success flag in `res` but never checks it. When the precompile returns `ExitError` (e.g., invalid NEAR recipient account ID, out-of-gas in the sub-call), the burn succeeds and the transaction completes normally, but no NEAR-side transfer is scheduled. The user's tokens are permanently destroyed with no corresponding credit on NEAR.

---

### Finding Description

In `EvmErc20.sol` (lines 53–63) and the identical pattern in `EvmErc20V2.sol` (lines 53–68):

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure path
    }
}
```

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `ExitError` (not `ExitFatal`) for several reachable conditions:

| Condition | Error returned |
|---|---|
| Recipient is not a valid NEAR account ID | `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")` |
| Amount > `u128::MAX` | `ExitError::Other("ERR_INVALID_AMOUNT")` |
| Out of gas in sub-call | `ExitError::OutOfGas` |

In SputnikVM (Aurora's EVM), `ExitError` from a precompile causes the sub-call to return `0` in the success flag — standard EVM semantics. The calling contract's assembly block captures this in `res` but does not branch on it, so execution continues past the failed call. The `_burn` is **not** reverted. The tokens are gone.

This is distinct from the paused-precompile path, which returns `ExitFatal::Other("ERR_PAUSED")` and causes the entire transaction to revert (confirmed by `test_executing_paused_precompile_throws_error`). The `ExitError` path has no such safety net.

The same unchecked pattern exists in `withdrawToEthereum` in both contracts, calling the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`.

---

### Impact Explanation

**Critical — Permanent freezing (destruction) of funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` with a recipient string that fails NEAR account-ID validation (e.g., contains uppercase letters, is too long, or is empty) will have their ERC-20 tokens burned with no corresponding NEP-141 credit issued on NEAR. The tokens are irrecoverably destroyed. There is no refund path in this code flow (the `error_refund` callback mechanism in the precompile is only triggered when the precompile itself schedules a callback, which it cannot do if it returns `ExitError` before reaching the promise-creation logic).

---

### Likelihood Explanation

**Low-Medium.** Any unprivileged EVM user can call `withdrawToNear` directly. NEAR account IDs have non-obvious validity rules (lowercase only, specific character set, length limits). A user who mistypes a recipient or passes a raw Ethereum address string as the NEAR recipient will silently lose funds. No admin action or special privilege is required.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile rejects the call for any reason, the `_burn` is also reverted and the user's tokens are preserved.

---

### Proof of Concept

1. Deploy an `EvmErc20` token (via `deploy_erc20_token` on the Aurora engine).
2. Mint tokens to address `A`.
3. From address `A`, call `withdrawToNear("INVALID ACCOUNT ID WITH SPACES", balance)`.
4. The `_burn` executes, reducing `A`'s balance to zero.
5. The precompile at `0xe921...` receives the input, calls `parse_recipient`, which calls `AccountId::parse` on `"INVALID ACCOUNT ID WITH SPACES"`, which fails, returning `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`.
6. The assembly `call` returns `res = 0`; since `res` is unchecked, `withdrawToNear` returns normally.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
