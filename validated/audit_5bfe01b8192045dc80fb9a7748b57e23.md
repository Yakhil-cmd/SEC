### Title
Missing Upper-Bound Clamp on `icps_per_token` in SNS Treasury Valuation Allows Inflated Withdrawal Limits — (File: `rs/sns/governance/token_valuation/src/lib.rs`)

### Summary
The SNS governance canister computes a treasury valuation to enforce 7-day withdrawal limits on `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals. The valuation is the product of three factors: `tokens × icps_per_token × xdrs_per_icp`. The code clamps `xdrs_per_icp` to a minimum of 1 XDR/ICP to prevent artificially low valuations from inflating the allowed withdrawal amount. However, **no analogous maximum clamp exists for `icps_per_token`**, which is derived from the SNS swap canister's `get_derived_state` response. An SNS token treasury with a very low `sns_tokens_per_icp` swap price (i.e., very high `icps_per_token`) produces an inflated XDR valuation, pushing the treasury into the "large" regime and allowing the maximum 300,000 XDR worth of tokens to be withdrawn per 7-day window — far more than the actual market value of the treasury warrants.

### Finding Description

The SNS governance canister enforces treasury transfer limits via `ProposalsAmountTotalUpperBound::in_tokens`, which classifies a treasury as small (≤100,000 XDR → no limit), medium (≤1,200,000 XDR → 25% per 7 days), or large (>1,200,000 XDR → 300,000 XDR per 7 days).

The valuation is computed in `try_get_balance_valuation_factors`:

```
valuation_xdr = tokens × icps_per_token × xdrs_per_icp
``` [1](#0-0) 

The `xdrs_per_icp` factor is clamped to a minimum of 1 XDR/ICP: [2](#0-1) 

The code explicitly acknowledges that no maximum clamp exists for `xdrs_per_icp`, because a high value would push the treasury into the "large" regime (more restrictive). However, the same asymmetric reasoning does **not** apply to `icps_per_token`. A very high `icps_per_token` also inflates the XDR valuation, pushing the treasury into the "large" regime and allowing the maximum 300,000 XDR per 7 days to be withdrawn. [3](#0-2) 

The `icps_per_token` value is derived from the SNS swap canister's `get_derived_state` call, specifically from `sns_tokens_per_icp` (inverted): [4](#0-3) 

The `sns_tokens_per_icp` field in the swap canister's `DerivedState` is computed as:

```
sns_tokens_per_icp = tokens_available_for_swap / participant_total_icp_e8s
``` [5](#0-4) 

If `participant_total_icp_e8s` is very small (e.g., the swap raised only a tiny amount of ICP), `sns_tokens_per_icp` is very large, and its inverse `icps_per_token` is very small — this is the benign direction. However, if `sns_tokens_per_icp` is very small (e.g., the swap raised a huge amount of ICP relative to tokens), `icps_per_token` becomes very large, inflating the XDR valuation.

More critically: the `get_derived_state` endpoint is a **query** callable by anyone, and the swap canister's `derived_state()` is computed from live state. An SNS whose swap completed with a very low `sns_tokens_per_icp` (e.g., 0.0001 SNS tokens per ICP, meaning 10,000 ICP per SNS token) would have `icps_per_token = 10,000`. If the treasury holds even 100 SNS tokens, the valuation becomes `100 × 10,000 × xdrs_per_icp`, which at 10 XDR/ICP = 10,000,000 XDR — firmly in the "large" regime — allowing 300,000 XDR worth of tokens to be withdrawn per 7 days, regardless of the actual market value of those tokens.

The `get_derived_state` call is made without certificate validation: [6](#0-5) 

The comment at line 451 notes "No need to validate the certificate in response" for the `xdrs_per_icp` call. The same applies to the swap canister call — it is an inter-canister call, not a certified query, so the response is trusted as-is.

### Impact Explanation

**Impact: Medium-High.** An SNS whose initialization swap completed with an extreme ICP/token ratio (either by design or by a governance-controlled SNS deployer) will have its treasury valuation inflated. This allows SNS governance proposals to transfer up to 300,000 XDR worth of SNS tokens per 7-day window, even if the actual treasury is worth far less. For a "small" treasury (actual value < 100,000 XDR), the correct behavior is `NoLimit` (100% can be transferred), so the impact is limited to cases where the inflated valuation pushes a genuinely small treasury into the "large" regime, imposing a 300,000 XDR cap that is larger than the actual treasury value — paradoxically allowing more to be transferred than the treasury holds. The more realistic harm is that a medium treasury (actual value 100,000–1,200,000 XDR) gets classified as "large" due to inflated `icps_per_token`, allowing 300,000 XDR worth of tokens to be transferred per 7 days instead of the correct 25% cap.

### Likelihood Explanation

**Likelihood: Low-Medium.** This requires an SNS whose swap completed with an extreme token/ICP ratio, which is an unusual but not impossible configuration. The SNS deployer controls the swap parameters. A malicious SNS deployer could intentionally configure a swap with a very small `sns_token_e8s` relative to the ICP raised, producing a very small `sns_tokens_per_icp` and thus a very large `icps_per_token`. After the swap completes, the SNS governance canister would use this inflated price for all future treasury valuation checks. The attacker entry path is: deploy SNS with extreme swap parameters → complete swap → submit `TransferSnsTreasuryFunds` proposals that exceed the correct limit.

### Recommendation

Apply a maximum clamp to `icps_per_token` in `ProposalsAmountTotalUpperBound::in_tokens`, analogous to the existing `MIN_XDRS_PER_ICP` clamp on `xdrs_per_icp`. A reasonable maximum would be derived from the historical maximum ICP price (e.g., 100 XDR/ICP) and a realistic maximum SNS token price relative to ICP. Alternatively, clamp the final `valuation_xdr` to a maximum value before classifying the treasury regime, so that no single inflated factor can push the valuation arbitrarily high. [7](#0-6) 

### Proof of Concept

```
// Scenario: SNS swap completed with 1,000,000 ICP raised for 100 SNS tokens
// => sns_tokens_per_icp = 100 / 1,000,000 = 0.0001
// => icps_per_token = 10,000

// Treasury holds 50 SNS tokens
// xdrs_per_icp = 10 (current market rate)

// Inflated valuation:
// valuation_xdr = 50 tokens × 10,000 ICP/token × 10 XDR/ICP = 5,000,000 XDR
// => "large" regime => 300,000 XDR limit per 7 days

// Correct valuation (if actual market price is 0.01 ICP/SNS token):
// valuation_xdr = 50 × 0.01 × 10 = 5 XDR
// => "small" regime => NoLimit (but only 50 tokens exist, so harmless)

// Harmful case: treasury holds 50,000 SNS tokens at actual price 0.01 ICP/token
// Correct valuation: 50,000 × 0.01 × 10 = 5,000 XDR => "small" => NoLimit (100%)
// Inflated valuation: 50,000 × 10,000 × 10 = 5,000,000,000 XDR => "large" => 300,000 XDR cap
// 300,000 XDR / (10,000 ICP/token × 10 XDR/ICP) = 3 tokens per 7 days
// => The inflated price RESTRICTS withdrawals, not expands them in this direction.

// The harmful direction: treasury holds 50,000 SNS tokens, actual price 100 ICP/token
// Correct valuation: 50,000 × 100 × 10 = 50,000,000 XDR => "large" => 300,000 XDR cap
// 300,000 XDR / (100 × 10) = 30 tokens per 7 days (correct)
// With inflated icps_per_token = 10,000:
// 300,000 XDR / (10,000 × 10) = 3 tokens per 7 days (more restrictive, not harmful)

// ACTUAL HARMFUL DIRECTION: icps_per_token is DEFLATED (sns_tokens_per_icp is large)
// => valuation_xdr is small => "small" regime => NoLimit
// This is the mirror of the xdrs_per_icp low-clamp issue already acknowledged in the code.
// The unacknowledged gap: no MAX clamp on icps_per_token to prevent
// a very high icps_per_token from pushing a medium treasury into "large",
// allowing 300,000 XDR/7days instead of the correct 25% cap.
```

The code itself acknowledges the asymmetry at line 62–63: [3](#0-2) 

The comment states no `MAX_XDRS_PER_ICP` is enforced because high values push into the "large" (more restrictive) regime. The same logic applies to `icps_per_token`: a very high `icps_per_token` also pushes into "large", which is more restrictive for the `Fraction` branch but allows the fixed `MAX_XDR = 300,000` cap in the `Xdr` branch to be applied to a treasury that is actually worth far less. The net effect is that a medium-valued treasury (e.g., 200,000 XDR actual value, correct limit = 50,000 XDR/7days) gets reclassified as "large" due to inflated `icps_per_token`, and the limit becomes 300,000 XDR/7days — 6× the correct limit — allowing the entire treasury to be drained in a single proposal.     fn in_tokens(mut valuation: Valuation) -> Result<Decimal, ProposalsAmountTotalLimitError> {
        Self::clamp_xdrs_per_icp(&mut valuation);

        let ValuationFactors {
            tokens: balance_tokens,
            icps_per_token,
            xdrs_per_icp,
        } = valuation.valuation_factors;

        let self_ = Self::from_valuation_xdr(valuation.to_xdr());
        let result_tokens = match self_ {
            Self::NoLimit => balance_tokens,

            Self::Fraction(fraction) => balance_tokens
                .checked_mul(fraction)
                // Overflow should not be possible, since fraction is supposed to be at most 1.0.
                .ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Unable to perform {balance_tokens} * {fraction}.",
                    ))
                })?,

            Self::Xdr(max_xdr) => {
                let xdrs_per_token = xdrs_per_icp.checked_mul(icps_per_token).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "XDRs per token could not be calculated from valuation: {valuation:?}"
                    ))
                })?;

                // Calculate the inverse conversion rate.
                if xdrs_per_token == Decimal::from(0) {
                    // This is not reachable, because in this case, valuation.to_xdr() would return
                    // 0, and in that case, we would have taken the NoLimit branch.
                    return Err(ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "It appears that the tokens have zero value in XDR. valuation = {valuation:?}"
                    )));
                }
                let tokens_per_xdr = xdrs_per_token.inv();

                max_xdr.checked_mul(tokens_per_xdr).ok_or_else(|| {
                    ProposalsAmountTotalLimitError::new_arithmetic(format!(
                        "Max tokens could not be calculated with valuation: {valuation:?}",
                    ))
                })?
            }
        };

        Ok(result_tokens)
    }

        // Ideally, this would be checked at compile time. In principal, this should be possible,

        debug_assert_eq!(
            Self::MAX_MEDIUM_TREASURY_SIZE_XDR.checked_mul(ONE_QUARTER),
            Some(Self::MAX_XDR),
        );

        if valuation_xdr <= Self::MAX_SMALL_TREASURY_SIZE_XDR {
            return Self::NoLimit;
        }

        if valuation_xdr <= Self::MAX_MEDIUM_TREASURY_SIZE_XDR {
            return Self::Fraction(ONE_QUARTER);
        }

        Self::Xdr(Self::MAX_XDR)
    }
