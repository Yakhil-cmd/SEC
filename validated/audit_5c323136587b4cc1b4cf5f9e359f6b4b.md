Audit Report

## Title
Unbounded ingress_expiries loop in `construction_payloads` causes OOM crash of Rosetta node — (File: `rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function accepts attacker-controlled `ingress_start` and `ingress_end` values from `ConstructionPayloadsRequestMetadata` and feeds them directly into an unbounded `while ingress_start < ingress_end` loop with no cap on the number of iterations. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes approximately 76 billion loop iterations, exhausting process memory and crashing the Rosetta node.

## Finding Description
The function signature at [1](#0-0)  accepts `metadata: Option<ConstructionPayloadsRequestMetadata>`. The `ingress_start` and `ingress_end` fields are plain `Option<u64>` with no range validation, as confirmed at [2](#0-1) .

The two guards before the loop are:
- Guard 1 at [3](#0-2)  rejects only if `ingress_start >= ingress_end` — passes with `0 < u64::MAX`.
- Guard 2 at [4](#0-3)  rejects only if `ingress_end < now + ingress_interval` — passes with `ingress_end=u64::MAX`.

Neither guard limits the *size* of the window `ingress_end - ingress_start`.

The unbounded loop at [5](#0-4)  then runs with `ingress_interval = (MAX_INGRESS_TTL − PERMITTED_DRIFT).as_nanos()`. From `rs/limits/src/lib.rs`, `MAX_INGRESS_TTL` is 5 minutes and `PERMITTED_DRIFT` is 1 minute, giving `ingress_interval ≈ 240,000,000,000 ns`. With `ingress_end=u64::MAX`, iterations ≈ `2^64 / 240_000_000_000 ≈ 76.8 billion`.

The `INGRESS_INTERVAL_OVERLAP` constant is 120 seconds [6](#0-5) , which reduces the effective step size slightly, making the iteration count even larger.

Each iteration pushes one entry into `ingress_expiries`, which is then passed to `handle_construction_payloads`, where a full `EnvelopeContent::Call` struct and a `SigningPayload` are allocated per entry. [7](#0-6)  The process OOMs long before completing.

## Impact Explanation
This matches the allowed High impact: **Application/platform-level DoS, crash** of the ICRC-1 Rosetta node. A single malformed request crashes the entire Rosetta process, making the ICRC-1 Rosetta API completely unavailable until manually restarted. The Rosetta node is a single-process service with no crash recovery. The IC itself is unaffected, but all clients depending on the Rosetta API lose access. This is a concrete, non-hypothetical availability impact on a listed in-scope financial integration component.

## Likelihood Explanation
The `/construction/payloads` endpoint is a public, unauthenticated Rosetta API endpoint. No credentials, governance role, or special access are required. The attack requires a single HTTP POST with two crafted integer fields (`ingress_start=0`, `ingress_end=18446744073709551615`). It is trivially reproducible and repeatable — the node can be crashed again immediately after restart.

## Recommendation
Add an explicit cap on the number of ingress expiries before the while loop:

```rust
const MAX_INGRESS_EXPIRIES: usize = 1000;

let mut ingress_expiries = vec![];
while ingress_start < ingress_end {
    if ingress_expiries.len() >= MAX_INGRESS_EXPIRIES {
        return Err(Error::processing_construction_failed(
            &"ingress window too large: exceeds maximum allowed expiry count"
        ));
    }
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
```

Alternatively, validate `ingress_end - ingress_start` does not exceed a reasonable maximum (e.g., 24 hours in nanoseconds) before entering the loop, consistent with the documented intent in `ConstructionPayloadsRequestMetadata` ("Must be within 24 hours from created_at_time"). [8](#0-7) 

## Proof of Concept

```rust
#[test]
fn test_construction_payloads_unbounded_ingress_window() {
    use std::time::SystemTime;
    let metadata = ConstructionPayloadsRequestMetadata {
        ingress_start: Some(0),
        ingress_end: Some(u64::MAX),
        created_at_time: None,
        memo: None,
    };
    // now=UNIX_EPOCH so now.as_nanos()=0, guard 2: u64::MAX >= 0 + interval => passes
    let result = construction_payloads(
        vec![/* valid transfer operation */],
        Some(metadata),
        &Principal::anonymous(),
        vec![/* valid public key */],
        SystemTime::UNIX_EPOCH,
    );
    // Expected: Err("ingress window too large")
    // Actual (unfixed): process hangs and OOMs
    assert!(result.is_err());
}
```
