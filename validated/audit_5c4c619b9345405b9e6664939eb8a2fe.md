### Title
`native_per_pubdata` Rounds to Zero via Integer Division, Making Pubdata Free for Users — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

In ZKsync OS's ZK transaction validation, `native_per_pubdata` is computed as an integer division `pubdata_price / native_price`. When `pubdata_price < native_price`, this division truncates to zero, silently making all pubdata free for the transaction. Any user who submits a transaction under these block-level pricing conditions can write unlimited storage slots (up to the block pubdata limit) without paying the associated L1 data cost, causing the operator to bear those costs uncompensated.

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`pubdata_price` and `native_price` are oracle-supplied `U256` values from `BlockMetadataFromOracle` (defined in `zk_ee/src/system/metadata/zk_metadata.rs`, lines 123–124). `wrapping_div` on `U256` is plain integer division with truncation. When `pubdata_price < native_price`, the result is `0`.

This zero value propagates through two distinct charging paths:

**Path 1 — Intrinsic pubdata charge** (`gas_helpers.rs`, line 352):
```rust
let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
```
With `native_per_pubdata = 0`, the intrinsic pubdata overhead is zero regardless of `intrinsic_pubdata`.

**Path 2 — Runtime pubdata charge** (`gas_helpers.rs`, lines 430–432):
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
```
With `native_per_pubdata = 0`, `native = 0` for any amount of pubdata spent. The function `get_resources_to_charge_for_pubdata` returns zero cost regardless of how many storage slots were written.

By contrast, `refund_calculation.rs` line 69 explicitly guards against a zero `native_per_gas`:
```rust
let delta_gas = if native_per_gas == 0 { 0 } else { ... };
```
No equivalent guard exists for `native_per_pubdata`.

The `native_per_pubdata` value is stored in `TxContextForPreAndPostProcessing` and reused throughout the transaction lifecycle (pre-charge, execution pubdata check, post-execution refund), so the zero value affects every pubdata accounting step for the transaction.

### Impact Explanation

When `pubdata_price < native_price` (a realistic condition — e.g., `pubdata_price = 5`, `native_price = 10`), any user submitting a ZK L2 transaction pays **zero** for pubdata. The user can write as many storage slots as the block's `pubdata_limit` allows, generating real L1 data costs that the operator must pay without any compensation from the transaction sender. This is a direct resource accounting bug: pubdata fees that should be collected are silently discarded due to integer truncation, causing financial loss to the operator proportional to the pubdata written.

### Likelihood Explanation

`pubdata_price` and `native_price` are independently set oracle values in `BlockMetadataFromOracle`. Their ratio is not guaranteed to be ≥ 1. During normal operation — especially when L1 data costs are low relative to native computational costs, or during price fluctuations — `pubdata_price < native_price` is a realistic steady-state condition, not an edge case. Any user can observe the block metadata and submit pubdata-heavy transactions (e.g., many SSTORE operations) to exploit the zero cost.

### Recommendation

Replace the bare integer division with a ceiling division or a minimum-of-1 guard when `pubdata_price > 0`:

```rust
// Option A: ceiling division (charges at least 1 native unit per pubdata byte when pubdata_price > 0)
let native_per_pubdata = if pubdata_price.is_zero() {
    0u64
} else {
    u256_try_to_u64(&pubdata_price.div_ceil(native_price))
        .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?
};
```

This mirrors the pattern already used for `native_per_gas` at line 135 of the same file, which uses `div_ceil`:
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```
Notably, `native_per_gas` already uses `div_ceil` to avoid rounding to zero, but `native_per_pubdata` uses plain `wrapping_div` — an inconsistency that is the direct root cause of this bug.

### Proof of Concept

**Setup:** Block oracle provides `pubdata_price = 5`, `native_price = 10`.

**Step 1:** `native_per_pubdata = 5 / 10 = 0` (integer truncation at line 142 of `validation_impl.rs`).

**Step 2:** User submits a transaction that performs 1000 SSTORE operations, generating 32,000 bytes of pubdata.

**Step 3:** `get_resources_to_charge_for_pubdata` computes `native = 32000 * 0 = 0`. The transaction passes the pubdata resource check with zero cost.
