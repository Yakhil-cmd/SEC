Audit Report

## Title
Unprivileged Caller Can Prematurely Fail SNS Upgrade Proposals via `fail_stuck_upgrade_in_progress` - (File: rs/sns/governance/canister/canister.rs)

## Summary
The `fail_stuck_upgrade_in_progress` update method on the SNS governance canister exposes no caller access control, allowing any unprivileged principal to invoke it once `mark_failed_at_seconds` has elapsed. If an upgrade completes in the final seconds before the 5-minute deadline but the periodic `check_upgrade_status` task has not yet run to confirm it, the attacker can force the proposal to be marked failed and leave `deployed_version` permanently stale at the pre-upgrade value, while the canister on-chain is already running the new WASM.

## Finding Description
The canister-level handler in `rs/sns/governance/canister/canister.rs` (lines 526–535) performs no caller check before delegating to the internal implementation:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(...))
}
```

The internal `fail_stuck_upgrade_in_progress` in `rs/sns/governance/src/governance.rs` (lines 6342–6357) fires whenever `now > pending_version.mark_failed_at_seconds`, calling `complete_sns_upgrade_to_next_version` with `deployed_version: None`:

```rust
if now > pending_version.mark_failed_at_seconds {
    self.complete_sns_upgrade_to_next_version(
        pending_version.proposal_id,
        status,
        message,
        None,   // deployed_version not updated
    );
}
```

`complete_sns_upgrade_to_next_version` (lines 6309–6313) clears `pending_version` and only updates `deployed_version` when `Some(version)` is passed; with `None` it leaves `deployed_version` at the pre-upgrade value:

```rust
self.proto.pending_version = None;
if let Some(deployed_version) = deployed_version {
    self.proto.deployed_version.replace(deployed_version);
}
```

The deadline is set to exactly 5 minutes after upgrade initiation (lines 2894–2899 and 5636–5641). The periodic task (`RUN_PERIODIC_TASKS_INTERVAL = 10s`, canister.rs line 76) calls `check_upgrade_status` every 10 seconds while `pending_version` is `Some`. The attack window is the gap between `mark_failed_at_seconds` elapsing and the next periodic task run confirming the upgrade — at most ~10 seconds, but only reachable if the upgrade completes in the final seconds before the deadline.

The existing test at `fail_stuck_upgrade_in_progress_tests.rs` lines 268–272 explicitly confirms the post-call state: `deployed_version` remains at `SNS_VERSION_1` (old version) after the call, even though the canister is running the new version.

## Impact Explanation
After a successful exploit: (1) the associated upgrade proposal is permanently marked failed via `set_proposal_execution_status` with an `ErrorType::External` error — SNS proposals are immutable once settled; (2) `deployed_version` remains at the pre-upgrade value while the canister runs the new WASM; (3) subsequent calls to `initiate_upgrade_if_sns_behind_target_version` use the stale `deployed_version` to compute upgrade steps, potentially triggering a redundant re-upgrade of the same canister. This constitutes a significant SNS governance state inconsistency with concrete protocol harm: a passed governance proposal is permanently recorded as failed, and the SNS version tracking diverges from reality. This matches the allowed impact: **High — Significant SNS security impact with concrete user or protocol harm** ($2,000–$10,000), though constrained by the narrow timing window, placing it at the lower end.

## Likelihood Explanation
The attack requires the upgrade to complete within the last ~10 seconds before the 5-minute deadline, and the attacker must submit the ingress call in that window. The `mark_failed_at_seconds` value is publicly readable via `get_running_sns_version` (canister.rs lines 504–524), so the attacker can observe the exact deadline. No special privileges are required — any principal including anonymous can call the method. The window is narrow and requires the upgrade to take nearly exactly 5 minutes, which is atypical. Likelihood: **Low-Medium** — deterministically reachable with no privilege requirement, but requires precise timing and an unusually slow upgrade.

## Recommendation
Add a caller access control check at the canister level in `rs/sns/governance/canister/canister.rs` before delegating to the internal implementation. At minimum, restrict to canister controllers:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    assert!(
        ic_cdk::api::is_controller(&ic_cdk::api::msg_caller()),
        "Only controllers may call fail_stuck_upgrade_in_progress"
    );
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(
        sns_gov_pb::FailStuckUpgradeInProgressRequest::from(request),
    ))
}
```

Alternatively, restrict to the SNS root canister ID analogously to `assert_eq_governance_canister_id` used in `rs/sns/root/canister/canister.rs` (lines 419–426).

## Proof of Concept
1. Submit an `UpgradeSnsToNextVersion` proposal to an SNS and vote it through.
2. Query `get_running_sns_version` to read `pending_version.mark_failed_at_seconds` (T+300s).
3. Monitor the SNS canister's running WASM hash (e.g., via `get_sns_canisters_summary`) to detect when the upgrade completes on-chain.
4. If the upgrade completes within the last ~10 seconds before T+300s, immediately send an ingress update call from any principal (including anonymous): `fail_stuck_upgrade_in_progress({})`.
5. Verify: `pending_version` is now `None`, the proposal shows `failed_timestamp_seconds != 0` with `failure_reason` containing "manually aborted", and `deployed_version` still reflects the pre-upgrade version hash — while the canister is running the new WASM.
6. A deterministic integration test can reproduce this by setting `mark_failed_at_seconds = env.now() - 1` with a `pending_version` whose `target_version` matches the currently running version, then calling `fail_stuck_upgrade_in_progress` and asserting `deployed_version` is not updated.
