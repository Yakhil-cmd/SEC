Audit Report

## Title
Cross-SNS Neurons' Fund Settlement via Insufficient Swap Canister Authorization — (`rs/nns/governance/src/governance.rs`)

## Summary
The `settle_neurons_fund_participation` endpoint on NNS Governance authorizes any legitimately deployed SNS swap canister to settle the Neurons' Fund participation for *any* proposal, not just the proposal that created that specific SNS. An attacker who controls a deployed SNS swap canister can target a different SNS's proposal, supply an attacker-controlled `sns_governance_canister_id` to redirect minted ICP, and permanently block the victim SNS's legitimate swap from ever settling.

## Finding Description

The public `#[update]` endpoint in `rs/nns/governance/canister/canister.rs` passes the raw `caller()` to `Governance::settle_neurons_fund_participation`: [1](#0-0) 

Inside `settle_neurons_fund_participation`, the only authorization check is a call to `is_canister_id_valid_swap_canister_id`, which queries SNS-W's `list_deployed_snses` and returns `Ok` if the caller appears as *any* swap canister in the entire list: [2](#0-1) 

The helper function confirms this — it iterates all deployed SNS instances and checks for any match: [3](#0-2) 

The code comment at line 7018 explicitly acknowledges the gap: *"Note that a Swap could settle each other's participation."* There is no check that `caller` is the swap canister recorded for the specific `nns_proposal_id` supplied in the request.

After authorization passes, the attacker-supplied `sns_governance_canister_id` from the `Committed` variant is used directly as the ICP mint destination: [4](#0-3) 

The minting call uses this value without any cross-check against the proposal's stored SNS canister IDs: [5](#0-4) 

The attacker-supplied `total_direct_participation_icp_e8s` is consumed at line 7208 to compute matched funding, maximizing the payout when set to `u64::MAX`: [6](#0-5) 

After the lifecycle is set to terminal at line 7235, the victim SNS's legitimate swap call falls into the "settlement in progress" error branch (lines 7154–7165), permanently blocking it: [7](#0-6) 

## Impact Explanation

A malicious SNS swap canister can: (1) redirect Neurons' Fund ICP minting to an attacker-controlled governance canister, stealing the full Neurons' Fund reservation for the victim SNS; and (2) permanently prevent the victim SNS's legitimate swap from settling by setting the proposal lifecycle to a terminal state before the legitimate swap can call the endpoint. The Neurons' Fund maturity of NNS neurons is irreversibly consumed. This matches the **High** impact class: unauthorized access to governance assets / significant NNS/SNS security impact with concrete user and protocol harm. The stolen amount is bounded by the victim SNS's Neurons' Fund reservation, which can be substantial (potentially millions of ICP depending on the SNS parameters).

## Likelihood Explanation

Exploitation requires controlling a legitimately deployed SNS swap canister, which requires passing an NNS governance vote. This is a meaningful but not insurmountable barrier — a project can propose a legitimate-looking SNS, get it deployed, and then immediately use its swap canister to attack any other open SNS proposal. The attack is fully deterministic, requires no privileged key material beyond the deployed swap canister's identity, and can be executed in a single call. The attack window is any time between SNS-B's proposal execution and SNS-B's legitimate swap calling `settle_neurons_fund_participation`.

## Recommendation

After loading `proposal_data`, extract the swap canister ID that was recorded when the SNS was deployed (stored in the proposal's execution result or `DeployedSns` data) and assert `caller == recorded_swap_canister_id` before proceeding. The `is_canister_id_valid_swap_canister_id` check should be replaced with a direct equality check against the swap canister ID stored in the specific proposal data, eliminating the cross-SNS settlement path entirely.

## Proof of Concept

1. Attacker deploys SNS-A via a legitimate NNS proposal; SNS-A's swap canister ID is now in `list_deployed_snses`.
2. SNS-B has an open `CreateServiceNervousSystem` proposal with `nns_proposal_id = P` and Neurons' Fund participation reserved.
3. SNS-A's swap canister calls `settle_neurons_fund_participation` with:
   ```
   nns_proposal_id: P,
   result: Committed {
     sns_governance_canister_id: SNS_A_governance,
     total_direct_participation_icp_e8s: u64::MAX,
     total_neurons_fund_participation_icp_e8s: 0,
   }
   ```
4. `is_canister_id_valid_swap_canister_id` passes because SNS-A's swap is a valid swap canister.
5. NNS Governance computes maximized matched funding, mints ICP to `SNS_A_governance`, and sets proposal `P`'s lifecycle to `Committed`.
6. SNS-B's legitimate swap subsequently calls `settle_neurons_fund_participation` for proposal `P` and receives the "settlement in progress" error (lines 7154–7165), permanently blocked.
7. A deterministic integration test using PocketIC can reproduce this by deploying two SNS instances and calling the endpoint from SNS-A's swap canister targeting SNS-B's proposal ID, then asserting the ICP balance of `SNS_A_governance` increased and SNS-B's legitimate call fails.
