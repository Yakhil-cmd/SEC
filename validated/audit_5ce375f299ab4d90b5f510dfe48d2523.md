Audit Report

## Title
Unbounded Heap Allocation via `to_bytes(body, usize::MAX)` in ICRC1 Rosetta Metrics Middleware â€” (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
The `extract_canister_id` function in `RosettaMetricsMiddleware` calls `to_bytes(body, usize::MAX)` to buffer the entire HTTP request body into heap memory with no size cap. The ICRC1 Rosetta axum server applies this middleware to every route with no upstream `DefaultBodyLimit` or `RequestBodyLimitLayer` anywhere in the router stack. An unauthenticated HTTP client can send an arbitrarily large POST body to any Rosetta endpoint, causing unbounded heap allocation that can OOM-kill the Rosetta process.

## Finding Description
In `rs/rosetta-api/common/rosetta_core/src/metrics.rs` at line 389, `extract_canister_id` unconditionally reads the entire request body with no size bound:

```rust
let bytes = to_bytes(body, usize::MAX).await?;
```

This is invoked for every incoming request inside `RosettaMetricsMiddleware::call` at lines 331â€“339, before the request is forwarded to any handler. The body is consumed and buffered into a contiguous heap `Bytes` allocation regardless of its size.

In `rs/rosetta-api/icrc1/src/main.rs` at lines 361â€“391, the axum router applies `metrics_layer` as the innermost middleware with no body-limiting layer anywhere in the chain â€” no `DefaultBodyLimit::max(N)`, no `RequestBodyLimitLayer::new(N)`. The server starts with a bare `axum::serve` call at lines 474â€“476. A grep across the entire `rs/rosetta-api/` tree for `RequestBodyLimitLayer`, `DefaultBodyLimit`, or any body-limit pattern returns zero matches, confirming no limit exists.

The exploit path is: attacker opens an HTTP connection â†’ sends a POST to any endpoint (e.g., `/network/list`) with a multi-hundred-MB or GB streaming body â†’ `RosettaMetricsMiddleware::call` invokes `extract_canister_id` â†’ `to_bytes(body, usize::MAX)` allocates a contiguous buffer of the full body size on the heap â†’ process is OOM-killed or experiences severe memory pressure before any handler logic runs.

## Impact Explanation
This is a **High** severity application/platform-level DoS. A single unauthenticated HTTP request can crash the ICRC1 Rosetta process, making the Rosetta API unavailable to all legitimate users. This matches the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."* The ICRC1 Rosetta server is a financial integration component serving ICRC ledger operations; its unavailability directly harms users relying on it for balance queries, block retrieval, and construction operations.

## Likelihood Explanation
The Rosetta HTTP port is publicly reachable by design. No authentication is required to POST to any endpoint. The exploit requires only a single HTTP connection with a large streaming body and is trivially reproducible with standard tools (`curl`, `dd`). No special privileges, victim interaction, or external compromise is needed. The only possible mitigation is a deployment-level reverse proxy with a body-size limit, which is not enforced by the code itself.

## Recommendation
Apply a body size limit before the metrics middleware reads the body. The preferred fix is adding `DefaultBodyLimit::max(N)` as the outermost layer in the axum router in `rs/rosetta-api/icrc1/src/main.rs`:

```rust
.layer(DefaultBodyLimit::max(4 * 1024 * 1024)) // 4 MB, outermost layer
```

Alternatively, replace `usize::MAX` with a reasonable constant in `extract_canister_id` in `rs/rosetta-api/common/rosetta_core/src/metrics.rs`:

```rust
const MAX_BODY_BYTES: usize = 4 * 1024 * 1024;
let bytes = to_bytes(body, MAX_BODY_BYTES).await?;
```

The router-level fix is preferred because it returns HTTP 413 before any allocation occurs.

## Proof of Concept
```bash
# Send a 512 MB body to the ICRC1 Rosetta /network/list endpoint
dd if=/dev/zero bs=1M count=512 | \
  curl -X POST http://<rosetta-host>:<port>/network/list \
       -H "Content-Type: application/json" \
       --data-binary @- \
       -v
```
The process will allocate ~512 MB on the heap inside `to_bytes` before returning any response. Repeating with concurrent connections or increasing body size to available RAM will OOM-kill the Rosetta process. This can also be verified with a local integration test by spawning the ICRC1 Rosetta binary, sending a large POST, and observing RSS growth or OOM termination.
