### Title
Fatal Block-Processing Halt via `notify_l2_asset_tracker` Push-Pattern Revert in L1→L2 Deposit Flow — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

Every L1→L2 transaction that carries a non-zero deposit causes the bootloader to call `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` up to three times (value mint, operator fee, refund). The helper `notify_l2_asset_tracker` explicitly treats any EVM-level revert from that call as a **fatal, unrecoverable error** that halts the entire block. This is the direct ZKsync OS analog of the Moloch.sol push-pattern bug: instead of a USDC blacklist blocking a token transfer and freezing proposal processing, a revert in the `L2AssetTracker` system contract blocks block processing for all L1→L2 deposits.

---

### Finding Description

`notify_l2_asset_tracker` (lines 855–915 of `process_l1_transaction.rs`) encodes a call to `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` on `L2_ASSET_TRACKER_ADDRESS` (`0x1000f`) and dispatches it via `run_single_interaction`. After the call returns, the code checks:

```rust
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    ).into());
}
``` [1](#0-0) 

This error propagates all the way up through `process_l1_transaction` and out of the block-level loop, halting the entire block — not just the individual transaction.

`notify_l2_asset_tracker` is invoked in three places per L1→L2 deposit:

1. **Value mint** — inside `execute_l1_transaction_and_notify_result` (line 634), called within the execution frame for the user's deposit value. [2](#0-1) 

2. **Operator fee** — after execution, via `mint_base_token` at line 290. [3](#0-2) 

3. **Refund** — after execution, via `mint_base_token` at line 338. [4](#0-3) 

The `l1_chain_id` passed to every call is read directly from `L2AssetTracker` storage slot 154 — a raw storage read, not a function call: [5](#0-4) 

The `L2AssetTracker` is a real, upgradeable EVM contract predeploy at `0x1000f`: [6](#0-5) 

Its `handleFinalizeBaseTokenBridgingOnL2` function has an `onlyBaseTokenHolderOrL2BaseToken` modifier, which the bootloader satisfies by spoofing `msg.sender = L2_BASE_TOKEN_ADDRESS` (0x800a). However, the function can still revert for contract-internal reasons:

- `isAssetRegistered[assetId] == false` (slot 153) — if the base token asset is deregistered or never registered, the function reverts.
- Arithmetic overflow in `chainBalance[fromChainId][assetId]` (slot 151) — if cumulative deposits overflow `uint256`, the function reverts.
- Any future upgrade to `L2AssetTracker` that introduces a revert path.
- `_needToForceSetAssetMigrationOnL2` logic that can revert under edge-case state.

The storage layout is documented in the code itself: [7](#0-6) 

The `should_make_frame = true` flag passed to `run_single_interaction` only isolates the `L2AssetTracker`'s own state changes on revert — it does **not** prevent the fatal error from propagating: [8](#0-7) 

---

### Impact Explanation

If `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` reverts for any reason during any L1→L2 deposit transaction, the entire block halts. No further transactions in that block — or any subsequent block containing an L1→L2 deposit — can be processed. This permanently freezes the L1→L2 bridge for the chain. Funds locked in the L1 bridge cannot be minted on L2, and the priority queue cannot be drained, which is a chain-halting condition.

---

### Likelihood Explanation

The `L2AssetTracker` is a real, upgradeable Solidity contract. Revert conditions include:

- **Asset deregistration**: If `isAssetRegistered[BASE_TOKEN_ASSET_ID]` is set to `false` (e.g., via a governance upgrade or a public deregistration function if one exists), every subsequent L1→L2 deposit call reverts.
- **`chainBalance` overflow**: An attacker who can submit L1 transactions with sufficiently large `total_deposited` values (enforced on L1, but the bootloader only checks `total_deposited >= tx_internal_cost`) could accumulate enough deposits to overflow `chainBalance[l1_chain_id][assetId]`, causing a revert on the next deposit.
- **Contract upgrade**: A governance upgrade that introduces any revert path in `handleFinalizeBaseTokenBridgingOnL2` immediately blocks all L1→L2 deposits.

The first two paths are reachable without privileged access (subject to L1 bridge constraints). The third requires governance but is a realistic operational risk given the contract is upgradeable.

---

### Recommendation

Replace the fatal-error design with a graceful-degradation approach:

1. **Pull pattern**: Instead of the bootloader pushing a notification to `L2AssetTracker` on every deposit, emit an L2 log/event that `L2AssetTracker` can process asynchronously. This decouples the accounting update from the critical block-processing path.

2. **Non-fatal error handling**: If the push pattern must be retained, treat a revert from `L2AssetTracker` as a non-fatal error for the individual transaction (mark the transaction as failed, skip the notification) rather than halting the entire block. The accounting inconsistency concern can be addressed by a reconciliation mechanism.

3. **Bounded retry**: Record failed notifications in a queue and allow the operator to replay them, rather than blocking block production.

The core issue is that a single external contract call — to a mutable, upgradeable contract — is on the critical path for all L1→L2 deposits, with no fallback.

---

### Proof of Concept

**Scenario: `chainBalance` overflow via large deposits**

1. Attacker submits a series of L1→L2 priority transactions, each with `total_deposited` close to `type(uint256).max / N` (where N is the number of transactions needed to overflow).
2. The L1 bridge enforces that `total_deposited` matches the actual ETH locked, but the attacker deposits real ETH.
3. After enough deposits, `chainBalance[l1_chain_id][BASE_TOKEN_ASSET_ID]` overflows in `L2AssetTracker`.
4. The next L1→L2 deposit transaction causes `handleFinalizeBaseTokenBridgingOnL2` to revert on the overflow.
5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
6. `process_l1_transaction` propagates the error, halting block processing.
7. All subsequent L1→L2 deposits are permanently blocked.

**Scenario: Asset deregistration (if a public deregistration path exists in `L2AssetTracker`)**

1. Attacker calls a public function on `L2AssetTracker` (e.g., `deregisterAsset(BASE_TOKEN_ASSET_ID)`) if such a function exists without access control.
2. `isAssetRegistered[BASE_TOKEN_ASSET_ID]` is set to `false`.
3. The next L1→L2 deposit causes `handleFinalizeBaseTokenBridgingOnL2` to revert at the `isAssetRegistered` check.
4. Block processing halts as above.

The root cause in ZKsync OS code is unambiguous regardless of which trigger is used: [9](#0-8)

/// Failure halts block processing — if the asset tracker reverts, the
/// chain's token accounting would be inconsistent, so we treat it as
/// fatal rather than silently continuing with incorrect bookkeeping.
///
/// If no contract is deployed at L2AssetTracker, the call succeeds silently
/// (a call to an empty address returns success with no returndata in EVM).
/// However, we are certain that L2AssetTracker is available after the upgrade.
fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }

            &l1_chain_id_slot,
