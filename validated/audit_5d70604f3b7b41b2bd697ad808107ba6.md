## Execution Flow Trace

**Step 1 — `begin_new_tx` vs `finish_tx` split in `GenericPubdataAwarePlainStorage`**

`current_tx_id` is incremented only in `finish_tx()`, not in `begin_new_tx()`: [1](#0-0) 

`begin_new_tx()` only calls `cache.commit()`. `finish_tx()` is the sole incrementer of `current_tx_id`.

The same pattern holds for the flat storage model's account cache: [2](#0-1) [3](#0-2) 

**Step 2 — Warmth check uses `current_tx_id`** [4](#0-3) 

An address is warm iff `last_touched_in_tx == Some(current_tx_id)`.

**Step 3 — Bootloader pre-warms coinbase and precompiles BEFORE the rollback snapshot**

In `generic_loop_op` (Ethereum bootloader): [5](#0-4) 

The coinbase and all precompile addresses are touched (setting `last_touched_in_tx = Some(current_tx_id)`) **before** `start_global_frame()` is called.

In the ZK bootloader, coinbase is also pre-warmed before the snapshot: [6](#0-5) 

**Step 4 — On validation failure, rollback does NOT undo the pre-warming** [7](#0-6) 

The rollback target (`pre_tx_rollback_handle`) was taken **after** the pre-warming. Rolling back to it leaves the warming intact.

**Step 5 — `finish_valid_tx()` is only called on success** [8](#0-7) [9](#0-8) 

`finish_valid_tx()` → `io.finish_tx()` → `storage.finish_tx()` → `current_tx_id += 1`. This is skipped entirely for failed transactions.

**Step 6 — Consequence for tx2**

When tx2 begins:
- `begin_new_tx()` calls `cache.commit()`, freezing the coinbase/precompile entries with `last_touched_in_tx = Some(0)` as the committed baseline
- `current_tx_id` is still `0` (never incremented)
- When tx2 accesses coinbase: `considered_warm(0)` → `last_touched_in_tx == Some(0)` → **`true`** → warm access cost (100 gas) instead of cold (2600 gas)

**Contrast with Ethereum storage model's account cache**

The Ethereum model's account cache increments `current_tx_number` in `begin_new_tx()`, not `finish_tx()`: [10](#0-9) 

This means the Ethereum model's account warmth is correctly reset per-tx regardless of success/failure. The flat storage model and `GenericPubdataAwarePlainStorage` do not have this protection.

### Title
Cross-Transaction Warmth Leak via Skipped `finish_tx` on Validation Failure — (`basic_system/src/system_implementation/caches/generic_pubdata_aware_plain_storage.rs`, `flat_storage_model/account_cache.rs`)

### Summary
When a transaction fails validation, `finish_valid_tx()` is never called, so `current_tx_id` is never incremented. Because the bootloader pre-warms coinbase and precompile addresses **before** the rollback snapshot, that warmth survives the rollback. The next transaction then observes those addresses as warm (same `current_tx_id`), paying 100 gas instead of 2600 gas for each cold access.

### Finding Description
The warmth invariant `considered_warm(current_tx_id)` relies on `current_tx_id` being incremented between transactions. In `GenericPubdataAwarePlainStorage` and the flat storage model's account cache, this increment lives exclusively in `finish_tx()`, which is only reached via `finish_valid_tx()` on the success path. The bootloader pre-warms coinbase (both loops) and all EVM precompile addresses (Ethereum loop) before taking the per-tx rollback snapshot. A validation failure rolls back to that snapshot, but the pre-warming writes are already committed to the cache history and are not undone. On the next iteration, `begin_new_tx()` calls `cache.commit()`, cementing those entries with `last_touched_in_tx = Some(current_tx_id)` where `current_tx_id` is unchanged. Every subsequent access to coinbase or a precompile in that next transaction is treated as warm.

### Impact Explanation
Each affected address saves 2500 EVM gas (2600 cold − 100 warm). With ~9 precompile addresses plus coinbase, a single failed-validation transaction can cause ~25,000 gas of undercharging in the following transaction. Repeated across many blocks, this allows an attacker to execute more computation per gas unit than the protocol permits, enabling fee drain and potential block-limit bypass.

### Likelihood Explanation
The precondition is two consecutive transactions where the first fails validation (invalid nonce, insufficient balance, validator filter, etc.). This is trivially achievable by any unprivileged user submitting a transaction with a deliberately wrong nonce. No privileged access, key material, or governance control is required.

### Recommendation
Increment `current_tx_id` inside `begin_new_tx()` rather than `finish_tx()`, mirroring the Ethereum storage model's account cache pattern (`self.current_tx_number += 1` in `begin_new_tx`). This ensures each new transaction boundary always advances the tx-id regardless of whether the previous transaction succeeded or failed, making warmth from one transaction's pre-warming invisible to the next.

### Proof of Concept
1. Deploy any EVM contract that reads `BALANCE` of `block.coinbase`.
2. Submit tx1 with an invalid nonce (e.g., nonce = 999 when account nonce = 0). Tx1 fails validation; `finish_valid_tx()` is not called; `current_tx_id` stays at 0.
3. Submit tx2 calling the contract. Measure gas consumed by the `BALANCE` opcode on coinbase.
4. Observe: tx2 pays 100 gas (warm) instead of 2600 gas (cold) for the coinbase `BALANCE`.
5. Repeat with a fresh block where tx1 succeeds: tx2 now pays 2600 gas, confirming the leak.
