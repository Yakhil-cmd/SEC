Audit Report

## Title
Stale ICP/XDR Rate Used Unconditionally for Cycle Minting Without Freshness Check - (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles()` function in the Cycles Minting Canister reads only `xdr_permyriad_per_icp` from the cached `icp_xdr_conversion_rate` state, never inspecting `timestamp_seconds`. When the Exchange Rate Canister (XRC) fails to return a fresh rate, the heartbeat logs the error but leaves the cached rate unchanged with no maximum-age enforcement. Any unprivileged caller can then send ICP to the CMC and receive excess cycles computed at the stale, inflated rate.

## Finding Description
`tokens_to_cycles()` at `rs/nns/cmc/src/main.rs` L1900–1922 extracts only `rate.xdr_permyriad_per_icp` and discards `rate.timestamp_seconds`:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);
```

This function is called unconditionally by all three minting entry points: `process_create_canister` (L1932), `process_mint_cycles` (L1965), and `process_top_up` (L1991).

The heartbeat at L2397–2428 calls `exchange_rate_canister::update_exchange_rate()`. When the XRC call fails, `update_exchange_rate` at `rs/nns/cmc/src/exchange_rate_canister.rs` L259–275 returns `UpdateExchangeRateError::FailedToRetrieveRate` without modifying `state.icp_xdr_conversion_rate`. The guard's `schedule_next_attempt` reschedules a retry at the next minute, but the cached rate remains at its last-known value indefinitely. On XRC failure the retry interval drops to one minute, but if the XRC remains unavailable the rate can grow arbitrarily stale.

`do_set_icp_xdr_conversion_rate` at L1009–1040 only enforces that a new rate's timestamp is strictly greater than the current one; it imposes no maximum age on the stored rate at read time. The `IcpXdrConversionRate` struct (`rs/nns/cmc/src/lib.rs` L488–497) carries `timestamp_seconds` precisely for freshness purposes, but that field is never consulted during minting.

The default state (`rs/nns/cmc/src/main.rs` L360–363) initialises `icp_xdr_conversion_rate` with `DEFAULT_ICP_XDR_CONVERSION_RATE_TIMESTAMP_SECONDS` (May 10 2021), meaning a CMC that has never successfully contacted the XRC would mint cycles at a four-year-old rate.

Exploit path:
1. XRC becomes transiently unavailable (inter-canister call error, XRC downtime, insufficient data sources).
2. CMC heartbeat fires; `update_exchange_rate` returns `FailedToRetrieveRate`; cached rate is unchanged.
3. ICP market price drops while the XRC is unavailable.
4. Unprivileged caller sends ICP to the CMC subaccount and calls `notify_top_up` / `notify_create_canister` / `notify_mint_cycles`.
5. `tokens_to_cycles` reads the stale high `xdr_permyriad_per_icp` with no timestamp check and mints excess cycles.
6. ICP is burned; caller retains cycles worth more than the current market value of the burned ICP.

No existing guard prevents this: the only check in `do_set_icp_xdr_conversion_rate` is a monotonicity check on incoming rates, not a staleness check on the stored rate at minting time.

## Impact Explanation
This constitutes illegal minting of cycles: callers receive more cycles per ICP than the current market rate justifies, representing a direct economic loss to the network's resource pricing model. The `base_cycles_limit` of `DEFAULT_CYCLES_LIMIT = 150e15` cycles per hour (`rs/nns/cmc/src/main.rs` L83) bounds the per-hour exposure but does not prevent exploitation within that limit. At current rates (~35,200 XDR permyriad per ICP), the hourly ceiling corresponds to tens of thousands of ICP worth of cycles; a sustained XRC outage combined with a significant ICP price drop could produce material excess minting across multiple hours. This matches the High impact class: "Significant XRC… security impact with concrete user or protocol harm."

## Likelihood Explanation
The XRC is an external inter-canister dependency; transient failures are explicitly anticipated and handled by the CMC's retry logic. A meaningful ICP price drop (10–30%) combined with even a brief XRC outage (minutes to hours) creates the exploitable window. The attacker requires no special privileges—only the ability to send ICP to a CMC subaccount and call a public update method. The window is unbounded in duration if the XRC remains unavailable, and the attacker can monitor on-chain state to detect when the cached rate diverges from the market price.

## Recommendation
Add a maximum-age check inside `tokens_to_cycles()`. Reject minting if `now_seconds() - rate.timestamp_seconds` exceeds a configurable threshold (e.g., a small multiple of `REFRESH_RATE_INTERVAL_SECONDS`):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let rate = state.icp_xdr_conversion_rate.as_ref();
        match rate {
            Some(rate) => {
                let age = now_seconds().saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: format!(
                            "ICP/XDR conversion rate is stale ({} seconds old)", age
                        ),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(/* existing None branch */),
        }
    })
}
```

`MAX_RATE_AGE_SECONDS` should be set to a value that tolerates normal heartbeat jitter (e.g., 2–3× `REFRESH_RATE_INTERVAL_SECONDS` = 10–15 minutes) while bounding the staleness window. Governance should also be able to override this threshold via a canister upgrade or configuration proposal.

## Proof of Concept
A deterministic PocketIC or StateMachine integration test can reproduce this without touching mainnet:

1. Deploy CMC with a known `exchange_rate_canister_id` pointing to a mock XRC.
2. Set an initial `icp_xdr_conversion_rate` with `xdr_permyriad_per_icp = 50_000` (5 XDR/ICP) and `timestamp_seconds = T`.
3. Configure the mock XRC to return an error on all subsequent calls.
4. Advance simulated time by `> REFRESH_RATE_INTERVAL_SECONDS` (5 minutes); trigger the heartbeat. Confirm the rate in state is unchanged and still timestamped at `T`.
5. Advance time further to simulate a 30% ICP price drop (the "correct" rate would now be ~35,000 XDR permyriad).
6. Call `notify_top_up` from an unprivileged principal with a known ICP amount.
7. Assert that the cycles credited equal `amount × 50_000 / 10_000 × cycles_per_xdr`—i.e., computed at the stale rate—rather than the market-correct `35_000`-based value.
