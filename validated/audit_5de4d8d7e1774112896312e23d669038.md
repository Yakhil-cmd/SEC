Audit Report

## Title
Anonymous Caller Bypasses ICRC-152 Burn/Mint Access Control When Anonymous Principal Is Canister Controller - (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary

`icrc152_burn_not_async` and `icrc152_mint_not_async` use `ic_cdk::api::is_controller(&caller)` as their sole caller-identity gate without ever rejecting the anonymous principal as a caller. Because the anonymous principal requires no key material and any HTTP client can send ingress messages under it, any deployment where the anonymous principal occupies the controller list exposes privileged burn and mint operations to fully unauthenticated callers. The integration test suite confirms the exploit path by using `PrincipalId::new_anonymous()` as the effective controller that successfully drives both endpoints.

## Finding Description

In `icrc152_burn_not_async` (L1009–1013), the only caller check is:

```rust
if !ic_cdk::api::is_controller(&caller) {
    return Err(Icrc152BurnError::Unauthorized(...));
}
```

The subsequent anonymous-principal check at L1025–1029 tests `args.from.owner == Principal::anonymous()` — the *target account*, not the *caller*. An anonymous caller whose principal appears in the controller list passes the `is_controller` gate unconditionally and is never rejected by the account check.

`icrc152_mint_not_async` has the symmetric gap: L916–920 checks `is_controller`, and L932–936 checks `args.to.owner == Principal::anonymous()` (the destination account), leaving the caller unchecked.

The `setup_icrc152` helper at L6177 calls `env.install_canister(ledger_wasm, args, None)`, which internally calls `execute_ingress` (anonymous sender) for `ProvisionalCreateCanisterWithCycles`. The IC management canister sets the caller as the default controller, so the anonymous principal becomes the canister's controller. The test at L6458 confirms this explicitly: `let controller = PrincipalId::new_anonymous();`, and the subsequent mint (L6465–6476) and burn (L6500–6512) calls both succeed. On mainnet, the same configuration arises whenever a deployer issues `create_canister` via an anonymous ingress call without specifying explicit controllers.

Because the IC protocol permits anonymous ingress calls to update methods, `is_controller(anonymous)` returning `true` is sufficient for any unauthenticated HTTP client to reach the privileged code paths.

## Impact Explanation

An attacker who observes that a deployed ICRC-152 ledger's controller list includes the anonymous principal can:

- Call `icrc152_burn` with `from` set to any real user account and `amount` equal to the full balance, irreversibly destroying that user's tokens. The ledger records the burn as an `AuthorizedBurn` block with `caller: Some(anonymous)`.
- Call `icrc152_mint` with `to` set to any non-anonymous, non-minting account, inflating total supply without authorization.

Both operations bypass the minimum-burn-amount floor and the fee mechanism that protect the standard `icrc1_transfer` burn path. This constitutes unauthorized access to ledger funds and illegal minting, matching the **High** impact category: unauthorized access to ledgers or canister-controlled funds where exploitation requires a specific but achievable precondition (anonymous principal in the controller list).

## Likelihood Explanation

The precondition — anonymous principal in the controller list — is not the default for typical `dfx`-based deployments (where the deployer's identity becomes the controller), but it is reachable via an anonymous ingress call to `create_canister` without specifying controllers, or by explicitly adding the anonymous principal as a controller. The StateMachine test environment defaults to this configuration. No key material, governance majority, or privileged network position is required; a single anonymous HTTP ingress call suffices once the precondition is met.

## Recommendation

Add an explicit rejection of the anonymous principal as a caller at the top of both `icrc152_burn_not_async` and `icrc152_mint_not_async`, before the `is_controller` check:

```rust
if caller == Principal::anonymous() {
    return Err(Icrc152BurnError::Unauthorized(
        "anonymous caller is not allowed".to_string(),
    ));
}
```

Apply the symmetric guard in `icrc152_mint_not_async`. Additionally, consider adding a canister-init-time assertion that rejects enabling ICRC-152 when the anonymous principal is the sole controller, and document that deployers must set a non-anonymous controller before enabling the `icrc152` feature flag.

## Proof of Concept

The existing integration test `test_icrc152_mint_and_burn` already demonstrates the exploit:

1. `setup_icrc152` installs the ledger with `icrc152: true` and no explicit controller override (`None`), causing the anonymous principal to become the controller.
2. `let controller = PrincipalId::new_anonymous()` — no key, no authentication.
3. `icrc152_mint(&env, canister_id, controller, &mint_args)` succeeds (L6465–6476).
4. `icrc152_burn(&env, canister_id, controller, &burn_args)` succeeds (L6500–6512), zeroing `p1`'s balance.

Both calls pass `is_controller(anonymous) == true` and are never rejected by the anonymous-caller check that does not exist.
