### Title
Unpermissioned `deploy_erc20_token` Allows Any Caller to Permanently Corrupt the NEP-141 → ERC-20 Registry - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `deploy_erc20_token` contract method has no caller access control. Any NEAR account can invoke it to register an arbitrary NEP-141 token with an attacker-deployed ERC-20 contract. Because `register_token` permanently rejects duplicate registrations with `TokenAlreadyRegistered` and no update/removal path exists, the attacker can front-run the protocol's legitimate deployment, permanently corrupting the NEP-141 → ERC-20 registry and blocking all future legitimate bridge setup for the targeted token.

---

### Finding Description

**Root cause — missing access control on `deploy_erc20_token`**

The public NEAR entrypoint `deploy_erc20_token` (exposed in `engine/src/lib.rs`) dispatches to `contract_methods::connector::deploy_erc20_token`. That function performs only a liveness check (`require_running`) and no caller restriction: [1](#0-0) 

The inline comment in the `WithMetadata` branch reads *"this transaction could be executed by the owner of the contract only"*, but no `require_owner_only` guard is present anywhere in the function body: [2](#0-1) 

Compare this with `mirror_erc20_token`, which correctly enforces `require_owner_only`: [3](#0-2) 

**Permanent state corruption via `register_token`**

Both `deploy_erc20_token` and `mirror_erc20_token` ultimately call `engine::deploy_erc20_token`, which calls `engine.register_token(address, nep141)`: [4](#0-3) 

`register_token` checks for an existing mapping and returns `TokenAlreadyRegistered` if one is found — with no update or removal path: [5](#0-4) 

Once a NEP-141 → ERC-20 entry is written into `nep141_erc20_map`, it is permanent. There is no `unregister_token` or `update_token` function anywhere in the codebase.

**Attack path**

1. Attacker monitors the NEAR blockchain for a new NEP-141 token that the protocol is about to bridge (e.g., by watching for the NEP-141 contract deployment or an off-chain announcement).
2. Attacker calls `deploy_erc20_token` (Legacy variant) with the target NEP-141 account ID before the protocol does.
3. The engine deploys an ERC-20 at an address determined by the attacker's nonce, using **default (empty) metadata** — name `""`, symbol `""`, decimals `0`: [6](#0-5) 

4. The NEP-141 → ERC-20 mapping is permanently written with the attacker-deployed ERC-20 address.
5. Every subsequent legitimate call to `deploy_erc20_token` or `mirror_erc20_token` for that NEP-141 fails with `ERR_NEP141_TOKEN_ALREADY_REGISTERED`: [7](#0-6) 

6. The `mirror_erc20_token_callback` path, which is the only way to register a Silo-mirrored ERC-20, is also permanently blocked for that NEP-141: [8](#0-7) 

---

### Impact Explanation

- **Permanent corruption of the bridge registry**: The NEP-141 → ERC-20 mapping is immutably set to the attacker-deployed ERC-20 with wrong metadata (decimals = 0, name/symbol empty). There is no admin function to correct it.
- **Permanent blocking of legitimate deployment**: The protocol can never deploy the correct ERC-20 (with proper metadata fetched via `WithMetadata`) for the targeted NEP-141. `mirror_erc20_token` on any Silo is also permanently blocked for that token.
- **Incorrect decimal accounting**: All DeFi protocols on Aurora that read `decimals()` from the ERC-20 will receive `0` instead of the correct value (e.g., 6 for USDC, 18 for WETH), causing permanent mispricing and potential fund loss in any protocol that relies on decimal-adjusted arithmetic.
- **No recovery path**: There is no admin function to remove or update a NEP-141 → ERC-20 mapping. A contract upgrade is the only remediation, which is a high-friction, time-delayed operation.

This constitutes **permanent freezing** of the bridge for the targeted NEP-141 token and permanent corruption of on-chain accounting for any user who bridges that token.

---

### Likelihood Explanation

- Any NEAR account can call `deploy_erc20_token` — no tokens, no stake, no special role required.
- The attack costs only a standard NEAR transaction fee (fractions of a cent).
- The attacker needs only to know the NEP-141 account ID of a token that has not yet been bridged, which is publicly observable on-chain.
- The attack is a single transaction and is irreversible.

---

### Recommendation

Add `require_owner_only` (or an equivalent admin-list check) to `deploy_erc20_token` in `engine/src/contract_methods/connector.rs`, mirroring the guard already present in `mirror_erc20_token`:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
+       require_owner_only(&state, &env.predecessor_account_id())?;
        // ...
    })
}
```

Additionally, consider adding an admin-controlled `unregister_token` or `update_token` function to allow recovery from accidental or malicious registrations.

### Proof of Concept

```
1. Protocol announces intent to bridge NEP-141 token "usdc.near".
2. Attacker submits NEAR transaction:
     aurora.deploy_erc20_token(DeployErc20TokenArgs::Legacy("usdc.near"))
   from any NEAR account.
3. Engine deploys ERC-20 at address X (attacker's nonce-derived address),
   metadata: name="", symbol="", decimals=0.
   nep141_erc20_map["usdc.near"] = X  (permanent).
4. Protocol submits:
     aurora.deploy_erc20_token(DeployErc20TokenArgs::WithMetadata("usdc.near"))
   → fails: ERR_NEP141_TOKEN_ALREADY_REGISTERED.
5. Protocol submits:
     silo.mirror_erc20_token({contract_id: "aurora", nep141: "usdc.near"})
   → mirror_erc20_token_callback calls engine::mirror_erc20_token
   → register_token returns TokenAlreadyRegistered
   → fails permanently.
6. All users bridging usdc.near receive tokens on ERC-20 at X with decimals=0.
   No DeFi protocol can correctly price or account for these tokens.
   No admin path exists to correct the registry.
```

pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
