### Title
Rosetta ICRC-1 `sync_blocks_interval` Stores Forked Blocks Before Trailing Hash Check, Enabling Permanent Fabricated Balance Injection — (`rs/rosetta-api/icrc1/src/ledger_blocks_synchronization/blocks_synchronizer.rs`)

---

### Summary

`sync_blocks_interval` fetches blocks backward from the certified tip and **commits each batch to SQLite before** performing the trailing parent-hash check that connects the new chain segment to the already-stored chain. Because `store_blocks` uses `INSERT OR IGNORE`, those rows are permanent. Because `start_synching_blocks` calls `update_account_balances` unconditionally (even after a sync error), the fabricated blocks are immediately reflected in the balance table. A canister controller who deploys a malicious ICRC-1 ledger can exploit this to permanently inject a forked chain into any Rosetta instance that syncs from their canister.

---

### Finding Description

**Step 1 — Certified tip is canister-controlled.**

`get_certified_chain_tip` in `packages/icrc-ledger-agent/src/lib.rs` verifies the IC certificate against the subnet's root key. [1](#0-0) 

The certificate is valid because the subnet signs whatever the canister places in `certified_data`. A canister controller can set `certified_data` to the hash of any block they choose; the subnet will faithfully sign it. The certificate verification therefore does not prevent a malicious canister from advertising an arbitrary tip hash.

**Step 2 — `sync_blocks_interval` works backward from the certified tip.**

The loop initialises `leading_block_hash` from the certified tip and fetches batches from the tip downward. [2](#0-1) 

`is_valid_blockchain` only verifies (a) that the last block in the batch matches `leading_block_hash` and (b) internal hash-chain consistency within the batch. It never checks the first block's parent hash against the already-stored chain. [3](#0-2) [4](#0-3) 

**Step 3 — Blocks are stored before the trailing check.**

After `is_valid_blockchain` passes, the batch is immediately committed to SQLite: [5](#0-4) 

The trailing parent-hash check — the only guard that connects the new segment to the existing chain — is evaluated **after** storage, and only on the final loop iteration: [6](#0-5) 

When the check fails, `bail!()` is called, but all fetched blocks are already durably written.

**Step 4 — `INSERT OR IGNORE` makes the injection permanent.**

The SQL used by `store_blocks` silently ignores any subsequent attempt to write a block at the same index: [7](#0-6) 

Once the malicious blocks occupy indices N…N+M-1, legitimate blocks at those indices can never overwrite them.

**Step 5 — `update_account_balances` runs unconditionally.**

`start_synching_blocks` calls `update_account_balances` regardless of whether `sync_from_the_tip` succeeded or failed: [8](#0-7) 

**Step 6 — Gap check does not detect hash-chain breaks.**

`update_account_balances` only guards against missing block indices, not broken hash chains: [9](#0-8) 

`get_blockchain_gaps` is a pure index-continuity check: [10](#0-9) 

If the malicious blocks fill indices N…N+M-1 contiguously, no gap is detected and balance computation proceeds over the forked chain.

---

### Impact Explanation

Any Rosetta instance configured to sync from the attacker's canister will permanently store the forked chain and derive account balances from it. Downstream exchanges or custodians that use Rosetta as their source of truth will observe fabricated balances. Because `INSERT OR IGNORE` prevents correction, the only recovery path is a full database wipe and re-sync — which itself re-syncs from the same malicious canister unless the operator intervenes.

---

### Likelihood Explanation

The attack requires only that the attacker control the ICRC-1 ledger canister that Rosetta is configured to sync from. On the Internet Computer, deploying a canister is permissionless. The realistic scenario is an exchange listing a new token and running Rosetta against the token's ledger: the token deployer is the canister controller and can execute this attack at any time without any additional privilege.

---

### Recommendation

1. **Perform the trailing parent-hash check before committing any batch to storage.** The simplest fix is to collect all fetched blocks in memory across all loop iterations and only call `store_blocks` once the trailing check passes.
2. **Gate `update_account_balances` on sync success.** Move the call inside the `Ok(_)` branch of `sync_from_the_tip`, or add an explicit `sync_failed` guard.
3. **Replace `INSERT OR IGNORE` with a check-then-insert pattern** that returns an error (rather than silently succeeding) when a block at a given index already exists with a different hash, so any future inconsistency is surfaced immediately.

---

### Proof of Concept

```
1. Rosetta syncs N real blocks (0..N-1) from the malicious ledger.
   stored_tip_hash = hash(block[N-1])

2. Attacker sets certified_data = hash(fake_block[N+M-1])
   (subnet signs this without inspection).

3. Attacker constructs a valid internal chain:
     fake_block[N].parent_hash   = <arbitrary, ≠ stored_tip_hash>
     fake_block[N+1].parent_hash = hash(fake_block[N])
     ...
     fake_block[N+M-1].parent_hash = hash(fake_block[N+M-2])
   hash(fake_block[N+M-1]) == certified tip hash  ✓

4. sync_blocks_interval fetches fake_block[N..N+M-1]:
   - is_valid_blockchain passes (internal chain + certified tip match)  ✓
   - store_blocks commits all M blocks via INSERT OR IGNORE
   - trailing check: fake_block[N].parent_hash ≠ stored_tip_hash → bail!()
   - M blocks are now permanently in the DB

5. update_account_balances runs (sync_failed is ignored):
   - does_blockchain_have_gaps() → false (indices 0..N+M-1 are contiguous)
   - balances computed from fake_block[N..N+M-1]
   - fabricated balances visible via Rosetta API
```

        leading_block_hash: &ByteBuf,

        if blockchain.is_empty() {
            return Ok(());
        }

        // Check that the leading block has the block hash that is provided.
        // Safe to call unwrap as the blockchain is guaranteed to have at least one element.
        if blockchain.last().unwrap().clone().get_block_hash().clone() != leading_block_hash {
            return Err(format!(
                "Invalid block at index {}",
                blockchain.last().unwrap().clone().index
            ));
        }

        let mut parent_hash = Some(blockchain[0].clone().get_block_hash().clone());
        // The blockchain has more than one element so it is safe to skip the first one.
        // The first element cannot be verified so we start at element 2.

            if block.get_parent_hash() != parent_hash {
                if block.index == 0 {
                    return Err("Block with index 0 found at different location".to_string());
                } else {
                    return Err(format!("Invalid block at index {}", block.index - 1));
                }
            }
            parent_hash = Some(block.clone().get_block_hash());
        }

        // No invalid blocks were found return true.
        Ok(())
