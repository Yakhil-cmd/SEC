Audit Report

## Title
Order-Dependent State Update in CMC: Cycles Minted Before ICP Burned, Stranding ICP on Burn Failure - (File: rs/nns/cmc/src/main.rs)

## Summary
In `process_mint_cycles`, `process_top_up`, and `process_create_canister`, cycles are fully credited to the recipient before `burn_and_log` is called to destroy the backing ICP. `burn_and_log` is explicitly designed to swallow all errors and never propagate failure. If the ICP ledger is unavailable when `burn_and_log` executes, the cycles remain in circulation, the notification is permanently marked as completed (non-retryable), and the ICP is stranded in the CMC's subaccount with no automated recovery path — permanently violating the ICP↔cycles conservation invariant.

## Finding Description
In all three notify paths the operation order is identical:

1. **Cycles credited first**: `do_mint_cycles` (L1966) calls the cycles ledger `deposit`, `deposit_cycles` (L1999) calls the management canister, and `do_create_canister` (L1943) calls the subnet management canister — all via `call_with_payment128`. These cross-canister calls transfer cycles to the recipient.
2. **ICP burn attempted second**: `burn_and_log(sub, amount)` (L1968, L2001, L1945) is called after the cycles credit succeeds.
3. **`burn_and_log` never propagates failure** (L2014–2049): the function's own comment states *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* On any `CallResult::Err`, it logs the error and returns `()`.

If the ICP ledger is stopped (e.g., during an NNS-governed upgrade), the `call_protobuf(ledger_canister_id, "send_pb", send_args)` at L2040 receives a rejection. `burn_and_log` logs it and returns. The caller (`process_mint_cycles`, etc.) then returns `Ok(...)` to the notify handler, which at L1305–1312 writes `NotificationStatus::NotifiedMint(Ok(...))` — permanently sealing the block index as non-retryable. The ICP in `Subaccount::from(&caller())` is never burned and is inaccessible to the user or any automated process.

Existing guards are insufficient: `is_transient_error` (L1310, L1219) only removes the block index from `blocks_notified` on transient errors returned by the *cycles* operations, not by `burn_and_log` (which returns `()` unconditionally). There is no retry queue, no reconciliation job, and no governance path to recover stranded subaccount ICP.

## Impact Explanation
This matches **High** impact: *"Significant ledger/NNS/SNS or infrastructure security impact with concrete user or protocol harm."* Each successful exploit permanently inflates the ICP total supply by the notified amount (ICP not burned) while the equivalent cycles remain in circulation. The ICP is locked in a CMC subaccount with no recovery mechanism. Repeated exploitation during a single ledger upgrade window (which can last tens of seconds) could strand a meaningful aggregate of ICP. The conservation invariant — that every cycle in existence corresponds to a burned ICP — is permanently broken for each affected transaction. This does not reach Critical because the attacker does not recover the ICP themselves (it is locked, not returned), so there is no direct theft; the harm is protocol-level supply inflation and accounting integrity loss.

## Likelihood Explanation
The trigger condition — ICP ledger unavailability during `burn_and_log` — is reachable by any unprivileged user who can observe NNS governance. ICP ledger upgrades are proposed and executed via public NNS proposals; the upgrade window (canister stopped → upgraded → restarted) is observable on-chain and typically lasts 10–60 seconds. An attacker pre-stages ICP in the CMC subaccount (via a normal ICP transfer with `MEMO_MINT_CYCLES`) before the upgrade, then calls `notify_mint_cycles` during the stopped window. The cycles ledger is a separate canister on a different subnet and is unaffected by the ICP ledger upgrade, so `do_mint_cycles` succeeds while `burn_and_log` fails. No special privileges are required beyond holding ICP.

## Recommendation
Reorder the operations so the ICP burn is confirmed before cycles are credited:

1. Call the ICP ledger `send_pb` to burn the ICP from the CMC subaccount.
2. Only if the burn succeeds, proceed to mint/deposit cycles to the recipient.
3. If the burn fails, return a retryable error (or refund) so the user can retry.

This requires `burn_and_log` to be replaced with a function that propagates errors. The concern about retryability (double-spend) is addressed by the existing `blocks_notified` deduplication: mark the block as `Processing` before the burn attempt, and only mark it `NotifiedMint` after both the burn and the cycles deposit succeed. Alternatively, implement a persistent retry queue for failed burns so stranded ICP is eventually recovered without requiring user action.

## Proof of Concept
**Precondition**: Attacker holds ICP. ICP ledger upgrade is pending (observable via NNS governance).

1. Attacker transfers `N` ICP to `CMC_subaccount = Subaccount::from(&attacker_principal)` with `MEMO_MINT_CYCLES`.
2. Attacker monitors the NNS for the ledger upgrade execution. When the ICP ledger canister transitions to `Stopping`/`Stopped`, attacker immediately calls `notify_mint_cycles { block_index: <transfer_block>, to_subaccount: None, deposit_memo: None }`.
3. CMC executes `process_mint_cycles` → `do_mint_cycles` → cycles ledger `deposit` succeeds → attacker's cycles ledger balance increases by `tokens_to_cycles(N)`.
4. CMC calls `burn_and_log(sub, N)` → `call_protobuf(ledger_canister_id, "send_pb", ...)` → ICP ledger is stopped → call returns `Err(CANISTER_STOPPED, ...)` → `burn_and_log` logs error, returns `()`.
5. `notify_mint_cycles` returns `Ok(NotifyMintCyclesSuccess { minted: cycles, ... })` to attacker.
6. `blocks_notified[block_index]` is set to `NotifiedMint(Ok(...))` — permanently non-retryable.
7. **Result**: Attacker has `tokens_to_cycles(N)` cycles. `N` ICP remains in CMC subaccount, never burned. ICP total supply is permanently inflated by `N`. No recovery path exists.

A deterministic integration test can reproduce this using PocketIC by: (a) setting up CMC + ICP ledger + cycles ledger, (b) stopping the ICP ledger canister after the ICP transfer but before `notify_mint_cycles` is processed, (c) calling `notify_mint_cycles`, and (d) asserting that the cycles balance increased while the ICP ledger burn block count did not increase and the CMC subaccount still holds the ICP.
