Audit Report

## Title
Unbounded DKG Unvalidated Pool Growth via Byzantine Peer Flooding with Deferred Remote Dealings — (`rs/consensus/dkg/src/lib.rs`, `rs/artifact_pool/src/dkg_pool.rs`)

## Summary
A single Byzantine subnet node can flood the DKG unvalidated pool by advertising an unbounded number of dealings referencing fabricated remote DKG target IDs. The `validate_dealings_for_dealer` function returns `Mutations::new()` (defers) without signature verification or removal for any dealing whose `target_subnet.is_remote()` and has no matching config. Combined with no pool size cap and `SLOT_TABLE_NO_LIMIT` for DKG, deferred dealings accumulate for up to 499 consensus rounds, enabling memory exhaustion and CPU amplification on victim replicas.

## Finding Description

**Root cause 1 — Unconditional deferral without signature verification:**

In `validate_dealings_for_dealer`, when `message_dkg_id.target_subnet.is_remote()` is true and no matching config exists, the function immediately returns `Mutations::new()` at line 210, before any signature check is performed. Signature verification only occurs later in the function (line 221+), which is never reached for this code path. [1](#0-0) 

**Root cause 2 — No pool size limit:**

`DkgPoolImpl::insert` unconditionally inserts into a `BTreeMap`-backed `PoolSection` with no maximum capacity, no per-peer counter, and no `max_count`/`max_bytes` guard analogous to the ingress pool's `exceeds_limit` check. [2](#0-1) 

**Root cause 3 — No P2P slot table limit for DKG:**

The DKG artifact channel is registered with `SLOT_TABLE_NO_LIMIT = usize::MAX`, meaning a single peer can advertise an unlimited number of DKG artifact slots. Compare this to ingress which uses `SLOT_TABLE_LIMIT_INGRESS = 50_000`. [3](#0-2) [4](#0-3) 

**Root cause 4 — DkgBouncer only filters by height:**

The bouncer accepts any artifact whose `height == current_start_height`, regardless of whether the `target_subnet` is a known remote ID, providing no pre-download filtering of invalid remote target IDs. [5](#0-4) 

**Root cause 5 — Purge only on DKG interval advance:**

The pool is only purged when `start_height > dkg_pool.get_current_start_height()`, i.e., when a new summary block is finalized. With `DKG_INTERVAL_HEIGHT = 499`, deferred artifacts accumulate for up to 499 consensus rounds. [6](#0-5) [7](#0-6) 

**Root cause 6 — CPU amplification via full pool iteration:**

`on_state_change` iterates over all unvalidated dealings on every call, so a large pool causes CPU exhaustion proportional to pool size on every consensus round. [8](#0-7) 

The existing test `test_remote_dealing_validation_is_deferred_until_context_exists` explicitly confirms the deferral behavior is intentional and exercised in practice. [9](#0-8) 

## Impact Explanation

A Byzantine subnet node can cause memory exhaustion and CPU exhaustion on honest replicas by flooding the DKG unvalidated pool with deferred dealings. If enough replicas are affected, consensus throughput degrades or the subnet halts. This matches the **High ($2,000–$10,000)** impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* NiDKG dealings are substantial cryptographic objects (kilobytes to megabytes depending on subnet size), so N fabricated dealings consume O(N × dealing_size) memory. The `on_state_change` CPU amplification compounds the impact on every consensus round for up to 499 rounds.

## Likelihood Explanation

- Requires only a single compromised subnet node, well within the Byzantine fault model (f < n/3).
- No cryptographic capability beyond the node's own valid signing key is needed; the attack exploits the deferral path that skips signature verification entirely.
- The attack is trivially automatable: generate unique 32-byte random target IDs, construct dealing content, sign with the node's key, advertise via P2P.
- No victim mistakes, social engineering, or external dependencies are required.
- The deferral behavior is confirmed by an existing unit test, meaning it is a stable, tested code path.

## Recommendation

1. **Verify the signature before deferring** remote DKG dealings. Move signature verification before the config lookup so only authenticated subnet nodes can contribute deferred entries.
2. **Reject remote DKG dealings with unrecognized target IDs after signature verification.** Once the signature is verified, dealings for unknown remote targets should be `HandleInvalid`-ed unless the target ID appears in a pending `SetupInitialDkg` context in the certified state.
3. **Add a pool size cap to `DkgPoolImpl`**, analogous to `ingress_pool_max_count`/`ingress_pool_max_bytes`, with per-peer accounting.
4. **Apply a per-peer slot table limit for DKG** (replace `SLOT_TABLE_NO_LIMIT` with a bounded constant, e.g., `2 × max_dealers_per_subnet × num_dkg_tags`).

## Proof of Concept

```
1. Identify current DKG start_height H from the consensus pool (public information).
2. For i in 1..N:
   a. Construct DealingContent {
        dealing: <random bytes>,
        dkg_id: NiDkgId {
          start_block_height: H,
          dealer_subnet: <attacker's subnet_id>,
          dkg_tag: NiDkgTag::LowThreshold,
          target_subnet: Remote(NiDkgTargetId::new(<random_32_bytes_i>))
        }
      }
   b. Sign with Byzantine node's valid node key → Message { content, signature }
   c. Advertise via P2P to victim replicas (no slot table limit blocks this)
3. Victim replicas: bouncer accepts (height == start_height), downloads, inserts into
   unvalidated pool (no size limit).
4. on_state_change: validate_dealings_for_dealer returns Mutations::new() for each
   (no config match, is_remote() == true, no signature check, no removal).
5. Assert: dkg_pool.get_unvalidated().count() == N after N insertions.
6. Assert: pool is not purged until next summary block (up to 499 rounds later).
7. Observe: replica memory grows by N × dealing_size; on_state_change CPU time
   grows linearly with N on every round.
```

The existing unit test at `rs/consensus/dkg/src/lib.rs:2093` (`test_remote_dealing_validation_is_deferred_until_context_exists`) already demonstrates steps 3–5 with two fabricated remote dealings from arbitrary peer IDs, confirming the pool accepts and retains them indefinitely. [10](#0-9)

    fn new_bouncer(&self, dkg_pool: &Pool) -> Bouncer<DkgMessageId> {
        let _timer = self.metrics.update_duration.start_timer();

        let start_height = dkg_pool.get_current_start_height();
        Box::new(move |id| {
            use std::cmp::Ordering;
            match id.height.cmp(&start_height) {
                Ordering::Equal => BouncerValue::Wants,
                Ordering::Greater => BouncerValue::MaybeWantsLater,
                Ordering::Less => BouncerValue::Unwanted,
            }
        })
    }
