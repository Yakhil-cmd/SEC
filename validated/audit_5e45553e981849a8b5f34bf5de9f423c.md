### Title
NEAR Attached to `fund_xcc_sub_account` Permanently Locked in Engine on Deployment Failure - (File: `engine/src/xcc.rs`)

### Summary
The `fund_xcc_sub_account` function accepts user-attached NEAR to create or upgrade XCC router sub-accounts. If the downstream batch action (`initialize` or `deploy_upgrade`) fails, NEAR Protocol refunds the attached NEAR to the Aurora engine contract (the promise creator), not to the original user. Because Aurora engine has no failure-path callback to track and return this NEAR, the user's funds are permanently frozen in the engine's balance.

### Finding Description
`fund_xcc_sub_account` in `engine/src/xcc.rs` reads the caller's attached deposit and forwards it to the router sub-account via a NEAR batch action. Two code paths are affected:

**Path 1 – `create_needed = true`** (new router account): [1](#0-0) 

`fund_amount` is forwarded via a `Transfer` action followed by a `FunctionCall` to `initialize`. If `initialize` panics or runs out of gas, the entire batch is rolled back by the NEAR runtime and the NEAR is refunded to the predecessor — Aurora engine — not the user.

**Path 2 – `create_needed = false`** (router upgrade): [2](#0-1) 

`fund_amount` is attached directly to the `deploy_upgrade` function call. If `deploy_upgrade` fails, NEAR Protocol refunds the attached NEAR to the predecessor (Aurora engine).

In both paths, the only callback attached is `factory_update_address_version`: [3](#0-2) 

This callback only updates the engine's internal version record. It does not refund the user on failure: [4](#0-3) 

When the deployment fails, `factory_update_address_version` returns `ERR_ROUTER_DEPLOY_FAILED` but performs no refund. The user's NEAR is now in Aurora engine's undifferentiated balance with no on-chain recovery path.

By contrast, `handle_precompile_promise` (the XCC precompile path) does include a `send_refund` callback for the storage staking amount it covers: [5](#0-4) 

No equivalent refund mechanism exists for the user-supplied `fund_amount` in `fund_xcc_sub_account`.

### Impact Explanation
**Critical – Permanent freezing of funds.** The user's attached NEAR (minimum `STORAGE_AMOUNT = 2_000_000_000_000_000_000_000_000` yoctoNEAR ≈ 2 NEAR for new accounts; arbitrary amounts for upgrades) is permanently locked in Aurora engine's balance. There is no on-chain mechanism for the user to reclaim it. Aurora engine's balance is not tracked per-user, so even the engine owner cannot attribute and return the funds without off-chain coordination.

### Likelihood Explanation
**Moderate.** The upgrade path (`create_needed = false`) is triggered whenever the engine owner calls `factory_update` to push a new router version, after which any user calling `fund_xcc_sub_account` for an existing router triggers `deploy_upgrade`. Failure modes include: insufficient gas allocation (`UPGRADE_GAS + INITIALIZE_GAS = 35 TGas`), a bug in the new router's `deploy_upgrade` or `initialize` logic, or WASM validation failure of the new bytecode. The function is publicly callable (no owner restriction when `wnear_account_id` is `None`): [6](#0-5) 

### Recommendation
Add a failure-path callback after the batch promise in `fund_xcc_sub_account` that refunds `fund_amount` to `env.predecessor_account_id()` if the deployment/upgrade batch fails. This mirrors the `send_refund` pattern already used in `handle_precompile_promise`. Alternatively, record the caller and amount in engine storage before dispatching the promise, and expose a claim function callable after a failed deployment.

### Proof of Concept
1. Engine owner calls `factory_update` with new router bytecode, incrementing the latest code version.
2. User calls `fund_xcc_sub_account` with 5 NEAR attached, targeting their existing (now outdated) router sub-account. `AddressVersionStatus::DeployNeeded { create_needed: false }` is selected.
3. Aurora engine dispatches a batch to the router sub-account: `FunctionCall { name: "deploy_upgrade", attached_yocto: 5 NEAR }`.
4. `deploy_upgrade` fails (e.g., gas exhaustion or panic in the new router code).
5. NEAR Protocol refunds 5 NEAR to Aurora engine (the promise creator / predecessor).
6. `factory_update_address_version` callback fires, detects failure, returns `ERR_ROUTER_DEPLOY_FAILED`, and exits — no refund issued.
7. User's 5 NEAR is permanently locked in Aurora engine's balance with no recovery path.
