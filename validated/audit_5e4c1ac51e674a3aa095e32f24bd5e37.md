Audit Report

## Title
`end_timestamp_seconds: None` in SNS `RewardEvent` Causes Permanent Reward Distribution DoS via Overflow Loop - (File: `rs/sns/governance/src/governance.rs`)

## Summary
When an SNS governance canister holds a `latest_reward_event` whose `end_timestamp_seconds` field is `None` (the protobuf default), both `should_distribute_rewards` and `distribute_rewards` silently treat the last reward timestamp as Unix epoch 0 via `unwrap_or_default()`. This causes `distribute_rewards` to compute `new_rounds_count ≈ now / round_duration_seconds` (~19,675 for a 1-day round), iterate that many rounds of reward arithmetic, overflow the `u64` rewards purse, and return early without ever updating `latest_reward_event`. Because the event is never updated, every subsequent periodic task call repeats the same failing computation, permanently breaking SNS reward distribution.

## Finding Description
`Governance::new` initializes `latest_reward_event` only when the field is `None`:

```rust
if proto.latest_reward_event.is_none() {
    proto.latest_reward_event = Some(RewardEvent {
        end_timestamp_seconds: Some(now),
        ...
    })
}
```

If the proto is supplied with `latest_reward_event: Some(RewardEvent { end_timestamp_seconds: None, ... })` — as occurs for any SNS deployed before `end_timestamp_seconds` was added to the protobuf schema — the guard is bypassed and `end_timestamp_seconds` remains `None` permanently.

`should_distribute_rewards` then evaluates:
```rust
let seconds_since_last_reward_event = now.saturating_sub(
    self.latest_reward_event().end_timestamp_seconds.unwrap_or_default(), // → 0
);
// seconds_since_last_reward_event == now >> round_duration_seconds → always true
```

`distribute_rewards` computes:
```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event().end_timestamp_seconds.unwrap_or_default(); // → 0
let new_rounds_count = now.saturating_sub(0).saturating_div(round_duration_seconds);
// ≈ 19,675 for 1-day rounds at current Unix time
```

The loop `for i in 1..=new_rounds_count` accumulates reward arithmetic over ~19,675 rounds into a `Decimal`. The accumulated value (on the order of 4×10²¹ e8s for a typical supply) exceeds `u64::MAX` (~1.8×10¹⁹), triggering:

```rust
Err(err) => {
    log!(ERROR, "Looks like the rewards purse ({}) overflowed u64: ...");
    return;  // ← latest_reward_event is NOT updated
}
```

Because `latest_reward_event` is never updated, `should_distribute_rewards` returns `true` again on the next timer tick, and the identical failure repeats indefinitely.

## Impact Explanation
This matches **High** impact: "Significant SNS security impact with concrete user or protocol harm." Reward distribution is permanently broken for any affected SNS canister: no neuron maturity is ever increased, the rewards purse is never distributed, and the canister burns instruction budget on every 10-second timer tick. The SNS governance canister remains otherwise functional (voting, proposals), but the core tokenomics incentive mechanism is completely disabled with no self-recovery path.

## Likelihood Explanation
The precondition — `latest_reward_event` present but `end_timestamp_seconds: None` — is met by any SNS instance deployed before `end_timestamp_seconds` was introduced to the `RewardEvent` protobuf message. No privileged access is required. The IC timer fires automatically every 10 seconds (`RUN_PERIODIC_TASKS_INTERVAL`), so the failure occurs without any user interaction. Any unprivileged message to the canister can also advance the timer queue. The condition is permanent once triggered: there is no self-healing path in the code.

## Recommendation
Replace `unwrap_or_default()` with an explicit fallback to `genesis_timestamp_seconds` in both `should_distribute_rewards` and `distribute_rewards`:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or(self.proto.genesis_timestamp_seconds);
```

Additionally, add a post-upgrade migration that sets `end_timestamp_seconds = Some(genesis_timestamp_seconds)` for any stored `RewardEvent` where the field is `None`, to remediate already-deployed affected SNS canisters.

## Proof of Concept
1. Deploy an SNS governance canister with `GovernanceProto { latest_reward_event: Some(RewardEvent { round: 0, end_timestamp_seconds: None, ..Default::default() }), voting_rewards_parameters: Some(<1-day round config>), .. }`.
2. The guard `if proto.latest_reward_event.is_none()` does not fire; `end_timestamp_seconds` remains `None`.
3. On the first `run_periodic_tasks` call (automatic, 10s interval), `should_distribute_rewards` computes `now.saturating_sub(0) > 86400` → `true`.
4. `distribute_rewards` computes `new_rounds_count = now / 86400 ≈ 19,675`.
5. The loop accumulates rewards over 19,675 rounds; for a supply of 1B tokens (1×10¹⁷ e8s) at a 1%/year rate, the purse ≈ 4×10²¹ e8s, overflowing `u64`.
6. The overflow early-return fires; `latest_reward_event` is not updated.
7. The next timer tick (10s later) repeats steps 3–6 identically. Reward distribution is permanently broken.
8. A deterministic PocketIC integration test can reproduce this by initializing governance with the above proto and advancing the IC time by one timer interval.
