Audit Report

## Title
Unprivileged Caller Can Force SNS Swap Participation for Any Buyer via Missing Authorization in `refresh_buyer_tokens` - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal without verifying that the caller is that buyer. Any unprivileged ingress sender can trigger participation registration for a victim who has already transferred ICP and holds an open ticket, deleting the ticket and preventing the victim from canceling via `notify_payment_failure`. If the swap commits, the victim's ICP is swept to SNS governance and they receive unwanted SNS tokens with no recovery path.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 127–143, the `refresh_buyer_tokens` handler resolves the target buyer from `arg.buyer` without any caller-equals-buyer check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no authorization check
};
``` [1](#0-0) 

The proto definition at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` lines 843–851 documents the field as "If not specified, the caller is used," but imposes no restriction on who may specify a different buyer. [2](#0-1) 

Inside `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs`, when the victim's subaccount balance meets or exceeds the ticket amount, the ticket is unconditionally deleted and participation is registered:

```rust
memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow_mut().remove(&principal));
``` [3](#0-2) 

The only cancellation path, `notify_payment_failure`, operates on the caller's own principal and returns `None` once the ticket is gone: [4](#0-3) 

After the ticket is deleted by the attacker's call, the victim's participation entry is written to `self.buyers` and `update_total_participation_amounts()` is called, making the registration permanent for the duration of the OPEN phase. [5](#0-4) 

The `confirmation_text` check provides no meaningful protection: it is a public value set at SNS initialization and readable by anyone. [6](#0-5) 

## Impact Explanation
This is an unauthorized state-mutating operation affecting user funds in an in-scope SNS core canister. A victim who has transferred ICP and holds an open ticket has their participation irrevocably registered without consent. If the swap commits, the victim's ICP is swept to SNS governance and they receive SNS tokens they did not intend to acquire; `error_refund_icp` only covers unregistered ICP and provides no remedy. This matches the High impact category: unauthorized access to/manipulation of governance assets and canister-controlled funds with concrete user harm.

## Likelihood Explanation
All preconditions are routinely met during any active SNS swap: victim principals are public on-chain, the normal participation flow requires transferring ICP before calling `refresh_buyer_tokens`, and the confirmation text (if any) is publicly visible in the SNS init config. No cycles, tokens, or special permissions are required by the attacker. The attack window is the entire OPEN phase and does not require front-running.

## Recommendation
Add a caller-equals-buyer check in the canister endpoint before dispatching:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Only the buyer can refresh their own tokens");
        }
        buyer
    };
    ...
}
```

Alternatively, remove the `buyer` field entirely and always use `caller_principal_id()`, since there is no legitimate use case for an unprivileged third party to register participation on behalf of another user without their consent. [7](#0-6) 

## Proof of Concept
1. User A calls `new_sale_ticket` to create a ticket for amount X ICP.
2. User A transfers X ICP to their swap subaccount on the ICP ledger.
3. User A decides to cancel and intends to call `notify_payment_failure`.
4. Attacker (any principal) calls `refresh_buyer_tokens` with `buyer = <User_A_principal>` (and the public `confirmation_text` if required).
5. Inside `refresh_buyer_token_e8s`: ledger balance = X ≥ ticket amount X → ticket deleted, participation registered for User A.
6. User A calls `notify_payment_failure` → returns `NotifyPaymentFailureResponse { ticket: None }` (ticket already gone); cancellation is impossible.
7. Swap commits → User A's ICP is swept to SNS governance; User A receives unwanted SNS tokens with no ICP recovery path via `error_refund_icp`.

A deterministic integration test using PocketIC can reproduce this by: creating a swap in OPEN state, issuing a ticket for principal A, transferring ICP from principal A's subaccount, then calling `refresh_buyer_tokens` from a distinct principal B with `buyer = A`, and asserting that principal A's subsequent `notify_payment_failure` call returns `ticket: None`.
