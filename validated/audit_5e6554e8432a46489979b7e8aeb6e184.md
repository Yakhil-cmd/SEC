Audit Report

## Title
Stale ICP/XDR Exchange Rate Used Without Freshness Check in Cycles Minting Operations - (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles()` function in the Cycles Minting Canister reads `state.icp_xdr_conversion_rate` and uses only `xdr_permyriad_per_icp`, never comparing `timestamp_seconds` against the current canister time. If the Exchange Rate Canister becomes unavailable and the CMC's heartbeat-driven refresh loop fails persistently, the cached rate can become arbitrarily stale. Any unprivileged user can then call `notify_top_up` or `notify_mint_cycles` to receive cycles computed at the stale (potentially much higher) ICP/XDR rate, minting more cycles per ICP than the current market rate justifies.

## Finding Description
`tokens_to_cycles()` at `rs/nns/cmc/src/main.rs` L1900–1922 extracts only `xdr_permyriad_per_icp` from the stored rate:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);  // timestamp_seconds never read
```

The `timestamp_seconds` field is present in `IcpXdrConversionRate` but is never compared to `now_seconds()` before the rate is used for minting. The only guard against stale rates is in `do_set_icp_xdr_conversion_rate()`, which enforces that a newly arriving rate must have a strictly greater timestamp than the currently stored one — it says nothing about the maximum age of the rate already in state at the time of a minting call.

The `validate_exchange_rate()` function in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` L111–129 checks only `base_asset_num_received_rates >= MINIMUM_ICP_SOURCES` and `quote_asset_num_received_rates >= MINIMUM_CXDR_SOURCES`. It performs no age check on the rate.

The refresh loop in `rs/nns/cmc/src/exchange_rate_canister.rs` targets `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS` (L16). On any retrieval failure, `schedule_next_attempt()` (L133–165) reschedules at the next minute and loops indefinitely without ever updating `state.icp_xdr_conversion_rate`. There is no circuit-breaker that rejects minting when the stored rate has not been refreshed within a bounded window.

Exploit path:
1. XRC canister becomes unavailable (subnet issue, canister trap loop, etc.).
2. CMC heartbeat calls `update_exchange_rate()`, receives `FailedToRetrieveRate`, reschedules at next minute, repeats indefinitely.
3. Real ICP market price drops significantly (e.g., 10 XDR/ICP → 5 XDR/ICP); CMC's cached rate remains at 10 XDR/ICP.
4. Unprivileged attacker sends ICP to CMC subaccount and calls `notify_top_up` (L1140, no access control).
5. `process_top_up` → `tokens_to_cycles()` reads the stale 10 XDR/ICP value and mints ~2× the correct number of cycles.

## Impact Explanation
The protocol mints cycles at an incorrect (stale, inflated) ICP/XDR rate. Users receive significantly more cycles per ICP than the current market rate justifies, constituting illegal over-minting of cycles. The magnitude scales with both the duration of XRC unavailability and the degree of ICP price movement during that window. This maps to the High impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm" — specifically, incorrect cycle issuance that undermines the economic integrity of the ICP/cycles conversion mechanism.

## Likelihood Explanation
The XRC is a live canister dependency on a separate subnet. Any sustained XRC outage, subnet degradation, or CMC heartbeat failure leaves the rate stale with no enforcement at the point of use. The attack requires no privileged access — any principal with ICP can call `notify_top_up` or `notify_mint_cycles`. The window of exploitability grows linearly with XRC downtime. ICP price volatility during an outage is a realistic and historically observed condition.

## Recommendation
In `tokens_to_cycles()`, retrieve the current canister time and compare it against `rate.timestamp_seconds`. Reject the conversion with a `NotifyError` if the rate age exceeds a defined maximum staleness bound (e.g., `REFRESH_RATE_INTERVAL_SECONDS * 2` or a configurable `MAX_RATE_AGE_SECONDS`):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) => {
                let age = now_seconds().saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: format!("ICP/XDR rate is stale ({age}s old)"),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(/* existing None arm */),
        }
    })
}
```

## Proof of Concept
1. Deploy CMC and XRC in a PocketIC or local replica test environment.
2. Record the initial `icp_xdr_conversion_rate` (e.g., 10 XDR/ICP) and its `timestamp_seconds`.
3. Stop or make the XRC canister unresponsive (trap on all calls).
4. Advance the replica clock by more than `REFRESH_RATE_INTERVAL_SECONDS * N` (e.g., 1 hour). Confirm CMC heartbeat fires repeatedly but `state.icp_xdr_conversion_rate.timestamp_seconds` does not advance.
5. Call `notify_top_up` with a valid ICP transfer block index from an unprivileged principal.
6. Assert that the returned `Cycles` value equals the amount computed at the original stale rate (10 XDR/ICP), not at any updated rate — confirming the stale rate is used without rejection.
