### Title
SNS `MintSnsTokens` Proposal Minting Cap Disabled — Unbounded Token Inflation via Governance — (`rs/sns/governance/src/proposal.rs`)

---

### Summary

The `MintSnsTokens` SNS governance proposal action has its treasury-valuation-based 7-day minting cap intentionally disabled in production code. The `recent_amount_total_upper_bound_tokens` implementation for `MintSnsTokens` unconditionally returns `Decimal::MAX`, meaning any SNS neuron majority can pass an unlimited number of `MintSnsTokens` proposals and inflate the SNS token supply without bound, breaking the invariant that minting is bounded by treasury size.

---

### Finding Description

In `rs/sns/governance/src/proposal.rs`, the `TokenProposalAction` trait requires each action to implement `recent_amount_total_upper_bound_tokens`, which is supposed to return the maximum tokens that may be minted/transferred within a rolling 7-day window, derived from the treasury valuation.

For `TransferSnsTreasuryFunds`, this is properly wired to `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens`, which enforces a cap based on treasury size (small/medium/large tiers, max 300,000 XDR per 7 days).

For `MintSnsTokens`, the implementation is:

```rust

---

### Impact Explanation

Any SNS neuron majority can pass repeated `MintSnsTokens` proposals without any 7-day cap being enforced. This allows:

1. **Unbounded token inflation**: The SNS token supply can be increased by an arbitrary amount within any time window, with no on-chain enforcement of the treasury-valuation-based limit.
2. **Broken invariant**: The system is designed so that minting is bounded by treasury size (analogous to the external report's "owner can mint beyond the 10,924 cap"). That invariant is not enforced for `MintSnsTokens`.
3. **Value destruction**: Token holders who expect the minting cap to protect token scarcity are exposed to unlimited dilution via governance proposals.

---

### Likelihood Explanation

The entry path requires only a neuron majority in the SNS — no privileged key, no admin access, no threshold corruption. Any SNS with a concentrated voting majority (e.g., a whale neuron, a founding team with majority stake, or a coordinated group) can exploit this immediately. The code is in production and the bypass is explicit (`Decimal::MAX`). The commented-out test confirms the second mint succeeds when it should fail. [6](#0-5) 

---

### Recommendation

Uncomment the actual cap implementation for `MintSnsTokens::recent_amount_total_upper_bound_tokens` and wire it to `mint_sns_tokens_7_day_total_upper_bound_tokens`, mirroring the `TransferSnsTreasuryFunds` implementation. Delete the `Decimal::MAX` stub. Uncomment the corresponding integration test assertion that verifies the second mint is rejected. [7](#0-6) 

---

### Proof of Concept

1. Deploy an SNS with a whale neuron holding majority voting power.
2. Submit a `MintSnsTokens` proposal for an amount exceeding the treasury-valuation-based 7-day cap (e.g., 100% of treasury value).
3. Pass the proposal. It executes successfully — `recent_amount_total_upper_bound_tokens` returns `Decimal::MAX`, so `proposal_amount_tokens > allowance_remainder_tokens` is never true.
4. Immediately submit another `MintSnsTokens` proposal for the same amount. It also passes.
5. Repeat indefinitely. The SNS token supply inflates without bound, with no on-chain enforcement of the cap.

The integration test at `rs/sns/integration_tests/src/sns_treasury.rs:930-966` already demonstrates step 4 — the "doomed" second proposal succeeds when it should fail, with the correct rejection logic commented out pending ticket NNS1-2982. [8](#0-7)
