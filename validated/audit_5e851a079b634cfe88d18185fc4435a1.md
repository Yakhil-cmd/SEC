### Title
`TreasuryManagerOperation::new_final` Always Produces Non-Final Steps, Breaking Audit Trail Finality — (File: rs/sns/treasury_manager/src/lib.rs)

### Summary

`TreasuryManagerOperation::new_final` is intended to construct a terminal step for a treasury operation, but it unconditionally sets `is_final: false` instead of `is_final: true`. Any code path that relies on `is_final` to determine whether a treasury operation has completed will never observe a final step, mirroring the original report's pattern of a parameter that is validated/named correctly but whose value is never actually applied.

### Finding Description

In `rs/sns/treasury_manager/src/lib.rs`, two constructors exist side-by-side:

```rust
// lines 352-361
pub fn new(operation: Operation) -> Self {
    Self {
        operation,
        step: Step { index: 0, is_final: false },
    }
}

// lines 363-371  ← BUG
pub fn new_final(operation: Operation) -> Self {
    Self {
        operation,
        step: Step { index: 0, is_final: false },  // should be true
    }
}
```

`new_final` is identical to `new`; the `is_final` field is hardcoded to `false` in both. The function name and its documented intent ("final step of an operation") are contradicted by the implementation. Any TreasuryManager implementation that calls `new_final` to signal completion will emit an audit-trail entry that is indistinguishable from an in-progress step.

The `is_final` field is consumed in multiple places across the SNS governance layer (`rs/sns/governance/src/proposal.rs`, `rs/sns/governance/src/governance.rs`) and in the treasury manager DID interface, where it is the only machine-readable signal that a multi-step operation has concluded.

### Impact Explanation

The SNS Treasury Manager is the mechanism by which an SNS DAO deposits and withdraws assets to/from external custodians (DEXes). The audit trail is the only on-chain record of these financial flows. Because `new_final` never sets `is_final: true`:

1. **Audit trail integrity is broken**: No treasury operation will ever be recorded as complete. Governance tooling and off-chain monitors that parse `is_final` to confirm settlement will always see operations as pending.
2. **State-machine logic is affected**: Any SNS governance code that gates subsequent actions (e.g., a second deposit, a withdrawal, or a reward issuance) on observing a final step will be permanently blocked, making those operations impossible — directly analogous to the original report's "settlement may become impossible" impact.
3. **Fund accountability gap**: Tokens transferred to an external custodian during a deposit will have no on-chain "final" confirmation, making it impossible to distinguish a completed deposit from a failed mid-flight one.

### Likelihood Explanation

The `TreasuryManager` library is the mandatory framework for all SNS treasury extensions. Any compliant implementation must call `new_final` to mark the last step of a `Deposit`, `Withdraw`, or `IssueReward` operation. The bug is triggered the first time any SNS DAO deploys a treasury manager and executes an operation — a governance-level action reachable by any SNS token holder who can pass a proposal.

### Recommendation

Change `is_final: false` to `is_final: true` in `new_final`:

```rust
pub fn new_final(operation: Operation) -> Self {
    Self {
        operation,
        step: Step { index: 0, is_final: true },
    }
}
```

Add a unit test asserting `TreasuryManagerOperation::new_final(op).step.is_final == true` to prevent regression.

### Proof of Concept [1](#0-0) 

The two constructors `new` and `new_final` produce byte-for-byte identical `Step` values. Any TreasuryManager implementation calling `new_final` to close an operation will emit `Step { index: 0, is_final: false }`, which is the same value emitted by `new` for the first non-final step. Governance and audit consumers reading `is_final` from the audit trail will never observe `true`, permanently preventing any treasury operation from being marked complete. [2](#0-1) 

The DID file itself acknowledges that slippage/settlement finality is a known risk surface for treasury operations, making correct `is_final` signalling critical for downstream safety checks. [3](#0-2)
