### Title
Unchecked Exit Precompile Call Return Value After Token Burn Causes Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens via `_burn`, then invoking the exit precompile via a low-level assembly `call`. The return value of that assembly `call` is captured in a local variable `res` but is **never checked**. If the precompile call fails (returns 0), the burn is not reverted, and the user's tokens are permanently destroyed with no corresponding NEP-141 transfer on the NEAR side.

---

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow this pattern:

```solidity
// withdrawToNear (lines 53–63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert if call fails
    }
}
``` [1](#0-0) 

```solidity
// withdrawToEthereum (lines 65–76)
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes20 recipient_b = bytes20(recipient);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient_b);
    uint input_size = 1 + 32 + 20;

    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert if call fails
    }
}
``` [2](#0-1) 

The identical pattern exists in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The sequence is:
1. `_burn(sender, amount)` — EVM state is modified; user's ERC-20 balance is reduced.
2. Low-level `call` to the exit precompile — if this sub-call fails (returns 0), execution continues without reverting.
3. `res` is assigned but never inspected; no `require(res == 1, ...)` or equivalent guard exists.

Because `_burn` executes before the assembly block and is not inside a try/catch or conditional, a failed precompile call does not undo the burn. The outer transaction succeeds, the ERC-20 tokens are gone, and no NEAR-side `ft_transfer` or `withdraw` promise is ever scheduled.

The exit precompile (`ExitToNear` / `ExitToEthereum`) can fail for several realistic reasons: the precompile is paused via `EnginePrecompilesPauser`, the input is malformed (e.g., recipient account ID too long or invalid UTF-8), or the sub-call runs out of gas. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

When the precompile call fails silently:
- The user's ERC-20 tokens are permanently burned (EVM accounting updated).
- No NEP-141 tokens are transferred on the NEAR side.
- The underlying NEP-141 balance remains locked in the Aurora engine contract with no mechanism for the user to recover it.

This constitutes **permanent freezing of user funds** — a Critical-severity impact under the defined scope.

---

### Likelihood Explanation

Any unprivileged ERC-20 token holder can trigger this by calling `withdrawToNear` or `withdrawToEthereum` directly. The failure condition is realistic:
- The exit precompile can be paused by the engine's pause mechanism.
- A malformed or oversized `recipient` argument causes the precompile to return an error (e.g., `ERR_INVALID_INPUT`, `ERR_INVALID_RECEIVER_ACCOUNT_ID`), which causes the sub-call to return 0.
- Gas exhaustion in the sub-call (the precompile costs are set to 0 EthGas but the NEAR-side gas budget is finite). [7](#0-6) 

A user who accidentally provides a recipient string that is too long (> `MAX_INPUT_SIZE` after encoding) will have their tokens burned with no recourse. [8](#0-7) 

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures that if the exit precompile call fails, the entire transaction reverts, including the `_burn`, so no tokens are lost.

---

### Proof of Concept

1. Deploy `EvmErc20` (or use the existing Aurora mainnet ERC-20 mirror contract).
2. Call `withdrawToNear(recipient, amount)` where `recipient` is a byte string that, after encoding with the `\x01` flag and 32-byte amount, exceeds `MAX_INPUT_SIZE` (1024 bytes).
3. The precompile's `validate_input_size` check returns `ExitError`, causing the sub-call to return 0.
4. The assembly block does not revert; `_burn` has already executed.
5. Observe: the caller's ERC-20 balance is reduced by `amount`, but no NEAR-side transfer occurs. The tokens are permanently lost. [1](#0-0) [9](#0-8)
