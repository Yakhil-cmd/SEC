Audit Report

## Title
NNS Governance Async Vote Cascade Reads Live Neuron Followees After Proposal Deadline, Enabling Post-Deadline Auto-Vote Manipulation - (File: `rs/nns/governance/src/voting.rs`)

## Summary
When a large following cascade hits the hard instruction limit during `cast_vote_and_cascade_follow`, the remaining cascade work is persisted in `VOTING_STATE_MACHINES` stable storage and deferred to a 3-second timer job (`process_voting_state_machines`). That timer reads **live** neuron followees from the neuron store rather than a snapshot taken at vote-cast time, and contains no check that the proposal deadline has passed. A neuron controller can call `follow()` after the deadline but before the timer processes their ballot, causing the cascade to fill in their ballot based on the new followees.

## Finding Description

**Deferral path.** `cast_vote_and_cascade_follow` (`rs/nns/governance/src/voting.rs:122-180`) loops, making self-calls via `noop_self_call_if_over_instructions` to reset the instruction counter. If the hard limit (`HARD_VOTING_INSTRUCTIONS_LIMIT = 750 * BILLION`) is hit, the loop breaks at line 174 with the cascade still in progress. The unfinished `ProposalVotingStateMachine` is saved to stable storage by `with_machine` (`voting.rs:331-357`) because `is_completely_finished()` returns false. The timer `schedule_vote_processing` (`canister.rs:94-98`) fires every 3 seconds and calls `process_voting_state_machines` (`voting.rs:225-268`), which picks up all pending machines unconditionally — there is no deadline guard anywhere in this path.

**Live follower reads.** Inside `continue_processing` (`voting.rs:506-573`):
1. `add_followers_to_check` (`voting.rs:462-475`) calls `neuron_store.get_followers_by_followee_and_topic`, reading the **current** follower index.
2. For each follower, `neuron_store.neuron_would_follow_ballots` (`neuron_store.rs:774-787`) loads the neuron's **current** `followees` field and passes it to `would_follow_ballots`.

**`follow()` updates immediately.** The `follow` function (`governance.rs:5718-5791`) calls `with_neuron_mut` to update `neuron.followees` synchronously. No check prevents calling `follow()` after a proposal deadline — only `register_vote` checks `accepts_vote`.

**`can_make_decision` defers finalization until cascade completes.** `can_make_decision` (`governance.rs:787-822`) checks `has_unprocessed_votes()` (line 797). While the cascade is pending, `votes_still_processing` is true, so `expired` is false, and the proposal cannot be decided by expiration alone. This means the proposal **must** wait for the cascade to complete — with whatever followees are live at that time — before `process_proposal` can finalize it. After the cascade completes, `process_voting_state_machines` calls `recompute_proposal_tally` and `process_proposal` (`voting.rs:250-252`), which can then decide the proposal using the manipulated tally.

**Attack flow:**
1. A triggering vote is cast on a close proposal near the deadline, initiating a large cascade.
2. The cascade hits the hard instruction limit; remaining work is deferred to the timer.
3. The proposal deadline passes.
4. The attacker calls `follow()` to add their neuron as a follower of the voting neuron, switch followees from a Yes-voter to a No-voter, or remove followees to suppress their ballot.
5. The timer fires, reads the updated follower index and followees, and fills in the attacker's ballot accordingly.
6. `process_proposal` finalizes the proposal using the post-deadline-manipulated tally.

**Existing guards are insufficient.** `cast_vote` (`voting.rs:488-503`) only prevents overwriting an already-cast ballot — it does not prevent filling in an `Unspecified` ballot. `register_vote` checks `accepts_vote` but `follow()` does not. `process_voting_state_machines` has no deadline check.

## Impact Explanation
A neuron controller with an `Unspecified` ballot can observe a pending cascade, wait for the deadline to pass, then call `follow()` to retroactively opt into a Yes or No vote, switch sides, or suppress their auto-vote. Because `can_make_decision` waits for the cascade to complete before finalizing the proposal, the manipulated ballot is counted in the final tally and can decide the outcome. This constitutes unauthorized influence over NNS governance proposals — a **High** severity impact matching "Significant NNS security impact with concrete user or protocol harm" ($2,000–$10,000).

## Likelihood Explanation
The NNS has hundreds of thousands of neurons with complex following graphs. The hard limit of 750 billion instructions accommodates approximately 750,000 neurons; cascades approaching this scale are realistic on mainnet. The attack requires: (1) a neuron with an `Unspecified` ballot, (2) a pending cascade deferred to the timer, (3) a `follow()` call timed between deadline passage and the next timer round (≥3 seconds). The attacker can observe pending cascade state via proposal ballot queries. The attack is repeatable on any sufficiently large cascade and requires no special privileges beyond neuron ownership.

## Recommendation
1. **Snapshot followees at vote-cast time.** Record the set of followers to check at the moment the triggering vote is cast (inside `record_neuron_vote`/`cast_vote`) rather than re-querying the live index in `add_followers_to_check`.
2. **Add a deadline guard in `process_voting_state_machines`.** Before processing a machine, check `proposal.accepts_vote(now, voting_period_seconds)`. If the deadline has passed and the cascade was not initiated before the deadline, skip or discard the machine.
3. **Alternatively, record the deadline at cascade creation time** and store it in `ProposalVotingStateMachine`, then enforce it during timer processing.

## Proof of Concept
A deterministic integration test using PocketIC or the existing `temporarily_set_over_soft_message_limit` test hook:

1. Create a governance instance with a proposal and a large following chain (use `temporarily_set_over_soft_message_limit(true)` to simulate hitting the instruction limit).
2. Cast a vote on neuron 1, triggering a cascade that is deferred to the timer (machine saved in `VOTING_STATE_MACHINES`).
3. Advance time past the proposal deadline.
4. Call `follow()` on neuron N (which has an `Unspecified` ballot) to set it to follow neuron 1.
5. Run the timer via `run_pending_timers_every_interval_for_count`.
6. Assert that neuron N's ballot is now `Yes` (matching neuron 1's vote), demonstrating post-deadline ballot manipulation.
7. Assert that `process_proposal` decided the proposal using the manipulated tally.
