Audit Report

## Title
Missing Length Validation on `Transform.context` and `Transform.method_name` Allows Replicated State Bloat via Canister HTTP Outcalls - (File: `rs/types/types/src/canister_http.rs`)

## Summary
The `CanisterHttpRequestContext` stored in replicated state contains a `Transform` struct whose `method_name` and `context` fields are never individually size-checked during request validation. While URL, headers, and body all have enforced length limits, the transform fields are silently accepted at any size up to the inter-canister message limit (~2 MiB). With `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT = 3000`, a malicious canister can bloat the subnet's replicated state by up to ~6 GiB from this map alone, causing state sync degradation, checkpoint pressure, and potential memory exhaustion.

## Finding Description
`CanisterHttpRequestContext::generate_from_args` (L510–519) and `generate_from_flexible_args` (L596–605) perform three validation steps:

1. `validate_transform_principal` — checks only that the principal ID matches the sender; no size check.
2. `validate_url_length` — enforces `MAX_CANISTER_HTTP_URL_SIZE` (8,192 bytes).
3. `validate_http_headers_and_body` — enforces `MAX_CANISTER_HTTP_HEADER_NUM`, `MAX_CANISTER_HTTP_HEADER_NAME_VALUE_LENGTH`, `MAX_CANISTER_HTTP_HEADER_TOTAL_SIZE`, and `MAX_CANISTER_HTTP_REQUEST_BYTES` on headers and body only.

None of these checks constrain `transform.method_name` or `transform.context`. The `validate_http_headers_and_body` function (L418–465) explicitly computes `request_total_bytes = headers_size_bytes + body_size_bytes` and does not include transform fields in this total.

The `variable_parts_size()` method (L494–508) does account for transform fields:
```rust
+ self.transform.as_ref().map_or(0, |transform| {
    transform.method_name.len() + transform.context.len()
});
```
but this is used only for fee calculation in `try_add_http_context_to_replicated_state` (L2122), not as a hard cap. After fee deduction, the context is unconditionally inserted into `SubnetCallContextManager.canister_http_request_contexts` (L2238–2239), which is part of the subnet's replicated state.

The only bound on the number of in-flight contexts is `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT = 3000` (L204–206 of `rs/config/src/execution_environment.rs`), checked at L2148–2162 of `execution_environment.rs`.

The upper bound on `transform.context` per request is the inter-canister message payload limit: `MAX_INTER_CANISTER_PAYLOAD_IN_BYTES = 2 MiB` (L54–66 of `rs/types/types/src/messages.rs`), since the `CanisterHttpRequestArgs` is carried as the `method_payload` of a `Request`. This gives a worst-case state bloat of 3,000 × 2 MiB ≈ 6 GiB from transform context data alone.

## Impact Explanation
This matches the allowed ICP bounty impact: **High — Application/platform-level DoS, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.**

Concrete consequences:
- **State sync degradation**: Nodes catching up must download and verify a multi-GiB metadata section, significantly slowing recovery and new-node onboarding.
- **Checkpoint pressure**: Periodic checkpointing of a bloated `canister_http_request_contexts` map increases latency for all canisters on the subnet.
- **Memory exhaustion**: Subnets with constrained memory may be unable to hold the full replicated state in memory, causing replica crashes or OOM kills.

Unlike URL and headers — which are bounded and cannot individually exceed their respective limits — the transform context has no enforced ceiling, making it the only unbounded field that persists in replicated state per HTTP outcall.

## Likelihood Explanation
Any canister on a subnet with HTTP outcalls enabled can trigger this. No privileged role, governance majority, or key material is required. The attacker only needs sufficient cycles to pay the fee for 3,000 requests with large transform contexts. The cycles cost scales with `variable_parts_size()`, so the attack is not free, but cycles cost is an economic deterrent, not a security control. The HTTP outcalls feature is enabled on production subnets.

## Recommendation
Enforce an explicit maximum length on `Transform.method_name` and `Transform.context` during request validation in `generate_from_args` and `generate_from_flexible_args`, alongside the existing `validate_url_length` call. Add a new validation function:

```rust
fn validate_transform_context_size(
    transform: &Option<TransformContext>,
) -> Result<(), CanisterHttpRequestContextError> {
    if let Some(tc) = transform {
        let method_len = tc.function.0.method.len();
        let context_len = tc.context.len();
        if method_len > MAX_CANISTER_HTTP_TRANSFORM_METHOD_NAME_LENGTH {
            return Err(...);
        }
        if context_len > MAX_CANISTER_HTTP_TRANSFORM_CONTEXT_LENGTH {
            return Err(...);
        }
    }
    Ok(())
}
```

Reasonable limits (e.g., 256 bytes for `method_name`, 64 KiB for `context`) would align with the existing header limits and prevent replicated state bloat while preserving legitimate use cases.

## Proof of Concept
1. Deploy a canister on a subnet with HTTP outcalls enabled.
2. From the canister, call `ic00::http_request` 3,000 times with:
   - `url`: any valid URL (e.g., `"https://example.com"`)
   - `headers`: empty
   - `body`: empty
   - `transform`: `TransformContext { function: <self>, context: vec![0u8; 2_000_000] }`
3. Each call passes `validate_transform_principal` (principal matches), `validate_url_length` (URL is short), and `validate_http_headers_and_body` (headers and body are empty — total = 0 bytes, well under `MAX_CANISTER_HTTP_REQUEST_BYTES`).
4. Each `CanisterHttpRequestContext` with a 2 MiB `transform.context` is inserted into `SubnetCallContextManager.canister_http_request_contexts`.
5. After 3,000 requests, the map holds ~6 GiB of transform context data in replicated state, causing state sync slowdowns and checkpoint pressure on the subnet.

A deterministic integration test using `StateMachine` or PocketIC can verify this by submitting 3,000 HTTP outcall requests with maximal transform contexts and asserting that the `canister_http_request_contexts` map size exceeds the expected threshold without any validation error being returned.
