Audit Report

## Title
Fee-on-Transfer ERC20 Token Deposit Mints Unbacked ckERC20 Tokens — (`rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`)

## Summary
Both deposit helper contracts emit the caller-supplied `amount` in their deposit events rather than the actual balance delta received by the minter address. The IC minter parses this event value verbatim and mints exactly that amount of ckERC20. For any fee-on-transfer ERC20 token, the minter receives `amount − fee` but mints `amount`, permanently breaking the 1:1 backing invariant and eventually making legitimate withdrawals impossible.

## Finding Description
**Step 1 — Helper contracts emit input `amount`, not received amount.**

`ERC20DepositHelper.sol` `deposit()` at lines 498–503:
```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
```
`DepositHelperWithSubaccount.sol` `depositErc20()` at lines 519–531:
```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```
Neither contract measures the balance delta at `minterAddress` before and after the transfer. For a fee-on-transfer token, `safeTransferFrom` succeeds but the minter receives only `amount − fee`; the emitted event still carries the full `amount`.

**Step 2 — The IC minter parses and mints the raw event value.**

`parser.rs` `ReceivedErc20LogParser::parse_log` at lines 86–103 reads `value_bytes` directly from `entry.data` and constructs `ReceivedErc20Event { value: Erc20Value::from_be_bytes(value_bytes), ... }` with no cross-check against on-chain balance.

`deposit.rs` `mint()` at lines 73–81 calls `client.transfer(TransferArg { amount: event.value(), ... })`, minting the event-logged amount unconditionally.

**Step 3 — No existing guard prevents this.**

The minter's token whitelist (`ckerc20_tokens`) gates which ERC20 logs are scraped, but it does not enforce that a whitelisted token is non-deflationary. Critically, a token that is standard at listing time can later have fees enabled by its own owner (e.g., via an upgradeable proxy) without any new NNS governance action. Once fees are active, every deposit by any user mints unbacked ckERC20.

## Impact Explanation
Every deposit of a fee-on-transfer token mints `amount` ckERC20 while the minter holds only `amount − fee` ERC20. The ckERC20 total supply grows faster than the ERC20 reserve. When users call `withdraw_erc20`, the minter constructs Ethereum transactions for the full ckERC20 burn amount but its Ethereum address lacks sufficient ERC20 balance. Withdrawals fail or drain reserves belonging to other depositors, constituting direct, permanent loss of funds for legitimate ckERC20 holders and protocol insolvency for the affected ckERC20 token. This matches the allowed impact: **illegal minting and protocol insolvency involving in-scope chain-key/ledger assets**.

## Likelihood Explanation
The precondition is that a whitelisted ckERC20 token has active transfer fees. This can occur without any new NNS governance action if a previously-whitelisted token's contract is upgraded by its own owner to enable fees (e.g., USDT has a configurable fee mechanism currently set to zero; ckUSDT is already whitelisted). NNS voters review token metadata and contract addresses during onboarding, not ongoing bytecode-level transfer semantics. Once fees are active, any unprivileged user can trigger the exploit by calling `depositErc20` or `deposit` — no special privileges are required for the exploit itself.

## Recommendation
1. **Measure actual balance delta in the helper contracts.** Record the minter's ERC20 balance before and after `safeTransferFrom` and emit the delta:
```solidity
uint256 before = IERC20(erc20Address).balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 received = IERC20(erc20Address).balanceOf(minterAddress) - before;
emit ReceivedEthOrErc20(erc20Address, msg.sender, received, principal, subaccount);
```
2. **Add an invariant check in the minter.** When scraping logs, cross-check the ERC20 balance of the minter's Ethereum address against the cumulative minted supply and halt minting if a discrepancy is detected.
3. **Enforce a no-fee-on-transfer check during token onboarding.** The ledger suite orchestrator or NNS proposal tooling should verify that `transferFrom(addr, addr, 1)` returns a balance delta of exactly `1` before a token is whitelisted.

## Proof of Concept
1. A fee-on-transfer ERC20 token `FEE` (2% fee) is already whitelisted, or a whitelisted token's owner enables a 2% fee via contract upgrade.
2. Attacker calls `FEE.approve(helperContract, 1000)`.
3. Attacker calls `depositErc20(FEE, 1000, principal, subaccount)` on `DepositHelperWithSubaccount`.
4. `safeTransferFrom` executes: minter receives `980 FEE`. The helper emits `ReceivedEthOrErc20(..., 1000, ...)`.
5. IC minter scrapes the log, parses `value = 1000`, mints `1000 ckFEE` to attacker.
6. Attacker holds `1000 ckFEE` backed by only `980 FEE`.
7. Attacker calls `withdraw_erc20` for `1000 ckFEE`. Minter burns `1000 ckFEE` and attempts to send `1000 FEE` — only `980 FEE` available, causing withdrawal failure or theft of `20 FEE` from other depositors' reserves.
8. Repeatable indefinitely; each deposit widens the insolvency gap.
