### Title
`CW20ERC20Pointer#transferFrom` Always Requires Allowance Even When `msg.sender == from` - (`contracts/src/CW20ERC20Pointer.sol`)

### Summary

`CW20ERC20Pointer.transferFrom` unconditionally dispatches a CW20 `transfer_from` message to the underlying CosmWasm contract, which always enforces an allowance check. When `msg.sender == from`, the call reverts unless the caller has pre-approved themselves, breaking ERC20 standard compatibility and causing tokens to become irreversibly stranded in any protocol that uses the pull-only `transferFrom(msg.sender, dst, amount)` pattern.

### Finding Description

The `transferFrom` implementation in `CW20ERC20Pointer` always routes through the CW20 `transfer_from` message:

```solidity
function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
    require(to != address(0), "ERC20: transfer to the zero address");
    string memory sender = _formatPayload("owner", _doubleQuotes(AddrPrecompile.getSeiAddr(from)));
    string memory recipient = _formatPayload("recipient", _doubleQuotes(AddrPrecompile.getSeiAddr(to)));
    string memory amt = _formatPayload("amount", _doubleQuotes(Strings.toString(amount)));
    string memory req = _curlyBrace(_formatPayload("transfer_from", _curlyBrace(_join(_join(sender, recipient, ","), amt, ","))));
    _execute(bytes(req));
    return true;
}
``` [1](#0-0) 

There is no `if (from != msg.sender)` guard. The CW20 `transfer_from` message on the CosmWasm side always validates that the caller (`msg.sender`, translated to a Sei address) has a sufficient allowance from `from`. When `msg.sender == from`, no such allowance exists by default, so the call reverts with `"CosmWasm execute failed"`.

By contrast, the ERC20 standard (and Sei's own `WSEI.sol` and `evmrpc/solidity/ERC20.sol`) correctly skip the allowance check when `src == msg.sender`:

```solidity
if (src != msg.sender && allowance[src][msg.sender] != type(uint).max) {
    require(allowance[src][msg.sender] >= wad);
    allowance[src][msg.sender] -= wad;
}
``` [2](#0-1) 

`NativeSeiTokensERC20` is unaffected because it inherits OpenZeppelin's `ERC20` and only overrides `_update`; OZ's `transferFrom` correctly handles the `spender == owner` case. [3](#0-2) 

### Impact Explanation

Any EVM protocol that uses the pull-only pattern — calling `token.transferFrom(msg.sender, address(this), amount)` instead of `token.transferFrom(user, address(this), amount)` after a prior `approve` — will always revert when the token is a CW20 pointer. Tokens deposited into such a protocol via a prior step cannot be moved out, leaving them permanently stranded. This maps to **Medium** under Sei's bounty criteria (fund loss/freeze < $5k).

### Likelihood Explanation

The pull-only `transferFrom(msg.sender, ...)` pattern is common in DeFi routers, vaults, and aggregators. Any such protocol deployed on Sei that integrates CW20 pointer tokens will silently break for this code path. The path is fully unprivileged — any user calling `transferFrom` with `from == msg.sender` triggers the revert.

### Recommendation

Add a self-transfer short-circuit in `CW20ERC20Pointer.transferFrom`: if `msg.sender == from`, use the CW20 `transfer` message (which does not require allowance) instead of `transfer_from`:

```solidity
function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
    require(to != address(0), "ERC20: transfer to the zero address");
    if (from == msg.sender) {
        // No allowance needed; use direct transfer
        string memory recipient = _formatPayload("recipient", _doubleQuotes(AddrPrecompile.getSeiAddr(to)));
        string memory amt = _formatPayload("amount", _doubleQuotes(Strings.toString(amount)));
        string memory req = _curlyBrace(_formatPayload("transfer", _curlyBrace(_join(recipient, amt, ","))));
        _execute(bytes(req));
    } else {
        string memory sender = _formatPayload("owner", _doubleQuotes(AddrPrecompile.getSeiAddr(from)));
        string memory recipient = _formatPayload("recipient", _doubleQuotes(AddrPrecompile.getSeiAddr(to)));
        string memory amt = _formatPayload("amount", _doubleQuotes(Strings.toString(amount)));
        string memory req = _curlyBrace(_formatPayload("transfer_from", _curlyBrace(_join(_join(sender, recipient, ","), amt, ","))));
        _execute(bytes(req));
    }
    return true;
}
```

### Proof of Concept

1. Deploy any CW20 token and its `CW20ERC20Pointer`.
2. Deploy a simple vault contract with: `token.transferFrom(msg.sender, address(this), amount)`.
3. Call `vault.deposit(amount)` — the call reverts with `"CosmWasm execute failed"` even though the caller owns the tokens, because no self-allowance exists.
4. Compare: calling `token.approve(address(vault), amount)` first and then `vault.deposit(amount)` succeeds — confirming the root cause is the missing `msg.sender == from` bypass in `CW20ERC20Pointer.transferFrom`. [1](#0-0)
