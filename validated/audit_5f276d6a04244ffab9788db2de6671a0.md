### Title
Silent Swallow of Failed `near_withdraw` Promise in `exit_to_near_precompile_callback` Leads to Permanent Loss of User wNEAR Tokens - (`engine/src/contract_methods/connector.rs`)

---

### Summary

`exit_to_near_precompile_callback` uses an asymmetric three-branch conditional to handle the result of the base promise (e.g. `near_withdraw`). When the base promise **fails** and `args.refund` is `None`, the function silently returns `Ok(None)` without refunding the user. This is structurally identical to the reported bug: a failed callback is swallowed in the "else" branch, while the user's ERC-20 wNEAR tokens have already been burned in the EVM.

---

### Finding Description

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` handles the result of the base promise as follows:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success: optionally transfer unwrapped NEAR
    if let Some(args) = args.transfer_near { ... }
    None
} else if let Some(args) = args.refund {
    // failure with refund args: refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
    Some(refund_result)
} else {
    None  // <-- silent swallow of failed promise
};
``` [1](#0-0) 

The callback is scheduled by the `ExitToNear` precompile whenever `callback_args != ExitToNearPrecompileCallbackArgs::default()`, which occurs when either `refund` or `transfer_near` is `Some`: [2](#0-1) 

`args.refund` is populated only when the `error_refund` compile-time feature is enabled:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [3](#0-2) 

`transfer_near` is `Some` specifically for the wNEAR unwrap path (`exit_erc20_token_to_near` with `:unwrap` suffix): [4](#0-3) 

**Vulnerable execution path (without `error_refund` feature):**

1. User calls the `ExitToNear` precompile with a wNEAR ERC-20 token and `:unwrap` suffix.
2. The EVM burns the user's wNEAR ERC-20 tokens.
3. The precompile schedules a `near_withdraw` call on the wNEAR NEP-141 contract, with `exit_to_near_precompile_callback` as the callback. `refund = None`, `transfer_near = Some(...)`.
4. The `near_withdraw` call fails (e.g. insufficient wNEAR balance on Aurora, contract paused, or any other revert).
5. `exit_to_near_precompile_callback` runs: `promise_result(0)` is `Some(PromiseResult::Failed)`, so the first branch is skipped. `args.refund` is `None`, so the second branch is skipped. The `else { None }` branch executes, returning `Ok(None)`.
6. No refund is issued. The user's wNEAR ERC-20 tokens are permanently destroyed with no NEAR received.

The `ExitToNearPrecompileCallbackArgs` struct confirms `refund` is `Option<RefundCallArgs>` and defaults to `None`: [5](#0-4) 

The `else { None }` branch at line 240–242 is the direct analog of the "empty catch" in the reported Distributor bug — a failed promise result is silently discarded with no error propagation and no state correction. [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing / theft of user funds.**

When `error_refund` is not compiled in (or in any deployment where `refund` is `None`), a user who calls the wNEAR unwrap exit path has their ERC-20 wNEAR tokens burned in the EVM before the NEAR-side promise is dispatched. If the `near_withdraw` call fails for any reason, the callback silently succeeds (`Ok(None)`), no refund EVM transaction is executed, and the user's tokens are permanently lost. The EVM state change (burn) is committed; the NEAR-side transfer never happens; no recovery path exists.

---

### Likelihood Explanation

The `near_withdraw` call can fail due to:
- Insufficient wNEAR balance held by the Aurora engine account (e.g. after a large concurrent exit drains it).
- The wNEAR contract being paused or upgraded.
- Gas exhaustion on the NEAR side.

Any of these conditions, which are reachable by normal protocol operation without admin compromise, trigger the silent swallow. The attacker does not need to do anything special — they simply call the wNEAR unwrap exit at a moment when the base promise will fail.

---

### Recommendation

The `else { None }` branch must not silently succeed when the base promise has failed. The fix is to explicitly check the promise result and return an error if it failed and no refund path is available:

```rust
} else {
    // If the promise failed and there are no refund args, this is an unrecoverable
    // state: the user's tokens were burned but cannot be refunded.
    if let Some(PromiseResult::Failed) = handler.promise_result(0) {
        return Err(errors::ERR_PROMISE_FAILED_NO_REFUND.into());
    }
    None
};
```

Alternatively, ensure that `refund` is always populated whenever a callback is scheduled (i.e. make `error_refund` a mandatory feature for any deployment that supports wNEAR unwrap), so the `else { None }` branch is structurally unreachable.

---

### Proof of Concept

**Setup:** Deploy Aurora without the `error_refund` feature. Bridge wNEAR to Aurora ERC-20. Ensure the Aurora engine's wNEAR NEP-141 balance is zero (or pause the wNEAR contract).

**Steps:**
1. User holds wNEAR ERC-20 tokens on Aurora.
2. User calls the `ExitToNear` precompile with flag `0x01`, amount `X`, recipient `user.near`, and `:unwrap` suffix.
3. The EVM burns `X` wNEAR ERC-20 tokens from the user's balance.
4. A `near_withdraw` promise is dispatched to the wNEAR NEP-141 contract. It fails (zero balance / paused).
5. `exit_to_near_precompile_callback` is invoked. `args.refund = None`, `promise_result(0) = Failed`.
6. The `else { None }` branch executes. The function returns `Ok(None)`.
7. **Result:** User's `X` wNEAR ERC-20 tokens are gone. No NEAR is received. No refund is issued. Funds are permanently lost. [7](#0-6) [8](#0-7)

    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);

                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
