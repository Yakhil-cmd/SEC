Audit Report

## Title
`ManageVotingPermission` Self-Propagation via `add_neuron_permissions` — (`rs/sns/governance/src/neuron.rs`, `rs/sns/governance/src/governance.rs`)

## Summary
`ManageVotingPermission` is included in `PERMISSIONS_RELATED_TO_VOTING`, which causes `is_exclusively_voting_related()` to return `true` for a permission list containing only `ManageVotingPermission`. Because `check_principal_authorized_to_change_permissions` accepts `ManageVotingPermission` as sufficient authority to grant any exclusively-voting-related permission, a principal holding only `ManageVotingPermission` can grant `ManageVotingPermission` to arbitrary third parties. Those parties are then in an identical position and can repeat the grant, expanding unauthorized voting control over the neuron.

## Finding Description

**Root cause — `PERMISSIONS_RELATED_TO_VOTING` includes `ManageVotingPermission`:** [1](#0-0) 

**`is_exclusively_voting_related()` therefore returns `true` for `[ManageVotingPermission]`:** [2](#0-1) 

**`check_principal_authorized_to_change_permissions` accepts `ManageVotingPermission` as sufficient when the list is exclusively voting-related:** [3](#0-2) 

**`add_neuron_permissions` calls these two checks in sequence with no additional guard:** [4](#0-3) 

**Exploit chain:**
1. Neuron N has principal A → `[ManageVotingPermission]`.
2. A calls `manage_neuron(N, AddNeuronPermissions { principal: B, permissions: [ManageVotingPermission] })`.
3. `[ManageVotingPermission].is_exclusively_voting_related()` → `true`.
4. A holds `ManageVotingPermission` → `check_principal_authorized_to_change_permissions` returns `Ok`.
5. `check_permissions_are_grantable` passes because production SNS deployments set `neuron_grantable_permissions` to all permissions.
6. B now holds `ManageVotingPermission` on N and can repeat steps 2–5 for C, D, … up to `max_number_of_principals_per_neuron`.
7. Any of B, C, … can grant `Vote`/`SubmitProposal` to themselves and cast votes with N's stake.

**Production `neuron_grantable_permissions` confirmation** — `SnsInitPayload::get_nervous_system_parameters` sets `neuron_grantable_permissions` to `NeuronPermissionType::all()`: [5](#0-4) 

**Existing guards are insufficient:** The only additional check, `max_number_of_principals_per_neuron` (default 5), limits fan-out but does not prevent the self-propagation or the subsequent unauthorized voting.

## Impact Explanation

A principal holding only `ManageVotingPermission` — the standard hotkey configuration for voting delegation — can expand voting control over a neuron to additional colluding principals without the neuron owner's consent. Each new principal can immediately grant themselves `Vote` and `SubmitProposal`, casting votes with the victim neuron's stake in SNS governance. If the original owner does not hold `ManagePrincipals` (e.g., swap-claimed neurons with only voting permissions), there is no path to revoke the rogue grants. This constitutes unauthorized access to neuron governance assets and matches the **High** bounty impact: *"Unauthorized access to neurons, governance assets … where exploitation requires meaningful per-target work or other constraints."*

## Likelihood Explanation

Both preconditions are routinely satisfied in production:
1. `ManageVotingPermission` is a standard hotkey permission granted to voting delegates.
2. `SnsInitPayload::get_nervous_system_parameters` sets `neuron_grantable_permissions` to all permissions for every SNS launched via the standard init path.

The attack requires no privileged access, no key compromise, and no social engineering — only a standard `manage_neuron` ingress call from a principal that already holds `ManageVotingPermission`.

## Recommendation

Remove `ManageVotingPermission` from `PERMISSIONS_RELATED_TO_VOTING` in `rs/sns/governance/src/neuron.rs`:

```rust
pub const PERMISSIONS_RELATED_TO_VOTING: &'static [NeuronPermissionType] = &[
    NeuronPermissionType::Vote,
    NeuronPermissionType::SubmitProposal,
    // ManageVotingPermission removed — granting it requires ManagePrincipals
];
```

This preserves the intended behavior (a `ManageVotingPermission` holder can grant/revoke `Vote` and `SubmitProposal`) while requiring `ManagePrincipals` to delegate `ManageVotingPermission` itself.

## Proof of Concept

A deterministic unit test using the existing `GovernanceCanisterFixture` infrastructure in `rs/sns/governance/tests/`:

```rust
// Setup: neuron with principal A holding only ManageVotingPermission
// neuron_grantable_permissions = all
let fixture = GovernanceCanisterFixtureBuilder::new()
    .with_neuron_grantable_permissions(NeuronPermissionList {
        permissions: NeuronPermissionType::all(),
    })
    .add_neuron_with_permissions(
        &[(principal_a, NeuronPermissionList {
            permissions: vec![NeuronPermissionType::ManageVotingPermission as i32],
        })],
        neuron_id.clone(),
    )
    .create();

// Step 1: A grants ManageVotingPermission to B — must succeed
let result = fixture.manage_neuron(
    &neuron_id, &principal_a,
    Command::AddNeuronPermissions(AddNeuronPermissions {
        principal_id: Some(principal_b),
        permissions_to_add: Some(NeuronPermissionList {
            permissions: vec![NeuronPermissionType::ManageVotingPermission as i32],
        }),
    }),
);
assert!(result.is_ok());

// Step 2: B grants Vote to itself — must succeed
let result = fixture.manage_neuron(
    &neuron_id, &principal_b,
    Command::AddNeuronPermissions(AddNeuronPermissions {
        principal_id: Some(principal_b),
        permissions_to_add: Some(NeuronPermissionList {
            permissions: vec![NeuronPermissionType::Vote as i32],
        }),
    }),
);
assert!(result.is_ok()); // B can now vote with neuron N's stake
```
