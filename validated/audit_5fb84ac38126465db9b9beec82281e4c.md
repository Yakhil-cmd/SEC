Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Breaks Conservation Invariant - (File: `rs/nns/cmc/src/main.rs`)

## Summary
In `rs/nns/cmc/src/main.rs`, the functions `process_mint_cycles`, `process_top_up`, and `process_create_canister` all call `burn_and_log(sub, amount).await;` after successfully depositing cycles, but `burn_and_log` is explicitly typed to return `()` and internally swallows all errors. If the ICP ledger call inside `burn_and_log` fails (e.g., during a ledger upgrade), cycles are permanently minted without a corresponding ICP burn, breaking the ICP/cycles conservation invariant.

## Finding Description
The root cause is in `burn_and_log` at line 2017, whose signature is `async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens)` — it returns unit and the comment at line 2015 explicitly states: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."*

This design conflates two separate concerns: (1) preventing duplicate notification retries, and (2) handling burn failures. The deduplication mechanism (`set_block_status_to_processing`) marks the block as `Processing` before the cycles deposit, so even if the caller returned an error on burn failure, the block would already be marked and could not be re-notified. The "prevent retry" justification does not require silently swallowing burn errors.

The exploit path in `process_top_up` (lines 1999–2002):
```
deposit_cycles(canister_id, cycles, ...) -> Ok(())   // cycles delivered
burn_and_log(sub, amount).await;                      // ICP burn silently fails
Ok(cycles)                                            // caller gets success
```
The same pattern exists in `process_mint_cycles` (line 1968) and `process_create_canister` (line 1945). On burn failure, `burn_and_log` logs the error at line 2044–2047 and returns. The ICP remains in the CMC's per-user subaccount with no recovery path, while the cycles are already in the recipient's account.

## Impact Explanation
This constitutes **illegal minting** of cycles: cycles are created and delivered to a recipient without the corresponding ICP being burned. The ICP is permanently locked in the CMC subaccount (the deduplication prevents re-notification, so there is no recovery path). The net effect is that the total cycles supply exceeds what the burned ICP supply should support, violating the protocol's conservation invariant. This matches the allowed impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles."* Severity is **High** given the timing constraint required to trigger it.

## Likelihood Explanation
An unprivileged user can trigger this by monitoring public NNS governance proposals for ICP ledger upgrade votes, sending ICP to the CMC subaccount in advance, and calling `notify_top_up` or `notify_mint_cycles` during the upgrade window when the ledger canister is briefly stopped. NNS upgrade proposals are public and their execution timing is predictable. The attack requires no special privileges, no social engineering, and no key compromise — only timing a standard CMC notification call during a known ledger maintenance window. The attack is repeatable across multiple upgrade windows.

## Recommendation
Separate the "prevent retry" concern from the "burn must succeed" concern. The block is already marked `Processing` before cycles are deposited, so returning an error on burn failure does not enable retries. The correct fix is one of:
1. **Retry burn asynchronously**: persist a "pending burn" record and retry via a timer task (analogous to how the CMC already retries other operations).
2. **Propagate the error**: change `burn_and_log` to return `Result<BlockIndex, String>`, propagate the error up through `process_top_up`/`process_mint_cycles`, and return a distinct error code to the caller indicating the operation is in an inconsistent state requiring operator intervention.
3. **At minimum**: if the burn fails, do not return `Ok` — return a new `NotifyError` variant (e.g., `BurnFailed`) so the inconsistency is surfaced and the block status can be set to a recoverable state.

## Proof of Concept
1. Observe a pending NNS proposal to upgrade the ICP ledger canister.
2. Transfer 10 ICP to the CMC subaccount for canister `C` with `MEMO_TOP_UP_CANISTER`.
3. Wait for the NNS proposal to execute (ledger canister stops briefly during upgrade).
4. During the upgrade window, call `notify_top_up` for canister `C`.
5. CMC executes `deposit_cycles(C, cycles, ...)` — succeeds; canister `C` receives cycles.
6. CMC calls `burn_and_log(sub, 10_ICP).await` — the ledger is stopped; `call_protobuf` returns a rejection; `burn_and_log` logs the error and returns `()`.
7. `process_top_up` returns `Ok(cycles)` to the caller.
8. **Result**: canister `C` holds the minted cycles; 10 ICP is permanently locked in the CMC subaccount; the block is marked processed and cannot be re-notified; cycles exist without a corresponding ICP burn.

A deterministic integration test can reproduce this by: deploying a mock ICP ledger that rejects `send_pb` calls, calling `notify_top_up`, and asserting that cycles were deposited while the CMC subaccount balance remains non-zero and no burn block was recorded. /// Attempt to burn the funds.
```rust
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
```
