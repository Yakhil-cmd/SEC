Audit Report

## Title
Unprivileged Caller Can Force Arbitrary Principal Into SNS Swap Participation — (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update method on the SNS Swap canister accepts an arbitrary `buyer` principal from any caller without verifying that the caller is the buyer. An attacker can fund a victim's subaccount on the ICP ledger and call `refresh_buyer_tokens` with `buyer=victim_principal`, registering the victim as a swap participant without their consent. If the swap commits, the victim receives unwanted SNS neurons and cannot recover their ICP; if the swap aborts, the attacker loses the ICP they sent with no recovery path.

## Finding Description
In `canister.rs` at L130–134, when `arg.buyer` is non-empty the caller identity is completely discarded:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // caller never checked
};
``` [1](#0-0) 

The resolved principal `p` is passed directly to `refresh_buyer_token_e8s` with no caller-equals-buyer assertion. [2](#0-1) 

Inside `refresh_buyer_token_e8s`, the only input validation is `validate_confirmation_text` — there is no check that the caller matches `buyer`. [3](#0-2) 

The function then queries the ICP ledger balance of `subaccount(this_canister, buyer)` and credits it to `buyer`: [4](#0-3) 

The buyer is unconditionally registered in `self.buyers` and participation totals are updated: [5](#0-4) 

The `confirmation_text` check is not a mitigation: the required text is set at SNS initialization and is publicly readable from the swap state. The proto comment at L844 documents the intended behavior as "If not specified, the caller is used," confirming the design intent was for the caller to be the buyer — but this is never enforced. [6](#0-5) 

Once registered, `error_refund_icp` is blocked for the victim while their ICP remains undisbursed: [7](#0-6) 

## Impact Explanation
This matches **High ($2,000–$10,000): Significant SNS security impact with concrete user or protocol harm.**

- A victim is registered as a swap participant without ever calling any method themselves.
- If the swap commits, `sweep_icp` transfers the victim's ICP to SNS Governance and mints SNS neurons for the victim. The victim receives unwanted SNS neurons (with potential tax, regulatory, or reputational consequences) and cannot recover the ICP.
- If the swap is near its `max_direct_participation_icp_e8s` target, the attacker can push it over the threshold, triggering early commitment and locking out other legitimate participants.
- The victim has no recourse while registered as a buyer with undisbursed ICP, as `error_refund_icp` explicitly blocks refunds in that state.

## Likelihood Explanation
- Requires no privileged role — any IC principal can call `refresh_buyer_tokens` via ingress.
- The only cost to the attacker is the ICP they send to the victim's subaccount, which they cannot recover regardless of outcome (if the swap aborts, the ICP is returned to the victim, not the attacker).
- The confirmation text, if required, is publicly readable from the swap state.
- The attack is fully executable on-chain with no off-chain dependencies and is repeatable against any principal.

## Recommendation
Add a caller-equals-buyer guard in `canister.rs` at the top of `refresh_buyer_tokens`. When `arg.buyer` is non-empty, verify that `PrincipalId::from_str(&arg.buyer).unwrap() == caller_principal_id()` and reject the call otherwise. This enforces the documented intent from the proto comment ("If not specified, the caller is used") and ensures only the buyer can register their own participation. [8](#0-7) 

## Proof of Concept
State-machine test:

```rust
// Setup: swap is OPEN, attacker != victim
let attacker = PrincipalId::new_user_test_id(1);
let victim   = PrincipalId::new_user_test_id(2);

// Step 1: attacker sends ICP to victim's subaccount on the swap canister
icp_ledger_transfer(
    attacker,
    Account { owner: swap_canister_id, subaccount: Some(principal_to_subaccount(&victim)) },
    min_participant_icp_e8s,
);

// Step 2: attacker calls refresh_buyer_tokens with buyer=victim
let response = state_machine.execute_ingress_as(
    attacker,
    swap_canister_id,
    "refresh_buyer_tokens",
    Encode!(&RefreshBuyerTokensRequest {
        buyer: victim.to_string(),
        confirmation_text: Some(publicly_known_confirmation_text),
    }).unwrap(),
);

// Step 3: victim is now registered as a buyer without ever calling anything
assert!(response.is_ok());
let buyer_state = swap.get_buyer_state(&GetBuyerStateRequest { principal_id: Some(victim) });
assert!(buyer_state.buyer_state.is_some());
```

The canister accepts this call and registers the victim. The fix is to assert `caller == buyer` before proceeding in `canister.rs`. [1](#0-0)
