Audit Report

## Title
Byzantine Block Proposer Injects Unvalidated Bitcoin Adapter Responses, Consuming Arbitrary Pending Contexts — (`rs/replicated_state/src/bitcoin.rs`, `rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary

`validate_payload` (the `BatchPayloadBuilder` impl for `BitcoinPayloadBuilder`) performs no cross-check of `callback_id` values against pending requests in replicated state, accepting any well-formed, size-bounded payload. Three of four arms in `push_response` also perform no such check, unconditionally enqueuing a `ConsensusResponse` into `state.consensus_queue`. A single Byzantine block proposer can therefore finalize a block containing fabricated `callback_id` values, causing the execution scheduler to consume arbitrary pending subnet call contexts and deliver spurious responses to requesting canisters.

## Finding Description

**Gap 1 — `validate_payload` performs no state lookup.**

`validate_payload` in `rs/bitcoin/consensus/src/payload_builder.rs` (lines 357–397) parses bytes into `Vec<BitcoinAdapterResponse>`, calls `validate_self_validating_payload_impl` which only measures byte size (lines 234–251), checks the size limit, and returns `Ok(())`. No step verifies that any `callback_id` in the payload corresponds to an entry in `bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`. The `delivered_ids` computed at line 369 is never used to reject unknown IDs.

**Gap 2 — `push_response` skips state verification for three response variants.**

In `rs/replicated_state/src/bitcoin.rs`:

- `GetSuccessorsResponse` (lines 28–63): checks `bitcoin_get_successors_contexts.get_mut(&callback_id).ok_or_else(|| StateError::BitcoinNonMatchingResponse{…})?` — **guarded**.
- `SendTransactionResponse` (lines 64–76): no lookup, unconditionally pushes `ConsensusResponse::new(callback_id, EmptyBlob)`.
- `GetSuccessorsReject` (lines 77–88): no lookup, unconditionally pushes reject payload.
- `SendTransactionReject` (lines 90–102): no lookup, unconditionally pushes reject payload.

**Gap 3 — Demux silently swallows errors.**

In `rs/messaging/src/routing/demux.rs` (lines 86–94), `push_response_bitcoin` errors are handled with `unwrap_or_else(|err| { debug!(...) })` — even the existing guard on `GetSuccessorsResponse` is effectively bypassed at the demux level.

**Exploit path:**

1. Byzantine proposer observes `next_callback_id` and the contents of `subnet_call_context_manager` from replicated state (fully observable by every node).
2. Proposer crafts a `BitcoinAdapterResponse { response: SendTransactionResponse {}, callback_id: <target_id> }` where `<target_id>` matches any pending context (e.g., a `SignWithThreshold` or `BitcoinSendTransactionInternal` entry).
3. Proposer includes this in a proposed block. Honest validators call `validate_payload`, which only checks byte size — block is accepted and finalized.
4. During execution, `push_response_bitcoin` is called; the `SendTransactionResponse` arm pushes `ConsensusResponse::new(target_id, EmptyBlob)` to `consensus_queue` without any state check.
5. The scheduler drains `consensus_queue` (lines 1299–1325 of `rs/execution_environment/src/scheduler.rs`), wrapping each entry as `SubnetMessage::Response { originator_reply_callback: response.callback, response_payload: response.payload, … }`.
6. `execute_subnet_message` calls `retrieve_context(target_id)` (lines 699–702 of `rs/execution_environment/src/execution_environment.rs`). `retrieve_context` searches all context maps — `setup_initial_dkg_contexts`, `sign_with_threshold_contexts`, `canister_http_request_contexts`, `bitcoin_get_successors_contexts`, `bitcoin_send_transaction_internal_contexts` — and removes the matching entry.
7. The consumed context's originating request is used to route an `EmptyBlob` (or reject) response back to the requesting canister via `push_subnet_output_response` (lines 771–781).

If `target_id` matches no context, `retrieve_context` returns `None` and execution returns `(state, Finished)` — silently dropped, no panic. If it matches a real context, that context is permanently consumed and the canister receives a fabricated response.

## Impact Explanation

A Byzantine block proposer can target any pending subnet call context by its observable `CallbackId`. Targeting a `SignWithThreshold` context causes the requesting canister to receive an `EmptyBlob` instead of a threshold signature, corrupting its execution state. Targeting a `BitcoinSendTransactionInternal` context causes the canister to receive a premature success or reject, with the real adapter response never deliverable (context already consumed). This constitutes execution integrity loss and application-level DoS constrained to specific canisters and pending operations — matching the **High** impact category: *"Memory corruption or execution integrity loss constrained to specific canisters, subnet conditions, or pre-existing properties"* and *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attacker is a **single Byzantine block proposer** — one compromised subnet node, explicitly within the BFT fault-tolerance bound (f < n/3). No threshold corruption, governance majority, or external infrastructure attack is required. The proposer can read `next_callback_id` and all pending context maps directly from the certified replicated state visible to every node. All honest validators accept the block because `validate_payload` performs no state cross-check. The attack is repeatable every round the Byzantine node is selected as proposer.

## Recommendation

1. **In `validate_payload`**: After parsing, retrieve state at `proposal_context.validation_context.certified_height` via `state_manager.get_state_at(...)` and reject any `callback_id` not present in `bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`. Also use the already-computed `delivered_ids` to reject duplicates. Mirror the pattern used in the HTTPS outcalls payload builder (`rs/https_outcalls/consensus/src/payload_builder`), which already performs this check and rejects unknown callback IDs.
2. **In `push_response`**: Add `.get(&callback_id).ok_or(StateError::BitcoinNonMatchingResponse{…})?` guards to the `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject` arms, mirroring the existing guard in the `GetSuccessorsResponse` arm.

## Proof of Concept

```rust
// Construct a ReplicatedState with no pending bitcoin requests.
// Call push_response with a SendTransactionResponse carrying callback_id = 9999.
// Observe: Ok(()) is returned and a ConsensusResponse(9999, EmptyBlob)
// is present in state.consensus_queue — no error, no state lookup.
let response = BitcoinAdapterResponse {
    response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
        SendTransactionResponse {}
    ),
    callback_id: 9999,
};
let result = push_response(&mut state, response);
assert!(result.is_ok());           // passes — should be Err(BitcoinNonMatchingResponse)
assert_eq!(state.consensus_queue.len(), 1); // spurious entry injected
```

A complementary integration test: create a `ReplicatedState` with a `SignWithThreshold` context at `callback_id = K`, inject a `SendTransactionResponse` with `callback_id = K` via `push_response_bitcoin`, execute a round, and assert that the threshold signing context has been consumed and the requesting canister received an `EmptyBlob` response.
