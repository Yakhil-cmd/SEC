Audit Report

## Title
Heap-Only `API_BOUNDARY_NODE_PRINCIPALS` Wiped on Upgrade Creates FullReadâ†’RestrictedRead Downgrade Window â€” (`rs/boundary_node/rate_limits/canister/storage.rs`, `canister.rs`)

## Summary
`API_BOUNDARY_NODE_PRINCIPALS` is stored exclusively in heap memory and is silently reset to an empty `HashSet` on every canister upgrade. The `post_upgrade` hook re-registers a periodic timer via `set_timer_interval`, which does not fire immediately but only after the first full interval elapses. During that window, every API boundary node principal is treated as `RestrictedRead`, causing confidential `rule_raw` and `description` fields to be redacted from responses, and replicated `get_config` calls to be trapped entirely.

## Finding Description
**Root cause 1 â€” heap-only storage:**
`API_BOUNDARY_NODE_PRINCIPALS` is declared as a plain `RefCell<HashSet<Principal>>` with no backing stable memory:
```rust
// storage.rs:149
pub static API_BOUNDARY_NODE_PRINCIPALS: RefCell<HashSet<Principal>> = RefCell::new(HashSet::new());
```
All other security-critical maps (`CONFIGS`, `RULES`, `INCIDENTS`, `AUTHORIZED_PRINCIPAL`) are backed by `StableBTreeMap` with explicit `MemoryId` allocations and survive upgrades. This one does not.

**Root cause 2 â€” timer fires after interval, not immediately:**
`post_upgrade` delegates to `init` (`canister.rs:101-103`), which calls `periodically_poll_api_boundary_nodes`. That function uses `ic_cdk_timers::set_timer_interval(interval, ...)` (`canister.rs:254`), scheduling the first execution only after `interval` seconds. There is no immediate one-shot poll on startup.

**Root cause 3 â€” access level degrades to `RestrictedRead`:**
`is_api_boundary_node_principal` consults the now-empty set and returns `false` (`state.rs:125-128`). `get_access_level` therefore falls through to `AccessLevel::RestrictedRead` (`access_control.rs:46-54`).

**Root cause 4 â€” confidential fields are redacted for `RestrictedRead`:**
In `ConfigGetter::get`, when `is_authorized_viewer` is `false`, the formatter is applied (`getter.rs:130-147`). The formatter sets `is_redacted = true` and nulls `rule_raw`/`description` for every non-disclosed rule (`confidentiality_formatting.rs:17-28`). The same redaction applies in `RuleGetter` and `IncidentGetter` (`getter.rs:234-244`).

**Additional impact â€” replicated-query path is fully blocked:**
For ingress calls to `get_config` (the `REPLICATED_QUERY_METHOD`), `inspect_message` checks `has_full_read_access`. With an empty set this is `false`, so the call is trapped entirely (`canister.rs:48-55`). Boundary nodes therefore receive either a trap (replicated query path) or a fully-redacted response (non-replicated query path) for the entire duration of the upgrade window.

## Impact Explanation
This is a **High** severity finding. The rate-limit canister is explicitly in-scope boundary node infrastructure. During every upgrade, API boundary nodes lose access to confidential (non-disclosed) rate-limit rules for the duration of the polling interval. Without `rule_raw`, a boundary node cannot parse or enforce those rules, meaning any traffic those rules were designed to rate-limit passes through unchecked. This constitutes a significant boundary/API security impact with concrete and repeatable harm to the IC's abuse-mitigation layer. The replicated query path is additionally fully blocked during this window.

## Likelihood Explanation
Every routine canister upgrade triggers the window unconditionally â€” no special attacker capability is required. An adversary need only observe an upgrade event (which is publicly visible on-chain via NNS proposals) and send traffic during the known window. The window length is determined by `registry_polling_period_secs`, a deployment parameter visible in `InitArg`, which could range from tens of seconds to several minutes.

## Recommendation
1. **Immediate fix**: Replace the heap-only `HashSet` with a stable-memory-backed structure (e.g., a `StableBTreeMap<StorablePrincipal, ()>` with a new `MemoryId`) so the set survives upgrades, consistent with how `AUTHORIZED_PRINCIPAL` is stored in `storage.rs`.
2. **Defense-in-depth**: Add a one-shot `set_timer(Duration::ZERO, ...)` in `post_upgrade`/`init` immediately before `set_timer_interval`, to trigger an immediate registry poll on startup. This reduces any residual window to a single async round-trip rather than a full polling interval.

## Proof of Concept
Integration test outline using PocketIC or the IC replica:
1. Install the canister with a known API boundary node principal `P` and at least one non-disclosed rule (`disclosed_at = None`).
2. Call `get_config` from `P`; assert `is_redacted = false` and `rule_raw` is present.
3. Upgrade the canister (before the timer fires).
4. Immediately call `get_config` from `P` again (before `registry_polling_period_secs` elapses).
5. Assert `is_redacted = true` and `rule_raw = None` â€” confirming the downgrade window.
6. Wait for the polling interval to elapse and the timer to fire.
7. Call `get_config` from `P` again; assert `is_redacted = false` and `rule_raw` is restored â€” confirming the window closes after the first timer tick.
