### Title
Inconsistent Silo Whitelist Enforcement Between `receive_base_tokens` and `receive_erc20_tokens` Allows ETH to Be Minted to Non-Whitelisted Addresses, Permanently Freezing Funds - (`engine/src/engine.rs`)

---

### Summary

In Silo mode, the Aurora Engine enforces an `Address` whitelist to restrict which EVM addresses may receive tokens. `receive_erc20_tokens` correctly checks this whitelist and redirects tokens to a configured fallback address when the recipient is not whitelisted. However, the parallel code path `receive_base_tokens` — triggered when bridged ETH (base tokens) arrive via `ft_on_transfer` from the ETH connector — performs **no whitelist check at all**. Any caller can bridge ETH to any arbitrary EVM address, bypassing the Silo access control entirely. Because the recipient address is not on the `Address` whitelist, it cannot submit transactions to move the funds, causing them to be permanently frozen.

---

### Finding Description

The Silo mode defines four whitelists. The `Address` whitelist (`WhitelistKind::Address`) governs which EVM addresses may submit transactions and receive ERC-20 tokens.

`is_allow_receive_erc20_tokens` checks only the `Address` whitelist: [1](#0-0) 

`receive_erc20_tokens` in `engine/src/engine.rs` correctly applies this check and redirects to the fallback address when the recipient is not whitelisted: [2](#0-1) 

However, `receive_base_tokens` — the parallel path for bridging ETH — performs **no whitelist check**. It directly mints ETH to whatever address is specified in the message: [3](#0-2) 

Both paths are dispatched from the same `ft_on_transfer` entrypoint. The branch is determined solely by whether the predecessor is the ETH connector account: [4](#0-3) 

The `assert_access` function, which enforces the `Address` whitelist for transaction submission, is only called during `submit`/`submit_with_args`, not during token receipt: [5](#0-4) 

This means ETH can be minted to any EVM address regardless of whitelist status, but that address cannot subsequently move the funds because `assert_access` will block all its transactions.

---

### Impact Explanation

**High — Permanent freezing of funds.**

When a Silo operator enables the `Address` whitelist to restrict which EVM addresses may participate, any user can still bridge ETH to a non-whitelisted address via the ETH connector's `ft_transfer_call → ft_on_transfer → receive_base_tokens` path. The ETH is minted to the non-whitelisted address. Because `assert_access` blocks all transaction submissions from non-whitelisted addresses, the minted ETH is irrecoverable — there is no fallback redirect mechanism for base tokens, unlike ERC-20 tokens. The funds are permanently frozen at the non-whitelisted address.

---

### Likelihood Explanation

**Medium.** The Silo mode with an enabled `Address` whitelist is a deliberate operator configuration. Any user who can call `ft_transfer_call` on the ETH connector (an unprivileged, publicly reachable NEAR call) can trigger this path. No special privileges are required. The attacker only needs to specify a non-whitelisted EVM address as the recipient in the transfer message.

---

### Recommendation

Apply the same whitelist-and-fallback logic to `receive_base_tokens` that is already applied in `receive_erc20_tokens`. Specifically, after parsing the recipient address, check `silo::is_allow_receive_erc20_tokens` (or an equivalent `is_allow_receive_base_tokens` function) and redirect to the configured fallback address if the recipient is not whitelisted. This mirrors the existing pattern:

```rust
// In receive_base_tokens, after parsing recipient:
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
{
    receipient = fallback_address;
}
```

This ensures consistent whitelist enforcement across all token-receipt paths in Silo mode.

---

### Proof of Concept

1. Operator deploys Aurora Engine in Silo mode: enables `Address` whitelist, sets a fallback address, and does **not** add victim address `V` to the whitelist.
2. Attacker calls `ft_transfer_call` on the ETH connector NEP-141 contract, specifying Aurora as `receiver_id` and `V`'s hex address as `msg`.
3. The ETH connector calls `ft_on_transfer` on Aurora. Since `predecessor_account_id == get_connector_account_id`, the engine dispatches to `receive_base_tokens`.
4. `receive_base_tokens` parses `V` from the message and calls `set_balance` to mint ETH to `V` — **no whitelist check occurs**.
5. `V` now holds ETH on Aurora but cannot call `submit` or any EVM transaction because `assert_access` returns `EngineErrorKind::NotAllowed` for non-whitelisted addresses.
6. The ETH is permanently frozen at `V` with no recovery path (unlike ERC-20 tokens, there is no fallback redirect for base tokens). [6](#0-5) [7](#0-6)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
