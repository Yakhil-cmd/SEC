Audit Report

## Title
Confidential Rule Lifecycle Metadata (`added_in_version` / `removed_in_version`) Leaked to Unprivileged Callers ‚Äî (`rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`)

## Summary
`RuleConfidentialityFormatter::format` only nulls `rule_raw` and `description` for non-disclosed rules; it never redacts `added_in_version` or `removed_in_version`. Because `get_rule_by_id` and `get_rules_by_incident_id` are `#[query]` methods, `inspect_message` never fires for them, so any anonymous caller can retrieve version lifecycle metadata for confidential rules. Combined with the fact that `incident_id` is already exposed via `get_config`, this allows any caller to determine exactly when each confidential security rule was introduced and retired.

## Finding Description
**Root cause:** `RuleConfidentialityFormatter::format` (confidentiality_formatting.rs L34‚Äì42) only sets `rule.description = None` and `rule.rule_raw = None` when `disclosed_at.is_none()`. The fields `added_in_version` and `removed_in_version` are never touched.

**Code path:**
1. `IncidentGetter::get` (getter.rs L182‚Äì189) constructs `OutputRuleMetadata` with all fields populated, including `added_in_version: stored_rule.added_in_version` and `removed_in_version: stored_rule.removed_in_version`.
2. For unauthorized viewers, it calls `self.formatter.format(output_rule)` (getter.rs L196), which returns the struct with version fields intact.
3. The same flaw exists in `RuleGetter::get` (getter.rs L224‚Äì243).
4. Both `get_rules_by_incident_id` (canister.rs L136) and `get_rule_by_id` (canister.rs L123) are `#[query]` methods. The `inspect_message` hook (canister.rs L34‚Äì68) only handles `UPDATE_METHODS` and `REPLICATED_QUERY_METHOD`; query calls bypass it entirely on the IC.
5. The prerequisite `incident_id` is freely obtainable: `ConfigConfidentialityFormatter::format` (confidentiality_formatting.rs L17‚Äì28) does not redact `incident_id` from `OutputRule`, so any caller can call `get_config` and collect all `incident_id` values, including those for non-disclosed rules.

**Why existing checks fail:** The `inspect_message` guard is irrelevant for query methods. The formatter is the only confidentiality enforcement mechanism for query responses, and it has an incomplete redaction set.

## Impact Explanation
This is a significant boundary/API infrastructure security impact with concrete protocol harm. The rate-limits canister's confidentiality model exists specifically to prevent adversaries from learning about active security incidents before public disclosure. Leaking `added_in_version` and `removed_in_version` reveals the exact config versions at which each confidential rule was introduced and retired. An attacker can correlate these version numbers with the publicly observable config history (via `get_config`) to determine the precise timing, duration, and scope of confidential security incidents ‚Äî the exact information the confidentiality model is designed to protect. This maps to the High impact class: "Significant boundary/API infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
Exploitation requires no credentials, no privileged role, and no network-level attack. Any IC user can issue query calls to the canister. The `incident_id` prerequisite is trivially satisfied via a single `get_config` query. The behavior is confirmed by the existing test suite and requires no fuzzing or special tooling to reproduce.

## Recommendation
Extend `RuleConfidentialityFormatter::format` to also redact `added_in_version` and `removed_in_version` when `disclosed_at.is_none()`:

```rust
if rule.disclosed_at.is_none() {
    rule.description = None;
    rule.rule_raw = None;
    rule.added_in_version = 0;      // redact
    rule.removed_in_version = None; // redact
}
```

Apply the same fix to `ConfigConfidentialityFormatter::format` if `incident_id` should also be hidden for non-disclosed rules in `get_config` responses. Update the unit tests in getter.rs to assert that `added_in_version` and `removed_in_version` are redacted (zeroed/None) for `RestrictedRead` callers.

## Proof of Concept
The existing unit test `test_get_incident_success` in getter.rs (L449‚Äì521) already proves the leak: with `AccessLevel::RestrictedRead`, the assertion at L500‚Äì508 confirms `added_in_version: 1` and `removed_in_version: Some(3)` are returned for a rule with `disclosed_at: None`. No additional test infrastructure is needed ‚Äî the test suite itself demonstrates the vulnerability.
