### Title
`factory_update` Stores Arbitrary Router Bytecode Without WASM Validity Check, Enabling Permanent Freezing of User NEAR in XCC Router Accounts - (File: engine/src/contract_methods/xcc.rs)

---

### Summary

`factory_update` accepts and stores arbitrary bytes as the XCC router WASM bytecode without any validation that the bytes constitute valid WASM or implement the required router interface. This is the direct analog of `BeaconDS.setImplementation` not checking for contract existence: the version counter is incremented immediately upon storage, and all subsequent XCC interactions deploy the stored bytes to user router sub-accounts. If the stored bytes are wrong-but-valid WASM (e.g., a WASM module that has `initialize` but lacks `execute`/`deploy_upgrade`), the deployment succeeds, the version is committed, the user's wNEAR is burned and NEAR is transferred to the router, and then the user's actual cross-contract call fails — leaving NEAR permanently frozen in a router account with no access keys and no recovery path.

---

### Finding Description

`factory_update` in `engine/src/contract_methods/xcc.rs` reads raw input bytes and passes them directly to `update_router_code` with no validation:

```rust
let bytes = io.read_input().to_vec();
let router_bytecode = xcc::RouterCode::new(bytes);
xcc::update_router_code(&mut io, &router_bytecode);
```

`update_router_code` in `engine/src/xcc.rs` immediately writes the bytes to storage and increments the global version counter:

```rust
io.write_storage(&key, &code.0);
let current_version = get_latest_code_version(io);
let latest_version = current_version.increment();
// ...
set_latest_code_version(io, latest_version);
```

No check is performed to verify:
- The bytes are non-empty
- The bytes are valid WASM (magic bytes `\0asm`)
- The WASM exports the required router interface (`initialize`, `execute`, `schedule`, `deploy_upgrade`, `send_refund`)

Once the version is incremented, `AddressVersionStatus::new` in `engine/src/xcc.rs` will classify every existing router whose recorded version is less than `latest_code_version` as `DeployNeeded { create_needed: false }`, triggering an upgrade attempt on the next XCC call. For new users, `target_code_version == None` triggers `DeployNeeded { create_needed: true }`.

The critical path for fund loss is the upgrade path (`create_needed: false`). In `handle_precompile_promise`, the upgrade is issued as a `FunctionCall` to `deploy_upgrade` on the existing router account. The router's `deploy_upgrade` method deploys the new code and calls `initialize`. If the wrong WASM exports `initialize` (causing the batch to succeed), `factory_update_address_version` records the new version. The promise chain then proceeds to `withdraw_wnear_to_router`, which burns the user's wNEAR ERC-20 balance and transfers real NEAR to the router sub-account. The final user call then executes on the router — which now has the wrong code — and fails. The NEAR transferred to the router is permanently inaccessible: router sub-accounts have no access keys and are controlled solely through cross-contract calls from the engine, but the wrong code has no `execute` or `send_refund` method to recover the funds.

---

### Impact Explanation

**High — Temporary/Permanent Freezing of Funds.**

User NEAR transferred to a router sub-account during a failed XCC call is permanently frozen if the router was upgraded to wrong-but-valid WASM that lacks the recovery interface. The router sub-account has no access keys; recovery requires the router to have `deploy_upgrade`, which the wrong code may not export. Even if the admin subsequently calls `factory_update` with correct bytes, the already-upgraded router (now at the new version) will be classified as `UpToDate` and will not be re-deployed, leaving the NEAR permanently locked.

---

### Likelihood Explanation

**Low.** The function is owner-only (`require_owner_only`), so exploitation requires an honest operational mistake by the Aurora Engine owner — for example, accidentally uploading the wrong WASM artifact. The specific impact (fund freeze rather than mere service disruption) requires the wrong WASM to export `initialize` but not the expected router methods. This is a realistic operational error (e.g., uploading a different NEAR contract's WASM), not a theoretical one.

---

### Recommendation

- **Short term:** Add a WASM magic-bytes check in `factory_update` before accepting new bytecode. Valid WASM modules begin with `\0asm` (bytes `0x00 0x61 0x73 0x6D`). Reject any input that does not start with these bytes.
- **Long term:** Add an interface validation step: after storing the bytes, verify (via a dry-run or export inspection) that the WASM exports the required entry points (`initialize`, `execute`, `schedule`, `deploy_upgrade`, `send_refund`) before incrementing the version counter.

---

### Proof of Concept

1. Admin accidentally calls `factory_update` with bytes from a different valid WASM contract that exports `initialize` but not `execute`, `schedule`, or `deploy_upgrade`.
2. `update_router_code` stores the bytes and increments the global version from `N` to `N+1`.
3. An existing user (whose router is at version `N`) calls the XCC precompile.
4. `handle_precompile_promise` detects `version N < latest N+1`, classifies as `DeployNeeded { create_needed: false }`.
5. Engine issues a batch: `FunctionCall { name: "deploy_upgrade", ... }` on the user's existing router.
6. The router's `deploy_upgrade` deploys the wrong WASM and calls `initialize` — both succeed.
7. `factory_update_address_version` callback sees `Some(true)`, records the user's router at version `N+1`.
8. `withdraw_wnear_to_router` callback executes: burns user's wNEAR ERC-20 balance and transfers NEAR to the router sub-account.
9. User's actual cross-contract call executes on the router — fails (no `execute` method).
10. The NEAR transferred in step 8 is now permanently frozen in the router sub-account, which has no access keys and no recovery method in the wrong code.

/// Set new router bytecode, and update increment the version by 1.
pub fn update_router_code<I: IO>(io: &mut I, code: &RouterCode) {
    let key = storage::bytes_to_key(KeyPrefix::CrossContractCall, CODE_KEY);
    io.write_storage(&key, &code.0);

    let current_version = get_latest_code_version(io);
    let latest_version = current_version.increment();

    // Store the latest version, this will be the first one where the
    // router contract is upgradable.
    let key = storage::bytes_to_key(KeyPrefix::CrossContractCall, FIRST_UPGRADABLE);
    if io.read_storage(&key).is_none() {
        let version_bytes = latest_version.0.to_le_bytes();
        io.write_storage(&key, &version_bytes);
    }

    set_latest_code_version(io, latest_version);
}

impl AddressVersionStatus {
    fn new<I: IO>(
        io: &I,
        latest_code_version: CodeVersion,
        target_code_version: Option<CodeVersion>,
    ) -> Self {
        let first_upgradable_version =
            get_first_upgradable_version(io).unwrap_or(CodeVersion::ZERO);
        match target_code_version {
            None => Self::DeployNeeded {
                create_needed: true,
            },
            Some(version) if version < first_upgradable_version => {
                // It is impossible to upgrade the initial XCC routers because
                // they lack the upgrade method.
                Self::UpToDate
            }
            Some(version) if version < latest_code_version => Self::DeployNeeded {
                create_needed: false,
            },
            Some(_version) => Self::UpToDate,
        }
    }
