### Title
Unprivileged Caller Can Permanently Break `mirror_erc20_token` by Pre-Registering a NEP-141 Token via `deploy_erc20_token` - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` has no access control beyond `require_running`, allowing any NEAR account to register any NEP-141 token to a freshly-deployed ERC-20 address. The owner-only `mirror_erc20_token` operation, which must deploy at a specific pre-determined EVM address and then register the same NEP-141, will permanently fail with `ERR_NEP141_TOKEN_ALREADY_REGISTERED` if an attacker pre-registers the NEP-141 first. There is no `unregister_token` function in the contract, so the incorrect mapping is irrecoverable without a full contract upgrade.

---

### Finding Description

`deploy_erc20_token` is the public entry point for bridging a NEP-141 token into Aurora as an ERC-20. Its implementation only checks `require_running` — no owner, whitelist, or role check is performed: [1](#0-0) 

After deploying the ERC-20 bytecode via the EVM, it calls `engine.register_token(address, nep141)` to record the NEP-141 → ERC-20 address mapping: [2](#0-1) 

`mirror_erc20_token` is the owner-only silo operation that mirrors an ERC-20 from the main Aurora contract into a silo at the **exact same EVM address** as on the main contract. After deploying at that specific address, it also calls `register_token`: [3](#0-2) 

`register_token` returns `RegisterTokenError::TokenAlreadyRegistered` (error constant `ERR_NEP141_TOKEN_ALREADY_REGISTERED`) if the NEP-141 is already mapped to any ERC-20 address: [4](#0-3) 

Because `mirror_erc20_token` propagates this error via `?`, the entire callback reverts. No `unregister_token` function exists anywhere in the contract, so the incorrect mapping is permanent.

The `mirror_erc20_token` entry point confirms the owner-only guard: [5](#0-4) 

---

### Impact Explanation

Once an attacker calls `deploy_erc20_token(nep141_id)`, the NEP-141 is permanently registered to a randomly-deployed ERC-20 address (determined by the EVM CREATE nonce at call time). The silo owner can never successfully execute `mirror_erc20_token` for that NEP-141 token. Any user who subsequently bridges the NEP-141 token into the silo via `ft_on_transfer` will receive tokens minted at the attacker-chosen ERC-20 address rather than the intended mirrored address. The silo's ERC-20 address for that token diverges permanently from the main Aurora contract's address, breaking any cross-contract logic that assumes address parity. Recovery requires a full contract upgrade with a state migration to overwrite the mapping — a significant operational burden.

**Impact category**: Permanent freezing of the mirroring functionality for the targeted NEP-141 token, with secondary risk of fund misrouting for users bridging that token into the silo.

---

### Likelihood Explanation

`deploy_erc20_token` is a public, permissionless NEAR function call. Any NEAR account can call it with an arbitrary NEP-141 `AccountId`. An attacker only needs to know which NEP-141 token the silo operator intends to mirror (observable from on-chain announcements, governance proposals, or by monitoring the main Aurora contract's token registry). The attack requires a single transaction with no special privileges, no funds, and no prior setup. Front-running is straightforward on NEAR because transaction ordering within a block is observable.

---

### Recommendation

Add an owner-only or admin-role access control check to `deploy_erc20_token`, consistent with the guard already applied to `mirror_erc20_token`:

```rust
require_owner_only(&state::get_state(&io)?, &env.predecessor_account_id())?;
```

Alternatively, introduce a separate permissioned entry point for silo mirroring that bypasses the public `deploy_erc20_token` path, and restrict the public path to non-mirrored tokens only (e.g., by checking that the NEP-141 is not already registered on the main contract before allowing registration in the silo).

---

### Proof of Concept

Attack sequence:

1. Silo operator announces intent to mirror NEP-141 token `token.near` into the silo.
2. Attacker calls `deploy_erc20_token` with `nep141 = "token.near"` on the silo contract. This deploys a new ERC-20 at address `A` (EVM CREATE nonce-derived) and writes the mapping `token.near → A` into storage.
3. Silo operator calls `mirror_erc20_token({ contract_id: "aurora", nep141: "token.near" })`.
4. The callback `mirror_erc20_token_callback` receives the correct mirrored address `B` from the main contract, deploys ERC-20 bytecode at `B` successfully, then calls `register_token(B, "token.near")`.
5. `register_token` finds `token.near` already mapped to `A` and returns `ERR_NEP141_TOKEN_ALREADY_REGISTERED`.
6. The callback propagates the error; the entire operation reverts.
7. The silo permanently holds the mapping `token.near → A` (wrong address). No recovery path exists without a contract upgrade. [6](#0-5) [7](#0-6)

pub fn mirror_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    args: MirrorErc20TokenArgs,
    erc20_address: Address,
    erc20_metadata: Erc20Metadata,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, Some(erc20_metadata));
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, Some(erc20_address), handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    assert_eq!(address, erc20_address);

    sdk::log!(
        "ERC-20 on: {} at address: {} has been mirrored",
        args.contract_id.as_ref(),
        address.encode()
    );
    engine
        .register_token(address, args.nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
