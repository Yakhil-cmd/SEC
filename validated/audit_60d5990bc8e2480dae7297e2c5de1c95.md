### Title
Missing Zero-Divisor Check in `i256_mod` Causes Panic on EVM SMOD Opcode with Zero Divisor — (`File: evm_interpreter/src/i256.rs`)

---

### Summary

`i256_mod` in `evm_interpreter/src/i256.rs` does not guard against a zero divisor before calling `div_rem`, which panics. The analogous function `i256_div` correctly handles this case. Any unprivileged user can deploy a contract that executes the EVM `SMOD` opcode with a zero divisor, triggering a panic that halts block processing.

---

### Finding Description

`i256_div` correctly checks for a zero divisor and returns `0` per EVM semantics: [1](#0-0) 

```rust
pub fn i256_div(dividend: &mut U256, divisor_or_quotient: &mut U256) {
    let divisor_sign = i256_sign::<true>(divisor_or_quotient);
    if divisor_sign == Sign::Zero {
        *divisor_or_quotient = U256::ZERO;
        return;
    }
```

`i256_mod`, however, calls `i256_sign::<true>` on the divisor but **discards the result** with `let _ = ...`, then unconditionally calls `div_rem`: [2](#0-1) 

```rust
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }

    let _ = i256_sign::<true>(divisor_or_remainder);  // result discarded!

    // this is unsigned division of moduli
    let (_q, r) = dividend.div_rem(*divisor_or_remainder);  // panics if divisor == 0
```

When `divisor_or_remainder` is `0`:
1. `i256_sign::<true>(divisor_or_remainder)` returns `Sign::Zero` but the result is thrown away.
2. `dividend.div_rem(0)` is called — `ruint::U256::div_rem` **panics** on a zero divisor, as confirmed by the codebase's own tests: [3](#0-2) 

The EVM specification mandates `SMOD(x, 0) = 0` for all `x`. The ZKsync OS implementation instead panics, diverging from the spec.

---

### Impact Explanation

A panic in the RISC-V proving path aborts proof generation entirely — the block cannot be proven and the state transition fails. In the forward (sequencer) path, the panic crashes the sequencer process. Either outcome is a **complete denial of service** for block processing: no further transactions in the block can be processed or proven until the block is discarded.

This is a **state-transition bug** with a **valid-execution unprovability** dimension: a transaction that is valid per EVM semantics (SMOD with zero divisor returns 0) cannot be proven by ZKsync OS.

---

### Likelihood Explanation

The trigger is trivially reachable. Any unprivileged user can deploy a one-instruction contract:

```
PUSH1 0x00   // divisor = 0
PUSH1 0x01   // dividend = 1
SMOD         // triggers i256_mod(1, 0) → panic
```

No special permissions, governance access, or oracle manipulation is required. The attacker only needs to submit a standard EVM transaction.

---

### Recommendation

Add the missing zero-divisor guard to `i256_mod`, mirroring the pattern already used in `i256_div`:

```rust
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    let divisor_sign = i256_sign::<true>(divisor_or_remainder);
    if divisor_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }

    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }

    let (_q, r) = dividend.div_rem(*divisor_or_remainder);
    // ... rest unchanged
}
```

---

### Proof of Concept

Deploy the following bytecode and call it from any account:

```
// SMOD(1, 0) — EVM spec: result = 0; ZKsync OS: panic
PUSH1 0x00   // 0x60 0x00
PUSH1 0x01   // 0x60 0x01
SMOD         // 0x07
STOP         // 0x00
```

Submitting a transaction to this contract causes `i256_mod` to be invoked with `divisor_or_remainder = 0` and a non-zero dividend. `div_rem` panics, aborting block execution and proof generation.
