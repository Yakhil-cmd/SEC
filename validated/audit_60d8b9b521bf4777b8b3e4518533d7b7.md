### Title
SNS Treasury Manager `DepositRequest` Contains No Slippage Protection, Exposing SNS Treasury Funds to Front-Running on DEX Deposits - (File: rs/sns/treasury_manager/treasury_manager.did, rs/sns/governance/src/extensions.rs)

### Summary

The SNS Treasury Manager framework defines a `DepositRequest` type with no minimum LP token output or price-ratio guarantee field. The SNS Governance canister's `execute_treasury_manager_deposit` function approves and calls `deposit` on a Treasury Manager canister without enforcing any minimum output. The `validate_deposit_operation_impl` function only checks that the requested amounts do not exceed 50% of the current treasury balance, but performs no minimum-output validation. This is structurally identical to the reported `_harvest` vulnerability: a token conversion (here, SNS tokens + ICP → DEX LP tokens) is executed with no slippage floor, making it exploitable by any actor who can observe the pending governance execution and manipulate the DEX pool price before the deposit settles.

### Finding Description

The `DepositRequest` type in the Treasury Manager API specification contains only `allowances` (the input amounts to deposit):

```candid
type DepositRequest = record {
  allowances : vec Allowance;
};
```

There is no `minimum_lp_tokens_out`, `minimum_price_ratio`, or any equivalent slippage-protection field. [1](#0-0) 

The IC codebase itself acknowledges this as a known security risk in the same file:

```
// Known Security Risks:
// ====================
// Some liquidity pools do not implement slippage protection
// for deposits. In other words, the price ratio at the time
// of execution may differ from the ratio at the time the proposal was
// approved.
``` [2](#0-1) 

The SNS Governance canister's `execute_treasury_manager_deposit` function approves the Treasury Manager to spend SNS tokens and ICP via ICRC-2, then calls `deposit` with no minimum output check:

```rust
// 1. Transfer funds from treasury to treasury manager
governance
    .approve_treasury_manager(extension_canister_id, treasury_allocation_sns_e8s, treasury_allocation_icp_e8s)
    .await?;

// 2. Call deposit on treasury manager
let balances = governance
    .env
    .call_canister(extension_canister_id, "deposit", arg_blob)
    ...
``` [3](#0-2) 

The proposal-time validation (`validate_deposit_operation_impl`) only enforces a 50%-of-balance cap on the input amounts; it performs no minimum-output validation whatsoever: [4](#0-3) 

The `approve_treasury_manager` function grants an ICRC-2 allowance with a 1-hour expiry, giving a wide window for exploitation: [5](#0-4) 

The governance proposal rendering even warns voters about this risk but provides no enforcement mechanism: [6](#0-5) 

### Impact Explanation

When an SNS DAO passes a `TreasuryManagerDeposit` proposal, the execution:
1. Grants the Treasury Manager an ICRC-2 allowance over SNS tokens and ICP from the treasury.
2. Calls `deposit` on the Treasury Manager, which in turn deposits into a DEX liquidity pool.

Because neither the `DepositRequest` type nor the governance execution enforces a minimum LP token output, the Treasury Manager canister (or the DEX it calls) can receive far fewer LP tokens than the deposited assets are worth. SNS treasury funds (both SNS tokens and ICP) are permanently lost to the difference. The IC codebase provides no on-chain mechanism to bound this loss.

**Vulnerability type**: Cycles/resource accounting bug (treasury asset conservation bug); analogous to a ledger conservation bug in the chain-fusion/SNS context.

### Likelihood Explanation

- SNS governance proposals are fully public. Any observer can read the deposit amounts and the target Treasury Manager canister before execution.
- IC governance proposal execution timing is predictable (voting period end + processing). A malicious canister registered on the same subnet as the DEX can act in the round immediately before the deposit executes.
- The 1-hour ICRC-2 allowance window further extends the exposure period.
- No privileged access is required to observe the proposal or to interact with the DEX pool.
- The IC codebase itself flags this as a "Known Security Risk," confirming the attack surface is real and acknowledged.

### Recommendation

1. **Add a `minimum_lp_tokens_out` (or equivalent) field to `DepositRequest`** in `rs/sns/treasury_manager/treasury_manager.did`. Treasury Manager implementations must enforce this bound before completing the DEX deposit.
2. **Enforce the minimum output in `execute_treasury_manager_deposit`** (`rs/sns/governance/src/extensions.rs`): after calling `deposit`, parse the returned `Balances` and verify that the `external_custodian` balance meets the minimum specified in the proposal.
3. **Include the minimum output in `validate_deposit_operation_impl`**: require proposers to specify a minimum LP token output as part of the `ExtensionOperationArg`, and validate it against current DEX pool state at proposal submission time.
4. Consider adding a **deadline** field to `DepositRequest` so that stale allowances cannot be exploited if execution is delayed.

### Proof of Concept

1. An SNS DAO submits and passes a `TreasuryManagerDeposit` proposal specifying `treasury_allocation_sns_e8s = X` and `treasury_allocation_icp_e8s = Y`.
2. A malicious canister observes the proposal (public) and the predicted execution round.
3. Just before execution, the malicious canister deposits a large amount of one asset into the target DEX pool, skewing the price ratio.
4. The SNS Governance canister calls `approve_treasury_manager` (granting ICRC-2 allowances) and then `deposit`. The Treasury Manager deposits X SNS tokens and Y ICP at the manipulated price, receiving far fewer LP tokens than fair value.
5. The malicious canister withdraws its position, capturing the difference. The SNS treasury has permanently lost value with no on-chain recourse, since `execute_treasury_manager_deposit` performs no minimum-output check on the returned `Balances`.

```rust
async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

    async fn approve_treasury_manager(
        &self,
        treasury_manager_canister_id: CanisterId,
        sns_amount_e8s: u64,
        icp_amount_e8s: u64,
    ) -> Result<(), GovernanceError> {
        let to = Account {
            owner: treasury_manager_canister_id.get().0,
            subaccount: None,
        };

        let expiry_time_sec = self.env.now().saturating_add(ONE_HOUR_SECONDS);
        let expiry_time_nsec = expiry_time_sec.saturating_mul(NANO_SECONDS_PER_SECOND);

        // If expected_allowance is None, the ledger *blindly* overwrites any existing
        // allowance (even if non-zero). Therefore, there is no risk of double spending.

        self.ledger
            .icrc2_approve(
                to,
                sns_amount_e8s,
                Some(expiry_time_nsec),
                self.transaction_fee_e8s_or_panic(),
                self.sns_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making SNS Token treasury transfer: {e}"),
                )
            })?;

        self.nns_ledger
            .icrc2_approve(
                to,
                icp_amount_e8s,
                Some(expiry_time_nsec),
                icp_ledger::DEFAULT_TRANSFER_FEE.get_e8s(),
                self.icp_treasury_subaccount(),
                None,
            )
            .await
            .map(|_| ())
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Error making ICP Token treasury transfer: {e}"),
                )
            })?;

        Ok(())
    }
