Audit Report

## Title
SNS Governance `distribute_rewards` Computes Inflated Reward Purse When `end_timestamp_seconds` Is `None` - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance `distribute_rewards` and `should_distribute_rewards` functions call `.unwrap_or_default()` on the optional `end_timestamp_seconds` field of `latest_reward_event`. When this field is `None` — as it would be for any SNS whose persisted `RewardEvent` was written before the field was introduced — the reward period start silently becomes Unix epoch (0), causing `new_rounds_count` to be computed as approximately `now / round_duration_seconds` (~2,800 for a weekly round). The resulting inflated purse is distributed in full to the first voters after the upgrade, constituting illegal maturity minting far exceeding the token supply.

## Finding Description

`end_timestamp_seconds` is declared `optional uint64` in the protobuf schema and maps to `Option<u64>` in Rust: [1](#0-0) [2](#0-1) 

`should_distribute_rewards` uses `.unwrap_or_default()` on this field: [3](#0-2) 

`distribute_rewards` does the same, then computes `new_rounds_count` from the result: [4](#0-3) 

When `end_timestamp_seconds` is `None`, both calls return `0`. With `now ≈ 1.7 × 10⁹` and `round_duration_seconds = 604800` (one week), `new_rounds_count ≈ 2,811`. The reward purse loop then accumulates ~2,811 rounds of `supply × reward_rate × round_duration`: [5](#0-4) 

The `Governance::new()` constructor only inserts a corrective dummy event when `latest_reward_event` is entirely absent (`is_none()`); it does **not** repair a present event whose `end_timestamp_seconds` is `None`: [6](#0-5) 

`canister_post_upgrade` calls `canister_init_` → `Governance::new()` with no additional migration for this field: [7](#0-6) 

No migration code exists anywhere in the codebase to repair a present-but-incomplete `RewardEvent`. If the inflated purse fits in `u64`, it is distributed in full to voters in the first settled proposal after the upgrade. If it overflows `u64`, the function returns early without updating `latest_reward_event`: [8](#0-7) 

This permanently stalls reward distribution, since `end_timestamp_seconds` remains `None` and `should_distribute_rewards` continues returning `true` on every heartbeat.

## Impact Explanation

For a 1 billion-token supply at a 10% annual rate with weekly rounds, the inflated purse is approximately 5.4 × 10¹⁷ e8s — roughly five times the entire token supply — distributed as maturity to a single cohort of voters. This value is below `u64::MAX` (~1.84 × 10¹⁹), so it does not overflow and is distributed in full. This constitutes illegal minting of maturity from thin air, a direct ledger-conservation violation. This matches the Critical impact class: **Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant in-scope ledger assets**.

## Likelihood Explanation

The `end_timestamp_seconds` field was added to `RewardEvent` after the initial SNS deployment (the proto comment at field 5 explicitly states it should be used *instead of* `round`, implying `round` predates it). Any SNS instance whose stable-memory snapshot predates this field will have `end_timestamp_seconds = None`. The vulnerability fires automatically on the very first `run_periodic_tasks` heartbeat after upgrade — no privileged action is required. An observer monitoring upgrade proposals can time their neuron's vote to be the sole beneficiary of the inflated purse.

## Recommendation

In `Governance::new()`, add a migration guard that repairs a present-but-incomplete `RewardEvent`:

```rust
if let Some(ref mut event) = proto.latest_reward_event {
    if event.end_timestamp_seconds.is_none() {
        event.end_timestamp_seconds = Some(proto.genesis_timestamp_seconds);
    }
}
```

Alternatively, replace the bare `.unwrap_or_default()` calls in both `distribute_rewards` and `should_distribute_rewards` with a fallback to `genesis_timestamp_seconds`:

```rust
let reward_start_timestamp_seconds = self
    .latest_reward_event()
    .end_timestamp_seconds
    .unwrap_or(self.proto.genesis_timestamp_seconds);
```

## Proof of Concept

1. Deploy an SNS while `RewardEvent` lacks `end_timestamp_seconds`; the canister runs normally, storing `latest_reward_event` with `end_timestamp_seconds = None`.
2. Upgrade the SNS canister to the current codebase version.
3. `canister_post_upgrade` → `canister_init_` → `Governance::new()`: because `latest_reward_event.is_some()`, the constructor skips the dummy-event insertion; `end_timestamp_seconds` remains `None`.
4. The next `run_periodic_tasks` heartbeat calls `should_distribute_rewards`: `now − 0 = now ≫ round_duration_seconds` → returns `true`.
5. `distribute_rewards` is called: `reward_start_timestamp_seconds = 0`, `new_rounds_count ≈ 2,811`.
6. The reward purse loop accumulates ~5.4 × 10¹⁷ e8s of maturity (for a 1B-token supply), which fits in `u64` and is distributed in full to voters on the first settled proposal.

A deterministic integration test can reproduce this by: (a) constructing a `Governance` proto with a `latest_reward_event` whose `end_timestamp_seconds` is `None`, (b) advancing the mock clock to a realistic `now`, (c) calling `distribute_rewards`, and (d) asserting that neuron maturity is inflated by a factor of ~2,811 relative to a single legitimate round.
