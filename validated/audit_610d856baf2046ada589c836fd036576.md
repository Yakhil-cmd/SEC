### Title
`ChainStateCommitment.hash()` Omits `chain_id`, Enabling Cross-Chain State Commitment Collision - (File: `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/public_input.rs`)

---

### Summary

`ChainStateCommitment`, whose blake2s hash is stored on the settlement layer as the canonical per-chain state anchor, does not include `chain_id` in its hash preimage. This is the direct structural analog of the ParaSpace `Credit` struct missing `MarketplaceAddress`: a committed structure that is supposed to uniquely identify a chain's state lacks the field that binds it to a specific chain. Two ZKsync OS chains with identical state parameters produce identical `ChainStateCommitment` hashes, enabling cross-chain proof replay against a settlement layer contract that validates `state_before` without separately enforcing `chain_id`.

---

### Finding Description

`ChainStateCommitment` is defined and hashed as follows:

```rust
pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
    // ← chain_id is ABSENT
}

pub fn hash(&self) -> [u8; 32] {
    let mut hasher = crypto::blake2s::Blake2s256::new();
    hasher.update(self.state_root.as_u8_ref());
    hasher.update(&self.next_free_slot.to_be_bytes());
    hasher.update(&self.block_number.to_be_bytes());
    hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
    hasher.update(&self.last_block_timestamp.to_be_bytes());
    hasher.finalize()
    // ← chain_id never hashed
}
``` [1](#0-0) 

This hash is used as `state_before` and `state_after` in `BatchPublicInput`, which is the final proof public input committed to the settlement layer:

```rust
let public_input = BatchPublicInput {
    state_before: chain_state_commitment_before.hash().into(),
    state_after: chain_state_commitment_after.hash().into(),
    batch_output: batch_output.hash().into(),
};
``` [2](#0-1) 

`chain_id` is only present in `BatchOutput`, a separate component:

```rust
pub struct BatchOutput {
    pub chain_id: U256,
    ...
}
``` [3](#0-2) 

The documentation explicitly states that `ChainStateCommitment` hashes are saved on the settlement layer and used to verify state continuity:

> *"This commitment(hash of its fields) will be saved on the settlement layer. With proofs, we'll ensure that the values used during block execution correspond to this commitment."* [4](#0-3) 

The `BatchPublicInput.hash()` is:

```
keccak256(state_before_hash || state_after_hash || batch_output_hash)
``` [5](#0-4) 

Because `state_before_hash` and `state_after_hash` are chain-agnostic, two chains with identical state parameters produce identical `ChainStateCommitment` hashes. A ZK proof generated for Chain A is structurally valid against Chain B's stored `state_before` commitment whenever the two hashes collide.

---

### Impact Explanation

**Cross-chain proof replay against the settlement layer.**

Consider two ZKsync OS chains (Chain A, `chain_id=X`; Chain B, `chain_id=Y`) that share the same genesis state (same `state_root`, `block_number=0`, `last_block_timestamp`, etc.). Their initial `ChainStateCommitment` hashes are byte-for-byte identical.

1. An attacker generates a valid ZK proof for Chain A's first batch. The proof's public input is `keccak256(state_before_A || state_after_A || batch_output_A)`, where `batch_output_A.chain_id = X`.
2. The attacker submits this proof to Chain B's settlement contract.
3. Chain B's contract verifies the ZK proof — it is cryptographically valid.
4. Chain B's contract checks `state_before_A == stored_commitment_B` — it matches, because `chain_id` is absent from both.
5. If Chain B's contract does not separately enforce `chain_id` from `batch_output`, it accepts the proof and advances Chain B's state to `state_after_A`.

The result is that Chain B's canonical state is overwritten with Chain A's post-batch state, corrupting all user balances and storage on Chain B. This is a direct funds-loss path for all users of Chain B.

The `batch_output.chain_id` field provides a partial mitigation only if the settlement layer contract explicitly validates it. The ZKsync OS public input design places the chain binding in `batch_output` rather than in the state commitment anchor itself, creating a split-validation requirement that is easy to implement incorrectly on the settlement layer side.

---

### Likelihood Explanation

**Medium.** The collision condition (same `state_root`, `block_number`, `last_block_timestamp`, `next_free_slot`, `last_256_block_hashes_blake`) is trivially satisfied at genesis for any two chains deployed from the same initial state — a common deployment pattern for testnets, staging environments, or forked chains. The attack does not require brute force or cryptographic breaks; it requires only that the settlement layer contract omit the `chain_id` check on `batch_output`, which is a natural implementation mistake given that the state commitment anchor itself carries no chain binding. The ZKsync OS codebase does not enforce this check internally.

---

### Recommendation

Include `chain_id` in `ChainStateCommitment` and its `hash()` function:

```rust
pub struct ChainStateCommitment {
    pub chain_id: u64,          // ← add this
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

pub fn hash(&self) -> [u8; 32] {
    let mut hasher = crypto::blake2s::Blake2s256::new();
    hasher.update(&self.chain_id.to_be_bytes()); // ← add this
    hasher.update(self.state_root.as_u8_ref());
    hasher.update(&self.next_free_slot.to_be_bytes());
    hasher.update(&self.block_number.to_be_bytes());
    hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
    hasher.update(&self.last_block_timestamp.to_be_bytes());
    hasher.finalize()
}
```

This makes the state commitment anchor chain-specific, so the settlement layer contract can safely validate `state_before` without needing a separate `chain_id` check on `batch_output`. It also removes the split-validation requirement that creates the replay surface.

The `chain_id` is already available at the call sites where `ChainStateCommitment` is constructed, sourced from `metadata.chain_id()`: [6](#0-5) [7](#0-6)

### Proof of Concept

1. Deploy two ZKsync OS chains from the same genesis: Chain A (`chain_id=37`) and Chain B (`chain_id=38`), both with `state_root=R`, `block_number=0`, `last_block_timestamp=0`, `next_free_slot=0`, `last_256_block_hashes_blake=H`.
2. Compute `ChainStateCommitment_A.hash()` and `ChainStateCommitment_B.hash()` — they are identical because `chain_id` is not hashed.
3. Execute one block on Chain A, generating a valid ZK proof with public input `PI_A = keccak256(state_before_A || state_after_A || batch_output_A)` where `batch_output_A.chain_id = 37`.
4. Submit `PI_A` and its proof to Chain B's settlement contract.
5. The contract verifies the proof (valid), checks `state_before_A == stored_commitment_B` (matches), and — if it does not separately validate `batch_output.chain_id` — accepts the proof and sets Chain B's stored commitment to `state_after_A`.
6. Chain B's state is now Chain A's post-batch state. All user balances and storage on Chain B are corrupted.

The root cause is entirely within ZKsync OS: `ChainStateCommitment.hash()` at `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/public_input.rs:33–41` omits `chain_id`, producing a chain-agnostic commitment that cannot serve as a unique per-chain anchor. [8](#0-7)
