Audit Report

## Title
SNS `ExecuteGenericNervousSystemFunction` Proposal Permanently Marked Executed Despite Application-Level `Err(...)` Reply from Target Canister - (File: rs/sns/governance/src/canister_control.rs)

## Summary
In `perform_execute_generic_nervous_system_function_call`, any Candid reply from the target canister — including a well-formed `Err(...)` — is unconditionally treated as `Ok(())`, causing `set_proposal_execution_status` to permanently set `executed_timestamp_seconds` on the proposal. The target method's documented return type is `Result<(), String>`, so application-level failures are a normal operational scenario, yet they are silently swallowed and the proposal is irreversibly recorded as successfully executed.

## Finding Description
`perform_execute_generic_nervous_system_function_call` in `rs/sns/governance/src/canister_control.rs` (L302–312) matches `Ok(_reply) => Ok(())` with an explicit TODO acknowledging the missing decode step. The validator path in the same file (L262–274) correctly decodes its reply as `Decode!(&reply, Result<String,String>)` and propagates `Err`; the execution path does not. The `Ok(())` propagates through `perform_execute_generic_nervous_system_function` (governance.rs L2547–2553), then through `perform_action` (L2172–2175), and finally into `set_proposal_execution_status` (L2242), which sets `proposal.executed_timestamp_seconds = self.env.now()` (L1725) and clears `failed_timestamp_seconds` and `failure_reason` (L1729–1730). Once `executed_timestamp_seconds != 0`, the guard at L1744 prevents any subsequent failure from being recorded. The proto documentation for `GenericNervousSystemFunction.target_method_name` (governance.proto L287–288, gen/ic_sns_governance.pb.v1.rs L347–348) explicitly states the required signature is `Result<(), String>`, confirming `Err(...)` replies are a first-class, expected outcome that the governance canister is contractually obligated to handle.

## Impact Explanation
This is a significant SNS governance integrity impact with concrete user and protocol harm. Any SNS DAO using `ExecuteGenericNervousSystemFunction` proposals (the primary extension mechanism for SNS governance) can have proposals permanently and incorrectly recorded as `Executed` when the intended on-chain action was never applied. Governance participants querying proposal status via `get_proposal` receive a false `ProposalDecisionStatus::Executed` signal. The SNS dapp can be left in an inconsistent state (e.g., a parameter update or treasury operation that was never applied is believed to have been applied), and the SNS community has no automated mechanism to detect or recover from this — they must notice the discrepancy externally and burn additional proposal fees and voting rounds to retry. This matches the allowed impact: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm**.

## Likelihood Explanation
Medium. The preconditions are: (1) an SNS has at least one registered `GenericNervousSystemFunction` (standard for SNS DAOs that extend governance), and (2) the target canister's method returns a Candid-encoded `Err(...)` reply — a normal operational outcome documented in the interface contract (e.g., precondition failure, state changed between proposal submission and execution). No privileged access beyond normal SNS token-holder voting power is required. The bug is triggered by the ordinary execution path of any passed `ExecuteGenericNervousSystemFunction` proposal whose target method signals failure via its return value rather than a canister-level reject.

## Recommendation
In `perform_execute_generic_nervous_system_function_call`, replace the `Ok(_reply) => Ok(())` arm with a decode of the reply bytes as `Result<(), String>` (matching the documented contract) and map `Err(msg)` to a `GovernanceError` so that `set_proposal_execution_status` records the proposal as `Failed`. This mirrors the existing pattern in `perform_execute_generic_nervous_system_function_validate_and_render_call` (canister_control.rs L262–274). Optionally, store the raw reply bytes in `ProposalData` for auditability, as the existing TODO suggests.

## Proof of Concept
1. Deploy a target canister whose `execute_fn` method returns `Encode!(&Err::<(), String>("precondition failed".to_string())).unwrap()`.
2. Submit and pass an `AddGenericNervousSystemFunction` proposal registering this canister/method (function ID ≥ 1000).
3. Submit and pass an `ExecuteGenericNervousSystemFunction` proposal targeting that function ID.
4. The governance canister calls the target method, receives the `Err(...)` Candid reply, hits `Ok(_reply) => Ok(())`, and calls `set_proposal_execution_status(proposal_id, Ok(()))`.
5. Query the proposal: `executed_timestamp_seconds > 0`, `failed_timestamp_seconds == 0`, `failure_reason == None`, status = `ProposalDecisionStatus::Executed`.
6. The target canister's intended action was never applied; the proposal is permanently and irreversibly marked as successfully executed with no automated recovery path.
This can be reproduced as a deterministic PocketIC integration test without any mainnet interaction.
