Audit Report

## Title
Unbounded Stable-Memory Scan in `get_neuron_ids_ready_to_finalize` Permanently Stalls NNS Maturity-Disbursement Timer on Instruction-Limit Trap — (`rs/nns/governance/src/maturity_disbursement_index.rs`)

## Summary
`get_neuron_ids_ready_to_finalize` performs an unbounded `range(..=max_key).collect()` over a `StableBTreeMap`, materialising every overdue disbursement entry into a heap `BTreeSet` in a single synchronous call. If the entry count grows large enough to exhaust the IC's 40 B per-message instruction limit, the Wasm execution traps. Because `RecurringAsyncTask` reschedules only after `execute().await` returns normally, a trap permanently removes the `FinalizeMaturityDisbursementsTask` from the timer queue, freezing all NNS maturity disbursements indefinitely.

## Finding Description

**Unbounded scan (root cause)** [1](#0-0) 
`get_neuron_ids_ready_to_finalize` issues `range(..=max_key).collect()` with no upper bound on the number of entries returned. Every `(timestamp, neuron_id)` pair whose timestamp ≤ `now_seconds` is loaded from stable memory and inserted into a heap `BTreeSet` in one synchronous call.

**O(1) alternative exists but is unused in the hot path** [2](#0-1) 
`get_next_entry()` calls `first_key_value()` and returns only the minimum-timestamp entry. It is not used by `next_maturity_disbursement_to_finalize`.

**Hot-path caller forces a full scan to find the first non-locked neuron** [3](#0-2) 
`next_maturity_disbursement_to_finalize` collects the entire ready set and then calls `.find()` on it, even though only the first non-locked entry is needed.

**Entry accumulation** [4](#0-3) 
Up to `MAX_NUM_DISBURSEMENTS = 10` entries per neuron are allowed. With ~500 000 NNS neurons each holding 10 pending disbursements, the index can reach 5 000 000 entries.

**Timer rescheduling is conditional on normal return** [5](#0-4) 
`schedule_with_delay` places `new_task.schedule_with_delay(...)` inside the spawned future, after `self.execute().await`. The module documentation explicitly states: *"if the task panics in one iteration, it will NOT be scheduled again."* A Wasm `InstructionLimitExceeded` trap is not a catchable Rust error; it unwinds the entire spawned future, so the rescheduling line is never reached.

**Instruction limit** [6](#0-5) 
`MAX_INSTRUCTIONS_PER_MESSAGE = 40 B`. Stable-BTreeMap node traversals cost thousands of instructions each; a scan over millions of entries can exhaust this budget.

**Task is scheduled unconditionally at startup** [7](#0-6) 

## Impact Explanation
If the instruction limit is exceeded, `FinalizeMaturityDisbursementsTask` traps and is permanently removed from the IC timer queue. No further invocations occur without a canister upgrade. All pending maturity disbursements across all NNS neurons are frozen: no ICP is minted, no disbursement record is popped, and the governance canister's maturity-disbursement subsystem is effectively DoS'd. This matches the **High** bounty impact: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant NNS … security impact with concrete user or protocol harm."*

## Likelihood Explanation
**Organic growth:** The NNS has ~500 000 neurons. If a meaningful fraction accumulate multiple pending disbursements simultaneously (e.g., after sustained high reward distribution), the index can grow into the millions of entries without any deliberate action.

**Deliberate attack:** An adversary controlling many neurons can call `disburse_maturity` up to 10 times per neuron (enforced by `MAX_NUM_DISBURSEMENTS`). Each disbursement has a 7-day delay, so filling 10 slots requires 70 days of patience. The cost is high but not impossible for a well-funded actor. No flash-loan shortcut exists because maturity must be earned over time.

Likelihood is **low-to-medium**: not trivially exploitable today, but a realistic operational risk as NNS adoption grows.

## Recommendation
Replace the full `collect()` scan with an incremental approach:

1. Use `get_next_entry()` (already implemented) to fetch the minimum-timestamp entry in O(1).
2. If that neuron is locked, iterate forward one entry at a time using `range((Excluded(current_key), Unbounded)).next()` until a non-locked neuron is found or a soft instruction-count budget is reached.
3. Alternatively, add a `get_first_n_ready(n: usize, now: u64)` method that returns at most `n` entries, and call it with a small constant (e.g., 100) so the per-invocation scan cost is bounded — consistent with how `unstake_maturity_of_dissolved_neurons` already applies a `max_num_neurons` bound. [8](#0-7) 

## Proof of Concept
A deterministic PocketIC integration test can demonstrate the stall:

1. Create N neurons (e.g., N = 50 000) each with 10 pending maturity disbursements whose `finalize_disbursement_timestamp_seconds` is set to `now - 1` (all overdue).
2. Advance the PocketIC clock past the finalization timestamp.
3. Trigger the `FinalizeMaturityDisbursementsTask` timer.
4. Observe that `get_neuron_ids_ready_to_finalize` is called with 500 000 entries in the index.
5. Assert that the instruction counter exceeds `MAX_INSTRUCTIONS_PER_MESSAGE` (40 B) during the `collect()` call, causing a trap.
6. Confirm that no subsequent timer invocation occurs (the task is no longer scheduled) and that no disbursement has been finalised.
