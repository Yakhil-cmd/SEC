### Title
Unauthorized Caller Can Register Swap Participation on Behalf of Any Buyer, Bypassing Confirmation-Text Consent — (`rs/sns/swap/canister/canister.rs`)

---

### Summary

The SNS Swap canister's `refresh_buyer_tokens` endpoint accepts a caller-supplied `buyer` principal string without verifying that the caller **is** that principal. Any unprivileged ingress sender can therefore invoke the endpoint on behalf of an arbitrary victim, supplying the (publicly readable) `confirmation_text` on the victim's behalf and committing the victim's pre-deposited ICP to the swap without the victim's explicit consent.

---

### Finding Description

`refresh_buyer_tokens` in `rs/sns/swap/canister/canister.rs` resolves the effective buyer as follows:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // ← no caller == buyer check
};
``` [1](#0-0) 

When `arg.buyer` is non-empty the function uses the attacker-supplied principal directly, with no assertion that `caller == p`. The downstream `refresh_buyer_token_e8s` then:

1. Reads the ICP balance of `buyer`'s subaccount on the swap canister.
2. Validates `confirmation_text` against the SNS-configured required text (a public value readable from canister state).
3. Writes the buyer's participation amount into `self.buyers`. [2](#0-1) 

The `confirmation_text` is the only gate that is supposed to represent the buyer's explicit consent to the swap terms. Because the text is public and the caller identity is never checked against the `buyer` field, an attacker can supply the correct text on the victim's behalf.

The `RefreshBuyerTokensRequest` proto documents the field as "If not specified, the caller is used," implying third-party calls are intentional for the no-text case, but the design does not account for the consent-text case. [3](#0-2) 

---

### Impact Explanation

**Governance authorization bug / ingress validation bypass.**

1. **Consent bypass.** The `confirmation_text` mechanism exists so that a buyer must explicitly agree to the swap terms before their ICP is committed. An attacker who knows the victim's principal and the (public) confirmation text can commit the victim's ICP to the swap without the victim ever calling the endpoint themselves.

2. **Swap-outcome manipulation.** If a swap is close to its minimum direct-participation threshold, an attacker can push it over the threshold by force-registering multiple victims who have already deposited ICP but have not yet chosen to confirm. This can cause a swap to commit that would otherwise abort, locking victims into receiving SNS tokens they did not intend to acquire at that time.

3. **Ticket deletion side-effect.** `refresh_buyer_token_e8s` also deletes an existing open ticket when the registered increment meets the ticket amount. An attacker can therefore silently delete a victim's ticket, disrupting the payment-flow protocol. [4](#0-3) 

---

### Likelihood Explanation

- **Entry path is fully unprivileged ingress.** Any Internet Computer user can send an update call to the swap canister's `refresh_buyer_tokens` method.
- **Required information is public.** The victim's principal is observable on-chain (ICP ledger subaccount deposits to the swap canister are public). The `confirmation_text` is readable from the swap canister's public state.
- **No special capability needed.** The attacker only needs to monitor the ICP ledger for deposits to the swap canister's subaccounts and then immediately call `refresh_buyer_tokens` with the victim's principal.

Likelihood is **medium-high** during an active SNS swap that uses a confirmation text.

---

### Recommendation

Add an explicit authorization check in `refresh_buyer_tokens`: if `arg.buyer` is non-empty, assert that `caller_principal_id() == p` before proceeding. Third-party triggering (for the no-confirmation-text case) can be preserved by only enforcing the check when a `confirmation_text` is required by the swap, or by removing the ability to supply an arbitrary `buyer` entirely and always using `caller_principal_id()`.

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let p = PrincipalId::from_str(&arg.buyer).unwrap();
    // Enforce that the caller is the buyer when a confirmation text is required.
    if swap().init_or_panic().confirmation_text.is_some() {
        assert_eq!(caller_principal_id(), p, "caller must be the buyer when confirmation_text is required");
    }
    p
};
```

---

### Proof of Concept

1. Alice sends 10 ICP to the SNS swap canister's subaccount derived from her principal (standard participation flow). She has not yet called `refresh_buyer_tokens`.
2. The SNS was initialized with `confirmation_text = "I agree to the terms"` (readable via `get_init`).
3. Attacker Eve sends an ingress update call:
   ```
   refresh_buyer_tokens({
     buyer: "<Alice's principal text>",
     confirmation_text: opt "I agree to the terms"
   })
   ```
4. The swap canister resolves `p = Alice`, validates the confirmation text (passes, because Eve supplied the correct public string), reads Alice's subaccount balance (10 ICP), and writes Alice's participation into `self.buyers`.
5. Alice's ICP is now committed to the swap. If the swap commits, Alice receives SNS tokens she never explicitly agreed to acquire at this moment. Her open ticket (if any) is also deleted. [1](#0-0) [5](#0-4) [6](#0-5)

        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

            };

                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
