### Title
MintSnsTokens 7-Day Minting Cap Unconditionally Disabled — (`rs/sns/governance/src/proposal.rs`)

---

### Summary

The `TokenProposalAction` implementation for `MintSnsTokens` permanently returns `Decimal::MAX` as the 7-day minting upper bound, effectively disabling the treasury-valuation-derived cap that is supposed to constrain how many SNS tokens can be minted via governance proposals within any rolling 7-day window. Any SNS governance participant with sufficient voting power can pass an unlimited number of `MintSnsTokens` proposals in rapid succession, minting an unbounded quantity of SNS tokens.

---

### Finding Description

The `TokenProposalAction` trait defines `recent_amount_total_upper_bound_tokens` as the mechanism that caps how many tokens may be minted within a 7-day window. For `TransferSnsTreasuryFunds`, the real cap function (`transfer_sns_treasury_funds_7_day_total_upper_bound_tokens`) is active and enforced. For `MintSnsTokens`, the equivalent function (`mint_sns_tokens_7_day_total_upper_bound_tokens`) is **commented out** and replaced with a stub that returns `Decimal::MAX`: [1](#0-0) [2](#0-1) 

The `treasury_valuation_if_proposal_amount_is_small_enough_or_err` function computes `allowance_remainder = max_tokens - spent_tokens` and rejects the proposal only if `proposal_amount > allowance_remainder`. With `max_tokens = Decimal::MAX`, this check can never trigger regardless of how many tokens have already been minted in the past 7 days. [3](#0-2) 

At execution time, `perform_mint_sns_tokens` performs no additional cap check — it calls `transfer_funds` directly: [4](#0-3) 

The integration test `sns_can_mint_funds_via_proposals` explicitly confirms this: the assertion that a second mint proposal should be rejected is commented out, and `doomed_make_proposal_result.unwrap()` is called instead, proving the second proposal succeeds on the current codebase: [5](#0-4) 

The real cap logic exists and is correct in the `proposals_amount_total_limit` crate — it is simply not wired in for `MintSnsTokens`: [6](#0-5) 

---

### Impact Explanation

An SNS governance participant controlling sufficient voting power can submit N `MintSnsTokens` proposals within a 7-day window, each for any `amount_e8s` up to `u64::MAX`. All proposals pass validation (cap = `Decimal::MAX`) and execute successfully, minting an unbounded quantity of SNS tokens. This causes:

- **Unbounded SNS token inflation**: total supply can be inflated to `u64::MAX` e8s per proposal, repeated indefinitely within the window.
- **Ledger conservation loss**: tokens are created from nothing, diluting all existing holders.
- **Treasury-valuation invariant broken**: the 7-day cap is specifically designed to limit minting to a fraction of treasury value; that invariant is entirely absent for `MintSnsTokens`.

---

### Likelihood Explanation

The precondition is controlling sufficient SNS voting power to pass proposals. This is a governance-majority requirement, which is a meaningful barrier. However:

- Many SNS DAOs have concentrated token distributions (e.g., founding teams, early investors) where a single entity or small coalition can pass proposals.
- The 7-day cap is specifically designed as a protocol-level safety guard that should constrain even a legitimate governance majority — its absence means the guard provides zero protection.
- The TODO comment and the integration test stub (`doomed_make_proposal_result.unwrap()`) confirm this is a known, unresolved gap in production code, not a theoretical edge case.

---

### Recommendation

Uncomment the real implementation of `recent_amount_total_upper_bound_tokens` for `MintSnsTokens` and the corresponding import, as tracked by ticket NNS1-2982:

In `rs/sns/governance/src/proposal.rs`:
- Uncomment `mint_sns_tokens_7_day_total_upper_bound_tokens` in the import block (line 52).
- Replace the `Decimal::MAX` stub (lines 1035–1041) with the commented-out real implementation (lines 1025–1033). [2](#0-1)

### Proof of Concept

State-machine test outline (mirrors the existing `sns_can_mint_funds_via_proposals` test):

1. Deploy an SNS with a whale neuron holding majority voting power.
2. Set ICP/XDR price so the SNS token treasury is valued at, say, 500,000 XDR ("medium" regime → 25% cap → ~12,500 XDR worth of tokens allowed per 7 days).
3. Submit and pass `MintSnsTokens` proposal #1 for `amount_e8s = u64::MAX / 2`.
4. Immediately submit and pass `MintSnsTokens` proposal #2 for `amount_e8s = u64::MAX / 2`.
5. Assert total minted ≈ `u64::MAX` e8s — orders of magnitude above any treasury-derived cap.
6. Confirm that removing the `Decimal::MAX` stub and enabling the real cap causes proposal #2 to be rejected with "Amount is too large … try again in a few days".

The existing integration test at `rs/sns/integration_tests/src/sns_treasury.rs:966` already demonstrates step 3–4 succeed today; the commented-out block at lines 942–965 is the expected post-fix behavior. [7](#0-6)
