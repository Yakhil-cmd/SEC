### Title
Silo Address Whitelist Not Enforced in `receive_base_tokens()`, Allowing Base ETH Minting to Non-Whitelisted Addresses — (File: `engine/src/engine.rs`)

---

### Summary

In Aurora Engine's Silo mode, `receive_erc20_tokens()` correctly checks the silo Address whitelist before minting ERC-20 tokens to a recipient, redirecting to a fallback address if the recipient is not whitelisted. However, the sibling function `receive_base_tokens()` — which mints native ETH (base tokens) — performs no such whitelist check. Any NEAR account can call `ft_transfer_call` on the ETH connector and specify an arbitrary non-whitelisted EVM address as the recipient, bypassing the silo's address access control and permanently freezing the deposited ETH at an address that cannot submit transactions.

---

### Finding Description

The Silo subsystem enforces an `Address` whitelist (`WhitelistKind::Address`) to control which EVM addresses are permitted to interact with the silo. This is checked in two places:

- `is_allow_submit()` — prevents non-whitelisted addresses from submitting EVM transactions.
- `is_allow_receive_erc20_tokens()` — prevents non-whitelisted addresses from receiving ERC-20 bridge tokens; if the recipient is not whitelisted and a fallback address is configured, the tokens are redirected to the fallback.

In `engine/src/engine.rs`, `receive_erc20_tokens()` correctly applies this check:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

However, `receive_base_tokens()` — which handles ETH deposits via the ETH connector — contains no equivalent check:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    // No whitelist check here
    set_balance(&mut self.io, &receipient, &new_balance);
    ...
}
``` [2](#0-1) 

The dispatch between the two paths occurs in `ft_on_transfer()` in `engine/src/contract_methods/connector.rs`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)   // ← no whitelist check
} else {
    engine.receive_erc20_tokens(...)    // ← whitelist enforced
};
``` [3](#0-2) 

The `is_allow_receive_erc20_tokens` function itself only checks the `Address` whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [4](#0-3) 

---

### Impact Explanation

**Impact: Critical — Permanent Freezing of Funds.**

When the silo Address whitelist is enabled, a non-whitelisted EVM address cannot submit transactions (`is_allow_submit` blocks it). If ETH is minted to such an address via `receive_base_tokens`, the ETH is permanently frozen: the address holds a balance it can never spend, and the depositor's NEP-141 ETH tokens are consumed by the connector. There is no recovery path — the silo operator cannot return the ETH to the depositor without a privileged intervention that is not part of the normal protocol flow.

Additionally, this is a whitelist bypass: the silo's intended invariant — that only whitelisted addresses hold or use assets — is violated for base ETH, while it is correctly enforced for ERC-20 tokens.

---

### Likelihood Explanation

**Likelihood: High.**

The entry path requires no special privileges. Any NEAR account holding NEP-141 ETH tokens can call `ft_transfer_call` on the ETH connector contract, specifying any EVM address (including a non-whitelisted one) as the recipient in the `msg` field. The silo Address whitelist is a production feature actively used in Silo deployments. The inconsistency between `receive_base_tokens` and `receive_erc20_tokens` is a straightforward oversight that any user — intentionally or accidentally — can trigger.

---

### Recommendation

Apply the same whitelist guard in `receive_base_tokens()` that is already present in `receive_erc20_tokens()`. Before crediting the recipient's balance, check `silo::is_allow_receive_erc20_tokens(&self.io, &receipient)` (or an equivalent `is_allow_receive_base_tokens` helper). If the recipient is not whitelisted and a fallback address is configured, redirect the ETH to the fallback address. If no fallback is configured, return an error so the NEP-141 tokens are refunded to the sender.

---

### Proof of Concept

**Setup:** Silo mode is active. The Address whitelist is enabled. Address `0xDEAD...` is not whitelisted. A fallback ERC-20 address is configured.

**Step 1:** Alice holds NEP-141 ETH tokens on NEAR.

**Step 2:** Alice calls `ft_transfer_call` on the ETH connector:
```json
{
  "receiver_id": "aurora",
  "amount": "1000000000000000000",
  "msg": "DEAD000000000000000000000000000000000000"
}
```

**Step 3:** The ETH connector calls `ft_on_transfer` on Aurora. Since `predecessor_account_id == connector_account_id`, `receive_base_tokens` is invoked. No whitelist check is performed. `0xDEAD...` receives 1 ETH in its EVM balance.

**Step 4:** Alice's NEP-141 ETH tokens are consumed (transferred to Aurora). `0xDEAD...` holds 1 ETH but cannot submit any EVM transaction (blocked by `is_allow_submit`). The ETH is permanently frozen.

**Contrast:** If Alice had instead bridged an ERC-20 token to `0xDEAD...`, `receive_erc20_tokens` would have detected the non-whitelisted recipient and redirected the tokens to the configured fallback address — protecting Alice's funds. No equivalent protection exists for base ETH.
