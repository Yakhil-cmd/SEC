Audit Report

## Title
Voting Reward Purse Permanently Lost When Settled Proposals Have Zero Voting Participation - (`rs/nns/governance/src/governance.rs`, `rs/sns/governance/src/governance.rs`)

## Summary
In both NNS and SNS governance, when a reward round contains settled proposals but no eligible votes were cast, the reward purse is neither distributed to neurons nor rolled over to the next round — it is silently and permanently destroyed. The root cause is that `rewards_rolled_over()` uses `settled_proposals.is_empty()` as its sole condition, but the distribution-skip guard fires when proposals are non-empty yet voting power is zero, leaving the purse in a state where neither distribution nor rollover occurs.

## Finding Description
**NNS (`rs/nns/governance/src/governance.rs`):**

At line 6696–6700, `sum_weighted_voting_power` is called over `considered_proposals`. If all neurons have zero deciding voting power at proposal creation (possible under `VotingPowerEconomics` after ~7 months without refresh), `total_voting_rights == 0.0`. The guard at line 6712 fires: `if total_voting_rights < 0.001 { ... None }`, setting `reward_distribution = None` and leaving `actually_distributed_e8s_equivalent = 0`. The `RewardEvent` at lines 6747–6757 is then stored with `settled_proposals: considered_proposals` — a **non-empty** list — and `distributed_e8s_equivalent: 0`.

In the next round, `e8s_equivalent_to_be_rolled_over()` (`rs/nns/governance/src/reward/calculation.rs`, lines 120–126) calls `rewards_rolled_over()` (line 145–147), which returns `self.settled_proposals.is_empty()`. Because proposals were settled, this returns `false`, so `e8s_equivalent_to_be_rolled_over()` returns `0`. The purse is permanently lost.

**SNS (`rs/sns/governance/src/governance.rs`):**

The identical structural flaw exists. At line 5946, `if total_reward_shares == dec!(0)` skips the distribution block. The `RewardEvent` at lines 6084–6092 is stored with `settled_proposals: considered_proposals` (non-empty) and `distributed_e8s_equivalent: 0`. In `rs/sns/governance/src/types.rs` lines 2054–2067, `rewards_rolled_over()` again returns `self.settled_proposals.is_empty()` — `false` — so `e8s_equivalent_to_be_rolled_over()` returns `0`. The SNS reward purse for that round is permanently destroyed.

The mismatch is precise: `rewards_rolled_over()` was designed to detect "no proposals were ready, so roll over," but it is also triggered (incorrectly returning `false`) in the case "proposals were ready but nobody voted."

## Impact Explanation
This is a permanent, irreversible loss of governance reward funds. For SNS, the entire reward purse computed for the affected round — derived from the live token supply and `VotingRewardsParameters` — is discarded with no recovery path. For NNS, the daily pool represents a fraction of total ICP supply (5–10% annualized), meaning hundreds of thousands of ICP-equivalent maturity per day could be destroyed. This matches the allowed impact: **significant SNS/NNS governance financial security impact with concrete and irreversible user/protocol harm** (High, $2,000–$10,000), and potentially Critical for NNS given the scale of the daily reward pool.

## Likelihood Explanation
**SNS (higher likelihood):** An SNS with few neurons where all neurons dissolve, or an early-stage SNS where no neuron holds sufficient dissolve delay to vote, will naturally produce proposals that expire with zero eligible votes. No adversarial action is required; this is a normal operational scenario. Any SNS project can hit this condition.

**NNS (lower but non-zero):** Requires all NNS neurons to have zero deciding voting power simultaneously (approximately 7+ months without any neuron refreshing under `VotingPowerEconomics`). This is unlikely for the full NNS given its large neuron population, but becomes more plausible during extended governance inactivity or for low-participation governance periods.

## Recommendation
Decouple the rollover condition from whether proposals were settled. The correct invariant is: **if rewards were not distributed for any reason, the purse must roll over.**

Concretely, in both NNS and SNS:
1. Add a boolean field to `RewardEvent` (e.g., `rewards_were_distributed: bool`), set to `true` only when the distribution block actually executes (i.e., `reward_distribution` is `Some(_)` in NNS, or `total_reward_shares != 0` in SNS).
2. Change `rewards_rolled_over()` in `rs/nns/governance/src/reward/calculation.rs` and `rs/sns/governance/src/types.rs` to return `!self.rewards_were_distributed` instead of `self.settled_proposals.is_empty()`.

Alternatively, when the zero-voting-power guard fires with non-empty proposals, record the `RewardEvent` with an empty `settled_proposals` list so the existing rollover logic fires correctly — though this leaves proposals in `ReadyToSettle` state indefinitely, which has its own drawbacks.

## Proof of Concept
**SNS path (most reproducible):**
1. Deploy an SNS with `VotingRewardsParameters` configured.
2. Create a proposal; ensure no neuron casts a vote (e.g., all neurons are dissolved or have insufficient dissolve delay).
3. Wait for the proposal to expire and enter `ReadyToSettle`.
4. Trigger `distribute_rewards()`. Observe `total_reward_shares == dec!(0)` at line 5946 — distribution block is skipped.
5. Inspect `latest_reward_event`: `settled_proposals` is non-empty, `distributed_e8s_equivalent == 0`, `total_available_e8s_equivalent == Y > 0`.
6. Trigger the next reward round. Observe `e8s_equivalent_to_be_rolled_over()` returns `0` (because `rewards_rolled_over()` returns `false`).
7. Confirm `Y` is absent from the new round's `rewards_purse_e8s` — permanently lost.

A deterministic integration test using PocketIC can reproduce this by controlling neuron dissolve state and proposal voting windows, then asserting that `total_available_e8s_equivalent` accumulates correctly across rounds when no votes are cast.
