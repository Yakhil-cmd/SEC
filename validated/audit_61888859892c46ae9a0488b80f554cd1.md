### Title
Rate limit slots consumed by no-op `update_subnet_admins` calls, enabling exhaustion without state change — (`rs/registry/canister/src/mutations/do_update_subnet_admins.rs`)

### Summary
`do_update_subnet_admins` reserves a rate-limit slot before computing the new admin list, and commits that slot whenever `compute_new_subnet_admins` returns `Ok(...)`. Because no-op operations (adding an already-present principal, removing an absent principal) return `Ok` with an unchanged list, they consume a slot identically to effective operations. All 10 daily slots for a target subnet can be exhausted by no-op calls, blocking legitimate admin updates for up to 24 hours.

### Finding Description
The reserve-then-commit pattern in `do_update_subnet_admins` has no guard that checks whether the resulting admin list differs from the current one before calling `commit`. The `compute_new_subnet_admins` function uses set union (for `Add`) and set difference (for `Remove`), both of which are total functions that silently absorb no-ops and return `Ok`. The rate limiter's `commit` is then unconditionally called on the `Ok` branch, consuming a slot regardless of whether any registry mutation was meaningful.

### Impact Explanation
With 10 slots per day and a replenishment interval of 8,640 seconds (~2.4 hours per slot), exhausting all 10 slots via no-op calls locks out legitimate subnet admin updates for the remainder of the replenishment window — up to ~24 hours. For a rented application subnet, this means the tenant cannot add or remove admins during that period.

### Likelihood Explanation
The subnet rental canister is the authorized caller. If the subnet rental canister's own logic retries operations (e.g., idempotent "ensure admin X is present" patterns), or if a user-facing interface of the subnet rental canister can be triggered to submit redundant add/remove calls, the no-op exhaustion occurs through normal operational paths without any adversarial key compromise. The test suite already demonstrates the no-op path succeeds without any rate-limit awareness.

### Recommendation
Before calling `commit`, compare the new admin list to the current one. Only commit the reservation if the lists differ:

```rust
let new_admins = compute_new_subnet_admins(...)?;
if new_admins != current_subnet_admins {
    subnet_record.subnet_admins = new_admins;
    self.maybe_apply_mutation_internal(mutations);
    UPDATE_SUBNET_ADMINS_RATE_LIMITER
        .with_borrow_mut(|limiter| limiter.commit(reservation, now));
} else {
    // Drop reservation — no state change, no slot consumed
    drop(reservation);
}
```

Alternatively, detect no-ops in `compute_new_subnet_admins` and return a distinct `Err` variant (e.g., `NoOp`) that causes the caller to drop the reservation without panicking.

### Proof of Concept
The existing test `can_not_add_existing_subnet_admins` already demonstrates the no-op path succeeds. Extending it to call `do_update_subnet_admins` with `Add([A])` (A already present) ten times and then asserting the eleventh call panics with `RateLimited` would confirm the exhaustion. The rate-limit unit test in `rate_limits.rs` confirms 10 commits exhaust the limiter: [8](#0-7)
