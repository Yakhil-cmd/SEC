### Title
Missing Storage Collision Check in CREATE/CREATE2 Allows Deployment at Addresses with Pre-existing Storage — (`evm_interpreter/src/ee_trait_impl.rs`)

### Summary

ZKsync OS's `before_executing_frame` function omits the EIP-684/EIP-7610 storage-non-empty collision check when processing CREATE and CREATE2 deployments. The code explicitly acknowledges this deviation. An unprivileged caller can exploit this to deploy a contract at an address that already has non-empty storage (zero code, zero nonce), causing the newly deployed contract to inherit attacker-controlled storage slots — the inverse of the external report's DOS pattern: instead of blocking creation, the attacker poisons the initial state of the created contract.

---

### Finding Description

EIP-684 (codified as EIP-7610 for the Paris fork) requires that CREATE and CREATE2 abort with a collision error if the target address has **non-empty code, non-zero nonce, OR non-empty storage**. ZKsync OS only checks the first two conditions.

In `before_executing_frame`:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // burn gas, emit CreateCollision, return Ok(false)
}
// ← storage is never checked
``` [1](#0-0) 

The comment is a self-admission that the third EIP-684 condition is unimplemented. The project's own test indexes confirm this with multiple disabled entries:

- `create2collisionStorageParis.json` — `enabled: false`, `comment: We do not check storage for collision`
- `RevertInCreateInInitCreate2Paris.json` — `enabled: false`, `comment: We do not check for storage collisions`
- `test_init_collision_create_opcode.json` — `enabled: false`, `comment: We do not check the storage`
- `test_init_collision_create_tx.json` — `enabled: false`, `comment: We do not check the storage` [2](#0-1) [3](#0-2) [4](#0-3) 

The `eip7610_create_collision` test group is entirely disabled in the `paris` section of the stable blockchain test index, covering both the opcode and transaction-level CREATE collision paths.

---

### Impact Explanation

**Attacker-controlled initial storage state in a newly deployed contract.**

The canonical exploit path (matching the `create2collisionStorageParis.json` scenario):

1. Attacker computes the deterministic CREATE2 address `X` for a victim factory's future deployment.
2. Attacker deploys a contract at `X` in transaction T₁, writes attacker-controlled values to storage slots (e.g., slot 0 = attacker address as "owner"), then calls `SELFDESTRUCT` within the same transaction.
3. Under EIP-6780 semantics, same-transaction SELFDESTRUCT clears code, nonce, and balance. In standard EVM (EIP-6780), **storage is NOT cleared**; address `X` is left with `code=∅`, `nonce=0`, `balance=0`, `storage={slot0: attacker}`.
4. In transaction T₂, the victim factory calls `CREATE2` at `X`. Standard EVM rejects this (EIP-684: non-empty storage). ZKsync OS accepts it because only `code_len` and `nonce` are checked.
5. The victim's constructor runs at `X` and the deployed contract inherits the attacker's pre-written storage. If slot 0 is an `owner` or `initialized` flag, the attacker now controls the contract.

Note: ZKsync OS's `finish_tx` calls `clear_state_impl` on SELFDESTRUCT in the same transaction, which may also clear storage (a separate EIP-6780 deviation). However, the missing storage collision check remains independently exploitable whenever any mechanism leaves non-empty storage at an address with zero code and zero nonce — including L1 priority transactions or system hooks that can write to arbitrary storage slots, or future changes to the SELFDESTRUCT handling. [5](#0-4) 

---

### Likelihood Explanation

The attack requires reaching a state where `code=∅`, `nonce=0`, `storage≠∅` at a predictable CREATE2 address. The primary path (same-tx SELFDESTRUCT leaving storage) is partially mitigated by ZKsync OS's own `clear_state_impl` call. However:

- The missing check is a **permanent structural gap** — any future code path, system hook, or L1 transaction that writes storage to an address without setting code/nonce bypasses the collision guard entirely.
- The `eip7610_create_collision` tests are disabled at the test-index level, confirming the gap is known and unresolved.
- Factory contracts using CREATE2 for deterministic proxy or vault deployment are common DeFi primitives and are the natural targets.

Likelihood is **medium**: the direct SELFDESTRUCT path is partially blocked by ZKsync OS's own storage-clearing behavior, but the structural absence of the check leaves a permanent attack surface for any alternative pre-population vector.

---

### Recommendation

Add a storage-emptiness check to the collision guard in `before_executing_frame`. The check must query whether the target address has any non-zero storage slots before allowing the constructor to execute. Because this requires a storage scan (potentially expensive), the standard approach is to track a per-account "storage is non-empty" flag (a single boolean in the account properties) that is set on the first `SSTORE` and cleared only when all slots are zeroed. This flag can then be read cheaply during the collision check alongside `code_len` and `nonce`.

```rust
// Proposed addition:
if deployee_code_len != 0 || deployee_nonce != 0 || deployee_has_storage {
    // emit CreateCollision
}
``` [1](#0-0)

### Proof of Concept

```
Transaction T1 (attacker):
  CREATE2(salt=S, initcode=<writes slot0=attacker_addr; SELFDESTRUCT(attacker)>)
  → deploys at address X
  → slot0 at X = attacker_addr
  → SELFDESTRUCT clears code/nonce/balance at X
  → (standard EVM EIP-6780: storage at X remains non-empty)

Transaction T2 (victim factory):
  CREATE2(salt=S, initcode=<proxy constructor>)
  → computes same address X
  → ZKsync OS: code_len=0, nonce=0 → collision check PASSES (storage not checked)
  → constructor executes at X
  → deployed proxy inherits slot0 = attacker_addr
  → attacker calls owner-only function on proxy → drains funds
```

Standard EVM at step T2: `CreateCollision` — constructor never runs.
ZKsync OS at step T2: constructor runs, proxy deployed with attacker-controlled storage. [6](#0-5) [7](#0-6) [4](#0-3)
