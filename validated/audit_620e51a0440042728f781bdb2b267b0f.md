Audit Report

## Title
Missing Subnet-Type Equality Check in `validate_request` Permits Cross-Type Canister Migration - (File: `rs/migration_canister/src/validation.rs`)

## Summary

`validate_request` in `rs/migration_canister/src/validation.rs` fetches both subnet types to reject `CloudEngine` subnets but never asserts that the two types are equal. An ordinary canister controller who owns canisters on an `Application` and a `VerifiedApplication` subnet can trigger a complete, irreversible migration between them. The governance-owned `prepare_canister_migration` path explicitly enforces type equality via `validate_subnets_consistency`, confirming this is a protocol-level requirement that the migration canister omits.

## Finding Description

**Root cause.** Step 2 of `validate_request` (lines 82–99 of `rs/migration_canister/src/validation.rs`) calls `get_subnet` on both subnets and returns early only when either equals `SubnetType::CloudEngine`. The two returned values are never compared to each other:

```rust
if get_subnet(migrated_canister_subnet).await? == SubnetType::CloudEngine { … }
if get_subnet(replaced_canister_subnet).await? == SubnetType::CloudEngine { … }
// ← no equality check between the two types
```

The `ValidationError` enum (`rs/migration_canister/src/lib.rs`, lines 46–84) contains no `SubnetTypesMismatch` variant, so no later stage can surface this error either.

**Contrast with the governance path.** `prepare_canister_migration` (`rs/registry/canister/src/mutations/prepare_canister_migration.rs`, lines 164–183) calls `validate_subnets_consistency`, which returns `Err(SubnetTypesMismatch)` when `source_subnet.subnet_type != destination_subnet.subnet_type`. The same file also restricts migrations to `[Application, VerifiedApplication]` via `SUPPORTED_SUBNET_TYPES` (line 11–12). The migration canister path enforces neither constraint.

**Registry layer also absent.** The registry's `do_migrate_canisters` (`rs/registry/canister/src/mutations/do_migrate_canisters.rs`, lines 25–46) only validates that the target subnet exists; it performs no subnet-type check. The migration canister is therefore the sole gatekeeper, and it is missing the check.

**Exploit flow.**
1. Attacker controls canister **A** on an `Application` subnet and canister **B** on a `VerifiedApplication` subnet.
2. Attacker adds the migration canister as a controller of both and stops both.
3. Attacker calls `migrate_canister { migrated_canister_id: A, replaced_canister_id: B }`.
4. `validate_request` fetches both subnet types, confirms neither is `CloudEngine`, and proceeds — no type-equality check is performed.
5. The state machine runs to completion: `process_accepted` takes exclusive control of both canisters; `process_stopped` calls `rename_canister` on the `VerifiedApplication` subnet; `process_renamed` calls the registry's `migrate_canisters` to update the routing table; `process_routing_table` deletes the original canister **A** from the `Application` subnet; `process_migrated_canister_deleted` restores controllers.
6. Canister **A**'s ID now resolves to the `VerifiedApplication` subnet (or vice versa). The original canister is permanently deleted; the operation cannot be reversed.

## Impact Explanation

The concrete harm is an irreversible security downgrade. `VerifiedApplication` subnets have SEV-SNP hardware attestation and memory encryption enabled; `Application` subnets do not. A canister migrated from `VerifiedApplication` to `Application` silently loses the hardware-level confidentiality and integrity guarantees that its users relied upon. The canister's controllers are restored and the migration is marked `Succeeded`, so there is no error signal and no recovery path. This constitutes a significant platform-level security impact with concrete user harm, matching the High impact class ($2,000–$10,000): "Significant … infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

No privileged role is required. The preconditions are:
- Caller is a controller of both canisters (ordinary canister ownership).
- Migration canister has been added as a controller of both (required by the protocol's own flow).
- Caller is on the allowlist, or the allowlist is `None` (the default deployment uses `allowlist: None`, granting access to all principals).

A developer who manages canisters across `Application` and `VerifiedApplication` subnets — an increasingly common scenario — can trigger this accidentally. The allowlist is a soft rollout gate, not a permanent security boundary; it is reconfigurable by the migration canister's controllers.

## Recommendation

Immediately after the `CloudEngine` checks in `validate_request` (`rs/migration_canister/src/validation.rs`, after line 98), add:

```rust
let migrated_subnet_type = get_subnet(migrated_canister_subnet).await?;
let replaced_subnet_type = get_subnet(replaced_canister_subnet).await?;
if migrated_subnet_type != replaced_subnet_type {
    return Err(ValidationError::SubnetTypesMismatch {
        migrated_subnet: migrated_canister_subnet,
        replaced_subnet: replaced_canister_subnet,
    });
}
```

Add the corresponding `SubnetTypesMismatch` variant to `ValidationError` in `rs/migration_canister/src/lib.rs` and expose it in `migration_canister.did`. Optionally, mirror `prepare_canister_migration`'s `SUPPORTED_SUBNET_TYPES` allowlist to also reject `System` subnet migrations.

## Proof of Concept

A deterministic PocketIC integration test can prove this:

1. Extend the existing `setup` function in `rs/migration_canister/tests/tests.rs` to accept a `VerifiedApplication` subnet alongside the existing `Application` subnet (using `PocketIcBuilder::with_verified_application_subnet()`).
2. Create canister **A** on the `Application` subnet and canister **B** on the `VerifiedApplication` subnet, both with the migration canister and a shared controller as controllers.
3. Stop both canisters.
4. Call `migrate_canister { migrated_canister_id: A, replaced_canister_id: B }` — expect `Ok(())` (no `SubnetTypesMismatch` error is returned).
5. Advance time and tick until `migration_status` returns `Succeeded`.
6. Assert that canister **A**'s ID now resolves to the `VerifiedApplication` subnet and that the original canister on the `Application` subnet no longer exists.
