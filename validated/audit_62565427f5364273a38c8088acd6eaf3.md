### Title
Permanent Fund Loss on Failed `ExitToNear` Push Transfer When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

### Summary

The `ExitToNear` precompile uses a push-payment model: it burns ERC-20 tokens (or deducts ETH balance) inside the EVM and then schedules an outbound `ft_transfer` / `ft_transfer_call` NEAR promise to the caller-supplied recipient account. When that promise fails (e.g., the recipient NEAR account is not registered with the NEP-141 token contract), the callback `exit_to_near_precompile_callback` receives `refund: None` in the default (non-`error_refund`) build and silently returns `Ok(None)`. The burned tokens are never re-minted and the user's funds are permanently lost.

### Finding Description

**Root cause — `engine-precompiles/src/native.rs` lines 449–453:**

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in the default production build
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not compiled in, `refund` is unconditionally `None`, so no refund address or amount is ever encoded into the callback arguments.

**Callback silently no-ops on failure — `engine/src/contract_methods/connector.rs` lines 214–242:**

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    ...
    None
} else if let Some(args) = args.refund {   // ← args.refund is always None without the feature
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← reached on every failure; no refund, no error
};
``` [2](#0-1) 

The `else` branch is reached whenever the outbound `ft_transfer` promise fails **and** `error_refund` is not enabled. The function returns `Ok(None)` — the EVM state change (token burn / ETH deduction) is already committed and is never reversed.

**The `error_refund` feature is opt-in and not part of the default feature set:** [3](#0-2) 

The test suite itself documents the permanent-loss behaviour when the feature is absent:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing / loss of user funds.**

Any EVM user who calls the `ExitToNear` precompile (directly or via an ERC-20 wrapper contract) with a recipient NEAR account that is not registered with the target NEP-141 contract will have their ERC-20 tokens burned inside the EVM with no corresponding NEP-141 credit and no on-chain refund. The tokens are irrecoverably destroyed. This matches the "permanent freezing of funds" impact class.

### Likelihood Explanation

**Medium.** NEAR NEP-141 tokens require explicit `storage_deposit` registration before an account can receive them. A user who specifies any unregistered NEAR account as the exit recipient — including a freshly created account, a mistyped account ID, or a valid account that simply has not called `storage_deposit` on that specific token — will silently lose their funds. No special attacker capability is required; the user's own transaction is the trigger.

### Recommendation

1. **Enable `error_refund` in the production build** so that `refund_call_args` populates the refund address and the callback can re-mint burned ERC-20 tokens (or return ETH) when the outbound promise fails.
2. As a defence-in-depth measure, validate at precompile entry time that the recipient account is registered with the target NEP-141 contract before burning any tokens, and revert the EVM call if it is not.
3. Alternatively, adopt a pull-pattern: hold the NEP-141 tokens in escrow on the Aurora contract and let the recipient claim them, so a failed push never results in a permanent burn.

### Proof of Concept

1. Deploy any NEP-141 token and bridge it to Aurora as an ERC-20.
2. As an EVM user, call the ERC-20's `withdrawToNear` (or equivalent) targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
3. The `ExitToNear` precompile burns the ERC-20 tokens and schedules `ft_transfer` to the unregistered account.
4. The `ft_transfer` promise fails (NEP-141 rejects unregistered recipients).
5. `exit_to_near_precompile_callback` is invoked; `args.refund` is `None` (no `error_refund` feature); the `else` branch returns `Ok(None)`.
6. Observe: ERC-20 balance is zero, NEP-141 balance of the recipient is zero, and no refund was issued — funds are permanently lost. [5](#0-4) [6](#0-5) [7](#0-6)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
