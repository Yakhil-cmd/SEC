Audit Report

## Title
Unbounded Loop via `Time::add_assign` u128→u64 Truncation Causes OOM Crash in `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The `/construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` values with no bounds validation. When `ingress_start` is set near `u64::MAX`, the `now += interval` step in the ingress expiry loop causes `Time::add_assign` to silently wrap `now` to zero via a truncating `u128 as u64` cast in `Time::from_duration`. This makes the loop termination condition permanently true, driving `ingress_expiries` to exhaust process memory and crash the Rosetta node.

## Finding Description

**Root cause — truncating cast in `Time::from_duration`:**

`Time::add_assign` at `rs/types/types/src/time.rs` lines 55–58 delegates to `Time::from_duration`:

```rust
fn add_assign(&mut self, other: Duration) {
    *self = Time::from_duration(Duration::from_nanos(self.0) + other)
}
``` [1](#0-0) 

`Time::from_duration` at lines 103–105 performs a silent truncating cast:

```rust
fn from_duration(t: Duration) -> Self {
    Time(t.as_nanos() as u64)   // u128 → u64, no overflow check
}
``` [2](#0-1) 

`Duration::as_nanos()` returns `u128`. When the sum exceeds `u64::MAX`, the `as u64` cast silently wraps to a small value.

**No bounds validation on attacker-supplied inputs:**

`ingress_start` and `ingress_end` are taken directly from the JSON request body and passed to the bare `Time(nanos)` constructor with no range check: [3](#0-2) [4](#0-3) 

**The unbounded loop:**

```rust
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;   // wraps to 0 on overflow
}
``` [5](#0-4) 

`interval` is `MAX_INGRESS_TTL(300s) − PERMITTED_DRIFT(60s) − 120s = 120_000_000_000 ns`: [6](#0-5) [7](#0-6) 

**Exploit flow:**

With `ingress_start = u64::MAX − 120_000_000_000 + 1` and `ingress_end = u64::MAX`:

- Iteration 1: `now < ingress_end` → true; `now += interval` computes `(u64::MAX − 120_000_000_000 + 1 + 120_000_000_000) = u64::MAX + 1 = 2^64` as u128; `as u64` → **0**.
- Iteration 2+: `0 < u64::MAX` → permanently true; loop runs ≈ `u64::MAX / 120_000_000_000 ≈ 153 billion` iterations, each pushing a `u64` (8 bytes) onto `ingress_expiries` → **≈1.2 TB allocation → OOM crash**.

**Existing safe alternative not used:** `Time::checked_add` exists at lines 122–128 and correctly handles overflow, but is not used in the loop: [8](#0-7) 

## Impact Explanation

A single unauthenticated HTTP POST to `/construction/payloads` with crafted `metadata.ingress_start` / `metadata.ingress_end` values causes the Rosetta node process to exhaust all available memory and crash. This is a deterministic, single-request application-level DoS against the ICP Rosetta API, which is an explicitly in-scope financial integration component. This matches the allowed impact: **"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS" → High ($2,000–$10,000)**.

## Likelihood Explanation

- `/construction/payloads` requires no authentication; any HTTP client can reach it.
- `metadata.ingress_start` and `metadata.ingress_end` are optional JSON integer fields; no schema-level constraint prevents near-`u64::MAX` values.
- No server-side guard exists between JSON deserialization and the loop.
- The exploit is deterministic, reproducible with a single HTTP request, and requires no special privileges or prior knowledge beyond the public Rosetta API spec.

## Recommendation

1. **Replace the truncating cast** in `Time::from_duration` with a checked or saturating conversion, or restrict it to internal use with a documented invariant.
2. **Use `Time::checked_add`** (already present at `rs/types/types/src/time.rs` lines 122–128) instead of `+=` in the loop body, breaking on `None`.
3. **Add an upper-bound guard** before the loop, e.g.:
   ```rust
   const MAX_EXPIRIES: u64 = 1000;
   if ingress_end > ingress_start + interval * MAX_EXPIRIES {
       return Err(ApiError::invalid_request("ingress window too large"));
   }
   ```
4. **Validate `ingress_start` and `ingress_end`** at deserialization time to reject values that would produce a window larger than a reasonable multiple of `interval`.

## Proof of Concept

```rust
// Unit test — no network required
use ic_types::time::Time;
use std::time::Duration;

let interval_ns: u64 = 120_000_000_000;
let ingress_start = u64::MAX - interval_ns + 1;
let ingress_end   = u64::MAX;

let mut count = 0usize;
let mut now = Time::from_nanos_since_unix_epoch(ingress_start);
let end     = Time::from_nanos_since_unix_epoch(ingress_end);
let dur     = Duration::from_nanos(interval_ns);

while now < end {
    count += 1;
    assert!(count <= 2, "wrap occurred: now wrapped to 0, loop is infinite (count={})", count);
    now += dur;
}
// With current code: assertion fires at count=3 because now wraps to 0 on first += and
// 0 < u64::MAX is permanently true.
```
