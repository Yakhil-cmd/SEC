Based on my analysis of the actual code, this is a valid vulnerability. Here is the full finding:

---

### Title
Cache Key Includes `ingress_expiry`, Enabling Anonymous Cache Pollution on Boundary Node — (`rs/boundary_node/ic_boundary/src/routes.rs`)

---

### Summary

The boundary node's query response cache uses `RequestContext` as its cache key. `RequestContext`'s `Hash` and `PartialEq` implementations both include `ingress_expiry`. For anonymous query requests, `ingress_expiry` is never validated at the boundary node, meaning an attacker can supply arbitrary `u64` values. Each distinct value produces a unique cache key, allowing an attacker to flood the finite LRU/moka cache with unique entries, evicting legitimate cached responses and forcing all subsequent legitimate queries to be cache misses.

---

### Finding Description

**Cache key includes `ingress_expiry`**

In `rs/boundary_node/ic_boundary/src/routes.rs`, both `Hash` and `PartialEq` for `RequestContext` include `ingress_expiry`:

```rust
impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.canister_id.hash(state);
        self.sender.hash(state);
        self.method_name.hash(state);
        self.ingress_expiry.hash(state);   // <-- included
        ...
    }
}

impl PartialEq for RequestContext {
    fn eq(&self, other: &Self) -> bool {
        let r = self.canister_id == other.canister_id
            && self.sender == other.sender
            && self.method_name == other.method_name
            && self.ingress_expiry == other.ingress_expiry;  // <-- included
        ...
    }
}
``` [1](#0-0) 

**`ingress_expiry` is never validated for anonymous queries at the boundary node**

`preprocess_request` in `rs/boundary_node/ic_boundary/src/http/middleware/process.rs` simply parses and stores `ingress_expiry` from the CBOR body with no range check:

```rust
let ctx = RequestContext {
    ...
    ingress_expiry: content.ingress_expiry,
    ...
};
``` [2](#0-1) 

The only place `ingress_expiry` is range-validated is in `rs/validator/src/ingress_validation.rs`, and that validation is **explicitly skipped for anonymous senders**:

```rust
impl<R> HttpRequestVerifier<Query, R> for HttpRequestVerifierImpl {
    fn validate_request(...) {
        if !request.sender().get().is_anonymous() {
            validate_ingress_expiry(request, current_time)?;  // skipped for anonymous
        }
        ...
    }
}
``` [3](#0-2) 

This validation runs at the replica, not at the boundary node cache layer. The cache middleware runs before the request ever reaches the replica.

**The cache key extractor uses the full `RequestContext`**

`KeyExtractorContext` in `cache.rs` uses `Arc<RequestContext>` directly as the cache key: [4](#0-3) 

**`ingress_expiry` is semantically irrelevant to query results**

For query calls, `ingress_expiry` is a client-side timestamp used for deduplication of update/ingress messages. It has no effect on the query execution result. Two anonymous queries identical in `canister_id`, `sender`, `method_name`, and `arg` but differing only in `ingress_expiry` will produce the same response from the replica, yet they are treated as entirely different cache entries.

---

### Impact Explanation

An attacker sends a continuous stream of anonymous `QueryV2` requests to the same canister/method/arg, each with a distinct `ingress_expiry` value (e.g., incrementing integers, or values spread across the valid `u64` range). Since anonymous queries skip expiry validation entirely, all values are accepted. Each request inserts a new entry into the finite LRU/moka cache (`cache_size` is a configured finite bound). Once the cache is full, new entries evict the oldest legitimate entries. Legitimate clients making repeated identical queries never achieve `CacheStatus::Hit`, forcing every query to be forwarded to the replica. This:

- Eliminates the caching benefit for all legitimate query traffic
- Increases replica load proportionally
- Constitutes a constrained availability degradation on the boundary node

The `MAX_INGRESS_TTL` is 5 minutes (`rs/limits/src/lib.rs`), but since anonymous queries skip expiry validation, the attacker is not constrained to any window — they can use values from `0` to `u64::MAX`. [5](#0-4) 

---

### Likelihood Explanation

- **No authentication required**: anonymous identity, no keys, no registration.
- **No rate limiting specific to `ingress_expiry` variation**: the boundary node has no guard that detects or throttles requests varying only in `ingress_expiry`.
- **Trivially scriptable**: a simple loop incrementing `ingress_expiry` in a CBOR-encoded query body is sufficient.
- **Continuously exploitable**: the attacker must maintain the stream to keep the cache polluted, but this is a low-cost operation.

---

### Recommendation

Remove `ingress_expiry` from the `Hash` and `PartialEq` implementations of `RequestContext` used as the cache key. `ingress_expiry` is semantically irrelevant to query results and should not differentiate cache entries. The cache key for query responses should be based only on `canister_id`, `sender`, `method_name`, and `arg` (or `http_request`). [1](#0-0) 

---

### Proof of Concept

```rust
// Pseudocode: drive N anonymous QueryV2 requests through cache_middleware
// with distinct ingress_expiry values and a small cache_size.
// Assert that a repeated legitimate query never achieves CacheStatus::Hit.

let cache_size = 10;
let n = cache_size + 100;

// Flood the cache with unique ingress_expiry values
for i in 0..n {
    let req = gen_request_with_params(
        CANISTER_1,
        RequestType::QueryV2,
        /*nonce=*/false,
        DEFAULT_SIZE,
        /*ingress_expiry=*/i as u64,  // distinct value each time
        /*anonymous=*/true,
        StatusCode::OK,
    );
    app.call(req).await.unwrap();
}

// Now send the "legitimate" repeated query (ingress_expiry=0)
let req = gen_request_with_params(CANISTER_1, RequestType::QueryV2,
    false, DEFAULT_SIZE, 0, true, StatusCode::OK);
let res = app.call(req).await.unwrap();
let cs = res.extensions().get::<CacheStatus<BypassReasonIC>>().cloned().unwrap();

// This will be Miss, not Hit — the entry was evicted
assert!(matches!(cs, CacheStatus::Miss(_)));
```

The test harness already exists in `cache.rs`'s `#[cfg(test)]` module with `gen_request_with_params` accepting an `ingress_expiry` parameter, making this directly reproducible. [6](#0-5)
