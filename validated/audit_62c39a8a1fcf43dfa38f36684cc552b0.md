### Title
Silo Mode Whitelist Bypass via `call` and `deploy_code` Methods — (`engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

Aurora Engine exposes two distinct paths for EVM execution: the `submit` path (Ethereum-signed RLP transactions) and the `call`/`deploy_code` path (NEAR-account-triggered EVM calls). The `submit` path enforces a Silo mode whitelist check (`assert_access`) before execution, but the `call` and `deploy_code` methods skip this check entirely. Any unprivileged NEAR account can bypass the Silo whitelist by using the `call` method instead of `submit`, executing EVM transactions in a restricted environment they are not authorized to access.

---

### Finding Description

`submit_with_alt_modexp` in `engine/src/engine.rs` performs a full validation pipeline before EVM execution:

1. Transaction parse + signature recovery
2. **`assert_access` — Silo whitelist check** (line 1052)
3. Chain ID validation
4. Nonce check
5. Intrinsic/floor gas check (EIP-7623)
6. `max_priority_fee > max_fee` check
7. EIP-3607 check
8. Gas payment [1](#0-0) 

The `call` contract method in `evm_transactions.rs` skips all of these steps and calls directly into `engine.call_with_args`: [2](#0-1) 

`call_with_args` itself passes `u64::MAX` as the gas limit and invokes `engine.call` with no prior validation: [3](#0-2) 

The same omission applies to `deploy_code`: [4](#0-3) 

`assert_access` checks both the NEAR predecessor account ID and the EVM sender address against the Silo whitelist: [5](#0-4) 

Neither `call` nor `deploy_code` invokes `assert_access`, so the Silo whitelist is never consulted on these paths.

---

### Impact Explanation

Silo mode is a restricted execution environment designed to limit which NEAR accounts and EVM addresses may submit transactions. The whitelist enforced by `assert_access` is the primary access-control boundary. Because `call` and `deploy_code` bypass this boundary, any NEAR account — regardless of whitelist status — can:

- Execute arbitrary EVM calls against any deployed contract, including contracts holding ETH or ERC-20 balances.
- Deploy new EVM contracts into the Silo environment.

If the Silo deployment holds user funds (ETH balances, ERC-20 tokens), an unauthorized caller can interact with those contracts and drain them. This satisfies the **direct theft of user funds** impact criterion.

---

### Likelihood Explanation

The `call` method is a standard public NEAR contract entrypoint with no caller restriction at the NEAR level. Any NEAR account can invoke it with a crafted `CallArgs` payload. No special privilege, leaked key, or governance action is required. The attacker only needs a NEAR account and knowledge of the target contract address and ABI. [6](#0-5) 

---

### Recommendation

Add an `assert_access`-equivalent check at the top of the `call` and `deploy_code` contract methods in `engine/src/contract_methods/evm_transactions.rs`, mirroring what `submit_with_alt_modexp` does. The check should gate on the NEAR predecessor account ID and the EVM address derived from it (via `predecessor_address`), consistent with the Silo whitelist semantics already implemented for `submit`.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode with a whitelist that excludes `attacker.near`.
2. Deploy an EVM contract holding ETH at address `0xVictim`.
3. From `attacker.near`, call the Aurora Engine's `call` NEAR method with:
   ```
   CallArgs::V2(FunctionCallArgsV2 {
       contract: 0xVictim,
       value: 0,
       input: <drain_funds_calldata>,
   })
   ```
4. The call succeeds: `assert_access` is never invoked, the Silo whitelist is not consulted, and the EVM call executes with the attacker's NEAR-derived address as `origin`.
5. Funds are transferred out of `0xVictim` to the attacker's EVM address, despite `attacker.near` not being whitelisted. [2](#0-1) [7](#0-6)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}
