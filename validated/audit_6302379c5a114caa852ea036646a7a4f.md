### Title
Wrong `burn_in_block` Index Used for ckERC20 Reimbursement State Update Causes Stuck Reimbursement Requests and Potential Double-Minting — (`rs/ethereum/cketh/minter/src/withdraw.rs`)

---

### Summary

In `process_reimbursement()`, the `Reimbursed.burn_in_block` field is populated with `reimbursement_request.ledger_burn_index`, which for `CkErc20` withdrawals equals the **ckETH** ledger burn index, not the ckERC20 ledger burn index. This value is then consumed in `audit.rs` as the `ckerc20_ledger_burn_index` when reconstructing the `ReimbursementIndex::CkErc20` key passed to `record_finalized_reimbursement`. When `cketh_ledger_burn_index ≠ ckerc20_ledger_burn_index` (the typical production case), the key lookup fails silently, the reimbursement request is never removed from `reimbursement_requests`, and the periodic timer re-processes it — minting ckERC20 tokens to the user a second time.

---

### Finding Description

**Root cause — wrong parameter in `process_reimbursement()`:**

In `rs/ethereum/cketh/minter/src/withdraw.rs`, the `Reimbursed` struct is constructed as:

```rust
let reimbursed = Reimbursed {
    burn_in_block: reimbursement_request.ledger_burn_index,  // ← ckETH burn index for CkErc20
    reimbursed_in_block: LedgerMintIndex::new(block_index),
    reimbursed_amount: reimbursement_request.reimbursed_amount,
    transaction_hash: reimbursement_request.transaction_hash,
};
``` [1](#0-0) 

For a `CkErc20` reimbursement, `reimbursement_request.ledger_burn_index` is the **ckETH** ledger burn index (the withdrawal ID), as confirmed by the test fixture: [2](#0-1) 

The actual `ckerc20_ledger_burn_index` is available in the `index` variable (`ReimbursementIndex::CkErc20 { ckerc20_ledger_burn_index, .. }`), but it is **never used** when building `Reimbursed`.

**Wrong key propagated into `audit.rs`:**

The emitted `EventType::ReimbursedErc20Withdrawal` carries `reimbursed.burn_in_block` = `cketh_ledger_burn_index`. In `audit.rs`, this value is used as `ckerc20_ledger_burn_index` when reconstructing the `ReimbursementIndex::CkErc20` key for `record_finalized_reimbursement`:

```rust
EventType::ReimbursedErc20Withdrawal {
    cketh_ledger_burn_index,
    ckerc20_ledger_id,
    reimbursed,
} => {
    state.eth_transactions.record_finalized_reimbursement(
        ReimbursementIndex::CkErc20 {
            cketh_ledger_burn_index: *cketh_ledger_burn_index,
            ledger_id: *ckerc20_ledger_id,
            ckerc20_ledger_burn_index: reimbursed.burn_in_block,  // ← ckETH index, not ckERC20!
        },
        *reimbursed_in_block,
    );
}
``` [3](#0-2) 

The actual key stored in `reimbursement_requests` is `ReimbursementIndex::CkErc20 { cketh_ledger_burn_index: X, ledger_id: Y, ckerc20_ledger_burn_index: Z }`. The reconstructed key has `ckerc20_ledger_burn_index: X` (wrong). When `X ≠ Z`, `BTreeMap::remove` returns `None` silently — the entry is never removed.

**Why the test suite does not catch this:**

The unit test `should_reimburse_tokens_when_ckerc20_withdrawal_fails` deliberately sets both indices to the same value (`LedgerBurnIndex::new(7)`), masking the mismatch: [4](#0-3)
