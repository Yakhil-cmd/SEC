Audit Report

## Title
Missing Response-Type/Request-Type Cross-Check in `validate_self_validating_payload_impl` and `push_response` Allows Byzantine Block Proposer to Deliver Mismatched Bitcoin Adapter Response — (`rs/bitcoin/consensus/src/payload_builder.rs`, `rs/replicated_state/src/bitcoin.rs`)

## Summary

`validate_self_validating_payload_impl` performs no semantic validation of a `SelfValidatingPayload`: it only checks whether the payload is empty and returns its byte size. A single Byzantine block proposer can include a `BitcoinAdapterResponse` whose response variant (`SendTransactionResponse`) does not match the request type stored in state for the given `callback_id` (a `GetSuccessors` context). Honest nodes will notarize and finalize the block. During batch delivery, the `SendTransactionResponse` arm of `push_response` unconditionally pushes `EmptyBlob` to the `consensus_queue` for the supplied `callback_id` without consulting `bitcoin_send_transaction_internal_contexts`, causing the bitcoin canister's `GetSuccessors` callback to receive `EmptyBlob` instead of a `BitcoinGetSuccessorsResponse`.

## Finding Description

**Gap 1 — Validation is a near-no-op.**

`validate_self_validating_payload_impl` at lines 234–251 of `rs/bitcoin/consensus/src/payload_builder.rs` checks only whether the payload equals `SelfValidatingPayload::default()` and, if not, returns the byte count. It never loads state, never verifies that a `callback_id` exists in either `bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`, and never verifies that the response variant matches the stored request type. [1](#0-0) 

The `BatchPayloadBuilder::validate_payload` path at lines 357–397 calls this same function after decoding, adding only a size check. [2](#0-1) 

**Gap 2 — `SendTransactionResponse` arm has no existence guard.**

In `rs/replicated_state/src/bitcoin.rs`, the `GetSuccessorsResponse` arm (lines 28–39) looks up `bitcoin_get_successors_contexts` and returns `StateError::BitcoinNonMatchingResponse` if the `callback_id` is absent. [3](#0-2) 

The `SendTransactionResponse` arm (lines 64–75) performs **no lookup at all**: it unconditionally constructs `EmptyBlob` and pushes a `ConsensusResponse` for whatever `callback_id` was supplied, without ever consulting `bitcoin_send_transaction_internal_contexts`. [4](#0-3) 

The two context maps are separate `BTreeMap`s in `SubnetCallContextManager`, so a `callback_id` that belongs to a `GetSuccessors` context will never appear in `bitcoin_send_transaction_internal_contexts`. [5](#0-4) 

**Attack path:**

1. Byzantine node proposes a block whose `SelfValidatingPayload` contains `BitcoinAdapterResponse { response: SendTransactionResponse {}, callback_id: X }` where `X` is a live `BitcoinGetSuccessorsContext` in state.
2. Honest nodes call `validate_payload` → `validate_self_validating_payload_impl`. The payload is non-empty and within size limits → **validation passes**.
3. Block is notarized (2/3 threshold) and finalized.
4. Batch delivery calls `push_response` with the mismatched response.
5. `SendTransactionResponse` arm fires: `EmptyBlob` is pushed to `consensus_queue` for `callback_id X`.
6. The `BitcoinGetSuccessorsContext` at `X` is **not touched** by the `SendTransactionResponse` arm.
7. Execution environment delivers `EmptyBlob` to the bitcoin canister's `GetSuccessors` callback.

Note: the `GetSuccessorsReject` and `SendTransactionReject` arms (lines 77–102) also lack existence guards, making them susceptible to the same cross-type injection pattern. [6](#0-5) 

## Impact Explanation

The bitcoin canister's `GetSuccessors` callback receives `EmptyBlob` where it expects a `BitcoinGetSuccessorsResponse`. This causes either a trap/panic (execution rolls back; the canister stalls for that request cycle) or silent misinterpretation (if the canister treats a decode failure as "no new blocks," it skips blocks, corrupting its UTXO set). A corrupted UTXO set is the precondition for false ckBTC minting. This maps to **High** ($2,000–$10,000): "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm." The Critical tier (false ckBTC minting) is plausible but requires confirming the bitcoin canister's decode-failure behavior, which is not demonstrated in the submitted PoC.

## Likelihood Explanation

The attacker must control one node on the bitcoin subnet. Subnet membership is governed by NNS, so this is not a zero-cost position. However, the IC's BFT model explicitly tolerates up to `f < n/3` malicious nodes, and a single malicious block proposer is squarely within that model. The exploit requires no threshold key, no governance majority, and no external dependency. It is fully local to one subnet round and is repeatable across rounds as long as the `GetSuccessorsContext` remains in state.

## Recommendation

1. **In `validate_self_validating_payload_impl`**: load the certified state and, for each `BitcoinAdapterResponse` in the payload, verify (a) the `callback_id` exists in the appropriate context map and (b) the response variant matches the stored request type (`GetSuccessors` context ↔ `GetSuccessorsResponse`/`GetSuccessorsReject`; `SendTransaction` context ↔ `SendTransactionResponse`/`SendTransactionReject`). Return `SelfValidatingPayloadValidationError` on mismatch.

2. **In `push_response` (`SendTransactionResponse` arm)**: add a lookup against `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if the `callback_id` is absent, mirroring the guard already present in the `GetSuccessorsResponse` arm. Apply the same fix to `GetSuccessorsReject` and `SendTransactionReject` arms.

## Proof of Concept

```rust
// State: BitcoinGetSuccessorsContext at callback_id=0
let mut state = ReplicatedState::new(SUBNET_ID, SubnetType::Application);
state.metadata.subnet_call_context_manager.push_context(
    SubnetCallContext::BitcoinGetSuccessors(BitcoinGetSuccessorsContext { ... }),
);

// Byzantine payload: SendTransactionResponse for a GetSuccessors callback_id
let mismatched = BitcoinAdapterResponse {
    response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
        SendTransactionResponse {}
    ),
    callback_id: 0,
};

// Validation passes — no type/existence check
let payload = SelfValidatingPayload::new(vec![mismatched.clone()]);
assert!(validator.validate_self_validating_payload_impl(&payload, &ctx).is_ok());

// After finalization, push_response delivers EmptyBlob to callback_id=0
state.push_response_bitcoin(mismatched).unwrap();

// EmptyBlob delivered to a GetSuccessors callback
assert_eq!(state.consensus_queue[0].callback_id, CallbackId::from(0));
assert_eq!(
    state.consensus_queue[0].payload,
    Payload::Data(EmptyBlob.encode())
);

// GetSuccessors context NOT removed — re-exploitable
assert!(state.metadata.subnet_call_context_manager
    .bitcoin_get_successors_contexts.contains_key(&CallbackId::from(0)));
```

    fn validate_self_validating_payload_impl(
        &self,
        payload: &SelfValidatingPayload,
        validation_context: &ValidationContext,
    ) -> Result<NumBytes, SelfValidatingPayloadValidationError> {
        let since = Instant::now();

        // An empty block is always valid.
        if *payload == SelfValidatingPayload::default() {
            return Ok(0.into());
        }

        self.metrics
            .observe_validate_duration(VALIDATION_STATUS_VALID, since);
        let size = NumBytes::new(payload.count_bytes() as u64);

        Ok(size)
    }
