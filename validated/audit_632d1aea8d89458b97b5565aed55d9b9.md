Audit Report

## Title
Hot Keys Not Cleared on Neuron Controller Change via `spawn_neuron` - (File: `rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with an explicit `new_controller` different from the parent neuron's controller, the parent's `hot_keys` are cloned verbatim onto the child neuron. Because NNS governance grants hot keys the ability to vote and set followees on behalf of a neuron, the original controller retains unauthorized governance influence over a neuron they no longer own.

## Finding Description
In `spawn_neuron`, the `child_controller` is resolved from `spawn.new_controller` at line 2651–2655: if a `new_controller` is provided, it is used as the child's controller; otherwise the parent's controller is used. The child neuron is then constructed at lines 2704–2718 with `child_controller` as the controller principal, but `.with_hot_keys(parent_neuron.hot_keys.clone())` at line 2714 unconditionally copies all of the parent's hot keys onto the child, regardless of whether the controller changed.

The authorization check in `is_authorized_to_vote` (`rs/nns/governance/src/neuron/types.rs`, lines 243–255) returns `true` for any principal that is either the controller or present in `hot_keys`. The `register_vote` path (`rs/nns/governance/src/governance.rs`, lines 5594–5602) and the `follow` path (lines 5725–5748) both gate on `is_authorized_to_vote`, meaning any inherited hot key principal retains full voting and following authority on the child neuron. No existing guard in `spawn_neuron` conditionally clears or filters hot keys when `new_controller != parent_controller`.

The `split_neuron` path (lines 2241–2257) has the same hot-key copy but always assigns `*caller` as the child controller, so the controller identity does not change and the issue is less harmful there.

## Impact Explanation
This is unauthorized access to a governance asset: the original controller (Owner A) retains the ability to cast NNS votes and redirect voting power (via `Follow`) on behalf of a neuron now legally owned by Owner B. This directly distorts NNS governance outcomes and constitutes unauthorized control over another user's neuron. This matches the High bounty impact: "Unauthorized access to neurons, governance assets … where exploitation requires meaningful per-target work or other constraints."

## Likelihood Explanation
The attack is fully self-contained and requires no special privilege beyond controlling a neuron with registered hot keys. Owner A initiates the spawn themselves, deliberately specifying `new_controller = Owner B`. The `Spawn` command is a standard, publicly accessible NNS governance operation. Owner B receives the neuron with no protocol-level warning and must proactively enumerate and remove inherited hot keys to close the exposure. The attack is repeatable for any neuron with maturity above the spawn threshold.

## Recommendation
In `spawn_neuron`, conditionally clear the hot-key list when the spawned neuron's controller differs from the parent's controller:

```rust
let hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(hot_keys)
```

This is consistent with the principle that hot keys are credentials registered by and for the current owner. Followees may reasonably be inherited (they represent voting preferences, not access credentials), but hot keys must not survive an ownership transfer.

## Proof of Concept
1. Owner A controls neuron N. Owner A calls `manage_neuron` → `Configure` → `AddHotKey` to register a secondary principal `HK_A` on N.
2. Owner A calls `manage_neuron` → `Spawn` with `new_controller = Owner B` and sufficient maturity. `spawn_neuron` executes and creates child neuron C with `controller = Owner B` and `hot_keys = [HK_A]` via `.with_hot_keys(parent_neuron.hot_keys.clone())` at line 2714.
3. An open NNS proposal P exists for which neuron C has a ballot.
4. `HK_A` calls `manage_neuron` on neuron C → `RegisterVote` for proposal P. The check at line 5594–5602 calls `neuron.is_authorized_to_vote(&HK_A)`, which returns `true` because `HK_A ∈ hot_keys`. The vote is cast successfully on behalf of Owner B's neuron.
5. Owner B, inspecting their neuron via `get_full_neuron`, observes `hot_keys = [HK_A]` — a principal they never authorized.

This can be reproduced as a unit test by extending the existing `test_neuron_spawn` test in `rs/nns/governance/tests/governance.rs`: add a hot key to the parent neuron before spawning, spawn with a different `new_controller`, then assert that the child neuron's `hot_keys` field is empty.
