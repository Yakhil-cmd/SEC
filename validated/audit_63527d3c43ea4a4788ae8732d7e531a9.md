Audit Report

## Title
CMC Rate-Limit Bypass via Cycles Donation to `ensure_balance` — (File: `rs/nns/cmc/src/main.rs`)

## Summary

`ensure_balance` computes `cycles_to_mint` as a saturating subtraction of the CMC's live balance from the requested amount. When the CMC's balance has been pre-funded by an external donation, `cycles_to_mint` saturates to zero, causing the rate limiter and `total_cycles_minted` counter to be charged zero. Any unprivileged canister can exploit this by depositing cycles to the CMC via the management canister's `deposit_cycles` endpoint, then converting ICP to cycles through `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` without ever consuming the hourly rate limit.

## Finding Description

`ensure_balance` at `rs/nns/cmc/src/main.rs:2306-2325` reads the live `canister_balance128()` and computes:

```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;  // saturating sub
```

`Cycles::Sub` is implemented as `saturating_sub` at `rs/types/cycles/src/cycles.rs:133-138`, so when `current_balance >= cycles`, `cycles_to_mint` becomes `Cycles(0)`.

Both the rate-limiter call at line 2316 and the accounting increment at line 2317 receive this zero value:

```rust
limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;  // charged 0
state.total_cycles_minted += cycles_to_mint;                        // incremented by 0
```

`CyclesMintingLimiterSelector::check_and_add_cycles` at `rs/nns/cmc/src/main.rs:403-419` delegates to `Limiter::check_and_add_cycles` at `rs/nns/cmc/src/limiter.rs:34-56`, which checks `count + cycles_to_mint > limit`. With `cycles_to_mint = 0`, this check always passes regardless of how many cycles have already been dispensed in the current window.

The caller `deposit_cycles` at `rs/nns/cmc/src/main.rs:2120-2128` then unconditionally sends the full `cycles` amount from the CMC's existing balance to the target via `call_with_payment128`. The rate limit is never consumed.

The attack path:
1. Attacker calls the IC management canister's `deposit_cycles` endpoint targeting the CMC's canister ID, depositing a large cycle balance.
2. CMC's `canister_balance128()` now exceeds any single `notify_top_up` request amount.
3. Each subsequent `notify_top_up` / `notify_mint_cycles` / `notify_create_canister` call invokes `ensure_balance`, which computes `cycles_to_mint = 0`, charges the limiter 0, and pays out from the donated balance.
4. ICP is burned, cycles are received, rate limit is untouched.

## Impact Explanation

This is a **High** severity finding. The hourly base rate limit (`base_cycles_limit`, default 150P cycles/hour) and the monthly subnet-rental limit (`subnet_rental_cycles_limit`) are the sole protocol controls preventing runaway ICP-to-cycles conversion. Bypassing them allows an attacker to convert an arbitrarily large ICP position to cycles within a single rate-limit window, undermining the protocol's cycles inflation controls. Additionally, `total_cycles_minted` — the on-chain accounting metric exposed via the `total_cycles_minted` query — will under-count all cycles dispensed from the donated balance, corrupting the audit trail for cycles inflation. This constitutes significant NNS/CMC infrastructure security impact with concrete protocol harm.

## Likelihood Explanation

The attack requires no privileged access, no governance vote, and no threshold-crypto compromise. Any canister on any subnet can call the management canister's `deposit_cycles` endpoint. The attacker's upfront cost is the donated cycles, but those cycles are not destroyed — they are re-dispensed to the attacker's chosen targets in exchange for ICP. The attack is repeatable: once the donated balance is exhausted, the attacker can re-donate. The only constraint is the attacker must hold sufficient cycles to pre-fund the CMC, which is obtainable from any cycles source (DEX, borrowing, prior accumulation).

## Recommendation

Always charge the rate limiter and accounting counter for the full dispensed amount, regardless of whether cycles are freshly minted or sourced from an existing balance:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    // Charge the full dispensed amount against the rate limit and accounting.
    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        state.total_cycles_minted += cycles;
        Ok::<_, String>(())
    })?;

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    if current_balance < cycles {
        let cycles_to_mint = cycles - current_balance;
        let _minted = ic0_mint_cycles128(cycles_to_mint);
    }
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

## Proof of Concept

A deterministic integration test using the existing `state_machine_builder_for_nns_tests` framework:

1. Set up NNS canisters with a funded ledger account.
2. Advance time to a fresh rate-limit window.
3. Call `notify_top_up` once to consume most of the 150P/hr base limit (e.g., 140P).
4. From a second canister, call the management canister's `deposit_cycles` targeting the CMC canister ID with 200P cycles.
5. Call `notify_top_up` again for another 140P worth of ICP — this should be rate-limited under correct behavior, but with the bug, `ensure_balance` computes `cycles_to_mint = 0` (CMC balance ≥ 140P due to donation) and the call succeeds.
6. Assert that `total_cycles_minted` has not increased by the second 140P, confirming the accounting corruption.

The existing test `cmc_notify_top_up_not_rate_limited_by_invalid_top_up` at `rs/nns/integration_tests/src/cycles_minting_canister.rs:1689-1743` demonstrates awareness of rate-limit manipulation but does not cover the cycles-donation vector. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }
