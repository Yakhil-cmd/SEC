### Title
Owner Can Temporarily Freeze All Bridged ETH by Resetting the ETH Connector Account Without Checking Active Deposits - (File: engine/src/contract_methods/connector.rs)

### Summary
`set_eth_connector_contract_account` allows the Aurora Engine owner to overwrite the stored ETH connector account ID at any time with no guard on whether user funds are currently held in the existing connector. Changing the connector account redirects every connector-proxied operation to the new account while all previously deposited ETH remains locked in the old connector contract, temporarily freezing those funds.

### Finding Description
`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` unconditionally overwrites the stored connector account ID:

```rust
// engine/src/contract_methods/connector.rs  lines 418-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);          // ← no guard
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}

The stored account ID is the sole routing target for every connector-proxied call. `return_promise` reads it unconditionally:

Every user-facing connector operation — `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw`, `ft_total_eth_supply_on_near`, `ft_balance_of`, `ft_metadata` — is routed through `return_promise` and therefore through the stored connector account ID. Additionally, `ft_on_transfer` uses the same stored ID to distinguish base-token (ETH) deposits from ERC-20 deposits:

```rust
// engine/src/contract_methods/connector.rs  line 81
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
};
``` [5](#0-4) 

There is no check analogous to `require(token == address(0) || totalStaking == 0)` before the overwrite.

### Impact Explanation
**High — Temporary freezing of funds.**

After the connector account is changed to account B:
- All `withdraw` calls are routed to account B; ETH held in account A is unreachable.
- All `ft_transfer` / `ft_transfer_call` calls are routed to account B.
- New ETH deposits sent by the old connector (account A) via `ft_on_transfer` are misclassified as ERC-20 transfers.

Every unit of ETH that was bridged through the old connector remains locked until the owner resets the connector account back to account A. If account B does not exist or is malicious, the freeze can become permanent.

### Likelihood Explanation
**Low-Medium.** The function is callable only by the owner or via a private (self) call. The scenario is an accidental misconfiguration during a connector migration or upgrade — the same class of mistake described in the reference report. No external attacker input is required; the owner's own (mistaken) transaction is sufficient to trigger the freeze.

### Recommendation
Add a guard before overwriting the connector account ID. The guard should verify that no ETH is currently held in the existing connector, or alternatively enforce a two-step migration (set a pending connector, then confirm after verifying zero balance). A minimal check would be to reject the call if the connector account is already set and the total ETH supply on NEAR is non-zero:

```rust
// Pseudocode
let current = get_connector_account_id(&io);
if current.is_ok() {
    // Ensure no funds are currently held before allowing a reset
    require!(total_eth_supply_on_near == 0, ERR_CONNECTOR_HAS_ACTIVE_DEPOSITS);
}
set_connector_account_id(io, &args.account);
```

### Proof of Concept
1. Owner deploys Aurora Engine and sets connector to account A via `set_eth_connector_contract_account`.
2. Users call `withdraw` / `ft_transfer_call` to bridge ETH; ETH is held in connector contract at account A.
3. Owner calls `set_eth_connector_contract_account` with account B (e.g., a new connector version or a typo).
4. `set_connector_account_id` overwrites the stored key with account B — no revert, no check. [6](#0-5) 

5. Any user calling `withdraw` now has their promise routed to account B; the ETH in account A is inaccessible.
6. The freeze persists until the owner calls `set_eth_connector_contract_account` again with account A. [7](#0-6)
