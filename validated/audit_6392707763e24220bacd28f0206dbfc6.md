### Title
Missing EIP-7610 Storage Collision Check Allows Pre-Deployment Storage Poisoning of CREATE2 Contracts — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter omits the EIP-7610 storage-non-empty collision check during `CREATE`/`CREATE2` deployment. An unprivileged attacker can pre-populate storage at a deterministic `CREATE2` address (by deploying a contract there, writing storage, then self-destructing), and when the victim later deploys their contract at that same address, the deployment succeeds with attacker-controlled initial storage values. This is a direct EVM semantic mismatch with a concrete funds-loss path.

---

### Finding Description

EIP-7610 (merged into the EVM specification) requires that a `CREATE` or `CREATE2` must revert if the target address has non-empty storage, in addition to the existing checks for non-zero nonce and non-empty code. This prevents an attacker from poisoning the initial storage of a contract before it is deployed.

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` function performs the collision check for constructor calls:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // ... burn gas, return CreateCollision error
}
```

The comment explicitly acknowledges the missing check. Only `code_len` and `nonce` are tested; the storage root is never consulted. This is confirmed by multiple disabled EIP-7610 test entries in the test index files, all annotated with `"We do not check the storage"` or `"We do not check for storage collisions"`. [1](#0-0) 

The test index files confirm this is a known, unimplemented check: [2](#0-1) [3](#0-2) 

---

### Impact Explanation

After a contract self-destructs, its account properties (code, nonce, balance) are cleared, but its storage slots persist in the flat storage model. ZKsync OS's `mark_for_deconstruction` only clears account metadata; it does not wipe storage slots. [4](#0-3) 

When a new contract is subsequently deployed at the same address via `CREATE2`, ZKsync OS's collision check passes (nonce = 0, code_len = 0), the constructor executes, and the deployed contract inherits the attacker-pre-written storage. If the contract uses storage slot 0 as an owner/admin address, a balance mapping, or an `initialized` flag, the attacker controls those values from the moment of deployment.

**Concrete impact:** Funds loss or complete access-control bypass in any contract deployed via `CREATE2` to an address whose storage was pre-poisoned by an attacker.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of the future `CREATE2` address (salt, deployer, initcode hash — all observable on-chain or predictable from protocol design).
2. A single unprivileged transaction to deploy a contract at that address, write storage, and self-destruct.
3. Waiting for the victim's deployment transaction.

No privileged access, no oracle manipulation, and no governance majority is required. The attacker entry path is a standard EVM transaction. Protocols that use `CREATE2` for deterministic factory deployments (DEX pools, proxy contracts, account abstraction wallets) are the primary targets.

---

### Recommendation

Implement the EIP-7610 storage-non-empty collision check in `before_executing_frame`. Before allowing a constructor to execute, read the storage root (or check whether any storage slot is non-zero) for the target address and treat a non-empty storage root as a `CreateCollision` failure, burning all gas and returning `false`, consistent with the existing nonce/code checks. [5](#0-4) 

---

### Proof of Concept

1. Attacker computes the `CREATE2` address `A` for a future victim deployment (known salt `S`, deployer `D`, initcode hash `H`).
2. Attacker deploys a contract at `A` using `CREATE2(S)` from `D` with initcode that:
   - Writes `attacker_address` to storage slot 0 (`SSTORE(0, attacker_address)`).
   - Self-destructs (`SELFDESTRUCT(attacker_address)`).
3. After the transaction: `A` has nonce=0, code=empty, balance=0, but storage slot 0 = `attacker_address`.
4. Victim deploys their contract (e.g., an upgradeable proxy whose slot 0 is the admin) at `A` via `CREATE2(S)`.
5. ZKsync OS checks: `code_len == 0` ✓, `nonce == 0` ✓ → no collision detected → constructor executes.
6. The deployed proxy contract has slot 0 = `attacker_address` → attacker is the admin from block 0.
7. Attacker calls `upgradeTo(malicious_impl)` and drains all funds. [6](#0-5)
