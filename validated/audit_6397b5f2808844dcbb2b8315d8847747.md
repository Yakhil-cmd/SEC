Audit Report

## Title
`estimate_withdrawal_fee` Permanently Drains `available_utxos` When Called via Inter-Canister Update Call - (`rs/bitcoin/ckbtc/minter/src/main.rs`)

## Summary

`estimate_withdrawal_fee` is annotated `#[query]` but internally calls `mutate_state`, passing `&mut s.available_utxos` to `estimate_retrieve_btc_fee`, which calls `utxos_selection` → `greedy`, permanently removing UTXOs from the live set. The inline comment claims state changes are discarded even in replicated mode; this is incorrect. When any canister calls a `#[query]` method via inter-canister call, the IC executes it in replicated mode and commits all state mutations. An unprivileged attacker canister can therefore permanently drain the minter's `available_utxos` without performing any Bitcoin withdrawal, causing a complete DoS on all ckBTC withdrawal requests.

## Finding Description

`estimate_withdrawal_fee` at `rs/bitcoin/ckbtc/minter/src/main.rs:219-248` is annotated `#[query]` and calls `mutate_state` with a closure that passes `&mut s.available_utxos` directly to `ic_ckbtc_minter::estimate_retrieve_btc_fee`. [1](#0-0) 

`estimate_retrieve_btc_fee` delegates to `queries::estimate_withdrawal_fee` at `rs/bitcoin/ckbtc/minter/src/queries.rs:46-76`, which calls `utxos_selection(withdrawal_amount, available_utxos, 1)`. [2](#0-1) 

`utxos_selection` at `rs/bitcoin/ckbtc/minter/src/lib.rs:1039-1060` calls `greedy`, which removes UTXOs from `available_utxos` via `available_utxos.remove(...)` and returns them in a `Vec`. [3](#0-2) 

`greedy` at `rs/bitcoin/ckbtc/minter/src/lib.rs:1070-1100` only re-inserts UTXOs into `available_utxos` in the failure branch (when there are not enough UTXOs). On success, the selected UTXOs are returned and never re-inserted. [4](#0-3) 

The selected UTXOs returned by `utxos_selection` are passed to `build_unsigned_transaction_from_inputs` for fee calculation only; they are never returned to `available_utxos`. The net effect of a successful call is that the selected UTXOs are permanently removed from the minter's live state.

The inline comment at line 221-222 states: *"This is a **query** endpoint, so mutating the state is not an issue (even when called in replicated mode) since any change will be discarded."* This is factually wrong. On the Internet Computer, when a canister calls another canister's `#[query]` method via an inter-canister call (originating from an update context), the callee executes in **replicated mode** and all state mutations are committed to the replicated state. The `#[query]` annotation only controls how the method is exposed to external users making direct query calls; it does not prevent state commits when the method is invoked in replicated mode via inter-canister call.

The ckDoge test harness confirms the correct design: its equivalent `estimate_withdrawal_fee` is called via `update_call`, indicating it is an `#[update]` endpoint. [5](#0-4) 

## Impact Explanation

An attacker deploys a canister and calls `estimate_withdrawal_fee` in a loop via inter-canister calls. Each call permanently removes UTXOs from `s.available_utxos`. Once the set is sufficiently depleted, all subsequent `retrieve_btc` and `retrieve_btc_with_approval` calls fail with `NotEnoughFunds` even though the minter holds sufficient Bitcoin on-chain. The minter's internal UTXO accounting diverges from the actual on-chain UTXO set, breaking the conservation invariant. Recovery requires a manual canister upgrade to re-scan and re-populate the UTXO set. This is a **High** impact: application/platform-level DoS on ckBTC withdrawals with concrete user harm — a significant Chain Fusion / ck-token security impact.

## Likelihood Explanation

No privileged access is required. The endpoint has no `check_anonymous_caller` guard and is publicly callable. The attacker only needs to deploy a canister and issue inter-canister calls in a loop. The cost is only cycles. The attack is repeatable and deterministic.

## Recommendation

Replace `mutate_state` with `read_state` and operate on a cloned UTXO set so no mutation of live state occurs:

```rust
#[query]
fn estimate_withdrawal_fee(arg: EstimateFeeArg) -> WithdrawalFee {
    read_state(|s| {
        let mut utxos_clone = s.available_utxos.clone();
        let fee_estimator = IC_CANISTER_RUNTIME.fee_estimator(s);
        let withdrawal_amount = arg.amount.unwrap_or(s.fee_based_retrieve_btc_min_amount);
        ic_ckbtc_minter::estimate_retrieve_btc_fee(
            &mut utxos_clone,
            withdrawal_amount,
            ...
        )
    })
}
```

Alternatively, annotate the endpoint `#[update]` consistent with the ckDoge minter design.

## Proof of Concept

```rust
// Attacker canister
#[update]
async fn drain_ckbtc_utxos(minter: Principal) {
    for _ in 0..1000 {
        let _: (WithdrawalFee,) = ic_cdk::api::call::call(
            minter,
            "estimate_withdrawal_fee",
            (EstimateFeeArg { amount: Some(50_000) },),
        ).await.unwrap();
        // Each inter-canister call runs in replicated mode,
        // permanently removing UTXOs from minter.available_utxos
    }
    // minter.available_utxos is now depleted;
    // all retrieve_btc calls fail with NotEnoughFunds
}
```

A deterministic integration test using PocketIC can confirm this by: (1) populating the minter with a known UTXO set, (2) calling `estimate_withdrawal_fee` via `update_call` from a second canister, (3) asserting that `available_utxos` shrinks after each call, and (4) asserting that a subsequent `retrieve_btc` call fails with `NotEnoughFunds`.
