Audit Report

## Title
Missing Staleness Check on CMC 30-Day Average ICP/XDR Rate in SNS Treasury Valuation - (`rs/sns/governance/token_valuation/src/lib.rs`)

## Summary

`CmcBased30DayMovingAverageXdrsPerIcpClient::get()` calls the CMC's `get_average_icp_xdr_conversion_rate` and extracts only `xdr_permyriad_per_icp`, silently discarding `response.data.timestamp_seconds`. The CMC returns whatever 30-day moving average was last computed with no expiry; if the XRC has been unreachable for an extended period while ICP market price moves significantly, the SNS governance enforces `TransferSnsTreasuryFunds` and `MintSnsTokens` 7-day spending caps against a stale valuation, potentially misclassifying the treasury regime and allowing a larger transfer than the protocol intends.

## Finding Description

**Root cause — `CmcBased30DayMovingAverageXdrsPerIcpClient::get()` discards the CMC timestamp**

In `rs/sns/governance/token_valuation/src/lib.rs` lines 435–458, after decoding the CMC response the code does:

```rust
let xdr_per_icp =
    Decimal::from(response.data.xdr_permyriad_per_icp) * *UNITS_PER_PERMYRIAD;
Ok(xdr_per_icp)
```

`response.data.timestamp_seconds` — which records when the 30-day average was last updated — is never read. The CMC's `get_average_icp_xdr_conversion_rate` (lines 891–912 of `rs/nns/cmc/src/main.rs`) simply returns `state.average_icp_xdr_conversion_rate` as stored, with no freshness check. If the XRC has been unavailable for weeks, that stored average is arbitrarily old.

**Why existing guards are insufficient**

`ProposalsAmountTotalUpperBound::clamp_xdrs_per_icp` (lines 137–140 of `rs/sns/governance/proposals_amount_total_limit/src/lib.rs`) enforces a floor of `MIN_XDRS_PER_ICP = 1`. The code comment explicitly acknowledges this is to prevent "wacky price quotes" from pushing the treasury into the `NoLimit` (small) regime. However, the floor only activates for rates below 1 XDR/ICP — a level the ICP/XDR rate has never historically reached. A stale rate of, e.g., 3 XDR/ICP (above the floor, but well below a current market price of 10+ XDR/ICP) passes through unmodified and can still misclassify the treasury regime.

**Spending cap misclassification**

`treasury_valuation_if_proposal_amount_is_small_enough_or_err` (lines 770–817 of `rs/sns/governance/src/proposal.rs`) calls `assess_treasury_balance`, which calls `try_get_icp_balance_valuation` / `try_get_sns_token_balance_valuation`, which use `new_standard_xdrs_per_icp_client`. The resulting `xdrs_per_icp` feeds directly into `from_valuation_xdr` (lines 116–135 of `proposals_amount_total_limit/src/lib.rs`):

- ≤ 100,000 XDR → `NoLimit` (full treasury transferable)
- ≤ 1,200,000 XDR → `Fraction(0.25)` (25% cap)
- \> 1,200,000 XDR → `Xdr(300_000)` (300,000 XDR cap)

A stale low rate can push a treasury that should be in the medium or large regime into the small (`NoLimit`) regime.

## Impact Explanation

If the XRC is unavailable for a sustained period while ICP price rises significantly, an SNS neuron holder can submit a `TransferSnsTreasuryFunds` proposal for an amount that exceeds the intended 25%/300,000 XDR cap. The validation check at proposal submission time uses the stale rate and incorrectly classifies the treasury as small, returning `NoLimit`. Once adopted by the SNS governance, `perform_transfer_sns_treasury_funds` executes the full transfer. This constitutes a significant SNS governance security impact with concrete protocol harm — the automated spending cap, which is the primary protection against rapid treasury drain, is bypassed. This fits the **Medium** impact tier: the attack requires strict target conditions (extended XRC outage, significant price movement, treasury in the right size range, governance adoption), but the impact — bypassing the SNS treasury spending cap — is meaningful and concrete.

## Likelihood Explanation

The CMC polls the XRC every 5 minutes (`REFRESH_RATE_INTERVAL_SECONDS`). The rate consumed by SNS governance is a 30-day moving average, so a short XRC outage barely shifts the average. For the stale rate to misclassify the treasury regime, the XRC would need to be unavailable for weeks while ICP price moves substantially. This is a low-frequency but non-zero event (XRC canister upgrades, bugs, or subnet issues). An attacker cannot force the XRC outage but can observe the CMC's `timestamp_seconds` via `get_average_icp_xdr_conversion_rate` and time a proposal submission opportunistically. The attack is repeatable across any SNS whose treasury is near a regime boundary.

## Recommendation

1. In `CmcBased30DayMovingAverageXdrsPerIcpClient::get()` (`rs/sns/governance/token_valuation/src/lib.rs`, after line 456), compare `response.data.timestamp_seconds` against the canister's current time. Return a `ValuationError::new_external(...)` if the rate is older than a configurable threshold (e.g., 48 hours for a 30-day average).

2. Optionally, add a staleness variant to `ValidateExchangeRateError` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` and check `exchange_rate.timestamp` against `now` in `validate_exchange_rate`, so the CMC and NNS governance also reject stale XRC responses at the point of ingestion.

## Proof of Concept

1. Observe that the XRC canister is unavailable (e.g., during a canister upgrade or subnet issue). Confirm via `get_average_icp_xdr_conversion_rate` on the CMC that `timestamp_seconds` has stopped advancing.

2. Wait until ICP market price has risen enough that the stale 30-day average, when applied to the SNS treasury balance, yields a valuation ≤ `MAX_SMALL_TREASURY_SIZE_XDR` (100,000 XDR), while the true current valuation would be in the medium or large regime.

3. Submit a `TransferSnsTreasuryFunds` proposal for the full treasury balance (or an amount exceeding the 25% / 300,000 XDR cap).

4. `validate_and_render_transfer_sns_treasury_funds` → `treasury_valuation_if_proposal_amount_is_small_enough_or_err` → `assess_treasury_balance` → `try_get_icp_balance_valuation` → `CmcBased30DayMovingAverageXdrsPerIcpClient::get()` returns the stale rate.

5. `from_valuation_xdr` classifies the treasury as small (≤ 100,000 XDR at the stale price) and returns `NoLimit`. The proposal passes validation.

6.
