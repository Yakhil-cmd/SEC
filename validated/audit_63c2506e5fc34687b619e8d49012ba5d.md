### Title
Existing SNS Swap Participant Incorrectly Blocked from Increasing ICP Participation When Neuron Basket Limit Is Reached - (File: rs/sns/swap/src/swap.rs)

### Summary

In `refresh_buyer_token_e8s()`, the guard that enforces the `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` ceiling unconditionally adds `+1` to the current participant count, treating every caller as a brand-new participant. When the limit is exactly at the boundary, an already-registered buyer who tries to top up their ICP contribution is rejected with the same error as a new entrant — even though the error message itself states "existing participants may still increase their ICP participation amount."

### Finding Description

`refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` is the public entry point through which any principal can notify the SNS Swap canister of an ICP deposit and register or increase their participation.

At lines 1179–1198, before any lookup of whether the caller is already a buyer, the code performs:

```rust
let num_direct_participants = self.buyers.len() as u64;
// ...
if (num_direct_participants + 1) * num_sns_neurons_per_basket
    > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS
{
    return Err(format!(
        "The swap has reached the maximum number of direct participants \
         ({num_direct_participants}) and does not accept new participants; \
         existing participants may still increase their ICP participation amount. …"
    ));
}
``` [1](#0-0) 

The `+1` is applied unconditionally. The check for whether the caller is a pre-existing buyer only appears later, at line 1275:

```rust
let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
``` [2](#0-1) 

Because `self.buyers.len()` already counts the existing buyer, adding `+1` for them inflates the projected participant count by one extra slot. When `num_direct_participants * num_sns_neurons_per_basket == MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` (i.e., the limit is exactly met), the expression `(num_direct_participants + 1) * num_sns_neurons_per_basket` exceeds the limit and the function returns an error — blocking the existing participant from topping up, contrary to the stated intent in the error message.

The correct fix mirrors the SNS governance `add_neuron_permissions` pattern: only apply the `+1` when the caller is not already present in `self.buyers`. [3](#0-2) 

### Impact Explanation

An existing swap participant who has already transferred ICP to their swap subaccount and registered in `self.buyers` cannot call `refresh_buyer_token_e8s` to increase their contribution once the neuron basket limit is exactly reached. Their newly transferred ICP is stranded in the swap subaccount until the swap closes (committed or aborted), at which point they must use `error_refund_icp` to recover it. This directly contradicts the protocol's documented guarantee that existing participants may always top up. In a popular SNS swap where the basket limit is hit, multiple existing participants could simultaneously lose the ability to increase their stake.

### Likelihood Explanation

The condition requires `num_direct_participants * num_sns_neurons_per_basket == MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` exactly. This is reachable in any SNS swap configured with a large `neuron_basket_count` (the test `test_refresh_buyer_tokens_no_sns_neuron_baskets_available` uses `33_000` to reach the limit with only 3 participants). [4](#0-3) 

Any unprivileged principal can call `refresh_buyer_token_e8s` on the publicly exposed SNS Swap canister. No special role or key is required. The scenario is realistic for high-demand SNS launches.

### Recommendation

Check whether the caller is already in `self.buyers` before deciding whether to add `+1` to the participant count:

```rust
let is_preexisting_buyer = self.buyers.contains_key(&buyer.to_string());
let projected_participants = if is_preexisting_buyer {
    num_direct_participants
} else {
    num_direct_participants + 1
};
if projected_participants * num_sns_neurons_per_basket > MAX_NEURONS_FOR_DIRECT_PARTICIPANTS {
    return Err(/* new-participant-only error */);
}
```

This mirrors the correct pattern already used in SNS governance's `add_neuron_permissions`, where the limit check is conditioned on `existing_permissions.is_none()`.

### Proof of Concept

1. Configure an SNS swap with `neuron_basket_count = 33_000`.
2. Have 3 participants join, reaching `3 * 33_000 = 99_000` neurons (assume `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS = 100_000`; the exact boundary depends on the constant value).
3. Participant 2 (already in `self.buyers`) transfers additional ICP to their swap subaccount and calls `refresh_buyer_token_e8s`.
4. The check evaluates `(3 + 1) * 33_000 = 132_000 > 100_000` → returns error "The swap has reached the maximum number of direct participants … existing participants may still increase their ICP participation amount."
5. Participant 2 is blocked despite being an existing buyer; their newly transferred ICP is stranded. [5](#0-4)

fn test_refresh_buyer_tokens_no_sns_neuron_baskets_available() {
    let user1 = PrincipalId::new_user_test_id(1);
    let user2 = PrincipalId::new_user_test_id(2);
    let user3 = PrincipalId::new_user_test_id(3);
    let user4 = PrincipalId::new_user_test_id(4);

    let mut swap = SwapBuilder::new()
        .with_sns_governance_canister_id(SNS_GOVERNANCE_CANISTER_ID)
        .with_lifecycle(Open)
        .with_swap_start_due(Some(START_TIMESTAMP_SECONDS), Some(END_TIMESTAMP_SECONDS))
        .with_min_participants(1)
        .with_min_max_participant_icp(2 * E8, 40 * E8)
        .with_min_max_direct_participation(5 * E8, 100 * E8)
        .with_sns_tokens(100_000 * E8)
        // An extremely large basket size, so we can reach MAX_NEURONS_FOR_DIRECT_PARTICIPANTS with
        // a relatively small number of participants.
        .with_neuron_basket_count(33_000)
        .with_neurons_fund_participation()
        .build();

    let params = swap.params.unwrap();

    let amount_user1_0 = 5 * E8;
    let amount_user2_0 = 40 * E8;
    let amount_user3_0 = 40 * E8;
    let amount_user4_0 = 99 * E8 - (amount_user2_0 + amount_user3_0);

    // All tokens but one should be already bought up by users 2 to 4 --> 99 Tokens were bought
    buy_token_ok(&mut swap, &user2, &amount_user2_0, &amount_user2_0);
    buy_token_ok(&mut swap, &user3, &amount_user3_0, &amount_user3_0);
    buy_token_ok(&mut swap, &user4, &amount_user4_0, &amount_user4_0);

    // Make sure the 99 tokens were registered
    assert_eq!(
        swap.get_buyers_total().buyers_total,
        amount_user2_0 + amount_user3_0 + amount_user4_0
    );

    // Make sure that only an amount smaller than the minimum amount to be bought per user is available
    assert!(
        params.max_direct_participation_icp_e8s.unwrap() - swap.get_buyers_total().buyers_total
            < params.min_participant_icp_e8s
    );

    // No user that has not participated in the swap yet can buy this one token left
    buy_token_err(
        &mut swap,
        &user1,
        &amount_user1_0,
        "The swap has reached the maximum number of direct participants",
    );

    // The one token should still be left fur purchase
    check_final_conditions(
        &mut swap,
        &user2,
        &amount_user2_0,
        &(params.max_direct_participation_icp_e8s.unwrap() - E8),
    );
