### Title
Global Cycles Minting Rate Limit Can Be Exhausted by a Single Actor, DoSing All Other Users - (File: rs/nns/cmc/src/main.rs)

---

### Summary

The Cycles Minting Canister (CMC) enforces a single global `base_limiter` that tracks all cycles minted across **all users** in a rolling 1-hour window. There is no per-account sub-limit. A single user holding sufficient ICP can exhaust the entire hourly minting quota through legitimate successful top-up or mint-cycles calls, blocking every other user from minting cycles for up to one hour.

---

### Finding Description

The CMC state holds one shared `Limiter` instance:

```rust
pub base_limiter: limiter::Limiter,   // rolling 1-hour window
pub base_cycles_limit: Cycles,        // 150P cycles / hour
``` [1](#0-0) 

Both public minting endpoints converge on `ensure_balance`, which atomically checks and increments this global counter before any cycles are minted:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;
    ...
}
``` [2](#0-1) 

`notify_top_up` reaches `ensure_balance` via `process_top_up` → `deposit_cycles`: [3](#0-2) [4](#0-3) 

`notify_mint_cycles` reaches it via `process_mint_cycles` → `do_mint_cycles`: [5](#0-4) 

The `Limiter::check_and_add_cycles` function rejects any request that would push the rolling total above `base_cycles_limit`, returning an error that propagates as `NotifyError::Refunded { reason: "…please try again later." }`: [6](#0-5) 

Because the limiter is a single global counter with no per-principal tracking, one actor who successfully mints up to the full hourly quota leaves zero capacity for every other user until the rolling window advances.

---

### Impact Explanation

Any user who can acquire ~1 500 ICP (≈ 150 P cycles at the default 100 XDR/ICP rate and 1 T cycles/XDR) can call `notify_top_up` or `notify_mint_cycles` in one or a few transactions, saturating `base_limiter`. For the remainder of the 1-hour rolling window every subsequent minting call from any other principal is rejected with a refund. This makes the cycles-minting service temporarily unavailable to the entire IC user base — canister creation, top-ups, and cycles-ledger deposits all fail — until the window rolls over.

The integration test explicitly confirms this behavior: [7](#0-6) 

---

### Likelihood Explanation

The attack requires no privileged access, no key material, and no governance majority. Any principal who holds or can borrow ~1 500 ICP on the open market can execute it. The attacker does not recover the ICP (it is burned), so there is a real economic cost, but the cost is bounded and predictable. The attack can be repeated every hour. On-chain ICP liquidity is public, so a motivated attacker can plan the acquisition in advance. Likelihood is **medium**: the economic barrier is non-trivial but the attack is fully reachable by an unprivileged ingress sender.

---

### Recommendation

1. **Per-principal sub-limits**: Track minting volume per `caller()` principal within the existing rolling window. Reject a request only when the per-principal quota is exceeded, not the global one.
2. **Reserve-commit pattern**: The `rs/nervous_system/rate_limits` library already implements a reserve-commit pattern that avoids counting failed operations against the limit. Apply the same pattern here so that the global counter is only committed after the downstream call (management canister or cycles ledger) succeeds. [8](#0-7) 

---

### Proof of Concept

1. Attacker acquires ≥ 1 500 ICP on the open market.
2. Attacker transfers 1 500 ICP to the CMC subaccount for their own canister and calls `notify_top_up`. `ensure_balance` mints ≈ 150 P cycles, incrementing `base_limiter` to its ceiling.
3. All subsequent calls to `notify_top_up` or `notify_mint_cycles` by any other principal within the same rolling hour window hit the check:

```rust
if count + cycles_to_mint > limit {
    return Err("More than {} cycles have been minted in the last {} seconds, please try again later.");
}
``` [9](#0-8) 

4. Every other user receives `NotifyError::Refunded { reason: "…try again later" }` for up to one hour.
5. The attacker repeats at the start of each new rolling window.

The existing test `cmc_notify_top_up_rate_limited` confirms that a single large successful top-up can exhaust the limit and block subsequent callers: [10](#0-9)

```rust
    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
    }

//! Rate limits library for nervous system components.
//!
//! This library uses a [Leaky Bucket Algorithm](https://en.wikipedia.org/wiki/Leaky_bucket) to
//! enforce rate limites, which allows for a configurable amount of spikiness around the average
//! limit desired.
//!
//! A reserve-commit pattern is used so that any failure would not affect the rate limit (otherwise
//! a DoS would be possible by deliberately triggering operations that would fail).
//!
//! Example:
//!
//! ```
//! let reservation = get_rate_limiter().try_reserve(now, key, requested_capacity)?;
//! do_something()?; // if this fails, the reservation is dropped without affecting the rate limit
//! get_rate_limiter().commit(now, reservation).expect("Failed to commit reservation"); // Or handle the error by logging. It should not fail unless there the resevation is invalid.
//! ```
//!
//! This pattern is also safe to use in an asynchronous context - if the reservation happens at the
//! first message of the async call, then the entire async call is protected by the rate limiting.
//! Although, in those cases, the user should consider whether the reservation should be committed
//! if the async call fails, depending on whether the rate limiting is trying to protect against the
//! inter-canister call (in which case, the reservation should be committed) or the some other
//! mutations inside the canister (in which case, the reservation should not be committed).
