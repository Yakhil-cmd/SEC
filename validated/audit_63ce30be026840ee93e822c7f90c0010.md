### Title
Missing EIP-7610 Storage Collision Check Allows Attacker to Pre-Poison CREATE Deployment Address — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter explicitly skips the EIP-7610 storage-non-empty collision check for `CREATE` deployments. An attacker can use `CREATE2` to deploy-and-selfdestruct a contract at a victim's predicted `CREATE` address, leaving behind attacker-controlled storage. When the victim later deploys their contract via `CREATE`, ZKsync OS allows the deployment to succeed on top of the poisoned storage, giving the attacker control over the victim contract's initial state and enabling funds theft.

---

### Finding Description

EIP-7610 (included in Prague/Osaka) requires that a `CREATE` deployment must fail if the target address already has non-empty storage. This prevents an attacker from pre-populating storage at a predictable `CREATE` address and then having a victim's contract inherit that state.

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` hook checks for deployment collisions but explicitly omits the storage check:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // ... fail
}
``` [1](#0-0) 

The comment acknowledges the missing check. The test index files confirm this is a known gap — EIP-7610 collision tests are explicitly disabled:

```yaml

      enabled: false

    test_init_collision_create_tx.json:
      enabled: false
      comment: We do not check the storage
``` [2](#0-1) 

The `CREATE` address derivation is standard `keccak256(rlp([deployer, nonce]))`, making the victim's future deployment address fully predictable from public state: [3](#0-2) 

The `CREATE` opcode reads the deployer's current nonce to derive the address: [4](#0-3) 

---

### Impact Explanation

An attacker who pre-populates storage at a victim's predicted `CREATE` address causes the victim's newly deployed contract to inherit attacker-controlled storage slots. If the victim's contract uses storage for:
- Initialization flags (e.g., `initialized = false` bypassed to `true`)
- Ownership records (e.g., `owner` slot pre-set to attacker's address)
- Token balances or allowances

…the attacker gains control over the contract's logic from the moment of deployment, enabling direct theft of any funds sent to or held by the contract.

---

### Likelihood Explanation

The attack is realistic and requires only public information:
1. The victim's address and current nonce are both publicly readable on-chain.
2. The attacker computes `keccak256(rlp([victim_address, victim_nonce]))` to get the target address.
3. The attacker uses `CREATE2` (with a chosen salt) to deploy a contract at that exact address, writes to storage, then calls `SELFDESTRUCT`.
4. The victim deploys via `CREATE` — ZKsync OS allows it to succeed on top of the poisoned storage.

`SELFDESTRUCT` is implemented in ZKsync OS: [5](#0-4) 

The attacker only needs to front-run the victim's deployment transaction, which is straightforward on any sequencer-based L2.

---

### Recommendation

Implement the EIP-7610 storage-non-empty check in `before_executing_frame`. Before allowing a `CREATE` constructor to execute, verify that the target address has no non-zero storage slots. If non-empty storage is detected, treat it as a collision (return `CreateCollision` error, burn gas) consistent with the existing code path at line 429–442. [1](#0-0) 

---

### Proof of Concept

1. Victim `V` has nonce `N`. Attacker computes target = `keccak256(rlp([V, N]))`.
2. Attacker deploys via `CREATE2` to `target`, writing `storage[0] = attacker_address`.
3. Attacker calls `SELFDESTRUCT` on the deployed contract. Code and nonce are cleared; storage at `target` remains.
4. Victim sends a `CREATE` deployment transaction (nonce `N`). ZKsync OS derives the same `target` address.
5. `before_executing_frame` checks: `code_len == 0` (cleared by selfdestruct) and `nonce == 0` (cleared) → **no collision detected**.
6. Victim's constructor executes. The contract reads `storage[0]` expecting zero (uninitialized), but finds `attacker_address`.
7. If the contract uses `storage[0]` as `owner`, the attacker is now the owner and can drain any funds.

        _resources: &mut <S as SystemTypes>::Resources,
        deployer_address: &<S::IOTypes as SystemIOTypesConfig>::Address,
        deployer_nonce: u64,
    ) -> Result<<S::IOTypes as SystemIOTypesConfig>::Address, EvmSubsystemError> {
        use crypto::sha3::{Digest, Keccak256};
        let mut buffer = [0u8; crate::utils::MAX_CREATE_RLP_ENCODING_LEN];
        let encoding_it = crate::utils::create_quasi_rlp(deployer_address, deployer_nonce);
        let encoding_len = ExactSizeIterator::len(&encoding_it);
        for (dst, src) in buffer.iter_mut().zip(encoding_it) {
            *dst = src;
        }

        #[allow(deprecated)]
        let new_address =
            B160::try_from_be_slice(&new_address.as_slice()[12..]).expect("must create address");

    pub fn selfdestruct(
        &mut self,
        system: &mut System<S>,
        tracer: &mut impl Tracer<S>,
    ) -> InstructionResult {
        self.gas
            .spend_gas_and_native(gas_constants::SELFDESTRUCT, SELFDESTRUCT_NATIVE_COST)?;

        if self.is_static_frame() {
            return Err(EvmError::StateChangeDuringStaticCall.into());
        }

        let beneficiary = u256_to_b160(self.stack.pop_1()?);

        let amount_transferred = system
            .io
            .mark_for_deconstruction(
                THIS_EE_TYPE,
                self.gas.resources_mut(),
                &self.address,
                &beneficiary,
            )
            .map_err(wrap_error!())?;

        tracer.evm_tracer().on_selfdestruct(
            beneficiary,
            amount_transferred,
            &InterpreterExternal::new_from(&self, system),
        );

        Err(ExitCode::SelfDestruct)
