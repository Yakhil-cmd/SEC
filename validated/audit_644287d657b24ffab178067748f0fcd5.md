### Title
`fee_based_minimum_withdrawal_amount` 2-Input Assumption Causes Unreimbursed `AmountTooLow` Finalization — (`rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

### Summary

`fee_based_minimum_withdrawal_amount` computes the minimum withdrawal amount using a hardcoded `PER_REQUEST_SIZE_BOUND = 374` bytes (2-input/2-output P2PKH) and `PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu. The actual minter fee for 2 inputs is `326_000_000` koinu — a **1000× underestimate** in the bound. When the minter's UTXO set is fragmented and the actual transaction requires many inputs, the real minter fee can exceed the withdrawal amount, triggering `BuildTxError::AmountTooLow`. Unlike `BuildTxError::InvalidTransaction(TooManyInputs)`, the `AmountTooLow` path **does not reimburse** the user — their ckDOGE is burned permanently.

### Finding Description

**Step 1 — Minimum amount computation (underestimates fee):** [1](#0-0) 

`PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu, but `evaluate_minter_fee(2, 2)` returns `146_000_000 * 2 + 4_000_000 * 2 + 26_000_000 = 326_000_000` koinu — 1000× larger. The formula's minter-fee term is negligible relative to the `min_withdrawal_amount_increment` (25 DOGE = 2,500,000,000 koinu), so the minimum always rounds to `retrieve_doge_min_amount` (50 DOGE) at normal fee rates, regardless of how many inputs the actual transaction will need.

**Step 2 — Actual fee check uses real input count:** [2](#0-1) 

`evaluate_minter_fee` is called with the actual `num_inputs`. For 35 inputs: `146_000_000 × 35 = 5_110_000_000` koinu = **51.1 DOGE > 50 DOGE minimum**. The check `if fee + minter_fee > amount` fires → `BuildTxError::AmountTooLow`.

**Step 3 — `AmountTooLow` path has no reimbursement:** [3](#0-2) 

The `AmountTooLow` branch calls `remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow, ...)` — no call to `reimburse_canceled_requests`. The burn is permanent.

**Contrast with `InvalidTransaction` path (which does reimburse):** [4](#0-3) 

`TooManyInputs` calls `reimburse_canceled_requests`. The asymmetry is the root cause.

**Step 4 — `TooManyInputs` check precedes `AmountTooLow` check:** [5](#0-4) 

If `num_inputs > max_num_inputs_in_transaction`, the user gets reimbursed. But if `num_inputs` is between 35 and `max_num_inputs_in_transaction`, the user hits `AmountTooLow` with no reimbursement.

### Impact Explanation

A user submitting a withdrawal at exactly `fee_based_minimum_withdrawal_amount` (50 DOGE at normal fee rates) when the minter holds 35+ small UTXOs (e.g., 35 UTXOs × 1.43 DOGE each) will have their ckDOGE burned with no reimbursement. The minter fee alone for 35 inputs (51.1 DOGE) exceeds the 50 DOGE withdrawal amount. The `RetrieveDogeStatus` becomes `AmountTooLow` permanently. [6](#0-5) 

The `Reimbursed` and `WillReimburse` statuses exist in the protocol but are never reached via the `AmountTooLow` finalization path.

### Likelihood Explanation

- UTXO fragmentation occurs naturally when many users deposit small amounts.
- An attacker can deliberately fragment the minter's UTXO set by depositing many small UTXOs (at `deposit_doge_min_amount` each), then submitting a withdrawal at the minimum amount.
- The minter's UTXO consolidation mechanism may not run before the withdrawal is processed.
- The user can observe the minter's available UTXOs via `get_known_utxos` to time the attack.
- The attacker loses the withdrawal amount (50 DOGE) but can cause the same loss to any other user who withdraws at the minimum while the UTXO set is fragmented.

### Recommendation

1. **Fix `PER_REQUEST_MINTER_FEE_BOUND`**: Change `326_000` to `326_000_000` koinu to match the actual 2-input minter fee.
2. **Account for worst-case input count**: Use a conservative upper bound on the number of inputs (e.g., based on `deposit_doge_min_amount` and `retrieve_doge_min_amount`) when computing the minimum withdrawal amount.
3. **Add reimbursement for `AmountTooLow`**: Mirror the `InvalidTransaction` path — reimburse the user (minus a processing fee) when `AmountTooLow` is triggered post-acceptance, since the amount passed the upfront check.
4. **Add invariant test**: Proptest with fragmented UTXO sets asserting no `AmountTooLow` finalization for amounts ≥ `fee_based_minimum_withdrawal_amount`.

### Proof of Concept

```
1. Minter has 35 UTXOs each worth 1.43 DOGE (total 50.05 DOGE).
2. fee_based_minimum_withdrawal_amount() returns 50 DOGE (2-input assumption).
3. User calls retrieve_doge_with_approval(50 DOGE) — accepted.
4. Greedy selects all 35 UTXOs to cover 50 DOGE.
5. evaluate_minter_fee(35, 2) = 146_000_000 * 35 + ... = 5_110_000_000 koinu = 51.1 DOGE.
6. 51.1 DOGE > 50 DOGE → BuildTxError::AmountTooLow.
7. remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow) — no reimbursement.
8. User's 50 DOGE ckDOGE is permanently burned.
``` [7](#0-6) [8](#0-7)

    fn fee_based_minimum_withdrawal_amount(&self, median_fee_rate: FeeRate) -> u64 {
        match self.network {
            Network::Mainnet => {
                //in Koinu
                const PER_REQUEST_RBF_BOUND: u64 = 374_000;
                // in Bytes
                // Size of a typical transaction made by the minter,
                // which is a P2PKH transaction with 2 inputs and 2 outputs
                const PER_REQUEST_SIZE_BOUND: u64 = 374;
                //in Koinu
                const PER_REQUEST_MINTER_FEE_BOUND: u64 = 326_000;

                let min_withdrawal_amount_increment = self.retrieve_doge_min_amount >> 1;
                ((PER_REQUEST_RBF_BOUND
                    + median_fee_rate.fee_ceil(PER_REQUEST_SIZE_BOUND)
                    + PER_REQUEST_MINTER_FEE_BOUND)
                    / min_withdrawal_amount_increment)
                    * min_withdrawal_amount_increment
                    + self.retrieve_doge_min_amount
            }
            Network::Regtest => self.retrieve_doge_min_amount,
        }
    }

    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
