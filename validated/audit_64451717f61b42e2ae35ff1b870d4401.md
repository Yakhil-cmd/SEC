Audit Report

## Title
Unprivileged Caller Can Abort In-Progress SNS Upgrade via `fail_stuck_upgrade_in_progress` - (File: `rs/sns/governance/canister/canister.rs`)

## Summary
The `fail_stuck_upgrade_in_progress` update endpoint in the SNS Governance canister is registered as a plain `#[update]` with no caller identity check. Any external principal can invoke it once `now > pending_version.mark_failed_at_seconds` (5 minutes after upgrade kick-off) to unconditionally clear `pending_version` and mark the associated upgrade proposal as `ExternalFailure`, regardless of whether the underlying canister upgrade actually succeeded. This allows any unprivileged actor to repeatedly abort SNS upgrade proposals at negligible cost.

## Finding Description
The canister endpoint at `rs/sns/governance/canister/canister.rs` lines 527–535 is declared `#[update]` with no `caller()` check and no `inspect_message` guard anywhere in the file:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(
        sns_gov_pb::FailStuckUpgradeInProgressRequest::from(request),
    ))
}
```

The governance logic at `rs/sns/governance/src/governance.rs` lines 6342–6358 checks only `now > pending_version.mark_failed_at_seconds` before calling `complete_sns_upgrade_to_next_version` with `deployed_version: None` and `Status::ExternalFailure`. `complete_sns_upgrade_to_next_version` (lines 6309, 6305–6306) unconditionally sets `self.proto.pending_version = None` and calls `set_proposal_execution_status(proposal_id, Err(...))`, marking the proposal failed. Because `deployed_version: None` is passed, `self.proto.deployed_version` is never updated to the new version.

The deadline is set to only 5 minutes after upgrade kick-off (`self.env.now() + 5 * 60`, line 2896). The public `get_running_sns_version` query (lines 503–524) exposes `mark_failed_at_seconds` to any observer, giving an attacker a precise trigger time. The test at `rs/sns/governance/src/governance/fail_stuck_upgrade_in_progress_tests.rs` lines 216–298 confirms that when `now > mark_failed_at_seconds`, calling the function clears `pending_version` and sets the proposal to failed with `ErrorType::External`.

No existing guard prevents this: there is no `inspect_message`, no `caller()` comparison, and no neuron/proposal-based authorization in the call path.

## Impact Explanation
This is a **High** severity SNS governance impact. Any unprivileged principal can force every SNS `UpgradeSnsToNextVersion` proposal into the `Failed` state after 5 minutes, even when the underlying canister upgrade succeeded. Because `deployed_version` is not updated, the SNS governance canister retains a stale view of the deployed version. Future `UpgradeSnsToNextVersion` proposals will compute upgrade paths from the stale `deployed_version`, potentially re-attempting already-applied upgrades or failing validation. The attack is cheap, repeatable on every upgrade attempt, and requires no tokens, neurons, or special privileges — matching the allowed impact class: *Significant SNS security impact with concrete user or protocol harm*.

## Likelihood Explanation
The attack window opens automatically 5 minutes after every adopted SNS upgrade proposal. The attacker only needs to poll the public `get_running_sns_version` query to read `mark_failed_at_seconds`, then submit a single ingress call immediately after the deadline. No stake, governance participation, or privileged access is required. The attack is deterministic and can be repeated for every future upgrade proposal.

## Recommendation
Add a caller check to `fail_stuck_upgrade_in_progress` in `rs/sns/governance/canister/canister.rs`. The function should only be callable by the SNS governance canister itself (`canister_self()`), consistent with how other self-invoked recovery operations are gated. Alternatively, restrict it to the SNS root canister or require a governance proposal action, mirroring the access control pattern used by `set_mode` (which passes `caller()` into the governance layer for validation). The `mark_failed_at_seconds` deadline should also be extended to give the periodic `check_upgrade_status` task sufficient time to confirm success before the recovery window opens.

## Proof of Concept
The existing unit test in `rs/sns/governance/src/governance/fail_stuck_upgrade_in_progress_tests.rs` (`test_fails_proposal_and_removes_upgrade_if_upgrade_attempt_is_expired`, lines 217–298) already demonstrates the full effect: with `env.now = UPGRADE_DEADLINE_TIMESTAMP_SECONDS + 1`, calling `fail_stuck_upgrade_in_progress` with no caller check clears `pending_version` and sets the proposal to `ErrorType::External`. To prove the missing access control, a PocketIC integration test should:
1. Deploy an SNS governance canister.
2. Adopt an `UpgradeSnsToNextVersion` proposal, triggering `pending_version` with a 5-minute deadline.
3. Advance time past `mark_failed_at_seconds`.
4. Call `fail_stuck_upgrade_in_progress` from an anonymous or arbitrary principal (not governance/root).
5. Assert the proposal is in `Failed` state and `deployed_version` remains at the old version.
6. Assert a subsequent `UpgradeSnsToNextVersion` proposal computes an incorrect upgrade path due to stale `deployed_version`.
