Audit Report

## Title
`sender_info` Canister Signature Signed Content Contains No Request-Binding Context, Enabling Indefinite Attestation Replay — (`rs/validator/src/ingress_validation.rs`)

## Summary

The `SenderInfoContent` signable type wraps only the raw `info` bytes under the domain separator `ic-sender-info`. The canister signature over this content is verified without binding it to any request-specific context (`ingress_expiry`, `MessageId`, or sender principal). Because IC canister signatures are persistent Merkle proofs against a historically certified state root, a once-issued `{info, sig}` pair remains cryptographically valid indefinitely, even after the signing canister has updated its state to revoke the attestation. Any user who holds a valid `{info, signer, sig}` triple can attach it to an unlimited number of future ingress messages with fresh `ingress_expiry` values, bypassing any attribute-based access control enforced by target canisters.

## Finding Description

**Root cause — signed content carries no time-bound context.**

`SenderInfoContent` is defined as a bare wrapper over a byte slice:

```rust
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);   // only the raw info blob
    }
}
```

The `SignatureDomain` prepends only the literal `ic-sender-info`:

```rust
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
```

The signed bytes are therefore exactly `\x0Eic-sender-info || info_bytes`. They contain no `ingress_expiry`, no `MessageId`, and no sender principal.

**Validator constructs and verifies only this content.**

`verify_sender_info_canister_sig` constructs `SenderInfoContent(&sender_info.info)` and verifies the canister signature against it. No expiry or binding check is performed anywhere in `validate_sender_info`:

```rust
let sender_info_content = SenderInfoContent(&sender_info.info);
let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));
verify_canister_sig_with_fallback!(validator, &canister_sig, &sender_info_content, ...);
```

**Why the outer envelope does not mitigate this.**

The `sender_info` fields (`info`, `signer`, `sig`) are included in the representation-independent hash that forms the `MessageId`, so the outer envelope signature does bind the specific `{info, signer, sig}` triple to one particular message. However, this does not prevent replay: the user simply constructs a new message M2 with a fresh `ingress_expiry` but the same `{info, signer, sig}`. M2 has a different `MessageId` (because `ingress_expiry` changed), so the outer envelope signature is freshly computed over M2. The `sender_info.sig` — the canister signature over `SenderInfoContent(info_bytes)` — is identical and still passes verification.

**Why canister signature revocation does not work.**

IC canister signatures are Merkle witnesses against a historically certified state root. The subnet's threshold signature over the old state root remains permanently valid. A verifier that accepts any validly certified state root (which is the standard IC canister signature verification model) will accept the old witness even after the signing canister has removed the corresponding entry from its current state tree. There is no protocol mechanism to invalidate a previously issued canister signature.

**Cross-user replay is constrained (claim partially overstated).**

The `sender_pubkey` in the envelope encodes `(canister_id, seed)`. The canister signature witness is at path `sig/<seed>/hash(message)`. A different user with a different seed would require a witness at a different path, so the same `sig` cannot be used by a colluding party with a different `sender_pubkey`. The cross-user replay vector described in the report is not directly exploitable. The temporal replay by the same user is the valid attack path.

## Impact Explanation

Any canister that reads `sender_info` via `msg_caller_info` and uses it for access-control decisions (e.g., KYC status, role membership, age verification) is exposed to stale-attestation replay. If the signing canister (e.g., Internet Identity) later determines that the user's attributes have changed or should be revoked, it has no protocol-level mechanism to invalidate the previously issued `{info, sig}` pair. The user can continue presenting the old attestation in every new ingress message they send, indefinitely bypassing attribute-based access control. This constitutes a **significant Internet Identity / identity-provider security impact with concrete user and protocol harm**, matching the High ($2,000–$10,000) impact class.

## Likelihood Explanation

The `sender_info` feature is explicitly designed for use by identity providers such as Internet Identity. Any canister that relies on `sender_info` for authorization — the primary intended use case — is immediately affected. An attacker needs only to have legitimately obtained a `sender_info` attestation once; no privileged access, key compromise, or network-level attack is required. The attack is repeatable without limit and requires no special tooling beyond constructing standard ingress messages.

## Recommendation

Bind the `sender_info` canister signature to request-specific context so that the attestation automatically expires. At minimum, include the `ingress_expiry` of the outer envelope in the signed content:

```
\x0Eic-sender-info || ingress_expiry_nanos (8 bytes, big-endian) || info_bytes
```

Ideally, include the full `MessageId` (which commits to sender, canister, method, args, expiry, and nonce) to make each attestation single-use:

```
\x0Eic-sender-info || message_id (32 bytes) || info_bytes
```

`SenderInfoContent` should be changed to carry the binding context alongside the `info` bytes, and `verify_sender_info_canister_sig` should pass the relevant request fields when constructing it.

## Proof of Concept

1. Deploy a signing canister S that implements IC canister signatures. Have S certify the path `sig/<seed>/hash(\x0Eic-sender-info || b"role:admin")` in its state tree.
2. User U (whose `sender_pubkey` encodes `(S, seed)`) constructs ingress message M1 to canister C with `sender_info = {info: b"role:admin", signer: S, sig: <witness>}` and `ingress_expiry = T1`. Sign M1 with U's key. The replica accepts it; C grants admin access.
3. S updates its state to remove the certified path (simulating revocation). S's current certified state no longer contains the witness.
4. U constructs a new ingress message M2 to C with `ingress_expiry = T2 > T1` (fresh, within `MAX_INGRESS_TTL`) but the identical `{info: b"role:admin", signer: S, sig: <same witness>}`. Sign M2 with U's key.
5. The replica's `verify_sender_info_canister_sig` verifies the canister signature over `SenderInfoContent(b"role:admin")` against the old (but still validly certified) state root embedded in the witness. Verification succeeds. C again grants admin access, contrary to S's intent.

A deterministic integration test using PocketIC can reproduce this by: (a) deploying a canister-signature canister, (b) capturing the witness at a specific certified height, (c) advancing the canister's state to remove the entry, and (d) confirming that the old witness still passes `verify_sender_info_canister_sig` when replayed in a new request with a fresh `ingress_expiry`.
