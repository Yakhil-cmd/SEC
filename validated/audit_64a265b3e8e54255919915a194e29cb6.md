### Title
Zero Native Resource Cost for User-Controlled Blake2F Round Loop Enables Prover Resource Exhaustion — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_152/impls.rs`)

### Summary

The `Blake2FPrecompile::execute` implementation hardcodes `cost_native = 0` while accepting a user-controlled `num_rounds` value (up to `u32::MAX = 4,294,967,295`). EVM gas is charged correctly at 1 gas per round, but the native resource — which models RISC-V proving cycles — is never charged regardless of how many rounds are requested. The RISC-V prover must still execute every round of the mixing function loop, creating a forward/proving divergence: the forward execution (x86) completes within EVM gas limits, but the prover is forced to execute computation whose actual RISC-V cycle cost is entirely unaccounted for in the native resource budget.

### Finding Description

**Root cause — `impls.rs` lines 66–72:**

```rust
let num_rounds = u32::from_be_bytes(input.as_chunks::<4>().0[0]);
let cost_ergs = Ergs(((num_rounds as u64) * GAS_PER_ROUND) * ERGS_PER_GAS);
let cost_native = 0;                          // ← hardcoded zero
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(cost_native),
))?;
```

After charging (with zero native cost), the code calls:

```rust
round_function_for_num_rounds(&mut extended_state, &message_block, num_rounds as usize);
```

Which in `mixing_function.rs` lines 144–148 executes:

```rust
for i in 0..num_rounds {
    let sigma = &SIGMAS[i % 10];
    mixing_function(extended_state, message_block, sigma);
}
```

Each `mixing_function` call invokes 8 `g_function` calls, each performing 8 arithmetic operations — a non-trivial per-round cost in RISC-V cycles.

**The correct native cost** is defined in `basic_system/src/cost_constants.rs`:
- `BLAKE2S_ROUND_NATIVE_COST = 340` native units per round
- `BLAKE2S_BASE_NATIVE_COST = 800`

With a 30M gas transaction, an attacker can request 30M rounds of Blake2F. The correct native cost would be `30,000,000 × 340 = 10,200,000,000` native units — approximately 30% of `MAX_NATIVE_COMPUTATIONAL = 1 << 35 = 34,359,738,368`. Since `cost_native = 0`, this entire proving workload is invisible to the native resource accounting system.

**Attacker-controlled entry path:**
1. Attacker submits an EVM transaction calling address `0x0009` (Blake2F precompile, `BLAKE_HOOK_ADDRESS_LOW`)
2. Calldata is exactly 213 bytes (`INPUT_LEN`) with the first 4 bytes encoding `num_rounds = u32::MAX` or any large value bounded only by the transaction's EVM gas limit
3. EVM gas is charged and depleted normally; native resource is not charged at all
4. The RISC-V prover executes all rounds, consuming actual cycles far beyond what the native resource budget accounts for

**Why existing checks fail:**
- The `resources.charge(...)` call at line 69 succeeds because `cost_native = 0` — it never returns an out-of-native error
- The EVM gas check bounds the maximum rounds to `gas_limit`, but does not protect the native resource budget
- `MAX_NATIVE_COMPUTATIONAL` is enforced only against the tracked native resource counter, which Blake2F never increments

### Impact Explanation

**Forward/proving divergence (critical):** The forward execution (x86 sequencer) completes the transaction successfully — EVM gas is consumed, the precompile returns a result. The RISC-V prover must then re-execute the same transaction and prove every Blake2F round. Because the native resource was not charged, the prover's actual cycle consumption can exceed `MAX_NATIVE_COMPUTATIONAL` without any in-execution guard triggering. This causes the prover to fail on a block that the sequencer accepted as valid — a state transition that cannot be proven.

**Block-level DoS:** A single transaction with maximum gas limit calling Blake2F with maximum rounds can force the prover to spend ~30% of its entire native budget on one precompile call. Multiple such transactions in a block can make the block unprovable.

**Attacker requirements:** None beyond the ability to submit a standard EVM transaction. No privileged role, no governance access, no validator status required.

### Likelihood Explanation

The `Blake2FPrecompile` in `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_152/` is a production implementation (not a mock). The `initialize_eip_152` function exists to register it. Once activated (EIP-152 is a standard Ethereum precompile), any unprivileged user can trigger this path with a single crafted transaction. The bug is trivially reproducible: set the first 4 bytes of the 213-byte input to `0xFF 0xFF 0xFF 0xFF` and provide any valid remaining state bytes.

### Recommendation

Replace the hardcoded `cost_native = 0` with a proportional native cost based on `num_rounds`:

```rust
let cost_native = (num_rounds as u64)
    .saturating_mul(BLAKE2S_ROUND_NATIVE_COST)
    .saturating_add(BLAKE2S_BASE_NATIVE_COST);
```

Where `BLAKE2S_ROUND_NATIVE_COST = 340` and `BLAKE2S_BASE_NATIVE_COST = 800` are already defined in `basic_system/src/cost_constants.rs`. This mirrors the pattern used by SHA-256 (`sha256.rs` line 49) and other precompiles that charge both ergs and native proportionally to input size/rounds.

### Proof of Concept

Craft a transaction calling address `0x0000000000000000000000000000000000000009` with exactly 213 bytes of calldata:
- Bytes 0–3: `0xFF 0xFF 0xFF 0xFF` (num_rounds = 4,294,967,295, bounded in practice by gas limit)
- Bytes 4–67: any 64 bytes (h state)
- Bytes 68–195: any 128 bytes (message block m)
- Bytes 196–203: any 8 bytes (t0)
- Bytes 204–211: any 8 bytes (t1)
- Byte 212: `0x00` or `0x01` (finalization flag)

With a gas limit of 30,000,000, `num_rounds` is effectively capped at 30,000,000. The EVM gas is consumed. The native resource counter remains unchanged. The RISC-V prover must execute 30,000,000 × 8 `g_function` calls — a workload of `10,200,000,000` native units that is entirely invisible to the resource accounting system.
