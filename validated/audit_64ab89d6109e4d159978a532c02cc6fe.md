### Title
ICRC-2 `icrc2_approve` Without `expected_allowance` Enables Allowance Race Condition ‚Äî (File: `rs/ledger_suite/common/ledger_core/src/approvals.rs`)

---

### Summary

The IC ICRC-2 ledger implementation exposes `icrc2_approve` with an **optional** `expected_allowance` field. When callers omit this field (pass `None`), the ledger blindly overwrites the existing allowance without any CAS-style check. This is the direct analog of the ERC20 `approve`/`transferFrom` race condition: a spender who observes a pending `icrc2_approve` in the IC ingress pool can submit `icrc2_transfer_from` to consume the old allowance before the new one takes effect, then consume the new allowance as well ‚Äî extracting more tokens than the approver intended.

---

### Finding Description

`ApproveArgs.expected_allowance` is declared as `Option<Nat>` with `#[serde(default)]`: [1](#0-0) 

When `None` is passed, the core `AllowanceTable::approve` function skips the current-allowance check entirely and unconditionally overwrites: [2](#0-1) [3](#0-2) 

The `None` branch at line 253‚Äì261 only rejects a non-zero `expected_allowance` when no allowance exists; it does **not** block the overwrite when an allowance already exists and `expected_allowance` is `None`. The `Some(old_allowance)` branch at line 278‚Äì292 similarly skips the check when `expected_allowance` is `None`.

Both the ICRC-1 ledger and the ICP ledger expose this path: [4](#0-3) [5](#0-4) 

Production callers omit `expected_allowance`. The SNS governance `approve_treasury_manager` explicitly passes `None` with a comment that incorrectly dismisses the double-spend risk: [6](#0-5) 

The comment states "there is no risk of double spending" ‚Äî but this only means the approve call itself will not fail; it does not prevent a spender from consuming both the old and new allowance.

The ckETH minter approval flow similarly omits `expected_allowance`: [7](#0-6) 

---

### Impact Explanation

A spender (Bob) who holds an existing allowance from Alice can:

1. Observe Alice's pending `icrc2_approve` (changing his allowance from X to Y) in the IC ingress pool ‚Äî the IC's P2P gossip layer makes pending ingress messages observable to replica nodes and monitoring infrastructure.
2. Submit `icrc2_transfer_from` for X tokens before Alice's approve is executed.
3. Because the IC does not guarantee ordering between ingress messages from different principals to the same canister, Bob's transfer may execute first.
4. After Alice's approve executes (setting allowance to Y), Bob submits another `icrc2_transfer_from` for Y tokens.
5. Total extracted: X + Y instead of Y.

This is a **ledger conservation bug**: the total tokens transferred out of Alice's account exceed what she authorized.

---

### Likelihood Explanation

**Medium-low.** The attack requires:
- The spender to have an existing non-zero allowance.
- The approver to call `icrc2_approve` without `expected_allowance` to change (not revoke) an existing allowance.
- The spender to observe the pending ingress message and react within the ~1‚Äì2 second IC block window.

The IC's ingress pool is observable by anyone running a replica node or monitoring P2P gossip ‚Äî this is not a privileged position requiring admin keys or governance majority. The ordering between messages from different principals to the same canister is not guaranteed by the IC protocol: [8](#0-7) 

The round-robin scheduling means Alice's approve and Bob's transfer_from, if submitted in the same round, have non-deterministic relative ordering. A malicious block maker (one of the subnet nodes) can deterministically reorder them.

---

### Recommendation

1. **Enforce `expected_allowance` in high-value callers**: Any caller that changes an existing non-zero allowance should supply `expected_allowance` equal to the current allowance. This turns the operation into a compare-and-swap, causing it to fail if the spender raced ahead.

2. **Fix the misleading comment in SNS governance**: The comment at `rs/sns/governance/src/extensions.rs:791‚Äì792` incorrectly states there is "no risk of double spending" when `expected_allowance` is `None`. This is false when the spender can observe and front-run the approve. The comment should be corrected and the call should use `expected_allowance`.

3. **Consider a two-step revoke-then-approve pattern**: Callers that cannot use `expected_allowance` (e.g., because they don't know the current allowance) should first set the allowance to 0, then set it to the new value ‚Äî analogous to the ERC20 zero-first mitigation.

---

### Proof of Concept

```
Setup:
  Alice has 300 tokens.
  Alice calls icrc2_approve(spender=Bob, amount=100, expected_allowance=None)
  ‚Üí Bob's allowance = 100

Attack:
  Alice submits icrc2_approve(spender=Bob, amount=150, expected_allowance=None)
  [This is now visible in the IC ingress pool]

  Bob observes Alice's pending approve and immediately submits:
    icrc2_transfer_from(from=Alice, to=Bob, amount=100)

  Round N execution order (non-deterministic between different principals):
    [Bob's transfer_from executes first]  ‚Üí Bob receives 100, Alice balance = 190
    [Alice's approve executes]            ‚Üí Bob's allowance = 150

  Bob then submits:
    icrc2_transfer_from(from=Alice, to=Bob, amount=150)
    ‚Üí Bob receives 150, Alice balance = 30

Result: Bob extracted 250 tokens; Alice intended to authorize only 150.
```

The root cause is confirmed at: [3](#0-2) 

When `expected_allowance` is `None`, the `if let Some(expected_allowance) = expected_allowance` guard at line 279 is never entered, and the allowance is unconditionally overwritten at line 300‚Äì307 ‚Äî with no protection against the race.
