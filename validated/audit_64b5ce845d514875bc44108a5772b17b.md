Audit Report

## Title
`sender_info.sig` Included in `MessageId` Hash Enables Deduplication Bypass via Distinct Valid Canister Signatures — (`rs/types/types/src/messages/http.rs`, `rs/types/types/src/messages/ingress_messages.rs`)

## Summary

`representation_independent_hash_call_or_query` includes the `sender_info.sig` field in the hash that produces the `MessageId`. Because the IC canister signature scheme produces structurally distinct byte strings for certificates obtained at different block heights (even over the same message), an attacker who controls a canister can submit two logically identical ingress messages carrying different valid `sender_info.sig` values. Each message receives a distinct `MessageId`, both pass `validate_sender_info`, and all deduplication guards — which key exclusively on `MessageId` — are bypassed, resulting in double execution of the same logical call.

## Finding Description

**Root cause — `sig` in the hash:**

`representation_independent_hash_call_or_query` in `rs/types/types/src/messages/http.rs` inserts the full `sender_info` sub-map, including `sig`, into the hash:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, sig }) = sender_info {
    map.insert(
        "sender_info",
        Map(btreemap! {
            "info"   => Bytes(info),
            "signer" => Bytes(signer),
            "sig"    => Bytes(sig),   // ← authentication proof in the content hash
        }),
    );
}
``` [1](#0-0) 

`SignedIngressContent::id()` passes `sender_info.sig` directly into this function: [2](#0-1) 

This is inconsistent with the existing design decision to exclude the envelope-level `sender_sig` from the `MessageId`, as confirmed by the `message_id_icf_reference` test (which passes `None` for `sender_info` and asserts that different envelope signatures produce the same `MessageId`): [3](#0-2) 

**Two distinct valid canister signatures over the same message are trivially obtainable:**

In `rs/crypto/iccsa/src/api.rs`, a canister signature is a certificate proving that `certified_data == hash(seed || message)` at a specific block height. The certificate embeds the subnet's threshold signature over the state root at that height: [4](#0-3) 

Two certificates obtained at heights H1 and H2 (with the same `certified_data`) are different byte strings — `sig1 ≠ sig2` — yet both verify correctly. The canister does not need to change its `certified_data` between the two requests.

**`validate_sender_info` accepts both:**

`verify_sender_info_canister_sig` verifies the signature against `SenderInfoContent(&sender_info.info)` and the canister public key. Both `sig1` and `sig2` verify correctly against the same content and key, so both pass: [5](#0-4) 

**All deduplication guards are keyed on `MessageId` only:**

- `ValidSetRuleImpl::is_duplicate` checks `state.get_ingress_status(&msg.content().id())`: [6](#0-5) 

- The ingress selector checks `past_ingress_set.contains(&ingress_id)` where `ingress_id` wraps the same `MessageId`: [7](#0-6) 

- The ingress handler checks `ingress_message_status(&ingress_object.message_id)`: [8](#0-7) 

Since `sig1 ≠ sig2` produces `MessageId1 ≠ MessageId2`, none of these guards detect the duplicate.

**`sender_info` is not gated behind any feature flag or registry setting** — no `cfg` attribute or registry key guards its use in the production code path. [9](#0-8) 

## Impact Explanation

An attacker who controls a canister can cause double execution of any ingress message they submit. Concretely:

- **Ledger double-spend**: Submit a transfer call twice with `sig1` and `sig2`. If both are inducted before the first execution settles the balance, both execute, draining more funds than the account holds. This constitutes illegal token transfer / ledger integrity violation.
- **Governance double-vote**: Submit a governance vote twice; both are inducted and counted, violating the one-vote-per-principal invariant.
- **General at-most-once violation**: Any state-changing call (SNS, ckBTC, ckETH, ICRC ledger) is subject to double execution.

This matches the **High** impact category: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." It may reach **Critical** for large-value ledger transfers exceeding $1M.

## Likelihood Explanation

Any principal who can deploy a canister on the IC (no special privileges required) can execute this attack. Obtaining two certificates at different block heights is a normal canister operation — no threshold corruption, no key leakage, no timing precision required. The two messages can be submitted seconds apart within the same `ingress_expiry` window. The attack is repeatable and requires no victim interaction.

## Recommendation

Remove `sig` from the representation-independent hash. The `MessageId` should be computed from the logical content of the request (`{info, signer}`) but not from the authentication proof (`sig`). This is consistent with the existing design decision to exclude the envelope-level `sender_sig`.

In `representation_independent_hash_call_or_query`:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, sig: _ }) = sender_info {
    map.insert(
        "sender_info",
        Map(btreemap! {
            "info"   => Bytes(info),
            "signer" => Bytes(signer),
            // sig excluded — authentication proof, not logical content
        }),
    );
}
```

The `sig` field must continue to be validated by `verify_sender_info_canister_sig` but must not influence the `MessageId`. The `message_id_icf_reference` test should be extended to cover the `sender_info` case, asserting that two messages with the same `{info, signer}` but different `sig` values produce the same `MessageId`.

## Proof of Concept

```rust
// Unit test (no network required)
let info_bytes = b"user attributes".to_vec();
let seed = b"seed";

// Canister C produces two valid signatures at different heights
// (same certified_data, different block heights → different certificate bytes)
let sig1 = canister_sign_at_height(&info_bytes, seed, height_1);
let sig2 = canister_sign_at_height(&info_bytes, seed, height_2);
assert_ne!(sig1, sig2); // different certificates, same logical content

let make_msg = |sig: Vec<u8>| build_signed_ingress(
    sender, canister_id, method_name.clone(), arg.clone(),
    ingress_expiry, nonce.clone(),
    Some(SignedSenderInfo { info: info_bytes.clone(), signer: C, sig }),
);

let msg1 = make_msg(sig1);
let msg2 = make_msg(sig2);

// Different MessageIds — deduplication is blind to them
assert_ne!(msg1.content().id(), msg2.content().id());

// Both pass validate_sender_info
assert!(validate_request(&msg1, ...).is_ok());
assert!(validate_request(&msg2, ...).is_ok());

// Feed both to ValidSetRuleImpl — both get inducted
let mut state = ReplicatedState::new(...);
valid_set_rule.induct_messages(&mut state, vec![msg1, msg2], round);
// Both messages appear in the canister's input queue — double execution
assert_eq!(ingress_queue_size(&state, canister_id), 2);
```
