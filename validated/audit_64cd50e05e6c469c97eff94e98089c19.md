### Title
Unbounded `active_assembles` / `artifact_processor_tasks` Growth via Malicious Peer Slot Updates — (`rs/p2p/consensus_manager/src/receiver.rs`)

---

### Summary

`handle_slot_update_receive` in `ConsensusManagerReceiver` inserts a new entry into the global `active_assembles` `HashMap` and spawns a new `process_slot_update` task for every unique artifact ID it has not seen before. The per-peer slot-table limit (`slot_limit`) is set to `usize::MAX` (`SLOT_TABLE_NO_LIMIT`) for consensus, certification, DKG, iDKG, and HTTPS-outcalls artifact types. A single authenticated subnet peer can therefore advertise an unbounded stream of unique artifact IDs, causing unbounded memory and task growth on every receiving node until the process is OOM-killed.

---

### Finding Description

**Slot-limit configuration** (`rs/replica/setup_ic_network/src/lib.rs`):

```
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;   // only ingress is bounded
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;     // consensus, cert, dkg, idkg, https
``` [1](#0-0) 

The comment on line 72 explicitly acknowledges the ingress limit exists "to protect against a malicious peer advertising many ingress messages," but the same protection is absent for all other artifact types. [2](#0-1) [3](#0-2) 

**The vulnerable path in `handle_slot_update_receive`:**

1. A new slot number (vacant entry) is accepted as long as `peer_slot_table_len < self.slot_limit`. With `slot_limit = usize::MAX` this check is always true. [4](#0-3) 

2. When the artifact ID has not been seen before (`active_assembles` miss), a new `watch` channel is created, the ID is inserted into the global `active_assembles` map, and a new async task is spawned into `artifact_processor_tasks`. [5](#0-4) 

3. The spawned `process_slot_update` task blocks until the peer counter for that ID reaches zero — i.e., until the advertising peer removes the ID from its slot table. A malicious peer simply never removes the IDs. [6](#0-5) 

**Result:** For every unique artifact ID the attacker advertises on a new slot number, one entry is permanently added to `active_assembles` and one live Tokio task is permanently added to `artifact_processor_tasks`. Both data structures are unbounded for the affected artifact types.

---

### Impact Explanation

- **Memory exhaustion**: Each entry in `active_assembles` holds a `watch::Sender<PeerCounter>` plus the cloned ID. Each task in `artifact_processor_tasks` holds its full async stack, a `watch::Receiver`, and the `FetchArtifact` assembler state (including a `Transport` arc and a `Bouncer` watch receiver). Millions of entries exhaust heap memory.
- **CPU exhaustion**: Each live task polls the bouncer, issues RPC fetch attempts, and awaits watch-channel changes. Thousands of tasks saturate the Tokio thread pool.
- **Node crash / consensus stall**: The receiving replica node is OOM-killed or becomes unresponsive, dropping it from active consensus participation. On a small subnet (13 nodes, f=4) a single attacker node can crash multiple honest nodes simultaneously.

---

### Likelihood Explanation

- The attacker must be an authenticated subnet member (QUIC/TLS peer identity is verified by the transport layer). Joining a subnet requires an NNS governance proposal, but the protocol explicitly tolerates up to f malicious nodes.
- The scope rules explicitly include "protocol peer behavior below the consensus fault threshold."
- The attack is trivially scriptable: send `SlotUpdate` messages with monotonically increasing `slot_id` and unique artifact IDs. No cryptographic material beyond the node's own TLS key is required.
- The ingress artifact type already has an explicit 50,000-slot cap with the exact same rationale; the absence of the same cap for consensus/cert/dkg/idkg/https is an oversight, not a design choice.

---

### Recommendation

Apply a per-peer slot-table limit to all artifact types, not just ingress. The existing `SLOT_TABLE_LIMIT_INGRESS` mechanism already works correctly; the fix is to pass a finite `slot_limit` instead of `SLOT_TABLE_NO_LIMIT` when registering consensus, certification, DKG, iDKG, and HTTPS-outcalls channels. The appropriate bound should reflect the maximum number of in-flight artifacts a legitimate node would ever advertise simultaneously for each artifact type (e.g., a few hundred for consensus, a few thousand for iDKG).

Additionally, consider adding a global cap on `active_assembles.len()` as a defense-in-depth measure, since the per-peer slot limit still allows `N_peers × slot_limit` total entries.

---

### Proof of Concept

```
// Attacker node sends to victim node over authenticated QUIC connection:
for slot_id in 0..u64::MAX {
    send SlotUpdate {
        slot_id,
        commit_id: slot_id,
        update: Update::Id(<unique artifact ID derived from slot_id>),
    }
}
// Each message passes the slot-limit check (usize::MAX) for consensus/cert/dkg/idkg/https,
// inserts a new entry into active_assembles, and spawns a new process_slot_update task.
// Tasks never terminate because the attacker never removes the IDs from its slot table.
// Victim node exhausts memory and crashes.
```

The `slot_table_limit_exceeded` unit test already demonstrates the guard works for ingress (slot limit = 2); the same test with `SLOT_TABLE_NO_LIMIT` shows zero entries are ever rejected. [7](#0-6)
