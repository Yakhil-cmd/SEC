### Title
Scheduled XCC Promises Lack Deadline/Expiry, Enabling MEV-Timed Execution of Time-Sensitive NEAR Cross-Contract Calls — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `CrossContractCallArgs::Delayed` path in the XCC precompile stores a `PromiseArgs` in the router contract via `schedule()` with no deadline or expiry field. Because `execute_scheduled` is intentionally callable by **any** NEAR account, a NEAR validator or MEV searcher can withhold execution until market conditions are maximally unfavorable for the user (e.g., peak slippage on a NEAR DEX swap), then trigger execution at that moment. This is the direct structural analog of the Reth.sol missing-deadline issue.

---

### Finding Description

**Root cause — `PromiseArgs` carries no deadline:** [1](#0-0) 

`CrossContractCallArgs::Delayed(PromiseArgs)` is the variant that stores a promise for later execution. `PromiseArgs` (and all its sub-types `PromiseCreateArgs`, `PromiseWithCallbackArgs`, `NearPromise`) contain no `deadline` or `not_after` field anywhere in the type hierarchy.

**Root cause — `schedule()` stores the promise with no expiry:** [2](#0-1) 

The `schedule` function inserts the promise into `scheduled_promises` keyed only by a monotonically increasing nonce. There is no timestamp, block height, or deadline stored alongside it.

**Root cause — `execute_scheduled` is open to any caller:** [3](#0-2) 

The comment explicitly states this is intentional. Any NEAR account can call `execute_scheduled(nonce)` at any future block, with no time constraint.

**Precompile dispatch — the `Delayed` path:** [4](#0-3) 

When the EVM user selects `CrossContractCallArgs::Delayed`, the precompile calls `schedule` on the router sub-account and returns immediately. The promise is now sitting in NEAR storage with no expiry.

---

### Impact Explanation

A user who schedules a time-sensitive NEAR cross-contract call — most concretely a swap on a NEAR-native DEX such as Ref Finance — has their funds at risk. The scheduled promise encodes the swap parameters (token amounts, minimum output) at the time of scheduling. If a NEAR validator or MEV searcher:

1. Observes the scheduled promise in the router's storage,
2. Waits until the token price moves adversely (or front-runs the execution with their own trade to move the price),
3. Then calls `execute_scheduled` at that moment,

the user's swap executes at maximum slippage, and the difference in value is extractable by the attacker. Because the promise is stored indefinitely, the attacker has unlimited time to find the optimal moment.

**Impact: High — Theft of user funds (in-motion) / theft of unclaimed yield**, depending on the operation encoded in the promise.

---

### Likelihood Explanation

- The `Delayed` XCC path is a documented, production-facing feature explicitly designed for use when gas is insufficient for an immediate call.
- `execute_scheduled` is callable by any NEAR account — no special privilege is required.
- NEAR validators have full visibility into contract storage and can observe pending scheduled promises.
- MEV infrastructure on NEAR (e.g., bots monitoring contract state) is actively deployed.
- Any Aurora EVM contract that uses the XCC precompile with `Delayed` for a DEX interaction is immediately exposed.

---

### Recommendation

1. Add an optional `deadline: Option<u64>` (NEAR block timestamp in nanoseconds) field to `PromiseArgs` or to a new `ScheduledPromise` wrapper stored by the router.
2. In `execute_scheduled`, check `env::block_timestamp() <= deadline` before executing; revert (and optionally delete the entry) if the deadline has passed.
3. Expose the deadline parameter through the XCC precompile ABI so EVM callers can set it when constructing `CrossContractCallArgs::Delayed`.

---

### Proof of Concept

1. Alice deploys an EVM contract on Aurora that uses the XCC precompile to schedule a swap on Ref Finance:
   ```
   CrossContractCallArgs::Delayed(PromiseArgs::Create(PromiseCreateArgs {
       target_account_id: "v2.ref-finance.near",
       method: "swap",
       args: /* swap 100 USDC → NEAR, min_out = 95 NEAR */,
       attached_balance: ...,
       attached_gas: ...,
   }))
   ```
2. The precompile calls `router.schedule(promise)` — stored at nonce `N` with no deadline.
3. A NEAR validator observes the router's `scheduled_promises[N]` in storage.
4. The validator front-runs by selling NEAR on Ref Finance, moving the price down.
5. The validator calls `router.execute_scheduled({"nonce": N})`.
6. Alice's swap executes at the depressed price; she receives far fewer NEAR than `min_out` would have protected (if `min_out` was set loosely) or the swap fails (if `min_out` was tight but the validator times it to just barely pass).
7. The validator profits from the price impact they created.

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
