Audit Report

## Title
ckBTC Minter Permanently Discards Burned ckBTC on `AmountTooLow` Without Reimbursement - (`File: rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary

When a user calls `retrieve_btc`, their ckBTC is burned atomically before the withdrawal request is queued. If Bitcoin network fees rise between request acceptance and transaction construction, `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, and the minter finalizes the request via `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` — with no reimbursement scheduled. The user's ckBTC is permanently lost: the ledger burn is irreversible, no BTC is sent, and no mint-back is ever triggered.

## Finding Description

**Step 1 — Burn precedes queuing.**

In `retrieve_btc.rs`, `burn_ckbtcs` is called at line 209 before `accept_retrieve_btc_request` at line 232. The burn is an irreversible ledger operation. The `reimbursement_account` is populated on the request (lines 218–221), so the infrastructure for reimbursement exists but is never invoked for `AmountTooLow`. [1](#0-0) 

**Step 2 — `AmountTooLow` is triggered at transaction-construction time using the current fee rate.**

In `build_unsigned_transaction_from_inputs`, the check at line 1306 compares `fee + minter_fee > amount` using the fee rate fetched at the moment `submit_pending_requests` runs — not the fee rate at request-acceptance time. [2](#0-1) 

**Step 3 — The `AmountTooLow` arm finalizes with no reimbursement.**

In `submit_pending_requests`, the `Err(BuildTxError::AmountTooLow)` arm (lines 412–434) calls only `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. No call to `reimburse_canceled_requests` is made. [3](#0-2) 

**Step 4 — Contrast with `InvalidTransaction`, which IS reimbursed.**

The immediately preceding arm (lines 400–410) calls `reimburse_canceled_requests`, which schedules a ckBTC mint-back to the user's `reimbursement_account`. The `AmountTooLow` arm has no equivalent. [4](#0-3) 

**Step 5 — `WithdrawalReimbursementReason` has no `AmountTooLow` variant.**

The enum in `reimbursement/mod.rs` only defines `InvalidTransaction`. There is no variant for fee-too-high, confirming the omission is structural, not just a missing call site. [5](#0-4) 

**Step 6 — The `fee_based_retrieve_btc_min_amount` guard is insufficient.**

The guard at lines 169–171 of `retrieve_btc.rs` validates the amount only against the fee estimate *at call time*. Already-queued requests are never re-validated when `fee_based_retrieve_btc_min_amount` is updated. A request accepted at fee level F can fail with `AmountTooLow` if fees rise to F' > amount before the timer fires. [6](#0-5) 

The `DustOutput` arm (lines 436–453) has the same defect for individual requests in a batch. [7](#0-6) 

## Impact Explanation

This is a permanent, irreversible loss of user ckBTC — a chain-key asset — with no recovery path. The ckBTC ledger burn is final; the minter records `FinalizedStatus::AmountTooLow` and schedules no reimbursement. A user querying `retrieve_btc_status_v2` will see `AmountTooLow` with no `WillReimburse` or `Reimbursed` state ever reachable. This matches the allowed impact: **"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm"** — High ($2,000–$10,000). The per-incident loss is bounded by the minimum withdrawal amount (~50,000 sat), but the condition is repeatable across all users with pending requests during any fee spike, and the loss is total for each affected user.

## Likelihood Explanation

Bitcoin fees are publicly known to spike sharply (e.g., during inscription/ordinal activity). The minter's timer introduces a mandatory window between burn and transaction construction. Any user who submits `retrieve_btc` just above the current minimum during a low-fee period is at risk if fees rise before the batch is processed. This is an unprivileged ingress path — `retrieve_btc` is a public update call requiring no special access. The condition is not adversarially controlled but arises naturally from Bitcoin fee market volatility, making it a realistic operational risk rather than a theoretical one.

## Recommendation

Add a reimbursement path for `BuildTxError::AmountTooLow` mirroring the `InvalidTransaction` handler in `lib.rs`. Add a new `WithdrawalReimbursementReason` variant (e.g., `FeesTooHigh`) in `reimbursement/mod.rs`, then replace the bare `remove_retrieve_btc_request` loop in the `AmountTooLow` arm with a call to `reimburse_canceled_requests(s, batch, reason, reimbursement_fee, runtime)`. Apply the same fix to the `DustOutput` arm for the individual request that is finalized as `AmountTooLow`. The penalty fee deducted should cover the cost of the failed attempt, consistent with the `InvalidTransaction` handler. [8](#0-7) 

## Proof of Concept

1. Deploy a local ckBTC minter instance (PocketIC or integration test harness).
2. Set `fee_based_retrieve_btc_min_amount` to 5,000 sat by configuring a low fee estimate.
3. Call `retrieve_btc { amount: 5,500, address: "bc1q..." }` as an unprivileged user. Observe `RetrieveBtcOk { block_index: N }` and confirm the ckBTC ledger burn at index N.
4. Before the minter timer fires, inject a high fee estimate (e.g., 200 sat/vbyte) so that `fee + minter_fee > 5,500`. This causes `build_unsigned_transaction` to return `BuildTxError::AmountTooLow`.
5. Trigger the minter timer (`submit_pending_requests`).
6. Assert: `retrieve_btc_status_v2(N)` returns `AmountTooLow`. Assert: `pending_withdrawal_reimbursements` does not contain an entry for burn index N. Assert: no ckBTC mint was issued to the user's account.

The invariant test to add: for every request finalized as `FinalizedStatus::AmountTooLow` where `reimbursement_account` is `Some`, a corresponding entry must exist in `reimbursed_transactions` or `pending_withdrawal_reimbursements`. [3](#0-2)

fn reimburse_canceled_requests<R: CanisterRuntime>(
    state: &mut state::CkBtcMinterState,
    requests: BTreeSet<state::RetrieveBtcRequest>,
    reason: WithdrawalReimbursementReason,
    total_fee: u64,
    runtime: &R,
) {
    assert!(!requests.is_empty());
    let fees = distribute(total_fee, requests.len() as u64);
    // This assertion makes sure the fee is smaller than each request amount
    assert!(
        fees[0] <= state.retrieve_btc_min_amount,
        "BUG: fees {fees:?} for {} withdrawal requests are larger than `retrieve_btc_min_amount` {}",
        requests.len(),
        state.retrieve_btc_min_amount
    );
    for (request, fee) in requests.into_iter().zip(fees.into_iter()) {
        if let Some(account) = request.reimbursement_account {
            let amount = request.amount.saturating_sub(fee);
            if amount > 0 {
                state::audit::reimburse_withdrawal(
                    state,
                    request.block_index,
                    amount,
                    account,
                    reason.clone(),
                    runtime,
                );
            }
        } else {
            log!(
                Priority::Info,
                "[reimburse_canceled_requests]: account is not found for retrieve_btc request ({:?})",
                request
            );
        }
    }
}
