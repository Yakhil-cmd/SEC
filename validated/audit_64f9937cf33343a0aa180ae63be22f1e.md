### Title
NEAR Permanently Stuck in XCC Router Sub-Account When Scheduled Promise Execution Fails — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

When a user schedules a cross-contract call (XCC) with `CrossContractCallArgs::Delayed` and a non-zero `attached_balance`, the required NEAR is unwrapped from the user's wNEAR ERC-20 balance and deposited into the router sub-account before the promise is executed. If the eventual `execute_scheduled` call results in a failed target receipt, NEAR protocol refunds the attached deposit back to the router sub-account. The router contract has no callback, no recovery function, and no mechanism to return this NEAR to the originating user. The funds are permanently frozen.

---

### Finding Description

The XCC delayed-promise flow proceeds as follows:

**Step 1 — XCC precompile charges the user:**
In `engine-precompiles/src/xcc.rs`, when `CrossContractCallArgs::Delayed` is used with `attached_balance > 0`, the precompile calls `transferFrom` on the wNEAR ERC-20 contract to move `required_near` from the user's EVM address to the engine's implicit address. [1](#0-0) 

**Step 2 — NEAR is sent to the router sub-account:**
`handle_precompile_promise` in `engine/src/xcc.rs` calls `withdraw_wnear_to_router`, which burns the wNEAR ERC-20 tokens from the engine's implicit address and sends the equivalent NEAR to the router sub-account. [2](#0-1) 

**Step 3 — Promise is stored, not executed:**
The router's `schedule` function stores the promise (with its `attached_balance`) in `scheduled_promises`. No NEAR is attached to the `schedule` call itself (`attached_balance: ZERO_YOCTO`). The NEAR already sits in the router's account balance. [3](#0-2) 

**Step 4 — `execute_scheduled` has no failure callback:**
When `execute_scheduled` is called, it removes the promise from storage and dispatches it via `promise_create`. If the target call fails, NEAR protocol refunds the attached deposit to the predecessor of the receipt — the router sub-account. The function has no `promise_then` callback to detect failure and no mechanism to return the NEAR to the user. [4](#0-3) 

**Step 5 — No general NEAR recovery exists:**
The only function that moves NEAR out of the router is `send_refund`, which sends a hardcoded `REFUND_AMOUNT` (2 NEAR) to the parent (Aurora engine). It is callable only by the parent and is not designed for user-deposited NEAR recovery. [5](#0-4) 

There is no `withdraw`, `recover`, or user-accessible refund function in the router contract.


---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's wNEAR ERC-20 balance is irreversibly reduced at the time the XCC precompile runs (Step 1). If the scheduled promise's target call fails, the NEAR is returned by NEAR protocol to the router sub-account and has no path back to the user. The router exposes no user-callable recovery function. The funds are permanently frozen in the router sub-account with no on-chain remedy available to the affected user.

---

### Likelihood Explanation

**Medium.** Any user who:
1. Uses `CrossContractCallArgs::Delayed` with `attached_balance > 0`, and
2. Has `execute_scheduled` called (by themselves or anyone, since it is open), and
3. Whose target call fails (non-existent account, panic, out-of-gas, etc.)

will lose their NEAR permanently. Target call failure is a realistic and common occurrence in cross-contract interactions. The `execute_scheduled` function is intentionally callable by anyone, so the trigger condition is reachable without any privileged access. [6](#0-5) 

---

### Recommendation

Add a `promise_then` callback to `execute_scheduled` that:
1. Checks whether the dispatched promise succeeded or failed.
2. On failure, mints the equivalent wNEAR ERC-20 tokens back to the originating EVM address (by calling back into the Aurora engine), or credits the user's ETH balance.

Alternatively, store the originating EVM address alongside each scheduled promise so that a recovery path can be invoked on failure.

---

### Proof of Concept

1. User holds 1 wNEAR as ERC-20 on Aurora and calls the XCC precompile with:
   ```
   CrossContractCallArgs::Delayed(PromiseArgs::Create(PromiseCreateArgs {
       target_account_id: "nonexistent.near",
       method: "some_method",
       args: b"",
       attached_balance: Yocto::new(1_000_000_000_000_000_000_000_000), // 1 NEAR
       attached_gas: ...,
   }))
   ```
2. XCC precompile calls `transferFrom` on wNEAR ERC-20: user's balance decreases by 1 NEAR equivalent.
3. Engine calls `withdraw_wnear_to_router`: 1 NEAR is deposited into the router sub-account.
4. Router's `schedule` stores the promise.
5. Anyone calls `execute_scheduled(nonce: 0)`.
6. Router dispatches a receipt to `nonexistent.near` with 1 NEAR attached.
7. Receipt fails (account does not exist); NEAR protocol refunds 1 NEAR to the router sub-account.
8. User's wNEAR ERC-20 balance remains at 0. The 1 NEAR is permanently locked in the router sub-account with no user-accessible recovery path. [4](#0-3) [2](#0-1) [7](#0-6)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
