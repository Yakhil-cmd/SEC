### Title
Pre-EIP-155 Legacy Transactions Bypass Chain-ID Validation, Enabling Cross-Chain Replay Attacks — (`engine-transactions/src/legacy.rs`, `engine/src/engine.rs`)

---

### Summary

Aurora Engine unconditionally accepts legacy transactions signed with `v = 27` or `v = 28` (pre-EIP-155 format). Because these transactions carry no chain-ID in their signature, `NormalizedEthTransaction.chain_id` is `None`, and the chain-ID guard in `submit_with_alt_modexp` is skipped entirely. Any pre-EIP-155 signed transaction that is valid on another EVM chain is also valid on Aurora, enabling a cross-chain replay attack that drains the victim's Aurora ETH balance.

---

### Finding Description

**Step 1 — `chain_id()` returns `None` for v = 27/28**

`LegacyEthSignedTransaction::chain_id()` in `engine-transactions/src/legacy.rs`:

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,           // v=27 and v=28 fall here
        _ => Some((self.v - 35) / 2),
    }
}
``` [1](#0-0) 

**Step 2 — `NormalizedEthTransaction` inherits `None` chain_id**

`TryFrom<EthTransactionKind> for NormalizedEthTransaction` in `engine-transactions/src/lib.rs` propagates the `None` directly:

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for v=27/28
    ...
}
``` [2](#0-1) 

**Step 3 — Chain-ID guard is skipped when `chain_id` is `None`**

In `submit_with_alt_modexp` (`engine/src/engine.rs`):

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [3](#0-2) 

When `transaction.chain_id` is `None`, the `if let Some(...)` arm never matches, the body is never entered, and execution continues unconditionally. There is no fallback rejection for the `None` case.

**Step 4 — Sender recovery also uses no chain binding for v = 27/28**

`LegacyEthSignedTransaction::sender()` hashes only the 6-field RLP `[nonce, gasPrice, gasLimit, to, value, data]` when `chain_id` is `None`, which is identical across every EVM chain: [4](#0-3) 

The recovered address is therefore the same on Aurora as on any other chain, and the signature is valid on all of them simultaneously.

---

### Impact Explanation

An attacker who observes a pre-EIP-155 signed transaction broadcast on any other EVM chain (e.g., Ethereum mainnet, BSC) can submit the identical RLP bytes to Aurora's `submit` endpoint. If the victim's Aurora account nonce matches the nonce in the replayed transaction, the EVM executes it: ETH is transferred, contracts are called, or deployments occur — all without the victim's knowledge or consent on Aurora. The most direct theft path: victim legitimately pays the attacker on Ethereum with a pre-EIP-155 transaction (nonce=N); attacker replays on Aurora when victim's Aurora nonce is also N, draining the victim's Aurora ETH to the attacker's address.

**Impact: Critical — Direct theft of user funds.**

---

### Likelihood Explanation

- Pre-EIP-155 signing (v=27/28) is still produced by certain hardware wallets in legacy mode, older tooling, and custom scripts. The test suite itself exercises this path (`test_eth_signed_no_chain_sender`, `create_eth_transaction` with `chain_id: None`).
- Nonce alignment is most trivially satisfied for fresh Aurora accounts (nonce=0), which is the common case for users who hold Aurora ETH but have not yet transacted on Aurora.
- The attacker needs only to monitor the public mempool or blockchain of another EVM chain — no private key compromise, no admin access, no social engineering.

**Likelihood: Medium** (requires pre-EIP-155 signing by the victim, but the nonce=0 case is common and the attack requires no privileged access).

---

### Recommendation

Reject legacy transactions that carry no chain-ID binding. Change the guard from an opt-in check to a mandatory one:

```rust
// Require chain_id to be present and match Aurora's chain_id
let Some(chain_id) = transaction.chain_id else {
    return Err(EngineErrorKind::InvalidChainId.into());
};
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

If backward compatibility with pre-EIP-155 transactions is required (e.g., for historical replay in the standalone engine), the rejection should be enforced at least in the `#[cfg(feature = "contract")]` production path.

---

### Proof of Concept

```rust
// 1. Sign a legacy transaction with chain_id = None (v = 27 or 28)
let tx = TransactionLegacy {
    nonce: U256::zero(),          // victim's Aurora nonce is also 0
    gas_price: U256::zero(),
    gas_limit: u64::MAX.into(),
    to: Some(attacker_address),   // attacker's address
    value: Wei::new_u64(1_000_000_000_000_000_000), // 1 ETH
    data: vec![],
};
let signed = sign_transaction(tx, None, &victim_secret_key); // chain_id = None → v = 27/28

// 2. Encode and submit to Aurora
let tx_bytes = rlp::encode(&signed).to_vec();
let args = SubmitArgs { tx_data: tx_bytes, .. };

// 3. Aurora accepts it: chain_id is None, guard is skipped, EVM executes
// Result: victim's Aurora ETH is transferred to attacker_address
// The same bytes also execute on Ethereum mainnet (or any other chain accepting pre-EIP-155 txs)
```

The existing test `test_eth_signed_no_chain_sender` in `engine-transactions/src/legacy.rs` already confirms that a transaction with `v = 28` decodes successfully and recovers a valid sender, demonstrating the path is live in unmodified production code. [5](#0-4)
