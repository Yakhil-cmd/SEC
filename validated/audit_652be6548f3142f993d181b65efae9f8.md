Audit Report

## Title
Cross-Subaccount Allowance Disclosure in `get_allowances` — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary

`get_allowances` performs a forward BTreeMap range scan over `ALLOWANCES_MEMORY` and terminates only when the **owner** field changes, with no guard on the **subaccount** field. Because `AccountSpender` keys are ordered by `(owner, effective_subaccount, spender)`, querying with a subaccount that is lexicographically smaller than any stored subaccount causes the scan to silently return allowances belonging to entirely different subaccounts of the same owner. Any unprivileged caller can enumerate all allowances for all subaccounts of any principal with a single read-only query.

## Finding Description

`get_allowances` in `rs/ledger_suite/icrc1/ledger/src/lib.rs` starts a range scan at `start_account_spender` and iterates forward: [1](#0-0) 

The only account-dimension termination guard is:

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

There is no corresponding check on `account_spender.account.effective_subaccount() == from.effective_subaccount()`.

`AccountSpender` ordering delegates to `Account::cmp`, which sorts by `(owner, effective_subaccount())`: [2](#0-1) [3](#0-2) 

`effective_subaccount()` maps `None` to `[0u8; 32]`, the lexicographic minimum. So a query with `subaccount: None` (effective `[0x00;32]`) starts the scan at the lowest possible key for that owner, and the loop will traverse and return every stored allowance for every subaccount of that owner — including `[0xFF;32]`, `[0x01;32]`, etc. — because the owner check never fires.

The result's `from_account` field is set directly from `account_spender.account`: [4](#0-3) 

So the response reveals the actual subaccount of each leaked entry.

The existing test suite explicitly acknowledges and accepts this cross-subaccount bleed-through as expected behavior, with the comment *"the result is the same as listing for current `from_account`"*, and the assertion `check_allowances(allowances, idx, from.owner)` validates only the owner field, not the full account: [5](#0-4) 

## Impact Explanation

This is a **significant ICRC ledger security impact with concrete user harm**, qualifying as **High** severity. The ICRC-103 invariant requires that results only contain entries where `from_account` exactly matches the queried account (owner **and** subaccount). The bug violates this invariant, enabling complete allowance disclosure across all subaccounts of any principal: which subaccounts have active approvals, which spenders are authorized, exact amounts, and expiry timestamps. This is financial privacy data that users have a reasonable expectation will not be disclosed when a different subaccount is queried. The impact does not reach Critical because no funds can be directly stolen or minted via this read-only path.

## Likelihood Explanation

The attack requires no privileges, no tokens, no prior state, and no special timing. It is a single read-only query call to a public endpoint. Any party who knows a victim's principal ID can execute it immediately and repeatedly at zero cost.

## Recommendation

Add a subaccount equality check immediately after the owner check in `get_allowances`:

```rust
if account_spender.account.owner != from.owner {
    break;
}
// Add:
if account_spender.account.effective_subaccount() != from.effective_subaccount() {
    break;
}
```

The test at lines 2836–2847 of `rs/ledger_suite/tests/sm-tests/src/lib.rs` that currently asserts cross-subaccount bleed-through as correct behavior must also be updated to assert an empty result when the queried subaccount differs from the stored one.

## Proof of Concept

```
1. icrc2_approve(
     from = { owner: X, subaccount: [0xFF;32] },
     spender = A,
     amount = 1_000_000
   )
   → stored key: (X, [0xFF;32], A)

2. icrc103_get_allowances({
     from_account: Some({ owner: X, subaccount: None }),  // effective = [0x00;32]
     prev_spender: None,
     take: None
   })

3. Range scan starts at (X, [0x00;32], ∅).
   First entry found: (X, [0xFF;32], A).
   Owner check: X == X → do not break.
   Subaccount check: MISSING.
   Entry pushed with from_account = { owner: X, subaccount: [0xFF;32] }.

4. Caller receives the allowance for a subaccount they never queried.
   Invariant violated: result[0].from_account ≠ queried from_account.
```
