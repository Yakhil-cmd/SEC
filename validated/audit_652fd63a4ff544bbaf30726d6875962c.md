### Title
Cross-Chain Signature Replay via Unprotected Legacy Transactions (v=27/28) - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

### Summary
ZKsync OS accepts pre-EIP-155 legacy transactions (where `v == 27` or `v == 28`) and computes their signing hash **without including the chain ID**. This allows an attacker to take a valid unprotected legacy transaction broadcast on any other EVM chain (e.g., Ethereum mainnet) and replay it verbatim on ZKsync OS, draining the victim's funds if their nonce and balance align.

### Finding Description
In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on `legacy_signature.is_eip155()`:

```rust
// legacy_tx.rs:129-131
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
```

When `v == 27` or `v == 28`, `is_eip155()` returns `false`, and the signing hash is computed over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` — **no chain ID is included**:

```rust
// legacy_tx.rs:89-94
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
```

The transaction is then stored as `Self::Legacy(tx, sig_data)` with no chain ID rejection:

```rust
// mod.rs:127-131
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    Self::Legacy(tx, sig_data)
};
```

The signature is subsequently verified in `validation_impl.rs` against this chain-ID-free hash via `ecrecover`. Because the hash is identical on every EVM chain, a signature valid on Ethereum mainnet is equally valid on ZKsync OS.

### Impact Explanation
An attacker who observes a victim's unprotected legacy transaction on Ethereum mainnet (or any other EVM chain) can submit the exact same raw bytes to ZKsync OS. If the victim holds the same nonce on ZKsync OS and has a non-zero balance, the transaction executes and transfers the victim's funds to the attacker-chosen destination. This is a direct, unprivileged loss-of-funds path requiring no special access.

### Likelihood Explanation
Pre-EIP-155 transactions are still broadcast on Ethereum mainnet today (e.g., by older wallets, hardware wallets with legacy firmware, or tooling that omits chain ID). Any such transaction is immediately replayable on ZKsync OS. The attacker only needs to monitor the Ethereum mempool or chain history, find a victim whose ZKsync nonce matches, and submit the transaction. No privileged access, oracle manipulation, or governance attack is required.

### Recommendation
Reject unprotected legacy transactions (v=27/28) at parse time by returning `InvalidTransaction::InvalidChainId` (or a new `MissingChainId` variant) when `is_eip155()` is false. All other typed transaction formats (EIP-2930, EIP-1559, EIP-4844, EIP-7702) already enforce `tx.chain_id != expected_chain_id` as a hard rejection. Legacy transactions should receive the same treatment:

```rust
// In parse_and_compute_signed_hash, legacy branch:
if !sig_data.is_eip155() {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

### Proof of Concept

1. On Ethereum mainnet, user Alice signs a legacy transfer with `v=27`:
   - `nonce=5, gasPrice=20gwei, gasLimit=21000, to=attacker, value=1 ETH`
   - Signature: `(v=27, r, s)`

2. Alice also has nonce=5 and ≥1 ETH on ZKsync OS.

3. Attacker takes the raw RLP bytes of Alice's Ethereum transaction and submits them to ZKsync OS.

4. `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` computes the signing hash over `[5, 20gwei, 21000, attacker, 1 ETH, ""]` — identical to the Ethereum hash.

5. `ecrecover(hash, 27, r, s)` recovers Alice's address. The `from` check passes.

6. Nonce 5 matches; the transaction executes; Alice loses 1 ETH on ZKsync OS.
