### Title
EIP-7702 Authorization List Duplicate Authority Enables Unbounded Gas Refund Inflation - (File: `basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

---

### Summary

The `parse_authorization_list_and_apply_delegations` function in ZKsync OS processes EIP-7702 authorization list entries without tracking which authority addresses have already been successfully applied within the same transaction. An attacker can craft a single EIP-7702 transaction containing multiple authorization entries that all resolve to the same authority address, each with a sequentially incremented nonce. Each successful entry triggers an unconditional gas refund addition via `add_to_refund_counter`. Because the refund counter accumulates without bound across all entries, an attacker can inflate the EVM gas refund to the maximum allowed (1/5 of gas used), effectively paying far less than the actual gas consumed.

---

Inside `validate_and_apply_delegation`, step 7 unconditionally adds to the refund counter whenever the authority account is non-empty:

```rust

        (evm_interpreter::gas_constants::NEWACCOUNT

``` [2](#0-1) 

After step 8 (`set_delegation`) and step 9 (`increment_nonce`), the authority's nonce is bumped by 1. [3](#0-2) 

The nonce check at step 6 reads the **current** (post-increment) nonce from the account cache on each iteration:

```rust
if account_properties.nonce.0 != auth_nonce {
    return Ok(false);
}
``` [4](#0-3) 

This means an attacker can include N authorization entries for the same authority, each signed with `nonce = k, k+1, k+2, ..., k+N-1`. Each entry passes the nonce check (because the previous entry incremented the nonce), applies the delegation, and adds a full refund credit. There is no deduplication or per-authority tracking across entries.

The refund is consumed in `compute_gas_refund`, capped at 1/5 of gas used per EIP-3529:

```rust
let evm_refund = {
    let full_refund_ergs = system.io.get_refund_counter().ergs();
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};
``` [5](#0-4) 

With enough entries, the attacker saturates the refund counter to always hit the 1/5 cap, receiving the maximum possible gas refund on every EIP-7702 transaction.

---

### Impact Explanation

An unprivileged transaction sender can craft an EIP-7702 transaction with a large authorization list (all entries pointing to the same authority with sequential nonces) to:

1. Inflate the EVM gas refund counter to its maximum (1/5 of gas used).
2. Systematically underpay for gas on every EIP-7702 transaction.
3. At scale, this constitutes a resource accounting bug: the operator/sequencer receives less fee than the actual computational work performed, and the block's gas accounting diverges from the true cost.

The impact is economic: repeated exploitation allows a user to execute transactions at up to 20% discount on gas cost, draining operator revenue and potentially enabling spam at reduced cost.

---

### Likelihood Explanation

- EIP-7702 is a standard transaction type (type 0x04) accepted from any unprivileged sender.
- The authorization list size is bounded only by the transaction's gas limit (each entry costs `NEWACCOUNT` gas = 25,000 gas intrinsically). With a block gas limit of ~30M gas, an attacker can include ~1,200 entries per transaction.
- The attack requires only a single pre-signed set of sequential authorizations for one EOA, which the attacker controls. No privileged access, oracle manipulation, or external dependency is needed.
- The attack is repeatable across blocks.

---

### Recommendation

Track which authority addresses have already been successfully applied within the current authorization list processing loop. Before adding to the refund counter (and before applying the delegation), check whether this authority was already processed in this transaction's authorization list. If so, skip the refund addition (and optionally skip the delegation re-application, per EIP-7702 spec intent).

Concretely, maintain a small set (or sorted list) of already-applied authority addresses within `parse_authorization_list_and_apply_delegations` and skip the refund credit for duplicate authorities:

```rust
// pseudocode
let mut seen_authorities: BTreeSet<B160> = BTreeSet::new();
for entry in auth_list.iter() {
    let success = validate_and_apply_delegation(...)?;
    if success {
        if seen_authorities.contains(&authority) {
            // skip refund for duplicate
        } else {
            seen_authorities.insert(authority);
            // apply refund
        }
    }
}
```

Alternatively, note that after the first successful application the authority's nonce is incremented, so a second entry with `nonce = k+1` will also pass — the fix must be at the refund-granting level, not just the nonce check.

### Proof of Concept

1. Attacker controls EOA `A` with nonce `N` and non-zero balance (so `is_empty = false`).
2. Attacker pre-signs 100 authorization entries: `[chain_id, delegate_addr, N]`, `[chain_id, delegate_addr, N+1]`, ..., `[chain_id, delegate_addr, N+99]`, all signed by `A`.
3. Attacker submits a single EIP-7702 transaction (from any sender) containing all 100 entries.
4. `parse_authorization_list_and_apply_delegations` iterates all 100 entries:
   - Entry 0: nonce check passes (`A.nonce == N`), delegation applied, nonce bumped to `N+1`, refund added.
   - Entry 1: nonce check passes (`A.nonce == N+1`), delegation applied, nonce bumped to `N+2`, refund added.
   - ... (repeated 100 times)
5. The refund counter now holds `100 × (NEWACCOUNT - PER_AUTH_BASE_COST)` ergs.
6. In `compute_gas_refund`, the refund is capped at `gas_used / 5`. With 100 entries × 25,000 gas = 2,500,000 gas intrinsic cost, the attacker's gas_used is ~2.5M, and the maximum refund is ~500,000 gas — the full 1/5 cap — effectively reducing the net gas cost by 20%.
7. The attacker repeats this with fresh sequential nonces in subsequent transactions. [6](#0-5) [7](#0-6) [8](#0-7)

pub fn parse_authorization_list_and_apply_delegations<S: EthereumLikeTypes>(
    system: &mut System<S>,
    resources: &mut S::Resources,
    auth_list: AuthorizationList<'_>,
) -> Result<(), TxError>
where
    S::IO: IOSubsystemExt,
{
    use crate::bootloader::transaction::rlp_encoded::AuthorizationEntry;
    let mut hasher = crypto::sha3::Keccak256::new();

    for entry in auth_list.iter() {
        let AuthorizationEntry {
            chain_id,
            address,
            nonce,
            y_parity,
            r,
            s,
        } = entry;
        let success = validate_and_apply_delegation(
            system,
            resources,
            &chain_id,
            nonce,
            address,
            (y_parity, r, s),
            &mut hasher,
        )?;
        system_log!(system, "Delegation success: {success}\n");

        if !success {}
    }
    Ok(())
}
