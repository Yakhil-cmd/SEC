### Title
SNS Governance Majority Can Instantly Reduce Voting Reward Rate to Zero Before Distribution - (File: rs/sns/governance/src/governance.rs)

### Summary
In the SNS governance canister, a majority voting-power holder can pass a `ManageNervousSystemParameters` proposal to set `voting_rewards_parameters` reward rates to zero. The parameter change takes effect **immediately** upon proposal execution with no additional timelock, allowing the majority holder to deprive neuron holders of their expected rewards for the current round — directly analogous to H10's `deployerCut` front-run.

### Finding Description
`perform_manage_nervous_system_parameters` unconditionally overwrites `self.proto.parameters` the moment a `ManageNervousSystemParameters` proposal executes:

```rust
// rs/sns/governance/src/governance.rs
fn perform_manage_nervous_system_parameters(
    &mut self,
    proposed_params: NervousSystemParameters,
) -> Result<(), GovernanceError> {
    let new_params = proposed_params.inherit_from(self.nervous_system_parameters_or_panic());
    match new_params.validate() {
        Ok(()) => {
            self.proto.parameters = Some(new_params);  // immediate, no delay
            Ok(())
        }
        ...
    }
}

The reward purse for the round is computed directly from those parameters:

```rust
// reward_rate_at uses initial_reward_rate_basis_points / final_reward_rate_basis_points
let neuron_reward_e8s =
    rewards_purse_e8s * (neuron_reward_shares / total_reward_shares);
``` [3](#0-2) 

Setting both `initial_reward_rate_basis_points = 0` and `final_reward_rate_basis_points = 0` in `VotingRewardsParameters` causes `reward_rate_at` to return zero for every instant, collapsing `rewards_purse_e8s` to zero and giving every neuron holder zero maturity for that round. [4](#0-3) 

`NervousSystemParameters` exposes `voting_rewards_parameters` as a directly settable field in `ManageNervousSystemParameters` proposals: [5](#0-4) 

There is **no post-execution timelock** that delays the effect of a reward-rate change until after the current reward round ends. The proposal voting period (minimum floor `INITIAL_VOTING_PERIOD_SECONDS_FLOOR`) is the only delay, and it can itself be reduced to its floor by a prior `ManageNervousSystemParameters` proposal. Once the voting period elapses and the proposal executes, the new rate is live for the very next `distribute_rewards` invocation.

### Impact Explanation
Neuron holders who staked SNS tokens expecting voting rewards receive **zero maturity** for the affected round. Because `distribute_rewards` is called periodically (every `round_duration_seconds`, default one day), a single well-timed proposal execution can silently zero out an entire round's rewards across all participants. The stolen value equals `total_token_supply × current_reward_rate × round_duration`, which for a live SNS can be substantial. Neuron holders have no on-chain recourse after the fact; the round is marked settled and cannot be re-run.

### Likelihood Explanation
Many SNS launches begin with the founding team holding a supermajority of voting power (developer neurons, vesting neurons, treasury neurons). With majority voting power, a proposal passes as soon as the proposer's neuron votes — the wait-for-quiet extension does not trigger because there is no opposing flip. The attacker only needs to submit the proposal so that it executes within the same `round_duration_seconds` window as the next `distribute_rewards` call. Because `distribute_rewards` runs on a predictable timer, the timing is observable and controllable. This is a realistic scenario for any SNS in its early post-launch phase.

### Recommendation
1. **Pending-change timelock**: When a `ManageNervousSystemParameters` proposal that modifies `voting_rewards_parameters` is executed, store the new parameters in a `pending_voting_rewards_parameters` field with an `effective_at` timestamp set to at least `now + round_duration_seconds`. `distribute_rewards` should only promote pending parameters to active after `effective_at` has passed.
2. **Alternatively**, treat `voting_rewards_parameters` changes as critical proposals (requiring the higher 67% threshold already defined in `CRITICAL_MINIMUM_YES_PROPORTION_OF_EXERCISED_VOTING_POWER`) and enforce a mandatory delay equal to at least one full reward round before the change takes effect.
3. Emit a canister log event when `voting_rewards_parameters` changes so neuron holders can observe the change before the next distribution.

### Proof of Concept
```
Precondition: Attacker controls neurons with >50% voting power in SNS X.
              round_duration_seconds = 86400 (1 day, default).
              Next distribute_rewards fires at T+1d.

Step 1 (T=0):
  Attacker submits ManageNervousSystemParameters proposal:
    voting_rewards_parameters {
      initial_reward_rate_basis_points: 0
      final_reward_rate_basis_points:   0
      round_duration_seconds:           86400
    }
  Attacker's neuron votes YES immediately → proposal reaches majority.

Step 2 (T = initial_voting_period_seconds):
  Proposal executes. perform_manage_nervous_system_parameters writes
  new params to self.proto.parameters with no delay.

Step 3 (T+1d):
  distribute_rewards fires. nervous_system_parameters_or_panic()
  returns the updated params. reward_rate_at() returns 0.
  rewards_purse_e8s = 0. All neuron maturity increments = 0.
  RewardEvent is recorded; round is settled. Neuron holders receive nothing.
```

    pub fn reward_rate_at(&self, now: Instant) -> RewardRate {
        let reward_rate_transition_duration_seconds = self
            .reward_rate_transition_duration_seconds
            .expect("reward_rate_transition_duration_seconds unset");

        let time_since_genesis = {
            let result = now - *GENESIS;
            // For the purposes of determining reward rate, treat times before
            // genesis the same as at genesis. This is not expected to occur in
            // practice. This code is just being extra defensive.
            if result.as_secs() < i2d(0) {
                Duration { days: i2d(0) }
            } else {
                result
            }
        };
        if reward_rate_transition_duration_seconds == 0
            || time_since_genesis.as_secs() >= i2d(reward_rate_transition_duration_seconds)
        {
            return self.final_reward_rate();
        }

        // s linearly varies from 1 -> 0 as seconds_since_genesis varies from 0
        // to reward_rate_transition_duration_seconds.
        let transition = LinearMap::new(
            dec!(0)..i2d(reward_rate_transition_duration_seconds),
            dec!(1)..dec!(0),
        );
        let s = transition.apply(time_since_genesis.as_secs());
        // s2 varies quadratically from 1 -> 0 (again, as seconds_since_genesis
        // varies from 0 to reward_rate_transition_duration_seconds), and
        // flattens out as seconds_since_genesis approaches
        // reward_rate_transition_duration_seconds.
        let s2 = s * s;

        // This looks backwards, but we think of variable rate as being added to
        // final growth rate, not initial, and the amount to add is up to
        // initial - final (where initial is thought of as being greater than
        // final).
        let dr = self.initial_reward_rate() - self.final_reward_rate();
        // variable_reward_rate varies from dr to 0 as round varies from
        // 1 to transition_round_count.
        let variable_reward_rate = s2 * dr;

        self.final_reward_rate() + variable_reward_rate

message VotingRewardsParameters {
  // The amount of time between reward events.
  //
  // Must be > 0.
  //
  // During such periods, proposals enter the ReadyToSettle state. Once the round is over, voting
  // for those proposals entitle voters to voting rewards. Such rewards are calculated in
  // the governance canister's run_periodic_tasks function.
  //
  // This is a nominal amount. That is, the actual time between reward
  // calculations and distribution cannot be guaranteed to be perfectly
  // periodic, but actual inter-reward periods are generally expected to be
  // within a few seconds of this.
  //
  // This supersedes super.reward_distribution_period_seconds.
  optional uint64 round_duration_seconds = 1;

  // No longer needed because we assume the start time is the SNS genesis.
  reserved "start_timestamp_seconds";
  reserved 2;

  // The amount of time that the growth rate changes (presumably, decreases)
  // from the initial growth rate to the final growth rate. (See the two
  // *_reward_rate_basis_points fields bellow.) The transition is quadratic, and
  // levels out at the end of the growth rate transition period.
  optional uint64 reward_rate_transition_duration_seconds = 3;

  // The amount of rewards is proportional to token_supply * current_rate. In
  // turn, current_rate is somewhere between `initial_reward_rate_basis_points`
  // and `final_reward_rate_basis_points`. In the first reward period, it is the
  // initial growth rate, and after the growth rate transition period has elapsed,
  // the growth rate becomes the final growth rate, and remains at that value for
  // the rest of time. The transition between the initial and final growth rates is
  // quadratic, and levels out at the end of the growth rate transition period.
  //
  // (A basis point is one in ten thousand.)
  optional uint64 initial_reward_rate_basis_points = 4;
  optional uint64 final_reward_rate_basis_points = 5;
}
