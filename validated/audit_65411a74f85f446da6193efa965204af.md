### Title
`max_gas_price` Enforced Only on `priority_fee_per_gas`, Not on `effective_gas_price` — Users Charged Beyond Their Stated Maximum - (File: `engine/src/engine.rs`)

---

### Summary

In `Engine::charge_gas()`, the user-supplied `max_gas_price` parameter (from `SubmitArgs`) is applied only to cap the `priority_fee_per_gas` component of the gas price. The final `effective_gas_price = priority_fee_per_gas + block_base_fee_per_gas` is never compared against `max_gas_price`. When `block_base_fee_per_gas > 0`, the actual per-gas cost charged to the sender exceeds the user's stated maximum, draining more ETH from their balance than they authorized.

---

### Finding Description

`submit_with_args` accepts a `SubmitArgs` struct containing an optional `max_gas_price: Option<u128>` field, documented as "Max gas price the user is ready to pay for the transaction." [1](#0-0) 

Inside `charge_gas()`, the enforcement of `max_gas_price` is:

```rust
let priority_fee_per_gas = max_gas_price.map_or(priority_fee_per_gas, |price| {
    price.min(priority_fee_per_gas)   // ← caps only the priority component
});
let effective_gas_price = priority_fee_per_gas + block_base_fee_per_gas;
``` [2](#0-1) 

The actual amount deducted from the sender is `gas_limit × effective_gas_price`: [3](#0-2) 

There is **no check** that `effective_gas_price ≤ max_gas_price`. When `block_base_fee_per_gas > 0`, the total per-gas cost paid is `priority_fee_per_gas_capped + block_base_fee_per_gas`, which can exceed `max_gas_price` by up to `block_base_fee_per_gas`.

The code already acknowledges that `block_base_fee_per_gas` can be non-zero — it guards the early-return path with `block_base_fee_per_gas.is_zero()` and rejects transactions where `max_fee_per_gas < block_base_fee_per_gas`: [4](#0-3) 

The `max_gas_price` is passed from `submit_with_args` through `submit_with_alt_modexp` into `charge_gas`: [5](#0-4) 

---

### Impact Explanation

**Impact: High — Theft of user funds (ETH over-charge beyond user-authorized maximum).**

A user who calls `submit_with_args` with `max_gas_price = P` expects to pay at most `P × gas_used` ETH in fees. When `block_base_fee_per_gas = B > 0`, they actually pay `(P + B) × gas_used` ETH. The excess `B × gas_used` ETH is deducted from their balance without their consent. For a transaction using 1,000,000 gas with `B = 10 Gwei`, the unauthorized over-charge is 0.01 ETH per transaction. This excess accrues to the relayer (or is burned), constituting a direct, measurable loss of user funds beyond what the user authorized.

---

### Likelihood Explanation

**Likelihood: Medium.**

The vulnerability is triggered whenever:
1. A user calls `submit_with_args` with a non-`None` `max_gas_price`, AND
2. The Aurora network's `block_base_fee_per_gas` is non-zero.

The `submit_with_args` entrypoint is a documented, production-facing function. The `block_base_fee_per_gas` being non-zero is a normal network condition under EIP-1559 semantics. Any relayer or user-facing application that uses `max_gas_price` as a slippage guard (its stated purpose) will be silently over-charged.

---

### Recommendation

After computing `effective_gas_price`, add an explicit check that it does not exceed `max_gas_price`:

```rust
let effective_gas_price = priority_fee_per_gas + block_base_fee_per_gas;

// Add this guard:
if let Some(price) = max_gas_price {
    if effective_gas_price > price {
        return Err(GasPaymentError::EffectiveGasPriceExceedsMax);
    }
}
```

This mirrors the Bond Protocol fix: just as `payout >= minAmountOut_` must be checked against the *actual* computed payout (not just the theoretical maximum), `effective_gas_price <= max_gas_price` must be checked against the *actual* total gas price (not just the priority component).

---

### Proof of Concept

**Setup:**
- `block_base_fee_per_gas = 10` (non-zero base fee active on the network)
- User submits via `submit_with_args` with `max_gas_price = Some(5)`
- Transaction: `max_fee_per_gas = 20`, `max_priority_fee_per_gas = 15`, `gas_limit = 100_000`

**Execution through `charge_gas()`:**

1. `block_base_fee_per_gas = 10` ✓ (non-zero, so no early return)
2. `max_fee_per_gas (20) >= block_base_fee_per_gas (10)` ✓ (passes the base-fee check)
3. `priority_fee_per_gas = min(15, 20 - 10) = min(15, 10) = 10`
4. After `max_gas_price` cap: `priority_fee_per_gas = min(5, 10) = 5`
5. `effective_gas_price = 5 + 10 = 15` ← **exceeds `max_gas_price = 5`**
6. `prepaid_amount = 100_000 × 15 = 1_500_000 Wei`

**Expected by user:** `100_000 × 5 = 500_000 Wei`
**Actual charge:** `1_500_000 Wei`
**Unauthorized over-charge:** `1_000_000 Wei` (2× the user's stated maximum)

The transaction proceeds without error, and the user's balance is silently reduced by `1_000_000 Wei` beyond their stated `max_gas_price` limit. [6](#0-5)
