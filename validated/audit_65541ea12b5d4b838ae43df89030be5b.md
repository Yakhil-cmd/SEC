Audit Report

## Title
Heap-Only `API_BOUNDARY_NODE_PRINCIPALS` Wiped on Upgrade Creates FullReadâ†’RestrictedRead Downgrade Window â€” (`rs/boundary_node/rate_limits/canister/storage.rs`, `canister.rs`)

## Summary
`API_BOUNDARY_NODE_PRINCIPALS` is stored exclusively in heap memory and is silently reset to an empty `HashSet` on every canister upgrade. The `post_upgrade` hook re-registers a periodic timer via `set_timer_interval`, which does not fire immediately but only after the first full interval elapses. During this window every API boundary node principal is treated as `RestrictedRead`, causing confidential `rule_raw` and `description` fields to be redacted from responses, and replicated `get_config` calls to be trapped entirely.

## Finding Description
**Root cause 1 â€” heap-only storage:**
`API_BOUNDARY_NODE_PRINCIPALS` is declared as a plain `thread_local! RefCell<HashSet<Principal>>` with no backing stable memory (`storage.rs` L149). All other security-critical maps (`CONFIGS`, `RULES`, `INCIDENTS`, `AUTHORIZED_PRINCIPAL`) are backed by `StableBTreeMap` over `VirtualMemory<DefaultMemoryImpl>` and survive upgrades (`storage.rs` L125â€“147). This one does not.

**Root cause 2 â€” timer fires after interval, not immediately:**
`post_upgrade` delegates unconditionally to `init` (`canister.rs` L100â€“103). `init` calls `periodically_poll_api_boundary_nodes` (`canister.rs` L85â€“88), which schedules the registry poll exclusively via `ic_cdk_timers::set_timer_interval(interval, ...)` (`canister.rs` L254). There is no `set_timer(Duration::ZERO, ...)` one-shot call. The first execution fires only after `registry_polling_period_secs` seconds have elapsed.

**Root cause 3 â€” access level degrades to `RestrictedRead`:**
`is_api_boundary_node_principal` reads the now-empty set and returns `false` (`state.rs` L125â€“128). `AccessLevelResolver::get_access_level` therefore falls through to `AccessLevel::RestrictedRead` (`access_control.rs` L46â€“54).

**Root cause 4 â€” confidential fields are redacted:**
In `ConfigGetter::get`, `is_authorized_viewer` is `false` when access level is `RestrictedRead`, so `self.formatter.format(config)` is applied (`getter.rs` L130â€“147). `ConfigConfidentialityFormatter::format` sets `is_redacted = true` and nulls `rule_raw`/`description` for every rule where `disclosed_at.is_none()` (`confidentiality_formatting.rs` L17â€“28). The same redaction path applies in `RuleGetter` and `IncidentGetter` (`getter.rs` L234â€“244, L192â€“197).

**Root cause 5 â€” replicated query path fully blocked:**
`inspect_message` checks `has_full_read_access` for the `REPLICATED_QUERY_METHOD` (`get_config`). With an empty set this is `false`, so `ic_cdk::api::trap(...)` is called (`canister.rs` L48â€“55). Boundary nodes using the replicated query path receive a trap for the entire duration of the window.

## Impact Explanation
During the upgrade window, API boundary nodes receive either a trap (replicated query path) or a fully-redacted response (non-replicated query path) for `get_config`. Without `rule_raw`, a boundary node cannot parse or enforce confidential rate-limit rules. Any traffic those rules were designed to rate-limit passes through unchecked for the duration of the window. This constitutes a significant boundary/API security impact with concrete protocol harm: the rate-limiting enforcement layer is silently disabled for confidential rules on every routine upgrade. This matches the **High ($2,000â€“$10,000)** impact class: "Significant boundary/API security impact with concrete user or protocol harm."

## Likelihood Explanation
Every routine canister upgrade unconditionally triggers the window. No special attacker capability is required beyond sending traffic during a known or observable upgrade event â€” upgrades are observable on-chain. The window length equals `registry_polling_period_secs` from `InitArg`, which can range from tens of seconds to several minutes. The attacker does not need to perform the upgrade; they only need to send traffic during the window.

## Recommendation
1. **Immediate fix**: Replace the heap-only `HashSet` with a stable-memory-backed structure (e.g., a `StableBTreeMap<StorablePrincipal, ()>` using a new `MemoryId`) so the set survives upgrades, consistent with how `AUTHORIZED_PRINCIPAL` is stored.
2. **Defense-in-depth**: Add a `ic_cdk_timers::set_timer(Duration::ZERO, ...)` one-shot call in `init`/`post_upgrade` to trigger an immediate registry poll before the first interval fires, reducing any residual window to a single async round-trip.

## Proof of Concept
Integration test outline using PocketIC or the local replica:
1. Install the canister with a known API boundary node principal `P` and at least one non-disclosed rule (`disclosed_at = None`).
2. Call `get_config` from `P`; assert `is_redacted = false` and `rule_raw` is `Some(...)`.
3. Upgrade the canister (before the timer fires).
4. Immediately call `get_config` from `P` again (before `registry_polling_period_secs` elapses).
5. Assert `is_redacted = true` and `rule_raw = None` â€” confirming the downgrade window.
6. Wait for the polling interval to elapse and the timer to fire.
7. Call `get_config` from `P` again; assert `is_redacted = false` and `rule_raw` is restored â€” confirming the window is bounded.
