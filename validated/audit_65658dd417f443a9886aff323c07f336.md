### Title
Asymmetric `deltaGas` Adjustment in `compute_gas_refund` Causes Systematic User Overcharge - (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

`compute_gas_refund` adjusts `gas_used` upward when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), but never adjusts it downward when EVM gas consumption exceeds native resource consumption (`delta_gas < 0`). This asymmetry causes users to be systematically overcharged whenever their EVM gas usage exceeds what their native (proving) resource consumption would imply, with the excess flowing to the operator. A `// TODO: return delta_gas to gas_used?` comment in the code explicitly acknowledges this unresolved asymmetry.

### Finding Description

The `compute_gas_refund` function in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements the `deltaGas` reconciliation described in `docs/double_resource_accounting.md`:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
```

The code handles only the positive case:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` — meaning EVM gas consumption exceeds what native resource consumption implies — `gas_used` is left at the higher EVM-gas-derived value. The user is then charged `gas_used * gas_price` and refunded `(gas_limit - gas_used) * gas_price`, both computed from the inflated `gas_used`. The operator receives the excess.

The documentation in `docs/double_resource_accounting.md` only describes the positive-delta case and is silent on the negative-delta case, consistent with the TODO comment being an unresolved design question rather than an intentional choice.

This `gas_used` value propagates directly into the refund and operator payment in `ZkTransactionFlowOnlyEOA::refund_and_commit_fee`:

```rust
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
// ...
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
```

And for L1 transactions in `process_l1_transaction`:

```rust
let pay_to_operator = U256::from(gas_used)
    .checked_mul(U256::from(gas_price))
```

### Impact Explanation

Any transaction where EVM gas consumption exceeds native resource consumption results in the user paying more fees than their actual resource usage warrants. The operator receives the excess. Concretely:

- A computation-heavy, storage-light transaction (high EVM gas, low native/pubdata) will have `delta_gas < 0`. The user is charged for the full EVM gas rather than the lower native-implied gas equivalent.
- The user's refund is reduced by `|delta_gas| * gas_price` compared to what accurate accounting would produce.
- The operator receives `|delta_gas| * gas_price` in excess fees per such transaction.

This is a direct, repeatable financial loss for users and a corresponding gain for the operator on every transaction where EVM gas dominates native resource consumption.

### Likelihood Explanation

The condition `delta_gas < 0` is reachable by any unprivileged transaction sender submitting a standard L2 EIP-1559 or legacy transaction. It occurs whenever a transaction performs significant EVM computation without proportional storage writes or pubdata generation — a common pattern for pure-computation contracts (e.g., cryptographic operations, sorting, in-memory data processing). No special privileges, governance access, or external oracle manipulation is required. The entry path is `ZkTransactionFlowOnlyEOA::before_refund` → `compute_gas_refund`.

### Recommendation

Apply the negative `delta_gas` symmetrically: when `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so that the reported gas used accurately reflects the maximum of EVM gas and native-implied gas, but also the minimum when native consumption is lower. The existing `minimal_gas_used` guard already prevents undercharging below the intrinsic cost floor:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption is lower than EVM gas consumption.
    // Reduce gas_used, but not below the minimal floor.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

### Proof of Concept

1. Deploy a computation-heavy, storage-light EVM contract (e.g., a loop performing arithmetic with no `SSTORE` calls).
2. Submit a transaction calling it with a gas limit large enough to succeed.
3. After execution, observe that `native_used` is small (no pubdata) while `gas_used` is large (EVM computation).
4. Compute `delta_gas = (native_used / native_per_gas) - gas_used`. This will be negative.
5. Observe that the reported `gas_used` in the `ZkTxResult` equals the EVM-gas-derived value, not the lower native-implied value.
6. Compute `expected_refund = (gas_limit - (native_used / native_per_gas)) * gas_price` vs `actual_refund = (gas_limit - gas_used) * gas_price`. The actual refund is smaller by `|delta_gas| * gas_price`.
