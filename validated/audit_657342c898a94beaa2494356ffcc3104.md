### Title
Unbounded Filter Map Growth via Unauthenticated `eth_newBlockFilter` / `eth_newFilter` — (`evmrpc/filter.go`)

---

### Summary

`NewBlockFilter` and `NewFilter` insert entries into an unbounded `map[ethrpc.ID]filter` with no cap on the total number of live filters and no rate limiting on the creation calls themselves. An unauthenticated attacker can flood the endpoint with `eth_newBlockFilter` (or `eth_newFilter`) requests, growing the map without bound until the RPC node is killed by OOM.

---

### Finding Description

**Root cause — no filter count cap:**

`a.filters` is a plain Go map with no size limit: [1](#0-0) 

Every call to `NewBlockFilter` unconditionally inserts a new entry: [2](#0-1) 

`NewFilter` does the same: [3](#0-2) 

Neither function checks `len(a.filters)` before inserting. There is no `maxFilters` field in `FilterConfig` and no such constant anywhere in the codebase: [4](#0-3) 

**Root cause — no rate limiting on creation calls:**

The `globalRPSLimiter` (30 req/s) is only consulted inside `GetLogs`, and only for block-range queries exceeding 100 blocks: [5](#0-4) 

`NewBlockFilter` and `NewFilter` never call `a.globalRPSLimiter.Allow()`. There is no per-IP or global rate limit on filter creation.

**Cleanup is too slow to defend against a flood:**

The cleanup goroutine fires every `timeout/2`. The default `FilterTimeout` is 120 s, so cleanup runs every 60 s: [6](#0-5) [7](#0-6) 

An attacker can create millions of entries in the 60-second window between cleanup passes. Each entry allocates a `filter` struct plus a `cancelCtx` from `context.WithCancel` (line 430), so memory grows linearly with the number of requests.

---

### Impact Explanation

Each filter entry is ~200–400 bytes (struct fields + heap-allocated `cancelCtx`). At a modest 10 000 req/s (easily achievable over a local or co-located network), the map grows by ~2–4 MB/s. Within minutes the process exhausts available memory and is killed by the OS OOM killer, crashing the RPC node. No funds are at risk, but the node becomes unavailable for the duration of the attack.

---

### Likelihood Explanation

- The endpoint is unauthenticated and enabled by default (`HTTPEnabled: true`, port 8545).
- `eth_newBlockFilter` takes zero parameters — the request body is 47 bytes.
- No authentication, no per-IP throttle, no connection-level rate limit was found in `server.go`.
- The attack requires no on-chain funds and no special knowledge.

---

### Recommendation

1. **Add a hard cap on live filters.** Before inserting, check `len(a.filters)` against a configurable `MaxFilters` (e.g. 10 000 default). Return an error if the cap is reached.
2. **Apply the existing `globalRPSLimiter` to `NewBlockFilter` and `NewFilter`**, or add a dedicated per-call rate limiter for filter-creation methods.
3. Optionally add per-source-IP rate limiting at the HTTP layer.

---

### Proof of Concept

```python
import socket, json, itertools, time

HOST, PORT = "127.0.0.1", 8545
req = json.dumps({"jsonrpc":"2.0","id":1,"method":"eth_newBlockFilter","params":[]}).encode()
http = (b"POST / HTTP/1.1\r\nHost: localhost\r\n"
        b"Content-Type: application/json\r\n"
        b"Content-Length: " + str(len(req)).encode() + b"\r\n\r\n" + req)

created = 0
start = time.time()
for _ in itertools.count():
    s = socket.create_connection((HOST, PORT))
    s.sendall(http)
    s.recv(4096)
    s.close()
    created += 1
    if created % 10000 == 0:
        print(f"{created} filters in {time.time()-start:.1f}s — watch RSS grow")
```

Each iteration inserts one entry into the unbounded `a.filters` map. After several hundred thousand iterations the node's RSS climbs into the gigabytes and the process is OOM-killed.
