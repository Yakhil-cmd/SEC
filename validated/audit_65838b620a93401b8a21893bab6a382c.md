### Title
Unprivileged Caller Can DoS Arbitrary Victim Accounts via Attacker-Controlled `owner` in `balance_update_guard` — (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

### Summary
`update_balance` constructs the guard key from the caller-supplied `args.owner` field rather than from the authenticated `caller` principal. An unprivileged attacker can submit 100 concurrent `update_balance` calls each specifying a distinct victim principal as `owner`, filling `update_balance_accounts` to `MAX_CONCURRENT=100` with attacker-chosen accounts. Legitimate owners of those accounts are then denied service for the duration of the attacker's in-flight calls.

### Finding Description
In `update_balance` (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, lines 164–168), the account passed to `balance_update_guard` is:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),  // attacker-controlled
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

`args.owner` is an ingress-supplied argument with no authorization check — any principal can set it to any value. The guard is then held across the `get_utxos` inter-canister call (lines 175–183), which has non-trivial latency (Bitcoin canister round-trips). Because the IC allows multiple update calls to be in-flight concurrently at `await` points, an attacker can submit 100 calls in rapid succession, each with a distinct victim `owner`, before any of them complete. Each call acquires a guard slot for the victim's account.

`Guard::new` (guard.rs lines 45–60) enforces two conditions:
- `AlreadyProcessing` — if the exact account is already in the set
- `TooManyConcurrentRequests` — if `accounts.len() >= 100`

Both conditions are triggered against legitimate victims once the attacker's calls are in-flight.

### Impact Explanation
- Any of the 100 targeted victim principals calling `update_balance` for their own account receives `AlreadyProcessing` (their specific account is locked) or `TemporarilyUnavailable` (set is full).
- This blocks ckBTC minting for those accounts for the entire duration of the attacker's calls (bounded by Bitcoin canister response time, but the attacker can re-submit continuously to maintain the DoS).
- The attacker pays only their own cycles; victims cannot mint ckBTC regardless of their own actions.

### Likelihood Explanation
- No privileged access required — any IC principal can call `update_balance` with arbitrary `owner`.
- Bitcoin canister `get_utxos` latency is routinely several seconds, easily long enough for 100 sequential ingress submissions to all be in-flight simultaneously.
- The attacker can cycle through victim sets continuously to maintain persistent DoS.
- Cost to attacker is bounded by cycles for 100 inter-canister calls per rotation, which is economically feasible.

### Recommendation
Restrict the `owner` field: if `args.owner` is `Some(p)` and `p != caller`, either reject the call or — if third-party minting is intentional — key the guard on the **caller** principal rather than the **owner** account. The guard's purpose is to prevent duplicate concurrent work for a given account, but the rate-limiting and DoS-prevention property must be tied to the authenticated caller, not the attacker-supplied argument. A minimal fix:

```rust
// Only allow owner override if caller == owner, or use caller as the guard key
let guard_account = Account {
    owner: caller,  // always use authenticated caller for guard
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(guard_account)?;
```

If third-party calls (caller ≠ owner) must be supported, add a per-caller rate limit separate from the per-account guard.

### Proof of Concept
State-machine test outline:
1. Initialize minter with a mock Bitcoin canister that delays `get_utxos` responses.
2. From a single attacker principal, submit 100 `update_balance` calls each with `owner = Some(victim_N)` for N in 0..100.
3. Assert all 100 calls are in-flight (guards held, `update_balance_accounts.len() == 100`).
4. From `victim_0`'s own principal, call `update_balance(owner=None)` (which resolves to `victim_0`).
5. Assert the result is `Err(UpdateBalanceError::AlreadyProcessing)` or `Err(TemporarilyUnavailable("too many concurrent requests"))`.

This directly matches the existing test structure in `guard.rs` lines 141–162, which already demonstrates that filling the set to 100 blocks further entries — the missing piece is that the set can be filled by a single attacker using distinct victim `owner` values.
