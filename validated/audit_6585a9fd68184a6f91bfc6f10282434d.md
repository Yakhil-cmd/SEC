### Title
Permanent Token Loss in `ExitToNear` Precompile When `ft_transfer` Fails on Unregistered Receiver Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile accepts a fully user-controlled `receiver_account_id` (NEAR account) as the destination for bridged tokens. When the `error_refund` compile-time feature is absent from the production build, no callback is attached to the outgoing `ft_transfer` promise. If that promise fails — which happens whenever the receiver has not registered storage with the NEP-141 contract — the ERC-20 tokens (or ETH) that were already burned/deducted on the Aurora side are permanently lost with no recovery path.

---

### Finding Description

The `ExitToNear` precompile is invoked when an EVM user wants to bridge tokens from Aurora back to NEAR. The precompile accepts a raw byte string from the caller, parses it as a NEAR `AccountId`, and schedules an `ft_transfer` (or `ft_transfer_call`) promise to the NEP-141 contract.

**Parsing — only syntactic validation:**

`parse_recipient` in `engine-precompiles/src/native.rs` validates only that the string is a syntactically legal NEAR account ID. It does not verify that the account exists or has registered storage with the target NEP-141 contract. [1](#0-0) 

**Promise construction — no callback when `error_refund` is off:**

After parsing, the precompile builds a `PromiseCreateArgs` targeting the NEP-141 contract. A refund callback is only included when the `error_refund` feature flag is compiled in. In the default build (`default = ["std"]`), `refund` is hardcoded to `None`, and `transfer_near_args` is also `None` for the standard ERC-20 exit path, so `callback_args == ExitToNearPrecompileCallbackArgs::default()` evaluates to `true` and no callback is attached. [2](#0-1) 

**Feature flag is not in the default set:** [3](#0-2) [4](#0-3) 

**ERC-20 exit path — tokens burned before promise fires:**

In the ERC-20 exit flow, the ERC-20 contract burns the tokens and then calls the precompile. The burn is committed as part of EVM execution. The `ft_transfer` promise is only scheduled afterward. If `ft_transfer` panics (e.g., receiver not registered), the NEAR runtime reverts the NEP-141 state change but the EVM burn is already final. [5](#0-4) 

**Test explicitly confirms the loss:**

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` sends tokens to `"unregistered.near"` (which has no storage registration) and asserts that without `error_refund`, the ERC-20 balance is reduced by `FT_EXIT_AMOUNT` with no refund — confirming permanent loss. [6](#0-5) 

The same loss applies to the ETH (base token) exit path, confirmed by `test_exit_to_near_eth_refund`. [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls `exitToNear` with a `receiver_account_id` that has not registered storage with the NEP-141 contract:

1. ERC-20 tokens are burned (or ETH is deducted) in the EVM — this is irreversible.
2. The `ft_transfer` promise to the NEP-141 contract panics because the receiver is not registered.
3. Without `error_refund`, no callback exists to re-mint the ERC-20 tokens or restore the ETH balance.
4. The tokens are permanently lost — neither the original sender nor the intended receiver holds them.

---

### Likelihood Explanation

**Medium-High.**

NEAR NEP-141 tokens require explicit `storage_deposit` registration before any account can receive them. This is a standard NEAR requirement that many users are unaware of, especially EVM-native users unfamiliar with NEAR's storage model. A user who provides a valid-looking but unregistered NEAR account ID (e.g., a freshly created account, a typo, or an account that simply never called `storage_deposit` on that specific token) will trigger this loss. The entry path requires only a standard EVM transaction calling the `exitToNear` precompile — no special privileges needed.

---

### Recommendation

1. **Enable `error_refund` in the production build** by adding it to the `default` feature set in `engine/Cargo.toml`, so that a refund callback is always attached to the outgoing `ft_transfer` promise.
2. **Alternatively**, validate at precompile execution time that the `receiver_account_id` has registered storage with the target NEP-141 contract before burning tokens (requires a cross-contract view call or a storage-registration check).
3. **At minimum**, document clearly that users must ensure the receiver account has registered storage before calling `exitToNear`, and consider making the `error_refund` feature non-optional.

---

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora as an ERC-20.
2. From an Aurora EVM account, call the ERC-20's `withdraw` function (which calls the `exitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with:
   - Flag `0x01` (ERC-20 exit)
   - Amount: any nonzero amount
   - `receiver_account_id`: any syntactically valid NEAR account that has **not** called `storage_deposit` on the NEP-141 contract (e.g., `"fresh-account.near"`)
3. The ERC-20 tokens are burned in the EVM.
4. The `ft_transfer` promise to the NEP-141 contract panics because `"fresh-account.near"` is not registered.
5. Without `error_refund`, no callback fires. The ERC-20 balance is permanently reduced; the NEP-141 tokens remain with the Aurora engine contract; the user has lost their funds.

This is directly confirmed by the existing test:

fn exit_erc20_token_to_near<I: IO>(
    context: &Context,
    exit_params: &Erc20TokenParams,
    io: &I,
) -> Result<
    (
        AccountId,
        String,
        events::ExitToNear,
        String,
        Option<TransferNearArgs>,
    ),
    ExitError,
> {
    // In case of withdrawing ERC-20 tokens, the `apparent_value` should be zero. In opposite way
    // the funds will be locked in the address of the precompile without any possibility
    // to withdraw them in the future. So, in case if the `apparent_value` is not zero, the error
    // will be returned to prevent that.
    if context.apparent_value != U256::zero() {
        return Err(ExitError::Other(Cow::from(
            "ERR_ETH_ATTACHED_FOR_ERC20_EXIT",
        )));
    }

    let erc20_address = context.caller; // because ERC-20 contract calls the precompile.
    let nep141_account_id = get_nep141_from_erc20(erc20_address.as_bytes(), io)?;

    let (nep141_account_id, args, method, transfer_near_args, event) = match exit_params.message {
        // wNEAR address should be set via the `factory_set_wnear_address` transaction first.
        Some(Message::UnwrapWnear) if erc20_address == get_wnear_address(io).raw() =>
        // The flow is following here:
        // 1. We call `near_withdraw` on wNEAR account id on `aurora` behalf.
        // In such way we unwrap wNEAR to NEAR.
        // 2. After that, we call callback `exit_to_near_precompile_callback` on the `aurora`
        // in which make transfer of unwrapped NEAR to the `target_account_id`.
        {
            (
                nep141_account_id,
                format!(r#"{{"amount":"{}"}}"#, exit_params.amount.as_u128()),
                "near_withdraw",
                Some(TransferNearArgs {
                    target_account_id: exit_params.receiver_account_id.clone(),
                    amount: exit_params.amount.as_u128(),
                }),
                events::ExitToNear::Legacy(ExitToNearLegacy {
                    sender: Address::new(erc20_address),
                    erc20_address: Address::new(erc20_address),
                    dest: exit_params.receiver_account_id.to_string(),
                    amount: exit_params.amount,
                }),
            )
        }
        // In this flow, we're just forwarding the `msg` to the `ft_transfer_call` transaction.
        Some(Message::Omni(msg)) => (
            nep141_account_id,
            ft_transfer_call_args(&exit_params.receiver_account_id, exit_params.amount, msg)?,
            "ft_transfer_call",
            None,
            events::ExitToNear::Omni(ExitToNearOmni {
                sender: Address::new(erc20_address),
                erc20_address: Address::new(erc20_address),
                dest: exit_params.receiver_account_id.to_string(),
                amount: exit_params.amount,
                msg: msg.to_string(),
            }),
        ),
        // The legacy flow. Just withdraw the tokens to the NEAR account id.
        // P.S. We use underscore here instead of `None` to handle the case when a user
        // could add the `unwrap` suffix for non wNEAR ERC-20 token by mistake.
        _ => {
            // There is no way to inject json, given the encoding of both arguments
            // as decimal and valid account id respectively.
            (
                nep141_account_id,
                format!(
                    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
                    exit_params.receiver_account_id,
                    exit_params.amount.as_u128()
                ),
                "ft_transfer",
                None,
                events::ExitToNear::Legacy(ExitToNearLegacy {
                    sender: Address::new(erc20_address),
                    erc20_address: Address::new(erc20_address),
                    dest: exit_params.receiver_account_id.to_string(),
                    amount: exit_params.amount,
                }),
            )
        }
    };
