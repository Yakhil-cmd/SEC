### Title
One-Sided `delta_gas` Adjustment Causes Systematic User Overpayment in Gas Refund Calculation - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` value — which reconciles EVM gas consumption with native resource consumption — is computed as a signed `i64` that can be both positive and negative, but only the positive branch is applied to `gas_used`. The negative branch is silently dropped (with a developer `// TODO` comment acknowledging the omission). This mirrors the original report's `clip()` bug exactly: a bidirectional correction is clamped to one direction, causing a systematic one-sided bias in a critical accounting accumulator.

---

### Finding Description

In `compute_gas_refund`, ZKsync OS implements a dual-resource reconciliation step. After execution, it computes:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

`delta_gas` is defined as `(native_used / native_per_gas) - gas_used`. This is a signed quantity:

- **Positive** when native resource consumption implies more gas than EVM gas tracking recorded → user should pay more → `gas_used` is increased. ✓ Applied.
- **Negative** when native resource consumption implies less gas than EVM gas tracking recorded → user should pay less (larger refund) → `gas_used` should be decreased. ✗ **Silently ignored.**

The `// TODO: return delta_gas to gas_used?` comment at line 80 is a developer acknowledgment that the negative case is unhandled.

The final refund is computed as:

```rust
let total_gas_refund = gas_limit - gas_used;
```

Because `gas_used` is never reduced when `delta_gas < 0`, the refund is systematically understated whenever EVM gas consumption exceeds native resource consumption. [1](#0-0) 

The documentation in `docs/double_resource_accounting.md` confirms the intended bidirectional semantics: `deltaGas := (nativeUsed / nativePerGas) - gasUsed` — but the implementation only acts on the positive half. [2](#0-1) 

---

### Impact Explanation

When `delta_gas < 0` (i.e., `native_used / native_per_gas < gas_used`), the user's `gas_used` is left inflated relative to what native resource consumption justifies. This means:

1. `total_gas_refund = gas_limit - gas_used` is **smaller than it should be**.
2. The user receives **less ETH/token refund** than they are entitled to.
3. The operator receives **excess fee payment** (`gas_used * gas_price`) beyond what the protocol intends.

This is a direct, quantifiable loss of user funds on every transaction where EVM gas consumption exceeds native resource consumption — for example, transactions that perform many EVM opcodes that are gas-expensive but proving-cheap (simple arithmetic, memory reads, etc.).

The magnitude of loss per transaction is: `|delta_gas| * gas_price` tokens, silently transferred to the operator instead of refunded to the user.

---

### Likelihood Explanation

The condition `delta_gas < 0` is triggered whenever a transaction's EVM gas consumption exceeds its native resource consumption (normalized by `native_per_gas`). This is a realistic and common scenario:

- Any transaction that uses EVM gas for operations with low RISC-V proving cost (e.g., memory-heavy but arithmetically simple operations) will produce `native_used / native_per_gas < gas_used`.
- Any unprivileged L2 transaction sender triggers this path via `before_refund` → `compute_gas_refund`.
- No special privileges, oracle manipulation, or governance access is required. [3](#0-2) 

---

### Recommendation

Apply `delta_gas` bidirectionally. When `delta_gas < 0`, reduce `gas_used` (increase the refund), subject to the `minimal_gas_used` floor already enforced by `core::cmp::max(gas_used, minimal_gas_used)`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native resource consumption implies less gas than EVM tracking recorded.
    // Return the difference to the user as a larger refund.
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

This resolves the `// TODO: return delta_gas to gas_used?` comment and makes the adjustment symmetric, matching the documented intent in `docs/double_resource_accounting.md`.

### Proof of Concept

**Setup:**
- `native_price = 1000`, `gas_price = 1000` → `native_per_gas = gas_price / native_price = 1`
- `gas_limit = 100_000`
- Transaction uses `gas_used_ergs = 80_000` EVM gas (ergs-based)
- Transaction uses `native_used = 50_000` native units (proving-cheap operations)

**Computation in `compute_gas_refund`:**
```
gas_used = gas_limit - remaining_ergs/ERGS_PER_GAS = 80_000
native_used = 50_000
delta_gas = (50_000 / 1) - 80_000 = -30_000   // negative
```

**Current behavior (buggy):**
- `delta_gas < 0` → branch not taken → `gas_used` stays at `80_000`
- `total_gas_refund = 100_000 - 80_000 = 20_000`
- User refunded for 20,000 gas units

**Correct behavior:**
- `gas_used` should be reduced by 30,000 → `gas_used = 50_000`
- `total_gas_refund = 100_000 - 50_000 = 50_000`
- User refunded for 50,000 gas units

**Loss:** User loses `30_000 * gas_price = 30_000 * 1000 = 30,000,000` token units per transaction, silently paid to the operator. [4](#0-3)
