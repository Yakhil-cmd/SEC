Audit Report

## Title
Unchecked Index Panic on Empty `EnvelopePair` Vec in `construction_hash` Crashes Rosetta ICP Process — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

## Summary
The `construction_hash` handler unconditionally accesses `envelope_pairs[0]` after finding a `RequestType::Send` entry via `find()`, without verifying the inner `Vec<EnvelopePair>` is non-empty. An unauthenticated attacker can POST a single crafted hex-CBOR `signed_transaction` containing `(RequestType::Send, vec![])` to `/construction/hash`, triggering a Rust index-out-of-bounds panic that terminates the Rosetta ICP process.

## Finding Description
`SignedTransaction::from_str` in `rs/rosetta-api/icp/src/models.rs` (lines 37–48) performs only CBOR deserialization with no structural validation — it accepts any well-formed CBOR, including entries with an empty `Vec<EnvelopePair>`. The type alias `pub type Request = (RequestType, Vec<EnvelopePair>)` (line 58) places no non-empty constraint on the inner vec.

In `construction_hash` (lines 18–28 of `rs/rosetta-api/icp/src/request_handler/construction_hash.rs`), the handler calls `.find(|(rt, _)| rt.is_transfer())` over the deserialized requests. `RequestType::Send` satisfies `is_transfer()`, so `find()` succeeds and returns the tuple `(request_type, envelope_pairs)` where `envelope_pairs` is the empty vec. The handler then unconditionally executes:

```rust
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pairs[0].update,  // panics: index 0 on empty Vec
)
```

This is a direct `Index` trait call on a `Vec`, which panics on out-of-bounds. There is no `.get(0).ok_or_else(...)` guard. By contrast, the `TryFrom<&models::Request> for Request` conversion in `rs/rosetta-api/icp/src/request.rs` (lines 237–239) correctly handles this with:

```rust
let payload: &models::EnvelopePair = calls
    .first()
    .ok_or_else(|| ApiError::invalid_request("No request payload provided."))?;
```

The `construction_hash` handler does not go through that path and has no equivalent guard.

## Impact Explanation
The Rosetta ICP node is a standalone HTTP server process. A Rust panic with the default panic handler (or under `panic = "abort"`, which is typical for production binaries) terminates the process immediately. A single malformed request crashes the entire Rosetta node, making it unavailable to all clients until manually restarted. This is a concrete, non-volumetric, single-request denial-of-service against the Rosetta API, which is an explicitly in-scope financial integration component. This matches the **High** impact class: "Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS" and "Significant... Rosetta... security impact with concrete user or protocol harm."

## Likelihood Explanation
The `/construction/hash` endpoint requires no authentication. The attacker needs only network access to the Rosetta node and the ability to POST JSON. Crafting the payload is trivial: serialize `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` with `serde_cbor`, hex-encode it, and submit it as the `signed_transaction` field. No keys, credentials, or privileged access are required. The attack is repeatable — the process will crash on every such request after restart.

## Recommendation
Replace the unchecked index at `construction_hash.rs` line 27 with a bounds-checked access that returns an `ApiError` on empty input, mirroring the existing safe pattern in `request.rs`:

```rust
let envelope_pair = envelope_pairs.first().ok_or_else(|| {
    ApiError::invalid_request("No envelope pairs provided for transfer request.")
})?;
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pair.update,
)
```

## Proof of Concept
```rust
#[test]
fn test_construction_hash_empty_envelope_pairs_panics() {
    use std::str::FromStr;
    use crate::models::{SignedTransaction, Request};
    use crate::request_types::RequestType;

    // Craft a SignedTransaction with RequestType::Send but empty EnvelopePairs
    let malicious = SignedTransaction {
        requests: vec![(RequestType::Send, vec![])],
    };
    let hex = malicious.to_string(); // hex-encoded CBOR

    let signed = SignedTransaction::from_str(&hex).unwrap();
    // find() succeeds because RequestType::Send satisfies is_transfer()
    let result = std::panic::catch_unwind(|| {
        let (_, envelope_pairs) = signed.requests.iter()
            .rev()
            .find(|(rt, _)| rt.is_transfer())
            .unwrap();
        let _ = &envelope_pairs[0]; // panics here
    });
    assert!(result.is_err(), "Index out of bounds panic confirmed");
}
```

Under `panic = "abort"` (production), `catch_unwind` does not apply — the process terminates immediately upon the first such request.
