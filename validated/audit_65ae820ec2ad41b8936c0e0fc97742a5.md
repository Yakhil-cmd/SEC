Audit Report

## Title
SNS-Specific Upgrade Path Entry Cannot Be Updated After Initial Set — (`rs/nns/sns-wasm/src/sns_wasm.rs`)

## Summary
`insert_sns_specific_upgrade_path_entry` in the `UpgradePath` impl silently discards all update attempts on an already-occupied entry: the `Entry::Occupied` branch logs an overwrite message but never calls `occupied.insert(to)`, so the first value written is permanently frozen. The caller returns a success response regardless, so every subsequent NNS Governance proposal to correct an SNS-specific upgrade path silently no-ops while reporting success.

## Finding Description
In `rs/nns/sns-wasm/src/sns_wasm.rs` at lines 1939–1947, the `Entry::Occupied` arm of `insert_sns_specific_upgrade_path_entry` only prints a log line and exits without mutating the map. The `mut` keyword is absent from the `occupied` binding, making it impossible to call `occupied.insert(to)`. The sibling function `insert_upgrade_path_entry` (lines 1956–1963) correctly declares `mut occupied` and calls `occupied.insert(to)`. The public entry point `insert_upgrade_path_entries` (lines 610–617) delegates to the broken function for every SNS-specific step and unconditionally returns `InsertUpgradePathEntriesResponse { error: None }` at line 627, giving callers no indication that the state was not changed.

## Impact Explanation
The SNS-specific upgrade path is the designated on-chain escape hatch for SNSes that cannot follow the standard upgrade path due to a bug (proto comment: "provides an escape hatch for if a particular SNS somehow has a bug that prevents upgrading on the standard path"). Once a `(governance_canister_id, current_version)` entry is set — even incorrectly — it can never be corrected via any on-chain mechanism. The affected SNS is permanently locked to the wrong `next_version` with no remedy. This constitutes a significant SNS governance impact with concrete, irreversible harm to SNS upgrade availability, matching the High bounty impact: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
Triggering the bug requires only two sequential NNS Governance `InsertUpgradePathEntries` proposals targeting the same `(sns_governance_canister_id, current_version)` pair — a routine operational sequence whenever an operator needs to correct a previously submitted path entry. No special privileges beyond the normal NNS proposal flow are required. The misleading log message ("is being overwritten") actively conceals the failure, making silent corruption the default outcome.

## Recommendation
Add `occupied.insert(to)` in the `Entry::Occupied` branch of `insert_sns_specific_upgrade_path_entry`, mirroring `insert_upgrade_path_entry`:

```rust
Entry::Occupied(mut occupied) => {
    println!(
        "Special Entry for {} from {:?} to {:?} is being overwritten with new value {:?}",
        sns_governance_canister_id,
        occupied.key(),
        occupied.get(),
        to
    );
    occupied.insert(to);  // ← add this line
}
```

## Proof of Concept
```rust
let mut path = UpgradePath::default();
let gov_id = CanisterId::from(1);
path.insert_sns_specific_upgrade_path_entry(from.clone(), to_v1.clone(), gov_id);
path.insert_sns_specific_upgrade_path_entry(from.clone(), to_v2.clone(), gov_id);
// Currently returns to_v1 (first-set value); correct behavior returns to_v2.
assert_eq!(path.get_next_version(from, gov_id.into()), Some(to_v2));
```
This unit test can be added directly to the existing test module in `rs/nns/sns-wasm/src/sns_wasm.rs` and will fail on the current code, confirming the bug.
