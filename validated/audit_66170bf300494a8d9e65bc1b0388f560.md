Audit Report

## Title
Ed25519 Small-Order Public Key Accepted in COSE WebAuthn Path Allows Signature Forgery — (`rs/crypto/internal/crypto_lib/basic_sig/cose/src/lib.rs`)

## Summary

`parse_eddsa_ed25519` calls `ic_ed25519::PublicKey::deserialize_raw` which, by documented design, accepts small-order (torsion) points without error and without any subsequent `is_torsion_free()` guard. The `verify_signature` implementation uses ZIP215 cofactor-8 verification (`mul_by_cofactor().is_identity()`), which collapses to a message-independent identity check when the public key is a small-order point. An unprivileged attacker can therefore forge a valid WebAuthn ingress signature for any self-authenticating principal derived from a small-order key, enabling them to call any canister as that principal and drain any funds it holds.

## Finding Description

**Root cause — COSE parsing accepts small-order keys.**

`parse_eddsa_ed25519` at `rs/crypto/internal/crypto_lib/basic_sig/cose/src/lib.rs` lines 188–195 calls `ic_ed25519::PublicKey::deserialize_raw(x)` and maps errors to `MalformedPublicKey`, but adds no `is_torsion_free()` guard after a successful parse:

```rust
let pk = ic_ed25519::PublicKey::deserialize_raw(x).map_err(|_| {
    CosePublicKeyParseError::MalformedPublicKey(AlgorithmId::Ed25519)
})?;
let der = pk.serialize_rfc8410_der();
Ok(Self::Ed25519(der))
```

The `deserialize_raw` documentation at `packages/ic-ed25519/src/lib.rs` lines 609–618 explicitly warns it does not check torsion. The library test at `packages/ic-ed25519/tests/tests.rs` lines 517–545 confirms all 18 known small-order/torsion points return `Ok` from `deserialize_raw` while `is_torsion_free()` returns `false`.

**Verification equation breaks for small-order keys.**

`verify_signature` at `packages/ic-ed25519/src/lib.rs` lines 709–727 implements:

```
(k·(−A) + s·B − R) · 8 == identity
```

where `k = H(R ∥ A ∥ M)`. If `A` is a small-order point (order divides 8), then `k·(−A)·8 = identity` for **any** scalar `k`. The check reduces to `(s·B − R)·8 == identity`, which is independent of the message. Choosing `s = 0` and `R` = any small-order point satisfies it: `(0·B − R)·8 = (−R)·8 = identity`.

**`Signature::from_slice` accepts this forged input.** At lines 449–472, `CompressedEdwardsY(r_bytes).decompress()` succeeds for valid small-order point encodings (confirmed by the torsion test), and `Scalar::from_canonical_bytes` accepts `s = 0` (zero is a canonical scalar, strictly less than the group order `l`).

**Full ingress path is reachable without any torsion check.**

`user_public_key_from_bytes` in `rs/crypto/standalone-sig-verifier/src/sign_utils.rs` lines 87–97 handles the COSE OID branch by calling `cose::parse_cose_public_key` (which calls `parse_eddsa_ed25519`) and then recursively calling itself on the resulting DER. The recursive call hits the `ed25519_algorithm_identifier()` branch at lines 50–62, which calls `deserialize_rfc8410_der` — also documented as not checking torsion — and stores the raw small-order bytes in `UserPublicKey.key`. Neither call site invokes `verify_public_key` (the function in `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs` lines 97–103 that correctly calls `is_torsion_free()`).

`verify_basic_sig_by_public_key` in `rs/crypto/standalone-sig-verifier/src/lib.rs` lines 20–44 also calls `deserialize_raw` without a torsion check before invoking `pk.verify_signature`.

**Ingress routing and principal check both pass.**

`validate_signature` in `rs/validator/src/ingress_validation.rs` lines 659–669 routes `Ed25519PublicKeyDerWrappedCose` to `validate_webauthn_sig`. That function at `rs/validator/src/webauthn.rs` lines 37–47 checks only that `envelope.challenge() == signed_bytes` — the attacker constructs the envelope themselves and sets the challenge to the correct message ID. `validate_user_id` at `rs/validator/src/ingress_validation.rs` lines 625–632 checks only that `sender == PrincipalId::new_self_authenticating(sender_pubkey)`, which passes trivially when the attacker sets the sender to the self-authenticating principal derived from the small-order key.

## Impact Explanation

An attacker can forge ingress signatures for any self-authenticating principal whose public key is one of the 18 known small-order Ed25519 points. They can call any canister as that principal, drain any ICP or token balance it holds, and exercise any canister permission granted to it. The attacker can create such a principal and fund it through normal protocol interactions (e.g., token faucets, ICRC-1 transfers from their own account) without social engineering. This matches **High ($2,000–$10,000): Unauthorized access to canister-controlled funds where exploitation requires meaningful per-target work or other constraints.**

## Likelihood Explanation

The attack requires no privileged access, no key material, and no network-level capability. It requires only: (1) knowledge of one of the 18 publicly documented small-order Ed25519 point encodings, (2) ability to submit a standard WebAuthn ingress message, and (3) construction of a 64-byte forged signature `(R_small_order ∥ 0x00…00)`. It is fully reproducible in a local replica or PocketIC environment.

## Recommendation

Add an `is_torsion_free()` guard in `parse_eddsa_ed25519` in `rs/crypto/internal/crypto_lib/basic_sig/cose/src/lib.rs` immediately after `deserialize_raw` succeeds:

```rust
if !pk.is_torsion_free() {
    return Err(CosePublicKeyParseError::MalformedPublicKey(AlgorithmId::Ed25519));
}
```

Add the same guard in the `AlgorithmId::Ed25519` branch of `verify_basic_sig_by_public_key` in `rs/crypto/standalone-sig-verifier/src/lib.rs` after `deserialize_raw` succeeds. The existing `verify_public_key` function in `rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs` already implements the correct check but is not called on the COSE ingress path.

## Proof of Concept

```rust
// 1. Pick any of the 18 known small-order points
let small_order_pk: [u8; 32] =
    hex!("c7176a703d4dd84fba3c0b760d10670f2a2053fa2c39ccc64ec7fd7792ac03fa");

// 2. Encode as COSE: {1:1, 3:-8, -1:6, -2:<pk>}
let cose_key: Vec<u8> = build_cose_ed25519_key(&small_order_pk);

// 3. DER-wrap with IC COSE OID 1.3.6.1.4.1.56387.1.1
let der_wrapped = der_wrap_cose(&cose_key);

// 4. Derive the self-authenticating principal
let principal = PrincipalId::new_self_authenticating(&der_wrapped);

// 5. Forged signature: R = small_order_pk, s = 0x00..00
let mut forged_sig = [0u8; 64];
forged_sig[..32].copy_from_slice(&small_order_pk);

// 6. Construct WebAuthn envelope with challenge = message_id.as_signed_bytes()
// 7. Submit ingress: sender=principal, sig=webauthn_envelope(forged_sig)
// -> verify_signature returns Ok(()) for any message because
//    (0·B − R)·8 = (−R)·8 = identity when R is a small-order point
```

A deterministic integration test using PocketIC or the state-machine test harness can confirm `validate_request` returns `Ok` for this forged ingress against any canister.
