### Title
Hardcoded NEAR Gas Stipend in `ExitToNear` Precompile Causes Permanent Fund Loss on Promise Failure - (File: `engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile attaches a hardcoded `FT_TRANSFER_GAS = 10 TGas` to the NEAR `ft_transfer` promise it creates when bridging ERC-20 tokens from Aurora to NEAR. This is the direct analog of Solidity's deprecated `transfer` with its fixed 2300 gas stipend. When the `error_refund` feature is not compiled in (it is absent from the `default` feature set), no callback is attached to the promise. If the NEP-141 contract's `ft_transfer` execution exhausts the 10 TGas stipend, the promise fails silently while the ERC-20 tokens have already been burned on the EVM side — resulting in permanent, unrecoverable fund loss.

### Finding Description

In `engine-precompiles/src/native.rs`, the `costs` module defines:

```rust
pub(super) const FT_TRANSFER_GAS: NearGas = NearGas::new(10_000_000_000_000); // 10 TGas
pub(super) const EXIT_TO_NEAR_CALLBACK_GAS: NearGas = NearGas::new(10_000_000_000_000); // 10 TGas
``` [1](#0-0) 

The `ExitToNear::run` method constructs the outbound NEAR promise with this fixed stipend:

```rust
let transfer_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method,
    args: args.into_bytes(),
    attached_balance: Yocto::new(1),
    attached_gas,   // ← hardcoded FT_TRANSFER_GAS or FT_TRANSFER_CALL_GAS
};
``` [2](#0-1) 

The callback (which would refund tokens on failure) is only attached when `callback_args != default()`. The `refund` field of `callback_args` is only populated under the `error_refund` compile-time feature:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← NO callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

Critically, `error_refund` is **not** in the `default` feature set:

```toml
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

For standard ERC-20 exits (non-wNEAR), `transfer_near_args` is also `None`, so `callback_args == default()` and the bare `PromiseArgs::Create` path is taken — no callback, no refund.

A secondary issue exists even when `error_refund` is enabled: the callback itself is allocated only `EXIT_TO_NEAR_CALLBACK_GAS = 10 TGas` to execute `refund_on_error`, which in turn runs a full EVM call (`engine.call(..., u64::MAX, ...)`) to re-mint the burned ERC-20 tokens. [5](#0-4) 

If the EVM execution inside the callback consumes more than 10 TGas of NEAR gas, the refund itself fails, leaving the user's tokens permanently lost even when `error_refund` is compiled in.

### Impact Explanation

The ERC-20 token burn on the EVM side is committed atomically within the NEAR transaction that executes the EVM call. The outbound `ft_transfer` NEAR promise is asynchronous. If that promise fails due to the 10 TGas stipend being exhausted:

- Without `error_refund`: no callback exists → burned ERC-20 tokens are permanently unrecoverable → **permanent fund loss (Critical)**.
- With `error_refund` but callback gas exhausted: the refund EVM call inside `exit_to_near_precompile_callback` fails → same permanent fund loss outcome.

This matches the **permanent freezing of funds** impact category.

### Likelihood Explanation

- Any NEP-141 token contract whose `ft_transfer` implementation requires more than 10 TGas (e.g., contracts with complex storage accounting, cross-contract hooks, or upgraded implementations) triggers this path.
- The `error_refund` feature is not in `default`, so any deployment compiled without explicitly enabling it is fully exposed.
- An unprivileged EVM user can trigger this by calling the `ExitToNear` precompile with any ERC-20 token mapped to such a NEP-141 contract.

### Recommendation

1. Replace the hardcoded `FT_TRANSFER_GAS` stipend with a dynamic value derived from the remaining prepaid NEAR gas at the time of promise creation (following the pattern already used in `calculate_attached_gas` in `connector.rs`): [6](#0-5) 

2. Ensure `error_refund` is always enabled in production builds, or make the callback unconditional.
3. Increase `EXIT_TO_NEAR_CALLBACK_GAS` to a value that accounts for the full EVM execution cost of `refund_on_error`, or derive it dynamically from remaining gas.
4. Add an explicit TODO resolution for `WITHDRAWAL_GAS` in `ExitToEthereum` (currently marked `// TODO(#332): Determine the correct amount of gas`), which has the same structural problem but no callback at all. [7](#0-6) 

### Proof of Concept

1. Deploy a NEP-141 token contract whose `ft_transfer` method performs storage-intensive operations requiring >10 TGas.
2. Bridge that token into Aurora as an ERC-20 (via `ft_on_transfer`).
3. From an EVM account, call the ERC-20 contract's `withdraw`/`burn` function, which internally calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`).
4. The ERC-20 tokens are burned on the EVM side and the NEAR transaction commits.
5. The asynchronous `ft_transfer` promise to the NEP-141 contract runs out of its 10 TGas stipend and fails.
6. With `error_refund` disabled (the default), no callback fires. The user's ERC-20 tokens are gone from Aurora and no NEP-141 tokens are received on NEAR — permanent fund loss.
