### Title
Silo Whitelist Bypass via `call()` and `deploy_code()` Skipping `assert_access()` - (File: `engine/src/contract_methods/evm_transactions.rs`)

### Summary

The Aurora Engine's silo mode enforces a whitelist (`Account` + `Address` for submissions, `Admin` + `EvmAdmin` for deployments) via `assert_access()`. However, the `call()` and `deploy_code()` contract methods never invoke `assert_access()`, allowing any unprivileged NEAR account to execute EVM calls or deploy EVM code in a silo deployment even when all whitelists are enabled.

### Finding Description

The `submit` and `submit_with_args` methods both route through `engine::submit()` → `submit_with_alt_modexp()`, which calls `assert_access()` before any EVM execution: [1](#0-0) 

`assert_access()` enforces the silo whitelist: [2](#0-1) 

`is_allow_submit` requires both the NEAR `Account` whitelist and the EVM `Address` whitelist to pass: [3](#0-2) 

`is_allow_deploy` requires both the `Admin` and `EvmAdmin` whitelists: [4](#0-3) 

However, the `call()` method constructs an `Engine` directly and calls `engine.call_with_args()` with **no call to `assert_access()`**: [5](#0-4) 

Similarly, `deploy_code()` calls `engine.deploy_code_with_input()` with **no call to `assert_access()`**: [6](#0-5) 

A grep across all production engine source confirms `assert_access`, `is_allow_submit`, and `is_allow_deploy` are referenced only inside `engine.rs` (the `submit_with_alt_modexp` path) and `silo/mod.rs` — never inside `call()` or `deploy_code()`. [7](#0-6) 

### Impact Explanation

In a silo deployment with all four whitelists enabled, the operator's intent is that only approved NEAR accounts and EVM addresses may submit transactions or deploy code. Because `call()` and `deploy_code()` skip `assert_access()`, any NEAR account — regardless of whitelist status — can:

1. **Call any EVM contract** via `call()`, using their NEAR-derived EVM address as `msg.sender`. If the silo hosts EVM contracts that hold user funds and rely on the silo whitelist as their sole access gate, an attacker can drain those contracts.
2. **Deploy arbitrary EVM bytecode** via `deploy_code()` without being in the `Admin`/`EvmAdmin` whitelist, enabling malicious contract deployment.

Impact: **Critical — direct theft of user funds** (for `call()`) and **Critical — unauthorized code deployment** (for `deploy_code()`).

### Likelihood Explanation

- The `call` and `deploy_code` NEAR contract methods are publicly callable by any NEAR account with no prerequisite.
- Silo mode is an explicitly supported production feature with documented whitelist semantics.
- The bypass requires no special privileges, no leaked keys, and no governance capture — only a standard NEAR function call.
- Likelihood: **High**.

### Recommendation

Add `assert_access()` (or an equivalent silo whitelist check) at the top of both `call()` and `deploy_code()` in `engine/src/contract_methods/evm_transactions.rs`, analogous to how `submit_with_alt_modexp()` enforces it. For `call()`, the check should use `is_allow_submit` (since `CallArgs` targets an existing contract). For `deploy_code()`, it should use `is_allow_deploy`.

### Proof of Concept

1. Deploy Aurora Engine in silo mode: enable all four whitelists (`Admin`, `EvmAdmin`, `Account`, `Address`).
2. Deploy an EVM contract holding ETH/ERC-20 tokens via the owner account (whitelisted).
3. From a **non-whitelisted** NEAR account, call the NEAR method `call` with `CallArgs` targeting the fund-holding EVM contract.
4. Observe that the call succeeds (no `NotAllowed` error), bypassing the silo whitelist entirely.
5. Confirm that calling `submit` from the same non-whitelisted account correctly returns `EngineErrorKind::NotAllowed`.

The root cause is the absence of `assert_access(&io, env, ...)` in `call()` at `engine/src/contract_methods/evm_transactions.rs:46` and `deploy_code()` at line 21, while `submit_with_alt_modexp()` at `engine/src/engine.rs:1052` correctly enforces it. [8](#0-7) [9](#0-8)

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
```

```rust
use aurora_engine_types::{
    borsh,

use function_name::named;

#[named]
pub fn deploy_code<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let input = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&env.predecessor_account_id()),
            current_account_id,
            io,
            env,
        );
        let result = engine.deploy_code_with_input(input, None, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}

#[named]
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}
```
