### Title
Missing G2 Subgroup Check in BN254 Pairing Precompile Allows EVM Semantic Mismatch — (`File: basic_system/src/system_functions/bn254_pairing_check.rs`)

---

### Summary

The `bn254_pairing_check_inner` function validates G2 input points using only `g2_point.check()`, which verifies on-curve membership but **not** prime-order subgroup membership. An attacker can supply a G2 point that lies on the BN254 twist curve E'(Fq2) but outside the prime-order subgroup, causing the precompile to compute and return an incorrect pairing result. This diverges from the EIP-197 specification and from Ethereum mainnet behavior, constituting an EVM semantic mismatch / state-transition bug.

---

### Finding Description

In `bn254_pairing_check_inner`, each G2 point is constructed with `G2Affine::new_unchecked` and then validated with `g2_point.check()`: [1](#0-0) 

The `check()` method (from the `ark_serialize::Valid` trait) only verifies that the point satisfies the curve equation y² = x³ + b over Fq2. It does **not** call `is_in_correct_subgroup_assuming_on_curve()`.

The subgroup check method is fully implemented for BN254 G2 in the codebase: [2](#0-1) 

It is never invoked from the pairing precompile path.

By contrast, the BLS12-381 pairing precompile (EIP-2537) correctly calls `parse_g2_with_subgroup_check`, which explicitly gates on `is_in_correct_subgroup_assuming_on_curve()`: [3](#0-2) 

The BN254 G2 group has a large cofactor h = 21888242871839275222246405745257275088844257914179612981679871602714643921549. Points on E'(Fq2) that are not in the prime-order subgroup pass the on-curve check but produce incorrect pairing values. EIP-197 requires G2 points to be in the correct subgroup; inputs that fail this check should cause the precompile to return failure (0 or revert), not proceed with computation.

---

### Impact Explanation

**Type:** EVM semantic mismatch / state-transition bug.

A G2 point outside the prime-order subgroup will pass `check()` and proceed to `Bn254::multi_pairing`. The pairing result for such a point is not guaranteed to equal 1 (the GT identity) even when the "intended" pairing equation holds, and conversely may equal 1 when it should not. This means:

- A pairing check that should return `0` (invalid) may return `1` (valid), or vice versa.
- ZKsync OS state transitions diverge from Ethereum mainnet for the same calldata.
- Smart contracts relying on `ecPairing` (address `0x08`) for cryptographic verification (e.g., ZK proof verifiers, BLS signature aggregation) can be made to accept forged proofs or reject valid ones. [4](#0-3) 

---

### Likelihood Explanation

**High.** The attacker entry path is a standard unprivileged EVM transaction calling the `ecPairing` precompile at address `0x08`. No privileged access is required. Constructing a BN254 G2 point that is on the curve but outside the prime-order subgroup is straightforward: take any generator of E'(Fq2) and multiply by a scalar that is not a multiple of the prime order r (e.g., multiply by the cofactor h to get a point of order r, or pick a random point and do not clear the cofactor). The `is_in_correct_subgroup_assuming_on_curve` method already exists in the codebase, confirming the check is known and intentional elsewhere.

---

### Recommendation

After constructing the G2 point and calling `g2_point.check()`, add an explicit subgroup membership check:

```rust
let g2_point = if g2_x.is_zero() && g2_y.is_zero() {
    G2Affine::zero()
} else {
    let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
    g2_point.check().map_err(|_| ())?;
    // ADD THIS:
    if !g2_point.is_in_correct_subgroup_assuming_on_curve() {
        return Err(());
    }
    g2_point
};
```

This mirrors the pattern already used in the BLS12-381 pairing precompile (`parse_g2_with_subgroup_check`) and the existing `is_in_correct_subgroup_assuming_on_curve` implementation for BN254 G2. [1](#0-0) 

---

### Proof of Concept

1. Construct a point Q on E'(Fq2) that is on the BN254 twist curve but not in the prime-order subgroup. One method: take the standard G2 generator and multiply by a scalar s such that s is not divisible by r (the prime order), but the resulting point still satisfies the curve equation. Alternatively, use a point of order h (the cofactor) by multiplying the generator by r.

2. Encode Q in Ethereum ABI format (128 bytes: x_c1 || x_c0 || y_c1 || y_c0, each 32 bytes big-endian) and prepend a valid G1 point (e.g., the G1 generator encoded as 64 bytes).

3. Call the `ecPairing` precompile (0x08) with this 192-byte input on ZKsync OS.

4. ZKsync OS: `g2_point.check()` passes (Q is on the curve), pairing proceeds, returns some result R.

5. Ethereum mainnet: the same input is rejected (G2 point not in subgroup), precompile returns `0x00...00`.

6. The two systems return different values for identical calldata, demonstrating the state-transition divergence. [5](#0-4)

fn bn254_pairing_check_inner<A: Allocator>(
    num_pairs: usize,
    src: &[u8],
    allocator: A,
) -> Result<bool, ()> {
    use crypto::ark_ec::pairing::Pairing;
    use crypto::ark_ff::{One, PrimeField};
    use crypto::bn254::curves::{Bn254, G1Affine, G2Affine};
    use crypto::bn254::fields::{Fq, Fq2};

    if num_pairs == 0 {
        return Ok(true);
    }

    let mut pairs = Vec::with_capacity_in(num_pairs, allocator);
    let mut src_iter = src.iter();

    for _ in 0..num_pairs {
        let mut buffer = [0u8; 192];
        for (dst, src) in buffer.iter_mut().zip(&mut src_iter) {
            *dst = *src;
        }
        let mut it = buffer.as_chunks::<32>().0.iter();
        unsafe {
            let mut g1_x = *it.next().unwrap_unchecked();
            let mut g1_y = *it.next().unwrap_unchecked();

            // NOTE: Ethereum serialization is strange
            let mut g2_x_c1 = *it.next().unwrap_unchecked();
            let mut g2_x_c0 = *it.next().unwrap_unchecked();
            let mut g2_y_c1 = *it.next().unwrap_unchecked();
            let mut g2_y_c0 = *it.next().unwrap_unchecked();

            bytereverse(&mut g1_x);
            bytereverse(&mut g1_y);

            let g1_x =
                <Fq as PrimeField>::BigInt::deserialize_uncompressed(&g1_x[..]).map_err(|_| ())?;
            let g1_y =
                <Fq as PrimeField>::BigInt::deserialize_uncompressed(&g1_y[..]).map_err(|_| ())?;
            let g1_x = Fq::from_bigint(g1_x).ok_or(())?;
            let g1_y = Fq::from_bigint(g1_y).ok_or(())?;

            let g1_point = if g1_x.is_zero() && g1_y.is_zero() {
                G1Affine::zero()
            } else {
                let g1_point = G1Affine::new_unchecked(g1_x, g1_y);
                g1_point.check().map_err(|_| ())?;
                g1_point
            };

            bytereverse(&mut g2_x_c0);
            bytereverse(&mut g2_x_c1);
            bytereverse(&mut g2_y_c0);
            bytereverse(&mut g2_y_c1);

            let g2_x_c0 = <Fq as PrimeField>::BigInt::deserialize_uncompressed(&g2_x_c0[..])
                .map_err(|_| ())?;
            let g2_x_c1 = <Fq as PrimeField>::BigInt::deserialize_uncompressed(&g2_x_c1[..])
                .map_err(|_| ())?;
            let g2_x_c0 = Fq::from_bigint(g2_x_c0).ok_or(())?;
            let g2_x_c1 = Fq::from_bigint(g2_x_c1).ok_or(())?;

            let g2_x = Fq2::new(g2_x_c0, g2_x_c1);

            let g2_y_c0 = <Fq as PrimeField>::BigInt::deserialize_uncompressed(&g2_y_c0[..])
                .map_err(|_| ())?;
            let g2_y_c1 = <Fq as PrimeField>::BigInt::deserialize_uncompressed(&g2_y_c1[..])
                .map_err(|_| ())?;
            let g2_y_c0 = Fq::from_bigint(g2_y_c0).ok_or(())?;
            let g2_y_c1 = Fq::from_bigint(g2_y_c1).ok_or(())?;

            let g2_y = Fq2::new(g2_y_c0, g2_y_c1);

            let g2_point = if g2_x.is_zero() && g2_y.is_zero() {
                G2Affine::zero()
            } else {
                let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
                g2_point.check().map_err(|_| ())?;
                g2_point
            };

            pairs.push((g1_point, g2_point));
        }
    }

    let g1_iter = pairs.iter().map(|(g1, _)| g1);
    let g2_iter = pairs.iter().map(|(_, g2)| g2);
    let result = Bn254::multi_pairing(g1_iter, g2_iter);
    let success = result.0.is_one();
    Ok(success)
}

fn parse_g2_with_subgroup_check(
    input: &[u8; G2_SERIALIZATION_LEN],
) -> Result<G2Affine, Bls12PrecompileSubsystemError> {
    let point = parse_g2(input)?;
    if point.is_zero() || point.is_in_correct_subgroup_assuming_on_curve() {
        Ok(point)
    } else {
        Err(interface_error!(
            Bls12PrecompileInterfaceError::PointNotInSubgroup
        ))
    }
}
