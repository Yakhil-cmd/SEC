### Title
Asymmetric `delta_gas` Adjustment Overcounts Gas Used When Native Consumption Exceeds EVM Gas ŌĆö (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the EIP-3529 refund cap (`gas_used / 5`) is computed and applied **before** the `delta_gas` native-resource adjustment is added to `gas_used`. When `delta_gas > 0` (native consumption implies more gas than EVM ergs alone), the extra gas is added to `gas_used` **after** the cap has already been applied. This means the final `gas_used` can exceed `gas_limit - evm_refund` in a way that is inconsistent with the EIP-3529 cap, and ŌĆö more critically ŌĆö the `delta_gas` adjustment is **one-sided**: it only ever increases `gas_used`, never decreases it. A user who pays a high priority fee (raising `native_per_gas`) to cover native costs will have their gas refund correctly reduced by `delta_gas`, but a user whose native consumption is *lower* than their EVM gas consumption receives no corresponding reduction in `gas_used` (i.e., no extra refund). This asymmetry is the direct analog of the HMX bug: a factor is applied in one direction only, causing users to be overcharged relative to their actual resource consumption.

---

### Finding Description

In `compute_gas_refund` (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`), the final `gas_used` is computed as follows:

1. Charge pubdata from native resources.
2. Compute `gas_used = gas_limit - remaining_ergs / ERGS_PER_GAS`.
3. Apply EIP-3529 EVM refund: `evm_refund = min(refund_counter / ERGS_PER_GAS, gas_used / 5)`.
4. Subtract: `gas_used -= evm_refund`.
5. Apply `minimal_gas_used` floor: `gas_used = max(gas_used, minimal_gas_used)`.
6. Compute `native_used = full_native_limit - remaining_native`.
7. Compute `delta_gas = (native_used / native_per_gas) as i64 - gas_used as i64`.
8. **If `delta_gas > 0`**: `gas_used += delta_gas`. ŌåÉ only positive branch taken.
9. **If `delta_gas <= 0`**: nothing happens. ŌåÉ negative delta is silently discarded (see the `TODO` comment).

The `TODO` comment at line 80 explicitly acknowledges this: `// TODO: return delta_gas to gas_used?`

This means:
- When native resource consumption is **higher** than EVM gas consumption, `gas_used` is increased (user pays more).
- When native resource consumption is **lower** than EVM gas consumption, `gas_used` is **not** decreased (user does not get a refund for the unused native budget).

The result is that users who consume fewer native resources than their EVM gas budget implies are overcharged: they pay for gas that corresponds to native resources they never consumed. [1](#0-0) 

The EIP-3529 cap is also applied to the pre-`delta_gas` value of `gas_used` (line 42), so when `delta_gas > 0` is later added, the final `gas_used` can exceed `gas_limit - evm_refund_capped`, violating the invariant that the refund cap is applied to the final gas used. [2](#0-1) 

---

### Impact Explanation

**Resource accounting bug** ŌĆö users are overcharged for gas when their native resource consumption is lower than their EVM gas consumption. Concretely:

- A user submits a transaction with a high `gas_limit` and a moderate `gas_price`.
- The EVM execution uses, say, 30,000 gas (ergs path).
- The native resource consumption corresponds to only 20,000 gas equivalent (`native_used / native_per_gas = 20,000`).
- `delta_gas = 20,000 - 30,000 = -10,000` (negative).
- The negative delta is **discarded**; `gas_used` stays at 30,000.
- The user is refunded for `gas_limit - 30,000` instead of `gas_limit - 20,000`.
- The user overpays by `10,000 * gas_price` tokens.

This is a direct financial loss to the transaction sender, benefiting the operator (coinbase) who receives the excess fee. The effect is reachable by any unprivileged L2 transaction sender whose transaction has a native-to-gas ratio below `native_per_gas` (i.e., the EVM execution is relatively more expensive in ergs than in native cycles). [3](#0-2) 

---

### Likelihood Explanation

This is reachable by any standard L2 transaction. The `native_per_gas` ratio is derived from `gas_price / native_price`. Transactions that are EVM-compute-heavy but native-light (e.g., tight arithmetic loops with few storage writes) will have `native_used / native_per_gas < gas_used`, triggering the discarded negative `delta_gas`. The condition is common in practice for any transaction where EVM gas dominates over proving/pubdata cost. The `TODO` comment confirms the developers are aware of this gap. [4](#0-3) 

---

### Recommendation

Apply `delta_gas` symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

Additionally, consider reordering so that the EIP-3529 cap (`gas_used / 5`) is applied **after** the `delta_gas` adjustment, so the cap is computed on the final `gas_used` value rather than an intermediate one.

---

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `native_per_gas = 10` (i.e., `gas_price / native_price = 10`)
- EVM execution uses 50,000 ergs-gas; `gas_used_from_ergs = 50,000`
- Native execution uses `400,000` native units; `native_used / native_per_gas = 40,000`
- No EVM refund counter entries.

**Current behavior:**
1. `gas_used = 50,000`
2. `evm_refund = 0`
3. `gas_used = max(50,000, minimal_gas_used)` ŌåÆ `50,000`
4. `delta_gas = 40,000 - 50,000 = -10,000` ŌåÆ **discarded**
5. Final `gas_used = 50,000`
6. Refund = `(100,000 - 50,000) * gas_price`

**Correct behavior:**
- `delta_gas = -10,000` should reduce `gas_used` to `40,000`
- Refund = `(100,000 - 40,000) * gas_price`
- User overpays by `10,000 * gas_price` tokens under the current code.

The entry path is any standard L2 transaction processed through `ZkTransactionFlowOnlyEOA::before_refund` ŌåÆ `compute_gas_refund`. [5](#0-4) [6](#0-5)

pub(crate) fn compute_gas_refund<S: EthereumLikeTypes>(
    system: &mut System<S>,
    to_charge_for_pubdata: S::Resources,
    gas_limit: u64,
    minimal_gas_used: u64,
    native_per_gas: u64,
    resources: &mut S::Resources,
) -> Result<RefundInfo, InternalError> {
    // Already checked
    resources.charge_unchecked(&to_charge_for_pubdata);

    let mut gas_used = gas_limit
        .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
        .ok_or(internal_error!("gas remaining > gas limit"))?;
    resources.exhaust_ergs();

    system_log!(system, "Gas used before refund calculations: {gas_used}\n");

    // Following EIP-3529, refunds are capped to 1/5 of the gas used
    let evm_refund = {
        let full_refund_ergs = system.io.get_refund_counter().ergs();
        let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
        let max_refund = gas_used / 5;
        core::cmp::min(full_refund_gas, max_refund)
    };

    system_log!(system, "Gas refund from refund counters = {evm_refund}\n");

    gas_used -= evm_refund;

    system_log!(
        system,
        "Minimal gas used from validation = {minimal_gas_used}\n"
    );

    #[allow(unused_mut)]
    let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

    // Note: for zero gas price, we use "unlimited native"
    let full_native_limit = if cfg!(feature = "unlimited_native") || native_per_gas == 0 {
        u64::MAX - 1
    } else {
        gas_limit.saturating_mul(native_per_gas)
    };
    let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());

    #[cfg(not(feature = "unlimited_native"))]
    {
        // Adjust gas_used with difference with used native
        let delta_gas = if native_per_gas == 0 {
            0
        } else {
            (native_used / native_per_gas) as i64 - (gas_used as i64)
        };

        if delta_gas > 0 {
            // In this case, the native resource consumption is more than the
            // gas consumption accounted for. Consume extra gas.
            gas_used += delta_gas as u64;
        }
        // TODO: return delta_gas to gas_used?
    }

    let total_gas_refund = gas_limit - gas_used;
    system_log!(system, "Refund after accounting for unused gas, refund counters and native cost: {total_gas_refund}\n");
    require_internal!(
        total_gas_refund <= gas_limit,
        "Gas refund greater than gas limit",
        system
    )?;
    let refund_info = RefundInfo {
        gas_used,
        evm_refund,
        native_used,
    };
    system_log!(system, "Final gas used: {gas_used}\n");
    Ok(refund_info)
}
