### Title
Hardcoded ERC-20 Withdrawal Gas Limit Causes Permanent Withdrawal Failures for Gas-Heavy Tokens - (`File: rs/ethereum/cketh/minter/src/withdraw.rs`)

---

### Summary

The ckETH minter hardcodes `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` to `65_000` gas units for every ERC-20 token withdrawal, regardless of the actual gas requirements of the specific token's `transfer()` function. ERC-20 tokens with complex transfer logic (e.g., fee-on-transfer tokens, tokens with hooks, or tokens like USDT under certain conditions) can require more than 65,000 gas. When this occurs, the Ethereum transaction runs out of gas and reverts, while the corresponding ckERC20 tokens have already been burned on the IC ledger, causing a failed withdrawal that must be reimbursed.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/withdraw.rs`, two gas limits are defined as compile-time constants:

```rust
pub const CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(21_000);
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);
``` [1](#0-0) 

The `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` constant of `65_000` is applied uniformly to all ERC-20 token withdrawal transactions, regardless of which ERC-20 token is being withdrawn. This constant is consumed by `create_transaction` in `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` and is referenced in `rs/ethereum/cketh/minter/src/main.rs` to build the EIP-1559 transaction that is signed via threshold ECDSA and submitted to Ethereum. [2](#0-1) 

The withdrawal flow is:
1. User calls `withdraw_erc20` on the minter canister (unprivileged ingress).
2. Minter burns the user's ckERC20 tokens on the IC ledger.
3. Minter creates an Ethereum EIP-1559 transaction with `gas_limit = 65_000`.
4. Transaction is signed via threshold ECDSA and submitted to Ethereum.
5. If the ERC-20 token's `transfer()` requires more than 65,000 gas, the transaction reverts on Ethereum.
6. The minter detects the failure and issues a reimbursement request, minting back the ckERC20 tokens.

The gas cost of ERC-20 `transfer()` varies widely across tokens:
- Standard ERC-20: ~21,000–50,000 gas
- USDT (Tether): ~65,000 gas (borderline)
- Tokens with transfer hooks, fee-on-transfer logic, or complex storage patterns: can exceed 65,000 gas

Because the gas limit is a hardcoded constant rather than a per-token configurable parameter, any supported ERC-20 token whose `transfer()` consistently exceeds 65,000 gas will have **all** of its withdrawals fail permanently, with no on-chain mechanism to fix this without a minter upgrade.

---

### Impact Explanation

**Vulnerability class:** Chain-fusion mint/burn/replay bug — hardcoded gas limit causes systematic ERC-20 withdrawal failures.

Any ERC-20 token supported by the ledger-suite orchestrator whose `transfer()` function requires more than 65,000 gas will have every withdrawal transaction revert on Ethereum. The user's ckERC20 tokens are burned before the Ethereum transaction is submitted, so the user is left without their tokens until the reimbursement flow completes. If the reimbursement flow itself encounters issues (e.g., ledger unavailability), funds can be temporarily locked. Additionally, the minter wastes Ethereum gas fees on every failed transaction, depleting the minter's ETH balance over time.

For tokens where 65,000 gas is borderline (e.g., USDT under certain network conditions), withdrawals may succeed intermittently, making the failure non-deterministic and harder to diagnose.

---

### Likelihood Explanation

The ckETH minter supports multiple ERC-20 tokens via the ledger-suite orchestrator. As new tokens are added (a governance-driven process open to any NNS proposal), the probability of adding a token whose `transfer()` exceeds 65,000 gas increases. The entry path is fully unprivileged: any user can call `withdraw_erc20` on the minter canister. The failure is triggered automatically by the minter's withdrawal processing loop, requiring no special attacker capability beyond holding ckERC20 tokens.

---

### Recommendation

Replace the single hardcoded constant with a per-token configurable gas limit stored in the minter's state alongside each supported ERC-20 token's metadata. The gas limit should be set when a token is added via governance and updatable via a subsequent governance proposal. As an interim measure, the constant should be raised to a safer upper bound (e.g., 150,000 gas) to accommodate tokens with more complex transfer logic, while a proper per-token configuration is implemented.

```rust
// Instead of:
pub const CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT: GasAmount = GasAmount::new(65_000);

// Store per-token gas limit in CkErc20Token:
pub struct CkErc20Token {
    // ... existing fields ...
    pub withdrawal_gas_limit: GasAmount,
}
```

### Proof of Concept

1. A governance proposal adds a new ERC-20 token (e.g., a fee-on-transfer token) to the ckETH minter's supported token list. The token's `transfer()` function requires 80,000 gas due to its fee distribution logic.

2. A user deposits the ERC-20 token to receive ckERC20 tokens on the IC.

3. The user calls `withdraw_erc20` on the minter canister (unprivileged ingress call).

4. The minter burns the user's ckERC20 tokens on the IC ledger (irreversible at this point).

5. The minter calls `create_transaction` with `gas_limit = CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000`. [1](#0-0) 

6. The signed EIP-1559 transaction is submitted to Ethereum. The ERC-20 `transfer()` call consumes 80,000 gas, exceeding the 65,000 gas limit. The transaction reverts with an out-of-gas error.

7. The minter detects the failed transaction receipt and creates a reimbursement request. The user must wait for the reimbursement flow to mint back their ckERC20 tokens, during which their funds are locked. The minter's ETH balance is reduced by the gas fees paid for the failed transaction.

8. Every subsequent withdrawal attempt for this token will fail identically, making the token effectively non-withdrawable without a minter canister upgrade.
