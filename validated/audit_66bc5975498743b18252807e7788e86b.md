### Title
`inspect_message` Guard Bypassed via Regular Query Call to `get_config`, Exposing Undisclosed Rule and Incident UUIDs — (`rs/boundary_node/rate_limits/canister/canister.rs`)

---

### Summary

The `inspect_message` hook is intended to restrict `get_config` to authorized callers (API boundary nodes and the `authorized_principal`). However, `get_config` is declared with `#[query]`, making it a regular (non-replicated) query. On the IC, `inspect_message` is **never invoked for regular query calls** — only for ingress/update messages. Any caller, including anonymous, can invoke `get_config` as a query call, bypassing the guard entirely. The `ConfigConfidentialityFormatter` then returns `OutputRule` entries with `rule_id` and `incident_id` populated even for undisclosed rules, enabling UUID enumeration.

---

### Finding Description

**Step 1 — The intended guard:**

`inspect_message` at lines 34–68 of `canister.rs` checks whether the caller is the `authorized_principal` or an API boundary node before allowing `get_config` to proceed as a replicated query: [1](#0-0) 

The constant `REPLICATED_QUERY_METHOD = "get_config"` and the guard logic make the intent clear: only privileged callers should be able to call `get_config`.

**Step 2 — The bypass:**

`get_config` is annotated with `#[query]`, not `#[update]`: [2](#0-1) 

Per the IC protocol specification, `canister_inspect_message` is **only invoked for ingress (update) messages**, never for query calls. A caller sending a regular query call to `get_config` bypasses `inspect_message` entirely — the hook simply does not fire.

**Step 3 — What the unprivileged caller receives:**

`ConfigGetter::get` applies `ConfigConfidentialityFormatter` for `RestrictedRead` callers: [3](#0-2) 

The formatter only redacts `description` and `rule_raw`, leaving `rule_id` and `incident_id` intact: [4](#0-3) 

The `OutputRule` struct confirms both fields are always present in the response: [5](#0-4) 

**Step 4 — The existing test proves the leak:**

The unit test for `getter_unauthorized` (RestrictedRead) explicitly asserts that `rule_id` and `incident_id` are populated for undisclosed rules: [6](#0-5) 

---

### Impact Explanation

An anonymous or unprivileged caller can:
1. Call `get_config(None)` as a regular query to retrieve the full list of `OutputRule` entries.
2. For every undisclosed rule, read its `rule_id` UUID and `incident_id` UUID (even though `rule_raw` and `description` are `None`).
3. Determine the exact count of undisclosed rules and which rules are grouped under the same incident.
4. Use the harvested `incident_id` values to call `get_rules_by_incident_id` (also a `#[query]` method, also bypasses `inspect_message`) to enumerate all rule IDs associated with a confidential incident.

The actual rule content (`rule_raw`, `description`) remains protected by the formatter. The exposure is limited to UUID enumeration and incident-to-rule correlation — but this directly violates the stated invariant that undisclosed rule IDs and incident IDs must not be enumerable by `RestrictedRead` callers.

---

### Likelihood Explanation

Trivially exploitable. Any IC user can send a query call to any canister method. No special tooling, keys, or privileges are required. The bypass is a direct consequence of using `#[query]` on a method that was intended to be gated by `inspect_message`.

---

### Recommendation

Either:
- Change `get_config` from `#[query]` to `#[update]` so that `inspect_message` fires for all calls (at the cost of consensus overhead and response certification), **or**
- Move the access control check **inside** `get_config` itself: if the caller's `AccessLevel` is `RestrictedRead`, return an authorization error rather than a redacted response. The `inspect_message` hook should be treated as a defense-in-depth layer only, not the primary access control boundary.

The formatter-based approach of returning redacted data (with UUIDs still present) is only safe if the method is truly unreachable by unprivileged callers — which is not the case for a `#[query]` method.
