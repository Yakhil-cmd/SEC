### Title
Truncating Division in `native_per_pubdata` Calculation Causes Pubdata to Be Completely Uncharged When `pubdata_price < native_price` — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's L2 transaction validation, `native_per_pubdata` is computed via integer floor division (`wrapping_div`) of `pubdata_price` by `native_price`. When `pubdata_price < native_price`, this division truncates to zero. As a result, every pubdata byte generated by the transaction costs zero native resources, allowing any transaction sender to generate pubdata at no native resource cost. This is a direct resource accounting analog to the Symmetrical rounding error: a proportional reduction silently becomes zero due to integer truncation.

---

### Finding Description

**Root cause — line 142 of `validation_impl.rs`:**

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is Rust's floor (truncating) integer division. When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `0`. The only guard is a `u64` range check; there is no minimum-value check.

**Propagation through the resource pipeline:**

1. `native_per_pubdata = 0` is passed into `create_resources_for_tx` as `native_per_pubdata_byte`:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0
``` [2](#0-1) 

2. At post-execution pubdata charging in `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // → 0 for any pubdata_spent
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

3. `check_enough_resources_for_pubdata` always returns `enough = true` since the required native is 0, so the post-execution pubdata revert path is never triggered. [4](#0-3) 

4. In `compute_gas_refund`, the `delta_gas` adjustment also uses `native_per_gas` (not `native_per_pubdata`), so the pubdata native cost is silently absent from `native_used`, and the user receives a larger gas refund than deserved:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [5](#0-4) 

The same truncating division also appears in `api/src/helpers.rs` line 427, used for off-chain fee estimation: [6](#0-5)

### Impact Explanation

When `pubdata_price < native_price`, every transaction in the block:
- Pays **zero** native resources for all pubdata it generates (SSTORE writes, events, L1 messages).
- Passes the post-execution pubdata sufficiency check unconditionally.
- Receives a larger gas refund than deserved (since `native_used` does not include pubdata cost).

The operator/protocol bears the full proving and L1 publication cost for that pubdata. A transaction sender can deliberately generate the maximum allowed pubdata per block (via many SSTORE operations) at no native resource cost. Over many blocks where `pubdata_price < native_price`, the accumulated undercharge is proportional to total pubdata generated × the true fractional cost per byte. This breaks the invariant that native resources must cover pubdata costs, and can lead to systematic underpayment for L1 data availability.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are independent block-level parameters set by the operator. `native_price` reflects RISC-V cycle cost; `pubdata_price` reflects L1 calldata/blob cost. During periods of low L1 gas prices or high proving costs, `pubdata_price < native_price` is a realistic and reachable configuration. No special privilege is required by the attacker — any transaction sender in such a block can exploit this. The condition is externally observable (both prices are in the block context), so an attacker can time submissions accordingly.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` (ceiling division) to match the treatment of `native_per_gas`, which already uses `div_ceil`:

```rust
// Current (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// Fix (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [7](#0-6) 

This ensures that even when `pubdata_price < native_price`, each pubdata byte costs at least 1 native unit, preventing complete undercharging. The same fix should be applied in `api/src/helpers.rs` line 427 for consistency with off-chain estimation.

---

### Proof of Concept

**Setup:** Block context with `native_price = 100`, `pubdata_price = 50`.

**Computation:**
- `native_per_pubdata = 50.wrapping_div(100) = 0`
- Correct value should be `ceil(50/100) = 1` native unit per pubdata byte.

**Attack:**
1. Submit a transaction that writes to 100 new storage slots (each slot write = 32 bytes pubdata = 3200 bytes total pubdata).
2. Expected native cost for pubdata: `3200 * 1 = 3200` native units.
3. Actual native cost charged: `3200 * 0 = 0` native units.
4. The post-execution check `check_enough_resources_for_pubdata` passes trivially (0 ≤ remaining native).
5. The transaction succeeds, the operator pays the full L1 publication cost, and the sender pays nothing for pubdata in native resources.

Repeating across many transactions or blocks where `pubdata_price < native_price` accumulates unbounded undercharging proportional to pubdata generated.
