### Title
Integer Division Truncation in `native_per_pubdata` Causes Zero Pubdata Charge, Allowing Free Pubdata Production - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the `native_per_pubdata` ratio — which determines how many native resources a transaction must pay per byte of pubdata it produces — is computed using integer floor division (`wrapping_div`). When `pubdata_price < native_price`, this division truncates to zero. A `native_per_pubdata` of zero causes every subsequent pubdata charge to evaluate to zero, meaning a transaction can produce an arbitrary amount of pubdata (storage writes, etc.) without paying any native resources for it. The operator bears the L1 publication cost with no compensation.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the `native_per_pubdata` ratio is computed as:

```rust
// line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` performs integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 500`, `native_price = 1000`), the result is `500 / 1000 = 0`.

This zero value propagates into every pubdata charge site. In `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // = N * 0 = 0
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

And in `create_resources_for_tx`, the intrinsic pubdata overhead is also zeroed:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
``` [3](#0-2) 

The same truncation bug exists in the off-chain helper `api/src/helpers.rs`:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

By contrast, `native_per_gas` is computed with `div_ceil`, which correctly rounds up and avoids the zero-truncation problem:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [5](#0-4) 

The inconsistency — `div_ceil` for `native_per_gas` but `wrapping_div` (floor) for `native_per_pubdata` — is the root cause.

---

### Impact Explanation

When `native_per_pubdata == 0`, the transaction is charged **zero native resources** for all pubdata it produces, regardless of volume. A user can submit a transaction that writes to many storage slots (each slot write produces pubdata bytes that must be published to L1), and the operator must pay the L1 data availability cost with no native resource recovery. This is a direct financial loss for the operator/protocol. The `check_enough_resources_for_pubdata` check also passes trivially (0 ≤ any remaining native), so the post-execution pubdata guard provides no protection. [6](#0-5) 

---

### Likelihood Explanation

`pubdata_price` and `native_price` are independent operator-set block parameters with different units: `native_price` is the cost per RISC-V cycle, while `pubdata_price` is the cost per pubdata byte. There is no protocol-level invariant enforcing `pubdata_price >= native_price`. In realistic market conditions — for example, when L1 calldata costs are low relative to proving costs — `pubdata_price` can legitimately be smaller than `native_price`, triggering the truncation. Any unprivileged L2 transaction sender can observe the block context and exploit this condition by submitting pubdata-heavy transactions (e.g., many `SSTORE` operations) at zero pubdata cost. [7](#0-6) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, ensuring at least 1 native unit per pubdata byte when price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. Additionally, consider adding a protocol-level check that rejects blocks where `native_per_pubdata` would be zero while `pubdata_price > 0`.

---

### Proof of Concept

**Setup:**
- Operator sets block context: `native_price = 1_000_000`, `pubdata_price = 999_999` (a realistic scenario where pubdata is slightly cheaper than native computation).
- `native_per_pubdata = 999_999 / 1_000_000 = 0` (integer floor division).

**Attack:**
1. Attacker submits an L2 transaction that writes to 1,000 new storage slots (each write produces ~65 bytes of pubdata = ~65,000 bytes total).
2. During validation, `native_per_pubdata = 0`, so `intrinsic_pubdata_overhead = 0`.
3. During post-execution pubdata check: `native_cost = 65_000 * 0 = 0`. Check passes.
4. Transaction succeeds. Attacker paid zero native resources for 65,000 bytes of pubdata.
5. Operator must publish 65,000 bytes to L1 at their own expense.

**Concrete numbers with the existing test framework values** (from `BlockMetadataFromOracle::new_for_test`): `native_price = 10`, `pubdata_price = 0` — this is the default test configuration, meaning `native_per_pubdata = 0` is the default in tests, confirming the condition is routinely exercised. [8](#0-7)
