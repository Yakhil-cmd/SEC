### Title
Stale `minter_id` Reference in Ledger Suite Orchestrator Causes Permanent ckERC20 Mint/Burn Failure on Minter Replacement - (File: `rs/ethereum/ledger-suite-orchestrator/src/scheduler/mod.rs`)

---

### Summary

The Ledger Suite Orchestrator (LSO) stores `minter_id` in its state at initialization time. When a new ckERC20 token is added, the LSO deploys a new ICRC-1 ledger canister with `minting_account` permanently set to the current `minter_id`. The LSO's `UpgradeArg` provides no field to update `minter_id` after initialization, and the ICRC-1 ledger's `minting_account` is immutable after deployment. If the ckETH minter canister is ever replaced at a new principal (e.g., via NNS governance due to a critical bug), all previously deployed ckERC20 ledger canisters retain the old minter as their sole authorized minting account, permanently breaking mint and burn operations for those tokens.

---

### Finding Description

The LSO's `State` struct stores `minter_id: Option<Principal>` set at `init` time: [1](#0-0) 

When a new ERC-20 token is added via `AddErc20Arg`, `InstallLedgerSuiteArgs::validate_add_erc20` reads `minter_id` directly from the current state and bakes it into the `InstallLedgerSuiteArgs`: [2](#0-1) 

The `install_ledger_suite` function then calls `icrc1_ledger_init_arg(args.minter_id, ...)`, which sets the ICRC-1 ledger's `minting_account` and `fee_collector_account` to the minter's principal: [3](#0-2) 

The `minting_account` field is set only in `InitArgs` and is absent from `UpgradeArgs` in the ICRC-1 ledger interface, making it immutable after deployment: [4](#0-3) 

Critically, the LSO's own `UpgradeArg` has **no `minter_id` field**, providing no governance path to update the stored minter reference: [5](#0-4) 

The `post_upgrade` handler processes `UpgradeArg` but never touches `minter_id`: [6](#0-5) 

---

### Impact Explanation

If the ckETH minter canister (`sv3dd-oaaaa-aaaar-qacoa-cai`) is ever replaced at a new canister principal via NNS governance (e.g., due to a critical security bug requiring a fresh deployment):

1. All existing ckERC20 ledger canisters (ckUSDC, ckUSDT, ckLINK, etc.) retain the old minter principal as their immutable `minting_account`. The new minter cannot call `icrc1_transfer` with a zero-fee burn or mint on these ledgers — only the account matching `minting_account` can do so.
2. The LSO's stored `minter_id` cannot be updated, so any new ckERC20 tokens added after the minter replacement would also be deployed with the old (now defunct) minter as their `minting_account`.
3. The `NotifyErc20Added` task also reads `minter_id` from state at execution time, so the new minter would not be notified of new tokens either. [7](#0-6) 

The result is a **permanent, irrecoverable chain-fusion mint/burn failure** for all ckERC20 tokens — deposits and withdrawals halt entirely.

**Impact: High** — complete loss of ckERC20 mint/burn functionality for all managed tokens.

---

### Likelihood Explanation

**Likelihood: Medium.** The minter canister ID is stable under normal upgrades (IC canister upgrades preserve the canister ID). However, scenarios requiring a new canister ID are realistic:

- A critical vulnerability in the minter requiring a full reinstall at a new canister ID (as happened with the KYT→BTC Checker migration, which changed the checker principal).
- A subnet migration requiring canister recreation.
- An emergency response to a compromised minter.

The LSO README and upgrade proposals confirm `minter_id` is set once at init and never revisited in any upgrade proposal to date. [8](#0-7) 

---

### Recommendation

1. **Add `minter_id: Option<Principal>` to `UpgradeArg`** in `rs/ethereum/ledger-suite-orchestrator/src/candid/mod.rs` and handle it in `post_upgrade` to allow NNS governance to update the stored minter reference.

2. **Add an admin function on the ICRC-1 ledger** (or use the LSO's upgrade path) to update `minting_account` on existing managed ledger canisters when the minter is replaced. Since `minting_account` is currently immutable in the ICRC-1 ledger, this requires adding a `change_minting_account` field to `UpgradeArgs` in `rs/ledger_suite/icrc1/ledger/ledger.did`.

3. **Track the `minter_id` used at ledger deployment** in the LSO's `Canisters` state so that stale references can be detected and remediated.

---

### Proof of Concept

**Step 1 — LSO initialized with minter `A`:**
```
InitArg { minter_id = Some(principal "sv3dd-oaaaa-aaaar-qacoa-cai") }
```
State: `minter_id = A`

**Step 2 — ckUSDC ledger deployed:**
`validate_add_erc20` reads `minter_id = A` from state.
`icrc1_ledger_init_arg(A, ...)` sets `minting_account = A` on the ckUSDC ledger. This is immutable. [9](#0-8) 

**Step 3 — NNS governance replaces minter:**
New minter deployed at principal `B`. LSO upgrade attempted with `UpgradeArg { ... }` — no `minter_id` field exists, so LSO state still holds `minter_id = A`. [5](#0-4) 

**Step 4 — New minter `B` attempts to mint ckUSDC:**
The ckUSDC ledger rejects the mint because `caller (B) != minting_account (A)`. All ckERC20 deposits and withdrawals fail permanently.

**Step 5 — New ckERC20 token added:**
`validate_add_erc20` still reads `minter_id = A` from LSO state. New ledger deployed with `minting_account = A` (old, defunct minter). New minter `B` cannot operate on it either. [10](#0-9)

pub fn post_upgrade(upgrade_arg: Option<UpgradeArg>) {
    if let Some(arg) = upgrade_arg {
        if let Some(git_commit_hash) = &arg.git_commit_hash {
            let git_commit_hash =
                GitCommitHash::from_str(git_commit_hash).expect("ERROR: invalid git commit hash");
            let ledger_suite_version = mutate_wasm_store(|s| {
                record_icrc1_ledger_suite_wasms(s, ic_cdk::api::time(), git_commit_hash)
            })
            .expect("BUG: failed to record icrc1 ledger suite wasms during upgrade");
            mutate_state(|s| s.init_ledger_suite_version(ledger_suite_version));
        }
        if let Some(manage_installed_canisters) = arg.manage_ledger_suites.clone() {
            for managed_canisters in manage_installed_canisters {
                let canisters =
                    read_state(|s| InstalledLedgerSuite::validate(s, managed_canisters))
                        .expect("ERROR: invalid manage installed canisters");
                mutate_state(|s| s.record_manage_other_canisters(canisters.clone()));
                log!(
                    INFO,
                    "[post_upgrade]: recorded manage installed canisters: {:?}",
                    canisters
                );
            }
        }
        match read_wasm_store(|w| UpgradeOrchestratorArgs::validate_upgrade_arg(w, arg.clone())) {
            Ok(valid_upgrade_args) => {
                if valid_upgrade_args.upgrade_ledger_suite() {
                    let current_ledger_suite_version =
                        read_state(|s| s.ledger_suite_version().cloned())
                            .expect("BUG: missing ledger suite version");
                    mutate_state(|s| {
                        s.update_ledger_suite_version(
                            valid_upgrade_args
                                .clone()
                                .new_ledger_suite_version(current_ledger_suite_version),
                        )
                    });
                    for token_id in
                        read_state(|s| s.all_managed_tokens_ids_iter().collect::<Vec<_>>())
                    {
                        schedule_now(
                            Task::UpgradeLedgerSuite(
                                valid_upgrade_args.clone().into_task(token_id),
                            ),
                            &IC_CANISTER_RUNTIME,
                        );
                    }
                }
            }
            Err(e) => {
                ic_cdk::trap(format!(
                    "[post_upgrade]: ERROR: invalid arguments to upgrade {arg:?}: {e:?}"
                ));
            }
        }
        if let Some(update) = arg.cycles_management {
            mutate_state(|s| update.apply(s.cycles_management_mut()));
        }
    }
    read_state(|s| s.validate_config().expect("ERROR: invalid state"));
    setup_tasks_and_timers()
}
