### Title
ERC-20 Tokens Permanently Burned Without Refund When NEAR-Side Transfer Fails in `ExitToNear` Precompile (Without `error_refund` Feature) - (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile burns ERC-20 tokens on Aurora and schedules a NEAR-side `ft_transfer` or `ft_transfer_call` promise with **no callback**. If the NEAR-side promise fails (e.g., recipient not registered with the NEP-141 contract, or `ft_on_transfer` rejects the transfer), the burned ERC-20 tokens are permanently lost with no recovery path.

---

### Finding Description

The `ExitToNear` precompile's `run()` function in `engine-precompiles/src/native.rs` constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `refund` is always `None`. For the ERC-20 exit paths (both legacy `ft_transfer` and Omni `ft_transfer_call`), `transfer_near_args` is also `None`. This makes `callback_args` equal to the default value, triggering the branch that schedules **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)  // No callback registered
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The ERC-20 token burn happens **before** the NEAR promise, inside `EvmErc20.withdrawToNear()`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);  // Irreversible burn
    // ... call ExitToNear precompile
}
``` [3](#0-2) 

The same pattern exists in `EvmErc20V2`: [4](#0-3) 

The `exit_erc20_token_to_near` function confirms that for both the legacy path and the Omni path, `transfer_near_args` is `None`: [5](#0-4) 

When `error_refund` IS enabled, `refund_call_args` populates a `RefundCallArgs` that causes `exit_to_near_precompile_callback` to call `refund_on_error`, which re-mints the burned tokens: [6](#0-5) 

Without `error_refund`, `exit_to_near_precompile_callback` is never scheduled, so `refund_on_error` is never called.

The test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

---

### Impact Explanation

**Permanent freezing/destruction of funds.** When the NEAR-side `ft_transfer` or `ft_transfer_call` fails:

- The user's ERC-20 tokens are **permanently burned** on Aurora (destroyed, not recoverable).
- The corresponding NEP-141 tokens remain in Aurora's account (not transferred to the user).
- There is no callback, no re-mint, and no recovery path.

This matches the external report's pattern: tokens are consumed on the source side, the destination-side transfer fails, and the user receives nothing.

---

### Likelihood Explanation

Any unprivileged EVM user holding an `EvmErc20` token can trigger this by calling `withdrawToNear` with:

- A NEAR account ID that is not registered with the NEP-141 contract (storage deposit not paid).
- An Omni message targeting a receiver whose `ft_on_transfer` rejects the transfer.
- A non-existent NEAR account.

These are all realistic, user-reachable conditions. The `withdrawToNear` function imposes no validation on the recipient. [3](#0-2) 

---

### Recommendation

The `ExitToNear` precompile should always register a callback (i.e., always use `PromiseArgs::Callback`) for ERC-20 exits, regardless of whether `error_refund` is enabled. The callback should, at minimum, detect a failed NEAR-side promise and re-mint the burned tokens to the original sender. The `error_refund` feature should be made a required default, or the no-callback branch should be removed for ERC-20 exits.

---

### Proof of Concept

1. Deploy a NEP-141 token and its corresponding ERC-20 mirror on Aurora (via `ft_transfer_call` to bridge tokens in).
2. Hold ERC-20 tokens in an Aurora address.
3. Call `EvmErc20.withdrawToNear("unregistered.near", amount)` — `"unregistered.near"` has not paid storage deposit to the NEP-141 contract.
4. The ERC-20 `_burn` executes atomically, destroying the tokens.
5. The `ExitToNear` precompile schedules `ft_transfer` to `"unregistered.near"` with **no callback** (when `error_refund` is not enabled).
6. The NEAR-side `ft_transfer` fails because `"unregistered.near"` is not registered.
7. No `exit_to_near_precompile_callback` is triggered; `refund_on_error` is never called.
8. The user's ERC-20 tokens are permanently gone; the NEP-141 tokens remain locked in Aurora's account. [8](#0-7)
