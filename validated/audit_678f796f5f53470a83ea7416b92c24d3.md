Audit Report

## Title
ICP Neurons' Fund Double-Mint via Lifecycle Restoration After Failed `mint_to_sns_governance` — (`rs/nns/governance/src/governance.rs`, `rs/ledger_suite/icp/ledger/src/lib.rs`)

## Summary
When `settle_neurons_fund_participation` mints ICP to an SNS treasury and the ledger call fails transiently, the proposal lifecycle is explicitly restored to its pre-terminal value and `final_neurons_fund_participation` is left unset. A subsequent call to `finalize_swap` (callable by any principal) re-enters "Ok case III" and issues a second mint. Because `IcpLedgerCanister::transfer_funds` always passes `created_at_time: None`, the ICP ledger substitutes the current round timestamp, producing a distinct transaction hash each call and bypassing its own deduplication — a known-broken invariant tracked as FI-541 with the deduplication test explicitly disabled.

## Finding Description

**Step 1 — ICP ledger deduplication is inoperative for `None` callers.**

`add_payment_with_timestamp` in `rs/ledger_suite/icp/ledger/src/lib.rs` unconditionally fills in the current timestamp when the caller omits `created_at_time`:

```rust
// TODO(FI-349): preserve created_at_time and memo the caller specified.
created_at_time: created_at_time.or(Some(now)),
``` [1](#0-0) 

The deduplication check in `rs/ledger_suite/common/ledger_canister_core/src/ledger.rs` only fires when `created_at_time` is `Some`, and compares the full transaction hash (which includes `created_at_time`): [2](#0-1) 

Because each call in a different round receives a distinct `now`, the hash never matches a prior entry. The deduplication window is populated but never triggered. The integration test confirming this behavior is suppressed:

```rust
#[ignore = "requires fix for FI-541"]
#[test]
fn test_tx_deduplication() { ... }
``` [3](#0-2) 

**Step 2 — `transfer_funds` always passes `created_at_time: None`.**

The shared nervous-system ledger client used by NNS Governance unconditionally omits `created_at_time`: [4](#0-3) 

**Step 3 — `mint_to_sns_governance` uses a fixed `memo = 0` and no `created_at_time`.**

Every Neurons' Fund ICP mint uses the same memo and relies entirely on the (broken) ledger deduplication: [5](#0-4) 

**Step 4 — Lifecycle is set to terminal *before* the mint, then restored on failure.**

The proposal lifecycle is advanced to a terminal state at line 7235, before the async mint at line 7266: [6](#0-5) [7](#0-6) 

If the mint returns an error, the lifecycle is explicitly reset and `final_neurons_fund_participation` is left unset:

```rust
let Ok(ref participated_reserves) = settlement_result else {
    // Reset the Proposal's lifecycle and complete the request. Note that the field
    // `final_neurons_fund_participation` remains unset in this case.
    proposal_data.sns_token_swap_lifecycle = Some(original_sns_token_swap_lifecycle as i32);
    return settlement_result;
};
``` [8](#0-7) 

**Step 5 — The state machine guards do not block a retry after lifecycle restoration.**

The two guards that would prevent re-entry are:

- **"Ok case I"** (line 7166): returns early only when `final_neurons_fund_participation` is already `Some`. After a failed mint it is `None`, so this guard does not fire.
- **"Err case 4"** (line 7154): blocks calls only when the lifecycle IS terminal AND `final_neurons_fund_participation` is `None`. After lifecycle restoration the lifecycle is NOT terminal, so this guard does not fire either. [9](#0-8) [10](#0-9) 

A second call therefore falls into **"Ok case III"** (line 7180) and proceeds to mint again: [11](#0-10) 

Because the second call lands in a different consensus round, `now` differs, the transaction hash differs, and the ICP ledger accepts the second mint unconditionally.

## Impact Explanation

A successful double-mint issues ICP from the NNS minting account to the SNS governance treasury a second time without any corresponding maturity deduction from Neurons' Fund neurons. This constitutes **illegal minting** of ICP, inflating the ICP supply by the full Neurons' Fund participation amount for the affected swap. For major SNS launches, Neurons' Fund participation has historically been in the range of millions of ICP e8s, placing the financial impact potentially above $1M. This matches the Critical/High allowed impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets."* Severity is assessed as **High** because the attack requires a transient ledger failure as a precondition, adding a meaningful constraint.

## Likelihood Explanation

`finalize_swap` is callable by any principal on a committed SNS swap. The attacker does not need any special privilege beyond the existence of a committed swap with Neurons' Fund participation. A transient ledger failure (`TxThrottled`, inter-canister call timeout, or replica restart) is a realistic trigger; an adversary can also attempt to induce `TxThrottled` by flooding the ICP ledger with transactions during the settlement window. Once the first call fails and the lifecycle is restored, the attacker simply calls `finalize_swap` again. The ICP ledger deduplication safety net is confirmed broken (FI-541, test disabled), removing the last backstop. The failure-then-retry path is not covered by any existing test.

## Recommendation

1. **Pass a stable, proposal-scoped `created_at_time` and a unique memo in `mint_to_sns_governance`** — e.g., derive `created_at_time` from the proposal creation timestamp stored in `proposal_data` and use the proposal ID as the memo. This makes the transaction hash idempotent across retries and allows the ICP ledger to reject a duplicate mint even if the application-level guard fails.
2. **Fix FI-541 and re-enable `test_tx_deduplication`** for the ICP ledger to restore the deduplication safety net.
3. **Do not restore the lifecycle on mint failure.** Instead, leave the lifecycle in its terminal state and surface the error for manual intervention (the comment at line 7319 already applies this philosophy to post-mint failures). Alternatively, set `final_neurons_fund_participation` to a sentinel value before the mint so that "Ok case I" fires on retry.
4. **Add an idempotency assertion** at the top of `settle_neurons_fund_participation` that verifies `final_neurons_fund_participation` is unset before proceeding past "Ok case III", and add an integration test covering the mint-failure-then-retry path.

## Proof of Concept

1. Deploy NNS Governance and ICP Ledger in a PocketIC or state-machine test environment.
2. Create a committed SNS swap proposal with non-zero Neurons' Fund participation.
3. Configure the mock ICP ledger to return `TxThrottled` on the first `transfer` call and succeed on the second.
4. Call `finalize_swap` (or directly `settle_neurons_fund_participation`) — observe it returns an error and the proposal lifecycle is restored to its pre-terminal value.
5. Call `finalize_swap` again — observe it returns success.
6. Query the ICP ledger block log and the SNS governance treasury balance — confirm two mint transactions are recorded for the same destination with the same amount, and the treasury balance is double the intended Neurons' Fund participation.
