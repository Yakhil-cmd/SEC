### Title
Unprotected `new()` Initialization Allows Front-Running to Hijack Aurora Engine Owner — (`engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine's `new()` initialization entrypoint carries no caller permission check. It only guards against double-initialization by checking whether state already exists. If the WASM deployment and the `new()` call are not submitted as a single atomic NEAR batch transaction, any unprivileged NEAR account can race to call `new()` first, supply their own `owner_id`, and permanently seize administrative control of the engine before the legitimate deployer can initialize it.

---

### Finding Description

The `new()` function in `engine/src/contract_methods/admin.rs` is the sole initialization path for the Aurora Engine contract. It accepts arbitrary `NewCallArgs` (including `owner_id`) from any caller and writes them directly to state, with no check on `env.predecessor_account_id()`: [1](#0-0) 

The only guard is a state-existence check:

```rust
if state::get_state(&io).is_ok() {
    return Err(b"ERR_ALREADY_INITIALIZED".into());
}
```

Once state is written by any caller, the function permanently rejects all subsequent calls. There is no `require_owner_only` or any predecessor check analogous to what is applied to every other privileged method (e.g., `set_owner`, `attach_full_access_key`). [2](#0-1) 

The public WASM entrypoint exposes this unconditionally: [3](#0-2) 

The deployment helper in `engine-workspace/src/lib.rs` demonstrates the non-atomic pattern in practice: `account.deploy()` is `await`-ed first, then `engine.new()` is `await`-ed in a separate async call — two distinct NEAR transactions with a gap between them: [4](#0-3) 

---

### Impact Explanation

The `owner_id` stored during `new()` is the root of all privilege in the engine. An attacker who controls `owner_id` can immediately call `attach_full_access_key` to add their own full-access key to the Aurora Engine NEAR account: [5](#0-4) 

A full-access key on the Aurora Engine account grants the attacker the ability to transfer all NEAR held by the account, redeploy arbitrary WASM, and drain all bridged ERC-20 / ETH balances held by the engine. This is direct, complete theft of all user funds at rest in the engine — **Critical**.

---

### Likelihood Explanation

NEAR Protocol supports atomic batch transactions (CreateAccount + DeployContract + FunctionCall in one batch), but nothing in the production contract enforces that `new()` must be called this way. The `engine-workspace` deployment helper uses two separate awaited calls, confirming the non-atomic pattern is used in practice. Any new Aurora Engine deployment (including Silo-mode instances) that follows this pattern is vulnerable during the window between WASM deployment and `new()` invocation. An attacker monitoring the NEAR chain for new contract deployments can trivially detect and exploit this window.

---

### Recommendation

Add a predecessor check to `new()` so that only the contract account itself (i.e., a self-call from a batch transaction) or a pre-designated deployer can initialize the engine:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Enforce that initialization must come from the contract account itself
    // (i.e., via a batch transaction: DeployContract + FunctionCall(new))
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    // ... rest of initialization
}
```

Alternatively, mandate that all Aurora Engine deployments use a single NEAR batch transaction containing both `DeployContract` and `FunctionCall("new", ...)` actions, making the deployment and initialization atomic by construction.

---

### Proof of Concept

1. Deployer submits transaction T1: `DeployContract` (uploads Aurora Engine WASM to `aurora.near`). T1 is confirmed.
2. Deployer submits transaction T2: `FunctionCall("new", {owner_id: "deployer.near", ...})`. T2 is pending.
3. Attacker observes T1 on-chain and submits transaction T3: `FunctionCall("new", {owner_id: "attacker.near", ...})` targeting `aurora.near`, with higher gas priority.
4. T3 executes before T2. `state::get_state` returns `Err` (no state yet), so the guard passes. State is written with `owner_id = attacker.near`.
5. T2 executes and fails with `ERR_ALREADY_INITIALIZED`.
6. Attacker calls `attach_full_access_key` on `aurora.near` with their own public key (permitted because `owner_id = attacker.near`). [5](#0-4) 

7. Attacker now holds a full-access key on `aurora.near` and can transfer all NEAR, redeploy arbitrary code, and drain all bridged user funds.

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
