### Title
Asymmetric `delta_gas` Adjustment Silently Discards User Refund When Native Cost Is Below EVM Gas Cost - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the double-resource accounting reconciliation step adjusts `gas_used` upward when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), but silently discards the symmetric case where native consumption is *below* EVM gas consumption (`delta_gas < 0`). The negative `delta_gas` — representing gas that was charged in EVM terms but not consumed in native terms — is never returned to the user. This is a resource accounting bug where a portion of the user's prepaid fee is neither refunded to the sender nor paid to the operator: it is simply lost.

---

### Finding Description

In `compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`, the reconciliation between EVM gas used and native resource used is one-sided:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The `// TODO: return delta_gas to gas_used?` comment at line 80 is a developer acknowledgment of the missing symmetric case. When `delta_gas < 0`, meaning `(native_used / native_per_gas) < gas_used`, the EVM gas meter has charged the user more gas than the native resource actually consumed. In this case, `gas_used` is not reduced, so the user is overcharged by `|delta_gas|` gas units.

The fee flow is:
1. At transaction start, the sender is debited `gas_limit * gas_price` (full prepayment).
2. At transaction end, the sender is refunded `(gas_limit - gas_used) * gas_price`.
3. The operator receives `gas_used * gas_price_for_operator`.

When `delta_gas < 0` and is not applied, `gas_used` is inflated relative to what native resource consumption justifies. The sender receives a smaller refund than they are owed, and the operator receives more than they earned. The gap is `|delta_gas| * gas_price` tokens — a direct, quantifiable loss to the transaction sender.

The same `compute_gas_refund` function is called for both ZK L2 transactions (via `ZkTransactionFlow::before_refund`) and L1→L2 priority transactions (via `process_l1_transaction`), so both transaction types are affected.

---

### Impact Explanation

**Funds loss to transaction senders.** Any transaction where EVM gas consumption exceeds the native-resource-implied gas equivalent results in the sender being overcharged. The overcharge amount is `|delta_gas| * gas_price` base tokens, which are neither refunded to the sender nor explicitly burned — they are transferred to the operator as excess fee. This is a direct, per-transaction loss of user funds proportional to the divergence between EVM gas and native resource consumption.

The magnitude depends on the `native_per_gas` ratio and the actual execution profile. Transactions that are EVM-gas-heavy but native-resource-light (e.g., computationally simple but calldata-heavy operations where calldata tokens inflate EVM gas via EIP-7623 floor cost) are most susceptible.

---

### Likelihood Explanation

The condition `delta_gas < 0` is reachable by any unprivileged transaction sender. It occurs whenever the EVM gas consumed by a transaction exceeds `native_used / native_per_gas`. This is a normal operating condition — the double-resource accounting model is explicitly designed to handle divergence in both directions, but only one direction is implemented. The developer `// TODO` comment confirms this is a known gap. Any L2 transaction or L1→L2 priority transaction can trigger this path.

---

### Recommendation

Apply the symmetric adjustment: when `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so that the user is refunded the gas that was charged in EVM terms but not consumed in native terms.

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption was less than EVM gas consumption.
    // Return the difference to the user, but not below the minimum.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

### Proof of Concept

**Setup:** Deploy a contract that performs a computationally cheap but EVM-gas-consuming operation (e.g., a large number of `SLOAD` on warm slots, which cost 100 gas each in EVM but are cheap in native/RISC-V cycles). Set `native_price` high enough that `native_per_gas` is small (e.g., 1), so `native_used / native_per_gas` is much less than `gas_used`.

**Expected (correct) behavior:** `gas_used` is reduced by `|delta_gas|`; sender receives a larger refund.

**Actual behavior:** `gas_used` is not reduced; sender receives a smaller refund; operator receives the excess.
