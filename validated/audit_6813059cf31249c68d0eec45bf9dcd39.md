### Title
`set_eth_connector_contract_account` Replaces Live Connector Without Checking Active Operations or Balance Migration - (File: engine/src/contract_methods/connector.rs)

### Summary

`set_eth_connector_contract_account` allows the owner to atomically overwrite the ETH connector account pointer stored in the engine without verifying that user balances have been migrated or that no operations are in flight. Because every deposit, withdrawal, and base-token exit path reads this pointer at call time, replacing it while funds are custodied by the old connector permanently severs the engine from those funds.

### Finding Description

`set_eth_connector_contract_account` writes a new connector `AccountId` and serialization type directly to storage with no preconditions beyond `require_running` and `require_owner_only`:

```rust
// engine/src/contract_methods/connector.rs:418-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

The stored connector account ID is the single source of truth consumed by every critical path:

1. **`ft_on_transfer`** (line 81) — distinguishes base-token deposits from ERC-20 deposits by comparing `predecessor_account_id` against `get_connector_account_id`. After the swap, the old connector's in-flight `ft_transfer_call` callbacks are misclassified as ERC-20 transfers and fail.

2. **`ExitToNear` precompile** (line 431 of `engine-precompiles/src/native.rs`) — reads `get_eth_connector_contract_account` at EVM execution time to build the `ft_transfer` promise target. After the swap, base-token exits target the new connector, which holds no user balances.

3. **`ExitToEthereum` precompile** (line 901 of `engine-precompiles/src/native.rs`) — similarly reads the connector account to build the `withdraw` promise target.

4. **`withdraw`** (line 58) — calls `return_promise(io, env, "engine_withdraw", ...)`, which dispatches to the connector account read from storage at call time.

None of these callers have any fallback to the old connector. The old connector retains all custodied ETH balances but the engine no longer routes any operation to it.

### Impact Explanation

**High — Permanent freezing of funds.**

All ETH (base token) balances custodied by the old connector become unreachable the moment the pointer is overwritten. Users cannot withdraw (the engine calls `engine_withdraw` on the new connector, which has no record of their balance), cannot receive base-token deposits (the old connector's callbacks are misrouted), and cannot exit via the precompile (the promise targets the new connector). The funds remain locked in the old connector contract indefinitely with no recovery path inside the engine.

### Likelihood Explanation

The operation is a legitimate administrative action (connector upgrade or migration). An owner performing a connector migration without first draining all user balances and migrating state would trigger this silently — there is no on-chain guard, no warning, and no minimum-balance check. The likelihood is low in absolute terms but the consequence of a single mistaken call is total loss of access to all custodied ETH.

### Recommendation

Before overwriting the connector account pointer:

1. **Verify zero outstanding balance** — read the old connector's `ft_total_supply` and reject the call if it is non-zero.
2. **Two-step migration** — introduce a pending-connector state; only commit the swap after the old connector confirms a zero balance.
3. **Grace period** — emit an event and enforce a time-lock so users can withdraw before the pointer changes.

### Proof of Concept

1. Users deposit ETH to Aurora; the old connector (`old.near`) holds 1 000 ETH in NEP-141 supply.
2. Owner calls `set_eth_connector_contract_account` pointing to `new.near` (a freshly deployed connector with zero balances).
3. A user calls `withdraw` on the engine. `return_promise` dispatches `engine_withdraw` to `new.near`. `new.near` has no record of the user's balance → call fails.
4. The user calls the `ExitToNear` precompile. `get_eth_connector_contract_account` returns `new.near`. The `ft_transfer` promise targets `new.near`, which has no balance → call fails.
5. The old connector (`old.near`) still holds 1 000 ETH in NEP-141 supply, but the engine will never route another call to it. All user funds are permanently frozen.
