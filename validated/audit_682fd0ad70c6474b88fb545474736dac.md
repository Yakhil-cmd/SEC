### Title
`floor_gas` Ignores Contract-Creation Transaction Type, Underestimating EIP-7623 Floor for Deploy Transactions - (File: `engine-transactions/src/lib.rs`)

---

### Summary

`NormalizedEthTransaction::floor_gas` unconditionally uses `config.gas_transaction_call` (21 000) as its base cost for every transaction, including contract-creation transactions. `NormalizedEthTransaction::intrinsic_gas`, by contrast, correctly branches on `self.to.is_none()` and uses `config.gas_transaction_create` (53 000) plus init-code cost for deployments. When a deployment carries enough calldata to make `floor_gas` the binding minimum (i.e. `floor_gas > intrinsic_gas`), the enforced gas-limit floor is 32 000 gas units lower than it should be, allowing a deployer to submit the transaction with a gas limit — and therefore a prepaid fee — that is 32 000 × `gas_price` Wei less than the protocol intends to collect.

---

### Finding Description

`intrinsic_gas` correctly distinguishes transaction types:

```rust
// engine-transactions/src/lib.rs  lines 165-171
let is_contract_creation = self.to.is_none();
let base_gas = if is_contract_creation {
    config.gas_transaction_create + init_code_cost(config, &self.data)?
} else {
    config.gas_transaction_call
};
```

`floor_gas` (EIP-7623) does not:

```rust
// engine-transactions/src/lib.rs  lines 238-246
let base_gas = config.gas_transaction_call;   // always 21 000, even for CREATE
let tokens_in_calldata = num_non_zero_bytes
    .checked_mul(4)
    .and_then(|gas| gas.checked_add(num_zero_bytes))
    .ok_or(Error::GasOverflow)?;
tokens_in_calldata
    .checked_mul(config.total_cost_floor_per_token)
    .and_then(|gas| gas.checked_add(base_gas))
    .ok_or(Error::GasOverflow)
```

The engine enforces the maximum of the two values as the minimum gas limit:

```rust
// engine/src/engine.rs  lines 1073-1081
let floor_gas = transaction.floor_gas(CONFIG)...;
if transaction.gas_limit < core::cmp::max(intrinsic_gas, floor_gas).into() {
    return Err(EngineErrorKind::IntrinsicGasNotMet.into());
}
```

For a deployment with `n` non-zero calldata bytes, `floor_gas` exceeds `intrinsic_gas` when roughly `n > 1 333`. At that point `floor_gas` becomes the binding check, but it is 32 000 gas units too low because it used 21 000 instead of 53 000 as the base. The deployer may therefore set `gas_limit = floor_gas_incorrect` and have the transaction accepted, prepaying 32 000 × `effective_gas_price` Wei less than the protocol should collect.

---

### Impact Explanation

**High — Theft of unclaimed yield / fee-revenue loss.**

For every qualifying contract-creation transaction (calldata > ~1 333 non-zero bytes), the protocol fails to collect up to `32 000 × gas_price` Wei in fees. A deployer who knows the threshold can craft large-calldata deployments, set the gas limit exactly at the (incorrect) floor, and consistently underpay the intended minimum fee. At a gas price of 1 Gwei this is ~32 000 Gwei (~0.000032 ETH) per transaction; at higher gas prices or in bulk the shortfall is proportionally larger. The protocol relayer/treasury receives less than the EIP-7623 floor mandates.

---

### Likelihood Explanation

**Medium.** The condition requires calldata larger than ~1 333 non-zero bytes, which is realistic for any non-trivial contract deployment. The exploit requires no special privilege — any EVM user submitting a `CREATE` transaction via the `submit` entry-point can trigger it. The attacker only needs to know the threshold and set `gas_limit` accordingly.

---

### Recommendation

In `floor_gas`, branch on `self.to.is_none()` exactly as `intrinsic_gas` does, and use `config.gas_transaction_create` (plus init-code cost) as the base for contract-creation transactions:

```rust
let base_gas = if self.to.is_none() {
    config.gas_transaction_create + init_code_cost(config, &self.data)?
} else {
    config.gas_transaction_call
};
```

This makes the two functions consistent and ensures the EIP-7623 floor is never lower than the intrinsic cost for any transaction type.

---

### Proof of Concept

1. Construct a `CREATE` transaction (`to = None`) with ~2 000 non-zero calldata bytes.
2. Compute `floor_gas` as the engine does: `4 × 2000 × 10 + 21 000 = 101 000`.
3. Compute `intrinsic_gas`: `53 000 + ceil(2000/32)×2 + 2000×16 = 53 000 + 126 + 32 000 = 85 126`.
4. `max(85 126, 101 000) = 101 000` → the engine accepts `gas_limit = 101 000`.
5. The correct floor should be `4 × 2000 × 10 + 53 000 = 133 000`.
6. The deployer therefore prepays `(133 000 − 101 000) × gas_price = 32 000 × gas_price` Wei less than intended.
7. If the transaction exhausts its 101 000-gas limit, the relayer collects only `101 000 × gas_price` instead of the correct `133 000 × gas_price`.
