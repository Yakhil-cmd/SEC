### Title
XCC Router `add_batch_actions` Is Unimplemented by Default, Permanently Breaking Batch Promise Execution - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `add_batch_actions` function in the XCC Router contract is compiled to `unimplemented!()` whenever the `all-promise-actions` Cargo feature is absent. Because the xcc-router's `default` feature set is empty (`default = []`), the production binary ships with this stub. Any call to `execute` or `execute_scheduled` that carries a `PromiseArgs::Recursive` containing a `SimpleNearPromise::Batch` variant will unconditionally panic, permanently breaking that execution path.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, two conditional-compilation blocks define `add_batch_actions`:

```rust
#[cfg(not(feature = "all-promise-actions"))]
fn add_batch_actions(_id: PromiseIndex, _actions: &[PromiseAction]) {
    unimplemented!("NEAR batch transactions are not supported. ...")
}

#[cfg(feature = "all-promise-actions")]
fn add_batch_actions(id: PromiseIndex, actions: &[PromiseAction]) { ... }
``` [1](#0-0) 

The xcc-router's `Cargo.toml` declares `default = []`, so `all-promise-actions` is never activated in a standard build: [2](#0-1) 

`add_batch_actions` is called from `recursive_promise_create` for both `SimpleNearPromise::Batch` arms: [3](#0-2) 

`recursive_promise_create` is reached from `promise_create`, which is called by the two public entry points `execute` and `execute_scheduled`: [4](#0-3) [5](#0-4) 

The `PromiseArgs::Recursive` variant (which wraps `NearPromise`, which in turn can contain `SimpleNearPromise::Batch`) is a first-class member of the `CrossContractCallArgs` type accepted by the XCC precompile: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Every call to `execute` or `execute_scheduled` that carries a batch-type recursive promise panics unconditionally. In NEAR, a panic reverts all state changes in the current call frame, so the scheduled promise entry is not permanently deleted, but the execution itself can never succeed. NEAR attached to the XCC call for storage staking or token transfers cannot be disbursed through the intended batch path. Because `send_refund` is restricted to the parent Aurora Engine account and not callable by the user directly, any NEAR locked in the router sub-account for a batch-promise workflow is effectively frozen until an operator intervenes.

**Impact: High — Temporary (potentially indefinite) freezing of funds.**

---

### Likelihood Explanation

The `all-promise-actions` feature is opt-in and absent from `default = []`. Any deployment built with the standard `cargo build` or the project's own `Makefile.toml` targets ships the broken stub. An EVM contract author who consults the `PromiseArgs` type definition and constructs a `Recursive` batch promise will trigger the panic on every attempt. No special privilege is required; any EVM address that calls the XCC precompile with a batch-type promise is sufficient.

---

### Recommendation

Either:
1. Add `all-promise-actions` to the `default` feature list in `etc/xcc-router/Cargo.toml` so the real implementation is always compiled in, or
2. Replace the `unimplemented!()` stub with a graceful `env::panic_str("ERR_BATCH_NOT_SUPPORTED")` and, more importantly, reject `SimpleNearPromise::Batch` inputs at the XCC precompile layer before they reach the router, so callers receive a clear error rather than an opaque panic.

---

### Proof of Concept

1. An EVM contract calls the XCC precompile with:
   ```
   CrossContractCallArgs::Eager(
       PromiseArgs::Recursive(
           NearPromise::Simple(
               SimpleNearPromise::Batch(PromiseBatchAction { ... })
           )
       )
   )
   ```
2. The Aurora Engine forwards this to the XCC Router's `execute` method.
3. `execute` → `promise_create` → `recursive_promise_create` matches `SimpleNearPromise::Batch` and calls `add_batch_actions`.
4. Because `all-promise-actions` is not in `default = []`, the compiled `add_batch_actions` body is `unimplemented!(...)`, which expands to a `panic!`.
5. The NEAR transaction reverts; the XCC call never executes; any NEAR deposited for the batch operation cannot be disbursed through this path. [1](#0-0) [2](#0-1) [8](#0-7)

    fn recursive_promise_create(promise: &NearPromise) -> PromiseIndex {
        match promise {
            NearPromise::Simple(x) => match x {
                SimpleNearPromise::Create(call) => Self::base_promise_create(call),
                SimpleNearPromise::Batch(batch) => {
                    let target = batch.target_account_id.as_ref().parse().unwrap();
                    let id = env::promise_batch_create(&target);
                    Self::add_batch_actions(id, &batch.actions);
                    id
                }
            },
            NearPromise::Then { base, callback } => {
                let base_index = Self::recursive_promise_create(base);
                match callback {
                    SimpleNearPromise::Create(call) => env::promise_then(
                        base_index,
                        call.target_account_id.as_ref().parse().unwrap(),
                        call.method.as_str(),
                        &call.args,
                        NearToken::from_yoctonear(call.attached_balance.as_u128()),
                        Gas::from_gas(call.attached_gas.as_u64()),
                    ),
                    SimpleNearPromise::Batch(batch) => {
                        let id = env::promise_batch_then(
                            base_index,
                            &batch.target_account_id.as_ref().parse().unwrap(),
                        );
                        Self::add_batch_actions(id, &batch.actions);
                        id
                    }
                }
            }
            NearPromise::And(promises) => {
                let indices: Vec<PromiseIndex> = promises
                    .iter()
                    .map(Self::recursive_promise_create)
                    .collect();
                env::promise_and(&indices)
            }
        }
    }
