### Title
Unbounded Ingress-Range Loop in ICP Rosetta `construction_payloads` Enables OOM DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

### Summary

The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from the request metadata and feeds them directly into an unbounded `while now < ingress_end` loop with no range guard. The ICRC1 Rosetta handler has an explicit guard for exactly this case; the ICP handler does not. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes the loop to iterate ~153 billion times, exhausting process memory and crashing the Rosetta node.

### Finding Description

**Attacker-controlled input path:**

`ConstructionPayloadsRequestMetadata` deserializes `ingress_start` and `ingress_end` as plain `Option<u64>` with no validation: [1](#0-0) 

These values are passed directly into `ic_types::time::Time::from_nanos_since_unix_epoch` with no bounds check: [2](#0-1) 

**The unbounded loop:**

`interval` is computed as `MAX_INGRESS_TTL - PERMITTED_DRIFT - 120s`: [3](#0-2) 

From `rs/limits/src/lib.rs`, `MAX_INGRESS_TTL = 300s` and `PERMITTED_DRIFT = 60s`, so `interval = 120s = 120,000,000,000 ns`. [4](#0-3) 

The loop then runs without any upper-bound guard: [5](#0-4) 

With `ingress_start=0` and `ingress_end=u64::MAX (18,446,744,073,709,551,615)`, the number of iterations before `now >= ingress_end` is:

```
u64::MAX / 120_000_000_000 ≈ 153,722,867,280 iterations
```

Each iteration pushes a `u64` (8 bytes) into `ingress_expiries`. At 8 bytes/entry, the process exhausts all available RAM (e.g., 64 GB at ~8 billion entries) long before the loop terminates.

**`Time::add_assign` behavior:**

The `+=` operator on `ic_types::time::Time` uses `Duration::from_nanos(self.0) + other`, which panics on `Duration` overflow in both debug and release builds. However, OOM occurs billions of iterations before any overflow is reached. [6](#0-5) 

**The missing guard — contrast with ICRC1:**

The ICRC1 handler has two explicit guards that prevent this: [7](#0-6) 

The ICP handler has no equivalent guards anywhere before the loop.

### Impact Explanation

The ICP Rosetta node is a public HTTP service. The `/construction/payloads` endpoint requires no authentication. A single malicious HTTP POST with `metadata.ingress_start=0` and `metadata.ingress_end=18446744073709551615` causes the Rosetta process to allocate memory until the OOM killer terminates it. This halts all ICP ledger transaction submissions and balance queries for exchanges that depend on the Rosetta node. The "theft via stale-state" secondary claim in the question is speculative and not directly caused by this bug; the primary impact is a reliable, single-request DoS.

### Likelihood Explanation

The endpoint is unauthenticated and publicly reachable. The exploit requires a single HTTP request with two integer fields set to boundary values. No special knowledge, credentials, or infrastructure is needed. The ICRC1 handler's guard code demonstrates that DFINITY developers were aware of the need to bound this range — the guard was simply not applied to the ICP handler.

### Recommendation

Add the same range validation present in the ICRC1 handler before the `while` loop in `construction_payloads`:

1. Reject if `ingress_start >= ingress_end`.
2. Reject if `ingress_end - ingress_start` exceeds a reasonable maximum (e.g., 24 hours = `86_400_000_000_000 ns`), or equivalently, cap the number of expiry entries (e.g., `<= 1440` entries for a 24-hour window at 1-minute intervals).
