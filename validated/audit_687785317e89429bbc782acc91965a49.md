### Title
Relayer Reward Over-Payment Due to Incorrect Gas Quantity in `refund_unused_gas` When `fixed_gas` Is Set - (File: `engine/src/engine.rs`)

### Summary
In Silo mode, when a `fixed_gas` amount is configured, the `refund_unused_gas` function computes the relayer's priority-fee reward using `fixed_gas` instead of the actual `gas_used`. This causes the relayer to receive `(fixed_gas − gas_used) × priority_fee_per_gas` more ETH than they are entitled to on every transaction, constituting a systematic theft of unclaimed yield from the sender.

### Finding Description

`refund_unused_gas` in `engine/src/engine.rs` defines a single closure `gas_to_wei` that is reused for both the sender's refund calculation (`spent_amount`) and the relayer's reward calculation (`reward_amount`):

```rust
let gas_to_wei = |price: U256| {
    fixed_gas
        .map_or_else(|| gas_used.into(), EthGas::as_u256)
        .checked_mul(price)
        .map(Wei::new)
        .ok_or(GasPaymentError::EthAmountOverflow)
};

let spent_amount  = gas_to_wei(gas_result.effective_gas_price)?;
let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;
``` [1](#0-0) 

When `fixed_gas = Some(x)`, the closure always substitutes `x` for the gas quantity, so:

- `spent_amount  = x × effective_gas_price`
- `reward_amount = x × priority_fee_per_gas`

The `spent_amount` path is intentionally designed this way: `charge_gas` already deducted `fixed_gas × effective_gas_price` from the sender, so `refund = prepaid_amount − spent_amount = 0` (no refund, by design). [2](#0-1) 

However, the `reward_amount` path is wrong. The relayer's reward should reflect the actual computational work performed (`gas_used × priority_fee_per_gas`), not the fixed billing unit. The closure incorrectly applies `fixed_gas` to both quantities, so the relayer receives:

```
reward_amount = fixed_gas × priority_fee_per_gas
             = gas_used × priority_fee_per_gas   +   (fixed_gas − gas_used) × priority_fee_per_gas
                  ↑ correct portion                        ↑ unearned surplus
```

The surplus `(fixed_gas − gas_used) × priority_fee_per_gas` is credited to the relayer via `add_balance` with no corresponding debit anywhere — it is simply extracted from the sender's prepaid amount that should otherwise be burned. [3](#0-2) 

`fixed_gas` is a Silo-mode parameter set by the operator via `set_silo_params` / `set_fixed_gas`. It is stored in NEAR state and read back on every `submit` call. [4](#0-3) 

In practice `fixed_gas` is set to a round number (e.g. `1_000_000` gas units in tests) while a simple ETH transfer consumes only ~21 000 gas, meaning the relayer collects roughly `(1_000_000 − 21_000) × priority_fee` in unearned reward on every such transaction. [5](#0-4) 

### Impact Explanation

Every transaction submitted through a Silo-mode Aurora Engine instance with `fixed_gas` configured results in the relayer receiving `(fixed_gas − gas_used) × priority_fee_per_gas` more ETH than they earned. This is a continuous, systematic drain: the sender's prepaid gas budget is partially redirected to the relayer rather than being burned. This constitutes **theft of unclaimed yield** (High severity) — the sender overpays the relayer on every transaction.

### Likelihood Explanation

Silo mode with `fixed_gas` is an explicitly supported, documented production feature of Aurora Engine. Once a Silo operator enables it, the bug fires on **every single transaction** submitted through that instance. No special attacker action is required beyond being the designated relayer (a role filled by an external, non-admin party). The relayer does not need to craft malicious calldata; the surplus accrues automatically whenever `gas_used < fixed_gas`, which is the common case for any transaction whose actual execution is cheaper than the fixed ceiling.

### Recommendation

Split the gas quantity used for `spent_amount` and `reward_amount`. When `fixed_gas` is set, `spent_amount` should still use `fixed_gas` (to zero out the sender's refund as intended), but `reward_amount` must always use `gas_used`:

```rust
let spent_gas = fixed_gas.map_or_else(|| gas_used.into(), EthGas::as_u256);
let spent_amount  = spent_gas
    .checked_mul(gas_result.effective_gas_price)
    .map(Wei::new)
    .ok_or(GasPaymentError::EthAmountOverflow)?;

let reward_amount = U256::from(gas_used)
    .checked_mul(gas_result.priority_fee_per_gas)
    .map(Wei::new)
    .ok_or(GasPaymentError::EthAmountOverflow)?;
```

### Proof of Concept

Consider a Silo instance with:
- `fixed_gas = 1_000_000`
- `priority_fee_per_gas = 1 Gwei` (1e9 wei)
- A simple ETH transfer: `gas_used = 21_000`

**Current behaviour** (buggy):
- Sender pays: `1_000_000 × gas_price` (deducted in `charge_gas`)
- Sender refund: `0`
- Relayer reward: `1_000_000 × 1e9 = 1_000_000_000_000_000 wei = 0.001 ETH`

**Correct behaviour**:
- Sender pays: `1_000_000 × gas_price`
- Sender refund: `0` (intentional in Silo mode)
- Relayer reward: `21_000 × 1e9 = 21_000_000_000_000 wei ≈ 0.000021 ETH`

**Surplus to relayer per transaction**: `(1_000_000 − 21_000) × 1e9 ≈ 0.000979 ETH`

Over 1 000 transactions the relayer extracts ~0.979 ETH in unearned fees. The entry path is the public `submit` NEAR method, callable by any relayer account without any privilege beyond being whitelisted in Silo mode. [6](#0-5) [7](#0-6) [8](#0-7)

pub fn refund_unused_gas<I: IO>(
    io: &mut I,
    sender: &Address,
    gas_used: u64,
    gas_result: &GasPaymentResult,
    relayer: &Address,
    fixed_gas: Option<EthGas>,
) -> Result<(), GasPaymentError> {
    if gas_result.effective_gas_price.is_zero() {
        return Ok(());
    }

    let (refund, relayer_reward) = {
        let gas_to_wei = |price: U256| {
            fixed_gas
                .map_or_else(|| gas_used.into(), EthGas::as_u256)
                .checked_mul(price)
                .map(Wei::new)
                .ok_or(GasPaymentError::EthAmountOverflow)
        };

        let spent_amount = gas_to_wei(gas_result.effective_gas_price)?;
        let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;

        let refund = gas_result
            .prepaid_amount
            .checked_sub(spent_amount)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        (refund, reward_amount)
    };

    if !refund.is_zero() {
        add_balance(io, sender, refund)?;
    }

    if !relayer_reward.is_zero() {
        add_balance(io, relayer, relayer_reward)?;
    }

    Ok(())
}

/// Return SILO parameters.
pub fn get_silo_params<I: IO>(io: &I) -> Option<SiloParamsArgs> {
    let params = get_fixed_gas(io)
        .and_then(|cost| get_erc20_fallback_address(io).map(|address| (cost, address)));

    params.map(|(cost, address)| SiloParamsArgs {
        fixed_gas: cost,
        erc20_fallback_address: address,
    })
}

/// Set SILO parameters.
pub fn set_silo_params<I: IO>(io: &mut I, args: Option<SiloParamsArgs>) {
    let (cost, address) = args.map_or((None, None), |params| {
        (Some(params.fixed_gas), Some(params.erc20_fallback_address))
    });

    set_fixed_gas(io, cost);
    set_erc20_fallback_address(io, address);
}

/// Return gas amount per transaction.
pub fn get_fixed_gas<I: IO>(io: &I) -> Option<EthGas> {
    let key = fixed_gas_key();
    io.read_storage(&key)
        .and_then(|bytes| bytes.to_value().ok())
}

/// Set an amount of gas per transaction.
pub fn set_fixed_gas<I: IO>(io: &mut I, gas: Option<EthGas>) {
    let key = fixed_gas_key();

    if let Some(gas) = gas {
        io.write_borsh(&key, &gas);
    } else {
        io.remove_storage(&key);
    }
}
