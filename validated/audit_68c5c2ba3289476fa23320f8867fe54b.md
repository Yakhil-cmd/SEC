### Title
Missing Zero-Address Validation in Minter Address Constructor - (`File: rs/ethereum/cketh/minter/EthDepositHelper.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

All three ckETH/ckERC20 deposit helper smart contracts (`CkEthDeposit`, `CkErc20Deposit`, `CkDeposit`) store the IC minter's Ethereum address in an `immutable` state variable set in the constructor, but none of them validate that the supplied address is non-zero. If any contract is deployed with `address(0)` as the minter address, every subsequent user deposit permanently burns the deposited ETH or ERC20 tokens while still emitting the deposit event that the IC minter scrapes to mint ckETH/ckERC20 — producing unbacked chain-fusion tokens.

---

### Finding Description

**`CkEthDeposit` (`EthDepositHelper.sol`, lines 17–19):**

```solidity
constructor(address _cketh_minter_main_address) {
    cketh_minter_main_address = payable(_cketh_minter_main_address);
}
```

No zero-address guard. `deposit()` at line 34 calls `cketh_minter_main_address.transfer(msg.value)`, sending ETH to `address(0)` if misconfigured. [1](#0-0) [2](#0-1) 

**`CkErc20Deposit` (`ERC20DepositHelper.sol`, lines 483–485):**

```solidity
constructor(address _cketh_minter_main_address) {
    cketh_minter_main_address = _cketh_minter_main_address;
}
```

No zero-address guard. `deposit()` at line 500 calls `erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount)`, sending ERC20 tokens to `address(0)` if misconfigured. [3](#0-2) [4](#0-3) 

**`CkDeposit` (`DepositHelperWithSubaccount.sol`, lines 488–490):**

```solidity
constructor(address _minterAddress) {
    minterAddress = payable(_minterAddress);
}
```

No zero-address guard. Both `depositEth()` (line 505) and `depositErc20()` (line 519–523) forward funds to `minterAddress`. Notably, `depositErc20()` validates that `erc20Address != ZERO_ADDRESS` (line 517) but applies no equivalent check to `minterAddress` itself — an inconsistency that highlights the oversight. [5](#0-4) [6](#0-5) [7](#0-6) 

Because `minterAddress` / `cketh_minter_main_address` is declared `immutable` in all three contracts, a zero-address deployment cannot be corrected post-deployment; the contract must be redeployed. [8](#0-7) [9](#0-8) [10](#0-9) 

---

### Impact Explanation

**Vulnerability class:** Ledger conservation bug (chain-fusion mint/burn).

If any helper contract is deployed with `address(0)`:

1. A user calls `deposit()` / `depositEth()` / `depositErc20()` and sends real ETH or ERC20 tokens.
2. The contract emits the `ReceivedEth` / `ReceivedErc20` / `ReceivedEthOrErc20` event normally.
3. The ETH or ERC20 is transferred to `address(0)` — permanently burned on Ethereum.
4. The IC ckETH minter scrapes the emitted event and mints the corresponding amount of ckETH or ckERC20 to the user's IC account.
5. The minter's Ethereum address never received the backing funds.

Result: ckETH or ckERC20 tokens are minted on the IC without any corresponding ETH/ERC20 collateral, breaking the 1:1 backing invariant that the entire chain-fusion system depends on. The IC minter's own validation (`validate_address_as_destination` rejects `Address::ZERO` for *withdrawal* destinations) does not protect against this deposit-side misconfiguration. [11](#0-10) 

---

### Likelihood Explanation

Likelihood is **low but non-negligible**. Deployment of these contracts requires a privileged deployer action, but:

- The constructor accepts any `address` without revert, so a typo or scripting error (e.g., passing an uninitialized variable, a zero-padded value, or a wrong environment's address) silently succeeds.
- The `immutable` keyword means the error is unrecoverable without redeployment and a new NNS/upgrade proposal to update the minter's configured helper contract address.
- The IC minter's log-scraping timer would immediately begin minting unbacked tokens as soon as the first user deposit arrives, with no on-chain signal of the misconfiguration beyond `getMinterAddress()` returning `address(0)`.
- Historical precedent: the minter upgrade proposal `minter_upgrade_2024_11_22.md` shows that helper contract addresses are set via upgrade args, meaning a new contract address is registered with the minter after deployment — a two-step process where a zero-address deployment could go undetected until the first deposit. [12](#0-11) 

---

### Recommendation

Add a zero-address guard in each constructor:

```solidity
// EthDepositHelper.sol
constructor(address _cketh_minter_main_address) {
    require(_cketh_minter_main_address != address(0), "CkEthDeposit: zero minter address");
    cketh_minter_main_address = payable(_cketh_minter_main_address);
}

// ERC20DepositHelper.sol
constructor(address _cketh_minter_main_address) {
    require(_cketh_minter_main_address != address(0), "CkErc20Deposit: zero minter address");
    cketh_minter_main_address = _cketh_minter_main_address;
}

// DepositHelperWithSubaccount.sol
constructor(address _minterAddress) {
    require(_minterAddress != address(0), "CkDeposit: zero minter address");
    minterAddress = payable(_minterAddress);
}
```

Additionally, the IC minter should verify that `getMinterAddress()` on a newly registered helper contract returns its own known Ethereum address before beginning to scrape logs from it.

---

### Proof of Concept

1. Deploy `CkEthDeposit` passing `address(0)` as `_cketh_minter_main_address` — deployment succeeds with no revert.
2. Register this contract address with the IC ckETH minter via an upgrade arg (`ethereum_contract_address`).
3. Call `deposit(principal_bytes32)` with `{value: 1 ether}`.
4. Observe: `ReceivedEth(msg.sender, 1 ether, principal_bytes32)` event is emitted; 1 ETH is sent to `address(0)` (burned); IC minter scrapes the event and mints 1 ckETH to the IC account identified by `principal_bytes32`.
5. The IC minter's Ethereum balance is unchanged; 1 ckETH is now unbacked. [13](#0-12) [14](#0-13) [15](#0-14)

pub fn validate_address_as_destination(address: &str) -> Result<Address, AddressValidationError> {
    let address =
        Address::from_str(address).map_err(|e| AddressValidationError::Invalid { error: e })?;
    if address == Address::ZERO {
        return Err(AddressValidationError::NotSupported(address));
    }
    if crate::blocklist::is_blocked(&address) {
        return Err(AddressValidationError::Blocked(address));
    }
    Ok(address)
