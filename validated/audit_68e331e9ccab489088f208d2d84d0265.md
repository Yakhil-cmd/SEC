Audit Report

## Title
Signed Cast Overflow in `add_expiration` Corrupts ICRC-21 Consent Message Expiration Display — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`add_expiration` casts a caller-supplied `expires_at: u64` to `i64` without bounds-checking at lines 232–233 and 265. For any `expires_at > i64::MAX`, the cast wraps to a negative value, causing the `GenericDisplayMessage` branch to emit `"Invalid nanosecond: 4294967295"` and the `FieldsDisplayMessage` branch to silently display Unix epoch (Jan 1, 1970) as the expiration. Because ICRC-21 consent messages are the security mechanism wallets use to show users what they are signing, a corrupted expiration display can cause users to approve allowances under false pretenses.

## Finding Description
In `packages/icrc-ledger-types/src/icrc21/responses.rs`, `add_expiration` performs two unchecked `as i64` casts:

**GenericDisplayMessage branch (lines 232–233):**
```rust
let seconds = (ts as i64) / 10_i64.pow(9);
let nanos   = ((ts as i64) % 10_i64.pow(9)) as u32;
```
For `ts = u64::MAX`:
- `u64::MAX as i64` → `-1` (wrapping)
- `(-1) / 1_000_000_000` → `0` (truncates toward zero)
- `(-1) % 1_000_000_000` → `-1` (Rust: sign follows dividend)
- `(-1_i64) as u32` → `4_294_967_295` (wraps to `u32::MAX`)
- `from_unix_timestamp(0)` → `Ok` (epoch)
- `replace_nanosecond(4_294_967_295)` → `Err` (> 999_999_999)
- Result: consent message contains `"Invalid nanosecond: 4294967295"`

**FieldsDisplayMessage branch (lines 265–270):**
```rust
let seconds = (expires_at as i64) / 10_i64.pow(9);  // = 0 for u64::MAX
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds as u64 },  // = 0 = Unix epoch
));
```
For `expires_at = u64::MAX`, `seconds = 0`, so `amount = 0`, displaying Jan 1, 1970 as the expiration when the actual expiration is year ~2554.

The `expires_at` value flows directly from caller-supplied `ApproveArgs` through `with_expires_at` (lib.rs line 463) into `add_expiration` with no sanitization. No existing guard checks whether `expires_at` fits in `i64`.

## Impact Explanation
ICRC-21 consent messages are the security layer wallets use to present human-readable summaries of what a user is signing. The `FieldsDisplayMessage` bug is the more serious path: a user sees `"Approval expiration: Jan 1, 1970"` and may reasonably conclude the allowance is already expired and therefore harmless, then approve an allowance that is actually valid for ~584 years. This constitutes a significant ICRC ledger security impact with concrete user harm — a user is deceived into granting a long-lived token allowance they believe to be inert. This fits the **High** impact category: "Significant ledger security impact with concrete user or protocol harm."

## Likelihood Explanation
Any unprivileged principal can call `icrc21_consent_message` with a crafted `icrc2_approve` argument containing `expires_at = u64::MAX` (or any value above `i64::MAX`). No special role, key, or governance majority is required. The endpoint is a standard ingress/query endpoint on every ICRC-2 ledger implementing ICRC-21. The attack is trivially repeatable and requires no prior access or elevated privilege.

## Recommendation
Replace the unchecked `as i64` casts in both branches with a checked conversion and return an error string if the value is out of range:

```rust
// GenericDisplayMessage branch
let ts_i64 = match i64::try_from(ts) {
    Ok(v) => v,
    Err(_) => return format!("Invalid timestamp: {ts}"),
};
let seconds = ts_i64 / 1_000_000_000;
let nanos   = (ts_i64 % 1_000_000_000) as u32; // safe: 0..999_999_999

// FieldsDisplayMessage branch
let seconds = match i64::try_from(expires_at) {
    Ok(v) => v / 1_000_000_000,
    Err(_) => { /* push error text or skip */ return; }
};
```

#[test]
fn test_add_expiration_u64_max() {
    let mut msg = ConsentMessage::GenericDisplayMessage(String::new());

        assert!(
