Audit Report

## Title
Silent ICP Burn Failure in Cycles Minting Canister Causes Unburned ICP Supply Inflation - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister is explicitly designed to swallow all errors from the ICP burn transfer. When the ICP ledger is temporarily unavailable (e.g., during an upgrade), cycles are already minted and deposited but the corresponding ICP is never burned, permanently stranding ICP in the CMC subaccount and inflating the effective ICP supply relative to circulating cycles.

## Finding Description
After a successful cycles operation, `process_top_up` (L1999–2002), `process_create_canister` (L1943–1946), and `process_mint_cycles` (L1966–1968) each call `burn_and_log` and then return `Ok(...)` unconditionally. The `burn_and_log` function (L2014–2049) issues a `send_pb` call to the ICP ledger and only logs the result — errors are never propagated and never retried:

```rust
// L2015-2016
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
```

```rust
// L2042-2047
match res {
    Ok(block) => print(format!("{msg} done in block {block}.")),
    Err((code, err)) => {
        print(format!("{msg} failed with code {code}: {err:?}"))
    }
}
```

After `burn_and_log` returns (regardless of outcome), the caller updates `blocks_notified` to `NotifiedTopUp`/`NotifiedMint`/`NotifiedCreateCanister` and returns `Ok(...)` to the user. The block is then permanently marked as processed, preventing any retry. The ICP remains in the CMC's subaccount — not burned, not refundable.

The existing deduplication guard (`set_block_status_to_processing`) correctly prevents double-spending but also prevents the user from retrying a notification whose burn silently failed.

## Impact Explanation
This is an illegal minting / ledger conservation violation: cycles enter circulation without the corresponding ICP being burned. The ICP total supply is not reduced as the protocol intends. The stranded ICP is permanently inaccessible to the user and is not automatically retried by the CMC. This matches the **High** impact class: "Significant ledger security impact with concrete user or protocol harm" — the ICP/cycles exchange invariant is broken for every affected notification, and the effect is permanent and cumulative across upgrade windows.

## Likelihood Explanation
The ICP ledger undergoes periodic upgrades on mainnet during which it briefly rejects calls. Any unprivileged user whose `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` call reaches the `burn_and_log` phase during this window triggers the condition. No attacker coordination, special privileges, or social engineering is required. The window is short but recurs with every ledger upgrade, and multiple users can be affected simultaneously.

## Recommendation
Replace the fire-and-forget pattern with one of:
1. **Persist and retry**: Record failed burns in CMC state and retry them in a heartbeat task.
2. **Pre-burn before minting**: Burn ICP before issuing cycles so a burn failure prevents cycles from being minted rather than leaving ICP unburned after the fact.
3. **Propagate with selective retry**: Only clear `Processing` status on transient ledger errors (reject codes indicating unavailability), allowing the user to retry; use permanent status only on definitive success.

## Proof of Concept
1. User sends N ICP to `CMC_SUBACCOUNT(canister_id)` with memo `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id }`.
3. CMC sets block status to `Processing`, calls `process_top_up`.
4. `deposit_cycles` succeeds — cycles are deposited to `canister_id`.
5. `burn_and_log` is called while the ICP ledger is being upgraded — `send_pb` returns an error.
6. `burn_and_log` logs the error and returns `()`.
7. `process_top_up` returns `Ok(cycles)`; CMC sets block status to `NotifiedTopUp(Ok(cycles))` and returns success to the user.
8. N ICP remain in `CMC_SUBACCOUNT(canister_id)`, unburned. The block is permanently processed; the user cannot retry. The ICP/cycles conservation invariant is violated by N ICP.

A deterministic integration test can reproduce this by: (a) deploying a mock ICP ledger that rejects `send_pb` calls, (b) calling `notify_top_up`, and (c) asserting that cycles were deposited while the CMC subaccount balance is unchanged and the block is marked `NotifiedTopUp`.
