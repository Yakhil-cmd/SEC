### Title
Multi-Block Batch Log Accumulator Overflow Causes Prover Panic - (`basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`)

### Summary

In the multi-block batch proving path, `ZKBatchDataKeeper.logs_storage` is a fixed-capacity `ArrayVec<Bytes32, 16384>` that accumulates L2→L1 log hashes across **all blocks** in a batch. The per-block limit allows exactly 16,384 logs per block. When a batch contains more than one block and the combined log count exceeds 16,384, `ArrayVec::push` panics, aborting the prover via `rust_abort()`. An unprivileged user can trigger this by emitting the maximum number of L2→L1 logs in one block and at least one log in a subsequent block of the same batch.

---

### Finding Description

**Root cause — capacity mismatch between per-block limit and batch accumulator:**

`MAX_NUMBER_OF_LOGS` is defined as 16,384 and is used as the per-block ceiling: [1](#0-0) 

The per-block enforcement in `check_for_block_limits` uses a strict `>` comparison, meaning a block is allowed to contain **exactly** 16,384 logs: [2](#0-1) 

The batch-level accumulator in `ZKBatchDataKeeper` is declared with the same fixed capacity of 16,384 — but it is meant to hold logs from **all blocks** in a multi-block batch: [3](#0-2) 

At the end of each block in the multi-block proving path, `apply_to_array_vec` is called to append that block's logs into the batch accumulator: [4](#0-3) 

`apply_to_array_vec` calls `ArrayVec::push` unconditionally for every log: [5](#0-4) 

`arrayvec::ArrayVec::push` **panics** when the array is already at capacity. In the RISC-V proving environment, a panic invokes `riscv_common::rust_abort()`, terminating the prover process.

**Attacker-controlled entry path:**

Any unprivileged user can call `sendToL1(bytes)` on the L1 Messenger system contract at address `0x8008`. This contract calls the L1 Messenger hook at `0x7001`, which calls `emit_l1_message`, which appends an entry to `logs_storage`: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

When a multi-block batch is being proven and the total L2→L1 log count across all blocks exceeds 16,384, the prover panics and aborts. This makes the entire batch **unprovable**, preventing the chain from settling on L1. The batch cannot be retried without excluding the offending transactions, which requires operator intervention. This is a **valid-execution unprovability** issue: the forward (sequencer) run succeeds, but the proving run aborts.

---

### Likelihood Explanation

The attack is straightforward and cheap:

1. The attacker fills one block with exactly 16,384 `sendToL1` calls (at the per-block limit). Each call costs ~9,202 gas (confirmed by test `test_l1_messenger_gas_charging`). At a 30M gas block limit, ~3,258 calls fit per block — so the attacker needs roughly 5 blocks to accumulate 16,384 logs.
2. Any single additional log in the next block of the same batch (even from a normal L1→L2 priority transaction result, which automatically emits a log via `emit_l1_l2_tx_log`) triggers the overflow.

The attacker does not need privileged access, leaked keys, or governance control. The only requirement is sufficient ETH to pay gas for the `sendToL1` calls.

---

### Recommendation

The batch-level `logs_storage` capacity must account for the maximum number of logs across all blocks in a batch. Either:

1. **Increase the `ArrayVec` capacity** in `ZKBatchDataKeeper` to `MAX_NUMBER_OF_LOGS * MAX_BLOCKS_PER_BATCH`, or
2. **Replace `ArrayVec::push` with a checked push** (e.g., `try_push`) in `apply_to_array_vec` and propagate an error instead of panicking, or
3. **Add a batch-level log count check** before calling `apply_to_array_vec` that rejects the block if adding its logs would overflow the batch accumulator.

---

### Proof of Concept

**Setup:** A multi-block batch with two blocks.

**Block 1:** Attacker submits transactions calling `sendToL1(bytes)` on `0x8008` until the block reaches exactly 16,384 L2→L1 logs (the per-block limit enforced by `check_for_block_limits`). Each transaction is accepted because `logs_used > 16384` is `false` when `logs_used == 16384`.

**Block 2:** Any single transaction that emits one L2→L1 log is included (e.g., any L1→L2 priority transaction, which automatically emits a log via `emit_l1_l2_tx_log`).

**Batch finalization (proving):** When `ZKHeaderStructurePostTxOpProvingMultiblockBatch::post_op` is called for Block 2, it calls:

```rust
io.logs_storage.apply_to_array_vec(&mut batch_data.logs_storage);
```

At this point `batch_data.logs_storage` already contains 16,384 entries from Block 1. The call to `ArrayVec::push` for the first log of Block 2 panics because the `ArrayVec<Bytes32, 16384>` is full. The RISC-V panic handler calls `rust_abort()`, terminating the prover. The batch is permanently unprovable. [5](#0-4) [3](#0-2) [2](#0-1)

pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;

    Ok(Ok(()))

    fn emit_l1_message(
        &mut self,
        _ee_type: ExecutionEnvironmentType,
        resources: &mut Self::Resources,
        address: &<Self::IOTypes as SystemIOTypesConfig>::Address,
        data: &[u8],
    ) -> Result<Bytes32, SystemError> {
        // TODO(EVM-1077): consider adding COMPUTATIONAL_PRICE_FOR_PUBDATA as in Era

        // We need to charge cost of hashing:
        // - keccak256_native_cost(L2_TO_L1_LOG_SERIALIZE_SIZE) and
        //   keccak256_native_cost(64) when reconstructing L2ToL1Log
        // - keccak256_native_cost(64) + keccak256_native_cost(data.len())
        //   when reconstructing Messages
        // - at most 1 time keccak256_native_cost(64) when building the
        //   Merkle tree (as merkle tree can contain ~2*N nodes, where the
        //   first N nodes are leaves the hash of which is calculated on the
        //   previous step).

        let hashing_native_cost =
            keccak256_native_cost::<Self::Resources>(L2_TO_L1_LOG_SERIALIZE_SIZE).as_u64()
                + 3 * keccak256_native_cost::<Self::Resources>(64).as_u64()
                + keccak256_native_cost::<Self::Resources>(data.len()).as_u64();

        // We also charge some native resource for storing the log
        let native = hashing_native_cost
            + EVENT_STORAGE_BASE_NATIVE_COST
            + EVENT_DATA_PER_BYTE_COST * (data.len() as u64);

        resources.charge(&R::from_native(R::Native::from_computational(native)))?;

        // TODO(EVM-1078): for Era backward compatibility we may need to add events for l2 to l1 log and l1 message

        // Compute data hash directly: the native cost for this keccak is already
        // pre-charged above (included in `hashing_native_cost`), and this function
        // must not charge ergs — EVM gas accounting is the caller's responsibility
        // (the L1Messenger system contract charges it before invoking the hook).
        use crypto::MiniDigest;
        let data_hash = Bytes32::from_array(crypto::sha3::Keccak256::digest(data));
        let data = UsizeAlignedByteBox::from_slice_in(data, self.allocator.clone());
        self.logs_storage
            .push_message(self.tx_number, address, data, data_hash)?;
        Ok(data_hash)
