Audit Report

## Title
MEGa `verify_pop` Missing Identity-Point Guard Allows Byzantine Dealer to Expose All Recipient Shares — (`rs/crypto/internal/crypto_lib/threshold_sig/canister_threshold_sig/src/idkg/mega.rs`)

## Summary
The `verify_pop` function in `mega.rs` performs no `is_infinity()` check on `ephemeral_key` or `pop_public_key` before delegating to `ProofOfDLogEquivalence::verify`. A Byzantine dealer can supply `ephemeral_key = identity` with a valid Sigma proof for witness `x = 0`, causing every recipient's `shared_secret` to collapse to the publicly-known identity element. This lets any on-chain observer recompute every `hm_r` mask and recover every recipient's plaintext share from the dealing, breaking MEGa ciphertext confidentiality for that dealing.

## Finding Description

**Step 1 — Identity deserialization is accepted.** `EccPoint::deserialize` explicitly returns `EccPoint::identity()` for an all-zero byte string on K256/P256, with no rejection. [1](#0-0) 

**Step 2 — `verify_pop` has no identity guard.** The function derives `pop_base` from the attacker-supplied `ephemeral_key` via a hash-to-point oracle (which never returns identity), then calls `pop_proof.verify(...)` with no prior `is_infinity()` check on either `ephemeral_key` or `pop_public_key`. [2](#0-1) 

**Step 3 — `ProofOfDLogEquivalence::verify` accepts a proof for `x = 0`.** `recover_commitment` computes `g_r = g*response + g_x*(-challenge)` and `h_r = h*response + h_x*(-challenge)`. When `g_x = h_x = identity`, both terms reduce to `g*response` and `h*response`. An attacker sets `x = 0`, picks random `r`, sets `response = r`, and computes `challenge = H(g, pop_base, identity, identity, g*r, pop_base*r, ad)`. Verification recomputes the same hash and passes unconditionally. [3](#0-2) 

**Step 4 — `check_validity` is the sole public-verification gate and contains no identity check.** `publicly_verify` → `check_validity` → `verify_pop` is the entire chain. [4](#0-3) [5](#0-4) 

**Step 5 — Decryption collapses to identity.** Each recipient computes `ubeta = ephemeral_key.scalar_mul(sk_r)`. Since `identity * sk_r = identity` for any scalar, `shared_secret = identity` for every recipient. [6](#0-5) 

**Step 6 — Private verification also passes.** Because the Byzantine dealer knows `shared_secret = identity`, they can compute every `hm_r = mega_hash_to_scalar(..., identity, identity)` and encrypt the correct polynomial shares. Recipients decrypt successfully and the commitment check passes. [7](#0-6) 

## Impact Explanation

Any external observer who sees the on-chain dealing can independently compute `hm_r` for every recipient (all inputs to `mega_hash_to_scalar` are public when `shared_secret = identity`) and recover every recipient's plaintext share. This breaks MEGa ciphertext confidentiality for the entire dealing. The shares exposed are the Byzantine dealer's polynomial evaluated at each recipient index. While the Byzantine dealer already knows their own polynomial, this attack makes it available to arbitrary third-party observers, violating the per-recipient confidentiality guarantee of the MEGa scheme. The concrete impact is disclosure of the Byzantine dealer's polynomial shares to any observer — a meaningful infrastructure security impact on the threshold signing subsystem. This maps to **Medium** severity: the attack requires node-level control (being an assigned dealer), and the exposed material is the Byzantine dealer's own polynomial contribution rather than honest dealers' shares, so it does not by itself reconstruct the subnet threshold key.

## Likelihood Explanation

The attacker must be a valid subnet node assigned as a dealer in an IDKG round — within the Byzantine fault tolerance of the protocol. No additional privileges, key material, or social engineering are required. The attack is fully local, deterministic, and repeatable for every dealing the Byzantine node participates in.

## Recommendation

Add an explicit identity-point rejection in `verify_pop` before the ZK proof is checked:

```rust
if ephemeral_key.is_infinity()? {
    return Err(CanisterThresholdError::InvalidPoint);
}
if pop_public_key.is_infinity()? {
    return Err(CanisterThresholdError::InvalidPoint);
}
```

This mirrors the existing `is_infinity` guard used in BIP32 key derivation. [8](#0-7) 

## Proof of Concept

A minimal unit test in `mega.rs::tests` can confirm this:

1. Construct `ephemeral_key = EccPoint::identity(EccCurveType::K256)` and `pop_public_key = EccPoint::identity(EccCurveType::K256)`.
2. Compute `pop_base = compute_pop_base(alg, ctype, curve, ad, dealer_index, &identity)`.
3. Create `proof = ProofOfDLogEquivalence::create(seed, alg, &EccScalar::zero(curve), &EccPoint::generator_g(curve), &pop_base, ad)`.
4. Call `verify_pop(alg, ctype, ad, dealer_index, &identity, &identity, &proof)` — assert it returns `Ok(())`.
5. For each recipient `r`, compute `hm_r = mega_hash_to_scalar(alg, plaintext_curve, dealer_index, r, ad, &pk_r, &identity, &identity)` and verify `ctext_r - hm_r` recovers the plaintext without knowledge of any recipient private key.
