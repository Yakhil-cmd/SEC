### Title
Wire Format Mismatch Between `EvmErc20.withdrawToNear` and `error_refund`-Enabled Engine Causes Permanent Fund Freeze — (`etc/eth-contracts/contracts/EvmErc20.sol`, `engine-precompiles/src/native.rs`)

---

### Summary

`EvmErc20` (V1) encodes its precompile call as `0x01 ++ amount_b(32) ++ recipient`, omitting the 20-byte `refund_address` field. When the engine is compiled with the `error_refund` feature, `parse_input` unconditionally consumes `input[1..21]` as `refund_address` and `input[21..]` as the remaining payload. This causes the engine to misparse `amount_b[20..32] ++ recipient[0..20]` as the token amount — a value that always exceeds `u128::MAX` for any real NEAR account ID — triggering `ERR_INVALID_AMOUNT`. The precompile call returns 0 (failure), but `EvmErc20.withdrawToNear` never checks the return value, so the `_burn` is committed with no corresponding NEAR-side promise. The NEP-141 tokens remain locked in Aurora's account on NEAR with no recovery path.

---

### Finding Description

**Step 1 — V1 wire format (no `refund_address`):**

`EvmErc20.withdrawToNear` burns tokens and then calls the precompile with:

```
input = 0x01 | amount_b (32 bytes) | recipient (N bytes)
``` [1](#0-0) 

The return value of the `call` opcode is captured in `res` but never inspected; there is no `require(res != 0)` or revert path.

**Step 2 — `error_refund` build's `parse_input` expects a 20-byte `refund_address` at `input[1..21]`:**

```rust
#[cfg(feature = "error_refund")]
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);          // reads amount_b[0..20] as refund_address
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))              // returns amount_b[20..32] ++ recipient as remainder
}
``` [2](#0-1) 

**Step 3 — Misparsed amount always exceeds `u128::MAX`:**

After `parse_input`, the ERC-20 branch reads `remainder[0..32]` as the amount:

```rust
0x1 => {
    let amount = parse_amount(&input[..32])?;   // input = amount_b[20..32] ++ recipient[0..20]
``` [3](#0-2) 

The 32-byte value is `amount_b[20..32]` (lower 12 bytes of the original amount) concatenated with `recipient[0..20]` (first 20 bytes of the ASCII account ID string). For any real NEAR account ID (e.g., `"alice.near"`, `"token.bridge.near"`), the lower 20 bytes are non-zero ASCII, making the resulting U256 far exceed `u128::MAX`. `parse_amount` rejects it with `ERR_INVALID_AMOUNT`.

**Step 4 — Burn committed, no promise scheduled:**

The precompile returns `Err(ExitError::Other("ERR_INVALID_AMOUNT"))`. In EVM semantics this causes the inner `call` to return 0 (failure) without reverting the caller. Since `EvmErc20.withdrawToNear` does not check `res`, the `_burn` executed at line 54 is permanently committed. [4](#0-3) 

No `PromiseArgs` is ever constructed or logged, so no `ft_transfer` is scheduled on the NEAR side. [5](#0-4) 

**Contrast with V2:** `EvmErc20V2.withdrawToNear` correctly inserts `sender` (20 bytes) between the flag and `amount_b`, matching the `error_refund` wire format:

```solidity
bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
``` [6](#0-5) 

V1 has no such field, creating the mismatch.

---

### Impact Explanation

Any user holding tokens in an ERC-20 mirror contract deployed from `EvmErc20` (V1) can call `withdrawToNear`. The `_burn` destroys their EVM-side balance. The precompile silently fails. The corresponding NEP-141 tokens remain locked in Aurora's NEAR account forever — there is no admin escape hatch, no refund callback (the `error_refund` callback is never reached because the precompile errors before scheduling any promise), and no re-entry path. This is **permanent freezing of funds**.

---

### Likelihood Explanation

The scenario requires the engine to be compiled with `error_refund` while V1 `EvmErc20` contracts remain deployed. This is a realistic protocol-upgrade window: `EvmErc20V2` was introduced precisely to carry the `refund_address` field, but on-chain V1 contracts cannot self-upgrade. Any user who calls `withdrawToNear` on a V1 mirror during or after such an upgrade loses their tokens permanently. The attacker needs no special privilege — `withdrawToNear` is a public, permissionless function callable by any token holder.

---

### Recommendation

1. **Guard the precompile against V1 callers:** In the `error_refund` build, detect the missing `refund_address` field (e.g., by checking total input length against the minimum expected for the `error_refund` format) and return an error *before* the burn is committed — or, better, revert the entire EVM transaction.
2. **Migrate all V1 `EvmErc20` deployments to V2** before enabling `error_refund` in production.
3. **Check the precompile return value in `EvmErc20.withdrawToNear`:** Add `require(res != 0, "EXIT_FAILED")` in the assembly block so that a precompile failure reverts the burn atomically.

### Proof of Concept

Deploy `EvmErc20` (V1) against an `error_refund`-enabled engine. Mint tokens to `alice`. Call `withdrawToNear("alice.near", 1e18)`.

Trace:
1. `_burn(alice, 1e18)` — EVM balance decremented.
2. Precompile receives `0x01 | [0x00…00 0xDE0B6B3A7640000] | "alice.near"` (42 bytes).
3. `parse_input` reads `input[1..21]` = first 20 bytes of `amount_b` as `refund_address`.
4. Remainder = `input[21..]` = `[0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x00]` (last 12 bytes of amount) + `"alice.near"` (10 bytes) = 22 bytes total — too short for a 32-byte amount field.
5. `parse_amount(&input[..32])` panics on an out-of-bounds slice, or if the input passes size validation with a longer recipient, the 32-byte misparsed amount has ASCII bytes in the lower 20 bytes → `U256 >> u128::MAX` → `ERR_INVALID_AMOUNT`.
6. Precompile returns `Err(...)` → inner `call` returns 0 → `res` ignored → function returns.
7. Assert: `erc20.balanceOf(alice) == 0` (burned). Assert: no `ft_transfer` promise was scheduled. Assert: NEP-141 balance of Aurora's account unchanged. Funds are permanently frozen. [7](#0-6)
