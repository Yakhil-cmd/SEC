### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Transfer â€” (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn ERC-20 tokens **before** calling the exit precompile, and never check the return value of that call. If the precompile rejects the input (e.g., an invalid NEAR account ID supplied as `recipient`), the burn is not rolled back, tokens are permanently destroyed, and no corresponding NEP-141 transfer is ever scheduled. This is the direct analog of the reported pattern: a fault-tolerant downstream call that silently swallows failure while an irreversible accounting action has already been committed.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` executes as follows:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // (1) tokens destroyed

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // (2) res is stored but NEVER checked â€” no revert on failure
    }
}
``` [1](#0-0) 

The same pattern appears in `EvmErc20V2.sol` for both `withdrawToNear` and `withdrawToEthereum`, and in `EvmErc20.sol`'s `withdrawToEthereum`. [2](#0-1) [3](#0-2) 

The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is `ExitToNear`. Its `run` method calls `ExitToNearParams::try_from(input)?`, which in turn calls `parse_recipient`, which validates the NEAR account ID:

```rust
fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    // ...
    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
``` [4](#0-3) 

`AccountId::validate` rejects any string with uppercase letters, spaces, `@`, leading/trailing separators, length < 2, or length > 64: [5](#0-4) 

When `parse_recipient` returns `Err(ExitError::Other(...))`, the precompile's `run` returns `Err(...)`, the EVM `call` opcode returns `0` (failure), but because `res` is never inspected in the Solidity assembly block, the outer transaction **does not revert**. The `_burn` at step (1) has already committed; ERC-20 supply is reduced with no NEAR promise ever emitted. [6](#0-5) 

---

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

Any ERC-20 tokens burned in step (1) are irrecoverably destroyed. No NEP-141 tokens are released on NEAR, and no refund path exists in the Solidity contract. The total bridged supply on Aurora decreases while the backing NEP-141 balance on NEAR is unchanged, creating a permanent accounting disbalance identical in structure to the reported collateralization disbalance.

---

### Likelihood Explanation

**Medium.** The `recipient` parameter is a raw `bytes` value; the Solidity contract performs zero validation before burning. Any of the following realistic scenarios triggers the loss:

- A user passes a NEAR account ID containing an uppercase letter, a space, or an `@` sign (e.g., copied from a display name).
- A frontend encodes the recipient incorrectly (e.g., passes raw binary instead of a UTF-8 account ID string).
- A smart contract integrator constructs the call programmatically with a malformed recipient.

The `BAD_ACCOUNT_IDS` list in the engine's own test suite confirms many plausible-looking strings are rejected: [7](#0-6) 

---

### Recommendation

Check the return value of the precompile `call` and revert on failure, **before** burning tokens, or at minimum after:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }   // <-- add this
    }
}
```

Applying the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol` is equally necessary.

---

### Proof of Concept

1. Deploy an `EvmErc20` mirror token (as Aurora does for every bridged NEP-141).
2. Mint tokens to address `0xAlice`.
3. `Alice` calls `withdrawToNear(bytes("Hello World"), 1000)` â€” the space character makes `"Hello World"` an invalid NEAR account ID.
4. `_burn(Alice, 1000)` executes; Alice's ERC-20 balance drops by 1000.
5. The `ExitToNear` precompile's `parse_recipient` returns `Err("ERR_INVALID_RECEIVER_ACCOUNT_ID")`.
6. The EVM `call` returns `0`; `res` is ignored; the transaction succeeds.
7. No NEAR promise is scheduled; no NEP-141 tokens are transferred.
8. Alice's 1000 tokens are permanently burned with no recourse. [1](#0-0) [8](#0-7)

    pub fn validate(account_id: &str) -> Result<(), ParseAccountError> {
        if account_id.len() < MIN_ACCOUNT_ID_LEN {
            Err(ParseAccountError::TooShort)
        } else if account_id.len() > MAX_ACCOUNT_ID_LEN {
            Err(ParseAccountError::TooLong)
        } else {
            // Adapted from https://github.com/near/near-sdk-rs/blob/fd7d4f82d0dfd15f824a1cf110e552e940ea9073/near-sdk/src/environment/env.rs#L819

            // NOTE: We don't want to use Regex here, because it requires extra time to compile it.
            // The valid account ID regex is /^(([a-z\d]+[-_])*[a-z\d]+\.)*([a-z\d]+[-_])*[a-z\d]+$/
            // Instead the implementation is based on the previous character checks.

            // We can safely assume that last char was a separator.
            let mut last_char_is_separator = true;

            for c in account_id.bytes() {
                let current_char_is_separator = match c {
                    b'a'..=b'z' | b'0'..=b'9' => false,
                    b'-' | b'_' | b'.' => true,
                    _ => {
                        return Err(ParseAccountError::Invalid);
                    }
                };
                if current_char_is_separator && last_char_is_separator {
                    return Err(ParseAccountError::Invalid);
                }
                last_char_is_separator = current_char_is_separator;
            }

            (!last_char_is_separator)
                .then_some(())
                .ok_or(ParseAccountError::Invalid)
        }
    }
