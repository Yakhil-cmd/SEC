### Title
Missing Emitter Address Validation in Event Hooks Allows Unauthorized Settlement Layer Chain ID and Interop Root Manipulation — (`system_hooks/src/event_hooks/system_context.rs`, `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

All call hooks in ZKsync OS explicitly validate the `caller` address before performing privileged state mutations. The two event hooks — `system_context_event_hook` and `interop_root_reporter_event_hook` — perform no equivalent check on the emitting contract's address within the hook body. The dispatch mechanism uses only the lower 32 bits of the emitting address (`address_low: u32`) as the lookup key. An unprivileged attacker can mine a CREATE2 salt to deploy a contract whose lower 32 bits match `SYSTEM_CONTEXT_ADDRESS_LOW` or `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW`, then emit the matching event topic to trigger `update_settlement_layer_chain_id` or `add_interop_root` with fully attacker-controlled values.

---

### Finding Description

**Root cause — asymmetric access control between call hooks and event hooks.**

Every call hook explicitly validates `caller` before acting:

```rust
// l1_messenger_hook
if caller != L1_MESSENGER_ADDRESS { return Ok((make_return_state_from_returndata_region(...), ...)); }

// mint_base_token_hook
if caller != L2_BASE_TOKEN_ADDRESS { return Ok((make_return_state_from_returndata_region(...), ...)); }

// set_bytecode_on_address_hook
if caller != CONTRACT_DEPLOYER_ADDRESS && caller != COMPLEX_UPGRADER_ADDRESS { ... }
``` [1](#0-0) [2](#0-1) [3](#0-2) 

The event hooks contain **no such check**. `system_context_event_hook` only inspects `topics[0]` for the event signature and immediately calls `update_settlement_layer_chain_id` with the attacker-supplied `topics[1]`:

```rust
pub fn system_context_event_hook<S: EthereumLikeTypes>(topics, data, caller_ee, system, resources) {
    if topics[0].as_u8_array() == SL_CHAIN_ID_UPDATED_EVENT_SIG {
        new_sl_chain_id_event_hook(topics, data, caller_ee, system, resources)
    }
}
// No emitter address check anywhere in the call chain.
``` [4](#0-3) [5](#0-4) 

`interop_root_reporter_event_hook` similarly checks only the event signature and then calls `add_interop_root` with attacker-controlled `root`, `chain_id`, and `block_or_batch_number`: [6](#0-5) 

**The dispatch mechanism provides only truncated address isolation.**

Event hooks are stored and looked up by `address_low: u32` — the lower 32 bits of the emitting contract's address:

```rust
pub struct HooksStorage<S: SystemTypes, A: Allocator + Clone> {
    call_hooks: BTreeMap<u16, SystemCallHook<S>, A>,
    event_hooks: BTreeMap<u32, SystemEventHook<S>, A>,  // only 32-bit key
}
``` [7](#0-6) [8](#0-7) 

The hooks are registered with:

```rust
hooks.add_event_hook(SYSTEM_CONTEXT_ADDRESS_LOW, SystemEventHook::new(system_context_event_hook))
hooks.add_event_hook(L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW, SystemEventHook::new(interop_root_reporter_event_hook))
``` [9](#0-8) 

System contract addresses (e.g., `0x000...0000800b`) have lower 32 bits of `0x0000800b`. An attacker can mine a CREATE2 salt to produce a contract address whose last 4 bytes match this value — a 32-bit search requiring ~2^32 keccak operations, feasible in seconds on commodity hardware. Once deployed, that contract emits `SettlementLayerChainIdUpdated(uint256)` or `InteropRootAdded(uint256,uint256,bytes32[])` with crafted parameters, and the hook fires as if the legitimate system contract emitted it.

---

### Impact Explanation

**`system_context_event_hook`:** An attacker can set the settlement layer chain ID to an arbitrary value. This corrupts the chain's cross-layer identity, breaking L1↔L2 message routing and potentially causing all subsequent cross-chain operations to reference the wrong chain, leading to permanent loss of bridged funds.

**`interop_root_reporter_event_hook`:** An attacker can inject an arbitrary `InteropRoot` (root hash, chain ID, block/batch number) into the system's interop root storage. If the interop root is used to verify cross-chain message proofs, a fake root enables the attacker to prove fraudulent messages and drain funds from cross-chain bridges.

Both impacts are direct, irreversible state corruption reachable by an unprivileged transaction sender with no special role.

---

### Likelihood Explanation

The attacker needs only to:
1. Mine a CREATE2 address with matching lower 32 bits (~2^32 hashes, seconds on a GPU).
2. Deploy a contract at that address.
3. Call a function that emits the target event with crafted parameters.

No privileged keys, governance access, or oracle manipulation is required. The attack is fully self-contained within a single transaction.

---

### Recommendation

Add an explicit emitter address check inside each event hook, mirroring the pattern used by all call hooks:

```rust
pub fn system_context_event_hook<S: EthereumLikeTypes>(
    topics, data, caller_ee, system, resources,
    emitter: &B160,  // pass full emitting address
) {
    if emitter != &SYSTEM_CONTEXT_ADDRESS {
        return Ok(());
    }
    // ... existing logic
}
```

Alternatively, pass the full 160-bit emitting address through `try_intercept_event` and validate it inside each hook, exactly as `caller` is validated in every call hook. The `SystemEventHook` function signature and `try_intercept_event` should be extended to carry the full emitter address. [10](#0-9) 

---

### Proof of Concept

1. Compute a CREATE2 salt `s` such that `keccak256(0xff ++ deployer ++ s ++ keccak256(bytecode))[12:]` ends in the 4-byte suffix matching `SYSTEM_CONTEXT_ADDRESS_LOW` (e.g., `0x0000800b`). This takes ~2^32 iterations.
2. Deploy the following contract at that address:
```solidity
contract FakeSystemContext {
    event SettlementLayerChainIdUpdated(uint256 newChainId);
    function attack(uint256 fakeChainId) external {
        emit SettlementLayerChainIdUpdated(fakeChainId);
    }
}
```
3. Call `attack(999)`. The EVM emits the event from an address whose lower 32 bits match `SYSTEM_CONTEXT_ADDRESS_LOW`. `try_intercept_event` finds the registered hook and dispatches to `system_context_event_hook`. The hook sees `topics[0] == SL_CHAIN_ID_UPDATED_EVENT_SIG` and calls `update_settlement_layer_chain_id(999)`.
4. The settlement layer chain ID is now `999`, corrupting all subsequent cross-chain operations.

The same attack applies to `interop_root_reporter_event_hook` using `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW` and a crafted `InteropRootAdded` event to inject a fake interop root.

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())
