### Title
Fee-on-Transfer ERC-20 Deposit Causes ckERC20 Over-Minting, Breaking 1:1 Backing Invariant — (`File: rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`)

---

### Summary

Both ckERC20 deposit helper contracts emit their deposit event using the caller-supplied `amount` argument rather than the actual tokens received by the minter address. The IC minter canister blindly mints ckERC20 tokens equal to the event's `amount` field. For any fee-on-transfer ERC-20 token in the supported list, the minter will mint more ckERC20 than it holds in actual ERC-20 collateral, permanently breaking the 1:1 backing invariant and enabling a drain of the minter's reserves.

---

### Finding Description

**Root cause — `ERC20DepositHelper.sol` (`CkErc20Deposit.deposit`):**

```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal); // ← emits input `amount`, not actual received
}
``` [1](#0-0) 

**Root cause — `DepositHelperWithSubaccount.sol` (`CkDeposit.depositErc20`):**

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount); // ← same flaw
}
``` [2](#0-1) 

For a fee-on-transfer ERC-20 token (e.g., one that deducts 5% on every transfer), `safeTransferFrom(user, minter, 1000)` causes only 950 tokens to arrive at `minterAddress`, but the event records `amount = 1000`.

**IC minter log scraping — `parser.rs`:**

The `ReceivedErc20LogParser` and `ReceivedEthOrErc20LogParser` parse the event's data field directly into `Erc20Value` with no cross-check against any on-chain balance: [3](#0-2) 

**IC minter minting — `deposit.rs`:**

The minter calls `icrc1_transfer` with `amount: event.value()`, which is the value read verbatim from the event log: [4](#0-3) 

There is no balance-before/balance-after check anywhere in the deposit pipeline. The minter's internal `erc20_balances` accounting is also updated from the same event value: [5](#0-4) 

---

### Impact Explanation

**Ledger conservation break.** For every deposit of a fee-on-transfer token, the minter mints `amount` ckERC20 but only holds `amount - fee` actual ERC-20. The cumulative shortfall grows with each deposit. When users withdraw ckERC20, the minter issues Ethereum ERC-20 transfers for the full face value; once the shortfall exceeds the minter's reserve, later withdrawals fail. Early depositors/withdrawers can drain the entire ERC-20 reserve held by the minter, leaving all remaining ckERC20 holders with unbacked tokens they cannot redeem. This is a direct, permanent loss of user funds.

---

### Likelihood Explanation

**Medium.** The minter enforces a whitelist of supported ERC-20 tokens via `ckerc20_tokens`, populated only through `add_ckerc20_token` which is restricted to the NNS-controlled orchestrator canister: [6](#0-5) 

Two realistic trigger paths exist without requiring a malicious governance majority:

1. **Honest NNS proposal adds a fee-on-transfer token.** Many ERC-20 tokens implement transfer fees (e.g., reflection tokens, auto-liquidity tokens). An NNS proposal submitted in good faith could add such a token. The code has no structural guard to detect this.

2. **An already-supported token's Ethereum contract is upgraded to add fees.** Several supported tokens (e.g., ckPEPE — PEPE is a meme token whose contract owner could add a fee mechanism) are upgradeable. If the underlying ERC-20 contract adds a transfer fee after the ckERC20 token is already supported, every subsequent deposit immediately over-mints. No IC-side action is required by the attacker; any ordinary depositor triggers the accounting error.

---

### Recommendation

In both `CkErc20Deposit.deposit` and `CkDeposit.depositErc20`, measure the actual balance change and emit that value rather than the caller-supplied `amount`:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

This mirrors the fix recommended in the original `RenovaQuest.sol` report. Alternatively, the NNS governance process for adding new ckERC20 tokens should include an explicit check that the candidate ERC-20 contract does not implement fee-on-transfer behavior.

---

### Proof of Concept

1. A fee-on-transfer ERC-20 token `FeeToken` (10% fee on every transfer) is added as a supported ckERC20 token via an NNS proposal (or an existing supported token upgrades to add fees).
2. Attacker calls `FeeToken.approve(helperContract, 1000)` on Ethereum.
3. Attacker calls `depositErc20(FeeToken, 1000, principal, subaccount)` on `CkDeposit`.
   - `safeTransferFrom(attacker, minter, 1000)` executes; minter receives **900** tokens (100 taken as fee).
   - `emit ReceivedEthOrErc20(FeeToken, attacker, 1000, ...)` fires with `amount = 1000`.
4. IC minter scrapes the log: `ReceivedEthOrErc20LogParser::parse_log` reads `value = 1000` from event data. [7](#0-6) 
5. IC minter calls `icrc1_transfer(..., amount: 1000)` on the ckFeeToken ledger — mints **1000** ckFeeToken to attacker. [4](#0-3) 
6. Minter holds 900 actual `FeeToken` but has issued 1000 ckFeeToken. Shortfall = 100 per deposit.
7. Attacker calls `withdraw_erc20` for 1000 ckFeeToken. Minter burns 1000 ckFeeToken and sends 1000 `FeeToken` on Ethereum — but only has 900. The withdrawal either fails or drains reserves belonging to other users.
8. Repeated deposits amplify the shortfall until the minter's entire `FeeToken` reserve is exhausted, leaving all remaining ckFeeToken holders with unbacked, non-redeemable tokens.
