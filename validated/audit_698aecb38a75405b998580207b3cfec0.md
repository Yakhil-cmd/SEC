### Title
Unbounded `move_to_block` Loop in Hashchain Causes System-Wide Gas-Exhaustion DoS After Any Inactivity Gap - (File: `engine-hashchain/src/hashchain.rs`, `engine/src/hashchain.rs`)

---

### Summary

When the Aurora Engine's hashchain feature is active, every transaction (EVM submit, deploy, call, connector operations) must first call `load_hashchain`, which invokes `move_to_block` to catch up the stored hashchain to the current NEAR block height. `move_to_block` contains an unbounded `while` loop that computes one keccak hash per skipped block. After any period of inactivity (e.g., a maintenance pause), the gap between the stored block height and the current block height can grow large enough to exhaust NEAR's 300 TGas gas limit. Because the hashchain is only persisted on transaction success, a gas-exhaustion abort leaves the stored block height unchanged, making every subsequent transaction face the same fatal loop. The result is a permanent, self-reinforcing system-wide DoS that freezes all EVM transactions, deposits, and withdrawals until an admin manually restarts the hashchain.

---

### Finding Description

**Root cause — unbounded loop in `move_to_block`:**

`engine-hashchain/src/hashchain.rs`, lines 67–88:

```rust
pub fn move_to_block(&mut self, next_block_height: u64) -> Result<(), BlockchainHashchainError> {
    if next_block_height <= self.current_block_height {
        return Err(BlockchainHashchainError::BlockHeightIncorrect);
    }
    while self.current_block_height < next_block_height {   // ← unbounded
        self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
            &self.chain_id,
            self.contract_account_id.as_bytes(),
            self.current_block_height,
            self.previous_block_hashchain,
        );
        self.block_hashchain_computer.clear_txs();
        self.current_block_height += 1;
    }
    Ok(())
}
```

Each iteration calls `compute_block_hashchain`, which concatenates ~120 bytes and calls `keccak`. NEAR's gas limit is 300 TGas; a keccak256 over ~120 bytes in WASM costs on the order of 1–5 TGas, so a gap of a few hundred blocks is sufficient to exhaust the budget.

**Called unconditionally before every transaction:**

`engine/src/hashchain.rs`, lines 88–96:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;   // ← called before any business logic
    }
    Ok(maybe_hashchain)
}
```

`load_hashchain` is the first thing called inside both `with_hashchain` and `with_logs_hashchain` (lines 33 and 66 of the same file), which wrap every user-facing method: `submit`, `submit_with_args`, `deploy_code`, `call`, `register_relayer`, `withdraw_wnear_to_router`, and connector operations.

**Hashchain is only saved on success:**

`engine/src/hashchain.rs`, lines 37–49:

```rust
let result = f(hashchain_io)?;   // ← if this panics/aborts, nothing below runs

if let Some(mut hashchain) = maybe_hashchain {
    ...
    save_hashchain(&mut io, &hashchain)?;   // ← never reached on gas exhaustion
}
```

When NEAR aborts a transaction for gas exhaustion, all state changes are rolled back. The stored `current_block_height` remains at its old value, so the next transaction faces the identical loop.

**Trigger scenario:**

1. Admin enables the hashchain via `start_hashchain` (requires pausing the contract first).
2. The contract is paused for maintenance (upgrade, migration, etc.). NEAR blocks continue at ~1 per second.
3. After N blocks of pause, the contract is unpaused.
4. The first user transaction calls `load_hashchain` → `move_to_block(current + N)` → N keccak iterations → gas exhaustion → abort → hashchain not saved.
5. Every subsequent transaction repeats step 4. The system is permanently DoS'd.

Recovery requires the admin to call `start_hashchain` again with `args.block_height = current_block_height - 1` (bypassing the loop), but this requires noticing the failure and acting.

---

### Impact Explanation

All user-facing Aurora Engine operations — EVM transaction submission, contract deployment, contract calls, ETH deposits, ETH withdrawals — are gated behind `with_logs_hashchain` or `with_hashchain`. Once the gap exceeds the gas threshold, none of these operations can succeed. User funds locked in the EVM or in transit through the connector are inaccessible until an admin manually intervenes. This matches **High: Temporary freezing of funds**.

---

### Likelihood Explanation

The hashchain is a production feature of Aurora Engine. Any routine maintenance pause (upgrade, state migration) that lasts more than a few hundred NEAR blocks (~5 minutes at 1 block/second) is sufficient to trigger the condition. No attacker action is required; the DoS is a natural consequence of the system design. Once triggered, it is self-sustaining and not visible to ordinary users, making delayed admin response likely.

---

### Recommendation

1. **Cap the loop**: In `move_to_block`, return an error (or a special status) if `next_block_height - current_block_height` exceeds a safe threshold (e.g., 100 blocks). Expose a dedicated admin method to advance the hashchain in bounded increments.
2. **Decouple hashchain advancement from transaction execution**: Advance and persist the hashchain in a separate, dedicated NEAR call so that a large gap does not block user transactions.
3. **Save partial progress**: If the loop is interrupted, persist the intermediate `current_block_height` so the next call resumes from where it left off rather than restarting from the original gap.

---

### Proof of Concept

**Step 1.** Deploy Aurora Engine and call `start_hashchain` (requires pausing first). The hashchain is now stored with `current_block_height = B`.

**Step 2.** Pause the contract for maintenance. Wait N NEAR blocks (N ≈ 300, ~5 minutes). Unpause.

**Step 3.** Submit any EVM transaction via `submit`. Execution enters:

```
submit
  └─ with_logs_hashchain          [engine/src/hashchain.rs:65]
       └─ load_hashchain           [engine/src/hashchain.rs:88]
            └─ move_to_block(B+N)  [engine-hashchain/src/hashchain.rs:75]
                 └─ while loop: N × compute_block_hashchain (keccak)
                      → NEAR gas exhaustion → transaction aborted
```

**Step 4.** The stored hashchain still has `current_block_height = B`. Repeat step 3 with any transaction — same result. The engine is permanently DoS'd.

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}
