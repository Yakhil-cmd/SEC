### Title
Owner Can Permanently Renounce Control by Setting `owner_id` to Empty `AccountId` - (File: `engine/src/contract_methods/admin.rs`)

### Summary

The `set_owner` function in the Aurora Engine does not reject an empty string as the `new_owner`. The `BorshDeserialize` implementation for `AccountId` explicitly allows empty strings (returning `AccountId::default()`) for backward compatibility. If the current owner calls `set_owner` with a Borsh-encoded empty account ID, `state.owner_id` is permanently set to an empty `AccountId`. Because no valid NEAR account can have an empty ID, `require_owner_only` can never be satisfied again, permanently locking out every owner-gated privileged function — including `resume_contract`. If the contract is paused (before or after this action), all user funds are permanently frozen with no recovery path.

---

### Finding Description

**Root cause — `set_owner` accepts an empty `AccountId`:**

`set_owner` in `engine/src/contract_methods/admin.rs` reads the new owner via Borsh deserialization and applies only one guard: the new owner must differ from the current owner.

```rust
// engine/src/contract_methods/admin.rs lines 103-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← no further validation
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [1](#0-0) 

There is no check that `args.new_owner` is a non-empty, reachable NEAR account.

**`BorshDeserialize` for `AccountId` explicitly allows empty strings:**

```rust
// engine-types/src/account_id.rs lines 89-101
impl BorshDeserialize for AccountId {
    fn deserialize_reader<R: io::Read>(reader: &mut R) -> io::Result<Self> {
        let account: String = BorshDeserialize::deserialize_reader(reader)?;
        // It's for saving backward compatibility.
        if account.is_empty() {
            return Ok(Self::default());   // ← empty AccountId accepted
        }
        Self::try_from(account)
            .map_err(|e| io::Error::new(io::ErrorKind::InvalidData, e.to_string()))
    }
}
``` [2](#0-1) 

`AccountId::default()` is `AccountId(Box<str>)` wrapping an empty string, because `Default` for `Box<str>` is `""`. [3](#0-2) 

**`require_owner_only` can never be satisfied with an empty `owner_id`:**

```rust
// engine/src/contract_methods/mod.rs lines 79-87
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
``` [4](#0-3) 

NEAR enforces a minimum account ID length of 2 characters (`MIN_ACCOUNT_ID_LEN = 2`). No real NEAR account can have an empty ID, so no caller can ever match an empty `owner_id`. [5](#0-4) 

**All owner-gated functions become permanently inaccessible**, including:

| Function | Effect of lockout |
|---|---|
| `resume_contract` | Cannot unpause a paused engine |
| `pause_contract` | Cannot pause in emergencies |
| `upgrade` / `stage_upgrade` | Cannot deploy contract fixes |
| `resume_precompiles` | Cannot re-enable paused precompiles |
| `set_key_manager` | Cannot manage relayer keys |
| `attach_full_access_key` | Cannot recover via FAK |
| `mirror_erc20_token` | Cannot mirror new ERC-20 tokens | [6](#0-5) 

---

### Impact Explanation

**Impact: Permanent freezing of funds (Critical)**

The concrete worst-case path:

1. Owner calls `pause_contract` → `state.is_paused = true`. All EVM execution (`submit`, `call`, `deploy_code`) is blocked; user funds are locked inside Aurora.
2. Owner calls `set_owner` with a Borsh-encoded `SetOwnerArgs { new_owner: "" }` → `state.owner_id` is set to the empty `AccountId`.
3. `resume_contract` requires `require_owner_only`. No NEAR account can satisfy this. The contract is **permanently paused**.
4. All ETH and ERC-20 token balances held inside Aurora are permanently frozen with no recovery path — not even via contract upgrade, since `upgrade` also requires `require_owner_only`. [7](#0-6) 

Even without a prior `pause_contract`, the loss of `upgrade` capability means any future critical bug cannot be patched, and the loss of `resume_precompiles` means any paused precompile (e.g., `exitToNear`, `exitToEthereum`) stays paused permanently, blocking all bridge withdrawals.

---

### Likelihood Explanation

**Likelihood: Low (2/10)**

This requires the owner account to call `set_owner` with an empty string argument. This could occur through:
- An operator scripting error that passes an empty string instead of a valid account ID.
- An intentional renouncement (analogous to `renounceOwnership()` in Solidity) where the operator mistakenly believes this is a safe operation.

The NEAR account ID validation in `AccountId::new` / `AccountId::validate` would reject an empty string through the normal `TryFrom<String>` path, but the `BorshDeserialize` path bypasses this with the explicit empty-string exception. This inconsistency makes the mistake non-obvious to a developer constructing the call. [8](#0-7) 

---

### Recommendation

1. **Remove the empty-string bypass in `AccountId::BorshDeserialize`**, or at minimum add a non-empty check before accepting the deserialized value in `set_owner`:

```rust
let args: SetOwnerArgs = io.read_input_borsh()?;
if args.new_owner.as_bytes().is_empty() {
    return Err(b"ERR_INVALID_OWNER".into());
}
```

2. **Add a zero-address / empty-account guard** in `set_owner` that explicitly rejects any `new_owner` whose string representation is empty or otherwise unreachable.

3. **Consider a two-step ownership transfer** (propose + accept) so that the new owner must actively confirm, preventing accidental transfers to non-existent accounts.

---

### Proof of Concept

```rust
// Construct a SetOwnerArgs with an empty new_owner via Borsh
// AccountId::BorshDeserialize accepts empty strings (backward compat path)
let args = SetOwnerArgs {
    new_owner: AccountId::default(), // empty string ""
};
let encoded = borsh::to_vec(&args).unwrap();

// Step 1 (optional, for fund-freeze): pause the contract first
runner.call("pause_contract", &current_owner, vec![]).unwrap();

// Step 2: set owner to empty AccountId — succeeds because:
//   - caller IS the current owner (require_owner_only passes)
//   - new_owner != current_owner (ERR_SAME_OWNER check passes)
//   - BorshDeserialize accepts empty string, returning AccountId::default()
runner.call("set_owner", &current_owner, encoded).unwrap();

// Step 3: attempt to resume — fails permanently
let result = runner.call("resume_contract", &current_owner, vec![]);
assert!(result.is_err()); // ERR_NOT_ALLOWED — owner_id is now ""

// No NEAR account can ever satisfy require_owner_only again.
// If paused: all user funds are permanently frozen.
// upgrade(), stage_upgrade(), attach_full_access_key() all permanently inaccessible.
