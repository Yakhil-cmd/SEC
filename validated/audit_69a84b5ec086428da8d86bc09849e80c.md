### Title
`native_per_pubdata` Truncates to Zero via Integer Floor Division When `pubdata_price < native_price`, Allowing Free Pubdata Usage - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the block-level `pubdata_price` is less than `native_price`, the integer floor division used to compute `native_per_pubdata` truncates to `0`. This makes all pubdata free for every L2 transaction in that block: no native resources are charged for any amount of pubdata written, and the post-execution pubdata sufficiency check always passes. An unprivileged sender can exploit this to spam state writes up to the block's `pubdata_limit` without paying the native resource cost, leaving the operator/prover uncompensated.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` for ZK (L2) transactions, `native_per_pubdata` is derived as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142-143
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer floor division. Whenever `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `0`. [1](#0-0) 

By contrast, `native_per_gas` is computed with ceiling division to guarantee a non-zero result:

```rust
// validation_impl.rs:135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The same floor-division pattern is duplicated in the public API helper:

```rust
// api/src/helpers.rs:427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

Once `native_per_pubdata == 0`, three downstream accounting sites all silently produce zero cost:

**1. Intrinsic pubdata overhead in `create_resources_for_tx`:**
```rust
// gas_helpers.rs:352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0
``` [4](#0-3) 

**2. Execution pubdata charge in `get_resources_to_charge_for_pubdata`:**
```rust
// gas_helpers.rs:430-431
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // → 0 for any pubdata_spent
``` [5](#0-4) 

**3. Post-execution sufficiency check in `check_enough_resources_for_pubdata`:**
```rust
// gas_helpers.rs:454
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata is always 0 → always true
``` [6](#0-5) 

The post-execution check is called from both the ZK L2 transaction flow and the L1 transaction flow: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

When `native_per_pubdata == 0`:

- Every storage write (SSTORE) in the transaction generates pubdata bytes that are published on-chain, but the sender pays **zero native resources** for them.
- The operator/prover must prove and publish all pubdata but receives no native-resource compensation for it.
- An attacker can fill the entire block pubdata budget (`pubdata_limit`) with state writes at no extra native cost beyond the EVM gas for the SSTOREs themselves.
- This is a direct resource accounting bypass: the operator's proving cost for pubdata is unrecovered, constituting a financial loss to the operator and a potential pubdata-spam DoS vector against other transactions in the same block.

---

### Likelihood Explanation

The `BlockContext` default in the test rig already demonstrates the condition: `pubdata_price = 0`, `native_price = 10`. [9](#0-8) 

In production, `pubdata_price` and `native_price` are independent operator-supplied block parameters with no enforced ordering constraint. Any block where `pubdata_price < native_price` — a common ratio when pubdata is cheap relative to computation — silently sets `native_per_pubdata = 0`. A transaction sender who observes the block context (both values are public) can immediately exploit this by submitting a pubdata-heavy transaction.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed. This ensures that any non-zero `pubdata_price` maps to at least 1 native unit per pubdata byte:

```rust
// validation_impl.rs:142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Optionally, add a validation gate that rejects blocks (or treats pubdata as intentionally free) only when `pubdata_price == 0` explicitly, rather than silently truncating any sub-unit price to zero.

---

### Proof of Concept

**Setup**: Block context with `pubdata_price = 5`, `native_price = 10`.

**Step 1** — `native_per_pubdata` computation:
```
pubdata_price.wrapping_div(native_price) = 5 / 10 = 0  (integer floor)
native_per_pubdata = 0
```

**Step 2** — Attacker submits a transaction calling a contract that performs 100 SSTOREs (each writing a new storage slot, generating ~64 bytes of pubdata each = ~6,400 bytes total).

**Step 3** — `get_resources_to_charge_for_pubdata` is called post-execution:
```
native = 6400 * 0 = 0
resources_for_pubdata = 0
has_enough(0) = true  // always
```

**Step 4** — Transaction succeeds. The attacker paid EVM gas for the SSTOREs but **zero native resources** for 6,400 bytes of pubdata. The operator publishes and proves the pubdata at their own cost.

**Step 5** — Repeat with maximum pubdata per block (`pubdata_limit`). All pubdata in the block is consumed for free, blocking other transactions from writing state.
