### Title
Unguarded `deploy_erc20_token` + `register_token` Uniqueness Constraint Allows Adversary to Permanently Brick NEP-141 → ERC-20 Bridge Mapping - (`engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` is a public NEAR entrypoint with **no caller access control**. It deploys a standard ERC-20 and calls `register_token`, which enforces a strict one-ERC-20-per-NEP-141 uniqueness constraint and permanently rejects any subsequent registration attempt for the same NEP-141 account. An adversary can front-run the legitimate deployer for any NEP-141 token, permanently seizing the NEP-141 → ERC-20 mapping with an ERC-20 deployed at an address of the adversary's choosing and with default (zeroed) metadata. The legitimate deployer can never overwrite this mapping.

---

### Finding Description

**Root cause — missing access control:**

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` checks only `require_running`; there is no `require_owner_only` or any other caller restriction: [1](#0-0) 

The comment on line 148–149 acknowledges the intent ("this transaction could be executed by the owner of the contract only") but the enforcement is absent.

**Root cause — uniqueness constraint:**

`register_token` in `engine/src/engine.rs` enforces a hard uniqueness constraint: if a NEP-141 account ID is already present in the bijection map, it returns `TokenAlreadyRegistered` and the entire `deploy_erc20_token` call reverts: [2](#0-1) 

There is no admin override or update path for this mapping anywhere in the codebase.

**Attack flow:**

1. Adversary calls `deploy_erc20_token` (Legacy variant) with `nep141 = "usdc.near"` before the legitimate deployer.
2. The engine deploys the standard `EvmErc20` bytecode at an address determined by the adversary's NEAR-account-derived EVM origin and its current nonce.
3. `register_token` writes `usdc.near → <adversary_chosen_address>` into the `Nep141Erc20Map` / `Erc20Nep141Map` bijection.
4. Every subsequent call to `deploy_erc20_token` for `usdc.near` hits `Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered)` and reverts.
5. The mapping is now permanently locked to the adversary's address. The `WithMetadata` path (which fetches real token metadata via a cross-contract call) is equally blocked. [3](#0-2) 

**Why the ERC-20 address matters:**

`deploy_erc20_token` uses `CreateScheme::Legacy` (nonce-based), so the deployed address depends on the caller's EVM origin and nonce: [4](#0-3) 

The adversary controls their NEAR account ID (and thus their EVM origin) and can choose their nonce, giving them full control over the resulting ERC-20 address.

**Metadata is permanently wrong:**

The Legacy path passes `metadata = None`, so `setup_deploy_erc20_input` uses `Erc20Metadata::default()` — empty name, empty symbol, `decimals = 0`: [5](#0-4) 

While `set_erc20_metadata` can update metadata fields, it cannot change the registered ERC-20 address, and the bijection map cannot be updated.

---

### Impact Explanation

**High — Permanent freezing of bridge functionality for targeted NEP-141 tokens.**

Once the adversary seizes the mapping, the NEP-141 token is permanently bound to an ERC-20 at an unexpected address with zeroed metadata (`decimals = 0`). Any DeFi protocol, DEX, or wallet that queries `get_erc20_from_nep141` will receive the adversary's address. Protocols that have pre-approved or hardcoded the expected ERC-20 address will be unable to interact with bridged tokens. The `decimals = 0` metadata causes all amount calculations in downstream protocols to be off by up to 10^18, effectively making bridged balances unusable in any protocol that reads decimals. The legitimate deployer has no recovery path — the mapping is immutable once set.

---

### Likelihood Explanation

**High.** The entrypoint is fully public on NEAR mainnet. The adversary needs only to:
- Know the NEP-141 account ID of a token about to be bridged (observable from NEAR chain or public announcements).
- Submit a NEAR transaction to `deploy_erc20_token` before the legitimate deployer.

No special privileges, funds beyond gas, or complex setup are required. The attack is a simple front-run.

---

### Recommendation

Add `require_owner_only` (or an equivalent allowlist check) to `deploy_erc20_token` in `engine/src/contract_methods/connector.rs`, mirroring the pattern already used in `mirror_erc20_token`:

```rust
// mirror_erc20_token already does this correctly:
require_owner_only(&state, &io.predecessor_account_id())?;
```

Additionally, consider adding an admin-only override path in `register_token` to allow correcting a poisoned mapping.

---

### Proof of Concept

```
1. Legitimate deployer announces intent to bridge "usdc.near" to Aurora.

2. Adversary submits NEAR tx:
     aurora.deploy_erc20_token(DeployErc20TokenArgs::Legacy("usdc.near"))
   from account "attacker.near".

3. Engine executes:
   - origin = near_account_to_evm_address(b"attacker.near")  // e.g. 0xDEAD...
   - deploys EvmErc20 at address 0xBAD0... (attacker's nonce 0)
   - register_token(0xBAD0..., "usdc.near") → writes mapping

4. Legitimate deployer submits:
     aurora.deploy_erc20_token(DeployErc20TokenArgs::WithMetadata("usdc.near"))

5. Callback calls engine::deploy_erc20_token → register_token →
     get_erc20_from_nep141("usdc.near") returns Ok(0xBAD0...)
     → returns Err(TokenAlreadyRegistered)
   → entire tx reverts.

6. "usdc.near" is now permanently mapped to 0xBAD0... with decimals=0.
   All ft_on_transfer calls for "usdc.near" mint tokens on 0xBAD0...,
   which has decimals=0, breaking every downstream protocol.
``` [6](#0-5) [7](#0-6) [8](#0-7)

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })

/// Used to bridge NEP-141 tokens from NEAR to Aurora. On Aurora the NEP-141 becomes an ERC-20.
pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
