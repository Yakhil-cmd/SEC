### Title
Pausing Engine During In-Flight `exit_to_near_precompile_callback` Causes Permanent Loss of User Funds - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The Aurora Engine's `pause_contract` mechanism places a `require_running` guard inside `exit_to_near_precompile_callback`, a NEAR promise callback that is the second step of a two-step ERC-20/ETH exit flow. If the engine is paused between the first step (burning ERC-20 tokens in the EVM) and the second step (the callback that either transfers NEAR to the user or refunds burned tokens on failure), the callback panics with `ERR_PAUSED`. The user's ERC-20 tokens are already burned and irrecoverable, while the corresponding NEAR value is either stuck in Aurora's account (wNEAR unwrap case) or the NEP-141 transfer failed without a refund (ERC-20 exit failure case). No recovery path exists.

---

### Finding Description

The exit-to-NEAR flow is a two-step asynchronous process:

**Step 1** — User calls the `ExitToNear` precompile from within the EVM. The precompile burns the user's ERC-20 tokens (or deducts ETH) and emits a log that schedules a NEAR promise. For the wNEAR unwrap path, the promise calls `near_withdraw` on the wNEAR contract; for a regular ERC-20 exit, it calls `ft_transfer` on the NEP-141 contract.

**Step 2** — The NEAR runtime executes the base promise and then calls `exit_to_near_precompile_callback` on the engine as a callback. This callback either:
- Transfers the unwrapped NEAR to the user (wNEAR path, success), or
- Re-mints the burned ERC-20 tokens as a refund (failure path, when `error_refund` feature is enabled).

The vulnerability is in `exit_to_near_precompile_callback`:

### Impact Explanation

**Critical — Permanent freezing of funds.**

**wNEAR unwrap path (always active):**
- User's wNEAR ERC-20 tokens are burned in the EVM (irreversible).
- `near_withdraw` executes on the wNEAR contract; NEAR is unwrapped and credited to Aurora's account.
- Callback fails → NEAR is never forwarded to the user.
- The NEAR is stranded in Aurora's account with no on-chain recovery path for the user.

**ERC-20 exit failure path (active when `error_refund` feature is enabled):**
- User's ERC-20 tokens are burned in the EVM.
- `ft_transfer` on the NEP-141 contract fails (e.g., recipient not registered).
- Callback fails → `refund_on_error` never re-mints the ERC-20 tokens.
- User permanently loses their ERC-20 tokens.

In both cases the user suffers an irreversible loss of funds with no self-service recovery mechanism.

---

### Likelihood Explanation

**Medium.** The engine owner can call `pause_contract` at any time for a legitimate upgrade. The NEAR protocol processes receipts asynchronously across blocks; a pause issued in block N will affect callbacks scheduled in block N-1 or earlier that have not yet been processed. During any upgrade window — which the CHANGES.md shows occurs regularly — any user who initiated a wNEAR unwrap or ERC-20 exit in the preceding blocks is at risk. No malicious intent is required from the admin; the vulnerability is structural. [6](#0-5) 

---

### Recommendation

Remove `require_running` from `exit_to_near_precompile_callback` and `withdraw_wnear_to_router`. These are private self-callbacks (`env.assert_private_call()` is already enforced) that complete a user-initiated flow whose first step has already committed irreversible state changes. Pausing the engine should not be able to strand funds that are mid-flight in a promise chain. Alternatively, implement a queuing mechanism so that failed callbacks due to `ERR_PAUSED` can be retried after the engine resumes.

---

### Proof of Concept

**Scenario: wNEAR unwrap blocked by pause**

1. User holds wNEAR ERC-20 tokens on Aurora and calls the `ExitToNear` precompile with the `UnwrapWnear` message.
2. The EVM burns the user's wNEAR ERC-20 tokens (irreversible).
3. The engine schedules a NEAR promise: `near_withdraw` on the wNEAR contract, with `exit_to_near_precompile_callback` as the callback.
4. Before the callback receipt is processed, the engine owner calls `pause_contract` (e.g., to stage an upgrade).
5. `near_withdraw` executes successfully; NEAR is unwrapped and credited to Aurora's account.
6. The NEAR runtime invokes `exit_to_near_precompile_callback` on the engine.
7. `require_running` returns `Err(ERR_PAUSED)` → `sdk_unwrap()` panics → callback receipt fails.
8. The NEAR that was unwrapped remains in Aurora's account. The user's wNEAR ERC-20 tokens are permanently burned. The user has lost their funds with no on-chain recourse. [7](#0-6) [8](#0-7)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
