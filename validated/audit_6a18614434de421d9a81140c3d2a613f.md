### Title
Missing Storage Collision Check in CREATE/CREATE2 Allows Deployment to Addresses with Non-Empty Storage (EIP-7610 Non-Compliance) - (File: `evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter does not check whether a target address has non-empty storage before allowing a CREATE or CREATE2 deployment to proceed. The EVM specification (EIP-7610) requires that contract creation must revert if the target address already has non-empty storage. ZKsync OS explicitly skips this check, as acknowledged in the source code and confirmed by multiple disabled test cases. This is a direct analog to the reported vulnerability class: a "singleton" state (contract storage at a deterministic address) can be "re-initialized" by a new deployment, causing the new contract to silently inherit attacker-controlled storage values.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` function performs the collision check for constructor calls:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // ... burn gas and return CreateCollision
}
``` [1](#0-0) 

The check only guards against `code_len != 0 || nonce != 0`. The storage emptiness check required by EIP-7610 is explicitly omitted with the comment "we cannot perform such a check for now."

This is confirmed by multiple disabled test cases across all test index files:

- `create2collisionStorageParis.json` — disabled: `"We do not check storage for collision"`
- `RevertInCreateInInitCreate2Paris.json` — disabled: `"We do not check for storage collisions"`
- `eip7610_create_collision / test_init_collision_create_opcode.json` — disabled: `"We do not check the storage"`
- `eip7610_create_collision / test_init_collision_create_tx.json` — disabled: `"We do not check the storage"` [2](#0-1) [3](#0-2) 

The `deploy_code` function in the flat storage model also carries a comment acknowledging the caller's responsibility to verify the address is usable, but no storage check is enforced:

### Impact Explanation

**Vulnerability class:** EVM semantic mismatch / state transition bug.

When a contract is selfdestructed in a pre-Cancun context (or in the same transaction as deployment under EIP-6780), its code and nonce are cleared but its storage slots persist in the state. ZKsync OS's collision check passes (`code_len == 0` and `nonce == 0`), so CREATE2 to the same address succeeds. The newly deployed contract silently inherits the attacker-pre-seeded storage.

**Concrete impact:**
1. A new contract deployed to address X inherits storage written by a prior contract at X.
2. If the new contract uses storage slot 0 as an "initialized" flag or "owner" field, it reads the attacker's pre-seeded value instead of the expected zero.
3. Initialization logic is bypassed, or ownership is hijacked — leading to direct fund loss or protocol manipulation.
4. The ZKsync OS state transition function diverges from Ethereum's expected output for the same transaction sequence, breaking EVM equivalence guarantees.

---

### Likelihood Explanation

**Attack path (pre-Cancun hardfork context or EIP-6780 same-transaction selfdestruct):**

1. Attacker deploys contract A to address X via CREATE2 (attacker controls salt and initcode).
2. Contract A writes attacker-controlled values to storage (e.g., slot 0 = attacker address as "owner").
3. Contract A selfdestructs — code and nonce are cleared; storage persists in the ZKsync OS state.
4. Victim (or attacker) deploys contract B to address X via CREATE2 with the same salt and initcode hash.
5. ZKsync OS's collision check passes (`code_len == 0`, `nonce == 0`); deployment succeeds.
6. Contract B's constructor runs against inherited storage — the "owner" slot already contains the attacker's address.
7. Any subsequent call to contract B that relies on the owner check is controlled by the attacker.

The attacker-controlled entry point is a standard unprivileged EVM transaction. No privileged role is required. The only precondition is that the attacker can predict or control the CREATE2 address (trivially achievable by controlling the salt).

---

### Recommendation

Implement the EIP-7610 storage emptiness check in `before_executing_frame` (and symmetrically in `before_reading_callee` / `constructor_pre_checks`). Before allowing a constructor frame to execute, verify that the target address has no non-zero storage slots. If any non-zero storage slot exists, the deployment must revert with `CreateCollision` and burn all gas, consistent with EIP-7610 semantics. [1](#0-0)
