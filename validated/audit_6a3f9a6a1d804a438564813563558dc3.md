### Title
Oversized `gas_limit` (> `u64::MAX`) causes permanent fund freeze via charge-then-overflow ordering — (`engine/src/engine.rs`)

### Summary
In `submit_with_alt_modexp`, `charge_gas` is called before `transaction.gas_limit` is narrowed to `u64`. A legacy transaction with `gas_limit > u64::MAX` passes all pre-charge validation, has its full gas cost deducted from the sender's balance, then causes an immediate `GasOverflow` return before `refund_unused_gas` is ever reached. The sender's ETH is permanently lost.

### Finding Description
`NormalizedEthTransaction.gas_limit` is stored as `U256`. [5](#0-4) 

The legacy transaction parser copies `gas_limit` verbatim from the RLP-decoded U256 field with no upper-bound check. [6](#0-5) 

Inside `submit_with_alt_modexp`, the ordering is:

1. Intrinsic gas check — compares `gas_limit (U256)` against `intrinsic_gas (u64 cast to U256)`, so any value above 21 000 passes.
2. `charge_gas` — multiplies `gas_limit (U256) × effective_gas_price (U256)`, deducts from sender balance, writes new balance to storage.
3. `gas_limit.try_into::<u64>()` — fails for any value > `u64::MAX`, returns `GasOverflow` immediately.
4. `refund_unused_gas` — unreachable.

The deducted funds are gone with no recovery path.

### Impact Explanation
**Critical — Permanent freezing of funds.**

An attacker who controls a sender account can self-inflict this to permanently destroy their own funds, or can be tricked into it. More critically, any user who accidentally or deliberately submits a transaction with `gas_limit > u64::MAX` (which is valid RLP and passes signature verification) loses their entire prepaid gas permanently. The ETH is not burned to a fee recipient; it simply disappears from the sender's balance with no corresponding credit anywhere.

### Likelihood Explanation
The attack requires the sender to hold at least `gas_limit × gas_price` Wei and to submit a transaction with `gas_limit > u64::MAX`. Constructing such a transaction is trivial — it is valid RLP and passes ECDSA recovery. A malicious relayer could craft and relay such a transaction on behalf of a victim if the victim's signed transaction bytes are intercepted or if the relayer constructs the transaction. The condition is fully attacker-controlled.

### Recommendation
Reject transactions with `gas_limit > u64::MAX` **before** calling `charge_gas`. Add an explicit guard immediately after parsing:

```rust
let gas_limit: u64 = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;
```

Place this check before line 1101 (before `charge_gas`), and pass the already-narrowed `u64` value to the EVM executor. This ensures the overflow is detected before any balance mutation occurs.

### Proof of Concept

```
sender_balance = u64::MAX + 1  (= 2^64 Wei, ~18.4 ETH — a realistic balance)
gas_limit      = u64::MAX + 1  (encoded as U256 in RLP)
gas_price      = 1 Wei
```

Execution:
1. `intrinsic_gas` = 21 000 (simple ETH transfer). `2^64 >= 21 000` → check passes.
2. `charge_gas`: `prepaid = (2^64) * 1 = 2^64 Wei`. `new_balance = 2^64 - 2^64 = 0`. `set_balance(sender, 0)` committed to storage.
3. `gas_limit.try_into::<u64>()`: `2^64` overflows `u64::MAX` → `Err(GasOverflow)` returned.
4. `refund_unused_gas` never called.
5. Assert: `get_balance(sender) == 0`. No refund. Funds permanently frozen.
