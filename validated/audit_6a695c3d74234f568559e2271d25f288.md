Audit Report

## Title
Unbounded `allowed_when_resources_are_low` Proposal Bypass Enables NNS Governance Heap Exhaustion - (File: `rs/nns/governance/src/governance.rs`)

## Summary
The `make_proposal` function in the NNS governance canister unconditionally skips both the heap-growth guard and the `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` (200) cap for any proposal whose action returns `true` from `allowed_when_resources_are_low()`. No separate cap exists for these "emergency" proposals. An unprivileged actor holding a neuron with ≥ 6-month dissolve delay and sufficient ICP stake can submit an unbounded stream of `InstallCode` proposals targeting protocol canisters, growing the governance canister's heap without limit until it becomes unresponsive and all NNS governance operations halt.

## Finding Description

**Gate 1 — heap growth check unconditionally skipped:**

At `rs/nns/governance/src/governance.rs` lines 5143–5145, the heap check is entirely omitted for exempt proposals:

```rust
if !action.allowed_when_resources_are_low() {
    self.check_heap_can_grow()?;
}
```

**Gate 2 — 200-proposal cap bypassed:**

At lines 5254–5269, the `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS` guard short-circuits on the same predicate:

```rust
if self.heap_data.proposals.values()
    .filter(|info| !info.ballots.is_empty() && !info.is_manage_neuron())
    .count()
    >= MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS
    && !action.allowed_when_resources_are_low()   // ← bypass
{
    return Err(...)
}
```

**Which proposals qualify:**

`InstallCode::allowed_when_resources_are_low()` (`rs/nns/governance/src/proposals/install_code.rs` lines 146–151) returns `true` for any canister whose topic resolves to `Topic::ProtocolCanisterManagement` (Governance, Root, Lifeline, Registry, Ledger, CMC, etc.). The same bypass is available for `UpdateCanisterSettings` and certain `ExecuteNnsFunction` variants via `ValidProposalAction::allowed_when_resources_are_low()` (`rs/nns/governance/src/proposals/mod.rs` lines 195–208).

**Fee mechanism does not prevent the attack:**

The submission fee is charged as `neuron.neuron_fees_e8s += proposal_submission_fee` (lines 5356–5358). The pre-submission check at line 5206 only requires `proposer_minted_stake_e8s >= proposal_submission_fee`. With 500 ICP staked and a 1 ICP fee per proposal, the attacker can submit ~500 proposals before the neuron's effective stake drops below the fee threshold. No per-neuron open-proposal count is enforced.

**Heap constants:**

`MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS = 200` (line 252) and `HEAP_SIZE_SOFT_LIMIT_IN_WASM32_PAGES` = 3.5 GiB (lines 260–268). Each exempt proposal allocates one ballot per eligible neuron (up to `MAX_NUMBER_OF_NEURONS = 500,000`), consuming ~10 MB of heap per proposal.

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

At ~350–450 proposals the governance canister's heap reaches the 3.5 GiB soft limit. Further state-modifying calls (voting, new proposals, reward distribution, neuron management) trap with an out-of-memory error, freezing all NNS governance operations. This is a complete, persistent governance freeze of the Internet Computer's root nervous system — not a transient disruption.

## Likelihood Explanation

No privileged role is required. The attacker needs:
- A neuron with ≥ 6-month dissolve delay (publicly available)
- ~350–450 ICP staked (non-trivial but realistic for a motivated adversary; current ICP price makes this a few thousand USD)
- Repeated `manage_neuron` calls submitting `InstallCode` proposals targeting any protocol canister

No governance majority, admin key, social engineering, or threshold corruption is needed. The economic barrier limits casual abuse but does not prevent a well-funded attacker. **Likelihood: Low-Medium.**

## Recommendation

1. **Add a separate cap** for `allowed_when_resources_are_low` proposals (e.g., `MAX_NUMBER_OF_EMERGENCY_PROPOSALS_WITH_BALLOTS = 10`). These are intended for rare recovery scenarios, not bulk submission.
2. **Apply the heap-growth check to all proposals**, using a higher threshold for emergency proposals (e.g., block at 95% utilization rather than the current soft limit, while still blocking at 100%).
3. **Add a per-neuron open-proposal limit** across all proposal types to prevent a single actor from monopolising the proposal queue.
4. Consider **increasing `reject_cost_e8s`** for `ProtocolCanisterManagement` proposals, since their ballot maps are the largest.

## Proof of Concept

```
1. Create neuron N:
     dissolve_delay = 6 months
     cached_neuron_stake_e8s = 500 * E8  (500 ICP)

2. Loop i = 1..500:
     submit manage_neuron {
       command: MakeProposal {
         action: InstallCode {
           canister_id: GOVERNANCE_CANISTER_ID,
           install_mode: Upgrade,
           wasm_module: [0x00, 0x61, 0x73, 0x6d, ...],  // minimal valid WASM
           wasm_module_hash: sha256(wasm_module),
         }
       }
     }

   Each call succeeds because:
     - allowed_when_resources_are_low() == true → heap check skipped (line 5143)
     - allowed_when_resources_are_low() == true → 200-proposal cap skipped (line 5261)
     - minted_stake_e8s (500 ICP - i ICP fees) >= reject_cost_e8s (1 ICP)

3. After ~350–450 iterations the governance canister heap is exhausted.
   Subsequent state-modifying calls trap with out-of-memory,
   freezing all NNS governance operations.

Reproducible as a PocketIC integration test: spin up a local NNS,
create the neuron, run the loop, assert that a subsequent vote call
returns a trap/out-of-memory error.
```
