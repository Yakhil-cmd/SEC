Audit Report

## Title
`PER_REQUEST_MINTER_FEE_BOUND` 1000× Underestimate Causes Unreimbursed `AmountTooLow` Permanent Fund Loss — (`rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

## Summary

`fee_based_minimum_withdrawal_amount` uses `PER_REQUEST_MINTER_FEE_BOUND = 326_000` koinu, but `evaluate_minter_fee(2, 2)` actually returns `326_000_000` koinu — a 1000× underestimate. This makes the computed minimum withdrawal amount (50 DOGE) insufficient when the minter's UTXO set is fragmented and many inputs are required. When the real minter fee exceeds the withdrawal amount, `BuildTxError::AmountTooLow` is returned, which finalizes the request with `FinalizedStatus::AmountTooLow` and **no reimbursement**, permanently burning the user's ckDOGE.

## Finding Description

**Root cause — `PER_REQUEST_MINTER_FEE_BOUND` is 1000× too small:** [1](#0-0) 

`evaluate_minter_fee(2, 2)` = `146_000_000 × 2 + 4_000_000 × 2 + 26_000_000` = **326,000,000 koinu**. The bound used in `fee_based_minimum_withdrawal_amount` is: [2](#0-1) 

`326_000` koinu — three orders of magnitude smaller. Because `min_withdrawal_amount_increment` = 25 DOGE = 2,500,000,000 koinu, the entire numerator (`PER_REQUEST_RBF_BOUND + fee_ceil(374) + PER_REQUEST_MINTER_FEE_BOUND` ≈ 700,374 koinu at normal rates) is less than one increment, so integer division yields 0 and the function always returns exactly `retrieve_doge_min_amount` = 50 DOGE regardless of fee rate. This is confirmed by the existing test: [3](#0-2) 

**Actual fee check uses real input count:** [4](#0-3) 

For 35 inputs: `evaluate_minter_fee(35, 2)` = `146_000_000 × 35 + 4_000_000 × 2 + 26_000_000` = **5,144,000,000 koinu = 51.44 DOGE**. Since `fee + minter_fee > amount` (51.44 DOGE > 50 DOGE), `BuildTxError::AmountTooLow` is returned.

**`TooManyInputs` check precedes `AmountTooLow` check, but the gap is large:** [5](#0-4) 

`DOGECOIN_MAX_NUM_INPUTS_IN_TRANSACTION = 500`: [6](#0-5) 

So any input count between ~35 and 500 hits `AmountTooLow` instead of `TooManyInputs`.

**`AmountTooLow` path has no reimbursement:** [7](#0-6) 

**`InvalidTransaction(TooManyInputs)` path does reimburse:** [8](#0-7) 

The test utility explicitly documents this asymmetry — reimbursement only occurs for `TooManyInputs`: [9](#0-8) 

The ckDOGE minter delegates all transaction processing to the shared ckBTC timer: [10](#0-9) 

## Impact Explanation

This is a **High** severity issue matching: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

A user withdrawing at the advertised minimum (50 DOGE) when the minter holds a fragmented UTXO set has their ckDOGE permanently burned with no reimbursement. The `RETRIEVE_DOGE_MIN_AMOUNT = 50 * DOGE` is confirmed as the production minimum: [11](#0-10) 

The `Reimbursed`/`WillReimburse` statuses are never reached via the `AmountTooLow` finalization path, making the fund loss permanent and unrecoverable.

## Likelihood Explanation

UTXO fragmentation occurs naturally when many users deposit small amounts (minimum deposit = 1 DOGE). An attacker can deliberately fragment the minter's UTXO set by depositing 35+ UTXOs at `deposit_doge_min_amount` (1 DOGE each), then submitting or inducing a victim to submit a withdrawal at the 50 DOGE minimum. The attacker can observe the minter's UTXO set via the public `get_known_utxos` query endpoint: [12](#0-11) 

The attack cost to the attacker is ~35 DOGE in deposits (which remain in the minter's UTXO set) plus the 50 DOGE withdrawal loss if they are the victim. The attack is repeatable and can target any user who withdraws at the minimum while the UTXO set is fragmented.

## Recommendation

1. **Fix `PER_REQUEST_MINTER_FEE_BOUND`**: Change `326_000` to `326_000_000` koinu in `fee_based_minimum_withdrawal_amount` to match the actual 2-input minter fee returned by `evaluate_minter_fee(2, 2)`.
2. **Use a conservative input-count upper bound**: Compute the worst-case minter fee based on `retrieve_doge_min_amount / deposit_doge_min_amount` (e.g., 50 inputs) rather than hardcoding 2 inputs.
3. **Add reimbursement for `AmountTooLow`**: Mirror the `InvalidTransaction` path — when `AmountTooLow` is triggered post-acceptance (i.e., the request passed the upfront minimum check), reimburse the user minus a processing fee.
4. **Add invariant/proptest**: Assert that for any UTXO set where `estimate_retrieve_doge_fee` succeeds for `fee_based_minimum_withdrawal_amount`, `build_unsigned_transaction` never returns `AmountTooLow`.

## Proof of Concept

```
1. Attacker deposits 50 UTXOs × 1.01 DOGE each into the ckDOGE minter.
2. Minter's available_utxos now contains 50 small UTXOs (total ~50.5 DOGE).
3. fee_based_minimum_withdrawal_amount() returns 50 DOGE (due to 1000× underestimate).
4. Victim calls retrieve_doge_with_approval(50 DOGE) — accepted (amount >= minimum).
5. Greedy UTXO selection picks all 50 UTXOs to cover 50 DOGE.
6. evaluate_minter_fee(50, 2) = 146_000_000 * 50 + 4_000_000 * 2 + 26_000_000
                              = 7,334,000,000 koinu = 73.34 DOGE.
7. fee + minter_fee (73.34+ DOGE) > amount (50 DOGE) → BuildTxError::AmountTooLow.
8. remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow) — no reimbursement.
9. Victim's 50 DOGE ckDOGE is permanently burned.

Minimal triggering case (35 UTXOs × 1.43 DOGE):
  evaluate_minter_fee(35, 2) = 5,144,000,000 koinu = 51.44 DOGE > 50 DOGE minimum.
```
