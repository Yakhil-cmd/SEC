### Title
Relayer NEAR Gas Costs Not Compensated in EVM Fee Accounting - (`engine/src/engine.rs`)

### Summary

The Aurora Engine's fee accounting in `refund_unused_gas` compensates relayers exclusively based on EVM gas metrics (`gas_used * priority_fee_per_gas` in ETH/Wei). However, relayers must pay NEAR gas (in yoctoNEAR) to submit the underlying NEAR transaction that calls `submit`. Because NEAR gas costs and EVM gas costs are explicitly documented as poorly correlated, relayers can suffer systematic losses, creating a disincentive to relay transactions and causing user transactions to be temporarily frozen.

### Finding Description

When a relayer submits an EVM transaction to Aurora, the flow is:

1. Relayer calls the `submit` NEAR method, paying NEAR gas out-of-pocket.
2. `engine::submit` calls `charge_gas` to deduct EVM gas prepayment from the EVM sender.
3. After EVM execution, `refund_unused_gas` pays the relayer `gas_used * priority_fee_per_gas` in ETH (EVM Wei), credited to the relayer's EVM mirror address.

The relayer's compensation is computed entirely in EVM terms:

```rust
// engine/src/engine.rs
let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;
// ...
if !relayer_reward.is_zero() {
    add_balance(io, relayer, relayer_reward)?;
}
```

There is no component in `charge_gas` or `refund_unused_gas` that accounts for the NEAR gas the relayer paid to the NEAR runtime. The relayer pays NEAR (a separate token, separate pricing system) and receives ETH (EVM Wei), with no mechanism to bridge the two cost dimensions.

The Aurora Engine's own benchmarks documentation explicitly states:

> "So far we have seen that this correlation [between NEAR gas and EVM gas] is not very strong, indicating that operations are costed very differently between our platform and Ethereum."

This is the direct analog of the Perennial H-6 finding: just as Optimism L2 keepers paid L1 calldata fees that were not reflected in the L2 gas compensation, Aurora relayers pay NEAR gas fees that are not reflected in the EVM priority-fee compensation.

### Impact Explanation

If the ETH priority fee received (converted to NEAR at market rates) is less than the NEAR gas cost paid, relayers operate at a loss. Rational relayers will stop submitting transactions. User EVM transactions submitted to Aurora will not be processed, resulting in temporary freezing of user funds and inability to exit positions or withdraw assets. This maps to **High: Temporary freezing of funds**.

### Likelihood Explanation

The NEAR gas price and ETH/NEAR exchange rate fluctuate independently. The benchmarks doc confirms the weak correlation between NEAR gas and EVM gas. For complex EVM transactions (e.g., DeFi interactions consuming 100–165 NEAR Tgas as shown in the benchmarks), the NEAR gas cost can be substantial while the EVM gas used may be modest. Any period where NEAR gas prices spike or the ETH/NEAR rate moves unfavorably makes relaying unprofitable. This condition is not rare — it is a structural property of two independent pricing systems.

### Recommendation

Introduce a NEAR gas cost component into the relayer compensation. The engine has access to `env.prepaid_gas()` and `env.used_gas()` (via the `Env` trait and the `prepaid_gas` precompile). The relayer reward should include a NEAR-gas-denominated component converted to ETH at a configurable exchange rate, or the protocol should maintain a separate NEAR-denominated relayer subsidy pool. Reference: Arbitrum's L1 pricing model and Optimism's L1 data fee mechanism.

### Proof of Concept

**Entry path:**

1. Relayer calls `submit` on the Aurora NEAR contract, paying e.g. 100 Tgas of NEAR gas.
2. `evm_transactions::submit` → `engine::submit` is called.
3. `charge_gas` deducts `gas_limit * effective_gas_price` in ETH from the EVM sender.
4. EVM execution runs; suppose `gas_used = 50_000` EVM gas, `priority_fee_per_gas = 1 Gwei`.
5. `refund_unused_gas` credits the relayer `50_000 * 1e9 = 50_000 Gwei = 0.00005 ETH`.
6. The relayer paid 100 Tgas of NEAR gas. At NEAR gas price of 100 Mgas/NEAR and NEAR = $3, this costs ~$0.03 in NEAR.
7. At ETH = $3000, the relayer received `0.00005 ETH = $0.15`. In this scenario the relayer profits.
8. But if NEAR gas price spikes (e.g., 1 Tgas/NEAR) or ETH/NEAR rate shifts, the relayer's 100 Tgas costs 100 NEAR = $300, while the ETH reward is still $0.15 — a $299.85 loss per transaction.
9. Rational relayers stop submitting; user transactions are stuck. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn charge_gas(
        &mut self,
        sender: &Address,
        transaction: &NormalizedEthTransaction,
        max_gas_price: Option<U256>,
        fixed_gas: Option<EthGas>,
    ) -> Result<GasPaymentResult, GasPaymentError> {
        let block_base_fee_per_gas = self.block_base_fee_per_gas();
        if transaction.max_fee_per_gas.is_zero()
            && fixed_gas.is_none()
            && block_base_fee_per_gas.is_zero()
        {
            return Ok(GasPaymentResult::default());
        }

        if transaction.max_fee_per_gas < block_base_fee_per_gas {
            return Err(GasPaymentError::MaxFeePerGasLessThanBaseFee);
        }

        let priority_fee_per_gas = transaction
            .max_priority_fee_per_gas
            .min(transaction.max_fee_per_gas - block_base_fee_per_gas);
        let priority_fee_per_gas = max_gas_price.map_or(priority_fee_per_gas, |price| {
            price.min(priority_fee_per_gas)
        });
        let effective_gas_price = priority_fee_per_gas + block_base_fee_per_gas;
        // First, we try to use `fixed_gas`. At this point we already know that the `fixed_gas` is
        // less than the `gas_limit`. It allows avoiding refunding unused gas to the sender later.
        let prepaid_amount = fixed_gas
            .map_or(transaction.gas_limit, EthGas::as_u256)
            .checked_mul(effective_gas_price)
            .map(Wei::new)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        let new_balance = get_balance(&self.io, sender)
            .checked_sub(prepaid_amount)
            .ok_or(GasPaymentError::OutOfFund)?;

        set_balance(&mut self.io, sender, &new_balance);

        self.gas_price = effective_gas_price;

        Ok(GasPaymentResult {
            prepaid_amount,
            effective_gas_price,
            priority_fee_per_gas,
        })
    }

pub fn refund_unused_gas<I: IO>(
    io: &mut I,
    sender: &Address,
    gas_used: u64,
    gas_result: &GasPaymentResult,
    relayer: &Address,
    fixed_gas: Option<EthGas>,
) -> Result<(), GasPaymentError> {
    if gas_result.effective_gas_price.is_zero() {
        return Ok(());
    }

    let (refund, relayer_reward) = {
        let gas_to_wei = |price: U256| {
            fixed_gas
                .map_or_else(|| gas_used.into(), EthGas::as_u256)
                .checked_mul(price)
                .map(Wei::new)
                .ok_or(GasPaymentError::EthAmountOverflow)
        };

        let spent_amount = gas_to_wei(gas_result.effective_gas_price)?;
        let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;

        let refund = gas_result
            .prepaid_amount
            .checked_sub(spent_amount)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        (refund, reward_amount)
    };

    if !refund.is_zero() {
        add_balance(io, sender, refund)?;
    }

    if !relayer_reward.is_zero() {
        add_balance(io, relayer, relayer_reward)?;
    }

    Ok(())
}

pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let tx_data = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let args = SubmitArgs {
            tx_data,
            ..Default::default()
        };
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}
