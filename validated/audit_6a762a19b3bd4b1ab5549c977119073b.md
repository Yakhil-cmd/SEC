### Title
Unprivileged Caller Can Force Victim's ICP Into SNS Swap Participation, Bypassing Confirmation-Text Consent - (`rs/sns/swap/canister/canister.rs`)

---

### Summary

The SNS Swap canister's `refresh_buyer_tokens` endpoint accepts an arbitrary `buyer` principal from any caller without verifying that the caller is the buyer. Because the swap's `confirmation_text` (the explicit consent mechanism) is publicly readable from the canister's state, any unprivileged ingress sender can call `refresh_buyer_tokens` on behalf of a victim who has already transferred ICP to the swap subaccount, forcing that victim's ICP into the swap participation without the victim's explicit consent at the time of registration.

---

### Finding Description

The `refresh_buyer_tokens` update method in the SNS Swap canister accepts a `buyer` field that can be set to any arbitrary principal by any caller:

```rust
// rs/sns/swap/canister/canister.rs
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()  // No check that caller == buyer
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
``` [1](#0-0) 

Inside `refresh_buyer_token_e8s`, the function:
1. Validates the `confirmation_text` against the SNS-configured text
2. Reads the ICP balance from the subaccount derived from the `buyer` principal
3. Records that balance as the buyer's participation in the swap [2](#0-1) 

The `confirmation_text` is the SNS's explicit consent mechanism — participants must send the exact text to confirm they agree to the swap terms. However, this text is stored in the swap canister's public `Init` state and is readable by anyone via `get_state`. The proto definition confirms it is an optional public field: [3](#0-2) 

The `RefreshBuyerTokensRequest` proto explicitly documents that the `buyer` field is caller-optional: [4](#0-3) 

The design intent of allowing any caller to specify any buyer is confirmed by a system test that explicitly calls `refresh_buyer_tokens` for a different user (the wealthy user) from the default identity: [5](#0-4) 

---

### Impact Explanation

The SNS swap participation flow is:
1. Victim transfers ICP to `swap_canister[subaccount=principal_to_subaccount(victim)]` on the ICP ledger
2. Victim is expected to call `refresh_buyer_tokens(buyer=self, confirmation_text=...)` to register participation

An attacker can:
1. Read the `confirmation_text` from the swap canister's public state (via `get_state`)
2. Observe on the ICP ledger that the victim has transferred ICP to the swap subaccount
3. Call `refresh_buyer_tokens(buyer=victim_principal, confirmation_text=correct_text)` before the victim does

The victim is now registered as a swap participant without having explicitly consented at the time of registration. If the swap commits, the victim receives SNS tokens instead of ICP — an outcome they may not have wanted (e.g., they changed their mind after the transfer, or they were waiting to see swap progress before committing). The `confirmation_text` consent mechanism is rendered meaningless because any caller can supply it on behalf of any buyer.

The victim's ICP is locked into the swap. After a committed swap, there is no path to recover ICP — only SNS tokens are distributed. [6](#0-5) 

---

### Likelihood Explanation

- The attacker needs no special privileges — any ingress sender can call `refresh_buyer_tokens`
- The `confirmation_text` is publicly readable from the swap canister state
- ICP transfers to swap subaccounts are visible on the public ICP ledger
- The attack window is the time between the victim's ICP transfer and their own `refresh_buyer_tokens` call
- The IC does not have a traditional mempool, but the attacker can poll the ledger and race the victim's notification call

---

### Recommendation

Enforce that the `buyer` field, when non-empty, must equal the caller's principal. If the caller is not the buyer, reject the call. This ensures that only the buyer themselves can register their own participation and provide the `confirmation_text` consent:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() {
        ic_cdk::trap("buyer must be the caller");
    }
    specified
};
```

This mirrors the fix recommended in the original report: binding the depositor identity to `msg.sender` rather than allowing it to be an arbitrary parameter.

---

### Proof of Concept

1. SNS swap is open with `confirmation_text = "I agree to the terms"` (readable via `get_state`)
2. Victim transfers 100 ICP to `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(victim) }` on the ICP ledger
3. Attacker (any principal) calls:
   ```
   refresh_buyer_tokens({
     buyer: "<victim_principal_text>",
     confirmation_text: Some("I agree to the terms")
   })
   ```
4. The swap canister reads the victim's subaccount balance (100 ICP), validates the confirmation text (passes, because the text matches), and registers the victim as a participant with 100 ICP
5. The victim's ICP is now committed to the swap without the victim having explicitly called `refresh_buyer_tokens` themselves
6. If the swap commits, the victim receives SNS tokens — an outcome they did not explicitly consent to at the time of registration [7](#0-6) [8](#0-7)

    pub async fn refresh_buyer_token_e8s(
        &mut self,
        buyer: PrincipalId,
        confirmation_text: Option<String>,
        this_canister: CanisterId,
        icp_ledger: &dyn ICRC1Ledger,
    ) -> Result<RefreshBuyerTokensResponse, String> {
        use swap_participation::*;

        // These two checks need to be repeated after awaiting the response from the ICP ledger.
        self.validate_lifecycle_is_open()
            .map_err(context_before_awaiting_icp_ledger_response)?;
        self.validate_possibility_of_direct_participation()
            .map_err(context_before_awaiting_icp_ledger_response)?;

        // User input validation doesn't expire after await, so this check doesn't need repetition.
        self.validate_confirmation_text(confirmation_text)?;

        // Look for the token balance of the specified principal's subaccount on 'this' canister.
        let e8s = {
            let account = Account {
                owner: this_canister.get().0,
                subaccount: Some(principal_to_subaccount(&buyer)),
            };
            icp_ledger
                .account_balance(account)
                .await
                .map_err(|x| x.to_string())?
                .get_e8s()
        };
