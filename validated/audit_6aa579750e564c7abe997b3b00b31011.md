Audit Report

## Title
Byzantine Dealer Can Trigger Unbounded BSGS Computation in `dec_chunks`, Blocking Crypto Vault Thread Pool — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

## Summary

A Byzantine subnet dealer below the fault threshold can craft a dealing with out-of-range plaintext chunks that passes all ZK proof verifications (`verify_ciphertext_integrity`, `verify_chunking`, `verify_sharing`). When any receiver calls `load_threshold_signing_key`, `dec_chunks` falls through to `CheatingDealerDlogSolver::solve()`, which runs an unbounded BSGS computation that the code itself acknowledges "may take hours." There is no timeout. The computation runs on the shared crypto vault rayon thread pool, and exhausting it blocks all crypto operations on the affected node.

## Finding Description

**Root cause — `check_instance` does not validate plaintext chunk ranges:**

`ChunkingInstance` holds only ciphertext chunks (G1 group elements), not plaintext values. Its `check_instance` method only validates structural properties:

```rust
pub fn check_instance(&self) -> Result<(), ZkProofChunkingError> {
    if self.public_keys.is_empty() || self.ciphertext_chunks.is_empty() || ... {
        return Err(ZkProofChunkingError::InvalidInstance);
    }
    if self.public_keys.len() != self.ciphertext_chunks.len() {
        return Err(ZkProofChunkingError::InvalidInstance);
    }
    Ok(())
}
``` [1](#0-0) 

`verify_chunking` only checks that `z_s < zz` where `zz = 2 * NUM_ZK_REPETITIONS * n * m * (CHUNK_SIZE - 1) * (scale_range - 1)` — a range astronomically larger than `[0, 65535]`. It never checks that the underlying plaintext chunks are in the honest range: [2](#0-1) 

**Root cause — `prove_chunking` succeeds for out-of-range chunks:**

The prover retries until `z_s < zz_big`. Since `zz` is vastly larger than the honest chunk range, a Byzantine dealer with out-of-range chunks (e.g., `chunk * delta` for small `delta`) can always find a valid `sigma` such that `z_s` falls within bounds, producing a proof that passes `verify_chunking`: [3](#0-2) 

**Trigger — `dec_chunks` falls through to `CheatingDealerDlogSolver`:**

When `HonestDealerDlogLookupTable` fails to find the discrete log (because the chunk is outside `[CHUNK_MIN, CHUNK_MAX]`), the code falls through to the BSGS solver with no timeout:

```rust
if dlogs.iter().any(|x| x.is_none()) {
    // Cheating dealer case
    let cheating_solver = CheatingDealerDlogSolver::new(n, m);
    for i in 0..dlogs.len() {
        if dlogs[i].is_none() {
            // TODO(CRP-2550) All BSGS could be run in parallel
            // It may take hours to brute force a cheater's discrete log.
            dlogs[i] = cheating_solver.solve(&powers[i]);
        }
    }
}
``` [4](#0-3) 

**BSGS range is astronomically large:**

`CheatingDealerDlogSolver::new` computes `bsgs_range = 2 * zz - 1` where for a 13-node subnet with 16 chunks: `zz ≈ 224 billion`. The outer loop iterates `scale_range = 256` times, each running a full BSGS lookup over this range: [5](#0-4) [6](#0-5) 

**Thread pool — no timeout, sized to CPU count:**

`load_threshold_signing_key` is dispatched to a rayon thread pool sized to `available_parallelism()` (number of CPUs) with no timeout: [7](#0-6) [8](#0-7) 

**Existing test confirms the scenario:**

The `should_decrypt_correctly_for_cheating_dealer` test confirms that `verify_ciphertext_integrity` passes for out-of-range chunks and that `dec_chunks` invokes the BSGS path. The `#[ignore]`d `print_time_for_cheating_dlog_solver_to_run` test exists specifically to measure the hours-long cost: [9](#0-8) [10](#0-9) 

## Impact Explanation

Every receiver node that calls `load_threshold_signing_key` for a transcript containing the Byzantine dealer's dealing will block one thread pool thread per out-of-range chunk (up to 16 per dealing) for hours. With `f = floor((n-1)/3)` Byzantine dealers allowed below the fault threshold, the effect is multiplied. Thread pool exhaustion (pool size = CPU count) blocks all subsequent crypto vault operations on the affected node — including threshold signing, key generation, and other DKG operations — matching the allowed impact: **Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS** (High, $2,000–$10,000).

## Likelihood Explanation

The attacker must be a registered subnet node acting as a dealer — a Byzantine protocol peer explicitly below the fault threshold and in-scope. The attack is straightforward: craft chunks outside `[0, 65535]` but within the ZK proof's slack range, generate a valid chunking proof (the prover loop succeeds because `zz` is much larger than the honest range), and submit the dealing. The dealing passes `verify_dealing` on all honest nodes. The `CheatingDealerDlogSolver` and the `#[ignore]`d timing test exist precisely because this scenario is a known design concern.

## Recommendation

1. **Add a timeout** to the BSGS computation in `dec_chunks`. If `CheatingDealerDlogSolver::solve` exceeds a bound (e.g., 60 seconds), return `Err(DecErr::InvalidChunk)` and exclude the dealing's contribution rather than blocking indefinitely.
2. **Reject the dealing at transcript assembly time** by adding a fast pre-check: after decryption, verify all chunks fall in `[CHUNK_MIN, CHUNK_MAX]` before including a dealing in the transcript.
3. **Run BSGS in a separate bounded thread** with a timeout, isolated from the main crypto vault thread pool.
4. **Tighten the ZK chunking proof** to more tightly bound plaintext chunks, reducing the slack that allows out-of-range chunks to pass `verify_chunking`.

## Proof of Concept

1. As a registered subnet dealer, create a dealing where one or more plaintext chunks for a target receiver are set to `chunk * delta` for small `delta` (2–10), pushing the value above `CHUNK_MAX = 65535` but within the ZK proof's `zz` slack range.
2. Call `enc_chunks` with `PlaintextChunks::new_unchecked` for the out-of-range chunks, then call `prove_chunking` — the prover loop will succeed because `zz` is much larger than the honest range.
3. Verify the dealing passes `verify_ciphertext_integrity` and `verify_chunking` (both confirmed by design: `check_instance` does not check plaintext ranges, and `verify_chunking` only checks `z_s < zz`).
4. Submit the dealing; it is included in the DKG transcript.
5. When any receiver calls `load_threshold_signing_key`, measure wall-clock time of `dec_chunks` — it will invoke `CheatingDealerDlogSolver::solve()` for each out-of-range chunk. Enable the `#[ignore]`d test `print_time_for_cheating_dlog_solver_to_run` to measure the per-chunk cost at production subnet sizes. [11](#0-10) [12](#0-11)

async fn execute_on_thread_pool<F, T>(thread_pool: &ThreadPool, job: F) -> T
where
    F: FnOnce() -> T,
    F: Send + 'static,
    T: Send + 'static,
{
    let (tx, rx) = tokio::sync::oneshot::channel();
    thread_pool.spawn(move || {
        if tx.is_closed() {
            // Do not start the job if the associated receiver handle was
            // dropped in the meanwhile (e.g., because the client closed
            // the connection or the client's RPC framework cancelled the
            // future due to a timeout).
            return;
        }
        let result = job();

        // Errors occur if the associated receiver handle was dropped and are considered
        // legitimate and are thus ignored.
        let _ = tx.send(result);
    });
    rx.await.expect("the sender was dropped")
}

#[test]
fn should_decrypt_correctly_for_cheating_dealer() {
    let epoch = Epoch::from(0);
    let rng = &mut reproducible_rng();
    let associated_data = rng.r#gen::<[u8; 10]>();

    let sys = SysParam::global();
    let key_gen_assoc_data = rng.r#gen::<[u8; 32]>();

    let nodes = 3;

    let mut keys = Vec::new();
    for _ in 0..nodes {
        let key_pair = kgen(&key_gen_assoc_data, sys, rng);
        keys.push(key_pair);
    }
    let pks: Vec<_> = keys.iter().map(|key| key.0.public_key().clone()).collect();

    let mut sij = {
        let mut sij = Vec::with_capacity(nodes);

        for _ in 0..nodes {
            let mut chunks = [0; NUM_CHUNKS];

            for i in 0..NUM_CHUNKS {
                // ensure that multiplying by delta pushes us out of Chunk range
                let chunk = (0x8000 | rng.r#gen::<u16>()) as isize;
                chunks[i] = chunk;
            }
            // this ensures that chunks is the encoding of a scalar less
            // than the group order:
            chunks[0] %= 0x73ee;

            sij.push(chunks);
        }

        sij
    };

    let cheating_i = rng.r#gen::<usize>() % nodes;
    let cheating_j = std::cmp::max(1, rng.r#gen::<usize>() % NUM_CHUNKS);

    let delta = (2 + rng.r#gen::<usize>() % 10) as isize;
    sij[cheating_i][cheating_j] *= delta; // doesn't overflow as delta is small and isize >> u16

    // however the new sij *is* larger than the maximum "legal" chunk
    assert!(sij[cheating_i][cheating_j] > CHUNK_MAX);

    let cheating_chunks = sij
        .iter()
        .map(|c| PlaintextChunks::new_unchecked(*c))
        .collect::<Vec<_>>();

    let pks_and_chunks = pks
        .iter()
        .cloned()
        .zip(cheating_chunks.iter().cloned())
        .collect::<Vec<_>>();

    let (crsz, _witness) = enc_chunks(&pks_and_chunks, epoch, &associated_data, sys, rng);

    // still a valid ciphertext
    assert!(verify_ciphertext_integrity(&crsz, epoch, &associated_data, sys).is_ok());

    for i in 0..keys.len() {
        let secret_key = &keys[i].1;
        let out = dec_chunks(secret_key, i, &crsz, epoch, &associated_data);
        assert_eq!(
            out.unwrap(),
            cheating_chunks[i].recombine_to_scalar(),
            "Message decrypted wrongly"
        );
    }
}
