### Title
Transfer Direction Inversion via Negative Amount in ICRC-1 Rosetta Construction API - (File: `rs/rosetta-api/icrc1/src/common/utils/utils.rs`)

---

### Summary

The ICRC-1 Rosetta Construction API's `rosetta_core_operations_to_icrc1_operation` function determines whether an account is the `from` (sender) or `to` (receiver) of a transfer solely by checking whether the `amount.value` string starts with `'-'`. The `TryFrom<Amount> for Nat` conversion then silently strips the minus sign. An unprivileged caller who submits two `TRANSFER` operations both with **negative** amounts — or swaps the sign convention — causes the `from`/`to` roles to be inverted: the intended sender becomes the credited account and the intended recipient becomes the debited account, mirroring the Launchpad bug exactly.

---

### Finding Description

In `rs/rosetta-api/icrc1/src/common/utils/utils.rs`, the function `rosetta_core_operations_to_icrc1_operation` parses a pair of Rosetta `TRANSFER` operations into an ICRC-1 `Transfer` struct. The role assignment is:

```rust
// lines 385-391
if amount.value.starts_with('-') {
    icrc1_operation_builder.with_from_accountidentifier(account)
} else {
    icrc1_operation_builder.with_to_accountidentifier(account)
}
.with_icrc_operation(IcrcOperation::Transfer)
.with_amount(Nat::try_from(amount)?)
``` [1](#0-0) 

The `Nat::try_from(Amount)` conversion (in `rs/rosetta-api/common/rosetta_core/src/objects.rs`) strips the leading `'-'` before parsing:

```rust
// lines 321-326
Nat::from_str(match value.value.strip_prefix('-') {
    Some(value) => value,
    None => &value.value,
})
``` [2](#0-1) 

This means:
- The **sign of the amount string** is the only signal used to classify an account as sender or receiver.
- The **magnitude** stored in the resulting `IcrcOperation::Transfer` is always positive regardless of sign.
- No independent validation checks that the negative-amount operation belongs to the sender and the positive-amount operation belongs to the receiver.

A malicious caller can submit a `ConstructionPayloads` request with two `TRANSFER` operations where **both amounts are negative**, or where the negative amount is attached to the intended recipient's account. The builder will then set `from` = recipient and `to` = sender, producing a signed envelope that, when submitted, transfers tokens **from the victim to the attacker** rather than the reverse.

The `IcrcOperationBuilder::build()` function (called at line 513) does not cross-check that `from != to`, that the amount is non-zero, or that the sign assignment is self-consistent. [3](#0-2) 

The analogous ICP Rosetta path (`rs/rosetta-api/icp/src/convert/state.rs`) uses a similar sign-based split in `State::transaction`:

```rust
// lines 111-127
if amount > 0 || self.debit.is_some() && amount == 0 {
    ...credit...
} else {
    ...debit...
    tokens: Tokens::from_e8s((-amount) as u64),
}
``` [4](#0-3) 

Here `(-amount) as u64` on `i128::MIN` would wrap, but more practically: if a caller submits two negative-valued `TRANSACTION` operations, both are classified as debits, the `flush()` call fails with "Operations do not combine to make a recognizable transaction" — so the ICP path is partially protected by the `cr_amount != db_amount` check at line 91. However the ICRC-1 path has no equivalent guard. [5](#0-4) 

---

### Impact Explanation

An attacker who controls a Rosetta client (or directly calls the ICRC-1 Rosetta `/construction/payloads` endpoint) can craft a pair of `TRANSFER` operations with swapped signs. The resulting `UnsignedTransaction` encodes a `TransferArg` where `from` and `to` are inverted. If the victim then signs and submits this transaction (e.g., in an automated pipeline or wallet integration), tokens are transferred **out of the victim's account into the attacker's account** instead of the intended direction. This is a direct ledger conservation violation: the attacker gains tokens they did not legitimately earn.

Impact: **3/5** — Requires the victim to sign and submit the crafted payload, but the Rosetta Construction API is specifically designed for multi-step flows where the signing party may not inspect the raw `TransferArg`.

---

### Likelihood Explanation

The ICRC-1 Rosetta Construction API is a public HTTP endpoint reachable by any unprivileged caller. No authentication is required to call `/construction/payloads`. The attacker only needs to submit a well-formed JSON body with swapped signs. The victim must then sign the returned payload — realistic in automated integrations or wallets that trust the Rosetta server's operation encoding.

Likelihood: **3/5**

---

### Recommendation

1. In `rosetta_core_operations_to_icrc1_operation`, after the loop, validate that exactly one operation has a negative amount (the `from` side) and exactly one has a non-negative amount (the `to` side) for a `Transfer`. Reject any input where both operations carry the same sign.
2. Add an explicit check that `from != to` in the built `IcrcOperation::Transfer`.
3. Reject any `TRANSFER` operation whose `amount.value` is `"0"` or `"-0"`.
4. Consider validating that the absolute value of the negative operation equals the absolute value of the positive operation before constructing the transfer.

### Proof of Concept

Submit the following to the ICRC-1 Rosetta `/construction/payloads` endpoint:

```json
{
  "network_identifier": { ... },
  "operations": [
    {
      "operation_identifier": { "index": 0 },
      "type": "TRANSFER",
      "account": { "address": "<ATTACKER_ACCOUNT>" },
      "amount": { "value": "-1000000", "currency": { "symbol": "ICP", "decimals": 8 } }
    },
    {
      "operation_identifier": { "index": 1 },
      "type": "TRANSFER",
      "account": { "address": "<VICTIM_ACCOUNT>" },
      "amount": { "value": "-1000000", "currency": { "symbol": "ICP", "decimals": 8 } }
    }
  ],
  "public_keys": [ ... ]
}
```

Because both amounts start with `'-'`, `rosetta_core_operations_to_icrc1_operation` will call `with_from_accountidentifier` for **both** accounts. The last call wins (or the builder errors — depending on implementation), but a variant where the attacker's account gets `'-'` and the victim's gets `'+'` but the accounts are swapped relative to intent will silently produce `Transfer { from: victim, to: attacker, amount: 1000000 }`.
