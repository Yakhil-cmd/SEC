### Title
ckETH/ckERC20 Withdrawal Queue Saturation Causes `ic_cdk::trap` DoS for All Users - (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`, `rs/ethereum/cketh/minter/src/main.rs`)

---

### Summary

The ckETH minter enforces a hard cap of `MAX_PENDING = 100` on pending withdrawal requests. Both `withdraw_eth` and `withdraw_erc20` call `ic_cdk::trap` — a hard canister panic — when this cap is reached, instead of returning a graceful error. An unprivileged attacker with 100 different principals can saturate the queue with minimum-amount withdrawal requests, causing every subsequent withdrawal call from any user to trap (system-level reject) until the minter's timer drains the queue.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/guard/mod.rs`, the `Guard::new` constructor enforces two limits:

- `MAX_CONCURRENT = 100`: maximum in-flight calls per principal set
- `MAX_PENDING = 100`: maximum total pending withdrawal requests in the queue [1](#0-0) 

When `pending_requests_count(s) >= MAX_PENDING`, the guard returns `Err(GuardError::TooManyPendingRequests)`: [2](#0-1) 

`pending_requests_count` for `PendingWithdrawalRequests` is implemented as `withdrawal_requests_len()`, which counts **all** pending withdrawal requests — both ckETH and ckERC20 — in the shared `pending_withdrawal_requests` queue: [3](#0-2) [4](#0-3) 

Both `withdraw_eth` and `withdraw_erc20` in `rs/ethereum/cketh/minter/src/main.rs` handle **all** guard errors — including the legitimate `TooManyPendingRequests` — by calling `ic_cdk::trap`, which is a hard canister panic that rolls back state and returns a system-level reject to the caller: [5](#0-4) [6](#0-5) 

The attack path is:

1. Attacker controls 100 principals P1…P100, each holding the minimum ckETH withdrawal amount.
2. Attacker calls `withdraw_eth` from each principal. Each call burns ckETH and appends a `WithdrawalRequest` to `pending_withdrawal_requests`. The guard is dropped after each call completes, but the request remains in the queue.
3. Once `pending_withdrawal_requests.len() == 100`, the `MAX_PENDING` check fires for any new caller.
4. Any user calling `withdraw_eth` or `withdraw_erc20` now hits `ic_cdk::trap`, receiving a system reject with no actionable error message.
5. The minter's timer eventually processes the queued requests via `withdrawal_requests_batch`, draining the queue. The attacker re-fills it to sustain the DoS.

Because the queue is shared between ckETH and ckERC20 requests, filling it with ckETH withdrawals also blocks all ckERC20 withdrawals, and vice versa.

Compare this to the ckBTC minter, which handles the analogous condition gracefully with a proper error return and a much larger cap (`MAX_CONCURRENT_PENDING_REQUESTS = 5000`): [7](#0-6) [8](#0-7) 

---

### Impact Explanation

All users are unable to call `withdraw_eth` or `withdraw_erc20` while the queue is saturated. The `ic_cdk::trap` produces a system-level reject (not a typed `WithdrawalError`), so callers receive no actionable feedback and cannot distinguish this condition from a canister bug. Funds are not at risk (the burn happens after the guard check, so no ckETH is burned on a trapped call), but withdrawal liveness is broken for all users for the duration of the attack.

---

### Likelihood Explanation

The attacker's net cost is only the Ethereum transaction fees paid by the minter on the 100 withdrawal transactions (the ETH principal is returned to the attacker's Ethereum address). The `MAX_PENDING = 100` cap is small enough to be saturated in a single burst of 100 ingress calls. The attacker can sustain the DoS by monitoring the queue depth (via the public `get_events` or dashboard endpoint) and re-submitting as the timer drains requests. No privileged access, governance majority, or threshold corruption is required — only 100 principals with the minimum ckETH balance.

---

### Recommendation

1. **Replace `ic_cdk::trap` with a graceful error return** for `TooManyPendingRequests` in both `withdraw_eth` and `withdraw_erc20`. Return `WithdrawalError::TemporarilyUnavailable` (or equivalent) so callers can retry.
2. **Increase `MAX_PENDING`** to a value commensurate with the minter's processing throughput (e.g., `MAX_NUM_PENDING_TRANSACTION_NONCES = 1000`, already used internally in `withdrawal_requests_batch`).
3. **Add per-principal pending request accounting** so a single attacker cannot monopolize the shared queue across multiple principals.

---

### Proof of Concept

```
1. Attacker creates principals P1..P100, each approved for min_withdrawal_amount ckETH.
2. Attacker calls withdraw_eth(min_amount, eth_addr) from each Pi in sequence.
   → Each call: guard acquired, ckETH burned, WithdrawalRequest pushed to pending_withdrawal_requests, guard dropped.
   → After 100 calls: pending_withdrawal_requests.len() == MAX_PENDING == 100.
3. Honest user calls withdraw_eth(any_amount, any_addr):
   → Guard::new checks pending_requests_count(s) >= 100 → Err(TooManyPendingRequests)
   → unwrap_or_else(|e| ic_cdk::trap(...)) → canister traps → system reject to user.
4. Same trap occurs for withdraw_erc20 (shared queue, same guard).
5. Minter timer fires, processes batch, queue drains.
6. Attacker repeats step 2 to re-saturate.
``` [9](#0-8) [10](#0-9)

impl<PR: RequestsGuardedByPrincipal> Guard<PR> {
    /// Attempts to create a new guard for the current code block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    fn new(principal: Principal) -> Result<Self, GuardError> {
        mutate_state(|s| {
            if PR::pending_requests_count(s) >= MAX_PENDING {
                return Err(GuardError::TooManyPendingRequests);
            }
            let principals = PR::guarded_principals(s);
            if principals.contains(&principal) {
                return Err(GuardError::AlreadyProcessing);
            }
            if principals.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            principals.insert(principal);
            Ok(Self {
                principal,
                _marker: PhantomData,
            })
        })
    }

```rust
    pub fn withdrawal_requests_batch(&self, requested_batch_size: usize) -> Vec<WithdrawalRequest> {
        // The number of pending transaction nonces is counted and not the number of pending transactions
        // because a nonce may be associated with several distinct transactions (due to re-submission and dynamic fees).
        // However, once a nonce is chosen for a withdrawal request, it's in our interest that the corresponding transaction be finalized asap.
        // Limiting the number of transactions would be counter-productive.
        const MAX_NUM_PENDING_TRANSACTION_NONCES: usize = 1000;
        let unique_pending_transaction_nonces: BTreeSet<_> =
            self.created_tx.keys().chain(self.sent_tx.keys()).collect();
        let actual_batch_size = min(
            MAX_NUM_PENDING_TRANSACTION_NONCES
                .saturating_sub(unique_pending_transaction_nonces.len()),
            requested_batch_size,
        );
        self.withdrawal_requests_iter()
            .take(actual_batch_size)
            .cloned()
            .collect()
    }
