### Title
Owner Can Immediately Upgrade Contract Bypassing Timelock, Enabling Theft of All Bridged Funds - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The Aurora Engine exposes two upgrade paths. The direct `upgrade()` function allows the owner to immediately replace the entire contract WASM with arbitrary code, with **no timelock whatsoever**. Separately, `set_upgrade_delay_blocks()` accepts any `u64` value including `0` with no minimum validation, allowing the owner to also nullify the timelock on the staged upgrade path. Because the Aurora Engine contract holds all bridged ETH and ERC-20 tokens, this gives the owner unrestricted, immediate ability to steal or permanently freeze all user funds.

### Finding Description

The engine implements two upgrade paths:

**Path 1 — Direct `upgrade()` (no delay):**

The `upgrade()` function checks only `require_owner_only()` and then immediately deploys arbitrary WASM via a NEAR batch promise. There is no reference to `upgrade_delay_blocks` and no timelock of any kind. [1](#0-0) 

**Path 2 — Staged `stage_upgrade()` + `deploy_upgrade()` (delay bypassable):**

The staged path computes `delay_block_height = env.block_height() + state.upgrade_delay_blocks`. If `upgrade_delay_blocks` is `0`, the staged code becomes deployable in the very next block. [2](#0-1) 

`set_upgrade_delay_blocks()` accepts any `u64` value with no minimum bound: [3](#0-2) 

The `deploy_upgrade()` guard uses a strict `<=` check, meaning `block_height == index` is blocked but `block_height > index` (i.e., the very next block when delay is 0) is allowed: [4](#0-3) 

The `EngineState` stores `upgrade_delay_blocks` as a plain `u64` with no enforced minimum: [5](#0-4) 

### Impact Explanation

The Aurora Engine contract is the custodian of all bridged ETH and ERC-20 tokens on Aurora. A malicious WASM upgrade can:

- Redirect all `withdraw` and `ft_transfer` calls to an attacker-controlled address, stealing all bridged ETH and NEP-141 tokens in motion or at rest.
- Remove withdrawal functionality entirely, permanently freezing all user funds.
- Alter the ERC-20 minting/burning logic to create unbacked tokens or destroy user balances.

This satisfies **Critical: Direct theft of any user funds** and **Critical: Permanent freezing of funds**.

### Likelihood Explanation

The `owner_id` is a single NEAR account with no on-chain multi-sig or governance requirement enforced at the contract level. The direct `upgrade()` path requires only one transaction from the owner account. There is no notice period, no community veto window, and no minimum timelock enforced in code. The `set_upgrade_delay_blocks()` function similarly requires only one owner transaction to reduce the delay to `0`.

### Recommendation

1. **Remove the direct `upgrade()` path** or subject it to the same `upgrade_delay_blocks` timelock as the staged path. The existence of a bypass-free immediate upgrade path makes the staged timelock security theater.
2. **Enforce a minimum value in `set_upgrade_delay_blocks()`** — reject any value below a protocol-defined minimum (e.g., equivalent to ~24 hours of NEAR blocks) so the delay cannot be zeroed out.
3. **Consider a multi-sig or DAO governance mechanism** for the `owner_id` role, so no single key can unilaterally upgrade the contract holding all user funds.

### Proof of Concept

**Attack via direct `upgrade()` (single transaction, zero delay):**

1. Owner calls `upgrade(malicious_wasm)` on the Aurora Engine contract.
2. The function passes `require_owner_only()`, then immediately schedules `PromiseAction::DeployContract { code: malicious_wasm }` followed by `state_migration()`.
3. In the same NEAR receipt batch, the malicious WASM is deployed, replacing all contract logic.
4. The malicious contract intercepts all subsequent `withdraw` / `ft_transfer` calls and redirects funds to the attacker. All bridged ETH and ERC-20 tokens are stolen.

**Attack via staged path with zeroed delay:**

1. Owner calls `set_upgrade_delay_blocks({ upgrade_delay_blocks: 0 })`.
2. Owner calls `stage_upgrade(malicious_wasm)` — `delay_block_height = current_block + 0 = current_block`.
3. In the next NEAR block (`block_height > index`), anyone calls `deploy_upgrade()` — the check `block_height <= index` is false, so `Runtime::self_deploy()` executes immediately.
4. Same outcome as above. [1](#0-0) [3](#0-2) [6](#0-5)

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}

    pub extern "C" fn deploy_upgrade() {
        // This function is intentionally not implemented in `contract_methods`
        // because it only makes sense in the context of the NEAR runtime.
        let mut io = Runtime;
        let state = state::get_state(&io).sdk_unwrap();
        require_running(&state)
            .map_err(ContractError::msg)
            .sdk_unwrap();
        let index = internal_get_upgrade_index();
        if io.block_height() <= index {
            sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
        }
        Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
    }
