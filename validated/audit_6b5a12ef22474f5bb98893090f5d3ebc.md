### Title
ERC-20/ETH Balance Permanently Burned Without Refund on `ft_transfer` Failure in `exit_to_near_precompile_callback` — (File: `engine/src/contract_methods/connector.rs`, `engine-precompiles/src/native.rs`)

---

### Summary

When a user calls the `ExitToNear` precompile to bridge ERC-20 tokens (or ETH) from Aurora to NEAR, the EVM balance is burned before the cross-contract `ft_transfer` call is made. If that call fails and the `error_refund` compile-time feature is not enabled, the `exit_to_near_precompile_callback` takes the `else { None }` branch and returns without restoring the burned balance. The user's funds are permanently destroyed with no on-chain recovery path.

---

### Finding Description

**Step 1 — Balance burned before the cross-contract call.**

In `engine-precompiles/src/native.rs`, the `ExitToNear` precompile burns the caller's ERC-20 (or ETH) balance during EVM execution and then schedules an `ft_transfer` / `ft_transfer_call` promise to the NEP-141 contract. A callback `exit_to_near_precompile_callback` is attached to handle the result: [1](#0-0) 

The `refund` field of `ExitToNearPrecompileCallbackArgs` is populated **only** when the `error_refund` feature is compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // <-- always None without the feature
    transfer_near: transfer_near_args,
};
```

**Step 2 — Callback does not restore state when `refund` is `None`.**

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` checks the promise result: [2](#0-1) 

When the `ft_transfer` promise fails **and** `args.refund` is `None` (i.e., `error_refund` is not compiled in), execution falls into the final `else { None }` branch. No re-mint or balance restoration occurs. The burned ERC-20 tokens (or ETH transferred to the `exit_to_near` precompile address) are irrecoverably lost.

**Step 3 — The refund path that should fire.**

`engine::refund_on_error` exists and correctly re-mints ERC-20 tokens or transfers ETH back: [3](#0-2) 

It is simply never reached when `error_refund` is absent.

**Step 4 — Test confirms the loss.**

The existing test explicitly acknowledges the fund loss when the feature is absent: [4](#0-3) 

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

---

### Impact Explanation

Any EVM user who calls the `ExitToNear` precompile targeting a NEAR account that is not registered with the NEP-141 contract (or any other condition that causes `ft_transfer` to fail) will have their ERC-20 tokens or ETH permanently burned with no refund. This is a **critical, permanent loss of user funds** reachable by any unprivileged EVM caller.

---

### Likelihood Explanation

The `ft_transfer` call fails whenever the recipient NEAR account has not called `storage_deposit` on the NEP-141 contract. This is a common, easily triggered condition — any user who exits to a fresh NEAR account or a contract that has not registered storage will trigger the failure. The attacker does not need any special privilege; they only need to call the `ExitToNear` precompile from an EVM contract or transaction.

---

### Recommendation

Remove the `#[cfg(feature = "error_refund")]` / `#[cfg(not(feature = "error_refund"))]` conditional and unconditionally populate `refund` in `ExitToNearPrecompileCallbackArgs`. The `refund_on_error` path already exists and is correct; it must always be reachable when the `ft_transfer` promise fails. Alternatively, ensure the production WASM is always compiled with `error_refund` enabled and enforce this in CI.

---

### Proof of Concept

1. Deploy an ERC-20 on Aurora backed by a NEP-141.
2. Bridge tokens to Aurora (ERC-20 balance credited).
3. Call the `ExitToNear` precompile targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141.
4. The `ft_transfer` promise fails.
5. `exit_to_near_precompile_callback` is invoked: `args.refund` is `None` (no `error_refund` feature), so the `else { None }` branch executes.
6. Observe: ERC-20 balance on Aurora is zero; NEP-141 balance on NEAR is unchanged; tokens are gone.

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) already demonstrates this exact outcome under the `#[cfg(not(feature = "error_refund"))]` branch. [5](#0-4)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
