### Title
Fee-on-Transfer NEP-141 Token Accounting Discrepancy Allows ERC-20 Over-Minting Leading to Bridge Insolvency - (File: `engine/src/engine.rs`)

### Summary
Aurora Engine's `ft_on_transfer` entry point blindly trusts `args.amount` from the NEP-141 callback to mint ERC-20 tokens. If the bridged NEP-141 token implements a transfer fee, Aurora Engine receives fewer tokens than `args.amount` but mints the full `args.amount` in ERC-20 tokens, creating unbacked ERC-20 supply and eventual bridge insolvency.

### Finding Description
When a NEP-141 token is bridged into Aurora via `ft_transfer_call`, the NEP-141 runtime calls `ft_on_transfer` on Aurora Engine with the `amount` field set to the value the *sender* specified — not the amount Aurora Engine actually received. If the NEP-141 token deducts a fee during transfer (a fee-on-transfer token), Aurora Engine's actual NEP-141 balance increases by `amount - fee`, but `ft_on_transfer` is called with `amount`.

The vulnerable path is:

1. `ft_on_transfer` in `engine/src/contract_methods/connector.rs` reads `args.amount` from the JSON input and dispatches to `engine.receive_erc20_tokens`. [1](#0-0) 

2. `receive_erc20_tokens` in `engine/src/engine.rs` takes `args.amount.as_u128()` directly and passes it to `setup_receive_erc20_tokens_input`, which encodes a `mint(recipient, amount)` call to the ERC-20 contract — no balance-before/after check is performed. [2](#0-1) 

3. `setup_receive_erc20_tokens_input` encodes the ERC-20 `mint` selector with the caller-supplied `amount`: [3](#0-2) 

4. The `EvmErc20.mint` function mints exactly the amount passed in, with no independent verification: [4](#0-3) 

At no point does the engine compare the NEP-141 balance of the Aurora contract before and after the transfer to determine the *actual* amount received.

### Impact Explanation
**Critical — Insolvency.** For every `ft_transfer_call` of a fee-on-transfer NEP-141 token, Aurora Engine mints `fee` more ERC-20 tokens than it holds NEP-141 backing. Over time (or in a single large deposit), the ERC-20 total supply exceeds the NEP-141 reserve held by Aurora. When users attempt to exit (burn ERC-20 → receive NEP-141 via `withdrawToNear`/`exit_to_near`), the last withdrawers cannot be satisfied because the NEP-141 balance is insufficient. This is a direct bridge insolvency: user funds are permanently unrecoverable for the last set of withdrawers.

### Likelihood Explanation
**Medium.** Any NEP-141 token that implements a transfer fee and is registered with Aurora Engine (via `deploy_erc20_token`) triggers this condition. The NEP-141 standard does not prohibit fee-on-transfer tokens. An unprivileged token holder only needs to call `ft_transfer_call` on such a token pointing to Aurora Engine — no special permissions required. The condition is latent until enough fee-bearing deposits accumulate to exhaust the reserve.

### Recommendation
In `receive_erc20_tokens`, record the Aurora Engine contract's NEP-141 balance before and after the transfer (using a cross-contract view or by relying on the NEP-141 `ft_resolve_transfer` callback return value) and mint only the *net received* amount. Alternatively, document and enforce that only zero-fee NEP-141 tokens may be registered via `deploy_erc20_token`, and add an on-chain guard that rejects registration of tokens whose transfer behavior cannot be verified.

### Proof of Concept
1. Deploy a NEP-141 token `fee_token.near` that deducts 1% on every `ft_transfer`/`ft_transfer_call`.
2. Call `deploy_erc20_token` on Aurora Engine to register `fee_token.near` → `EvmErc20` at address `0xFEE...`.
3. Call `ft_transfer_call` on `fee_token.near` with `receiver_id = aurora`, `amount = 1000`, `msg = <evm_recipient_hex>`.
4. `fee_token.near` transfers 990 tokens to Aurora Engine (deducts 10 as fee) but calls `ft_on_transfer` with `amount = 1000`.
5. Aurora Engine's `receive_erc20_tokens` mints 1000 `EvmErc20` tokens to `<evm_recipient>`.
6. Aurora Engine holds 990 NEP-141 tokens but has 1000 ERC-20 tokens in circulation.
7. Repeat: after 100 such deposits the ERC-20 supply exceeds NEP-141 reserve by 10% — the last ~10% of ERC-20 holders cannot withdraw, their funds are permanently frozen. [5](#0-4) [6](#0-5)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[must_use]
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);

    [selector, tail.as_slice()].concat()
}
