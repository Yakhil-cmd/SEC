### Title
`native_per_pubdata` Truncates to Zero via Floor Division When `pubdata_price < native_price`, Causing Pubdata to Be Uncharged in Native Resources - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's double resource accounting model, the `native_per_pubdata` ratio is computed using floor (truncating) integer division. When the operator-set `pubdata_price` is less than `native_price`, this ratio silently truncates to zero. As a result, every byte of pubdata published by any transaction in that block costs zero native resources, allowing users to publish pubdata without consuming any of their native resource budget — a systematic undercharge analogous to the funding-fee precision loss in the external report.

---

### Finding Description

**Root cause — floor division in `native_per_pubdata`:**

In `validate_and_compute_fee_for_transaction` (L2 transactions):

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's standard integer floor division. When `pubdata_price < native_price`, the result is `0`.

The same floor division appears in the off-chain helper used by RPC nodes:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

**Contrast with `native_per_gas`**, which correctly uses ceiling division to avoid undercharging:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

**Downstream propagation — pubdata becomes free:**

`native_per_pubdata = 0` flows into two critical charging sites:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   // basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, line 352
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // → 0 * intrinsic_pubdata = 0
   ```

2. **Execution pubdata charging** in `get_resources_to_charge_for_pubdata`:
   ```rust
   // basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, line 430-431
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)  // → 0 for any pubdata_spent
       .ok_or(out_of_native_resources!())?;
   ```

When `native_per_pubdata = 0`, the entire pubdata cost in native resources is zero regardless of how many storage slots the transaction writes. The user's native resource budget (`native_per_gas * gas_limit`) is never reduced by pubdata consumption.

**Concrete numeric example:**

| Parameter | Value |
|---|---|
| `native_price` | 10 |
| `pubdata_price` | 9 |
| `native_per_pubdata` (floor) | **0** |
| `native_per_pubdata` (ceil, correct) | **1** |

With `pubdata_price = 9` and `native_price = 10`, the operator intends to charge 9/10 of a native unit per pubdata byte. Floor division silently drops this to 0. A transaction writing 1,000 storage slots (≈ 32,000 pubdata bytes) pays **0** native resources for pubdata instead of the intended ~28,800 native units.

---

### Impact Explanation

- **Resource accounting bug / funds-loss path**: The protocol systematically fails to collect native resource fees for pubdata whenever `pubdata_price < native_price`. The operator's intended pubdata pricing is silently ignored.
- **User-exploitable**: Any unprivileged user who observes the current block's `pubdata_price` and `native_price` (both are public block metadata) can submit storage-heavy transactions and publish pubdata at zero native cost, exhausting the block's pubdata budget without paying for it.
- **Severity analog**: The external report showed a 1% fee loss from a similar truncation. Here the loss is 100% of pubdata fees whenever `pubdata_price mod native_price != 0` and `pubdata_price < native_price` — a strictly worse outcome.

---

### Likelihood Explanation

The operator sets `pubdata_price` and `native_price` independently. The condition `pubdata_price < native_price` is reachable in normal operation:
- The default test metadata sets `pubdata_price = 0`, `native_price = 10`, confirming `native_per_pubdata = 0` is a supported runtime state.
- Any operator configuration where pubdata is priced below the native cycle cost (e.g., during low-congestion periods) triggers the truncation.
- No on-chain enforcement prevents `pubdata_price < native_price`.

---

### Recommendation

Replace floor division with ceiling division for `native_per_pubdata`, mirroring the existing pattern used for `native_per_gas`:

```rust
// Before (floor division — undercharges):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Alternatively, add a minimum value of 1 for `native_per_pubdata` when `pubdata_price > 0`, analogous to the `MIN_FEE` mitigation suggested in the external report.
