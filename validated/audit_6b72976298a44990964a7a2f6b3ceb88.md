### Title
Calldata-Supplied Sender Address in `send_to_l1_inner` Bypasses Actual Caller Identity — (`system_hooks/src/call_hooks/l1_messenger.rs`)

### Summary

`send_to_l1_inner` reads the message sender address from the first 20 bytes of attacker-influenced calldata and records it verbatim as the L2→L1 message originator. The `callers_caller` field — the actual on-chain identity of whoever triggered the L1 messenger contract — is present in `ExternalCallRequest` but is explicitly discarded. Any code path inside the L1 messenger system contract (0x8008) that lets a caller supply an arbitrary sender address will produce a forged L2→L1 log whose `key` field (the "message sender address" committed to the Merkle tree and pubdata) is fully attacker-controlled.

### Finding Description

`l1_messenger_hook` correctly gates execution on `caller == L1_MESSENGER_ADDRESS`: [1](#0-0) 

After that gate passes, control flows to `send_to_l1_inner`, which reads the sender from calldata bytes `[0..20]`: [2](#0-1) 

The `callers_caller` field — the address that called the L1 messenger contract and therefore the true originator of the message — is destructured and silently dropped with `_`: [3](#0-2) 

`emit_l1_message` stores the calldata-supplied address as the `address` of a `UserMsgData` entry: [4](#0-3) 

When the `LogContent` is serialised into an `L2ToL1Log`, that stored address becomes the `key` field — explicitly documented as "message sender address (padded)": [5](#0-4) 

The `key` is then encoded into the 88-byte log, hashed into the Merkle tree, and published in pubdata: [6](#0-5) 

There is no cross-check that the calldata-supplied address equals `callers_caller`. The hook comment even acknowledges the trust dependency: *"the L1 messenger system contract should guarantee correct usage"*: [7](#0-6) 

### Impact Explanation

The `key` field of every user L2→L1 log is the value that L1 bridge and messaging contracts use to verify which L2 address sent a message. If an attacker can cause the L1 messenger contract (0x8008) to forward an arbitrary address in the first 20 bytes of its hook calldata, the resulting log will carry a forged sender identity that is cryptographically committed to the batch Merkle tree and accepted by L1 verifiers. This enables impersonation of any L2 address (e.g., a token bridge contract) in cross-chain messages, potentially allowing unauthorised withdrawals or state changes on L1 that are gated on the L2 sender identity. [8](#0-7) 

### Likelihood Explanation

The hook is correctly restricted to `L1_MESSENGER_ADDRESS` (0x8008), so direct exploitation requires a code path inside the L1 messenger EVM contract that lets a caller supply an arbitrary sender address. The L1 messenger contract source is not present in this repository, so direct exploitability cannot be confirmed from the in-scope files alone. However, the hook's design is structurally unsound: it ignores the cryptographically-derived `callers_caller` and instead trusts a mutable calldata field. Any future upgrade, bug, or alternative call path in the L1 messenger contract immediately becomes exploitable without any change to the hook. The pattern is directly analogous to the reported `notionalCallback` bug — a caller-controlled parameter substituted for actual caller identity.

### Recommendation

Replace the calldata-read `address_sender` with the `callers_caller` value extracted from `ExternalCallRequest`, which is the actual address that invoked the L1 messenger contract and cannot be forged by the caller:

```rust
// In l1_messenger_hook, capture callers_caller instead of discarding it:
let ExternalCallRequest {
    caller,
    callers_caller,   // <-- capture, do not discard
    input: calldata,
    ...
} = request;

// In send_to_l1_inner (or inline), use callers_caller as the sender:
system.io.emit_l1_message(
    ExecutionEnvironmentType::NoEE,
    resources,
    &callers_caller,   // authoritative sender
    message,
)?;
```

Alternatively, add an assertion that the calldata-supplied address equals `callers_caller` before accepting it.

### Proof of Concept

1. Deploy a contract `Forger` on ZKsync OS L2.
2. `Forger.forge(address victim, bytes message)` calls the L1 messenger contract at `0x8008` with calldata constructed as `abi.encodePacked(victim, message)` — i.e., it manually prepends an arbitrary address instead of `msg.sender`.
3. If the L1 messenger contract forwards this calldata verbatim to the hook (or has any function that accepts a caller-supplied sender), `send_to_l1_inner` reads `victim` as `address_sender` at line 147 and records it in the L2→L1 log.
4. The resulting `L2ToL1Log.key` equals `victim`, not `Forger`'s address.
5. On L1, a bridge contract that checks `log.key == trustedL2Bridge` will accept the forged message as if it originated from `victim`. [9](#0-8) [10](#0-9)

```rust
pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;

    Ok(Ok(()))
}
```
