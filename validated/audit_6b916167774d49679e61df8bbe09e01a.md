### Title
Cycles Permanently Burned on Ingress to Stopping/Stopped Canister Before Status Check — (`rs/messaging/src/scheduling/valid_set_rule.rs`)

### Summary

In `ValidSetRuleImpl::enqueue`, the ingress induction cost is **charged before** the canister status is checked. When the canister is `Stopping` or `Stopped`, the function returns an error without refunding the already-deducted cycles, permanently burning them for a message that is never enqueued.

### Finding Description

In `enqueue` (lines 244–337 of `rs/messaging/src/scheduling/valid_set_rule.rs`), the execution order is:

1. **Charge cycles** via `charge_ingress_induction_cost` (lines 305–315) — this calls `consume_with_threshold` for any canister without a paused execution, immediately and permanently deducting cycles from the canister's balance.
2. **Check canister status** (lines 318–331) — only after the charge, the code checks `canister.status()`. If `Stopping` or `Stopped`, it returns `Err(IngressInductionError::CanisterStopping/Stopped)` with **no refund**. [1](#0-0) 

In `charge_ingress_induction_cost` (lines 316–357 of `rs/cycles_account_manager/src/cycles_account_manager.rs`), for a canister without a paused execution (which is the normal case for `Stopped` and most `Stopping` canisters), `consume_with_threshold::<IngressInduction>` is called directly — a permanent, irreversible deduction: [2](#0-1) 

There is no refund path anywhere in `enqueue` after the status check fails. [3](#0-2) 

The existing test `ingress_to_stopping_canister_is_rejected` confirms the error is returned but **does not assert that the canister's cycle balance is unchanged**, leaving the burn undetected: [4](#0-3) 

### Impact Explanation

An unprivileged attacker who knows the ID of a `Stopping` or `Stopped` canister can repeatedly submit ingress messages to it. Each submission causes the canister to pay the ingress induction cost (base fee + per-byte fee) while the message is never placed in the input queue and never executed. The cycles are reported as consumed (`IngressInduction` use-case), meaning they are burned — not transferred to anyone. Repeated submissions drain the target canister's cycles balance without any useful work being performed.

### Likelihood Explanation

- Canister IDs are public. Any canister that has ever been stopped or is in the process of stopping is a valid target.
- No special privileges are required; any anonymous or authenticated user can submit ingress messages.
- The `Stopped` state is persistent until the canister is explicitly restarted, giving an indefinite attack window.
- The `Stopping` state window is bounded by in-flight call completion, but can be prolonged if the canister has many open calls.
- The attack is rate-limited by the ingress pool and subnet throughput, but even at modest rates the cumulative burn is real.

### Recommendation

Move the canister status check **before** the call to `charge_ingress_induction_cost`. The corrected order in `enqueue` should be:

1. Resolve `effective_canister_id`
2. Compute `induction_cost`
3. **Check `canister.status()`** — return `Err(CanisterStopping/Stopped)` immediately if not `Running`
4. Only then call `charge_ingress_induction_cost`
5. Call `state.push_ingress`

This matches the logical invariant: a canister should only pay for messages it will actually receive.

### Proof of Concept

State-machine test outline:

// 2. Stop the canister (status = Stopped)
// 3. Send K ingress messages via valid_set_rule.induct_messages(...)

// 5. Assert: canister balance == freeze_threshold + N - K * induction_cost

fn ingress_to_stopping_canister_is_rejected() {
    let ingress_history_writer = Arc::new(NoopIngressHistoryWriter);
    let metrics_registry = MetricsRegistry::new();
    let valid_set_rule = ValidSetRuleImpl::new(
        ingress_history_writer,
        Arc::new(CyclesAccountManagerBuilder::new().build()),
        &metrics_registry,
        no_op_logger(),
    );

    let mut state = ReplicatedStateBuilder::new()
        .with_canister(
            CanisterStateBuilder::new()
                .with_canister_id(canister_test_id(0))
                .with_status(CanisterStatusType::Stopping)
                .build(),
        )
        .build();

    let ingress = SignedIngressBuilder::new()
        .canister_id(canister_test_id(0))
        .build();
    assert_eq!(
        valid_set_rule.enqueue(&mut state, ingress),
        Err(IngressInductionError::CanisterStopping(canister_test_id(0)))
    );
}
