### Title
Unprotected Legacy Transactions Accepted Without Chain-ID Binding, Enabling Cross-Chain Replay - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS explicitly accepts pre-EIP-155 ("unprotected") legacy transactions — those with `v ∈ {27, 28}` — without any chain-ID binding in the signed hash. A signature produced on Ethereum mainnet (or any other EVM chain) for a legacy transaction with the same `nonce`, `to`, `value`, and `data` fields is cryptographically valid on ZKsync OS and will be accepted and executed, draining the sender's funds on ZKsync without their knowledge or consent.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code explicitly branches on whether the signature is EIP-155-protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: v must match 35 + 2*chainId (+ {0,1})
    ...
}
``` [1](#0-0) 

When `v == 27` or `v == 28`, `is_eip155()` returns `false`:

```rust
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
``` [2](#0-1) 

The unprotected branch computes the signing hash over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` — **no chain ID is included**. This is the same hash that Ethereum mainnet (and every other pre-EIP-155 chain) would compute for the same transaction. The parsed transaction is stored as `RlpEncodedTxInner::Legacy` (not `LegacyWithEIP155`): [3](#0-2) 

The `chain_id()` accessor returns `None` for this variant, confirming no chain binding: [4](#0-3) 

The signature is then verified in `validate_and_compute_fee_for_transaction` using the chain-ID-free hash, and if `ecrecover` recovers the correct `from` address, the transaction is accepted and executed: [5](#0-4) 

The nonce check that follows does prevent literal replay of the *same nonce* on ZKsync, but it does **not** prevent replay of a transaction that was signed on another chain at the same nonce value, because the signed hash is identical across all chains for unprotected legacy transactions.

---

### Impact Explanation

An attacker who observes an unprotected legacy transaction on Ethereum mainnet (or any other EVM chain) with `v ∈ {27, 28}` can submit the identical raw bytes to ZKsync OS. If the sender's ZKsync account nonce matches the transaction nonce, the transaction will be accepted and executed on ZKsync, transferring value or executing arbitrary calldata — **without the sender ever authorizing a ZKsync transaction**. This constitutes unauthorized movement of user funds on ZKsync.

The impact is direct theft or unauthorized movement of assets. The attacker needs no privileged access; they only need to observe a public Ethereum transaction and submit it to ZKsync.

---

### Likelihood Explanation

Pre-EIP-155 transactions are still broadcast on Ethereum mainnet today (hardware wallets, old tooling, and some DeFi protocols still produce them). Any user whose Ethereum nonce on ZKsync matches the nonce of an unprotected legacy transaction they sent on mainnet is vulnerable. Because ZKsync is an EVM-compatible L2 where many users have the same address and nonce sequence as on mainnet, this overlap is realistic and not rare.

---

### Recommendation

Reject unprotected legacy transactions (those with `v ∈ {27, 28}`) at the parsing stage. All modern EVM tooling uses EIP-155 signatures. Accepting unprotected legacy transactions provides no meaningful user benefit while creating a cross-chain replay surface. The fix is a single check in `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`:

```rust
if legacy_signature.is_eip155() == false {
    return Err(InvalidTransaction::InvalidEncoding.into());
}
```

Alternatively, if unprotected legacy transactions must be supported for compatibility, document the security property explicitly and ensure the operator-level chain configuration makes the nonce-space divergence between ZKsync and mainnet a reliable protection (it currently is not guaranteed).

---

### Proof of Concept

1. Alice sends an unprotected legacy transaction on Ethereum mainnet at nonce `N`:
   - `v = 27`, `r`, `s` (standard secp256k1 signature over `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))`)
   - This transaction is publicly visible on-chain.

2. Alice also has a ZKsync OS account at the same address with nonce `N` (common for new users bridging from mainnet).

3. Attacker takes the raw RLP bytes of Alice's mainnet transaction and submits them to ZKsync OS.

4. `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` detects `v == 27`, enters the unprotected branch, and computes `sig_hash = keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the mainnet hash. [6](#0-5) 

5. `validate_and_compute_fee_for_transaction` calls `ecrecover(sig_hash, v=27, r, s)` and recovers Alice's address, matching `from`. Signature check passes. [7](#0-6) 

6. Nonce check: Alice's ZKsync nonce is `N`, transaction nonce is `N` — passes. [8](#0-7) 

7. The transaction executes on ZKsync, sending Alice's ZKsync funds to the `to` address — an action Alice never authorized on ZKsync.
