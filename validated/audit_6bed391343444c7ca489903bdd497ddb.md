Audit Report

## Title
Uncancellable `wait_for` After `Insert` Blocks `Remove` Emission and Leaks Unvalidated Pool Memory — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary
In `process_slot_update`, after the outer `select!` resolves with `AssembleResult::Done`, the cancellation token future is dropped and execution falls into sequential code containing `peer_rx.wait_for(|p| p.is_empty()).await` at line 500 (and line 521 for `Unwanted`) with no enclosing `select!` on the cancellation token. A Byzantine peer below the BFT fault threshold that advertises an artifact and never overwrites that slot keeps the `PeerCounter` non-zero indefinitely, blocking `wait_for` forever, preventing `UnvalidatedArtifactMutation::Remove` from being emitted, and permanently leaking both the unvalidated pool entry and the spawned tokio task.

## Finding Description
`process_slot_update` constructs a three-arm `select!` at line 480: cancellation, `assemble_artifact`, and `all_peers_deleted_artifact`. When `assemble_artifact` wins, the cancellation arm is dropped. Execution then proceeds sequentially to line 500:

```rust
// wait for deletion from peers
// TODO: NET-1774
let _ = peer_rx.wait_for(|p| p.is_empty()).await;
```

This `wait_for` is not wrapped in any new `select!` with the cancellation token. `peer_rx` is a `watch::Receiver<PeerCounter>`. `wait_for` returns `Err` only when the corresponding `watch::Sender` is dropped. That sender lives in `active_assembles` (line 188) and is removed only inside `handle_artifact_processor_joined` at line 318 — which is called only when the task finishes. This creates a circular dependency: the task cannot finish until `wait_for` returns; `wait_for` returns only when the counter reaches zero or the sender is dropped; the sender is dropped only when the task finishes.

The `PeerCounter` is decremented in exactly two places: a slot overwrite from the same peer (line 427, `sender.send_if_modified(|h| h.remove(peer_id))`) and a topology update removing the peer (line 562, `peers_sender.send_if_modified(...)`). A Byzantine peer that stays connected and never sends another slot update for the occupied slot triggers neither path. The slot limit at line 373 (`peer_slot_table_len < self.slot_limit`) bounds per-peer impact but does not prevent the deadlock.

## Impact Explanation
Each stuck task holds a live tokio task and a `watch::Receiver`, and the corresponding artifact remains permanently in the unvalidated pool because `UnvalidatedArtifactMutation::Remove` is never sent. With `f` Byzantine peers each filling `slot_limit` slots, the total leaked artifacts scale as `f × slot_limit`. Large artifacts (consensus blocks, state-sync chunks) amplify memory pressure. Sustained accumulation can exhaust heap memory on the replica node, causing it to crash and drop out of consensus, degrading subnet liveness. This matches the allowed impact: **High — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
The attack requires only a single Byzantine replica below the fault threshold to connect (normal protocol behavior), send one slot update per slot up to `slot_limit`, and then go silent on those slots. No cryptographic material, admin access, or majority corruption is needed. The `// TODO: NET-1774` comments at lines 499 and 520 are explicit developer acknowledgements of the deficiency. The attack is trivially repeatable and persistent for the lifetime of the node process.

## Recommendation
Wrap both post-assembly `wait_for` calls in a `select!` that also listens to the cancellation token:

```rust
select! {
    _ = cancellation_token.cancelled() => { return (peer_rx, id_c); }
    result = peer_rx.wait_for(|p| p.is_empty()) => { let _ = result; }
}
```

Additionally, resolve the referenced `NET-1774` ticket and consider adding a bounded timeout so that even a non-cancelled stuck task eventually unblocks and emits `Remove`.

## Proof of Concept
Using the existing test harness pattern in the file:

1. Create a `ConsensusManagerReceiver` with a mock assembler that returns `AssembleResult::Done` immediately (matching the pattern in `overwrite_slot_send_remove`).
2. Call `handle_slot_update_receive` for `NODE_1`, slot 1, artifact ID 0.
3. Assert `UnvalidatedArtifactMutation::Insert` arrives on the channel.
4. Do **not** send any further slot update (no overwrite, no topology change removing `NODE_1`).
5. Assert that `UnvalidatedArtifactMutation::Remove` is **never** received within a 5-second timeout.
6. Assert `artifact_processor_tasks.len() == 1` (task is alive and blocked at line 500).

This is the inverse of the existing `overwrite_slot_send_remove` test, which demonstrates `Remove` is emitted only when a slot overwrite occurs — the no-overwrite case is the exploit scenario and is currently unguarded.
