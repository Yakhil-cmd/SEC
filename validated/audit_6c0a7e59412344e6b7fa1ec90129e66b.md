### Title
Gas Prepayment Permanently Lost When `gas_limit > u64::MAX` Due to Missing Refund on `GasOverflow` — (`engine/src/engine.rs`)

---

### Summary

In `submit_with_alt_modexp`, `charge_gas` is called **before** `transaction.gas_limit.try_into::<u64>()`. When a legacy transaction carries a `gas_limit` that is a `U256` value strictly greater than `u64::MAX` but small enough that `gas_limit * gas_price` does not overflow `U256`, `charge_gas` successfully deducts the full prepaid amount from the sender's balance. The subsequent `try_into::<u64>()` conversion then fails, returning `EngineErrorKind::GasOverflow` via an early `?`-return that completely bypasses the `refund_unused_gas` call. The deducted ETH is permanently removed from the sender's balance with no recovery path.

---

### Finding Description

The execution sequence inside `submit_with_alt_modexp` is:

1. Parse transaction — `gas_limit` is stored as raw `U256`, no upper-bound check.
2. `check_nonce` — passes.
3. Intrinsic-gas check at line 1079:
   ```rust
   if transaction.gas_limit < core::cmp::max(intrinsic_gas, floor_gas).into() {
   ```
   Both `intrinsic_gas` and `floor_gas` are `u64`; their `.into()` produces a `U256` ≤ `u64::MAX`. A `gas_limit` of `u64::MAX + 1` is larger, so the condition is `false` and execution continues. [1](#0-0) 

4. `charge_gas` at line 1101 — uses `transaction.gas_limit` as a raw `U256` in the multiplication:
   ```rust
   let prepaid_amount = fixed_gas
       .map_or(transaction.gas_limit, EthGas::as_u256)
       .checked_mul(effective_gas_price)
       .map(Wei::new)
       .ok_or(GasPaymentError::EthAmountOverflow)?;
   ```
   With `gas_limit = u64::MAX + 1` and `gas_price = 1`, `checked_mul` returns `u64::MAX + 1` which fits in `U256`. `set_balance` then writes the reduced balance to storage. [2](#0-1) [3](#0-2) 

5. `gas_limit.try_into::<u64>()` at line 1107 — **fails** because `u64::MAX + 1 > u64::MAX`, returns `Err(EngineErrorKind::GasOverflow)` immediately via `?`. [4](#0-3) 

6. `refund_unused_gas` at line 1148 — **never reached**. The prepaid ETH is gone. [5](#0-4) 

`NormalizedEthTransaction::try_from` for a `Legacy` transaction copies `gas_limit` verbatim from the RLP-decoded `U256` field with no bounds check, so any value up to `U256::MAX` is accepted. [6](#0-5) 

---

### Impact Explanation

The prepaid ETH (`gas_limit * effective_gas_price` wei) is subtracted from the sender's balance via `set_balance` and is never credited to any address — not to the relayer, not back to the sender. The funds are permanently frozen inside the engine contract with no administrative recovery path. This satisfies the **Critical — Permanent freezing of funds** impact criterion.

---

### Likelihood Explanation

The attack is reachable through the standard `submit` / `submit_with_args` public NEAR contract methods, which are the primary EVM transaction entry points. No admin access, key compromise, or external dependency failure is required. The attacker only needs:

- A funded Aurora address with at least `(u64::MAX + 1) * gas_price` wei.
- The ability to craft and sign a valid legacy RLP transaction with an oversized `gas_limit`.

Both conditions are trivially satisfiable. The RLP decoder accepts any `U256` for the `gas` field. [7](#0-6) 

---

### Recommendation

Move the `gas_limit.try_into::<u64>()` conversion to **before** the `charge_gas` call, so that an oversized gas limit is rejected before any balance mutation occurs:

```rust
// Validate gas_limit fits in u64 BEFORE charging
let gas_limit: u64 = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;

// Now charge gas (safe: gas_limit is a valid u64)
let prepaid_amount = match engine.charge_gas(&sender, &transaction, max_gas_price, fixed_gas) { ... };
```

This ensures the invariant: *any ETH deducted during gas prepayment is either consumed or refunded*.

### Proof of Concept

```rust
// Construct a legacy transaction with gas_limit = u64::MAX + 1, gas_price = 1
let gas_limit = U256::from(u64::MAX) + U256::one(); // 2^64
let gas_price = U256::one();

let tx = TransactionLegacy {
    nonce: U256::zero(),
    gas_price,
    gas_limit,
    to: Some(some_address),
    value: Wei::zero(),
    data: vec![],
};

// Fund sender with exactly gas_limit * gas_price = 2^64 wei
add_balance(&mut io, &sender, Wei::new(gas_limit));

// Submit the signed transaction
let result = submit_with_alt_modexp(io, env, &args, state, account_id, relayer, &mut handler);

// Assert: error is GasOverflow
assert!(matches!(result.unwrap_err().kind, EngineErrorKind::GasOverflow));

// Assert: sender balance is now 0 (funds deducted, never refunded)
assert_eq!(get_balance(&io, &sender), Wei::zero());

// Assert: relayer balance is still 0 (no reward paid either)
assert_eq!(get_balance(&io, &relayer), Wei::zero());
```

The sender's `2^64` wei is permanently lost. No refund event occurs. The error kind is `GasOverflow`.
