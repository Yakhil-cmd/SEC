Audit Report

## Title
Hot Keys Not Cleared on NNS Neuron Ownership Transfer via `spawn_neuron` - (File: `rs/nns/governance/src/governance.rs`)

## Summary

When `spawn_neuron` is called with a `new_controller` field assigning the spawned neuron to a different principal, the child neuron unconditionally inherits the parent neuron's full `hot_keys` list with no guard on controller change. Because hot keys are authorized to vote and set following on behalf of a neuron, the original owner's hot keys retain unauthorized governance influence over a neuron they no longer control, without the new owner's knowledge or consent.

## Finding Description

In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` from the optional `spawn.new_controller` field (lines 2651–2655). When `new_controller` is set, `child_controller` differs from `parent_neuron.controller()`. However, the child neuron is then constructed at lines 2704–2718 with `.with_hot_keys(parent_neuron.hot_keys.clone())` unconditionally — there is no branch checking whether `child_controller == parent_neuron.controller()` before copying hot keys.

The authorization model in `rs/nns/governance/src/neuron/types.rs` line 255 defines `is_hotkey_or_controller` as `self.is_controlled_by(principal) || self.hot_keys.contains(principal)`. This is used by `is_authorized_to_vote` (line 243–245), which gates both `register_vote` (lines 5594–5602) and `follow` (lines 5725–5748). A hot key holder can therefore cast votes and set following rules on any neuron whose `hot_keys` list contains their principal.

No existing guard in `spawn_neuron` clears or conditionally omits hot keys when the controller changes. A grep for any such guard (`child_controller.*parent.*controller`, `hot_keys.*clear`, `hot_keys.*empty`, `spawn.*hot_key`) returns no matches in `governance.rs`.

The `Spawn` message in `rs/nns/governance/proto/ic_nns_governance/pb/v1/governance.proto` lines 862–870 explicitly documents `new_controller` as the mechanism for cross-principal spawning, confirming this is an intended, reachable code path.

**Exploit flow:**
1. Attacker holds NNS neuron `N` with sufficient maturity and registers a malicious hot key `H` on it via `AddHotKey`.
2. Attacker calls `manage_neuron` → `Spawn { new_controller: Some(victim), ... }`.
3. `spawn_neuron` executes: `child_controller = victim`, but `.with_hot_keys(parent_neuron.hot_keys.clone())` copies `[H]` into the child neuron. The child neuron is stored with `controller = victim` and `hot_keys = [H]`.
4. `H` calls `manage_neuron` → `RegisterVote` or `Follow` targeting the child neuron. Governance authorizes `H` because `hot_keys.contains(H)` is true.
5. The victim's neuron votes or follows according to the attacker's direction. The victim sees `hot_keys: [H]` only if they proactively call `get_full_neuron`.

Existing checks are insufficient: `spawn_neuron` validates that the caller is the parent controller (line 2631–2633) and that the child controller is self-authenticating (line 2720 comment), but neither check addresses hot key inheritance on controller change.

## Impact Explanation

This is an unauthorized governance authorization bug affecting NNS. An attacker can cause a neuron they no longer own to vote on NNS governance proposals or change its following configuration without the new owner's consent. NNS voting power is proportional to stake × dissolve delay × age bonus; a spawned neuron with significant converted maturity carries real voting weight. This constitutes unauthorized access to a governance asset (the victim's neuron's voting power) and concrete NNS governance harm.

This matches the High impact tier: **"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints"** ($2,000–$10,000). The constraint is that the attacker must sacrifice their own maturity to fund the spawned neuron, and must know the victim's principal.

## Likelihood Explanation

- `spawn_neuron` with `new_controller` is a documented, unprivileged ingress path callable by any NNS neuron controller with sufficient maturity.
- The attacker controls the entire setup: they choose when to add hot keys and when to spawn.
- The victim receives the neuron without any acceptance step or notification; the neuron simply appears under their principal.
- The victim has no automatic defense; they must proactively inspect `hot_keys` via `get_full_neuron`.
- The attack is repeatable against multiple victims from a single parent neuron (by spawning multiple times with different `new_controller` values and different `nonce` values), limited only by available maturity.
- The attacker does incur a cost (loss of maturity), which constrains likelihood to **Medium-High**.

## Recommendation

In `spawn_neuron`, when `child_controller != parent_neuron.controller()`, initialize the child neuron with an empty hot-keys list:

```rust
let hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};

let child_neuron = NeuronBuilder::new(...)
    .with_hot_keys(hot_keys)
    ...
```

Additionally, review `split_neuron` (lines 2241–2257) for the same `.with_hot_keys(parent_neuron.hot_keys.clone())` pattern. While `split_neuron` always uses `*caller` as the child controller (same principal), the pattern should be audited for consistency and future-proofing.

## Proof of Concept

**Minimal integration test using `PocketIC` or the existing `GovernanceCanisterFixture`:**

1. Create governance with two principals: `alice` (attacker) and `charlie` (victim).
2. Create neuron `N` controlled by `alice` with maturity above `neuron_minimum_spawn_stake_e8s`.
3. Call `manage_neuron(alice, AddHotKey { new_hot_key: bob })` on neuron `N`.
4. Call `manage_neuron(alice, Spawn { new_controller: Some(charlie), nonce: Some(1), percentage_to_spawn: Some(100) })`.
5. Assert the returned child neuron ID exists and `child_neuron.controller == charlie`.
6. Assert `child_neuron.hot_keys.contains(bob)` — **this will pass, demonstrating the bug**.
7. Submit an open NNS proposal; call `manage_neuron(bob, RegisterVote { proposal: ..., vote: Yes })` targeting the child neuron ID.
8. Assert the vote succeeds — **this will pass, demonstrating unauthorized governance influence**.

The root cause line is `rs/nns/governance/src/governance.rs:2714` — `.with_hot_keys(parent_neuron.hot_keys.clone())` inside `spawn_neuron` with no guard on `child_controller == parent_neuron.controller()`.
