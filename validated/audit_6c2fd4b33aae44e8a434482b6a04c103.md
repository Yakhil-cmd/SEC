### Title
`native_per_pubdata` Truncates to Zero When `pubdata_price < native_price`, Allowing Transactions to Write Pubdata for Free — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` value is computed via truncating integer division (`pubdata_price.wrapping_div(native_price)`). When `pubdata_price < native_price`, the result is zero. This zero propagates through the entire pubdata fee pipeline, causing all pubdata costs to be silently waived. Any transaction sender operating in a block where this condition holds can write the maximum allowed pubdata without paying any native resource cost for it.

---

### Finding Description

In `validation_impl.rs` line 142, the per-byte pubdata cost in native resources is:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This is a truncating integer division. When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `0`.

The zero value propagates through three critical accounting sites:

**1. Intrinsic pubdata overhead in `create_resources_for_tx` (`gas_helpers.rs:352`):**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
```
No native resources are deducted for intrinsic pubdata, giving the transaction a larger native budget than intended.

**2. Execution pubdata cost in `get_resources_to_charge_for_pubdata` (`gas_helpers.rs:430-432`):**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // N * 0 = 0
    .ok_or(out_of_native_resources!())?;
```
The pubdata cost is always zero regardless of how much pubdata the transaction writes.

**3. Post-execution pubdata check in `check_enough_resources_for_pubdata` (`gas_helpers.rs:454`):**
```rust
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata = 0, so always true
```
The post-execution pubdata check always passes; transactions are never reverted for insufficient pubdata payment.

The identical truncating division also appears in the off-chain validation helper at `api/src/helpers.rs:427`:
```rust

### Impact Explanation

When `pubdata_price < native_price`, any transaction sender can write the maximum allowed pubdata (bounded only by the block's `pubdata_limit`) without paying any native resources for it. The operator must still pay L1 data-availability costs for this pubdata but collects zero pubdata fees from users. This creates a direct financial loss for the operator and enables pubdata-spam attacks that exhaust the block's pubdata budget at minimal cost to the attacker. Because `native_per_pubdata = 0` also zeroes out the intrinsic pubdata overhead deducted at transaction setup, the transaction additionally receives a larger computational native budget than it is entitled to.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` can arise in normal operation when the native token price is high relative to the cost of L1 data. Notably, the default test-rig `BlockContext` sets `pubdata_price: U256::default()` (zero), which always triggers this condition. In production, any block where the operator sets `pubdata_price < native_price` exposes every transaction in that block to this issue. The attacker does not need to control the block context; they only need to submit transactions while the condition holds. [6](#0-5) 

---

### Recommendation

Replace the truncating division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is already computed:

```rust
// native_per_gas already uses div_ceil:
u256_try_to_u64(&gas_price.div_ceil(native_price))

// native_per_pubdata should do the same:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Alternatively, add an explicit guard: if `pubdata_price > 0` but `native_per_pubdata == 0` after division, clamp to `1` to prevent complete fee bypass. [7](#0-6) 

---

### Proof of Concept

1. Set block context: `pubdata_price = 5`, `native_price = 10`.
2. `native_per_pubdata = 5 / 10 = 0` (truncated to zero).
3. Submit a transaction that writes 10,000 bytes of pubdata (e.g., ~312 SSTORE operations to new storage slots).
4. `get_resources_to_charge_for_pubdata` computes `10000 * 0 = 0` native resources.
5. `check_enough_resources_for_pubdata` returns `true` (0 ≤ any remaining balance).
6. Transaction succeeds, paying zero native resources for 10,000 bytes of pubdata.
7. Operator pays L1 data-availability costs for 10,000 bytes without collecting any pubdata fees from the sender.
8. Repeat across the full `pubdata_limit` of the block to maximally drain operator revenue at minimal cost.
