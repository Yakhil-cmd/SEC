Audit Report

## Title
Unauthenticated Caller Can Force SNS Swap Participation and Delete Tickets on Behalf of Any Principal via `refresh_buyer_tokens` - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method on the SNS Swap canister accepts an arbitrary `buyer` principal string from any caller with no authorization check verifying the caller equals the specified buyer. This allows any unprivileged ingress sender to irreversibly delete another principal's open sale ticket and force-register their swap participation — including supplying the (publicly known) confirmation text — without the victim's explicit consent.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 128–143, the canister handler resolves the buyer principal as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // attacker-controlled, no check
};
```

No check is performed to verify `p == caller_principal_id()`. The resolved `p` is passed directly to `refresh_buyer_token_e8s`.

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` lines 1248–1270), the function:
1. Looks up the ICP ledger balance of the **victim's** subaccount (not the caller's).
2. If the balance satisfies the ticket amount, **irreversibly deletes the victim's open ticket** from `OPEN_TICKETS_MEMORY`.
3. Creates or updates a `BuyerState` entry for the victim, registering their participation.

The ticket deletion is keyed on the victim's principal and is permanent:
```rust
memory::OPEN_TICKETS_MEMORY.with(|m| m.borrow_mut().remove(&principal));
```

The `confirmation_text` required by some SNS swaps is public (stored in swap init parameters), so an attacker trivially supplies the correct text. No existing check in the call path verifies caller identity against the buyer argument.

## Impact Explanation
This is a **High** severity finding matching the allowed impact class: *"Significant SNS security impact with concrete user or protocol harm."*

1. **Irreversible ticket deletion griefing**: A victim creates a sale ticket via `new_sale_ticket` (caller-authenticated), transfers ICP to their swap subaccount, and an attacker calls `refresh_buyer_tokens` with the victim's principal. The ticket is permanently deleted from `OPEN_TICKETS_MEMORY`, disrupting the ticketing payment flow with no recovery path within the swap round.

2. **Forced participation bypassing explicit consent**: The confirmation text mechanism is designed to ensure users explicitly agree to swap terms before participation is registered. An attacker can supply the correct (public) confirmation text and register the victim's participation without the victim ever taking that action, undermining the consent mechanism.

3. **Swap outcome manipulation**: By force-registering many principals who have ICP in their subaccounts, an attacker can artificially advance the direct participation total, potentially triggering premature swap commitment or affecting Neurons' Fund matched funding calculations.

## Likelihood Explanation
- `refresh_buyer_tokens` is a publicly callable `#[update]` method; any principal can call it.
- The `buyer` field is a plain Candid string, trivially set to any target principal's text representation.
- A victim's principal ID is publicly observable on-chain; their swap subaccount balance is queryable via the ICP ledger.
- The attack requires zero ICP from the attacker and no special privileges.
- The confirmation text is public in swap init parameters, removing any meaningful barrier.
- The attack is repeatable across any number of victims during the swap's open lifecycle.

## Recommendation
Enforce that the resolved buyer principal equals the actual caller. Either remove the `buyer` field and always use `caller_principal_id()`, or add an explicit identity check:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, requested, "Caller may only refresh their own buyer tokens");
        requested
    };
    ...
}
```

If third-party notification (e.g., by the NNS dapp) is a required use case, restrict the `buyer` override to a whitelist of trusted canister principals (e.g., NNS governance, NNS dapp canister IDs).

## Proof of Concept
**Preconditions:**
1. SNS swap is in `Open` lifecycle state.
2. Victim calls `new_sale_ticket(amount_icp_e8s: X)` → ticket stored in `OPEN_TICKETS_MEMORY[victim]`.
3. Victim transfers `X` ICP to `swap_canister_subaccount(victim)` on the ICP ledger.

**Attack:**
4. Attacker (any principal) calls:
   ```
   refresh_buyer_tokens({ buyer: "<victim_principal>", confirmation_text: <public_swap_text> })
   ```

**Result:**
5. Inside `refresh_buyer_token_e8s`: ledger balance of victim's subaccount = X ≥ `min_participant_icp_e8s` → passes minimum check.
6. Ticket found: `amount_ticket = X ≤ requested_increment_e8s = X` → ticket deleted from `OPEN_TICKETS_MEMORY`.
7. `BuyerState` for victim set to X e8s; participation registered with attacker-supplied confirmation text.
8. Victim's ticket is permanently destroyed. Victim's participation is registered without their explicit action or consent.

**Reproducible test**: A PocketIC or `StateMachine` integration test (analogous to those in `rs/sns/integration_tests/src/payment_flow.rs`) can confirm this by: (a) creating a ticket as user A, (b) transferring ICP as user A, (c) calling `refresh_buyer_tokens` as user B with `buyer = user_A.to_string()`, and (d) asserting the ticket is deleted and `BuyerState` is set for user A.
