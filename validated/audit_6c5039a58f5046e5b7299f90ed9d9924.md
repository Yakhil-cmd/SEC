### Title
Heap-Only `API_BOUNDARY_NODE_PRINCIPALS` State Lost on Every Upgrade Causes Post-Upgrade Authorization Blackout - (File: `rs/boundary_node/salt_sharing/canister/canister.rs`, `rs/boundary_node/salt_sharing/canister/storage.rs`)

---

### Summary

The salt-sharing canister stores its authorized-reader set (`API_BOUNDARY_NODE_PRINCIPALS`) exclusively in heap memory (`thread_local!`). Because the IC wipes the heap on every upgrade, and because `post_upgrade` only schedules an async timer to repopulate this set (not an immediate synchronous restore), there is a guaranteed window after every canister upgrade during which every API boundary node is unauthorized and `get_salt` returns `GetSaltError::Unauthorized` for all callers.

---

### Finding Description

`API_BOUNDARY_NODE_PRINCIPALS` is declared as a `thread_local!` `HashSet<Principal>` in heap memory: [1](#0-0) 

This is **not** backed by stable memory. On every canister upgrade the IC discards the heap, so this set is reset to an empty `HashSet::new()`.

The `post_upgrade` hook simply delegates to `init`: [2](#0-1) 

`init` schedules `init_async` as a zero-delay timer (not a synchronous call): [3](#0-2) 

Inside `init_async`, the registry poll that repopulates `API_BOUNDARY_NODE_PRINCIPALS` is registered as a **periodic interval timer**, meaning the first execution fires only after `registry_polling_interval_secs` seconds have elapsed: [4](#0-3) 

The production upgrade proposal uses `registry_polling_interval_secs = 300` (5 minutes). During this entire window, `API_BOUNDARY_NODE_PRINCIPALS` is empty.

Both the query path and the update path enforce authorization against this empty set: [5](#0-4) [6](#0-5) 

The `inspect_message` hook rejects all update calls to `get_salt` that are not from a known API boundary node principal. Since the set is empty post-upgrade, every update call is rejected before consensus. The query path returns `GetSaltError::Unauthorized` for the same reason.

The salt value itself is stored in stable memory via `StableBTreeMap` and survives the upgrade correctly: [7](#0-6) 

So the salt exists but is completely inaccessible to all legitimate callers for up to 5 minutes after every upgrade.

---

### Impact Explanation

Every upgrade of the salt-sharing canister causes a complete service blackout for all API boundary nodes for the duration of `registry_polling_interval_secs` (5 minutes in production). During this window:

- All `get_salt` query calls return `GetSaltError::Unauthorized`.
- All `get_salt` update calls are rejected at the `inspect_message` phase.
- The `AnonymizationSaltFetcher` running on each boundary node logs warnings and retains its last cached salt value, but any boundary node that restarted during this window starts with `None` and cannot obtain a salt until the canister recovers. [8](#0-7) 

The anonymization salt is used to hash sensitive user data (e.g., IP addresses) before logging. A boundary node with no salt cannot perform this anonymization, degrading privacy guarantees for users whose requests are handled by that node during the outage window.

---

### Likelihood Explanation

This is triggered deterministically by every routine canister upgrade — a normal, expected governance operation. No adversarial action is required beyond submitting a standard NNS upgrade proposal for the salt-sharing canister. The production upgrade proposal already demonstrates `registry_polling_interval_secs = 300`, confirming the 5-minute blackout window is reproducible on mainnet on every upgrade. [9](#0-8) 

---

### Recommendation

Persist `API_BOUNDARY_NODE_PRINCIPALS` in stable memory (e.g., using a `StableBTreeMap` backed by the `MemoryManager`, analogous to how `SALT` is stored). In `post_upgrade`, restore the set synchronously from stable memory before any message is processed, eliminating the authorization gap entirely. Alternatively, trigger an immediate `poll_api_boundary_nodes()` call (via `set_timer(Duration::ZERO, ...)`) in addition to the periodic interval, so the set is repopulated in the first round after upgrade rather than after the full polling interval.

---

### Proof of Concept

1. Deploy the salt-sharing canister with `regenerate_now: true`, `registry_polling_interval_secs: 300`.
2. Confirm an API boundary node principal can call `get_salt` successfully.
3. Submit an NNS upgrade proposal for the canister with `regenerate_now: false`, `registry_polling_interval_secs: 300`.
4. Immediately after the upgrade executes, call `get_salt` from the same API boundary node principal.
5. Observe `GetSaltError::Unauthorized` is returned — the principal is no longer in `API_BOUNDARY_NODE_PRINCIPALS` because the heap was wiped and the registry poll has not yet fired.
6. Wait 300+ seconds and confirm the call succeeds again once `poll_api_boundary_nodes` completes.

The integration test confirms the upgrade path exists and uses `regenerate_now: false`: [10](#0-9)

fn get_salt() -> GetSaltResponse {
    let caller_id = caller();
    if is_api_boundary_node_principal(&caller_id) {
        let stored_salt = SALT
            .with(|cell| cell.borrow().get(&()))
            .ok_or(GetSaltError::SaltNotInitialized)?;

        return Ok(SaltResponse {
            salt: stored_salt.salt,
            salt_id: stored_salt.salt_id,
        });
    }
    Err(GetSaltError::Unauthorized)
}

pub async fn init_async(init_arg: InitArg) {
    if (!is_salt_init() || init_arg.regenerate_now)
        && let Err(err) = try_regenerate_salt().await
    {
        log!(P0, "[init_regenerate_salt_failed]: {err}");
    }
    // Start salt generation schedule based on the argument.
    match init_arg.salt_generation_strategy {
        SaltGenerationStrategy::StartOfMonth => schedule_monthly_salt_generation(),
    }
    // Set up periodical job to get all API boundary node IDs from the registry.
    let period = Duration::from_secs(init_arg.registry_polling_interval_secs);
    set_timer_interval(period, async || poll_api_boundary_nodes().await);
}
