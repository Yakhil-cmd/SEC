Audit Report

## Title
SNS Governance `perform_execute_generic_nervous_system_function` Lacks Re-entrancy Guard During Inter-Canister Proposal Execution - (File: `rs/sns/governance/src/canister_control.rs`)

## Summary
`perform_execute_generic_nervous_system_function_call` makes an unguarded `await` to an arbitrary community-registered external canister. While governance is suspended at that await, the IC runtime can deliver and fully process other queued messages, including `manage_neuron` calls. A malicious target canister can exploit this by calling back into governance's `manage_neuron` endpoint to cast a deciding vote on a second pending proposal, triggering its execution â€” including a second call to the same target canister â€” before the first proposal's execution has completed.

## Finding Description

**Root cause â€” no re-entrancy guard on the inter-canister call:**

`perform_execute_generic_nervous_system_function_call` in `rs/sns/governance/src/canister_control.rs` makes an unguarded `await` to an arbitrary external canister with no in-progress flag or lock:

```rust
let result = env
    .call_canister(
        valid_function.target_canister_id,
        &valid_function.target_method,
        call.payload,
    )
    .await;  // governance is suspended here; other messages are processed
```

Its caller `perform_execute_generic_nervous_system_function` in `rs/sns/governance/src/governance.rs` sets no in-progress flag before delegating to this function. No `executing_proposals` set, no per-function-id lock, and no global execution guard exists for `ExecuteGenericNervousSystemFunction` actions. The `in_flight_commands` mechanism (visible in `rs/sns/governance/tests/interleaving_tests.rs`) only protects neuron-level operations, not proposal execution.

**Re-entrancy path through `manage_neuron` â†’ `register_vote` â†’ `process_proposal` â†’ `start_proposal_execution`:**

The public `#[update]` endpoint `manage_neuron` in `rs/sns/governance/canister/canister.rs` is reachable by any canister. `register_vote` synchronously calls `process_proposal` at line 3944. `process_proposal` checks only `proposal_data.status() != ProposalDecisionStatus::Open` â€” proposal B is still `Open` at attack time, so this check does not block execution. `process_proposal` then calls `start_proposal_execution`, which uses `unsafe { std::mem::transmute(self) }` to obtain a `'static` reference and calls `spawn_in_canister_env(governance.perform_action(proposal_id, action))`. The spawned future runs and issues a second `call_canister` to the same target canister before the first call has returned.

**Concrete execution sequence:**

1. Proposal A (`ExecuteGenericNervousSystemFunction(id=1000)`) is adopted; governance calls canister X and suspends at the `await`.
2. Canister X, during its execution, sends `manage_neuron { RegisterVote { proposal: B, vote: Yes } }` to governance.
3. Governance processes the `manage_neuron` message: `register_vote` â†’ `process_proposal(B)` â†’ B reaches quorum â†’ `start_proposal_execution(B)` â†’ `spawn_in_canister_env(perform_action(B))`.
4. The spawned future runs: governance calls canister X again for proposal B and suspends.
5. Governance replies to canister X's `manage_neuron` call; `manage_neuron` processing completes.
6. Canister X receives governance's call for proposal B, processes it, and replies.
7. Governance resumes proposal B's execution â†’ marks B `Executed`.
8. Canister X replies to governance's original call for proposal A.
9. Governance resumes proposal A's execution â†’ marks A `Executed`.

Proposal B has executed inside proposal A's execution window, with canister X receiving both calls in an order it fully controls.

## Impact Explanation

This is a **High** severity finding. The concrete impact is a significant SNS governance security violation: an external callee can re-enter the governance execution path and cause two governance-authorized `ExecuteGenericNervousSystemFunction` calls to be delivered to the same target canister in an attacker-controlled order. If the target canister controls treasury funds, token minting, or permission grants, each call may produce side effects (e.g., token minting, fund transfers, permission escalation) that are doubled or reordered. This constitutes concrete user and protocol harm within the SNS governance framework, matching the "Significant SNS security impact with concrete user or protocol harm" High-severity impact class.

## Likelihood Explanation

The attack requires: (1) a canister registered as a `NervousSystemFunction` target (requires a prior governance proposal to pass, but is common in SNS launches where dapp canisters are registered); (2) the attacker controls a neuron with deciding voting power on a second proposal (developer neurons with significant voting power are standard in SNS DAOs); (3) a second proposal pre-voted to threshold minus one vote (achievable by the attacker if they control enough neurons). No privileged role, key compromise, or subnet-majority is required. The attack is repeatable as long as the conditions hold.

## Recommendation

1. **Add a per-function-id execution lock**: Before the `await` in `perform_execute_generic_nervous_system_function`, insert the `function_id` into a `BTreeSet<u64>` stored in governance state; remove it after the `await` completes (success or failure). Reject any concurrent `ExecuteGenericNervousSystemFunction` call for the same `function_id` with `ErrorType::ResourceExhausted`.

2. **Alternatively, add a global execution-in-progress flag** (analogous to `finalize_swap_in_progress` in the SNS Swap canister) that blocks new `ExecuteGenericNervousSystemFunction` proposal executions while any such execution is awaiting a reply.

3. **At minimum, document** that target canisters must not call back into governance's `manage_neuron` during execution, and that SNS DAOs should not register canisters holding voting neurons as `GenericNervousSystemFunction` targets.

## Proof of Concept

**Setup:**
- SNS has `NervousSystemFunction` id=1000 targeting canister X.
- Canister X controls neuron N with deciding voting power.
- Proposal A: `ExecuteGenericNervousSystemFunction { function_id: 1000, payload: b"action_a" }` â€” adopted, awaiting execution.
- Proposal B: `ExecuteGenericNervousSystemFunction { function_id: 1000, payload: b"action_b" }` â€” open, all neurons except N have voted Yes.

**Attack steps:**
1. Governance heartbeat triggers `process_proposals` â†’ proposal A adopted â†’ `start_proposal_execution(A)` â†’ `perform_execute_generic_nervous_system_function` â†’ `env.call_canister(X, "execute", b"action_a")` â†’ governance suspends.
2. Canister X's `execute` method receives `b"action_a"`. Before replying, it issues `manage_neuron { subaccount: N.id, command: RegisterVote { proposal: B, vote: Yes } }` to governance.
3. Governance processes `manage_neuron`: `register_vote` casts N's vote on B â†’ `process_proposal(B)` â†’ B reaches quorum â†’ `start_proposal_execution(B)` â†’ `spawn_in_canister_env(perform_action(B))` â†’ spawned future runs â†’ `env.call_canister(X, "execute", b"action_b")` â†’ governance suspends.
4. Governance replies to canister X's `manage_neuron` call.
5. Canister X receives governance's call for proposal B (`b"action_b"`), processes it, replies.
6. Governance resumes proposal B â†’ `set_proposal_execution_status(B, Ok)` â†’ B marked `Executed`.
7. Canister X replies to governance's original call for proposal A.
8. Governance resumes proposal A â†’ `set_proposal_execution_status(A, Ok)` â†’ A marked `Executed`.

**Reproducible test plan:** Write a PocketIC integration test with a mock canister X that, upon receiving `b"action_a"`, issues a `manage_neuron / RegisterVote` call to governance for proposal B before replying. Assert that proposal B's execution (receipt of `b"action_b"` by canister X) occurs before proposal A's execution completes, demonstrating the ordering violation.
