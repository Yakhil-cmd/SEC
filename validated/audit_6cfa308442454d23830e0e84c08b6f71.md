Audit Report

## Title
Single `NetworkMessage::Unknown` Permanently Removes Peer Address, Enabling Address Book Depletion — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary

`process_unknown_message` unconditionally calls `internal_discard` on any peer that sends a single unknown Bitcoin P2P message. This triggers permanent removal of that peer's address from the address book when DNS seeds are configured (the normal production deployment), with no rate limiting, counter, or grace period. An attacker operating enough Bitcoin-protocol-speaking nodes can deplete the adapter's address book and force continuous re-discovery from DNS seeds, disrupting the IC's Bitcoin integration (ckBTC, canister Bitcoin API).

## Finding Description

Every step of the call chain is confirmed in production code:

**Step 1 — Router dispatch:** The router passes every received `NetworkMessage::Unknown` to `connection_manager.process_bitcoin_network_message`. Because `process_unknown_message` returns `Ok(())`, the outer `discard` guard in the router is never triggered — the discard happens entirely inside the handler. [1](#0-0) 

**Step 2 — Message dispatch:** `process_bitcoin_network_message` matches `Unknown` with no prior state check and calls `process_unknown_message`. [2](#0-1) 

**Step 3 — Unconditional discard:** `process_unknown_message` logs the message and unconditionally calls `internal_discard`. There is no counter, threshold, time window, or connection-state guard. [3](#0-2) 

**Step 4 — State transition:** `internal_discard` calls `conn.discard()`, which sets `ConnectionState::AdapterDiscarded` and aborts the task handle. [4](#0-3) [5](#0-4) 

**Step 5 — Reap loop:** On the next `tick()` → `manage_connections()` → `reap_disconnected()`, the `AdapterDiscarded` branch calls `address_book.discard()`. [6](#0-5) 

**Step 6 — Permanent removal:** `address_book.discard()` removes the address from both `active_addresses` and `known_addresses` when `has_seeds()` is true (i.e., DNS seeds are configured, which is the normal production deployment). The address is gone permanently — it is not returned to `known_addresses` as `remove_from_active` would do. [7](#0-6) 

**Existing guards are insufficient:** There is no misbehavior counter, no minimum-address-book floor, and no connection-state check before `process_unknown_message` is invoked. The `NodeDisconnected` path (soft disconnect) returns the address to `known_addresses`; the `AdapterDiscarded` path (triggered here) does not. [8](#0-7) 

**Fallback behavior:** When `known_addresses` is empty, `make_connection` falls back to `resolve_next_seed()`, rebuilding from DNS seeds. This means the adapter is not permanently bricked, but it enters a re-discovery phase during which it may fall below `min_connections = 2` available peers, causing `send_to_all` to silently drop outgoing messages and the adapter to be unable to serve Bitcoin data. [9](#0-8) 

## Impact Explanation

The Bitcoin adapter is the production component backing ckBTC minting/redemption and the canister Bitcoin API. When the adapter cannot maintain `min_connections` (2) `HandshakeComplete` peers, `has_enough_active_connections()` returns false and `send_to_all` silently no-ops, stalling block header and transaction propagation. An attacker who can continuously inject their addresses into the adapter's address book and then trigger the `Unknown` discard path on each connection can sustain this degraded state indefinitely.

This matches the allowed impact: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token infrastructure security impact with concrete user or protocol harm** (ckBTC deposits and withdrawals stall; canister Bitcoin API calls time out or return stale data).

## Likelihood Explanation

Running Bitcoin-protocol-speaking nodes is unprivileged and low-cost. The attacker's addresses enter the adapter's address book via standard `addr` message propagation from any peer the adapter is already connected to. The adapter initiates outbound connections, so the attacker must wait for the adapter to select their address from the book — but with enough nodes, this is probabilistically reliable. No admin access, key material, governance majority, BGP/DNS hijack, or victim mistake is required. The attack is repeatable: after each DNS re-discovery cycle, the attacker re-injects their addresses and repeats.

## Recommendation

1. **Add a misbehavior counter per address.** Only call `internal_discard` after N unknown messages within a time window, not on the first occurrence.
2. **Treat `Unknown` as a soft disconnect.** Call `conn.disconnect()` instead of `conn.discard()`, which transitions to `NodeDisconnected` and returns the address to `known_addresses` via `remove_from_active`, preserving the address book.
3. **Add a minimum address book floor.** In `address_book.discard()`, if `known_addresses.len() + active_addresses.len()` would drop below `min_addresses`, demote to `remove_from_active` instead of permanently removing the entry.

## Proof of Concept

State-machine test (no network required):

1. Create a `ConnectionManager` with DNS seeds configured (so `has_seeds()` returns true).
2. Insert 10 `Connection` objects with `AddressEntry::Discovered` and `ConnectionState::HandshakeComplete` into `manager.connections`; insert their addresses into `address_book.active_addresses` directly (simulating the state after `pop()` was called for each).
3. For each address `addr_i`, call:
   ```rust
   manager.process_bitcoin_network_message(
       addr_i,
       &NetworkMessage::Unknown {
           command: "exploit".parse().unwrap(),
           payload: vec![],
       },
   ).unwrap();
   ```
4. Call `manager.reap_disconnected()`.
5. Assert `manager.address_book.known_addresses.is_empty()` and `manager.address_book.active_addresses.is_empty()` — all 10 addresses have been permanently removed with zero legitimate justification.
