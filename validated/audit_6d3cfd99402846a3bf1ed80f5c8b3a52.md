### Title
Unchecked Precompile Call Return Value in `EvmErc20` Causes Silent Token Burn With No NEAR-Side Credit — (`etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

Both `EvmErc20.withdrawToNear` and `EvmErc20.withdrawToEthereum` execute `_burn` before making a low-level assembly `call` to the exit precompile. The return value `res` of that `call` is declared but **never checked**. When the precompile fails (e.g., due to an invalid NEAR recipient account ID), the EVM sub-call returns 0, the caller's execution continues without reverting, and the user's tokens are permanently destroyed with no corresponding NEP-141 `ft_transfer` or `withdraw` promise scheduled on the NEAR side.

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow the same pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // (1) tokens destroyed

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // (2) res is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) [2](#0-1) 

The `ExitToNear` precompile's `run` function returns `Err(ExitError)` in several reachable conditions, including `ERR_INVALID_RECEIVER_ACCOUNT_ID` when the recipient bytes are not valid UTF-8 or do not form a valid NEAR account ID, and `ERR_INVALID_AMOUNT` when `amount > u128::MAX`. [3](#0-2) 

In `engine-precompiles/src/lib.rs`, `process_precompile` maps every `ExitError` to `PrecompileFailure::Error`:

```rust
p.run(input, gas_limit.map(EthGas::new), context, is_static)
    .map_err(|exit_status| PrecompileFailure::Error { exit_status })
``` [4](#0-3) 

`PrecompileFailure::Error` is the SputnikVM variant that causes the `call` opcode to return 0 to the caller **without propagating the failure up the call stack**. This is distinct from `PrecompileFailure::Fatal` (used only for the paused-precompile case), which does abort the entire transaction. [5](#0-4) 

Because `res` is never inspected, the Solidity function returns normally after the failed sub-call. The `_burn` is not rolled back. No promise log is emitted, so `filter_promises_from_logs` in the engine schedules nothing on the NEAR side. [6](#0-5) 

### Impact Explanation

The invariant that every ERC-20 burn must be matched by a scheduled NEP-141 `ft_transfer` or `withdraw` promise is violated. The user's ERC-20 balance is permanently reduced, no NEAR-side tokens are credited, and there is no recovery path. This constitutes **permanent destruction of user funds** (permanent freezing / direct loss of in-motion funds).

### Likelihood Explanation

Any call to `withdrawToNear` with a recipient that is not a valid NEAR account ID (e.g., empty bytes, non-UTF-8 bytes, a string exceeding 64 characters, or a string containing invalid characters) triggers the failure path. This can happen through:
- A user mistake in a dApp UI
- A smart contract that constructs the recipient incorrectly
- A deliberate call with a crafted invalid recipient (self-inflicted but irreversible)

The `withdrawToEthereum` path is less exposed because the recipient is a fixed-size 20-byte Ethereum address, but the same structural defect exists and could be triggered by other precompile error conditions (e.g., `ERR_TARGET_TOKEN_NOT_FOUND` if storage is in an unexpected state).

### Recommendation

Add an explicit revert on precompile call failure in both functions:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, restructure the functions to call the precompile **before** `_burn`, and only burn if the precompile call succeeds. This is the safer ordering because it avoids the need to roll back state.

### Proof of Concept

1. Deploy a valid `EvmErc20` (registered in the NEP-141 map) on a local Aurora sandbox.
2. Mint tokens to address `A`.
3. From address `A`, call `withdrawToNear(bytes("!!!invalid!!!"), amount)` — `!!!invalid!!!` is not a valid NEAR account ID.
4. Observe: `_burn` fires (balance of `A` decreases), the assembly `call` returns 0 (precompile rejected with `ERR_INVALID_RECEIVER_ACCOUNT_ID`), `res` is not checked, the transaction succeeds.
5. Observe: no `PromiseArgs` log is emitted from the exit precompile address, so no `ft_transfer` promise is scheduled.
6. Result: tokens are permanently destroyed with no NEAR-side credit. [3](#0-2) [1](#0-0)
