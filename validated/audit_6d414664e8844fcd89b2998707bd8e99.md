### Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_start`/`ingress_end` in ICP Rosetta `construction_payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The `construction_payloads` handler in the ICP Rosetta node accepts caller-supplied `ingress_start` and `ingress_end` nanosecond timestamps with no server-side range validation. These values directly control the iteration count of a `while` loop that pushes entries into a `Vec<u64>`. An unauthenticated attacker can supply `ingress_start=0` and `ingress_end=u64::MAX` to force ~153 billion loop iterations, exhausting process memory and crashing the Rosetta node.

---

### Finding Description

`ConstructionPayloadsRequestMetadata` declares both fields as raw `Option<u64>` with no constraints: [1](#0-0) 

In `construction_payloads`, these values are consumed directly without any bounds check: [2](#0-1) 

The `interval` is:

```
MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120_000_000_000 ns
``` [3](#0-2) 

With `ingress_start=0` and `ingress_end=u64::MAX (18_446_744_073_709_551_615)`:

```
iterations ≈ 18_446_744_073_709_551_615 / 120_000_000_000 ≈ 153,722,867,280
memory     ≈ 153 billion × 8 bytes ≈ 1.2 TB
```

The process OOMs before the loop completes. Additionally, if `ic_types::time::Time` addition wraps in release mode, `now` wraps back to a small value and the loop becomes **infinite** since `now < u64::MAX` is always true after wrapping.

**Critical contrast:** The ICRC1 Rosetta implementation of the same endpoint explicitly validates the range and rejects requests where `ingress_end` is not within one interval of the current time, bounding the loop to at most 1–2 iterations: [4](#0-3) 

The ICP Rosetta implementation has **no equivalent guard**.

---

### Impact Explanation

The Rosetta node is a standalone HTTP service. A single malformed POST request to `/construction/payloads` causes the process to exhaust all available memory and crash (OOM kill). This:

- Blocks all ICP ledger operations routed through Rosetta (transfers, neuron management, etc.)
- Can be repeated immediately after restart, making the node permanently unavailable
- Requires no authentication, no privileged role, and no on-chain state

---

### Likelihood Explanation

The endpoint is publicly reachable. The payload is a standard JSON POST body. No cryptographic material, tokens, or prior state is required. The attack is a single HTTP request. Any party aware of the Rosetta API spec can trigger it.

---

### Recommendation

Add an explicit upper bound on `ingress_end - ingress_start` before the loop, mirroring the ICRC1 Rosetta guard. For example:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 60 * 60); // 24h
if ingress_end > ingress_start + MAX_INGRESS_WINDOW.as_nanos() as u64 {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed window",
    ));
}
```

Alternatively, cap `ingress_expiries` with a hard `Vec` size limit (e.g., 720 entries for a 24-hour window at 120s intervals) and return an error if the computed count exceeds it.
