Audit Report

## Title
Delegated `Disburse`/`DisburseMaturity` Permission Defaults Destination to Caller's Account, Enabling Fund Theft - (File: rs/sns/governance/src/governance.rs)

## Summary
In SNS governance, `disburse_neuron` and `disburse_maturity` both resolve the default transfer destination to `Account { owner: caller.0, ... }` when `to_account` is `None`. Because `NeuronPermissionType::Disburse` and `NeuronPermissionType::DisburseMaturity` are independently grantable to arbitrary principals via `AddNeuronPermissions`, a delegated principal can call either function with `to_account: None` and silently redirect the neuron's entire stake or accrued maturity to their own wallet rather than the neuron owner's.

## Finding Description
**Root cause â€” `disburse_neuron`** (`rs/sns/governance/src/governance.rs`, lines 1127 and 1143â€“1147):

Authorization is checked against the caller holding `NeuronPermissionType::Disburse`:
```rust
neuron.check_authorized(caller, NeuronPermissionType::Disburse)?;
```
Then the destination is resolved using `caller`, not the neuron owner:
```rust
let to_account = match disburse.to_account.as_ref() {
    None => Account { owner: caller.0, subaccount: None },
    ...
};
```
The same pattern exists in **`disburse_maturity`** (lines 1616 and 1619â€“1623):
```rust
neuron.check_authorized(caller, NeuronPermissionType::DisburseMaturity)?;
let to_account: Account = match disburse_maturity.to_account.as_ref() {
    None => Account { owner: caller.0, subaccount: None },
    ...
};
```
**Why existing checks are insufficient:** The authorization check only verifies that the caller holds the `Disburse` (or `DisburseMaturity`) permission bit â€” it does not verify that the caller is the neuron owner. Because `neuron_grantable_permissions` in `NervousSystemParameters` can include `NEURON_PERMISSION_TYPE_DISBURSE` and `NEURON_PERMISSION_TYPE_DISBURSE_MATURITY`, any principal holding `ManagePrincipals` can grant these permissions to an arbitrary third party via `add_neuron_permissions`. Once granted, that third party can call with `to_account: None` and receive the funds.

**Contrast with NNS governance** (`rs/nns/governance/src/governance.rs`, lines 1970â€“1978): NNS gates `disburse_neuron` entirely behind `is_controlled_by(caller)`, making caller == neuron owner a hard invariant. SNS deliberately separates these roles, but the default-destination logic was not updated to reflect that separation.

## Impact Explanation
A principal granted `NeuronPermissionType::Disburse` can drain the full cached stake of a dissolved SNS neuron to their own ICRC-1 account via a single `manage_neuron` ingress call with `Command::Disburse { amount: None, to_account: None }`. A principal granted `NeuronPermissionType::DisburseMaturity` can drain all accrued maturity similarly. Both are irreversible on-chain ledger transfers. This constitutes **unauthorized access to neuron/governance assets and theft of user funds**, matching the **High ($2,000â€“$10,000)** impact tier: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The constraint is that the attacker must first have been granted the relevant permission â€” a realistic scenario given that SNS explicitly supports and documents delegation of these permissions.

## Likelihood Explanation
`NEURON_PERMISSION_TYPE_DISBURSE` (value 5) and `NEURON_PERMISSION_TYPE_DISBURSE_MATURITY` (value 8) are standard entries in `neuron_grantable_permissions`. The SNS framework and documentation encourage granting fine-grained permissions to dapp frontend canisters, co-signers, and automation bots. Any such delegatee â€” whether malicious from the start or later compromised â€” can execute the attack with a single standard `manage_neuron` ingress call. No key compromise, no threshold attack, no governance majority, and no privileged infrastructure access is required beyond holding the granted permission.

## Recommendation
When `to_account` is `None` and the caller is not the neuron owner, default to the neuron owner's account rather than the caller's. The neuron owner can be identified as the principal holding `NeuronPermissionType::ManagePrincipals` (the claimer). Apply the same fix to both `disburse_neuron` and `disburse_maturity`:

```rust
// Suggested fix for disburse_neuron and disburse_maturity
let to_account = match disburse.to_account.as_ref() {
    None => {
        // Use the neuron owner's account, not the caller's
        let owner = neuron
            .permissions
            .iter()
            .find(|p| p.permission_type.contains(&(NeuronPermissionType::ManagePrincipals as i32)))
            .and_then(|p| p.principal)
            .ok_or_else(|| GovernanceError::new_with_message(
                ErrorType::NotFound,
                "Cannot determine neuron owner for default disbursement destination",
            ))?;
        Account { owner: owner.0, subaccount: None }
    }
    Some(ai_pb) => Account::try_from(ai_pb.clone())...
};
```
Alternatively, require `to_account` to be explicitly specified whenever the caller does not hold `ManagePrincipals`.

## Proof of Concept
1. Alice claims an SNS neuron with 100 SNS tokens; the neuron dissolves.
2. Alice calls `manage_neuron` with `Command::AddNeuronPermissions` granting `NeuronPermissionType::Disburse` to Bob.
3. Bob sends ingress `manage_neuron`:
   ```
   Command::Disburse(Disburse { amount: None, to_account: None })
   ```
4. `disburse_neuron` passes `neuron.check_authorized(&Bob, Disburse)` âś“
5. `to_account` resolves to `Account { owner: Bob, subaccount: None }` (lines 1144â€“1146).
6. The ledger transfer sends ~100 SNS tokens to Bob's account.
7. Alice's neuron stake is zero; Bob holds the funds.

A deterministic unit test can be written using the existing `GovernanceCanisterFixtureBuilder` pattern (see `rs/sns/governance/tests/governance.rs`): create a neuron owned by `user_principal_A`, grant `Disburse` to `user_principal_B`, call `disburse_neuron` as `B` with `to_account: None`, and assert that `B`'s account balance increased while `A`'s did not. The same pattern applies to `disburse_maturity` using the `DisburseMaturity` command.
