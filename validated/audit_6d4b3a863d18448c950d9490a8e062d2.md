### Title
SNS Swap Token Conservation Bug: Integer Division Truncation in `Swap::scale` Permanently Locks SNS Tokens - (File: rs/sns/swap/src/swap.rs)

### Summary
The `Swap::scale` function in the SNS Swap canister uses integer (floor) division when computing each participant's SNS token allocation. Because the sum of per-participant floor-divided allocations is strictly less than the total SNS tokens offered whenever the division is not exact, a residual amount of SNS tokens remains permanently locked in the Swap canister's ledger account after finalization, with no redistribution mechanism.

### Finding Description
During swap finalization, `create_sns_neuron_recipes` calls `Swap::scale` for every direct participant and every Neurons' Fund neuron to compute their SNS token allocation:

```rust
fn scale(amount_icp_e8s: u64, total_sns_e8s: u64, total_icp_e8s: NonZeroU64) -> u64 {
    let r = (amount_icp_e8s as u128)
        .saturating_mul(total_sns_e8s as u128)
        .div(NonZeroU128::from(total_icp_e8s));
    r as u64
}
``` [1](#0-0) 

This computes `floor(buyer_icp_e8s × total_sns_e8s / total_icp_e8s)` for each participant. Because integer division truncates, the sum of all per-participant allocations satisfies:

```
Σ floor(buyer_i × total_sns / total_icp) ≤ total_sns
```

The code itself acknowledges this residual in a log message:

```rust
log!(INFO, "SNS Neuron Recipes Created; ... Participants receive a total of {} out of {} (change {});",
    total_sns_tokens_sold_e8s, sns_being_offered_e8s,
    sns_being_offered_e8s - total_sns_tokens_sold_e8s);
``` [2](#0-1) 

The "change" tokens remain in the Swap canister's SNS ledger subaccount. After finalization, `sweep_sns` transfers only the exact `amount_e8s` from each neuron recipe to the corresponding neuron account. No step redistributes the remainder to the SNS treasury or any participant. The Swap canister is eventually deleted, permanently removing these tokens from circulation.

The maximum residual is bounded by `N − 1` e8s, where N is the total number of participants (direct + Neurons' Fund neurons). With thousands of participants this can reach thousands of e8s.

### Impact Explanation
SNS tokens are permanently locked in the Swap canister's ledger account after finalization and are burned when the canister is deleted. This is a ledger conservation violation: the total SNS tokens distributed to participants is strictly less than `sns_token_e8s` whenever the per-participant ICP contributions do not divide `total_sns_e8s` evenly. The magnitude is bounded by `N − 1` e8s (at most one e8 lost per participant), so for a swap with 10,000 participants the maximum loss is 9,999 e8s. These tokens are irrecoverable.

### Likelihood Explanation
This occurs in every SNS decentralization swap where the per-participant ICP amounts do not divide `sns_token_e8s` exactly — which is the common case for any swap with more than one participant and non-round contribution amounts. Any unprivileged user who participates in a swap (via `refresh_buyer_tokens`) contributes to triggering this path. Finalization is called by anyone after the swap commits.

### Recommendation
After computing all per-participant allocations, calculate the remainder:

```rust
let remainder = sns_being_offered_e8s - total_sns_tokens_sold_e8s;
```

and distribute it to the SNS governance treasury account (or add it to the last participant's allocation, analogous to the `apportion_approximately_equally` pattern already used in `generate_vesting_schedule`). The `apportion_approximately_equally` function in the same file already demonstrates the correct pattern: compute `quotient = total / len`, then distribute the `remainder = total % len` by incrementing the last `remainder` buckets by 1. [3](#0-2) 

### Proof of Concept
- `sns_token_e8s = 10`, `total_icp_e8s = 3`, two participants each contributing 1 ICP and one contributing 1 ICP (total = 3 ICP).
- Participant A (1 ICP): `floor(1 × 10 / 3) = 3` SNS e8s
- Participant B (1 ICP): `floor(1 × 10 / 3) = 3` SNS e8s
- Participant C (1 ICP): `floor(1 × 10 / 3) = 3` SNS e8s
- Total distributed: 9 SNS e8s. Remainder: 1 SNS e8 permanently locked in Swap canister.

The `create_sns_neuron_recipes` loop iterates over `self.buyers` and `self.cf_participants`, calling `Swap::scale` for each, accumulating `total_sns_tokens_sold_e8s` which will be less than `sns_being_offered_e8s` in this case. [4](#0-3) [5](#0-4)

pub fn apportion_approximately_equally(total: u64, len: u64) -> Result<Vec<u64>, String> {
    let quotient = total
        .checked_div(len)
        .ok_or_else(|| format!("Unable to divide total={total} by len={len}"))?;
    let remainder = total % len; // For unsigned integers, % cannot overflow.

    // So far, we have only apportioned quotient * len. To reach the desired
    // total, we must still somehow add remainder (per Euclid's Division
    // Theorem). That is accomplished right after this.
    let mut result = vec![quotient; len as usize];

    // Divvy out the remainder: Starting from the last element, increment
    // elements by 1. The number of such increments performed here is remainder,
    // bringing our total back to the desired amount.
    if remainder >= result.len() as u64 {
        return Err(format!("Could not apportion {total} into {len} pieces"));
    }
    let mut iter_mut = result.iter_mut();
    for _ in 0..remainder {
        let element: &mut u64 = iter_mut
            .next_back()
            // We can prove that this will not panic:
            // The number of iterations of this loop is total % len.
            // This must be < len (by Euclid's Division Theorem).
            // Thus, the number of iterations that this loop goes through is < len.
            // Thus, the number of times next_back is called is < len.
            // next_back only returns None after len calls.
            // Therefore, next_back does not return None here.
            // Therefore, this expect will never panic.
            .ok_or_else(
                || format!("Ran out of elements to increment. total={total}, len={len}",),
            )?;

        // This cannot overflow because the result must be <= total. Thus, this
        // will not panic.
        *element = element.checked_add(1).ok_or_else(|| {
            format!("Incrementing element by 1 resulted in overflow. total={total}, len={len}",)
        })?;
    }

    Ok(result)
