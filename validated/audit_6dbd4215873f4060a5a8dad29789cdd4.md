Audit Report

## Title
Unbounded Index Access on Empty `Vec<EnvelopePair>` Causes Rosetta Node Panic — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

## Summary
The `construction_hash` handler unconditionally accesses `envelope_pairs[0]` after finding a `RequestType::Send` entry via `find`, with no check that the `Vec<EnvelopePair>` is non-empty. An attacker can craft a CBOR-encoded `SignedTransaction` containing `(RequestType::Send, vec![])`, causing an index-out-of-bounds panic that crashes the Rosetta node process. The same pattern exists in `construction_parse.rs` at `updates[0]`. No authentication is required.

## Finding Description
`SignedTransaction::from_str` performs a plain `serde_cbor::from_slice` with no structural validation on the caller-supplied hex string: [1](#0-0) 

`Request` is a plain tuple alias with no invariant enforcement — the `Vec<EnvelopePair>` may be empty: [2](#0-1) 

In `construction_hash`, after `find` succeeds on a `RequestType::Send` entry, `envelope_pairs[0]` is accessed without a bounds check: [3](#0-2) 

`RequestType::Send` unconditionally satisfies `is_transfer()`, so the `find` always succeeds when the attacker supplies a `Send` entry: [4](#0-3) 

When `envelope_pairs` is an empty slice, `envelope_pairs[0]` panics with an index-out-of-bounds. The same unchecked `updates[0]` pattern exists in `construction_parse`: [5](#0-4) 

No `catch_unwind` or panic hook is present anywhere in the Rosetta API codebase, so the panic propagates and terminates the process.

## Impact Explanation
The Rosetta node process crashes and becomes unavailable. Any operator or exchange relying on the ICP Rosetta node for ledger monitoring, balance queries, or transaction submission loses service until the process is manually restarted. Because the endpoint is unauthenticated and the payload is trivially constructable, the attack can be repeated immediately after each restart, constituting a sustained DoS. This matches the allowed impact: **High ($2,000–$10,000) — Significant Rosetta/infrastructure security impact with concrete user or protocol harm**, and **High — Application/platform-level DoS not based on raw volumetric DDoS**.

## Likelihood Explanation
The `POST /construction/hash` endpoint is a public, unauthenticated HTTP API. Crafting the malicious payload requires only CBOR-encoding a two-element array `["TRANSACTION", []]` and hex-encoding the result — achievable with any CBOR library in any language. No privileged access, key material, or protocol participation is required. The attack is repeatable with zero cost after each restart.

## Recommendation
Replace the unchecked `envelope_pairs[0]` index in `construction_hash.rs` with a bounds-checked accessor that returns an `ApiError` on failure:

```rust
let first = envelope_pairs.first().ok_or_else(|| {
    ApiError::invalid_transaction("Transfer request has no envelope pairs")
})?;
TransactionIdentifier::try_from_envelope(request_type.clone(), &first.update)
```

Apply the same fix to `updates[0]` in `construction_parse.rs` — replace the direct index with `.first()` and propagate an error if the slice is empty, rather than panicking.

## Proof of Concept
```rust
// Craft: SignedTransaction { requests: [(RequestType::Send, vec![])] }
// RequestType::Send serializes as "TRANSACTION" (serde rename)
let malicious: Vec<(String, Vec<()>)> = vec![("TRANSACTION".to_string(), vec![])];
let cbor = serde_cbor::to_vec(&malicious).unwrap();
let hex_payload = hex::encode(&cbor);

// POST /construction/hash
// { "network_identifier": {...}, "signed_transaction": "<hex_payload>" }
// => thread panics: index out of bounds: the len is 0 but the index is 0
// => Rosetta node process terminates
```

A unit test can be written directly against `RosettaRequestHandler::construction_hash` by constructing a `ConstructionHashRequest` whose `signed_transaction` field is the hex-encoded CBOR of `[("TRANSACTION", [])]` and asserting that the call returns an `Err(ApiError)` rather than panicking.
