### Title
Integer Underflow in `chain_state_commitment_before.block_number` at Genesis Block — (`basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/post_tx_op_proving_singleblock_batch.rs` and `post_tx_op_proving_multiblock_batch.rs`)

---

### Summary

Both proving post-tx operations compute `chain_state_commitment_before.block_number` as `metadata.block_number() - 1` with no guard for the genesis case (`block_number == 0`). When the genesis block is proven, this unchecked `u64` subtraction either panics (debug builds) or silently wraps to `u64::MAX` (release/RISC-V proving builds), producing an incorrect `chain_state_commitment_before` hash and therefore an incorrect public input. The settlement layer will reject the resulting proof, permanently blocking L1 finalization from genesis.

---

### Finding Description

In `post_tx_op_proving_singleblock_batch.rs` and `post_tx_op_proving_multiblock_batch.rs`, the `chain_state_commitment_before` struct is built as:

```rust
let chain_state_commitment_before = ChainStateCommitment {
    state_root: state_commitment.root,
    next_free_slot: state_commitment.next_free_slot,
    block_number: metadata.block_number() - 1,   // ← unchecked subtraction
    last_256_block_hashes_blake: blocks_hasher.finalize().into(),
    last_block_timestamp,
};
``` [1](#0-0) [2](#0-1) 

`metadata.block_number()` is sourced directly from `BlockMetadataFromOracle.block_number`, which is `u64`. [3](#0-2) 

When `block_number == 0` (genesis block):

- **Debug builds**: Rust panics with `"attempt to subtract with overflow"`, aborting the proving run.
- **Release builds (RISC-V proving target)**: `0u64 - 1` wraps to `u64::MAX`. The `chain_state_commitment_before` is hashed with `block_number = u64::MAX` instead of the correct genesis value, producing a wrong `state_before` field in `BatchPublicInput`. [4](#0-3) 

The `state_before` hash is what the settlement layer checks against its stored chain state commitment. A wrong value causes proof verification to fail.

The analogous genesis guard already exists in `form_block_header` for the block header path:

```rust
let previous_block_hash = if block_number == 0 {
    Bytes32::ZERO
} else {
    system.get_blockhash(block_number - 1)?
};
``` [5](#0-4) 

That guard is absent in both proving post-tx operations.

---

### Impact Explanation

The `chain_state_commitment_before` hash is embedded in `BatchPublicInput.state_before`, which is the on-chain commitment the settlement layer verifies against its stored state. [6](#0-5) 

A corrupted `state_before` (containing `block_number = u64::MAX`) will never match the genesis commitment stored on the settlement layer, causing every proof of the genesis block to be rejected. The chain cannot advance past genesis on L1. This is a **valid-execution unprovability** bug: the genesis block executes correctly in forward mode but cannot be proven.

---

### Likelihood Explanation

This is a **deterministic, 100% reproducible** failure for every new chain deployment. The genesis block (`block_number = 0`) is always the first block proven. No special attacker action is required; the prover/forward execution input of `block_number = 0` is the normal operational path. The `BlockMetadataFromOracle` default even sets `block_number: 1` in tests, which masks the bug in the test suite. [7](#0-6) 

---

### Recommendation

Replace the bare subtraction with a saturating or guarded form in both proving post-tx operations:

```rust
block_number: metadata.block_number().saturating_sub(1),
```

This mirrors the existing guard in `form_block_header` and produces `0` for the genesis block, matching the expected settlement-layer genesis commitment.

---

### Proof of Concept

1. Construct a `BlockMetadataFromOracle` with `block_number = 0` and a valid `ProofData` (any non-zero `state_root_view`, `last_block_timestamp = 0`).
2. Run `ZKHeaderStructurePostTxOpProvingSingleblockBatch::post_op` in a release build.
3. Observe that `chain_state_commitment_before.block_number` is `u64::MAX` (= `18446744073709551615`).
4. The resulting `BatchPublicInput.state_before` hash will differ from the settlement layer's genesis commitment, causing proof rejection.
5. In a debug build, the process panics at line 143 of `post_tx_op_proving_singleblock_batch.rs` with `"attempt to subtract with overflow"`. [8](#0-7) [9](#0-8)
