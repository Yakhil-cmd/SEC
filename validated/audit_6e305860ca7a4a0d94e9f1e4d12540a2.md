### Title
Pausing the Engine After XCC Precompile Execution Permanently Freezes User wNEAR in the Engine's Implicit Address — (File: `engine/src/contract_methods/xcc.rs`)

---

### Summary

When a user calls the XCC precompile with `required_near > 0`, their wNEAR ERC-20 balance is transferred to the engine's implicit address atomically during EVM execution. A subsequent NEAR promise callback (`withdraw_wnear_to_router`) is responsible for unwrapping that wNEAR and forwarding NEAR to the router sub-account. Because `withdraw_wnear_to_router` enforces `require_running`, pausing the engine between the EVM execution and the callback causes the callback to fail, leaving the user's wNEAR permanently frozen in the engine's implicit address with no recovery path.

---

### Finding Description

**Step 1 — wNEAR is transferred from the user to the engine's implicit address inside the EVM.**

In `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the XCC precompile calls `transferFrom` on the wNEAR ERC-20 contract to move the user's wNEAR to the engine's implicit address: [1](#0-0) 

This ERC-20 debit is committed atomically as part of the EVM transaction. The user's wNEAR balance is reduced immediately.

**Step 2 — A NEAR promise chain is scheduled that includes a `withdraw_wnear_to_router` callback.**

In `engine/src/xcc.rs`, `handle_precompile_promise` constructs a promise chain. When `required_near > 0`, it schedules a call back to the engine's own `withdraw_wnear_to_router` method: [2](#0-1) 

This callback executes in a subsequent NEAR receipt (a later block), creating a window between the EVM execution and the callback.

**Step 3 — `withdraw_wnear_to_router` enforces `require_running`.**

In `engine/src/contract_methods/xcc.rs`, the callback checks the engine's pause state before doing anything: [3](#0-2) 

If the engine is paused between the EVM execution (Step 1) and this callback (Step 3), the callback returns an error and the NEAR promise chain is broken. The wNEAR that was already debited from the user in Step 1 remains in the engine's implicit address as an ERC-20 balance.

**Step 4 — No recovery mechanism exists.**

The engine's implicit address (`near_account_to_evm_address(engine_account_id.as_bytes())`) is not controlled by any private key. There is no function in the Aurora Engine to recover ERC-20 tokens stranded in this address. The user's wNEAR is permanently frozen.

---

### Impact Explanation

**Permanent freezing of user funds.** The user's wNEAR ERC-20 balance is debited before the promise chain executes. If the engine is paused before `withdraw_wnear_to_router` runs, the wNEAR is irrecoverably locked in the engine's implicit EVM address. No admin function exists to sweep or refund these tokens.

---

### Likelihood Explanation

**Low.** The attack window is the gap between the EVM transaction receipt and the `withdraw_wnear_to_router` callback receipt (one or more NEAR blocks). The engine owner must call `pause_contract` within this window. This can happen legitimately during a security incident response, an upgrade, or an emergency pause — scenarios that are uncommon but realistic. The `exit_to_near_precompile_callback` has an analogous `require_running` check with a refund path; `withdraw_wnear_to_router` has no such refund path.

---

### Recommendation

Remove the `require_running` check from `withdraw_wnear_to_router`, since it is a private callback (`env.assert_private_call()` already enforces that only the engine itself can invoke it) and its sole purpose is to complete a fund movement already committed in the EVM. Alternatively, add a refund path: if `require_running` fails, re-credit the wNEAR back to the original sender's ERC-20 balance, mirroring the refund logic in `exit_to_near_precompile_callback`. [4](#0-3) 

---

### Proof of Concept

1. User calls the XCC precompile with `CrossContractCallArgs::Eager` targeting a NEAR contract, with `attached_near > 0` or with no existing router sub-account (so `required_near = attached_near + STORAGE_AMOUNT`).
2. The XCC precompile executes `transferFrom` on the wNEAR ERC-20 contract, moving `required_near` worth of wNEAR from the user's address to the engine's implicit address. The user's ERC-20 balance is reduced. [5](#0-4) 
3. The EVM transaction completes. A NEAR promise chain is scheduled: `[deploy router if needed] → [withdraw_wnear_to_router] → [execute on router]`. [6](#0-5) 
4. Before the `withdraw_wnear_to_router` receipt executes (in the next NEAR block), the engine owner calls `pause_contract`. [7](#0-6) 
5. The `withdraw_wnear_to_router` receipt executes. `require_running` returns `ERR_PAUSED`. The callback fails. The promise chain is broken. [3](#0-2) 
6. The wNEAR remains in the engine's implicit EVM address. The user's wNEAR is permanently frozen. No recovery function exists in the contract.

pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
