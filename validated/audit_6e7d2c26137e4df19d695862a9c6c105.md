### Title
Incomplete `delta_gas` Negative-Branch Refund in `compute_gas_refund` Causes Systematic User Overcharge - (File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs)

---

### Summary

The `compute_gas_refund` function implements ZKsync OS's double resource accounting model but contains an acknowledged incomplete implementation: when `delta_gas < 0` (EVM gas consumption exceeds native resource consumption in gas-equivalent terms), the function does **not** reduce `gas_used`, causing users to be charged for more gas than their actual native resource consumption warrants. The excess fee is permanently transferred to the operator (coinbase) with no recovery path for the user. A `TODO` comment at the exact branch explicitly flags this as unresolved.

---

### Finding Description

ZKsync OS uses a double resource accounting model where both EVM gas (ergs) and native resources (RISC-V proving cycles) are tracked. After execution, `compute_gas_refund` computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
```

The documentation in `docs/double_resource_accounting.md` specifies:

> "If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. [...] Finally, any remaining gas left is refunded as usual."

The code correctly handles `delta_gas > 0` (native cost exceeds EVM gas cost — charge extra gas):

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs, lines 66-81
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // In this case, the native resource consumption is more than the
        // gas consumption accounted for. Consume extra gas.
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?
}
```

The `delta_gas < 0` branch — where EVM gas consumption exceeds native resource consumption — is entirely unimplemented. The `TODO` comment at line 80 is the developer's own acknowledgment that this branch is incomplete. When `delta_gas < 0`, `gas_used` is left at its full EVM-derived value. The downstream `refund_and_commit_fee` in `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs` (lines 452–544) then:

1. Refunds the user only `gas_price * (gas_limit - gas_used)` — a smaller refund than warranted.
2. Pays the operator `gas_price * gas_used` — a larger payment than warranted.

The user's "missing" refund of `|delta_gas| * gas_price` is permanently transferred to the operator with no mechanism to recover it.

---

### Impact Explanation

Every L2 (ZK) transaction where EVM gas consumption exceeds native resource consumption results in the user being overcharged. The overcharge amount is:

```
overcharge = |delta_gas| * gas_price
           = (gas_used - native_used / native_per_gas) * gas_price
```

This is a direct, permanent loss of user funds. The excess is not burned or locked — it is transferred to the operator's coinbase address. There is no recovery mechanism: the user cannot reclaim the overcharged amount, and the protocol provides no correction path.

The docs state "any remaining gas left is refunded as usual," implying the `delta_gas < 0` case should produce a larger refund. The code does not implement this, creating a divergence between the documented specification and the actual behavior.

---

### Likelihood Explanation

`delta_gas < 0` occurs when:

```
native_used / native_per_gas < gas_used
```

i.e., when a transaction's EVM gas consumption is higher than its native resource consumption expressed in gas units. This is reachable by any unprivileged L2 transaction sender. Concretely, it occurs when:

- A user pays a high `gas_price` (increasing `native_per_gas = gas_price / native_price`), which lowers the native-to-gas ratio, making it easier for EVM gas to exceed the native-equivalent.
- A transaction performs EVM-expensive but proving-cheap operations (e.g., arithmetic-heavy contracts with minimal storage writes).

The `native_per_gas` is derived from `gas_price / native_price` (operator-set). During periods of elevated gas prices, `native_per_gas` rises, making `native_used / native_per_gas` smaller and `delta_gas` more likely to be negative. This is a normal, recurring network condition — not an edge case.

---

### Recommendation

Implement the `delta_gas < 0` branch by reducing `gas_used` when native resource consumption is lower than EVM gas consumption:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native resource consumption is less than EVM gas consumption.
    // Reduce gas_used so the user is refunded the excess.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction);
    // Ensure gas_used does not fall below minimal_gas_used (already enforced above).
    gas_used = gas_used.max(minimal_gas_used);
}
```

Remove the `TODO` comment once the branch is implemented and covered by tests.

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `gas_price = 10_000` (high, e.g., during congestion)
- `native_price = 100` (operator-set)
- `native_per_gas = gas_price / native_price = 100`
- Transaction executes and uses `gas_used_evm = 50_000` EVM gas
- Transaction uses `native_used = 2_000_000` native units

**Computation:**
```
delta_gas = (2_000_000 / 100) - 50_000 = 20_000 - 50_000 = -30_000
```

**Current behavior (buggy):**
- `delta_gas < 0` → branch not taken → `gas_used` stays at `50_000`
- User refund: `(100_000 - 50_000) * 10_000 = 500_000_000`
- Operator payment: `50_000 * 10_000 = 500_000_000`

**Correct behavior (per spec):**
- `gas_used` should be reduced to `50_000 - 30_000 = 20_000`
- User refund: `(100_000 - 20_000) * 10_000 = 800_000_000`
- Operator payment: `20_000 * 10_000 = 200_000_000`

**User loss per transaction:** `300_000_000` wei (0.3 ETH at these prices), permanently transferred to the operator with no recovery path.

The entry point is any standard L2 EIP-1559 transaction processed through `ZkTransactionFlowOnlyEOA::before_refund` → `compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` lines 66–81. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
