Audit Report

## Title
Unbounded `candid::Int` in `Value::hash()` Causes Unconditional Panic, Permanently DoS-ing index-ng Canister - (File: `packages/icrc-ledger-types/src/icrc/generic_value.rs`)

## Summary

`Value::hash()` unconditionally panics via `.expect()` when called on a `Value::Int` whose inner `candid::Int` (arbitrary-precision `BigInt`) does not fit in 128 bits. A malicious ICRC-1 ledger canister can return blocks containing oversized `Int` values in extra map fields, causing any index-ng canister pointed at it to trap on every periodic timer tick when it calls `original_block.hash()` inside `append_block`, resulting in a permanent crash loop and complete unavailability of the index-ng canister.

## Finding Description

**Root cause — `Value::hash()` panics on oversized `Int`:**

In `packages/icrc-ledger-types/src/icrc/generic_value.rs` lines 127–131, the `Value::Int` arm calls `.to_i128().expect(...)`:

```rust
Value::Int(int) => {
    let v = int
        .0
        .to_i128()
        .expect("BUG: blocks cannot contain integers that do not fit into the 128-bit representation");
```

`candid::Int` is backed by an arbitrary-precision `BigInt`. If the integer exceeds 128 bits, `to_i128()` returns `None` and `.expect()` panics unconditionally. The comment at line 133 acknowledges this is a known limitation.

**Injection path — `icrc1_block_from_value` accepts unbounded `NEG_BIGNUM`:**

In `rs/ledger_suite/icrc1/src/blocks.rs` lines 150–158, the `NEG_BIGNUM` CBOR tag arm constructs a `BigInt` from raw bytes with no size guard:

```rust
CiboriumValue::Tag(NEG_BIGNUM, value) => {
    use num_bigint::{BigInt, BigUint, Sign};
    let value_bytes = value.into_bytes()...;
    Ok(GenericValue::Int(Int(BigInt::from_biguint(
        Sign::Minus,
        BigUint::from_bytes_be(&value_bytes),
    ) - 1)))
}
```

A 17-byte (136-bit) payload produces a `BigInt` that exceeds `i128::MAX`, which will later cause `to_i128()` to return `None`.

**Trigger — `append_block` calls `original_block.hash()` on the unfiltered block:**

In `rs/ledger_suite/icrc1/index-ng/src/main.rs` lines 875–916, `append_block` clones the raw block as `original_block`, then calls `original_block.hash()` at line 907:

```rust
let original_block = block.clone();
// ...
if original_block.hash() != decoded_value.hash() {
```

The oversized `Int` survives in `original_block` because:
1. `generic_block_to_encoded_block(block)` (line 879) succeeds — it serializes the oversized `Int` as a `NEG_BIGNUM` CBOR tag without panicking.
2. `Block::<Tokens>::decode(block.clone())` (line 893) succeeds — the unknown key `"x"` is silently ignored by the typed decoder.
3. `original_block.hash()` (line 907) is then called on the unmodified block still containing the oversized `Value::Int` → **panic / Wasm trap**.

**Why existing checks are insufficient:**

The unknown-field detection logic at line 907 is the check that is supposed to catch extra fields — but it panics *before* it can return an error, because `hash()` is called as part of the comparison itself. There is no pre-validation step that rejects `Value::Int` values exceeding 128 bits before `hash()` is invoked.

## Impact Explanation

When the index-ng canister traps inside its periodic timer callback, the IC rolls back all state changes for that execution. The canister's block index pointer is not advanced, so on the next timer tick it fetches the same malicious block and traps again. This creates a **permanent crash loop**: the index-ng canister becomes completely unavailable — all balance queries, transaction history lookups, and ICRC-3 block serving are denied indefinitely. This matches the allowed High impact: *"Application/platform-level DoS, crash … or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

Any IC principal can deploy a canister that implements the `icrc3_get_blocks` interface and returns blocks with oversized `Int` values in extra map fields. No privileged access, governance vote, or threshold key is required — only cycles to deploy the malicious canister. The index-ng canister is architecturally designed to index arbitrary ICRC-1 ledgers, so any index-ng instance pointed at an attacker-controlled ledger is vulnerable. The attack is deterministic and repeatable with zero per-attempt cost after initial deployment.

## Recommendation

- **Short term:** In `Value::hash()`, replace the `.expect()` with a checked conversion and return a `Result<Hash, HashError>` (or at minimum, return an error hash / propagate an error) instead of panicking. Guard the `Value::Int` arm against integers that do not fit in 128 bits.
- **Short term:** In `append_block`, add a pre-validation step that walks `original_block` and rejects any `Value::Int` whose inner `BigInt` does not fit in 128 bits *before* calling `hash()`, analogous to the existing unknown-field detection logic.
- **Long term:** Change `Value::hash()` signature to `fn hash(&self) -> Result<Hash, HashError>` so all callers can handle oversized integers gracefully without panicking.
- **Long term:** Add a size guard in `icrc1_block_from_value` for the `NEG_BIGNUM` arm that rejects payloads exceeding 16 bytes, preventing oversized integers from entering the `GenericBlock` representation at all.

## Proof of Concept

1. Deploy a canister on the IC implementing `icrc3_get_blocks` that returns:
   ```
   ICRC3Value::Map({
     "ts": ICRC3Value::Nat(1_000_000_000),
     "tx": ICRC3Value::Map({ "op": ICRC3Value::Text("mint"), ... }),
     "x":  ICRC3Value::Int(candid::Int from a BigInt with 17+ bytes, e.g. -(2^136))
   })
   ```
2. Deploy an index-ng canister configured to index the malicious ledger canister.
3. Observe that on the first timer tick, `append_block` is called with the malicious block.
4. `generic_block_to_encoded_block` succeeds (serializes the oversized Int as `NEG_BIGNUM` with 17-byte payload).
5. `Block::<Tokens>::decode` succeeds (unknown key `"x"` is ignored).
6. `original_block.hash()` is called → `Value::Int` arm → `to_i128()` returns `None` → `.expect()` panics → Wasm trap.
7. State is rolled back; block index pointer is not advanced.
8. On every subsequent timer tick, steps 3–7 repeat → permanent crash loop.

A deterministic unit test can reproduce this without a live canister:
```rust
#[test]
fn test_oversized_int_panics() {
    use num_bigint::{BigInt, Sign};
    let big = BigInt::from_biguint(Sign::Minus, num_bigint::BigUint::from_bytes_be(&[0xffu8; 17])) - 1;
    let value = Value::Int(candid::Int(big));
    value.hash(); // panics
}
```
