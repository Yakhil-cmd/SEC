Audit Report

## Title
Byzantine Peer Stalls State Sync via Unconditional `NoContent` Peer Removal and No Restart Penalization — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
A Byzantine subnet peer can repeatedly stall state sync on a lagging replica by advertising a valid state, returning `NO_CONTENT` for every chunk request (causing its own unconditional removal from `active_downloads`), waiting for the sync to self-terminate due to an empty peer set, then re-advertising to restart the cycle. No rate-limiting, cooldown, or peer penalization exists anywhere in the state sync manager module, confirmed by code inspection and grep.

## Finding Description

All four mechanical steps of the cycle are confirmed in the actual code:

**Step 1 — `NoContent` unconditionally removes the peer and decrements the download budget.** [1](#0-0) 
There is no distinction between a peer that never had the chunk vs. one that is actively lying; both are removed identically.

**Step 2 — An empty `active_downloads` map immediately breaks the event loop, completing the `Shutdown` handle.** [2](#0-1) 

**Step 3 — `handle_advert` detects `shutdown.completed()`, clears `ongoing_state_sync`, then unconditionally falls through to `maybe_start_state_sync` for the same peer's next advert.** [3](#0-2) 
The `StateSyncClient` interface contract states that `maybe_start_state_sync` returns `Some` again once the prior `Chunkable` object is dropped, which happens when the `run` future completes. [4](#0-3) 

**Step 4 — The `Entry::Vacant` guard preventing duplicate peer insertion is reset on every new `start_ongoing_state_sync` call.** [5](#0-4) 

**Step 5 — Zero rate-limiting, blacklisting, cooldown, or penalization logic exists in the module.** A grep across `rs/p2p/state_sync_manager/**` for `blacklist|rate.limit|penali|cooldown|backoff` returns no matches.

The exact cycle:
```
Byzantine peer sends advert
  → handle_advert: ongoing_state_sync is None → maybe_start_state_sync → new OngoingStateSync spawned
  → ongoing.run: peer added to active_downloads, PARALLEL_CHUNK_DOWNLOADS dispatched
  → peer returns NO_CONTENT on first chunk
  → handle_downloaded_chunk_result: active_downloads.remove(&peer_id), allowed_downloads -= 10
  → active_downloads.is_empty() == true → break (Shutdown completes, Chunkable dropped)
  → Byzantine peer re-sends advert
  → handle_advert: ongoing.shutdown.completed() == true → ongoing_state_sync = None
  → maybe_start_state_sync called again → cycle repeats indefinitely
```

The advert channel capacity of 20 only throttles the queue, not the cycle rate. [6](#0-5) 
The cycle speed is bounded by `CHUNK_DOWNLOAD_TIMEOUT` (10 seconds), yielding ~6 restart cycles per minute. [7](#0-6) 

## Impact Explanation

When the Byzantine peer is the sole advertiser of the target state (e.g., a new node joining a subnet where only one peer holds the relevant checkpoint), the lagging replica is permanently unable to complete state sync. It cannot advance its certified state, cannot participate in consensus, and is effectively removed from the subnet for the duration of the attack. Even when honest peers co-advertise, the Byzantine peer causes repeated sync restarts, wasting CPU in the advert handler and the `OngoingStateSync` event loop, and delaying completion proportionally to the fraction of chunk downloads assigned to the Byzantine peer.

This matches the allowed impact: **Application/platform-level DoS / subnet availability impact not based on raw volumetric DDoS** — High severity ($2,000–$10,000).

## Likelihood Explanation

The attacker must be a legitimate subnet node operating below the Byzantine fault threshold. No special privileges beyond subnet membership are required. The attacker needs only: (a) send a valid `StateSyncArtifactId` advert (correct height + hash, which is public information from the consensus chain), and (b) respond to all chunk RPCs with HTTP `204 NO_CONTENT`. Both are trivially achievable by a compromised node. The sole-advertiser condition is realistic for new nodes joining a subnet where only one peer holds the relevant checkpoint.

## Recommendation

1. **Peer penalization**: Track per-peer `NoContent` counts across sync instances (e.g., a `HashMap<NodeId, (u32, Instant)>` in `StateSyncManager`). After N consecutive `NoContent` failures, refuse to add that peer for a configurable backoff window.
2. **Sync restart cooldown**: Record the last time `maybe_start_state_sync` was called for a given `StateSyncArtifactId`. Enforce a minimum interval (e.g., 30 seconds) before restarting a sync for the same height.
3. **Termination reason tracking**: Before breaking on empty `active_downloads`, distinguish between `NoContent`-driven removal and legitimate disconnect, and feed that signal into reputation scoring.

```rust

    Ok(Response::builder()
        .status(StatusCode::NO_CONTENT)
        .body(Bytes::new())
        .unwrap())
});

// Send advert N times from Byzantine peer
for _ in 0..10 {
    handler_tx.send((advert.clone(), BYZANTINE_NODE)).await.unwrap();
    tokio::time::sleep(Duration::from_millis(100)).await;
}

// Assert: state sync never completed AND Byzantine peer was not rate-limited
// (metrics.state_syncs_total == 10, state never advanced)
```

This is a deterministic integration test requiring no external dependencies, directly exercising the `handle_advert` → `start_ongoing_state_sync` → `NoContent` → termination → restart cycle.
