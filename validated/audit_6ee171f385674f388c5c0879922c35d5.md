### Title
Deterministic Subaccount Squatting Blocks `spawn_neuron` with User-Supplied Nonce — (`rs/nns/governance/src/governance.rs`)

---

### Summary

`spawn_neuron` in NNS Governance computes the child neuron's subaccount using `compute_neuron_staking_subaccount(child_controller, nonce)` — the **same hash domain** (`"neuron-stake"`) used by `claim_or_refresh_neuron_by_memo_and_controller`. Any unprivileged caller can pre-occupy that exact subaccount by calling `claim_or_refresh` with the same `(controller, memo)` pair before the victim's `spawn_neuron` executes. The subsequent `ensure_subaccount_available` check then permanently rejects the victim's spawn for that nonce.

---

### Finding Description

**Deterministic subaccount computation in `spawn_neuron`:**

When a user provides an optional nonce to `spawn_neuron`, the child neuron's subaccount is computed deterministically:

```rust
// rs/nns/governance/src/governance.rs ~2678-2688
let to_subaccount = match spawn.nonce {
    None => self.neuron_store.new_neuron_subaccount(&mut *self.randomness)?,
    Some(nonce_val) => {
        let to_subaccount =
            ledger::compute_neuron_staking_subaccount(child_controller, nonce_val);
        self.neuron_store
            .ensure_subaccount_available(to_subaccount)?
    }
};
``` [1](#0-0) 

`compute_neuron_staking_subaccount` uses domain `b"neuron-stake"`:

```rust
// rs/nervous_system/common/src/ledger.rs

    compute_neuron_domain_subaccount_bytes(controller, b"neuron-stake", nonce)
}
``` [2](#0-1) 

**The same domain is used by `claim_or_refresh`:**

```rust
// rs/nns/governance/src/governance.rs ~5860
let subaccount = ledger::compute_neuron_staking_subaccount(controller, memo);
``` [3](#0-2) 

**`ensure_subaccount_available` fails immediately on collision:**

```rust
// rs/nns/governance/src/neuron_store.rs
pub fn ensure_subaccount_available(
    &self,
    subaccount: Subaccount,
) -> Result<Subaccount, NeuronStoreError> {
    if self.has_neuron_with_subaccount(subaccount) {
        return Err(NeuronStoreError::SubaccountAlreadyExists { subaccount });
    }
    Ok(subaccount)
}
``` [4](#0-3) 

The code comment explicitly acknowledges this is a hard failure: *"Unlike random subaccounts (which retry on collision), deterministic subaccounts must fail immediately since retrying would produce the same result."* [5](#0-4) 

**Attack path:**

1. Alice owns a neuron with maturity and intends to call `spawn_neuron` with `new_controller = Bob` and `nonce = 42`.
2. Mallory (attacker) learns Alice's intended `(child_controller=Bob, nonce=42)` — e.g., from a public announcement, a dapp UI, or by observing Alice's prior on-chain behavior.
3. Mallory transfers the minimum neuron stake (1 ICP) to the governance canister's subaccount `compute_neuron_staking_subaccount(Bob, 42)`.
4. Mallory calls `claim_or_refresh_neuron_from_account` (publicly accessible, no whitelist) with `controller=Bob, memo=42`, creating a neuron at that subaccount.
5. Alice's `spawn_neuron` call now hits `ensure_subaccount_available` → `SubaccountAlreadyExists` → returns an error. Alice's maturity is not moved, but the spawn is blocked for nonce 42.

The `claim_or_refresh_neuron_from_account` endpoint is publicly callable:

```rust
// rs/nns/governance/canister/canister.rs
#[update]
async fn claim_or_refresh_neuron_from_account(
    claim_or_refresh: ClaimOrRefreshNeuronFromAccount,
) -> ClaimOrRefreshNeuronFromAccountResponse {
``` [6](#0-5) 

**Why the other deterministic operations are NOT vulnerable to the same cross-domain squatting:**

- `disburse_to_neuron` uses `compute_neuron_disburse_subaccount_bytes` → domain `b"neuron-split"` — different from `"neuron-stake"`, so `claim_or_refresh` cannot pre-occupy it.
- `split_neuron` uses `compute_neuron_split_subaccount_bytes` → domain `b"split-neuron"` — also different.

Only `spawn_neuron` with a user-supplied nonce shares the `"neuron-stake"` domain with `claim_or_refresh`, making it uniquely vulnerable. [7](#0-6) 

---

### Impact Explanation

A victim's `spawn_neuron` call with a specific nonce is permanently blocked for that nonce. The victim's maturity is not lost (the function returns before modifying state), but the victim is forced to use a different nonce. An attacker who systematically pre-occupies many nonces for a target `child_controller` can force the victim to exhaust all predictable nonces. The attacker's ICP is locked in a neuron controlled by `child_controller` (not the attacker), making this a costly griefing attack rather than a theft.

---

### Likelihood Explanation

The attacker must:
1. Know the victim's intended `(child_controller, nonce)` pair — feasible if the victim uses predictable nonces (e.g., 0, 1, 2…) or announces their intent.
2. Pay at least the minimum neuron stake (1 ICP on NNS) per blocked nonce — the attacker's ICP is locked in a neuron they do not control.

The IC has no traditional mempool, but the attacker can submit their `claim_or_refresh` message before the victim's `spawn_neuron` is processed. Likelihood is **low** for targeted attacks but rises if the victim uses predictable nonces.

---

### Recommendation

When `spawn_neuron` is called with a user-supplied nonce, use a domain-separated subaccount that cannot be pre-occupied via `claim_or_refresh`. Specifically, use `compute_neuron_split_subaccount_bytes` (domain `"split-neuron"`) or a dedicated `"spawn-neuron"` domain instead of `compute_neuron_staking_subaccount` (domain `"neuron-stake"`). This mirrors the fix already applied to `split_neuron` and `disburse_to_neuron`, which use distinct domains.

Alternatively, if the nonce-based subaccount is intentional (so the spawned neuron is discoverable on the ledger), document the griefing risk and recommend that callers use random nonces rather than predictable ones.

---

### Proof of Concept

```
// Attacker pre-occupies Alice's intended spawn subaccount:

// Step 1: Compute target subaccount
target = SHA256(0x0c || "neuron-stake" || Bob_principal || nonce_42_be_bytes)

// Step 2: Fund the subaccount
ledger.icrc1_transfer({
    to: { owner: governance_canister_id, subaccount: Some(target) },
    amount: 100_000_000,  // 1 ICP minimum stake
})

// Step 3: Claim the neuron (publicly callable, no whitelist)
governance.claim_or_refresh_neuron_from_account({
    controller: Some(Bob),
    memo: 42,
})
// → Neuron created at `target`, controlled by Bob

// Step 4: Alice's spawn now fails
governance.manage_neuron({
    command: Spawn { new_controller: Some(Bob), nonce: Some(42), ... }
})
// → Err(SubaccountAlreadyExists)
