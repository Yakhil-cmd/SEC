Audit Report

## Title
Unbounded `ingress_end` Causes OOM DoS in ICRC1 and ICP Rosetta `construction_payloads` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`, `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The `construction_payloads` function in both the ICRC1 and ICP Rosetta servers accepts a user-supplied `ingress_end: u64` from JSON metadata with no upper-bound validation. The function then enters a `while` loop that pushes one `u64` entry per step until `ingress_start` reaches `ingress_end`. Supplying `ingress_end = u64::MAX` causes approximately 139 billion iterations, exhausting server memory and crashing the process. No authentication is required.

## Finding Description

In `rs/rosetta-api/icrc1/src/construction_api/services.rs`, `ingress_end` is read directly from attacker-controlled JSON metadata: [1](#0-0) 

The only guards applied before the loop are: [2](#0-1) 

Neither check bounds `ingress_end` from above. With `ingress_end = u64::MAX`, both checks pass: `u64::MAX` is not less than `ingress_start`, and `u64::MAX` is not less than `now + ingress_interval`. The function then enters the unbounded loop: [3](#0-2) 

The step size per iteration is:
```
ingress_interval - INGRESS_INTERVAL_OVERLAP
= (MAX_INGRESS_TTL - PERMITTED_DRIFT) - INGRESS_INTERVAL_OVERLAP
= (300s - 60s) - 120s = 120s = 120_000_000_000 ns
``` [4](#0-3) 

With `ingress_end = u64::MAX ≈ 1.84×10¹⁹ ns` and `now ≈ 1.7×10¹⁸ ns` (year 2024–2026), the loop runs approximately `(1.84×10¹⁹ − 1.7×10¹⁸) / 1.2×10¹¹ ≈ 139 billion` iterations, each pushing a `u64` (8 bytes) into `ingress_expiries`. Total heap allocation: ~1.1 TB, causing OOM process termination.

The identical structural flaw exists in the ICP Rosetta server, with the same unbounded loop and no upper-bound guard on `ingress_end`: [5](#0-4) 

## Impact Explanation

A single unauthenticated HTTP POST request crashes the Rosetta server process via OOM. Any operator running either the ICRC1 or ICP Rosetta server as a gateway for ledger transfers is fully denied service for the duration of the crash and restart cycle. This is a repeatable, zero-cost attack. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash** and **Significant Rosetta/ledger infrastructure security impact with concrete user or protocol harm**.

## Likelihood Explanation

The `/construction/payloads` endpoint is a public HTTP endpoint requiring no authentication. A single POST request with `"ingress_end": 18446744073709551615` in the JSON body is sufficient. The attack requires no special knowledge beyond the public Rosetta API specification, is trivially repeatable, and affects any deployed instance of either Rosetta server.

## Recommendation

Add an explicit upper-bound check immediately after parsing `ingress_end`, before the loop, in both files:

```rust
const MAX_INGRESS_WINDOW: u64 = 24 * 60 * 60 * 1_000_000_000; // 24 hours in ns
if ingress_end.saturating_sub(ingress_start) > MAX_INGRESS_WINDOW {
    return Err(Error::processing_construction_failed(
        "ingress_end - ingress_start exceeds maximum allowed window"
    ));
}
```

Apply the equivalent fix to `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs` at the same logical point (before the `while now < ingress_end` loop at line 101).

## Proof of Concept

```bash
curl -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"<ledger-id>"},
    "operations": [<valid transfer operation>],
    "public_keys": [<valid public key>],
    "metadata": {
      "ingress_start": 1700000000000000000,
      "ingress_end": 18446744073709551615
    }
  }'
# Server OOMs and is killed before responding.
```

A deterministic unit test can also be written by calling `construction_payloads(...)` directly with `ingress_end = u64::MAX` and `ingress_start = now`, verifying the function returns an error rather than hanging/OOMing.
