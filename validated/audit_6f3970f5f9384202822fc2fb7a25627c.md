Audit Report

## Title
Unbounded CBOR Deserialization Without Body Size Limit Enables Single-Request DoS — (`rs/rosetta-api/icrc1/src/construction_api/types.rs`)

## Summary
`SignedTransaction::from_str` and `UnsignedTransaction::from_str` call `serde_cbor::from_slice` on attacker-controlled bytes with no preceding size check. The axum router in `main.rs` registers all construction routes with no body size limit layer. An unauthenticated attacker can POST a single crafted CBOR payload to any of four construction endpoints, triggering unbounded heap allocation and crashing the ICRC1 Rosetta process.

## Finding Description
Both deserialization entry points directly decode attacker-supplied hex into bytes and pass them to `serde_cbor::from_slice` with no length guard:

- `SignedTransaction::from_str` at [1](#0-0) 
- `UnsignedTransaction::from_str` at [2](#0-1) 

The deserialized types hold unbounded `Vec` fields — `SignedTransaction.envelopes: Vec<Envelope<'a>>` and `UnsignedTransaction.envelope_contents: Vec<EnvelopeContent>` — each element of which contains multiple heap-allocated fields (`sender`, `canister_id`, `method_name`, `arg`, etc.): [3](#0-2) [4](#0-3) 

All four affected service functions call these parsers before any application-level validation: [5](#0-4) [6](#0-5) 

The axum router in `main.rs` registers `/construction/parse`, `/construction/hash`, `/construction/submit`, and `/construction/combine` with no `DefaultBodyLimit`, `RequestBodyLimitLayer`, or any equivalent guard: [7](#0-6) 

A grep for `RequestBodyLimitLayer`, `DefaultBodyLimit`, `body_limit`, `MAX_REQUEST`, `JsonConfig`, and `.limit(` across the entire `rs/rosetta-api/icrc1/` tree returns zero matches. By contrast, the ICP Rosetta server explicitly caps JSON bodies at 4 MB via actix-web's `JsonConfig`: [8](#0-7) 

## Impact Explanation
A single unauthenticated HTTP POST to any of the four construction endpoints with a crafted CBOR payload containing a large `Vec` of `EnvelopeContent::Call` entries causes O(N) heap allocations before any validation runs, exhausting process memory and crashing the ICRC1 Rosetta server. This takes the Rosetta API offline for all users of that deployment, blocking all transaction construction, submission, and hash queries until the process is manually restarted. This matches the allowed High impact: **Application/platform-level DoS** and **Significant Rosetta security impact with concrete user or protocol harm** ($2,000–$10,000).

## Likelihood Explanation
The endpoint is publicly reachable with no authentication. The attack requires only knowledge of CBOR encoding (well-documented, standard tooling available), a single HTTP request, and no privileged access, key material, or network-level capability. The missing body-size guard is a straightforward omission. The attack is repeatable: after the process restarts, a single new request re-triggers the crash.

## Recommendation
1. Apply `axum::extract::DefaultBodyLimit::max(4 * 1024 * 1024)` (or `tower_http::limit::RequestBodyLimitLayer`) to the axum router, matching the 4 MB cap used by the ICP Rosetta server.
2. Add an explicit byte-length check on the hex-decoded CBOR slice before calling `serde_cbor::from_slice` in both `SignedTransaction::from_str` and `UnsignedTransaction::from_str`.
3. Consider switching to `serde_cbor::from_reader` with a `Read` adapter that enforces a byte limit, or a deserializer with configurable allocation/recursion limits.

## Proof of Concept
```python
import cbor2, binascii, requests

minimal_call = {
    "request_type": "call",
    "sender": b'\x04',
    "canister_id": b'\x00' * 10,
    "method_name": "x",
    "arg": b'',
    "ingress_expiry": 9999999999999999999,
}
payload = cbor2.dumps({"envelope_contents": [minimal_call] * 500_000})
hex_payload = binascii.hexlify(payload).decode()

body = {
    "network_identifier": {"blockchain": "Internet Computer", "network": "<ledger_id>"},
    "transaction": hex_payload,
    "signed": False,
}
r = requests.post("http://<rosetta-host>:8080/construction/parse", json=body, timeout=120)
print(r.status_code)  # process OOM-killed before response
```

A deterministic unit test can confirm the absence of any guard by calling `UnsignedTransaction::from_str` with a CBOR-encoded struct containing a large `envelope_contents` array and asserting that the call returns an error (size exceeded) rather than allocating unboundedly.

    currency: Currency,

                .collect(),

            .map_err(|err| Error::parsing_unsuccessful(&err))?;
        (
            unsigned_transaction.get_lowest_ingress_expiry(),
            unsigned_transaction.get_highest_ingress_expiry(),

        )
