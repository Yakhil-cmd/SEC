Audit Report

## Title
Missing Callback-ID Validation in `validate_self_validating_payload_impl` and Three `push_response` Arms Allows Byzantine Block Proposer to Inject Spurious Bitcoin Responses — (`rs/bitcoin/consensus/src/payload_builder.rs`, `rs/replicated_state/src/bitcoin.rs`)

## Summary

`validate_self_validating_payload_impl` performs no content validation on a non-empty `SelfValidatingPayload`, returning `Ok(size)` immediately after the empty-payload check. Three of the four arms in `push_response` (`SendTransactionResponse`, `GetSuccessorsReject`, `SendTransactionReject`) push a `ConsensusResponse` into `state.consensus_queue` without verifying that the supplied `callback_id` exists in `subnet_call_context_manager`. A Byzantine block proposer below the consensus fault threshold can therefore craft a payload carrying a spurious response for any known `callback_id`, have it pass consensus validation, and cause execution to deliver fabricated data to the targeted canister — with the original context entry left intact, enabling a subsequent double-delivery when the legitimate response arrives.

## Finding Description

**Layer 1 — Consensus validation is absent.**

`validate_self_validating_payload_impl` (lines 234–251 of `rs/bitcoin/consensus/src/payload_builder.rs`) reads:

```rust
if *payload == SelfValidatingPayload::default() {
    return Ok(0.into());
}
// records metric, returns Ok(size) — no content check
```

For any non-empty payload the function immediately returns `Ok(size)`. It never reads the certified state, never loads `subnet_call_context_manager`, and never checks whether any `callback_id` in the payload corresponds to a pending request. `validate_payload` (lines 357–397) calls this function and adds only a byte-size check; neither path performs callback-ID verification.

**Layer 2 — Three of four `push_response` arms have no callback-ID guard.**

In `rs/replicated_state/src/bitcoin.rs`:

- `GetSuccessorsResponse` (lines 28–63): **protected** — calls `.get_mut(&callback_id).ok_or_else(|| StateError::BitcoinNonMatchingResponse {...})?`, so an unknown ID returns an error.
- `SendTransactionResponse` (lines 64–75): **unprotected** — derives `callback_id` from the response field and unconditionally pushes `ConsensusResponse::new(callback_id, payload)` to `state.consensus_queue`. No lookup in `bitcoin_send_transaction_internal_contexts`.
- `GetSuccessorsReject` (lines 77–88): **unprotected** — same pattern, no lookup.
- `SendTransactionReject` (lines 90–101): **unprotected** — same pattern, no lookup.

None of the three unprotected arms remove the context entry from `subnet_call_context_manager`, so the legitimate response (when it arrives) will also be pushed to the queue, producing a double-delivery.

**Layer 3 — Errors are silently swallowed in the demux.**

`DemuxImpl::process_payload` in `rs/messaging/src/routing/demux.rs` (lines 86–93) calls `push_response_bitcoin` with `.unwrap_or_else(|err| debug!(...))`. Even if an error were returned for the unprotected arms, it would only be logged; execution continues with whatever was already pushed.

**Exploit flow:**

1. Byzantine proposer reads the certified state to obtain a live `callback_id` (e.g., for a pending `send_transaction` call from the ckBTC minter).
2. Proposer crafts a `SelfValidatingPayload` containing a `SendTransactionResponse` (or `GetSuccessorsReject` / `SendTransactionReject`) with that `callback_id`.
3. Consensus validation passes — only size is checked.
4. `push_response` pushes a `ConsensusResponse` with the fabricated payload into `state.consensus_queue`.
5. Execution delivers the spurious response to the targeted canister (e.g., ckBTC minter), which executes its reply/reject handler on fabricated data.
6. The context entry remains; the legitimate adapter response later pushes a second `ConsensusResponse` for the same `callback_id`, but the callback has already been consumed, so the legitimate response is silently dropped.

## Impact Explanation

The Bitcoin subnet hosts the ckBTC minter. Delivering a fabricated `SendTransactionResponse` (always `EmptyBlob`) or a spurious reject to the ckBTC minter for a pending `send_transaction` callback causes the minter to treat a Bitcoin transaction as confirmed (or failed) based on fabricated data, producing incorrect replicated state. The subsequent drop of the legitimate response means the minter never receives the real outcome. This constitutes a significant Chain Fusion / ck-token security impact with concrete protocol harm, matching the **High ($2,000–$10,000)** impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

Exploitation requires controlling a block-proposing replica on the Bitcoin subnet — i.e., being a Byzantine node within the f < n/3 fault threshold. This is within the IC's explicit threat model for protocol-peer behavior. No external network access, governance majority, key material, or victim interaction is required beyond node compromise. The attack is repeatable every round the Byzantine node is the block proposer, and the target `callback_id` is directly readable from the certified state.

## Recommendation

1. **In `validate_self_validating_payload_impl`**: load the certified state at `validation_context.certified_height` via `self.state_manager.get_state_at(...)` and verify that every `callback_id` in the payload exists in `subnet_call_context_manager.bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`. Return `ValidationError::InvalidArtifact(InvalidSelfValidatingPayloadReason::...)` for any unknown ID.

2. **In `push_response` for `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject`**: add the same `.get(&callback_id).ok_or_else(|| StateError::BitcoinNonMatchingResponse { callback_id: callback_id.get() })?` guard that already protects the `GetSuccessorsResponse` arm, and remove the matched context entry to prevent double-delivery.

## Proof of Concept

```rust
// State with no pending Bitcoin requests.
let mut state = ReplicatedState::new(subnet_id, SubnetType::Bitcoin);
// subnet_call_context_manager.bitcoin_send_transaction_internal_contexts is empty.

// Byzantine proposer crafts a spurious SendTransactionResponse.
let result = push_response(
    &mut state,
    BitcoinAdapterResponse {
        response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
            SendTransactionResponse {}
        ),
        callback_id: 999, // non-existent
    },
);
assert!(result.is_ok()); // succeeds — no guard for this arm
assert_eq!(state.consensus_queue.len(), 1); // spurious entry present

// Consensus-layer validation also passes:
let spurious_payload = SelfValidatingPayload::new(vec![BitcoinAdapterResponse {
    response: BitcoinAdapterResponseWrapper::SendTransactionResponse(SendTransactionResponse {}),
    callback_id: 999,
}]);
let validation_result = builder.validate_self_validating_payload(
    &spurious_payload, &validation_context, &[],
);
assert!(validation_result.is_ok()); // BUG: should be Err(InvalidArtifact)
```

A state-machine or PocketIC integration test can confirm end-to-end delivery by checking that `state.consensus_queue` contains the spurious `ConsensusResponse` after `process_payload` is called with the crafted batch.
