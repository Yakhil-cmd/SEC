### Title
Unauthenticated `canister_status` Proxy Allows Any Caller to Leak Cycles, Module Hash, and Controllers of SNS-Controlled Canisters — (`rs/sns/root/canister/canister.rs`)

### Summary
The `canister_status` update method on SNS root forwards an arbitrary caller-supplied `CanisterIdRecord` directly to the IC management canister with no authorization guard. Because SNS root is a controller of all registered dapp canisters, the management canister accepts the call and returns the full `CanisterStatusResult` — including cycles balance, module hash, memory size, and controller list — to the unprivileged caller.

### Finding Description
`canister_status` is declared as an `#[update]` method, making it reachable via ingress from any principal including anonymous. [1](#0-0) 

It contains no caller check. Compare with every other sensitive update method in the same file, all of which gate on `assert_eq_governance_canister_id`: [2](#0-1) [3](#0-2) 

The management canister's `canister_status` is restricted to controllers of the target canister. SNS root is the sole controller of every registered dapp canister (enforced at registration time): [4](#0-3) 

By calling SNS root's `canister_status` with a registered dapp's canister ID, any caller borrows root's controller privilege and receives the full status response.

### Impact Explanation
The returned `CanisterStatusResult` exposes:
- **Exact cycles balance** — enables targeted attacks on canisters near the freezing threshold
- **Module hash** — allows fingerprinting of deployed WASM versions to identify unpatched canisters
- **Controller list** — reveals governance/operational structure of the SNS

All registered dapp canisters and SNS framework canisters (governance, ledger, index) that root controls are affected.

### Likelihood Explanation
The attack requires only a standard ingress call to a public update method with a known canister ID. No privileged access, no key material, no social engineering. Canister IDs of SNS dapps are publicly discoverable via `list_sns_canisters` (a query method with no auth check). [5](#0-4) 

### Recommendation
Add a caller authorization check at the top of `canister_status`, restricting it to the SNS governance canister (or at minimum to the set of SNS framework canisters), consistent with the pattern used by all other privileged methods:

```rust
async fn canister_status(id: CanisterIdRecord) -> CanisterStatusResult {
    assert_eq_governance_canister_id(PrincipalId(ic_cdk::api::caller()));
    // ...
}
```

### Proof of Concept
State-machine test outline:
1. Deploy SNS root with a known governance canister ID.
2. Register a dapp canister (controlled by root) via governance.
3. From an **anonymous principal**, send an ingress `canister_status` call to SNS root with the dapp's `CanisterIdRecord`.
4. Assert the call succeeds and the response contains a non-zero `cycles` value and a non-`None` `module_hash`.

No privileged setup is needed beyond step 2, which reflects normal SNS operation. [1](#0-0)
