Audit Report

## Title
Unbounded Neuron Iteration in SNS Governance `compute_ballots_for_new_proposal` Causes Instruction-Limit DoS on Proposal Submission — (File: `rs/sns/governance/src/governance.rs`)

## Summary
`SNS Governance::compute_ballots_for_new_proposal` iterates over every entry in `self.proto.neurons` without any instruction-limit guard, computing `dissolve_delay_seconds` and `voting_power` per neuron. This function is called synchronously inside `make_proposal`, an update call reachable by any SNS neuron holder. When the neuron count grows large enough to exhaust the IC's per-message instruction limit (~5 billion instructions on application subnets), every `make_proposal` call traps, permanently freezing governance for the affected SNS.

## Finding Description
`compute_ballots_for_new_proposal` at `rs/sns/governance/src/governance.rs:5255–5280` iterates the full `self.proto.neurons` `BTreeMap`, calling `v.dissolve_delay_seconds(now_seconds)` and `v.voting_power(...)` per neuron with no batching, no early exit, and no instruction-limit check. There is no `is_message_over_threshold` guard, no `noop_self_call_if_over_instructions` call, and no `MAX_NUMBER_OF_NEURONS` enforcement anywhere in `rs/sns/governance/src/governance.rs` (confirmed by search). The function is called unconditionally at `rs/sns/governance/src/governance.rs:3557–3559` inside `make_proposal`, which is the synchronous update-call handler reachable by any principal holding SNS tokens. The same function is also called from `compute_cached_metrics` at `rs/sns/governance/src/governance.rs:5388`. By contrast, NNS governance replaced this pattern with a pre-computed snapshot (`compute_voting_power_snapshot_for_standard_proposal` + `VOTING_POWER_SNAPSHOTS`) computed in a periodic timer task (`SnapshotVotingPowerTask` in `rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`), and `make_proposal` reads from the snapshot instead of iterating neurons live. SNS governance has not received this mitigation.

## Impact Explanation
Once the neuron count in an SNS crosses the threshold where a single pass through `self.proto.neurons` exceeds ~5 billion instructions, every call to `make_proposal` traps before any state mutation occurs. No proposal can ever be submitted again; the SNS governance canister is permanently unable to accept new proposals. This is a complete governance freeze of an in-scope SNS canister, matching the allowed High impact: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS" ($2,000–$10,000).

## Likelihood Explanation
SNS canisters accumulate neurons organically through token swaps, staking, and airdrops. There is no enforced neuron cap in `rs/sns/governance/src/governance.rs`. Any unprivileged principal holding SNS tokens can stake a neuron and call `make_proposal`; no special privilege is required to trigger the loop. A popular SNS with tens of thousands of neurons is realistic. The attacker does not need to cause the neuron count to grow themselves — organic growth suffices — but can accelerate it by staking many small neurons if the SNS minimum stake is low.

## Recommendation
1. **Cache the electoral roll** using the same snapshot mechanism already deployed in NNS governance: compute the voting-power snapshot in a periodic timer task (analogous to `SnapshotVotingPowerTask`) with an instruction-limit guard, and read from the snapshot inside `make_proposal` instead of iterating `self.proto.neurons` live.
2. **Add an instruction-limit guard** inside the loop so that if the loop cannot complete in one message it fails gracefully rather than trapping.
3. **Enforce a maximum neuron count** for SNS governance, analogous to the NNS `MAX_NUMBER_OF_NEURONS` limit, to bound the worst-case iteration cost.

## Proof of Concept
1. Deploy an SNS governance canister (or use an existing one).
2. Stake enough neurons (via token swaps, airdrops, or direct staking) to push `self.proto.neurons.len()` past the instruction-limit threshold.
3. Any principal with a neuron holding `SubmitProposal` permission calls `manage_neuron` with a `MakeProposal` command.
4. `make_proposal` calls `compute_ballots_for_new_proposal` synchronously at `rs/sns/governance/src/governance.rs:3557`.
5. The loop at `rs/sns/governance/src/governance.rs:5255` iterates all N neurons; when N is large enough the update message traps.
6. Every subsequent `make_proposal` call also traps — the SNS is governance-frozen.

A deterministic integration test can reproduce this by pre-populating `self.proto.neurons` with a large synthetic neuron set in a PocketIC or `StateMachine` environment and asserting that `make_proposal` returns a trap/error once the count exceeds the instruction budget.
