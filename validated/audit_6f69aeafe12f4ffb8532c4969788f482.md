### Title
Fully Deterministic `blockhash` Opcode Enables Pre-computation of All Past and Future Block Hashes, Allowing Exploitation of Any EVM Contract Using It as Entropy - (File: engine/src/engine.rs)

### Summary

Aurora Engine's implementation of the EVM `BLOCKHASH` opcode returns a value that is a pure function of three public constants: `chain_id`, `engine_account_id`, and `block_height`. All inputs are globally known on Aurora mainnet (`chain_id = 1313161554`, `account_id = "aurora"`). Any EVM contract deployed on Aurora that uses `blockhash(N)` as a source of randomness — a common Solidity pattern — is completely gameable by any unprivileged attacker who can pre-compute the hash for any past or future block.

### Finding Description

The `compute_block_hash` function in `engine/src/engine.rs` computes Aurora's block hash as:

```
sha256(BLOCK_HASH_PREFIX || chain_id || account_id || block_height)
``` [1](#0-0) 

All inputs are public constants for any given block height. The `block_hash` EVM Backend method calls this function for any block within the last 256 blocks: [2](#0-1) 

Unlike Ethereum, where `blockhash` depends on actual block content (transactions, state root, etc.) and is only known after the block is produced, Aurora's `blockhash(N)` is a deterministic function of `N` alone. An attacker can pre-compute `blockhash(N)` for any future block `N` before submitting any transaction.

The `get_block_hash` NEAR method also exposes this computation publicly to any caller: [3](#0-2) 

The `BLOCK_HASH_PREFIX`, `chain_id`, and `account_id` are all fixed constants: [4](#0-3) 

Additionally, the `RandomSeed` precompile at `0xc104f4840573bed437190daf5d2898c2bdf928ac` explicitly documents that it returns the **same seed for all transactions in the same block**: [5](#0-4) 

This precompile also has a gas cost of exactly zero (`RANDOM_BYTES_GAS = EthGas::new(0)`), with an unresolved TODO: [6](#0-5) 

### Impact Explanation

**Critical — Direct theft of user funds.**

Any EVM contract deployed on Aurora that uses `blockhash(block.number - 1)` or `blockhash(N)` as a source of randomness (a standard Solidity pattern for lotteries, NFT minting, gaming, or commit-reveal schemes) is completely gameable. An attacker can:

1. Pre-compute `sha256(0x00 || chain_id_bytes || b"aurora" || N.to_be_bytes())` for any target block `N`.
2. Know the exact "random" value the contract will receive before submitting any transaction.
3. Selectively submit transactions only when the pre-computed hash produces a favorable outcome (e.g., winning a lottery, minting a rare NFT, winning a game).

This directly steals funds from other participants in such contracts. The `get_block_hash` NEAR view method makes this pre-computation trivially accessible to any off-chain script.

### Likelihood Explanation

**High.** The `blockhash` opcode is a standard Solidity primitive that developers commonly use as a cheap entropy source. Developers porting Ethereum contracts to Aurora may not be aware that Aurora's `blockhash` is fully deterministic and pre-computable, unlike Ethereum's. The `get_block_hash` public view method makes the pre-computation trivially accessible to any attacker with a NEAR RPC call. No special privileges are required.

### Recommendation

1. **Document prominently** that `blockhash(N)` on Aurora is a deterministic function of `N` and must never be used as a source of randomness. The existing documentation reference (`doc.aurora.dev/develop/compat/evm#blockhash`) should include an explicit security warning.
2. **Encourage use of the `RandomSeed` precompile** (`0xc104f4840573bed437190daf5d2898c2bdf928ac`) as the intended entropy source, while also documenting its per-block-constant property and block-producer influence.
3. **Assign a non-zero gas cost** to the `RandomSeed` precompile (resolve `TODO(#483)`) to prevent free entropy-extraction loops.
4. Consider incorporating the `action_hash` (per-transaction) into the `RandomSeed` precompile output so that different transactions in the same block receive different seeds, reducing block-producer influence.

### Proof of Concept

An attacker can pre-compute Aurora's `blockhash` for any future block `N` entirely off-chain:

```python
import hashlib, struct

BLOCK_HASH_PREFIX = b'\x00'
CHAIN_ID = (1313161554).to_bytes(32, 'big')  # Aurora mainnet chain_id
ACCOUNT_ID = b'aurora'

def compute_aurora_blockhash(block_height: int) -> bytes:
    data = BLOCK_HASH_PREFIX + CHAIN_ID + ACCOUNT_ID + struct.pack('>Q', block_height)
    return hashlib.sha256(data).digest()

# Pre-compute blockhash for the next 10,000 blocks
current_height = 120_000_000  # example current NEAR block height
for i in range(10_000):
    h = compute_aurora_blockhash(current_height + i)
    print(f"blockhash({current_height + i}) = {h.hex()}")
```

A Solidity lottery contract on Aurora using `uint256(blockhash(block.number - 1)) % 100 == 0` to determine winners can be gamed by any attacker who runs this script and submits their transaction only when the pre-computed hash produces a winning outcome. [1](#0-0) [2](#0-1)

pub fn compute_block_hash(chain_id: [u8; 32], block_height: u64, account_id: &[u8]) -> H256 {
    debug_assert_eq!(BLOCK_HASH_PREFIX_SIZE, size_of_val(&BLOCK_HASH_PREFIX));
    debug_assert_eq!(BLOCK_HEIGHT_SIZE, size_of_val(&block_height));
    debug_assert_eq!(CHAIN_ID_SIZE, size_of_val(&chain_id));
    let mut data = Vec::with_capacity(
        BLOCK_HASH_PREFIX_SIZE + BLOCK_HEIGHT_SIZE + CHAIN_ID_SIZE + account_id.len(),
    );
    data.push(BLOCK_HASH_PREFIX);
    data.extend_from_slice(&chain_id);
    data.extend_from_slice(account_id);
    data.extend_from_slice(&block_height.to_be_bytes());

    sdk::sha256(&data)
}
```

        let idx = U256::from(self.env.block_height());
        if idx.saturating_sub(U256::from(256)) <= number && number < idx {
            // since `idx` comes from `u64` it is always safe to downcast `number` from `U256`

                number.low_u64(),
