### Title
Unchecked Return Value of `ExitToNear` Precompile Call in `withdrawToNear` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary
The `withdrawToNear` function in `EvmErc20.sol` and `EvmErc20V2.sol` burns the caller's ERC-20 tokens **before** calling the `ExitToNear` precompile via a low-level assembly `call`. The return value of this `call` is captured in `res` but never checked. If the precompile call fails for any reason â€” including an oversized `recipient` byte array that exceeds the precompile's enforced `MAX_INPUT_SIZE` of 1024 bytes â€” the tokens are permanently burned with no corresponding NEP-141 transfer, resulting in irreversible fund loss.

### Finding Description

In `EvmErc20.sol` lines 53â€“63:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens burned first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}

The identical pattern appears in `EvmErc20V2.sol` lines 53â€“64, where the input is `1 + 20 + 32 + recipient.length` bytes: [2](#0-1) 

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` enforces a hard `MAX_INPUT_SIZE` of 1024 bytes: [3](#0-2) 

When the precompile's `validate_input_size` check fails, it returns an `ExitError`. In the EVM, a `call` to a precompile that returns an error causes the `call` opcode to push `0` onto the stack â€” but since `res` is never inspected, execution continues. The `_burn` that already executed is not rolled back, and no NEP-141 transfer is ever scheduled.

For `EvmErc20.sol`: input size = `1 + 32 + recipient.length`; failure threshold is `recipient.length > 991`.
For `EvmErc20V2.sol`: input size = `1 + 20 + 32 + recipient.length`; failure threshold is `recipient.length > 971`.

The `ExitToNear` precompile run path confirms the size validation is the first gate before any promise is scheduled: [4](#0-3) 

### Impact Explanation

**Critical â€” Permanent freezing of funds.** Any user who calls `withdrawToNear` with a `recipient` byte array exceeding the threshold has their ERC-20 tokens burned with no recovery path. The NEP-141 tokens remain locked in the Aurora contract, and the ERC-20 supply is reduced. The user suffers a total, unrecoverable loss of the withdrawn amount.

### Likelihood Explanation

Low-to-medium. A valid NEAR account ID is at most 64 bytes, far below the 971â€“991 byte threshold. However:
- No input validation exists in the Solidity contract before `_burn` executes, so a buggy frontend, a crafted contract interaction, or a user passing raw bytes can trivially exceed the limit.
- The precompile can also fail due to insufficient gas forwarded through the `call` (gas is passed as `gas()` with no minimum guarantee), creating a second failure path that is harder to predict.
- The contract provides zero protection or revert-on-failure semantics, making the silent loss a systemic design flaw rather than an edge case.

### Recommendation

Check the return value of the precompile `call` and revert the entire transaction (including the burn) if it fails:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Additionally, validate `recipient.length` before calling `_burn` to give users a clear error rather than a silent loss.

### Proof of Concept

1. Attacker or user calls `withdrawToNear(recipient, amount)` where `recipient` is a byte array of length > 991 (e.g., 992 bytes of arbitrary data).
2. `_burn(_msgSender(), amount)` executes and permanently reduces the caller's ERC-20 balance.
3. The assembled input (`1 + 32 + 992 = 1025` bytes) is passed to the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`.
4. The precompile's `validate_input_size` check fails (`1025 > MAX_INPUT_SIZE = 1024`), returning an `ExitError`. The EVM `call` opcode pushes `0` onto the stack.
5. `res` is never read; the assembly block exits silently.
6. The transaction succeeds. The caller's ERC-20 tokens are gone. No NEP-141 transfer promise is ever created. Funds are permanently lost.
