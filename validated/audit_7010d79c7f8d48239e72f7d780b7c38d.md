### Title
ckERC20 Withdrawal Accounting Trusts EVM Transaction Status Without Verifying ERC20 `transfer` Return Value - (File: rs/ethereum/cketh/minter/src/state.rs)

### Summary
The ckETH minter finalizes ERC20 withdrawal accounting and considers a withdrawal complete based solely on the EVM-level `TransactionStatus::Success` from the transaction receipt. It never inspects the actual return value of the ERC20 `transfer(address,uint256)` call. For non-standard ERC20 tokens that return `false` on failure without reverting (causing EVM status = `0x1` / Success), the minter incorrectly burns the user's ckERC20 tokens, decrements its internal ERC20 balance, and skips reimbursement — while no actual ERC20 transfer occurred on Ethereum.

### Finding Description

The ckETH minter sends ERC20 withdrawals as raw EIP-1559 transactions encoding the `transfer(address,uint256)` selector. After the transaction is mined, `finalize_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs` fetches the receipt via `eth_getTransactionReceipt` and calls `process_event(EventType::FinalizedTransaction{...})`. [1](#0-0) 

The receipt struct contains only: `block_hash`, `block_number`, `effective_gas_price`, `gas_used`, `status`, `transaction_hash`. There is no `output` / return-data field. [2](#0-1) 

`update_balance_upon_withdrawal` then branches exclusively on `receipt.status`:

```rust
if receipt.status == TransactionStatus::Success && !tx.transaction_data().is_empty() {
    let TransactionCallData::Erc20Transfer { to: _, value } = TransactionCallData::decode(
        tx.transaction_data(),
    )...;
    self.erc20_balances.erc20_sub(*tx.destination(), value);
}
``` [3](#0-2) 

Simultaneously, `record_finalized_transaction` only enqueues a reimbursement when `receipt.status == TransactionStatus::Failure`:

```rust
WithdrawalRequest::CkErc20(request) => {
    if receipt.status == TransactionStatus::Failure {
        self.record_reimbursement_request(...);
    }
}
``` [4](#0-3) 

The EVM `status = 0x1` (Success) only means the transaction did not revert. It does **not** mean the ERC20 `transfer` returned `true`. For tokens that return `false` without reverting, the EVM status is `0x1` yet no tokens moved. The minter has no mechanism to distinguish this case because `eth_getTransactionReceipt` does not expose the call's return data.

The minter encodes only the standard `transfer(address,uint256)` selector: [5](#0-4) 

### Impact Explanation

For any supported ckERC20 token whose `transfer` implementation returns `false` on failure without reverting:

1. The user's ckERC20 tokens are burned on the IC ledger (irreversible).
2. The minter decrements `erc20_balances` for that token contract.
3. No reimbursement is triggered (only fires on `TransactionStatus::Failure`).
4. The actual ERC20 balance on Ethereum is unchanged — the user receives nothing.
5. The minter's internal ERC20 accounting permanently undercounts its real Ethereum holdings, causing future legitimate withdrawals to be incorrectly rejected or miscounted.

This is a **chain-fusion burn/accounting bug**: ckERC20 tokens are destroyed on the IC without a corresponding ERC20 transfer on Ethereum, breaking the 1:1 peg invariant for affected tokens.

### Likelihood Explanation

Currently supported tokens (ckUSDC, ckUSDT, ckPEPE, etc.) revert on failure and do not exhibit this behavior. However:

- New ckERC20 tokens are added via NNS governance proposals without on-chain verification that the token's `transfer` is revert-on-failure compliant.
- The minter documentation explicitly warns that the helper contract does not enforce a whitelist and that the minter only filters by supported token address — not by token behavior. [6](#0-5) 

- Any future token added via governance that has non-standard `transfer` semantics (returning `false` without reverting) would silently trigger this bug for every user withdrawal of that token.

Likelihood is **low** given current token set, but the code path is reachable by any unprivileged chain-fusion user withdrawing a non-standard ckERC20 token.

### Recommendation

1. After a withdrawal transaction is confirmed with `TransactionStatus::Success`, the minter should additionally verify the ERC20 `transfer` return value by issuing an `eth_call` replay of the same transaction at the confirmed block number and decoding the returned bytes. If the return data is non-empty and decodes to `false`, treat the withdrawal as failed and trigger reimbursement.
2. Alternatively, require that all supported ERC20 tokens emit a `Transfer(from, to, value)` event and validate its presence in the transaction receipt logs as a secondary confirmation that tokens actually moved.
3. Document and enforce (at the governance proposal validation layer) that only ERC20 tokens whose `transfer` reverts on failure are eligible for ckERC20 support.

### Proof of Concept

1. An NNS proposal adds a ckERC20 token whose Ethereum `transfer(address,uint256)` returns `false` on insufficient balance without reverting (e.g., a token modeled after early non-standard ERC20 implementations).
2. A user deposits 100 units of this token via the helper contract → minter mints 100 ckTOKEN.
3. User calls `withdraw_erc20` for 100 units → minter burns 100 ckTOKEN on the IC ledger.
4. Minter sends the EIP-1559 `transfer` transaction to Ethereum. The token contract returns `false` (e.g., due to a fee-on-transfer mechanism that leaves balance insufficient) but does not revert → EVM receipt `status = 0x1`.
5. `update_balance_upon_withdrawal` sees `TransactionStatus::Success`, calls `erc20_balances.erc20_sub(...)`, and records `FinalizedTransaction` with no reimbursement.
6. User's 100 ckTOKEN are gone; their Ethereum address received 0 tokens. The minter's `erc20_balances` for this token is now negative relative to reality, corrupting future withdrawal capacity. [7](#0-6) [8](#0-7)

            WithdrawalRequest::CkEth(req) => req

                .expect("BUG: withdrawal amount MUST always be at least the transaction amount"),
            WithdrawalRequest::CkErc20(req) => req.max_transaction_fee,
        };

            "BUG: charged transaction fee MUST always be at least the effective transaction fee",
        );
        let debited_amount = match receipt.status {
            TransactionStatus::Success => tx
                .transaction()
                .amount
                .checked_add(tx_fee)
                .expect("BUG: debited amount always fits into U256"),
            TransactionStatus::Failure => tx_fee,
        };

        self.eth_balance.total_effective_tx_fees_add(tx_fee);
        self.eth_balance.total_unspent_tx_fees_add(unspent_tx_fee);

        if receipt.status == TransactionStatus::Success && !tx.transaction_data().is_empty() {
            let TransactionCallData::Erc20Transfer { to: _, value } = TransactionCallData::decode(
                tx.transaction_data(),
            )
            .expect("BUG: failed to decode transaction data from transaction issued by minter");
            self.erc20_balances.erc20_sub(*tx.destination(), value);

// First 4 bytes of keccak256(transfer(address,uint256))
const ERC_20_TRANSFER_FUNCTION_SELECTOR: [u8; 4] = hex_literal::hex!("a9059cbb");

#[derive(Clone, Eq, PartialEq, Debug)]
pub enum TransactionCallData {
    Erc20Transfer { to: Address, value: Erc20Value },
}

impl TransactionCallData {
    /// Encode the transaction call data to interact with an Ethereum smart contract.
    /// See the [Contract ABI Specification](https://docs.soliditylang.org/en/develop/abi-spec.html#contract-abi-specification).
    pub fn encode(&self) -> Vec<u8> {
        match self {
            TransactionCallData::Erc20Transfer { to, value } => {
                let mut data = Vec::with_capacity(68);
                data.extend(ERC_20_TRANSFER_FUNCTION_SELECTOR);
                data.extend(<[u8; 32]>::from(to));
                data.extend(value.to_be_bytes());
                data
            }
        }
    }
