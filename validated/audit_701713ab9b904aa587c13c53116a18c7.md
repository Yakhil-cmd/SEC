Audit Report

## Title
`burn_and_log` Silently Discards ICP Ledger Burn Failure, Enabling Cycles Minting Without ICP Destruction - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The CMC function `burn_and_log` issues a `send_pb` call to the ICP ledger to destroy ICP after cycles have already been minted and delivered. On any call error, the function only logs and returns `()` — no error is propagated, no retry is scheduled, and no compensating action is taken. Because the block is marked as fully processed before `burn_and_log` is awaited, the user cannot re-notify to trigger the burn again, leaving ICP permanently unburned in the CMC subaccount while the corresponding cycles remain in circulation.

## Finding Description
Three notification flows call `burn_and_log` as their final step after the primary operation has already succeeded and the block status has been advanced past `Processing`:

- `process_create_canister` (L1943–1946): after `do_create_canister` returns `Ok`
- `process_mint_cycles` (L1966–1968): after `do_mint_cycles` returns `Ok`
- `process_top_up` (L1999–2001): after `deposit_cycles` returns `Ok`

`burn_and_log` (L2017–2049) issues `call_protobuf(ledger_canister_id, "send_pb", send_args).await` and on `Err` only prints a log line. The code comment at L2014–2016 explicitly acknowledges this: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."*

The block-status machinery (`set_block_status_to_processing` / `clear_block_processing_status`) ensures that once a block reaches `NotifiedTopUp` / `NotifiedCreateCanister` / `NotifiedMint`, any subsequent notification attempt is rejected with `InvalidTransaction`. There is no pending-burn queue, no heartbeat retry, and no governance recovery path. A failed burn is therefore permanent.

The claim that the burn is idempotent and safe to retry is correct: a second `send_pb` from a subaccount that was already successfully burned would fail with insufficient funds, not produce a double burn. The stated rationale for suppressing errors applies to the primary operation (cycles delivery), not to the burn step.

## Impact Explanation
When `burn_and_log` fails, cycles have been minted and delivered while the corresponding ICP remains unburned in the CMC subaccount. This constitutes illegal minting of cycles without the required ICP destruction, breaking the ICP↔cycles supply conservation invariant. The unburned ICP accumulates across occurrences and cannot be recovered without a governance-level intervention. This matches the allowed High impact: *"Significant … ledger … security impact with concrete user or protocol harm."*

## Likelihood Explanation
The ICP ledger is upgraded periodically via NNS governance proposals. During an upgrade the ledger canister is briefly stopped and rejects all calls. The upgrade proposal and execution are observable on-chain, making the timing window predictable. Any unprivileged user who submits a `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` call whose primary operation completes just before or during the ledger upgrade window will trigger this path. No special privilege, key material, or attacker-controlled infrastructure is required. The window is short but repeatable across every ledger upgrade.

## Recommendation
Redesign `burn_and_log` to handle failures durably:

1. Before marking the block as fully processed, persist the pending burn (subaccount + amount) in CMC stable state.
2. Retry the burn in a heartbeat or timer task until it succeeds. A duplicate burn on an already-burned subaccount will fail with insufficient funds, so retries are safe.
3. Alternatively, expose a governance-callable endpoint to drain accumulated unburned subaccounts.

The no-retry rationale in the comment applies only to the primary operation (cycles delivery). The burn step is idempotent and does not risk double-minting.

## Proof of Concept
1. User transfers ICP to the CMC subaccount for a target canister with `MEMO_TOP_UP_CANISTER`.
2. An NNS upgrade proposal for the ICP ledger is submitted and passes; the ledger enters its brief stopped state.
3. User calls `notify_top_up` on the CMC.
4. CMC calls `deposit_cycles` → succeeds; cycles are delivered to the target canister.
5. CMC calls `burn_and_log` → `call_protobuf` to the stopped ledger returns a reject code.
6. `burn_and_log` logs the error and returns `()`.
7. CMC records `NotificationStatus::NotifiedTopUp(Ok(cycles))` for the block.
8. Any subsequent `notify_top_up` for the same block is rejected with `InvalidTransaction("Block has already been processed")`.
9. The ICP remains in the CMC subaccount indefinitely; the cycles are in the target canister.

A deterministic integration test can reproduce this by: (a) deploying CMC + ledger in a PocketIC instance, (b) stopping the ledger canister to simulate an upgrade, (c) calling `notify_top_up`, and (d) asserting that the CMC subaccount balance is non-zero while the target canister received cycles.
