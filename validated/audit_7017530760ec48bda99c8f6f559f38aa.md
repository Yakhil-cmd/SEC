Audit Report

## Title
Predictable ChaCha20 RNG Seed Combined with Existence Oracle in `get_rule_by_id` Enables Full Confidential Rule Enumeration — (`rs/boundary_node/rate_limits/canister/random.rs`, `rs/boundary_node/rate_limits/canister/getter.rs`)

## Summary
The rate-limit canister seeds its ChaCha20 RNG with only 64 bits of public entropy (the IC block timestamp), leaving the remaining 24 bytes as the constant `0x2a`. Because the IC block timestamp is publicly readable via `read_state`, any observer can reconstruct the exact seed and predict every rule UUID the canister will ever generate. Combined with the fact that `get_rule_by_id` returns `Ok` — not `Err(NotFound)` — for confidential undisclosed rules, leaking `incident_id`, `added_in_version`, and `removed_in_version`, an unprivileged attacker can enumerate all confidential rules and harvest their metadata without any authorization, breaking the stated confidentiality invariant of the canister.

## Finding Description

**Root cause 1 — Weak RNG seed (`random.rs` L8–14):**
The `thread_local!` initializer fills all 32 seed bytes with `42` (`0x2a`), then overwrites only the first 8 bytes with `time().to_le_bytes()`. The resulting seed is `T.to_le_bytes() || [0x2a; 24]`, where `T` is the IC nanosecond block timestamp at the time of first RNG access. The IC block timestamp is part of the certified state tree and is publicly readable by any caller via `read_state` — it is not a secret. The RNG is never re-seeded after initialization.

**Root cause 2 — UUID generation routes through the weak RNG (`add_config.rs` L186–193):**
`generate_random_uuid()` calls `getrandom::getrandom`, which on `wasm32-unknown-unknown` is registered to `custom_getrandom_bytes_impl` (`random.rs` L22–29). All UUID bytes therefore come from the same deterministic ChaCha20 stream seeded with the public timestamp.

**Root cause 3 — `get_rule_by_id` is an existence oracle for confidential rules (`getter.rs` L219–245):**
For a non-existent rule the call returns `Err(NotFound)`. For an existing but confidential (non-disclosed) rule, the code constructs a full `OutputRuleMetadata` with all fields populated, then calls `formatter.format()`. The `RuleConfidentialityFormatter` (`confidentiality_formatting.rs` L34–41) only sets `rule_raw = None` and `description = None` when `disclosed_at.is_none()` — it does **not** suppress `rule_id`, `incident_id`, `added_in_version`, or `removed_in_version`. The function then returns `Ok(output_rule.into())`. There is no access-level check that returns `Err(NotFound)` for confidential rules when called by an unauthorized principal.

This is confirmed by the unit test at `getter.rs` L433–446, which asserts that an unauthorized (`RestrictedRead`) caller receives `Ok` with `incident_id`, `added_in_version`, and `removed_in_version` populated for a non-disclosed rule.

**Exploit flow:**
1. Read the IC block timestamp `T` from `read_state` (public, certified). The `active_since` field of the first config version that added rules also exposes this value.
2. Reconstruct the 32-byte seed: `T.to_le_bytes() || [0x2a; 24]`.
3. Instantiate `ChaCha20Rng::from_seed(seed)` locally and generate the same UUID sequence the canister produces.
4. For each predicted UUID, call the public `get_rule_by_id` query.
5. Distinguish `Ok` (rule exists, confidential) from `Err(NotFound)` (rule not yet added).
6. From every `Ok` response, harvest `incident_id`, `added_in_version`, and `removed_in_version` — revealing the existence, timing, and scope of active security incidents tied to undisclosed rate-limit rules.

**Existing checks are insufficient:** The `WithAuthorization` wrapper only guards write operations (`add_config`, `disclose_rules`). The `get_rule_by_id` query is publicly callable by any principal with `RestrictedRead` access (the default for all callers). The formatter redacts content fields but does not suppress the `Ok` response itself.

## Impact Explanation
This is a **High** severity finding. The rate-limit canister is a boundary node component whose stated security property is that undisclosed rules are invisible to unauthorized callers. This vulnerability breaks that invariant: any unprivileged IC user can enumerate all confidential rules and learn their `incident_id`, `added_in_version`, and `removed_in_version` — revealing the existence, timing, and scope of active security incidents being mitigated by undisclosed rate-limit rules. This constitutes a significant boundary/API security impact with concrete harm to the confidentiality of security incident response. It maps to the allowed High impact: "Significant... boundary/API... security impact with concrete user or protocol harm."

## Likelihood Explanation
The attack requires no privileges, no key material, and no network-level capabilities. The only input needed — the IC block timestamp — is freely available from any IC node via `read_state`. The exploit is entirely local and deterministic. The canister's RNG state is never re-seeded after initialization (no upgrade re-seed, no `raw_rand` injection), so the full UUID sequence for the canister's lifetime is fixed from the moment of first RNG access. Any IC user can execute this attack at any time.

## Recommendation
**Fix 1 — Replace the time-seeded RNG:** Use `ic_cdk::api::management_canister::main::raw_rand()` asynchronously during `init`/`post_upgrade` to obtain at least 128 bits of unpredictable entropy and store it as the RNG seed. The seed must not be derivable from any publicly observable state.

**Fix 2 — Eliminate the existence oracle:** In `RuleGetter::get`, return `Err(GetEntityError::NotFound(...))` for confidential (non-disclosed) rules when called by an unauthorized principal, rather than returning `Ok` with redacted fields. The check should occur before constructing the `OutputRuleMetadata` response, so that the response is indistinguishable from a genuinely absent rule.

use rand_chacha::rand_core::{RngCore, SeedableRng};
use uuid::Uuid;

fn predict_uuids(first_add_config_timestamp_ns: u64, count: usize) -> Vec<Uuid> {
    let mut seed = [42u8; 32];
    seed[..8].copy_from_slice(&first_add_config_timestamp_ns.to_le_bytes());
    let mut rng = ChaCha20Rng::from_seed(seed);
    (0..count).map(|_| {
        let mut buf = [0u8; 16];
        rng.fill_bytes(&mut buf);
        Uuid::from_slice(&buf).unwrap()
    }).collect()
}

//    - Err(NotFound) => not yet added
