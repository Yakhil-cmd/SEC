Audit Report

## Title
Permanent ckBTC Loss on `AmountTooLow` Finalization Due to Missing Reimbursement Path - (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary
The ckBTC minter burns the full user-specified ckBTC amount synchronously at request submission time, but determines the actual Bitcoin network fee only at batch processing time using the then-current fee rate. When the computed fee exceeds the withdrawal amount (`BuildTxError::AmountTooLow`), the request is finalized with `FinalizedStatus::AmountTooLow` and no ckBTC reimbursement is issued — permanently destroying the user's funds. This is an asymmetric treatment: the `BuildTxError::InvalidTransaction` path in the same function does trigger `reimburse_canceled_requests`, but `AmountTooLow` does not.

## Finding Description

**Step 1 — ckBTC burned immediately and irrevocably at request time:**

`burn_ckbtcs` is called before the request is enqueued. The `RetrieveBtcRequest` struct stores only `amount`, `address`, `block_index`, `received_at`, `kyt_provider`, and `reimbursement_account` — no `max_fee` or `min_receive_amount` field exists. [1](#0-0) [2](#0-1) 

**Step 2 — Fee determined at batch processing time using the current fee rate:**

`submit_pending_requests` calls `estimate_fee_per_vbyte` at the moment of batch construction, not at request submission time. [3](#0-2) 

**Step 3 — `AmountTooLow` path silently drops requests with no reimbursement:**

When `fee + minter_fee > amount`, `BuildTxError::AmountTooLow` is returned. The handler calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` and issues no reimbursement. The code comment explicitly states "There is no point in retrying the request because the amount is too low" — but this rationale does not address the missing reimbursement. [4](#0-3) [5](#0-4) 

**Step 4 — Asymmetry: `InvalidTransaction` path DOES reimburse:**

In the same `match` block, `BuildTxError::InvalidTransaction` calls `reimburse_canceled_requests`, demonstrating that the reimbursement infrastructure exists and is used for other error paths — but is deliberately absent for `AmountTooLow`. [6](#0-5) 

**Step 5 — Batching delay creates the fee-spike exposure window:**

`can_form_a_batch` allows requests to sit in the queue until `max_time_in_queue_nanos` elapses for the oldest request, creating a window of potentially hours during which Bitcoin fees can change dramatically. [7](#0-6) 

## Impact Explanation

A user submitting a `retrieve_btc` request during a low-fee period who has their request processed during a high-fee spike (e.g., Ordinals inscription waves, halving events) can have their entire ckBTC balance permanently destroyed with zero BTC received and zero reimbursement. The burned ckBTC is removed from the ledger supply with no corresponding BTC output and no recovery path. This constitutes a permanent, non-recoverable loss of in-scope ck-token assets for affected users. This matches the allowed High impact: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."

## Likelihood Explanation

Bitcoin fee spikes of 10–100× are well-documented recurring events on mainnet. The `max_time_in_queue_nanos` parameter is configurable and used in production, meaning requests legitimately queue for hours. Any unprivileged user submitting a near-minimum withdrawal during a low-fee period is exposed. No special privileges, social engineering, or external compromise is required — the user only needs to call `retrieve_btc` or `retrieve_btc_with_approval`. The `estimate_withdrawal_fee` query endpoint is advisory only and does not bind the minter to any fee ceiling.

## Recommendation

Add an optional `max_fee` field to `RetrieveBtcArgs` and `RetrieveBtcWithApprovalArgs`, and store it in `RetrieveBtcRequest`. In the `BuildTxError::AmountTooLow` handler in `submit_pending_requests`, instead of silently finalizing, call `reimburse_canceled_requests` (as is already done for `InvalidTransaction`) to return the burned ckBTC to the user's `reimbursement_account`. Alternatively, reschedule the request back to the pending queue (as ckETH does for `InsufficientTransactionFee`) rather than permanently dropping it, so the request can be retried when fees normalize. [8](#0-7) 

## Proof of Concept

1. Bitcoin mainnet fee rate is 5 sat/vbyte. User calls `retrieve_btc_with_approval(address="bc1q...", amount=100_000)`. The full 100,000 satoshi of ckBTC is burned immediately via `burn_ckbtcs`.
2. The request enters `pending_retrieve_btc_requests` with `received_at = T`.
3. No other requests arrive. The minter waits until `max_time_in_queue_nanos` elapses (hours).
4. During this window, a Bitcoin congestion event occurs; fee rate rises to 800 sat/vbyte.
5. `submit_pending_requests` fires, calls `estimate_fee_per_vbyte` → returns 800 sat/vbyte.
6. `build_unsigned_transaction` computes `fee ≈ 800 * ~140 vbytes = ~112,000 satoshi`.
7. Since `112,000 + minter_fee > 100,000`, `BuildTxError::AmountTooLow` is returned.
8. The `AmountTooLow` handler calls `remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow)` — no `reimburse_canceled_requests` call.
9. The user has permanently lost 100,000 satoshi of ckBTC with zero BTC received and zero reimbursement.

A deterministic integration test can reproduce this by: (a) setting up a minter state with a single pending request at a low amount, (b) injecting a high fee rate via the mock runtime's `fee_estimator`, (c) calling `submit_pending_requests`, and (d) asserting that `retrieve_btc_status` returns `AmountTooLow` and that no reimbursement event was emitted in the event log. [4](#0-3) [9](#0-8)

        }

        if let Some(req) = self.pending_retrieve_btc_requests.first()
            && self.max_time_in_queue_nanos < now.saturating_sub(req.received_at)

            return true;
        }

        false
