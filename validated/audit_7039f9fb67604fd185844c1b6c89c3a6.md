### Title
Staged Upgrade Cancellation Can Be Front-Run by Any NEAR Account - (File: `engine/src/lib.rs`)

### Summary

The Aurora Engine implements a two-step upgrade process: `stage_upgrade` (owner-only) stores new contract code and records a minimum deployment block height, and `deploy_upgrade` (callable by **anyone**) deploys the staged code once the delay has passed. If the owner discovers a problem with the staged code after the delay has elapsed and attempts to overwrite it by calling `stage_upgrade` again, any NEAR account can race to call `deploy_upgrade` first, forcing deployment of the original (potentially buggy or malicious) code before the owner's corrective transaction is processed.

### Finding Description

`stage_upgrade` is restricted to the owner and stores the new WASM code at `CODE_KEY` while writing the minimum deployment block height to `CODE_STAGE_KEY`: [1](#0-0) 

`deploy_upgrade` has **no caller restriction**. It only checks that the contract is running and that the current block height exceeds the stored index: [2](#0-1) 

This was made intentionally open per the v2.9.3 changelog: [3](#0-2) 

The race window is: from the block at which `block_height > index` becomes true, until the owner's replacement `stage_upgrade` transaction is included in a block. During this window, any NEAR account can call `deploy_upgrade` and deploy the originally staged code, bypassing the owner's attempt to cancel or replace it.

The only mechanism available to the owner to "cancel" a staged upgrade is to call `stage_upgrade` again with different code, which overwrites `CODE_KEY` and resets the delay. However, once the delay has already elapsed, this race is losable.

### Impact Explanation

If the owner stages code that is later found to contain a vulnerability (e.g., a bug that allows fund theft or a logic error that freezes balances), and the delay has already passed, any external NEAR account can force deployment of that code before the owner can overwrite it. The Aurora Engine controls all ETH and ERC-20 token balances bridged from Ethereum. Deploying malicious or buggy WASM code to the engine contract can result in:

- **Critical**: Direct theft of all bridged user funds held in the engine.
- **Critical**: Permanent freezing of all bridged funds.

### Likelihood Explanation

The precondition is that the owner has staged code that they subsequently want to cancel. This is a realistic operational scenario (e.g., a post-staging security review reveals a bug). Once the delay elapses, the race window is open to any NEAR account. On NEAR, block times are approximately one second, so the window between the delay expiring and the owner's corrective transaction being included is narrow but real, especially if the owner is not monitoring continuously. A motivated attacker watching the mempool/chain state can reliably win this race.

### Recommendation

Add a caller restriction to `deploy_upgrade` so that only the owner (or the contract itself) can trigger deployment, consistent with how `stage_upgrade` is protected. Alternatively, introduce an explicit `cancel_upgrade` function (owner-only) that clears `CODE_KEY` and `CODE_STAGE_KEY`, giving the owner an atomic cancellation path that cannot be front-run.

```rust
// In deploy_upgrade, add before the block-height check:
require_owner_only(&state, &io.predecessor_account_id())
    .map_err(ContractError::msg)
    .sdk_unwrap();
```

### Proof of Concept

1. Owner calls `stage_upgrade` with code `X` at block `B`. `upgrade_delay_blocks = D`. The stored index is `B + D`.
2. Block `B + D + 1` arrives. The delay has elapsed; `deploy_upgrade` is now callable.
3. Owner discovers a critical bug in code `X` and broadcasts a new `stage_upgrade` transaction with safe code `Y`.
4. Before the owner's transaction is included, an attacker calls `deploy_upgrade`.
5. `deploy_upgrade` passes all checks (contract is running, `block_height > index`) and calls `Runtime::self_deploy` with code `X`. [4](#0-3) 

6. The engine is now running code `X`. The owner's subsequent `stage_upgrade(Y)` transaction either fails (if code `X` removed the method) or succeeds but the damage is already done.
7. If code `X` contains a fund-draining or fund-freezing bug, all bridged assets are at risk.

```rust
#[named]
pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}
```

```rust
    /// Deploy staged upgrade.
    #[unsafe(no_mangle)]
    pub extern "C" fn deploy_upgrade() {
        // This function is intentionally not implemented in `contract_methods`
        // because it only makes sense in the context of the NEAR runtime.
        let mut io = Runtime;
        let state = state::get_state(&io).sdk_unwrap();
        require_running(&state)
            .map_err(ContractError::msg)
            .sdk_unwrap();
        let index = internal_get_upgrade_index();
        if io.block_height() <= index {
            sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
        }
        Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
    }
```
