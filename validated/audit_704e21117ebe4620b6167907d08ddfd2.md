### Title
`new_settlement_layer_chain_id_storage` Snapshot Not Included in `finish_io_frame` Rollback — (`File: basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIO` holds six sub-storages. `start_io_frame` / `finish_io_frame` snapshot and roll back five of them, but silently omit `new_settlement_layer_chain_id_storage`. Because `NewSettlementLayerChainIdStorage` exposes `start_frame` / `finish_frame` backed by a `HistoryCounter`, the omission is not intentional design — it is a missing call. When a frame that triggered a settlement-layer chain-ID update is rolled back, the update survives in `new_settlement_layer_chain_id_storage` while the underlying storage slot write is correctly reverted. At batch finalization the consistency assertion in `read_batch_context_inputs` then panics, making the entire batch unprovable.

---

### Finding Description

`FullIO` owns six sub-storages: [1](#0-0) 

`FullIOStateSnapshot` captures handles for five of them: [2](#0-1) 

`start_io_frame` calls `start_frame` on five sub-storages but never on `new_settlement_layer_chain_id_storage`: [3](#0-2) 

`finish_io_frame` rolls back five sub-storages but never calls `new_settlement_layer_chain_id_storage.finish_frame(...)`: [4](#0-3) 

`NewSettlementLayerChainIdStorage` explicitly provides `start_frame` and `finish_frame` backed by a `HistoryCounter`, proving rollback was intended: [5](#0-4) 

The chain-ID update is written by `update_settlement_layer_chain_id`, which is called from the `system_context_event_hook` whenever a `SettlementLayerChainIdUpdated` event is observed: [6](#0-5) 

At batch finalization, `read_batch_context_inputs` asserts that the value stored in `new_settlement_layer_chain_id_storage` matches the chain-ID slot read from persistent storage: [7](#0-6) 

If the frame that emitted the event is rolled back, the persistent storage write is correctly reverted (via `storage.finish_frame`), but `new_settlement_layer_chain_id_storage` retains the new value. The assertion therefore compares the new chain ID (stale, from the reverted frame) against the old chain ID (correctly reverted in storage), and panics.

---

### Impact Explanation

**Classification:** Storage rollback bug / state-transition divergence.

A reverted frame that contained a `SettlementLayerChainIdUpdated` event leaves `new_settlement_layer_chain_id_storage` in a permanently inconsistent state for the remainder of the block. The `assert_eq!` in `read_batch_context_inputs` panics unconditionally, making the batch impossible to finalize or prove. This is a liveness / validity break for the entire batch.

---

### Likelihood Explanation

The trigger requires a service transaction that:
1. Calls `setSettlementLayerChainId` on the SystemContext contract (emitting the event, which fires the hook and writes to `new_settlement_layer_chain_id_storage`), and
2. Reverts after that point (e.g., runs out of gas, hits a REVERT, or a sub-call fails).

Service transactions are operator-controlled, but the bug can also be reached accidentally (e.g., gas exhaustion after the LOG opcode but before the transaction completes). Because `NewSettlementLayerChainIdStorage::update` enforces a single-update-per-block invariant, a second legitimate update attempt in the same block would also fail with an internal error, compounding the impact.

---

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and call its `start_frame` / `finish_frame` in `start_io_frame` / `finish_io_frame`, mirroring the pattern already used for `interop_root_storage`:

```rust

// finish_io_frame
self.new_settlement_layer_chain_id_storage
```

### Proof of Concept

1. Operator submits a service transaction that calls `setSettlementLayerChainId(newId)` on the SystemContext contract.
2. The EVM executes the LOG opcode for `SettlementLayerChainIdUpdated(newId)`.
3. `system_context_event_hook` fires → `update_settlement_layer_chain_id(newId)` → `new_settlement_layer_chain_id_storage` now holds `Some(newId)`.
4. The transaction subsequently REVERTs (e.g., gas exhaustion or explicit revert).
5. `finish_io_frame` is called with a rollback handle:
   - `storage.finish_frame(...)` reverts the storage slot write → chain-ID slot still holds `oldId`.
   - `events_storage.finish_frame(...)` removes the emitted event.
   - `new_settlement_layer_chain_id_storage.finish_frame(...)` is **never called** → value remains `Some(newId)`.
6. At batch finalization, `read_batch_context_inputs` reads `settlement_layer_chain_id = oldId` from storage and checks `assert_eq!(newId, &oldId)` → **panic** → batch cannot be finalized or proven. [8](#0-7) [5](#0-4) [7](#0-6)
