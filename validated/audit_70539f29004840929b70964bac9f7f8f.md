### Title
Unvalidated Zero `refund_recipient` in L1→L2 Transactions Causes Permanent Gas-Refund Fund Loss - (`basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The ZKsync OS bootloader processes L1→L2 (priority) transactions without validating that the `refund_recipient` field (`reserved[1]`) is non-zero. When this field is `address(0)` — which is the default when not explicitly set — the unused-gas refund is minted to `address(0)` and permanently burned. The `validate_structure` function contains an explicit `// TODO: validate address?` comment acknowledging this gap. This is a direct analog to the RouterV2 zero-receiver bug: a recipient field that defaults to zero is used without validation, causing permanent fund loss.

---

### Finding Description

L1→L2 transactions in ZKsync OS carry a `refund_recipient` address encoded in `reserved[1]` of the ABI-encoded transaction format. After execution, the bootloader computes the unused-gas refund and mints it to this address via `mint_base_token` → `transfer_from_treasury`.

The `validate_structure` function in `AbiEncodedTransaction` explicitly skips validation of `reserved[1]`:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
``` [1](#0-0) 

The refund recipient is then extracted and used unconditionally:

```rust
let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
mint_base_token::<S, Config>(
    system, system_functions, memories.reborrow(),
    &to_refund_recipient,
    &refund_recipient,   // ← can be B160::ZERO
    ...
)
``` [2](#0-1) 

The `u256_to_b160_checked` helper only asserts that the upper bits are zero (valid address encoding), but does **not** reject `B160::ZERO`:

```rust
pub fn u256_to_b160_checked(src: U256) -> B160 {
    assert!(src.as_limbs()[3] == 0 && src.as_limbs()[2] < (1u64 << 32));
    // ... no zero-address check
}
``` [3](#0-2) 

`transfer_from_treasury` also performs no zero-address check before crediting the recipient: [4](#0-3) 

The test infrastructure confirms this path is live: `L1TxBuilder::new()` defaults `refund_recipient` to `address(0)` via `unwrap_or_default()`, and the regression test explicitly uses `address(0)` as the refund recipient and asserts the refund is sent there: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Any L1→L2 transaction submitted with `reserved[1] = 0` (the default when the caller does not explicitly set a refund recipient) will have its entire unused-gas refund minted to `address(0)` and permanently burned. The refund amount is `(gas_limit - gas_used) * gas_price`. For transactions with large gas limits and low actual gas consumption, this can represent a substantial loss of base tokens. The funds are irrecoverable.

---

### Likelihood Explanation

The `L1TxBuilder` test helper defaults `refund_recipient` to `address(0)`. Any integration, bridge contract, or user that submits an L1→L2 transaction without explicitly setting `reserved[1]` will silently burn their refund. The `// TODO: validate address?` comment in `validate_structure` confirms the developers identified this gap but left it unresolved. The path is reachable by any unprivileged sender of an L1→L2 priority transaction.

---

### Recommendation

1. In `validate_structure`, reject transactions where `reserved[1]` decodes to `B160::ZERO` (remove the `TODO` and add the check).
2. Alternatively, in `process_l1_transaction`, fall back to `transaction.from` when `reserved[1]` is zero, mirroring the intent of the `_to` parameter in the RouterV2 analog.
3. Add a zero-address guard in `transfer_from_treasury` as a defense-in-depth measure.

---

### Proof of Concept

1. Submit an L1→L2 transaction with `reserved[1] = U256::ZERO` (refund recipient not set), `gas_limit = 100_000`, `gas_price = 1_000`, and a simple ETH transfer that uses ~21,000 gas.
2. The bootloader computes `to_refund_recipient = (100_000 - 21_000) * 1_000 = 79,000,000` tokens.
3. `refund_recipient = u256_to_b160_checked(U256::ZERO)` → `B160::ZERO`.
4. `mint_base_token(..., &79_000_000, &B160::ZERO, ...)` → `transfer_from_treasury` credits `address(0)` with 79,000,000 tokens.
5. The sender receives no refund; 79,000,000 tokens are permanently burned.

The existing regression test at line 1843 already demonstrates this exact scenario and treats it as correct behavior, confirming the path is live and unguarded. [7](#0-6)

pub fn u256_to_b160_checked(src: U256) -> B160 {
    assert!(src.as_limbs()[3] == 0 && src.as_limbs()[2] < (1u64 << 32));
    let mut result = B160::ZERO;
    unsafe {
        result.as_limbs_mut()[0] = src.as_limbs()[0];
        result.as_limbs_mut()[1] = src.as_limbs()[1];
        result.as_limbs_mut()[2] = src.as_limbs()[2];
    }

    result
}
