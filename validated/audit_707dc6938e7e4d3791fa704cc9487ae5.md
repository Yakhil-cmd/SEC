### Title
Missing Error-Handling Callback on NEAR Transfer in wNEAR Unwrap Flow Causes Permanent Fund Freeze — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The `exit_to_near_precompile_callback` function, which handles the wNEAR-to-NEAR unwrap path of the `ExitToNear` precompile, creates a final NEAR `Transfer` promise with no attached error-handling callback. If that transfer fails, the user's wNEAR ERC-20 tokens are already burned and the unwrapped NEAR is permanently stranded in Aurora's contract balance with no recovery path.

---

### Finding Description

The `ExitToNear` precompile supports a two-step wNEAR unwrap flow:

1. The user calls `withdrawToNear(recipient, amount)` (with `:unwrap` suffix) on the wNEAR ERC-20 contract.
2. `EvmErc20.sol` burns the caller's ERC-20 tokens and calls the `ExitToNear` precompile.
3. The precompile schedules a `near_withdraw` call on the wNEAR NEP-141 contract, with `exit_to_near_precompile_callback` as the callback.
4. Inside the callback, if `near_withdraw` succeeded, a bare `PromiseAction::Transfer` is created to send the unwrapped NEAR to the user-supplied `target_account_id`.

The critical gap is at step 4. The NEAR transfer promise is created with **no callback**:

```rust
// engine/src/contract_methods/connector.rs  lines 214-228
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    if let Some(args) = args.transfer_near {
        let action = PromiseAction::Transfer {
            amount: Yocto::new(args.amount),
        };
        let promise = PromiseBatchAction {
            target_account_id: args.target_account_id,
            actions: vec![action],
        };
        let promise_id = handler.promise_create_batch(&promise);
        handler.promise_return(promise_id);   // ← no error callback attached
    }
    None
``` [1](#0-0) 

If the `Transfer` action fails (e.g., `target_account_id` does not exist on NEAR — a bare `Transfer` to a non-existent account panics in the NEAR runtime), the NEAR that was successfully withdrawn from the wNEAR contract is refunded to the Aurora contract's own balance. There is no subsequent promise or state mutation to re-mint the burned ERC-20 tokens or otherwise return value to the user.

The `transfer_near` args originate from the user-controlled `receiver_account_id` field parsed in `exit_erc20_token_to_near`:

```rust
// engine-precompiles/src/native.rs  lines 587-608
Some(Message::UnwrapWnear) if erc20_address == get_wnear_address(io).raw() => {
    (
        nep141_account_id,
        format!(r#"{{"amount":"{}"}}"#, exit_params.amount.as_u128()),
        "near_withdraw",
        Some(TransferNearArgs {
            target_account_id: exit_params.receiver_account_id.clone(),
            amount: exit_params.amount.as_u128(),
        }),
        ...
    )
}
``` [2](#0-1) 

The `ExitToNearPrecompileCallbackArgs` struct carries the `transfer_near` field but has no `refund` field populated for the wNEAR path (the `refund` field is only populated for the `error_refund` feature, and only for the base-token and ERC-20 non-wNEAR paths):

```rust
// engine-precompiles/src/native.rs  lines 449-455
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [3](#0-2) 

Even with `error_refund` enabled, `refund_call_args` for the wNEAR path returns a `RefundCallArgs` that re-mints ERC-20 tokens — but this refund is only triggered when the **first** promise (`near_withdraw`) fails, not when the **second** promise (the bare NEAR `Transfer`) fails. There is no third-level callback.

---

### Impact Explanation

When the final NEAR `Transfer` fails:

- The user's wNEAR ERC-20 tokens are **permanently burned** (the burn happened atomically in the EVM before any promise was scheduled).
- The unwrapped NEAR is **permanently stranded** in Aurora's contract balance; no function exists to reclaim it for the original user.
- This constitutes **permanent freezing of user funds** (Critical).

---

### Likelihood Explanation

The `target_account_id` is fully user-controlled. A NEAR `Transfer` action to a non-existent account fails at the NEAR runtime level. A user who mistypes their NEAR account ID, uses a deleted account, or is directed to do so by a malicious front-end will trigger this path. The scenario is realistic and requires no special privilege.

---

### Recommendation

Attach a second-level callback to the NEAR `Transfer` promise created inside `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-mint the burned wNEAR ERC-20 tokens to the original sender (analogous to what `refund_on_error` does for the base-token and ERC-20 paths). The `transfer_near` args already carry the amount; the ERC-20 address for wNEAR is available via `get_wnear_address`.

---

### Proof of Concept

1. Deploy wNEAR ERC-20 on Aurora and bridge wNEAR into it.
2. Call `withdrawToNear("nonexistent.near:unwrap", amount)` on the wNEAR ERC-20 contract from an EVM account holding wNEAR ERC-20 tokens.
3. Observe: wNEAR ERC-20 tokens are burned; `near_withdraw` on the wNEAR NEP-141 succeeds; the subsequent `Transfer` to `nonexistent.near` fails.
4. Observe: Aurora's NEAR balance increases by `amount` yoctoNEAR; the user's wNEAR ERC-20 balance is zero; no refund is issued. The NEAR is permanently inaccessible to the user.

The test suite itself documents the analogous ERC-20 case (no refund when `ft_transfer` fails without `error_refund`):

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 656-660
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The wNEAR unwrap path has no equivalent feature flag at all — the second-level transfer is always unguarded.
