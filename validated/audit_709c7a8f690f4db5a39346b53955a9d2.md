### Title
L1 Priority-Queue Transaction with Insufficient Deposit Permanently Halts Chain via Unguarded `require_internal!` — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`process_l1_transaction` explicitly acknowledges that L1 priority-queue transactions **cannot be invalidated** because doing so would halt the chain. The helper `prepare_and_check_resources` therefore uses saturating arithmetic for every overflow/underflow case. However, immediately after that safe helper returns, `process_l1_transaction` performs two additional checks using `require_internal!` / `checked_*` that propagate a fatal `internal_error!` on failure — bypassing the saturating-arithmetic safety net. An L1 transaction whose `total_deposited` field is less than `gas_price × gas_limit` (a condition the L1 contracts are supposed to prevent but may fail to enforce after a protocol upgrade) triggers one of these unguarded paths, returns a `BootloaderSubsystemError`, and permanently stalls every subsequent transaction in the priority queue.

---

### Finding Description

`prepare_and_check_resources` is the designated "resilience layer" for L1 transactions. Its own doc-comment states:

> *"This function handles cases that for L2 transactions would be validation errors, as 'invalidating' an L1 transaction can halt the chain (due to the priority queue). … The approach is to use saturating arithmetic and emit a system log if this situation ever happens."*

Every arithmetic operation inside that function uses `.unwrap_or_else(|| { system_log!(…); u64::MAX })` or `.min(gas_limit)` to guarantee forward progress. [1](#0-0) 

Immediately after `prepare_and_check_resources` returns, `process_l1_transaction` computes `tx_internal_cost` and checks the deposit with **hard-failing** arithmetic:

```rust
let tx_internal_cost = gas_price
    .checked_mul(U256::from(gas_limit))
    .ok_or(internal_error!("gp*gl"))?;          // ← fatal on overflow
let total_deposited = transaction.reserved[0].read();
require_internal!(
    total_deposited >= tx_internal_cost,
    "Deposited amount too low",
    system
)?;                                               // ← fatal if deposit < cost
``` [2](#0-1) 

The same pattern recurs inside `execute_l1_transaction_and_notify_result`:

```rust
let max_fee_commitment = gas_price
    .checked_mul(U256::from(transaction.gas_limit.read()))
    .ok_or(internal_error!("gp*gl"))?;           // ← fatal on overflow
let to_transfer = total_deposited
    .checked_sub(max_fee_commitment)
    .ok_or(internal_error!("td-mfc"))?;          // ← fatal if deposit < fee
``` [3](#0-2) 

And in the post-execution refund calculation:

```rust
total_deposited
    .checked_sub(pay_to_operator)
    .ok_or(internal_error!("td-pto"))            // ← fatal
prepaid_fee
    .checked_sub(pay_to_operator)
    .ok_or(internal_error!("pf-pto"))            // ← fatal
``` [4](#0-3) 

All of these `internal_error!` values have a root cause that is **not** `RootCause::Runtime(FatalRuntimeError)`. The error-handling arm in `process_l1_transaction` that converts out-of-native errors into a graceful revert only catches `FatalRuntimeError`; every other error variant falls through to `return Err(e)`, propagating the error up to the block-level loop:

```rust
_ => {
    system.finish_global_frame(Some(&rollback_handle))?;
    return Err(e);   // ← propagates, halts block
}
``` [5](#0-4) 

The block-level transaction loop in `tx_loop.rs` does not have a recovery path for `BootloaderSubsystemError` on L1 transactions; the error surfaces as a block-level failure, making the block unproducible and the chain unable to advance past that priority-queue entry. [6](#0-5) 

The regression-test suite itself documents the invariant that motivates the vulnerability:

> *"L1 transactions cannot be invalidated (doing so would halt the chain due to the priority queue)."* [7](#0-6) 

---

### Impact Explanation

An L1 priority-queue transaction that satisfies `total_deposited < gas_price × gas_limit` (or triggers any of the other unguarded `internal_error!` paths) causes `process_l1_transaction` to return a fatal error. Because L1 transactions must be processed in strict queue order and cannot be skipped or removed, the chain is permanently stalled at that entry. Every subsequent L1 transaction — including legitimate user deposits and protocol upgrades — is blocked indefinitely. This is a complete, irreversible chain halt achievable by a single crafted L1 transaction.

---

### Likelihood Explanation

The code comment in `prepare_and_check_resources` explicitly acknowledges the scenario:

> *"Note that the 'validation errors' are practically unreachable, as gas_limit, gas_price and gas_per_pubdata are either checked or set by the L1 contracts. We decide to handle these cases as a fallback **in case the L1 contracts aren't properly updated to reflect a change in ZKsync OS**."* [8](#0-7) 

This is precisely the scenario described in the external report: the vulnerability can be triggered **naturally** (e.g., after a protocol upgrade that changes gas accounting on L2 without a corresponding L1 contract update) or **intentionally** by an attacker who discovers a discrepancy between L1 validation arithmetic and L2 validation arithmetic and submits a transaction that passes L1 checks but fails the L2 `require_internal!` guard. The cost to the attacker is a single L1 transaction fee.

---

### Recommendation

Apply the same saturating-arithmetic pattern used in `prepare_and_check_resources` to every arithmetic operation in `process_l1_transaction` that currently uses `internal_error!` on failure:

1. Replace `checked_mul(…).ok_or(internal_error!(…))?` with `.saturating_mul(…)` (or log-and-saturate via `unwrap_or_else`).
2. Replace `require_internal!(total_deposited >= tx_internal_cost, …)?` with a log-and-clamp: if the invariant is violated, log a system warning, set `tx_internal_cost = total_deposited` (or zero out the user's resources), and continue processing so the transaction is recorded as failed rather than causing a chain halt.
3. Apply the same treatment to the `checked_sub` operations for `to_transfer`, `to_refund_recipient`, and `prepaid_fee - pay_to_operator`.

This mirrors the existing resilience philosophy and ensures that even a malformed or edge-case L1 transaction is processed (as a failed/reverted transaction) rather than stalling the entire chain.

---

### Proof of Concept

1. Attacker submits an L1→L2 priority-queue transaction on L1 with:
   - `gas_price = P` (e.g., `u128::MAX`)
   - `gas_limit = G` (e.g., `u64::MAX`)
   - `total_deposited = P × G − 1` (one wei below the required deposit)
2. The L1 contract validates `total_deposited >= P × G` using its own arithmetic; due to a discrepancy (e.g., integer truncation, a recent upgrade that changed the formula on L2 but not L1), the L1 check passes.
3. The transaction enters the priority queue and is eventually included in an L2 block.
4. `prepare_and_check_resources` runs and saturates all overflows — it returns successfully.
5. `process_l1_transaction` then executes:
   ```rust
   let tx_internal_cost = gas_price.checked_mul(U256::from(gas_limit))
       .ok_or(internal_error!("gp*gl"))?;   // succeeds: P*G fits in U256
   require_internal!(
       total_deposited >= tx_internal_cost,  // FAILS: P*G-1 < P*G
       "Deposited amount too low", system
   )?;                                        // returns Err(internal_error!)
   ```
6. The `internal_error!` propagates through the block loop as a `BootloaderSubsystemError`.
7. The block cannot be finalized. The sequencer retries the same block, hits the same error, and the chain is permanently halted at this priority-queue entry. [9](#0-8)
