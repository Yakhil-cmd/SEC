Audit Report

## Title
Unbounded Per-Peer Slot Table Growth via `SLOT_TABLE_NO_LIMIT` Enables Byzantine Node to OOM Honest Replicas — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

The per-peer slot table guard in `handle_slot_update_receive` is gated on `peer_slot_table_len < self.slot_limit`. For consensus, certifier, DKG, and iDKG artifact channels, `slot_limit` is set to `SLOT_TABLE_NO_LIMIT` (effectively `usize::MAX`), making the guard a no-op. A single Byzantine node with valid subnet membership can send an unbounded stream of slot updates with monotonically increasing `SlotNumber` values, causing unbounded growth of `slot_table`, `active_assembles`, and `artifact_processor_tasks` until the honest replica OOMs and halts.

## Finding Description

The guard at `receiver.rs` line 373 is:

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => {
    empty_slot.insert(new_slot_entry);
    ...
}
Entry::Vacant(_) => {
    warn!(..., "Peer {} tries to exceed slot limit {}. Dropping slot update", ...);
    (false, None)
}
``` [1](#0-0) 

When `slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice — a `usize` counter cannot reach `usize::MAX` before the process exhausts memory. The guard is therefore a no-op.

Each new `SlotNumber` with a unique artifact ID causes three allocations:
1. A `SlotEntry` inserted into `slot_table: HashMap<NodeId, HashMap<SlotNumber, SlotEntry<...>>>` [2](#0-1) 
2. A `watch::Sender<PeerCounter>` inserted into `active_assembles` [3](#0-2) 
3. A live tokio task spawned into `artifact_processor_tasks` [4](#0-3) 

All three structures grow without bound. Topology pruning in `handle_topology_update` only removes nodes that **leave** the subnet topology — a Byzantine node that remains a subnet member is never pruned. [5](#0-4) 

In production, `SLOT_TABLE_NO_LIMIT` is passed for consensus (both code paths), certifier, DKG, and iDKG channels: [6](#0-5) 

Only the ingress channel uses a finite `SLOT_TABLE_LIMIT_INGRESS`. [7](#0-6) 

The inbound channel has capacity 100, but the event loop drains it continuously in `start_event_loop`, so backpressure does not prevent the attack. [8](#0-7) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

A single Byzantine node can crash one or more honest replicas via OOM, halting their participation in consensus. If multiple honest replicas are targeted simultaneously, subnet-wide consensus finalization stalls. The attack is not volumetric DDoS — it exploits a protocol-level unbounded allocation path.

## Likelihood Explanation

- **Precondition**: attacker controls one node with valid subnet membership (authenticated TLS). This is within the IC's stated BFT threat model (f < n/3 Byzantine nodes); no majority, governance action, key compromise, or privileged access is required.
- **Automation**: trivially automatable — a tight loop sending slot updates with incrementing `SlotNumber` values and unique artifact IDs.
- **No victim interaction**: the attack is entirely attacker-driven over the existing P2P protocol channel.
- **Repeatability**: the attack can be re-launched after any replica restart.

## Recommendation

Replace `SLOT_TABLE_NO_LIMIT` for consensus, certifier, DKG, and iDKG channels with a finite, protocol-justified per-peer bound derived from the maximum number of in-flight artifacts a legitimate node can advertise per protocol round. The existing guard logic at line 373 is correct; only the limit value needs to change. The ingress channel already demonstrates this pattern with `SLOT_TABLE_LIMIT_INGRESS`. [7](#0-6) 

## Proof of Concept

The existing test `slot_table_limit_exceeded` at line 846 confirms the guard works correctly with a finite limit of 2. [9](#0-8) 

A minimal extension to prove the `usize::MAX` case:

```rust
#[tokio::test]
async fn slot_table_no_limit_unbounded_growth() {
    let (mut mgr, _channels) = ReceiverManagerBuilder::new()
        .with_slot_limit(usize::MAX)  // SLOT_TABLE_NO_LIMIT
        .build();
    let cancellation = CancellationToken::new();

    for i in 0..10_000u64 {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(i),
                commit_id: CommitId::from(i),
                update: Update::Id(i),  // unique artifact ID per slot
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    // All three structures grow to 10_000 — no limit is enforced
    assert_eq!(mgr.slot_table.get(&NODE_1).unwrap().len(), 10_000);
    assert_eq!(mgr.active_assembles.len(), 10_000);
    assert_eq!(mgr.artifact_processor_tasks.len(), 10_000);
}
```

At realistic scale (millions of iterations), the process OOMs before the loop completes. The `ReceiverManagerBuilder` test harness already supports `with_slot_limit(...)`, so this test requires no new infrastructure. [10](#0-9)
