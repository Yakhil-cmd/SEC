Audit Report

## Title
Replica Accepts Unvalidated `OutputRequest` Entries from Compromised Sandbox via IPC — (`rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs`)

## Summary

The replica controller accepts `SystemStateModifications` from the sandbox process over a Unix domain socket IPC channel and applies the `requests` field directly to the canister's output queue without verifying that those requests were actually produced by legitimate Wasm execution. A compromised sandbox process (post-sandbox-escape) can craft a forged `ExecutionFinishedRequest` containing arbitrary `OutputRequest` entries, causing the replica to enqueue unauthorized inter-canister calls on behalf of the canister. The only guard (`validate_cycle_change`) checks internal cycle accounting consistency and is trivially satisfied by setting `payment = 0` and `cycles_balance_change = zero()`. The sender identity check in `push_output_request` is a regular `assert_eq!` that a compromised sandbox bypasses by setting `sender` to the known canister ID.

## Finding Description

**Step 1 — Sandbox sends forged `ExecutionFinishedRequest` over IPC.**
The sandbox calls `controller.execution_finished(ExecutionFinishedRequest { exec_id, exec_output })` over the Unix domain socket. The `exec_output.state.system_state_modifications.requests` field is fully attacker-controlled. The `ExecutionFinishedRequest` struct is `#[derive(Deserialize, Serialize)]` and carries `SandboxExecOutput` which carries `StateModifications` which carries `SystemStateModifications`. [1](#0-0) [2](#0-1) 

**Step 2 — `ControllerServiceImpl::execution_finished` passes it through without validation.**
The only check is whether `exec_id` exists in the registry. If it does, the raw `exec_output` is passed directly to the completion closure with no inspection of `system_state_modifications.requests`. [3](#0-2) 

**Step 3 — `take_system_state_modifications` filtering is in-sandbox and bypassable.**
The filtering logic in `SystemApiImpl::take_system_state_modifications` that strips `requests` for non-update API types runs inside the sandbox process. A compromised sandbox bypasses this entirely and sends a hand-crafted `SystemStateModifications` directly over IPC. [4](#0-3) 

**Step 4 — `validate_cycle_change` is the only guard and is trivially bypassed.**
`apply_changes` calls `validate_cycle_change` which only verifies that `cycles_balance_change == sum(req.payment) + consumed + reserved`. A compromised sandbox sets `payment = 0` on all forged requests and `cycles_balance_change = CyclesBalanceChange::zero()`, making the check pass trivially. [5](#0-4) 

**Step 5 — `apply_changes` enqueues the forged request.**
For each request in `self.requests`, `apply_changes` calls `push_message` → `system_state.push_output_request(msg, time)` with no additional validation of request origin. [6](#0-5) 

**Step 6 — The sender check in `push_output_request` is bypassable.**
`SystemState::push_output_request` asserts `request.sender == self.canister_id`. A compromised sandbox knows the canister ID (it was passed in `SandboxExecInput.sandbox_safe_system_state`) and sets `sender` accordingly, so this assertion passes. [7](#0-6) 

**Step 7 — `OutputRequest` is fully serializable with all fields attacker-controlled.**
`OutputRequest` is `#[derive(Deserialize, Serialize)]` with public fields including `receiver`, `method_name`, `method_payload`, `payment`, `call_context_id`, and `on_reply`/`on_reject` Wasm closures. [8](#0-7) 

**Root cause:** The replica's IPC handler treats `SystemStateModifications.requests` as trusted output from the sandbox, but the sandbox is an untrusted process by design. The SELinux policy explicitly lists "control replica" as a critically disallowed sandbox operation, yet the IPC protocol provides no mechanism for the replica to independently verify that the `requests` field reflects actual `ic0.call_perform` invocations made during Wasm execution. [9](#0-8) 

## Impact Explanation

Once the forged `OutputRequest` is enqueued in the canister's output queue, message routing delivers it to the target canister (e.g., the ICP ledger). The ledger sees a legitimate inter-canister call from the attacker's canister and executes it. If the canister holds ICP tokens, the attacker can call `transfer` to move them to an arbitrary account. The `on_reply`/`on_reject` Wasm closures in the forged request point to attacker-controlled Wasm table indices, enabling further exploitation upon response. This matches the Critical impact class: **Theft, permanent loss, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets**.

## Likelihood Explanation

The precondition is compromise of the sandbox process via a memory-safety vulnerability in wasmtime or the sandbox binary. This is non-trivial but explicitly within the ICP bug bounty scope (sandbox escape scenarios). The SELinux documentation itself acknowledges sandbox escape as a known risk vector and discusses its consequences. Once sandbox escape is achieved, the IPC injection is deterministic and straightforward — the attacker needs only to craft a valid `ExecutionFinishedRequest` with a known `exec_id` (observable from the IPC channel) and a forged `requests` field with `payment = 0`. No brute-forcing or probabilistic exploitation is required.

## Recommendation

1. **Cross-check request count on the replica side**: The `WasmExecutionOutput.system_api_call_counters` already tracks `call_perform_count`. The replica should verify that `system_state_modifications.requests.len() == system_api_call_counters.call_perform_count` before applying changes.
2. **Validate `call_context_id` on the replica side**: Before enqueuing, verify that each `request.call_context_id` exists in the canister's `CallContextManager` and was not already consumed.
3. **Harden the sender check**: Convert the `assert_eq!` in `push_output_request` to a hard error (`return Err(...)`) rather than a panic, and add an equivalent check in `apply_changes` before calling `push_message`.
4. **Consider moving request construction to the replica side**: The replica could reconstruct `OutputRequest` entries from the system API call log rather than accepting them wholesale from the sandbox.

    fn validate_cycle_change(&self, is_cmc_canister: bool) -> HypervisorResult<()> {
        let mut expected_change = CyclesBalanceChange::zero();

        if let Some((_, call_context_balance_taken)) = self.call_context_balance_taken {
            expected_change =
                expected_change + CyclesBalanceChange::added(call_context_balance_taken);
        }

        for req in self.requests.iter() {
            expected_change = expected_change + CyclesBalanceChange::removed(req.payment);
        }

        for amount in self.consumed_cycles_by_use_case.iter_over_real() {
            expected_change = expected_change + CyclesBalanceChange::removed(amount);
        }

        expected_change = expected_change + CyclesBalanceChange::removed(self.reserved_cycles);

        // If the canister is not the cycles minting canister, then the balance
        // change coming from the Wasm execution must match the expected balance
        // change that we just computed.
        if is_cmc_canister || self.cycles_balance_change == expected_change {
            Ok(())
        } else {
            Err(HypervisorError::WasmEngineError(
                WasmEngineError::FailedToApplySystemChanges(format!(
                    "Invalid cycle change: expected {:?}, got {:?}",
                    expected_change, self.cycles_balance_change
                )),
            ))
        }
    }
