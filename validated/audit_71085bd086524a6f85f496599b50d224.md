Audit Report

## Title
Unauthenticated `reset_timers` Allows Any Caller to Disrupt SNS Periodic Task Scheduling - (Files: `rs/sns/governance/canister/canister.rs`, `rs/sns/root/canister/canister.rs`, `rs/sns/swap/canister/canister.rs`)

## Summary

The `reset_timers` update method on SNS Governance, SNS Root, and SNS Swap canisters performs no caller authentication. Any unprivileged ingress sender can invoke it. In all three canisters, the cool-down guard is wrapped in a doubly-nested `if let Some(...)` chain: when `timers` or `last_reset_timestamp_seconds` is `None`, the guard is silently skipped and `init_timers()` executes unconditionally. Even when the `Option` fields are populated, the absence of any `caller()` check means any principal can trigger a timer reset once per cool-down window, disrupting SNS periodic task scheduling.

## Finding Description

**No caller authentication (primary issue):**

In `rs/sns/governance/canister/canister.rs` at L644–661, `rs/sns/root/canister/canister.rs` at L487–506, and `rs/sns/swap/canister/canister.rs` at L348–365, the `reset_timers` function is decorated `#[update]` with no `caller()` check, no `inspect_message` guard, and no principal allowlist. The function is listed in the public Candid interface of all three canisters and is callable by any principal including anonymous.

**`None`-bypass of cool-down guard (secondary issue):**

In `rs/sns/governance/canister/canister.rs` L648–656:

```rust
if let Some(timers) = governance_mut().proto.timers
    && let Some(last_reset_timestamp_seconds) = timers.last_reset_timestamp_seconds
{
    assert!(
        now_seconds().saturating_sub(last_reset_timestamp_seconds)
            >= reset_timers_cool_down_interval_seconds, ...
    );
}
init_timers();  // always reached when either guard fails
```

The same pattern appears in `rs/sns/root/canister/canister.rs` L491–500. The `Timers` struct stores `last_reset_timestamp_seconds` as `Option<u64>` (`rs/nervous_system/proto/src/gen/ic_nervous_system.pb.v1.rs` L199). If `proto.timers` is `None` or `last_reset_timestamp_seconds` is `None`, both guards fail, the `assert!` is never reached, and `init_timers()` runs unconditionally. This applies to canisters upgraded from a version that predates the `timers` field.

The SNS Swap variant at L352–360 uses a three-way `&&` that panics only when all three sub-conditions are true; if either `timers` or `last_reset_timestamp_seconds` is `None`, the panic is skipped and `init_timers()` runs.

**Effect of `init_timers()`:**

`init_timers()` clears the existing `TimerId`, registers a fresh `set_timer_interval`, and resets `last_reset_timestamp_seconds` to `now()`. For SNS Root (`rs/sns/root/canister/canister.rs` L466–485), the periodic task interval is 1 day (`RUN_PERIODIC_TASKS_INTERVAL = Duration::from_secs(60 * 60 * 24)`) and the cool-down is 1 week (`RESET_TIMERS_COOL_DOWN_INTERVAL = Duration::from_secs(60 * 60 * 24 * 7)`). Each reset shifts the next root periodic task execution forward by up to 24 hours. For SNS Governance (`rs/sns/governance/canister/canister.rs` L76), the periodic task interval is 10 seconds and the cool-down is 600 seconds, so the per-reset delay is bounded by 10 seconds — negligible in practice.

## Impact Explanation

The concrete, exploitable impact is on SNS Root: any unprivileged caller can delay SNS Root periodic tasks (canister-status polling, dapp-controller management) by up to 24 hours, repeatable once per week. This constitutes a significant SNS security impact with concrete protocol harm. The governance/swap impact is present but overstated in the claim — the actual delay per reset is bounded by the 10-second periodic task interval, not the 600-second cool-down. This matches: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm.**

## Likelihood Explanation

The method is listed in the public Candid interface of all three canisters, callable by any ingress sender including anonymous principal. No staking, neuron ownership, or privileged key is required. The attacker needs only the canister ID and an empty `ResetTimersRequest`. The attack is trivially discoverable from on-chain Candid metadata and requires no special technical skill. For SNS Root, a single call per week is sufficient to sustain the maximum delay.

## Recommendation

1. **Restrict the caller**: Add a `caller()` check to `reset_timers` in all three canisters, restricting invocation to a known set of principals (e.g., the SNS root canister, NNS governance, or the canister itself). At minimum, reject the anonymous principal.

2. **Harden the `None` guard**: Replace the `if let Some(...)` bypass with an explicit rejection when `timers` or `last_reset_timestamp_seconds` is `None`:

```rust
let timers = governance_mut().proto.timers
    .expect("timers not initialized");
let last_reset = timers.last_reset_timestamp_seconds
    .expect("last_reset_timestamp_seconds not set");
assert!(
    now_seconds().saturating_sub(last_reset) >= reset_timers_cool_down_interval_seconds,
    "..."
);
```

## Proof of Concept

```bash
# Any principal (including anonymous) can call reset_timers on SNS Root:
dfx canister --network ic call <sns_root_canister_id> reset_timers '(record {})'
# Returns: (record {})  -- succeeds with no authorization check

# Call once per week to delay SNS Root periodic tasks by up to 24 hours.
# Repeat indefinitely to sustain the maximum delay.
```

For the `None`-bypass: deploy an SNS canister from a WASM that omits the `timers` field in its state proto, then upgrade to the current WASM. On the first `reset_timers` call after upgrade (before `post_upgrade` sets `timers`), the cool-down check is skipped entirely and `init_timers()` executes unconditionally regardless of how recently the previous reset occurred.
