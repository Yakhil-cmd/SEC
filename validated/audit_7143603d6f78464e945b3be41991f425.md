### Title
Malicious NEP-141 Contract Can Mint Unbacked ERC-20 Tokens via `ft_on_transfer` Without Transferring Tokens — (File: `engine/src/contract_methods/connector.rs`, `engine/src/engine.rs`)

---

### Summary

The `ft_on_transfer` entry point in Aurora Engine unconditionally trusts `args.amount` supplied by the calling NEP-141 contract and mints the corresponding ERC-20 tokens on Aurora without verifying that the actual NEP-141 tokens were transferred to Aurora's account. Any NEAR account that has a registered ERC-20 on Aurora (i.e., is "whitelisted" by having called `deploy_erc20_token`) can call `ft_on_transfer` directly with an arbitrary `args.amount`, causing Aurora to mint unbacked ERC-20 tokens. This is the direct analog of the LSSVMPairERC20 finding: a trusted/registered counterparty can skip the actual token transfer while the protocol still credits the full amount.

---

### Finding Description

The `ft_on_transfer` public entry point in `engine/src/contract_methods/connector.rs` dispatches to one of two minting paths based on the caller's identity:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)          // mints ETH
} else {
    engine.receive_erc20_tokens(               // mints ERC-20
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

In the ERC-20 path, `receive_erc20_tokens` (engine/src/engine.rs:796–844) performs only one guard — checking that a registered ERC-20 exists for the caller — and then immediately mints `args.amount` tokens:

```rust
let erc20_token = get_erc20_from_nep141(&self.io, token)?;   // only check
let erc20_admin_address = current_address(current_account_id);
let result = self.call(
    &erc20_admin_address,
    &erc20_token,
    Wei::zero(),
    setup_receive_erc20_tokens_input(&recipient, amount),     // mints amount
    u64::MAX,
    Vec::new(),
    Vec::new(),
    handler,
).and_then(submit_result_or_err)?;
```

There is no before-after check on Aurora's NEP-141 balance to confirm that the tokens were actually received. The NEP-141 standard's `ft_transfer_call` flow is supposed to transfer tokens before calling `ft_on_transfer`, but the NEAR runtime does not enforce this ordering — any NEAR account can call `ft_on_transfer` on Aurora directly, setting itself as `predecessor_account_id`, without going through `ft_transfer_call` at all.

`deploy_erc20_token` has no access control, so any attacker can register their own NEP-141 contract and obtain a corresponding ERC-20 on Aurora:

### Impact Explanation

An attacker who controls a NEP-141 contract with a registered ERC-20 on Aurora can:

1. Call `ft_on_transfer` on Aurora directly from their NEP-141 contract with an arbitrarily large `args.amount` — no actual NEP-141 tokens are transferred.
2. Aurora mints the full `args.amount` of ERC-20 tokens to the attacker's address.
3. The attacker swaps the unbacked ERC-20 tokens for real tokens (ETH, other bridged assets) in Aurora DeFi liquidity pools.
4. Liquidity providers lose their deposited funds.

This constitutes direct theft of user funds held in Aurora DeFi protocols. The ERC-20 token supply on Aurora becomes unbacked, breaking the 1:1 peg invariant between NEP-141 tokens held by Aurora and ERC-20 tokens in circulation.

---

### Likelihood Explanation

The attack is reachable by any unprivileged NEAR account:

- `deploy_erc20_token` is permissionless — any account can register a NEP-141 and obtain a corresponding ERC-20 on Aurora.
- `ft_on_transfer` is a public entry point with no caller authentication beyond checking for a registered ERC-20.
- The NEAR runtime does not prevent direct calls to `ft_on_transfer` outside of the `ft_transfer_call` flow.

Profitability requires the attacker's ERC-20 to have liquidity on Aurora. For a newly created malicious token this requires attracting liquidity providers. For an attacker who already controls a legitimate NEP-141 token with existing Aurora liquidity, the attack is immediately profitable. Likelihood is **Medium**.

---

### Recommendation

Add a before-after balance check on Aurora's NEP-141 balance inside `ft_on_transfer`. Before minting, record Aurora's current NEP-141 balance via a view call to the NEP-141 contract; after the call completes (via callback), verify that the balance increased by at least `args.amount`. If the balance did not increase by the expected amount, revert the mint and return the full `args.amount` to the sender.

Alternatively, track the total ERC-20 supply minted for each NEP-141 and periodically reconcile it against Aurora's actual NEP-141 balance, reverting any discrepancy.

### Proof of Concept

**Entry path:**

1. Attacker deploys contract `attacker.near` implementing a minimal NEAR contract with a method that calls `ft_on_transfer` on Aurora.
2. Attacker calls `deploy_erc20_token` on Aurora with `attacker.near` as the NEP-141 account — this succeeds with no access control.
3. Attacker calls `ft_on_transfer` on Aurora from `attacker.near` with:
   ```json
   { "sender_id": "attacker.near", "amount": "1000000000000000000000", "msg": "<victim_evm_address_hex>" }
   ```
4. Aurora executes `receive_erc20_tokens` → `get_erc20_from_nep141` succeeds (ERC-20 is registered) → `setup_receive_erc20_tokens_input` mints `1000000000000000000000` ERC-20 tokens to the attacker's EVM address. No NEP-141 tokens were transferred.
5. Attacker swaps the unbacked ERC-20 tokens for ETH or other bridged assets in an Aurora DEX, draining liquidity providers' funds.

- `engine/src/contract_methods/connector.rs` lines 111–158 (`deploy_erc20_token`, no access control)
