### Title
Inconsistent Rounding Direction in `native_per_gas` vs `native_per_pubdata` Unit Conversion Causes Systematic Pubdata Underpayment - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

ZKsync OS computes two critical unit-conversion ratios from the same pair of operator-supplied prices (`gas_price` / `native_price` and `pubdata_price` / `native_price`). The two computations use **opposite rounding directions** with no documented justification: `native_per_gas` is rounded **up** (ceiling), while `native_per_pubdata` is rounded **down** (floor/truncating). This asymmetry causes pubdata to be systematically underpriced relative to gas, and the downstream `delta_gas` refund calculation compounds the error, resulting in the operator being underpaid for every transaction that generates pubdata when `pubdata_price % native_price ≠ 0`.

---

### Finding Description

**Location 1 — `native_per_gas` uses ceiling division:** [1](#0-0) 

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

**Location 2 — `native_per_pubdata` uses floor (truncating) division:** [2](#0-1) 

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

The same inconsistency is replicated in the API helper: [3](#0-2) 

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

**Location 3 — `delta_gas` refund uses floor division against a ceiling-rounded divisor:** [4](#0-3) 

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

Because `native_per_gas = ceil(gas_price / native_price)`, the expression `native_used / native_per_gas` (floor division) systematically underestimates the gas equivalent of native consumed. This means `delta_gas` is smaller than it should be, so the user's `gas_used` is understated and the refund is overstated.

**The two errors compound:**
1. `native_per_pubdata` is truncated → user is charged fewer native units per pubdata byte than the operator intended.
2. `delta_gas` is underestimated → the gas-equivalent of native consumed is understated → the user receives a larger-than-correct refund.

Both effects benefit the transaction sender at the expense of the operator/protocol.

---

### Impact Explanation

- **Resource accounting bug / public funds-loss path.** The operator sets `pubdata_price` and `native_price` to reflect the real cost of publishing data to L1. The truncation in `native_per_pubdata` means every transaction that generates pubdata when `pubdata_price % native_price ≠ 0` pays fewer native units than the operator intended. The shortfall per pubdata byte is at most `native_price − 1` native units. For a transaction generating `N` pubdata bytes, the total underpayment is up to `(native_price − 1) × N` native units. At scale (many transactions, large pubdata), this represents a systematic drain on operator revenue.
- The `delta_gas` underestimation further reduces the gas charged to the user, meaning the user's token refund is slightly larger than correct, compounding the loss.

---

### Likelihood Explanation

- **High.** The condition `pubdata_price % native_price ≠ 0` is the common case for any real-world price pair. The operator has no mechanism to enforce divisibility. Every L2 transaction that writes storage (generating pubdata) is affected. The entry path is any unprivileged EVM transaction that modifies state.

---

### Recommendation

1. Use a **consistent rounding direction** for all unit conversions in the resource accounting model. Since `native_per_gas` uses `div_ceil` (conservative, operator-protective), `native_per_pubdata` should also use `div_ceil`:
   ```rust
   let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
       .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
   ```
2. Apply the same fix in `api/src/helpers.rs` to keep the off-chain validation helper consistent with the on-chain bootloader.
3. Document the chosen rounding direction and the rationale (e.g., "always round up to protect the operator") at each conversion site, as recommended by the original report.
4. Review the `delta_gas` calculation in `refund_calculation.rs` to ensure the floor division of `native_used / native_per_gas` is intentional and does not over-refund users when `native_per_gas` was ceiling-rounded.

---

### Proof of Concept

**Concrete numeric example:**

Let `pubdata_price = 101`, `native_price = 10`, `gas_price = 1000`.

- `native_per_gas = ceil(1000 / 10) = 100` ✓ (exact, no rounding error here)
- `native_per_pubdata = floor(101 / 10) = 10` ← truncated; exact value is 10.1

For a transaction generating 1,000 pubdata bytes:
- Intended native charge for pubdata: `10.1 × 1000 = 10,100`
- Actual native charge: `10 × 1000 = 10,000`
- **Underpayment: 100 native units per transaction**

For `pubdata_price = 999`, `native_price = 10`:
- `native_per_pubdata = floor(999 / 10) = 99` (exact: 99.9)
- For 1,000 pubdata bytes: underpayment = `0.9 × 1000 = 900` native units per transaction

At high throughput (e.g., 10,000 TPS as targeted by ZKsync OS), the cumulative underpayment to the operator is significant. [2](#0-1) [5](#0-4) [6](#0-5)
