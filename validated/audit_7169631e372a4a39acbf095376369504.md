Audit Report

## Title
Provider Rate Limit Exhaustion via No-Op `do_update_node_operator_config_directly` Blocks `add_node`/`remove_node` — (`rs/registry/canister/src/rate_limits.rs`, `rs/registry/canister/src/mutations/do_update_node_operator_config_directly.rs`)

## Summary

`do_update_node_operator_config_directly_` charges only the `NODE_PROVIDER_RATE_LIMITER`, while `do_add_node_` and `do_remove_node_directly_` charge **both** `NODE_OPERATOR_RATE_LIMITER` and `NODE_PROVIDER_RATE_LIMITER`. Because there is no guard rejecting a submission where the new `node_provider_id` equals the current value, a node provider can issue 140 state-preserving no-op calls to exhaust the shared provider pool, permanently blocking all `add_node` and `remove_node` operations for every operator under that provider for up to ~7 days.

## Finding Description

**Asymmetric rate-limit charging:**

`do_update_node_operator_config_directly_` (L67–70) calls only `try_reserve_capacity_for_node_provider_operation`:

```rust
let current_node_provider = caller;
let reservation =
    self.try_reserve_capacity_for_node_provider_operation(now, current_node_provider, 1)?;
```

`do_add_node_` (L60–61) and `do_remove_node_directly_` (L53–54) both call `try_reserve_capacity_for_node_operator_operation`, which internally reserves from **both** the operator and provider limiters (rate_limits.rs L114–127):

```rust
let operator_reservation = with_node_operator_rate_limiter(|rate_limiter| {
    rate_limiter.try_reserve(now, node_operator_key(node_operator_id), requested_capacity)
})?;
let provider_reservation = with_node_provider_rate_limiter(|rate_limiter| {
    rate_limiter.try_reserve(now, node_provider_key(node_provider_id), requested_capacity)
})?;
```

**No-op exhaustion vector:**

The only guard in `do_update_node_operator_config_directly_` against a trivial call is `node_provider_id != node_operator_id` (L77–81). There is no check that the submitted `node_provider_id` differs from the *current* `node_provider_principal_id` stored in the record. Submitting the same provider ID that is already set writes the same bytes back (L83, L86–93) — a state-preserving no-op — while still consuming one unit of provider capacity. This is directly confirmed by the existing test `test_update_node_operator_config_directly_affects_rate_limits` (L173–210), which calls with the current provider ID and asserts `available - 1 == next_available`.

**Spike cap:**

`NODE_PROVIDER_MAX_SPIKE = NODE_PROVIDER_MAX_AVG_OPERATIONS_PER_DAY * 7 = 20 * 7 = 140` (rate_limits.rs L18–19). The provider pool starts full at 140 and refills at 1 unit per `NODE_PROVIDER_CAPACITY_ADD_INTERVAL_SECONDS = ONE_DAY_SECONDS / 20 ≈ 4320 s (~72 min)`, so full recovery from 0 takes ~7 days.

**Attack flow:**
1. Attacker is the `node_provider_principal_id` for at least one operator record.
2. Attacker calls `do_update_node_operator_config_directly` 140 times with `node_provider_id = current_node_provider_principal_id` (≠ `node_operator_id`). Each call is a no-op mutation but consumes 1 unit from `NODE_PROVIDER_RATE_LIMITER`.
3. Provider capacity reaches 0.
4. Any subsequent `do_add_node_` or `do_remove_node_directly_` call by **any** operator under that provider fails at the provider reservation step inside `try_reserve_capacity_for_node_operator_operation`, even though the operator-level limiter still has full capacity.

## Impact Explanation

All `add_node` and `remove_node` operations for every node operator under the targeted provider are blocked for up to ~7 days. The operator-level rate limits are untouched, making the block invisible from the operator's perspective. This is a concrete, sustained application/platform-level DoS on NNS registry node-management functionality — matching the **High** bounty impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

The attacker must be a registered node provider (i.e., hold the `node_provider_principal_id` for at least one operator record). No governance majority, admin key, or subnet compromise is required. The attack requires only 140 ingress messages to the registry canister, is cheap, and is immediately effective. The effect is repeatable: once the limiter partially refills, the attacker can drain it again.

## Recommendation

Replace the call in `do_update_node_operator_config_directly_` from `try_reserve_capacity_for_node_provider_operation` to `try_reserve_capacity_for_node_operator_operation`, consistent with `do_add_node_` and `do_remove_node_directly_`. Alternatively, add an early-return guard that rejects calls where the submitted `node_provider_id` equals the current `node_provider_principal_id` in the record, eliminating the no-op exhaustion vector entirely.

## Proof of Concept

Mirrors the existing test patterns in `test_do_add_node_fails_when_rate_limits_exceeded` and `test_update_node_operator_config_directly_fails_when_rate_limits_exceeded`:

```rust
// 1. Register node_provider P and node_operator O (O.node_provider = P, O != P)
// 2. For i in 0..140:
//        call do_update_node_operator_config_directly_(
//            UpdateNodeOperatorConfigDirectlyPayload {
//                node_operator_id: Some(O),
//                node_provider_id: Some(P),  // same as current — no-op mutation
//            },
//            caller = P, now
//        ).unwrap();  // succeeds, consumes 1 provider unit each time
// 3. assert_eq!(registry.get_available_node_provider_op_capacity(P, now), 0);
// 4. assert_eq!(registry.get_available_node_operator_op_capacity(O, now), 140); // untouched
// 5. let err = registry.do_add_node_(payload, O, now).unwrap_err();
// 6. assert_eq!(err, "Rate Limit Capacity exceeded. Please wait and try again later.");
```
