Audit Report

## Title
Unbounded Tokio Task Accumulation via `SLOT_TABLE_NO_LIMIT` and Indefinitely Stalled `MaybeWantsLater` Loop — (`rs/p2p/consensus_manager/src/receiver.rs`, `rs/replica/setup_ic_network/src/lib.rs`)

## Summary

A Byzantine subnet peer below the consensus fault threshold can cause an honest replica to spawn an unbounded number of stalled tokio tasks by sending slot updates with distinct artifact IDs at heights that cause the consensus bouncer to return `MaybeWantsLater`. Because consensus, certifier, DKG, IDKG, and HTTPS-outcalls artifact types are configured with `slot_limit = usize::MAX`, the per-peer slot table guard is effectively disabled, and each distinct `(slot_number, artifact_id)` pair spawns a new `process_slot_update` task that blocks indefinitely. This leads to heap exhaustion or tokio thread-pool starvation, degrading subnet availability.

## Finding Description

**Root cause 1 — No effective per-peer slot limit for consensus artifacts.**

`rs/replica/setup_ic_network/src/lib.rs` defines:
```rust
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
```
All artifact types except ingress pass `SLOT_TABLE_NO_LIMIT` to `abortable_broadcast_channel`: consensus (both paths), certifier, DKG, IDKG, and HTTPS-outcalls. Only ingress uses the finite `SLOT_TABLE_LIMIT_INGRESS = 50_000`.

**Root cause 2 — Guard condition is unreachable with `slot_limit = usize::MAX`.**

In `handle_slot_update_receive` (`rs/p2p/consensus_manager/src/receiver.rs`):
```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { ... }
Entry::Vacant(_) => { /* drop */ }
```
`peer_slot_table_len` is a `usize` that can never reach `usize::MAX` in practice, so the drop branch is unreachable. Every new `(peer_id, slot_number)` pair is accepted unconditionally.

**Root cause 3 — Each new distinct artifact ID spawns a new task.**

After the slot table insert, if `active_assembles.get(&id)` returns `None` (i.e., the artifact ID has not been seen before), a new `process_slot_update` task is spawned unconditionally via `artifact_processor_tasks.spawn_on(...)`.

**Root cause 4 — Tasks stall indefinitely for far-future artifact heights.**

`FetchArtifact::assemble_message` calls `should_download`, which loops:
```rust
while let BouncerValue::MaybeWantsLater = bouncer_value {
    let _ = bouncer_watcher.changed().await;
    bouncer_value = bouncer_watcher.borrow_and_update()(id);
}
```
The consensus bouncer (`rs/consensus/src/consensus/priority.rs`) returns `MaybeWantsLater` for any non-CUP artifact at `height > next_cup_height + ACCEPTABLE_NOTARIZATION_CUP_GAP`. For far-future heights, this condition is permanently true; the bouncer refreshes every ~3 seconds but always re-evaluates to `MaybeWantsLater`.

**Root cause 5 — Task termination requires peer disconnect or slot overwrite.**

`process_slot_update` selects on `cancellation_token.cancelled()`, `assemble_artifact` (stuck in `should_download`), and `all_peers_deleted_artifact` (fires only when `PeerCounter` becomes empty). As long as the Byzantine peer holds the slot open and remains connected, the `PeerCounter` stays non-empty and the task lives indefinitely.

**Exploit flow:**
1. Byzantine peer (authenticated subnet member) connects via QUIC.
2. Sends N slot updates: `SlotUpdate { slot_number: i, commit_id: 1, update: Id(far_future_consensus_id(i)) }` for `i in 0..N`.
3. Each update passes the slot table guard (no limit), inserts a new `(peer_id, slot_i)` entry, finds no existing `active_assembles` entry for the new ID, and spawns a new `process_slot_update` task.
4. Each task enters `should_download` and blocks on `bouncer_watcher.changed()` indefinitely.
5. N tasks accumulate, each holding a `watch::Receiver<Bouncer<Id>>`, a `watch::Receiver<PeerCounter>`, and async stack frames.
6. Tasks only terminate if the peer disconnects (topology update removes the peer) or the cancellation token fires.

## Impact Explanation

This is a **High** severity finding matching the impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

A single Byzantine subnet peer can accumulate O(rate × time) stalled tokio tasks on an honest replica. With `f` Byzantine peers, the accumulation is O(f × rate × time). Heap exhaustion or tokio thread-pool starvation degrades or halts the replica's participation in consensus, reducing subnet availability. This is a protocol-level resource exhaustion attack, not raw volumetric DDoS.

## Likelihood Explanation

The attacker must be an authenticated subnet member (a Byzantine node with `f < fault_threshold`). This is the standard Byzantine threat model for IC subnets and requires no compromised cryptographic material. The attacker sends well-formed, protocol-compliant slot updates over its legitimate QUIC connection. The rate is bounded only by network bandwidth; there is no code-level cap for consensus artifacts. The path is deterministic and locally reproducible.

## Recommendation

1. Apply a finite per-peer slot limit for all artifact types. The comment at lines 72–74 of `rs/replica/setup_ic_network/src/lib.rs` explicitly acknowledges the ingress limit exists "to protect against a malicious peer advertising many ingress messages" — the same rationale applies to consensus, DKG, IDKG, certifier, and HTTPS-outcalls.
2. In `should_download` (`rs/p2p/artifact_downloader/src/fetch_artifact/download.rs`), add a configurable timeout or maximum retry count so that tasks stalled on `MaybeWantsLater` beyond a threshold self-terminate rather than waiting indefinitely.
3. Add a global cap on `artifact_processor_tasks.len()` in `ConsensusManagerReceiver` as a defense-in-depth measure, in addition to the per-peer slot cap.

## Proof of Concept

```
1. Instantiate a test replica with a mock subnet topology containing one Byzantine peer.
2. From the Byzantine peer, send N=100_000 slot updates over its QUIC connection:
     for i in 0..100_000:
         SlotUpdate {
             slot_number: i,
             commit_id: 1,
             update: Id(ConsensusMessageId { height: current_height + 10_000, hash: NotarizationShare(unique_hash(i)) })
         }
3. Assert active_assembles.len() == 100_000 and artifact_processor_tasks.len() == 100_000.
4. Wait 60 seconds; assert tasks remain alive (bouncer keeps returning MaybeWantsLater).
5. Measure heap allocation and tokio task count growth.
6. Disconnect the Byzantine peer; assert all tasks terminate (PeerCounter empties via topology update).
```
