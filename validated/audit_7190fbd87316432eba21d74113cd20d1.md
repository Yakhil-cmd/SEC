### Title
Unbounded neuron iteration in SNS `compute_ballots_for_new_proposal` allows instruction-limit DoS on proposal creation — (`rs/sns/governance/src/governance.rs`)

---

### Summary

The SNS Governance canister's `compute_ballots_for_new_proposal` iterates over every neuron in the governance state in a single synchronous message execution with no instruction-limit guard. Any neuron holder (unprivileged ingress sender) can trigger this path via `make_proposal`. If the SNS has been populated with neurons up to its configured `max_number_of_neurons`, the loop can exhaust the IC's per-message instruction limit (~40 billion instructions), causing `make_proposal` to trap and permanently blocking new proposal creation. A secondary unbounded BFS in `cast_vote_and_cascade_follow` (SNS-specific, no instruction-limit pause unlike the NNS counterpart) compounds the risk on the voting path.

---

### Finding Description

**Primary: `compute_ballots_for_new_proposal`**

`make_proposal` calls `compute_ballots_for_new_proposal`, which performs a full linear scan of `self.proto.neurons` — a `BTreeMap<String, Neuron>` held in heap memory — to build the electoral roll: [1](#0-0) 

There is no `is_over_instructions_limit()` check, no pagination, and no deferred/background processing. The entire scan must complete within one message execution.

**Secondary: `cast_vote_and_cascade_follow`**

The SNS `cast_vote_and_cascade_follow` runs a BFS over the follower graph in a plain `while !induction_votes.is_empty()` loop: [2](#0-1) 

The NNS counterpart was explicitly retrofitted with an `is_over_instructions_limit()` guard that pauses and resumes across heartbeat rounds: [3](#0-2) 

The SNS version has no equivalent protection.

**Contrast with NNS**

The NNS governance caps neurons at `MAX_NUMBER_OF_NEURONS = 500_000` and uses stable storage with a snapshot mechanism for ballot creation, plus the instruction-limit-aware voting state machine. The SNS stores all neurons in heap memory and has no analogous safeguards. [4](#0-3) 

The SNS `max_number_of_neurons` is a governance-controlled parameter with a ceiling (`MAX_NUMBER_OF_NEURONS_CEILING`); its exact value was not surfaced in the index, but the parameter is explicitly configurable and can be raised via proposal: [5](#0-4) 

---

### Impact Explanation

If `compute_ballots_for_new_proposal` exceeds the ~40 billion instruction limit, `make_proposal` traps with `CanisterInstructionLimitExceeded`. Because the SNS governance canister is the only path for submitting proposals, this permanently blocks:

- New governance proposals (upgrades, parameter changes, treasury transfers)
- Any SNS canister upgrade (which requires a proposal)

The SNS becomes effectively frozen. The only recovery path would be a direct NNS intervention to upgrade the SNS governance canister — a slow, high-friction process.

The secondary BFS in `cast_vote_and_cascade_follow` can similarly cause `register_vote` to trap, preventing vote cascading from completing and leaving proposals in a permanently unresolved state.

---

### Likelihood Explanation

- Any neuron holder can call `make_proposal` — no privileged role required.
- An attacker who controls or coordinates many neuron accounts can populate the SNS up to `max_number_of_neurons` (each neuron requires only the minimum stake).
- Once the neuron count is high enough, every subsequent `make_proposal` call will trap. The attacker does not need to be the one calling `make_proposal`; the DoS is persistent and affects all users.
- The cost to the attacker is proportional to `max_number_of_neurons × neuron_minimum_stake_e8s`, which for many SNS deployments is economically feasible.
- The NNS having survived at scale does not protect SNS instances, which have independent neuron stores and no equivalent instruction-limit mitigations.

---

### Recommendation

1. **Cap `max_number_of_neurons` conservatively** and benchmark `compute_ballots_for_new_proposal` at that cap to verify it stays well under the instruction limit.
2. **Migrate ballot creation to a chunked/background approach** (similar to the NNS voting-power snapshot mechanism) so that the full neuron scan is not required in a single message execution.
3. **Add an `is_over_instructions_limit()` guard to SNS `cast_vote_and_cascade_follow`** and implement the same pause-and-resume state machine that the NNS uses, so that vote cascading can span multiple heartbeat rounds.
4. **Move SNS neurons to stable storage** (as the NNS did) to reduce per-neuron access cost and enable range queries that are cheaper than full heap iteration.

---

### Proof of Concept

1. Deploy an SNS with `max_number_of_neurons` set to a large value (e.g., 100,000) and `neuron_minimum_stake_e8s` set to the minimum.
2. Create neurons up to the cap by staking the minimum amount from many accounts.
3. Call `make_proposal` from any neuron holder.
4. Observe that the call traps with `CanisterInstructionLimitExceeded` (error code 522) because `compute_ballots_for_new_proposal` iterates over all 100,000 neurons in a single message.
5. Confirm that no further proposals can be submitted, freezing the SNS governance canister.

The exact neuron count at which the limit is hit depends on per-neuron instruction cost; the NNS benchmark infrastructure (`rs/nns/governance/src/neuron_store/benches.rs`) provides a template for measuring this: [6](#0-5)
