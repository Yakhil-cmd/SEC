### Title
Unchecked Assembly `call` Return Value in `withdrawToNear`/`withdrawToEthereum` Enables Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn user tokens **before** making a low-level assembly `call` to the exit precompile addresses. The return value of that `call` is captured in a local variable (`res`) but is **never checked**. If the precompile call fails for any reason, execution continues silently, the tokens are permanently destroyed, and no corresponding NEP-141 or Ethereum withdrawal is ever initiated.

---

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow the same pattern:

```solidity
// EvmErc20.sol – withdrawToNear (lines 53–63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is assigned but never read; no revert if res == 0
    }
}
``` [1](#0-0) 

```solidity
// EvmErc20.sol – withdrawToEthereum (lines 65–76)
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is assigned but never read; no revert if res == 0
    }
}
``` [2](#0-1) 

`EvmErc20V2.sol` contains the identical pattern in its `withdrawToNear` (lines 53–63) and `withdrawToEthereum` (lines 66–77). [3](#0-2) [4](#0-3) 

The analog to the reported vulnerability class is direct: the original report shows that a Solidity low-level `call` to a non-existent address returns `true`, causing silent success. Here, the assembly `call` result is never inspected at all — whether the precompile returns `0` (failure) or `1` (success), the ERC-20 function always returns without reverting. Per EVM specification, a `call` to a non-existent address also returns `1` (success) with empty return data, compounding the risk if the precompile address is ever absent.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can legitimately return failure in several cases — for example, if `get_nep141_from_erc20` cannot find the ERC-20 registration, if the recipient account ID is invalid, or if the precompile is called in an unexpected context. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

The `_burn` call is irreversible. If the subsequent assembly `call` to the exit precompile returns `0` (failure), the ERC-20 tokens are permanently destroyed with no corresponding release of NEP-141 tokens on NEAR or ETH on Ethereum. The user has no recourse: the transaction succeeds from the EVM's perspective, the tokens are gone, and no bridge event is emitted.

---

### Likelihood Explanation

**Medium.** The precompile addresses are hardcoded and exist in normal Aurora operation. However, failure paths inside the precompile are reachable:

- If the ERC-20 contract's address is not registered in the NEP-141 mapping (e.g., a newly deployed or misconfigured token), `get_nep141_from_erc20` returns an error and the precompile call fails.
- If the recipient account ID bytes are malformed, the precompile returns an error.
- Per EVM spec, if the precompile address were ever absent (e.g., a fork or alternative deployment), the `call` returns `1` with empty data — silent success with no withdrawal.

Any token holder can trigger this by calling `withdrawToNear` or `withdrawToEthereum` directly.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is `0`. The burn must only be committed after confirming the precompile call succeeded:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, restructure the functions to call the precompile first and only burn on confirmed success, eliminating the burn-before-call ordering risk entirely.

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora with a NEP-141 token that is **not** registered in the Aurora engine's ERC-20 mapping (or use a recipient account ID that causes the precompile to reject the call).
2. Mint tokens to an attacker-controlled address.
3. Call `withdrawToNear(recipient, amount)`.
4. `_burn` executes and destroys `amount` tokens from the caller.
5. The assembly `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` returns `0` (precompile error).
6. The `res` value is never checked; the function returns without reverting.
7. The caller's ERC-20 balance is zero; no NEP-141 tokens are released on NEAR. Funds are permanently lost. [1](#0-0) [3](#0-2)
