### Title
Hardcoded `L1_TX_NATIVE_PRICE = 10` Is Disproportionately Small Relative to Actual Proving Cost, Allowing L1→L2 Transactions to Consume Native Resources Far Below Fair Cost - (File: `basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded at `10` with an explicit `TODO` comment acknowledging it is not calibrated. This constant is used to derive the native resource limit for all L1→L2 priority transactions. Because the operator-set `native_price` for L2 transactions reflects the actual proving cost and can be orders of magnitude higher, L1→L2 transactions receive a disproportionately large native resource budget per unit of gas paid. An unprivileged attacker can submit L1→L2 transactions that consume maximum native resources (proving cycles) at a fraction of the cost that equivalent L2 transactions would pay, creating a systematic resource accounting imbalance and a subsidized proving-cost DoS vector.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant used as the native price for all L1→L2 priority transactions is:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
```

This constant is consumed in `prepare_and_check_resources` inside `process_l1_transaction.rs`:

```rust
let native_price = L1_TX_NATIVE_PRICE;
let native_per_gas = if is_priority_op {
    ...
    u256_try_to_u64(&gas_price.div_ceil(native_price)).unwrap_or_else(...)
    ...
};
let native_prepaid_from_gas = native_per_gas.checked_mul(gas_limit)...;
```

The resulting `native_limit = native_per_gas * gas_limit` is the budget of native resources (RISC-V proving cycles) the transaction is allowed to consume.

For L2 transactions, the equivalent computation uses the operator-supplied `native_price` (read from block metadata), which is calibrated to the actual proving cost. If the operator sets `native_price = 1000` (a plausible value), the disparity is:

| Transaction type | `native_per_gas` at `gas_price=1000` | `native_limit` at `gas_limit=1,000,000` |
|---|---|---|
| L2 tx | `1000 / 1000 = 1` | `1,000,000` |
| L1→L2 tx | `1000 / 10 = 100` | `100,000,000` |

The L1→L2 transaction receives **100× more native resources** for the same gas price and gas limit. `MAX_NATIVE_COMPUTATIONAL = 2^35 ≈ 3.4 × 10^10` is the per-execution ceiling, so a sufficiently high gas limit allows a single L1→L2 transaction to consume the full block's native resource budget at a fraction of the fair cost.

---

### Impact Explanation

Native resources model the off-chain proving cost (RISC-V cycles). When `L1_TX_NATIVE_PRICE` is far below the actual proving cost:

1. **Resource accounting bug**: L1→L2 transactions are systematically underpriced for their proving cost. The protocol/operator absorbs the proving cost that the submitter should pay.
2. **Subsidized proving-cost DoS**: An attacker can submit L1→L2 transactions with high gas limits that consume up to `MAX_NATIVE_COMPUTATIONAL` native resources per transaction while paying only `1/N` of the fair fee (where `N = operator_native_price / 10`). This makes blocks expensive to prove at low attacker cost.
3. **Block native resource exhaustion**: Because native resources are not refunded when a transaction reverts (the full gas limit is charged), an attacker can repeatedly fill blocks' native resource budgets at minimal L1 cost.

---

### Likelihood Explanation

- The `TODO (EVM-1157)` comment is an explicit developer acknowledgment that the value is a placeholder and not calibrated.
- Any unprivileged user can submit L1→L2 priority transactions; no special role is required.
- The disparity between `L1_TX_NATIVE_PRICE = 10` and a realistic operator `native_price` (e.g., 100–10,000) is large and grows as the operator adjusts pricing to reflect actual proving costs.
- The attack requires only L1 gas to submit transactions, which is a known, bounded cost.

---

### Recommendation

Replace the placeholder `L1_TX_NATIVE_PRICE = 10` with a value derived from the same calibration process used for the operator-set `native_price`. At minimum, `L1_TX_NATIVE_PRICE` should be set to a value that ensures L1→L2 transactions receive no more native resources per unit of gas than L2 transactions at the expected operator `native_price`. Resolve `TODO (EVM-1157)` before production deployment.

---

### Proof of Concept

1. Operator sets `native_price = 1000` (block context).
2. Attacker submits an L1→L2 priority transaction with `gas_price = 10000`, `gas_limit = 30,000,000`.
   - L1→L2 path: `native_per_gas = 10000 / 10 = 1000`; `native_limit = 1000 * 30,000,000 = 3 × 10^10 ≈ MAX_NATIVE_COMPUTATIONAL`.
   - Equivalent L2 tx: `native_per_gas = 10000 / 1000 = 10`; `native_limit = 10 * 30,000,000 = 3 × 10^8`.
3. The L1→L2 transaction consumes up to `MAX_NATIVE_COMPUTATIONAL` proving cycles while paying the same fee as an L2 transaction that would be limited to `3 × 10^8` cycles — a **100× proving-cost subsidy**.
4. By submitting multiple such transactions, the attacker exhausts the block's native resource budget at 1/100th the fair cost, making each block disproportionately expensive to prove.
