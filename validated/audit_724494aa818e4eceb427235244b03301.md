Audit Report

## Title
Unauthenticated Caller Can Abort Any In-Progress SNS Upgrade via `fail_stuck_upgrade_in_progress` - (File: rs/sns/governance/canister/canister.rs)

## Summary
`fail_stuck_upgrade_in_progress` is registered as a public `#[update]` canister method with no caller authorization check at either the canister entry point or the underlying governance implementation. After the 5-minute `mark_failed_at_seconds` deadline elapses, any anonymous ingress sender can call this function to forcibly abort an adopted SNS upgrade proposal, marking it `Failed` and clearing the `pending_version` lock. An attacker can repeat this indefinitely to permanently prevent any SNS from upgrading its canisters.

## Finding Description
The canister entry point at `rs/sns/governance/canister/canister.rs` lines 526–535 exposes the function as a plain `#[update]` with no `caller()` check:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(
        sns_gov_pb::FailStuckUpgradeInProgressRequest::from(request),
    ))
}
```

There is no `inspect_message` hook in the canister file that would filter ingress callers. The underlying implementation at `rs/sns/governance/src/governance.rs` lines 6327–6361 also performs no authorization check — the request parameter is explicitly ignored (`_: FailStuckUpgradeInProgressRequest`). The only guard is a time check: `if now > pending_version.mark_failed_at_seconds`. When an `UpgradeSnsToNextVersion` proposal is adopted and execution begins, `pending_version` is set with `mark_failed_at_seconds = self.env.now() + 5 * 60` (line 2896). After those 5 minutes, the function unconditionally calls `complete_sns_upgrade_to_next_version`, which clears `proto.pending_version` and sets the associated proposal's status to `Failed` with `ErrorType::External`. The incomplete authorization design is acknowledged by the developer comment at line 6337: `// Maybe, we should look at the checking_upgrade_lock field and only proceed if it is false, or the request has force set to true.`

## Impact Explanation
An attacker can permanently block all SNS upgrades by polling governance state and calling `fail_stuck_upgrade_in_progress` immediately after each 5-minute deadline. Each call irreversibly marks the adopted upgrade proposal as `Failed`, forces the SNS community to re-submit and re-vote, and produces a misleading `ExternalFailure` journal entry. Because the attack is repeatable at zero cost with no stake or neuron required, security patches, bug fixes, and feature upgrades can be indefinitely prevented for any targeted SNS. This constitutes a significant SNS governance liveness impact with concrete protocol harm, matching the High ($2,000–$10,000) bounty tier: "Significant SNS... security impact with concrete user or protocol harm."

## Likelihood Explanation
The function is unconditionally reachable via any ingress message. No neuron, stake, or privileged role is required. The 5-minute window is predictable and observable on-chain via `get_running_sns_version` or `get_upgrade_journal`. Any motivated attacker can automate this with a simple polling loop at negligible cost, and can repeat it indefinitely.

## Recommendation
Add a caller authorization check at the canister entry point in `rs/sns/governance/canister/canister.rs` before delegating to governance logic. Acceptable callers should be restricted to, for example, the SNS root canister principal or any neuron holder (requiring a valid neuron subaccount). At minimum, the `caller()` should be checked against a whitelist of authorized principals. The incomplete developer comment at line 6337 of `rs/sns/governance/src/governance.rs` should also be resolved to enforce a clear precondition policy.

## Proof of Concept
1. Submit an `UpgradeSnsToNextVersion` proposal to any SNS governance canister and get it adopted.
2. Observe that `pending_version` is set with `mark_failed_at_seconds = T + 300` via `get_running_sns_version`.
3. At time `T + 301`, send an ingress update call to `fail_stuck_upgrade_in_progress` (with empty record `{}`) from any anonymous principal — no neuron required.
4. Observe that `pending_version` is cleared and the proposal transitions to `Failed` with `ErrorType::External` and a message containing `"manually aborted"` (confirmed by the unit test at `rs/sns/governance/src/governance/fail_stuck_upgrade_in_progress_tests.rs` lines 216–299, which exercises exactly this path).
5. Repeat from step 1 to permanently block all SNS upgrades.
