Audit Report

## Title
Permanent ckETH Fee Loss on ckERC20 Withdrawal Out-of-Gas Failure Due to Fixed `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` - (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

The ckETH minter hardcodes `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 65_000` for every ckERC20 withdrawal transaction. When a supported ERC-20 token's `transfer()` consumes more than 65,000 gas and the Ethereum transaction reverts out-of-gas, the minter reimburses only the ckERC20 withdrawal amount; the ckETH burned upfront to pay the transaction fee is permanently lost with no reimbursement path. This is confirmed by both the production code path and the existing test `should_reimburse_tokens_when_ckerc20_withdrawal_fails`.

## Finding Description

**Hardcoded gas limit:**
`CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` is a compile-time constant of `65_000` applied uniformly to every ckERC20 withdrawal regardless of token. [1](#0-0) 

`estimate_gas_limit` returns this constant for all `CkErc20` requests with no per-token override. [2](#0-1) 

**ckETH fee burned before transaction is sent:**
`estimate_erc20_transaction_fee()` computes the fee using `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` and immediately burns it from the user's ckETH account via `burn_from`. This burn is irreversible. [3](#0-2) [4](#0-3) 

**Transaction built with the fixed gas limit:**
The `gas_limit` field of the EIP-1559 transaction is set directly from the constant. [5](#0-4) 

**On Ethereum transaction failure, only ckERC20 tokens are reimbursed — ckETH fee is not:**
`record_finalized_transaction` for `CkErc20` creates a `ReimbursementRequest` with `reimbursed_amount: request.withdrawal_amount.change_units()` — this is the ERC-20 token amount only. The `max_transaction_fee` (ckETH) field of the `Erc20WithdrawalRequest` is never included in any reimbursement. [6](#0-5) 

**Contrast with ckETH withdrawal failure handling:**
For ckETH withdrawals, the reimbursement is `finalized_tx.transaction_amount()` — the withdrawal amount minus the actual fee paid — so the user recovers the unspent portion. No equivalent recovery exists for the ckETH fee in ckERC20 withdrawals. [7](#0-6) 

**Confirmed by existing test:**
The test `should_reimburse_tokens_when_ckerc20_withdrawal_fails` explicitly asserts that on `TransactionStatus::Failure`, only `withdrawal_amount` (ckERC20) is reimbursed — the `max_transaction_fee` ckETH is silently consumed. [8](#0-7) 

**Documentation acknowledges the fixed limit and non-reimbursement:**
The docs state the gas limit "should be sufficient for standard ERC-20 contracts" (the qualifier "standard" is critical) and that "Overcharged transaction fees are not reimbursed." The latter statement addresses the normal success case where `gas_used < gas_limit`; it does not address the out-of-gas failure case where the full fee is consumed and the transaction reverts. [9](#0-8) 

## Impact Explanation

When a governance-approved ckERC20 token's `transfer()` requires more than 65,000 gas:

1. User's ckETH is burned upfront (`max_transaction_fee` = `65_000 * max_fee_per_gas * safety_margin`).
2. Ethereum transaction is submitted with `gas_limit = 65_000`.
3. EVM exhausts all 65,000 gas and reverts out-of-gas.
4. Ethereum charges the full `65_000 * effective_gas_price` fee to the minter's address.
5. Minter reimburses the ckERC20 withdrawal amount to the user.
6. The ckETH burned for the fee is **permanently lost** — no reimbursement path exists in the code.

This matches the allowed impact: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token security impact with concrete user fund harm.** The per-user loss equals `max_transaction_fee` (typically 0.001–0.05 ETH at current gas prices), with no recourse and no retry mechanism that recovers the lost fee.

## Likelihood Explanation

- **Entry path:** Any unprivileged IC principal can call `withdraw_erc20` on the ckETH minter — a fully public ingress endpoint.
- **Trigger condition:** A governance-approved ERC-20 token whose `transfer()` consumes ≥ 65,000 gas. Real-world examples include USDT (blacklist checks, can approach this threshold on cold storage slots), fee-on-transfer tokens, and tokens with complex storage layouts. The 65,000 limit is at the boundary for several production tokens.
- **No special privileges required:** The victim is any user withdrawing a borderline token; no attacker action is needed beyond initiating a normal withdrawal.
- **Likelihood:** Medium. Depends on which tokens are governance-approved. As the set of supported ckERC20 tokens grows, the probability of including a token that exceeds 65,000 gas increases. Currently supported tokens (USDC, USDT) can approach this threshold under certain conditions (e.g., first transfer to a cold address).

## Recommendation

1. **Per-token gas limit configuration:** Allow governance to set a per-token `gas_limit` override when adding a new ckERC20 token via `add_ckerc20_token`, rather than using a single global constant.
2. **Dynamic gas estimation:** Before submitting the withdrawal transaction, use `eth_estimateGas` via the EVM RPC canister to estimate the actual gas required for the specific token and amount.
3. **Reimburse ckETH on out-of-gas failure:** Detect when `gas_used == gas_limit` in the receipt (a strong signal of out-of-gas) and reimburse the unused portion of `max_transaction_fee` minus the actual fee paid, mirroring the existing ckETH withdrawal reimbursement logic at `record_finalized_transaction` for `CkEth`.
4. **Increase the default limit conservatively:** Raise `CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` to a safer value (e.g., 100,000–150,000) to accommodate a wider range of standard tokens.

## Proof of Concept

The existing unit test `should_reimburse_tokens_when_ckerc20_withdrawal_fails` already proves the reimbursement gap — it asserts that on `TransactionStatus::Failure`, only the ckERC20 `withdrawal_amount` is reimbursed and no ckETH reimbursement request is created. [10](#0-9) 

To reproduce the out-of-gas scenario specifically, extend this test:

1. Set `gas_used = gas_limit = CKERC20_WITHDRAWAL_TRANSACTION_GAS_LIMIT` (65,000) in the receipt to simulate out-of-gas.
2. Set `receipt.status = TransactionStatus::Failure`.
3. Call `record_finalized_transaction`.
4. Assert that `reimbursement_requests` contains only a `ReimbursementIndex::CkErc20` entry for the ckERC20 token amount.
5. Assert that **no** `ReimbursementIndex::CkEth` entry exists for the `max_transaction_fee`.
6. Assert that `max_transaction_fee` (ckETH) is permanently unaccounted for.

This test will pass against the current code, confirming the permanent ckETH loss on out-of-gas failure.

```text
. The ckETH minter attempts to estimate the current transaction fee and tries to burn the necessary amount of ckETH to pay for the transaction. The `gas_limit` for ckERC20 withdrawals is currently fixed to `65_000` and should be sufficient for standard ERC-20 contracts. This estimate must include some safety margin to ensure that the minter can resubmit the transaction if necessary, which requires an increase of at least 10% in the max priority fee per gas. If the burn fails (e.g., insufficient funds), the withdrawal request will be rejected. If the burn succeeds, the burn transaction index is used as the request identifier.
. The minter attempts to burn the specified token amount from the user account on the ckERC20 ledger. If the burn succeeds, the minter schedules a withdrawal task. If the burn fails (e.g., insufficient funds), the minter schedules the reimbursement of the burnt ckETH amount from the previous step minus some (small) penalty fee.
. The ckETH minter constructs a 0-ETH amount transaction containing the ERC-20 withdrawal (in `data` field) to the Ethereum network.
. The user can query the withdrawal status using the identifier from the erc20_withdraw response.
. Once the transaction gets enough confirmations, the minter considers the transaction finalized.
. The minter retrieves the receipt of the finalized transaction (as done currently by the ckETH minter) and will reimburse the ckERC20 tokens in case the transaction failed. Overcharged transaction fees are not reimbursed.
```
