### Title
Unprivileged Callers Can Trigger Unbounded Parallel `canister_status` Calls via `get_sns_canisters_summary` — (`rs/sns/root/canister/canister.rs`)

### Summary

The `get_sns_canisters_summary` update method has no access control when called with `update_canister_list: false`. Any unprivileged principal can invoke it, causing the SNS root canister to fan out parallel `canister_status` calls to the management canister for every registered dapp and archive canister. This constitutes a constrained cycles-drain and message-queue-pressure vector.

### Finding Description

In `rs/sns/root/canister/canister.rs`, the `get_sns_canisters_summary` handler is declared as an `#[update]` method (ingress-accessible by any caller): [1](#0-0) 

The only guard present is: [2](#0-1) 

When `update_canister_list` is `false` (the default), **no caller check is performed**. The function then delegates to `SnsRootCanister::get_sns_canisters_summary`, which (as confirmed by the single `join!` match in `lib.rs`) fans out parallel `canister_status` calls to the management canister for all registered dapp and archive canisters. [3](#0-2) 

### Impact Explanation

Each invocation with N registered dapps + archives sends N concurrent `canister_status` calls to the management canister, all paid from the SNS root canister's cycles balance. Repeated concurrent invocations by unprivileged callers:

1. **Cycles drain**: Management canister calls are charged to the calling canister (SNS root). With 100 dapps + archives, each invocation multiplies the cycles cost. Sustained concurrent calls accelerate depletion toward the freezing threshold.
2. **Output queue pressure**: The IC caps a canister's outgoing message queue. With many concurrent `get_sns_canisters_summary` calls each sending 100+ messages, the queue can fill, causing legitimate outgoing calls (governance, upgrades) to be dropped with `SYS_TRANSIENT` errors.

### Likelihood Explanation

The attack path is fully concrete: any principal sends repeated update calls to `get_sns_canisters_summary` with `update_canister_list: false`. No privileged access, no key material, no social engineering required. The IC's ingress rate limiting at boundary nodes provides partial mitigation, but multiple principals or subnets can coordinate. The IC's freezing threshold prevents complete cycles depletion but does not prevent the canister from becoming unresponsive to legitimate traffic while under sustained load.

### Recommendation

Add a caller check or convert this to a query call (which does not consume SNS root cycles for outgoing calls in the same way), or implement rate limiting. At minimum, restrict `get_sns_canisters_summary` to governance, swap, or known trusted callers, consistent with the pattern used by all other state-mutating methods on this canister. [4](#0-3) 

### Proof of Concept

State-machine test: register 100 dapp canisters via governance, then send concurrent `get_sns_canisters_summary` calls from an anonymous principal with `update_canister_list: false`. Measure cycles balance of SNS root before and after, and observe outgoing message queue depth. The cycles delta per call scales linearly with the number of registered canisters, confirming the amplification.

async fn get_sns_canisters_summary(

) -> GetSnsCanistersSummaryResponse {
    log!(INFO, "get_sns_canisters_summary");
    let update_canister_list = request.update_canister_list.unwrap_or(false);
    // Only governance can set this parameter to true
    if update_canister_list {
        assert_eq_governance_canister_id(PrincipalId(ic_cdk::api::caller()));
    }

    let canister_env = CanisterEnvironment {};
    SnsRootCanister::get_sns_canisters_summary(
        &STATE,
        &ManagementCanisterClientImpl::<CanisterRuntime>::new(None),
        &create_ledger_client(),
        &canister_env,
        update_canister_list,
        PrincipalId(ic_cdk::api::id()),
    )
    .await
}
