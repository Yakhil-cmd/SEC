### Title
`set_eth_connector_contract_account` Accepts a Mismatched Connector Without Back-Reference Validation, Freezing All Connector Operations - (File: engine/src/contract_methods/connector.rs)

---

### Summary

`set_eth_connector_contract_account` stores an arbitrary connector account ID with no cross-contract check that the new connector's internal `aurora_engine_account_id` field equals the current engine's own account ID. If the owner sets a connector whose `aurora_engine_account_id` points to a different engine, every subsequent call the engine makes to that connector (`engine_withdraw`, `engine_ft_transfer`, `engine_ft_transfer_call`) is rejected by the connector with "Method can be called only by aurora engine," permanently blocking all user withdrawals and token transfers until the misconfiguration is corrected.

---

### Finding Description

The engine maintains a stored connector account ID (key `EthConnectorStorageId::EthConnectorAccount`). Every user-facing connector operation — `withdraw`, `ft_transfer`, `ft_transfer_call` — resolves to a cross-contract promise dispatched to that stored account via `return_promise`. [1](#0-0) 

The connector contract enforces that only the engine whose account ID it was initialized with may call `engine_withdraw`. This is confirmed by the integration test: [2](#0-1) 

The function that changes the stored connector account is: [3](#0-2) 

It performs no cross-contract query to verify that `args.account`'s `aurora_engine_account_id` equals `env.current_account_id()`. It simply writes the new account ID to storage unconditionally: [4](#0-3) 

Additionally, `ft_on_transfer` uses the stored connector account ID to distinguish base-token deposits from ERC-20 transfers: [5](#0-4) 

If the stored connector account is replaced with a mismatched one, the old connector's `ft_on_transfer` calls are no longer recognized as base-token deposits, silently misrouting them.

---

### Impact Explanation

After `set_eth_connector_contract_account` is called with a connector whose `aurora_engine_account_id != engine.current_account_id()`:

1. Every `withdraw` call dispatches `engine_withdraw` to the mismatched connector, which rejects it — all user ETH withdrawals fail.
2. Every `ft_transfer` / `ft_transfer_call` dispatches `engine_ft_transfer` / `engine_ft_transfer_call` to the mismatched connector, which also rejects them — all NEP-141 token transfers fail.
3. Deposits via the old connector's `ft_on_transfer` are misclassified as ERC-20 transfers, breaking the deposit flow.

All bridged funds held in the connector become inaccessible to users. This constitutes **High — Temporary freezing of funds** (resolvable only by the owner calling `set_eth_connector_contract_account` again with a correctly configured connector; if the owner key is unavailable, the freeze is permanent).

---

### Likelihood Explanation

The function is callable by the engine owner or via a private (self) call: [6](#0-5) 

The owner is a legitimate privileged actor, not a compromised one. The scenario is a configuration mistake — deploying a new connector instance (e.g., for an upgrade) that was initialized with a different `aurora_engine_account_id` — which the code provides no guard against. The external report's judge accepted an identical pattern (admin calling `changeController` with a mismatched factory reference) as a valid medium/high finding. Likelihood is **Low** but realistic during connector upgrades or migrations.

---

### Recommendation

Before persisting the new connector account ID, perform a cross-contract view call to `get_aurora_engine_account_id` on the candidate connector and assert the returned value equals `env.current_account_id()`. This mirrors the check recommended in the external report for `VaultFactory.createNewMarket`. Alternatively, require the caller to supply the expected `aurora_engine_account_id` as a parameter and validate it locally, avoiding an async round-trip.

---

### Proof of Concept

1. Owner deploys a new eth-connector instance initialized with `aurora_engine_account_id = "other-engine.near"` (e.g., a staging engine).
2. Owner calls `set_eth_connector_contract_account` on the production engine pointing to this new connector. The call succeeds — no validation occurs.
3. Any user calls `withdraw` on the production engine. The engine dispatches `engine_withdraw` to the new connector.
4. The new connector checks `predecessor_account_id == "other-engine.near"` but receives `"aurora"` — rejects with "Method can be called only by aurora engine."
5. All user withdrawals, `ft_transfer`, and `ft_transfer_call` operations revert. Bridged ETH and NEP-141 tokens are frozen. [3](#0-2) [1](#0-0) [7](#0-6)

#[named]
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
