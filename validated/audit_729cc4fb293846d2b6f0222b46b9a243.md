### Title
Floor-Division Truncation of `native_per_pubdata` Causes Pubdata to Be Undercharged When `pubdata_price < native_price` - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When computing `native_per_pubdata` — the number of native resource units charged per byte of pubdata — the bootloader uses integer floor division (`wrapping_div`) of `pubdata_price` by `native_price`. Whenever `pubdata_price < native_price`, this division truncates to zero, making pubdata completely free in terms of native resources for the entire transaction. Any user can exploit this by submitting storage-heavy transactions that generate large amounts of pubdata at zero native cost.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (ZK transaction path), the key ratio is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 500`, `native_price = 1000`), the result is `0`. This zero propagates through the entire fee pipeline:

**1. Intrinsic pubdata overhead is zeroed** in `create_resources_for_tx`:
```rust
// gas_helpers.rs, line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0; native_limit is not reduced
```

**2. Post-execution pubdata charge is zeroed** in `get_resources_to_charge_for_pubdata`:
```rust
// gas_helpers.rs, line 430-431
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // → N * 0 = 0
    .ok_or(out_of_native_resources!())?;
```

**3. The `delta_gas` adjustment is suppressed** in `compute_gas_refund`:
```rust
// refund_calculation.rs, line 64
let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());
// native_used is low (no pubdata native charged), so delta_gas ≤ 0 → no extra gas charged
```

### Summary

When computing `native_per_pubdata` — the number of native resource units charged per byte of pubdata — the bootloader uses integer floor division (`wrapping_div`). Whenever `pubdata_price < native_price`, the result truncates to zero, making all pubdata completely free in terms of native resources. Any user can then write arbitrary amounts of pubdata (up to the block pubdata limit) without paying the native cost the operator intended to charge.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142, the ZK transaction validation path computes:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer floor division. When `pubdata_price < native_price`, the quotient is zero. The same truncation appears in the off-chain helper `api/src/helpers.rs` at line 427:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

`native_per_pubdata = 0` then propagates into two downstream accounting functions:

**1. `create_resources_for_tx` (`gas_helpers.rs` line 352):**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
let native_limit = match native_limit.checked_sub(intrinsic_pubdata_overhead) { ... };
```
With `native_per_pubdata_byte = 0`, no native is withheld upfront for intrinsic pubdata.

**2. `get_resources_to_charge_for_pubdata` (`gas_helpers.rs` line 430):**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
```
With `native_per_pubdata = 0`, the native charge for all pubdata written during execution is zero.

The `delta_gas` adjustment in `refund_calculation.rs` (line 72) also fails to recover the missing pubdata cost:
```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
```
Because `native_used` does not include pubdata native (it was never charged), `delta_gas` does not compensate, and the user receives a full gas refund for pubdata they consumed.

**Concrete example:**
- Operator sets `native_price = 1000`, `pubdata_price = 999`
- Intended: charge ~1 native unit per pubdata byte
- Actual: `999 / 1000 = 0` → pubdata is free
- A user submitting a transaction with 10,000 pubdata bytes pays 0 native for pubdata instead of ~10,000 native units

---

### Impact Explanation

This is a **resource accounting bug**. When `pubdata_price < native_price` (a plausible operator configuration), `native_per_pubdata` silently becomes zero. Every transaction in that block can write pubdata at zero native cost. The operator's intended pubdata pricing is completely bypassed. Users are undercharged, operator revenue from pubdata fees is lost, and the native resource budget is not correctly enforced for pubdata consumption. The block pubdata limit (`pubdata_limit`) is the only remaining guard, but it is a count limit, not a cost limit — it does not restore correct fee accounting.

---

### Likelihood Explanation

The operator sets both `native_price` and `pubdata_price` independently via the oracle/block metadata. These represent different physical quantities (cost per RISC-V cycle vs. cost per pubdata byte), and there is no protocol-level constraint requiring `pubdata_price >= native_price`. Any deployment where the operator configures `pubdata_price < native_price` — which is a natural configuration when pubdata is cheap relative to computation — silently activates this bug for every transaction in every block. No special attacker capability is required; any standard transaction with storage writes exploits the condition.

---

### Recommendation

Replace floor division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Current (floor division — truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// Fix (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that any non-zero `pubdata_price` results in at least 1 native unit charged per pubdata byte, eliminating the silent truncation-to-zero.

**Setup:**

- User submits a transaction that writes 100 storage slots (each slot write = 64 pubdata bytes → 6400 pubdata bytes total)

   `native_per_pubdata = 500u256.wrapping_div(1000u256) = 0`

   → Full native limit allocated to computation, nothing withheld for pubdata.

3. Transaction executes, writes 6400 bytes of pubdata.

4. `gas_helpers.rs` line 430 (`get_resources_to_charge_for_pubdata`):
   `native = 6400 * 0 = 0`
   → Zero native charged for 6400 bytes of pubdata.

   `delta_gas` does not compensate for the missing pubdata cost.
