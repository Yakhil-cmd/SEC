## The Invariant Mismatch

The per-block limit allows exactly `16_384` logs in a single block. The batch-level `ArrayVec` has capacity `16_384` for the **entire batch**. Therefore:

- Block 1 with exactly 16,384 logs: fills `batch_data.logs_storage` to capacity.
- Block 2 with ≥1 log: `apply_to_array_vec` calls `array_vec.push(...)` on a full `ArrayVec` → **panic**.

The forward/sequencing mode uses `LogsStorage` backed by a growable `HistoryList` (not an `ArrayVec`), so it never panics. The proving mode uses the fixed-capacity `ArrayVec` and panics. This is a forward/proving inconsistency. [5](#0-4)

## Attacker Reachability

L1→L2 priority transactions are submitted from L1 by any unprivileged user. Each such transaction emits exactly one log via `push_l1_l2_tx_log` (called unconditionally for priority ops). [6](#0-5) 

User messages via the L1 messenger system hook are also accessible to any contract. [7](#0-6)

## Verdict

### Title
Unchecked `ArrayVec::push` in `apply_to_array_vec` Causes Proving Panic in Multiblock Batches — (`basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`, `zk_ee/src/common_structs/logs_storage.rs`)

### Summary
The per-block log limit allows exactly `MAX_NUMBER_OF_LOGS = 16_384` logs per block, but the batch-level `ArrayVec<Bytes32, 16384>` accumulates logs across all blocks with the same fixed capacity. In a multiblock batch, if block 1 reaches the per-block limit (16,384 logs) and block 2 has ≥1 log, `apply_to_array_vec` panics in proving mode. Forward/sequencing mode uses a growable structure and never panics, creating a forward/proving inconsistency that makes the batch unprovable.

### Finding Description
`ZKBatchDataKeeper::logs_storage` is `ArrayVec<Bytes32, 16384>`. After each block in a multiblock batch, `LogsStorage::apply_to_array_vec` iterates all per-block logs and calls `array_vec.push(log.hash())` with no capacity check. `ArrayVec::push` panics on overflow. The per-block guard (`logs_used > MAX_NUMBER_OF_LOGS`) permits exactly 16,384 logs per block — exactly the batch accumulator's total capacity. Any second block with ≥1 log after a full first block overflows the accumulator.

### Impact Explanation
The proving run aborts with a Rust panic. The batch is unprovable. Forward mode (sequencing) succeeds because it uses a growable `HistoryList`. This is a forward/proving mismatch requiring a code fix.

### Likelihood Explanation
Any operator constructing a multiblock batch with a first block at the log limit (achievable via 16,384 L1→L2 priority transactions, submittable by any unprivileged L1 user) and a second block with any log triggers the panic. The path is concrete and locally testable.

### Recommendation
In `apply_to_array_vec`, check remaining capacity before each push and return an error, or assert `self.list.len() + array_vec.len() <= 16384` before iterating. Alternatively, enforce a **batch-level** log limit (not just per-block) so the total across all blocks in a batch never exceeds 16,384.

### Proof of Concept
1. Submit 16,384 L1→L2 priority transactions targeting block 1 of a multiblock batch.
2. Submit 1 additional L1→L2 transaction targeting block 2.
3. Run the prover in multiblock batch mode.
4. `apply_to_array_vec` panics at the first `push` for block 2's log.
5. The same batch succeeds in forward/sequencing mode.
