Audit Report

## Title
Silent ICP Burn Failure in Cycles Minting Canister Allows Cycles Minting Without ICP Conservation — (`rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the Cycles Minting Canister (CMC) intentionally swallows all errors from the ICP ledger burn call after cycles have already been irrevocably deposited or minted. If the ICP ledger returns a transient error during `burn_and_log`, cycles are delivered to the recipient while the corresponding ICP remains permanently stranded in the CMC's subaccount — unburned and unrecoverable — violating the ICP-to-cycles conservation invariant. The notification block is then marked as fully processed, preventing any retry.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` (L2014–2049) explicitly does not propagate errors from the ledger `send_pb` call:

```rust
match res {
    Ok(block) => print(format!("{msg} done in block {block}.")),
    Err((code, err)) => {
        let code = code as i32;
        print(format!("{msg} failed with code {code}: {err:?}"))
        // error is only logged; function returns ()
    }
}
```

This function is called on the success path of all three notification handlers, after cycles have already been irrevocably delivered:

- `process_create_canister` (L1943–1946): after `do_create_canister` succeeds
- `process_mint_cycles` (L1966–1968): after `do_mint_cycles` succeeds
- `process_top_up` (L1999–2001): after `deposit_cycles` succeeds

In all three cases, the caller's notification block is subsequently recorded as `NotifiedCreateCanister`, `NotifiedMint`, or `NotifiedTopUp` respectively, permanently closing the notification and preventing any retry. The `burn_and_log` call uses the `amount` from the current notification only — it does not sweep the full subaccount balance — so any ICP left unburned from a failed call is never retried and accumulates permanently in the CMC's per-canister or per-principal subaccount.

The design rationale (stated in the comment at L2015–2016) is to avoid double-minting: if the burn failure caused the notification to return an error, the caller could retry and receive cycles again. However, the consequence of the chosen design is that a transient ledger error at this exact point causes a permanent conservation violation with no recovery path.

## Impact Explanation
Cycles are minted without the corresponding ICP being burned. The ICP-to-cycles conservation invariant — that every cycle in circulation is backed by burned ICP — is violated. The unburned ICP accumulates permanently in CMC subaccounts, inaccessible to users and not burned. Over time, repeated transient ledger failures during `burn_and_log` inflate the effective cycles supply relative to burned ICP. This constitutes illegal minting of cycles and a significant NNS/CMC financial integrity impact with concrete protocol harm, matching the High impact tier ($2,000–$10,000): significant NNS/CMC ledger security impact with concrete user or protocol harm.

## Likelihood Explanation
The ICP ledger is a system canister that undergoes scheduled upgrades (evidenced by the upgrade history in `rs/ledger_suite/icp/UPGRADES.md`). During an upgrade, the ledger canister is stopped/starting and will return a transient call error to any caller. The `TemporarilyUnavailable` variant is a first-class error in the ICP ledger's transfer interface (defined in `packages/icrc-ledger-types/src/icrc1/transfer.rs` L68). Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during a ledger upgrade window triggers the condition without any special privilege. The window is narrow but is a real, recurring operational condition (ledger upgrades happen regularly), not a theoretical one.

## Recommendation
1. **Persist failed burns and retry them.** When `burn_and_log` fails, store the `(subaccount, amount)` pair in stable state and retry in a heartbeat or timer until the burn succeeds, similar to how ckBTC minter handles pending reimbursements (`rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`).
2. **Alternatively**, expose a privileged governance endpoint to trigger a re-burn for a specific subaccount, allowing the conservation invariant to be restored after a ledger outage.
3. At minimum, emit a certified metric or counter for failed burns so that conservation violations are observable and auditable.

## Proof of Concept
1. User sends N ICP to the CMC subaccount corresponding to canister `C` with `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id: C }` timed to coincide with an ICP ledger upgrade (ledger is stopped/starting).
3. CMC calls `deposit_cycles(C, cycles, true, limiter_to_use)` — this succeeds; cycles are now in canister `C`.
4. CMC calls `burn_and_log(sub, amount)` — the ICP ledger is temporarily unavailable (mid-upgrade) and returns a call error.
5. `burn_and_log` logs the error and returns `()` — no error is propagated.
6. CMC records `NotifiedTopUp(Ok(cycles))` for the block — the notification is permanently closed.
7. Result: canister `C` received cycles; the N ICP in the CMC subaccount is never burned and is permanently stranded. The cycles supply is inflated relative to burned ICP by N ICP worth of cycles.

A deterministic integration test using PocketIC can reproduce this by: installing the CMC and a mock ICP ledger, having the mock ledger return a `CanisterStopping`/`CanisterStopped` reject code on the `send_pb` call, triggering `notify_top_up`, and then asserting that the CMC subaccount balance is non-zero while the target canister's cycle balance increased.
