Audit Report

## Title
VIP Caller Protection Bypass via Slot Exhaustion in NNS Root `canister_status` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary

`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` unconditionally rejects all callers when `available_slot_count == 0`, including NNS governance callers that are designated as VIP and should never consume a slot. Any unprivileged principal can exhaust all 167 slots by flooding the public, access-control-free `canister_status` endpoint on NNS Root, temporarily blocking governance-initiated calls to `update_canister_settings`, `take_canister_snapshot`, `load_canister_snapshot`, and SNS-W-initiated `change_canister_controllers`.

## Finding Description

**Root cause — `try_borrow_slot` in `rs/nervous_system/clients/src/management_canister_client.rs` (L264–287):**

The function correctly sets `used_slot_count = 0` for VIP callers, meaning they should never deplete the pool. However, the zero-check fires unconditionally before the VIP path takes effect:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {   // ← fires for VIP callers too
                return Err((RejectCode::SysTransient as i32, "Unavailable...".to_string()));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
``` [1](#0-0) 

**Public entrypoint with no access control — `canister_status` (L88–98):**

```rust
#[update]
async fn canister_status(canister_id_record: CanisterIdRecord) -> CanisterStatusResult {
    let client = new_management_canister_client();
    ...
}
``` [2](#0-1) 

Any principal can call this. Each call holds a slot for the duration of the inter-canister call to the management canister (the canister yields at the `await` point).

**Slot pool — 167 slots (L47–51):** [3](#0-2) 

**VIP determination (L57–66):** NNS canister principals get `is_caller_vip = true`; all others get `false` and consume 1 slot per call. [4](#0-3) 

**Affected governance functions** — all call `new_management_canister_client()` and are therefore subject to the slot guard:

- `update_canister_settings` (L221–230) — callable by NNS Governance
- `take_canister_snapshot` (L244–254) — callable by NNS Governance
- `load_canister_snapshot` (L258–268) — callable by NNS Governance
- `change_canister_controllers` (L207–216) — callable by SNS-W [5](#0-4) [6](#0-5) 

**Not affected** — these bypass `ManagementCanisterClient` entirely:

- `change_nns_canister` calls `change_canister()` from `ic_nervous_system_root` directly.
- `stop_or_start_nns_canister` calls `start_canister`/`stop_canister` from `ic_nervous_system_root` directly.
- `create_canister_and_install_code` explicitly bypasses rate limiting (comment at L261–268 in `canister_management.rs`). [7](#0-6) [8](#0-7) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

An unprivileged attacker can block NNS Governance from executing `update_canister_settings`, `take_canister_snapshot`, and `load_canister_snapshot` proposals, and block SNS-W from executing `change_canister_controllers`. These are legitimate governance operations with real protocol impact. The primary NNS canister upgrade path (`change_nns_canister`) is not blocked, which limits severity to High rather than Critical.

## Likelihood Explanation

The attack requires no special privileges. Any principal can call `canister_status` on NNS Root. Sending 167 concurrent update calls exhausts the slot pool. The attacker must sustain the flood (replenish calls as slots free up) to maintain the DoS. Each update call costs cycles, imposing a cost on the attacker, but the cost is not prohibitive given the value of blocking governance operations. The IC's single-threaded execution model allows all 167 slots to be held simultaneously across interleaved message processing.

## Recommendation

Fix `try_borrow_slot` to skip the zero-check for VIP callers:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                let message = "Unavailable. Maybe, try again later?".to_string();
                return Err((code, message));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

This ensures VIP callers always succeed regardless of the current slot count, matching the stated design intent.

## Proof of Concept

A state-machine test using `StateMachine` / PocketIC would:

1. Deploy NNS with all canisters including Root and Governance.
2. From a non-NNS principal, submit 167 concurrent `canister_status` update calls to NNS Root targeting a valid canister ID, without waiting for responses (slots are held at the `await` point).
3. While those calls are in flight, submit a Governance-originated `update_canister_settings` call to NNS Root.
4. Assert the governance call returns `Err(SysTransient, "Unavailable. Maybe, try again later?")` — demonstrating VIP protection is broken.
5. Allow the 167 `canister_status` calls to complete (slots freed), retry the governance call, and assert it succeeds. [9](#0-8)
