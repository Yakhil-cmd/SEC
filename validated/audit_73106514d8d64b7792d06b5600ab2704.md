### Title
SMOD Opcode Panics on Zero Divisor with Non-Zero Dividend — (`evm_interpreter/src/i256.rs`)

### Summary

`i256_mod`, which implements the EVM `SMOD` opcode, does not guard against a zero divisor when the dividend is non-zero. The underlying `div_rem` call panics on division by zero. The EVM specification requires `SMOD(x, 0) = 0`; instead, the implementation causes a fatal panic that halts block execution, enabling a DoS by any unprivileged caller who deploys a contract executing `SMOD(nonzero, 0)`.

### Finding Description

In `evm_interpreter/src/i256.rs`, `i256_mod` checks whether the **dividend** is zero and returns early, but never checks whether the **divisor** is zero:

```rust
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {          // ← only dividend is guarded
        *divisor_or_remainder = U256::ZERO;
        return;
    }

    let _ = i256_sign::<true>(divisor_or_remainder); // does NOT return on Sign::Zero

    // this is unsigned division of moduli
    let (_q, r) = dividend.div_rem(*divisor_or_remainder); // ← PANICS when divisor == 0
``` [1](#0-0) 

By contrast, `i256_div` (the `SDIV` implementation) correctly handles a zero divisor as its **first** action:

```rust
pub fn i256_div(dividend: &mut U256, divisor_or_quotient: &mut U256) {
    let divisor_sign = i256_sign::<true>(divisor_or_quotient);
    if divisor_sign == Sign::Zero {
        *divisor_or_quotient = U256::ZERO;
        return;
    }
``` [2](#0-1) 

The `div_rem` primitive is explicitly documented to panic on a zero divisor, and the test suite confirms this:

```rust
#[test]
#[should_panic]
fn naive_divrem_by_zero_panics() {
    let mut x = naive::U256::one();
    let mut y = naive::U256::zero();
    naive::U256::div_rem(&mut x, &mut y);
}
``` [3](#0-2) [4](#0-3) 

### Impact Explanation

The EVM Yellow Paper specifies `SMOD(x, 0) = 0` for any `x`. When a contract executes `SMOD` with a non-zero stack value as dividend and `0` as divisor, `i256_mod` reaches `dividend.div_rem(0)` and panics. In the ZKsync OS RISC-V proving environment, a Rust panic is a fatal, unrecoverable error — it propagates out of the EVM interpreter through the bootloader and aborts block execution/proving. This is a **block-level DoS**: the entire block containing the offending transaction cannot be proven or finalized.

### Likelihood Explanation

The trigger is trivially reachable. Any unprivileged user can:
1. Deploy a contract containing the two-byte sequence `PUSH1 0x00; SMOD` (or any variant that leaves a non-zero value below a zero divisor on the stack).
2. Submit a transaction calling that contract.

No privileged access, oracle manipulation, or external dependency is required. The attacker pays only standard gas fees.

### Recommendation

Mirror the zero-divisor guard from `i256_div` into `i256_mod`. Add the check immediately after the dividend-zero early return:

```rust
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }

    // ADD: guard against zero divisor (EVM spec: SMOD(x, 0) = 0)
    let divisor_sign = i256_sign::<true>(divisor_or_remainder);
    if divisor_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }

    // now safe: both dividend and divisor are non-zero
    let (_q, r) = dividend.div_rem(*divisor_or_remainder);
    ...
}
``` [5](#0-4) 

### Proof of Concept

**Minimal EVM bytecode to trigger the panic:**

```
PUSH1 0x00   // divisor = 0
PUSH1 0x01   // dividend = 1
SMOD         // calls i256_mod(1, 0) → panic
```

**Attack flow:**
1. Attacker deploys a contract with the above bytecode.
2. Attacker submits a transaction calling the contract.
3. The bootloader executes the transaction via the EVM interpreter.
4. `i256_mod` is invoked with `dividend = 1`, `divisor = 0`.
5. Line 136 check: `dividend_sign != Sign::Zero` → no early return.
6. Line 141: `i256_sign::<true>(divisor_or_remainder)` returns `Sign::Zero` but does **not** return early.
7. Line 144: `dividend.div_rem(0)` → **panic**.
8. The panic propagates fatally through the bootloader; the block cannot be proven. [6](#0-5) [7](#0-6)
