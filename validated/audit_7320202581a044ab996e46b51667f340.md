### Title
Query Cache Coherency Break: `ic0.canister_liquid_cycle_balance128()` Not Tracked for Balance-Change Invalidation — (`File: rs/execution_environment/src/query_handler/query_cache.rs`)

### Summary

The IC replica-side query cache determines whether a cached result can be reused despite a changed canister balance by checking whether the query ever called `ic0.canister_cycle_balance()` or `ic0.canister_cycle_balance128()`. A newer System API call, `ic0.canister_liquid_cycle_balance128()`, is tracked in `SystemApiCallCounters` but is **not** included in the `ignore_canister_balances` guard. A query that exclusively uses `ic0.canister_liquid_cycle_balance128()` will have `ignore_canister_balances = true` set, causing the cache to serve stale balance data to all subsequent callers even after the canister's cycle balance has materially changed.

### Finding Description

`EntryValue::new()` in `rs/execution_environment/src/query_handler/query_cache.rs` computes the `ignore_canister_balances` flag as:

```rust
let ignore_canister_balances = system_api_call_counters.canister_cycle_balance == 0
    && system_api_call_counters.canister_cycle_balance128 == 0;
``` [1](#0-0) 

`SystemApiCallCounters` has a dedicated counter for `ic0.canister_liquid_cycle_balance128()`:

```rust
pub canister_liquid_cycle_balance128: usize,
``` [2](#0-1) 

That counter is incremented correctly in the embedder:

```rust
self.call_counters.canister_liquid_cycle_balance128 += 1;
``` [3](#0-2) 

But `ignore_canister_balances` never reads `canister_liquid_cycle_balance128`. Consequently, a query that calls only `ic0.canister_liquid_cycle_balance128()` (and neither of the two older balance APIs) will have `ignore_canister_balances = true` stored in its `EntryValue`.

The `is_valid()` check then allows the cached result to be returned even when `all_canister_balances_are_valid` is `false`:

```rust
&& (all_canister_balances_are_valid || self.ignore_canister_balances)
``` [4](#0-3) 

The liquid balance is computed from the canister's total balance minus the freeze threshold, which itself depends on memory usage and the current time (via `base_per_second_fee`). Therefore the liquid balance is sensitive to **both** balance changes and time advances. Because `ignore_batch_time` is also set to `true` when `ic0.time()` is not called, a query that uses only `ic0.canister_liquid_cycle_balance128()` will have both `ignore_batch_time = true` and `ignore_canister_balances = true`, meaning the cache will serve the same stale result regardless of how much time has passed or how much the balance has changed — up to the 600-second `QUERY_CACHE_MAX_EXPIRY_TIME` hard ceiling. [5](#0-4) 

The `query_cache_future_proof_test` in the test suite explicitly lists `CanisterLiquidCycleBalance128` as a known System API call and warns that adding new balance-dependent calls requires updating the cache coherency logic — but the fix was never applied for this call. [6](#0-5) [7](#0-6) 

### Impact Explanation

Any canister that exposes a query method reporting its liquid cycle balance (e.g., a DeFi vault, a cycles-dispensing canister, or any canister that uses `ic0.canister_liquid_cycle_balance128()` to gate logic) will return a stale, incorrect balance to all callers for up to 10 minutes after the balance changes. Callers relying on this value to decide how many cycles to attach to subsequent calls, or to assess canister solvency, receive misleading data. This is a direct analog to the Sablier finding: time/state-sensitive data is embedded in a cached response and served stale to consumers.

### Likelihood Explanation

`ic0.canister_liquid_cycle_balance128()` is the recommended modern API for querying spendable cycles (the older `ic0.canister_cycle_balance128()` does not subtract the freeze threshold). Any canister author who follows current best practices and uses only the liquid balance API in a query method will silently trigger this cache coherency break. The entry path requires only an unprivileged query call to such a canister.

### Recommendation

Add `canister_liquid_cycle_balance128` to the `ignore_canister_balances` guard in `EntryValue::new()`:

```rust
let ignore_canister_balances = system_api_call_counters.canister_cycle_balance == 0
    && system_api_call_counters.canister_cycle_balance128 == 0
    && system_api_call_counters.canister_liquid_cycle_balance128 == 0;
``` [1](#0-0) 

Additionally, add a corresponding test analogous to `query_cache_returns_different_results_for_different_canister_balances` that exercises `ic0.canister_liquid_cycle_balance128()` to prevent regression.

### Proof of Concept

1. Deploy a canister with a query method that calls only `ic0.canister_liquid_cycle_balance128()` and returns the result.
2. Issue a query call — this populates the cache with `ignore_canister_balances = true`.
3. Transfer cycles into or out of the canister (via an update call), changing its balance.
4. Issue the same query call again within 600 seconds.
5. The replica returns the cached (stale) liquid balance from step 2, not the updated value, because `is_valid()` passes with `ignore_canister_balances = true` despite `all_canister_balances_are_valid` being `false`. [8](#0-7)
