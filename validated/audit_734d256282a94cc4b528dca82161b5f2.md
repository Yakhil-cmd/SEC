Audit Report

## Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_end` Crashes Rosetta Node — (`rs/rosetta-api/icrc1/src/construction_api/services.rs` and `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
Both the ICRC1 and ICP Rosetta `construction_payloads` handlers accept an attacker-controlled `ingress_end` field from unauthenticated HTTP POST requests and use it as the upper bound of a `while ingress_start < ingress_end` loop that allocates one entry per ~120–240 second interval. Setting `ingress_end = u64::MAX` causes the loop to execute ~70–140 million iterations, each allocating heap objects, exhausting the Rosetta process's RAM and crashing it. A single HTTP request with no credentials is sufficient.

## Finding Description

**ICRC1 path — `rs/rosetta-api/icrc1/src/construction_api/services.rs`:**

`ingress_interval = (MAX_INGRESS_TTL − PERMITTED_DRIFT).as_nanos() as u64 = (300s − 60s) × 10⁹ = 240,000,000,000 ns`. `INGRESS_INTERVAL_OVERLAP = 120s`, so the effective loop step is `240,000,000,000 − 120,000,000,000 = 120,000,000,000 ns`.

Two guards precede the loop:

- **Guard 1** (`ingress_start >= ingress_end`): With `ingress_start = now` and `ingress_end = u64::MAX`, evaluates to `false` — does not fire.
- **Guard 2** (`ingress_end < now + ingress_interval`): With `ingress_end = u64::MAX ≈ 1.844×10¹⁹` and `now + ingress_interval ≈ 1.7×10¹⁸ + 2.4×10¹¹`, evaluates to `false` because `u64::MAX` is the largest representable value — does not fire.

Neither guard caps the window size; both are minimum-value checks. The unguarded loop then runs:

```
iterations ≈ (u64::MAX − now) / 120,000,000,000 ≈ 139 million
```

Each iteration pushes a `u64` to `ingress_expiries`. Subsequently, `handle_construction_payloads` iterates over `ingress_expiries` and for each entry allocates an `EnvelopeContent::Call` struct containing `canister_method_args.clone()` (a heap-allocated `Vec<u8>`), plus a `SigningPayload`.

**ICP path — `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`:**

`interval = MAX_INGRESS_TTL − PERMITTED_DRIFT − 120s = 120s`. There are **no guards** between the metadata parse and the loop. With `ingress_end = u64::MAX` nanoseconds, the loop runs for `(u64::MAX − ingress_start) / interval_ns` iterations — potentially hundreds of billions — before any arithmetic overflow. If `ic_types::time::Time`'s `AddAssign<Duration>` wraps in release mode, the loop becomes infinite.

## Impact Explanation
The Rosetta node is a single OS process. Allocating 139+ million heap objects (each containing a cloned `Vec<u8>`) exhausts available RAM, triggering an OOM kill or process crash. This constitutes a **High** severity application-level DoS: "Application/platform-level DoS, crash … or subnet availability impact not based on raw volumetric DDoS" and "Significant … Rosetta … infrastructure security impact with concrete user or protocol harm." Rosetta is the canonical ICP financial integration bridge; crashing it disrupts ICP/ICRC1 ledger access for exchanges and custodians.

## Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication. The `ingress_end` field is a plain `u64` in the JSON metadata with no server-side cap. The exploit requires a single small HTTP POST request, no prior state, no volumetric traffic, and no credentials. The faulty Guard 2 (`ingress_end < now + ingress_interval`) appears protective at a glance but is logically inert for any value near `u64::MAX`. Both paths are affected independently.

## Recommendation
1. After computing `ingress_start` and `ingress_end`, add a window-size cap in both handlers before the loop:
   ```rust
   const MAX_INGRESS_WINDOW: u64 = 24 * 3600 * 1_000_000_000; // 24 hours in ns
   if ingress_end.saturating_sub(ingress_start) > MAX_INGRESS_WINDOW {
       return Err(...);
   }
   ```
2. Use `saturating_add` for `ingress_start` inside the ICRC1 loop to prevent wrap-around infinite loops in release builds.
3. Apply the same fix to both `services.rs` (ICRC1) and `construction_payloads.rs` (ICP).
4. Optionally assert `ingress_expiries.len() <= SOME_REASONABLE_MAX` after the loop as a defense-in-depth check.

## Proof of Concept
```rust
#[test]
fn test_construction_payloads_unbounded_ingress_window() {
    use std::time::SystemTime;
    let now = SystemTime::now();
    // Without the fix, this OOMs or loops forever.
    let result = construction_payloads(
        vec![/* valid transfer operation */],
        Some(ConstructionPayloadsRequestMetadata {
            ingress_start: None,         // defaults to now
            ingress_end: Some(u64::MAX), // attacker-controlled
            created_at_time: None,
            memo: None,
        }),
        &some_ledger_principal,
        vec![valid_public_key()],
        now,
    );
    assert!(result.is_err()); // must return Err with the fix
}
```
