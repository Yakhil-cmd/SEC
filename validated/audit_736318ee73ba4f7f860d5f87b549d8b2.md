Audit Report

## Title
SNS Governance `max_number_of_neurons` Exhaustion Griefing — No Rate Limiting on Neuron Creation - (File: rs/sns/governance/src/governance.rs)

## Summary
SNS governance enforces a hard cap (`max_number_of_neurons`, ceiling 200,000) on total neurons but has no rate limiter on neuron creation, unlike NNS governance which uses a token-bucket limiter (`MAX_SUSTAINED_NEURONS_PER_HOUR = 15`, burst `MAX_NEURON_CREATION_SPIKE = 300`). An unprivileged attacker can bulk-create neurons with the minimum stake and dissolve delay 0 until the cap is reached, permanently blocking all other principals from staking new neurons. The attacker's tokens are fully recoverable via disburse, making the attack low-cost and repeatable.

## Finding Description
**Root cause:** `add_neuron` in `rs/sns/governance/src/governance.rs` (L940–975) only calls `check_heap_can_grow()` and `check_neuron_population_can_grow()` before inserting a neuron. There is no rate-limiting call analogous to the NNS `rate_limiter.try_reserve(...)` at `rs/nns/governance/src/governance.rs` L5992–5996.

**Exploit path:**
1. Attacker calls `manage_neuron { ClaimOrRefresh { MemoAndController } }` in a loop, each time transferring `neuron_minimum_stake_e8s` SNS tokens to the neuron subaccount.
2. Each call reaches `add_neuron`, which passes both guards and inserts the neuron with `DissolveState::DissolveDelaySeconds(0)` (L4347), meaning the neuron is immediately dissolvable.
3. Once `proto.neurons.len() == max_number_of_neurons`, `check_neuron_population_can_grow` (L6365–6379) returns `PreconditionFailed: "Cannot add neuron. Max number of neurons reached."` for every subsequent caller.
4. The `claim_swap_neurons` path (L4535–4546) also calls `add_neuron` and hits the same cap, meaning an active SNS swap whose neurons have not yet been claimed would also be blocked.
5. Attacker recovers all tokens by disbursing the dissolved neurons at will.

**Existing checks are insufficient:** `check_neuron_population_can_grow` is a simple count comparison with no temporal or per-principal dimension. The NNS rate limiter (`new_rate_limiter`, L1261–1271) and its constants (`MAX_SUSTAINED_NEURONS_PER_HOUR`, `MAX_NEURON_CREATION_SPIKE`, L231–240) have no equivalent in the SNS codebase.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … SNS … security impact with concrete user or protocol harm."* Concrete harms include: (a) all new token holders are permanently locked out of SNS governance participation; (b) an in-progress SNS swap's `claim_swap_neurons` call fails, potentially breaking the swap completion flow for all participants.

## Likelihood Explanation
- **Fully unprivileged:** any principal can call `manage_neuron { ClaimOrRefresh }` on any SNS governance canister.
- **Cost is low and recoverable:** the attacker only needs `max_number_of_neurons × neuron_minimum_stake_e8s` tokens temporarily locked; all are returned on disburse.
- **No rate limiting exists** in the SNS path, confirmed by absence of any `rate_limiter` reference in `rs/sns/governance/src/`.
- **Repeatable:** after the community raises the cap via governance proposal, the attacker can immediately fill the new headroom again.

## Recommendation
Add a token-bucket rate limiter to SNS `claim_neuron`, mirroring the NNS implementation:
- Introduce `max_neuron_creation_spike` and `max_sustained_neurons_per_hour` fields to `NervousSystemParameters` (or hard-code conservative defaults).
- Gate `add_neuron` behind a `rate_limiter.try_reserve(...)` call analogous to `rs/nns/governance/src/governance.rs` L5992–5996.
- Alternatively, enforce a per-principal neuron creation rate limit so a single attacker cannot exhaust the global budget in a single burst.

## Proof of Concept
```rust
// Pseudocode — attacker principal calls SNS governance manage_neuron in a loop
let min_stake = nervous_system_parameters.neuron_minimum_stake_e8s;
let max_neurons = nervous_system_parameters.max_number_of_neurons; // up to 200_000

for memo in 0..max_neurons {
    sns_ledger.icrc1_transfer({
        to: neuron_subaccount(attacker_principal, memo),
        amount: min_stake,
    });
    // No rate limit check — goes directly to add_neuron
    sns_governance.manage_neuron({
        subaccount: neuron_subaccount(attacker_principal, memo),
        command: ClaimOrRefresh { by: MemoAndController { memo, controller: None } },
    });
    // Neuron created with DissolveDelaySeconds(0) — immediately dissolvable
}

// Any legitimate user now gets:
// GovernanceError { PreconditionFailed: "Cannot add neuron. Max number of neurons reached." }
victim.manage_neuron({ command: ClaimOrRefresh { ... } }); // → PreconditionFailed

// Attacker recovers all tokens:
for each attacker_neuron {
    sns_governance.manage_neuron({ command: Disburse { ... } });
}
```
A deterministic integration test can be written using PocketIC: initialize an SNS with a small `max_number_of_neurons` (e.g., 10), have a single attacker identity fill all slots, then assert that a second identity's `ClaimOrRefresh` returns `PreconditionFailed`, and that `claim_swap_neurons` also fails.

    fn add_neuron(&mut self, neuron: Neuron) -> Result<(), GovernanceError> {
        let neuron_id = neuron
            .id
            .as_ref()
            .expect("Neuron must have a NeuronId")
            .clone();

        // New neurons are not allowed when the heap is too large.
        self.check_heap_can_grow()?;

        // New neurons are not allowed when the maximum configured is reached
        self.check_neuron_population_can_grow()?;

        if self.proto.neurons.contains_key(&neuron_id.to_string()) {
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Cannot add neuron. There is already a neuron with id: {neuron_id}"),
            ));
        }

        GovernanceProto::add_neuron_to_principal_to_neuron_ids_index(
            &mut self.principal_to_neuron_ids_index,
            &neuron,
        );

        add_neuron_to_function_followee_index(
            &mut self.function_followee_index,
            &self.proto.id_to_nervous_system_functions,
            &neuron,
        );

        add_neuron_to_follower_index(&mut self.topic_follower_index, &neuron);

        self.proto.neurons.insert(neuron_id.to_string(), neuron);

        Ok(())
