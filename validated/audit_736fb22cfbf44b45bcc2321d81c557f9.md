### Title
Unprivileged Injection of Arbitrary Interop Roots into Batch Public Input via Spoofed `InteropRootAdded` Event — (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` fires on **any** `InteropRootAdded` event emitted by **any** contract, with no check on the emitting address. An unprivileged user can deploy a contract that emits a crafted `InteropRootAdded` event, causing arbitrary `InteropRoot` entries to be injected into the in-memory `interop_root_storage`. These fake roots are then folded into the `interop_roots_rolling_hash` field of the batch public input, corrupting the cryptographic commitment that the settlement layer verifies.

---

### Finding Description

The `interop_root_reporter_event_hook` function in `system_hooks/src/event_hooks/interop_root_reporter.rs` is registered as a global event hook. It intercepts events matching the `InteropRootAdded` signature and calls `system.io.add_interop_root()` to record the root in the system's in-memory `InteropRootStorage`.

The hook signature is:

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<...>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
``` [1](#0-0) 

The hook receives no emitting-contract address parameter. Its only guard is a check on the event signature in `topics[0]`: [2](#0-1) 

After validating the ABI layout (offset == 32, len == 1, topics.len() == 3), it unconditionally calls:

```rust
system.io.add_interop_root(
    ExecutionEnvironmentType::NoEE,
    resources,
    InteropRoot { root, block_or_batch_number, chain_id },
)?;
``` [3](#0-2) 

`add_interop_root` in the IO subsystem performs no authenticity check — it simply pushes the root into the list: [4](#0-3) 

`push_root` in `InteropRootStorage` also has no ordering or deduplication check: [5](#0-4) 

At block finalization, `calculate_interop_roots_rolling_hash` iterates over **all** roots in `interop_root_storage` — including any injected by the attacker — and folds them into the `interop_roots_rolling_hash` field of `BatchOutput`: [6](#0-5) 

This rolling hash is committed into the batch public input: [7](#0-6) [8](#0-7) 

The legitimate path for adding interop roots is through operator-only service transactions (`ServiceTx`, type `0x7d`) calling `addInteropRootsInBatch` on `L2_INTEROP_ROOT_STORAGE_ADDRESS`: [9](#0-8) 

However, the event hook fires for **any** contract emitting the matching event signature — not just `L2_INTEROP_ROOT_STORAGE_ADDRESS`. A regular EVM transaction can deploy a contract that emits `InteropRootAdded(uint256,uint256,bytes32[])` with attacker-controlled `chain_id`, `block_or_batch_number`, and `root`, bypassing the operator-only service transaction gate entirely.

---

### Impact Explanation

An unprivileged user can corrupt the `interop_roots_rolling_hash` in the batch public input by injecting fake `InteropRoot` entries. This has two concrete consequences:

1. **Batch rejection / chain halt**: If the settlement layer verifies `interop_roots_rolling_hash` against an expected value derived from legitimate operator-submitted roots, the proof will be invalid and the batch will be rejected, halting the chain.
2. **Fake cross-chain message proofs**: If the settlement layer uses `interop_roots_rolling_hash` to verify inclusion of cross-chain messages, injected fake roots could be used to prove the existence of messages that were never sent on the source chain, enabling fraudulent cross-chain asset transfers.

---

### Likelihood Explanation

Any user who can submit a regular EVM transaction can trigger this. The attacker only needs to:
1. Deploy a contract (or call any existing contract) that emits an event with topic[0] = `0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0`, topic[1] = attacker-chosen `chain_id`, topic[2] = attacker-chosen `block_or_batch_number`, and 96 bytes of data encoding `(32, 1, fake_root)`.
2. The hook fires automatically with no further preconditions.

This is directly reachable from an unprivileged EVM transaction sender.

---

### Recommendation

Add an emitting-address check to `interop_root_reporter_event_hook`. The hook should only process events emitted by `L2_INTEROP_ROOT_STORAGE_ADDRESS`. Pass the emitting contract address as a parameter to the event hook and reject events from any other address:

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    emitter: &B160,           // add this parameter
    topics: &arrayvec::ArrayVec<...>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError> {
    if emitter != &L2_INTEROP_ROOT_STORAGE_ADDRESS {
        return Ok(());
    }
    // ... existing logic
}
```

This mirrors the fix applied to the ECO Protocol bug: binding the state-update path to a specific trusted source rather than accepting it from any caller.

---

### Proof of Concept

1. Attacker deploys a contract `FakeEmitter` with a function:
   ```solidity
   function emit_fake_root(uint256 chainId, uint256 blockNum, bytes32 root) external {
       emit InteropRootAdded(chainId, blockNum, [root]);
   }
   ```
   where `InteropRootAdded` has signature `0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0`.

2. Attacker submits a regular EVM transaction calling `FakeEmitter.emit_fake_root(attacker_chain_id, attacker_block_num, attacker_root)`.

3. The `interop_root_reporter_event_hook` fires (no address check), passes all format validations, and calls `system.io.add_interop_root(InteropRoot { root: attacker_root, block_or_batch_number: attacker_block_num, chain_id: attacker_chain_id })`.

4. At block finalization, `calculate_interop_roots_rolling_hash` folds the fake root into `interop_roots_rolling_hash`.

5. The batch public input now contains a `interop_roots_rolling_hash` that includes the attacker's fake root, diverging from the value the settlement layer expects based on legitimate operator-submitted roots. [10](#0-9) [8](#0-7)

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
