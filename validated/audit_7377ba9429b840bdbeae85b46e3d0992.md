Audit Report

## Title
Unbounded Prometheus Label Cardinality via Attacker-Controlled `subnet_id` in Subnet API Routes — (`rs/boundary_node/ic_boundary/src/metrics.rs`)

## Summary
An unprivileged attacker can send HTTP requests to `/api/v2/subnet/{subnet_id}/read_state` (and v3/v4 equivalents) with an unbounded number of syntactically valid but non-existent `subnet_id` Principal values. Each unique value is unconditionally registered as a distinct Prometheus label combination in the in-process registry before subnet existence is checked, causing unbounded memory growth and eventual OOM crash of the boundary node process.

## Finding Description
`validate_subnet_request` in `rs/boundary_node/ic_boundary/src/http/middleware/validate.rs` (L82–89) parses the URL path `{subnet_id}` as a `PrincipalId` using `Principal::from_text` — a format-only check — and inserts the resulting `SubnetId` into request extensions with no existence verification.

The `subnet_layers` stack in `rs/boundary_node/ic_boundary/src/core.rs` (L1033–1040) is ordered as:
1. `validate_subnet_request`
2. `common_layers` (which contains `middleware_metrics` at L1008)
3. `middleware_subnet_read_state_cache`
4. `middleware_subnet_lookup`

Because `common_layers` wraps `middleware_subnet_lookup`, the metrics middleware executes before subnet existence is verified. In `metrics_middleware` (`rs/boundary_node/ic_boundary/src/metrics.rs` L493–495), the attacker-supplied `SubnetId` is read from request extensions and converted to a string before `next.run(request)` is called. After `next.run()` returns — even when `subnet_lookup` fails with `SubnetNotFound` — the middleware unconditionally calls `with_label_values(labels)` on all four metric objects (L596–603), creating a new in-memory time series for each novel `subnet_id` string.

`remove_stale_metrics` (L81–135) only filters the `Vec<MetricFamily>` snapshot passed to the Prometheus text encoder at export time. It does not call `remove_label_values` on the underlying `HistogramVec`/`IntCounterVec` objects, so the registry's internal hash maps accumulate every unique label combination ever seen and are never pruned.

## Impact Explanation
This matches the allowed High impact: **Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS.** Each unique `subnet_id` label value creates 4 new metric series (counter, durationer, request_sizer, response_sizer), each with allocated bucket arrays. At scale, the boundary node process exhausts available memory and crashes, causing a denial of service for all traffic routed through that node. This is a code-level resource exhaustion bug, not a volumetric flood.

## Likelihood Explanation
The attack requires no authentication and no special privileges. Valid-format Principal IDs are trivially generated (29-byte values with correct CRC32 checksum). Per-IP rate limiting slows accumulation but does not prevent it — at 10 req/s, an attacker accumulates ~864,000 unique label combinations per day. The per-subnet rate limiter is ineffective because each request uses a different subnet ID. The attack is repeatable and low-cost.

## Recommendation
1. **Move `subnet_lookup` before `metrics_middleware`** in the layer stack, or record `subnet_id` only from the `Arc<Subnet>` response extension (which is only set on successful lookup), so unknown IDs never become label values.
2. **Alternatively**, in `metrics_middleware`, replace `subnet_id_str` with `SUBNET_ID_UNKNOWN` when the response contains `ErrorCause::SubnetNotFound`, preventing unrecognized IDs from ever entering the label set.
3. **Defense-in-depth**: Call `remove_label_values` on the underlying metric objects (not just filter the export snapshot) when stale subnet IDs are detected during the `remove_stale_metrics` pass.

## Proof of Concept
```python
import struct, base64, requests

def crc32(data):
    import binascii
    return binascii.crc32(data) & 0xFFFFFFFF

def make_principal(n: int) -> str:
    raw = b'\x04' + n.to_bytes(28, 'big')
    crc = struct.pack('>I', crc32(raw))
    encoded = base64.b32encode(crc + raw).decode().lower().rstrip('=')
    return '-'.join(encoded[i:i+5] for i in range(0, len(encoded), 5))

for i in range(1_000_000):
    sid = make_principal(i)
    requests.post(
        f"https://<boundary-node>/api/v2/subnet/{sid}/read_state",
        data=b'\xa0',
        headers={"Content-Type": "application/cbor"}
    )
# Monitor boundary node RSS: it grows proportionally to i.
# Scraping /metrics will show stale IDs filtered from export,
# but process memory continues to grow, confirming the registry leak.
```

A local integration test can confirm the bug without mainnet access: construct a test axum router with the same `subnet_layers` stack, send N requests with N distinct valid-format but non-existent subnet IDs, then assert that `registry.gather()` returns cardinality proportional to N and that process RSS grows accordingly.
