### Title
`receive_base_tokens` Ignores Relayer Fee in Backward-Compatible Message Format, Causing Theft of Relayer Fees — (`engine/src/engine.rs`)

### Summary

`Engine::receive_base_tokens` parses `FtTransferMessageData` from the bridge message, which still accepts the legacy format `"{relayer_id}:{fee_hex}{address_hex}"` for backward compatibility. The parsed `fee` field is silently discarded: the full `args.amount` is minted to the recipient, and the relayer receives nothing. Any bridge deposit using the old message format with a non-zero fee results in the relayer's fee being permanently lost to the recipient.

### Finding Description

`FtTransferMessageData::try_from` in `engine-types/src/parameters/connector.rs` still fully parses the legacy format and populates the `fee` field: [1](#0-0) 

The parsed struct is returned with `fee: Some(FtTransferFee { relayer, amount })`. However, `receive_base_tokens` in `engine/src/engine.rs` only reads `message_data.recipient` and unconditionally mints the full `args.amount` to the recipient, never touching `message_data.fee`: [2](#0-1) 

The engine already has the infrastructure to look up a relayer's EVM address (`get_relayer`) and to credit balances (`add_balance`/`set_balance`), but `receive_base_tokens` never calls either for the fee path. The `ft_on_transfer` entry point dispatches to `receive_base_tokens` whenever the caller is the registered ETH connector: [3](#0-2) 

### Impact Explanation

When a bridge deposit arrives with the legacy message format and a non-zero fee:

- The recipient's EVM balance is credited with the **full** `args.amount` instead of `args.amount − fee`.
- The relayer's EVM balance is never credited with `fee`.
- The fee amount is permanently locked inside the recipient's balance with no recovery path.

This constitutes **theft of unclaimed yield** (the relayer's earned fee) and an **accounting discrepancy** between the NEP-141 total supply (which includes the fee) and the intended distribution of ETH on Aurora.

The test file itself acknowledges the gap: `// NOTE: We don't test relayer fee`. [4](#0-3) 

### Likelihood Explanation

The legacy message format is still accepted by the parser (it is not rejected), so any relayer or bridge integration that has not yet migrated to the 40-character address-only format can trigger this path. The `test_relayer_charge_fee` test (which exercises the analogous ERC-20 fee path) is already marked `#[ignore]`, indicating the fee path is under-tested and the regression surface is real. [5](#0-4) 

### Recommendation

In `receive_base_tokens`, after parsing `message_data`, check for a non-zero fee and apply it:

1. Look up the relayer's EVM address via `self.get_relayer(fee.relayer.as_bytes())`.
2. Compute `net_amount = args.amount − fee.amount` (with underflow protection).
3. Credit `net_amount` to `message_data.recipient` and `fee.amount` to the relayer's EVM address.

Alternatively, if the fee mechanism is fully retired, reject the legacy format at parse time (return an error for messages longer than 40 hex characters) so callers receive an explicit failure rather than silent fee loss.

### Proof of Concept

1. Register a relayer NEAR account → EVM address mapping via `register_relayer`.
2. Call `ft_on_transfer` from the ETH connector account with:
   - `amount = 1000`
   - `msg = "relayer.near:" + hex(fee=100) + hex(recipient_address)`
3. Observe that `get_balance(recipient) == 1000` (should be `900`) and `get_balance(relayer_evm) == 0` (should be `100`).

The relayer's 100-unit fee is silently absorbed by the recipient, matching the root cause in the external report where fees accrued inside a vault were not subtracted from the supply limit, causing them to be inaccessible to the rightful claimant. [6](#0-5) [7](#0-6)

            .ok_or(errors::ParseOnTransferMessageError::TooManyParts)?;

        // Check relayer account id from 1-th data element
        let account_id = account
            .parse()
            .map_err(|_| errors::ParseOnTransferMessageError::InvalidAccount)?;

        // Decode message array from 2-th element of data array
        // Length = fee[32] + eth_address[20] bytes
        let mut data = [0; 52];
        hex::decode_to_slice(msg, &mut data).map_err(|e| match e {
            hex::FromHexError::InvalidHexCharacter { .. } | hex::FromHexError::OddLength => {
                errors::ParseOnTransferMessageError::InvalidHexData
            }
            hex::FromHexError::InvalidStringLength => {
                errors::ParseOnTransferMessageError::WrongMessageFormat
            }
        })?;

        // Parse the fee from the message slice.
        // The fee is expected to be represented as a 32-byte value in the message.
        // However, it will be parsed and converted to u128 for further processing.
        // This parsing logic is implemented to ensure compatibility

            .try_into()
            .map_err(|_| errors::ParseOnTransferMessageError::OverflowNumber)?;

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }
