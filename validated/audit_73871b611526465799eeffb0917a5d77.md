### Title
SNS Governance Voting Rewards Permanently Lost When All Voters Have Zero Voting Power During a Reward Round - (File: `rs/sns/governance/src/governance.rs`)

### Summary
In `rs/sns/governance/src/governance.rs`, the `distribute_rewards` function permanently discards the entire rewards purse for a round when all neurons that voted on `ReadyToSettle` proposals have zero voting power (`total_reward_shares == 0`). The rollover mechanism only triggers when `settled_proposals` is empty, but in this scenario proposals are settled with zero distribution, causing the rewards purse to be silently dropped with no recovery path.

### Finding Description

The `distribute_rewards` function in SNS governance computes a `rewards_purse_e8s` from the token supply and reward rate, then sums `total_reward_shares` from the `voting_power` field of each ballot on `ReadyToSettle` proposals. [1](#0-0) 

When `total_reward_shares == 0`, the maturity distribution loop is skipped entirely, leaving `distributed_e8s_equivalent = 0`. However, the function continues to settle all `considered_proposals` by writing `reward_event_end_timestamp_seconds` and clearing ballots: [2](#0-1) 

The final `RewardEvent` is then written with `settled_proposals: considered_proposals` (non-empty) and `distributed_e8s_equivalent: 0`: [3](#0-2) 

The rollover mechanism in `RewardEvent::e8s_equivalent_to_be_rolled_over` only returns the purse when `rewards_rolled_over()` is true, which requires `settled_proposals` to be empty: [4](#0-3) 

Because `settled_proposals` is non-empty (proposals were settled), `rewards_rolled_over()` returns `false`, `e8s_equivalent_to_be_rolled_over()` returns `0`, and the entire `rewards_purse_e8s` is permanently lost — never credited to any neuron and never rolled over to the next round.

The same structural bug exists in NNS governance's `calculate_voting_rewards` / `distribute_voting_rewards_to_neurons` path: [5](#0-4) [6](#0-5) 

### Impact Explanation

The entire `rewards_purse_e8s` for the affected round — computed as `supply * reward_rate * round_duration` plus any rolled-over amount from prior rounds — is permanently unrecoverable. It is never minted as maturity for any neuron and is not carried forward. For an SNS with a meaningful token supply and active reward rate, this can represent a significant loss of governance token maturity per affected round. The loss is irreversible because the proposals are marked `Settled` and cannot be re-processed.

### Likelihood Explanation

A neuron's `voting_power` is zero when `cached_neuron_stake_e8s <= neuron_fees_e8s`. The `neuron_fees_e8s` field accumulates from rejected governance proposals. The `voting_power` recorded in each ballot is snapshotted at proposal creation time. If all neurons that cast ballots on a `ReadyToSettle` proposal had zero voting power at proposal creation, `total_reward_shares` will be zero at distribution time.

This is demonstrated by the existing test `zero_total_reward_shares` in the integration tests: [7](#0-6) 

In a small SNS with few neurons, an unprivileged participant can engineer this condition by:
1. Creating a neuron and submitting proposals that get rejected, accumulating `neuron_fees_e8s` until `neuron_fees_e8s >= cached_neuron_stake_e8s` (voting power = 0).
2. Ensuring this depleted neuron is the only one that voted on a proposal that reaches `ReadyToSettle`.
3. When `distribute_rewards` runs, `total_reward_shares == 0` with non-empty `considered_proposals`, and the purse is dropped.

### Recommendation

When `total_reward_shares == 0` but `considered_proposals` is non-empty, the rewards purse should be rolled over to the next round rather than silently discarded. The simplest fix is to treat this case identically to the no-proposals case: do not settle the proposals (or, alternatively, set `distributed_e8s_equivalent = 0` and override `rewards_rolled_over()` to also return `true` when `distributed_e8s_equivalent == 0` regardless of `settled_proposals`). A more targeted fix is to add a third condition to `e8s_equivalent_to_be_rolled_over`: roll over when `distributed_e8s_equivalent == 0` and `total_available_e8s_equivalent > 0`, regardless of whether proposals were settled.

### Proof of Concept

**State setup:**
- SNS has one neuron `N` with `cached_neuron_stake_e8s = 1_000_000` and `neuron_fees_e8s = 1_000_000` (net stake = 0, voting power = 0).
- One proposal `P` is created while `N` has zero voting power; `N`'s ballot records `voting_power: 0`.
- Proposal `P` is decided and enters `ReadyToSettle`.

**Execution trace through `distribute_rewards`:**

1. `rewards_purse_e8s` is computed as, e.g., `100_000_000` e8s (non-zero). [8](#0-7) 

2. `considered_proposals = [P]` (non-empty). [9](#0-8) 

3. `total_reward_shares = 0` (only ballot has `voting_power: 0`). [10](#0-9) 

4. Maturity distribution is skipped; `distributed_e8s_equivalent = 0`. [11](#0-10) 

5. Proposal `P` is settled (ballots cleared, `reward_event_end_timestamp_seconds` set). [12](#0-11) 

6. New `RewardEvent` written: `settled_proposals = [P]`, `distributed_e8s_equivalent = 0`, `total_available_e8s_equivalent = 100_000_000`. [3](#0-2) 

7. Next round: `e8s_equivalent_to_be_rolled_over()` checks `rewards_rolled_over()` → `settled_proposals.is_empty()` → `false` → returns `0`. The `100_000_000` e8s purse is gone. [4](#0-3)

    /// Calculates the rounds_since_last_distribution in this event that should
    /// be "rolled over" into the next `RewardEvent`.
    ///
    /// Behavior:
    /// - If rewards were distributed for this event, then no rounds should be
    ///   rolled over, so this function returns 0.
    /// - Otherwise, this function returns
    ///   `rounds_since_last_distribution`.
    pub(crate) fn rounds_since_last_distribution_to_be_rolled_over(&self) -> u64 {

            0
        }
