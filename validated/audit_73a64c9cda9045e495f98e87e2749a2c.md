### Title
Missing Staleness Check on CMC ICP/XDR Rate Used for SNS Treasury Spending Limit Enforcement - (File: rs/sns/governance/token_valuation/src/lib.rs)

### Summary

The SNS governance canister fetches the ICP/XDR conversion rate from the Cycles Minting Canister (CMC) to enforce 7-day spending limits on `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals. The rate is consumed directly without any check on the `timestamp_seconds` field of the response. If the CMC's rate is stale — because the Exchange Rate Canister (XRC) has been unavailable and the CMC has not received fresh data — the SNS governance will silently enforce the spending limit against an outdated price, allowing proposals to transfer or mint more value than the intended limit permits.

### Finding Description

`CmcBased30DayMovingAverageXdrsPerIcpClient::get()` in `rs/sns/governance/token_valuation/src/lib.rs` calls `get_average_icp_xdr_conversion_rate` on the CMC and immediately extracts `response.data.xdr_permyriad_per_icp` without inspecting `response.data.timestamp_seconds`: [1](#0-0) 

The returned `IcpXdrConversionRateCertifiedResponse` carries a `timestamp_seconds` field that indicates when the rate was last updated, but the code ignores it entirely. The comment at line 451 only explains why the certificate need not be verified (canister-to-canister update call), not why the age of the data is irrelevant.

The CMC's average rate is populated by `update_exchange_rate`, which calls the XRC every five minutes: [2](#0-1) 

If the XRC is unavailable for an extended period, the CMC stops updating its `average_icp_xdr_conversion_rate`. The CMC's `get_average_icp_xdr_conversion_rate` query returns whatever is in state with no staleness guard: [3](#0-2) 

The stale rate then flows through `assess_treasury_balance` → `treasury_valuation_if_proposal_amount_is_small_enough_or_err` → the 7-day spending limit calculation for both `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals: [4](#0-3) [5](#0-4) 

### Impact Explanation

The 7-day spending limit is denominated in XDR value. If ICP's market price drops sharply while the CMC's rate is stale (still reflecting the old, higher price), the limit expressed in tokens is computed as:

```
max_tokens = XDR_limit / (stale_xdrs_per_icp * icps_per_token)
```

A stale high `xdrs_per_icp` produces a smaller `max_tokens` denominator, meaning the limit in token terms is *lower* than it should be — this is the conservative direction. However, the inverse is also true: if ICP price has risen since the last CMC update, the stale low rate produces a *higher* token limit than intended, allowing more tokens to be transferred or minted than the current market value of the treasury warrants. In either direction, the limit is computed against a price that does not reflect reality, defeating the purpose of the rate-based cap.

### Likelihood Explanation

The XRC is a system canister that aggregates prices from external HTTP sources via canister HTTP outcalls. It can be temporarily unavailable due to subnet issues, rate limiting, or canister upgrades. The CMC retries on failure but has no fallback that marks its stored rate as expired. A multi-day XRC outage — realistic given the dependency on external HTTP endpoints — would leave the CMC's average rate stale. Any SNS token holder can submit a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal during this window; no privileged access is required.

### Recommendation

In `CmcBased30DayMovingAverageXdrsPerIcpClient::get()`, after receiving the CMC response, compare `response.data.timestamp_seconds` against the current canister time. If the rate is older than an acceptable threshold (e.g., 48 hours for a 30-day moving average), return a `ValuationError::new_external` rather than proceeding with the stale value. This mirrors the pattern already used in the CMC itself, where `REFRESH_RATE_INTERVAL_SECONDS` gates how long the CMC will wait before re-querying the XRC: [2](#0-1) 

### Proof of Concept

1. The XRC canister becomes unavailable (e.g., subnet degradation or persistent HTTP outcall failures).
2. The CMC stops receiving fresh rates; `average_icp_xdr_conversion_rate.timestamp_seconds` freezes at the last successful update.
3. ICP market price rises 2× while the CMC rate is stale.
4. An SNS token holder submits a `MintSnsTokens` proposal for an amount that would exceed the intended XDR-denominated limit at the current price.
5. `CmcBased30DayMovingAverageXdrsPerIcpClient::get()` returns the stale (half the current) `xdrs_per_icp`.
6. `mint_sns_tokens_7_day_total_upper_bound_tokens` computes a token limit that is 2× larger than the current market value warrants.
7. The proposal passes validation and, if adopted, mints tokens beyond the intended economic guardrail. [6](#0-5) [7](#0-6)

#[query(hidden = true)]
fn get_average_icp_xdr_conversion_rate(_: ()) -> IcpXdrConversionRateCertifiedResponse {
    with_state(|state| {
        let witness_generator = convert_data_to_mixed_hash_tree(state);
        let average_icp_xdr_conversion_rate = state
            .average_icp_xdr_conversion_rate
            .as_ref()
            .expect("average_icp_xdr_conversion_rate is not set");

        let payload = convert_conversion_rate_to_payload(
            average_icp_xdr_conversion_rate,
            Label::from(LABEL_AVERAGE_ICP_XDR_CONVERSION_RATE),
            witness_generator,
        );

        IcpXdrConversionRateCertifiedResponse {
            data: average_icp_xdr_conversion_rate.clone(),
            hash_tree: payload,
            certificate: ic_cdk::api::data_certificate().unwrap_or_default(),
        }
    })
}
