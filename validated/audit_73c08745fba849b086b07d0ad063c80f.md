Audit Report

## Title
Permanent Fund Lock with No Reimbursement Path for Withdrawal Requests with Insufficient Fee - (`rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

When Ethereum gas fees rise above a user's withdrawal budget after tokens have already been burned, the ckETH minter indefinitely reschedules the withdrawal request to the back of the pending queue with no cancellation path, no timeout, and no reimbursement. For ckETH withdrawals, the full `withdrawal_amount` is burned upfront; for ckERC20 withdrawals, both the ckETH gas-fee tokens and the ckERC20 tokens are burned upfront. Neither is ever returned while the request cycles in the pending queue, resulting in permanent fund lock with no user recourse.

## Finding Description

**Root cause — no reimbursement on `InsufficientTransactionFee` in `create_transactions_batch`:**

In `rs/ethereum/cketh/minter/src/withdraw.rs`, `create_transactions_batch` iterates over pending requests and calls `create_transaction`. When the current gas fee exceeds the user's budget, `create_transaction` returns `CreateTransactionError::InsufficientTransactionFee`. The sole handler for this error is:

```rust
mutate_state(|s| s.eth_transactions.reschedule_withdrawal_request(request));
```

No reimbursement is triggered. The request is moved to the back of the FIFO queue and re-evaluated on every timer tick indefinitely.

**`reschedule_withdrawal_request` has no state change:**

`reschedule_withdrawal_request` simply removes and re-appends the request with no retry counter, no expiry timestamp, and no status transition:

```rust
self.remove_withdrawal_request(&request);
self.record_withdrawal_request(request);
```

**ckETH path — full amount burned before gas check:**

In `rs/ethereum/cketh/minter/src/main.rs`, `withdraw_eth` burns the entire `withdrawal_amount` on the ckETH ledger before any gas estimate is applied. The check `withdrawal_amount.checked_sub(max_transaction_fee)` in `create_transaction` (L1125) only runs later, during `create_transactions_batch`. If it returns `None`, the error arm reschedules — the burned ckETH is never returned.

**ckERC20 path — both ckETH gas fee and ckERC20 tokens burned before enqueue:**

In `withdraw_erc20`, the ckETH gas fee (`erc20_tx_fee`) is burned first, then the ckERC20 tokens are burned, and only then is the request enqueued. The `max_transaction_fee` field of `Erc20WithdrawalRequest` is fixed at the burned amount and never updated. If `actual_min_max_fee_per_gas > request_max_fee_per_gas` (L1160), `InsufficientTransactionFee` is returned and the request is rescheduled — neither the ckETH gas fee nor the ckERC20 tokens are reimbursed.

**Existing reimbursement paths do not cover this case:**

The only reimbursement paths that exist are:
1. `FailedErc20WithdrawalRequest` — fires only when the ckERC20 ledger burn fails at call time (not during queue processing).
2. `record_finalized_transaction` with `TransactionStatus::Failure` — fires only after a transaction is sent and mined with a failure receipt.

Neither path is reachable from the `reschedule_withdrawal_request` branch. The documentation explicitly confirms: "Overcharged transaction fees are not reimbursed."

**No cancellation endpoint exists** for users to recover funds from a stuck pending request.

## Impact Explanation

This is a **High** severity finding matching: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

- **ckETH:** A user who submits `withdraw_eth` with `amount = minimum_withdrawal_amount` (now `5_000_000_000_000_000` wei, ~$10) during a low-fee period has their ckETH burned immediately. If Ethereum fees spike above this amount, the request loops in the queue forever. The user's ckETH is permanently burned with no ETH delivered and no refund path.
- **ckERC20:** Both the ckETH gas-fee tokens and the ckERC20 tokens (which can be worth significantly more than $10) are burned upfront. If fees spike above `max_transaction_fee`, both are permanently locked with no reimbursement path.
- **Queue pollution / availability:** An attacker can submit many near-minimum withdrawal requests during low-fee periods. When fees spike, these requests cycle on every timer tick, consuming minter cycles and degrading throughput for all users. This constitutes a low-cost, sustained application-level DoS against the minter's withdrawal queue.

## Likelihood Explanation

Any unprivileged user can trigger this by calling `withdraw_eth` or `withdraw_erc20` with a near-minimum amount. No special access is required. The recent reduction of the minimum ckETH withdrawal amount from 0.03 ETH to 0.005 ETH (~$10) with only a ~10× safety margin makes this significantly more likely: a moderate gas spike (historically common during NFT mints or network congestion) can push the required fee above a near-minimum withdrawal amount. The ckERC20 case is additionally severe because the `max_transaction_fee` is fixed at call time, so even a temporary spike that later subsides may leave the request permanently unprocessable if fees do not return below the original estimate.

## Recommendation

1. **Add a reimbursement path in `create_transactions_batch`:** When `InsufficientTransactionFee` is returned, instead of (or after N retries of) rescheduling, trigger a `FailedErc20WithdrawalRequest` event (for ckERC20) or a `CkEth` reimbursement request (for ckETH), mirroring the existing call-time failure paths.
2. **Add a maximum retry count or expiry timestamp** to `WithdrawalRequest`. After N failed attempts or after a configurable deadline, automatically reimburse the burned tokens.
3. **Expose a user-callable cancellation endpoint** that allows the original requester to cancel a pending request and receive a reimbursement after a minimum waiting period.
4. **For ckERC20 specifically**, consider allowing `max_transaction_fee` to be updated (with user consent) rather than being permanently fixed at call-time value.

## Proof of Concept

1. Observe current Ethereum `base_fee_per_gas` is low (e.g., 1 gwei). Call `eip_1559_transaction_price` to confirm a low `max_fee_per_gas`.

* Reduce the minimum ETH withdrawal amount by a factor of 6, from 0.03 ETH (`30_000_000_000_000_000` wei) to 0.005 ETH (`5_000_000_000_000_000` wei) — approximately $10 at current prices. The reasoning is as follows:
    * The current minimum dates back to December 2023, when the ckETH minter was installed (see proposal [126171](https://dashboard.internetcomputer.org/proposal/126171)). At that time ETH traded in a similar USD range (around $2000), but Ethereum mainnet transaction fees were averaging $5–$10 per transaction ([source](https://bitinfocharts.com/comparison/ethereum-transactionfees.html#3y)).
    * Today, Ethereum mainnet fees are in the order of cents and rarely exceed $1.
    * As explained [here](https://github.com/dfinity/ic/blob/14382b5abb14b8e7de2bd4a3fb402ba069b82861/rs/ethereum/cketh/docs/cketh.adoc?plain=1#L208), an order-of-magnitude safety margin is preserved so the minter can always submit the transaction even when the Ethereum network is congested and one or more resubmissions are needed (each resubmission requires at least a 10% fee bump). With current Ethereum fees of ~$0.10–$1, a $10 minimum still preserves the ~10× safety margin even after several fee bumps.
