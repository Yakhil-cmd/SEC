Audit Report

## Title
Malicious Block Proposer Can Include Oversized `BatchPayload` That Passes Consensus Validation - (File: `rs/consensus/src/consensus/payload_builder.rs`)

## Summary
`PayloadBuilderImpl::validate_payload` enforces a hard rejection only when the combined payload size exceeds **twice** `max_block_payload_size` (8 MiB for app subnets). Payloads between 4 MiB and 8 MiB are silently accepted with only a metric increment. A malicious block proposer can craft a `BatchPayload` in this range by filling multiple independent sections to their individual limits simultaneously, forcing every validator to download, fully validate, and permanently store an oversized block.

## Finding Description
In `rs/consensus/src/consensus/payload_builder.rs` at lines 144–168, `validate_payload` accumulates the byte-size returned by each section validator and applies a two-tier check:

```rust
// lines 153-168
if accumulated_size > max_block_payload_size {
    error!(..., CRITICAL_ERROR_PAYLOAD_TOO_LARGE);
    self.metrics.critical_error_payload_too_large.inc();
}
if accumulated_size > max_block_payload_size * 2 {
    return Err(ValidationError::InvalidArtifact(
        InvalidPayloadReason::PayloadTooBig { ... },
    ));
}
```

The first branch is a soft guard — it logs and increments a metric but returns `Ok(())`. The hard rejection threshold is `2 × max_block_payload_size`. With `MAX_BLOCK_PAYLOAD_SIZE = 4 MiB` (`rs/limits/src/lib.rs` line 71), the hard limit is 8 MiB.

Each section validator enforces its own independent ceiling:
- Ingress: up to `MAX_INGRESS_BYTES_PER_BLOCK = 4 MiB` (`rs/limits/src/lib.rs` line 77)
- CanisterHttp: up to `MAX_CANISTER_HTTP_PAYLOAD_SIZE = 2 MiB` (`rs/types/types/src/batch/canister_http.rs` line 25)
- Additional sections (query_stats, chain_key, xnet, bitcoin) contribute further bytes

A malicious proposer bypasses `get_payload` entirely and constructs a `BatchPayload` struct directly, filling ingress to 4 MiB and canister-HTTP to 2 MiB simultaneously. The combined ~6 MiB payload passes every section validator individually and passes the top-level check because `6 MiB < 8 MiB`.

The test at lines 497–502 explicitly documents this behavior: a 6 MiB ingress payload combined with ~2 MiB from other sections (`expects_soft_error: true, expects_hard_error: false`) passes validation. Only at 7 MiB ingress (~9 MiB total) does the hard rejection trigger.

`validate_payload` is called inside `check_block_validity` at `rs/consensus/src/consensus/validator.rs` lines 1305–1320 before any notarization signature is cast, meaning every honest validator must process and store the oversized block.

## Impact Explanation
**Medium.** Every honest replica on the subnet must download the oversized block over P2P (up to 2× the normal maximum), run all section validators against it, and permanently store the finalized block on disk. Repeated across many rounds, this increases per-round CPU and I/O cost and shrinks available disk space on every node. This constitutes a limited subnet availability impact matching the Medium bounty tier: a Byzantine node below the fault threshold can degrade subnet performance without halting it outright.

## Likelihood Explanation
Block proposers are selected in rank order from subnet membership. Any single malicious node will be selected as rank-0 proposer roughly once per `n` rounds (where `n` is subnet size). No collusion is required; a single compromised or adversarial node suffices. The crafted payload requires no special cryptographic material beyond the node's consensus key. The attack is repeatable every time the malicious node is selected as proposer.

## Recommendation
Replace the 2× safety margin with a strict hard limit at `max_block_payload_size`:

```rust
if accumulated_size > max_block_payload_size {
    return Err(ValidationError::InvalidArtifact(
        InvalidPayloadReason::PayloadTooBig {
            expected: max_block_payload_size,
            received: accumulated_size,
        },
    ));
}
```

If a grace margin is needed for serialization overhead, it should be a small absolute value (e.g., a few KB), not a full 2× multiplier that allows an attacker to double the effective block size.

## Proof of Concept
The existing test at `rs/consensus/src/consensus/payload_builder.rs` lines 497–569 directly demonstrates the issue. The test case `#[case(6 * MB, true, false)]` with the note "payloads other than the ingress payload sum to a little below 2 MB" confirms that a ~8 MiB combined payload (`expects_hard_error: false`) passes validation. A malicious proposer constructs:

```rust
let oversized_payload = BatchPayload {
    ingress: /* 4 MiB of valid ingress messages */,
    canister_http: /* 2 MiB of valid canister-HTTP bytes */,
    query_stats: vec![0u8; 512 * 1024],
    chain_key: vec![0u8; 512 * 1024],
    xnet: XNetPayload::default(),
    self_validating: SelfValidatingPayload::default(),
};
// Combined ≈ 7.5 MiB > max_block_payload_size (4 MiB)
//                     < 2 × max_block_payload_size (8 MiB) → passes validate_payload
```

The proposer wraps this in a `BlockPayload::Data`, signs it with its consensus key, and broadcasts the `BlockProposal`. Every validator calls `validate_payload`, which returns `Ok(())`, and the block is notarized, finalized, and stored by all replicas.
