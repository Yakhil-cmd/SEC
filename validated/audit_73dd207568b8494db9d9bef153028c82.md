Audit Report

## Title
Timer-Driven Migration Pipeline Bypasses `migrations_disabled()` Guard — (`rs/migration_canister/src/processing.rs`, `rs/migration_canister/src/lib.rs`)

## Summary
The migration canister exposes `disable_api()` as an emergency pause callable by controllers and the NNS governance canister. However, `migrations_disabled()` is only checked in the `migrate_canister()` ingress handler. The nine timer-driven processing callbacks registered in `start_timers()` never consult this flag, so any request already in `REQUESTS` at the time of a pause will advance through every state — including irreversible steps such as `delete_canister` and `migrate_canisters` on the registry — until it reaches `RestoredControllers` and is finalized.

## Finding Description

**Single read-site of `DISABLED`:**
`migrations_disabled()` is defined in `canister_state.rs` at L56-58 and is called in exactly two places in `migration_canister.rs`: the guard at L63-65 inside `migrate_canister()` and the metrics encoder at L150. No call site exists in `processing.rs` or anywhere in the timer path.

**`start_timers()` registers nine unconditional callbacks (`lib.rs` L460-523):**
Each callback calls `process_all_by_predicate` with one of `process_accepted`, `process_controllers_changed`, `process_stopped`, `process_renamed`, `process_updated`, `process_routing_table`, `process_migrated_canister_deleted`, `process_all_succeeded`, or `process_all_failed`. None of these functions import or call `migrations_disabled()`.

**Irreversible operations in the pipeline (`processing.rs`):**
- `process_renamed` (L243): calls `migrate_canister()` on the registry canister, updating the routing table — this is a registry mutation.
- `process_routing_table` (L298-299): calls `delete_canister()` on the migrated canister's original subnet — permanent deletion.
- `process_accepted` (L84, L99): calls `set_exclusive_controller()` on both canisters, stripping original controllers.

**Exploit flow:**
1. A legitimate user (controller of both canisters, on the allowlist) calls `migrate_canister()` while the API is enabled. The request is inserted into `REQUESTS` as `RequestState::Accepted`.
2. Governance submits a `PauseCanisterMigrations` NNS proposal; `disable_api()` sets `DISABLED = true`.
3. New calls to `migrate_canister()` are rejected at L63-65. However, the timer fires every second and calls `process_accepted` → `process_controllers_changed` → `process_stopped` → `process_renamed` → `process_updated` → `process_routing_table` → `process_migrated_canister_deleted` → `process_all_succeeded`. None check `migrations_disabled()`.
4. The migration completes in full: the migrated canister is permanently deleted from its original subnet, the routing table is updated in the registry, and controllers are restored — all while `DISABLED = true`.

**Existing test gap:**
`validation_fails_disabled` in `tests/tests.rs` (L1243-1278) only verifies that `migrate_canister()` is rejected when disabled. It does not advance the timer or verify that in-flight requests are halted.

## Impact Explanation
This is a **High** severity finding. The migration canister is NNS-governed infrastructure performing irreversible cross-subnet operations. The `disable_api()` / `PauseCanisterMigrations` mechanism is the sole emergency brake for the migration pipeline. Its failure to halt in-flight requests means:
- `delete_canister` on the migrated canister's original subnet proceeds despite the pause, permanently destroying the canister.
- The registry routing table is mutated via `migrate_canisters` despite the pause, with no rollback path.
- Controllers of both canisters are held exclusively by the migration canister during the window, with no guarantee of recovery if the process is interrupted mid-flight after the pause.

This matches the allowed impact: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm"* (High, $2,000–$10,000).

## Likelihood Explanation
The precondition for the user (being a controller of both canisters and on the allowlist) is the **normal operating condition** for any legitimate migration user — no privilege escalation is required. The race window is not tight: any request accepted before the pause, regardless of how far in advance, will run to completion. The `PauseCanisterMigrations` NNS function exists precisely for emergency use (e.g., a bug discovered in `process_renamed` or `process_routing_table`), making this scenario the primary motivation for the pause feature. The finding is deterministically reproducible in a PocketIC environment.

## Recommendation
Add a `migrations_disabled()` guard at the earliest processing stage. At minimum, `process_accepted` should return `ProcessingResult::NoProgress` when the flag is set, stalling the request without advancing it:

```rust
pub async fn process_accepted(
    request: RequestState,
) -> ProcessingResult<RequestState, RequestState> {
+   if migrations_disabled() {
+       return ProcessingResult::NoProgress;
+   }
    let RequestState::Accepted { request } = request else { ... };
    ...
}
```

Blocking at `process_accepted` prevents all irreversible actions, since controller stripping, routing table mutation, and canister deletion all occur in later states. Depending on the desired semantics of the pause, guards should also be added to `process_controllers_changed` and subsequent stages to halt requests that have already passed `Accepted`.
