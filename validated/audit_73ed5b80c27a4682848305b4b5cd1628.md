Audit Report

## Title
ckETH/ckERC20 Withdrawal Queue Saturation Causes `ic_cdk::trap` DoS for All Users - (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`, `rs/ethereum/cketh/minter/src/main.rs`)

## Summary
The ckETH minter enforces a hard cap of `MAX_PENDING = 100` on pending withdrawal requests. When this cap is reached, both `withdraw_eth` and `withdraw_erc20` call `ic_cdk::trap` — a hard canister panic — instead of returning a graceful typed error. An unprivileged attacker controlling 100 principals, each holding the minimum ckETH withdrawal amount, can saturate the shared queue and cause every subsequent withdrawal call from any user to receive a system-level reject until the minter's timer drains the queue.

## Finding Description
In `rs/ethereum/cketh/minter/src/guard/mod.rs`, `MAX_PENDING` is set to 100. [1](#0-0) 

`Guard::new` checks `pending_requests_count(s) >= MAX_PENDING` and returns `Err(GuardError::TooManyPendingRequests)` when the cap is reached. [2](#0-1) 

`pending_requests_count` for `PendingWithdrawalRequests` delegates to `withdrawal_requests_len()`, which counts all entries in the shared `pending_withdrawal_requests` queue — covering both ckETH and ckERC20 requests. [3](#0-2) 

The `Guard` is dropped after each successful call, removing the principal from `pending_withdrawal_principals`, but the `WithdrawalRequest` remains in `pending_withdrawal_requests` until the timer processes it. This means 100 sequential calls from 100 distinct principals permanently fill the queue.

In `withdraw_eth`, the guard is acquired at line 274 — **before** the `burn_from` call at line 301. When the guard fails, `ic_cdk::trap` is called unconditionally, producing a system-level reject with no typed `WithdrawalError`. [4](#0-3) 

The same pattern applies to `withdraw_erc20`. [5](#0-4) 

Because the queue is shared, filling it with ckETH withdrawals also blocks all ckERC20 withdrawals and vice versa. The `AlreadyProcessing` guard (which tracks in-flight concurrent calls per principal) does not prevent this attack since the attacker uses 100 distinct principals submitting sequentially.

## Impact Explanation
All users are unable to call `withdraw_eth` or `withdraw_erc20` while the queue is saturated. The `ic_cdk::trap` produces a system-level reject rather than a typed `WithdrawalError`, so callers receive no actionable feedback and cannot distinguish this from a canister bug. This constitutes an application/platform-level DoS of the ckETH and ckERC20 withdrawal functionality — a significant ck-token security impact with concrete user harm. No funds are permanently lost (the guard fires before the burn), but withdrawal liveness is broken for all users for the duration of the attack. This matches the **High** impact tier.

## Likelihood Explanation
The attacker requires 100 principals each holding the minimum ckETH withdrawal amount. The ckETH burned by each call is eventually sent to the attacker's Ethereum address, so the net cost is only Ethereum gas fees for 100 outbound transactions. No privileged access, governance majority, or threshold corruption is required. The attacker can monitor queue depth via the public `get_events` endpoint and re-submit as the timer drains requests, sustaining the DoS indefinitely at low recurring cost.

## Recommendation
1. **Replace `ic_cdk::trap` with a graceful error return** for `TooManyPendingRequests` in both `withdraw_eth` and `withdraw_erc20`. Return `WithdrawalError::TemporarilyUnavailable` (or `WithdrawErc20Error::TemporarilyUnavailable`) so callers can retry with actionable feedback.
2. **Increase `MAX_PENDING`** to a value commensurate with the minter's processing throughput (e.g., aligned with `MAX_NUM_PENDING_TRANSACTION_NONCES = 1000` already used in `withdrawal_requests_batch`).
3. **Add per-principal pending request accounting** to prevent a single attacker from monopolizing the shared queue across multiple principals, consistent with the ckBTC minter's approach.

## Proof of Concept
```
1. Attacker creates principals P1..P100, each approved for min_withdrawal_amount ckETH.
2. Attacker calls withdraw_eth(min_amount, eth_addr) from each Pi sequentially.
   → Each call: guard acquired (principal added to pending_withdrawal_principals),
     ckETH burned, WithdrawalRequest pushed to pending_withdrawal_requests,
     guard dropped (principal removed from pending_withdrawal_principals).
   → After 100 calls: pending_withdrawal_requests.len() == MAX_PENDING == 100.
3. Honest user calls withdraw_eth(any_amount, any_addr):
   → Guard::new: pending_requests_count(s) >= 100 → Err(TooManyPendingRequests)
   → unwrap_or_else(|e| ic_cdk::trap(...)) → canister traps → system reject to user.
4. Same trap occurs for withdraw_erc20 (shared queue, same guard).
5. Minter timer fires, processes batch, queue drains below 100.
6. Attacker repeats step 2 to re-saturate.

Verification: unit test in guard/tests.rs — populate 100 withdrawal requests in
eth_transactions, then call retrieve_withdraw_guard(new_principal) and assert
Err(GuardError::TooManyPendingRequests) is returned.
```
