### Title
Silo Whitelist Bypass via `call` and `deploy_code` Methods Allows Unauthorized EVM Execution — (File: `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

The `call` and `deploy_code` NEAR contract entry points in Aurora Engine execute EVM transactions without invoking the silo whitelist access check (`assert_access`) that the `submit` path enforces. In Silo mode, any unprivileged NEAR account can call these methods directly to execute arbitrary EVM transactions against any EVM contract, bypassing the Account and Address whitelist restrictions entirely.

---

### Finding Description

**Root cause — missing `assert_access` call in `call` and `deploy_code`:**

The `submit` path calls `engine::submit()`, which at line 1052 of `engine/src/engine.rs` invokes `assert_access` before any EVM execution: [1](#0-0) 

`assert_access` enforces the Silo whitelist: [2](#0-1) 

It calls `silo::is_allow_submit` (for calls) or `silo::is_allow_deploy` (for deployments) against the NEAR predecessor account and the EVM target address. If either check fails, the transaction is rejected with `EngineErrorKind::NotAllowed`.

By contrast, the `call` method in `evm_transactions.rs` only checks `require_running` and then calls `engine.call_with_args()` directly — `assert_access` is never invoked: [3](#0-2) 

Likewise, `deploy_code` only checks `require_running` and calls `engine.deploy_code_with_input()` directly, bypassing `is_allow_deploy`: [4](#0-3) 

**Attacker-controlled entry path:**

Both `call` and `deploy_code` are public NEAR contract methods. In NEAR, any account can invoke any public method on a contract. Neither method has any caller-identity restriction beyond `require_running`. The silo whitelist tests in the test suite exercise only the `submit_transaction` path: [5](#0-4) 

There is no analogous test confirming that `call` is blocked by the whitelist, because it is not.

---

### Impact Explanation

In Silo mode, the Account and Address whitelists are the primary access-control boundary. They determine which NEAR accounts may submit EVM transactions and which EVM contract addresses may be targeted. Bypassing them via `call` allows a non-whitelisted NEAR account to:

1. Call any EVM contract address, including token contracts (ERC-20 mirrors of bridged assets) that the Silo operator intended to restrict.
2. Deploy arbitrary EVM contracts without being on the deploy whitelist.

Because the EVM sender is derived from the attacker's NEAR account (`predecessor_address(&predecessor_account_id)`), the attacker executes as their own EVM address. This is sufficient to interact with any EVM contract — including transferring tokens held at the attacker's own EVM address to arbitrary recipients, or calling contracts whose logic does not require the caller to be a specific address. In a Silo deployment where the whitelist is the sole barrier preventing unauthorized token movement, this constitutes a direct path to fund theft from the EVM state.

**Impact category:** High — whitelist/pause bypass enabling unauthorized EVM execution; fund theft from EVM contracts reachable by the attacker's EVM address.

---

### Likelihood Explanation

High. The `call` method is unconditionally public. No privilege, key, or deposit is required. Any NEAR account — including one freshly created with no history — can invoke it. The bypass requires only knowledge of the `CallArgs` encoding format, which is public.

---

### Recommendation

Add an `assert_access`-equivalent check to both `call` and `deploy_code` before delegating to the engine. For `call`, construct a synthetic `NormalizedEthTransaction`-compatible check using the `CallArgs` target address and the NEAR predecessor account, mirroring the logic already present in `engine::submit`. For `deploy_code`, apply the `is_allow_deploy` silo check. Alternatively, gate both methods behind the same silo whitelist lookup used in `assert_access`.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode; enable Account and Address whitelists.
2. Create a NEAR account `attacker.near` that is **not** on either whitelist.
3. From `attacker.near`, call `submit` with a signed EVM transaction targeting any EVM contract → rejected with `EngineErrorKind::NotAllowed` (whitelist enforced).
4. From `attacker.near`, call `call` with `CallArgs` targeting the same EVM contract → **succeeds**; `assert_access` is never reached; the EVM transaction executes.
5. The attacker has executed an EVM transaction against a contract the Silo operator intended to restrict, with no whitelist validation applied.

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}
