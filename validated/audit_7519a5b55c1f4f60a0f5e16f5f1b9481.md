### Title
Non-Whitelisted EVM Addresses Can Receive Base ETH Tokens in Silo Mode, Bypassing the `Address` Whitelist - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, the `WhitelistKind::Address` whitelist is intended to control which EVM addresses can both submit transactions **and receive tokens**. While `is_allow_submit` enforces this for transaction submission and `is_allow_receive_erc20_tokens` exists for ERC-20 receipt, the `receive_base_tokens()` function — which mints ETH to a recipient during bridge deposits — contains **no whitelist check at all**. Any unprivileged user can deposit ETH to any non-whitelisted EVM address in a Silo instance, bypassing the access control mechanism entirely.

---

### Finding Description

The Silo mode `Address` whitelist (`WhitelistKind::Address`) is documented as controlling "which EVM addresses can submit transactions or receive tokens." [1](#0-0) 

The `is_allow_receive_erc20_tokens` function enforces this for ERC-20 token receipt in `receive_erc20_tokens`: [2](#0-1) 

However, `receive_base_tokens` — the function that mints ETH to a recipient when the ETH connector calls `ft_on_transfer` — performs **no whitelist check**: [3](#0-2) 

It directly calls `set_balance` on the recipient address without consulting `is_allow_receive_erc20_tokens` or any equivalent check. The dispatch between the two paths happens in `ft_on_transfer`: [4](#0-3) 

When `predecessor_account_id` matches the registered ETH connector account, `receive_base_tokens` is called — and the whitelist is never consulted.

A secondary bypass also exists in `receive_erc20_tokens`: the whitelist check is gated behind a Rust let-chain that only evaluates `is_allow_receive_erc20_tokens` when `get_erc20_fallback_address` returns `Some`. If no fallback address is configured, the entire check is skipped and any address receives ERC-20 tokens regardless of whitelist status. [2](#0-1) [5](#0-4) 

---

### Impact Explanation

In a Silo deployment with `WhitelistKind::Address` enabled, a non-whitelisted EVM address that receives ETH via `receive_base_tokens` cannot spend it: `is_allow_submit` blocks all transaction submission from non-whitelisted addresses. [6](#0-5) 

The deposited ETH is therefore frozen in the non-whitelisted address until the Silo operator intervenes (adds the address to the whitelist or disables the whitelist). This constitutes **temporary freezing of funds** (High impact). If the operator never acts, the freeze is permanent (Critical).

---

### Likelihood Explanation

Any user who can interact with the ETH bridge can trigger this. No special privileges are required — the attacker simply deposits ETH to Aurora specifying a non-whitelisted EVM address as recipient. This is a standard bridge operation available to any token holder. The Silo mode with `Address` whitelist enabled is a production configuration explicitly supported by the engine. [7](#0-6) 

---

### Recommendation

Add a whitelist check inside `receive_base_tokens` mirroring the pattern used in `receive_erc20_tokens`. When the `Address` whitelist is enabled and the recipient is not whitelisted, either reject the deposit (returning the amount to the sender) or redirect to a configured fallback address. Additionally, the `receive_erc20_tokens` bypass should be fixed so that the whitelist check is enforced independently of whether a fallback address is configured — i.e., if the whitelist is enabled and no fallback is set, reject deposits to non-whitelisted addresses rather than silently allowing them.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode.
2. Enable the `Address` whitelist via `set_whitelist_status` with `WhitelistKind::Address, active: true`.
3. Do **not** add EVM address `0xDEAD...` to the whitelist.
4. From any NEAR account, deposit ETH to Aurora via the bridge, specifying `0xDEAD...` as the recipient in the `msg` field of `ft_on_transfer`.
5. The ETH connector calls `ft_on_transfer` on the Aurora engine contract with itself as `predecessor_account_id`.
6. `ft_on_transfer` routes to `receive_base_tokens` (line 82 of `connector.rs`).
7. `receive_base_tokens` calls `set_balance` on `0xDEAD...` with no whitelist check — ETH is credited.
8. Attempt to submit any EVM transaction from `0xDEAD...`: it is rejected with `NotAllowed` because `is_allow_submit` checks the `Address` whitelist.
9. The ETH deposited to `0xDEAD...` is frozen. [3](#0-2) [4](#0-3) [6](#0-5)

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }
