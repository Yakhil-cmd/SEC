Audit Report

## Title
SNS Governance `distribute_rewards` Early Return on `u64` Overflow Leaves `latest_reward_event` Stale, Permanently Blocking Reward Distribution - (File: rs/sns/governance/src/governance.rs)

## Summary
In `distribute_rewards`, when `rewards_purse_e8s` exceeds `u64::MAX`, the function returns early at line 5888 without updating `self.proto.latest_reward_event`. Because `should_distribute_rewards` and `new_rounds_count` both derive their baseline from `latest_reward_event.end_timestamp_seconds`, the stale timestamp causes `new_rounds_count` to grow on every subsequent heartbeat, making the overflow self-perpetuating. All proposals in `ReadyToSettle` state are never settled, neuron maturity is never increased, and the accumulated rewards purse is permanently discarded.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 5878‚Äď5890, after computing `rewards_purse_e8s` as a `Decimal` accumulation of rolled-over rewards plus per-round interest, the code attempts a narrowing cast:

```rust
let total_available_e8s_equivalent = Some(match u64::try_from(rewards_purse_e8s) {
    Ok(ok) => ok,
    Err(err) => {
        log!(ERROR, "Looks like the rewards purse ({}) overflowed u64: {}. \
             Therefore, we stop the current attempt to distribute voting rewards.",
             rewards_purse_e8s, err);
        return;   // ‚Üź early return, no state update
    }
});
```

The `return` exits `distribute_rewards` before the block at lines 6083‚Äď6092 that writes `self.proto.latest_reward_event`. Consequently:

1. `latest_reward_event.end_timestamp_seconds` retains its old value.
2. `should_distribute_rewards` (lines 5735‚Äď5753) reads `end_timestamp_seconds` from the stale event; since real time keeps advancing, it returns `true` on every heartbeat.
3. `new_rounds_count` (lines 5812‚Äď5814) is computed as `(now - reward_start_timestamp_seconds) / round_duration_seconds`; with a frozen `reward_start_timestamp_seconds`, this count grows monotonically.
4. On the next call, `rewards_purse_e8s` is recomputed from the same stale rollover value plus a larger `new_rounds_count`, producing an even larger overflow.
5. `e8s_equivalent_to_be_rolled_over()` (types.rs lines 2054‚Äď2060) reads `total_available_e8s_equivalent` from the last successfully committed event ‚ÄĒ this value is bounded by `u64::MAX` and does not itself grow, but the `supply * rate * new_rounds_count` term grows without bound.

The loop is self-perpetuating: every heartbeat re-enters the overflow branch, and `latest_reward_event` is never advanced. All `considered_proposals` (line 5822‚Äď5823) remain in `ReadyToSettle` indefinitely; their ballots are never cleared (line 6080) and neuron maturity is never credited.

The maximum allowed `initial_reward_rate_basis_points` is `10_000` (100 %/year, `reward.rs` line 145). At that ceiling with a token supply of 10¬Ļ¬≤ tokens (10¬≤‚Āį e8s) and a 1-day round, each round adds ‚Čą 2.74 √ó 10¬Ļ‚Ā∑ e8s to the purse. Overflow occurs after ‚Čą 67 missed rounds (‚Čą 67 days of canister unavailability). Once the overflow state is entered, no code path resets it without a direct state migration.

## Impact Explanation
This matches **High ($2,000‚Äď$10,000): Significant SNS security impact with concrete user or protocol harm**. Specifically:

- **Permanent loss of voting rewards**: The entire rewards purse ‚ÄĒ potentially representing a significant fraction of the token supply ‚ÄĒ is silently discarded on every heartbeat. SNS neuron holders who voted on proposals receive zero maturity, violating the economic invariant of the SNS reward model.
- **Proposals permanently stuck in `ReadyToSettle`**: Ballots are never cleared, consuming unbounded memory. Once `max_number_of_proposals_with_ballots` is reached, new proposals are blocked, halting SNS governance entirely.
- The impact is irreversible without an out-of-band canister upgrade or state migration; no in-protocol recovery path exists.

## Likelihood Explanation
Triggering the overflow requires a large token supply (‚Č• ~10¬Ļ¬≤ tokens at max reward rate) combined with an extended period of missed reward rounds (‚Č• ~67 days at max rate and 10¬Ļ¬≤ token supply). This is reachable without any privileged action: a canister upgrade failure, a prolonged IC subnet issue, or a governance-approved `ManageNervousSystemParameters` proposal reducing `round_duration_seconds` to a small value (causing `new_rounds_count` to spike on the next heartbeat) can all trigger the condition. The `ManageNervousSystemParameters` path requires only a passing SNS governance vote ‚ÄĒ no special system privileges. Once triggered, the state is self-perpetuating and requires no further attacker action.

## Recommendation
Replace the early return with a saturating cast so that time always advances and reward distribution is never permanently blocked:

```rust
let total_available_e8s_equivalent = Some(
    u64::try_from(rewards_purse_e8s).unwrap_or(u64::MAX)
);
```

Alternatively, if the overflow must abort distribution, still advance `latest_reward_event.end_timestamp_seconds` to `reward_event_end_timestamp_seconds` so that `new_rounds_count` resets to zero on the next call and the overflow does not self-perpetuate. A third complementary guard is to validate in `ManageNervousSystemParameters` that the new reward rate combined with the current rolled-over purse and token supply cannot immediately overflow.

## Proof of Concept
1. Deploy an SNS with `initial_reward_rate_basis_points = 10_000` and a token supply of 10¬Ļ¬≤ tokens (10¬≤‚Āį e8s).
2. Pause the governance canister's heartbeat (or submit a `ManageNervousSystemParameters` proposal setting `round_duration_seconds` to `1`) for 67+ days, causing `new_rounds_count` to reach ‚Č• 67 on the next `distribute_rewards` call.
3. Resume the heartbeat. `distribute_rewards` computes `rewards_purse_e8s ‚Čą 10¬≤‚Āį √ó (10000/10000/year) √ó (67 days) ‚Čą 1.84 √ó 10¬Ļ‚ĀĻ e8s`, which exceeds `u64::MAX`. The `u64::try_from` fails; the function returns at line 5888 without updating `latest_reward_event`.
4. On the next heartbeat, `should_distribute_rewards` returns `true` again (stale `end_timestamp_seconds`). `new_rounds_count` is now larger. The overflow recurs.
5. Observe via canister state inspection that `latest_reward_event.end_timestamp_seconds` never advances, all `ReadyToSettle` proposals remain unsettled, and no neuron maturity is ever increased.
6. A deterministic integration test using PocketIC can reproduce this by setting the mock clock forward 67+ days after SNS initialization with the above parameters and asserting that `latest_reward_event` is unchanged after repeated `run_periodic_tasks` calls.
