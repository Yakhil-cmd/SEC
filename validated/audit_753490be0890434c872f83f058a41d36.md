### Title
`MintSnsTokens` 7-Day Spending Limit Unenforced Due to `Decimal::MAX` Upper Bound — (`File: rs/sns/governance/src/proposal.rs`)

---

### Summary

The `MintSnsTokens` implementation of `TokenProposalAction::recent_amount_total_upper_bound_tokens` returns `Decimal::MAX` instead of a real treasury-based limit. Because the shared validation function `treasury_valuation_if_proposal_amount_is_small_enough_or_err` computes `allowance_remainder = max_tokens - spent_tokens`, and `max_tokens` is `Decimal::MAX`, the already-minted `spent_tokens` counter is effectively never subtracted from the available allowance. The 7-day minting cap is therefore completely unenforced for `MintSnsTokens` proposals.

---

### Finding Description

`rs/sns/governance/src/proposal.rs` defines a shared trait `TokenProposalAction` used by both `TransferSnsTreasuryFunds` and `MintSnsTokens` to enforce a 7-day treasury spending limit. [1](#0-0) 

The shared validation function computes:

```
allowance_remainder_tokens = max_tokens - spent_tokens
```

and rejects the proposal if `proposal_amount_tokens > allowance_remainder_tokens`.

For `TransferSnsTreasuryFunds`, `recent_amount_total_upper_bound_tokens` correctly returns a treasury-valuation-based limit: [2](#0-1) 

For `MintSnsTokens`, however, the implementation returns `Decimal::MAX`: [3](#0-2) 

This means `allowance_remainder_tokens ≈ Decimal::MAX` regardless of how much has already been minted. The `spent_tokens` value — correctly computed by `total_minting_amount_tokens` — is subtracted from `Decimal::MAX` and the result is still effectively unbounded. The guard `proposal_amount_tokens > allowance_remainder_tokens` can never trigger for any realistic mint amount.

Additionally, unlike `TransferSnsTreasuryFunds`, there is no execution-time re-check analogous to `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` for `MintSnsTokens`, so there is no second layer of protection. [4](#0-3) 

---

### Impact Explanation

Any SNS neuron holder with sufficient voting power can submit and execute `MintSnsTokens` proposals of arbitrary size, bypassing the intended 7-day treasury minting cap entirely. This allows unlimited inflation of the SNS token supply, diluting all existing token holders and potentially draining treasury value. The `spent_tokens` accounting variable is computed correctly but is never meaningfully compared against a real limit — a direct analog to the original `tokensRepurchased` not being subtracted from `getAmountRepurchasable`.

---

### Likelihood Explanation

Exploiting this requires controlling enough SNS voting power to pass a `MintSnsTokens` proposal. In SNS DAOs with concentrated neuron ownership (e.g., early-stage projects, developer-controlled neurons), this threshold can be low. The entry path is a standard governance ingress call (`manage_neuron` → `MakeProposal`) available to any neuron holder — no privileged access is required beyond voting power.

---

### Recommendation

Replace the `Decimal::MAX` stub with the real treasury-valuation-based limit by uncommenting the intended implementation (tracked under `TODO(NNS1-2982)`): [5](#0-4) 

Until that is done, the `spent_tokens` variable computed by `total_minting_amount_tokens` is silently ignored, leaving the minting rate entirely unconstrained. Additionally, add an execution-time re-check for `MintSnsTokens` mirroring `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`.

---

### Proof of Concept

1. An SNS neuron holder submits a `MintSnsTokens` proposal for `u64::MAX` e8s.
2. `treasury_valuation_if_proposal_amount_is_small_enough_or_err` is called:
   - `spent_tokens` = sum of already-minted amounts in past 7 days (e.g., 0)
   - `max_tokens` = `Decimal::MAX` (from `recent_amount_total_upper_bound_tokens`)
   - `allowance_remainder_tokens` = `Decimal::MAX - 0` = `Decimal::MAX`
   - Check: `u64::MAX / E8 > Decimal::MAX`? → **false** → proposal passes validation
3. Proposal is adopted and executed; `u64::MAX` tokens are minted.
4. Step 1–3 can be repeated immediately with no cooldown, since `spent_tokens` from step 3 is still subtracted from `Decimal::MAX` and the check still passes. [6](#0-5) [7](#0-6)

async fn treasury_valuation_if_proposal_amount_is_small_enough_or_err<MyTokenProposalAction>(
    env: &dyn Environment,
    sns_ledger_canister_id: CanisterId,
    swap_canister_id: CanisterId,
    proposals: impl Iterator<Item = &ProposalData>,
    action: &MyTokenProposalAction,
) -> Result<Valuation, String>
where
    MyTokenProposalAction: TokenProposalAction,
{
    let spent_tokens = action.recent_amount_total_tokens(proposals, env.now())?;

    // Get valuation of the tokens in the treasury.
    let token = action.token()?;
    let valuation = assess_treasury_balance(
        token,
        env.canister_id(),
        sns_ledger_canister_id,
        swap_canister_id,
    )
    .await?;

    // From valuation, determine limit on the total from the past 7 days.
    let max_tokens = MyTokenProposalAction::recent_amount_total_upper_bound_tokens(&valuation)
        // Err is most likely a bug.
        .map_err(|treasury_limit_error| {
            format!("Unable to validate amount: {treasury_limit_error:?}",)
        })?;

    // Finally, inspect the proposal's amount: it must not exceed max - spent (remainder). Or if
    // you prefer, equivalently, amount + spent must be <= max.
    let allowance_remainder_tokens = max_tokens.checked_sub(spent_tokens).ok_or_else(|| {
        format!("Arithmetic error while performing {max_tokens} - {spent_tokens}",)
    })?;
    let proposal_amount_tokens = action.proposal_amount_tokens()?;
    if proposal_amount_tokens > allowance_remainder_tokens {
        // Although it might not be obvious to the user, their proposal is invalid, and we
        // consider it to be "their fault".
        return Err(format!(
            "Amount is too large. Within the past 7 days, a total of {spent_tokens} tokens has already \
             been executed in like proposals. Whereas, at most {max_tokens} is allowed. An additional \
             {proposal_amount_tokens} tokens from this proposal would cause that upper bound to be exceeded. \
             Maybe, try again in a few days?"
        ));
    }

    Ok(valuation)

pub(crate) fn transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err<'a>(
    transfer: &TransferSnsTreasuryFunds,
    valuation: Valuation,
    proposals: impl Iterator<Item = &'a ProposalData>,
    now_timestamp_seconds: u64,
) -> Result<(), GovernanceError> {
    let allowance_tokens = transfer_sns_treasury_funds_7_day_total_upper_bound_tokens(valuation)
        .map_err(|err| {
            // This should not be possible, because valuation was already used the same way during
            // proposal submission/creation/validation.
            GovernanceError::new_with_message(
                ErrorType::InconsistentInternalData,
                format!(
                    "Unable to determined upper bound on the amount of \
                     TransferSnsTreasuryFunds proposals: {err:?}\nvaluation:{valuation:?}",
                ),
            )
        })?;

    // The total calculated here _could_ be different from what was calculated at proposal
    // submission/creation time. A difference would result from the execution of (another)
    // TransferSnsTreasuryFunds proposal between now and then.
    let spent_tokens = total_treasury_transfer_amount_tokens(
        proposals,
        transfer.from_treasury(),
        now_timestamp_seconds - 7 * ONE_DAY_SECONDS,
    )
    .map_err(|message| {
        GovernanceError::new_with_message(ErrorType::InconsistentInternalData, message)
    })?;

    let remainder_tokens = allowance_tokens - spent_tokens;
    let transfer_amount_tokens = denominations_to_tokens(transfer.amount_e8s, E8)
        // This Err cannot be provoked, because we are dividing a u64 (amount_e8s) by a positive
        // integer (E8).
        .ok_or_else(|| {
            GovernanceError::new_with_message(
                ErrorType::UnreachableCode,
                format!(
                    "Unable to convert proposals amount {} e8s to tokens.",
                    transfer.amount_e8s,
                ),
            )
        })?;
    if transfer_amount_tokens > remainder_tokens {
        return Err(GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!(
                "Executing this proposal is not allowed at this time, because doing \
                 so would cause the 7 day upper bound of {allowance_tokens} tokens to be exceeded. \
                 Maybe, try again later? The total amount transferred in the past \
                 7 days stands at {spent_tokens} tokens, and the amount in this proposal is {transfer_amount_tokens} \
                 tokens. The upper bound is based on treasury valuation factors at \
                 the time of proposal submission: {valuation:?}",
            ),
        ));
    }

    Ok(())
