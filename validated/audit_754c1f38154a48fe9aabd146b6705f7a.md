### Title
Fixed 2300-Gas Stipend in `WSEI.withdraw()` Can Permanently Freeze Funds — (`contracts/src/WSEI.sol`)

---

### Summary

`WSEI.sol` uses `payable(msg.sender).transfer(wad)` in its `withdraw()` function. Post-Istanbul, `transfer()` forwards a hard-capped 2300 gas stipend. Any contract caller whose `receive()`/`fallback()` consumes more than 2300 gas (e.g., a multisig, a vault, or any contract that performs a single `SLOAD` at 2100 gas) will have the withdrawal revert every time, permanently freezing the deposited SEI inside the WSEI contract.

---

### Finding Description

In `contracts/src/WSEI.sol`, the `withdraw()` function sends native SEI back to the caller using Solidity's `transfer()`: [1](#0-0) 

`transfer()` hard-codes a 2300 gas forward. Post-EIP-1884 (Istanbul), a single `SLOAD` costs 2100 gas. Any contract `receive()` that reads one storage slot exhausts the stipend, causing the call to revert. Because `transfer()` propagates the revert, `withdraw()` fails unconditionally for such callers, and their balance in `balanceOf[msg.sender]` is already decremented before the call — wait, actually the decrement happens before the transfer: [2](#0-1) 

The balance is decremented at line 29, then `transfer()` is called at line 30. If `transfer()` reverts, the entire transaction reverts (Solidity reverts state on revert), so the balance is restored. The net effect is that the withdrawal **always reverts** for contract callers with non-trivial `receive()` hooks — their funds remain locked in WSEI with no alternative withdrawal path.

The same 2300-gas issue exists in `contracts/src/MultiSender.sol` via `.send()`: [3](#0-2) [4](#0-3) 

Here a single contract recipient in a batch can cause the entire batch to revert (DoS), but since the transaction reverts no funds are lost — this is a griefing/DoS issue rather than a freeze.

---

### Impact Explanation

Any contract account (multisig wallet, DeFi vault, DAO treasury) that:
1. Calls `WSEI.deposit()` (or sends SEI directly via `fallback`/`receive`), and
2. Has a `receive()` or `fallback()` that performs any storage read

…will be **permanently unable to call `withdraw()`**. Their wrapped SEI balance is irrecoverable through the contract's own interface. The amount frozen scales with how much SEI the contract deposited — it can trivially exceed $5k for any protocol-level integrator.

Severity mapping: **Critical** (fund freeze ≥ $5k possible) or at minimum **Medium** (fund freeze < $5k).

---

### Likelihood Explanation

WSEI is a wrapped-native token intended for DeFi composability. Multisigs (Gnosis Safe), yield aggregators, and lending protocols are the primary integrators — all of which have non-trivial `receive()` hooks. The likelihood of a contract caller hitting this is **high** in any real DeFi deployment on Sei EVM.

---

### Recommendation

Replace `transfer()` with a low-level `call{}` and check the return value:

```solidity
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    (bool success, ) = payable(msg.sender).call{value: wad}("");
    require(success, "WSEI: ETH transfer failed");
    emit Withdrawal(msg.sender, wad);
}
```

Apply the same fix to `MultiSender.sol`'s `batchTransferEqualAmount` and `batchTransfer` functions, replacing `.send()` with `.call{value: ...}("")`.

---

### Proof of Concept

1. Deploy a contract `Vault` with a `receive()` that reads one storage slot (`uint256 x = storedValue;`).
2. From `Vault`, call `WSEI.deposit{value: 1 ether}()`.
3. From `Vault`, call `WSEI.withdraw(1 ether)`.
4. The call reverts because `transfer()` forwards 2300 gas; the single `SLOAD` in `Vault.receive()` costs 2100 gas, leaving only 200 gas for the rest of the execution — insufficient.
5. `Vault`'s 1 SEI is permanently unrecoverable via `WSEI.withdraw()`.
