### Title
Permanent Fund Freeze When NEAR Transfer Fails in wNEAR Unwrap Exit Flow — (`engine/src/contract_methods/connector.rs`)

---

### Summary

When a user exits wNEAR ERC-20 tokens via the `ExitToNear` precompile with the `:unwrap` suffix, the flow burns the ERC-20 tokens, calls `near_withdraw` on the wNEAR NEP-141 contract, and then schedules a native NEAR `Transfer` action to the user-specified recipient account. If that final NEAR transfer fails (e.g., the target named NEAR account does not exist or has been deleted), the NEAR is refunded to the Aurora engine by the NEAR runtime, but the ERC-20 tokens are already permanently burned with no re-mint or claim mechanism. The user's funds are frozen forever.

---

### Finding Description

The wNEAR unwrap exit flow is:

1. User calls `withdrawToNear(recipient + ":unwrap", amount)` on the wNEAR ERC-20 contract.
2. The ERC-20 contract burns the tokens and calls the `ExitToNear` precompile.
3. In `exit_erc20_token_to_near()`, the `Message::UnwrapWnear` branch constructs a `TransferNearArgs { target_account_id: exit_params.receiver_account_id, amount }` and sets `method = "near_withdraw"`. [1](#0-0) 

4. A `PromiseWithCallbackArgs` is scheduled: base = `near_withdraw` on the wNEAR NEP-141 contract; callback = `exit_to_near_precompile_callback` on Aurora engine. [2](#0-1) 

5. When `near_withdraw` succeeds, `exit_to_near_precompile_callback` runs and schedules a `PromiseAction::Transfer` to `args.target_account_id`: [3](#0-2) 

**The critical gap:** `handler.promise_create_batch(&promise)` creates the NEAR transfer promise, and `handler.promise_return(promise_id)` sets it as the return value — but **no callback is attached to this promise**. If the NEAR transfer fails (e.g., the named account `args.target_account_id` does not exist), the NEAR runtime refunds the NEAR back to Aurora engine, but:

- The wNEAR ERC-20 tokens are already burned (irreversible EVM state change).
- The `refund` field in `ExitToNearPrecompileCallbackArgs` is only triggered when the **base** promise (`near_withdraw`) fails, not when the subsequent NEAR transfer fails. [4](#0-3) 

- There is no `claims` mapping or any other mechanism for the user to recover the NEAR that was refunded to Aurora engine.

The `error_refund` feature flag only guards re-minting of ERC-20 tokens when `near_withdraw` itself fails — it does not cover the case where `near_withdraw` succeeds but the subsequent NEAR transfer fails. [5](#0-4) 

---

### Impact Explanation

**Permanent freezing of funds.** The user's wNEAR ERC-20 tokens are burned and the corresponding NEAR is permanently stranded inside the Aurora engine contract with no recovery path. The NEAR runtime refunds the failed transfer amount back to Aurora engine (the predecessor), but Aurora engine has no logic to credit it back to the user or allow a claim. This matches the Critical impact tier: permanent freezing of user funds.

---

### Likelihood Explanation

The trigger condition is that the user-specified NEAR `receiver_account_id` does not exist at the time the NEAR transfer executes. This can happen in realistic scenarios:

- A user mistypes a named NEAR account (e.g., `alice.neer` instead of `alice.near`).
- A user specifies an account that existed when they submitted the transaction but was deleted before the promise executes (NEAR accounts can be deleted).
- A user programmatically constructs the exit call with an unregistered account.

The entry path is fully unprivileged: any EVM user holding wNEAR ERC-20 tokens can trigger this by calling `withdrawToNear` with `:unwrap` and a non-existent named NEAR account as recipient.

---

### Recommendation

Attach a callback to the NEAR `Transfer` promise scheduled inside `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-mint the equivalent amount of wNEAR ERC-20 tokens to the original sender (analogous to the existing `refund_on_error` path for the `near_withdraw` failure case). Alternatively, implement a `claims` mapping inside Aurora engine that records the stranded NEAR amount keyed by the original EVM sender address, and expose a `claim_near` method that allows the sender to retry the transfer to a different (valid) NEAR account.

---

### Proof of Concept

**Step-by-step:**

1. Alice holds wNEAR ERC-20 tokens on Aurora.
2. Alice calls `withdrawToNear("nonexistent.near:unwrap", 1_000_000)` on the wNEAR ERC-20 contract.
3. The ERC-20 contract burns Alice's tokens and invokes the `ExitToNear` precompile.
4. Aurora engine schedules: `near_withdraw(amount=1_000_000)` → callback `exit_to_near_precompile_callback`.
5. `near_withdraw` succeeds: wNEAR NEP-141 tokens are burned, 1 NEAR is unlocked to Aurora engine.
6. `exit_to_near_precompile_callback` runs (promise result = `Successful`), schedules `PromiseAction::Transfer { amount: 1_000_000 }` to `"nonexistent.near"`. [6](#0-5) 

7. The NEAR transfer fails because `"nonexistent.near"` does not exist. NEAR runtime refunds 1 NEAR to Aurora engine.
8. Alice's wNEAR ERC-20 tokens are permanently burned. The 1 NEAR is now inside Aurora engine with no claim mechanism.
9. Alice has permanently lost her funds.
