### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Users to Systematically Overpay Transaction Fees - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

The `compute_gas_refund` function implements a one-sided reconciliation between EVM gas consumption and native resource consumption. It increases `gas_used` when native resource consumption exceeds gas consumption (`delta_gas > 0`), but never decreases `gas_used` when gas consumption exceeds native resource consumption (`delta_gas < 0`). This asymmetric accounting causes users to overpay transaction fees whenever their EVM gas consumption is higher than what their native resource consumption warrants. A `TODO` comment in the code explicitly acknowledges this gap.

---

### Finding Description

In `compute_gas_refund`, the double-resource accounting reconciliation block (lines 66–81) reads:

```rust
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // native > gas: charge extra gas to cover native costs
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?
}
```

`delta_gas` represents the difference between the gas equivalent of native resource consumption and the EVM gas consumed:

```
delta_gas = (native_used / native_per_gas) - gas_used
```

where `native_per_gas = gas_price / native_price` (per the double-resource accounting docs).

**When `delta_gas > 0`** (native cost > gas cost): `gas_used` is increased so the user pays enough to cover native proving costs. This is correct.

**When `delta_gas < 0`** (native cost < gas cost): `gas_used` is **not** decreased. The user pays the full EVM-derived `gas_used` even though their native resource consumption only warrants a lower charge. The `TODO` comment at line 80 explicitly acknowledges this missing adjustment.

The resulting `gas_used` is then used directly in both fee paths:

- **ZK/L2 tx operator payment** (`zk/mod.rs` line 514): `token_to_pay_operator = gas_used * gas_price_for_operator`
- **ZK/L2 tx user refund** (`zk/mod.rs` line 458): `token_to_refund = (tx_gas_limit - gas_used) * gas_price`
- **ETH tx operator payment** (`ethereum/mod.rs` line 583+): same pattern

When `delta_gas < 0`, the user is charged `|delta_gas| * gas_price` more than the actual resource consumption warrants, and the operator receives this excess.

**Concrete scenario:** A transaction performs many EVM arithmetic/logic operations (high EVM gas) but writes to zero or few storage slots (low pubdata → low native). For example:
- `gas_used` (EVM) = 100,000
- `native_used / native_per_gas` = 60,000
- `delta_gas` = −40,000 → **not applied**
- User overpays by `40,000 * gas_price`

---

### Impact Explanation

Users who submit computation-heavy but storage-light transactions are charged more than the actual resource consumption warrants. The excess fee is transferred to the operator (coinbase). This is a direct, measurable financial loss for users — the magnitude is `|delta_gas| * gas_price = (gas_used − native_used/native_per_gas) * gas_price` per transaction. The funds are not permanently locked (they go to the operator), but the user's balance is reduced by more than the protocol should charge.

This is a resource accounting bug: the aggregate `gas_used` counter is updated in one direction (upward) but the symmetric downward correction is absent, mirroring the original report's pattern of an incomplete counter update.

---

### Likelihood Explanation

The condition `delta_gas < 0` is triggered whenever EVM gas consumption exceeds the gas-equivalent of native resource consumption. This is a common pattern:

- Any transaction with heavy EVM computation (loops, cryptographic operations, complex logic) but few or no storage writes will have low pubdata and thus low native consumption.
- The condition is reachable by any unprivileged transaction sender simply by crafting or submitting such a transaction.
- The `unlimited_native` feature gate means this only applies in production builds (where the feature is off), making it a live issue.

---

### Recommendation

Apply the `delta_gas` adjustment symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` so users receive a refund for the excess gas they paid relative to native costs:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Return excess gas to user when native cost < gas cost
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Ensure gas_used does not fall below minimal_gas_used (already applied above)
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

This resolves the TODO at line 80 and makes the double-resource accounting symmetric.

---

### Proof of Concept

1. Deploy a contract that performs many EVM arithmetic operations (e.g., a loop of 1,000 `MUL`/`ADD` instructions) but writes to no storage slots.
2. Submit an EIP-1559 transaction calling this contract with `gas_limit = 200,000`, `gas_price = 1,000`, and a `native_price` such that `native_per_gas` is low (e.g., `native_price` is high).
3. After execution, observe:
   - `gas_used` ≈ 150,000 (EVM computation)
   - `native_used / native_per_gas` ≈ 80,000 (low native due to no storage writes)
   - `delta_gas` = 80,000 − 150,000 = −70,000 → **not applied**
4. The user is charged `150,000 * 1,000 = 150,000,000` instead of `80,000 * 1,000 = 80,000,000`.
5. The operator receives the excess `70,000,000` that the user should have been refunded.
