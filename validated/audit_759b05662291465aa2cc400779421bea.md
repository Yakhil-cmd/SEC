### Title
Unprivileged Caller Can Inject Fake Interop Roots via Arbitrary Event Emission — (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` adds interop roots to the system's interop root storage based solely on the emitted event signature, without verifying that the event was emitted by the authorized `L2_INTEROP_ROOT_STORAGE_ADDRESS` contract. Any unprivileged user can deploy a contract that emits a `LOG3` with the `InteropRootAdded` event signature and attacker-controlled topic/data values, causing fake interop roots to be injected into the block's rolling hash commitment.

---

### Finding Description

The `interop_root_reporter_event_hook` in `system_hooks/src/event_hooks/interop_root_reporter.rs` is an event hook that fires whenever any EVM contract emits a log whose first topic matches `INTEROP_ROOT_ADDED_EVENT_SIG` (`0x6b451b84...`). [1](#0-0) 

The function signature accepts only `topics`, `data`, `_caller_ee`, `system`, and `resources`. There is no `caller` or `emitter` address parameter: [2](#0-1) 

The only guard is a check on `topics[0]`: [3](#0-2) 

If the signature matches, the hook unconditionally parses `chain_id`, `block_or_batch_number`, and `root` from the topics/data and calls `system.io.add_interop_root(...)`: [4](#0-3) 

Compare this to all call hooks (`l1_messenger_hook`, `set_bytecode_on_address_hook`, `mint_base_token_hook`), which all receive a `caller: B160` field from `ExternalCallRequest` and explicitly reject unauthorized callers: [5](#0-4) [6](#0-5) [7](#0-6) 

The event hook has no equivalent guard. The intended flow is that only `L2_INTEROP_ROOT_STORAGE_ADDRESS` emits `InteropRootAdded` events (triggered by service transactions calling `addInteropRootsInBatch`): [8](#0-7) 

But any EVM contract can emit a `LOG3` with the same signature and arbitrary topic values.

The injected roots are stored in `InteropRootStorage` and iterated over during block finalization to compute the rolling hash: [9](#0-8) [10](#0-9) 

---

### Impact Explanation

An attacker can corrupt the interop root rolling hash that is committed to in the block's public inputs. This hash is used by the settlement layer to verify cross-chain interoperability roots. Injecting fake roots with attacker-controlled `chain_id`, `block_or_batch_number`, and `root` values will:

1. Corrupt the rolling hash commitment, causing legitimate cross-chain message verification to fail (DoS of interop).
2. Potentially allow the attacker to craft a specific fake root that, when combined with the rolling hash, produces a value that passes downstream verification for a forged cross-chain message.

The state change is permanent within the block — `InteropRootStorage` does not deduplicate or validate roots against any on-chain source of truth at insertion time. [11](#0-10) 

---

### Likelihood Explanation

Any unprivileged user with ETH to pay gas can:
1. Deploy a minimal EVM contract that emits `LOG3(INTEROP_ROOT_ADDED_EVENT_SIG, chainId, blockNum)` with arbitrary data.
2. Call it in a regular (non-service) transaction.
3. The event hook fires unconditionally.

No special privileges, leaked keys, or governance access are required. The attack is cheap and repeatable within any block.

---

### Recommendation

Add an emitter address check to the event hook dispatch mechanism, or pass the emitting contract's address into `interop_root_reporter_event_hook` and reject events not originating from `L2_INTEROP_ROOT_STORAGE_ADDRESS`. This mirrors the pattern already used by all call hooks:

```rust
if emitter != L2_INTEROP_ROOT_STORAGE_ADDRESS {
    return Ok(());
}
```

Alternatively, restrict the event hook to only fire during service-block transactions, where the only permitted callee is already `L2_INTEROP_ROOT_STORAGE_ADDRESS`.

---

### Proof of Concept

1. Deploy the following minimal EVM contract on ZKsync OS:
```solidity
contract FakeInteropRoot {
    // InteropRootAdded(uint256,uint256,bytes32[]) sig = 0x6b451b84...
    bytes32 constant SIG = 0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0;
    function inject(uint256 chainId, uint256 blockNum, bytes32 root) external {
        // Emit LOG3: topics = [SIG, chainId, blockNum], data = abi.encode(32, 1, root)
        assembly {
            let ptr := mload(0x40)
            mstore(ptr,        0x20)          // offset
            mstore(add(ptr,32), 0x01)         // length = 1
            mstore(add(ptr,64), root)         // root value
            log3(ptr, 96, SIG, chainId, blockNum)
        }
    }
}
```

2. Call `inject(999, 1, 0xdeadbeef...)` from any EOA in a regular transaction.

3. Observe that `interop_root_reporter_event_hook` fires (topics[0] matches `INTEROP_ROOT_ADDED_EVENT_SIG`, topics length == 3, data length == 96), parses `chain_id=999`, `block_or_batch_number=1`, `root=0xdeadbeef...`, and calls `system.io.add_interop_root(...)`.

4. The fake root is included in the block's rolling hash computation at finalization, corrupting the interop root commitment in the block's public inputs. [12](#0-11)

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
```
