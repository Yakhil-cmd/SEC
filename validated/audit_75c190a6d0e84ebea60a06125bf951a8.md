### Title
Silo Address Whitelist Removal Permanently Freezes Previously Deposited User Funds - (`engine/src/engine.rs`)

### Summary

In Aurora Engine's Silo mode, the `assert_access` check applied to every EVM transaction submission makes no exception for withdrawal/exit operations. When a Silo owner removes a user's EVM address from the `Address` whitelist after the user has deposited ETH or ERC-20 tokens, the user loses all ability to submit any EVM transaction — including calls to the `exit_to_near` precompile — permanently freezing their on-chain funds.

### Finding Description

Aurora Engine's Silo mode enforces an `Address` whitelist on every EVM transaction via `assert_access` in `submit_with_alt_modexp`: [1](#0-0) 

`assert_access` unconditionally calls `silo::is_allow_submit` for all non-deploy transactions: [2](#0-1) 

`is_allow_submit` delegates to `is_address_allowed`, which checks the `Address` whitelist: [3](#0-2) 

The `Address` whitelist entry for any user can be removed at any time by the Silo owner via `remove_entry_from_whitelist`: [4](#0-3) [5](#0-4) 

There is **no exception** in `assert_access` for withdrawal transactions (e.g., EVM calls to the `exit_to_near` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`). The whitelist check is applied identically to all EVM transactions regardless of their purpose.

The deposit path (`ft_on_transfer` → `receive_erc20_tokens` / `receive_base_tokens`) does **not** require the recipient to be whitelisted — it only redirects to a fallback address if a fallback is configured: [6](#0-5) 

This means a user can receive funds into their EVM address (e.g., via a direct bridge transfer where the fallback is not set, or while they were whitelisted), and later be removed from the whitelist, leaving their funds permanently inaccessible.

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a user's EVM address is removed from the `Address` whitelist in Silo mode, they cannot submit any EVM transaction. This includes:
- Calling the `exit_to_near` precompile to bridge ETH back to NEAR.
- Calling any ERC-20 `transfer` or `approve` function to move tokens.
- Any other EVM interaction that would allow recovery of funds.

The funds (ETH balance and/or ERC-20 token balances) remain locked in the Aurora EVM state with no recovery path for the user.

### Likelihood Explanation

Silo mode is an explicitly supported deployment mode for private/enterprise Aurora instances. Address whitelist management (adding and removing entries) is a routine administrative operation. The scenario where a user is removed from the whitelist after having deposited funds — whether due to policy changes, user offboarding, or operational error — is realistic and has no safeguard in the current code. The `remove_entry_from_whitelist` entrypoint performs no balance check before removing an address.

### Recommendation

`assert_access` should exempt withdrawal/exit transactions from the whitelist check, or at minimum check whether the sender holds a non-zero balance before blocking the transaction. Concretely:

- Before removing an address from the `Address` whitelist, the engine should verify the address holds zero ETH balance and zero ERC-20 balances, or
- `assert_access` should allow transactions whose `to` field targets the `exit_to_near` precompile address (`exit_to_near::ADDRESS`) regardless of whitelist status, so users can always recover their funds.

### Proof of Concept

1. Silo mode is active with the `Address` whitelist enabled.
2. User's address `A` is added to the `Address` whitelist.
3. User bridges 1 ETH to address `A` via `ft_on_transfer` → `receive_base_tokens`. ETH balance is credited to `A`.
4. Silo owner calls `remove_entry_from_whitelist` for address `A`.
5. User attempts to submit an EVM transaction calling `exit_to_near` to recover their ETH.
6. `submit_with_alt_modexp` calls `assert_access` → `silo::is_allow_submit` → `is_address_allowed` → `Whitelist::is_exist` returns `false` → returns `EngineErrorKind::NotAllowed`.
7. The user's 1 ETH is permanently frozen with no recovery path. [7](#0-6) [8](#0-7) [9](#0-8)

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
```
