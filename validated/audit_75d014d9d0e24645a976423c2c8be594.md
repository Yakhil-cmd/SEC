Audit Report

## Title
Unbounded Linear Scan Over `processed_withdrawal_requests` in `withdrawal_status` Query Causes Permanent Instruction-Limit DoS — (`File: rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary

`EthTransactions::withdrawal_status` performs a full linear scan over `processed_withdrawal_requests`, a `BTreeMap` that is populated on every withdrawal and **never pruned**. Because the endpoint is an unrestricted `#[query]` callable by any user, and the IC query instruction limit is 5 billion instructions, the endpoint's per-call cost grows monotonically with cumulative withdrawal volume. Once the map is large enough, every call traps with `CanisterInstructionLimitExceeded`, permanently disabling withdrawal status queries for all users of the ckETH/ckERC20 minter.

## Finding Description

`EthTransactions::withdrawal_status` at lines 805–841 of `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` iterates every value in `processed_withdrawal_requests` unconditionally:

```rust
let processed = self
    .processed_withdrawal_requests
    .values()
    .filter(|r| r.match_parameter(parameter))
    ...
```

This scan is O(n) in the total number of processed withdrawals, regardless of the search parameter type (`ByWithdrawalId`, `ByRecipient`, or `BySenderAccount`). Notably, for `ByWithdrawalId` the map key is `LedgerBurnIndex`, so a direct O(log n) `.get()` lookup is possible but unused.

`processed_withdrawal_requests` is a plain `BTreeMap<LedgerBurnIndex, WithdrawalRequest>` with no eviction policy. Entries are inserted in `record_created_transaction` (line 548) and there is no corresponding removal anywhere in the codebase — confirmed by the absence of any `processed_withdrawal_requests.remove` or `processed_withdrawal_requests.retain` call.

The public query endpoint `withdrawal_status` in `rs/ethereum/cketh/minter/src/main.rs` (line 348) is annotated `#[query]` with no caller authentication or access control, making it callable by any unprivileged user. The DID interface confirms this: `withdrawal_status : (WithdrawalSearchParameter) -> (vec WithdrawalDetail) query;`.

The IC query instruction limit is `MAX_INSTRUCTIONS_PER_QUERY_MESSAGE = 5 * B` (5 billion instructions), confirmed in `rs/config/src/subnet_config.rs` line 41.

## Impact Explanation

This matches the allowed impact: **High — Application/platform-level DoS with concrete user and protocol harm**. Once `processed_withdrawal_requests` accumulates enough entries, every call to `withdrawal_status` traps with `CanisterInstructionLimitExceeded`, permanently disabling the endpoint for all callers. Users cannot query the status of their own withdrawals. The `retrieve_eth_status` update endpoint (40B limit) follows the same growth trajectory but has a higher headroom.

## Likelihood Explanation

No adversarial action is required for the natural degradation path: every finalized withdrawal permanently adds one entry to `processed_withdrawal_requests`. The minimum ckETH withdrawal amount was recently reduced to 0.005 ETH (~$10 at current prices), lowering the cost per entry. An attacker wishing to accelerate degradation can submit many minimum-value withdrawals; the cost is bounded only by the minimum withdrawal amount multiplied by the number of entries needed to exhaust 5B instructions. The exact instruction cost per BTreeMap element traversal (including `match_parameter` comparisons) determines the threshold, but the growth is monotonic and unbounded regardless.

## Recommendation

1. **Prune `processed_withdrawal_requests`**: Remove entries once a withdrawal reaches a terminal state (finalized + reimbursement settled). Retain only minimal historical data in a separate bounded structure if needed.
2. **Index by search key**: For `ByWithdrawalId`, use `processed_withdrawal_requests.get(&burn_index)` directly (O(log n)). For `ByRecipient` and `BySenderAccount`, maintain secondary `BTreeMap<Address, BTreeSet<LedgerBurnIndex>>` indexes to avoid full scans.
3. **Add a result cap and pagination**: Bound per-call instruction cost regardless of total state size.
4. **Fix `record_withdrawal_request` duplicate check**: Replace the O(n) linear scan over `pending_withdrawal_requests` with a shadow `BTreeSet<LedgerBurnIndex>` for O(log n) membership tests.

## Proof of Concept

1. Over time (or via deliberate minimum-value withdrawals), `processed_withdrawal_requests` accumulates N entries.
2. Any caller invokes `withdrawal_status(ByRecipient("0x..."))` — an unrestricted query with no rate limiting.
3. `EthTransactions::withdrawal_status` iterates all N entries in `processed_withdrawal_requests.values()` regardless of how many match.
4. When N is large enough that the per-call instruction count exceeds 5B, the query traps with `CanisterInstructionLimitExceeded`, permanently disabling the endpoint.

A deterministic integration test can reproduce this by inserting a large number of synthetic withdrawal records into `processed_withdrawal_requests` via `record_created_transaction` in a PocketIC or StateMachine test environment, then calling `withdrawal_status` and observing the instruction count grow linearly with N until the limit is exceeded.
