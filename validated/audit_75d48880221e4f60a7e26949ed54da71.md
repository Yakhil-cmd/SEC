Audit Report

## Title
Private Custom Section Existence Oracle via Differential HTTP Status Codes in `can_read_canister_metadata` — (`rs/http_endpoints/public/src/read_state.rs`)

## Summary
`can_read_canister_metadata` returns `Err(403)` exclusively when a private custom section with the queried name exists and the caller is not a controller. All other cases — canister absent, no execution state, section absent, or section public — return `Ok(())`, which propagates to HTTP 200 with an Absent leaf. This asymmetry allows any unprivileged caller to determine whether a specific private custom section name exists on any canister by observing the HTTP status code.

## Finding Description
The function `can_read_canister_metadata` (lines 593–630) implements the following decision tree:

- Canister not found → `Ok(())` (line 601)
- Canister found, no execution state → `Ok(())` (line 628)
- Canister found, section not found → `Ok(())` (line 611)
- Canister found, section **private**, caller not controller → `Err(HttpError { status: 403 })` (lines 618–623)
- Canister found, section public → `Ok(())` (line 626)

The error is propagated directly to the HTTP response at lines 270–282 without normalization. The 403 path is the only distinguishable outcome, and it fires only when the private section actually exists. An attacker probing section name `"secret"` receives HTTP 403 if and only if the canister has a private custom section named `"secret"` and the caller is not a controller. All other states return HTTP 200. The existing guard (lines 615–616) correctly enforces access control on the section *content*, but it inadvertently leaks section *existence* through the error code.

## Impact Explanation
This is a limited information disclosure: an unprivileged caller can enumerate whether specific private custom section names exist on any canister. Private sections are explicitly marked private by canister developers; their existence (e.g., `candid`, `dfx`, application-specific metadata names) may reveal implementation details, tooling choices, or application structure. The impact does not extend to section content. This fits the Medium bounty tier: meaningful security impact (violation of the privacy guarantee of private custom sections) achievable without any privileges, but limited to existence disclosure rather than content exposure.

## Likelihood Explanation
Exploitation requires no privilege, no key, and no special conditions. The entry point is the public `/api/v2/canister/{id}/read_state` endpoint, reachable by any anonymous caller. The probe is a single HTTP POST with a valid (including anonymous) identity. No side effects, no cycles consumed. For well-known section names (`candid` is a universal convention), the attack is immediately effective against any canister on any subnet.

## Recommendation
Restructure `can_read_canister_metadata` so that a non-controller always receives `Ok(())` for any private section path, regardless of whether the section exists. The certified state tree will naturally return an Absent leaf for paths the caller is not entitled to read, without leaking existence. Concretely, move the controller check before the section lookup: if the caller is not a controller, return `Ok(())` unconditionally for any metadata path, and let the state tree handle the Absent response. This eliminates the distinguishable 403 path for non-controllers entirely.

## Proof of Concept
```
// 1. Install a canister with a private custom section named "secret"
// 2. As a non-controller (anonymous identity), POST to:
//    /api/v2/canister/<canister_id>/read_state
//    paths: [["canister", <canister_id_bytes>, "metadata", "secret"]]
// Observe: HTTP 403 → section "secret" exists and is private

// 3. Probe a section name that does not exist, e.g. "nonexistent"
//    Same endpoint, same caller
// Observe: HTTP 200 with Absent leaf → section does not exist

// The 403 vs 200 distinction is a certified oracle for private section existence.

// delete_canister while asserting that a non-controller probe of the section
```
