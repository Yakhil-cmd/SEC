Audit Report

## Title
Third-Party Caller Can Bypass `confirmation_text` Consent Gate and Force SNS Swap Participation on Behalf of Any Pre-Funded Buyer - (File: `rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal in its request body and performs no check that the IC caller matches the specified buyer. When an SNS swap is configured with a `confirmation_text` (a documented consent mechanism), any unprivileged caller can supply a victim's principal as `buyer` along with the publicly readable `confirmation_text`, forcing the victim's pre-deposited ICP into the swap without their explicit consent. If the swap commits, the victim's ICP is swept to the SNS treasury and they receive SNS tokens with no recourse to reclaim ICP.

## Finding Description

In `rs/sns/swap/canister/canister.rs` (L127–143), the canister entry point resolves the buyer principal from the request body rather than from the IC caller:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // attacker-controlled
};
```

No check enforces `caller == p` when `arg.buyer` is non-empty. The resolved principal `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` (L1134–1163), which calls `validate_confirmation_text` (L1150) to compare the supplied text against the SNS-configured text stored in `Init`. This comparison is purely textual — it does not verify that the caller is the buyer:

```rust
pub fn validate_confirmation_text(&self, confirmation_text: Option<String>) -> Result<(), String> {
    match (self.init_or_panic().confirmation_text.as_ref(), confirmation_text) {
        (Some(expected_text), Some(text)) => {
            if &text != expected_text { Err(...) } else { Ok(()) }
        }
        ...
    }
}
```

The `confirmation_text` is set at SNS initialization and is publicly readable via `get_state` / `get_init`. The proto documentation at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` (L322–325) describes it as: *"An optional text that swap participants should confirm before they may participate in the swap."* The code comment at `rs/sns/swap/src/swap.rs` (L1123–1126) states *"the caller of this function must accept this confirmation"*, but this invariant is never enforced in code.

**Exploit flow:**
1. SNS swap is deployed with `confirmation_text = "I agree to the terms"` and is in `Open` state.
2. Victim transfers ICP to `subaccount(swap_canister, victim_principal)` on the ICP ledger as a preparatory step, but has not yet called `refresh_buyer_tokens` (e.g., they changed their mind and plan to reclaim via `error_refund_icp` after the swap closes).
3. Attacker reads `confirmation_text` from the public swap state.
4. Attacker submits `refresh_buyer_tokens({ buyer: "<victim_principal>", confirmation_text: "I agree to the terms" })` from their own principal.
5. The swap canister reads the balance from `subaccount(swap_canister, victim_principal)` on the ICP ledger and registers the victim's ICP as committed participation (`rs/sns/swap/src/swap.rs` L1285–1288).
6. If the swap commits, the victim's ICP is swept to the SNS treasury and SNS tokens are distributed to the victim. The victim cannot reclaim ICP via `error_refund_icp`, which only operates on unregistered or post-disbursement residual balances (`rs/sns/swap/src/swap.rs` L1906–1960).

**Why existing checks fail:** The `validate_confirmation_text` check only validates the text string value, not the identity of who provided it. The `buyer` field is intentionally designed to allow third-party notification (proto L844: *"If not specified, the caller is used"*), but this design conflicts with the `confirmation_text` consent requirement, which is documented as a per-participant gate.

## Impact Explanation

This is a **High** severity finding. A victim who has deposited ICP into the swap subaccount but has not yet consented to participate (and intends to reclaim their ICP) can have their participation forcibly committed by any unprivileged attacker. If the swap commits, the victim permanently loses their ICP and receives SNS tokens instead, with no on-chain recourse. This constitutes unauthorized control over user funds held in the SNS swap canister, matching the allowed impact: *"Significant SNS infrastructure security impact with concrete user or protocol harm"* and *"Unauthorized access to... canister-controlled funds."*

## Likelihood Explanation

The attack requires no privileged access. All preconditions are observable on public infrastructure: the `confirmation_text` is readable from the swap canister's public state, and the victim's ICP transfer to the swap subaccount is visible on the public ICP ledger. The attacker needs only to submit a single update call. The swap's `Open` window (days to weeks) provides ample time. The only constraint is that the victim must have already transferred ICP to the swap subaccount, which is a realistic scenario (e.g., a user who deposited ICP but then decided not to participate before calling `refresh_buyer_tokens`).

## Recommendation

Enforce that when `confirmation_text` is configured, the `buyer` field must equal the IC caller's principal. The minimal fix in `rs/sns/swap/canister/canister.rs`:

```rust
#[update]
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let requested = PrincipalId::from_str(&arg.buyer).unwrap();
        if swap().init_or_panic().confirmation_text.is_some() {
            assert_eq!(caller, requested,
                "caller must match buyer when confirmation_text is configured");
        }
        requested
    };
    ...
}
```

If third-party notification is intentionally desired even when `confirmation_text` is set, the consent must be recorded separately per-buyer via a signed mechanism rather than a publicly readable string comparison.

## Proof of Concept

A deterministic PocketIC or `StateMachine` integration test can prove this:

1. Deploy an SNS swap with `confirmation_text = Some("I agree to the terms")` and advance to `Open` state.
2. As `victim_principal`, transfer `min_participant_icp_e8s` ICP to `subaccount(swap_canister, victim_principal)` on the ICP ledger. Do **not** call `refresh_buyer_tokens` as the victim.
3. As `attacker_principal` (a different identity), call:
   ```
   refresh_buyer_tokens({
     buyer: victim_principal.to_string(),
     confirmation_text: Some("I agree to the terms".to_string())
   })
   ```
4. Assert the call succeeds and `get_buyer_state(victim_principal)` returns a non-zero `icp_accepted_participation_e8s`.
5. Advance the swap to `Committed` state and call `finalize_swap`.
6. Assert the victim's ICP subaccount balance is zero and the victim holds SNS neurons, confirming forced participation without the victim ever calling `refresh_buyer_tokens`.
7. Assert that `error_refund_icp` for the victim returns a precondition error or zero balance, confirming no recourse.
