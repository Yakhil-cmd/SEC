Audit Report

## Title
ICP Burn Silently Ignored After Successful Cycles Minting, Breaking Conservation Invariant - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the Cycles Minting Canister is explicitly designed to swallow all errors from the ICP ledger burn that must follow a successful cycles mint or canister creation. If the burn call fails transiently, cycles have already been deposited to the caller, the ICP remains unburned in the CMC's subaccount, and the notification is permanently marked complete — preventing any retry. This breaks the ICP/cycles conservation invariant: cycles are created without the corresponding ICP being destroyed.

## Finding Description
Three processing functions call `burn_and_log` **after** the irreversible action has already succeeded:

- `process_top_up` (lines 1999–2002): calls `deposit_cycles`, then `burn_and_log`
- `process_mint_cycles` (lines 1966–1968): calls `do_mint_cycles`, then `burn_and_log`
- `process_create_canister` (lines 1943–1945): calls `do_create_canister`, then `burn_and_log` [1](#0-0) [2](#0-1) [3](#0-2) 

The `burn_and_log` function is explicitly annotated: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* It catches all ledger call errors and returns `()` regardless of outcome: [4](#0-3) 

Specifically, the burn silently no-ops if:
1. `minting_account_id` is `None` (lines 2020–2023)
2. `amount < DEFAULT_TRANSFER_FEE` (lines 2027–2030)
3. The `call_protobuf` call to the ICP ledger returns any `Err` (lines 2044–2047)

After `burn_and_log` returns, the caller stores the result as a terminal `NotificationStatus` (`NotifiedTopUp`, `NotifiedMint`, or `NotifiedCreateCanister`), permanently closing the notification: [5](#0-4) [6](#0-5) [7](#0-6) 

The `is_transient_error` check only removes the notification entry when the result is a transient error. A successful `Ok(cycles)` result from `process_top_up` is never transient, so the entry is always permanently stored even when the burn failed silently.

The design rationale in the comment is to prevent double-spending of cycles via retry, but this creates the inverse problem: a failed burn leaves ICP unburned in the CMC's subaccount with no recovery path, while the cycles are already in circulation.

## Impact Explanation
When `burn_and_log` fails silently:
- The user has already received cycles (deposited to their canister or cycles-ledger account).
- The ICP is **not** burned — it remains in the CMC's subaccount keyed by the user's principal/canister ID.
- The notification is permanently marked complete; neither the user nor the protocol can retry.
- The ICP/cycles conservation invariant is violated: new cycles exist without the corresponding ICP being destroyed, inflating the effective cycles supply relative to the ICP supply.

This constitutes **illegal minting** — cycles are created without the corresponding ICP being destroyed — matching the High bounty impact: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."* Repeated occurrences (e.g., during a ledger upgrade window) compound the invariant violation.

## Likelihood Explanation
The ICP ledger is a production canister on the NNS subnet. Transient inter-canister call failures (reject codes from the subnet, ledger temporarily stopping for upgrade, message queue full) are realistic and have occurred historically on mainnet. No privileged access, key compromise, or majority attack is required — only a timing coincidence with a ledger transient error. Any unprivileged user who calls `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` during such a window will receive cycles while the ICP burn is silently dropped. The window is particularly wide during NNS ledger upgrades, which are routine governance operations.

## Recommendation
1. **Do not silently discard burn failures.** If `burn_and_log` fails, the notification should remain in `Processing` state (allowing retry) or be stored with a distinct `BurnFailed` status that a privileged operator can later resolve.
2. **Separate the burn from the success response.** The burn should be confirmed before the notification is finalized, or the notification should only be finalized after the burn is confirmed.
3. **Add an invariant monitor** (e.g., in `heartbeat`) that compares the sum of all CMC subaccount balances against expected zero, alerting on any residual ICP that was not burned.

## Proof of Concept
1. User sends `N` ICP to the CMC's subaccount (keyed by their canister ID) with memo `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id }`.
3. CMC fetches the transaction, sets `blocks_notified[block_index] = Processing`, and calls `process_top_up`.
4. `deposit_cycles` succeeds — the target canister receives cycles.
5. `burn_and_log` is called. At this moment, the ICP ledger is transiently unavailable (e.g., mid-upgrade). The `call_protobuf` call returns `Err(...)`. `burn_and_log` logs the error and returns `()`.
6. `process_top_up` returns `Ok(cycles)`. The CMC stores `NotifiedTopUp(Ok(cycles))`.
7. The user's canister has received cycles. The `N` ICP remains unburned in the CMC's subaccount. The notification is permanently closed.
8. The ICP/cycles conservation invariant is broken: `N` ICP was not destroyed but `N * rate` cycles were created.

A deterministic integration test can reproduce this by deploying a mock ICP ledger that returns a reject error on the `send_pb` call, then verifying that the CMC subaccount balance is non-zero after a successful `notify_top_up` call.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
