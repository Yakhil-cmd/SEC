### Title
XCC Router Upgrade Failure Permanently Blocks All Future XCC Calls for an Address - (`engine/src/xcc.rs`)

### Summary

When the XCC router deployment or upgrade batch fails on-chain (e.g., due to insufficient gas, NEAR runtime error, or wNEAR registration failure), the `factory_update_address_version` callback correctly does **not** update the stored version for that address. However, every subsequent XCC call from that address will again attempt the same upgrade batch — and because the upgrade batch is chained as a **prerequisite callback** before the user's actual call, any persistent failure in the upgrade step permanently blocks all XCC calls from that address. The user's intended cross-contract call is never executed, and there is no fallback path to execute it without the upgrade succeeding first.

### Finding Description

In `engine/src/xcc.rs`, the function `handle_precompile_promise` constructs a chained promise sequence for every XCC call:

1. **Deploy/upgrade batch** → `factory_update_address_version` callback (to record the new version)
2. → `withdraw_wnear_to_router` (if NEAR is needed)
3. → The user's actual `execute` or `schedule` call on the router [1](#0-0) 

The `AddressVersionStatus::new` function determines whether a deploy is needed by comparing the address's stored version against `latest_code_version`: [2](#0-1) 

If the deploy batch fails, `factory_update_address_version` returns `ERR_ROUTER_DEPLOY_FAILED` and does **not** update the stored version: [3](#0-2) 

Because the stored version is never updated, every subsequent XCC call from that address will again evaluate `AddressVersionStatus::DeployNeeded`, again prepend the upgrade batch, and again chain the user's call as a dependent callback. If the root cause of the failure persists (e.g., the router code is too large for the gas budget, or the wNEAR account is misconfigured), **all future XCC calls from that address are permanently blocked** — the user's call is always chained after the failing upgrade and is never reached.

The `execute_scheduled` path in the XCC router is callable by anyone and does not check preconditions: [4](#0-3) 

However, this only helps for already-scheduled promises. New XCC calls via the precompile always go through `handle_precompile_promise`, which re-evaluates the version status and re-chains the upgrade.

The `fund_xcc_sub_account` path has the same structural issue: [5](#0-4) 

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

Any EVM user whose XCC router upgrade batch fails is unable to execute any further cross-contract calls. Since XCC is the mechanism by which EVM contracts interact with the NEAR ecosystem (including unwrapping wNEAR, calling DeFi protocols, etc.), a blocked XCC path means:

- Funds locked in the router sub-account (wNEAR, NEAR) cannot be retrieved via XCC.
- Any EVM contract that depends on XCC for its operation (e.g., a contract that uses XCC to liquidate positions or release collateral) is permanently broken for that address.
- The user cannot self-recover: there is no engine method to reset the stored version or bypass the upgrade chain.

### Likelihood Explanation

**Medium.** The upgrade batch can fail due to:
- Insufficient gas allocated to `UPGRADE_GAS + INITIALIZE_GAS` for a large router binary.
- The wNEAR account not being registered (the `storage_deposit` promise in `initialize` can fail silently or run out of gas).
- NEAR runtime-level failures during contract deployment.

These are realistic on-chain conditions, not theoretical. Any user who triggers an XCC call at the moment a new router version is deployed by the engine owner is exposed to this race condition.

### Recommendation

1. **Decouple the upgrade from the user call.** Do not chain the user's `execute`/`schedule` call as a callback of the upgrade batch. Instead, if an upgrade is needed, only perform the upgrade in this transaction and require the user to retry their XCC call in a subsequent transaction once the upgrade has succeeded.

2. **Alternatively**, if the upgrade fails (i.e., `factory_update_address_version` receives `Some(false)`), fall back to treating the address as `UpToDate` for the purpose of the current call, so the user's call can still proceed against the existing (older) router.

3. Add a recovery method (callable by the address owner or anyone) to manually reset or retry the version update for a stuck address.

### Proof of Concept

**Setup:**
- Engine owner calls `factory_update` to deploy a new router version (e.g., version 2). `latest_code_version` is now 2.
- User Alice has an existing router at version 1 (stored in engine state).

**Attack/Failure Scenario:**
1. Alice submits an EVM transaction that triggers the XCC precompile (`CrossContractCallArgs::Eager`).
2. `handle_precompile_promise` is called. It sees `sender_code_version = Some(1)` and `latest_code_version = 2`, so `AddressVersionStatus::DeployNeeded { create_needed: false }`.
3. The promise chain is constructed: `[deploy_upgrade batch] → [factory_update_address_version] → [withdraw_wnear_to_router] → [execute on router]`.
4. The `deploy_upgrade` batch fails (e.g., out of gas, or the new router binary is malformed).
5. `factory_update_address_version` is called as a callback, sees `Some(false)`, returns `ERR_ROUTER_DEPLOY_FAILED`. Alice's stored version remains 1.
6. Alice retries her XCC call. Step 2 repeats identically — `AddressVersionStatus::DeployNeeded` again — and the same failing upgrade is prepended. Alice's actual call is never reached.
7. Alice's funds in the router sub-account are inaccessible via XCC indefinitely. [6](#0-5) [7](#0-6)

impl AddressVersionStatus {
    fn new<I: IO>(
        io: &I,
        latest_code_version: CodeVersion,
        target_code_version: Option<CodeVersion>,
    ) -> Self {
        let first_upgradable_version =
            get_first_upgradable_version(io).unwrap_or(CodeVersion::ZERO);
        match target_code_version {
            None => Self::DeployNeeded {
                create_needed: true,
            },
            Some(version) if version < first_upgradable_version => {
                // It is impossible to upgrade the initial XCC routers because
                // they lack the upgrade method.
                Self::UpToDate
            }
            Some(version) if version < latest_code_version => Self::DeployNeeded {
                create_needed: false,
            },
            Some(_version) => Self::UpToDate,
        }
    }

#[named]
pub fn factory_update_address_version<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &H,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        // The function is only set to be private, otherwise callback error will happen.
        env.assert_private_call()?;
        let check_deploy: Result<(), &[u8]> = match handler.promise_result_check() {
            Some(true) => Ok(()),
            Some(false) => Err(b"ERR_ROUTER_DEPLOY_FAILED"),
            None => Err(b"ERR_ROUTER_UPDATE_NOT_CALLBACK"),
        };
        check_deploy?;
        let args: xcc::AddressVersionUpdateArgs = io.read_input_borsh()?;
        xcc::set_code_version_of_address(&mut io, &args.address, args.version);
        Ok(())
    })
