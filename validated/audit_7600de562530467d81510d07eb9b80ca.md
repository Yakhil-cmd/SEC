### Title
Rebasing NEP-141 Token Rewards Permanently Locked in Aurora Engine Bridge — (`engine/src/engine.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

When a rebasing NEP-141 token (e.g., a yield-bearing liquid-staking token like stNEAR) is bridged to Aurora via `ft_on_transfer`, the engine mints ERC-20 mirror tokens 1:1 against the deposited amount. Because the ERC-20 supply is fixed at deposit time and the exit precompile transfers exactly the burned ERC-20 amount back to NEAR, any additional NEP-141 balance that accrues inside the Aurora Engine contract through rebasing is permanently unclaimable. There is no share-based accounting and no recovery path.

---

### Finding Description

The NEP-141 → ERC-20 bridge flow works as follows:

**Deposit path:**
A user calls `ft_transfer_call` on a NEP-141 contract, routing tokens to the Aurora Engine. The NEP-141 contract then calls `ft_on_transfer` on Aurora, which dispatches to `receive_erc20_tokens`:

```rust
// engine/src/engine.rs  ~line 803
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount)
```

This calls `mint(recipient, amount)` on the ERC-20 mirror contract, creating exactly `amount` ERC-20 tokens.

**Exit path:**
A user calls `withdrawToNear(recipient, amount)` on `EvmErc20.sol`:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol  line 53-62
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    // calls ExitToNear precompile
}
```

The `ExitToNear` precompile then calls `ft_transfer` on the NEP-141 contract for exactly `amount` tokens:

```rust
// engine-precompiles/src/native.rs  ~line 630-636
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()
)
```

**The gap:** If the bridged NEP-141 token is rebasing (its total supply grows over time, increasing every holder's balance proportionally), the Aurora Engine contract's NEP-141 balance grows beyond the sum of all deposited amounts. However, the ERC-20 mirror token's total supply is fixed at the sum of deposits. The surplus NEP-141 tokens accumulate in the Aurora Engine account with no corresponding ERC-20 tokens to claim them, and no administrative function exists to sweep or distribute them.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / Permanent freezing of funds.**

Every rebasing reward cycle increases the NEP-141 balance held by the Aurora Engine contract, but the ERC-20 mirror supply does not grow. Users can only ever withdraw the exact amount they deposited. The delta — all accrued yield — is permanently locked inside the Aurora Engine NEAR account with no recovery mechanism. Over time, for a popular yield-bearing NEP-141 token, this could represent a substantial and growing sum of permanently inaccessible funds.

---

### Likelihood Explanation

The NEAR ecosystem has multiple rebasing or yield-accruing NEP-141 tokens (e.g., liquid staking derivatives). Any such token can be permissionlessly bridged to Aurora by any user via `ft_transfer_call` → `ft_on_transfer` → `deploy_erc20_token`. No admin action is required to trigger the vulnerability; it activates automatically as soon as the underlying NEP-141 rebases after a deposit. The entry path is fully unprivileged and reachable by any token holder.

---

### Recommendation

1. **Share-based accounting:** Track each user's deposit as a proportion of the total NEP-141 balance held by the engine at deposit time. On exit, compute the withdrawal amount as `(user_shares / total_shares) * current_nep141_balance`. This distributes rebasing rewards proportionally.

2. **Minimal mitigation:** Explicitly reject bridging of known rebasing NEP-141 tokens, or document that only non-rebasing tokens are supported and enforce this via an allowlist.

---

### Proof of Concept

1. Deploy a rebasing NEP-141 token `rebase.near` whose balance for each holder increases by 10% per epoch.
2. Alice holds 1000 `rebase.near` tokens and calls `ft_transfer_call` to bridge 1000 tokens to Aurora. Aurora mints 1000 `rebase.near` ERC-20 tokens for Alice.
3. One epoch passes. The Aurora Engine's `rebase.near` balance is now 1100 (10% rebase). Alice's ERC-20 balance is still 1000.
4. Alice calls `withdrawToNear(alice.near, 1000)` on the ERC-20 contract. `EvmErc20.withdrawToNear` burns 1000 ERC-20 tokens and the `ExitToNear` precompile calls `ft_transfer` for exactly 1000 `rebase.near` tokens.
5. Alice receives 1000 tokens. The remaining 100 rebased tokens sit in the Aurora Engine's `rebase.near` balance permanently. No ERC-20 tokens exist to claim them, and no contract function can release them.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
