Audit Report

## Title
VIP Caller Starvation via Unconditional Slot-Guard in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` checks `available_slot_count == 0` unconditionally before applying the VIP distinction, causing NNS canisters (e.g., Governance) to be rejected with `SysTransient` when all 167 slots are occupied. Because NNS Root's `canister_status` endpoint is public and auth-free, any unprivileged principal can exhaust the slot pool and persistently block Governance's management-canister calls.

## Finding Description
In `rs/nervous_system/clients/src/management_canister_client.rs` at lines 264–287, `try_borrow_slot` first computes `used_slot_count = 0` for VIP callers, but then immediately enters a `with_borrow_mut` closure that returns `Err` whenever `*available_slot_count == 0` — before the VIP `used_slot_count` value is ever consulted:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };  // VIP = 0

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {          // fires for VIPs too
                return Err((RejectCode::SysTransient as i32,
                            "Unavailable. Maybe, try again later?".to_string()));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The `SlotLoan::drop` implementation adds `used_slot_count` back to the counter, so a VIP call that succeeds returns 0 slots (no net consumption). But when the counter is already 0, the guard fires and the VIP call never reaches the subtraction step — the VIP distinction is entirely bypassed.

The attacker entry point is `canister_status` in `rs/nns/handlers/root/impl/canister/canister.rs` (lines 88–98), which is explicitly documented as public ("anyone can get the status of any NNS canister") and performs no caller check. The slot pool is fixed at 167 (line 50). VIP classification (lines 57–60) marks all `ALL_NNS_CANISTER_IDS` members, including `GOVERNANCE_CANISTER_ID`, as VIP — with the stated intent that they are never slot-limited.

An attacker sends 167 concurrent `canister_status` calls from non-NNS principals, each awaiting a management canister response. While those calls are in-flight, the counter is 0. Any subsequent call from Governance hits the guard and receives `SysTransient`. The attacker replenishes slots as old ones complete, sustaining the condition indefinitely with only 167 concurrent in-flight calls at any moment.

## Impact Explanation
NNS Governance relies on `NnsRootCanisterClientImpl::canister_status` (which calls Root's public `canister_status` endpoint) to check canister statuses during governance operations such as canister upgrades. Persistent rejection of these calls with `SysTransient` disrupts NNS governance operations. This matches the High bounty impact: **Application/platform-level DoS with concrete NNS governance harm, not based on raw volumetric DDoS**.

## Likelihood Explanation
No privileges are required. The `canister_status` endpoint on NNS Root is explicitly public. Holding 167 concurrent in-flight calls is feasible: each call borrows a slot and suspends at the `await` point while the management canister processes the request; multiple such calls can be simultaneously in-flight across execution rounds. The attacker needs only to continuously replenish the 167 slots as responses arrive — no high-bandwidth flood is required.

## Recommendation
Add a VIP bypass to the guard in `try_borrow_slot`:

```rust
if !self.is_caller_vip && *available_slot_count == 0 {
    let code = RejectCode::SysTransient as i32;
    let message = "Unavailable. Maybe, try again later?".to_string();
    return Err((code, message));
}
```

This preserves the invariant that VIP callers are never rejected by the slot-limit guard while non-VIP callers remain rate-limited.

## Proof of Concept
State-machine / PocketIC test:
1. Deploy NNS Root with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Submit 167 concurrent `canister_status` update calls from 167 distinct non-NNS principals targeting a canister Root controls; hold all 167 in-flight simultaneously (awaiting management canister response in the same round).
3. Before any response is delivered, submit a `canister_status` call from `GOVERNANCE_CANISTER_ID`.
4. Assert the Governance call returns `Err((SysTransient, "Unavailable. Maybe, try again later?"))`.

The failure is directly observable at line 269 of `management_canister_client.rs`: the `*available_slot_count == 0` branch is taken regardless of `is_caller_vip`.
