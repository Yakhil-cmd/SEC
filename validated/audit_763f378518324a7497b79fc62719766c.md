Audit Report

## Title
Missing Upper-Bound Enforcement on `limit` in `search_transactions` Allows Memory Exhaustion via Unbounded SQLite Query — (`rs/rosetta-api/icrc1/src/data_api/services.rs`)

## Summary
The `search_transactions` handler resolves the caller-supplied `limit` using `unwrap_or`, which only substitutes the `MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST = 10000` default when `limit` is absent. An explicit `limit` of `i64::MAX` bypasses the default entirely, converts to `u64` without error, and is forwarded verbatim to the SQLite `LIMIT` clause, causing the database to return all matching rows and loading them into the Rosetta process heap. A single unauthenticated HTTP POST can exhaust process memory and crash or severely degrade the Rosetta API.

## Finding Description
`MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST` is defined as `10000` at [1](#0-0)  but is applied only as a fallback default via `unwrap_or` at [2](#0-1)  When `request.limit` is `Some(9223372036854775807)` (`i64::MAX`), the `unwrap_or` branch is never taken. The subsequent `.try_into::<u64>()` succeeds silently because `i64::MAX` is representable as `u64`. The only post-resolution check is `if limit == 0` at [3](#0-2)  which does not constrain large values. The uncapped `limit` is then cast back to `i64` and injected into the SQLite `LIMIT` clause at [4](#0-3)  SQLite treats `i64::MAX` as an effectively unbounded limit, returning all matching rows. These are deserialized into a `Vec<BlockTransaction>` on the heap with no size guard.

## Impact Explanation
On a Rosetta node with a large number of indexed blocks (e.g., 100k+), a single request causes the process to allocate memory proportional to the full matching block set. This can exhaust available memory, crashing or severely degrading the Rosetta API process. This matches the allowed High impact: *"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant... Rosetta... security impact with concrete user or protocol harm."* Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation
The Rosetta HTTP API is publicly reachable with no authentication required. The exploit requires a single crafted HTTP POST with one field set to `i64::MAX`. Any attacker aware of the Rosetta API specification can trigger this. The attack is repeatable and requires no special privileges, prior knowledge of the node state, or victim interaction.

## Recommendation
After resolving `limit`, enforce the cap unconditionally regardless of whether the caller supplied a value:

```rust
let limit: u64 = request
    .limit
    .unwrap_or(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST as i64)
    .try_into()
    .map_err(|err| {
        Error::request_processing_error(&format!("Limit has to be a valid u64: {err}"))
    })?;

// Enforce cap unconditionally
let limit = limit.min(MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST);
```

Alternatively, reject requests that exceed the maximum with an explicit error.

## Proof of Concept
Send the following to a Rosetta instance with at least one indexed block:

```json
POST /search/transactions
{
  "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
  "limit": 9223372036854775807
}
```

The handler will pass `i64::MAX` as the SQLite `LIMIT`, returning all rows in the `blocks` table. On a node with 100k+ blocks, the process will allocate memory proportional to the full block set, causing OOM or severe latency. A bounded response or an explicit error is the expected correct behavior. This can be verified with a local integration test by populating the SQLite store with a large number of blocks and confirming the response count exceeds `MAX_TRANSACTIONS_PER_SEARCH_TRANSACTIONS_REQUEST`.
