### Title
EIP-155 Not Enforced for Legacy Transactions Allows Cross-Chain Replay Attacks - (File: engine-transactions/src/legacy.rs)

### Summary

Aurora Engine accepts legacy Ethereum transactions with `v = 27` or `v = 28` (pre-EIP-155 signatures) without enforcing chain ID validation. Because the chain ID check in `engine/src/engine.rs` is conditional on `transaction.chain_id` being `Some`, a pre-EIP-155 legacy transaction signed on Ethereum mainnet (or any other EVM network) can be replayed on Aurora to steal funds from users who have balances on Aurora.

### Finding Description

In `engine-transactions/src/legacy.rs`, the `sender()` function decodes the `v` field of a legacy signed transaction. When `v` is 27 or 28, it sets `chain_id = None` and signs the message hash without including the chain ID — the pre-EIP-155 behavior:

```rust
// engine-transactions/src/legacy.rs, lines 68-78
let (chain_id, rec_id) = match self.v {
    0..=26 | 29..=34 => return Err(Error::InvalidV),
    27..=28 => (
        None,   // <-- No chain ID: pre-EIP-155 path
        u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
    ),
    _ => (
        Some((self.v - 35) / 2),
        u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
    ),
};
```

The `chain_id()` method also returns `None` for `v ∈ {27, 28}`:

```rust
// engine-transactions/src/legacy.rs, lines 88-93
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
```

When a `Legacy` transaction is normalized in `engine-transactions/src/lib.rs`, the `chain_id` field is set directly from `tx.chain_id()`, which is `None` for pre-EIP-155 transactions:

```rust
// engine-transactions/src/lib.rs, lines 106-118
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for v=27/28
    ...
}
```

Finally, in `engine/src/engine.rs`, the chain ID validation is gated on `chain_id` being `Some`:

```rust
// engine/src/engine.rs, lines 1054-1059
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

When `chain_id` is `None`, the entire check is skipped. The transaction proceeds to execution with no chain binding.

### Impact Explanation

**Critical — Direct theft of user funds.**

An attacker who finds a pre-EIP-155 legacy transaction (v=27 or v=28) that was previously broadcast on Ethereum mainnet or another EVM network can replay it verbatim on Aurora. If the original signer has a balance on Aurora, the replayed transaction executes with the same sender, nonce, recipient, and value — transferring the victim's Aurora ETH to the original recipient (who may be the attacker, or whose private key the attacker controls). The nonce check (`check_nonce`) does not prevent this because the Aurora nonce for that account may be at the same value as when the original transaction was sent.

### Likelihood Explanation

Pre-EIP-155 transactions are permanently recorded on Ethereum mainnet (before block ~2,675,000, circa 2016). Any Aurora user who:
1. Holds ETH on Aurora, and
2. Previously sent a pre-EIP-155 transaction on Ethereum mainnet (v=27 or v=28)

is vulnerable. An attacker only needs to scan Ethereum mainnet history for such transactions and replay them via Aurora's public `submit` endpoint. No special privileges are required — any NEAR account can call `submit`.

### Recommendation

Reject legacy transactions that do not include a chain ID (i.e., where `v = 27` or `v = 28`) in `submit_with_alt_modexp`. Add an explicit check after normalization:

```rust
if transaction.chain_id.is_none() {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

Alternatively, enforce this at the `LegacyEthSignedTransaction` parsing level by returning `Error::InvalidV` for `v = 27` or `v = 28`, making pre-EIP-155 transactions entirely unacceptable.

### Proof of Concept

1. Find any pre-EIP-155 Ethereum mainnet transaction where `v = 27` or `v = 28` (e.g., from Etherscan, filtering transactions before block 2,675,000).
2. Confirm the original sender has a balance on Aurora.
3. Submit the raw RLP-encoded transaction bytes directly to Aurora's `submit` NEAR method.
4. The engine parses the transaction: `v = 27` → `chain_id = None` in `LegacyEthSignedTransaction::sender()`.
5. `NormalizedEthTransaction::chain_id` is `None`.
6. The chain ID check at `engine/src/engine.rs:1055` is skipped (`if let Some(chain_id)` does not match).
7. The nonce check passes if the Aurora account nonce matches.
8. The transaction executes, transferring the victim's Aurora ETH to the original recipient.
