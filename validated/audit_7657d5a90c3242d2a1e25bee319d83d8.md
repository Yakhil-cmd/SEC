### Title
ckETH Gas Fee Burned Upfront Not Reimbursed When ckERC20 Withdrawal Ethereum Transaction Fails - (File: rs/ethereum/cketh/minter/src/state/transactions/mod.rs)

### Summary
When a user calls `withdraw_erc20` on the ckETH minter, ckETH is burned upfront to cover the Ethereum gas fee before the transaction is submitted. If the resulting Ethereum transaction fails (e.g., the ERC20 contract reverts), the ckERC20 withdrawal amount is reimbursed to the user, but the ckETH gas fee is permanently lost. This is a direct analog to the external report: the "fee" (ckETH gas) is committed to an irrecoverable state before the operation completes, while the "main amount" (ckERC20 tokens) is returned on failure.

### Finding Description
In `withdraw_erc20` (`rs/ethereum/cketh/minter/src/main.rs`), the minter first burns `erc20_tx_fee` ckETH from the user's account to pay for Ethereum gas, then burns `ckerc20_withdrawal_amount` ckERC20 tokens. Both burns happen before the Ethereum transaction is submitted. [1](#0-0) 

When the Ethereum transaction is later finalized, `record_finalized_transaction` in `EthTransactions` handles the failure case. For `CkErc20` requests, it creates a reimbursement only for `request.withdrawal_amount` (the ckERC20 tokens), with no reimbursement for `request.max_transaction_fee` (the ckETH gas fee): [2](#0-1) 

Contrast this with the `CkEth` failure path, which reimburses `finalized_tx.transaction_amount()` — the ETH value that was not consumed: [3](#0-2) 

The asymmetry is confirmed by the test `should_reimburse_tokens_when_ckerc20_withdrawal_fails`, which asserts that only `withdrawal_request.withdrawal_amount` (ckERC20) is in the reimbursement request, with no ckETH entry: [4](#0-3) 

The documentation explicitly acknowledges this: *"Overcharged transaction fees are not reimbursed."* [5](#0-4) 

The minter's `update_balance_upon_withdrawal` tracks the unspent fee (`max_transaction_fee - effective_tx_fee`) as `total_unspent_tx_fees` in its internal accounting, confirming the difference is retained by the minter and never returned: [6](#0-5) 

### Impact Explanation
Any user whose ckERC20 Ethereum withdrawal transaction fails loses the entire `max_transaction_fee` ckETH, even though the Ethereum network only consumed `effective_tx_fee` ≤ `max_transaction_fee`. The overcharged portion (`max_transaction_fee - effective_tx_fee`) is retained by the minter's ETH balance and never minted back to the user. Given that `max_transaction_fee` for ckERC20 withdrawals is estimated with a safety margin (to allow for resubmissions at higher gas prices), the overcharge can be substantial. This is a ledger conservation bug: ckETH is burned in excess of what is actually consumed on Ethereum, with no mechanism to return the surplus to the user on transaction failure.

### Likelihood Explanation
Any ckERC20 Ethereum transaction that results in `TransactionStatus::Failure` triggers this path. This occurs when the ERC20 contract reverts (e.g., due to a transfer restriction, paused contract, or insufficient on-chain allowance at execution time), or when the transaction is submitted but fails for other Ethereum-level reasons. This is reachable by any unprivileged user calling `withdraw_erc20` — no special role is required. The scenario is realistic given that ERC20 contracts can have transfer restrictions that are not detectable at withdrawal request time.

### Recommendation
When a ckERC20 Ethereum transaction fails, compute the unused gas fee (`max_transaction_fee - effective_tx_fee`) and schedule a ckETH reimbursement for that amount in addition to the ckERC20 token reimbursement. This mirrors how ckETH withdrawals handle failure: the user is charged only for what was actually consumed. The `update_balance_upon_withdrawal` function already computes `unspent_tx_fee` correctly; this value should be used to mint ckETH back to the user via a `ReimbursedEthWithdrawal` event alongside the `ReimbursedErc20Withdrawal` event.

### Proof of Concept
1. User calls `withdraw_erc20` with `max_transaction_fee = 10_000_000 wei` ckETH and `withdrawal_amount = 100 USDC` ckUSDC.
2. Minter burns `10_000_000 wei` ckETH and `100 USDC` ckUSDC.
3. Ethereum transaction is submitted; the ERC20 contract reverts (e.g., transfer is paused).
4. `record_finalized_transaction` is called with `TransactionStatus::Failure`; `effective_tx_fee = 4_000_000 wei`.
5. A reimbursement request is created for `100 USDC` ckUSDC only.
6. `process_reimbursement` mints `100 USDC` ckUSDC back to the user.
7. The `10_000_000 wei` ckETH is never returned. The minter's `total_unspent_tx_fees` increases by `6_000_000 wei` (`max_transaction_fee - effective_tx_fee`), retained permanently. [7](#0-6)

        fn should_reimburse_tokens_when_ckerc20_withdrawal_fails() {
            let mut transactions = EthTransactions::new(TransactionNonce::ZERO);
            let cketh_ledger_burn_index = LedgerBurnIndex::new(7);
            let ckerc20_ledger_burn_index = LedgerBurnIndex::new(7);
            let withdrawal_request = ckerc20_withdrawal_request_with_index(
                cketh_ledger_burn_index,
                ckerc20_ledger_burn_index,
            );
            transactions.record_withdrawal_request(withdrawal_request.clone());
            let created_tx = create_and_record_transaction(
                &mut transactions,
                withdrawal_request.clone(),
                gas_fee_estimate(),
            );
            let signed_tx = create_and_record_signed_transaction(&mut transactions, created_tx);
            let receipt = TransactionReceipt {
                gas_used: GasAmount::from(40_000_u32),
                effective_gas_price: WeiPerGas::from(100_u16),
                ..transaction_receipt(&signed_tx, TransactionStatus::Failure)
            };
            assert_eq!(
                receipt.effective_transaction_fee(),
                Wei::from(4_000_000_u32)
            );
            transactions.record_finalized_transaction(cketh_ledger_burn_index, receipt.clone());
            let expected_ckerc20_reimbursed_amount = withdrawal_request.withdrawal_amount;

            assert_eq!(transactions.maybe_reimburse, btreeset! {});

                transactions.reimbursement_requests,
                btreemap! {
                    ReimbursementIndex::CkErc20 {
                        cketh_ledger_burn_index,
                        ledger_id: withdrawal_request.ckerc20_ledger_id,
                        ckerc20_ledger_burn_index } =>
                    ReimbursementRequest {
                        ledger_burn_index: cketh_ledger_burn_index,
                        reimbursed_amount: expected_ckerc20_reimbursed_amount.change_units(),
                        to: withdrawal_request.from,
                        to_subaccount: withdrawal_request.from_subaccount,
                        transaction_hash: Some(receipt.transaction_hash),
