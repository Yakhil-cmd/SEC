### Title
Secp256k1 Private Key Passed as Plaintext CLI Argument Enables Rate-Limit Canister Impersonation - (File: rs/boundary_node/rate_limits/canister_client/src/main.rs)

### Summary
The `rate-limiting-canister-client` binary accepts a raw Secp256k1 private key (SEC1 PEM format) as a plaintext `--identity-key` command-line string argument. This key is the sole credential authorizing calls to the on-chain rate-limiting canister's `add_config` update method. Any local user or log-aggregation system with read access to the operator's shell history file can extract the key and impersonate the authorized principal to inject arbitrary rate-limit rules, causing targeted denial-of-service against any canister on the IC.

### Finding Description
In `rs/boundary_node/rate_limits/canister_client/src/main.rs`, the `Cli` struct declares `identity_key` as a plain `Option<String>` clap argument:

```rust
/// Identity key
#[arg(long)]
identity_key: Option<String>,
``` [1](#0-0) 

The README documents the intended invocation as:

```
./rate-limiting-canister-client \
    --config-file=ratelimits.yml \
    --canister-id=pawub-syaaa-aaaam-qb7zq-cai \
    --identity-key="XYZ"
``` [2](#0-1) 

The string value is consumed directly as a SEC1 PEM private key and used to construct a `Secp256k1Identity` that signs the `add_config` update call:

```rust
let identity = Secp256k1Identity::from_private_key(
    SecretKey::from_sec1_pem(&identity_key).context("failed to parse the identity key")?,
);
``` [3](#0-2) 

Because the full PEM block is passed on the command line, it is recorded verbatim in the shell history file (e.g., `~/.bash_history`, `~/.zsh_history`), in `/proc/<pid>/cmdline`, and in process-listing output (`ps aux`).

### Impact Explanation
The rate-limiting canister enforces an `AccessLevel::FullAccess` check keyed to a single `authorized_principal` set at install time:

```rust
if self.access_resolver.get_access_level() == AccessLevel::FullAccess {
    return self.inner.add_config(input_config, time);
}
Err(AddConfigError::Unauthorized)
``` [4](#0-3) 

An attacker who recovers the private key from shell history can call `add_config` as the authorized principal and push arbitrary rate-limit rules. API boundary nodes fetch and enforce these rules:

```rust
let rule = RateLimitRule::from_bytes_yaml(&raw)
    .context(format!("unable to decode raw rule with id {}", x.rule_id))?;
``` [5](#0-4) 

A malicious rule can set `Action::Block` for any canister ID, including critical NNS canisters, causing a targeted denial-of-service for all users reaching the IC through API boundary nodes. The `submit_config` function calls the `add_config` update method directly:

```rust
let result = agent
    .update(&canister_id, "add_config")
    .with_arg(args)
    .call_and_wait()
    .await
``` [6](#0-5) 

### Likelihood Explanation
The operator running this tool is a privileged DFINITY employee. Shell history files are world-readable by default on many Linux configurations, accessible to co-tenants on shared build machines, captured by CI/CD log systems, and visible in audit logs. The key is also visible in `ps aux` output during the brief window the process runs. Likelihood is **Low-Medium**: it requires local access to the operator's environment, but that environment is a shared build/ops machine in practice.

### Recommendation
- Accept the identity key via a file path argument (e.g., `--identity-key-file <path>`) so the secret never appears on the command line or in history.
- Alternatively, read the key from an environment variable or from stdin with echo disabled, following the pattern already used elsewhere in the repo:

```bash
read -s DFX_HSM_PIN
export DFX_HSM_PIN
``` [7](#0-6) 

- Apply `#[arg(hide_env_values = true)]` if an env-var approach is used with clap, to suppress the value from `--help` output.

### Proof of Concept
1. Operator invokes the tool as documented:
   ```
   ./rate-limiting-canister-client \
       --config-file=ratelimits.yml \
       --canister-id=pawub-syaaa-aaaam-qb7zq-cai \
       --identity-key="-----BEGIN EC PRIVATE KEY-----\nMHQCAQEE..."
   ```
2. Shell records the full command, including the PEM block, in `~/.bash_history`.
3. Attacker with read access to the history file (co-tenant, log aggregator, or compromised CI) extracts the PEM string.
4. Attacker constructs a `Secp256k1Identity` from the recovered key and calls `add_config` on the rate-limiting canister with a rule setting `Action::Block` for the NNS governance canister ID.
5. All API boundary nodes fetch the new rule and begin blocking ingress to the governance canister, preventing NNS proposals and votes until the authorized principal pushes a corrective config.
