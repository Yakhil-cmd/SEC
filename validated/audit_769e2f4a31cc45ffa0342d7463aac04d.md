### Title
Unbounded Iteration in `move_to_block` Causes Permanent Fund Freeze After Engine Idle Period — (`engine-hashchain/src/hashchain.rs`)

---

### Summary

The `Hashchain::move_to_block` function iterates one-by-one through every NEAR block that elapsed since the last Aurora transaction, performing a keccak hash per block. This function is called unconditionally on every EVM transaction submission and every token bridging operation. If the Aurora Engine is idle for a sufficiently long period, the loop exceeds NEAR's 300 TGas per-transaction gas limit, causing every subsequent transaction to fail and permanently freezing all funds held in the engine.

---

### Finding Description

`Hashchain::move_to_block` in `engine-hashchain/src/hashchain.rs` contains an unbounded `while` loop:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
``` [1](#0-0) 

Each iteration calls `compute_block_hashchain`, which concatenates several byte arrays and computes a keccak hash. [2](#0-1) 

This loop is triggered from `load_hashchain` whenever the stored block height lags behind the current NEAR block height:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
``` [3](#0-2) 

`load_hashchain` is called from both `with_hashchain` and `with_logs_hashchain`, which wrap every critical Aurora operation:

- **EVM transaction submission** (`submit`, `submit_with_args`) via `with_logs_hashchain`
- **Token bridging** (`ft_on_transfer`) via `with_hashchain`
- **ERC-20 deployment** (`deploy_erc20_token`, `deploy_erc20_token_callback`) via `with_hashchain`
- **Token mirroring** (`mirror_erc20_token_callback`) via `with_hashchain` [4](#0-3) 

NEAR produces approximately one block per second. After just ~5 minutes of engine inactivity (~300 blocks), the loop already performs 300 keccak hashes. After one day (~86,400 blocks), the loop is orders of magnitude larger. Because the hashchain state is only updated on a *successful* transaction, and every transaction fails once the gas limit is exceeded, the engine becomes permanently stuck: no transaction can ever succeed again, because each attempt re-enters the same loop from the same stale block height.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

All ETH and ERC-20 tokens bridged into the Aurora Engine (held as NEAR contract state) become permanently inaccessible. No EVM transaction can be submitted, and no token bridge operation (`ft_on_transfer`, `withdraw`) can complete. The hashchain state is never advanced because the transaction that would advance it always runs out of gas before completing. This is a self-reinforcing deadlock with no on-chain recovery path.

---

### Likelihood Explanation

**Medium.** NEAR produces blocks continuously regardless of Aurora activity. Any realistic idle period — market downturns, low-usage deployments, scheduled maintenance windows, or simply a new deployment that sees low initial traffic — creates the precondition. The threshold for triggering the bug depends on the per-iteration NEAR gas cost of `compute_block_hashchain`, but given NEAR's 300 TGas limit and the cost of keccak hashing, even a few hours of inactivity may be sufficient. No attacker action is required; the passage of time alone is the trigger.

---

### Recommendation

Replace the per-block iteration with an O(1) computation. The hashchain does not need to hash every empty intermediate block individually. Instead:

1. Store only the last-seen block height and the cumulative hash.
2. When advancing multiple blocks, compute a single hash that encodes the span (e.g., hash over `(previous_hash, start_height, end_height)` for the empty-block range), rather than iterating one block at a time.
3. Alternatively, cap the number of blocks processed per call and allow the hashchain to be advanced incrementally via a dedicated admin method, separate from the critical transaction path.

---

### Proof of Concept

1. Deploy Aurora Engine on NEAR with hashchain initialized.
2. Submit one successful EVM transaction (hashchain records block height `H`).
3. Allow the engine to be idle for `N` NEAR blocks (e.g., `N = 10,000`, approximately 2.8 hours).
4. Submit any EVM transaction or call `ft_on_transfer`.
5. `load_hashchain` is called; `block_height = H + N > hashchain.current_block_height = H`, so `move_to_block(H + N)` is invoked.
6. The `while` loop runs `N` iterations, each computing a keccak hash.
7. NEAR gas limit (300 TGas) is exhausted mid-loop; the NEAR transaction aborts.
8. The hashchain state on-chain is unchanged (still at height `H`).
9. Every subsequent transaction repeats steps 4–8 and also fails.
10. All funds in the engine are permanently frozen. [5](#0-4) [6](#0-5)

    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,

    let cache = RefCell::new(IOCache::default());
