### Title
Single-Step Ownership Transfer Permanently Locks All Owner-Gated Administrative Functions - (`engine/src/contract_methods/admin.rs`)

---

### Summary

The `set_owner` function in the Aurora Engine performs an immediate, single-step ownership transfer with no confirmation from the new owner. If the current owner supplies an incorrect, mistyped, or inaccessible NEAR `AccountId`, ownership is irrecoverably transferred to an account that cannot sign transactions, permanently locking every function guarded by `require_owner_only` — including `upgrade`, `pause_contract`, `resume_contract`, and `attach_full_access_key`.

---

### Finding Description

`set_owner` reads a `SetOwnerArgs` struct containing `new_owner: AccountId` and immediately overwrites `state.owner_id` in a single atomic step:

```rust
// engine/src/contract_methods/admin.rs, lines 104-121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, unconfirmed write
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [1](#0-0) 

There is no `pending_owner` staging step and no requirement that the new account confirm acceptance. The only guard is `require_owner_only`, which checks `state.owner_id != predecessor_account_id`: [2](#0-1) 

Once `state.owner_id` is written with an inaccessible account, `require_owner_only` will permanently reject every subsequent privileged call.

---

### Impact Explanation

Every function gated by `require_owner_only` or `require_owner_and_running` becomes permanently inaccessible:

| Function | Effect of loss |
|---|---|
| `upgrade` | Contract code can never be patched |
| `stage_upgrade` | No new WASM can be staged |
| `pause_contract` | Emergency halt impossible |
| `resume_contract` | Contract permanently stuck if paused |
| `resume_precompiles` | Paused precompiles can never be re-enabled |
| `attach_full_access_key` | No new full-access key can be added |
| `set_key_manager` | Relayer key management frozen |
| `set_erc20_fallback_address` / `set_silo_params` | Silo configuration frozen | [3](#0-2) [4](#0-3) 

The most critical consequence is that `upgrade` is permanently blocked. Aurora Engine is a live EVM on NEAR that holds user ETH and ERC-20 balances. If a fund-draining vulnerability is discovered after ownership is lost, the contract cannot be patched or paused, leading to **permanent freezing or theft of all user funds** with no recovery path.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The current owner is a human-operated NEAR account. NEAR account IDs are arbitrary UTF-8 strings (e.g., `aurora.near`, `aurora-v2.near`). A single character typo (e.g., `auroa.near`) produces a syntactically valid `AccountId` that passes Borsh deserialization and the `ERR_SAME_OWNER` check, but corresponds to an account with no known keys. The operation is irreversible the moment the transaction is finalized. This is a realistic operational error during a routine ownership handoff (e.g., multisig migration, DAO takeover). [5](#0-4) 

---

### Recommendation

Implement a two-step ownership transfer:

1. **Step 1 — `set_pending_owner(new_owner)`**: The current owner nominates a `pending_owner` stored in `EngineState`. This is correctable if the wrong address is used.
2. **Step 2 — `accept_ownership()`**: The `pending_owner` account calls this function. `require_owner_only` checks `predecessor == pending_owner`, then promotes `pending_owner` to `owner_id` and clears the pending slot.

This ensures ownership only transfers to an account that demonstrably controls its private key. [1](#0-0) 

---

### Proof of Concept

1. Current owner (`aurora.near`) calls `set_owner` with `new_owner = "auroa.near"` (typo).
2. `require_owner_only` passes (caller is current owner).
3. `ERR_SAME_OWNER` check passes (`aurora.near != auroa.near`).
4. `state.owner_id` is written as `auroa.near` and persisted via `state::set_state`.
5. `auroa.near` does not exist / has no known keys.
6. All subsequent calls to `upgrade`, `pause_contract`, `resume_contract`, `attach_full_access_key`, etc. fail with `ERR_NOT_ALLOWED`.
7. The contract can never be upgraded or paused. Any latent fund-draining bug becomes permanently unexploitable-to-fix, resulting in permanent loss of all bridged user funds. [6](#0-5) [2](#0-1)
