Audit Report

## Title
Unbounded Recursive CBOR Deserialization Causes Stack Overflow and Process Abort on `/_/catch_up_package` — (`rs/http_endpoints/public/src/catch_up_package.rs`)

## Summary

The unauthenticated `POST /_/catch_up_package` handler calls `serde_cbor::from_slice::<CatchUpPackageParam>(&body)` directly on attacker-controlled bytes with no recursion depth guard. Because `CatchUpPackageParam` lacks `#[serde(deny_unknown_fields)]`, unknown CBOR map keys are silently skipped via `serde::de::IgnoredAny`, which recurses one Rust stack frame per nesting level. A maximally nested CBOR payload within the 5 MB body limit produces enough recursion to exhaust the Tokio worker thread's stack, triggering Rust's stack overflow handler which calls `abort()` and terminates the replica process.

## Finding Description

**Root cause:** `serde_cbor` (version 0.11.x, now deprecated) performs recursive-descent parsing with no built-in depth limit. When deserializing a struct with unknown fields, serde's derived `Deserialize` calls `deserializer.deserialize_any(IgnoredAnyVisitor)` for each unknown value. For a CBOR map value, this calls `visit_map`, which calls `next_value::<IgnoredAny>`, which calls `deserialize_any` again — one Rust call frame per nesting level, unbounded.

**Confirmed code path:**

- `catch_up_package.rs` line 71: `serde_cbor::from_slice::<CatchUpPackageParam>(&body)` — no depth guard, no `spawn_blocking`, called synchronously on the Tokio worker thread. [1](#0-0) 

- `CatchUpPackageParam` is a flat two-field struct with `#[derive(Deserialize)]` and no `#[serde(deny_unknown_fields)]`, so any unknown CBOR key triggers the `IgnoredAny` recursion path. [2](#0-1) 

- The body size limit is 5 MB (`max_request_size_bytes: 5 * 1024 * 1024`), which is the attacker's budget. [3](#0-2) 

**Why existing mitigations fail:**

- **`RequestBodyLimitLayer` (5 MB):** Admits the payload — this is the attacker's attack surface, not a defense against recursion depth. [4](#0-3) 

- **300-second `timeout`:** Wraps the async future but cannot preempt a synchronous blocking call. The stack overflow occurs in microseconds. [5](#0-4) 

- **`max_catch_up_package_concurrent_requests: 100`:** Limits simultaneous in-flight requests but does not prevent a single request from overflowing the stack. [6](#0-5) 

- **`verify_cbor_content_header` middleware:** Only checks `Content-Type: application/cbor`; passes any conforming payload. [7](#0-6) 

**Stack overflow mechanics:** A CBOR one-element map (`0xA1`) with a 1-byte key and a nested map as value costs ~3 bytes per level. At 5 MB, an attacker can produce ~1.7 M nesting levels. At even a conservative 50 bytes per Rust stack frame, this requires ~85 MB of stack — far exceeding the OS default of 8 MB for Tokio worker threads. Rust's stack overflow handler (installed via `sigaltstack`) catches the resulting SIGSEGV and calls `abort()`, terminating the entire replica process — not just the task.

## Impact Explanation

Process abort terminates the replica. The orchestrator restarts it, but the attacker can immediately re-send the payload, keeping the replica in a crash-restart loop. Targeting multiple replicas in a subnet simultaneously can reduce live replicas below the consensus fault threshold, halting the subnet. This matches the allowed impact: **Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS — High ($2,000–$10,000).**

## Likelihood Explanation

The endpoint is unauthenticated and publicly reachable on port 8080. The payload is trivial to construct (a CBOR map nested inside itself, repeated to fill 5 MB). No special privileges, cryptographic keys, network position, or victim interaction are required. The attack is repeatable and cheap. A single attacker can target all replicas in a subnet sequentially or in parallel.

## Recommendation

1. **Add `#[serde(deny_unknown_fields)]` to `CatchUpPackageParam`.** This causes `serde_cbor` to return an error immediately on the first unknown key rather than recursively skipping its value, eliminating the recursion path entirely for unexpected inputs. This is the lowest-effort, highest-impact fix. [2](#0-1) 

2. **Add a CBOR recursion depth pre-scan before deserialization.** Reject any payload whose CBOR nesting depth exceeds a small fixed limit (e.g., 10) before passing it to `serde_cbor`. The `ciborium` crate (the recommended replacement for `serde_cbor`) supports configurable recursion limits natively.

3. **Move deserialization into `tokio::task::spawn_blocking`.** This isolates the blocking call to a dedicated thread pool with a controlled stack size and prevents starvation of the async executor, though it does not prevent the crash itself without one of the above depth guards.

## Proof of Concept

```python
import requests

# Build a 5 MB deeply nested CBOR one-element map:
# 0xA1 = map(1), 0x61 0x61 = text("a"), value = next level
level = b'\x00'  # innermost: integer 0
while len(level) < 5 * 1024 * 1024:
    level = b'\xa1\x61\x61' + level  # map(1) {"a": <level>}

requests.post(
    "http://<replica-ip>:8080/_/catch_up_package",
    data=level[:5 * 1024 * 1024],
    headers={"Content-Type": "application/cbor"},
    timeout=30,
)
# Expected: replica process aborts (stack overflow → abort()), connection reset
```

A local integration test can be written using `ic-http-endpoints-public`'s existing test harness (`CatchUpPackageService::new_router` with a `MockConsensusPoolCache`) by sending this payload via `tower::ServiceExt::oneshot` and asserting the process does not abort — or by running it in a subprocess and observing the exit code.
