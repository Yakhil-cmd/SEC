Audit Report

## Title
Unauthenticated Caller Can Abort Any In-Progress SNS Upgrade via `fail_stuck_upgrade_in_progress` - (File: rs/sns/governance/canister/canister.rs)

## Summary
The SNS governance canister exposes `fail_stuck_upgrade_in_progress` as a public `#[update]` method with no caller authorization check at either the canister entry point or the underlying governance implementation. Any anonymous ingress sender can invoke it after the 5-minute upgrade deadline to forcibly abort an adopted-but-executing SNS upgrade proposal, marking it as `Failed` and clearing the `pending_version` lock. An attacker can repeat this indefinitely to prevent any SNS from upgrading its canisters.

## Finding Description
The canister entry point at `rs/sns/governance/canister/canister.rs` lines 526–535 registers the function as a public update method with no `caller()` check:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(
        sns_gov_pb::FailStuckUpgradeInProgressRequest::from(request),
    ))
}
```

The underlying implementation at `rs/sns/governance/src/governance.rs` lines 6327–6361 also performs no authorization check. It accepts the request from any caller, checks only whether `now > pending_version.mark_failed_at_seconds`, and if so calls `complete_sns_upgrade_to_next_version` to mark the associated proposal as `Failed` with `ErrorType::External` and clear `proto.pending_version`. The incomplete authorization design is acknowledged by the developer comment at line 6337: `// Maybe, we should look at the checking_upgrade_lock field and only proceed if it is false, or the request has force set to true.`

When an `UpgradeSnsToNextVersion` or `AdvanceSnsTargetVersion` proposal is adopted and execution begins, `pending_version` is set with `mark_failed_at_seconds = now + 5 * 60`. After those 5 minutes elapse, the exploit window opens unconditionally to any caller. The DID interface at `rs/sns/governance/canister/governance.did` line 1050 confirms the method is publicly listed: `fail_stuck_upgrade_in_progress : (record {}) -> (record {});`.

The unit tests in `rs/sns/governance/src/governance/fail_stuck_upgrade_in_progress_tests.rs` lines 216–299 confirm the exact behavior: calling the function with `env.now = UPGRADE_DEADLINE_TIMESTAMP_SECONDS + 1` clears `pending_version` and sets the proposal to `Failed` with an error message containing `"manually aborted"`.

## Impact Explanation
This is a governance liveness attack against SNS. An attacker who monitors SNS governance state can call `fail_stuck_upgrade_in_progress` immediately after the 5-minute deadline on every SNS upgrade attempt. Each call irreversibly marks the adopted upgrade proposal as `Failed`, clears `pending_version`, and forces the SNS community to re-submit and re-vote on a new upgrade proposal. The attacker can repeat this indefinitely, permanently blocking all SNS canister upgrades with zero cost and no required stake or neuron. This matches the allowed High impact: **Significant SNS security impact with concrete user or protocol harm** — specifically, systematic denial of the SNS upgrade governance mechanism.

## Likelihood Explanation
The function is unconditionally reachable via any ingress message to the SNS governance canister. No neuron, stake, or privileged role is required. The 5-minute window is predictable and observable on-chain via `get_running_sns_version` or `get_upgrade_journal`. An attacker can automate this with a simple polling loop that watches for `pending_version` to appear and then calls the function after `mark_failed_at_seconds` elapses. The attack is repeatable at negligible cost (only cycles for the ingress call).

## Recommendation
Add a caller authorization check to `fail_stuck_upgrade_in_progress` at the canister entry point in `rs/sns/governance/canister/canister.rs`. Acceptable callers should be restricted to at least one of:
- The SNS root canister principal
- Any neuron holder (requiring a valid neuron subaccount, enforcing skin-in-the-game)
- A governance proposal action (making abort itself a voted action)

At minimum, pass `caller()` into the governance method and validate it against a whitelist before delegating to the abort logic. The developer comment at line 6337 already signals that the precondition design was left incomplete and should be revisited.

## Proof of Concept
1. Submit an `UpgradeSnsToNextVersion` proposal to any SNS governance canister and get it adopted.
2. Observe that `get_running_sns_version` returns a non-null `pending_version` with `mark_failed_at_seconds = T + 300`.
3. At time `T + 301`, send an ingress update call to `fail_stuck_upgrade_in_progress` from any anonymous principal (no neuron required).
4. Observe via `get_proposal` that the proposal transitions to `Failed` with `ErrorType::External` and `failure_reason.error_message` containing `"manually aborted"`, and that `pending_version` is cleared.
5. Repeat from step 1 to permanently block all SNS upgrades.

This is directly reproducible using the existing test harness in `rs/sns/governance/src/governance/fail_stuck_upgrade_in_progress_tests.rs` by adding a test that calls the function from a different principal identity and observing the same `Failed` outcome.
