### Title
Pre-EIP-155 Legacy Transaction Cross-Chain Replay Bypasses Chain ID Validation, Enabling Unauthorized Fund Transfer - (`engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

---

### Summary

The Aurora engine intentionally accepts pre-EIP-155 legacy transactions (v=27 or v=28) for EIP-1820 compatibility. However, the chain ID validation in `submit_with_alt_modexp` is gated on `if let Some(chain_id) = transaction.chain_id`, which means it is **entirely skipped** when `chain_id` is `None`. A pre-EIP-155 transaction signed on any EVM network (Ethereum mainnet, testnets, etc.) can be submitted to Aurora and executed as-is, provided the sender's Aurora nonce matches the transaction nonce.

---

### Finding Description

**Step 1 — Transaction parsing and normalization**

When raw bytes arrive at `submit`, they are parsed via `EthTransactionKind::try_from`. A legacy transaction (no typed-envelope prefix byte) is decoded as `EthTransactionKind::Legacy`. [1](#0-0) 

It is then normalized via `NormalizedEthTransaction::try_from`. For the `Legacy` arm, `chain_id` is set to `tx.chain_id()`: [2](#0-1) 

**Step 2 — `chain_id()` returns `None` for v=27/28**

`LegacyEthSignedTransaction::chain_id()` returns `None` for any `v` in `0..=34`, which covers both 27 and 28: [3](#0-2) 

`sender()` also correctly recovers the signer for v=27/28 by hashing only the 6-field RLP (no chain_id), matching the pre-EIP-155 signing convention used on Ethereum mainnet: [4](#0-3) 

**Step 3 — Chain ID validation is skipped when `chain_id` is `None`**

In `submit_with_alt_modexp`, the chain ID check uses an `if let Some(chain_id)` guard. When `transaction.chain_id` is `None`, the entire block is skipped and execution continues unconditionally: [5](#0-4) 

**Step 4 — Remaining guards do not prevent the attack**

- `assert_access` checks the silo whitelist. Whitelists are **disabled by default** (`!list.is_enabled()` → allow all), so on a standard Aurora deployment any address passes: [6](#0-5) 
- `check_nonce` requires the transaction nonce to match the sender's current Aurora nonce. This is the only real barrier, but it is trivially satisfied for fresh accounts (nonce=0).

**Step 5 — Intentional design confirmed**

The CHANGES.md explicitly documents that pre-EIP-155 transactions were re-enabled for EIP-1820 compatibility: [7](#0-6) 

---

### Impact Explanation

An attacker who obtains a pre-EIP-155 signed transaction from a victim (e.g., from Ethereum mainnet history, or by tricking the victim into signing with a non-EIP-155 wallet) can submit it to Aurora. If the victim's Aurora nonce matches the transaction nonce and the transaction transfers value to an attacker-controlled address, the victim's Aurora balance is drained without their consent. This constitutes **direct theft of user funds** (Critical).

---

### Likelihood Explanation

**Low-to-Medium.** The attack requires:
1. A valid pre-EIP-155 signed transaction from the victim (v=27 or v=28).
2. The victim's Aurora nonce must match the transaction nonce.

Condition 2 is trivially satisfied for accounts that have never transacted on Aurora (nonce=0). Condition 1 is satisfied by any transaction the victim signed before EIP-155 was widely adopted (2016–2017 era), or by tricking the victim into using a wallet/dApp that does not enforce EIP-155. Both conditions are realistic for users who have been in the Ethereum ecosystem since before EIP-155.

---

### Recommendation

After the `sender()` recovery succeeds, enforce that pre-EIP-155 transactions are only permitted for the specific EIP-1820 deployer address and nonce (the only legitimate use case), or reject all `chain_id == None` transactions that carry a nonzero `value` or non-EIP-1820 `data`. Alternatively, add an explicit check:

```rust
// Reject pre-EIP-155 transactions unless they are the EIP-1820 deployer tx
if transaction.chain_id.is_none() && !transaction.value.is_zero() {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

A stricter fix is to only allow `chain_id == None` for the exact EIP-1820 deployment transaction (known sender, nonce, and data), and reject all others.

### Proof of Concept

```rust
// Locally testable — no mainnet required
#[test]
fn test_pre_eip155_replay_on_aurora() {
    use engine_tests::utils;
    use aurora_engine_transactions::legacy::{LegacyEthSignedTransaction, TransactionLegacy};
    use aurora_engine_types::types::Wei;
    use aurora_engine_types::U256;

    let mut runner = utils::deploy_runner();

    // Victim key — same address on Ethereum mainnet and Aurora
    let victim_sk = libsecp256k1::SecretKey::parse_slice(
        &hex::decode("YOUR_TEST_KEY_HERE").unwrap()
    ).unwrap();
    let victim_addr = utils::address_from_secret_key(&victim_sk);
    let attacker_addr = utils::address_from_hex("0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef");

    // Fund victim on Aurora at nonce=0
    runner.create_address(victim_addr, Wei::new_u64(1_000_000_000), U256::zero());

    // Sign a pre-EIP-155 transaction (chain_id=None → v=27/28)
    // This simulates a tx the victim signed on Ethereum mainnet
    let tx = TransactionLegacy {
        nonce: U256::zero(),
        gas_price: U256::from(1u64),
        gas_limit: U256::from(21_000u64),
        to: Some(attacker_addr),
        value: Wei::new_u64(500_000_000),
        data: vec![],
    };
    let signed = utils::sign_transaction(tx, None /* no chain_id → pre-EIP-155 */, &victim_sk);
    assert_eq!(signed.v, 27u64.wrapping_add(/* rec_id */ 0)); // v=27 or 28

    // Attacker submits the mainnet-signed tx to Aurora
    let tx_bytes: Vec<u8> = rlp::encode(&signed).to_vec();
    let outcome = runner.call(utils::SUBMIT, "relay.aurora", tx_bytes).unwrap();
    let result = aurora_engine_types::parameters::engine::SubmitResult::try_from_slice(
        &outcome.return_data.as_value().unwrap()
    ).unwrap();

    // Engine accepts and executes the transaction
    assert!(result.status.is_ok(), "engine must accept pre-EIP-155 tx");

    // Victim's balance decreased without their Aurora consent
    assert!(runner.get_balance(victim_addr) < Wei::new_u64(1_000_000_000));
    // Attacker received the funds
    assert_eq!(runner.get_balance(attacker_addr), Wei::new_u64(500_000_000));
}
```

The test uses `utils::sign_transaction(tx, None, &sk)` which produces v=27 or v=28 (pre-EIP-155), exactly as the `sign_transaction` utility already supports: [8](#0-7)
