### Title
`native_per_pubdata` Silently Truncates to Zero via Integer Division When `pubdata_price < native_price`, Bypassing Pubdata Resource Accounting — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, the `native_per_pubdata` ratio is computed using integer floor division (`wrapping_div`). When the operator-set `pubdata_price` is less than `native_price`, the result silently truncates to zero. No zero-check is applied afterward. This causes all pubdata costs to be treated as free in native resources for the lifetime of the transaction, allowing any unprivileged user to generate pubdata (storage writes) without paying the intended native resource cost.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142-143
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `0`. The `.ok_or(...)` guard only catches the case where the value overflows `u64`; it does **not** catch the zero case. There is no subsequent `require!(native_per_pubdata > 0, ...)` guard.

This is structurally inconsistent with how `native_per_gas` is computed in the same function:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:135-138
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

`native_per_gas` uses `div_ceil`, which rounds up and guarantees a non-zero result when `gas_price > 0`. `native_per_pubdata` uses `wrapping_div`, which rounds down and can produce zero.

The zero value propagates through the entire resource accounting pipeline:

**1. `create_resources_for_tx` — intrinsic pubdata overhead becomes zero:**
```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0, no native deducted for intrinsic pubdata
```

**2. `get_resources_to_charge_for_pubdata` — execution pubdata cost becomes zero:**
```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // → N * 0 = 0
    .ok_or(out_of_native_resources!())?;
```

**3. `check_enough_resources_for_pubdata` — always returns `has_enough = true`:**
```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:454
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata = 0 native → always enough
```

The transaction body is never reverted for pubdata exhaustion, and the post-execution pubdata charge is zero regardless of how many storage slots were written.

---

### Impact Explanation

When `pubdata_price < native_price`, any L2 transaction can generate an arbitrary amount of pubdata (up to the block's `pubdata_limit`) without consuming any native resources for it. The native resource cost that the operator intended to charge per pubdata byte is silently zeroed out by integer truncation. Consequences:

- **Resource accounting divergence**: The operator sets `pubdata_price > 0` intending to charge for pubdata, but the system charges zero native resources per pubdata byte. This is a state-transition resource accounting bug.
- **Economic loss**: The operator/protocol does not collect the intended native resource payment for pubdata generation.
- **Block stuffing**: An attacker can fill blocks with storage-write-heavy transactions at a lower native resource cost than intended, degrading throughput for other users.
- **Forward/proving divergence risk**: The native resource accounting recorded on-chain diverges from the actual proving cost, since pubdata is a significant component of proving cost.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic in production. These two values are set by the operator in the block context and represent different units (pubdata byte cost vs. RISC-V cycle cost). For example, if `native_price = 1000` and `pubdata_price = 500`, the intended ratio is 0.5 native units per pubdata byte, but integer division yields 0. The default test configuration already demonstrates this: `pubdata_price = 0`, `native_price = 10`, confirming the system is designed to operate with `pubdata_price < native_price`. Any non-zero `pubdata_price` that is still less than `native_price` triggers the silent truncation. Once the condition holds, any unprivileged transaction sender can exploit it by writing to storage.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// In validate_and_compute_fee_for_transaction
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This ensures that any non-zero `pubdata_price` produces at least 1 native unit per pubdata byte, preventing silent truncation to zero. Additionally, consider adding an explicit check:

```rust
if pubdata_price > U256::ZERO && native_per_pubdata == 0 {
    return Err(TxError::Validation(InvalidTransaction::PubdataPriceTooLow));
}
```

### Proof of Concept

**Setup**: Operator sets `pubdata_price = 5`, `native_price = 10` in the block context.

**Step 1**: `native_per_pubdata = 5.wrapping_div(10) = 0` (integer truncation). [1](#0-0) 

**Step 2**: `create_resources_for_tx` is called with `native_per_pubdata_byte = 0`. The intrinsic pubdata overhead is `0 * L2_TX_INTRINSIC_PUBDATA = 0`. No native resources are deducted for intrinsic pubdata. [2](#0-1) 

**Step 3**: Attacker submits a transaction that writes to 100 storage slots (generating ~3200 bytes of pubdata). During execution, `check_enough_resources_for_pubdata` is called with `native_per_pubdata = 0`. `get_resources_to_charge_for_pubdata` returns `(3200, 0_native)`. `has_enough(0_native)` is always `true`. [3](#0-2) 

**Step 4**: In `before_refund`, `get_resources_to_charge_for_pubdata` again returns 0 native for all pubdata. `compute_gas_refund` charges 0 native for pubdata. The attacker's transaction completes with zero native resource cost for pubdata, despite the operator having set `pubdata_price = 5`. [4](#0-3) 

**Contrast with `native_per_gas`**: The same function uses `div_ceil` for `native_per_gas`, ensuring it is never silently zero when `gas_price > 0`. The inconsistency confirms this is an unintended truncation, not a deliberate design choice. [5](#0-4)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
