Audit Report

## Title
Unprivileged Caller Can Repeatedly Drain ckBTC Minter Cycles by Triggering Re-evaluation of Permanently-Quarantined UTXOs - (File: `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
The `update_balance` endpoint accepts an arbitrary `owner` field with no check that it matches the caller, allowing any non-anonymous principal to trigger re-evaluation of quarantined UTXOs belonging to any account. `processable_utxos_for_account` re-queues `Quarantined` UTXOs after 24 hours identically to `ValueTooSmall` UTXOs, causing `check_utxo` to call `check_transaction` on the Bitcoin checker canister at a cost of 40 billion cycles from the minter's own balance. Because the SDN list is immutable without an NNS upgrade, the result is always `Failed`, the UTXO is re-quarantined with a fresh timestamp, and the cycle drain repeats every 24 hours indefinitely.

## Finding Description

**No caller-owner binding.** `update_balance` in `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` constructs the target account as:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
``` [1](#0-0) 

There is no assertion that `args.owner` equals `caller`. The only access control is `check_anonymous_caller()` at the `#[update]` entry point. [2](#0-1) 

**Quarantined UTXOs re-queued after 24 hours.** `processable_utxos_for_account` in `rs/bitcoin/ckbtc/minter/src/state.rs` treats `Quarantined` and `ValueTooSmall` UTXOs identically: both are re-queued into `processable_utxos` once 24 hours have elapsed since `last_time_checked`. [3](#0-2) 

`ProcessableUtxos::iter()` chains `new_utxos`, `previously_ignored_utxos`, and `previously_quarantined_utxos`, so re-queued quarantined UTXOs are included in `satoshis_to_mint` (line 215), bypassing the early-return guard. [4](#0-3) 

**Cycle cost on every re-evaluation.** `check_utxo` first checks `checked_utxos` for a cache hit. Quarantined UTXOs are stored in `suspended_utxos`, not `checked_utxos`, so the cache always misses. The function then calls `check_transaction` with `CHECK_TRANSACTION_CYCLES_REQUIRED` (40 billion cycles) attached from the minter's balance, up to `MAX_CHECK_TRANSACTION_RETRY = 10` times. [5](#0-4) 

**Timestamp reset enables indefinite repetition.** When `check_transaction` returns `Failed`, `quarantine_utxo` is called with the current `now` timestamp. `SuspendedUtxos::insert` unconditionally updates `last_time_checked_cache` before checking whether the UTXO is already quarantined for the same reason, so the 24-hour cooldown resets even when no new event is emitted. [6](#0-5) [7](#0-6) 

## Impact Explanation

Each `update_balance` call targeting an account with N quarantined UTXOs aged past 24 hours drains at least `N × 40B` cycles (up to `N × 400B` if the checker returns `NotEnoughCycles` repeatedly) from the ckBTC minter canister. If the minter's cycle balance reaches zero, it stops executing — halting all ckBTC deposits and withdrawals for every user. This is a concrete, platform-level DoS of a core Chain Fusion / ck-token component, matching the **High** impact class: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."*

## Likelihood Explanation

The attack requires only a freely-obtainable non-anonymous IC principal and knowledge of accounts with quarantined UTXOs. The ckBTC minter's public dashboard exposes quarantined UTXOs directly. [8](#0-7) 

The attacker pays only ingress message fees; the minter pays 40B cycles per UTXO per day. The 24-hour cooldown limits rate per UTXO but not across the full set of quarantined UTXOs. As ckBTC adoption grows, the quarantined UTXO set grows, amplifying the drain rate.

## Recommendation

1. **Skip re-evaluation of `Quarantined` UTXOs** in `processable_utxos_for_account`. Unlike `ValueTooSmall` UTXOs (which can become processable if the minimum deposit amount changes), `Quarantined` UTXOs are permanently tainted until the SDN list changes via NNS upgrade and should not be re-evaluated on a user-triggered timer.
2. **Alternatively**, cache the `Tainted` result in `checked_utxos` so that `check_utxo` returns immediately without spending cycles on a repeat `check_transaction` call.
3. **Restrict `args.owner`** to require that the caller matches the owner, preventing third-party triggering of re-evaluation for arbitrary accounts.

## Proof of Concept

1. Identify an account `A` with a quarantined UTXO via the public ckBTC minter dashboard or `get_events` query.
2. Wait until the UTXO's `last_time_checked` timestamp is at least 24 hours old.
3. From any non-anonymous principal, call `update_balance(record { owner = opt A; subaccount = null })`.
4. The minter calls `check_transaction` on the Bitcoin checker canister, spending 40B cycles from the minter's balance. The checker returns `Failed` (tainted). `quarantine_utxo` is called with a fresh `now` timestamp, resetting the cooldown.
5. Repeat steps 2–4 every 24 hours. With K quarantined UTXOs across the system, an attacker drains at least `K × 40B` cycles per day from the minter at negligible personal cost.
