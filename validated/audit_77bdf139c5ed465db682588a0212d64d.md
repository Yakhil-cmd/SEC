### Title
Missing `r` Scalar Range Validation in EIP-7702 Authorization Signature Verification Allows Authority Forgery - (File: engine-transactions/src/eip_7702.rs)

### Summary

The `authorization_list()` method in `SignedTransaction7702` validates the `s` scalar and `parity` of each EIP-7702 authorization signature, but **never validates that `r` is in the range `(0, n)`** where `n` is the secp256k1 group order. This is an incomplete signature validation analogous to the external report's finding: just as the original bug omitted curve-point validation enabling birthday-attack forgery, this omission allows an attacker to craft authorization tuples with `r = 0` or `r >= n` that pass the pre-`ecrecover` validity gate and are forwarded to the NEAR host's `ecrecover`, which may recover an arbitrary address. A successfully forged authorization installs arbitrary EVM bytecode delegation (`EF0100 || attacker_contract`) on a victim EOA, enabling the attacker to drain that EOA's funds.

### Finding Description

In `engine-transactions/src/eip_7702.rs`, the `authorization_list()` method performs the following pre-`ecrecover` validation on each `AuthorizationTuple`:

```rust
let mut is_valid = if auth.s > SECP256K1N_HALF {
    false
} else {
    (auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
        && auth.parity <= U256::one()
};
```

The checks performed are:
- `s <= SECP256K1N_HALF` (EIP-2 low-s)
- `chain_id` matches
- `parity` is 0 or 1

**There is no check that `r > 0` and `r < n` (the secp256k1 group order `n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141`).**

The `SECP256K1N_HALF` constant defined in the file is only half the group order. The full group order `n` is never checked against `auth.r`. An attacker can supply `r = 0` or `r >= n` in an `AuthorizationTuple`. The `is_valid` flag remains `true` (assuming `s`, `chain_id`, and `parity` are valid), and `ecrecover` is called with the malformed `r`. The NEAR host `ecrecover` (or `libsecp256k1` in non-contract mode) may either return an error (setting `is_valid = false`) or — depending on the implementation's handling of out-of-range `r` — recover an address from a degenerate point, yielding an attacker-controlled `authority` address.

Critically, the EIP-7702 specification (step 3) requires `0 < r < n` and `0 < s < n`. The `s` bound is enforced (via `SECP256K1N_HALF`), but the `r` bound is entirely absent.

### Impact Explanation

EIP-7702 authorization tuples grant the `authority` address (recovered via `ecrecover`) a code delegation: the EVM writes `EF0100 || auth.address` as the code of the `authority` account. If an attacker can forge the `authority` to be a victim EOA address they do not control, the victim's account code is overwritten with a delegation to an attacker-controlled contract. On the next call to the victim's address, the attacker's contract code executes in the victim's context, allowing complete theft of the victim's ETH balance.

**Impact: Critical — direct theft of user funds.**

### Likelihood Explanation

Any unprivileged EVM user can submit an EIP-7702 transaction (type `0x04`) via the Aurora Engine's `submit` endpoint. The `authorization_list` is attacker-controlled calldata. The attacker needs to find an `r` value (zero or `>= n`) such that `ecrecover` returns a target victim address. With `r = 0`, the NEAR host `ecrecover` returns an error and `is_valid` is set to `false` — this specific case is safe. However, with `r >= n` but `r` still fitting in 32 bytes (e.g., `r = n`, `r = n+1`, ...), the behavior depends on the NEAR runtime's `ecrecover` host function, which performs its own modular reduction. If the NEAR host reduces `r mod n` before point recovery, then `r = n` maps to `r' = 0` (failure), but `r = n + k` for small `k` maps to `r' = k`, a valid small `r`. This means an attacker can choose any small target `r'` by submitting `r = n + r'`, bypassing the missing range check entirely and recovering an arbitrary address determined by the chosen `r'`, `s`, and `parity`. The attacker can brute-force `s` values to target a specific victim address, reducing the attack to a birthday-attack scenario analogous to the external report (approximately 2^80 operations for a 0.63 probability collision).

### Recommendation

Add an explicit check that `auth.r > U256::zero()` and `auth.r < SECP256K1N` (the full group order) in the `authorization_list()` validation block, mirroring the existing `s` check:

```rust
let mut is_valid = if auth.s > SECP256K1N_HALF || auth.r.is_zero() || auth.r >= SECP256K1N {
    false
} else {
    (auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
        && auth.parity <= U256::one()
};
```

Define `SECP256K1N` as the full group order constant alongside `SECP256K1N_HALF`.

### Proof of Concept

1. Attacker constructs an EIP-7702 `Transaction7702` targeting Aurora Engine's `submit` endpoint.
2. The `authorization_list` contains one `AuthorizationTuple` with:
   - `chain_id` = Aurora chain ID (valid)
   - `address` = attacker's malicious contract address
   - `nonce` = victim's current nonce
   - `parity` = 0 or 1 (valid)
   - `r` = `SECP256K1N + r'` for a chosen small `r'` (bypasses the missing range check; `is_valid` remains `true`)
   - `s` = brute-forced value such that `ecrecover(hash, parity, r', s)` returns the victim's address
3. The `authorization_list()` method passes `is_valid = true` and calls `ecrecover` with the out-of-range `r`.
4. The NEAR host reduces `r mod n = r'` internally, recovering the victim's address as `authority`.
5. The EVM processes the authorization: writes `EF0100 || attacker_contract` as the victim's code.
6. Attacker calls the victim's address; attacker's contract executes in victim's context and transfers all ETH to the attacker.
