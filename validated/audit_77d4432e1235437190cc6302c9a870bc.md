### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Users to Overpay Gas Fees — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` applies the `deltaGas` correction only in one direction: when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), `gas_used` is increased and the user pays more. When native resource consumption is *lower* than EVM gas consumption (`delta_gas < 0`), `gas_used` is never reduced. The user is therefore overcharged, receiving a smaller gas refund than the double-accounting model entitles them to. A `// TODO: return delta_gas to gas_used?` comment at the exact branch confirms this omission is unresolved.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements the `deltaGas` mechanism described in `docs/double_resource_accounting.md`:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
```

The documentation states that if `deltaGas > 0` the difference is added to `gasUsed`. It is silent on the `deltaGas < 0` case. The implementation reflects this silence:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0`, the branch is skipped entirely. `gas_used` retains its EVM-derived value even though the native resource accounting implies a lower cost. The final refund is `gas_limit - gas_used`, so a higher-than-correct `gas_used` directly reduces the token amount returned to the sender and increases the amount paid to the operator.

This is structurally identical to the external report's root cause: a derived value (there, `adjustedAmount`; here, `gas_used`) is computed with an asymmetric formula that only corrects in one direction, causing the party that should receive more to receive less.

---

### Impact Explanation

Every transaction where `(native_used / native_per_gas) < gas_used` — i.e., where EVM execution consumed more gas than the native resource model implies — results in the user being charged for gas they did not actually consume in native terms. The excess is silently transferred to the operator as additional fee revenue. Over many transactions this constitutes a systematic, protocol-level overcharge of users and corresponding enrichment of the operator, with no mechanism for the user to detect or avoid it.

---

### Likelihood Explanation

The condition `delta_gas < 0` arises whenever a transaction is EVM-gas-heavy but native-resource-light — for example, transactions that perform many cheap EVM arithmetic opcodes, or transactions submitted with a high `gasPrice` (raising `native_per_gas`) relative to their actual native consumption. This is a routine occurrence for ordinary EVM transactions. The code path is unconditionally active for all L2 and L1 transactions processed through `compute_gas_refund` when the `unlimited_native` feature is not enabled.

---

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), mirroring the symmetric treatment already applied for the positive case. The TODO comment at line 80 already identifies this gap. Concretely:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // delta_gas < 0: native consumption implies less gas than EVM used.
    // Reduce gas_used, but never below minimal_gas_used.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

Add a property-based test asserting that for any transaction the sum of `gas_used * gas_price` paid to the operator plus the refund equals exactly `gas_limit * gas_price` (conservation of fees).

---

### Proof of Concept

**Setup:**
- `native_price = 1000`, `gas_price = 10_000` → `native_per_gas = 10`
- `gas_limit = 100_000`
- Transaction consumes `50_000` EVM gas (ergs-derived) and `200_000` native units

**Calculation:**
- `gas_used` (from ergs) = `50_000`
- `native_used` = `full_native_limit - remaining` = `1_000_000 - 800_000` = `200_000`
- `delta_gas = (200_000 / 10) - 50_000 = 20_000 - 50_000 = -30_000`

**Current behaviour:** `delta_gas < 0` branch is skipped. `gas_used` stays at `50_000`. Refund = `(100_000 - 50_000) * gas_price`.

**Correct behaviour:** `gas_used` should be reduced to `max(50_000 - 30_000, minimal_gas_used) = 20_000`. Refund = `(100_000 - 20_000) * gas_price`.

The user is overcharged by `30_000 * gas_price` tokens per transaction in this scenario. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
