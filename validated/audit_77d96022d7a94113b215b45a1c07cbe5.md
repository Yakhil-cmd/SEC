### Title
Missing Deadline on XCC Scheduled Promises Enables Adversarial Execution Timing - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `schedule` function stores cross-contract call promises indefinitely with no expiry timestamp. The `execute_scheduled` function is explicitly open to any caller and performs no deadline validation before executing the stored promise. This allows an adversary to monitor the chain for scheduled promises and trigger execution at a strategically chosen time — for example, when a user's swap slippage tolerance permits a profitable sandwich — extracting value at the user's expense. There is no cancellation mechanism in the router, so the user cannot revoke a scheduled promise once submitted.

---

### Finding Description

When a user submits a `CrossContractCallArgs::Delayed` XCC call through the cross-contract call precompile, the engine routes it to the router's `schedule` method:

**`engine-precompiles/src/xcc.rs`, lines 159–172:**
```rust
CrossContractCallArgs::Delayed(call) => {
    let attached_near = call.total_near();
    let promise = PromiseCreateArgs {
        target_account_id,
        method: consts::ROUTER_SCHEDULE_NAME.into(),   // "schedule"
        args: borsh::to_vec(&call)...,
        attached_balance: ZERO_YOCTO,
        attached_gas: costs::ROUTER_SCHEDULE,
    };
    (promise, attached_near)
}
```

The router's `schedule` method stores the `PromiseArgs` in a `LookupMap<u64, PromiseArgs>` keyed only by a monotonically increasing nonce. No expiry timestamp is stored alongside the promise:

**`etc/xcc-router/src/lib.rs`, lines 136–144:**
```rust
pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();
    let nonce = self.nonce.get().unwrap_or_default();
    self.scheduled_promises.insert(nonce, promise);
    self.nonce.set(&(nonce + 1));
    near_sdk::log!("Promise scheduled at nonce {}", nonce);
}
```

The `execute_scheduled` function is explicitly callable by **anyone** (the comment on line 146 states: *"It is intentional that this function can be called by anyone (not just the parent)"*). It removes and immediately executes the stored promise with no deadline check:

**`etc/xcc-router/src/lib.rs`, lines 149–156:**
```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The `PromiseCreateArgs` type (used inside `PromiseArgs`) carries only `target_account_id`, `method`, `args`, `attached_balance`, and `attached_gas` — no deadline field exists anywhere in the data model. Furthermore, there is no `cancel_scheduled` or equivalent function in the router; the only way to remove a scheduled promise from storage is to execute it via `execute_scheduled`.

The attacker-controlled entry path is:
1. User calls the XCC precompile (`engine-precompiles/src/xcc.rs`, `CrossContractCall::run_with_handle`) with `CrossContractCallArgs::Delayed`, scheduling a time-sensitive NEAR cross-contract call (e.g., a swap on Ref Finance).
2. The promise is stored in the router's `scheduled_promises` map with no expiry.
3. An adversary (MEV bot) monitors the NEAR chain for `schedule` calls on router sub-accounts.
4. The adversary waits for market conditions where the user's swap parameters (e.g., loose `min_amount_out`) allow a profitable sandwich.
5. The adversary calls `execute_scheduled(nonce)` on the router at the optimal moment, front-running and back-running the user's swap to extract value.
6. The user receives fewer tokens than expected; the difference accrues to the adversary.

Because there is no cancellation path, the user cannot protect themselves after the fact.

---

### Impact Explanation

A user who schedules a cross-contract swap via the XCC `Delayed` path has their swap parameters (including slippage tolerance) locked in storage indefinitely. An adversary can execute the swap at any future time when market conditions make a sandwich attack profitable. The adversary profits by the exact amount the user loses on the swap output. This constitutes **direct theft of user funds in motion** (Critical) or at minimum **theft of unclaimed yield** (High) depending on the nature of the scheduled call.

The absence of a cancellation mechanism compounds the severity: unlike Ethereum pending transactions (which can be replaced by resubmitting with the same nonce), a user who realizes their scheduled promise is stale has no recourse.

---

### Likelihood Explanation

Likelihood is **Medium**. The `execute_scheduled` function is explicitly open to any caller by design, and all scheduled promises are visible on-chain. MEV infrastructure on NEAR is less mature than on Ethereum, but it exists and is growing. The attacker must pay gas to call `execute_scheduled`, but this cost is negligible relative to the potential profit from a large swap sandwich. The attack is only profitable when the user's slippage tolerance is loose enough to permit it, which limits but does not eliminate the attack surface.

---

### Recommendation

1. Add an optional `deadline: Option<u64>` (block timestamp in nanoseconds) field to `PromiseArgs` or as a separate parameter to `schedule`.
2. In `execute_scheduled`, check `env::block_timestamp() <= deadline` before executing; revert if the deadline has passed.
3. Add a `cancel_scheduled(nonce: U64)` function callable only by the parent (Aurora Engine), so users can revoke stale scheduled promises via a new EVM transaction.

---

### Proof of Concept

1. Alice calls the XCC precompile with `CrossContractCallArgs::Delayed` encoding a swap on Ref Finance: swap 1000 wNEAR for TOKEN with `min_amount_out = 900 TOKEN` (10% slippage).
2. The router stores this as `scheduled_promises[0]`. No deadline is recorded.
3. Market conditions shift: TOKEN price drops 8%. Alice's `min_amount_out` still allows execution.
4. A MEV bot detects the scheduled promise on-chain.
5. The bot front-runs: buys TOKEN on Ref Finance, pushing the price up.
6. The bot calls `router.execute_scheduled(U64(0))` — succeeds with no deadline check (`etc/xcc-router/src/lib.rs:150–156`).
7. Alice's swap executes at the inflated price, receiving only ~900 TOKEN.
8. The bot back-runs: sells TOKEN, profiting ~80 TOKEN at Alice's expense.
9. Alice has no way to cancel the scheduled promise before step 6 because no `cancel_scheduled` function exists. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
