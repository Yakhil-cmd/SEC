Audit Report

## Title
Insecure-by-default `access_controls_enabled: false` in `SnsWasmCanisterInitPayloadBuilder` allows any unprivileged principal to poison the SNS upgrade path — (`rs/nns/sns-wasm/src/init.rs`)

## Summary
`SnsWasmCanisterInitPayloadBuilder::new()` initializes `access_controls_enabled` to `false`. Both `add_wasm` and `insert_upgrade_path_entries` gate their NNS Governance caller check with `if access_controls_enabled && ...`, meaning when the flag is `false` the check is never evaluated and any caller proceeds to the mutation logic. The production NNS deployment CLI (`rs/nns/init/src/main.rs`) constructs `NnsInitPayloadsBuilder::new()` — which embeds `SnsWasmCanisterInitPayloadBuilder::new()` — and never calls `with_access_controls_enabled(true)`, leaving any deployment via this path with open mutation endpoints.

## Finding Description
**Root cause — insecure default:**
`SnsWasmCanisterInitPayloadBuilder::new()` sets `access_controls_enabled: false` at `rs/nns/sns-wasm/src/init.rs` lines 16–24. [1](#0-0) 

**Conditional, bypassable guard:**
In `rs/nns/sns-wasm/canister/canister.rs`, `add_wasm` (line 305) and `insert_upgrade_path_entries` (line 318) both use `if access_controls_enabled && caller() != GOVERNANCE_CANISTER_ID.into()`. When `access_controls_enabled` is `false`, the entire `if` branch is skipped and the caller proceeds unconditionally to the mutation logic. [2](#0-1) 

Note the contrast with `update_sns_subnet_list` at line 407, which unconditionally checks `caller() != GOVERNANCE_CANISTER_ID` with no flag gate. [3](#0-2) 

**Deployment tool never overrides the default:**
`create_init_payloads` in `rs/nns/init/src/main.rs` (lines 176–254) calls `NnsInitPayloadsBuilder::new()`, which at `rs/nns/test_utils/src/common.rs` line 89 embeds `SnsWasmCanisterInitPayloadBuilder::new()` directly. Neither function ever calls `with_access_controls_enabled(true)` or `with_sns_wasm_access_controls(true)`. [4](#0-3) [5](#0-4) 

**Persistence through upgrades:**
`access_controls_enabled` is serialized into `StableCanisterState` on `pre_upgrade` and deserialized on `post_upgrade` via the `From<StableCanisterState> for SnsWasmCanister` impl. Whatever value was set at `canister_init` time persists indefinitely through all future upgrades. [6](#0-5) [7](#0-6) 

## Impact Explanation
With `access_controls_enabled: false`, any unprivileged principal can call `add_wasm` to register an attacker-controlled WASM binary for any `SnsCanisterType` (e.g., `Governance`), then call `insert_upgrade_path_entries` to wire that malicious hash into the canonical SNS upgrade path. Every subsequent `deploy_new_sns` call will install the attacker's WASM into newly created SNS governance canisters, giving the attacker full control over all future SNS deployments. This constitutes a **High** impact: significant SNS security impact with concrete protocol harm — unauthorized manipulation of NNS/SNS governance assets and the SNS deployment pipeline.

## Likelihood Explanation
Any deployment of the SNS-WASM canister using `NnsInitPayloadsBuilder::new()` without an explicit `with_sns_wasm_access_controls(true)` call inherits the insecure default. The `ic-nns-init` CLI tool supports targeting any IC URL (including mainnet via `--url`) and HSM signing (`--use-hsm`), making it a realistic deployment path. No special privileges are required to trigger the exploit once the canister is initialized with the default. The attack is repeatable and requires no victim interaction.

## Recommendation
1. Change the default in `SnsWasmCanisterInitPayloadBuilder::new()` to `access_controls_enabled: true` in `rs/nns/sns-wasm/src/init.rs`.
2. Remove the conditional flag entirely from `add_wasm` and `insert_upgrade_path_entries` in `rs/nns/sns-wasm/canister/canister.rs` — these endpoints should unconditionally require `caller() == GOVERNANCE_CANISTER_ID`, matching the pattern already used by `update_sns_subnet_list`.
3. Update `rs/nns/init/src/main.rs` to explicitly call `with_sns_wasm_access_controls(true)` regardless of the builder default.
4. Add a `canister_init` assertion that panics if `access_controls_enabled` is `false` in non-test builds.

## Proof of Concept
The existing test suite already proves the exploit path is reachable. `test_add_wasm_can_be_called_directly_if_access_controls_are_disabled` in `rs/nns/sns-wasm/tests/add_wasm.rs` (lines 47–66) demonstrates that with `access_controls_enabled: false`, any caller can successfully register a WASM binary. [8](#0-7) 

`insert_upgrade_path_entries_callable_by_anyone_when_access_controls_disabled` in `rs/nns/sns-wasm/tests/upgrade_sns_instance.rs` (lines 1022–1054) demonstrates that an unprivileged principal (`PrincipalId::new_user_test_id(1)`) can call `insert_upgrade_path_entries` and reach the mutation logic (the response error is from the business logic, not the access control check). [9](#0-8) 

A full exploit test would extend these by: (1) calling `add_wasm` with an attacker-controlled WASM binary from an unprivileged principal, (2) calling `insert_upgrade_path_entries` to wire the malicious hash into the upgrade path, and (3) calling `deploy_new_sns` to confirm the attacker's WASM is installed into a new SNS governance canister — all without any NNS Governance involvement.

fn add_wasm(add_wasm_payload: AddWasmRequest) -> AddWasmResponse {
    let access_controls_enabled =
        SNS_WASM.with(|sns_wasm| sns_wasm.borrow().access_controls_enabled);
    if access_controls_enabled && caller() != GOVERNANCE_CANISTER_ID.into() {

impl<M: StableMemory + Clone + Default> From<StableCanisterState> for SnsWasmCanister<M> {
    fn from(stable_canister_state: StableCanisterState) -> Self {
        let StableCanisterState {
            wasm_indexes,
            upgrade_path,
            sns_subnet_ids,
            deployed_sns_list,
            access_controls_enabled,
            allowed_principals,
            nns_proposal_to_deployed_sns,
        } = stable_canister_state;

            .into_iter()
            .map(|index| (vec_to_hash(index.hash.clone()).unwrap(), index))

            wasm_indexes,
            sns_subnet_ids,
            stable_memory,
            deployed_sns_list,
            upgrade_path,
            access_controls_enabled,
            allowed_principals,
            nns_proposal_to_deployed_sns,
        }

#[test]
fn test_add_wasm_can_be_called_directly_if_access_controls_are_disabled() {
    let machine = StateMachine::new();

    let nns_init_payload = NnsInitPayloadsBuilder::new()
        .with_sns_dedicated_subnets(machine.get_subnet_ids())
        .with_sns_wasm_access_controls(false)
        .build();

    let sns_wasm_canister_id = install_sns_wasm(&machine, &nns_init_payload);

    let root_wasm = build_root_sns_wasm();
    let root_hash = root_wasm.sha256_hash();
    let response = add_wasm(&machine, sns_wasm_canister_id, root_wasm, &root_hash);

    assert_eq!(
        response.result.unwrap(),
        add_wasm_response::Result::Hash(root_hash.to_vec())
    );
}

#[test]
fn insert_upgrade_path_entries_callable_by_anyone_when_access_controls_disabled() {
    let machine = StateMachineBuilder::new().with_current_time().build();

    let nns_init_payload = NnsInitPayloadsBuilder::new()
        .with_sns_wasm_access_controls(false)
        .with_initial_invariant_compliant_mutations()
        .with_test_neurons()
        .with_sns_dedicated_subnets(machine.get_subnet_ids())
        .build();

    setup_nns_canisters(&machine, nns_init_payload);

    let response: InsertUpgradePathEntriesResponse = update_with_sender(
        &machine,
        SNS_WASM_CANISTER_ID,
        "insert_upgrade_path_entries",
        InsertUpgradePathEntriesRequest {
            upgrade_path: vec![],
            sns_governance_canister_id: None,
        },
        PrincipalId::new_user_test_id(1),
    )
    .unwrap();

    // We get an error past the access controls (request was actually processed)
    assert_eq!(
        response,
        InsertUpgradePathEntriesResponse::error(
            "No Upgrade Paths in request. No action taken.".to_string()
        )
    );
}
