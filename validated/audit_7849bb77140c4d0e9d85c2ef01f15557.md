### Title
Unvalidated User-Supplied Ethereum Destination Address in `withdraw_eth` Allows Funds to Be Sent to Smart Contracts Where They Are Permanently Lost - (File: `rs/ethereum/cketh/minter/src/main.rs`)

---

### Summary

The ckETH minter's `withdraw_eth` function accepts a user-supplied Ethereum `recipient` address and sends real ETH to it via a threshold-ECDSA-signed EIP-1559 transaction with a fixed gas limit of 21,000. The minter only validates that the address is non-zero and not on a static blocklist, but does **not** detect or reject smart contract addresses. The Candid interface itself documents this risk: *"The current gas limit is set to 21,000 for a transaction so withdrawals to smart contract addresses will likely fail."* However, the minter still accepts the request, burns the user's ckETH, and dispatches the Ethereum transaction. When the destination is a smart contract whose fallback/receive function requires more than 21,000 gas, the Ethereum transaction reverts, the ETH is consumed by fees, and the ckETH has already been burned — resulting in permanent loss of funds with no reimbursement path for the ETH.

---

### Finding Description

In `rs/ethereum/cketh/minter/src/main.rs`, the `withdraw_eth` update function calls `validate_address_as_destination` on the user-supplied `recipient` string:

```rust
let destination = validate_address_as_destination(&recipient).map_err(|e| match e {
    AddressValidationError::Invalid { .. } | AddressValidationError::NotSupported(_) => {
        ic_cdk::trap(e.to_string())
    }
    AddressValidationError::Blocked(address) => WithdrawalError::RecipientAddressBlocked { ... },
})?;
```

`validate_address_as_destination` in `rs/ethereum/cketh/minter/src/address.rs` only checks:
1. That the address parses as a valid 20-byte hex address.
2. That it is not `Address::ZERO`.
3. That it is not on the static `ETH_ADDRESS_BLOCKLIST`.

There is **no check** for whether the address is an Ethereum smart contract (i.e., has code deployed at it). The minter then immediately burns the caller's ckETH on the ledger and queues the withdrawal. The Ethereum transaction is constructed in `create_transaction` with a fixed `gas_limit` of `GasAmount::from(21_000_u32)` and `data: Vec::new()`. If the destination is a smart contract whose `receive()` or `fallback()` function requires more than 21,000 gas (which is the case for virtually all non-trivial contracts), the Ethereum transaction will revert. The ETH is not delivered, the gas fee is consumed, and the ckETH has already been permanently burned with no reimbursement.

The Candid interface at line 719 of `rs/ethereum/cketh/minter/cketh_minter.did` explicitly acknowledges this:
> `// IMPORTANT: The current gas limit is set to 21,000 for a transaction so withdrawals to smart contract addresses will likely fail.`

Yet the minter accepts the call, burns ckETH, and dispatches the transaction anyway.

---

### Impact Explanation

**Chain-fusion ledger conservation bug / chain-fusion mint/burn/replay bug.**

A user who calls `withdraw_eth` with a smart contract address as `recipient` will:
1. Have their ckETH burned on the ckETH ledger (irreversible).
2. Have the minter dispatch an Ethereum transaction to the contract address with 21,000 gas and no calldata.
3. If the contract's `receive()` or `fallback()` requires more than 21,000 gas (e.g., any proxy, multisig, or DeFi contract), the transaction reverts on Ethereum.
4. The ETH is not delivered. The gas fee is consumed from the minter's ETH balance.
5. The ckETH is already burned. There is no reimbursement path for the ETH value (unlike the ckERC20 path which reimburses ckETH on ERC-20 burn failure).

The result is permanent loss of the user's funds. The ckETH supply decreases without a corresponding ETH delivery, breaking the 1:1 peg conservation invariant.

---

### Likelihood Explanation

**Medium.** Any unprivileged user who calls `withdraw_eth` with a smart contract address (e.g., a Gnosis Safe, Uniswap router, or any DeFi protocol address) triggers this. This is a realistic and common mistake — users frequently want to withdraw to a multisig or smart contract wallet. The minter accepts the call without warning at the protocol level (only a comment in the Candid file), burns the ckETH, and the loss is permanent. No special access or privilege is required.

---

### Recommendation

1. **Detect and reject smart contract destinations at the minter level**: Before accepting a `withdraw_eth` request, the minter should query the Ethereum network (via the EVM RPC canister) for the bytecode at the destination address using `eth_getCode`. If the result is non-empty, the minter should reject the request with a clear error (e.g., `RecipientIsSmartContract`).

2. **Alternatively, increase the gas limit for ETH withdrawals** to a value sufficient for common smart contract wallets (e.g., 100,000 gas), so that standard multisig and proxy wallets can receive ETH.

3. **Add a reimbursement path**: If the Ethereum transaction reverts (status = 0 in the receipt), reimburse the user's ckETH (minus the gas fee), analogous to the ckERC20 reimbursement path already implemented.

4. **Update the Candid comment** from a passive warning to an active enforcement note, or remove the comment and enforce the restriction in code.

---

/// Validate whether the given address can be used as the destination of an Ethereum transaction.
pub fn validate_address_as_destination(address: &str) -> Result<Address, AddressValidationError> {
    let address =
        Address::from_str(address).map_err(|e| AddressValidationError::Invalid { error: e })?;
    if address == Address::ZERO {
        return Err(AddressValidationError::NotSupported(address));
    }
    if crate::blocklist::is_blocked(&address) {
        return Err(AddressValidationError::Blocked(address));
    }
    Ok(address)
}
