### Title
Unprotected `new()` Initializer Allows Any Caller to Seize Engine Ownership — (`File: engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine's `new()` initializer imposes no restriction on who may call it. Because NEAR contract deployment and initialization are two separate transactions, any unprivileged NEAR account that submits a `new()` call before the legitimate deployer can install itself as `owner_id`, gaining full administrative control over the engine and all bridged user funds.

---

### Finding Description

`engine/src/contract_methods/admin.rs` exposes the `new()` function as the sole initialization entry point for the Aurora Engine:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;
    // ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

The only guard is a check that state does not already exist. There is no check on `env.predecessor_account_id()`. The `owner_id` written into `EngineState` is taken entirely from caller-supplied calldata (`NewCallArgs`), not from the deployer's account identity. [2](#0-1) 

The deployment flow in `engine-workspace/src/lib.rs` confirms that deployment and initialization are two **separate** transactions — `account.deploy(...).await?` is followed by a distinct `engine.new(...).transact().await` call: [3](#0-2) 

This creates a window between the two transactions during which any NEAR account can call `new()` and supply an arbitrary `owner_id`.

The public WASM entrypoint in `engine/src/lib.rs` confirms `new()` is unconditionally reachable by any NEAR account: [4](#0-3) 

---

### Impact Explanation

The `owner_id` field in `EngineState` gates every privileged operation in the engine:

- `upgrade()` — deploys arbitrary WASM to the Aurora account, enabling complete replacement of engine logic and theft of all bridged ETH and ERC-20 tokens. [5](#0-4) 
- `attach_full_access_key()` — adds a full-access key to the Aurora account, giving the attacker permanent account-level control. [6](#0-5) 
- `pause_contract()` — permanently freezes all user funds. [7](#0-6) 

All of these are gated solely by `require_owner_only`: [8](#0-7) 

An attacker who wins the initialization race becomes the unconditional owner and can immediately call `upgrade()` to deploy malicious WASM, draining all user funds held in the Aurora bridge. **Impact: Critical — direct theft of all user funds at rest.**

---

### Likelihood Explanation

NEAR Protocol has a public transaction pool. Contract deployment and initialization are provably non-atomic in this codebase (two separate `await` calls). An attacker passively monitoring the NEAR RPC for `DeployContract` actions on the `aurora` account can immediately submit a `new()` call in the same or next block. NEAR's ~1-second block time provides a narrow but real window. No special privileges, leaked keys, or social engineering are required — only a valid NEAR account and knowledge of the `NewCallArgs` encoding (which is public). **Likelihood: Medium** (requires active monitoring and timely submission, but is fully permissionless).

---

### Recommendation

Perform deployment and initialization atomically in a single NEAR batch transaction, exactly as the XCC router does (Create + Deploy + Initialize in one batch): [9](#0-8) 

Alternatively, restrict `new()` so it can only be called by `env.current_account_id()` (i.e., a self-call from within a deploy batch), or by the account that holds the contract's access key at deployment time. A check such as:

```rust
if env.predecessor_account_id() != env.current_account_id() {
    return Err(b"ERR_NOT_ALLOWED".into());
}
```

would ensure only a batch-initiated self-call can initialize the engine.

---

### Proof of Concept

1. Deployer broadcasts transaction T1: `DeployContract { code: aurora_engine.wasm }` to account `aurora`.
2. Attacker observes T1 in the NEAR transaction pool.
3. Attacker broadcasts transaction T2 (same or next block): call `new` on `aurora` with `NewCallArgs::V2 { chain_id: ..., owner_id: "attacker.near", upgrade_delay_blocks: 0 }`.
4. If T2 is processed before the deployer's `new` call, `state::set_state` writes `owner_id = "attacker.near"` into storage. [10](#0-9) 
5. Deployer's `new` call fails: `ERR_ALREADY_INITIALIZED`.
6. Attacker calls `upgrade()` from `attacker.near`, deploying malicious WASM that transfers all ETH connector balances to the attacker. [5](#0-4)

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
```
