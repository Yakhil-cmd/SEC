Audit Report

## Title
ckERC20 Withdrawal Permanently Stuck and Cascading Nonce Blockage When Gas Fees Exceed Locked `max_transaction_fee` - (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, `rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

When a user calls `withdraw_erc20`, the minter immediately burns ckETH as `max_transaction_fee` and ckERC20 tokens, locking both amounts into an `Erc20WithdrawalRequest` with no mutable fee field. If Ethereum gas fees subsequently spike above that ceiling, the pending request is rescheduled indefinitely with no reimbursement path. If a transaction was already sent and cannot be resubmitted at the higher fee, the minter silently stops resubmission, the stuck nonce blocks all higher-nonce transactions, and every subsequent ckERC20 withdrawal for every user is frozen until a governance upgrade intervenes.

## Finding Description

**Fee locked at creation.** In `main.rs` lines 448–492, `erc20_tx_fee` is burned from the user's ckETH account and stored immutably as `max_transaction_fee` in `Erc20WithdrawalRequest`. The struct (defined in `mod.rs` lines 144–177) has no mutable fee field after creation, and no endpoint exists to increase it post-creation (confirmed by grep: no `top_up`, `cancel_withdrawal`, or `increase_fee` symbol anywhere in the minter source).

**Indefinite reschedule with no reimbursement.** `create_transaction` in `mod.rs` lines 1155–1168 computes `request_max_fee_per_gas = max_transaction_fee / gas_limit` and returns `CreateTransactionError::InsufficientTransactionFee` when `actual_min_max_fee_per_gas > request_max_fee_per_gas`. The handler in `withdraw.rs` lines 281–291 calls `reschedule_withdrawal_request`, moving the request to the back of the queue with no reimbursement event emitted. The reimbursement mechanism in `record_finalized_transaction` (`mod.rs` lines 733–745) only fires on `TransactionStatus::Failure` from an on-chain receipt — the reschedule path never reaches that code.

**Sent-transaction stuck path cascades.** `SignedTransactionRequest::resubmit` in `tx.rs` lines 169–173 returns `ResubmitTransactionError::InsufficientTransactionFee` when the new fee would exceed `allowed_max_transaction_fee` (set from `ckerc20.max_transaction_fee`). `resubmit_transactions_batch` in `withdraw.rs` lines 242–244 only logs the error and continues. Critically, `create_resubmit_transactions` in `mod.rs` lines 618–631 explicitly `return`s on the first `InsufficientTransactionFee` error, because — as the code comment at lines 589–592 states — all higher-nonce transactions are blocked by the stuck nonce. This means one stuck ckERC20 withdrawal freezes every subsequent withdrawal for every user sharing the minter's nonce sequence.

**Existing guards are insufficient.** The only reimbursement paths are: (a) ckERC20 ledger burn failure before queuing (`main.rs` lines 506–531), and (b) on-chain `TransactionStatus::Failure` receipt (`mod.rs` lines 733–745). Neither covers the indefinite-reschedule loop or the silent-abandon-on-resubmit path. The `maybe_reimburse` set is only populated when a transaction is created (`mod.rs` line 552) and only cleared on finalization (`mod.rs` line 710) — a request stuck in `pending_withdrawal_requests` never enters `maybe_reimburse` at all.

## Impact Explanation

This matches **High** impact: "Significant Chain Fusion / ck-token security impact with concrete user or protocol harm" and "Application/platform-level DoS… not based on raw volumetric DDoS."

- **User fund loss**: A user's burned ckETH (gas fee) and burned ckERC20 tokens are irrecoverable for the duration of the stuck state. If gas fees remain elevated indefinitely, the loss is permanent with no protocol-level remedy available to the user.
- **Minter-wide DoS**: A single stuck sent transaction blocks all subsequent nonces, freezing every other user's pending ckERC20 withdrawal until a governance NNS upgrade is executed. This is a concrete, user-triggered availability impact on a production Chain Fusion component.

## Likelihood Explanation

Ethereum gas fee spikes are routine (NFT mints, DeFi liquidation cascades, network congestion). The EVM-RPC canister adds multiple HTTPS outcall and consensus-round latencies between the user's `eip_1559_transaction_price` query and the minter's actual transaction construction, widening the window for a spike. The 2024-08-05 ckETH minter upgrade note explicitly states a prior fix for "ckERC20 withdrawals which were unnecessarily delayed as soon as the estimated transaction fees increased," confirming this class of bug has already manifested in production. No special privileges are required — any unprivileged user calling `withdraw_erc20` during a gas spike can trigger the indefinite-reschedule path; the cascading nonce blockage requires only that one such request reach the sent-transaction stage.

## Recommendation

1. **Add a reimbursement path for the indefinite-reschedule case**: after a configurable number of failed reschedule attempts or a timeout, emit a `FailedErc20WithdrawalRequest` event and reimburse both the ckETH gas fee and the ckERC20 tokens to the user.
2. **Add a reimbursement path for the sent-transaction stuck case**: when `ResubmitTransactionError::InsufficientTransactionFee` is returned in `resubmit_transactions_batch`, emit a reimbursement event for the stuck withdrawal rather than only logging, and unblock the nonce sequence.
3. **Add a `top_up_withdrawal_fee` endpoint** allowing the original caller to burn additional ckETH to increase the `max_transaction_fee` of a pending `Erc20WithdrawalRequest`.

## Proof of Concept

1. Call `eip_1559_transaction_price`; receive a fee estimate `F`.
2. Approve the minter for `F` on the ckETH ledger; call `withdraw_erc20`. Minter burns `F` ckETH and `W` ckERC20.
3. Simulate a 3× gas spike (e.g., via `StateMachine` test by injecting a `GasFeeEstimate` with `base_fee_per_gas` tripled).
4. Advance the state machine timer to trigger `process_retrieve_eth_requests` → `create_transactions_batch`.
5. Observe `create_transaction` returns `InsufficientTransactionFee`; `reschedule_withdrawal_request` is called; no reimbursement event is emitted.
6. Repeat timer ticks; confirm the request cycles indefinitely in `pending_withdrawal_requests` with no reimbursement.
7. For the cascading case: inject the spike after the transaction reaches `sent_tx`; advance the timer; observe `resubmit_transactions_batch` logs `"Failed to resubmit transaction"` and all higher-nonce withdrawals remain frozen in `pending_withdrawal_requests`.
