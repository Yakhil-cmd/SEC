### Title
Silent NEAR-Side Promise Failure in `ExitToNear` Precompile Causes Permanent EVM Token Loss When `error_refund` Is Disabled - (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns EVM tokens (ETH or ERC-20) and schedules a NEAR-side `ft_transfer`/`ft_transfer_call` promise. When the `error_refund` compile-time feature is absent — which is the case for production builds, since the `contract` feature does not include it — no failure callback is attached to the NEAR promise. If the NEAR-side call fails (e.g., the recipient account does not exist on NEAR, or is not registered with the NEP-141 contract), the EVM tokens are permanently destroyed with no refund path. This is the direct Aurora analog of the Solidity low-level-call-to-nonexistent-contract vulnerability: the EVM-side call returns `Succeed` while the cross-chain transfer silently fails.

---

### Finding Description

**Root cause — conditional callback omission**

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` method constructs a `callback_args` struct whose `refund` field is gated on the `error_refund` feature flag:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the standard ETH-exit and ERC-20-exit paths, `transfer_near_args` is also `None`, so `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The branch that follows then emits a bare `PromiseArgs::Create` — a fire-and-forget NEAR promise with **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The precompile then returns `Ok(PrecompileOutput { ... })`, so the EVM transaction is marked `Succeed` and the token burn is committed, regardless of what the NEAR promise will do. [3](#0-2) 

**Why `error_refund` is absent in production**

The `error_refund` feature is defined in `engine-precompiles/Cargo.toml` as a standalone opt-in feature with no inclusion in `default`:

```toml
[features]
default = ["std"]
error_refund = []
``` [4](#0-3) 

The production `contract` feature in `engine/Cargo.toml` propagates only `log`, `aurora-engine-sdk/contract`, and `aurora-engine-precompiles/contract` — `error_refund` is never pulled in:

```toml
contract = ["log", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [5](#0-4) 

**Confirmed by the test suite**

The integration test `test_exit_to_near_refund` explicitly documents the two outcomes:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

The same pattern is confirmed for the ETH-exit path in `test_exit_to_near_eth_refund`:

```rust
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [7](#0-6)

### Impact Explanation

**Impact: Critical — Permanent freezing / direct theft of user funds.**

When a user calls `ExitToNear` with a NEAR recipient that does not exist or is not registered with the NEP-141 token contract:

1. The EVM-side token burn is committed (ERC-20 balance decremented, or ETH deducted).
2. The NEAR `ft_transfer` promise is scheduled with no callback.
3. The NEAR runtime executes the promise; it fails (account not found / not registered).
4. No `exit_to_near_precompile_callback` is ever invoked.
5. The user's EVM tokens are gone with no recovery path.

The funds are not recoverable by the user, the protocol, or any admin function. This constitutes permanent loss of user funds.

---

### Likelihood Explanation

**Likelihood: High.**

The attacker-controlled entry path is fully unprivileged:

- Any EVM account can call the `ExitToNear` precompile at address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` directly or via a Solidity contract.
- The `receiver_account_id` is user-supplied bytes parsed from calldata; any syntactically valid NEAR account ID is accepted, including accounts that do not exist on-chain.
- No admin role, no special permission, no governance action is required.
- Ordinary user error (mistyped NEAR account) is sufficient to trigger the loss, and a malicious contract can deliberately route victim funds to a dead address. [8](#0-7) 

---

### Recommendation

1. **Enable `error_refund` in the production `contract` feature** in `engine/Cargo.toml`:
   ```toml
   contract = ["log", "error_refund", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
   ```
2. Alternatively, unconditionally attach the `exit_to_near_precompile_callback` as a failure handler on every `ft_transfer`/`ft_transfer_call` promise emitted by `ExitToNear`, regardless of the feature flag.
3. Consider also adding a no-callback guard for the `transfer_near` (wNEAR-unwrap) path in `exit_to_near_precompile_callback`, where a failed `PromiseAction::Transfer` to a non-existent NEAR account similarly has no further recovery callback. [9](#0-8) 

---

### Proof of Concept

**Attacker steps (ERC-20 exit path):**

1. Attacker holds ERC-20 tokens on Aurora (bridged NEP-141).
2. Attacker (or victim, by mistake) calls `withdrawToNear(bytes recipient, uint256 amount)` on the ERC-20 contract, supplying a NEAR account ID that does not exist (e.g., `"nonexistent12345.near"`).
3. The ERC-20 contract calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x01`, the amount, and the recipient.
4. The precompile burns the ERC-20 tokens and emits a `PromiseArgs::Create(ft_transfer_promise)` log — no callback.
5. `filter_promises_from_logs` in `engine/src/engine.rs` schedules the bare NEAR promise.
6. The NEAR runtime executes `ft_transfer` on the NEP-141 contract; it panics because `nonexistent12345.near` is not registered.
7. The promise result is `Failed`; no callback fires; no refund is issued.
8. The user's ERC-20 balance is permanently zero. [10](#0-9)
