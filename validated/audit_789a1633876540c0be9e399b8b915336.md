### Title
Asymmetric `delta_gas` Application in `compute_gas_refund` Causes Permanent User Overpayment - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` correction is applied **only when native resource consumption exceeds EVM gas consumption** (`delta_gas > 0`), but is **never applied** when EVM gas consumption exceeds native resource consumption (`delta_gas < 0`). This one-directional accumulator update is structurally identical to the Fei Protocol bug: one direction of an operation adjusts a shared counter, but the reverse direction does not symmetrically restore it. The result is that users always pay at least the full EVM gas amount even when native resource consumption would justify a lower charge, causing a permanent, per-transaction loss of user funds.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements ZKsync OS's double-resource reconciliation step. After computing `gas_used` from remaining ergs and applying the EVM refund counter cap, it computes:

```
delta_gas = (native_used / native_per_gas) as i64 - gas_used as i64
```

The intent is to reconcile the two resource dimensions. However, the adjustment is applied asymmetrically:

```rust
if delta_gas > 0 {
    // native cost > EVM gas cost → charge extra gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` (EVM gas consumed > native resource consumption), `gas_used` is **not reduced**. The developer-authored `TODO` comment directly acknowledges this gap. The documentation in `docs/double_resource_accounting.md` only describes the `delta_gas > 0` path:

> "If `deltaGas > 0`, we add it to `gasUsed`…"

The negative case is absent from the specification, confirming it is an unresolved gap rather than intentional design.

**Structural analog to Fei Protocol H03:**

| Fei Protocol | ZKsync OS |
|---|---|
| `timeWeightInfo.weight` accumulates over time | `gas_used` is computed per-transaction |
| Partial buy reduces weight by `remainingRatio` | `delta_gas > 0` increases `gas_used` |
| Sell does **not** increase weight by inverse ratio | `delta_gas < 0` does **not** decrease `gas_used` |
| Incentive decays with volatility | User always pays ≥ full EVM gas amount |

In both cases, one direction of an operation updates a counter, but the reverse direction does not symmetrically restore it, causing the counter to drift in one direction permanently.

---

### Impact Explanation

For any transaction where EVM gas consumption exceeds native resource consumption:

- `gas_used` remains at the EVM-computed value instead of being reduced by `|delta_gas|`
- The user pays `gas_price × gas_used` to the operator
- The user receives a refund of `gas_price × (gas_limit − gas_used)`
- The overpayment per transaction is `|delta_gas| × gas_price`

**Concrete example:**
- `gas_limit = 200,000`, EVM `gas_used = 100,000`, `native_per_gas = 10`
- `native_used = 500,000` (implying only 50,000 gas worth of native cost)
- `delta_gas = 50,000 − 100,000 = −50,000`
- Without fix: user pays for 100,000 gas
- With symmetric fix: user would pay for 50,000 gas
- Overpayment: `50,000 × gas_price` tokens per transaction

This is a direct, quantifiable loss of user funds on every transaction where EVM gas exceeds native resource consumption.

---

### Likelihood Explanation

The `delta_gas < 0` case is triggered whenever EVM gas consumption exceeds native resource consumption. This is the **common case** for most standard EVM transactions: EVM gas models computation at a coarser granularity than the RISC-V native proving cost. The documentation itself states:

> "We expect the base fee to be enough to cover most transactions without the need of additional gas."

This implies `delta_gas > 0` (the case requiring extra gas) is the **rare** case, meaning `delta_gas < 0` (the unhandled case) is the **normal** case. Every standard EVM transaction submitted by any unprivileged user is affected.

---

### Recommendation

Apply `delta_gas` symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, subject to the existing `minimal_gas_used` floor:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // Apply symmetric reduction, bounded by minimal_gas_used
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction);
}
// The existing max(gas_used, minimal_gas_used) floor already handles the lower bound
```

### Proof of Concept

1. Deploy any EVM contract with moderate computation (e.g., a loop with arithmetic operations).
2. Submit a transaction with `gas_limit = 200,000`, `gas_price = 1000`, `native_price` set such that `native_per_gas` is low (e.g., 5).
3. After execution, observe `gas_used = 100,000` (EVM gas consumed) and `native_used` implying only 40,000 gas equivalent.
4. `delta_gas = 40,000 − 100,000 = −60,000` → not applied.
5. User is charged for 100,000 gas instead of 40,000 gas.
6. Overpayment: `60,000 × 1000 = 60,000,000` base token units per transaction.
7. Repeat across any number of transactions to demonstrate the systematic nature of the overpayment.
