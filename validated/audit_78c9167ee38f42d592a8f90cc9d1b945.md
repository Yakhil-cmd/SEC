### Title
Unchecked Subtraction `gas_limit - gas_used` in `compute_gas_refund` Can Underflow When `minimal_gas_used` Exceeds `gas_limit` — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` contains an unchecked subtraction `gas_limit - gas_used` at line 83. When the `eip_7623` feature is not compiled in, a transaction with large calldata and a low `gas_limit` can pass L2 ZK validation while carrying a `minimal_gas_used` value that exceeds `gas_limit`. The subsequent `core::cmp::max(gas_used, minimal_gas_used)` call at line 56 then sets `gas_used > gas_limit`, making the subtraction at line 83 underflow. In Rust release mode this wraps to a near-`u64::MAX` value, which immediately trips the `require_internal!` guard and returns an `InternalError` — a class of error that propagates out of the transaction flow and can abort block processing entirely.

---

### Finding Description

`compute_gas_refund` is the final accounting step for every processed transaction. Its signature is:

```rust
pub(crate) fn compute_gas_refund<S: EthereumLikeTypes>(
    system: &mut System<S>,
    to_charge_for_pubdata: S::Resources,
    gas_limit: u64,
    minimal_gas_used: u64,   // ← EIP-7623 floor, caller-supplied
    native_per_gas: u64,
    resources: &mut S::Resources,
) -> Result<RefundInfo, InternalError>
```

The relevant sequence inside the function is:

```rust
// line 31-33: gas_used ≤ gas_limit here (checked_sub)
let mut gas_used = gas_limit
    .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
    .ok_or(internal_error!("gas remaining > gas limit"))?;

// line 48: safe, evm_refund ≤ gas_used/5
gas_used -= evm_refund;

// line 56: gas_used may now EXCEED gas_limit if minimal_gas_used > gas_limit
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

// ... delta_gas block (feature-gated) ...

// line 83: UNCHECKED subtraction — panics in debug, wraps in release
let total_gas_refund = gas_limit - gas_used;
``` [1](#0-0) 

The `minimal_gas_used` argument originates from `compute_calldata_tokens` in the ZK validation path:

```rust
// validation_impl.rs line 98-104
let (calldata_tokens, minimal_gas_used) = compute_calldata_tokens(system, calldata);
#[cfg(feature = "eip_7623")]          // ← guard is ABSENT without the feature
require!(
    minimal_gas_used <= tx_gas_limit,
    InvalidTransaction::EIP7623IntrinsicGasIsTooLow,
    system
)?;
``` [2](#0-1) 

Without the `eip_7623` feature, no check enforces `minimal_gas_used ≤ tx_gas_limit`. The transaction passes validation and reaches `compute_gas_refund` with `minimal_gas_used > gas_limit`.

By contrast, the L1 transaction path explicitly clamps the value before passing it:

```rust
// process_l1_transaction.rs line 537
let minimal_gas_used = minimal_gas_used.min(gas_limit);
``` [3](#0-2) 

No equivalent clamp exists for L2 ZK transactions.

---

### Impact Explanation

In Rust release mode the subtraction wraps to approximately `u64::MAX - (gas_used - gas_limit)`. The immediately following guard:

```rust
require_internal!(
    total_gas_refund <= gas_limit,
    "Gas refund greater than gas limit",
    system
)?;
``` [4](#0-3) 

fails because the wrapped value far exceeds `gas_limit`, and `compute_gas_refund` returns `Err(InternalError)`. `InternalError` is not a user-level rejection; it propagates out of the transaction flow and can abort block processing entirely, constituting a **state-transition DOS**: a single crafted transaction can prevent the sequencer from finalising a block.

In debug / proving mode (overflow checks enabled) the subtraction panics directly, with the same block-halting effect.

---

### Likelihood Explanation

The attack requires only:
1. Submitting an L2 ZK transaction with calldata large enough that `compute_calldata_tokens` returns `minimal_gas_used > gas_limit`.
2. Setting `gas_limit` below that floor.

Both fields are fully attacker-controlled in a standard transaction. No privileged role, oracle manipulation, or key compromise is needed. The only prerequisite is that the production build does not compile in the `eip_7623` feature, which is an optional compile-time flag and not enforced by default.

---

### Recommendation

Apply the same clamp used for L1 transactions to the L2 ZK path, unconditionally (not only under `eip_7623`):

```rust
// After computing minimal_gas_used, before passing to compute_gas_refund:
let minimal_gas_used = minimal_gas_used.min(tx_gas_limit);
```

Alternatively, replace the bare subtraction at line 83 of `refund_calculation.rs` with a checked variant and return an `InternalError` with a clear message if the invariant is violated:

```rust
let total_gas_refund = gas_limit
    .checked_sub(gas_used)
    .ok_or(internal_error!("gas_used exceeds gas_limit in refund calculation"))?;
``` [5](#0-4) 

---

### Proof of Concept

1. Craft an L2 ZK transaction whose calldata contains enough tokens that `compute_calldata_tokens` returns `minimal_gas_used = M`.
2. Set `gas_limit = M - 1` (one unit below the EIP-7623 floor).
3. Submit the transaction to a node built **without** the `eip_7623` feature.
4. Validation passes (no floor check).
5. Execution completes; `compute_gas_refund` is called with `minimal_gas_used = M`, `gas_limit = M - 1`.
6. Line 56: `gas_used = max(gas_used_from_ergs, M) = M > M - 1 = gas_limit`.
7. Line 83: `(M - 1) - M` underflows → wraps to `u64::MAX - 0` in release mode.
8. `require_internal!(total_gas_refund <= gas_limit)` fails → `InternalError` returned → block processing aborted.
