### Title
Anonymous Principal Permanently Blocked from Removing Its Own Approvals via `remove_approval` — (`rs/ledger_suite/icp/ledger/src/main.rs`)

### Summary
`remove_approval` constructs a dummy `ApproveArgs` with `spender.owner = Principal::anonymous()` and passes it to `icrc2_approve_not_async`. When the caller is the anonymous principal, the self-approval guard (`from_account.owner == arg.spender.owner`) evaluates `anonymous == anonymous → true` and traps unconditionally. The `override_spender` parameter that carries the real spender is only consulted after this guard, so it cannot rescue the call. The anonymous principal can grant approvals but can never revoke them.

### Finding Description
In `remove_approval` (line 1438–1460), the function builds a synthetic `ApproveArgs` to reuse `icrc2_approve_not_async` as the revocation mechanism. It sets `spender: Account { owner: Principal::anonymous(), subaccount: None }` as a placeholder, intending the real spender to be supplied via the `override_spender` argument. However, `icrc2_approve_not_async` checks `from_account.owner == arg.spender.owner` at line 1328 before it ever consults `override_spender` (line 1331). When `caller()` is `Principal::anonymous()`, both sides of the comparison are `Principal::anonymous()`, the guard fires, and the canister traps with `"self approval is not allowed"`. The approval is never removed.

### Impact Explanation
The anonymous principal is a fully supported account holder on the ICP ledger (`can_send` returns `true` unconditionally; `test_anonymous_approval` confirms approvals succeed). Any approval the anonymous principal grants — including ones with no expiry — becomes permanently irrevocable through the `remove_approval` endpoint. A spender holding such an allowance can drain the anonymous principal's balance via `icrc2_transfer_from` at any future time, and the anonymous principal has no on-chain mechanism to stop it.

### Likelihood Explanation
The anonymous principal's account is accessible to any unauthenticated caller, so any party can deposit funds into it and grant approvals from it. The bug is deterministic and 100% reproducible: every `remove_approval` call from the anonymous principal traps. No special conditions, timing, or privileged access are required.

### Recommendation
Move the self-approval check in `icrc2_approve_not_async` to compare against the effective spender (i.e., `override_spender` when present) rather than `arg.spender.owner`. Alternatively, `remove_approval` should not use `Principal::anonymous()` as the dummy spender; it should use a sentinel value that cannot collide with any real caller, or restructure the revocation path to bypass the self-approval guard entirely.
