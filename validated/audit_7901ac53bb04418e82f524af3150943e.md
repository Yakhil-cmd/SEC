Audit Report

## Title
Unbounded `ingress_expiries` Loop Allows OOM DoS via Attacker-Controlled `ingress_start`/`ingress_end` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The `construction_payloads` handler in ICP Rosetta accepts attacker-supplied `ingress_start` and `ingress_end` nanosecond timestamps from the JSON request body with no server-side span cap. The while-loop that builds `ingress_expiries` iterates `(ingress_end − ingress_start) / interval` times with no bound check, allowing a single unauthenticated HTTP POST to exhaust Rosetta process memory and crash the node.

## Finding Description
The step size `interval` is computed at line 59–60 as `MAX_INGRESS_TTL − PERMITTED_DRIFT − 120 s = 300 s − 60 s − 120 s = 120 s = 120 000 000 000 ns`. [1](#0-0) 

`ingress_start` and `ingress_end` are deserialized directly from the JSON body as plain `Option<u64>` fields with no validation or clamping before use: [2](#0-1) [3](#0-2) 

The loop then runs without any cap: [4](#0-3) 

With `ingress_start = 0` and `ingress_end = u64::MAX = 18 446 744 073 709 551 615`, the loop executes ≈ 153 722 867 280 iterations, each pushing one `u64` (8 bytes) → ≈ 1.23 TB for `ingress_expiries` alone. The `add_payloads` function then iterates over `ingress_expiries` for every transaction and pushes **two** `SigningPayload` structs per entry, multiplying the allocation further: [5](#0-4) 

The ICRC1 sibling has a partial guard (`ingress_start >= ingress_end` check) but still lacks an upper-bound cap on the span, confirming the pattern is a known gap: [6](#0-5) 

There is no authentication, rate limiting, or span cap anywhere in the ICP Rosetta path.

## Impact Explanation
A single unauthenticated POST to `/construction/payloads` causes the Rosetta process to attempt allocating hundreds of gigabytes to terabytes of memory, triggering an OOM kill or indefinite CPU spin. This crashes the Rosetta node, denying service to all users relying on it for ICP transfers and neuron management. This matches the allowed High impact: **"Application/platform-level DoS, crash … or subnet availability impact"** and **"Significant … Rosetta … security impact with concrete user or protocol harm"** ($2,000–$10,000). The ICP Rosetta API is explicitly in-scope under Financial integrations.

## Likelihood Explanation
The Rosetta HTTP endpoint is publicly reachable by design with no authentication or API key. The exploit requires a single HTTP request with two integer fields set to extreme values. It is trivially reproducible by any unauthenticated client. The attack can be repeated indefinitely after each OOM kill, making sustained denial of service straightforward.

## Recommendation
Add a server-side cap on the ingress window span before entering the loop:

```rust
const MAX_INGRESS_WINDOW: Duration = Duration::from_secs(24 * 3600);
if ingress_end > ingress_start + MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end exceeds ingress_start by more than 24 hours"
    ));
}
```

Alternatively, cap `ingress_expiries` length to a fixed constant (e.g., `MAX_INGRESS_TTL / interval * 24 * 3 ≈ 2160` entries) and return an error if the requested span exceeds it. The same fix should be applied to the ICRC1 sibling in `rs/rosetta-api/icrc1/src/construction_api/services.rs`. [4](#0-3) 

## Proof of Concept
```bash
curl -s -X POST http://<rosetta-host>:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION",
      "account":{"address":"<valid-account>"},
      "amount":{"value":"-100","currency":{"symbol":"ICP","decimals":8}}}],
    "public_keys": [{"hex_bytes":"<valid-pubkey>","curve_type":"edwards25519"}],
    "metadata": {
      "ingress_start": 0,
      "ingress_end": 18446744073709551615
    }
  }'
```

The Rosetta process will enter the unbounded loop and be killed by the OS OOM killer before returning a response. Confirm with `dmesg | grep -i oom` or by observing process memory growth with `watch -n0.1 'ps -o rss= -p <pid>'`. A deterministic unit test can also be written by calling `construction_payloads` directly with these metadata values and asserting it returns an error rather than attempting allocation.
