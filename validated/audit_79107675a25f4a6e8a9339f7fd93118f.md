### Title
Confidential Rule Lifecycle Metadata (`added_in_version` / `removed_in_version`) Leaked to Unprivileged Callers via `get_rules_by_incident_id` and `get_rule_by_id` — (`rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`)

---

### Summary

`RuleConfidentialityFormatter` only redacts `rule_raw` and `description` for non-disclosed rules. It never redacts `added_in_version` or `removed_in_version`. Both `get_rules_by_incident_id` and `get_rule_by_id` are `#[query]` methods that `inspect_message` never gates (it only fires for ingress update messages). Any anonymous caller can therefore retrieve version lifecycle metadata for confidential rules, violating the intended confidentiality model.

---

### Finding Description

**Step 1 — `inspect_message` does not gate query calls.**

`inspect_message` is an IC system hook that only runs for ingress **update** messages. Non-replicated query calls bypass it entirely. The canister defines:

```
UPDATE_METHODS = ["add_config", "disclose_rules"]
REPLICATED_QUERY_METHOD = "get_config"
``` [1](#0-0) 

The `else` branch of `inspect_message` traps all other update-mode calls, but `get_rules_by_incident_id` and `get_rule_by_id` are declared `#[query]`, so `inspect_message` is never invoked for them. [2](#0-1) 

**Step 2 — `RuleConfidentialityFormatter` does not redact version metadata.**

The formatter only nulls `rule_raw` and `description` when `disclosed_at.is_none()`. `added_in_version` and `removed_in_version` pass through unconditionally: [3](#0-2) 

**Step 3 — `IncidentGetter::get` constructs the full `OutputRuleMetadata` before formatting.**

The struct is built with all fields populated, then the formatter is called for unauthorized viewers — but the formatter leaves `added_in_version` and `removed_in_version` intact: [4](#0-3) 

**Step 4 — `incident_id` is obtainable from `get_config`.**

`ConfigConfidentialityFormatter` only redacts `rule_raw` and `description`; it does not redact `incident_id` from `OutputRule`. Any caller can call `get_config` as a query and harvest `incident_id` values for all rules, including non-disclosed ones: [5](#0-4) 

The `OutputRule` struct includes `incident_id` as a non-optional field: [6](#0-5) 

**Step 5 — The existing test confirms the leak.**

The unit test for `IncidentGetter` with `AccessLevel::RestrictedRead` asserts that `added_in_version: 1` and `removed_in_version: Some(3)` are returned for a non-disclosed rule (`disclosed_at: None`): [7](#0-6) 

The same leak exists in `RuleGetter` / `get_rule_by_id`: [8](#0-7) 

---

### Impact Explanation

An unprivileged (including anonymous) caller can:
1. Call `get_config` (query) → collect `incident_id` values for all rules, including confidential ones.
2. Call `get_rules_by_incident_id(incident_id)` (query) → receive `added_in_version` and `removed_in_version` for every rule linked to that incident, even non-disclosed ones.

This reveals **when** each confidential rule was introduced into the config and **when** it was retired, enabling inference of incident timing, duration, and scope — information the confidentiality model is explicitly designed to protect.

---

### Likelihood Explanation

The exploit requires no credentials, no privileged role, and no network-level attack. Any IC user can send query calls to the canister. The `incident_id` prerequisite is trivially satisfied via `get_config`. The behavior is confirmed by the existing test suite itself.

---

### Recommendation

Extend `RuleConfidentialityFormatter::format` to also null `added_in_version` (set to a sentinel such as `0`) and `removed_in_version` (set to `None`) when `disclosed_at.is_none()`:

```rust
if rule.disclosed_at.is_none() {
    rule.description = None;
    rule.rule_raw = None;
    rule.added_in_version = 0;       // redact
    rule.removed_in_version = None;  // redact
}
```

Apply the same fix to `ConfigConfidentialityFormatter` if `incident_id` should also be hidden for non-disclosed rules in `get_config` responses. [9](#0-8) 

---

### Proof of Concept

```rust
// Existing test already proves this — getter.rs test_get_incident_success
// RestrictedRead caller receives:
// rule_1: { rule_raw: None, description: None, added_in_version: 1, removed_in_version: Some(3) }
// The version metadata is NOT redacted despite the rule being non-disclosed.

let getter_unauthorized = IncidentGetter::new(
    canister_state,
    RuleConfidentialityFormatter,
    create_mock_access_resolver(AccessLevel::RestrictedRead),
);
let response = getter_unauthorized.get(&incident_id.0.to_string()).unwrap();
// added_in_version and removed_in_version are present for the non-disclosed rule
assert_eq!(response[0].added_in_version, 1);
assert_eq!(response[0].removed_in_version, Some(3));
assert_eq!(response[0].rule_raw, None); // correctly redacted
``` [10](#0-9)

impl ConfidentialityFormatting for ConfigConfidentialityFormatter {
    type Input = OutputConfig;

    fn format(&self, config: OutputConfig) -> OutputConfig {
        let mut config = config;
        config.is_redacted = true;
        // Redact (hide) fields of non-disclosed rules
        config.rules.iter_mut().for_each(|rule| {
            if rule.disclosed_at.is_none() {
                rule.description = None;
                rule.rule_raw = None;
            }
        });
        config
    }
