### Title
`receive_base_tokens()` Skips Silo Address-Whitelist Check That `receive_erc20_tokens()` Enforces, Permanently Freezing ETH in Non-Whitelisted Addresses - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode with the `Address` whitelist enabled, `receive_erc20_tokens()` redirects tokens to a configured fallback address when the intended recipient is not whitelisted. The parallel path `receive_base_tokens()`, called when the ETH connector deposits native ETH, performs no such check. Any caller can therefore mint base ETH to an arbitrary non-whitelisted EVM address. Because that address is blocked from submitting any EVM transaction by the same whitelist, the deposited ETH is permanently frozen with no on-chain recovery path.

---

### Finding Description

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to one of two handlers depending on whether the predecessor is the ETH connector: [1](#0-0) 

When the predecessor **is** the ETH connector, `receive_base_tokens()` is called: [2](#0-1) 

`receive_base_tokens()` reads the recipient address from the transfer message and unconditionally mints ETH to it. There is no silo whitelist check of any kind.

When the predecessor is **any other NEP-141 token**, `receive_erc20_tokens()` is called instead: [3](#0-2) 

`receive_erc20_tokens()` explicitly calls `silo::is_allow_receive_erc20_tokens()` and, if the recipient is not whitelisted, redirects the tokens to the configured `erc20_fallback_address`. `receive_base_tokens()` contains no equivalent guard.

The silo `Address` whitelist is checked in two places that gate all subsequent use of the minted ETH:

- `is_allow_submit` — blocks every EVM transaction from a non-whitelisted address.
- `is_allow_receive_erc20_tokens` — blocks ERC-20 minting (already guarded). [4](#0-3) 

Because submitting an EVM transaction is the only on-chain mechanism to move or exit base ETH (via `ExitToNear` or `ExitToEthereum` precompiles), ETH minted to a non-whitelisted address in Silo mode is irrecoverable without an out-of-band admin action to add that address to the whitelist.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Base ETH minted to a non-whitelisted address in Silo mode cannot be spent, transferred, or bridged out. The address is blocked from submitting any EVM transaction by `is_allow_submit`, which gates the only available exit paths (`ExitToNear`, `ExitToEthereum`). The ETH is permanently frozen unless an admin manually whitelists the address — an action that may never occur and that the depositor cannot compel.

---

### Likelihood Explanation

**Medium.**

Silo mode with an active `Address` whitelist is the intended production configuration for permissioned Aurora deployments. Any user who can call `ft_transfer_call` on the ETH connector NEP-141 contract — which is a standard, public NEP-141 interface — can trigger this path by specifying any non-whitelisted EVM address as the recipient. No special privilege is required. The depositor may be acting in good faith (e.g., sending ETH to a freshly created address not yet added to the whitelist), making accidental freezes likely in addition to deliberate ones.

---

### Recommendation

Apply the same silo whitelist guard in `receive_base_tokens()` that already exists in `receive_erc20_tokens()`. If the recipient address is not whitelisted and a fallback address is configured, redirect the ETH mint to the fallback address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut recipient = message_data.recipient;

    // Mirror the silo guard from receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
    {
        recipient = fallback_address;
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    sdk::log!("Mint {amount} base tokens for: {}", recipient.encode());
    Ok(None)
}
```

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a non-zero `fixed_gas` and a valid `erc20_fallback_address`.
2. Enable the `Address` whitelist: call `set_whitelist_status` with `kind = Address, active = true`.
3. Do **not** add victim address `0xVICTIM` to the whitelist.
4. As any NEAR account, call `ft_transfer_call` on the ETH connector NEP-141 contract with:
   - `receiver_id` = Aurora Engine account
   - `amount` = any positive amount
   - `msg` = hex encoding of `0xVICTIM`
5. The ETH connector calls `ft_on_transfer` on Aurora Engine. Because `predecessor == connector_account_id`, `receive_base_tokens()` is invoked.
6. `receive_base_tokens()` mints ETH to `0xVICTIM` with no whitelist check.
7. Attempt to submit any EVM transaction from `0xVICTIM`. The call is rejected with `NotAllowed` because `is_allow_submit` checks the `Address` whitelist.
8. Attempt to call `ExitToNear` or `ExitToEthereum` from `0xVICTIM`. Both require submitting an EVM transaction and are therefore also blocked.
9. The ETH is permanently frozen at `0xVICTIM`.

Contrast with step 4 using an ERC-20 NEP-141 token instead of the ETH connector: `receive_erc20_tokens()` is called, detects `0xVICTIM` is not whitelisted, and redirects the tokens to `erc20_fallback_address` — no freeze occurs. [2](#0-1) [3](#0-2) [1](#0-0) [5](#0-4)

pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
