### Title
No Callback on `ExitToEthereum` Precompile Causes Permanent Fund Loss When Connector `withdraw` Fails - (File: engine-precompiles/src/native.rs)

### Summary
The `ExitToEthereum` precompile burns user funds (ETH or ERC-20 tokens) inside the Aurora EVM and schedules a cross-contract `withdraw` call to the ETH connector, but registers **no callback and no refund path**. If the connector's `withdraw` call fails for any reason—including concurrent exits that temporarily exhaust the connector's balance—the user's funds are permanently destroyed with no recovery mechanism.

### Finding Description
In `ExitToEthereum::run()`, after the EVM deducts the user's ETH or burns their ERC-20 tokens, the precompile constructs a promise:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

This is always `PromiseArgs::Create` — a fire-and-forget call with **no callback**. There is no `PromiseArgs::Callback` wrapping, no `exit_to_ethereum_precompile_callback`, and no refund logic anywhere in the codebase for this precompile.

Contrast this with `ExitToNear`, which, when the `error_refund` feature is compiled in, wraps the transfer promise with a `PromiseWithCallbackArgs` pointing to `exit_to_near_precompile_callback`:

```rust
PromiseArgs::Callback(PromiseWithCallbackArgs {
    base: transfer_promise,
    callback: PromiseCreateArgs {
        method: "exit_to_near_precompile_callback".to_string(),
        ...
    },
})
``` [2](#0-1) 

That callback invokes `engine::refund_on_error` to re-mint burned ERC-20 tokens or return ETH from the precompile address: [3](#0-2) 

`ExitToEthereum` has **no equivalent**. The `ExitToNear` refund path is itself gated on the `error_refund` compile-time feature, and even when enabled it only covers the NEAR-exit direction. The Ethereum-exit direction is unconditionally unprotected.

The race-condition scenario that mirrors the original report:

1. Multiple users simultaneously call `ExitToEthereum` (flag `0x00` for ETH, flag `0x01` for ERC-20).
2. Each EVM execution atomically deducts the user's balance or burns their ERC-20 tokens.
3. Each execution schedules a `withdraw` cross-contract call to the ETH connector.
4. The connector processes these calls sequentially. If its internal balance or storage state is insufficient to satisfy all concurrent withdrawals (e.g., due to an accounting lag, a concurrent deposit reversal, or a connector-side invariant violation), some `withdraw` calls fail.
5. Because there is no callback registered for `ExitToEthereum`, the failed calls produce no on-chain effect. The user's Aurora-side funds are already gone; the Ethereum-side funds never arrive.

The entry point is the `ExitToEthereum` precompile at address `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`, callable by any unprivileged EVM user or contract: [4](#0-3) 

### Impact Explanation
- **ETH exit**: the ETH value is transferred to the precompile's EVM address via `context.apparent_value` before the promise is created. If `withdraw` fails, that ETH is permanently stranded at the precompile address with no owner able to recover it — **permanent fund freeze**.
- **ERC-20 exit**: the ERC-20 contract burns the tokens before calling the precompile (the precompile receives `context.caller == erc20_address` and `amount` from the burn). If `withdraw` fails, the tokens are permanently destroyed — **permanent fund loss / insolvency**. [5](#0-4)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
}
