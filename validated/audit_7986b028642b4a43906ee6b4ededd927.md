### Title
Missing Storage Collision Check Allows CREATE/CREATE2 Deployment Over Accounts With Pre-Existing Storage — (`File: evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter does not check whether the target address of a `CREATE` or `CREATE2` operation has non-empty storage before proceeding with deployment. The EVM specification (EIP-7610) requires that deployment must fail if the target address has non-empty storage. Because this check is absent, an attacker can pre-populate storage at a deterministic `CREATE2` address and cause a victim's contract to be deployed on top of attacker-controlled state, corrupting the contract from its first moment of existence.

---

### Finding Description

The collision guard in `before_executing_frame` only checks `deployee_code_len != 0 || deployee_nonce != 0`. The code itself contains an explicit acknowledgment that the storage check is missing:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    ...
    tracer.evm_tracer().on_call_error(&EvmError::CreateCollision);
    return Ok(false);
}
``` [1](#0-0) 

The omission is confirmed by multiple disabled EVM test cases across all test index files, each carrying the comment `"We do not check storage for collision"` or `"We do not check for storage collisions"`: [2](#0-1) [3](#0-2) [4](#0-3) 

EIP-7610 (`eip7610_create_collision`) tests are also explicitly disabled: [5](#0-4) 

The `CREATE`/`CREATE2` address derivation is standard and deterministic: [6](#0-5)

### Impact Explanation

An attacker who can predict or control the `CREATE2` address of a future deployment can:

1. Pre-write storage slots at that address (e.g., via a contract that writes to storage and then self-destructs, leaving storage behind).
2. When the victim factory later deploys to that address via `CREATE2`, ZKsync OS proceeds with deployment because `code_len == 0` and `nonce == 0` — the only two checks performed.
3. The newly deployed contract inherits the attacker-controlled storage values from the moment of deployment.

Concrete damage: if the deployed contract uses storage slot 0 for an `owner` or `admin` variable (extremely common in OpenZeppelin-style contracts), the attacker can pre-set it to their own address, giving themselves ownership of the contract from block 1. Any funds deposited into or managed by that contract are immediately at risk.

This is a **state-transition bug / EVM semantic mismatch** with a direct path to public funds loss.

---

### Likelihood Explanation

`CREATE2` addresses are fully deterministic and publicly computable from `(deployer, salt, initcode_hash)`. Any factory contract that uses a fixed salt or a predictable salt (e.g., derived from a user-supplied token address) is vulnerable. The attacker only needs to:

- Observe or predict the factory's next `CREATE2` parameters (trivial for open-source factory contracts).
- Deploy a contract to that address first, write storage, then self-destruct (leaving storage intact, nonce reset to 0, code cleared).
- Wait for the victim factory to deploy.

This is a realistic, low-cost attack requiring only standard EVM transactions. No privileged access, oracle manipulation, or governance majority is needed.

---

### Recommendation

Implement the EIP-7610 storage-emptiness check before allowing `CREATE`/`CREATE2` to proceed. Before executing the constructor frame, query whether the target address has any non-zero storage slots. If any exist, treat it as a collision and return `EvmError::CreateCollision`, burning all gas as is done for the existing code/nonce collision path.

The check must be added in `before_executing_frame` in `evm_interpreter/src/ee_trait_impl.rs`, alongside the existing `deployee_code_len != 0 || deployee_nonce != 0` guard. [1](#0-0) 

---

### Proof of Concept

**Setup:**

```solidity
// Attacker contract
contract Poisoner {
    function poison(address target) external {
        // Write to storage slot 0 at the target address
        assembly {
            sstore(0, caller()) // slot 0 = attacker address
        }
        selfdestruct(payable(msg.sender)); // clear code+nonce, leave storage
    }
}
```

**Step 1:** Attacker computes the CREATE2 address that `VictimFactory` will deploy to:
```
target = keccak256(0xff ++ factory_addr ++ salt ++ keccak256(initcode))[12:]
```

**Step 2:** Attacker deploys `Poisoner` at `target` (using CREATE2 with matching parameters from a different deployer, or via a helper), calls `poison(target)`, which writes `slot[0] = attacker_address` and self-destructs. After self-destruct: `code = empty`, `nonce = 0`, `storage[0] = attacker_address`.

**Step 3:** Victim calls `VictimFactory.deploy(salt)`. ZKsync OS checks `code_len == 0` ✓ and `nonce == 0` ✓, so deployment proceeds. The victim's contract is deployed with `storage[0]` already set to the attacker's address.

**Step 4:** If the victim contract is an `Ownable`-style contract reading `owner` from slot 0, the attacker is now the owner and can drain any funds.

On Ethereum mainnet, Step 3 would revert with `CreateCollision` per EIP-7610. On ZKsync OS, it succeeds.

    pub fn derive_address_for_deployment_create2(
        system: &mut System<S>,
        resources: &mut <S as SystemTypes>::Resources,
        salt: &U256,
        deployer_address: &<S::IOTypes as SystemIOTypesConfig>::Address,
        deployment_code: &[u8],
    ) -> Result<<S::IOTypes as SystemIOTypesConfig>::Address, EvmSubsystemError> {
        use crypto::sha3::{Digest, Keccak256};
        // we need to compute address based on the hash of the code and salt
        let mut initcode_hash = ArrayBuilder::default();
        resources
            .with_infinite_ergs(|inf_resources| {
                S::SystemFunctions::keccak256(
                    deployment_code,
                    &mut initcode_hash,
                    inf_resources,
                    system.get_allocator(),
                )
            })
            .map_err(|e| -> EvmSubsystemError {
                match e.root_cause() {
                    RootCause::Runtime(e @ RuntimeError::FatalRuntimeError(_)) => {
                        e.clone_or_copy().into()
                    }
                    _ => internal_error!("Keccak in create2 cannot fail").into(),
                }
            })?;
        let initcode_hash = Bytes32::from_array(initcode_hash.build());

        let mut create2_buffer = [0xffu8; 1 + 20 + 32 + 32];
        create2_buffer[1..(1 + 20)]
            .copy_from_slice(&deployer_address.to_be_bytes::<{ B160::BYTES }>());
        create2_buffer[(1 + 20)..(1 + 20 + 32)]
            .copy_from_slice(&salt.to_be_bytes::<{ U256::BYTES }>());
        create2_buffer[(1 + 20 + 32)..(1 + 20 + 32 + 32)]
            .copy_from_slice(initcode_hash.as_u8_array_ref());

        let new_address = Keccak256::digest(&create2_buffer);
        #[allow(deprecated)]
        let new_address =
            B160::try_from_be_slice(&new_address.as_slice()[12..]).expect("must create address");

        Ok(new_address)
    }
