### Title
EIP-7623 Floor Gas Not Enforced for L1→L2 Transactions Enables Underpayment for L2 Resources - (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

When the `eip_7623` feature is enabled, `calculate_tx_intrinsic_gas` uses `CALLDATA_TOKEN_GAS_COST = 4` per token for L1→L2 transactions, while the EIP-7623 floor is `TOTAL_COST_FLOOR_PER_TOKEN = 10` per token. The `minimal_gas_used` (EIP-7623 floor) is computed but then **silently capped at `gas_limit`** for L1 transactions rather than enforced. This means an attacker can submit L1→L2 transactions with large calldata and a `gas_limit` set just above the old 4-per-token intrinsic cost but below the EIP-7623 floor, consuming L2 resources for less than the protocol-mandated minimum cost.

---

### Finding Description

In `prepare_and_check_resources` inside `process_l1_transaction.rs`, the following sequence occurs:

**Step 1** — `compute_calldata_tokens` returns the EIP-7623 floor gas:

```
minimal_gas_used = TX_INTRINSIC_GAS + tokens * TOTAL_COST_FLOOR_PER_TOKEN
                 = 21_000 + tokens * 10
``` [1](#0-0) 

**Step 2** — `calculate_tx_intrinsic_gas` computes the actual intrinsic gas deducted from resources using the **old** 4-per-token cost:

```
intrinsic_gas = TX_INTRINSIC_GAS + tokens * CALLDATA_TOKEN_GAS_COST
              = 21_000 + tokens * 4
``` [2](#0-1) [3](#0-2) 

**Step 3** — `minimal_gas_used` is **silently capped** at `gas_limit` instead of being enforced:

```rust
if gas_limit < minimal_gas_used {
    system_log!(system, "L1 tx gas limit below intrinsic cost, using saturated arithmetic instead");
}
let minimal_gas_used = minimal_gas_used.min(gas_limit);
``` [4](#0-3) 

Compare this to L2 transactions, where the EIP-7623 floor is a hard rejection:

```rust
#[cfg(feature = "eip_7623")]
require!(
    minimal_gas_used <= tx_gas_limit,
    InvalidTransaction::EIP7623IntrinsicGasIsTooLow,
    system
)?;
``` [5](#0-4) 

**Step 4** — In `compute_gas_refund`, `minimal_gas_used` is used as a floor for `gas_used`:

```rust
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);
``` [6](#0-5) 

Because `minimal_gas_used` was already capped at `gas_limit`, when `gas_limit < EIP-7623 floor`, the operator is paid for only `gas_limit` gas — not the EIP-7623 floor. The attacker paid `gas_limit × gas_price` on L1, which is less than the floor cost.

The code comment explicitly acknowledges this dependency on L1 contracts:

> *"Note that the 'validation errors' are practically unreachable, as gas_limit, gas_price and gas_per_pubdata are either checked or set by the L1 contracts. We decide to handle these cases as a fallback in case the L1 contracts aren't properly updated to reflect a change in ZKsync OS."* [7](#0-6) 

---

### Impact Explanation

If the ZKsync L1 Mailbox/Diamond contracts are not updated to account for EIP-7623 when computing the minimum `gas_limit` for L1→L2 transactions (analogous to Optimism's `minimumGasLimit()` issue), an attacker can:

1. Submit L1→L2 transactions with large calldata and `gas_limit` in the range `[21000 + tokens×4, 21000 + tokens×10)`.
2. The transaction is accepted on L1 (L1 contracts use old formula).
3. On L2, the transaction has only `gas_limit − intrinsic_gas` gas for execution — potentially just a few units — and fails with out-of-gas.
4. The operator is paid for `gas_limit` gas, which is below the EIP-7623 floor. The attacker consumed L2 bootloader resources (hash computation, frame setup, refund logic) for less than the protocol-mandated minimum cost.
5. Repeated at scale, this constitutes a resource-underpriced DoS against the L2 sequencer.

**Concrete example** with 1000 non-zero calldata bytes (`tokens = 4000`):
- Old intrinsic gas: `21000 + 4000×4 = 37000`
- EIP-7623 floor: `21000 + 4000×10 = 61000`
- Attacker sets `gas_limit = 37001` → pays for 37001 gas on L1
- L2 execution gas available: `37001 − 37000 = 1` → immediate OOG failure
- Operator paid 37001 gas; EIP-7623 floor is 61000 gas → **39% underpayment**

---

### Likelihood Explanation

The ZKsync OS codebase itself acknowledges that L1 contract validation is the intended gatekeeper for this invariant. If the ZKsync L1 contracts have not been updated to use the EIP-7623 calldata cost formula (10 per token instead of 4 per non-zero byte / 1 per zero byte) in their minimum gas limit computation, this path is directly reachable by any unprivileged L1 transaction sender. The Optimism ecosystem encountered exactly this class of omission post-Pectra, making it a realistic and precedented failure mode.

---

### Recommendation

1. **Update ZKsync L1 contracts**: The Mailbox/Diamond `minimumGasLimit` (or equivalent) function must be updated to use the EIP-7623 floor formula (`max(21000 + tokens×4, 21000 + tokens×10)`) when computing the minimum acceptable `gas_limit` for L1→L2 transactions.

2. **Enforce the floor in ZKsync OS**: Rather than silently capping `minimal_gas_used` at `gas_limit`, ZKsync OS should enforce the EIP-7623 floor for L1 transactions as well. Since L1 transactions cannot be invalidated, the correct behavior is to treat a `gas_limit` below the floor as if `gas_limit = minimal_gas_used` for resource allocation purposes, ensuring the operator is always compensated at the floor rate.
