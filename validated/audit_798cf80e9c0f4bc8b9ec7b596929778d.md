### Title
Missing Access Control in `ft_on_transfer` Enables Unbacked ERC-20 Token Minting — (File: engine/src/contract_methods/connector.rs)

---

### Summary

`ft_on_transfer` is a public NEAR entry point on the Aurora Engine that is intended to be called only by legitimate NEP-141 token contracts as a callback after `ft_transfer_call`. It contains no check that the caller is a registered, trusted token contract, and no check that actual tokens were transferred. Combined with the equally unrestricted `deploy_erc20_token` function, any unprivileged NEAR account can mint arbitrary amounts of ERC-20 tokens on Aurora without depositing any NEP-141 tokens.

---

### Impact Explanation

An attacker can mint an unbounded quantity of ERC-20 tokens on Aurora that are supposed to be backed 1:1 by NEP-141 tokens, but with zero actual deposit. These tokens are indistinguishable from legitimately bridged tokens and can be:

- Swapped on Aurora DEXes for real assets, draining liquidity pools.
- Deposited as collateral in Aurora lending protocols to borrow real assets.
- Transferred back to NEAR via the bridge if the bridge does not independently verify backing.

This constitutes **direct theft of user funds** and **insolvency** of any Aurora DeFi protocol that accepts the affected ERC-20 token. Impact: **Critical**.

---

### Likelihood Explanation

The attack requires only a standard NEAR account and the ability to call two public contract functions. No privileged keys, governance access, or oracle manipulation is needed. The steps are deterministic and can be executed in two transactions. Likelihood: **High**.

---

### Recommendation

Add caller validation to `ft_on_transfer` analogous to how TBTC fixed `requestNewKeep`: verify that `predecessor_account_id` is a registered, approved NEP-141 token contract before minting. Concretely:

1. Maintain an allowlist of NEP-141 token accounts that are permitted to call `ft_on_transfer` (populated only via `deploy_erc20_token`, which itself should require owner authorization).
2. At the top of `ft_on_transfer`, reject any call whose `predecessor_account_id` is not in that allowlist (and is not the connector account).

This mirrors the TBTC fix: checking that `msg.sender` is an existing `TBTCDepositToken` before allowing `requestNewKeep` to proceed.

---

### Proof of Concept

1. Attacker creates NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token` on Aurora Engine with `attacker.near` as the NEP-141 token. No access control prevents this. Aurora deploys an ERC-20 and stores the mapping `attacker.near → <erc20_address>`.
3. Attacker calls `ft_on_transfer` directly on Aurora Engine (public NEAR method) with JSON input:
   ```json
   {"sender_id": "attacker.near", "amount": "1000000000000000000", "msg": "<attacker_evm_address_hex>"}
   ```
4. Inside `ft_on_transfer`, `predecessor_account_id = attacker.near` ≠ connector account, so `receive_erc20_tokens` is called with `token = attacker.near` and `amount = 1e18`.
5. `get_erc20_from_nep141` succeeds (mapping exists from step 2). The ERC-20 contract is called with `erc20_admin_address` as the caller, minting `1e18` tokens to the attacker's EVM address.
6. Attacker holds `1e18` unbacked ERC-20 tokens and can swap them for real assets on any Aurora DEX. [5](#0-4) [6](#0-5)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
