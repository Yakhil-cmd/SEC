Audit Report

## Title
SNS Governance `register_vote` Off-by-One Allows Vote and WFQ Extension at Exact Deadline Second While Proposal Is `ReadyToSettle` - (`rs/sns/governance/src/governance.rs`)

## Summary
In SNS governance, `register_vote` rejects late votes only when `now_seconds > deadline`, while `accepts_vote` uses `now_seconds < deadline`. At the exact second `now_seconds == deadline`, `register_vote` accepts the vote while `accepts_vote` simultaneously returns `false`, causing `reward_status` to return `ReadyToSettle`. Additionally, `evaluate_wait_for_quiet`'s guard (`now_seconds > current_deadline`) also fails to fire at the boundary, meaning a vote cast at the exact deadline second can flip the tally and trigger a wait-for-quiet deadline extension on a proposal that the protocol simultaneously considers expired and eligible for reward settlement.

## Finding Description
**Root cause:** Three boundary comparisons in SNS governance are mutually inconsistent at `now_seconds == deadline`:

1. `register_vote` (`rs/sns/governance/src/governance.rs`, line 3916): uses `now_seconds > deadline`, so the check passes (no error) when `now_seconds == deadline`.
2. `accepts_vote` (`rs/sns/governance/src/proposal.rs`, line 2102): uses `now_seconds < deadline`, returning `false` when `now_seconds == deadline`.
3. `evaluate_wait_for_quiet` (`rs/sns/governance/src/proposal.rs`, line 2136): skips extension only when `now_seconds > current_deadline`, so the guard does not fire when `now_seconds == deadline`.

**Exploit flow:**

- A neuron controller observes a proposal's deterministic `deadline = T` from public state.
- At `now_seconds = T`, the controller submits a `register_vote` ingress. The `> deadline` check passes; the vote is recorded and `cast_vote_and_cascade_follow` executes.
- Simultaneously, `reward_status(T)` returns `ReadyToSettle` (because `accepts_vote(T)` returns `false`), so `ready_to_be_settled_proposal_ids` includes this proposal.
- If the vote flips the tally at `now_seconds = T`, `evaluate_wait_for_quiet` is invoked. Because `T > T` is false, the WFQ guard does not skip extension. The deadline is extended beyond `T`, contradicting the fact that `accepts_vote` already declared the proposal expired at `T`.
- If `distribute_rewards` also runs at `now_seconds = T` (it is called periodically and can coincide), the proposal is settled while the boundary vote is being processed, creating a race between vote recording and reward settlement.

**Why existing checks fail:** NNS governance avoids this entirely by delegating the deadline check in `register_vote` directly to `accepts_vote` (lines 5630–5637 of `rs/nns/governance/src/governance.rs`), making the two checks structurally identical. SNS governance duplicates the check with a different operator, introducing the one-second inconsistency window.

## Impact Explanation
The concrete impact is twofold. First, a neuron controller can trigger a wait-for-quiet deadline extension on a proposal that the SNS governance protocol simultaneously considers expired and `ReadyToSettle`. This allows an attacker to extend the voting period of a governance proposal beyond its stated deadline, potentially altering the outcome of a vote by giving additional time for follow-on votes to accumulate. Second, if `distribute_rewards` coincides with the boundary second, a vote is recorded against a proposal that is being settled, creating an inconsistency between the recorded ballot state and the rewarded state. Both impacts affect SNS governance correctness and constitute a significant SNS governance security impact with concrete protocol harm, qualifying as High severity under the "Significant SNS security impact with concrete user or protocol harm" category.

## Likelihood Explanation
IC consensus time advances in deterministic one-second increments. Every proposal's `deadline` is publicly readable from canister state. Any neuron controller — requiring no special privileges, no key compromise, and no majority — can observe the deadline and submit a `register_vote` ingress timed to arrive in the consensus round where `env.now() == deadline`. The IC's predictable block timing makes this boundary reliably targetable. The WFQ extension arm additionally requires the vote to flip the tally, which constrains the attack to scenarios where the attacker holds or coordinates sufficient voting power to flip the result, but this is a normal capability of any sufficiently large neuron holder.

## Recommendation
Change the deadline check in SNS `register_vote` from strict-greater-than to greater-than-or-equal, aligning it with `accepts_vote`:

```diff
- if now_seconds > deadline {
+ if now_seconds >= deadline {
```

Alternatively, delegate directly to `accepts_vote` as NNS does:

```rust
if !proposal.accepts_vote(now_seconds) {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Proposal deadline has passed.",
    ));
}
```

Also fix `evaluate_wait_for_quiet`'s guard from `now_seconds > current_deadline` to `now_seconds >= current_deadline` so that WFQ extension cannot fire once the deadline is reached.

## Proof of Concept
A deterministic unit test suffices. Given a `ProposalData` with `proposal_creation_timestamp_seconds = 0`, `initial_voting_period_seconds = T`, and no WFQ state (so `deadline = T`):

1. Assert `proposal.accepts_vote(T - 1)` returns `true` and `reward_status(T - 1)` returns `AcceptVotes`.
2. Assert `proposal.accepts_vote(T)` returns `false` and `reward_status(T)` returns `ReadyToSettle`.
3. Invoke the `register_vote` code path with `now_seconds = T`; assert it does **not** return `Err("Proposal deadline has passed.")` — confirming the vote is accepted at the boundary.
4. Steps 2 and 3 together demonstrate the inconsistency: the same `now_seconds = T` simultaneously produces `ReadyToSettle` and a successful vote registration.
5. For the WFQ arm: construct a `ProposalData` with `wait_for_quiet_state` set and `current_deadline_timestamp_seconds = T`; call `evaluate_wait_for_quiet` with `now_seconds = T` and a tally that has flipped; assert the deadline is extended — confirming WFQ fires at the boundary despite `accepts_vote` returning `false`.
