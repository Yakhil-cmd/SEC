## Title
SNS Token Valuation Mixes Swap-Only Token Supply with Total Initial Supply in Exchange Rate Calculation - (File: `rs/sns/governance/token_valuation/src/lib.rs`)

## Summary
`IcpsPerSnsTokenClient::fetch_icps_per_sns_token` computes the ICP-per-SNS-token price by dividing an initial price denominated in swap tokens only by an inflation factor denominated in the total initial supply. Because `total_initial_supply > swap_tokens` always (developer, airdrop, and treasury allocations are also minted at genesis), the resulting valuation is inflated by `total_initial_supply / swap_tokens`. This inflated valuation is stored at proposal submission time and reused at execution time to enforce 7-day transfer and minting limits for `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals, causing legitimate proposals to be incorrectly rejected.

## Finding Description

**Root cause — mismatched denominators in `fetch_icps_per_sns_token`:**

`fetch_icps_per_sns_token` concurrently fetches three values:

1. `initial_sns_tokens_per_icp` — from `swap.get_derived_state()`. Inside the swap canister (`rs/sns/swap/src/swap.rs` lines 2992–2995), this is computed as `tokens_available_for_swap / participant_total_icp_e8s`, where `tokens_available_for_swap` is the swap allocation only — a strict subset of the total initial supply. [1](#0-0) 

2. `initial_supply_e8s` — from `initial_supply_e8s()`, which scans the SNS ledger and sums **all** mint transactions sharing the first block timestamp. This includes developer, airdrop, treasury, and swap allocations — the full total initial supply. [2](#0-1) 

3. `current_supply_e8s` — from `icrc1_total_supply`. [3](#0-2) 

The math assembled at lines 389–414:

```rust
let initial_icps_per_sns_token = 1 / initial_sns_tokens_per_icp;
//  = buyer_icp / swap_tokens   ← denominator is swap tokens only

let total_inflation = current_supply_e8s / initial_supply_e8s;
//  = current_supply / total_initial_supply  ← denominator is total supply

initial_icps_per_sns_token / total_inflation
//  = (buyer_icp / swap_tokens) / (current_supply / total_initial_supply)
//  = buyer_icp × total_initial_supply / (swap_tokens × current_supply)
//  ≠ buyer_icp / current_supply  (correct)
``` [4](#0-3) 

The result is inflated by the factor `total_initial_supply / swap_tokens`. For a typical SNS where the swap allocation is 20–50% of the total initial supply, this is a 2–5× overstatement.

**Why the existing unit test does not catch this:**

`test_icps_per_sns_token_client` mocks the ledger with a single mint transaction whose amount equals the entire `INITIAL_SNS_TOKEN_SUPPLY_E8S`, and the swap returns `sns_tokens_per_icp` derived from that same value. The test therefore never exercises the real-world case where the ledger's initial supply is larger than the swap's token allocation. [5](#0-4) 

**How the inflated valuation propagates to proposal enforcement:**

The `icps_per_token` field of `ValuationFactors` is set to the inflated value. `to_xdr()` multiplies `tokens × icps_per_token × xdrs_per_icp`, so the treasury's XDR valuation is inflated by the same factor. [6](#0-5) 

`from_valuation_xdr` classifies the treasury as small (≤ 100,000 XDR → NoLimit), medium (≤ 1,200,000 XDR → 25% of balance), or large (> 1,200,000 XDR → 300,000 XDR cap). An inflated XDR value pushes the treasury into a higher, more restrictive tier. [7](#0-6) 

In the `Xdr` (large) branch, `tokens_per_xdr` is derived from the inflated `icps_per_token`, so the token cap is further reduced. [8](#0-7) 

The valuation is stored in `action_auxiliary` at submission time and reused at execution time via `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err`, so both submission and execution checks use the wrong value. [9](#0-8) [10](#0-9) 

## Impact Explanation

**Impact: Medium.** The treasury valuation is inflated by `total_initial_supply / swap_tokens` (typically 2–5×). This causes treasuries that should be classified as "small" (NoLimit) to appear "medium" (25% cap) or "large" (300,000 XDR cap), and "medium" treasuries to appear "large". Legitimate `TransferSnsTreasuryFunds` and `MintSnsTokens` governance proposals are incorrectly rejected at both submission and execution time. No funds can be stolen or illegally minted — the bug only makes limits more restrictive than the protocol intends, constituting a governance denial-of-service for SNS DAOs. This matches the allowed impact: *Significant SNS security impact with concrete user or protocol harm.*

## Likelihood Explanation

Every SNS that mints developer, airdrop, or treasury tokens at genesis is affected — which is virtually all production SNS deployments. The swap allocation is always a strict subset of the total initial supply, so the inflation factor is always wrong. The bug is triggered by any governance participant submitting a `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal; no special privilege is required. No attacker action is needed — the miscalculation occurs automatically on every valuation call.

## Recommendation

Replace the two-step formula with a consistent denominator. The simplest correct form uses `buyer_total_icp_e8s` directly from the derived state response and divides by `current_supply_e8s`:

```rust
// Correct: buyer_icp / current_supply
let buyer_total_icp_e8s = i2d(get_derived_state_response.buyer_total_icp_e8s
    .ok_or_else(|| ...)?);
buyer_total_icp_e8s
    .checked_div(current_supply_e8s)
    .ok_or_else(|| ...)?
```

Alternatively, if the inflation-based framing is preferred, use the swap token supply as the baseline for both the initial price and the inflation factor:

```rust
let initial_icps_per_sns_token = 1 / initial_sns_tokens_per_icp; // buyer_icp / swap_tokens
let inflation = current_supply_e8s / swap_supply_e8s;            // consistent denominator
initial_icps_per_sns_token / inflation                            // = buyer_icp / current_supply
```

The unit test `test_icps_per_sns_token_client` should be updated to model a realistic SNS genesis where the ledger's initial supply is larger than the swap allocation (e.g., 1,000,000 total with 200,000 in the swap), and assert the correct `buyer_icp / current_supply` result.

## Proof of Concept

**Setup (matches the PoC in the claim):**
- Total initial supply: 1,000,000 SNS tokens (developer 400k + treasury 400k + swap 200k)
- Swap tokens: 200,000; ICP raised: 100,000 ICP
- Current supply (after 10% inflation): 1,100,000 SNS tokens
- Treasury balance: 300,000 SNS tokens; `xdrs_per_icp`: 10

**Correct price:** `100,000 / 1,100,000 ≈ 0.0909 ICP/token`
**Correct treasury XDR:** `300,000 × 0.0909 × 10 = 272,727 XDR` → Medium tier → 25% cap = 75,000 tokens/7 days

**Buggy price:** `(100,000 / 200,000) / (1,100,000 / 1,000,000) = 0.5 / 1.1 ≈ 0.4545 ICP/token`
**Buggy treasury XDR:** `300,000 × 0.4545 × 10 = 1,363,636 XDR` → Large tier → 300,000 XDR cap
**Buggy token cap:** `300,000 / (0.4545 × 10) ≈ 66,000 tokens/7 days`

A proposal to transfer 70,000 tokens (within the correct 75,000-token limit) is rejected because the buggy cap is 66,000 tokens.

**Reproducible test:** Extend `test_icps_per_sns_token_client` in `rs/sns/governance/token_valuation/src/tests.rs` to mock the ledger with multiple genesis mint transactions totalling 1,000,000 tokens (swap allocation = 200,000), set `sns_tokens_per_icp = 200,000 / 100,000 = 2.0`, set `current_supply = 1,100,000`, and assert the result equals `100,000 / 1,100,000 ≈ 0.0909`. The current code will instead return `≈ 0.4545`, demonstrating the 5× inflation.
