Audit Report

## Title
Missing Anonymous Principal Validation in Neuron Controller Assignment via `claim_or_refresh_neuron_by_memo_and_controller` - (File: rs/nns/governance/src/governance.rs)

## Summary
The NNS Governance canister's `claim_or_refresh_neuron_by_memo_and_controller` function accepts a caller-supplied `controller` principal without validating that it is not the anonymous principal. Because the IC protocol permits the anonymous principal to send ingress update calls (no signature required), any party can claim a neuron with the anonymous principal as its controller and subsequently exercise full neuron control from any anonymous identity, including casting NNS votes and disbursing the staked ICP.

## Finding Description
In `rs/nns/governance/src/governance.rs` at line 5858, the controller is resolved as:

```rust
let controller = memo_and_controller.controller.unwrap_or(*caller);
```

If `memo_and_controller.controller` is `Some(PrincipalId::new_anonymous())`, the anonymous principal is used directly with no rejection. The function then calls `claim_neuron(subaccount, controller, ...)` which persists this principal as the neuron's authoritative controller. No validation of the controller value occurs anywhere in the call chain: `manage_neuron_internal` dispatches directly to this function for `By::MemoAndController` without pre-validation, and `Neuron::set_controller` blindly assigns whatever principal is provided.

The IC protocol explicitly allows the anonymous principal to send ingress update calls without a signature. This is confirmed by the integration tests in `rs/nns/integration_tests/src/neuron_voting.rs`: `anonymous_principal_cannot_vote_on_nonexistent_proposal` and `anonymous_principal_cannot_vote_on_existent_proposal` show that anonymous ingress messages reach the governance canister and receive governance-level authorization errors (not protocol-level rejections). The error "Caller is not authorized to vote for neuron" is returned because the anonymous principal is not the controller of the target neuron — but if a neuron were claimed with the anonymous principal as controller, `is_controlled_by(anonymous_principal)` would return `true` and the authorization check would pass.

The identical missing validation exists in SNS Governance at `rs/sns/governance/src/governance.rs` line 4215.

## Impact Explanation
This matches the High bounty impact: **Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds**. An attacker can create a neuron whose controller is the anonymous principal. Because the anonymous principal requires no private key and any unauthenticated caller is treated as anonymous, the neuron's full management surface — `RegisterVote`, `Disburse`, `Follow`, `MakeProposal`, `Configure` — becomes exercisable by any party on the network. The staked ICP is at risk of disbursement, and the neuron's voting power can be wielded by any anonymous caller to influence NNS governance outcomes.

## Likelihood Explanation
The attack requires no privileged access, no key compromise, and no social engineering. The only prerequisite is funding the ledger subaccount computed from `(anonymous_principal, memo)` with at least `neuron_minimum_stake_e8s` ICP. The `ClaimOrRefresh` call can be submitted by any principal (including a non-anonymous one) with `controller: Some(anonymous_principal)`. Once claimed, any subsequent anonymous ingress call to `manage_neuron` targeting that neuron will pass the controller authorization check. The attack is repeatable and cheap relative to the governance impact of a well-funded neuron.

## Recommendation
Add an explicit rejection of the anonymous principal before assigning the controller in `claim_or_refresh_neuron_by_memo_and_controller` in both NNS and SNS Governance:

```rust
let controller = memo_and_controller.controller.unwrap_or(*caller);
if controller == PrincipalId::new_anonymous() {
    return Err(GovernanceError::new_with_message(
        ErrorType::InvalidPrincipal,
        "The anonymous principal cannot be set as a neuron controller.",
    ));
}
```

Apply the same guard in `rs/sns/governance/src/governance.rs` at the equivalent location. Consider adding a defense-in-depth check inside `Neuron::set_controller` as well.

## Proof of Concept
1. Compute `subaccount = compute_neuron_staking_subaccount(PrincipalId::new_anonymous(), memo=42)`.
2. Transfer `neuron_minimum_stake_e8s` ICP to the governance canister's subaccount address for that subaccount (can be done by any funded principal).
3. Submit an ingress `manage_neuron` call (as any principal) to NNS Governance with:
   ```
   ManageNeuron {
     command: ClaimOrRefresh {
       by: MemoAndController { memo: 42, controller: Some(<anonymous_principal>) }
     }
   }
   ```
4. The call succeeds; a neuron is created with `controller = anonymous_principal`.
5. Submit a subsequent `manage_neuron` call as the anonymous principal (e.g., `RegisterVote` or `Disburse`). The governance canister's `is_controlled_by` check passes because `neuron.controller() == anonymous_principal == caller`, and the operation executes successfully.

This can be reproduced as a unit test by adapting the existing `test_neuron_with_non_self_authenticating_controller_is_now_allowed` test in `rs/nns/governance/tests/governance.rs` to use `PrincipalId::new_anonymous()` as the controller and verifying that a subsequent `manage_neuron` call with the anonymous principal as caller succeeds.
