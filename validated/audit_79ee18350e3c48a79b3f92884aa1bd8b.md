### Title
Missing Storage Collision Check in CREATE/CREATE2 Deployment Allows Attacker-Controlled Storage Inheritance - (`evm_interpreter/src/ee_trait_impl.rs`)

### Summary

ZKsync OS's EVM interpreter explicitly skips the EIP-7610 storage-emptiness check during CREATE and CREATE2 deployments. An unprivileged attacker can pre-populate storage at a deterministic CREATE2 address (by deploying, writing storage, then selfdestructing), and when a victim later deploys a new contract to the same address, the new contract silently inherits the attacker-controlled storage. This is the ZKsync OS analog of the external report's vulnerability class: **insufficient collision/uniqueness checks in deterministic deployment leading to fund theft**.

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` function checks for address collision before executing a constructor, but explicitly omits the storage-emptiness check mandated by EIP-7610:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // collision → revert
}
``` [1](#0-0) 

The omission is confirmed across all test index files, where every EIP-7610 and storage-collision test case is explicitly disabled: [2](#0-1) [3](#0-2) 

The address derivation logic itself is correct — `derive_address_for_deployment_create2` uses `keccak256(0xff ++ deployer ++ salt ++ keccak256(initcode))` — but the post-derivation collision guard is incomplete: [4](#0-3) 

### Impact Explanation

**Vulnerability class**: EVM semantic mismatch / valid-execution unprovability.

Attack scenario:
1. Attacker computes the CREATE2 address `X` for a known future victim deployment (deployer `D`, salt `S`, initcode `I`).
2. Attacker deploys a contract to `X` (using the same `D`, `S`, `I` or any combination that resolves to `X`), writes malicious values to critical storage slots (e.g., `owner`, `balances[attacker]`), then selfdestructs the contract. After selfdestruct, `code_len == 0` and `nonce == 0`, but storage persists.
3. Victim deploys their contract to `X` via CREATE2. In standard EVM (EIP-7610), this would revert because storage is non-empty. In ZKsync OS, the check is absent, so the deployment succeeds.
4. The victim's newly deployed contract now has attacker-controlled storage values. If the contract uses those slots for access control, token balances, or initialization flags, the attacker can drain funds or take ownership.

This directly enables theft of user funds — the same impact class as the external report.

### Likelihood Explanation

- Requires no privileged access; any EOA can call CREATE2 and SELFDESTRUCT.
- Factory patterns (UniswapV2, Abracadabra-style pools, proxy deployers) are the primary target: their CREATE2 salts are often predictable or publicly computable from on-chain parameters.
- ZKsync OS is an L2 rollup; L2s are known to have reorg windows during sequencer transitions, making pre-positioning attacks more feasible than on L1.
- The missing check is a permanent, unconditional omission — not a race condition — so the attacker has unlimited time to pre-populate storage before the victim deploys.

### Recommendation

Implement the EIP-7610 storage-emptiness check before allowing a constructor to execute. Concretely, in `before_executing_frame`, after checking `deployee_code_len` and `deployee_nonce`, also verify that the target address has no non-zero storage slots. If the storage model cannot support a full enumeration check, at minimum track whether any storage was ever written to an address (a single boolean flag per address in the account metadata) and treat a non-zero flag as a collision.

### Proof of Concept

```
// Step 1: Attacker pre-populates storage at the victim's future CREATE2 address
// Attacker deploys AttackerContract to address X = CREATE2(D, S, initcode_victim)
// Inside AttackerContract constructor:
//   sstore(0, attacker_address)   // poison the "owner" slot
//   selfdestruct(attacker)        // clear code+nonce, storage persists

// Step 2: Victim deploys VictimContract to address X via CREATE2(D, S, initcode_victim)
// ZKsync OS: deployee_code_len == 0, deployee_nonce == 0 → no collision detected
// Deployment succeeds; VictimContract.owner() == attacker_address

// Step 3: Attacker calls VictimContract.withdrawAll() as "owner" → drains funds
```

```rust
    pub fn derive_address_for_deployment_create2(
        system: &mut System<S>,
        resources: &mut <S as SystemTypes>::Resources,
        salt: &U256,
        deployer_address: &<S::IOTypes as SystemIOTypesConfig>::Address,
        deployment_code: &[u8],
    ) -> Result<<S::IOTypes as SystemIOTypesConfig>::Address, EvmSubsystemError> {
        use crypto::sha3::{Digest, Keccak256};
        // we need to compute address based on the hash of the code and salt
        let mut initcode_hash = ArrayBuilder::default();
        resources
            .with_infinite_ergs(|inf_resources| {
                S::SystemFunctions::keccak256(
                    deployment_code,
                    &mut initcode_hash,
                    inf_resources,
                    system.get_allocator(),
                )
            })
            .map_err(|e| -> EvmSubsystemError {
                match e.root_cause() {
                    RootCause::Runtime(e @ RuntimeError::FatalRuntimeError(_)) => {
                        e.clone_or_copy().into()
                    }
                    _ => internal_error!("Keccak in create2 cannot fail").into(),
                }
            })?;
        let initcode_hash = Bytes32::from_array(initcode_hash.build());

        let mut create2_buffer = [0xffu8; 1 + 20 + 32 + 32];
        create2_buffer[1..(1 + 20)]
            .copy_from_slice(&deployer_address.to_be_bytes::<{ B160::BYTES }>());
        create2_buffer[(1 + 20)..(1 + 20 + 32)]
            .copy_from_slice(&salt.to_be_bytes::<{ U256::BYTES }>());
        create2_buffer[(1 + 20 + 32)..(1 + 20 + 32 + 32)]
            .copy_from_slice(initcode_hash.as_u8_array_ref());

        let new_address = Keccak256::digest(&create2_buffer);
        #[allow(deprecated)]
        let new_address =
            B160::try_from_be_slice(&new_address.as_slice()[12..]).expect("must create address");

        Ok(new_address)
    }
