Audit Report

## Title
CMC `burn_and_log` Failure Silently Ignored, Enabling Cycles Minting Without ICP Burn — (File: `rs/nns/cmc/src/main.rs`)

## Summary
In `process_mint_cycles`, `process_top_up`, and `process_create_canister`, the CMC dispenses value (cycles, canister, top-up) before burning the backing ICP. The `burn_and_log` function is explicitly typed to return `()` and swallows all errors by design. If the ICP ledger call inside `burn_and_log` fails, cycles are credited (or a canister is created/topped up) while the ICP remains unburned and permanently stranded in the CMC's subaccount. The block index is then sealed as `NotifiedMint(Ok(...))`, preventing any retry or refund, and permanently violating the ICP/cycles conservation invariant.

## Finding Description
`burn_and_log` at line 2017 is declared `async fn burn_and_log(...) -> ()`. The comment at lines 2014–2016 explicitly states the intent: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* This means the function was designed to swallow errors to prevent double-dispensing on retry, but the consequence is that a failed burn is indistinguishable from a successful one at the call site.

In all three process functions the pattern is identical:
- `process_create_canister` (L1943–1946): `do_create_canister` succeeds → `burn_and_log(sub, amount).await;` → `Ok(canister_id)` returned unconditionally.
- `process_mint_cycles` (L1966–1973): `do_mint_cycles` succeeds → `burn_and_log(sub, amount).await;` → `Ok(NotifyMintCyclesSuccess{...})` returned unconditionally.
- `process_top_up` (L1999–2002): `deposit_cycles` succeeds → `burn_and_log(sub, amount).await;` → `Ok(cycles)` returned unconditionally.

The callers (`notify_mint_cycles` L1305–1312, `notify_create_canister` L1411–1418) then call `is_transient_error` on the result. Because the result is always `Ok(...)` regardless of burn outcome, `is_transient_error` never triggers removal, and the block index is permanently sealed as a successful notification. No refund path exists and no retry is possible.

The exploit path:
1. Attacker transfers N ICP to `AccountIdentifier(CMC_ID, Subaccount::from(&caller))` with `memo = MEMO_MINT_CYCLES`.
2. Attacker calls `notify_mint_cycles` during a window when the ICP ledger is temporarily unavailable (e.g., mid-upgrade on the NNS subnet).
3. `do_mint_cycles` succeeds — cycles are credited on the cycles ledger.
4. `burn_and_log` calls `call_protobuf(ledger_canister_id, "send_pb", ...)` (L2040), which fails with a rejection; the error is logged and discarded (L2044–2047).
5. `process_mint_cycles` returns `Ok(NotifyMintCyclesSuccess{...})`.
6. Block index is sealed as `NotifiedMint(Ok(...))`.
7. Result: cycles exist on the cycles ledger; N ICP sits unburned and permanently stranded in the CMC's subaccount; the block index cannot be re-notified or refunded.

## Impact Explanation
The ICP/cycles system relies on the invariant that every unit of cycles minted corresponds to an equal-value ICP burn. A successful `burn_and_log` failure causes the cycles supply to increase without a corresponding decrease in the ICP supply (the ICP is locked in the CMC's subaccount, not burned). Repeated occurrences — each requiring only a narrow ledger-unavailability window — compound the imbalance. The stranded ICP is unrecoverable without a governance-level CMC upgrade. This constitutes illegal minting of cycles and permanent loss of the ICP/cycles backing invariant, matching the High bounty impact tier: significant ledger/financial security impact with concrete protocol harm.

## Likelihood Explanation
The ICP ledger and CMC both reside on the NNS subnet. During a ledger canister upgrade, the ledger is briefly stopped and inter-canister calls to it are rejected with `CanisterStopped` or `CanisterInstallCodeRateLimit`. NNS upgrade proposals are publicly visible on the NNS dashboard, so an unprivileged attacker can observe the upgrade schedule, pre-stage the ICP transfer, and time the `notify_mint_cycles` call to the upgrade window. No special privilege is required. The window is narrow (seconds to low minutes) but recurs with every ledger upgrade cycle.

## Recommendation
1. **Burn before dispensing (preferred):** Restructure all three process functions to call the ICP burn first and only proceed to dispense cycles/canister/top-up if the burn succeeds. This eliminates the window entirely and matches the correct acquire-before-dispense ordering.
2. **Propagate burn errors:** Change `burn_and_log` to return `Result<(), String>` and propagate failures. If the burn fails, return a transient error so `is_transient_error` removes the block index entry and the user can retry after the ledger recovers.
3. **At minimum:** If dispensing-first is kept, record the notification as a transient failure (not `Ok`) when `burn_and_log` fails, so the block index is not permanently sealed and the user retains a retry/refund path.

## Proof of Concept
**Local PocketIC / integration test plan:**
1. Deploy a mock ICP ledger that accepts the initial ICP transfer but rejects all subsequent `send_pb` calls (simulating mid-upgrade unavailability).
2. Deploy the CMC pointing at this mock ledger and a real (or mock) cycles ledger.
3. Call `notify_mint_cycles` with a valid block index.
4. Assert: `do_mint_cycles` succeeds and cycles are credited; `burn_and_log` call to the mock ledger is rejected; `notify_mint_cycles` returns `Ok(NotifyMintCyclesSuccess{...})`.
5. Assert: the block index is now in `blocks_notified` as `NotifiedMint(Ok(...))` — a second call returns `InvalidTransaction` (not `Processing`), confirming the block is permanently sealed.
6. Assert: the CMC's subaccount still holds N ICP (no burn occurred), while the cycles ledger shows the minted cycles balance.
7. Repeat for `notify_create_canister` and `notify_top_up` with analogous mock setups.
