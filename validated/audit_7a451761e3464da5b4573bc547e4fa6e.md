### Title
Unchecked Return Value from `burn_and_log` ICP Burn Call Allows ICP to Remain Unburned After Successful Cycles Minting or Top-Up - (File: rs/nns/cmc/src/main.rs)

---

### Summary

The Cycles Minting Canister (CMC) contains a `burn_and_log` function that is called after a successful cycles minting or canister top-up operation to burn the corresponding ICP from the CMC's subaccount. The function explicitly discards its own error return and only logs failures. If the ICP burn call to the ledger fails, the ICP remains in the CMC's subaccount permanently — it is neither burned nor refunded — while the user has already received their cycles. This is a direct analog to the M-02 "unchecked return value from low-level call" vulnerability class.

---

### Finding Description

In `rs/nns/cmc/src/main.rs`, the function `burn_and_log` is called after a successful `deposit_cycles` (in `process_top_up`) or `do_mint_cycles` (in `process_mint_cycles`). The function's own comment explicitly acknowledges the design intent:

> "Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried." [1](#0-0) 

The function makes a call to the ICP ledger's `send_pb` endpoint and matches on the result, but only logs the error — it does not propagate it, does not retry, and does not trigger any compensating action: [2](#0-1) 

This `burn_and_log` is called in two critical success paths:

1. In `process_top_up`, after `deposit_cycles` succeeds: [3](#0-2) 

2. In `process_mint_cycles`, after `do_mint_cycles` succeeds: [4](#0-3) 

Both `process_top_up` and `process_mint_cycles` are called from the publicly reachable `notify_top_up` and `notify_mint_cycles` update endpoints: [5](#0-4) [6](#0-5) 

The notification status is recorded as `NotifiedTopUp` or `NotifiedMint` (success) before `burn_and_log` is called, so the block index is permanently marked as processed. If the burn fails, the ICP stays in the CMC's subaccount indefinitely with no recovery path.

---

### Impact Explanation

If the ICP ledger's `send_pb` call fails transiently (e.g., ledger temporarily unavailable, queue full, or ledger upgrade in progress) at the moment `burn_and_log` is invoked:

- The user has already received their cycles (deposited to their canister or cycles ledger account).
- The ICP is **not burned** — it remains in the CMC's subaccount.
- The notification block index is permanently marked as `NotifiedTopUp`/`NotifiedMint` (success), so the user cannot retry to trigger the burn.
- The ICP is effectively **locked** in the CMC subaccount with no automated recovery mechanism.

This results in a ledger conservation violation: cycles are minted (or deposited) without the corresponding ICP being burned, inflating the effective cycles supply relative to the ICP backing it. The locked ICP in the CMC subaccount is unrecoverable without manual operator intervention.

---

### Likelihood Explanation

The ICP ledger is a high-availability canister, but transient failures are possible during:
- Ledger canister upgrades (which cause a brief unavailability window).
- Output queue saturation on the CMC's subnet during high load.
- Any inter-canister call rejection due to subnet-level resource pressure.

Any unprivileged user can trigger this path by calling `notify_top_up` or `notify_mint_cycles` with a valid ICP payment block index. The window of failure is narrow but real, and the consequence is permanent once the notification status is committed.

---

### Recommendation

1. **Do not mark the notification as permanently successful before the burn completes.** The burn should be attempted before committing `NotifiedTopUp`/`NotifiedMint` to state, or the state should remain `Processing` until the burn is confirmed.
2. **Alternatively**, if the fire-and-forget design is intentional to avoid retry abuse, introduce a separate persistent queue of "pending burns" that a background timer task retries until confirmed, similar to the reimbursement queue pattern used in the ckETH minter (`process_reimbursement` in `rs/ethereum/cketh/minter/src/withdraw.rs`).
3. At minimum, if `burn_and_log` fails, the CMC should emit a certified event or store the failed burn in recoverable state so operators can detect and remediate the discrepancy. [7](#0-6) 

---

### Proof of Concept

**Attacker-controlled entry path:**

1. An unprivileged user sends ICP to the CMC's subaccount with `MEMO_TOP_UP_CANISTER` or `MEMO_MINT_CYCLES`.
2. The user calls `notify_top_up` (or `notify_mint_cycles`) with the resulting block index.
3. CMC calls `deposit_cycles` (or `do_mint_cycles`) — this succeeds, cycles are delivered.
4. CMC calls `burn_and_log` to burn the ICP. If the ICP ledger call fails at this moment (e.g., ledger is mid-upgrade), the error is only logged.
5. CMC records `NotificationStatus::NotifiedTopUp(Ok(cycles))` — permanently marking the block as processed.
6. The ICP remains in the CMC's subaccount. The user cannot re-notify (duplicate check blocks it). The ICP is locked.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
