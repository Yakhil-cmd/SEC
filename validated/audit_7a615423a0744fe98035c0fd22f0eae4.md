Audit Report

## Title
Unbounded Per-Stream Heap Accumulation via Concurrent QUIC Streams Enables Replica OOM — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
The IC QUIC transport layer accepts up to 1,000 concurrent bidirectional streams per authenticated peer connection and spawns an unbounded `JoinSet` task for each stream. Every task calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)` where `MAX_MESSAGE_SIZE_BYTES = 128 MiB`, accumulating received bytes into a heap `Vec<u8>`. QUIC flow-control windows throttle in-flight data but do not cap total heap accumulation across concurrent `read_to_end` calls, allowing a malicious authenticated subnet peer to drive the receiving replica into OOM and crash it out of consensus.

## Finding Description

**`MAX_MESSAGE_SIZE_BYTES = 128 MiB`** is the single global ceiling applied to every stream: [1](#0-0) 

**`MAX_CONCURRENT_BIDI_STREAMS = 1,000`** is the QUIC-enforced per-connection stream limit: [2](#0-1) 

**The `JoinSet` is unbounded** — every accepted stream immediately spawns a task with no capacity check: [3](#0-2) 

The code even contains a comment acknowledging the unbounded buffering risk but treats it as a performance concern rather than a security one: [4](#0-3) 

**Each spawned task calls `read_to_end(MAX_MESSAGE_SIZE_BYTES)`**, which dynamically grows a `Vec<u8>` on the heap as data arrives: [5](#0-4) 

The same pattern exists on the response path: [6](#0-5) 

**Why flow-control windows are insufficient**: `STREAM_RECEIVE_WINDOW = 4 MiB` and `RECEIVE_WINDOW = 200 MiB` limit how much data can reside in QUIC receive buffers at any moment. However, `read_to_end` continuously drains those buffers into the `Vec<u8>`, causing the QUIC stack to emit window-update frames that permit the sender to transmit more data. The windows govern in-flight bytes, not total heap bytes already moved into the Vecs. Over time, each of the 1,000 concurrent Vecs can grow to 128 MiB independently, for a theoretical total of 128 GiB from a single peer connection.

## Impact Explanation
A Byzantine subnet peer can crash a replica via OOM, removing it from consensus participation. This is a concrete subnet availability impact — a node outage caused by a protocol-level DoS that is not raw volumetric DDoS. This matches the allowed High impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation
The attacker must hold a valid TLS certificate as an authenticated subnet peer — a Byzantine node below the consensus fault threshold, which is explicitly within the IC bounty attacker model. No governance majority, leaked key, or external oracle is required. The attack uses only standard QUIC stream primitives: open 1,000 streams and pace data delivery to stay within flow-control windows. The constants `MAX_CONCURRENT_BIDI_STREAMS = 1,000` and `MAX_MESSAGE_SIZE_BYTES = 128 MiB` are both reachable without any special privilege beyond subnet membership. The attack is repeatable and sustained.

## Recommendation
1. **Cap the `JoinSet`**: Enforce a maximum number of concurrently inflight request tasks per connection (e.g., 50–100), back-pressuring `accept_bi` when the cap is reached. The existing comment at lines 52–56 of `request_handler.rs` already identifies this as the correct architectural fix.
2. **Per-peer memory budget**: Track total bytes currently buffered across all in-flight `read_to_end` calls for a given connection; reject or reset new streams when the budget is exceeded.
3. **Per-handler message size limits**: Replace the single global `MAX_MESSAGE_SIZE_BYTES` with per-URI/handler limits derived from the actual maximum artifact size for each protocol handler, rather than the worst-case 128 MiB ceiling applied uniformly.
4. **Reduce `MAX_MESSAGE_SIZE_BYTES`**: The comment cites 40-node subnet summary blocks as justification; a tighter per-handler limit should be enforced at the router layer instead.

## Proof of Concept
```
1. Obtain a valid IC TLS certificate as an authenticated subnet peer.
2. Establish a QUIC connection to the victim replica.
3. Open 1,000 bidirectional streams (the MAX_CONCURRENT_BIDI_STREAMS limit).
4. On each stream, send a protobuf-encoded HttpRequest whose body is
   127 MiB of arbitrary bytes, paced to stay within QUIC flow-control
   windows (STREAM_RECEIVE_WINDOW=4 MiB, RECEIVE_WINDOW=200 MiB).
5. The victim spawns 1,000 tokio tasks via the unbounded JoinSet, each
   executing: recv_stream.read_to_end(134_217_728).await
   Each task's Vec<u8> grows as data arrives and window updates are sent.
6. As each Vec drains the QUIC receive buffer, window-update frames
   replenish the connection window, permitting continued data delivery.
7. Total heap allocation grows toward 1,000 × 128 MiB = 128 GiB.
8. The OS OOM-killer terminates the replica; the node drops out of consensus.

Reproducible as a local integration test: spin up two replica nodes with
PocketIC or a local testnet, configure one as a Byzantine peer, and use
a modified quinn client to open MAX_CONCURRENT_BIDI_STREAMS streams each
sending MAX_MESSAGE_SIZE_BYTES of data. Observe heap growth via /proc/PID/status
VmRSS until OOM or replica crash.
```
