### Title
Owner Can Redirect ETH Connector to Arbitrary Account Without Migration, Permanently Freezing User Funds - (File: `engine/src/contract_methods/connector.rs`)

### Summary
The `set_eth_connector_contract_account` function in the Aurora Engine allows the `owner` to atomically replace the stored ETH connector `AccountId` with any arbitrary account at any time, with no migration guard, no balance check, and no interface verification. Every user-facing token operation (`withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw`) resolves the connector account at call time via `get_connector_account_id`. Replacing the connector while user balances exist severs the accounting link between the engine and the connector, permanently freezing all bridged ETH and NEP-141 token balances held on behalf of users.

### Finding Description
`set_eth_connector_contract_account` is callable by the owner with no preconditions beyond the engine being running: [1](#0-0) 

The function unconditionally overwrites the stored connector `AccountId` and serialization type: [2](#0-1) 

Every user-facing token operation then resolves the connector at call time through `return_promise → get_connector_account_id`: [3](#0-2) 

The affected operations are `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, and `storage_withdraw` — all of which call `return_promise` with the live connector account: [4](#0-3) [5](#0-4) 

There is no check that the new account has existing user balances, implements the expected NEP-141 interface, or that a migration has been completed. The `EngineState` stores only `owner_id` and no connector-related invariants: [6](#0-5) 

The owner is a single `AccountId` with no time-lock or multi-sig enforcement: [7](#0-6) 

This is the direct analog of the FujiVault finding: the owner can "switch out the debt-management token while one is already configured without any migration," creating an inconsistent state where "no one will be able to withdraw their now non-existent token."

### Impact Explanation
**High — Permanent freezing of funds.**

After `set_eth_connector_contract_account` is called with a non-functional or non-existent account:
- All `withdraw` calls route to the new account and fail (or are silently consumed by a malicious account).
- All `ft_transfer` / `ft_transfer_call` calls fail identically.
- All bridged ETH and NEP-141 balances recorded inside the engine become inaccessible.
- There is no on-chain recovery path: the engine holds no ETH itself; the connector holds the NEP-141 supply. Changing the pointer without migrating the supply orphans all user balances permanently.

### Likelihood Explanation
The owner is a single NEAR account with no time-lock, no multi-sig requirement, and no governance delay enforced at the contract level. The function is exposed as a standard NEAR contract method (`set_eth_connector_contract_account`) callable in a single transaction. A legitimate upgrade attempt (e.g., deploying a new connector version) without a coordinated migration window is sufficient to trigger the freeze — no malicious intent is required. The design explicitly acknowledges the need for a special role via a TODO comment in the adjacent `set_erc20_metadata` function: [8](#0-7) 

### Recommendation
1. **Add a migration guard**: Before accepting a new connector account, verify that the current connector's total supply is zero, or require an explicit migration transaction that atomically transfers all balances.
2. **Separate the role**: Introduce a dedicated `connector_admin` role distinct from `owner_id`, so that connector replacement requires a separate, independently controlled key — mirroring the segregation-of-duties principle the external report recommends.
3. **Enforce a time-lock**: Stage the connector account change (analogous to `stage_upgrade` / `upgrade_delay_blocks`) so users have time to exit before the switch takes effect.
4. **Ensure `owner_id` is a time-locked multi-sig or governance contract**, consistent with the external report's recommendation.

### Proof of Concept
1. Users bridge ETH to Aurora; the engine records their balances and the connector holds the NEP-141 supply.
2. The owner calls `set_eth_connector_contract_account` with a new account `new-connector.near` that does not exist or does not implement `engine_withdraw` / `ft_transfer`.
3. Any user calls `withdraw(recipient_eth_address, amount)` → `return_promise` resolves `get_connector_account_id` → returns `new-connector.near` → the NEAR promise targets a non-existent account and fails.
4. The same failure occurs for `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, and `storage_withdraw`.
5. All user ETH balances inside the engine are permanently inaccessible; the old connector still holds the NEP-141 supply but the engine no longer points to it.

#[named]
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

    env: &E,

    deposit: Yocto,
) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
        target_account_id: get_connector_account_id(&io)?,
        method: method.to_string(),
        args,
        attached_balance: deposit,
        attached_gas: calculate_attached_gas(env),
    };
    let promise_id = io.promise_create_call(&promise_args);

    io.promise_return(promise_id);

    Ok(())
