### Title
Silent Precompile Failure + No Refund Path Burns ERC-20 Tokens With No Exit — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` burns the caller's tokens **before** calling the exit precompile, and never checks the return value of that call. When the precompile call fails (invalid recipient bytes) or the downstream NEAR-side `ft_transfer` fails, the `error_refund` feature — the only recovery mechanism — is **not enabled in the production build**. The result is permanently burned ERC-20 tokens with no corresponding NEP-141 transfer and no user-accessible recovery path.

---

### Finding Description

**Step 1 — Burn-before-call with unchecked return value**

`EvmErc20.withdrawToNear` executes `_burn` first, then calls the precompile via inline assembly. The return value `res` is stored but never inspected; no `require(res != 0)` or revert follows. [1](#0-0) 

If the `call` opcode returns `0` (precompile error), the EVM does **not** automatically revert the already-committed `_burn`. The tokens are gone.

**Step 2 — When does the precompile call fail?**

`parse_recipient` in the precompile validates the recipient bytes as UTF-8 and as a valid NEAR account ID. Any non-UTF-8 byte sequence or syntactically invalid account ID causes an `ExitError`, which makes the sub-call return `0`. [2](#0-1) 

A user who passes malformed bytes (e.g., `\xc2` alone, or `test@.near`) triggers this path. The burn is committed; no promise is ever created.

**Step 3 — Even with a valid recipient, NEAR-side failure has no refund**

When the recipient is valid, the precompile creates a `ft_transfer` promise. If that promise fails on the NEAR side (e.g., unregistered account), the refund path is gated behind the `error_refund` compile-time feature: [3](#0-2) 

Without `error_refund`, `refund` is hardcoded to `None`. When `callback_args` equals the default (both fields `None`), no callback is attached at all — just a bare `PromiseArgs::Create`: [4](#0-3) 

The `exit_to_near_precompile_callback` refund branch is therefore unreachable: [5](#0-4) 

**Step 4 — `error_refund` is NOT enabled in the production build**

`engine/Cargo.toml` shows `default = ["std"]` and `contract = ["log", ...]`. Neither includes `error_refund`. [6](#0-5) 

The production WASM is compiled with the `contract` feature. `error_refund` is absent.

**Step 5 — The test suite explicitly confirms the loss**

The existing test `test_exit_to_near_refund` documents the exact behavior: without `error_refund`, the ERC-20 balance is reduced by the full exit amount with no refund. [7](#0-6) 

---

### Impact Explanation

Two concrete loss paths exist:

| Path | Trigger | Outcome |
|---|---|---|
| A | `withdrawToNear` called with non-UTF-8 or invalid NEAR account bytes | Precompile sub-call returns 0; burn committed; no promise created; NEP-141 tokens remain locked in Aurora's NEAR account forever |
| B | Valid recipient, but NEAR-side `ft_transfer` fails (unregistered account) | Promise fires and fails; no callback; no refund; same permanent loss |

In both cases the ERC-20 tokens are burned and the corresponding NEP-141 tokens are permanently stranded in Aurora's NEAR account with no user-accessible recovery. This meets **Permanent freezing of funds** (and arguably direct loss of user funds).

---

### Likelihood Explanation

- Path A is reachable by any user who passes arbitrary bytes as `recipient` — the parameter type is `bytes memory` with no Solidity-level validation.
- Path B is reachable whenever the NEAR-side recipient account is not registered with the NEP-141 token contract, a common operational condition.
- No admin compromise, key leak, or external oracle is required.
- The production binary provably lacks `error_refund`.

---

### Recommendation

1. **Check the precompile return value** in `EvmErc20.withdrawToNear` and revert if it is zero, so the burn is rolled back atomically:
   ```solidity
   assembly {
       let res := call(...)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Enable `error_refund`** in the production build feature set so that NEAR-side failures trigger the existing callback refund path.
3. Apply the same fix to `withdrawToEthereum` and `EvmErc20V2`, which have the identical unchecked-return pattern. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IEvmErc20 {
    function withdrawToNear(bytes memory recipient, uint256 amount) external;
    function balanceOf(address) external view returns (uint256);
}

contract PoC {
    function exploit(IEvmErc20 token, uint256 amount) external {
        // Pass a non-UTF-8 byte sequence — precompile will return ExitError,
        // sub-call returns 0, but EvmErc20 never checks res.
        // _burn already executed; tokens are gone.
        bytes memory invalidRecipient = hex"c2"; // lone continuation byte, invalid UTF-8
        token.withdrawToNear(invalidRecipient, amount);

        // Caller's ERC-20 balance is now reduced by `amount`.
        // No NEP-141 transfer was ever scheduled.
        // No refund callback exists (error_refund not compiled in).
        // Funds are permanently frozen in Aurora's NEAR account.
        assert(token.balanceOf(address(this)) == 0); // tokens burned
    }
}
```

The same outcome is achieved with a syntactically valid UTF-8 string that is not a valid NEAR account ID (e.g., `"INVALID ACCOUNT!"`), or with a valid account ID that is not registered with the NEP-141 contract.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
