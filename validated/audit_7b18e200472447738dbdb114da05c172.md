### Title
Unbounded `ingress_expiries` Loop via Client-Controlled `ingress_start`/`ingress_end` Causes OOM Crash of ICP Rosetta Server — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

---

### Summary

The ICP Rosetta server's `RosettaRequestHandler::construction_payloads` function accepts client-supplied `ingress_start` and `ingress_end` values from the request metadata without any range validation. These values directly control a `while` loop that pushes entries into a `Vec<u64>`. Setting `ingress_start = 0` and `ingress_end = u64::MAX` causes the loop to execute ~153 million iterations before `Time` wraps around (due to truncating `as u64` cast), after which the loop becomes infinite. The result is unbounded heap allocation and an OOM crash of the Rosetta server process, blocking all ICP exchange operations.

---

### Finding Description

**Vulnerable loop** — `construction_payloads.rs` lines 59–107:

```rust
let interval =
    ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT - Duration::from_secs(120);
// interval = 300s - 60s - 120s = 120 seconds = 120_000_000_000 ns

let ingress_start = meta
    .and_then(|m| m.ingress_start)
    .map(ic_types::time::Time::from_nanos_since_unix_epoch)
    .unwrap_or_else(ic_types::time::current_time);   // no validation

let ingress_end = meta
    .and_then(|m| m.ingress_end)
    .map(ic_types::time::Time::from_nanos_since_unix_epoch)
    .unwrap_or_else(|| ingress_start + interval);    // no validation

let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {                            // ← no bound check
    ingress_expiries.push(...);
    now += interval;                                 // ← wraps on overflow
}
``` [1](#0-0) 

**`MAX_INGRESS_TTL` = 5 min, `PERMITTED_DRIFT` = 60 s**, making `interval` = 120 seconds = 120,000,000,000 ns. [2](#0-1) 

**`Time::add_assign` uses a truncating `as u64` cast:**

```rust
impl std::ops::AddAssign<Duration> for Time {
    fn add_assign(&mut self, other: Duration) {
        *self = Time::from_duration(Duration::from_nanos(self.0) + other)
    }
}
fn from_duration(t: Duration) -> Self {
    Time(t.as_nanos() as u64)   // ← truncates, does NOT panic or saturate
}
``` [3](#0-2) [4](#0-3) 

When `now` reaches near `u64::MAX` and `interval` is added, `t.as_nanos()` (a `u128`) exceeds `u64::MAX`, and the `as u64` truncation wraps `now` back to a small value (~120 seconds). Since that small value is still `< ingress_end` (`u64::MAX`), the loop continues indefinitely.

**The endpoint is unauthenticated and publicly registered:**

```rust

The JSON body size limit is 4 MB, but the malicious payload is tiny — just two u64 values in the metadata JSON. [6](#0-5) 

**The ICRC1 Rosetta counterpart has the fix; the ICP version does not.** The ICRC1 `construction_payloads` explicitly validates the range before the loop:

```rust
if ingress_start >= ingress_end {
    return Err(Error::processing_construction_failed(...));
}
if ingress_end < now + ingress_interval {
    return Err(Error::processing_construction_failed(...));
}
``` [7](#0-6) 

The ICP Rosetta handler has no equivalent guards.

---

### Impact Explanation

With `ingress_start = 0` and `ingress_end = u64::MAX` (18,446,744,073,709,551,615):

- **Before overflow:** `u64::MAX / 120_000_000_000 ≈ 153,722,867` iterations. Each pushes 8 bytes → **~1.2 GB** allocated in `ingress_expiries` alone.
- **After overflow:** `now` wraps to ~120 seconds, still `< u64::MAX`, loop continues → **infinite loop / OOM**.
- The Rosetta server process is killed by the OS OOM killer.
- All ICP exchange operations (transfers, neuron management, staking) routed through this Rosetta instance are blocked until the process is restarted.
- A single unauthenticated HTTP POST is sufficient; the attack is repeatable.

---

### Likelihood Explanation

- The `/construction/payloads` endpoint requires no authentication or privileged role.
- The malicious request is a standard JSON POST with two integer fields; no special tooling is needed.
- The attack is deterministic and reproducible locally.
- The ICRC1 Rosetta already has the fix, confirming the developers are aware of the pattern — the ICP version was simply not updated.

---

### Recommendation

Add range validation before the loop in `RosettaRequestHandler::construction_payloads`, mirroring the ICRC1 implementation:

1. Reject if `ingress_start >= ingress_end`.
2. Reject if `ingress_end` exceeds `ingress_start` by more than a reasonable maximum (e.g., 24 hours, matching the ledger's `created_at_time` window).
3. Cap `ingress_expiries` length with an explicit `MAX_INGRESS_EXPIRIES` constant and return an error if the computed count would exceed it. [8](#0-7)
