### Title
Unchecked Exit-Precompile Return Value in `EvmErc20` Burns User Tokens Without Completing Cross-Chain Transfer — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` and `EvmErc20.withdrawToEthereum` irreversibly burn the caller's ERC-20 tokens **before** invoking the exit precompile via a low-level assembly `call`, but **never check the return value**. If the precompile call fails, the burn is not reverted and no cross-chain transfer is initiated, permanently destroying the user's funds.

---

### Finding Description

Both exit functions share the same flawed pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← irreversible state change

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← irreversible state change
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [2](#0-1) 

The low-level EVM `call` opcode returns `0` on failure **without reverting the caller's context**. Because `res` is declared but immediately discarded, a precompile failure is completely invisible to the contract. The `_burn` that preceded it is not rolled back.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) explicitly returns errors for a range of conditions — invalid recipient account ID format, malformed input, or internal state errors — any of which will cause the `call` to return `0`. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of funds.**

When the precompile call fails:
- The user's ERC-20 mirror tokens are burned (supply reduced, balance zeroed).
- No NEP-141 or Ethereum tokens are transferred to the recipient.
- There is no refund mechanism; the tokens are unrecoverable.

This is a direct analog to the reported vulnerability class: a fund-transfer step that silently fails after an irreversible accounting change, leaving the user with neither the original tokens nor the destination tokens.

---

### Likelihood Explanation

**Medium.** The `ExitToNear` precompile rejects inputs with invalid NEAR account IDs (e.g., too long, illegal characters) and can fail on insufficient forwarded gas. A user supplying a malformed `recipient` bytes argument — or a scenario where the precompile encounters an internal error — triggers the silent failure path. The `recipient` parameter is an unvalidated `bytes memory` value accepted directly from the caller. [4](#0-3) 

---

### Recommendation

Check the return value of every low-level precompile `call` and revert on failure, so the `_burn` is rolled back atomically:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum`. This ensures the burn and the cross-chain transfer are effectively atomic: either both succeed or neither takes effect.

---

### Proof of Concept

1. User calls `withdrawToNear(recipient, amount)` where `recipient` is a byte string that the `ExitToNear` precompile rejects (e.g., an oversized or malformed NEAR account ID).
2. `_burn(_msgSender(), amount)` executes and permanently reduces the user's ERC-20 balance.
3. The assembly `call` to the `ExitToNear` precompile returns `0` (failure).
4. `res` is never read; the function returns normally with no revert.
5. The user's ERC-20 tokens are gone; no NEP-141 tokens arrive on NEAR. Funds are permanently lost. [1](#0-0) [3](#0-2)
