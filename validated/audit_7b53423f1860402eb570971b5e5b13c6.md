Audit Report

## Title
ckBTC Minter Address Self-Send Guard Bypassed via Uppercase Bech32 Address - (File: `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary
Both `retrieve_btc` and `retrieve_btc_with_approval` guard against withdrawals to the minter's own Bitcoin address using a case-sensitive string equality check. Because bech32 addresses are case-insensitive by specification, supplying the minter's address in uppercase bypasses the guard entirely. The address parses successfully to the minter's own `BitcoinAddress`, ckBTC is burned on the ledger, and BTC is sent back to the minter — the caller permanently loses their ckBTC with no recourse.

## Finding Description
In `retrieve_btc` (lines 156–160) and identically in `retrieve_btc_with_approval` (lines 254–258) of `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`:

```rust
let main_address_str = state::read_state(|s| runtime.derive_minter_address_str(s));
if args.address == main_address_str {          // case-sensitive ==
    ic_cdk::trap("illegal retrieve_btc target");
}
// ...
let parsed_address = BitcoinAddress::parse(&args.address, btc_network)?;
```

`derive_minter_address_str` calls `display()` → `encode_bech32()` → `bech32::encode`, which always emits a **lowercase** string. The guard therefore only blocks the exact lowercase representation.

`BitcoinAddress::parse` explicitly routes uppercase-prefixed addresses (`'B'`, `'T'`) into `parse_bip173_address` (address.rs lines 81–83). Inside `parse_bip173_address`, the HRP is normalised with `.to_lowercase()` (line 344), and the data portion is decoded through the case-insensitive bech32 bit-conversion. The in-tree test at address.rs lines 471–480 confirms that `"BC1QW508D6QEJXTDG4Y5R3ZARVARY0C5XW7KV8F3T4"` and its lowercase equivalent decode to the **identical** `BitcoinAddress::P2wpkhV0([117, 30, ...])`.

Exploit flow:
1. Attacker queries the minter's address (lowercase bech32, e.g. `bc1q...xyz`).
2. Attacker converts it to uppercase: `BC1Q...XYZ`.
3. Attacker calls `retrieve_btc` with `address = "BC1Q...XYZ"`.
4. String guard at line 158 is **not triggered** (`"BC1Q...XYZ" != "bc1q...xyz"`).
5. `BitcoinAddress::parse` at line 173 succeeds, returning `P2wpkhV0(minter_pkhash)`.
6. `burn_ckbtcs` burns the caller's ckBTC on the ledger (line 210).
7. A Bitcoin transaction is constructed and submitted with the minter's own address as the output.
8. The minter later sweeps that UTXO as change via `fetch_main_utxos`.
9. Caller receives **no BTC** and has permanently lost their ckBTC.

The TLA+ spec at `rs/bitcoin/ckbtc/spec/Ck_BTC.tla` lines 1415–1428 (`Prevent_Retrievals_To_Change_Address`) explicitly models this as a forbidden state, confirming the design intent that no withdrawal should target the minter's address.

## Impact Explanation
This is a permanent, irreversible loss of user ckBTC funds with no reimbursement path: ckBTC supply decreases without a corresponding reduction in the minter's BTC holdings, breaking the 1:1 peg invariant for the affected user. The attack is trivially repeatable against any user who can be induced to call `retrieve_btc` with the uppercase form of the minter's address. This matches the **High** impact class: "Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm." At scale (many users or large amounts), it approaches the **Critical** threshold of "Theft, permanent loss… involving in-scope chain-key/ledger assets."

## Likelihood Explanation
The minter's Bitcoin address is publicly queryable via `get_btc_address`. Bech32 addresses are routinely displayed in uppercase by QR-code generators, hardware wallets, and block explorers. A user who copies the minter's address from such a source and calls `retrieve_btc` would silently lose funds. An adversarial front-end could also deliberately inject the uppercase form. The entry path requires only a standard unprivileged ingress call to `retrieve_btc` or `retrieve_btc_with_approval` — no special privileges, no governance majority, no key compromise.

## Recommendation
Replace the string comparison with a comparison of the **parsed** `BitcoinAddress` value, which is representation-independent:

```rust
let main_address = state::read_state(|s| runtime.derive_minter_address(s));
let parsed_address = BitcoinAddress::parse(&args.address, btc_network)?;
if parsed_address == main_address {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

Apply the same fix in both `retrieve_btc` and `retrieve_btc_with_approval`. This eliminates the case-sensitivity gap and is consistent with how the rest of the minter compares addresses structurally.

## Proof of Concept
1. Query the minter's address: `dfx canister call ckbtc_minter get_btc_address '(record { owner = opt principal "minter-id"; subaccount = null })'` → returns e.g. `bc1q...xyz`.
2. Convert to uppercase: `BC1Q...XYZ`.
3. Transfer ckBTC to the minter's withdrawal subaccount for your principal.
4. Call `retrieve_btc '(record { address = "BC1Q...XYZ"; amount = <amount> })'`.
5. Observe: the call succeeds (no trap), ckBTC is burned on the ledger (confirmed via ledger `query_blocks`), and a Bitcoin transaction is submitted sending BTC to the minter's own address. The caller receives no BTC.

A deterministic unit test can be written by mocking `derive_minter_address_str` to return a known lowercase bech32 address, then calling `retrieve_btc` with the uppercase equivalent and asserting that the call traps (currently it does not).
