Audit Report

## Title
Unbounded Per-Connection Aggregate Memory via Concurrent QUIC Streams — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
`start_stream_acceptor` spawns an unbounded number of Tokio tasks into a `JoinSet` — one per accepted bidirectional QUIC stream — with no aggregate in-flight memory cap. Each task calls `read_to_end(MAX_MESSAGE_SIZE_BYTES)` (128 MiB), and because Quinn's flow-control windows are sliding windows that advance as data is drained into the application `Vec`, a Byzantine peer holding a valid subnet TLS certificate can open up to `MAX_CONCURRENT_BIDI_STREAMS = 1,000` streams and drive the replica's heap toward ~128 GiB, triggering an OOM kill.

## Finding Description

**Confirmed constants** (`rs/p2p/quic_transport/src/lib.rs` L74, `rs/p2p/quic_transport/src/connection_manager.rs` L71–75):
- `MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024`
- `MAX_CONCURRENT_BIDI_STREAMS = VarInt::from_u32(1_000)`
- `STREAM_RECEIVE_WINDOW = VarInt::from_u32(4_000_000)` (4 MiB per stream)
- `RECEIVE_WINDOW = VarInt::from_u32(200_000_000)` (200 MiB connection-level)

**Vulnerable acceptor loop** (`rs/p2p/quic_transport/src/request_handler.rs` L50, L79–90): Every accepted `accept_bi()` stream unconditionally spawns a new task into `inflight_requests: JoinSet` with no bound on concurrent tasks or their combined memory footprint. The code comment at L52–56 explicitly acknowledges this: *"The extreme result of a slow handler is that the stream limit will be reached, hence having buffered up to the stream limit number of messages/requests."*

**Vulnerable read site** (`rs/p2p/quic_transport/src/request_handler.rs` L173–178):
```rust
let request_bytes = recv_stream
    .read_to_end(MAX_MESSAGE_SIZE_BYTES)
    .await
    ...?;
```
`read_to_end` repeatedly calls `read_chunk`, moving data from Quinn's internal receive buffer into an application-owned `Vec<u8>`. Each drain frees Quinn's internal buffer and causes Quinn to emit `MAX_STREAM_DATA`/`MAX_DATA` window-extension frames back to the sender. The QUIC flow-control windows (`STREAM_RECEIVE_WINDOW = 4 MiB`, `RECEIVE_WINDOW = 200 MiB`) are therefore **sliding windows that throttle rate, not total heap**. As each stream's `Vec` grows, the sender is continuously granted new send credits. With 1,000 concurrent streams each allowed to grow to 128 MiB, the aggregate heap can reach ~128 GiB before any stream completes.

**Why existing guards are insufficient:**
- The QUIC `RECEIVE_WINDOW` (200 MiB) only limits how much data is simultaneously in Quinn's internal buffers; it does not cap the sum of all application-side `Vec` allocations.
- There is no semaphore, permit, or byte-counter gating stream acceptance or task spawning.
- `MAX_CONCURRENT_BIDI_STREAMS = 1,000` is the only limit, and it is the multiplier in the worst-case calculation.

## Impact Explanation
A Byzantine subnet member (single node below the consensus fault threshold) can crash any peer replica it is connected to via OOM. The Linux OOM killer terminates the replica process, removing it from consensus participation. The attack is non-volumetric (it exploits a protocol-level memory accounting gap, not raw bandwidth) and repeatable — the attacker can re-establish the connection and repeat after each crash. This matches the High bounty impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation
- **Precondition:** Valid subnet node TLS certificate — i.e., the attacker is a Byzantine subnet member. This is within the single-node fault model and below the consensus fault threshold.
- **No authentication bypass, rate-limiting bypass, or victim mistake required.** The connection is already established by normal protocol operation.
- **Repeatable:** After an OOM kill the attacker reconnects and repeats.
- **No special hardware needed:** A single malicious node with a normal network connection suffices.

## Recommendation
1. **Add a per-connection in-flight byte semaphore.** Before spawning a task for a new stream, acquire `MAX_MESSAGE_SIZE_BYTES` permits from a `tokio::sync::Semaphore` initialized to a safe per-connection budget (e.g., 256–512 MiB). Release permits when the task completes. Reject or back-pressure streams that would exceed the budget.
2. **Reduce `MAX_CONCURRENT_BIDI_STREAMS`** to a value consistent with actual application concurrency (e.g., 10–50), reducing the worst-case multiplier.
3. **Use a bounded task pool** (e.g., `tokio::sync::Semaphore`-guarded spawn) in `start_stream_acceptor` so that at most `N` streams read concurrently, where `N × MAX_MESSAGE_SIZE_BYTES` is within a safe memory envelope.
4. **Consider streaming deserialization** (e.g., `prost::decode` from a length-prefixed chunked reader) to avoid materializing the full 128 MiB body in one allocation.

## Proof of Concept
```
1. Obtain a valid subnet node TLS certificate (Byzantine node precondition).
2. Establish a QUIC connection to the victim replica using the certificate.
3. Open 1,000 bidirectional streams simultaneously (within MAX_CONCURRENT_BIDI_STREAMS).
4. On each stream, send data at the STREAM_RECEIVE_WINDOW rate (4 MiB chunks),
   keeping each stream open (no FIN). As the victim's read_to_end drains Quinn's
   buffer, the flow-control window advances; continue sending up to 128 MiB per stream.
5. Monitor victim RSS via /proc/<pid>/status or Prometheus metrics.
6. Expected: victim heap exceeds available RAM; OOM killer terminates the replica process.
```
A deterministic integration test can simulate this with a local Quinn endpoint acting as the Byzantine peer, asserting that heap usage grows proportionally to the number of open streams without any server-side rejection.
