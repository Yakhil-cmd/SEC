Audit Report

## Title
Unbounded Synchronous BFS Traversal in SNS Governance Vote Cascade Enables Permissionless DoS - (File: rs/sns/governance/src/governance.rs)

## Summary
SNS governance's `cast_vote_and_cascade_follow` performs a fully synchronous BFS over the neuron follower graph with no instruction-limit guard. An unprivileged attacker can create up to `MAX_NUMBER_OF_NEURONS_CEILING` (200,000) neurons each following a target neuron, then trigger a vote cascade that exhausts the per-message instruction budget, causing the update call to trap and permanently preventing the target neuron from voting or submitting proposals.

## Finding Description
`cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs` (L3687–3836) is a plain synchronous `fn` with a `while !induction_votes.is_empty()` BFS loop. A grep for any instruction-limit keyword (`instruction`, `over_soft`, `is_message_over`, `HARD_VOTING`, `SOFT_VOTING`) across the entire file returns exactly one unrelated match — confirming there is no guard of any kind. [1](#0-0) 

By contrast, the NNS governance version (`rs/nns/governance/src/voting.rs`, L122–180) is `async`, checks `over_soft_message_limit()` after each processed neuron, and calls `noop_self_call_if_over_instructions` with a soft limit of 1 billion and a hard limit of 750 billion instructions to break work across messages. [2](#0-1) 

The SNS function is called synchronously from two public update-call paths:

1. `register_vote` → `cast_vote_and_cascade_follow` at L3931 [3](#0-2) 

2. `make_proposal` → `cast_vote_and_cascade_follow` at L3658 [4](#0-3) 

Both are reachable from the public `manage_neuron` update endpoint. [5](#0-4) 

The `max_followees_per_function` cap (ceiling 15) limits how many neurons a single neuron can *follow*, but the **follower** side is entirely uncapped — any number of neurons can follow a single target neuron. With `MAX_NUMBER_OF_NEURONS_CEILING = 200_000`, an attacker can create up to 200,000 neurons each following the target. [6](#0-5) [7](#0-6) 

## Impact Explanation
When the target neuron votes or submits a proposal, the synchronous BFS must complete within a single message execution. With 200,000 follower neurons, each requiring `BTreeMap` lookups and `vote_from_ballots_following` calls, the traversal exhausts the IC per-message instruction limit. The entire update call traps and is rolled back: the vote is never recorded, and repeated calls always trap. The target neuron is permanently unable to vote or propose on any open proposal. This constitutes a **High** severity application/platform-level DoS on SNS governance with concrete protocol harm, matching the allowed impact: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation
**Medium.** The attacker must acquire N × `neuron_minimum_stake_e8s` governance tokens and create N neurons each following the target. For SNS tokens with low market value this is economically feasible. The attack is also self-reinforcing: a popular neuron (e.g., a well-known validator) naturally accumulates followers over time without attacker involvement, eventually triggering the same failure organically. The attack is permissionless, requires no special privileges, and is repeatable.

## Recommendation
Apply the same fix used in NNS governance:

1. Introduce a `ProposalVotingStateMachine` equivalent for SNS governance, analogous to `rs/nns/governance/src/voting.rs`.
2. Make `cast_vote_and_cascade_follow` `async` and check `is_message_over_threshold(SOFT_VOTING_INSTRUCTIONS_LIMIT)` after each neuron processed in the BFS loop.
3. Use `noop_self_call_if_over_instructions` with a hard limit to break work across messages, and drain remaining cascade work via a timer task.

As a simpler interim mitigation: add a hard cap on the total number of neurons processed per BFS traversal, returning early and deferring remaining work to a timer.

## Proof of Concept
1. Deploy an SNS with default parameters (`max_number_of_neurons = 200_000`, `neuron_minimum_stake_e8s = 1 token`).
2. Create target neuron T with a valid stake.
3. Create N attacker neurons A₁…Aₙ, each calling `manage_neuron::Follow` to follow T on the relevant `function_id`.
4. Have T call `RegisterVote` on any open proposal:
```
manage_neuron({
  subaccount: T_subaccount,
  command: RegisterVote({ proposal: Some(proposal_id), vote: Yes })
})
```
5. Observe: the update call traps due to instruction exhaustion. The vote is never recorded. All subsequent `RegisterVote` and `MakeProposal` calls from T also trap. A PocketIC integration test can verify this by setting N to a value that exceeds the instruction budget and asserting the call returns a trap/reject rather than `Ok`.
