### Title
`Operation::Approve` Fee Never Routed to `fee_collector` in ICRC-1 Ledger — (`File: rs/ledger_suite/icrc1/src/lib.rs`)

---

### Summary

The ICRC-1 ledger supports a configurable `fee_collector` account that is supposed to receive transaction fees instead of burning them. However, the `Approve` operation in the ledger's `apply` function always **burns** the fee via `balances_mut().burn()`, completely ignoring the configured `fee_collector`. This is directly analogous to M-01: a fee-collection mechanism exists and is configurable, but a specific operation type silently bypasses it.

---

### Finding Description

In `rs/ledger_suite/icrc1/src/lib.rs`, the `Transaction::apply` function retrieves the `fee_collector` at the top of the function: [1](#0-0) 

For `Operation::Transfer`, the `fee_collector` is correctly passed to `balances_mut().transfer()`: [2](#0-1) 

However, for `Operation::Approve`, the fee is unconditionally burned via `balances_mut().burn()`, and `fee_collector` is never consulted: [3](#0-2) 

The `burn()` function always routes tokens to the `token_pool` (i.e., destroys them), with no path to the `fee_collector`: [4](#0-3) 

Compare with `transfer()`, which explicitly branches on `fee_collector`: [5](#0-4) 

The `fee_collector` is set via the ledger's upgrade mechanism: [6](#0-5) 

The `icrc2_approve` endpoint calls `apply_transaction` which calls `apply`, confirming the reachable path: [7](#0-6) 

The asymmetry is explicitly documented in the index-ng canister's `process_balance_changes`, which uses `get_fee_collector_107()` (not the legacy `get_fee_collector()`) for `Approve` blocks: [8](#0-7) 

And confirmed by the test comment: [9](#0-8) 

---

### Impact Explanation

Any ICRC-1/ICRC-2 ledger that has a `fee_collector` configured (via `change_fee_collector` in upgrade args) will silently burn all `Approve` fees instead of routing them to the `fee_collector`. The `fee_collector` account receives zero revenue from `icrc2_approve` calls. This is a **ledger conservation bug**: tokens are permanently destroyed that should instead be credited to the fee_collector. For high-volume ledgers with active ICRC-2 approval usage, this represents a continuous, unrecoverable loss of fee revenue for the fee_collector operator.

---

### Likelihood Explanation

The likelihood is **high**. Any unprivileged user calling `icrc2_approve` on a ledger with a configured `fee_collector` triggers this path. No special conditions are required — the standard ICRC-2 approval flow is sufficient. The entry path is a normal ingress update call to `icrc2_approve`.

---

### Recommendation

In the `Operation::Approve` branch of `Transaction::apply` in `rs/ledger_suite/icrc1/src/lib.rs`, replace the unconditional `balances_mut().burn()` call with logic that routes the fee to `fee_collector` when one is configured, mirroring the `Transfer` branch. Specifically, introduce a helper analogous to `balances_mut().transfer()` that accepts an optional `fee_collector`, or inline the same `match fee_collector { None => burn, Some(fc) => credit(fc) }` pattern used in `Balances::transfer`.

---

### Proof of Concept

1. Deploy an ICRC-1/ICRC-2 ledger canister.
2. Upgrade it with `change_fee_collector: Some(ChangeFeeCollector::SetTo(collector_account))`.
3. Call `icrc2_approve` from any user account with a non-zero fee.
4. Observe that `icrc1_balance_of(collector_account)` remains 0 and `icrc1_total_supply` decreases by the fee amount — confirming the fee was burned, not collected.
5. Call `icrc1_transfer` from the same user account.
6. Observe that `icrc1_balance_of(collector_account)` increases by the fee — confirming Transfer correctly routes to the fee_collector.

The discrepancy between steps 4 and 6 proves the root cause. [10](#0-9)
