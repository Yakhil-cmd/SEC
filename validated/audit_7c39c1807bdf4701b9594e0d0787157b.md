Audit Report

## Title
ckDOGE Minter Never Schedules `ConsolidateUtxos` Task, Enabling UTXO Fragmentation DoS on Large Withdrawals — (`rs/dogecoin/ckdoge/minter/src/main.rs`)

## Summary
ckDOGE's `setup_tasks()` omits `schedule_now(TaskType::ConsolidateUtxos, ...)` that ckBTC includes, so `consolidate_utxos()` is never invoked at runtime. The `utxo_consolidation_threshold` init parameter is therefore dead configuration. As the minter's `available_utxos` set grows monotonically from deposits, any withdrawal requiring more than `max_num_inputs_in_transaction` (500) inputs will fail with `BuildTxError::InvalidTransaction(TooManyInputs)`, rendering large withdrawals impossible until a canister upgrade is applied.

## Finding Description
`rs/dogecoin/ckdoge/minter/src/main.rs` `setup_tasks()` (lines 46–49) schedules only `TaskType::ProcessLogic` and `TaskType::RefreshFeePercentiles`. The ckBTC equivalent at lines 54–58 of `rs/bitcoin/ckbtc/minter/src/main.rs` additionally calls `schedule_now(TaskType::ConsolidateUtxos, &IC_CANISTER_RUNTIME)`.

The task queue is a `thread_local! { static TASKS: RefCell<TaskQueue> }` defined in `rs/bitcoin/ckbtc/minter/src/tasks.rs` (line 14), which starts empty and is populated only via explicit `schedule_now`/`schedule_after` calls. Without the initial `schedule_now`, `TaskType::ConsolidateUtxos` is never inserted into `TASKS`, so the self-rescheduling loop in `run_task` (lines 167–189 of `tasks.rs`) that calls `consolidate_utxos(&runtime).await` and re-enqueues itself every 3600 seconds is never bootstrapped.

The full consolidation infrastructure — `consolidate_utxos()` in `rs/bitcoin/ckbtc/minter/src/lib.rs` (lines 1431–1457), `utxo_consolidation_threshold` state field, `select_utxos_to_consolidate` — is present and shared with ckDOGE but unreachable. Every `update_balance` call for a new deposit inserts a UTXO into `available_utxos`. With no consolidation, this set grows monotonically. When `retrieve_doge_with_approval` is called, `build_unsigned_transaction` enforces `DOGECOIN_MAX_NUM_INPUTS_IN_TRANSACTION = 500` (defined at `rs/dogecoin/ckdoge/minter/src/lib.rs` line 49). If covering the requested withdrawal amount requires more than 500 inputs, the transaction build fails. The staging deployment explicitly configures `utxo_consolidation_threshold = opt (10_000 : nat64)` (`rs/dogecoin/ckdoge/staging/README.md` line 58), confirming the parameter was intended to be active.

## Impact Explanation
Large withdrawals (ckDOGE → DOGE) fail with `TooManyInputs` when the minter's UTXO set is sufficiently fragmented. Users whose withdrawal requests fail are reimbursed, so funds are not permanently lost. However, the minter becomes unable to service large withdrawals until a canister upgrade adds the missing `schedule_now` call. This matches the **High** bounty impact: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm" — specifically, a platform-level DoS on large ckDOGE withdrawals, a live chain-key token on the Internet Computer.

## Likelihood Explanation
The ckBTC upgrade history (`rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_12_12.md` lines 19–23) documents that exactly this failure mode occurred in production: 60k fragmented UTXOs caused large withdrawal failures and required an emergency consolidation upgrade. For ckDOGE, organic usage alone (many small legitimate deposits) will naturally fragment the UTXO set over time without any attacker. A deliberate attacker can accelerate fragmentation by depositing many small UTXOs; at `retrieve_doge_min_amount = 5_000_000_000 koinus` (50 DOGE), creating 501 UTXOs costs ~25,050 DOGE. The omission is a latent operational vulnerability that activates under normal growth and is trivially exploitable.

## Recommendation
Add `schedule_now(TaskType::ConsolidateUtxos, &DOGECOIN_CANISTER_RUNTIME);` to ckDOGE's `setup_tasks()` function in `rs/dogecoin/ckdoge/minter/src/main.rs` (after line 48), mirroring the ckBTC implementation at `rs/bitcoin/ckbtc/minter/src/main.rs` lines 54–58. This bootstraps the hourly self-rescheduling consolidation loop and makes the already-configured `utxo_consolidation_threshold` parameter effective.

## Proof of Concept
A state-machine/PocketIC test can:
1. Initialize the ckDOGE minter with `max_num_inputs_in_transaction = 500` and `utxo_consolidation_threshold = 10_000`.
2. Simulate 501 `update_balance` calls, each depositing a UTXO worth exactly `deposit_doge_min_amount`, populating `available_utxos` with 501 entries.
3. Assert `available_utxos.len() > max_num_inputs_in_transaction` and that no `ConsolidateUtxos` task is present in the `TASKS` queue (inspectable via `global_timer` remaining at 0 for that task type).
4. Call `retrieve_doge_with_approval` for an amount requiring all 501 UTXOs.
5. Assert the result is `Err(BuildTxError::InvalidTransaction(TooManyInputs { .. }))`.
