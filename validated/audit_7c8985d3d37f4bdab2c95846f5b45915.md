Audit Report

## Title
Byzantine Peer Permanently Blocks `process_slot_update` After Assembly via Unguarded `peer_rx.wait_for` — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

After `AssembleResult::Done` is returned by the assembler, `process_slot_update` exits the outer `select!` and calls `peer_rx.wait_for(|p| p.is_empty()).await` at line 500 with no cancellation guard. A Byzantine subnet peer that advertises an artifact and never sends a slot retraction keeps the `PeerCounter` permanently non-empty, causing the task to block indefinitely. The assembled artifact is inserted into the unvalidated pool via `UnvalidatedArtifactMutation::Insert` but the corresponding `Remove` is never sent, and the task does not terminate even after `cancellation_token.cancel()` is called.

## Finding Description

In `process_slot_update` (lines 480–536), the outer `select!` races three futures: cancellation, assembly, and all-peers-deleted. Once the `assemble_result` branch wins, the `select!` is resolved and the cancellation token is no longer polled. Inside `AssembleResult::Done`:

1. `UnvalidatedArtifactMutation::Insert` is sent at line 489.
2. `peer_rx.wait_for(|p| p.is_empty()).await` is called at line 500 — outside any `select!`, with no cancellation arm.
3. `UnvalidatedArtifactMutation::Remove` at line 505 is only reached if step 2 returns.

The same unguarded pattern exists for `AssembleResult::Unwanted` at line 521.

`peer_rx.wait_for` unblocks only when the `PeerCounter` becomes empty or when the `watch::Sender` is dropped. The sender lives in `active_assembles` and is only removed in `handle_artifact_processor_joined` at line 318 — but that function is only called when the task has already joined. Since the task is blocked at `wait_for`, it never joins, so the sender is never dropped, and `wait_for` never returns via the error path either.

The only other unblocking path is `handle_topology_update` (lines 560–563), which removes peers that leave the subnet topology. A Byzantine peer that remains a valid subnet member and simply withholds the slot retraction is never removed by this path.

Both call sites are already flagged with `// TODO: NET-1774` (lines 499 and 519), confirming developer awareness of the missing cancellation guard.

## Impact Explanation

This maps to **High ($2,000–$10,000): Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

Each Byzantine peer can occupy up to `slot_limit` slots. For each slot where assembly completes, one Tokio task is permanently suspended and one unvalidated pool entry is never cleaned up. With multiple Byzantine peers (still below the consensus fault threshold), leaked tasks and stale pool entries accumulate for the lifetime of the replica process. This degrades the P2P consensus manager's ability to process legitimate artifacts, pollutes the unvalidated pool consumed by the consensus layer, and breaks the shutdown/cancellation invariant of the event loop.

## Likelihood Explanation

The attacker is a Byzantine subnet peer operating below the consensus fault threshold — a standard adversary in the IC threat model. The required action is trivial: send one slot advertisement (with or without the artifact body) and then stop responding. No cryptographic material, no privileged access, and no coordination with other nodes is required. The `slot_limit` per peer bounds the damage from a single peer, but does not prevent accumulation across multiple Byzantine peers or across reconnections.

## Recommendation

Replace the bare `peer_rx.wait_for(...)` calls at lines 500 and 521 with a `select!` that also arms the cancellation token:

```rust
select! {
    _ = cancellation_token.cancelled() => { return (peer_rx, id_c); }
    _ = peer_rx.wait_for(|p| p.is_empty()) => {}
}
```

This is already tracked as `TODO: NET-1774`. The fix must be applied at both call sites (lines 500 and 521). For the `AssembleResult::Done` arm, early return on cancellation means `Remove` is not sent — this is acceptable since the `JoinSet` abort on shutdown already skips cleanup, and the pool is reset on replica restart.

## Proof of Concept

```rust
#[tokio::test]
async fn byzantine_peer_blocks_task_after_assembly() {
    use tokio::time::{timeout, Duration};

    fn make_assembler() -> MockArtifactAssembler {
        let mut a = MockArtifactAssembler::default();
        a.expect_assemble_message()
            .returning(|id, _, _: PeerWatcher| {
                Box::pin(async move {
                    AssembleResult::Done {
                        message: U64Artifact::id_to_msg(id, 100),
                        peer_id: NODE_1,
                    }
                })
            });
        a
    }

    let (mut mgr, mut channels) = ReceiverManagerBuilder::new()
        .with_artifact_assembler_maker(make_assembler)
        .build();
    let cancellation = CancellationToken::new();

    // Byzantine peer advertises artifact on slot 1 but never retracts.
    mgr.handle_slot_update_receive(
        SlotUpdate { slot_number: SlotNumber::from(1), commit_id: CommitId::from(1), update: Update::Id(0) },
        NODE_1, ConnId::from(1), cancellation.clone(),
    );

    // Assembly completes; Insert arrives.
    assert!(matches!(
        channels.unvalidated_artifact_receiver.recv().await.unwrap(),
        UnvalidatedArtifactMutation::Insert(_)
    ));

    // Cancel — task should terminate.
    cancellation.cancel();

    // Task does NOT terminate: blocked at peer_rx.wait_for with no cancellation guard.
    let result = timeout(
        Duration::from_secs(5),
        mgr.artifact_processor_tasks.join_next(),
    ).await;
    assert!(result.is_ok(), "Task leaked: did not terminate after cancellation");

    // Remove is never sent; artifact leaks in unvalidated pool.
    assert!(
        timeout(Duration::from_millis(100),
            channels.unvalidated_artifact_receiver.recv()).await.is_err(),
        "Remove was never sent"
    );
}
```

The test can be added directly to the `#[cfg(test)]` module in `rs/p2p/consensus_manager/src/receiver.rs` using the existing `ReceiverManagerBuilder` and `MockArtifactAssembler` infrastructure already present in the file.
