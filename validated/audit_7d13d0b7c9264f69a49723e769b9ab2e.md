### Title
Whitelist Bypass in `receive_base_tokens` Allows Permanent Freezing of Bridged ETH in Silo Mode - (File: `engine/src/engine.rs`)

### Summary

In Silo mode, `receive_base_tokens` mints native ETH to any recipient address without performing any whitelist check. This is inconsistent with `receive_erc20_tokens`, which does check the whitelist and redirects to a fallback address. An unprivileged user can bridge ETH to a non-whitelisted address, permanently freezing those funds because the non-whitelisted address cannot submit transactions to move them.

### Finding Description

Aurora Engine's Silo mode enforces an `Address` whitelist. When enabled, only whitelisted EVM addresses may submit transactions (`is_allow_submit`). The bridge entry point `ft_on_transfer` dispatches to one of two minting paths depending on whether the predecessor is the ETH connector: [1](#0-0) 

For ERC-20 tokens, `receive_erc20_tokens` applies the whitelist check and redirects to a configured fallback address when the recipient is not whitelisted: [2](#0-1) 

For base ETH tokens, `receive_base_tokens` performs **no whitelist check whatsoever** — it directly credits the recipient's balance: [3](#0-2) 

The whitelist check function `is_allow_receive_erc20_tokens` exists and is used only in the ERC-20 path: [4](#0-3) 

`receive_base_tokens` has no equivalent guard and no fallback address redirection.

### Impact Explanation

When the Address whitelist is enabled in Silo mode, a non-whitelisted address that receives base ETH via `receive_base_tokens` cannot submit any transaction to move those funds — `is_allow_submit` will reject every attempt: [5](#0-4) 

There is no administrative recovery path: the engine has no function to forcibly transfer or burn ETH from an arbitrary address. The funds are permanently frozen. This meets the **Critical — Permanent freezing of funds** impact threshold.

### Likelihood Explanation

The attack requires no special privilege. Any user who can initiate a NEAR `ft_transfer_call` on the ETH connector (a standard, publicly accessible NEP-141 operation) can specify any EVM address in the `msg` field. If that address is not on the Silo whitelist, the ETH is minted and immediately frozen. The Silo whitelist is an opt-in operator feature, so this only affects Silo deployments, but within those deployments the path is trivially reachable by any token holder.

### Recommendation

Apply the same whitelist-and-fallback logic used in `receive_erc20_tokens` to `receive_base_tokens`. Before crediting the recipient balance, check `silo::is_allow_receive_erc20_tokens`; if the check fails and a fallback address is configured, redirect the mint to the fallback address. If no fallback is configured and the recipient is not whitelisted, return the tokens to the sender (return the full amount) rather than minting to a frozen address.

### Proof of Concept

1. Deploy Aurora in Silo mode; enable the `Address` whitelist; configure a fallback address for ERC-20 tokens but leave `receive_base_tokens` unmodified.
2. As an unprivileged NEAR account, call `ft_transfer_call` on the ETH connector with `receiver_id = aurora`, `amount = X`, and `msg` encoding a non-whitelisted EVM address `VICTIM`.
3. The ETH connector calls `ft_on_transfer` on Aurora; `predecessor_account_id == connector`, so `receive_base_tokens` is invoked.
4. `receive_base_tokens` calls `set_balance(&mut self.io, &VICTIM, &new_balance)` with no whitelist check.
5. `VICTIM` now holds `X` ETH on Aurora. Any attempt by `VICTIM` to call `submit` is rejected by `is_allow_submit` because `VICTIM` is not on the whitelist.
6. The ETH is permanently frozen: `VICTIM` cannot send it, and no admin function exists to recover it. [3](#0-2) [2](#0-1) [1](#0-0)

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }
