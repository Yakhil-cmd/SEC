Audit Report

## Title
Missing Concurrency Limit on Synchronous Call Endpoints Enables Unbounded Blocking Thread and Connection Exhaustion - (File: `rs/http_endpoints/public/src/lib.rs`)

## Summary
The IC replica's HTTP server applies `GlobalConcurrencyLimitLayer` to `/api/v2/canister/.../call` but explicitly omits it for the synchronous call endpoints `/api/v3`, `/api/v4`, and `/api/v4/subnet`. Each unguarded request spawns a blocking task for cryptographic signature verification and, for the synchronous variants, holds an open connection waiting for certified execution results. An unprivileged attacker can flood these endpoints to exhaust Tokio's blocking thread pool and connection resources without triggering the 429 load-shedding that protects v2.

## Finding Description
In `make_router()` at `rs/http_endpoints/public/src/lib.rs` lines 603–609, `call_v2_router` is wrapped with `service_builder(GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests))`, while `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` are merged bare, with a developer TODO acknowledging the gap:

```rust
// call_v2 gets a concurrency limit
.merge(http_handler.call_v2_router.layer(service_builder(
    GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests),
)))
// TODO(CON-1574): see if there is any reasonable explicit concurrency limit we could use here.
.merge(http_handler.call_v3_router)   // NO LIMIT
.merge(http_handler.call_v4_router)   // NO LIMIT
.merge(http_handler.subnet_call_v4_router)  // NO LIMIT
```

Every request reaching these routers calls `IngressValidator::validate_ingress_message()` (`rs/http_endpoints/public/src/call.rs` lines 211–370), which unconditionally executes `tokio::task::spawn_blocking` for cryptographic signature verification (lines 327–341). The synchronous handler in `call_sync.rs` then holds the connection open waiting for `wait_for_certification()` with a configurable timeout (lines 310–327 of `call_sync.rs`).

The two existing guards — `ingress_throttler.exceeds_threshold()` (line 230) and `ingress_tx.capacity() == 0` (line 239) — check whether the ingress *pool* or submission *channel* is full. They do not bound the number of requests concurrently in the validation/waiting phase. An attacker whose requests are still being validated (before any submission) bypasses both checks entirely.

One claim in the submission is overstated: the `ingress_filter` mutex at line 343 (`ingress_filter.lock().unwrap().clone()`) is held only for the duration of a clone, not for the full request lifetime. Mutex contention is therefore minimal and not a meaningful attack vector. The real resource exhaustion vectors are the blocking thread pool (via unbounded `spawn_blocking`) and long-lived connection slots (via the certification wait timeout in the sync handler).

## Impact Explanation
This is an application/platform-level DoS against a single replica node, not raw volumetric DDoS. The attack exploits a specific missing protection: the concurrency limit that exists for v2 is absent for v3/v4. Each concurrent request consumes a Tokio blocking thread (default pool cap: 512) for crypto verification, and the synchronous handler additionally holds an HTTP connection open for up to `ingress_message_certificate_timeout_seconds`. Flooding with enough concurrent connections exhausts the blocking thread pool, stalling all blocking work on the node (including v2 call processing, state reads, and other blocking tasks), and degrades or halts the node's ability to accept legitimate ingress. This matches the allowed impact: **High — Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
The `/api/v3` and `/api/v4` endpoints are the primary call endpoints used by `ic-agent` and current tooling. They are publicly reachable through boundary nodes. No special privileges are required — any internet user can send HTTP POST requests. The attack requires only enough concurrent connections to saturate the blocking thread pool (≥512 concurrent requests in the spawn_blocking phase), which is achievable with modest resources. The TODO comment confirms this is a known, unresolved gap rather than an intentional design choice.

## Recommendation
Apply `GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests)` (with `load_shed()`) to `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` in `make_router()`, consistent with how `call_v2_router` is protected. This directly resolves the acknowledged TODO at line 606. If a different limit is appropriate for synchronous endpoints (given their longer per-request lifetime), a separate config value (e.g., `max_sync_call_concurrent_requests`) should be introduced and applied.

## Proof of Concept
1. Identify a replica node's public HTTPS endpoint.
2. Open ≥512 concurrent HTTP connections, each sending a POST to `/api/v3/canister/aaaaa-aa/call` with a valid CBOR-encoded body (well-formed enough to pass body-limit and content-type checks and reach `validate_ingress_message`).
3. Observe that unlike `/api/v2` (which returns `429 Too Many Requests` once `max_call_concurrent_requests` is exceeded), the v3/v4 endpoints accept all concurrent connections without bound.
4. Each connection spawns a `spawn_blocking` task for crypto verification; once the blocking thread pool is saturated, all blocking work on the node stalls.
5. Confirm: legitimate `/api/v2` and `/api/v3` call processing stalls; the node's ingress acceptance degrades.
6. A deterministic integration test can be written using PocketIC or a local replica: spawn N concurrent requests to `/api/v3/.../call` exceeding `max_call_concurrent_requests`, assert that the node continues to respond to `/api/v2/.../call` with 429 (load-shed) while v3 accepts all N without shedding, and measure blocking thread pool saturation via metrics.
