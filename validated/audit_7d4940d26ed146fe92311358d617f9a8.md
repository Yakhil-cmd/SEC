Audit Report

## Title
`merge_neurons` Silently Launders Stale Deciding Voting Power Into Fresh Neuron — (File: `rs/nns/governance/src/governance/merge_neurons.rs`)

## Summary
`MergeNeuronsTargetEffect::apply()` never updates `voting_power_refreshed_timestamp_seconds` on the target neuron. When a stale source neuron (whose `deciding_voting_power` has been linearly reduced toward zero due to inactivity exceeding `DEFAULT_START_REDUCING_VOTING_POWER_AFTER_SECONDS` = 6 months) is merged into a freshly-refreshed target neuron, the combined neuron retains the target's recent timestamp. The source neuron's entire stake is thereby credited with full deciding voting power without the owner ever voting or calling `RefreshVotingPower`, bypassing the staleness-penalty mechanism introduced for NNS governance.

## Finding Description
`potential_and_deciding_voting_power()` in `rs/nns/governance/src/neuron/types.rs` computes `deciding_voting_power` by multiplying `potential_voting_power` by an `adjustment_factor` derived exclusively from `self.voting_power_refreshed_timestamp_seconds`:

```rust
// types.rs L389-399
let adjustment_factor: Decimal = {
    let time_since_last_refreshed = Duration::from_secs(
        now_seconds.saturating_sub(self.voting_power_refreshed_timestamp_seconds),
    );
    voting_power_economics
        .deciding_voting_power_adjustment_factor(time_since_last_refreshed)
};
let deciding_voting_power = adjustment_factor * potential_voting_power.floor();
```

The adjustment factor is 1.0 for neurons refreshed within 6 months, then decreases linearly to 0.0 over the following month (`DEFAULT_START_REDUCING_VOTING_POWER_AFTER_SECONDS` = 6 months, `DEFAULT_CLEAR_FOLLOWING_AFTER_SECONDS` = 1 month, confirmed at `rs/nns/governance/src/network_economics.rs` L296–298).

`MergeNeuronsTargetEffect::apply()` at `merge_neurons.rs` L119–130 only updates `dissolve_state_and_age`, `maturity_e8s_equivalent`, staked maturity, and `eight_year_gang_bonus_base_e8s` — `voting_power_refreshed_timestamp_seconds` is never touched:

```rust
impl MergeNeuronsTargetEffect {
    pub fn apply(self, target_neuron: &mut Neuron) {
        target_neuron.set_dissolve_state_and_age(self.dissolve_state_and_age);
        target_neuron.maturity_e8s_equivalent = ...;
        target_neuron.add_staked_maturity(self.add_staked_maturity);
        target_neuron.eight_year_gang_bonus_base_e8s = ...;
        // voting_power_refreshed_timestamp_seconds: never set
    }
}
```

A `grep` for `voting_power_refreshed_timestamp_seconds` in `merge_neurons.rs` returns **zero matches**, confirming the field is entirely absent from the merge logic.

`calculate_merge_neurons_effect()` at L303–315 constructs `MergeNeuronsEffect` with no field for the source neuron's refresh timestamp, so the information is discarded before `apply()` is even called.

`validate_request_and_neurons()` at L486–603 checks dissolve state, spawning status, Neurons' Fund membership, managers, KYC, and neuron type — but imposes **no constraint** on the relative staleness of the two neurons' `voting_power_refreshed_timestamp_seconds`.

After `merge_neurons()` applies the effect at `governance.rs` L2496–2510, the target neuron's `voting_power_refreshed_timestamp_seconds` is whatever it was before the merge — the source neuron's staleness is silently discarded.

**Exploit flow:**
1. Attacker holds large stale neuron S (inactive ≥7 months → `adjustment_factor = 0`, `deciding_voting_power = 0`).
2. Attacker maintains small fresh neuron F (voted recently → `adjustment_factor = 1`).
3. Attacker calls `manage_neuron` → `Merge { source_neuron_id: S }` targeting F.
4. After merge: F holds combined stake of S+F, retains F's recent `voting_power_refreshed_timestamp_seconds`.
5. F's `deciding_voting_power = 1.0 × potential_voting_power(S+F)` instead of the correct `0 × potential_voting_power(S) + 1.0 × potential_voting_power(F)`.

## Impact Explanation
This is a **High** severity NNS governance security impact. The deciding voting power reduction mechanism — designed to penalize inactive neurons and encourage participation — can be fully bypassed by any neuron controller who owns both a stale and a fresh neuron. A sufficiently large stale neuron merged into a minimal fresh neuron can recover up to 100% of the deciding voting power that the staleness penalty was designed to remove, without casting a single vote or calling `RefreshVotingPower`. Because `deciding_voting_power` governs NNS proposal tallying (confirmed by 25 uses in `voting.rs`), this directly affects governance outcomes. This matches the allowed impact: **"Significant NNS security impact with concrete user or protocol harm."**

## Likelihood Explanation
**Medium.** The attacker needs only to: (1) control two neurons under the same principal — standard for large ICP holders; (2) allow one neuron to go stale for ≥7 months — trivially achieved by stopping voting/following; (3) keep a second neuron fresh by voting or calling `RefreshVotingPower` periodically; (4) issue a single standard `ManageNeuron::Merge` ingress message. No privileged access, threshold corruption, or social engineering is required. The operation is available to any neuron controller on mainnet and is repeatable.

## Recommendation
In `MergeNeuronsTargetEffect` (and its construction in `calculate_merge_neurons_effect`), carry the source neuron's `voting_power_refreshed_timestamp_seconds` and, when applying the effect to the target, set the target's `voting_power_refreshed_timestamp_seconds` to `min(source.voting_power_refreshed_timestamp_seconds, target.voting_power_refreshed_timestamp_seconds)`. This ensures the combined neuron's deciding voting power correctly reflects the staleness of the less-active constituent. Alternatively, prohibit merging neurons whose `voting_power_refreshed_timestamp_seconds` differ by more than `start_reducing_voting_power_after_seconds`, analogous to existing checks that prohibit merges with mismatched dissolve states.
