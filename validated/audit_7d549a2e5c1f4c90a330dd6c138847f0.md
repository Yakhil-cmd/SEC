Audit Report

## Title
Unbounded `BouncerValue::Wants` Window in `CertifierBouncer::new_bouncer` Enables Unvalidated Pool Flooding by a Byzantine Peer — (`rs/consensus/certification/src/certifier.rs`)

## Summary

`CertifierBouncer::new_bouncer` returns `BouncerValue::Wants` for every `CertificationMessageId` at any height ≥ `cup_height` not already in `certified_heights`, with no upper-bound guard. Combined with `SLOT_TABLE_NO_LIMIT` for the certification channel and the absence of any size cap on the unvalidated pool, a single Byzantine subnet peer below the f < n/3 fault threshold can flood the unvalidated certification pool with syntactically valid but cryptographically invalid artifacts at arbitrarily large heights, causing unbounded memory growth and eventual OOM-induced replica crashes.

## Finding Description

**1. Unbounded bouncer window (confirmed)**

`CertifierBouncer::new_bouncer` at [1](#0-0)  returns `Wants` for every height ≥ `cup_height` not in `certified_heights`, with no upper-bound check. By contrast, the consensus bouncer caps acceptance at `next_cup_height + ACCEPTABLE_NOTARIZATION_CUP_GAP` and uses `LOOK_AHEAD = 10` for per-type windows [2](#0-1) , and the IDkg bouncer uses `LOOK_AHEAD` for every message type [3](#0-2) . The certification bouncer has no equivalent guard.

**2. No slot-table limit for certification (confirmed)**

The certification channel is registered with `SLOT_TABLE_NO_LIMIT = usize::MAX` [4](#0-3)  at [5](#0-4) . The slot-table enforcement in `handle_slot_update_receive` only drops new entries when `peer_slot_table_len >= slot_limit` [6](#0-5) ; with `usize::MAX` that check never fires.

**3. Unvalidated pool has no size limit (confirmed)**

`CertificationPoolImpl::insert` unconditionally inserts into an in-memory `BTreeMap` with no capacity check [7](#0-6) .

**4. Certifier only validates heights the state manager requests (confirmed)**

`CertifierImpl::on_state_change` calls `validate` only over `state_heights_to_aggregate_and_validate`, derived from `list_state_hashes_to_certify` / `list_state_heights_to_certify` [8](#0-7) . Artifacts at heights far above the current execution tip are never validated and therefore never removed from the unvalidated pool.

**5. Purger only removes artifacts *below* `cup_height − MINIMUM_CHAIN_LENGTH` (confirmed)**

`get_purge_height` issues `RemoveAllBelow(cup_height − MINIMUM_CHAIN_LENGTH)` [9](#0-8) . Artifacts at heights far *above* `cup_height` are never touched by the purger.

**6. Download loop retries indefinitely (confirmed)**

`download_artifact` uses `with_max_elapsed_time(None)` [10](#0-9) . Download tasks for future-height adverts never become `Unwanted` because the bouncer window is unbounded, so they loop forever consuming resources.

**Exploit flow:**

1. Byzantine peer advertises `CertificationMessageId`s on slots 1…N (N unbounded) at heights H >> `cup_height`.
2. Bouncer returns `Wants` for every one; N download tasks are spawned and N entries are added to `active_assembles`.
3. The peer responds to fetch requests with syntactically valid `CertificationShare` artifacts constructed with its own legitimate signing key at arbitrary heights. Each artifact passes the `message.id() == id` check in the downloader.
4. Each artifact is delivered to the unvalidated pool via `UnvalidatedArtifactMutation::Insert` and stored unconditionally.
5. The certifier never validates or purges these artifacts because the state manager never requests those heights.
6. The unvalidated `BTreeMap` and `HeightIndex` structures grow without bound → OOM → replica crash.

## Impact Explanation

A single Byzantine subnet peer (below the f < n/3 fault threshold) can crash all honest replicas on a subnet via OOM, halting the subnet and disrupting certified state, xnet stream delivery, and state sync. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

- Requires one compromised subnet node — explicitly within the threat model.
- No threshold-majority corruption needed; a single node suffices.
- No cryptographic forgery required; the attacker uses its own legitimate signing key to produce syntactically valid shares at arbitrary heights.
- The attack is mechanically simple: send many slot-update messages with distinct slot numbers and respond to fetch requests with crafted artifacts.
- Repeatable and persistent: the attack continues until the victim crashes or the bouncer is patched.

## Recommendation

1. **Add an upper-bound guard to `CertifierBouncer::new_bouncer`** analogous to the consensus and IDkg bouncers. Return `MaybeWantsLater` for heights beyond `cup_height + LOOK_AHEAD` (or a certification-specific constant).
2. **Set a finite `SLOT_TABLE_LIMIT_CERTIFICATION`** (similar to `SLOT_TABLE_LIMIT_INGRESS = 50_000`) for the certification channel in `rs/replica/setup_ic_network/src/lib.rs`.
3. **Add a size cap to the unvalidated certification pool** and drop insertions that exceed it.
4. **Periodically validate and discard unvalidated artifacts at heights far above the current execution tip**, rather than waiting for the state manager to request them.

## Proof of Concept

```rust
// Unit test: confirm unbounded Wants window
let cup_height = Height::from(0);
let certified_heights: HashSet<Height> = HashSet::new();
let bouncer = move |id: &CertificationMessageId| {
    let h = id.height;
    if h < cup_height || certified_heights.contains(&h) {
        BouncerValue::Unwanted
    } else {
        BouncerValue::Wants
    }
};
for h in [1u64, 1_000, 1_000_000, u64::MAX - 1] {
    let id = CertificationMessageId {
        height: Height::from(h),
        hash: CertificationMessageHash::CertificationShare(
            CryptoHashOf::from(CryptoHash(vec![]))
        ),
    };
    assert_eq!(bouncer(&id), BouncerValue::Wants,
        "height {h} should be Wants — no upper bound exists");
}
// All assertions pass: the window is truly unbounded.
```

For an integration-level PoC: configure a local subnet with a malicious peer that advertises `N = 100_000` distinct `CertificationShare` slot updates at heights `cup_height + 1` through `cup_height + 100_000`, responds to each fetch with a well-formed but cryptographically invalid share, and observe the victim replica's RSS grow without bound until OOM.
