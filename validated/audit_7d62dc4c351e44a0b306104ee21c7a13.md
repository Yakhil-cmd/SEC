### Title
Unvalidated Zero-Address `coinbase` Causes Permanent Loss of Operator Fee Funds - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In ZKsync OS, the `coinbase` address used as the operator fee recipient is sourced directly from the block context metadata with no zero-address validation. When `coinbase` is `B160::ZERO` (the default value), all operator fees minted from the treasury are permanently transferred to the zero address and are irrecoverably lost. This is an exact structural analog to the reported `feeReceiver` unset bug: an uninitialized/default-zero fee recipient address causes public funds to be silently burned.

---

### Finding Description

In `process_l1_transaction.rs`, after computing `pay_to_operator`, the bootloader calls `system.get_coinbase()` and immediately passes the result to `mint_base_token` without any zero-address check:

```rust
let coinbase = system.get_coinbase();
// Mint operator fee portion of the deposit to coinbase.
mint_base_token::<S, Config>(
    system,
    system_functions,
    memories.reborrow(),
    &pay_to_operator,
    &coinbase,   // <-- no zero-address guard
    ...
)?;
```

The `coinbase` value comes from `BlockMetadataFromOracle`, whose `new_for_test()` constructor explicitly sets `coinbase: B160::ZERO`, and the `BlockContext::default()` implementation also sets `coinbase: B160::default()` (i.e., all-zeros).

The same pattern exists in the ZK transaction flow's `refund_and_commit_fee` in `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs` (line 518) and in the Ethereum transaction flow in `basic_bootloader/src/bootloader/transaction_flow/ethereum/mod.rs` (line 594).

The `mint_base_token` function calls `transfer_from_treasury`, which performs a real balance deduction from `BASE_TOKEN_HOLDER_ADDRESS` and a credit to the zero address — permanently destroying those tokens.

---

### Impact Explanation

Any L1→L2 priority transaction processed in a block where `coinbase` is the zero address results in the operator fee (`gas_used * gas_price`) being minted to `address(0)` from the treasury. These tokens are permanently inaccessible. For a block with many L1→L2 transactions or high gas prices, this can represent a significant loss of base token funds from the treasury. The treasury balance is finite (initialized to `2^128 - 1`), so repeated losses compound.

**Impact class**: Public funds loss path — treasury tokens are irreversibly transferred to the zero address.

---

### Likelihood Explanation

The `BlockContext::default()` sets `coinbase` to `B160::ZERO`. Any sequencer or block producer that omits setting the coinbase field (e.g., during initialization, misconfiguration, or a protocol upgrade transition) will trigger this path. The `BlockMetadataFromOracle::new_for_test()` also hardcodes `coinbase: B160::ZERO`, confirming this is a realistic default state. The attacker-controlled entry path is simply submitting any L1→L2 transaction to a block with an unset coinbase — no special privileges required.

---

### Recommendation

Add a zero-address guard before calling `mint_base_token` with the coinbase address in `process_l1_transaction`. If `coinbase.is_zero()`, either return an internal error (halting block processing) or skip the fee transfer. The same guard should be applied in `refund_and_commit_fee` for both the ZK and Ethereum transaction flows.

```rust
let coinbase = system.get_coinbase();
if coinbase == B160::ZERO {
    return Err(internal_error!("coinbase address is zero").into());
}
mint_base_token::<S, Config>(..., &coinbase, ...)?;
```

---

### Proof of Concept

1. Submit any L1→L2 priority transaction with `gas_price > 0` and `gas_limit > 0` to a block where `coinbase` is not set (defaults to `B160::ZERO`).
2. In `process_l1_transaction`, `pay_to_operator = gas_used * gas_price > 0`.
3. `system.get_coinbase()` returns `B160::ZERO`.
4. `mint_base_token` is called with `to = B160::ZERO`, which calls `transfer_from_treasury` — deducting `pay_to_operator` from `BASE_TOKEN_HOLDER_ADDRESS` and crediting `address(0)`.
5. The treasury balance decreases permanently; the operator receives nothing; the funds are unrecoverable.

    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
