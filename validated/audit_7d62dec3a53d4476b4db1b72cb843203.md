Audit Report

## Title
Unbounded Body Buffering in `RosettaMetricsMiddleware` Allows Unauthenticated OOM Crash of ICRC-1 Rosetta Process — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
`RosettaMetricsMiddleware` unconditionally buffers the full HTTP request body via `to_bytes(body, usize::MAX)` for every non-`/metrics` request. The ICRC-1 Rosetta Axum router applies no body-size limit anywhere in its layer stack. An unauthenticated attacker can send a single arbitrarily large POST body to any endpoint, exhausting the process heap and causing an OOM kill of the ICRC-1 Rosetta service.

## Finding Description
In `rs/rosetta-api/common/rosetta_core/src/metrics.rs`, `extract_canister_id` reads the entire request body with no upper bound:

```rust
let bytes = to_bytes(body, usize::MAX).await?;  // line 389
```

This is called unconditionally from `RosettaMetricsMiddleware::call` for every request that is not `GET /metrics` (lines 304–339), before the request is dispatched to any route handler. The body is consumed and buffered entirely into heap memory.

In `rs/rosetta-api/icrc1/src/main.rs` (lines 361–391), the router is built with no body-size guard anywhere in the layer stack — no `DefaultBodyLimit`, no `RequestBodyLimitLayer`, no `Content-Length` rejection. Axum's extractor-level default body limit (applied to `Json<T>` etc.) is irrelevant here because the middleware consumes the body before any handler extractor runs.

By contrast, the ICP Rosetta (Actix-web) enforces a 4 MB `JsonConfig` limit at `rs/rosetta-api/icp/src/rosetta_server.rs` lines 296–303, confirming the ICRC-1 omission is unintentional.

## Impact Explanation
Taking down the ICRC-1 Rosetta process denies all ICRC-1 token users access to the Rosetta API, a listed in-scope financial integration component. This is an application-level DoS crash triggered by a single unauthenticated request — not raw volumetric DDoS — matching the High impact class: *"Application/platform-level DoS, crash... not based on raw volumetric DDoS"* and *"Significant... Rosetta... security impact with concrete user or protocol harm."*

## Likelihood Explanation
The attack requires only a TCP connection to the Rosetta HTTP port. No credentials, tokens, or prior knowledge are needed beyond the open port. A single streaming POST to any listed endpoint (e.g., `/construction/submit`) is sufficient. The condition is trivially repeatable after each operator restart.

## Recommendation
Add an explicit body-size limit as the outermost layer in the Axum router in `rs/rosetta-api/icrc1/src/main.rs`:

```rust
use axum::extract::DefaultBodyLimit;

let app = Router::new()
    // ... routes ...
    .layer(metrics_layer)
    .layer(add_request_span())
    .layer(RequestIdLayer)
    .layer(DefaultBodyLimit::max(4 * 1024 * 1024)) // 4 MB, consistent with ICP Rosetta
    .with_state(token_app_states.clone());
```

Alternatively, replace `usize::MAX` with a bounded constant in `extract_canister_id` in `rs/rosetta-api/common/rosetta_core/src/metrics.rs` so the middleware is safe regardless of router configuration.

## Proof of Concept
```bash
# Stream a 2 GB body to any Rosetta endpoint; observe OOM kill before any 413 is returned
dd if=/dev/zero bs=1M count=2048 | \
  curl -X POST http://<rosetta-host>:8080/construction/submit \
       -H "Content-Type: application/json" \
       --data-binary @- \
       -v
# Confirm: dmesg | grep -i oom  OR  journalctl -u rosetta
```

A deterministic integration test can be written by spawning the ICRC-1 Rosetta server locally and sending a POST with a body larger than available heap, asserting the process exits with a non-zero status before returning any HTTP response.
