### Title
Missing Response-Body Timeout in `send_post_request` Allows Indefinite Hang of Rosetta API Submission Path - (File: rs/rosetta-api/icp/src/ledger_client.rs)

### Summary
The `send_post_request` helper in the ICP Rosetta API applies a per-request timeout only to the `.send().await` call (i.e., until response headers are received). The subsequent `resp.bytes().await` body-collection call carries no timeout. A replica or boundary node that sends HTTP headers promptly but then stalls on streaming the response body will cause the Rosetta API worker to block indefinitely on that call, exhausting available async tasks and preventing further ICP ledger transaction submissions.

### Finding Description
In `rs/rosetta-api/icp/src/ledger_client.rs`, the function `send_post_request` is:

```rust
async fn send_post_request(
    http_client: &reqwest::Client,
    url: &str,
    body: Vec<u8>,
    timeout: Duration,
) -> Result<(Vec<u8>, reqwest::StatusCode), String> {
    let resp = http_client
        .post(url)
        .header(reqwest::header::CONTENT_TYPE, "application/cbor")
        .body(body)
        .timeout(timeout)   // ← covers only until headers are received
        .send()
        .await
        .map_err(|err| format!("sending post request failed with {err}: "))?;
    let resp_status = resp.status();
    let resp_body = resp
        .bytes()            // ← NO timeout; hangs indefinitely if body stalls
        .await
        .map_err(|err| format!("receive post response failed with {err}: "))?
        .to_vec();
    Ok((resp_body, resp_status))
}
``` [1](#0-0) 

In reqwest, `.timeout()` on a `RequestBuilder` is applied to the `send()` future, which resolves when response headers are received. Once `send().await` returns a `Response`, the timeout is no longer active. The subsequent `resp.bytes().await` call — which streams and collects the full response body — is a separate, unguarded future with no deadline.

This function is called from `do_request` in the same file, which drives the ICP ledger transaction submission loop: [2](#0-1) 

The caller computes a `wait_timeout` from a 20-second `TIMEOUT` constant and passes it to `send_post_request`, but that timeout only protects the header-receipt phase. [3](#0-2) 

### Impact Explanation
If the IC replica (or an intermediate network element) sends HTTP response headers immediately but then stalls on transmitting the body bytes, `resp.bytes().await` will block the Rosetta API async task indefinitely. Because `do_request` is called inside the `submit` path of `LedgerAccess`, all in-flight ICP ledger transaction submissions routed through that Rosetta instance will be blocked. Repeated triggering exhausts the async runtime's task capacity, causing a full denial-of-service against the Rosetta API's transaction submission endpoint. This directly impacts ICP ledger users and exchange integrations that rely on Rosetta for ICP transfers.

### Likelihood Explanation
The Rosetta API connects to a configurable IC replica URL. Any network-level stall (packet loss, TCP window exhaustion, a slow replica under load, or a malicious boundary node that deliberately withholds body bytes after sending headers) can trigger the hang. No authentication or special privilege is required to cause a replica to be slow; a ledger user submitting a high volume of transactions can indirectly induce replica load. The missing timeout is a structural gap that will manifest under realistic adverse network or load conditions.

### Recommendation
Wrap the `resp.bytes().await` call in an explicit `tokio::time::timeout`, using the same `timeout` duration already passed to the function, mirroring the pattern used elsewhere in the codebase (e.g., `rs/canister_client/src/http_client.rs` `wait_for_one_http_request` which applies `tokio::time::timeout_at` to both the response future and the body-collection future): [4](#0-3) 

Concretely, replace the unguarded `resp.bytes().await` with:
```rust
tokio::time::timeout(timeout, resp.bytes())
    .await
    .map_err(|_| format!("receive post response timed out after {timeout:?}"))?
    .map_err(|err| format!("receive post response failed with {err}: "))?
    .to_vec()
```

### Proof of Concept
1. Stand up a TCP server that immediately sends a valid HTTP/1.1 `200 OK` response with `Content-Length: 1000` but then writes no body bytes and holds the connection open.
2. Point the Rosetta API at this server as its IC replica URL.
3. Submit any ICP ledger transaction via the Rosetta `/construction/submit` endpoint.
4. Observe that `send_post_request` never returns; the Rosetta API worker is permanently blocked on `resp.bytes().await` with no timeout firing, while the 20-second `TIMEOUT` deadline in `do_request` is never enforced for the body-read phase.
