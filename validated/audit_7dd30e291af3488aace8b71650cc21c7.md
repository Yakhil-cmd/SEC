Audit Report

## Title
TOCTOU Snapshot Griefing Permanently Fails Migration Requests via `FatalFailure` in `process_controllers_changed` - (`rs/migration_canister/src/processing.rs`)

## Summary

The Migration Canister checks for the absence of snapshots on the replaced canister during upfront validation (`validate_request`) and again inside `process_controllers_changed`. Between these two checks, original controllers of the replaced canister retain full control and can call `take_canister_snapshot`. When `process_controllers_changed` re-checks and finds a snapshot, it unconditionally returns `ProcessingResult::FatalFailure`, permanently transitioning the migration to `RequestState::Failed`. A malicious co-controller can repeat this on every retry, indefinitely blocking migration of the targeted canister pair.

## Finding Description

**Check 1 – Upfront validation (passes):**

`validate_request` in `rs/migration_canister/src/validation.rs` at line 156 calls `assert_no_snapshots(replaced_canister)`. If no snapshot exists, validation succeeds and the request is inserted as `RequestState::Accepted`.

**Attack window:**

`process_accepted` in `rs/migration_canister/src/processing.rs` (lines 83–109) calls `set_exclusive_controller` on the migrated canister first, then on the replaced canister. Until the second `set_exclusive_controller` call completes, original controllers of the replaced canister retain controller status and can call the management canister's `take_canister_snapshot`. The timer fires every 1 second (`rs/migration_canister/src/lib.rs` line 461), so the window is seconds-wide but observable via the `migration_status` query.

**Check 2 – Fatal re-check in `process_controllers_changed`:**

`rs/migration_canister/src/processing.rs` lines 160–170:
```rust
match assert_no_snapshots(request.replaced_canister).await {
    ProcessingResult::Success(_) => {}
    ProcessingResult::NoProgress => return ProcessingResult::NoProgress,
    ProcessingResult::FatalFailure(_) => {
        return ProcessingResult::FatalFailure(RequestState::Failed {
            request,
            recovery_state: RecoveryState::new(),
            reason: "Replaced canister has snapshots.".to_string(),
        });
    }
}
```

`assert_no_snapshots` in `rs/migration_canister/src/external_interfaces/management.rs` lines 265–288 returns `ProcessingResult::FatalFailure` for any non-empty snapshot list. This causes `process_controllers_changed` to emit `FatalFailure`, which `transition` in `processing.rs` lines 511–514 converts to `RequestState::Failed`. `process_failed` then runs controller recovery and, once done, emits `EventType::Failed` to HISTORY, permanently closing the request.

**Why existing guards are insufficient:**

By the time `process_controllers_changed` runs, the migration canister is already the exclusive controller of the replaced canister (set in `process_accepted`). It therefore has the authority to call `delete_canister_snapshot` on any existing snapshots. Instead of doing so, the code fatally fails. The `FatalFailure` path is not recoverable for that request — the user must resubmit, giving the attacker another window.

The execution-layer `rename_canister` check (`rs/execution_environment/src/canister_manager.rs` lines 2979–2981) independently rejects rename if snapshots exist, confirming the snapshot must be cleared before the migration can proceed, but the migration canister never attempts deletion.

## Impact Explanation

A co-controller of the replaced canister can permanently block migration of a specific canister pair by repeatedly taking a snapshot in the seconds-wide window between validation and `process_accepted`. Each retry by the legitimate user gives the attacker another opportunity. The migration canister restores original controllers on failure, so the attacker regains controller status after each failed attempt and can repeat indefinitely. This constitutes a targeted, repeatable application-level DoS against the migration service for the affected canister pair, matching the **Medium ($200–$2,000)** impact class: meaningful security impact with a concrete constraint (co-controller status required).

## Likelihood Explanation

- **Attacker entry path**: Any co-controller of the replaced canister. `take_canister_snapshot` requires only controller status, which is a normal operational role, not a special governance privilege.
- **Timing**: The attack window is the interval between `migrate_canister` returning `Ok` and `process_accepted` calling `set_exclusive_controller(replaced_canister)` — observable via the public `migration_status` query.
- **Cost**: Zero. The attacker can delete and re-create snapshots on every retry.
- **Repeatability**: Unlimited. Controller status is restored by the migration canister's own recovery logic after each failure.

## Recommendation

1. **Primary fix**: After `set_exclusive_controller(replaced_canister)` succeeds in `process_accepted` (or at the start of `process_controllers_changed`), enumerate and delete all snapshots of the replaced canister using `delete_canister_snapshot` before proceeding. The migration canister is already the exclusive controller at this point and has the authority to do so.

2. **Minimum mitigation**: Downgrade the `FatalFailure` branch in `process_controllers_changed` for the snapshot case to `NoProgress` (retry), and add a snapshot deletion step before the re-check, so the migration retries rather than permanently failing.

3. **Do not** rely solely on the upfront `assert_no_snapshots` check in `validate_request` — the TOCTOU window cannot be eliminated without proactive deletion.

## Proof of Concept

```
1. User A (controller of both canisters) calls migrate_canister(migrated, replaced).
   → validate_request passes (no snapshots exist).
   → Request inserted as RequestState::Accepted.
   → migrate_canister returns Ok(()).

2. Attacker B (co-controller of `replaced`) immediately calls:
   management_canister.take_canister_snapshot({ canister_id: replaced, replace_snapshot: None })
   (callable by any controller; no special privilege required)

3. Timer fires → process_accepted runs:
   → set_exclusive_controller(migrated) succeeds.
   → set_exclusive_controller(replaced) succeeds.
   → State transitions to RequestState::ControllersChanged.
   (snapshot already exists; set_exclusive_controller does not check for snapshots)

4. Timer fires → process_controllers_changed runs:
   → assert_no_snapshots(replaced) finds snapshot → ProcessingResult::FatalFailure
   → RequestState::Failed { reason: "Replaced canister has snapshots." }

5. process_failed runs → restores original controllers of both canisters
   → EventType::Failed recorded in HISTORY.
   → Attacker B is a controller of `replaced` again.

6. User A resubmits migrate_canister → Attacker B repeats step 2.
   Migration is permanently blocked.
```

**Reproducible test plan**: Using PocketIC (already used in `rs/migration_canister/tests/tests.rs`), create two canisters with two controllers each, call `migrate_canister`, immediately call `take_canister_snapshot` from the second controller, advance time, and assert that `migration_status` returns `Failed { reason: "Replaced canister has snapshots." }`. pub async fn assert_no_snapshots(canister_id: Principal) -> ProcessingResult<(), ValidationError> {
    match list_canister_snapshots(&ListCanisterSnapshotsArgs { canister_id }).await {
        Ok(snapshots) if snapshots.is_empty() => ProcessingResult::Success(()),
        Ok(_) => {
            ProcessingResult::FatalFailure(ValidationError::ReplacedCanisterHasSnapshots(Reserved))
        }
        Err(CallError::CallRejected(e))
            if e.reject_code() == Ok(RejectCode::DestinationInvalid) =>
        {
            println!(
                "Call `list_canister_snapshots` for {} returned DestinationInvalid, treating as success",
                canister_id
            );
            ProcessingResult::Success(())
        }
        Err(e) => {
            println!(
                "Call `list_canister_snapshots` for {} failed: {:?}",
                canister_id, e
            );
            ProcessingResult::NoProgress
        }
    }
}
