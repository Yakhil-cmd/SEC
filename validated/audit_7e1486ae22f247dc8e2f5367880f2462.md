Audit Report

## Title
Unbounded Synchronous Iteration in SNS `distribute_rewards` Can Permanently Halt Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary

SNS governance's `distribute_rewards` function iterates over all ballots of all `ReadyToSettle` proposals and all voting neurons in a single synchronous execution with no instruction-limit guard. NNS governance fixed this identical class of bug by moving reward distribution to a batched timer with an `is_over_instructions_limit` check. SNS governance has no equivalent protection. At the ceiling parameter values (700 proposals × 200,000 neurons), the computation far exceeds the IC's 5-billion-instruction message limit, causing the heartbeat to trap and permanently blocking reward distribution.

## Finding Description

**Root cause — `distribute_rewards` (no batching, no instruction guard):**

In `rs/sns/governance/src/governance.rs` at lines 5892–5997, `distribute_rewards` performs two nested unbounded loops with no instruction-limit check:

```rust
// Loop 1: over all considered proposals (up to 700)
for proposal_id in &considered_proposals {
    if let Some(proposal) = self.get_proposal_data(*proposal_id) {
        // Loop 2: over all ballots per proposal (up to 200,000)
        for (voter, ballot) in &proposal.ballots {
            // Decimal arithmetic + HashMap insert/update per entry
            *neuron_id_to_reward_shares.entry(neuron_id).or_insert_with(|| dec!(0)) += reward_shares;
        }
    }
}
// Loop 3: over all rewarded neurons — neuron mutation per entry
for (neuron_id, neuron_reward_shares) in neuron_id_to_reward_shares {
    neuron.maturity_e8s_equivalent += neuron_reward_e8s;
}
```

A grep search for `is_message_over_threshold`, `is_over_instructions_limit`, and any instruction-limit guard in `rs/sns/governance/src/*.rs` returns **no matches**. The function is called synchronously from `run_periodic_tasks` (line 5513), which is invoked from the heartbeat.

**Confirmed ceiling constants** (`rs/sns/governance/src/types.rs`, lines 386 and 390):
- `MAX_NUMBER_OF_NEURONS_CEILING = 200_000`
- `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700`

Worst-case ballot entries: 700 × 200,000 = **140 million** entries, each requiring HashMap operations and `Decimal` arithmetic. At a conservative ~50 instructions per entry, this is 7 billion instructions — 40% over the 5-billion-instruction message limit.

**Contrast with NNS governance (fixed):**

`rs/nns/governance/src/reward/distribution.rs` lines 154–188 shows the NNS fix: `continue_processing` pops rewards one at a time and calls `is_over_instructions_limit()` after each neuron, breaking and resuming in the next timer invocation. SNS has no equivalent.

**Secondary surface — `compute_ballots_for_new_proposal`** (lines 5255–5280): iterates over all neurons synchronously during `make_proposal` with no guard. The NNS benchmark (`canbench_results.yml`, line 47) records 2,450,000 instructions for a small neuron set; extrapolating to 200,000 neurons approaches the 5-billion limit for a single update call.

## Impact Explanation

When the heartbeat traps during `distribute_rewards`, the state is rolled back — the reward event is never created. On every subsequent heartbeat, `should_distribute_rewards()` returns true again, `distribute_rewards` is called again, and the trap recurs. There is no fallback path. **Voting rewards permanently stop being distributed to all SNS participants.** This is a concrete, permanent application-level DoS of the SNS reward mechanism, matching the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* at High severity ($2,000–$10,000).

## Likelihood Explanation

No special privileges are required. Any unprivileged user can stake SNS tokens and claim neurons via `manage_neuron` → `ClaimOrRefresh`. The condition worsens monotonically as the SNS grows. No attacker coordination is needed — organic growth of a successful SNS with a high `max_number_of_neurons` parameter is sufficient. The SNS governance parameter `max_number_of_neurons` is bounded by `MAX_NUMBER_OF_NEURONS_CEILING = 200,000`, which is a legitimate configuration value explicitly documented in the codebase.

## Recommendation

1. Move SNS `distribute_rewards` out of the synchronous `run_periodic_tasks` heartbeat into a dedicated periodic timer task, mirroring the NNS fix documented in `rs/nns/governance/CHANGELOG.md`.
2. Add an `is_over_instructions_limit` (or equivalent IC instruction-counter) guard inside the reward-distribution loop so work is broken into multiple messages, resuming from a stable checkpoint (e.g., a persisted `RewardsDistributionInProgress` state as NNS uses in `rs/nns/governance/src/reward/distribution.rs`).
3. Add a similar guard or neuron-count cap to `compute_ballots_for_new_proposal` to prevent proposal submission from trapping at high neuron counts.

## Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_number_of_proposals_with_ballots = 700` (both within the defined ceilings).
2. Have users stake SNS tokens and call `manage_neuron` → `ClaimOrRefresh` until the neuron count approaches 200,000. This is an unprivileged, standard operation.
3. Submit and allow 700 proposals to reach `ReadyToSettle` state. Each proposal records a ballot for every eligible neuron at creation time.
4. When the reward round ends, the heartbeat calls `run_periodic_tasks` → `distribute_rewards`. The function must iterate over 700 × 200,000 = 140 million ballot entries plus update maturity for every rewarded neuron — far exceeding the 5-billion-instruction limit.
5. The heartbeat traps. The reward event is never written. `should_distribute_rewards()` returns true on the next heartbeat. The trap recurs indefinitely. Reward distribution is permanently frozen.

A deterministic integration test using PocketIC can reproduce this by populating the SNS neuron store to the ceiling, creating 700 settled proposals, advancing the clock past the reward round boundary, and asserting that `latest_reward_event` does not advance across multiple heartbeat invocations.
