### Title
EVM Refund Counter Not Reset Between Transactions Enables Cross-Transaction Gas Refund Inflation - (File: `basic_system/src/system_implementation/caches/generic_pubdata_aware_plain_storage.rs`)

---

### Summary

The `evm_refunds_counter` field in `GenericPubdataAwarePlainStorage` accumulates EVM SSTORE gas refunds across the lifetime of a block but is **never reset to zero between transactions**. Refunds earned in one transaction carry over and inflate the refund applied to subsequent transactions in the same block, causing incorrect gas accounting and financial loss to the operator.

---

### Finding Description

ZKsync OS tracks EVM gas refunds (EIP-2929/EIP-3529) in `evm_refunds_counter`, a `NonEmptyHistoryCounter` stored inside `GenericPubdataAwarePlainStorage`. [1](#0-0) 

It is initialized to `R::empty()` (zero) when the storage model is constructed: [2](#0-1) 

During every SSTORE, `apply_write_impl` calls `refund_for_storage_write` and then commits the updated counter value: [3](#0-2) 

The refund logic in `refund_for_storage_write` adds 4,800 gas to the counter for each non-zero-to-zero SSTORE, and up to 19,900 gas for restoring a slot to its original value: [4](#0-3) 

At the end of each transaction, `compute_gas_refund` reads the counter and applies it as a gas refund, capped at 1/5 of `gas_used` per EIP-3529: [5](#0-4) 

The `StorageSnapshotId` includes the counter for frame snapshot/rollback: [6](#0-5) 

When a transaction's frame is committed via `finish_global_frame(None)`, the counter's current value becomes the new committed baseline. **There is no call that resets `evm_refunds_counter` to zero before the next transaction begins.** The `current_tx_id` is incremented per transaction (for warm/cold access tracking), but the refund counter is not touched. The next transaction therefore starts with the accumulated counter value from all prior transactions in the block.

This is structurally identical to the TribeRedeemer vulnerability: a stored accounting variable (`redeemBase` / `evm_refunds_counter`) diverges from the correct per-operation value because it is not reset when a new operation begins, causing subsequent operations to use an inflated value.

---

### Impact Explanation

An attacker who controls a transaction earlier in a block can pre-load the refund counter by clearing pre-existing non-zero storage slots. Any subsequent transaction in the same block then reads an inflated counter and receives up to 20% of its `gas_used` as a refund it did not earn. The operator collects less fees than the actual computation cost warrants. This is a direct resource accounting bug causing financial loss to the operator and a deviation from EVM semantics (the EVM specification requires the refund counter to be per-transaction).

---

### Likelihood Explanation

Any unprivileged EVM transaction sender can trigger this. The attacker only needs pre-existing non-zero storage slots (settable in a prior block at standard SSTORE cost). No privileged access, oracle manipulation, or governance control is required. The attack is executable within a single block by any externally-owned account.

---

### Recommendation

Reset `evm_refunds_counter` to `R::empty()` at the start of each transaction. Add a `reset_refund_counter` method to `GenericPubdataAwarePlainStorage` and call it from the bootloader's per-transaction initialization path — for example, in `before_execute_transaction_payload` in `ZkTransactionFlowOnlyEOA`, at the same point where `current_tx_id` is incremented. [7](#0-6) 

---

### Proof of Concept

1. **Setup (prior block):** Attacker deploys contract C and writes 100 storage slots to non-zero values. Cost: ~100 × 20,000 = 2,000,000 gas (one-time).

2. **Tx A (block N):** Attacker calls C to SSTORE all 100 slots back to zero. Each generates 4,800 gas refund via `refund_for_storage_write`. After Tx A commits, `evm_refunds_counter` = 480,000 gas (committed baseline for the block).

3. **Tx B (block N, any sender):** A transaction using 1,000,000 gas executes. `compute_gas_refund` reads the counter = 480,000 gas (plus any refunds from Tx B's own SSTOREs). The EIP-3529 cap is 1,000,000 / 5 = 200,000 gas. Tx B receives a 200,000 gas refund it did not earn.

4. **Net result:** The attacker spent ~100 × 200 = 20,000 gas net in Tx A (5,000 SSTORE cost − 4,800 refund per slot) to give Tx B a 200,000 gas refund — a 10× leverage on fee reduction. The operator is underpaid by 200,000 × `gas_price`. [8](#0-7) [9](#0-8)
