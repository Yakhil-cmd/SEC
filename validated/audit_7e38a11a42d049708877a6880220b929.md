Audit Report

## Title
Unbounded Body Buffering in `RosettaMetricsMiddleware` Enables Unauthenticated OOM DoS — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
`extract_canister_id` passes `usize::MAX` to `axum::body::to_bytes`, meaning the entire HTTP request body is buffered into heap memory with no size cap before any handler logic runs. Because `RosettaMetricsMiddleware::call` invokes this function for every non-`/metrics` request and no outer middleware imposes a body size limit, any unauthenticated client can stream an arbitrarily large body to any Rosetta POST endpoint, exhausting process heap and triggering an OS-level OOM kill that terminates both the HTTP server and all ledger sync threads.

## Finding Description
In `extract_canister_id` ( [1](#0-0) ), `to_bytes(body, usize::MAX).await?` is called unconditionally. The `usize::MAX` argument is the axum/hyper sentinel meaning "no limit — buffer everything." This function is called for every non-`/metrics` request inside `RosettaMetricsMiddleware::call` ( [2](#0-1) ). The purpose is purely observational — extracting `network_identifier.network` from the JSON body for metrics labeling — but it does so by consuming the entire body stream into a contiguous heap allocation before the request is forwarded to any handler.

The Axum middleware stack in `main.rs` ( [3](#0-2) ) applies layers in this order (outermost to innermost): `RequestIdLayer` → `add_request_span()` → `metrics_layer` → handlers. Neither `RequestIdLayer` nor the tracing layer imposes any body size limit. There is no `tower_http::limit::RequestBodyLimitLayer` or equivalent anywhere in the stack. The error branch at line 338 (`Err(_) => (None, Body::empty())`) does not protect against OOM because the OS-level kill occurs before any Rust error can be returned.

The server binds to `0.0.0.0` ( [4](#0-3) ), making it reachable by any unauthenticated network client. The sync threads and HTTP server share the same OS process ( [5](#0-4) ), so an OOM kill terminates both.

## Impact Explanation
This is an application-level DoS exploiting a specific code defect (missing body size bound), not a raw volumetric flood. A single TCP connection sending a chunked-transfer-encoded POST body with no `Content-Length` to any endpoint (e.g., `/block`, `/account/balance`, `/construction/submit`) causes the Tokio runtime to allocate heap memory proportional to the body size. At sufficient scale the Linux OOM killer terminates the process, killing the HTTP server, all `tokio::spawn`-ed ledger sync threads, and the watchdog threads. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user and protocol harm** to a financial integration component (ICRC ledgers, ckBTC, ckETH Rosetta).

## Likelihood Explanation
The attack requires only a TCP connection to the Rosetta port and the ability to send a streaming HTTP POST — no authentication, credentials, or privileged access. The server is intended to be reachable by exchange integrators. A single attacker with a fast uplink can sustain the DoS indefinitely by reconnecting after each OOM kill, since no rate limiting or connection-level body cap exists.

## Recommendation
Replace `usize::MAX` with a reasonable bound in `extract_canister_id`:

```rust
// rs/rosetta-api/common/rosetta_core/src/metrics.rs, line 389
const MAX_BODY_BYTES: usize = 10 * 1024 * 1024; // 10 MB
let bytes = to_bytes(body, MAX_BODY_BYTES).await?;
```

Additionally, add `tower_http::limit::RequestBodyLimitLayer` as the outermost middleware in `main.rs` to enforce the limit at the HTTP layer before any middleware processes the body.

## Proof of Concept
```python
import socket

HOST, PORT = "rosetta.example.com", 8080
s = socket.create_connection((HOST, PORT))
s.sendall(
    b"POST /block HTTP/1.1\r\n"
    b"Host: rosetta\r\n"
    b"Transfer-Encoding: chunked\r\n"
    b"Content-Type: application/json\r\n\r\n"
)
chunk = b"a" * 65536
header = f"{len(chunk):x}\r\n".encode()
while True:
    s.sendall(header + chunk + b"\r\n")
```
Assert: process RSS grows linearly with bytes sent; no HTTP 413 is returned before OOM kill.
