### Title
Controller Can Burn Arbitrary Amounts from Any Account via `icrc152_burn` - (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

### Summary

The `icrc152_burn` endpoint in the ICRC-1 ledger canister allows any canister controller to burn an arbitrary amount of tokens from any arbitrary account without the account owner's consent or approval. This is a direct analog to M-05: a privileged role can unilaterally destroy token balances belonging to third parties.

### Finding Description

The `icrc152_burn` function (part of the ICRC-152 feature) accepts a caller-supplied `from` account and `amount`, checks only that the caller is a controller via `ic_cdk::api::is_controller`, and then executes an `Operation::AuthorizedBurn` that directly debits the specified account — bypassing the normal ICRC-2 allowance mechanism entirely.

```rust
// rs/ledger_suite/icrc1/ledger/src/main.rs
fn icrc152_burn_not_async(caller: Principal, args: Icrc152BurnArgs) -> Result<u64, Icrc152BurnError> {
    let block_idx = Access::with_ledger_mut(|ledger| {
        if !ledger.feature_flags().icrc152 { ... }
        if !ic_cdk::api::is_controller(&caller) {
            return Err(Icrc152BurnError::Unauthorized(...));
        }
        // No allowance check, no owner consent — burns from args.from directly
        let tx = Transaction {
            operation: Operation::AuthorizedBurn {
                from: args.from,   // <-- arbitrary account supplied by caller
                amount,
                ...
            },
            ...
        };
        apply_transaction(ledger, tx, now, Tokens::zero())
    })
}
``` [1](#0-0) 

The `AuthorizedBurn` operation is explicitly designed to bypass allowance checks, as confirmed by the in-memory ledger consumer which processes it with `spender: &None`: [2](#0-1) 

The `FeatureFlags` struct shows `icrc152` defaults to `false`, but once a ledger deployer enables it, all controllers gain this power permanently: [3](#0-2) 

The public canister interface exposes `icrc152_burn` as an update call accepting any `from: Account`: [4](#0-3) 

### Impact Explanation

A malicious or compromised controller of an ICRC-152-enabled ledger canister can call `icrc152_burn` with any token holder's account as `from` and any amount up to that holder's balance. This allows:

- Complete destruction of any user's token balance without their knowledge or consent
- Draining of liquidity pool accounts (e.g., a DEX pool holding the token)
- Effective theft of value, since burning reduces total supply and inflates the relative value of remaining tokens held by the attacker

The `reason` field provides no on-chain enforcement — it is a free-form string with no validation beyond length. [5](#0-4) 

### Likelihood Explanation

The feature is opt-in (`icrc152: false` by default), so only ledgers that explicitly enable ICRC-152 are affected. However, once enabled, every controller of the ledger canister has this power with no further access control granularity. On the IC, a canister can have multiple controllers (including governance canisters, upgrade tooling, or developer principals), any one of which could be compromised. The attack requires no threshold corruption, no subnet majority, and no social engineering beyond compromising a single controller key.

### Recommendation

1. Remove the ability for `icrc152_burn` to target an arbitrary `from` account. Instead, require the account owner to pre-approve the burn via ICRC-2 `approve`, and have the controller call `icrc2_transfer_from` to the minting account (which triggers a burn), analogous to the M-05 fix.
2. If regulatory forced-burn semantics are genuinely required, introduce a separate, more narrowly scoped role (e.g., a dedicated compliance principal stored in ledger state) rather than reusing the broad controller set.
3. At minimum, emit a strongly-typed on-chain event that cannot be suppressed, so token holders can detect unauthorized burns.

### Proof of Concept

1. Deploy an ICRC-1 ledger with `feature_flags: Some(FeatureFlags { icrc2: true, icrc152: true })`.
2. Mint tokens to victim account `V`.
3. As a controller of the ledger canister, call:
   ```
   icrc152_burn(Icrc152BurnArgs {
       from: V,
       amount: <V's entire balance>,
       created_at_time: <current_time>,
       reason: Some("compliance"),
   })
   ```
4. Observe that `V`'s balance is now zero, with no approval from `V` ever having been granted.

This is confirmed by the integration test `test_icrc152_mint_and_burn` which demonstrates a controller burning from an arbitrary account `p1` without any prior approval from `p1`: [6](#0-5)
