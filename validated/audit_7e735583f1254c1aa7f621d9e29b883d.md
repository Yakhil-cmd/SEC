Audit Report

## Title
ckETH Withdrawal Permanently Stuck With No Reimbursement When Gas Fee Exceeds Withdrawal Amount — (`rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary
When a ckETH withdrawal request is accepted and the user's ckETH is burned, a subsequent gas fee spike can cause `create_transaction` to return `InsufficientTransactionFee`. The `create_transactions_batch` handler responds by rescheduling the request indefinitely with no reimbursement path ever triggered. The user's ckETH is permanently burned with no ETH sent and no refund issued, violating ckETH ledger conservation.

## Finding Description
The burn occurs immediately in `withdraw_eth` (`rs/ethereum/cketh/minter/src/main.rs`, L300–312) before any gas fee check. The static minimum withdrawal guard at L291–296 only compares against `cketh_minimum_withdrawal_amount`; it does not compare against the dynamic Ethereum gas fee estimate.

Later, `create_transactions_batch` (`rs/ethereum/cketh/minter/src/withdraw.rs`, L249–293) calls `create_transaction`. Inside `create_transaction` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, L1125–1133), if `request.withdrawal_amount.checked_sub(max_transaction_fee)` returns `None`, `CreateTransactionError::InsufficientTransactionFee` is returned.

The error arm in `create_transactions_batch` (L281–291) only calls `reschedule_withdrawal_request`; it never adds the withdrawal ID to `maybe_reimburse`. The `maybe_reimburse.insert(withdrawal_id)` call at `mod.rs` L552 is exclusively inside `record_created_transaction`, which is only reached on the `Ok` path. No reschedule counter, staleness deadline, or timeout mechanism exists (confirmed by absence of any such field in the codebase). The request therefore cycles in the queue on every timer tick until gas fees drop, which may never happen.

The `panic!` in the `AmountTooLow` ledger-burn error handler (`rs/ethereum/cketh/minter/src/endpoints.rs`, L241–243) confirms the developers assumed the withdrawal amount always exceeds the ledger fee — a correct assumption for the ICRC ledger fee but not for the Ethereum network gas fee, which is a completely separate and volatile quantity.

## Impact Explanation
This is a concrete permanent loss of user funds in an in-scope ck-token / Chain Fusion component. Any user whose withdrawal amount falls between `cketh_minimum_withdrawal_amount` and the current `max_transaction_fee` during a gas spike loses their ckETH with no ETH received and no reimbursement. The ckETH supply decreases without a corresponding ETH transfer, breaking the 1:1 reserve invariant. This matches the allowed High impact: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation
The attack requires no privileges — any user calling `withdraw_eth` is exposed. Ethereum gas fees are publicly observable and historically spike 10–100× within minutes. The recent reduction of the minimum withdrawal amount from 0.03 ETH to 0.005 ETH (proposal 139665, confirmed in `rs/ethereum/cketh/mainnet/minter_upgrade_2026_05_29.md`, L21–24) significantly narrows the safety margin, making the scenario where `max_transaction_fee > withdrawal_amount` far more reachable than before. No oracle manipulation, social engineering, or privileged access is required.

## Recommendation
In the `InsufficientTransactionFee` error arm of `create_transactions_batch`, add a reschedule counter to `EthWithdrawalRequest`. Once the counter exceeds a configurable threshold (or a wall-clock deadline is exceeded), instead of calling `reschedule_withdrawal_request`, add the withdrawal ID to `maybe_reimburse` and create a `ReimbursementRequest` (deducting a small penalty fee), mirroring the pattern already used for failed ERC-20 withdrawals. Alternatively, record a creation timestamp on each `EthWithdrawalRequest` and reimburse any request that has been pending longer than a configurable staleness window.

## Proof of Concept
1. Ethereum gas fees are low; user calls `withdraw_eth` with `amount = 5_000_000_000_000_000 wei` (the new minimum after proposal 139665).
2. ckETH is burned; `EthWithdrawalRequest` is queued (`rs/ethereum/cketh/minter/src/main.rs`, L314–330).
3. Ethereum gas spikes: `max_transaction_fee = 21_000 * 300_000_000_000 = 6_300_000_000_000_000 wei > 5_000_000_000_000_000 wei`.
4. `create_transactions_batch` calls `create_transaction`; `checked_sub` returns `None`; `InsufficientTransactionFee` is returned (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, L1125–1133).
5. `reschedule_withdrawal_request` is called; `maybe_reimburse` is never populated (`rs/ethereum/cketh/minter/src/withdraw.rs`, L281–291).
6. Steps 4–5 repeat on every timer tick indefinitely. User's ckETH is gone; no ETH is ever sent; no refund is ever issued.

A deterministic integration test can reproduce this by: (a) initializing the minter with `cketh_minimum_withdrawal_amount = 5_000_000_000_000_000`, (b) submitting a withdrawal at that amount, (c) injecting a `GasFeeEstimate` where `gas_limit * max_fee_per_gas > 5_000_000_000_000_000`, (d) advancing the timer, and (e) asserting that the withdrawal request remains in the pending queue indefinitely with no reimbursement entry created.
