### Title
Unprotected Pre-EIP-155 Legacy Transactions Accepted Without Chain ID Binding, Enabling Cross-Chain Replay — (`basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`) whose signing hash is computed over only `[nonce, gasPrice, gasLimit, to, value, data]` — with no chain ID committed. A transaction signed on Ethereum mainnet (or any other EVM-compatible chain) without EIP-155 protection is cryptographically valid verbatim on ZKsync OS, enabling cross-chain replay attacks that can drain user funds or execute unintended contract calls.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155-protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: chain_id is included in the hash
    ...
}
``` [1](#0-0) 

`is_eip155()` returns `false` when `v == 27 || v == 28`: [2](#0-1) 

When the unprotected branch is taken, `inner_slice` contains only the 6-field RLP payload `[nonce, gasPrice, gasLimit, to, value, data]`. No chain ID is mixed into the hash. The EIP-155 branch, by contrast, appends `chain_id` and two empty bytes to the preimage: [3](#0-2) 

The parsed transaction is stored as the `RlpEncodedTxInner::Legacy` variant, and `chain_id()` explicitly returns `None` for it, confirming no chain binding: [4](#0-3) 

The outer dispatcher in `parse_and_compute_signed_hash` accepts this variant without any chain ID check — unlike all typed transactions (EIP-2930, EIP-1559, EIP-4844, EIP-7702) which explicitly validate `tx.chain_id != expected_chain_id`: [5](#0-4) 

The signature is then verified in `validate_and_compute_fee_for_transaction` using `ecrecover` against this chain-ID-free hash: [6](#0-5) 

---

### Impact Explanation

An attacker who observes any pre-EIP-155 transaction broadcast on Ethereum mainnet (or any other EVM chain) can replay it on ZKsync OS without modification, provided:

1. The sender's nonce on ZKsync OS equals the nonce in the replayed transaction.
2. The sender has sufficient balance on ZKsync OS.

Because Ethereum addresses are derived from the same private key across all EVM chains, condition (1) is trivially satisfied for new accounts (nonce = 0) and is achievable in general whenever the ZKsync OS nonce happens to match. The result is an unauthorized fund transfer or contract call on ZKsync OS that the user never intended for this chain.

The reverse direction also holds: a pre-EIP-155 transaction signed for ZKsync OS can be replayed on Ethereum mainnet.

---

### Likelihood Explanation

Pre-EIP-155 transactions are rare in modern wallets but are still produced by some hardware wallets, older software, and certain institutional signing setups. The nonce constraint limits opportunistic replay but does not prevent targeted attacks, especially against new accounts (nonce = 0) or accounts whose ZKsync OS nonce is known to match a previously broadcast Ethereum transaction. The attacker entry path is fully unprivileged: any party can submit an RLP-encoded transaction to the bootloader.

---

### Recommendation

Reject pre-EIP-155 legacy transactions entirely by returning an error when `is_eip155() == false`. ZKsync OS is a new chain with its own chain ID; there is no legitimate reason to accept signatures that carry no chain binding. The change is minimal:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    return Err(InvalidTransaction::InvalidChainId.into()); // reject unprotected legacy
} else {
    // EIP-155 protected path (unchanged)
    ...
};
```

Alternatively, if pre-EIP-155 compatibility must be preserved for specific use cases, the signing hash for the unprotected branch should be augmented with the expected chain ID to prevent cross-chain replay, analogous to how EIP-155 works.

---

### Proof of Concept

1. Alice signs a pre-EIP-155 ETH transfer on Ethereum mainnet:
   - Fields: `[nonce=0, gasPrice=50gwei, gasLimit=21000, to=Bob, value=1ETH, data=]`
   - Signature: `v=27, r=R, s=S`
   - Signing hash: `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — **no chain ID**

2. Alice has the same address on ZKsync OS (same private key → same address), nonce = 0, and a balance of 1 ETH.

3. Attacker submits the identical RLP-encoded transaction bytes to ZKsync OS.

4. `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` detects `v=27`, takes the unprotected branch, computes the identical 6-field hash, and returns it as `sig_hash`.

5. `validate_and_compute_fee_for_transaction` calls `ecrecover(sig_hash, v=27, R, S)` → recovers Alice's address → nonce check passes (both = 0) → transaction executes.

6. Alice's ZKsync OS balance is drained to Bob without Alice's knowledge or consent. [7](#0-6) [8](#0-7)
