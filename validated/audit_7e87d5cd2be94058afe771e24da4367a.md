### Title
`native_per_pubdata` Truncates to Zero via Floor Division, Allowing Free Pubdata Generation - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

When the operator-set `pubdata_price` is less than `native_price`, the floor division used to compute `native_per_pubdata` produces zero. This causes the system to charge zero native resources for all pubdata generated by a transaction, allowing any user to generate unlimited pubdata without paying the corresponding native resource cost. The protocol/operator bears the L1 data availability cost that should have been charged to the user.

### Finding Description

In `validation_impl.rs` line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs floor (truncating) integer division. When `pubdata_price < native_price` — a realistic condition, since these are independently set values in different units — the result is `native_per_pubdata = 0`.

This zero value propagates through the entire resource accounting pipeline:

**1. Intrinsic pubdata overhead becomes zero** in `create_resources_for_tx`:
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
```
The user's native limit is not reduced by intrinsic pubdata at all.

**2. Runtime pubdata charge becomes zero** in `get_resources_to_charge_for_pubdata`:
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
// = current_pubdata_spent * 0 = 0
```
Any amount of pubdata generated during execution costs zero native resources.

**3. The pubdata sufficiency check always passes** in `check_enough_resources_for_pubdata`:
```rust
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata = 0 native → always true
```

The same truncation bug is also present in the public API helper `validate_l2_tx_intrinsic_native_resources` in `api/src/helpers.rs` line 427:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

This mirrors the bootloader behavior, confirming the bug is systemic.

### Impact Explanation

When `pubdata_price < native_price`, any user can submit a transaction that generates large amounts of pubdata (e.g., many `SSTORE` operations writing to new storage slots) without consuming any native resources for it. The user pays only the EVM gas fee, while the operator must absorb the full L1 data availability cost for the generated pubdata. This is a direct financial loss to the operator/protocol proportional to the amount of pubdata generated. A single transaction writing to many new storage slots (each costing 32+ bytes of pubdata) can generate kilobytes of pubdata at zero native cost.

### Likelihood Explanation

The condition `pubdata_price < native_price` is operator-controlled but observable on-chain. `native_price` reflects the cost per RISC-V proving cycle, while `pubdata_price` reflects the cost per pubdata byte — these are independent values in different units. It is entirely plausible in normal operation that `pubdata_price < native_price` (e.g., `native_price = 1000`, `pubdata_price = 500` → `native_per_pubdata = 0`). An attacker can monitor these values and exploit the window whenever the condition holds. No privileged access is required; any unprivileged transaction sender can trigger this.

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Current (floor division — truncates to zero when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// Fixed (ceiling division — ensures at least 1 native unit charged per pubdata byte
// when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This is consistent with the existing pattern for `native_per_gas`:
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

Additionally, add a guard that rejects transactions when `native_per_pubdata == 0` and `pubdata_price > 0`, to prevent silent under-charging. Apply the same fix to `api/src/helpers.rs` line 427.

### Proof of Concept

**Attacker-controlled entry path:**

1. Observe that `pubdata_price = 50` and `native_price = 100` are set in the block context (both operator-set, publicly observable).
2. `native_per_pubdata = 50 / 100 = 0` (floor division).
3. Submit an L2 transaction that writes to 1000 new storage slots (generating ~32,000 bytes of pubdata).
4. During validation, `intrinsic_pubdata_overhead = 0 * 134 = 0` — native limit is not reduced.
5. During execution, `get_resources_to_charge_for_pubdata` returns 0 native for 32,000 bytes of pubdata.
6. `check_enough_resources_for_pubdata` returns `has_enough = true` trivially.
7. Transaction succeeds. User paid only EVM gas. Operator must pay L1 DA cost for 32,000 bytes.
