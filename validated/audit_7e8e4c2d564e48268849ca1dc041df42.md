### Title
Integer Division Truncation in `native_per_pubdata` Calculation Allows Free Pubdata Consumption — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary
When the operator-set `pubdata_price` is less than `native_price`, the integer division `pubdata_price.wrapping_div(native_price)` truncates to zero. This makes `native_per_pubdata = 0`, causing all pubdata consumption to be completely free in native resource terms. Any unprivileged transaction sender can write unlimited storage without paying the native cost for pubdata, while the operator bears the full proving and L1-publishing cost.

### Finding Description

In `validate_and_compute_fee_for_transaction`, the per-pubdata-byte native cost is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This is a floor (truncating) division. When `pubdata_price < native_price`, the result is `0`. The same truncating pattern appears in the public API helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

This is inconsistent with how `native_per_gas` is computed — which correctly uses ceiling division to avoid truncation to zero:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [3](#0-2) 

When `native_per_pubdata = 0`, the downstream function `get_resources_to_charge_for_pubdata` computes:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // = current_pubdata_spent * 0 = 0
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

So **zero native resources are charged for any amount of pubdata consumed**, regardless of how many storage slots the transaction writes.

The `delta_gas` adjustment in `compute_gas_refund` also cannot compensate for this, because `native_used` itself will not include pubdata costs when `native_per_pubdata = 0`:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [5](#0-4) 

The documentation explicitly states that `deltaGas := (nativeUsed / nativePerGas) - gasUsed` is the mechanism to ensure native costs are reflected in gas used — but this only works if `native_used` correctly includes pubdata costs, which it does not when `native_per_pubdata = 0`. [6](#0-5) 

### Impact Explanation

When `pubdata_price < native_price`, pubdata consumption is entirely free in native resource terms. A user can write as many storage slots as the pubdata limit allows without consuming any native resources for pubdata. The operator/sequencer must still prove and publish this pubdata to L1, bearing the full cost without receiving native resource compensation. This breaks the resource accounting invariant that native resources model the off-chain proving cost.

### Likelihood Explanation

`pubdata_price` and `native_price` are independent operator-set block parameters. The condition `pubdata_price < native_price` is a realistic and common configuration — for example, if the operator sets `native_price = 1000` (reflecting expensive proving) and `pubdata_price = 500` (reflecting cheaper L1 data availability). Any block where this ratio holds triggers the bug for every transaction in that block.

### Recommendation

Use ceiling division for `native_per_pubdata`, consistent with `native_per_gas`:

```diff
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
      .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427.

### Proof of Concept

1. Operator sets block context: `native_price = 1000`, `pubdata_price = 999`, `eip1559_basefee = 1000`.
2. `native_per_pubdata = 999 / 1000 = 0` (integer truncation).
3. Attacker submits a transaction that writes 1,000 storage slots (generating ~32,000 bytes of pubdata).
4. `get_resources_to_charge_for_pubdata` charges `32000 * 0 = 0` native resources for pubdata.
5. The transaction's native resource budget is consumed only by computational work, not pubdata.
6. The transaction succeeds; the operator must publish 32,000 bytes to L1 with zero native resource compensation.
7. Compared to the intended behavior (`native_per_pubdata = 1`), the attacker avoids paying `32,000` native resource units — the full pubdata cost.
