### Title
Unprotected `new` Initialization Allows Front-Running to Seize Engine Ownership — (File: `engine/src/contract_methods/admin.rs`)

### Summary
The `new` function that initializes the Aurora Engine state has no caller restriction. Any NEAR account can call it before the legitimate deployer, supplying an attacker-controlled `owner_id` and permanently seizing administrative ownership of the engine. The victim's subsequent `new` call is rejected with `ERR_ALREADY_INITIALIZED`, with no recovery path.

### Finding Description
`admin::new` is the sole initialization entry point for the Aurora Engine contract. It writes the `EngineState` — including `owner_id` — from caller-supplied `NewCallArgs`. The only guard is a check that the state has not already been written:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

There is no check on `env.predecessor_account_id()`. The `owner_id` is taken entirely from the attacker-supplied `args`, not from the caller identity. The public WASM export exposes this unconditionally: [2](#0-1) 

The test harness in `engine-tests-connector` demonstrates the two-step deployment pattern (deploy contract, then call `new` in a separate transaction) that creates the exploitable window: [3](#0-2) 

### Impact Explanation
Whoever calls `new` first becomes `owner_id`. The owner controls:
- `pause_contract` / `resume_contract` — permanent fund freeze for all EVM users
- `set_upgrade_delay_blocks` + contract upgrade — potential arbitrary code execution and fund theft
- `set_eth_connector_contract_account` — redirect the bridge to a malicious contract

The attacker-set owner can immediately call `pause_contract`, permanently freezing all ETH and ERC-20 balances held in the Aurora EVM. This satisfies **Critical: Permanent freezing of funds**. [4](#0-3) 

### Likelihood Explanation
The attack window exists whenever the deployer issues the `DeployContract` action and the `new` call as two separate NEAR transactions rather than a single atomic batch. The test infrastructure in this repository does exactly that (separate `.deploy()` and `.call("new")` calls). Any Silo operator or third-party deployer following the same pattern is vulnerable. A NEAR block producer or any monitoring account can observe the deploy receipt and submit a `new` call in the next block before the legitimate operator does. No special privilege is required.

### Recommendation
**Short term:** Restrict `new` to be callable only by the contract account itself (`env.predecessor_account_id() == env.current_account_id()`), or require a one-yocto deposit from the contract account as proof of self-call.

**Long term:** Enforce that deployment and initialization are always performed in a single atomic NEAR batch transaction (`DeployContract` + `FunctionCall` to `new` in the same batch action list), making the two-step window structurally impossible. Document this requirement for all Silo deployments.

### Proof of Concept
1. Attacker monitors the NEAR chain for a `DeployContract` action targeting a new Aurora Engine account.
2. In the next block, attacker submits a call to `new` on that account with `owner_id = attacker.near` and any valid `chain_id`.
3. Attacker's `new` call succeeds; `EngineState.owner_id` is set to `attacker.near`.
4. Legitimate deployer's subsequent `new` call returns `ERR_ALREADY_INITIALIZED` and is rejected.
5. Attacker calls `pause_contract` as the owner, freezing all user funds in the EVM permanently. [1](#0-0) [2](#0-1)

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}
