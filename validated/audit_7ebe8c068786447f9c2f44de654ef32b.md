### Title
Unbounded `move_to_block()` Loop in Hashchain Subsystem Causes NEAR Gas Exhaustion, Temporarily Freezing All Aurora Engine Transactions - (File: engine-hashchain/src/hashchain.rs)

### Summary

The `move_to_block()` function in the Aurora Engine hashchain subsystem contains an unbounded `while` loop that iterates once per NEAR block between the stored hashchain height and the current NEAR block height. This function is invoked on every hashchain-enabled transaction (including `submit`, the primary EVM transaction entry point) via `load_hashchain()`. When the gap between the stored hashchain height and the current block height grows large enough — approximately 90–110 blocks (~1.5–2 minutes of inactivity) — the cumulative NEAR gas cost of the loop exceeds the 300 Tgas per-transaction limit. Every subsequent transaction then fails before executing, permanently blocking all user activity until the key manager manually resets the hashchain via `start_hashchain`.

### Finding Description

`move_to_block()` in `engine-hashchain/src/hashchain.rs` iterates over every intermediate block height, calling `compute_block_hashchain()` (which invokes `aurora_engine_sdk::keccak`, a NEAR host function) once per block:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
``` [1](#0-0) 

`compute_block_hashchain()` concatenates `chain_id` (32 bytes) + `contract_account_id` (~10 bytes) + `current_block_height` (8 bytes) + `previous_block_hashchain` (32 bytes) + `txs_hash` (32 bytes) + `txs_logs_bloom` (256 bytes) = ~370 bytes, then calls `keccak()`: [2](#0-1) 

The NEAR host function `keccak256` costs approximately `keccak256_base` (~1.77 Tgas) + `keccak256_byte` × 370 (~0.49 Tgas) ≈ **2.26 Tgas per iteration**. With the 300 Tgas transaction limit and ~50–100 Tgas consumed by the actual transaction logic, only ~88–110 iterations are possible before gas exhaustion.

This loop is triggered on every transaction through `load_hashchain()`, called unconditionally by both `with_hashchain()` and `with_logs_hashchain()`:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
``` [3](#0-2) 

`with_hashchain` and `with_logs_hashchain` wrap all critical contract methods including `submit` (EVM transaction submission), `call`, `deploy`, `ft_on_transfer`, and others: [4](#0-3) 

The hashchain's `current_block_height` is only advanced when a transaction is successfully processed. If no transactions are submitted for ~90+ NEAR blocks (~1.5 minutes), the next transaction will call `move_to_block()` with a gap of 90+ iterations, consuming ~203+ Tgas in the loop alone — exhausting the gas budget before the actual EVM logic executes. Because the transaction fails and is rolled back, the hashchain state is **never updated**, so every subsequent transaction faces the same (or larger) gap and also fails. The engine enters a state where no user transaction can succeed.

### Impact Explanation

**High. Temporary freezing of funds.**

All hashchain-enabled entry points — including `submit` (the primary EVM transaction path), `call`, `deploy`, and token bridge operations — become permanently unreachable. Users cannot transfer, withdraw, or interact with any EVM contract on Aurora. Funds are frozen until the key manager manually intervenes by pausing the contract and calling `start_hashchain` with `args.block_height = current_block_height - 1` (which avoids the gap). This is a temporary but complete freeze of all user funds in the Aurora Engine.

### Likelihood Explanation

**High.**

NEAR produces approximately one block per second. A gap of only ~90–110 blocks (~1.5–2 minutes of zero transaction activity) is sufficient to trigger the failure. This is a realistic condition during:
- Off-peak hours with low traffic
- Any planned or unplanned maintenance pause (the contract must be paused before `start_hashchain`, during which no transactions are processed, guaranteeing a gap equal to the pause duration)
- Network congestion or relayer downtime

No attacker action is required. The vulnerability is triggered automatically by the first legitimate user transaction submitted after any sufficiently long quiet period. The longer the gap, the larger the gas cost, and the more certain the failure.

### Recommendation

Cap the number of iterations in `move_to_block()` to a safe maximum per transaction (e.g., 50 blocks). If the gap exceeds the cap, either:
1. Return an error requiring the key manager to call `start_hashchain` to re-anchor the hashchain at a recent block height, or
2. Process the gap incrementally across multiple transactions by storing a "catch-up" pointer in contract state.

Additionally, `start_hashchain` should validate that `args.block_height` is close to the current block height to prevent the same OOG issue in that path.

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled (via `NewCallArgsV4` with `initial_hashchain = Some([0u8; 32])`).
2. Submit one transaction to initialize the hashchain at block height `H`.
3. Wait for NEAR to produce 110+ blocks without submitting any Aurora transactions (gap = 110 blocks).
4. Submit any EVM transaction via `submit`. The call enters `with_logs_hashchain` → `load_hashchain` → `move_to_block(H + 110)`.
5. `move_to_block` iterates 110 times, each calling `keccak` on ~370 bytes (~2.26 Tgas each) = ~249 Tgas consumed in the loop alone, plus transaction overhead, exceeding the 300 Tgas limit.
6. The transaction fails with gas exhaustion. The hashchain state is rolled back to height `H`.
7. Every subsequent transaction faces the same (now larger) gap and also fails.
8. The engine is frozen until the key manager calls `pause_contract` then `start_hashchain` with `block_height = current_block_height - 1`. [5](#0-4) [3](#0-2) [6](#0-5)

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }
