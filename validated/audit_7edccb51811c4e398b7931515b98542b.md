Now I have all the information needed for a complete analysis. Let me trace every step of the claimed attack.

**Step 1 — 15-byte path: does `insert_err` fire?**

`decoded_wasm_size` → `wasm_encoding_and_size`: a 15-byte buffer starting with `\x1f\x8b\x08` hits the `module_bytes.len() < 16` guard and returns `Err(DecodingError("gzip stream is too short"))`. [1](#0-0) 

Back in `Hypervisor::create_execution_state`, `wasm_size_result` is `Err`, so the code immediately calls `compilation_cache.insert_err` and returns — before the wasm executor is ever invoked. [2](#0-1) 

**Step 2 — 16-byte malformed-gzip path: does `insert_err` fire?**

For a 16-byte buffer, `wasm_encoding_and_size` returns `Ok` (reads the last 4 bytes as the uncompressed size). `wasm_size_result` is `Ok`, so the early-return branch is skipped. The code falls through to `self.wasm_executor.create_execution_state`. [3](#0-2) 

In the **sandboxed** production path, the compiler receives the raw bytes and calls `decode_wasm` internally. If `libflate::gzip::Decoder::new` fails, the compiler returns `Err`, and `compilation_cache.insert_err` is called. [4](#0-3) 

In the **non-sandboxed** path (`WasmExecutorImpl`), `decode_wasm` is called with `?` and the error propagates without ever touching `insert_err`. [5](#0-4) 

**Step 3 — Cache structure and eviction**

`CompilationCache` is an LRU map keyed by `WasmHash`. `DEFAULT_MAX_ENTRIES = 500_000`. Both `insert_err` and `insert_ok` call `cache.pop_lru()` when the limit is reached, so error entries directly compete with valid compiled-module entries for slots. [6](#0-5) [7](#0-6) 

**Step 4 — The "alternating two inputs" framing is wrong**

The cache key is the hash of the module bytes. Submitting the same two inputs repeatedly just overwrites the same two slots — it cannot fill the cache. The actual attack requires **500,000 distinct** malformed modules. The question's own proof idea ("submit 10,000 distinct 16-byte malformed gzip inputs") is the correct framing.

**Step 5 — Rate limiting**

`install_code_rate_limit` is per-canister (`install_code_debit` accumulates per canister and is decremented by `install_code_rate_limit` each round). [8](#0-7) [9](#0-8) 

For a 15-byte module the cost is `create_execution_state_base_cost + cost_to_compile_wasm_instruction × 15`. The base cost dominates. As long as this is ≤ `install_code_rate_limit`, one call per round per canister is allowed. The attacker can use N canisters to multiply throughput — creating 1,000 canisters costs ~100T cycles, enabling cache saturation in ~500 rounds (~8 minutes at 1 round/s). [10](#0-9) 

**Step 6 — Impact**

Once the 500,000-entry LRU is filled with error entries, every `compilation_cache.get` for a legitimate canister returns `None`, forcing a full recompile on every `install_code` or first execution. Recompilation is expensive; sustained cache poisoning degrades subnet throughput measurably. [11](#0-10)

### Title
Compilation Cache Poisoning via Distinct Malformed Gzip Modules in `install_code` — (`rs/execution_environment/src/hypervisor.rs`, `rs/embedders/src/compilation_cache.rs`)

### Summary
An unprivileged canister developer can submit `install_code` messages carrying distinct malformed gzip-prefixed Wasm modules. Each submission inserts an error entry into the subnet-wide `CompilationCache` (keyed by module hash). With enough distinct inputs the 500,000-entry LRU cache fills with error entries, evicting valid compiled modules for legitimate canisters and forcing them to recompile on every execution, degrading subnet throughput.

### Finding Description
`Hypervisor::create_execution_state` calls `decoded_wasm_size` before invoking the wasm executor. When the result is `Err` (e.g., a ≤15-byte gzip-prefixed buffer), it immediately calls `self.compilation_cache.insert_err(&canister_module, …)` and returns. [2](#0-1) 

`insert_err` unconditionally pushes an `Err` entry into the shared LRU cache, evicting the least-recently-used entry (which may be a valid compiled module for another canister) when `cache.len() >= max_entries`. [7](#0-6) 

The cache has `DEFAULT_MAX_ENTRIES = 500_000`. Each distinct malformed module (different bytes → different hash) occupies one slot. An attacker controlling N canisters can submit one `install_code` per round per canister, generating N distinct error entries per round. [6](#0-5) 

### Impact Explanation
Once the cache is saturated with error entries, every `compilation_cache.get` for a legitimate canister returns `None` (cache miss), triggering a full recompile. Recompilation is the most expensive part of `install_code` and first-execution setup. Sustained poisoning forces continuous recompilation for all canisters on the subnet, measurably degrading throughput. Correctness is not violated, but availability/performance is.

### Likelihood Explanation
The attack requires only: (a) the ability to create canisters and call `install_code` (unprivileged, open to any principal with cycles), and (b) enough cycles to create ~1,000 canisters (~100T cycles) and pay per-call execution costs. The per-call cost for a 15-byte module is minimal. The rate limit is per-canister, so it is trivially bypassed by spreading calls across many canisters. The attack is locally testable and repeatable.

### Recommendation
1. **Do not cache decoding errors.** Decoding failures (`WasmValidationError::DecodingError`) are cheap to reproduce and carry no compilation work; they should not occupy cache slots. Remove the `insert_err` call in the `wasm_size_result` error branch in `Hypervisor::create_execution_state`.
2. **Separate error entries from compiled-module entries** in the LRU, or cap error entries at a small fixed fraction of `max_entries`.
3. **Charge cycles proportional to actual work done** for malformed-module submissions to raise the economic cost of cache flooding.
