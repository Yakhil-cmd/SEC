Audit Report

## Title
Permanent Slot Exhaustion via Unauthenticated `canister_status` on Non-Controlled Canister — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary

The public `canister_status` endpoint on NNS Root has no caller authentication and calls `.unwrap()` on the management canister response. When the management canister rejects (e.g., Root is not a controller of the target), `.unwrap()` causes a Wasm trap in the callback execution context. The `SlotLoan` RAII guard does execute its `drop()` — restoring the slot count — but that restoration is a state mutation made during the callback, which the IC rolls back when the trap occurs. The pre-await slot decrement was already committed and survives. Repeating this 167 times permanently exhausts all slots, blocking every management-canister operation routed through NNS Root, including those from NNS Governance.

## Finding Description

**Entry point:** `canister_status` at `rs/nns/handlers/root/impl/canister/canister.rs` lines 88–98 is a `#[update]` endpoint with no authentication check. [1](#0-0) 

**Slot accounting:** `new_management_canister_client()` wraps the real client in `LimitedOutstandingCallsManagementCanisterClient`, which calls `try_borrow_slot()` before forwarding the call. `try_borrow_slot()` decrements `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` (initialized to 167) and returns a `SlotLoan` RAII guard. [2](#0-1) [3](#0-2) 

**RAII guard:** `SlotLoan::drop()` increments the count back when the guard is dropped. [4](#0-3) 

**The bug — IC execution phase boundary:** In `LimitedOutstandingCallsManagementCanisterClient::canister_status`, `_loan` is created and the slot is decremented *before* the `.await`. The IC commits all state at the await point when the outgoing inter-canister call is enqueued. The slot decrement is now durable. [5](#0-4) 

When the management canister rejects, the callback resumes. `_loan` goes out of scope and `SlotLoan::drop()` *does* execute, writing the restored count back to the `thread_local`. However, this write is a state mutation performed *during the callback execution context*. Control then returns to the outer `canister_status` function, which calls `.unwrap()` on the `Err`, causing a Wasm trap. [6](#0-5) 

Per the IC execution model, a Wasm trap in a callback causes the IC to roll back all state changes made during that callback. The slot restoration written by `SlotLoan::drop()` is rolled back. The pre-await decrement, already committed, is not. The net effect is a permanent decrement of one slot per call.

**VIP callers are not exempt:** When `available_slot_count == 0`, `try_borrow_slot()` returns `Err` for all callers unconditionally. The VIP path only sets `used_slot_count = 0`, but the zero-count guard fires before that subtraction. [7](#0-6) 

**Acknowledged design:** The codebase explicitly documents that NNS Root's `canister_status` uses trapping as its error-propagation mechanism, confirming this is the production behavior. [8](#0-7) 

## Impact Explanation

Once all 167 slots are exhausted, every call to `try_borrow_slot()` returns `Err((SysTransient, "Unavailable. Maybe, try again later?"))`. This blocks all management-canister operations routed through NNS Root: canister upgrades, settings changes, stop/start, snapshot operations, and controller changes. NNS Governance, which relies on NNS Root for these operations, is fully blocked from executing governance proposals that touch canister management. The slot count cannot self-recover because the only recovery path (`SlotLoan::drop()`) is always rolled back when the count is 0 (all calls trap). Recovery requires an NNS Root canister upgrade to reset the `thread_local` — an operation that itself requires the NNS upgrade machinery to be functional.

This matches the allowed High impact: **Application/platform-level DoS with significant NNS security impact and concrete protocol harm**.

## Likelihood Explanation

- No authentication required; any principal including anonymous can call `canister_status`
- No special cycles cost beyond normal update-call fees
- 167 sequential calls suffice; no concurrency or race conditions required
- The management canister deterministically rejects `canister_status` for any canister Root does not control
- The trap is deterministic and reproducible

## Recommendation

Change `canister_status` to return `Result<CanisterStatusResult, String>` and propagate errors without trapping:

```rust
#[update]
async fn canister_status(canister_id_record: CanisterIdRecord) -> Result<CanisterStatusResult, String> {
    let client = new_management_canister_client();
    client
        .canister_status(canister_id_record)
        .await
        .map(CanisterStatusResult::from)
        .map_err(|(_, msg)| msg)
}
```

This ensures `SlotLoan` is dropped normally (restoring the slot count) in all cases, because the function returns without trapping. The slot restoration write occurs in the callback context and is committed rather than rolled back.

## Proof of Concept

State-machine or PocketIC integration test:

1. Deploy NNS with the Root canister.
2. Choose any canister ID not controlled by Root (e.g., `CanisterId::from_u64(9999)`).
3. From the anonymous principal, call `canister_status` on Root 167 times with that canister ID. Each call returns a replica-level error (trap).
4. Verify `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT == 0` via the `/metrics` HTTP endpoint or by observing that a subsequent call from any principal (including an NNS canister) returns `SysTransient / "Unavailable. Maybe, try again later?"`.
5. Have NNS Governance attempt any management-canister operation through Root (e.g., `change_nns_canister`). Assert it fails with the slot-exhaustion error, confirming governance is blocked.

        let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

            .with_borrow_mut(|available_slot_count| {

/// Increments available_slot_count by used_slot_count when dropped.
struct SlotLoan {
    available_slot_count: &'static LocalKey<RefCell<u64>>,
    used_slot_count: u64,
}

impl Drop for SlotLoan {
    fn drop(&mut self) {
        self.available_slot_count
            .with_borrow_mut(|available_slot_count| {
                *available_slot_count = available_slot_count.saturating_add(self.used_slot_count);
            });
    }
}
