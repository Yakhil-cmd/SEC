### Title
Permanent Fund Freeze When `ft_transfer` Fails in `ExitToNear` Precompile Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens or ETH inside the EVM and then schedules an `ft_transfer` (or `near_withdraw`) promise on the NEP-141 contract. When the `error_refund` compile-time feature is **not** enabled, the `refund` field in the callback args is hardcoded to `None`. For the common ERC-20 and ETH exit paths this causes the promise to be scheduled with **no callback at all**. If the `ft_transfer` promise fails, the burned assets are permanently unrecoverable — an exact structural analog to the reported `sendNFTs()` silent-catch pattern.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` function constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the two most common exit paths — **ETH base-token exit** and **ERC-20 token exit** — `transfer_near_args` is also `None` (returned by `exit_base_token_to_near` and the non-wNEAR branch of `exit_erc20_token_to_near`). [2](#0-1) [3](#0-2) 

When both fields are `None`, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the branch that attaches a callback is skipped entirely:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← NO callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [4](#0-3) 

The result is a bare `ft_transfer` promise with no `then`-callback. If that promise fails (e.g., recipient account not registered with the NEP-141 contract, insufficient gas forwarded, or any other NEP-141 rejection), the NEAR runtime simply discards the failure — there is no `exit_to_near_precompile_callback` invocation, no `refund_on_error` call, and no re-mint of the already-burned ERC-20 tokens or ETH.

The `exit_to_near_precompile_callback` handler that *would* perform the refund is:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [5](#0-4) 

But this handler is never scheduled when `error_refund` is disabled and `transfer_near` is `None`.

The wNEAR-unwrap path does attach a callback (because `transfer_near_args` is `Some`), but the callback's failure branch still silently returns `None` when `args.refund` is `None`, so wNEAR burns are equally unrecoverable on `near_withdraw` failure.

The own test suite explicitly acknowledges the loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `withdrawToNear` on a bridged ERC-20 (or uses the ETH exit precompile) when the `error_refund` feature is absent from the production binary will have their tokens burned in the EVM with no possibility of recovery if the downstream `ft_transfer` promise fails. The NEP-141 balance held by the Aurora contract is unaffected (the transfer simply did not happen), but the EVM-side balance is gone. The discrepancy is permanent and irrecoverable without an admin intervention that the protocol does not provide.

---

### Likelihood Explanation

**Medium.**

`ft_transfer` on a NEP-141 contract fails in several realistic, user-reachable scenarios:

- The recipient NEAR account is not registered with the NEP-141 contract (storage deposit missing) — the test itself uses this exact trigger.
- The Aurora engine account's NEP-141 balance is insufficient (e.g., drained by a concurrent operation).
- The NEP-141 contract is paused or has custom transfer restrictions.
- Gas exhaustion in the promise chain.

None of these require any privileged access; an ordinary EVM user can trigger the failure simply by specifying an unregistered recipient.

---

### Recommendation

1. **Make `error_refund` a default (or unconditional) feature** in the production engine build. The refund logic in `refund_on_error` and `exit_to_near_precompile_callback` already exists and is correct; it simply must not be gated behind an opt-in flag.
2. Alternatively, remove the `#[cfg(not(feature = "error_refund"))] refund: None` branch entirely and always populate `refund` with the appropriate `RefundCallArgs`, ensuring a callback is always attached for every exit path.
3. Add a CI check that the production WASM artifact is always compiled with `error_refund` enabled.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the `#[cfg(not(feature = "error_refund"))]` branch is active).
2. Bridge a NEP-141 token to Aurora; user holds ERC-20 balance.
3. User calls `withdrawToNear("unregistered.near", amount)` on the ERC-20 contract from their EVM address.
4. The ERC-20 contract calls the `ExitToNear` precompile; the precompile burns the ERC-20 tokens in the EVM and emits a `PromiseArgs::Create(ft_transfer)` log — **no callback**.
5. The engine schedules the bare `ft_transfer` promise to the NEP-141 contract targeting `unregistered.near`.
6. The NEP-141 contract rejects the transfer (account not registered).
7. NEAR runtime discards the failure silently; no callback fires.
8. **Result**: ERC-20 tokens are gone from the EVM; NEP-141 balance of Aurora is unchanged; user has lost their funds permanently with no recovery path.
