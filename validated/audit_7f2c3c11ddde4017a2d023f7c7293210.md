### Title
Unchecked Precompile Return Value in `withdrawToNear` Allows Token Burn Without NEAR Credit - (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`withdrawToNear` in `EvmErc20V2.sol` burns the caller's tokens before invoking the `ExitToNear` precompile, but never checks the return value of the low-level `call`. When an invalid NEAR account ID is supplied as `recipient`, the precompile returns an `ExitError` (EVM `call` returns `0`), the burn is not reverted, and no NEAR-side promise is scheduled. The user's tokens are permanently destroyed with no corresponding credit.

---

### Finding Description

**Step 1 — Burn before precompile call, no return-value check.**

`withdrawToNear` burns tokens unconditionally on line 55, then calls the precompile on line 62. The assembly block captures `res` but never acts on it:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert on failure
    }
}
``` [1](#0-0) 

**Step 2 — Invalid NEAR account ID causes `ExitError` inside the precompile.**

`parse_recipient` in the Rust precompile validates the recipient bytes as a NEAR `AccountId`. If `str::from_utf8` succeeds but `AccountId::parse` fails (e.g., for `test@.near`, `UPPERCASE`, `a..b`), it returns `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`:

```rust
fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    ...
    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        ...
    })
}
``` [2](#0-1) 

This `ExitError` propagates through `ExitToNearParams::try_from` (line 419: `ExitToNearParams::try_from(input)?`) back to `ExitToNear::run`, which returns `Err(ExitError)`. [3](#0-2) 

The EVM translates a precompile `Err(ExitError)` into a `call` return value of `0`. Because `res` is never checked in the Solidity assembly block, the outer `withdrawToNear` transaction succeeds.

**Step 3 — No promise is ever scheduled.**

A successful precompile run emits a `promise_log` containing the serialized `PromiseArgs` that the Aurora engine later executes to credit the NEAR recipient. When the precompile fails, this log is never produced: [4](#0-3) 

The `error_refund` callback path (`exit_to_near_precompile_callback`) only fires when the precompile itself succeeds but the downstream NEAR call fails — it cannot recover from a precompile-level `ExitError`. [5](#0-4) 

---

### Impact Explanation

Any token holder can call `withdrawToNear` with a syntactically valid UTF-8 string that is an invalid NEAR account ID. Their ERC-20 tokens are burned on the Aurora/EVM side, but no `ft_transfer` or `ft_transfer_call` promise is ever scheduled on NEAR. The tokens are irrecoverably destroyed with no NEAR-side credit — a permanent loss of user funds matching the **High: Temporary freezing of funds** scope (and arguably permanent destruction).

---

### Likelihood Explanation

- No privilege is required; any token holder can call `withdrawToNear`.
- The invalid-recipient condition is trivially reachable: any uppercase letter, `@`, consecutive dots, or leading/trailing hyphens in the recipient bytes suffices.
- The existing unit test already confirms the failure path: `assert!(parse_recipient(b"test@.near").is_err())`. [6](#0-5) 
- The bug is in production Solidity code with no runtime guard.

---

### Recommendation

Add a revert on precompile call failure inside the assembly block:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, restructure `withdrawToNear` to validate the recipient before burning (call the precompile first, or perform account-ID validation in Solidity), so that a failed precompile call never leaves a burned balance without a corresponding promise.

The same unchecked-`res` pattern exists in `withdrawToEthereum` (line 75) and should be fixed there as well. [7](#0-6) 

---

### Proof of Concept

1. Deploy `EvmErc20V2` on a local Aurora sandbox.
2. Mint tokens to an attacker address.
3. Call `withdrawToNear(b"INVALID@ACCOUNT", amount)` from the attacker address.
4. Observe: `_burn` succeeds, ERC-20 balance drops to zero, the outer transaction succeeds (no revert).
5. Observe: no `ExitToNear` promise log is emitted, no NEAR-side `ft_transfer` is scheduled, and the attacker's NEAR account receives nothing.
6. The tokens are gone with no recovery path.

The unit test `test_parse_invalid_recipient` already proves step 4 at the precompile level:
```rust
assert!(parse_recipient(b"test@.near").is_err());   // precompile returns ExitError
assert!(parse_recipient(b"test@.near:msg").is_err());
``` [6](#0-5)
