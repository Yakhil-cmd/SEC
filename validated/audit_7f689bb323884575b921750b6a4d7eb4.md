### Title
Unauthenticated `new` Initializer Allows Any Caller to Seize Engine Ownership and Drain All Funds - (File: engine/src/contract_methods/admin.rs)

---

### Summary
The `new` function that initializes the Aurora Engine state performs no check on who the caller is. Any NEAR account that calls `new` before the legitimate deployer can supply an arbitrary `owner_id`, seizing full administrative control of the engine. Because the owner can subsequently call `attach_full_access_key`, the attacker gains a full-access key to the Aurora Engine NEAR account and can drain all bridged ETH and ERC-20 tokens.

---

### Finding Description
The `new` entrypoint in `engine/src/contract_methods/admin.rs` is the sole initialization path for the Aurora Engine. It reads `owner_id` entirely from caller-supplied input and writes it directly to state:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    // owner_id comes entirely from args — no check on env.predecessor_account_id()
    let state: EngineState = args.into();
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

There is no assertion that `env.predecessor_account_id() == env.current_account_id()` (i.e., that the caller is the contract account itself). The only guard is the already-initialized check on line 57, which is irrelevant before first initialization. [2](#0-1) 

`NewCallArgs` (all versions V1–V4) accept `owner_id` as a plain caller-supplied field: [3](#0-2) 

The resulting `EngineState.owner_id` is then used as the sole authority gate for every privileged operation, including `attach_full_access_key`: [4](#0-3) 

`require_owner_only` simply compares `state.owner_id` against `predecessor_account_id`: [5](#0-4) 

Contrast this with the XCC router's `initialize`, which correctly records the first caller as the immutable parent and rejects all other callers on subsequent invocations: [6](#0-5) 

The main engine's `new` has no equivalent protection.

---

### Impact Explanation
**Critical — Direct theft of all user funds.**

An attacker who wins the initialization race becomes the engine owner. As owner they can immediately call `attach_full_access_key` to add their own ED25519 full-access key to the Aurora Engine NEAR account. A full-access key grants unrestricted control over the account: the attacker can transfer all NEAR, call any method, and redeploy arbitrary WASM — effectively stealing every bridged ETH and ERC-20 token held in the contract. [4](#0-3) 

---

### Likelihood Explanation
**Medium.** The window exists whenever the `DeployContract` action and the `new` call are submitted in separate transactions rather than as a single atomic NEAR batch. Any NEAR account monitoring the chain for uninitialized Aurora Engine deployments (e.g., during an upgrade or a new shard deployment) can exploit this. The attack requires no special privileges, no leaked keys, and no social engineering — only the ability to submit a NEAR transaction before the deployer's initialization call is confirmed.

---

### Recommendation
Add a caller-identity check at the top of `new` that requires the predecessor to be the contract account itself:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Ensure only the contract account itself (i.e., the deployer acting
    // via a batch transaction) can initialize the engine.
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    // ... rest of initialization
}
```

Alternatively, enforce that `DeployContract` and the `new` call are always submitted in a single atomic NEAR batch transaction, making the window structurally impossible.

---

### Proof of Concept

1. Deployer submits a `DeployContract` action for the Aurora Engine WASM in transaction T1 (without bundling `new` in the same batch).
2. Attacker observes T1 on-chain and immediately submits a call to `new` with:
   ```json
   {
     "chain_id": 1313161554,
     "owner_id": "attacker.near",
     "upgrade_delay_blocks": 1,
     "key_manager": "attacker.near"
   }
   ```
3. Attacker's `new` call is included before the deployer's own `new` call. `state::get_state` returns `Err` (uninitialized), so the guard passes. `owner_id` is written as `attacker.near`.
4. Deployer's `new` call now fails with `ERR_ALREADY_INITIALIZED`.
5. Attacker calls `attach_full_access_key` with their own public key. `require_owner_only` passes because `state.owner_id == "attacker.near" == predecessor_account_id`.
6. Attacker now holds a full-access key to the Aurora Engine NEAR account and can transfer all bridged assets. [1](#0-0) [4](#0-3) [5](#0-4)

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
