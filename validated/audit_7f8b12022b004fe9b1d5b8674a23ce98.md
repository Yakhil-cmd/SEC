Audit Report

## Title
Coinbase Transaction Input Causes Permanent Retriable Error Loop in ckBTC Checker â€” (`rs/bitcoin/checker/src/state.rs`, `rs/bitcoin/checker/src/fetch.rs`)

## Summary
`TransactionCheckData::from_transaction` stores coinbase inputs verbatim without any coinbase sentinel check, causing `check_fetched` to unconditionally attempt to fetch the phantom txid `[0;32]` from external providers. Every provider returns a non-200 response for this non-existent txid, which is classified as a retriable `TransientInternalError`. The `Provider::next()` function cycles through all three providers in a round-robin loop with no termination condition, so the deposit is permanently stuck and the ckBTC minter drains cycles on every retry.

## Finding Description

**Step 1 â€” No coinbase guard in `from_transaction`**

`TransactionCheckData::from_transaction` in `state.rs` (L92â€“99) maps every transaction input to a `PreviousOutput` verbatim, including the Bitcoin coinbase sentinel (`previous_output.txid = [0;32]`, `vout = 0xFFFFFFFF`). There is no `is_coinbase()` check anywhere in the checker codebase (confirmed: zero matches for "coinbase" across all checker source files).

**Step 2 â€” `check_fetched` unconditionally fetches every input**

In `fetch.rs` (L196â€“203), for every input where `fetched.input_addresses[index].is_none()`, `try_fetch_tx(input.txid)` is called. For a coinbase transaction, `input.txid` is `[0;32]`, which is a valid `Txid` key in the `FetchTxCache`.

**Step 3 â€” Non-200 HTTP status maps to a retriable error**

In `main.rs` (L426â€“431), any non-200 HTTP status (including 404 for the non-existent txid `[0;32]`) returns `HttpGetTxError::Rejected { code: 2, message: ... }`. In `fetch.rs` (L25â€“27), `Rejected` maps to `CheckTransactionRetriable::TransientInternalError` â€” a retriable, not irrecoverable, response.

**Step 4 â€” Provider cycling never terminates**

In `providers.rs` (L69â€“83), `Provider::next()` cycles Btcscan â†’ Blockstream â†’ MempoolSpace â†’ Btcscan indefinitely with no exhaustion state. In `fetch.rs` (L85â€“90), `FetchTxStatus::Error(err)` always advances to `err.provider.next()` and retries. Since all three providers return 404 for txid `[0;32]`, the cycle never produces a `Fetched` or irrecoverable result.

**Why existing checks fail**

The `FetchTxCache` eviction at `MAX_FETCH_TX_ENTRIES = 10,000` does not help: if the `[0;32]` entry is evicted, the next call simply re-adds it and restarts the cycle from the first provider. The `ResponseTooLarge` irrecoverable path is never reached because the 404 response body is small. The `TxidMismatch` irrecoverable path is never reached because the fetch never succeeds.

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Significant Chain Fusion, ck-token, ledger â€¦ security impact with concrete user or protocol harm."*

- The coinbase UTXO deposit is permanently stuck: the checker never returns `Passed` or `CheckTransactionIrrecoverableError`, only `TransientInternalError`.
- The ckBTC minter retries `check_transaction` indefinitely, paying `CHECK_TRANSACTION_CYCLES_SERVICE_FEE` on every call (accepted from the minter's cycle balance at `main.rs` L147â€“151).
- The checker canister also pays cycles for each HTTP outcall to the three providers.
- A single coinbase UTXO deposit is sufficient to trigger and sustain the loop indefinitely.

## Likelihood Explanation

Coinbase UTXOs mature after 100 confirmations and are freely spendable by any miner or secondary market buyer. Sending a mature coinbase output to the ckBTC minter deposit address is an unprivileged, standard on-chain transaction requiring no special access, no victim interaction, and no repeated attacker effort. The loop is self-sustaining once triggered.

## Recommendation

In `TransactionCheckData::from_transaction` (`state.rs` L88â€“113), detect coinbase inputs using the Bitcoin protocol sentinel values (`txid == [0;32]` and `vout == 0xFFFFFFFF`) and return `CheckTransactionIrrecoverableError::InvalidTransaction("coinbase transactions cannot be checked")` immediately. This surfaces the condition explicitly to the minter, allowing it to handle coinbase deposits with a defined policy rather than silently looping. Alternatively, skip coinbase inputs so `check_fetched` never attempts to fetch them, but the explicit irrecoverable error is safer as it prevents the minter from retrying indefinitely.

## Proof of Concept

1. Obtain a mature coinbase UTXO (any mined block reward after 100 confirmations).
2. Send it to the ckBTC minter deposit address.
3. The minter calls `check_transaction(coinbase_txid)`.
4. The checker fetches the coinbase transaction successfully (it exists on-chain).
5. `check_fetched` calls `try_fetch_tx([0;32])`.
6. All three providers return 404 â†’ `HttpGetTxError::Rejected` â†’ `TransientInternalError`.
7. On every subsequent `check_transaction` call, `try_fetch_tx` sees `FetchTxStatus::Error` for `[0;32]`, advances to the next provider via `Provider::next()`, fetches again, gets 404 again.
8. The response is always `TransientInternalError`; the deposit never resolves; cycles drain on every retry.

A deterministic integration test can reproduce this by injecting a mock coinbase transaction (one input with `txid=[0;32]`, `vout=0xFFFFFFFF`) into the checker's `check_transaction` endpoint with a mock HTTP provider that returns 404 for `[0;32]`, and asserting the response is `CheckTransactionIrrecoverableError` rather than `TransientInternalError` after three provider cycles.
