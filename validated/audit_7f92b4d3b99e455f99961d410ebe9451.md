Based on my full read of the relevant code, here is my analysis:

**Code trace:**

`socket_read_messages` in `rs/canister_sandbox/src/transport.rs` (lines 434–484):
- Allocates `fds = Vec::<RawFd>::new()` with no capacity bound [1](#0-0) 
- The main loop first tries to decode complete frames; only on success does it call `install_file_descriptors` to drain `fds` [2](#0-1) 
- Then calls `receive_message`, which unconditionally appends every SCM_RIGHTS FD from `recvmsg` into `fds` with no bound check [3](#0-2) 

`FrameDecoder::decode` returns `None` (and does not consume the buffer) whenever fewer than 8 bytes or fewer than `size` bytes are present: [4](#0-3) 

`install_file_descriptors` is **only** reached when `decode` returns `Some`. When it does run, excess FDs beyond what the message needs are left in `fds` as raw integers — `RawFd` is `i32`, so `drain()` drops the integer value without calling `close()`: [5](#0-4) 

**Attack path:**

A compromised sandbox process (the sender side of the Unix socket) can:
1. Open many FDs locally (e.g., repeated `open("/dev/null")`)
2. Send partial frames — e.g., exactly 1–7 bytes of payload per `sendmsg` call — so `FrameDecoder::decode` always returns `None`
3. Attach up to `MAX_NUM_FD_PER_MESSAGE = 16` SCM_RIGHTS FDs per call [6](#0-5) 
4. The replica's `receive_message` receives and opens those FDs into the replica process on every iteration, pushing them onto `fds`
5. Since no complete frame is ever assembled, `install_file_descriptors` is never called, and `fds` grows without bound — all received FDs remain open in the replica process

With a typical per-process FD limit of 65536, ~4096 such partial-frame `sendmsg` calls (each carrying 16 FDs) exhaust the replica's FD table. After that, the replica cannot open new sockets, pipes, or files — breaking consensus socket connections, state sync, and any other I/O.

**No existing guard prevents this.** There is no check on `fds.len()` before or after `receive_message`, no cap on accumulated FDs, and no `close()` call on FDs that are never consumed.

### Title
Unbounded FD accumulation in `socket_read_messages` allows compromised sandbox to exhaust replica process file descriptors — (`rs/canister_sandbox/src/transport.rs`)

### Summary
The `socket_read_messages` function accumulates received SCM_RIGHTS file descriptors into an unbounded `Vec<RawFd>`. A compromised sandbox process can send arbitrarily many partial frames, each carrying up to 16 FDs via `SCM_RIGHTS`, causing the replica controller process to exhaust its open-file-descriptor limit.

### Finding Description
In `socket_read_messages`, the `fds: Vec<RawFd>` vector is populated by `receive_message` on every `recvmsg` call. File descriptors are only drained from this vector inside `install_file_descriptors`, which is only called when `FrameDecoder::decode` returns a complete frame. A compromised sandbox can deliberately send partial frames (e.g., 1–7 bytes of payload) so that `decode` perpetually returns `None`. Each such partial `sendmsg` can carry up to 16 SCM_RIGHTS FDs. The replica's kernel-side `recvmsg` opens those FDs into the replica process and `receive_message` pushes them onto `fds` with no bound. Because `RawFd` is a plain `i32`, neither `Vec::drain` nor `Vec::clear` closes the underlying file descriptors; they remain open and counted against the process FD limit indefinitely.

### Impact Explanation
Once the replica's FD table is exhausted (`EMFILE`), it cannot open new TCP/Unix sockets, pipes, or files. This breaks:
- Consensus socket connections to peers
- State sync and artifact pool I/O
- Any new canister execution that requires file handles

The replica node effectively stops participating in consensus, constituting a targeted denial-of-service against a single replica node.

### Likelihood Explanation
The attack requires only a compromised sandbox process — i.e., a canister that has achieved a sandbox escape. The sandbox escape is the harder prerequisite, but the IC's defense-in-depth model explicitly treats a compromised sandbox as an in-scope threat. Once the sandbox is compromised, the FD exhaustion attack is trivial: ~4096 `sendmsg` calls with 16 FDs each suffice against a 65536-FD limit, and the sandbox can replenish its own FDs by repeatedly opening `/dev/null`.

### Recommendation
1. **Bound `fds.len()`**: Before pushing new FDs in `receive_message`, check that `fds.len()` does not exceed a small constant (e.g., `MAX_NUM_FD_PER_MESSAGE * 2`). If exceeded, close the excess FDs with `libc::close()` and terminate or reset the connection.
2. **Close unconsumed FDs**: In `install_file_descriptors`, explicitly `close()` any FDs remaining in `fds` after the message's slots are filled, rather than leaving them as raw integers.
3. **Close FDs on connection teardown**: When `socket_read_messages` exits its loop, iterate over `fds` and close each entry.
