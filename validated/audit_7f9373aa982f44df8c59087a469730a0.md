### Title
Unsafe `u64`-to-`u16` Truncation of Callee Address in Special-Address Hook Dispatch — (`File: basic_bootloader/src/bootloader/runner.rs`)

---

### Summary

In `basic_bootloader/src/bootloader/runner.rs`, the callee address's lowest 64-bit limb is silently truncated to `u16` (without bounds checking) when deciding whether to route a call to a system hook. This is a direct structural analog of the reported asset-index truncation bug: a wider integer is narrowed to a smaller type via `as`, discarding high bits. If `SPECIAL_ADDRESS_SPACE_BOUND` exceeds `u16::MAX` (65 535), an unprivileged caller can craft a call to an address whose low 16 bits alias an existing hook address, causing the wrong hook to be invoked with attacker-controlled calldata.

---

### Finding Description

Two adjacent sites in `execute_call` perform the same unsafe cast:

**Site 1 — hook-presence check** [1](#0-0) 

```rust
let is_call_to_special_address = external_call_launch_params.external_call.callee.as_uint()
    < SPECIAL_ADDRESS_BOUND.as_uint()
    && self.hooks.has_hook_for(
        external_call_launch_params.external_call.callee.as_limbs()[0] as u16,
    );
```

**Site 2 — hook dispatch** [2](#0-1) 

```rust
let (res, remaining_memory) = self.hooks.try_intercept(
    external_call_launch_params.external_call.callee.as_limbs()[0] as u16,
    external_call_launch_params.external_call,
    caller_ee_type as u8,
    self.system,
    return_memory,
)?;
```

`as_limbs()[0]` returns the lowest `u64` limb of the `B160` address. The subsequent `as u16` cast silently discards bits 16–63. For any address `A` where `A < SPECIAL_ADDRESS_SPACE_BOUND` but `A > 0xFFFF`, the effective hook key becomes `A & 0xFFFF`, which may alias a completely different registered hook address.

The special-address bound is defined as: [3](#0-2) 

```rust
const SPECIAL_ADDRESS_BOUND: B160 = B160::from_limbs([SPECIAL_ADDRESS_SPACE_BOUND, 0, 0]);
```

If `SPECIAL_ADDRESS_SPACE_BOUND > 0x10000`, addresses such as `0x10000` pass the bound check but truncate to `0x0000` when cast to `u16`.

---

### Impact Explanation

An unprivileged transaction sender who calls address `H + k·0x10000` (where `H` is a registered hook address and `k ≥ 1`, and the result is still `< SPECIAL_ADDRESS_SPACE_BOUND`) causes:

1. `has_hook_for(H)` returns `true` → the call is routed to `call_to_special_address_execute_callee_frame` instead of the normal EVM execution path.
2. `try_intercept(H, actual_call_params, ...)` invokes the hook registered at `H` with attacker-controlled calldata.

Depending on what hook `H` implements (e.g., token transfers, pubdata accounting, interop), this can produce incorrect state transitions, bypass hook-level access controls, or corrupt accounting — all triggered by a normal user transaction with no privileged access required.

---

### Likelihood Explanation

Exploitability is conditional on `SPECIAL_ADDRESS_SPACE_BOUND > 0xFFFF`. If the bound is exactly `0x10000`, all special addresses fit in `u16` and no truncation occurs. However, the cast is structurally unsafe regardless, and any future increase of the bound (or a current value already above `0xFFFF`) immediately opens the attack surface. The attacker entry path requires only the ability to send a transaction — no keys, governance, or oracle manipulation needed.

---

### Recommendation

Replace the unchecked `as u16` casts with explicit bounds-checked conversions. Before using the address as a hook key, verify that the lowest limb fits in `u16`:

```rust
let addr_low = external_call_launch_params.external_call.callee.as_limbs()[0];
let hook_key: u16 = addr_low.try_into().expect("special address exceeds u16 range");
```

Or, equivalently, assert at the point where `SPECIAL_ADDRESS_SPACE_BOUND` is defined that it does not exceed `u16::MAX + 1`, and document this invariant explicitly.

---

### Proof of Concept

Assume `SPECIAL_ADDRESS_SPACE_BOUND = 0x20000` and a hook is registered at address `0x0001` (a common system-contract address).

1. Attacker sends a transaction calling address `0x10001`.
2. `0x10001 < SPECIAL_ADDRESS_BOUND` → passes the bound check.
3. `0x10001_u64 as u16 = 0x0001` → `has_hook_for(0x0001)` returns `true`.
4. `try_intercept(0x0001, call_params_for_0x10001, ...)` is called.
5. The hook at `0x0001` executes with attacker-supplied calldata, producing state changes that should never have been triggered by a call to `0x10001`.

The root cause — silent integer truncation without bounds checking — is structurally identical to the reported asset-index truncation bug. [1](#0-0)
