### Title
Unbounded EIP-6110 Deposit Event Loop in Block Finalization Inflates Proving Cost Without Gas Bound - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

`eip6110_events_parser` iterates over every event emitted in the block during post-block finalization with no cap on the number of deposit events processed. The analogous EIP-7002 and EIP-7251 system-part functions both enforce explicit per-block maximums (`MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16` and `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2`). EIP-6110 has no such guard, creating an asymmetric resource accounting bug where an attacker can inflate the unmetered finalization work by packing a block with deposit-contract calls.

---

### Finding Description

During block finalization, `post_op` calls three EIP-specific processors in sequence: [1](#0-0) 

The EIP-7002 processor reads the queue length and immediately clamps it: [2](#0-1) 

The EIP-7251 processor does the same: [3](#0-2) 

The EIP-6110 processor has no equivalent cap. It iterates over the entire block event log and processes every matching deposit event: [4](#0-3) 

For each matching event, `validate_and_write_event_data` performs SHA-256 hashing over 192 bytes of event data. This work is performed entirely outside of gas accounting — there is no `with_infinite_ergs` guard needed because the function never charges any resources at all. The number of deposit events is bounded only by the block gas limit, not by any protocol-level cap analogous to those applied to EIP-7002 and EIP-7251.

The function is invoked in both sequencing and proving modes: [5](#0-4) [6](#0-5)

### Impact Explanation

The finalization phase runs outside the block's gas/native resource budget. An attacker who fills a block with deposit-contract calls forces the prover to execute an unbounded amount of SHA-256 hashing work during `post_op` that is not reflected in the block's gas cost. In the RISC-V proving mode this directly inflates the RISC-V cycle count for the block, increasing proving cost and latency proportionally to the number of deposit events. At the extreme, a block filled to the gas limit with deposit calls could make the block significantly more expensive to prove than its gas usage implies, creating a cost asymmetry between the attacker's on-chain spend and the protocol's proving overhead. This is a **resource accounting bug** with a secondary **DoS / proving-cost inflation** impact.

---

### Likelihood Explanation

The deposit contract is a standard predeploy at a fixed address. Any unprivileged EOA can call it. The only economic friction is the per-call gas cost and any minimum deposit amount enforced by the contract bytecode. Because the proving overhead per deposit event is unmetered, the ratio of attacker cost to protocol cost is unfavorable to the protocol. The attack requires no privileged access, no leaked keys, and no governance majority.

---

### Recommendation

Apply a `MAX_DEPOSIT_REQUESTS_PER_BLOCK` constant in `eip6110_events_parser`, consistent with the caps already present in `eip7002_system_part` (`MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16`) and `eip7251_system_part` (`MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2`). The EIP-6110 specification itself does not mandate processing all deposits in a single block; a per-block cap is both spec-compatible and already the established pattern in the codebase.

---

### Proof of Concept

1. Deploy or use the existing deposit contract predeploy at `DEPOSIT_CONTRACT_ADDRESS`.
2. Submit a block filled with transactions that each call the deposit contract and emit a `DepositEvent` (topic `0x649bbc62...`). With a 30 M gas block limit and ~70 K gas per deposit call, ~430 deposit events can be packed into one block.
3. Observe that `eip6110_events_parser` iterates over all 430 events and calls `validate_and_write_event_data` for each, hashing 192 bytes per event (~82 KB total SHA-256 input) with no gas charged.
4. Compare to a block with 430 EIP-7002 withdrawal requests: `eip7002_system_part` would process at most `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16` of them, leaving the rest for future blocks.
5. The EIP-6110 path processes all 430 in the same block, with no analogous bound, inflating the unmetered proving work by a factor of ~27× relative to the capped EIP-7002 path. [7](#0-6) [8](#0-7) [9](#0-8)

pub fn eip6110_events_parser<S: EthereumLikeTypes>(
    system: &System<S>,
    requests_hasher: &mut impl crypto::sha256::Digest,
) -> Result<bool, SystemError>
where
    S::IO: IOSubsystemExt + IOTeardown<S::IOTypes>,
{
    // we can not easily get the number from one scan, so we will accumulate into hasher directly

    let mut event_encountered = false;
    let mut logger = system.get_logger();
    for event in system.io.events_iterator() {
        if event.address != &DEPOSIT_CONTRACT_ADDRESS {
            continue;
        }
        if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
            if event_encountered == false {
                event_encountered = true;
                requests_hasher.update(&[DEPOSIT_REQUEST_EIP_7685_TYPE]);
            }
            let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
                panic!("invalid deposit event structure");
            };
        }
    }

    Ok(event_encountered)
