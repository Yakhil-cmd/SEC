### Title
User-Controlled `refund_address` in `ExitToNear` Precompile Enables Refund Redirection to Arbitrary Addresses - (File: engine-precompiles/src/native.rs)

---

### Summary

When the `error_refund` feature is enabled, the `ExitToNear` precompile accepts a `refund_address` directly from user-supplied calldata (bytes 1–20 of the input) with no validation that it matches `context.caller`. Any caller — including an intermediary contract — can set `refund_address` to an arbitrary EVM address. If the downstream NEAR `ft_transfer` or `ft_transfer_call` promise fails, the engine re-mints or transfers the bridged tokens to that arbitrary address rather than to the actual token owner. This is the direct Aurora Engine analog of the ZetaChain M-13 bug, where the `Sender` of a cctx was set to `tx.origin` instead of the emitting contract, causing refunds to reach the wrong recipient.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Consider an intermediary contract (e.g., a DEX router or aggregator) that:
1. Holds pooled ETH or ERC-20 tokens from multiple depositors.
2. Calls `ExitToNear` on behalf of a user, setting `refund_address` = `tx.origin` (the transaction signer) rather than its own address — a natural pattern for "refund the user on failure."

An attacker can:
1. Call the intermediary contract, triggering an `ExitToNear` call that uses pool funds and sets `refund_address` = attacker's address.
2. Engineer the NEAR-side transfer to fail (e.g., supply an unregistered NEAR account ID as the recipient).
3. Receive the full refund at their address, draining pool funds that belong to other depositors.

Additionally, if `refund_address` is set to `address(0)` — either by mistake or by a griefing attacker who controls the calldata — the refunded tokens are permanently frozen at the zero address, satisfying the permanent fund freeze impact criterion.

---

### Likelihood Explanation

**Medium.** The `error_refund` feature must be compiled in (it is a named Cargo feature, not always active). When it is active, the attack requires:
- An intermediary contract that forwards user-influenced calldata to `ExitToNear`, or that sets `refund_address` = `tx.origin` / `msg.sender` of the outer call rather than its own address.
- The ability to cause the downstream NEAR promise to fail (trivially achieved by supplying an unregistered NEAR account ID).

Both conditions are realistic in production DeFi usage of Aurora.

---

### Recommendation

Derive `refund_address` from `context.caller` inside the precompile rather than accepting it from calldata. Replace the user-supplied field with:

```rust
let refund_address = Address::new(context.caller);
```

This mirrors the correct fix suggested in the ZetaChain report: use the address of the entity that actually initiated the bridge operation, not an address supplied by an untrusted caller. If callers legitimately need to redirect refunds, a separate authenticated mechanism (e.g., a signed permit) should be required.

### Proof of Concept

**Setup (ETH base-token exit via intermediary contract):**

```
Attacker EOA  ──calls──►  IntermediaryContract.bridge(near_account="unregistered.near")
                                │
                                │  calls ExitToNear precompile with:
                                │    value          = 100 ETH (from pool balance)
                                │    input[0]       = 0x00  (ETH flag)
                                │    input[1..21]   = attacker_address  ← refund_address
                                │    input[21..]    = "unregistered.near"
                                ▼
                          ExitToNear::run()
                            parse_input() → refund_address = attacker_address  ✓ (no check vs context.caller)
                            schedules ft_transfer("unregistered.near", 100 ETH)
                                │
                                │  NEAR promise FAILS (account not registered)
                                ▼
                          exit_to_near_precompile_callback()
                            refund_on_error(recipient_address = attacker_address)
                                │
                                ▼
                          100 ETH transferred to attacker_address
```

The intermediary contract's pool has lost 100 ETH; the attacker has gained 100 ETH.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
