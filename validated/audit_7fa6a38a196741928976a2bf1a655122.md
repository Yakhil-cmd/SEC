Audit Report

## Title
Missing Staleness Validation in CMC Exchange Rate Acceptance Path Allows Stale ICP/XDR Rate to Drive Cycles Minting — (`rs/nns/cmc/src/exchange_rate_canister.rs`, `rs/nervous_system/clients/src/exchange_rate_canister_client.rs`)

## Summary
The Cycles Minting Canister (CMC) accepts ICP/XDR exchange rates from the XRC without verifying that the returned rate's timestamp is within an acceptable age of the current time. `validate_exchange_rate` checks only source counts, and `do_set_icp_xdr_conversion_rate` enforces only non-zero value and strictly-greater-than-stored-timestamp. When the XRC serves a cached rate whose timestamp is newer than the CMC's stored rate but significantly behind the current time, the CMC accepts and certifies it, and all subsequent `notify_top_up` / `notify_create_canister` calls mint cycles at the wrong price.

## Finding Description

**Root cause — `validate_exchange_rate` omits a staleness check**

`validate_exchange_rate` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` (L110–129) validates only `base_asset_num_received_rates` and `quote_asset_num_received_rates`. There is no check that `exchange_rate.timestamp` is within an acceptable age of the current time.

**CMC acceptance path — no maximum-age guard**

`update_exchange_rate` in `rs/nns/cmc/src/exchange_rate_canister.rs` (L236–280) calls `xrc_client.get_icp_to_xdr_exchange_rate(None)` (requesting the latest rate), calls `validate_exchange_rate`, then passes the result directly to `do_set_icp_xdr_conversion_rate`. No age check is performed between these steps.

`do_set_icp_xdr_conversion_rate` in `rs/nns/cmc/src/main.rs` (L1009–1040) enforces exactly two guards:
1. `xdr_permyriad_per_icp != 0`
2. `proposed_conversion_rate.timestamp_seconds > current_conversion_rate.timestamp_seconds`

Neither guard compares `proposed_conversion_rate.timestamp_seconds` against `env.now_timestamp_seconds()`. A rate whose timestamp is hours behind the current time is accepted as long as it is strictly newer than the previously stored rate.

**Exploit window mechanics**

The "strictly greater timestamp" guard provides partial protection in the steady state: once the CMC stores a stale rate, subsequent polls returning the same cached XRC timestamp are rejected. However, the initial acceptance of a stale rate is not blocked. The window opens when:
- The XRC's HTTP outcalls to exchanges fail or are rate-limited, causing the XRC to serve a cached rate with timestamp `T_stale`.
- `T_stale` is greater than the CMC's currently stored rate timestamp (so the guard at L1023–1030 passes).
- `T_stale` is significantly less than `now` (e.g., 30–120 minutes behind).

Once the CMC stores the stale rate, it remains in effect until the XRC recovers and returns a rate with a timestamp strictly greater than `T_stale`. During this entire window, all cycles-minting operations use the stale price.

**Contrast with the Governance path**

`fetch_and_validate_rate` in `rs/nns/governance/src/timer_tasks/update_icp_xdr_rate_related_data.rs` (L289–297) additionally verifies `exchange_rate.timestamp == timestamp` (the exact requested historical day), providing a tight timestamp bound. The CMC path, which requests the current rate with `None`, has no equivalent guard.

## Impact Explanation

The CMC's stored `icp_xdr_conversion_rate` is the direct input to cycles minting for every `notify_top_up` and `notify_create_canister` call. When the stale rate is lower than the true current rate (ICP price rose while the XRC cache was stale), any caller can convert ICP to cycles at a favorable price, receiving more cycles than the current market rate warrants. This constitutes unauthorized cycles minting — a concrete protocol accounting violation. The CMC also certifies the stale rate via `set_certified_data`, so downstream consumers of the certified rate are also affected.

This matches the allowed High impact: **Significant XRC and NNS/CMC security impact with concrete user or protocol harm** ($2,000–$10,000).

## Likelihood Explanation

The XRC aggregates prices via HTTP outcalls to external exchanges. HTTP outcall failures, rate limiting, or exchange downtime are normal operational events on the IC. When they occur, the XRC may serve a cached rate. The CMC polls every `REFRESH_RATE_INTERVAL_SECONDS = 5 * ONE_MINUTE_SECONDS` (L15–16 of `exchange_rate_canister.rs`). Any principal holding ICP can call `notify_top_up` or `notify_create_canister` — no privileged access is required. An attacker who monitors the CMC's certified rate and the real market price can detect the discrepancy and act within the stale window. The condition cannot be forced by the attacker but is a realistic operational scenario.

## Recommendation

Add a maximum-age check in `update_exchange_rate` immediately after receiving the XRC response, before calling `validate_exchange_rate`:

```rust
let max_age_seconds: u64 = 30 * 60; // configurable
let rate_age = now_timestamp_seconds.saturating_sub(exchange_rate.timestamp);
if rate_age > max_age_seconds {
    return Err(UpdateExchangeRateError::InvalidRate(format!(
        "Rate timestamp {} is {} seconds old (max {})",
        exchange_rate.timestamp, rate_age, max_age_seconds
    )));
}
```

Alternatively, extend `validate_exchange_rate` to accept a `now_timestamp_seconds` parameter and perform the check there, keeping the validation logic centralized. The threshold should be stored in canister state so it can be adjusted via upgrade without a code change.

## Proof of Concept

1. Deploy a local replica with the CMC and a mock XRC.
2. Initialize the CMC with a stored rate at timestamp `T0`.
3. Configure the mock XRC to return a rate with timestamp `T0 + 60` (one minute newer than stored) but set the environment clock to `T0 + 7200` (two hours later).
4. Trigger `update_exchange_rate`. Observe that `validate_exchange_rate` passes (source counts are valid) and `do_set_icp_xdr_conversion_rate` passes (T0+60 > T0). The CMC stores the rate with timestamp `T0+60` even though it is 7140 seconds stale.
5. Call `notify_top_up` with ICP. Observe that cycles are minted at the stale rate.
6. A unit test mirroring the existing `test_periodic_calls_the_xrc_and_sets_the_rate` pattern in `rs/nns/cmc/src/exchange_rate_canister.rs` can reproduce this by setting `env.now_timestamp_seconds` to `T0 + 7200` while the mock XRC returns a rate with timestamp `T0 + 60`, and asserting that the call is rejected — which it currently is not.
