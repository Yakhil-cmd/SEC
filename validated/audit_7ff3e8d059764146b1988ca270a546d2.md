Audit Report

## Title
Missing Minimum Fee Floor in ckDOGE Minter Allows Anomalously Low Fee Rates to Produce Stuck Dogecoin Withdrawal Transactions - (File: rs/dogecoin/ckdoge/minter/src/fees/mod.rs)

## Summary
`DogecoinFeeEstimator::estimate_nth_fee` returns the raw fee percentile value with no minimum floor, unlike `BitcoinFeeEstimator::estimate_nth_fee` which applies `.max(self.minimum_fee_per_vbyte())`. The ckBTC minter suffered a confirmed production incident on 2025-06-21 where an anomalously low fee estimate caused three BTC withdrawal transactions to get stuck; a stop-gap fix was applied to ckBTC but was never ported to ckDOGE. If the same anomaly recurs on the Dogecoin canister, ckDOGE withdrawal transactions will be built with near-zero fees, get stuck or dropped from the Dogecoin mempool, and users will have their ckDOGE burned with no DOGE received until a governance upgrade is executed.

## Finding Description
In `rs/dogecoin/ckdoge/minter/src/fees/mod.rs` lines 53–66, `DogecoinFeeEstimator::estimate_nth_fee` returns the raw percentile value for Mainnet:

```rust
Network::Mainnet => {
    if fee_percentiles.len() < 100 || nth >= 100 {
        return None;
    }
    Some(fee_percentiles[nth])   // no floor applied
}
```

In contrast, `rs/bitcoin/ckbtc/minter/src/fees/mod.rs` lines 99–113 applies a floor after the same pattern:

```rust
median_fee.map(|f| f.max(self.minimum_fee_per_vbyte()))
```

where `minimum_fee_per_vbyte()` (lines 80–87) returns 1,500 millis/byte for Mainnet. This floor was introduced specifically because the Bitcoin canister returned a fee of 142 millisatoshis/vbyte on 2025-06-21 instead of the expected ~2,000, as documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` lines 26–30. The ckDOGE minter calls `dogecoin_canister::dogecoin_get_fee_percentiles` via `DogeCanisterRuntime::get_current_fee_percentiles` (`rs/dogecoin/ckdoge/minter/src/lib.rs` lines 74–80) and feeds the result directly into `estimate_nth_fee` with no subsequent floor check. If the Dogecoin canister returns an anomalously low percentile array, the minter will use that value to price and submit Dogecoin transactions below the network relay threshold.

## Impact Explanation
This matches **High ($2,000–$10,000): Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.** Users who submit valid `retrieve_doge` requests have their ckDOGE burned on the IC ledger before the Dogecoin transaction is broadcast. If the transaction is priced below the relay threshold it will be dropped or sit unconfirmed indefinitely. The minter's RBF resubmission logic increments by `MIN_RELAY_FEE_RATE_INCREASE` (100,000 millikoinu/byte), but if the initial fee is near zero, many resubmission rounds will still be below the relay threshold, prolonging the stuck state. Recovery requires a governance upgrade proposal, exactly as was required for ckBTC.

## Likelihood Explanation
The root cause of the ckBTC anomaly is explicitly unresolved — the upgrade proposal states "We currently don't have a satisfying explanation for how this low median fee was computed." The ckDOGE minter uses the same architectural pattern (`get_current_fee_percentiles` → `estimate_nth_fee`) over a different but analogous canister. No privileged access or attacker action is required; the anomaly is triggered by the same unknown condition that affected the Bitcoin canister. Likelihood is **medium**.

## Recommendation
Add a `minimum_fee_per_byte()` const method to `DogecoinFeeEstimator` in `rs/dogecoin/ckdoge/minter/src/fees/mod.rs` returning a network-appropriate floor (e.g., 1,000,000 millikoinu/byte for Mainnet, matching the Dogecoin minimum relay fee of 1,000 koinu/byte), and apply it in `estimate_nth_fee` via `.max(self.minimum_fee_per_byte())` before returning, mirroring the ckBTC fix at line 112 of `rs/bitcoin/ckbtc/minter/src/fees/mod.rs`.

## Proof of Concept
A unit test analogous to the ckBTC test at `rs/bitcoin/ckbtc/minter/tests/tests.rs` lines 2882–2887 can demonstrate the issue:

1. Construct a `DogecoinFeeEstimator` for `Network::Mainnet`.
2. Call `estimate_nth_fee(&[FeeRate::from_millis_per_byte(1); 100], 50)`.
3. Observe the returned value is `Some(FeeRate::from_millis_per_byte(1))` — far below any Dogecoin relay threshold — with no floor applied.
4. Contrast with the ckBTC equivalent, which returns `Some(FeeRate::from_millis_per_byte(1_500))` due to the floor.

The existing ckBTC test at lines 2882–2887 already serves as a template; an identical test for ckDOGE would fail (demonstrating the missing floor) until the fix is applied.
