### Title
`native_per_pubdata` Truncates to Zero via Floor Division, Allowing Pubdata to Be Written for Free - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When `pubdata_price < native_price`, the `native_per_pubdata` ratio computed via `wrapping_div` (floor/truncating division) rounds down to zero. This causes the entire pubdata charging mechanism to charge **zero** native resources for any amount of pubdata written during a transaction, directly analogous to the WooFi `gamma = 0` bug where a rounding-to-zero bypasses a critical state update.

---

### Finding Description

In `validation_impl.rs` at line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` performs floor (truncating) integer division. When `pubdata_price < native_price`, the result is `0`. This is in direct contrast to `native_per_gas`, which uses `div_ceil` to prevent rounding to zero:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

The same floor-division pattern is replicated in `api/src/helpers.rs`:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

**When `native_per_pubdata = 0`, the following cascade occurs:**

**1. Intrinsic pubdata overhead becomes zero** — no native is deducted for the fixed per-transaction pubdata cost:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
``` [4](#0-3) 

**2. Pubdata cost check always passes** — `get_resources_to_charge_for_pubdata` returns zero native for any amount of pubdata written:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // = 0 for any pubdata_spent
    .ok_or(out_of_native_resources!())?;
``` [5](#0-4) 

**3. `check_enough_resources_for_pubdata` always returns `true`** — the transaction is never reverted for pubdata costs regardless of how much pubdata it writes: [6](#0-5) 

**4. `delta_gas` adjustment is suppressed** — since pubdata consumes no native resources, `native_used` is artificially low, and the `delta_gas` correction that would otherwise charge extra gas for pubdata-heavy transactions does not fire:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [7](#0-6) 

The design intent of `native_per_pubdata` is documented explicitly:

> `nativePerGas := gasPrice/nativePrice` … `deltaGas := (nativeUsed / nativePerGas) - gasUsed` [8](#0-7) 

When `native_per_pubdata = 0`, the entire pubdata dimension of the double-resource accounting model is silently disabled.

---

### Impact Explanation

When `pubdata_price < native_price`, any user submitting transactions that write storage (SSTORE), emit events, or otherwise generate pubdata pays **zero** native resources for that pubdata. The operator must still publish all state diffs to L1 at real L1 gas cost, but receives no compensation for it through the native resource mechanism. A user can repeatedly submit pubdata-heavy transactions (e.g., writing many storage slots) up to the block pubdata limit, paying only EVM gas for computation but nothing for the L1 data publication cost. This is a direct financial loss for the operator and a resource accounting invariant violation.

---

### Likelihood Explanation

The trigger condition is `pubdata_price < native_price`. Both values are operator-set block-level parameters. In normal operation `pubdata_price >> native_price` (L1 calldata is expensive). However, if the operator raises `native_price` to make computation cheap (e.g., during a period of low proving costs), the ratio can round to zero without the operator realizing it — exactly the same class of inadvertent misconfiguration that the WooFi protocol team acknowledged. The inconsistency between `native_per_gas` (uses `div_ceil`) and `native_per_pubdata` (uses `wrapping_div`) suggests this is an unintentional asymmetry rather than a deliberate design choice.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — can produce 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. Additionally, add an explicit guard that rejects or specially handles the case where `native_per_pubdata == 0` but `pubdata_price > 0`, mirroring the existing guard for `native_price == 0`.

---

### Proof of Concept

**Setup**: Configure a block with `native_price = 1000` and `pubdata_price = 999`.

**Computation**:
- `native_per_pubdata = 999 / 1000 = 0` (floor division)
- `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0`
- For a transaction writing 100 storage slots: `native_for_pubdata = 100 * 0 = 0`

**Result**: The transaction writes 100 storage slots (generating ~3200 bytes of pubdata), passes `check_enough_resources_for_pubdata` with zero native charged, and the `delta_gas` adjustment does not compensate for the pubdata cost. The operator publishes ~3200 bytes to L1 at real cost while the user pays nothing for it. Repeating this across many transactions in a block exhausts the pubdata limit at no native cost to the user.

**File:** docs/double_resource_accounting.md (L37-50)
```markdown
First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. We expect the base fee to be enough to cover most transactions without the need of additional gas.
```
