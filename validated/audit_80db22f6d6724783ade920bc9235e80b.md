### Title
Silo Whitelist Bypass in `receive_base_tokens` Allows ETH Minting to Non-Whitelisted Addresses, Permanently Freezing Funds - (File: `engine/src/engine.rs`)

### Summary

In Silo mode, the Aurora Engine enforces an `Address` whitelist to restrict which EVM addresses may receive tokens. The `receive_erc20_tokens` path correctly checks this whitelist and redirects tokens to a configured fallback address when the recipient is not whitelisted. However, the `receive_base_tokens` path — invoked when the ETH connector calls `ft_on_transfer` — performs no whitelist check at all, minting ETH directly to any recipient address. ETH minted to a non-whitelisted address is permanently frozen because `is_allow_submit` blocks non-whitelisted addresses from submitting transactions to spend it.

### Finding Description

**Root cause:** `receive_base_tokens` in `engine/src/engine.rs` does not call `silo::is_allow_receive_erc20_tokens` or any equivalent whitelist check before crediting the recipient's balance.

`receive_erc20_tokens` (lines 818–822) correctly applies the Silo whitelist:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

`receive_base_tokens` (lines 773–790) has no such check — it directly sets the balance for any recipient:

```rust
set_balance(&mut self.io, &receipient, &new_balance);
``` [2](#0-1) 

The dispatch in `ft_on_transfer` routes to `receive_base_tokens` when the caller is the ETH connector account, and to `receive_erc20_tokens` otherwise: [3](#0-2) 

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which checks the `Address` whitelist: [4](#0-3) 

The same `Address` whitelist is also checked by `is_allow_submit`, which gates all EVM transaction submission: [5](#0-4) 

**Exploit flow:**

1. Silo mode is active; the `Address` whitelist is enabled; a fallback address is configured.
2. An attacker (or any NEAR user) calls `ft_transfer_call` on the ETH connector NEP-141 contract, specifying the Aurora engine as the receiver and a non-whitelisted EVM address as the recipient in the message.
3. The ETH connector calls `ft_on_transfer` on the Aurora engine with `predecessor_account_id == connector_account_id`.
4. `ft_on_transfer` dispatches to `receive_base_tokens`.
5. `receive_base_tokens` mints ETH to the non-whitelisted address with no whitelist check.
6. The non-whitelisted address now holds ETH but cannot call `submit` (blocked by `assert_access` → `is_allow_submit` → `is_address_allowed`), so the ETH is permanently frozen. [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH minted to a non-whitelisted address in Silo mode cannot be spent: `assert_access` rejects any `submit` call from an address not in the `Address` whitelist. There is no admin function to forcibly move ETH between EVM addresses. The funds are irrecoverably locked. The Silo operator's intent — that only whitelisted addresses may hold and use tokens — is violated for the base ETH asset class while being correctly enforced for ERC-20 tokens.

### Likelihood Explanation

**Medium.** Silo mode with an active `Address` whitelist is a supported, documented production configuration. Any NEAR account holding ETH-connector NEP-141 tokens can trigger this path by calling `ft_transfer_call` with an arbitrary EVM recipient. No privileged access is required. The scenario can occur accidentally (a user specifying the wrong recipient address) or deliberately.

### Recommendation

Apply the same whitelist-and-fallback logic to `receive_base_tokens` that is already applied in `receive_erc20_tokens`. When a fallback address is configured and the intended recipient is not in the `Address` whitelist, redirect the ETH credit to the fallback address instead of the original recipient:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
{
    receipient = fallback_address;
}
```

This mirrors the existing pattern in `receive_erc20_tokens` and ensures consistent authorization enforcement across both token-receiving code paths.

### Proof of Concept

1. Deploy Aurora in Silo mode; enable the `Address` whitelist; configure a fallback EVM address; do **not** add address `0xDEAD...` to the whitelist.
2. From any NEAR account holding ETH-connector NEP-141 tokens, call:
   ```
   ft_transfer_call(receiver_id: "aurora", amount: "1000", msg: "0xDEAD...")
   ```
3. Observe that `ft_on_transfer` calls `receive_base_tokens`, which credits `0xDEAD...` with 1000 wei — no whitelist check, no fallback redirect.
4. Attempt to submit any EVM transaction from `0xDEAD...`; observe `NotAllowed` error from `assert_access` → `is_allow_submit`.
5. Confirm the 1000 wei is permanently frozen: no EVM transaction can spend it, and no admin method can recover it. [7](#0-6) [8](#0-7)

    pub fn receive_base_tokens(

        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;

        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
