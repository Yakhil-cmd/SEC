Audit Report

## Title
Dust-Triggered Participation Rejection in SNS Swap Canister - (File: rs/sns/swap/src/swap.rs)

## Summary
In `refresh_buyer_token_e8s`, the raw ICP ledger balance `e8s` is used directly to compute `requested_increment_e8s` without first capping it to the buyer's remaining per-participant allowance. When the swap's remaining capacity (`max_increment_e8s`) is smaller than `min_participant_icp_e8s`, an attacker can send a trivial dust amount to a victim's deterministic subaccount, causing the `actual_increment_e8s` to be capped so low that `new_balance_e8s` falls below `min_participant_icp_e8s`, triggering a rejection at line 1241–1245. The victim's ICP is locked in the subaccount until the swap closes.

## Finding Description
The vulnerable code path in `rs/sns/swap/src/swap.rs`:

1. **Line 1153–1163**: `e8s` is read as the raw ledger balance of the buyer's subaccount — this includes any unsolicited dust sent by a third party.
2. **Line 1177**: `max_increment_e8s = self.available_direct_participation_e8s()` — this is the remaining swap capacity, which can be arbitrarily small near the end of a swap.
3. **Line 1202**: `if e8s < params.min_participant_icp_e8s` — this check passes because the dust-inflated balance meets the minimum.
4. **Line 1223**: `requested_increment_e8s = e8s - old_amount_icp_e8s` — computed from the dust-inflated balance.
5. **Line 1224**: `actual_increment_e8s = min(max_increment_e8s, requested_increment_e8s)` — capped by the tiny remaining capacity.
6. **Line 1225**: `new_balance_e8s = old_amount_icp_e8s + actual_increment_e8s` — for a new buyer with `old_amount_icp_e8s = 0`, this equals `actual_increment_e8s`, which is at most `max_increment_e8s`.
7. **Line 1237**: `new_balance_e8s = min(new_balance_e8s, max_participant_icp_e8s)` — no change when `new_balance_e8s` is already tiny.
8. **Line 1241–1245**: `if new_balance_e8s < params.min_participant_icp_e8s` → **Error returned**.

The root cause is that `e8s` is never capped to `old_amount_icp_e8s + max_participant_icp_e8s` before computing `requested_increment_e8s`. Any excess balance — whether from dust or overpayment — inflates `requested_increment_e8s`, which then gets truncated by `max_increment_e8s`, producing a `new_balance_e8s` that is below `min_participant_icp_e8s`. No existing guard prevents a third party from sending ICP to a buyer's subaccount, and no guard prevents `max_increment_e8s` from being smaller than `min_participant_icp_e8s`.

The `error_refund_icp` function at line 1932 confirms that refunds are only available when the swap is `ABORTED` or `COMMITTED`, meaning the victim's ICP is locked for the entire remaining duration of the swap.

## Impact Explanation
This is a **High** severity finding matching: *"Significant SNS security impact with concrete user or protocol harm."*

- A targeted buyer is permanently denied participation in the SNS swap for the swap's remaining duration.
- The victim's ICP is locked in the swap subaccount until the swap closes; it is not permanently lost but is inaccessible during the swap.
- An attacker can selectively block specific buyers (e.g., large participants) to prevent the swap from reaching `min_participants` or `min_direct_participation_icp_e8s`, causing the swap to abort rather than commit — directly harming the SNS project and all legitimate participants.
- The attack is repeatable: every time the victim retries `refresh_buyer_tokens` while the dust remains in the subaccount and `max_increment_e8s < min_participant_icp_e8s`, the call will fail again.

## Likelihood Explanation
- **No privileges required**: The ICP ledger is permissionless; any principal can transfer to any subaccount.
- **Deterministic target**: The victim's subaccount is `principal_to_subaccount(buyer_principal)` on the swap canister — publicly computable from the buyer's principal.
- **Observable trigger condition**: The remaining swap capacity is publicly queryable via `get_derived_state`. An attacker can monitor the swap state and time the attack precisely.
- **Negligible cost**: The attacker spends 10,001 e8s (1 dust e8 + 10,000 e8 ICP transfer fee ≈ 0.0001 ICP) per targeted victim.
- **Realistic end-of-swap condition**: Near the end of a popular SNS swap, `available_direct_participation_e8s` being less than `min_participant_icp_e8s` is a common and expected state.

## Recommendation
Before computing `requested_increment_e8s` at line 1223, cap `e8s` to the buyer's maximum allowable balance:

```rust
// Cap e8s to prevent dust from inflating requested_increment_e8s
let e8s = std::cmp::min(e8s, old_amount_icp_e8s.saturating_add(max_participant_icp_e8s));
```

This ensures that any balance beyond the per-participant cap (whether from dust or overpayment) is silently ignored rather than propagated into the increment logic. The subsequent `max_increment_e8s` cap then operates on a bounded `requested_increment_e8s`, and `new_balance_e8s` will never be less than `min_participant_icp_e8s` unless the buyer genuinely has insufficient funds.

## Proof of Concept

**Setup:**
- SNS swap is `Open` with `min_participant_icp_e8s = 100_000_000`, `max_participant_icp_e8s = 500_000_000`, `max_direct_participation_icp_e8s = 1_000_000_000`.
- Current total direct participation: `999_999_999` e8s, so `max_increment_e8s = 1`.

**Steps:**
1. Victim (new buyer, `old_amount_icp_e8s = 0`) transfers `100_000_000` e8s to their swap subaccount.
2. Attacker sends `1` e8 dust to the same subaccount (address: `principal_to_subaccount(victim_principal)` on the swap canister).
3. Victim calls `refresh_buyer_tokens`.

**Trace through `refresh_buyer_token_e8s`:**
- `e8s = 100_000_001`
- Line 1202: `100_000_001 >= 100_000_000` → passes
- `old_amount_icp_e8s = 0`
- `requested_increment_e8s = 100_000_001`
- `actual_increment_e8s = min(1, 100_000_001) = 1`
- `new_balance_e8s = 0 + 1 = 1`
- Line 1237: `min(1, 500_000_000) = 1`
- Line 1241: `1 < 100_000_000` → **Error: "Rejecting participation of effective amount 1; minimum required to participate: 100000000"**

**Result:** Victim's 100,000,000 e8s are locked in the swap subaccount. Attacker cost: 10,001 e8s. A unit test can reproduce this by constructing a `Swap` state with `direct_participation_icp_e8s = max_direct_participation_icp_e8s - 1`, calling `refresh_buyer_token_e8s` with a mock ledger returning balance `min_participant_icp_e8s + 1`, and asserting the `Err` result.
