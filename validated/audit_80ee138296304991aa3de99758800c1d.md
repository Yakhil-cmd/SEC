Audit Report

## Title
Spawned Neuron Unconditionally Inherits Parent Hot Keys When Controller Changes — (`rs/nns/governance/src/governance.rs`)

## Summary
When `spawn_neuron` is called with a `new_controller` that differs from the parent neuron's controller, the child neuron is created with a verbatim copy of the parent's `hot_keys` list. Those hot keys were authorized by the original controller and were never approved by the new controller. As a result, principals who were hot keys on the parent neuron retain full hot-key privileges (vote, follow, join Neuron Fund) on the child neuron without the new controller's knowledge or consent.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `spawn_neuron` resolves `child_controller` from the optional `spawn.new_controller` field:

```rust
let child_controller = if let Some(child_controller) = &spawn.new_controller {
    *child_controller
} else {
    parent_neuron.controller()
};
``` [1](#0-0) 

The child neuron is then built with `child_controller` as its controller but unconditionally clones the parent's entire `hot_keys` list:

```rust
let child_neuron = NeuronBuilder::new(
    child_nid,
    to_subaccount,
    child_controller,
    ...
)
.with_hot_keys(parent_neuron.hot_keys.clone())   // ← no guard on controller change
.with_followees(parent_neuron.followees.clone())
...
.build();
``` [2](#0-1) 

There is no conditional branch that clears or filters `hot_keys` when `child_controller != parent_neuron.controller()`. The normal hot-key management path (`add_hot_key`) enforces that only the controller can authorize new hot keys: [3](#0-2) 

`spawn_neuron` bypasses this gate entirely by directly cloning the parent's list into the child, so the new controller's consent is never obtained.

## Impact Explanation
Hot keys on an NNS neuron can vote on NNS governance proposals, set followees, join or leave the Neuron Fund, and refresh voting power. A principal (Bob) who was a hot key on the parent neuron retains all of these capabilities on the child neuron after `spawn_neuron` with `new_controller = Dave`, without Dave's approval. Dave can only remediate by discovering and individually removing each inherited hot key via `RemoveHotKey`. This constitutes **unauthorized access to a governance neuron** — matching the High ($2,000–$10,000) bounty impact: *"Unauthorized access to neurons, governance assets … where exploitation requires meaningful per-target work or other constraints."*

## Likelihood Explanation
`spawn_neuron` with `new_controller` is a documented, standard use case (e.g., spawning maturity rewards to a cold-storage principal). Any neuron controller can trigger it via a normal ingress `manage_neuron` call. A parent neuron may legitimately carry multiple hot keys (WebAuthn device keys, hardware wallet keys). The attacker (Bob) needs only to have been a hot key on any neuron whose controller later spawns to a different principal — a realistic and repeatable condition requiring no special privileges beyond being a hot key.

## Recommendation
When `spawn_neuron` is called with a `new_controller` that differs from the parent's controller, the child neuron should be initialized with an empty hot-keys list. The new controller can then add their own hot keys explicitly:

```rust
let child_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};

let child_neuron = NeuronBuilder::new(...)
    .with_hot_keys(child_hot_keys)
    ...
    .build();
``` [2](#0-1) 

## Proof of Concept
1. Alice controls neuron N and adds Bob as a hot key via `AddHotKey`.
2. Alice calls `manage_neuron` → `Spawn { new_controller: Some(Dave), percentage_to_spawn: 100, nonce: None }` on N with sufficient maturity.
3. Governance creates child neuron C with `controller = Dave` and `hot_keys = [Bob]`.
4. Bob submits `manage_neuron` → `RegisterVote { proposal: <any open proposal>, vote: Yes }` for neuron C. The call succeeds because Bob is in `C.hot_keys`.
5. Dave's neuron has voted without Dave's knowledge or consent.
6. Alternatively, Bob calls `manage_neuron` → `Follow { topic: 0, followees: [Bob's neuron] }` on C, redirecting all of Dave's automatic votes to Bob's neuron.

A deterministic integration test can be written using PocketIC: create two principals (Alice/Dave), stake a neuron for Alice, add Bob as hot key, call `spawn_neuron` with `new_controller = Dave`, then assert that Bob's `RegisterVote` or `Follow` call on the child neuron returns `Ok` — confirming the unauthorized access.
