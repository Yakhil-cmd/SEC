Audit Report

## Title
Global One-at-a-Time `CreateServiceNervousSystem` Restriction Enables Indefinite DoS of SNS Creation - (`File: rs/nns/governance/src/governance.rs`)

## Summary

The `validate_create_service_nervous_system` function in NNS governance enforces a global, type-level uniqueness constraint: if any non-final `CreateServiceNervousSystem` proposal exists, all new ones are rejected with `PreconditionFailed`. Because this check is keyed only on proposal type and not on the proposer's identity, any single NNS neuron holder meeting the minimum stake and dissolve-delay requirements can submit a malicious proposal to block all legitimate SNS creation proposals for the duration of each voting period, and repeat the attack indefinitely at low cost.

## Finding Description

`validate_create_service_nervous_system` in `rs/nns/governance/src/governance.rs` calls `select_nonfinal_proposal_ids` to enumerate all open proposals of type `Action::CreateServiceNervousSystem`, and if any exist, immediately returns `PreconditionFailed`: [1](#0-0) 

The predicate passed to `select_nonfinal_proposal_ids` matches solely on action type: [2](#0-1) 

There is no check on the proposer's `NeuronId`, no per-neuron cooldown, and no mechanism to remove or supersede the blocking proposal. The `make_proposal` entry point calls `validate_proposal` (which calls `validate_proposal_action`, which calls `validate_create_service_nervous_system`) before any state mutation occurs, so the block is enforced atomically on every submission attempt. [3](#0-2) 

The integration test in `rs/nns/integration_tests/src/create_service_nervous_system.rs` explicitly confirms the blocking behavior: the second proposal fails with `ErrorType::PreconditionFailed` and an error message containing `"another open"`: [4](#0-3) 

**Exploit flow:**
1. Attacker holds an NNS neuron with ≥6 months dissolve delay and ≥`reject_cost_e8s` (currently 10 ICP) minted stake.
2. Attacker submits a `CreateServiceNervousSystem` proposal with adversarial parameters (wrong `dapp_canisters`, misleading description, or parameters designed to fail execution).
3. All subsequent legitimate `CreateServiceNervousSystem` proposals from any neuron are rejected with `PreconditionFailed` for the entire voting period (minimum 4 days).
4. After the voting period ends and the malicious proposal is rejected, the attacker immediately re-submits. The cycle repeats indefinitely.

No existing guard prevents this: the only checks before the uniqueness test are proposal format validity, neuron authorization, stake sufficiency, and dissolve delay — all of which the attacker satisfies legitimately.

## Impact Explanation

This is a **High** severity finding matching: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."* An attacker can permanently prevent any new SNS from being launched through the NNS governance proposal process for as long as the attack is maintained. The attacker does not need to win the vote; they only need the proposal to remain open. The neuron stake is preserved throughout, and voting rewards continue to accrue, making the net cost per attack cycle only the `reject_cost_e8s` (10 ICP) — a negligible sum relative to the impact of blocking an SNS launch that may represent millions of ICP in fundraising.

## Likelihood Explanation

The NNS has thousands of neurons meeting the minimum requirements (≥6 months dissolve delay, ≥10 ICP stake). The attack requires only one standard `manage_neuron` ingress call every few days. A motivated attacker — such as a competitor to a specific SNS project — has clear economic incentive and faces no increasing cost, no cooldown, and no technical barrier beyond holding a qualifying neuron.

## Recommendation

1. **Include the proposer's `NeuronId` in the uniqueness check**: allow at most one open `CreateServiceNervousSystem` proposal *per neuron*, so a malicious proposal from one neuron does not block proposals from all other neurons.
2. **Allow multiple concurrent proposals and resolve conflicts at execution time**: the first adopted proposal executes; subsequent ones fail gracefully with a refund.
3. **Add a per-neuron post-rejection cooldown** for `CreateServiceNervousSystem` proposals to raise the cost of repeated attacks.

## Proof of Concept

The existing integration test at `rs/nns/integration_tests/src/create_service_nervous_system.rs` lines 37–102 already demonstrates the blocking behavior deterministically. [5](#0-4) 

To demonstrate the DoS specifically:
1. Run the test environment with production governance (no `feature = "test"` flag, which bypasses the uniqueness check at line 5069).
2. Submit a `CreateServiceNervousSystem` proposal from neuron A with an invalid `dapp_canisters` field (a canister not controlled by NNS root).
3. Attempt to submit a legitimate `CreateServiceNervousSystem` proposal from neuron B — observe `PreconditionFailed` with `"another open"`.
4. Wait for the voting period to expire and the first proposal to be rejected.
5. Immediately re-submit from neuron A — the block is reinstated.

The `#[cfg(not(feature = "test"))]` guard at line 5069 confirms the restriction is active in production builds only, making local reproduction straightforward by removing that cfg gate in a test build. [6](#0-5)
