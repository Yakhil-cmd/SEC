Audit Report

## Title
Caller Identity Not Verified in `refresh_buyer_tokens` Allows Forced SNS Swap Participation and Confirmation Text Bypass - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal in its request argument and uses it directly without verifying it matches `caller()`. Any unprivileged ingress sender can call this method with a victim's principal as the `buyer` field, supplying the public confirmation text on the victim's behalf, and thereby register the victim's ICP deposit as confirmed participation — bypassing the SNS-configured confirmation text consent mechanism. If the swap commits, the victim's ICP is swept to the SNS treasury and they receive SNS tokens without ever having consented.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 127–143, the `refresh_buyer_tokens` endpoint resolves the effective buyer principal from the caller-supplied `arg.buyer` string with no check that `arg.buyer == caller()`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // ← no caller == buyer check
};
``` [1](#0-0) 

The resolved principal `p` is passed directly to `refresh_buyer_token_e8s`, which (1) validates the caller-supplied `confirmation_text` against the SNS-configured text, (2) reads the ICP balance from the subaccount derived from `buyer`, and (3) writes the buyer into `self.buyers`:

```rust
self.validate_confirmation_text(confirmation_text)?;
// ...
let account = Account {
    owner: this_canister.get().0,
    subaccount: Some(principal_to_subaccount(&buyer)),
};
``` [2](#0-1) [3](#0-2) 

The `RefreshBuyerTokensRequest` proto explicitly documents the `buyer` field as an optional override defaulting to the caller, with no authorization guard: [4](#0-3) 

Once a buyer is registered in `self.buyers` with `transfer_success_timestamp_seconds == 0`, `error_refund_icp` blocks any refund with a precondition error: [5](#0-4) 

When the swap commits, `sweep_icp` transfers all registered buyers' ICP to the SNS governance treasury, making the outcome irreversible: [6](#0-5) 

## Impact Explanation
The confirmation text is an explicit per-participant consent signal required by the SNS before accepting ICP. Because the text is set in public SNS initialization parameters and visible on-chain, any attacker who observes a victim's ICP deposit in the swap subaccount can call `refresh_buyer_tokens` with the victim's principal and the correct confirmation text, registering the victim as a confirmed participant without the victim's knowledge or action. If the swap reaches `COMMITTED`, the victim's ICP is permanently swept to the SNS treasury and replaced with SNS tokens — an irreversible financial outcome the victim never consented to. This constitutes a **High** impact: significant SNS security impact with concrete, irreversible user-funds harm, matching the allowed impact class "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The attack requires no special privilege, no key material, and no majority. The only precondition is that the victim has transferred ICP to the swap subaccount — a normal, observable on-chain action. The confirmation text is public. The call is a standard ingress update callable by any principal. The attack is repeatable for every victim who has deposited ICP but not yet called `refresh_buyer_tokens` themselves.

## Recommendation
Add a caller authorization check in the canister endpoint before using the `buyer` field:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Caller {} is not authorized to refresh tokens for buyer {}", caller, buyer);
        }
        buyer
    };
    // ...
}
```

Alternatively, remove the `buyer` override entirely and always derive the principal from `caller()`, since the only legitimate use case is a participant registering their own deposit. If third-party notification is intentionally desired (e.g., for bots), the confirmation text field must be removed from the third-party path or the confirmation text must be treated as a per-call secret rather than a public string.

## Proof of Concept
1. An SNS swap is open with `confirmation_text = "I agree to the terms"` set in its init parameters (publicly visible on-chain).
2. Alice transfers ICP to the SNS Swap canister subaccount derived from her principal (the normal first step of participation). She has not yet called `refresh_buyer_tokens`.
3. Eve (any principal) calls:
   ```
   refresh_buyer_tokens({
     buyer: "<Alice's principal text>",
     confirmation_text: Some("I agree to the terms")
   })
   ```
4. The swap canister reads Alice's subaccount balance, validates the confirmation text (supplied by Eve), and records Alice in `self.buyers` with her ICP amount.
5. Alice's `error_refund_icp` call is now blocked with "ICP cannot be refunded as principal … has … ICP (e8s) in escrow."
6. When the swap commits, `sweep_icp` transfers Alice's ICP to the SNS treasury and she receives SNS tokens — without ever having called `refresh_buyer_tokens` herself or consented to participation.

A deterministic integration test using `PocketIC` or `StateMachine` can reproduce this by: (a) opening a swap with a confirmation text, (b) crediting Alice's subaccount on the mock ICP ledger, (c) calling `refresh_buyer_tokens` as Eve with Alice's principal and the correct confirmation text, (d) asserting Alice appears in `self.buyers`, and (e) asserting `error_refund_icp` returns a precondition error for Alice.
