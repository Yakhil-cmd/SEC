Audit Report

## Title
Unbounded Loop in `distribute_rewards` Can Exhaust Instruction Limit, Permanently Blocking SNS Reward Distribution - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's `distribute_rewards` function iterates over all ready-to-settle proposals, all ballots per proposal, and all rewarded neurons in a single synchronous call with no instruction-limit guard. If the accumulated set of proposals and voting neurons is large enough, the function traps on the instruction limit every time it is invoked. Because `latest_reward_event` is only updated on successful completion, `should_distribute_rewards()` returns `true` again on the next heartbeat, causing a permanent livelock where voting rewards are never distributed.

## Finding Description
`distribute_rewards` at `rs/sns/governance/src/governance.rs:5763` is called synchronously from `run_periodic_tasks` (the canister heartbeat) at line 5513 whenever `should_distribute_rewards()` returns true. The function contains three sequential unbounded loops with no instruction-limit check anywhere in `rs/sns/governance/`:

1. **Outer loop over all ready-to-settle proposals** (line 5894): `for proposal_id in &considered_proposals`
2. **Inner loop over all ballots per proposal** (line 5896): `for (voter, ballot) in &proposal.ballots`
3. **Final loop over all rewarded neurons** (line 5954): `for (neuron_id, neuron_reward_shares) in neuron_id_to_reward_shares`

The `latest_reward_event` is only updated after all three loops complete (line 6011). If the message traps on the instruction limit, the state rolls back entirely, `latest_reward_event` is unchanged, and the next heartbeat re-enters the same path and traps again — indefinitely.

This is in direct contrast to the NNS governance canister, which was explicitly fixed: `rs/nns/governance/src/reward/distribution.rs:43` uses `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` inside `continue_processing` (line 184), and `rs/nns/governance/src/timer_tasks/distribute_rewards.rs:43-55` drains the queue in bounded batches via a periodic timer task running every 2 seconds.

A grep search for `is_message_over_threshold`, `instruction_limit`, and `DISTRIBUTION_MESSAGE_LIMIT` across all of `rs/sns/governance/` returns zero matches, confirming no guard exists.

## Impact Explanation
This matches the **High ($2,000–$10,000)** impact class: *Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS.* Concretely: voting rewards are permanently frozen for the affected SNS. Neuron maturity never increases, breaking the core economic incentive of the SNS staking model. The canister itself remains alive but its reward mechanism is effectively dead for as long as the proposal/ballot set remains large. This is a permanent, self-reinforcing failure state, not a transient one.

## Likelihood Explanation
Any SNS token holder (unprivileged) can submit proposals and vote on them. A moderately active SNS with 1,000 neurons and 50 proposals settling in the same reward round produces 50,000 ballot iterations plus 1,000 neuron maturity writes. SNS neurons are stored in heap memory (unlike NNS neurons which use stable structures), but the Decimal arithmetic per ballot and the HashMap operations still consume significant instructions at scale. The condition is reachable organically through normal governance activity, not just adversarial action, and once triggered it is self-sustaining.

## Recommendation
Apply the same batching pattern already implemented in NNS governance:

1. In `distribute_rewards`, compute per-neuron reward shares and store them in a persistent queue (analogous to `RewardsDistribution` in `rs/nns/governance/src/reward/distribution.rs`).
2. Introduce a separate periodic timer task (analogous to `DistributeRewardsTask` in `rs/nns/governance/src/timer_tasks/distribute_rewards.rs`) that drains the queue in bounded batches, checking `is_message_over_threshold` after each neuron write and breaking when the limit is approached.
3. Only advance `latest_reward_event` after the queue is fully drained, or use a two-phase commit (mark event as "pending distribution" first, then "settled" after the queue empties).

## Proof of Concept
1. Deploy an SNS with 2,000+ neurons.
2. Submit 30+ proposals and have neurons vote on all of them (30 × 2,000 = 60,000 ballot entries).
3. Allow proposals to pass their voting period, entering `ReadyToSettle` state.
4. When the reward round ends, `run_periodic_tasks` calls `should_distribute_rewards()` → `true`, then calls `distribute_rewards(supply)` at line 5513.
5. `distribute_rewards` enters the nested loops at lines 5894/5896/5954 and traps on the instruction limit.
6. State rolls back; `latest_reward_event` is unchanged.
7. On the next heartbeat, steps 4–6 repeat indefinitely. No rewards are ever distributed.

A deterministic integration test using PocketIC can reproduce this by: creating an SNS with a large neuron/proposal set, advancing the clock past a reward round boundary, triggering heartbeats, and asserting that `latest_reward_event` never advances and neuron maturity never increases.
