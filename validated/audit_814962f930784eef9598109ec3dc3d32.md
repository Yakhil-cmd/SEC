### Title
Silent Failure of Exit Precompile Call After Token Burn Causes Permanent Fund Loss - (File: etc/eth-contracts/contracts/EvmErc20.sol)

### Summary
In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` burn the caller's ERC-20 tokens **before** invoking the exit precompile via an inline assembly `call`. The return value of that assembly call is captured in `res` but is never inspected. If the precompile call fails for any reason, the EVM transaction still succeeds, the tokens are permanently destroyed, and no corresponding NEAR-side release is ever scheduled — resulting in irreversible fund loss for the user.

### Finding Description

`withdrawToNear` in `EvmErc20.sol` follows this sequence:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum`: [2](#0-1) 

On the Rust side, the `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return an `ExitError` in several reachable paths:

- `parse_recipient` fails if the `recipient` bytes are not valid UTF-8 or do not form a valid NEAR account ID.
- `validate_input_size` fails if `recipient.length` causes the total input to exceed `MAX_INPUT_SIZE = 1024`.
- `get_nep141_from_erc20` fails if the ERC-20 → NEP-141 mapping is absent. [3](#0-2) [4](#0-3) 

When the precompile returns `ExitError`, the EVM `call` opcode returns `0` to the caller. Because `res` is never tested, the Solidity function does **not** revert. The `_burn` state change is committed, but no `PromiseArgs` log is emitted, so no NEAR-side `ft_transfer` or `near_withdraw` promise is ever created.

The existing `error_refund` callback mechanism (`exit_to_near_precompile_callback`) only fires when a promise was successfully **created** but later **failed** on the NEAR side. It provides no protection when the precompile itself fails before any promise is scheduled. [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's ERC-20 tokens are irreversibly burned inside the EVM. Because no exit promise is scheduled, the corresponding NEP-141 balance on the NEAR side is never released. The funds are permanently destroyed with no recovery path available to the user or the protocol.

### Likelihood Explanation

**Medium.**

The `recipient` parameter of `withdrawToNear` is typed as `bytes memory`, accepting arbitrary byte sequences. Any of the following realistic inputs trigger the silent failure:

- A NEAR account ID containing uppercase letters or other disallowed characters (NEAR account IDs are lowercase only).
- Raw Ethereum address bytes (20 bytes of binary data) passed by mistake instead of a NEAR account string.
- A recipient string longer than ~991 bytes, exceeding `MAX_INPUT_SIZE`.
- Any byte sequence that is not valid UTF-8.

These are plausible user errors, especially for developers integrating the contract programmatically. No privileged access is required; any token holder can trigger the loss on their own funds.

### Recommendation

Check the assembly `call` return value and revert on failure in both withdrawal functions:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, restructure the functions to validate the recipient and invoke the precompile **before** burning tokens, so that a failed precompile call causes the entire transaction to revert with no state change.

### Proof of Concept

1. Deploy `EvmErc20` and mint `1000` tokens to Alice.
2. Alice calls `withdrawToNear(bytes(hex"deadbeef"), 1000)` — passing 4 bytes of non-UTF-8 binary data as the recipient.
3. `_burn(Alice, 1000)` executes; Alice's EVM balance drops to 0.
4. The assembly `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` invokes `ExitToNear::run`, which calls `parse_recipient` on `0xdeadbeef`. `str::from_utf8` fails → `ExitError` is returned → the EVM `call` returns `0`.
5. `res == 0` is never checked; the Solidity function returns normally.
6. The EVM transaction is committed: Alice has 0 tokens, no NEAR-side promise was scheduled, and no NEP-141 tokens are released.
7. Alice's 1000 tokens are permanently destroyed. [1](#0-0) [6](#0-5)

fn validate_input_size(input: &[u8], min: usize, max: usize) -> Result<(), ExitError> {
    if input.len() < min || input.len() > max {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_INPUT")));
    }
    Ok(())
}

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
