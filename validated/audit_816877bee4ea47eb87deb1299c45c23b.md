### Title
Tautological Safety Guard in `compute_gas_refund` Checks Wrong Variable — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

`compute_gas_refund` contains a safety guard that checks `total_gas_refund <= gas_limit`, but since `total_gas_refund` is defined as `gas_limit - gas_used` (both `u64`), this condition is a mathematical tautology — it is always true whenever `gas_used >= 0`, which is guaranteed by the unsigned type. The check never catches the actual invariant it is meant to enforce (`gas_used <= gas_limit`), mirroring the original report's pattern of checking the derived value instead of the source value.

---

### Finding Description

In `compute_gas_refund`, after all adjustments to `gas_used` (EVM refund, `minimal_gas_used` floor, and native `delta_gas` bump), the code computes:

```rust
let total_gas_refund = gas_limit - gas_used;   // line 83
require_internal!(
    total_gas_refund <= gas_limit,             // line 86
    "Gas refund greater than gas limit",
    system
)?;
``` [1](#0-0) 

Because `total_gas_refund = gas_limit - gas_used` and both operands are `u64`, the expression `total_gas_refund <= gas_limit` reduces to `gas_limit - gas_used <= gas_limit`, which is equivalent to `gas_used >= 0` — always true for an unsigned integer. The guard is a no-op in every non-overflow execution path.

The intended invariant is `gas_used <= gas_limit`. The check should be placed **before** the subtraction:

```rust
require_internal!(gas_used <= gas_limit, "gas_used exceeds gas_limit", system)?;
let total_gas_refund = gas_limit - gas_used;
```

The misleading comment in the L1 transaction flow reinforces that callers believe this check is meaningful:

```rust
// We already checked that total_gas_refund <= gas_limit
let pay_to_operator = U256::from(gas_used)
    .checked_mul(U256::from(gas_price))
    ...
``` [2](#0-1) 

`compute_gas_refund` is called from three sites: the ZK L2 transaction flow (`before_refund`), the Ethereum L2 transaction flow (`before_refund`), and the L1 transaction processor. [3](#0-2) [4](#0-3) [5](#0-4) 

---

### Impact Explanation

`gas_used` can exceed `gas_limit` through two paths inside `compute_gas_refund`:

1. **`minimal_gas_used` floor** (line 56): `gas_used = max(gas_used, minimal_gas_used)`. For L1 transactions, `minimal_gas_used` is explicitly capped at `gas_limit` before the call (line 537 of `process_l1_transaction.rs`), but for L2 ZK transactions the cap is only enforced when the `eip_7623` feature flag is active. [6](#0-5) 

2. **Native `delta_gas` bump** (line 78): `gas_used += delta_gas`. If `native_used / native_per_gas` exceeds the ergs-derived `gas_used`, the bump can push `gas_used` above `gas_limit`. [7](#0-6) 

When `gas_used > gas_limit`, the subtraction `gas_limit - gas_used` on line 83 **underflows**. In a release (non-debug) build this wraps to a very large `u64`, causing the tautological check to accidentally fire and return an `InternalError`. In a debug build it panics. Either outcome halts block processing for the affected transaction, constituting a resource-accounting / state-transition bug reachable by any transaction sender who can drive high native resource consumption (e.g., via pubdata-heavy storage writes).

The downstream consequence is incorrect fee accounting: `pay_to_operator` and the refund amount are computed from the corrupted `gas_used` value, potentially over-charging or under-refunding users, or causing the block to abort with an internal error.

---

### Likelihood Explanation

The `delta_gas` path is reachable by any unprivileged L2 transaction sender: a transaction that consumes native resources disproportionately to its EVM gas (e.g., heavy pubdata writes with a low `native_price`) can produce `native_used / native_per_gas > gas_limit`, triggering the overflow. The `minimal_gas_used` path is reachable when `eip_7623` is disabled (the default compile-time configuration). Both paths are exercised in normal transaction processing.

---

### Recommendation

Replace the tautological post-subtraction check with a pre-subtraction guard on the correct variable:

```diff
-    let total_gas_refund = gas_limit - gas_used;
-    require_internal!(
-        total_gas_refund <= gas_limit,
-        "Gas refund greater than gas limit",
-        system
-    )?;
+    require_internal!(
+        gas_used <= gas_limit,
+        "gas_used exceeds gas_limit",
+        system
+    )?;
+    let total_gas_refund = gas_limit - gas_used;
```

Additionally, remove the misleading comment in `process_l1_transaction.rs` that asserts the old check was meaningful.

---

### Proof of Concept

Consider an L2 ZK transaction with:
- `gas_limit = 100_000`
- `native_per_gas = 1`
- `full_native_limit = 100_000`
- After execution, `resources.native().remaining() = 0` → `native_used = 100_000`
- EVM ergs remaining = 50_000 ergs → `gas_used = 100_000 - (50_000 / ERGS_PER_GAS)`
  With `ERGS_PER_GAS = 256`: `gas_used = 100_000 - 195 = 99_805`
- `evm_refund = 0` (no SSTORE refunds)
- `minimal_gas_used = 0` (eip_7623 disabled)
- `delta_gas = (100_000 / 1) - 99_805 = 195` → `gas_used = 100_000`

In this normal case `gas_used == gas_limit`, so `total_gas_refund = 0` and the tautological check passes trivially. Now increase native consumption slightly beyond the limit (e.g., via `charge_unchecked` on pubdata) so `native_used = 100_001`:

- `delta_gas = 100_001 - 99_805 = 196` → `gas_used = 100_001 > gas_limit`
- `gas_limit - gas_used` underflows → wraps to `u64::MAX - 0` in release
- `total_gas_refund <= gas_limit` → `u64::MAX <= 100_000` → **false** → `InternalError` returned

The tautological check accidentally fires via underflow rather than catching the invariant violation directly, confirming the wrong-variable comparison. [8](#0-7)

pub(crate) fn compute_gas_refund<S: EthereumLikeTypes>(
    system: &mut System<S>,
    to_charge_for_pubdata: S::Resources,
    gas_limit: u64,
    minimal_gas_used: u64,
    native_per_gas: u64,
    resources: &mut S::Resources,
) -> Result<RefundInfo, InternalError> {
    // Already checked
    resources.charge_unchecked(&to_charge_for_pubdata);

    let mut gas_used = gas_limit
        .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
        .ok_or(internal_error!("gas remaining > gas limit"))?;
    resources.exhaust_ergs();

    system_log!(system, "Gas used before refund calculations: {gas_used}\n");

    // Following EIP-3529, refunds are capped to 1/5 of the gas used
    let evm_refund = {
        let full_refund_ergs = system.io.get_refund_counter().ergs();
        let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
        let max_refund = gas_used / 5;
        core::cmp::min(full_refund_gas, max_refund)
    };

    system_log!(system, "Gas refund from refund counters = {evm_refund}\n");

    gas_used -= evm_refund;

    system_log!(
        system,
        "Minimal gas used from validation = {minimal_gas_used}\n"
    );

    #[allow(unused_mut)]
    let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

    // Note: for zero gas price, we use "unlimited native"
    let full_native_limit = if cfg!(feature = "unlimited_native") || native_per_gas == 0 {
        u64::MAX - 1
    } else {
        gas_limit.saturating_mul(native_per_gas)
    };
    let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());

    #[cfg(not(feature = "unlimited_native"))]
    {
        // Adjust gas_used with difference with used native
        let delta_gas = if native_per_gas == 0 {
            0
        } else {
            (native_used / native_per_gas) as i64 - (gas_used as i64)
        };

        if delta_gas > 0 {
            // In this case, the native resource consumption is more than the
            // gas consumption accounted for. Consume extra gas.
            gas_used += delta_gas as u64;
        }
        // TODO: return delta_gas to gas_used?
    }

    let total_gas_refund = gas_limit - gas_used;
    system_log!(system, "Refund after accounting for unused gas, refund counters and native cost: {total_gas_refund}\n");
    require_internal!(
        total_gas_refund <= gas_limit,
        "Gas refund greater than gas limit",
        system
    )?;
    let refund_info = RefundInfo {
        gas_used,
        evm_refund,
        native_used,
    };
    system_log!(system, "Final gas used: {gas_used}\n");
    Ok(refund_info)
}
