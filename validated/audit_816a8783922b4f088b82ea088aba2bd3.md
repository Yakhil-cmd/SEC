Audit Report

## Title
Sandbox RSS Eviction Permanently Blind to Page-Allocator (Stable Memory) RSS — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

## Summary
In `monitor_and_evict_sandbox_processes`, only the anonymous RSS (`get_anon_rss`) is pushed into `sandbox_processes_rss` and subsequently written into each backend's `stats.rss`. The page-allocator RSS (`get_page_allocator_rss`), which accounts for all memfd/file-backed stable memory, is measured for histogram metrics only and never stored in `stats.rss`. As a result, `total_sandboxes_rss` permanently undercounts actual sandbox memory, and the RSS-based eviction gate in `trigger_sandbox_eviction` can never fire for canisters whose memory footprint is dominated by stable memory, enabling a canister developer to drive a node toward OOM and cause subnet unavailability.

## Finding Description
At lines 1398–1405, `get_anon_rss` is called and its result is pushed to `sandbox_processes_rss`: [1](#0-0) 

At lines 1409–1417, `get_page_allocator_rss` is called but its result is only accumulated into `total_memfd_rss` and `process_rss` (used for histogram metrics). It is never appended to `sandbox_processes_rss`: [2](#0-1) 

`update_sandbox_processes_rss` (lines 2009–2018) writes only those anon-RSS values into `stats.rss`: [3](#0-2) 

`total_sandboxes_rss` (lines 2022–2030) sums only `stats.rss` for active backends, which contains only anon RSS: [4](#0-3) 

The RSS-based eviction gate in `trigger_sandbox_eviction` (lines 1542–1560) checks `total_sandboxes_rss > self.max_sandboxes_rss()`, which is permanently near-zero for stable-memory-heavy canisters: [5](#0-4) 

Additionally, the background monitoring thread (lines 1482–1490) passes `u64::MAX` as `max_sandboxes_rss`, so it never triggers RSS-based eviction at all — that path is exclusively in `trigger_sandbox_eviction`, which is broken: [6](#0-5) 

The count-based eviction path (`active_sandboxes > max_sandbox_count`) is unaffected but does not bound per-sandbox memory consumption.

## Impact Explanation
A canister with large stable memory (up to the 400 GiB per-canister protocol maximum) but minimal heap will have near-zero anon RSS. Its `stats.rss` will be near zero, `total_sandboxes_rss` will be near zero, and the condition `total_sandboxes_rss > max_sandboxes_rss` will never be satisfied. The IC's own eviction mechanism will not proactively evict such sandboxes based on memory pressure. Under sustained stable-memory allocation across one or more canisters, the node can be driven into OOM conditions, crashing the replica process and causing subnet unavailability. This matches the **High** bounty impact: *Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS*.

## Likelihood Explanation
Any unprivileged canister developer can deploy a canister that calls `stable_grow` and writes to every page to force stable memory into RSS. The cost is cycles, which are purchasable. The effect is deterministic: once stable memory is allocated and paged in, the sandbox process's anon RSS remains small while its page-allocator RSS is large, and the eviction gate never fires. No special privileges, victim mistakes, or external dependencies are required.

## Recommendation
In the loop inside `monitor_and_evict_sandbox_processes`, accumulate both anon RSS and page-allocator RSS before pushing to `sandbox_processes_rss`:

```rust
let mut total_rss_kib: u64 = 0;
if let Ok(kib) = process_os_metrics::get_anon_rss(pid) {
    total_rss_kib += kib;
    total_anon_rss += kib;
    process_rss += kib;
    metrics.sandboxed_execution_subprocess_anon_rss.observe(kib as f64);
}
if let Ok(kib) = process_os_metrics::get_page_allocator_rss(pid) {
    total_rss_kib += kib;
    total_memfd_rss += kib;
    process_rss += kib;
    metrics.sandboxed_execution_subprocess_memfd_rss.observe(kib as f64);
}
if total_rss_kib > 0 {
    sandbox_processes_rss.push((*canister_id, NumBytes::new(total_rss_kib * 1024)));
}
```

This ensures `stats.rss` and therefore `total_sandboxes_rss` reflect actual sandbox memory consumption including stable memory.

## Proof of Concept
1. Deploy a canister that calls `stable_grow` to allocate N GiB of stable memory and writes to every page to force it into RSS.
2. Configure `max_sandboxes_rss` to a value smaller than N GiB.
3. Mock `available_memory` to return a value below `DEFAULT_MIN_MEM_AVAILABLE_TO_EVICT_SANDBOXES` (250 GiB).
4. Call `trigger_sandbox_eviction`.
5. Observe: no eviction occurs because `total_sandboxes_rss` (anon-only) is near zero, so the first condition of the AND-gate is false.
6. Patch `sandbox_processes_rss.push` to also include page-allocator RSS and repeat: eviction now fires correctly.

fn total_sandboxes_rss(backends: &HashMap<CanisterId, Backend>) -> NumBytes {
    backends
        .values()
        .map(|backend| match backend {
            Backend::Active { stats, .. } => stats.rss,
            Backend::Evicted { .. } | Backend::Empty => 0.into(),
